## Summary

Keep open: this PR proposes a new external bootstrap-file trust boundary, while current main still intentionally enforces workspace-only guarded reads. It also lacks inspectable real behavior proof and has blocking correctness/security review findings.

## What This Changes

The PR adds a `bootstrapTrustExternal` opt-in to `bootstrap-extra-files` for reading configured bootstrap paths outside the workspace, plus an unrelated `extensions/runs-registry` lockfile importer.

## Best Possible Solution

If maintainers want this capability, ship it as a documented trust-boundary feature with byte-capped opened reads, default-rejection and opt-in coverage, docs/changelog, and no unrelated lockfile drift; otherwise keep the workspace-only contract.

## Reproduction Assessment

Not applicable; this PR proposes a new opt-in capability, not a bug with a failing current-main reproduction path. Current main is source-verifiable as workspace-only and has regression tests for rejecting outside paths.

## Solution Assessment

No/unclear as written; the direct external `fs.readFile` path, undocumented config key, and unrelated lockfile importer are not the narrowest maintainable way to change this trust boundary. A safer version would preserve guarded byte enforcement and document/test the explicit opt-in.

  - confidence: 0.9
