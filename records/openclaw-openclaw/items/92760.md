## Summary

Keep open: current main and the latest release still route CLI status context-window resolution through the cache-only resolver, while open closing PRs https://github.com/openclaw/openclaw/pull/92930 and https://github.com/openclaw/openclaw/pull/94210 are active fix candidates. This issue should close only after one candidate merges or maintainers choose a different canonical fix.

## What This Changes

The issue asks for CLI status context-window resolution to use the shared agent context resolver instead of a cache-only status runtime path that can fall back to 200K.

## Best Possible Solution

Review and land one focused canonical fix that makes CLI status resolve discovered model context windows through the shared context path, then close this issue through the merged PR.

## Reproduction Assessment

Yes from source inspection, though not from a fresh live command: current main routes CLI status through a cache-only resolver and passes `allowAsyncLoad: false`, so discovery-only model windows can still fall back to `DEFAULT_CONTEXT_TOKENS`. The open candidate PRs also include after-fix behavior claims for the same path.

## Solution Assessment

Yes, delegating CLI status to the shared context resolver or otherwise making one canonical status path warm/read discovered context windows is the narrow maintainable fix. Because two open closing PRs already target the issue, the best next step is maintainer review of those PRs rather than a new repair job.
