## Summary

Current main and the latest release still contain the reported status/context resolver split, and this issue has open closing PR candidates. Keep it open until maintainers land or reject the canonical fix path rather than closing it early.

## What This Changes

The issue reports that CLI `openclaw status` can show a 200k context window for discovered models because the CLI status summary runtime uses the cache-only context resolver path.

## Best Possible Solution

Keep this issue open and resolve it by landing a focused status/context fix that preserves read-only startup behavior while proving `openclaw status` reports discovered model context windows correctly on first use.

## Reproduction Assessment

Yes, by source inspection rather than a live CLI run: current main routes CLI status through the cache-only resolver, passes `allowAsyncLoad: false`, and skips cache loading when `cfg` is supplied. I did not run a provider-backed `openclaw status` repro in this read-only cleanup review.

## Solution Assessment

The right fix is to unify status context resolution with the shared resolver/cache path, but import delegation alone is not enough on current main. The best path is a focused PR that fixes the cfg-backed cache-warming/status contract and includes real redacted CLI status proof.
