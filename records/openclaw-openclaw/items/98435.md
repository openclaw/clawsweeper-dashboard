## Summary

Keep open: current main still has a credible restart-recovery gap where session content can be resumed while the Claude CLI bundle-MCP bridge is not proven reconnected, and no merged PR or current release evidence closes this exact post-restart loopback failure.

## What This Changes

The issue reports that gateway restart recovery restores a Claude CLI-backed session but leaves the OpenClaw MCP loopback transport stale until the user manually reconnects it.

## Best Possible Solution

Preserve Claude CLI conversation reuse while adding an explicit bundle-MCP reconnection/reload signal or downgrading recovery status until post-restart MCP tool readiness is proven.

Yes at source/field-report level, but not with a live destructive replay in this read-only pass. Current source can report `recovered=1` after scheduling the resume while the stable CLI MCP resume path intentionally avoids invalidating the session on loopback port churn.

Unclear: the suggested supervisor/reconnect approach is plausible, but maintainers need to choose the boundary that preserves Claude CLI memory without keeping a stale MCP client after gateway restart.
