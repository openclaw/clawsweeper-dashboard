## Summary

Keep open: current main still reports main-session restart recovery independently of CLI-side MCP transport readiness, and I found no merged fix, exact duplicate, or current release evidence that closes the post-restart loopback reconnect gap.

## What This Changes

The issue requests automatic MCP loopback reconnection or clearer partial-recovery semantics after a gateway restart so recovered sessions do not fail the next CLI-side MCP tool call.

## Best Possible Solution

Preserve Claude CLI conversation reuse while adding an explicit bundle-MCP reconnect/readiness contract, or downgrade recovery status until post-restart MCP tool readiness is proven.
