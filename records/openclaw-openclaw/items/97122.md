## Summary

Keep this maintainer PR open for normal release-owner review. The workflow change is focused and useful, but the patched docs advertise `openclaw/openclaw:latest` while the live Docker Hub repository currently has no tags, and the release workflow now becomes fail-closed on Docker Hub credentials and registry availability.

## What This Changes

The branch adds Docker Hub publishing, manifests, attestation verification, docs, and string-level workflow coverage to the Docker release path while keeping GHCR as the primary registry.

## Best Possible Solution

Land the workflow only after release owners either publish/backfill the initial Docker Hub tags before docs go live or change the docs to describe the mirror as pending until the first successful publish.

## Reproduction Assessment

Yes for the PR finding: live Docker Hub metadata currently reports zero tags for `openclaw/openclaw`, while the patched docs advertise `openclaw/openclaw:latest`. This is an after-patch documentation/runtime mismatch, not a current-main bug.

## Solution Assessment

No, not as-is. The workflow belongs in the right release surface, but the docs should not advertise a pull command until the first Docker Hub tags exist or the text clearly states that the mirror becomes available after the first successful backfill/release.
