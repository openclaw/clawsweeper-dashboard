## Summary

Keep open. Current main still forces every isolated cron-style agent run to create a fresh session, and this PR is a narrow implementation for the linked webhook session regression; the only concrete review finding is the missing required changelog entry for a user-facing fix.

## What This Changes

The PR changes isolated cron session resolution so hook/webhook-sourced isolated agent runs do not force a new session, and adds a regression test for webhook session reuse.

## Best Possible Solution

Land this PR or a narrow equivalent after adding the changelog entry: hook/webhook-sourced isolated agent turns with stable, policy-approved hook provenance should reuse fresh stored sessions, while ordinary scheduled isolated cron runs should keep fresh-session behavior.

## Reproduction Assessment

Yes. A direct /hooks/agent request with a stable permitted sessionKey reaches dispatchAgentHook, becomes an isolated CronJob, and current main passes forceNew: true, causing resolveCronSession to bypass the fresh-session reuse branch.

## Solution Assessment

Yes for the functional code path. The PR uses existing hook provenance and preserves ordinary isolated-cron freshness; the safer remaining adjustment is process-only, adding the required changelog entry rather than changing the implementation shape.
