## Summary

Current main implements the requested behavior through the runtime catalog: explicit `models.providers.<provider>.models[]` entries are now appended as catalog rows with their `input` metadata, so the existing Gateway image-capability resolver sees configured vision models even when provider discovery or Qwen Coding Plan suppression omits them.

## What This Changes

WebChat should preserve image attachments for an explicitly configured Qwen `qwen3.6-plus` model on `coding.dashscope.aliyuncs.com` even when the built-in Qwen catalog omits that model.

## Best Possible Solution

Keep the catalog-level implementation: explicit provider model definitions should be first-class Gateway catalog rows, which lets all runtime capability checks honor configured modalities without adding Qwen-specific model-name shims.

## Reproduction Assessment

Yes. The original path is concrete: configure a provider model that is absent from discovery but declares image input, then send a WebChat image attachment; current main no longer reproduces the drop because the configured model is now present in the runtime catalog and covered by focused tests.

## Solution Assessment

Yes. Solving this at the runtime model catalog boundary is the narrowest maintainable fix because the existing resolver, WebChat path, and provider-specific suppression rules can remain generic.
