## Summary

Close as implemented on current main. The Qwen Coding Plan catalog still avoids advertising `qwen3.6-plus` by default, but current main now merges explicit `models.providers.*.models` entries into the gateway model catalog and uses normalized image-capability lookup, so configured vision models are treated as image-capable by the WebChat attachment path.

## What This Changes

The issue asks WebChat/gateway image-capability detection to honor explicitly configured Qwen/custom model `input: ["text", "image"]` metadata when the built-in catalog omits the model for a Coding Plan DashScope base URL.

## Best Possible Solution

Keep the current generic catalog-based implementation: explicit provider model metadata should flow into the gateway catalog, Qwen Coding Plan catalog suppression should remain the default for unconfigured `qwen3.6-plus`, and the main-only fix should ship in the next release.

## Reproduction Assessment

Yes. The issue has a clear source-level reproduction path: configure a provider model with `input: ["text", "image"]`, send a WebChat image attachment, and verify the gateway catalog/resolver keeps the image path; current tests cover the analogous configured custom vision and Qwen suppression cases.

## Solution Assessment

Yes. The current-main fix is more maintainable than a Qwen-name heuristic because it preserves the existing image-capability resolver contract while making explicit configured model metadata available to that resolver.
