## Summary

Keep open: current main still does not expose MiniMax-M3 adaptive/xhigh/max through the provider thinking profile, and the MiniMax stream wrapper still defaults Anthropic-compatible MiniMax requests to disabled thinking unless an explicit payload already exists. The issue is valid as a provider capability gap, but the exact xhigh/max budget-token request needs maintainer/provider judgment because MiniMax’s current Anthropic-compatible schema documents disabled/adaptive, not enabled budget mode.

## What This Changes

The issue requests MiniMax-M3 thinking-profile support for adaptive, xhigh, and max levels so they appear in /think menus and map correctly to MiniMax API calls.

## Best Possible Solution

Add provider-owned MiniMax-M3 adaptive support only after proving the stream format is safe, and document or reject xhigh/max unless MiniMax publishes those request modes for the Anthropic-compatible API.

## Reproduction Assessment

Yes, from source inspection: MiniMax-M3 is reasoning-capable, but MiniMax providers do not declare a custom thinking profile, so OpenClaw falls back to off/minimal/low/medium/high. I did not run a live MiniMax request because no credentials were available in this read-only review.

## Solution Assessment

No as written: adaptive support appears upstream-backed, but xhigh/max mapped to budget tokens is not supported by the current MiniMax Anthropic-compatible schema I inspected. A safer fix is MiniMax-owned adaptive support with tests and live proof before expanding additional levels.
