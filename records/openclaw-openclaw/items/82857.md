## Summary

Keep open: the PR is aimed at the right source-proven bug, but the new CLI followup branch drops existing CLI-runner session-isolation and metadata contracts, and the external PR still only has mocked test proof.

## What This Changes

This PR adds a CLI-runtime dispatch branch to `createFollowupRunner` and a regression test that verifies CLI-backed queued followups call `runCliAgent` instead of `runEmbeddedPiAgent`.

## Best Possible Solution

Rework the followup CLI branch to share or faithfully mirror the primary CLI dispatch context, including room-event transient handling, CLI session metadata, and provenance/static-prompt forwarding, then add real CLI-backed queued followup proof.

## Reproduction Assessment

Yes. The linked bug is source-reproducible on current main because `createFollowupRunner` still calls the embedded runner directly while the primary path has a CLI dispatch branch; I did not run a live CLI/channel scenario in this read-only review.

## Solution Assessment

No, not as-is. Routing CLI-backed followups through `runCliAgent` is the right direction, but the branch should preserve the primary path's room-event isolation and CLI metadata contracts before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid reusing CLI sessions for room-event followups:** `src/auto-reply/reply/followup-runner.ts:413-415`
  - body: This always passes the active CLI session binding into queued CLI followups. The primary CLI path deliberately passes no binding for `room_event` turns and strips any returned binding so ambient/system events do not reuse or persist chat CLI sessions. A queued room-event followup on a CLI runtime would now attach to the existing session and can persist the returned binding via followup session accounting; guard `queued.currentInboundEventKind === "room_event"` the same way before passing `cliSessionId`/`cliSessionBinding`.
  - confidence: 0.9
- **[P2] Forward static prompt metadata to CLI followups:** `src/auto-reply/reply/followup-runner.ts:397-409`
  - body: The queued run already carries `inputProvenance` and `extraSystemPromptStatic`, and the primary CLI path forwards both. This new branch only forwards the dynamic `extraSystemPrompt`; `prepareCliRunContext` hashes that full prompt when the static value is absent, so per-message inbound metadata can invalidate CLI session reuse on followup turns and provenance annotations are lost. Pass `run.inputProvenance` and `run.extraSystemPromptStatic` alongside the prompt.
  - confidence: 0.84
