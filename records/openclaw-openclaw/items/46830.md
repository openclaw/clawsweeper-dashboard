## Label Justifications

- `P2`: The issue targets meaningful channel reliability and user trust, but it is a bounded product/API improvement rather than an emergency regression.
- `impact:message-loss`: The reported failure mode is generated replies or message-tool sends not reaching users after channel send errors.

## Summary

Keep open. Current main has durable delivery-status primitives, plugin observation hooks, and a Teams-only system-event path, but it still lacks the requested shared cross-channel agent-visible delivery-failure contract for ordinary final replies and message-tool sends.

## What This Changes

The issue asks for channel send failures to become visible to the agent, either through injected context or structured delivery status in message-tool responses.

## Best Possible Solution

Define an additive shared delivery-failure contract for final replies and message-tool sends, reuse the existing durable send outcome primitives, map channel-specific failures through SDK-owned seams, and add cross-channel regression coverage.

## Reproduction Assessment

Yes, source-level: current main still lacks a shared agent-visible delivery-failure signal across Feishu, Telegram, and generic message-tool/default final-reply sends. I did not run a live failed channel send.

## Solution Assessment

Unclear as a concrete implementation because maintainers still need to choose the shared contract shape. The maintainable direction is additive and shared, built on durable delivery outcomes rather than another channel-local patch.
