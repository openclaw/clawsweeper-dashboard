## Summary

Keep open. Current main has the system-event primitive and a Microsoft Teams-specific partial implementation, but the shared reply dispatcher/outbound delivery path plus Feishu, Telegram, and Discord normal reply paths still do not inject a model-visible delivery-failure event or return delivery status to the agent.

## What This Changes

Request model-visible notification or delivery-status feedback when channel message delivery fails, so the agent can retry, acknowledge the gap, or recover from an undelivered reply.

## Best Possible Solution

Implement a generic delivery-failure notification contract that reuses the existing system-event queue, carries compact channel/target/error/retry/message-preview context, and is wired through the normal shared reply/outbound path plus bundled Feishu, Telegram, Discord, and Teams reply delivery with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make reply delivery reject in the shared dispatcher or bundled channel send path: current main reaches `onError`, logging, fallback, or `message_sent` hooks, but Feishu, Telegram, Discord, and the shared outbound path do not enqueue a system event for the next agent turn.

## Solution Assessment

Yes, the system-event direction is the best available path because it already feeds the next prompt and matches the Teams precedent. Logging, observer hooks, or same-channel fallback alone are not sufficient because they do not reliably inform the agent that its previous reply was not delivered.
