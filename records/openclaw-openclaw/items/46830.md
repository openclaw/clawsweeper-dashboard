## Summary

Keep open. Current main and the latest release have partial delivery-status handling and a Teams-specific system-event implementation, but they still do not provide the requested shared, cross-channel agent-visible delivery-failure contract for Feishu, Telegram, and generic message-tool sends.

## What This Changes

The issue requests an agent-visible notification or tool response status when channel message delivery fails, so the agent can recover instead of assuming the user received the message.

## Best Possible Solution

Define an additive shared delivery-failure contract for channel replies and message-tool sends, then map channel-specific failures into it with compatibility tests for successful, partial, suppressed, and failed delivery shapes.

## Reproduction Assessment

Yes for the source-level feature gap: current main still has Feishu and generic message-tool paths without a shared agent-visible delivery-failure signal. I did not reproduce a live failed send, so full runtime breadth remains unverified.

## Solution Assessment

Unclear as a concrete implementation. A shared additive delivery-failure contract is preferable to one-off channel injections, but maintainers need to choose event shape, retry semantics, and compatibility behavior first.
