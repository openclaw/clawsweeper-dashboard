## Label Justifications

- `P2`: The issue describes a real reliability improvement with user-visible message loss, but it needs design work and has limited immediate blast radius compared with an active outage.
- `impact:message-loss`: The reported failure mode is that generated replies can fail to reach the channel while the agent proceeds as if delivery succeeded.
- `impact:session-state`: The requested fix would change what delivery-failure state is injected into or preserved for the agent session context.

## Summary

Keep open. Current main has partial delivery-status and channel-specific work, but it still does not provide a general cross-channel way for an agent to learn that a visible channel reply failed; the remaining behavior needs maintainer product/API direction rather than cleanup closure.

## What This Changes

Request a cross-channel delivery-failure notification path so agents can detect failed sends and recover or report them.

## Best Possible Solution

Define an additive shared delivery-failure contract that channel reply paths and message-tool sends can report consistently, then preserve channel-specific successes and add focused regressions for failed, partial, suppressed, and successful delivery shapes.

## Reproduction Assessment

Yes for the source-level feature gap: Feishu and generic message-tool paths still lack a shared agent-visible delivery-failure signal. I did not reproduce a live failed channel send, so runtime breadth remains unproven.

## Solution Assessment

Unclear as a concrete implementation. A shared additive contract is preferable to one-off channel injections, but maintainers need to choose event shape, retry semantics, and compatibility behavior before a safe implementation PR.
