## Label Justifications

- `P2`: The issue describes normal-priority reliability work for lost channel replies and agent state drift with limited but real user impact.
- `impact:message-loss`: The reported failure mode is a generated reply that the user never receives after a channel API failure.

## Summary

Keep open. Current main has delivery outcomes and warning-style handling for narrower paths, but it still does not provide the shared agent-visible cross-channel delivery-failure notification or tool-status contract requested here; the remaining work is a product/API design decision rather than a conservative cleanup close.

## What This Changes

Request a shared mechanism that tells the agent when channel message delivery fails so it can retry, acknowledge the failure, or preserve the failure in context.

## Best Possible Solution

Define a maintainer-approved shared delivery outcome contract at the agent/channel boundary, then implement channel adapters so failed final replies become agent-visible without duplicating channel-specific policy.

## Reproduction Assessment

Yes for the source-level failure class: current code has narrower delivery status paths, but no generic mechanism that injects failed channel delivery into agent context. I did not run a live Feishu, Telegram, Discord, or Teams failure scenario for this review.

## Solution Assessment

Unclear as proposed: system-message injection is plausible, but the best fix needs maintainer agreement on the shared channel/agent API contract and how retries or memory updates should behave across transports.
