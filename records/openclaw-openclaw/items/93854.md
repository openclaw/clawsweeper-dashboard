## Summary

Keep open: current main and v2026.6.9 still build duplicate-prone Docker mount arguments for this case, and two open closing PRs already cover the fix path.

## What This Changes

The issue reports Docker sandbox startup failure when custom Docker binds target the same container paths as OpenClaw's read-only workspace skill mounts.

## Best Possible Solution

Land one canonical sandbox mount fix that prevents duplicate Docker targets across shell and browser Docker paths while keeping protected skill roots read-only and authoritative.

## Reproduction Assessment

Yes. Current main source builds custom `-v` args before protected read-only skill `-v` args in shell and browser sandbox creation, and existing tests assert the duplicate-prone ordering; I did not run a live Docker daemon in this read-only review.

## Solution Assessment

No. Current main is not the best solution because it can still emit duplicate Docker destinations; the best fix is one linked PR or replacement that removes the duplicate target while preserving the read-only skill boundary.
