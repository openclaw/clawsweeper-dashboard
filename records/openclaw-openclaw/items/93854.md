## Summary

Keep open: current main and the latest release still build duplicate Docker mount targets for this conflict, and several open PRs explicitly target this issue. The right cleanup is to keep this issue as the canonical bug until one safe fix lands.

## What This Changes

The issue asks OpenClaw to avoid Docker sandbox startup failures when a custom bind targets the same container path as an internal read-only skill mount.

## Best Possible Solution

Land one canonical fix that prevents duplicate Docker mount targets while preserving protected read-only skill mounts across shell, browser, and filesystem-bridge surfaces.

## Reproduction Assessment

Yes from source, though not from a live Docker daemon run in this read-only review. Current main appends a user bind and then a protected read-only skill mount for the same target, and Docker upstream rejects duplicate bind destinations.

## Solution Assessment

Yes, the best direction is to dedupe before docker create, but the fix should keep protected skill mounts authoritative. Skipping or rejecting the colliding custom bind is safer than letting it shadow read-only skill sources.
