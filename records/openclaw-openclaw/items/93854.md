## Label Justifications

- `P2`: The report is a concrete sandbox startup bug with limited blast radius around custom Docker binds and reserved skill paths.
- `impact:security`: The conflict involves protected read-only skill mount destinations, so the repair must preserve the sandbox skill write boundary.

## Summary

This issue should stay open: current main still appends custom Docker binds and protected read-only skill mounts as separate `-v` entries, and the linked fixes are open candidate PRs rather than merged implementation.

## What This Changes

The issue reports Docker sandbox startup failing with `Duplicate mount point` when a user bind targets the same container path as internal read-only skill mounts.

## Best Possible Solution

Review and land one narrow candidate PR that makes duplicate container mount destinations impossible while documenting the protected-skill precedence policy across Docker, browser, config hash, and filesystem path resolution.

## Reproduction Assessment

Yes for source reproduction: when reserved-target validation is explicitly bypassed, current main can still construct both a custom bind and a protected skill bind for `/workspace/skills` or `/workspace/.agents/skills`. I did not run a live Docker daemon repro because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the security policy for collisions. The safest maintainable direction is to reject or filter collisions before `docker create` while keeping protected skill mounts authoritative unless maintainers intentionally choose otherwise.
