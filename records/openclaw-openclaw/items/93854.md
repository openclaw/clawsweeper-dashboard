- `impact:security`: The affected behavior is a sandbox mount boundary that protects workspace skill instructions from writable custom binds.
- `impact:crash-loop`: When reached, the duplicate mount target prevents sandbox container creation and blocks agent runs that need that sandbox.

## Summary

Keep open. Current main and v2026.6.9 still append custom binds before protected read-only skill mounts, so the duplicate Docker target remains possible when reserved-target validation is explicitly bypassed; several open PRs reference this issue as candidate fixes, but none has merged.

## What This Changes

The issue requests fixing Docker sandbox startup when a user-configured bind targets the same container path as OpenClaw's internal read-only workspace skill mounts.

## Best Possible Solution

Land one canonical fix that rejects or filters colliding custom binds while preserving protected read-only skill mounts across shell, browser, config hashing, and filesystem bridge modeling, then close this issue after redacted live Docker proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes when reserved-target validation is explicitly bypassed: current main can append two `-v` entries for the same skill container path. Live Docker reproduction was not run, and the issue body does not mention the required dangerous override.

## Solution Assessment

Is this the best way to solve the issue? No for the suggested fix as written: skipping default protected mounts would weaken the existing read-only skill boundary. The safer solution is to normalize targets and reject or skip the colliding custom bind while keeping protected skill mounts authoritative.
