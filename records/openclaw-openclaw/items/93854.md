## Repair Work Prompt

Fix the Docker sandbox duplicate mount target bug reported in https://github.com/openclaw/openclaw/issues/93854. Current main should already reject ordinary binds under `/workspace` through reserved-target validation; preserve that fail-closed behavior. For the explicit `dangerouslyAllowReservedContainerTargets` path, current Docker and browser create args can append a user bind and then a protected read-only skill mount for the same destination (`/workspace/skills`, `/workspace/.agents/skills`, or `/workspace/.openclaw/sandbox-skills/skills`), which Docker can reject as a duplicate mount point. Keep the fix in the shared sandbox mount validation/create path, preserve protected skill security semantics, cover both `src/agents/sandbox/docker.ts` and `src/agents/sandbox/browser.ts`, and keep `buildSandboxFsMounts` consistent. Do not add a new config option. Add focused regression coverage for colliding user bind destinations and assert the user-facing error or safe mount outcome happens before Docker daemon duplicate-mount failure.
