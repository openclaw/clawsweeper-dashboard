## Summary

Keep open: current main and the latest release still append configured Docker binds before read-only workspace skill overlays without de-duplicating identical container targets, and this issue already has open fixing PR candidates that need sandbox/security review.

## What This Changes

The issue reports Docker sandbox startup failure when a custom bind in `openclaw.json` targets the same container path as OpenClaw's internal read-only workspace skill mounts.

## Best Possible Solution

Land or close a single canonical sandbox fix that makes duplicate Docker mount targets impossible while preserving the protected read-only skill boundary across shell, browser, config-hash, and filesystem-bridge paths.

## Reproduction Assessment

Yes for source reproduction: current main and `v2026.6.9` still build Docker args that can include a custom bind and a read-only skill overlay for the same container target. I did not run a live Docker daemon in this read-only review, but the emitted argument path is clear from source and tests.

## Solution Assessment

No merged solution exists yet. The issue's suggested fix is only safe if it de-duplicates without weakening protected read-only skill mounts; current tests show that preserving protected overlays is an existing sandbox security invariant.
