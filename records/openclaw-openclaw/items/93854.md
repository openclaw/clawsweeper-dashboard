## Summary

Keep this issue open: current main still builds Docker create args with a custom bind followed by an internal read-only skill mount for the same container path, and there are several open candidate PRs linked with closing syntax that should be reviewed instead of closing the issue early.

## What This Changes

The issue reports Docker sandbox startup failure when custom Docker binds target the same container paths as internal read-only skill mounts.

## Best Possible Solution

Land one narrow linked PR that deduplicates shell, browser, and fs mount handling while preserving protected read-only skill mounts and adding regression coverage for the conflicting-bind case.

## Reproduction Assessment

Yes for the narrower current-main path: with reserved-target validation explicitly bypassed, source and existing tests show two Docker `-v` args can target `/workspace/skills`. The issue's exact snippet is incomplete because default validation blocks reserved `/workspace` targets before Docker.

## Solution Assessment

No, the suggested fix of skipping internal mounts is not the best fix by itself because current tests and docs treat skill sources as protected read-only overlays. The better repair is to prevent duplicate Docker destinations while preserving the protected mount invariant, likely by rejecting or skipping colliding custom binds consistently across shell, browser, and fs mapping.
