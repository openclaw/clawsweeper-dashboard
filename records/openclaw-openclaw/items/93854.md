## Label Justifications

- `P2`: The bug blocks sandbox startup for a specific custom-bind configuration but has a narrow, localized agents/sandbox surface.
- `impact:security`: The affected paths are protected skill mounts and reserved sandbox targets, so the repair must preserve the read-only security boundary.
- `impact:crash-loop`: The reported behavior prevents Docker sandbox containers from starting when the duplicate mount target reaches the Docker daemon.

## Summary

This issue is still necessary: current main still constructs duplicate Docker bind targets for the reported custom-bind/protected-skill-mount case, and two open closing PRs already own candidate fixes rather than a merged implementation solving it.

## What This Changes

The issue asks OpenClaw to avoid Docker sandbox startup failures when a custom `sandbox.docker.binds` entry targets the same container path as OpenClaw's internal read-only skill mounts.

## Best Possible Solution

Review and land one narrow fix that prevents duplicate Docker targets by rejecting or skipping the colliding user bind while keeping protected skill mounts read-only, then close this issue from the merged PR.

## Reproduction Assessment

Yes. The source path is high-confidence: current main appends a user bind for `/workspace/skills` and then appends a protected read-only mount for the same target, and tests assert both entries are present; only live Docker daemon proof is missing.

## Solution Assessment

Yes, the best fix shape is narrow: preserve the protected read-only skill mount and filter or reject colliding user binds before Docker receives duplicate targets. A user-bind-precedence fix would need extra security review because it can make skill paths writable when dangerous reserved-target flags are enabled.
