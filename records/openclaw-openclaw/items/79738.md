## Summary

Keep open: the related in-process restart snapshot bug has been fixed on current main, but that does not prove the reported `u4s-openclaw` wrapper/plugin-enable startup sequence no longer rewrites access-control config. Current source still shows active config writers and no specific guard against losing owner/elevated/WhatsApp allowlist settings.

## What This Changes

The issue asks OpenClaw to stop wrapper/startup restarts from rewriting `openclaw.json` and discarding WhatsApp allowlists, owner allowlists, elevated webchat restrictions, and disabled-channel settings.

## Best Possible Solution

Trace the wrapper restart with per-write config diffs on current main, then harden the config write or startup boundary so explicit access-control settings cannot be overwritten by stale or derived baselines.

## Reproduction Assessment

No, not for current main. The reporter supplied detailed audit evidence and source inspection confirms active writer and guard-gap paths, but the exact `u4s-openclaw` wrapper/plugin-enable sequence has not been reproduced after the related restart fix.

## Solution Assessment

Unclear. The merged consume-once startup snapshot fix is the right narrow fix for the related SIGUSR1 bug, but this issue still needs proof of which current-main writer supplies or persists the minimal baseline before a maintainable repair can be chosen.
