## Summary

Keep open: the report describes a plausible config-corruption bug with security impact, but current OpenClaw source does not show `plugins enable` or gateway startup rewriting the reported channel/security fields, and the `u4s-openclaw` wrapper itself is not in this checkout.

## What This Changes

The issue reports that restarting through `u4s-openclaw` rewrites `openclaw.json`, dropping WhatsApp allowlists, command owners, elevated webchat restrictions, and disabled channel choices.

## Best Possible Solution

Identify the first process that supplies the reduced baseline, then either fix the wrapper to preserve user-authored config or add a focused OpenClaw regression/guard for the confirmed stale-write path.

## Reproduction Assessment

No: the issue has detailed deployment logs, but the checked-out repo does not include `u4s-openclaw`, so there is no high-confidence current-main reproduction path for the actual restart loop. Source inspection confirms the affected config surfaces and that current `plugins enable`/gateway startup code does not obviously perform the reported destructive rewrite.

## Solution Assessment

Unclear: investigating the wrapper/startup owner is the right next step, but the best code fix depends on which process first supplies the reduced config. If the wrapper is external, the safest solution is to fix that wrapper rather than adding broad core clobber policy.
