## Summary

Keep open: current main contains the related consume-once startup snapshot fix, but that only proves the narrower in-process restart bug from https://github.com/openclaw/openclaw/issues/79947 is fixed. The reported `u4s-openclaw` startup sequence still needs tracing because `plugins enable` remains a config writer, WhatsApp channel config remains a no-op reload path, and no current-main proof shows the wrapper/plugin-enable baseline rewrite is gone.

## What This Changes

Prevent the `u4s-openclaw` restart/startup sequence from rewriting explicit WhatsApp allowlists, owner/elevated allowlists, and channel enable settings in `openclaw.json`.

## Best Possible Solution

Trace the wrapper restart with per-write config diffs on current main, then harden the config write or startup boundary so explicit access-control and channel-enable settings cannot be overwritten by stale or derived baselines.

## Reproduction Assessment

No, not for current main. The reporter supplied detailed audit evidence and source inspection confirms adjacent write/reload paths, but the exact `u4s-openclaw` wrapper/plugin-enable sequence has not been reproduced after the merged startup snapshot fix.

## Solution Assessment

No complete current solution is proven. The merged startup snapshot fix is the right narrow fix for https://github.com/openclaw/openclaw/issues/79947, but this issue still needs proof that wrapper startup writes no longer replay a baseline over explicit access-control config.
