## Summary

Keep open: the related in-process restart snapshot fix is on current main, but it does not prove the reported `u4s-openclaw` startup `plugins enable` sequence and broader access-control baseline rewrite are fixed.

## What This Changes

The issue reports that restarting under `u4s-openclaw` rewrites `openclaw.json`, dropping WhatsApp allowlists, owner allowlists, elevated webchat restrictions, and disabled channel settings.

## Best Possible Solution

Trace the wrapper restart with per-write config diffs on current main, then harden the startup/write boundary so explicit access-control settings cannot be overwritten by stale or derived baselines.

## Reproduction Assessment

No, not yet for current main. The reporter supplied detailed audit evidence and a related source-proven SIGUSR1 stale-snapshot bug was fixed, but the exact `u4s-openclaw` wrapper plus `plugins enable` sequence has not been reproduced after that fix.

## Solution Assessment

No complete solution is proven. The merged one-shot startup snapshot fix is the right narrow fix for the related SIGUSR1 bug, but this issue still needs proof that wrapper startup writes no longer replay a baseline over explicit access-control config.
