## Summary

Keep open: current main does not show a shipped fix, and source inspection supports a plausible OpenClaw-owned config write/reload failure even though the external `u4s-openclaw` wrapper is not in this repository.

## What This Changes

The issue reports that restart under `u4s-openclaw` rewrites `openclaw.json` to a minimal baseline, dropping WhatsApp allowlists, owner allowlists, elevated webchat restrictions, and channel enablement choices.

## Best Possible Solution

Trace the restart sequence with per-write config diffs, then harden the OpenClaw config-write/reload path so stale startup baselines cannot overwrite explicit access-control settings or be promoted as last-known-good.

## Reproduction Assessment

No, not at high confidence: the issue includes detailed audit evidence and current source supports the failure mode, but I did not run the `u4s-openclaw` container restart path or isolate the exact writer on current main.

## Solution Assessment

No current fix is present. The narrowest maintainable fix is likely at the config-write/reload boundary, with targeted instrumentation or rejection for stale baseline writes, rather than a WhatsApp-only allowlist workaround.
