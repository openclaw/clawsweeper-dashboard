## Summary

Keep open: current main includes the related consume-once startup snapshot fix, but the reported `u4s-openclaw` startup sequence still has not been proven fixed, and the broader access-control baseline rewrite remains credible and security-sensitive.

## What This Changes

The issue reports that the `u4s-openclaw` startup wrapper runs plugin-enable and gateway startup commands that rewrite `openclaw.json`, dropping WhatsApp allowlists, owner allowlists, elevated webchat restrictions, and disabled-channel choices after restart.

## Best Possible Solution

Trace the `u4s-openclaw` restart with per-write config diffs on current main, then harden the startup/config-write boundary so explicit access-control and channel enablement settings cannot be overwritten by stale or derived baselines.

## Reproduction Assessment

No, not yet for current main. The reporter supplied detailed audit evidence and current source supports adjacent writer/reload paths, but the exact wrapper restart sequence after the merged SIGUSR1 fix has not been reproduced or reduced to a single current-main writer.

## Solution Assessment

No complete solution is proven. The merged startup snapshot fix is the right narrow fix for the related in-process restart bug, but this issue still needs proof that wrapper startup writes and plugin-enable reconciliation no longer replay a minimal baseline over user config.
