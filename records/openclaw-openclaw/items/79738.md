## Summary

Keep open: current main has the merged fix for the related stale in-process restart snapshot, but the reported u4s wrapper/plugin-enable startup sequence is not proven fixed and the wrapper itself is not in this repository. The remaining report is a credible access-control config overwrite that needs maintainer tracing, especially while the adjacent WhatsApp hot-activation issue remains open.

## What This Changes

The issue reports that restarting OpenClaw under the u4s wrapper rewrites openclaw.json and drops WhatsApp allowlists, owner allowlists, elevated webchat restrictions, and channel enabled settings.

## Best Possible Solution

Trace the wrapper, `plugins enable`, and `gateway run` write sequence with per-write config diffs on current main, then harden the OpenClaw writer/reload boundary so startup writes cannot drop explicit access-control keys; coordinate any WhatsApp unloaded-plugin behavior with https://github.com/openclaw/openclaw/issues/78404.

## Reproduction Assessment

No. The reporter provided detailed v2026.5.7 container audit evidence, but I did not reproduce the u4s wrapper sequence on current main after the merged in-process restart fix, and the exact wrapper is outside this source tree.

## Solution Assessment

Unclear. The merged in-process restart fix is the right solution for https://github.com/openclaw/openclaw/issues/79947, but this issue still needs tracing of the u4s/plugin-enable startup sequence before a narrow OpenClaw fix can be selected.
