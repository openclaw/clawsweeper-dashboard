## Summary

Keep open: current main has the merged consume-once startup snapshot fix for https://github.com/openclaw/openclaw/issues/79947, but that does not prove this wrapper-startup `plugins enable` sequence and broader access-control baseline rewrite are fixed. The remaining report still needs maintainer tracing against the `u4s-openclaw` restart path and the adjacent WhatsApp reload gap in https://github.com/openclaw/openclaw/issues/78404.

## What This Changes

The issue reports restart-time config rewriting under the `u4s-openclaw` wrapper that drops WhatsApp allowlists, owner allowlists, elevated webchat restrictions, and channel enabled settings.

## Best Possible Solution

Trace the wrapper restart with per-write config diffs on current main, then harden the config write or startup boundary so explicit access-control settings cannot be overwritten by stale or derived baselines.

## Reproduction Assessment

No, not yet for current main. The reporter supplied detailed audit evidence, and a related source-reproducible SIGUSR1 stale-snapshot bug was fixed, but the exact `u4s-openclaw` wrapper/plugin-enable sequence has not been reproduced after that fix.

## Solution Assessment

No proven complete solution is present. The merged consume-once startup snapshot fix is the right narrow fix for the related in-process restart bug, but this issue still needs proof that the wrapper startup writes no longer replay a baseline over user config.
