The PR changes `initFastReplySessionState` to clear thinking, verbose, reasoning, TTS, model, provider, and auth-profile overrides on `/reset`, with two fast-path tests for reset and non-reset turns.

## Best Possible Solution

Align the fast-test bootstrap with production reset semantics: preserve user-owned behavior/model/auth choices, clear only auto-owned routing state, and treat any reset-all change as an explicit maintainer product decision.

## Reproduction Assessment

Yes for the review finding from source inspection: the PR clears every reset override in `initFastReplySessionState`, while current main’s production initializer and tests preserve user-owned overrides and clear only auto-sourced selections. I did not run the PR branch in this read-only review.

## Solution Assessment

No. Clearing every override in the fast bootstrap is not the narrowest maintainable fix against current main; the safer path is to mirror `initSessionState` and `resolveResetPreservedSelection`, or first get maintainer approval to change reset semantics globally.
