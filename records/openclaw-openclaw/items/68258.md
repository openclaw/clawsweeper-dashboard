- `P1`: A Gateway self-restart during OAuth refresh can drop in-flight WebSocket or agent work for real users.
- `impact:crash-loop`: The issue is about config reload scheduling a Gateway SIGTERM/restart from a benign auth-profile write.
- `impact:auth-provider`: The trigger is persisted OAuth auth-profile refresh state under `auth.profiles.*`.

## Summary

Keep open. Current main still source-reproduces the reload-planner path where `auth.profiles.*` has no matching rule and therefore requests a Gateway restart, and the paired fix PR at https://github.com/openclaw/openclaw/pull/68257 remains open rather than merged.

## What This Changes

Requests preventing OAuth auth-profile refresh writes under `auth.profiles.*` from causing Gateway process restarts during config reload.

## Best Possible Solution

Classify persisted auth-profile refresh writes as no-op reload paths with regression coverage, while preserving `gateway.auth.*` restart behavior and the conservative unknown-path restart default.

## Reproduction Assessment

Yes. Source inspection on current main shows `auth.profiles.openai-codex:user@example.com` has no matching reload rule and takes the unmatched-path restart branch; the issue discussion adds production-log confirmation.

## Solution Assessment

Yes, with a scope caveat. The narrow maintainable fix is a reload-rule classification plus focused tests, while maintainers can choose whether the no-op prefix should be `auth.profiles` or all `auth.*`.
