## Summary

Keep open. Current main still leaves auth.profiles.* unmatched in the gateway reload planner, so the reported path still takes the default gateway-restart branch; the issue is also paired with open PR #68257, which is the right maintainer review path before closing.

## What This Changes

The issue asks for OAuth auth profile config updates under auth.profiles.* to be treated as dynamic/no-op reloads instead of unmatched changes that restart the gateway.

## Best Possible Solution

Keep this issue open until PR #68257 or an equivalent patch lands with an explicit dynamic/no-op classification for auth profile metadata updates and regression tests proving OAuth profile refreshes do not restart the gateway while gateway.auth.token and gateway.auth.mode still do.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to pass auth.profiles.openai-codex:user@example.com into buildGatewayReloadPlan on current main; because no auth rule matches, the no-rule branch marks restartGateway and records that path as a restart reason.

## Solution Assessment

Yes, if the fix is scoped to auth profile metadata rather than changing the unknown-path default. The paired PR direction is the narrow maintainable path, with the remaining judgment being whether the rule should cover all auth.* or only auth.profiles/auth.order.
