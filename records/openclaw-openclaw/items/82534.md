Overall confidence: 0.9

Full review comments:

- **[P1] Recurse compile scans for nested page-group files:** `extensions/memory-wiki/src/compile.ts:372`
  - body: The PR advertises recursive custom vault support, but compile still feeds each configured dir into `collectMarkdownFiles`, whose helper only returns immediate child `.md` files. A page like `资料/技术/AI/foo.md` will be absent from `pages`, so root indexes, directory indexes, dashboards, and digests omit it even though the proof claims nested pages are indexed.
  - confidence: 0.92
- **[P1] Recurse wiki search scans for configured dirs:** `extensions/memory-wiki/src/query.ts:249-256`
  - body: `listWikiMarkdownFiles` now iterates configured page-group dirs, but each dir is still scanned with a single `fs.readdir` and `entry.isFile()` filter. Nested custom pages therefore never enter `wiki_search` fallback scans, which breaks the central recursive-scanning behavior described in the PR.
  - confidence: 0.91
- **[P1] Pass pageGroups through wiki_get lookups:** `extensions/memory-wiki/src/query.ts:1563-1568`
  - body: `getMemoryWikiPage` still reads digest and fallback wiki pages without `effectiveConfig.pageGroups`. For custom dirs such as `资料/foo.md`, `toWikiPageSummary` falls back to English hard-coded prefixes and returns null, so direct `wiki_get` lookups can fail even after search/status know about the custom dir.
  - confidence: 0.89
- **[P2] Mirror pageGroups dir validation in the manifest:** `extensions/memory-wiki/openclaw.plugin.json:215-218`
  - body: Runtime config rejects absolute paths and `..` traversal for pageGroup dirs, but the plugin manifest only says `dir` is a non-empty string. That lets setup/discovery metadata accept config that the runtime later rejects or silently drops, so the new public config contract is inconsistent.
  - confidence: 0.84
