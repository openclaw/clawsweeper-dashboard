## Summary

Keep open. The feature is relevant to the open recursive memory-wiki request, but this PR is not mergeable: the branch has a TypeScript syntax break, removes session-visibility protections from wiki search/get, accepts unsafe page-group paths, and has no after-fix real vault proof.

## What This Changes

The PR adds configurable memory-wiki page groups, a standalone vault `.wiki-page-groups.json`, recursive compile scanning, dynamic directory lookup, docs, and related tests.

## Best Possible Solution

Rework the feature on top of the existing memory-wiki implementation with vault-relative path validation, preserved session visibility, one shared recursive collector for compile/query/status, focused tests, and real vault-level proof.

## Reproduction Assessment

Yes. Source inspection of the PR branch reproduces the patch regressions: `vault.ts` is invalid TypeScript, wiki search/get no longer carry session-visibility filtering, and configured dirs are accepted without path-safety checks.

## Solution Assessment

No. Configurable groups and recursive traversal may be the right product direction, but this implementation is not the narrowest maintainable solution because it breaks existing safety and query behavior while leaving the config precedence unclear.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep session visibility filtering in wiki search/get:** `extensions/memory-wiki/src/query.ts:1269-1272`
  - body: This rewrites wiki search/get so shared-memory results are mapped directly and the `sandboxed` caller context plus session visibility guards are gone. That reopens session-sourced memory hits and direct reads that current main filters before returning them to an agent.
  - confidence: 0.93
- **[P1] Constrain page-group dirs to vault-relative paths:** `extensions/memory-wiki/src/config.ts:132`
  - body: `dir` only requires a non-empty string, but the new directory values are later joined into vault read/write paths. A `.wiki-page-groups.json` or config entry like `../outside` or an absolute path can make initialization, ingest, or apply write outside the vault, especially after this PR replaces `fsRoot` writes with direct `fs` calls.
  - confidence: 0.91
- **[P1] Remove the leftover array terminator:** `extensions/memory-wiki/src/vault.ts:10`
  - body: The branch leaves a standalone `] as const;` after the imports in `vault.ts`. That is not valid TypeScript, so the memory-wiki package cannot compile or load with this patch.
  - confidence: 0.99
- **[P2] Keep existing page readers working without config:** `extensions/memory-wiki/src/query.ts:248-250`
  - body: Existing callers such as `apply.ts`, `memory-palace.ts`, and `import-insights.ts` still call `readQueryableWikiPages(rootDir)` with no config. With the new `?? []` fallback, those callers scan zero directories and can no longer resolve existing wiki pages.
  - confidence: 0.88
- **[P2] Honor configured page-group precedence:** `extensions/memory-wiki/src/config.ts:312`
  - body: The resolver appends defaults before user and vault groups, while `findDirForKind` returns the first matching kind. A user who configures a custom `source` or `synthesis` dir still gets the built-in directory for new writes, contradicting the documented priority and the PR’s main feature.
  - confidence: 0.86
- **[P2] Use recursive discovery for wiki query/get paths:** `extensions/memory-wiki/src/query.ts:232-237`
  - body: The compile path now walks nested directories, but `listWikiMarkdownFiles` for query/get still reads only direct children of each configured directory. Nested pages can be compiled but remain invisible to search/get fallback when no digest candidate points at them.
  - confidence: 0.82
