## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P1] Recurse page-group scans before accepting nested vaults:** `extensions/memory-wiki/src/query.ts:249-256`
  - body: `listWikiMarkdownFiles` still reads only the immediate entries of each configured page-group directory and filters out subdirectories, so `wiki_search`, `wiki_get`, and callers using `readQueryableWikiPages` will miss the nested pages shown in the PR proof. The compile path has the same flat `collectMarkdownFiles` shape, so nested paths such as `资料/技术/AI/foo.md` can be counted by status but stay out of the index/digest/query surfaces.
  - confidence: 0.93
- **[P2] Filter index sections by group directory:** `extensions/memory-wiki/src/compile.ts:998-1004`
  - body: The root index now iterates every page group, but each section includes `pages.filter((page) => page.kind === group.kind)`. With the documented multi-dir setup where `synthesis` maps to `综合`, `视图`, and `模板`, every synthesis page appears under every one of those headings instead of only the matching directory, so the compiled index is duplicated and incorrectly grouped.
  - confidence: 0.9
- **[P2] Mirror pageGroups path constraints in the manifest:** `extensions/memory-wiki/openclaw.plugin.json:215-218`
  - body: The runtime Zod schema rejects absolute and `..` traversal dirs, but the plugin manifest exposes `pageGroups[].dir` as any non-empty string. That lets setup/config metadata accept values the runtime later strips or ignores, which is especially risky for a new filesystem-directory config surface; add equivalent manifest constraints or route validation through the same contract.
  - confidence: 0.78
