## Summary

Keep this PR open: page-group configurability is a useful feature direction, but the latest head still breaks memory-wiki search/apply behavior, drops sandboxed session visibility propagation, and introduces an unsafe path/config contract without real behavior proof.

## What This Changes

The PR adds configurable memory-wiki page groups via manifest/vault config, recursive compile scanning, dynamic directory resolution across vault paths, docs, and tests.

## Best Possible Solution

Ship a validated relative-dir pageGroups contract that preserves built-in defaults and reports, passes resolved groups through every read/write/status/apply path, keeps sandbox and package-boundary guards intact, and proves a real vault with nested/custom groups.

## Reproduction Assessment

Yes for the PR regressions by source inspection: latest head routes search/apply paths without resolved page groups, drops sandboxed propagation, and accepts unvalidated dirs used in direct filesystem paths. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the narrowest maintainable solution until it preserves current defaults, session visibility, path validation, recursive read/status parity, and package-boundary checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Restore sandboxed propagation in wiki tools:** `extensions/memory-wiki/src/tool.ts:143`
  - body: Current main passes `memoryContext.sandboxed` into both wiki search/get calls, and the downstream visibility guard only enforces sandboxed session filtering when that flag or a session key is present. Dropping it lets sandboxed calls without a session key skip the current session-memory filter.
  - confidence: 0.96
- **[P1] Pass resolved page groups into wiki search:** `extensions/memory-wiki/src/query.ts:1329-1342`
  - body: `searchWikiCorpus` still has no pageGroups/config parameter, so it calls `readQueryableWikiPages` without config and `listWikiMarkdownFiles` with `[]`. `wiki_search` can therefore scan no ordinary pages unless a digest candidate happens to match, and it cannot see custom dirs.
  - confidence: 0.94
- **[P1] Reject page group dirs that escape the vault:** `extensions/memory-wiki/src/config.ts:131-134`
  - body: `dir` accepts any non-empty string, but those values feed direct `path.join`, `fs.mkdir`, and `fs.writeFile` paths. Absolute paths and `../outside` need to be rejected or rooted through the fs-safe helper before any read/write path is built.
  - confidence: 0.95
- **[P2] Pass page groups when resolving apply targets:** `extensions/memory-wiki/src/apply.ts:177-178`
  - body: `resolveWritablePage` calls `readQueryableWikiPages` without config, but the reader now defaults to an empty group list. Existing `wiki_apply` metadata updates by id/path will fail to find current wiki pages.
  - confidence: 0.9
- **[P2] Scan query and status page groups recursively:** `extensions/memory-wiki/src/query.ts:238-245`
  - body: Compile now walks nested directories, but query and status still use one-level `readdir` loops. A page under a configured nested subdir can be compiled into indexes while `wiki_search`, lookup, and status counts miss it.
  - confidence: 0.88
- **[P2] Keep reports in the default page groups:** `extensions/memory-wiki/src/config.ts:118-123`
  - body: Current main indexes and counts `reports/`, and this branch still writes dashboard reports there when no report group is configured. Because defaults omit `report`, generated reports disappear from summaries, counts, and group indexes despite the upgrade doc claiming backward-compatible reports.
  - confidence: 0.88
- **[P2] Let configured groups override built-in defaults:** `extensions/memory-wiki/src/config.ts:315`
  - body: The docs say user/vault groups have priority, but the resolved config appends them after `DEFAULT_PAGE_GROUPS` and `findDirForKind` returns the first kind match. A configured `source` or `synthesis` dir cannot become the default write target.
  - confidence: 0.86
- **[P2] Keep memory-wiki opted into package-boundary checks:** `extensions/memory-wiki/tsconfig.json:2-9`
  - body: The package-boundary collector only treats an extension as opted in when `tsconfig.extends` is `../tsconfig.package-boundary.base.json`, and the contract expects every extension tsconfig to opt in. Replacing that extends drops memory-wiki from the public-SDK boundary canary.
  - confidence: 0.9
