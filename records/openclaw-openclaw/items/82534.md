## Summary

Keep open, but the patch is incorrect. It targets a real memory-wiki recursive-scanning/configurability request, but the current branch removes session visibility guards, accepts unsafe page-group paths, breaks existing default page lookup callers, and lacks real behavior proof.

## What This Changes

The PR adds memory-wiki `pageGroups` configuration, vault-local `.wiki-page-groups.json` loading, recursive scanning, dynamic output directories, docs/tests, and changes memory query/vault file I/O paths.

## Best Possible Solution

Rework this as a smaller memory-wiki change that preserves session access control, validates page-group dirs as safe vault-relative paths, shares one deterministic recursive collector across compile/query/status/lint, updates docs/tests, and includes real vault proof.

## Reproduction Assessment

Yes for the PR regressions by source inspection: unchanged current-main tests require session filtering, while the PR removes those filters, and existing callers still omit config while the PR defaults page discovery to no directories. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. Recursive collection is directionally useful, but this branch is not the narrowest maintainable solution until it preserves access controls, validates path inputs, keeps existing callers working, and settles the config contract.
