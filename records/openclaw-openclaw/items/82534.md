## Summary

Keep open: the latest head is still not mergeable because it breaks wiki search/read defaults, bypasses fs-safe vault writes, and leaves the new page-group contract inconsistent with the documented behavior.

## What This Changes

The PR adds configurable memory-wiki page groups, vault-local `.wiki-page-groups.json` loading, recursive compile scanning, dynamic directory resolution, docs, and tests.

## Best Possible Solution

Land a smaller owner-reviewed implementation that resolves page groups once, preserves fs-safe and package-boundary contracts, and threads the resolved groups consistently through compile, query, status, apply, and docs.

## Reproduction Assessment

Yes for the PR regressions by source inspection: latest head still passes empty page groups in wiki search/read paths and replaces rooted fs-safe writes with direct absolute fs writes. I did not run tests in this read-only review.

## Solution Assessment

No. The requested capability is useful, but this branch is not the best mergeable shape until it preserves default readers, recursive query parity, fs-safe writes, schema validation, package-boundary checks, and clear precedence semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Pass configured groups through wiki search:** `extensions/memory-wiki/src/query.ts:1390-1403`
  - body: `searchWikiCorpus` receives `pageGroups`, but candidate reads, fallback reads, and remaining path listing still pass `[]` or omit config. With the new reader defaulting to an empty group list, `wiki_search` can see an empty wiki corpus even when the resolved config has groups.
  - confidence: 0.97
- **[P1] Keep reader defaults backward compatible:** `extensions/memory-wiki/src/query.ts:252-258`
  - body: `readQueryableWikiPages(rootDir)` now maps missing config to `[]`. Existing callers such as apply metadata, import insights, and memory palace still call it without config, so they lose all wiki pages instead of using the built-in directories.
  - confidence: 0.96
- **[P1] Keep vault writes behind fs-safe:** `extensions/memory-wiki/src/apply.ts:168-169`
  - body: The branch writes pages through absolute `fs.mkdir`/`fs.writeFile` instead of the rooted fs-safe API used on main. Dir validation does not preserve the symlink/root safety checks expected for existing vault write targets.
  - confidence: 0.91
- **[P2] Walk query directories recursively:** `extensions/memory-wiki/src/query.ts:238-245`
  - body: Compile now walks nested directories, but query listing still only reads immediate markdown files in each group. Nested pages can be indexed while `wiki_search` and `wiki_get` miss them.
  - confidence: 0.9
- **[P2] Let configured groups take precedence:** `extensions/memory-wiki/src/config.ts:347-360`
  - body: Docs claim configured groups outrank built-ins, but resolved config appends defaults first and `findDirForKind` returns the first match. Custom dirs are created and scanned, yet write/report defaults continue to choose the built-in dirs.
  - confidence: 0.9
- **[P2] Validate vault page-group entries with the schema:** `extensions/memory-wiki/src/config.ts:260-279`
  - body: The standalone file loader only filters `dir`; invalid `kind` values are accepted into `config.pageGroups`, despite the docs promising malformed files fall back. Reuse the page-group schema for vault-loaded entries before compile/query see them.
  - confidence: 0.88
- **[P2] Keep memory-wiki on the package-boundary tsconfig:** `extensions/memory-wiki/tsconfig.json:2-14`
  - body: Replacing the shared package-boundary base with local compiler options removes the public SDK/path boundary checks for this bundled plugin. Keep the shared `extends` and adjust includes/excludes within that contract.
  - confidence: 0.88
