## Summary

Keep open: the feature direction is useful, and the author updated the body with copied live output, but the current PR still regresses wiki read/search behavior, sandboxed session visibility, vault path safety, report defaults, recursive query coverage, and package-boundary checks.

## What This Changes

The PR adds memory-wiki page group configuration via vault-local `.wiki-page-groups.json`/manifest config, dynamic directory resolution, recursive compile scanning, docs, and tests.

## Best Possible Solution

Ship page groups only after validating vault-relative dirs, preserving reports/defaults, passing resolved groups through every reader/writer/status path, keeping session and fs-safe guards, and proving a real nested/custom vault after the fixes.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the PR head defaults missing page groups to `[]`, passes no groups through wiki search and some callers, removes sandboxed propagation, and accepts unvalidated dirs that flow into filesystem paths. I did not run tests because this review is read-only.

## Solution Assessment

No. The requested direction is useful, but this branch is not the best mergeable shape until it preserves current defaults, security boundaries, package-boundary checks, and the full wiki read/search/status contract.
