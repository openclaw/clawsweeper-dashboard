## Summary

## What This Changes

The PR adds configurable memory-wiki page groups through `openclaw.json` and a vault `.wiki-page-groups.json`, then threads configured directories through initialization, compile, query, status, apply, manifest, tests, and docs.

## Best Possible Solution

Use one canonical page-group resolver and recursive walker across init, compile, query/get, status, lint, and report writers, then add runtime/manifest validation parity plus default and custom-dir upgrade tests.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: a nested configured page such as `资料/技术/AI/foo.md` is compiled/status-counted but not listed by `wiki_get`/fallback search, and a custom `report` group still receives generated dashboards under `reports/`. No local tests were run because this was a read-only review.

## Solution Assessment

No, this is not the best solution yet; it spreads page-group handling across local helpers and leaves query, index, report, lint, and manifest validation paths inconsistent. The safer fix is one shared directory contract used by every memory-wiki read/write surface.

- **[P1] Recurse page-group directories for wiki_get and search:** `extensions/memory-wiki/src/query.ts:253-256`
  - body: `listWikiMarkdownFiles` still reads only direct child files from each configured directory. With the advertised nested layout, compile/status can see `资料/技术/AI/foo.md`, but `readQueryableWikiPages` powers `wiki_get` and fallback wiki search from this non-recursive list, so nested pages become unreachable through those tools.
  - confidence: 0.93
- **[P1] Filter index sections by the configured directory:** `extensions/memory-wiki/src/compile.ts:995-1000`
  - body: After switching the index loop to `deriveAllPageGroups`, each section still renders `pages.filter((page) => page.kind === group.kind)`. Multiple directories for one kind, such as `综合`, `视图`, and `模板`, will each show all synthesis pages instead of only pages under that group directory.
  - confidence: 0.9
- **[P1] Write generated reports to the configured report directory:** `extensions/memory-wiki/src/compile.ts:83`
  - body: Dashboard report definitions still hardcode `reports/...`, and lint reports also write to `reports/lint.md`. If a user configures `report` as `报告`, generated report pages are written outside the configured report group and can be missed by the new page-group index/query model.
  - confidence: 0.91
- **[P2] Mirror pageGroups dir validation in the manifest schema:** `extensions/memory-wiki/openclaw.plugin.json:215-218`
  - body: The runtime Zod schema rejects absolute and traversal paths for `pageGroups[].dir`, but the manifest schema accepts any non-empty string. Setup and metadata validation should enforce the same filesystem boundary before this becomes a plugin config contract.
  - confidence: 0.84
