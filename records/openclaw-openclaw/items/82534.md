## Summary

Keep open. The latest head is still an implementation PR for a new memory-wiki config surface, but source review finds build, read-path, and security regressions, and the external contributor has not provided real vault proof.

## What This Changes

The PR adds memory-wiki page group configuration, a vault-local `.wiki-page-groups.json`, recursive compile scanning, docs/tests, and dynamic directory resolution across memory-wiki read and write paths.

## Best Possible Solution

Ship a narrower page-groups contract that validates relative dirs, threads resolved groups through every read/write/status path, preserves existing defaults and package-boundary checks, and proves a real vault with nested/custom groups.

## Reproduction Assessment

Yes for the PR regressions by source inspection: latest head imports a missing SDK export, defaults page reads to empty groups for existing callers, removes sandboxed propagation, and accepts path-escaping dirs. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The requested direction is useful, but this branch is not the narrowest maintainable solution until it preserves current security, default behavior, package-boundary checks, and the full wiki read/search contract.

  - confidence: 0.94
