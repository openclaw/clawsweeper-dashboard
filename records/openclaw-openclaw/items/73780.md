## Repair Work Prompt

Fix the Telegram repeated provider-error delivery bug tracked by https://github.com/openclaw/openclaw/issues/73780. Current main sends final Telegram `ReplyPayload` values marked `isError: true` through `deliverReplies` before `channels.telegram.errorPolicy` can suppress them; `errorPolicy` currently only affects delivery-failure logging in the dispatcher `onError` callback. Implement the fix inside the Telegram plugin boundary, likely around `extensions/telegram/src/bot-message-dispatch.ts` and `extensions/telegram/src/error-policy.ts`: before calling `deliverReplies`, resolve account/group/topic/thread `errorPolicy` and `errorCooldownMs`, suppress `silent`, suppress repeated `once` messages per account/chat/thread/message, and keep normal non-error replies, successful media replies, existing delivery-failure logging, and `silentErrorReplies` notification behavior intact. Add focused tests for repeated final `isError` rate-limit/overload payloads under `once` and `silent`, including group/topic inheritance where feasible. Update `docs/channels/telegram.md` to match the actual `always`/`once`/`silent` config contract and default cooldown, and add a user-facing `CHANGELOG.md` fix entry with reporter credit. Do not change provider fallback/cooldown semantics, do not add Z.ai-specific core behavior, and do not require live Telegram or provider credentials.
