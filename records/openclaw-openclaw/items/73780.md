Keep open. Current main still routes successful Telegram `isError` provider replies through normal reply delivery before `errorPolicy` can suppress them, so the repeated Telegram-visible rate-limit message behavior remains valid; the reporter's Z.ai account-limit escalation is separate upstream context, not a reason to close the OpenClaw channel bug.

## What This Changes

The issue requests reducing repeated Telegram-visible API rate-limit/high-concurrency error replies when an upstream provider reports a limit error.

## Best Possible Solution

Add Telegram-owned pre-send handling for provider/agent error payloads: resolve account/group/topic/thread `errorPolicy` and `errorCooldownMs` before `deliverReplies`, suppress `silent` and repeated `once` errors, preserve normal successful replies plus delivery-failure logging, and align docs, tests, and changelog with the `always`/`once`/`silent` contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text under `errorPolicy: "once"` or `"silent"`; current main forwards them to `deliverReplies` before any policy suppression can apply.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram dispatch boundary because provider fallback/cooldown handling already exists upstream, while the remaining observable bug is repeated successful delivery of error-message replies to Telegram chats.
