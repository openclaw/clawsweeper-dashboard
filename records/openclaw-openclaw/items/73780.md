## Summary

Keep open. Current main still sends final Telegram `ReplyPayload` values with `isError: true` through normal delivery, while the Telegram `errorPolicy` is only consulted for Telegram delivery failures, so repeated provider rate-limit or overload replies can still flood Telegram despite the suppression controls from #51914. The reporter's Z.ai account-reset path is external support, but the OpenClaw suppression gap is narrow and source-backed.

## What This Changes

The issue asks OpenClaw to stop repeated API rate-limit or overload errors from flooding Telegram replies.

## Best Possible Solution

Apply the existing Telegram error policy at the Telegram dispatch boundary before provider-error replies are delivered, with regression coverage and docs/changelog alignment.

## Reproduction Assessment

Yes, at source level. Dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` without policy suppression.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned pre-send suppression using the existing policy and cooldown helpers, without changing provider failover, retry behavior, or adding a new config knob.
