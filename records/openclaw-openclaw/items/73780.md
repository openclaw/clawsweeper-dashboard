## Summary

Keep open. Current main still delivers Telegram `ReplyPayload` values with `isError: true` before applying `channels.telegram.errorPolicy`, so repeated provider rate-limit or overload messages can still be posted into Telegram chats.

## What This Changes

The issue reports repeated Telegram-visible API rate-limit errors from an OpenClaw Telegram workflow using Z.ai.

## Best Possible Solution

Add Telegram-owned pre-send suppression for provider/agent error replies using existing `errorPolicy` and `errorCooldownMs`, align the Telegram docs with the current enum, and leave provider account remediation outside this repository.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` before policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable repair is at the Telegram dispatch boundary because provider cooldown and fallback behavior already exists upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.
