## Summary

Keep open. Current main still does not apply Telegram `errorPolicy` before provider/rate-limit error replies are delivered, so the existing suppression contract does not cover the reported Telegram-visible error flood.

## What This Changes

The issue asks OpenClaw to stop repeatedly posting API/provider rate-limit errors into Telegram chats.

## Best Possible Solution

Apply the existing Telegram error policy at the Telegram send boundary for provider/error replies, with regression coverage and docs/changelog alignment.

## Reproduction Assessment

Yes, at source level. Dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` without policy suppression.

## Solution Assessment

Yes. The narrowest maintainable repair is Telegram-owned pre-send suppression using the existing policy and cooldown helpers, without changing provider failover, core retry behavior, or adding another config knob.
