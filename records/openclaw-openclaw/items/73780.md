## Summary

Keep open. Current main still forwards Telegram `ReplyPayload` values with `isError: true` through normal delivery before applying `channels.telegram.errorPolicy`, so repeated provider rate-limit or overload messages can still be posted into Telegram chats.

## What This Changes

The issue asks OpenClaw to stop repeated Telegram-visible API rate-limit error messages when a provider/account is rate-limited.

## Best Possible Solution

Add Telegram-owned pre-send suppression for provider/agent error replies using the existing `errorPolicy` and cooldown config, while preserving normal reply delivery, useful non-repeated errors, and Telegram send-failure logging.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` before any policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider cooldown and fallback behavior already exists upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.
