## Summary

Keep open. Current main still applies Telegram `errorPolicy` only after Telegram delivery throws, while successful outbound `isError` provider/agent replies are forwarded through normal Telegram delivery, so the OpenClaw-visible repeated error-message behavior remains valid despite the reporter's separate upstream Z.ai escalation.

## What This Changes

The issue asks OpenClaw to stop repeatedly surfacing Telegram-visible API rate-limit errors from provider failures.

## Best Possible Solution

Add Telegram pre-send handling for provider/agent `isError` rate-limit and overload replies: resolve the existing account/group/topic/thread `errorPolicy` and `errorCooldownMs` scope before calling `deliverReplies`, suppress `silent` and repeated `once` errors, preserve normal successful replies and delivery-failure logging, and update Telegram tests, docs, and changelog to match the `always`/`once`/`silent` contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit/overload text under `errorPolicy: "once"` or `"silent"`; current main still forwards those payloads to `deliverReplies` before any policy suppression can apply.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram dispatch boundary because provider fallback/cooldown handling already exists upstream, while the remaining observable bug is repeated successful delivery of error-message replies to Telegram chats.
