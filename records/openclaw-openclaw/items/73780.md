## Summary

Keep open. Current main still sends successful Telegram `isError` provider replies through normal delivery before `errorPolicy` can suppress them, so the repeated Telegram-visible rate-limit message behavior remains valid even though the reporter also has a separate upstream Z.ai account-limit escalation.

## What This Changes

The issue reports repeated Telegram-visible API rate-limit errors when a provider returns rate-limit or overload failures.

## Best Possible Solution

Add Telegram-owned pre-send handling for provider/agent rate-limit and overload `isError` replies: resolve account/group/topic/thread `errorPolicy` and `errorCooldownMs` before `deliverReplies`, suppress `silent` and repeated `once` errors, preserve normal successful replies and delivery-failure logging, and align docs/tests/changelog with the `always`/`once`/`silent` contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit/overload text under `errorPolicy: "once"` or `"silent"`; current main forwards those payloads to `deliverReplies` before any policy suppression can apply.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram dispatch boundary because provider fallback/cooldown handling already exists upstream, while the remaining observable bug is repeated successful delivery of error-message replies to Telegram chats.
