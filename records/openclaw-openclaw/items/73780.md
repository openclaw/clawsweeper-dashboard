## Summary

Current main still has an OpenClaw-side Telegram delivery gap: provider/agent rate-limit replies marked `isError` are sent through normal Telegram delivery before `errorPolicy` can suppress them. The reporter’s Z.ai account-limit escalation is external context, but the repeated Telegram-visible error-message behavior remains a valid, narrow source-level bug.

## What This Changes

Request to stop repeated Telegram-visible API rate-limit errors from spamming Telegram chats when an upstream provider such as Z.ai reports a rate-limit or high-concurrency failure.

## Best Possible Solution

Add Telegram-owned pre-send handling for provider/agent rate-limit and overload `isError` replies: resolve account/group/topic/thread `errorPolicy` and `errorCooldownMs` before `deliverReplies`, suppress `silent` and repeated `once` errors, preserve normal successful replies and delivery-failure logging, and update focused tests, docs, and changelog for the current `always`/`once`/`silent` contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text under `errorPolicy: "once"` or `"silent"`; current main forwards them to `deliverReplies` before policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider fallback/cooldown handling already exists upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.

## Repair Work Prompt

Fix the Telegram repeated provider-error delivery bug tracked by https://github.com/openclaw/openclaw/issues/73780. Current main sends final Telegram `ReplyPayload` values marked `isError: true` through `deliverReplies` before `channels.telegram.errorPolicy` can suppress them; `errorPolicy` currently only affects delivery-failure logging in the dispatcher `onError` callback. Implement the fix inside the Telegram plugin boundary, likely around `extensions/telegram/src/bot-message-dispatch.ts` and `extensions/telegram/src/error-policy.ts`: before calling `deliverReplies`, resolve account/group/topic/thread `errorPolicy` and `errorCooldownMs`, suppress `silent`, suppress repeated `once` messages per account/chat/thread/message, and keep normal non-error replies, successful media replies, existing delivery-failure logging, and `silentErrorReplies` notification behavior intact. Add focused tests for repeated final `isError` rate-limit/overload payloads under `once` and `silent`, including group/topic inheritance where feasible. Update `docs/channels/telegram.md` to match the actual `always`/`once`/`silent` config contract and default cooldown, and add a user-facing `CHANGELOG.md` fix entry with reporter credit. Do not change provider fallback/cooldown semantics, do not add Z.ai-specific core behavior, and do not require live Telegram or provider credentials.
