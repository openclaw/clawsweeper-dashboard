## Summary

Keep open. Current main still sends Telegram `isError: true` provider rate-limit/overload replies through normal delivery before the existing Telegram `errorPolicy` can suppress or de-duplicate them; the Z.ai account reset request is external, but the repeated Telegram-visible error behavior is a narrow OpenClaw bug.

## What This Changes

Suppress or de-duplicate repeated Telegram-visible provider rate-limit and overload error replies using the existing Telegram error-policy controls.

## Best Possible Solution

Apply a Telegram dispatch-layer fix that resolves account/group/topic/thread `errorPolicy` before sending provider/agent `isError` replies, suppresses `silent` and repeated `once` errors, and updates tests, docs, and changelog without changing provider fallback or Z.ai account handling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `channels.telegram.errorPolicy` is `once` or `silent`; current main forwards them to `deliverReplies` before policy suppression runs.

## Solution Assessment

Yes. The narrowest maintainable solution is in the Telegram dispatch boundary because provider cooldown/fallback handling already exists upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.
