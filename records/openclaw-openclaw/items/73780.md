Keep open. Current main still has the source-level Telegram bug: final provider/rate-limit error replies marked `isError` are delivered normally, while `errorPolicy` suppression only runs for Telegram delivery failures; the reporter's Z.ai account reset request is external but does not remove the OpenClaw-side spam issue.

## What This Changes

The issue reports repeated provider API rate-limit errors being sent through Telegram instead of being suppressed according to the existing Telegram error policy.

## Best Possible Solution

Apply Telegram `errorPolicy` in the active plugin before final provider-error replies are sent, keep provider account reset outside this repo, and align docs/tests with the existing enum contract.

Yes, at source level: dispatch repeated final Telegram `ReplyPayload` values marked `isError: true` while `errorPolicy` is `once` or `silent`; current main sends them through normal delivery without consulting the policy.

## Solution Assessment

Yes. The narrowest maintainable fix is Telegram-owned pre-send suppression using the existing policy and cooldown helper, without changing provider failover, Z.ai behavior, or adding a new config knob.
