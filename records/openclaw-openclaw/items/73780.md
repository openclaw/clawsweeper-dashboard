## What This Changes

Request to stop repeated provider API rate-limit errors from being posted through Telegram.

## Solution Assessment

## Repair Work Prompt

Fix repeated Telegram provider-error replies for https://github.com/openclaw/openclaw/issues/73780. Current main marks rate-limit/overload failures as final `ReplyPayload` values with `isError: true`, but `extensions/telegram/src/bot-message-dispatch.ts` only applies `errorPolicy` in the delivery-failure `onError` path. Reuse `resolveTelegramErrorPolicy`, `shouldSuppressTelegramError`, and `buildTelegramErrorScopeKey` at the final provider-error dispatch boundary so `silent` suppresses these replies and `once` sends only the first matching error per account/chat/thread cooldown. Do not change provider failover, Z.ai account handling, or add a new config option. Add/update focused tests for final error payload suppression and keep docs/changelog aligned with the existing `always`/`once`/`silent` contract.
