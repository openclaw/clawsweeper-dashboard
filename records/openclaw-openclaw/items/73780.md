## What This Changes

The issue reports repeated Telegram-visible API rate-limit errors when a provider returns rate-limit or overload failures.

## Best Possible Solution

Add Telegram pre-send handling for provider/agent rate-limit and overload `isError` replies: resolve account/group/topic/thread `errorPolicy` and `errorCooldownMs` before `deliverReplies`, suppress `silent` and repeated `once` errors, preserve normal successful replies and delivery-failure logging, and update Telegram tests, docs, and changelog to match the `always`/`once`/`silent` contract.

Yes. ## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram dispatch boundary because provider fallback/cooldown handling already exists upstream, while the remaining observable bug is repeated successful delivery of error-message replies to Telegram chats.
