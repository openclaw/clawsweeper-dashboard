## Summary

Keep open. Current main still routes final Telegram `ReplyPayload` values with `isError: true` through normal delivery, while `errorPolicy` is only applied to Telegram delivery failures, so repeated provider rate-limit or overload messages can still be posted to chat.

## What This Changes

Suppress or throttle repeated Telegram API/provider rate-limit error replies using the existing Telegram error policy instead of repeatedly posting them to chat.

## Best Possible Solution

Apply the existing Telegram `errorPolicy` at the dispatch boundary for final provider-error replies, with focused regression coverage and docs/changelog alignment while leaving provider account resets to Z.ai support.

Yes, at source level. Dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches normal delivery without policy suppression.

## Solution Assessment

Yes. The narrowest maintainable fix is Telegram-owned pre-send suppression using the existing policy and cooldown helper, without changing provider failover, Z.ai behavior, or adding a new config knob.
