## Summary

Keep open. Current main still sends Telegram `ReplyPayload` values with `isError: true` through normal reply delivery before applying `channels.telegram.errorPolicy`; the policy helper is only consulted for Telegram send failures, so repeated provider rate-limit or overload replies can still be posted. The reporter's Z.ai escalation is external, but the OpenClaw-owned Telegram reply behavior remains actionable.

## What This Changes

Prevent repeated Telegram API rate-limit or overload error replies from spamming chats by honoring the existing Telegram error policy before delivery.

## Best Possible Solution

Apply Telegram-owned pre-send suppression for provider/agent error replies using the existing `errorPolicy` and cooldown settings, while preserving normal delivery for successful replies and send-failure logging.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`. Current main reaches `deliverReplies` before any policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider cooldown/fallback already exists upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.
