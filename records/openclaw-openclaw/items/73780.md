## Summary

Keep open. Current main still sends Telegram `ReplyPayload` values with `isError: true` through normal delivery before applying `channels.telegram.errorPolicy`, so repeated provider rate-limit or overload replies can still be posted to Telegram chats; the Z.ai account escalation is external, but the OpenClaw Telegram reply behavior remains actionable.

## What This Changes

Telegram should suppress or cooldown repeated provider/agent error replies according to the existing Telegram error policy controls.

## Best Possible Solution

Add Telegram-owned pre-send suppression for final provider/agent error payloads using the existing `errorPolicy` and cooldown helpers, while preserving normal delivery for `always` and existing Telegram send-failure logging.

## Reproduction Assessment

Yes, at source level. Dispatch repeated final Telegram payloads with `isError: true` and rate-limit or overload copy while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` before any policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider fallback/cooldown already exists upstream and the remaining observable bug is repeated successful delivery of error replies into Telegram chats.
