## Summary

Keep open. Current main still delivers Telegram `isError` replies before applying Telegram `errorPolicy`, so provider rate-limit or overload failures can repeatedly appear in chats despite an existing suppression config; the Z.ai account block itself is external provider support scope.

## What This Changes

Request to stop repeated provider API rate-limit errors from being posted into Telegram chats.

## Best Possible Solution

Apply Telegram-owned pre-send suppression for repeated final `isError` provider/agent replies using the existing `errorPolicy` and `errorCooldownMs`, with regression coverage and any needed docs/config wording alignment.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` before any policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider cooldown/fallback behavior already exists upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.
