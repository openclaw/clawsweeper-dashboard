## Summary

Keep open. Current main still forwards Telegram agent/provider error payloads to normal delivery; the existing Telegram `errorPolicy` path only throttles Telegram delivery exceptions, not successful delivery of repeated `isError` rate-limit replies. The reporter's Z.ai account-side 1302 block is external, but the OpenClaw-visible Telegram error-message spam remains a narrow valid bug.

## What This Changes

The issue asks OpenClaw to stop repeatedly sending Telegram-visible API rate-limit error replies when the upstream provider is rate-limited or blocked.

## Best Possible Solution

Add Telegram pre-send handling for agent/provider error payloads: apply the existing account/group/topic/thread `errorPolicy` and `errorCooldownMs` before calling `deliverReplies`, preserve normal successful replies and `silentErrorReplies`, and align Telegram docs/tests with the actual `always`/`once`/`silent` config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to deliver repeated Telegram `ReplyPayload` values with `isError: true` and rate-limit text under `errorPolicy: "once"` or `"silent"`; current main still calls `deliverReplies` for those payloads instead of suppressing or coalescing them before Telegram delivery.

## Solution Assessment

Yes. The narrowest maintainable repair belongs at the Telegram dispatch/delivery boundary, because provider fallback/cooldown behavior is already tracked separately and the remaining OpenClaw bug is successful Telegram delivery of repeated error replies.
