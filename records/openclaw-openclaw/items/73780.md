## Summary

Keep open. Current main still sends Telegram `ReplyPayload` values with `isError: true` through normal delivery, so repeated provider rate-limit or overload messages can reach the chat before `channels.telegram.errorPolicy` can suppress them; the Z.ai account-limit escalation is external, but the Telegram-visible repeat-error behavior remains a narrow OpenClaw bug.

## What This Changes

The issue requests suppression or throttling of repeated Telegram-visible API rate-limit errors during provider failures.

## Best Possible Solution

Implement Telegram-owned pre-send handling for provider and agent error replies, applying account/group/topic/thread `errorPolicy` before delivery while keeping normal replies and delivery-failure logging intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text under `errorPolicy: "once"` or `"silent"`; current main reaches `deliverReplies` before policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider fallback and cooldown handling already exist upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.

## Repair Work Prompt

Fix the Telegram repeat-error behavior reported in openclaw/openclaw#73780. Current main sends final `ReplyPayload` values with `isError: true` through `deliverReplies` before `channels.telegram.errorPolicy` can suppress repeated provider rate-limit or overload messages. Keep the fix owned by the Telegram plugin: resolve account/group/topic/thread error policy before sending successful provider/agent error replies, suppress `silent`, suppress repeated matching `once` errors within `errorCooldownMs`, and preserve normal replies, media replies, delivery-failure logging, preview cleanup, and `silentErrorReplies` notification semantics. Do not change provider fallback/cooldown behavior or attempt to solve the external Z.ai account reset. Add focused regression tests for `errorPolicy: "silent"` and repeated `"once"` final `isError` payloads in Telegram dispatch, update `docs/channels/telegram.md` if the documented values/defaults are stale, and add a user-facing `CHANGELOG.md` fix entry.
