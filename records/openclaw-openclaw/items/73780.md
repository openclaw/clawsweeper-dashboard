## Summary

Keep open. Current main still sends Telegram `ReplyPayload` values marked `isError: true` through normal delivery before applying `channels.telegram.errorPolicy`, so the external Z.ai account escalation does not resolve the OpenClaw-owned repeated Telegram-visible error replies.

## What This Changes

Suppress repeated Telegram-visible provider/API rate-limit error replies according to Telegram error-policy settings.

## Best Possible Solution

Apply Telegram-owned pre-send suppression for provider/agent error replies using the existing error-policy and cooldown config, while preserving normal replies, delivered-error notification behavior, and Telegram send-failure logging.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit/overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` before policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider cooldown/fallback behavior already exists upstream and the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.

## Repair Work Prompt

Fix the Telegram plugin bug where provider/agent error `ReplyPayload` values with `isError: true` are delivered normally before `channels.telegram.errorPolicy` is applied. Implement pre-send suppression in `extensions/telegram/src/bot-message-dispatch.ts` using the existing Telegram-owned `resolveTelegramErrorPolicy`, `buildTelegramErrorScopeKey`, and `shouldSuppressTelegramError` helpers with account/group/topic/DM scope and configured cooldown. Preserve `silentErrorReplies` semantics for error payloads that are still delivered, preserve normal non-error replies and Telegram send-failure logging, and do not move provider cooldown/fallback policy into core. Add focused tests in `extensions/telegram/src/bot-message-dispatch.test.ts` and `extensions/telegram/src/error-policy.test.ts` for `silent`, `once`, cooldown expiry, distinct messages/scopes, and normal reply delivery. Update `docs/channels/telegram.md` and `CHANGELOG.md` for the user-facing fix, and align docs with the actual schema values `always | once | silent`.
