## Summary

Keep open. Current main still sends Telegram `ReplyPayload` values marked `isError: true` through normal reply delivery before applying `channels.telegram.errorPolicy`, so repeated provider rate-limit or overload messages can continue to post into Telegram chats; the reporter's Z.ai escalation is external support context, but the OpenClaw-owned Telegram reply behavior remains actionable.

## What This Changes

Suppress repeated Telegram provider/API error replies according to the existing Telegram error-policy settings.

## Best Possible Solution

Add Telegram-owned pre-send suppression for provider/agent error replies using the existing policy and cooldown semantics, while preserving default error delivery and Telegram send-failure logging.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` before policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider cooldown/fallback logic already exists upstream, while the observable bug is repeated successful delivery of provider-error replies into Telegram chats.

## Repair Work Prompt

Fix this issue by making Telegram suppress repeated provider/agent error replies before they are sent. In `extensions/telegram/src/bot-message-dispatch.ts`, apply the existing Telegram `errorPolicy`/`errorCooldownMs` resolution to final `ReplyPayload` values with `isError: true` before calling `deliverReplies`: `silent` should suppress, `once` should deliver the first matching error per account/chat/thread/cooldown and suppress repeats, and the default policy should preserve current delivery. Keep the existing `onError` handling for Telegram send failures, preserve `silentErrorReplies` semantics for delivered error replies, and do not change provider cooldown/fallback behavior or add config knobs. Add regression coverage for repeated rate-limit/overload `isError` payloads in Telegram dispatcher tests and any helper tests needed. Align `docs/channels/telegram.md` with the actual public policy values if touched, and add a single-line `CHANGELOG.md` fix entry for this issue.
