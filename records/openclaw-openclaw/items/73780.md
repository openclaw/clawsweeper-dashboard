## Summary

Keep open. Current main still forwards Telegram `isError` provider replies through normal delivery before Telegram `errorPolicy` can suppress repeated rate-limit text; the reporter's Z.ai escalation is separate upstream context, but the OpenClaw Telegram-visible repeat-error behavior remains valid.

## What This Changes

Request to stop Telegram from repeatedly posting provider rate-limit error replies by honoring Telegram error suppression and cooldown settings for successful outbound error payloads.

## Best Possible Solution

Add Telegram-owned pre-send handling for provider/agent rate-limit and overload `isError` replies: resolve account/group/topic/thread `errorPolicy` and `errorCooldownMs` before `deliverReplies`, suppress `silent` and repeated `once` errors, preserve normal successful replies and delivery-failure logging, and update tests, docs, and changelog to match the `always`/`once`/`silent` contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text under `errorPolicy: "once"` or `"silent"`; current main forwards those payloads to `deliverReplies` before suppression can apply.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram dispatch boundary because provider fallback and cooldown handling already exists upstream, while the remaining observable bug is repeated successful delivery of error-message replies to Telegram chats.
