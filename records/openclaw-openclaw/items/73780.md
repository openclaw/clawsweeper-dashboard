## Summary

Keep open. Current main still applies Telegram `errorPolicy` only after Telegram delivery failures, while final error payloads, including provider rate-limit or overload copy, are delivered through the normal Telegram send path.

## What This Changes

The issue asks OpenClaw to stop repeated Telegram-facing API rate-limit error replies when a provider reports rate-limit or overload failures.

## Best Possible Solution

Apply the existing Telegram `errorPolicy`/`errorCooldownMs` before sending Telegram error payloads, with focused regression coverage and docs/changelog alignment while leaving provider retry and failover semantics unchanged.

## Reproduction Assessment

Yes, at source level. Dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` without policy suppression.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned pre-send suppression using the existing policy and cooldown helpers, not a provider retry change or new config option.

Fix https://github.com/openclaw/openclaw/issues/73780. Current Telegram dispatch applies `errorPolicy` only after Telegram delivery failures, so final `ReplyPayload` values marked `isError: true` with provider rate-limit or overload copy still get sent normally. Implement Telegram-owned pre-send suppression for error payloads using the existing `errorPolicy` and `errorCooldownMs` inheritance at account, group/direct, and topic scope.
