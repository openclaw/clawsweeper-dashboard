## Summary

Keep open. Current main still forwards Telegram `ReplyPayload` values with `isError: true` through normal delivery before applying `channels.telegram.errorPolicy`, so repeated provider rate-limit or overload messages can still be posted into Telegram chats.

## What This Changes

The issue requests suppression or throttling of repeated Telegram-visible API rate-limit errors during provider failures.

## Best Possible Solution

Add Telegram-owned pre-send suppression for provider/agent error replies using the existing `errorPolicy` and cooldown config, while preserving normal reply delivery and Telegram send-failure logging.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` before policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider cooldown and fallback behavior already exists upstream, while the remaining observable bug is repeated successful delivery of provider-error replies into Telegram chats.

## Repair Work Prompt

Fix the Telegram repeat-error behavior reported at https://github.com/openclaw/openclaw/issues/73780. Current main sends final `ReplyPayload` values with `isError: true` through `deliverReplies` before `channels.telegram.errorPolicy` can suppress repeated provider rate-limit or overload messages. Keep the fix owned by the Telegram plugin: resolve account/group/topic/thread error policy before sending successful provider/agent error replies, suppress `silent`, suppress repeated matching `once` errors within `errorCooldownMs`, and preserve normal replies, media replies, delivery-failure logging, preview cleanup, and `silentErrorReplies` notification semantics. Do not change provider fallback/cooldown behavior or attempt to solve the external Z.ai account reset. Add focused regression tests for `errorPolicy: "silent"` and repeated `"once"` final `isError` payloads in Telegram dispatch, update `docs/channels/telegram.md` so documented values/defaults match the current contract, and add a single-line user-facing `CHANGELOG.md` fix entry.
