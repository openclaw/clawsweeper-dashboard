## Summary

Keep open. Current main still sends Telegram `ReplyPayload` values with `isError: true` through the normal delivery path before applying `channels.telegram.errorPolicy`, so repeated provider rate-limit or overload messages can still be posted into Telegram chats even though the config/docs expose suppression and cooldown controls.

## What This Changes

The issue reports repeated Telegram chat messages for upstream API rate-limit/high-concurrency failures and expects existing Telegram error-reply controls to suppress repeated errors.

## Best Possible Solution

Add Telegram-owned pre-send suppression for repeated provider/agent error replies using the existing `errorPolicy` and cooldown config, while preserving normal reply delivery, distinct error messages, and Telegram send-failure logging.

## Reproduction Assessment

Yes for source-level reproduction: dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`. Current main reaches `deliverReplies` before policy suppression can run.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram dispatch boundary because provider fallback/cooldown behavior already exists upstream, while the remaining user-visible problem is repeated successful delivery of provider-error replies into Telegram chats.

## Repair Work Prompt

Fix the OpenClaw-owned part of https://github.com/openclaw/openclaw/issues/73780: repeated Telegram delivery of provider/agent error replies. Current main applies `channels.telegram.errorPolicy` only to Telegram send failures in `onError`; successful final `ReplyPayload` values with `isError: true` still go through `deliverReplies`, so repeated rate-limit/overload copy can spam chats. Add Telegram-owned pre-send suppression for repeated provider/agent error replies using the existing `resolveTelegramErrorPolicy`, `shouldSuppressTelegramError`, account/chat/thread scope, and `errorCooldownMs`. Preserve the default `always` behavior, `silentErrorReplies` notification semantics, native command error delivery unless intentionally covered by the same policy, distinct error messages, normal non-error replies, and Telegram send-failure logging. Do not change core provider fallback/cooldown policy or add new config. Add regression coverage in Telegram dispatch/error-policy tests and update docs/changelog if the user-facing behavior or documented values change.
