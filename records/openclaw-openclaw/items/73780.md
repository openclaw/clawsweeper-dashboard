## Summary

Keep open. Current main still applies Telegram `errorPolicy` only to Telegram delivery failures, while final `ReplyPayload` values with `isError: true` are sent through normal delivery, so repeated provider rate-limit or overload errors can still be posted to Telegram despite the existing suppression controls introduced by #51914.

## What This Changes

The issue asks Telegram to stop repeatedly posting provider API rate-limit or overload error replies to chats.

## Best Possible Solution

Apply the existing Telegram error policy at the Telegram dispatch boundary before provider-error replies are sent, with focused regression coverage and docs/changelog alignment.

## Reproduction Assessment

Yes, at source level. Dispatch repeated final Telegram `ReplyPayload` values with `isError: true` and rate-limit or overload text while `errorPolicy` is `once` or `silent`; current main reaches `deliverReplies` without policy suppression.

## Solution Assessment

Yes. The narrowest maintainable fix is Telegram-owned pre-send suppression using the existing policy and cooldown helpers, without changing provider failover, core retry behavior, or adding a new config knob.

## Repair Work Prompt

Fix the Telegram repeated provider-error reply bug reported in https://github.com/openclaw/openclaw/issues/73780. Current main applies `channels.telegram.errorPolicy` only from the Telegram delivery `onError` callback, while final `ReplyPayload` values with `isError: true` are sent normally via `deliverReplies`, so provider rate-limit/overload text can repeat in chats. Keep the fix in the Telegram plugin boundary: before sending final error payloads, resolve account/group/topic `errorPolicy` and `errorCooldownMs`, suppress `silent`, and suppress repeated matching errors for `once` using the existing scope/cooldown helpers. Preserve default `always` behavior, do not change provider failover/cooldown semantics, and do not add a new config knob. Add regression coverage for `errorPolicy: "once"` and `"silent"` on final `isError` replies while keeping normal non-error replies and distinct error messages deliverable. Update `docs/channels/telegram.md` if the current docs wording still disagrees with the schema, and add a user-facing `CHANGELOG.md` fix entry.
