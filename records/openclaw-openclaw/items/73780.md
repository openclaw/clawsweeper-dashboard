## Summary

Keep open. Current main still has the Telegram-side bug: final provider/rate-limit replies marked `isError` are delivered normally, while the configured Telegram `errorPolicy` is only consulted after Telegram delivery itself fails.

## What This Changes

The issue asks OpenClaw to stop repeated Telegram-facing API/provider rate-limit error replies from spamming chats.

## Best Possible Solution

Apply Telegram `errorPolicy` before final provider-error replies are sent, keep provider account reset outside this repo, and update tests/docs to match the existing enum contract.

## Reproduction Assessment

Yes, at source level: dispatch repeated final Telegram `ReplyPayload` values marked `isError: true` while `errorPolicy` is `once` or `silent`; current main sends them through normal delivery without consulting the policy. I did not run a live Telegram/Z.ai reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is Telegram-owned pre-send suppression using the existing policy and cooldown helper, without changing provider failover, Z.ai behavior, or adding a new config knob.

Current main sends final Telegram `ReplyPayload` values with `isError: true` through normal delivery without applying `channels.telegram.errorPolicy`, so repeated provider/API rate-limit failures can spam Telegram chats even when the config is `once` or `silent`.
