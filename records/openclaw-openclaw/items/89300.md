## Summary

Keep open. Current main and the latest release still make the normal `[model-fetch]` start/response diagnostics opt-in at info level, so the reported behavior is real and not implemented away; because the current docs and changelog also frame this as targeted diagnostics, the remaining decision is the intended default logging contract.

## What This Changes

The issue asks OpenClaw to restore default info-level `[model-fetch]` request/response diagnostics, or at least make the new `OPENCLAW_DEBUG_MODEL_TRANSPORT` flag prominent in release notes.

## Best Possible Solution

Maintainers should define the supported model-transport logging contract: if the old default is intended, keep bounded request/response timing at `info` by default while leaving payload/SSE details behind explicit flags; if the opt-in behavior is intended, make the release-note/docs wording explicit enough for upgraded users.

## Reproduction Assessment

Yes. Source inspection shows that with no debug env flags, current main and `v2026.5.28` send `[model-fetch]` start/response diagnostics to `log.debug`, and default file logging is `info`, so those lines are filtered from the default log.

## Solution Assessment

Unclear pending maintainer policy. The mechanical fix is small if default info-level transport timing is a contract, but current docs and changelog describe the new behavior as targeted diagnostics, so the safest next step is an owner decision on the default.
