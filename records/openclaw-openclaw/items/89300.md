Keep open. Current `main` and the latest release still make default `[model-fetch]` start/response diagnostics opt-in at info level, and the linked fixing PR is still open, cleanly mergeable, and explicitly closes this issue if merged.

## What This Changes

The issue asks to restore default info-level `[model-fetch]` request/response diagnostics or document the opt-in env flag prominently.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/89648; the clean end state is explicit docs plus a helper contract that promotes only `[model-fetch]` metadata to info while leaving payload/SSE diagnostics gated.

## Reproduction Assessment

Yes. Source inspection shows current `main` sends `[model-fetch]` start/response diagnostics through `emitModelTransportDebug`, which falls back to `debug` when no model transport debug env flag is set; I did not run a live gateway in this read-only cleanup review.

## Solution Assessment

Unclear pending maintainer policy, but the linked PR is the narrowest plausible fix if default info-level metadata is intended. It separates `[model-fetch]` metadata from payload/SSE diagnostics instead of raising all model transport diagnostics by default.
