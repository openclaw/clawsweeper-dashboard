- `P2`: This is a normal-priority memory-core regression fix with limited blast radius, but it still needs correction before merge.

- `merge-risk: 🚨 session-state`: Memory index identity, chunk rows, cache entries, and vector search state can remain mismatched or dirty if the empty model is still written outside the identity helper.

## Summary

Keep open: the patch is not merge-ready because it only changes the identity fallback while active providers with an empty model can still propagate that empty model into provider keys, cache rows, chunk rows, vector-search filters, and index metadata; the supplied proof also does not exercise that real runtime path.

## What This Changes

The PR changes memory-core identity comparison and pre-init fallback model resolution so empty or whitespace provider models fall back to `fts-only`, with focused identity-state tests.

## Best Possible Solution

Normalize or reject empty active provider models at the provider creation/manager boundary, then use one resolved model value consistently for identity, provider key, cache, chunk, meta, and vector-search paths, with manager-level regression and real runtime proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction shape: construct a memory embedding provider that returns `model: ""`, then run sync/status and observe the raw model reaching identity, meta, chunks, cache, and search. I did not execute that repro because this review is read-only.

## Solution Assessment

No. The PR is a plausible mitigation for one identity comparison, but the best fix is to normalize the resolved provider model once at the provider/manager boundary and prove the full runtime reindex path no longer writes or searches empty-model state.
