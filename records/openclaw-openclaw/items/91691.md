## Summary

Keep open: the memory empty-model bug is real, but this PR is not the best fix because it treats an active empty-model provider as `fts-only` only in the identity helper while current runtime paths still write, cache, key, and search with the raw empty model.

## What This Changes

The PR changes memory-core identity comparison and pre-init fallback model resolution so empty or whitespace provider models compare as `fts-only`, and adds helper-level identity-state tests.

## Best Possible Solution

Normalize or reject empty provider models once at the provider creation or manager boundary, reuse that normalized model through identity, metadata, cache, chunks, and search, then land with manager-level regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction shape: return a provider with `model: ""`, then manager identity, metadata, cache, chunk, and vector-search paths consume that raw model; I did not execute it because this review is read-only.

## Solution Assessment

No. This PR is a plausible identity-layer mitigation, but the maintainable fix is provider-boundary normalization or rejection, with https://github.com/openclaw/openclaw/pull/91660 already proposing the cleaner root-cause path.
