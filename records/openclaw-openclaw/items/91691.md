## Summary

Keep this PR open, but do not merge it as-is: the new fallback changes an empty model under a configured semantic provider into `fts-only`, so the reported semantic index can still stay dirty instead of comparing against the configured or adapter-default embedding model. The supplied proof also does not exercise the real empty-provider-model path.

## What This Changes

The PR changes memory-core index identity comparison to treat empty or whitespace provider models as `fts-only` and adds three unit tests for that fallback.

## Best Possible Solution

Resolve or backfill the provider model before identity comparison, and keep `fts-only` reserved for a truly null provider; add regression coverage and real proof for an empty provider model with an existing semantic OpenAI or Gemini index.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the patch defect: with a semantic `meta.model` and `provider.model === ""`, the proposed code compares against `fts-only` and still returns a mismatch. I did not run tests because this is a read-only review.

## Solution Assessment

No. The identity helper cannot know the configured or adapter-default semantic model from `{ id, model }` alone, so the better fix is to normalize/backfill the provider model before this comparison or pass a resolved expected model from the caller.
