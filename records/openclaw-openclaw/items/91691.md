## Summary

Keep this PR open, but not merge-ready: the empty-model symptom is real from source, yet the patch handles it at the identity consumer by mapping an active provider’s empty model to `fts-only` while the indexing and metadata write paths still use the raw empty provider model. The best fix needs to normalize or reject the active provider result before identity, chunk, cache, and meta paths observe it.

## What This Changes

The PR changes memory-core identity comparison and pre-init provider fallback handling so empty provider models become `fts-only`, and adds three identity-state unit tests.

## Best Possible Solution

Normalize or reject empty active provider models at the provider creation boundary, then keep `fts-only` reserved for the no-provider state so identity, cache keys, chunk writes, and metadata all share one canonical model.

## Reproduction Assessment

Yes for the narrow empty-string path from source: current main reads `params.provider.model` directly, so `model: ""` produces an empty expected model. I did not reproduce the full upgrade/runtime failure in a live setup during this read-only review.

## Solution Assessment

No. The patch is a plausible symptom guard, but the maintainable fix is to normalize the provider model where the provider result is created, or fail that provider state, before index identity and write paths consume it.
