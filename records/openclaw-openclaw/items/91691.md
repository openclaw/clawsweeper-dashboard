## Summary

Keep this PR open, but do not merge it as-is: it changes the empty-model identity fallback to `fts-only`, which makes the reason more readable but does not restore semantic/vector indexes that should compare against the configured embedding model.

## What This Changes

The PR changes memory-core index identity comparison so empty or whitespace-only `provider.model` falls back to `fts-only`, and adds three unit tests for that fallback.

## Best Possible Solution

Backfill or resolve the configured/default embedding model at the provider identity boundary, then prove that an existing semantic index no longer reports an empty or `fts-only` expected model when the provider object has an empty model.

## Reproduction Assessment

Yes from source inspection: current main compares a provider object with `model: ""` against index metadata and then search returns no results when identity is mismatched. I did not run a live upgrade reproduction in this read-only review.

## Solution Assessment

No. The PR is a plausible guard for the empty string, but the best fix is to compare against or backfill the resolved configured/default embedding model rather than treating a configured provider as `fts-only`.
