## Summary

The PR fixes the malformed empty expected-model text, but substitutes the explicit providerless `fts-only` identity when a configured adapter is unavailable; current memory lifecycle code treats that as a distinct degraded mode, so the patch can report the wrong index identity and does not solve the configured-provider recovery path safely.

## What This Changes

The PR maps empty or whitespace-only memory embedding model identities to `fts-only` in identity comparison and pre-initialization fallback resolution, and adds three regression tests.

## Best Possible Solution

Carry the canonical configured or adapter-resolved embedding model into identity checks, or represent the unresolved configured-provider state explicitly as pending; reserve `fts-only` for an intentional provider-none or approved degraded lifecycle and test the transition before and after adapter activation.

## Reproduction Assessment

Yes. Current main can construct an empty pre-initialization model for an unregistered configured adapter, and the identity consumer compares that value directly; source inspection reproduces the empty expected-model condition with high confidence.

## Solution Assessment

No. Eliminating the empty string is necessary, but `fts-only` is an explicit providerless/degraded lifecycle with special safeguards, so using it as a generic unresolved-model sentinel does not preserve the configured-provider contract.
