## Summary

Keep open: the `displayPath` direction is useful, but the patch currently rewrites every digit-only validation path segment as a row label, which can misrepresent numeric string record keys, and the contributor proof is still test-only.

## What This Changes

The branch adds an optional `displayPath` to config validation issues and prefers it in human-facing CLI, Gateway, and Control UI formatting while keeping raw `path` for machine consumers.

## Best Possible Solution

Keep raw `path` canonical, compute `displayPath` only for schema-confirmed array positions, preserve numeric record keys, refresh the branch, and attach redacted real invalid-config proof.

## Reproduction Assessment

Yes. Current main joins Zod path segments with `.`, the Control UI labels array rows with `#${idx + 1}`, and the issue panel prints raw issue paths, so the `.3` versus `#4` mismatch is source-reproducible.

## Solution Assessment

No. Keeping raw `path` plus a human `displayPath` is the right direction, but text-only numeric segment rewriting is not the best fix because config paths can contain numeric-looking record keys.
