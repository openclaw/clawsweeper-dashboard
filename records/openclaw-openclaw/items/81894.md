## Summary

Keep open: current main still has a source-reproducible agent `update.run` path that can run the package update inside the live Gateway, and the linked fixing PR is still open rather than merged.

## What This Changes

The issue reports that agent-triggered OpenClaw self-updates on macOS can leave LaunchAgent-backed gateways offline or fail before completing the package swap to v2026.5.12.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/81945 that moves package-managed agent `update.run` into a detached/supervised CLI handoff, preserves service identity, verifies LaunchAgent health, and reports durable completion through `update.status`/restart sentinel state.

## Reproduction Assessment

Yes at source level: current main lets an agent call Gateway `update.run`, and the handler can call `runGatewayUpdate` inside the live Gateway before scheduling restart. I did not run the destructive macOS self-update scenario, but the reporter's three-host evidence matches that path.

## Solution Assessment

Yes in direction, but not yet complete on main: the linked PR is the right owner-boundary fix path, while the current issue should stay open until that PR is repaired, proven on a managed macOS install, and merged.
