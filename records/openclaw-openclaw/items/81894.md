## Summary

Keep open: current main still exposes the agent/control-plane `update.run` path that performs the package update in the live gateway process and only then schedules a restart, while the safer managed-service stop/restart and LaunchAgent recovery checks live on the shell `openclaw update` path.

## What This Changes

Investigate macOS LaunchAgent self-update failures when `update.run` is invoked from inside an OpenClaw-managed agent session.

## Best Possible Solution

Preserve the documented `update.run` self-update contract by moving package-manager updates into a detached or supervised handoff that reuses the CLI service-stop/restart and LaunchAgent recovery checks, then reports completion through `update.status` and the continuation sentinel.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live in this review; the source path is clear and matches the three-host report, but the destructive macOS LaunchAgent package-update scenario still needs a real managed-install run.

## Solution Assessment

Is this the best way to solve the issue? The shell-only safeguards are not enough because the documented agent `update.run` path remains separate; the narrow maintainable fix is a detached/supervised package-update handoff with LaunchAgent health verification, not removing the public action.
