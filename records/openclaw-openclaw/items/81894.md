## Summary

Keep open: the report has concrete three-host macOS evidence, and current main still exposes an agent/control-plane `update.run` path that runs the package update inside the live gateway before scheduling a restart. The shell `openclaw update` path now has useful managed-service safeguards, but those do not cover the documented agent `update.run` self-update flow that this issue implicates.

## What This Changes

Ensure agent/control-plane self-update on macOS safely hands off package updates, reloads the LaunchAgent, and reports durable completion, or refuses with a clear outside-shell command.

## Best Possible Solution

Preserve the documented `update.run` self-update contract by moving package-manager updates into a detached/supervised handoff that reuses the CLI service-stop/restart and LaunchAgent recovery checks, then reports completion through `update.status` and the continuation sentinel.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live, but the source path is clear: an agent can invoke gateway `update.run`, which runs the package update in the live gateway before scheduling restart, matching the reporter's macOS LaunchAgent failure class.

## Solution Assessment

Is this the best way to solve the issue? The current shell-only CLI protections are not sufficient because the documented agent `update.run` path remains separate; the safer fix is a detached/supervised package-update handoff with LaunchAgent health verification rather than removing the public action.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81894. Current agents are prompted to use gateway `update.run`, and `src/agents/tools/gateway-tool.ts` forwards that to `src/gateway/server-methods/update.ts`, where package-manager updates run inside the live gateway via `runGatewayUpdate` before scheduling restart. Make package-manager `update.run` for a managed local gateway use a detached/supervised handoff that reuses the safe CLI update lifecycle where possible: stop or hand off outside the live gateway before package replacement, refresh/restart the service from the updated install, recover an installed-but-not-loaded macOS LaunchAgent, verify expected gateway version/reachability, and keep `update.status`, `continuationMessage`, and restart sentinel behavior durable. Do not remove the public `update.run` contract, do not add a config flag, and do not weaken the existing shell `openclaw update` ancestry/service safeguards. Add focused tests around the gateway `update.run` package-manager path, agent gateway-tool metadata, and macOS LaunchAgent recovery; update docs/changelog if operator-visible behavior changes.
