## Summary

Keep this issue open: current main still has the reported ACP background-task completion behavior, and open PR #74103 explicitly targets this issue with closing syntax. The issue should remain open until that PR, or an equivalent reviewed fix, lands.

## What This Changes

The issue asks for ACP background-task completions to use a useful sanitized progress summary as the terminal summary and to raise the ACP progress-summary cap from 240 characters to about 1200 characters.

## Best Possible Solution

Review and land PR #74103, or replace it with an equivalent narrow ACP/task patch that raises the bounded progress cap, derives successful `terminalSummary` only from cleaned user-safe progress, preserves blocked outcome handling, and adds regression coverage for both useful completion summaries and suppressed internal/control text.

## Reproduction Assessment

Yes. The current-main source path is deterministic: non-thought ACP output fills `taskProgressSummary`, ordinary non-blocked success returns `{}`, terminal finalization stores a null summary, and delivery formats only `terminalSummary`.

## Solution Assessment

Yes. The narrow maintainable fix is in the ACP/task completion boundary: raise the progress cap and use sanitized successful progress as the terminal fallback while preserving the existing blocked/internal-progress protections.
