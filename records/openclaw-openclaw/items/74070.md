## Summary

Keep open: current main and the latest release still leave ordinary successful ACP background-task progress without `terminalSummary` and still cap progress at 240 characters, but the direct progress-buffer fallback has been rejected by a member as unsafe because it crosses the task-registry privacy boundary.

## What This Changes

The issue asks for ACP background-task completions to surface useful terminal summaries instead of empty success payloads and to use a longer progress summary cap.

## Best Possible Solution

Add an owner-approved terminal-only ACP completion result path and thread that into detached-task `terminalSummary`; keep live progress bounded and private unless maintainers explicitly approve a different exposure model.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce current-main behavior: ordinary successful ACP progress still resolves to no `terminalSummary`, and the progress cap remains 240 characters.

## Solution Assessment

No. The direct fallback proposed in the issue is not the best fix after the related maintainer rejection; the safer path is a terminal-only ACP result signal or adapter-owned final payload.
