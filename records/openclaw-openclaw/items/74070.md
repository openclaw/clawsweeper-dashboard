## Summary

Keep open: current main and the latest release still have the reported ACP background-task summary behavior, and open PR #74103 already targets this issue with closing syntax. The issue should remain open until that PR, or an equivalent reviewed fix, lands.

## What This Changes

The issue requests that ACP background-task completions populate `terminalSummary` from useful `progressSummary` output and raise `ACP_BACKGROUND_TASK_PROGRESS_MAX_LENGTH` from 240 to about 1200 characters.

## Best Possible Solution

Land PR #74103 after review, or replace it with an equivalent narrow ACP/task patch that raises the bounded progress cap, derives successful `terminalSummary` only from cleaned user-safe progress, preserves blocked outcome handling, and adds regression coverage for both useful completion summaries and suppressed internal/control text.

## Reproduction Assessment

Yes. The current-main source path is deterministic: non-thought ACP output fills `taskProgressSummary`, ordinary non-blocked success returns `{}`, terminal finalization stores a null summary, and delivery formats only `terminalSummary`.

## Solution Assessment

Yes, the fix boundary is the narrow maintainable one: use sanitized successful progress as the terminal fallback and raise the progress cap. The safer alternative to a blind fallback is to preserve the current blocked/internal-progress protections with focused tests.
