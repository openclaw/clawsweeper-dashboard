## Summary

Keep open: current `main` and the latest release still have the reported empty success `terminalSummary` and 240-character progress cap, but the direct progress-to-terminal fallback was explicitly rejected in related maintainer discussion as crossing the task-registry privacy boundary.

## What This Changes

The issue requests preserving useful ACP background-task completion text as `terminalSummary` and increasing the progress-summary cap beyond 240 characters.

## Best Possible Solution

Add an owner-approved terminal-only ACP completion result path and thread that into detached-task `terminalSummary`; keep live progress bounded and private unless maintainers explicitly approve a different exposure model.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the current-main behavior: ordinary successful ACP progress still resolves to no `terminalSummary`, and the cap is still 240 characters. I did not run tests because this review is read-only and test commands can create artifacts.

## Solution Assessment

No, the direct suggested fallback is not the best fix after the maintainer close note on the related PR. The better path is a terminal-only result signal or adapter-owned final payload so OpenClaw does not expose live progress as a completion summary.
