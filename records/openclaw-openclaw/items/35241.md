## Summary

Keep open: current main still does not expose `--session-key` on `openclaw agent`, and the downstream integration comments show a real use case. The branch is not mergeable as-is because it is conflicting, lacks after-fix real behavior proof, and exposes inconsistent session-key validation between Gateway and embedded paths.

## What This Changes

This branch adds a `--session-key <key>` option to `openclaw agent` and forwards it into the Gateway agent CLI session resolver.

## Best Possible Solution

Land a rebased version that exposes the flag with shared session-key validation and CLI/docs coverage, leaving `--no-fallback` for a separate follow-up.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection gives a high-confidence baseline: current main does not expose `--session-key` on `openclaw agent`, while the resolver and Gateway protocol already accept explicit session keys.

## Solution Assessment

No, not as-is. The direction is narrow and consistent with existing lower-level contracts, but the mergeable solution should validate explicit keys consistently before Gateway dispatch, `--local`, and embedded fallback, then document and prove the new selector.
