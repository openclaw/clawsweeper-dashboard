## Summary

Keep this PR open because the requested flag is still not on current main and downstream comments describe a real integration need, but the branch needs validation work and real CLI proof before merge.

## What This Changes

The PR adds a `--session-key <key>` option to `openclaw agent` and forwards that value through the agent CLI session-resolution path.

## Best Possible Solution

Expose `--session-key` on the agent CLI with shared pre-dispatch validation so gateway and embedded execution reject malformed or mismatched explicit keys the same way.

## Reproduction Assessment

Source-level reproduction is clear, but I did not run a live CLI turn in this read-only review. After this PR, explicit session keys enter both gateway and embedded paths, while current source validates malformed `agent:` keys only on the gateway side.

## Solution Assessment

No, not as-is. Surfacing the existing protocol field is the right narrow direction, but the PR should add shared validation before either gateway dispatch or embedded fallback/local execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate session keys before local dispatch:** `src/commands/agent-via-gateway.ts:37`
  - body: This flag makes explicit session keys user-reachable in the embedded path via the shared `localOpts` spread, but only the gateway currently rejects malformed `agent:` keys. For example, a malformed key can be rejected by the gateway path while `--local` proceeds into session resolution with the raw key, leaving different behavior for the same CLI input. Add one shared validation step before dispatching to either path.
