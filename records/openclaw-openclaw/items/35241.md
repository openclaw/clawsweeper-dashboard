## Summary

Keep open. The requested CLI surface is still missing on current main and the downstream discussion gives a concrete integration need, but this PR is not merge-ready because it exposes user-provided session keys without shared pre-dispatch validation and does not include the required release/docs follow-through.

## What This Changes

The PR adds an `openclaw agent --session-key <key>` option and threads `sessionKey` through `AgentCliOpts` into the Gateway/local session-key resolution path.

## Best Possible Solution

Land this as a small public CLI feature only after adding shared session-key validation before both Gateway dispatch and embedded fallback, preserving arbitrary non-`agent:` namespaces such as `agentgram:<room_id>`, adding focused Gateway/local/fallback tests, and updating CLI docs plus the changelog. Keep the requested `--no-fallback` behavior as a separate follow-up.

## Reproduction Assessment

Yes. Static inspection is enough: current main lacks the CLI flag, and the proposed diff can route a malformed `agent:` key into Gateway validation and then into the catch-all embedded fallback path.

## Solution Assessment

No, not as currently proposed. The direction is the narrow maintainable fix, but the implementation should validate explicit session keys before dispatch/fallback and include the required docs, changelog, and regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate session keys before gateway fallback:** `src/commands/agent-via-gateway.ts:121`
  - body: Adding this option threads arbitrary CLI input into the Gateway request, but `agentCliCommand` falls back to embedded mode on Gateway errors and the embedded path does not reject malformed `agent:` keys. Validate the key before either dispatch path or prevent fallback for validation failures.
