## Summary

Keep open: the requested CLI surface is still absent from current main and downstream comments describe a real integration need, but this PR is not merge-ready because it exposes explicit session keys without shared embedded-path validation and lacks real behavior proof.

## What This Changes

The PR adds a `--session-key <key>` option to `openclaw agent` and passes it through the CLI gateway request/session resolver.

## Best Possible Solution

Expose the selector through CLI, Gateway, and embedded paths with one shared explicit-session-key validation rule, focused gateway/local/CLI parsing coverage, docs/help updates, and maintainer changelog handling at landing.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection shows current main lacks the CLI flag while the underlying session resolver and Gateway protocol already accept `sessionKey`.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the best fix validates malformed canonical keys before both Gateway and embedded execution and documents/tests the new selector.
