## Summary

Keep open: current main still does not expose `--session-key` on `openclaw agent`, and downstream comments show a real session-isolation use case, but this branch is not merge-ready because it lacks real behavior proof and exposes explicit keys without shared malformed-key validation.

## What This Changes

This PR adds a `--session-key <key>` selector to `openclaw agent` and threads it through `AgentCliOpts` into session-key resolution.

## Best Possible Solution

Rebase and land a version that exposes `--session-key` through the existing session-routing contract, validates explicit key shape and agent mismatch before gateway/local dispatch, and includes focused tests plus redacted CLI proof.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR would make raw explicit session keys user-reachable in `--local` and embedded fallback, while Gateway rejects malformed agent-prefixed keys and the embedded path only checks agent mismatch. I did not run a live CLI command in this read-only review.

## Solution Assessment

No, not as-is. The CLI flag fits existing lower-level contracts, but the patch should share Gateway-style explicit session-key validation across gateway and local execution before it is merge-ready.
