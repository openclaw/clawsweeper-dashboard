## Summary

Keep open: current main still does not expose the requested `openclaw agent --session-key` flag, and downstream comments describe real integration use cases. The PR direction fits existing lower-level support, but the patch still needs validation parity before Gateway dispatch/fallback plus release-note coverage before merge.

## What This Changes

The PR adds `--session-key <key>` to `openclaw agent` and threads `sessionKey` through CLI option handling into Gateway/local session-key resolution.

## Best Possible Solution

Land the flag as a small public CLI feature after adding shared session-key validation before both Gateway dispatch and embedded fallback, preserving arbitrary non-`agent:` namespaces, adding focused tests, and updating docs plus the changelog; keep `--no-fallback` separate.

## Reproduction Assessment

Yes. Static source inspection is enough: current main lacks the CLI flag, and the PR diff would make malformed explicit `agent:` keys user-reachable before the embedded fallback path that lacks the Gateway's shape validation.

## Solution Assessment

No, not as currently proposed. The feature direction is narrow and compatible with existing lower-level support, but the implementation should validate explicit session keys before either dispatch path and include docs, changelog, and regression coverage.
