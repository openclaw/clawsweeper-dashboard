## Summary

Keep open: current main still lacks the requested CLI flag, and the PR is useful, but it should not merge until explicit session-key validation is shared before Gateway dispatch/fallback and the public docs/changelog are updated.

## What This Changes

The PR adds `--session-key <key>` to `openclaw agent` and forwards that value through the CLI Gateway command path.

## Best Possible Solution

Land the flag as a small CLI feature after adding shared session-key validation before both Gateway and embedded execution, focused tests, docs, and a changelog entry; keep the requested `--no-fallback` companion out of this PR.

## Reproduction Assessment

Yes. Static source inspection is enough: current main lacks the CLI flag, while the PR would make explicit session keys user-reachable before the embedded fallback path has the Gateway's malformed-key validation.

## Solution Assessment

No, not as currently proposed. The feature direction is narrow and fits existing lower-level support, but the implementation should validate explicit keys before either dispatch path and include docs, changelog, and regression coverage.
