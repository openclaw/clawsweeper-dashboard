## Summary

Keep open: current main still has no `openclaw agent --session-key` CLI surface, and downstream comments show a real integration use case, but the branch should not merge as-is because it exposes raw keys to local/fallback execution without the Gateway's malformed-key validation and still lacks real behavior proof.

## What This Changes

The PR adds an `openclaw agent --session-key <key>` option and threads it through agent CLI session-key resolution.

## Best Possible Solution

Land a rebased version that exposes `--session-key` through the existing session-routing contract, validates explicit key shape and agent mismatch before gateway/local dispatch, and includes focused tests plus redacted real CLI proof.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR would make raw explicit keys user-reachable in `--local` and embedded fallback while Gateway rejects malformed agent-prefixed keys. I did not run a live CLI command in this read-only review.

## Solution Assessment

No, not as-is. The CLI flag fits existing lower-level contracts, but the patch should share Gateway-style session-key validation across gateway and local execution before it is merge-ready.
