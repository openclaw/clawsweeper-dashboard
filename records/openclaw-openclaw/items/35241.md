## Summary

Keep open: the feature is still useful and not implemented on current main, but the PR is conflicting, lacks required real behavior proof, and exposes a validation inconsistency between Gateway and embedded/local execution.

## What This Changes

The branch adds an `--session-key <key>` option to `openclaw agent`, threads it through `AgentCliOpts`, includes it in the target-selection guard, and passes it to `resolveSessionKeyForRequest()`.

## Best Possible Solution

Rebase the branch, add shared pre-dispatch validation for explicit session keys, update CLI help and docs for accepted key shapes, keep `--no-fallback` as separate follow-up scope, and provide redacted real CLI proof before merge.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection gives a high-confidence baseline: current main does not expose `--session-key` on `openclaw agent`, while the lower-level resolver and Gateway protocol already accept explicit session keys.

## Solution Assessment

No, not as-is. The direction is narrow and useful, but the mergeable solution should validate explicit keys consistently before Gateway dispatch, `--local`, and embedded fallback, then document and prove the new selector.
