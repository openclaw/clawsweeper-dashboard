## Summary

Keep open. Current main still does not expose `--session-key` on `openclaw agent`, and downstream discussion shows a real integration need, but this PR is not merge-ready because it lacks shared validation, real behavior proof, and a rebase against the current gateway CLI path.

## What This Changes

The PR adds an `openclaw agent --session-key <key>` option and threads `sessionKey` into agent CLI session resolution.

## Best Possible Solution

Rebase the PR onto current main, add shared validation for explicit session-key shape and agent mismatch before gateway/local dispatch, add focused gateway/local tests, and post redacted real CLI proof.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR would make explicit keys user-reachable through `--local` and embedded fallback while current Gateway code rejects malformed agent-prefixed keys. I did not run a live CLI command in this read-only review.

## Solution Assessment

No, not as-is. The flag fits the existing lower-level contract, but the maintainable path is to validate the explicit key once before either gateway or local execution and then cover both paths with focused tests.
