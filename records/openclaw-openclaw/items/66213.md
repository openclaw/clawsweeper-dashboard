## Summary

Keep open. Current main still lacks this post-publish macOS asset verifier and the root appcast CODEOWNERS rule, but the proposed polling loop still needs two narrow fixes before it can be a trustworthy release guard.

## What This Changes

The PR adds a stable-release GitHub Actions verifier for macOS assets and a root `appcast.xml` CODEOWNERS rule.

## Best Possible Solution

Land a release-owner-approved post-publish guard that verifies stable macOS assets and appcast consistency without masking API failures or missing the final polling window, while keeping broader updater reliability tracked by #48306.

## Reproduction Assessment

Yes. Current main lacks the verifier and appcast CODEOWNERS entry, and the public release API reproduces the motivating `v2026.4.12` zero-asset case; the two PR defects are reproducible by static inspection of the shell loop.

## Solution Assessment

No, not yet. The PR direction is narrow and maintainable, but the loop should fail clearly on real `gh api` errors and check the full intended polling window before timing out.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fail real API errors instead of counting them as zero assets:** `.github/workflows/verify-release-assets.yml:53-54`
  - body: The `|| echo "0"` fallback makes every `gh api` failure look like a release with no assets. A 403, rate limit, outage, or missing release will burn the full 45-minute loop and report the wrong incident cause instead of failing with the real API error.
  - confidence: 0.91
- **[P2] Recheck assets before declaring the timeout:** `.github/workflows/verify-release-assets.yml:60-62`
  - body: The loop sleeps after the 45th poll and then fails without another API call. If the private publish uploads assets during that final minute, the workflow reports a missing-asset failure even though the assets arrived within the intended waiting window.
  - confidence: 0.88
