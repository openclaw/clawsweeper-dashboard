## Summary

Keep open: this is a focused release guard for a real macOS release failure class, but the proposed workflow still has two correctness defects and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds a GitHub Actions workflow that polls stable macOS release assets after publication and adds a root `appcast.xml` CODEOWNERS rule.

## Best Possible Solution

Land or replace a release-owner-approved verifier that preserves real `gh api` failures, polls through the full intended window, and keeps broader updater atomicity work under https://github.com/openclaw/openclaw/issues/48306.

## Reproduction Assessment

Yes for the release-artifact failure class and PR defects: historical `v2026.4.12` and `v2026.3.13-1` still have zero assets, current main lacks the verifier, and the workflow defects are visible in source. I did not reproduce a current-latest updater outage because `v2026.5.7` has the expected assets.

## Solution Assessment

No, not yet. The direction is narrow, but the workflow should fail real API errors, avoid the final-window false negative, and include real dispatch proof before merge.
