## Summary

Keep open. The PR targets a real stable macOS release verification gap that current main still lacks, but the proposed polling loop can hide GitHub API failures and can false-fail assets that arrive during the final minute.

## What This Changes

The PR adds a release-published and manually dispatched GitHub Actions workflow to poll for macOS release assets, warns on missing appcast entries, and adds a CODEOWNERS rule for root `appcast.xml`.

## Best Possible Solution

Land a release-owner-approved post-publish guard that verifies the required stable macOS assets and appcast state without masking API failures or missing the final polling window; keep broader updater atomicity in #48306.

## Reproduction Assessment

Yes. Current main lacks the verifier and appcast CODEOWNERS entry, the public API still reproduces the `v2026.4.12` zero-asset case, and the PR defects are reproducible by static inspection of lines 53-62.

## Solution Assessment

No, not yet. The direction is narrow and maintainable, but the loop should preserve real `gh api` errors and observe the full wait window before merge.
