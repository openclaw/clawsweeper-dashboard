## Summary

Keep open: current main still only exposes the standard Xiaomi API-key onboarding path, while Xiaomi's official Token Plan docs now confirm the regional OpenAI-compatible and Anthropic-compatible endpoint families and say OpenClaw currently requires manual config for Token Plan.

## What This Changes

Add Xiaomi MiMo Token Plan onboarding choices for CN, Singapore, and Europe with OpenAI-compatible and Anthropic-compatible endpoint presets.

## Best Possible Solution

Add plugin-local Xiaomi Token Plan presets, tests, docs, and changelog while preserving the existing pay-as-you-go xiaomi-api-key path and avoiding any core special-casing.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection gives high confidence that current main lacks the requested Xiaomi Token Plan auth-choice IDs and endpoint presets.

## Solution Assessment

Yes, with one implementation detail to settle: plugin-local auth choices using the existing provider wizard/auth seam are the narrow maintainable path. The fix should stop if Anthropic Token Plan support requires a new transport seam or a broader provider-ID model decision.

## Repair Work Prompt

Implement Xiaomi MiMo Token Plan regional onboarding presets in the bundled Xiaomi provider plugin. Add explicit auth-choice IDs xiaomi-token-plan-cn, xiaomi-token-plan-sgp, xiaomi-token-plan-ams, xiaomi-token-plan-anthropic-cn, xiaomi-token-plan-anthropic-sgp, and xiaomi-token-plan-anthropic-ams. Use Xiaomi's official Token Plan endpoints: OpenAI-compatible /v1 endpoints for CN/SGP/AMS with api openai-completions, and /anthropic endpoints for CN/SGP/AMS with api anthropic-messages. Keep the existing xiaomi-api-key pay-as-you-go path unchanged. Prefer a plugin-local endpoint map and onboarding helper updates, following the Z.AI regional auth-choice pattern and existing provider wizard/auth metadata; do not add core provider special-casing or unrelated Xiaomi TTS changes. Update manifest providerAuthChoices/runtime auth registration, Xiaomi onboarding tests, any static auth-choice option expectations, docs/providers/xiaomi.md, and CHANGELOG.md because this is user-facing provider onboarding. Preserve existing manual baseUrl override behavior. Stop and ask for maintainer guidance if supporting simultaneous pay-as-you-go and Token Plan requires introducing new provider IDs beyond the requested auth-choice presets, or if the Anthropic-compatible endpoint needs a new transport/API seam.
