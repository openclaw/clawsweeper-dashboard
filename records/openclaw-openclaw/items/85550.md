## Summary

Keep this open: the report describes a serious billing/session-isolation symptom, and current source confirms default-on trajectory sidecars record compiled prompt context, but I did not find proof that those sidecars are actually loaded back into model context.

## What This Changes

The issue asks OpenClaw to stop prior trajectory sidecars from inflating context and provider input tokens for new `openclaw chat --session <key>` sessions.

## Best Possible Solution

Reproduce the Windows/Gemini new-session flow with request-payload or provider-token evidence, then either block any trajectory sidecar from entering model context or adjust trajectory defaults/limits/docs if the tokens come from normal prompt/tool context instead.

## Reproduction Assessment

No. The source shows default-on trajectory sidecars and compiled-context recording, but I did not establish a current-main path where `.trajectory.jsonl` files are loaded back into model context.

## Solution Assessment

Unclear. The safest fix depends on proving whether the sidecar is actually fed to the provider; current source leans toward a separate capture artifact, so the next step is request-payload/token proof before changing runtime behavior.
