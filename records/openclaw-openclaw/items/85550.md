- `P1`: The report describes a fresh, user-visible runaway input-token and billing problem during normal chat sessions.

## Summary

Keep this open: the reported runaway token and billing symptom is serious and recent, but current source inspection confirms only that trajectory sidecars record compiled prompt context, not that those sidecars are fed back into provider input on new sessions.

## What This Changes

The issue asks OpenClaw to stop prior `.trajectory.jsonl` sidecars from being loaded into new `--session` chat context and driving runaway provider input tokens.

## Best Possible Solution

Reproduce the Windows/Gemini new-session flow with request payload or provider-token evidence, then either block trajectory sidecars from model input or adjust trajectory capture defaults and limits if the sidecars are only diagnostic artifacts.

## Reproduction Assessment

No. The reporter provides plausible steps, but I did not run the Windows/Gemini flow and source inspection did not reveal a current-main path where `.trajectory.jsonl` is read into provider context.

## Solution Assessment

Unclear. If payload proof shows sidecar replay, the narrow fix is to block sidecars from context; if the tokens come from normal compiled context or trajectory defaults, maintainers need to choose the intended diagnostic/billing tradeoff.
