## Summary

Keep open. Current main still has in-process Discord reconnect and READY timeout handling, but it does not classify certificate-expiry/TLS trust-store failures for process-level recovery and it still lacks the requested post-READY operator identity log; the newer READY issue is related but does not cover the unique CA-cache recovery policy.

## What This Changes

The issue requests recovery from Discord certificate-expiry zombie states that in-process reconnects cannot escape, plus restoration of a post-READY Discord identity log for operators.

## Best Possible Solution

Split the work into a maintainer-approved Discord TLS/certificate recovery policy that can escape process-local TLS state, plus lifecycle-backed READY/RESUMED identity logging with focused monitor tests and operator docs.

## Reproduction Assessment

No for the full live CA-rotation scenario; this review did not have the affected long-running host and Discord credentials. Source inspection does show the missing READY log path and the absence of a Discord certificate-expiry recovery classifier on current main.

## Solution Assessment

No for current main. In-process reconnects, readiness polling, and status patches are not enough for a process-local TLS trust-store failure, and the observability fix should be a separate READY/RESUMED lifecycle log.
