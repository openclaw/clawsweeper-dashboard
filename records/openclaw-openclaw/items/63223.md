## Summary

Keep open. Current main has more Discord READY timeout and reconnect handling than the reported 2026.4.5 build, but certificate-expiry errors still do not trigger a recovery path that escapes process-local TLS state, and the requested operator-visible READY identity log is not emitted after lifecycle readiness.

## What This Changes

The issue reports Discord gateway certificate-expiry recovery failure and missing READY operator logging.

## Best Possible Solution

Add Discord-owned certificate-expiry escalation to a maintainer-approved channel or process respawn path, and add lifecycle-backed READY/RESUMED identity logging coordinated with the active READY regression thread.

## Reproduction Assessment

No, not for the full live CA-rotation scenario. Source inspection does show the reported `certificate has expired` signature is not a special recovery class and post-READY identity logging is absent on current main.

## Solution Assessment

No for current main. In-process reconnects, channel-local health restarts, and status patches help related hangs but do not escape stale process-local TLS state or provide the requested operator-visible READY log.
