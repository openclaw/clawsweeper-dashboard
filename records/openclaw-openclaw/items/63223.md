## Summary

Keep open. Current main still lacks a Discord TLS certificate-expiry/system-CA recovery path that can escape process-local state, and the Discord READY transition still updates internal status without emitting a post-READY operator log line. The issue discussion also contains concrete macOS and Debian reports with the same process-local certificate-failure signature.

## What This Changes

Request to make the long-running Discord Gateway recover from process-local TLS certificate-expiry failures and restore a post-READY Discord identity log line for watchdogs.

## Best Possible Solution

Define a maintainer-approved fresh-process recovery policy for repeated Discord TLS certificate-expiry failures, then add a lifecycle-backed READY/RESUMED identity log and focused tests/docs for both operational paths.

## Reproduction Assessment

Unclear for a high-confidence full reproduction of the CA-rotation zombie state in this read-only review. The missing post-READY operator log and absent certificate-expiry classification are directly verifiable in current-main source, and the issue discussion provides concrete cross-platform runtime evidence.

## Solution Assessment

No, current main is not the best way to solve the issue because it relies on in-process reconnect/channel restarts and status updates. A safer direction is to separate the low-risk READY log restoration from a maintainer-approved service/process recovery policy for TLS trust-store failures.
