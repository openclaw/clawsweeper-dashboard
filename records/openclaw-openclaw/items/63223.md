## Summary

Keep open. ## What This Changes

The issue reports a Discord gateway outage where certificate-expiry TLS failures survive in-process reconnects and the READY identity log no longer appears for operator health checks.

## Best Possible Solution

Add a Discord-owned TLS certificate-error recovery path that hands off to a maintainer-approved supervised full-process restart policy, and emit identity-rich READY/RESUMED logs when the gateway lifecycle actually reaches readiness.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection does show the reported failure class remains plausible: certificate-expiry text is not a special stopping event, reconnects remain in-process, and READY identity logging is startup-only.

## Solution Assessment

Unclear until maintainers choose the recovery policy. A hard process exit, supervised restart handoff, or operator hook can all address process-local TLS state, but that availability behavior needs an owner decision before an automated fix PR.
