## Summary

Keep open. Current main still lacks a certificate-expiry or system-CA recovery path that can escape process-local TLS state, and the Discord READY transition still updates status/presence without emitting the requested post-READY operator log line.

## What This Changes

The issue requests process-level recovery for Discord gateway TLS certificate-expiry failures after system CA rotation and restoration of a Discord READY identity log.

## Best Possible Solution

Keep this open and split the fix into two maintainer-approved parts: add an explicit Discord TLS certificate-expiry recovery policy that can trigger a fresh process when repeated in-process reconnects cannot recover, and add a lifecycle-backed READY/RESUMED identity log with focused Discord monitor tests and operator docs.

## Reproduction Assessment

Unclear for the full live CA-rotation scenario because this read-only pass did not run a long-lived Discord gateway through certificate churn. The missing READY log and absent certificate-expiry classifier are directly reproducible by inspecting current-main source and tests.

## Solution Assessment

No. Current main has in-process reconnects, channel health restarts, and status updates, but the reported CA-cache failure needs a process-level recovery decision and the READY observability gap needs a separate lifecycle log.
