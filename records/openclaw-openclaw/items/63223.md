## Summary

Keep open. Current main still lacks a Discord certificate-expiry/TLS recovery class that can escape process-local CA state, and the requested post-READY operator identity log is still not emitted after readiness is reached.

## What This Changes

Request recovery from Discord gateway TLS certificate-expiry zombie states and restore an operator-visible Discord READY log line.

## Best Possible Solution

Adopt a maintainer-approved Discord TLS failure escalation that can escape process-local CA state, and add READY/RESUMED identity logging with focused monitor tests and operator docs.

## Reproduction Assessment

No for the full live CA-rotation scenario; this pass did not have the affected long-running host and Discord credentials. Source inspection does show the missing Discord certificate-expiry recovery classifier and the missing post-READY operator log on current main.

## Solution Assessment

No for current main. In-process reconnects and readiness polling are not enough for process-local TLS trust-store failures, and the observability fix should be a lifecycle-backed READY/RESUMED log rather than only the pre-lifecycle startup line.
