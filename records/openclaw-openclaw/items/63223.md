## Summary

Keep open. Current main still handles Discord failures with in-process reconnects, READY timeouts, and status updates, but it does not classify certificate-expiry/TLS trust-store failures for process-level recovery, and the READY path still lacks the requested post-READY operator identity log.

## What This Changes

Request robust Discord Gateway recovery after long-lived TLS certificate expiration or system-CA churn and restore a post-READY log line for operator health checks.

## Best Possible Solution

Split the fix into a maintainer-approved Discord TLS recovery policy that can escape process-local TLS state, plus lifecycle-backed READY/RESUMED identity logging with focused monitor tests and operator docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the full live CA-rotation scenario in this read-only pass, but source inspection does reproduce the missing READY log path and the absence of a Discord certificate-expiry recovery classifier.

## Solution Assessment

Is this the best way to solve the issue? No for current main: in-process reconnects and READY status updates are not enough for a process-local TLS cache failure, and the observability fix should be a separate READY/RESUMED lifecycle log.
