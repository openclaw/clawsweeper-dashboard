## Label Justifications

- `P1`: The issue describes a real Discord channel outage where reconnect loops leave users unable to send or receive messages until an external restart.
- `impact:crash-loop`: The gateway/plugin enters repeated reconnect and terminal disposed states instead of recovering cleanly.
- `impact:message-loss`: Discord channel interactions are reported dead while the process remains superficially running.

## Summary

Keep open. Current main has improved Discord startup readiness handling, but it still does not special-case certificate-expiry/TLS trust-store failures into a recovery path that can leave the current process, and the READY identity log remains incomplete for the late-ready path.

## What This Changes

The issue reports Discord gateway TLS certificate-expiry zombie behavior plus missing READY identity logging after startup.

## Best Possible Solution

Add a Discord-owned certificate-expiry escalation path that can escape process-local TLS state, and coordinate lifecycle-backed READY/RESUMED identity logging with the open Discord READY regression work.

## Reproduction Assessment

No high-confidence live reproduction of system CA rotation was run in this read-only review. Source inspection does show that certificate-expiry is not a special recovery class and the current reconnect/health paths remain in-process.

## Solution Assessment

Unclear until maintainers choose the recovery policy. In-process reconnects and channel-local restarts are useful for related hangs, but they are not a complete solution for a process-local TLS trust-store failure.
