## Label Justifications

- `P1`: The issue reports a high-impact Discord gateway outage that leaves channel traffic dead until an external restart.
- `impact:crash-loop`: The reported gateway repeatedly reconnects, exhausts attempts, and enters a terminal disposed/zombie state.
- `impact:message-loss`: Discord channel interactions are reported dead while the gateway process appears to remain running.

## Summary

Keep open. Current main improved Discord READY timeout/reconnect handling, but certificate-expiry errors are still not treated as a recovery class that can escape process-local TLS state, and the READY identity log is still only emitted during the initial startup status check rather than when lifecycle later reaches READY.

## What This Changes

The issue asks OpenClaw to recover from Discord gateway TLS certificate-expiry zombie states with a full-process escape path and to restore an operator-visible Discord READY log line.

## Best Possible Solution

Add a Discord-owned certificate-expiry escalation path that can hand off to the existing supervised full-process restart mechanism, and emit an identity-rich READY/RESUMED log when lifecycle actually reaches gateway readiness.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the reported failure class is still plausible: certificate-expiry text is not a special stopping event, reconnects remain in-process, and READY logging is startup-only.

## Solution Assessment

Unclear until maintainers choose the recovery policy. A hard process exit, supervised restart handoff, or operator hook can all address process-local TLS state, but that availability behavior needs an owner decision before an automated fix PR.
