## Summary

Keep open. Current main and the latest release still lack a Discord certificate-expiry recovery path that can escape process-local TLS state, and the READY identity log is still only emitted during startup before the lifecycle wait rather than from the READY listener itself.

## What This Changes

The issue requests recovery from Discord TLS certificate-expiry zombie states and restoration of an operator-visible Discord READY identity log.

## Best Possible Solution

Add owner-approved Discord TLS error escalation that can recover outside stale process TLS state, and restore lifecycle-backed READY/RESUMED identity logging with focused tests and live proof.

## Reproduction Assessment

No, not for the full live CA-rotation scenario. Source inspection does show the current-main failure path: `certificate has expired` is not a recovery class and the READY listener does not emit the requested identity log.

## Solution Assessment

No for current main. In-process reconnects and channel health restarts help related hangs but do not escape process-local TLS state; the maintainable fix needs a maintainer-chosen respawn/escalation policy plus a narrow READY/RESUMED log path.
