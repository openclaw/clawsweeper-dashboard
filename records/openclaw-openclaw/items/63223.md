## Summary

Keep open. Current main has Discord READY timeout and reconnect improvements, but it still lacks a certificate-expiry recovery path that can escape process-local TLS state, and it still does not emit the requested post-READY operator identity log after readiness is reached.

## What This Changes

The issue requests Discord Gateway recovery from process-local certificate-expiry/TLS failures plus a READY/RESUMED identity log usable by operator watchdogs.

## Best Possible Solution

Add owner-approved Discord TLS error escalation that can trigger recovery outside the stale TLS context, and emit lifecycle-backed READY/RESUMED identity logging with focused tests and docs.

## Reproduction Assessment

No, not for the full live CA-rotation scenario. The current-main source path does show that the reported `certificate has expired` signature is not a recovery class and that the READY listener does not emit the requested operator identity log after readiness.

## Solution Assessment

No for current main. In-process reconnects and READY timeouts help related hangs but do not escape process-local TLS state; the maintainable fix needs a maintainer-chosen respawn/escalation policy plus a narrow READY/RESUMED log path.
