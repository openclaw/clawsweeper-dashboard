## Summary

Keep this PR open: the inbound-peer send-policy direction is useful and well-covered for the core Telegram behavior, but the shared delivery default now policy-checks existing explicit agent delivery callers that do not pass `sendPolicyMode`. That is a merge-blocking compatibility gap before maintainer cleanup can land it.

## What This Changes

The PR adds `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"`, composition helpers, inbound-peer context propagation for automatic/durable replies, durable queue replay mode persistence, docs, and focused tests.

## Best Possible Solution

Land the native inbound-peer predicate after explicit operator/API delivery paths are either marked `sendPolicyMode: "explicit"` or the shared default preserves old behavior while automatic reply callers opt into policy enforcement.

## Reproduction Assessment

Yes from source inspection: PR head defaults missing `sendPolicyMode` to `automatic`, while the documented `openclaw agent --deliver` path calls `sendDurableMessageBatch` without that flag. A deny-by-default session policy would therefore suppress a delivery current main sends.

## Solution Assessment

No: the native predicate is the right core shape, but defaulting every unannotated durable send to automatic is not the safest compatibility boundary. The better fix is to explicitly classify operator/API delivery callers or preserve the old default and opt automatic reply paths into policy.
