## Summary

Keep this PR open, but not merge-ready: the native predicate is a useful direction, yet the current branch leaves explicit send paths without inbound-peer context and widens shared config schemas beyond the evaluators that understand the new fields.

## What This Changes

The branch adds `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"`, recursive match composition, blocked-send suppression metadata, reply-routing context propagation, tests, docs, and a Vitest CI heap default.

## Best Possible Solution

Land a native relational send-policy predicate only after the authoritative inbound peer is carried through every outbound send surface meant to be protected, and keep the public schema/docs/help metadata aligned with the exact supported contract.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: PR head durable delivery reads `params.session?.inboundPeer`, while the gateway `message.send` path still builds session context without that field. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. A native predicate is a plausible maintainable solution, but this implementation must either cover all protected send surfaces or explicitly narrow the public contract before merge.
