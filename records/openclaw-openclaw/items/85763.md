## Summary

Keep open: the native inbound-peer predicate is useful and has strong live proof, but the current patch still policy-enforces unannotated durable sends and leaves documented `openclaw agent --deliver` delivery unclassified, which can suppress existing operator-requested sends under deny-by-default `session.sendPolicy`.

## What This Changes

Adds `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"`, recursive match composition, inbound-peer propagation for automatic/durable replies, queued `sendPolicyMode` replay semantics, docs, and tests.

## Best Possible Solution

Land the inbound-peer predicate after operator/API delivery callers are explicitly classified while automatic reply and automatic queue replay paths remain policy-enforced.

## Reproduction Assessment

Yes, from source inspection: PR head defaults missing `sendPolicyMode` to `automatic`, while documented `openclaw agent --deliver` still calls `sendDurableMessageBatch` without that flag. A deny-by-default policy would suppress a delivery current main sends.

## Solution Assessment

No. The native predicate is the right core shape, but defaulting every unannotated durable send to automatic is not the safest compatibility boundary; explicit operator/API callers should be classified before merge.
