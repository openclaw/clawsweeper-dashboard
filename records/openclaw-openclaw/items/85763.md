- `P1`: The PR changes message delivery and security-boundary behavior for an active cross-peer privacy guard, with upgrade-sensitive effects for existing sendPolicy users.

## Summary

Keep open: the PR is a useful implementation of the inbound-peer predicate, but it is not merge-ready because it changes explicit-send enforcement for existing configs, widens a reused config schema beyond the send-policy surface, and the real-behavior proof is stale relative to the latest head.

## What This Changes

Adds `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"`, propagates inbound peer context through reply and explicit send paths, suppresses denied sends, and updates docs, tests, and config metadata.

## Best Possible Solution

Land only after maintainers choose the explicit-send enforcement contract or an opt-in compatibility path, keep relational predicates scoped to `session.sendPolicy`, and refresh proof against the latest head.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main leaves explicit message-tool sends available under sendPolicy suppression, while the PR adds explicit-send suppression paths and widens a reused schema export. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The core predicate is useful, but the safest merge path needs an explicit compatibility decision plus a schema split so non-sendPolicy config surfaces do not accept unsupported relational predicates.
