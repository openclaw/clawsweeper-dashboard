## Summary

This PR is useful but not merge-ready: the new delivery-layer guard depends on `session.inboundPeer`, and the main durable inbound-reply bridge still builds its outbound session without that field, so a wrong-peer final reply through that path can bypass the new relational predicate. Live Telegram proof is also still absent for a Telegram-visible delivery guard.

## What This Changes

The branch adds `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"`, propagates inbound-peer context through several reply/tool/outbound paths, preserves explicit-send bypass behavior, and updates docs plus focused tests.

## Best Possible Solution

Land the new predicate only after the durable inbound-reply bridge carries trusted inbound peer candidates into `buildOutboundSessionContext`, regression coverage proves that path denies wrong-peer automatic final replies, and a maintainer or contributor posts redacted live Telegram proof.

## Reproduction Assessment

Yes for the review finding: source inspection on the PR head shows the new policy check reads `params.session?.inboundPeer`, while `deliverInboundReplyWithMessageSendContext` builds the session without that field. I did not run tests because this was a read-only review.

## Solution Assessment

No, not yet. The maintainable fix is to carry inbound peer candidates into every automatic durable-send boundary, especially `src/channels/turn/durable-delivery.ts`, and prove both same-peer allow and wrong-peer denial.
