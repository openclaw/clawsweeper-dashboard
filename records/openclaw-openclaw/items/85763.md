## Summary

Keep open: the branch adds a plausible native send-policy predicate, but as-is it leaves explicit message tool sends without inbound-peer context and only provides test-only proof for a security-sensitive routing guard.

## What This Changes

The PR adds `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"`, recursive match composition, blocked-send suppression metadata, reply-routing context propagation, tests, and docs.

## Best Possible Solution

Ship the native predicate only after the authoritative turn peer is carried through every outbound path covered by the public contract, the config metadata/docs are aligned, and a redacted real-transport proof demonstrates both allowed and suppressed delivery.

## Reproduction Assessment

Yes for the main review finding by source inspection: the PR head checks `params.session?.inboundPeer` in durable delivery, while the direct `sendMessage` path still builds its outbound session without that field. I did not run tests because this review is read-only.

## Solution Assessment

No as-is: a native relational predicate is a maintainable direction, but the patch must either cover all claimed outbound paths or explicitly narrow the public contract before merge.
