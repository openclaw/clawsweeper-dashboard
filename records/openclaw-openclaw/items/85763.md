## Summary

Keep open: the feature is useful, but the current patch still leaves the advertised peer guard bypassable on explicit sends, changes existing sendPolicy behavior for explicit message delivery without an upgrade decision, and broadens a shared config schema beyond what all consumers implement.

## What This Changes

Adds `peerEquals: "inboundPeer"` send-policy matching, recursive match composition, inbound-peer propagation for routed replies, delivery suppression metadata, docs, and focused unit coverage.

## Best Possible Solution

Land a send-policy-specific contract that carries the authoritative inbound peer through every protected outbound surface, preserves or deliberately migrates explicit-send behavior, and keeps shared config schemas aligned with each consumer.

## Reproduction Assessment

Yes for the review findings by source inspection: PR head central delivery consumes only `params.session?.inboundPeer`, while explicit message sends build session context without that field. I did not run tests because this review is read-only.

## Solution Assessment

No. The right direction is plausible, but this patch should first settle explicit-send scope, thread the inbound peer through every protected send surface, and split send-policy-only schema keys away from shared allow/deny consumers.
