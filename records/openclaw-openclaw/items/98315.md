## Summary

Keep open: current main still leaves this as a security-boundary/product decision, because per-agent skill filtering is applied to OpenClaw skill discovery and prompt/runtime paths, not to arbitrary external processes launched through an allowed shell command.

## What This Changes

The issue reports that an agent with allowed shell access can invoke the external `mcporter` CLI with a global config path and reach MCP servers outside its per-agent skills/MCP allowlist.

## Best Possible Solution

Maintainers should decide whether per-agent skills/MCP visibility is a hard shell-time authorization boundary; if yes, implement engine/sandbox/credential enforcement with focused regression proof, otherwise document the required isolation model clearly.

## Reproduction Assessment

No live high-confidence reproduction was run. The report is source-reproducible enough to keep open: current code applies per-agent skills to OpenClaw skill discovery/runtime prompt surfaces, while an allowed shell command can launch an external `mcporter` runtime outside that filter path.

## Solution Assessment

Unclear. Rejecting or wrapping `mcporter --config` may be a mitigation, but the best fix depends on a maintainer security decision about whether this is an OpenClaw-enforced shell-time authorization boundary or a deployment isolation requirement.
