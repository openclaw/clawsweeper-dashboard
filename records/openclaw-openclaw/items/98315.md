## Summary

Keep open: current main does not implement an engine-level guard that would make per-agent skills/MCP visibility authoritative against an allowed shell command, and the report is security-sensitive enough for maintainer review rather than cleanup closure.

## What This Changes

The issue reports that an agent with legitimate shell access can bypass per-agent `mcporter.json` and `skills` allowlists by invoking `mcporter call --config` against a globally readable mcporter registry.

## Best Possible Solution

Maintainers should define whether per-agent skills/MCP configuration is intended to constrain allowed shell-time MCP clients; if yes, add an engine-owned enforcement design with tests, otherwise document the required sandbox, OS-user, or per-agent credential isolation model.

## Reproduction Assessment

Unclear for a live reproduction; the reporter provides concrete steps, and source review shows skill filtering does not mediate arbitrary external `exec` commands, but I did not run the CentOS/mcporter setup.

## Solution Assessment

Unclear; rejecting external `mcporter --config` paths may be a mitigation, but the best fix depends on a maintainer security decision about whether this is a hard OpenClaw authorization boundary or a deployment-isolation requirement.
