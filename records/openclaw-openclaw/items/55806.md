## Summary

Keep open. Current main has partial tool execution diagnostics, sanitized agent-event args, duration/errorCategory spans, and apply_patch summaries, but it does not provide the requested end-to-end core behavior for INFO/default logs, structured toolResult error fields, or general file/memory writeback tracking; the broader open observability thread is complementary, not a safe duplicate.

## What This Changes

Request adds production tool-call observability through sanitized input summaries, structured tool error categories, and context/writeback tracking for tools that mutate files or memory.

## Best Possible Solution

Define a bounded core diagnostic/writeback schema first, keep raw tool content opt-in, preserve plugin hook observability for fleet-specific logging, and split implementation into small surfaces that align JSONL, OTEL/Prometheus, docs, and tests.

## Reproduction Assessment

No, not as a broken-behavior repro; this is a feature/design request. Source inspection does give a high-confidence current-state check that main only partially covers the requested observability and lacks general context writeback tracking.

## Solution Assessment

Unclear: the direction is valid, but the proposed raw INFO-log/toolResult-field shape should not be implemented before maintainers settle schema, redaction-before-truncation, cardinality, and plugin-versus-core boundaries. A bounded diagnostics schema with exporter mappings is the safer first step.
