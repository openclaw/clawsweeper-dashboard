## Summary

Keep open. Current main has useful partial observability through low-cardinality diagnostic tool execution events, redacted Control UI agent events, OTel tool spans, trajectory export, and apply_patch summaries, but it still does not implement the full requested audit surface: INFO/file-log sanitized argument summaries, durable toolResult errorType/errorRecoverable metadata, or generic context/file/memory writeback tracking. The related implementation PR #55846 is closed and unmerged, so there is no merged canonical fix to close against.

## What This Changes

Add privacy-preserving tool-call observability with sanitized input summaries, structured tool error metadata, and generic context/file/memory writeback audit tracking.

## Best Possible Solution

Define one privacy-preserving tool audit contract that reuses or strengthens the existing redaction path, then extend diagnostics/agent events/OTel and any durable transcript metadata additively. Generic writeback events should describe mutations with bounded metadata rather than contents, with tests proving secrets, message bodies, and file data are not exposed.

## Reproduction Assessment

Not applicable as a feature/audit gap rather than a runtime crash. The high-confidence verification path is static plus behavioral: inspect the tool execution, diagnostic, transcript, and patch summary paths, then run a tool call and confirm the requested sanitized INFO log, durable errorType/errorRecoverable fields, and generic writeback records are absent.

## Solution Assessment

Unclear. The issue’s direction is valid, but a standalone INFO log plus new toolResult fields is not necessarily the safest long-term shape; the safer solution is an additive, centrally redacted observability contract spanning the existing diagnostic and agent-event surfaces.
