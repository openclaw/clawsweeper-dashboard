## Summary

Keep open. Current main has partial tool-call observability through diagnostic events, OTel attributes, plugin hooks, and redacted UI events, but the requested durable sanitized INFO/file log, toolResult errorType/errorRecoverable contract, and generic writeback audit trail are still not implemented and need maintainer product/security design.

## What This Changes

Request durable tool-call observability for sanitized input summaries, structured error categories/recoverability, and context or file writeback tracking.

## Best Possible Solution

Define a privacy-preserving observability contract across diagnostics, durable logs/transcripts, plugin events, OTel, and mutating-tool audit events, then add focused regression coverage for redacted args, categorized errors, and bounded writeback metadata.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/audit gap. Static inspection of current main gives a high-confidence verification path showing the durable argument log, toolResult error metadata, and generic writeback audit fields are absent.

## Solution Assessment

Unclear. The requested direction is valid, but standalone INFO logs and transcript fields are not necessarily the safest solution; maintainers should first approve a shared redaction and observability contract.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because new durable observability fields could persist sensitive tool inputs, outputs, messages, or write contents.

  - body: sanitizeToolArgs currently protects the Control UI event path, but durable INFO/file logs or transcript fields would be new exposure points. Any implementation should reuse or strengthen this redaction boundary before persisting parameters.
  - confidence: 0.86
