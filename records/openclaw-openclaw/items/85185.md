## What This Changes

The PR lets Google’s lightweight provider-policy artifact expose Gemini 3 thinking levels and lets opted-in provider profiles override stale `reasoning:false` catalog metadata, with focused tests for the shared thinking resolver and Google policy artifact.

## Best Possible Solution

Land the provider-profile fix after maintainers accept the additive provider API flag, or move the stale-catalog override behind an internal bundled-policy contract if that should not be public SDK surface.

## Reproduction Assessment

Yes. Source inspection shows current main returns an off-only profile before provider profile lookup when catalog metadata says `reasoning:false`, and cron then downgrades unsupported thinking with the exact warning path; I did not run a live systemd cron job in this read-only review.

## Solution Assessment

Yes on the fix boundary: routing the decision through provider-owned thinking policy and the lightweight Google provider-policy surface is narrower than a cron-only special case. The remaining solution-fit question is whether the new stale-catalog opt-in should be public plugin API or internal bundled-provider metadata.
