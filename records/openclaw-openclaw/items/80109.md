## Summary

Keep open. Current main still routes arbitrary unknown CLI roots through the plugin allowlist diagnostic when `plugins.allow` is restrictive, and a current regression test explicitly preserves that behavior for `totally-unknown`; this matches the reported bug rather than an implemented fix.

## What This Changes

Route truly unknown CLI primary tokens to normal unknown-command diagnostics instead of suggesting `plugins.allow` changes for arbitrary strings.

## Best Possible Solution

Add a small CLI primary-token classification step so only known plugin-owned roots get allowlist remediation, while unmatched roots fall through to the existing unknown-command formatter.

## Reproduction Assessment

Yes. Source inspection shows the exact path: `resolveMissingPluginCommandMessage('totally-unknown', { plugins: { allow: [...] } })` is currently expected to return the misleading allowlist message, and `runCli` can throw it before Commander handles the unknown command.

## Solution Assessment

Yes. The narrow maintainable fix is the classification layer described in the issue: resolve real plugin roots, manifest aliases, runtime slash aliases, and tools first, then let truly unknown primaries use normal CLI unknown-command handling instead of expanding the reserved list one token at a time.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80109 as a narrow CLI diagnostics bug. In `src/cli/run-main-policy.ts` and the immediate CLI routing path, stop using the generic missing-plugin allowlist fallback for truly unknown primary tokens. Preserve existing behavior for real bundled/plugin CLI roots, manifest `commandAliases`, runtime slash aliases, and manifest-declared tools, but when no built-in command, plugin CLI metadata, alias, or tool owner matches, let the existing Commander/unknown-command diagnostic path handle the token. Do not make `plugins.allow` accept arbitrary command names, do not grow the reserved-root list as the primary solution, and do not add new config options. Add focused regression coverage for an arbitrary token such as `nonexistent-fake-command` or `approval` with restrictive `plugins.allow`, plus coverage that real plugin-owned roots and aliases still get the correct parent-plugin guidance. Add a user-facing changelog fix entry if the branch is shipped.
