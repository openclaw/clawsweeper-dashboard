## Summary

This is a valid current-main CLI diagnostic bug, but the same remaining arbitrary-unknown-command allowlist fallback is already tracked by open repair PRs, especially https://github.com/openclaw/openclaw/pull/64112 with overlapping maintainer-labelled work in https://github.com/openclaw/openclaw/pull/63336.

## What This Changes

The issue asks for arbitrary unknown CLI root tokens to use normal unknown-command diagnostics instead of suggesting that users add those tokens to `plugins.allow`.

## Best Possible Solution

Land one current-main fix that emits plugin allowlist diagnostics only for real plugin CLI roots or manifest-owned command aliases, while arbitrary unknown roots fall through to normal unknown-command handling.

## Reproduction Assessment

Yes. The source-level reproduction is the existing `resolveMissingPluginCommandMessage("totally-unknown", { plugins: { allow: ["quietchat"] } }, ...)` policy test, and `runCli` still calls the same helper for unregistered primary commands.

## Solution Assessment

Yes. The narrow maintainable solution is the one already tracked by the open PRs: classify whether a token is a real plugin CLI root or manifest-owned alias before emitting any `plugins.allow` remediation.
