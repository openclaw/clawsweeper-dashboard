## Summary

Keep open. Current `main` still sends arbitrary unowned CLI roots through the `plugins.allow` fallback, and the existing test suite explicitly preserves that behavior for `totally-unknown`; the prior `tool` fix only covered a reserved token, not the generic unknown-command path.

## What This Changes

The issue asks for truly unknown CLI primary tokens to use normal unknown-command diagnostics instead of suggesting that arbitrary strings be added to `plugins.allow`.

## Best Possible Solution

Add a small CLI primary-token classification step so only known plugin-owned roots get allowlist remediation, manifest aliases resolve to their owning plugin, agent tools get tool-use guidance, and unowned roots fall through to the existing unknown-command formatter.

## Reproduction Assessment

Yes, by source inspection: `resolveMissingPluginCommandMessage("totally-unknown", { plugins: { allow: ["quietchat"] } })` is currently expected to return the misleading allowlist message, and `runCli` can throw it before Commander parses the unknown command. I did not run a packaged CLI repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is the classification layer described in the issue, not continuing to grow the reserved-token set one command at a time.
