## Summary

Keep this PR open, but it is not merge-ready: the numeric-input direction is reasonable, yet the branch removes request-boundary coercion while the stored default is still string-backed, uses `0` as a blank sentinel, is now conflicting against current main, and lacks after-fix real behavior proof.

## What This Changes

The PR changes the Control UI Sessions Limit filter from a text field to a numeric input and changes selected Sessions limit state/test types.

## Best Possible Solution

Finish the narrow UI fix with one consistent state model, preserve editable blank state, only send positive integer limits to `sessions.list`, rebase on current main, and add after-fix browser proof.

## Reproduction Assessment

Yes. Source inspection and the attached screenshot show current main allows arbitrary text in the Sessions Limit field, while the Gateway `sessions.list` schema requires a positive integer limit.

## Solution Assessment

No. A numeric input is a reasonable direction, but this patch drops request-boundary coercion before the stored state/defaults are safely migrated and uses `0` for blank/invalid input.

- **[P1] Coerce stored limit before sessions.list:** `ui/src/ui/controllers/sessions.ts:159`

Preserve an editable blank/undefined state, or keep string state and only coerce positive integers at the request boundary.
  - confidence: 0.86
