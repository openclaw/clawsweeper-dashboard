## Summary

Keep open: current main still has the text-input Sessions Limit bug, but this PR is not mergeable as-is because it forwards possibly string-backed default state directly to `sessions.list`, stores blank/invalid input as `0`, and lacks after-fix real behavior proof.

## What This Changes

The branch changes the Control UI Sessions Limit filter from a text-backed string field to a number input, changes selected Sessions UI/controller state types to number, updates focused tests, and includes CSS formatting churn.

## Best Possible Solution

Finish the narrow UI fix with one consistent state model: either keep editable strings and coerce at the request boundary, or migrate all defaults/app/view/controller limit state to `number | undefined` while only sending positive integers.

## Reproduction Assessment

Yes, by source inspection. On current main, the Sessions Limit control is a plain text input that forwards raw text into filter state, while `sessions.list` only accepts a positive integer limit at the Gateway boundary.

## Solution Assessment

No. A numeric input is a reasonable direction, but this implementation removes request-boundary coercion before all state/default paths are numeric and uses `0` as a blank sentinel instead of preserving omitted/blank state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Coerce stored limit before sessions.list:** `ui/src/ui/controllers/sessions.ts:159`
  - body: `loadSessions` now forwards `state.sessionsFilterLimit` directly when no override is passed, but the app still initializes that property from the string default `DEFAULT_SESSIONS_FILTERS.limit`, and opening/refreshing Sessions calls `loadSessions(app)` without overrides. That can send `limit: "200"` to `sessions.list`, violating the schema-required integer until the user edits the field.
  - confidence: 0.92
- **[P2] Preserve blank limits without a zero sentinel:** `ui/src/ui/views/sessions.ts:293-294`
  - body: The input handler maps empty, invalid, or non-positive values to `0`, so clearing the field stores an invalid sentinel and turns the next request into an omitted/unbounded limit path. Keep the editable blank state as blank/undefined, or preserve string state and only send positive integers at the request boundary.
  - confidence: 0.84
