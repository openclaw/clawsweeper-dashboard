## Summary

Keep this PR open, but it needs changes before merge. Current main still has the reported text-backed Sessions Limit field, and the PR moves in the right direction, but its latest diff leaves the app default as a string while removing controller-side coercion, stores blank/invalid input as a zero sentinel, omits the required changelog entry, and includes unrelated formatting churn.

## What This Changes

The PR changes the Control UI Sessions Limit field toward a numeric input, migrates the Sessions view/controller limit types to numbers, updates two Sessions tests, and includes unrelated CSS/app-render formatting changes.

## Best Possible Solution

Finish a focused Sessions Limit fix with one consistent model: either keep editable string filter state and parse only at the controller request boundary, or migrate the full app/view/controller chain to `number | undefined`. The final patch should render a numeric input with `min="1"` and `step="1"`, preserve blank editing without a `0` sentinel, only send positive integers to `sessions.list`, add targeted tests and a changelog entry, and remove unrelated formatting churn.

## Reproduction Assessment

Yes. Current main reproduces the original issue by source inspection: the Limit control is a plain text input that forwards raw string values. The PR-specific regression is also reproducible from the diff because stored refresh paths can still supply the app's string default while the controller no longer coerces it before `sessions.list`.

## Solution Assessment

No. The requested direction is reasonable, but this patch is not the best implementation because it partially migrates the state model and relies on `0` for blank/invalid input; a parse-boundary fix or complete `number | undefined` migration is safer and narrower.
