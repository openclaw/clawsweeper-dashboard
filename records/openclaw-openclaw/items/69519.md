## Summary

Keep open: this PR targets a source-reproducible Control UI validation issue, but the current branch still has blocking correctness findings and lacks after-fix real behavior proof before merge.

## What This Changes

The PR changes the Control UI Sessions Limit filter from a string text input to a numeric input, migrates selected state types/tests, and includes CSS formatting churn.

## Best Possible Solution

Finish the narrow UI fix with one consistent state model: either keep editable strings and coerce at the request boundary, or migrate all defaults/app/view/controller limit state to `number | undefined` while only sending positive integers.

## Reproduction Assessment

Yes, by source inspection. Current main renders the Sessions Limit control as a plain input that forwards string values into filter state, while the Gateway `sessions.list` contract requires a positive integer limit.

## Solution Assessment

No. A numeric input is a reasonable direction, but this implementation removes request-boundary coercion before all default/state paths are numeric and uses `0` as a blank sentinel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Coerce stored limit before sessions.list:** `ui/src/ui/controllers/sessions.ts:159`
  - body: `loadSessions` now forwards `state.sessionsFilterLimit` directly when no override is passed, but this branch still initializes that property from a string default and refresh paths call `loadSessions` without overrides. That can send `limit: "120"` to `sessions.list`, violating the integer schema until the user edits the field.
  - confidence: 0.92
- **[P2] Preserve blank limits without a zero sentinel:** `ui/src/ui/views/sessions.ts:293`
  - body: The input handler maps empty, invalid, or non-positive values to `0`, so clearing the field stores an invalid sentinel and turns later refreshes into an omitted-limit path. Keep editable blank state as blank/undefined, or preserve string state and only send positive integers at the request boundary.
  - confidence: 0.84
