## Summary

Keep open. The requested Sessions Limit fix is valid on current main, but this PR is not mergeable as-is because the branch removes request-boundary coercion while the app still initializes the stored limit as a string, and it uses a zero sentinel for blank or invalid input instead of one consistent editable-state model.

## What This Changes

The PR changes the Control UI Sessions Limit field toward a numeric input and number-backed view/controller state, updates two tests, and includes unrelated CSS/app-render formatting churn.

## Best Possible Solution

Finish a focused Sessions Limit fix with one consistent model: either keep editable string UI state and parse only at the controller boundary, or migrate the full app/view/controller chain to `number | undefined`. The final patch should render a numeric input with `min="1"` and `step="1"`, preserve blank editing, only send positive integers to `sessions.list`, add targeted tests and a changelog entry, and remove unrelated formatting churn.

## Reproduction Assessment

Yes. Current main reproduces the original issue by inspection: the Sessions Limit control is a plain text input that forwards raw string values. The PR-specific regression is also reproducible from the diff because the controller would forward stored state directly while `app.ts` still defaults that state to `"120"`.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best implementation because it partially migrates the type chain and uses `0` as an invalid/blank sentinel; a narrow parse-boundary fix or complete `number | undefined` migration is safer.
