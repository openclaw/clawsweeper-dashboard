## Summary

Keep this PR open. Current main still has the string-backed Sessions Limit field, so the intended fix is still relevant, but the PR branch is not merge-ready because it partially migrates the filter to numeric state while the app initializer remains string-backed and the controller then forwards that stored value directly to `sessions.list`.

## What This Changes

The PR changes the Control UI Sessions Limit field toward a numeric input, updates related Sessions view/controller state types and tests, and includes unrelated CSS/app-render formatting churn.

## Best Possible Solution

Finish a focused Sessions Limit fix that chooses one consistent model: either keep editable string UI state and validate/parse only at the controller boundary, or migrate the full app/view/controller state chain to `number | undefined`. The final behavior should render a numeric input with `min="1"` and `step="1"`, preserve blank editing, only send positive integers to `sessions.list`, cover the view/controller paths, and remove unrelated formatting churn.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the original issue: the Sessions Limit input is a plain text input that forwards raw strings, and the PR-specific regression is visible from the diff because the controller forwards stored state directly while `app.ts` still initializes that state as `"120"`.

## Solution Assessment

No. The direction is reasonable, but this PR is not the best current solution because it only partially migrates the type chain and relies on `0` as an invalid/blank sentinel; a narrower, consistent parse boundary or full `number | undefined` migration would be safer.

Full review comments:

- **[P1] Coerce the stored limit before sending sessions.list:** `ui/src/ui/controllers/sessions.ts:156`
  - body: This change forwards `state.sessionsFilterLimit` directly when no override is provided, but the app still initializes that field as the string `"120"` and Sessions tab refresh calls `loadSessions(app)` without a limit override. That path can now send `limit: "120"` to `sessions.list`, which violates the protocol schema's integer limit contract and can fail before the user edits the filter. Keep request-boundary coercion or complete the full state migration before removing it.
  - confidence: 0.92
- **[P2] Avoid storing zero for an empty Limit field:** `ui/src/ui/views/sessions.ts:304`
  - body: The input handler writes `0` for blank or invalid values even though the field has `min="1"` and the PR says empty/invalid input should fall back to `undefined`. That leaves filter state carrying a value that is invalid for the control and depends on the downstream `limit > 0` guard to avoid sending it. Use `number | undefined` consistently, or keep editable string state and omit invalid values only at the request boundary.
  - confidence: 0.82
