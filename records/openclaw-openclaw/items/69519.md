## Summary

Keep open: current main still has a text-backed Sessions Limit field, so the PR addresses a real UI bug, but the patch still has a request-boundary type regression and lacks convincing after-fix real behavior proof.

## What This Changes

The PR changes the Control UI Sessions Limit filter toward a numeric input and number-backed state, updates focused tests, and includes unrelated CSS/app-render formatting churn.

## Best Possible Solution

Finish a narrow Sessions Limit fix with one consistent state model: keep editable string UI state and parse at the controller boundary, or fully migrate the app/view/controller chain to `number | undefined`, while preserving blank editing and sending only positive integers to `sessions.list`.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: Control -> Sessions renders a plain Limit text input that forwards raw string input, while the protocol requires a positive integer limit.

## Solution Assessment

No. The PR direction is right, but this implementation is not the best fix because it partially migrates the state chain and removes the controller coercion that protects refresh paths with string-backed state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Coerce the stored limit before sessions.list:** `ui/src/ui/controllers/sessions.ts:156`
  - body: This change forwards `state.sessionsFilterLimit` directly when no override is provided, but the app still initializes that value from the string `DEFAULT_SESSIONS_FILTERS.limit`. Opening or refreshing Sessions can therefore send `limit: "200"` to `sessions.list`, which violates the Gateway schema's integer limit contract.
  - confidence: 0.93
- **[P2] Represent blank limits without a zero sentinel:** `ui/src/ui/views/sessions.ts:304`
  - body: The input handler writes `0` for blank or invalid input even though the control has `min="1"` and the PR says invalid or empty values fall back to `undefined`. That leaves filter state carrying an invalid value and relies on downstream guards instead of representing the omitted limit explicitly.
  - confidence: 0.81
