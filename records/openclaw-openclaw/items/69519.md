## Summary

Keep open: the reported Sessions Limit UI problem is valid, but this PR is not merge-ready because it partially migrates limit state, can forward a string limit on refresh/open paths, stores blank input as a zero sentinel, and still lacks after-fix real behavior proof.

## What This Changes

This PR changes the Control UI Sessions Limit filter from a string-backed text input toward a numeric input and number-backed state, with focused test updates plus formatting churn.

## Best Possible Solution

Finish a narrow Sessions Limit fix with one consistent state model: either keep editable string UI state and parse at the controller boundary, or fully migrate app/view/controller state to `number | undefined` while preserving blank input and only sending positive integers.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the Sessions Limit field is a plain text input that forwards raw strings, while the controller and Gateway schema show that only positive integer limits should reach `sessions.list`.

## Solution Assessment

No. The PR direction is reasonable, but this implementation is not the best fix because it removes request-boundary coercion before all state/default paths are numeric and uses `0` as a blank/invalid sentinel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Coerce stored limit before sessions.list:** `ui/src/ui/controllers/sessions.ts:156`
  - body: `loadSessions` now uses `state.sessionsFilterLimit` directly when no override is passed. The app still initializes that field from string `DEFAULT_SESSIONS_FILTERS.limit`, and tab/refresh paths call `loadSessions(app)` without overrides, so the request can send `limit: "200"` instead of the schema-required integer.
  - confidence: 0.93
- **[P2] Represent blank limits without a zero sentinel:** `ui/src/ui/views/sessions.ts:304`
  - body: The handler maps blank or invalid input to `0` even though the input has `min="1"` and the PR says blank/invalid should become `undefined`. That stores an invalid limit value and relies on downstream guards, making cleared input behave as an implicit unbounded request rather than an omitted limit.
  - confidence: 0.82
