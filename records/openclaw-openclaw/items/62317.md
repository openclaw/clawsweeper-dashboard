## Summary

Keep this issue open. Current main still has a catalog path that accepts provider/model rows from the pi registry without filtering aggregator-prefix shadow entries, and same-repo fix PRs are already open with closing references, so the issue should remain open until one lands or is rejected.

## What This Changes

The issue requests that `/model status` stop showing a phantom native provider, such as `google`, when the only configured provider is an aggregator like OpenRouter with a slashed model id.

## Best Possible Solution

Land one narrow catalog-layer fix that removes only registry-inferred aggregator-prefix duplicates, preserves explicitly configured native providers, and adds regression coverage for an OpenRouter `google/...` model id.

## Reproduction Assessment

Yes. The issue gives a concrete configuration, and current source inspection shows the catalog still has a path where a registry-emitted `google` row would survive beside the configured `openrouter` row.

## Solution Assessment

Yes. A catalog-layer filter is the best direction because it fixes all consumers of `loadModelCatalog`; a status-rendering-only skip would hide one symptom while leaving phantom rows in picker/list/catalog consumers.
