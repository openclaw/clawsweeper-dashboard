## Summary

Keep open for normal PR review: the current main and latest shipped release still use a detached child for unmanaged ordinary gateway restarts, while this PR makes that path use the existing in-process restart fallback and preserves the separate update respawn path. I found no blocking functional or security findings in the proposed diff.

## What This Changes

The PR changes ordinary unmanaged gateway restarts to return the run loop's in-process fallback, updates focused respawn tests, and revises low-power host doctor/docs wording plus the changelog.

## Best Possible Solution

Review and land this focused PR through the normal maintainer path so ordinary unmanaged restarts stay in-process while update restarts continue using the fresh-process replacement path.

## Reproduction Assessment

Yes, at source level: current main and `v2026.5.12` detach-spawn for unmanaged ordinary restarts, and the run loop exits on `mode: "spawned"`; a PID-tracking custom supervisor can then lose the replacement process and collide on the gateway port. I did not run a live supervisor loop in this read-only review.

## Solution Assessment

Yes: returning `disabled` for ordinary unmanaged restarts is the narrowest maintainable fix because the run loop already has the in-process fallback, and the PR keeps `update.run` on the separate fresh-process respawn helper.
