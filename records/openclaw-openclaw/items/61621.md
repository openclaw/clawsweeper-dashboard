## Summary

Keep open: current main still creates Feishu agent-facing text with a standalone `[message_id: ...]` line followed by a sender label, and the user-facing reply sanitizer still does not strip that copied Feishu scaffold before delivery. The earlier ClawSweeper comment’s source path still matches current `main`, and no merged or direct open fix was found for this exact visible-output leak.

## What This Changes

The issue asks OpenClaw to keep Feishu agent-only metadata such as `[message_id: ...]` and `ou_...:` out of user-visible quote/reply messages.

## Best Possible Solution

Preserve Feishu IDs in agent context, then add a narrow user-facing sanitizer or Feishu reply transform with regression coverage for copied `[message_id: ...]` plus adjacent Feishu sender-label scaffolding.

## Reproduction Assessment

Yes, at source level. `buildFeishuAgentBody` can produce `[message_id: om_x...]\nou_x...: ...`, and current reply normalization calls `sanitizeUserFacingText`, which does not strip that Feishu legacy scaffold before delivery.

## Solution Assessment

Yes, the best fix is a targeted user-facing sanitizer or Feishu outbound transform, not a new default-visible config option for internal scaffolding. It should keep the agent-facing metadata that existing Feishu media/reply workflows need.
