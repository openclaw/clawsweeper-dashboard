## Summary

Keep this open: current main still constructs Feishu agent-visible input with the legacy `[message_id: ...]` line plus sender-label text, and the shared user-facing reply sanitizer still does not strip that exact echoed shape before delivery.

## What This Changes

The issue asks OpenClaw to prevent Feishu quote/reply metadata such as `[message_id: ...]` and sender IDs from appearing in user-visible chat content.

## Best Possible Solution

Preserve Feishu IDs in structured inbound context and reply-targeting fields, then add targeted user-facing sanitization plus regression coverage for copied Feishu `[message_id: ...]` and adjacent sender-label scaffolding.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an assistant reply that echoes `[message_id: om_x...]\nou_x...: ...` flows through the current Feishu reply path and is not removed by `sanitizeUserFacingText`; I did not run a live Feishu chat.

## Solution Assessment

Yes. The narrowest maintainable solution is a Feishu/user-facing sanitization guard that preserves structured metadata, not a new default-visible `showMetadata` option for internal agent scaffolding.
