## Summary

Keep this open: current main still builds Feishu agent-facing text with the reported standalone `[message_id: ...]` plus sender-label shape, and the generic user-facing sanitizer still does not strip that shape if the model echoes it back.

## What This Changes

The issue requests hiding Feishu internal metadata such as `[message_id: ...]` and `ou_...` sender labels from user-visible quoted or reply messages.

## Best Possible Solution

Keep Feishu IDs available through structured inbound context and reply-target fields, then strip copied legacy `[message_id: ...]` lines plus adjacent Feishu sender-label scaffolding from user-visible replies with targeted regression coverage.

## Reproduction Assessment

Yes, at source level: a Feishu assistant reply that echoes `[message_id: om_x...]\nou_...: ...` flows through `sanitizeUserFacingText` without a message-id or sender-label stripping step. I did not run a live Feishu tenant reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is a targeted user-facing sanitizer guard that preserves structured Feishu metadata for the agent, not a new default-visible `showMetadata` option.
