## Summary

Keep this open: current main still carries Feishu `message_id` and sender-id-shaped labels in the agent-facing prompt path, and the visible reply sanitizer does not remove that legacy shape if the model echoes it back. The newer structured inbound metadata path helps, but it has not replaced or guarded the older Feishu prompt body that matches the reported leak.

## What This Changes

The issue asks OpenClaw to keep Feishu reply/quote metadata such as `[message_id: ...]` and `ou_...` sender ids out of user-visible message content, or make that display configurable.

## Best Possible Solution

Keep Feishu IDs available through structured inbound context and transport fields, then add a narrow user-facing sanitization guard for copied legacy `[message_id: ...]` plus adjacent Feishu sender-id prompt labels, with Feishu reply/quote regression coverage. A user-facing `showMetadata` option is not the best default for internal agent scaffolding.

## Reproduction Assessment

Yes. A source-level reproduction is clear: send a Feishu turn whose model reply echoes `[message_id: om_x...]\nou_x...: ...`; current main normalizes and delivers assistant text without stripping that legacy shape.

## Solution Assessment

No current fix is present on main. The narrowest maintainable solution is to filter copied legacy metadata before Feishu-visible delivery while keeping the structured metadata and reply target fields intact for tool use.
