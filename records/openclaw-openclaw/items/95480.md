## Summary

Keep open: this PR is a focused candidate fix for the canonical Windows gateway Scheduled Task visible-console bug, and current main plus v2026.6.9 still leave generated gateway installs on direct `gateway.cmd` unless the hidden-launcher flag is present in the scheduler render env. The patch uses OpenClaw's existing `gateway.vbs` path and has real Windows Task Scheduler proof, but maintainers still need to accept the WScript default compatibility tradeoff before merge.

## What This Changes

The PR makes generated Windows gateway service installs set and consume the existing hidden launcher env so Scheduled Tasks run `gateway.vbs`, preserves caller-selected task identity, and updates focused tests and Windows docs.

## Best Possible Solution

Land this PR after required checks pass and a maintainer accepts the VBS/WScript default for Windows gateway Scheduled Tasks.

## Reproduction Assessment

Yes. Source inspection shows generated gateway env lacks the hidden-launcher flag on current main and v2026.6.9, while scheduler action resolution only switches to `.vbs` when that flag is present; the linked issue and PR body add Windows runtime evidence.

## Solution Assessment

Yes, with a maintainer compatibility decision. Reusing the existing WScript hidden launcher at the generated service-env source and resolving the task action from that generated env is narrower than adding a second launcher mechanism or broad Windows refactor.
