## Summary

Keep open: this is a real Windows gateway Scheduled Task bug with sufficient live Task Scheduler proof, but the patch still needs a fix because the new render-env merge can override caller-selected task identity.

## What This Changes

This PR defaults generated Windows gateway Scheduled Tasks to the existing `gateway.vbs` hidden launcher, resolves task rendering from generated service env, adds daemon coverage, and updates Windows docs.

## Best Possible Solution

Keep the hidden VBS launcher as the gateway default, but compose task render env so generated launcher defaults apply without replacing caller-owned task identity keys.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.9 leave gateway services without the hidden-launcher default while `schtasks.ts` only selects `gateway.vbs` when the flag is truthy. I did not run a live Windows repro in this read-only review, but the PR body adds live Windows Task Scheduler after-fix output.

## Solution Assessment

No. Reusing the existing VBS launcher is the right direction, but this implementation should preserve caller-selected task identity and leave the WSH-default compatibility tradeoff explicit for maintainers.
