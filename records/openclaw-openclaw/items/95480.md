## Label Justifications

- `P2`: The PR addresses a real Windows gateway managed-startup bug with limited platform blast radius but availability impact when users close the visible console.
- `merge-risk: 🚨 compatibility`: The default generated Windows gateway task would use a `.vbs` WScript wrapper, which can fail on hosts that block Windows Script Host or `.vbs` execution.
- `merge-risk: 🚨 availability`: The patch can leave a hidden Startup-folder login item behind after uninstall, causing the gateway to keep starting unexpectedly.

## Summary

Keep open: the latest PR revision fixes the earlier scheduler-env boundary, but it introduces an install/uninstall asymmetry where hidden `.vbs` launchers can be created from generated service env and then missed by normal uninstall paths that use plain `process.env`.

## What This Changes

The PR changes Windows gateway scheduled-task installs to default to the existing `gateway.vbs` hidden launcher, updates focused daemon tests, and documents Windows cleanup/startup behavior.

## Best Possible Solution

Make hidden-launcher cleanup independent of the caller env flag, then land the merged-env Scheduled Task fix if maintainers accept the WScript default.

## Reproduction Assessment

Yes. Source inspection shows current main omits the gateway hidden-launcher marker and resolves the Scheduled Task action from the scheduler env, so generated gateway installs can still point at `gateway.cmd`.

## Solution Assessment

No. The merged-env Scheduled Task action fix is the right core shape, but the patch must also make hidden `.vbs` cleanup symmetric because install can now create launchers that normal uninstall paths cannot see.
