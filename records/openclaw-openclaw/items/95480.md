## Label Justifications

- `P2`: This is a focused Windows managed-startup bugfix with platform-limited blast radius, but it still affects gateway availability when the visible window is closed.
- `merge-risk: 🚨 compatibility`: The intended default changes newly generated Windows gateway tasks from direct `.cmd` launch to a VBS/WScript wrapper, which can affect locked-down hosts and upgrades.
- `merge-risk: 🚨 availability`: If the scheduler boundary remains wrong or WScript is blocked, the managed gateway can still launch visibly, fail to relaunch, or be killed by users closing the console.

## Summary

Keep open: the PR is still not a correct fix because the added flag lands in the generated service environment, while the Windows Scheduled Task action is selected from the separate install/control env before that generated environment is written.

## What This Changes

Adds the Windows hidden-launcher env marker to generated gateway service environments, adds focused scheduled-task coverage, and updates Windows/uninstall docs for the generated `gateway.vbs` wrapper.

## Best Possible Solution

Set the hidden-launcher marker in the scheduler-control env or make task action resolution consume the merged install env, then prove the actual `openclaw gateway install` path on Windows and document any WSH compatibility decision.

## Reproduction Assessment

Yes. Current main and `v2026.6.9` are source-reproducible because gateway service env omits the hidden-launcher marker while `schtasks.ts` selects `.vbs` only when the scheduler env flag is truthy; I did not run a live Windows desktop repro in this read-only review.

## Solution Assessment

No. Reusing the existing VBS launcher is the right direction, but adding the flag only to the generated service environment is not the best or complete fix because the task action is resolved from a separate install env.
