## Summary

Keep open: the root-cause direction is good and the Windows proof is now materially stronger, but the PR still has a lifecycle parity bug where generated hidden-launcher installs can create VBS artifacts or fallback entries that normal gateway management paths may not discover or remove.

## What This Changes

The PR defaults generated Windows gateway service installs to the existing hidden VBS launcher, resolves Scheduled Task rendering from merged install/service environment, and updates daemon tests plus Windows docs.

## Best Possible Solution

Land the hidden Windows gateway launcher fix with lifecycle parity: successful Scheduled Task installs and Startup-folder fallback installs should remain discoverable, stoppable, restartable, and fully uninstallable from normal gateway commands.

## Reproduction Assessment

Yes. Source inspection shows current main only switches to gateway.vbs when OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER is truthy, while generated gateway service env lacks that marker; the linked issue also supplies Windows reproduction steps.

## Solution Assessment

No as-is. Reusing the existing VBS hidden launcher and resolving the action from generated service env is the right root-cause shape, but the PR must also align gateway status/stop/restart/uninstall and fallback discovery with the new default.

  - body: Passing `staged.taskEnv` into activation means a generated gateway install now carries `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER=1` into both the normal Task Scheduler path and the Startup-folder fallback path.
