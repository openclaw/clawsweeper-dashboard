## Summary

Keep open: the PR now fixes the prior scheduler-env boundary and includes useful live Windows Task Scheduler proof, but the new full env merge can override caller-selected Windows task identity, so it needs a narrow compatibility repair before merge.

## What This Changes

The PR defaults generated Windows gateway Scheduled Tasks to the existing `gateway.vbs` hidden launcher, resolves task rendering from merged caller/service env, adds daemon tests, and updates Windows docs.

## Best Possible Solution

Keep the existing hidden VBS launcher as the gateway default, but compose task render env so generated launcher defaults are applied without overriding caller-owned service identity keys.

## Reproduction Assessment

Yes. Source inspection and the linked issue show current gateway Scheduled Tasks can target `gateway.cmd`, while the existing hidden-launcher path only engages when the env flag is truthy; the PR body also includes live Windows Task Scheduler output after the patch.

## Solution Assessment

No. Reusing the existing VBS hidden launcher is the right fix direction, but this implementation should preserve caller-owned task identity fields while applying generated service-env launcher defaults.
