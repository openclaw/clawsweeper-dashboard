## Label Justifications

- `P2`: This is a focused Windows managed-startup bugfix with platform-limited blast radius, but it affects gateway availability when users close the visible console.
- `merge-risk: 🚨 compatibility`: The intended default changes newly generated Windows gateway tasks from direct `.cmd` launch to a VBS/WScript wrapper, which can affect locked-down Windows hosts and upgrades.
- `merge-risk: 🚨 availability`: If the scheduler boundary remains wrong or WScript is blocked, the managed gateway can still launch visibly, fail to relaunch, or be killed by users closing the console.

## Summary

Keep open: the PR is still not merge-ready because it adds the hidden-launcher marker to the generated service environment, while the Windows Scheduled Task action is resolved from the separate scheduler-control env used by the real install paths.

## What This Changes

This PR adds the hidden Windows task launcher marker to generated gateway service environments, adds focused service-env/schtasks coverage, and updates Windows uninstall/platform docs for `gateway.vbs`.

## Best Possible Solution

Make the actual scheduler-control env select `gateway.vbs`, or make task action resolution consume the merged install environment, then prove the real Windows CLI install/query/run path and document the WSH compatibility expectation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main and `v2026.6.9` omit the gateway hidden-launcher marker while `schtasks.ts` only switches to `.vbs` when the scheduler env flag is truthy. I did not run a live Windows desktop repro in this read-only review.

## Solution Assessment

No. Reusing the existing VBS launcher is the right direction, but setting the marker only in generated service env is not the best fix because the task action is resolved from a separate env in the real install flows.
