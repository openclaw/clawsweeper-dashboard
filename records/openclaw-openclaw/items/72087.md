## Summary

Keep open. The report is recent, has concrete same-host controls plus a second macOS reproduction, and current main still has the distinct packaged entry/bootstrap path before `runCli()` while the Codex image provider remains a lower-level guarded HTTP path.

## What This Changes

Report that packaged `openclaw.mjs`/`dist/entry.js` invocations fail Codex OAuth `gpt-image-2` image generation while direct `runCli()` and provider calls succeed.

## Best Possible Solution

Isolate the exact packaged-entry side effect, then land a narrow launcher/main-entry parity fix with regression coverage for Codex OAuth image generation through `openclaw.mjs` or `dist/entry.js`.

## Reproduction Assessment

Yes, there is a high-confidence reported reproduction path: run the packaged `openclaw infer image generate --model openai/gpt-image-2` command with Codex OAuth and compare it to direct `runCli()`/provider controls. This review did not execute live OAuth, but current main still preserves the distinct entry/main path required by that matrix.

## Solution Assessment

Unclear until the exact side effect is pinned. The maintainable direction is entry/bootstrap parity with packaged-entry coverage; a provider-only fix would ignore the direct provider and direct `runCli()` controls.
