## Summary

Keep open. The report is recent, has detailed Linux and macOS packaged-CLI failure evidence, and current main still has a distinct entry/bootstrap path before `runCli()` while the Codex image provider path remains separately documented and tested.

## What This Changes

The issue reports that packaged `openclaw infer image generate` fails for Codex OAuth image generation while direct `runCli()` and provider-level calls succeed on the same host.

## Best Possible Solution

Trace packaged entry and direct `runCli()` under the same Codex OAuth profile, compare argv/env/proxy/dispatcher/fetch error state, then land a narrow startup parity fix with packaged CLI live coverage.

## Reproduction Assessment

Yes. The high-confidence path is packaged `openclaw infer image generate --model openai/gpt-image-2` with a Codex OAuth-only profile, compared against direct `runCli()` and provider controls; I did not run it here because it requires private OAuth credentials and the live backend.

## Solution Assessment

Unclear until the side effect is pinned down. A provider-only change would miss the reported boundary, so the maintainable fix should be a narrow entry/startup parity change based on traced packaged-entry evidence.
