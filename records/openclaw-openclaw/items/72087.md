Keep open. The report is recent, has detailed cross-platform packaged-CLI versus direct-run evidence, and current main still has a distinct entry/bootstrap path before `runCli()` while the Codex image provider path remains separate.

The issue asks OpenClaw to fix a packaged CLI entry/bootstrap path that makes Codex OAuth `openai/gpt-image-2` image generation fail while direct `runCli()` and provider calls succeed.

## Best Possible Solution

Instrument packaged entry and direct `runCli()` under the same real Codex OAuth profile, compare argv/env/proxy/dispatcher/fetch error state, and land a narrow startup parity fix with packaged CLI coverage.

## Reproduction Assessment

Yes. The high-confidence path is to run packaged `openclaw infer image generate --model openai/gpt-image-2` with a Codex OAuth-only profile and compare it with direct `runCli()` and provider controls; I did not live-run the credentialed path here.

## Solution Assessment

Unclear until the side effect is pinned. A provider-only patch would not address the reported boundary; the safest fix is a narrow entry/startup parity change based on traced packaged-entry evidence.
