## Summary

Keep open. The report has a concrete packaged-entry versus direct-run differential, a second macOS reproduction, and current `main` still has a distinct entry bootstrap path plus global `process.argv` reads before the Codex OAuth image provider path; the related hosted-tool and timeout work does not prove the entry/direct-run parity problem is fixed.

## What This Changes

The issue asks OpenClaw to make packaged CLI entry invocations of `openclaw infer image generate --model openai/gpt-image-2` behave like direct `runCli()` and direct provider calls under Codex OAuth.

## Best Possible Solution

Trace packaged `openclaw`, `node dist/entry.js`, direct `runCli(process.argv)`, direct `runCli(argv)`, and direct provider calls under one Codex OAuth profile, then land a narrow startup parity or provider-dispatcher fix with live packaged CLI proof while coordinating backend availability with https://github.com/openclaw/openclaw/issues/76690.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live against current `main`; the reported packaged CLI repro is concrete for the cited releases and source inspection shows the same entry/direct-run boundary, but a private Codex OAuth profile is needed to verify the failure now.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the exact entry/bootstrap side effect is isolated; a generic provider or network change would be speculative unless tracing proves it removes the packaged-entry parity gap.
