## Summary

Keep open. The report is recent and still credible: current main preserves a distinct packaged entry/bootstrap path before `runCli()`, while the OpenAI Codex image provider remains a lower-level guarded HTTP path; the related open Codex OAuth image issue is a different backend tool-contract failure, not a closure path for this entry/bootstrap differential.

## What This Changes

The issue reports that packaged `dist/entry.js`/`openclaw.mjs` Codex OAuth image generation fails with `fetch failed` while direct `runCli()` and provider calls succeed on the same host.

## Best Possible Solution

Isolate the exact packaged-entry side effect, then land a narrow entry/bootstrap parity fix with packaged CLI coverage for Codex OAuth image generation, keeping separate backend-tool-contract behavior tracked independently.

## Reproduction Assessment

Yes, there is a high-confidence reported path: run packaged `openclaw infer image generate --model openai/gpt-image-2` under a Codex OAuth-only profile and compare it with direct `runCli()` and provider controls. I did not run that credentialed path here, so the structured status is source-reproducible rather than live reproduced.

## Solution Assessment

Unclear until the side effect is pinned. The maintainable solution should compare packaged-entry argv/env/dispatcher/proxy state against the direct controls; a provider-only fix would ignore the reported lower-layer success.
