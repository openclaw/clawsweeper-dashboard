The issue reports that packaged `dist/entry.js` CLI invocation breaks Codex OAuth `openai/gpt-image-2` generation while direct `runCli()` and provider calls succeed on the same host.

## Best Possible Solution

Trace packaged entry, direct `runCli(process.argv)`, direct `runCli(argv)`, and direct provider execution under the same Codex OAuth profile, then fix the exact argv/env/proxy/dispatcher/fetch state divergence or deliberately route the remaining provider-level failure to the canonical image issue.

## Solution Assessment

Unclear. The best fix should come after credentialed tracing identifies the entry/startup side effect; a provider-only or generic network repair is not sufficient unless it also proves packaged-entry parity.
