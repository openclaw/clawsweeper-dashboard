Keep open. The report has detailed packaged-CLI failure evidence plus a second macOS reproduction, and current main still has a distinct main-module entry prelude before `runCli()` with no source, test, release, or live proof that the entry-specific differential is fixed. ## What This Changes

The issue reports that packaged `openclaw infer image generate --model openai/gpt-image-2` fails through `dist/entry.js` while direct `runCli()` and provider calls succeed with the same Codex OAuth profile.

## Best Possible Solution

Trace packaged entry and direct `runCli()` under the same Codex OAuth profile, compare argv/env/proxy/dispatcher/fetch state, then land the narrowest startup parity fix with packaged CLI live coverage.

## Reproduction Assessment

Yes for the reported releases: use packaged `openclaw infer image generate --model openai/gpt-image-2` with a Codex OAuth-only profile and compare direct `runCli()` plus provider controls. I do not have a high-confidence current-main local reproduction because it needs private OAuth credentials and live backend access.

## Solution Assessment

Unclear. A provider-only or generic network fix is not the best answer unless tracing proves it covers the entry boundary; the maintainable fix should target the exact entry/startup side effect that changes the request behavior.
