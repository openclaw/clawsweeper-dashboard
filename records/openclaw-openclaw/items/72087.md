## Summary

Keep open. Current main still has a distinct packaged launcher/main-entry path before `runCli()`, the OpenAI Codex image provider still uses the lower-level guarded HTTP path the reporter says succeeds directly, and the current live CLI coverage does not exercise packaged `openclaw.mjs` or Codex OAuth image generation.

## What This Changes

The issue reports that packaged `openclaw`/`openclaw.mjs`/`dist/entry.js` image generation fails with OpenAI Codex OAuth while direct `runCli()` and provider calls succeed.

## Best Possible Solution

Make packaged `openclaw.mjs`/`dist/entry.js` behaviorally equivalent to direct `runCli()` for OpenAI Codex OAuth image generation, with targeted regression coverage for that launcher boundary.

## Reproduction Assessment

Yes. The report provides exact failing packaged commands, successful same-host lower-layer controls, and a second macOS reproduction; this read-only review did not execute the live OAuth path.

## Solution Assessment

Yes. The narrow maintainable direction is to fix the launcher/main-entry bootstrap boundary and add packaged-entry parity coverage; a provider-only change would not explain the direct provider and direct `runCli()` controls.
