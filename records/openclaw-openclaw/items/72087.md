## Summary

Keep open. The report has detailed packaged-entry failure evidence, a second macOS reproduction, and current main still has a distinct main-module entry prelude plus global argv reads before the Codex OAuth image provider path; related OpenAI image fixes do not prove this entry/direct-run parity bug is fixed.

## What This Changes

The issue reports that packaged/main `dist/entry.js` invocation of `openclaw infer image generate` fails for Codex OAuth `openai/gpt-image-2` while direct `runCli()` and provider calls succeed.

## Best Possible Solution

Trace packaged entry, `node dist/entry.js`, direct `runCli(process.argv)`, and direct provider calls under the same Codex OAuth profile, then land the smallest startup parity fix with packaged CLI live proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not against current main in this review; the reported packaged-CLI repro is concrete for the cited releases, and source inspection shows the entry/direct-run boundary still exists, but live OAuth credentials are needed to verify it now.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the exact entry/bootstrap side effect is isolated; a provider-only or generic network change would be too speculative unless tracing proves it closes the entry parity gap.
