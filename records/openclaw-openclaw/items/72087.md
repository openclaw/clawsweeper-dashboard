## Summary

Keep open. Current `main` still has a distinct packaged launcher/main-entry bootstrap before `runCli()`, while the documented OpenAI Codex image path still goes through the lower-level guarded request path that the reporter verified directly.

## What This Changes

The issue reports that packaged CLI entry execution fails OpenAI Codex OAuth image generation with `fetch failed` while direct `runCli()`, provider, and guarded HTTP helper calls succeed.

## Best Possible Solution

Add a narrow packaged-entry parity fix and regression coverage proving `openclaw.mjs`/`dist/entry.js` behaves like direct `runCli()` for OpenAI Codex OAuth image generation.

## Reproduction Assessment

Yes, for a maintainer with Codex OAuth credentials. The issue gives exact failing packaged commands, same-host lower-layer controls, and a second macOS reproduction; this review confirmed current `main` still preserves the distinct entry/main path.

## Solution Assessment

Yes. The narrow maintainable direction is launcher/main-entry bootstrap parity plus coverage; a provider-only change would miss the direct provider and direct `runCli()` controls.
