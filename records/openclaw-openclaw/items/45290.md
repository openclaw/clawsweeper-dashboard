## Summary

Keep open: current main does not implement the PR's central delivery fallback and recovery changes, and the branch still has high-confidence correctness blockers plus a routing security concern. The related replacement PR #52275 covers only one queue-recovery subset, not this whole mixed transport/session/Telegram patch.

## What This Changes

The PR branch broadens outbound delivery fallback/recovery, mirrored session channel selection, context-overflow matching, plain-text outbound sanitization, Argus session metadata handling, and Telegram polling logs.

## Best Possible Solution

Land focused, owner-reviewed patches instead of this broad branch: fail closed for explicit channel selection, add a real retry/drain path before deferring silent deliveries, resolve Argus state through the configured session store if Argus proceeds, and keep any sanitizer or overflow classifier fixes narrowly tested.

## Reproduction Assessment

Yes. The blockers have high-confidence source reproduction paths: enqueue a silent delivery while user-visible work exists, call sendMessage with an invalid explicit channel plus mirrored delivery context, and use a configured session.store outside the transcript directory before Argus lock inspection.

## Solution Assessment

No. The branch targets real reliability gaps, but the current diff is too broad and introduces routing and recovery regressions; narrower patches with fail-closed channel handling and configured-store lookup are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:493-501`
  - body: This returns after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Because recovery drains pending snapshots on startup/reconnect rather than continuously watching this queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Reject explicit channel failures before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: Passing the mirrored fallback into resolveMessageChannelSelection also applies it when params.channel is explicit. A typo or unavailable explicit channel can therefore deliver through the mirrored session channel instead of failing fast, which can route content to an unintended channel.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus checks:** `src/agents/session-write-lock.ts:313-317`
  - body: resolveSessionStorePathForLock always reads sessions.json beside the session file, but session.store supports configured paths and templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can still reclaim locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR can misroute explicit outbound content through a mirrored fallback channel when explicit channel resolution fails.

Concerns:

- **[medium] Explicit channel fallback can misroute outbound content:** `src/infra/outbound/message.ts:113`
  - body: The proposed sendMessage fallback applies even when the caller supplied an explicit channel. If that channel is unavailable or malformed, message content can be routed through the mirrored session channel instead of failing, which can expose content on an unintended channel or recipient path.
  - confidence: 0.82
