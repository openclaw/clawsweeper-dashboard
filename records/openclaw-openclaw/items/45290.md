## What This Changes

The PR changes agent/gateway outbound delivery fallback, delivery queue recovery, session-lock Argus metadata, plain-text sanitization, context-overflow detection, and Telegram polling restart logs.

## Best Possible Solution

Rebase and split the work into narrow current-main PRs: land queue metadata through the focused replacement PR, handle explicit-channel fallback only for implicit/no-channel cases, and add separate proven fixes for sanitizer and overflow detection.

## Reproduction Assessment

No high-confidence live reproduction is available from the PR. Source inspection does give a clear reproduction path for the blockers: enqueue a silent delivery while a user-visible queue entry is pending, or call message.send with an unavailable explicit channel plus a mirrored session fallback.

## Solution Assessment

No. The broad mixed branch is not the best current fix because it combines useful narrow bug fixes with stale paths, new session metadata, and fallback behavior that can route explicit sends incorrectly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not leave deferred silent deliveries pending:** `src/infra/outbound/deliver.ts:501`
  - body: This returns after enqueueing the silent delivery, but nothing schedules that newly queued item for immediate retry. Current recovery/drain code processes pending entries on startup or channel drains, so normal runtime traffic can leave the silent follow-up stuck until a later restart or reconnect.
  - confidence: 0.88
- **[P2] Reject explicit channel errors before session fallback:** `src/infra/outbound/message.ts:115-116`
  - body: Passing a mirror-derived fallback while an explicit channel is present lets an unavailable or mistyped channel be replaced by the session channel. That can send a caller's message to a different channel instead of returning the existing unknown/unavailable-channel error.
  - confidence: 0.84
- **[P2] Resolve Argus context from the configured session store:** `src/agents/session-write-lock.ts:314-315`
  - body: This assumes the store is `sessions.json` next to the session file, but `session.store` can be configured or templated. In those setups Argus-protected sessions are missed, so stale-lock cleanup can reclaim locks that the new metadata was meant to protect.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the explicit-channel fallback change can reroute user content to a different channel.
