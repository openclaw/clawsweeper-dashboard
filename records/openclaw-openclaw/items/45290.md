## Summary

Keep open: the branch still contains some unique reliability fixes that are not fully present on current main, but the proposed PR remains broad, conflicting, lacks real behavior proof, and has source-level correctness and security blockers.

## What This Changes

The PR changes outbound channel fallback, delivery queue recovery, session-lock Argus metadata, context-overflow detection, plain-text sanitization, Telegram polling logs, and related tests.

## Best Possible Solution

Split this into focused owner-reviewed patches: fail closed for explicit channel selection, keep deferred deliveries drainable, resolve lock/session state through configured stores, and land sanitizer/overflow/logging fixes separately with targeted real proof.

## Reproduction Assessment

Yes, for the PR-introduced blockers by source inspection: invalid explicit channel plus mirrored context, silent delivery overlap with a user-visible pending entry, and configured session.store outside dirname(sessionFile)/sessions.json exercise the risky paths.

## Solution Assessment

No. The patch targets real reliability gaps, but it mixes broad routing, protocol, queue, Argus, sanitizer, and Telegram changes while introducing regressions; narrower fail-closed patches with live proof are the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:496-501`
  - body: This returns after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Because recovery drains a snapshot rather than continuously watching the queue, overlap with a user-visible delivery can leave the follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Fail explicit channel errors before falling back:** `src/infra/outbound/message.ts:113-120`
  - body: Passing the mirrored fallback into channel selection also applies it when params.channel is explicit. A typo or unavailable explicit channel can therefore deliver through the mirrored session channel instead of failing fast, which can route content to an unintended channel.
  - confidence: 0.86
- **[P2] Resolve Argus locks from configured session stores:** `src/agents/session-write-lock.ts:316-317`
  - body: resolveSessionStorePathForLock always reads sessions.json beside the session file, but session.store supports configured paths and templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can reclaim locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review
