## What This Changes

The PR changes outbound channel fallback and recovery, session-lock Argus metadata, gateway session patch protocol, context-overflow detection, plain-text sanitization, and Telegram polling logs/tests.

## Best Possible Solution

Split this into focused owner-reviewed patches: fail closed for explicit channel selection, add a real live drain/retry contract before deferring silent deliveries, resolve or defer Argus store semantics, and land sanitizer/overflow/logging fixes separately with targeted proof.

## Reproduction Assessment

Yes, by source inspection. The blockers can be exercised with an invalid explicit outbound channel plus mirrored delivery context, a silent delivery enqueued while user-visible work is pending, and a configured session.store outside dirname(sessionFile)/sessions.json.

## Solution Assessment

No. The branch targets real reliability gaps, but the current diff is too broad and introduces routing, queue-drain, and configured-store regressions; narrower patches are the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:496-501`
  - body: This returns after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Because recovery drains a snapshot rather than continuously watching the queue, overlap with a user-visible delivery can leave the follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Fail explicit channel errors before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: Passing the mirrored fallback into channel selection also applies it when params.channel is explicit. A typo or unavailable explicit channel can therefore deliver through the mirrored session channel instead of failing fast, which can route content to an unintended channel.
  - confidence: 0.86
- **[P2] Resolve Argus locks from the configured session store:** `src/agents/session-write-lock.ts:316-317`
  - body: resolveSessionStorePathForLock always reads sessions.json beside the session file, but session.store supports configured paths and templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can reclaim locks the new protection is meant to preserve.
  - confidence: 0.84
