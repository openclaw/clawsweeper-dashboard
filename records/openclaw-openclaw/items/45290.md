## Summary

Keep open. The PR still has unresolved correctness and security blockers, lacks after-fix real behavior proof, and is broad/conflicting enough that it should be split or replaced rather than cleaned up as obsolete.

## What This Changes

The branch broadens outbound delivery fallback/recovery, mirrored session channel selection, Argus session metadata handling, context-overflow matching, plain-text sanitization, and Telegram polling logs.

## Best Possible Solution

Split this into focused owner-reviewed patches: fail closed on explicit channel selection, add a real drain/retry path before deferring silent deliveries, resolve Argus state through the configured session store if Argus proceeds, and land sanitizer/overflow/logging fixes separately with targeted proof.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths for the blockers: enqueue a silent delivery while user-visible work is pending, call sendMessage with an invalid explicit channel plus mirrored delivery context, and use a configured session.store outside dirname(sessionFile)/sessions.json for Argus lock inspection.

## Solution Assessment

No. The branch targets real reliability gaps, but the current diff is too broad and introduces routing, queue-drain, and configured-store regressions; narrower patches are the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:493-501`
  - body: This returns after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Since recovery drains pending snapshots rather than continuously watching this queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Reject explicit channel failures before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: Passing the mirrored fallback into resolveMessageChannelSelection also applies it when params.channel is explicit. A typo or unavailable explicit channel can therefore deliver through the mirrored session channel instead of failing fast, which can route content to an unintended channel.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus checks:** `src/agents/session-write-lock.ts:313-317`
  - body: resolveSessionStorePathForLock always reads sessions.json beside the session file, but session.store supports configured paths and templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can still reclaim locks the new protection is meant to preserve.
  - confidence: 0.84
