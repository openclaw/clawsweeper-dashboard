## Summary

Keep open: the branch targets real delivery-reliability gaps, but it still introduces source-reproducible regressions in queue draining, explicit channel routing, and Argus store lookup, and current main does not already implement the PR's intended changes.

## What This Changes

The PR broadens outbound delivery fallback and queue recovery, adds Argus session-protection metadata/protocol handling, updates plain-text sanitization and overflow matching, and improves Telegram polling logs.

## Best Possible Solution

Land focused, owner-reviewed patches: fail closed for explicit channel selection, add a real retry/drain path before deferring silent deliveries, resolve Argus state through configured session.store if Argus proceeds, and keep sanitizer or overflow classifier fixes narrowly tested.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths: defer a silent delivery while user-visible work is pending, call sendMessage with an invalid explicit channel plus mirrored delivery context, and use a configured session.store outside the transcript directory before Argus lock inspection.

## Solution Assessment

No. The branch aims at valid reliability problems, but the current diff is too broad and introduces routing, recovery, and store-resolution regressions; narrower patches with fail-closed channel handling and configured-store lookup are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Drain deferred silent deliveries before returning:** `src/infra/outbound/deliver.ts:493-501`
  - body: This returns after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Since queue recovery is startup/reconnect oriented rather than a continuous worker, a normal runtime overlap with user-visible delivery can leave the silent follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Fail explicit channel selection before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: Passing the mirrored fallback into channel resolution also applies it when params.channel is explicit. A typo or unavailable explicit channel can therefore deliver through the mirrored session channel instead of failing fast, which can route content to an unintended channel.
  - confidence: 0.86
- **[P2] Resolve Argus state from the configured store:** `src/agents/session-write-lock.ts:313-317`
  - body: The new lock lookup always reads sessions.json beside the session file, but session.store supports custom paths and templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can still reclaim locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete routing-security concern because explicit outbound channel failures can fall through to mirrored session fallback delivery.

Concerns:

- **[medium] Explicit channel fallback can misroute outbound content:** `src/infra/outbound/message.ts:113`
  - body: The proposed sendMessage fallback applies even when the caller supplied an explicit channel. If that channel is unavailable or malformed, message content can be routed through the mirrored session channel instead of failing, which can expose content on an unintended channel or recipient path.
