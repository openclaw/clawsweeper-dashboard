## Summary

Keep this open: current main and v2026.6.8 still force non-internal Slack `room_event` turns to `message_tool_only` before configured `visibleReplies` can apply, and Slack still wires typing/status callbacks on that path. The related Discord and Slack delivery fixes are adjacent, not a merged fix for this Slack room-event status mismatch.

## What This Changes

The issue asks OpenClaw to make Slack `room_event` reply visibility honor or clearly document `messages.groupChat.visibleReplies: "automatic"`, and to align Slack typing/status indicators when final source replies are suppressed.

## Best Possible Solution

Keep strict ambient-room delivery unless maintainers intentionally revise that policy, and make a focused maintainer decision on whether Slack should suppress, alter, or document typing/status feedback during tool-only room events.

## Reproduction Assessment

Yes, at source level. Current main still forces non-internal `room_event` turns to `message_tool_only` before `visibleReplies` is applied, and Slack still supplies typing/status callbacks in that path; no live Slack scenario was run in this read-only review.

## Solution Assessment

Unclear. Honoring `automatic` for all room events would conflict with the current ambient-room contract, while the narrower Slack typing/status alignment needs a maintainer UX and privacy decision.
