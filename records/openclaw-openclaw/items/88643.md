## Summary

Keep this open: current `main` and the latest release still force non-internal room events into message-tool-only delivery, and Slack still wires typing/status callbacks for that tool-only path. The room-event delivery policy is now documented as intentional ambient-room behavior, so the remaining Slack status mismatch needs maintainer direction rather than cleanup closure.

## What This Changes

The issue reports that Slack room-event turns ignore `messages.groupChat.visibleReplies: "automatic"` by forcing message-tool-only delivery while still showing typing/status indicators.

## Best Possible Solution

Keep ambient room events strict unless maintainers revise that contract, and align Slack typing/status behavior or docs/tests so tool-only room events do not imply an automatic final reply.

## Reproduction Assessment

Yes, at source level: current main still forces non-internal `room_event` turns to `message_tool_only`, and Slack still wires typing/status callbacks for that path. I did not run a live Slack scenario.

## Solution Assessment

Unclear as a product direction: the broad request to make `room_event` honor automatic replies conflicts with current ambient-room docs, while the Slack typing/status alignment looks like the narrowest remaining fix to evaluate.
