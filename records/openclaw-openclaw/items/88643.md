## Summary

Keep open: current `main` and `v2026.6.6` still combine strict Slack room-event source delivery with visible Slack typing/status callbacks, and the remaining fix needs maintainer direction because strict room-event delivery is now documented as intentional ambient-room behavior.

## What This Changes

The issue asks OpenClaw to align Slack room-event visible-reply delivery and typing/status behavior when group/channel `visibleReplies` is configured as `automatic`.

## Best Possible Solution

Keep ambient room events strict unless maintainers revise that contract, and align Slack typing/status behavior or docs/tests so tool-only room events do not imply an automatic final reply.

## Reproduction Assessment

Yes, at source level: current main still forces non-internal `room_event` turns to `message_tool_only`, and Slack still wires typing/status callbacks for that path. I did not run a live Slack scenario.

## Solution Assessment

Unclear as a product direction: making `room_event` honor automatic replies conflicts with current ambient-room docs, while Slack typing/status alignment looks like the narrower remaining fix to evaluate.
