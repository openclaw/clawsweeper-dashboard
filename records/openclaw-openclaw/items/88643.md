## Label Justifications

- `P2`: The issue affects visible Slack reply/status behavior in a bounded channel path but does not make the core runtime unusable.
- `impact:message-loss`: The reported impact is that Slack can show progress feedback while the final assistant text remains transcript-only unless the model uses the message tool.

## Summary

Keep this open: current main and v2026.6.6 still force external room events into message-tool-only delivery before Slack group-chat automatic replies can apply, and Slack still wires typing/status callbacks for that same tool-only path. The remaining issue is a maintainer product decision: keep ambient room-event strictness and align Slack progress visibility, or intentionally revise the room-event delivery contract.

## What This Changes

The issue asks OpenClaw to stop or document Slack room-event suppression of automatic visible replies and to align Slack typing/status indicators with the final delivery policy.

## Best Possible Solution

Keep the strict ambient room-event delivery contract unless maintainers intentionally revise it, and make Slack typing/status behavior or docs match that contract with focused coverage for tool-only room events.

## Reproduction Assessment

Yes, at source level: current main still resolves external `room_event` inputs to `message_tool_only` and still wires Slack typing/status callbacks in that path. I did not run a live Slack scenario, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as final product direction. Removing the room-event force is too broad without maintainer approval, while suppressing or documenting Slack typing/status for tool-only room events is the narrower maintainable path if maintainers want strict ambient rooms to stay strict.
