## Label Justifications

- `P1`: Long but valid sessions can become repeatedly unusable whenever compaction consistently exceeds the whole-operation timeout.
- `impact:session-state`: The failure prevents an existing session from compacting and continuing while discarding completed summarization work.
- `impact:crash-loop`: Subsequent turns can repeat the same bounded compaction attempt and timeout without any convergence mechanism.

## Summary

This remains a valid core compaction bug: current main applies one 180-second timeout to the complete operation, timeout errors bypass the existing partial-summary fallback, and ordinary user-facing failures omit the timeout cause and configuration hint. The related open reports and https://github.com/openclaw/openclaw/pull/81190 cover adjacent delivery, escalation, or tool-truncation behavior but do not own this entire no-progress path.

Likely related people: `wangmiao0668000666` introduced the current default, and `velvet-shark` reviewed that change; both are high-confidence routing candidates.

## What This Changes

Prevent whole-pipeline compaction timeouts from causing identical no-progress retries, and make timeout failures identify the cause and relevant configuration.

## Best Possible Solution

Adopt one bounded, monotonic recovery policy that cannot repeat identical compaction work forever—preferably by combining earlier deterministic reduction or stage-aware limits with a terminal actionable error—before considering canonical persisted resumability for completed chunks.

## Reproduction Assessment

Yes, at high confidence from current source: one outer timer covers the complete compaction and timeout errors bypass partial-summary handling. The supplied synthetic test is not a live provider run, so real-world timing and prevalence remain to be measured.

## Solution Assessment

No, the persisted-progress suggestion is not yet established as the best fix because it adds durable session-state semantics. Maintainers should first compare stage-aware timeouts, earlier truncation or fallback escalation, and a terminal no-retry outcome against the lock-duration constraint that motivated the 180-second default.
