## Label Justifications

- `P1`: Affected sessions can become repeatedly unusable on subsequent turns until the operator changes configuration or abandons session history.
- `impact:session-state`: Completed summarization work is discarded when the operation-wide timeout aborts, preventing the session from advancing toward recovery.
- `impact:crash-loop`: An unchanged large session and timeout can enter the same deterministic compaction failure path on every turn.

## Summary

Current main still has the reported operation-wide 180-second compaction timeout, discards in-memory chunk progress when that timeout aborts the pipeline, and hides the actionable timeout detail by default. The bug remains necessary to address, but the durable recovery contract needs maintainer judgment because timeout scoping, persisted progress, and earlier fallback have different session-state consequences.

## What This Changes

Prevent long chunked compactions from repeatedly timing out without progress and expose the timeout cause and existing configuration knob to users.

## Best Possible Solution

Define one canonical compaction recovery policy that guarantees bounded forward progress without parallel state paths—preferably by carrying validated progress through the existing session/compaction owner or escalating deterministically—then make timeout failures explicitly name the cause and existing `agents.defaults.compaction.timeoutSeconds` setting.

## Reproduction Assessment

Yes at source level: current main retains the single operation-wide default timeout, immediate timeout propagation, local-only chunk summary, and generic non-verbose error described by the report. This read-only review did not perform a live slow-provider compaction run.

## Solution Assessment

Unclear. The diagnosis is accurate, but the proposed alternatives are not interchangeable; maintainers should select one session-state recovery contract rather than independently adding persistence, per-stage timers, and fallback behavior.
