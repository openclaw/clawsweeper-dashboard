## Label Justifications

- `P1`: Affected long-running sessions can repeatedly fail the same compaction and remain unusable on subsequent turns until operator intervention or session replacement.
- `impact:session-state`: The failure prevents a large transcript from advancing into a usable compacted session state and discards completed summarization work.
- `impact:crash-loop`: With unchanged input and budget, subsequent turns can repeat the same timeout path indefinitely without converging.

## Summary

Current main still has the reported failure shape: the default 180-second timer bounds the entire compaction operation, timeout aborts bypass partial-summary recovery, and the ordinary user reply omits both the timeout cause and the compaction timeout setting. The latest stable release still used the older 900-second default, so the 180-second regression is current-main-only, but the underlying whole-operation timeout and lost-progress behavior remains actionable rather than fixed or superseded.

## What This Changes

Make long multi-chunk compaction recover or fail actionably instead of restarting from scratch under one opaque whole-operation timeout.

## Best Possible Solution

Define one bounded recovery contract that retains an overall availability limit while checkpointing usable completed work in the canonical session path or escalating to a deterministic reduction strategy, and make timeout replies name the cause plus `agents.defaults.compaction.timeoutSeconds`.

## Reproduction Assessment

Yes at source level: current main has a concrete deterministic path where one timer covers sequential chunks and timeout bypasses partial-summary preservation. The supplied fake-timer test is a faithful focused model of that path, but no live slow-provider run was independently executed during this read-only review.

## Solution Assessment

Unclear. The diagnosis is strong, but choosing durable checkpoints, stage budgets, earlier truncation, or a hybrid requires an explicit recovery and lock-availability decision; simply removing or multiplying the timeout is not the safest fix.
