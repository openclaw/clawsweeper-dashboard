## Label Justifications

- `P1`: A legitimately long default-config compaction can deterministically fail on every turn and leave an active session unusable until manual intervention.
- `impact:session-state`: The failure prevents large session history from being compacted into a usable continuation state.
- `impact:crash-loop`: Each subsequent turn can repeat the same timed-out compaction from its initial chunk without forward progress.

## Summary

Keep open: current main still applies one 180-second budget to the complete compaction operation, timeout aborts bypass the existing partial-summary fallback, and non-verbose users receive no timeout-specific recovery guidance. No merged or viable canonical item resolves that combined failure.

**Likely related people**

- `wangmiao0668000666` — introduced the current timeout default; high confidence.
- `velvet-shark` — reviewed and merged the timeout-default change; medium confidence.

## What This Changes

Make long multi-chunk compactions recoverable across the safety timeout and provide users with actionable timeout guidance.

## Best Possible Solution

Implement one canonical bounded recovery path—preferably stage-aware budgeting or resumable progress in an existing canonical session/database owner—then expose timeout-specific guidance including `agents.defaults.compaction.timeoutSeconds`, without adding parallel fallback readers or file-based state.

## Reproduction Assessment

Yes at source level: current main retains the operation-wide timer, and its chunk loop explicitly discards partial progress on timeout. The supplied focused fake-timer reproduction models that path, although this read-only review did not execute it.

## Solution Assessment

Unclear: the bug is well supported, but per-stage budgeting, canonical persisted progress, and alternate recovery ordering have different locking and state-ownership consequences, so maintainers must choose the permanent contract before implementation.
