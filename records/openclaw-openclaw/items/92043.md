## Label Justifications

- `P1`: Affected sessions can repeatedly fail before a turn starts and remain unusable until manual configuration or session replacement.
- `impact:session-state`: The failure prevents an oversized session from advancing through compaction while preserving its accumulated transcript.
- `impact:crash-loop`: The same bounded operation can be retried on later turns with no retained progress, producing a persistent availability loop.

## Summary

Keep open: current main and v2026.6.6 still use one 180-second safety timeout for the whole compaction operation, timeout aborts still discard chunk progress, and the non-verbose recovery message still hides the timeout-specific operator action.

## What This Changes

The issue requests recovery for long compactions that exceed the new 180-second default without restarting the full chunk pipeline or hiding timeout guidance.

## Best Possible Solution

Define one canonical compaction recovery state machine that preserves bounded forward progress in existing session state, keeps lock holds bounded, and surfaces timeout plus agents.defaults.compaction.timeoutSeconds.

## Reproduction Assessment

Yes at source level: current main wraps the full sequential compaction promise in one 180-second timeout and immediately propagates timeout aborts before the local rolling summary can be reused.

## Solution Assessment

Unclear: the report identifies a real failure, but durable checkpoints, per-stage budgets, earlier deterministic truncation, and diagnostics-only mitigation have different lock, state-retention, and upgrade consequences.
