## Label Justifications

- `P1`: Affected sessions can repeatedly fail before a turn starts and remain unusable until manual configuration or session replacement.
- `impact:session-state`: The failure prevents an oversized session from advancing through compaction while preserving its accumulated transcript.
- `impact:crash-loop`: The same bounded compaction operation can be retried on later turns with no retained progress, creating a persistent availability loop.

## Summary

Keep open: current main still uses one 180-second safety timeout for a whole compaction operation, abort/timeout paths still discard in-call chunk progress, and the non-verbose recovery text still hides the timeout cause and compaction timeout knob. Related items cover adjacent lock, provider, and message-delivery symptoms, but none is a merged or clearly canonical replacement for this structural recovery problem.

## What This Changes

The issue requests a compaction recovery path that avoids deterministic long-compaction timeout loops and surfaces the timeout cause plus `agents.defaults.compaction.timeoutSeconds`.

## Best Possible Solution

Define one canonical compaction recovery contract that preserves bounded forward progress without reintroducing long lock starvation, and always reports timeout-specific recovery guidance using the existing compaction timeout knob.

## Reproduction Assessment

Yes, at source level: current main wraps the complete compaction operation in one 180-second timeout and timeout aborts propagate before local chunk progress can be reused. I did not run a live slow-provider compaction in this read-only pass.

## Solution Assessment

Unclear. The report identifies a real failure, but maintainers need to choose between durable progress checkpoints, per-stage budgets, earlier deterministic reduction, and diagnostics-only mitigation because each changes lock and session-state behavior.

## Risks / Open Questions

- Persisting chunk summaries across attempts would introduce durable compaction progress state, cleanup, and validity semantics that need an explicit SQLite/session ownership design.
- Applying the timeout per chunk or scaling it by chunk count could reintroduce the long session-lock holds that the 180-second default was meant to reduce.
- A diagnostics-only fix would help users recover manually but would not address the deterministic no-progress loop for slow or local compaction providers.
