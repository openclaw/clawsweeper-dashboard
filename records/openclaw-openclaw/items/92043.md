## Label Justifications

- `P1`: Affected sessions can repeatedly fail before a turn starts and remain unusable until manual configuration or session replacement.
- `impact:session-state`: The failure prevents an oversized session from advancing through compaction while preserving its accumulated transcript.
- `impact:crash-loop`: The same bounded operation can be retried on later turns with no retained progress, producing a persistent availability loop.

## Summary

Keep open: current main still applies one 180-second budget to the complete compaction operation, drops the local rolling summary when timeout aborts chunk processing, and gives non-verbose users no timeout-specific recovery guidance; no merged or viable canonical item now resolves that combined failure.

## What This Changes

Make long compactions recover across bounded attempts, or otherwise avoid repeatedly discarding completed chunk work, while clearly identifying timeout recovery options.

## Best Possible Solution

Define one canonical compaction recovery state machine that preserves bounded forward progress in the existing session store, releases locks predictably, and reports the timeout cause plus the supported operator action without introducing parallel fallback state.

## Reproduction Assessment

Yes, at source level: current main demonstrably wraps the full sequential compaction promise in one 180-second timeout and immediately propagates timeout aborts before the local rolling summary can be reused. This read-only review did not run a live slow-provider compaction.

## Solution Assessment

Unclear. The report correctly identifies the failure, but durable progress checkpoints, per-stage budgets, earlier deterministic truncation, and diagnostics-only mitigation have different lock, state-retention, and upgrade consequences that maintainers must choose between.
