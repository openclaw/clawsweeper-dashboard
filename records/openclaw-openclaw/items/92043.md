## Summary

Keep open: current main and v2026.6.6 still use one 180-second safety timeout for the complete compaction operation, discard in-call chunk progress on timeout, and hide timeout-specific recovery guidance in the default failure text; no merged canonical fix was found in the provided related context or successful live search pass.

## What This Changes

The issue requests making long compaction pipelines recoverable under the 180-second default by reusing partial progress or changing timeout budgeting, and by surfacing timeout-specific recovery guidance.

## Best Possible Solution

Define one canonical compaction recovery state machine that preserves bounded progress in the existing session/state model, releases locks predictably, and reports the timeout cause plus supported operator action without adding sidecar fallback state.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.6 still wrap the full compaction promise in one timeout and rethrow timeout aborts before partial summaries can be reused. This read-only review did not run the proposed Vitest.

## Solution Assessment

Unclear: the report identifies a real failure, but maintainers need to choose the durable progress, timeout-budget, truncation, or diagnostics contract because each changes lock and session-state behavior.
