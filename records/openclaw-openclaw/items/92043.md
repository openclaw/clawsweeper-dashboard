- `impact:session-state`: The reported failure discards completed summarization progress and leaves the same oversized session state failing on repeated turns.

## Summary

Keep open: current main still has the reported 180s whole-compaction safety timeout, timeout/abort paths do not preserve completed chunk-summary progress, and the default user-facing preflight failure text still hides the timeout reason unless details are enabled. The closest merged PR introduced the 180s tradeoff for a different lock-hold problem; related open PRs address diagnostics, tool-result truncation, or per-agent overrides rather than this resumable-progress/default-failure behavior.

## What This Changes

The issue reports that the 180s default compaction timeout applies to the whole chunked compaction pipeline, discards completed chunk summaries on timeout, and hides the timeout/config knob from users.

## Best Possible Solution

Keep this open for a core compaction recovery design that preserves or resumes completed progress, keeps timeout behavior bounded, and surfaces timeout-specific recovery guidance to users.

## Reproduction Assessment

Yes at source level: current main still has a 180s default whole-compaction timeout, timeout/abort paths bypass partial-summary preservation, and the non-verbose preflight failure copy hides the timeout reason. I did not run the proposed Vitest because this review was read-only.

## Solution Assessment

Unclear as an implementation choice: the issue identifies real broken behavior, but the durable fix should be chosen by maintainers because per-stage budgets and persisted/resumable chunk progress have different runtime-state and compatibility tradeoffs.
