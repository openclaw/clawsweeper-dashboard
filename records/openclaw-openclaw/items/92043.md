## Label Justifications

- `P1`: Affected sessions can repeatedly fail before a turn starts and remain unusable until manual configuration or session replacement.
- `impact:session-state`: The failure prevents an oversized session from advancing through compaction while preserving its accumulated transcript.
- `impact:crash-loop`: The same bounded operation can be retried on later turns with no retained progress, producing a persistent availability loop.

## Summary

Keep open: current main and v2026.6.6 still apply one 180-second budget to the complete compaction operation, discard the in-memory rolling summary when timeout aborts chunk processing, and omit timeout-specific recovery guidance for non-verbose users; no canonical issue or PR resolves that combined failure.

## What This Changes

Correct compaction’s whole-pipeline timeout loop, lost partial progress, and opaque timeout guidance.

## Best Possible Solution

Define one canonical compaction recovery state machine that preserves bounded forward progress in the existing session store, releases locks predictably, and reports the timeout cause plus supported operator action without introducing parallel fallback state.

## Reproduction Assessment

Yes, at source level: current main wraps the complete sequential compaction promise in one timeout, and timeout aborts bypass the partial-summary fallback before any progress can be reused. This read-only review did not run a live slow-provider compaction.

## Solution Assessment

Unclear. The report identifies the failure accurately, but durable checkpoints, per-stage budgets, earlier deterministic truncation, and diagnostics-only mitigation have different lock, state-retention, and upgrade consequences that maintainers must choose between.
