- `impact:session-state`: The issue concerns an oversized session that cannot transition into a usable compacted state.
- `impact:crash-loop`: Subsequent turns can repeatedly restart the same compaction work and hit the same deterministic timeout.

## Summary

Current main still has the combined failure: one 180-second deadline covers the complete compaction callback, timeout aborts bypass the existing in-call partial-summary fallback, and the default user-facing response omits both the timeout cause and its configuration knob. The issue should remain open because related reports cover adjacent delivery, truncation-order, or silent-failure symptoms, while no merged fix or viable canonical item resolves this specific bounded-progress problem.

## What This Changes

Make long compactions recoverable without restarting completed chunk work and expose actionable timeout guidance when recovery fails.

## Best Possible Solution

Define one canonical recovery state machine that preserves a bounded total lock budget while carrying safe compaction progress in the appropriate session database, then add focused fresh-session and repeated-attempt tests plus timeout-specific guidance naming `agents.defaults.compaction.timeoutSeconds`.

## Reproduction Assessment

Yes, at source level with high confidence: current main applies one deadline outside the sequential chunk loop, stores progress only locally, and immediately propagates timeout aborts. The supplied fake-timer reproduction accurately demonstrates that control-flow invariant, though a live slow-provider run would still be useful for measuring prevalence.

## Solution Assessment

No single remedy in the report is automatically the best fix; durable progress reuse is preferable only if maintainers define its SQLite ownership and cleanup semantics, while per-stage deadlines alone risk recreating the lock-starvation behavior that motivated the 180-second cap.
