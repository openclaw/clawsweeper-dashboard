## Summary

Keep open: current main still places one 180-second safety budget around the complete compaction operation, chunk summaries are only retained inside the active call, and ordinary preflight failure text does not identify the timeout or its supported setting. The related reports cover delivery retries, recovery ordering, or silent overflow outcomes, but no merged or viable canonical item owns this combined forward-progress problem.

## What This Changes

Make long compactions either retain forward progress or recover gracefully when the shared timeout expires, and show actionable timeout guidance to affected users.

## Best Possible Solution

Define one core-owned compaction recovery contract that preserves bounded session-lock time while guaranteeing measurable forward progress, uses canonical session/database state if progress must survive attempts, covers fallback and quality-guard stages, and emits timeout-specific guidance naming `agents.defaults.compaction.timeoutSeconds`.

## Reproduction Assessment

Yes, at source level with high confidence: current main wraps the complete operation in one timeout, retains rolling chunk progress only in local memory, and hides timeout details in the ordinary reply. This review did not run a live slow-provider compaction because the checkout is read-only and the report's focused reproduction already models the observable control flow.

## Solution Assessment

No, neither merely raising the default nor persisting an ad hoc partial string is the best complete fix. Maintainers should first choose a canonical recovery/state contract that also preserves the bounded-lock objective of https://github.com/openclaw/openclaw/pull/91361 and covers sibling fallback and recovery paths.
