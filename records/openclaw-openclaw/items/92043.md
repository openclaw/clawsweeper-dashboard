## Summary

Keep open: current main and the latest release still show the reported compaction timeout shape, and the only direct fix path is the open linked PR at https://github.com/openclaw/openclaw/pull/92899 rather than a merged implementation.

## What This Changes

The issue reports that compaction uses one 180-second wall-clock budget across the whole chunk summarization pipeline, discards completed chunk progress on timeout, and hides the timeout/config knob from users.

## Best Possible Solution

Review and land the focused linked PR if maintainers accept its bounded timeout-progress contract; otherwise choose a different canonical compaction recovery state machine and close or replace that PR before changing this issue state.

## Reproduction Assessment

Yes, at source level: current main uses one safety timeout around the complete compaction operation and throws timeout errors before completed chunk summaries can be retained. I did not run a live slow-provider compaction in this read-only review.

## Solution Assessment

Unclear until maintainer review of the linked PR completes. The issue is real, but the permanent recovery contract affects session-state ownership and timeout/lock policy, so the open PR should be accepted or replaced deliberately.
