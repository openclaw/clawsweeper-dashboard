## Label Justifications

- `P1`: Affected sessions can repeatedly fail compaction before a turn starts and remain unusable until manual configuration or session replacement.
- `impact:session-state`: The issue is about preserving and recovering an oversized session transcript through compaction.
- `impact:crash-loop`: The same bounded compaction attempt can be retried on later turns with no retained progress, creating a persistent availability loop.

## Summary

Keep open: current main and the latest release still use one 180-second safety timeout for the full compaction operation, timeout aborts discard in-call chunk progress, and the normal preflight recovery text still hides the timeout cause and compaction timeout knob. Related issues and PRs cover adjacent lock, tool-truncation, and visibility problems, but none is a merged canonical fix for this combined failure.

## What This Changes

The issue reports that compaction uses one 180-second wall-clock budget across the full multi-chunk summarization pipeline, discards completed chunk summaries on timeout, and hides the timeout/config knob from non-verbose recovery text.

## Best Possible Solution

Define one canonical compaction recovery path that preserves bounded forward progress or uses stage-scoped budgets without restoring long lock starvation, then surface timeout-specific guidance with the supported config knob.

## Reproduction Assessment

Yes, at source level: current main wraps the complete sequential compaction promise in one 180-second timeout, and timeout/abort errors propagate before the rolling chunk summary can be reused. I did not run a live slow-provider compaction in this read-only review.

## Solution Assessment

Unclear. The report correctly identifies the failure, but durable checkpoints, per-stage budgets, earlier deterministic truncation, and diagnostics-only mitigation have different lock, state-retention, and upgrade consequences.
