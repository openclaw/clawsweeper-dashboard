## Summary

This issue should stay open: current main still applies the 180s compaction safety timeout as one wall-clock cap around a full compaction operation, timeout/abort paths do not persist completed chunk summaries for a later attempt, and the default preflight failure copy still hides the timeout cause and compaction timeout knob. The report is source-reproducible and fits the open compaction-timeout cluster, but the repair changes session-state and timeout/default behavior enough to need maintainer direction before automation.

## What This Changes

The issue asks OpenClaw to make long chunked compaction recoverable under the 180s default by reusing partial progress or changing timeout budgeting, and to surface timeout cause/config guidance to users.

## Best Possible Solution

Keep this issue open for a core compaction design decision that preserves the shorter lock-hold default while making long, healthy compactions either resume from saved progress or fail with clear timeout guidance.

## Reproduction Assessment

Yes, at source level: current main still resolves unconfigured compaction to 180s, applies that budget around the whole compaction call, and rethrows timeout/abort from chunk summarization without carrying completed chunk work into a later attempt. I did not run the temporary reproduction file because this review is read-only.

## Solution Assessment

Unclear as an implementation choice: the report identifies a real recovery gap, but maintainers need to choose between persisted chunk checkpoints, per-stage budgets, explicit timeout escalation, or default policy changes without restoring long lock holds.
