## Summary

Keep open: current main still gives the complete compaction pipeline one 180-second budget, drops its in-memory rolling chunk summary when timeout aborts the attempt, and gives non-verbose users no timeout-specific recovery guidance; no merged or viable canonical item resolves that combined failure.

## What This Changes

Prevent legitimately long compactions from restarting indefinitely under one 180-second wall clock and provide actionable timeout recovery guidance.

## Best Possible Solution

Adopt an owner-reviewed staged recovery contract that bounds session-lock time, preserves safe forward progress through canonical state if checkpointing is selected, and gives non-verbose users the timeout cause plus the existing configuration escape hatch.

## Reproduction Assessment

Yes at source level: current main applies one 180-second timeout to the complete compaction callback, and timeout propagation discards the local rolling chunk summary before reusable progress is returned. The submitted fake-timer reproduction accurately models that control flow, though it is not a live slow-provider run.

## Solution Assessment

No, the issue's possible remedies are not interchangeable: simply increasing the timeout conflicts with the lock-availability rationale, while checkpointing, per-stage budgets, and earlier deterministic reduction require a maintainer-approved state and recovery contract.
