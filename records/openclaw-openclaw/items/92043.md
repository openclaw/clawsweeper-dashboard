## Label Justifications

- `P1`: Affected sessions can repeatedly fail compaction and remain unusable until manual configuration or session replacement.
- `impact:session-state`: The failure prevents an oversized session from advancing through compaction while preserving accumulated transcript context.
- `impact:crash-loop`: The same bounded compaction attempt can retry with no retained progress, causing a persistent availability loop.

## Summary

Keep open: current main and latest release still use one 180-second timeout around the full compaction operation, drop chunk-local progress on timeout/abort, and omit timeout-specific recovery guidance in the normal preflight failure text; no merged or safe canonical item resolves that combined failure.

## What This Changes

The issue asks OpenClaw to avoid deterministic repeated compaction failure when long chunked compaction exceeds the default 180-second wall-clock budget and to surface the timeout cause and recovery knob.

## Best Possible Solution

Define one canonical compaction recovery state machine that makes bounded forward progress, preserves session data through the existing canonical store, releases locks predictably, and tells users when the compaction timeout knob is the supported operator action.

## Reproduction Assessment

Yes, at source level: current main wraps the whole compaction operation in one 180-second safety timeout, and timeout/abort exits the chunk loop before any partial summary can be reused. I did not run the supplied synthetic Vitest because this review must keep the checkout read-only.

## Solution Assessment

Unclear. The report identifies a real failure, but the best permanent fix depends on maintainer choice between durable checkpointing, per-stage budgets, earlier deterministic truncation, or a narrower diagnostics/default adjustment.
