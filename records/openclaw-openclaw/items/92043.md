## Label Justifications

- `P1`: Affected sessions can repeatedly fail compaction before a turn starts and remain unusable until manual configuration or session replacement.
- `impact:session-state`: The failure prevents an oversized session from advancing through compaction while preserving accumulated transcript context.
- `impact:crash-loop`: The same compaction attempt can be retried on later turns with no retained progress, producing a persistent availability loop.

## Summary

Keep open: current main and v2026.6.6 still use one 180-second compaction wall clock, do not persist chunk progress across attempts, and keep the non-verbose preflight failure text generic; no merged or viable canonical PR now owns the structural recovery choice.

## What This Changes

The issue asks OpenClaw to make long compactions recoverable by reusing partial progress or budgeting per stage, and to surface timeout-specific recovery guidance.

## Best Possible Solution

Define one canonical compaction recovery state machine that preserves bounded forward progress in session-owned storage, releases locks predictably, and reports timeout-specific recovery guidance without adding parallel fallback state.

## Reproduction Assessment

Yes, at source level: current main wraps the full sequential compaction operation in one 180-second timeout and does not persist completed chunk summaries across attempts. I did not run a live slow-provider compaction in this read-only review.

## Solution Assessment

Unclear. The report identifies a real failure, but the best permanent fix depends on maintainer choice between durable progress checkpoints, per-stage budgets, earlier deterministic truncation, and diagnostics-only mitigation.
