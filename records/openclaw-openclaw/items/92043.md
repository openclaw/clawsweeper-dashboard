## What This Changes

Request to prevent long compaction runs from looping after the 180-second wall-clock timeout by reusing partial chunk progress or otherwise making timeout recovery actionable.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92899 if maintainers accept its bounded timeout-progress design, or replace it with a maintainer-approved compaction recovery state-machine fix before closing this issue.

## Reproduction Assessment

Yes. Source inspection and current tests show that after one chunk succeeds, timeout-shaped failures are rethrown before the partial summary can be reused; I did not run a live slow-provider compaction in this read-only review.

## Solution Assessment

Likely yes for the linked PR path, pending maintainer acceptance. It fixes the timeout-progress loss inside the compaction/session owners without changing the public config shape, while durable checkpointing or per-stage budgets would be larger state and lock-policy changes.
