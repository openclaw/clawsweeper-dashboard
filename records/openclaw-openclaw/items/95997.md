## Summary

Keep open: this is a protected maintainer issue, current main still has the source-backed CLI/shared-auth gap, and https://github.com/openclaw/openclaw/pull/96002 is the open draft closing PR for the exact fix path.

## What This Changes

The issue reports that read-scoped local CLI Gateway probes can leave durable CLI device scope state at `operator.read`, blocking later write-scoped local CLI agent calls.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96002, or an equivalent narrow fix, that omits durable device identity only for loopback CLI shared-token/password calls while preserving remote and device-token scope-upgrade enforcement.

## Reproduction Assessment

Yes, at source level with high confidence: CLI calls can still carry durable device identity under loopback shared-token/password auth, `health` is read-scoped, `agent` is write-scoped, and the server-side shared-auth scope-preservation carve-out is currently backend-only. I did not run the live Gateway sequence in this read-only review.

## Solution Assessment

Yes, the linked PR shape appears to be the narrow maintainable solution: move only loopback CLI shared-token/password calls out of durable device identity and preserve missing-device scopes only for that first-party local CLI shared-auth case. The safer alternative is not to relax scope-upgrade enforcement for paired device-token or remote clients.
