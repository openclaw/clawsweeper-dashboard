## Summary

Keep open: this is a protected maintainer issue, current main still has the source-backed CLI shared-auth scope poisoning path, and the exact fix is already tracked by an open draft closing PR.

## What This Changes

The issue reports that read-scoped local CLI Gateway probes can persist a durable read-only device baseline that blocks later write-scoped CLI agent calls.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96002, or an equivalent narrow fix that keeps local CLI shared-token/password calls off durable device identity while preserving explicit scope-upgrade approval for remote and device-token paths.

## Reproduction Assessment

Yes, at source level: current main maps health to operator.read and agent to operator.write, keeps default CLI shared-token calls on durable device identity, and preserves missing-device scopes only for the backend gateway-client path. I did not run the live Gateway sequence in this read-only review.

## Solution Assessment

Yes, the linked PR is the narrow maintainable fix shape because it changes only loopback CLI shared-token/password handling and adds matching client/server regression coverage. A broader silent local scope-upgrade change, such as https://github.com/openclaw/openclaw/pull/96381, would need separate security/product review because it changes the paired-device upgrade boundary.
