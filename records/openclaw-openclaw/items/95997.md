## Summary

Keep open: this is a protected maintainer issue with a source-reproducible current-main Gateway auth bug and an open closing PR that is not in main or the latest release.

## What This Changes

The issue reports that a local CLI read-scoped Gateway probe can persist a read-only device baseline and then block a later write-scoped CLI agent call.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96002, or an equivalent narrow fix, while preserving paired-device scope-upgrade enforcement outside local CLI shared-token/password calls.

## Reproduction Assessment

Yes, at source level: current main maps `health` to `operator.read` and `agent` to `operator.write`, keeps durable device identity for local CLI shared-token/password calls, and preserves missing-device scopes only for backend gateway-client. I did not run a live Gateway sequence in this read-only review.

## Solution Assessment

Yes, the linked PR's direction appears to be the narrow maintainable fix: omit durable identity only for loopback CLI shared-token/password calls and preserve scopes server-side only for that local CLI shared-auth path. Weakening paired-device scope-upgrade enforcement would be the wrong layer.
