## Summary

Keep open: this is a protected maintainer issue with a source-backed Gateway auth bug and an open closing PR that is not on current main. Current main still omits durable device identity only for backend gateway-client shared auth, while CLI-mode shared-token/password calls remain the unresolved path.

## What This Changes

The issue asks OpenClaw to stop local CLI shared-token/password read probes from persisting or reusing a read-scoped device baseline that blocks later write-scoped Gateway calls.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96002, or an equivalent narrow fix, so loopback CLI shared-token/password calls avoid durable device identity while remote and device-token scope upgrades still require approval.

## Reproduction Assessment

Yes. Source inspection shows gateway health resolves to operator.read, agent resolves to operator.write, and current CLI-mode shared-token calls still carry durable device identity while the server scope-preservation exception is backend-only.

## Solution Assessment

Yes. The linked PR’s shape is the narrow maintainable path because it targets only loopback CLI shared-token/password identity omission and matching server scope preservation while leaving remote and device-token scope-upgrade checks intact.
