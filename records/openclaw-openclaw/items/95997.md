## Label Justifications

- `P1`: The issue reports a release-blocking local CLI Gateway auth regression that can block the primary agent path after a read probe.
- `impact:security`: The affected behavior sits on the device pairing and scope-upgrade authz boundary and must preserve explicit approval for real scope widening.
- `impact:auth-provider`: The failure is caused by token/password, device identity, and stored device-token scope selection in Gateway authentication.

## Summary

Keep open: current main still has the reported CLI shared-auth/device-scope interaction, and the issue is maintainer-labeled, member-authored, assigned, and already paired with an open draft fix PR at https://github.com/openclaw/openclaw/pull/96002.

## What This Changes

The issue reports that read-scoped local CLI Gateway probes can leave a durable CLI paired-device baseline that blocks later write-scoped local CLI agent calls.

## Best Possible Solution

Land a narrow Gateway auth fix that omits durable device identity for loopback CLI shared-token/password calls, preserves declared scopes only for local non-browser CLI shared auth, and keeps paired-device scope-upgrade enforcement unchanged.

## Reproduction Assessment

Yes by source, though I did not execute the live sequence: current main maps `health` to `operator.read`, maps `agent` to `operator.write`, keeps device identity for explicit CLI loopback shared-token auth, and requires pairing when requested scopes exceed a paired device baseline.

## Solution Assessment

Yes, the requested fix shape is the narrowest maintainable direction: handle local CLI shared auth at the Gateway call/auth boundary while preserving generic device-less scope clearing and remote/device-token scope-upgrade approval.
