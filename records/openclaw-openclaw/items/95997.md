## Summary

Keep open: this protected maintainer issue is still source-reproducible on current main and in the latest release, and https://github.com/openclaw/openclaw/pull/96002 is the open draft closing candidate for the narrow fix path.

## What This Changes

Ensure local CLI shared-token/password Gateway probes do not persist a read-scoped device baseline that blocks later write-scoped calls while preserving paired-device scope-upgrade enforcement.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96002 or an equivalent narrow fix that omits durable device identity only for loopback CLI shared-token/password calls and preserves paired-device scope-upgrade enforcement elsewhere.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main maps `health` to `operator.read`, `agent` to `operator.write`, keeps durable device identity for local CLI shared-token calls, and only preserves missing-device scopes for backend gateway-client.

## Solution Assessment

Yes. The narrow linked-fix shape is the best direction because it keeps paired-device scope-upgrade enforcement intact while moving only loopback CLI shared-token/password calls out of durable device identity with targeted regression coverage.
