## Summary

Keep open: this is a maintainer-labeled P1 Gateway auth bug with source-backed current-main behavior and an open draft closing PR. Current main still only omits durable device identity and preserves device-less shared-auth scopes for the backend gateway-client path, not for local non-browser CLI shared-token/password calls.

## What This Changes

The issue reports that local CLI Gateway read-scoped probes can persist a durable read-only device baseline that blocks later write-scoped agent calls.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96002, or an equivalent narrow fix, while preserving paired-device scope-upgrade enforcement for remote, browser, and device-token paths.

## Reproduction Assessment

Yes, at source level with high confidence: health resolves to operator.read, agent resolves to operator.write, current CLI shared-token calls still carry device identity, and the server preserves missing-device scopes only for backend gateway-client. I did not run the live Gateway sequence in this read-only review.

## Solution Assessment

Yes, the linked PR shape is the narrow maintainable fix: move only loopback CLI shared-token/password calls out of durable device identity and add a server-side CLI scope-preservation exception, without weakening remote or device-token scope-upgrade checks.
