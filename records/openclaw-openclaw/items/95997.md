## Label Justifications

## Summary

Keep open: current main still has the source-backed local CLI shared-auth scope poisoning path, and this protected maintainer issue already has an open linked candidate fix that has not merged or shipped.

## What This Changes

Local CLI Gateway read probes can persist a read-scoped durable device baseline that blocks later write-scoped CLI agent calls.

## Best Possible Solution

Review and land PR https://github.com/openclaw/openclaw/pull/96002, or an equivalent narrow fix, that omits durable device identity only for loopback CLI shared-token/password calls and preserves scope-upgrade enforcement elsewhere.

## Reproduction Assessment

Yes, at source level: health resolves to operator.read, agent resolves to operator.write, current local CLI shared-token calls still load a device identity, and the server only preserves missing-device scopes for backend gateway-client shared auth. I did not run a live Gateway sequence because this review is read-only.

## Solution Assessment

Yes, the linked maintainer PR shape is the best narrow fix: omit durable identity for loopback CLI shared-token/password calls and preserve scopes only for that local non-browser CLI shared-auth path, without weakening paired-device scope-upgrade enforcement.
