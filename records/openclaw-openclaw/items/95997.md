## Summary

Keep open: this is a protected maintainer issue with a source-backed current-main bug and an open draft closing PR. Current main still only omits durable device identity for the backend gateway-client local path, while the linked fix commit is not contained in main or the latest release.

## What This Changes

Local CLI shared-token/password Gateway calls should avoid persisting or reusing a read-scoped paired-device baseline while preserving normal scope-upgrade approval for real paired devices.

## Best Possible Solution

Review and land the linked draft PR, or an equivalent narrow fix, that omits durable device identity only for loopback CLI shared-token/password calls and preserves scope-upgrade enforcement elsewhere.

## Reproduction Assessment

Yes, at source level with high confidence: health resolves to operator.read, agent resolves to operator.write, current CLI shared-token calls still carry device identity, and the server preserves missing-device scopes only for backend gateway-client. I did not run the live Gateway sequence in this read-only review.

## Solution Assessment

Yes, the linked PR's shape is the narrow maintainable fix: move only loopback CLI shared-token/password calls out of durable device identity and add a server-side CLI scope-preservation exception, without weakening remote or device-token scope-upgrade checks.
