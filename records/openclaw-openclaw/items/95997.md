## Summary

Keep open: current main still has the reported source-level Gateway auth bug, and the issue is a protected maintainer item with an open draft closing PR rather than an implemented fix.

## What This Changes

Fix local CLI Gateway shared-token/password read probes that can persist a read-scoped device baseline and block later write-scoped agent calls.

## Best Possible Solution

Review and land the linked draft PR, or an equivalent narrow fix, that keeps loopback CLI shared-token/password calls out of durable device identity while preserving scope-upgrade enforcement elsewhere.

## Reproduction Assessment

Yes, at source level: health resolves to operator.read, agent resolves to operator.write, current CLI shared-token calls still carry device identity, and device-less scope preservation is backend-only. I did not run the live Gateway sequence in this read-only review.

## Solution Assessment

Yes, the linked PR's shape is the narrow maintainable fix: move only loopback CLI shared-token/password calls out of durable device identity and preserve scopes only for local non-browser CLI shared-secret auth. Weakening paired-device scope-upgrade enforcement would be the wrong layer.
