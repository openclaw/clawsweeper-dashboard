## Summary

Keep open: current main still has the source-level CLI shared-token scope poisoning path, and live GitHub shows the exact closing PR is still open. The issue is also maintainer-authored and protected by the maintainer/security labels, so cleanup should not close it before the fix lands.

## What This Changes

The issue requests a Gateway auth fix so local CLI shared-token/password read probes do not persist a read-scoped device baseline that blocks later write-scoped local CLI calls.

## Best Possible Solution

Land the linked Gateway fix or an equivalent narrow repair that keeps local CLI shared-token/password calls out of durable device scope baselines while preserving paired-device scope-upgrade enforcement everywhere else.

## Reproduction Assessment

Yes. ## Solution Assessment

Yes. The linked PR's direction is the narrowest maintainable fix because it removes durable CLI identity only for loopback shared-token/password calls and adds a matching local CLI scope-preservation exception without weakening paired-device scope-upgrade checks.
