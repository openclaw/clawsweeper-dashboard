## Summary

Keep open. The PR is directionally useful, but latest head still lets field-level compaction consumers re-inherit global defaults even though the branch now documents and tests `agents.list[].compaction: {}` as the way to clear inherited compaction behavior.

## What This Changes

The PR adds schema, docs, runtime wiring, and tests for per-agent compaction and contextPruning overrides across direct, queued, gateway, manual compaction, and memory-flush paths.

## Best Possible Solution

Make the agent-scoped config resolver carry the inheritance decision once: absent block means inherit, object block means replace or clear, and all compaction/pruning consumers should use that contract with inherited, partial replacement, and `{}` clear regression tests.

## Reproduction Assessment

Yes at source level. The PR docs and resolver test define `{}` as clearing inherited defaults, while the PR-head memory-flush planner still falls back to `agents.defaults.compaction.memoryFlush` when the per-agent block omits `memoryFlush`; I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The implementation should converge on one public inheritance contract and route all consumers through it instead of mixing replacement semantics in the resolver with field-level fallback in runtime helpers.
