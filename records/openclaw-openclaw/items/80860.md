## Summary

Keep this PR open. The bug is valid and the direction is useful, but the current head still has a blocking timeout-budget regression: it applies the reduced readiness budget to the whole direct bridge invocation, not only to waiting for bridge registration.

## What This Changes

The PR makes the native hook relay CLI wait longer for direct bridge readiness, reserve fallback/output timeout budget, emit richer unavailable diagnostics, and add CLI regression tests.

## Best Possible Solution

Keep the readiness wait, but cap only the missing-registry wait; once a bridge record is found, allow the full remaining hook budget minus response reserve and add regression coverage for a slow but reachable bridge invocation.

## Reproduction Assessment

Yes. Current main is source-reproducible: the CLI passes registrationTimeoutMs: 100, while the PR regression registers the relay after 150 ms and expects the direct path to recover.

## Solution Assessment

No. The PR is solving the right bug, but the current implementation uses the readiness budget as the entire direct bridge timeout, which can abort a reachable relay before the external hook timeout is exhausted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the bridge invocation on the full hook budget:** `src/cli/native-hook-relay-cli.ts:67`
  - body: This passes the reduced readiness budget as the entire direct bridge timeout. Once the registry exists, a default 5s hook gives the relay handler only 3.75s, so a reachable PreToolUse/PermissionRequest that would finish inside Codex's 5s window can be timed out and sent to fallback or fail-closed unnecessarily. Limit the reduced budget to registration waiting and keep the connected bridge invocation on the remaining hook budget minus the response reserve.
  - confidence: 0.86
