## Summary

Keep open: the PR targets a real source-reproducible startup race, but the patch can spend the full hook timeout waiting for bridge registration before fallback or fail-closed output, and the supplied real behavior proof is still not inspectable.

## What This Changes

The PR changes the native hook relay CLI to wait for delayed direct bridge registration, add unavailable diagnostics, and add regression coverage.

## Best Possible Solution

Keep the direct bridge readiness wait, but reserve enough command budget for gateway fallback and fail-closed responses under the default Codex hook timeout, with regression coverage for that budget.

## Reproduction Assessment

Yes, from source: current main waits only 100 ms for direct bridge registration, while the PR regression registers the bridge after 150 ms and expects the direct CLI path to succeed. I did not execute it because this review is read-only.

## Solution Assessment

No. Waiting for the full CLI timeout addresses delayed bridge registration, but it can exhaust the same 5 second provider hook budget before gateway fallback or fail-closed JSON is emitted.
