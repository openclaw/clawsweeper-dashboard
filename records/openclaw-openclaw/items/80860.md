## Summary

Keep open: the startup-readiness bug is plausible and the diff targets the right surface, but the patch spends the full hook command timeout waiting for bridge registration, which can leave no time for gateway fallback or fail-closed output on the default Codex hook settings.

## What This Changes

The PR changes `openclaw hooks relay` to wait longer for the direct native hook bridge, adds unavailable diagnostics, and adds CLI regression coverage for delayed bridge registration.

## Best Possible Solution

Keep the direct bridge readiness wait, but reserve enough command budget for gateway fallback and fail-closed responses under default Codex hook timing, with a regression that models that timeout budget.

## Reproduction Assessment

Yes, from source: current main gives bridge registration only 100ms before fallback, while the PR’s regression starts the CLI before `registerNativeHookRelay` and registers the bridge after 150ms. I did not run the test because the review was read-only.

## Solution Assessment

No. The narrow fix should wait long enough for startup races while preserving fallback and fail-closed output within the external Codex hook timeout; using the entire timeout as registration wait does not preserve that default budget.
