## Summary

## What This Changes

The PR updates the native hook relay CLI timing and diagnostics, adds focused relay tests, and records a changelog entry.

## Best Possible Solution

Keep the startup readiness wait and diagnostics, but revise direct bridge budgeting so stale bridge records still leave a meaningful gateway fallback window.

## Reproduction Assessment

Yes, source-reproducible. Current main gives bridge registration only 100 ms before fallback, while the harness uses that registration timeout to decide when a not-found bridge stops polling; the remaining review finding is also source-reproducible from retryable stale-record errors consuming the direct bridge timeout.

## Solution Assessment

No, not yet. The proposed direction is narrow, but the direct bridge timeout still needs to be capped or otherwise reserve fallback time for stale registry records.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Reserve fallback time after stale bridge retries:** `src/cli/native-hook-relay-cli.ts:171`
  - body: `calculateNativeHookRelayDirectBridgeTimeoutMs` returns the whole remaining hook timeout minus only the response reserve. Cap the direct bridge attempt, for example with the direct-bridge fraction or an explicit fallback reserve.
  - confidence: 0.86
