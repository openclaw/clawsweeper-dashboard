## Summary

Keep this PR open. It fixes a real native hook relay startup-readiness race, but the current patch still lets retrying direct-bridge calls consume nearly the whole hook timeout, which can leave the gateway fallback with only 1 ms for stale or unreachable bridge records.

## What This Changes

The PR updates the native hook relay CLI timeout budgeting and unavailable diagnostics, and adds focused regression tests for delayed bridge registration, absent bridge fallback, and provider-compatible fail-closed output.

## Best Possible Solution

Keep the bounded startup-readiness wait and improved diagnostics, but preserve a meaningful gateway fallback budget for retryable direct-bridge failures before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main waits only 100 ms for a bridge record that is written asynchronously, while the PR regression registers after 150 ms and would miss that old window.

## Solution Assessment

No. The direction is right, but the current helper gives retrying direct-bridge calls the full remaining budget, so stale bridge records can leave the gateway fallback with only 1 ms instead of a usable timeout slice.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Cap retrying bridge calls before fallback:** `src/cli/native-hook-relay-cli.ts:171`
  - body: `calculateNativeHookRelayDirectBridgeTimeoutMs` returns the full remaining budget. For an unexpired stale bridge record, `invokeNativeHookRelayBridge` treats `ECONNREFUSED` and `EAGAIN` as retryable and can spend that whole budget before throwing, so the gateway fallback is called with only 1 ms left. Please cap the direct bridge slice, or otherwise reserve a real fallback budget, and cover the retryable stale-bridge path.
  - confidence: 0.86
