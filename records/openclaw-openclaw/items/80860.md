## Summary

Keep open: the bug is valid and the PR is on the right surface, but the latest head still caps the entire direct bridge invocation to the readiness budget, and the live proof is not inspectable enough for the external-PR proof gate.

## What This Changes

The PR updates the native hook relay CLI to wait longer for bridge readiness, reserve fallback/output budget, emit richer unavailable diagnostics, and add CLI regression tests.

## Best Possible Solution

Keep the startup-readiness wait and diagnostics, but split the bridge registration-wait budget from the connected bridge invocation budget and add a slow reachable bridge regression.

## Reproduction Assessment

Yes, source-reproducible. Current main gives direct bridge registration only 100 ms while the PR's delayed-registration regression waits 150 ms before registering, matching the startup race.

## Solution Assessment

No. The PR targets the right code path, but the latest head still applies the readiness/fallback reserve to the entire direct bridge invocation; the narrow fix is to cap only registration waiting and leave the connected bridge on the remaining hook budget minus response reserve.
