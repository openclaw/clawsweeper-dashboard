## Summary

Keep this PR open, but do not merge the current head yet: the fix is aimed at a real native hook relay startup race, but the new timeout split makes the existing gateway fallback path wait up to 75% of the hook timeout when no bridge registry exists.

## What This Changes

The PR adjusts the native hook relay CLI to wait for delayed direct bridge records, reserve fallback and response timeout budget, emit detailed unavailable diagnostics, and add CLI relay regression coverage.

## Best Possible Solution

Keep the richer diagnostics and delayed-registration handling, but cap absent-registry polling separately so fallback-only relay paths remain responsive.

## Reproduction Assessment

Yes, source-reproducible. Current main caps missing bridge registration polling at 100 ms while the PR's delayed-registration regression registers after 150 ms; I did not execute tests because this review is read-only.

## Solution Assessment

No for the current patch. The split-budget approach is directionally right, but missing-registry polling needs a small cap while real direct-bridge invocation can use the remaining hook budget.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap the missing-bridge poll before fallback:** `src/cli/native-hook-relay-cli.ts:158`
  - body: When no bridge registry exists, `invokeNativeHookRelayBridge` only breaks after `registrationTimeoutMs`; with the default 5000 ms timeout this calculation waits about 3750 ms before trying the existing gateway fallback. That regresses embedded/local or older registrations that intentionally rely on `nativeHook.invoke`, adding multi-second latency to each hook before fallback can run. Keep the startup-readiness wait bounded separately from the overall direct-bridge invocation budget.
  - confidence: 0.88
