## Summary

Keep open. The PR targets a real source-reproducible native hook relay readiness bug, but the latest diff still has a fallback-budget bug that should be fixed before merge.

## What This Changes

The PR changes the native hook relay CLI timing budget, unavailable diagnostics, regression tests, and changelog entry for Codex app-server hook relay readiness.

## Best Possible Solution

Fix the direct-bridge budget so retryable stale bridge failures still leave a usable gateway fallback window, then land the readiness wait, diagnostics, and regression tests together.

## Reproduction Assessment

Yes. Current main has a 100 ms direct bridge registration window, and the PR's delayed-registration regression waits 150 ms before registering, which source-reproduces the missed startup-readiness path.

## Solution Assessment

No, not yet. The readiness wait and diagnostics are the right narrow direction, but the direct bridge retry budget should be capped or otherwise reserve real time for gateway fallback when a stale bridge record is retryable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reserve fallback time for stale bridge retries:** `src/cli/native-hook-relay-cli.ts:171`
  - body: The new direct-bridge timeout uses the whole remaining hook budget minus only the response reserve. Because `invokeNativeHookRelayBridge` treats `ECONNREFUSED` as retryable until `timeoutMs`, a stale bridge record can spend almost the entire hook timeout before gateway fallback, leaving the fallback with only 1 ms and defeating the missing/disappeared-relay recovery this PR is trying to add.
