## Summary

Keep open: current main has the Telegram richMessages opt-in/runtime fix, but the latest stable release is still v2026.6.8 and the stable tag matches the reported schema/runtime mismatch.

## What This Changes

The issue reports that v2026.6.8 still sends Telegram text via rich messages and rejects the documented richMessages config key while current Telegram docs describe rich messages as opt-in.

## Best Possible Solution

Publish or backport a stable release containing the Telegram richMessages opt-in gate and schema, or clearly caveat v2026.6.8 until the stable package matches the docs.

## Reproduction Assessment

Yes. Source comparison shows v2026.6.8 lacks the richMessages schema key and sends normal Telegram text through sendRichMessage, while current main has the opt-in gate; two issue comments also report live stable deployments reproducing the rejection and Telegram client symptom. I did not run a live Telegram Desktop packaged repro in this read-only review.

## Solution Assessment

No, this should not close yet. The main-side implementation from https://github.com/openclaw/openclaw/pull/93279 is the right source fix, but this issue tracks the remaining stable-release/docs mismatch and needs a maintainer release/backport or documentation-caveat decision.
