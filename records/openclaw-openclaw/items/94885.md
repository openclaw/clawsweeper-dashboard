## Summary

Keep open: current main has the Telegram rich-message opt-in/runtime fix, but this issue is specifically tracking that the latest stable release, v2026.6.8, still ships the older rich-message default and schema mismatch. The second live forum-topic confirmation strengthens the stable-release impact rather than making this a duplicate of the closed main-fixed reports.

## What This Changes

The issue asks maintainers to resolve the mismatch between v2026.6.8 stable Telegram rich-message behavior and current docs that describe rich messages as an explicit opt-in.

## Best Possible Solution

Publish a stable release or backport that includes the merged Telegram rich-message opt-in fix, or temporarily caveat the v2026.6.8 stable docs/release notes until the stable package matches the docs.

## Reproduction Assessment

Yes, source and user evidence give a high-confidence reproduction for v2026.6.8: inspect the tag to see the missing schema key and unconditional sendRichMessage path, or run the reported config dry-run on that stable package. I did not run a live packaged Telegram Desktop repro in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes for the code direction, but not as a close: current main’s opt-in gate is the right runtime/schema fix, while the remaining issue is the maintainer release/backport or stable-docs decision.
