## Summary

Keep open. Current main has tightened the macOS LaunchAgent scanner, but the doctor cleanup renderer still emits canonical gateway removal commands without receiving the detected service context, and the issue is paired with open unmerged implementation PRs that reference it with closing syntax.

## What This Changes

`openclaw doctor` should stop suggesting removal of the active canonical gateway LaunchAgent when reporting extra gateway-like services and should handle `exec:` SecretRef failures as doctor warnings rather than hard errors.

## Best Possible Solution

Keep this issue open and review PR #52515, or a smaller rebased equivalent, until the cleanup hints are either detected-service-specific or suppressed for the active canonical gateway, with focused regressions for macOS LaunchAgent targeting and the reported `exec:keychain:` doctor behavior.

## Reproduction Assessment

Yes for the cleanup-hint path: current source and tests provide a high-confidence static reproduction where a custom macOS LaunchAgent running `openclaw gateway` is reported as extra and then the doctor renderer still emits canonical cleanup hints. The exact `exec:keychain:` exit-code path remains only partially reproducible from current-main evidence because no targeted test covers that specific SecretRef source.

## Solution Assessment

No current-main solution is complete yet. The scanner tightening is useful, but the maintainable fix still needs detected-service context or suppression of canonical active-gateway cleanup hints, plus a narrow doctor SecretRef contract for `exec:` refs.
