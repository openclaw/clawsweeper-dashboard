## Label Justifications

- `P2`: This is a normal-priority doctor/status bug with limited blast radius but real gateway availability impact if users follow bad cleanup guidance.
- `impact:crash-loop`: The reported cleanup hint can unload or remove the running gateway service, causing process-level downtime.

## Summary

Keep open: current main and the latest release still render profile-owned gateway cleanup commands whenever extra gateway-like services are detected, and the implementation candidate at https://github.com/openclaw/openclaw/pull/52515 is open and conflicting. Recent code has narrowed some false-positive detection cases, but the unsafe cleanup-hint path is still present.

## What This Changes

The issue asks `openclaw doctor` and gateway status to avoid cleanup guidance that can unload the active macOS gateway LaunchAgent, and to treat unavailable `exec:keychain:` SecretRefs as doctor warnings instead of hard failures.

## Best Possible Solution

Land a revised https://github.com/openclaw/openclaw/pull/52515 or a narrow replacement that carries detected service metadata into doctor/status cleanup hints, suppresses active canonical gateway cleanup guidance on macOS, and handles doctor SecretRef inspection without weakening strict runtime resolution.

## Reproduction Assessment

Yes, at source level for the cleanup-hint symptom: current tests still produce an extra macOS gateway service, and current doctor/status code renders cleanup hints without service context. The `exec:keychain:` subclaim was not live-reproduced here.

## Solution Assessment

No, current main is not a complete solution. The maintainable fix is to render cleanup hints from actual detected service metadata and use explicit doctor read-only SecretRef semantics rather than weakening strict helpers globally.
