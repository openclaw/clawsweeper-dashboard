## Summary

Keep open. Current main and the latest shipped release still render cleanup commands for the profile-owned gateway after detecting extra gateway-like services, and the linked implementation PR remains open and conflicting rather than merged.

## What This Changes

The issue asks doctor and gateway status to stop cleanup guidance from targeting the active macOS gateway LaunchAgent and to handle runtime-only exec SecretRefs without failing read-only doctor checks.

## Best Possible Solution

Land a revised https://github.com/openclaw/openclaw/pull/52515 or a replacement that carries detected service metadata into doctor/status cleanup hints, suppresses active gateway cleanup guidance, honors service scope, and preserves strict SecretRef runtime helpers.

## Reproduction Assessment

Yes, at source level for the cleanup-hint symptom: current tests show custom LaunchAgents executing `openclaw gateway` are reported as extra services, and current doctor/status code renders cleanup hints without detected service context. The exec:keychain symptom is not live-reproduced here.

## Solution Assessment

No, current main is not a complete solution and the linked PR is not ready as-is. The maintainable fix is to render cleanup hints from actual detected service metadata and use explicit SecretRef inspection semantics rather than weakening strict helpers.
