## Summary

Keep open: current main and the latest release still render cleanup hints from the active gateway profile after detecting extra gateway-like services, and the linked closing PR remains open, conflicting, and previously reviewed with blockers.

## What This Changes

The issue reports that `openclaw doctor` can suggest cleanup commands for the active macOS gateway LaunchAgent and can fail on runtime-resolved `exec:keychain:` SecretRefs.

## Best Possible Solution

Land or replace the linked PR with a narrow fix that renders cleanup hints from actual detected stale-service metadata in both doctor and status, honors service scope, and keeps strict SecretRef behavior outside read-only doctor/status paths.

## Reproduction Assessment

Yes, at source level: current main reports nonempty extra gateway-like services and then calls `renderGatewayServiceCleanupHints()` without detected-service metadata, which renders active-profile cleanup commands. I did not run a live macOS LaunchAgent reproduction in this read-only Linux checkout.

## Solution Assessment

Yes on direction, not yet on the open implementation: cleanup hints should be rendered from actual detected service records while strict SecretRef helpers remain strict outside read-only doctor/status paths. The open linked PR needs repair or replacement before this issue can close.
