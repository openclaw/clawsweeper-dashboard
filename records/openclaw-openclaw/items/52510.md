## Label Justifications

- `P2`: This is a normal-priority doctor/status bug with limited blast radius but real gateway availability impact if users follow bad cleanup guidance.
- `impact:crash-loop`: The issue is about cleanup guidance that can unload or remove the running gateway service and cause process-level downtime.
- `impact:auth-provider`: The issue also covers `exec:keychain:` SecretRef handling in doctor, which is credential and SecretRef resolution behavior.

## Summary

Keep open. Current main and the latest shipped release still render cleanup commands for the profile-owned gateway after detecting extra gateway-like services, and the linked implementation PR remains open and conflicting rather than merged.

## What This Changes

The issue asks `openclaw doctor` to stop suggesting cleanup commands that remove the active macOS gateway LaunchAgent and to avoid hard-failing on runtime-only `exec:keychain:` SecretRefs.

## Best Possible Solution

Land a revised https://github.com/openclaw/openclaw/pull/52515 or replacement that carries detected service metadata into doctor/status cleanup hints, suppresses active gateway cleanup guidance on macOS, honors service scope, and preserves strict SecretRef runtime helpers.

## Reproduction Assessment

Yes, at source level for the cleanup-hint symptom: current tests show custom LaunchAgents executing `openclaw gateway` are reported as extra services, and current doctor/status code renders cleanup hints without detected service context. The `exec:keychain:` symptom was not live-reproduced here.

## Solution Assessment

No, current main is not a complete solution and the linked PR is not ready as-is. The maintainable fix is to render cleanup hints from actual detected service metadata and use explicit SecretRef inspection semantics rather than weakening strict helpers.
