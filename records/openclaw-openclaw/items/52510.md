## Summary

Keep open. Current `main` has improved some false-positive detection and read-only SecretRef handling, but the cleanup-hint path can still report a detected extra service and then print commands for the active profile gateway; the issue is also paired with an open fixing PR that explicitly closes it.

## What This Changes

The issue asks `openclaw doctor` to stop suggesting removal of the active macOS gateway LaunchAgent plist and to avoid failing on runtime-only `exec:keychain:` SecretRefs.

## Best Possible Solution

Revise or replace the linked PR so doctor and status render cleanup hints from actual detected service metadata, honor user/system scope and active state, and keep SecretRef warnings in explicit read-only inspection paths.

## Reproduction Assessment

Yes, source-level. On current `main`, a non-empty `extraServices` list reaches `maybeScanExtraGatewayServices()`, which calls `renderGatewayServiceCleanupHints()` without service metadata; on macOS that helper formats commands for the active gateway label instead of the detected plist.

## Solution Assessment

Yes, the requested direction is the right fix, but it should land through the linked PR or a narrow replacement. The maintainable boundary is detected-service-specific cleanup guidance plus read-only SecretRef inspection, not process-wide leniency or profile-derived cleanup commands.
