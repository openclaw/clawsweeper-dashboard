## Summary

Keep open: current main and v2026.6.8 still render cleanup commands from the active gateway profile whenever extra gateway-like services are reported, so the core cleanup-hint footgun remains despite partial detection and SecretRef improvements.

## What This Changes

The issue asks doctor/status cleanup hints to avoid targeting the active macOS gateway LaunchAgent and to keep exec-backed SecretRef checks non-fatal in read-only doctor paths.

## Best Possible Solution

Land or replace the linked PR with a narrow fix that renders cleanup hints from the actual detected stale service, honors service scope, updates doctor and status together, and keeps strict SecretRef behavior outside read-only doctor/status paths.

## Reproduction Assessment

Yes, at source level: current main reports nonempty extra gateway-like services and then calls `renderGatewayServiceCleanupHints()` without detected-service metadata, which renders active-profile cleanup commands. I did not run a live macOS LaunchAgent reproduction in this read-only Linux checkout.

## Solution Assessment

Yes on direction, not yet on implementation: cleanup hints should be rendered from actual detected service records while strict SecretRef helpers remain strict outside read-only doctor/status paths. The open linked PR needs repair or replacement before this issue can close.
