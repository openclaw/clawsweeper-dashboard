- `impact:crash-loop`: The reported cleanup hint can lead users to unload or remove the running gateway service, causing process availability downtime.
- `impact:auth-provider`: The issue also covers SecretRef resolution behavior in doctor, which falls under credential and SecretRef handling.

## Summary

Keep open. Current main has partial hardening for false-positive detection, but doctor and gateway status still render cleanup commands from the active profile instead of the detected extra service, and the open linked fix PR is still unmerged and conflicting.

## What This Changes

The issue asks doctor to stop suggesting removal of an active or unrelated macOS gateway LaunchAgent plist and to avoid treating runtime-only exec SecretRefs as fatal doctor errors.

## Best Possible Solution

Land a revised version of https://github.com/openclaw/openclaw/pull/52515 or a replacement that renders cleanup hints from the actual detected service metadata, suppresses active/current gateway cleanup guidance, and keeps read-only SecretRef inspection degraded without weakening runtime strictness.

## Reproduction Assessment

Yes, source-level. Current tests prove a custom LaunchAgent executing openclaw gateway is reported as an extra service, and current doctor/status callers then render cleanup hints without service context; I did not run tests because this review is read-only.

## Solution Assessment

No, current main is not a complete solution. The maintainable fix is to carry detected service metadata into doctor/status cleanup hints while preserving the existing strict-versus-inspect SecretRef contract.
