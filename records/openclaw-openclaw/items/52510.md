## Summary

Keep this open: current main still renders doctor/status cleanup hints for the active profile service whenever an extra gateway-like service is reported, and the linked closing PR is still open and conflicting rather than merged.

## What This Changes

The issue asks doctor/status cleanup hints to avoid targeting the active macOS gateway LaunchAgent plist, and also asks doctor to treat exec SecretRef-backed credentials as read-only warnings instead of fatal errors.

## Best Possible Solution

Render cleanup hints from actual detected service metadata, including platform and user/system scope, and keep this issue as the canonical tracker until the linked PR is fixed and merged or superseded by a narrow replacement.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: a custom macOS LaunchAgent that executes `openclaw gateway` is reported as extra, then doctor/status call the cleanup helper without passing that service, and the helper emits active-profile cleanup commands.

## Solution Assessment

No. The current implementation is still too broad; the best fix is to build cleanup hints from detected service metadata and scope, while preserving the already safer exec SecretRef read-only behavior.
