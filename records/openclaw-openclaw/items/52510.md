## Summary

Keep open: current main and v2026.6.8 still render cleanup commands from the active gateway profile whenever extra gateway-like services are reported, and the linked implementation PR remains open, conflicting, and not merged.

## What This Changes

The issue asks doctor to stop suggesting removal of the active macOS gateway LaunchAgent plist and to handle exec-backed SecretRefs in doctor without a fatal exit.

## Best Possible Solution

Land a corrected version of the linked PR, or a narrow replacement, that renders cleanup guidance from actual detected service metadata with correct scope, never targets the active launcher, updates doctor and status together, and keeps SecretRef degradation explicit to read-only doctor/status paths.

## Reproduction Assessment

Yes, at source level: current main reports nonempty extra gateway-like services and then calls renderGatewayServiceCleanupHints() without detected service metadata, which renders active-profile cleanup commands. I did not run a live macOS launchctl repro in this read-only Linux checkout.

## Solution Assessment

Yes on direction, not yet on implementation: cleanup hints should be rendered from the actual detected service records while strict SecretRef helpers remain strict outside read-only doctor/status paths. The open linked PR needs repair or replacement before this issue can close.
