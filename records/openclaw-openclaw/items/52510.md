## Label Justifications

- `P2`: This is a normal-priority doctor/status bug with limited blast radius but real gateway availability impact if users follow the bad cleanup hint.
- `impact:crash-loop`: The cleanup hint can unload or remove the running gateway service, causing process-level availability downtime.
- `impact:auth-provider`: The issue also covers `exec:keychain:` SecretRef handling in doctor, which is part of credential and SecretRef resolution behavior.

## Summary

Keep open. Current main still emits cleanup commands for the profile-owned gateway after detecting extra gateway-like services, and the linked implementation PR remains open and conflicting rather than merged.

## What This Changes

The issue reports that `openclaw doctor` can print cleanup commands that remove the active gateway LaunchAgent plist and can fail hard on `exec:keychain:` SecretRefs.

## Best Possible Solution

Land a revised https://github.com/openclaw/openclaw/pull/52515 or a replacement that carries detected service metadata into doctor/status cleanup hints, suppresses active gateway cleanup guidance, and preserves the strict SecretRef runtime boundary.

## Reproduction Assessment

Yes, source-level. Current tests show custom LaunchAgents executing `openclaw gateway` are reported as extra services, and current doctor/status code then renders cleanup hints without that detected service context.

## Solution Assessment

No, current main is not a complete solution. The maintainable fix is to render cleanup hints from actual detected service metadata and use explicit SecretRef inspection semantics instead of weakening strict helpers.
