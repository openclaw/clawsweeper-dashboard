## Summary

Keep open: current main and v2026.6.8 still render cleanup commands from the active gateway profile whenever extra gateway-like services are reported, and the open linked implementation PR is still unmerged and conflicting.

## What This Changes

The issue asks doctor to stop suggesting cleanup of the active gateway LaunchAgent and to keep exec-backed SecretRefs from making read-only doctor checks exit non-zero.

## Best Possible Solution

Land a corrected version of the linked fix, or a narrow replacement, that renders cleanup guidance from actual detected service metadata with the right scope, never targets the active launcher, updates doctor and status together, and keeps read-only SecretRef degradation out of strict runtime helpers.

## Reproduction Assessment

Yes, at source level: current main reports nonempty extra gateway-like services and then calls `renderGatewayServiceCleanupHints()` without passing those services, which renders active-profile cleanup commands. I did not run a live macOS `launchctl` repro in this read-only Linux checkout.

## Solution Assessment

Yes on direction, not yet on implementation: rendering cleanup hints from actual detected services and keeping read-only SecretRef degradation explicit is the right fix shape, but https://github.com/openclaw/openclaw/pull/52515 needs repair or replacement before this issue can close.
