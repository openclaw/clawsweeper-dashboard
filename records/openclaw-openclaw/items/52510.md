## Label Justifications

- `P2`: This is a normal-priority doctor/status bug with limited blast radius but real gateway availability impact when users follow the hint.
- `impact:crash-loop`: The reported cleanup hint can lead users to unload or remove the running gateway service, causing process availability downtime.
- `impact:auth-provider`: The issue also covers doctor behavior around SecretRef resolution for configured credentials.

## Summary

Keep open. Current main has hardened some false-positive detection and the exec SecretRef crash class appears separately addressed, but doctor/status still render cleanup commands from the active profile rather than the detected extra service, and an open linked PR explicitly closes this issue.

## What This Changes

The issue asks doctor to stop suggesting cleanup commands that remove the active macOS gateway LaunchAgent and to avoid failing on runtime-only exec SecretRefs.

## Best Possible Solution

Land a revised version of https://github.com/openclaw/openclaw/pull/52515 or a replacement that renders cleanup hints from actual detected service metadata in doctor and status, suppresses active/current gateway cleanup guidance, and keeps SecretRef read-only inspection explicit without weakening runtime strictness.

## Reproduction Assessment

Yes, source-level. Current tests show a custom LaunchAgent executing openclaw gateway is reported as an extra service, and current doctor/status callers then render cleanup hints without carrying that detected service context.

## Solution Assessment

No, current main is not a complete solution. The maintainable fix is to carry detected service metadata into doctor/status cleanup hints while preserving the existing strict-versus-inspect SecretRef contract.
