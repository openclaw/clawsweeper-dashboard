## Summary

Keep open: this is a useful draft, but it is not mergeable as-is because it drops an existing successful-delivery cleanup signal and still reports intentional hook-cancelled sends as failures. The supplied proof is also too indirect for an external PR because it runs a standalone mirrored script rather than patched OpenClaw behavior.

## What This Changes

This PR adds `deliveryStatus` tracking and non-JSON delivery-failure warnings to `deliverAgentCommandResult`, wraps delivery calls for status tracking, expands delivery tests, and adds a changelog entry.

## Best Possible Solution

Preserve or migrate the existing cleanup contract, consume delivery outcome cancellation metadata from https://github.com/openclaw/openclaw/pull/57843 before classifying empty results, and then land with focused regression tests plus patched-OpenClaw proof.

## Reproduction Assessment

Yes, at source level: current main can receive an empty delivery result and return only `deliverySucceeded` without structured status or a zero-result diagnostic. I did not establish a live stale-channel reproduction in this read-only review.

## Solution Assessment

No. The direction is maintainable, but this patch is not the best mergeable form until it preserves the durable cleanup signal and distinguishes hook cancellations from actual delivery failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the deliverySucceeded cleanup signal:** `src/agents/command/delivery.ts:458-462`
  - body: `agent-command.ts` still clears `pendingFinalDelivery` only when `deliveryResult?.deliverySucceeded === true`, but this return object no longer includes that field. Successful main-session deliveries would leave the durable retry marker set, so restart/recovery can replay a reply that already reached the channel; keep the compatibility flag or update the caller and regression coverage together.
  - confidence: 0.94
- **[P2] Treat hook cancellations as non-failures:** `src/agents/command/delivery.ts:416`
  - body: `deliverySucceeded = results.length > 0` reports every empty result set as failed delivery, but `deliverOutboundPayloads` intentionally returns `[]` when `message_sending` hooks cancel all payloads. That creates false failure status and warning logs for expected policy suppression, so consume the delivery outcome cancellation signal or keep this PR sequenced behind that work.
  - confidence: 0.88
