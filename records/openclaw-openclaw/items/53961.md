## Summary

Keep open. Current main still lacks structured deliveryStatus tracking, so the PR is not obsolete, but the draft branch is dirty and the proposed patch still breaks the JSON contract on strict delivery failures while misclassifying expected hook cancellations as failed delivery.

## What This Changes

This PR adds deliveryStatus tracking, diagnostic delivery-failure logging, JSON deliveryStatus output, and command delivery tests around deliverAgentCommandResult.

## Best Possible Solution

Keep the delivery observability direction, but land it only after the branch is rebased onto current main, preserves a single parseable --json stdout object on strict delivery errors, and consumes DeliveryOutcome-style cancellation metadata. Coordinate the JSON deliveryStatus surface with #57755 so the repository lands one coherent contract.

## Reproduction Assessment

Yes. Code inspection gives high-confidence paths: strict --json --deliver can rethrow before the PR's delayed JSON block runs, and an all-cancelled message_sending hook returns [] on current main, which the PR maps to succeeded: false.

## Solution Assessment

No. The proposed direction is useful, but this patch is not the best mergeable solution until it emits JSON before rethrowing delivery failures and distinguishes expected hook cancellations from actual zero-result delivery failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit JSON before rethrowing delivery failures:** `src/agents/command/delivery.ts:330`
  - body: The PR suppresses the existing early JSON envelope whenever `--deliver` is set. In strict delivery mode, `deliverOutboundPayloads` can rethrow before the later JSON block runs, so `openclaw agent --json --deliver` can produce no parseable stdout even though the command reserves stdout for JSON output.
  - confidence: 0.9
- **[P2] Treat hook cancellations as non-failures:** `src/agents/command/delivery.ts:426`
  - body: `deliverySucceeded = results.length > 0` treats every empty result set as a failed send, but `deliverOutboundPayloads` intentionally returns `[]` when `message_sending` hooks cancel all payloads. That creates false failure telemetry/status for expected cancellations, so this should consume cancellation metadata or wait for the DeliveryOutcome work.
  - confidence: 0.86
