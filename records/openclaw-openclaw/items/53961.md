## Summary

Keep open: this is a useful draft direction for delivery observability, but the current branch still drops a current-main cleanup contract and misclassifies intentional hook cancellations as failed delivery.

## What This Changes

The branch adds `deliveryStatus` return data and `[delivery]` warning logs in `deliverAgentCommandResult`, with command delivery tests for success, missing target, empty result, throw, partial failure, and internal-channel paths.

## Best Possible Solution

Keep this as a draft implementation candidate, preserve or migrate the `deliverySucceeded` cleanup contract, consume delivery outcome cancellation metadata before treating empty results as failures, and add real after-fix proof from patched OpenClaw.

## Reproduction Assessment

Yes, at source level: current main can call `deliverOutboundPayloads` and return only `deliverySucceeded` without exposing structured status or a zero-result diagnostic. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No. The direction is maintainable, but this patch is not mergeable as-is because it removes a caller-visible cleanup signal and intentionally treats hook-cancelled sends as delivery failures until the related outcome metadata lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the deliverySucceeded cleanup signal:** `src/agents/command/delivery.ts:458-462`
  - body: `agent-command.ts` still clears `pendingFinalDelivery` only when `deliveryResult?.deliverySucceeded === true`, but this return object no longer includes that field. Successful main-session deliveries would leave the durable pending final reply set, so restart/recovery paths can replay a reply that already reached the channel; keep the compatibility flag or update the caller and regression coverage together.
  - confidence: 0.93
- **[P2] Treat hook cancellations as non-failures:** `src/agents/command/delivery.ts:416`
  - body: `deliverySucceeded = results.length > 0` reports every empty result set as failed delivery, but current `deliverOutboundPayloads` intentionally returns no result when `message_sending` hooks cancel all payloads. This will create false failure status and warning logs for expected policy suppression, so consume the DeliveryOutcome cancellation signal or keep this PR sequenced behind that work.
  - confidence: 0.88
