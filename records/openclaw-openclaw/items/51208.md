## Label Justifications

- `P2`: The issue is a concrete hook/Codex session-continuity defect with real workflow impact, but it is limited to webhook-driven agent sessions and needs product/security review.
- `impact:session-state`: Stable-key hook runs lose CLI thread and transcript continuity because each event is forced into a fresh isolated session.
- `impact:data-loss`: Prior hook-run context becomes unavailable to later runs despite persisted session metadata, effectively losing accumulated task history for the agent.

## Summary

Keep open: current main and the latest release still force mapped hook-agent runs through isolated cron sessions, so a stored Codex CLI session id is suppressed on later stable-key hook events. Open implementation candidates exist, including a closing fork PR and the explicit session-mode PR, but none has landed.

## What This Changes

This issue requests that stable-key mapped hook-agent runs with `codex-cli` reuse persisted `cliSessionIds.codex-cli` and invoke `codex exec resume` on subsequent hook events.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that keeps isolation as the default, preserves hook session-key policy gates, and proves repeated stable-key hook reuse with runtime evidence.

## Reproduction Assessment

Yes, source-level. Mapped hook dispatch preserves the stable key but hard-codes `sessionTarget: "isolated"`, the cron runner turns that into `forceNew: true`, and the CLI executor skips stored CLI session ids when `isNewSession` is true.

## Solution Assessment

Unclear as a final product choice. Explicit opt-in `sessionMode: "isolated" | "persistent"` is the safest evidenced direction, while automatic stable-key reuse needs stronger compatibility and security approval.
