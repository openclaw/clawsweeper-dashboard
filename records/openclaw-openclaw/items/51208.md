## Label Justifications

- `P2`: This is a concrete hook/Codex session-continuity defect with real workflow impact, but it is limited to webhook-driven agent sessions and needs product/security review.
- `impact:session-state`: Stable-key hook runs lose CLI thread and transcript continuity because each event is forced into a fresh isolated session.
- `impact:data-loss`: Prior hook-run context becomes unavailable to later runs despite persisted session metadata, effectively losing accumulated task history for the agent.

## Summary

Keep open: current main and the latest release still route stable-key hook-agent calls through isolated cron sessions, which suppresses stored Codex CLI resume IDs. Open implementation candidates exist, but none has landed, and the safe fix still needs explicit product/security handling around persistent hook sessions.

## What This Changes

The issue asks for repeated hook-mapped agent runs with a stable session key to resume the persisted Codex CLI thread instead of starting a fresh `codex exec` session.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that keeps isolation as the default, preserves hook session-key policy gates, and proves repeated stable-key hook reuse in a real runtime path.

## Reproduction Assessment

Yes, source-level. Hook dispatch hard-codes `sessionTarget: "isolated"`, the cron runner turns that into `forceNew`, and the CLI executor skips stored CLI session ids when `isNewSession` is true.

## Solution Assessment

Unclear as a final product choice. Explicit opt-in `sessionMode: "isolated" | "persistent"` is the safest evidenced direction, while automatic stable-key reuse needs stronger compatibility and security approval.
