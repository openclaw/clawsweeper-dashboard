## Summary

Keep open. Current main still hardcodes hook-dispatched agent work as isolated, so stable mapped hook session keys still force a fresh cron session and suppress stored CLI session IDs; the open fork PR with closing syntax should be reviewed before any cleanup close.

## What This Changes

The issue requests stable hook-mapped agent sessions to resume stored codex-cli thread IDs on later webhook events instead of starting fresh codex exec runs.

## Best Possible Solution

Review and land a narrow explicit hook continuation control, with config/schema/docs/tests, while keeping isolated cron jobs fresh by default.

## Reproduction Assessment

Yes. The issue includes live repro steps, and current main has a deterministic source path: mapped hooks pass a stable sessionKey but dispatch an isolated job, force a new cron session, and suppress stored CLI session IDs.

## Solution Assessment

Yes. The maintainable fix is an explicit mapped-hook continuation setting rather than relaxing all isolated job behavior; the linked open fork PR should be reviewed against that boundary.
