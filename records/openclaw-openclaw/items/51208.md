## Summary

Keep open: current main still sends hook agent runs through the isolated cron path, which forces a new session and suppresses CLI resume, and the issue is paired with an open cross-repo fix PR that has not merged.

## What This Changes

The issue requests hook-mapped agent runs with stable session keys to reuse persisted codex-cli CLI sessions on later hook events.

## Best Possible Solution

Review, land, or replace the linked fix with a bounded hook-session continuation change that preserves isolated defaults and gates any external control over persistent session targets.

## Reproduction Assessment

Yes, at source level. Current main hardcodes hook agent jobs as isolated, turns that into forceNew, and then suppresses the stored CLI session id before CLI execution; the issue also includes live argv evidence for the same path.

## Solution Assessment

Unclear. Letting config-owned mapped hooks opt into persistent session targets is a plausible narrow fix, but the implementation should avoid unrestricted direct webhook payload control over main or persistent sessions.
