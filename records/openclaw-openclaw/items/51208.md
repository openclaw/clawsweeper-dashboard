## Summary

Keep this open. Current main still routes hook-dispatched agent work through an isolated cron job, and the isolated cron executor still suppresses stored CLI session IDs whenever the resolved cron session is new. The provided discussion and related open fork PR point to a real core/gateway continuation bug, not a ClawHub or stale-support item.

## What This Changes

The issue asks for mapped `/hooks/<name>` agent runs with a stable `sessionKey` to reuse persisted `cliSessionIds.codex-cli` and invoke Codex CLI resume on later hook events.

## Best Possible Solution

The desired end state is an opt-in mapped-hook continuation control, likely `hooks.mappings[].sessionTarget`, plumbed through config types/schema/help, mapping normalization, transform overrides, replay scoping, and gateway dispatch while keeping isolated cron jobs fresh by default.

## Reproduction Assessment

Yes. The issue includes concrete live steps and current main has a deterministic source-level reproduction: mapped hooks can pass a stable `sessionKey`, but dispatch hardcodes `sessionTarget: "isolated"`, which forces `isNewSession=true` and suppresses CLI resume.

## Solution Assessment

Yes, the narrow maintainable direction is an explicit hook continuation setting rather than relaxing all isolated job behavior. The open implementation candidate should be reviewed against the session-isolation invariants before this issue is closed.
