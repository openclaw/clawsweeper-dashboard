## Summary

Keep open: current main still hardcodes hook-dispatched agent jobs as isolated, which forces a fresh cron session and prevents the CLI executor from reading the persisted Codex CLI session id. There is also an open linked implementation PR, so this issue should remain open until that PR or a replacement lands.

## What This Changes

The issue reports that mapped `/hooks/<name>` agent runs with a stable `sessionKey` store `cliSessionIds.codex-cli` but still launch fresh Codex CLI executions instead of resuming the existing thread.

## Best Possible Solution

Land a narrow hook session-continuation fix with regression coverage, preserving isolated defaults and adding an explicit policy boundary for externally supplied session targets, then close this issue after the fix merges.

## Reproduction Assessment

Yes, from source inspection. The mapped hook path can resolve a stable session key, but current main hardcodes the job to `isolated`, which forces `isNewSession=true` and suppresses stored CLI session id reuse.

## Solution Assessment

No, closing now is not the best way to solve the issue. The maintainable direction is to review the linked PR or a replacement that plumbs an explicit hook session target while keeping external hook session access gated.
