## Summary

Keep open: current main still routes hook-dispatched agent jobs through an isolated cron session, so the persisted Codex CLI session id is not read on later hook events. There is also an open linked implementation PR that uses closing syntax for this issue, so the issue should stay open until that fix or a replacement lands.

## What This Changes

Hook-mapped agent runs with a stable session key should resume the persisted Codex CLI thread instead of starting a fresh `codex exec` on later hook events.

## Best Possible Solution

Land a narrow hook session-continuation fix with regression coverage, preserving isolated defaults and adding an explicit policy boundary for non-isolated hook targets.

## Reproduction Assessment

Yes, from source inspection. The mapped hook path can resolve a stable session key, but current main hardcodes the job to `isolated`, which forces `isNewSession=true` and suppresses stored CLI session id reuse.

## Solution Assessment

Unclear as a merge-ready solution. The linked `sessionTarget` plumbing is the right general direction, but the external hook policy boundary needs maintainer review before landing.
