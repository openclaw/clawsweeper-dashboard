## Label Justifications

- `P2`: The issue is a concrete hook/Codex session-continuity defect with real workflow impact, but it is limited to webhook-driven agent sessions and needs maintainer product/security review.
- `impact:session-state`: Stable-key hook runs lose CLI thread and transcript continuity because each event is forced into a fresh isolated session.
- `impact:data-loss`: Prior hook-run context becomes unavailable to later runs, so the agent effectively loses accumulated task/session history despite persisted session metadata.

## Summary

Keep open: current main still sends mapped hook-agent runs through an isolated cron session, so the stored Codex CLI session id is suppressed on later events. Open implementation candidates are unmerged, including a closing PR for this issue, so the issue should stay open until one lands or is replaced.

## What This Changes

The issue asks for stable-key hook-mapped agent runs to resume stored `codex-cli` sessions instead of starting fresh `codex exec` sessions.

## Best Possible Solution

Land one scoped hook session-continuation implementation, preferably an explicit `sessionMode: "isolated" | "persistent"` path that keeps isolation as the default and proves stable-key hook reuse with regression coverage.

## Reproduction Assessment

Yes, source-level. Mapped hook dispatch preserves the stable key but hard-codes `sessionTarget: "isolated"`, the cron runner turns that into `forceNew: true`, and the CLI executor skips the stored session id when `isNewSession` is true.

## Solution Assessment

Unclear as a final merge choice. The explicit persistent-session mode is the safest evidenced direction, but maintainers still need to choose it over automatic stable-key reuse and verify the hook security boundary.
