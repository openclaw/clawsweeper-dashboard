## Label Justifications

- `P2`: This is a real hook/Codex session-continuity defect with production workflow impact, but it is bounded to webhook-driven agent sessions and has canonical open implementation work.
- `impact:session-state`: Stable-key hook runs lose CLI thread and transcript continuity because every event is forced into a fresh isolated session.
- `impact:data-loss`: Earlier hook-run context becomes unavailable to later runs despite persisted session metadata, effectively losing accumulated task history for the agent.

## Summary

Close as duplicate/superseded: the Codex CLI resume symptom comes from the same forced-new hook-session root cause already tracked by the canonical persistent hook-session issue and active implementation PRs. Current main and the latest release are still affected, so this should close as a satellite, not as fixed.

## What This Changes

Stable hook-mapped agent events with the same sessionKey should resume the stored codex-cli thread instead of starting a fresh codex exec run.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/11665 as the canonical tracker, reconcile the open implementation PRs, and land an explicit persistent hook-session design that preserves default isolation and session-key gates.

## Reproduction Assessment

Yes. The source path is clear: mapped hook dispatch hard-codes isolated cron execution, the runner passes forceNew, and the CLI executor suppresses stored cliSessionIds when the resulting cron session is new.

## Solution Assessment

Yes for closing this duplicate, not for claiming the bug is fixed. The maintainable implementation should be owned by the canonical persistent hook-session tracker and should use an explicit opt-in mode rather than competing one-off behavior.
