## Summary

Keep open. Current main still lacks the requested cron-owned per-attempt `cron.delivery.sent` telemetry with job/run/agent/account attribution and platform message IDs; the existing outbound diagnostics are generic and do not solve the forensic attribution gap.

## What This Changes

Request to emit one structured `cron.delivery.sent` log line for each outbound cron delivery attempt with cron/job/run attribution, account/channel target, status, and Telegram/Discord message IDs.

## Best Possible Solution

Add a privacy-safe, cron-owned telemetry emission at the durable outbound send boundary for direct cron announcements and cron failure notifications, reusing existing durable send results for platform message IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows the cron dispatch inputs and durable send result message IDs exist, while searches and the outbound send path show no `cron.delivery.sent` or equivalent cron-attributed per-attempt event.

## Solution Assessment

Yes, directionally. Emitting near the durable platform send boundary is the narrow maintainable solution, but maintainers should own the final event schema, retry/error semantics, and privacy contract.
