## Summary

Keep open. Current main still lacks the requested cron-owned per-attempt `cron.delivery.sent` telemetry with job/run/agent/account attribution and platform message IDs; existing diagnostics are generic outbound events and cron run traces, not forensic send attribution.

## What This Changes

Request to emit one structured privacy-safe log event for every outbound cron delivery attempt, including cron job/run identity, agent/account attribution, channel target, platform message ID, and success/error status.

## Best Possible Solution

Define a bounded cron-owned telemetry contract, then emit it from the direct cron announce and failure-notification send boundaries using durable send results for platform message IDs and existing cron/run/account context for attribution.

## Reproduction Assessment

Yes. Source inspection on current main shows the cron dispatch inputs and durable send result IDs exist, while the requested `cron.delivery.sent` event and `deliveryTelemetry` plumbing are absent.

## Solution Assessment

Yes, directionally, but not ready for an automated fix. Emitting near the durable cron send boundary is the narrow maintainable path, while maintainers should decide the public schema, retry semantics, and privacy contract first.
