## Summary

Keep open. Current main still lacks the requested cron-owned `cron.delivery.sent` structured log record; the existing cron run trace and generic outbound hooks do not provide one privacy-safe per-attempt log tying a platform message ID back to cron job/run attribution.

## What This Changes

Request to emit a privacy-safe structured `cron.delivery.sent` log line for every outbound cron delivery attempt with cron job/run IDs, agent/account IDs, target, platform message ID, and status.

## Best Possible Solution

Add cron-owned telemetry at the isolated direct-delivery boundary: carry `{ cronJobId, cronRunId, agentId }` plus resolved `{ accountId, channel, to }`, emit exactly one structured success/error record per outbound attempt after the channel send path, map returned platform message IDs into channel-specific fields, and keep message bodies, prompts, tokens, and other sensitive content out of the log.

## Reproduction Assessment

Yes. The missing behavior is reproducible by code inspection or a focused unit test around `dispatchCronDelivery`: current main calls `deliverOutboundPayloads` and records delivered state, but no logger receives `cron.delivery.sent` with cron job/run IDs and returned message IDs.

## Solution Assessment

Yes. A cron-owned telemetry wrapper at the direct outbound send boundary is the narrowest maintainable fix because the generic message hooks lack cron job/run attribution and intentionally include message content for hook consumers.
