## Summary

This is a useful optional cross-Gateway orchestration idea, but current OpenClaw policy and extension surfaces point it to a ClawHub plugin rather than a bundled core feature.

## What This Changes

Request support for asynchronous agent-to-agent messaging across separate Gateway instances, including remote routing, persistent inboxes, or remote agentTurn triggering.

## Best Possible Solution

Keep OpenClaw core focused on the existing Gateway, App SDK, and plugin seams; publish the remote inbox/router as a ClawHub plugin that owns remote Gateway credentials, routing IDs, ACK/TTL behavior, persistence, and audit UX.

## Reproduction Assessment

Not applicable: this is a new cross-Gateway communication feature request, not a broken existing behavior. The issue itself distinguishes existing same-Gateway multi-agent communication from the requested remote Gateway capability.

## Solution Assessment

Yes for cleanup: the narrow maintainable path is a ClawHub plugin because the App SDK already targets explicit remote Gateways and the Plugin SDK can expose tools, services, routes, and discovery hooks. Core should only reopen this direction if plugin authors identify a concrete missing generic API.
