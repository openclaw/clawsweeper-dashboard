## Summary

Keep open: current main still records the cron timeout as an error and can recover tasks only from cron run logs or stored cron state, with no declared external-artifact reconciliation path or canonical merged fix for the reported side-effect-then-timeout case.

## What This Changes

The issue requests cron support for post-timeout artifact reconciliation or clearer diagnostics when a scheduled Codex turn performs side effects before losing its final assistant artifact.

## Best Possible Solution

Define an opt-in cron completion-artifact contract that can verify declared external artifacts after a timeout, record verified partial/success outcomes without hiding real failures, and surface terminal-idle timeout diagnostics separately from pre-side-effect timeouts.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the current-main path where cron timeout/error states are recorded without any external artifact verifier that could reconcile an already-created GitHub PR.

## Solution Assessment

Unclear: the reconciliation hook is plausible, but the durable artifact declaration, trust model, and state semantics need maintainer design. A safe direction is to improve terminal-idle diagnostics first, then add an explicit opt-in artifact verifier contract.
