## Summary

Keep open: current main still lacks a declared external-artifact reconciliation path for scheduled jobs. The Codex app-server path now has better side-effect-aware timeout messaging, and https://github.com/openclaw/openclaw/pull/85107 covers adjacent missing-completion recovery, but cron still records the run as an error and alerts without checking whether the declared completion artifact already exists.

## What This Changes

The issue requests scheduled-agent timeout handling that can distinguish or reconcile side effects already performed before final assistant artifact delivery.

## Best Possible Solution

Define an opt-in generic cron completion-artifact contract that can verify declared artifacts after timeout, record verified partial/success outcomes without hiding real failures, and keep service-specific GitHub checks outside core unless exposed through a safe plugin/runtime seam.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection does show the current-main path where cron timeout/error states are recorded without any external artifact verifier that could reconcile an already-created pull request.

## Solution Assessment

Unclear: the reconciliation hook is plausible, but durable artifact declaration, trust/idempotency rules, and cron state semantics need maintainer design. A safer first slice is the narrower terminal-idle diagnostic work tracked at https://github.com/openclaw/openclaw/issues/85242.
