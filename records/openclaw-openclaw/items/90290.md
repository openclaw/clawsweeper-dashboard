## Summary

Keep this PR open, but it is not merge-ready: the branch conflicts with the current `tool.loop` diagnostic level contract and is unlikely to quiet the real isolated-cron path because production cron run keys are agent-scoped, not raw `cron:` prefixes.

## What This Changes

The PR threads trigger/sessionKey context into agent loop detection and downgrades loop warning logs to `info` when a session key starts with `cron:`.

## Best Possible Solution

Thread an explicit isolated-cron/session-target signal into loop-warning logging, keep `tool.loop` diagnostic typing coherent, and prove the result with redacted real gateway output for isolated and non-isolated cron sessions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible for the PR defects: current code shows isolated cron run keys become `agent:<id>:cron:<job>:run:<sessionId>`, so a raw `startsWith("cron:")` branch misses production; the original log flood still needs live gateway-log proof.

## Solution Assessment

Is this the best way to solve the issue? No: the maintainable fix should use explicit isolated-cron/session-target metadata or the existing scoped cron-key helper, not infer from a raw prefix that production no longer exposes.
