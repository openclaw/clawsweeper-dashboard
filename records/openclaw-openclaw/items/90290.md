## Summary

Keep open: the PR addresses a real linked log-noise problem, but it is not merge-ready because a second warn-level hook log remains and the new cron check is broader than the isolated-session boundary described in the report.

## What This Changes

This PR passes the run trigger into before-tool-call loop detection and logs three cron-triggered loop-warning detector messages at info instead of warn.

## Best Possible Solution

Centralize the loop-warning log-level decision so only fresh isolated cron warning results become info, while critical and non-isolated warnings stay warn/error, then add focused coverage and redacted gateway-log proof.

## Reproduction Assessment

Yes from source inspection: warning-level loop results still flow through runBeforeToolCallHook's log.warn path, and cron runtime has non-isolated session targets. I did not run a live cron job in this read-only review.

## Solution Assessment

No. The patch is a plausible partial mitigation, but the maintainable fix should carry explicit fresh-isolated cron context and make one context-aware warning emission decision for the actual file-log path.
