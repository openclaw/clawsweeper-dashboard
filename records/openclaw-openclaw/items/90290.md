## Summary

Keep open; this PR is not merge-ready because it downgrades only the detector's direct logs, leaves the before-tool-call wrapper and diagnostic loop event at warning level, broadens the downgrade to every cron-triggered run instead of fresh isolated cron sessions, and has no real after-fix cron log proof.

## What This Changes

The branch adds a trigger field to HookContext, passes it into detectToolCallLoop, changes three detector warning logs to info when trigger is cron, and adds a focused unit test.

## Best Possible Solution

Thread an explicit isolated/fresh cron severity mode through the hook context, detector, wrapper logging, and diagnostic event logging; keep warnings for main/custom cron sessions, add wrapper-level regression coverage, and include real redacted cron log proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the PR defects: warning-level loop results still flow through runBeforeToolCallHook's warn branch, and cron trigger is separate from session isolation. I did not run a live cron job in this read-only review.

## Solution Assessment

No. This is a plausible partial fix, but the best fix needs an isolation-aware severity decision carried through the wrapper and diagnostic logging paths, not trigger-only branching inside the detector.
