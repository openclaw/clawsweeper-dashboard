## Summary

Keep open. Current main has partially addressed adjacent Discord discovery behavior, but Telegram embedded prompt-prep still passes raw config into account/token resolution where non-env SecretRefs can throw before model output, and the open closing PR has not merged.

## What This Changes

The issue reports embedded Telegram/Discord reply runs crashing before model output when prompt/discovery code reads unresolved SecretRef-backed channel credentials from raw config.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75445 with a fix that makes optional Telegram/Discord prompt and discovery paths use resolved runtime data or fail-soft inspection while preserving strict credential failures for real send/probe paths.

## Reproduction Assessment

Yes, at source level. Configure Telegram with a non-env SecretRef botToken and trigger an embedded reply run; current main still routes prompt capabilities and reaction guidance from raw params.config into strict Telegram token resolution before model output.

## Solution Assessment

Yes for the direction, but it is not completed on main. Optional prompt/discovery should degrade safely or use the active runtime snapshot; the linked PR is the current implementation path and should be reviewed, rebased, or replaced before closing this issue.
