## Summary

Keep open. The report is concrete, current main has not changed the inspected reply/discovery files since the reported v2026.4.29 release, and source still shows embedded prompt-discovery paths that can call Telegram credential resolution through prompt/reaction helpers if raw channel config reaches them.

## What This Changes

The issue reports that embedded Telegram/Discord reply runs can crash before model output when channel credentials are configured as non-env SecretRefs, even though channel transport itself works.

## Best Possible Solution

Harden the embedded reply prompt-discovery boundary so Telegram/Discord prompt capabilities, reaction guidance, message hints, and action discovery either use the resolved active runtime snapshot or degrade credential-only discovery failures without aborting the agent run, while preserving strict SecretRef behavior for real sends/startup/reload.

## Reproduction Assessment

Yes. The issue gives a concrete operator path and sanitized logs, and source inspection confirms that raw non-env SecretRefs still throw in Telegram/Discord credential helpers while embedded prompt discovery can call those helpers if handed raw config.

## Solution Assessment

No. The existing caller-side runtime/scoped resolution is useful but not sufficient evidence for closure; the safer maintainable fix is to make the embedded discovery boundary itself runtime-snapshot-aware or failure-tolerant for credential-only discovery.
