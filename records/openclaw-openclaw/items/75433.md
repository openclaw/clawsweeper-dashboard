## Summary

Keep open. The report is concrete, current `main` still has embedded prompt-discovery paths that can read channel config during Telegram/Discord capability discovery, and an open closing PR (#75445) is already paired with this issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

Embedded Telegram/Discord reply runs should avoid crashing when channel credentials are configured with non-env SecretRefs that are resolved in the active gateway runtime snapshot.

## Best Possible Solution

Review #75445 as the implementation candidate, and require the final fix to make embedded prompt/action discovery use the resolved runtime snapshot or degrade optional credential-only discovery without weakening strict SecretRef handling on real transport/send paths.

## Reproduction Assessment

Yes. The issue provides concrete steps and sanitized logs, and current source inspection confirms raw non-env SecretRefs can still throw if they reach Telegram/Discord discovery or prompt helpers during embedded prompt preparation.

## Solution Assessment

Unclear for the open PR until reviewed end to end. The maintainable solution is narrow: keep runtime SecretRef resolution strict for transport, but make optional embedded discovery runtime-snapshot-aware or failure-tolerant.
