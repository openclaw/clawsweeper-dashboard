## Summary

Keep open. Current main intentionally keeps auto fallback overrides selected until reset, and the merged fix at https://github.com/openclaw/openclaw/pull/82363 addresses recovered fallback provenance rather than periodic return to the configured primary.

## What This Changes

Requests that auto fallback-pinned sessions retry or return to the configured primary instead of using the persisted fallback indefinitely.

## Best Possible Solution

Define auto fallback pins as soft runtime state with a throttled configured-primary reconciliation path, while preserving strict user-selected model overrides and documenting reset/status behavior.

## Reproduction Assessment

Yes, for the current behavior: source and tests show auto-failover overrides persist across turns until reset and are applied before the configured primary. I did not run a live Slack/provider reproduction.

## Solution Assessment

Unclear as an automatic fix: current docs intentionally avoid probing a known-bad primary every message, so the right solution needs a maintainer decision on throttled reconciliation rather than a blanket clear of fallback pins.
