## Summary

Keep open. Current main and the latest release still derive the WebChat New messages button directly from scroll state, while the merged auto-scroll selector intentionally preserves that catch-up affordance; the remaining request is a separate banner/indicator preference that needs maintainer UX/config direction.

## What This Changes

Request to add a WebChat setting or configuration path to disable, replace, or make the New messages banner less intrusive.

## Best Possible Solution

Add a maintainer-approved browser-local WebChat new-message indicator mode that preserves an accessible jump-to-latest affordance and avoids retired WebChat config unless maintainers explicitly choose that surface.

## Reproduction Assessment

Yes, at source level: current main sets chatNewMessagesBelow when chat does not auto-stick, renders the New messages button from that state, and has no indicator preference in persisted UI settings or WebChat docs.

## Solution Assessment

No for the proposed openclaw.json WebChat shape because that config surface is retired. A browser-local persisted indicator setting is the narrower likely direction, but maintainers first need to choose the indicator and catch-up semantics.
