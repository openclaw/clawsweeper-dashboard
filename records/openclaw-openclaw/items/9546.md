## Summary

Current main still treats the New messages indicator as an independent manual catch-up affordance, and no open canonical issue or PR supersedes this request. The feature is bounded but needs a maintainer UX decision because hiding the indicator can leave users with auto-scroll disabled without a clear route back to live output.

## What This Changes

Add a persisted WebChat preference to hide, customize, or replace the New messages indicator.

## Best Possible Solution

Define one browser-local persisted indicator-style preference and ensure every mode retains an accessible manual jump-to-latest action; avoid adding a gateway configuration key for this client-only UX choice.

## Reproduction Assessment

Yes at source level: the scheduler and focused tests show that new content raises the indicator whenever output is not followed, including with the shipped auto-scroll `off` mode. This review did not run a live browser scenario.

## Solution Assessment

No, the suggested `openclaw.json` configuration is broader than necessary. The existing persisted Control UI settings path is the narrowest maintainable boundary, but maintainers must first decide the hidden-mode catch-up UX.
