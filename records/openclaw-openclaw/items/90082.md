## Label Justifications

- `P1`: The issue reports a current-release hidden-context pollution path that can mislead the main model across repeated user turns when Active Memory is enabled.
- `impact:session-state`: The affected behavior writes misleading Active Memory context into the main session prompt path and can persist across subsequent prompt builds.
- `impact:auth-provider`: The report also concerns model choice/fallback behavior when the active-memory model returns empty or transiently fails.

## Summary

Keep open: current main has documented the breaker keys and uses a 60s cooldown, but the central source path can still promote the embedded agent's generic error payload into Active Memory's hidden prompt prefix, and the default timeout threshold remains 3.

## What This Changes

The issue requests Active Memory to use less aggressive circuit-breaker behavior, avoid injecting fallback error text into main-session prompt context, and expose breaker diagnostics/configuration more clearly.

## Best Possible Solution

Treat embedded error/fallback payloads as failed or no-recall output before prompt-prefix construction, keep a clear diagnostic status, and decide any breaker-default or fallback-model policy change explicitly with focused tests and docs.

## Reproduction Assessment

No live MiniMax run was established, but the source path is reproducible: the embedded runner can emit the exact fallback error text, Active Memory currently accepts payload text as summary, and non-empty summaries are injected into prependContext.

## Solution Assessment

No, not as a single mechanical change: filtering/error-classifying embedded fallback payloads is the narrow bug fix, while raising defaults, adding doctor health state, and runtime fallback-model behavior need maintainer policy decisions.
