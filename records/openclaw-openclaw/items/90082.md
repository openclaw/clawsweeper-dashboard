## Summary

Keep open: current main and v2026.6.8 mitigate the original fallback-prompt injection path and now document the breaker knobs, but the breaker still defaults to 3 timeouts and Active Memory still uses single-model resolution rather than runtime provider failover, so the remaining behavior needs maintainer policy rather than cleanup close.

## What This Changes

The issue requests safer Active Memory circuit-breaker defaults, diagnostics, runtime failover behavior, and prevention of fallback/error text entering the main prompt context.

## Best Possible Solution

Keep this open until maintainers split or approve the remaining Active Memory degradation policy: preserve the no-error-payload prompt-context invariant, choose breaker defaults/diagnostics, and align runtime failover with the canonical provider-failover work.

## Reproduction Assessment

No high-confidence live reproduction was run. The breaker default and single-model Active Memory failover semantics are source-reproducible on current main, while the exact current fallback-string pollution path appears partially mitigated and still needs live degraded-provider proof.

## Solution Assessment

Unclear as a single implementation: filtering embedded error payloads is the narrow bug invariant, but breaker thresholds, diagnostics, and runtime model failover need maintainer-approved policy. A safer path is to keep this issue open and coordinate the failover portion with https://github.com/openclaw/openclaw/issues/79611.
