## Summary

Keep open: current main and v2026.6.8 have fixed the schema/docs and open-breaker no-injection parts, but the default breaker threshold remains 3 and fallback-text filtering/runtime failover policy still need maintainer judgment.

## What This Changes

The issue reports that Active Memory's timeout circuit breaker and embedded fallback handling can cause repeated recall skips and prompt-context pollution after provider or empty-response failures.

## Best Possible Solution

Keep this issue open until maintainers choose the Active Memory degradation policy, add a regression guard that generic embedded-agent fallback text never becomes recalled context, and coordinate runtime failover or latency telemetry with the open fallback issue.

## Reproduction Assessment

Yes at source level for the remaining gaps: the breaker threshold is still 3, and accepted subagent rawReply can become prompt context when memory evidence exists. No live MiniMax/quota-exhaustion run was performed, so the exact field reproduction remains medium confidence.

## Solution Assessment

Unclear as one bundled fix. Filtering generic fallback payloads is a narrow bug fix, but changing breaker defaults, surfacing health, and adding runtime failover need maintainer policy and should align with the related fallback issue.
