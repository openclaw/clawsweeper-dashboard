## Summary

Keep open: current main and v2026.6.8 fixed part of the report by exposing breaker settings and skipping prompt injection when the breaker is open, but the default threshold remains low and the fallback-text/runtime-failover policy still needs maintainer judgment.

## What This Changes

The issue reports that Active Memory opens its circuit breaker too easily and can inject generic fallback text into main-session prompt context, with missing diagnostics and breaker-setting visibility.

## Best Possible Solution

Keep this issue open until maintainers choose the Active Memory degradation policy, add a regression guard for generic embedded-agent fallback text becoming recalled context, and coordinate runtime failover or latency telemetry with the broader fallback issue.

Yes at source level for the remaining gaps: current main still defaults the breaker threshold to 3, and accepted subagent raw replies can still become prompt context when memory evidence exists. No live MiniMax/OAuth quota-exhaustion reproduction was run in this read-only review.

No as a single automated fix. Filtering generic fallback payloads is a narrow bug fix, but changing breaker defaults, surfacing health, and adding runtime failover need maintainer policy and should align with the related fallback issue.
