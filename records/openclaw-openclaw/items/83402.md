## Summary

Keep open. The report is credible and not fully implemented on current main: the Xiaomi provider-local empty-array-items compat exists, but a custom `xiaomi-coding` provider configured as `openai-completions` can still miss the Xiaomi plugin/model compat before cron embedded tools are normalized.

## What This Changes

The issue reports Xiaomi MiMo token-plan custom-provider cron embedded runs failing with `400 Param Incorrect` schema rejection while normal interactive runs still work.

## Best Possible Solution

Make Xiaomi-native custom OpenAI-completions routes resolve the same provider-owned model/tool-schema compatibility as the bundled `xiaomi` provider before embedded-run tool normalization, with focused regression coverage and no cron-only workaround.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The reporter supplied concrete cron failure logs, and current source shows a plausible custom-provider path where Xiaomi schema compat is not applied before tool normalization.

## Solution Assessment

Yes. The maintainable fix is to resolve Xiaomi-native custom-provider compatibility at the provider/model layer before tool schemas are built, rather than adding cron-specific stripping or a public config knob.
