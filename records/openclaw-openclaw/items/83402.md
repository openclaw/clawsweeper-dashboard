## Summary

Keep open. Current main fixes the supported Xiaomi and Xiaomi Token Plan provider paths, but the reported `xiaomi-coding` custom provider can still miss Xiaomi-owned `omitEmptyArrayItems` model compatibility before embedded cron tools are normalized.

## What This Changes

The issue requests MiMo `mimo-v2.5-pro` cron embedded runs through a custom `xiaomi-coding` OpenAI-completions provider to submit MiMo-compatible tool schemas instead of failing with `400 Param Incorrect`.

## Best Possible Solution

Decide whether custom Xiaomi aliases like `xiaomi-coding` remain supported; then either migrate/diagnose them to canonical `xiaomi-token-plan` or make known Xiaomi endpoints receive the same model compat as the bundled provider with cron regression coverage.

## Reproduction Assessment

Yes, source-reproducible: reporter logs show the failing cron path, and current source still lets a custom `xiaomi-coding` core OpenAI-completions provider miss Xiaomi `omitEmptyArrayItems` compat before tool normalization.

## Solution Assessment

No, current main is not enough for the exact reported path: it fixes canonical Xiaomi/Token Plan provider behavior, but maintainers still need to choose migration versus custom-alias support for known Xiaomi endpoints.
