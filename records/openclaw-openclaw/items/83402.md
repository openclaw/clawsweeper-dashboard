## Summary

Keep open. The report has concrete cron logs, and current main still only applies Xiaomi's empty-array-items tool-schema cleanup on Xiaomi-owned resolved models, so custom Token Plan provider ids can miss `omitEmptyArrayItems` before embedded cron tools are normalized.

## What This Changes

Cron embedded agent runs using a custom Xiaomi MiMo Token Plan OpenAI-completions provider reject tool schemas with `400 Param Incorrect` while normal sessions still work.

## Best Possible Solution

Extend Xiaomi-owned provider/model compatibility so custom MiMo Token Plan OpenAI-completions routes get the same empty-array-items tool-schema cleanup before cron embedded tools are built, with focused regression coverage for custom provider ids and Token Plan hosts.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83402. Current main applies `compat.omitEmptyArrayItems` for the bundled `xiaomi` provider, but custom Xiaomi Token Plan providers such as `xiaomi-coding` or `xiaomi-token` with `api: "openai-completions"` and `token-plan-*.xiaomimimo.com` can miss that compat before cron embedded tools are normalized. Add a provider-owned/shared compat path so MiMo model ids on known Xiaomi-native Token Plan endpoints receive the empty-array-items cleanup without adding a new config option and without special-casing cron. Keep typed `items` intact, do not strip schemas for unrelated providers, and do not try to solve the separate single-turn visible-output issue in https://github.com/openclaw/openclaw/issues/83467 unless the same regression test proves it. Add focused regression coverage for custom provider ids, Token Plan host detection, and embedded/runtime tool schema normalization; add a changelog entry because this is user-visible.
