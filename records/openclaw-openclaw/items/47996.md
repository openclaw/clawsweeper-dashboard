## Summary

Keep open: the useful stale-expiry idea is not implemented on current main, but this PR branch is not mergeable as-is because the new direct outbound dedupe crosses caller/recovery boundaries and the recovery expiry is stale relative to current queue-claim contracts.

## What This Changes

The PR adds 10-minute stale queued-outbound expiry during recovery, a 1.5s direct reply/mirror outbound dedupe cache, targeted outbound tests, one setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after claiming and re-reading pending entries, preserves all current queued metadata, and treats direct-reply dedupe as a separate design only if every per-call semantic boundary is preserved.

## Reproduction Assessment

Yes. Source inspection of the PR head gives high-confidence reproduction paths: two skipQueue recovery replays with the same fingerprint can coalesce, and two reply-shaped sends that differ only by session/scopes/callbacks can reuse one result; I did not run tests because this was a read-only review.

## Solution Assessment

No. The stale-expiry idea is useful, but this branch is not the best fix because it is stale relative to current recovery ownership/metadata contracts and the direct dedupe design changes security- and session-sensitive delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key includes target/payload/mirror fields but not session, gatewayClientScopes, abortSignal, or per-call callbacks. A cache or inflight hit can return another request's result without running deliverOutboundPayloadsCore, skipping session-scoped media roots, hooks, authorization-sensitive side effects, and callback delivery for a distinct send.
  - confidence: 0.88
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:400-418`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue true for each stored entry. Since the key only records the skipQueue boolean and not the queued entry identity, two legitimate queued reply-shaped entries with the same target and payload can coalesce inside the 1.5s window, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The recovery loop acts on the startup snapshot directly; current main re-reads each queue file under an in-process claim before replaying or failing it. Without that guard, startup recovery can race reconnect/live delivery drains and move or ack an entry after another path has taken ownership.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main recovery replay passes stored replyToMode, renderedBatchPlan, formatting, session, queue id/state dir, gateway scopes, and deferred commit hooks through to delivery. This branch's recovery params omit several of those fields, so recovered messages can route, hook, scope media, or commit differently from the original send.
  - confidence: 0.84
