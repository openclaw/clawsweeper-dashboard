## Summary

Keep open, but do not merge as-is. Current main still lacks the stale-age expiry requested by the PR, so this is not implemented-on-main; however the PR head introduces unsafe outbound coalescing, is stale against newer recovery ownership protections, is merge-conflicted, and includes unrelated Control UI churn.

## What This Changes

The PR branch adds 10-minute queued-outbound recovery expiry, a 1.5-second direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting changes.

## Best Possible Solution

Prepare a narrow outbound-only replacement or revision that adds age-based recovery expiry after claiming and re-reading the current pending entry, preserves current queued replay metadata and reconnect-drain safeguards, and leaves direct-reply dedupe for a separate design only if it can preserve per-call delivery semantics and queue identity.

## Reproduction Assessment

Yes. There are high-confidence source-level reproduction paths: replay two identical queued reply-shaped entries with skipQueue during recovery to hit the dedupe cache, and let startup recovery act on a stale pending snapshot after reconnect/live drain has claimed or acked the same entry.

## Solution Assessment

No. The stale-expiry idea is plausible, but this branch is not the best solution because it combines recovery expiry with security-sensitive direct-send caching, omits current-main ownership and replay metadata protections, and carries unrelated UI churn.

  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and per-call callbacks even though delivery uses them for media roots, adapter scopes, cancellation, hooks, and callback delivery.   - body: Recovery replays call deliverOutboundPayloads with skipQueue: true for each stored entry. Since the key records skipQueue but not queue entry identity, two legitimate queued replies with the same target and payload can coalesce, letting the later entry be acked without a real send.
  - confidence: 0.9

  - body: The recovery loop acts on the startup snapshot directly; current main re-reads each queue file under an in-process claim before replaying or failing it. Without that guard, startup recovery can race reconnect/live delivery drains and move or ack an entry after another path has taken ownership.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main's recovery replay passes stored replyToMode and session through to delivery. This branch's buildRecoveryDeliverParams omits both, so recovered messages can route, hook, or scope media differently from the original queued send.

## Security Review

Status: needs_attention

Summary: The diff introduces outbound coalescing and recovery movement that do not preserve all caller/session and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits session and gatewayClientScopes while delivery passes both into media access scoping, adapters, hooks, and authorization-sensitive side effects. A cache or inflight hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:400`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path without queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than the claimed and re-read queue entry, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
