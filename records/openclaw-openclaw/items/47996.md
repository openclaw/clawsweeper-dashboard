## Summary

Keep open: current main does not implement the requested stale-expiry or direct-reply dedupe behavior, and the PR branch introduces correctness and security regressions against newer outbound recovery safeguards.

## What This Changes

The PR adds 10-minute stale queued delivery expiry, a 1.5s direct reply/mirror dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting changes.

## Best Possible Solution

Prepare a narrow outbound-only revision that expires old entries only after claim-and-re-read, preserves queued metadata, and splits direct-reply dedupe into a separate design with proven per-call isolation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: replay two queued reply-shaped entries with identical payloads during recovery, or issue two matching reply sends with different session/scope/abort/callback context inside the 1.5s window.

## Solution Assessment

No. The stale-expiry goal is plausible, but this PR is not the best solution because it combines that policy with unsafe cross-call caching, drops current-main recovery metadata, and includes unrelated UI churn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and per-call callbacks even though delivery uses them for media scoping, adapter permissions, hooks, cancellation, and callback delivery. A cache or inflight hit can return another call's result without running this call's delivery semantics.
  - confidence: 0.88
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:644`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue: true for each stored entry. Since the dedupe key records skipQueue but not queue entry identity, two legitimate queued replies with the same target and payload can coalesce and let the later entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-199`
  - body: The stale-expiry branch acts on the startup snapshot directly; current main re-reads each queue file under an active claim before replaying or failing it. Without that guard, startup recovery can race reconnect or live delivery drains and move an entry another path has already owned.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main passes stored replyToMode and session through recovery replay, but this branch's buildRecoveryDeliverParams omits both. Recovered messages can route, hook, or scope media differently from the original queued send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces outbound coalescing and recovery movement that do not preserve all caller, session, scope, and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits session and gatewayClientScopes while delivery passes both into media access scoping, adapters, hooks, and authorization-sensitive side effects. A cache or inflight hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:644`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path without queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than the claimed and re-read queue entry, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
