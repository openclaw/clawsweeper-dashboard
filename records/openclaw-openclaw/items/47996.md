## Summary

Keep open for maintainer review: current main has not implemented the requested stale-expiry/dedupe behavior, but the branch would regress recovery ownership and per-call delivery isolation and lacks real behavior proof.

## What This Changes

The PR adds stale queued-outbound expiry during recovery, direct reply/mirror outbound dedupe, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after claim-and-re-read, preserves all queued metadata and reconnect-drain protections, and splits any direct-reply dedupe into a separate design only if per-call isolation can be proven.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths: replay two queued reply-shaped entries with the same payload during recovery, or issue two matching reply sends with different session, scope, abort, or callback context inside the 1.5s window.

## Solution Assessment

No. The stale-expiry goal may be useful, but this branch combines it with unsafe cross-call caching, stale recovery semantics, missing queued metadata, and unrelated UI churn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and per-call callbacks even though delivery uses those values for media scoping, adapter permissions, hooks, cancellation, and callback delivery. A cache or inflight hit can return another call's result without running this call's delivery semantics.
  - confidence: 0.88
- **[P1] Bypass direct dedupe for recovery replays:** `src/infra/outbound/deliver.ts:644`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue: true for each stored entry. Since the dedupe key records skipQueue but not queue entry identity, two legitimate queued replies with the same target and payload can coalesce and let the later entry be acked without a real send.
  - confidence: 0.9
- **[P2] Claim and re-read before expiring stale entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-199`
  - body: The stale-expiry branch acts on the startup snapshot directly. Current main claims and re-reads each queue file before replaying or failing it, so this branch can race reconnect or live delivery drains and move an entry another path has already owned.
  - confidence: 0.86
- **[P2] Replay stored reply mode and session metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main's recovery replay passes stored replyToMode and session through to delivery. This branch's buildRecoveryDeliverParams omits them, so recovered messages can route, hook, or scope media differently from the original queued send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces outbound coalescing and recovery movement that do not preserve all caller, session, scope, and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits session and gatewayClientScopes while delivery passes both into media access scoping, adapters, hooks, and authorization-sensitive side effects. A cache or inflight hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:644`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path without queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than a claimed and re-read queue entry, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
