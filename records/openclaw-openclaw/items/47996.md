## Summary

Keep open: current main still lacks the requested stale-age expiry and direct-reply dedupe, but this branch remains unsafe to merge because the dedupe can coalesce distinct delivery contexts, recovery expiry bypasses current ownership safeguards, and real behavior proof is still missing.

## What This Changes

The PR adds 10-minute stale queued-outbound expiry during recovery, a 1.5s direct reply/mirror outbound dedupe cache, targeted tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that expires stale recovery entries only after re-reading and claiming pending entries, preserves queued metadata and commit hooks, and treats direct-reply dedupe as a separate context-complete design.

## Reproduction Assessment

Yes, by source inspection: two skipQueue recovery replays with the same reply-shaped payload can coalesce, and direct sends that differ by session, scopes, abort state, or callbacks can reuse the first result. I did not run a live outbound setup, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The branch wraps all delivery with a broad cache and expires stale recovery entries before current-main claim/re-read safeguards; a smaller recovery-only expiry path plus separately reviewed dedupe design is safer.

  - body: The dedupe key includes target/payload/mirror fields but not session, gatewayClientScopes, abortSignal, or per-call callbacks.   - confidence: 0.9

  - body: The stale-expiry check acts on the startup snapshot directly. Current main claims and re-reads each queue file before replaying or failing it, so expiring the old snapshot can race reconnect or live delivery ownership and move an entry another path has taken.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
