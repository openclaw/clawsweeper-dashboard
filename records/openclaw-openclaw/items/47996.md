## Summary

Keep open: the branch still contains blocking outbound delivery regressions and lacks after-fix real behavior proof, so it is not a conservative cleanup close candidate.

## What This Changes

The PR adds 10-minute outbound queue recovery expiry, a 1.5s reply/mirror send dedupe cache, outbound tests, a channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement: add age-based recovery expiry after claim-and-re-read, preserve all queued metadata and reconnect-drain protections, and split direct-reply dedupe into a separate design only if per-call isolation can be proven.

## Reproduction Assessment

Yes, by source inspection: replay two queued reply-shaped entries with the same target and payload during recovery, or issue two matching reply sends with different session, scopes, abort state, or callbacks inside the 1.5s window.

## Solution Assessment

No. The stale-expiry goal is useful, but this branch combines it with unsafe cross-call caching, stale recovery ownership semantics, missing queued metadata, and unrelated UI churn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits `session`, `gatewayClientScopes`, `abortSignal`, and per-call callbacks, but current delivery uses those values for media access, adapter scope, hooks, cancellation, and callback delivery. A cache or inflight hit can return another call's result without running this call's delivery semantics.
  - confidence: 0.9
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:416-418`
  - body: Recovery replays call `deliverOutboundPayloads` with `skipQueue: true` for each stored entry. Since the new key records `skipQueue` but not queue entry identity, two legitimate queued replies with the same target and payload can coalesce and let the second entry be acked without a real send.
  - confidence: 0.91
- **[P2] Claim and reload before expiring recovery entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-199`
  - body: The stale-expiry branch acts on the startup snapshot directly. Current main claims and re-reads each queue file before replaying or failing it, so this can race reconnect or live delivery drains and move an entry after another path owns it.
  - confidence: 0.86
- **[P2] Replay stored delivery metadata during recovery:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: `buildRecoveryDeliverParams` omits metadata current main now persists and replays, including `replyToMode`, formatting, identity/session context, rendered batch plans, and deferred commit-hook queue fields. Recovered messages can therefore route, hook, or scope media differently from the original send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive outbound coalescing and recovery movement that do not preserve caller, scope, session, and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits `session` and `gatewayClientScopes` while delivery uses both for media access, adapter behavior, hooks, and authorization-sensitive side effects. A cache hit can reuse a result produced under another caller or session context.
  - confidence: 0.9
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:416`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path without queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks an ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than a claimed and re-read queue entry, so it can race another drain that already owned or acked the same queue file.
  - confidence: 0.86
