## Summary

Keep this PR open but block merge: source review confirms the branch still conflicts with current outbound recovery ownership contracts, adds a security-sensitive direct-delivery cache with incomplete per-call boundaries, and lacks required real behavior proof.

## What This Changes

The PR adds stale queued-outbound expiry, a short direct reply/mirror outbound dedupe path, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after re-reading and claiming pending entries, preserves all queued metadata and reconnect-drain protections, and treats direct-reply dedupe as a separate design only if it preserves every per-call delivery boundary.

## Reproduction Assessment

Yes, by source inspection: two skipQueue recovery replays with the same reply-shaped payload can coalesce, and two direct sends that differ by session, scopes, abort state, or callbacks can reuse the first result. I did not run a live outbound setup, so the reproduction is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The branch wraps all delivery with a broad cache and expires stale recovery entries before current-main claim/re-read safeguards; the safer fix is a smaller recovery-only expiry path plus a separately reviewed dedupe design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Scope direct dedupe to per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and callback boundaries. A cache or inflight hit can return another request's result without running delivery for its own authorization, media roots, cancellation, hooks, or callback side effects.
  - confidence: 0.9
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:641-644`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue: true, and this wrapper applies the direct dedupe path to those calls too. Two legitimate queued reply-shaped entries with the same target and payload can coalesce, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The stale-expiry check acts on the startup snapshot directly. Current main claims and re-reads each queue file before replaying or failing it, so expiring the old snapshot can race reconnect or live delivery ownership and move an entry another path has taken.
  - confidence: 0.86
- **[P2] Replay stored queue metadata during recovery:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main's recovery replay passes stored replyToMode, rendered batch plan, session, deliveryQueueId/stateDir, and deferred commit hooks into delivery. This branch's buildRecoveryDeliverParams omits several of those fields, so recovered messages can route, hook, scope media, or commit differently from the original queued send.
  - confidence: 0.84
