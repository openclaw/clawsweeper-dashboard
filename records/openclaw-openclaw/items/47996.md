## Summary

Keep this PR open, but it is not mergeable as-is: current main still lacks the requested stale-age expiry, while the branch introduces blocking outbound dedupe/recovery regressions and has no real behavior proof.

## What This Changes

The PR adds 10-minute queued-outbound recovery expiry, a 1.5s direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after current-main claim/re-read safeguards, preserves queued metadata and reconnect-drain protections, and splits direct-reply dedupe into a separate design only if it preserves every per-call delivery boundary.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths: two skipQueue recovery replays with the same reply-shaped payload can coalesce, and two direct sends that differ by session, scopes, abort state, or callbacks can reuse the first result.

## Solution Assessment

No. The current PR is not the best fix because it adds a broad cache around delivery and applies stale expiry on a stale recovery loop instead of current main's claim/re-read queue contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Scope direct dedupe to per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key includes target, payload, mirror, and a few flags, but not session, gatewayClientScopes, abortSignal, or per-call callbacks. A cache or inflight hit can return another request's result without running delivery for its own authorization, media roots, cancellation, hooks, or callback side effects.
  - confidence: 0.9
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:641-644`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue: true, and this wrapper applies the direct dedupe path to those calls too. Since the key records only skipQueue and not the queued entry identity, two legitimate queued reply-shaped entries can coalesce and the second entry can be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The stale-expiry check runs against the startup snapshot directly. Current main claims and re-reads each queue file before replaying or failing it, so expiring the old snapshot can race reconnect or live delivery ownership and move an entry another path has taken.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main's recovery replay carries stored replyToMode, rendered batch plan, session, deliveryQueueId, stateDir, and deferred commit hooks into delivery. This branch's buildRecoveryDeliverParams omits several of those fields, so recovered messages can route, hook, or scope media differently from the original queued send.
  - confidence: 0.84
