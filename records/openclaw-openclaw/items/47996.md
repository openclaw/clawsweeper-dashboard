## Summary

Keep open: the requested reliability work is plausible, but this PR head is not mergeable as-is because it adds outbound dedupe across per-call context boundaries, applies stale recovery expiry on an outdated recovery loop, and lacks real behavior proof.

## What This Changes

The PR adds 10-minute queued-outbound recovery expiry, a 1.5s reply-shaped outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds stale-age expiry after current main's claim and re-read path, preserves queued metadata and commit-hook handling, and defers direct-reply dedupe unless it can preserve every per-call delivery boundary.

## Reproduction Assessment

Yes, source inspection gives high-confidence reproduction paths: two skipQueue recovery replays with the same reply-shaped payload can coalesce, and two direct sends that differ only by session, scopes, abort state, or callbacks can reuse the first result. Focused tests should exercise those exact paths before rework merges.

## Solution Assessment

No. The current PR is not the best fix because it adds a broad cache around delivery and is based on an older recovery loop; the safer solution is a smaller patch on current main's recovery and queue-claim contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Scope direct dedupe to every per-call delivery boundary:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and callbacks even though delivery uses those for media access, hooks, authorization-sensitive adapter behavior, cancellation, and caller notifications. A cache or in-flight hit can therefore return another request's result without executing the second request's required side effects.
  - confidence: 0.9
- **[P1] Bypass direct dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:644`
  - body: Recovery replays call this path with skipQueue: true for each stored queue entry. Since the dedupe key has no queued entry identity, two legitimate queued reply-shaped entries with the same target and payload can coalesce inside the 1.5s window, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve recovery claims before expiring stale entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-199`
  - body: The new stale-expiry check runs against the startup snapshot in a recovery loop that lacks current main's per-entry claim and re-read guard. That can race reconnect or live delivery drains and move an entry to failed after another path has already taken ownership or changed the queue file.
  - confidence: 0.86
- **[P2] Replay stored reply and session metadata during recovery:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: buildRecoveryDeliverParams omits stored replyToMode and session context that current main now persists and replays. Recovered messages can therefore use different reply routing, hook routing, or media policy than the original queued send.
  - confidence: 0.84
