## Summary

Keep open: the stale-recovery idea is not implemented on current main, but this PR head is not mergeable as-is because the direct dedupe can coalesce distinct caller contexts and recovery replays, while the recovery changes are stale relative to current active-claim/re-read safeguards.

## What This Changes

The PR branch adds 10-minute queued-outbound recovery expiry, a 1.5s direct reply/mirror outbound dedupe cache, focused outbound tests, one setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after claim/re-read while preserving current queue metadata and treats direct-reply dedupe as a separate design unless every per-call semantic boundary can be preserved.

## Reproduction Assessment

Yes, source inspection gives high-confidence reproduction paths for the review findings: two skipQueue recovery replays with the same fingerprint can coalesce, and two reply-shaped sends that differ only by session/scopes/callbacks can reuse one result. I did not run tests because this was a read-only review.

## Solution Assessment

No. The stale-expiry part may be useful, but this branch is not the best fix because it is stale relative to current recovery ownership/metadata contracts and the direct dedupe design changes security- and session-sensitive delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe to each caller context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits behavior-changing per-call context such as `session`, `gatewayClientScopes`, `abortSignal`, and callbacks. A cache or in-flight hit can therefore return another caller's result without running the second delivery's media-root scoping, hooks, authorization-sensitive adapter side effects, or callback stream.
  - confidence: 0.9
- **[P1] Bypass direct dedupe for replayed queue entries:** `src/infra/outbound/deliver.ts:404-418`
  - body: Recovery replays call this delivery path with `skipQueue: true`, but the dedupe key only records that boolean and not the queued entry identity. Two legitimate queued reply-shaped entries with the same target and payload can coalesce inside the 1.5s window, causing the second queue entry to be acked without a real send.
  - confidence: 0.9
- **[P2] Claim and re-read queue entries before expiring them:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The new expiry check acts on the startup snapshot directly. Current main claims each entry and re-reads the queue file before replaying or failing it, which prevents racing reconnect/live drains from moving or acking an entry after another path has taken ownership.
  - confidence: 0.86
- **[P2] Replay all stored recovery metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Recovery replay now needs to preserve stored metadata such as `replyToMode`, rendered batch plan, formatting, session, queue id/state dir, and deferred commit hooks. This branch's recovery params omit several of those fields, so recovered messages can route, hook, scope media, or commit differently from the original send.
  - confidence: 0.84

## Security Review

Status: needs_attention
