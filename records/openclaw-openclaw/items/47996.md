## Summary

Keep open: current main does not implement the central stale-expiry or direct-reply dedupe changes, but the PR has blocking source-reproducible outbound delivery and security defects plus missing real behavior proof.

## What This Changes

The branch adds age-based outbound queue recovery expiry, direct reply/mirror outbound dedupe, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds stale recovery expiry after claim/re-read safeguards, preserves all queued replay metadata, and treats direct-reply dedupe as a separate context-complete design.

## Reproduction Assessment

Yes, by source inspection: PR-head direct dedupe runs before queue handling and keys without full session, scopes, abort, queue-entry, and callback context, so legitimate sends can coalesce incorrectly. I did not run a live outbound setup, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. A smaller recovery-expiry fix after current-main claim/re-read protections plus a separately reviewed, context-complete dedupe design is safer than this mixed-scope branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by full delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits session, gatewayClientScopes, abort state, and per-call callback context. A cache or inflight hit can return another request's result without running delivery for its own media roots, scopes, hooks, cancellation, or callbacks.
  - confidence: 0.9
- **[P1] Bypass direct dedupe for recovery replays:** `src/infra/outbound/deliver.ts:641-644`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue true, but the wrapper still applies direct dedupe. Two legitimate queued reply-shaped entries with the same target and payload can coalesce, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The stale-expiry branch acts on the startup snapshot directly. Current main claims and re-reads each queue file before retrying or failing it, so expiring the old snapshot can race reconnect or live delivery ownership.
  - confidence: 0.86
- **[P2] Replay stored recovery metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main replays stored replyToMode, rendered batch plan, session, delivery queue id/state dir, and deferred commit hooks. This branch omits several of those fields, so recovered messages can route, hook, scope media, or commit differently from the original queued send.
  - confidence: 0.84
