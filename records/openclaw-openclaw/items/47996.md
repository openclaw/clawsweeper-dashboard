## Summary

Keep open: current main still lacks the stale-age recovery expiry, but this PR head has blocking outbound dedupe and recovery-race regressions, is dirty against main, and still lacks real behavior proof.

## What This Changes

The branch adds 10-minute queued-outbound expiry during recovery, a 1.5s direct reply/mirror outbound dedupe path, targeted outbound tests, one setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement or rework that adds age-based recovery expiry after claiming and re-reading pending entries, preserves current queued metadata and live-claim protections, and only adds reply dedupe if every per-call semantic boundary is included.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths: two skipQueue recovery replays with the same fingerprint can coalesce, and two reply-shaped sends that differ only by session, scopes, abort state, or callbacks can reuse one result.

## Solution Assessment

No. The stale-expiry idea is useful, but this branch is not the best solution because it is stale relative to current recovery ownership/metadata contracts and the direct dedupe design changes security- and session-sensitive delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key includes target, payload, mirror, and flags but not session, gatewayClientScopes, abortSignal, or per-call callbacks. A cache or inflight hit can return another request's result without running delivery for that caller, skipping session-scoped media roots, hooks, authorization-sensitive side effects, and callback delivery.
  - confidence: 0.88
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:400-418`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue true for each stored entry. Since the key records only the skipQueue boolean and not the queued entry identity, two legitimate queued reply-shaped entries with the same target and payload can coalesce and the second entry can be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:190-198`
  - body: The recovery loop expires the startup snapshot entry directly. Current main re-reads each queue file under an in-process claim before replaying or failing it, so this can race reconnect/live delivery drains and move or ack an entry after another path has taken ownership.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main recovery replay passes stored replyToMode, renderedBatchPlan, formatting, session, gateway scopes, queue id/state dir, and deferred commit hooks through to delivery. This branch's stale recovery base omits several of those fields, so recovered messages can route, hook, scope media, or commit differently from the original send.
  - confidence: 0.84
