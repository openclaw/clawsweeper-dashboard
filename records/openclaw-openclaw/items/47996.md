## Summary

Keep open: current main does not contain the proposed stale-expiry or direct-reply dedupe behavior, but this branch still has correctness and security blockers, lacks real behavior proof, and carries unrelated UI churn.

## What This Changes

The PR adds 10-minute stale outbound recovery expiry, a 1.5s direct reply/mirror dedupe cache, outbound tests, a channel setup type loosening, and broad Control UI formatting changes.

## Best Possible Solution

Prepare a narrow outbound-only replacement that expires stale recovery entries after current-main claim/re-read safeguards, preserves queued metadata and commit hooks, and treats direct-reply dedupe as a separate context-complete design.

## Reproduction Assessment

Yes, by source inspection: PR head applies direct dedupe before queue handling and keys it without session/scopes/abort/callback context, so queued replays and direct sends can coalesce incorrectly. I did not run a live outbound setup, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. A smaller recovery expiry path after current-main claim/re-read protections plus a separately reviewed, context-complete dedupe design is safer than this broad cache and UI-churn branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by full delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits `session`, `gatewayClientScopes`, `abortSignal`, and per-call callbacks. A cache or inflight hit can return another request's result without running delivery for its own media roots, authorization-sensitive scopes, cancellation, hooks, or callback side effects.
  - confidence: 0.9
- **[P1] Bypass direct dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:641-644`
  - body: Recovery replays call `deliverOutboundPayloads` with `skipQueue: true`, but this wrapper still applies direct dedupe. Two legitimate queued reply-shaped entries with the same target and payload can coalesce, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Claim and re-read before expiring stale entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The stale-expiry branch acts on the startup snapshot directly. Current main claims and re-reads each queue file before replaying or failing it, so expiring the old snapshot can race reconnect or live delivery ownership and move an entry another path already took.
  - confidence: 0.86
- **[P2] Replay the stored recovery metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main replays stored `replyToMode`, rendered batch plan, session, delivery queue id/state dir, and deferred commit hooks. This branch's recovery params omit several of those fields, so recovered messages can route, hook, scope media, or commit differently from the original queued send.
  - confidence: 0.84
