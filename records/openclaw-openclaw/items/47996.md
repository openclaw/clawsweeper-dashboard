## Summary

Keep this PR open, but it should not merge as-is: the outbound dedupe cache can merge distinct delivery contexts, recovery replays can be acked without a real send, stale expiry is based on an older recovery loop, and the PR still lacks real behavior proof.

## What This Changes

The PR branch adds 10-minute queued-outbound recovery expiry, a 1.5s direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after re-reading and claiming pending entries, preserves queued metadata and reconnect-drain protections, and splits direct-reply dedupe into a separate design only if it can preserve every per-call delivery boundary.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths: two skipQueue recovery replays with the same reply-shaped payload can coalesce, and two direct sends that differ by session, scopes, abort state, or callbacks can reuse the first result.

## Solution Assessment

No. The current PR is not the best fix because it adds a broad cache around delivery and applies stale expiry on an older recovery loop instead of current main's claim/re-read queue contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Scope direct dedupe to every per-call delivery boundary:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key includes target/payload/mirror fields but not `session`, `gatewayClientScopes`, `abortSignal`, or per-call callbacks. A cache or inflight hit can return another request's result without running `deliverOutboundPayloadsCore`, skipping session-scoped media roots, hooks, authorization-sensitive side effects, cancellation, and callback delivery for a distinct send.
  - confidence: 0.9
- **[P1] Bypass direct dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:641-644`
  - body: Recovery replays call `deliverOutboundPayloads` with `skipQueue: true` for each stored entry. Since the dedupe key records only `skipQueue` and not queued entry identity, two legitimate queued reply-shaped entries with the same target and payload can coalesce inside the 1.5s window, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-199`
  - body: The new stale-expiry check runs against the startup snapshot directly. Current main re-reads each queue file under an in-process claim before replaying or failing it; without that guard, startup recovery can race reconnect/live delivery drains and move an entry after another path has taken ownership.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main's recovery replay passes stored `replyToMode` and `session` through to delivery. This branch omits both in `buildRecoveryDeliverParams`, so recovered messages can route, hook, or scope media differently from the original queued send.
  - confidence: 0.84
