## Summary

Keep open: the scoped rejected-write restart guard is useful, but this PR is now dirty against current main's SQLite restart-sentinel contract and still needs maintainer acceptance of the empty-diff `config.apply` compatibility change before merge.

## What This Changes

Adds a zero-`changedPaths` guard in the gateway config restart helper and regression tests that rejected `config.patch`/`config.apply` calls do not call the SIGUSR1 scheduler or change the config hash.

## Best Possible Solution

Land a rebased guard and tests only if maintainers accept that no-op `config.apply` is not a supported restart trigger, while keeping the startup-window and invalid-on-disk recovery work separate.

## Reproduction Assessment

No. The full downstream zombie-draining incident was not reproduced on current main; source inspection shows rejected RPCs return before commit/restart, while successful empty-diff `config.apply` currently still reaches the restart helper.

## Solution Assessment

Unclear until maintainer signoff and rebase. The shared helper guard is a narrow backstop for rejected/no-op writes, but it also changes successful empty-diff `config.apply` semantics and must be updated for the current `sentinelPersisted` contract.
