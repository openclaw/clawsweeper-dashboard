## Summary

Keep open: the attribution bug is source-reproducible and the related Telegram/operator comments support the need for a focused fix, but this PR is not mergeable as written because the normalized runtime params also feed dynamic tool construction and can change `tools.byProvider` policy matching. The contributor proof is still mock-only because the PR body supplies fake app-server Vitest output rather than a real after-fix run.

## What This Changes

This PR normalizes Codex app-server local runtime attribution for canonical OpenAI model turns to `openai-codex` / `openai-codex-responses` and adds a focused regression test.

## Best Possible Solution

Separate local runtime-attribution metadata from selected model/provider inputs so transcript, trajectory, prompt-report, and context-engine records use the Codex runtime family while dynamic tool policy and routing continue to use the original selected provider; then require real after-fix proof or a maintainer override.

## Reproduction Assessment

Yes for source-level reproduction: current main records Codex app-server transcript and trajectory metadata from the selected `openai` / `openai-responses` params, and the canonical tracker includes operator evidence for the same mismatch. I did not run a live current-main Telegram/OAuth session in this read-only review.

## Solution Assessment

No as written: the attribution direction is narrow, but the implementation should not pass rewritten attribution params into tool-policy-sensitive construction. A safer fix is to pass separate attribution metadata only to record/report surfaces.
