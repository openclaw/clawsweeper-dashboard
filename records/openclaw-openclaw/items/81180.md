## Summary

Keep open: the attribution bug is source-reproducible and the PR targets the right Codex app-server surface, but the current patch also changes dynamic tool provider-policy inputs and still lacks external real behavior proof.

## What This Changes

The PR normalizes Codex app-server transcript, trajectory, prompt-report, context-engine, thread, and dynamic-tool metadata for canonical OpenAI model turns and adds a focused run-attempt regression test.

## Best Possible Solution

Keep the attribution normalization for diagnostics and persisted records, but keep dynamic tool construction on the selected model provider or pass separate attribution metadata, then add redacted real-run proof.

## Reproduction Assessment

Yes for a source-level reproduction: current main feeds canonical `openai` / `openai-responses` params into transcript and trajectory recorders. I did not establish a live current-main reproduction in this read-only review.

## Solution Assessment

No as written: the attribution direction is narrow, but the implementation should not let normalized runtime-attribution params alter dynamic tool construction and `tools.byProvider` policy matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep tool policy on the selected model provider:** `extensions/codex/src/app-server/run-attempt.ts:530`
  - body: This call now builds tools with `codexRuntimeParams`, after the helper rewrites `params.model.provider` from `openai` to `openai-codex`. `buildDynamicTools` forwards that value as `modelProvider` to `createOpenClawCodingTools`, and provider-specific policy lookup uses that key, so existing `tools.byProvider.openai` or `tools.byProvider.openai/gpt-5.5` restrictions will no longer apply to canonical OpenAI Codex runs. Keep the attribution rewrite to record/report surfaces, or pass separate attribution metadata that does not change tool-policy inputs.
  - confidence: 0.86
