## Summary

This PR should stay open because the attribution fix is useful but the current diff still lets attribution-normalized Codex params flow into dynamic-tool construction, which can change provider-scoped tool policy for canonical OpenAI Codex runs.

## What This Changes

The PR rewrites Codex app-server local runtime attribution for canonical OpenAI Responses runs to `openai-codex` / `openai-codex-responses` and adds a regression test for transcript and trajectory records.

## Best Possible Solution

Keep the attribution normalization for transcript, trajectory, prompt-report, and context-engine records, but keep model/provider policy inputs separate so tool construction still honors the selected canonical provider.

## Reproduction Assessment

Yes for a source-level reproduction: current main records Codex app-server assistant and trajectory metadata from canonical OpenAI params, so a Codex-owned canonical OpenAI run is locally stamped as OpenAI Responses. I did not establish a live runtime reproduction in this read-only review.

## Solution Assessment

No as written: the runtime-attribution direction is narrow, but the implementation should not pass attribution-normalized params into dynamic-tool construction because that changes provider-scoped policy inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep tool policy on the selected model provider:** `extensions/codex/src/app-server/run-attempt.ts:530`
  - body: This call now uses `codexRuntimeParams`, where the helper rewrites `params.model.provider` from `openai` to `openai-codex`. `buildDynamicTools` forwards that value as `modelProvider`, and tool construction applies `tools.byProvider` from that provider, so canonical `openai/gpt-5.5` Codex runs can stop honoring existing `tools.byProvider.openai` policy. Keep the attribution rewrite to record/report surfaces, or pass separate attribution metadata that does not change tool-policy inputs.
  - confidence: 0.88
