## Summary

Keep open: this is a focused candidate fix for the linked bug, and current main/latest release still have the mismatch. I found no blocking code defect, but the implicit CLI default change remains compatibility-sensitive and needs maintainer acceptance before merge.

## What This Changes

The branch makes `sessions export-trajectory` resolve its implicit store through `getRuntimeConfig().session?.store` and `resolveStorePath`, updates help text, and adds command/resolver fallback tests.

## Best Possible Solution

Land the focused resolver change after maintainers accept that export without `--store` should match sibling session commands, with explicit `--store` remaining the override.

## Reproduction Assessment

Yes. Source inspection on current main shows export without `--store` uses the default agent store while sibling session commands resolve configured `session.store`; the linked issue and PR body also include configured-store reproduction/proof.

## Solution Assessment

Yes. Reusing `getRuntimeConfig` plus `resolveStorePath` in the single-session export command is the narrowest maintainable fix; routing through multi-target session selection would add broader list/tail semantics this command does not need.
