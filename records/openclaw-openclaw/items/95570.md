## Summary

Keep open: the PR is a focused, source-backed fix for the linked issue and the patch looks correct, but it changes a compatibility-sensitive implicit CLI default that should get maintainer acceptance before merge.

## What This Changes

The branch makes `sessions export-trajectory` resolve its implicit store through `getRuntimeConfig().session?.store` and `resolveStorePath`, updates the `--store` help text, and adds fallback/configured-store tests.

## Best Possible Solution

Land the focused resolver change once maintainers accept that export without `--store` should follow configured `session.store` like sibling session commands, with explicit `--store` remaining the override.

## Reproduction Assessment

Yes. Source inspection on current main shows export without `--store` uses `resolveDefaultSessionStorePath` while sessions list resolves through runtime config and session-store targets; the issue and PR also include temporary configured-store reproduction/proof.

## Solution Assessment

Yes. Reusing `getRuntimeConfig` plus `resolveStorePath` is the narrowest owner-boundary fix; routing through multi-target session selection would add broader list/tail semantics to a single-session export command.
