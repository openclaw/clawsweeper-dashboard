## Summary

Keep open: current main and v2026.6.9 still resolve `sessions export-trajectory` through the default agent session store when `--store` is omitted, while sibling session commands use configured store resolution. The PR is a narrow candidate fix for the linked bug, and I found no blocking patch or security findings.

## What This Changes

The PR changes `sessions export-trajectory` to resolve the implicit session store through runtime `session.store`, then updates the command help and regression tests.

## Best Possible Solution

Land this narrow PR, or an equivalent fix, so trajectory export uses the same configured store resolution as session list and tail while preserving explicit `--store`.

## Reproduction Assessment

Yes. The source path is high-confidence: current main exports from `resolveDefaultSessionStorePath` when `--store` is omitted, while session list/tail use configured store resolution; a temp-home setup with `session.store` pointed elsewhere would reproduce the miss.

## Solution Assessment

Yes. Reusing `resolveStorePath(getRuntimeConfig().session?.store, { agentId })` in the export command is the narrow owner-boundary fix and preserves explicit `--store` as the higher-priority override.
