## Summary

Keep open. Current main and v2026.5.19 still route TypeScript plugin source entries through the jiti source-transform path; the later Windows fast paths cover compiled JavaScript artifacts, not the remaining source-TS loader latency reported here.

## What This Changes

Fix pathological Windows latency when production plugin-loading paths import TypeScript source plugin entries through jiti.

## Best Possible Solution

Keep the compiled-JS Windows fast paths and design a Windows-safe TypeScript source-loading strategy with before/after profiling from real source-checkout CLI and gateway paths.

## Reproduction Assessment

No high-confidence live Windows reproduction was run in this read-only review. The remaining path is source-reproducible by inspection because current main still sends `.ts`/`.tsx` plugin entries through jiti when native loading is not selected.

## Solution Assessment

Unclear. The native-import direction is plausible, but the best fix needs maintainer loader-design review because it crosses sync registration contracts, SDK alias rewriting, and Windows TS runtime behavior.
