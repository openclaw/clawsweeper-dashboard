## Summary

Keep open. Current main and v2026.5.22 still preserve the jiti/source-transform path for TypeScript plugin source entries; the landed Windows fast paths and related open PRs address compiled JavaScript artifacts, not the remaining source-TS loader latency reported here.

## What This Changes

The issue reports pathological Windows latency when production plugin bootstrap/status/CLI call sites load TypeScript source plugins through the synchronous jiti loader path.

## Best Possible Solution

Keep the compiled-JS native fast paths, then design a Windows-safe TypeScript source-loading strategy with before/after profiling from real source-checkout CLI, status/inspect, and gateway bootstrap paths.

## Reproduction Assessment

Unclear for live behavior: the report gives detailed Windows timings, and the current source path still routes TypeScript plugin entries through jiti, but I did not run a Windows production-profile reproduction in this read-only review.

## Solution Assessment

Unclear: the async native-import TS-aware direction is plausible, but the best fix needs maintainer loader-design review because the current plugin loader is synchronous and relies on jiti alias behavior.
