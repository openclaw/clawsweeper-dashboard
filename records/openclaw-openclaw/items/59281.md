## Summary

Keep open. Current main and the latest release still keep TypeScript plugin source entries on the synchronous jiti source-transform path; the merged Windows/native fast paths cover compiled JavaScript artifacts and adjacent duplicate-evaluation cases, not the central Windows TS source-loading latency report.

## What This Changes

The issue asks OpenClaw to avoid pathological Windows latency when production plugin-loading call sites load TypeScript source plugin entries through the current jiti-based loader.

## Best Possible Solution

Design the plugin-loader fix at the shared loader boundary, preserve documented plugin SDK/manifest behavior, and prove it with live Windows production call sites plus focused loader and alias tests.

## Reproduction Assessment

No, not as a live current-main reproduction in this review. The source path is clear: current main and v2026.6.10 still route TypeScript source plugin entries through the jiti source-transform path, but the Windows latency numbers need live profiling.

## Solution Assessment

Unclear. The requested async native-import TS-aware path is plausible, but the best fix needs maintainer review because plugin loading is currently synchronous and tied to SDK alias/module-identity contracts.
