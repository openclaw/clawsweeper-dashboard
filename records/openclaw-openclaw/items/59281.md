## Label Justifications

- `P2`: The report describes severe platform-specific startup/import latency in plugin bootstrap paths, but not data loss, security bypass, or a universal runtime outage.
- `impact:crash-loop`: Several-second to several-minute synchronous plugin loads can make CLI and gateway startup paths appear hung or unusable for affected Windows setups.

## Summary

Keep open. Current main and the latest release still route TypeScript source plugin entries through the jiti source-transform path; the native fast path only covers compiled JavaScript artifacts, so the central Windows source-loading latency report remains unresolved.

## What This Changes

The issue requests a Windows-safe redesign of TypeScript source plugin loading so real CLI, status, gateway, and registry callers avoid pathological synchronous jiti latency.

## Best Possible Solution

Keep this open until maintainers choose a Windows-safe TypeScript source-loading design and validate it with focused loader tests plus live Windows before/after profiling across CLI, status, gateway, and registry callers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the implicated path: current main still accepts source TypeScript plugin entries and routes them through jiti, but this review did not run a live Windows latency reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The issue's async native-import direction is plausible, but the best fix needs maintainer design review because current tests intentionally preserve jiti for TypeScript sources and SDK alias fallback behavior.
