## Label Justifications

- `P2`: The issue describes severe platform-specific plugin bootstrap latency, but not data loss, security bypass, or a universal runtime outage.
- `impact:crash-loop`: Several-second to several-minute synchronous plugin loads can make CLI and gateway startup paths appear hung or unusable for affected Windows setups.

## Summary

Keep open. Current main and v2026.6.6 still route linked/local TypeScript plugin entries through the jiti source-transform path while native loading is scoped to compiled JavaScript artifacts, so the central Windows TS source-loading latency report is not implemented on main.

## What This Changes

The issue reports pathological Windows latency when production plugin-loading call sites load TypeScript source plugins through the synchronous jiti path.

## Best Possible Solution

Keep this open until maintainers choose a Windows-safe TypeScript source-loading design and validate it with focused loader tests plus live Windows profiling across CLI, status/inspect, gateway bootstrap, and registry preload callers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the implicated path: current main accepts linked/local TypeScript plugin entries and routes them through jiti, but this review did not run a live Windows latency reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear; the async native-import direction is plausible, but current tests and docs intentionally preserve jiti for TypeScript source fallback, so maintainers need to approve the loader design before a fix PR.
