## Summary

Keep open. The April 30 ClawSweeper review still matches current main: Windows compiled JavaScript plugin loading has a native fast path now, but TypeScript source plugin entries still route through the synchronous jiti transform path described in this issue.

## What This Changes

The issue asks OpenClaw to avoid pathological Windows latency when production plugin bootstrap paths load TypeScript source plugin entries through jiti.

## Best Possible Solution

Keep the compiled-JS Windows fast path, then design and validate a Windows-safe source-TypeScript strategy: async TS-aware native import where appropriate, or deterministic routing to built plugin artifacts when available.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The source path is still reproducible by inspection because current main routes source .ts/.tsx plugin entries through jiti when tryNative is false.

## Solution Assessment

Unclear. The issue's native-import direction matches the reported evidence, but maintainers should choose the exact design because changing source TS loading affects loader synchrony, alias rewriting, and Windows-only performance validation.
