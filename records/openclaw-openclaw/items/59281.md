## Summary

Keep open. Current main and latest release still keep source TypeScript plugin entries on the jiti/source-transform path; the merged Windows fast-path work only covered compiled JavaScript artifacts, so the central Windows TS source-loading latency report remains unresolved.

## What This Changes

The issue requests a fix for pathologically slow Windows TypeScript source plugin loading through the synchronous jiti-based plugin loader path.

## Best Possible Solution

Keep this issue open until maintainers choose a Windows-safe TypeScript source-loading design and validate it with focused loader tests plus live Windows before/after profiling across CLI, status, gateway, and registry callers.

Yes at source level: current main and latest release still support source TypeScript plugin entries for local development and route non-native plugin loads through jiti. No live Windows latency reproduction was run in this read-only review.

Unclear. A native-import TS-aware path is plausible, but current tests and merged PR history intentionally preserve jiti for TypeScript sources, so the best fix needs maintainer design review and live Windows proof.
