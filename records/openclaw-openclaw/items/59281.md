## Summary

Keep open. Current main and the latest release still keep TypeScript source plugin entries on the synchronous jiti source-transform path; recent merged work improved compiled JavaScript and plugin-sdk dist fast paths on Windows, but did not close the remaining source-TS performance problem.

## What This Changes

The issue requests avoiding pathological Windows latency when production plugin-loading paths load TypeScript source plugin entries through the synchronous jiti path.

## Best Possible Solution

Keep the compiled-JS Windows fast paths, then design and validate a Windows-safe TypeScript source-loading strategy with before/after profiling from a real Windows source checkout.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The source route is reproducible by inspection because current main still sends `.ts`/`.tsx` plugin entries through jiti when `tryNative` is false.

## Solution Assessment

Unclear. The issue's native-import direction is plausible, but the exact fix should be chosen by maintainers because it crosses synchronous loader contracts, alias rewriting, and Windows-only performance validation.
