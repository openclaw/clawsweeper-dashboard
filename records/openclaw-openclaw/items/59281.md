## Summary

Keep open. Current main and v2026.5.22 still route TypeScript source plugin loads through the jiti source-transform path; the merged Windows fast paths cover compiled JavaScript artifacts, not the source-TS latency reported here.

## What This Changes

The issue asks OpenClaw to avoid pathological Windows latency when real production plugin-loading call sites load TypeScript source plugins through the synchronous jiti path.

## Best Possible Solution

Keep the compiled-JS Windows fast paths, then design and profile a Windows-safe TypeScript source-loading strategy for source-checkout CLI, status/inspect, gateway bootstrap, and registry preload paths.

## Reproduction Assessment

No high-confidence live Windows reproduction was run in this read-only review. The remaining path is source-reproducible by inspection because current main still sends TypeScript source plugin entries through the jiti source-transform loader when native loading is not selected.

## Solution Assessment

Unclear. The async/native-import direction is plausible, but the best fix needs maintainer loader-design review because it touches sync registration contracts, SDK alias rewriting, and Windows TS runtime behavior.
