## Label Justifications

- `P2`: The report describes severe Windows-specific plugin bootstrap latency, but not data loss, security bypass, or a universal runtime outage.
- `impact:crash-loop`: Several-second to several-minute synchronous plugin loading can make startup and plugin bootstrap paths appear hung for affected Windows source-plugin setups.

## Summary

Keep open. Current main and the latest release still route TypeScript source plugin entries through the Jiti source-transform path; the merged Windows fast paths cover compiled JavaScript and plugin-sdk alias cases, not the central TS-source latency report.

## What This Changes

The issue asks OpenClaw to avoid pathological Windows latency when production plugin-loading call sites load TypeScript source plugins through the synchronous Jiti path.

## Best Possible Solution

Keep this issue open until maintainers choose a Windows-safe TypeScript source-loading design and validate it with focused loader tests plus live Windows profiling across CLI, status, gateway, and registry callers.

## Reproduction Assessment

Yes at source level, but not as a fresh live Windows timing run. Current main still routes TypeScript source entries away from native loading and into the Jiti source-transform fallback.

## Solution Assessment

Unclear. A native-import TS-aware path is plausible, but current docs, tests, and merged PR scope intentionally preserve Jiti for TypeScript sources, so the best fix needs maintainer design review and live Windows proof.
