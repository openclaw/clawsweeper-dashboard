## Summary

Keep open. Current main still keeps source TypeScript plugin entries on the jiti path; the Windows native fast path covers compiled JavaScript artifacts, but not the source-checkout/dev-runtime lane described in this issue.

## What This Changes

The issue requests removing pathological Windows latency when production plugin-loading call sites import TypeScript source plugin entries through the synchronous jiti loader path.

## Best Possible Solution

Keep the compiled-JS Windows fast path, then choose and validate a Windows-safe strategy for source TypeScript plugin loading: either an async TS-aware native import path for source callers or deterministic routing from source entries to built runtime artifacts where available.

## Reproduction Assessment

No: this review did not run a Windows source checkout. The static reproduction path remains high-signal because current main still routes source .ts plugin entries through jiti in the loader path named by the report.

## Solution Assessment

Unclear. The issue's async native-import direction matches the reported performance evidence, but the current source-graph aliasing and synchronous loader contract mean maintainers should choose the exact design before implementation.
