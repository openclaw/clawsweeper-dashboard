## Label Justifications

- `P2`: This is a real performance bug with a limited blast radius around Windows source-TS plugin loading rather than a crash, data loss, or security emergency.
- `impact:crash-loop`: The reported loader latency can make plugin bootstrap and CLI/gateway flows appear hung for affected Windows setups.

## Summary

Keep open. Current main and v2026.5.18 still route TypeScript plugin source entries through the jiti source-transform path; the merged Windows improvements cover compiled JavaScript fast paths, not the remaining source-TS performance problem.

## What This Changes

The issue asks OpenClaw to eliminate pathological Windows latency when production plugin-loading paths import TypeScript source plugins through synchronous jiti.

## Best Possible Solution

Keep the compiled-JS Windows fast paths, then design and validate a Windows-safe TypeScript source-loading strategy with before/after profiling from real Windows source-checkout CLI and gateway paths.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The remaining route is source-reproducible by inspection because current main still sends `.ts`/`.tsx` plugin entries through jiti when native loading is not selected.

## Solution Assessment

Unclear. The native-import direction is plausible, but the best fix needs maintainer review because it changes loader behavior across sync contracts, SDK alias rewriting, and Windows TS runtime handling.
