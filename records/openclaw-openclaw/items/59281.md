## Label Justifications

- `P2`: The report describes severe Windows-specific plugin bootstrap latency, but not data loss, security bypass, or a universal runtime outage.
- `impact:crash-loop`: Several-second to several-minute synchronous plugin loading can make startup and plugin bootstrap paths appear hung for affected Windows setups.

## Summary

Keep open. Current main and the latest release still keep TypeScript plugin source entries on the jiti/source-transform path; the merged Windows fast-path work covers compiled JavaScript artifacts only, so the central Windows TS-source latency report remains unresolved.

## What This Changes

The issue asks OpenClaw to avoid pathological Windows latency when production plugin bootstrap and registry paths load TypeScript source plugin entries through jiti.

## Best Possible Solution

Keep this issue open until maintainers choose a Windows-safe TS source-loading design and validate it with focused loader tests plus live Windows profiling across CLI, status, gateway, and registry callers.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still support local TypeScript source plugin entries and route non-native plugin loads through jiti. No live Windows latency reproduction was run in this read-only review.

## Solution Assessment

Unclear. A native-import TS-aware path is plausible, but current tests and merged PR history intentionally preserve jiti for TypeScript sources, so the best fix needs maintainer design review and live Windows proof.
