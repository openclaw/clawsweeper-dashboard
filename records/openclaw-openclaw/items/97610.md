## Summary

Keep open: the patch uses the right bounded helper for the shared `readUsageJson` success path, and the submitted terminal proof covers that path, but the provider-usage OOM hardening is incomplete while sibling provider-usage fetchers still call `response.json()` directly.

## What This Changes

The PR changes `readUsageJson` in the shared provider-usage fetch helper to parse responses through `readProviderJsonResponse` instead of `Response.json()`.

## Best Possible Solution

Use the existing bounded provider JSON helper for every provider-usage JSON body read that can receive provider-controlled data, while preserving the existing MiniMax error string and Claude missing-scope fallback behavior.

## Reproduction Assessment

Yes. Current main calls `response.json()` in `readUsageJson`, and sibling provider-usage fetchers also have direct `response.json()` reads, so oversized provider-controlled JSON can be fully buffered from source inspection; I did not run the contributor's local harness.

## Solution Assessment

No. Swapping `readUsageJson` to `readProviderJsonResponse` is the right primitive for the shared success path, but the best fix should also cover the remaining provider-usage JSON readers or explicitly split that work into a tracked follow-up before merge.
