## Label Justifications

- `impact:security`: The issue concerns permissions on OpenClaw state and session directories that may contain config, credentials-adjacent data, transcripts, and logs.

## Summary

Keep open: the exact existing-700 reset path is not reproduced, but current main and v2026.6.8 still have managed state/session directory creation paths that omit private directory modes, and the related fixing PRs are open and not merged.

## What This Changes

The issue reports that agent startup can loosen `~/.openclaw` permissions through recursive directory creation without an explicit private mode.

## Best Possible Solution

Centralize private creation and hardening for managed state/session directories, preserve caller-owned custom paths, and close this issue only after https://github.com/openclaw/openclaw/pull/45901 or a narrower reviewed replacement lands with upgrade and live permission proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the exact existing-700 reset path: a focused Node probe preserved an existing 700 directory. Yes from source for the remaining bug class: current managed session/setup/store/lock paths can create missing parents without a private mode, and the probe showed missing parents land as 755 under the default umask.

## Solution Assessment

Is this the best way to solve the issue? The suggested `mode: 0o700` direction is right but too narrow if applied one call at a time. The better fix is a shared managed-state/session directory helper plus regression coverage and explicit preservation of custom caller-owned paths.
