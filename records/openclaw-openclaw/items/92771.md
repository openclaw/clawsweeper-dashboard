## Summary

Keep open: current main still has source-level state/session directory paths that can create OpenClaw-owned parents without private permissions, and the related fix work is open but not landed. The exact existing-700 reset was not live-reproduced in this read-only pass, but the docs and security audit contract require the state directory to remain private, so this remains a valid security hardening bug.

## What This Changes

The issue asks OpenClaw to keep `~/.openclaw` private during agent startup instead of creating or relaxing state/session parents through default-umask directory creation.

## Best Possible Solution

Centralize OpenClaw-owned state and managed session directory creation behind a private-mode helper, keep `security audit --fix` and doctor as repair paths for already-open directories, and land focused POSIX regression coverage before closing this issue.

## Reproduction Assessment

Partly: there is a high-confidence source-level path for missing state/session parents being created without `0o700`, but this review did not run a live repro. The exact report that an already-700 directory is reset by recursive mkdir remains unproven here.

## Solution Assessment

No, the issue's single call-site `mode` suggestion is directionally right but likely incomplete. The best fix should cover the shared state/session creation boundary and the atomic writer default without forcing arbitrary caller-supplied custom paths.
