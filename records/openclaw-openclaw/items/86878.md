## Label Justifications

- `P1`: The report describes a persisted Codex thread state corruption that made every later ChatGPT-OAuth call fail for the affected session.
- `impact:session-state`: The issue is about poisoned persisted Codex rollout/session history being replayed into later agent turns.

## Summary

Current main and the latest release now cover the OpenClaw side of the failure: the Codex plugin uses a managed Codex app-server 0.139.0 by default, sanitizes invalid inline image payloads before replay/projection, and clears the poisoned Codex thread binding after invalid-image API errors so the broken native rollout is not reused indefinitely.

## What This Changes

Reports that Codex app-server can corrupt persisted rollout `input_image` base64 with exec stdout/truncation markers, causing repeated ChatGPT-OAuth request failures until thread recovery.

## Best Possible Solution

Keep the shipped OpenClaw mitigation and managed Codex app-server default; leave the deeper Rust buffer/root-cause investigation in the upstream OpenAI Codex tracker.

## Reproduction Assessment

No, not against current main: I did not reproduce the original rare corruption live. The report has detailed post-mortem evidence, and current source/tests now cover the OpenClaw mitigation paths for invalid image replay and poisoned binding recovery.

## Solution Assessment

Yes for the OpenClaw side: the maintainable fix is to use the managed current Codex binary by default, sanitize replayed image payloads, and rotate away from poisoned native bindings. The exact Rust root-cause remains best handled in the upstream OpenAI Codex issue at https://github.com/openai/codex/issues/24582.
