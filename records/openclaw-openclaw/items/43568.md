## Summary

Keep open for maintainer product/security review: current main has global and model-provider proxy routing, but auth profiles remain credential-only and still do not define profile-scoped proxy selection.

## What This Changes

The issue requests an optional `proxyUrl` field in auth profiles so different provider credentials can use different outbound proxies.

## Best Possible Solution

Define a credential-safe profile-scoped transport contract, or choose a config-level alternative, with explicit precedence, redaction, SecretRef, docs, and regression coverage before adding the field.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Source inspection on current main shows auth profiles have no profile-scoped proxy field while provider request config already has separate proxy override support.

## Solution Assessment

No. A small `proxyUrl` passthrough in `auth-profiles.json` is not the best safe fix until maintainers decide credential storage, redaction, precedence, and network egress semantics.
