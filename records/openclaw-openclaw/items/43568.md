## Summary

Keep open: current main has global and provider-level proxy routing, but auth-profile credentials do not carry per-profile transport, and runtime auth hooks currently reject proxy/TLS overrides. This is a new config and security-sensitive product decision, not an already-implemented bug fix.

## What This Changes

Add an optional `proxyUrl` field to auth-profiles so requests made with a selected provider profile route through that profile's proxy.

## Best Possible Solution

If accepted, design a sanctioned transport contract that keeps proxy URLs secret-aware and redacted, resolves them with auth-profile selection, and applies them consistently through LLM/provider plugin request paths with docs and regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main lacks an auth-profile `proxyUrl` field while existing proxy support is process-level or provider-config-scoped.

## Solution Assessment

No, not as a 20-30 line auth-profile field. The maintainable path needs schema, credential/SecretRef handling, transport propagation, docs, tests, and security review; existing provider-level proxy config covers some deployments but not same-provider profile rotation.
