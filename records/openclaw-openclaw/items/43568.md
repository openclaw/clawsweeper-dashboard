## Summary

Keep open for maintainer product/security review: current main and v2026.5.22 have managed process-level proxy routing and provider-level `request.proxy`, but auth profiles remain credential-only and still have no profile-scoped transport contract.

## What This Changes

The issue requests an optional `proxyUrl` field on auth profiles so provider requests can be routed through a proxy selected by the active credential profile.

## Best Possible Solution

Define a credential-safe profile-scoped transport contract, or an approved config-level alternative, with explicit precedence, SecretRef/redaction handling, docs, and regression coverage.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source and release inspection confirm auth profiles do not expose a profile-scoped proxy field while provider-level and process-level proxy paths exist separately.

## Solution Assessment

No for the proposed small passthrough. The maintainable solution needs an approved credential storage model, SecretRef/redaction policy, precedence rules, and egress semantics before code is safe to land.
