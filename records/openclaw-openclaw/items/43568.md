## Summary

Keep open: current main supports global and provider-scoped model request proxying, but not auth-profile-scoped proxy routing. The request would add a new credential/config contract and needs maintainer security/product direction before implementation.

## What This Changes

Request adding an optional `proxyUrl` field to auth profiles so different credentials for the same provider can route through separate HTTP proxies.

## Best Possible Solution

Define an approved auth-profile transport contract, or document an approved provider-alias alternative, that keeps proxy values secret-aware, redacted, and consistently applied through selected-profile provider requests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms current main has provider/global proxy routing but no auth-profile `proxyUrl` contract or profile-scoped transport propagation.

## Solution Assessment

No, not as a small direct field addition. The maintainable solution needs schema, SecretRef/redaction, transport precedence, runtime auth behavior, docs, and security review, with provider aliases considered as an alternative.
