## Summary

Keep open for maintainer product/security review: current main supports global and model-provider proxy routing, but auth profiles remain credential-only and do not support profile-scoped proxy selection.

## What This Changes

The issue requests adding a `proxyUrl` field to auth-profiles so each provider credential profile can route API traffic through a dedicated proxy.

## Best Possible Solution

Make an explicit maintainer product/security decision first; if accepted, define a credential-safe profile-scoped transport contract with precedence, SecretRef/redaction coverage, docs, and tests across auth-profile selection and provider request dispatch.

## Reproduction Assessment

Not applicable as a feature request. Source inspection shows current main lacks a profile-level proxy field while provider proxy routing exists under model-provider request config.

## Solution Assessment

No. The proposed small `proxyUrl` passthrough is not the best safe fix yet because current contracts separate credentials from transport policy and runtime auth overrides explicitly reject proxy/TLS settings.
