## Summary

Current main provides provider-scoped proxy transport but does not bind proxy selection to auth-profile rotation; this remains a valid feature request, though the proposed legacy file field is not the right implementation shape.

## What This Changes

Add a per-auth-profile proxy setting so provider requests use a distinct proxy whenever that credential profile is selected.

## Best Possible Solution

Keep the existing provider request proxy as the canonical transport mechanism and, if maintainers approve profile binding, add a SecretRef-capable profile-to-request-policy reference with explicit precedence and lifecycle coverage rather than a raw proxyUrl credential field.

## Reproduction Assessment

Not applicable as a bug reproduction: current source clearly shows provider-scoped proxy configuration and profile-scoped credential rotation, but no supported contract joining them.

## Solution Assessment

No, the proposed raw proxyUrl field in the legacy auth-profile file shape is not the best solution; any implementation should securely bind profiles into the existing typed request-policy surface and define precedence and request scope.

## Risks / Open Questions

- A raw proxy URL stored with a credential may contain usernames and passwords, creating new SecretRef, redaction, audit, export, backup, migration, and copy-to-agent obligations.
- A profile-level proxy beside models.providers.*.request.proxy needs explicit precedence rules to avoid competing transport-policy sources during profile rotation.
- The desired scope is ambiguous for requests outside inference, including OAuth refresh, usage probes, model discovery, and other provider-owned HTTP calls.
