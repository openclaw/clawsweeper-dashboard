## Summary

Close as duplicate/superseded: current main already supports provider-level request proxies, auth profiles are intentionally credential-only, and the remaining per-account routing/product/security design is already tracked by the broader provider-profile routing issue.

## What This Changes

The issue requests an optional `proxyUrl` auth-profile field so different provider accounts can route through different proxies.

## Best Possible Solution

Track per-account proxy and transport requirements under the broader provider-profile routing contract, while keeping auth profiles credential-only and using provider request transport config for direct proxy cases.

## Reproduction Assessment

Not applicable as a feature request. Source review shows current main lacks `proxyUrl` on auth-profile credentials while already supporting provider-level `request.proxy`.

## Solution Assessment

No: adding `proxyUrl` directly to auth profiles is not the best standalone path because current docs and types keep auth profiles credential-only. The maintainable path is to carry the proxy/IP-isolation use case into the broader provider-profile routing design.
