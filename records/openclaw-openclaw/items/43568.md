## Summary

Keep open. Current main has route-level model-provider proxy support, but it does not attach proxy routing to the selected auth profile; adding that would change public credential/config contracts and needs maintainer/security design.

## What This Changes

Request adding optional `proxyUrl` to auth-profiles so each provider credential profile can send model API traffic through a dedicated proxy.

## Best Possible Solution

Define an approved model-provider/auth-profile proxy contract that preserves auth-profiles as credential-only unless maintainers intentionally expand it, reuses guarded transport overrides, and covers redaction, precedence, docs, and tests.

## Reproduction Assessment

Not applicable as a classic bug because this is a feature request. Source inspection gives high confidence in current behavior: proxy transport is provider-route scoped, while auth-profile types and docs have no proxy field.

## Solution Assessment

Unclear. The requested `proxyUrl` field is plausible, but the current architecture places transport under `models.providers.*.request`, so maintainers should decide whether profile-scoped proxying belongs in credential storage or in a route/default policy.
