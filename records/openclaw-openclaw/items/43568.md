## Summary

Keep open. Current main still supports process-level and provider-request proxy routing, but not auth-profile-scoped transport policy, and the request affects secret-bearing profile data plus provider egress behavior that needs maintainer design approval.

## What This Changes

Add optional per-auth-profile proxy routing so different provider credentials can send model-provider API requests through distinct proxy URLs.

## Best Possible Solution

Define a first-class auth-profile transport policy that is secret-aware, validates proxy URLs, documents egress precedence, preserves redaction and agent-copy semantics, and threads the selected profile policy into provider request construction.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high confidence because current credential types, strict config schema, resolved-auth type, runtime-auth sanitization, docs, and request-proxy tests show no profile-scoped proxy channel on current main.

## Solution Assessment

No, adding a raw proxyUrl directly to credential records is not yet the best complete solution. The safer path is a designed profile transport policy that composes with secrets/redaction and existing egress precedence before implementation.
