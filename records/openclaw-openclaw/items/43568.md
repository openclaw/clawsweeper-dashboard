## Summary

Keep open. Current main still has no auth-profile-scoped proxy field or flow; adjacent model-provider `request.proxy` and global managed proxy support do not attach proxy routing to the selected auth profile, and adding that public credential/config surface needs maintainer/security design.

## What This Changes

Request adding an optional `proxyUrl` field to auth-profile credentials so requests made with each profile can use a dedicated proxy.

## Best Possible Solution

Define an approved profile-or-route proxy contract that preserves secret redaction and guarded egress, then update auth/config types, docs, schema, and focused provider transport tests together.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a broken documented behavior. Source inspection on current main gives high confidence that profile-scoped proxy routing is absent while route-level `models.providers.*.request.proxy` exists.

## Solution Assessment

Unclear. The requested `proxyUrl` field is plausible, but the maintainable solution needs a maintainer decision on whether transport belongs in credential storage or should remain under route-level provider request policy.
