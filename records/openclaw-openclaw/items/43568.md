## Summary

Keep open for maintainer product/security review: current main has global managed proxy routing and provider-local request proxy overrides, but auth profiles remain credential-only and do not define profile-scoped transport selection.

## What This Changes

Request adding an optional `proxyUrl` field to auth-profiles so provider requests can use a profile-specific proxy.

## Best Possible Solution

Define a credential-safe profile-scoped transport contract, or choose a config-level alternative, with explicit precedence, SecretRef/redaction handling, docs, and regression coverage before adding auth-profile proxy routing.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection on current main gives high confidence that auth profiles have no profile-scoped proxy field while provider request config already has separate proxy override support.

## Solution Assessment

No; a small `proxyUrl` passthrough in `auth-profiles.json` is not the best safe fix until maintainers decide credential storage, redaction, precedence, and network egress semantics.
