## Summary

Keep open. Current main has process-level and provider-request proxy controls, but selected auth profiles still resolve only credentials/source/mode, so auth-profile-scoped proxy routing remains unimplemented and needs maintainer/security design.

## What This Changes

Request to add an optional auth-profile proxy policy so provider calls made with each profile can use a dedicated proxy instead of process/provider-wide proxy settings.

## Best Possible Solution

Define a secret-aware auth-profile transport policy with validation, redaction, inheritance/copy semantics, and documented egress precedence, then thread the selected profile policy into provider request construction.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high confidence because current auth-profile types, strict config schema, resolved-auth type, profile resolution path, docs, and proxy tests show no profile-scoped proxy channel on current main.

## Solution Assessment

No, adding a raw proxyUrl field directly to credential records is not yet the best complete solution. A structured profile transport policy is safer because it can compose with SecretRefs, redaction, agent inheritance, and existing proxy precedence.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would add profile-scoped proxy credentials and provider egress routing to secret-bearing auth data.

Concerns:

- **[medium] Design proxy credential storage and redaction:** `src/agents/auth-profiles/types.ts:19`
  - body: Profile-level proxy URLs may include userinfo credentials, while auth profiles already participate in copying, read-through inheritance, status, debug, backups, and snapshots; those behaviors need a SecretRef/redaction contract before the field is added.
  - confidence: 0.91
- **[medium] Define egress precedence before implementation:** `docs/security/network-proxy.md:56`
  - body: Per-profile proxies need explicit precedence against proxy.enabled, OPENCLAW_PROXY_URL, provider request proxy/TLS overrides, and runtime auth override rules so operator egress policy is not accidentally bypassed or shadowed.
  - confidence: 0.88
