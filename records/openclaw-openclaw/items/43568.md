## Summary

Keep open. Current main supports global/env and model-provider-route proxying, but it still does not attach proxy routing to the selected auth profile; adding `proxyUrl` would expand the public credential/config contract and needs maintainer/security design.

## What This Changes

Add an optional per-auth-profile `proxyUrl` field so different provider credentials can route model API requests through different proxies.

## Best Possible Solution

Define an approved profile-or-route proxy contract that preserves safe redaction and guarded provider transport behavior, then update the credential/config types, docs, schema/help, and focused provider transport tests together.

## Reproduction Assessment

Not applicable as a classic bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks profile-scoped proxy routing while supporting provider-route and global/env proxy paths.

## Solution Assessment

Unclear. The requested `proxyUrl` field is plausible, but the maintainable solution needs a maintainer decision on whether transport belongs in credential storage or should remain under route-level provider request policy.

## Risks / Open Questions

- Proxy URLs can contain credentials, so a new auth-profile field needs explicit redaction across config snapshots, logs, status output, diagnostics, UI forms, and docs.
- The feature needs defined precedence against `models.providers.*.request.proxy`, trusted env-proxy behavior, `proxy.proxyUrl`/`OPENCLAW_PROXY_URL`, and provider-owned runtime/auth hooks.
- Some multi-account deployments may already be modeled as distinct provider route IDs with `request.proxy`, so maintainers need to decide whether profile-scoped routing is worth expanding credential storage.
