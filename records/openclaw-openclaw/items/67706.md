## Summary

Keep open. The PR addresses a real current-main observability gap, but it should not merge as-is because the proposed end-event semantics diverge from the canonical compaction contract and can make failed recovery attempts count as completed memory compactions.

## What This Changes

The PR adds compaction start/end agent-event emission from embedded runner overflow and timeout recovery helpers.

## Best Possible Solution

Merge a revised patch that emits recovery compaction lifecycle events with the same result-based completion contract as the subscribe path, hardens memory accounting for failed end events, adds focused tests, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main calls `contextEngine.compact` during timeout and overflow recovery without emitting `stream: "compaction"`, so subscribers cannot observe those recovery compactions.

## Solution Assessment

No. Emitting recovery lifecycle events is the right direction, but this patch should first align `completed` with the canonical result/abort contract and update consumers/tests so failed recovery events are not persisted as successful compactions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align completed with the canonical result contract:** `src/agents/pi-embedded-runner/run.ts:908`
  - body: The new recovery end event reports `completed` as `ok && compacted`, while the subscribe handler reports completion only when compaction produced a result and was not aborted. A context engine can therefore return `ok: true` and `compacted: true` without a result and make subscribers see a completed/retrying compaction contrary to the existing contract.
  - confidence: 0.86
- **[P2] Count only completed recovery compactions in memory:** `src/agents/pi-embedded-runner/run.ts:916`
  - body: The patch emits a compaction end event for failed timeout/overflow recovery attempts, but `agent-runner-memory` currently treats any compaction end callback as completed before incrementing the stored count. Please either avoid failed end events on this path or harden the consumer to require `completed === true`.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `CHANGELOG.md:15`
  - body: This is a user-visible fix for Control UI compaction liveness, but the PR only changes runner code. Add a single-line Unreleased Fixes entry so the release notes cover the behavior change.
  - confidence: 0.9
