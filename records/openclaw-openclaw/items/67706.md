## Summary

Keep this PR open: current main still lacks standard compaction lifecycle events in the overflow and timeout recovery helpers, but the proposed patch has two blocking contract/consumer issues and still lacks contributor-provided real behavior proof.

## What This Changes

The PR adds `stream: "compaction"` start/end agent-event emission from embedded runner overflow and timeout recovery helpers in `src/agents/pi-embedded-runner/run.ts`.

## Best Possible Solution

Emit recovery lifecycle events using the same result-based completion contract as the subscribe path, update memory-flush counting to honor `completed === true`, add focused regression coverage, and collect redacted real subscriber or UI proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main runs timeout and overflow recovery compaction through helpers that fire hooks but do not emit the standard compaction start/end events expected by subscribers.

## Solution Assessment

No. Emitting lifecycle events from the recovery helpers is the right direction, but this patch should first align `completed` with the canonical result/non-abort contract and prevent failed recovery events from being counted as successful memory compactions.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P2] Align completed with the canonical result contract:** `src/agents/pi-embedded-runner/run.ts:593`
  - body: The new recovery end event reports completion as `ok && compacted`, while the subscribe handler reports completion only when compaction produced a result and was not aborted. Because `CompactResult.result` is optional, a context engine can return `ok: true` and `compacted: true` without a result and make subscribers see a completed compaction contrary to the existing contract.
  - confidence: 0.88
- **[P2] Gate memory counting on completed compactions:** `src/agents/pi-embedded-runner/run.ts:597`
  - body: This patch emits `phase: "end"` for failed timeout/overflow recovery attempts too, but `agent-runner-memory` currently treats every compaction end as completed before incrementing stored compaction count. Please either avoid failed end events in this path or update the consumer and tests to require `completed === true`.
  - confidence: 0.87
