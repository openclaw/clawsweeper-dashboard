## Summary

Keep open. Current main still lacks compaction lifecycle event emission from embedded-runner overflow and timeout recovery, so this PR addresses a real Control UI observability gap, but the patch needs revision for completion semantics, memory compaction accounting, targeted tests, and the required changelog entry before merge.

## What This Changes

The PR adds `stream: "compaction"` start/end event emission from `runEmbeddedPiAgent` recovery compaction helpers in `src/agents/pi-embedded-runner/run.ts`.

## Best Possible Solution

Merge a revised version that emits recovery compaction lifecycle events consistently with the canonical subscribe path, hardens memory accounting to count only completed compactions, and adds focused overflow/timeout recovery tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger overflow or timeout recovery on current main and observe that the recovery helpers call `contextEngine.compact` without emitting `stream: "compaction"`; this read-only review did not run tests because repo policy for this task forbids commands that may write artifacts.

## Solution Assessment

No. Emitting lifecycle events from the recovery helpers is the right direction, but this exact patch should not merge until `completed` matches the canonical result-based contract and consumers/tests are updated so failed recovery events do not increment memory compaction state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Align completed with the canonical compaction result:** `src/agents/pi-embedded-runner/run.ts:908`
  - body: The new recovery helper reports `completed` as `ok && compacted`, but the canonical `handleCompactionEnd` contract uses result presence and abort state. If a context engine returns `ok: true` and `compacted: true` without a result, subscribers will see a completed compaction even though the canonical path would not.
  - confidence: 0.82
- **[P2] Harden memory counts for failed recovery events:** `src/agents/pi-embedded-runner/run.ts:916`
  - body: This emits a compaction `phase: "end"` event even for failed timeout/overflow recovery attempts. `agent-runner-memory` currently treats any compaction end callback as completed and increments the compaction count, so failed recovery compactions can now be persisted as successful unless the consumer checks `completed === true` or failed end events are not emitted on this path.
  - confidence: 0.87

  - body: This is a user-visible fix for Control UI liveness during compaction, but the PR only changes runner code. Repo policy requires user-facing fixes to add a single-line `CHANGELOG.md` entry under `Unreleased` with contributor credit.
  - confidence: 0.9
