## Summary

This PR is a plausible fix for a real source-level gap, but it should stay open because the branch still emits a looser completion contract than the existing compaction path, can make memory flush count failed recovery compactions as successful, and lacks after-fix real behavior proof.

## What This Changes

The PR imports `emitAgentEvent` and emits `stream: "compaction"` start/end events from embedded runner overflow and timeout recovery helpers.

## Best Possible Solution

Rebase the branch, emit recovery lifecycle events with the same result-based completion contract as the subscribe path, update strict consumers/tests to respect `completed`, and add redacted runtime proof showing a subscriber or UI receives the new events.

## Reproduction Assessment

Yes, source-level: current main timeout and overflow recovery call `contextEngine.compact` through helpers that run hooks but do not emit the standard compaction lifecycle events. I did not run tests because this was a read-only review.

## Solution Assessment

No, not yet. Emitting the events is the right direction, but this patch should align `completed` with the existing result/non-abort contract and prevent failed recovery end events from being counted as successful memory compactions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align completed with the compaction result contract:** `src/agents/pi-embedded-runner/run.ts:596`
  - body: The recovery end event reports `completed` as `ok && compacted`, while the subscribe handler reports completion only when compaction produced a result and was not aborted. Because `CompactResult.result` is optional, an engine can return `ok: true` and `compacted: true` without a result and make subscribers observe a completed compaction contrary to the existing contract.
  - confidence: 0.9
- **[P2] Gate memory flush counting on completed compactions:** `src/agents/pi-embedded-runner/run.ts:605`
  - body: This emits `phase: "end"` for failed timeout/overflow recovery attempts, but memory flush still treats every compaction end event as completed before incrementing stored compaction count. Either emit failed end events only after updating the consumer contract, or make the consumer require `completed === true` before rotating/incrementing compaction state.
  - confidence: 0.88
