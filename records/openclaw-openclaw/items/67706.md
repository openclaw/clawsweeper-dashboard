## Summary

Keep open: the bug is valid and the PR still contains useful work, but the current patch introduces two correctness issues in the compaction completion contract and still lacks real after-fix behavior proof.

## What This Changes

The PR adds compaction start/end agent-event emission from embedded runner overflow and timeout recovery helpers.

## Best Possible Solution

Rebase the patch, emit recovery lifecycle events using the same result-based completion contract as the subscribe path, harden memory counting to require completed compactions, add focused regression coverage, and collect redacted real behavior proof from a real subscriber or UI path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main calls `contextEngine.compact` during timeout and overflow recovery, but those recovery helpers only fire hooks and do not emit standard compaction lifecycle events to subscribers.

## Solution Assessment

No. Emitting recovery lifecycle events is the right direction, but this patch should align `completed` with the canonical result/non-abort contract and prevent failed recovery end events from being counted as successful memory compactions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align completed with the canonical result contract:** `src/agents/pi-embedded-runner/run.ts:596`
  - body: The new recovery end event reports completion as `ok && compacted`, while the subscribe handler reports completion only when compaction produced a result and was not aborted. Because `CompactResult.result` is optional, a context engine can return `ok: true` and `compacted: true` without a result and make subscribers see a completed compaction contrary to the existing contract.
  - confidence: 0.88
- **[P2] Gate memory counting on completed compactions:** `src/agents/pi-embedded-runner/run.ts:605`
  - body: This new end-event emission also fires for failed timeout/overflow recovery attempts, but `agent-runner-memory` currently treats every compaction `phase: "end"` as completed before incrementing stored compaction count. Please either avoid failed end events in this path or update the consumer/tests to require `completed === true`.
  - confidence: 0.87
