## Summary

Keep open. Current main still lacks standard compaction lifecycle events in the embedded runner recovery helpers, but this PR is currently conflicting, has only test/lint proof, and its completion predicate can make failed or incomplete recovery compactions look successful to memory/session consumers.

## What This Changes

The PR imports `emitAgentEvent` and emits `stream: "compaction"` start/end events from embedded runner overflow and timeout recovery helpers.

## Best Possible Solution

Rebase the branch, emit recovery lifecycle events with the same result-based completion contract as the subscribe path, update memory/session consumers and tests to require `completed === true`, and add redacted runtime proof showing a subscriber or UI receives the new events.

## Reproduction Assessment

Yes, source-level. Current main timeout and overflow recovery call `contextEngine.compact` through helpers that only run hooks and never emit the standard compaction start/end events.

## Solution Assessment

No. Emitting lifecycle events is the right direction, but this patch should align `completed` with the existing result/non-abort contract and keep memory/session consumers from acting on incomplete end events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align completed with the canonical result contract:** `src/agents/pi-embedded-runner/run.ts:596`
  - body: The recovery end event reports completion as `ok && compacted`, while the subscribe handler reports completion only when compaction produced a result and was not aborted. Because `CompactResult.result` is optional, a context engine can return `ok: true` and `compacted: true` without a result and make subscribers see a completed compaction contrary to the existing contract.
  - confidence: 0.88
- **[P2] Gate memory flush on completed compactions:** `src/agents/pi-embedded-runner/run.ts:605`
  - body: This emits `phase: "end"` for failed timeout/overflow recovery attempts, but `agent-runner-memory` still treats every compaction end event as completed before incrementing stored compaction state. Either avoid failed end events in this path or update the consumer/tests to require `completed === true`.
  - confidence: 0.87
