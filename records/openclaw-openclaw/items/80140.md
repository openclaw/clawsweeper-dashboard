## Summary

Keep this PR open, but it is not mergeable as written: the watchdog direction matches the linked systemd feature gap, while the runtime heartbeat currently calls an unsupported `systemd-notify --watchdog` option and the supplied proof does not exercise the positive Linux/systemd notification path.

## What This Changes

The branch adds `WatchdogSec`/`NotifyAccess` to generated and documented gateway systemd units, creates a `systemd-notify` helper, wires it into gateway maintenance ticks, and adds tests plus a changelog entry.

## Best Possible Solution

Use the documented `WATCHDOG=1` notification payload or a direct `sd_notify` implementation, keep the unit change narrow, and add redacted Linux/systemd proof that the service accepts heartbeat pings from the gateway tick.

## Reproduction Assessment

Yes for the PR blocker: source inspection plus the local systemd 255 toolchain show `systemd-notify --watchdog` is unsupported. The linked feature gap is also source-clear because current main has neither `WatchdogSec` in the rendered unit nor a systemd notifier in the maintenance tick.

## Solution Assessment

No as written. Reusing the existing gateway tick is a reasonable direction, but the notifier must send `WATCHDOG=1` and the positive systemd path needs real Linux proof before this is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Send WATCHDOG=1 instead of using --watchdog:** `src/infra/systemd-notify.ts:79`
  - body: `systemd-notify` does not support a `--watchdog` option on the checked systemd 255 toolchain; watchdog keepalives are sent as `WATCHDOG=1`.
