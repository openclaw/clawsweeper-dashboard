## Summary

Keep this PR open: the feature is useful and linked to an open feature request, but the current patch is not mergeable because its watchdog path calls an unsupported `systemd-notify --watchdog` option and the supplied proof does not exercise the positive Linux/systemd notification path.

## What This Changes

This PR adds generated systemd watchdog directives, a `systemd-notify` helper, gateway maintenance-loop heartbeat calls, docs, tests, and a changelog entry for gateway hang recovery.

## Best Possible Solution

Keep the PR open, send watchdog pings with the documented `WATCHDOG=1` notification payload or direct `sd_notify` equivalent, and add redacted Linux/systemd proof that the user service accepts heartbeat pings.

## Reproduction Assessment

Yes. Source inspection plus the local systemd 255 command-line contract reproduce the patch defect: `systemd-notify --watchdog` is unsupported, while watchdog keepalives are documented as `WATCHDOG=1`; I did not run a live hung-gateway recovery test.

## Solution Assessment

No as written. Reusing the gateway tick for systemd watchdog liveness is a plausible narrow direction, but this implementation must use the supported watchdog notification payload and prove the positive Linux/systemd path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Send WATCHDOG=1 instead of using --watchdog:** `src/infra/systemd-notify.ts:78`
  - body: `systemd-notify` does not support a `--watchdog` flag on the checked systemd 255 toolchain; watchdog keepalives are sent as `WATCHDOG=1`. With this argument, the first managed watchdog call exits nonzero, disables the notifier, and a unit with `WatchdogSec=90s` may restart a healthy gateway because it never receives keepalives.
  - confidence: 0.94
