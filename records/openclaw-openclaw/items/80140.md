## Summary

Keep this PR open: the requested systemd watchdog capability is valid and current main still lacks it, but this branch sends watchdog pings with an unsupported `systemd-notify --watchdog` option and the supplied proof does not exercise the positive Linux/systemd notification path.

## What This Changes

The branch adds systemd `WatchdogSec`/`NotifyAccess` unit rendering, a `systemd-notify` helper, gateway maintenance-loop heartbeat wiring, docs, tests, and a changelog entry.

## Best Possible Solution

Replace the watchdog notification with systemd's supported `WATCHDOG=1` payload or direct `sd_notify`, keep the unit timeout aligned with the heartbeat cadence, and add focused Linux/systemd proof plus regression coverage.

## Reproduction Assessment

Yes for the PR defect: the diff sends `systemd-notify --watchdog`, and systemd 255 rejects that flag while `systemd.service(5)` documents `WATCHDOG=1` for keepalives. I did not run a live managed gateway hang-recovery scenario.

## Solution Assessment

No as written. Reusing a gateway liveness tick is a plausible direction, but the notifier must use the supported systemd watchdog payload and prove the positive Linux/systemd path before merge.
