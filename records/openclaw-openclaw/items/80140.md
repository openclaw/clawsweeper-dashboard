## Summary

Keep open. The linked watchdog feature is valid and current main still lacks it, but this PR has a blocking runtime defect: the notifier calls `systemd-notify --watchdog`, which is not a supported option on the checked systemd tool, so managed units with `WatchdogSec` would not receive keepalive pings.

## What This Changes

The PR adds generated systemd watchdog directives, a `systemd-notify` helper, maintenance-loop heartbeat calls, docs, tests, and a changelog entry for gateway liveness recovery.

## Best Possible Solution

Keep the PR open, send watchdog pings with the documented `WATCHDOG=1` notification path or a direct `sd_notify` implementation, then add redacted Linux/systemd proof for the positive notification path before merge.

## Reproduction Assessment

Yes. Source inspection plus the local systemd 255 toolchain reproduce the PR defect: the new notifier calls an unsupported `--watchdog` flag instead of sending `WATCHDOG=1`.

## Solution Assessment

No. The overall maintenance-loop watchdog direction is reasonable, but this implementation must use the documented notification field and prove it on a Linux/systemd positive path before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Send WATCHDOG=1 instead of using --watchdog:** `src/infra/systemd-notify.ts:78`
  - body: `systemd-notify` does not support a `--watchdog` flag on the checked systemd 255 toolchain; watchdog keepalives are sent as `WATCHDOG=1`. With this argument the first managed watchdog call exits nonzero, disables the notifier, and a unit with `WatchdogSec=90s` can restart a healthy gateway because it never receives keepalives.
  - confidence: 0.94
