## Summary

Keep open. Current main still lacks the requested Plugin SDK session-slot projection and trusted-policy session-extension read surface, and the PR head implements the intended additive seam with focused tests and API/changelog updates; however, one slot-key hardening gap remains and the latest head is dirty against main, so it is not ready to merge or close as implemented.

## What This Changes

This PR adds plugin-owned `sessionEntrySlotKey` and `sessionEntrySlotSchema` registration fields, mirrors projected plugin session-extension values into guarded top-level `SessionEntry` slots, adds scoped trusted-policy session-extension reads, and updates contract tests, changelog, and SDK API baseline files.

## Best Possible Solution

Keep the additive Plugin SDK slot-projection design, but finish the slot-key guard by reserving inherited `Object.prototype` names with focused contract coverage, then rebase onto current main and require the targeted Plugin SDK contract/API/lint checks to pass on the exact head.

## Reproduction Assessment

Yes. Static inspection is enough: on the PR head, `normalizeSessionEntrySlotKey("toString")` passes the regex and reserved-set check, and `patchPluginSessionExtension` writes the projected value as an own `SessionEntry.toString` property when a plugin registers that slot.

## Solution Assessment

Mostly yes. The PR’s read-only mirror and scoped trusted-policy session-extension accessor are narrow, additive Plugin SDK seams; the safer implementation needs the remaining prototype-name guard before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Reserve inherited Object slot names:** `src/plugins/session-entry-slot-keys.ts:3-6`
  - body: `normalizeSessionEntrySlotKey` rejects declared `SessionEntry` fields plus `__proto__`, `constructor`, and `prototype`, but inherited Object names like `toString`, `hasOwnProperty`, and `valueOf` still pass the identifier regex. A plugin can register one of those as a promoted slot, and `patchPluginSessionExtension` will write it as an own top-level `SessionEntry` property, undermining the stated prototype-key collision guard. Add the `Object.prototype` names to the reserved set, or derive them, and cover them in the contract test.
  - confidence: 0.84
