## Summary

Keep open. Current main does not have the proposed SessionEntry slot projection seam, and this PR is a fresh split from the closed predecessor stack rather than obsolete cleanup. The diff is directionally useful, but it has a concrete reserved-key guard hole for the existing `subagentRecovery` SessionEntry field plus a required changelog gap before it should merge.

## What This Changes

The PR adds `sessionEntrySlotKey` and `sessionEntrySlotSchema` to plugin session extension registrations, mirrors projected extension state into guarded top-level `SessionEntry` slots on patch/unset, validates slot keys, adds contract coverage, and refreshes the plugin SDK API hash baseline.

## Best Possible Solution

Keep the additive SDK seam, but make the slot guard cover every current core SessionEntry field that can be overwritten, add a regression test for `subagentRecovery`, add the required Unreleased changelog entry, then rerun the targeted contract test plus plugin SDK API and type/lint checks.

## Reproduction Assessment

Yes. A static reproduction path is clear: current main defines `SessionEntry.subagentRecovery`, while the PR's `normalizeSessionEntrySlotKey` reserved set omits it, so a test plugin can register that slot key and `patchPluginSessionExtension` will write or delete the core field.

## Solution Assessment

No, not as-is. The overall direction is the narrow maintainable place for this SDK seam, but the guard must reserve the existing recovery field and the public SDK change needs a changelog entry before merge.
