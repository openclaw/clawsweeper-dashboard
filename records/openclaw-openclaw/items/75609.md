## Summary

Keep open. Current main still lacks the requested `sessionEntrySlotKey` / `sessionEntrySlotSchema` projection surface, so this is the active implementation slice rather than cleanup-close material. The latest head appears to address the duplicate-slot collision feedback, but it still has a security-sensitive SDK trust-boundary problem because `OpenClawConfig` is added to the shared ordinary `before_tool_call` context, and the slot-key guard still allows inherited Object prototype names.

## What This Changes

The PR adds Plugin SDK session-extension slot metadata, mirrors projected plugin session state into guarded top-level `SessionEntry` slots with cleanup, exposes scoped trusted-policy session-extension reads, and updates contract coverage, changelog, and SDK API baseline hashes.

## Best Possible Solution

Keep the additive session-slot projection seam, but repair the authority boundary before merge: keep `OpenClawConfig` out of public/shared tool hook contexts, expose session-extension reads only through a trusted-policy-only path or private policy parameter, and extend slot-key validation to reject inherited Object prototype names with focused regression coverage.

## Reproduction Assessment

Yes. Static tracing is enough: at PR head, `config` is copied into `toolContext` and that same object is later passed to `hookRunner.runBeforeToolCall`; `normalizeSessionEntrySlotKey("toString")` also passes the current regex and reserved-key set.

## Solution Assessment

No. The session projection direction is reasonable, but widening the ordinary plugin hook context is not the narrowest maintainable way to support trusted-policy reads; a private trusted-policy context or explicit scoped reader would preserve the existing external-plugin boundary.
