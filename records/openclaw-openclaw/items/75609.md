## Summary

Keep open: current main does not contain the new session-entry slot projection API, and the PR head still has a restart cleanup bug that can drop valid promoted SessionEntry mirrors for restarted plugins that keep the same slot.

## What This Changes

The PR adds plugin-owned SessionEntry slot projection, scoped trusted-policy session-extension reads across tool surfaces, API baseline updates, tests, and a changelog entry.

## Best Possible Solution

Compare previous and next slot ownership during restart cleanup, preserving unchanged mirrors while clearing only obsolete promoted slots, then cover same-slot, changed-slot, and dropped-slot restart cases.

## Reproduction Assessment

Yes, source-reproducible: a plugin can patch a promoted slot, restart with the same sessionEntrySlotKey, and the restart cleanup path deletes the mirror even though the plugin still owns it. The mirror is only restored by a later patchPluginSessionExtension call.

## Solution Assessment

No, not yet. The additive SDK direction is reasonable, but restart cleanup should compare previous versus next ownership instead of clearing every previous promoted slot for restarted plugins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve unchanged promoted slots on restart:** `src/plugins/host-hook-cleanup.ts:267-273`

If the next registry still declares the same sessionEntrySlotKey, there is no startup path that repopulates the mirror; it is only written during patchPluginSessionExtension, so consumers see missing promoted state until another patch happens.
