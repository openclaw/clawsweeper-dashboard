## Summary

Keep open: current main does not yet contain the requested Plugin SDK slot surface, and the PR is still the active implementation path, but latest head still has a restart cleanup bug that can delete preserved promoted slots and leave stale ownership metadata.

## What This Changes

This PR adds plugin-owned `SessionEntry` slot projection for session extensions, scoped trusted-policy session-extension reads across tool surfaces, cleanup/persistence metadata, SDK baseline and changelog updates, regression tests, and a small Korean locale text correction.

## Best Possible Solution

Keep the additive SDK seam, but make restart cleanup treat the passed stale slot set as the exact target and prune corresponding ownership metadata while preserving still-declared mirrors.

## Reproduction Assessment

Yes for the review finding. Source inspection of PR head shows a restart stale-key set is expanded with all persisted slot ownership for the plugin, so a multi-slot restart that drops one slot can delete another still-declared mirror.

## Solution Assessment

No, not yet. The API direction is narrow, but restart cleanup should update only obsolete slot mirrors and ownership entries instead of deriving the delete set from all persisted metadata.

## Repair Work Prompt

On the PR branch for https://github.com/openclaw/openclaw/pull/75609, fix promoted `SessionEntry` slot cleanup on plugin restart. The bug: restart cleanup computes stale promoted slot keys, but `clearPromotedSessionEntrySlots` / `hasPromotedSessionEntrySlot` recompute from all persisted `pluginExtensionSlotKeys`, which can delete still-declared mirrors and leave dropped ownership metadata. Keep plugin extension state on restart; delete only stale top-level mirrors, prune matching `pluginExtensionSlotKeys` ownership entries, and preserve still-declared mirrors and metadata. Add a regression with two session extensions for the same plugin where one slot is dropped or changed and one slot is preserved. Do not expand the public SDK surface, change trusted-policy/config behavior, touch dependencies, or hand-edit unrelated generated locale files. The existing changelog entry is enough unless the repair changes user-facing scope beyond this PR.
