## Summary

Keep open. Current main still lacks the Plugin SDK session-entry slot projection surface, and this PR is the active implementation slice, but the latest head still permits inherited Object prototype names as promoted top-level slot keys. That is a narrow hardening issue before merge, alongside the already-pending rebase/exact-head validation.

## What This Changes

The PR adds Plugin SDK session-entry slot metadata, mirrors projected plugin session-extension state into guarded top-level SessionEntry fields, exposes scoped trusted-policy session-extension reads, and updates contract tests, changelog, and SDK API baselines.

## Best Possible Solution

Keep the additive session-entry projection and scoped trusted-policy read design, but reject inherited Object prototype names before merge and then rebase/validate the exact PR head with the focused Plugin SDK and Gateway checks.

## Reproduction Assessment

Yes. Static inspection is sufficient: normalizeSessionEntrySlotKey("toString") passes the PR head identifier regex and is not in the reserved set, so patchPluginSessionExtension can write an own SessionEntry.toString mirror.

## Solution Assessment

Mostly yes. The PR's additive API and internal trusted-policy context are the narrow maintainable direction, but the slot-key guard should derive from Object.prototype or explicitly reserve those inherited names before this lands.
