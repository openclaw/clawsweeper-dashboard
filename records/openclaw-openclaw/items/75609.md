## Summary

Keep open. Current main still lacks the requested Plugin SDK session-slot projection, and this PR is the active implementation slice, but the latest head still has a trusted-policy bypass on MCP tool calls plus a remaining promoted-slot key guard gap.

## What This Changes

This PR adds Plugin SDK session-extension slot projection, scoped trusted-policy reads of plugin session-extension state, cleanup and registration guards, Matrix DM allowlist refresh, gateway tools.invoke config propagation, tests, changelog, and SDK API baseline updates.

## Best Possible Solution

Keep the additive Plugin SDK design, but make session-extension reads available consistently on every before-tool-call surface that carries a sessionKey, especially MCP loopback, without exposing full config to public plugin hook context. Also reserve inherited Object prototype names for promoted SessionEntry slots, add focused contract/MCP coverage, then validate the exact rebased head.

## Reproduction Assessment

Yes. Static inspection is enough: MCP loopback supplies a sessionKey but no config, while the new getSessionExtension helper returns undefined when options.config is absent; the slot-key validator also accepts names like toString because they match the identifier regex and are not reserved.

## Solution Assessment

No. The broad approach is a narrow maintainable SDK seam, but the latest implementation is not complete until trusted-policy session reads work across MCP and the reserved-key guard covers Object prototype names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Pass config into MCP trusted policies:** `src/plugins/trusted-tool-policy.ts:31-37`
  - body: The new `getSessionExtension` helper returns `undefined` whenever `options.config` is missing. `src/gateway/mcp-http.ts` still calls `handleMcpJsonRpc` with a `sessionKey` but no config, so a trusted policy that blocks based on session extension state is bypassed for MCP `tools/call` even though it works on surfaces that pass config. Thread `cfg` or an equivalent session-state snapshot through the MCP hook path and add coverage for this surface.
  - confidence: 0.88
- **[P3] Reserve inherited Object slot names:** `src/plugins/session-entry-slot-keys.ts:3-6`
  - body: `normalizeSessionEntrySlotKey` rejects real `SessionEntry` fields plus `__proto__`, `constructor`, and `prototype`, but inherited names such as `toString`, `hasOwnProperty`, and `valueOf` still pass. A plugin can register one of those as a promoted slot and the patch path will create an own top-level property on `SessionEntry`, which defeats the stated prototype-key collision guard. Reserve the Object prototype names, preferably derived from `Object.prototype`, and cover them in the contract test.
  - confidence: 0.84
