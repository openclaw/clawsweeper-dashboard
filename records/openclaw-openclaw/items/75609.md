## Summary

Keep open. Current main still lacks the requested Plugin SDK session-slot projection surface, so this PR is the active implementation slice, but the latest head still has a blocking trust-boundary regression: it adds `config` to the shared tool-hook context that is later passed to ordinary `before_tool_call` hooks, even though the new session-extension read helper only needs that authority inside bundled trusted policies.

## What This Changes

The PR adds Plugin SDK session extension slot metadata, mirrors projected plugin session extension values into guarded top-level `SessionEntry` fields with cleanup, exposes scoped trusted-policy session-extension reads, and updates contract tests, changelog, and SDK API baseline hashes.

## Best Possible Solution

Keep the additive session-slot projection seam, but repair the authority boundary before merge: pass config only through a private trusted-policy path or dedicated trusted-policy parameter, keep ordinary `before_tool_call` contexts on the existing public shape, and extend slot-key validation to reject inherited Object prototype names with regression coverage.

## Reproduction Assessment

Yes. Static tracing is enough: the PR adds `config` to the shared `toolContext`, and current code passes that object to both `runTrustedToolPolicies` and `hookRunner.runBeforeToolCall`, so an ordinary `before_tool_call` plugin can observe `ctx.config`. The prototype-key gap is also reproducible from the new validator because `toString` and `hasOwnProperty` pass the identifier regex and are not reserved.

## Solution Assessment

No. The session projection direction is reasonable, but widening the public/shared hook context is not the narrowest maintainable fix for trusted-policy session reads; a private trusted-policy context or explicit `runTrustedToolPolicies` parameter would preserve the existing external-plugin boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep config out of normal tool hooks:** `src/agents/pi-tools.before-tool-call.ts:482`
  - body: The added `config` property is placed on the shared `toolContext`, and that same object is later passed to `hookRunner.runBeforeToolCall`. This gives every ordinary `before_tool_call` plugin access to the full `OpenClawConfig`, even though the new read helper only needs config inside bundled trusted policies. Pass config through a private trusted-policy context or strip it before running normal hooks.
  - confidence: 0.89
- **[P3] Reserve inherited object slot names:** `src/plugins/session-entry-slot-keys.ts:1-3`
  - body: The new reserved-key set only blocks `__proto__`, `constructor`, and `prototype` from the object-prototype family. Identifier keys such as `toString` and `hasOwnProperty` still pass validation and can be persisted as top-level `SessionEntry` own properties, despite the guard's collision goal. Seed the guard from `Object.prototype` names or add those names explicitly with regression coverage.
  - confidence: 0.74
