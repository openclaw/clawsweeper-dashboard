## Summary

Keep open: the PR is still useful and current main does not have this SDK surface, but the latest head still has a restart cleanup regression that can delete a still-declared promoted SessionEntry mirror.

## What This Changes

The PR adds plugin-owned SessionEntry slot projection for session extensions, scoped trusted-policy session extension reads across tool surfaces, SDK API baseline and changelog updates, and regression coverage.

## Best Possible Solution

Keep the additive SDK seam, but make restart cleanup treat the stale slot set as the exact target and prune only those obsolete ownership entries while preserving still-declared mirrors.

## Reproduction Assessment

Yes for the review finding. PR-head source shows the restart path passes an exact previous-minus-next stale slot set, but the helper expands it with all persisted slot ownership for the plugin before deletion.

## Solution Assessment

No, not yet. The API direction is narrow, but restart cleanup should update only obsolete promoted slots and ownership metadata rather than deriving deletion from every persisted slot owned by the plugin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep restart slot cleanup scoped to stale slots:** `src/plugins/host-hook-cleanup.ts:56-58`
  - body: On restart, callers pass only the promoted slot keys removed from the replacement registry, but this helper expands that set with every persisted `pluginExtensionSlotKeys` value for the plugin before deleting fields. If one namespace drops its slot while another keeps a different slot, cleanup deletes the still-declared mirror and leaves the dropped namespace's ownership metadata behind; use the restart set as the exact target and prune only those ownership entries.
