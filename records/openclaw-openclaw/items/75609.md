## Summary

Keep open: current main does not contain this SDK slot-projection surface, and the latest PR head still has a source-reproducible stale slot ownership cleanup bug after a plugin restarts without its old slot key.

## What This Changes

The PR adds plugin-owned SessionEntry slot projection for session extensions, scoped trusted-policy session-extension reads across tool surfaces, validation/cleanup hardening, tests, API baseline updates, and a changelog entry.

## Best Possible Solution

Keep the additive SDK seam, but make restart cleanup remove or update obsolete persisted slot ownership metadata while preserving pluginExtensions state, then re-run the focused plugin SDK gates.

## Reproduction Assessment

Yes, source-reproducible: plugin A can write a promoted slot, restart with the same namespace but no slot key, plugin B can later claim that freed slot, and disabling plugin A will collect A's stale pluginExtensionSlotKeys entry and delete B's mirror.

## Solution Assessment

No, not yet. The PR direction is a narrow additive SDK seam, but the cleanup metadata must track current slot ownership instead of leaving restart tombstones that future cleanup treats as active ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Drop stale slot ownership on restart:** `src/plugins/host-hook-cleanup.ts:222`
  - body: When a restarted plugin drops or changes its `sessionEntrySlotKey`, this path removes the old top-level mirror but leaves `pluginExtensionSlotKeys` for that plugin/namespace. That stale owner map is later consumed by ordinary disable/delete cleanup; if another plugin legitimately claims the now-free slot in the interim, cleanup of the first plugin deletes the new plugin's mirror. Remove or update the persisted ownership entry when restart cleanup clears obsolete promoted slots.
  - confidence: 0.87
