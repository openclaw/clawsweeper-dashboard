## Summary

Keep open: current main does not include this additive SDK seam, and the PR still has a source-reproducible cleanup defect where promoted top-level SessionEntry slots can survive after the plugin registration that created them is no longer available.

## What This Changes

The branch adds plugin-owned SessionEntry slot projection for session extensions, guarded slot-key validation, scoped trusted-policy session-extension reads across tool surfaces, SDK API baseline updates, tests, and a changelog entry.

## Best Possible Solution

Keep the additive SDK direction, but make slot cleanup independent of live registry availability, then land after API review and exact-head validation.

## Reproduction Assessment

Yes, source-reproducible: write a promoted slot, remove the creating registration from the active registry, then run reset/delete cleanup. The code still clears pluginExtensions but has no retained slot key to remove the top-level mirror.

## Solution Assessment

No, not yet. The PR is the right narrow SDK seam, but cleanup should persist or otherwise retain slot ownership metadata so reset/delete/disable can clear mirrors even when the original plugin registration is gone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear promoted slots after plugin removal:** `src/plugins/host-hook-cleanup.ts:213-215`
  - body: `runPluginHostCleanup` now discovers promoted slot keys only from the supplied or active registry. If a plugin wrote `SessionEntry[slotKey]` and is later removed while OpenClaw is down, reset/delete cleanup runs with a registry that no longer contains that registration; `pluginExtensions` is cleared, but the top-level mirror cannot be identified and survives as stale plugin-owned state. Persist slot ownership or otherwise make cleanup remove orphaned promoted mirrors without relying only on live registry metadata.
