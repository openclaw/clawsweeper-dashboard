## Summary

Keep open. Current main does not have the requested SessionEntry slot projection or trusted-policy session-extension read surface, and the PR is the active implementation slice. The latest head fixes the earlier config-plumbing problems across gateway, MCP, native relay, and client-tool paths, but one low-priority guard gap remains in the promoted slot-key validation.

## What This Changes

The PR adds plugin-owned SessionEntry slot projection for session extensions, scoped trusted-policy session extension reads, config plumbing for before-tool surfaces, contract coverage, SDK API baseline updates, and a changelog entry.

## Best Possible Solution

Keep the additive SDK mirror design, but reserve inherited Object property names before merge, then re-run the targeted plugin contract/API/lint checks and let normal maintainer review decide the public SDK surface.

## Reproduction Assessment

Yes. A static PR-head reproduction is clear: register a session extension with sessionEntrySlotKey set to an inherited Object name such as toString, then patch it; the current validator accepts that identifier and the mirror writer creates an own top-level SessionEntry property.

## Solution Assessment

Mostly yes. The PR is a narrow maintainable way to add SessionEntry mirrors and scoped trusted-policy reads without exposing host config publicly, but the slot-key guard should also reject Object prototype names to match the stated collision-safety goal.
