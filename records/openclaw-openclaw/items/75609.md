## Summary

Keep open: current main does not implement this Plugin SDK slot-projection surface, and the latest PR head still has a source-reproducible restart cleanup bug that can delete preserved promoted session slots.

## What This Changes

The PR adds plugin-owned `SessionEntry` slot projection and scoped trusted-policy session-extension reads, with registry validation, cleanup/state handling, SDK baseline, changelog, and focused tests.

## Best Possible Solution

Keep the additive SDK seam, but make restart cleanup use the computed stale slot set exactly and preserve still-declared mirrors and ownership metadata.

## Reproduction Assessment

Yes. Source inspection of PR head shows `cleanupReplacedPluginHostRegistry` computes an exact stale-slot set, but `clearPromotedSessionEntrySlots` expands it with every persisted slot key for the plugin before deletion.

## Solution Assessment

No, not yet. The API direction is narrow, but the restart cleanup implementation should delete only obsolete promoted slots and prune only those ownership entries, with mixed stale/preserved namespace coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep restart slot cleanup scoped to stale slots:** `src/plugins/host-hook-cleanup.ts:56-58`
  - body: On restart, callers pass only the promoted slot keys removed from the replacement registry, but this helper expands that set with every persisted `pluginExtensionSlotKeys` value for the plugin before deleting fields. If one namespace drops its slot while another keeps a different slot, cleanup deletes the still-declared mirror and leaves the dropped namespace's ownership metadata behind; use the restart set as the exact target and prune only those ownership entries.

## Repair Work Prompt

On PR #75609 (`split/plugin-sdk-session-entry-projection`), fix the restart promoted `SessionEntry` slot cleanup bug. The caller already computes stale promoted slot keys as previous registry minus next registry; make the restart cleanup path delete exactly those stale top-level fields and prune only their `pluginExtensionSlotKeys` ownership entries, while preserving still-declared promoted mirrors and their metadata. Add a regression in `src/plugins/contracts/session-entry-projection.contract.test.ts` where one plugin has two session-extension namespaces, one drops its `sessionEntrySlotKey` on restart and the other keeps a different slot; assert the stale slot is removed, the preserved slot remains, plugin extension state remains, and only stale ownership metadata is removed. Do not broaden restart cleanup to delete `pluginExtensions`, scheduler jobs, or unrelated plugin-owned state. Keep the existing changelog entry unless the user-facing API text changes.
