## Summary

Keep open: current main does not implement this SDK seam, and the latest PR head still has a source-reproducible restart cleanup bug that can delete still-declared promoted SessionEntry mirrors.

## What This Changes

The PR adds plugin-owned SessionEntry slot projection, scoped trusted-policy session-extension reads, cleanup/validation coverage, SDK API baseline updates, and a changelog entry.

## Best Possible Solution

Keep the additive SDK seam, but make restart cleanup use the computed stale slot set exactly and prune only obsolete ownership metadata while preserving still-declared mirrors.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the review finding: restart cleanup computes exact stale keys, then the helper unions them with all persisted plugin-owned slot keys before deletion. I did not run tests in this read-only review.

## Solution Assessment

No, not yet. The API direction is narrow, but the restart cleanup implementation should update only obsolete promoted slots and ownership entries rather than deriving deletion from every persisted slot owned by the plugin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep restart slot cleanup scoped to stale slots:** `src/plugins/host-hook-cleanup.ts:56-58`
  - body: On restart, callers pass only the promoted slot keys removed from the replacement registry, but this helper expands that set with every persisted `pluginExtensionSlotKeys` value for the plugin before deleting fields. If one namespace drops its slot while another keeps a different slot, cleanup deletes the still-declared mirror and leaves the dropped namespace's ownership metadata behind; use the restart set as the exact target and prune only those ownership entries.
