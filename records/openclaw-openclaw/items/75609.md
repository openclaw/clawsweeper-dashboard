## Summary

Keep open. This is an active additive Plugin SDK PR, and source inspection found one remaining PR-introduced stale-data path on plugin restart; earlier duplicate-slot, config exposure, and config-plumbing findings appear addressed in the current head.

## What This Changes

The PR adds plugin-owned SessionEntry slot projection, scoped trusted-policy session extension reads, config plumbing across tool surfaces, API baseline/changelog updates, and contract coverage.

## Best Possible Solution

Keep the additive SDK direction, but reconcile promoted SessionEntry mirrors during plugin restart/replacement by clearing stale old slots and reprojecting any current slots from preserved pluginExtensions without deleting restart-preserved plugin state.

## Reproduction Assessment

Yes, at source level. The PR head skips persistent cleanup for restart while adding top-level slot mirrors, and the existing restart contract preserves plugin session state, leaving old mirrors stale when the next plugin registration removes or changes the slot.

## Solution Assessment

No, not yet. The API shape and guarded mirror approach look maintainable, but the implementation should not land until restart/reload keeps promoted mirrors consistent with the active plugin registration.
