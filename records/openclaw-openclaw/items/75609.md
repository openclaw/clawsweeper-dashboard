## Summary

Keep open: the PR is a useful focused SDK slice, but source review found a restart cleanup gap that can leave stale promoted top-level SessionEntry slots after a plugin reload.

## What This Changes

The PR adds plugin session-extension slot projection, scoped trusted-policy session-extension reads, config threading for policy lookup across tool surfaces, API baseline updates, tests, and a changelog entry.

## Best Possible Solution

Fix the restart stale-slot cleanup gap, keep the additive SDK surface guarded and mirror-only, then land after maintainer API review and exact-head validation.

## Reproduction Assessment

Not applicable as a user bug, but the PR defect is source-reproducible: the branch writes top-level slot mirrors on patch, then the plugin-restart cleanup path skips persistent store cleanup, leaving old mirrors if a later plugin version removes or renames the slot.

## Solution Assessment

No, not yet. The overall direction is maintainable, but restart handling needs to clear or reproject promoted slot mirrors while preserving the documented durable pluginExtensions state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear promoted slots on plugin restart:** `src/plugins/host-hook-cleanup.ts:181-189`
  - body: `patchPluginSessionExtension` now persists a top-level mirror, but when a loaded plugin is replaced, `cleanupReplacedPluginHostRegistry` calls this path with `reason: "restart"` for plugins that still exist. This branch collects the previous slot keys and then skips persistent cleanup for restarts, so if the new plugin version drops or renames `sessionEntrySlotKey` or changes its projector, the old `SessionEntry[slotKey]` remains visible until a later patch. Clear or reproject promoted slots during restart while preserving the documented durable `pluginExtensions` state.
