## Summary

Keep open. Current main still lacks the requested Plugin SDK SessionEntry slot projection, and this PR is the active implementation slice, but the latest head still has a blocking policy-enforcement gap: session-extension reads depend on an optional config value and still return `undefined` on the native hook relay pre-tool path. Prior review threads usefully tightened duplicate slot registration, cleanup, Gateway/MCP config plumbing, and type drift, but one surface plus the inherited Object-name slot guard still need repair before merge.

## What This Changes

The PR adds plugin-owned `SessionEntry` slot projection for session extensions, scoped trusted-policy session-extension reads, cleanup/registry guards, focused contract tests, SDK API baseline updates, and a changelog entry.

## Best Possible Solution

Keep the additive SDK mirror design, but make trusted-policy session-extension reads reliable on every before-tool-call path that carries a session key, then finish the slot-key guard so plugin-owned mirrors cannot shadow inherited Object names.

## Reproduction Assessment

Yes. A static reproduction path is enough: register a trusted policy that blocks when `ctx.getSessionExtension("policy")` returns state, persist that plugin session extension, then invoke the native pre-tool relay with a `sessionKey`; the PR head passes no config there, so the new helper returns `undefined` and the policy does not block.

## Solution Assessment

No. The proposed API direction is narrow and maintainable, but the current implementation should not silently disable policy state reads when config is absent; the safer repair is to plumb/fallback config internally without exposing full host config on the public plugin hook context.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Provide config for all session-extension policy reads:** `src/plugins/trusted-tool-policy.ts:31-37`
  - body: The new `getSessionExtension` path returns `undefined` whenever `options?.config` is absent. `runNativeHookRelayPreToolUse` still calls `runBeforeToolCallHook` with a `sessionKey` but no config, so a trusted policy that blocks or requires approval based on session extension state is bypassed for native `pre_tool_use` calls. Plumb an internal config/fallback through that surface, or make the lookup config-independent, and cover the native relay path.
  - confidence: 0.9
- **[P3] Reserve inherited Object slot names:** `src/plugins/session-entry-slot-keys.ts:3-6`
  - body: `normalizeSessionEntrySlotKey` rejects declared `SessionEntry` fields plus `__proto__`, `constructor`, and `prototype`, but inherited names like `toString`, `hasOwnProperty`, and `valueOf` still pass the identifier regex and can be written as own top-level mirrors. Add the inherited Object names to the reserved set, or derive them, and add focused contract coverage.
  - confidence: 0.84
