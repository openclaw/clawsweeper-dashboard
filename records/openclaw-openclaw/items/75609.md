## Summary

Keep this PR open: current main does not have the requested SessionEntry slot projection surface, and the PR is useful, but the latest head still leaves stale slot-ownership metadata after a restart where a plugin drops its promoted slot.

## What This Changes

The PR adds plugin-owned SessionEntry slot projection, scoped trusted-policy session extension reads, cleanup/validation hardening, tests, SDK API baseline updates, and a changelog entry.

## Best Possible Solution

Keep the additive SDK seam, but make restart cleanup remove or update obsolete `pluginExtensionSlotKeys` ownership metadata while preserving plugin extension state and same-slot restart mirrors.

## Reproduction Assessment

Yes, source-reproducible: plugin A can write a promoted slot, restart with the same namespace but no slot key, plugin B can later claim that freed slot, and cleanup of plugin A can still delete plugin B's mirror because A's stale `pluginExtensionSlotKeys` entry remains.

## Solution Assessment

No, not yet. The PR direction is narrow and maintainable, but restart cleanup needs to treat persisted slot ownership as current ownership metadata, not leave tombstones that future cleanup consumes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Drop stale slot ownership on restart:** `src/plugins/host-hook-cleanup.ts:271-276`
  - body: When a restarted plugin drops or changes `sessionEntrySlotKey`, this branch clears the old top-level mirror but leaves that plugin's persisted `pluginExtensionSlotKeys` entry behind. If another plugin later claims the now-free slot, ordinary cleanup for the first plugin will collect the stale stored key and delete the new plugin's mirror, so restart cleanup should remove or update obsolete ownership metadata when it clears obsolete promoted slots.
