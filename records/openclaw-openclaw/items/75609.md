## Summary

Keep open. Current main still lacks the requested SessionEntry slot-projection API, and this PR is the active implementation slice, but the latest head still leaves inherited Object prototype names valid as promoted slot keys and the branch is dirty against main.

## What This Changes

The PR adds plugin-owned `sessionEntrySlotKey`/`sessionEntrySlotSchema` metadata, mirrors projected plugin session-extension state into guarded top-level `SessionEntry` fields, adds cleanup and contract coverage, exposes scoped trusted-policy session-extension reads, and refreshes changelog/API baseline files.

## Best Possible Solution

Keep the additive slot-projection and scoped trusted-policy read design, but finish the slot-key hardening by rejecting inherited Object prototype names, rebase the branch onto current main, and require the targeted Plugin SDK tests plus API/lint checks to pass on the exact head.

## Reproduction Assessment

Yes. Static inspection is enough: `normalizeSessionEntrySlotKey("toString")` passes the PR head regex and reserved-set check, and `patchPluginSessionExtension` would write the projected value as an own top-level `SessionEntry.toString` property.

## Solution Assessment

Mostly yes. The PR’s additive API and internal trusted-policy config path are the narrow maintainable direction, but the reserved-key implementation should derive from `Object.prototype` or explicitly include those inherited names before merge.

## Review Findings

Full review comments:

- **[P3] Reserve Object prototype slot names:** `src/plugins/session-entry-slot-keys.ts:3-6`
  - body: `normalizeSessionEntrySlotKey` rejects only `__proto__`, `constructor`, `prototype`, and declared `SessionEntry` fields.
