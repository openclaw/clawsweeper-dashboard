## Summary

Keep open. Current main still lacks the requested Plugin SDK slot-projection surface, and this PR is the active implementation slice; most prior review threads appear addressed, but the slot-key guard still accepts inherited Object method names and the exact head is not merge-ready yet.

## What This Changes

The PR adds plugin-owned `SessionEntry` slot projection for session extensions, scoped trusted-policy session-extension reads, cleanup/registry hardening, contract coverage, API baseline updates, and a changelog entry.

## Best Possible Solution

Keep the additive Plugin SDK design, but harden `sessionEntrySlotKey` validation against inherited Object names, add a focused contract test for that guard, then rebase and validate the exact head before merge.

## Reproduction Assessment

Yes. Static inspection is enough: on PR head, `normalizeSessionEntrySlotKey("toString")` passes the identifier regex and is absent from the reserved set, so a plugin can register it as a promoted top-level slot.

## Solution Assessment

Mostly yes. The additive slot-projection and scoped trusted-policy read path fits the existing host-hook architecture; the safer final form is to derive or explicitly include inherited Object prototype names in the reserved-key set before merge.
