## Summary

Keep open. Current main does not yet implement the proposed Plugin SDK session-entry slot seam, and this draft PR is the active fresh-main slice replacing the closed consolidated stack, but the latest patch still needs changes for slot ownership collisions and trusted-policy session reads on direct tool invocation paths.

## What This Changes

The PR adds sessionEntrySlotKey/sessionEntrySlotSchema to plugin session extensions, mirrors projected extension values into guarded top-level SessionEntry slots, exposes scoped session-extension reads to trusted tool policies, and updates tests, changelog, and the plugin SDK API baseline.

## Best Possible Solution

Keep this as the active review surface, but make the additive SDK seam enforce unique normalized top-level slot ownership, preserve trusted-policy session reads across every production tool-call path that has cfg/sessionKey, and keep the API baseline, changelog, and focused contract tests aligned.

## Reproduction Assessment

Yes. A static reproduction is clear: register two plugins with the same sessionEntrySlotKey, patch both namespaces, and the second write overwrites the shared top-level slot; cleanup for one plugin can then delete the slot while the other plugin state still exists. A second static check shows tools.invoke has cfg but does not pass config into runBeforeToolCallHook, so trusted-policy getSessionExtension returns undefined on that surface.

## Solution Assessment

No, not as-is. The overall additive SDK direction is the right narrow surface, but the implementation needs registry-level slot-key uniqueness and complete config propagation for trusted policy reads before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject duplicate SessionEntry slot keys:** `src/plugins/registry.ts:1528-1532`
  - body: The registry validates that a slot key is shaped correctly and does not collide with core SessionEntry fields, but it still accepts the same sessionEntrySlotKey from multiple plugins. Since patchPluginSessionExtension writes directly to entry[slotKey] and cleanup deletes collected slot keys, one plugin can overwrite or remove another plugin's mirror while that other plugin's extension state still exists. Reject duplicate normalized slot keys and add a regression test.
  - confidence: 0.9
- **[P2] Pass config to direct trusted-policy hooks:** `src/plugins/trusted-tool-policy.ts:29-31`
  - body: getSessionExtension returns undefined whenever ctx.config is absent, but the Gateway tools.invoke path already has cfg and calls runBeforeToolCallHook without adding it to ctx. A trusted policy that can read session extension state during ordinary agent tool calls loses that state on direct tool invocation, which makes policy behavior surface-dependent. Thread cfg through this hook context and cover the tools.invoke path.
  - confidence: 0.82
