## Summary

This looks like a valid current-main bug with a small, well-bounded fix: the ACP client subcommand still logs caught failures with `String(err)`, while the parent ACP command already uses the shared formatter.

## What This Changes

Use `formatErrorMessage` for ACP client subcommand failures instead of `String(err)`, with focused regression coverage.

## Best Possible Solution

Replace only the ACP client catch handler with `formatErrorMessage(err)` and add a regression test that mocks a plain-object rejection.

## Reproduction Assessment

Yes. Source inspection shows `runAcpClientInteractive` is awaited inside a catch block that logs `String(err)`, so a mocked rejection with `{ code: 42 }` will produce `[object Object]` instead of the formatter output.

## Solution Assessment

Yes. Using the already-imported `formatErrorMessage` is the narrowest maintainable fix and aligns the client subcommand with the parent ACP command.
