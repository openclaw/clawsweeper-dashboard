## Summary

Keep open: current main still has the requested Telegram polling and session persistence seams, but it does not expose `polling_lifecycle` or `session_persistence` hook families to plugins. Existing session/transcript events are adjacent but do not provide the before/after persistence, watchdog, restart, or timing payloads requested here, so this remains a new plugin API contract rather than a cleanup-close candidate.

## What This Changes

Expose plugin-observable polling lifecycle and session persistence hook events for instrumentation plugins.

## Best Possible Solution

Define a stable, generic plugin hook contract for channel liveness/polling and session persistence telemetry, then wire Telegram and session-store/transcript emitters with focused tests and public plugin docs.

## Reproduction Assessment

Not applicable; this is a feature request for new plugin-observable lifecycle events, not a broken existing behavior. Source inspection shows the named polling and persistence seams exist on current main but the matching hook families do not.

## Solution Assessment

Unclear; the requested direction is plausible, but the narrow maintainable solution needs maintainer decisions on generic channel coverage, event names, stable payloads, privacy, and failure policy before implementation.
