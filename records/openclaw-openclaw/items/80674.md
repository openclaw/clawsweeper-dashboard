## Summary

Keep open: current main and v2026.6.8 still expose adjacent compaction, session, message, subagent, and transcript-update hooks, but not the requested polling lifecycle or before/after session persistence hook families. ## What This Changes

Request to add plugin hook events around Telegram or channel polling watchdog cycles and session-store/transcript persistence boundaries.

## Best Possible Solution

Define a minimal, security-reviewed plugin observability contract for polling and persistence boundaries, reuse existing typed hook policy/timeout/access gates and transcript update subscriptions where possible, then implement only the approved generic seams.

## Reproduction Assessment

Yes, at source level: current hook type unions, docs, Telegram polling code, session-store save code, and transcript event helpers lack the requested polling_lifecycle and session_persistence hook families. This is a missing capability rather than a broken existing contract.

## Solution Assessment

Unclear pending maintainer design. Dedicated hook families are plausible, but maintainers need to choose generic versus Telegram-specific ownership, payload minimization, timeout/failure semantics, and how this relates to existing transcript update subscriptions and durable-write hook proposals.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it proposes new plugin/workspace hook execution points with access to polling, session, transcript, timing, and path metadata.

Concerns:

- **[medium] Minimize trusted hook payloads:** `src/hooks/loader.ts:46`
  - body: Workspace hooks are loaded as trusted local code in the Gateway process, so any new lifecycle event should expose only bounded fields and avoid raw transcripts, credentials, private channel identifiers, or unnecessary filesystem paths unless maintainers explicitly accept that contract.
  - confidence: 0.9
- **[medium] Define hot-path hook timing and failure semantics:** `src/hooks/internal-hooks.ts:286`
  - body: Internal hook dispatch awaits handlers in registration order; firing equivalent hooks from polling watchdog or session-store write paths needs clear timeout, error, and non-blocking behavior so observability cannot degrade transport or persistence reliability.
  - confidence: 0.88
- **[medium] Align with plugin access gates:** `src/plugins/registry.ts:2555`
  - body: The typed plugin registry already gates conversation-sensitive hooks for non-bundled plugins; polling and transcript persistence events need an equivalent policy decision if they expose session or conversation metadata.
  - confidence: 0.82
