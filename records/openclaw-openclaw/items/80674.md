## Summary

Keep open: current main still has the underlying Telegram polling, session-store, transcript, and hook infrastructure, but the requested plugin-observable polling and session persistence hook families are not implemented and would add a new public plugin API contract.

## What This Changes

The issue requests new plugin-observable hook families for polling lifecycle events and session persistence/transcript boundaries.

## Best Possible Solution

Define a stable, generic plugin hook contract for channel/polling liveness and session persistence telemetry, then wire Telegram and session-store emitters with focused tests, SDK exports, and plugin docs.

## Reproduction Assessment

Not applicable; this is a feature request for new plugin API events, not a broken existing behavior. Source inspection shows the named operational seams exist and are not exposed as hook families on current main.

## Solution Assessment

Unclear; the requested direction is plausible, but the narrow maintainable solution needs maintainer decisions on event names, payload contracts, privacy, and whether polling events should be generic channel lifecycle hooks rather than Telegram-only hooks.
