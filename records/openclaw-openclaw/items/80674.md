## Summary

Keep open: current main has the Telegram polling watchdog, session-store writes, transcript rewrite/rotation, and existing hook infrastructure, but the requested plugin-observable polling and persistence event families are not implemented and would extend the plugin API contract.

## What This Changes

Feature request to expose plugin-observable polling lifecycle and session persistence hook events.

## Best Possible Solution

Define a stable, generic plugin hook contract for polling/channel liveness and session persistence telemetry, then wire Telegram and session-store emitters with focused tests and docs.

## Reproduction Assessment

Not applicable; this is a feature request for new plugin-observable lifecycle events, not a broken existing behavior. Source inspection shows the named operational seams exist and are not currently exposed as hook families.

## Solution Assessment

Unclear; the requested direction is plausible, but the best implementation needs maintainer decisions on event naming, generic channel coverage, payload stability, privacy, and hook failure policy before code should be queued.
