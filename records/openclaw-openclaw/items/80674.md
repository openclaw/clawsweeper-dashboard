## Summary

Keep open. Current main has adjacent plugin hooks, Telegram polling diagnostics, and session persistence operations, but it does not expose the requested `polling_lifecycle` or `session_persistence` hook families; this is new public API work that needs maintainer design rather than cleanup closure.

## What This Changes

The issue requests plugin-observable polling lifecycle and session persistence hook events for correlating Telegram polling stalls with session-store and transcript IO.

## Best Possible Solution

Design an additive, documented plugin hook contract for sanitized polling and session persistence lifecycle events, with generic seams where possible, Telegram-owned details where necessary, and focused tests/docs for dispatch ordering, timeout, and payload boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives high confidence that current main lacks the requested hook names while the relevant polling and persistence seams exist.

## Solution Assessment

Unclear as proposed. The direction is plausible, but hook names, payload shape, privacy gates, dispatch timing, and whether the polling side should be channel-generic require maintainer product/API judgment.
