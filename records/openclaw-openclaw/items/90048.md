## Best Possible Solution

Centralize the scoped, guarded provider-row projection and reuse it from both `resolvePluginProviders` and `findProviderRuntimePluginInRegistry`, with regression coverage for list resolution and direct provider runtime lookup.

Yes, source inspection gives a high-confidence reproduction path: current main directly maps provider registration rows, so a throwing `provider` getter aborts resolution. I did not run tests because this review is read-only.

## Solution Assessment

No: the helper is a good fix shape for `resolvePluginProviders`, but the same projection invariant still needs to cover the loaded-registry hook lookup. The narrowest maintainable solution is to share one guarded projection path across both runtime entry points.
