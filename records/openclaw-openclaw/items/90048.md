Keep open: current main still projects provider registry rows directly, and this draft maintainer PR only guards `resolvePluginProviders`; the loaded-registry provider hook lookup can still throw on the same malformed row shape.

## What This Changes

The PR adds a guarded provider registration projection helper in `src/plugins/providers.runtime.ts` and a regression test for unreadable and out-of-scope provider rows in `resolvePluginProviders`.

## Best Possible Solution

Share one guarded, scoped provider-row projection across `resolvePluginProviders` and loaded provider hook lookup, then cover both list resolution and `resolveProviderRuntimePlugin` active-registry paths.

## Reproduction Assessment

Yes. Source inspection gives high confidence: current main maps `registry.providers` directly in both `resolvePluginProviders` and `findProviderRuntimePluginInRegistry`, so a throwing provider getter aborts resolution before later healthy rows are considered.

## Solution Assessment

No. The helper is a good fix shape for provider list resolution, but the best fix is to reuse the same guarded projection in direct loaded-registry hook lookup as well.
