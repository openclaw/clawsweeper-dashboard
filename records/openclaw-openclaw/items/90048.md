## Summary

This PR should stay open: the provider-row projection helper is useful, but the fix is incomplete because loaded active/channel provider hook lookup still dereferences provider rows directly before filtering. The maintainer-labeled draft needs the sibling hook path repaired and rerun validation, not cleanup closure.

## What This Changes

The branch adds guarded provider-registration projection in `src/plugins/providers.runtime.ts` and a regression test in `src/plugins/providers.test.ts` for unreadable and out-of-scope runtime provider rows.

## Best Possible Solution

Share one guarded, scoped provider-row projection across `resolvePluginProviders` and loaded provider hook lookup, with provider-runtime coverage for active/channel registries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: put an unreadable provider registration before a healthy matching row in a loaded active or channel registry, then call a provider hook such as `prepareProviderExtraParams`; the loaded-registry scan reads `entry.provider` directly before it can reach the healthy row.

## Solution Assessment

No. Guarding `resolvePluginProviders` is a good direction, but the best fix must apply the same guarded/scoped projection to loaded provider hook lookup and prove that sibling entry point with `provider-runtime` coverage.
