## Summary

Keep open. Current main and the latest release still collapse Telegram voice notes into generic audio/transcript context, and no open canonical PR or broader issue supersedes this focused request for agent-visible source provenance.

## What This Changes

Expose whether an inbound agent turn originated from a voice message, preferably through a generic source-modality/source-traits context field rather than a Telegram-only flag.

## Best Possible Solution

Add a backward-compatible generic inbound origin contract, populate it from Telegram before STT collapses voice to text, render it in agent-visible metadata, and document/test voice notes versus ordinary audio and typed text.

## Reproduction Assessment

Yes, as a source-level reproduction: a Telegram `msg.voice` turn is normalized to `<media:audio>`, STT produces only transcript/index metadata, and the final inbound metadata contains no voice or modality field. I did not run a live Telegram turn because this review is read-only and the missing plumbing is explicit in current source.

## Solution Assessment

Yes. The maintainable direction is the generic `source_modality`/`source_traits` approach raised in discussion, rather than a Telegram-only `is_voice` flag or raw Telegram message exposure.
