## Summary

Keep open. Current main still normalizes Telegram voice notes into generic audio/transcript context, and the shared inbound metadata rendered to agents has no voice-note or native message-origin marker; the broader Telegram metadata issue remains open but does not supersede this narrower request.

## What This Changes

Expose whether an inbound message originated as a Telegram voice note rather than typed text or generic audio in agent-visible context metadata.

## Best Possible Solution

Add a backward-compatible generic inbound message-origin field, populate it from Telegram before msg.voice is normalized to generic audio, render it for agents, and document/test voice-note versus text/audio behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is an additive metadata request. Source inspection gives a high-confidence verification path: a Telegram voice message currently reaches STT/media context but no agent-visible voice-origin field is carried or rendered.

## Solution Assessment

Yes. The narrow maintainable path is a bounded generic message-origin field in shared inbound context, not exposing raw Telegram message objects or adding a Telegram-only prompt convention.
