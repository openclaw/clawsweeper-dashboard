## Summary

Keep open. Current main still treats Telegram voice notes as generic audio/transcript turns and does not expose a voice-origin or native message-type field in the shared context or rendered inbound metadata; broader #40838 is related but does not supersede this narrower voice-turn-taking request.

## What This Changes

Expose whether an inbound agent turn originated from a Telegram voice note, preferably through a generic message-type or modality field in agent-visible inbound context.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to the shared context contract, populate it from Telegram before msg.voice is normalized to generic audio, render it for agents in dynamic conversation metadata, and document it with regression coverage for voice notes versus ordinary audio/text.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a Telegram msg.voice reaches STT and MediaTranscribedIndexes, but current source and tests show no agent-visible voice-origin or native message-type field in MsgContext or rendered inbound metadata.

## Solution Assessment

Yes. The narrow maintainable direction is additive context plumbing with a generic field shape; dumping raw Telegram message JSON or adding Telegram-only prompt prose would be less stable and harder for other channel plugins to follow.
