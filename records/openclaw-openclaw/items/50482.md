## Summary

Keep open. Current main still collapses Telegram voice notes into generic audio/transcript context, and neither the shared message context nor the rendered inbound metadata exposes a voice-origin or native message-type marker. Related #40838 is broader Telegram metadata work and can inform the field shape, but it does not supersede this narrower voice-note turn-taking request.

## What This Changes

Expose whether an inbound agent turn originated from a Telegram voice note, preferably through a generic message-type or modality field in the agent-visible inbound context.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to the shared context contract, populate it from Telegram before msg.voice is normalized to generic audio, render it for agents in the dynamic conversation metadata, and document it with regression coverage for voice notes versus ordinary audio/text.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a Telegram msg.voice reaches STT and MediaTranscribedIndexes, but current source and tests show no agent-visible voice-origin or native message-type field in MsgContext or rendered inbound metadata.

## Solution Assessment

Yes. The best direction is narrow additive context plumbing with a generic field shape; a Telegram-only prompt string or exposing raw Telegram message JSON would be less maintainable and harder for other channel plugins to follow.
