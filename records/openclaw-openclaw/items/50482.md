## Summary

Keep open. Current main still transcribes Telegram voice notes but does not expose a voice-origin or native message-type marker to the shared context or rendered inbound metadata, and related #40838 is broader Telegram metadata work rather than a superseding fix.

## What This Changes

Expose whether an inbound agent turn originated from a voice note rather than typed text or generic audio.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin/modality field, populate it from Telegram before voice notes collapse to generic audio, render it in agent-visible metadata, and document it with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: msg.voice reaches Telegram STT and MediaTranscribedIndexes, but the shared context and rendered inbound metadata have no voice-origin or native message-type field.

## Solution Assessment

Yes. The maintainable solution is a narrow additive generic context field; raw Telegram JSON or Telegram-only prompt prose would be less stable and harder for other channel plugins to follow.

## Repair Work Prompt

Implement the voice-origin context request from this issue. Current main transcribes Telegram msg.voice but loses the voice-note origin before agent-visible metadata is rendered. Add a backwards-compatible generic field to the shared inbound context contract, populate it from Telegram when msg.voice is present before resolveTelegramPrimaryMedia normalizes it to <media:audio>, and render it in the dynamic conversation metadata for agents. Keep the naming generic enough for other channels; do not dump raw Telegram message JSON, add a config option, or change STT/mention-gating behavior. Add regression coverage for Telegram voice notes versus regular audio/text, update docs, and add a user-facing CHANGELOG.md entry under the active version.
