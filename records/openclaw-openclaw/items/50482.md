## Summary

Keep open. Current main still normalizes Telegram voice notes into generic audio/transcript context and does not expose an agent-visible voice-origin or native message-type marker; related #40838 is adjacent broader Telegram metadata work, not a replacement for this voice turn-taking request.

## What This Changes

Expose whether an inbound Telegram message originated as a voice note, preferably through a generic message-origin or modality field visible to agents.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to shared context, populate it from Telegram before voice is normalized to generic audio, render it in agent-visible conversation metadata, and document/test voice versus ordinary audio/text behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a Telegram msg.voice reaches STT and MediaTranscribedIndexes, but current source and tests show no agent-visible voice-origin or native message-type field in MsgContext or rendered inbound metadata.

## Solution Assessment

Yes. The narrow maintainable solution is additive generic context plumbing; dumping raw Telegram message JSON or adding Telegram-only prompt prose would be less stable for other channel plugins.
