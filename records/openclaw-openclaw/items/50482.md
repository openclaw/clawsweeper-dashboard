## Summary

Keep open. Current main and the latest release still normalize Telegram voice notes into generic audio/transcript context without an agent-visible source modality, source traits, or voice-origin marker; the broader Telegram metadata issue is adjacent but does not supersede this focused request.

## What This Changes

Expose a voice/source-modality indicator in inbound agent context so agents can distinguish dictated voice notes from typed text after STT.

## Best Possible Solution

Add a backward-compatible generic source-modality and source-traits contract, populate it from Telegram before voice is normalized to audio, render it in inbound metadata, and document/test voice notes against ordinary audio and text.

## Reproduction Assessment

Yes for source-level verification, not as a runtime bug. Current `origin/main` shows Telegram voice notes are normalized to generic audio/transcript context and rendered inbound metadata lacks a voice/source-modality marker.

## Solution Assessment

Yes. The maintainable path is a generic shared source-modality/traits field populated by Telegram, not raw Telegram message exposure or a Telegram-only `is_voice` core concept.
