## What This Changes

Expose voice-note/source-modality provenance in the inbound agent context after Telegram audio is transcribed.

## Best Possible Solution

Add a backwards-compatible generic source-provenance contract, populate it from Telegram before voice is normalized to audio, render it in dynamic agent metadata, and cover voice notes versus ordinary audio and typed text.

## Reproduction Assessment

Yes. A source-level reproduction is clear: Telegram voice notes become generic audio/transcript context, and current main has no voice/source-modality field in the shared context or rendered metadata.

## Solution Assessment

Partly. Exposing provenance is the right direction, but the discussion-backed generic `source_modality` plus `source_traits` shape is more maintainable than a Telegram-only `is_voice` flag.
