## Summary

Keep open. Current main still collapses Telegram voice notes into generic audio/transcript context and does not expose a voice-origin or native message-type marker to the agent; the prior ClawSweeper comments and related #40838 are useful context, but #40838 is broader sender/message metadata work and does not supersede this focused voice-turn-taking request.

## What This Changes

Expose whether an inbound agent turn originated from a Telegram voice note, preferably through a generic message-type or modality field in agent-visible inbound context.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to the shared context contract, populate it from Telegram before msg.voice is normalized to generic audio, render it for agents in volatile inbound conversation metadata, and document it with regression coverage for voice notes versus ordinary audio/text.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a Telegram msg.voice reaches STT and MediaTranscribedIndexes, but current source and tests show no agent-visible voice-origin or native message-type field in MsgContext or rendered inbound metadata.

## Solution Assessment

Yes, if implemented as additive generic context plumbing. A raw Telegram JSON dump or Telegram-only prompt prose would be less maintainable and harder for other channel plugins to adopt.
