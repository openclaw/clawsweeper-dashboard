## Summary

Keep open. Current main still collapses Telegram voice notes into generic audio/transcript context and does not expose a voice-origin or native message-type marker; the adjacent open Telegram metadata issue can inform naming but does not supersede this narrower turn-taking request.

## What This Changes

Expose whether an inbound agent turn originated from a Telegram voice note, preferably through a generic message-type or modality field in agent-visible inbound context.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field, populate it from Telegram before voice is normalized to audio, render it for agents, and document it with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to follow a Telegram msg.voice through media resolution, STT preflight, final context assembly, and inbound metadata rendering; current main exposes only generic audio/transcript state.

## Solution Assessment

Yes. Additive shared context plumbing with a generic field is the narrow maintainable path; exposing raw Telegram JSON or prompt-only Telegram prose would be less stable for other channel plugins.
