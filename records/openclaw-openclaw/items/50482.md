## Summary

Keep open. Current main and v2026.6.6 still collapse Telegram voice notes into generic audio/transcript context, and the shared inbound context plus rendered agent metadata do not expose a voice/source-modality marker.

## What This Changes

The issue requests agent-visible inbound metadata that distinguishes voice notes from typed text or other source modalities.

## Best Possible Solution

Add a narrow generic inbound source-modality/message-origin field, populate Telegram voice-note origin before it collapses to generic audio, render only stable agent-useful metadata, and cover voice-note versus ordinary audio/text behavior with focused tests.

## Reproduction Assessment

Yes, at source level: Telegram `msg.voice` is available in current code, but it is projected as generic audio and the rendered inbound metadata has no voice/source-modality marker. I did not run live Telegram in this read-only review.

## Solution Assessment

Yes, if kept narrow: the maintainable solution is shared source-modality plumbing plus Telegram population and prompt metadata tests, not a Telegram-only prompt string or broad STT telemetry.
