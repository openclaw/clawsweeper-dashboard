## Summary

Keep open. Current main and v2026.5.19 still collapse Telegram voice notes into generic audio/transcript context, and the shared inbound context plus rendered metadata do not expose voice/source-modality provenance; the broader Telegram metadata issue can inform naming but does not replace this focused request.

## What This Changes

Expose whether an inbound agent turn originated from a voice note, preferably through generic source-modality metadata rather than a Telegram-only flag.

## Best Possible Solution

Add a backward-compatible generic `source_modality` plus `source_traits` contract, populate it from Telegram before voice is normalized to audio, render it in inbound metadata, and document/test voice notes against ordinary audio and text.

## Reproduction Assessment

Yes for source-level verification, not as a runtime bug. Current main shows Telegram voice notes are normalized to generic audio/transcript context and rendered inbound metadata lacks a voice or source-modality marker.

## Solution Assessment

Yes. The maintainable path is the generic `source_modality`/`source_traits` direction raised in discussion, not a Telegram-only `is_voice` core field or raw Telegram message exposure.
