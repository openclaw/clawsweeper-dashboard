## Summary

Keep open. Current main and v2026.6.6 still collapse Telegram voice notes into generic audio/transcript context without an agent-visible source-modality or voice-origin fact; the broader Telegram metadata thread at https://github.com/openclaw/openclaw/issues/40838 is useful naming context but does not supersede this narrower request.

## What This Changes

Request to expose whether an inbound Telegram audio message originated as a voice note in agent context metadata so agents can adjust turn-taking.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin/source-modality fact, populate it from Telegram before `msg.voice` is normalized to generic audio, and render the minimal agent-useful metadata in the dynamic conversation-info block with tests.

## Reproduction Assessment

Yes, at source level: Telegram `msg.voice` is available, but current main projects it as generic audio/transcript context without a voice-origin field. I did not run a live Telegram repro in this read-only cleanup review.

## Solution Assessment

Yes, if kept narrow and generic: the maintainable fix is shared source-modality plumbing plus Telegram population and prompt projection tests. A Telegram-only `is_voice` field in the core prompt would be a weaker long-term shape.
