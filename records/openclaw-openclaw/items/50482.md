## Summary

Keep open. Current main and the latest shipped release still collapse Telegram voice notes into generic audio/transcript context, and the agent-visible inbound metadata has no source-modality or voice-note provenance; the broader Telegram metadata issue at https://github.com/openclaw/openclaw/issues/40838 is related but does not supersede this focused request.

## What This Changes

Expose voice-note or generic source-modality provenance in inbound agent context so agents can distinguish dictated voice messages from typed text.

## Best Possible Solution

Add a backward-compatible generic source_modality/source_traits-style inbound context contract, populate it from Telegram before voice is normalized to audio, render it in agent-visible inbound metadata, and document/test voice notes against ordinary audio and text.

## Reproduction Assessment

Yes for source-level verification, not as a runtime bug. Current main shows Telegram voice notes are normalized to generic audio/transcript context and rendered inbound metadata lacks a voice or source-modality marker.

## Solution Assessment

Yes, with the generic shape discussed in the issue. A source_modality/source_traits contract is more maintainable than a Telegram-only is_voice field or raw Telegram message exposure.
