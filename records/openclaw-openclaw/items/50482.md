## Summary

Keep open. Current main and v2026.5.19 still collapse Telegram voice notes into generic audio/transcript context, and the agent-visible inbound metadata has no voice/source-modality provenance; the broader Telegram metadata issue is related but not a replacement for this focused request.

## What This Changes

Expose voice-note provenance, or a generic message source-modality field, in inbound agent context after Telegram STT transcription.

## Best Possible Solution

Add a backward-compatible generic source_modality/source_traits contract, populate it from Telegram before voice is normalized to audio, render it in inbound metadata, and document/test voice notes against ordinary audio and text.

## Reproduction Assessment

Yes for source-level verification, not as a runtime bug. Current main shows Telegram voice notes are normalized to generic audio/transcript context and rendered inbound metadata lacks a voice or source-modality marker.

## Solution Assessment

Yes. The maintainable path is the generic source_modality/source_traits direction raised in discussion, not a Telegram-only is_voice core field or raw Telegram message exposure.
