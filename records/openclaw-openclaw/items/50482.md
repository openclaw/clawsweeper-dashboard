## Summary

Keep open. Current main and v2026.5.22 still collapse Telegram voice notes into generic audio/transcript context, and the agent-visible inbound metadata has no native voice/modality marker. Related https://github.com/openclaw/openclaw/issues/40838 is broader sender/message metadata work and should inform naming, but it does not supersede this narrower modality provenance request.

## What This Changes

Expose whether an inbound agent turn originated from a voice message, preferably as generic source-modality metadata rather than a Telegram-only flag.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin/modality field, populate it from Telegram before voice is normalized to generic audio, render it in dynamic agent metadata, and cover voice notes versus typed text/audio with focused tests.

## Reproduction Assessment

Yes, as a source-level reproduction: a Telegram `msg.voice` turn is normalized to `<media:audio>`, STT produces only transcript/index metadata, and the final inbound metadata contains no voice or modality field. I did not run a live Telegram turn because this review is read-only and the missing plumbing is explicit in current source.

## Solution Assessment

Partly. Exposing provenance is the right direction, but a generic `source_modality` plus bounded traits is more maintainable than adding only `is_voice` to core metadata.
