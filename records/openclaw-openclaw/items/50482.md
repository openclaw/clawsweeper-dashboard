## Summary

Keep open. Current main and v2026.6.6 still expose Telegram voice notes as an audio transcript plus transcribed-media index, but they do not expose a voice-origin or source-modality fact to the agent; the broader Telegram metadata issue is useful context but does not supersede this narrower request.

## What This Changes

Requested exposing a voice-origin or modality flag in inbound agent context so agents can distinguish Telegram voice-note transcripts from typed text.

## Best Possible Solution

Add a shared inbound message modality/provenance seam, project Telegram `msg.voice` into it, and render the fact in agent-visible inbound metadata while keeping the broader sender/bot metadata issue separate.

## Reproduction Assessment

Yes, at source level: Telegram voice-note tests pass `msg.voice` and audio media through preflight transcription, while current `MsgContext` and inbound metadata expose only a transcript/media index and no voice/source-modality fact. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes, if implemented through the shared inbound context/media contract rather than a Telegram-only prompt hack. The m13v comment correctly points toward `source_modality` plus traits as the more maintainable shape than one-off flags.
