## Summary

Keep open. Current main still transcribes Telegram voice notes but only exposes generic audio/transcript context to the agent; neither the shared context type nor rendered inbound metadata carries a voice-origin or native message-type marker. Related #40838 is broader Telegram metadata work and does not supersede this narrower request.

## What This Changes

Expose whether an inbound agent context originated from a voice message, preferably through a generic message-origin or message-type field usable by Telegram and other channels.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or message-type field, populate it from Telegram before voice notes are normalized to audio, render it in agent-visible metadata, and cover voice versus audio/text with focused tests and docs.

## Reproduction Assessment

Yes for source-level reproduction. Current main shows msg.voice reaches Telegram media/STT handling, but the shared context and inbound metadata paths have no voice-origin or native message-type field to expose to the agent.

## Solution Assessment

Yes. A narrow additive generic context field is the maintainable direction; exposing raw Telegram JSON or adding Telegram-only prompt prose would be less stable and harder for other channels to follow.
