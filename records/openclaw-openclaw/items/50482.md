## Summary

Keep open. Current main and v2026.5.18 still collapse Telegram voice notes into generic audio/transcript context without an agent-visible voice or source-modality marker; https://github.com/openclaw/openclaw/issues/40838 is related broader metadata work, not a replacement for this focused request.

## What This Changes

The issue requests preserving voice-note origin in inbound agent metadata so agents can distinguish dictated voice messages from typed text.

## Best Possible Solution

Add a backward-compatible generic source-modality/source-traits contract, populate it from Telegram before voice notes are normalized to audio transcripts, render it in inbound user metadata, and cover voice-note versus typed/audio behavior in tests and docs.

## Reproduction Assessment

Yes for source-level verification: current main handles Telegram voice messages and STT, but the shared context and rendered inbound metadata omit any voice-origin or source-modality marker. No runtime test was run because this was a read-only review.

## Solution Assessment

Yes, if the fix uses a generic modality/source-traits field rather than raw Telegram message exposure or a one-off Telegram-only flag. The latest comment usefully points toward a durable schema shape that can also cover forwarded, edited, captioned, and media-origin cases.
