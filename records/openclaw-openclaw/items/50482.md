## Summary

Keep open. Current main and the latest release still collapse Telegram voice notes into generic audio/transcript context without an agent-visible voice-origin or source-modality marker, and the latest discussion usefully narrows the preferred shape to generic provenance fields rather than a Telegram-only flag.

## What This Changes

Request to expose inbound message provenance so agents can distinguish Telegram voice notes from typed text after STT normalization.

## Best Possible Solution

Add a backward-compatible generic inbound provenance contract, populate it from Telegram before STT/media normalization, render it in inbound metadata, and cover voice-note versus ordinary audio/text behavior in tests and docs.

## Reproduction Assessment

Yes for source-level verification: current Telegram voice handling reaches STT and generic media metadata, while shared context and rendered inbound metadata omit any voice-origin or source modality marker. This is not a runtime bug reproduction because the request is for a new agent-visible field.

## Solution Assessment

Yes. The maintainable solution is a generic shared source_modality/source_traits-style contract populated by Telegram before normalization, not raw Telegram message exposure or a prompt-only convention.
