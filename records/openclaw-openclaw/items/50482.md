## Summary

Keep open. Current main still collapses Telegram voice notes into generic audio/STT context and renders only generic route, conversation, and media metadata to the agent, so the requested voice-origin signal remains missing. The open #40838 is broader Telegram metadata work and can inform naming, but it does not supersede this narrower request.

## What This Changes

Expose whether an inbound Telegram message originated as a voice note in agent-visible inbound context metadata.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to the shared context contract, populate it from Telegram before msg.voice is normalized to generic audio, render it for agents in the volatile inbound conversation metadata, and document it with regression coverage for voice notes versus ordinary audio/text.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a Telegram msg.voice reaches STT and MediaTranscribedIndexes, but current source and tests show no agent-visible voice-origin or native message-type field in MsgContext or rendered inbound metadata.

## Solution Assessment

Yes. The narrow maintainable direction is additive context plumbing with a generic field shape; a Telegram-only prompt string or hard-coded core special case would be less reusable and harder for other channel plugins to follow.
