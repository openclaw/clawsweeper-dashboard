## Summary

Keep this open: the report is recent and plausibly points at a real Google provider/config-resolution bug, but current source evidence is mixed rather than closeable. The Google transport already suppresses `thinkingConfig` when the resolved model has `reasoning:false`, while the reported `compat.supportsThinking` flag is not part of the current config schema and Gemma 4 forward-compat resolution can force `reasoning:true`.

## What This Changes

The issue reports Google/Gemma model requests from Telegram failing with a Google 400 because OpenClaw appears to send thinking/reasoning parameters despite `reasoning:false` and compat opt-outs.

## Best Possible Solution

Add a focused Google-provider regression that drives the reporter's model config through model resolution into the outgoing Gemini request, then fix the narrow path that loses `reasoning:false` or improve diagnostics for invalid `models.json`/unsupported compat keys.

## Reproduction Assessment

No. Current source shows a plausible path for thinking payloads on Google/Gemma models, but it also shows `model.reasoning:false` should suppress them if the model override is actually resolved, so the exact current-main failure path is still unproven.

## Solution Assessment

Unclear. The maintainable fix is to verify whether the configured model override reaches the Google transport and only then adjust model resolution or payload shaping, instead of broadly stripping thinking from all Google requests.
