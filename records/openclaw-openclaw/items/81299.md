- `P1`: The PR addresses a reported LINE message-loss path where generated table Flex content can be silently dropped for users with exhausted Push quota.

## Summary

Keep open. This is a focused candidate fix for a real current-main LINE message-loss path: current main and v2026.6.8 still reply with text first and then Push the extracted table Flex message, while this PR is the active linked repair and is not superseded by the broader open LINE PRs.

## What This Changes

The PR changes LINE auto-reply delivery to batch rich/media messages with text chunks into the first Reply API call up to the five-message limit, then Push only overflow, with regression tests for batching, overflow, and Reply failure fallback.

## Best Possible Solution

Land a narrow LINE plugin batching fix once maintainers accept or obtain real LINE delivery proof and explicitly accept the mixed-message ordering tradeoff; keep broader LINE retry/quota/loading work in separate canonical PRs.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here. Current main extracts Markdown tables into Flex messages, sends text through Reply first, and then routes the Flex message through Push, matching the reporter's exhausted-Push-quota failure path.

## Solution Assessment

Yes for the narrow no-quick-replies bug: batching eligible rich/media/text before consuming the reply token is the right owner-local fix, while retry, quota monitoring, and loading animation belong to separate LINE reliability work. The main remaining solution question is whether maintainers accept rich/media-first ordering or want text-first batching before merge.
