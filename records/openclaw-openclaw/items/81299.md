## Summary

Keep open. This PR targets a real LINE message-loss path and the code change is plausibly in the right delivery layer, but the external proof still uses stubbed LINE send adapters and the merge changes which messages get the scarce Reply API slots when Push quota is exhausted.

## What This Changes

This PR changes LINE auto-reply delivery to batch rich/media/table Flex messages with text into one Reply API call up to the 5-message limit, push overflow, and update focused regression tests.

## Best Possible Solution

Land a LINE delivery fix that batches eligible rich/media/table messages into Reply API slots before Push overflow, with live LINE proof or an explicit maintainer proof override and an accepted overflow/quick-reply policy.

## Reproduction Assessment

Yes, source-level. Current main converts Markdown tables to Flex messages, sends text through Reply API first, then pushes the Flex/table messages, matching the reported 429 loss path when Push quota is exhausted.

## Solution Assessment

Partly yes. The patch is in the narrow delivery helper that owns the bug, but live LINE proof and maintainer acceptance of the rich-first overflow and quick-reply limitation are still needed before this is the best merge-ready solution.
