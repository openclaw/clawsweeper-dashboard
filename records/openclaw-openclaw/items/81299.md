## Summary

Keep this PR open. Current main and the latest release still have the reported LINE Reply-then-Push split, and the open related LINE PRs are broader, unmerged, and not safe superseding targets for this focused fix; the remaining blockers are real LINE proof, CI recheck, and maintainer acceptance of mixed-message ordering.

## What This Changes

The PR changes LINE auto-reply delivery so rich/media/text messages are packed into Reply API slots before falling back to Push overflow, with focused regression tests.

## Best Possible Solution

Keep this focused PR open for maintainer review; land it only after live LINE proof or explicit proof override and an accepted ordering policy for mixed rich/media/text batches.

## Reproduction Assessment

Yes, source-reproducible: current main extracts Markdown tables into Flex messages, sends text chunks through Reply first, and then sends Flex/media through Push in the chunks-present path. I did not run a live exhausted-quota LINE account.

## Solution Assessment

Yes for the narrow no-quick-replies bug: batching eligible rich/media/text before consuming the reply token is the right LINE-plugin-local fix. The main remaining solution question is whether maintainers accept rich/media-first ordering or want text-first batching before merge.
