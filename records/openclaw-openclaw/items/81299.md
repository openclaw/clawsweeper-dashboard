## Label Justifications

- `P1`: The PR addresses a reported LINE message-loss path where generated table Flex content can be silently dropped for users with exhausted Push quota.
- `merge-risk: 🚨 message-delivery`: The diff changes Reply-vs-Push allocation, ordering, overflow, and error fallback behavior for mixed LINE rich/media/text replies.

## Summary

Keep open. Current main still has the focused LINE message-loss bug, and this PR is the right narrow direction, but the new Reply-error fallback can still lose text when the rich/media item is what made the combined batch fail, and the contributor proof is stubbed rather than live LINE delivery proof.

## What This Changes

The PR changes LINE auto-reply delivery to batch rich/media messages with text in the first LINE Reply API call up to five messages, then Push only overflow.

## Best Possible Solution

Keep the owner-local Reply-slot batching fix, but split the Reply-error fallback so text remains deliverable when rich/media is invalid, then require live LINE proof or maintainer proof override before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main extracts Markdown tables into Flex messages, replies with text first, then routes Flex through Push; a live exhausted-quota LINE account would be needed for full end-to-end reproduction.

## Solution Assessment

No, not as currently written. Batching eligible rich/media/text before consuming the reply token is the right layer for the focused bug, but the fallback after Reply API failure should preserve text delivery instead of retrying the same mixed batch wholesale.
