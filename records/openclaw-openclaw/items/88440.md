## Summary

Keep open: this is a focused bug-fix PR with useful fake-CLI coverage, but it is a draft external PR and still lacks required real CLI/Gateway behavior proof. The retry policy also changes session binding and provider fallback behavior, so maintainers should review the upgrade risk before merge.

## What This Changes

The branch moves stale resumed CLI session recovery into the CLI runner lifecycle, adds a command-path pre-retry clearing callback, tags retryable empty timeout/exit failures, and expands CLI reliability tests.

## Best Possible Solution

Land a bounded runner-owned retry only after redacted real CLI/Gateway proof confirms stale resumed session_expired, empty timeout, and empty unknown-exit recovery without masking auth, billing, or rate-limit failures.

## Reproduction Assessment

Yes, source-reproducible: current main only retries session_expired while timeout and unknown FailoverErrors clear-and-rethrow, and the linked issue documents that gap. I did not run a live CLI repro in this read-only review.

## Solution Assessment

Mostly yes: moving retry inside the runner lifecycle is a maintainable way to preserve final hook ordering. It still needs real behavior proof and maintainer review of the bounded retry policy before merge.
