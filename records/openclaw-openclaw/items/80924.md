## Summary

Keep open: the localization direction is useful and the screenshots show real zh-CN behavior, but the PR currently expands the English locale key set without regenerating every shipped locale, which breaks the Control UI i18n contract.

## What This Changes

The PR moves Dreaming diary labels, explainer text, Memory Wiki empty-state copy, reload actions, and restart confirmation text into English and Simplified Chinese locale entries.

## Best Possible Solution

Keep the i18n extraction, but regenerate the Control UI locale bundles and `.i18n` metadata through the existing sync pipeline so every shipped locale has the new keys while preserving the intended zh-CN wording where appropriate.

## Reproduction Assessment

Yes. The PR's current failure is reproducible from source and CI: adding the new English keys without updating every shipped locale makes `ui/src/i18n/test/translate.test.ts` fail because locale key sets no longer match.

## Solution Assessment

No as submitted. Moving the hardcoded Dreaming copy into i18n is the right direction, but the maintainable fix is to use the established Control UI i18n sync workflow and commit the generated outputs instead of updating only `en` and `zh-CN`.
