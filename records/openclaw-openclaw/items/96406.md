## Summary

Keep open: this is a focused Matrix poll PR, but it is not merge-ready because MSC3381 says poll responses are truncated to max_selections before duplicate entries are collapsed, and the PR changes that contract without maintainer-approved divergence or real Matrix/OpenClaw proof.

## What This Changes

The PR moves Matrix poll response max_selections truncation to after duplicate answer-id deduplication and adds a regression test for that ordering.

## Best Possible Solution

Preserve Matrix truncation-before-deduping semantics unless maintainers explicitly choose and document a product-level divergence from MSC3381.

## Reproduction Assessment

Yes from source: current main truncates ["a1","a1","a2"] to ["a1","a1"] under max_selections=2 before Set dedupe, so only a1 is counted. The Matrix MSC text indicates that behavior is expected contract behavior, not a confirmed bug.

## Solution Assessment

No: moving slice after Set is a plausible user-friendly interpretation, but it is not the best fix while OpenClaw tracks MSC3381 semantics. A spec-locking test or maintainer-approved divergence would be safer.
