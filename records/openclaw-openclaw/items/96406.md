## Summary

Keep open, but the patch should not merge as-is: the behavior it changes is aligned with the Matrix poll processing contract, so the proposed dedupe-before-truncate order would make over-limit duplicate responses affect displayed totals.

## What This Changes

The PR changes Matrix poll result normalization to dedupe duplicate answer IDs before enforcing max_selections and adds a regression test for that ordering.

## Best Possible Solution

Preserve Matrix truncation-before-deduping semantics unless maintainers explicitly choose and document a product-level divergence from MSC3381.

## Reproduction Assessment

Yes from source: current main truncates the filtered answer array before Set deduping, so ["a1","a1","a2"] under max_selections=2 produces only a1. The Matrix MSC text indicates that behavior is the expected processing contract rather than a confirmed bug.

## Solution Assessment

No: moving slice after Set is a plausible user-friendly interpretation, but it is not the best fix while OpenClaw tracks MSC3381 semantics. A spec-locking test or maintainer-approved divergence would be safer.
