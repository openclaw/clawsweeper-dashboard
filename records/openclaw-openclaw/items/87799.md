## Summary

Keep this PR open: it targets a real documented installer failure, but the current patch still has a fallback stdin compatibility defect and does not yet include sufficient real behavior proof for the changed install paths.

## What This Changes

The PR redirects installer subprocess stdin away from the piped script stream in gum, fallback, npm, dashboard, plugin-update, daemon-restart, and onboarding paths, with matching installer test assertions.

## Best Possible Solution

Keep the targeted stdin hardening, but base fallback isolation on `NO_PROMPT` or non-TTY stdin and add redacted full installer proof for piped gum mode plus a direct logged or interactive fallback path.

## Reproduction Assessment

Yes, source-reproducible: the documented curl-to-bash path feeds `install.sh` through stdin, current main routes subprocesses through gum/fallback wrappers, and gum v0.17.0 passes `os.Stdin` to the wrapped command. I did not run a live full installer reproduction in this read-only review.

## Solution Assessment

No. The fix direction is right, but the best version needs a stdin-focused fallback predicate instead of the stdout-sensitive helper, plus full redacted real install-path proof.
