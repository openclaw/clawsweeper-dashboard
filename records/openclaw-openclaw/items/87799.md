## Summary

Keep open: the PR targets a real source-reproducible installer stdin bug, but the current diff still uses a stdout-sensitive helper for fallback stdin isolation and the supplied proof is simulation output rather than a full PR-script install path.

## What This Changes

The PR adds stdin redirects around installer subprocesses and updates installer assertions so piped curl-to-bash installs cannot have child processes consume the script stream.

## Best Possible Solution

Keep the targeted stdin hardening, but use a pipe-safety predicate based on `NO_PROMPT` or non-TTY stdin, then verify a redacted full piped install with gum plus a direct logged/interactive fallback path.

## Reproduction Assessment

Yes, source-reproducible: the documented curl-to-bash path feeds `install.sh` through stdin, current main routes subprocesses through gum/fallback wrappers, and gum v0.17.0 passes `os.Stdin` to the wrapped command. I did not run a live full installer reproduction in this read-only review.

## Solution Assessment

No. The fix direction is right, but the best version needs a stdin-focused predicate instead of the existing stdout-sensitive helper, plus full redacted real install-path proof.
