## Summary

Keep this PR open: current main still leaves the installer’s gum-wrapped and fallback child-command paths able to inherit piped script stdin, while this branch is the active focused fix for the linked installer stdin-corruption bug. I found no blocking correctness defect, but merge still needs a maintainer call on the compatibility-sensitive installer stdin contract and whether the supplied terminal proof is enough.

## What This Changes

The PR adds installer stdin-isolation logic for piped and NO_PROMPT runs, preserves stdin for direct interactive runs, redirects several noninteractive installer subprocesses from `/dev/null`, and adds focused install-sh tests.

## Best Possible Solution

Land the focused stdin-consumer hardening after maintainer acceptance of the installer compatibility tradeoff, then close https://github.com/openclaw/openclaw/issues/90008 as implemented.

## Reproduction Assessment

Yes. Source inspection shows current main still runs gum/fallback child commands with inherited stdin, the public installer command can be piped through stdin, and the gum v0.17.0 source forwards `os.Stdin` to wrapped commands.

## Solution Assessment

Yes, this is the best observed fix direction. It hardens the actual stdin consumers instead of reviving the broader pipe-buffer guard that was closed as too broad; the remaining question is compatibility and proof acceptance.
