## Summary

Keep open: the reported CLI/Gateway status mismatch is real, but this patch still reads remote eligibility inside the standalone CLI process, where the Gateway-populated remote node cache is empty, so the central bug can remain unfixed.

## What This Changes

The PR adds remote eligibility inputs to the CLI skills status report builder and adds CLI command test mocks for the new lazy imports.

## Best Possible Solution

Make `openclaw skills list`, `check`, and `info` prefer Gateway `skills.status` when reachable, keep the local builder as the offline fallback, and add tests for remote-aware Gateway results across those commands.

## Reproduction Assessment

Yes from source inspection: the standalone CLI path can still see an empty process-local remote node cache while the Gateway owns the connected macOS node records. I did not run a live paired macOS node setup in this read-only review.

## Solution Assessment

No: mirroring `getRemoteSkillEligibility()` in the CLI is only a plausible local refactor, not the best fix for a Gateway-owned live state boundary. The maintainable path is to read `skills.status` from the Gateway when available and preserve local/offline fallback behavior.
