## Summary

Keep open: the patch wires remote eligibility into the standalone CLI process, but the connected remote-node state is populated in the Gateway process, so the reported Linux Gateway plus macOS node flow can still return no remote eligibility. The PR also still lacks real after-fix proof from a live connected macOS node setup.

## What This Changes

The branch adds lazy imports for remote skill eligibility and exec-node policy to the CLI skills status loader, plus test mocks for those imports.

## Best Possible Solution

Have skills list, info, and check prefer Gateway skills.status when reachable, preserve local/offline fallback, and validate with a Linux Gateway plus connected macOS node.

## Reproduction Assessment

Yes. From source inspection, the standalone CLI status commands build local status directly while live remote-node state is populated in Gateway runtime code; I did not run a live connected macOS node in this read-only review.

## Solution Assessment

No. Passing getRemoteSkillEligibility inside the CLI is only a partial refactor; the maintainable fix is to consume Gateway skills.status when reachable because that process owns connected remote-node state.
