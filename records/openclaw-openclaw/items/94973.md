## Summary

Keep open: this is a focused bug-fix PR, but the patch is not merge-ready because it still asks a one-shot CLI process to read Gateway-owned remote-node state, so the linked CLI/Gateway skills status bug can remain unfixed.

## What This Changes

The branch adds lazy imports for remote skill eligibility and exec-node policy to the CLI skills status loader, plus test mocks for those imports.

## Best Possible Solution

Have `openclaw skills list`, `info`, and `check` prefer Gateway `skills.status` when reachable, preserve local/offline fallback, and validate with a Linux Gateway plus connected macOS node.

## Reproduction Assessment

Yes. From source inspection, the standalone CLI status commands build local status directly while live remote-node state is populated in Gateway runtime code; I did not run a live connected macOS node in this read-only review.

## Solution Assessment

No. Passing `getRemoteSkillEligibility()` inside the CLI is only a plausible partial refactor; the best fix is to consume Gateway `skills.status` when reachable because that process owns connected remote-node state.
