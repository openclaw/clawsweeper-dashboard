## Summary

Keep open: this targets the right regression, but the patch is likely ineffective because it reads process-local remote-node state from the one-shot CLI instead of querying the Gateway status source that owns connected macOS nodes.

## What This Changes

The PR adds remote eligibility to the CLI skills status report builder and mocks the new lazy imports in the CLI command tests.

## Best Possible Solution

Make the skills CLI prefer Gateway `skills.status` when reachable, keep the local status builder as an offline fallback, and coordinate with https://github.com/openclaw/openclaw/pull/94972 if its `system.which`-only gate is also confirmed.

## Reproduction Assessment

Yes from source inspection: current CLI status is local-only, while the remote node cache is populated in Gateway handlers. I did not run a live paired macOS node, but the process boundary makes the current PR path source-reproducible as insufficient.

## Solution Assessment

No: adding `getRemoteSkillEligibility()` to the standalone CLI is only a plausible fix because the CLI process does not own the Gateway connected-node cache. The maintainable fix is to query Gateway `skills.status` when reachable and keep local status as fallback.
