## Summary

Keep open: current main still writes adjacent numbered config backups, the direct directory-and-datetime implementation was closed unmerged, and the active related pull requests only address timestamp suffixes beside the config file. This remains a valid feature/design item because moving credential-bearing backups changes recovery, retention, permissions, legacy cleanup, audit output, and docs.

## What This Changes

Store config backups in a dedicated backup directory using datetime-based names instead of adjacent numbered `.bak` files.

## Best Possible Solution

Adopt one unified backup-layout design that moves managed config backups into a dedicated directory with collision-safe datetime names, preserves or migrates legacy adjacent backups safely, and updates recovery, audit, docs, changelog, and focused tests together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high confidence that config writes still create adjacent `.bak` and `.bak.N` files through `maintainConfigBackups`.

## Solution Assessment

No current implementation solves the whole issue. The timestamp-only pull requests are plausible partial work, but the maintainable solution needs an explicit directory, migration, recovery, retention, and permission contract before merge.
