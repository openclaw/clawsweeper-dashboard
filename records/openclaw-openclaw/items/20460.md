## Summary

Keep open: current main still writes adjacent `openclaw.json.bak` plus numbered `.bak.N` backups, while the requested dedicated `config-backup/` directory with datetime names has not landed. The direct implementation at https://github.com/openclaw/openclaw/pull/20877 was closed unmerged, and the active related PRs only cover timestamp suffixes beside the config file.

## What This Changes

The issue requests moving OpenClaw config backups into a dedicated backup directory and replacing numbered backup names with datetime-based names.

## Best Possible Solution

Adopt one unified backup-layout design that moves managed config backups into a dedicated directory, uses collision-safe datetime names, preserves or safely migrates legacy adjacent backups, keeps bounded retention and `0o600` permissions, and updates recovery, audit, docs, changelog, and focused tests together.

## Reproduction Assessment

Yes for the current behavior: source inspection shows config writes call `maintainConfigBackups`, which writes `${configPath}.bak` and rotates numbered neighbors. This is a feature request rather than a failing bug reproduction, and no live write was run because the review is read-only.

## Solution Assessment

No current implementation solves the whole issue. The open timestamp PRs are plausible partial work, but the best maintainable path needs a maintainer-chosen directory, migration, recovery, and security contract before merge.

- Config backups can contain credentials, so a new directory scheme must preserve owner-only permissions, bounded retention, and safe legacy/orphan cleanup.
- Open timestamp-only PRs overlap part of the naming request but not the dedicated-directory request, so maintainers should choose a single backup-format direction before merging partial work.
