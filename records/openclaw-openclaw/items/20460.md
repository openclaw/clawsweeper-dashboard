## Summary

Keep open: current main and v2026.5.19 still store config backups beside the active config as `.bak` plus numbered `.bak.N` files. The direct directory-and-datetime PR closed unmerged, while the open related PRs only cover timestamp suffixes and do not settle the requested dedicated backup directory layout.

## What This Changes

Requests moving config backup files from adjacent `.bak` and `.bak.N` files into a dedicated `config-backup/` directory with datetime-stamped names.

## Best Possible Solution

Adopt one coherent config-backup layout with a dedicated directory, collision-safe datetime names, bounded retention, owner-only permissions, safe legacy handling, and matching recovery, audit, docs, tests, and changelog coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection of current main and v2026.5.19 confirms config writes still create adjacent `.bak` and numbered `.bak.N` files through `maintainConfigBackups`.

## Solution Assessment

Unclear until maintainers choose the backup-layout contract. A dedicated directory may be the right end state, but the maintainable fix must update creation, retention, permissions, legacy handling, recovery, audit output, docs, and tests together.
