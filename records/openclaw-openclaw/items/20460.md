## Summary

Keep open: current main still uses adjacent numbered config backups, and the requested dedicated `config-backup/` directory with datetime names has not landed. Related open PRs only target timestamp suffixes beside the config file, while the direct directory PR was closed unmerged.

## What This Changes

Request to store OpenClaw config backups in a dedicated directory with datetime-based filenames instead of adjacent numbered `.bak` files.

## Best Possible Solution

Adopt one unified backup-layout design that moves managed config backups into a dedicated directory, uses collision-safe datetime names, safely handles legacy adjacent backups, preserves `0o600` permissions and retention, and updates recovery, audit, docs, changelog, and focused tests together.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms config writes still create adjacent `${configPath}.bak` plus numbered `.bak.N` files.

## Solution Assessment

No current implementation solves the full issue. The maintainable path is a maintainer-chosen layout and migration contract rather than landing a timestamp-only partial change.

## Risks / Open Questions

- Config backups can contain credentials, so a new directory scheme must preserve owner-only permissions, bounded retention, path safety, and orphan cleanup.
- Recovery, audit, and doctor paths currently assume an adjacent `${configPath}.bak`; moving backups without a unified contract could regress restore and diagnostics.
- Open timestamp-only PRs overlap part of the request but not the dedicated-directory behavior, so maintainers should choose one backup-format direction before implementation.
