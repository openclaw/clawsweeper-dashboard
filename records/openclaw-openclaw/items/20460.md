## Summary

Keep open: current main and v2026.6.8 still store config backups beside the active config as `.bak` plus numbered `.bak.N` files. The direct directory-layout PR is closed unmerged, while the open timestamp-only issue and PR are partial overlap because they do not move backups into a dedicated `config-backup/` directory.

## What This Changes

The issue asks OpenClaw to move config backup files into a dedicated `config-backup/` directory and replace numbered backup suffixes with datetime-based names.

## Best Possible Solution

Define one canonical config-backup layout, then implement directory placement and collision-safe datetime naming with bounded retention, owner-only permissions, legacy adjacent-backup handling, recovery/audit/doctor updates, and focused tests.

## Reproduction Assessment

Yes for the current behavior claim: source and focused tests on current main show adjacent `.bak` plus `.bak.1` through `.bak.4` backups. This is a feature request rather than a broken existing behavior report.

## Solution Assessment

Unclear until maintainers choose the canonical backup layout and legacy handling. A timestamp-only suffix change is not the full fix because this issue also asks to move backups into their own directory and current recovery/audit paths assume an adjacent primary backup.
