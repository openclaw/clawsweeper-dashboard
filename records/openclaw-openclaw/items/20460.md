## Summary

Keep open. Current main still writes config backups beside the active config as a numbered `.bak` ring, and the write, recovery, audit, docs, and tests still depend on that adjacent layout. The direct implementation PR #20877 was closed unmerged, while open related PR #40392 only covers datetime suffixes for adjacent backups, so the dedicated backup directory request remains unresolved.

## What This Changes

Request to store OpenClaw configuration backup files in a dedicated `config-backup/` directory and use datetime-based backup filenames instead of numbered suffixes.

## Best Possible Solution

Implement a single config-backup path helper used by write, rotation, recovery, audit, and docs; store managed backups under a dedicated `config-backup/` directory with sortable UTC datetime names, collision handling, bounded retention, owner-only permissions, and a safe legacy adjacent-backup migration or fallback path.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence reproduction path: any existing config write calls `maintainConfigBackups`, which creates `${configPath}.bak` and rotates numbered adjacent backups that the test suite explicitly expects.

## Solution Assessment

No. Current main does not solve the requested behavior, and the open datetime-only PR is a partial solution unless it also moves backups into the dedicated directory and updates recovery, audit, docs, and legacy handling.
