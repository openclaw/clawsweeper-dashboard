## Summary

Keep open: current `main` and the latest release still store config backups beside the active config as `.bak` plus numbered `.bak.N` files. The direct directory-layout PR was closed unmerged, and the open timestamp PR is only partial overlap because it keeps backups adjacent to the config file.

## What This Changes

Request to store configuration backups in a dedicated `config-backup/` directory with datetime-based backup names.

## Best Possible Solution

Define one canonical config-backup layout, then implement directory placement and collision-safe datetime naming together with bounded retention, owner-only permissions, legacy-backup handling, recovery selection, audit output, doctor docs, and focused upgrade tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms current `main` and `v2026.6.6` still create adjacent `.bak` and `.bak.N` backups rather than a `config-backup/` directory.

## Solution Assessment

Unclear until maintainers choose the canonical backup layout and legacy handling. A timestamp-only suffix change is not the full fix because this issue also asks to move backups into their own directory and current recovery/audit paths assume an adjacent primary backup.
