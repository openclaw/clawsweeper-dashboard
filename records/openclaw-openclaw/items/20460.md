## Summary

Current main and v2026.6.5 still use the adjacent five-file numbered backup ring, while recovery, audit output, permissions, and orphan cleanup all depend on that layout. The direct directory implementation closed unmerged, and the open timestamp proposal only renames adjacent files, so the requested directory move remains necessary and needs a maintainer-owned upgrade contract.

## What This Changes

Requests storing configuration backups in a dedicated directory with datetime-based filenames instead of adjacent numbered `.bak` files.

## Best Possible Solution

Implement one core-owned backup directory with collision-safe sortable timestamps, bounded retention, owner-only permissions, and atomic updates to backup creation, recovery, audit output, and docs; use `openclaw doctor --fix` as the sole owner for migrating or safely retiring legacy adjacent backups.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source and tests nevertheless establish the current adjacent numbered-ring behavior with high confidence.

## Solution Assessment

No, moving or renaming files in isolation is not the best solution. The narrowest maintainable fix is a single coordinated backup-storage contract covering creation, recovery, cleanup, security, legacy migration, and operator-facing documentation.

## Risks / Open Questions

- Moving backups without updating suspicious-config recovery could leave existing or newly created recovery points undiscoverable.
- Legacy adjacent backups could be silently deleted, stranded, or retained indefinitely unless doctor owns an explicit migration or retirement policy.
- Configuration backups may contain API keys and tokens, so directory creation, every backup file, and cleanup must preserve owner-only permissions and bounded retention.
- Timestamp naming needs deterministic ordering and collision handling for rapid consecutive writes; the open timestamp-only PR does not resolve the directory-wide upgrade contract.
