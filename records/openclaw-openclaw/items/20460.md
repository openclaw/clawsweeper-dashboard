## Summary

Keep open: current main and the latest release still create adjacent `.bak` plus numbered `.bak.N` config backups, while this request asks for a dedicated backup directory with datetime filenames. The direct directory implementation was closed unmerged, and the active related work only covers timestamp suffixes beside the config file.

## What This Changes

Move config backups from adjacent `.bak`/`.bak.N` files into a dedicated `config-backup/` directory with datetime-based filenames.

## Best Possible Solution

Adopt one coherent config-backup layout with a dedicated directory, collision-safe datetime names, bounded retention, owner-only permissions, safe legacy handling, and matching recovery, audit, doctor docs, tests, and changelog coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection of current main and `v2026.5.12` gives high confidence that config writes still create adjacent `.bak` and `.bak.N` files through `maintainConfigBackups`.

## Solution Assessment

No current implementation solves the full issue. The maintainable path is a maintainer-chosen backup-layout contract that updates creation, retention, permissions, legacy migration, recovery, audit, docs, and focused tests together.
