## Summary

Keep open: current main and the latest release still keep config backups beside the active config as `.bak` plus numbered `.bak.N` files. The open related timestamp PR is only a partial overlap because it keeps adjacent backup files and does not implement the requested dedicated backup directory.

## What This Changes

Request to move config backup files into a dedicated `config-backup/` directory and use datetime-based backup names.

## Best Possible Solution

Define one canonical config-backup layout, then implement directory placement and collision-safe datetime naming together with bounded retention, owner-only permissions, legacy-backup handling, recovery selection, audit output, doctor docs, and focused upgrade tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence current-main check: config writes still create adjacent `.bak` and `.bak.N` backups rather than a `config-backup/` directory.

## Solution Assessment

Unclear until maintainers choose the canonical backup layout and legacy handling. A timestamp-only suffix change is not the full fix because this issue also asks to move backups into their own directory and current recovery/audit paths assume an adjacent primary backup.
