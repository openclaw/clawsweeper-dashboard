## Summary

Current main still uses adjacent numbered config backups, and the open linked work changes only suffixes, so the dedicated-directory request remains necessary. The complete change is compatibility-sensitive because backup retention, recovery lookup, audit output, permissions, orphan cleanup, and legacy recovery points share this path; maintainers should decide whether to broaden the active datetime work or retain this as the canonical layout request.

## What This Changes

Move configuration backups from adjacent `.bak`/`.bak.N` files into a dedicated backup directory and use datetime-based filenames.

## Best Possible Solution

Define one canonical core backup layout, then implement directory placement and collision-safe datetime naming together with bounded retention, owner-only permissions, explicit legacy-backup migration or preservation, updated recovery selection and audit output, doctor guidance, and focused fresh-install plus upgrade tests.

## Reproduction Assessment

Yes for the reported current behavior: current source and focused tests deterministically create adjacent `.bak` plus numbered backup files. The requested layout is an enhancement to established behavior, not a failure of an existing documented contract.

## Solution Assessment

No, the active datetime-only approach is not the best complete solution for this issue because it leaves backups in the root config directory and does not coordinate every recovery and upgrade consumer. A unified layout, migration, retention, permission, and recovery design is safer than parallel partial solutions.
