## Summary

Keep open: current main still writes adjacent `.bak` plus numbered `.bak.N` config backups, while the requested dedicated `config-backup/` directory with datetime naming has not landed. The direct implementation PR was closed unmerged, and the open related PRs only cover timestamp suffixes beside the config file.

## What This Changes

Store config backup files under a dedicated `config-backup/` directory and name them with datetimes instead of adjacent numeric `.bak` rotation files.

## Best Possible Solution

Adopt one unified backup-layout design that moves managed config backups into a dedicated directory, uses deterministic collision-safe datetime names, preserves or safely migrates legacy adjacent backups, keeps bounded retention and `0o600` permissions, and updates recovery, audit, docs, changelog, and focused tests together.

## Reproduction Assessment

Yes for the current behavior: source inspection shows existing config writes call `maintainConfigBackups`, which writes `${configPath}.bak` and rotates numbered neighbors. This is a feature request rather than a failing bug reproduction, and no live write was run because the review is read-only.

## Solution Assessment

No current implementation solves the whole issue. The open timestamp PRs are plausible partial work, but the best maintainable path needs a maintainer-chosen directory, migration, recovery, and security contract before merge.
