## Summary

Keep open. The PR fixes the initial direct stdout QR path, but it is incomplete because the existing 515 restart path only preserves QR handling when `loginWeb` passes `onQr` into `waitForWhatsAppLoginResult`; this patch does not, so replacement sockets can still lose runtime-routed QR output.

## What This Changes

The PR changes WhatsApp `loginWeb` to disable session-level QR printing, render terminal QR art locally, log the heading and art through the injected runtime, and add an Unreleased changelog entry.

## Best Possible Solution

Hoist one QR-rendering handler in `loginWeb`, use it for the initial socket and pass it into `waitForWhatsAppLoginResult`, then cover both the initial QR and restart-created socket cases with focused WhatsApp login tests.

## Reproduction Assessment

Yes, by source inspection. Current main routes `loginWeb` QR output through `console.log` and `process.stdout.write`, and the PR diff leaves the restart-created socket without the new runtime QR handler.

## Solution Assessment

No. The direction is right, but the fix needs to pass the same `onQr` handler through the login-result restart path so socket replacement cannot reintroduce missing QR output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass the QR handler through restart sockets:** `extensions/whatsapp/src/login.ts:26`
  - body: The new `onQr` callback is only installed on the first `createWaSocket` call. If `waitForWhatsAppLoginResult` hits the 515 restart path, it recreates the socket with `onQr: params.onQr`, but `loginWeb` still does not provide that parameter, so any QR emitted by the replacement socket will not be logged through the injected runtime. Hoist this handler and pass it to `waitForWhatsAppLoginResult` as well.
