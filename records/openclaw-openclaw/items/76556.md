## Summary

Keep open. The PR moves the initial WhatsApp QR output into the injected runtime, but source inspection shows the existing 515 restart path still creates the replacement socket from `waitForWhatsAppLoginResult` using `params.onQr`, and `loginWeb` does not pass the new callback to that waiter, so restart-generated QR codes would still be dropped.

## What This Changes

The PR changes WhatsApp `loginWeb` to disable direct terminal QR printing, render QR ASCII with `renderQrTerminal`, send the heading/art through `runtime.log`, and records the fix in the changelog.

## Best Possible Solution

Keep the PR open until it forwards the runtime-routed QR callback through the 515 restart path and adds focused regression coverage for that path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make `waitForWaConnection` fail once with status 515, let `waitForWhatsAppLoginResult` create the replacement socket, and observe that `params.onQr` is undefined because `loginWeb` did not pass the new QR callback into the waiter.

## Solution Assessment

No. The direction is right, but the narrow maintainable fix is to define one QR runtime-output callback and pass it to both the initial socket and `waitForWhatsAppLoginResult` so the restart socket gets the same behavior.

## Review Findings
