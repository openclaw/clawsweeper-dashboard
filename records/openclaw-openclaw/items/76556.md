## Summary

Keep open for normal PR handling. Current main still routes WhatsApp login QR output through the direct terminal printer, while the updated PR now carries the runtime-routed QR callback through both the initial socket and the 515 restart path with no remaining blocking finding from source review.

## What This Changes

The PR changes WhatsApp `loginWeb` to render QR terminal art through `renderQrTerminal`, log the QR heading/art through the injected runtime for initial and restart sockets, and add a changelog fix entry.

## Best Possible Solution

Land this PR after standard validation so explicit WhatsApp login QR output consistently goes through the injected runtime, including the restart socket path.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: when `loginWeb` receives a QR event, it uses `createWaSocket(true, ...)`, and `session.ts` then prints the heading/art via `console.log` and `process.stdout` instead of the injected runtime.

## Solution Assessment

Yes. The updated PR is the narrow maintainable fix: disable the direct printer, define one runtime-routed `onQr` handler, and pass it to both the initial socket and `waitForWhatsAppLoginResult` so 515 restarts keep the same behavior.
