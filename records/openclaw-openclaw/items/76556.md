## Summary

Keep open: current main still sends WhatsApp login QR output through direct terminal printing, while this PR now routes initial and 515-restart QR callbacks through the injected runtime with focused coverage and no remaining blocking review finding.

## What This Changes

The PR changes WhatsApp `loginWeb` to disable direct QR printing, render terminal QR art via `renderQrTerminal`, pass one runtime-routed `onQr` callback to initial and restart sockets, add restart coverage, and add a changelog entry.

## Best Possible Solution

Land this PR after normal CI and mergeability checks; it is the narrow WhatsApp-owned fix and should close #76213 once merged.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main calls `createWaSocket(true, ...)` from `loginWeb`, and `session.ts` prints QR output via `console.log` and `process.stdout.write` instead of the injected runtime. The 515 restart path is also source-reproducible because replacement sockets only receive `params.onQr`.

## Solution Assessment

Yes. Hoisting one runtime-routed `onQr` callback and passing it to both the initial socket and `waitForWhatsAppLoginResult` uses the existing WhatsApp socket seam and avoids core API or config changes.
