## Summary

Keep open: this maintainer-labeled member PR changes exec authorization and the current head still has a concrete approval race. The planner direction looks useful, but delaying registration until after command analysis makes caller-supplied approval IDs unresolvable during that window and the live head check is already failing the node.invoke approval-bypass coverage for that path.

## What This Changes

The PR adds a planner-backed command authorization path and routes POSIX allowlist, allow-always persistence, approval summaries, control-command rejection, and shell rendering through it.

## Best Possible Solution

Keep the planner-backed authorization direction, but preserve the current immediate explicit-ID registration/resolve contract while attaching command analysis before request delivery, then rerun the failing gateway and node-host approval checks.

## Reproduction Assessment

Yes for the PR regression: source review shows explicit IDs are not registered until after command analysis, and the live head check fails the existing node.invoke approval-bypass test at the approved invoke assertion.

## Solution Assessment

No, not as currently implemented. Centralizing through the planner is maintainable, but the approval request flow should not trade away the existing immediate explicit-ID resolution behavior.
