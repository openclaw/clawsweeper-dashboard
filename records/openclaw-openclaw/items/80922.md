## Summary

Keep open: this is a protected maintainer PR touching command authorization, and the diff has a blocking argv normalization bug that should be fixed before merge.

## What This Changes

This PR adds an internal `src/infra/command-authorization` planner contract, implementation, exports, and corpus tests for argv, POSIX shell, unsupported shell dialects, and prompt-only outcomes.

## Best Possible Solution

Preserve literal argv tokens in the planner, add a corpus case for whitespace-bearing argv arguments, then route the security-sensitive contract through maintainer review before merge.

## Reproduction Assessment

Yes for the review finding. Source inspection shows `planArgvCommand` trims argv entries before analysis while current `analyzeArgvCommand` preserves nonempty argv strings, so an argv argument with leading or trailing spaces would be planned as different input.

## Solution Assessment

No. The planner should filter empty argv entries without rewriting nonempty tokens, and the corpus should lock that behavior before this authorization contract is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve argv tokens before planning:** `src/infra/command-authorization/plan.ts:53`
  - body: This planner trims every argv element before analysis. For tokenized argv input, leading and trailing spaces are literal argument bytes; `['printf', ' x ']` would be authorized and reported as `['printf', 'x']`, so the approval plan no longer describes the command being executed. Filter empty entries without rewriting non-empty tokens, and add a corpus case for this boundary.
  - confidence: 0.9
