## Summary

Keep this PR open: the branch is active, maintainer-labeled, and still has a concrete correctness issue in the new authorization planner where argv inputs are rewritten before planning.

## What This Changes

The PR adds an internal command-authorization planner and routes POSIX allow-always persistence through planner units, with planner corpus and approval-store coverage.

## Best Possible Solution

Keep the planner direction, preserve literal argv tokens in the new planner, add a corpus regression for whitespace-bearing argv arguments, then send the exact head back through maintainer review.

## Reproduction Assessment

Yes for the review finding from source inspection. PR-head `planArgvCommand` trims argv entries before planning, while current main preserves nonempty argv token strings in `analyzeArgvCommand`.

## Solution Assessment

No. The planner should filter empty argv entries without rewriting nonempty tokens, and the corpus should lock that boundary before this authorization contract is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve argv tokens before planning:** `src/infra/command-authorization/plan.ts:49`
  - body: This planner trims every argv element before analysis. For tokenized argv input, leading and trailing spaces are literal argument bytes; `['printf', ' x ']` would be authorized and reported as `['printf', 'x']`, so the approval plan no longer describes the command being executed. Filter empty entries without rewriting nonempty tokens, and add a corpus case for this boundary.
  - confidence: 0.91
