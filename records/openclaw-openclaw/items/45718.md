## Summary

Keep open: current main and the latest release externalize large `skillsSnapshot.prompt` payloads, but they still persist full `systemPromptReport` objects in durable session entries; fresh v2026.6.10 field data confirms that remaining payload is now the dominant bloat source.

## What This Changes

The issue asks OpenClaw to stop durable session entries from accumulating duplicate `skillsSnapshot` and `systemPromptReport` payloads across repeated runs.

## Best Possible Solution

Keep this issue as the canonical tracker and design a session metadata policy that preserves only minimal durable warning-dedupe state while compacting or omitting bulky prompt-report details from retained session rows.

## Reproduction Assessment

Yes, at source level: current main still persists systemPromptReport into durable session entries after runs, and a fresh v2026.6.10 report shows that field dominating sessions.json size. I did not run a live long-running gateway workload in this read-only review.

## Solution Assessment

No, the issue is not solved yet: current main solves the largest duplicated skillsSnapshot.prompt payload but preserves full systemPromptReport data. The best fix should separate minimal warning-dedupe state from bulky diagnostic report details instead of either keeping or dropping the whole report blindly.
