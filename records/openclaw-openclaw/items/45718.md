## Summary

Keep open. Current main still persists the reported bulky session-entry fields through the normal session-store write path, and open PR #48425 already uses closing syntax for this issue, so the issue should not be closed until that PR or an equivalent scoped replacement lands.

## What This Changes

The issue asks to prevent `sessions.json` growth by avoiding redundant persistence and rebuilds of `skillsSnapshot` and `systemPromptReport` across agent runs.

## Best Possible Solution

Land #48425 or a scoped replacement that strips transient `skillsSnapshot` from disk serialization, keeps disk-budget calculations aligned with the exact persisted shape, and explicitly documents or tests the intended persisted subset of `systemPromptReport` needed for warning dedupe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: save a session entry containing `skillsSnapshot` or `systemPromptReport` through `saveSessionStore`; current main serializes the full entry into `sessions.json`.

## Solution Assessment

Unclear. A shared session-store serialization helper is the narrowest maintainable boundary for `skillsSnapshot`, but the `systemPromptReport` part should preserve the warning-dedupe contract or replace it with a smaller persisted shape.
