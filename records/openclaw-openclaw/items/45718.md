## Summary

Keep open. Current main only partially mitigates the bloat by stripping `skillsSnapshot.resolvedSkills`; it still persists the remaining `skillsSnapshot` shell and `systemPromptReport`, the stale-session path still forces snapshot rebuild behavior, and open PR #48425 uses closing syntax for this issue.

## What This Changes

The issue requests preventing per-run session-store bloat by avoiding repeated or bulky `skillsSnapshot` and `systemPromptReport` data in `sessions.json`.

## Best Possible Solution

Land #48425 or a scoped replacement that defines the durable session-entry shape, keeps disk-budget measurement aligned with serialization, and preserves only the minimal prompt-report state needed for warning dedupe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction remains: save or update a session entry containing `skillsSnapshot` or `systemPromptReport` through the current session-store path and current main will persist those fields, while stripping only `resolvedSkills`.

## Solution Assessment

Unclear. The current main mitigation is useful for the largest skill-cache payload, but the best complete fix should be the shared serialization boundary proposed by #48425 or a replacement that also settles the `systemPromptReport` persistence contract.
