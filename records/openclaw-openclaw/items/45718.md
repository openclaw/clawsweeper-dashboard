## Summary

Keep open: current main and v2026.6.6 ship the large `skillsSnapshot.prompt` blob/ref mitigation, but current source still persists `systemPromptReport` from run metadata and still treats stale unrequested sessions as new for skill snapshot reuse. The open closing PR is still conflicting and does not settle the remaining report-retention contract.

## What This Changes

The issue requests stopping repeated persisted `skillsSnapshot` and `systemPromptReport` session metadata bloat so session stores do not grow into context overflow failures.

## Best Possible Solution

Keep the shipped promptRef/blob design, define the minimal durable `systemPromptReport` state needed for warning dedupe, and update or close the linked PR around that narrower session-store contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level for the remaining behavior: current main still rebuilds reports with a fresh timestamp, persists them into session entries, and bypasses existing skill snapshots for stale unrequested sessions; I did not run a live long-uptime size-growth repro.

## Solution Assessment

Is this the best way to solve the issue? No for closing or landing the linked whole-field stripping path as-is; the best fix is a narrower durable report-state contract layered on top of the already-shipped prompt blob storage.
