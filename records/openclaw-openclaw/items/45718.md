## Summary

Keep open. Current main shipped the important `resolvedSkills` bloat mitigation, but the specific reported persistence paths still retain `skillsSnapshot` metadata and `systemPromptReport`, and the target-repo implementation PR for stripping the whole field was closed unmerged as too broad.

## What This Changes

The issue reports unbounded `sessions.json` growth from repeated persisted `skillsSnapshot` and `systemPromptReport` session fields.

## Best Possible Solution

Define a scoped session-persistence policy that keeps required warning-dedupe and snapshot-resume metadata while avoiding large recomputable prompt or skill payloads in `sessions.json`.

## Reproduction Assessment

Yes. Source inspection on current main shows the save, run-completion, and stale-session snapshot paths still retain the reported fields, while tests prove only `resolvedSkills` is stripped; I did not run a live gateway workload.

## Solution Assessment

Unclear. Removing whole fields at serialization is too broad, but a narrower persistence/serialization contract that trims recomputable payloads while preserving warning dedupe and snapshot metadata is the maintainable direction.
