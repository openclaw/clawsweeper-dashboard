## Summary

Keep open: this is a plausible Telegram reliability fix with now-supplied live-output proof, but the patch is not merge-ready because the polling-start status change can hide a restarted account that never reaches a successful getUpdates cycle.

## What This Changes

The PR enables TCP keepalive on Telegram undici connect options, stops notePollingStart from publishing connected:false, updates Telegram polling tests, and widens shared channel connect grace from 120s to 300s.

## Best Possible Solution

Keep the dependency-supported keepalive change only with a lifecycle status design that preserves startup failure detection, and scope or prove any longer connect-grace policy before changing the shared default.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows a patch-merged connected:true state can survive while transport timestamps are cleared, then evaluateChannelHealth can fall through as healthy after grace. The original NAT-stall failure was not independently reproduced with an induced network scenario.

## Solution Assessment

No as written. The undici keepalive option is supported, but the polling-start status and shared grace changes need to preserve health-monitor observability and avoid broad default drift.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep startup failures observable after polling restarts:** `extensions/telegram/src/polling-status.ts:18-22`
  - body: After a previously connected account restarts, setRuntime preserves omitted fields. This patch clears the transport timestamps without clearing connected, so once startup grace expires a startup that hangs before the next successful getUpdates can remain healthy instead of being restarted. Keep an explicit connecting/disconnected signal or add a stale-startup path before dropping connected:false here.
  - confidence: 0.91
- **[P2] Avoid widening every channel's startup grace:** `src/gateway/channel-health-policy.ts:52`
  - body: Changing the shared connect grace from 120s to 300s delays disconnected recovery for every channel, not only Telegram. Scope the longer grace to the implicated Telegram startup path or add focused cross-channel upgrade proof that the longer default will not mask real disconnects.
  - confidence: 0.82
