## Summary

Keep open: the linked prior Telegram recovery PRs support the problem boundary, and the undici keepalive option is dependency-supported, but this PR is not merge-ready because it lacks real after-fix Telegram/prod proof and still introduces a health-state regression.

## What This Changes

The PR enables TCP keepalive in Telegram undici connect options, stops polling-start from setting `connected:false`, updates polling status/session tests, and raises shared channel connect grace to 300s.

## Best Possible Solution

Keep the keepalive direction only with a lifecycle status design that preserves startup failure detection, plus redacted live Telegram or production-log proof for the stall scenario.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: patch-merged `connected:true` can survive while transport activity is cleared, and health can then fall through as healthy after grace. The original NAT-stall production failure was not independently reproduced here.

## Solution Assessment

No as written. The undici keepalive option is contract-supported, but the polling-start status change needs an explicit connecting/stale-startup liveness path before dropping `connected:false`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep startup health observable after reconnects:** `extensions/telegram/src/polling-status.ts:18-23`
  - body: `setRuntime` patch-merges account status, so after a previously connected Telegram account restarts this patch can preserve `connected:true` while clearing `lastTransportActivityAt`. Once startup grace expires, `evaluateChannelHealth` only checks stale sockets when that timestamp is non-null, so a startup that hangs before the next successful `getUpdates` can remain healthy instead of being restarted. Preserve a lifecycle liveness signal or add an explicit connecting/stale-startup path before dropping `connected:false` here.
