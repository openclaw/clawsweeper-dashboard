## Summary

Keep open: the keepalive change is dependency-supported and the production proof is adequate, but the PR still has a blocking polling lifecycle health regression and a status-diagnostic grace mismatch.

## What This Changes

The PR enables Telegram undici TCP keepalive, changes polling-start status patches to preserve `connected`, updates polling tests, and widens shared channel connect grace to 300s.

## Best Possible Solution

Keep the keepalive change, but change polling lifecycle status so startup hangs after reconnect become unhealthy after the grace window, then align Telegram status diagnostics with the same grace and add focused tests.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows a prior `connected:true` can survive restart, have transport timestamps cleared, and then pass health after grace if no first `getUpdates` succeeds. The original NAT-expiry stall was not independently reproduced in this read-only review.

## Solution Assessment

No as written. The undici keepalive part is a supported dependency use, but the status change needs an explicit connecting, disconnected, or stale-startup liveness signal instead of preserving `connected:true` with no transport timestamp.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve startup liveness after reconnects:** `extensions/telegram/src/polling-status.ts:18-23`
  - body: `setRuntime` patch-merges account status, and channel startup refreshes `lastStartAt` without clearing an old `connected:true`. With this patch, `notePollingStart` clears transport timestamps but leaves `connected` untouched, so after the 300s grace `evaluateChannelHealth` skips both disconnected and stale-socket checks and a startup hung before first `getUpdates` can stay healthy indefinitely. Preserve an explicit connecting/disconnected signal or add a stale-startup branch with coverage.
  - confidence: 0.91
- **[P2] Align Telegram status grace with health grace:** `src/gateway/channel-health-policy.ts:55`
  - body: `DEFAULT_CHANNEL_CONNECT_GRACE_MS` moves to 300s, but Telegram's runtime issue collector still suppresses disconnected polling issues for only 120s. During the newly allowed startup window, `openclaw channels status` can still tell users to restart even though the health monitor is intentionally waiting; update or share the grace constant and cover it in status tests.
  - confidence: 0.84
