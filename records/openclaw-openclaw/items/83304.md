## Summary

Keep open: the PR has sufficient copied live-output proof now, but the proposed polling-start status change introduces a blocking health-state regression for previously connected Telegram accounts that hang before the next successful getUpdates.

## What This Changes

The PR enables Telegram undici keepalive with a 30s initial delay, stops polling start from marking the account disconnected, updates polling tests, and raises shared channel connect grace to 300s.

## Best Possible Solution

Keep the keepalive and widened grace only with a lifecycle status design that still exposes startup failure after grace, and align Telegram status diagnostics with the same grace window.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows a previously connected account can restart with connected:true, have transport timestamps cleared, and then evaluate healthy after grace if startup hangs before getUpdates. The original NAT-stall production failure was not independently reproduced here.

## Solution Assessment

No as written. The undici keepalive option is dependency-supported, but the polling-start status change needs an explicit connecting, disconnected, or stale-startup liveness signal instead of preserving connected:true with no transport timestamp.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve startup liveness after reconnects:** `extensions/telegram/src/polling-status.ts:18-23`
  - body: `setRuntime` patch-merges account status, and channel startup refreshes `lastStartAt` without clearing an old `connected:true`. With this patch, `notePollingStart` clears `lastTransportActivityAt` but leaves `connected` untouched, so after the 300s grace `evaluateChannelHealth` skips both the disconnected and stale-socket checks and a startup hung before first `getUpdates` can stay healthy indefinitely. Preserve an explicit connecting/disconnected signal or add a stale-startup branch with coverage.
  - confidence: 0.91
- **[P2] Align Telegram status grace with health grace:** `src/gateway/channel-health-policy.ts:55`
  - body: `DEFAULT_CHANNEL_CONNECT_GRACE_MS` moves to 300s, but Telegram's status issue collector still uses `TELEGRAM_POLLING_CONNECT_GRACE_MS = 120_000`. During the newly allowed 2-3 minute startup window, `openclaw channels status` can still report polling as broken and tell users to restart even though the health monitor is intentionally waiting. Update that grace/tests or share the policy constant.
  - confidence: 0.84
