## Summary

Keep this PR open, but it is not merge-ready: the keepalive change is dependency-supported and the production proof is useful, while the polling-start status change can hide a failed reconnect after the widened grace window.

## What This Changes

The PR enables TCP keepalive for Telegram undici connections, changes polling lifecycle status patches, updates focused Telegram tests, and widens the shared channel connect grace from 120s to 300s.

## Best Possible Solution

Keep the TCP keepalive direction, but repair lifecycle status so failed reconnect startup becomes unhealthy after grace and align Telegram status diagnostics with the same grace window.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows a previously connected account can restart, preserve connected:true, clear transport timestamps, and then evaluate healthy after grace if startup hangs before the first getUpdates. The original NAT-stall failure is supported by production proof but was not independently induced here.

## Solution Assessment

No as written. The keepalive option is dependency-supported, but the status lifecycle needs an explicit connecting/disconnected/stale-startup signal and matching status diagnostics before the grace increase is safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve startup liveness after reconnects:** `extensions/telegram/src/polling-status.ts:18-23`
  - body: `setRuntime` patch-merges account status, and channel startup refreshes `lastStartAt` without clearing an old `connected:true`. With this patch, `notePollingStart` clears transport timestamps but leaves `connected` untouched, so after the 300s grace a startup hung before first `getUpdates` skips both disconnected and stale-socket checks and stays healthy. Publish an explicit connecting/disconnected signal or add a stale-startup branch with coverage.
  - confidence: 0.91
- **[P2] Align Telegram status grace with health grace:** `src/gateway/channel-health-policy.ts:55`
  - body: The health policy grace moves to 300s, but Telegram status issues still use a 120s polling connect grace. During the newly allowed 2-3 minute startup window, `openclaw channels status` can report polling as broken and tell users to restart while the health monitor is intentionally waiting. Share or update the grace and cover the status behavior.
  - confidence: 0.84
