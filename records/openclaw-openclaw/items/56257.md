## Summary

Keep this PR open, but it is not merge-ready: the declaration idea is still unimplemented on current main, the branch is conflicting with current plugin-audit refactors, and the proposed audit signal can miss enabled plugins while reporting disabled ones.

## What This Changes

The branch adds an optional plugin manifest `capabilities` declaration, a `plugins.capabilities.legacy_unrestricted` security-audit warning, and an RFC for phased runtime capability enforcement.

## Best Possible Solution

Land a maintainer-approved declaration contract in the current `audit-plugins-trust` path only after manifest roots, enabled-plugin scoping, docs wording, focused regression tests, and redacted real audit output are all covered.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the PR-introduced audit defects: `pluginDirs` are basenames, `loadPluginManifest` expects a plugin root, and the new loop bypasses current enabled-plugin scoping.

## Solution Assessment

No. A declaration-only phase may be reasonable, but this branch should first move onto the current audit module, fix root/scoping defects, avoid implying enforcement before runtime gates exist, and provide real audit output proof.

## Review Findings

Full review comments:

- **[P1] Resolve manifests from installed plugin roots:** `src/security/audit-extra.async.ts:833`
  - body: `pluginDirs` are directory names from the extensions directory, but this passes each bare name to `loadPluginManifest`, so installed plugins under `<stateDir>/extensions/<id>` will usually be skipped and the new warning can miss enabled legacy plugins. Join each id with `extensionsDir` before loading the manifest.
  - confidence: 0.94
- **[P2] Scope legacy capability warnings to enabled plugins:** `src/security/audit-extra.async.ts:831-834`
  - body: The loop scans every discovered extension directory instead of the enabled plugin ids already derived from config allow/deny/global-disabled state, so disabled or denied plugins can be reported as running legacy unrestricted. Scope this warning to enabled plugins or relabel it as installed inventory.
  - confidence: 0.88
- **[P2] Clarify that manifest capabilities are not enforced yet:** `docs/proposals/plugin-capability-model.md:90`
  - body: The RFC says plugins with a `capabilities` field enforce declared boundaries, but this PR only parses declarations and adds an audit warning. Mark that behavior as Phase 2 future work so operators do not treat a manifest field as an active runtime security boundary.
  - confidence: 0.82
