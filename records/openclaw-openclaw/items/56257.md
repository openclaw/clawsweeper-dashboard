## Summary

Keep open: current main does not implement this top-level plugin runtime capability model, but this PR is not merge-ready because it adds a security-sensitive manifest contract without real behavior proof and its new audit warning is wired to the wrong plugin roots and scope.

## What This Changes

The PR adds a `capabilities` field to plugin manifests, normalizes it, adds a legacy-unrestricted plugin audit warning, and documents a phased capability/budget RFC.

## Best Possible Solution

Define the capability schema through maintainer-approved security design, implement the audit in `src/security/audit-plugins-trust.ts` using resolved enabled plugin manifests, add focused tests/docs, and provide real `openclaw security audit` output proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: PR head passes installed plugin basenames to `loadPluginManifest`, and current main already has enabled-plugin scoping that the new warning bypasses.

## Solution Assessment

No. A declaration-only phase may be viable, but this implementation should first rebase onto the current trust-audit split, resolve manifest roots correctly, scope findings to enabled plugins, and get security/product approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve plugin manifests from the extensions root:** `src/security/audit-extra.async.ts:833`
  - body: `pluginDirs` are directory names from the extensions directory, but this passes each basename directly to `loadPluginManifest`. That makes the loader look under the process cwd instead of `<stateDir>/extensions/<plugin>`, so real installed legacy plugins can be skipped and the new security warning may not fire.
  - confidence: 0.94
- **[P2] Scope legacy capability warnings to enabled plugins:** `src/security/audit-extra.async.ts:831-834`
  - body: This scans every discovered plugin directory, including globally disabled, denied, or per-entry disabled plugins. The finding says those plugins run without capabilities, so inactive plugins can produce false security warnings and reduce operator trust in the audit output.
  - confidence: 0.88
