## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Load manifests from the installed plugin root:** `src/security/audit-extra.async.ts:831`
  - body: `pluginDirs` are directory names, not root paths. Passing the bare name to `loadPluginManifest` resolves relative to the process CWD, so installed legacy plugins can be silently omitted from the new audit warning; join each name with `extensionsDir` before loading.
  - confidence: 0.96
- **[P1] Do not let declarations hide unrestricted runtime access:** `src/security/audit-extra.async.ts:832`
  - body: This PR adds no runtime enforcement, but the audit stops warning as soon as `manifest.capabilities` exists. A plugin can add a declaration and still run with the full `PluginRuntime`, so the warning/remediation needs to report declaration-only status instead of implying the runtime boundary is hardened.
  - confidence: 0.9
- **[P2] Scope legacy capability warnings to enabled plugins:** `src/security/audit-extra.async.ts:829`
  - body: The new check scans every discovered plugin directory, while the existing trust audit computes enabled extension ids from `plugins.enabled`, allow/deny, and per-entry disabled state. Disabled plugins can be reported as running unrestricted even when they are not loaded, which makes the security audit noisy and misleading.
  - confidence: 0.9
