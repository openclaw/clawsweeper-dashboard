## Summary

Keep open: current main does not have the proposed manifest permission-declaration field or audit warning, and the PR is still an unmerged security-facing feature, but the current diff has blocking audit correctness issues before it can be reviewed for merge.

## What This Changes

The PR adds an optional `capabilities` field to native plugin manifests, normalizers for that declaration, a legacy-unrestricted plugin audit finding, and an RFC for phased runtime enforcement.

## Best Possible Solution

Define the capability schema with maintainer/secops approval, rebase onto the current trust-audit module, audit only enabled plugins using resolved manifest roots, and add focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection is enough: installed plugin discovery returns directory names, `loadPluginManifest` expects a root path, and current main has an enabled-plugin resolver that the PR bypasses for the new warning.

## Solution Assessment

No. A declaration-only phase can be reasonable, but this diff is not the best mergeable implementation until it fixes manifest path resolution, scopes warnings to enabled plugins, and gets maintainer security/product approval.
