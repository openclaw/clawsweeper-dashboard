## What This Changes

The PR adds an optional `PluginManifest.capabilities` schema and normalizer, a legacy-unrestricted plugin audit warning, and an RFC for phased runtime enforcement.

## Best Possible Solution

Rebase onto the current audit collector, land a maintainer-approved capability declaration contract, and make the audit load manifests from resolved enabled plugin roots with focused tests plus redacted real audit output.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the PR-introduced audit defects: `pluginDirs` are basenames, `loadPluginManifest` expects a plugin root, and the new loop bypasses enabled-plugin scoping.

## Solution Assessment

No. A declaration-only phase may be reasonable, but this implementation needs a clean rebase, fixed audit root/scoping, focused coverage, real audit output, and explicit maintainer approval for the public security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Resolve manifests from the installed plugin root:** `src/security/audit-extra.async.ts:831`
  - body: `pluginDirs` are directory names from the extensions directory, but this calls `loadPluginManifest(pluginDir, false)`, so installed plugins under `<stateDir>/extensions/<id>` will usually be skipped and the new security warning can miss enabled legacy plugins. Join each id with `extensionsDir` or use the current manifest registry path before loading the manifest.
  - confidence: 0.94
- **[P2] Scope legacy warnings to enabled plugins:** `src/security/audit-extra.async.ts:829-832`
  - body: The loop scans every discovered extension directory instead of the enabled plugin ids already derived from config allow/deny/global disabled state, so disabled or denied plugins can be reported as running legacy unrestricted. Scope this to enabled plugins or relabel it as an installed inventory check.
  - confidence: 0.88
