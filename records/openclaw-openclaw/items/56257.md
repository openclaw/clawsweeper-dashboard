## Summary

Keep open. Current main still does not implement this plugin manifest capability contract, and the PR has actionable security-audit correctness blockers plus missing real behavior proof for the new audit behavior.

## What This Changes

The PR adds optional plugin-manifest `capabilities` declarations, a `plugins.capabilities.legacy_unrestricted` security audit finding, and an RFC for phased capability enforcement.

## Best Possible Solution

Define the capability schema through maintainer-approved security design, implement the audit in the current trust-audit module using resolved enabled plugin manifests, add focused tests and docs, then provide real audit output proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current helpers return installed plugin basenames while the PR passes those basenames to `loadPluginManifest` and scans every discovered directory instead of the enabled plugin set.

## Solution Assessment

No. A declaration-only phase may be reasonable, but this implementation should first rebase onto the current audit split, resolve manifest roots correctly, scope findings to enabled plugins, and get security/product approval.
