## Summary

Keep open: the PR is a plausible security hardening direction, but current main does not already implement it and the branch still has blocking correctness, proof, conflict, and security/product-design gaps.

## What This Changes

The PR adds optional plugin-manifest runtime capability declarations, a legacy-unrestricted plugin audit warning, and an RFC for phased plugin capability enforcement.

## Best Possible Solution

Define the capability schema through maintainer-approved security design, implement the audit in the current trust-audit module using resolved enabled plugin manifests, add focused tests and docs, then provide real `openclaw security audit` output proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current helpers return installed plugin basenames, while the PR passes those basenames to `loadPluginManifest` and scans every discovered directory instead of the enabled set.

## Solution Assessment

No. A declaration-only phase may be reasonable, but this implementation should first rebase onto the current audit split, resolve manifest roots correctly, scope findings to enabled plugins, and get security/product approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve plugin manifests from the extensions root:** `src/security/audit-extra.async.ts:833`
  - body: `pluginDirs` are directory names from the extensions directory, but this passes each basename directly to `loadPluginManifest`. That makes the loader look under the process cwd instead of `<stateDir>/extensions/<plugin>`, so real installed legacy plugins can be skipped and the new security warning may not fire.
  - confidence: 0.94
- **[P2] Scope legacy capability warnings to enabled plugins:** `src/security/audit-extra.async.ts:831-834`
  - body: This scans every discovered plugin directory, including globally disabled, denied, or per-entry disabled plugins. The finding says those plugins run without capabilities, so inactive plugins can produce false security warnings and reduce operator trust in the audit output.
  - confidence: 0.88
