## Summary

Keep open: current main does not implement this capability model, and the PR still has blocking audit correctness issues plus a public security-contract decision for maintainers.

## What This Changes

The PR adds a public plugin-manifest capabilities declaration, capability normalizers, a legacy-unrestricted plugin audit warning, and an RFC for phased runtime enforcement.

## Best Possible Solution

Define the capability schema with maintainer security approval, rebase onto the current trust-audit module, audit only enabled plugins using resolved manifest paths, and add focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an installed plugin under `<stateDir>/extensions/demo/openclaw.plugin.json` can be skipped when the PR passes only `demo` to `loadPluginManifest`, and disabled installed plugins are still scanned because the loop uses all discovered plugin directories.

## Solution Assessment

No. A declaration-only phase can be reasonable, but this patch is not the best implementation until it is rebased onto the current audit module, fixes manifest path resolution, scopes findings to enabled plugins, and receives maintainer approval for the public security contract.

## Review Findings

Full review comments:

- **[P1] Join plugin dirs before loading manifests:** `src/security/audit-extra.async.ts:831`
  - body: `pluginDirs` are entry names from the extensions directory, but `loadPluginManifest(pluginDir, false)` resolves `openclaw.plugin.json` under that bare name from the process cwd.   - body: The loop scans every installed plugin directory, including plugins disabled globally, denied, or disabled per entry. The finding says those plugins run without declared capabilities, which is false for inactive plugins and creates noisy security audit output.
  - confidence: 0.88
