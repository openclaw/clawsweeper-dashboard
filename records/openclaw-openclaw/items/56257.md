## Summary

Keep open. Current main still lacks this declarative runtime-permissions manifest model, but the PR is not merge-ready because its new security audit warning can miss normal installed plugins, can report disabled plugins as running, and defines a public security-facing schema that needs maintainer product/security review.

## What This Changes

This PR adds an optional plugin manifest `capabilities` model, normalization types, a legacy-unrestricted plugin audit warning, and an RFC for phased runtime enforcement.

## Best Possible Solution

Rebase the work onto current main, settle the public capability schema and audit semantics with maintainers, then implement the manifest contract and enabled-plugin trust audit in the current audit module with regression tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the PR's audit loop receives directory names from `listInstalledPluginDirs`, passes them directly to `loadPluginManifest`, and scans all installed directories instead of the enabled-plugin set.

## Solution Assessment

No. A declaration-only phase could be maintainable, but this patch should first be rebased to the current audit module, load manifests from joined plugin roots, scope warnings to enabled plugins, and get security/product approval for the public contract.
