## Summary

Keep open: current main does not implement this capability model, and the PR still has blocking audit correctness issues plus missing real behavior proof for a security-sensitive contract.

## What This Changes

The PR adds an optional plugin manifest `capabilities` declaration, normalizes capability data, emits a legacy-unrestricted plugin security audit warning, and adds an RFC for phased enforcement.

## Best Possible Solution

Define the capability schema through security design, implement the audit against resolved enabled plugin manifests in the current trust-audit module, add focused tests/docs, and include real `openclaw security audit` output proof.

## Reproduction Assessment

Yes. Source inspection is enough for the PR-introduced audit bug: current helpers return extension basenames, while the PR passes those basenames into `loadPluginManifest` and scans all discovered directories.

## Solution Assessment

No. A declaration-only phase may be reasonable, but this implementation should first use the current trust-audit module, resolve enabled manifest records correctly, and get security/product approval.
