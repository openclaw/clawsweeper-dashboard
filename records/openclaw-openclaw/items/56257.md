## Summary

Keep open. The PR addresses a real plugin-security direction, but current main still lacks the proposed manifest capability contract and the PR’s audit implementation has concrete false-negative and false-positive behavior before it can merge.

## What This Changes

The PR adds an optional plugin manifest `capabilities` model, normalizers, a legacy-unrestricted plugin audit warning, and an RFC for phased runtime enforcement.

## Best Possible Solution

Rebase onto current main, get maintainer approval for the capability schema and rollout semantics, then implement the declaration contract in manifest code/docs and the enabled-plugin trust audit in `src/security/audit-plugins-trust.ts` with focused tests and changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an installed plugin at `<stateDir>/extensions/demo/openclaw.plugin.json` is skipped when the PR passes only `demo` to `loadPluginManifest`, and disabled installed plugins are still scanned because the loop uses all `pluginDirs`.

## Solution Assessment

No. A declaration-only first phase can be a maintainable approach, but this patch is not the best implementation until it is rebased, moved to the current audit module, scoped to enabled plugins, and backed by tests/docs for the public security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Join plugin dirs before loading manifests:** `src/security/audit-extra.async.ts:831`
  - body: `pluginDirs` are entry names from the extensions directory, but `loadPluginManifest(pluginDir, false)` resolves `openclaw.plugin.json` under the passed root. For a normal install under `<stateDir>/extensions/<id>`, this searches from the process cwd and can silently miss legacy plugins, so the new warning may not fire.
  - confidence: 0.94
- **[P2] Scope legacy warnings to enabled plugins:** `src/security/audit-extra.async.ts:829`
  - body: This loop scans every installed plugin directory, including plugins disabled globally, denied, or disabled per entry. The finding says those plugins run without declared capabilities, which is false for inactive plugins and creates noisy security audit results.
  - confidence: 0.88
