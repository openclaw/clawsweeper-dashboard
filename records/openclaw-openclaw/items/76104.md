## Summary

Keep open: current `main` still has a source-level route-resolution gap for the reported Feishu DM shape. Direct Feishu session keys do not resolve from the key alone, and the sessions-list fallback can normalize a stored `channel: "webchat"` ahead of the explicit Feishu `deliveryContext`.

## What This Changes

The issue asks for `sessions_send` replies from Feishu DMs to route back to the originating Feishu conversation instead of WebChat.

## Best Possible Solution

Make explicit persisted external `deliveryContext` authoritative for announce/direct delivery when the stored session channel is internal or stale, and cover the Feishu DM session shape with a regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: seed a session row for `agent:main:feishu:direct:ou_x` with `channel: "webchat"` and `deliveryContext: { channel: "feishu", to: "user:ou_x" }`, then assert `resolveAnnounceTarget` or `sessions_send` announce delivery resolves Feishu.

## Solution Assessment

Yes for the requested direction. The narrow maintainable fix is generic delivery resolution for explicit external `deliveryContext`, not a Feishu-specific branch or a new config flag.

## Repair Work Prompt

Fix the Feishu `sessions_send` announce-routing bug from https://github.com/openclaw/openclaw/issues/76104. Current source inspection shows direct keys like `agent:main:feishu:direct:ou_x` do not parse as announce targets, then the sessions-list fallback normalizes `channel: "webchat"` ahead of explicit `deliveryContext: { channel: "feishu", to: "user:ou_x" }`. Make explicit persisted external `deliveryContext` win when the competing stored session channel is the internal WebChat channel or otherwise stale/internal; do not add a Feishu-specific core branch or a config flag. Add regression coverage for the reported Feishu DM shape in the smallest appropriate unit tests, and preserve existing cross-channel carryover behavior for non-internal conflicts. Add a single-line `CHANGELOG.md` fix entry because this is user-facing.
