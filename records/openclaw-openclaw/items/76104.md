## Summary

Keep open: current `main` still has a source-level route-resolution gap for the reported Feishu DM shape. Prior webchat-routing issues were fixed for adjacent paths, but a direct Feishu session with stale `channel: "webchat"` can still lose the explicit Feishu delivery context during announce resolution.

## What This Changes

`sessions_send` replies from a Feishu DM should route back to the originating Feishu conversation instead of webchat.

## Best Possible Solution

Make persisted external `deliveryContext` authoritative for announce/direct delivery when the stored session channel is internal or stale, and cover the reported Feishu DM row with a regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: seed a session row for `agent:main:feishu:direct:ou_x` with `channel: "webchat"` and `deliveryContext: { channel: "feishu", to: "user:ou_x" }`, then assert `resolveAnnounceTarget` or `sessions_send` sends through Feishu.

## Solution Assessment

Yes. The narrow maintainable solution is generic delivery resolution for explicit external `deliveryContext`, not a Feishu-specific branch or a new config flag.

## Repair Work Prompt

Fix the Feishu direct `sessions_send` announce routing bug reported in https://github.com/openclaw/openclaw/issues/76104. Current source inspection shows `agent:main:feishu:direct:ou_x` does not resolve from the session key, and `deliveryContextFromSession` / `normalizeSessionDeliveryFields` lets stored `channel: "webchat"` take precedence over explicit external `deliveryContext.channel: "feishu"`. Implement a generic route-resolution fix so explicit persisted external delivery context is authoritative when the stored session channel is internal or stale; do not add a Feishu-only branch or a new config flag. Add a user-facing changelog entry. Validate with targeted delivery-context and sessions announce/send tests, then run the changed gate in Testbox before handoff.
