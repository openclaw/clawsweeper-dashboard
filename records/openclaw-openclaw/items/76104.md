## Summary

Keep open: the report identifies a still-plausible Feishu direct-session routing gap that is not covered by the earlier generic `sessions_send` route-preservation fixes. Current code preserves Feishu delivery for `chat.send` when `lastChannel` is already Feishu, but `sessions_send` announce fallback for direct session keys can still depend on normalized session-list delivery fields that may prefer the stored `channel` over `deliveryContext`.

## What This Changes

The issue asks for `sessions_send` replies from Feishu DMs to route back to the originating Feishu conversation instead of the internal webchat surface.

## Best Possible Solution

Make delivery resolution treat an explicit persisted `deliveryContext` as authoritative over an internal/webchat session `channel` for direct channel sessions, with a regression test for the exact Feishu DM shape.

## Reproduction Assessment

Yes. A focused unit or gateway-tool test can seed `agent:main:feishu:direct:ou_x` with `channel: "webchat"` and `deliveryContext.channel: "feishu"`, then assert `sessions_send` announce delivery resolves Feishu rather than webchat.

## Solution Assessment

Unclear until patched, but the safest direction is a generic delivery-context precedence fix or announce-target fallback fix, not Feishu-specific core branching.
