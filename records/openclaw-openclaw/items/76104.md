## Label Justifications

- `P1`: Recent users report a broken Feishu channel workflow where sessions_send replies are delivered to webchat instead of the active Feishu conversation.
- `impact:message-loss`: The originating Feishu user receives no response because the reply is misrouted to WebChat.
- `impact:session-state`: The failure is driven by persisted session routing state where channel/webchat conflicts with deliveryContext/feishu.

## Summary

Keep open: current main still normalizes persisted session route fields before the explicit Feishu deliveryContext, so a session row with stale/internal webchat channel data can still resolve the sessions_send announce target to webchat. Recent issue comments also corroborate the same Feishu/WebChat conflict on OpenClaw 2026.5.7, and no merged PR or canonical open PR appears to own this exact repair.

## What This Changes

Fix sessions_send reply routing so Feishu-originated sessions reply to the Feishu origin channel instead of stale/internal webchat session metadata.

## Best Possible Solution

Make explicit persisted external deliveryContext authoritative for sessions_send announce/direct delivery when stored session route fields point at stale or internal webchat, with regressions for Feishu DM, Feishu group, and control-ui-triggered routes.

## Reproduction Assessment

Yes, at source level. Seed a sessions.list row with a Feishu session key, stored channel or lastChannel as webchat, and deliveryContext { channel: "feishu", to: "user:ou_x" }; deliveryContextFromSession currently keeps the stale/internal primary route over the explicit Feishu context.

## Solution Assessment

Yes. The narrow maintainable fix is a generic stale/internal-route precedence repair for persisted external deliveryContext, not a Feishu-specific branch or a new config flag.
