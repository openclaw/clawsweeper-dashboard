## Summary

Keep open: current main still has the same source-level route precedence gap. For the reported Feishu shape, sessions_send announce delivery can fall back to the stored session row and normalize the stale/internal webchat channel ahead of the explicit Feishu deliveryContext.

## What This Changes

The issue asks for sessions_send replies from Feishu-origin sessions to route back to Feishu instead of webchat when stored session.channel conflicts with deliveryContext.channel.

## Best Possible Solution

Make explicit persisted external deliveryContext authoritative for announce/direct delivery when stored session route fields point at stale/internal webchat, and add regression coverage for the Feishu DM plus related group/control-ui shapes.

## Reproduction Assessment

Yes, by source inspection: seed a session row for agent:main:feishu:direct:ou_x with channel webchat and deliveryContext { channel: feishu, to: user:ou_x }, then resolve sessions_send announce delivery and observe that current normalization can keep webchat primary. I did not run a live Feishu websocket setup in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is generic deliveryContext precedence for stale/internal stored routes, not a Feishu-specific branch or a new config flag.
