## Summary

Keep open: current main still has a source-level routing gap for the reported Feishu direct-session shape. Earlier generic route-preservation work covers adjacent webchat-overwrites-external-channel cases, but `sessions_send` announce fallback can still lose an explicit Feishu `deliveryContext` when the stored session channel is `webchat`.

## What This Changes

The issue asks for `sessions_send` replies triggered from Feishu direct messages to route back to the Feishu origin instead of the dashboard webchat channel.

## Best Possible Solution

Make persisted external `deliveryContext` authoritative for direct/announce delivery when the stored session channel is internal or stale, with a generic regression test for the reported Feishu DM shape.

## Reproduction Assessment

Yes. A focused unit test can seed a session row for `agent:main:feishu:direct:ou_x` with `channel: "webchat"` and `deliveryContext: { channel: "feishu", to: "user:ou_x" }`, then assert `sessions_send` announce delivery resolves Feishu.

## Solution Assessment

Yes for the requested direction, but the safest fix is generic delivery resolution for explicit external `deliveryContext`, not a Feishu-specific core branch or a new config flag.

Fix the Feishu direct-session `sessions_send` announce route reported at https://github.com/openclaw/openclaw/issues/76104.
