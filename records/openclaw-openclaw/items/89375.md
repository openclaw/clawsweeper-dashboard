## Summary

Keep open: current main and the latest release still use listener-scoped stale gateway cleanup and conservative live OpenClaw session-lock preservation, so the reported no-longer-listening orphaned gateway holding a session write lock is not fixed or superseded.

## What This Changes

The issue reports that gateway restart stale-process cleanup can miss an orphaned old gateway process, leaving a session write lock held until later requests time out.

## Best Possible Solution

Keep this open for a maintainer-owned restart reliability fix that identifies same-instance orphaned gateway lock holders without killing legitimate active writers, with live macOS/service proof and coordinated session-lock regression tests.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The report's logs are concrete and current source makes the miss plausible because stale gateway cleanup is listener-scoped while live OpenClaw-looking session lock owners are preserved.

## Solution Assessment

Unclear: the requested verification and cleanup direction is plausible, but the best fix needs maintainer-chosen ownership rules for detecting and retiring same-instance orphaned gateways without stealing legitimate transcript writes.
