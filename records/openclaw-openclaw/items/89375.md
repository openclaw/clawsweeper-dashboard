## Summary

Keep open: current main still only cleans stale gateway PIDs discovered as verified listeners on the configured port, while current session-lock cleanup intentionally preserves live OpenClaw-owned stale locks, so the reported orphaned gateway lock holder is not fixed or superseded.

## What This Changes

The issue reports a SIGUSR1 gateway restart missing an orphaned old gateway process that continues holding a session write lock and blocks later requests.

## Best Possible Solution

Keep this issue open until maintainers choose and prove a gateway restart/session-lock policy that handles same-instance orphaned gateway lock holders beyond listener-only PID cleanup.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The issue logs are concrete and current source makes the failure plausible because restart cleanup is listener-scoped while live OpenClaw lock owners are preserved.

## Solution Assessment

Unclear: broader stale process detection and lock reclamation are both plausible, but the best fix needs a maintainer-chosen safety boundary for retiring orphaned OpenClaw owners without corrupting active transcript writes.
