## Summary

Current `main` still appears to have the reported gap: the security audit continues to flag installed extensions when `plugins.allow` is unset, and the current auto-enable path intentionally does not create a new `plugins.allow` list during config materialization or upgrade-like startup flows.

## Evidence

- **Audit finding still present on main:** `runPluginTrustAudit` still emits `plugins.extensions_no_allowlist` with title `Extensions exist but plugins.allow is not set` when extension directories exist and no explicit allowlist is configured.
  - file: [src/security/audit-plugins-trust.ts:336](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/security/audit-plugins-trust.ts#L336)
  - command: `nl -ba src/security/audit-plugins-trust.ts | sed -n '320,355p'`
  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Allowlist helper cannot create a missing allowlist:** `ensurePluginAllowlisted` returns the config unchanged unless `cfg.plugins.allow` is already an array, so it only appends to an existing allowlist and never initializes one.
  - file: [src/config/plugins-allowlist.ts:11](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/config/plugins-allowlist.ts#L11)
