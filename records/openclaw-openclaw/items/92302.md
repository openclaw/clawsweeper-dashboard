## Summary

Keep this issue open: current `main` and latest `v2026.6.6` still route `memory.qmd.command` through POSIX-style shell parsing, and the live clean fix path is the open linked PR https://github.com/openclaw/openclaw/pull/92560.

## What This Changes

The issue reports that Windows absolute `memory.qmd.command` paths for the QMD memory backend are parsed as shell escapes and then resolved incorrectly, making QMD unusable from OpenClaw on Windows.

## Best Possible Solution

Review and land the linked PR after adequate Windows behavior proof, then close this issue when the fix reaches `main`.

## Reproduction Assessment

Yes for source-level reproduction: current `main` still sends `memory.qmd.command` through `splitShellArgs`, whose unquoted backslash handling strips Windows separators. I did not run a live Windows QMD setup in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix is at the QMD command normalization boundary before Windows spawn resolution, and the open linked PR targets that boundary. It still needs PR review and real Windows behavior proof before this issue should close.
