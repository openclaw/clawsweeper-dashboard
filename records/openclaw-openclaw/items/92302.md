## Summary

Keep open: current `main` and the latest release still pass `memory.qmd.command` through POSIX-style parsing that strips unquoted Windows backslashes, while the direct fix attempts are closed unmerged.

## What This Changes

The issue reports that native Windows QMD memory users cannot use absolute `memory.qmd.command` paths because OpenClaw mangles the path before spawning QMD.

## Best Possible Solution

The end state should preserve Windows drive-letter and UNC executable paths before shell parsing, with focused regression tests and real Windows QMD proof before closing the issue.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a raw Windows `C:\...` command reaches `splitShellArgs`, whose unquoted backslash handling strips separators before spawn resolution. I did not run a live Windows QMD repro.

## Solution Assessment

Yes: fixing the QMD command-normalization boundary before spawn is the narrow maintainable solution. A shell fallback or new config option is not needed for the reported path-mangling bug.
