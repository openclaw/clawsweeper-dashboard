## Summary

Keep this issue open: current main and latest v2026.6.6 still pass `memory.qmd.command` through the backslash-consuming shell parser, and the active fix path is the open linked PR https://github.com/openclaw/openclaw/pull/92867.

## What This Changes

The issue reports that Windows absolute paths configured in `memory.qmd.command` are mangled before QMD is spawned, making the QMD memory backend unusable on the reported Windows setup.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/92867, then close this issue once main preserves Windows absolute QMD command paths with adequate proof.

## Reproduction Assessment

Yes for source reproduction: current main passes raw `memory.qmd.command` through `splitShellArgs`, and that parser consumes backslashes outside quotes. I did not run a live Windows gateway/QMD setup in this read-only review.

## Solution Assessment

Yes in direction: the fix belongs at `resolveMemoryBackendConfig` before QMD spawn resolution, but the current open PR needs its review blocker and Windows behavior proof handled before merge.
