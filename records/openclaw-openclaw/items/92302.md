## Summary

Keep open: current main and v2026.6.6 still route `memory.qmd.command` through POSIX-style shell parsing, and the issue now has an open linked fix PR that should be reviewed and landed rather than closing the issue early.

## What This Changes

Fix QMD memory backend command resolution on Windows so configured absolute `memory.qmd.command` paths and QMD invocation work without path mangling.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92560 or a successor after it preserves configured Windows absolute commands and proves real Windows QMD indexing/search, including the reported default-command behavior.

## Reproduction Assessment

Yes for the configured-path failure: current source sends a Windows backslash path through `splitShellArgs`, whose parser consumes unquoted backslashes. I did not run a live Windows QMD setup, so the omitted-command branch remains unproven in a real environment.

## Solution Assessment

Yes in direction: the narrow fix belongs before `splitShellArgs` in QMD command resolution. The open PR should still be reviewed for Windows runtime proof and coverage of the default bare `qmd` invocation.
