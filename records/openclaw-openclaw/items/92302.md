## Summary

Keep this issue open: current main and the latest release still parse `memory.qmd.command` through POSIX-style shell splitting, and the active fix path is the open closing PR at https://github.com/openclaw/openclaw/pull/92867.

## What This Changes

The issue reports that Windows absolute paths in `memory.qmd.command` are mangled before QMD is spawned, leaving the QMD memory backend unusable on the reported Windows setup.

## Best Possible Solution

Land a narrow command-normalization fix in the QMD backend resolver with focused regression coverage and real Windows QMD behavior proof, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes for source-level reproduction: current main still routes `memory.qmd.command` through a parser that consumes unquoted backslashes, matching the reported Windows path mangling. I did not run a live Windows QMD setup in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix belongs at the QMD command normalization boundary before `splitShellArgs`. The current issue should stay open while the linked PR is reviewed, proofed, and either landed or replaced.
