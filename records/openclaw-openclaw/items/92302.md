## Summary

Keep this issue open: current main and the latest release still parse `memory.qmd.command` with the POSIX-style shell tokenizer, so the reported Windows backslash path mangling remains until one of the linked fix PRs lands.

## What This Changes

The issue reports that QMD memory backend command paths configured as absolute Windows paths lose separators before QMD spawn resolution, making QMD indexing/search unusable on Windows.

## Best Possible Solution

Review and land one canonical backend-config command-normalization fix, with `https://github.com/openclaw/openclaw/pull/95274` currently the strongest proof-positive candidate, then close this issue as fixed.

## Reproduction Assessment

Yes, source-level. Current main still routes unquoted `memory.qmd.command` through `splitShellArgs`, and the parser consumes Windows backslashes before QMD spawn resolution.

## Solution Assessment

Yes, the best fix is to preserve Windows drive-letter and UNC executable command tokens before shell-style parsing in backend config. Adding a shell fallback or changing the shared parser globally would be broader and riskier.
