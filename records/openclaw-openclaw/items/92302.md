## Summary

Keep this issue open: current main and the latest release still pass `memory.qmd.command` through POSIX-style shell parsing, so unquoted Windows backslash paths can still be mangled before QMD spawn resolution.

## What This Changes

Windows QMD users report that absolute `memory.qmd.command` paths are parsed as shell text, stripping separators and making the QMD memory backend unusable despite a working QMD CLI.

## Best Possible Solution

Add a focused command-normalization fix in the QMD backend config path that preserves Windows drive-letter and UNC absolute commands before shell parsing, with regression tests and Windows QMD behavior proof before closing this issue.

## Reproduction Assessment

Yes, source-level: current main still routes `memory.qmd.command` through `splitShellArgs`, whose unquoted backslash handling strips Windows separators. I did not run a live Windows QMD repro.

## Solution Assessment

No current solution is on main. The best fix is at the QMD command-normalization boundary before shell parsing and Windows spawn resolution, not a new config option or shell fallback.
