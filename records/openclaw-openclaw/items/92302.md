## Summary

Keep this issue open: current main and v2026.6.9 still route `memory.qmd.command` through the POSIX shell splitter that strips unquoted Windows backslashes, and https://github.com/openclaw/openclaw/pull/95274 is the live closing candidate.

## What This Changes

Fix Windows QMD memory backend command resolution so absolute `memory.qmd.command` paths keep their separators before QMD spawn resolution.

## Best Possible Solution

Land one canonical backend-config command-token fix, with https://github.com/openclaw/openclaw/pull/95274 as the current live candidate if review remains clean, then close this issue as implemented.

## Reproduction Assessment

Yes, source-level. Current main routes unquoted `memory.qmd.command` through `splitShellArgs`, whose unquoted-backslash handling strips Windows separators before QMD spawn resolution; the issue discussion also includes Windows output using the exported parser.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve Windows drive-letter and UNC executable command tokens in backend config before shell-style parsing, rather than changing the shared parser globally or trying to recover the command later in spawn resolution.
