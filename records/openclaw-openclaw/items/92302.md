## Summary

Keep this issue open: current main and the latest release still have the reported QMD command parser path, and two open clean PRs currently claim this issue with the same backend-config fix boundary.

## What This Changes

The issue reports that the QMD memory backend on Windows mangles absolute `memory.qmd.command` paths through POSIX-style parsing, leaving QMD index/search unusable.

## Best Possible Solution

Land one canonical backend-config command-token fix, preferably https://github.com/openclaw/openclaw/pull/95274 if review remains clean, then close this issue as implemented.

## Reproduction Assessment

Yes. Current main routes `memory.qmd.command` through `splitShellArgs`, and that parser consumes unquoted backslashes before QMD spawn code sees the command; the issue discussion also includes Windows output using the exported parser.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve Windows drive-letter and UNC executable command tokens in backend config before shell-style parsing, rather than changing the shared parser globally or trying to repair the command later in spawn resolution.
