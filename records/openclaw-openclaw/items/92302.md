## Summary

Keep open: current main and the latest release still route the configured QMD command through POSIX-style parsing, and the open linked PR https://github.com/openclaw/openclaw/pull/92560 is the active implementation candidate for this issue.

## What This Changes

The issue reports that Windows absolute `memory.qmd.command` paths are mangled before QMD spawn, leaving QMD memory indexing and search unusable on Windows despite a working QMD CLI.

## Best Possible Solution

Review and land a narrow fix that preserves Windows absolute `memory.qmd.command` before shell parsing, validates explicit path and bare `qmd` behavior on Windows, and then closes this issue from the merged PR.

## Reproduction Assessment

Yes for source-level reproduction: current main sends a backslash-delimited Windows `memory.qmd.command` through `splitShellArgs`, which consumes unquoted backslashes before QMD spawn. I did not run a live Windows QMD indexing flow in this read-only review.

## Solution Assessment

Yes, keeping this issue open is the right path because GitHub recognizes an open closing PR as the implementation candidate. The repair should stay at the command-resolution boundary, not become a new config workaround or a competing fix PR.
