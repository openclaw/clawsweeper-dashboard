## Summary

Keep this issue open: current main and the latest release still route `memory.qmd.command` through the POSIX shell splitter, while the open focused fix PR is the active path to resolve it.

## What This Changes

Fix Windows QMD memory backend command parsing so absolute `memory.qmd.command` paths keep their separators and can be spawned.

## Best Possible Solution

Review and land the focused backend-config command-token fix in https://github.com/openclaw/openclaw/pull/95274, or reject it and replace it with an equally narrow fix at the same boundary.

## Reproduction Assessment

Yes, from source: an unquoted `C:\...` command reaches `splitShellArgs`, whose outside-quote backslash branch consumes separators before `resolveMemoryBackendConfig` stores the command token. I did not run a live Windows gateway repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to preserve Windows drive-letter and UNC executable command tokens in backend config before shell-style parsing, rather than changing the shared parser globally or trying to repair the command later in spawn resolution.
