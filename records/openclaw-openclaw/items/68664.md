## Summary

Keep open: current main still rejects obsolete `memorySearch` keys through the strict config schema before doctor legacy migrations run in the startup/read paths, and the issue is paired with the open implementation PR https://github.com/openclaw/openclaw/pull/68685.

## What This Changes

The issue requests a recovery path for legacy `memorySearch` config keys that currently cause upgrade/startup validation failures.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land, replace, or close https://github.com/openclaw/openclaw/pull/68685 with an explicit doctor-only versus startup/read self-healing decision.

## Reproduction Assessment

Yes. Source inspection shows obsolete `memorySearch` keys reach a strict schema before doctor migrations run in startup/read paths, and current main lacks an obsolete-key cleanup migration; I did not run a live repro because this review is read-only.

## Solution Assessment

No, not as broadly filed. A narrow doctor migration for obsolete keys is plausible, but globally reordering startup validation or adding update-time migration needs maintainer approval because current docs say startup refuses legacy config and does not rewrite it.
