## Summary

Keep open: current main still has strict `memorySearch` validation and no obsolete-key cleanup before startup/read validation, and the issue is paired with the still-open implementation PR https://github.com/openclaw/openclaw/pull/68685.

## What This Changes

The issue asks OpenClaw to prevent legacy `memorySearch` keys such as `chunkSize`, `chunkOverlap`, and root-level `maxResults` from blocking service startup after config schema changes.

## Best Possible Solution

Keep this issue as the canonical tracker while maintainers choose doctor-only obsolete-key cleanup versus startup/read self-healing, then resolve it through the linked PR or a replacement with real runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main shows obsolete `memorySearch` keys reach a strict schema before any doctor legacy migration can strip them; I did not run a live service repro because this review is read-only.

## Solution Assessment

No, not as broadly filed. A narrow obsolete-key migration is clear, but moving doctor migrations into startup/update changes the documented migration boundary and needs maintainer approval; the linked open PR addresses cleanup but not the full startup/read behavior.
