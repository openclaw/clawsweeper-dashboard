## Summary

Keep this issue open. Current main still has the specific stale `agents.defaults.memorySearch` sub-key gap, and the provided context shows open PR #68685 with closing syntax for the same bug, so the issue should wait for that PR to merge, be replaced, or be closed by maintainer decision.

## What This Changes

The issue requests upgrade-safe config migration so obsolete `memorySearch` keys are removed before strict schema validation can crash gateway startup.

## Best Possible Solution

Land #68685 or an equivalent focused fix that strips only the obsolete `agents.defaults.memorySearch` keys during the existing legacy migration pass, with regression coverage; handle update-command automation separately if maintainers want that broader product behavior.

## Reproduction Assessment

Yes. A config containing `agents.defaults.memorySearch.chunkSize`, `chunkOverlap`, or root-level `maxResults` reaches a strict `MemorySearchSchema` with no current-main migration that removes those keys; this was verified statically under the read-only review constraints.

## Solution Assessment

Yes for the narrow bug direction. A focused legacy migration plus regression test is more maintainable than changing the global validation order, because current main already runs runtime legacy migrations before validation.
