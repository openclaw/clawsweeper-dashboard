## Label Justifications

- `P1`: The report describes an upgrade-triggered config validation failure that can keep the gateway in a restart loop for existing users.
- `impact:crash-loop`: The affected startup path can reject config before the gateway runs, causing repeated service restarts until manual repair.

## Summary

Keep open: current main and the latest release still reject the reported flat obsolete `memorySearch` keys, and no current migration strips them before startup validation. The linked closing PR remains open, unmerged, and currently conflicting, so it is the active fix path rather than a reason to close this issue.

## What This Changes

The issue requests a safe config migration path so obsolete `agents.defaults.memorySearch` keys like `chunkSize`, `chunkOverlap`, and flat `maxResults` do not crash gateway startup after schema changes.

## Best Possible Solution

Land or recreate the narrow doctor/runtime migration from the linked PR so the obsolete memorySearch keys are stripped with regression coverage, then handle any broader startup or update auto-repair policy as a separate maintainer decision.

## Reproduction Assessment

Yes, from source: current main has strict memorySearch validation and no specific migration for the reported flat keys, so that config remains invalid before startup can proceed. I did not run a live gateway repro because this is a read-only review.

## Solution Assessment

No, current main is not the best end state because the specific obsolete-key migration is still absent. The best fix direction is the narrow doctor migration in the linked PR or an equivalent replacement, with separate maintainer judgment for broader automatic startup/update migration behavior.
