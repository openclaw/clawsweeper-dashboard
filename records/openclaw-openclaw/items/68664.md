## Summary

Keep open: current main still validates the startup/read config path before doctor legacy migrations and still lacks cleanup for obsolete `memorySearch` keys, while the linked closing PR remains open and unmerged.

## What This Changes

The issue asks OpenClaw to migrate or strip legacy `memorySearch` config keys before schema validation can block gateway startup.

## Best Possible Solution

Keep the issue open until a maintainer-approved config recovery path strips known-obsolete `memorySearch` keys before validation blocks startup, or explicitly scopes the linked PR to doctor-only cleanup and tracks startup self-healing separately.

## Reproduction Assessment

Yes from source inspection. A config with stale `memorySearch` keys reaches strict `MemorySearchSchema` in `loadConfig` and `readConfigFileSnapshot` before doctor migrations, and current tests assert no read-time auto-migration for top-level `memorySearch`.

## Solution Assessment

No as a simple validation-order reorder. Current policy and tests make startup self-healing a maintainer decision, so the safer path is an approved migration boundary with regression coverage and real runtime proof.
