Keep open: current main and v2026.6.6 still lack a specific doctor migration for the reported obsolete flat `memorySearch` keys, and the linked closing PR remains open and unmerged.

## What This Changes

The issue asks OpenClaw to prevent upgrade-time gateway crash loops when strict config validation rejects obsolete `agents.defaults.memorySearch` keys before the config is repaired.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68685 or a narrow replacement that strips the obsolete memorySearch keys through the doctor migration path with regression coverage, then decide separately whether any startup/update auto-repair behavior belongs in core.

## Solution Assessment

No, current main is not the best end state because the specific obsolete-key migration remains absent. The maintainable path is a focused doctor migration plus regression tests, with maintainer judgment reserved for any broader startup/update auto-migration behavior.
