## Summary

Keep open: current main still rejects the legacy `memorySearch` shapes on the normal startup/read validation path, and the issue is paired with an open, unmerged closing PR that needs maintainer review rather than pre-merge closure.

## What This Changes

The issue requests that legacy config migrations handle obsolete `memorySearch` keys before strict validation can crash startup after schema changes.

## Best Possible Solution

Keep this issue open while maintainers review the linked PR and decide whether the accepted scope is doctor-only obsolete-key cleanup or an explicit startup/read self-healing change with regression coverage.

## Reproduction Assessment

Yes. Source inspection shows obsolete `memorySearch` keys reach a strict schema on the startup/read validation path before doctor migrations run, and current tests preserve the no-read-time-auto-migration boundary.

## Solution Assessment

No as filed. Reordering global startup validation or adding automatic update migration conflicts with the documented doctor-owned repair boundary; the narrower maintainable path is a maintainer-approved PR with clear doctor-only or startup-self-healing scope.
