## Summary

Keep open: current main still validates the startup/read config path before doctor legacy migrations can strip or move legacy `memorySearch` settings, and the issue is paired with an open, unmerged closing PR that should be reviewed before this tracker closes.

## What This Changes

The issue reports that obsolete legacy `memorySearch` config keys can fail strict validation before the migration path removes them, causing startup/service failure after schema changes.

## Best Possible Solution

Keep this issue open while maintainers review, land, replace, or close https://github.com/openclaw/openclaw/pull/68685, with an explicit scope decision between doctor-only cleanup and startup/read-time self-healing.

## Reproduction Assessment

Yes. Source inspection shows obsolete `memorySearch` keys reach a strict schema during load/read validation before doctor migrations run, and current tests explicitly preserve the no-read-time-auto-migration boundary.

## Solution Assessment

No as filed. Reordering global validation or adding startup/update auto-migration is broader than the documented doctor-owned legacy repair policy; the maintainable path is maintainer scope approval plus focused regression coverage and runtime proof.
