## Summary

Keep open. Current main has post-failure billing/rate-limit cooldowns, model fallback, provider usage snapshots, and a LiteLLM spend-limit workaround, but no native pre-request spend threshold that can block or skip a provider before inference; the right owner boundary needs maintainer product/API review.

## What This Changes

The issue requests an opt-in pre-request budget or spend-limit check that can skip a model provider before paid API calls when a configured quota or cost threshold is exceeded.

## Best Possible Solution

Design an opt-in generic provider budget guard that can make a pre-inference block/fallback decision, preferably reusing provider usage snapshots or a narrow provider-runtime hook, while documenting LiteLLM virtual-key spend limits as the current workaround.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source inspection confirms current main has post-failure cooldowns and usage display, but no configured pre-request spend threshold.

## Solution Assessment

Unclear; the proposed spendLimit config is plausible but not yet clearly the best shape. Maintainers should first decide whether this belongs in core config, a provider-runtime hook, or documented external proxy spend limits.
