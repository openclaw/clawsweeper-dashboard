## Summary

The PR correctly centralizes the safety policy and provides strong real CLI proof, but it introduces an upgrade regression: any legacy server containing a blocked env key can no longer use unrelated configure or tool-filter commands, while the CLI provides no supported way to remove that key. The PR should remain open for a focused compatibility repair. Likely related people: @steipete for the MCP transport seam and Marcus Castro for the MCP CLI/config mutation surface.

## What This Changes

The PR centralizes MCP stdio environment safety policy, rejects blocked env keys before config writes and probes, reports legacy blocked configs in status/doctor, and removes repeated runtime warnings.

## Best Possible Solution

Continue rejecting newly introduced blocked stdio env keys, but add a supported cleanup path for legacy entries and avoid rejecting unrelated mutations that do not introduce or retain a newly supplied unsafe key.

## Reproduction Assessment

Yes. Current main accepts blocked stdio env keys at the MCP config-write boundary and filters them during transport resolution, matching the linked issue; the proposed legacy mutation failure is also directly reproducible from the complete-server validation path.

## Solution Assessment

No, not in its current form. Centralizing policy and rejecting unsafe new writes is the right boundary, but the implementation must preserve a supported legacy cleanup path and unrelated configuration mutations.
