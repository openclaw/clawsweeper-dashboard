## Summary

Keep this PR open. Current main still renders Android chat text through the markdown path with no Adaptive Card parser or renderer, but the PR diff is not merge-ready because it introduces source-confirmed crashes, non-functional actions, ordering/data-loss issues, missing template application, and no Android tests or changelog entry.

## What This Changes

The PR adds Android parsing for `<!--adaptive-card-->` marker blocks and a Jetpack Compose Adaptive Card renderer wired into chat message rendering.

## Best Possible Solution

Decide the rich-message boundary first, then either route Android through the shared presentation/client contract or intentionally adopt native Adaptive Card rendering with ordered mixed text/card segments, multiple-card support, template expansion, explicit action policy, depth and numeric guards, focused Android tests, and a changelog entry.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main renders marker text as markdown, while PR head can be exercised with intro + card + suffix text, multiple marker blocks, separate template-data markers, invalid column widths, negative `Rating.max`, deeply nested containers, and Submit/Execute actions.

## Solution Assessment

No. The proposed direct marker parser plus Compose renderer is not the best mergeable solution as written because it lacks action routing, ordered segmentation, template handling, multi-card support, defensive rendering guards, tests, and clear alignment with the shared presentation direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire submit actions before enabling buttons:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:543-545`
  - body: `Action.Submit` and `Action.Execute` render enabled buttons, but tapping them only writes a debug log. Cards that depend on submit or execute workflows will look interactive and then do nothing, so wire a real route or render these actions disabled/unsupported.
  - confidence: 0.95
- **[P1] Validate column weights before applying them:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:191-196`
  - body: Card widths such as `"0"`, `"-1"`, or `"NaN"` are parsed and passed straight to `Modifier.weight(weight)`. Compose requires a finite positive weight, so malformed or model-generated cards can crash chat rendering instead of degrading.
  - confidence: 0.93
- **[P1] Bound recursive card rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:100-105`
  - body: `RenderElement` can recurse through containers, column sets, table cells, and action sets without carrying a depth limit. A deeply nested card payload can exhaust the stack and crash Android chat rendering; add a maximum depth and fallback/skip beyond it.
  - confidence: 0.9
- **[P2] Preserve card and text order:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:123-128`
  - body: `fallbackText` contains all non-card text after stripping markers, but `ChatMessageBody` renders that entire fallback before the card. Messages shaped as intro + card + trailing explanation display in the wrong order, so return and render ordered segments.
  - confidence: 0.94
- **[P2] Render all card blocks or preserve unmatched blocks:** `apps/android/app/src/main/java/ai/openclaw/app/chat/AdaptiveCardParser.kt:23-39`
  - body: `parseAdaptiveCardMarkers` captures only the first card block, while fallback generation removes every card marker block. A message with multiple adaptive cards renders the first and silently drops the rest; parse/render all matches or leave unrendered blocks visible as fallback.
  - confidence: 0.91
- **[P2] Apply parsed template data before rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:122-128`
  - body: The parser extracts `templateData`, but `ChatMessageBody` passes only `parsed.card` into `AdaptiveCardView`. Cards using `${...}` template fields will render unresolved placeholders even when valid data markers are present.
  - confidence: 0.89
- **[P2] Clamp rating max before coercing:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:474-480`
  - body: `RenderRating` uses card-provided `max` directly as the upper bound for `value.toInt().coerceIn(0, max)`. A malformed card with a negative max throws `IllegalArgumentException`; clamp or reject invalid max values first.
  - confidence: 0.88
- **[P2] Handle numeric column widths:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:191-193`
  - body: `RenderColumnSet` only reads `column["width"]` as a `String`, but weighted Adaptive Card widths may arrive from JSON as numbers. Those cards fall back to equal columns, so parse numeric and string width values before defaulting.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:121-128`
  - body: This is a user-facing Android feature, but the PR changes only Android source files and has no `CHANGELOG.md` entry. Add a single-line entry in the active changelog section with appropriate contributor credit before merge.
  - confidence: 0.87

## Security Review

Summary: The diff adds no dependency or workflow supply-chain change, but it introduces native rendering of untrusted card payloads with concrete denial-of-service and action-data handling concerns.
