## Summary

Keep this PR open. Current main still lacks Android inline Adaptive Card parsing/rendering, so the contribution is not obsolete, but the proposed branch is not merge-ready: it adds a marker-based Android renderer that drops or reorders mixed content, ignores extracted template data, leaves submit/execute actions as enabled no-op buttons, and can crash on malformed card numeric/layout data. The external openclaw-adaptive-cards plugin and companion client PRs are useful context, but Android-native rendering needs maintainer product/security review before merging or moving behind a documented client-renderer API.

## What This Changes

The PR adds Android chat-side parsing of `<!--adaptive-card-->` marker blocks and a Jetpack Compose renderer for Adaptive Card elements in chat bubbles.

## Best Possible Solution

Decide the product boundary first: either adopt native mobile card rendering through a documented, testable client-renderer or `MessagePresentation` path, or keep Adaptive Cards in the external plugin and add the missing Android client extension API. If native Android rendering is desired, the implementation should preserve ordered mixed text/card segments, support multiple cards and template data, make submit/execute actions either real or clearly non-interactive, enforce render limits and numeric guards, and add focused Android parser/render tests.

## Reproduction Assessment

Yes. The review findings can be reproduced from PR head 51f5f23 by rendering text with prefix/card/suffix or multiple card blocks, a card with template-data markers, a card with `Column.width` of `0`, `-1`, or `NaN`, a negative rating `max`, or a deeply nested `Container`/`ColumnSet`; current main also has a source-backed reproduction for the missing feature because adaptive-card markers still flow through plain `ChatMarkdown`.

## Solution Assessment

No. The proposed marker parser plus direct Compose renderer is not the best mergeable solution as written because it bypasses the current shared presentation direction, loses ordered content, lacks action policy/routing, and trusts malformed card values too much. A narrower maintainable solution should either extend the client/plugin API or implement Android rendering with explicit ordered segments, template expansion, action semantics, limits, and tests.

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or workflow supply-chain change, but it does introduce native rendering of untrusted plugin/model card payloads with concrete crash and action-data handling concerns.

Concerns:

- **[medium] Malformed cards can crash Android chat rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:196`
  - body: The renderer feeds untrusted card values into recursive Compose rendering, `Modifier.weight`, and `coerceIn` without sufficient limits or validation. A plugin/model-supplied card can therefore crash the Android chat view via deep nesting, invalid column weights, or negative rating bounds.
  - confidence: 0.9
- **[low] Action payload data is written to debug logs:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:545`
  - body: The submit/execute handler logs `action["data"]` on tap. Adaptive Card action data can contain workflow context or user-entered values, so the branch should avoid logging raw action payloads when implementing routing or disabled states.
  - confidence: 0.72
