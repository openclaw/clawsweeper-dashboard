## Summary

Keep open for maintainer review: Android Adaptive Card rendering is not present on current main, but this PR still has blocking correctness, security, product-scope, and missing real-behavior-proof issues before it should merge.

## What This Changes

The PR adds an Android Adaptive Card marker parser, a Jetpack Compose renderer, and chat message wiring to render marked card JSON inline in chat bubbles.

## Best Possible Solution

Land a rebased Android renderer only after native-card scope is approved, action routing or disabled-state semantics are settled, URL and payload guards are implemented, focused Android coverage exists, and real device or emulator proof is provided.

## Reproduction Assessment

Yes for the PR review blockers: source inspection of the PR head shows log-only actions, direct URI opening, reordered fallback rendering, dropped templateData, single-card parsing, and malformed-card crash paths. I did not run Android UI tests in this read-only review.

## Solution Assessment

No. Native Android rendering may be useful with the linked external plugin, but this branch is not the narrowest maintainable merge shape until action policy, URL policy, fallback ordering, template binding, multi-card behavior, and payload hardening are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire card actions before enabling buttons:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:543-545`
  - body: `Action.Submit` and `Action.Execute` render enabled buttons, but tapping them only writes a debug log. Cards that depend on submit or execute actions expose controls that do nothing, so route these actions to a real handler or render them disabled/unsupported.
  - confidence: 0.93
- **[P2] Allow only safe URL schemes before opening links:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:529-532`
  - body: `Action.OpenUrl` sends the card-provided string straight to `LocalUriHandler.openUri`. A model- or plugin-supplied card can trigger non-web Android handlers from chat, so allowlist supported schemes before enabling the button.
  - confidence: 0.88
- **[P2] Preserve fallback text order around cards:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:124-128`
  - body: `fallbackText` removes the card block from the whole message, but the integration renders all fallback text before the card. For intro + card + trailing explanation, the trailing text moves above the card and can change the response meaning.
  - confidence: 0.9
- **[P2] Apply template data before rendering cards:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:128`
  - body: The parser extracts `templateData`, but `ChatMessageBody` passes only `parsed.card` to `AdaptiveCardView`. Messages with `<!--adaptive-card-data-->` markers will render unresolved placeholders unless the data is bound or the unsupported extraction path is removed.
  - confidence: 0.87
- **[P2] Bound recursive card rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:100-105`
  - body: Nested elements recurse through `RenderElement` without a depth limit. A deeply nested model- or plugin-generated card can exhaust the stack and crash chat rendering, so pass a depth counter through nested renderers and bail out at a fixed maximum.
  - confidence: 0.86
- **[P2] Validate column widths before applying weight:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:191-196`
  - body: Column width strings are converted directly into `Modifier.weight(weight)` without checking that the value is finite and positive, and numeric JSON widths are ignored. Malformed widths such as `0`, `-1`, or `NaN` can crash or degrade layout instead of falling back safely.
  - confidence: 0.84
- **[P2] Clamp rating maxima before coerceIn:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:474-480`
  - body: `max` comes directly from card data and is used as the upper bound in `value.toInt().coerceIn(0, max)`. If a malformed card supplies a negative max, Kotlin throws instead of degrading gracefully.
  - confidence: 0.84
- **[P2] Avoid dropping additional card blocks:** `apps/android/app/src/main/java/ai/openclaw/app/chat/AdaptiveCardParser.kt:23-39`
  - body: `parseAdaptiveCardMarkers` uses `find` to capture only the first card block, but fallback text strips all card markers from the message. If a response contains multiple card blocks, later cards are silently removed instead of rendered or preserved as fallback.
  - confidence: 0.82
