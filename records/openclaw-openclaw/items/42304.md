## Summary

Keep open: current main does not implement Android Adaptive Card rendering, and the PR still has blocking correctness and security-sensitive review issues before it can merge.

## What This Changes

The PR adds an Android marker parser and Jetpack Compose Adaptive Card renderer, then wires parsed cards into chat message rendering.

## Best Possible Solution

Make an explicit maintainer call on Android native card support, then land a revised renderer that preserves message order, applies template data or removes it, safely degrades malformed cards, validates URL schemes, defines action semantics, and adds Android coverage.

## Reproduction Assessment

Yes for the review blockers: the PR head source shows enabled Submit/Execute buttons that only log, recursive rendering without a depth guard, dropped templateData, and direct URL opening. I did not run Android UI tests because this was a read-only review.

## Solution Assessment

No. The direction may be useful, especially with the linked external Adaptive Cards plugin context, but this PR is not the narrowest maintainable merge as-is because action behavior, malformed payload handling, and URL policy remain unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire action buttons before enabling them:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:447-450`
  - body: `Action.Submit` and `Action.Execute` render enabled buttons, but the click handler only logs. Cards that depend on submit or execute actions will show tappable controls that do nothing, so either route these actions to a real handler or render them disabled/unsupported until routing exists.
  - confidence: 0.93
- **[P2] Render trailing fallback text in the original order:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:123-127`
  - body: `fallbackText` is built by removing the card block from the whole message, then this branch renders all fallback text before the card. For `intro + card + trailing text`, the trailing text moves above the card and can change the response meaning; the renderer needs to preserve prefix/card/suffix order.
  - confidence: 0.9
- **[P2] Apply template data before rendering cards:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:126-127`
  - body: The parser extracts `templateData`, but this integration passes only `parsed.card` to `AdaptiveCardView`. Messages that include `<!--adaptive-card-data-->` markers will still render unresolved `${...}` placeholders, so bind the data before rendering or remove the unsupported extraction path.
  - confidence: 0.88
- **[P2] Bound recursive card rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:95-108`
  - body: Nested elements recurse through `RenderElement` with no depth limit. A deeply nested model- or plugin-generated card can exhaust the stack and crash chat rendering, so pass a depth counter through nested renderers and bail out at a fixed maximum.
  - confidence: 0.86
- **[P2] Validate URL schemes before opening card links:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:436-439`
  - body: `Action.OpenUrl` sends the card-provided string directly to `LocalUriHandler.openUri`. A crafted card can launch non-web schemes such as `intent:`, `file:`, or `content:` from chat; restrict allowed schemes, preferably to `https`/`http`, before enabling the button.
  - confidence: 0.87
- **[P2] Guard invalid rating maxima:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:405`
  - body: `max` comes directly from card data and is used as the upper bound in `value.toInt().coerceIn(0, max)`. If a malformed card supplies a negative max, Kotlin throws instead of degrading gracefully, so clamp or reject `max` before calling `coerceIn`.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds no new dependencies, but the new untrusted card renderer opens arbitrary URL schemes and can be crashed by malformed/deep payloads.

Concerns:

- **[medium] Unvalidated card URL schemes:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:436`
  - body: Action.OpenUrl opens the card-provided URL without checking the scheme, allowing crafted cards to trigger non-web Android handlers from chat.
  - confidence: 0.87
- **[medium] Malformed cards can crash rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:95`
  - body: Recursive rendering has no depth bound and some numeric fields are trusted directly, so model/plugin-produced card data can crash the Android chat UI instead of falling back.
  - confidence: 0.82
