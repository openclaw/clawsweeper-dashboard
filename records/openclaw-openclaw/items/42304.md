## Summary

Keep this PR open. Current main still does not implement Android Adaptive Card rendering, but the PR is not merge-ready because the proposed Android renderer has blocking correctness and security issues around untrusted card payloads, action semantics, content ordering, template data, multiple cards, and required changelog coverage.

## What This Changes

The PR adds Android parsing for `<!--adaptive-card-->` marker blocks and a Jetpack Compose renderer for Adaptive Card elements inside chat message bubbles.

## Best Possible Solution

Decide the product boundary first: either route this through a documented shared presentation/client extension API, or deliberately adopt native Android Adaptive Card rendering with ordered mixed text/card segments, multiple-card support, template expansion, explicit action policy, render-depth and numeric guards, focused Android tests, and a changelog entry.

## Reproduction Assessment

Yes. On PR head, source-level reproduction is clear with intro + card + suffix text, multiple marker blocks, separate template-data markers, `Column.width` values of `0`, `-1`, `NaN`, or JSON numbers, negative `Rating.max`, deeply nested containers, and Submit/Execute actions; current main also reproduces the missing feature by rendering markers as plain markdown text.

## Solution Assessment

No. The proposed marker parser plus direct Compose renderer is not the best mergeable solution as written because it bypasses the shared presentation direction and lacks action semantics, ordered segmentation, template handling, multi-card support, and defensive rendering guards.

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or workflow supply-chain change, but it introduces native rendering of untrusted card payloads with concrete crash and action-data handling concerns.

Concerns:

- **[medium] Untrusted cards can crash Android chat rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:196`
  - body: The renderer accepts plugin/model-provided card maps and applies unchecked column weights, rating bounds, and recursive rendering. Malformed cards can crash the chat render path, so depth and numeric validation are needed before this can safely ship.
  - confidence: 0.92
- **[low] Submit and execute action data is logged without policy:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:545`
  - body: Enabled Submit/Execute buttons log the action payload instead of routing through an explicit interaction policy. Besides being non-functional, this can expose plugin-provided action data in logs when users tap controls.
