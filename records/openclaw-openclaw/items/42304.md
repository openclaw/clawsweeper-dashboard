## Summary

Keep this PR open, but it is not mergeable as written. Current main does not implement Android Adaptive Card rendering, and the linked external plugin context is useful, but the patch introduces a standalone untrusted renderer with action-routing, ordering, multi-card, template-data, numeric-validation, recursion-depth, and changelog blockers that need maintainer design review before landing.

## What This Changes

The PR adds Android parsing for `<!--adaptive-card-->` marker blocks, a Jetpack Compose Adaptive Card renderer, and chat message integration that renders parsed cards before markdown.

## Best Possible Solution

Decide the rich-message boundary first, then either route Android through the shared presentation/client contract or intentionally adopt native Adaptive Card rendering with ordered mixed text/card segments, multiple-card support, template expansion, explicit action policy, depth and numeric guards, focused Android tests, and a changelog entry.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main renders marker text as markdown, while the PR head can be exercised with intro + card + suffix text, multiple marker blocks, separate template-data markers, invalid column widths, negative `Rating.max`, deeply nested containers, and Submit/Execute actions.

## Solution Assessment

No. The proposed direct marker parser plus Compose renderer is not the best mergeable solution as written because it lacks action routing, ordered segmentation, template handling, multi-card support, defensive rendering guards, tests, and clear alignment with the shared presentation direction.

  - body: `Action.Submit` and `Action.Execute` render enabled buttons, but tapping them only writes a debug log. Cards that depend on submit or execute workflows will look interactive and then do nothing, so wire a real route or render these actions disabled/unsupported.

  - body: Card widths such as `"0"`, `"-1"`, or `"NaN"` are parsed and passed straight to `Modifier.weight(weight)`. Compose requires a finite positive weight, so malformed or model-generated cards can crash chat rendering instead of degrading.
  - confidence: 0.93

  - body: `RenderElement` can recurse through containers, column sets, table cells, and action sets without carrying a depth limit. A deeply nested card payload can exhaust the stack and crash Android chat rendering; add a maximum depth and fallback/skip beyond it.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or workflow supply-chain change, but it introduces native rendering of untrusted card payloads with concrete denial-of-service and action-data handling concerns.

Concerns:

- **[medium] Untrusted cards can crash Android chat rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:100`
  - body: The renderer recursively walks card-provided containers without a depth bound and passes unvalidated numeric fields into Compose/layout helpers. A malicious or malformed card can crash the chat surface instead of falling back to text.
  - confidence: 0.91
- **[low] Action data is logged without a real route:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:543`
  - body: Submit and Execute actions log raw action data on tap while appearing enabled. That is both a misleading action surface and an avoidable local log exposure for card-provided data.
  - confidence: 0.82
