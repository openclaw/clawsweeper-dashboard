## Summary

Keep open: current main still renders Android chat text as Markdown only, while this PR adds a broad new Android Adaptive Card feature with unresolved correctness/security blockers and no real Android behavior proof. The linked external plugin is useful ecosystem context, but it does not make this branch a safe core Android merge as-is.

## What This Changes

The branch adds an Android Adaptive Card marker parser, a Jetpack Compose renderer, and chat message wiring to render marked card JSON inline.

## Best Possible Solution

Land a revised Android renderer only after scope approval, with safe action semantics, URL scheme allowlisting, ordered fallback rendering, template handling, malformed-payload guards, focused Android coverage, and real-device or emulator proof.

## Reproduction Assessment

Yes for the PR review blockers: source inspection of the PR head shows enabled log-only actions, direct URI opening, reordered fallback rendering, dropped templateData, and malformed-card crash paths. I did not run Android UI tests in this read-only review.

## Solution Assessment

No. The direction may be useful with the linked external plugin, but this branch is not the narrowest maintainable solution until action routing, URL policy, fallback ordering, template binding, and payload hardening are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Wire card actions before enabling buttons:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:543-545`
  - body: `Action.Submit` and `Action.Execute` render enabled buttons, but tapping them only writes a debug log. Cards that depend on submit or execute actions will expose interactive controls that do nothing, so route these actions to a real handler or render them disabled/unsupported.
  - confidence: 0.93
- **[P2] Allow only safe URL schemes before opening links:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:529-532`
  - body: `Action.OpenUrl` sends the card-provided string straight to `LocalUriHandler.openUri`. A model- or plugin-supplied card can trigger non-web Android handlers from chat, so allowlist supported schemes before enabling the button.
  - confidence: 0.88
- **[P2] Preserve fallback text order around cards:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:124-128`
  - body: `fallbackText` is built by removing the card block from the whole message, but the integration renders all fallback text before the card. For intro + card + trailing explanation, the trailing text moves above the card and can change the response meaning.
  - confidence: 0.9
- **[P2] Apply template data or remove unsupported markers:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatMessageViews.kt:128`
  - body: The parser extracts `templateData`, but `ChatMessageBody` passes only `parsed.card` to `AdaptiveCardView`. Messages with `<!--adaptive-card-data-->` markers will render unresolved placeholders unless the data is bound or the unsupported extraction path is removed.
  - confidence: 0.87
- **[P2] Bound recursive card rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:100-105`
  - body: Nested elements recurse through `RenderElement` without a depth limit. A deeply nested model- or plugin-generated card can exhaust the stack and crash chat rendering, so pass a depth counter through nested renderers and bail out at a fixed maximum.
  - confidence: 0.86
- **[P2] Validate column widths before applying weight:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:191-196`
  - body: Column width strings are converted directly into `Modifier.weight(weight)` without checking that the value is finite and positive, and numeric JSON widths are ignored. Malformed widths such as `0`, `-1`, or `NaN` can crash or degrade layout instead of falling back safely.
  - confidence: 0.84
- **[P2] Clamp rating maxima before coerceIn:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:474-480`
  - body: `max` comes directly from card data and is used as the upper bound in `value.toInt().coerceIn(0, max)`. If a malformed card supplies a negative max, Kotlin throws instead of degrading gracefully.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Needs attention: the new renderer consumes model/plugin-supplied card payloads and currently opens raw URI strings and can crash on malformed structures.

Concerns:

- **[medium] Unvalidated card URL schemes:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:532`
  - body: `Action.OpenUrl` opens the card-provided URL without checking the scheme, allowing crafted cards to trigger non-web Android handlers from chat.
  - confidence: 0.88
- **[medium] Malformed cards can crash rendering:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/AdaptiveCardComposable.kt:100`
  - body: Recursive rendering has no depth bound and numeric layout/rating fields are trusted directly, so plugin- or model-produced card data can crash Android chat instead of falling back.
  - confidence: 0.84
