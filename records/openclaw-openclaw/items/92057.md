## Summary

Keep this issue open: it provides fresh v2026.6.5 evidence of effective gateway unavailability at a realistic large multi-agent scale, while current main still synchronously loads and merges all targeted session stores and no merged fix or canonical issue clearly covers this exact large-store recurrence. Prior performance work narrows several known costs, but it does not establish that the reported RPC timeouts and event-loop stalls are resolved.

## What This Changes

Investigate and prevent gateway RPC, message-delivery, and agent-execution stalls under large multi-agent session, transcript, and log stores.

## Best Possible Solution

Use the existing `gateway.sessions.list.*` timeline spans plus a sanitized stability bundle and CPU profile during a stall, add a representative large multi-agent store regression benchmark, and then move the measured blocking metadata operation to the canonical SQLite/indexed or asynchronous owner path without adding new fallback/config surfaces.

No, not yet at high confidence: the report contains repeated, concrete v2026.6.5 field measurements, and current source exposes measurable candidate phases, but this review did not reproduce the 4,820-file workload or identify one failing handler conclusively.

## Solution Assessment

Yes, a profiling-led canonical hot-path repair is the best direction; implementing all suggested pagination, cleanup, rotation, and maintenance features before isolating the dominant phase would be broader than the evidence supports.
