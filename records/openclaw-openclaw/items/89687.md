## Summary

Keep open: current main still queues Feishu inbound work at the whole-chat level before topic routing, and this PR is a plausible fix, but it lacks required real Feishu/OpenClaw proof and needs an explicit maintainer decision on topic-scoped `/stop` and `/btw` command lanes.

## What This Changes

The PR makes Feishu inbound sequential queue keys asynchronous and derives group queue scope from configured group-topic session state, including topic `thread_id` hydration before enqueue.

## Best Possible Solution

Land one canonical Feishu queue-scoping fix that reuses the same group-session peer invariant as routing, explicitly decides command-lane scope, and includes real Feishu topic-group proof with private details redacted.

## Reproduction Assessment

Yes from source: current main queues Feishu inbound messages by account and chat before routing splits `group_topic` and `group_topic_sender` sessions, so unrelated topics in one group can share a queue. I did not run live Feishu proof.

## Solution Assessment

Unclear: deriving the queue key from the same group-session peer invariant is the right shape, but this PR also changes `/stop` and `/btw` lane scope and should make that product decision explicit before merge.
