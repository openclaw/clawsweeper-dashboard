## Summary

Keep open: the OAuth bug is real and the supplied behavior proof is sufficient, but the patch is not mergeable as written because the runner change exposes a broad auth profile store to plugin-owned harnesses and conflicts with the scoped projection now on current main.

## What This Changes

The PR scopes Codex app-server OAuth lookup for plugin-owned harnesses, forwards harness auth context, and makes the live Codex command probe opt-in.

## Best Possible Solution

Apply the Codex app-server scoped external-CLI lookup on top of current main while preserving the scoped plugin harness auth-store projection from https://github.com/openclaw/openclaw/pull/80017 and adding coverage for runtime-only Codex CLI OAuth without exposing unrelated profiles.

## Reproduction Assessment

Yes. Source inspection shows current main can miss runtime-only scoped Codex CLI OAuth when the app-server resolves from a supplied or unscoped store, and the contributor supplied before/after live harness proof for the failing and fixed OAuth path.

## Solution Assessment

No, not as written. The app-server scoped lookup is the right direction, but the runner change should not pass a full merged auth store to plugin-owned harnesses and should instead keep the current-main projected-store boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
