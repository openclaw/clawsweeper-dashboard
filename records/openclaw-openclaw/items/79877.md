## Summary

Keep open: the OAuth routing problem is valid, but this patch widens plugin-owned harness access to persisted auth profiles and should not merge as-is.

## What This Changes

The PR scopes Codex app-server OAuth lookup to Codex external CLI profiles, forwards harness auth context, and makes live Codex harness command probes opt-in.

## Best Possible Solution

Keep the scoped Codex app-server external-CLI lookup, but pass plugin-owned harnesses a minimal projected authProfileStore containing only the selected Codex harness profile and cover that isolation in tests.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main passes an empty store to plugin-owned harness attempts while Codex app-server startup resolves from a supplied store first, so scoped OAuth profiles can be missed.

## Solution Assessment

No, not as written. The scoped Codex app-server lookup is the right direction, but the embedded runner should project only the intended harness auth profile instead of handing a merged persisted store to plugin-owned harness code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Project plugin harness auth stores:** `src/agents/pi-embedded-runner/run.ts:562-566`
  - body: For plugin-owned harnesses this builds `authStore` with `ensureAuthProfileStore(...)`, which starts from merged main/agent persisted profiles before adding scoped external CLI entries. That same store is passed to `runAttempt`, so a plugin harness can inspect unrelated credentials; pass a projected store containing only the selected or forwarded harness profile across this boundary.
  - confidence: 0.88
