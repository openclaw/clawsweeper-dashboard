## Summary

This PR is a plausible fix for a real Codex app-server OAuth harness bug, but it should not merge as written because it broadens credential visibility across the plugin harness boundary.

## What This Changes

The branch scopes Codex app-server OAuth profile lookup to Codex external CLI providers, forwards harness auth context from the embedded runner, and makes live Codex harness command probes opt-in.

## Best Possible Solution

Keep the scoped Codex app-server external-CLI lookup, but pass plugin-owned harnesses a minimal projected authProfileStore containing only the selected Codex harness profile and cover that isolation.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main passes an empty store to plugin-owned harness attempts while Codex app-server startup resolves from a supplied store first, so scoped OAuth profiles can be missed.

## Solution Assessment

No, not as written. The scoped Codex app-server lookup is the right direction, but the embedded runner should project the selected harness profile instead of handing a full merged auth store to plugin-owned code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Project plugin harness auth stores:** `src/agents/pi-embedded-runner/run.ts:562-566`
  - body: For plugin-owned harnesses this builds `authStore` with `ensureAuthProfileStore(...)`, which starts from the merged main/agent persisted auth store before adding scoped external CLI entries. The same store is passed to `runAttempt`, so plugin-owned harness code can inspect credentials unrelated to the forwarded Codex profile; build a minimal store containing only the forwarded or explicit harness profile before crossing this boundary.
  - confidence: 0.88
