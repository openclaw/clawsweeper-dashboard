## Summary

Keep open; the PR addresses a real Codex OAuth harness auth gap, but the current branch introduces a security-sensitive credential-scope regression by passing a merged auth store to plugin-owned harness code.

## What This Changes

The PR changes Codex app-server OAuth profile resolution for plugin-owned harnesses, forwards harness auth context through the embedded runner, and makes the live Codex command probe opt-in.

## Best Possible Solution

Keep the scoped Codex app-server OAuth lookup, but project the plugin-owned harness authProfileStore down to only the selected forwarded or explicit Codex harness profile before passing it across the plugin boundary.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main supplies an empty store to plugin-owned harness attempts while Codex app-server startup resolves auth from a supplied store first, so scoped OAuth profiles can be missed. I did not live-run the harness in this read-only review.

## Solution Assessment

No, not as written. The scoped Codex lookup is the narrow maintainable direction, but the embedded runner should project the selected harness profile instead of passing a full merged persisted store to plugin-owned harnesses.
