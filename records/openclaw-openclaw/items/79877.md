## Summary

Keep open. The PR addresses a credible Codex OAuth routing gap, but it is still draft, lacks attached real behavior proof, and introduces a security-sensitive auth-store scoping regression.

## What This Changes

The PR scopes Codex app-server OAuth lookup, forwards plugin harness auth context from the embedded runner, and makes live Codex harness command probes opt-in with broader accepted status text.

## Best Possible Solution

Keep the scoped Codex OAuth bridge direction, but pass only the intended Codex auth profile/provider to plugin-owned harnesses and attach redacted live OAuth harness output before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence reproduction is available in this review record; source inspection on current main shows the app-server path uses unscoped auth-store loading while plugin harnesses receive an empty store, matching the reported gap.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The auth-bridge scoping is narrow, but the embedded-runner store change should be constrained to the intended Codex auth profile/provider instead of passing a merged store to plugin harnesses.
