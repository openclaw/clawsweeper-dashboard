## Summary

Keep this PR open: it targets a real Codex app-server OAuth harness gap and supplies convincing real behavior proof, but the current branch still has a credential-scope regression and is now conflicting with current main's scoped-store follow-up.

## What This Changes

The PR scopes Codex app-server OAuth lookup to Codex external CLI auth, forwards harness auth context from the embedded runner, and makes live Codex harness command probes opt-in.

## Best Possible Solution

Preserve current main's one-profile plugin harness store boundary while adding the Codex app-server external-CLI scoped lookup needed for runtime-only Codex OAuth profiles, with regression coverage for both routing and credential isolation.

## Reproduction Assessment

Yes at source level: current main's external CLI discovery only bootstraps missing Codex CLI auth when provider/profile scope is supplied, while the app-server auth bridge still uses an unscoped store. The contributor also supplied before/after recordings and live output, but I did not run the live harness here.

## Solution Assessment

No, not as written. The Codex app-server scoped lookup is the right direction, but the embedded runner must project only the selected harness profile instead of passing a full merged auth store across the plugin-owned harness boundary.
