## Summary

Keep open: the PR addresses a real Codex app-server OAuth harness bug and includes adequate live-output proof, but the runner change widens plugin-owned harness access to persisted auth profiles and needs a scoped-store repair before merge.

## What This Changes

The branch scopes Codex app-server OAuth profile lookup, forwards harness auth context from the embedded runner, and makes the live Codex harness command probe opt-in.

## Best Possible Solution

Keep the scoped Codex app-server external-CLI lookup, but pass plugin-owned harnesses only a projected store containing the selected or forwarded Codex profile and cover that isolation contract.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main passes an empty store to plugin-owned harness attempts while the Codex app-server startup path resolves from a supplied store first, so scoped OAuth profiles can be missed. I did not live-run the harness in this read-only review.

## Solution Assessment

No, not as written. The Codex app-server scoped lookup is the right direction, but the embedded runner should project the selected harness auth profile instead of passing a full merged persisted auth store to plugin-owned harness code.
