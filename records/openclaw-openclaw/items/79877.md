## Summary

Keep open: the Codex app-server OAuth direction is useful and the real behavior proof is sufficient, but the branch still broadens plugin-owned harness access to the merged auth profile store and should not merge as written.

## What This Changes

The PR scopes Codex app-server OAuth-store loading, forwards plugin-owned harness auth profile context, and makes Codex live harness command probes opt-in.

## Best Possible Solution

Keep the scoped Codex app-server external-CLI lookup, but pass plugin-owned harnesses a minimal authProfileStore containing only the forwarded or explicit Codex harness profile, with regression coverage for unrelated profile isolation.

## Reproduction Assessment

Source-reproducible, but not live-reproduced in this read-only review. Current main passes an empty store to plugin-owned harness attempts while Codex app-server startup resolves from a supplied store first, so scoped OAuth profiles can be missed unless the harness profile is forwarded safely.

## Solution Assessment

No, not as written. The Codex app-server scoped lookup is the narrow direction, but the embedded runner should project the selected harness profile instead of passing the full merged auth store to plugin-owned harnesses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Project plugin harness auth stores:** `src/agents/pi-embedded-runner/run.ts:570-575`

Build a minimal store containing only the forwarded or explicit harness profile before passing it across this boundary.
