## Summary

Keep open: the Codex OAuth routing problem is credible and the PR now includes real after-fix proof, but the embedded-runner change still broadens credential visibility to plugin-owned harnesses instead of projecting only the intended Codex profile.

## What This Changes

The PR scopes Codex app-server OAuth-store loading, forwards harness auth profile context to plugin-owned Codex runs, and makes live harness command probes opt-in.

## Best Possible Solution

Keep the Codex app-server scoped external-CLI lookup, but pass a minimal auth store containing only the resolved or forwarded Codex harness profile to plugin-owned runAttempt calls.

## Reproduction Assessment

Source-reproducible, but not live-reproduced in this review. Current main passes an empty store into Codex run-attempt while the PR body supplies after-fix local and Docker OAuth harness output.

## Solution Assessment

No, not as written. The Codex app-server auth-store scoping is the right direction, but the embedded-runner should project the selected harness profile instead of passing the full merged auth store to plugin-owned harnesses.
