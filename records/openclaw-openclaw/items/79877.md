## Summary

Keep open: the branch addresses a real Codex app-server OAuth harness bug, but the current patch broadens plugin-owned harness visibility to the merged persisted auth store, which is a concrete security regression before merge.

## What This Changes

The PR scopes Codex app-server OAuth lookup to scoped external CLI profiles, forwards harness auth context to plugin-owned Codex transports, and makes live harness command probes opt-in.

## Best Possible Solution

Keep the scoped Codex app-server external-CLI lookup, but pass plugin-owned harnesses a projected authProfileStore containing only the selected Codex harness profile and add isolation coverage.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main passes an empty store to plugin-owned harness attempts while Codex app-server startup resolves from a supplied store first, so scoped OAuth profiles can be missed; I did not live-run the harness in this read-only review.

## Solution Assessment

No, not as written. The Codex app-server scoped lookup is the narrow maintainable direction, but the embedded runner should project the selected harness profile instead of passing a full merged auth store to plugin-owned harnesses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Project plugin harness auth stores:** `src/agents/pi-embedded-runner/run.ts:562-566`

The same store is then passed to runAttempt, so plugin harness code can inspect credentials unrelated to the forwarded Codex profile; build a minimal store containing only the selected harness profile before passing it across this boundary.
