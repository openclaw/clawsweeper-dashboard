## Best Possible Solution

Keep the controller-level dedupe, but separate duplicate renderable payload handling from no-payload fallback, add stale-stream regression coverage, and require redacted real WebChat proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live browser reproduction was run for the intermittent linked issue, but source inspection gives a high-confidence branch-local failure path when a duplicate renderable payload falls into stream fallback with different stale `chatStream`.

## Solution Assessment

Is this the best way to solve the issue? No; the direction is narrow, but the implementation should treat duplicate renderable payloads as handled instead of falling through to missing-payload stream fallback.
