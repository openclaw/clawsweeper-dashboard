## Summary

Keep open: the patch is aimed at the right WebChat race, but duplicate-suppressed final and aborted payloads can still fall through to stream fallback, and the PR still has only mock/unit proof rather than real WebChat behavior proof.

## What This Changes

The PR adds tail-duplicate checks to WebChat `handleChatEvent` assistant-message append paths and controller tests for final, aborted, other-run, and stream-fallback cases.

## Best Possible Solution

Keep the controller-level fix direction, but make duplicate payload suppression terminal for that event, add regression tests for duplicate payload plus differing stream fallback, refresh against current main, and include redacted real WebChat proof.

## Reproduction Assessment

Yes for the patch defect from source inspection: a duplicate final or aborted payload makes the guarded append condition false, then the stream fallback can still append if `chatStream` has a distinct signature. The original reported race is source-backed by the linked issue and controller flow, but I did not live-reproduce it here.

## Solution Assessment

No. The tail-signature guard is a plausible repair direction, but the fallback should be gated on absence of a usable payload rather than on the payload append condition that also includes duplicate suppression.
