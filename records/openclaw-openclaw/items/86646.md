## Best Possible Solution

Keep the controller-level dedupe, but separate duplicate renderable payload handling from absent/nonrenderable payload fallback, add a stale-stream regression test, and require redacted real WebChat proof before merge.

## Reproduction Assessment

No live high-confidence reproduction was run; the linked issue reports an intermittent browser path and current source shows a plausible `chat.history` reload plus final-event append race. The PR regression is source-reproducible from the new branch logic because duplicate payloads become fallback-eligible.

## Solution Assessment

No; the proposed direction is narrow, but this implementation should distinguish duplicate renderable payloads from missing/nonrenderable payloads before using stream fallback. The safer solution is a small control-flow fix plus the missing real WebChat proof.
