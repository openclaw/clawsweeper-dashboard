## Summary

Keep open: the fix direction is useful, but the current branch can still append a stale stream fallback when a duplicate final payload is skipped, and the supplied proof is terminal/source-level rather than a real WebChat or gateway run.

## What This Changes

The PR adds a WebChat controller tail-duplicate guard for same-run final, aborted, and stream-fallback assistant messages, plus focused controller tests.

## Best Possible Solution

Separate duplicate visible final/aborted payloads into explicit no-op terminal handling before stream fallback, add regression coverage for stale stream state, and require redacted WebChat/gateway proof before merge.

## Reproduction Assessment

Yes for the patch defect at source level: a state with duplicate tail final text and different nonempty `chatStream` reaches the stream fallback after the new duplicate guard. No live WebChat/gateway reproduction of the intermittent original race was supplied.

## Solution Assessment

No. The narrow idea is reasonable, but the implementation should treat a duplicate visible terminal payload as handled instead of falling through to stream fallback, and it still needs real WebChat/gateway proof.
