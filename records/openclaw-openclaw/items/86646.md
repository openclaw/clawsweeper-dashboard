## Summary

Keep open: the branch fixes the earlier user-tail and stale-stream problems, but the current guard can still suppress a legitimate same-content terminal reply when WebChat adopts a selected-session final/aborted event that was not already an active local run. ## Best Possible Solution

Limit terminal dedupe to the active-run history race, preserve observed terminal messages for untracked runs, and add real WebChat/Gateway proof after the repair.

## Reproduction Assessment

Yes, from source inspection: construct a selected-session final or aborted event with no active run and a previous assistant tail with the same content; the PR's adopted-run dedupe suppresses the new terminal message. The original duplicate report is also source-plausible from loadChatHistory updating the tail before the final event appends.

## Solution Assessment

No. The controller-local guard is a plausible direction, but the best fix must scope dedupe to the active-run history race instead of every same-content assistant tail after run-id adoption.
