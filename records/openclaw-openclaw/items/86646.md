## Summary

Keep open: the PR targets a real WebChat final-event/history race, but the proposed guard is not merge-safe because it compares only message content and can drop legitimate assistant replies that echo the preceding user message. The proof is useful terminal-level signal, but it does not show the visible WebChat race fixed in a real UI/gateway run.

## What This Changes

The PR adds a tail-duplicate guard in WebChat `handleChatEvent` for same-run final/aborted terminal assistant messages and adds controller tests for duplicate terminal payloads and stale stream fallback.

## Best Possible Solution

Keep the focused WebChat fix direction, but make terminal duplicate detection assistant-tail and role-aware, preserve legitimate user echoes and cross-run finals, add regression coverage for those cases, and then provide redacted live WebChat proof.

## Reproduction Assessment

Yes for the introduced patch defect: with `chatMessages` ending in a user message and a same-content assistant final payload, `isTailDuplicate` compares only `content` and the append path is skipped. The original intermittent WebChat race was not live-reproduced here.

## Solution Assessment

No. The controller is the plausible repair boundary, but the helper must be role-aware and current-main reconciliation-aware before this is the best fix.
