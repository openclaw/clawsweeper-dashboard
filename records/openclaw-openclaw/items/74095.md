## Summary

Keep this open: current main still exposes task and TaskFlow status, delivery status, terminal outcome, summaries, and operational audit codes, but not a shared machine-readable blocker/failure taxonomy. The related delivery-reason request #64413 remains open and narrower, so it does not supersede this broader task taxonomy work.

## What This Changes

Request a centralized blocker/failure taxonomy for task states so blocked, failed, waiting, and transient conditions have reusable machine-readable classes.

## Best Possible Solution

Define an additive shared taxonomy contract with explicit unknown handling, then consistently map TaskRecord, TaskFlow, delivery, audit, and plugin-visible DTO outcomes into it.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has status/prose/error fields and audit codes but no normalized blocker/failure taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable solution. The exact names, persistence model, DTO exposure, and public compatibility rules should be settled before implementation.
