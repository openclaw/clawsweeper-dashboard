## Summary

Keep this open: current main still exposes task status, delivery status, terminal outcome, blocked summaries, wait JSON, and operational audit codes, but not a shared machine-readable blocker/failure class. Related #64413 is still open and narrower, covering delivery failure reasons rather than superseding the broader task taxonomy request.

## What This Changes

Add a shared machine-readable blocker/failure taxonomy for background tasks and reuse it across audits, summaries, status output, and future retry decisions.

## Best Possible Solution

Define an additive shared classification contract with explicit unknown handling, map TaskRecord, Task Flow, delivery, audit, and agent-run outcomes into it, and expose it compatibly through persisted records, runtime DTOs, CLI JSON/status output, and docs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has status/prose fields and operational audit codes but no shared blocker/failure taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable path. The exact names, storage shape, and public compatibility rules should be chosen before implementation.
