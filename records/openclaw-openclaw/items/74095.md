## Summary

Keep open: current main still models task and TaskFlow conditions with coarse statuses, prose summaries, delivery status, and operational audit codes rather than a shared machine-readable blocker/failure taxonomy. The related delivery issue remains open and narrower, so it does not supersede this task-wide request: https://github.com/openclaw/openclaw/issues/64413.

## What This Changes

Request to add shared task blocker/failure classes and reuse them across task records, audits, summaries, status output, and future retry decisions.

## Best Possible Solution

Define an additive shared taxonomy contract with explicit unknown handling, then map TaskRecord, TaskFlow, delivery, audit, plugin DTO, CLI/JSON, and status-rendering outcomes into it consistently.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence current-main verification path: the relevant task records, TaskFlow records, audits, plugin DTOs, and docs still expose coarse status/prose fields without the requested shared taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable direction. The exact classes, storage/API shape, compatibility rules, and relationship to the narrower delivery issue need maintainer agreement before implementation.
