## Summary

Keep this open. Current main still has separate task status, delivery status, terminal outcome, Task Flow blocked/waiting state, audit codes, and prose error/summary fields, but no shared machine-readable blocker/failure taxonomy; related #64413 remains a narrower delivery-failure issue rather than a superseding tracker for this broader task-system contract.

## What This Changes

The issue requests a shared machine-readable blocker/failure taxonomy for task states and failures so audits, summaries, status output, and future retry logic can branch on normalized classes.

## Best Possible Solution

Define an additive shared task blocker/failure classification contract with explicit unknown handling, map task, Task Flow, delivery, audit, and agent-run outcomes into it, then expose it through compatible persisted records, runtime DTOs, CLI JSON/status output, and docs while coordinating the delivery-specific subset with #64413.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has status/prose/audit surfaces but no shared blocker/failure taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable path. The exact public contract, field names, persistence migration, DTO exposure, and rollout boundaries should be chosen by maintainers before implementation.
