## Summary

Keep this open: current main still exposes task/flow status, delivery status, audit codes, summaries, and errors, but not a shared machine-readable blocker/failure taxonomy. Related #64413 remains open and narrower, covering delivery failure reasons rather than superseding the broader task taxonomy request.

## What This Changes

Add shared task blocker/failure classification types and helpers, then reuse them across task records, audits, summaries, and user-facing status output.

## Best Possible Solution

Define an additive shared task blocker/failure classification contract with explicit unknown handling, map TaskRecord, Task Flow, delivery, audit, and agent-run outcomes into it, and expose it compatibly through persisted records, runtime DTOs, CLI JSON/status output, and docs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has status/prose fields and operational audit codes but no shared blocker/failure taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable path. The exact names, persistence model, and public compatibility rules should be chosen by maintainers before implementation.
