## Summary

Keep open: current main still exposes task and flow statuses, delivery states, summaries, and audit codes, but not the requested shared machine-readable blocker/failure taxonomy. The related delivery-reason issue is narrower and still open, so it does not supersede this broader task-system request.

## What This Changes

Request a shared blocker/failure taxonomy for task records, task flows, audits, summaries, status output, and future retry logic.

## Best Possible Solution

Define an additive, versioned taxonomy contract with explicit unknown handling, then map TaskRecord, TaskFlow, delivery, audit, CLI/status, and plugin runtime DTO outcomes to it consistently.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence evidence that current main has coarse status/prose/audit fields but no shared blocker/failure taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable path. The exact class names, persistence shape, plugin DTO exposure, and relationship to #64413 need maintainer decision before implementation.
