## Summary

Keep this open: current main still exposes coarse task/TaskFlow statuses, delivery status, summaries, errors, and audit codes, but not a shared machine-readable blocker/failure taxonomy. The related #64413 is open and narrower around delivery failure reasons, so it does not supersede this broader task taxonomy request.

## What This Changes

Request a shared blocker/failure taxonomy for tasks and TaskFlow state so audits, summaries, status output, and future retry logic can consume normalized reason classes.

## Best Possible Solution

Define an additive, versioned taxonomy contract with explicit unknown handling, then map TaskRecord, TaskFlow, delivery, audit, CLI JSON/status output, plugin DTOs, and docs onto that contract while coordinating the delivery subset with #64413.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has status/prose/error fields and operational audit codes but no shared blocker/failure taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable solution. The exact names, persistence model, public DTO shape, and compatibility rules should be settled before implementation.
