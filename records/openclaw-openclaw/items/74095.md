## Summary

Keep this open: current main still has task/TaskFlow statuses, delivery status, summaries, errors, and operational audit codes, but no shared machine-readable blocker/failure taxonomy. The related delivery-focused #64413 remains open and narrower, so it does not supersede this broader task taxonomy request.

## What This Changes

Request a shared blocker/failure taxonomy for task records, TaskFlow state, audits, summaries, status output, and future retry decisions.

## Best Possible Solution

Define an additive shared taxonomy contract with explicit unknown handling, then map TaskRecord, TaskFlow, delivery, audit, and agent-run outcomes into it across persisted records, runtime DTOs, CLI JSON/status output, and docs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has status/prose/error fields and operational audit codes but no shared blocker/failure taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable solution. The exact names, persistence model, and public compatibility rules should be settled before implementation.
