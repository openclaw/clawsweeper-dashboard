## Summary

Keep open: current main still has coarse task/TaskFlow statuses, prose error/summary fields, operational audit codes, and a CLI delivery reason that is explicitly free-form rather than the requested shared blocker/failure taxonomy. The related delivery tracker is open and narrower, so it does not supersede this task-wide feature request.

## What This Changes

Add a shared machine-readable blocker/failure taxonomy for tasks and reuse it across task records, TaskFlow, audits, summaries, and user-facing status output.

## Best Possible Solution

Define an additive shared taxonomy contract with explicit unknown handling, then map TaskRecord, TaskFlow, delivery, audit, plugin DTO, CLI/JSON, status rendering, and docs into it consistently.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence current-main verification path: task records, TaskFlow records, audits, plugin DTOs, and CLI docs still expose coarse status/prose/free-form fields without the requested taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable direction. The exact classes, storage/API shape, compatibility rules, and relationship to the narrower delivery issue need maintainer agreement before implementation.
