## Summary

Keep this open. Current main still exposes coarse task status, delivery status, terminal outcome, operational audit codes, and prose error/summary fields, but it does not have the requested shared machine-readable blocker/failure taxonomy; related #64413 is open and narrower delivery-failure work, not a superseding tracker.

## What This Changes

The issue requests a shared task blocker/failure taxonomy so blocked, waiting, failed, and transient failure states can be classified consistently across task records, audits, summaries, status output, and future retry logic.

## Best Possible Solution

Define an additive shared task blocker/failure classification contract with explicit unknown handling, map task lifecycle, Task Flow, delivery, audit, and agent-run outcomes into it, and expose it consistently through persisted records, plugin/runtime DTOs, CLI JSON/status output, and docs while coordinating delivery-specific classes with #64413.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has status/prose fields and operational audit codes but no shared causal taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable solution. The exact public contract and rollout boundaries should be chosen by maintainers before implementation so consumers do not get another ad hoc field.
