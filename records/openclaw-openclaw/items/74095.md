## Summary

Keep open: current main still exposes coarse task and TaskFlow status fields, prose summaries/errors, operational audit codes, and a free-form CLI delivery reason rather than the requested shared blocker/failure taxonomy. The related delivery-reason issue and cron warning PR are narrower and do not supersede this task-wide API/design request.

## What This Changes

The issue requests a shared machine-readable taxonomy for task blockers, failures, waiting states, and recoverable transient failures.

## Best Possible Solution

Define an additive shared taxonomy contract with explicit unknown handling, then map TaskRecord, TaskFlow, delivery, audit, plugin DTO, CLI/JSON, status rendering, and docs into it consistently.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence current-main verification path: task records, TaskFlow records, audits, plugin DTOs, and CLI/docs still expose coarse status, prose, or free-form fields without the requested taxonomy.

## Solution Assessment

Yes at the design-direction level: a centralized additive taxonomy is the maintainable direction. The exact classes, storage/API shape, compatibility rules, and relationship to the narrower delivery issue need maintainer agreement before implementation.
