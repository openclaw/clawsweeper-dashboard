## Summary

Keep open and do not merge as-is: the patch removes the streaming usage request for the custom/local providers implicated by the linked llama.cpp zero-usage bug, while current stream handling still only records usage from chunks yielded by the OpenAI SDK. Existing tests and docs expect local OpenAI-compatible backends such as llama.cpp to preserve usage visibility, and the supplied proof only verifies a simulated parameter object.

## What This Changes

The PR gates `stream_options.include_usage` in `buildOpenAICompletionsParams` on `compat.supportsUsageInStreaming` for OpenAI Chat Completions streaming requests.

## Best Possible Solution

Fix the OpenAI-completions streaming path to preserve llama.cpp usage by consuming trailing usage/timings before finalization or adding a robust timings fallback, with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main depends on SDK-yielded chunks, OpenAI Node v6.37.0 skips SSE data after `[DONE]`, and the linked issue supplies llama.cpp logs plus direct curl evidence; I did not run a live llama.cpp server.

## Solution Assessment

Is this the best way to solve the issue? No: the patch suppresses the usage request for affected false-compat/custom providers instead of preserving or recovering the usage data needed by `/status` and compaction.
