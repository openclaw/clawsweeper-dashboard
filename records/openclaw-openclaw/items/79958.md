## Summary

Keep open: the PR suppresses the `stream_options.include_usage` request for the reported custom/local OpenAI-compatible path, which can leave the llama.cpp zero-usage bug unresolved and contradicts current main's documented/tested behavior.

## What This Changes

This PR changes OpenAI Chat Completions streaming request construction to send `stream_options.include_usage` only when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Keep requesting streamed usage and fix the stream-consumption or fallback path so trailing llama.cpp usage metadata reaches normalized transcript and session usage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main only records SDK-yielded usage chunks, the OpenAI SDK skips SSE after `[DONE]`, and the linked issue includes llama.cpp log/curl evidence for a trailing usage chunk.

## Solution Assessment

Is this the best way to solve the issue? No: omitting `stream_options.include_usage` can stop affected custom/local backends from emitting the usage metadata needed to fix zero saved usage.
