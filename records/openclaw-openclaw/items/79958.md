## Summary

Keep open: the patch suppresses the request field that current code, tests, docs, and feature history rely on for local OpenAI-compatible usage reporting, so it can leave the linked llama.cpp zero-usage bug unresolved; the supplied proof also does not exercise the real OpenClaw path.

## What This Changes

The PR gates Chat Completions streaming `stream_options.include_usage` on `compat.supportsUsageInStreaming` in `src/agents/openai-transport-stream.ts`.

## Best Possible Solution

Keep requesting streaming usage for local/custom OpenAI-compatible backends, then preserve the final usage chunk or llama.cpp timing metadata with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main records only SDK-yielded usage chunks, OpenAI Node 6.37.0 skips post-`[DONE]` SSE messages, and the linked issue supplies llama.cpp logs plus direct curl evidence.

## Solution Assessment

Is this the best way to solve the issue? No: gating off `stream_options` removes the request that makes the affected backend emit usage or timing data, so zero usage can remain instead of being fixed.
