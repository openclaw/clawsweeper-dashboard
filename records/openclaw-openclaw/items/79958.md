## Summary

Keep open: the PR targets a real source-plausible llama.cpp usage bug, but this diff withholds `stream_options.include_usage` from the same custom/local class that needs the request in order to emit usage metadata.

## What This Changes

The PR changes OpenAI Chat Completions streaming request construction to include `stream_options.include_usage` only when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Keep requesting streamed usage and fix stream consumption or a fallback path so trailing llama.cpp usage or timings data reaches saved usage, with regression coverage and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main records only SDK-yielded usage chunks, OpenAI Node skips SSE messages after `[DONE]`, and the linked issue includes llama.cpp log and curl evidence for a trailing usage chunk.

## Solution Assessment

Is this the best way to solve the issue? No: omitting `stream_options.include_usage` can stop affected custom/local backends from emitting the usage metadata needed to fix zero saved usage.
