## Summary

Keep open: the PR has a blocking correctness issue because it disables `stream_options` for the same class of custom/llama.cpp endpoints that need it to emit usage, so the linked zero-usage bug would remain; the supplied proof is only synthetic.

## What This Changes

The PR changes `buildOpenAICompletionsParams` so OpenAI Chat Completions streaming requests include `stream_options: { include_usage: true }` only when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Preserve usage reporting for llama.cpp/custom OpenAI-compatible streams by capturing the final usage or llama.cpp timings data, then add focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main depends on SDK-yielded chunks while OpenAI Node 6.37.0 skips SSE data after `[DONE]`, and the linked issue supplies llama.cpp logs plus direct curl evidence.

## Solution Assessment

Is this the best way to solve the issue? No: gating `stream_options` removes the usage request for the affected custom backend instead of preserving usage, so the linked zero-usage behavior can remain or get worse.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
