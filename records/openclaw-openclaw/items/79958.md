## Summary

Keep open: the patch is not safe to merge because it removes the usage request needed by custom/local OpenAI-compatible providers and reverts the shipped behavior from https://github.com/openclaw/openclaw/pull/68746 without fixing the post-[DONE] parsing path from the linked llama.cpp report.

## What This Changes

The PR changes `buildOpenAICompletionsParams` to include `stream_options.include_usage` only when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Keep requesting streaming usage for OpenAI-compatible streams, then fix the llama.cpp post-[DONE] case by consuming the provider’s trailing usage safely or adding a targeted fallback that produces nonzero saved usage without regressing https://github.com/openclaw/openclaw/pull/68746.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: the linked llama.cpp issue reports usage after `[DONE]`, current main uses the OpenAI SDK parser that skips post-[DONE] SSE messages, and OpenClaw only updates saved usage from yielded chunks. I did not live-run llama.cpp here.

## Solution Assessment

No. Gating `stream_options` removes the request required to receive streaming usage for the affected custom/local providers, so it does not solve saved usage staying at zero; the narrower maintainable fix is in stream parsing or a targeted usage fallback while preserving the request.
