## Summary

Keep open: the PR targets a real usage-accounting bug, but the patch suppresses the usage request for the affected false-compat backends instead of preserving or recovering the missing llama.cpp usage data.

## What This Changes

The PR changes `buildOpenAICompletionsParams` so `stream_options: { include_usage: true }` is only sent when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Keep requesting usage where local backends can provide it, then fix OpenAI-compatible stream handling or fallback normalization so llama.cpp token counts are captured with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current `main` requests streaming usage and the OpenAI SDK drops SSE messages after `[DONE]`, while the linked issue provides llama.cpp logs and saved `0/0/0` usage but no runnable live server was available here.

## Solution Assessment

Is this the best way to solve the issue? No: the patch removes the usage request from the affected false-compat path instead of making OpenClaw consume or recover the usage data needed by `/status` and compaction.
