## Summary

Keep open: the latest one-line change still does not fix the linked zero-usage bug because affected llama.cpp/custom OpenAI-compatible endpoints need the usage request to emit trailing usage data, while the real failure is that the SDK-backed consumer cannot see usage sent after `[DONE]`. The supplied proof is synthetic payload construction rather than an after-fix OpenClaw run against the reported backend.

## What This Changes

This PR changes OpenAI Chat Completions streaming request construction to send `stream_options.include_usage` only when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Preserve the usage request for affected OpenAI-compatible streams and fix the stream consumption or fallback path so llama.cpp timing/usage data reaches normalized transcript usage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main records usage only from SDK-yielded chunks, the linked report supplies llama.cpp log/curl evidence, and the OpenAI SDK source skips SSE data after `[DONE]`.

## Solution Assessment

Is this the best way to solve the issue? No: gating off `stream_options.include_usage` removes the request that makes affected OpenAI-compatible backends emit usage metadata, so saved usage can still remain zero.
