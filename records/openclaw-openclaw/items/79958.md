## Summary

Keep open: the PR targets a real OpenAI-compatible streaming usage bug, but the proposed one-line gate would remove the request field that the affected local/custom backends need in order to emit usage metadata, so it does not fix the linked problem.

## What This Changes

This PR changes OpenAI Chat Completions streaming request construction to send `stream_options.include_usage` only when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Keep requesting streaming usage for OpenAI-compatible local/custom backends, then fix the consumer to preserve or recover the trailing llama.cpp usage/timings with focused regression coverage and real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main records only SDK-yielded usage chunks, the linked issue supplies llama.cpp logs and curl evidence, and the dependency iterator skips post-`[DONE]` SSE data.

## Solution Assessment

Is this the best way to solve the issue? No: gating off `stream_options.include_usage` removes the request that makes affected local/custom backends emit usage metadata, so saved usage can still remain zero.
