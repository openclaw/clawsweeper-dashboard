## Summary

Keep this PR open, but do not merge it as-is. The diagnosis that the OpenAI SDK can skip post-`[DONE]` SSE data is plausible, but the patch suppresses `stream_options.include_usage` for the affected custom/local backend instead of preserving nonzero llama.cpp usage, and the supplied proof only simulates object construction rather than showing an OpenClaw llama.cpp turn saving corrected usage.

## What This Changes

The PR changes `buildOpenAICompletionsParams` so `stream_options: { include_usage: true }` is only included when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Preserve usage visibility for llama.cpp by fixing the OpenAI-completions stream handling to capture the final usage chunk or normalize llama.cpp timing metadata, with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main always requests streaming usage, the linked issue shows saved `0/0/0` usage from llama.cpp, and the parser only records yielded usage chunks; I did not have a live llama.cpp server to run the full turn.

## Solution Assessment

Is this the best way to solve the issue? No: the patch removes the usage request for the affected false-compat backends rather than making OpenClaw consume or recover the usage data that `/status` and compaction need.
