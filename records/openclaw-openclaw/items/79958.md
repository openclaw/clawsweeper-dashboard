## Summary

Keep open: the PR targets a real source-reproducible usage-accounting bug, but this patch removes the `stream_options.include_usage` request for the reported custom/local OpenAI-compatible class, so it is unlikely to fix the zero-usage symptom. The supplied proof is a synthetic payload-construction check rather than an after-fix OpenClaw/llama.cpp run.

## What This Changes

This PR changes OpenAI Chat Completions streaming request construction to send `stream_options.include_usage` only when `compat.supportsUsageInStreaming` is true.

## Best Possible Solution

Keep requesting streaming usage for affected OpenAI-compatible backends, then fix the stream consumer or fallback path to preserve the trailing llama.cpp usage/timings before saved usage stays zero.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main records only SDK-yielded usage chunks, the linked issue supplies llama.cpp log/curl evidence, and OpenAI SDK v6.37.0 ignores SSE data after `[DONE]`.

## Solution Assessment

Is this the best way to solve the issue? No: gating off `stream_options.include_usage` removes the request that makes affected local/custom backends emit usage metadata, so the saved usage can still remain zero.
