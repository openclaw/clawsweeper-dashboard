## Label Justifications

- `P1`: The issue reports a recent upgrade regression where Control UI webchat can fail to deliver any assistant reply even though the same model/auth setup works from the CLI.
- `impact:message-loss`: The affected webchat turn can end with every fallback candidate aborted and no assistant message returned.

## Summary

Keep open. Current main and the latest release still have the reported abort shape: Control UI webchat registers one timeout-backed chat-send abort controller, forwards that signal into the embedded agent path, and the fallback layer still treats timeout-shaped aborts as retryable candidate failures. The linked open PR is useful for the misleading fallback cascade, but its own discussion says it does not address why the Control UI timeout can fire before embedded startup.

## What This Changes

The issue reports that Control UI webchat can reuse an already-timed-out chat-send abort signal across all embedded Codex fallback candidates, producing an `All models failed` timeout cascade instead of a useful timeout boundary.

## Best Possible Solution

Keep this open until the gateway/agent abort boundary separates request cleanup from model-attempt execution timeout, while preserving explicit user/client abort behavior and landing focused regression coverage for delayed embedded startup plus fallback.

## Reproduction Assessment

No, not as a live end-to-end reproduction. The issue is source-reproducible because current main and `v2026.5.20` wire one timeout-backed webchat abort signal into the embedded fallback path, and timeout-shaped aborts are still handled as retryable fallback failures.

## Solution Assessment

Yes, the right fix direction is a bounded gateway/agent abort-boundary repair rather than a new setting. The linked PR is useful for the fallback cascade, but the remaining best solution should also address the Control UI timeout source that can fire before embedded startup.
