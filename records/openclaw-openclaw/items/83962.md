## Summary

Keep open. The report is recent, matches current-main source wiring, and no merged post-release fix separates Control UI chat-run timeout/cleanup from the embedded fallback attempt abort signal.

## What This Changes

The issue asks for Control UI webchat fallback handling to avoid one expired webchat abort signal aborting every fallback candidate before the embedded Codex run starts.

## Best Possible Solution

Separate chat-run lifetime cleanup from per-attempt model execution so timeout expiry cannot pre-abort all fallback candidates, while explicit user abort remains terminal.

## Reproduction Assessment

No live high-confidence reproduction was run. The issue is source-reproducible because current main wires one chat-send abort controller through dispatch and embedded fallback, then immediately aborts attempts when that parent signal is already fired.

## Solution Assessment

Yes, the best direction is a bounded abort-ownership fix rather than a new setting or fixed WebChat timeout cap. The fix should distinguish timeout expiry from explicit user aborts and add regression coverage for delayed startup followed by fallback.
