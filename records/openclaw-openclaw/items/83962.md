## Label Justifications

- `P1`: The report describes an upgrade regression where Control UI webchat can fail to deliver an assistant reply even though the same model/auth setup works through the CLI.
- `impact:message-loss`: The affected webchat turn can end with every fallback candidate aborted and no assistant response delivered.
- `impact:auth-provider`: The failure involves model/provider fallback routing and timeout classification across configured fallback candidates.

## Summary

Keep open. Current main still has a source-visible path where a Control UI `chat.send` timeout-backed abort signal can reach the embedded fallback chain, and the fallback boundary still treats timeout-shaped aborts as retryable candidate failures; the related terminal-abort PR is open and clarifies the cascade but has not landed or proven the webchat timeout source fixed.

## What This Changes

The issue requests that Control UI webchat timeout handling stop aborting every fallback candidate before the embedded Codex run can start.

## Best Possible Solution

Land the terminal-abort boundary fix or an equivalent, then add a corrected WebChat timeout/backstop fix only if live or focused regression proof shows the pre-start timeout still occurs under the existing `chat.send` timeout contract.

## Reproduction Assessment

Yes at source level, but not as a live Control UI run. Current main wires one timeout-backed `chat.send` abort signal into dispatch and fallback while timeout-shaped aborts remain retryable candidate failures.

## Solution Assessment

Yes for the terminal-abort boundary direction, with a webchat-root-cause caveat. The maintainable solution is to distinguish terminal run/request aborts from retryable provider timeouts, while any WebChat timeout backstop must reuse the existing timeout contract rather than adding a new 30s policy.
