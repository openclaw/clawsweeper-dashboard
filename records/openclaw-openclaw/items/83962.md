## Summary

Keep open. The report matches a current-main source path: Control UI `chat.send` can carry one timeout-backed abort signal through the whole embedded agent/fallback chain, so an expired chat run can make every candidate fail as aborted before useful fallback work starts.

## What This Changes

The issue requests that Control UI webchat timeouts not reuse one already-fired abort signal across all embedded Codex fallback candidates.

## Best Possible Solution

Separate chat-run lifetime cleanup from per-attempt model execution timeout so a server-side chat-send timeout cannot poison every fallback candidate, while explicit user aborts still stop the full run.

## Reproduction Assessment

No live high-confidence reproduction was run. The issue is source-reproducible: current main wires one chat-send abort controller through dispatch and embedded fallback, and the embedded runner immediately aborts when that signal is already fired.

## Solution Assessment

Yes, the best direction is a bounded gateway/agent abort-boundary fix rather than a new user setting. The implementation should distinguish timeout expiry from explicit user aborts and add regression coverage for delayed startup followed by fallback.

## Repair Work Prompt

Fix the Control UI webchat fallback abort regression reported at https://github.com/openclaw/openclaw/issues/83962. Current main registers one timeout-backed `chatAbortControllers` signal in `chat.send`, passes that signal through `dispatchInboundMessage`, and the agent fallback/embedded runner path can reuse the already-aborted signal across all fallback candidates. Make fallback attempts avoid being poisoned by a chat-send timeout that fired before an embedded candidate starts, while preserving explicit user aborts from `chat.abort`, stop commands, and session aborts. Prefer a bounded gateway/agent abort-boundary refactor; do not add a new config option. Add regression coverage that simulates a delayed or timed-out first candidate before embedded startup and verifies a later fallback candidate can start when the user did not explicitly abort, plus coverage that explicit `chat.abort` still cancels the whole run.
