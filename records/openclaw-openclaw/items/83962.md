## Summary

Keep open. Current main and the latest shipped release still show a single Control UI `chat.send` timeout-backed abort signal flowing into the embedded agent and model-fallback path, so the report remains a plausible current bug rather than stale cleanup.

## What This Changes

Control UI webchat can time out a `chat.send` run and then abort every configured embedded Codex fallback candidate before useful execution starts.

## Best Possible Solution

Separate chat-run lifetime cleanup from per-attempt model execution timeout so server-side timeout expiry cannot poison every fallback candidate, while explicit user aborts still cancel the full run.

## Reproduction Assessment

No live high-confidence reproduction was run. The source path is reproducible on current main: one timeout-backed `chat.send` abort signal is passed through dispatch into the embedded fallback chain, and abort-shaped errors can be normalized as timeout failures.

## Solution Assessment

Yes, the maintainable fix is a bounded gateway/agent abort-boundary change rather than a new timeout setting. It should distinguish server-side timeout expiry from explicit user abort and add regression coverage for delayed startup followed by fallback.
