## Summary

Keep open: this is a plausible gateway/Gmail reload bug fix, but the proposed cancellation path still allows a cancelled Gmail startup to run Tailscale endpoint mutation before noticing cancellation, and the external PR only provides unit/mock-level proof.

## What This Changes

This PR moves Gmail watcher startup into cancellable post-ready sidecar handles, stops those handles during reload/close, threads abort state into command execution, and adds gateway/process/Gmail tests.

## Best Possible Solution

Keep the targeted abort/sidecar direction, but make Tailscale setup cancellable or serialize restart with the cancelled startup, then verify a real Gmail/Tailscale reload before merge.

## Reproduction Assessment

No live Gmail/Tailscale reload was run in this read-only review. The review blocker is high-confidence from source: in the PR, cancellation during `ensureTailscaleEndpoint` is only observed after the endpoint commands finish.

## Solution Assessment

No. The exec abort normalization is narrow, but the Gmail restart fix is incomplete until Tailscale setup is abortable or the replacement watcher waits for stale setup to settle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Abort Tailscale setup before stale reloads can win:** `src/hooks/gmail-watcher.ts:223-224`
  - body: The new stop handle can mark a Gmail startup as cancelled while `ensureTailscaleEndpoint` is still running, but this block checks cancellation only after the `tailscale serve/funnel --bg --set-path` command can mutate the endpoint. On reload, the old startup can finish after the fresh watcher starts and overwrite the public path/target with stale config; pass cancellation into the Tailscale commands or wait for the cancelled startup before starting the replacement.
  - confidence: 0.88
