## Summary

Keep open: the PR targets a real gateway reload/abort race, but it is draft, still has a blocking Tailscale cancellation gap, and the external real behavior proof is only unit/mock proof.

## What This Changes

The PR moves Gmail watcher/model startup into cancellable post-ready sidecars, stops queued sidecars before Gmail hot reload, threads AbortSignal support through runCommandWithTimeout, and adds focused unit tests.

## Best Possible Solution

Keep a focused gateway fix that cancels or drains Gmail post-ready startup across Tailscale setup, watch registration, and serve startup before hot reload starts the replacement watcher.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection shows the stale-sidecar race and the PR's remaining Tailscale cancellation gap, but the provided verification is targeted unit/mock proof rather than a real Gmail/Tailscale reload run.

## Solution Assessment

No, not yet. The direction is narrow and maintainable, but the implementation should either make ensureTailscaleEndpoint abortable or wait for in-flight stopped sidecars before starting the replacement watcher.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Abort Tailscale setup before restart can race:** `src/hooks/gmail-watcher.ts:223-224`
  - body: The new reload flow stops the post-ready Gmail sidecar and immediately starts a fresh watcher, but this cancellation check only runs after `ensureTailscaleEndpoint` finishes. If the old sidecar is inside `tailscale serve/funnel --bg --set-path`, it can still write the stale path/target after reload has begun and race with the replacement; pass cancellation through Tailscale setup or wait for the stopped sidecar to finish before restarting.
