## Summary

Keep this draft PR open, but it is not ready to merge: the abort/sidecar direction is useful, yet the current head still lets a cancelled Gmail startup run non-abortable Tailscale endpoint commands after reload, and the submitted proof is unit/mock-only.

## What This Changes

The PR moves Gmail watcher startup into cancellable post-ready sidecar handles, stops those handles during reload/close, adds abort-aware command execution, and expands gateway/process/Gmail tests.

## Best Possible Solution

Make Gmail/Tailscale startup cancellation cover the Tailscale commands or wait for stale startup to settle, refresh the branch, and verify a real hot reload with redacted evidence.

## Reproduction Assessment

No live Gmail/Tailscale reload was run, so there is not a high-confidence end-to-end reproduction. The review blocker is high-confidence from source because the PR checks cancellation only after non-abortable Tailscale commands can mutate the endpoint.

## Solution Assessment

No. The exec abort normalization is narrow, but the Gmail restart fix is incomplete until Tailscale setup is abortable or the replacement watcher waits for the cancelled startup to settle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Abort Tailscale setup before stale reloads can win:** `src/hooks/gmail-watcher.ts:223-224`
  - body: The new cancellation check runs after `ensureTailscaleEndpoint`, but that helper invokes `tailscale status` and `tailscale serve/funnel --bg --set-path` without a signal. A reload can stop the old post-ready sidecar and immediately start a fresh watcher while the old Tailscale command still mutates the public endpoint, so pass cancellation into the helper or await the cancelled startup before replacement.
  - confidence: 0.88
