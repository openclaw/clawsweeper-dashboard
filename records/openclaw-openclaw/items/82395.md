## Summary

Keep open: current main does not already implement this fix, and the PR targets a real gateway/Gmail lifecycle race, but the current head is incomplete because stale cancelled startup work can still mutate the Tailscale endpoint before cancellation is checked.

## What This Changes

The PR moves Gmail watcher startup into cancellable post-ready sidecar handles, adds abort-signal handling to command execution, and adds gateway/process/Gmail tests.

## Best Possible Solution

Make Gmail/Tailscale startup cancellation cover the Tailscale commands or wait for stale startup to settle before replacement, then verify the real reload path with redacted live evidence.

## Reproduction Assessment

No high-confidence live reproduction was provided or run. Source inspection does reproduce the remaining race: cancellation is checked after ensureTailscaleEndpoint can run uncancelled tailscale commands.

## Solution Assessment

No. The abort normalization is narrow, but the Gmail restart fix is not complete until Tailscale setup is abortable or the replacement watcher waits for cancelled startup work to settle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Abort Tailscale setup before stale reloads can win:** `src/hooks/gmail-watcher.ts:223-224`
  - body: The new cancellation check runs after `ensureTailscaleEndpoint`, but that helper runs `tailscale status` and `tailscale serve/funnel --bg --set-path` without a signal. A reload can stop the old sidecar and immediately start a fresh watcher while the old Tailscale command still mutates the endpoint, so pass cancellation into the helper or wait for cancelled startup to settle before replacement.
  - confidence: 0.88
