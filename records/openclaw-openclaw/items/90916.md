## Summary

Keep open: current main and the latest release have session labels, parent linkage, thread/topic routing, and memory primitives, but not the requested first-class topic-session family model or explicit family-level memory handoff boundary. The adjacent UI and shared-session issues remain open and do not supersede this narrower product/security scoping request.

## What This Changes

Add an explicit topic-session family model so one assistant/persona can manage multiple named isolated topic lanes with controlled durable memory sharing and visible chat/UI lifecycle.

## Best Possible Solution

Keep this as a maintainer RFC/scoping tracker, decide the family data model and memory/security boundary first, then split implementation into protocol/store, chat commands, UI, memory handoff/docs, and routing tests.
