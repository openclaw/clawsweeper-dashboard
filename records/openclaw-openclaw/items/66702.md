## Summary

Keep open. Current main has shipped a narrower old-history marker scrub and separates some runtime context from the visible auto-reply prompt, but it still preserves recent prior image blocks in model replay and does not implement the requested current-turn-only media policy for Telegram image turns.

## What This Changes

The issue requests that multimodal Telegram turns send only current-turn media to vision models unless the user explicitly asks to compare earlier images, avoiding stale images from injected context or recent history.

## Best Possible Solution

Add an embedded-runner media policy for image-bearing turns: keep current-turn attachments available, mask or omit prior replay image blocks and hydratable media markers by default, and provide a deliberate path for explicit comparisons with earlier images. Fold in the memory/untrusted-context protection from the related PR or a replacement implementation, with regression tests for Telegram topic sessions, active-memory/relevant-memories refs, and recent image-heavy history.

## Reproduction Assessment

Yes, for a source-level reproduction: current main preserves recent image blocks in replay while separately adding current prompt images, so an image-heavy session can present prior and current screenshots together to the model. A live Telegram run was not performed in this read-only review.

## Solution Assessment

No, current main is not the best solution yet. The #71868 scrub is a good old-history mitigation, but the narrow maintainable fix still needs a current-turn media boundary plus explicit comparison semantics; the open memory-block PR is only one slice of that fix.
