## Summary

Keep this PR open as a draft feature: the direction of reusing `tools.invoke` is consistent with current Gateway contracts, but the patch has a fail-closed validation bug and the external PR does not include real behavior proof.

## What This Changes

The PR adds `executeToolPlan`, a new `openclaw tools execute-plan` CLI, command registration, tests, and CLI documentation for replaying JSON tool plans through Gateway `tools.invoke`.

## Best Possible Solution

Keep the PR open as a draft feature only if maintainers want this core CLI surface, then make plan parsing fail closed, use real tool IDs in docs, and add redacted real Gateway CLI proof before merge.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: a present-but-invalid `input`, `args`, or `arguments` value reaches `normalizeArgs` and is dispatched as `args: {}`. I did not run it because this review was read-only.

## Solution Assessment

No: reusing `tools.invoke` is the right boundary, but this patch should validate plan argument shapes before dispatch and document real tool IDs before it is a maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed plan arguments before invoking tools:** `src/tools/execute-plan.ts:62-64`
  - body: `normalizeArgs` turns any non-object or array `input`/`args`/`arguments` value into `{}`, and `tools-cli` then sends that empty object to `tools.invoke`. That bypasses the Gateway schema's record-shaped `args` validation and can execute a different empty-argument tool call instead of failing the bad plan before side effects. Treat present-but-invalid argument payloads as plan validation errors.
  - confidence: 0.89
- **[P3] Use existing tool IDs in the CLI examples:** `docs/cli/tools.md:31-32`
  - body: The new CLI page demonstrates `openclaw.version` and `deploy.production`, but those are not current OpenClaw tool IDs. Users copying the documented plan will get `not_found` rather than the shown completed/blocked flow, so the examples should use real low-risk tool names or clearly mark fictional ones as placeholders.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The patch preserves the Gateway policy boundary, but malformed plan argument payloads are silently coerced before arbitrary tool invocation.

Concerns:

- **[medium] Malformed args can become empty tool calls:** `src/tools/execute-plan.ts:62`
  - body: Because `normalizeArgs` converts invalid argument shapes to `{}`, the CLI can bypass `tools.invoke`'s record-shaped `args` validation and dispatch an unintended empty-argument call. For a command designed to replay tool plans, plan validation should fail closed before any tool body or approval path is reached.
  - confidence: 0.86
