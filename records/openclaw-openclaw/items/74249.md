## Summary

Keep open. Current main still has only global and explicit timeout overrides; there is no typed `channelTimeoutSeconds` config surface, no channel-aware timeout resolver, and the docs still describe `agents.defaults.timeoutSeconds` as the agent runtime default.

## What This Changes

Add per-channel agent run timeout overrides for Telegram, Discord, WebChat, and cron while preserving existing timeout fallback behavior.

## Best Possible Solution

Add a backward-compatible typed channel timeout map to agent defaults and entries, thread normalized channel identity into interactive timeout resolution, keep cron job `payload.timeoutSeconds` first, and document/test the fallback order.

## Reproduction Assessment

Yes for the repository-level gap: current main has no `channelTimeoutSeconds` key and no channel parameter in the shared timeout resolver. The live OAuth soft-throttle hang was not reproduced, but source inspection is enough to verify the missing configuration capability.

## Solution Assessment

Yes for the generic channel-aware resolver direction, because it keeps the behavior in shared agent timeout policy instead of channel-specific forks. The remaining product choice is whether to ship opinionated default values or leave the new map opt-in.
