## Summary

Keep this issue open. Current main still has only a global agent runtime timeout plus explicit per-call overrides; there is no `channelTimeoutSeconds` config key, no schema/type support for defaults or per-agent entries, and the interactive reply/webchat/cron paths do not pass channel identity into the shared resolver. The related Telegram and active-memory reports are useful context for the pain, but they do not supersede this missing generic config/runtime capability.

## What This Changes

Request support for per-channel agent timeout overrides across Telegram, Discord, webchat, and cron while preserving existing global timeout behavior.

## Best Possible Solution

Add a generic typed timeout-resolution path that supports `agents.defaults.channelTimeoutSeconds` and per-agent `agents.list[].channelTimeoutSeconds`, threads request channel identity into interactive agent runs, preserves cron `payload.timeoutSeconds` as the explicit highest-priority override, and documents the config without changing existing defaults unless maintainers choose new per-channel defaults.

## Reproduction Assessment

Yes for the repository-level gap: static inspection on current main shows the proposed key is absent from code/docs, strict schemas do not accept it, and all inspected interactive paths use the single shared timeout resolver result. The OpenAI soft-throttle hang itself was not live-reproduced, but it is not needed to verify the missing config capability.

## Solution Assessment

Yes, a generic resolver extension is the narrow maintainable direction. The safer implementation is to ship opt-in overrides first and leave built-in per-channel default values unchanged unless a maintainer explicitly approves the behavior change.
