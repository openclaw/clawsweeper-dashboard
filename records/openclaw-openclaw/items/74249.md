## Summary

Keep this issue open. Current main still has only global/default agent timeout resolution plus explicit per-request or per-cron-job overrides; there is no validated `channelTimeoutSeconds` config surface, and Gateway/WebChat/Cron call sites do not pass channel context into timeout resolution. The related Telegram and active-memory issues are useful evidence for the latency pain, but they do not supersede or implement this requested config granularity.

## What This Changes

The issue requests per-channel agent timeout overrides for Telegram, Discord, WebChat, and cron fallback behavior while preserving existing global timeout semantics.

## Best Possible Solution

Add a shared channel-aware timeout resolver for Gateway agent runs, WebChat/chat.send, and cron isolated-agent fallback paths; keep explicit request and cron job timeouts first, preserve existing global behavior when no channel override is configured, and update config types, strict schemas, generated config metadata/help, docs, changelog, and focused regression tests together.

## Reproduction Assessment

Yes for the config/runtime gap: current main can be checked by searching for `channelTimeoutSeconds`, inspecting the strict schemas, and following Gateway/WebChat/Cron timeout calls. The exact silent provider-hang reproduction remains environment-dependent.

## Solution Assessment

Unclear until maintainers approve the public config shape and default timeout values. The maintainable implementation direction is a shared resolver plus schema/docs/test coverage, not separate transport-level timeout tweaks in each channel.
