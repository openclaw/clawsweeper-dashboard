## Summary

Keep this open. Current main still has only the single agent runtime timeout resolver, no `channelTimeoutSeconds` config surface, and no channel-aware timeout input in the interactive reply path; cron already has the explicit per-job override, but that covers only the first priority in the requested resolution order. The related Telegram stall reports are useful supporting context, not a duplicate or shipped fix for this missing core config capability.

## What This Changes

Add per-channel agent runtime timeout overrides under `agents.defaults` and per-agent config, with cron job `payload.timeoutSeconds` remaining the highest-priority override.

## Best Possible Solution

Add a generic typed timeout-resolution path for `agents.defaults.channelTimeoutSeconds` and `agents.list[].channelTimeoutSeconds`, normalize channel ids at the runtime boundary, preserve cron `payload.timeoutSeconds` as the explicit top priority, keep existing `timeoutSeconds` behavior unchanged when no channel override is configured, and document the resolution order in the agent config reference.

## Reproduction Assessment

Yes for the repository-level gap: adding `channelTimeoutSeconds` today has no schema or runtime path, and static inspection shows interactive agent runs use the single global resolver. The provider soft-throttle hang itself was not live-reproduced, but live provider behavior is not needed to verify the missing configuration capability.

## Solution Assessment

Yes for the override mechanism if it is implemented as a generic resolver extension rather than channel-specific branches. The proposed built-in defaults are the only unclear part; the safer first fix is to support configured overrides while preserving current defaults unless maintainers explicitly approve new per-channel defaults.
