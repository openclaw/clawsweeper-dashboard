## Summary

Keep this issue open. Current main still has global/default agent-run timeout resolution plus explicit request or cron payload overrides, but no channel-aware timeout config for Telegram, Discord, WebChat, or cron defaults. The related Telegram stall and active-memory reports are useful context for the pain point, but they do not supersede or implement this requested public config surface.

## What This Changes

The issue requests `channelTimeoutSeconds` overrides in global and per-agent config so Telegram, Discord, WebChat, and cron can use different agent-run timeout defaults while preserving existing `timeoutSeconds` semantics.

## Best Possible Solution

Add one shared channel-aware timeout resolver used by Gateway agent runs, WebChat/chat.send, and cron fallback paths; preserve existing global behavior when no override is configured, keep cron `payload.timeoutSeconds` first, and update config schemas, generated metadata/help, docs, and focused regression tests together.

## Reproduction Assessment

Yes for the config/runtime gap: current main can be verified by searching for `channelTimeoutSeconds`, inspecting strict schemas, and following Gateway/WebChat/cron timeout calls. The provider soft-throttle hang itself remains environment-dependent.

## Solution Assessment

Unclear until maintainers approve the public API shape and default values. The maintainable implementation direction is a shared resolver plus schema/docs coverage, not separate transport-level timeout tweaks in each channel.
