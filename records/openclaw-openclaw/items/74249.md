## Summary

Keep open. Current main still has only global agent timeout resolution plus explicit per-call and cron overrides; there is no typed per-channel timeout map or channel-aware resolver path.

## What This Changes

Request a backward-compatible per-channel agent timeout configuration for Telegram, Discord, WebChat, and cron-related agent runs.

## Best Possible Solution

Add a typed channel timeout map to agent defaults and entries, thread normalized channel identity into shared timeout resolution, keep cron `payload.timeoutSeconds` first, and document/test the fallback order.

## Reproduction Assessment

Yes for the repository-level gap: current main has no `channelTimeoutSeconds` key and no channel parameter in the shared timeout resolver. The live provider soft-throttle scenario was not reproduced.

## Solution Assessment

Yes for the shared resolver direction, because it avoids channel-specific forks while preserving existing explicit overrides. The open product choice is whether to ship opinionated channel defaults or make the map opt-in.
