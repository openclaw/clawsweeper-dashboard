## Summary

Keep this issue open. Current main still has only a global agent runtime timeout plus explicit per-run overrides, while `channelTimeoutSeconds` is absent from config schema/types, runtime resolution, and docs; the linked Telegram stall/event-loop issues are related operational context but do not supersede this remaining configuration gap.

## What This Changes

The issue requests schema-validated per-channel agent runtime timeout overrides for Telegram, Discord, WebChat, and cron while preserving existing `timeoutSeconds` fallback behavior.

## Best Possible Solution

Add an optional, schema-documented agent runtime timeout resolver that can choose a channel-specific timeout from the current request channel before falling back to existing agent/global timeout behavior, with cron job `payload.timeoutSeconds` and explicit RPC overrides still taking priority.

## Reproduction Assessment

Yes for the configuration gap: current main has no `channelTimeoutSeconds` schema/type/runtime/docs support, and the central resolver only reads `agents.defaults.timeoutSeconds`. The provider soft-hang production scenario itself remains externally dependent and was not live reproduced.

## Solution Assessment

Yes, the requested direction is the right shape if implemented as a generic core runtime/config seam rather than Telegram-specific behavior. The safer path is to preserve existing defaults initially and let maintainers decide whether to ship shorter built-in channel defaults later.
