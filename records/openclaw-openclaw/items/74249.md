## Summary

Keep this issue open. Current main still has one shared agent-run timeout resolver plus explicit request/cron payload overrides, but no validated channel-aware timeout config for Telegram, Discord, WebChat, or cron fallback behavior; the related Telegram stall reports provide useful context but do not implement or supersede this public config surface.

## What This Changes

The issue requests channel-specific agent timeout overrides for Telegram, Discord, WebChat, and cron while preserving existing global timeout fallback semantics.

## Best Possible Solution

Add a single channel-aware timeout resolver used by Gateway agent runs, `chat.send`/WebChat, and cron fallback paths; keep current behavior when no channel override is configured, preserve cron `payload.timeoutSeconds` as highest priority, and update schemas, generated config metadata/help, docs, and focused regression tests together.

## Reproduction Assessment

Yes for the config/runtime gap: current main can be reproduced by searching for `channelTimeoutSeconds`, inspecting the strict config schemas, and following the Gateway/WebChat/cron timeout call sites. The provider soft-throttle hang itself remains unclear without a live throttled OpenAI OAuth profile.

## Solution Assessment

Unclear until maintainers approve the public API shape and default channel values. The maintainable implementation direction is a shared resolver plus schema/docs/test coverage, not separate transport-level timeout tweaks in each channel.
