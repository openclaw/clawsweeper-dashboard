## Summary

Keep this open. Current main still lacks the requested per-channel agent runtime timeout config: there is no `channelTimeoutSeconds` key in code/docs, the shared timeout helper reads only the global agent timeout plus explicit per-call overrides, and cron’s existing per-job `payload.timeoutSeconds` covers only the first priority in the proposed order. The related Telegram stall reports are useful context, but they do not supersede this missing config/runtime capability.

## What This Changes

The issue requests per-channel agent timeout overrides for Telegram, Discord, webchat, and cron while preserving existing global timeout behavior.

## Best Possible Solution

Add a generic typed timeout-resolution path that supports `agents.defaults.channelTimeoutSeconds` and per-agent `agents.list[].channelTimeoutSeconds`, threads the originating channel into interactive agent runs, preserves cron `payload.timeoutSeconds` as the highest-priority explicit override, and keeps existing behavior unchanged when no per-channel override is configured.

## Reproduction Assessment

Yes for the repository-level gap: adding `channelTimeoutSeconds` today has no schema or runtime path, and static inspection shows all interactive agent runs use the single global resolver result. The provider soft-throttle scenario is not live-reproduced, but it is not required to verify that the requested config capability is missing.

## Solution Assessment

Yes, the proposed direction is maintainable if implemented as a generic resolver extension rather than channel-specific branches. The safer variant is to ship the override mechanism first and only change default timeout values with explicit maintainer approval.
