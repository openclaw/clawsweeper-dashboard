## Summary

Keep this open. Current main still has no `channelTimeoutSeconds` config key, the shared timeout resolver only reads the global agent default plus explicit overrides, and the main interactive reply paths still do not pass channel identity into timeout resolution. The related Telegram stall reports are useful supporting context, but they do not supersede this missing core config/runtime capability.

## What This Changes

Add per-channel agent runtime timeout overrides under `agents.defaults` and `agents.list[]`, while keeping cron job `payload.timeoutSeconds` as the highest-priority explicit override.

## Best Possible Solution

Add a generic typed timeout-resolution path for `agents.defaults.channelTimeoutSeconds` and `agents.list[].channelTimeoutSeconds`, normalize channel ids at the runtime boundary, preserve explicit cron/request timeout overrides as top priority, preserve existing default behavior when no channel override is configured, and document the resolution order in the agent config reference.

## Reproduction Assessment

Yes for the repository-level gap: adding `channelTimeoutSeconds` today has no schema or runtime path, and static inspection shows interactive agent runs use the single shared resolver. The provider soft-throttle hang itself was not live-reproduced, but live provider behavior is not needed to verify the missing configuration capability.

## Solution Assessment

Yes for the configured override mechanism if it is implemented as a generic resolver extension rather than channel-specific branches. The proposed built-in defaults are the unclear part; the safer first fix is support for configured overrides while preserving current defaults unless maintainers approve new channel defaults.
