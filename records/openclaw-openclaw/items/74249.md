## Summary

Keep open. Current main still has no `channelTimeoutSeconds` config/schema/docs surface or channel-aware runtime resolver; the existing implementation is a global agent timeout plus explicit per-request or cron-job overrides.

## What This Changes

Request per-channel agent runtime timeout overrides for Telegram, Discord, webchat, and cron fallback behavior.

## Best Possible Solution

Add a backward-compatible typed `channelTimeoutSeconds` surface for defaults and agent entries, thread normalized channel identity into the timeout resolver, keep cron `payload.timeoutSeconds` as the highest priority, and document/test the fallback order without changing built-in defaults unless maintainers approve.

## Reproduction Assessment

Yes for the repository-level gap: `channelTimeoutSeconds` is absent and timeout resolution has no channel input on current main. The live OAuth soft-throttle hang was not reproduced, but it is not needed to verify the missing configuration capability.

## Solution Assessment

Yes, a generic channel-aware resolver extension is the narrow maintainable direction. The proposed built-in default values are the only part that should be treated as separate maintainer policy.
