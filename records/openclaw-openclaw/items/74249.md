## Summary

Keep this open. Current main still has no `channelTimeoutSeconds` config key, the shared timeout resolver only reads explicit overrides plus `agents.defaults.timeoutSeconds`, and the interactive reply path computes one timeout for all originating channels. Cron already preserves the explicit per-job timeout, but that covers only the top precedence case and does not solve the requested channel-aware fallback behavior.

## What This Changes

Request per-channel agent runtime timeout overrides for Telegram, Discord, Webchat, and cron, with schema/docs support and a defined precedence order above the existing global timeout.

## Best Possible Solution

Add a generic timeout-resolution path keyed by normalized channel id for `agents.defaults` and `agents.list[]`, keep cron `payload.timeoutSeconds` as the explicit top priority, preserve existing global fallback behavior when no channel override is configured, and document the precedence in the agent config reference.

## Reproduction Assessment

Yes for the repository-level gap: adding `channelTimeoutSeconds` today has no schema, type, docs, or runtime resolver path. The external soft-throttle scenario is provider-dependent, but it is not needed to reproduce the missing OpenClaw configuration capability.

## Solution Assessment

Yes, the best fix is a generic resolver extension threaded through the existing channel context, not Telegram/Discord/Webchat-specific branches. Built-in shorter defaults are the only product-policy piece; opt-in configured overrides are the safer first implementation.

## Repair Work Prompt

Implement this issue as an opt-in per-channel agent timeout configuration. Add typed/schema-validated `channelTimeoutSeconds` maps under `agents.defaults` and `agents.list[]`; also address the requested `agents.list[].timeoutSeconds` fallback if maintainers want the full stated precedence. Extend the shared timeout resolver, or add a small wrapper around it, so precedence is: explicit request override such as cron `payload.timeoutSeconds`, per-agent channel override, global channel override, per-agent timeout, global timeout, built-in default. Thread normalized originating channel identity from the interactive reply context into timeout resolution without hard-coding Telegram/Discord/Webchat branches. Preserve current behavior when no channel override is configured, and do not ship shorter built-in per-channel defaults without maintainer approval. Update config types, strict zod schemas, generated schema/help surfaces as needed, docs in `docs/gateway/config-agents.md`, and CHANGELOG with user-facing attribution. Keep core channel-agnostic and avoid plugin-specific policy in the resolver.
