## Summary

Keep open. Current main still lacks a `channelTimeoutSeconds` config, schema, docs, or runtime path; the implemented pieces are the global agent timeout resolver and cron’s explicit per-job override, so the requested per-channel fail-fast behavior remains unshipped.

## What This Changes

Request per-channel agent runtime timeout overrides for Telegram, Discord, webchat, and cron fallback paths while preserving existing global `timeoutSeconds` behavior.

## Best Possible Solution

Add a generic typed `channelTimeoutSeconds` surface for defaults and agent entries, thread normalized channel identity into timeout resolution, keep cron `payload.timeoutSeconds` as the explicit top priority, and document/test the fallback order without changing defaults unless maintainers approve.

## Reproduction Assessment

Yes for the repository-level gap: `channelTimeoutSeconds` is absent and current timeout resolution has no channel input. The provider soft-throttle hang was not live-reproduced, but it is not needed to verify the missing configuration capability.

## Solution Assessment

Yes for the configured override mechanism if implemented as a generic resolver extension. The proposed built-in default values are the only part that needs maintainer product judgment.
