## Summary

Keep open: current main still exposes only global or explicit per-run agent timeout controls, with no typed per-channel timeout map or channel-aware timeout resolver. The request is a plausible feature, but it needs maintainer decisions on public config shape, default values, and channel ID semantics.

## What This Changes

Request per-channel agent timeout overrides for Telegram, Discord, Webchat, and cron while preserving existing global and cron payload timeout behavior.

## Best Possible Solution

Add a backward-compatible typed channel timeout map to agent defaults and entries, thread normalized channel identity into timeout resolution, preserve cron `payload.timeoutSeconds` precedence, and document/test the fallback order.

## Reproduction Assessment

Yes for the repository-level gap: current main has no `channelTimeoutSeconds` key and no channel parameter in the shared timeout resolver. The live provider soft-throttle hang was not reproduced.

## Solution Assessment

Yes for the generic channel-aware resolver direction, because timeout policy should stay centralized instead of being forked across channel implementations. The unresolved part is maintainer policy on default values and channel key semantics.
