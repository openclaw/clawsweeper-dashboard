## Summary

Keep open. Current main still has only global and explicit agent-run timeout resolution; there is no `channelTimeoutSeconds` config/schema/docs surface and no channel-aware resolver path for Telegram, Discord, webchat, or cron fallback behavior.

## What This Changes

Request a backward-compatible per-channel agent-run timeout configuration for agent defaults and agent entries across Telegram, Discord, webchat, and cron.

## Best Possible Solution

Add typed `channelTimeoutSeconds` maps on agent defaults and entries, pass normalized channel identity through interactive/gateway timeout resolution, preserve cron `payload.timeoutSeconds` precedence, and document/test the fallback order without changing built-in defaults unless maintainers approve them.

## Reproduction Assessment

Yes for the repository-level gap: current main has no `channelTimeoutSeconds` key and no channel argument in timeout resolution. The live OAuth soft-throttle hang was not reproduced, but that is not required to verify the missing configuration capability.

## Solution Assessment

Yes, a generic channel-aware resolver is the narrow maintainable direction. The only open product choice is whether OpenClaw should ship opinionated channel-specific defaults or keep the new map opt-in.

## Repair Work Prompt

Implement per-channel agent-run timeout configuration for this issue. Add a backward-compatible `channelTimeoutSeconds` map to `agents.defaults` and `agents.list[]`, with positive integer seconds keyed by normalized channel id. Update the shared timeout resolver so precedence is: explicit cron/job/request override first, per-agent channel map, global channel map, per-agent/global `timeoutSeconds` as currently supported or newly added, then the existing built-in default. Thread channel identity through interactive reply and gateway paths without hard-coding bundled channel behavior in core. Preserve existing `timeoutSeconds` behavior when no channel map is configured, and do not introduce opinionated built-in Telegram/Discord/webchat defaults unless a maintainer explicitly asks. Update generated/config help surfaces and user docs, and add a single-line CHANGELOG entry because this is user-facing config behavior.
