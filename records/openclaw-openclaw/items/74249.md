## Summary

Keep open. Current main still has global agent-run timeouts plus explicit request/job overrides, but no typed per-channel agent timeout map or channel-aware timeout resolver; this is a valid feature/config request that needs maintainer product judgment before implementation.

## What This Changes

Add per-channel agent timeout overrides for Telegram, Discord, Webchat, and cron while preserving existing `timeoutSeconds` fallback behavior.

## Best Possible Solution

Add a backward-compatible, typed channel timeout map to agent defaults and agent entries, thread normalized channel identity into shared agent timeout resolution, keep cron job `payload.timeoutSeconds` first, and document/test the fallback order.

## Reproduction Assessment

Yes for the source-level gap: current main has no `channelTimeoutSeconds` key, and the shared resolver has no channel input. The live OAuth soft-throttle hang was not reproduced, but source inspection is enough to verify the missing capability.

## Solution Assessment

Yes for the requested generic resolver direction: channel-aware timeout policy belongs in shared agent runtime/config rather than separate Telegram or Discord forks. The unresolved part is maintainer choice on shipped defaults and accepted channel keys.
