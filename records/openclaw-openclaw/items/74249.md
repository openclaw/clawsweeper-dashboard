## Summary

Keep open. The earlier ClawSweeper comment still matches current main: OpenClaw has global/default timeout resolution and explicit per-run overrides, but no parsed, documented, or runtime-applied per-channel `channelTimeoutSeconds` surface for Telegram, Discord, webchat, cron fallback, or third-party channels.

## What This Changes

Request per-channel timeout overrides for agent defaults and per-agent config so interactive channels can fail fast without shortening longer webchat or cron runs.

## Best Possible Solution

Add a typed generic channel-timeout map to defaults and agent entries, thread normalized channel identity into timeout resolution, preserve existing `timeoutSeconds` fallback behavior, keep explicit cron/request overrides first, and document/test the complete fallback order.

## Reproduction Assessment

Yes. The repository-level reproduction is a direct current-main check: `channelTimeoutSeconds` is absent from schema, docs, and source, and the active resolver has no channel input; the provider soft-throttle hang does not need live reproduction to verify the missing capability.

## Solution Assessment

Yes for the configured override mechanism if it is implemented as a generic, backward-compatible resolver extension. The proposed built-in default values are the part that needs maintainer product judgment.
