## Summary

Keep open. Current main still has only global agent timeout defaults plus explicit request/job overrides; there is no typed per-channel timeout config, no channel-aware timeout resolver, and no documented `channelTimeoutSeconds` surface.

## What This Changes

The issue requests per-channel agent timeout overrides for Telegram, Discord, Webchat, and cron while preserving the current global timeout fallback.

## Best Possible Solution

Add a backward-compatible typed channel timeout map to agent defaults and entries, thread normalized channel identity into runtime timeout resolution, preserve cron `payload.timeoutSeconds` priority, and document/test the fallback order.

## Reproduction Assessment

Yes for the repository-level gap: current main has no `channelTimeoutSeconds` key and no channel-aware parameter in the shared timeout resolver. The live OAuth soft-throttle hang was not reproduced, but source inspection verifies the missing configuration capability.

## Solution Assessment

Yes for the generic channel-aware resolver direction, because timeout policy belongs in shared agent runtime config instead of Telegram or Discord forks. The remaining product choice is channel key semantics and whether defaults should be opinionated or opt-in.
