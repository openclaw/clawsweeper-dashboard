## Summary

Keep open. Current main still has only global and explicit timeout overrides; there is no typed `channelTimeoutSeconds` config surface, no channel-aware resolver, and no canonical duplicate that owns this narrower request.

## What This Changes

Add per-channel agent timeout overrides for Telegram, Discord, Webchat, and cron while preserving existing `timeoutSeconds` fallback behavior.

## Best Possible Solution

Add a backward-compatible typed channel timeout map to agent defaults and entries, thread normalized channel identity into timeout resolution, preserve cron `payload.timeoutSeconds` precedence, and document/test the fallback order.

## Reproduction Assessment

Yes for the repository-level gap: current main has no `channelTimeoutSeconds` key and no channel parameter in the shared timeout resolver. The live provider soft-throttle hang was not reproduced.

## Solution Assessment

Yes for the generic channel-aware resolver direction, because it keeps timeout policy centralized instead of forking channel code. The unresolved part is product policy around default values and channel key semantics.
