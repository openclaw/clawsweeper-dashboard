## Summary

Keep open: current main still has only global or explicit per-run timeout controls, not a typed per-channel timeout map or channel-aware resolver. This is a plausible feature, but it needs maintainer decisions on public config shape, default values, and channel-id semantics before implementation.

## What This Changes

Request adding `channelTimeoutSeconds` overrides under agent defaults and per-agent config, with channel-aware timeout resolution for Telegram, Discord, Webchat, and cron fallback behavior.

## Best Possible Solution

Add a backward-compatible typed channel timeout map to agent defaults and entries, pass normalized channel identity into centralized timeout resolution, preserve cron `payload.timeoutSeconds` precedence, and document/test the fallback order.

## Reproduction Assessment

Yes for the repository-level gap: current main has no `channelTimeoutSeconds` key and no channel parameter in the shared timeout resolver. The live provider soft-throttle hang was not reproduced.

## Solution Assessment

Yes on the central resolver direction: timeout policy should remain shared instead of being forked into each channel. The unresolved part is maintainer policy for default values and stable channel keys.
