## Summary

Keep open. Current main still lacks the generic doctor warning this PR is trying to add, but the PR is not merge-ready because the check can suppress the reported misconfiguration, can warn on valid aliases, and still has a small terminal-output security issue.

## What This Changes

The PR adds a config-validation helper, tests, and a doctor health contribution to warn when default or per-agent model primaries are missing from the loaded model catalog.

## Best Possible Solution

Land a focused doctor/config diagnostic that uses runtime-equivalent model-ref resolution, distinguishes provider/static/runtime catalog evidence from self-declared config rows, sanitizes terminal output, covers the #39811 and #53590 shapes, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The PR defects are reproducible by source inspection: a config that declares an invalid model under `models.providers.openai.models` and then selects it as primary will be treated as present, while an alias such as `fast` can be resolved differently from runtime unless the alias index is used.

## Solution Assessment

No. A non-blocking doctor warning is the right direction, but this implementation is not the narrowest maintainable solution until it matches runtime resolution semantics and validates against catalog evidence that is not supplied by the same config value being checked.
