## Summary

Keep open: the user problem is still real on current main, but this PR is not mergeable as-is because it can suppress the intended warning, emit false positives for valid aliases, and prints config-derived text unsafely. It also still needs real behavior proof before merge.

## What This Changes

The branch adds `collectModelConfigWarnings`, a `doctor:model-config` health contribution, and targeted tests for configured agent model primaries missing from the catalog.

## Best Possible Solution

Land or replace this with a focused doctor/config diagnostic that uses runtime-equivalent model resolution, distinguishes trusted catalog evidence from user-declared config rows, sanitizes terminal output, and coordinates with adjacent config-set warning work.

## Reproduction Assessment

Yes. Source inspection shows current main still lacks a generic default/per-agent model warning path, and the PR defects are visible in the live diff without running the branch.

## Solution Assessment

No. Warning-only validation is the right direction, but this implementation is not the best fix until it uses trusted catalog semantics, runtime-equivalent alias resolution, sanitized output, and real doctor proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })` into the warning helper, but that catalog appends `models.providers.*.models` from the same config. In the linked report's reproduction, the bogus model is first added under `models.providers.openai.models`, so it will be present and no warning is emitted even though runtime/provider calls can still fail. Use a catalog/source that distinguishes provider-discovered rows from user-declared config rows, or carry origin metadata into the check.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as a default-provider model and warned as missing. Runtime and model-picker paths build `agents.defaults.models` aliases before resolving, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path/message through the existing terminal sanitizers before display.
  - confidence: 0.82
