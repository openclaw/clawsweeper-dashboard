## Summary

Keep open: current main still lacks the generic configured-model doctor/config validation this PR targets, but the live PR diff still has correctness and terminal-output issues plus missing real behavior proof before merge.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when configured default or per-agent primary model refs are absent from the loaded model catalog.

## Best Possible Solution

Land or replace this with a focused doctor/config diagnostic that uses runtime-equivalent model resolution, distinguishes trusted provider/static catalog evidence from user-declared config rows, sanitizes terminal output, and coordinates with the config-set warning path while leaving fallback notifications to the separate issue.

## Reproduction Assessment

Yes, by source inspection: current main structurally accepts configured agent model strings and lacks a generic doctor/config diagnostic for unknown default or per-agent model refs. The PR defects are also visible in the live diff against current catalog and alias-resolution contracts.

## Solution Assessment

No. Advisory doctor validation is a reasonable direction, but this implementation should use runtime-equivalent alias/model resolution, avoid treating user-declared catalog rows as provider proof, sanitize output, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against trusted catalog rows:** `src/flows/doctor-health-contributions.ts:407`
  - body: `runModelConfigHealth` passes `loadModelCatalog({ config: ctx.cfg })`, but current catalog loading appends `models.providers.*.models` from that same config. A bogus model first added under `models.providers.openai.models` can therefore be considered present and skip the warning, which misses the linked misconfiguration class.
  - confidence: 0.88
- **[P2] Resolve aliases before warning on model refs:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without the config-built alias index that runtime model resolution uses. Valid configured aliases such as `fast` can be parsed under the default provider and incorrectly warned as missing.
  - confidence: 0.9
- **[P3] Sanitize model warning output:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning text includes config-derived paths and model identifiers and passes them directly to `note()`. Route the path and message through the existing terminal sanitizer so ANSI/control characters cannot spoof or hide doctor output.
  - confidence: 0.82
