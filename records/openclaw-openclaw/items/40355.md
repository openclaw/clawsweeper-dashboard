## Summary

Keep open. The PR targets a real configured-model diagnostic gap that is still absent from current main, but the live head still has correctness issues around catalog authority and alias resolution, plus missing real `openclaw doctor` behavior proof. The linked issue discussion also has a useful external workaround in https://github.com/exisz/openclaw-extensions, but that does not close the upstream gap.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when default or per-agent primary model refs are missing from the model catalog.

## Best Possible Solution

Land a focused diagnostic that resolves model refs the same way runtime/model-picker paths do, distinguishes trusted catalog rows from config-declared rows, sanitizes terminal output, and coordinates with the separate config-set and fallback-visibility trackers.

## Reproduction Assessment

Yes. Source inspection on current main shows doctor only validates `hooks.gmail.model`, config schema accepts string/object model refs, and config validation only rejects static suppressions; the PR defects are visible from the live diff without running the branch.

## Solution Assessment

No. A warning-only diagnostic is the right direction, but this implementation is not the best fix because it can treat self-declared config rows as catalog proof, miss runtime-supported aliases, and render config-derived text unsafely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })`, but that catalog appends `models.providers.*.models` from the same config. In the linked reproduction, the bogus model is declared there first, so this can suppress the warning the PR is meant to emit.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index. Runtime paths resolve `agents.defaults.models` aliases first, so a valid alias such as `fast` can be reported as a missing default-provider model.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning path and message include local config-derived model text and are passed directly to `note()`. A model id containing ANSI or control characters can spoof doctor output; use the existing terminal sanitizer before rendering.
  - confidence: 0.82
