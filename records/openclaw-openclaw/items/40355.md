## Summary

Keep open: current main still lacks the generic doctor warning this PR targets, but the patch should not merge as-is because it can suppress the linked misconfiguration, can false-positive valid aliases, still lacks real behavior proof, and prints config-derived warning text unsafely.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when default or per-agent model primaries are absent from the loaded model catalog.

## Best Possible Solution

Land or replace this with a focused doctor/shared diagnostic that uses runtime-equivalent model resolution, distinguishes trusted catalog evidence from self-declared config rows, sanitizes terminal output, and coordinates with adjacent config/model command warning work.

## Reproduction Assessment

Yes. Source inspection of current main and the PR head gives a high-confidence path: current main has only hooks.gmail.model doctor validation, while the PR validates through a config-inclusive catalog and omits runtime alias context.

## Solution Assessment

No. A warning-only diagnostic is a reasonable direction, but this patch is not the best fix until it uses trusted catalog semantics, runtime-equivalent alias resolution, and sanitized terminal rendering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against trusted catalog rows:** `src/flows/doctor-health-contributions.ts:408`
  - body: The new doctor check validates against `loadModelCatalog({ config: ctx.cfg })`, but that catalog appends `models.providers.*.models` from the same config. In the linked report's reproduction, the bogus model is first declared there, so this can hide the warning this PR is meant to emit.
  - confidence: 0.88
- **[P2] Resolve aliases before checking primaries:** `src/config/validation.ts:1344`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index. A valid configured alias such as `fast` can resolve at runtime through `agents.defaults.models`, but this helper checks it as a default-provider model and emits a false warning.
  - confidence: 0.9
- **[P3] Sanitize model warning output:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning path and message include model text derived from local config and are passed directly to `note()`. A model id containing control characters or ANSI escapes can spoof doctor output, so these values should be sanitized before rendering.
  - confidence: 0.82
