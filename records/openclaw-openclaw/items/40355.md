## What This Changes

The PR adds a doctor health contribution, `collectModelConfigWarnings`, and tests to warn when default or per-agent model primaries are absent from the loaded model catalog.

## Best Possible Solution

Revise or replace this PR with a focused doctor/config diagnostic that uses runtime-equivalent model resolution, trusted catalog semantics, sanitized terminal rendering, and real `openclaw doctor` proof while coordinating adjacent config-set/check-config work.

## Reproduction Assessment

Yes, by source inspection rather than a live run: origin/main has only `hooks.gmail.model` doctor validation and structural agent model schemas, while the PR defects are visible in the diff.

## Solution Assessment

No. Advisory doctor validation is a reasonable direction, but this patch should align with runtime alias/model resolution, avoid config-sourced catalog rows as proof, and sanitize output before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })`, but that catalog appends `models.providers.*.models` from the same config. In the linked report's reproduction, the bogus model is first added under `models.providers.openai.models`, so it will be present and no warning is emitted even though runtime/provider calls can still fail. Use a catalog/source that distinguishes provider-discovered rows from user-declared config rows, or carry origin metadata into the check.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without config or an alias index, so a valid configured alias such as `fast` is parsed as the default-provider model instead of resolving through `agents.defaults.models[*].alias`. That makes `doctor` warn on supported configs that runtime/model-picker paths handle correctly.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers and paths derived from local config and passes them straight to `note()`. A model id containing ANSI or control characters can spoof or hide doctor output; route the path and message through the existing terminal sanitizer before rendering.
  - confidence: 0.82
