## Best Possible Solution

Land or replace this with a focused doctor/config diagnostic that follows runtime-equivalent model resolution, separates trusted catalog evidence from user-declared config rows, covers primary and fallback refs, and sanitizes terminal output.

- **[P2] Validate against trusted catalog rows:** `src/flows/doctor-health-contributions.ts:407`
  - body: `loadModelCatalog({ config: ctx.cfg })` appends `models.providers.*.models` from the same config being checked.   - confidence: 0.9

  - body: The helper only reads primary values, so invalid `agents.defaults.model.fallbacks` or per-agent fallbacks still pass doctor silently. The linked report includes fallback misconfiguration, and current main already has collector coverage for fallback paths.
  - confidence: 0.84
- **[P3] Sanitize model warning output:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning text includes model identifiers from local config and passes them straight to `note()`. Route the path and message through the existing terminal sanitizer or config-issue formatter so control characters cannot spoof doctor output.
  - confidence: 0.82
