## Summary

Keep open. The underlying bug is valid and current main still lacks a generic doctor/config warning for invalid default or per-agent model primaries, but this PR is not safe to merge as-is because the check can self-validate against user-declared catalog rows, misses runtime alias semantics, and prints config-derived warning text without terminal sanitization.

## What This Changes

The PR adds a `collectModelConfigWarnings` helper, tests, and a `doctor:model-config` health contribution to warn when configured default or per-agent model primaries are absent from the loaded model catalog.

## Best Possible Solution

Land a focused validation path that uses the same model-ref and alias semantics as runtime/model-picker code, distinguishes provider/static/runtime catalog evidence from user-declared config rows when warning about unsupported models, sanitizes all terminal output, and covers the #39811 reproduction plus valid alias configurations. Keep fallback-user notification as a separate product thread unless the maintainer chooses to expand scope.

## Reproduction Assessment

Yes. The review can reproduce the current PR defects through deterministic source inspection: current main lacks the generic doctor check, while the PR uses a config-inclusive catalog, omits alias resolution, and prints config-derived text directly.

## Solution Assessment

No. Warning through `openclaw doctor` is a reasonable direction, but this patch is not the best maintainable fix until it matches runtime alias/provider resolution, avoids self-validating against config-provided catalog rows, and sanitizes displayed warnings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })` into the warning helper, but that catalog appends `models.providers.*.models` from the same config. In the linked #39811 shape, a bogus model first added there can appear present and suppress the warning even though provider/runtime calls still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as a default-provider model and warned as missing. Runtime and model-picker paths resolve `agents.defaults.models` aliases first, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
