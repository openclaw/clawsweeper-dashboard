## Summary

Keep open. The underlying invalid-model warning problem is still real on current main, but this PR is not merge-ready because its new doctor check can validate against model rows declared by the same user config, does not match runtime alias resolution, and still prints config-derived warning text without terminal sanitization.

## What This Changes

The PR adds a config-validation helper, tests, and a `doctor:model-config` health contribution to warn when default or per-agent model primaries are absent from the loaded model catalog.

## Best Possible Solution

Land a focused doctor or shared validation path that resolves model refs using the same alias and provider rules as runtime/model-picker code, validates against provider/static/runtime evidence rather than self-declared config rows, sanitizes all terminal output, and covers both the #39811 shape and valid alias configurations. Keep fallback user notification as a separate product thread unless maintainers intentionally expand scope.

## Reproduction Assessment

Yes. The PR defects are reproducible by source inspection: current main lacks the generic doctor check, while the PR’s diff uses a config-inclusive catalog, omits the alias index used by runtime resolution, and emits config-derived text through `note()`.

## Solution Assessment

No. Warning through `openclaw doctor` is a reasonable direction, but this implementation is not the best maintainable fix until it matches runtime resolution semantics and does not validate an invalid model against rows supplied by the same config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })` into the warning helper, but that catalog appends `models.providers.*.models` from the same config. In the linked #39811 shape, a bogus model first added there can appear present and suppress the warning even though provider/runtime calls still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as a default-provider model and warned as missing. Runtime and model-picker paths resolve `agents.defaults.models` aliases first, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
