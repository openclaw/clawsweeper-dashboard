## Summary

Keep open: current main still lacks the generic default/per-agent model diagnostic, but this PR has correctness and security findings plus missing real behavior proof, so it needs contributor and maintainer follow-up rather than cleanup closure.

## What This Changes

The branch adds `collectModelConfigWarnings`, a `doctor:model-config` health contribution, and tests to warn when default or per-agent primary model refs are absent from the loaded catalog.

## Best Possible Solution

Land or replace this with a focused doctor/config diagnostic that uses runtime-equivalent model resolution, distinguishes trusted provider/runtime/static catalog rows from user-declared config rows, sanitizes terminal output, and coordinates with the related config-set warning path.

## Reproduction Assessment

Yes, by source inspection. Current main has only a hook-scoped model doctor warning and structural/suppression-only config validation, while the PR’s false-negative and false-positive paths are visible in the diff and surrounding model-catalog contracts.

## Solution Assessment

No. Advisory doctor validation is a reasonable direction, but this implementation is not the best fix until it uses runtime-equivalent alias resolution, trusted catalog semantics, sanitized output, and real `openclaw doctor` proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })`, but that catalog appends `models.providers.*.models` from the same config. In the linked report's reproduction, the bogus model is first added under `models.providers.openai.models`, so it will be present in this catalog and no warning is emitted even though provider/runtime calls can still fail. Use catalog origin metadata or a trusted provider/runtime/static catalog source for this warning.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as `anthropic/fast` and warned as missing. Runtime and model-picker paths build `agents.defaults.models` aliases before resolving, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof or hide doctor output; route the path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
