## Summary

Keep open: current main still lacks a generic configured-model doctor/config warning, but this PR head remains incorrect because it uses the config-augmented catalog as authority, diverges from runtime alias resolution, and lacks real behavior proof. The discussion shows the author intended follow-up fixes, but the live PR head still contains the earlier problematic implementation.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when configured default or per-agent primary models are absent from the loaded model catalog.

## Best Possible Solution

Land or replace this with a focused diagnostic that uses runtime-equivalent model resolution, distinguishes trusted catalog evidence from user-declared config rows, sanitizes terminal output, and covers the linked misconfiguration path without broadening runtime behavior.

## Reproduction Assessment

Yes, by source inspection. Current main lacks a generic configured-model doctor/config warning, and the PR defects are visible in the live head diff at dc0edbcb29eaa42faec61b0f00d7852ba77a56ab.

## Solution Assessment

No. Advisory doctor validation is a reasonable direction, but this implementation should use runtime-equivalent alias/model resolution, distinguish config-authored catalog rows from trusted catalog rows, and sanitize output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:408`
  - body: `loadModelCatalog({ config: ctx.cfg })` appends `models.providers.*.models` from the same config being validated. In the linked https://github.com/openclaw/openclaw/issues/39811 reproduction, a bogus model can be declared under `models.providers.openai.models`, making this diagnostic treat it as present and skip the warning even though runtime/provider calls can still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1344`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as the default-provider model instead of resolving through `agents.defaults.models[*].alias`. Doctor would warn on supported configs that runtime accepts.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and passes them directly to `note()`. Route the path/message through the existing terminal sanitizers so control characters or ANSI escapes cannot spoof doctor output.
  - confidence: 0.82
