## Summary

Keep open: current main still lacks a generic model-config doctor check, and this active contributor PR is not mergeable as-is because the diagnostic can miss the reported misconfiguration, can warn on valid aliases, lacks real behavior proof, and still prints config-derived text unsanitized.

## What This Changes

The PR adds a config-validation helper, doctor health contribution, and tests to warn when default or per-agent model primaries are absent from the loaded model catalog.

## Best Possible Solution

Land or replace this with a focused doctor/config diagnostic that uses runtime-equivalent model resolution, distinguishes trusted provider/runtime catalog evidence from user-declared config rows, sanitizes terminal output, and coordinates with the related config-set warning path.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main lacks the generic default/per-agent model diagnostic, and the PR defects are visible in the diff and surrounding model-catalog contracts.

## Solution Assessment

No. Advisory doctor validation is a reasonable direction, but this patch should resolve aliases the same way runtime paths do, avoid treating config-sourced catalog rows as provider proof, sanitize output, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against trusted catalog rows:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new check passes loadModelCatalog({ config: ctx.cfg }), but that catalog includes models declared under the same local config via models.providers.*.models. In the linked report's reproduction, the bogus model is first added there, so this check can mark it present and emit no warning even though provider/runtime calls can still fail. Use catalog origin metadata or a trusted provider/runtime catalog source for this warning.
  - confidence: 0.88
- **[P2] Resolve aliases before checking primaries:** `src/config/validation.ts:1341`
  - body: collectModelConfigWarnings calls resolveModelRefFromString without cfg or aliasIndex, so a valid alias such as fast can be parsed as the default provider's literal model instead of resolving through agents.defaults.models aliases. That makes doctor warn on configs the runtime/model-picker paths accept.
  - confidence: 0.9
- **[P3] Sanitize model warning text before rendering:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning text includes config-derived paths and model IDs and passes them directly to note(). A model ID containing ANSI/control characters can spoof or hide doctor output; route both path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
