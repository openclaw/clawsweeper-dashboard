## Summary

Keep open: current main still lacks the generic default/per-agent model warning this PR targets, so the central work is not obsolete, but the branch still has blocking catalog-authority, alias-resolution, and terminal-output issues plus missing real behavior proof.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when configured default or per-agent primary model refs are absent from the loaded model catalog.

## Best Possible Solution

Land or replace this with a focused doctor/config diagnostic that uses runtime-equivalent model resolution, distinguishes trusted provider/runtime/static catalog evidence from user-declared config rows, sanitizes terminal output, and coordinates with the related config-set warning path.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main structurally accepts configured model strings and lacks a generic doctor/config diagnostic for unknown default or per-agent model refs, while the PR defects are visible in the diff and surrounding catalog contracts.

## Solution Assessment

No. Advisory doctor validation is a reasonable direction, but this implementation should resolve aliases like runtime paths do, avoid treating config-sourced catalog rows as provider proof, sanitize output, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes loadModelCatalog({ config: ctx.cfg }) into the warning helper, but that catalog appends models.providers.*.models from the same config. In the linked reproduction, the bogus model is first added under models.providers.openai.models, so it can be present in this catalog and no warning is emitted even though provider/runtime calls can still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: collectModelConfigWarnings calls resolveModelRefFromString without cfg or an alias index, so a valid configured alias such as fast can be parsed as anthropic/fast and warned as missing. Runtime and model-picker paths build agents.defaults.models aliases before resolving, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers and paths derived from local config and passes them straight to note(). A model id containing ANSI or control characters can spoof or hide doctor output; route the path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
