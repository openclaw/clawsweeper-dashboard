## Summary

Keep open. Current main and the latest release still route Feishu table-heavy auto/card replies through interactive-card sends without a 230099/11310 table-limit classifier or plain-text retry, and https://github.com/openclaw/openclaw/pull/50429 directly closes this issue while https://github.com/openclaw/openclaw/pull/73651 covers related outbound delivery.

## What This Changes

The issue requests Feishu to avoid dropping table-heavy card replies by falling back to plain text/table conversion or another safe rendering path when the card table limit is hit.

## Best Possible Solution

Land or replace a coordinated Feishu plugin fix that classifies 230099/11310 table-limit failures and retries affected final reply and outbound card delivery through the existing plain-text/table-conversion path; table-to-image rendering can remain a separate enhancement.

## Reproduction Assessment

Yes. The issue includes concrete Feishu 230099/11310 logs, and current main still routes Markdown-table auto/card replies and outbound sends into card delivery without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

Yes for the fallback direction, but it is not implemented on current main. The narrow maintainable fix is to classify the Feishu table-limit error and reuse the existing sendMessageFeishu markdown-table conversion path rather than making image rendering part of the bug fix.
