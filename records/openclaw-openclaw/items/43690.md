## What This Changes

The issue requests that Feishu replies with too many Markdown tables avoid silent card-delivery failure by falling back to text/post mode or another table-safe rendering.

## Best Possible Solution

Land or explicitly supersede the narrow Feishu static-card table-limit fallback with focused reply-dispatcher and outbound coverage, then close this issue with the merge commit and release provenance.

## Reproduction Assessment

Yes, at source level: current main routes Markdown-table replies into Feishu card send paths without a table-limit guard, and the issue plus linked PRs include matching 230099/11310 failure logs. I did not run a live Feishu API reproduction in this read-only review.

## Solution Assessment

Yes: the best current solution is a narrow Feishu plugin preflight or fallback that avoids over-limit static interactive cards and reuses the existing post/text table conversion path. Streaming CardKit behavior should remain separate unless maintainers intentionally broaden the fix.
