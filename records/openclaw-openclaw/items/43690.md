## Summary

Keep open: current main and the latest release still route Feishu Markdown-table replies through interactive-card paths without a five-table guard or table-limit fallback, and the active fix candidate is still open and dirty rather than merged.

## What This Changes

The issue asks OpenClaw to prevent Feishu card table-limit failures from dropping table-heavy replies, preferably by falling back to text/post mode or otherwise avoiding over-limit cards.

## Best Possible Solution

Land or replace the narrow Feishu static-card table-limit fallback against current main, with focused outbound and reply-dispatcher tests; track streaming CardKit separately unless the same limit is proven there.

## Reproduction Assessment

Yes. Current main still routes Markdown-table replies through Feishu card paths without a five-table preflight, and the issue plus linked PR proof include 230099/11310 failure logs; I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes for the static-card path. A narrow preflight/fallback to the existing post sender is the maintainable fix because sendMessageFeishu already converts Markdown tables; rendering tables as images would add a new media-rendering surface.
