## Summary

Keep this issue open: current main and the latest release still route over-limit Markdown-table replies into Feishu card paths without a table-count guard, and an open same-repository PR is the current fix candidate rather than a completed implementation.

## What This Changes

The issue reports Feishu message loss when replies with too many Markdown tables are sent as interactive cards and asks for a fallback or safer table rendering.

## Best Possible Solution

Rebase and land https://github.com/openclaw/openclaw/pull/86901 or a narrow equivalent that guards Feishu static-card table counts, reuses the existing post/text fallback, adds focused tests, and leaves distinct streaming CardKit behavior to separate review if it still reproduces.

## Reproduction Assessment

Yes, at source level: current main routes Markdown-table replies to Feishu static card send paths without a table-count guard, and the issue plus linked PR include Feishu 230099/11310 failure logs. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, the narrow static-card preflight/fallback is the best current repair shape because it avoids sending a Feishu payload the platform rejects and reuses the existing post/table conversion path. The issue should remain open until the linked PR or equivalent lands and any streaming-specific remainder is verified separately.
