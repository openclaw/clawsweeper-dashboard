## Summary

Keep open: current main and the latest release still route table-heavy Feishu replies into interactive-card paths without a table-count guard, while https://github.com/openclaw/openclaw/pull/86901 and an open closing fork PR are active fix candidates.

## What This Changes

The issue requests Feishu delivery fallback or table-safe rendering when Markdown-heavy replies exceed Feishu interactive-card table limits.

## Best Possible Solution

Land or replace the focused Feishu static-card table-limit fallback so over-limit table replies use the existing post/text path with targeted reply-dispatcher and outbound regression coverage.

## Reproduction Assessment

Yes, at source level: current main routes Markdown tables into Feishu interactive-card sends without checking Feishu's table limit, and reporter/PR logs show 230099/11310 failures. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, the best narrow fix is to prevent or catch over-limit static card sends and reuse the existing Feishu post/text path that already converts Markdown tables. Rendering tables as images is a larger feature and is not required to fix the delivery bug.
