## Summary

Keep open: current main and v2026.6.6 still route Feishu markdown-table replies into static interactive cards without a five-table guard, so the reported Feishu message loss remains valid. The useful fix path is already represented by an open proof-positive PR, but that PR is currently conflicting and needs refresh before this issue can close.

## What This Changes

The issue asks Feishu delivery to avoid dropping replies when interactive-card rendering exceeds Feishu's table limit, by falling back to text/post mode or otherwise rendering tables outside the card limit.

## Best Possible Solution

Rebase or narrowly recreate the proof-positive static-card preflight fix so every Feishu reply and outbound card decision keeps interactive cards at five or fewer markdown tables and routes larger replies through the existing post/text conversion path, with focused tests for auto mode, explicit card mode, and reply/thread routing.

## Reproduction Assessment

Yes at source level. Current main selects static card delivery for explicit card mode or auto-detected markdown tables without a table-count guard, and the linked production logs plus Feishu documentation establish rejection above five table components.

## Solution Assessment

Yes, after conflict resolution. A table-count preflight at the existing static-card selection boundaries is narrower and safer than image rendering or a catch-all retry, because it reuses the established post/text delivery path only for content that cannot fit in Feishu cards.
