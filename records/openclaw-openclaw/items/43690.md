## Summary

Keep open. Current main and latest release still route Feishu table-heavy auto/card replies through interactive-card delivery without a 230099/11310 table-limit classifier or text retry, and the issue is already paired with open fix work at https://github.com/openclaw/openclaw/pull/50429 plus related outbound work at https://github.com/openclaw/openclaw/pull/73651.

## What This Changes

The issue requests graceful Feishu delivery for responses with many Markdown tables by falling back from rejected cards to deliverable text/table-converted output instead of dropping the message.

## Best Possible Solution

Land or replace a coordinated Feishu plugin fix that classifies 230099/11310 table-limit failures and retries affected reply, streaming fallback, and outbound card delivery through the existing plain-text/table-conversion path; table-to-image rendering can remain a separate enhancement.

## Reproduction Assessment

Yes. The issue includes concrete Feishu 230099/11310 logs, and current main still routes table-heavy auto/card replies and outbound sends into card delivery without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

Yes for the fallback direction, but it is not implemented on current main. The narrow maintainable solution is to classify the Feishu table-limit error and reuse the existing sendMessageFeishu markdown-table conversion path rather than making image rendering part of the bug fix.
