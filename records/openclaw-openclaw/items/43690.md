## Summary

Keep open. Current main still routes Feishu markdown-table replies and outbound sends through interactive-card delivery without a table-limit classifier or text retry, and this issue is already paired with open implementation PRs, including a direct closing PR, so the right maintainer action is to review or consolidate that work rather than close the issue or queue another repair branch.

## What This Changes

The issue requests a Feishu fallback or table-safe rendering path when markdown-heavy card delivery fails with the platform's table-count limit.

## Best Possible Solution

A consolidated Feishu plugin fix should classify CardKit/table-limit failures such as 230099/11310, retry affected reply-card, streaming-card fallback, and outbound card sends through the existing plain-text/table-conversion path or another bounded renderer, preserve thread and mention routing, rethrow unrelated card failures, and add focused regression coverage for each delivery surface.

## Reproduction Assessment

Yes. The issue provides concrete Feishu 230099/11310 logs and steps, and current-main source inspection shows table-heavy auto/card replies still enter card delivery without a table-limit fallback; a live Feishu bot would be the final end-to-end confirmation.

## Solution Assessment

Yes for the fallback direction, but current main does not yet implement it. The narrowest maintainable fix is to classify Feishu table-limit errors and reuse sendMessageFeishu because it already applies markdown table conversion; native table or image rendering can remain follow-up polish.
