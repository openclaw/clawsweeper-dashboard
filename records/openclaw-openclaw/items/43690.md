## Summary

Keep open: current main and the latest release still route Feishu markdown-table-heavy replies into interactive/streaming card paths without a table-count guard, and the linked fix PRs are open but conflicting/unmerged.

## What This Changes

The issue asks Feishu to avoid losing replies when markdown-table-heavy content exceeds the interactive-card table limit by falling back to text/post mode or another safe rendering path.

## Best Possible Solution

Resolve or narrowly recreate the Feishu table-count fallback so more than five markdown tables use the existing post/text conversion path while preserving reply/thread routing; keep any streaming CardKit limit work tied to the separate streaming report if it remains reproducible.

## Reproduction Assessment

Yes, at source level: current main still sends markdown-table content through Feishu card paths without a table-count guard, and the issue plus linked PRs provide 230099/11310 logs. I did not run a live Feishu API reproduction in this read-only review.

## Solution Assessment

Yes, the best current direction is a narrow preflight/fallback to the existing post/text table conversion path. Rendering tables as images or catching all card failures would be broader and more likely to hide unrelated Feishu card errors.
