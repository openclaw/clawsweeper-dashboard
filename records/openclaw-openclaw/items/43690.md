## Label Justifications

- `P2`: This is a bounded Feishu channel delivery bug where table-heavy replies can fail to reach the chat.
- `impact:message-loss`: The reported failure leaves the Feishu chat with no visible reply even though the agent response exists in the control UI.

## Summary

Keep open: current main and the latest release still route table-heavy Feishu replies into interactive-card paths without a table-count guard, and the active same-repo fix candidate is still open rather than merged.

## What This Changes

Feishu should avoid silently dropping replies with too many Markdown tables by falling back from interactive cards or converting the tables before delivery.

## Best Possible Solution

Land a current-main Feishu fix that preflights the documented five-table card limit and falls back to the existing post/text table-conversion path for static card delivery, while keeping streaming CardKit changes separate unless maintainers explicitly broaden the scope.

## Reproduction Assessment

Yes, at source level: current main routes Markdown tables to Feishu static card sends without a table-count guard, and the reporter plus linked PR provide matching Feishu 230099/11310 logs. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, the narrow static-card preflight and fallback to the existing post/text table-conversion path is the best current fix shape. Rendering tables as images would be a broader feature and is not needed to stop message loss.
