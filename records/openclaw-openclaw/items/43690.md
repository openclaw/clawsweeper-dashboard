## Summary

Keep open. Current main and the latest release still route Feishu table-heavy replies into card delivery without a 230099/11310 table-limit classifier or plain-text retry, and an open PR directly closes this issue.

## What This Changes

The issue requests Feishu to avoid dropped replies when card rendering hits the platform table-count limit by falling back to plain text or otherwise avoiding card table limits.

## Best Possible Solution

Land or replace a coordinated Feishu plugin fix that classifies the 230099/11310 table-limit failure and retries affected card delivery through the existing plain-text/table-conversion path; image rendering can remain a separate enhancement.

## Reproduction Assessment

Yes, source-level. The issue includes concrete Feishu 230099/11310 logs, and current main still routes Markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

Yes for the fallback direction, but it is not implemented on current main. The narrow maintainable fix is to classify Feishu table-limit errors and reuse sendMessageFeishu/table conversion rather than making image rendering part of the required bug fix.
