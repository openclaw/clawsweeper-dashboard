## Summary

Keep open: current main and v2026.6.10 still route Feishu Markdown-table replies through interactive-card paths without a five-table guard or table-limit fallback, and the active same-repo fix candidate is open but dirty/conflicting rather than merged.

## What This Changes

The issue asks Feishu delivery to avoid dropping replies when Markdown tables exceed the interactive-card table limit, preferably by falling back to plain/post text or otherwise converting tables before card delivery.

## Best Possible Solution

Land or replace the narrow Feishu static-card table-limit fallback against current main, with focused outbound and reply-dispatcher tests, while keeping streaming CardKit behavior separate unless it is proven to share the same limit.

## Reproduction Assessment

Yes, source-reproducible: current main still routes Markdown-table replies through Feishu card paths without a five-table preflight, and the issue plus linked PR proof include 230099/11310 failure logs. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes for the static-card path: a narrow preflight or fallback to the existing post/text sender is the maintainable fix because `sendMessageFeishu` already converts Markdown tables. Rendering tables as images would add a new media-rendering surface, and streaming CardKit should stay separate unless maintainers prove it shares this limit.
