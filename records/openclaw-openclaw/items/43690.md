## Summary

Keep open: current main and the latest release still route table-heavy Feishu replies through interactive-card delivery without a table-limit guard or fallback. The open related PR is a plausible fix candidate, but it is unmerged and currently dirty, so this issue is still needed as the canonical tracker.

## What This Changes

Feishu should fall back to post/plain-text delivery, or otherwise avoid over-limit interactive cards, when replies contain too many Markdown tables.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/86901, or a narrow replacement, with table-count fallback coverage for Feishu reply-dispatcher and outbound static-card paths while keeping unrelated streaming CardKit behavior separate.

## Reproduction Assessment

Yes. Source inspection shows current main still routes Markdown tables to Feishu card sends without a table-count guard or 230099/11310 fallback, and related reports include matching Feishu API logs.

## Solution Assessment

Yes. A narrow fallback to the existing post/text path is the best current repair because that path already converts Markdown tables, while image rendering would add broader new surface.
