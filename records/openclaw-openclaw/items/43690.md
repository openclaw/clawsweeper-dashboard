## Summary

Keep open: current main and v2026.6.6 still route Feishu markdown-table replies into interactive cards without a table-count guard or table-limit fallback, and the remaining fix is already represented by open PRs that have not merged.

## What This Changes

The issue reports Feishu message loss when markdown-table-heavy replies exceed the platform card table limit and asks for fallback, image rendering, or table conversion before card delivery.

## Best Possible Solution

Land or narrowly recreate the static-card preflight fix so Feishu keeps card mode for five or fewer markdown tables and routes larger table-heavy content through the existing post/text conversion path with reply and outbound regression tests.

## Reproduction Assessment

Yes, at source level: current main still sends Feishu markdown-table content through the card path without a table-count guard, and linked issue/PR logs show the Feishu 230099/11310 rejection. I did not run a live Feishu API reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is a preflight at the existing static-card selection boundaries that falls back to the already supported post/text table conversion path. Rendering tables as images or catching all card failures would be broader and more likely to hide unrelated Feishu card errors.
