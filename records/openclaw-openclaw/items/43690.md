## Best Possible Solution

Land or replace the narrow Feishu static-card table-limit fallback against current main, with focused outbound and reply-dispatcher tests, while keeping streaming CardKit behavior separate unless proven to share the same limit.

## Solution Assessment

Yes for the static-card problem: a narrow preflight or fallback to the existing post/text path is the best maintainable fix because sendMessageFeishu already converts Markdown tables. Rendering tables as images would add a new media-rendering surface, and streaming CardKit should stay separate unless maintainers confirm it shares the limit.
