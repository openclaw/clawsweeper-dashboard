## Summary

Keep open: current main and the latest tagged release still route table-heavy Feishu replies into static interactive cards without a table-count guard, while the linked open fix PR is not merged yet.

## What This Changes

The issue requests a Feishu fallback or table-safe rendering path when markdown-table replies exceed Feishu card limits and otherwise disappear from chat.

## Best Possible Solution

Land or replace the narrow Feishu static-card table-limit fallback covering reply-dispatcher and outbound paths with focused tests; keep streaming CardKit work separate unless it is freshly reproduced.

## Reproduction Assessment

Yes at source level: current main routes markdown-table replies into Feishu static card paths without a table-count preflight, and the issue plus linked PR body provide Feishu 230099/11310 logs. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, the best current fix is a narrow static-card preflight or fallback to post/text delivery because it avoids a known Feishu card contract limit and reuses the existing table conversion path. Rendering tables as images would be broader than needed to stop message loss.
