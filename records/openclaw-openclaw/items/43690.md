## Summary

Keep open. Current main and v2026.5.12 still lack a Feishu 230099/11310 table-limit classifier or plain-text retry on the reply card path, and https://github.com/openclaw/openclaw/pull/50429 directly closes this issue while still open.

## What This Changes

Feishu interactive-card delivery should recover when Markdown-table-heavy replies hit the platform table-count limit, preferably by falling back to the existing plain-text/table-conversion path.

## Best Possible Solution

Land or replace a coordinated Feishu plugin fix that classifies the 230099/11310 table-limit failure and retries affected card delivery through the existing plain-text/table-conversion path; table images or native table rendering can remain separate enhancements.

## Reproduction Assessment

Yes, source-level. The issue includes concrete Feishu 230099/11310 logs, and current main still routes Markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

Yes for the fallback direction, but it is not implemented on current main. The narrow maintainable fix is to classify Feishu table-limit errors and reuse sendMessageFeishu/table conversion rather than making image rendering part of the required bug fix.
