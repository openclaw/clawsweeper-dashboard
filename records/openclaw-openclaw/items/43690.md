## Summary

Keep open: current main still routes Feishu markdown-table replies into structured-card delivery without a 230099/11310 table-limit classifier or plain-text retry, and the direct closing fix remains open at https://github.com/openclaw/openclaw/pull/50429. ## What This Changes

Add a Feishu fallback so table-heavy interactive card replies that hit the platform table limit are still delivered, preferably through the existing plain-text table-conversion path.

## Best Possible Solution

Land or replace a coordinated Feishu fix that classifies only the 230099/11310 card table-limit failure and retries affected final reply/card delivery through the existing plain-text table-conversion path.

## Reproduction Assessment

Yes, source-level. The reporter provided concrete Feishu 230099/11310 logs, and current main still routes markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

Yes for a table-limit-specific plain-text fallback; no for making table-image rendering the required bug fix. The narrow maintainable path is to reuse sendMessageFeishu and existing Markdown-table conversion for the affected failure only.
