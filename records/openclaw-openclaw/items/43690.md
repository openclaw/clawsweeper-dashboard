## Summary

Current main still sends table-heavy Feishu replies through static interactive-card paths without enforcing the platform’s five-table limit, so the reported message loss remains necessary to fix. The proof-positive canonical PR is still open but currently conflicts with main; this issue should remain paired with that work until a focused implementation lands.

## What This Changes

Make Feishu static-card replies exceeding the five-table platform limit fall back to the existing post/text delivery path instead of being silently dropped.

## Best Possible Solution

Rebase or narrowly recreate the proof-positive preflight fix so every Feishu static-card decision keeps card delivery at five or fewer tables and routes larger replies through the existing post/text conversion path, with focused tests for reply, outbound, explicit card mode, auto mode, and thread routing.

## Reproduction Assessment

Yes. Current main clearly selects static cards for table-bearing replies without a table-count guard, and multiple production reports plus the linked PR’s evidence establish rejection above five table components.

## Solution Assessment

Yes, after conflict resolution. A table-count preflight at all static-card selection boundaries is narrower and more deterministic than image rendering or a broad catch-all retry, while reusing the established post/text delivery path.
