- `impact:message-loss`: The reported failure drops the Feishu reply after the agent has produced it, leaving the chat without the useful response.

## Summary

Keep open: current main and the latest release still route markdown-table-heavy Feishu replies through interactive-card paths without a table-count guard, and the available fix paths are open or unmerged.

## What This Changes

The issue asks Feishu delivery to avoid silently dropping replies when markdown tables exceed the card table limit, preferably by falling back to the existing text/post rendering path.

## Best Possible Solution

Land or recreate a narrow Feishu plugin guard/fallback so replies with more than five markdown tables use the existing post/text conversion path while preserving reply and thread routing; keep streaming CardKit behavior separate unless it is proven to share the same limit.

## Reproduction Assessment

Yes, at source level: current main still routes markdown-table content through Feishu card paths without a table-count guard, and the issue plus linked PRs provide 230099/11310 logs. I did not run a live Feishu API reproduction in this read-only review.

## Solution Assessment

Yes, the best fix direction is a narrow preflight/fallback to the existing post/text table conversion path. Rendering tables as images or catching all card failures would be broader and more likely to hide unrelated Feishu card errors.
