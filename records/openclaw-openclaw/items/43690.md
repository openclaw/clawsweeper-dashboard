## Summary

Keep open: current main and the latest release still route table-heavy Feishu replies into interactive-card paths without a five-table guard or 230099/11310 fallback. The active same-repo implementation candidate is https://github.com/openclaw/openclaw/pull/86901, but it is still open and conflicting, so this issue should remain the canonical tracker until a rebased fix lands or maintainers choose a replacement.

## What This Changes

The issue requests Feishu to avoid silent message loss when markdown-table-heavy replies exceed the interactive-card table limit, preferably by falling back to text/post mode or converting tables before card delivery.

## Best Possible Solution

Land or replace the narrow Feishu static-card table-limit fallback against current main, with focused outbound and reply-dispatcher tests, while tracking streaming CardKit behavior separately unless it is proven to share the same limit.

## Reproduction Assessment

Yes, at source level: current main still routes markdown-table replies to Feishu card delivery without a table-count guard, and the issue plus linked duplicate reports include 230099/11310 failure logs. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes for the core static-card problem: a narrow preflight or fallback to the existing post/text path is the most maintainable fix because sendMessageFeishu already converts markdown tables. Rendering tables as images would add a new media-rendering surface, and streaming CardKit should stay separate unless maintainers confirm the same API limit.
