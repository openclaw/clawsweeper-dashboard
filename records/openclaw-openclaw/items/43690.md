## Summary

Current main and v2026.6.5 still route table-heavy Feishu replies through interactive-card paths without enforcing Feishu's five-table card limit, so the reported message loss remains actionable. The best current fix path is the open upstream PR https://github.com/openclaw/openclaw/pull/86901, but it is still unmerged and conflicting, and the open fork PR with closing syntax is also unmerged.

## What This Changes

Feishu replies with too many Markdown tables should fall back to deliverable text/post output instead of silently failing when interactive-card creation is rejected.

## Best Possible Solution

Land a narrow static-card preflight or equivalent fallback that keeps every Feishu card send at five or fewer Markdown tables and routes larger replies through the existing post/text conversion path, with focused reply/outbound tests and preserved thread routing.

## Reproduction Assessment

Yes, at source level. Current main selects static Feishu cards for Markdown-table replies without a table-count guard, and the official Feishu docs plus linked production proof establish rejection above five table components; I did not run a live Feishu setup in this read-only review.

## Solution Assessment

Yes, the open PR's preflight-and-post fallback direction is the best known fix after conflict resolution. It is narrower than rendering tables as images or retrying every card error, and it reuses the existing Feishu post/text path that already converts Markdown tables.
