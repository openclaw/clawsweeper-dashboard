## Label Justifications

- `P2`: This is a bounded Feishu channel delivery bug where generated responses can fail to reach the user when table-heavy content exceeds the platform card limit.
- `impact:message-loss`: The reported failure drops or replaces the actual Feishu reply content after the agent has produced it, leaving the chat without the useful response.

## Summary

Keep open: current main and v2026.6.6 still route Feishu markdown-table replies into static interactive cards without a five-table guard, so the reported message-loss bug remains. The fix path is already represented by https://github.com/openclaw/openclaw/pull/86901 and an older open closing fork PR, but neither has merged.

## What This Changes

Feishu messages containing too many markdown tables should avoid or recover from interactive-card table-limit failures and still deliver the response through the existing post/text path or another safe fallback.

## Best Possible Solution

Land or narrowly recreate the static-card preflight fix so Feishu reply and outbound card decisions keep interactive cards at five or fewer markdown tables and route larger content through the existing post/text conversion path with focused tests for auto mode, explicit card mode, and reply/thread routing.

## Reproduction Assessment

Yes, at source level: current main selects Feishu interactive cards for explicit card mode or auto-detected markdown tables without a five-table guard, and the issue/linked PR logs plus official Feishu documentation establish the over-limit rejection. I did not run a live Feishu repro in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow preflight at the existing static-card selection boundaries that falls back to the already supported post/text table conversion path. Image rendering or a broad catch-all retry would be larger and more likely to hide unrelated card failures.
