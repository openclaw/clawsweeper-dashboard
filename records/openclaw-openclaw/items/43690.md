## What This Changes

The issue asks OpenClaw to avoid silently dropping Feishu replies when interactive card creation fails on Feishu's table-count limit.

## Best Possible Solution

Land or replace a coordinated Feishu fix that classifies the 230099/11310 card table-limit failure and retries affected reply-dispatcher card delivery through the existing plain-text table-conversion path.

## Reproduction Assessment

Yes, source-level. The issue includes concrete Feishu 230099/11310 logs, and current main still routes markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit catch or retry; I did not run a live Feishu send.

## Solution Assessment

Yes for a narrow table-limit-specific plain-text fallback in the reply dispatcher; no for making table-image rendering the required bug fix. The maintainable path is to reuse sendMessageFeishu and existing markdown-table conversion for the affected failure only.
