## Summary

Keep open. Current main and the latest release still send Feishu table-heavy auto/card replies through interactive-card delivery without a 230099/11310 table-limit classifier or plain-text retry, and https://github.com/openclaw/openclaw/pull/50429 directly closes this issue while https://github.com/openclaw/openclaw/pull/73651 covers only the related outbound surface.

## What This Changes

Request to make Feishu recover from interactive-card table-limit failures by falling back to text/table conversion or another deliverable representation.

## Best Possible Solution

Land or replace a coordinated Feishu plugin fix that classifies 230099/11310 table-limit failures and retries affected final reply and outbound card delivery through the existing plain-text/table-conversion path; table-to-image rendering can remain a separate enhancement.

## Reproduction Assessment

Yes, source-level. The issue includes concrete Feishu 230099/11310 logs, and current main still routes Markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

Yes for the fallback direction, but it is not implemented on current main. The narrow maintainable fix is to classify the Feishu table-limit error and reuse the existing sendMessageFeishu markdown-table conversion path rather than making image rendering part of the bug fix.
