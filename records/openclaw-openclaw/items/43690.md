## Summary

Keep open: current main still sends Feishu Markdown-table replies through the structured-card path without a 230099/11310 table-limit classifier or plain-text retry, and an open closing PR already targets this reply-dispatcher bug.

## What This Changes

The issue requests Feishu delivery to fall back or otherwise render safely when card creation fails with the platform table-count limit.

## Best Possible Solution

Land or replace a coordinated Feishu fix that classifies 230099/11310 card table-limit failures and retries only affected reply/card delivery through the existing plain-text table-conversion path.

## Reproduction Assessment

Yes, source-level. The reporter provided Feishu 230099/11310 logs, and current main still routes Markdown-table auto/card replies to sendStructuredCardFeishu without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

Yes for a table-limit-specific fallback; no for making table-image rendering part of the required bug fix. The narrow maintainable path is to reuse sendMessageFeishu and existing Markdown-table conversion for the affected failure only.
