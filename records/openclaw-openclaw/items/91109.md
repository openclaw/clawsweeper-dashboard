## Summary

Keep open: the linked Xiaomi provider bug is real, but this branch is still incomplete because it misses the documented Xiaomi base hosts/custom-provider path, leaves Xiaomi on OpenAI-style thinking payload defaults, and still has only mock/unit-test proof.

## What This Changes

The PR adds Xiaomi provider-id/base-URL detection in `src/llm/providers/openai-completions.ts` so provider-layer `detectCompat` sets `requiresReasoningContentOnAssistantMessages` for Xiaomi as well as DeepSeek.

## Best Possible Solution

Use the same endpoint-attribution-backed Xiaomi compat defaults for both provider and agent paths, add focused provider payload tests for pay-as-you-go, Token Plan, and custom-base-url Xiaomi routes, then require redacted live Xiaomi multi-turn proof before merge.

## Reproduction Assessment

Yes for source-level reproduction, but not live API reproduction: current main has separate provider and agent compat paths, and the provider path lacks Xiaomi endpoint/default handling while the agent path and docs identify Xiaomi-native routes. A live Xiaomi multi-turn session has not been proven.

## Solution Assessment

No: this is a plausible narrow mitigation, but the best fix should align the provider-layer Xiaomi detection/defaults with the endpoint-attribution-backed agent compat rather than adding one stale host check and one replay flag.
