## Summary

Keep open: current main still delegates WhatsApp account startup to plugin code that uses Baileys multi-file auth state without a yielding/chunking wrapper, and the direct fix PR was closed unmerged.

## What This Changes

The issue requests that WhatsApp account startup stop monopolizing the Node event loop during Baileys auth-state and signal-store bootstrap so reconnects do not cascade.

## Best Possible Solution

Fix the WhatsApp plugin bootstrap/auth path to yield or offload large Baileys auth-state and key rehydrate work, while leaving the broader production auth-store replacement to https://github.com/openclaw/openclaw/issues/9544.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The reporter supplied concrete logs and source inspection maps the phase to the WhatsApp plugin/Baileys auth path, but I did not rerun a large paired WhatsApp session on Windows.

## Solution Assessment

Yes for the repair boundary: the narrow maintainable fix belongs in the WhatsApp plugin auth/socket bootstrap, not in the generic channel manager. The implementation should prove large auth or signal-key work stops monopolizing the event loop and should avoid the full auth-store migration unless maintainers redirect it.
