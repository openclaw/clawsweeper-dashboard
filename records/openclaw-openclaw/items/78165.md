## Summary

Keep open: current main and the latest release still route the reported `channels.whatsapp.start-account` phase into the WhatsApp plugin's Baileys multi-file auth/socket bootstrap without the signal-key yielding wrapper proposed by the closed direct fix PR. This issue is the canonical tracker for the WhatsApp start-account stall; related reports either point back here or track separate follow-up symptoms.

## What This Changes

The issue requests a fix for WhatsApp account startup blocking the Node event loop long enough to trigger liveness warnings and reconnect storms.

## Best Possible Solution

Fix the WhatsApp plugin bootstrap/auth path to yield or offload large Baileys auth-state and signal-key rehydrate work, while leaving the broader durable auth-store replacement to the separate auth-store tracker.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The reporter supplied concrete liveness logs, and source plus upstream Baileys inspection maps the diagnostic phase to a still-unfixed WhatsApp auth/socket bootstrap path.

## Solution Assessment

Yes for the repair boundary: the narrow maintainable fix belongs in the WhatsApp plugin auth/socket bootstrap, not in the generic gateway phase wrapper. A full SQL/durable auth-store migration should remain separate unless maintainers explicitly redirect the scope.
