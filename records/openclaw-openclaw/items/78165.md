## Summary

Keep open. Current main and latest release still route WhatsApp startup through Baileys multi-file auth and signal-key reads without the proposed yielding wrapper or a replacement auth store, while the only linked fix PR closed unmerged and the broader auth-store tracker remains open.

## What This Changes

The issue requests that WhatsApp account startup stop blocking the Node event loop during Baileys auth/signal-key bootstrap so reconnects do not cascade.

## Best Possible Solution

Land a WhatsApp plugin-owned startup/auth-state fix that prevents long event-loop stalls during reconnect, aligns with the broader auth-store migration plan, and includes live WhatsApp reconnect proof plus focused session/auth regression tests.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection still traces WhatsApp account startup into Baileys multi-file auth and unchunked signal-key reads, and the issue includes concrete liveness logs from v2026.5.4.

## Solution Assessment

Unclear. The closed yielding PR is a plausible mitigation, but the best durable fix may be coordinated with the broader WhatsApp auth-store replacement and needs maintainer/security direction first.
