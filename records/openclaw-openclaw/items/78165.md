## Summary

Keep open. Current main and v2026.6.10 still route WhatsApp startup through Baileys multi-file auth and cacheable signal-key reads without a batching/yielding wrapper or replacement auth store, while the direct yielding PR closed unmerged and the broader auth-store work remains open.

## What This Changes

The issue asks for the WhatsApp account startup path to stop blocking the Node event loop long enough to cause liveness warnings, missed keepalives, and reconnect storms.

## Best Possible Solution

Land a WhatsApp plugin-owned startup/auth-state fix that prevents long event-loop stalls during reconnect, coordinates with the broader auth-store plan, and includes live WhatsApp reconnect proof plus focused session/auth regression tests.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source inspection still traces WhatsApp account startup into Baileys multi-file auth and signal-key reads, and the issue provides concrete v2026.5.4 liveness logs.

## Solution Assessment

Unclear. Chunking/yielding large signal-key reads is a plausible mitigation, but the durable fix may need to align with the broader WhatsApp auth-store replacement and security/migration plan.
