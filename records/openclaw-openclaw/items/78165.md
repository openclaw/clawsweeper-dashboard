## Summary

Keep open. Current main and latest release still initialize WhatsApp through Baileys multi-file auth and signal-key reads, the linked yielding fix was closed unmerged, and the broader auth-store tracker does not by itself prove this reconnect-stall bug is fixed.

## What This Changes

The issue asks OpenClaw to stop WhatsApp `startAccount` from blocking the Node event loop long enough to trigger liveness warnings and reconnect storms.

## Best Possible Solution

Choose the WhatsApp auth/startup fix shape, then change the plugin so large auth or signal-key stores do not monopolize the event loop and prove a real reconnect completes without liveness warnings or reconnect churn.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection traces the reported startup phase into the same Baileys multi-file auth and signal-key read path, and the issue includes concrete v2026.5.4 liveness logs.

## Solution Assessment

Unclear. The closed batching/yielding PR is a plausible mitigation, but the best permanent fix may be the broader plugin-owned auth-store replacement tracked separately and needs maintainer/security direction.
