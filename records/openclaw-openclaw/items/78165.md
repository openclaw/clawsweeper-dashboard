## Summary

Keep open: current main still routes WhatsApp startup through the plugin path that uses Baileys multi-file auth state without a yielding/chunking wrapper around large signal-key reads, and the direct fix pull request was closed unmerged. The related open registry/transport performance pull request may reduce adjacent reconnect pressure, but it does not touch the Baileys auth/key bootstrap path reported here.

## What This Changes

Fix WhatsApp startup so large Baileys auth/signal-store bootstrap work does not block the Node event loop long enough to trigger liveness warnings and reconnect storms.

## Best Possible Solution

Fix the WhatsApp plugin auth/bootstrap path so large Baileys signal-key reads or auth replay yield or are offloaded, with a focused large-store regression, while leaving the full auth-store replacement to https://github.com/openclaw/openclaw/issues/9544.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The report supplies concrete liveness logs and source inspection maps the phase to the current WhatsApp plugin/Baileys auth path, but this review did not rerun a large paired WhatsApp session.

## Solution Assessment

Yes on the repair boundary: the narrow maintainable fix belongs in the WhatsApp plugin auth/socket bootstrap, not in the generic channel manager wrapper. The implementation should prove that large signal-key reads or auth replay stop monopolizing the event loop without changing the broader auth-store design.
