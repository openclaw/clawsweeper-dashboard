## Summary

Keep open. Current main and v2026.6.10 still start WhatsApp accounts through Baileys multi-file auth and cacheable signal-key reads, the direct yielding fix closed unmerged, and the remaining repair touches WhatsApp auth/session state enough to need maintainer strategy and live proof.

## What This Changes

The issue reports WhatsApp account startup blocking the Node event loop for 30-40s during Baileys auth/signal-store bootstrap, causing keepalive timeouts and reconnect storms.

## Best Possible Solution

Land a WhatsApp plugin-owned startup/auth-state fix that prevents long event-loop stalls during reconnect, aligns with the broader auth-store and sync-FS work, and includes live WhatsApp reconnect proof plus focused session/auth regression tests.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection still traces WhatsApp account startup into Baileys multi-file auth and signal-key reads, and the issue provides concrete v2026.5.4 liveness logs.

## Solution Assessment

Unclear. Chunking or yielding large signal-key reads is a plausible mitigation, but the durable fix may need to align with the broader WhatsApp auth-store replacement and security/migration plan.
