## Summary

Keep open. Current main and the latest release still route WhatsApp startup through Baileys multi-file auth and signal-key reads, the narrow fix at https://github.com/openclaw/openclaw/pull/78178 was closed unmerged, and the broader open auth-store tracker does not by itself prove this event-loop stall is fixed.

## What This Changes

The issue reports WhatsApp account startup blocking the Node event loop for 30-40 seconds during Baileys auth/signal-store bootstrap, causing keepalive timeouts and reconnect storms.

## Best Possible Solution

Fix the WhatsApp plugin’s auth/signal-key startup path so large stores do not monopolize the event loop, then verify a real reconnect completes without liveness warnings or keepalive-triggered reconnect churn.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows current main and v2026.6.6 still exercise the reported Baileys multi-file auth and signal-key path, and the issue includes concrete v2026.5.4 liveness logs.

## Solution Assessment

Unclear whether the closed chunking patch is the best permanent fix. A narrow yielding wrapper may mitigate the stall, but the broader production auth-store direction at https://github.com/openclaw/openclaw/issues/9544 needs maintainer and security design before becoming the canonical solution.
