## Summary

Keep open: the report is new, concrete, and maps to the current WhatsApp startup path; I did not find an implemented fix on current main.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp account startup from monopolizing the Node event loop during Baileys auth/session bootstrap and causing reconnect storms.

## Best Possible Solution

Make the WhatsApp plugin’s auth/bootstrap path yield or offload heavy Baileys key replay so start-account cannot monopolize the gateway event loop, then cover it with a large-store regression and real Windows proof if available.

## Reproduction Assessment

No, not as a live current-main reproduction in this review. The issue provides a concrete 40s liveness diagnostic and current source maps that phase to WhatsApp plugin startup, but a large paired WhatsApp session was not available to rerun it.

## Solution Assessment

Unclear on the exact mechanism, but the best boundary is the WhatsApp plugin auth/socket bootstrap rather than the generic core channel manager. A maintainable fix should preserve the existing plugin contract and avoid adding timing knobs for a CPU-bound startup stall.
