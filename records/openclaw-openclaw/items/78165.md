## Summary

Keep open: current main and the latest release still route the reported `channels.whatsapp.start-account` phase into the WhatsApp plugin's Baileys multi-file auth/socket bootstrap without a yielding/offload wrapper for large auth-state or signal-key reads. The closed direct fix PR was not merged, and the open registry/discovery performance PR is adjacent rather than a clear replacement for the Baileys auth bootstrap stall.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp `start-account` from blocking the Node event loop long enough to trigger liveness warnings and reconnect storms.

## Best Possible Solution

Fix the WhatsApp plugin bootstrap/auth path so large Baileys auth-state and signal-key rehydrate work yields or moves off the event loop, while leaving the broader durable auth-store replacement to https://github.com/openclaw/openclaw/issues/9544.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The reporter supplied concrete liveness logs, and source plus upstream Baileys inspection maps the diagnostic phase to a still-present WhatsApp auth/socket bootstrap path.

## Solution Assessment

Yes for the repair boundary: the narrow maintainable fix belongs in the WhatsApp plugin auth/socket bootstrap, not in the generic gateway phase wrapper. A full SQL/durable auth-store migration should remain separate unless maintainers explicitly redirect the scope.
