- `impact:security`: The reported failure mode crosses the tool-execution boundary by treating reasoning-derived content as actionable host tool intent.

## Summary

Keep open. Current main still turns structured OpenAI-compatible `delta.tool_calls` into actionable `toolCall` blocks regardless of the final `finish_reason`, and the linked closing fix PR is open and conflicting rather than merged.

## What This Changes

The issue asks OpenClaw to ignore tool-call XML that appears only in model reasoning/thinking output so it cannot be executed as a real tool call.

## Best Possible Solution

Review, repair if needed, and land https://github.com/openclaw/openclaw/pull/85248 or an equivalent focused fix that strips tool-call blocks unless the provider signals tool-use intent, with regression coverage for `stop` and `length` finish reasons.

## Reproduction Assessment

No high-confidence live reproduction is attached for the llama.cpp/Qwen setup. There is a clear source-level reproduction path for a streamed non-empty `delta.tool_calls` sequence followed by `finish_reason: "stop"` or `"length"`, which current main would leave as actionable `toolCall` blocks.

## Solution Assessment

Mostly yes. Ignoring reasoning-only tool syntax is the right boundary, and the linked PR’s finish-reason guard is the narrow likely fix, but maintainers still need to review the compatibility tradeoff for OpenAI-compatible providers that emit tool deltas without a tool-call finish reason.
