## Label Justifications

- `P2`: This is a real memory-search usability failure for a specific remote embedding configuration, with limited blast radius and a product/API decision still needed.
- `impact:session-state`: The reported failure happens during post-compaction session indexing and can leave memory search out of sync with session transcripts.
- `impact:auth-provider`: The issue centers on embedding provider credential fallback, bearer headers, and SecretRef/API-key behavior for `memorySearch.remote`.

## Summary

Keep open: current main still cannot create a no-auth OpenAI-compatible memory embedding client when `memorySearch.remote.apiKey` is absent. Source also shows `remote.baseUrl` is still preferred, so the remaining issue is the auth/header contract rather than a proven fallback to the OpenAI URL.

## What This Changes

The issue requests support for self-hosted OpenAI-compatible memory embeddings without `memorySearch.remote.apiKey`, including post-compaction session sync.

## Best Possible Solution

Define an explicit no-auth or header-only contract for OpenAI-compatible memory endpoints, or land the plugin-owned OpenAI-compatible provider path after boundary review while preserving existing key inheritance.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a focused reproduction of the no-auth gap: absent `remote.apiKey` cannot produce a no-auth request because the shared client falls back to provider auth and always emits a bearer header.

## Solution Assessment

Unclear as stated. Blanket no-auth whenever `remote.baseUrl` is set would be upgrade-sensitive; an explicit no-auth/header-only mode or the plugin-owned `openai-compatible` provider path is the safer maintainable direction.
