## Label Justifications

- `P2`: This is a user-facing memory sync failure for a specific remote embedding configuration, with limited blast radius and no core runtime outage.
- `impact:session-state`: The reported failure occurs during post-compaction session transcript indexing, which can leave memory search out of sync with session state.
- `impact:auth-provider`: The issue centers on memory embedding provider auth, credential fallback, and endpoint routing behavior for `memorySearch.remote`.

## Summary

Keep open: current main still requires a resolved bearer credential before the shared remote embedding client can return, while post-compaction session indexing uses that same memory sync path. The remaining decision is the supported contract for unauthenticated or header-only OpenAI-compatible embedding endpoints, because current docs say remote embeddings require an API key and existing configs may rely on provider-key inheritance.

## What This Changes

The issue requests support for post-compaction memory embedding sync when `memorySearch.remote.baseUrl` points at an unauthenticated OpenAI-compatible endpoint and `memorySearch.remote.apiKey` is unset.

## Best Possible Solution

Define an explicit no-auth or header-only contract for OpenAI-compatible memory endpoints, preserve existing provider-key inheritance for current configs, and add post-compaction/session-sync coverage for the chosen behavior.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a focused reproduction path for the credential failure: omit `memorySearch.remote.apiKey`, use the OpenAI memory embedding path, and the shared client still requires provider auth before post-compaction sync can embed.

## Solution Assessment

Unclear as stated. Allowing no auth whenever `remote.baseUrl` is set is likely too broad; an explicit no-auth/header-only mode or the related OpenAI-compatible provider direction is safer because current docs and some configs rely on API-key inheritance.
