## Label Justifications

- `P2`: This is a real user-facing memory-search sync failure for a specific remote embedding configuration, but it is not a core runtime outage.
- `impact:auth-provider`: The issue centers on memory embedding provider auth and credential fallback behavior for `memorySearch.remote`.
- `impact:session-state`: The reported failure occurs during post-compaction session transcript indexing, which can leave memory search out of sync with session state.

## Summary

Keep open: current main still forces bearer auth for the shared OpenAI-compatible remote embedding client when `memorySearch.remote.apiKey` is absent, and post-compaction session indexing reaches that same path. The remaining maintainer decision is how to support unauthenticated or header-only OpenAI-compatible embedding endpoints without breaking existing key-inheriting remote configs.

## What This Changes

The issue requests that post-compaction memory embedding sync use a configured OpenAI-compatible `memorySearch.remote.baseUrl` without requiring or falling back to OpenAI credentials.

## Best Possible Solution

Define the supported no-auth or header-only contract for OpenAI-compatible memory endpoints, then implement it in the shared remote embedding client with post-compaction session-sync coverage.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a focused reproduction path for the credential failure: omit `memorySearch.remote.apiKey`, set provider `openai`, and the shared client still requires provider auth before post-compaction sync can embed.

## Solution Assessment

Unclear as stated. Allowing no-auth whenever `remote.baseUrl` is set is probably too broad; a documented explicit no-auth or header-only mode is safer because current docs say remote embeddings require an API key.
