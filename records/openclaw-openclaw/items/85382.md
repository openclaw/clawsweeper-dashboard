## Label Justifications

- `P2`: This is a real memory sync usability failure for a specific remote embedding configuration, but it is not a core runtime outage.
- `impact:session-state`: The reported failure happens during post-compaction session transcript indexing and can leave memory search out of sync with session state.
- `impact:auth-provider`: The issue centers on embedding provider credential fallback, `remote.apiKey`, and auth header behavior.

## Summary

Keep open: current main still requires a memory embedding API key or inherited provider key before building remote embedding requests, while docs do not define unauthenticated OpenAI-compatible memory endpoints. The source-level failure path is clear, but the fix needs a product/config contract decision rather than a safe autonomous bug patch.

## What This Changes

The issue requests support for post-compaction memory embedding sync against a self-hosted OpenAI-compatible `memorySearch.remote.baseUrl` when `remote.apiKey` is omitted.

## Best Possible Solution

Define and document the supported no-auth or header-only OpenAI-compatible memory endpoint path, then cover the shared remote embedding client and post-compaction session sync with focused regression tests.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows a focused current-main path: with `memorySearch.remote.baseUrl` set and no `remote.apiKey` or provider key, the shared client calls `requireApiKey` before any request; the exact 500/fallback-to-OpenAI part is not proven.

## Solution Assessment

Unclear as stated. Treating every missing `remote.apiKey` as no-auth would risk breaking configs that intentionally inherit provider credentials, so an explicit no-auth/header-only contract or the generic OpenAI-compatible provider path is safer.
