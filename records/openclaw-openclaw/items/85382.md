## Label Justifications

- `P2`: This is a real user-facing memory-sync/provider-auth gap for a specific remote embedding configuration, but it is not a core runtime outage.
- `impact:session-state`: The failure occurs during post-compaction session indexing and can leave session memory search out of sync.
- `impact:auth-provider`: The issue centers on provider credential fallback and bearer-header behavior for `memorySearch.remote` embedding requests.

## Summary

Keep open: current main still requires a resolved API key before the shared OpenAI remote embedding client can be used, and post-compaction session sync reaches that same embedding path. The exact claim that the URL falls back to OpenAI is not source-confirmed, because `remote.baseUrl` is preferred, but the no-auth/header-only contract remains unresolved.

## What This Changes

The issue requests support for post-compaction memory embedding sync against a self-hosted OpenAI-compatible `memorySearch.remote.baseUrl` when `memorySearch.remote.apiKey` is omitted.

## Best Possible Solution

Define a maintainer-approved no-auth or header-only contract for OpenAI-compatible memory endpoints, then align the remote embedding client, docs, and post-compaction coverage with that contract.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a focused reproduction path for the auth failure: omit `memorySearch.remote.apiKey`, use the OpenAI memory embedding provider, and the shared client still requires provider auth before post-compaction sync can embed.

## Solution Assessment

Unclear as stated. Automatically treating every `remote.baseUrl` without `remote.apiKey` as unauthenticated is compatibility-sensitive; an explicit no-auth/header-only contract or the generic OpenAI-compatible provider path is safer.
