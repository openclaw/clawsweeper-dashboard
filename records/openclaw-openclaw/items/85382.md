## Label Justifications

- `P2`: This is a user-facing memory embedding sync failure for a specific remote endpoint configuration, but not a full core runtime outage.
- `impact:session-state`: The failing path is post-compaction session transcript indexing, which can leave memory search stale relative to session state.
- `impact:auth-provider`: The issue centers on embedding provider credential fallback, auth headers, and SecretRef/provider-key behavior for `memorySearch.remote`.

## Summary

Keep open: current main and the latest shipped release still require a bearer credential when `memorySearch.remote.apiKey` is absent, while related OpenAI-compatible embedding work is still open and unmerged. The remaining decision is the supported contract for unauthenticated or header-only OpenAI-compatible memory endpoints.

## What This Changes

The issue asks OpenClaw to support post-compaction memory embedding sync against a self-hosted OpenAI-compatible `memorySearch.remote.baseUrl` when no `remote.apiKey` is configured.

## Best Possible Solution

Define an explicit no-auth or header-only remote embedding contract, preserve existing provider-key inheritance unless maintainers intentionally change it, then update the shared client, docs, and regression tests for post-compaction session sync.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a focused current-main path: configure `memorySearch.provider: "openai"` with `remote.baseUrl` and no `remote.apiKey`, then post-compaction sync reaches a shared client that requires provider auth before embedding.

## Solution Assessment

Unclear as reported. A blanket no-auth fallback whenever `remote.baseUrl` is set is probably not the best fix because current docs and existing configs support provider-key inheritance; an explicit no-auth/header-only contract is safer.
