## Label Justifications

- `P2`: This is a real user-facing memory indexing failure for a specific remote embedding configuration, but not a core runtime outage.
- `impact:session-state`: The failure occurs during post-compaction session transcript indexing, which can leave memory search out of sync with session state.
- `impact:auth-provider`: The issue centers on memory embedding provider auth, API key fallback, and custom endpoint credential behavior.

## Summary

Keep open: current main still requires bearer auth when `memorySearch.remote.apiKey` is absent, while post-compaction session indexing reaches that embedding path. The remaining decision is the supported contract for unauthenticated or header-only OpenAI-compatible memory endpoints, especially with the related generic provider PR still open.

## What This Changes

The issue asks OpenClaw to support post-compaction memory embedding sync against a self-hosted OpenAI-compatible endpoint configured with `memorySearch.remote.baseUrl` and no API key.

## Best Possible Solution

Define an explicit no-auth or header-only OpenAI-compatible memory endpoint contract, then cover the shared remote embedding client and post-compaction session sync path with focused tests.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a focused reproduction path for the credential failure: configure `memorySearch.provider: "openai"` with `remote.baseUrl` and no `remote.apiKey`, then post-compaction session sync reaches the shared client that requires provider auth before embedding.

## Solution Assessment

Unclear as stated. Allowing no auth whenever `remote.baseUrl` is set is probably too broad; an explicit documented no-auth/header-only mode or the maintainer-approved OpenAI-compatible provider path is safer.
