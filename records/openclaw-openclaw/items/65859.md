## Summary

Keep open. Current main has partial WebChat audio support, but source inspection still shows two remaining gaps: explicit audio files with `.webm`/`.mp4` names can be filtered out during upload, and local TTS audio is emitted as base64 `type: "audio"` content blocks while the Control UI render path only turns `attachment` blocks into playable audio cards.

## What This Changes

The issue requests WebChat audio MIME upload support and inline playback for received audio attachments.

## Best Possible Solution

Land or replace the active WebChat TTS playback fixes so generated audio becomes URL-based trusted audio attachments, and separately fix the picker classifier to accept explicit `audio/*` MIME while preserving video rejection.

## Reproduction Assessment

Yes, at source level. The upload path rejects `audio/webm` or `audio/mp4` files with `.webm`/`.mp4` names, and the TTS path emits `type: "audio"` base64 content blocks that the Control UI normalizer does not convert into playable audio attachments.

## Solution Assessment

Yes, in direction. The narrow maintainable path is to honor explicit audio MIME in the upload helper and keep the playback fix at the gateway/UI attachment contract boundary rather than broadening media policy or adding configuration.
