## Summary

Keep open. Current main now covers the upload filter and generic inline audio rendering, but the issue's auto-TTS WebChat path is still not fully wired because auto-TTS payloads do not set the internal local-media trust flag required by the WebChat audio embedding path.

## What This Changes

Add WebChat audio uploads and inline playback for audio attachments, including generated TTS audio replies.

## Best Possible Solution

Keep the existing upload/player implementation, then land or replace the narrow auto-TTS trust-flag fix with real WebChat playback proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: auto-TTS payloads omit `trustedLocalMedia`, while WebChat local audio embedding requires that flag before it will expose the audio file.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable remaining fix is to preserve the current WebChat attachment/player path and add the missing trust flag to auto-TTS payloads, as the open PR already proposes.
