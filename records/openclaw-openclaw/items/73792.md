## Summary

Keep open: the reported stale MCP runtime problem is real on current main, but this PR is not merge-ready because its close handler also fires during intentional startup cleanup, current-main test imports no longer resolve, and the user-facing fix lacks a changelog entry.

## What This Changes

The PR adds MCP transport close/error callbacks to session MCP runtimes, evicts the cached runtime through the manager on transport close, logs the retirement, and adds a stdio child SIGKILL regression test.

## Best Possible Solution

Land a rebased narrow fix that distinguishes unexpected transport closes from intentional cleanup, preserves manager-local eviction, uses current test helpers, and adds the required changelog entry.

## Reproduction Assessment

Yes. The original bug is reproducible by materializing a session-scoped stdio MCP server, killing its child process, and observing current main keep returning the cached runtime; the PR regression is reproducible with a multi-server startup where one failed server is intentionally closed while another should remain usable.

## Solution Assessment

No, not as submitted. The callback-before-connect seam matches the SDK contract, but the implementation must guard intentional close() cleanup and update the stale test/changelog surfaces before it is the narrow maintainable fix.
