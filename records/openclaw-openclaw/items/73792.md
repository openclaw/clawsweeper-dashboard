## Summary

Keep open: current main still has the stale session-MCP runtime cache path, but this PR is not mergeable as submitted because the close handler also fires during intentional cleanup, the new test imports helpers that current main does not export, and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR wires MCP transport close/error callbacks into session MCP runtimes, disposes the cached runtime through the manager on close, logs runtime retirement, and adds a stdio child SIGKILL regression test.

## Best Possible Solution

Land a rebased narrow fix that evicts only unexpected transport closes, preserves multi-server startup behavior, uses current test helper boundaries, and records the user-facing MCP recovery fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes from source inspection. Current main can materialize a cached stdio MCP session, the child close path fires `onclose` and lets the SDK clear `_transport`, and a later `getOrCreate` still returns the same cached runtime because only workspace/config fingerprint are checked.

## Solution Assessment

No, not as submitted. The handler-before-connect seam matches the SDK contract, but the patch must guard intentional cleanup closes, fix the stale test imports, and add the required changelog entry before it is the narrow maintainable fix.
