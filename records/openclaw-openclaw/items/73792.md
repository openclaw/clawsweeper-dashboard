## Summary

Keep open: this PR targets a valid production-facing MCP staleness bug, and wiring before SDK connect matches the dependency contract, but the submitted patch still needs narrow fixes for intentional close handling, current-main test imports, and required changelog coverage.

## What This Changes

The PR wires MCP transport close/error callbacks into session MCP runtimes, evicts the cached runtime through the manager when a transport closes, logs the retirement, and adds a stdio child SIGKILL regression test.

## Best Possible Solution

Land a rebased narrow fix that evicts only unexpected transport closes, preserves healthy multi-server startup, adapts the current test helpers, and adds the required changelog entry.

## Reproduction Assessment

Yes. The original issue is reproducible by materializing a session-scoped stdio MCP server, killing the child process, and observing current main keep returning the cached runtime; the PR regression is reproducible with a multi-server config where one startup failure intentionally closes while another server should stay usable.

## Solution Assessment

No, not as submitted. The callback-before-connect direction is the right dependency-aware seam, but the implementation must guard intentional closes and update the stale test/changelog surface before it is the narrow maintainable fix.
