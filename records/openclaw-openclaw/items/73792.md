## Summary

Keep open. The stale cached MCP runtime bug is real on current main and the PR has the right overall direction, but the current patch is not merge-ready because its unconditional close handler also fires during intentional startup-failure cleanup and can retire the whole session runtime. The PR is focused and maintainer-modifiable, so a narrow repair is feasible.

## What This Changes

The PR wires MCP transport close/error callbacks into session MCP runtimes, routes unexpected close events through manager-level session disposal with an operator log line, and adds a SIGKILL regression test for stale stdio MCP child sessions.

## Best Possible Solution

Land a revised version that preserves unexpected transport-close eviction, guards intentional cleanup paths, adds partial-startup regression coverage, and includes the required changelog entry before merge.

## Reproduction Assessment

Yes. The reported stale-runtime bug is reproducible by materializing a session stdio MCP runtime, killing the child process, and reusing the cached runtime; the PR regression is reproducible with a multi-server config where one server fails during startup/listing while another should remain usable.

## Solution Assessment

No, not yet. Assigning the callback before connect matches the SDK chaining contract, but the handler must distinguish unexpected closure from intentional cleanup, for example by removing/marking the session before disposeSession(session) and no-oping onclose when the session is already absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard intentional transport closes before eviction:** `src/agents/pi-bundle-mcp-runtime.ts:277-279`
  - body: This handler also runs when OpenClaw intentionally closes a transport in the startup-failure catch path below. The pinned MCP Transport contract says `onclose` fires from `close()`, and `OpenClawStdioClientTransport.close()` waits for the child's `close` event; because the catch closes before deleting the session, a failed server can call `onTransportClosed` and retire the whole session runtime, aborting catalog materialization or closing healthy server sessions. Mark/remove the session before intentional close and have this handler no-op when the session is already gone.
