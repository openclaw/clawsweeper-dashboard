## Summary

Keep open. Current main and the latest release still have the reported custom-list behavior, while this PR is a small, relevant implementation patch that should get normal maintainer review rather than cleanup closure.

## What This Changes

The PR updates `parseBooleanValue` to guard custom truthy/falsy options with `Array.isArray`, compare custom string entries case-insensitively, and add unit tests for mixed-case, malformed, and mixed-type inputs.

## Best Possible Solution

Land a focused hardening patch for the shared helper with the proposed regression tests, filter custom-list entries to strings before lowercasing for clarity, and make an explicit maintainer call on syncing or deferring the browser plugin copy.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence path: `parseBooleanValue("yes", { truthy: ["YES"] })` misses because the input is lowercased but the custom set is not, and `parseBooleanValue("no", { falsy: { not: "an array" } as any })` reaches `new Set` with a non-iterable object.

## Solution Assessment

Mostly yes. The PR is the narrowest core fix for the reported utility behavior, with the safer landing version being a small cleanup that filters non-string custom entries and resolves the browser-copy scope question.
