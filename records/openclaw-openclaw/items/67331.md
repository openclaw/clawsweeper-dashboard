Full review comments:

- **[P1] Preserve ENOENT for missing sandbox reads:** `src/agents/sandbox/fs-bridge-path-safety.ts:80-84`
  - body: `openRootFile` reports ordinary missing paths as `reason: "path"` with the underlying ENOENT-style error, but this branch wraps every non-ok open as `E_FS_PERMISSION_DENIED`. Current sandbox bridge coverage expects missing reads to remain ENOENT/no-such-file, so keep the path/not-found branch unchanged and wrap only validation or true permission denials.
  - confidence: 0.9
- **[P2] Classify allowed-mount escapes as workspace_boundary:** `src/agents/fs-permission-denied.ts:22-26`
  - body: The branch emits `Sandbox path escapes allowed mounts` from the sandbox and remote bridge paths, but the classifier only recognizes `escapes sandbox root`, `outside workspace root`, and `outside sandbox root`. Those new denials will be labeled `reason=unknown`, defeating the standardized reason surface this PR is adding.
  - confidence: 0.93
- **[P1] Make the write/edit size cap an accepted policy first:** `src/agents/pi-tools.read.ts:50-51`
  - body: The new constants impose an unconditional 2 MiB ceiling on write and edit text payloads before mutation. Current file tools do not have that default cap, so existing workflows that generate larger files can start failing unless maintainers explicitly accept the limit and the PR adds the corresponding docs and upgrade/compatibility tests.
  - confidence: 0.82
