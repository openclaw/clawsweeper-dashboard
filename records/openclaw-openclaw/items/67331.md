- **[P1] Do not hard-code the mutation payload cap without a policy:** `src/agents/pi-tools.read.ts:47-48`
  - body: Existing write/edit tools do not have a 2 MiB text payload cap, but this wrapper throws before host and sandbox mutations execute. Large existing mutation workflows can fail at runtime unless maintainers explicitly accept and document the new limit or replace it with a compatible policy path.
  - confidence: 0.82
- **[P2] Keep missing sandbox reads as not-found errors:** `src/agents/sandbox/fs-bridge-path-safety.ts:80-84`
  - body: `openReadableFile` now wraps every failed boundary open in `createFsPermissionDeniedError`, including ordinary path misses. Current coverage expects missing sandbox reads to remain ENOENT/no-such-file before Docker runs, so this changes diagnostics and errno-style handling for non-existent files.
  - confidence: 0.92
- **[P2] Classify allowed-mount escapes as workspace_boundary:** `src/agents/fs-permission-denied.ts:21-25`
  - body: The classifier does not recognize `Sandbox path escapes allowed mounts`, even though the patch wraps local and remote sandbox boundary errors using that exact cause text. Those denials become `reason=unknown`, defeating the standardized reason metadata this PR is adding.
  - confidence: 0.95
- **[P3] Preserve the boundary-open failure cause:** `src/agents/apply-patch.ts:383-386`
  - body: `assertBoundaryRead` creates a synthetic `Failed boundary read ...` cause for validation failures instead of passing through the boundary-open error when available. That drops symlink, hardlink, or boundary wording the new classifier needs and can still report `reason=unknown` for actionable denials.
  - confidence: 0.78
