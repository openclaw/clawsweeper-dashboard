## Summary

Keep open: current main does not yet contain this standardization work, but the proposed patch still has concrete correctness blockers in sandbox filesystem error semantics and reason classification before it should merge.

## What This Changes

The PR adds an agent filesystem permission-denied helper, wraps apply_patch and sandbox filesystem boundary/read-only errors with structured reason metadata, adds write/edit text payload size and NUL-byte guards, and updates related tests.

## Best Possible Solution

Request changes on this PR and land it only after sandbox mount escapes classify as workspace-boundary denials, ordinary missing sandbox reads preserve not-found semantics, the payload guard limits are accepted by the relevant owner, and focused regression coverage proves host plus sandbox read/write/edit behavior.

## Reproduction Assessment

Yes. The blockers are reproducible by focused code checks: pass an Error containing "Sandbox path escapes allowed mounts" through the PR classifier, and read a missing file through SandboxFsBridge while asserting ENOENT/no-such-file semantics are preserved.

## Solution Assessment

No, not as currently proposed. Centralizing filesystem permission-denied metadata is a maintainable direction, but the patch should not merge until it preserves path-miss semantics and recognizes the existing sandbox mount-boundary message shape.

## Review Findings

Overall correctness: patch is incorrect

  - confidence: 0.91

  - body: The classifier does not recognize "Sandbox path escapes allowed mounts", but this PR wraps that exact message from local and remote sandbox bridge boundary checks. Those denials become `reason=unknown` instead of `workspace_boundary`, which defeats the standardization the helper introduces.
  - confidence: 0.94
