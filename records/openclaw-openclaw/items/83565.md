## Summary

Keep open. Current main and latest release still expose persistent session model overrides only: `sessions.patch` mutates stored session override state, `chat.send` has no request-local model field, and the ACK does not return the resolved model. This is a real product/API request, but the contract needs maintainer choice before implementation because the two proposed API shapes have different authorization, hook, idempotency, reporting, and concurrency semantics.

## What This Changes

Issue requests an atomic one-turn model override for `sessions.patch` or `chat.send` that does not persist session model state.

## Best Possible Solution

Define one additive request-local model override contract, implement it at the Gateway/session/runner boundary, and cover non-persistence, resolved-model reporting, provider/auth validation, hooks, docs, and focused protocol tests.

## Reproduction Assessment

Yes, at the source-contract level. Current main shows only persistent `sessions.patch` model changes, no `chat.send` model field, no `agent` modelOverrideSource, and no resolved-model ACK for `chat.send`.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable solution is one canonical additive protocol contract, not continuing the sticky two-patch workaround or landing the closed mixed-message PR branch.
