## Summary

Keep open. Current main and the latest shipped tag still expose persistent session model overrides only; the requested one-turn `sessions.patch` scope or `chat.send` model field is a new public Gateway/session API that needs maintainer product direction before implementation.

## What This Changes

The issue requests an additive one-turn model override API for `sessions.patch` or `chat.send` that does not persist session state and reports the resolved model for that turn.

## Best Possible Solution

Choose one additive Gateway protocol contract, then implement it at the gateway/runner boundary with schema, docs, and regression coverage proving no durable session mutation and correct resolved-model reporting.

## Reproduction Assessment

Yes at the source-contract level: current schemas and handlers show `sessions.patch` persists model changes and `chat.send` has no caller model field. A live repro is not needed because this is a missing feature/API surface rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the API shape. The maintainable direction is a single additive protocol contract with session-store non-mutation proof, not a hidden sticky-switch workaround.
