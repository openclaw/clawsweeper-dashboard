## Summary

Keep open. Current main and v2026.6.6 still expose only persistent session model overrides, and the related open issues cover broader turn limits or fallback persistence rather than this exact `sessions.patch` or `chat.send` one-turn API. The request changes public Gateway protocol, session state, and provider/auth routing semantics, so it needs maintainer API/product review before implementation.

## What This Changes

The issue requests an atomic per-turn model override for `sessions.patch` or `chat.send` that does not mutate persistent session model state and reports the resolved turn model.

## Best Possible Solution

Choose one additive Gateway contract, implement it at the session/runner boundary, and cover non-persistence, resolved-model reporting, auth/provider checks, hooks, docs, and focused protocol tests before merging.

## Reproduction Assessment

Yes, at the source-contract level. Current schemas and handlers show `sessions.patch` persists model changes and `chat.send` has no caller-supplied model field, so the requested API surface is absent on current main.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is one canonical additive protocol contract rather than extending the sticky two-patch workaround.
