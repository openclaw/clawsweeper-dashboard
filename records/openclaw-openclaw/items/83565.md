## Summary

Keep open. Current main and the latest shipped tag still only expose persistent session model overrides, while related open items cover fallback persistence or broader turn limits rather than this exact `sessions.patch`/`chat.send` one-turn API. The request changes public Gateway protocol, session state semantics, and model auth/routing behavior, so it needs maintainer product/API review instead of cleanup closure.

## What This Changes

The issue requests an atomic one-turn model override for `sessions.patch` or `chat.send` that reports the resolved model without changing persistent session state.

## Best Possible Solution

Choose one canonical public contract, then implement it at the Gateway protocol/runner boundary with SDK/docs/tests proving no durable session mutation, correct resolved-model reporting, and preserved model auth/allowlist checks.

## Reproduction Assessment

Yes at the source-contract level: current schemas and handlers show `sessions.patch` persists model changes and `chat.send` has no caller model field. A live repro is not needed to establish the missing API surface because this is a feature request, not a regression report.

## Solution Assessment

Unclear until maintainers choose the API shape. The maintainable direction is a single additive protocol contract with session-store non-mutation proof rather than continuing the two-patch sticky-switch workaround.
