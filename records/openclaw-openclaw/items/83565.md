## Summary

Keep open. Current main and the latest shipped tag still only expose persistent session model overrides, and the closest related open items cover fallback persistence or broader turn limits rather than this exact one-turn Gateway API. The request changes public protocol, session-state semantics, and provider/model authorization behavior, so it needs maintainer API review rather than cleanup closure.

## What This Changes

Request to add an atomic single-turn model override through `sessions.patch` or `chat.send` without persisting session state.

## Best Possible Solution

Choose one additive public Gateway contract, then implement it at the session/runner boundary with SDK/docs/tests proving no durable session mutation, correct resolved-model reporting, and preserved auth/allowlist checks.

## Reproduction Assessment

Yes, at the source-contract level: current schemas and handlers show `sessions.patch` persists model changes and `chat.send` has no caller model field. A live repro is not needed to establish the missing API surface because this is a feature request, not a regression report.

## Solution Assessment

Unclear until maintainers choose the API shape. The maintainable direction is a single additive protocol contract with session-store non-mutation proof rather than continuing the two-patch sticky-switch workaround.
