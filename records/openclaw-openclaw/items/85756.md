## Summary

Keep open: the PR targets a real protected-avatar failure, but the proposed fix moves reusable gateway credentials into browser-visible avatar URLs and opens the avatar endpoint to query-token auth, which conflicts with the existing bearer-header plus blob-URL contract.

## What This Changes

The PR appends a gateway token query parameter to local Control UI avatar URLs and enables query-token auth on the gateway avatar route.

## Best Possible Solution

Fix the avatar preview by reusing the existing authenticated fetch-to-blob flow or the related blob-path PR, keeping `/avatar/<agentId>` bearer-authenticated and avoiding reusable gateway credentials in URLs.

## Reproduction Assessment

Yes, source-reproducible for the protected-route class: current main can pass a protected `/avatar/<agentId>` URL to Quick Settings while the documented route requires authenticated fetches. I did not run a live Tailscale browser reproduction because this review is read-only.

## Solution Assessment

No. The maintainable fix is to reuse the existing bearer-authenticated fetch-to-blob path rather than adding long-lived credential query parameters to avatar image URLs.
