## Summary

Keep this PR open, but it is not merge-ready: the revised branch removed the rendered tokenized image URL but still broadens `/avatar` to accept reusable query-token credentials, while current code and docs already point avatar rendering through bearer-header fetches and local blob URLs.

## What This Changes

The PR adds `allowQueryToken` plumbing to the Control UI avatar handler and enables query-token auth for `/avatar`, with a stale helper left from the reverted frontend token-in-URL approach.

## Best Possible Solution

Keep avatar rendering on bearer-header fetches plus local blob URLs, or use a short-lived avatar ticket similar to assistant media if a browser-native image URL is truly required.

## Reproduction Assessment

No, not for the full Tailscale UI failure in a real browser. Source inspection does confirm the narrower query-token symptom: current avatar auth ignores `?token=`, while the existing chat avatar path already fetches with bearer headers and blob URLs.

## Solution Assessment

No. Enabling reusable query-token auth on the avatar route is not the safest fix for a browser rendering problem; the maintainable path is bearer-header fetches with blob URLs or a short-lived ticketed URL if native image loading is required.
