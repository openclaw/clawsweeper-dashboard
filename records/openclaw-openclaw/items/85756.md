Keep open and do not merge as-is: the final diff still enables reusable query-string token auth on the protected avatar route, while current main already defines the safe Control UI path as bearer-header fetches converted to blob URLs. The PR adds an `allowQueryToken` option to the Control UI avatar handler, enables it for the gateway `/avatar` route, and leaves `ui/src/ui/app-render.ts` with formatting-only changes.

## Best Possible Solution

Keep `/avatar` on the documented bearer-header plus blob URL path, diagnose the actual Tailscale request that drops auth, and only consider browser-native avatar URLs through a scoped short-lived ticket contract if live proof shows headers cannot work.

No high-confidence full reproduction is available for the Tailscale UI failure. Source inspection confirms current main rejects avatar `?token=`, but current docs define bearer/blob auth, and the PR only includes pre-fix curl output rather than after-fix browser proof.

## Solution Assessment

No. Backend query-token acceptance is not the best fix for a documented bearer/blob avatar flow; the safer path is to repair the actual frontend auth request or introduce a scoped short-lived avatar ticket after maintainer approval.
