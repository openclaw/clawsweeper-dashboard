## Summary

Keep open, but this PR is not merge-ready: current main already implements the documented bearer-header/blob avatar flow, while the remaining diff exposes reusable query-token auth on `/avatar/<agentId>` and does not change the URL path that would make a direct unauthenticated image request succeed.

## What This Changes

This PR adds an optional query-token fallback to the Control UI avatar handler, enables it for the gateway avatar route, and only reformats the Control UI avatar route call site.

## Best Possible Solution

Keep the avatar route on bearer-authenticated fetch/blob rendering, or introduce a short-lived scoped avatar ticket if a native image URL path is truly needed; do not authorize avatars with reusable gateway or device tokens in query strings.

## Reproduction Assessment

No. The linked curl output proves unauthenticated avatar requests return 401 and bearer requests return 200, which matches docs and tests; the screenshots show rendering but not the failing current-main Tailscale request or the after-fix network path.

## Solution Assessment

No. Enabling reusable query-token auth for avatars widens a protected auth surface and does not change the Control UI's direct image URL path; the safer fix is bearer/blob rendering or a scoped short-lived ticket if a native image URL is required.
