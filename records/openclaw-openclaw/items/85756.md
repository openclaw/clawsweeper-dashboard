## Summary

Keep open: the reported avatar-auth problem is plausible, but this PR changes the auth boundary by putting gateway tokens in avatar URLs and lacks after-fix real UI/Tailscale proof.

## What This Changes

The PR forwards query-token auth into the Control UI avatar route and appends the current Control UI auth token to local assistant avatar URLs.

## Best Possible Solution

Keep avatar auth on bearer-header fetches plus local blob URLs, and fix any remaining Tailscale avatar surface by reusing that authenticated fetch path with focused coverage.

## Reproduction Assessment

No, not as a live Tailscale reproduction. Source inspection shows a direct Quick Settings avatar URL can render without headers, while the current chat avatar path already fetches with bearer headers and converts to a blob URL.

## Solution Assessment

No. The PR’s query-token approach conflicts with the documented and tested bearer-header/blob contract; the safer fix is to extend that existing authenticated fetch path to the affected rendering surface.
