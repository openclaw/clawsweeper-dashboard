## What This Changes

The PR adds sandbox-only Python/Node research-script import-template preflight, focused tests, and sandboxing docs for direct script exec commands.

## Best Possible Solution

Rebase onto current main's exec preflight path, keep the current parser safeguards, harden import detection with multiline-aware parsing and validated local-import closure or a clearly narrower documented boundary, then add focused bypass regressions and real sandbox exec proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse Node imports across multiline statements:** `src/agents/sandbox-execution-templates.ts:442-457`
  - body: `validateNodeImports` scans one trimmed line at a time, so a normal multiline import such as `import { execSync }\nfrom "node:child_process"` never sends the denied specifier to `assertNodeImportAllowed`. That lets sandboxed Node research scripts bypass the new template preflight with standard syntax.
  - confidence: 0.94
- **[P1] Handle whitespace before dynamic import call parens:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: JavaScript permits whitespace between the callee and `(`, but these guards only match `require(` and `import(`. Calls like `require ("node:child_process")` avoid both the literal-import check and the dynamic-call rejection, so denied runtime modules can still execute in sandboxed scripts.
  - confidence: 0.9
- **[P1] Validate local helper imports before allowing them:** `src/agents/bash-tools.exec.ts:134-139`
  - body: The preflight validates only the direct entry script content, while the template allows workspace-local helpers without scanning them. A script can import a local helper that imports `node:child_process` or `subprocess`, bypassing the stated sandbox import template before execution.
  - confidence: 0.86
