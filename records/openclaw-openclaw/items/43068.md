## Summary

Keep open: the hardening goal is useful, but current main does not implement the research import-template behavior and the proposed patch has source-confirmed parser bypasses in a security-sensitive exec path.

## What This Changes

Adds sandbox-only Python/Node research import-template validation to exec script preflight, with focused tests and sandboxing documentation.

## Best Possible Solution

Implement this as a sandbox-only layer on the current exec preflight path with parser-backed or multiline-aware Node extraction, explicit Python bare-relative denied-name checks, regression tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows valid Node multiline imports and whitespace-before-call forms can evade the line scanner, and `from . import subprocess` reaches the Python bare-relative gap.

## Solution Assessment

No. The sandbox-only direction is plausible, but the PR's regex line scanner is not robust enough for command-execution hardening and the branch needs to be reconciled with current main's preflight behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Parse Node imports across multiline declarations:** `src/agents/sandbox-execution-templates.ts:436-471`
  - body: The validator scans one trimmed line at a time, so `import { ... } from "node:child_process"` split across lines never sends the denied specifier to `assertNodeImportAllowed`. Multiline imports and re-exports are normal JavaScript syntax, so sandboxed Node scripts can bypass the new template preflight.
  - confidence: 0.95
- **[P1] Treat spaced require/import calls as calls:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: JavaScript permits `require ("node:child_process")` and `import ("node:child_process")`, but these regexes only recognize `require(` and `import(`. Those calls are neither validated as literal imports nor rejected as dynamic calls, letting denied modules pass sandbox preflight.
  - confidence: 0.93
- **[P2] Check bare relative Python imported names:** `src/agents/sandbox-execution-templates.ts:287-289`
  - body: For `from . import subprocess`, the source specifier becomes empty after stripping dots and returns before checking the imported name. That allows denied-looking local wrappers through even though the template intends to reject denied process/runtime module names.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `docs/gateway/sandboxing.md:220`
  - body: This changes user-visible exec behavior and the PR marks a feature/bug fix, but the file list does not include `CHANGELOG.md`. Repo policy requires changelog coverage for user-facing fix, feature, and perf changes before merge.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch narrows sandboxed command execution but leaves concrete import-validation bypasses.

Concerns:

- **[high] Node template bypass through valid import syntax:** `src/agents/sandbox-execution-templates.ts:436`
  - body: Line-by-line same-line regex matching misses multiline imports/re-exports and whitespace before `require` / `import` call parentheses, allowing denied modules such as `node:child_process` to evade sandbox import validation.
