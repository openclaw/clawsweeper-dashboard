## Summary

Keep open: current main still lacks the sandbox import-template feature, but this PR should not merge as-is because the added validator has source-reproducible import bypasses, lacks real behavior proof, and is dirty against newer main.

## What This Changes

This PR adds sandbox-only Python and Node research-script import-template validation before direct `exec`, with focused tests and sandboxing docs.

## Best Possible Solution

Rebase the PR onto current main, replace the fragile line-oriented scanner with parser-like or multiline-aware validation, add regressions for the bypasses, and post redacted real exec output proving sandbox rejection and host behavior.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows deterministic bypass inputs for multiline Node imports, spaced dynamic calls, and Python bare-relative imports. I did not run tests because this cleanup review is read-only.

## Solution Assessment

No. The security-hardening direction is useful, but the current line-oriented regex implementation is not robust enough for the stated sandbox preflight guarantee.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Parse multiline Node imports before allowing execution:** `src/agents/sandbox-execution-templates.ts:442-457`
  - body: `validateNodeImports` splits the script into lines and matches each trimmed line independently, so a normal formatted import like `import {\n  execSync\n} from "node:child_process"` never has both `import` and the module specifier on the same line. That lets denied Node modules through the new sandbox preflight.
  - confidence: 0.95
- **[P1] Accept whitespace before require/import parentheses:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: The literal and dynamic-call checks only recognize `require(` and `import(`. JavaScript allows whitespace before the parenthesis, so `require ("node:child_process")` and `import ("node:child_process")` avoid both the literal import check and the dynamic-call guard.
  - confidence: 0.92
- **[P2] Check imported names for bare relative Python imports:** `src/agents/sandbox-execution-templates.ts:287-290`
  - body: For `from . import subprocess`, the source specifier is `.`, which is stripped to an empty root and returns before any denied-name check. If a generated script also drops a local wrapper with a denied-looking name, this bypasses the template's stated intent to reject those imports before exec.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR narrows sandbox exec in concept, but the added validator has concrete import-bypass bugs that weaken the security guarantee.

Concerns:

- **[high] Multiline Node imports bypass denied-module checks:** `src/agents/sandbox-execution-templates.ts:442`
  - body: Because the validator scans one trimmed line at a time, standard multiline `import ... from "node:child_process"` syntax can pass without calling `assertNodeImportAllowed`.
  - confidence: 0.95
- **[high] Spaced require/import calls bypass dynamic guards:** `src/agents/sandbox-execution-templates.ts:473`
  - body: JavaScript permits `require (...)` and `import (...)`; the PR checks only the no-space forms, leaving another straightforward way around the new preflight.
  - confidence: 0.92
- **[medium] Bare relative Python imports skip denied-name review:** `src/agents/sandbox-execution-templates.ts:287`
  - body: `from . import subprocess` passes `.` into the checker, which becomes an empty root and returns early before checking the imported name.
  - confidence: 0.82
