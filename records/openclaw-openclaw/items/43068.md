## Summary

Keep open: current main does not implement the requested research-script import template, but this PR cannot merge as-is because the proposed validator has concrete parser bypasses and predates current exec preflight hardening.

## What This Changes

The PR adds sandbox-only Python/Node research import-template validation to exec preflight, plus focused tests and sandboxing documentation.

## Best Possible Solution

Implement this as a sandbox-only layer on the current exec preflight path, using parser-backed or multiline-aware Node import extraction, explicit Python bare-relative denied-name checks, bypass regressions, docs, and the required changelog entry.

## Reproduction Assessment

Yes. On PR head, a sandboxed Node script using a multiline import or `require ("node:child_process")` bypasses the proposed line scanner, and `from . import subprocess` exercises the Python gap; this review is source-based rather than test-run based.

## Solution Assessment

No. The sandbox-only direction is plausible, but the PR's regex line scanner is not robust enough for command-execution hardening; parser-backed or multiline-aware extraction on current main is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Parse Node imports across multiline declarations:** `src/agents/sandbox-execution-templates.ts:436-471`
  - body: The validator scans one trimmed line at a time, so `import { ... } from "node:child_process"` split across lines never passes the denied specifier to `assertNodeImportAllowed`. Multiline imports and re-exports are normal JavaScript syntax, so sandboxed Node scripts can bypass the new template preflight.
  - confidence: 0.95
- **[P1] Treat spaced require/import calls as calls:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: JavaScript permits `require ("node:child_process")` and `import ("node:child_process")`, but these regexes only recognize `require(` and `import(`. Those calls are neither validated as literal imports nor rejected as dynamic calls, letting denied modules pass sandbox preflight.
  - confidence: 0.93
- **[P2] Check bare relative Python imported names:** `src/agents/sandbox-execution-templates.ts:287-289`
  - body: For `from . import subprocess`, the source specifier becomes empty after stripping dots and returns before checking the imported name. That allows denied-looking local wrappers through even though the template intends to reject denied process/runtime module names.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch narrows sandboxed command execution but leaves concrete import-validation bypasses.
