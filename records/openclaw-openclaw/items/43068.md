## Summary

Keep open: the proposed hardening addresses a command-execution gap that is not implemented on current main, but this PR should not merge yet because the new validator has concrete bypasses and the external PR lacks inspectable real behavior proof.

## What This Changes

This PR adds sandbox-only Python/Node research-script import-template preflight logic, tests for sandbox-vs-host behavior, and sandboxing documentation.

## Best Possible Solution

Rebase onto current main, preserve the existing fail-closed and safe-read preflight behavior, replace the fragile line-by-line import scan with multiline-aware or parser-backed extraction, add bypass regression tests, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection shows direct sandboxed Node scripts using multiline imports or spaced require/import calls would not reach assertNodeImportAllowed. I did not execute the branch because this was a read-only review.

## Solution Assessment

No. The hardening direction is plausible, but a security preflight should not rely on line-local regexes that miss common JavaScript syntax and Python bare-relative import names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse Node imports across multiline statements:** `src/agents/sandbox-execution-templates.ts:442-444`
  - body: `validateNodeImports` splits the file into lines and applies the import regex to each trimmed line, so a normal formatted import with `import {` on one line and `} from "node:child_process"` later never calls `assertNodeImportAllowed`. That lets denied Node builtins pass the new sandbox template preflight.
  - confidence: 0.94
- **[P1] Reject spaced require and import calls:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: JavaScript permits whitespace before the call parenthesis, but the literal and dynamic guards only recognize `require(` and `import(`. Calls like `require ("node:child_process")` or `import (name)` therefore bypass both the allowlist check and the dynamic-call rejection.
  - confidence: 0.91
- **[P2] Check denied names in bare Python relative imports:** `src/agents/sandbox-execution-templates.ts:287-289`
  - body: For `from . import subprocess`, the specifier is `.`; after stripping dots, `rootSpecifier` is empty and the function returns before checking denied names. If a generated script also drops a local wrapper with a denied name, the text-level sandbox template no longer enforces its stated rule.
  - confidence: 0.83
