## What This Changes

The PR adds sandbox-only Python/Node direct-script import-template validation to exec, with focused tests and sandboxing documentation.

## Best Possible Solution

Rebase onto current main and implement parser-like or multiline-aware fail-closed import validation for sandboxed direct scripts while preserving host exec and current fs-safe preflight behavior.

## Solution Assessment

No. The security-hardening direction is useful, but the current line-oriented regex scanner is not robust enough for a sandbox preflight guarantee at the exec boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Parse multiline Node imports before allowing execution:** `src/agents/sandbox-execution-templates.ts:442-457`
  - body: validateNodeImports scans each trimmed line separately, so a standard multiline import never presents both the import keyword and module specifier to the validator. That lets denied modules like node:child_process pass the new sandbox preflight.
  - confidence: 0.95
- **[P1] Accept whitespace before require/import parentheses:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: The literal and dynamic-call checks only recognize require( and import(. JavaScript permits require ("node:child_process") and import ("node:child_process"), which bypass both the literal import check and dynamic-call guard.
  - confidence: 0.92
- **[P2] Check imported names for bare relative Python imports:** `src/agents/sandbox-execution-templates.ts:287-290`
  - body: For from . import subprocess, the source specifier becomes an empty root and returns before denied names are checked. A local wrapper with a denied name can therefore avoid the template's stated text-level block.
  - confidence: 0.82
