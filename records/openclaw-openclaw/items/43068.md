## Summary

Keep open: the security-hardening goal is not implemented on current main and remains useful, but this PR head has concrete import-validator bypasses and lacks real behavior proof, so it should not merge as-is.

## What This Changes

The PR adds sandbox-only Python and Node research-script import-template validation before direct exec, plus focused tests and sandboxing documentation.

## Best Possible Solution

Rebase onto current main, implement parser-like or multiline-aware import validation with bypass regressions, preserve the sandbox-only boundary, and require redacted real exec proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head gives deterministic bypass inputs for multiline Node imports, spaced require/import calls, and Python bare-relative imports. I did not run tests because this cleanup review is read-only.

## Solution Assessment

No. The hardening belongs near exec preflight, but the current line-oriented regex scanner is not robust enough for a security-sensitive sandbox import guarantee.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Parse multiline Node imports before allowing execution:** `src/agents/sandbox-execution-templates.ts:442-457`
  - body: `validateNodeImports` splits the script into lines and matches each trimmed line independently, so `import {\n  execSync\n} from "node:child_process"` never has both the import keyword and module specifier on the same line. That lets denied Node modules through the new sandbox preflight.
  - confidence: 0.95
- **[P1] Accept whitespace before require/import parentheses:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: JavaScript allows `require ("node:child_process")` and `import ("node:child_process")`, but the PR only recognizes `require(` and `import(`. Those valid forms avoid both the literal import check and the dynamic-call guard.
  - confidence: 0.92
- **[P2] Check imported names for bare relative Python imports:** `src/agents/sandbox-execution-templates.ts:287-290`
  - body: For `from . import subprocess`, the captured specifier is `.`, which is stripped to an empty root and returned without checking the imported name. A local wrapper with a denied-looking name can bypass the template's stated import review.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR narrows sandbox exec in concept, but the added validator has concrete import-bypass bugs in a security-sensitive execution path.

Concerns:

- **[high] Multiline Node imports bypass denied-module checks:** `src/agents/sandbox-execution-templates.ts:442`
  - body: Because the validator scans one trimmed line at a time, standard multiline `import ... from "node:child_process"` syntax can pass without calling `assertNodeImportAllowed`.
  - confidence: 0.95
- **[high] Spaced require/import calls bypass dynamic guards:** `src/agents/sandbox-execution-templates.ts:473`
  - body: JavaScript permits whitespace before call parentheses, but the PR checks only no-space `require(` and `import(` forms, leaving a straightforward bypass.
  - confidence: 0.92
- **[medium] Bare relative Python imports skip denied-name review:** `src/agents/sandbox-execution-templates.ts:287`
  - body: `from . import subprocess` passes `.` into the checker, which becomes an empty root and returns before checking the imported name.
  - confidence: 0.82
