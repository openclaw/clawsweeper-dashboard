Keep open. The security-hardening goal is still relevant and not implemented on current main, but the branch is conflicting, lacks after-fix real behavior proof, and introduces concrete import-scanner bypasses in the new exec preflight path.

## What This Changes

The PR adds sandbox-only Python/Node research-script import-template validation to direct exec preflight, with focused tests and sandboxing docs.

## Best Possible Solution

Rebase onto current main, preserve the fs-safe/fail-closed preflight path, and replace the scanner with parser-like or multiline-aware validation plus regressions and redacted real exec proof.

Yes for the review findings: PR-head source inspection gives deterministic bypass inputs for multiline Node imports, spaced require/import calls, and Python bare-relative imports. I did not run tests because this cleanup review is read-only.

## Solution Assessment

No. The hardening goal is useful, but the current line-oriented regex scanner is not robust enough for a sandbox preflight guarantee at the exec security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Parse multiline Node imports before allowing execution:** `src/agents/sandbox-execution-templates.ts:442-457`
  - body: `validateNodeImports` scans each trimmed line separately, so `import {\n  execSync\n} from "node:child_process"` never presents both the import keyword and module specifier to the validator. That lets denied Node modules pass the new sandbox preflight.
  - confidence: 0.95
- **[P1] Accept whitespace before require/import parentheses:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: The literal and dynamic-call checks only recognize `require(` and `import(`. JavaScript permits `require ("node:child_process")` and `import ("node:child_process")`, which bypasses both the literal import check and dynamic-call guard.
  - confidence: 0.92
- **[P2] Check imported names for bare relative Python imports:** `src/agents/sandbox-execution-templates.ts:287-290`
  - body: For `from . import subprocess`, the source specifier is `.`; stripping leading dots produces an empty root and returns before denied names are checked. A local wrapper with a denied name can therefore avoid the template's stated text-level block.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention because the PR changes the exec security boundary and the added import scanner has concrete bypass cases.

Concerns:

  - confidence: 0.95

  - confidence: 0.92

  - body: `from . import subprocess` passes `.` as the specifier, which becomes an empty root and returns before the imported name is checked against the denied list.
