## Summary

Keep open. Current main still lacks sandbox-only research import templates, so the hardening is not obsolete, but this PR is not mergeable as-is because the proposed text scanner has concrete import-validation bypasses and the branch predates current exec preflight refactors.

## What This Changes

The PR adds sandbox-only Python/Node research import-template validation to exec preflight, with focused tests and sandboxing documentation.

## Best Possible Solution

Implement the feature as a sandbox-only layer on the current exec preflight path, using parser-backed or robust multiline-aware import extraction for Node, explicit checks for Python bare-relative denied names, focused bypass regressions, updated docs, and the required user-facing changelog entry.

## Reproduction Assessment

Yes. On PR head, a sandboxed Node script with `import {\n  execSync\n} from "node:child_process"` or `require ("node:child_process")` exercises the validator bypass; `from . import subprocess` exercises the Python gap. Current-main verification is source-based because this read-only review did not run tests.

## Solution Assessment

No. The direction is plausible, but the current regex line scanner is not the safest way to enforce a sandbox import policy; a parser-backed or multiline-aware extractor with explicit dynamic-call handling is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Parse Node imports across multiline declarations:** `src/agents/sandbox-execution-templates.ts:436-471`
  - body: The validator scans one trimmed line at a time, so `import {\n  execSync\n} from "node:child_process"` never passes the denied specifier to `assertNodeImportAllowed`. Multiline imports and re-exports are normal JavaScript syntax, so sandboxed Node scripts can bypass the new template preflight.
  - confidence: 0.95
- **[P1] Treat spaced require/import calls as calls:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: JavaScript permits `require ("node:child_process")` and `import ("node:child_process")`, but these regexes only recognize `require(` and `import(`. Those calls are neither validated as literal imports nor rejected as dynamic calls, letting denied modules pass sandbox preflight.
  - confidence: 0.93
- **[P2] Check bare relative Python imported names:** `src/agents/sandbox-execution-templates.ts:287-289`
  - body: For `from . import subprocess`, the source specifier becomes empty after stripping dots and returns before checking the imported name. That allows denied-looking local wrappers through even though the template intends to reject denied process/runtime module names.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention: the patch targets command-execution sandbox hardening, but the proposed parser has concrete bypasses and must be rebased onto current preflight safeguards.

Concerns:

- **[high] Node template bypass through valid import syntax:** `src/agents/sandbox-execution-templates.ts:436`
  - body: Line-by-line same-line regex matching misses multiline imports/re-exports and whitespace before `require` / `import` call parentheses, allowing denied modules such as `node:child_process` to evade sandbox import validation.
