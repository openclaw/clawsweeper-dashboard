## Summary

Keep open. Current main does not already contain this sandbox import-template validation, but the PR is not mergeable as-is because the added scanner has concrete bypasses in ordinary Node/Python import syntax and the external PR still lacks real behavior proof.

## What This Changes

The PR adds sandbox-only Python/Node research-script import allow/deny validation before direct `exec`, plus focused tests and sandboxing documentation.

## Best Possible Solution

Rebase onto current main and implement parser-like or multiline-aware import validation with regressions for the known bypasses while preserving host exec behavior.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection gives deterministic bypass inputs for multiline Node imports, spaced dynamic calls, and Python bare-relative imports. I did not run tests because this cleanup review is read-only.

## Solution Assessment

No. The security-hardening direction is useful, but the current line-oriented regex implementation is not robust enough for the sandbox preflight guarantee.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Parse multiline Node imports before allowing execution:** `src/agents/sandbox-execution-templates.ts:442-457`
  - body: `validateNodeImports` scans one trimmed line at a time, so a normal formatted import like `import {\n  execSync\n} from "node:child_process"` never has both the `import` keyword and module specifier in the same checked string. That lets denied Node modules pass the new sandbox preflight.
  - confidence: 0.95
- **[P1] Accept whitespace before require/import parentheses:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: The literal and dynamic-call checks only recognize `require(` and `import(`. JavaScript allows whitespace before the parenthesis, so `require ("node:child_process")` and `import ("node:child_process")` avoid both the literal import check and dynamic-call guard.
  - confidence: 0.92
- **[P2] Check imported names for bare relative Python imports:** `src/agents/sandbox-execution-templates.ts:287-290`
  - body: For `from . import subprocess`, the source specifier is `.`, which is stripped to an empty root and returns before denied-name checks. A generated script with a same-named local wrapper can bypass the template's stated intent to reject denied import names before exec.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention because the PR changes the exec security boundary and the added scanner has concrete import-bypass cases.

Concerns:

- **[high] Multiline Node imports bypass denied-module checks:** `src/agents/sandbox-execution-templates.ts:442`
  - body: The validator scans each trimmed line independently, so standard multiline `import ... from "node:child_process"` syntax can pass without calling `assertNodeImportAllowed`.
  - confidence: 0.95
- **[high] Spaced require/import calls bypass dynamic guards:** `src/agents/sandbox-execution-templates.ts:473`
  - body: JavaScript permits `require (...)` and `import (...)`, but the PR checks only the no-space forms, leaving a direct way around the new preflight.
  - confidence: 0.92
- **[medium] Bare relative Python imports skip denied-name review:** `src/agents/sandbox-execution-templates.ts:287`
  - body: `from . import subprocess` passes `.` as the specifier, which becomes an empty root and returns before the imported name is checked against denied imports.
  - confidence: 0.82
