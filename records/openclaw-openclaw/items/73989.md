## Summary

Keep this PR open: current main still lacks reconnect grace for recently disconnected running channels, and the proposed diff is a narrow gateway fix with focused tests and changelog coverage. The remaining blocker is not code correctness from this review, but the current dirty/unmergeable PR head and a needed changed-gate run after rebase.

## What This Changes

This PR adds shared gateway reconnect-grace handling, projects typed `lastDisconnect.at` into health snapshots, threads `reconnectGraceMs` through health monitor/readiness policy, adds regression tests, and updates the changelog with #45712 credit.

## Best Possible Solution

Keep the shared reconnect-grace implementation in the gateway health policy and use it consistently from both background health monitoring and readiness checks. After the dirty head is repaired, the PR should land only with a clean changed gate on the exact rebased head.

## Reproduction Assessment

Yes. A current-main unit-level fixture with `running: true`, `connected: false`, `lastStartAt` older than startup grace, and a recent typed `lastDisconnect.at` still evaluates as `disconnected`, which drives monitor restarts and readiness failure before this PR.

## Solution Assessment

Yes. Centralizing the timing window in `channel-health-policy.ts` and projecting typed runtime snapshots at the monitor/readiness call sites is the narrowest maintainable fix; no owner-specific Discord special case or broader plugin API change is needed for the behavior shown here.
