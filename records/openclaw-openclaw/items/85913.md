## Label Justifications

- `P1`: The report describes user-visible dropped replies in active channel and heartbeat workflows, and current source still has the same coordination gap.
- `impact:message-loss`: The reported failure mode is an embedded run failing before a user-visible reply, which can silently drop chat responses.
- `impact:session-state`: The root cause is concurrent embedded lanes mutating and invalidating the same session transcript file and fence.

## Summary

Keep open: current main still has a source-supported transcript-file contention gap where separate embedded lanes can share one session JSONL and trip the takeover fence; the related fallback fix prevents model-chain churn but does not prevent the upstream race.

## What This Changes

The issue reports that embedded heartbeat, channel, direct, cron, or stuck-recovery lanes can concurrently use the same session JSONL file, causing `EmbeddedAttemptSessionTakeoverError` and occasional user-visible dropped replies.

## Best Possible Solution

Add a transcript-file-scoped embedded prompt ownership guard that serializes, waits, or returns a typed contention result before duplicate logical lanes can mutate the same session file, while keeping the existing fence and fallback classification as safety nets.

## Reproduction Assessment

Yes at source level, but not as a live run in this review. Current code and tests show that another owner changing the same session file while the prompt lock is released triggers `EmbeddedAttemptSessionTakeoverError`; a full heartbeat/channel harness mapping two lanes to one transcript file is still missing.

## Solution Assessment

Unclear as a final implementation: the core boundary is right, but maintainers need to choose the contention behavior before automation changes lock semantics. Changing `heartbeat.isolatedSession` defaults should be treated as a separate compatibility decision, not the narrow bug fix.
