## Label Justifications

- `P1`: The issue describes dropped replies in active embedded channel/heartbeat workflows, and current source still has the coordination gap.
- `impact:message-loss`: The reported failure mode is an embedded run failing before a user-visible reply, which can silently drop chat responses.
- `impact:session-state`: The root cause is concurrent embedded lanes mutating and invalidating the same session transcript file/fence.

## Summary

Keep open: current source still allows the reported session-file contention. The fallback classification fix for the related closed issue is present, but it only aborts model fallback after a takeover error; it does not prevent heartbeat/channel lanes that resolve to the same transcript file from racing.

## What This Changes

The issue asks OpenClaw to prevent concurrent embedded heartbeat, channel, direct, or recovery lanes from sharing and mutating the same resolved session transcript file.

## Best Possible Solution

Add transcript-file-scoped embedded prompt ownership so duplicate logical lanes are serialized or redirected before they can mutate the same transcript, while keeping the existing fence as a safety net.

## Reproduction Assessment

Yes at source level, but not as a live run in this review. Current tests and code show that a second owner changing the same session file while the prompt lock is released triggers `EmbeddedAttemptSessionTakeoverError`; the missing piece is a live heartbeat/channel harness that maps two lanes to one transcript file.

## Solution Assessment

Unclear until maintainers choose the contention policy. The right boundary is the resolved transcript file before provider streaming, but wait-vs-steer-vs-typed-failure is a product/runtime behavior decision.
