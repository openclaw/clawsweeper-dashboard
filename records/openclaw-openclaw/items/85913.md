## Summary

Keep open: current main still lacks a transcript-file-scoped active embedded prompt holder, so the same session JSONL can be entered by another logical run while the original prompt lock is released. The older fallback and owned-write fixes reduce related symptoms, but they do not prevent duplicate prompt holders from racing on the same file.

## What This Changes

The issue requests a transcript-file-scoped guard so concurrent embedded prompt runs cannot race on the same session JSONL and drop replies.

## Best Possible Solution

Add a resolved-session-file active prompt ownership guard before provider streaming, then make stuck-session recovery wait, steer, fail cleanly, or choose a fresh session rather than re-entering a JSONL file that already has an active holder.

## Reproduction Assessment

Yes at source level: current main still releases the prompt lock, records a file fence, and throws when the same file changes; the reporter also supplied real gateway logs for the same failure. I did not run a live heartbeat/channel harness in this read-only review.

## Solution Assessment

Yes for the narrow fix boundary: prevent or serialize duplicate embedded prompt holders by resolved session file before provider streaming. Changing heartbeat isolated-session defaults is a separate config/product choice and should not be required for the core bug fix.
