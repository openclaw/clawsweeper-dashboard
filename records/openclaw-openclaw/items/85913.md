## Best Possible Solution

Add a resolved-session-file ownership guard around embedded prompt submission and make stuck-session recovery wait, steer, or fail cleanly instead of creating a second prompt holder for the same JSONL.

## Repair Work Prompt

Fix the session-file contention race reported in https://github.com/openclaw/openclaw/issues/85913. Add a resolved transcript-file-scoped embedded prompt ownership guard before provider streaming so a second embedded prompt holder for the same JSONL waits, steers, or fails with a typed coordination outcome before mutating the file; keep `EmbeddedAttemptSessionTakeoverError` as the safety fence, not the normal coordination path. Update stuck-session recovery so it does not clear/restart work onto a session file with an active embedded prompt holder. Do not change heartbeat isolation defaults or plugin/channel public APIs unless maintainers explicitly request that product/config change. Inspect adjacent open PRs touching `attempt.session-lock.ts` so the fix does not duplicate or regress event-write/false-positive fence work. Add focused regression coverage for the file-scoped ownership guard and recovery behavior; include a changelog entry if the fix is user-visible.
