## Label Justifications

- `P1`: The report describes user-visible dropped replies in active channel/heartbeat workflows, and current source still has the coordination gap.
- `impact:message-loss`: The reported failure mode is an embedded run failing before a user-visible reply, which can silently drop chat responses.
- `impact:session-state`: The root cause is concurrent embedded lanes mutating and invalidating the same session transcript file/fence.

## Summary

The source path still permits the reported class of contention: session locks intentionally release around provider streaming and detect later file mutations, while active embedded-run bookkeeping is keyed by session id/key rather than the resolved session file. The related fallback classification work is present, but it only stops model fallback churn; it does not prevent two logical lanes that map to the same transcript file from racing.

## What This Changes

The issue asks OpenClaw to prevent concurrent embedded lanes, especially heartbeat and channel/direct lanes, from sharing and racing on the same session JSONL file.

## Best Possible Solution

Add a transcript-file-scoped embedded prompt ownership guard that makes duplicate logical lanes wait, steer, or fail with a typed coordination result before they can mutate the same session file, while keeping the existing takeover fence as a safety net.

## Reproduction Assessment

Yes at source level, but not as a live run in this review. Current tests and code show that a second owner changing the same session file while the prompt lock is released triggers `EmbeddedAttemptSessionTakeoverError`; the missing piece is a real heartbeat/channel harness that maps two lanes to one transcript file.

## Solution Assessment

Yes for the core fix boundary: prevent or serialize duplicate embedded prompt holders for the same resolved session file before provider streaming. Changing `heartbeat.isolatedSession` defaults is a separate product/config decision and should not be required for the narrow bug fix.

## Repair Work Prompt

Fix the embedded session takeover race where two logical embedded lanes can resolve to the same session JSONL file and one lane throws `EmbeddedAttemptSessionTakeoverError` after the prompt lock is released. Keep the fix generic in core embedded/session coordination: add a session-file-scoped active prompt ownership guard or equivalent serialization before provider streaming, integrate it with stuck-session recovery so recovery does not force a duplicate run on a file with an active embedded prompt holder, and leave `heartbeat.isolatedSession` default changes out unless maintainers explicitly decide otherwise. Preserve the existing fence/takeover detection as a safety net and add focused regression coverage proving two controllers/runs for the same resolved session file do not race while unrelated session files still run independently. Related refs: https://github.com/openclaw/openclaw/issues/85913 and https://github.com/openclaw/openclaw/issues/83510. Changelog expected because the fix is user-visible reliability. Do not change plugin APIs or channel-specific routing policy.
