## Summary

Keep this open. Current main still lacks a shared tool-response freshness contract; the named tools and common helpers use ordinary `content`/`details` results or ad hoc cache flags, and the remaining work needs core/SDK design rather than cleanup closure.

## What This Changes

Standardize tool response freshness metadata such as `cached`, `age_ms`, `ttl_ms`, and fetch timestamps so agents can reason about stale cached tool data.

## Best Possible Solution

Define a backwards-compatible optional core/SDK freshness metadata contract, document model-visible versus runtime-only semantics, add replay/compaction/persistence coverage, then adopt it in targeted cached tools such as `session_status`, `memory_search`, cron status/runs, and cached web/provider tools.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an RFC/feature request. The gap is high-confidence by source inspection because current main has no shared freshness type or envelope and the named tool implementations still return ordinary payloads.

## Solution Assessment

Unclear until maintainers settle the contract shape. A core/SDK optional metadata surface is the likely maintainable direction, but a details-only implementation would not meet the requested agent-aware behavior because details are stripped from LLM-facing compaction/display paths.
