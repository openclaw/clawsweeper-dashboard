## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata envelope, and this is a cross-core/SDK feature contract that needs maintainer API/product direction before implementation.

## What This Changes

Request a standardized freshness metadata envelope for tool responses, including cached state, age, TTL, and fetched timestamp, with adoption by high-value cached tools.

## Best Possible Solution

Define an optional, documented core/SDK freshness metadata contract with compatibility rules, then adopt it in high-value cached tools with focused tests and prompt guidance.

## Reproduction Assessment

Not applicable: this is a feature RFC, not a broken existing behavior. Source inspection of current main gives high confidence that the requested standardized freshness envelope is still absent from shared tool-result and named adoption surfaces.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional contract is likely the maintainable direction; ad hoc per-tool fields would be less consistent across plugins and runtimes.
