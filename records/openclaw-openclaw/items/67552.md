## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata envelope, and this is a cross-core/SDK feature contract that needs maintainer API/product direction before implementation.

## What This Changes

The issue requests an additive freshness metadata contract for tool responses, including `cached`, `age_ms`, `ttl_ms`, and `timestamp`, with adoption in session status, memory search, and cron status tools.

## Best Possible Solution

Define an optional, documented core/SDK freshness metadata contract with compatibility rules, then adopt it in high-value cached tools with focused tests and prompt guidance.

## Reproduction Assessment

Not applicable: this is a feature RFC, not a broken existing behavior. Source inspection on current main shows the requested standardized freshness envelope is absent from shared tool-result helpers and the named adoption tools.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional contract is likely the maintainable direction; ad hoc per-tool fields would not standardize behavior across plugins and runtimes.
