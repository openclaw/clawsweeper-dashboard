## Summary

Keep open. Current main and v2026.6.10 still expose direct tool-result payloads without a shared freshness metadata contract, and the remaining work is a cross-core/plugin-SDK/gateway API decision rather than conservative cleanup.

## What This Changes

The issue requests a standardized freshness metadata envelope for cached tool responses, including cached state, age, TTL, and fetch timestamp.

## Best Possible Solution

Have maintainers choose a documented optional core/SDK freshness metadata contract, then adopt it in high-value cached tools and gateway/plugin docs/tests while coordinating separately with context provenance work.

## Reproduction Assessment

Not applicable: this is a feature RFC for a new shared metadata contract, not a broken existing behavior with a failing runtime path. Source inspection confirms the requested standardized freshness envelope is absent today.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional core/SDK metadata contract is the maintainable direction; ad hoc per-tool payload fields or an unconditional data/freshness wrapper could create drift or compatibility surprises.
