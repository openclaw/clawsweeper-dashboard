## Summary

Keep open. Current main and the latest release still expose direct tool-result payloads without a shared freshness contract, and the remaining work is a cross-core/SDK/gateway/plugin API product decision rather than conservative cleanup.

## What This Changes

The issue requests a standardized freshness metadata contract for tool responses so agents can distinguish cached or stale data from fresh results.

## Best Possible Solution

Have maintainers choose a documented optional core/SDK freshness metadata contract, then adopt it in high-value cached tools and gateway/SDK docs/tests while coordinating separately with context provenance work.

## Reproduction Assessment

Not applicable: this is a feature RFC for a new shared metadata contract, not a broken existing behavior with a current-main failing path. Source inspection confirms the requested standardized freshness envelope is absent today.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional core/SDK metadata contract is the maintainable direction; ad hoc per-tool payload fields or an unconditional data/freshness wrapper could create drift or compatibility surprises.
