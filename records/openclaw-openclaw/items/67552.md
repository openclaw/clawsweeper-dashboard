## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata envelope; the existing tool-result helpers and middleware expose `content`/`details` results or ad hoc cache flags, and this is a cross-core/SDK feature RFC rather than a stale, duplicate, or ClawHub-only item.

## What This Changes

Request a standardized freshness metadata envelope with `cached`, `age_ms`, `ttl_ms`, and `timestamp` for tool responses, starting with `session_status`, `memory_search`, and cron status.

## Best Possible Solution

Define an optional core/SDK tool-result freshness contract with clear model-visible semantics, then adopt it in high-value cached tools with focused compatibility and regression tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature RFC. Source inspection on current main gives a high-confidence gap because no shared freshness type/envelope exists and the named tools still return ordinary result payloads.

## Solution Assessment

Unclear until maintainers decide the API shape. A documented optional core/SDK metadata surface is likely more maintainable than adding ad hoc freshness fields inside individual tool payloads.
