## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata contract, and the related context-provenance issue/PR overlap with stale-context reasoning but do not supersede this tool-result API request.

## What This Changes

Requests a standardized freshness metadata contract for tool responses, with fields like `cached`, `age_ms`, `ttl_ms`, and `timestamp`, plus adoption in `session_status`, `memory_search`, and cron status.

## Best Possible Solution

Define an optional, documented core/SDK freshness metadata contract with compatibility rules, adopt it in high-value cached tools, and coordinate it with the context-provenance RFC instead of closing either tracker prematurely.

## Reproduction Assessment

Not applicable: this is a feature RFC rather than a broken existing behavior. Source inspection gives high confidence that current main still lacks the requested standardized freshness envelope.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional contract is likely the maintainable direction, but the exact placement and compatibility rules need product/API review.
