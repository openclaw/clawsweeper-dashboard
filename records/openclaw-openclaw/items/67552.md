## Summary

Keep this open. Current main still lacks a shared tool-response freshness contract; the matching surfaces use ordinary `content`/`details` tool results or ad hoc per-tool cache markers, and making freshness agent-visible crosses core/SDK, provider replay, compaction, and plugin compatibility boundaries that need maintainer design.

## What This Changes

Standardize tool responses so tools can expose cache/freshness metadata such as cached state, age, TTL, and source timestamp.

## Best Possible Solution

Define a backwards-compatible optional freshness metadata contract in the core/SDK tool-result surface, document which parts are model-visible versus runtime-only, add regression coverage for replay/compaction/persistence, then adopt it in targeted cached tools such as `session_status`, `memory_search`, cron status/runs, and cached web/provider tools.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an RFC/feature request. The gap is high-confidence by source inspection: current main has no shared freshness type/envelope, and the named tools still return ordinary `content`/`details` or JSON payloads.

## Solution Assessment

Unclear until maintainers settle the contract shape. A core/SDK optional metadata surface is the likely maintainable direction, but a details-only implementation would not satisfy the requested agent-aware behavior because `details` is stripped before model replay and compaction.
