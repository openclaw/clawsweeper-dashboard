## Summary

Keep open. Current main still does not have a standardized tool-response freshness metadata envelope; the shared tool result helpers, Pi adapter, middleware types, and named adoption surfaces still use ordinary result payloads. This remains a cross-core/SDK feature RFC that needs maintainer API/product direction, not a stale or duplicate cleanup item.

## What This Changes

The issue requests a standardized tool response freshness metadata contract with fields like `cached`, `age_ms`, `ttl_ms`, and `timestamp`, plus adoption in tools such as `session_status`, `memory_search`, and cron status.

## Best Possible Solution

Define an optional, documented core/SDK freshness metadata contract with compatibility semantics, then adopt it in high-value cached tools with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature RFC. Source inspection against current main gives a high-confidence gap because no shared freshness metadata type/envelope exists and the named tools still return ordinary result payloads.

## Solution Assessment

Unclear until maintainers choose the API shape. A documented optional core/SDK metadata surface is likely more maintainable than adding ad hoc freshness fields inside individual tool payloads.
