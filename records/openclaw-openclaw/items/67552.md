## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata envelope, and this is a cross-core/SDK feature contract that needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The request proposes a standardized optional freshness envelope for tool responses, including `cached`, `age_ms`, `ttl_ms`, and source timestamp metadata.

## Best Possible Solution

Define a documented optional core/SDK freshness metadata contract with compatibility rules, then adopt it in high-value cached tools with focused tests and prompt guidance.

## Reproduction Assessment

Not applicable: this is a feature RFC, not a broken existing behavior. Source inspection gives high confidence that current main lacks the requested standardized freshness envelope.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional core/SDK contract is likely the maintainable direction; ad hoc per-tool fields would not standardize behavior across plugins and runtimes.
