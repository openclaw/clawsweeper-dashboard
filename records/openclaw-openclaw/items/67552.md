## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata envelope, and the requested change is a cross-core/SDK feature contract that needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The issue requests a standardized freshness metadata contract for tool responses, including fields such as cached state, age, TTL, and fetch timestamp.

## Best Possible Solution

Define an optional, documented core/SDK freshness metadata contract with compatibility rules, then adopt it in high-value cached tools with focused tests and prompt guidance.

## Reproduction Assessment

Not applicable: this is a feature RFC, not a broken existing behavior. Source inspection gives high confidence that current main still lacks the requested standardized freshness envelope on shared tool results and the named adoption surfaces.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional core/SDK contract is likely the maintainable direction; ad hoc freshness fields inside individual tool payloads would be less consistent.
