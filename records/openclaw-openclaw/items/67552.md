## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata contract; adjacent tool metadata and tool-result middleware surfaces exist, but they do not define the requested `cached`/`age_ms`/`ttl_ms` envelope for model-visible tool results.

## What This Changes

Request standard freshness metadata on tool responses so agents can distinguish fresh, cached, and stale data.

## Best Possible Solution

Define a backwards-compatible optional core/SDK freshness metadata contract, document model-visible versus runtime-only semantics, and adopt it first in cached session_status, memory_search, cron, and web/provider tools with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an RFC/feature request. Source inspection gives a high-confidence gap because current main has no shared freshness type or envelope and the named tools still return ordinary content/details or ad hoc cache flags.

## Solution Assessment

Unclear until maintainers settle the contract shape. A core/SDK optional metadata surface is likely the maintainable direction, while only adding fields inside individual tool payloads would not standardize agent-visible freshness across runtimes and plugins.
