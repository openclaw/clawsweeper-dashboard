## Summary

Keep open. Current main and the latest release still expose direct tool-result payloads without the requested shared freshness metadata contract; the merged tool-result middleware is adjacent infrastructure, but it does not settle the API/product decision for standardized freshness semantics.

## What This Changes

The issue requests a standard freshness metadata envelope for tool responses so agents can reason about cached or stale data.

## Best Possible Solution

Have maintainers choose a documented optional core/SDK freshness metadata contract, then adopt it in high-value cached tools and gateway/plugin docs/tests while coordinating separately with context provenance work.

## Reproduction Assessment

Not applicable: this is a feature RFC for a new shared metadata contract, not a broken existing behavior with a failing runtime path. Source inspection confirms the requested standardized freshness envelope is absent today.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional core/SDK metadata contract is maintainable, but an unconditional data/freshness wrapper could create compatibility surprises across existing tools and gateway clients.
