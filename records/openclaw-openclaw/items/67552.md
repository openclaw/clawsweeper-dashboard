## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata contract; the remaining work is a cross-core/SDK/API design decision rather than conservative cleanup.

## What This Changes

The issue requests a standard freshness metadata envelope for tool responses, including cached state, age, TTL, and fetch timestamp.

## Best Possible Solution

Define a documented optional core/SDK freshness metadata contract, then adopt it in high-value cached tools with focused compatibility tests and prompt guidance.

## Reproduction Assessment

Not applicable: this is a feature RFC, not a broken existing behavior. Source inspection confirms current main lacks the requested standardized freshness envelope.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional core/SDK contract is the maintainable direction; ad hoc per-tool fields would not standardize behavior across plugins, gateway callers, and runtimes.
