## Summary

Keep open. Current main still lacks a standardized tool-response freshness metadata envelope; shared tool-result helpers and SDK middleware still expose the Pi `content`/`details` result shape, and the named tools return ordinary payloads without a shared `cached`/`age_ms`/`ttl_ms` contract. This is a cross-core/SDK feature RFC, not a stale issue, duplicate, or ClawHub-only request.

## What This Changes

Requests standardized freshness metadata on tool responses, including `cached`, `age_ms`, `ttl_ms`, and `timestamp`, for cached outputs such as `session_status`, `memory_search`, and cron status.

## Best Possible Solution

Define an optional, documented core/SDK freshness metadata contract with clear model-visible compatibility semantics, then adopt it in high-value cached tools with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature RFC. Source inspection on current main gives a high-confidence gap because no shared freshness metadata type/envelope exists and the named tools still return ordinary result payloads.

## Solution Assessment

Unclear until maintainers choose the API shape. A documented optional core/SDK metadata surface is likely more maintainable than adding ad hoc freshness fields inside individual tool payloads.
