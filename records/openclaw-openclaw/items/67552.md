## Summary

Keep open: current main and v2026.6.10 still lack a shared tool-response freshness metadata contract, and the merged tool-result middleware is only an adjacent extension seam rather than the requested API/product decision.

## What This Changes

Request to add a standardized freshness metadata envelope (`cached`, `age_ms`, `ttl_ms`, `timestamp`) to tool responses and adopt it in high-value cached tools.

## Best Possible Solution

Have maintainers define a documented optional freshness metadata contract at the shared tool-result, gateway, and SDK boundary, then adopt it in `session_status`, memory search, and cron status with focused compatibility tests.

## Reproduction Assessment

Not applicable: this is a feature RFC for a new shared metadata contract, not a broken existing behavior with a runtime reproduction path. Source inspection confirms the requested standardized envelope is absent today.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared optional core/SDK metadata contract is the maintainable direction; ad hoc per-tool fields or an unconditional wrapper could create drift or compatibility surprises.
