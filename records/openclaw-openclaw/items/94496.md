## Summary

Keep this PR open, but it is not merge-ready: the revised scoped-cleanup direction is useful, yet the default no-service path now runs a Tailscale `off` command that upstream source treats as an error when no handler exists, so fresh Serve startup can fail before the route is enabled.

## What This Changes

The PR adds a pre-enable Tailscale Serve cleanup in `enableTailscaleServe`, using `serve --https=443 off` for the default node route and `serve clear <serviceName>` for named services, with updated helper tests.

## Best Possible Solution

Keep the scoped route-preserving cleanup idea, but make missing default-route cleanup a harmless no-op, add focused regression coverage, and require redacted live Tailscale first-start plus restart proof before merge.

## Reproduction Assessment

Yes, at source-inspection level. The PR runs `tailscale serve --https=443 off` before the default Serve enable, and upstream Tailscale returns `handler does not exist` when that route is absent; I did not run a live daemon in this read-only review.

## Solution Assessment

No. Scoped cleanup is the right direction compared with a global reset, but the implementation must be idempotent for the no-existing-route case before it is the best fix.
