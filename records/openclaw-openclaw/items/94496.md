## Summary

Keep open: current main still lacks the Tailscale Serve duplicate-entry fix, but this PR is not merge-ready because the new default cleanup can fail on first start when no default HTTPS handler exists, and the supplied proof is tests plus fake CLI capture rather than a real Tailscale daemon restart path.

## What This Changes

Adds a Tailscale Serve pre-cleanup inside `enableTailscaleServe`, using `serve --https=443 off` for default routes and `serve clear <serviceName>` for named services, plus updated helper tests.

## Best Possible Solution

Keep the scoped route-preserving cleanup direction, but make missing default-route cleanup a harmless no-op, add focused regression coverage, and require redacted live Tailscale first-start plus restart proof before merge.

## Reproduction Assessment

Yes, at source-inspection level. The PR runs `tailscale serve --https=443 off` before the default Serve enable, and upstream Tailscale returns `handler does not exist` when that route is absent; I did not run a live daemon in this read-only review.

## Solution Assessment

No. Scoped cleanup is the right direction compared with a global reset, but the implementation must be idempotent for the no-existing-route case before it is the best fix.
