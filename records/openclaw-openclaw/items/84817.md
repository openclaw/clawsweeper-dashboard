## Summary

Keep open. Current main and the latest release still build the managed gateway command around the resolved dist entrypoint when no wrapper is configured, while the related closed items cover entrypoint drift, generated-wrapper overwrite warnings, or a different errno -11 read path rather than this launchd plugin-load failure.

## What This Changes

Ensure macOS `openclaw gateway install --force` installs a launchd entrypoint that does not make plugins fail to load and send the gateway into a restart loop.

## Best Possible Solution

Choose and prove a launchd-safe canonical gateway service entrypoint, then update install, update, doctor, and macOS launchd validation so plugin loading works without regressing stable upgrade paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the full live macOS launchd errno in this review, but the source path is clear: current install planning still emits the wrapperless dist entrypoint that the reporter identified as the broken service command.

## Solution Assessment

Is this the best way to solve the issue? Unclear until a maintainer chooses the entrypoint policy; the reporter's wrapper workaround is plausible, but the existing dist-entrypoint path was introduced to protect service/update stability.
