Keep open: current main still has the reported subagent history/delivery surfaces, an open maintainer PR only covers the thread-bound spawn-default piece, and a maintainer comment explicitly says the stale announce/session-history path still needs resolution.

## What This Changes

Prevent stale subagent completion announces from becoming user-visible replies and ensure new subagent sessions do not inherit unrelated requester history.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/79050 for thread-bound isolation, then add a separate generic subagent announce freshness/lineage guard or align it with the durable completion design in https://github.com/openclaw/openclaw/issues/67777.

No live end-to-end reproduction was run in this read-only review. Source and tests still show the thread-bound default-fork path and the announce queue/direct delivery path that can inject `subagent_announce` work back into the requester session.

## Solution Assessment

No, current main is not yet the best solution because the inherited-history default is still present and the stale announce freshness path is still open. The active maintainer PR is the right immediate fix for one branch, but it does not replace a focused announce lineage guard.
