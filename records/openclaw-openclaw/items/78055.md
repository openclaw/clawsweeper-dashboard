## Summary

Keep open: current main still has the original subagent announce/session-history failure class. The custom OpenAI WebSocket branch was superseded by https://github.com/openclaw/openclaw/pull/79726, but that did not fix stale `subagent_announce` delivery or the thread-bound spawn context path, and https://github.com/openclaw/openclaw/pull/79050 is still open for part of the remaining work.

## What This Changes

The issue asks OpenClaw to prevent stale subagent completion announcements from becoming user-visible replies and to keep newly spawned subagent sessions from inheriting unrelated prior history.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79050 for the thread-bound isolation half, then add a focused announce freshness/session-lineage guard with regression tests for stale requester-session delivery.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live Telegram replay in this read-only pass, but the current-main source path is clear: omitted thread-bound context still resolves to `fork`, and queued announces are deliberately delivered after the busy-parent defer cap.

## Solution Assessment

Is this the best way to solve the issue? The current merged transport refactor is not sufficient; the maintainable path is the narrow thread-bound isolation fix plus a separate freshness/session-lineage check in subagent announce delivery.
