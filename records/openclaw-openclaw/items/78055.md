## Label Justifications

- `P1`: The report describes stale subagent output being delivered into a live requester conversation, which is a broken user-facing agent/channel workflow.
- `impact:session-state`: The central failure involves child sessions inheriting unrelated requester transcript state and stale lineage across parent/child session boundaries.
- `impact:message-loss`: The completion delivery path can misroute or duplicate stale subagent output into the user-visible channel instead of the current conversation result.

## Summary

Keep open: current main still preserves the native failure shape. The deleted custom OpenAI WebSocket path covers only one branch of the discussion, while thread-bound native subagent spawns still default to forked parent history and subagent completion delivery can still hand `subagent_announce` output back through the requester path.

## What This Changes

The issue reports stale subagent completion announcements reaching a requester session and child subagent sessions inheriting unrelated transcript history.

## Best Possible Solution

Keep this issue open for a maintainer-approved native subagent repair that makes thread-bound child context explicit or safe by default and adds freshness/session-lineage validation before requester-visible subagent_announce delivery.

## Reproduction Assessment

Yes, at source level: current main still defaults thread-bound subagent spawns to fork and still sends subagent_announce completion handoffs into the requester path. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No complete fix is on main. The best fix needs a maintainer decision on thread-bound fork compatibility and on the canonical stale-completion delivery contract before code should change behavior.
