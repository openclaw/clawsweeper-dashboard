## Summary

Keep this PR open. Current main still appears to have the reported Anthropic OAuth tool-name dispatch gap, but the submitted patch is not merge-ready because it imports non-existent dependency symbols and exposes new filesystem discovery tools without the existing workspace, sandbox, and tool-policy controls.

## What This Changes

This PR adds Anthropic OAuth aliases for Claude Code-style `Glob`/`Grep`/`LS` names and appends `find`/`grep`/`ls` tools to the OpenClaw agent tool set.

## Best Possible Solution

Land a revised, reviewed implementation that keeps the OAuth name mapping small, constructs discovery tools through supported dependency factories or OpenClaw-owned wrappers, applies workspace/sandbox filesystem boundaries, updates the core tool catalog and user docs if these become `group:fs` tools, and adds regression tests for alias dispatch plus allow/deny and `tools.fs.workspaceOnly` behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with an OAuth token, an inbound Anthropic `tool_use` named `Glob` flows through `fromClaudeCodeName()`, does not match a same-named local tool, and current main does not register local `find`/`grep`/`ls` tools for dispatch.

## Solution Assessment

No. The submitted implementation is not the best way to solve it because it would not typecheck against the pinned dependency API and it skips OpenClaw's filesystem guard and policy catalog layers; the safer path is a guarded, catalog-aligned implementation with targeted tests.
