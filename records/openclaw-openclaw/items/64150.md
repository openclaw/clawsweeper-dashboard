## Summary

The PR targets a real current-main OAuth dispatch gap, but it should stay open because the submitted patch has blocking dependency, filesystem-guard, and policy-catalog issues, plus no after-fix real behavior proof from the contributor.

## What This Changes

The PR adds Claude Code OAuth tool-name aliasing for `Glob`/`LS` and registers upstream `find`/`grep`/`ls` discovery tools in the default OpenClaw coding tool set.

## Best Possible Solution

Land a revised implementation that keeps OAuth name aliasing narrow and exposes any new filesystem discovery tools only through guarded factory-created tools, catalog/profile/docs updates, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `fromClaudeCodeName("Glob", tools)` cannot resolve to local `find`, while the default OpenClaw tool construction path does not register `find`, `grep`, or `ls`.

## Solution Assessment

No. The PR points at the right user problem, but the submitted patch is not the best solution because it skips the current dependency factory APIs and bypasses OpenClaw filesystem guard and policy catalog layers.
