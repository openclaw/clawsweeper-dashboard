## Summary

Keep open: the PR targets a real source-reproducible OAuth tool-dispatch bug, but the submitted patch is not correct because it imports unavailable tool constants and exposes raw filesystem discovery tools without OpenClaw workspace/sandbox guards or catalog policy coverage.

## What This Changes

The PR adds Anthropic OAuth Claude Code tool-name aliasing for `Glob`/`LS` and appends upstream `find`/`grep`/`ls` discovery tools to the OpenClaw agent tool set.

## Best Possible Solution

Land a revised implementation that keeps the OAuth alias small and exposes discovery tools only through guarded factory-created tools, catalog/profile/docs updates, and targeted regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with an Anthropic OAuth token, an inbound `tool_use.name === "Glob"` is returned unchanged by `fromClaudeCodeName()`, while the default OpenClaw tool set does not register a local `find` tool.

## Solution Assessment

No. The submitted patch is not the best way to solve the issue because it skips the pinned dependency factories and bypasses the existing filesystem guard and policy catalog layers.
