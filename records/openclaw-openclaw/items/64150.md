## Summary

Keep open: the reported OAuth `Glob` dispatch failure is source-reproducible on current main, but this PR is not correct as-is because it exposes raw filesystem discovery tools outside the existing workspace/sandbox and tool-policy catalog path, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds Anthropic OAuth Claude Code tool-name aliases, adds `LS` to the Claude Code tool list, and registers `find`/`grep`/`ls` discovery tools in the default coding tool set.

## Best Possible Solution

Land a revised fix that keeps the OAuth aliasing small and exposes discovery tools only through guarded factory-created wrappers with catalog/profile/docs/tests aligned, or choose a narrower transport fallback if core discovery tools are not approved.

## Reproduction Assessment

Yes. Source inspection shows that an Anthropic OAuth inbound `tool_use.name === "Glob"` reaches `fromClaudeCodeName()`, which only case-matches registered tools, while current main registers no default local `find`/`grep`/`ls` discovery tools.

## Solution Assessment

No. The alias direction is reasonable, but registering raw discovery tools is not the narrow maintainable fix because it bypasses existing filesystem guard and tool-policy catalog conventions.
