## Summary

Keep open: current main still appears to lack the requested OAuth `Glob` to `find` dispatch path, but this PR is not merge-ready because it imports non-exported tool constants and exposes filesystem discovery tools outside OpenClaw's guard and policy layers.

## What This Changes

This PR adds Anthropic OAuth Claude Code name aliases for `Glob`/`LS` and appends upstream `find`/`grep`/`ls` filesystem discovery tools to the OpenClaw agent tool set.

## Best Possible Solution

Land a revised implementation that keeps the OAuth alias small and exposes discovery tools only through OpenClaw-owned guarded factories, catalog/profile/docs updates, and targeted regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with an Anthropic OAuth token, inbound `tool_use.name === "Glob"` flows through `fromClaudeCodeName()`, current main does not map it to `find`, and the default OpenClaw tool set does not register `find`/`grep`/`ls`.

## Solution Assessment

No. The submitted patch is not the best way to solve the issue because it fails the pinned dependency API and skips filesystem guard/catalog layers; the safer fix is a guarded, catalog-aligned wrapper path with tests.
