## Summary

Keep open. The underlying OAuth `Glob` dispatch problem is still source-reproducible on current main, but this PR is not mergeable as-is because the added discovery tools use unguarded/static filesystem surfaces, are not in the policy catalog, and the external contributor has not posted after-fix real behavior proof.

## What This Changes

Adds Claude Code `LS`/`Glob` name mapping in the Anthropic OAuth transport and appends upstream `find`/`grep`/`ls` discovery tools to the default agent tool set.

## Best Possible Solution

Land a revised fix that keeps the OAuth aliasing, constructs discovery tools from guarded workspace-aware factories, updates policy/catalog/docs/tests, and includes real OAuth proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: with an Anthropic OAuth token, inbound `tool_use.name === "Glob"` reaches `fromClaudeCodeName()`, current main does not map it to `find`, and the default tool set does not register `find`. I did not run a live OAuth session in this read-only review.

## Solution Assessment

No. The alias direction is plausible, but adding raw/static discovery tools is not the safest fix because it bypasses workspace/sandbox guards and policy catalog semantics; the maintainable path is guarded factory-built tools plus catalog and regression coverage.
