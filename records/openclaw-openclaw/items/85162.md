## What This Changes

The PR adds `openclaw models auth paste-api-key`, which reads an API key from prompt or stdin and stores an `api_key` auth profile plus matching config mode, with focused auth command coverage.

## Best Possible Solution

Land the focused API-key paste path with matching docs so headless users can create `api_key` profiles without changing OAuth/token paste behavior.

## Reproduction Assessment

Yes from source inspection: current main's `paste-token` writes token profiles, the API-key resolver consumes `key` on `api_key` profiles, and the Codex app-server bridge treats token profiles as token auth. I did not run a live credentialed Codex request in this read-only review.

## Solution Assessment

Yes, the proposed shape is the narrow maintainable fix: add an explicit API-key paste path using the existing auth-profile schema while leaving `paste-token` semantics alone. The remaining fit issue is documentation, not the core implementation.
