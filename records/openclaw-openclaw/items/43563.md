## Summary

Keep open. Current main has Feishu broadcast handling for platform-delivered inbound group messages, but it still does not implement the requested send-time synthetic fan-out for bot-authored Feishu group sends; the remaining work needs maintainer design on synthetic event semantics, loop limits, dedupe, and overlap with the broader Feishu multi-agent report.

## What This Changes

Add opt-in Feishu cross-bot broadcast so bot-authored group messages are synthetically delivered to other registered OpenClaw Feishu bots in the same group, with loop protection and config.

## Best Possible Solution

Keep this as a Feishu plugin-owned channel design item. If maintainers approve it, implement fan-out only after successful Feishu group sends, mark injected turns as synthetic bot-originated, enforce dedupe and consecutive-bot-turn limits, expose documented opt-in config, and cover both final replies and message-tool sends with focused Feishu tests.

## Reproduction Assessment

Yes. The reporter gives a concrete five-bot Feishu group setup, and current source gives a high-confidence code-level reproduction: outbound Feishu group sends return after the API call and no current path injects a peer-bot inbound event after delivery.

## Solution Assessment

Unclear. Send-time fan-out is a plausible narrow compensation for Feishu's platform behavior, but the best maintainable fix depends on maintainer decisions about synthetic-event metadata, loop safety, dedupe, and how this should relate to existing inbound broadcast groups.
