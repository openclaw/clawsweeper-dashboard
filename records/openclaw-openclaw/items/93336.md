## Summary

Keep open: current main and the latest release still reject raw `sessions.send` payloads that use `sessionKey`, and the linked closing PR is open, unmerged, and still carries the unresolved API-boundary decision.

## What This Changes

The issue reports that subagent `sessions.send` calls using `sessionKey` are rejected by the gateway schema while the caller still reports notification success.

## Best Possible Solution

Trace the raw caller that emits `sessionKey`, then either normalize to canonical `key` at that owner boundary or get explicit maintainer approval for a gateway protocol alias with schema, handler, and regression proof aligned.

## Reproduction Assessment

Yes, at source level: current main validates `sessions.send` against a required `key` schema with `additionalProperties: false`, so a raw `sessionKey` payload is rejected before handler logic. I did not run a live subagent workflow in this read-only review.

## Solution Assessment

Unclear: accepting `sessionKey` at the raw gateway method is a plausible compatibility path, but prior related reviews argue the better fix is to identify and repair the caller emitting the malformed raw payload. The best implementation needs maintainer API-boundary direction first.
