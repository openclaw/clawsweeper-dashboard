## Summary

Keep open: current main and v2026.6.9 still reject a `sessionKey`-only raw `sessions.send` payload before handler logic, and the only linked closing PR is open with unresolved boundary/proof review.

## What This Changes

The issue reports that subagent `sessions.send` calls using `sessionKey` are rejected by the gateway schema and then logged as successful despite failed delivery.

## Best Possible Solution

The end state should have one canonical boundary: either raw `sessions.send` officially accepts and tests `sessionKey`, or the actual raw caller normalizes to `key`, with failed notification responses no longer reported as success.

## Reproduction Assessment

Yes at source level: current main requires `key` in `SessionsSendParamsSchema`, closes the schema with `additionalProperties: false`, and validates before handler logic. I did not run a live current-main subagent that emits the malformed payload.

## Solution Assessment

Unclear: the bug is real, but maintainers still need to choose whether the best fix is caller-side normalization or an approved additive raw RPC alias. The open PR is a candidate, not a completed solution.
