## Summary

Keep open: current main and the latest release still define raw `sessions.send` as `{ key, message }` and validate before handler logic, while the linked open fix candidate is unmerged and still needs maintainer boundary review.

## What This Changes

The issue asks OpenClaw to stop losing explicit subagent `sessions.send` notifications when callers provide `sessionKey` and to avoid reporting failed sends as success.

## Best Possible Solution

Trace the raw caller and either normalize `sessionKey` to canonical `key` at that owner boundary or explicitly approve and test an additive gateway alias, then make failed notifications visible enough to avoid false success reports.

## Reproduction Assessment

Yes, at source level: current main requires `key` in `SessionsSendParamsSchema` and validates before handler logic, so a raw `sessionKey` payload is rejected. I did not run a live current-main subagent that emits the malformed payload.

## Solution Assessment

Unclear: the fix belongs in the caller/protocol boundary, but maintainers still need to choose between preserving raw `sessions.send` as `key` only and approving an additive `sessionKey` alias with tests and proof.
