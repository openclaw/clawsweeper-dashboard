## Summary

Keep open: current main and the latest release still reject raw `sessions.send` payloads that use `sessionKey`, and the exact subagent caller/failure-surfacing path needs maintainer direction before choosing a gateway alias, SDK remap, or caller fix.

## What This Changes

The issue reports that subagent `sessions.send` notifications use `sessionKey` while the gateway RPC requires `key`, causing rejected notifications to be treated as successful.

## Best Possible Solution

Trace the raw subagent caller first, then fix the canonical boundary with focused tests: either remap `sessionKey` to `key` before raw `sessions.send`, or explicitly approve an additive gateway alias and make failed sends tool-visible without leaking noisy chat warnings.

## Reproduction Assessment

Yes, at source level: current `sessions.send` validation requires `key` and rejects `sessionKey` before handler logic. I did not establish a live current-main subagent run that emits the bad raw payload.

## Solution Assessment

Unclear: accepting both names at the gateway is plausible but may be the wrong layer if the bug is an internal caller using the raw RPC contract incorrectly. The best fix depends on maintainer choice between preserving the raw protocol and adding an explicit compatibility alias.
