## Summary

Keep open: current main and the latest release still require canonical `key` for raw `sessions.send`, so `sessionKey`-only payloads remain source-reproducible failures; the open linked PR is the current candidate path but still needs maintainer API-boundary review.

## What This Changes

The issue reports raw `sessions.send` payloads using `sessionKey` being rejected before delivery while the subagent path can still report the failed notify as successful.

## Best Possible Solution

Keep this issue open until maintainers decide the raw RPC contract, then land either a corrected alias implementation with focused proof or a caller-side normalization that prevents malformed raw payloads and reports failed sends accurately.

## Reproduction Assessment

Yes, at source level: current main requires `key` in `SessionsSendParamsSchema`, rejects additional `sessionKey`, and validates before handler logic. I did not run a live subagent workflow that emits the malformed raw payload.

## Solution Assessment

Unclear. A gateway alias is plausible but conflicts with prior maintainer boundary feedback unless explicitly approved; the safer default is to trace the raw caller first and fix the canonical owner boundary.
