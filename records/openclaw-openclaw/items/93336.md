## Summary

Keep open: current main and the latest release still define raw `sessions.send` as `key`-only and validate before handler logic, while the linked fix PR remains open and review-blocked on the API-boundary choice.

## What This Changes

The issue reports that raw `sessions.send` rejects `sessionKey` payloads with `INVALID_REQUEST`, and the subagent-side notification path can treat that failed send as success.

## Best Possible Solution

Review or replace the linked implementation so `sessionKey` is either normalized to canonical `key` at the raw caller boundary or accepted as an approved, tested gateway alias, with failed notifications reported honestly.

## Reproduction Assessment

Yes, at source level: current main requires `key` in `SessionsSendParamsSchema`, closes the schema with `additionalProperties: false`, and validates before handler logic. I did not run a live current-main subagent that emits the malformed payload.

## Solution Assessment

Unclear: the issue needs a maintainer boundary choice between normalizing the raw caller to canonical `key` and approving an additive raw gateway alias. Either path should include focused key-only/sessionKey-only coverage and proof that failed notifications do not self-report as success.
