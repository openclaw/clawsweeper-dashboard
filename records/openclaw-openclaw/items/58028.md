## Summary

Keep this open. Current main still lacks a top-level `/resume` command or an arbitrary previous-session picker/restore flow; the shipped restore surface is limited to compaction checkpoints, and the related direct implementation attempt was closed unmerged while a separate `/restore` PR remains open for maintainer review.

## What This Changes

The issue requests a top-level `/resume` slash command that lists prior sessions and restores a selected or most recent conversation after `/clear` or reset.

## Best Possible Solution

Review or replace the open archived-session restore work with a first-party chat command design that safely lists recent reset/archived sessions, supports selection by id/index/last, restores the transcript into the active session boundary, and updates docs/tests.

## Reproduction Assessment

Yes. The static reproduction path is to inspect the current registry, handler list, docs, reset archive path, and checkpoint restore API; together they show `/resume` is not registered or implemented on current main.

## Solution Assessment

Unclear. A first-party command is the right user-facing direction, but maintainers should settle the `/resume` versus `/restore` naming and security boundary before another automated implementation proceeds.
