## Summary

Keep open. Current main still lacks a top-level `/resume` slash command or previous-session picker/restore flow; the only documented resume command is `/codex resume`, and the shipped Gateway restore API is limited to compaction checkpoints. The exact related implementation PR #36481 is closed unmerged, while adjacent restore/session-management work remains open for maintainer review.

## What This Changes

The issue requests a first-party `/resume` slash command that lists and restores previous conversation sessions after `/clear`, `/new`, or `/reset`.

## Best Possible Solution

Implement or review a focused first-party session resume/restore flow that reconciles this request with open PR #51889: register a clear chat command, list recent reset/archived sessions for the current session scope, support safe selection by id or `last`, restore through Gateway-owned session APIs, and update docs plus targeted command/Gateway tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The high-confidence verification path is static: current command registry, handler loading, docs, and Gateway APIs do not expose the requested top-level `/resume` behavior.

## Solution Assessment

No current-main behavior solves the issue yet. Keeping this open is the best triage outcome because closing it would conflate `/codex resume` or checkpoint restore with arbitrary previous-session restoration.
