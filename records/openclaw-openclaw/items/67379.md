## Summary

Keep open. Current main still has the reported QMD scope failure path: spawned subagent keys are created as `agent:<id>:subagent:<uuid>`, QMD scope normalization drops the `subagent:*` suffix, and the default deny policy rejects the search before QMD runs. The related #70297 context does not supersede this because that PR was closed unmerged and addressed direct-vs-channel defaults, not subagent classification.

## What This Changes

Subagent sessions spawned through `sessions_spawn` should retain an intentional QMD memory-search scope instead of resolving to missing channel/chatType metadata and being denied by default.

## Best Possible Solution

Keep the issue open and implement a narrow, fail-closed QMD scope contract for `agent:<id>:subagent:<uuid>` sessions. The fix should preserve requester intent, add regression coverage for default scope and explicit raw-prefix overrides, and update the QMD memory docs once the chosen semantics are settled.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: evaluate the default QMD scope path for `agent:marketing-coo:subagent:<uuid>` and normalization returns no parsed channel/chatType, causing `isQmdScopeAllowed()` to fall through to `default: "deny"`.

## Solution Assessment

Unclear until maintainers choose the exact subagent inheritance semantics. A narrow requester-scope classification with tests is safer than a broad default `rawKeyPrefix: "agent:"` allow rule because QMD recall is a memory-access boundary.
