## Summary

Keep open. Current main still has the reported QMD scope path: spawned subagent keys are reduced to `subagent:<uuid>`, QMD scope normalization drops them, and the default deny policy rejects the search before QMD runs. The related #70297 context does not supersede this because it concerned direct-vs-channel default policy and was closed unmerged, not subagent key classification.

## What This Changes

The issue asks OpenClaw to make QMD memory scope resolution handle `sessions_spawn` child keys like `agent:<id>:subagent:<uuid>` so spawned subagents keep the intended QMD recall path instead of falling through default-deny.

## Best Possible Solution

Keep the deny-by-default boundary until maintainers choose explicit subagent semantics, then implement a targeted QMD scope classification for `agent:<id>:subagent:<uuid>` that uses parent/requester context or an intentional subagent classification rather than a broad raw-prefix allow. Add mirrored scope tests, memory-manager coverage for denied/allowed subagent searches, and update the memory config docs to match the final policy.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: evaluate the default QMD scope against `agent:marketing-coo:subagent:<uuid>` and the current normalization returns no channel/chatType, causing `isQmdScopeAllowed()` to fall through to default deny.

## Solution Assessment

Unclear until a maintainer chooses the exact inheritance semantics. The safest direction is a narrow classification/inheritance fix with tests; a blanket `rawKeyPrefix: "agent:"` default allow is not the best solution because it can broaden memory access beyond the requester’s original surface.
