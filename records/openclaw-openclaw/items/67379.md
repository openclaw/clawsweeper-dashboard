## Summary

Keep open. Current main still has the reported QMD scope failure for `agent:<id>:subagent:<uuid>` sessions, and the latest release still contains the same subagent-normalization branch; fixing it changes a memory-access boundary, so this needs maintainer semantics rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to preserve usable QMD memory search for `sessions_spawn` subagent sessions instead of resolving their scope to unknown channel/chatType and denying the search.

## Best Possible Solution

Define a fail-closed subagent QMD scope contract that preserves the requester/session classification, then add focused QMD scope and manager regression tests plus any needed memory-scope docs cleanup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_spawn` creates `agent:<id>:subagent:<uuid>`, QMD normalization drops that key to no parsed scope, and the default-deny QMD scope rejects the search before QMD runs.

## Solution Assessment

Unclear until maintainers choose the memory-access semantics. Preserving requester scope or adding explicit subagent classification is safer than a broad raw `agent:` allow rule.
