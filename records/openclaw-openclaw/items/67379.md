- `impact:session-state`: The failure comes from subagent session-key classification losing the session context needed for QMD memory recall.

## Summary

Keep open. Current main and the latest release still normalize `agent:<id>:subagent:<uuid>` to `subagent:*` and then drop it, so default-deny QMD scope rejects subagent recall; the remaining fix needs an intentional memory-scope policy choice.

## What This Changes

The issue requests fixing QMD scope parsing so subagent sessions spawned by `sessions_spawn` can use QMD memory search instead of being denied and falling back.

## Best Possible Solution

Define fail-closed subagent QMD scope semantics that preserve requester or parent classification, then add focused regression coverage for default scope, subagent keys, and explicit raw-prefix overrides.

## Reproduction Assessment

Yes, at source level. `sessions_spawn` emits `agent:<id>:subagent:<uuid>`, QMD normalization strips that to `subagent:*` and returns no scope, and the default-deny direct rule rejects the search.

## Solution Assessment

Unclear until maintainers choose the memory-access semantics. Preserving requester or parent scope, or adding explicit subagent classification, is safer than a broad raw `agent:` allow rule.
