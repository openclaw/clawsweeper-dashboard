## Summary

Keep open. Current main still drops `agent:<id>:subagent:<uuid>` keys during QMD scope normalization, so spawned subagents get no parsed channel/chatType and default-deny QMD scope blocks search; the remaining fix needs maintainer judgment because it changes a memory-access boundary.

## What This Changes

Subagent QMD searches should resolve an intentional scope instead of falling through default deny for `agent:<id>:subagent:<uuid>` session keys.

## Best Possible Solution

Define a fail-closed subagent QMD scope contract that carries the requester/session classification into QMD search, then add regression coverage and align the QMD scope docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_spawn` creates `agent:<id>:subagent:<uuid>`, QMD normalization strips the agent prefix and returns no parsed scope for `subagent:*`, then default-deny scope rejects the search.

## Solution Assessment

Unclear until maintainers choose the memory-access semantics. Scope inheritance or explicit subagent classification looks safer than a blanket raw `agent:` allow rule.
