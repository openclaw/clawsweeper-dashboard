## Summary

Keep open. Current main and the latest release still drop `agent:<id>:subagent:<uuid>` keys during QMD scope parsing, so default-deny QMD scope rejects subagent memory search; the fix should be chosen carefully because it changes a memory-access boundary.

## What This Changes

Subagent sessions spawned with `sessions_spawn` should retain enough QMD scope context for memory search instead of resolving to unknown channel/chatType and falling through default-deny.

## Best Possible Solution

Define an intentional, fail-closed QMD scope contract for subagents that preserves requester or parent session classification, then add focused scope and manager regression coverage.

## Reproduction Assessment

Yes, at source level. `sessions_spawn` creates `agent:<id>:subagent:<uuid>`, QMD normalization drops that key to no parsed scope, and default-deny QMD scope rejects it before QMD search runs.

## Solution Assessment

Unclear until maintainers choose the memory-access semantics. Preserving requester/parent scope or adding explicit subagent classification is safer than a broad raw `agent:` allow rule.
