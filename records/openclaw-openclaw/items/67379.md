## Summary

Keep open. Current main and the latest release still have the reported QMD scope failure for `agent:<id>:subagent:<uuid>` sessions, and the fix needs an explicit maintainer decision because it changes a memory-access boundary.

## What This Changes

The issue asks for QMD memory search to work from `sessions_spawn` subagent sessions whose keys use `agent:<id>:subagent:<uuid>`.

## Best Possible Solution

Define a fail-closed subagent QMD scope contract that preserves requester channel/chatType classification, then add focused regression tests and any needed memory-scope docs update.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `sessions_spawn` creates `agent:<id>:subagent:<uuid>`, QMD normalization drops the `subagent:*` suffix after stripping the agent prefix, and default-deny scope rejects the search before QMD runs.

## Solution Assessment

Unclear until maintainers choose the memory-access semantics. Preserving requester scope or adding explicit subagent classification is safer than a broad raw `agent:` allow rule.
