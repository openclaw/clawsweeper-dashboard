## Summary

Keep open. Current main still has the reported QMD scope failure path for spawned subagents, and the related #70297 context does not supersede it because that PR was closed unmerged and addressed direct-vs-channel defaults rather than subagent classification.

## What This Changes

Subagent sessions spawned via `sessions_spawn` should be able to use QMD memory search without resolving to unknown channel/chatType and being denied by the default scope.

## Best Possible Solution

Keep this open and implement an explicit, fail-closed QMD scope contract for `agent:<id>:subagent:<uuid>` sessions, with requester-aware semantics and regression coverage for default scope plus rawKeyPrefix overrides.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: evaluate default QMD scope for `agent:marketing-coo:subagent:<uuid>` and normalization returns no parsed channel/chatType, so `isQmdScopeAllowed()` falls through to `default: "deny"`.

## Solution Assessment

Unclear until maintainers choose the subagent inheritance semantics. A narrow requester-scope classification is safer than a blanket `rawKeyPrefix: "agent:"` allow rule because QMD recall is a memory-access boundary.
