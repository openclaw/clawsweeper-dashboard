## Summary

Keep open. Current main and the latest release still have the reported QMD scope failure for `agent:<id>:subagent:<uuid>` sessions, and the safe fix changes a memory-access policy boundary that needs maintainer semantics rather than cleanup closure.

## What This Changes

The issue reports that spawned subagent session keys lose QMD scope classification and are denied by the default QMD scope.

## Best Possible Solution

Define a fail-closed subagent QMD scope contract that preserves requester or parent session classification, then add focused QMD scope and manager regression tests.

## Reproduction Assessment

Yes. Source inspection shows `sessions_spawn` creates `agent:<id>:subagent:<uuid>`, QMD normalization drops that key to no parsed scope, and default-deny QMD scope rejects it before QMD search runs.

## Solution Assessment

Unclear until maintainers choose the memory-access semantics. Preserving requester scope or adding explicit subagent classification is safer than a broad raw `agent:` allow rule.
