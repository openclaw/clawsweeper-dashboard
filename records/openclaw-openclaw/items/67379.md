## Summary

Keep open. Current main still has the reported QMD scope failure for `agent:<id>:subagent:<uuid>` sessions, and the safe fix needs maintainer choice because it changes a memory-access boundary.

## What This Changes

The issue requests that QMD memory search work for spawned subagent sessions instead of being denied after their session keys normalize to no channel/chatType.

## Best Possible Solution

Define a fail-closed subagent QMD scope contract that carries requester/session classification into QMD search, then add focused regression coverage and any necessary scope documentation update.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: `sessions_spawn` creates `agent:<id>:subagent:<uuid>`, QMD normalization strips the agent prefix and drops `subagent:*`, then default-deny scope rejects the search before QMD runs.

## Solution Assessment

Unclear until maintainers choose the memory-access semantics. Inheriting requester scope or adding explicit subagent classification looks safer than a blanket `rawKeyPrefix: "agent:"` default allow rule.
