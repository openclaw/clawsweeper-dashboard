## Summary

Close: current main already contains the Telegram isolated-polling lane drain fix that lets different forum-topic lanes enter handling concurrently, and related live proof shows the same v2026.5.12 symptom no longer reproduces with that fix applied.

## What This Changes

Telegram forum-topic messages in one supergroup should run as separate sessions and use `agents.defaults.maxConcurrent` instead of serializing one topic at a time.

## Best Possible Solution

Keep the merged isolated-polling lane-drain implementation and direct affected stable users to v2026.5.14-beta.1 or a later release containing it.

## Reproduction Assessment

Yes for the affected v2026.5.12 path: the related live Telegram report reproduces the serial topic behavior and shows it disappearing after applying the lane-drain fix. I did not reproduce a failing path on current main because the fix is already present there.

## Solution Assessment

Yes. Reusing the existing Telegram sequential key in the isolated polling spool drain preserves same-topic ordering while letting unrelated topics, DMs, and control lanes proceed independently, which is narrower than adding new concurrency settings or changing session semantics.
