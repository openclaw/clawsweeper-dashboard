## Summary

Current main already removes the shared auto-managed runtime-deps pruning class: bundled plugin dependency staging is now legacy cleanup, gateway startup does not install plugin dependencies, and the memory packages are represented by root/plugin manifests instead of a per-plugin pruning pass.

## What This Changes

The issue reports that memory-core and other plugin runtime dependencies are installed into a shared plugin-runtime-deps tree and then pruned by later plugin lifecycle passes, breaking cold imports.

## Best Possible Solution

Keep the current manifest-backed dependency model and legacy cleanup path; older 2026.4.x and 2026.5.x users should move to a build containing current main rather than patching plugin-runtime-deps trees manually.

## Reproduction Assessment

Yes for the affected release line: the issue body and follow-up give concrete disappearing node_modules and import-failure evidence, and related reports show the shared-prune failure mode across plugins. I did not live-run an old packaged install, but current main source no longer has the startup staging/prune path that caused it.

## Solution Assessment

Yes. Removing runtime package-manager mutation and relying on package manifests plus explicit install/update/doctor flows is narrower and safer than trying to make competing per-plugin pruning manifests converge.
