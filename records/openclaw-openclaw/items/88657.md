## Summary

Keep open: current main and v2026.6.9 still skip non-visible retry once tool activity marks the attempt replay-unsafe, then surface the same verify-before-retrying incomplete-turn warning. Recent merged work covers adjacent replay-safe empty-final and finish-reason/tool-call cases, but the central replay-unsafe post-tool recovery policy remains unresolved.

## What This Changes

The issue reports DeepSeek/OpenRouter post-tool agent turns ending with `payloads=0`, `tools>0`, `replaySafe=no`, and an incomplete-turn error instead of a usable assistant reply.

## Best Possible Solution

Define a maintainer-approved replay-safety policy for `tools>0 payloads=0` post-tool turns, then add narrow embedded-runner recovery and tests for safe cases while keeping side-effecting attempts fail-closed.

## Reproduction Assessment

Yes, at source level: current main still suppresses non-visible retries when replay metadata has potential side effects, then returns the reported incomplete-turn warning. I did not run a live DeepSeek/OpenRouter turn in this read-only review.

## Solution Assessment

No for the requested blanket retry when `replaySafe=no`; that could duplicate tool side effects. The maintainable solution is a narrow replay policy that proves which post-tool attempts are safe before retrying.
