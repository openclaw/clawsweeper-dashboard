## Summary

Keep open: current main and v2026.6.9 still suppress non-visible retries whenever the attempt has potential side effects, then surface the same incomplete-turn warning for the reported `payloads=0 tools>0 replaySafe=no` state. The newly linked DeepInfra PR is useful partial provider work, but it does not solve the OpenRouter or generic replay-unsafe runner policy.

## What This Changes

The issue requests recovery for DeepSeek/OpenRouter post-tool incomplete turns where `payloads=0`, `tools>0`, and `replaySafe=no` currently surface an error without retrying.

## Best Possible Solution

Keep the fail-closed default for side-effectful tools, but define a maintainer-approved recovery rule for `payloads=0 tools>0` post-tool turns and add focused runner/provider coverage for the safe cases.

## Reproduction Assessment

Yes, at source level: current main still skips non-visible retries when replay metadata records side effects, then returns the reported incomplete-turn warning. I did not run a live DeepSeek/OpenRouter reproduction in this read-only review.

## Solution Assessment

No for the requested blanket retry when `replaySafe=no`; that would risk replaying side-effectful tools. The better fix is a narrow, maintainer-approved replay policy that proves which post-tool attempts are safe before retrying.
