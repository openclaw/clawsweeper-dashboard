## Summary

Keep open: current main still intentionally refuses the reported retry path when the attempt is not replay-safe, and the related OpenRouter/DeepSeek issues in the provided context are older closed issues that do not own this specific remaining behavior.

## What This Changes

The issue reports that DeepSeek V4 Flash via OpenRouter can emit partial tool-call JSON that OpenClaw classifies as an incomplete turn and does not retry because the turn is not replay-safe.

## Best Possible Solution

Keep the side-effect safety gate by default, then fix the OpenRouter/DeepSeek partial-tool-call path narrowly with live provider proof and regression coverage for safe retries or provider classification.

## Reproduction Assessment

No high-confidence live reproduction is available from this read-only review. Source inspection does reproduce the reported no-retry branch for zero-payload incomplete turns after replay-unsafe tool activity.

## Solution Assessment

No, a blanket retry for `replaySafe=no` is not the best fix because it can duplicate side effects. The safer direction is provider/finish-reason handling plus retry only when the tool history is proven replay-safe.
