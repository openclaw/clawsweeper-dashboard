## Summary

Keep open: current main still matches the reported Slack presentation downgrade. The shared CLI/core path preserves `presentation`, and Slack has a Block Kit renderer, but the active Slack channel outbound facade does not expose that renderer to core delivery, so core falls back to text before Slack can render Block Kit.

## What This Changes

The issue requests that Slack `presentation` payloads from `openclaw message send` render as Slack Block Kit instead of plain text.

## Best Possible Solution

Expose Slack's existing `presentationCapabilities` and `renderPresentation` through the active `slackChannelOutbound` facade, then add focused coverage for the core delivery path so `presentation` reaches Block Kit before fallback text.

## Reproduction Assessment

Yes, source-reproducible with high confidence: `presentation` is parsed into the outbound payload, but the active Slack delivery handler lacks a presentation renderer, causing core to strip it into fallback text. I did not run a live Slack workspace send in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to bridge Slack's existing renderer and capability declaration into the active outbound facade. Changing core fallback semantics or adding a config option would be broader than needed for the documented contract.
