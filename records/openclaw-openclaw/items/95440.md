## Label Justifications

- `P2`: The bug deterministically breaks a documented Slack rich-message path, but the blast radius is limited to Slack presentation payloads rather than all message delivery.
- `impact:message-loss`: Structured presentation blocks are silently lost by downgrading the payload to plain text before Slack Block Kit rendering.

## Summary

Keep open: current main still has the reported Slack presentation downgrade. The CLI/core send path can carry `presentation`, and the Slack renderer exists, but the active Slack channel outbound facade does not expose `presentationCapabilities` or `renderPresentation`, so core falls back to text before Slack can build Block Kit.

## What This Changes

The issue reports that Slack `presentation` payloads sent through `openclaw message send` are delivered as plain text instead of Block Kit cards.

## Best Possible Solution

Expose Slack's existing presentation capabilities and renderer through the active Slack channel outbound facade so core delivery renders Block Kit before fallback text is considered, then add regression coverage for the CLI/core send path returning a card receipt.

## Reproduction Assessment

Yes, source-reproducible. The documented CLI path carries `presentation` into the send payload, but current main's active Slack delivery handler lacks a presentation renderer, so `renderPresentationForDelivery` strips the presentation and sends fallback text.

## Solution Assessment

Yes, the narrow maintainable fix is to bridge Slack's existing renderer/capabilities into the active Slack channel outbound facade. Adding a new config option or changing core fallback behavior would be broader than needed for this documented contract.
