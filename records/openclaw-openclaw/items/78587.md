## Summary

Keep open: the PR has a plausible reliability direction, but it currently needs real behavior proof and blocks on a compatibility regression where existing `gateway.reload.deferralTimeoutMs` restart-drain settings would be ignored.

## What This Changes

The PR adds a `gateway.restart` drain policy, zombie task promotion, diagnostic event, docs, changelog, and tests for bounded gateway restart recovery.

## Best Possible Solution

Keep the bounded zombie promotion design only if it preserves the existing restart-drain config contract, clearly documents any deprecation path, and is backed by a real gateway restart proof artifact.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `gateway.reload.deferralTimeoutMs` and trigger a direct or config-driven gateway restart; current main reads that key, while the PR readers use only `gateway.restart`. I did not run the PR because this review is read-only, but the changed call sites are explicit.

## Solution Assessment

No. The maintainable path is to bridge or migrate `gateway.reload.deferralTimeoutMs` while adding `gateway.restart`, then prove the real restart/zombie path with logs or terminal output.
