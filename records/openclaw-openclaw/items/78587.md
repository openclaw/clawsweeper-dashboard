## Summary

Keep open: the PR targets a real gateway restart failure corroborated by user runtime logs, but the patch still drops the existing public restart-deferral config on both direct and config-triggered restarts and lacks after-fix real behavior proof from the PR branch.

## What This Changes

The PR adds a `gateway.restart` drain policy/config, promotes stale running tasks to failed during gateway restart drains, emits diagnostics/logging, and updates docs, config schema, changelog, and tests.

## Best Possible Solution

Bridge `gateway.reload.deferralTimeoutMs` into the new restart policy or define an explicit migration/deprecation path, keep zombie promotion scoped to genuinely stale running tasks, and require redacted real restart logs before merge.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows current main reads `gateway.reload.deferralTimeoutMs` for both direct and config-triggered restarts, while the PR reads only `gateway.restart`. The underlying zombie-task restart problem is corroborated by v2026.5.7 runtime logs in a PR comment, but the after-fix PR behavior has not been demonstrated.

## Solution Assessment

No. The bounded drain and zombie promotion direction is plausible, but the implementation needs to preserve the existing public config contract and provide real after-fix gateway restart proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve the existing restart deferral config:** `src/cli/gateway-cli/run-loop.ts:280`
  - body: This switches direct restart drains to read only `gateway.restart`, so an existing config like `gateway.reload.deferralTimeoutMs: 0` no longer gives an indefinite wait and custom millisecond budgets are ignored. Users who set `0` to wait indefinitely or a custom timeout would unexpectedly fall back to the new default, so this path needs the same compatibility bridge.
  - confidence: 0.9
