## Best Possible Solution

Land the resolver-based fix that derives Feishu group session/tool context from the channel-owned chat id while preserving speaker attribution, then close or supersede the overlapping alternative PR.

## Reproduction Assessment

Yes, at source level. Current main Feishu group turns use `From: feishu:<sender>` with `OriginatingTo`/`To: chat:<group>`, while `resolveGroupSessionKey` currently derives the group id from `From`, matching the linked warning path.

## Solution Assessment

Yes. The resolver-based fix is the narrowest maintainable path because `MsgContext` has no supported `GroupId` field and changing `From` would lose the speaker-shaped transcript identity; it preserves the tool-policy fail-closed guard.
