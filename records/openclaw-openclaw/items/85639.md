## Summary

Keep open: current main still has a source-visible path where stale-session recovery can report `queued_work_without_active_run` and then skip with `active_reply_work`, and the open related PR only targets one likely orphaned-reply-operation root cause rather than proving the Telegram group wedge is fully fixed.

## What This Changes

Investigate and fix Telegram group session lanes that queue messages indefinitely when stuck-session recovery treats stale `activeSessionId` reply work as active.

## Best Possible Solution

Keep the issue open while maintainers either land or expand https://github.com/openclaw/openclaw/pull/80427, then add coverage that stale `active_reply_work` cannot indefinitely block a queued channel lane under the existing stuck-session abort threshold.

## Reproduction Assessment

Yes at source level, but not by live execution: current main has the stale-session request path without `allowActiveAbort` and the recovery branch that skips with `active_reply_work` when the reply registry still says the session is active.

## Solution Assessment

Yes directionally: recovery should not trust stale active reply state forever; the narrow reset-cleanup PR is a likely part of the fix, but the final solution should prove Telegram/group lanes recover and add regression coverage for the stuck recovery path.
