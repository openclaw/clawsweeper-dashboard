## Summary

## What This Changes

The issue reports Telegram group session lanes staying in `processing` because stale `active_reply_work` prevents stuck-session recovery from freeing queued messages.

## Best Possible Solution

Keep the issue open while maintainers land or expand https://github.com/openclaw/openclaw/pull/80427, then add coverage proving stale `active_reply_work` cannot indefinitely block a queued channel lane under the existing stuck-session abort threshold.

Yes at source level, but not by live execution: current main has the stale-session request path without `allowActiveAbort` and a recovery branch that skips with `active_reply_work` when the reply registry still says the session is active.

## Solution Assessment

Yes directionally: recovery should not trust stale active reply state forever; the narrow reset-cleanup PR is a likely part of the fix, but the final solution should prove Telegram group lanes recover and add regression coverage for the stuck recovery path.
