## Summary

Keep open. Current main still leaves `zai/glm-4.6v` image requests on the generic OpenAI-compatible image completion path, with the same abort-signal and streaming behavior described by the report, and I found no merged ZAI-specific endpoint, payload, transport, or abort fix. The duplicate abort report and related 404 route issue add useful corroboration, but the open route PR is not merged or proven against this abort symptom.

## What This Changes

The issue reports that `agents.defaults.imageModel = "zai/glm-4.6v"` makes the image tool fail with `Request was aborted` even though direct Z.ai API calls with the same image and API key succeed.

## Best Possible Solution

Keep this open as the canonical ZAI `glm-4.6v` abort report. The best end state is a focused regression for the image tool against the reported ZAI coding-endpoint configuration, followed by the narrowest ZAI-owned provider fix or a documented shared image transport seam if the failure proves generic across OpenAI-compatible providers.

## Reproduction Assessment

Yes. The issue and duplicate #66608 give a concrete path: configure the ZAI coding endpoint, set `agents.defaults.imageModel` to `zai/glm-4.6v`, and send an image through Telegram or the image tool; current main still follows that reported generic completion path.

## Solution Assessment

Unclear. The route-only approach in #70915 is plausible for the related 404 reports, but this abort report needs live or mocked proof before choosing endpoint rewriting, payload transformation, streaming settings, or timeout handling as the maintainable fix.
