## Summary

Keep open. Current main still advertises `zai/glm-4.6v` image understanding and routes it through the generic OpenAI-compatible streaming image path, with no merged ZAI-specific endpoint, payload, streaming, or abort fix for the reported failure; #66608 corroborates the same abort symptom, while #70886/#70915 are related but currently focused on a different 404 route theory.

## What This Changes

The issue reports that ZAI `glm-4.6v` image-tool calls abort through OpenClaw even though direct ZAI API calls with the same image and key succeed.

## Best Possible Solution

Keep this as the canonical ZAI `glm-4.6v` abort report, add focused proof for the coding-endpoint image path, then land the narrowest ZAI-owned provider fix or documented shared image transport seam that the proof supports.

## Reproduction Assessment

Yes. The issue and duplicate #66608 give a concrete path: configure the ZAI coding endpoint, set `agents.defaults.imageModel` to `zai/glm-4.6v`, and send an image through the image tool; current main still follows that provider-to-generic-streaming path.

## Solution Assessment

Unclear. A route-only change like #70915 may be right for 404 reports, but this abort report includes direct coding-endpoint success, so the maintainable fix needs live or mocked proof before choosing endpoint rewriting, streaming settings, payload transformation, or abort handling.
