## Summary

Keep open. The PR targets a real, source-backed bug in the non-verbose external-chat failure path, but the proposed classifier would now label generic CLI timeout strings from any CLI backend as Claude-specific.

## What This Changes

The PR adds a non-verbose external-chat classifier for CLI no-output and overall timeout messages, tests it in the agent reply path, and records the user-facing fix in CHANGELOG.

## Best Possible Solution

Land the narrow timeout-classification fix with provider-aware metadata or backend-neutral wording, leaving timeout defaults and broader diagnostics to separate follow-up work.

## Reproduction Assessment

Yes. Source inspection shows current main can reproduce the linked bug by rejecting a non-verbose external-chat run with a generated CLI timeout string, and the introduced regression can be reproduced by using the same generic timeout string for a non-Claude CLI backend.

## Solution Assessment

No, not yet. The carve-out is the right narrow direction, but using Claude-specific copy from generic CLI timeout text alone is not the most maintainable fix.

## Repair Work Prompt

On PR #77015, keep the narrow external-chat CLI timeout fix, but repair the provider-specific wording. The generic `CLI produced no output...` and `CLI exceeded timeout...` strings come from `src/agents/cli-runner/execute.ts` for any CLI backend, so do not render `Claude CLI backend` based on message text alone. Prefer passing typed FailoverError provider/model metadata into `buildExternalRunFailureReply` and only using Claude-specific copy when the provider/backend is known to be Claude, or use backend-neutral `CLI backend` wording for generic runner strings. Add regression coverage in `src/auto-reply/reply/agent-runner-execution.test.ts` proving a non-Claude CLI timeout does not mention Claude, while preserving the Claude timeout coverage. Do not change timeout defaults, auth, model fallback policy, or Telegram delivery behavior.
