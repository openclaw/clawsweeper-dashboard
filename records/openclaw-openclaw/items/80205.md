## Summary

Keep open: current main still has managed npm peer-link repair in install/update/uninstall paths, but it does not directly audit missing or wrong `node_modules/openclaw` peer links in `openclaw plugins doctor`, and gateway startup still loads runtime plugins without a peer-link repair pass. The related doctor and boot-time repair PRs are open and cover parts of the request, so this should stay available for coordinated maintainer follow-up.

## What This Changes

Ensure managed npm plugins that declare `peerDependencies.openclaw` have missing, stale, or wrong plugin-local peer links diagnosed and repaired before Discord/gateway runtime failures surface as `configured_unavailable`.

## Best Possible Solution

Coordinate the open doctor and startup repair PRs, or replace them with one shared peer-link audit/repair helper that covers `plugins doctor`, `doctor --fix`, and any maintainer-approved gateway startup repair policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a managed npm plugin declaring `peerDependencies.openclaw` with a missing or wrong `node_modules/openclaw` link, then current `plugins doctor` has no direct peer-link audit and gateway startup loads runtime plugins without repairing it.

## Solution Assessment

Unclear. The audit/repair helper and doctor diagnostics are the right direction, but startup-time repair needs maintainer confirmation because current docs say startup does not repair dependency trees.
