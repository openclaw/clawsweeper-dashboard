## Summary

Keep open: current main still repairs managed npm `openclaw` peer links only in install/update/uninstall paths. It does not audit these links in `openclaw plugins doctor`, wrong-target links can pass as healthy, and gateway startup still loads runtime plugins without pre-load peer-link repair; the related open PR covers the doctor path but not the startup/diagnostic decision behind the Discord outage.

## What This Changes

The issue asks OpenClaw to detect and repair managed npm plugin `openclaw` peer-link drift so Discord and other SDK-using plugins do not degrade to `configured_unavailable`.

## Best Possible Solution

Consolidate with https://github.com/openclaw/openclaw/pull/77412 or a focused follow-up that adds one canonical peer-link audit/repair helper, wires diagnostics into `plugins doctor` and `doctor --fix`, detects wrong-target links, and only adds startup repair if maintainers accept that runtime mutation boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a managed npm plugin declaring `peerDependencies.openclaw` with a missing or wrong-target `node_modules/openclaw` link, then current `plugins doctor` has no direct audit and gateway startup loads plugins without repairing the link.

## Solution Assessment

Unclear as written. Doctor diagnostics and a shared repair helper are the narrow maintainable path, but startup-time repair conflicts with the documented verify-only startup model and should be an explicit maintainer decision; if startup repair is rejected, runtime diagnostics still need to point at the broken plugin peer link.
