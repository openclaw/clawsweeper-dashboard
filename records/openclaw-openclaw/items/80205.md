## Summary

Keep open: current main still has the reported gaps in `openclaw plugins doctor`, gateway startup, and wrong-target peer-link detection. The related open PRs cover parts of the recovery story, but neither has merged and they do not fully cover the Discord diagnostic failure yet.

## What This Changes

Report that managed npm plugin `openclaw` peer links can drift missing or stale, causing Discord contract imports to fail and tokens to show `configured_unavailable`.

## Best Possible Solution

Consolidate this with the open doctor and boot-repair work around one canonical peer-link audit/repair helper, then explicitly decide whether startup-time repair is allowed or should remain a diagnostic-only path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a managed npm plugin declaring `peerDependencies.openclaw` with a missing or wrong `node_modules/openclaw` link, then current `plugins doctor` has no direct audit and gateway startup loads runtime plugins without repairing it. I did not run the live reproduction because this review is read-only.

## Solution Assessment

Unclear as a complete fix: a canonical audit/repair helper and doctor diagnostics are the right direction, but startup-time repair needs maintainer confirmation against the documented no-runtime-repair boundary. A focused follow-up should reuse the existing repair paths and avoid adding another independent peer-link checker.
