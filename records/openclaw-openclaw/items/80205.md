## Summary

Keep open: current main still lacks direct managed-npm `openclaw` peer-link audit in `openclaw plugins doctor`, does not repair peer links before gateway runtime plugin loading, and treats any existing `node_modules/openclaw` path as healthy even if it points at the wrong host package.

## What This Changes

The issue asks OpenClaw to audit and repair missing or stale npm-managed plugin `openclaw` peer links so Discord does not degrade to `configured_unavailable` after plugin dependency drift.

## Best Possible Solution

Consolidate this with the open doctor and startup repair work into one canonical peer-link audit/repair helper, wire read-only diagnostics into `plugins doctor`, and only add gateway startup repair if maintainers approve that runtime mutation boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: use a managed npm plugin declaring `peerDependencies.openclaw` with a missing or wrong `node_modules/openclaw` link, then current `plugins doctor` has no direct audit and gateway startup reaches runtime plugin loading without repairing it.

## Solution Assessment

Unclear as written. The reusable audit/repair helper and doctor diagnostics are the narrow maintainable direction, but startup-time repair needs maintainer confirmation because it changes the documented startup dependency-repair boundary.
