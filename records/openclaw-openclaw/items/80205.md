## Summary

Keep open: current main still has peer-link repair in install/update/uninstall paths, but it does not audit missing or wrong-target links in `openclaw plugins doctor` and does not repair managed npm peer links before gateway runtime plugin loading. The related doctor and boot-time repair PRs are still open, so the remaining work needs maintainer coordination rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to diagnose and repair managed npm plugin `openclaw` peer-link drift so Discord tokens do not degrade to `configured_unavailable` when plugin runtime imports cannot resolve the host package.

## Best Possible Solution

Consolidate the doctor and startup paths around one peer-link audit/repair helper, wire diagnostics into `plugins doctor` and `doctor --fix`, and land boot-time repair only if maintainers accept that startup mutation boundary.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a managed npm plugin declaring `peerDependencies.openclaw` with a missing or wrong `node_modules/openclaw` link, then current `plugins doctor` has no direct audit and gateway startup loads runtime plugins without repairing it. I did not run the live reproduction because this review is read-only.

## Solution Assessment

Unclear as a complete fix direction: canonical audit/repair plus doctor diagnostics is the right path, but startup-time repair needs maintainer confirmation against the current dependency-resolution contract. The safest route is to coordinate the open doctor and boot-time repair PRs rather than opening a third overlapping fix.
