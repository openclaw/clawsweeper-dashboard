## Summary

Keep open: current main repairs npm peer links during install/update/uninstall, but `plugins doctor` and gateway startup still do not audit or repair missing/stale plugin-local `openclaw` peer links. The report is source-reproducible; the startup auto-repair part needs owner judgment because current docs keep dependency repair out of runtime startup.

## What This Changes

The issue requests doctor/startup handling for drifted npm plugin `node_modules/openclaw` peer links so Discord token contract imports do not degrade into `configured_unavailable`.

## Best Possible Solution

Add a managed npm peer-link audit that detects missing, unreadable, and wrong-target host links, wire it into plugin diagnostics/doctor repair, and decide whether startup should auto-repair or emit a clear pre-runtime diagnostic consistent with the dependency-resolution contract.

## Reproduction Assessment

Yes at source level: current main has the peer-link helper, but `plugins doctor` and gateway startup do not call it, and the existing update probe only checks that `node_modules/openclaw` exists. I did not run the full Discord gateway incident locally.

## Solution Assessment

Unclear for the full proposed direction. Doctor/audit coverage is the narrow maintainable fix, but automatic gateway startup repair should be confirmed by plugin/runtime owners against the documented startup dependency contract.
