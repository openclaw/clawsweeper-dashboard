## Summary

Keep open: current main has peer-link repair in install/update/uninstall paths, but it does not audit these links in `openclaw plugins doctor`, does not repair before gateway plugin runtime load, and treats wrong-target links as healthy. The overlapping open PR only covers `openclaw doctor --fix`, so it does not fully cover the Discord startup/diagnostic failure reported here.

## What This Changes

This issue requests managed npm plugin `openclaw` peer-link audit and repair so Discord secret contract imports do not degrade configured tokens to `configured_unavailable`.

## Best Possible Solution

Consolidate this with https://github.com/openclaw/openclaw/pull/77412 or a focused follow-up that adds one canonical peer-link audit/repair helper, wires diagnostics into `plugins doctor` and `doctor --fix`, and only adds startup repair if maintainers approve that runtime mutation boundary.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a managed npm plugin declaring `peerDependencies.openclaw` with a missing or wrong `node_modules/openclaw` link, then current `plugins doctor` has no direct audit and gateway startup loads plugins without repairing it. I did not run the live reproduction because this review is read-only.

## Solution Assessment

Unclear as written: the audit/repair helper and doctor diagnostics are the right direction, but startup-time repair needs maintainer confirmation because current docs say startup does not repair dependency trees. The safest path is to reuse one canonical helper and coordinate with the existing open doctor repair PR.
