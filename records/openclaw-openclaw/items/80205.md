## Summary

Keep open. Current main repairs npm plugin `openclaw` peer links during install/update/uninstall paths, but `openclaw plugins doctor` and gateway startup still do not audit or repair missing/stale managed npm peer links before runtime plugin loading.

## What This Changes

Request that OpenClaw audit and repair npm-managed plugin `node_modules/openclaw` peer-link drift in plugin diagnostics and gateway startup so Discord tokens do not degrade to `configured_unavailable`.

## Best Possible Solution

Add managed npm peer-link audit/repair as a shared plugin maintenance primitive, surface broken links in `openclaw plugins doctor`, and run a lightweight managed-root repair before gateway runtime plugin loading.

## Reproduction Assessment

Yes, at source level. A managed npm plugin that declares `peerDependencies.openclaw` but has a missing or wrong `node_modules/openclaw` link will not be repaired by current gateway startup or reported by current plugin doctor, and the reporter supplied a live Discord failure and recovery trace.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing peer-link repair boundary, extend it with audit/wrong-target detection, and wire it into managed npm plugin diagnostics and pre-runtime gateway startup without adding a new config flag.

Managed npm plugin `node_modules/openclaw` peer links can be missing or stale after drift, leaving Discord secret/runtime imports unable to resolve the host package.
