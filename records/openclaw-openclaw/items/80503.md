## Summary

Keep open: current main still follows the package-entry path for global plugins when `package.json#openclaw.extensions` names a TypeScript entry without a built JavaScript peer, while docs and tests intentionally enforce the compiled-runtime contract. The report is actionable, but the right fix requires a plugin-loader contract decision rather than conservative cleanup.

## What This Changes

Request that plugin discovery handle TS-source local plugins with both `package.json` and `openclaw.plugin.json` without silently dropping them under managed-package validation.

## Best Possible Solution

Maintainers should define the package-shaped local plugin contract, then either allow a manifest/index TypeScript fallback for explicit local-development paths or keep the compiled-runtime rule and surface louder diagnostics plus docs guidance.

## Reproduction Assessment

Yes. Source inspection on current main shows a global plugin directory whose `package.json#openclaw.extensions` points at a TypeScript entry without a built peer takes the package path and is excluded with the reported warning; I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear. Preferring `openclaw.plugin.json` may be the right operator experience, but it changes the current managed-package runtime-output contract, so maintainers should choose the contract first and then add focused discovery tests/docs.
