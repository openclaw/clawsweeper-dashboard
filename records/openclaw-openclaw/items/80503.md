## Summary

Keep open: current `main` still follows the package-entry path when `package.json#openclaw.extensions` points at a TypeScript entry under the global extensions root, and existing tests/docs intentionally require compiled JS there. The report is actionable, but changing precedence between package metadata and manifest/index fallback is a plugin-loader contract decision rather than a conservative auto-close or automation-safe bug fix.

## What This Changes

The issue requests that local plugin discovery respect `openclaw.plugin.json` when a plugin directory also has `package.json`, so TS-source plugins are not dropped as package installs needing compiled JS.

## Best Possible Solution

Maintainers should define the package-shaped local plugin contract, then either allow a manifest/index TypeScript fallback for explicit local-development paths or keep the compiled-runtime rule and surface a louder diagnostic plus docs guidance.

## Reproduction Assessment

Yes, from source for a global plugin directory whose `package.json#openclaw.extensions` points at `index.ts` without a built JS peer: discovery takes the package-entry path and skips the candidate. I did not run a live repro because this review is read-only and the existing focused test already encodes the current behavior.

## Solution Assessment

Unclear: preferring `openclaw.plugin.json` may be the right operator experience, but it changes an intentional package-install contract. A safer maintainer-owned path is to decide the contract first, then add focused discovery tests and update docs/diagnostics with the chosen behavior.
