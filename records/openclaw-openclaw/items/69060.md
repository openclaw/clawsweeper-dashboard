## Summary

Keep open. The PR is a plausible narrow improvement for the QMD warm-session part of the linked latency issue, but the patch still exposes the new header through the shared Responses resolver without a documented/tested Responses contract, and the docs heading still violates the scoped Mintlify rule.

## What This Changes

The PR adds an `x-openclaw-session` Gateway HTTP header, Chat Completions precedence tests, OpenAI HTTP docs, and a changelog entry for stable agent-scoped sessions.

## Best Possible Solution

Make the header contract explicit first, then either scope the hint to Chat Completions or update Responses continuation scoping, docs, and tests consistently; also rename the docs heading to a Mintlify-safe form.

## Reproduction Assessment

Yes, at source level. Current main falls back to a random UUID when callers omit `user`, and the PR body includes after-fix curl/log output showing stable session-key reuse and a warm-session QMD skip.

## Solution Assessment

No, not yet. The stable hint is plausible, but adding it in the shared resolver changes Responses without a matching public contract, continuation-scope update, or Responses tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope the session hint before sharing the resolver:** `src/gateway/http-utils.ts:154-156`
  - body: `resolveSessionKey` is used by both Chat Completions and Responses, so adding `x-openclaw-session` here silently creates a `/v1/responses` header contract too. Either keep the hint Chat Completions-only or document/test Responses and include the hint in the continuation-scope contract.
  - confidence: 0.9

  - body: The new heading contains an em dash, but `docs/AGENTS.md` says docs headings should avoid em dashes because Mintlify anchors are brittle. Rename the heading so docs checks and anchors stay stable.
  - confidence: 0.87
