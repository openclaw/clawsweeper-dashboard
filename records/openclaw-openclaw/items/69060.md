## Summary

Keep this PR open: the stable session hint is useful and still missing from current main, but the branch adds a public session-routing header through a shared helper without settling the `/v1/responses` contract, still has release-owned changelog churn, and is currently dirty against the base.

## What This Changes

Adds a validated `x-openclaw-session` Gateway HTTP session header, documents it for Chat Completions, adds precedence tests, and edits `CHANGELOG.md`.

## Best Possible Solution

Make one explicit Gateway HTTP session-routing contract for Chat Completions and Responses, then either scope `x-openclaw-session` to Chat Completions or document and test the same precedence for both endpoints before merge.

## Reproduction Assessment

Yes, at source level: current main falls back to a random UUID session key when neither `x-openclaw-session-key` nor `user` is present, so repeated stateless Chat Completions calls cannot reuse a warm session. I did not run a live gateway benchmark during this read-only review.

## Solution Assessment

Not yet. The new header is a reasonable shape for the Chat Completions warm-session problem, but the implementation should first settle whether the same public header applies to `/v1/responses`, align docs/tests with that choice, and remove the release-owned changelog edit.
