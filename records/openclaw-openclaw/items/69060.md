## Summary

Keep open: the stable session hint is useful and the PR body has sufficient runtime proof, but the latest diff still adds `x-openclaw-session` in the shared Gateway resolver, which makes `/v1/responses` accept it without a documented Responses contract or continuation-scope handling.

## What This Changes

The branch adds an `x-openclaw-session` Gateway HTTP header in shared session resolution, documents it for OpenAI Chat Completions, adds precedence tests, and updates the changelog.

## Best Possible Solution

Set one public Gateway session-header contract: either keep `x-openclaw-session` Chat Completions-only, or make it Responses-wide with docs, continuation-scope matching, and regression tests in the same change.

## Reproduction Assessment

Yes, at source level. Current main falls back to a random UUID when no explicit session key or `user` is present, and the PR body includes after-fix curl/log output showing stable Chat Completions reuse with the new header.

## Solution Assessment

No, not yet. The stable hint is a plausible narrow feature, but placing it in the shared resolver creates an undocumented Responses behavior unless endpoint scope is narrowed or Responses is fully updated.
