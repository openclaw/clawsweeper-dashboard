## What This Changes

Adds a validated `x-openclaw-session` header in the shared Gateway HTTP request-context path, documents it for Chat Completions, adds precedence tests, and edits `CHANGELOG.md`.

## Best Possible Solution

Define one explicit Gateway HTTP session-routing contract for Chat Completions and Responses, then either scope `x-openclaw-session` to Chat Completions or document and test it consistently for both endpoints before merge.

## Reproduction Assessment

Yes, at source level: current main falls back to a random UUID session key when neither `x-openclaw-session-key` nor `user` is present, so repeated stateless Chat Completions calls cannot reuse a warm session. I did not run a live gateway benchmark during this read-only review.

## Solution Assessment

Not yet. The new header is a plausible shape for the Chat Completions warm-session problem, but the implementation should first settle endpoint scope, session-key precedence, and release-owned changelog cleanup.
