## Summary

The PR is plausible but not merge-ready: it still wires a new public session header through shared OpenAI-compatible request context, so Responses behavior changes without matching docs, tests, or continuation-scope semantics.

## What This Changes

The PR adds a stable `x-openclaw-session` Gateway HTTP header, related docs/changelog text, and Chat Completions tests for agent-scoped session reuse.

## Best Possible Solution

Make the new header's endpoint contract explicit: either plumb it only through Chat Completions, or document and test it for Responses and include it in Responses continuation scope where needed.

## Reproduction Assessment

Yes, at source level. Current main falls back to a random UUID when OpenAI-compatible callers omit `user`, and the PR body includes after-fix terminal curl/log output showing stable session-key reuse and a warm-session result.

## Solution Assessment

No, not as currently patched. The stable hint is a plausible narrow fix for the QMD re-index component, but the shared resolver makes it a broader Responses contract without matching docs/tests/scope handling.
