## Summary

Keep open. The PR addresses a real source-level stable-session gap and includes sufficient terminal proof, but it still needs a maintainer API-scope decision because the implementation changes the shared request-context resolver used by Responses while the docs/tests frame the header as Chat Completions-only.

## What This Changes

The PR adds a public `x-openclaw-session` Gateway HTTP header, docs, changelog text, and Chat Completions tests for stable agent-scoped sessions when callers cannot set the OpenAI `user` field.

## Best Possible Solution

Land the stable Chat Completions session hint only after explicitly scoping it to Chat Completions or documenting/testing it as a Responses contract too; keep https://github.com/openclaw/openclaw/issues/68920 open for the larger light-context work.

## Reproduction Assessment

Yes, at source level. Current main falls back to a random UUID when Chat Completions callers omit `user`, and the PR body includes after-fix curl/log proof that the new header reuses a warm session; I did not run a live Gateway in this read-only review.

## Solution Assessment

No, not as currently patched. The stable session hint is a reasonable narrow direction, but the implementation needs an explicit endpoint-scope decision plus docs formatting before it is the maintainable solution.
