## Label Justifications

- `P2`: This is a normal-priority Gateway HTTP API improvement with limited blast radius but real session-routing implications.
- `merge-risk: 🚨 compatibility`: The diff adds a new public request header in a shared resolver, which can change behavior for existing OpenAI-compatible Responses clients that send the same header.
- `merge-risk: 🚨 session-state`: Responses continuation matching can reuse an older session even when the new session hint changes, risking mis-associated agent session state.

## Summary

Keep open: the PR still needs an endpoint-scope/session-precedence decision because the new header is added in the shared resolver and would silently affect Responses without docs or continuation-scope handling.

## What This Changes

The PR adds a validated `x-openclaw-session` header in the shared Gateway request-context resolver, documents it for Chat Completions, adds precedence tests, and adds a changelog entry.

## Best Possible Solution

Make the header contract explicit first, then either keep `x-openclaw-session` Chat Completions-only or update Responses continuation scope, docs, and tests so the header participates consistently.

## Reproduction Assessment

Yes, at source level. Current main falls back to `randomUUID()` when `x-openclaw-session-key` and `user` are absent, and the PR body includes after-fix curl/log output showing stable session-key reuse with the new header.

## Solution Assessment

No, not yet. The stable hint is a plausible narrow mitigation, but placing it in the shared resolver creates an undocumented Responses contract unless the PR also updates Responses session scope, docs, and tests.
