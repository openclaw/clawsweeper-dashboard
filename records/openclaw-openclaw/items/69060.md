## Summary

Keep open: the stable session hint is a plausible narrow fix for the QMD warm-session part, and the PR body now includes usable curl/log proof, but the patch still needs maintainer attention because the shared resolver also changes `/v1/responses` without matching docs/tests and the branch is currently merge-conflicting.

## What This Changes

Adds a validated `x-openclaw-session` header in the shared Gateway HTTP request-context resolver, documents it for Chat Completions, adds Chat Completions precedence tests, and updates the changelog.

## Best Possible Solution

Make one explicit Gateway HTTP session contract for Chat Completions and Responses, then either scope `x-openclaw-session` to Chat Completions or document and test the same precedence for both endpoints before merge.

## Reproduction Assessment

Yes, at source level: current main returns a random UUID session key when neither `x-openclaw-session-key` nor `user` is present, so repeated stateless Chat Completions calls cannot reuse a warm session. I did not run a live gateway benchmark during this read-only review.

## Solution Assessment

Not yet. The new header is a reasonable shape for the Chat Completions warm-session problem, but the implementation should first settle whether the same public header applies to `/v1/responses` and align docs/tests with that choice.
