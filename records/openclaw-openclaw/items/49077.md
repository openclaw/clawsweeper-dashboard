## Summary

Keep open. Current main still does not expose or enforce a configurable delay between model API request starts, and the linked open PR remains the active implementation candidate rather than a merged fix.

## What This Changes

Add a configurable delay between consecutive model API requests so users can avoid strict provider rate limits, especially for GLM-like providers.

## Best Possible Solution

Keep the issue open and land #49221 or an equivalent replacement that adds validated global and per-provider model request-delay config, applies it consistently to normal embedded runs and compaction, documents the public config surface, and covers provider-scoped spacing, zero disablement, override precedence, and abort behavior with focused tests.

## Reproduction Assessment

Not applicable as a conventional runtime bug reproduction because this is a feature request. Source-level verification is high confidence: current config types, strict schema, docs, guarded fetch, embedded-run stream setup, and compaction setup all lack request-start pacing.

## Solution Assessment

Unclear until the open PR is fully reviewed. A generic core model-request pacing wrapper with global and provider-specific config appears like the maintainable direction, but it should prove shared per-provider state, zero overrides, current-main integration, and abortable sleeps before merge.
