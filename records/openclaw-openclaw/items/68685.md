## Summary

Keep open: the branch adds useful doctor migration coverage, but it does not implement the claimed gateway/read-path self-healing because current main validates config before applying doctor migrations, and the PR still lacks real after-fix runtime proof.

## What This Changes

The PR adds a doctor legacy config migration and tests to strip obsolete memorySearch keys from agents.defaults.memorySearch and agents.list[].memorySearch.

## Best Possible Solution

Decide the intended config policy, then either land this as doctor-only cleanup without closing the startup bug or implement a maintainer-approved read/startup migration with focused coverage and real runtime proof.

## Reproduction Assessment

Yes. Source inspection of current main shows stale memorySearch keys reach a strict schema during load/read validation before doctor migrations run, and current tests preserve the no-read-time-auto-migration boundary.

## Solution Assessment

No as currently framed. Targeted stale-key stripping is reasonable, but placing it only in the doctor migration registry does not solve the claimed gateway startup path.
