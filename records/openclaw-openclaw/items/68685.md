## What This Changes

Adds a legacy doctor config migration and tests to strip obsolete `chunkSize`, `chunkOverlap`, and `maxResults` keys from `agents.defaults.memorySearch` and `agents.list[].memorySearch`.

## Best Possible Solution

Either land this as doctor-only cleanup without closing the startup bug, or implement a maintainer-approved startup/read stale-key cleanup with focused coverage and real runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main routes stale `memorySearch` keys into a strict schema during load/read validation before doctor migrations run, and the existing test suite preserves that no-read-time-auto-migration boundary.

## Solution Assessment

No. Targeted stale-key stripping is reasonable, but putting it only in the doctor migration registry does not solve the claimed gateway startup path; the safer path is an explicit maintainer decision on doctor-only cleanup versus startup/read self-healing.
