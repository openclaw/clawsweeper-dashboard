The PR adds a runtime legacy config migration and tests to strip retired `chunkSize`, `chunkOverlap`, and top-level `maxResults` keys from `agents.defaults.memorySearch` and `agents.list[].memorySearch` before strict config validation.

## Best Possible Solution

Land a narrow runtime/doctor migration for these known retired memory-search keys while preserving strict config validation and avoiding a generic unknown-key stripping policy. The final PR should include a schema-accurate regression test and a one-line changelog entry for the startup recovery fix.

## Reproduction Assessment

Yes. A config containing `agents.defaults.memorySearch.chunkSize` or `agents.list[0].memorySearch.maxResults` reaches the current read-time migration path and then the strict `MemorySearchSchema`, where current main has no strip step for those keys.

## Solution Assessment

Yes, with small cleanup. The PR’s scoped strip-after-move migration is the narrowest maintainable fix for this legacy shape; changing global schema strictness or validation ordering would be broader and riskier.
