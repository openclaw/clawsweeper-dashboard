The PR changes `openclaw secrets configure` so the one-way migration confirmation is based on the final `shouldApply` state after either `--apply` or interactive approval, instead of only on the `--apply` flag.

## Best Possible Solution

Land the focused prompt-gating fix with real CLI prompt proof, plus a small regression assertion and docs alignment for the new prompt sequence if maintainers want the behavior fully documented.

## Reproduction Assessment

Yes. Source inspection on current main shows `shouldApply` can be set by the interactive apply prompt while `needsIrreversiblePrompt` still depends on `opts.apply`, so running `openclaw secrets configure` without `--apply` and answering yes would skip the one-way warning.

## Solution Assessment

Yes for the core bug: deriving the warning from the final `shouldApply` state is the narrow maintainable fix. The merge package still needs real CLI proof, and a focused regression assertion would make the behavior harder to regress.
