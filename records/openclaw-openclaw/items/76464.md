## Summary

Keep this issue open. Current `main` still builds the local `model.run` simple-completion context with only a user message, while the OpenAI Responses request layer only sends Codex `instructions` when `context.systemPrompt` is present; the related #73031/#74019 work covers a separate tool-allowlist failure, not this missing-instructions/no-error-surfacing path.

## What This Changes

The issue reports that local `openclaw infer model run --model openai-codex/gpt-5.5` omits instructions for the Codex Responses transport and then hides the provider error behind a generic no-text message.

## Best Possible Solution

Add a narrow local `model.run` instructions default and provider-error surfacing in `src/cli/capability-cli.ts`, with regression coverage for Codex Responses and empty error results.

## Reproduction Assessment

Yes from source inspection and the reporter's live command: current `main` still omits `systemPrompt` on the local simple-completion path for an `openai-codex-responses` model. I did not run the live Codex OAuth call myself.

## Solution Assessment

Yes. The narrow maintainable fix is to keep the lean local completion path, add minimal instructions to that context, and include `stopReason`/`errorMessage` detail when no text is returned.
