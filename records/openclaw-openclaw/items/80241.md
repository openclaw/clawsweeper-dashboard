## Summary

Keep this PR open, but it is not merge-ready: the memory-core fix direction is supported by the linked bug report and by a later reviewer comment, but the proposed classifier can misroute quota/provider failures into skip-file cleanup and the PR still lacks real low-limit endpoint proof.

## What This Changes

The PR adds adaptive inline memory embedding batch splitting, per-file skip behavior for irreducible embedding size failures, docs/changelog updates, an SDK export, and regression tests.

## Best Possible Solution

Land a memory-core fix that adaptively splits concrete per-batch/input-size failures, preserves quota/rate/provider failures as command failures, and includes redacted real endpoint proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can batch unknown OpenAI-compatible embedding chunks under its 8000 cap while the reported endpoint rejects batches around 2000. I did not run the live Qianfan endpoint in this read-only review.

## Solution Assessment

No, not as currently written: adaptive splitting belongs in memory-core, but the matcher must exclude quota/rate/provider failures before skip-file cleanup is safe. The PR also needs real after-fix endpoint proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Narrow the batch-size error matcher:** `extensions/memory-core/src/memory/manager-embedding-policy.ts:96-98`

Please exclude quota/rate messages or match only concrete per-input/per-batch size errors.
