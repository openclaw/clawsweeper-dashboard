## Summary

Keep open: the fix direction is right, and the linked bug is source-reproducible, but this PR still misclassifies provider quota errors as file-skippable batch-size failures, adds a public SDK export without updating the API baseline, and lacks required real low-limit endpoint proof.

## What This Changes

This PR changes memory-core indexing to adaptively split oversized embedding batches, skip irreducible per-file embedding failures, export the embedding max-token resolver, and update tests, docs, and changelog.

## Best Possible Solution

Land a memory-core fix that adaptively splits only concrete per-input or per-batch size failures, preserves quota/rate/provider failures as command failures, updates SDK API drift artifacts if a public export remains, and includes redacted real endpoint proof.

## Reproduction Assessment

Yes, at source level: current main can batch unknown OpenAI-compatible embedding chunks up to the generic 8192/8000 budget while the linked low-limit endpoint reports a 2000-size batch rejection. I did not run the live Qianfan endpoint in this read-only review.

## Solution Assessment

No, not as currently written: the architecture matches the right model, but the matcher must not convert quota/provider failures into skip-file behavior, and the new public SDK export needs baseline follow-through plus real after-fix proof.
