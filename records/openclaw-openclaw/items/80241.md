## Summary

Keep this PR open: it targets a valid memory indexing bug and follows the right adaptive-splitting direction, but the proposed error classifier is too broad and the external PR still lacks real after-fix proof from a low-limit embedding endpoint.

## What This Changes

The branch changes memory-core inline embedding to adaptively split oversized batches, skip irreducible file-level input-size failures, and updates OpenAI embedding tests, SDK exports, docs, and changelog.

## Best Possible Solution

Land a memory-core fix that adaptively splits concrete per-input or per-batch size failures, preserves quota/rate/provider failures as command failures, and includes redacted real low-limit endpoint proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main can batch unknown OpenAI-compatible embedding chunks up to the generic 8k budget while the reported endpoint rejects combined batches around 2k; I did not run the live Qianfan endpoint.

## Solution Assessment

No, not as currently written. Adaptive splitting belongs in memory-core, but the matcher must only catch concrete input or batch-size failures and must not route quota/rate/provider failures into skip-file handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Narrow the batch-size error matcher:** `extensions/memory-core/src/memory/manager-embedding-policy.ts:96-98`
  - body: The new matcher treats broad messages such as `rate limit exceeded` or quota text containing `exceeded` as input-size failures. After retries are exhausted, `resolveIndexFileFailureAction` can return `skip-file`, clear indexed data, and continue instead of failing the command for a provider/quota outage. Please exclude quota/rate messages or match only concrete per-input/per-batch size errors.
  - confidence: 0.88
