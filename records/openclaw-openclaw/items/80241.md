## Summary

Keep this PR open, but not merge-ready: the adaptive split direction matches the linked memory regression and an independent commenter agreed with that model, but the current head still misclassifies quota-style provider failures as skippable file-size failures and lacks real low-limit endpoint proof.

## What This Changes

The PR changes memory indexing to adaptively split inline embedding batches on provider size errors, skip irreducible per-file embedding failures, export the max-token resolver, and update tests, docs, and changelog.

## Best Possible Solution

Land a focused memory-core fix that adaptively splits concrete per-input or per-batch size failures, preserves quota/rate/provider failures as command failures, and includes real low-limit endpoint proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can batch unknown OpenAI-compatible embedding chunks up to the generic 8k budget while the reported endpoint rejects a lower combined batch size. I did not run the live Qianfan endpoint in this read-only review.

## Solution Assessment

No, not as currently written: adaptive splitting belongs in memory-core, but the matcher must not route quota/provider failures into skip-file handling. The PR also needs real after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Narrow the batch-size error matcher:** `extensions/memory-core/src/memory/manager-embedding-policy.ts:96`
  - body: The new matcher treats `Too many tokens per day` as an input-size failure. Current retry policy already treats `tokens per day` as a provider quota/rate-limit condition, so after retries are exhausted this path can split, then skip the file and clear indexed data instead of failing the command for a provider-level outage. Please exclude quota/rate messages or match only concrete per-input/per-batch size errors.
  - confidence: 0.88
