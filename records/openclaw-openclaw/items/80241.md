## Summary

Keep open: the memory indexing bug remains source-reproducible on current main, and this PR is a plausible direction, but the patch still overclassifies quota errors as skip-file and lacks required real low-limit endpoint proof.

## What This Changes

The PR changes memory indexing to split low-limit embedding batches, skip irreducible per-file embedding failures, export the embedding limit resolver, and update docs, tests, and changelog.

## Best Possible Solution

Land a memory-core fix that adaptively splits concrete per-batch or per-input size failures, preserves quota/provider failures as command-fatal, updates SDK baseline if the export remains, and includes real endpoint proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure an OpenAI-compatible embedding endpoint that rejects combined batches above roughly 2000, then index enough chunks for current main to build a batch below 8000 but above that provider cap. I did not run the live Qianfan endpoint.

## Solution Assessment

No, not as currently written: adaptive splitting and per-file isolation are the right boundary, but the matcher must not classify quota/provider failures as skip-file and the public SDK baseline must stay aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow the batch-size error matcher:** `extensions/memory-core/src/memory/manager-embedding-policy.ts:96`
  - body: The new matcher treats any `too many tokens` message as an input-size failure. Because the sync wrapper now clears indexed file data and continues for `skip-file`, a provider quota error such as `tokens per day` can be downgraded from command-fatal to skipped-file behavior after retries. Please exclude quota/rate failures or match only concrete per-input/per-batch size errors.
  - confidence: 0.88
- **[P2] Regenerate the Plugin SDK API baseline:** `src/plugin-sdk/memory-core-host-engine-embeddings.ts:28`
  - body: This adds `resolveEmbeddingMaxInputTokens` to a public plugin SDK subpath, but the PR does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. The API baseline check is meant to catch exactly this public-surface drift, so the intentional export needs the generated hash update or the export should stay private.
  - confidence: 0.81
