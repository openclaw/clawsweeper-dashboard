## Summary

Keep open: the PR targets a valid source-reproducible memory indexing bug, but the current diff can treat provider quota errors as skip-file input-size failures and still lacks after-fix real behavior proof.

## What This Changes

This PR adaptively splits memory-core inline embedding batches on provider size errors, skips irreducible input-size file failures, exports the embedding max-token resolver, and updates tests, docs, and changelog.

## Best Possible Solution

Land a memory-core fix that adaptively splits concrete per-input or per-batch size failures, preserves quota/rate/provider failures as command failures, and includes real low-limit endpoint proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can batch unknown OpenAI-compatible embedding chunks up to the generic 8k budget while the reported endpoint rejects combined batches around 2k. I did not run the live Qianfan endpoint in this read-only review.

## Solution Assessment

No, not as currently written: adaptive splitting belongs in memory-core, but the matcher must not route quota/provider failures into skip-file handling. The PR also needs real after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Narrow the batch-size error matcher:** `extensions/memory-core/src/memory/manager-embedding-policy.ts:96`

Please exclude quota/rate messages or match only concrete per-input/per-batch size errors.
