## Summary

Keep open: this PR targets a real source-backed memory batching bug and a later reviewer comment supports the general direction, but the current head still has a blocking classifier issue and no after-fix proof from a real low-limit embedding endpoint.

## What This Changes

The branch adds adaptive inline memory embedding batch splitting, per-file skip handling for irreducible embedding size errors, SDK exports, docs, changelog, and regression tests.

## Best Possible Solution

Land a memory-core implementation that adaptively splits concrete provider batch-size failures, preserves quota/rate/provider failures as command failures, and includes real low-limit endpoint proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can batch unknown OpenAI-compatible embedding chunks up to the generic 8k budget while the linked issue reports a provider rejecting combined batches around 2k. I did not run the live Qianfan endpoint in this read-only review.

## Solution Assessment

No, not as currently written: adaptive splitting is the right memory-core direction, but the matcher must not route quota or provider failures into skip-file handling. The PR also needs real after-fix endpoint proof before merge.
