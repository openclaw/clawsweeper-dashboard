## Summary

Keep this PR open. The reported WhatsApp reconnect failure is still visible in current main, but the branch edits obsolete `src/web` paths while the shipped WhatsApp runtime now lives under `extensions/whatsapp/src`, so the useful fix needs to be retargeted before it can merge.

## What This Changes

The PR changes the old WhatsApp Web reconnect implementation under `src/web` to default `maxAttempts` to unlimited, replace max-attempt shutdown with degraded slow retries, and update the old web reconnect tests.

## Best Possible Solution

Retarget the useful behavior into the WhatsApp plugin: make the default retry policy long-lived, route capped retry exhaustion through an explicit degraded retry mode, keep terminal auth/conflict exits intact, and update plugin tests plus config help around the chosen `maxAttempts: 0` semantics.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a positive WhatsApp `maxAttempts`, trigger retryable closes until the cap, and the controller returns `stop` while the monitor shuts down and breaks; the current e2e test asserts that behavior.

## Solution Assessment

No, not as written. The behavioral direction is useful, but the PR edits obsolete `src/web` files; the maintainable fix is to move the policy, controller, monitor, test, and config-help changes into the current WhatsApp plugin surface.
