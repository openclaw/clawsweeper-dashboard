## Summary

Keep open: current main and the latest release still preserve sender identity only as current-turn, display, or pending-history metadata; durable persisted/replayed user messages and provider-bound model messages still have no per-message sender attribution contract.

## What This Changes

The issue asks OpenClaw to carry group-chat sender attribution on durable model-message history entries so attribution survives continuation-style rolling context.

## Best Possible Solution

Define one additive provider-agnostic sender attribution contract for user turns, persist it safely, and serialize it consistently through replay, compaction, and provider adapters without inline body pollution.

## Reproduction Assessment

Yes, at source level: current code carries sender metadata in active-turn context, but persisted/replayed user messages lack sender fields and historical inbound metadata blocks are stripped before provider serialization.

## Solution Assessment

Partly: per-message sender metadata is the right class of solution, but the permanent field shape, trust boundary, and provider mapping need maintainer approval; inline body prefixes and channel-only workarounds are not the best fix.
