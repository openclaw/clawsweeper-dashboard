## Summary

Keep open: current main has structured sender attribution for the current prompt, but it still stores the transcript prompt as clean body text and strips hidden runtime-context messages before LLM conversion, so durable per-message group sender attribution is not implemented.

## What This Changes

The issue asks OpenClaw to persist group sender attribution on each LLM user-message entry so it survives continuation-skip rolling context.

## Best Possible Solution

Define a generic persisted sender-attribution field for user transcript messages, map it safely at the LLM boundary per provider, and cover continuation-skip group replay with regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: group SenderName is available for the current prompt, but the persisted transcript prompt remains clean body text and hidden runtime-context custom messages are stripped before provider replay. I did not run a live group-chat reproduction.

## Solution Assessment

Unclear: message-array-level attribution is the right layer, but maintainers should choose the durable transcript schema and provider mapping before implementation. A BodyForAgent prefix would be a narrower workaround but not the best long-term design.
