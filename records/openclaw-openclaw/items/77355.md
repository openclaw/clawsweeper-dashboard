## Label Justifications

- `P2`: This is a normal-priority group-chat attribution/session-history correctness issue with limited blast radius and a design-dependent fix.
- `impact:session-state`: The issue is about sender identity being lost from durable conversation state when historical user turns are replayed to the model.

## Summary

Keep open: current main and v2026.6.6 still carry group sender identity as current-turn inbound context, not as durable per-message metadata, so historical user turns can replay without sender attribution after metadata stripping.

## What This Changes

The issue requests durable sender attribution on LLM user-message entries for group chats so speaker identity survives rolling context reuse.

## Best Possible Solution

Define one provider-agnostic durable sender-attribution contract for user turns, persist it with transcript messages, and serialize it consistently through replay, compaction, and provider adapters without polluting BodyForAgent.

## Reproduction Assessment

Yes, at source level. Current main stores user turns without sender metadata and strips historical inbound Sender/Conversation blocks before provider replay; no live multi-user channel repro was run.

## Solution Assessment

Unclear without maintainer design input. Per-message sender metadata is cleaner than inline BodyForAgent prefixes, but the fix needs a durable transcript/provider contract rather than a channel-only patch.
