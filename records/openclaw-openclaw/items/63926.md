## Summary

Keep open: current main and v2026.6.10 still expose browser CLI/RPC primitives and narrow plugin UI descriptors, but no visible Control UI browser-login handoff panel. The discussion also treats this as the canonical same-browser-state login handoff tracker, with the FSB reference as useful design context rather than a replacement.

## What This Changes

Request a visible Control UI browser section for managed-browser status, tab access, login handoff, tab actions, and same-session agent resume.

## Best Possible Solution

Design and ship an operator-scoped browser handoff surface with typed profile/tab context, read-only status by default, explicit write controls, redaction, and same-state resume semantics coordinated with the broader preview and plugin UI work.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 have browser-control primitives but no Control UI browser route, tab, or handoff renderer. This is a missing feature rather than a runtime bug needing live reproduction.

## Solution Assessment

Unclear until maintainers scope the UX and security boundary. The requested direction is useful, but the best solution should be a secured browser handoff surface rather than an ad hoc service-specific panel or unrestricted co-browsing view.
