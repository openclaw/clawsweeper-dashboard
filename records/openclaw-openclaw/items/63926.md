## Summary

Keep open. Current main still has browser CLI/tool/Gateway control and Google Meet manual-action metadata, but it does not provide the requested visible Control UI browser handoff panel, and a maintainer explicitly marked this as the canonical browser-login handoff surface to keep separate from the Google Meet plugin.

## What This Changes

Add a visible Control UI browser panel for managed-browser status, active tabs, tab actions, login handoff, and resuming agent work after manual login or permission steps.

## Best Possible Solution

Add a generic first-class Browser area in the Control UI backed by the existing browser control surface or an additive typed Gateway method. It should show profile/status/readiness, active tabs with open/focus/close actions, exact node/profile/tab context for manual blockers, browser doctor/inspect diagnostics, and a clear resume path after login or permission prompts, while keeping service-specific blocker detection inside plugins such as Google Meet.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. Source inspection gives high-confidence evidence of the gap: current main exposes browser CLI/tool/Gateway control and Google Meet manual-action data, but has no Control UI browser route or rendered panel.

## Solution Assessment

Not applicable because there is no proposed patch to evaluate. The requested direction is the right product-level fix, but the maintainable implementation should be generic Control UI browser handoff infrastructure rather than a Google Meet-only panel or another CLI/documentation workaround.
