## Summary

Keep open. Current main still lacks the requested Control UI browser login handoff panel, the issue discussion explicitly treats this as the canonical handoff surface, and the remaining work is a new UX/security/product surface rather than conservative cleanup.

## What This Changes

Add a visible Control UI browser section for managed-browser login handoff, tab/status management, and resume flow after manual user action.

## Best Possible Solution

Ship a generic admin-scoped Browser area in Control UI, backed by browser.request or additive typed gateway methods, that shows profile/readiness, active tabs, manual-action node/profile/tab context, diagnostics, safe tab actions, and a clear resume path after login or permission prompts.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection gives a high-confidence absence check: browser primitives and manual-action metadata exist, but no Control UI browser route, view, or rendered handoff panel exists on current main.

## Solution Assessment

Not applicable because there is no proposed patch to evaluate. The requested direction is useful, but the maintainable solution should be a generic Control UI browser handoff surface rather than a Google Meet-only panel or another CLI-only workaround.
