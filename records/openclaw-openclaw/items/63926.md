## Summary

Keep open: current main and the latest release still lack the requested Control UI browser-login handoff panel, and the discussion explicitly treats this as canonical feature work needing UX and security scoping.

## What This Changes

The issue requests a visible Control UI browser panel for managed-browser status, tabs, open/focus/close actions, manual login handoff, and resume against the same browser state.

## Best Possible Solution

Ship a generic admin-scoped Browser area in Control UI that reuses `browser.request` or additive typed gateway methods to show profile/readiness, active tabs, manual-action node/profile/tab context, diagnostics, safe tab actions, and a clear resume path after login or permission prompts.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection on current main confirms there is no Control UI browser route, view, or rendered handoff panel while backend browser and Google Meet manual-action primitives exist.

## Solution Assessment

Unclear until maintainers scope the UX and security boundaries. The requested direction is useful, but the maintainable solution should be a generic secured Control UI browser handoff surface rather than a Google Meet-only panel or CLI-only workaround.
