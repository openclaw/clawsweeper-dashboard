Keep open: current main and v2026.6.10 still have browser CLI/tool/gateway primitives, but no visible Control UI browser-login handoff panel; the issue discussion also explicitly treats this as the canonical browser-login handoff tracker.

## What This Changes

Request a visible Control UI browser section for managed-browser status, active tabs, user login handoff, tab actions, and same-state agent resume.

## Best Possible Solution

Design and ship an operator-scoped browser handoff surface with typed profile/tab context, read-only status by default, explicit write controls, redaction, and same-state resume semantics coordinated with https://github.com/openclaw/openclaw/issues/27574 and https://github.com/openclaw/openclaw/pull/80388.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request for a missing Control UI surface. Source and v2026.6.10 inspection confirm browser-control primitives exist but no browser-login handoff panel is present.

## Solution Assessment

Unclear until maintainers scope the UX and security boundary. The requested direction is useful, but the best solution should be a secured browser handoff surface rather than an ad hoc service-specific panel or unrestricted co-browsing view.
