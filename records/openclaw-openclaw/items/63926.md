## Summary

Keep open: current main still lacks the requested Control UI browser-login handoff panel, and the live discussion treats this as canonical feature work that needs UX and security scoping.

## What This Changes

The issue asks for a visible Control UI browser panel that lets users inspect and manage the managed browser session during manual login handoff workflows.

## Best Possible Solution

Ship a generic admin-scoped Browser area in Control UI that reuses `browser.request` or additive typed gateway methods to show profile/readiness, active tabs, manual-action node/profile/tab context, diagnostics, safe tab actions, and a clear resume path after login or permission prompts.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Current-main source inspection confirms no Control UI browser route, view, or rendered handoff panel while backend browser and Google Meet manual-action primitives exist.

## Solution Assessment

Unclear until maintainers scope the UX and security boundaries. The requested direction is useful, but the maintainable solution should be a generic secured Control UI browser handoff surface rather than a Google Meet-only panel or CLI-only workaround.
