## Summary

Keep open: current main and the latest release still have browser-control primitives but no Control UI browser-login handoff panel, and the discussion explicitly treats this as canonical product/security-scoped feature work.

## What This Changes

Add a visible Control UI browser section for managed-browser status, tab management, manual login handoff, and agent resume against the same browser state.

## Best Possible Solution

Ship a generic admin-scoped Browser area in Control UI that reuses `browser.request` or additive typed gateway methods to show readiness, profiles, tabs, manual-action context, safe tab actions, and a clear resume path after login or permission prompts.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection on current main confirms there is no Control UI browser route, rendered browser panel, or documented handoff surface.

## Solution Assessment

Unclear until maintainers scope the UX and security boundaries. The requested direction is useful, but the maintainable path should be a secured generic browser handoff surface rather than a Google Meet-only workaround or a repeat of the rejected Terminal/Browser co-working PR shape.
