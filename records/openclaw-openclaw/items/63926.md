## Summary

Keep open: current main and v2026.5.18 still lack the requested Control UI browser-login handoff panel, and the discussion explicitly keeps this as canonical feature work needing UX and security scoping.

## What This Changes

Add a visible Control UI browser panel for managed-browser status, tabs, user login handoff, and resuming agent work against the same browser state.

## Best Possible Solution

Ship a generic admin-scoped Browser area in Control UI that reuses `browser.request` or additive typed gateway methods for profile/readiness, active tabs, diagnostics, safe tab actions, manual-action context, and a clear resume path after login or permission prompts.

Not applicable as a runtime bug reproduction because this is a feature request. Source and release inspection give high confidence that OpenClaw has browser-control primitives but no Control UI browser route, view, or login-handoff panel.

## Solution Assessment

Unclear until maintainers scope the UX and security boundaries. The requested direction is useful, but the maintainable solution should be a generic secured Control UI browser handoff surface rather than a Google Meet-only panel or CLI-only workaround.
