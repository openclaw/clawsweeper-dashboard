## Summary

Keep open. Current main still exposes browser CLI/tool/Gateway control and Google Meet manual-action metadata, but it does not provide the requested visible Control UI browser login handoff panel, and a maintainer explicitly marked this as the canonical surface to keep open.

## What This Changes

Add a visible Control UI browser handoff panel with managed-browser status, active tabs, tab actions, login completion, and agent-resume support.

## Best Possible Solution

Add a generic first-class Browser area in Control UI backed by browser.request or an additive typed Gateway method, showing profile/readiness, active tabs, safe tab actions, exact manual-blocker node/profile/tab context, diagnostics, and a clear resume path after login or permission prompts.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. Source inspection gives high-confidence verification of the gap: current main has browser control APIs and manual-action metadata, but no Control UI route or rendered panel for browser handoff.

## Solution Assessment

Not applicable because there is no proposed patch to evaluate. The requested direction is right, but the maintainable solution should be generic Control UI browser handoff infrastructure rather than a Google Meet-only panel or another CLI-only workaround.
