## Summary

Keep open. Current main already runs FTS/BM25 separately from vector search, so the stated sequential-KNN root cause does not match current source, but there is still a plausible memory-search bug because keyword results are model-filtered and can be absent from the hybrid merge, leaving vector-only session hits with `textScore: 0`.

## What This Changes

The issue asks for session-memory `memory_search` to surface exact FTS/BM25 transcript matches even when vector KNN misses dialogue-format chunks.

## Best Possible Solution

Land a focused memory-core fix that makes lexical FTS candidate generation independent of KNN and inappropriate model gating, with regression coverage using session transcript chunks and source filters preserved.

## Reproduction Assessment

No high-confidence end-to-end replay was done. The issue provides detailed external evidence, and current source gives a medium-confidence failure path where FTS rows can be excluded from hybrid results and vector-only hits keep `textScore: 0`.

## Solution Assessment

No. Making FTS parallel is not the right fix by itself because current main already does that; the safer direction is to remove or relax the keyword-only model filter and add focused session-memory regression coverage, possibly by rebasing or replacing https://github.com/openclaw/openclaw/pull/48328.
