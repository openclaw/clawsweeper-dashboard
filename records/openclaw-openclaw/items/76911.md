## Summary

Keep open: the PR targets a valid Discord SecretRef crash path, but the proposed shared normalizer change erases configured-unavailable state and can let an explicit `channels.discord.token` SecretRef fall through to the default env token.

## What This Changes

The PR changes Discord token normalization and its unit test so unresolved SecretRef token objects return no token instead of throwing.

## Best Possible Solution

Preserve a distinct configured-unavailable token state for read-only runtime/status snapshots while keeping Discord execution and startup token use fail-closed unless the configured ref resolves.

## Reproduction Assessment

Yes from source inspection: current `resolveDiscordToken` passes a SecretRef to strict `normalizeResolvedSecretInputString`, and current tests assert the unresolved SecretRef throw. I did not execute tests because this review is read-only.

## Solution Assessment

No; returning `undefined` from a shared token normalizer fixes one throw path but loses the difference between missing and configured-unavailable credentials. A safer fix should use inspect-mode or tagged token resolution for status/health while preserving fail-closed execution behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve configured SecretRefs before env fallback:** `extensions/discord/src/token.ts:16-17`
  - body: Returning `undefined` for a SecretRef makes `channels.discord.token` look absent, so the default-account path can continue to `DISCORD_BOT_TOKEN`. That violates the documented config-over-env precedence and can run the wrong Discord bot when the configured ref is unavailable; carry a configured-unavailable state instead.
