## Summary

Keep open: this PR targets a real Discord SecretRef crash path, but the patch is not safe because it turns a configured top-level SecretRef into a missing token and can fall through to the default `DISCORD_BOT_TOKEN`.

## What This Changes

The PR changes Discord token normalization so unresolved SecretRef token objects return no token instead of throwing, and updates the corresponding unit test.

## Best Possible Solution

Keep separate read-only inspection semantics for status/health snapshots while preserving fail-closed runtime token resolution, so configured SecretRefs never fall back to env when unresolved.

## Reproduction Assessment

Yes. Source inspection shows current `resolveDiscordToken` passes SecretRef objects into strict secret normalization, and current tests assert the unresolved SecretRef throw; I did not run tests because this was a read-only review.

## Solution Assessment

No. Returning `undefined` from the shared normalizer is not the best fix because callers treat `undefined` as missing and may continue to env fallback; the safer repair should carry a configured-unavailable state only for read-only status paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve configured SecretRefs before env fallback:** `extensions/discord/src/token.ts:16-17`
  - body: Returning `undefined` for a SecretRef makes `channels.discord.token` look absent, so the default-account path can continue to `DISCORD_BOT_TOKEN`. That breaks the existing config-over-env behavior and can run the wrong Discord bot when the configured ref is unavailable; keep configured-unavailable distinct from missing instead.
  - confidence: 0.91
