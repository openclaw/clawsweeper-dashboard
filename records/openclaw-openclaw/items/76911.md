## Summary

Keep open: the patch fixes the observed crash path too broadly by turning configured SecretRefs into missing tokens, which can bypass Discord config-over-env precedence and use `DISCORD_BOT_TOKEN` for the default account.

## What This Changes

This PR changes Discord token normalization so unresolved SecretRef objects return no token and updates the token test to expect `source: "none"` instead of a throw.

## Best Possible Solution

Use a distinct configured-unavailable token state or separate read-only inspection resolver for status/health paths, while keeping Discord execution and startup token resolution strict and fail-closed.

## Reproduction Assessment

Yes, from source inspection. Current main passes unresolved SecretRef objects into strict token normalization, and the PR diff can be followed to an env fallback when `DISCORD_BOT_TOKEN` is present.

## Solution Assessment

No. Returning `undefined` from the shared normalizer fixes the throw but loses the difference between missing and configured-unavailable credentials; a tagged/inspect-mode resolver is the safer narrow fix.
