## Summary

Keep open: the PR addresses a real source-level crash path, but the proposed `undefined` return makes a configured Discord SecretRef indistinguishable from a missing token and can fall through to `DISCORD_BOT_TOKEN`.

## What This Changes

The PR changes Discord token normalization so unresolved SecretRef objects return no Discord token instead of throwing, and updates the corresponding unit test.

## Best Possible Solution

Preserve a distinct configured-unavailable token state for read-only status/health paths while keeping Discord startup and execution token resolution fail-closed unless the configured SecretRef resolves.

## Reproduction Assessment

Yes. Current source passes unresolved SecretRef objects into the strict secret normalizer, and current tests assert that unresolved Discord config tokens throw; I did not execute tests because this review is read-only and `pnpm` is unavailable here.

## Solution Assessment

No. Returning `undefined` from the shared normalizer loses the difference between missing and configured-unavailable credentials; a safer fix should use inspect-mode or tagged token resolution only for read-only/status surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve configured SecretRefs before env fallback:** `extensions/discord/src/token.ts:16-17`
