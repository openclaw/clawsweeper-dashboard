## Summary

Keep open; the PR addresses a real current-main Discord SecretRef crash path, but the implementation is not mergeable because it collapses an unresolved configured SecretRef into a missing token and can fall through to env fallback.

## What This Changes

The PR changes Discord token normalization so an unresolved SecretRef returns no token instead of throwing, and updates the matching unit test expectation.

## Best Possible Solution

Keep read-only/status snapshots graceful with a distinct configured-unavailable token state, while preserving fail-closed execution/startup behavior unless the selected SecretRef resolves.

## Reproduction Assessment

Yes, from source inspection: current main passes unresolved Discord SecretRef objects into strict token normalization, and current tests assert the throw. I did not execute tests because this review is read-only.

## Solution Assessment

No. Returning undefined from the shared normalizer fixes the throw path but also makes a configured-unavailable token indistinguishable from no config token, so a tagged/inspect-mode path is safer.
