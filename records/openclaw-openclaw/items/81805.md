## Label Justifications

- `P2`: This is a focused performance improvement attempt with real user impact, but it changes channel/secrets control-plane behavior and needs review before merge.
- `merge-risk: 🚨 compatibility`: The new opt-in mode changes shared doctor and secret contract semantics for operators who enable it, which can diverge from existing configure/audit expectations.
- `merge-risk: 🚨 security-boundary`: The secrets loader shortcut can suppress bundled channel credential target discovery from audit/configure registry flows under the opt-in mode.

## Summary

Keep open: the performance problem is credible and the PR has useful real-output proof, but the patch puts the lazy-mode shortcut in shared doctor/secrets contract loaders where it can hide credential registry data and still fall back to full channel runtime loading.

## What This Changes

Adds a changelog entry and an opt-in `OPENCLAW_LAZY_BUNDLED_CHANNEL_ARTIFACTS=1` short-circuit to bundled channel doctor and secrets public-artifact loaders.

## Best Possible Solution

Keep `openclaw status --json` and other cold control-plane discovery on metadata or light artifacts while preserving complete channel secret and doctor contracts for configure, audit, doctor, and runtime flows; coordinate broader channel lazy-loading behavior with https://github.com/openclaw/openclaw/issues/65444.

## Reproduction Assessment

Yes for the PR-introduced regressions by source inspection: with the env var set, the changed loaders return `undefined`, and current callers either omit secret target entries or fall through to bootstrap runtime loading. The original 4.27 timing problem is supported by copied live output in the PR body, but I did not rerun that install locally.

## Solution Assessment

No. The safer fix is a metadata/light-artifact status path or a narrower caller option, not a shared contract-loader shortcut that changes audit/configure and doctor semantics.
