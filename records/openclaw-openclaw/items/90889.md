## Summary

The patch is still necessary and appears correct: current main lets a positive override overstate a smaller known model window, while this branch centralizes the cap without changing unknown-model behavior. Exact-head CI is green, but merging intentionally changes effective configuration and persisted session metadata for affected users, so maintainer acceptance of that compatibility-visible correction remains appropriate.

## What This Changes

The PR makes positive session context-token overrides act as ceilings against known model windows and persists the corrected effective window after agent runs.

## Best Possible Solution

Keep the shared resolver as the single authority, cap an override only against an actual fixed/configured/discovered model limit, preserve it when no limit is known, and use the same result for status and persisted session accounting.

## Reproduction Assessment

## Solution Assessment

Yes. The shared resolver is the narrowest maintainable owner because status and persistence already depend on it; the branch also preserves lower overrides and unknown-model behavior.
