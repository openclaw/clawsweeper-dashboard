## Summary

Keep open: current main still has a source-visible gap where native DeepSeek V4 exposes `xhigh|max` through the runtime provider hook but not through the lightweight provider-policy artifact used by non-runtime thinking resolution paths.

## What This Changes

The issue reports that the Control UI `/think` argument picker omits `xhigh` and `max` for native DeepSeek V4 models.

## Best Possible Solution

Publish the DeepSeek V4 thinking profile through the lightweight DeepSeek provider-policy artifact, preferably by sharing one local helper with the runtime hook so the two surfaces cannot drift.

## Reproduction Assessment

Yes, by source inspection: a fallback/non-runtime thinking-resolution path for `deepseek/deepseek-v4-pro` reaches `resolveBundledProviderPolicySurface("deepseek")`, whose current policy artifact lacks `resolveThinkingProfile` and therefore cannot return `xhigh` or `max`. I did not establish a live browser reproduction in this read-only review.

## Solution Assessment

Yes: the issue's proposed direction is the narrowest maintainable fix because the Control UI is already provider-profile driven and should not gain DeepSeek-specific regex logic. The fix belongs in the DeepSeek plugin's lightweight provider policy surface plus targeted regression coverage.
