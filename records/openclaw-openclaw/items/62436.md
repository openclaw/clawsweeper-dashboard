## Summary

Keep open. Current main still implements `/v1/chat/completions` as an agent-first Gateway run with generated session keys and no passthrough/no-persistence config surface, so this is a valid new Gateway contract request rather than conservative cleanup.

## What This Changes

Request a lightweight `/v1/chat/completions` mode that keeps Gateway auth, model routing, and fallback while bypassing agent sessions, transcript writes, skills, memory, and other persistent state.

## Best Possible Solution

Define a first-class documented relay or ephemeral-session contract for OpenAI-compatible HTTP calls, covering routing, fallback, auth, prompts, tools, memory, usage, streaming, and persistence lifecycle.

## Reproduction Assessment

Yes. A source-level path is high-confidence: send `POST /v1/chat/completions` without `user` or `x-openclaw-session-key`, and current main generates an `agent:<id>:openai:<uuid>` session key before dispatching through `agentCommandFromIngress`; I did not run it live because this read-only review would create session artifacts.

## Solution Assessment

Unclear. Passthrough mode is plausible, but maintainers should choose between raw provider relay, ephemeral agent sessions, and cleanup/TTL/session-management semantics before changing the public Gateway contract.
