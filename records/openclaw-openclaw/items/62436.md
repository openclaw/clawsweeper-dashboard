## Label Justifications

- `impact:session-state`: The issue is about unwanted session keys, session-store entries, transcript files, and persisted per-request agent state.
- `impact:security`: A no-persistence relay mode would define privacy, auth, prompt, tool, memory, and audit boundaries for an authenticated Gateway surface.

## Summary

Keep open. Current main and the latest release still execute OpenAI-compatible Chat Completions as normal agent runs with generated session keys and durable session metadata, and there is no accepted passthrough/no-persistence config surface; the remaining work is a new Gateway contract that needs maintainer product and security design.

## What This Changes

The issue requests a no-persistence OpenAI-compatible Chat Completions relay mode that preserves auth, model routing, and provider fallback while skipping the normal agent/session pipeline.

## Best Possible Solution

Define a first-class documented relay or ephemeral-session contract for OpenAI-compatible HTTP calls, covering routing, fallback, auth, prompts, tools, memory, usage, streaming, and persistence lifecycle.

## Reproduction Assessment

Yes, at source level. An anonymous POST to /v1/chat/completions without user or x-openclaw-session-key generates an agent:<id>:openai:<uuid> session key before entering agentCommandFromIngress; I did not run it live because this read-only review should not create session artifacts.

## Solution Assessment

Unclear. Passthrough mode is plausible, but maintainers should choose between raw provider relay, ephemeral agent sessions, cleanup/TTL behavior, and REST session-management semantics before changing the public Gateway contract.
