## Summary

Keep open. Current main still treats `/v1/chat/completions` as an agent-first Gateway run with generated session keys and persistent session metadata, and the requested passthrough/no-persistence mode is a new public API/config decision rather than cleanup that can be closed.

## What This Changes

Requested a lightweight OpenAI-compatible Chat Completions relay mode that preserves model routing, fallback, and auth while skipping session/transcript persistence and other agent pipeline state.

## Best Possible Solution

Define a first-class documented relay or ephemeral-session contract for OpenAI-compatible HTTP calls, including routing/fallback/auth plus explicit behavior for tools, prompts, memory, usage, streaming, and lifecycle.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send `/v1/chat/completions` without `user` or `x-openclaw-session-key`, and current main generates an `agent:<id>:openai:<uuid>` session key before dispatching through `agentCommandFromIngress`; I did not run it live because this read-only review must not create session artifacts.

## Solution Assessment

Unclear. Passthrough mode is plausible, but maintainers should choose between raw provider relay, ephemeral agent sessions, and cleanup/TTL/session-management semantics before changing the public Gateway contract.
