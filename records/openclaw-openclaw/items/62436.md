## Summary

Keep open. Current main still routes `/v1/chat/completions` through the normal Gateway agent command path, assigns anonymous requests generated session keys, and persists session metadata/transcripts; the related reports are adjacent but do not replace this narrower no-persistence LLM relay request.

## What This Changes

The issue requests a lightweight Chat Completions passthrough mode that keeps Gateway auth, model routing, and fallback while skipping session persistence, transcripts, skills/tools, memory, and system prompt injection.

## Best Possible Solution

Define a first-class, documented HTTP relay or ephemeral-session contract that separates provider routing/fallback/auth from agent session persistence, with explicit semantics for streaming, usage accounting, tools, system prompts, memory, model overrides, and compatibility with existing OpenAI clients.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `/v1/chat/completions` without `user` or `x-openclaw-session-key`, and the handler generates an `agent:<id>:openai:<uuid>` session key before entering the normal persistence path. I did not run a live request because this review is read-only and the path would create session artifacts.

## Solution Assessment

Unclear. Passthrough mode is plausible for this use case, but maintainers should choose between a raw provider relay, an ephemeral agent session, and cleanup/TTL semantics before an implementation changes the public Gateway contract.
