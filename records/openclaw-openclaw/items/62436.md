## Label Justifications

- `P2`: This is a normal-priority Gateway/API enhancement with real session-bloat impact, but it is not an emergency outage or strict regression.
- `impact:session-state`: The issue is about unwanted session keys, session-store entries, transcript files, skills snapshots, and persisted per-request agent state.
- `impact:security`: A no-persistence relay mode would define privacy, auth, prompt, tool, memory, usage, and audit boundaries for an authenticated Gateway surface.

## Summary

Keep open. Current main and the latest release still run OpenAI-compatible Chat Completions through the normal agent pipeline with generated session keys and durable session metadata, and there is no accepted passthrough or no-persistence Gateway contract.

## What This Changes

The issue requests a lightweight passthrough or ephemeral mode for OpenAI-compatible Chat Completions so one-shot LLM relay calls can avoid durable session persistence.

## Best Possible Solution

Define a first-class documented relay or ephemeral-session contract for OpenAI-compatible HTTP calls, including routing, fallback, auth/scopes, prompts, tools, memory, usage, streaming, observability, and persistence lifecycle.

## Reproduction Assessment

Yes, at source level. An anonymous POST to /v1/chat/completions without user or x-openclaw-session-key still gets an agent:<id>:openai:<uuid> session key before the handler calls agentCommandFromIngress; I did not run it live because this read-only review should not create session artifacts.

## Solution Assessment

Unclear. Passthrough mode is plausible, but maintainers should choose between raw provider relay, ephemeral agent sessions, cleanup/TTL behavior, and REST session-management semantics before changing the public Gateway contract.
