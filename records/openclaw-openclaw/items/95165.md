## Label Justifications

- `P1`: The report describes a real messaging workflow where watchdog abort loops can leave WhatsApp sessions unusable until gateway restart.

## Summary

Keep open: current main and v2026.6.9 still have no pre-first-output or pre-first-token liveness signal that would let slow Claude/Anthropic turns stay distinguishable from a locally wedged embedded run, and the remaining fix needs maintainer policy on watchdog recovery versus provider failover.

## What This Changes

The issue asks for embedded-run or model-call progress during slow Anthropic/Claude first-token waits so stuck-session recovery does not abort active WhatsApp sessions before model failover can run.

## Best Possible Solution

Add an owner-approved liveness signal for provider or Claude live-session calls waiting for their first response, bounded by the existing CLI/provider timeout and failover contracts so truly hung local processes remain recoverable.

## Reproduction Assessment

Yes at source level, but not as a full live repro. Current main has no pre-first-output progress heartbeat for model/Claude live waits, and recovery can abort stale active work after `stuckSessionAbortMs`; the exact WhatsApp/Anthropic timing still needs live confirmation.

## Solution Assessment

Unclear until maintainers choose the liveness policy. A first-token heartbeat is the direct fix, but it must be tied to existing CLI no-output timeout and model failover semantics so genuinely hung subprocesses are still recovered.
