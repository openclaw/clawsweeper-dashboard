- `impact:crash-loop`: The reported failure is a repeated abort/retry loop that requires process-level restart to break.

## Summary

Keep open: current main still has no pre-first-token progress heartbeat for active model calls, so a slow upstream Anthropic/Claude turn can remain silent until diagnostic recovery treats it as stalled and aborts it before model-layer failover finishes. The secondary `released=0` lane-cleanup concern overlaps with an already-open lane-release issue/PR, but that does not cover the first-token heartbeat gap.

## What This Changes

Investigate watchdog aborts of slow Anthropic embedded runs before first token, plus a secondary queued-session cleanup concern after recovery reports `released=0`.

## Best Possible Solution

Add a narrow, owner-approved liveness signal for provider calls waiting for the first response chunk, tied to existing model/CLI timeout and failover semantics, while leaving true local hangs recoverable by the stuck-session watchdog.

## Reproduction Assessment

Yes at source level for the no-pre-first-token progress gap: current main records model-call start and stream progress only after chunks/results, while diagnostic recovery can abort stale active model calls. I did not run a live Anthropic/WhatsApp reproduction, and the reporter's exact `lastProgress=embedded_run:started` signature still needs runtime confirmation on current main.

## Solution Assessment

Unclear until maintainers choose the policy boundary. A first-token heartbeat is the most direct fix, but it must be bounded by the provider/CLI timeout and failover contract so genuinely hung local subprocesses still recover.
