## Best Possible Solution

Keep the no-auth non-loopback refusal, then prove the real update/service-install mutation path and make that path preserve omitted `gateway.bind` or stop before restart with an actionable auth/bind diagnostic.

## Reproduction Assessment

Unclear for the full issue: source and tests prove the exit 78 startup failure once `gateway.bind=lan` and `auth.mode:none` are present, but source inspection did not prove that `openclaw update` writes `lan` for an omitted bind. ## Solution Assessment

Unclear: preserving the omitted bind/default loopback is the narrowest fix if the updater mutates config, while `openclaw config lock` is a separate product feature. The best immediate path is to reproduce the mutation and repair that exact writer without weakening the safety guard.
