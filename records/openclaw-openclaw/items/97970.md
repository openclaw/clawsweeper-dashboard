## Summary

Keep open: current source proves the exit-78 guard and documented loopback default, but current main does not yet identify the update step that writes `gateway.bind=lan`, so the upgrade path still needs maintainer reproduction instead of cleanup closure.

## What This Changes

The issue reports that `openclaw update` writes a missing `gateway.bind` as `lan`, conflicting with `gateway.auth.mode:"none"` and causing gateway exit 78.

## Best Possible Solution

Preserve an omitted `gateway.bind` across update and service-refresh paths so the documented loopback default remains effective; if a path must set a non-loopback bind, it should require explicit auth or abort before restart with an actionable diagnostic.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the startup failure path: current source and tests show `gateway.bind=lan` with `auth.mode:none` exits 78 with the reported message. No for the full update mutation path: source inspection did not identify where `openclaw update` writes `gateway.bind=lan` for an omitted bind.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the writer is found, but preserving the omitted bind/default loopback is the narrowest maintainable fix if the updater is mutating config. The suggested `config lock` is a separate product feature and should not block the bug fix.
