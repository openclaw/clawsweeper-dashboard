## Summary

Keep open: current main and the latest release still hardcode the public npm registry for update metadata, so mirror-configured installs can still time out before the package-manager path runs.

## What This Changes

The issue requests that OpenClaw update checks respect the user’s npm registry mirror configuration instead of always querying registry.npmjs.org.

## Best Possible Solution

Centralize npm registry base resolution for update metadata fetches, honor existing npm registry environment/config where safe, and fall back to the public registry only when no mirror is configured.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: with `npm_config_registry` set, the update metadata path still constructs `https://registry.npmjs.org/openclaw/<target>` before making the fetch.

## Solution Assessment

Yes in substance, but the narrowest maintainable fix belongs in the central npm metadata fetch path rather than `update-channels.ts`, so all startup/status/update callers inherit the same registry resolver.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79140. Current update metadata helpers hardcode `https://registry.npmjs.org/openclaw/<target>` and ignore npm registry mirror configuration. Add a central registry-base resolver for OpenClaw npm metadata fetches, use it from `fetchNpmPackageTargetStatus`/tag/channel helpers, honor existing npm registry environment such as `npm_config_registry` and `NPM_CONFIG_REGISTRY` first, and support bounded npmrc-derived registry config if it can be done without an unbounded package-manager subprocess in Gateway startup. Normalize mirror URLs with the URL API, handle trailing slashes, reject unsafe/malformed values by falling back to `https://registry.npmjs.org/`, and avoid logging credentials.
