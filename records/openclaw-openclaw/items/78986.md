## Summary

Keep open: current main's recall hook already honors `config.agents`, but the session-scoped `/active-memory status` command ignores that gate and can report `on` for a non-targeted agent, matching the user-visible complaint.

## What This Changes

The issue asks for active-memory agent scoping to be reflected correctly when `/active-memory status` is run from agents excluded by `plugins.entries.active-memory.config.agents`.

## Best Possible Solution

Status and session-toggle responses should report effective Active Memory availability for the current agent/session, including when the agent is not targeted, while leaving the existing recall gate intact.

## Reproduction Assessment

Yes from source inspection: configure Active Memory for `sandbox`, run the session-scoped status command from another agent, and the handler can reach the `on for this session` branch because it only checks the session toggle. I did not execute that path live in this read-only review.

## Solution Assessment

Yes; the narrow maintainable fix is in the Active Memory command/status path, with regression coverage for an untargeted agent. The recall runtime should not be broadened or reworked because it already checks the configured agent list.
