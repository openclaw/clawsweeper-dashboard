## Summary

Keep open: current main still treats memory flush as a pre-compaction/threshold mechanism, not as an action-based session-end continuity safeguard. The open reset-memory thread covers part of the same lifecycle gap, but this issue adds distinct product choices around post-work nudges, startup preludes, and optional short-idle session stitching.

## What This Changes

Request to add runtime continuity safeguards such as an action-based session-end memory nudge, startup prelude, and optional sticky sessionId mode for short idle rotations.

## Best Possible Solution

Design one lifecycle continuity policy that coordinates reset-time flush, post-work/session-end nudging, startup awareness, and any sticky-session mode with explicit config, auditability, timeouts, sandbox limits, and docs.

## Reproduction Assessment

Not as a strict bug repro; this is a feature request. Source inspection gives a high-confidence path for the gap: idle/daily reset can rotate the session, while the only agentic memory flush is compaction/threshold-gated and there is no action-based post-turn flush or sticky-session mode.

## Solution Assessment

Unclear. The direction is plausible, but the best implementation needs maintainer choices between post-work nudge, pre-reset flush, startup prelude, sticky session stitching, and how those interact with https://github.com/openclaw/openclaw/issues/45608 and https://github.com/openclaw/openclaw/issues/79903.
