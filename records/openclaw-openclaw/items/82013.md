## Summary

Keep open. Current main still has idle/daily session rotation and a compaction-scoped memory flush, but no significant-work session-end nudge, continuity prelude, or sticky-sessionId mode; the nearby reset-memory thread does not cover this whole product decision.

## What This Changes

Request a continuity mechanism that prompts or stitches sessions so significant in-flight work is written to durable memory before a rotated session loses context.

## Best Possible Solution

Keep this as a design item for a narrow, audited continuity mechanism that reuses memory-flush infrastructure and recorded tool-effect metadata without committing to sticky sessionId semantics until maintainers choose the UX and config contract.

## Reproduction Assessment

No, not as a high-confidence contract bug reproduction. Source inspection confirms the current rotation and compaction-scoped flush behavior, but current docs/tests do not promise short-idle stitching or significant-work session-end nudges.

## Solution Assessment

Unclear pending product decision. A significant-work session-end nudge looks narrower than sticky sessionId reuse, but maintainers need to define trigger policy, failure visibility, and whether this belongs before session.ended, on the next session start, or in reset-time flushing.
