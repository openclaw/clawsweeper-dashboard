## Summary

Keep open. Current main intentionally rotates expired idle/daily sessions and only runs the automatic memory-flush turn around compaction pressure, so the requested session-end continuity nudge or sticky session behavior is not already implemented and needs a product/ownership decision.

## What This Changes

The issue requests runtime support to preserve continuity across short sessionId rotations by nudging memory flushes before session end, injecting a continuity prelude on the next session, or adding an optional sticky-session mode.

## Best Possible Solution

Keep this as a design item for a narrow continuity mechanism, likely reusing the existing memory-flush machinery plus recorded tool metadata, while avoiding default sticky-session semantics until maintainers choose the UX and config contract.

## Reproduction Assessment

Not applicable as a strict bug reproduction: source inspection confirms the current idle rotation and compaction-only memory flush behavior that the request wants to change, but current docs/tests do not promise short-idle session stitching or session-end nudging.

## Solution Assessment

Unclear pending product decision: the session-end nudge appears narrower than sticky sessionId reuse, but maintainers need to define significant-work triggers, cost/latency limits, and whether this belongs before `session.ended` or as a next-session prelude.
