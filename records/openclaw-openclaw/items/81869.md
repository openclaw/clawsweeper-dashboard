## Summary

Keep open: the underlying CLI context-engine gap is real, but the PR still has a blocking lifecycle cleanup bug for background-maintenance engines.

## What This Changes

The PR resolves a non-legacy context engine for CLI runs, finalizes successful CLI attempts through the shared context-engine lifecycle, and adds focused regression tests.

## Best Possible Solution

Land the CLI lifecycle fix after assigning clear disposal ownership for context engines handed to deferred maintenance.

## Reproduction Assessment

Yes, source-level: current main returns successful CLI results without resolving or finalizing a non-legacy context engine, and the linked report provides a concrete CLI-vs-embedded live A/B. I did not run live claude-cli plus LCM in this read-only review.

## Solution Assessment

No, not as written: reusing the shared finalizer is the right direction, but the cleanup path leaks engines when turn maintenance is deferred. The safer fix is to dispose after deferred maintenance completes or explicitly transfer disposal ownership to the scheduler.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Dispose deferred CLI context engines:** `src/agents/cli-runner.ts:157-158`
  - body: When a successful CLI turn schedules background context-engine maintenance, this cleanup returns without disposing the engine. The deferred scheduler only clears its tracking map and reschedules; it never disposes the run-scoped engine instance, so engines with DB handles or timers can leak per successful background-maintenance CLI turn. Please dispose after the deferred worker finishes or transfer ownership to the scheduler.
  - confidence: 0.86
