## Summary

Keep open. The PR targets a real current-main CLI context-engine lifecycle gap and now includes plausible terminal real-behavior proof, but the latest diff still has a blocking resource-lifecycle bug for background context engines that should be fixed before merge.

## What This Changes

The PR resolves a non-legacy context engine during CLI preparation, bootstraps and finalizes successful CLI turns through the shared context-engine lifecycle, carries context-engine fields in the prepared CLI context, and adds focused regression tests.

## Best Possible Solution

Land the CLI lifecycle fix with a bounded cleanup adjustment that preserves background maintenance deferral and disposes resolved engines once their foreground or deferred work is no longer using them.

## Reproduction Assessment

Yes, source-level: current main's CLI runner returns successful CLI results without resolving or finalizing a non-legacy context engine, while the linked issue provides a concrete live A/B between CLI and embedded providers. I did not rerun the live claude-cli plus LCM setup in this read-only review.

## Solution Assessment

No, not as written: reusing the shared lifecycle is the right direction, but avoiding the deferred-maintenance race by never disposing background engines leaves a resource leak. A safer fix should tie disposal to deferred maintenance completion or the no-deferred cleanup path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Dispose background engines after deferred work:** `src/agents/cli-runner.ts:152-153`
  - body: This return makes every CLI-resolved background context engine skip disposal forever. `prepareCliRunContext` resolves an engine instance for the run, and the deferred maintenance scheduler keeps that same instance but only clears scheduler state when the task finishes; it never calls `dispose()`. A background LCM-style engine that opens DB handles or timers will leak an instance per CLI turn, and failed or blocked turns leak too because no deferred turn task is scheduled. Please preserve the deferred-maintenance race fix, but arrange disposal after the last foreground/deferred maintenance path instead of returning here.
  - confidence: 0.84
