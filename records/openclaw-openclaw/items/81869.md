## Summary

Keep open: this is a plausible bug-fix PR for the CLI context-engine lifecycle, and the supplied terminal proof is sufficient, but the latest head still has a P2 lifecycle bug where coalesced background maintenance can dispose the rerun engine before its deferred worker runs.

## What This Changes

The PR resolves non-legacy context engines during CLI preparation, bootstraps and finalizes successful CLI attempts through the shared context-engine lifecycle, threads deferred maintenance callbacks, and adds CLI context-engine regression tests.

## Best Possible Solution

Land the CLI lifecycle fix after deferred-maintenance disposal ownership is tied to the actual worker or rerun that uses each context-engine instance.

## Reproduction Assessment

Yes, source-level: current main returns successful CLI results without resolving or finalizing a non-legacy context engine, and the linked report provides a concrete CLI-vs-embedded live A/B. I did not rerun live claude-cli plus LCM in this read-only review.

## Solution Assessment

No, not as written: reusing the shared lifecycle is the right direction, but the latest deferred-disposal path can tear down a coalesced rerun engine before its maintenance executes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep coalesced deferred engines alive:** `src/agents/cli-runner.ts:164-166`
  - body: When a second CLI turn schedules background maintenance while the same session already has an active deferred run, the scheduler stores the second run's engine in `latestParams` but returns the first active promise. This `finally` then disposes that second engine as soon as the first promise settles, exactly when the scheduler queues the rerun that will use it, so rapid consecutive CLI turns can run maintenance against a disposed DB/timer-backed engine. Tie disposal to the specific deferred worker/rerun, or move disposal ownership into the scheduler.
  - confidence: 0.86
