## Summary

Keep open. Current main still has the stale-snapshot gap for offline skills config edits followed by restart: the disable filter works when a snapshot rebuilds, but persisted snapshot validity is still based on process-local version state plus the agent skill filter, and #71497 remains an unmerged draft parent.

## What This Changes

The issue asks for existing long-lived sessions to stop reusing stale skillsSnapshot data after an offline `skills.entries.<name>.enabled=false` config change and gateway restart.

## Best Possible Solution

Land restart-stable skills snapshot invalidation based on the effective snapshot inputs, or update and land #71497 with proof that an offline disabled-skill config edit removes an already-snapshotted skill after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, a gateway restart that returns `getSkillsSnapshotVersion()` to 0, and another turn on the same session.

## Solution Assessment

Yes for the issue's durable input-fingerprint direction, because it directly models snapshot validity across restarts and offline edits. The narrower startup-version seeding in #71497 is acceptable only if it explicitly covers this disabled-skill case across gateway reply and CLI agent paths.
