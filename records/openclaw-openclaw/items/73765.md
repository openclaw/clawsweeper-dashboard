## Summary

Keep open. Current main still has the restart/offline-edit invalidation gap: `enabled:false` is honored when a skills snapshot is rebuilt, but existing persisted snapshots can be reused after restart because snapshot validity is still based on process-local numeric versions and agent skill filter only.

## What This Changes

The issue asks OpenClaw to invalidate persisted session skills snapshots when `skills.entries.<skill>.enabled:false` is added after an existing long-lived session already cached that skill.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, preferably a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots or entries. If maintainers prefer the smaller version-seeding repair, it should prove that an already-snapshotted skill disappears after an offline `enabled:false` config edit and restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an existing session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, then gateway restart and another turn on the same session; current main computes version `0` again and can reuse the stale snapshot.

## Solution Assessment

Yes for the issue's durable inputs-fingerprint direction, because it directly models snapshot validity across restarts and offline edits. The narrower #71497 version-seeding path is a plausible smaller fix only if it explicitly covers this disabled-skill case.
