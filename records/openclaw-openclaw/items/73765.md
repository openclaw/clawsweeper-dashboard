## Summary

Keep open. Current main still reuses persisted `version: 0` skills snapshots after restart or offline config edits because snapshot freshness is checked only against the process-local numeric version and agent skill filter; the open draft PR #71497 targets the same restart root cause but has not landed.

## What This Changes

The issue asks OpenClaw to invalidate existing session skills snapshots when `skills.entries.<skill>.enabled: false` is added so disabled skills stop appearing in long-lived session prompts after restart or offline config edits.

## Best Possible Solution

Add restart-stable skills snapshot invalidation keyed to the effective inputs that make a snapshot valid, preferably a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries. If maintainers choose the narrower #71497 version-seeding repair, it should include a regression for disabling an already-snapshotted skill after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an existing session with `skillsSnapshot.version: 0`, a `skills.entries.<name>.enabled=false` edit made while the gateway is stopped, then a restart and another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes for the issue's durable invalidation direction: it directly models when the cached prompt is stale instead of relying only on an in-memory watcher bump. The existing watcher-only mitigation is not sufficient after restart or offline edits.
