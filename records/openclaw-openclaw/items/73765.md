## Summary

Keep open. Current main still reuses a persisted skills snapshot after an offline skills config edit plus restart because snapshot validity is based on an in-process numeric version and agent skill filter, not a restart-stable config or skill-root fingerprint.

## What This Changes

The issue reports that `skills.entries.<name>.enabled: false` can be ignored in existing long-lived sessions when the disabled-skill config edit happens outside the running gateway watcher.

## Best Possible Solution

Land restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, or adjust #71497 so it proves an offline `enabled:false` edit removes an already-snapshotted skill after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, a gateway restart, and another turn on the same session while `getSkillsSnapshotVersion()` is again `0`.

## Solution Assessment

Yes for the issue's durable inputs-fingerprint direction, because it directly models snapshot validity across restarts and offline edits. The narrower version-seeding approach in #71497 is acceptable only if it explicitly covers this disabled-skill case.
