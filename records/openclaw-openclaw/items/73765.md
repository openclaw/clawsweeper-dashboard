## Summary

Keep open. Current main still has the restart/offline invalidation gap: persisted skills snapshots are validated by a process-local numeric version and agent skill filter, while live `skills.*` config reload only helps when the running gateway observes the edit.

## What This Changes

Existing long-lived sessions should stop advertising a skill after `skills.entries.<name>.enabled=false`, including after an offline config edit followed by gateway restart.

## Best Possible Solution

Land restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, with regression coverage for an offline `enabled:false` edit on an existing session after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, gateway restart, and another turn on the same session while `getSkillsSnapshotVersion()` is again 0.

## Solution Assessment

Yes, durable snapshot-input invalidation is the right direction because it models validity across restarts and offline edits; the current live watcher bump is only a partial mitigation.
