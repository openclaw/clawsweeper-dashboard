## Summary

Keep open. Current main still can reuse a persisted `skillsSnapshot.version: 0` after an offline `skills.entries.*.enabled: false` edit and restart because snapshot freshness is process-local and no durable skills-config or input fingerprint is stored; draft maintainer PR #71497 is the active route but has not merged or explicitly closed this report.

## What This Changes

The issue asks OpenClaw to invalidate existing session skill snapshots when a skill is disabled in config before a gateway restart.

## Best Possible Solution

Land restart-stable snapshot invalidation based on effective skills config and skill roots, or update #71497 so it explicitly proves `enabled:false` removes a skill from an already-persisted session after restart.

## Reproduction Assessment

Yes, source-level. A persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled:false` edit, process restart, and another turn while `getSkillsSnapshotVersion()` is again `0` follows the current predicates without rebuilding.

## Solution Assessment

Yes for the durable input-fingerprint direction because it models snapshot validity across restarts and offline edits. The narrower version-seeding direction in #71497 is acceptable only if it explicitly covers this disabled-skill case.
