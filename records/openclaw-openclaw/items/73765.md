## Summary

Keep open. Current main still reuses an existing skills snapshot after an offline `skills.entries.<name>.enabled=false` edit followed by restart, because the refresh predicate is based on process-local version state and skill-filter equality rather than durable skills config inputs.

## What This Changes

The issue asks OpenClaw to invalidate persisted skills snapshots for existing sessions when a skill is disabled in config while the gateway is not watching the edit.

## Best Possible Solution

Land restart-stable skills snapshot invalidation based on the effective skills config and resolved skill roots, or update the existing draft PR so it proves the disabled-skill offline edit case across both gateway replies and direct agent runs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, a gateway restart, and another turn on the same session while `getSkillsSnapshotVersion()` is again `0`.

## Solution Assessment

Yes for the issue's durable input-fingerprint direction, because it models the inputs that make a snapshot valid across process restarts. The narrower version-seeding direction in #71497 is acceptable only if it explicitly covers this disabled-skill case.
