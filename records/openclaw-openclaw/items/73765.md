## Summary

Keep open. Current main still has a restart/offline-edit gap: persisted `version: 0` skills snapshots are reused when the process-local skills version also starts at `0`, so a skill disabled in config can remain in an existing session prompt until a separate refresh path rebuilds the snapshot. Draft PR #71497 targets the same restart-version root cause but is unmerged and should be reviewed before this issue is closed.

## What This Changes

The issue requests durable invalidation of existing session `skillsSnapshot` data when `skills.entries.<skill>.enabled` changes, especially across gateway restarts or config edits made while the gateway is stopped.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, ideally a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots or entries. If maintainers prefer the narrower #71497 version-seeding fix, it should include a focused regression proving an already-snapshotted disabled skill is removed after restart or an offline config edit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an existing session with `skillsSnapshot.version: 0`, a `skills.entries.<name>.enabled=false` edit made while the gateway is stopped, then a restart and another turn on the same session; current main computes version `0` again and reuses the old snapshot.

## Solution Assessment

Yes for the issue's durable inputs-fingerprint direction because it directly models snapshot validity across restarts. The open #71497 version-seeding approach may be an acceptable narrower repair if it proves the disabled-skill restart path and maintainers accept the smaller scope.
