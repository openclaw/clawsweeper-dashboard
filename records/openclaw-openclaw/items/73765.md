## Summary

Keep open. Current main still reuses persisted skills snapshots when both the cached snapshot version and the restarted process-local version are 0, so a skill disabled after the snapshot was saved can remain advertised in an existing session. The docs and rebuild-time filter are correct; the remaining gap is restart-stable invalidation, and #71497 is still an unmerged draft.

## What This Changes

Existing long-lived sessions should rebuild skills snapshots when `skills.entries.<name>.enabled` is set to `false` after the snapshot was persisted, including config edits made before a gateway restart.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, preferably a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots or entries. If maintainers prefer the narrower version-seeding repair in #71497, it should land with a regression proving an already-snapshotted disabled skill is removed after restart or offline config edit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an existing session with `skillsSnapshot.version: 0`, a later `skills.entries.<name>.enabled=false` edit while the gateway is stopped, then a restart and another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes for the issue's durable input-fingerprint direction because it directly models snapshot validity across restarts. The open #71497 version-seeding approach may be an acceptable narrower fix if it proves this disabled-skill restart path and maintainers accept the scope.
