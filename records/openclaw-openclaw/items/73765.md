## Summary

Keep open. Current main still has the offline-edit/restart invalidation gap: disabled skills are filtered only when a snapshot rebuilds, while existing session snapshots can still be reused after restart because validity is process-local version plus agent skill filter, with no durable config or skill-root fingerprint.

## What This Changes

The issue requests restart-stable invalidation of persisted skills snapshots so `skills.entries.<name>.enabled: false` removes disabled skills from existing long-lived sessions after offline config edits and gateway restart.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, or adjust and land #71497 with regression coverage proving an offline `enabled:false` edit removes an already-snapshotted skill after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, restart, and another turn on the same session; current main can compute version `0` again and reuse the cached snapshot.

## Solution Assessment

Yes for the issue's durable inputs-fingerprint direction, because it directly models snapshot validity across restarts and offline edits. The narrower version-seeding approach in #71497 is acceptable only if it explicitly covers this disabled-skill case.
