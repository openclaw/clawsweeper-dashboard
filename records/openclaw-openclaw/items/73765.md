## Summary

Keep open. Current main still reuses a persisted skills snapshot after an offline `skills.entries.<name>.enabled=false` edit followed by restart because the refresh predicate only sees process-local version state and the agent-level skill filter.

## What This Changes

The issue requests restart-stable invalidation of persisted skills snapshots when per-skill config disables a skill after an existing session already cached it.

## Best Possible Solution

Land restart-stable skills snapshot invalidation in the shared refresh path, with regression coverage for an offline `skills.entries.<name>.enabled=false` edit after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline per-skill `enabled:false` config edit, a gateway restart, and another turn on the same session while `getSkillsSnapshotVersion()` is again `0`.

## Solution Assessment

Yes. The durable input-fingerprint direction suggested in the issue is the narrowest robust fix; the version-seeding approach in #71497 is also acceptable if it proves this disabled-skill restart case.
