## Summary

Keep open. Current main still has the offline-edit-after-restart stale snapshot gap: the `enabled:false` filter works when a snapshot is rebuilt, but existing persisted snapshots can still be reused when both the cached snapshot version and the process-local version are `0`.

## What This Changes

The issue requests restart-stable invalidation of persisted skills snapshots so `skills.entries.<name>.enabled:false` takes effect for existing long-lived sessions after offline config edits.

## Best Possible Solution

Land restart-stable snapshot invalidation based on the effective skills config and skill roots, or update the active restart-refresh PR so it explicitly proves disabled skills are removed from already-persisted sessions after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled:false` edit, process restart, and another turn while `getSkillsSnapshotVersion()` is again `0`.

## Solution Assessment

Yes for the durable input-fingerprint direction because it directly models snapshot validity across restarts and offline edits. The narrower version-seeding direction in #71497 is acceptable only if it explicitly covers this disabled-skill case.
