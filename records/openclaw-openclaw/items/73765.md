## Summary

Keep open. Current main still has the stale-snapshot gap for offline skills config edits followed by restart: the disable filter works when a snapshot rebuilds, but the persisted snapshot validity check is still based on a process-local version and skill filter only.

## What This Changes

The issue asks OpenClaw to refresh existing session skill snapshots when `skills.entries.<skill>.enabled=false` is added after the snapshot was persisted and the gateway restarts.

## Best Possible Solution

Land restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, or update #71497 so it proves an offline disabled-skill config edit removes an already-snapshotted skill after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, a gateway restart, and another turn on the same session while `getSkillsSnapshotVersion()` is again `0`.

## Solution Assessment

Yes for the issue's durable input-fingerprint direction, because it directly models snapshot validity across restarts and offline edits. The narrower version-seeding direction in #71497 is acceptable only if it explicitly covers this disabled-skill case.
