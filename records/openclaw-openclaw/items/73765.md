## Summary

Keep open. Current main still reuses persisted version-0 skills snapshots after restart/offline config edits, so a skill disabled in config can remain in an existing session prompt until some other refresh path rebuilds the snapshot. Open draft PR #71497 targets the same restart/version root cause, but it is not merged.

## What This Changes

The issue reports that `skills.entries.<name>.enabled=false` is ignored for existing long-lived sessions when the persisted `skillsSnapshot` predates the config change.

## Best Possible Solution

Add restart-stable skills snapshot invalidation based on the effective inputs that make a snapshot valid, ideally a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries. If maintainers choose #71497's narrower startup-version approach, it should include regression coverage for disabled-skill snapshots after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist a session with `skillsSnapshot.version: 0`, add `skills.entries.<name>.enabled=false` while the gateway is stopped, restart, and send another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes for the issue's durable input-hash direction, because it directly models snapshot staleness. The open draft startup-seeding fix is a plausible narrower repair, but it is not merged and may not cover every stale input change as precisely.
