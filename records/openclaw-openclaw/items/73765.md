## Summary

Keep open. Current main still reuses persisted version-0 skills snapshots after a restart/offline config edit, so a skill disabled through config can remain in an existing session prompt until another refresh path rebuilds the snapshot. The related draft PR #71497 targets the same restart/version root cause but is unmerged and should be reviewed or linked before this issue is closed.

## What This Changes

The issue requests restart-stable invalidation of persisted skills snapshots so `skills.entries.<name>.enabled: false` applies to existing long-lived sessions after config edits made while the gateway is stopped or restarted.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, preferably a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries. If maintainers choose the narrower #71497 version-seeding approach, it should include a focused regression for disabling an already-snapshotted skill after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an existing session with `skillsSnapshot.version: 0`, a `skills.entries.<name>.enabled=false` edit made while the gateway is stopped, then a restart and another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes for the issue's durable inputs-hash direction; it directly models when the persisted snapshot is stale. The open draft startup-version fix may be an acceptable narrower repair if it proves the disabled-skill restart path and maintainers accept the scope.
