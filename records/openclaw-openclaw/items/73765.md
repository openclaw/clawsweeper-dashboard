## Summary

Keep open. Current main still has the offline-edit/restart invalidation gap: `enabled: false` is honored when a skills snapshot rebuilds, but existing persisted snapshots can be reused after restart because validity is still a process-local numeric version plus agent skill filter, without a durable config or skill-root fingerprint.

## What This Changes

The issue asks for `skills.entries.<skill>.enabled=false` changes to invalidate persisted skills snapshots for existing long-lived sessions across gateway restarts and offline config edits.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective skills inputs, or land an adjusted #71497 with regression coverage proving an offline `enabled:false` edit removes an already-snapshotted skill after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, restart, and another turn on the same session; current main can compute version `0` again and reuse the cached snapshot.

## Solution Assessment

Yes for the issue's durable inputs-fingerprint direction, because it models snapshot validity across restarts and offline edits. The narrower version-seeding approach in #71497 is acceptable if it explicitly covers this disabled-skill case.
