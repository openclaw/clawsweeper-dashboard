## Summary

Keep open. Current main still has a restart/offline-edit cache-invalidation gap: persisted skills snapshots are refreshed from a process-local numeric version and agent skill filter only, so a version-0 snapshot can survive a restart even after `skills.entries.<name>.enabled=false` changes.

## What This Changes

The issue requests durable invalidation of persisted skills snapshots so disabled skills stop appearing in existing long-lived sessions after config edits made while the gateway is stopped or after restart.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective inputs that make a skills snapshot valid, ideally a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries. If maintainers prefer #71497’s narrower version-seeding approach, it should prove the disabled-skill-after-restart path with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an existing session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, then a restarted gateway turn in the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes for the issue’s durable inputs-hash direction because it directly models snapshot validity across restarts. The open draft startup-version fix may be acceptable if maintainers intentionally choose the narrower scope and add the disabled-skill regression.
