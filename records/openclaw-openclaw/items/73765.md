## Summary

Keep open. Current main still has the offline-edit/restart invalidation gap: disabled skills are filtered correctly when a snapshot rebuilds, but existing persisted session snapshots can still be reused after restart because validity is only the process-local numeric version plus the agent skill filter.

## What This Changes

Existing sessions should stop exposing skills disabled by `skills.entries.<name>.enabled=false` after an offline config edit and gateway restart, without requiring `/new`.

## Best Possible Solution

Add restart-stable snapshot invalidation based on the effective skills snapshot inputs, or update and land #71497 with regression coverage for the disabled-skill offline-edit case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a persisted session with `skillsSnapshot.version: 0`, an offline `skills.entries.<name>.enabled=false` edit, restart, and another turn on the same session; current main can compute version `0` again and reuse the cached snapshot.

## Solution Assessment

Yes for the issue's durable input-fingerprint direction, because it directly models snapshot validity across restarts and offline edits. A version-seeding fix is acceptable only if it proves this disabled-skill case across both agent-command and auto-reply paths.
