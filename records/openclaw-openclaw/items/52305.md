## Summary

Keep open. Current main still leaves the external `openclaw system event` / gateway `wake` path without an explicit session-targeting surface, while the linked implementation PRs are open and unmerged; cleanup policy should not close this issue before that candidate or a replacement lands.

## What This Changes

The issue asks for reliable async completion reporting by preserving the originating session for external `system event` / `wake` callbacks, or by adding a first-class completion notification path.

## Best Possible Solution

Keep the issue open and resolve it through PR #50818 or a narrower replacement that adds an explicit, validated target-session path for external completion callbacks, preserves default main-session behavior for existing callers, threads the target through protocol, CLI, gateway wake, cron wake, `enqueueSystemEvent`, and `requestHeartbeatNow`, and covers the CLI/gateway/internal exec paths with focused tests.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: the CLI has no `--session-key`, sends only `{ mode, text }`, and the wake/cron path does not propagate caller target context into the heartbeat request.

## Solution Assessment

Unclear. The open PR is a plausible implementation candidate, but the safest end state is a narrowly validated external session-targeting or first-class completion API rather than relying on the existing unqualified `wake` path.
