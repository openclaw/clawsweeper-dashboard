## Summary

Keep open. Current main still lacks an explicit session-targeting surface for external `openclaw system event` / `wake` calls, while the provided GitHub context shows open, unmerged closing PRs that should be reviewed or replaced before this issue is cleaned up.

## What This Changes

The issue asks OpenClaw to preserve or accept originating session context so async task completion events reach the conversation that launched them.

## Best Possible Solution

Resolve this through the existing implementation candidates or a narrower replacement that adds a validated target-session completion path while preserving main-session defaults and covering CLI, gateway wake, cron wake, and async exec/subagent paths with focused tests.

## Reproduction Assessment

Yes from source inspection, though not live-reproduced. The current CLI sends only `{ mode, text }`, the wake schema and handler do not carry a typed session target, and cron wake enqueues/request heartbeats without a session key.

## Solution Assessment

Unclear. The linked PRs are plausible implementation candidates, but the maintainable end state should be a narrowly validated external session-targeting or first-class completion API rather than expanding unqualified wake behavior without routing checks.
