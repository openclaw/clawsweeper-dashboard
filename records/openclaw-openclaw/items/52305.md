## Summary

Keep open. Current main has session-scoped internal event/heartbeat plumbing, but the external `openclaw system event` / `wake` CLI path still cannot carry a caller-supplied target session, and open closing PRs remain the right implementation candidates to review first.

## What This Changes

The issue asks for async/background task completion reports to preserve or accept originating session context instead of relying on main-session wake inference.

## Best Possible Solution

Land or replace the existing session-targeting implementation with a narrow additive CLI/RPC completion path that validates session keys, preserves the current main-session default, and covers CLI wake, gateway wake, cron wake, exec, and subagent completion paths with focused tests.

## Reproduction Assessment

Yes from source inspection, though not live-reproduced. The CLI sends only `{ mode, text }`, the wake schema and handler do not carry a typed session target, and manual wake still enqueues/requests heartbeat without session context.

## Solution Assessment

Unclear. The linked PRs are plausible candidates, but the maintainable end state should be a narrowly validated external session-targeting or completion API rather than broadening unqualified wake behavior without routing checks.
