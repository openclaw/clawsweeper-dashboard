## Summary

Keep open. Current main still has a valid remaining gap for external `openclaw system event` / `wake` callers: internal exec completion paths now carry session keys, but the CLI/protocol wake path still cannot accept and preserve an explicit target session, and open closing PRs are already tracking candidate fixes.

## What This Changes

Request session-targeted async completion reporting so external runners and background tasks can notify the originating OpenClaw conversation reliably.

## Best Possible Solution

Resolve this through the existing implementation candidates or a narrower replacement that adds a validated target-session completion path while preserving main-session defaults and covering CLI, wake protocol, cron wake, and async exec/subagent boundaries with focused tests.

## Reproduction Assessment

Yes from source inspection, though not live Telegram/WhatsApp reproduction. The current CLI sends only `{ mode, text }`, the wake schema and handler do not carry a typed session target, and manual cron wake requests heartbeat without a `sessionKey`.

## Solution Assessment

Unclear. Explicit session targeting or a first-class completion callback is the right product direction, but maintainers should choose the smallest supported surface rather than expanding unqualified wake behavior without routing and validation checks.
