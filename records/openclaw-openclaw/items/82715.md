## Summary

Keep open: the PR is focused on the right launcher surface, but the patched containment check still does not handle the fresh `NODE_COMPILE_CACHE` base shape documented for users, and the PR does not provide real packaged-launcher before/after proof.

## What This Changes

The PR adds a private helper that treats a compile-cache directory equal to or nested under the packaged cache root as already anchored and uses it before respawning.

## Best Possible Solution

Revise the launcher fix around the actual fresh packaged process path, then prove it with a packaged fixture or real CLI run that shows the before/after respawn behavior while preserving cache scoping across OpenClaw versions.

## Reproduction Assessment

Yes for the review finding: source inspection plus a local Node command show `NODE_COMPILE_CACHE=<base>` produces `<base>/v...`, while OpenClaw computes `<base>/openclaw/<version>/<marker>`. That means the proposed helper still returns false for the documented fresh-invocation path.

## Solution Assessment

No. The containment helper is only useful when the active cache is already below the computed scoped root, but the launcher computes that root from the current env base, so the reported fresh process remains outside it.
