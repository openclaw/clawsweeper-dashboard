## Summary

Keep this PR open, but it is not currently mergeable: the proposed containment check still returns false for the documented fresh NODE_COMPILE_CACHE base path, so it does not prove the repeated packaged-launcher respawn is fixed.

## What This Changes

The PR adds a launcher helper in openclaw.mjs that skips packaged compile-cache respawn when the active cache directory is equal to or nested under the computed desired cache root.

## Best Possible Solution

Revise the packaged launcher logic around the actual Node env cache root shape, add focused launcher coverage for OPENCLAW_PACKAGED_COMPILE_CACHE_RESPAWNED, and preserve cache scoping across OpenClaw versions.

## Reproduction Assessment

Yes for the review finding: source inspection plus a local Node command show that a fresh NODE_COMPILE_CACHE base produces a Node-versioned sibling of OpenClaw's scoped desired root, making the PR helper return false. I did not run the full packaged launcher e2e path.

## Solution Assessment

No. A child-of-desired containment check only helps after the active cache is already below the scoped root; the reported fresh env-enabled packaged process remains outside that root and still respawns.
