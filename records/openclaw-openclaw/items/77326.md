## Summary

This PR should stay open because the proposed sessions fix changes the dashboard `sessions.create` path, while the reported Telegram/Discord `/new` path is handled by auto-reply session reset code that still preserves stale `model`/`modelProvider` fields when it merges the new entry over the existing store row.

## What This Changes

The PR changes `sessions.create` to send `model: null` when no model is provided, adds a sessions-patch regression test and changelog entry, and also includes unrelated BlueBubbles per-DM system-prompt docs/config/runtime/test changes.

## Best Possible Solution

Fix the channel reset path so `/new` and `/reset` clear or recompute stale runtime `model`/`modelProvider` fields when no user model override is preserved, with regression coverage in the auto-reply session tests.

Yes, source inspection gives a high-confidence reproduction path: seed a channel session row with `model`/`modelProvider` and no overrides, change the configured default, then process `/new`; the reset merge leaves the stale runtime fields in the store. No. The PR's `sessions.create` change may help dashboard-created sessions, but it is not the narrow fix for the reported channel `/new` bug; the safer repair is in `src/auto-reply/reply/session.ts` or a shared reset helper used by that path.
