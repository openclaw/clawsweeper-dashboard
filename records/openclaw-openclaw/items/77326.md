## Summary

Keep open: the PR targets the Gateway `sessions.create` RPC, but the reported chat `/new` path is handled by auto-reply reset code that can still merge stale runtime `model`/`modelProvider` fields back into the session store. The branch also mixes unrelated BlueBubbles per-DM system-prompt work into the sessions fix.

## What This Changes

The PR changes `sessions.create` model patching, adds a sessions-patch regression and changelog entry, and includes BlueBubbles per-DM system-prompt config/docs/runtime/test changes.

## Best Possible Solution

Fix the auto-reply `/new` and `/reset` write path so it clears or recomputes runtime `model`/`modelProvider` when no user model override is preserved, with focused `initSessionState` regression coverage; handle the BlueBubbles feature separately.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a channel session row with `model`/`modelProvider` and no overrides, change the configured default, then process `/new`; the auto-reply reset write can preserve the stale runtime fields.

## Solution Assessment

No. Changing `sessions.create` may help dashboard-created sessions, but it is not the narrow repair for the reported channel `/new` path; the safer fix belongs in the auto-reply reset write path or a shared reset helper used there.
