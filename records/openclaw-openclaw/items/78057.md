## Summary

Current main still rejects `.zip` host-local message-tool attachments with the reported `application/zip` path, and the affected code is a file-read security boundary, so this should stay open for maintainer review rather than cleanup close.

## What This Changes

The issue asks to restore sending `.zip` files as Discord DM/message-tool attachments after 2026.5.x began rejecting host-local zip media as `application/zip`.

## Best Possible Solution

Restore safe `.zip` attachment sends through the shared media loader, likely by allowing buffer-verified archive MIME only within the existing host-read policy and adding regression coverage for Discord/message action attachment hydration.

## Reproduction Assessment

Yes from source inspection: host-mode message-tool attachment hydration enables `hostReadCapability`, and `assertHostReadMediaAllowed` rejects a sniffed `application/zip` with the reported error. I did not perform a live Discord send in this read-only sweep.

## Solution Assessment

Unclear until maintainer review: adding `.zip` to the buffer-verified document allowlist is the narrow likely fix, but the archive policy should preserve the host-local read guard instead of broadly allowing arbitrary binary files.
