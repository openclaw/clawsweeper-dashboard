- `P2`: The PR addresses a real but bounded Docker/browser sandbox startup failure for users with overlapping custom binds.

## Summary

Keep open: the fix is focused and the protected-mount-wins direction matches the existing sandbox contract, but the raw destination comparison still misses path aliases and the external PR has only unit/constructed-argument proof rather than live Docker behavior proof.

## What This Changes

The PR filters Docker and browser sandbox custom bind mounts when their container destination exactly matches a protected read-only workspace skill mount destination.

## Best Possible Solution

Keep protected skill mounts authoritative, normalize custom bind container destinations through the existing container-path helper before filtering in both shell and browser sandboxes, add alias regression coverage, and require redacted live Docker create proof before merge.

## Reproduction Assessment

Yes from source, though not from a live Docker run in this review. Current main appends custom binds and then protected skill mounts for the same destination, matching the linked duplicate-mount report when reserved-target validation is deliberately bypassed.

## Solution Assessment

No, not yet. Filtering colliding custom binds is the right layer and safer than dropping protected mounts, but the filter must normalize container destinations and prove the real Docker path before it is the best fix.
