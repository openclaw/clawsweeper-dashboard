## Label Justifications

- `P1`: The PR targets a broken QMD memory-search workflow for mcporter 0.10+ users that can affect real agent memory searches.
- `merge-risk: 🚨 compatibility`: The diff changes mcporter config discovery, generated config routing, environment inheritance, and daemon identity for existing QMD setups.
- `merge-risk: 🚨 session-state`: Misclassifying a mcporter server can make QMD memory search read from or write to the wrong agent-scoped index or daemon state.
- `merge-risk: 🚨 availability`: Incorrect generated/external classification can leave QMD memory search failing or falling back for users with supported external configs.

## Summary

Keep open: current main and v2026.6.8 still use the shared QMD-scoped env for mcporter, so this PR remains useful, but it still has a blocking compatibility gap around mcporter-supported JSONC syntax in `mcporter.json`.

## What This Changes

The PR changes QMD mcporter routing so bare QMD servers use generated per-agent mcporter configs while user/workspace-owned mcporter definitions stay external.

## Best Possible Solution

Keep the generated/external design, but make the raw mcporter probe follow mcporter-compatible JSONC parsing for both `.json` and `.jsonc` before merge.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 pass QMD-scoped XDG env into mcporter, while mcporter 0.10 reads XDG_CONFIG_HOME for config discovery. The remaining review finding is also source-reproducible from mcporter's JSONC parser and the PR's `.jsonc`-only raw fallback.

## Solution Assessment

No, not yet: the generated/external split is the right owner-boundary fix, but raw config probing must match mcporter's JSONC contract before this is the best mergeable solution.
