## Summary

Keep open: current main and the latest shipped release still use the misleading `target-none` diagnostic for the default disabled-delivery heartbeat path, and the two linked fix PRs are closed unmerged rather than active canonical landing paths.

## What This Changes

The issue requests clearer heartbeat skipped diagnostics when `agents.defaults.heartbeat.target` defaults to `none`, without changing heartbeat delivery behavior.

## Best Possible Solution

Preserve `target-none` as the stable machine reason and add a human-readable disabled-delivery hint or presentation-layer message, with focused resolver, heartbeat-event, CLI/status formatting, and regression tests.

## Reproduction Assessment

Yes. Current main source shows the default or explicit `target: "none"` path still emits `reason: "target-none"`, and the gateway `last-heartbeat` method returns that stored event directly; no runtime command was run because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the public diagnostic contract. The additive hint/status-rendering shape from the closed related PR looks safer than renaming the machine reason, but it still needs maintainer acceptance.
