## Summary

Keep open: current main and the latest release still expose the confusing `target-none`/missing-channel diagnostic, and the two direct fix PRs were closed unmerged rather than superseding the issue.

## What This Changes

The issue asks for clearer heartbeat diagnostics when heartbeat delivery is disabled by the default `target: none` setting.

## Best Possible Solution

A maintainer should either sponsor a small additive hint/status-display change that preserves `reason: "target-none"`, or explicitly decide that the existing machine reason is acceptable and close the issue as not planned.

## Reproduction Assessment

Yes by source inspection: default target resolution still returns `reason: "target-none"`, and deep status still substitutes `unknown` when the skipped event has no channel. I did not run the CLI because this review is read-only and must not create artifacts.

## Solution Assessment

The best technical shape appears to be the additive hint/status-row approach from https://github.com/openclaw/openclaw/pull/83279, not a raw rename, because it avoids breaking consumers that parse the machine `reason` string.
