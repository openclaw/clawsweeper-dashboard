## Summary

Current main still emits `target-none` for the default or explicit heartbeat `target: "none"` path, and the last-heartbeat RPC exposes that raw reason. The linked fix PR was closed unmerged, so this remains a narrow diagnostic cleanup rather than a close candidate.

## What This Changes

Clarify the heartbeat skipped reason shown when heartbeat delivery is disabled by default so users do not interpret it as missing target resolution.

## Best Possible Solution

Rename only the disabled-by-configuration skip reason to a clearer value such as `delivery-disabled`, update focused tests and changelog, and leave heartbeat scheduling and routing unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows unset and explicit `target: "none"` still return `target-none`, and the runner exposes that reason through the last-heartbeat event; I did not run the CLI in this read-only review.

## Solution Assessment

Yes. A narrow reason-string rename with matching focused tests is the smallest maintainable fix; adding a config option or changing delivery resolution would be broader than needed.
