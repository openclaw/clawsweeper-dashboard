## Summary

Keep open: current main still emits the misleading `target-none` diagnostic for the default-disabled heartbeat path, and the two direct fix PRs are closed unmerged rather than canonical landing paths.

## What This Changes

The issue asks for clearer heartbeat diagnostics when delivery is intentionally disabled by the default `agents.defaults.heartbeat.target: "none"` configuration.

## Best Possible Solution

Keep `target-none` as the stable machine reason and add a human-readable disabled-delivery hint or presentation layer text, with focused resolver, event, CLI/status formatting tests.

## Reproduction Assessment

Yes. Current main source and tests show the default-disabled heartbeat target still emits `reason: "target-none"`, and `system heartbeat last` returns that raw event; no CLI run was performed because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the public diagnostic contract. The additive hint/status rendering shape from the closed related PR looks safer than renaming the machine reason, but it still needs maintainer acceptance.
