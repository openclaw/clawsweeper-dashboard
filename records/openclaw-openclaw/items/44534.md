## Summary

Current main and the latest release still emit `reason: "target-none"` for default disabled heartbeat delivery, so this issue is still valid; the remaining decision is the exact diagnostic/API shape rather than whether the confusion exists.

## What This Changes

Requests clearer heartbeat skipped diagnostics when the default heartbeat delivery target disables external delivery.

## Best Possible Solution

Preserve `target-none` as the machine-readable reason and, if maintainers accept the additive diagnostic surface, add a human-readable disabled-delivery hint to heartbeat events and status output.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` both return `reason: "target-none"` for disabled heartbeat delivery and format missing last-heartbeat channels as `unknown`. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear until maintainers confirm the diagnostic contract. The safest candidate is the prior additive-hint/status-row approach because it improves human output while preserving the existing machine reason.
