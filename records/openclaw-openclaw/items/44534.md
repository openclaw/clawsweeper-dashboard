## Summary

Keep open: current main and the latest release still use `target-none` for the default heartbeat delivery-disabled case, and the linked direct fix was closed unmerged for PR-count management rather than because the behavior was resolved.

## What This Changes

The issue asks for clearer heartbeat skip wording when the default `target: "none"` disables external delivery.

## Best Possible Solution

Land a narrow diagnostic cleanup that keeps heartbeat delivery behavior unchanged while making the disabled-by-configuration case read as intentional.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: unset/default heartbeat target resolves to `target === "none"`, returns `reason: "target-none"`, and the heartbeat runner emits that reason in the skipped heartbeat event.

## Solution Assessment

Yes. The narrowest maintainable fix is a diagnostic wording change, such as renaming the no-delivery reason to `delivery-disabled` or adding a CLI hint, without changing routing, scheduling, defaults, or delivery policy.
