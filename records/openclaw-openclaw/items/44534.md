## What This Changes

The issue asks for the default heartbeat skipped reason to read as intentional delivery-disabled configuration instead of a missing target/routing failure.

## Best Possible Solution

Land a narrow diagnostic-wording fix that keeps heartbeat delivery behavior unchanged, updates the focused expectations, and documents the new or clarified skipped reason if it remains user-visible.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: unset/default heartbeat target resolves to `target === "none"`, returns `reason: "target-none"`, and the heartbeat runner emits that reason in skipped heartbeat events.

## Solution Assessment

Yes. The narrowest maintainable fix is to rename the target-none skip reason to a configuration-oriented value such as `delivery-disabled` and update focused tests/docs, without changing routing, scheduling, or defaults.
