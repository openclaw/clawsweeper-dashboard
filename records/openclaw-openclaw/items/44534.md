## Summary

Keep open: current main and the latest release still expose `reason: "target-none"` for the intentional default heartbeat delivery-disabled path, while both direct fix PRs are closed unmerged. The remaining work is choosing the compatibility-safe diagnostic contract, not gathering more reproduction data.

## What This Changes

The issue asks OpenClaw to make the default disabled heartbeat delivery diagnostic clearer than `target-none` in `openclaw system heartbeat last`, status output, or heartbeat event fields.

## Best Possible Solution

Keep `target-none` as the stable machine reason unless maintainers approve a rename, and add an additive human-readable explanation to heartbeat event/status/CLI surfaces with focused regression coverage.

## Reproduction Assessment

Yes. Current main source and tests show unset or explicit `target: "none"` resolves to `channel: "none"` with `reason: "target-none"`, and `openclaw system heartbeat last` exposes the raw last-heartbeat payload.

## Solution Assessment

Unclear until maintainers choose the diagnostic contract. The safest observed path is an additive hint/status rendering change because it preserves `target-none` for machines while clarifying disabled delivery for humans.
