## Summary

Current main still exposes the misleading heartbeat skip reason. `resolveHeartbeatDeliveryTarget` defaults unset or explicit `target: "none"` to a no-delivery target with `reason: "target-none"`, and the heartbeat runner persists that reason into the `last-heartbeat` event consumed by `openclaw system heartbeat last`. The docs confirm this is intentional no-delivery behavior, but the user-facing reason has not been clarified. The narrow rename PR #46870 is closed unmerged, and no open PR directly referencing this issue was found.

## Best Possible Solution

Keep this issue open and land a narrow heartbeat DX fix. Make the no-delivery-by-configuration path explicit, either by renaming the skipped reason to `delivery-disabled`/`target-config-none` or by adding target metadata and CLI display text that says delivery is disabled because `target=none`. Do not change heartbeat scheduling, routing, prompt execution, or the default no-delivery behavior.
