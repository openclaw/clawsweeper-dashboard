## Summary

Keep open: current main and v2026.6.10 still emit raw `reason: "target-none"` with no explanatory field/status hint for default disabled heartbeat delivery, and the two direct fix PRs are closed unmerged; maintainer choice on the diagnostic contract is still needed.

## What This Changes

The issue requests clearer heartbeat skipped diagnostics when default heartbeat delivery is disabled by `target: "none"`.

## Best Possible Solution

Keep `target-none` as the stable machine reason unless maintainers explicitly approve a rename, and add an additive human-readable explanation to heartbeat event/status/CLI surfaces with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows unset or explicit `target: "none"` resolves to `reason: "target-none"`, the heartbeat runner forwards that reason into skipped events, and the CLI/gateway surfaces expose the raw last-heartbeat event.

## Solution Assessment

Unclear until maintainers choose the diagnostic contract. The safest observed path is an additive hint/status rendering change because it preserves `target-none` for machines while clarifying disabled delivery for humans.
