## Summary

Keep open: current main and v2026.6.8 still expose the confusing `target-none` disabled-delivery path without an operator-facing hint, and both direct fix PRs are closed unmerged rather than viable canonical landing paths.

## What This Changes

The issue asks for heartbeat skipped output under default `target: "none"` delivery to be less misleading than `reason: "target-none"`.

## Best Possible Solution

Keep `target-none` as the stable machine reason and add optional human-readable disabled-delivery context plus CLI/status rendering that avoids the misleading `unknown` channel for skipped no-delivery heartbeats.

## Reproduction Assessment

Yes, from source inspection: current main still resolves default or explicit disabled heartbeat delivery to `reason: "target-none"`, emits that skipped event reason, and formats missing last-heartbeat channels as `unknown`. I did not run a live gateway heartbeat in this read-only review.

## Solution Assessment

Yes for the additive direction, not for a direct reason-string rename without maintainer approval. Preserving `target-none` and adding optional human-readable context is the narrowest maintainable path because it improves operator clarity without changing a machine-readable diagnostic value.
