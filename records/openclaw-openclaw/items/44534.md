## Summary

Keep open. Current main still exposes `target-none` for the default heartbeat no-delivery path, the docs confirm that `target: "none"` is intentional default behavior, and the only linked fix PR is closed unmerged rather than superseding the issue.

## What This Changes

The issue requests clearer heartbeat skipped output when delivery is intentionally disabled by the default `target: "none"` configuration.

## Best Possible Solution

Land a narrow heartbeat DX change in the delivery-target resolver and its tests so intentional `target: "none"` no-delivery is reported as configuration-disabled behavior, without changing scheduling, prompt execution, routing, or the default no-delivery setting.

## Reproduction Assessment

Yes. The current resolver path can be reproduced with unset heartbeat config or `heartbeat.target: "none"`; existing tests already assert the current `target-none` result and the runner forwards that reason into the last heartbeat event.

## Solution Assessment

Yes. The maintainable fix is a small resolver/event-output change, with CLI display help or additive metadata only if reason-string compatibility is a concern; broader heartbeat routing changes are unnecessary.

## Repair Work Prompt

Fix the heartbeat skipped reason shown when heartbeat delivery is intentionally disabled by config. Current main returns `reason: "target-none"` from `resolveHeartbeatDeliveryTarget` when the resolved heartbeat target is `none`, and `runHeartbeatOnce` forwards that into `last-heartbeat`. Make this path explicit as delivery-disabled/config-disabled behavior, preferably by renaming the no-delivery-by-config reason to `delivery-disabled` to match the closed unmerged source PR #46870. Do not change heartbeat scheduling, prompt execution, target resolution for `last` or explicit channels, delivery routing, or the default `target: "none"` behavior. Update focused tests that currently expect `target-none`; add or adjust CLI/docs coverage only if the implementation chooses a human-readable hint or additive metadata instead of a reason rename. Add a user-facing `CHANGELOG.md` fix entry with appropriate non-bot credit, including Thanks @Congregalis and, if using the prior PR direction, Thanks @Br1an67.
