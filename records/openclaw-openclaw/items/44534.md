## Summary

Keep open: current main still reports the intentional default heartbeat no-delivery path as `target-none`, and the only linked fix PR is closed unmerged, so there is still a narrow DX cleanup to land.

## What This Changes

The issue asks for heartbeat skipped output to make the default disabled-delivery configuration clearer than `reason: "target-none"`.

## Best Possible Solution

Land a narrow heartbeat diagnostic change that reports the intentional `target: "none"` path as configuration-disabled delivery while preserving the default no-delivery behavior and existing routing semantics.

## Reproduction Assessment

Yes, from source inspection. The resolver returns `target-none` for unset or explicit `target: "none"`, existing tests assert that result, and the runner forwards the reason into skipped heartbeat events.

## Solution Assessment

Yes. The narrow maintainable fix is in heartbeat target resolution plus last-heartbeat display/tests; if reason-string compatibility matters, additive target metadata and a CLI hint would be safer than changing delivery behavior.
