## Summary

Keep open: current main and the latest release still expose the confusing `target-none` path without an operator-facing explanation, and both linked fix PRs are closed unmerged rather than canonical landing paths.

## What This Changes

The issue asks for heartbeat skip output for default disabled delivery to be less misleading than a bare `target-none` reason.

## Best Possible Solution

Keep `target-none` as the stable machine reason and, if maintainers approve the direction, add an optional human-readable disabled-delivery hint plus status/CLI rendering that avoids the misleading `unknown` channel for skipped no-delivery heartbeats.

## Reproduction Assessment

Yes from source inspection: current main still resolves default or explicit disabled delivery to `reason: "target-none"`, and deep status still substitutes `unknown` when the skipped event has no channel. I did not run a live gateway heartbeat in this read-only review.

## Solution Assessment

Yes for the additive direction, not for a raw rename. Preserving `target-none` and adding an optional human-readable hint/status rendering is the narrowest maintainable path because it improves operator clarity without changing the existing machine reason contract.
