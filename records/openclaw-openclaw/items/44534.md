## Summary

Keep open: current main and the latest release still expose the confusing `target-none` diagnostic for default disabled heartbeat delivery, while both linked fix PRs were closed unmerged. The remaining question is the maintainer-approved compatibility shape, not whether the source behavior exists.

## What This Changes

The issue requests clearer heartbeat diagnostics when default heartbeat delivery is disabled by `target: "none"`.

## Best Possible Solution

Keep the stable machine-readable `target-none` reason unless maintainers explicitly choose a rename, and add a narrow human-readable hint/status rendering path for the disabled-delivery case.

## Reproduction Assessment

Yes, source-level only: current main still resolves unset or explicit heartbeat `target: "none"` to `reason: "target-none"` without an operator-facing hint, and the status formatter has no disabled-delivery wording. I did not run a live heartbeat command in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. The narrowest safer fix appears to be the additive hint/status rendering shape from the closed unmerged fix candidate, because it preserves the machine reason while improving human diagnostics.
