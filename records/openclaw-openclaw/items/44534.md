## Summary

Keep open: current main and the latest release still expose the confusing `target-none` diagnostic for default disabled heartbeat delivery, and both linked fix PRs are closed unmerged. The remaining work is a maintainer-approved diagnostic contract, not missing reproduction evidence.

## What This Changes

Request clearer heartbeat skipped output when the default `target: "none"` intentionally disables delivery.

## Best Possible Solution

Keep `target-none` as the machine reason unless maintainers approve a rename, and add an additive human-readable explanation to heartbeat event/status/CLI surfaces with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main source and tests show unset or explicit `target: "none"` resolves to `channel: "none"` with `reason: "target-none"`, and the gateway exposes that raw last-heartbeat payload.

## Solution Assessment

Unclear until maintainers choose the diagnostic contract. The safest observed path is the closed additive hint/status rendering approach, because it preserves `target-none` for machines while clarifying disabled delivery for humans.
