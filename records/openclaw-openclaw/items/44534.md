## Summary

Keep open: current `main` and latest release still emit the confusing `target-none` diagnostic for default-disabled heartbeat delivery, and the two related fix PRs were closed unmerged.

## What This Changes

The issue asks for heartbeat skip diagnostics to explain that delivery is disabled by configuration when the heartbeat target is the default `none`.

## Best Possible Solution

Keep the stable machine reason unless maintainers explicitly approve a contract rename, and add an operator-facing hint/status explanation for configuration-disabled heartbeat delivery with focused regression tests.

## Reproduction Assessment

Yes, source-level only: current `main` still resolves unset or explicit heartbeat `target: "none"` to `reason: "target-none"`, and status output still has no disabled-delivery wording. I did not run a live gateway command because this review is read-only.

## Solution Assessment

The safest apparent solution is the additive hint/status rendering shape from the closed related PR, not a direct machine-reason rename. Maintainer confirmation is still needed because this touches operator-facing diagnostic/API shape.
