## Summary

Keep open. Current main still replays old first-attempt outbound queue entries during gateway startup recovery with no queued-age or expiry gate, and the focused implementation is still open/conflicting while the TTL policy remains a maintainer decision.

## What This Changes

The issue asks OpenClaw to prevent stale outbound delivery-queue messages from flooding channels after a gateway restart, ideally through a configurable TTL or quarantine policy.

## Best Possible Solution

Land or replace a focused outbound delivery expiry policy that quarantines stale queued replies while preserving recent crash recovery, reconnect drains, recovery budgets, and uncertain-send reconciliation.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount: 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the TTL duration, default, expire action, and config surface need maintainer alignment before the open PR or a replacement can close this.
