## Summary

Keep open: current main and the latest v2026.5.20 release still recover old pending outbound queue entries without an age or expiry gate, while the focused fix PR remains open and the TTL/default/config policy is still tracked separately.

## What This Changes

The issue reports that outbound delivery recovery can replay stale queued channel messages after a gateway restart and asks for a TTL or discard policy.

## Best Possible Solution

Land or replace a backward-compatible outbound delivery expiry policy shared by startup recovery and reconnect drains, preserving recent crash recovery while covering config/default/expire action with focused tests.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but TTL duration, default behavior, expire action, config surface, and reconnect-drain consistency need maintainer alignment before the open candidate or a replacement can close this.
