## Summary

Keep this issue open. Current main still starts outbound delivery recovery on gateway activation and accepts pending first-attempt delivery entries with no stale-age TTL or restart-generation cutoff; the linked implementation candidate in PR #54900 is still open, so the issue should remain open until that or a replacement lands.

## What This Changes

The issue asks OpenClaw to avoid replaying stale queued outbound gateway messages after a restart, for example by dropping old queued entries or adding a gateway delivery TTL setting.

## Best Possible Solution

Add a core outbound recovery freshness policy that skips, fails, or quarantines old never-attempted queued replies during startup recovery while preserving intentional retry recovery for messages that already failed delivery; cover the behavior in delivery-queue recovery tests and update config/docs/changelog only if an operator-facing knob is exposed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enqueue an old delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current tests and policy code show that path is treated as eligible and delivered.

## Solution Assessment

Yes, the narrowest maintainable fix belongs in the shared delivery-queue recovery policy rather than a Telegram-specific workaround. PR #54900 appears to follow that direction, but maintainers should still verify the stale-age default and whether it should be configurable.
