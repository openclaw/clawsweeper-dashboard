## Summary

Keep open. Current main still replays old pending outbound delivery entries on gateway startup without an age, expiry, or restart-generation cutoff, and the focused fix path remains open and unmerged.

## What This Changes

The issue requests preventing stale queued outbound messages from being replayed as fresh channel messages after a gateway restart.

## Best Possible Solution

Land or replace the focused outbound recovery fix so stale never-attempted queued replies are skipped or quarantined while legitimate recent crash recovery remains intact, with any operator-facing TTL/config semantics coordinated through https://github.com/openclaw/openclaw/issues/16555.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is the right place, but the TTL default/action, optional config exposure, and stale predicate after recovery-budget deferral need maintainer alignment before merge.
