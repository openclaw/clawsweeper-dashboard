## Summary

Keep open. Current main and the latest v2026.5.19 release still recover first-attempt outbound delivery queue entries after startup with no queued-age expiry, while the focused fix remains open/conflicting and the broader TTL/config policy is still tracked separately.

## What This Changes

The issue requests preventing stale outbound delivery queue entries from flooding channels after gateway restart, preferably through a configurable delivery TTL or discard/quarantine policy.

## Best Possible Solution

Land or replace a focused outbound delivery expiry policy that prevents stale queue replay while preserving recent crash recovery, reconnect drains, recovery budgets, backoff, and unknown-send reconciliation semantics.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but TTL duration, default behavior, expire action, and config surface need maintainer alignment before the open candidate or a replacement can close this.
