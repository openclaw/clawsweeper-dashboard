## Summary

Keep this issue open. Current main still runs outbound delivery recovery during gateway startup and treats first-attempt queued entries as immediately eligible without a stale-age TTL or restart-generation cutoff. PR #54900 is open as the focused implementation candidate, so the issue should remain open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to avoid replaying stale queued outbound gateway messages after restart, for example by dropping old entries or adding a delivery TTL setting.

## Best Possible Solution

Review, repair, and land PR #54900 or replace it with a narrow shared outbound recovery freshness fix that skips or quarantines old never-attempted queued replies while preserving intentional retry recovery for real transient send failures. Add regression coverage around startup recovery and add docs/config/changelog only for any user-facing knob or shipped behavior change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an old queued delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current policy allows that entry and recovery will deliver it if the send succeeds.

## Solution Assessment

Yes, the right boundary is the shared delivery-queue recovery policy rather than a Telegram-specific workaround. The open PR follows that direction, but it still needs maintainer review or replacement before this issue can close.
