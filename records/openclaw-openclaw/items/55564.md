## Summary

Keep open. Current main and v2026.5.18 still replay old first-attempt outbound delivery queue entries on gateway startup without an age/expiry gate, and the focused open implementation still needs rebase, proof, and TTL policy alignment.

## What This Changes

The issue asks for stale outbound delivery-queue entries to be dropped, quarantined, or controlled by a TTL after gateway restart.

## Best Possible Solution

Land or replace a focused outbound delivery expiry policy that quarantines stale queued replies while preserving recent crash recovery, reconnect drains, recovery budgets, and uncertain-send reconciliation.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount: 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the TTL default, expiry action, config surface, and open PR predicate still need maintainer alignment before this can close.
