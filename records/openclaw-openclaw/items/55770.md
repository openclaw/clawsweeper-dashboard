## Summary

Keep this PR open, but do not merge it as-is. Current main still has the hard-coded exec display truncation, while the branch introduces stale session-state behavior in TUI and leaves WebChat rendering out of sync after setting the new limit.

## What This Changes

The PR adds a session-persisted `/verbose limit <n>` setting and threads that value through gateway/session rows, agent run context, exec tool display rendering, TUI, WebChat, and messaging directive handling.

## Best Possible Solution

Keep the feature direction, but revise the branch so session rows are the single source of truth: TUI should explicitly clear `verboseLimit` when absent, WebChat should refresh or apply the returned session patch immediately after setting it, and the change should add focused tests plus docs for the new command surface.

## Reproduction Assessment

Yes. Set a limit in one TUI session, switch/load a session with no `verboseLimit`, and the PR path keeps the previous limit; in WebChat, run `/verbose limit 300` and the current render state will not reload because the slash-command result has no refresh action.

## Solution Assessment

No. The proposed plumbing is close, but the UI state synchronization is not the narrowest maintainable solution until it clears missing optional values and refreshes session-backed render state consistently.

- **[P2] Refresh sessions after setting the web verbose limit:** `ui/src/ui/chat/slash-command-executor.ts:270`
  - body: `dispatchSlashCommand` only reloads chat/session state when `result.action === "refresh"`, but this new branch patches `verboseLimit` and returns only content. Since `renderChat` reads `activeSession?.verboseLimit`, tool cards keep using the previous limit until some unrelated refresh or navigation occurs.
  - confidence: 0.92
- **[P2] Clear the TUI limit when session info omits it:** `src/tui/tui-session-actions.ts:180-183`
  - body: `sessions.patch` can clear optional fields by returning an entry without `verboseLimit`, but this guard skips both `next.verboseLimit` and `chatLog.setVerboseLimit(undefined)`. After clearing or refreshing a session with no override, TUI tool cards continue using the previous limit.
  - confidence: 0.93
- **[P2] Do not carry a previous limit into loaded sessions:** `src/tui/tui-session-actions.ts:310-311`
  - body: On history load, `record.verboseLimit ?? state.sessionInfo.verboseLimit` keeps the old session's limit whenever the newly loaded session has no override. Because `chatLog.setVerboseLimit` is called immediately after, the new session renders exec cards with another session's truncation setting.
  - confidence: 0.94
