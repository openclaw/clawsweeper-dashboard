Keep open: current main still has the fixed 120-character exec display truncation and the PR remains useful, but the branch has merge-blocking state-refresh, stale TUI state, parser, command-menu, and docs/changelog gaps.

## What This Changes

The PR adds a persisted `/verbose limit <n>` setting and threads `verboseLimit` through session rows, gateway patching, agent run context, TUI, WebChat, and exec tool display formatting.

## Best Possible Solution

Repair the branch so session rows remain the source of truth across WebChat, TUI, and messaging, preserve existing `/verbose full` behavior, add docs/changelog/regression coverage, then let maintainers decide whether to merge the new setting.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: set a verbose limit and the PR code shows WebChat will not refresh its session snapshot, TUI can retain stale limits when entries omit the field, invalid messaging suffixes can become cleaned user text, and native command choices drop `full`.

## Solution Assessment

No. The feature direction is plausible, but this is not the best mergeable solution until refresh, clearing, session-switch, parser, existing-choice, docs/changelog, and regression-test gaps are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep full in verbose command choices:** `src/auto-reply/commands-registry.shared.ts:654`
  - body: The patch changes the native `/verbose` command choices from `["on", "off", "full"]` to `["on", "off", "limit"]`. `/verbose full` is still a supported mode and has regression coverage, so command menus would stop offering the full mode while the handler still accepts it.
  - confidence: 0.91
- **[P2] Refresh chat after setting the web verbose limit:** `ui/src/ui/chat/slash-command-executor.ts:266-267`
  - body: `dispatchSlashCommand` only reloads session state when the result has `action: "refresh"`, but this branch patches `verboseLimit` and returns only content. Since rendering reads `activeSession?.verboseLimit`, WebChat keeps using the old truncation length until an unrelated refresh or navigation occurs.
  - confidence: 0.92
- **[P2] Clear the TUI limit when session info omits it:** `src/tui/tui-session-actions.ts:180-183`
  - body: `sessions.patch` can clear optional fields by returning an entry without `verboseLimit`, but this guard skips both `next.verboseLimit` and `chatLog.setVerboseLimit(undefined)`. After clearing or refreshing an unset session, TUI tool cards continue using the previous limit.
  - confidence: 0.93
- **[P2] Reset the TUI limit when loading another session:** `src/tui/tui-session-actions.ts:310-311`
  - body: On history load, `record.verboseLimit ?? state.sessionInfo.verboseLimit` keeps the old session's limit whenever the newly loaded session has no override. Because that value is immediately applied to `chatLog`, the new session renders exec cards with another session's truncation setting.
  - confidence: 0.94
- **[P2] Consume invalid verbose-limit suffixes before stripping:** `src/auto-reply/reply/directives.ts:53-61`
  - body: The parser only extends `rawVerboseLevel` when the suffix after `limit` is digits. For directive-only messages like `/verbose limit abc` or `/verbose limit -1`, the invalid suffix is left in `cleaned`, so the directive cleanup path can send the suffix as normal user text instead of returning the invalid-limit error.
  - confidence: 0.86
- **[P3] Document the new verbose limit command:** `src/tui/commands.ts:150`
  - body: This adds a new user-facing command, but the PR file list has no `CHANGELOG.md` or docs update. The published slash-command docs still only describe `/verbose on|off|full`, so the new setting would ship without the required user-facing documentation and release note.
  - confidence: 0.88
