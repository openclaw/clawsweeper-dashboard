Full review comments:

- **[P1] Preserve exec detail options while adding the limit:** `src/agents/tool-display-common.ts:400`
  - body: Current main passes `detailMode` through `resolveToolDisplay` into `resolveExecDetail` so explain/raw exec summaries differ. This patch changes that call path to pass only a numeric length, so a rebase must extend the existing options object instead of replacing it or exec tool cards will lose the current detail-mode behavior.
  - confidence: 0.9
- **[P2] Refresh WebChat after setting the verbose limit:** `ui/src/ui/chat/slash-command-executor.ts:267-272`
  - body: The new Web `/verbose limit` branch patches `sessions.patch` but returns only content. `dispatchSlashCommand` refreshes session rows only when `result.action === "refresh"`, so `activeSession?.verboseLimit` stays stale and tool cards keep the old/default limit until a manual refresh.
  - confidence: 0.91
- **[P2] Clear TUI limits when session info omits them:** `src/tui/tui-session-actions.ts:177-181`
  - body: `sessions.patch` supports clearing `verboseLimit` with null, but this guard only updates ChatLog when the incoming entry has a number. A cleared or unset session leaves the old ChatLog limit applied to future tool cards.
  - confidence: 0.9
- **[P2] Avoid leaking limits across TUI session loads:** `src/tui/tui-session-actions.ts:310-311`
  - body: When loading another session, `record.verboseLimit ?? state.sessionInfo.verboseLimit` keeps the previous session's limit whenever the new session is unset. That makes the setting leak across sessions instead of falling back to the default.
  - confidence: 0.9
- **[P2] Consume invalid verbose limit arguments before validating:** `src/auto-reply/reply/directives.ts:52-59`
  - body: The parser only consumes digits after `limit`, so `/verbose limit abc` is parsed as a bare `limit` plus leftover text instead of an invalid limit argument. The validation branch then cannot reliably reject the bad value before the rest of the message is handled.
  - confidence: 0.83
