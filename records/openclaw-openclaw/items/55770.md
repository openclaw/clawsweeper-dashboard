  - body: Current main routes both `exec` and `bash` through `resolveExecDetail(args, { detailMode: params.toolDetailMode })`, but this branch's edited call only handles `exec` and replaces the options object with a numeric limit. A rebase that lands this shape would drop bash summaries and the documented raw/explain detail mode, so thread the limit through the existing options object instead.
  - confidence: 0.86
- **[P2] Return refresh after setting verbose limit:** `ui/src/ui/chat/slash-command-executor.ts:269-270`
  - body: `dispatchSlashCommand` reloads chat/session state only when the slash result has `action: "refresh"`, but this branch patches `verboseLimit` and returns only content. Because chat rendering reads `activeSession?.verboseLimit`, WebChat keeps using the old truncation length until some later refresh or navigation.
  - confidence: 0.9
- **[P2] Clear TUI verboseLimit when session info omits it:** `src/tui/tui-session-actions.ts:180-183`
  - body: This branch updates `chatLog` only when `entry?.verboseLimit !== undefined`, so a session entry without an override never clears a previously applied limit. After a clear/null patch or refresh to an unset session, future tool cards still render with the stale truncation length.
  - confidence: 0.9
- **[P2] Reset TUI verboseLimit from loaded session data:** `src/tui/tui-session-actions.ts:310-311`
  - body: When loading a different session, `record.verboseLimit ?? state.sessionInfo.verboseLimit` preserves the prior session's limit whenever the new session has no override. That leaks a per-session display setting across sessions and makes TUI tool cards disagree with persisted state.
  - confidence: 0.9
