## Summary

Keep this PR open. Current main still has the hard-coded exec command display truncation and no `verboseLimit` session field, so the feature is not obsolete; however, the submitted patch has blocking state-synchronization bugs in TUI and WebChat that can leave rendered exec tool cards using stale limits after session changes or `/verbose limit` updates.

## What This Changes

The PR adds a session-persisted `/verbose limit <n>` setting and threads `verboseLimit` through gateway session rows, agent run context, exec display rendering, TUI, WebChat, and messaging directive handling.

## Best Possible Solution

Keep the feature direction, but revise the branch so session rows remain the source of truth: TUI must clear missing `verboseLimit` values, WebChat must refresh or locally mirror the session update after `/verbose limit`, and the command should ship with focused parser/session/UI tests plus docs and changelog coverage.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: set a limit in one TUI session, switch or load a session with no `verboseLimit`, and the PR path keeps the old limit; in WebChat, run `/verbose limit 300` and the active session snapshot does not reload because the result lacks `action: "refresh"`.

## Solution Assessment

No. The proposed plumbing is close, but it is not the best mergeable solution until optional session values are cleared consistently and the WebChat update path refreshes the state that rendering reads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Refresh sessions after setting the web verbose limit:** `ui/src/ui/chat/slash-command-executor.ts:270`
  - body: `dispatchSlashCommand` reloads chat/session state only when `result.action === "refresh"`, but this branch patches `verboseLimit` and returns only content. Since `renderChat` reads `activeSession?.verboseLimit`, exec tool cards keep using the previous limit until an unrelated refresh or navigation occurs.
  - confidence: 0.92
- **[P2] Clear the TUI limit when session info omits it:** `src/tui/tui-session-actions.ts:180-183`
  - body: `sessions.patch` can clear optional fields by returning an entry without `verboseLimit`, but this guard skips both `next.verboseLimit` and `chatLog.setVerboseLimit(undefined)`. After clearing or refreshing a session with no override, TUI tool cards continue using the previous limit.
  - confidence: 0.93
- **[P2] Do not carry a previous limit into loaded sessions:** `src/tui/tui-session-actions.ts:310-311`
  - body: On history load, `record.verboseLimit ?? state.sessionInfo.verboseLimit` keeps the old session's limit whenever the newly loaded session has no override. Because `chatLog.setVerboseLimit` is called immediately after, the new session renders exec cards with another session's truncation setting.
  - confidence: 0.94
