## Summary

Keep open: current main still has a source-backed Discord startup/status probe path that can hit `/users/@me` with a short fixed timeout, and the WSL2 stabilization failure has not been live-verified or clearly superseded by an existing fix.

## What This Changes

Investigate a WSL2 regression where Discord REST startup probes time out on OpenClaw 2026.4.29 and later while 2026.4.23 works.

## Best Possible Solution

Keep the fix inside the Discord plugin by making startup probes tolerant of transient REST timeouts, reusing token-derived identity where possible, and improving diagnostics without adding a new config knob unless the owner decides it is needed.

## Reproduction Assessment

Yes from source for the named timeout path: current main still calls `probeDiscord` during Discord account startup, and that probe fetches `/users/@me` through `fetchWithTimeout` with a 2.5s budget. No live WSL2 reproduction was established, so the exact stabilization failure remains only partially reproduced.

## Solution Assessment

No close-out solution exists on current main: the monitor identity path avoids `/users/@me` for parseable tokens, but the earlier startup probe still does not. A safer fix is a narrow Discord-owned change with regression tests around hung/transient probe requests before considering a public REST timeout option.
