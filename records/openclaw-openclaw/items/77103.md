## Summary

Keep open: current main still exercises the reported Discord `/users/@me` REST probe with a short fixed timeout, and the existing token-derived identity/application-id mitigations do not fully remove that startup/status probe path.

## What This Changes

This issue reports a WSL2 Discord regression after upgrading from 2026.4.23 to 2026.4.29 or 2026.5.2, with Node fetch timing out on Discord REST while curl succeeds.

## Best Possible Solution

Keep the fix inside the Discord plugin by making startup/status probes tolerant of transient REST timeouts, preferring token/config-derived identity where possible, and adding diagnostics plus regression tests before considering a public timeout knob.

## Reproduction Assessment

Partially: current main source still has the named `/users/@me` timeout path with a 2.5s startup probe, but I did not live-reproduce the full WSL2 messaging failure.

## Solution Assessment

No current-main close-out is available. The narrow maintainable direction is Discord-plugin hardening around transient probe timeouts and clearer diagnostics, not a core change or immediate new config surface.
