## Summary

Keep open: current main still has the Pi `apply_patch` gate tied to the literal `openai`/`openai-codex` provider ids, while qa-lab's `mock-openai` lane is an OpenAI Responses-compatible provider with a `mock-openai/gpt-5.5` model id. The related harness PR at https://github.com/openclaw/openclaw/pull/80323 records this as a known-broken fixture, but it is open and does not close or fix this issue.

## What This Changes

The issue asks OpenClaw to align Pi and Codex runtime exposure of `apply_patch` for the mock OpenAI runtime parity fixture, or explicitly document expected drift.

## Best Possible Solution

Teach the Pi tool-construction path to recognize the qa-lab OpenAI-compatible mock provider for `apply_patch`, while preserving the existing disable, allowModels, workspace-only, sandbox, shell-tool, and policy gates.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not by executing the private QA fixture in this read-only review, but source inspection gives a clear reproduction path: Pi sees `modelProvider: "mock-openai"`, the provider-id gate rejects it, and `apply_patch` is omitted.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable fix is to align the QA OpenAI-compatible Pi lane with the existing `apply_patch` contract rather than broadening the tool to unrelated providers or documenting accidental drift.
