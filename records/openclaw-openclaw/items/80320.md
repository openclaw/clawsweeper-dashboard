Keep open: the reported Pi/Codex `apply_patch` parity gap is still plausible on current `main`, the related parity-harness PR is still open, and there is no merged narrower fix or shipped implementation to close against.

## What This Changes

The issue asks OpenClaw to make the Pi runtime parity QA surface expose an `apply_patch` capability comparable to Codex, or explicitly mark that drift as intentional.

## Best Possible Solution

Align the QA OpenAI-compatible Pi tool profile with the `apply_patch` contract while preserving production disable, allowModels, workspace-only, sandbox, and policy gates; if Pi intentionally should not expose it, document that as expected drift in the parity fixture.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as an executed current-main repro in this read-only pass; source inspection shows `mock-openai` cannot pass the current Pi `apply_patch` provider gate, and the issue supplies concrete QA artifact output for the drift.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the intended tool-profile contract, but the narrow maintainable fix would be to recognize the mock OpenAI-compatible QA lane without broadening `apply_patch` to unrelated providers.
