## Summary

Keep open: current main still has the Pi `apply_patch` availability gate tied to raw provider ids, while the reported QA mock OpenAI lane uses `mock-openai` with an OpenAI Responses-compatible provider config. The related runtime-parity PR and umbrella issues are still open, so this is not implemented or superseded on main.

## What This Changes

The issue asks OpenClaw to resolve a Codex-vs-Pi QA parity gap where Pi omits `apply_patch` for the mock OpenAI runtime lane while Codex exposes it.

## Best Possible Solution

Align Pi tool construction for the OpenAI-compatible mock QA lane so `apply_patch` is exposed under the same safety gates, or explicitly document and encode the drift if maintainers decide Pi should not expose that tool there.

## Reproduction Assessment

Yes, as a source-level reproduction: `createOpenClawCodingTools` with `modelProvider: "mock-openai"` cannot pass the current `openai`/`openai-codex` provider-id gate, while the QA mock provider is configured as `openai-responses`. I did not execute the private runtime-pair fixture in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to use the prepared provider/model API facts to treat the mock OpenAI QA lane as OpenAI-compatible for `apply_patch`, while preserving the existing disable, model allowlist, workspace-only, sandbox, and policy gates.
