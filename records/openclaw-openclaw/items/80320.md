## Summary

Keep open: current main still gates Pi `apply_patch` creation on exact `openai`/`openai-codex` provider IDs, while the linked same-author harness PR only preserves this as known-broken evidence and does not merge a fix. The older open provider-gating report at https://github.com/openclaw/openclaw/issues/45269 also covers the broader trust-boundary question, so this should not be swept closed or auto-repaired without maintainer direction.

## What This Changes

The issue reports that the Phase 2 runtime parity apply-patch fixture sees Codex expose `apply_patch` while Pi omits it for the `mock-openai` lane.

## Best Possible Solution

Define the trusted provider/capability contract for OpenAI-compatible QA and routed providers, then update Pi tool construction or the fixture drift marker without broadening `apply_patch` to untrusted providers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: construct Pi tools with `modelProvider: "mock-openai"` and the exact provider gate prevents `apply_patch` from being created, while `openai`/`openai-codex` pass; the full runtime-pair fixture was not executed because that fixture is not on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer review chooses the trust boundary: a QA-only mock-provider allowance, a documented provider capability, or explicit expected drift are safer than broad provider-name matching.
