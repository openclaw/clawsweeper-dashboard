## Summary

Keep open: current main and v2026.5.28 reduce the blast radius by defaulting QMD to BM25 and falling back to builtin memory after QMD failures, but they do not specifically detect or document the reported Alder Lake-N / 0xC0000005 native-Windows semantic QMD crash path.

## What This Changes

The issue asks OpenClaw to handle qmd/node-llama-cpp Windows access violations on Alder Lake-N so semantic QMD memory avoids hard crashes or gives a clear workaround.

## Best Possible Solution

Keep the issue open until maintainers choose a Windows/Alder Lake-N policy: detect or disable QMD semantic modes with a clear diagnostic, document the source-build/WSL2 workaround, or move binary handling upstream with QMD/node-llama-cpp.

## Reproduction Assessment

Unclear for live reproduction: the issue gives a specific hardware path, and source confirms QMD `query`/`vsearch` still enter semantic modes, but I did not have native Windows Alder Lake-N hardware to reproduce 0xC0000005 on current main.

## Solution Assessment

No: the current shipped defaults and fallback are good mitigations, but they are not the complete fix for users who intentionally enable semantic QMD on affected Windows CPUs.
