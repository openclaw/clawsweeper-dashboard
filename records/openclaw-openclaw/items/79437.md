## Summary

Keep open. ## What This Changes

The issue reports a Windows access-violation crash when OpenClaw's qmd LLM path loads node-llama-cpp GGUF models on Intel Alder Lake-N hardware.

## Best Possible Solution

Keep this open for maintainer-owned product/dependency handling: prove and route an upstream QMD/node-llama-cpp binary fix, add hardware-aware graceful fallback, or document the Windows E-core limitation with source-build or service-provider workarounds.

## Reproduction Assessment

Partly. Source shows semantic QMD and local GGUF paths still execute external QMD/node-llama-cpp model loading, but this Linux checkout cannot reproduce the Windows Alder Lake-N access violation itself.

## Solution Assessment

No. The current BM25-only default and docs are useful mitigations, not a complete fix for users who intentionally enable QMD semantic search on affected Windows hardware.
