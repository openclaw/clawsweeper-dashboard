## Summary

Keep open. Current main and v2026.6.8 mitigate the default QMD path by using BM25-only `search`, but configured semantic QMD and local GGUF paths still rely on native QMD/node-llama-cpp model loading, with no Alder Lake-N detection, safe binary selection, upstream dependency bump, or live Windows proof showing the crash is fixed.

## What This Changes

The issue reports a Windows access-violation crash in QMD semantic/native GGUF model loading on Intel Alder Lake-N hardware and asks for detection, fallback, safer binaries, or documentation.

## Best Possible Solution

Keep the BM25 default mitigation, then make a maintainer-backed dependency/product decision for semantic QMD on affected Windows CPUs: prove an upstream safe binary, add a hardware-aware graceful fallback, or document a source-build/remote-provider workaround with live hardware validation.

## Reproduction Assessment

Partly. Source shows the semantic QMD and local GGUF paths still execute native QMD/node-llama-cpp model loading, but this review did not run on Windows Alder Lake-N hardware to reproduce the access violation itself.

## Solution Assessment

No. The current BM25-only default is a useful mitigation, but it is not the complete fix for users who intentionally enable QMD `query` or `vsearch` on the affected Windows CPUs.
