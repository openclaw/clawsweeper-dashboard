## Summary

Keep open: the PR correctly reuses the Feishu text/card fallback helper for direct non-thread image/file sends, but it misses the production dispatcher path that computes the top-level fallback allowance for normal quoted group replies. That leaves one reported media-loss path still broken while text/card would recover.

## What This Changes

The PR exports Feishu reply fallback helpers, routes image/file media sends through them, adds an optional media fallback flag, and adds media reply fallback tests.

## Best Possible Solution

Forward the dispatcher allowTopLevelReplyFallback decision into sendMediaFeishu, keep native topic replies strict, and add regression coverage for media in the allowed fallback path before merge.

## Reproduction Assessment

Yes, source-level. Current main shows text/card using the withdrawn-target fallback while media reply senders assert the direct reply result, and the PR tests model the same Feishu 230011/231003 shapes without live Feishu execution.

## Solution Assessment

No. Reusing sendReplyOrFallbackDirect is the right fix shape, but the PR is incomplete until the production dispatcher forwards allowTopLevelReplyFallback into media sends and tests that allowed-thread case.
