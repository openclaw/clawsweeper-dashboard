## Summary

Keep this open. Current main and the latest release still lack opt-in, domain-scoped agent attestation for arbitrary outbound HTTP requests, while the existing comments and labels keep it as a product/security design tracker rather than a cleanup close candidate.

## What This Changes

Request to add opt-in static or signed OpenClaw agent attestation headers for outbound external HTTP requests, optionally scoped by domain or skill metadata.

## Best Possible Solution

Define the trust model and outbound path scope first, then implement an opt-in, domain-scoped attestation config/API with privacy-preserving defaults, docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested attestation config, outbound header injection, and skill metadata surfaces rather than showing a broken existing contract.

## Solution Assessment

Unclear until maintainers settle the product/security design. Static headers, signed JWTs, and per-skill metadata are plausible directions, but a local header patch is not the best solution until the trust model and outbound scope are explicit.

## Security Review

Status: needs_attention

Summary: The request is identity and attestation infrastructure, so it needs explicit security design before implementation.

Concerns:

- **[medium] Define attestation trust and replay controls first:**
  - body: A signed attestation header needs a clear issuer, key distribution model, audience binding, expiry/replay window, and rotation story before OpenClaw can safely emit verifiable identity claims.
  - confidence: 0.9
- **[medium] Prevent cross-service identity leakage:**
  - body: Static or signed headers could leak stable agent or user identifiers to unintended hosts unless config and runtime enforce strict opt-in domain scoping, redirect handling, and non-leakage across fallback paths.
  - confidence: 0.88
