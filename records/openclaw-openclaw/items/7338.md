## Summary

Keep open: current main and v2026.6.8 still lack a general opt-in, domain-scoped agent-origin attestation surface for arbitrary outbound HTTP requests. Existing provider attribution and static provider headers are narrower transport attribution mechanisms, not a signed or allowlisted agent-origin trust contract.

## What This Changes

The issue requests opt-in agent-origin attestation headers or signed tokens for outbound OpenClaw HTTP requests, scoped by domain or skill metadata.

## Best Possible Solution

Keep this open for maintainer security/product design, then implement an opt-in, domain-scoped attestation contract with privacy-preserving defaults, schema/docs, and tests for allowlisted injection and non-leakage.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug report. Source inspection gives high confidence that the requested attestation surface is absent, but there is no failing existing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the trust model. A signed, opt-in, domain-scoped design is directionally safer than static global headers, but existing provider attribution and static provider headers are not the best complete solution for arbitrary external requests.
