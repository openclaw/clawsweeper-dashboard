## Summary

Keep open: current main and v2026.6.5 still do not provide a general opt-in, domain-scoped agent-origin attestation contract for outbound HTTP requests, and the remaining work is security/product design rather than conservative cleanup.

## What This Changes

The issue requests optional OpenClaw agent attestation headers or signed tokens for external HTTP requests, with domain scoping and possible per-skill configuration.

## Best Possible Solution

Keep this as the canonical product/security design tracker, define trust and outbound scope first, then implement a privacy-preserving opt-in attestation contract through the right config/API/plugin seam with docs and non-leakage tests.

## Reproduction Assessment

Not applicable: this is a feature/security design request rather than broken existing behavior. Source, docs, release-tag, and related-item checks show the requested general attestation headers, config, and skill metadata surfaces do not exist today.

## Solution Assessment

Unclear until maintainers settle the product/security design. Static headers, signed JWTs, and per-skill metadata are plausible inputs, but a local header patch is not the best solution until trust, privacy, redirects, and outbound scope are explicit.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it would send identity or proof material to external services and needs a trust/privacy design before implementation.

Concerns:

- **[medium] Define attestation trust and replay controls:**
  - body: A signed attestation header needs an issuer, signing authority, public-key distribution model, audience binding, expiry/replay window, and rotation story before OpenClaw can safely emit verifiable identity claims.
  - confidence: 0.92

  - body: Static or signed headers could leak stable agent, user, channel, session, or skill identifiers to unintended hosts unless config and runtime enforce opt-in domain scoping, redirect handling, and non-leakage across fallback paths.
