## Summary

Keep open. Current main still lacks a general, opt-in, domain-scoped agent attestation mechanism for outbound external HTTP requests, and the existing provider attribution path is intentionally narrower than the requested agent-origin attestation surface. Prior discussion also treats this as product/security design work rather than conservative cleanup.

## What This Changes

The issue requests opt-in OpenClaw agent attestation headers or signed attestations for outbound external HTTP requests made by agents, skills, or tools.

## Best Possible Solution

Define the identity and trust model first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, signing or explicitly limited non-cryptographic semantics, redirect/fallback protections, docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence current-main verification path: inspect web_fetch headers, runtime config/schema, skill metadata, docs, and provider-attribution behavior to confirm the requested general attestation surface is absent.

## Solution Assessment

Unclear until maintainers settle the design. Static headers alone are not the best solution for agent-only services; the maintainable path needs scoped consent, cryptographic or explicitly limited semantics, redirect protections, and non-leakage tests.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how services obtain verification keys, how replay is prevented, and what issuer/audience claims mean for OpenClaw agents.
  - confidence: 0.93
- **[medium] Prevent cross-domain identity leakage:** `src/agents/tools/web-fetch.ts:431`
  - body: Any attestation must be opt-in and allowlisted so agent ids, versions, skill context, or signed tokens are not attached to arbitrary external requests, redirects, custom proxies, browser traffic, or fallback providers.
  - confidence: 0.91
