## Summary

Keep open: current main and v2026.6.5 still lack a general opt-in, domain-scoped OpenClaw agent attestation contract for outbound HTTP requests; existing provider attribution is narrower vendor/provider metadata, and the related LLM-header proposal was closed unmerged as a duplicate of this issue.

## What This Changes

The issue requests opt-in OpenClaw agent attestation headers or signed tokens for outbound HTTP requests so external services can verify agent-origin traffic.

## Best Possible Solution

Keep this as the canonical product/security design tracker, define trust and outbound scope first, then implement a privacy-preserving opt-in attestation contract through the right config, API, or plugin seam with docs and non-leakage tests.

## Reproduction Assessment

Not applicable: this is a feature/security design request rather than broken existing behavior. Source, docs, release-tag, and related-item checks show the requested general attestation headers, config, and skill metadata surfaces do not exist today.

## Solution Assessment

Unclear until maintainers settle the product/security design. Static headers, signed JWTs, and per-skill metadata are plausible inputs, but a local header patch is not the best solution until trust, privacy, redirects, and outbound scope are explicit.
