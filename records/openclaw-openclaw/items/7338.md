## Summary

Keep open. Current main still lacks the requested general agent attestation surface, and the existing maintainer-authored review keeps this as a product/security design tracker rather than a stale cleanup item.

## What This Changes

Request opt-in, domain-allowlisted static or cryptographic OpenClaw agent attestation headers for external HTTP requests and skill-owned domains.

## Best Possible Solution

Define the trust model and covered outbound paths first, then add an opt-in domain-scoped config/API surface with docs and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable. This is a feature/security design request, and source inspection confirms current main does not expose the requested web_fetch, config, or skill-metadata attestation surfaces.

## Solution Assessment

Unclear. The requested capability is plausible, but the best solution needs design approval for signing authority, replay windows, audience/domain allowlists, and covered outbound paths before implementation.

## Security Review

Status: needs_attention

Summary: The request changes outbound identity and privacy semantics, so maintainers should review the trust model before any implementation.

Concerns:

- **[medium] Define signing and replay semantics first:**
  - body: Cryptographic attestation needs a clear issuer, key distribution model, audience binding, expiry, and replay policy before OpenClaw can safely send verifiable identity headers.
  - confidence: 0.9
- **[medium] Prevent cross-service metadata leakage:**
  - body: Any static or signed agent headers must be opt-in and domain-scoped so agent ids, versions, or skill context are not leaked to unrelated external services.
  - confidence: 0.88
