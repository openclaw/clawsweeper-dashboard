## Summary

Keep open. Current main and the latest release still lack a general opt-in, domain-scoped agent-origin attestation surface for arbitrary external HTTP requests; existing provider attribution and provider custom headers are narrower than the requested trust, privacy, signing, and outbound-scope design.

## What This Changes

The issue requests optional OpenClaw agent attestation headers or signed tokens for outbound HTTP requests, configurable globally or per skill/domain.

## Best Possible Solution

Keep this open for maintainer security/product design, then implement an opt-in, domain-scoped attestation contract with privacy-preserving defaults, schema/docs, and tests for injection and non-leakage.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug report. Source inspection gives high confidence that the requested attestation surface is absent, but there is no failing existing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the trust model. A signed, opt-in, domain-scoped design is directionally safer than static global headers, but existing provider attribution and static provider headers are not the best complete solution for arbitrary external requests.
