## Summary

Keep this open: current main still lacks the requested opt-in, domain-scoped agent attestation surface, and the related provider/header work is narrower than arbitrary external-request attestation.

## What This Changes

The issue requests optional static or cryptographic OpenClaw agent attestation headers for outbound HTTP requests, with user/skill-scoped configuration and privacy controls.

## Best Possible Solution

Define the security model and outbound scope first, then add a generic opt-in config/API contract with domain allowlisting, privacy-preserving defaults, docs, and non-leakage tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new identity/security capability request. Source and docs inspection gives high confidence that current main does not implement the requested general attestation surface.

## Solution Assessment

Unclear until maintainer design approval. A narrow header injection patch is not the best fix yet because trust, privacy, signing, verifier semantics, and outbound-path scope need to be settled first.

Status: needs_attention

Concerns:

  - confidence: 0.9
