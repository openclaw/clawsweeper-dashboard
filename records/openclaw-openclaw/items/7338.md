## Summary

Keep open: current main and v2026.6.10 still only send ordinary web_fetch headers and provider-specific attribution, not general opt-in, domain-scoped agent-origin attestation; this remains a security/product design tracker rather than a cleanup-closeable item.

## What This Changes

Request an opt-in, domain-scoped agent attestation mechanism for outgoing HTTP requests, including possible static or signed headers and per-skill metadata/config.

## Best Possible Solution

Keep this open for maintainer security/product design, then implement an opt-in, domain-scoped attestation contract with privacy-preserving defaults, schema/docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main and v2026.6.10 lack the requested general outbound attestation surface.

## Solution Assessment

Unclear until maintainers choose the trust model. A signed, opt-in, domain-scoped design is directionally safer than static global headers, but existing provider attribution is not the complete solution for arbitrary external requests.
