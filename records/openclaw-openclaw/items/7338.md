## Summary

Keep open: current main and the latest release still lack opt-in, domain-scoped agent-origin attestation for arbitrary outbound HTTP requests. Existing provider attribution, plugin trust hooks, and sender identity context are adjacent but narrower and do not solve the requested static or signed attestation header surface.

## What This Changes

The issue requests optional static or signed OpenClaw agent attestation headers for outbound HTTP requests, configurable by domain and skill metadata.

## Best Possible Solution

Define the trust model and outbound path scope first, then design an opt-in, domain-scoped plugin/config/API surface with privacy-preserving defaults and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable as a feature/security design request; there is no broken existing behavior to reproduce. Source inspection on current main verifies the requested attestation capability is absent rather than failing intermittently.

## Solution Assessment

Unclear until maintainers choose the trust model and scope. The requested direction is plausible, but the best implementation should avoid one-off core headers and instead settle signing, audience, domain allowlisting, and plugin/config ownership first.
