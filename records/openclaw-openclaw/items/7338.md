## Summary

Keep open. Current main and v2026.6.8 still lack opt-in, domain-scoped agent-origin attestation for arbitrary external HTTP requests; existing provider attribution, provider custom headers, policy attestation, and plugin trust hooks are adjacent but narrower than the requested security/product surface.

## What This Changes

Requests optional static or cryptographic OpenClaw agent-attestation headers for outbound external HTTP requests, with domain allowlisting and possible skill metadata/config opt-in.

## Best Possible Solution

Keep this open for maintainer security/product design, then implement an opt-in, domain-scoped attestation contract with privacy-preserving defaults, schema/docs, and tests for injection and non-leakage.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug report. Source inspection gives high confidence that the requested attestation surface is absent, but there is no failing existing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the trust model. A signed, opt-in, domain-scoped design is directionally safer than static global headers, but current provider attribution and static provider headers are not a complete solution for arbitrary external requests.
