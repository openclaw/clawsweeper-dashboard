## Summary

Keep this open. Current main still lacks a general opt-in, domain-scoped agent attestation surface for external HTTP requests, and the existing labels/discussion mark this as a product and security design decision rather than a cleanup close candidate.

## What This Changes

The issue requests optional OpenClaw agent attestation headers, preferably signed and domain-scoped, for outbound HTTP requests made by agents or skills.

## Best Possible Solution

Define the trust model and outbound scope first, then implement an opt-in, domain-scoped attestation contract with privacy-preserving defaults, schema/docs, and tests for injection and non-leakage.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source inspection confirms the requested attestation surface is absent on current main, but there is no failing existing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the trust model. A signed, opt-in, domain-scoped design is directionally safer than static global headers, but the exact implementation should wait for security/product approval.
