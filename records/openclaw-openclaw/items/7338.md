## Summary

Keep this open: current main still lacks a general opt-in, domain-scoped agent attestation surface, and the existing discussion treats the request as a product/security design tracker rather than cleanup-closeable work.

## What This Changes

The issue requests optional OpenClaw agent attestation headers or signed JWTs on outbound HTTP requests, scoped by user or skill configuration and domain allowlists.

## Best Possible Solution

Define the security model and outbound scope first, then add a generic opt-in config/API contract with domain allowlisting, privacy-preserving defaults, docs, and non-leakage tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new identity/security capability request. Source and docs inspection gives high confidence that current main does not implement the requested general attestation surface.

## Solution Assessment

Unclear until maintainer design approval. A narrow header injection patch is not the best fix yet because trust, privacy, signing, verifier semantics, and outbound-path scope need to be settled first.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive and needs explicit trust-model and privacy review before implementation.

Concerns:

- **[high] Define verifier trust and signing model:**
  - body: A signed attestation header would need clear issuer, key distribution, replay protection, audience binding, expiration, and rotation semantics before services can safely rely on it.
  - confidence: 0.9
- **[medium] Prevent stable identity leakage:**
  - body: Agent ids, skill ids, workspace identifiers, or session-derived values in outbound headers could enable cross-service tracking unless domain allowlisting, minimal claims, and user opt-in are enforced.
