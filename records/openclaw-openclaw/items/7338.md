## Summary

Keep this open. Current main still lacks opt-in, domain-scoped agent attestation for arbitrary external HTTP requests, and the existing discussion/labels point to a product and security design decision rather than a stale or already-implemented cleanup item.

## What This Changes

Request opt-in static or signed OpenClaw agent-attestation headers for outbound HTTP requests, with user/domain and skill-scoped configuration.

## Best Possible Solution

Define the trust model and outbound path scope first, then add an opt-in, domain-scoped config/API surface with privacy-preserving defaults, docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection does confirm current main lacks the requested attestation config, outbound header injection, and skill metadata surfaces.

## Solution Assessment

Unclear until maintainers settle the product/security design. A local header patch is not the best path yet because the signing authority, audience/domain scope, identifier privacy, and exact outbound request paths need to be defined first.

## Security Review

Status: needs_attention

Summary: The requested capability is identity and attestation infrastructure, so implementation needs explicit security design before code changes.

Concerns:

- **[medium] Define attestation trust and replay controls first:**
  - body: A signed attestation header would need a clear issuer, key distribution model, audience binding, expiry/replay window, and rotation story before OpenClaw can safely emit verifiable identity claims.
  - confidence: 0.9
- **[medium] Prevent cross-service identity leakage:**
  - body: Static or signed headers could leak stable agent or user identifiers to unintended hosts unless the config and runtime enforce strict opt-in domain scoping and non-leakage across redirects/fallbacks.
  - confidence: 0.88
