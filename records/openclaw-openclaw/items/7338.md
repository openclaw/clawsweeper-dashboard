## Summary

Keep open. Current main still lacks a general, opt-in, domain-scoped agent attestation mechanism for outbound external HTTP requests, and the existing provider attribution path is narrower model-provider transport attribution; prior maintainer-facing review also treats this as product/security design work.

## What This Changes

Request to add opt-in OpenClaw agent attestation headers or signed tokens for outbound HTTP requests, scoped by user config and possible skill metadata.

## Best Possible Solution

Define the identity, trust, and outbound-scope model first, then implement a privacy-preserving opt-in attestation surface with strict domain allowlisting, redirect protections, docs, and non-leakage tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence current-main verification path: inspect web_fetch headers, runtime config/schema, skill metadata, docs, and provider-attribution behavior to confirm the requested general attestation surface is absent.

## Solution Assessment

Unclear until maintainers settle the design. Static headers alone are not a sufficient solution for agent-only services; the maintainable path needs scoped consent, cryptographic or explicitly limited semantics, replay protection, and non-leakage guarantees.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

  - body: Any attestation must be opt-in and allowlisted so agent ids, versions, skill context, or signed tokens are not attached to arbitrary external requests, redirects, custom proxies, browser traffic, or fallback providers.
