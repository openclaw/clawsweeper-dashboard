## Security Review

Status: needs_attention

Summary: The requested feature needs security architecture review before implementation because it creates an authentication-like claim and new sensitive outbound metadata.

Concerns:

- **[high] Define what the attestation actually proves:**
  - body: A self-issued header or JWT cannot distinguish an autonomous agent from a human or program controlling the same OpenClaw installation, so verifier-facing semantics must not overstate the assurance.
  - confidence: 0.96
- **[high] Design signing-key and replay lifecycle:**
  - body: Cryptographic attestation requires explicit key generation, storage, rotation, revocation, recovery, audience binding, expiry, and replay handling before it can be treated as a security boundary.
  - confidence: 0.95
- **[medium] Prevent cross-origin identity leakage:** `src/agents/tools/web-fetch.ts:422`
  - body: Domain matching, redirects, proxies, plugin tools, and independently owned outbound clients need a clear policy so identifiers or signatures cannot escape their intended audience.
  - confidence: 0.93

## Risks / Open Questions

- Static identity headers are copyable and therefore cannot prove that a request originated from an autonomous OpenClaw agent rather than a human or another HTTP client.
- Cryptographic assertions require an approved signing authority, protected key custody, rotation and recovery, verifier distribution, revocation, replay limits, and audience binding.
- Stable identifiers or reusable signing keys could enable cross-service correlation or leak through redirects, proxies, browsers, plugin tools, and independently owned service clients.
- The proposed scope spans web_fetch, model-provider calls, plugin-owned tools, MCP, browser automation, and other transports; a partial global mechanism could create inconsistent or misleading trust semantics.
