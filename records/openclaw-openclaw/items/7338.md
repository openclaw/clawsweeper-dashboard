## Summary

Keep this open: current main and v2026.5.12 still lack a general opt-in, domain-scoped agent attestation surface for arbitrary external HTTP requests, and the adjacent provider/header work is narrower than agent-origin attestation. The live labels and prior review also keep this as a product/security design tracker rather than a cleanup-close candidate.

## What This Changes

The issue requests optional static or signed OpenClaw attestation headers for outgoing external HTTP requests, with user/skill opt-in and domain scoping.

## Best Possible Solution

Define the trust model and outbound path scope first, then implement an opt-in, domain-scoped attestation config/API with privacy-preserving defaults, docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not broken existing behavior. Source inspection of current main and v2026.5.12 gives high confidence that OpenClaw lacks the requested attestation headers, config, and skill metadata surfaces.

## Solution Assessment

Unclear until maintainers settle the product/security design. Static headers, signed JWTs, and per-skill metadata are plausible directions, but a local header patch is not the best solution until trust, privacy, redirect behavior, and outbound scope are explicit.

## Security Review

Status: needs_attention

Summary: The requested identity and attestation infrastructure needs explicit security design before implementation.

Concerns:

- **[medium] Define attestation trust and replay controls first:**
  - body: A signed attestation header needs a clear issuer, signing authority, public-key distribution model, audience binding, expiry/replay window, and rotation story before OpenClaw can safely emit verifiable identity claims.
  - confidence: 0.92
- **[medium] Prevent cross-service identity leakage:**
  - body: Static or signed headers could leak stable agent, user, channel, or session identifiers to unintended hosts unless config and runtime enforce strict opt-in domain scoping, redirect handling, and non-leakage across fallback paths.
  - confidence: 0.9
