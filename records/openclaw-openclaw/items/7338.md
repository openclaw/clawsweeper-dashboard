## Summary

Keep this open as the canonical product and security design tracker: current main still lacks general, audience-scoped agent attestation, and the existing provider attribution and web-fetch extension surfaces do not solve the requested cross-cutting behavior.

## What This Changes

Add optional, domain-scoped identity or cryptographic attestation to outbound requests made by OpenClaw agents.

## Best Possible Solution

Define a truthful attestation claim and exact covered request classes, then implement audience-scoped opt-in behavior in a plugin, adding only the smallest generic core hook if existing extension seams cannot support it safely.

## Reproduction Assessment

Not applicable; this requests a new identity-attestation capability rather than reporting a failure of established current behavior.

## Solution Assessment

No, blanket static headers or an OpenClaw-signed JWT are not yet a defensible best solution because the security claim, signing authority, audience rules, and plugin/core ownership boundary remain undefined.
