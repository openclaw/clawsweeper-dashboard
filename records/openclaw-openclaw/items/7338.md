## Summary

Keep this open. Current main still has no opt-in, domain-scoped agent attestation surface for arbitrary external HTTP requests, and the live issue state treats this as a product/security design item rather than conservative cleanup.

## What This Changes

Request to add optional OpenClaw agent attestation headers or signed tokens to outbound external HTTP requests, with domain allowlisting and possible skill metadata.

## Best Possible Solution

Define the attestation trust model and outbound path scope first, then implement an opt-in, domain-scoped config/API surface with privacy-preserving defaults, docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main lacks the requested attestation config, outbound header injection, and skill metadata surfaces.

## Solution Assessment

Unclear until maintainers settle the product/security design. Static headers, signed JWTs, and per-skill metadata are plausible directions, but a local header patch is not the best solution until the trust model and outbound scope are explicit.

## Security Review

Status: needs_attention

Summary: The requested capability is identity and attestation infrastructure, so it needs explicit security design before code changes.

Concerns:

- **[medium] Define attestation trust and replay controls:**
  - body: A signed attestation header needs a clear issuer, key distribution model, audience binding, expiry, replay window, and rotation story before OpenClaw can safely emit verifiable identity claims.
  - confidence: 0.9
- **[medium] Prevent cross-service identity leakage:**
  - body: Static or signed headers could leak stable agent or user identifiers unless config and runtime enforce strict opt-in domain scoping, redirect handling, and non-leakage across fallback paths.
  - confidence: 0.88
