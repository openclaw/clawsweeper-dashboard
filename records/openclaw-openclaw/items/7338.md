## Summary

Keep open: current main and v2026.6.10 still lack a general opt-in, domain-scoped agent-origin attestation surface for arbitrary outbound HTTP requests. Existing provider attribution and plugin trust hooks are adjacent, but they do not solve signed or static agent-origin attestation for external services, so this still needs product and security design.

## What This Changes

The issue requests opt-in OpenClaw agent-origin attestation headers or signed tokens for outbound HTTP requests, with domain scoping and skill/config controls.

## Best Possible Solution

Define the trust model and exact outbound paths first, then implement an opt-in, domain-scoped attestation contract with privacy-preserving defaults, schema/docs, and tests for allowlisted injection and non-leakage.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug report. Source and release inspection show the requested attestation surface is absent in current main and v2026.6.10.

## Solution Assessment

Unclear until maintainers choose the trust model. A signed, opt-in, domain-scoped design is directionally safer than static global headers, but existing provider attribution is not a complete solution for arbitrary external requests.
