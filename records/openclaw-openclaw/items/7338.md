## Summary

Keep open: current main and v2026.6.5 still lack a general opt-in, domain-scoped OpenClaw agent attestation contract for arbitrary outbound HTTP requests; the existing provider attribution and inbound gateway headers are narrower surfaces, and the discussion already treats this as product/security design work.

## What This Changes

The issue requests optional static or signed agent attestation headers for outgoing HTTP requests, with user/domain scoping and possible skill metadata declarations.

## Best Possible Solution

Keep this as the canonical product/security design tracker, define trust and outbound scope first, then implement a privacy-preserving opt-in attestation contract through the right config/API/plugin seam with docs and non-leakage tests.

## Reproduction Assessment

Not applicable: this is a feature/security design request rather than broken existing behavior. Source, docs, release-tag, and related-item checks show the requested general attestation headers, config, and skill metadata surfaces do not exist today.

## Solution Assessment

Unclear until maintainers settle the product/security design. Static headers, signed JWTs, and per-skill metadata are plausible inputs, but a local header patch is not the best solution until trust, privacy, redirects, and outbound scope are explicit.
