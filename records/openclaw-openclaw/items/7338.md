## Summary

Keep this open: current main and the latest shipped tag still lack a general opt-in, domain-scoped agent attestation surface, and the active discussion/labels keep this in product and security design rather than cleanup-close territory.

## What This Changes

Request an opt-in OpenClaw agent attestation mechanism for outbound HTTP requests, including possible static headers, signed attestations, per-skill metadata, and domain allowlisting.

## Best Possible Solution

Define the trust model and outbound path scope first, then implement an opt-in, domain-scoped attestation config/API with privacy-preserving defaults, docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested attestation config, outbound header injection, and skill metadata surfaces rather than showing a broken existing contract.

## Solution Assessment

Unclear until maintainers settle the product/security design. Static headers, signed JWTs, and per-skill metadata are plausible directions, but a local header patch is not the best solution until trust, privacy, and outbound scope are explicit.

## Security Review

Status: needs_attention

Summary: The requested identity and attestation infrastructure needs explicit security design before implementation.

Concerns:

- **[medium] Define attestation trust and replay controls first:**
  - body: A signed attestation header needs a clear issuer, signing authority, public-key distribution model, audience binding, expiry/replay window, and rotation story before OpenClaw can safely emit verifiable identity claims.
  - confidence: 0.9
- **[medium] Prevent cross-service identity leakage:**
  - body: Static or signed headers could leak stable agent or user identifiers to unintended hosts unless config and runtime enforce strict opt-in domain scoping, redirect handling, and non-leakage across fallback paths.
  - confidence: 0.88
