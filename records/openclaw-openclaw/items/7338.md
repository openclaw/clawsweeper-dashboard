## Summary

Keep open: current main still lacks opt-in, domain-scoped agent attestation for arbitrary outbound HTTP requests. The existing provider attribution code is vendor-transport attribution, while web_fetch, runtime config, and skill metadata still have no agent id, audience, signing, or allowlist surface.

## What This Changes

The issue requests optional OpenClaw agent attestation headers for outbound HTTP requests, with domain-scoped config and possible skill metadata.

## Best Possible Solution

Define the trust model and outbound scope first, then add an opt-in, domain-scoped attestation config/API with privacy defaults, docs, and tests for allowlisted injection plus non-leakage.

## Reproduction Assessment

Not applicable. This is a feature/security design request, and source inspection confirms the current code lacks the requested config, metadata, and outbound header paths rather than violating an existing contract.

## Solution Assessment

No for a direct header patch. The maintainable path needs maintainer product and security decisions before implementation, especially around signing, audience/domain scope, and privacy-preserving identifiers.
