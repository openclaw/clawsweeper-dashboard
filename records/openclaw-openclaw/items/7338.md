## Summary

Keep open: current main and v2026.6.10 still lack a general opt-in, domain-scoped OpenClaw agent attestation surface for arbitrary outbound HTTP requests. Existing provider attribution and trust/plugin hooks are adjacent but narrower, so the remaining work needs product and security design rather than cleanup closure.

## What This Changes

Request to add opt-in, domain-scoped static or signed OpenClaw agent attestation headers for outgoing HTTP requests, with skill metadata and config controls.

## Best Possible Solution

Define the trust model and exact outbound paths first, then implement an opt-in, domain-scoped attestation contract with privacy-preserving defaults, schema/docs, and tests for allowlisted injection and non-leakage.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug report. Source inspection of current main and v2026.6.10 gives high confidence that the requested attestation surface is absent.

## Solution Assessment

Unclear until maintainers choose the trust model. A signed, opt-in, domain-scoped design is directionally safer than static global headers, but existing provider attribution and trust hooks are not a complete solution for arbitrary external requests.
