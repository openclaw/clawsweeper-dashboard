## Summary

Keep this open as the canonical product/security tracker: current main still lacks general, opt-in agent attestation for external HTTP requests, and the required trust model and outbound scope are unresolved.

## What This Changes

The issue requests opt-in OpenClaw agent attestation headers or signed tokens for outbound HTTP requests to agent-only services.

## Best Possible Solution

Define the verifier trust model and exact request-owner boundary first, then add the smallest opt-in, audience-bound plugin or tool signing seam with privacy-preserving defaults and non-leakage tests.

## Reproduction Assessment

Not applicable; this issue requests a new identity and attestation capability rather than reporting broken existing behavior. Source inspection confirms the proposed surfaces are absent today.

## Solution Assessment

No, not as currently proposed: static headers are not a trustworthy attestation mechanism, and the cryptographic/config design still lacks a defined trust root, key lifecycle, request-owner boundary, redirect policy, and safe verifier semantics.

## Security Review

Status: needs_attention

Summary: The issue itself is security-sensitive because it asks OpenClaw to make externally verifiable origin claims for outbound requests.

Concerns:

- **[high] Define what attestation proves:**
  - body: A self-issued header or JWT cannot prove that a request came from an autonomous OpenClaw agent rather than a human or script controlling the same installation.
  - confidence: 0.96

  - body: Cryptographic attestation needs explicit key generation, storage, rotation, revocation, recovery, audience binding, expiration, and replay handling before it can be a security boundary.
  - confidence: 0.95

- The outbound scope spans web_fetch, provider transports, plugin tools, skills, MCP, browser automation, redirects, and proxies, so a partial implementation could leak stable identifiers or create inconsistent trust semantics.
