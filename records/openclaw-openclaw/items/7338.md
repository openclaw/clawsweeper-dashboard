## Summary

Current main still lacks general, opt-in agent attestation for external requests, and this remains the canonical tracker for a capability that requires product and security design before implementation.

## What This Changes

Requests opt-in, domain-scoped headers or cryptographic attestations that let external services identify requests as originating from an OpenClaw agent.

## Best Possible Solution

Keep this as the canonical design tracker, define the attested principal and threat model, then specify the smallest opt-in plugin/tool contract with audience allowlisting, ephemeral identifiers, redirect-safe non-leakage, protected key lifecycle, and verifier documentation.

## Reproduction Assessment

Not applicable: this is a new capability request rather than a failure of an existing contract; current-main source inspection establishes that the proposed general attestation surface is absent.

## Solution Assessment

No, the alternatives are not yet a complete solution: static headers are forgeable, while signed attestations require a defined trusted principal, issuer, key lifecycle, verifier model, and enforceable request scope.

## Security Review

Status: needs_attention

Summary: The requested identity mechanism needs an explicit threat model and trust-boundary decision before implementation.

Concerns:

- **[high] Define what the attestation actually proves:**
  - body: A signature from a user-controlled OpenClaw installation can prove possession of that installation's key, but does not inherently distinguish autonomous agent action from a human operating the same instance.
  - confidence: 0.96
- **[high] Prevent identifier and credential leakage:**
  - body: Domain matching, redirects, subdomains, proxies, plugin-owned requests, and browser navigation need explicit rules so identifiers or signed tokens cannot reach unintended recipients.
  - confidence: 0.95
- **[high] Specify signing-key lifecycle and replay controls:**
  - body: Cryptographic attestation requires secure key generation and storage, rotation, revocation, audience binding, short expiry, nonce or replay policy, and public-key discovery before verifiers can rely on it.
  - confidence: 0.97

## Risks / Open Questions

- A stable agent identifier or broadly injected header could enable cross-service tracking and leak identity across redirects or unintended destinations.
- Cryptographic attestation has no agreed signing authority, key custody and rotation model, revocation mechanism, audience binding, replay window, or verifier trust contract.
- The requested scope spans web_fetch, browsers, plugin-owned HTTP, skill subprocesses, model-provider SDKs, and arbitrary tools that core cannot uniformly attest without a precise enforcement boundary.
- Calling a request agent-originated may provide false assurance when a human controls the same OpenClaw instance, credentials, prompts, tools, or signing key.
