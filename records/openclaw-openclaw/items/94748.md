## Summary

Keep open: current main still has the reported overflow path in `fitCodexProjectedContextForTurnStart`, and no exact duplicate or merged follow-up fix was found.

## What This Changes

The issue reports that Codex projected context fitting can exceed `maxChars` when non-context wrapper text already fills the turn-start input budget.

## Best Possible Solution

Patch the fitter so its final return is always bounded to `maxChars`, including when the non-context wrapper and current request already exceed the limit, with focused regression coverage for that edge case.

## Reproduction Assessment

Yes at source level: with `beforeContext.length + afterContext.length > maxChars`, `contextBudget` is non-positive, `truncateOlderContext` returns an empty string, and the function returns the still-over-limit non-context text. I did not run the Vitest repro because this review is constrained to read-only inspection.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: keep the existing context-range trimming path, but add a non-positive-budget branch that bounds the final prompt and tests both small-cap and default-limit cases.
