## Summary

Keep open. Current main has static `extra_body` pass-through and a newer native transport-turn metadata seam for some OpenAI Responses/WebSocket paths, but it still does not provide session-aware OpenRouter/OpenAI-compatible `user` or `trace` request-body attribution, and related PRs #54725 and #64813 remain open for maintainer design review.

## What This Changes

Request to pass provider request metadata, especially OpenRouter/OpenAI-compatible `user` or `trace` fields derived from session, graph, task, or agent context.

## Best Possible Solution

Add an opt-in, sanitized attribution design that threads approved session/run metadata through a documented provider SDK/runtime seam, defines precedence with static `extra_body`, supports OpenRouter/OpenAI-compatible `user` and `trace` payload shapes where appropriate, and adds focused tests plus docs.

## Reproduction Assessment

Yes. Static inspection of current main gives a high-confidence reproduction path: configure an OpenRouter or OpenAI-compatible completions model without hard-coded `extra_body`, then no code path moves `sessionKey`/`sessionId` into provider extra params, the OpenRouter wrapper, or a `user`/`trace` request body.

## Solution Assessment

Unclear. The request is valid, but auto-setting raw `user = sessionKey` is not the safest default; a maintainer-approved opt-in attribution seam with sanitization and clear precedence is the narrower maintainable direction.
