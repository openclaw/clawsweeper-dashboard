## Summary

Keep open. Current main still lacks session-aware provider request-body attribution: it supports static OpenAI-compatible `extra_body` pass-through and native transport metadata, but the active provider extra-param and OpenRouter wrapper paths do not receive or inject `sessionKey`/`sessionId` as `user` or `trace`, and the related implementation PRs remain open for design review.

## What This Changes

Request to pass per-session or per-task request metadata such as `user` or `trace` into API provider request bodies for cost attribution.

## Best Possible Solution

Add an opt-in, sanitized attribution design through a documented provider SDK/runtime seam or provider config path, define precedence with static `extra_body`, support provider-specific `user`/`trace` payload shapes where appropriate, and cover it with focused tests and docs.

## Reproduction Assessment

Yes for the current feature gap: source inspection of current main shows no path from `sessionKey` or `sessionId` into provider extra-param contexts, OpenRouter wrapping, or OpenAI-compatible `user`/`trace` request bodies. No live provider call is needed to confirm that absence.

## Solution Assessment

Unclear. The requested attribution is useful, but automatically setting raw `user = sessionKey` is not clearly the safest default; an opt-in sanitized seam with documented precedence is the more maintainable direction.
