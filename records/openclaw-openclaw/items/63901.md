## Summary

Keep open. Current main still only supports static `extra_body` pass-through for OpenAI-compatible completions and the provider hook contexts remain session-agnostic; the two related PRs are open and have unresolved design/review blockers.

## What This Changes

Request to pass per-session or per-request attribution metadata such as `trace` or `user` through API provider requests so provider dashboards can distinguish agents, graphs, tasks, or sessions.

## Best Possible Solution

Define an additive provider metadata contract that is opt-in, cache-safe, and documented, then land the chosen static `user` passthrough and/or sanitized session attribution path with focused payload-shape tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request, not a broken documented behavior. Source inspection does give high-confidence current-main proof that dynamic/session-aware provider metadata is not implemented.

## Solution Assessment

No. Current static `extra_body` support is only a partial workaround, and automatic raw session-key forwarding would need an explicit opt-in API, cache isolation, provider-family guards, and privacy documentation.

## Risks / Open Questions

- Forwarding raw session keys can expose channel, peer, task, tenant, or agent identifiers to third-party provider logs, so the API should be opt-in and privacy-conscious.
- Static `extra_body` already covers hard-coded metadata for one API family, so remaining work should stay focused on dynamic/session-aware attribution rather than broad arbitrary payload injection.
