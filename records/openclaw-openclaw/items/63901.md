## Summary

Keep open. Current main still lacks a session-aware provider request metadata path for the reported OpenRouter cost-attribution use case: static `extra_body` can pass fixed JSON on `openai-completions`, and native Responses transports have a separate turn-state metadata seam, but OpenRouter LLM calls do not inject per-session `user` or `trace` data and the extra-param provider hooks still do not receive `sessionKey`/`sessionId`.

## What This Changes

Request to pass per-session, per-graph, or per-task metadata such as OpenRouter `trace` or `user` into provider API request bodies for cost attribution.

## Best Possible Solution

Keep the issue open until maintainers define a privacy-safe, opt-in provider metadata contract, then land guarded OpenRouter/LiteLLM-compatible `user`/`trace` handling with focused payload tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that the requested dynamic attribution path is absent: current main has only static `extra_body` passthrough on `openai-completions`, while the extra-param provider contexts and OpenRouter wrapper lack session-aware `user`/`trace` injection.

## Solution Assessment

Unclear. Option B is plausible, but automatically forwarding session keys to providers needs opt-in and sanitization; the maintainable direction is an additive provider/request metadata contract plus guarded OpenRouter payload behavior.
