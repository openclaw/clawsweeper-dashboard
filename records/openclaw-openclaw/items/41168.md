## Summary

Current main and the shipped release already address the central report: TUI relaunch restores a remembered non-heartbeat user session, and session persistence no longer repeats full resolved skill bodies or large skill prompts inline in each session row.

## What This Changes

The issue reports TUI relaunch selecting the heartbeat/main session instead of the last active user session, plus session storage bloat from duplicated skill context.

## Best Possible Solution

Keep the shipped TUI last-session restore and skill-prompt dedupe; broader session-store OOM or systemPromptReport work should stay on the canonical tracker at https://github.com/openclaw/openclaw/issues/55334.

## Reproduction Assessment

Yes, at source/history level: the old behavior is covered by current regression tests for remembered-session restoration, heartbeat filtering, and session-store skill stripping. I did not run a live Windows TUI reproduction in this read-only cleanup pass.

## Solution Assessment

Yes. The current implementation fixes the owner boundaries directly: TUI owns non-heartbeat session restoration, while session persistence owns stripping and deduplicating skill context before it reaches durable sessions.json rows.
