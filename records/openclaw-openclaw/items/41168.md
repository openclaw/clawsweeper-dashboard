## Summary

Current main still has the reported behavior, so this should stay open. The TUI still initializes an empty session argument to the configured main session and only refreshes metadata for that already-selected key, while session rows still expose `updatedAt` rather than a user-activity timestamp; the storage half is also not fixed on main because `skillsSnapshot` remains part of `SessionEntry` and the store write path serializes the full store.

## What This Changes

The issue requests that TUI startup without an explicit session resume the latest real user-active session instead of a heartbeat/main session, and that saved session storage stop duplicating full skill context.

## Best Possible Solution

Add a TUI startup path for the no-explicit-session case that waits for agent metadata, lists sessions for the selected agent, uses a user-activity signal such as `lastInteractionAt`, filters heartbeat/cron/system-only rows, and selects the newest real user session with fallback to the configured main session. Preserve explicit `--session` and global-scope behavior, update `docs/web/tui.md`, and resolve the storage bloat through #48425 or an equivalent merged serialization change.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: starting the TUI without `--session` in per-sender scope reaches `resolveTuiSessionKey` with an empty raw value, selects the agent main key, and then loads history for that key. The storage bloat path is also visible statically because `skillsSnapshot` remains on `SessionEntry` and the write path serializes the whole store.

## Solution Assessment

No, current main is not the best solution because it still uses the main-session default and `updatedAt`-based session metadata. A maintainable fix should expose or derive `lastInteractionAt` in the session-list contract and use that for TUI startup selection, while keeping the `skillsSnapshot` serialization fix coordinated with the existing storage PR.
