## Summary

Keep open. Current main has made useful progress on both halves, but it still does not implement latest-active user-session selection and the remaining storage cleanup overlaps open PR #48425 rather than a merged fix.

## What This Changes

The issue asks for TUI startup to resume the latest active user session instead of the heartbeat/main session and for session persistence to stop multiplying full skill context in `sessions.json`.

## Best Possible Solution

Define the TUI startup policy explicitly, expose or derive user-activity freshness in the session-list contract, filter heartbeat/system-only rows when appropriate, and land #48425 or an equivalent serialization cleanup for transient skill snapshot data.

## Reproduction Assessment

Yes for source-level reproduction: current main still initializes an empty TUI session to the agent main key and only restores a remembered key, while session listing exposes/sorts `updatedAt` rather than user activity. The storage bloat from full `resolvedSkills` is fixed, but complete `skillsSnapshot` exclusion remains unmerged in #48425.

## Solution Assessment

No, current main is only a partial solution: last-selected resume and `resolvedSkills` stripping reduce the reported pain, but they do not implement latest-active user-session selection or finish the serialization cleanup. The narrower maintainable path is to split TUI policy from the storage PR and review each on its own contract.
