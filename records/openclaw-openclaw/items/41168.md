## Summary

Keep open. Current main has important partial fixes, but TUI startup still restores the last selected session rather than choosing the latest active non-heartbeat user session, and the remaining session-store bloat work overlaps an open storage PR with review blockers.

## What This Changes

The issue asks TUI relaunch to resume the most recently active non-heartbeat user session and to stop sessions.json from growing by embedding full skill context in each saved session.

## Best Possible Solution

Define the TUI startup policy explicitly; if latest-active resume is desired, add a durable user-activity freshness signal with heartbeat filtering, and resolve storage bloat with a narrow metadata-preserving session-store serialization change.

## Reproduction Assessment

Yes, for source-level reproduction. With no explicit session, current TUI startup resolves to the agent main key; with a remembered key, it validates only that key and never chooses the newest non-heartbeat user session.

## Solution Assessment

No. Current main is only a partial solution: last-selected resume and resolvedSkills stripping address important symptoms, but latest-active selection and the remaining storage serialization policy still need maintainer direction.
