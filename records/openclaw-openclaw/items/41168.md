## Summary

Keep open. Current main and v2026.5.22 include useful mitigations for last-selected TUI resume, heartbeat-session rejection, and `resolvedSkills` stripping, but they still do not implement latest-active non-heartbeat TUI selection and still persist large `skillsSnapshot.prompt` / `systemPromptReport` payloads while https://github.com/openclaw/openclaw/issues/55334 and https://github.com/openclaw/openclaw/pull/78259 remain active storage follow-up paths.

## What This Changes

The issue asks TUI startup to resume the latest active user session instead of a heartbeat/main session and to stop repeated session skill-context metadata from bloating persisted storage.

## Best Possible Solution

Define the TUI resume contract around a reliable user-active signal that excludes heartbeat/system sessions, and sequence prompt/report compaction through the canonical session-store bloat work with migration and cold-resume coverage.

## Reproduction Assessment

Yes, at source level. Current main restores only an exact remembered TUI session, falls back to main/global when no usable remembered key exists, and still persists `skillsSnapshot.prompt` plus `systemPromptReport`; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or a simple latest-by-time sort. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-store design rather than deleting retained metadata broadly.
