## Summary

Keep open. Current main has useful partial fixes, but it still implements exact last-selected TUI restore rather than a latest-active non-heartbeat startup contract, and it still persists large prompt/report session metadata while the broader storage fix remains open in https://github.com/openclaw/openclaw/issues/55334 and https://github.com/openclaw/openclaw/pull/78259.

## What This Changes

The issue asks for TUI startup to resume the latest active non-heartbeat user session and for session storage to avoid repeatedly persisting full skill context.

## Best Possible Solution

Define a TUI resume contract around a reliable user-active signal that excludes heartbeat/system sessions, then sequence the canonical session-store prompt/report compaction with migration and cold-resume coverage.

## Reproduction Assessment

Yes, at source level. Current main starts from main/global when no explicit session is supplied, restores only an exact remembered key, and still persists skillsSnapshot.prompt plus systemPromptReport; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or a simple latest-by-time sort. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-store design instead of deleting retained metadata broadly.
