## Summary

Keep open: current main mitigates the worst skill-cache bloat and now remembers the last TUI-selected session, but it still does not implement the requested latest-active user-session selection. The TUI startup path can still fall back to the agent main session, and the session-list contract does not expose user-activity freshness for choosing around heartbeat/bookkeeping writes.

## What This Changes

The issue requests TUI startup to select the latest real user session instead of the heartbeat/main session and to stop sessions.json skill-context bloat.

## Best Possible Solution

Define the intended TUI startup policy, then implement a no-explicit-session path that can choose a real user-active session using lastInteractionAt while filtering heartbeat/system-only rows; keep the resolvedSkills mitigation and coordinate any broader skillsSnapshot serialization work with the open storage PR.

## Reproduction Assessment

Yes for a source-level reproduction: start gateway-mode TUI without --session and without a remembered TUI key; the resolver selects the agent main session and the later restore path only checks a remembered key. The session-list rows also do not expose lastInteractionAt, so current main cannot choose the newest real user-active session from source alone.

## Solution Assessment

No; current main is not the full fix because it resumes the last selected TUI session and strips resolvedSkills, but it does not implement latest-active user-session startup selection. A safer path is to split the TUI policy decision from the storage cleanup and add an additive activity-freshness field if maintainers choose latest-active behavior.
