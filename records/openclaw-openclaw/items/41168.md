## Summary

Keep open. Current main and v2026.5.19 still implement last-selected TUI restore, not latest-active non-heartbeat startup selection, and the remaining session-store payload bloat is still tracked by open storage issues and an open prompt-stripping PR.

## What This Changes

The issue requests TUI startup to resume the latest active non-heartbeat user session and for session persistence to stop accumulating full skill context per saved session.

## Best Possible Solution

Define the TUI resume contract explicitly, implement latest-active non-heartbeat selection only with a reliable user-activity signal, and resolve session payload bloat through the canonical session-store schema and prompt-rebuild work.

## Reproduction Assessment

Yes, at source level. Current main starts with main/global when no explicit session is supplied, restores only an exact remembered key, and still persists `skillsSnapshot.prompt` plus `systemPromptReport`; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or simple sort patch. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-bloat design instead of deleting retained metadata broadly.
