## Summary

Keep open. Current main has a useful last-selected TUI session mitigation, but it still does not implement the requested latest-active non-heartbeat startup selection, and the sessions.json storage bloat remains covered by active canonical storage work rather than fully fixed here.

## What This Changes

The issue requests that TUI startup choose the latest active user session instead of a heartbeat/main session and that sessions.json stop growing from repeated full skill context payloads.

## Best Possible Solution

Define the TUI resume policy explicitly, add latest-active non-heartbeat selection only if maintainers approve a reliable user-activity signal, and keep static session payload reduction centralized in the canonical storage-bloat work.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main resumes an exact remembered key or falls back to the agent main/global key, while persisted session rows still retain skillsSnapshot prompt/systemPromptReport payloads.

## Solution Assessment

No, not as a direct updatedAt sort patch. The TUI half needs an explicit resume-policy decision and a user-activity signal, while the storage half should continue through the canonical storage-bloat issue and open prompt-stripping work.
