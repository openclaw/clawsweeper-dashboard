## Summary

Keep open. Current main has partial TUI and storage mitigations, but it still implements last-selected TUI restore rather than a latest-active non-heartbeat startup contract, and the retained session-store prompt/report bloat is still tracked by open canonical storage work.

## What This Changes

The report asks the TUI to resume the most recent active non-heartbeat user session on restart and to stop duplicating large skill prompt/session metadata in `sessions.json`.

## Best Possible Solution

Define a TUI resume contract around a reliable user-active signal that excludes heartbeat/system sessions, then land the canonical session-store prompt/report compaction with migration and resume-proof coverage.

## Reproduction Assessment

Yes, at source level. Current main starts from main/global when no explicit session is supplied, restores only an exact remembered key, and still persists `skillsSnapshot.prompt` plus `systemPromptReport`; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or simple latest-by-time sort. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-store design instead of deleting retained metadata broadly.
