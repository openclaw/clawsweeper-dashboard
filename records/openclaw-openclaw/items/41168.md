## Summary

Keep open. Current main has useful partial fixes, but the TUI startup contract is still last-selected rather than latest-active user session, and the remaining session-store bloat work is still represented by open storage issues and PR coordination.

## What This Changes

The issue asks for TUI startup to resume the latest active user session instead of a heartbeat/main session and for session storage to stop duplicating skill context across saved sessions.

## Best Possible Solution

Define the TUI startup policy explicitly, add a durable user-interaction freshness signal if latest-active resume is desired, and resolve the remaining session-store serialization shape through the existing storage-bloat PR or canonical tracker.

## Reproduction Assessment

Yes for source-level reproduction. With no explicit session and no remembered key, current TUI startup resolves to the agent main key; with a remembered key, it restores only that exact last-selected key and never searches for the latest active non-heartbeat user session.

## Solution Assessment

No. Current main is only a partial solution: last-selected resume avoids some silent resets and resolvedSkills stripping removes the largest skill-body duplication, but latest-active selection and the broader persisted snapshot policy are still unresolved.
