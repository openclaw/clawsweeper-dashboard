## Summary

Keep open. Current main and the latest release have partial fixes, but TUI startup still resumes the last selected or exact remembered session instead of selecting the latest active non-heartbeat user session, and the broader session-store bloat work is still tracked in open storage follow-ups.

## What This Changes

The issue asks for TUI startup to resume the latest active non-heartbeat user session and for session-store writes to stop growing by repeatedly persisting skill context.

## Best Possible Solution

Define the TUI resume policy explicitly, add latest-active non-heartbeat selection only with a durable user-activity freshness signal, and resolve storage bloat through the canonical session-store schema work.

## Reproduction Assessment

Yes, for source-level reproduction. Current main defaults to the agent main session with no explicit session and validates only an exact remembered key, so it never chooses the newest non-heartbeat user session by activity time.

## Solution Assessment

No. Current main is only a partial fix, and switching directly to updatedAt ordering would be risky because system events and heartbeat bookkeeping can update session rows; maintainers need to define the intended user-activity signal first.
