## Summary

Keep open. Current main has partial TUI and storage mitigations, but it still restores only an exact remembered TUI session instead of choosing the latest active non-heartbeat user session, and session persistence still retains large prompt/report metadata while the broader storage work remains open.

## What This Changes

The issue asks for TUI startup to resume the latest active non-heartbeat user session and for session persistence to avoid repeated full skill-context storage.

## Best Possible Solution

Define a TUI resume contract around a reliable user-active signal that excludes heartbeat/system sessions, then sequence session-store prompt/report compaction through the canonical storage work with migration and cold-resume coverage.

## Reproduction Assessment

Yes, at source level. Current main starts from main/global when no explicit session is supplied, restores only an exact remembered key, and still persists skillsSnapshot.prompt plus systemPromptReport; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or a simple latest-by-time sort. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-store design instead of deleting retained metadata broadly.
