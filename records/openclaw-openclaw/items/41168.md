## Summary

Keep open. Current main fixes the older TUI relaunch-continuity slice by remembering a selected TUI session and refusing heartbeat restores, but it still does not implement latest-active non-heartbeat startup selection, and the remaining storage-bloat work is already tracked by canonical session-bloat work with an open PR.

## What This Changes

The issue asks the TUI to resume the latest active user session instead of heartbeat/main sessions and to prevent repeated full skill-context persistence from bloating saved sessions.

## Best Possible Solution

Define the TUI resume contract explicitly, then implement latest-active non-heartbeat selection only with a reliable user-activity signal while resolving session payload bloat through the canonical session-store work.

## Reproduction Assessment

Yes, at source level. Current main restores an exact remembered TUI key or falls back to main/global, and session persistence still retains prompt/report metadata beyond the stripped resolvedSkills cache.

## Solution Assessment

No. Closing this as implemented or patching a simple updatedAt sort would skip the product decision about user-active session semantics, and the storage portion should follow the canonical session-bloat contract work instead of deleting fields broadly.
