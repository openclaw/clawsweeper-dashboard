## Summary

Keep open. Current main and v2026.5.19 still implement documented last-selected TUI restore rather than latest-active non-heartbeat startup selection, and the session-store bloat half remains only partially mitigated with canonical storage issues and an open prompt-stripping PR still active.

## What This Changes

The issue asks for TUI startup to resume the latest active user session instead of heartbeat/main sessions and for persisted sessions to stop accumulating full skill context payloads.

## Best Possible Solution

Define the TUI resume contract around a reliable user-active signal, then resolve session payload growth through the canonical session-store schema and prompt-rebuild work already tracked by the open storage issues and PR.

## Reproduction Assessment

Yes, at source level. Current main starts with main/global when no explicit session is supplied, restores only an exact remembered key, and still persists skillsSnapshot.prompt plus systemPromptReport; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or simple latest-by-time sort. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-bloat design instead of deleting retained metadata broadly.
