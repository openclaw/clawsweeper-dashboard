## Summary

Keep open. Current main has partial fixes for TUI last-selected restore, heartbeat restore rejection, and some session-store bloat, but it still does not implement latest-active non-heartbeat startup selection and the remaining storage-schema work is tracked by open session-bloat issues and an open PR.

## What This Changes

The issue requests that TUI startup choose the latest active non-heartbeat user session and that saved sessions stop duplicating large skill context in `sessions.json`.

## Best Possible Solution

Define the TUI resume contract explicitly, implement latest-active non-heartbeat selection only with a reliable user-activity signal, and resolve session payload bloat through the canonical session-store work.

## Reproduction Assessment

Yes, at source level. Current main starts with main/global when no explicit session is supplied, restores only an exact remembered key, and still persists `skillsSnapshot.prompt` plus `systemPromptReport` in session entries; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or simple sort patch. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-bloat design instead of deleting retained metadata broadly.
