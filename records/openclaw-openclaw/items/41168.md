## Summary

Keep open. Current main has partial mitigations for both halves, but it still does not implement latest-active non-heartbeat TUI startup selection, and the retained `skillsSnapshot.prompt` / `systemPromptReport` bloat is still owned by active session-store follow-up work.

## What This Changes

The issue requests that TUI startup resume the latest active user session instead of a heartbeat/main session and that repeated full skill-context payloads stop bloating `sessions.json`.

## Best Possible Solution

Define the TUI resume policy explicitly, then implement latest-active non-heartbeat selection only with a reliable user-activity signal while landing the storage reduction through the canonical session-store bloat work.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main resumes an exact remembered key or falls back to main/global, and the session store still persists retained `skillsSnapshot.prompt` plus `systemPromptReport` payloads.

## Solution Assessment

No, not as a direct `updatedAt` sort patch. The TUI half needs an explicit resume-policy decision and user-activity signal, while the storage half should continue through the canonical storage-bloat issue and open prompt-stripping PR.
