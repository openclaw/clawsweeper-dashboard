## Summary

Keep open. Current main has partial TUI continuity and session-store mitigations, but it still implements documented last-selected TUI restore rather than latest-active non-heartbeat startup selection, and the duplicated session metadata work remains covered by open storage follow-ups.

## What This Changes

The issue requests TUI startup to choose the latest active non-heartbeat user session instead of the heartbeat/main session and to stop duplicating large skill context in persisted sessions.

## Best Possible Solution

Define the TUI resume contract around a reliable user-active signal that excludes heartbeat/system sessions, then sequence the session-store prompt/report compaction through the canonical storage tracker and focused open PR.

## Reproduction Assessment

Yes, at source level. Current main starts from main/global when no explicit session is supplied, restores only an exact remembered key, and still persists skillsSnapshot.prompt plus systemPromptReport; I did not run a live Windows/npm TUI workload.

## Solution Assessment

No, not as a cleanup close or simple latest-by-updatedAt sort. The TUI half needs a maintainer-approved user-active resume contract, and the storage half should follow the canonical session-store design and open prompt-stripping work.
