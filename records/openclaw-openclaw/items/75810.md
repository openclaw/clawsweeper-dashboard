## Summary

Keep open. Current main still has the managed-markdown duplicate-block bug and this PR targets the right helper, but the proposed cleanup rewrites whitespace across unmanaged markdown outside the matched block, so it should not merge as-is.

## What This Changes

The PR changes `replaceManagedMarkdownBlock` to match CRLF headed blocks and replace duplicate managed blocks, adds unit coverage for those cases, and adds an Unreleased changelog entry.

## Best Possible Solution

Land a scoped helper fix that matches CRLF boundaries and collapses repeated managed blocks while preserving every byte outside the matched managed-block ranges, with a regression test for unmanaged blank-line preservation.

## Reproduction Assessment

Yes. The current-main bug is reproducible with pure string inputs: a headed CRLF block misses the existing pattern, and repeated matching blocks leave later copies because replacement is not global. The PR finding is also reproducible with user markdown containing three blank lines outside the managed markers.

## Solution Assessment

No. The CRLF/global replacement direction is the right narrow boundary, but whole-document newline normalization is not; the safer implementation removes duplicate managed-block matches without normalizing unrelated markdown.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Limit cleanup to removed duplicate blocks:** `src/plugin-sdk/memory-host-markdown.ts:42`
  - body: The added cleanup runs over the entire `replaced` document whenever any existing block matches. That means user-authored content elsewhere with three or more blank lines, or intentional trailing blank lines, is collapsed even when no duplicate was removed. Scope the whitespace cleanup to gaps left by removed duplicate managed blocks, or avoid global blank-line normalization entirely.
