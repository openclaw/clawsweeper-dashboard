## Label Justifications

- `P1`: The report describes a broken real channel workflow where restart catch-up cron work can occupy a live chat lane and delay or suppress replies.
- `impact:session-state`: The suspected cause is cron work being attached to a live chat session lane through an inherited sessionKey.
- `impact:message-loss`: The observed user-visible failure is inbound WhatsApp messages being ingested while visible replies are delayed or absent.

## Summary

Keep open: current main and the latest release still bind omitted-target systemEvent cron jobs to the caller chat session key, and the open linked fix PR owns the implementation but has not merged.

## What This Changes

The issue requests stopping cron.add from binding implicit main/systemEvent jobs created from live chat to the caller's chat session key.

## Best Possible Solution

Land or replace the linked cron-only fix so implicit main/systemEvent jobs do not inherit caller chat session keys, while explicit sessionKey and current isolated cron-owned session behavior remain covered by regression tests.

## Reproduction Assessment

Yes, at source level: omitted-target systemEvent jobs normalize to main and cron.add stamps the resolved caller sessionKey whenever the target is not isolated. I did not rerun the live restart scenario.

## Solution Assessment

Yes. The narrow tool-level guard in a rebased linked PR or replacement PR is the best direction; opening a second repair lane while the closing PR remains open would duplicate the fix path.
