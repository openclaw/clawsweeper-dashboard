## Label Justifications

- `P1`: The report describes a broken real channel workflow where restart catch-up cron work can delay or suppress inbound chat replies.
- `impact:message-loss`: The observed failure is inbound WhatsApp messages being ingested while visible replies are delayed or absent.
- `impact:session-state`: The suspected cause is cron work being attached to the live chat session lane through an inherited sessionKey.

## Summary

Keep this open: current main and the latest release still allow the cron tool to stamp the caller chat session key onto implicit main/systemEvent jobs, and an open closing PR already owns the proposed implementation but has not merged.

## What This Changes

The issue asks OpenClaw to stop implicit cron main/systemEvent jobs from inheriting live chat session keys so restart catch-up work cannot occupy the same lane as inbound replies.

## Best Possible Solution

Land or replace the linked cron-only fix so implicit main/systemEvent jobs no longer inherit caller chat session keys, while explicit sessionKey and isolated agentTurn delivery behavior remain covered by regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an omitted-target systemEvent cron job normalizes to main and then cron.add stamps the caller sessionKey for non-isolated jobs; I did not rerun the live WhatsApp restart scenario.

## Solution Assessment

Yes. The narrow tool-level guard is the best fix direction for this trigger; the open linked PR should be cleaned up, landed, or replaced rather than opening a second repair lane now.
