## Summary

Keep open. Current main still has the source-level behavior: implicit systemEvent cron jobs default to the main session, cron.add then stamps the caller sessionKey when the job omits one, and main cron execution uses that sessionKey for system-event enqueue and heartbeat targeting; the open fixing PR at https://github.com/openclaw/openclaw/pull/81376 explicitly closes this issue and should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks OpenClaw to stop implicit main/systemEvent cron jobs created from live chat from inheriting the caller chat session key while preserving explicit session binding and isolated agentTurn context.

## Best Possible Solution

Review and land the narrow fix in https://github.com/openclaw/openclaw/pull/81376, or a replacement, so implicit main/systemEvent jobs stay unbound while explicit job.sessionKey and isolated agentTurn delivery context continue to work.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: create the cron tool with an agentSessionKey and add an implicit systemEvent job, and current main stamps the caller sessionKey before calling cron.add. I did not execute a live restart/WhatsApp repro because this review is read-only.

## Solution Assessment

Yes, the best fix boundary is the cron tool stamping guard proposed by the linked PR: keep explicit session binding and isolated agentTurn context, but do not attach a live chat session key to implicit main/systemEvent jobs. The broader lane-concurrency PR is adjacent but is not the narrow fix for this trigger.
