## Summary

Keep open. Current main still stamps the caller session key onto implicit main/systemEvent cron jobs, and the linked closing PR remains open and conflicting, so this should close only after that fix or a clean replacement lands.

## What This Changes

The issue reports cron jobs created from live chat inheriting a chat session key as implicit main/systemEvent work, which can occupy the same lane as inbound replies after restart.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81376 with a scoped cron-tool normalization fix and regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: current main defaults implicit systemEvent cron jobs to main, stamps a missing job.sessionKey from the caller session, and then uses that key for main cron enqueue/wake. I did not run the full live WhatsApp restart path in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to avoid caller-session stamping for implicit main/systemEvent jobs while preserving explicit job.sessionKey and isolated agentTurn context.
