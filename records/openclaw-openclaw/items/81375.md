## Summary

Keep this issue open: current main still has the source-reproducible cron/session-key binding path, and an open linked PR already targets this exact bug and references this issue for closure.

## What This Changes

The issue reports that cron.add can persist a live chat sessionKey on implicit main/systemEvent jobs, letting missed-job catch-up occupy the same chat lane after restart.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/81376; main should stop stamping caller sessionKey on implicit main/systemEvent cron jobs while preserving explicit keys and isolated agentTurn context.

## Reproduction Assessment

Yes, by source inspection: cron.add normalizes an implicit systemEvent job to main, stamps the caller sessionKey when missing, and main cron execution uses that sessionKey for enqueue and wake targeting. I did not rerun the live macOS restart scenario in this read-only review.

## Solution Assessment

Yes, the narrow tool-level guard in the linked PR is the best direction because it preserves explicit sessionKey and isolated agentTurn behavior while removing the accidental live-chat binding for implicit main/systemEvent jobs.
