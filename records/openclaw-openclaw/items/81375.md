## Label Justifications

- `P2`: The issue affects inbound chat responsiveness through a bounded cron/session-lane path and already has a focused fix candidate.
- `impact:message-loss`: The reported failure mode is that real inbound chat replies are delayed or effectively not delivered while cron work occupies the same lane.
- `impact:session-state`: The root path is cron jobs inheriting a live chat session key and reusing that session lane after restart.

## Summary

Keep open. Current main and v2026.5.18 still stamp the caller session key onto implicit main/systemEvent cron jobs, and the linked closing pull request remains open and conflicting.

## What This Changes

The issue asks for cron job normalization to avoid binding implicit main/systemEvent jobs created from live chat to the caller chat session key.

## Best Possible Solution

Keep explicit sessionKey and intentional current-session binding available, but make implicit main/systemEvent cron jobs avoid live chat sessionKey stamping and cover both paths with focused cron-tool regression tests.

## Reproduction Assessment

Yes, at source level: current main defaults systemEvent cron jobs to main, stamps a missing job.sessionKey from the caller session, and then uses that key for main cron enqueue/wake. I did not run the full live WhatsApp restart scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to guard implicit main/systemEvent jobs from caller sessionKey stamping while preserving explicit job.sessionKey and isolated agentTurn context.
