## Summary

Keep open. Current main still contains the cron tool path that stamps the caller session key onto an implicit main/systemEvent job, and an open linked PR with closing syntax targets this exact bug, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue reports that cron jobs created from live chat can inherit a chat session key as implicit main/systemEvent work and block inbound replies after restart.

## Best Possible Solution

Review and land a narrowed version of https://github.com/openclaw/openclaw/pull/81376 that keeps the cron tool guard and regression coverage while dropping unrelated dependency or fixture churn as needed.

## Reproduction Assessment

Yes for the tool-level binding path: current main defaults systemEvent jobs to main and then stamps the caller sessionKey onto missing sessionKey jobs. I did not reproduce the full post-restart WhatsApp stall live.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix: prevent implicit main/systemEvent cron jobs from inheriting live chat session keys while preserving explicit sessionKey and isolated agentTurn behavior. The issue should stay open until the linked PR is merged or superseded.
