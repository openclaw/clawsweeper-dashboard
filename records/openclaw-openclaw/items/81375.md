## Label Justifications

- `P1`: The original bug could make a live WhatsApp/chat workflow appear non-responsive after restart while cron catch-up occupied the same lane.
- `impact:message-loss`: The issue is about inbound chat replies being delayed or effectively suppressed behind cron work.
- `impact:session-state`: The root symptom involved cron work and human chat work sharing the same session lane state.

## Summary

Current main and v2026.6.6 already fix the reported lane collision: main-session cron work now runs on a cron-owned run lane while preserving the target chat delivery context. The still-open PR #81376 targets the older tool-stamping path, but the user-visible restart/catch-up symptom is superseded by the merged runtime fix.

## What This Changes

Prevent chat-created cron main/systemEvent jobs from occupying the live chat session lane during missed-job catch-up.

## Best Possible Solution

Keep the runtime cron-owned wake lane as canonical; handle any residual tool-stamping cleanup through #81376 only if maintainers still want it after the shipped runtime fix.

## Reproduction Assessment

Yes for the historical path: the issue and related #82766 identify the source-level shared-lane behavior, while current main source/tests show that path now routes through a cron-owned run lane.

## Solution Assessment

Yes. Moving execution at the cron runtime layer is the better fix because it covers existing persisted main-session cron jobs and all creation paths, not just future cron.add tool calls.
