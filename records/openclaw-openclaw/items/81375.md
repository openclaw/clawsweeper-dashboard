## Summary

Keep open. Current main and the latest shipped release still have the cron-tool path that stamps the caller session key onto implicit main/systemEvent jobs, and the linked closing pull request is still open, so this issue should close only after that fix or a cleaned replacement merges.

## What This Changes

The issue reports that cron.add can attach a live chat sessionKey to implicit main/systemEvent cron jobs, letting restart catch-up work occupy the same lane as inbound chat replies.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81376 with a scoped cron-tool normalization fix and focused regression coverage, then close this issue after the fix merges.

## Reproduction Assessment

Yes, at source level: current main defaults systemEvent cron jobs to main, stamps a missing job.sessionKey from the caller session, and then uses that key for main cron enqueue and wake. I did not run the full live WhatsApp restart scenario in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to stop implicit main/systemEvent jobs from inheriting the caller chat sessionKey while preserving explicit job.sessionKey and isolated agentTurn context; the linked PR follows that direction but still needs review or replacement.
