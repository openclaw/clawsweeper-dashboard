## Label Justifications

- `P1`: The report describes a real channel workflow where inbound WhatsApp replies can be delayed behind cron catch-up work after restart.
- `impact:message-loss`: The observable failure is that inbound WhatsApp messages are ingested but visible replies can be delayed or absent.
- `impact:session-state`: The suspected root cause is cron work being attached to the same session lane via sessionKey.

## Summary

Keep this open: current main and latest release still have the reported cron/session-key behavior, and an open closing PR already owns the proposed implementation but has not merged.

## What This Changes

The issue requests that chat-created implicit main/systemEvent cron jobs stop inheriting the caller chat session key so restart catch-up cannot occupy the live chat lane ahead of inbound replies.

## Best Possible Solution

Land a clean cron-tool fix that prevents implicit main/systemEvent jobs from binding to live chat session keys while preserving explicit sessionKey and detached agentTurn delivery semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main normalizes omitted systemEvent targets to main and then stamps the caller session key unless the target is isolated. I did not rerun the live WhatsApp restart scenario in this read-only review.

## Solution Assessment

Yes, the narrow tool-level guard is the best fix direction for this reported trigger. The existing linked PR should be cleaned up or replaced before merge, rather than opening a separate issue fix lane now.
