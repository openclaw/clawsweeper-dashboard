## Label Justifications

- `P1`: The report covered a real channel/session workflow where background cron work could delay or suppress user-visible chat replies after restart.
- `impact:message-loss`: The observed symptom was an inbound WhatsApp message being ingested without a timely visible reply while cron work occupied the lane.
- `impact:session-state`: The root problem was cron work sharing or confusing the same session lane/state used by live chat turns.

## Summary

Current main and the shipped v2026.6.6 release now run main-session cron work on cron-owned run lanes while preserving delivery context, so the reported live-chat lane collision is fixed even though the separate open guard PR did not merge.

## What This Changes

Report that chat-created main/systemEvent cron jobs can inherit a live chat session key and delay inbound replies after restart.

## Best Possible Solution

Keep the shipped cron-owned wake-lane behavior and close this report; any remaining cleanup to remove the obsolete linked PR should happen on that PR.

## Reproduction Assessment

Yes. The original source path and live logs showed main-session cron work sharing a live lane, and current regression tests now assert cron-owned run keys instead of the human main key.

## Solution Assessment

Yes. The merged runtime fix is broader and safer than only suppressing `cron.add` session-key stamping because it guarantees main cron executions use a cron-owned lane while preserving reply delivery context.
