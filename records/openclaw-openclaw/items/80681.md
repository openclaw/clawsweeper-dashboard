## Summary

Keep open: the direction is useful, but the core recorder still resolves the new cap from ambient `process.env` instead of the recorder env, and the PR body does not yet show after-fix real behavior proof for an oversized event preserved under the override.

## What This Changes

This PR adds a suffix-aware `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES` override for core and Codex trajectory event truncation, plus tests, docs, and a changelog entry.

## Best Possible Solution

Resolve the cap once from the recorder env for both writers, keep docs and focused tests aligned, and merge only after redacted after-fix proof shows an oversized event preserved under the override.

## Reproduction Assessment

Yes for source-level reproduction: current main has fixed 256 KiB caps in both core and Codex trajectory writers, and the PR head has a focused injected-env failure path in the core recorder. I did not run tests or a live session because this review is read-only.

## Solution Assessment

No, not yet: the Codex side resolves the cap from recorder env at startup, but the core side still reads ambient `process.env` inside truncation. The narrow fix is to compute the core event cap from `params.env` in `createTrajectoryRuntimeRecorder` and pass it into truncation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve the core cap from the recorder env:** `src/trajectory/runtime.ts:121`
  - body: `createTrajectoryRuntimeRecorder` already accepts an injected `env` and uses it for trajectory enablement and path resolution, but this new truncation path calls `resolveTrajectoryRuntimeEventMaxBytes()` with no env. That means `createTrajectoryRuntimeRecorder({ env: { OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES: "512" } })` still reads ambient `process.env`, and an active recorder can change behavior if `process.env` mutates mid-session. Resolve the cap from the recorder env once and pass that value into truncation.
  - confidence: 0.9
