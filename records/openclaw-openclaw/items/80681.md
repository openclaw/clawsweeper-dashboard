## Summary

Keep open: the updated PR now reaches both trajectory writers, but the Codex writer mis-parses documented suffix values such as `2mb`, and the PR still lacks after-fix real behavior proof from a real OpenClaw run.

## What This Changes

The PR adds an `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES` env override for core and Codex trajectory event truncation, plus tests, docs, and a changelog entry.

## Best Possible Solution

Use one shared byte-size resolver, or equivalent parser behavior, for both trajectory writers; cover raw and suffix values in core and Codex tests; keep docs aligned with the accepted config surface; and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes for the source behavior: current main hard-codes 256 KiB in both core and Codex trajectory writers, and the PR body shows production trajectory files with `truncated:true`. I did not run a live OpenClaw session in this read-only review.

## Solution Assessment

No. The direction is reasonable, but the Codex writer should share the same byte-size parsing contract as core, and the contributor still needs after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Parse Codex suffix values as byte sizes:** `extensions/codex/src/app-server/trajectory.ts:39`
  - body: The PR promises suffixes like `512kb` and `2mb`, and the core resolver implements them, but the Codex resolver uses `parseInt`. In Codex sessions `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES=2mb` resolves to `2`, so the cap shrinks to two bytes and events are dropped or truncated instead of preserved. Reuse the core parser or add equivalent suffix handling and tests here.
  - confidence: 0.96
