## Summary

Keep open. The PR is a focused config-surface feature, but the core runtime still resolves the new cap from ambient `process.env` instead of the recorder env, and the supplied proof does not show after-fix oversized-event preservation.

## What This Changes

The branch adds a suffix-aware `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES` override to core and Codex trajectory writers, plus tests, docs, and a changelog entry.

## Best Possible Solution

Resolve the core event cap once from the recorder env, pass that limit into truncation while keeping file caps intact, and land only after maintainer acceptance of the env surface plus redacted after-fix proof.

## Reproduction Assessment

Yes for source-level reproduction: current main hard-codes 256 KiB caps in both trajectory writers, and oversized events follow the truncation paths shown in source and tests. I did not run a live session because this review is read-only.

## Solution Assessment

No, not yet: the Codex writer resolves the cap from recorder env, but the core writer still calls the resolver without env per event. The narrow fix is to resolve from `params.env` in `createTrajectoryRuntimeRecorder` and pass that value into truncation.
