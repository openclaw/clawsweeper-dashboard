## Summary

Keep open: the override is useful and current main still has fixed 256 KiB caps, but PR head still reads ambient `process.env` in the core truncation path instead of the recorder env, and the supplied proof does not show after-fix preservation of an oversized event.

## What This Changes

The PR adds a suffix-aware `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES` env override for core and Codex trajectory event line caps, with tests, docs, and changelog coverage.

## Best Possible Solution

Resolve the cap once from the recorder env for both writers, keep docs/tests aligned with the accepted env surface, and merge only after redacted after-fix proof shows an oversized event preserved under the override.

## Reproduction Assessment

Yes for source-level reproduction: current main has fixed 256 KiB caps in both core and Codex trajectory writers, and oversized events flow through those truncation paths. I did not run tests or a live session because this review is read-only.

## Solution Assessment

No, not yet: the Codex side follows the recorder env, but the core side still resolves the cap from ambient `process.env` inside truncation. The narrow fix is to compute the core event cap from `params.env` when creating the recorder and pass that value into truncation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the recorder env into core truncation:** `src/trajectory/runtime.ts:121`
  - body: `createTrajectoryRuntimeRecorder` still accepts an injected `env` and uses it for enablement/path resolution, but this new cap path calls `resolveTrajectoryRuntimeEventMaxBytes()` with no env. A recorder created with `env: { OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES: "512" }` will read ambient `process.env` instead, and a long-lived recorder can change behavior if `process.env` mutates mid-session. Resolve the cap from the recorder env once and pass it into truncation.
  - confidence: 0.91
