## Summary

The current head is technically coherent, mergeable, fully checked, and backed by convincing live recorder output, but it adds a permanent operator environment contract and larger trajectory-retention behavior that require explicit owner/product acceptance before merge. Current `main` still has the fixed 256 KiB limit, no merged or viable canonical PR supersedes this work, and the latest branch has removed the unrelated QA-Lab change identified in the earlier review.

## What This Changes

The PR adds a startup-scoped environment-variable override for the per-event trajectory byte cap in both core and Codex trajectory recorders, with size parsing, file-budget clamping, tests, and documentation.

## Best Possible Solution

Approve the operational need and larger-retention tradeoff explicitly, then keep one clearly owned definition of the byte-size and cap semantics—exposed through an appropriate lightweight SDK seam if both recorders must consume it—while preserving the 256 KiB default and existing file budgets.

## Reproduction Assessment

Not applicable: this PR proposes a new operator capability rather than repairing broken behavior defined by an existing contract. The supplied real recorder output does provide a high-confidence demonstration of the proposed default and override paths.

## Solution Assessment

Unclear pending maintainer direction: startup-scoped resolution, safe fallback, and file-budget clamping are sound, but maintaining the same new configuration contract in two local implementations is not clearly the best long-term boundary.
