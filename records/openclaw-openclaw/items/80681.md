## Summary

Keep open: current main still has fixed 256 KiB trajectory event caps and this PR is relevant, but the head still resolves the core cap from ambient `process.env` per event instead of the recorder env, and the supplied proof does not show after-fix behavior.

## What This Changes

The PR adds a suffix-aware `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES` override for core and Codex trajectory event truncation, with tests, docs, and a changelog entry.

## Best Possible Solution

Resolve one validated event cap from each writer's startup env, pass it through the core truncation path, document the operator contract, and land after after-fix proof shows oversized events preserved under the override.

## Reproduction Assessment

Yes for source-level reproduction: current main has fixed 256 KiB caps in both core and Codex trajectory writers, and the PR head has a focused injected-env failure path in the core recorder. I did not run tests or a live session because this review is read-only.

## Solution Assessment

No, not yet: the direction is reasonable, but the core runtime should resolve the cap from the recorder env at creation and pass that value into truncation. Merge should also wait for maintainer acceptance of the new env surface and stronger after-fix proof.
