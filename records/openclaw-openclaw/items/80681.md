## Summary

Keep open: the PR is a useful focused feature, but the current patch still has a core env-contract bug and the supplied proof does not show after-fix oversized-event preservation.

## What This Changes

The PR adds a suffix-aware `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES` override to core and Codex trajectory event truncation, with tests, docs, and a changelog entry.

## Best Possible Solution

Land a narrow env override after resolving the cap from the same env contract in both writers, adding core runtime regression coverage, documenting startup/units/fallback behavior, and attaching redacted after-fix runtime proof.

## Reproduction Assessment

Yes for source-level reproduction: current main has fixed 256 KiB caps in both core and Codex trajectory writers, and both truncation paths use those caps. I did not run a live OpenClaw session because this review is read-only.

## Solution Assessment

No for the current PR as-is: the direction is maintainable, but the core runtime should resolve the cap from the recorder env rather than `process.env` implicitly. The safer fix is to mirror the Codex writer pattern and add a core runtime test for injected env behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the recorder env for the core event cap:** `src/trajectory/runtime.ts:121`
  - body: `createTrajectoryRuntimeRecorder` already resolves `env` from `params.env` and uses it for `OPENCLAW_TRAJECTORY` and path resolution, but this call reads from `process.env` instead. Any caller or regression test that passes `env: { OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES: ... }` to the core recorder will still get the default 256 KiB cap. Resolve the cap from the recorder env and pass that value into the bounded-line path, matching the Codex writer behavior.
  - confidence: 0.88
