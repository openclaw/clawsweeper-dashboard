## Summary

Keep open: current main still has fixed 256 KiB trajectory event caps, and the PR is a plausible fix, but the latest diff has a core env-injection bug and the supplied proof still does not show an after-fix oversized event preserved with the override.

## What This Changes

The PR adds a suffix-aware `OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES` override for core and Codex trajectory event truncation, plus tests, docs, and a changelog entry.

## Best Possible Solution

Resolve the cap once from the recorder env for both writers, document the env contract clearly, and land only after redacted after-fix runtime proof shows an oversized event preserved under the override.

## Reproduction Assessment

Yes for source-level reproduction: current main has fixed 256 KiB caps in both core and Codex trajectory writers, and both truncation paths use that cap. I did not run a live session in this read-only review.

## Solution Assessment

No, not yet: the direction is reasonable, but the core runtime should use the recorder env/startup value instead of ambient `process.env` per event. Merge should also wait for maintainer acceptance of the new config surface and stronger after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor the recorder env when resolving the core cap:** `src/trajectory/runtime.ts:121`
  - body: The core recorder already accepts an injected `env` and uses it for `OPENCLAW_TRAJECTORY` and `OPENCLAW_TRAJECTORY_DIR`, but this new path calls `resolveTrajectoryRuntimeEventMaxBytes()` without passing that env. As a result, `createTrajectoryRuntimeRecorder({ env: { OPENCLAW_TRAJECTORY_RUNTIME_EVENT_MAX_BYTES: "512" } })` still uses ambient `process.env` or the default, and an active recorder can change behavior if `process.env` mutates mid-session. Resolve the cap from the recorder env once and pass that value into truncation.
  - confidence: 0.86
