## Summary

Keep open. Current main has the useful runtime/account merge for refreshed health snapshots, but the plain non-probe health path can still return a fresh cached snapshot that was primed before channel startup populated Telegram runtime state; the latest reporter confirmation plus current source ordering make this a valid remaining diagnostics bug.

## What This Changes

The issue asks for `openclaw health --json` to stop reporting Telegram `running=false` and null lifecycle state when probe/status/doctor paths show the channel healthy.

## Best Possible Solution

Keep the shared runtime/account snapshot merge, then refresh or invalidate the public health cache after channel startup completes so plain `health --json` cannot serve a pre-channel-start channel summary. The fix should stay generic in the gateway health/cache lifecycle rather than adding Telegram-specific behavior.

## Reproduction Assessment

Yes. Configure Telegram, restart the gateway, let the plain health cache be primed during early startup, then compare `openclaw health --json` with `openclaw status --deep`, `openclaw doctor`, or `openclaw channels status --json --probe`; current source still supports that stale-cache timing path.

## Solution Assessment

No. Current main is only a partial solution: refreshed snapshots merge runtime state correctly, but startup can still prime and serve a stale cached health snapshot before `startChannels()` publishes runtime state.
