## Summary

Keep open. Current main and the latest shipped tag still create native `sessions_spawn` children as fresh UUID-backed sessions, keep `resumeSessionId` ACP-only, and Microsoft Teams still has no `subagent_spawning` hook path; the closed warm-reuse report points back to this as the canonical native subagent reuse tracker.

## What This Changes

Request native `sessions_spawn` session reuse for expert subagents, especially on Teams, instead of creating a fresh child session for every run-mode spawn.

## Best Possible Solution

Define an explicit native subagent reuse contract or Teams thread-binding path with isolation scope, TTL/invalidation, concurrency locking, cleanup, tool-schema/docs updates, and focused regression coverage.

## Reproduction Assessment

No, not as a strict bug reproduction because this is a feature request. Source inspection gives a high-confidence missing-capability path: native run-mode spawns still choose UUID child sessions, native resume stays ACP-only, and Teams has no `subagent_spawning` hook.

## Solution Assessment

Unclear. Registry-based reuse is plausible, but maintainers need to choose the boundary between core TTL reuse, a Teams hook, and explicit existing-session continuation before implementation.

## Risks / Open Questions

- A naive native reuse key or pool could mix unrelated expert histories, workspace state, channel context, tool permissions, model settings, or credentials unless the isolation scope is explicit.
- Concurrent spawns need liveness and locking semantics so an active child session is not reused unsafely.
- The remaining design choice is product/API-level: core registry reuse, a Teams hook, and explicit continuation through an existing session each create different user contracts.
