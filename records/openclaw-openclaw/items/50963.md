## Summary

Keep open. Current main and the latest shipped tag still lack native run-mode subagent session reuse, Teams still has no Discord/Feishu-style subagent thread-binding hook, and the related warm-reuse report was closed in favor of this issue rather than resolving it.

## What This Changes

The issue requests a native way for expert subagents, especially on Teams, to reuse prior session state instead of creating a fresh session on every run-mode spawn.

## Best Possible Solution

Define an explicit native subagent reuse contract or a Teams thread-binding path, including isolation scope, TTL/invalidation, concurrency locking, cleanup, tool-schema/docs updates, and regression coverage, while keeping this issue as the canonical native `sessions_spawn` reuse tracker.

## Reproduction Assessment

No, not as a strict bug repro because this is a feature request. Source inspection gives a high-confidence missing-feature path: native run-mode spawns still create UUID-backed child sessions, native resume is ACP-only, and Teams has no subagent thread-binding hook registration.

## Solution Assessment

Unclear. Registry-based reuse is plausible, but maintainers should first choose the boundary between core TTL reuse, a Teams hook, and explicit `sessions_send(sessionKey=...)` continuation semantics.
