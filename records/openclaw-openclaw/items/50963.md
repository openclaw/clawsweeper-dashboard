## Summary

Keep open: current main and the latest release still create native subagent child sessions with fresh UUIDs and keep resume behavior ACP-only. Teams exposes thread capability but has no subagent spawning hook, so the remaining work is a real product/API decision around core reuse versus channel binding, isolation, TTL, locking, and cleanup.

## What This Changes

Request native expert-subagent session reuse for repeated `sessions_spawn` calls, especially for Teams where thread-bound session mode is not available today.

## Best Possible Solution

Define an explicit native subagent reuse contract, or a Teams-compatible thread-binding hook path, with isolation, liveness/TTL, locking, cleanup, docs, and tests before implementation.

## Reproduction Assessment

Yes for the source-level gap: native `sessions_spawn` still selects a fresh UUID child key and has no native reuse or resume field. I did not run a live Teams/provider cost reproduction in this read-only review.

## Solution Assessment

No as a ready-to-implement direction: the request is valid, but adding a reuse parameter or automatic pooling before deciding ownership, isolation, liveness, and locking would be premature. The safer path is maintainer design first, then a focused implementation with tests.
