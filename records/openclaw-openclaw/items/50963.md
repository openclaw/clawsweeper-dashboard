## Summary

Keep open. Current main and the latest release still create native subagent runs with fresh UUID-backed child sessions, keep `resumeSessionId` ACP-only, and do not expose a native reuse or pooling contract; the remaining work is a product/API design decision rather than cleanup-closeable stale work.

## What This Changes

Request to let expert/native subagents reuse an existing session instead of creating a fresh session for every `sessions_spawn(mode="run")`.

## Best Possible Solution

Define an explicit native subagent reuse contract, then implement it in the core spawn path with documented isolation, TTL/liveness, locking, cleanup, channel/thread behavior, and focused tests.

## Reproduction Assessment

Yes at source level: current native `sessions_spawn` still creates a fresh UUID-backed child session key and does not accept native resume or reuse input. I did not run a live Teams/provider-cost reproduction in this read-only review.

## Solution Assessment

Unclear. Registry-based reuse is plausible and likely cleaner than per-channel hooks for `mode="run"`, but maintainers still need to choose isolation, TTL, locking, cleanup, and API shape before implementation.
