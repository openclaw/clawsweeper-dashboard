## Summary

Keep open. Current main and the latest release still create native subagent sessions with UUID-backed child keys, keep `resumeSessionId` ACP-only, and do not expose a native reuse/pooling contract; the closed warm-reuse issue now points back here as the canonical tracker, while the Active Memory issue is a broader adjacent performance request.

## What This Changes

Allow native expert subagents, especially Teams workflows without thread-bound session hooks, to reuse an existing session instead of creating a fresh child session on every `sessions_spawn(mode="run")`.

## Best Possible Solution

Define an explicit native subagent reuse contract, then implement it in the core spawn path with documented isolation, TTL/liveness, locking, cleanup, and channel/thread behavior.

## Reproduction Assessment

Yes at the source level: native `sessions_spawn` still chooses a fresh UUID child session key and the schema/docs/tests keep native resume or reuse absent. I did not run a live Teams or provider-cost reproduction in this read-only review.

## Solution Assessment

Unclear. Core registry-based reuse is plausible and likely better than per-channel hooks for `mode="run"`, but maintainers still need to choose isolation, TTL, locking, cleanup, and API shape before implementation.
