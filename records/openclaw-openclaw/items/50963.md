## Summary

Keep open. Current main and the latest shipped tag still create fresh native `sessions_spawn` child sessions, keep resume behavior ACP-only, and Teams still has no Discord/Feishu-style `subagent_spawning` hook; the closed warm-reuse report points back to this as the canonical native subagent reuse tracker.

## What This Changes

Request native `sessions_spawn` or Teams subagent session reuse so expert agents can continue prior context instead of starting a UUID-backed child session for every spawn.

## Best Possible Solution

Define an explicit native subagent reuse contract or Teams thread-binding path with isolation scope, TTL/invalidation, concurrency locking, cleanup, tool-schema/docs updates, and focused regression coverage.

## Reproduction Assessment

No, not as a strict bug reproduction because this is a feature request. Source inspection gives a high-confidence missing-feature path: native run-mode spawns still create UUID-backed child sessions, native resume remains ACP-only, and Teams has no subagent thread-binding hook registration.

## Solution Assessment

Unclear. Registry-based reuse is plausible, but maintainers should first choose the boundary between core TTL reuse, a Teams hook, and explicit `sessions_send(sessionKey=...)` continuation semantics.

## Risks / Open Questions

- A naive native reuse key or pool could mix unrelated expert histories, workspace state, channel context, model settings, tool permissions, or credentials unless the isolation scope is explicit.
- Concurrent spawns need locking and liveness semantics so an active child session is not reused unsafely.
- The remaining design choice is product/API-level: core registry reuse, a Teams hook, or explicit `sessions_send(sessionKey=...)` continuation each creates a different user contract.
