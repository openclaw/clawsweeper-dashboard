## Summary

Keep open: current main and v2026.6.10 still create native run-mode subagent sessions with fresh UUID-backed keys, keep resume behavior ACP-only, and do not provide a Teams-compatible native reuse or pooling contract.

## What This Changes

Requested native subagent session reuse for repeated expert spawns, especially Teams flows where thread-bound persistent subagent sessions are not available.

## Best Possible Solution

Define an explicit native subagent reuse or Teams-compatible session-binding contract with scoping, TTL/liveness, locking, cleanup, docs, and regression tests before implementation.

## Reproduction Assessment

Yes for a source-level reproduction: native sessions_spawn still chooses a fresh UUID child session key and has no native reuse or resume field. I did not run a live Teams/provider scenario in this read-only review.

## Solution Assessment

No immediate proposed option is clearly best yet: adding only a Teams hook is likely too narrow, while registry reuse needs explicit isolation, liveness, locking, and cleanup semantics. A maintainer-approved API contract should come first.
