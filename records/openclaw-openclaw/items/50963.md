## Summary

Keep open: current main still creates native subagent run sessions with fresh UUID-backed child keys and keeps reuse/resume behavior out of native `runtime="subagent"`; the remaining work is a product/API decision, not a cleanup close.

## What This Changes

Request adding a native subagent session-reuse path so repeated expert-agent spawns can preserve context instead of creating a new child session each time.

## Best Possible Solution

Define an explicit native subagent reuse contract, or a Teams-compatible session-binding adapter path, with isolation, liveness/TTL, locking, cleanup, docs, and tests before implementation.

## Reproduction Assessment

Yes for a source-level reproduction: current main's native `sessions_spawn` path still chooses a fresh UUID child session key and has no native reuse or resume field. I did not run a live Teams/provider cost scenario in this read-only review.

## Solution Assessment

No as an immediate implementation direction: adding a raw session key parameter or a Teams-only hook is not the best permanent fix without maintainer decisions on isolation, TTL, locking, cleanup, and channel binding. The best path is design first, then a focused implementation against the chosen core/session-binding seam.
