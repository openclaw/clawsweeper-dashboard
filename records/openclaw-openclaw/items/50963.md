## Summary

Keep open: current main and the latest release still create native run-mode subagent sessions with fresh UUID-backed keys, keep resume behavior ACP-only, and do not expose a native reuse/pooling contract. The linked KeepMyClaw comment is useful operational backup context, but it is not a live subagent session-reuse implementation, so the remaining work is still a maintainer product/API decision.

## What This Changes

Request native expert subagent session reuse so repeated `sessions_spawn(mode="run")` calls can continue prior context instead of always starting a fresh child session.

## Best Possible Solution

Define an explicit native subagent reuse contract, or a Teams-compatible session-binding adapter path, with isolation, liveness/TTL, locking, cleanup, docs, and tests before implementation.

## Reproduction Assessment

Yes for a source-level reproduction: current main's native `sessions_spawn` path still chooses a fresh UUID child session key and has no native reuse or resume field. I did not run a live Teams/provider cost scenario in this read-only review.

## Solution Assessment

No as an immediate implementation direction: adding a Teams `subagent_spawning` hook is not the best long-term path because that hook is deprecated, and automatic reuse needs maintainer decisions on ownership, isolation, liveness, locking, and cleanup. The best path is design first, then a focused implementation against the chosen core/session-binding seam.
