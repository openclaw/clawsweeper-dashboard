## Summary

Keep open. Current main still creates a fresh native subagent session for each `sessions_spawn` run and only allows persistent native subagent sessions through thread-binding hooks; Microsoft Teams advertises thread chat capability but does not currently register a `subagent_spawning` hook or expose a core reuse parameter. The KeepMyClaw comment is useful operational context, and #67000 is related warm-session work, but neither supersedes this Teams/session-spawn gap.

## What This Changes

Request subagent session reuse so expert agents, especially on Microsoft Teams, can continue an existing child session instead of each `sessions_spawn(mode="run")` creating a fresh one.

## Best Possible Solution

Keep this open until maintainers choose the contract. The maintainable end state is an explicit, documented persistent or reusable native subagent path that works for Teams, with tests around `sessions_spawn`, subagent registry cleanup, session ownership, thread delivery, and docs; if the fix is Teams-only, it should live in the Teams plugin using existing hook seams.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: native `sessions_spawn` produces a random child session key on each spawn, and `mode="session"` requires `thread=true` plus a `subagent_spawning` hook; Teams has no such hook in current main. I did not run a live Teams tenant test because the code path and tests already show the gating behavior.

## Solution Assessment

Unclear. The requested core registry reuse could be the right long-term fix, but it changes cross-channel session lifecycle semantics; a Teams hook is narrower but may not satisfy non-thread reuse or #67000-style warm-session goals. Maintainer design review should decide before an automated fix PR.
