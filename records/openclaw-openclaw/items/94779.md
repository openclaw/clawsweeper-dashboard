## Summary

Keep open: this PR targets a real source-visible serialized startup path, but it is not merge-ready because it changes runtime-level MCP/LSP initialization failures into missing tool families and still lacks real after-fix slow-MCP/LSP timing proof.

## What This Changes

The PR parallelizes bundled MCP catalog loading, bundled LSP server startup, and top-level MCP/LSP initialization in embedded agent attempts, with post-hoc LSP tool deduplication.

## Best Possible Solution

Parallelize independent MCP/LSP startup only after preserving existing runtime failure semantics and adding real multi-server timing proof; keep lazy loading and warm pooling as separate work.

## Reproduction Assessment

Yes from source for the latency shape: current main serializes MCP materialization before LSP startup, and the MCP catalog loop awaits each configured server in order. I did not run a live slow-server before/after trace in this read-only review.

## Solution Assessment

No, not as submitted: parallelizing independent startup is plausible, but the current patch silently drops configured tool families on runtime-level init errors. The better path preserves the existing failure contract, then adds real timing proof for the fan-out change.
