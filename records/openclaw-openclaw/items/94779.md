## Summary

Keep open: this is useful latency work, but the patch is not merge-ready because it changes runtime-level MCP/LSP initialization failures into missing tool families and still lacks a real after-fix multi-server timing trace.

## What This Changes

The PR parallelizes bundled MCP catalog loading, bundled LSP startup, and top-level MCP/LSP preparation in embedded agent attempts, then deduplicates LSP tools against MCP tools after both runtimes initialize.

## Best Possible Solution

Parallelize independent MCP/LSP startup only after preserving existing runtime-level failure propagation, explicitly accepting or bounding startup fan-out, and adding redacted real multi-server timing proof.

## Reproduction Assessment

Yes, from source for the latency shape: current main serializes MCP materialization before LSP startup, and the MCP catalog loop awaits each configured server in order. I did not run a live slow-server before/after trace in this read-only review.

## Solution Assessment

No, not as submitted: parallelizing independent startup is plausible, but the current patch silently drops configured tool families on runtime-level init errors. The safer path preserves the existing failure contract, then adds real timing proof for the fan-out change.
