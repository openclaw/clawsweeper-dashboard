## Summary

Keep open: this PR targets a real source-level latency shape, but it is not merge-ready because it converts whole MCP/LSP runtime initialization failures into missing configured tool families, and the supplied proof does not demonstrate the affected real slow multi-server path.

## What This Changes

The branch parallelizes bundle MCP catalog loading, bundle LSP startup, and top-level MCP/LSP tool preparation in embedded agent attempts, then deduplicates LSP tools against MCP names after both initialize.

## Best Possible Solution

Parallelize independent startup only after preserving the existing runtime-level failure contract, explicitly accepting or bounding startup fan-out, and adding redacted real multi-server timing proof.

## Reproduction Assessment

Yes for the source-level latency shape: current main awaits MCP catalog materialization before LSP startup and awaits each MCP server in sequence. I did not establish a live before/after multi-server timing reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Parallelizing independent startup is plausible, but the current patch silently drops configured tool families on runtime-level init errors and still needs real after-fix timing proof.
