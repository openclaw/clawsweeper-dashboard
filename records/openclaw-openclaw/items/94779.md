## Summary

Keep this PR open, but it is not merge-ready: the latency problem is real from source inspection, while the patch still changes whole-family MCP/LSP initialization failures into missing tools and the after-fix proof remains test/mock-only.

## What This Changes

The PR parallelizes bundled MCP catalog loading, bundled LSP startup, and top-level MCP/LSP tool preparation in embedded agent attempts, then deduplicates LSP tools against MCP tools after both runtimes initialize.

## Best Possible Solution

Parallelize independent startup only after preserving the existing runtime-level failure contract, bounding or explicitly accepting startup fan-out, and adding redacted real multi-server timing proof.

## Reproduction Assessment

Yes. Current main shows the latency shape from source: MCP materialization completes before LSP startup, and MCP catalog loading awaits each configured server in sequence.

## Solution Assessment

No, not as submitted. Parallelizing independent startup is plausible, but the current patch silently drops configured tool families on runtime-level init errors and still needs real after-fix timing proof.
