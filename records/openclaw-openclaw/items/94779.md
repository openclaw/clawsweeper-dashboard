## Summary

Keep open: the current main code still has the serialized MCP-before-LSP and per-MCP-server startup path this PR targets, but the patch is not merge-ready because it changes configured tool-runtime failure behavior and still lacks real slow multi-server after-fix proof.

## What This Changes

The PR parallelizes session MCP catalog loading, bundle LSP startup, and top-level MCP/LSP tool preparation, then deduplicates LSP tools after both initialize.

## Best Possible Solution

Parallelize independent MCP/LSP startup while preserving current runtime-level failure propagation and server-specific diagnostics, then prove the improvement with a redacted real multi-server trace.

## Reproduction Assessment

Yes for the source-level latency shape: current main awaits MCP catalog materialization before LSP startup and awaits each MCP server in sequence. I did not establish a live before/after multi-server timing reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Parallelizing independent startup is plausible, but the current patch silently drops configured tool families on runtime-level init errors and still needs real after-fix timing proof.
