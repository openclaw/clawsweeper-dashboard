## Summary

Keep open: this is a useful latency-fix candidate, but it is not merge-ready because the top-level catches change enabled tool-runtime failure behavior and the PR still lacks after-fix real multi-server timing proof.

## What This Changes

The branch parallelizes bundled MCP catalog loading, bundled LSP startup, and top-level MCP/LSP tool preparation in embedded agent attempts, then deduplicates LSP tools against MCP tool names after both initialize.

## Best Possible Solution

Parallelize independent startup only after preserving the existing runtime-level failure contract, bounding or explicitly accepting startup fan-out, and adding redacted real multi-server timing proof.

## Reproduction Assessment

Yes. Current main shows the latency shape from source: MCP materialization completes before LSP startup, and MCP catalog loading awaits each configured server in sequence; I did not establish a live before/after latency reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Parallelizing independent startup is plausible, but the current patch silently drops configured tool families on runtime-level init errors and still needs real after-fix timing proof.
