## Summary

Keep this PR open, but not mergeable as written: the docs change documents a real failure mode too broadly and conflicts with current auth-none local-backend behavior already covered by source and tests.

## What This Changes

The PR expands the `gateway.auth.mode: "none"` configuration-reference bullet to warn that no shared token is generated and backend WebSocket clients should use token mode.

## Best Possible Solution

Revise the docs to distinguish no shared token under auth-none, the direct-local `gateway-client` backend exception, and the browser-origin/remote/device-required failure path, recommending token mode only for clients that need shared-secret auth.

## Reproduction Assessment

Yes for the review finding: current-main source and existing tests give a high-confidence auth-none matrix for direct-local backend success versus browser-origin backend rejection. I did not run tests because this was a read-only review.

## Solution Assessment

No; the proposed wording is not the best solution because it turns a specific failure mode into a blanket backend-client warning. A narrower docs update matching the current auth matrix is safer.
