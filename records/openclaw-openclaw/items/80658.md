## Summary

Keep this PR open for revision: it targets a real auth-mode nuance, but the proposed wording says backend clients will fail with `gateway.auth.mode: "none"` even though current main has a scoped local-backend bypass and regression coverage proving that path succeeds.

## What This Changes

The PR expands the `gateway.auth.mode: "none"` entry in the Gateway configuration reference to warn that no shared token is generated and to recommend token mode for backend clients.

## Best Possible Solution

Clarify that `none` does not create shared-token credentials, but narrow the warning to token-dependent or nonlocal clients and preserve the current local backend/device-identity exceptions.

## Reproduction Assessment

No. Current-main source and tests instead show direct-local backend gateway clients can connect under `gateway.auth.mode: "none"` without device identity; any remote or token-dependent failure path still needs narrower proof.

## Solution Assessment

No. The docs should document the missing shared secret, but the proposed all-backend-client failure statement is broader than the current runtime contract.
