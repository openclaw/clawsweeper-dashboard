## Summary

Close: the docs change is superseded by https://github.com/openclaw/openclaw/pull/75781; current main and shipped v2026.5.12 allow auth-none local non-browser backend clients, so the proposed blanket warning would document the wrong behavior.

## What This Changes

This PR replaces the `gateway.auth.mode: "none"` configuration-reference bullet with a warning that backend WebSocket clients fail without a shared token and should use token mode.

## Best Possible Solution

Close this PR and keep the shipped https://github.com/openclaw/openclaw/pull/75781 behavior; any future docs should describe the scoped local backend versus browser/remote boundary instead of recommending token mode for all backend clients.

## Reproduction Assessment

No for the PR's blanket backend failure claim: current main source and regression tests show auth-none local non-browser backend clients succeed, while browser-origin backend clients still fail with `device identity required`. I did not run the tests during this read-only review.

## Solution Assessment

No; the proposed wording is not the best solution because it turns a fixed, scoped failure mode into a broad backend-client warning. The maintainable docs direction would state the exact no-auth local backend and browser/remote/device-identity boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Narrow the auth-none backend warning:** `docs/gateway/configuration-reference.md:517`
  - body: This sentence says backend clients will fail whenever `gateway.auth.mode` is `none`, but current main has an explicit local backend exception: local non-browser `gateway-client` backend connects with `authMethod === "none"` can skip device pairing, and the compat baseline asserts that path can invoke admin RPCs. The docs should distinguish direct-local backend clients from browser-origin or remote clients instead of recommending token mode for all backend compatibility.
