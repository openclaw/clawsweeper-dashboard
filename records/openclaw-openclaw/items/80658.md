## Summary

Keep open: the docs intent is useful, but the proposed wording is too broad because current main intentionally supports reserved direct-loopback `gateway-client` backend RPCs under `gateway.auth.mode: "none"`; the branch also still needs the maintainer-requested refresh before Dependency Guard can run.

## What This Changes

This PR expands the `gateway.auth.mode: "none"` configuration-reference entry to warn that no shared token is generated and backend WebSocket clients should use token mode to avoid `device identity required`.

## Best Possible Solution

Merge a narrower config-reference note that says auth-none provides no shared token while preserving the reserved direct-loopback `gateway-client` backend exception and steering custom, remote, or browser-origin clients to device identity/pairing, token/password auth, trusted proxy, or an HTTP RPC path as appropriate.

## Reproduction Assessment

Yes. Source and current tests show direct-local reserved `gateway-client` backend calls can run under auth-none without device identity, while browser-origin or non-exempt backend clients still hit `device identity required`.

## Solution Assessment

No as written. The maintainable fix is narrower documentation that separates the true no-shared-token fact from the supported local backend exception and the custom/remote/browser-origin failure cases.
