## Summary

Keep open: the config-reference clarification is useful, but the branch overstates current Gateway behavior because current main supports reserved direct-loopback `gateway-client` backend RPCs under `gateway.auth.mode: "none"`.

## What This Changes

The branch changes the Gateway configuration reference note for `gateway.auth.mode: "none"` to warn that no shared token is generated and to recommend token mode for backend client compatibility.

## Best Possible Solution

Merge a narrower config-reference note: auth-none does not provide a shared token, custom or remote/browser-origin WebSocket clients still need device identity or authenticated mode, and the reserved direct-loopback `gateway-client` backend helper remains supported.

## Reproduction Assessment

Yes. Source and tests show direct local reserved `gateway-client` backend calls can run under auth-none without device identity, while browser-origin or non-exempt clients still hit `device identity required`.

## Solution Assessment

No as written. The best fix is narrower documentation that separates the true no-shared-token fact from the supported local backend exception and the failing custom/remote/browser-origin cases.
