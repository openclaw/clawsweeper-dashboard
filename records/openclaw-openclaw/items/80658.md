## Summary

Keep open: the docs clarification targets a real auth-mode confusion, but the proposed wording is too broad and contradicts current main's direct-local backend exception for `gateway-client` backend RPCs.

## What This Changes

The PR changes the gateway configuration reference to warn that `gateway.auth.mode: "none"` does not provide a shared token and that backend WebSocket clients should use token mode.

## Best Possible Solution

Document the precise matrix: auth-none has no shared token and must stay trusted/private, direct-local `gateway-client` backend RPCs may run without device identity, and browser-origin or remote clients that need shared-secret auth should use token/password mode.

## Reproduction Assessment

Yes for the review finding: current-main source and existing tests establish that auth-none direct-local backend clients succeed, while browser-origin backend clients fail with `device identity required`. I did not execute tests in this read-only review.

## Solution Assessment

No; the PR is not the best solution as written because it turns a specific failure mode into a blanket backend-client warning. A narrower docs update matching `shouldSkipLocalBackendSelfPairing` and the compat tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the auth-none backend warning:** `docs/gateway/configuration-reference.md:517`
  - body: This sentence says backend clients will fail whenever `gateway.auth.mode` is `none`, but current main explicitly allows direct-local `gateway-client` backend connects with `authMethod === "none"`, and the compat test asserts that path can invoke admin RPCs. Please distinguish direct-local backend clients from browser-origin/remote/device-required clients instead of recommending token mode for all backend compatibility.
