## Summary

Keep open: the docs-only PR is trying to document a real auth-none/token distinction, but its current wording incorrectly says backend clients fail in all auth-none setups even though current main allows direct-local backend clients and rejects only browser-origin/remote/device-required paths.

## What This Changes

The PR replaces the `gateway.auth.mode: "none"` configuration-reference note with a longer warning that no shared token is generated and backend WebSocket clients should use token mode.

## Best Possible Solution

Revise the note to distinguish no shared-token generation from the direct-local backend exception, and recommend token mode only for clients that require shared-secret auth or are outside the trusted local backend path.

## Reproduction Assessment

Yes, source-level: current main and existing tests show the direct-local auth-none backend path succeeds while browser-origin backend connections fail with `device identity required`. I did not run tests because this was a read-only review.

## Solution Assessment

No; the current PR wording is too broad. The maintainable fix is a narrower documentation update that matches the shipped local-backend versus browser-origin/remote boundary.
