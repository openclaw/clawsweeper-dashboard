## Summary

Keep open with requested changes: the warning is directionally useful, but the proposed wording overstates current Gateway behavior because current `main` allows direct-local backend `gateway-client` connections in auth-none mode while rejecting browser-origin/remote missing-device paths.

## What This Changes

The PR replaces the one-line `gateway.auth.mode: "none"` configuration-reference note with a longer warning that no shared token is generated and backend WebSocket clients should use token mode.

## Best Possible Solution

Revise the note to say auth-none does not provide a shared token and to document the exact direct-local backend versus browser-origin/remote/device-identity boundary, keeping token mode as the recommendation for clients that require shared-secret auth.

## Reproduction Assessment

Yes for the review finding: current-main source and tests provide a high-confidence matrix for auth-none backend behavior. I did not run the tests, but the existing regression tests cover both the allowed local backend path and the rejected browser-origin path.

## Solution Assessment

No; the PR is not the best solution as written because it turns a specific failure mode into a blanket backend-client warning. A narrower docs update matching the existing auth matrix would be maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow the auth-none backend warning:** `docs/gateway/configuration-reference.md:517`
  - body: This sentence says backend clients will fail whenever `gateway.auth.mode` is `none`, but current main has an explicit local backend exception: `shouldSkipLocalBackendSelfPairing` returns true for direct-local `gateway-client` backend connects with `authMethod === "none"`, and `server.auth.compat-baseline.test.ts` asserts that this path can invoke admin RPCs. The docs should distinguish direct-local backend clients from browser-origin/remote/device-required clients instead of recommending token mode for all backend compatibility.
