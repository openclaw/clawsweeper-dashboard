## Summary

Keep this PR open, but it needs a wording fix before merge: the proposed note correctly notices that auth-none does not create a shared token, but it incorrectly says backend clients broadly fail even though current and shipped Gateway behavior preserves a reserved direct-local `gateway-client` backend path.

## What This Changes

The PR replaces the `gateway.auth.mode: "none"` configuration-reference note with a warning that no shared token is generated and backend clients should use token mode.

## Best Possible Solution

Merge a narrower configuration-reference note that says auth-none does not provide a shared token while preserving the reserved direct-loopback `gateway-client` backend exception and naming the non-exempt cases that still fail.

## Reproduction Assessment

Yes. Source and existing tests show auth-none local reserved backend clients can connect without device identity, while browser-origin and otherwise non-exempt clients still hit `device identity required`.

## Solution Assessment

No as written. The maintainable fix is a narrower docs note that distinguishes the absence of a shared token from the supported direct-local backend exception.
