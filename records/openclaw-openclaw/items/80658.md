## Summary

Keep open: the docs topic is useful, but this PR overstates current auth-none behavior because current main allows direct-local backend `gateway-client` connections while rejecting browser-origin or remote device-less paths, and the external PR still lacks sufficient after-fix real behavior proof.

## What This Changes

The PR expands the `gateway.auth.mode: "none"` configuration-reference note to warn that backend WebSocket clients fail without a shared token and should use token mode.

## Best Possible Solution

Revise the note to say auth-none does not provide a shared token while documenting the exact boundary: direct-local `gateway-client` backend RPCs can work without device identity, but browser-origin, remote, or shared-secret-dependent clients should use token or password mode.

## Reproduction Assessment

Yes for the review finding: current-main source and tests define the auth-none backend matrix, including the allowed direct-local backend path and the rejected browser-origin path. I did not run the tests in this read-only checkout.

## Solution Assessment

No; the PR is not the best docs fix as written because it turns a specific failure mode into a blanket backend-client warning. A narrower configuration-reference note matching the existing auth matrix would be maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the auth-none backend warning:** `docs/gateway/configuration-reference.md:517`
