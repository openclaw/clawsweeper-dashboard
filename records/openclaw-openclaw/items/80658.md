## Summary

Keep open: the docs clarification is useful, but the proposed wording overstates current Gateway behavior because auth-none direct-local reserved backend helper connections are supported while browser-origin or non-exempt device-less backend paths still fail.

## What This Changes

The PR replaces the `gateway.auth.mode: "none"` configuration-reference note with a longer warning that backend WebSocket clients fail without a shared token and should use token mode.

## Best Possible Solution

Merge a narrower docs note that says auth-none does not issue a shared token while preserving the reserved direct-local `gateway-client` backend exception and warning only about remote, browser-origin, custom, or non-exempt device-less clients.

## Reproduction Assessment

Yes. Current source and tests show auth-none direct-local reserved backend helper connects succeed without device identity, while browser-origin backend connects still fail with `device identity required`.

## Solution Assessment

No as written. The maintainable solution is narrower documentation that separates the true no-shared-token fact from the specific client classes that still require device identity, pairing, or shared-secret auth.
