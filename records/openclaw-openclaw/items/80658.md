## Summary

Keep this PR open, but it should not merge as written: the docs update points at a real auth-none troubleshooting area, yet the proposed warning overstates current behavior by saying backend clients broadly fail in no-auth mode.

## What This Changes

The PR expands the `gateway.auth.mode: "none"` configuration-reference bullet to warn that no shared token is generated and backend WebSocket clients should use token mode.

## Best Possible Solution

Revise the reference note to say auth-none does not provide a shared token, is only for trusted local loopback, and distinguish the direct-local `gateway-client` backend exception from browser-origin, remote, or shared-secret-required clients.

## Reproduction Assessment

Yes for the review finding: current-main source and existing regression tests show direct-local auth-none backend connects can succeed while browser-origin backend connects without device identity fail. I did not run the tests because this review was read-only.

## Solution Assessment

No; the PR is not the best solution as written because it turns a specific failure mode into a blanket backend-client warning. A narrower docs update matching the existing auth matrix would be maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89
