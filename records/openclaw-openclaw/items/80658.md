## Summary

Keep this PR open: the docs clarification is salvageable, but the current wording overstates auth-none backend failure and conflicts with current main's reserved direct-local `gateway-client` backend path.

## What This Changes

The PR changes one line in `docs/gateway/configuration-reference.md` to warn that `gateway.auth.mode: "none"` provides no shared token and recommends token mode for backend WebSocket clients.

## Best Possible Solution

Merge a narrower docs note that says auth-none provides no shared token while preserving the reserved direct-local `gateway-client` exception and steering custom, remote, or browser-origin clients to device identity, token/password auth, or admin HTTP RPC.

## Reproduction Assessment

Yes. Current source and tests show auth-none direct-local `gateway-client` backend connects succeed without device identity, while browser-origin device-less backend connects fail with `device identity required`.

## Solution Assessment

No as written. The maintainable fix is narrower documentation that separates the true no-shared-token fact from the client classes that still need device identity, shared-secret auth, or an HTTP RPC surface.
