## Summary

Keep open. The PR addresses a real auth-none documentation pitfall, but the proposed wording is too broad because current main deliberately allows direct-local backend `gateway-client` connections in auth-none mode while still rejecting browser-origin or remote/device-less paths.

## What This Changes

The PR expands the `gateway.auth.mode: "none"` configuration-reference bullet to warn that auth-none does not generate a shared token and that backend WebSocket clients should use token mode.

## Best Possible Solution

Revise the note to document that auth-none provides no shared token while preserving the exact direct-local backend versus browser-origin/remote/device-identity boundary, and leave broader service-identity design to https://github.com/openclaw/openclaw/issues/69066.

## Reproduction Assessment

Yes for the review finding: current-main source and tests show direct-local auth-none backend clients succeed, while browser-origin backend clients without device identity fail with `device identity required`. I did not run the tests in this read-only review.

## Solution Assessment

No; the PR as written overstates the failure by treating all backend clients as incompatible with auth-none. A narrower docs update matching the existing auth matrix is the maintainable fix.
