## Summary

Keep open: the docs clarification is useful in spirit, but the proposed wording is materially too broad because current `main` supports reserved direct-local `gateway-client` backend RPCs under `gateway.auth.mode: "none"`. The PR should be narrowed rather than closed, and the maintainer note about refreshing the branch still applies.

## What This Changes

The PR expands the `gateway.auth.mode: "none"` configuration-reference entry to warn that no shared token is generated and backend WebSocket clients may fail with `device identity required`.

## Best Possible Solution

Merge a narrower config-reference note that says auth-none provides no shared token while preserving the reserved direct-local `gateway-client` backend exception and steering custom, remote, or browser-origin clients to the appropriate auth path.

## Reproduction Assessment

Yes. Source and tests on current `main` show direct-local `gateway-client` backend RPCs can run under auth-none without device identity, while non-exempt clients still reach the `device identity required` path.

## Solution Assessment

No as written. The best fix is narrower documentation that separates the true no-shared-token fact from the reserved local backend exception and the custom/remote/browser-origin failure cases.
