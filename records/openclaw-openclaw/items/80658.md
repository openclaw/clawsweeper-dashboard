## Summary

Keep open: the docs-only change preserves a useful no-token clarification, but the added wording overstates current Gateway behavior because local non-browser `gateway-client` backend connects can work under `gateway.auth.mode: "none"` while remote/browser/custom device-less cases can still fail with `device identity required`.

## What This Changes

This PR replaces the `gateway.auth.mode: "none"` config-reference bullet with a longer note saying no shared token is generated and backend WebSocket clients should use token mode.

## Best Possible Solution

Merge only a scoped docs note: `none` does not generate a shared token, but only remote/browser/custom device-less backend clients that rely on shared credentials should expect `device identity required`; the reserved local `gateway-client` backend path remains an exception.

## Reproduction Assessment

Yes. The docs finding is source-reproducible: the PR line says backend clients fail under auth-none, while current helper code and tests show the local reserved backend path succeeds and remote/browser cases fail.

## Solution Assessment

No as written. The maintainable fix is narrower documentation that separates the true no-shared-token fact from the specific device-less client cases that still need device identity or shared-secret auth.
