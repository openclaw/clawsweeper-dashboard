## Summary

Keep open: the docs clarification is useful, but the proposed wording is too broad because current main supports auth-none for reserved direct-loopback `gateway-client` backend RPCs while still rejecting browser-origin or non-exempt backend clients without device identity.

## What This Changes

This PR changes the Gateway configuration reference to warn that `gateway.auth.mode: "none"` does not provide a shared token and recommends token mode for backend client compatibility.

## Best Possible Solution

Merge a narrower config-reference note: auth-none provides no shared token, custom/remote/browser-origin WebSocket clients need device identity or an authenticated path, and the reserved direct-loopback `gateway-client` backend helper remains supported.

## Reproduction Assessment

Yes. Source and current tests show direct-local reserved `gateway-client` backend calls can run under auth-none without device identity, while browser-origin or non-exempt backend clients still hit `device identity required`.

## Solution Assessment

No as written. The maintainable fix is narrower documentation that separates the true no-shared-token fact from the supported local backend exception and the custom/remote/browser-origin failure cases.
