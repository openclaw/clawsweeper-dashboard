## Best Possible Solution

Merge a narrower config-reference note that separates the true no-shared-token behavior from the supported local backend exception and the failing custom, remote, browser-origin, or otherwise non-exempt cases.

## Reproduction Assessment

Yes. Source and existing compatibility tests show direct local reserved `gateway-client` backend calls can run under auth-none without device identity, while browser-origin or non-exempt clients still hit `device identity required`.

## Solution Assessment

No as written. The best fix is narrower documentation that states auth-none does not provide a shared token while preserving the reserved direct-loopback backend exception.
