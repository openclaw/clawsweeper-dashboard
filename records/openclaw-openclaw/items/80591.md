## Summary

This should stay open: current main and the reported release still make Feishu setup try the scan-to-create QR flow first, while the report says that QR is not accepted by the China domestic Feishu app. I did not find a merged fix or a duplicate/canonical issue covering this exact setup regression.

## What This Changes

Investigate a reported regression where binding the Feishu channel in the 2026-05-07 OpenClaw release shows a QR code that the China domestic Feishu app does not respond to on Windows 10.

## Best Possible Solution

Verify the QR flow with a real China Feishu mobile client and either correct the registration parameters/domain handling or make manual credential entry immediately selectable so Feishu setup is not blocked by a QR that cannot be scanned.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection does show the reported QR-first setup path on current main and v2026.5.7, but the scan no-response symptom needs a real China Feishu mobile client and tenant to prove end to end.

## Solution Assessment

No, the current QR-first setup path is not sufficient for this report because manual credential entry is only reached after failure or timeout. The safer direction is to validate the Feishu registration QR and add an explicit manual fallback/skip path if the external scan flow is unreliable.
