## Summary

Keep open: current main still creates Feishu dynamic-agent direct bindings without `match.accountId`, while both DM and comment flows re-resolve routes with the active Feishu account id, so named-account traffic can keep falling back to `agent:main`. The closed related fix PR correctly identified the account-aware binding fix, but it was not merged and no current open canonical PR owns this exact regression.

## What This Changes

Make Feishu `dynamicAgentCreation` route each named-account DM user to an independent dynamic agent session instead of falling back to `agent:main`.

## Best Possible Solution

Land a focused Feishu plugin fix that passes normalized `accountId` into `maybeCreateDynamicAgent`, uses it in binding detection and creation, and preserves default-account legacy binding semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows named-account Feishu DM/comment traffic creates account-less direct bindings, then route resolution looks for that same peer under the named account id and misses the generated binding.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin: make dynamic-agent binding detection and creation account-aware using the public account-id normalizer while leaving core routing and default-account semantics unchanged.
