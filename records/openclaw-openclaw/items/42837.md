## Summary

Current main still has the reported Feishu named-account routing gap: dynamic agent creation writes direct-message bindings without `match.accountId`, while the router treats account-less bindings as default-account-only. The closed related PR identifies the right narrow fix direction, but it was not merged.

## What This Changes

Feishu `dynamicAgentCreation` should route each named-account DM user to an independent dynamic agent session instead of falling back to `agent:main`.

## Best Possible Solution

Land a narrow Feishu plugin fix that writes account-aware dynamic bindings for named accounts, preserves legacy default account-less binding behavior, and adds focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: a named-account Feishu DM with `dynamicAgentCreation.enabled` creates a binding without `match.accountId`, and the current router only evaluates that account-less binding for default-account traffic. I did not run live Feishu.

## Solution Assessment

Yes, the narrow maintainable fix is to pass and normalize the Feishu account id through `maybeCreateDynamicAgent`, include it in generated bindings, and test named-account plus legacy-default behavior; core routing already has the right account contract.
