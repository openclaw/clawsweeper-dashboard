## Summary

Keep open: current main still has the Feishu named-account dynamic-agent routing gap, and the latest release still contains the same account-blind helper path. A merged docs PR now documents the limitation and points back to this issue, but no runtime fix has landed.

## What This Changes

The issue requests that Feishu dynamicAgentCreation route each DM user to a separate dynamic agent/session instead of falling back to agent:main for named-account traffic.

## Best Possible Solution

Land a focused Feishu plugin runtime patch that carries the normalized accountId into dynamic-agent binding detection and creation for DM and comment flows while keeping default-account legacy behavior intact.

## Reproduction Assessment

Yes, at source level. A named Feishu account resolves routes with account.accountId, but dynamicAgentCreation writes an account-less binding that current routing tests define as default-account-only, so re-resolution can fall back to agent:main.

## Solution Assessment

Yes. The maintainable fix belongs in the Feishu plugin by making maybeCreateDynamicAgent account-aware through the existing account-id normalization contract and updating both DM and comment callers without changing core routing semantics.
