## Summary

Keep open: current main still writes Feishu dynamic-agent direct bindings without `match.accountId`, then re-resolves named-account DM/comment traffic with `account.accountId`, so generated bindings still miss named accounts and fall back to `agent:main`.

## What This Changes

Feishu `dynamicAgentCreation` should route each named-account DM user to an independent generated agent/session instead of continuing to use the main agent.

## Best Possible Solution

Land a focused Feishu plugin fix that makes dynamic-agent binding detection and creation account-aware while preserving default-account legacy bindings and core routing semantics.

## Reproduction Assessment

Yes by source inspection: named-account Feishu DM/comment traffic calls `maybeCreateDynamicAgent`, which writes an account-less direct binding, then routing re-resolves with the named account and will not match that default-only binding. I did not run a live Feishu tenant flow.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the active Feishu account id into `maybeCreateDynamicAgent`, normalize it through the public account-id helper, and write `match.accountId` on generated direct bindings without changing core routing or group/broadcast behavior.
