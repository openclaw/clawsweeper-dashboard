## Summary

Current main still has the reported Feishu named-account dynamic routing gap: dynamic agent bindings are created without `match.accountId`, while route resolution treats account-less bindings as default-account-only. The closed related PR is useful prior art, but it was not merged.

## What This Changes

A bug report that Feishu `dynamicAgentCreation` creates per-user agents and bindings, but named-account DMs still route to `agent:main` instead of independent sessions.

## Best Possible Solution

Land a narrow Feishu plugin fix that threads a normalized account id through `maybeCreateDynamicAgent`, writes `match.accountId` on generated dynamic bindings, preserves legacy default account-less behavior, and covers both bot and comment call sites with regression tests.

## Reproduction Assessment

Yes, source-reproducible: a named-account Feishu DM with `dynamicAgentCreation.enabled` creates an account-less direct binding, and the current router only considers that binding for the default account. I did not run a live Feishu tenant.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the active Feishu account id into dynamic-agent creation, use the shared account-id normalization contract, and keep core routing semantics unchanged.
