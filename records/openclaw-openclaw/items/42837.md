## Summary

Keep open: current `main` still creates Feishu dynamic-agent DM bindings without `match.accountId`, then re-resolves named-account traffic with `account.accountId`, so non-default accounts can still fall back to `agent:main`. The related fix at https://github.com/openclaw/openclaw/pull/42886 is closed unmerged, and targeted searches did not find another open PR owning this exact dynamic-agent account-scope fix.

## What This Changes

Fix the Feishu `dynamicAgentCreation` regression where named-account DM traffic creates agents and bindings but continues routing to `agent:main` instead of per-user dynamic sessions.

## Best Possible Solution

Make Feishu dynamic-agent binding creation account-aware inside the Feishu plugin while preserving default-account legacy behavior and wildcard routing semantics.

## Reproduction Assessment

Yes, source-level. With `dynamicAgentCreation` enabled for a non-default Feishu account, the helper writes a direct binding without `match.accountId`, and the subsequent account-scoped route resolution cannot match it because omitted account ids are default-only.

## Solution Assessment

Yes. The narrow maintainable repair is to pass the active Feishu account id into `maybeCreateDynamicAgent`, include account scope in existing-binding checks and generated bindings, and add regression coverage without changing core routing policy.
