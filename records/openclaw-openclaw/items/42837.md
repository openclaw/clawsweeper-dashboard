## Summary

Keep open: current main and the latest release still create Feishu dynamic-agent direct bindings without `match.accountId`, so named-account traffic re-resolves with an account id that cannot match the generated binding.

## What This Changes

The issue asks for Feishu `dynamicAgentCreation` DMs to route each user to an independent dynamic agent session instead of falling back to `agent:main` on named accounts.

## Best Possible Solution

Land a focused Feishu plugin fix that makes dynamic-agent binding detection and creation account-aware using SDK account normalization while preserving default-account legacy bindings and core routing semantics.

## Solution Assessment

## Repair Work Prompt

Fix the Feishu `dynamicAgentCreation` named-account routing bug reported at https://github.com/openclaw/openclaw/issues/42837. On current main, `maybeCreateDynamicAgent` creates direct bindings without `match.accountId`; named-account DM/comment flows then re-resolve with `account.accountId` and fall back to `agent:main`. Implement this inside the Feishu plugin only: pass the active account id from `extensions/feishu/src/bot.ts` and `extensions/feishu/src/comment-handler.ts` into `maybeCreateDynamicAgent`, normalize account ids with the public SDK account-id helper, make existing-binding detection account-aware, write normalized `match.accountId` for generated direct bindings, and preserve legacy account-less bindings as default-account-only while keeping wildcard account bindings working. Use https://github.com/openclaw/openclaw/pull/42886 as reference, but adapt it to the current `configWritesAllowed`/`replaceConfigFile` helper shape. Do not change core routing semantics, group/broadcast behavior, non-Feishu channels, or add startup migrations. Add focused regression coverage for existing-agent missing-binding and brand-new-agent creation on named accounts, plus default-account legacy behavior; add a single-line `CHANGELOG.md` fix entry if this becomes a maintainer repair PR.
