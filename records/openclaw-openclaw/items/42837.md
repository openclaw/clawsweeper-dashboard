## Summary

Keep open: current main still creates Feishu dynamic-agent bindings without `match.accountId`, while named-account routing only considers bindings for the active account or wildcard account, so named/non-default Feishu DM traffic can still fall through to the main agent. The merged docs PR documents the limitation rather than fixing it, and the latest release still contains the same account-less dynamic binding creation path.

## What This Changes

The issue reports that Feishu `dynamicAgentCreation` creates agents and bindings but DM messages still route to `agent:main` instead of per-user agents.

## Best Possible Solution

Add a focused Feishu plugin fix that passes the active account id into dynamic binding lookup and creation, writes `match.accountId` for named accounts, preserves default-account legacy behavior, and adds regression coverage for named-account DM routing.

## Reproduction Assessment

Yes, from source inspection: Feishu named-account messages route with an `accountId`, generated dynamic bindings omit `match.accountId`, and core routing ignores account-less bindings for non-default accounts. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu dynamic-agent path: pass account id into binding detection and writes, with regression coverage for both default and named-account behavior.
