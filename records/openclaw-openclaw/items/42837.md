## Label Justifications

- `P2`: This is a concrete Feishu channel routing regression with limited blast radius, not a core-runtime outage.
- `impact:session-state`: Named-account Feishu DM users can be routed into shared `agent:main` state instead of their intended per-user dynamic agent/session.
- `impact:message-loss`: Accepted Feishu DM messages can be misrouted to the wrong agent target when the generated binding cannot match the active account.

## Summary

Keep open: current main and the latest release still write Feishu dynamic-agent bindings without `match.accountId`, while named-account routing only considers bindings for the active account or `*`; the docs PR documents this limitation rather than fixing it.

## What This Changes

Feishu `dynamicAgentCreation` should route each DM user on named Feishu accounts to an independent dynamic agent/session instead of falling back to `agent:main`.

## Best Possible Solution

Pass the normalized active Feishu account ID into dynamic binding lookup and creation, and keep legacy account-less bindings default-account-only.

## Reproduction Assessment

Yes, from source: current main creates Feishu dynamic bindings without `match.accountId`, while named-account route resolution uses `account.accountId` and ignores account-less bindings. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu dynamic-agent path: pass accountId into binding detection and writes, with regression coverage for default and named-account behavior.
