## Summary

Keep open: current main still creates Feishu dynamic-agent DM bindings without `match.accountId`, while Feishu message and comment routing re-resolve with the active account id and route indexing treats account-less bindings as default-account-only. The focused related fix was closed unmerged, and the later merged config-write-policy work did not address named-account binding creation.

## What This Changes

The issue asks for Feishu `dynamicAgentCreation` to route each DM user on named Feishu accounts to their own dynamic agent session instead of falling back to `agent:main`.

## Best Possible Solution

Make Feishu dynamic-agent binding detection and creation account-aware in the Feishu plugin, using the shared account-id normalizer, preserving legacy account-less bindings as default-account-only, and covering DM plus comment flows with regression tests.

## Reproduction Assessment

Yes by source inspection: a named Feishu account with `dynamicAgentCreation.enabled=true` writes a direct binding without `match.accountId`, then the caller re-resolves with the named `accountId`, which current routing will not match. I did not run a live Feishu tenant flow, so this is source-reproducible rather than executed.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the active Feishu account id into `maybeCreateDynamicAgent`, check existing bindings with account-aware semantics, and write `match.accountId` on new dynamic bindings without changing core routing semantics.
