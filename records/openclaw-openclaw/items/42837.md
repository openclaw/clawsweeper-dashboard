## Summary

Keep open: current main and v2026.5.20 still create Feishu dynamic-agent bindings without account scope while named-account DM/comment routing re-resolves with the active account id, so traffic can still fall back to agent:main. The closed runtime PR at https://github.com/openclaw/openclaw/pull/42886 has the right account-aware shape to learn from, while the open docs PR at https://github.com/openclaw/openclaw/pull/82793 does not change runtime behavior.

## What This Changes

The issue asks for Feishu dynamicAgentCreation to route each DM user to the generated per-user agent/session instead of continuing to use agent:main.

## Best Possible Solution

Land a focused Feishu plugin runtime patch that passes normalized accountId into dynamic-agent binding detection and creation for DM and comment flows, writes account-scoped bindings for named accounts, and keeps default-account legacy behavior intact.

## Reproduction Assessment

Yes, at source level. A named Feishu account resolves routes with account.accountId, but dynamicAgentCreation writes an account-less binding that current routing tests define as default-account-only, so re-resolution can fall back to agent:main.

## Solution Assessment

Yes. The maintainable fix belongs in the Feishu plugin by making maybeCreateDynamicAgent account-aware through the existing account-id normalization contract and updating both DM and comment callers without changing core routing semantics.

Do not change core routing semantics, group routing, broadcast logic, or unrelated path-template/security policy. Add a user-facing CHANGELOG.md fix entry if the repair branch is prepared for merge.
