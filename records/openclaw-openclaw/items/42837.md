## Summary

Keep open: current main still creates Feishu dynamic-agent DM bindings without `match.accountId`, while Feishu DM and comment routing re-resolve with the active account id. The related fix was closed unmerged, and the later merged config-write-policy PR did not address named-account binding creation.

## What This Changes

Feishu dynamic agent creation should route each DM user on named Feishu accounts to an independent dynamic agent/session instead of falling back to `agent:main`.

## Best Possible Solution

Make Feishu dynamic-agent binding detection and creation account-aware in the Feishu plugin, preserving legacy account-less bindings as default-account-only and covering DM plus comment flows with regression tests.

## Reproduction Assessment

Yes by source inspection: a named Feishu account with `dynamicAgentCreation.enabled=true` writes a direct binding without `match.accountId`, then the caller re-resolves with that named `accountId`, which current routing will not match. I did not run a live Feishu tenant flow.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the active Feishu account id into `maybeCreateDynamicAgent`, use the shared account-id normalizer for matching, and write `match.accountId` on new dynamic bindings without changing core routing semantics.
