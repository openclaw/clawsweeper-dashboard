## Summary

Keep open: current main still creates Feishu dynamic DM bindings without `match.accountId`, while the shared routing contract treats account-less bindings as default-account-only. The related fix PR #42886 identified the same root cause but was closed unmerged, so the named-account regression still appears valid.

## What This Changes

The issue reports that Feishu DM dynamic agent creation creates agents and bindings but still routes messages to `agent:main` instead of per-user sessions.

## Best Possible Solution

Make Feishu dynamic agent creation account-aware for message and comment flows, writing `match.accountId` for the active account while preserving default-account legacy behavior and adding focused regression coverage.

## Reproduction Assessment

Yes, source-level. With a non-default Feishu account, the helper writes a direct binding without `match.accountId`, then Feishu re-resolves routing with that named account; the shared routing test confirms such bindings only match the default account.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the active Feishu account id into `maybeCreateDynamicAgent`, include it in generated/existing-binding checks, and cover named-account plus default-account behavior without changing group, broadcast, or core routing policy.
