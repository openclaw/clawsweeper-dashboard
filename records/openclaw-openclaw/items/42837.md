## Summary

Keep open: current main still creates Feishu dynamic DM bindings without `match.accountId`, while Feishu message and comment routing re-resolves with the active account id. The closed, unmerged fix PR at https://github.com/openclaw/openclaw/pull/42886 identified the same named-account root cause, so the regression remains actionable.

## What This Changes

The issue reports that Feishu `dynamicAgentCreation` creates agents and bindings but DM traffic still routes to `agent:main` instead of per-user sessions.

## Best Possible Solution

Make Feishu dynamic-agent binding creation account-aware inside the Feishu plugin while preserving default-account legacy behavior and wildcard routing semantics.

## Reproduction Assessment

Yes, source-level. With a non-default Feishu account, the helper writes or recognizes direct bindings without `match.accountId`, then message/comment flows re-resolve with that non-default account and the routing contract treats the unscoped binding as default-only.

## Solution Assessment

Yes. The narrow fix is to pass the active Feishu account id into `maybeCreateDynamicAgent`, include account scope in existing-binding checks and generated bindings, and add regression coverage for non-default, default legacy, and comment/message flows without changing core routing policy.

## Repair Work Prompt

Fix the Feishu dynamic-agent named-account regression tracked by https://github.com/openclaw/openclaw/issues/42837. Current main writes Feishu dynamic DM bindings without `match.accountId`, while message and comment routing resolve with `account.accountId`, so non-default account traffic does not match the generated binding and falls back to `agent:main`. Update `extensions/feishu/src/dynamic-agent.ts` so `maybeCreateDynamicAgent` accepts the active account id, normalizes it with the public `openclaw/plugin-sdk/account-id` helper, matches existing bindings by channel + direct peer + account scope, treats legacy account-less bindings as default-account-only, preserves `match.accountId: "*"` semantics, and writes normalized `match.accountId` on new/missing dynamic bindings. Pass the active Feishu account id from both `extensions/feishu/src/bot.ts` and `extensions/feishu/src/comment-handler.ts`. Preserve the current `configWritesAllowed` behavior and do not change group routing, broadcast behavior, core routing policy, or non-Feishu channels. Add focused regression coverage for non-default account creation, default-account legacy bindings, existing-agent missing-binding repair, and the message/comment call paths as needed. Add a single-line `CHANGELOG.md` fix entry because this is user-visible.
