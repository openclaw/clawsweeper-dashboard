## Summary

Keep open: current main still creates Feishu dynamic-agent DM bindings without `match.accountId`, while message/comment routing re-resolves with the active Feishu account id and the routing contract treats omitted account ids as default-only. The related closed, unmerged PR identified the same named-account root cause, so this remains an actionable regression.

## What This Changes

The issue asks Feishu `dynamicAgentCreation` to route each DM user to an independent per-user agent/session instead of `agent:main` when using named/non-default Feishu accounts.

## Best Possible Solution

Make Feishu dynamic-agent binding creation account-aware inside the Feishu plugin while preserving default-account legacy behavior and wildcard routing semantics.

## Reproduction Assessment

Yes, source-level. With `dynamicAgentCreation` enabled for a non-default Feishu account, the helper writes a direct Feishu binding without `match.accountId`, and the subsequent account-scoped route resolution falls through to `agent:main` because omitted account ids are default-only.

## Solution Assessment

Yes. The narrow repair is to pass the active Feishu account id into `maybeCreateDynamicAgent`, include account scope in existing-binding checks and generated bindings, and add regression coverage without changing core routing policy.
