## Summary

Keep open: current main still creates Feishu dynamic-agent bindings without `match.accountId`, while Feishu DM and comment routing re-resolve with the active account id, so named-account traffic can still fall back to `agent:main`. The related code PR had the right account-aware shape but is closed unmerged; the open docs PR documents default-account behavior and does not fix this runtime path.

## What This Changes

Feishu dynamicAgentCreation should create account-scoped per-user bindings so named-account DMs route to separate dynamic agents instead of `agent:main`.

## Best Possible Solution

Land a focused Feishu plugin fix that passes normalized account id into dynamic-agent binding detection and creation, preserves default-account legacy binding semantics, and adds targeted regression coverage.

## Reproduction Assessment

Yes, at source level. Current main creates account-less Feishu dynamic bindings and then re-runs route resolution with the named account id, while the routing contract treats account-less bindings as default-account-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin: make dynamic-agent binding detection and writes account-aware using the existing SDK account-id normalizer, without changing core routing or default-account semantics.
