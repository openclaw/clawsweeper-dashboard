## Summary

Keep open: current main and latest v2026.5.22 still have the Feishu dynamic-agent account-scope gap. The helper creates account-less bindings, while DM and comment routing re-resolve with the active Feishu account id, so named-account traffic can still fall back to agent:main; https://github.com/openclaw/openclaw/pull/42886 remains useful prior art but was closed unmerged, and https://github.com/openclaw/openclaw/pull/82793 only documented the limitation.

## What This Changes

The issue reports that Feishu dynamicAgentCreation creates agents and bindings but still routes direct messages to agent:main instead of per-user dynamic agents.

## Best Possible Solution

Land a focused Feishu plugin runtime patch that makes dynamic-agent binding detection and creation account-aware for DM and comment flows, updates the docs limitation once fixed, and preserves legacy default-account behavior.

## Reproduction Assessment

Yes, at source level. A named Feishu account resolves routes with account.accountId, but dynamicAgentCreation writes an account-less binding that the routing contract treats as default-account-only, so re-resolution can fall back to agent:main.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin by passing normalized accountId into maybeCreateDynamicAgent and both callers, without changing core routing semantics.
