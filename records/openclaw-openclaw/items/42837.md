## Summary

Keep open: current main still creates Feishu dynamic-agent direct bindings without `match.accountId`, while named-account DM and comment traffic re-resolves routes with the active Feishu account id and therefore misses the generated binding.

## What This Changes

The issue reports that Feishu dynamic-agent creation creates agents and bindings but named-account DM traffic still routes to `agent:main` instead of per-user sessions.

## Best Possible Solution

Land a focused Feishu plugin fix that passes the active account id into dynamic-agent creation, writes normalized `match.accountId` on generated direct bindings, and preserves default-account legacy binding semantics.

## Reproduction Assessment

Yes, by source inspection on current main: named-account Feishu DM/comment traffic creates account-less direct bindings, then re-resolves with a named account id that cannot match those bindings. I did not run a live Feishu tenant flow, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is account-aware binding detection and creation inside the Feishu plugin, using the public account-id normalizer and leaving core routing/default-account semantics unchanged.
