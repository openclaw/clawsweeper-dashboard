## What This Changes

The issue reports that Feishu dynamicAgentCreation creates agents and bindings but named-account DM messages still route to agent:main instead of per-user dynamic sessions.

## Best Possible Solution

Update the Feishu plugin so dynamic agent creation receives the normalized runtime accountId, writes it into generated direct-message bindings, treats legacy account-less bindings as default-account-only while preserving wildcard account bindings, and covers both message and comment dynamic-agent paths with regression tests.

## Reproduction Assessment

Yes, from source inspection: configure a non-default Feishu account with dynamicAgentCreation enabled, let the first DM create a binding without match.accountId, then re-resolve that DM under the named account and core routing cannot see the default-account binding. I did not run a live Feishu tenant flow.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-owned account-aware dynamic binding creation using the existing plugin-sdk account-id helper; core routing already has the correct account-scoped contract.
