## Summary

Keep open: current `main` and the latest release still create Feishu dynamic-agent DM bindings without `match.accountId`, while routing intentionally treats account-less bindings as default-account-only. The closed related PR identifies the same root cause and contains useful source work, but it was not merged and is stale against current APIs.

## What This Changes

The issue reports that Feishu `dynamicAgentCreation` creates agents and bindings but named-account DMs still route to `agent:main` instead of per-user dynamic sessions.

## Best Possible Solution

Land a current-main Feishu dynamic-agent fix that writes account-scoped bindings for named accounts while preserving the latest config-write policy and adding named-account regression coverage.

## Reproduction Assessment

Yes, for a source-level reproduction. On current `main`, configure a named Feishu account with dynamic agent creation enabled: the dynamic-agent path writes a binding without `match.accountId`, and the route resolver only matches that binding for the default account.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the active Feishu `account.accountId` into dynamic-agent creation, normalize it with the existing account-id helper, and write `match.accountId` on generated bindings rather than changing the generic routing contract.
