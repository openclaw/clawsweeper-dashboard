## Summary

Keep this open. Current main still has the Feishu named-account dynamic-agent routing gap: the helper creates direct-peer bindings without `match.accountId`, while DM and comment ingress resolve routes with the runtime Feishu account id. The directly related fix PR #42886 was closed unmerged, and no newer open PR found in the review specifically closes this issue.

## What This Changes

The issue reports that Feishu `dynamicAgentCreation` creates agents and bindings but still routes named-account DM users to `agent:main` instead of isolated per-user agents/sessions.

## Best Possible Solution

Make Feishu dynamic-agent creation account-aware inside the Feishu plugin: pass the normalized runtime `accountId` from DM and comment ingress into the helper, include `match.accountId` when checking and writing dynamic bindings for named accounts, preserve legacy account-less bindings for default-account traffic, and add focused regression tests for named accounts, default compatibility, missing-binding repair, and duplicate-binding avoidance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable Feishu `dynamicAgentCreation` on a named account, send a new DM, observe that the helper writes an account-less direct-peer binding, then route resolution with that named `accountId` cannot see the generated binding and falls back to the default route.

## Solution Assessment

Yes. The narrow maintainable fix is in the Feishu plugin’s dynamic-agent helper and DM/comment call sites; core routing already has explicit account matching semantics and should not be loosened for this Feishu-specific generated-binding bug.
