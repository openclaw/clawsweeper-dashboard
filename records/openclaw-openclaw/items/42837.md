## Summary

Keep open: current main still creates Feishu dynamic-agent bindings without `match.accountId` while named-account DM and comment routing re-resolves with `account.accountId`, so named-account traffic can still fall back to `agent:main`. The prior runtime fix was closed unmerged, and the merged docs change now documents the limitation rather than fixing it.

## What This Changes

The issue asks Feishu `dynamicAgentCreation` to route each DM user on named accounts to its generated per-user agent instead of `agent:main`.

## Best Possible Solution

Land a focused Feishu plugin runtime patch that passes normalized `accountId` into dynamic-agent binding detection and creation for DM and comment flows, writes account-scoped bindings for named accounts, and keeps default-account legacy behavior intact.

## Reproduction Assessment

Yes, at source level. A named Feishu account resolves routes with `account.accountId`, but `dynamicAgentCreation` writes an account-less binding that current routing tests and docs define as default-account-only, so re-resolution can fall back to `agent:main`.

## Solution Assessment

Yes. The maintainable fix belongs in the Feishu plugin by making `maybeCreateDynamicAgent` account-aware through the existing account-id normalization contract and updating both DM and comment callers without changing core routing semantics.
