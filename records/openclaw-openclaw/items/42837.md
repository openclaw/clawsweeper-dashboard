## Summary

Keep open: current `main` and latest release `v2026.6.6` still create Feishu dynamic-agent bindings without `match.accountId`, while named-account routing resolves with the active account ID and can fall back to `agent:main`. The focused account-aware repair was closed unmerged, and the merged docs PR documents the remaining limitation rather than fixing it.

## What This Changes

The issue reports that Feishu dynamic agent creation creates per-user agents and bindings but routes DM messages to `agent:main` instead of independent per-user agents.

## Best Possible Solution

Carry the active Feishu account ID into dynamic binding lookup and creation, while preserving default-account behavior for legacy account-less bindings.

## Repair Work Prompt

Fix Feishu `dynamicAgentCreation` routing for named accounts. Current `main` passes `account.accountId` to route resolution in `extensions/feishu/src/bot.ts`, but `maybeCreateDynamicAgent` in `extensions/feishu/src/dynamic-agent.ts` neither accepts accountId nor writes `match.accountId`, so generated direct bindings only match the default account. Implement the fix inside the Feishu plugin boundary: pass the active account ID into dynamic binding lookup/creation, write account-scoped bindings for named accounts, preserve legacy account-less bindings for the default account, and add focused regression coverage for default, named account, existing-agent/missing-binding, and new-agent creation paths. Do not add new config flags or broad routing changes; do not edit CHANGELOG.md. Related refs: https://github.com/openclaw/openclaw/issues/42837 and the closed unmerged attempted repair https://github.com/openclaw/openclaw/pull/42886.
