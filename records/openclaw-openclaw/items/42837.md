## Summary

Keep open: current main still creates Feishu dynamic-agent direct bindings without `match.accountId`, while DM and comment routing re-resolves with the runtime `account.accountId`, so named-account traffic can miss the generated binding and fall back to `agent:main`. The closed related code PR identified the right account-aware shape, while the open docs PR documents the default-account behavior and does not fix named accounts.

## What This Changes

The issue requests that Feishu `dynamicAgentCreation` route each DM user to an independent per-user agent/session instead of continuing to use `agent:main`.

## Best Possible Solution

Land a focused Feishu plugin fix that passes normalized `accountId` into dynamic-agent binding detection and creation, preserves default-account legacy binding semantics, and updates targeted regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows named-account Feishu DM/comment traffic creates account-less direct bindings, then route resolution looks for that same peer under the named account id and can fall back to `agent:main`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin: make dynamic-agent binding detection and creation account-aware using the public account-id normalizer while leaving core routing and default-account semantics unchanged.
