## Summary

Keep open: current main still lacks WhatsApp read-only config, and the PR head is a useful but incomplete feature branch with three source-visible blockers plus missing real behavior proof.

## What This Changes

The PR adds WhatsApp `readOnly` config at channel/account scopes and gates outbound messages, polls, reactions, typing/presence, read receipts, and pairing replies in the WhatsApp transport.

## Best Possible Solution

Land a repaired WhatsApp-owned read-only mode that gates all outbound transport actions consistently, still records pairing requests for operator approval, aligns config metadata/docs, and has redacted live WhatsApp proof.

## Reproduction Assessment

Yes for the PR review defects: PR-head source shows the empty-send early return before the guard, the read-only composing no-op, and the pairing branch that skips request creation. The requested read-only mode itself is a new feature/config surface rather than a current-main bug reproduction.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right owner boundary, but the branch needs consistent blocked-send behavior, preserved pairing persistence, conflict resolution, and live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` still returns a success-shaped no-op for empty text with no media before `assertWhatsAppWritable` runs. Move config/account resolution and the writable assertion before this early return so every outbound send attempt follows the blocked-send contract.
  - confidence: 0.91
- **[P2] Throw for read-only composing attempts:** `extensions/whatsapp/src/inbound/monitor.ts:546-548`
  - body: The listener `sendComposing` closure silently returns in read-only mode while reply/media sends and the listener send API throw a read-only error. Callers that use the blocked-send error as the signal will treat composing indicators differently, so align this closure with the rest of the outbound contract.
  - confidence: 0.88
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:139-143`
  - body: When default pairing access meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer`, so no pending pairing request is stored for operator approval. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
