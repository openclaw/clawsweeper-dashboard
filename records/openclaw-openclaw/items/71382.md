## Summary

Keep open. Current main still lacks WhatsApp `readOnly` configuration and transport enforcement, while the PR is useful but still has concrete merge-blocking gaps in guard ordering, composing behavior, pairing persistence, and user-facing docs/changelog coverage.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` config, threads it through account resolution and the Web listener, gates several outbound WhatsApp paths, and updates focused tests plus generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that documents `readOnly` at channel and account scope, resolves it per account, enforces one consistent hard outbound gate across message/media/poll/reaction/typing/presence/read-receipt/send-API paths, preserves inbound processing and pairing request creation without sending WhatsApp replies, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. The PR diff itself gives a high-confidence reproduction path: an empty-body/no-media `sendMessageWhatsApp` call returns before the read-only guard, `sendComposing` silently returns while other blocked sends throw, and a read-only pairing DM skips the helper that creates the pending pairing request.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right direction and is narrower than relying on generic `sendPolicy`, but this patch needs the guard ordering, composing contract, pairing persistence, and docs/changelog fixes before merge.
