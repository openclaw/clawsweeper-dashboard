Full review comments:

- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:139-144`
  - body: When `dmPolicy` resolves to pairing, this new `readOnly` branch skips `createChannelPairingChallengeIssuer(...)` entirely, so `upsertChannelPairingRequest` never runs. A new DM sender is blocked without creating the pending request an operator can approve, turning read-only mode into permanent deny instead of silent-but-approvable pairing. Keep request creation and suppress only the outbound pairing reply.
  - confidence: 0.95
- **[P3] Run the read-only guard before empty-send success:** `extensions/whatsapp/src/send.ts:86-93`
  - body: `sendMessageWhatsApp` still returns a success-shaped no-op for empty text/no media before it loads config and calls `assertWhatsAppWritable`. In read-only mode that makes one send path look successful while the rest of the transport reports a blocked send, so move the guard ahead of this return or make this path use the same blocked-send contract.
  - confidence: 0.91
- **[P3] Use one blocked-send signal for composing:** `extensions/whatsapp/src/inbound/monitor.ts:545-548`
  - body: The inbound `sendComposing` callback silently returns in read-only mode, while `sendTrackedMessage` and the exposed `sendPresenceUpdate` throw the read-only error. Callers that rely on a blocked-send error will see different behavior for composing than for replies/media/presence, so align this with the chosen read-only contract.
  - confidence: 0.88
- **[P3] Document the readOnly config surface:** `src/config/types.whatsapp.ts:56-57`
  - body: The PR adds `channels.whatsapp.readOnly` and account-level `readOnly` to the config type/schema, but the WhatsApp docs and schema help/UI hints do not describe the option or its scopes. This is a new user-facing config contract, so add the public docs/help entry that tells operators exactly what is blocked and how account inheritance works.
  - confidence: 0.82
