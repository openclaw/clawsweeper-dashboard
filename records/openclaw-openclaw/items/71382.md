## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve pairing requests while suppressing replies:** `extensions/whatsapp/src/inbound/access-control.ts:139-144`
  - body: This branch skips `createChannelPairingChallengeIssuer` entirely when `readOnly` is set. That helper both upserts the pending pairing request and sends the outbound pairing reply, so new DM senders become permanently blocked with no request for an operator to approve. Keep the request creation and make only `sendPairingReply` silent/blocked under read-only mode.
  - confidence: 0.93
- **[P2] Run the read-only guard before empty-message success:** `extensions/whatsapp/src/send.ts:93`
  - body: `sendMessageWhatsApp` returns `{ messageId: "", toJid }` for empty text/no-media before resolving config and calling `assertWhatsAppWritable`. In read-only mode that gives callers a success-shaped response instead of the same blocked-send signal as normal message sends, so the hard gate is not uniform.
  - confidence: 0.89
- **[P2] Make composing use the same blocked-send signal:** `extensions/whatsapp/src/inbound/monitor.ts:546-548`
  - body: The new `sendComposing` branch silently returns under read-only mode, while `sendTrackedMessage` and the listener `sendPresenceUpdate` wrapper throw `createWhatsAppReadOnlySendError`. Callers that treat that error as the proof that outbound traffic was blocked will get different behavior for typing indicators than for other sends.
  - confidence: 0.82
- **[P3] Document the new WhatsApp config option:** `src/config/types.whatsapp.ts:57`
  - body: The patch adds a user-facing `readOnly` config field to the WhatsApp schema/types, but the WhatsApp channel docs do not explain the channel/account scopes or how it interacts with `sendReadReceipts`, pairing, and agent runs. New config surface should be documented before becoming a supported contract.
  - confidence: 0.8
