## Summary

Keep open. The feature is still needed because current main lacks WhatsApp `readOnly` config or transport gating, but this PR has narrow merge-blocking contract gaps already identified in review comments.

## What This Changes

The PR adds channel/account WhatsApp `readOnly` config, resolves it per account, gates outbound WhatsApp transport paths, and updates focused tests plus generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that keeps inbound processing, enforces one consistent hard outbound gate across all WhatsApp send paths, preserves pairing request persistence without sending WhatsApp replies, and documents the new config.

## Reproduction Assessment

Yes. The PR diff and review comments give a high-confidence path: empty message sends return before the read-only guard, `sendComposing` silently no-ops instead of throwing like other blocked helpers, and the read-only pairing branch skips the pairing issuer that upserts pending requests.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right direction and narrower than relying on generic `sendPolicy`, but this patch needs the guard ordering, composing contract, pairing persistence, docs, and changelog fixes before merge.

  - body: With `readOnly` enabled, `sendMessageWhatsApp` can still return `{ messageId: "", toJid }` for empty text with no media before reaching the writable assertion. Move config/account resolution and the read-only assertion ahead of the empty-content return so every send attempt follows the same blocked-send contract.
  - confidence: 0.91
- **[P2] Throw for read-only composing attempts:** `extensions/whatsapp/src/inbound/monitor.ts:543-548`
  - body: The inbound `sendComposing` closure silently returns in read-only mode while reply/media sends and the listener send API throw a read-only error. Callers that use the blocked-send error as a signal will treat composing indicators differently, so align this closure with the rest of the outbound contract.
  - confidence: 0.87
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:136-141`
  - body: When `dmPolicy: "pairing"` meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer` entirely, so no pending pairing request is upserted for the operator to approve. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
- **[P3] Document the new readOnly config:** `src/config/types.whatsapp.ts:56`
  - body: This adds public `channels.whatsapp.readOnly` and account-level config, but the PR does not update the WhatsApp docs or `CHANGELOG.md`. OpenClaw policy requires behavior/API docs and a user-facing changelog entry for a discoverable feature.
  - confidence: 0.86
