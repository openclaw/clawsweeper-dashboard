## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep pairing requests while suppressing replies:** `extensions/whatsapp/src/inbound/access-control.ts:139-144`
  - body: When `readOnly` is true, this branch skips `createChannelPairingChallengeIssuer(...)` entirely. That helper is what upserts the pending DM pairing request before sending the reply, so new default-policy DM senders are blocked without an approval request operators can act on; create the request and suppress only the outbound reply.
  - confidence: 0.91
- **[P2] Run the writable guard before empty sends:** `extensions/whatsapp/src/send.ts:93`
  - body: The empty text/no-media return still happens before config resolution and `assertWhatsAppWritable`, so `sendMessageWhatsApp(..., "", { readOnly: true })` resolves with a success-shaped no-op instead of enforcing the hard read-only send contract. Move the guard ahead of this return or make the return explicitly read-only aware.
  - confidence: 0.86
- **[P2] Align composing with other read-only sends:** `extensions/whatsapp/src/inbound/monitor.ts:546-547`
  - body: `sendComposing` silently returns under read-only mode, while `sendTrackedMessage` and the exposed `sendPresenceUpdate` throw the read-only error. Callers that treat the read-only error as proof of a blocked outbound action will get a false success for typing indicators, so use the same blocked-send behavior here.
  - confidence: 0.82
- **[P2] Document the new WhatsApp config option:** `src/config/types.whatsapp.ts:56-57`
  - body: This adds a user-facing `readOnly` config option but does not update the WhatsApp channel docs, whose configuration pointer still lists delivery fields without the new mode. Add public docs for scope, restart/upgrade expectations, and what outbound actions are suppressed before making this a supported config contract.
  - confidence: 0.78
