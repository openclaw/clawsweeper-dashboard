## Summary

Keep open: current main still lacks a WhatsApp read-only transport/config mode, and this PR is the active implementation candidate, but PR head still has correctness issues around blocked-send consistency and pairing persistence plus missing real behavior proof.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` config, resolves it into account state, and gates WhatsApp outbound messages, polls, reactions, typing/presence, read receipts, and pairing replies in transport paths.

## Best Possible Solution

Land a repaired WhatsApp-owned read-only implementation that preserves inbound processing and pairing approval, blocks outbound WhatsApp side effects consistently, documents the config, and includes real WhatsApp proof.

## Reproduction Assessment

Yes for the PR review defects by source inspection of PR head: empty sends bypass the guard, composing silently no-ops, and read-only pairing skips request creation. The requested WhatsApp read-only mode itself is a new feature/config surface rather than a current-main bug reproduction.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right owner boundary, but the branch needs the blocked-send contract and pairing persistence fixed before merge plus real setup proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` still returns `{ messageId: "", toJid }` for empty text with no media before reaching this writable assertion. Move config/account resolution and the read-only assertion before the empty-content return so every outbound send attempt follows the blocked-send contract.
  - confidence: 0.91
- **[P2] Throw for read-only composing attempts:** `extensions/whatsapp/src/inbound/monitor.ts:546-548`
  - body: The listener `sendComposing` closure silently returns in read-only mode while reply/media sends and the send API presence path throw a read-only error. Align this closure with the rest of the outbound contract so callers can reliably detect blocked outbound attempts.
  - confidence: 0.88
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:139-143`
  - body: When `dmPolicy: "pairing"` meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer` entirely, so no pending pairing request is upserted for the operator to approve. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
