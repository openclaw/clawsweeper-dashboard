## Summary

Keep open: the PR addresses a real WhatsApp safety feature that current main still lacks, but the branch has three actionable correctness blockers and no after-fix real WhatsApp/OpenClaw proof.

## What This Changes

The PR adds WhatsApp `readOnly` config at channel/account scopes, resolves it through account state, and gates outbound WhatsApp sends, presence, receipts, reactions, polls, and pairing replies with focused tests and generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned read-only implementation that preserves inbound processing and pairing approval, blocks outbound WhatsApp side effects consistently, documents the config, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes for the PR review defects: PR-head source shows the empty-send early return before the read-only guard, the read-only composing no-op, and the read-only pairing branch that skips request creation. The requested WhatsApp read-only mode itself is a new feature/config surface rather than a current-main bug reproduction.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right owner boundary and narrower than relying on session sendPolicy alone, but the branch needs consistent blocked-send behavior and pairing persistence before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` still returns `{ messageId: "", toJid }` for empty text with no media before reaching `assertWhatsAppWritable`. Move config/account resolution and the writable assertion before this early return so every outbound send attempt follows the blocked-send contract.
  - confidence: 0.91
- **[P2] Throw for read-only composing attempts:** `extensions/whatsapp/src/inbound/monitor.ts:546-548`
  - body: The listener `sendComposing` closure silently returns in read-only mode while reply/media sends and the listener send API throw a read-only error. Callers that use the blocked-send error as the signal will treat composing indicators differently, so align this closure with the rest of the outbound contract.
  - confidence: 0.88
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:139-143`
  - body: When `dmPolicy: "pairing"` meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer`, so no pending pairing request is upserted for the operator to approve. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
