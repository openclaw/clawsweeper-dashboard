- **[P1] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:139-144`
  - body: With the default DM `pairing` policy, this new branch logs and skips the whole pairing issuer before `upsertChannelPairingRequest` can run. That suppresses the outbound reply, but it also leaves the sender with no pending request for an operator to approve, making read-only mode behave like a permanent deny path. Keep creating the pairing request and suppress only the WhatsApp reply send.
  - confidence: 0.93
- **[P2] Guard empty sends before returning success:** `extensions/whatsapp/src/send.ts:86-93`
  - body: `sendMessageWhatsApp()` still returns `{ messageId: "", toJid }` for empty text/no media before resolving config and checking `assertWhatsAppWritable`. In read-only mode that gives callers a success-shaped result for a blocked outbound attempt, which violates the hard gate contract and can hide attempted sends from callers that rely on the error.
  - confidence: 0.88
- **[P3] Use the same read-only signal for composing:** `extensions/whatsapp/src/inbound/monitor.ts:545-548`
  - body: The inbound `sendComposing` callback silently returns under `readOnly`, while `reply`, `sendMedia`, and the listener-exposed `sendPresenceUpdate` throw the read-only error. Callers that treat the thrown error as the blocked-send signal will see different behavior for typing indicators than for the other outbound helpers.
  - confidence: 0.78
- **[P3] Document the readOnly config key:** `src/config/types.whatsapp.ts:56-57`
  - body: This adds a user-facing WhatsApp config option at channel and account scope, but the WhatsApp docs and high-signal config list do not describe it. Add docs for both scopes, the hard outbound side effects it blocks, and the expected pairing/read-receipt behavior so users do not rely on an undocumented security-sensitive knob.
  - confidence: 0.82
