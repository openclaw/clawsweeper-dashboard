## Summary

Keep this PR open. Current main still lacks WhatsApp read-only mode, and the branch is a plausible implementation direction, but the proposed patch has concrete P2 correctness blockers in the read-only contract and still needs real WhatsApp behavior proof before merge.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` config, resolves it per account, and gates outbound WhatsApp send, presence, receipt, composing, poll, reaction, and pairing-reply paths with focused tests and generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that keeps inbound processing silent, blocks every outbound WhatsApp transport action consistently, preserves pairing-request persistence without sending WhatsApp replies, and documents the new config surface.

## Reproduction Assessment

Yes for the PR defects: the diff gives a high-confidence source reproduction for the empty-send early return, the silent composing no-op, and the pairing branch that skips the issuer which persists pending requests. The requested WhatsApp read-only mode itself is a new feature/config surface, not a broken existing contract on current main.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right owner boundary and narrower than relying on generic sendPolicy alone, but the patch needs the guard ordering, blocked-send contract, pairing persistence, docs, and real behavior proof addressed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` can still return `{ messageId: "", toJid }` for empty text with no media before reaching `assertWhatsAppWritable`. Move config/account resolution and the writable assertion before this early return so every outbound send attempt follows the blocked-send contract.
  - confidence: 0.91
- **[P2] Throw for read-only composing attempts:** `extensions/whatsapp/src/inbound/monitor.ts:543-548`
  - body: The listener `sendComposing` closure silently returns in read-only mode while reply/media sends and the listener send API throw a read-only error. Callers that use the blocked-send error as the signal will treat composing indicators differently, so align this closure with the rest of the outbound contract.
  - confidence: 0.88
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:136-141`
  - body: When `dmPolicy: "pairing"` meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer` entirely, so no pending pairing request is upserted for the operator to approve. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
