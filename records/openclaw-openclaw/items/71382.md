## Summary

Keep open. Current main still lacks WhatsApp `readOnly`, and this PR is the active WhatsApp-specific implementation, but PR head still has three correctness blockers in the blocked-send contract and pairing flow plus no real WhatsApp behavior proof.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` config, resolves it into account state, and gates outbound WhatsApp messages, polls, reactions, presence/read receipts, pairing replies, and send APIs with focused tests and config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned read-only implementation that preserves inbound processing and pairing approval, blocks outbound WhatsApp side effects consistently, documents the config, and includes real WhatsApp proof.

## Reproduction Assessment

Yes for the PR review defects: source inspection of PR head shows the empty-send early return before the guard, the read-only composing no-op, and the read-only pairing branch that skips request creation. The requested WhatsApp read-only mode itself is a new feature/config surface rather than a current-main bug reproduction.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right owner boundary and narrower than relying on session sendPolicy alone, but the branch needs the blocked-send contract and pairing persistence fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` still returns `{ messageId: "", toJid }` for empty text with no media before reaching `assertWhatsAppWritable`. Move config/account resolution and the writable assertion before this early return so every outbound send attempt follows the blocked-send contract.
  - confidence: 0.91
- **[P2] Throw for read-only composing attempts:** `extensions/whatsapp/src/inbound/monitor.ts:546-548`
  - body: The listener `sendComposing` closure silently returns in read-only mode while reply/media sends and the listener send API throw a read-only error. Callers that use the blocked-send error as the signal will treat composing indicators differently, so align this closure with the rest of the outbound contract.
  - confidence: 0.88
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:139-143`
  - body: When `dmPolicy: "pairing"` meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer` entirely, so no pending pairing request is upserted for the operator to approve. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
