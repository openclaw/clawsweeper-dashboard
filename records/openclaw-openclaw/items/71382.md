## Summary

Keep open: current main still lacks a WhatsApp read-only transport gate, and the PR branch is a plausible implementation path, but it still has three source-level blockers plus missing real behavior proof before merge.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` config, resolves it into account state, and gates WhatsApp outbound send, presence, receipt, poll, reaction, and pairing paths with focused tests and generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned read-only implementation that preserves inbound processing, blocks outbound WhatsApp side effects consistently, keeps pairing approval possible, updates docs/config metadata, and includes real setup proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows the empty-send early return, silent composing no-op, and read-only pairing branch that skips the issuer. The requested WhatsApp read-only mode itself is a new feature/config surface, not a broken current-main contract.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right owner boundary and narrower than relying on session sendPolicy alone, but the branch needs the blocked-send contract, pairing persistence, docs, and real behavior proof fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` still returns `{ messageId: "", toJid }` for empty text with no media before reaching `assertWhatsAppWritable`.   - confidence: 0.91

  - body: The listener `sendComposing` closure silently returns in read-only mode while reply/media sends and the listener send API throw a read-only error. Callers that use the blocked-send error as the signal will treat composing indicators differently, so align this closure with the rest of the outbound contract.
  - confidence: 0.88
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:139-143`
  - body: When `dmPolicy: "pairing"` meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer` entirely, so no pending pairing request is upserted for the operator to approve. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
