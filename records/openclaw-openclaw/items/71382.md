## Summary

I verified the PR head still has the same three functional blockers called out in prior review comments, and the PR body still provides tests/typecheck only rather than real WhatsApp behavior proof. ## What This Changes

The PR adds WhatsApp channel/account `readOnly` config, resolves it into account state, and gates WhatsApp outbound messages, reactions, polls, presence, read receipts, pairing replies, and send APIs.

## Best Possible Solution

Land a repaired WhatsApp-owned read-only implementation that preserves inbound processing and pairing approval, blocks outbound WhatsApp side effects consistently, documents the config, and includes real WhatsApp proof.

## Reproduction Assessment

Yes for the PR review defects: source inspection of PR head shows the empty-send early return before the guard, the read-only composing no-op, and the read-only pairing branch that skips request creation. The requested WhatsApp read-only mode itself is a new feature/config surface rather than a current-main bug reproduction.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right owner boundary and narrower than relying on session sendPolicy alone, but the branch needs the blocked-send contract, pairing persistence, and real behavior proof before merge.

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` still returns `{ messageId: "", toJid }` for empty text with no media before reaching `assertWhatsAppWritable`.   - confidence: 0.91

  - body: The listener `sendComposing` closure silently returns in read-only mode while reply/media sends and the listener send API throw a read-only error.   - confidence: 0.88

  - body: When `dmPolicy: "pairing"` meets `readOnly`, this branch skips `createChannelPairingChallengeIssuer`, so no pending pairing request is upserted for the operator to approve. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
