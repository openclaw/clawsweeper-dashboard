## Summary

Keep this PR open. Current main still lacks a WhatsApp `readOnly` config and hard transport gate, and the PR is directionally useful, but the proposed patch remains merge-blocked by three concrete read-only contract gaps plus missing user-facing docs/changelog coverage.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` config, resolves it through account state, threads it into the WhatsApp Web listener, gates outbound WhatsApp sends/presence/read-receipts paths, and updates focused WhatsApp tests plus generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that exposes `readOnly` at channel and account scope, resolves it per account, enforces one consistent hard outbound gate across message/media/poll/reaction/typing/presence/read-receipt/send-API paths, preserves inbound processing and pairing request creation while suppressing WhatsApp replies, and documents the feature with a changelog entry.

## Reproduction Assessment

Yes. Current main can be checked by reading the WhatsApp config/schema and transport send paths, and the PR's remaining failures are visible in the compact diff: empty sends bypass the guard, composing indicators use a different blocked-send contract, and read-only pairing DMs skip pending-request creation.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right narrower solution compared with relying on generic `session.sendPolicy`, but the PR needs the guard-ordering, composing-contract, pairing-persistence, docs, and changelog fixes before it is the best maintainable implementation.
