## Summary

Keep open. Current main still misses the documented Discord elevated allowFrom fallback, and this PR addresses a valid linked bug, but the latest head is not ready because it regresses the human `sandbox explain` diagnostics output, puts Discord-specific fallback policy in core instead of the channel plugin seam, and omits the required changelog entry.

## What This Changes

The PR adds a Discord elevated allowFrom fallback resolver, uses it from runtime elevated authorization and `sandbox explain` with account context, and adds focused regression tests.

## Best Possible Solution

Land a rebased fix that implements the Discord fallback through the Discord channel plugin or another extension-owned adapter, shares a generic fallback-resolution path between runtime auth and `sandbox explain` with account context, preserves the current human and JSON diagnostics contract, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. The issue is reproducible from source-level config resolution on current main: docs promise the Discord fallback, runtime elevated auth only calls the plugin fallback hook, the Discord plugin has no adapter, and `sandbox explain` only reads `tools.elevated.allowFrom[channel]`.

## Solution Assessment

No. The intended behavior is correct, but the current PR shape is not the narrowest maintainable fix because it regresses existing diagnostics output and moves Discord-owned fallback policy into core instead of using the existing plugin seam.
