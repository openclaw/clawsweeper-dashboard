## Summary

Keep open: the reported Discord elevated fallback bug is still valid on current main and this PR targets the right behavior, but the branch is conflicting, lacks external real-behavior proof, and still has blocking review findings around elevated-auth ownership and human sandbox diagnostics.

## What This Changes

The PR adds a Discord elevated allowFrom fallback resolver, uses it from runtime authorization and `sandbox explain`, derives account context for diagnostics, and adds focused regression tests.

## Best Possible Solution

Land a rebased fix that implements Discord fallback through the Discord channel plugin `elevated.allowFromFallback` adapter, shares that generic path with `sandbox explain` including account context, preserves current human diagnostics, and includes regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection rather than a live run: the docs promise the Discord fallback, current runtime only asks the plugin fallback hook, the Discord plugin lacks that adapter, and `sandbox explain` reads only explicit elevated allowlists.

## Solution Assessment

No. The intended behavior is correct, but this PR currently special-cases Discord in core and compacts existing human diagnostics; the narrower maintainable solution is a Discord plugin adapter plus preserved formatter behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move Discord fallback into the channel adapter:** `src/auto-reply/reply/reply-elevated.ts:50-55`
  - body: This helper special-cases `discord` in core and reads `cfg.channels.discord` even though `ChannelPlugin` already exposes `elevated.allowFromFallback`. Elevated authorization is plugin-owned, security-sensitive policy, so putting it in core makes the core plugin-aware and easier to drift from Discord account semantics; move the fallback into the Discord channel plugin and keep runtime/diagnostics on the generic hook.
  - confidence: 0.88
- **[P2] Preserve existing human sandbox explain output:** `src/commands/sandbox-explain.ts:349-360`
  - body: The new non-JSON formatter drops current diagnostics such as `mainSessionKey` and the non-main mode hint. In normal `openclaw sandbox explain` use, users troubleshooting `sandbox.mode=non-main` lose the direct session key, so this fallback fix regresses the command's existing troubleshooting workflow.
  - confidence: 0.87
