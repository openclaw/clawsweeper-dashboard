## Summary

Keep open. The PR targets a real documented Discord elevated fallback bug that is still present on current main, but the current head should not merge because it hard-codes Discord fallback policy in core and regresses the human `sandbox explain` diagnostics.

## What This Changes

This PR adds a Discord elevated `allowFrom` fallback resolver, wires it into runtime elevated authorization and `sandbox explain` with account context, and adds focused regression tests.

## Best Possible Solution

Land a rebased fix that puts Discord-specific fallback resolution on the Discord channel plugin adapter, lets runtime auth and `sandbox explain` share the generic fallback path with account context, preserves the existing human diagnostics contract, and adds a user-facing changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure `channels.discord.allowFrom`, enable `tools.elevated`, omit `tools.elevated.allowFrom.discord`, then current main has no Discord `allowFromFallback` adapter for runtime auth and `sandbox explain` reads only the explicit elevated allowlist path.

## Solution Assessment

No. The intended behavior is correct, but this PR is not the best shape because it moves Discord-owned policy into core and removes existing human `sandbox explain` details; the safer path is a Discord plugin adapter plus generic diagnostics reuse.
