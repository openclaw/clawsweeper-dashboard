## Summary

Keep open: current main still has a source-reproducible Discord elevated allowFrom fallback bug, and this external PR is a plausible fix path, but the head is not merge-ready because it puts Discord fallback policy in core, regresses human `sandbox explain` output, and lacks real behavior proof.

## What This Changes

The PR adds a Discord elevated allowFrom fallback resolver used by runtime auth and `sandbox explain`, infers Discord account context for diagnostics, and adds focused unit tests.

## Best Possible Solution

Land a rebased fix that implements Discord fallback through the Discord plugin `elevated.allowFromFallback` adapter, lets runtime auth and `sandbox explain` share the generic path with account context, preserves current human diagnostics, and includes regression tests plus redacted real Discord/gateway proof.

## Reproduction Assessment

Yes. By source inspection on current main, configuring only `channels.discord.allowFrom` while omitting `tools.elevated.allowFrom.discord` leaves runtime fallback undefined because Discord has no `elevated.allowFromFallback` adapter, and `sandbox explain` reads only explicit elevated allowlists.

## Solution Assessment

No. The intended behavior is correct, but this PR hard-codes Discord fallback policy in core and regresses existing human diagnostics; the narrower maintainable solution is a Discord plugin adapter plus preserved formatter behavior.
