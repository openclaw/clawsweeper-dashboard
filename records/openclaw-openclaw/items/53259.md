## Summary

Keep open: this PR targets a valid documented Discord elevated fallback bug, but the current head still has a blocking `sandbox explain` output regression, lacks after-fix real behavior proof, is conflicting against main, and needs security-aware review of the elevated authorization boundary.

## What This Changes

The PR adds Discord `allowFrom` fallback resolution for elevated auth and `sandbox explain`, infers account context for diagnostics, and adds focused unit tests for fallback and explicit-empty override behavior.

## Best Possible Solution

Implement the fallback in the Discord channel `elevated.allowFromFallback` adapter, share that generic path with `sandbox explain`, preserve the existing human output contract, and add real CLI or Discord/gateway proof.

## Reproduction Assessment

Yes from source inspection, though I did not run a live repro in this read-only review. The docs promise the Discord fallback, while current main only reads plugin fallback data and the Discord plugin does not provide that adapter.

## Solution Assessment

No. The target behavior is correct, but this PR hard-codes Discord fallback policy in core and regresses `sandbox explain` human output; the narrower maintainable fix is adapter-owned fallback with the existing output preserved.
