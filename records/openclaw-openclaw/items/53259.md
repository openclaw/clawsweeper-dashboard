## Summary

Keep open: the linked Discord elevated fallback bug is real and this PR is useful, but the current patch still regresses human `sandbox explain` output, hard-codes Discord policy in core, lacks the required changelog entry, and is dirty against current main.

## What This Changes

This PR adds a Discord elevated `allowFrom` fallback resolver, uses it from runtime elevated authorization and `sandbox explain` with account context, and adds focused regression tests.

## Best Possible Solution

Land a rebased or replacement fix that implements Discord fallback through the Discord channel plugin adapter, shares generic fallback resolution with diagnostics, preserves human `sandbox explain` output, and adds regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows the docs promise the fallback, runtime only asks the plugin fallback hook, the Discord plugin has no such adapter, and `sandbox explain` reads only the explicit elevated allowlist path.

## Solution Assessment

No. The intended behavior is correct, but the PR is not the best shape because it puts Discord-owned policy in core and regresses the existing human diagnostics contract; the safer fix is a Discord plugin adapter plus preserved output.
