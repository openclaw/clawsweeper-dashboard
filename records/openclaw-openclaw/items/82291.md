## Summary

Keep open. The report matches a real bootstrap/workspace boundary: inbound messages run workspace bootstrap setup before the agent turn, and current code still relies on successfully removing a stale `BOOTSTRAP.md` before the workspace can stop treating that file as active bootstrap context.

## What This Changes

The issue reports a 2026.5.12 Docker regression where inbound Telegram and Slack messages fail because a legacy `workspace/BOOTSTRAP.md` is treated as an active bootstrap path.

## Best Possible Solution

Make completed workspace setup state authoritative so a leftover legacy `BOOTSTRAP.md` cannot block message processing or keep bootstrap pending, even when cleanup cannot remove the file immediately.

## Reproduction Assessment

No high-confidence end-to-end Docker reproduction was run, but the source path is clear enough for a focused regression test: normal inbound replies call workspace setup, stale bootstrap repair depends on deleting `BOOTSTRAP.md`, and surviving `BOOTSTRAP.md` content can still mark the run as bootstrap-pending.

## Solution Assessment

No, the current implementation is not robust enough because it couples completion state to successful file deletion. A safer fix is to record or honor completed setup independently, skip stale `BOOTSTRAP.md` during bootstrap-file selection/routing once complete, and leave cleanup failure as a warning or doctor repair path.
