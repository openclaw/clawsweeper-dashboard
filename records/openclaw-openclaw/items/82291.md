## Summary

Keep open. Current main still shows the stale workspace-bootstrap boundary: normal inbound replies ensure workspace setup before the agent turn, stale completion repair depends on successfully deleting `BOOTSTRAP.md`, and surviving file content can still keep bootstrap routing active.

## What This Changes

The issue asks OpenClaw to stop failing inbound messages after upgrading when a legacy immutable `workspace/BOOTSTRAP.md` remains.

## Best Possible Solution

Make completed workspace setup state authoritative, tolerate stale `BOOTSTRAP.md` cleanup failure, and keep leftover legacy bootstrap content out of normal inbound routing once completion evidence exists.

## Reproduction Assessment

Yes for a focused source-level reproduction: simulate an upgraded workspace with completion evidence and a `BOOTSTRAP.md` that cannot be removed, then verify workspace setup and normal bootstrap routing do not fail or remain pending. I did not run the full Docker upgrade flow in this read-only review.

## Solution Assessment

No, the current implementation is not robust because it couples setup completion to successful file deletion. The safer fix is to persist or honor completion independently, skip stale `BOOTSTRAP.md` for normal runs after completion evidence, and leave physical cleanup to a warning or doctor repair path.
