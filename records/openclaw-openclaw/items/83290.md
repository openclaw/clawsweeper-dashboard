## Summary

Keep open: current main has the source-level inconsistency the report describes, but the requested fix would expand startup auto-migration into channel auth state that current docs/tests describe as doctor-owned. This needs maintainer product/owner judgment before a conservative automated fix.

## What This Changes

The issue requests that `autoMigrateLegacyState` treat detected channel legacy migration plans as actionable and apply them in its returned migration result.

## Best Possible Solution

Align the contract explicitly: either keep channel credential migrations doctor-only and avoid collecting them in the auto startup helper, or update docs/tests and `autoMigrateLegacyState` to run channel plans with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: with `channelPlans.hasLegacy` true and no legacy sessions or agent dir, `autoMigrateLegacyState` returns before any channel plan runner is called; I did not execute a filesystem repro because this review is read-only.

## Solution Assessment

Is this the best way to solve the issue? Unclear: including channel plans in the auto path is mechanically narrow, but it conflicts with the documented doctor-only migration boundary for channel auth, so maintainers should decide the intended contract first.
