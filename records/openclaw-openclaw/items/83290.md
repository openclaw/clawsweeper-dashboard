## Summary

Keep open: current main and the latest release still have the source-level skip described in the report, but resolving it requires deciding whether channel credential and pairing migrations may run at startup despite the current doctor-only documentation.

## What This Changes

The issue asks for channel legacy migration plans detected by `autoMigrateLegacyState` to be included in the auto-migration decision and returned migration result.

## Best Possible Solution

Align the contract explicitly: either keep channel migrations doctor-only and avoid treating them as startup auto-migration work, or update `autoMigrateLegacyState`, docs, and regression tests so channel plans are deliberately applied at startup.

## Reproduction Assessment

Yes, from source inspection: with `channelPlans.hasLegacy` true and no legacy sessions or agent dir, `autoMigrateLegacyState` returns before calling the channel plan runner. I did not run a filesystem repro because this review is read-only.

## Solution Assessment

Unclear: including `channelPlans` in the auto path is mechanically narrow, but it changes the documented startup boundary for channel auth and pairing state, so maintainer product/owner judgment should come first.
