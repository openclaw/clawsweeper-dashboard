## Summary

Keep open: current main still detects channel legacy migration plans but the startup auto-migration path ignores channel-only detections. The requested behavior conflicts with the documented narrower startup scope for sessions and agent-dir migration, so maintainers should decide the intended contract before a repair is queued.

## What This Changes

Request that `autoMigrateLegacyState` include detected channel legacy migration plans in its early-return decision and apply them.

## Best Possible Solution

Make the migration contract explicit: either keep channel credential migrations doctor-only and avoid treating them as startup auto work, or update docs/tests and `autoMigrateLegacyState` to run channel plans intentionally.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: with `channelPlans.hasLegacy` true and no legacy sessions or agent dir, `autoMigrateLegacyState` returns before any channel plan runner is called.

## Solution Assessment

Is this the best way to solve the issue? Unclear: including channel plans in the auto path is mechanically narrow, but it changes the documented startup migration boundary for channel auth state.
