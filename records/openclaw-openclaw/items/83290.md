## Label Justifications

- `P2`: The source-level issue affects legacy state migration behavior with limited blast radius and needs normal maintainer prioritization rather than emergency handling.
- `impact:session-state`: The affected migration path decides whether legacy session/agent state is migrated or left behind during startup.
- `impact:auth-provider`: The skipped channel plans include auth and pairing state such as WhatsApp credentials and Telegram allowFrom stores.

## Summary

Keep open: current main still has the source-level mismatch, but automatically applying channel credential/pairing migrations at startup conflicts with the documented doctor-only boundary and needs maintainer product/owner judgment before a fix is queued.

## What This Changes

The issue asks for `autoMigrateLegacyState` to include detected channel legacy migration plans in its early-return decision and returned migration results.

## Best Possible Solution

Align the migration contract explicitly: either keep channel credential migrations doctor-only and avoid collecting them for the startup helper, or update docs/tests and `autoMigrateLegacyState` to run channel plans with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: when `channelPlans.hasLegacy` is true and both sessions and agent-dir legacy flags are false, `autoMigrateLegacyState` returns before any channel plan runner is called; this read-only review did not execute a filesystem repro.

## Solution Assessment

Is this the best way to solve the issue? Unclear: including channel plans in the auto path is mechanically narrow, but it changes the documented doctor-only boundary for channel auth state, so maintainers should decide the intended contract first.
