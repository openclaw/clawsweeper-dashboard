## Summary

Keep this PR open as a candidate fix for the Windows ACPX spawn failure, but it is not merge-ready: the patch can bypass the fail-closed Windows wrapper policy, can hide acpx's built-in installed-package launch path, and still lacks proof from the patched Windows ACPX session path.

## What This Changes

The PR wraps the ACPX agent registry on Windows to rewrite command wrappers through the plugin SDK Windows spawn helpers before acpx receives the agent command.

## Best Possible Solution

Keep acpx's built-in launch planning canonical, fix Windows wrapper resolution or rejection at the ACPX launch layer, and require redacted Windows ACPX session proof before merge.

## Reproduction Assessment

Yes by source inspection: current main delegates ACPX session startup to upstream acpx, Node rejects direct .cmd/.bat spawn without shell, and the linked reports include Windows spawn EINVAL logs. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No. This is a plausible mitigation, but the best fix should preserve acpx's built-in launch planner and fail closed or resolve wrappers at the launch layer instead of returning rejected wrapper commands to acpx.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not fall back after rejecting a wrapper:** `extensions/acpx/src/runtime.ts:707-709`
  - body: The catch block returns the original command even when `applyWindowsSpawnProgramPolicy({ allowShellFallback: false })` rejected an unresolved wrapper. That hands the wrapper back to acpx, whose Windows spawn helper can use shell fallback for `.cmd`/`.bat`, so the intended fail-closed no-shell policy is bypassed.
  - confidence: 0.9
- **[P1] Preserve acpx built-in launch resolution:** `extensions/acpx/src/runtime.ts:748`
  - body: Wrapping `options.agentRegistry` changes the exact built-in command string before `BaseAcpxRuntime` sees it. In acpx 0.10.0 those exact strings are how `resolveBuiltInAgentLaunch` chooses the installed adapter package or current-Node npm bridge, so Windows built-ins can fall back to package-manager command strings instead of the packaged launch path.
  - confidence: 0.87
