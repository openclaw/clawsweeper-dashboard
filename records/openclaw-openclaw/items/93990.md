- `P1`: The linked bug makes the Windows ACP runtime path unusable for affected users, which is a broken agent workflow.

## Summary

Keep this PR open, but do not merge it as-is: it is a focused Windows ACPX bug-fix attempt, yet the patch can still return unsafe wrapper commands after the Windows spawn policy rejects them, and the supplied proof does not exercise the patched ACPX runtime spawn path.

## What This Changes

The branch wraps the ACPX agent registry on Windows so resolved agent command strings are rewritten through `openclaw/plugin-sdk/windows-spawn` before being passed to the upstream acpx runtime.

## Best Possible Solution

Keep one safe Windows spawn normalization path that resolves or fails closed for `.cmd`, `.bat`, and `.ps1` wrappers before acpx spawn, with focused ACPX tests and redacted Windows `sessions_spawn` proof before landing.

## Reproduction Assessment

Yes by source inspection for the PR defect: `applyWindowsSpawnProgramPolicy(... allowShellFallback: false)` throws for unresolved wrappers, but the PR catches that path and returns the original command. I did not run a live Windows gateway spawn in this read-only review.

## Solution Assessment

No. Registry string rewriting is a plausible mitigation, but the best fix must preserve the SDK fail-closed contract and cover PowerShell shims/generated wrapper paths instead of silently falling back to unsafe command strings.
