Full review comments:

- **[P1] Resolve the Claude executable shim too:** `extensions/acpx/src/runtime.ts:748`
  - body: Wrapping the registry only changes the adapter process command. acpx still resolves `claude` on Windows and sets `CLAUDE_CODE_EXECUTABLE`; https://github.com/openclaw/openclaw/issues/95310 shows that can be `claude.cmd`, and `claude-agent-acp` prefers that env path during `session/new`, so the same `spawn EINVAL` can remain.
  - confidence: 0.9
- **[P1] Preserve ACPX built-in launch resolution:** `extensions/acpx/src/runtime.ts:748`
  - body: `acpx@0.10.0` matches built-in agents by exact fallback command before selecting installed/package-exec launch. After this wrapper rewrites `npx ...` into `node ...npx-cli.js`, `BaseAcpxRuntime` no longer sees the exact built-in command, so packaged or offline installs can lose the ACPX-owned launch path.
  - confidence: 0.88
- **[P1] Do not return rejected wrappers to ACPX:** `extensions/acpx/src/runtime.ts:707-709`
  - body: `applyWindowsSpawnProgramPolicy({ allowShellFallback: false })` throws for unresolved wrappers, but this catch returns the original command. That passes the rejected wrapper to acpx, whose Windows spawn options use shell fallback for `.cmd/.bat`, bypassing the no-shell policy.
  - confidence: 0.89
