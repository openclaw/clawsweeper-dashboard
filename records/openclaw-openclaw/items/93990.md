## Summary

Keep open: this is a focused fix for a real Windows ACPX bug, but it is not merge-ready because it only rewrites the outer registry command, can still leave the `CLAUDE_CODE_EXECUTABLE` `.cmd` failure path intact, and bypasses the fail-closed wrapper policy on resolver failure.

## What This Changes

The PR adds a Windows-only ACPX agent-registry wrapper in `extensions/acpx/src/runtime.ts` that rewrites resolved agent command strings through the plugin SDK Windows spawn resolver before constructing the upstream ACPX runtime.

## Best Possible Solution

Fix the Windows ACPX launch at the ACPX-owned planner boundary: preserve built-in registry identity, resolve or reject unsafe `CLAUDE_CODE_EXECUTABLE` wrappers, fail closed for unresolved wrappers, and prove a real Windows ACPX Claude session after the patch.

## Reproduction Assessment

Yes by source inspection and linked Windows logs, though I did not run a live Windows gateway in this read-only review. Current main delegates ACPX session startup to acpx, and the dependency path can inject a `.cmd` Claude executable that the Claude ACP adapter prefers.

## Solution Assessment

No. This is a plausible mitigation for the outer adapter command, but the best fix should preserve acpx's built-in launch ownership and make the Claude executable path safe or explicitly fail closed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve the Claude executable shim too:** `extensions/acpx/src/runtime.ts:748`
  - body: Wrapping the registry only changes the adapter process command. acpx still resolves `claude` on Windows and injects it as `CLAUDE_CODE_EXECUTABLE`, and the newer duplicate report shows that value can be `claude.cmd`; `claude-agent-acp` then prefers that path for Claude Code. The patched path can therefore still hit the same `spawn EINVAL` during `session/new`, so the fix needs to make that executable path safe or fail closed in the ACPX launch layer.
  - confidence: 0.9
- **[P1] Preserve ACPX built-in launch resolution:** `extensions/acpx/src/runtime.ts:748`
  - body: By rewriting the exact built-in command string before `BaseAcpxRuntime` sees it, this wrapper prevents acpx from matching the built-in `claude`/`codex` fallback command and selecting its installed-adapter or current-Node npm CLI launch path. That can push packaged or offline Windows installs back through a package-manager command string instead of the pinned ACPX-owned launch plan.
  - confidence: 0.88
- **[P1] Do not return rejected wrappers to ACPX:** `extensions/acpx/src/runtime.ts:707-709`
  - body: `applyWindowsSpawnProgramPolicy({ allowShellFallback: false })` throws when a `.cmd`/`.bat` wrapper cannot be resolved safely, but this catch returns the original command. That hands the rejected wrapper to acpx, whose Windows spawn options can use shell fallback for batch files, bypassing the no-shell policy this patch tried to apply.
  - confidence: 0.89
