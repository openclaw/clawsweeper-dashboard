Keep open: the PR targets a real session-stall bug and the chosen recovery owner boundary is plausible, but the central fix is ineffective under the production default timing because the heartbeat returns before the new idle-zombie predicate can run.

## What This Changes

Adds idle zombie embedded-run detection to the diagnostic heartbeat, plus focused tests for reclaiming leaked ACTIVE_EMBEDDED_RUNS entries.

## Best Possible Solution

Keep the existing abort/drain recovery path, but make idle zombie activity keep the heartbeat scan alive until the abort threshold or run recovery before the early return, with a default-threshold regression test.

## Reproduction Assessment

Yes at source level: with default timing, an idle queueDepth=0 zombie is neither recent activity nor open work by the time stuckSessionAbortMs is reached, so the heartbeat returns before scanning it. I did not run tests because this review is read-only.

## Solution Assessment

No: the recovery coordinator path is the right layer, but the detection is placed behind a gate that excludes the default stale idle case. The narrow fix is to include this state in the heartbeat work predicate or move the recovery scan before the early return.
