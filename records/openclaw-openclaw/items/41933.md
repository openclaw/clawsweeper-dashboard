## Summary

Current main and the shipped v2026.5.12 release solve the reported Webchat gap: configured agents are exposed through the chat agent filter, and switching to an agent with no stored session falls back to that agent's main session.

## What This Changes

The issue requests that Webchat expose all configured agents, not only agents with existing session-store rows, in the session selection flow.

## Best Possible Solution

Keep the shipped Control UI agent-filter design as the canonical solution and close the older duplicate workaround threads.

## Reproduction Assessment

No against current main. Source and regression tests show configured agents are available through the agent filter, and selecting an agent without stored session rows falls back to that agent's main session.

## Solution Assessment

Yes. The shipped design is more maintainable than adding synthetic configured-agent rows to the session picker because it separates agent choice from per-agent session selection while preserving the expected fallback.
