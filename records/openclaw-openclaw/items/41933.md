## Label Justifications

- `P2`: The issue affected a bounded Webchat agent selection workflow without making the core runtime unusable.
- `impact:session-state`: The reported failure was about session selector state omitting configured agents with no saved session.

## Summary

Close as implemented. Current main and the v2026.5.20 release now provide an agent filter populated from configured agents, and selecting an agent with no stored sessions falls back to that agent's main session.

## What This Changes

Show all configured Webchat agents in the chat selection flow, including agents without existing session rows.

## Best Possible Solution

Keep the shipped split: configured agents are selected through the agent filter, while the session picker stays scoped to the active agent and stored session rows.

## Reproduction Assessment

No, not against current main. The current source and regression test cover an `alpha`/`beta` setup where only `alpha` has a stored session, and selecting `beta` switches to `agent:beta:main`.

## Solution Assessment

Yes. The shipped solution uses `agents.list` for agent choice and keeps `sessions.list` store-backed, which is the narrowest maintainable boundary for this UI behavior.
