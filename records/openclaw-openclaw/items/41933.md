- `P2`: The report affected a bounded Webchat agent/session selection workflow with normal user impact.
- `impact:session-state`: The bug was about selecting the correct configured agent/session target when no stored session row existed yet.

## Summary

Close as implemented. Current main and the latest release now let Webchat select configured agents even when they have no stored session by rendering a global agent selector from `agents.list` and falling back to that agent's main session.

## What This Changes

Show all configured Webchat agents in the selector path, including agents that do not yet have stored sessions.

## Best Possible Solution

Keep the shipped split-selector design and close the older duplicate implementation attempts as superseded by the release.

## Reproduction Assessment

No current-main failure remains. The current source and regression test cover the reported alpha/beta scenario where beta is configured but has no stored session, and selecting beta now targets `agent:beta:main`.

## Solution Assessment

Yes. Separating configured-agent selection from stored-session selection is the maintainable fix because `agents.list` owns available agents while `sessions.list` remains store-backed.
