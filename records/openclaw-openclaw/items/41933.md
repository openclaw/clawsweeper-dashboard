## Label Justifications

- `P2`: The original bug affected WebChat multi-agent selection but had limited blast radius and is now implemented.
- `impact:session-state`: The issue was about session picker state omitting configured agents without existing session rows.

## Summary

Current main already solves the reported WebChat problem with a dedicated configured-agent filter before the scoped session picker, including fallback to an agent's main session when no stored session row exists; the implementation is also present in the latest release tag.

## What This Changes

The issue requests that WebChat let users select all configured agents even when some agents do not yet have stored sessions.

## Best Possible Solution

Keep the shipped agent-filter-before-session-picker implementation as the canonical solution; open a new issue only for a distinct UX request beyond selecting configured sessionless agents.

## Reproduction Assessment

No against current main. Source and regression coverage show selecting a configured beta agent with no stored session row falls back to `agent:beta:main`.

## Solution Assessment

Yes. The current implementation separates agent selection from per-agent session selection, which is cleaner than synthesizing configured agents into the session picker itself.
