## Summary

Current main still lacks the requested operator-owned paired-device naming workflow: it can display client-supplied displayName metadata or the deviceId, but there is no durable user label, rename command, Gateway method, or Control UI rename action.

## What This Changes

Request to let users assign and view human-friendly names for paired devices in the web UI and openclaw devices CLI instead of relying on cryptographic device IDs.

## Best Possible Solution

Add an additive paired-device label workflow with durable operator-owned storage, Gateway protocol/method support, CLI and Control UI rename actions, and clear precedence between label, client displayName, and deviceId.

## Reproduction Assessment

Yes, at source level: current main lists paired devices with client displayName metadata or deviceId and has no CLI, Gateway, or UI path to set a durable operator-owned label. This is a feature gap rather than a failing existing contract.

## Solution Assessment

Yes, the requested additive rename workflow is the right shape; simply reusing the existing displayName field would be the wrong fix because reconnect metadata can refresh that field.
