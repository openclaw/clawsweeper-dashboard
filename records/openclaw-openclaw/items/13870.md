## Summary

Keep open: current main still lacks durable operator-owned paired-device labels and any paired-device rename path. Prior maintainer discussion also keeps this as the canonical rename workflow, separate from broader #23916 metadata display.

## What This Changes

Request a user-assignable human-friendly name for paired devices in `openclaw devices list` and the Control UI.

## Best Possible Solution

Introduce an additive paired-device label stored separately from client `displayName`, with protocol/schema/server/CLI/UI/docs/tests and clear precedence rules.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence proof of the current gap: paired-device lists can display client `displayName`, but current main has no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive `device.pair.rename` workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
