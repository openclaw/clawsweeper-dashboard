## Summary

Current main still lacks a durable operator-owned paired-device name and an end-to-end rename workflow; existing `displayName` data is client-owned metadata, while the related items only cover broader metadata or timestamps.

## What This Changes

Add durable user-assigned names for paired devices and expose rename and display support through Gateway, CLI, and Control UI.

## Best Possible Solution

Add a distinct durable operator-owned label with precedence over client `displayName`, expose one additive and authorized paired-device rename method, and use it consistently from the CLI and Control UI.

## Reproduction Assessment

Yes, at source level. Current main has no operator-owned label field or rename operation across persistence, protocol, CLI, and Control UI, so the missing capability is directly verifiable without a runtime failure reproduction.

## Solution Assessment

Yes. A separate operator-owned label is safer and more maintainable than repurposing client `displayName`, because the latter is reconnect-refreshed metadata with a different owner and lifecycle.
