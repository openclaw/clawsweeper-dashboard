## Summary

Keep open: current main and v2026.6.8 still lack a durable operator-owned paired-device label and any paired-device rename flow. The existing `displayName` is client-supplied metadata and reconnect paths can refresh it, so it is not a safe substitute for the requested user label.

## What This Changes

The issue requests human-friendly operator-assigned names for paired devices in the CLI and web UI.

## Best Possible Solution

Add one operator-owned paired-device label field with explicit label-over-client-displayName-over-deviceId precedence, exposed through Gateway protocol, CLI, Control UI, and docs with focused authorization coverage.

## Reproduction Assessment

Yes, at source level: current main has no paired-device rename method, command, UI action, or operator-owned label field. This is a missing feature rather than a runtime failure needing live reproduction.

## Solution Assessment

Yes. A separate operator-owned field with explicit display precedence is the narrow maintainable solution; reusing client `displayName` would create competing writers from reconnect metadata.
