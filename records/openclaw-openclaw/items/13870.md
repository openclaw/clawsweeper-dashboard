## Summary

Keep open: current main and v2026.6.8 still have client-supplied displayName/deviceId display, but no durable operator-owned paired-device label or rename workflow; https://github.com/openclaw/openclaw/issues/23916 is broader metadata display rather than the canonical rename feature.

## What This Changes

Request to let operators assign and display human-friendly names for paired devices in the CLI and Control UI instead of relying on hash IDs or reconnect metadata.

## Best Possible Solution

Add one additive operator-owned paired-device label with explicit label-over-client-displayName-over-deviceId precedence, exposed through Gateway protocol, CLI, Control UI, focused authz coverage, and docs.

## Reproduction Assessment

Yes, at source level: current storage, protocol, Gateway handlers, CLI, Control UI, and docs have no durable operator-owned paired-device label or rename action. This is a missing feature rather than a failing runtime behavior that needs live reproduction.

## Solution Assessment

Yes. A separate operator-owned label field with explicit display precedence is the narrow maintainable shape; reusing client displayName would create two writers and allow reconnect metadata to overwrite the operator preference.
