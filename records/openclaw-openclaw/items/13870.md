## Summary

Keep open: current main still lacks the requested durable, operator-owned paired-device naming workflow. The code only stores and displays client-supplied `displayName` metadata or the device id, and there is no paired-device label field, Gateway rename method, CLI rename command, or Control UI rename action; the related open metadata-display issue and PRs do not replace this rename workflow.

## What This Changes

The issue requests durable human-friendly names for paired devices in the web UI and `openclaw devices list`, including a CLI rename path.

## Best Possible Solution

Add an additive paired-device label stored separately from client reconnect metadata, expose it through Gateway protocol, server authorization, CLI, and Control UI, document display precedence, and keep broader metadata display tracked separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus an additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
