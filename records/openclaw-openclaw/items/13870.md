## Summary

Keep open: current main and the latest release still lack a durable operator-owned paired-device label, a paired-device rename protocol/schema/handler, a CLI rename command, and a Control UI rename action. The broader device metadata discussion at https://github.com/openclaw/openclaw/issues/23916 is related but does not replace this naming workflow.

## What This Changes

The issue requests human-friendly names for paired devices, settable by users and shown in the CLI and Control UI instead of relying on long device IDs.

## Best Possible Solution

Add an additive paired-device label workflow with a durable operator-owned label separate from reconnect displayName, expose it through Gateway protocol, CLI, and Control UI, and document display precedence while keeping broader metadata display separate.

## Reproduction Assessment

Yes, for source-level absence of the requested capability: current main has no durable operator label, paired-device rename method/schema/handler, CLI command, or Control UI action. This is a feature gap rather than a failing bug reproduction.

## Solution Assessment

Yes, the requested direction is maintainable if implemented as a separate operator-owned label instead of overloading reconnect-owned displayName. The narrow path is additive protocol, storage, CLI, UI, docs, and authorization coverage.
