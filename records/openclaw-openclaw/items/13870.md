## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and any rename path in storage, Gateway protocol, server handlers, CLI, or Control UI. The related metadata-display issue and open display-only PR cover useful identification details, but they do not supersede the user-settable naming workflow.

## What This Changes

Request user-assignable human-friendly names or labels for paired devices in the CLI and Control UI, backed by durable storage and a rename path.

## Best Possible Solution

Add an additive paired-device label stored separately from client `displayName`, expose it through Gateway protocol, CLI, and Control UI, and document display precedence without folding in broader metadata display work.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus an additive `device.pair.rename` workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
