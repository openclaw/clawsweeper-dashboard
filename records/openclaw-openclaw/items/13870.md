## Summary

Keep open: current main and v2026.6.8 still lack a durable operator-owned paired-device label and rename workflow; the existing `displayName` is client metadata that reconnects can refresh, and the related metadata issue does not replace this naming workflow.

## What This Changes

Request durable human-friendly names for paired devices in the CLI and Control UI, with a user-settable rename flow.

## Best Possible Solution

Add one optional operator-owned paired-device label with explicit label-over-client-displayName-over-deviceId precedence, exposed through Gateway protocol, CLI, Control UI, and docs with focused authorization tests.

## Reproduction Assessment

Yes, at source level: storage, protocol, Gateway handlers, CLI, Control UI, and docs have no durable operator-owned paired-device label or rename action. This is a missing feature rather than a failing runtime behavior needing live reproduction.

## Solution Assessment

Yes. A separate operator-owned field with explicit display precedence is the narrowest maintainable solution; reusing client `displayName` would create two writers and allow reconnect metadata to overwrite the operator preference.
