## Summary

Keep open: current main still lacks a durable, operator-owned paired-device label/rename workflow. The existing code only stores and displays client-supplied `displayName` metadata or the raw device id, while related display work remains open and does not implement user-settable labels.

## What This Changes

The request is to add human-friendly paired-device names that users can assign and see in the CLI and Control UI.

## Best Possible Solution

Add an additive paired-device label stored separately from client reconnect metadata, expose it through Gateway protocol, CLI, and Control UI, and document display precedence while keeping broader metadata display separate.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable user label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
