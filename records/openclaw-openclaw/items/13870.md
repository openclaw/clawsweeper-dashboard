## Summary

Keep open: this issue is still necessary because current main and v2026.6.8 still lack a durable operator-owned paired-device label and any rename flow across storage, Gateway protocol, CLI, Control UI, or docs. The existing client `displayName` is not a safe substitute because reconnect metadata can refresh it.

## What This Changes

Request a human-friendly, user-assignable name for paired devices in `openclaw devices list` and the Control UI Devices/Nodes panel.

## Best Possible Solution

Add one operator-owned paired-device label field with explicit label-over-client-displayName-over-deviceId precedence, exposed through Gateway protocol, CLI, Control UI, and docs with focused authorization coverage.

## Reproduction Assessment

Yes, at source level: current main has no operator-owned paired-device label or rename flow across storage, protocol, Gateway handlers, CLI, UI, or docs. This is a missing feature rather than a live runtime failure needing reproduction.

## Solution Assessment

Yes. A separate operator-owned field with explicit display precedence is the narrowest maintainable solution; reusing client `displayName` would create two writers and let reconnect metadata overwrite the operator preference.
