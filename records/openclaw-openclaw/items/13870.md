## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and rename workflow. The related metadata-display issue and open CLI PR cover broader `devices list` presentation, not editable paired-device names.

## What This Changes

Request adding human-friendly paired-device labels, including a CLI rename command and Control UI display/action, so users do not have to identify devices by cryptographic IDs.

## Best Possible Solution

Add an additive paired-device label stored separately from reconnect metadata, expose it through Gateway protocol/server methods, CLI rename/list, and Control UI, and keep broader metadata display tracked separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label field or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
