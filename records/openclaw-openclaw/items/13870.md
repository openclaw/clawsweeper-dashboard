## Summary

Keep open: current main still lacks an operator-owned paired-device label and any rename workflow across storage, Gateway protocol, server handlers, CLI, and Control UI. The related metadata-display issue and open display-only PRs are useful context, but they do not replace this user-settable naming request.

## What This Changes

The issue asks for human-friendly, user-settable labels for paired devices in `openclaw devices list` and the web UI instead of relying on device IDs or client-supplied metadata.

## Best Possible Solution

Add an additive paired-device label stored separately from client `displayName`, expose it through Gateway protocol, CLI, and Control UI, and document display precedence while keeping broader metadata-display work separate.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus an additive `device.pair.rename` workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
