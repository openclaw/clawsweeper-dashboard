## Summary

Keep open: current main still does not provide a durable, operator-owned paired-device name/label workflow. The source, protocol, CLI, Control UI, docs, tests, and issue discussion all point to the same remaining gap, and the maintainer comment already defines this as the canonical editable paired-device naming request rather than broader metadata display work.

## What This Changes

The issue asks for user-settable human-friendly names for paired devices, displayed in the web UI and `openclaw devices list`, with a CLI command such as `openclaw devices rename`.

## Best Possible Solution

Add an additive paired-device label feature: persist an operator-owned label separate from client `displayName`, define display precedence as label then client displayName then deviceId, expose a typed Gateway rename method with authorization, add CLI and Control UI rename actions, and document the behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives a high-confidence gap because storage, protocol, server methods, CLI, Control UI, docs, and tests still lack the editable paired-device label workflow.

## Solution Assessment

Yes: a separate operator-owned label is the narrowest maintainable direction because client `displayName` is reconnect metadata and can be overwritten, while `node.rename` targets the separate legacy node-pairing store.
