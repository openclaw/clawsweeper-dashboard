## Summary

Keep open: current main still lacks a durable, operator-owned paired-device label or rename workflow. The existing code stores and displays client-supplied `displayName` metadata or the raw device id, and the related display-only issues/PRs do not replace this rename feature.

## What This Changes

Request a human-friendly paired-device naming workflow across paired-device storage, `openclaw devices list`, and the Control UI Devices/Nodes panel.

## Best Possible Solution

Add an additive paired-device label stored separately from client reconnect metadata, expose it through Gateway protocol, server authorization, CLI, and Control UI, and document display precedence while keeping broader metadata display tracked separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
