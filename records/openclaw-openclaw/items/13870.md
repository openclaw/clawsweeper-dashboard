## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and there is no paired-device rename workflow in storage, Gateway protocol, CLI, Control UI, or docs. Related items cover broader metadata display or closed unmerged attempts, so this remains the canonical rename/label feature.

## What This Changes

Add human-friendly, user-settable names for paired devices in the device list, CLI, and Control UI.

## Best Possible Solution

Add one optional operator-owned paired-device label with explicit label-over-client-displayName-over-deviceId precedence, exposed through the Gateway protocol, CLI, and Control UI with focused authz coverage and docs.

## Reproduction Assessment

Yes, at source level: current storage, protocol, CLI, Control UI, and docs have no durable operator-owned paired-device label or rename action. This is a missing feature rather than a failing runtime behavior needing live reproduction.

## Solution Assessment

Yes. A separate operator-owned field with explicit display precedence is the narrowest maintainable solution; reusing client `displayName` would create two writers and allow reconnect metadata to overwrite the operator preference.
