## Summary

Keep open: current main still lacks the durable, operator-owned paired-device naming workflow requested here. The existing surfaces can show client-supplied `displayName` metadata or the raw device id, while the related open display work does not implement a rename/label feature.

## What This Changes

Request adds human-friendly user-assigned names or labels for paired devices in CLI and Control UI device lists.

## Best Possible Solution

Add an additive paired-device label stored separately from client reconnect metadata, expose it through Gateway protocol, CLI, and Control UI, and document display precedence while keeping broader metadata display tracked separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
