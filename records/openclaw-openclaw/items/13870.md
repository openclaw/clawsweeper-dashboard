## Summary

Keep open: current main still lacks the requested durable, operator-owned paired-device naming workflow. The existing code stores and displays client-supplied `displayName` metadata or the raw device id, but there is no separate label field, paired-device rename method, CLI command, or Control UI action.

## What This Changes

Request a human-friendly name or label for paired devices that can be set by users and shown in the CLI and web UI instead of relying on hash-like device IDs.

## Best Possible Solution

Add an additive paired-device label stored separately from reconnect metadata, expose it through Gateway protocol/server authorization, CLI, and Control UI, and document/test display precedence.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
