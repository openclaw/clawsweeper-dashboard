## Summary

Keep open: current main and the latest shipped tag still lack the durable operator-owned paired-device label and rename workflow needed for human-friendly device names. Related device-list metadata work is useful context, but it does not replace the requested end-to-end naming feature.

## What This Changes

Request to let users assign and display human-friendly names for paired devices in the CLI and Control UI.

## Best Possible Solution

Add an additive paired-device label workflow with a durable operator-owned label separate from reconnect displayName, expose it through Gateway protocol, CLI, and Control UI, and document display precedence and authorization behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main has no durable label, no device.pair.rename schema/handler/descriptor, no CLI rename command, and no Control UI action. This is a feature gap rather than a failing bug.

## Solution Assessment

Yes. A separate operator-owned label is the maintainable path because current displayName is reconnect metadata and can be refreshed by clients; the implementation should stay separate from broader metadata-display work.
