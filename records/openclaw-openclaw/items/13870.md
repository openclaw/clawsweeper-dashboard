## What This Changes

The issue requests human-friendly names for paired devices in the CLI and web UI, including a rename workflow instead of relying on opaque device ids or client-supplied metadata.

## Best Possible Solution

Land a narrow additive operator-owned paired-device label workflow through Gateway RPC, CLI, Control UI, docs, and authorization coverage, using https://github.com/openclaw/openclaw/pull/94517 as the current candidate or replacing it if maintainers close that path.

## Reproduction Assessment

Yes. At source level, current main has no operator-owned paired-device label field, no device.pair.rename method/schema, no devices rename command, and no Control UI rename action.

## Solution Assessment

Yes. A separate operator-owned label with explicit precedence over client displayName is the narrow maintainable solution; reusing displayName would create competing client reconnect writes.
