## Summary

Keep open: current main still lacks the requested paired-device naming workflow. The code only has client-supplied `displayName` metadata or the raw `deviceId`; there is no durable operator-owned label, `device.pair.rename` method, CLI rename command, or Control UI rename action. The related #23916 remains broader metadata display work, while this issue is the canonical paired-device rename request.

## What This Changes

Add durable human-friendly labels for paired devices, settable from CLI and visible in the CLI and Control UI paired-device lists.

## Best Possible Solution

Add an additive paired-device label workflow with label precedence, protocol schema/handler support, CLI and Control UI rename actions, docs, and regression coverage while leaving `node.rename` as a separate node surface.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence proof of the current gap: the paired-device list can display `displayName`, but current main has no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive `device.pair.rename` workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.

Add an additive `device.pair.rename` protocol schema, server handler, method listing, and method scope using the existing device-pairing authorization pattern.
