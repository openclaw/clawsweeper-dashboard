## Summary

Keep open: current main still lacks the durable, operator-owned paired-device label and rename workflow requested here. Existing surfaces show client-supplied `displayName` or the raw `deviceId`, and the related metadata-display issue/PR do not implement an editable paired-device label.

## What This Changes

Request to add human-friendly, operator-assigned labels for paired devices in the CLI and Control UI, including a rename workflow.

## Best Possible Solution

Add an additive paired-device label stored separately from reconnect metadata, expose it through Gateway protocol, CLI, and Control UI, and document label/display precedence while keeping broader metadata display separate.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label field or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
