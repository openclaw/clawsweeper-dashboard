## Summary

Keep open: current main still lacks the durable, operator-owned paired-device label and rename workflow requested here. Existing surfaces can display client-supplied `displayName` or a raw `deviceId`, while related metadata-display work at https://github.com/openclaw/openclaw/issues/23916 and https://github.com/openclaw/openclaw/pull/57449 does not implement user-managed paired-device names.

## What This Changes

Add human-friendly user-assigned names for paired devices in the web UI and `openclaw devices list`, with a CLI rename workflow.

## Best Possible Solution

Add an additive paired-device label stored separately from reconnect metadata, expose it through Gateway protocol, CLI, and Control UI, and document label/display precedence while keeping broader metadata display tracked separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label field or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
