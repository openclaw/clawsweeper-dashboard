## What This Changes

Request a human-friendly, user-settable paired-device label shown in the CLI and web UI instead of relying on cryptographic device IDs.

## Best Possible Solution

Land a narrow additive operator-owned paired-device label workflow with Gateway RPC, CLI, Control UI, docs, and authorization/log-sanitization coverage, or keep the open PR from closing this issue if it remains a narrower CLI-only slice.

## Reproduction Assessment

Yes. At source level, current main has no operator-owned paired-device label field, no device.pair.rename method, no devices rename command, and the Control UI still renders displayName or deviceId.

## Solution Assessment

Yes. A separate operator-owned label with explicit precedence over client displayName is the narrow maintainable solution; reusing displayName would create competing client reconnect writes.
