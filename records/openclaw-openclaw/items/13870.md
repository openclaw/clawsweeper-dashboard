## Summary

Current main still lacks the requested operator-owned paired-device rename workflow, and live discussion keeps this issue as the canonical request rather than the broader metadata-display issue. The source boundary is clear enough for a focused repair, but it spans persisted pairing state, protocol, CLI, and Control UI, so it is a medium-sized feature rather than a strict bug.

## What This Changes

Add durable, operator-assigned human-friendly names for paired devices and expose rename and display support in the CLI and Control UI.

## Best Possible Solution

Add one optional operator-owned label to the canonical paired-device record, define label-over-client-displayName-over-deviceId precedence, and expose the same authenticated rename operation through Gateway protocol, CLI, and Control UI while leaving node naming and broader metadata display separate.

## Reproduction Assessment

Yes, at source level: current paired-device storage, protocol, CLI, Control UI, and docs have no durable operator-owned label or rename action. This is a missing feature rather than a failing current behavior suitable for runtime reproduction.

## Solution Assessment

Yes. A separate operator-owned field with explicit display precedence is the narrowest maintainable solution; reusing client `displayName` would create two writers and allow reconnect metadata to overwrite the operator preference.
