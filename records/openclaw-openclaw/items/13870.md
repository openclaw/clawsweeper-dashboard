## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and any rename path in storage, Gateway protocol, CLI, or Control UI. The related metadata-display issue https://github.com/openclaw/openclaw/issues/23916 and open display-only PR https://github.com/openclaw/openclaw/pull/57449 do not supersede this rename workflow.

## What This Changes

Request to add human-friendly, user-set labels for paired devices in the devices list and Control UI, with a CLI rename flow.

## Best Possible Solution

Add an additive paired-device label stored separately from client `displayName`, with protocol/schema/server/CLI/Control UI/docs/tests and clear display precedence.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive `device.pair.rename` workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
