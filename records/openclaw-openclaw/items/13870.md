## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and any rename path in storage, Gateway protocol, CLI, or Control UI. The broader metadata-display discussion at https://github.com/openclaw/openclaw/issues/23916 and open display-only PR https://github.com/openclaw/openclaw/pull/57449 do not supersede this paired-device rename workflow.

## What This Changes

Request to let operators assign durable human-readable names to paired devices and show them in the CLI and Control UI device lists.

## Best Possible Solution

Introduce an additive paired-device label stored separately from client displayName, with protocol/schema/server/CLI/UI/docs/tests and clear precedence rules.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high-confidence proof of the current gap: paired-device lists can display client displayName, but current main has no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive device.pair.rename workflow is the narrow maintainable direction; reusing client displayName would conflict with reconnect metadata updates.
