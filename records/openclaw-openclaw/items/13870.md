## Summary

Current main still lacks the requested operator-owned paired-device naming workflow; existing `displayName` is client metadata, not a durable user label, and there is no rename protocol, CLI command, or Control UI action.

## What This Changes

Request adds human-friendly names or labels for paired devices in the web UI and `openclaw devices list`, including a possible CLI rename command.

## Best Possible Solution

Add one optional operator-owned label to the canonical paired-device record, define label-over-client-displayName-over-deviceId display precedence, and expose the same authenticated rename operation through Gateway protocol, CLI, and Control UI while keeping broader metadata display separate.

## Reproduction Assessment

Yes, at source level: current storage, protocol, CLI, Control UI, and docs have no durable operator-owned paired-device label or rename action. This is a missing feature rather than a failing runtime behavior needing live reproduction.

## Solution Assessment

Yes. A separate operator-owned field with explicit display precedence is the narrowest maintainable solution; reusing client `displayName` would create two writers and allow reconnect metadata to overwrite the operator preference.
