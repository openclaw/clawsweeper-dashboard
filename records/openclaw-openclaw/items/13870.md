## Summary

Keep open: current main still has no operator-owned paired-device label or rename workflow. The existing implementation displays client-supplied `displayName` or `deviceId`, while the related metadata-display issue and open display-only PRs do not replace this user-settable naming request.

## What This Changes

The issue requests user-settable, human-friendly labels for paired devices in the web UI and `openclaw devices list`.

## Best Possible Solution

Add an additive paired-device label stored separately from client `displayName`, expose it through Gateway protocol, server handlers/scopes, CLI, and Control UI, document display precedence, and leave broader metadata display tracked separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that paired devices can display client `displayName` or `deviceId`, but have no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus an additive paired-device rename workflow is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
