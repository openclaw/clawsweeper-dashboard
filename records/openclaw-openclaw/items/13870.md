## Summary

Keep open: current main still lacks a durable, operator-owned paired-device label and rename workflow; it only displays client-supplied displayName metadata or falls back to the deviceId.

## What This Changes

The issue requests human-friendly, user-editable names for paired devices in the web UI and `openclaw devices list`.

## Best Possible Solution

Add an additive paired-device label feature: persist an operator-owned label separate from client displayName, define precedence as label then displayName then deviceId, expose a typed authorized Gateway rename method, and wire CLI, Control UI, docs, tests, and changelog together.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives a high-confidence gap because storage, protocol, server methods, CLI, Control UI, docs, and tests still lack the editable paired-device label workflow.

## Solution Assessment

Yes: a separate operator-owned label is the narrowest maintainable direction because client displayName is reconnect metadata and can be overwritten, while node.rename targets the separate node-pairing surface.
