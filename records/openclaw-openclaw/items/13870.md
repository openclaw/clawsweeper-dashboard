## Summary

Keep open: current main still lacks the requested durable paired-device naming workflow. The existing surfaces display client-supplied `displayName` or the raw `deviceId`, but there is no operator-owned label field, paired-device rename protocol method, CLI rename command, or Control UI rename action.

## What This Changes

Request durable human-friendly labels for paired devices, with CLI/UI display and rename support.

## Best Possible Solution

Add an additive paired-device label workflow with a durable operator-owned label, explicit precedence over client metadata, `device.pair.rename` protocol/schema/handler support, CLI and Control UI rename actions, docs, changelog, and regression coverage while leaving `node.rename` separate.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence proof of the current gap: paired-device lists can display `displayName`, but current main has no durable label or rename path.

## Solution Assessment

Yes. A separate operator-owned label plus additive paired-device rename method is the narrow maintainable direction; reusing client `displayName` would conflict with reconnect metadata updates.
