## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and has no rename workflow in storage, Gateway protocol, CLI, Control UI, or docs; existing client `displayName` metadata is not a safe substitute because reconnect metadata can rewrite it.

## What This Changes

Request to let operators assign human-friendly names or labels to paired devices and show them in the CLI and Control UI.

## Best Possible Solution

Add one optional operator-owned paired-device label with explicit label-over-client-displayName-over-deviceId precedence, exposed through Gateway protocol, CLI, Control UI, and docs with focused authorization tests.

## Reproduction Assessment

Yes, at source level: current storage, protocol, Gateway handlers, CLI, Control UI, and docs have no durable operator-owned paired-device label or rename action. This is a missing feature rather than a failing runtime behavior needing live reproduction.

## Solution Assessment

Yes. A separate operator-owned field with explicit display precedence is the narrowest maintainable solution; reusing client `displayName` would create two writers and allow reconnect metadata to overwrite the operator preference.
