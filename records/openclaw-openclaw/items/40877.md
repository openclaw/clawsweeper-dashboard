## Summary

Keep open. Current main still does not contain this iOS permission-snapshot work, and the PR remains a plausible fix for the reported Speech/CoreLocation warning path, but the latest diff still appears to leave one CoreLocation main-thread probe and a timing-sensitive async test helper that should be addressed before merge.

## What This Changes

The PR makes iOS gateway connect-option creation asynchronous, moves speech authorization into a detached utility-priority task, reuses a stored CLLocationManager, adds generation/in-flight guards around async gateway refresh/reconnect paths, and expands GatewayConnectionController tests.

## Best Possible Solution

Land this PR or a tightened equivalent after removing the remaining main-thread CoreLocation services-enabled probe, making the async test wait deterministic, and preserving the async speech probe plus gateway refresh/reconnect race guards with focused iOS tests and device warning validation.

## Reproduction Assessment

Yes for the source-level issue: current main deterministically routes makeConnectOptions() through currentPermissions(), which performs the reported Speech/CoreLocation permission queries on the main-actor controller path. The actual console warning symptom still needs device or simulator verification.

## Solution Assessment

No, not as currently proposed. The async speech probe and stale-task guards are the right direction, but the patch should also remove the remaining CoreLocation locationServicesEnabled() main-thread call and replace the fixed-yield test wait before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.76

Full review comments:

- **[P2] Remove the remaining main-thread CoreLocation probe:** `apps/ios/Sources/Gateway/GatewayConnectionController.swift:983-985`
  - body: `currentPermissions()` is still isolated to `GatewayConnectionController`, so after the awaited speech probe it resumes on the main actor and still calls `CLLocationManager.locationServicesEnabled()`. That leaves a CoreLocation main-thread probe in the permission snapshot path this PR is meant to quiet; route through the existing location service/status snapshot or remove the redundant services-enabled check instead.
  - confidence: 0.78

  - body: `_test_startAutoConnect` returns after 100 `Task.yield()` iterations even though `startAutoConnect` now awaits async permission construction. On a slower scheduler this helper can fall through before the config is applied, making the new reconnect tests assert too early; use the deadline-based drain helper or an explicit completion signal here too.
  - confidence: 0.73
