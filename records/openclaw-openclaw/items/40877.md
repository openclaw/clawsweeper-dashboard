## Summary

Keep this PR open: current main still has the iOS permission snapshot warning path, but the latest PR head still leaves a CoreLocation main-thread probe, two scheduler-dependent test waits, and no visible after-fix real-device/runtime proof.

## What This Changes

The PR makes iOS gateway permission snapshots async for Speech authorization, reuses a stored CLLocationManager, adds reconnect/refresh race guards, and expands GatewayConnectionController tests.

## Best Possible Solution

Land a tightened iOS controller patch that moves Speech off-main, removes the remaining CoreLocation main-thread probe while preserving permission semantics, waits deterministically in async tests, and includes redacted iOS runtime proof.

## Reproduction Assessment

Yes for a source-level path: current main routes gateway option creation through currentPermissions(), which directly probes Speech and CoreLocation authorization APIs from the main-actor controller path. I did not establish a live Xcode/CoreLocation warning reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed. The branch improves Speech authorization handling and several reconnect races, but it should remove the remaining CoreLocation warning probe and replace fixed-yield async waits before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Remove the remaining CoreLocation main-thread probe:** `apps/ios/Sources/Gateway/GatewayConnectionController.swift:985`
  - body: currentPermissions() still calls CLLocationManager.locationServicesEnabled() after moving authorizationStatus to a stored manager. Apple documents globally disabled Location Services as denied authorization and says this call is not required for authorization handling, so this keeps part of the warning-prone path alive without changing the permission boolean.
  - confidence: 0.86
- **[P2] Wait on completion instead of fixed yields:** `apps/ios/Sources/Gateway/GatewayConnectionController.swift:1115-1123`
  - body: _test_startAutoConnect() still returns after 100 Task.yield() iterations even though startAutoConnect awaits async option construction before applying the config. On slower schedulers the helper can return before the config is applied, making the new assertions flaky; wait on an explicit condition with a deadline instead.
  - confidence: 0.82
- **[P3] Replace the refresh test's fixed-yield drain:** `apps/ios/Tests/GatewayConnectionControllerTests.swift:289-292`
  - body: The new refreshActiveGatewayRegistrationPreservesLatestAuthAcrossAwait test gives the unstructured refresh task only 200 Task.yield() calls before asserting. That can still assert before the task commits on slower runners, so use a deadline or explicit completion condition instead of a fixed yield count.
  - confidence: 0.74
