## Summary

Keep open. Current main still has the targeted iOS permission-snapshot warning path, and this remains the canonical open PR, but the latest branch still has focused code/test blockers and lacks real behavior proof.

## What This Changes

The PR makes iOS gateway connect option assembly async for Speech authorization, reuses a stored CLLocationManager, adds reconnect/refresh race guards, and expands GatewayConnectionController tests.

## Best Possible Solution

Land a tightened iOS controller patch that avoids the remaining CoreLocation main-thread probe while preserving permission semantics, replaces scheduler-dependent waits with deterministic async synchronization, and includes redacted runtime proof.

## Reproduction Assessment

Yes for a source-level path: current main routes gateway option creation through currentPermissions(), which directly probes Speech and CoreLocation authorization APIs on the controller path. I did not establish a live Xcode or device warning reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed. The branch improves the Speech path and adds useful reconnect guards, but it still leaves a CoreLocation warning-prone call in the target path and keeps fixed-yield async waits in new tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid the remaining CoreLocation main-thread probe:** `apps/ios/Sources/Gateway/GatewayConnectionController.swift:985`
  - body: `currentPermissions()` still calls `CLLocationManager.locationServicesEnabled()` while building gateway permissions. Apple documents denied authorization as covering globally disabled Location Services and says this call is optional for distinguishing global disablement, so leaving it here keeps a warning-prone probe in the path the PR is trying to quiet.
  - confidence: 0.86
- **[P2] Wait for the connect task by condition, not yields:** `apps/ios/Sources/Gateway/GatewayConnectionController.swift:1115-1123`
  - body: `_test_startAutoConnect()` returns after 100 `Task.yield()` attempts even though `startAutoConnect()` now awaits permission work before applying the config. On slower schedulers this helper can return before the config is applied, making the new reconnect assertions flaky; wait on an explicit condition with a deadline instead.
  - confidence: 0.82
- **[P3] Make the refresh regression test wait for commit:** `apps/ios/Tests/GatewayConnectionControllerTests.swift:289-292`
  - body: `refreshActiveGatewayRegistrationPreservesLatestAuthAcrossAwait` only yields 200 times before checking credentials. If the refresh task has not applied yet, the test can pass without exercising the stale-overwrite path it is meant to guard; wait on a visible completion or state change with a timeout.
  - confidence: 0.75
