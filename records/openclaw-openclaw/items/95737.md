## Summary

Keep open: current main and the latest release still expose only sender-id and sender-name to the MS Teams group allowlist matcher, while the open linked PR is the focused candidate fix for conversation-id matching.

## What This Changes

The issue asks for Microsoft Teams `groupAllowFrom` allowlist entries to match normalized group/channel conversation IDs, including `;messageid=` suffix stripping and legacy `@thread.skype` IDs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95764, or an equivalent Teams plugin fix that adds a non-DM conversation-id allowlist alias with suffix normalization and regression coverage after maintainer acceptance of the auth semantics.

## Reproduction Assessment

Yes, source-reproducible: current main normalizes the Teams conversation ID but never exposes it as a subject identity alias, while the shared matcher only checks declared identity fields. I did not run a live Teams tenant repro in this read-only review.

## Solution Assessment

Yes, with one maintainer decision: adding a non-DM `conversation-id` alias in the Teams plugin is the narrowest code path if `groupAllowFrom` should authorize conversations. If maintainers want `groupAllowFrom` to stay sender-only, the docs should instead point users to `channels.msteams.teams` for channel scoping.
