## Summary

Close: current main and v2026.6.9 already normalize Teams conversation IDs for route matching, while source and docs define `groupAllowFrom` as a sender allowlist; channel conversation IDs belong under `channels.msteams.teams`, so this is configuration guidance rather than an actionable source bug.

## What This Changes

The issue asks OpenClaw to treat MS Teams channel conversation IDs in `groupAllowFrom` as allowlisted groups/channels after stripping any `;messageid=` suffix.

## Best Possible Solution

Keep the current split: use `channels.msteams.teams` to scope Teams/team/channel conversation IDs, and use `groupAllowFrom` only for sender AAD object IDs or access groups.

## Reproduction Assessment

Yes for the observed drop under the submitted config: source shows `groupAllowFrom` is evaluated against the sender identity, so a Teams channel conversation ID there will not match. No current source path treats `groupAllowFrom` as the channel route allowlist.

## Solution Assessment

No; overloading `groupAllowFrom` to also mean channel conversation IDs would duplicate the existing `channels.msteams.teams` route allowlist and blur sender versus route authorization. The narrow supported solution is to configure Teams channel IDs under `teams` and sender IDs or `"*"` under `groupAllowFrom` as needed.
