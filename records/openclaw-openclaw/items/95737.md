## Summary

Keep open: current main still resolves MS Teams group sender allowlists against sender ID/name only, while an open Fixes-linked PR owns the conversation-ID normalization repair.

## What This Changes

The issue reports MS Teams group/channel messages being dropped when `groupAllowFrom` contains a Teams conversation ID, especially with `;messageid=...` suffixes or legacy `@thread.skype` IDs.

## Best Possible Solution

Land a narrow MS Teams access resolver fix that exposes normalized non-DM conversation IDs as allowlist match material, with coverage for `;messageid=...`, `@thread.tacv2`, `@thread.skype`, and DM isolation.

## Reproduction Assessment

Yes, at source level: current main only supplies sender ID/name to the ingress matcher, so a `groupAllowFrom` entry containing the normalized conversation ID has no matching subject. I did not run a live Teams tenant repro in this read-only review.

## Solution Assessment

Yes, likely: adding a normalized non-DM conversation-id alias in the MS Teams access resolver is the narrow owner-boundary fix. The issue should stay open until the linked PR or an equivalent repair merges.
