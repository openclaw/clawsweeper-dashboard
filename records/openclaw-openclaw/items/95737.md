## Summary

Keep this issue open: current main and the latest release still route MS Teams groupAllowFrom through sender identity matching, and the linked open fix PR now owns the candidate repair. The only review caveat is product shape: current docs describe groupAllowFrom as a sender allowlist while channel/conversation scoping is documented under teams, so maintainers should settle that config contract on the linked PR before this issue closes.

## What This Changes

The issue requests that MS Teams groupAllowFrom accept normalized group/channel conversation IDs, including @thread.tacv2, @thread.skype, and runtime ;messageid= suffix cases.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/95764 or replace it with an equivalent MS Teams-owned fix that aligns docs and tests with the accepted groupAllowFrom conversation-ID contract.

## Reproduction Assessment

Yes. Source inspection shows current main normalizes the inbound conversation ID but does not include it in MS Teams sender identity material, so a groupAllowFrom entry containing that conversation ID cannot match and the group_policy_not_allowlisted path drops the message.

## Solution Assessment

Unclear. Adding a normalized non-DM conversation-id alias is the narrow code fix if maintainers want groupAllowFrom to accept conversation IDs; otherwise the documented shape is to use channels.msteams.teams for channel scoping and keep groupAllowFrom sender-only.
