## Summary

Keep open: current main and the latest release still hard-code the Gateway-visible TUI label as `openclaw-tui`, with no CLI, TUI option, relaunch, or docs path to override it. The Gateway already accepts `clientDisplayName`, so the remaining work is a TUI UX/product decision rather than a missing protocol seam.

## What This Changes

Request adds a TUI CLI option or generated naming behavior so multiple Gateway-connected TUI windows show distinct client display names.

## Best Possible Solution

Add a TUI-owned, sanitized client display-name option that feeds `GatewayClient.clientDisplayName`, update relaunch/docs/tests, and keep the stable TUI client id plus unique instance id semantics unchanged.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the public flag name, generated-name policy, and sanitization rules before implementation.
