## Summary

Keep open: current main and the latest release still hard-code the TUI Gateway display label as `openclaw-tui`, and there is no TUI CLI/options/docs surface for a custom client name. This is a concrete usability feature gap, but it needs maintainer UX policy for the public flag name, generated defaults, and display-label sanitization before implementation.

## What This Changes

Request adding a custom display-name CLI option or generated unique display name for Gateway-backed TUI clients.

## Best Possible Solution

Add a TUI-owned, maintainer-approved, sanitized display-name option that feeds `GatewayClient.clientDisplayName` while preserving stable client id and unique instance id semantics, with focused CLI/client coverage and docs.

## Reproduction Assessment

Yes, as a source-level feature gap: `registerTuiCli` exposes no name/display-name option, `TuiOptions` cannot carry one, and `GatewayChatClient` always passes `clientDisplayName: "openclaw-tui"`. A live repro would connect two TUIs to one Gateway and inspect presence, but the code path is direct.

## Solution Assessment

Unclear until maintainers choose the CLI UX and sanitization policy. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, so the work should stay in TUI CLI/options/handoff rather than protocol changes.
