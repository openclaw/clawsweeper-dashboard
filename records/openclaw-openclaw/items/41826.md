## Summary

Keep open: current main and the latest shipped tag still hardcode the TUI Gateway display label and expose no CLI or docs surface for overriding it, so this remains a valid user-facing feature request rather than cleanup-close material.

## What This Changes

The issue requests a custom `openclaw tui` client display name, likely through a new CLI flag or automatic per-window naming.

## Best Possible Solution

Add a TUI-owned display-name option that feeds `GatewayClient.clientDisplayName`, update the public TUI docs and focused tests, and keep gateway client ids and instance ids unchanged.

## Reproduction Assessment

Yes, from source inspection: current main always sends `clientDisplayName: "openclaw-tui"`, and `openclaw tui` has no option to override it. A live repro would start two gateway-backed TUI instances and inspect `system-presence`, but the code path is already clear.

## Solution Assessment

Unclear pending maintainer UX choice. A TUI-specific `--client-name` style flag is the narrowest likely implementation, while `--username` or automatic hostname/TTY defaults need product direction before code work.
