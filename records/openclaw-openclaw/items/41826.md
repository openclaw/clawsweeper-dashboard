## Summary

Keep open. Current main still lacks a custom TUI client display-name path: the CLI does not parse a client-name/display-name flag, TUI option types cannot carry it, the Gateway connect path does not pass it, and `GatewayChatClient` still hard-codes `clientDisplayName: "openclaw-tui"`. The Gateway protocol/client layer already supports display names, so this is a focused TUI wiring/docs/tests task rather than a core protocol gap.

## What This Changes

The issue requests a custom display-name option for TUI clients so multiple TUI connections can be distinguished in Gateway presence/session lists.

## Best Possible Solution

Add a focused TUI feature that accepts a user-provided display name, sanitizes it, preserves the stable TUI client id, passes the value through the TUI option/relaunch/connect layers into `GatewayChatClient`, and documents the new flag on the public TUI CLI/docs pages.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: inspect the `openclaw tui` option registration, `TuiOptions`, `runTui` Gateway connection, and `GatewayChatClient` construction path; the value is not accepted or propagated and the sent display name is fixed.

## Solution Assessment

Yes, a narrow TUI wiring fix is the best path. The core Gateway protocol already has the generic display-name seam, so this should not become a broader Gateway protocol or presence redesign.
