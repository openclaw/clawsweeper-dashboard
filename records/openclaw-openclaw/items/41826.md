## Summary

Keep open: current main and v2026.6.10 still hardcode the Gateway-backed TUI display label as `openclaw-tui`, and the TUI CLI/docs/options expose no custom display-name path. The implementation seam already exists in Gateway client metadata, but adding a public TUI flag needs maintainer direction on naming, validation, Gateway-only scope, and privacy defaults.

## What This Changes

Request to let `openclaw tui` instances advertise a user-chosen Gateway-visible client display name instead of always using `openclaw-tui`.

## Best Possible Solution

Add an explicit opt-in Gateway-mode TUI display-name override that keeps `openclaw-tui` as the default, validates and sanitizes the label, and documents privacy-safe usage after maintainers choose the public option shape.

## Reproduction Assessment

Yes, at source level: current main still hardcodes the Gateway-backed TUI display label as `openclaw-tui`, while the TUI CLI, docs, and `TuiOptions` expose no override path. I did not run a live multi-TUI Gateway session in this read-only review.

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `displayName` seam is the narrow technical route, but the public flag/config name, validation, Gateway-only scope, and privacy defaults need product direction first.
