## Summary

Keep open: current main and the latest release still expose no TUI display-name override, while the Gateway client already has a `clientDisplayName` seam. This is a real, source-reproducible feature gap, but the fix needs a maintainer UX decision because it adds a public TUI flag and user-controlled Gateway-visible label policy.

## What This Changes

The issue requests a TUI option for setting a custom Gateway-visible client display name when multiple TUI instances are connected.

## Best Possible Solution

Add a TUI-owned, sanitized maintainer-approved display-name option that feeds `GatewayClient.clientDisplayName`, update docs and focused TUI/Gateway tests, and keep the stable TUI client id plus unique instance id semantics unchanged.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live reproduction would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name policy, and sanitization rules before implementation.
