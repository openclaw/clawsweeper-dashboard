## Summary

Keep open: current main and the latest release still hardcode the TUI Gateway display label and expose no TUI CLI/docs override, while the Gateway already has a display-name seam. This is a concrete feature request that needs maintainer UX/policy review, not conservative cleanup closure.

## What This Changes

Request adding a custom TUI client display name flag or auto-generated label so multiple Gateway-backed TUI windows can be distinguished.

## Best Possible Solution

Add a TUI-owned sanitized display-name option that feeds `GatewayClient.clientDisplayName`, with focused CLI/client tests and docs, while preserving the existing client id and unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: current main and `v2026.5.12` always send `clientDisplayName: "openclaw-tui"`, and neither the TUI CLI nor `TuiOptions` has an override. A live repro would start two Gateway-backed TUI instances and inspect presence, but the code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing Gateway `clientDisplayName` seam is the narrow implementation path, but maintainers should choose the flag name, scope, defaulting, and sanitization policy before implementation.
