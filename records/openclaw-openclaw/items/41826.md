## Summary

Keep open: current main and the latest release still hardcode the TUI Gateway display label and expose no CLI/docs override. The Gateway protocol already accepts a client display name, so the remaining work is a TUI UX/product decision rather than cleanup-close material.

## What This Changes

The issue asks for a TUI option or automatic policy so each `openclaw tui` instance can send a custom Gateway display name instead of all using `openclaw-tui`.

## Best Possible Solution

Add a TUI-owned display-name option that feeds `GatewayClient.clientDisplayName`, with focused CLI/client tests and TUI docs, while leaving Gateway client ids and instance ids unchanged.

## Reproduction Assessment

Yes, as a source-level feature gap rather than a failing bug: current main always sends `clientDisplayName: "openclaw-tui"`, and `openclaw tui` has no override option. A live repro would start two Gateway-backed TUI instances and inspect presence, but the code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing Gateway display-name seam is the narrow implementation path, but maintainers should choose the flag name, scope, and sanitization policy before implementation.
