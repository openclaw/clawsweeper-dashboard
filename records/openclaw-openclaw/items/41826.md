## Summary

Keep open: current main still hard-codes the TUI Gateway client display label as `openclaw-tui`, and there is no CLI, option type, relaunch, or docs surface for users to override it. The Gateway client/protocol already has a `clientDisplayName` field, so the remaining work is a TUI-owned UX/product decision rather than a missing Gateway protocol capability.

## What This Changes

The issue requests a public `openclaw tui` option such as `--client-name` or `--username` so multiple Gateway-connected TUI windows can show distinct display names.

## Best Possible Solution

Add a TUI-owned, sanitized client display-name option that feeds `GatewayClient.clientDisplayName`, update relaunch/docs/tests, and preserve the stable TUI client id plus unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the public flag name, generated-name policy, and sanitization rules before implementation.
