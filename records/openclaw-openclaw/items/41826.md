## Summary

Keep open: current main and the latest release still hardcode the TUI Gateway display label and expose no CLI or docs path for overriding it. The Gateway client contract already accepts a display name, so the remaining work is a TUI-owned UX/product decision rather than cleanup-close material.

## What This Changes

The issue requests a configurable display name for `openclaw tui` clients so multiple TUI windows are distinguishable in Gateway presence.

## Best Possible Solution

Add a TUI-owned, maintainer-selected display-name flag that feeds `GatewayClient.clientDisplayName`, with focused tests and public TUI docs, while keeping Gateway client ids and instance ids unchanged.

## Reproduction Assessment

Yes, as a source-level feature gap rather than a bug repro: current main always sends `clientDisplayName: "openclaw-tui"`, and `openclaw tui` has no override option. A live repro would start two Gateway-backed TUI instances and inspect `system-presence`, but the code path is already clear.

## Solution Assessment

Unclear pending maintainer UX choice. A TUI-specific `--client-name` or `--display-name` that feeds the existing Gateway display-name seam is the narrow likely solution, while `--username` or automatic hostname/TTY defaults need product direction.
