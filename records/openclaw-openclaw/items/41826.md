## Summary

Keep open: current main and the latest shipped release still hard-code the TUI Gateway display label, while the Gateway client/protocol already has a display-name field. This is a concrete TUI feature request, but it needs maintainer UX/policy choice before implementation.

## What This Changes

Allow `openclaw tui` instances to use a custom or generated Gateway display name instead of every TUI presenting as `openclaw-tui`.

## Best Possible Solution

Add a TUI-owned sanitized display-name option that feeds `GatewayClient.clientDisplayName`, with focused CLI/client tests and docs, while preserving the stable client id and unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start two Gateway-backed TUI instances and inspect presence, but the code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing Gateway `clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, scope, generated-name policy, and sanitization rules before implementation.
