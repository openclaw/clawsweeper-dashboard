## Summary

Keep open. Current main still lacks the requested TUI display-name option: the CLI has no client-name/username flag, `TuiOptions` cannot carry such a value, and the Gateway TUI client still sends the hard-coded `openclaw-tui` display name even though the Gateway client already supports `clientDisplayName`.

## What This Changes

Add a user-facing `openclaw tui` option that lets each TUI instance send a distinct Gateway client display name.

## Best Possible Solution

Add a focused TUI display-name flag that feeds a sanitized label into the existing Gateway `clientDisplayName` contract while preserving the stable TUI client id.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but yes for the missing capability: current-main source inspection shows no TUI CLI flag, no `TuiOptions` field, no connect-path propagation, and a hard-coded display name.

## Solution Assessment

Is this the best way to solve the issue? Yes. A narrow TUI flag plus options/connect plumbing is the maintainable path because the Gateway client already accepts `clientDisplayName`, so no protocol redesign is needed.
