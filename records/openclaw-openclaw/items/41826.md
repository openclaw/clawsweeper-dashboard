## Summary

Keep open: current main and the latest release still hardcode the Gateway-backed TUI display label as `openclaw-tui`, and no CLI/docs/options path exposes the requested custom client label. The feature is source-reproducible but still needs maintainer direction because it adds public CLI/config surface and has privacy-sensitive naming choices.

## What This Changes

Request to add a custom display name option, such as `openclaw tui --client-name`, so multiple TUI clients connected to the same Gateway are distinguishable.

## Best Possible Solution

Add an explicit opt-in TUI display-name override that keeps `openclaw-tui` as the default, reuses the existing Gateway `displayName` field, validates/trims the label, and documents privacy-safe usage after maintainers choose the public option shape.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.10` still hardcode the Gateway-backed TUI display label as `openclaw-tui`, while the TUI CLI, docs, and `TuiOptions` expose no override path. I did not run a live Gateway/TUI session in this read-only review.

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `displayName` field is the narrow technical route, but the public flag/config name, validation, Gateway-only scope, and privacy defaults need product direction first.
