## Summary

Keep open. Current main still lacks a TUI display-name option: the CLI exposes no client-name/display-name flag, `TuiOptions` cannot carry one, and the Gateway TUI path still sends the hard-coded `openclaw-tui` label even though the Gateway client already supports `clientDisplayName`.

## What This Changes

Request a TUI CLI option for a custom Gateway display name so multiple connected TUI windows can be distinguished.

## Best Possible Solution

Add a focused TUI display-name flag that feeds a sanitized label into the existing Gateway `clientDisplayName` contract while preserving the stable TUI client id.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but source inspection gives high-confidence proof of the missing capability: there is no TUI flag, no option field, no connect-path propagation, and a hard-coded display name.

## Solution Assessment

Is this the best way to solve the issue? Yes. A narrow TUI flag plus option/connect plumbing is the maintainable path because the Gateway client already accepts `clientDisplayName`, so no protocol or Gateway redesign is needed.
