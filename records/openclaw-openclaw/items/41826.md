## Summary

Keep open: current main and the latest release still expose only the hard-coded TUI Gateway label, while the Gateway client/protocol already has a display-name field. This remains a valid feature gap, but it needs maintainer UX policy before implementation.

## What This Changes

The request asks for a custom or generated display name for each `openclaw tui` Gateway client so multiple TUI windows are distinguishable.

## Best Possible Solution

Add a TUI-owned, sanitized display-name option that feeds `GatewayClient.clientDisplayName`, documents the behavior, and preserves stable client id plus unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing Gateway `clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name policy, and sanitization rules before implementation.
