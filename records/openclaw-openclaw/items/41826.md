## Summary

Keep open: current main still hardcodes the TUI gateway display name and exposes no TUI flag for overriding it. The gateway protocol already carries `client.displayName`, so this is a narrow feature/config request, but the public flag name and default behavior need maintainer judgment.

## What This Changes

Request to let `openclaw tui` instances use distinct custom display names in gateway presence instead of all appearing as `openclaw-tui`.

## Best Possible Solution

Add a documented TUI display-name option that threads into `GatewayChatClient.clientDisplayName` and gateway presence without changing the existing gateway protocol.

## Reproduction Assessment

Yes, from source inspection: the TUI command has no display-name option and `GatewayChatClient` sends a fixed `clientDisplayName`. I did not run a live multi-TUI gateway session in this read-only pass.

## Solution Assessment

Unclear. The technical path is narrow because the gateway already accepts `client.displayName`, but a maintainer should choose the public flag name and default behavior before implementation.
