## Summary

Keep open. Current `main` still lacks a TUI display-name flag or option plumbing, and the Gateway TUI client still sends the hard-coded `openclaw-tui` display label even though the Gateway client contract already supports `clientDisplayName`.

## What This Changes

The issue requests a custom display-name option for `openclaw tui` so multiple TUI clients connected to one Gateway can be distinguished.

## Best Possible Solution

Add a focused TUI display-name flag that feeds a sanitized label into the existing Gateway `clientDisplayName` contract while preserving the stable TUI client id.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but source inspection gives high-confidence proof of the missing capability: no CLI flag, no option field, no connect-path propagation, and a hard-coded display name.

## Solution Assessment

Is this the best way to solve the issue? Yes. A narrow TUI flag plus option/connect plumbing is the maintainable path because the Gateway client already accepts `clientDisplayName`, so no protocol or Gateway redesign is needed.
