## Summary

Keep open: current main and v2026.5.12 still hardcode the Gateway-visible TUI label and expose no TUI option or docs path for overriding it. The Gateway client and presence contracts already accept a display name, so this is a valid TUI UX/product feature request rather than cleanup-close material.

## What This Changes

Request a TUI CLI option or automatic naming policy so multiple Gateway-connected TUI instances can use distinct visible display names.

## Best Possible Solution

Add a TUI-owned maintainer-selected display-name flag that threads through `TuiOptions` to `GatewayClient.clientDisplayName`, with sanitation, focused tests, and docs.

## Reproduction Assessment

Yes. The source path is high-confidence: current main hardcodes `clientDisplayName: "openclaw-tui"`, `openclaw tui` has no override flag, and Gateway presence consumes that display name.

## Solution Assessment

Unclear pending maintainer UX choice. The existing Gateway display-name seam is the right lower-level hook, but the flag name, mode behavior, and sanitation policy need product review before implementation.
