## Summary

Keep open. Current main still has the focused TUI gap described in the issue: the Gateway client layer supports a display name, but `openclaw tui` has no user-facing option and the TUI client still sends `openclaw-tui` unconditionally.

## What This Changes

Request adds an `openclaw tui` option to set a custom Gateway client display name so multiple TUI windows can be distinguished.

## Best Possible Solution

Add a narrow TUI display-name flag that preserves the stable TUI client id and plumbs a sanitized label through the existing Gateway `clientDisplayName` contract.

## Reproduction Assessment

Yes for the missing capability. Current-main source inspection shows no TUI CLI option, no `TuiOptions` field, no propagation into `GatewayChatClient.connect`, and a hard-coded `clientDisplayName`.

## Solution Assessment

Yes. A focused TUI flag and plumbing change is the narrowest maintainable solution because Gateway display names already exist and no protocol redesign is needed.

## Repair Work Prompt

Fix openclaw/openclaw#41826 by adding a custom display-name option for `openclaw tui`. Add a user-facing flag such as `--client-name <name>` to the TUI CLI, thread it through `TuiOptions`, `runTui`, `GatewayChatClient.connect`, and `launchTuiCli`, and pass it as `clientDisplayName` while preserving `GATEWAY_CLIENT_NAMES.TUI` / `openclaw-tui` as the stable client id. Trim the value, omit or reject empty values, apply a reasonable length bound, and strip terminal control/ANSI sequences before sending it to the Gateway. Update TUI docs and add a user-facing changelog entry. Do not change Gateway protocol ids or make display names authorization identifiers.
