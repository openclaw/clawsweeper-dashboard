## Summary

Keep open: current main and latest release still hardcode Gateway-backed TUI clients as `openclaw-tui`, and there is no TUI CLI/docs/options path for a custom label. The request is source-reproducible, but it adds public CLI/config surface and needs maintainer direction on naming, validation, Gateway-only scope, and privacy defaults before implementation.

## What This Changes

The issue requests a custom display name for TUI Gateway clients so multiple TUI windows can be distinguished.

## Best Possible Solution

Add an explicit opt-in TUI display-name override that keeps `openclaw-tui` as the default, reuses the existing Gateway `displayName` seam, and documents privacy-safe usage after maintainers choose the public option shape.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still hardcode the Gateway-backed TUI display label as `openclaw-tui`, while the TUI CLI, docs, and `TuiOptions` expose no override path. I did not run a live Gateway/TUI session in this read-only review.

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `displayName` field is the narrow technical route, but the public flag/config name, validation, Gateway-only scope, and privacy defaults need product direction first.
