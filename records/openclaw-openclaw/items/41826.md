## Summary

Keep open: current main and the latest release still hardcode Gateway-backed TUI clients as `openclaw-tui`, and there is no public TUI option or config path for a custom display label. The request is valid, but it adds public CLI surface and needs a maintainer product decision for flag naming, validation, and privacy-safe behavior before a repair PR should be queued.

## What This Changes

Request to add a custom display name for `openclaw tui` clients so multiple TUI windows can be distinguished in Gateway presence/session views.

## Best Possible Solution

Add an explicit opt-in TUI display-name flag that flows through `TuiOptions` into the existing Gateway `clientDisplayName` field, keep `openclaw-tui` as the default, and avoid auto-publishing host/path-derived metadata.

## Reproduction Assessment

Yes, source-reproducible: current main hardcodes the Gateway-backed TUI display label and exposes no CLI, config, or option type that can override it. I did not run a live multi-TUI Gateway session because the connection construction and option plumbing directly establish the limitation.

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `clientDisplayName` field is the narrowest technical path, but the public flag name, validation, docs, and privacy-safe default need a product decision first.
