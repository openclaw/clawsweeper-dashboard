## Summary

Keep open: current main and the latest release still hard-code the Gateway-visible TUI client label as `openclaw-tui`, and no `openclaw tui` flag or option path carries a custom display name into the Gateway handshake. The implementation seam already exists, but this is a new UX/config surface that needs a maintainer decision before a fix PR.

## What This Changes

The issue requests a TUI option such as `--client-name` or `--username` so each Gateway-connected TUI window can show a distinct client display name.

## Best Possible Solution

Add a TUI-owned, sanitized display-name override wired into `GatewayClient.clientDisplayName`, while preserving stable `GATEWAY_CLIENT_NAMES.TUI` client-id and unique `instanceId` semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name behavior, and sanitization/privacy policy first.
