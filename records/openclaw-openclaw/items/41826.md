## Summary

Keep open: current main and v2026.5.19 still expose only the fixed TUI Gateway label, while the Gateway protocol already has a display-name seam. This is a concrete TUI feature request that needs maintainer UX/policy choice before implementation.

## What This Changes

The issue requests a TUI option such as `--client-name` or `--username` so multiple Gateway-connected TUI windows can show distinct display names.

## Best Possible Solution

Add a TUI-owned sanitized display-name option that feeds `GatewayClient.clientDisplayName`, with focused CLI/client tests and docs, while preserving the existing client id and unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: current main and v2026.5.19 always send `clientDisplayName: "openclaw-tui"`, and neither the TUI CLI nor `TuiOptions` has an override. A live repro would start two Gateway-backed TUI instances and inspect `system-presence`, but the code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The Gateway `clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, scope, defaulting, and sanitization policy before implementation.
