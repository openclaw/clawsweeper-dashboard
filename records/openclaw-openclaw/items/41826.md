## Summary

Keep open: current main and v2026.6.10 still hardcode the Gateway-backed TUI client display label as `openclaw-tui`, and no CLI option, TUI option field, config key, or docs path exposes the requested custom label. The source path is clear, but adding a new public TUI flag/config surface needs maintainer direction on naming, validation, Gateway-only scope, and privacy-safe defaults.

## What This Changes

Issue requests a custom display name option for Gateway-backed `openclaw tui` clients so multiple TUI windows can be distinguished.

## Best Possible Solution

Add an explicit opt-in TUI display-name override that keeps `openclaw-tui` as the default, reuses the existing Gateway `displayName` seam, and documents privacy-safe usage after maintainers choose the public option shape.

## Reproduction Assessment

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `displayName` field is the narrow technical route, but the public flag/config name, validation, Gateway-only scope, and privacy defaults need product direction first.
