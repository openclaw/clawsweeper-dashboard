## Summary

Keep open. Current main still lacks the requested TUI client display-name option: the CLI exposes no `--client-name`/`--username` flag, `TuiOptions` cannot carry it, and the Gateway TUI client still sends the hard-coded `openclaw-tui` display name even though the Gateway protocol already has a `displayName` field.

## What This Changes

Add a user-supplied display-name option for `openclaw tui` so multiple connected TUI windows can be distinguished in Gateway presence/session surfaces.

## Best Possible Solution

Add a focused TUI display-name flag that feeds a sanitized label into the existing Gateway `clientDisplayName` contract while preserving the stable TUI client id.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but the missing capability is source-proven: current main has no TUI flag, no `TuiOptions` field, no connect-path propagation, and a hard-coded display name.

## Solution Assessment

Is this the best way to solve the issue? Yes. A narrow CLI option plus TUI option/connect plumbing is the maintainable path because the Gateway client and protocol already accept `clientDisplayName`.
