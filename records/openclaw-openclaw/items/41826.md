## Summary

Keep open: current main and v2026.6.6 still hardcode Gateway-backed TUI clients as `openclaw-tui`, with no CLI, docs, or `TuiOptions` path for a custom display label. The request is useful and source-reproducible, but it adds public CLI surface and needs maintainer product direction on naming, validation, and privacy-safe behavior before implementation.

## What This Changes

Add a public way for `openclaw tui` users to set a custom Gateway-visible client display name for each TUI instance.

## Best Possible Solution

Add an explicit opt-in TUI display-name flag with a maintainer-approved name, keep `openclaw-tui` as the default, flow the value through `TuiOptions` into `GatewayChatClient`'s existing `clientDisplayName`, and document privacy-safe usage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main hardcodes the Gateway-backed TUI display label and has no CLI, docs, or option type that can override it. I did not run a live multi-TUI Gateway session because this read-only review established the limitation directly from the connection path.

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `clientDisplayName` field is the narrowest technical route, but the public flag name, validation, docs, and privacy defaults need a product decision first.
