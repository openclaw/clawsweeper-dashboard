## Summary

Keep open: current main and the latest release still hardcode the TUI Gateway display label and expose no CLI/docs path for overriding it, so this remains a valid feature request rather than cleanup-close material.

## What This Changes

The issue requests a custom display-name option for `openclaw tui` so multiple TUI clients are distinguishable in Gateway presence/session lists.

## Best Possible Solution

Add a TUI-owned, maintainer-selected display-name flag that feeds `GatewayClient.clientDisplayName`, with focused tests and public TUI docs, while keeping Gateway client ids and instance ids unchanged.

## Reproduction Assessment

Yes, from source inspection: current main always sends `clientDisplayName: "openclaw-tui"`, and `openclaw tui` has no override option. A live repro would start two Gateway-backed TUI instances and inspect `system-presence`, but the code path is already clear.

## Solution Assessment

Unclear pending maintainer UX choice. A TUI-specific client display-name flag is the narrowest likely solution, while `--username` or automatic hostname/TTY defaults need product direction before implementation.
