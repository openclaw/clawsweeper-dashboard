## Summary

Keep open: current main and v2026.6.1 still make Gateway-connected TUI clients present the fixed display label `openclaw-tui`, while the lower-level Gateway client contract already has a diagnostic `displayName` field that the TUI does not expose.

## What This Changes

Request to let `openclaw tui` instances set a custom Gateway-visible client display name, such as via `--client-name` or `--username`.

## Best Possible Solution

Expose an explicit opt-in TUI display label through the existing Gateway `clientDisplayName` path, document it, and keep the default label unchanged unless maintainers approve privacy-safe auto naming.

## Reproduction Assessment

Yes, source-reproducible: current main registers no `--client-name` or `--username` option and sends `clientDisplayName: "openclaw-tui"` for TUI Gateway connections. I did not run a live multi-TUI Gateway session.

## Solution Assessment

Unclear. The existing `clientDisplayName` wire field is the right lower-level path, but the best user surface needs a maintainer choice between an explicit flag, auto-generated labels, or no new public surface.
