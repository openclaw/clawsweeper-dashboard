## Summary

Keep open: current main and v2026.5.22 still hard-code the Gateway-backed TUI display label as `openclaw-tui` and expose no TUI CLI/docs option for overriding it. The Gateway client already has a `clientDisplayName` seam, so the remaining work is a small TUI-owned feature after maintainers choose the public flag name, default/generated-name behavior, and sanitization/privacy rules.

## What This Changes

The issue requests a custom display-name option for `openclaw tui` so multiple Gateway-connected TUI windows are distinguishable.

## Best Possible Solution

Add a TUI-owned, sanitized display-name override wired into `GatewayClient.clientDisplayName`, while preserving the stable TUI client id and unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name policy, and sanitization rules before implementation.
