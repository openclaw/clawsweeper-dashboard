## Summary

Keep open: current main and v2026.5.20 still hard-code the Gateway-backed TUI presence label as `openclaw-tui` and expose no TUI display-name override, while the Gateway client already supports `clientDisplayName`. This remains a valid, bounded TUI UX feature request, but maintainers still need to choose the public flag name, generated-name behavior, and sanitization/privacy policy before implementation.

## What This Changes

The issue requests a TUI option such as `--client-name` or `--username` so multiple Gateway-connected TUI windows can show distinct display names.

## Best Possible Solution

Add a TUI-owned, sanitized display-name override wired into `GatewayClient.clientDisplayName`, while preserving stable TUI client id and unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name policy, and sanitization rules before implementation.
