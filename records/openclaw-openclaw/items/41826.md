## Summary

Keep open: current main and v2026.5.12 still hardcode the TUI Gateway display label and expose no TUI CLI/docs override. This remains a concrete, source-verifiable feature request that needs a maintainer UX decision rather than conservative cleanup closure.

## What This Changes

The issue requests a TUI option or automatic policy to give each Gateway-connected TUI instance a custom display name.

## Best Possible Solution

Add a TUI-owned sanitized display-name option that feeds `GatewayClient.clientDisplayName`, with focused CLI/client tests and docs, while preserving the existing client id and unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: current main always sends `clientDisplayName: "openclaw-tui"`, and `openclaw tui` has no override option or TuiOptions field. A live repro would start two Gateway-backed TUI instances and inspect presence, but the code path is already direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing Gateway display-name seam is the narrow implementation path, but maintainers should choose the flag name, scope, defaulting, and sanitization policy before implementation.
