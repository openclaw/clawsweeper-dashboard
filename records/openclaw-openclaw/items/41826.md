## Summary

Keep open: current main and the latest release still hardcode every Gateway-backed TUI as `openclaw-tui`, and no public TUI option or option type carries a custom display label. The request is valid and bounded, but it adds public CLI surface and needs a maintainer product decision for the flag name, validation, and privacy-safe defaults.

## What This Changes

Request adding a custom display-name option for `openclaw tui` so multiple Gateway-connected TUI windows can be distinguished.

## Best Possible Solution

Add an explicit opt-in TUI display-name flag that flows through `TuiOptions` into the existing Gateway `clientDisplayName` field, keep `openclaw-tui` as the default, and avoid auto-publishing host/path-derived metadata.

## Reproduction Assessment

Yes, source-reproducible: current main hardcodes the Gateway-backed TUI display label and exposes no matching CLI or TUI option. I did not run a live multi-TUI Gateway session, but the connection construction and option plumbing directly establish the limitation.

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `clientDisplayName` field is the narrowest technical path, but the public flag name, validation, docs, and privacy-safe default need a product decision first.
