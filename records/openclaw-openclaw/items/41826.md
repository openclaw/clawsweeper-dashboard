## Summary

Keep open: current main and v2026.6.8 still hardcode Gateway-backed TUI clients as `openclaw-tui` and expose no CLI, docs, or `TuiOptions` override. The request is source-reproducible and useful, but it adds public TUI CLI surface and needs maintainer direction on naming, validation, and privacy before implementation.

## What This Changes

Request adding a custom display-name or username option for `openclaw tui` so multiple Gateway-connected TUI windows can be distinguished.

## Best Possible Solution

Add an explicit opt-in TUI display-name flag after maintainer approval, keep `openclaw-tui` as the default, and flow the value through `TuiOptions` into `GatewayChatClient` with focused docs and tests.

## Reproduction Assessment

Yes, at source level: current main hardcodes the Gateway-backed TUI `clientDisplayName` and has no CLI, docs, or `TuiOptions` path for an override. I did not run a live multi-TUI Gateway session because this read-only review established the limitation directly from the connection path.

## Solution Assessment

Unclear until maintainers pick the public behavior. The existing Gateway `clientDisplayName` field is the narrow technical seam, but the flag name, validation, gateway/local applicability, and privacy guidance need product approval first.
