## Summary

Keep open: current main still hardcodes the gateway-backed TUI display name and has no supported CLI/config surface for naming individual TUI windows. The implementation path is clear, but the user-facing option name, sanitization, and auto-generated-name behavior need maintainer product judgment before a repair lane should act.

## What This Changes

The issue requests a configurable display name for `openclaw tui` clients so multiple TUI windows can be distinguished in gateway presence/session views.

## Best Possible Solution

Define a supported TUI identity surface that passes a sanitized user-selected or generated display name into the existing gateway client display-name field and documents how it appears in presence.

## Reproduction Assessment

Yes, at source level: current main hardcodes `clientDisplayName: "openclaw-tui"` and the CLI exposes no client-name option. A live repro would start two gateway-backed TUI instances and inspect `system-presence` or the Control UI presence list.

## Solution Assessment

Unclear pending product choice. Passing a new TUI option into the existing `GatewayClientOptions.clientDisplayName` path is technically narrow, but maintainers should decide the option name, sanitization, and whether auto-generated labels are preferable.
