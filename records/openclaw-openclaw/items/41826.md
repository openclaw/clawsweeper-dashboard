## Summary

Keep open. Current main still has the same focused TUI gap: the Gateway client layer can send a display name, but `openclaw tui` has no custom display-name flag and the TUI Gateway client still sends `openclaw-tui` unconditionally.

## What This Changes

The issue requests a user-settable TUI Gateway client display name, such as `openclaw tui --client-name "Office-Terminal"`, so multiple TUI windows can be distinguished in Gateway presence/session views.

## Best Possible Solution

Add a narrow TUI feature that accepts a custom client display name, preserves the stable `openclaw-tui` client id, propagates the value through launch/connect layers, and documents the new flag.

## Reproduction Assessment

Yes for the reported missing capability. Current-main source inspection shows no CLI option, no `TuiOptions` field, no propagation into `GatewayChatClient.connect`, and a hard-coded `clientDisplayName`.

## Solution Assessment

Yes. A focused TUI flag and plumbing fix is the narrow maintainable solution because the generic Gateway display-name contract already exists and should not require a protocol redesign.
