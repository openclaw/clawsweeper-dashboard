Keep open: current `main` and v2026.5.22 still expose only the fixed Gateway-visible TUI label, while the existing Gateway protocol/client seam can carry a display name once maintainers choose the public UX and privacy policy.

## What This Changes

The issue requests a TUI option such as `--client-name` or `--username` to customize the Gateway-visible display name for each TUI instance.

## Best Possible Solution

Add a TUI-owned, sanitized display-name override wired into `GatewayClient.clientDisplayName`, while preserving stable TUI client identity and per-connection `instanceId` semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI/options path does not expose a display-name override, while `GatewayChatClient` always sends the fixed `openclaw-tui` display name. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name behavior, and sanitization/privacy policy first.
