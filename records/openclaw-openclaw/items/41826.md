## Summary

Keep open: current main and v2026.5.20 still hard-code the Gateway-backed TUI display label as `openclaw-tui`, while the TUI CLI, launch options, runtime handoff, and docs expose no custom label override. Gateway already has a `clientDisplayName` seam, so the remaining work is a TUI UX/product decision rather than a missing protocol capability.

## What This Changes

Request to add a custom display-name option or generated unique label for Gateway-backed `openclaw tui` instances so multiple TUI windows are distinguishable in Gateway presence/session views.

## Best Possible Solution

Add a TUI-owned, sanitized client display-name option that feeds `GatewayClient.clientDisplayName`, update docs and focused TUI/Gateway tests, and keep the stable TUI client id plus unique instance id semantics unchanged.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect Gateway presence, but the current code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing `GatewayClient.clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name policy, and sanitization rules before implementation.
