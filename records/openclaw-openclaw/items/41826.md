## Summary

Keep open: current main still has the TUI Gateway display name hard-coded as `openclaw-tui`, while the existing Gateway client/protocol can already carry a display name. This is a concrete feature gap, but it needs maintainer UX policy for the public flag name, generated defaults, and sanitization before implementation.

## What This Changes

The issue requests a TUI option such as `--client-name` or `--username` so multiple Gateway-connected TUI windows can show distinct client display names.

## Best Possible Solution

Add a TUI-owned, sanitized display-name option that feeds `GatewayClient.clientDisplayName`, with focused CLI/client coverage and docs while preserving stable client id and unique instance id semantics.

## Reproduction Assessment

Yes, as a source-level feature gap: the TUI CLI has no display-name flag, `TuiOptions` cannot carry one, and `GatewayChatClient` always sends `clientDisplayName: "openclaw-tui"`. A live repro would start multiple Gateway-backed TUI instances and inspect presence, but the code path is direct.

## Solution Assessment

Unclear pending maintainer UX choice. The existing Gateway `clientDisplayName` seam is the narrow implementation path, but maintainers should decide the flag name, generated-name policy, and sanitization rules before implementation.
