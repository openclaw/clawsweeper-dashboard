## Summary

Keep open: current main and latest release v2026.6.8 still hardcode Gateway-backed TUI clients as `openclaw-tui`, with no CLI, docs, or `TuiOptions` path for a custom display label. The request is source-reproducible and useful, but it adds public TUI CLI/config surface and needs maintainer direction on naming, validation, scope, and privacy-safe behavior before implementation.

## What This Changes

Request a public `openclaw tui` option or equivalent behavior that lets each TUI instance advertise a custom Gateway-visible display name.

## Best Possible Solution

Add an explicit opt-in TUI display-name override that keeps `openclaw-tui` as the default, reuses the existing Gateway display-name seam, and documents privacy-safe usage after maintainers choose the public option shape.

## Reproduction Assessment

Yes. Source inspection on current main shows the Gateway-backed TUI hardcodes `openclaw-tui`, while the CLI docs/options and `TuiOptions` expose no override path.

## Solution Assessment

Unclear pending maintainer approval. Reusing the existing Gateway `displayName` field is the narrow technical route, but the public flag/config name, validation, Gateway-only behavior, and privacy defaults need product direction first.
