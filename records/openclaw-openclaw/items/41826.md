## Summary

Keep open: current main still hardcodes the TUI gateway display label and has no CLI or docs surface for overriding it, so this is a valid user-facing feature request rather than an implemented or stale cleanup close.

## What This Changes

The issue requests a configurable or automatically unique display name for `openclaw tui` clients so multiple TUI windows can be distinguished in gateway presence/session lists.

## Best Possible Solution

Add a TUI-owned display-name contract that flows from the CLI into `GatewayClient.clientDisplayName`, update the public TUI docs and focused tests, and leave gateway client IDs/instance IDs unchanged.

## Reproduction Assessment

Yes, from source inspection: current main always sends `clientDisplayName: "openclaw-tui"`, and `openclaw tui` has no option to override it. A live repro would be to start two gateway-backed TUI instances and inspect `system-presence`, but the code path is already clear.

## Solution Assessment

Unclear pending maintainer UX choice. A TUI-specific display-name option is the narrowest likely implementation, while `--username` or automatic hostname/TTY defaults need a product decision before code work.
