## Label Justifications

- `P2`: This is a normal, bounded TUI usability feature with limited blast radius and no urgent regression signal.
- `impact:other`: The concrete impact is operator confusion when multiple TUI clients have the same Gateway-visible label, which does not fit a more specific owned impact label.

## Summary

Current main still gives every Gateway-connected TUI the fixed display label `openclaw-tui`, and neither the TUI CLI nor its option types expose a custom label. The existing Gateway transport already supports `clientDisplayName`, so the request remains technically bounded, but adding a public flag and choosing privacy-safe defaults are product decisions rather than a broken-contract bug.

## What This Changes

Add a way for each TUI instance to use a custom Gateway-visible client display name instead of always appearing as `openclaw-tui`.

## Best Possible Solution

Expose an explicit opt-in TUI display-name option through `TuiOptions` and the existing Gateway `clientDisplayName` field, preserve `openclaw-tui` as the default, and do not auto-publish hostname, TTY, or working-directory metadata.

## Reproduction Assessment

Yes, source-reproducible: current main hardcodes the TUI Gateway display label and exposes no matching CLI or TUI option. A live multi-TUI Gateway session was not run, but the connection construction directly establishes the reported limitation.

## Solution Assessment

Unclear pending product approval. Reusing the existing Gateway display-name field is the narrowest technical solution, but maintainers must choose the public option name, validation, and privacy-safe default before implementation.
