## Summary

Keep open. Current main and the latest release still pass the parent reply operation signal into preflight compaction and memory flush, preflight failures can escape into the outer `run_failed` path before the main turn starts, and the only linked fix PR is closed unmerged with an incomplete signal-only approach.

## What This Changes

The issue asks for preflight compaction and memory-flush auxiliary work to stop cascading cancellation or failure into the main reply operation when maintenance work fails or times out.

## Best Possible Solution

Make preflight compaction and memory flush true auxiliary work: parent abort should still cancel them, but their own timeout or failure should be caught, logged, covered by regression tests, and unable to fail or mutate the parent reply operation.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection does show a current-main path where preflight compaction can throw before the main turn and reach the outer `run_failed` handler, matching the reported failure class.

## Solution Assessment

No for the closed signal-only PR. The safer repair is to downgrade auxiliary preflight failures and isolate maintenance-run operation/backend state, while keeping parent/user abort propagation and avoiding unrelated gateway config changes.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79390 on current main. The narrow bug is that pre-run maintenance is not isolated enough from the parent reply operation: `runPreflightCompactionIfNeeded` passes `params.replyOperation.abortSignal` to `compactEmbeddedPiSession` and awaits it before the main turn, and `runMemoryFlushIfNeeded` passes both the parent signal and `replyOperation` into the maintenance embedded run. Add focused regression coverage that simulates preflight compaction timeout/rejection and memory-flush abort/failure, proving that auxiliary maintenance failures are logged/downgraded and the parent turn continues unless the parent/user aborts. Implement the smallest isolation needed so parent abort still cancels maintenance, but maintenance timeout/failure cannot fail/abort or attach mutable backend state to the parent reply operation. Do not add gateway config allowlist changes, compaction threshold changes, or unrelated behavior. Add a changelog entry if the fix is user-visible under repo policy.
