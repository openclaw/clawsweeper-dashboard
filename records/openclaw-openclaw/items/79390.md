## Summary

Keep open. Current main and the latest release still pass the parent reply operation signal into preflight compaction and memory flush, preflight errors can still escape into the outer run failure path, and the related fix PR is closed unmerged with no canonical replacement found.

## What This Changes

The issue reports that preflight compaction and memory flush share the main reply operation abort signal, allowing auxiliary compaction failures to abort the main turn.

## Best Possible Solution

Make preflight compaction and memory flush true auxiliary work: parent/user abort should still cancel them, but their own timeout or failure should be logged, covered by regression tests, and unable to fail or mutate the parent turn's reply operation.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection on current main shows a clear preflight failure path where compaction is awaited before the main turn without a local catch and can reach the outer `run_failed` handler.

## Solution Assessment

No for the closed PR's signal-only approach. The safer fix is to catch and downgrade auxiliary preflight failures and isolate any mutable maintenance-run operation/backend state, without unrelated gateway config changes.
