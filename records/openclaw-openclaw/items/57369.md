## Summary

Keep open. Current main and the latest checked release still expose only `daily` and `idle` session reset modes, and the related `off` implementation attempts are closed unmerged while this issue remains the canonical manual-only reset request.

## What This Changes

Add a first-class `session.reset.mode: "never"` option for base, type, and channel reset policies so sessions only reset through explicit user/operator actions.

## Best Possible Solution

Define one additive manual-only reset mode across exported types, Zod validation, runtime freshness, generated config metadata/help, focused tests, docs, and changelog while preserving existing daily, idle, legacy idle, provider-owned implicit-expiry, and explicit `/new`/`/reset` behavior.

## Reproduction Assessment

Yes, by source inspection. A config using `mode: "never"` reaches the session reset schema, which accepts only `daily` and `idle`, and the runtime freshness policy has no manual-only branch.

## Solution Assessment

Yes in principle: a first-class manual-only reset mode is narrower and clearer than a huge idle timeout sentinel. The exact public literal and strictness around unused fields should be decided before implementation.
