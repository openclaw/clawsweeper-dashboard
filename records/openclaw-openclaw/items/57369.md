## Summary

Keep open. Current main and the latest shipped tag still expose only `daily` and `idle` reset modes, and the related `off` request was closed in favor of this canonical manual-only reset thread; implementation still needs a maintainer API/product decision for the public literal and validation semantics.

## What This Changes

The issue requests a first-class `session.reset.mode: "never"` option for manual-only session reset behavior across base, type, and channel policies.

## Best Possible Solution

Add one explicit manual-only reset mode across exported types, Zod validation, runtime freshness evaluation, config help/docs, focused session tests, and changelog after maintainers choose the public literal and field semantics.

## Reproduction Assessment

Yes, by source inspection. A config using `mode: "never"` reaches a strict schema that accepts only `daily` and `idle`, and the runtime freshness policy has no manual-only branch.

## Solution Assessment

Yes in principle: a first-class manual-only reset mode is clearer than giant idle timeout sentinels. The exact public literal and validation semantics need maintainer API/product direction before implementation.
