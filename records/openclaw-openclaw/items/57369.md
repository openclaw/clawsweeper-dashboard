## Summary

Keep open. Current main still lacks a public or runtime `mode: "never"` session reset policy: config types, zod validation, generated schema/help, runtime policy typing, tests, and docs remain limited to `daily` and `idle`. The request is valid, narrow, and suitable for a focused repair PR.

## What This Changes

Add a first-class `mode: "never"` session reset policy for base, type, and channel overrides so sessions reset only through explicit operator action.

## Best Possible Solution

Implement `never` as an additive session reset mode across public config types, zod validation, generated schema/help, runtime policy resolution, freshness evaluation, docs, and focused precedence tests while preserving existing daily, idle, legacy `idleMinutes`, `/new`, and `/reset` behavior.

## Reproduction Assessment

Yes. A config with `session.reset.mode: "never"`, or the same value under `resetByType` or `resetByChannel`, follows a deterministic source path into `SessionResetConfigSchema`, which currently accepts only `daily` and `idle`.

## Solution Assessment

Yes. A first-class additive union member and explicit no-expiry policy branch is the narrowest maintainable fix; the giant idle timeout workaround remains semantically wrong and is not represented in public config validation or docs.
