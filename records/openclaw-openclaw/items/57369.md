## Summary

Keep open. Current main still supports only `daily` and `idle` session reset modes in the public type, Zod validation, runtime freshness policy, config help, and docs; the earlier `off` implementation PR closed unmerged, and the duplicate `off/manual` issue points back to this canonical request.

## What This Changes

Request support for `session.reset.mode: "never"` in base, type, and channel reset policies for manual-only session reset behavior.

## Best Possible Solution

Add one explicit manual-only reset mode across exported types, validation, runtime freshness, generated config metadata/help, tests, docs, and changelog while preserving existing daily, idle, legacy idle, provider-owned implicit-expiry, and explicit `/new`/`/reset` behavior.

## Reproduction Assessment

Yes, by source inspection. A config using `mode: "never"` reaches `SessionResetConfigSchema`, which currently accepts only `daily` and `idle`, and the runtime policy has no manual-only freshness branch.

## Solution Assessment

Yes, if maintainers accept the requested literal. A first-class additive union member with an always-fresh runtime branch is narrower and clearer than encoding manual-only intent with a huge idle timeout.
