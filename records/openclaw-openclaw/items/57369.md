## Summary

Keep open. Current main and the latest release still expose only `daily` and `idle` reset modes, the previous `off` implementation PR was closed unmerged, and the related duplicate issue was already closed in favor of this canonical manual-only reset request.

## What This Changes

Request adds a first-class `session.reset.mode: "never"` manual-only reset policy for base, type, and channel session reset overrides.

## Best Possible Solution

Add one explicit manual-only mode across public types, Zod validation, runtime freshness, SDK exports, config help/docs, focused tests, and changelog while preserving daily/idle, legacy idle, provider-owned implicit-expiry, and explicit `/new` or `/reset` behavior.

## Reproduction Assessment

Yes, by source inspection. A config using `mode: "never"` reaches a schema that accepts only `daily` and `idle`, and the runtime freshness policy has no manual-only branch.

## Solution Assessment

Yes in principle: a first-class manual-only reset mode is clearer and safer than giant idle timeout sentinels. The exact public literal and strictness around unused fields need maintainer API/product direction before implementation.
