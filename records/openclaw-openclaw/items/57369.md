## Summary

Keep open. Current main and v2026.5.2 still expose only `daily` and `idle` session reset modes in public types, validation, generated schema/help, runtime freshness, and docs; the duplicate #76534 is closed to this canonical request and PR #64028 did not ship.

## What This Changes

Add a first-class `mode: "never"` session reset policy for base, type, and channel overrides so sessions reset only through explicit operator action.

## Best Possible Solution

Implement `never` as an additive reset mode across config types, Zod validation, generated schema/help, runtime policy resolution, docs, changelog, and focused precedence tests while preserving existing daily, idle, legacy idle, `/new`, and `/reset behavior.

## Reproduction Assessment

Yes, by source inspection rather than a live run. A config with `session.reset.mode: "never"`, or the same value under `resetByType` or `resetByChannel`, deterministically reaches `SessionResetConfigSchema`, which currently accepts only `daily` and `idle`.

## Solution Assessment

Yes. A first-class additive union member and explicit no-expiry policy branch is the narrowest maintainable solution; the giant idle-timeout workaround is semantically wrong and is not represented in validation or docs.
