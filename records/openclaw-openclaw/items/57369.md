## Summary

Keep open. Current main still exposes only `daily` and `idle` reset modes in the public config type, Zod validation, runtime freshness policy, config help, and docs; the related `off` implementation PR was closed unmerged and the duplicate `off/manual` issue was closed to this canonical request.

## What This Changes

Request support for `session.reset.mode: "never"` across base, type, and channel reset policies so automatic resets are disabled until an explicit `/new` or `/reset`.

## Best Possible Solution

Add one canonical manual-only reset mode to the public config contract and runtime policy, preserving existing daily, idle, legacy idle, provider-owned implicit-expiry behavior, and explicit `/new`/`/reset` semantics.

## Reproduction Assessment

Yes, by source inspection. A config using `mode: "never"` under `session.reset`, `session.resetByType`, or `session.resetByChannel` reaches `SessionResetConfigSchema`, which currently accepts only `daily` and `idle`.

## Solution Assessment

Yes. A first-class additive union member with an explicit always-fresh runtime branch is the narrowest maintainable fix; the large-idle workaround is not a clear or durable public contract.
