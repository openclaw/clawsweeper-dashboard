## Summary

Keep open. Current main and the latest release still only support `daily` and `idle` reset policies, and related cleanup has already treated this issue as the canonical narrow tracker for first-class manual-only reset mode.

## What This Changes

The issue requests an explicit `session.reset.mode: "never"` value for base, type, and channel reset policies so sessions reset only by explicit user/operator action.

## Best Possible Solution

Keep this as the canonical narrow manual-only reset-mode issue, then implement the approved public literal consistently across schema, exported types, resolver behavior, docs/help metadata, and regression tests.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the current gap: config validation accepts only `daily | idle`, the public reset type is only `daily | idle`, and the freshness evaluator only implements daily and idle automatic expiry.

## Solution Assessment

Unclear until maintainers choose the public config contract. A manual-only mode is a plausible narrow solution, but the best fix should settle the literal name and field-validation semantics before implementation.
