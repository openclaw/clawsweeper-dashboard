## Summary

Keep open. Current main and v2026.5.2 still expose only `daily` and `idle` reset modes across public types, validation, generated schema/help, runtime freshness, and docs; the related PR #64028 was closed unmerged and #76534 was closed to this canonical request.

## What This Changes

Add a first-class `session.reset.mode: "never"` policy for base, type, and channel reset config so sessions reset only through explicit manual mechanisms.

## Best Possible Solution

Implement an additive explicit no-expiry reset policy that keeps existing daily, idle, legacy idle, channel/type precedence, `/new`, and `/reset behavior unchanged.

## Reproduction Assessment

Yes, by source inspection. A config with `session.reset.mode: "never"`, or the same value under `resetByType` or `resetByChannel`, reaches `SessionResetConfigSchema`, which currently accepts only `daily` and `idle`.

## Solution Assessment

Yes. A first-class additive union member plus an explicit freshness branch that never marks the session stale is the narrowest maintainable fix; large idle-timeout sentinels should remain unnecessary workarounds.

Implement `session.reset.mode: "never"` as a backward-compatible explicit manual-only reset mode for openclaw/openclaw. Update the public config type, Zod schema, generated config schema/help, reset policy resolution/freshness evaluation, docs, changelog, and focused tests. The new mode must apply to `session.reset`, `session.resetByType`, and `session.resetByChannel`; it must prevent daily and idle automatic resets while preserving manual `/new` and `/reset`, legacy `session.idleMinutes`, existing daily/idle semantics, channel-over-type-over-base precedence, provider-owned CLI session behavior, and reset archive/maintenance behavior. Do not change default reset behavior or add broad session lifecycle policy changes.
