## Summary

Keep open. Current main still exposes only `daily` and `idle` session reset modes in the public type, zod validation, generated config schema, runtime policy, and docs, so the requested manual-only reset mode remains unimplemented; the closed #76534 appears to be the same need under different naming, making this the better canonical thread.

## What This Changes

Add a first-class `session.reset.mode: "never"` manual-only reset policy, including `resetByType` and `resetByChannel` overrides.

## Best Possible Solution

Implement `never` as an additive reset mode across config types, zod validation, generated schema/help, runtime freshness evaluation, docs, changelog, and focused precedence tests while preserving existing daily, idle, legacy idle, `/new`, and `/reset` behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: any `session.reset.mode: "never"` or equivalent type/channel override reaches `SessionResetConfigSchema`, whose mode union only accepts `daily` and `idle`.

## Solution Assessment

Yes. A first-class additive union member with an explicit no-expiry freshness path is the narrowest maintainable solution; the large idle-timeout workaround is semantically wrong and not documented as a real policy.
