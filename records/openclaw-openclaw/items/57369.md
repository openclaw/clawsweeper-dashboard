## Summary

Keep open. Current main and the latest shipped tag still expose only `daily` and `idle` reset modes, so the requested manual-only session reset behavior remains unimplemented and needs maintainer API/product direction.

## What This Changes

The issue requests adding `session.reset.mode: "never"` for base, type, and channel reset policies so sessions reset only through explicit user/operator action.

## Best Possible Solution

Add one explicit manual-only reset mode across exported types, Zod validation, runtime freshness evaluation, generated config help/schema, docs, focused session tests, and changelog while preserving existing `daily` and `idle` behavior.

## Reproduction Assessment

Yes, by source inspection. A config using `mode: "never"` reaches a strict schema that accepts only `daily` and `idle`, and the runtime freshness policy has no manual-only branch.

## Solution Assessment

Yes in principle: a first-class manual-only reset mode is clearer than giant idle timeout sentinels. The exact public literal and validation semantics need maintainer API/product direction before implementation.
