## Summary

Keep open. Current main still has the agent-run assistant-media append path that can persist fallback assistant text with an `:assistant-media` idempotency key when a media-bearing payload resolves to no playable media, and open unmerged PRs already target this issue.

## What This Changes

Prevent Control UI/WebChat from persisting duplicate gateway-injected assistant replies through the assistant-media/TTS transcript supplement path when no playable media is resolved.

## Best Possible Solution

Land one canonical gateway fix with a focused regression test and changelog entry that only appends supplemental agent-run assistant-media transcript entries when resolved non-text media content exists, while preserving normal model transcripts and non-agent command/TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final or block payload with visible text plus a stale or unembeddable media URL: `isMediaBearingPayload` passes, the media builder returns `null`, and the append path can persist fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. Guarding the supplemental append on resolved media content is the narrowest maintainable fix; because #73962 and #75481 already target that path, the right next step is to review and land one canonical PR rather than queue another repair branch.
