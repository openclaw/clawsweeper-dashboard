## Summary

Keep open. Current main still allows the agent-run assistant-media transcript path to append a gateway-injected assistant message from fallback text when the media reference does not resolve, and two open PRs already target the bug with closing syntax, so this issue should stay open until one canonical fix lands.

## What This Changes

Stop Control UI/WebChat from persisting duplicate gateway-injected assistant replies through the assistant-media/TTS delivery path when no playable media is available.

## Best Possible Solution

Land one canonical gateway fix with a focused regression test and changelog entry so supplemental agent-run assistant-media transcript entries are appended only when resolved non-text media content exists, while preserving normal model transcripts and non-agent command/TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final or block payload with visible text plus a stale or unembeddable media URL: `isMediaBearingPayload` passes, the media builder returns `null`, and current main can persist fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. The linked PR direction, gating the supplemental assistant-media append on resolved media content, is the narrowest maintainable fix because the normal model transcript already carries the agent text and command/non-agent media fallback can remain separate.
