## Summary

Keep open. Current main still has the gateway assistant-media transcript path that can persist a text-only fallback with an `:assistant-media` idempotency key when media references do not resolve, and open unmerged PRs #73962 and #75481 already target this issue with closing syntax.

## What This Changes

Fix the Control UI/WebChat assistant-media delivery path so stale or disabled-TTS media supplements do not persist duplicate text-only assistant replies.

## Best Possible Solution

Land one canonical gateway fix with a focused regression test and changelog entry that appends supplemental agent-run assistant-media transcript entries only when resolved non-text media content exists, while preserving normal model transcripts and non-agent command/TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final/block payload with visible text plus a stale or unembeddable media URL: `isMediaBearingPayload` passes, the media builder returns `null`, and the append path can persist fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. The linked PR direction, gating the supplemental assistant-media append on resolved media content, is the narrowest maintainable fix; the issue should stay open until one of those PRs or an equivalent patch lands.
