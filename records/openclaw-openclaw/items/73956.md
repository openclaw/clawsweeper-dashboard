## Summary

Keep open. Current main still has the agent-run assistant-media append path that can fall back to visible assistant text and persist a `:assistant-media` transcript entry when no embeddable media resolves, and open unmerged PRs already explicitly target this issue.

## What This Changes

Control UI/WebChat should stop creating duplicate assistant transcript messages from the gateway assistant-media delivery path when there is no playable media to persist.

## Best Possible Solution

Land one canonical gateway fix with a focused regression test and changelog entry: supplemental agent-run assistant-media transcript entries should only be appended when resolved non-text media content exists, while normal model transcripts and non-agent command/TTS fallback behavior remain intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final or block payload with visible text plus a stale or unembeddable media URL: the payload passes `isMediaBearingPayload`, the media builder can return `null`, and current main can still append fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. Guarding the supplemental append on resolved media content is the narrowest maintainable fix; since #73962 and #75481 already target this path, the right next step is to review and land one canonical PR rather than queue another repair branch.
