## Summary

Keep open. Current main still has a source-level path where an agent-run media-bearing reply can append a supplemental `:assistant-media` assistant transcript using fallback text when no playable media resolves, and two open PRs already target this issue with closing syntax.

## What This Changes

Control UI/WebChat should avoid storing a second gateway-injected assistant transcript entry when the assistant-media path has no playable media after TTS has been disabled.

## Best Possible Solution

Land one canonical, narrow gateway fix that appends the agent-run assistant-media supplemental transcript only when resolved non-text media content exists, while preserving normal model transcripts and non-agent command/TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final payload with visible text plus a stale or unembeddable media URL: the media-bearing guard passes, media resolution returns no media message, and the append path can persist fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes for the linked PR direction, no for current main. Guarding the supplemental append on resolved media content is the narrowest maintainable fix; maintainers should review and land one of the existing canonical PRs rather than queue another repair.
