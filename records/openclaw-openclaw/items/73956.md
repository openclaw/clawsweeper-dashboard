## Summary

Keep open. Current main still has the assistant-media append path that can persist a fallback text transcript when a media-bearing agent payload does not resolve to playable media, and open unmerged PRs #73962 and #75481 already target this issue with the right narrow fix direction.

## What This Changes

The issue asks OpenClaw to stop Control UI/WebChat from persisting duplicate assistant replies through the assistant-media transcript path when TTS/media is disabled or unresolved.

## Best Possible Solution

Land one canonical gateway fix with a focused stale-media regression test and changelog entry, preserving resolved media supplements and command/non-agent TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final/block payload with visible text plus a stale or unembeddable media URL: isMediaBearingPayload passes, the media builder returns null, and the append path can persist fallback text with an :assistant-media idempotency key.

## Solution Assessment

Yes. Gating the supplemental assistant-media append on resolved non-text media content, as the linked PRs propose, is the narrowest maintainable fix for this path.
