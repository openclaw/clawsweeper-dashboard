## Summary

Keep open. Current main still has a source-level path that can persist a text fallback as a `:assistant-media` assistant message when media resolution fails, and open unmerged PRs #73962 and #75481 already target the narrow fix.

## What This Changes

The report asks OpenClaw to stop Control UI/WebChat from storing duplicate assistant replies through the `:assistant-media` transcript path after TTS/media becomes disabled or unplayable.

## Best Possible Solution

Land one canonical gateway fix that appends `:assistant-media` only when resolved non-text media content exists, with stale-media regression coverage and a changelog entry while preserving command/non-agent TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final/block payload with visible text plus a stale or inaccessible `mediaUrl`: `isMediaBearingPayload` passes, media message construction returns null, and the append helper can persist fallback text with a `:assistant-media` key.

## Solution Assessment

Yes. Gating the supplemental append on resolved display media is the narrowest maintainable fix because the normal model transcript already carries the assistant text and command/non-agent TTS fallback can remain unchanged.
