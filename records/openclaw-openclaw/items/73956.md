## Summary

Keep open. Current main still has the gateway assistant-media transcript path that can persist fallback text as a supplemental assistant message, and the issue is paired with open closing PRs that should be reviewed or superseded before the issue closes.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies caused by the gateway assistant-media delivery path persisting a second text-only assistant message after TTS/media state changes.

## Best Possible Solution

Land one canonical gateway fix that appends supplemental agent-run assistant-media transcript entries only when resolved non-text media content exists, while preserving normal model transcripts and non-agent command/TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final or block payload with visible text plus a stale or unembeddable media URL: the media-bearing guard passes, the media builder returns no media message, and the append path can persist fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. Guarding the supplemental append on resolved media content is the narrowest maintainable direction; #73962, #74502, and #75481 already target that path, so maintainers should review and land one canonical PR rather than open another repair lane.
