## Summary

Keep open. The PR addresses a real current-main CLI/update mismatch, but the proposed terminal output still renders dirty worktree entries without sanitizing user-controlled filenames and leaves one misleading clean-check failure label.

## What This Changes

The branch forwards update-step stdout tails, uses clean-check stdout to report dirty entries in progress and skipped output, updates dirty-worktree guidance, adds focused tests, and adds a changelog fix entry.

## Best Possible Solution

Keep the stdoutTail plumbing and dirty-entry guidance, but sanitize every dirty status line before coloring/logging and use a failure-specific clean-check label before merge.

## Reproduction Assessment

Yes. A high-confidence reproduction is a git checkout where `git status --porcelain` emits dirty stdout while exiting 0; current main then reports the clean-check spinner as clean while the runner skips with `reason: dirty`.

## Solution Assessment

No, not as-is. The proposed implementation shape is narrow and maintainable, but it needs terminal-safe rendering for dirty entries and a neutral clean-check failure label before it is the best fix.

## Review Findings

  - body: This dirty warning path renders `git status --porcelain` lines directly through terminal coloring. Filenames can contain terminal control text, so sanitize each line before applying `theme.warn()`/`theme.muted()` or logging it.
  - confidence: 0.86

  - body: `resolveStepDisplay()` falls through to `getStepLabel()` for a nonzero `clean check`, which can make the step summary say `Working directory is clean` with a failure icon. Return a failure-specific label such as `Failed to check working directory`.
