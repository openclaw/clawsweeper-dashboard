## Summary

Keep open. The PR addresses a real current-main CLI/update mismatch, but the proposed dirty-entry output prints `git status --porcelain` filenames without terminal sanitization and leaves a misleading clean-check failure label path before merge.

## What This Changes

The PR forwards update-step `stdoutTail`, uses it to show dirty clean-check progress and skipped-result entries, updates focused tests, and adds a changelog fix entry.

## Best Possible Solution

Land the narrow stdout-tail dirty-check fix after sanitizing every dirty status line with the existing terminal-safe helper and returning a neutral clean-check failure label, with focused update CLI tests covering both paths.

## Reproduction Assessment

Yes. A high-confidence reproduction is a git install where `git status --porcelain` emits an untracked or modified entry while exiting 0; current main renders the clean-check spinner from exit code while the runner skips with `reason: dirty`.

## Solution Assessment

No, not as-is. The stdout-tail plumbing is the right shape, but the rendering must sanitize untrusted filenames before theme coloring and avoid the clean success label for clean-check failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

  - body: This new dirty warning path logs `git status --porcelain` lines directly through `theme.warn(line)`. Filenames can contain ANSI/control text, so sanitize each line before applying theme coloring or logging it.

  - confidence: 0.88

  - body: For a non-zero `clean check`, `resolveStepDisplay()` falls through to `getStepLabel()`, which still says `Working directory is clean`. Return a failure-specific label such as `Failed to check working directory`.
