## Summary

Keep this issue open. Current main has useful chat-history bounding, sanitization, and a separate paginated REST/SSE session-history endpoint, but the affected Control UI WebChat path still uses the WebSocket `chat.history` latest-window request, has no cursor/hasMore UI path, still materializes the transcript before trimming, and still renders a capped Lit list instead of paginated or virtualized history.

## What This Changes

The issue requests WebChat session-history loading and rendering changes so accumulated chat history no longer freezes the Control UI.

## Best Possible Solution

Keep the issue open and implement a current-main-native WebChat history window: add an additive cursor/hasMore contract to the WebSocket `chat.history` path or route Control UI history through the existing paginated `/sessions/:key/history` endpoint, then add load-earlier or infinite-scroll UI behavior plus virtualization or chunked rendering coverage for realistic long transcripts with large tool results.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is to follow Control UI `loadChatHistory()` to WebSocket `chat.history`: current main still requests `limit: 200`, lacks cursor state, and renders a capped non-virtualized list; a live browser freeze would require a large transcript fixture outside this read-only review.

## Solution Assessment

No current-main fix is sufficient yet. The existing response caps and REST/SSE paginated endpoint reduce some blast radius, but the reported WebChat surface still needs an integrated paginated/virtualized UI path rather than only hidden server-side limits.
