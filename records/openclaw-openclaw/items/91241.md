Keep open for maintainer merge review: the patch is a narrow, source-backed fix for retry accounting on unattempted outbound recovery budget deferrals, and I found no blocking code or security findings. ## What This Changes

The branch removes the outbound recovery path that calls `failDelivery` for entries deferred solely by `maxRecoveryMs` and updates the recovery regression to expect `retryCount` to remain `0`.

## Best Possible Solution

Land the narrow retry-accounting fix after required checks pass, and leave the transport-readiness/restart recovery behavior tracked in https://github.com/openclaw/openclaw/issues/91212.

## Reproduction Assessment

Yes for a source-level reproduction: current main enters the `maxRecoveryMs` deadline branch before any delivery call and calls `failDelivery` on every remaining entry, whose storage contract increments `retryCount` and `lastAttemptAt`. I did not execute the scenario locally because this review is read-only.

## Solution Assessment

Yes. Removing `failDelivery` only from budget deferral is narrower than changing failed-attempt accounting globally, and the session-delivery sibling already treats budget exhaustion as a non-mutating deferral.
