## Summary

Keep this PR open. The security hardening goal is valid, but the patch still enforces a protected-file policy before the final host/sandbox/apply_patch mutation target is known, lacks an explicit approval/exception contract, and has no after-fix real behavior proof from an external contributor.

## What This Changes

The PR adds protected-instruction-file guards for write/edit and apply_patch tools, plus tests for SOUL.md/CLAUDE.md/MEMORY.md-style targets.

## Best Possible Solution

Enforce one protected-instruction-file mutation policy at the resolved host, sandbox bridge, and apply_patch write/remove/move boundaries, with documented approval and system-flow exception semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a non-protected raw or lexical path can pass the new checks and later resolve through write/edit/apply_patch to a protected instruction file.

## Solution Assessment

No. Wrapper-level basename checks are not the best security boundary; the check needs to run where the final mutation target is known, with explicit approval semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve write targets before checking protected names:** `src/agents/pi-tools.instruction-file-guard.ts:50-53`
  - body: `safeRealpath(filePath)` runs on the raw tool argument before write/edit resolves it relative to the workspace, sandbox root, container workdir, or bridge. A benign-looking alias can pass this wrapper while the actual write still targets a protected instruction file, so the guard needs the same resolved target that the write will use.
  - confidence: 0.86

  - body: This guard checks parsed hunk paths before `resolvePatchPath` decides the host or sandbox target written by `fileOps`. That leaves add/update/delete/move operations with a lexical check instead of one canonical protected-target policy at the mutation boundary.
  - confidence: 0.85
- **[P2] Add an approval path for legitimate instruction edits:** `src/agents/pi-tools.instruction-file-guard.ts:59-63`
  - body: The PR says protected instruction files are blocked without explicit approval, but the implementation always throws. Current docs still describe agent-driven SOUL.md and MEMORY.md-style updates, so the approval or scoped exception contract needs to be defined before this can merge.
  - confidence: 0.78
