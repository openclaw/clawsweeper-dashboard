## Summary

Keep this PR open. Current main and the latest release still lack an equivalent instruction-file mutation guard, but the proposed patch is not merge-ready because its checks are split across tool wrappers and patch parsing rather than enforced at the final resolved mutation boundary.

## What This Changes

The PR adds protected-basename guards for agent write/edit tools, adds lexical apply_patch hunk checks for the same instruction files, and adds regression tests for those paths.

## Best Possible Solution

Keep the security hardening effort, but enforce it as a central resolved-target mutation policy shared by host write/edit, sandbox write/edit, apply_patch add/update/delete/move, and the sandbox bridge. Add an explicit operator approval or scoped exception model for legitimate persona, bootstrap, and memory promotion workflows, then document the behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: on current main, createOpenClawCodingTools exposes write/edit and apply_patch mutation paths without protected-instruction checks; on the PR branch, alias/canonical-target bypasses follow from the new checks running before the same final path resolution used by the existing mutation code.

## Solution Assessment

No. The PR is directionally useful, but the current implementation is not the best fix because it wraps selected tools and parses patch text instead of enforcing one post-resolution write policy with an approval/exception contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve instruction paths before canonical guard checks:** `src/agents/pi-tools.instruction-file-guard.ts:50`
  - body: `safeRealpath(filePath)` runs on the raw tool argument before the write/edit tool resolves it relative to the workspace or sandbox root. For relative agent paths, the realpath probe can check the gateway process cwd instead of the file the tool will mutate, so alias paths can pass the guard and still write through the underlying host/sandbox operation. Move this check to the resolved mutation target or pass the root-aware resolver into the guard.
  - confidence: 0.86
- **[P1] Check apply_patch targets after path resolution:** `src/agents/apply-patch.ts:155`
  - body: The apply_patch guard only checks `hunk.path` and `hunk.movePath` lexically, then later `resolvePatchPath` decides the actual file that is written, removed, or moved. When apply_patch is allowed outside the workspace, a benign-looking alias path can resolve to a protected instruction file after this check. The guard needs to run on the resolved target, using the same canonical/inode policy as write/edit.
  - confidence: 0.84
- **[P2] Add an approval path for legitimate instruction edits:** `src/agents/pi-tools.instruction-file-guard.ts:61-66`
  - body: The PR says instruction-file writes are blocked without explicit approval, but the code always throws and does not provide an approval or scoped exception path. Existing docs tell users to have the agent rewrite `SOUL.md`, and memory promotion flows write `MEMORY.md`, so this would regress legitimate operator-directed workflows unless those paths can be approved or excluded deliberately.
  - confidence: 0.78
