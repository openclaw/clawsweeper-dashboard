## Summary

Keep open: the security goal is valid, but the proposed control is still enforced before the final write target is known and it lacks the approval contract and real behavior proof needed for a security-sensitive PR.

## What This Changes

The branch adds an instruction-file guard module, wraps agent write/edit tools, checks apply_patch hunk paths, and adds tests for protected bootstrap filenames.

## Best Possible Solution

Enforce a shared resolved-target instruction-file mutation policy at the actual host, sandbox, bridge, and apply_patch write boundaries, with an explicit operator approval or trusted system-flow exception contract and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR checks raw or parsed paths before current main resolves write/edit/apply_patch targets, so a benign-looking input can differ from the final mutation target.

## Solution Assessment

No. Wrapper-level basename checks are not the best way to solve this security issue; the narrow maintainable fix is a resolved-target policy at the mutation boundary plus explicit approval semantics.

  - confidence: 0.85

  - body: The PR says protected instruction files are blocked without explicit approval, but the implementation always throws. Current docs still describe agent-driven `SOUL.md` rewrites and `MEMORY.md` distillation, so the approval or trusted-flow exception contract needs to be defined before merge.
  - confidence: 0.78
