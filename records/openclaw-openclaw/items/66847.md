## Summary

Keep open. The security hardening direction is useful, but the PR still checks protected instruction files before the final resolved mutation boundary and does not define the explicit approval path its own description promises.

## What This Changes

The PR adds protected-basename guards for agent write/edit tools, adds lexical apply_patch hunk checks for the same instruction files, and adds regression tests for those paths.

## Best Possible Solution

Enforce one central resolved-target mutation policy shared by host write/edit, sandbox write/edit, apply_patch add/update/delete/move, and sandbox bridge writes, with an explicit operator approval or scoped system-flow exception model and documentation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: on the PR branch, a non-protected alias path can pass the new lexical/raw-path checks and then be resolved or followed by the underlying write/edit/apply_patch operation to mutate a protected instruction file.

## Solution Assessment

No. The PR is directionally useful, but wrapping selected tools and parsing patch text before final path resolution is not the narrowest maintainable security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve write targets before checking protected names:** `src/agents/pi-tools.instruction-file-guard.ts:49-52`
  - body: `safeRealpath(filePath)` runs on the raw tool argument before the write/edit tool resolves it relative to the workspace, sandbox root, container workdir, or host path rules. A relative or alias path with a non-protected basename can pass this wrapper and still be written by the underlying operation to a protected instruction file, so the guard needs to run on the same resolved target the write actually mutates.
  - confidence: 0.86
- **[P1] Check apply_patch targets after path resolution:** `src/agents/apply-patch.ts:148-153`
  - body: The new apply_patch guard checks `hunk.path` and `hunk.movePath` lexically, then `resolvePatchPath` later decides the file that is written, removed, or moved. A benign-looking alias or resolved sandbox path can therefore bypass the protected-basename check, so apply_patch needs the same post-resolution target policy as write/edit.
  - confidence: 0.85
- **[P2] Add an approval path for legitimate instruction edits:** `src/agents/pi-tools.instruction-file-guard.ts:59-64`
  - body: The PR says instruction-file writes are blocked without explicit approval, but the implementation always throws and offers no approval or scoped exception path. Existing bootstrap and memory workflows legitimately write SOUL.md, IDENTITY.md, and MEMORY.md, so this would regress operator-directed or system-owned updates unless that contract is defined.
  - confidence: 0.78
