## Summary

Keep open: current main still lacks an equivalent instruction-file write guard, but this PR remains unsafe as a security control because it checks raw or lexical paths before the final mutation target is resolved and does not define the promised approval path.

## What This Changes

The PR adds protected-basename guards for agent write/edit tools, adds apply_patch hunk checks for those instruction files, and adds regression tests.

## Best Possible Solution

Enforce one central resolved-target mutation policy shared by host write/edit, sandbox write/edit, apply_patch, and sandbox bridge writes, with an explicit operator approval or scoped system-flow exception model and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists on the PR branch: a non-protected raw or lexical path can pass the new wrapper/check and then be resolved by the underlying write/edit/apply_patch path to a protected instruction file.

## Solution Assessment

No. The PR is directionally useful, but selected tool wrappers and pre-resolution patch parsing are not the best security boundary; the policy needs to run where the final mutation target is known.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve write targets before checking protected names:** `src/agents/pi-tools.instruction-file-guard.ts:48-53`
  - body: `safeRealpath(filePath)` runs on the raw tool argument before write/edit resolves it relative to the workspace, sandbox root, container workdir, or bridge. A benign-looking alias can pass this wrapper and still mutate a protected instruction file, so the guard needs the same resolved target that the write will actually touch.
  - confidence: 0.86
- **[P1] Check apply_patch targets after path resolution:** `src/agents/apply-patch.ts:151-154`
  - body: This guard checks parsed hunk paths before `resolvePatchPath` decides the host or sandbox target that fileOps writes/removes. That leaves the security policy separate from the canonical patch mutation boundary, so add/update/delete/move targets need to be checked after resolution.
  - confidence: 0.85
- **[P2] Add an approval path for legitimate instruction edits:** `src/agents/pi-tools.instruction-file-guard.ts:59-64`
  - body: The PR says protected instruction files are blocked without explicit approval, but the implementation always throws. Existing bootstrap and memory workflows legitimately write SOUL.md, IDENTITY.md, and MEMORY.md, so the approval or scoped exception contract needs to be defined before this can merge.
  - confidence: 0.78
