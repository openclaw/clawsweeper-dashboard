## Summary

Keep open: this is security-sensitive core behavior, but the proposed guard is still bypassable before final path resolution, lacks the explicit approval semantics described by the PR, and has no real behavior proof from a real agent session.

## What This Changes

Adds protected-instruction-file guards for write/edit and apply_patch tools, with tests for SOUL.md/CLAUDE.md/MEMORY.md-style targets.

## Best Possible Solution

Enforce a shared resolved-target mutation policy at the actual host, sandbox, bridge, and apply_patch write boundaries, with an explicit operator approval or system-flow exception contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-protected raw or lexical path can pass the new checks and later resolve through write/edit/apply_patch to a protected instruction file.

## Solution Assessment

No. Wrapper-level basename checks are not the best security boundary; the check needs to run where the final mutation target is known, with explicit approval semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve write targets before checking protected names:** `src/agents/pi-tools.instruction-file-guard.ts:50-53`
  - body: `safeRealpath(filePath)` runs on the raw tool argument before write/edit resolves it relative to the workspace, sandbox root, container workdir, or bridge. A benign-looking alias can pass this wrapper while the actual write still targets a protected instruction file, so the guard needs the same resolved target that the write will use.
  - confidence: 0.86
- **[P1] Check apply_patch targets after path resolution:** `src/agents/apply-patch.ts:151-155`
  - body: This guard checks parsed hunk paths before `resolvePatchPath` decides the host or sandbox target written by `fileOps`. That leaves add/update/delete/move operations with a lexical check instead of one canonical protected-target policy at the mutation boundary.
  - confidence: 0.85
- **[P2] Add an approval path for legitimate instruction edits:** `src/agents/pi-tools.instruction-file-guard.ts:59-63`
  - body: The PR says protected instruction files are blocked without explicit approval, but the implementation always throws. Current docs still describe agent-driven SOUL.md and MEMORY.md updates, so the approval or scoped exception contract needs to be defined before this can merge.
  - confidence: 0.78
