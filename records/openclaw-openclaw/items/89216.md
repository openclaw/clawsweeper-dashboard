## Label Justifications

- `P1`: The report describes a broken Matrix channel workflow where room messages can consistently route to the wrong agent for affected users.
- `impact:session-state`: The failure creates or uses Matrix room sessions under `main` instead of the configured per-room agent.
- `impact:message-loss`: Misrouting Matrix room messages can send responses through the wrong agent or prevent the intended agent from handling the room.

## Summary

Keep open. Current main and v2026.6.8 still preserve omitted `match.accountId` as default-account only while Matrix routes with the active Matrix account id, so the reported named-account room binding can still miss; the open linked fix PR is a candidate, not a reason to close before merge.

## What This Changes

The issue requests that Matrix per-room agent bindings route incoming room messages to the configured agent instead of falling back to `main`.

## Best Possible Solution

Land a narrow Matrix-owned routing or account-selection fix that makes omitted-account room bindings work for named default or only Matrix accounts while preserving the shared default-account binding contract.

## Reproduction Assessment

Yes, by source inspection. Matrix passes its effective account id into `resolveAgentRoute()`, while omitted `match.accountId` bindings are indexed under `DEFAULT_ACCOUNT_ID`, so a named Matrix account can miss the reported room binding shape.

## Solution Assessment

Yes for the issue direction, but not for the broad wildcard implementation currently proposed in the linked PR. The best fix is Matrix-scoped handling that preserves explicit `accountId: "*"` as the shared any-account contract.
