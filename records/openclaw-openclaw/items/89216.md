## Label Justifications

- `P1`: The report describes a broken Matrix channel workflow where affected room messages can consistently route to the wrong agent.
- `impact:session-state`: The failure creates or uses Matrix room sessions under `main` instead of the configured per-room agent.
- `impact:message-loss`: Misrouted Matrix room messages can prevent the intended agent from handling the room and can send replies from the wrong session.

## Summary

Keep open. Current main still indexes omitted `match.accountId` bindings under the default account while Matrix routes with the active Matrix account id, and the open closing PR is a candidate fix rather than a reason to close this issue before merge.

## What This Changes

Matrix room messages should honor per-room agent bindings instead of silently routing bound rooms to the default `main` agent.

## Best Possible Solution

Land a narrow Matrix-owned routing or account-selection fix that makes omitted-account room bindings work for named default or only Matrix accounts while preserving the shared account-id binding contract.

## Reproduction Assessment

Yes, source-level. Matrix passes its effective account id into `resolveAgentRoute()`, while omitted `match.accountId` bindings are indexed under the default account, so a named Matrix account can miss the reported room binding shape.

## Solution Assessment

Yes for keeping this issue open, but the broad wildcard approach in the linked PR is not the safest fix. The maintainable path is Matrix-scoped handling that preserves explicit `accountId: "*"` as the shared any-account contract.
