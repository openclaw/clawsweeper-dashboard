## Label Justifications

- `P1`: The issue describes a broken Matrix channel workflow where room messages can consistently route to the wrong agent.
- `impact:session-state`: The failure creates or uses Matrix room sessions under `main` instead of the configured per-room agent.
- `impact:message-loss`: Misrouted Matrix room messages can prevent the intended agent from handling the room and can send replies from the wrong session.

## Summary

Keep open. Current main and the latest release still preserve default-only omitted account bindings while Matrix can route with a named effective account, and the open closing PR is only a candidate fix until it merges.

## What This Changes

Matrix per-room agent bindings should route incoming room messages to the configured agent instead of the default `main` agent.

## Best Possible Solution

Preserve the documented shared account-binding contract and land or rework a Matrix-scoped fix through https://github.com/openclaw/openclaw/pull/93338 or a narrow replacement.

## Reproduction Assessment

Yes, source-level. Matrix passes its effective account id into shared routing, while omitted `match.accountId` bindings are indexed under the default account, so a named Matrix account can miss the reported room binding shape.

## Solution Assessment

No for the broad wildcard approach in the linked PR. The best fix should be Matrix-scoped or account-selection scoped while preserving explicit `accountId: "*"` as the shared any-account contract.
