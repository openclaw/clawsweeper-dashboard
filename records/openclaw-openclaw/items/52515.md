## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep exec SecretRef leniency off global process env:** `src/commands/doctor.ts:81`
  - body: Setting `OPENCLAW_ALLOW_EXEC_SECRETREF_FAILURES` for the whole doctor run makes every strict `normalizeResolvedSecretInputString` caller downgrade unresolved exec refs while doctor is running, including plugin SDK/API-key paths. Current main already has explicit `allowExecSecretRefs` / `--allow-exec` handling for doctor gateway token audits, so this leniency should stay scoped to the specific doctor check that needs it.
  - confidence: 0.87
- **[P2] Update status cleanup hints after the signature change:** `src/daemon/inspect.ts:29-30`
  - body: After this helper returns `[]` for the default call, `src/cli/daemon-cli/status.print.ts` still reports `extraServices` but calls `renderGatewayServiceCleanupHints()` with no services, so `openclaw status --deep` loses its cleanup guidance. Pass the detected services through the status rendering path or keep a compatible helper for that caller.
  - confidence: 0.9
- **[P2] Honor system-scoped services in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: Deep scans can report system launchd/systemd services, but these commands always target `gui/$UID` and `systemctl --user`. For `/Library/LaunchDaemons` or `/etc/systemd/system` entries, the printed recipe does not unload the service that was actually detected.
  - confidence: 0.82
- **[P2] Keep hints when legacy cleanup is skipped:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every `legacy` service assumes they were removed, but the prompt can be declined and system-scoped legacy services are reported as skipped. In those paths the legacy service remains and the user loses the manual cleanup commands.
  - confidence: 0.8
