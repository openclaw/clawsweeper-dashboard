## Summary

Keep this PR open: it addresses a real doctor/gateway false-positive problem, but the branch is conflicting, targets the old inline doctor command, has blocking cleanup-hint regressions, changes security-sensitive SecretRef semantics, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes doctor cleanup hints to render from detected gateway services and changes doctor SecretRef handling so unresolved exec SecretRefs warn instead of failing.

## Best Possible Solution

Rebase the useful cleanup-hint fix onto current main so doctor and status render hints from actual detected service metadata, preserve legacy fallback guidance, honor service scope, and handle doctor SecretRefs through the existing explicit inspect path while strict exported helpers stay fail-closed.

## Reproduction Assessment

Yes, source-level. Current main can detect concrete extra gateway services, but doctor and status still call cleanup-hint rendering without passing that detected service context, while the current helper derives commands from the active profile.

## Solution Assessment

No. The direction is useful, but this PR needs a current-main rebase, status wiring, legacy and system-scope handling, and SecretRef handling through the existing inspect-mode contract rather than process-wide strict-helper leniency.
