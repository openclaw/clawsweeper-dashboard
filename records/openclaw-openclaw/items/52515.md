## Summary

Keep open. The PR targets a valid linked doctor false-positive, but current main has not implemented the cleanup-hint fix and the proposed patch still has blocking regressions in status output, legacy fallback guidance, service-scope handling, and security-sensitive SecretRef leniency.

## What This Changes

The branch changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in a process env flag that downgrades unresolved exec SecretRefs to warnings, and updates generated plugin SDK API baselines.

## Best Possible Solution

Revise and rebase the fix onto current main so doctor and status render cleanup hints from actual detected service metadata, cleanup commands honor user versus system scope, unremoved legacy services keep actionable manual guidance, and doctor SecretRef tolerance uses explicit read-only inspection/gateway resolution paths while exported strict helpers stay strict.

## Reproduction Assessment

Yes. The patch regressions are reproducible from source: current main still calls the cleanup helper with no detected-service arguments in both doctor and status, while the PR helper returns no hints for no-argument calls; the original bug remains source-reproducible whenever extra services are reported because main still renders active-profile cleanup commands.

## Solution Assessment

No. The PR is aimed at the right user problem, but this is not the best implementation because it misses the status path, drops legacy fallback hints, ignores service scope, and handles doctor SecretRef tolerance through process-wide exported-helper leniency instead of a narrow inspection path.
