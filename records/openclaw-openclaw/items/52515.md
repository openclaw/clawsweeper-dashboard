Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep strict SecretRef helpers strict:** `src/config/types.secrets.ts:147-151`
  - body: `assertSecretInputResolved` is the strict path behind the exported SDK helper, but this branch returns successfully for exec refs whenever a process-wide doctor flag is set. Doctor runs shared config/plugin readers in that same process, so strict callers can get `undefined` instead of an unresolved-secret error; use an explicit read-only doctor inspection path instead of weakening the strict helper.
  - confidence: 0.86
- **[P2] Keep status cleanup hints wired to detected services:** `src/daemon/inspect.ts:29-30`
  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when called without a service list, but `status.print.ts` still calls it with no arguments after reporting `extraServices`. That makes `openclaw status --deep` report extra gateway-like services without cleanup hints, regressing the sibling troubleshooting path.
  - confidence: 0.93
- **[P2] Preserve manual hints for unremoved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every `svc.legacy` before rendering hints drops the only manual cleanup guidance when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. In those paths doctor still reports the legacy service but no longer gives the user a command to remove it.
  - confidence: 0.88
- **[P2] Honor service scope in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The new helper receives `svc.scope` but always emits user-domain `launchctl` and `systemctl --user` commands. For `doctor --deep` findings from `/Library/LaunchDaemons` or system systemd directories, those commands target the wrong manager and leave the detected system service installed.
  - confidence: 0.9
