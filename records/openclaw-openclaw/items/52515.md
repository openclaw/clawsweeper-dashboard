## Summary

Keep open. The PR targets a real doctor/status cleanup-hint bug, but the branch is conflicting with current main, misses related callers and scope cases, lacks after-fix real behavior proof, and changes security-sensitive SecretRef strict-helper behavior.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in an env flag to downgrade unresolved exec SecretRefs to warnings, and updates generated plugin SDK API baselines.

## Best Possible Solution

Revise or replace the PR so doctor and status render cleanup hints from actual detected service metadata, honor user versus system scope, preserve legacy manual cleanup guidance, and handle doctor SecretRef checks through an explicit read-only warning path while exported strict helpers remain strict.

## Reproduction Assessment

Yes, source-level. Current main can report extra gateway-like services, then doctor and status render cleanup hints without passing the detected service metadata into the helper.

## Solution Assessment

No. The direction is useful, but this patch misses the status caller, drops legacy fallback guidance, ignores system scope, targets the old inline doctor flow, and weakens strict SecretRef behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Rebase SecretRef tolerance onto doctor health flow:** `src/commands/doctor.ts:79-81`
  - body: Current main's doctor command lazy-loads `src/flows/doctor-health.ts`, but this patch wraps the old inline doctor body in a process env flag. The exec SecretRef warning path needs to live in the current health flow or an explicit read-only inspection path, not in the stale entry-point body.
  - confidence: 0.9
- **[P2] Pass detected services into status cleanup hints:** `src/daemon/inspect.ts:28-30`
  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when called without a service list, but `status.print.ts` still calls it with no arguments after reporting `extraServices`. That makes status output report extra gateway-like services without cleanup hints.
  - confidence: 0.93
- **[P2] Keep legacy services in manual cleanup hints:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering every `svc.legacy` before rendering hints drops the fallback cleanup guidance when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. Doctor can still report the legacy service but no longer gives the user an actionable manual path.
  - confidence: 0.88
- **[P2] Use scope-specific cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The helper receives each service's `scope` but always emits user launchd/systemd commands. For LaunchDaemons or system systemd units found by deep scans, those commands target the wrong service manager and leave the detected service installed.
  - confidence: 0.9
- **[P2] Keep strict SecretRef helpers fail-closed:** `src/config/types.secrets.ts:143-151`
  - body: The patch lets an exported strict SecretRef helper return successfully for unresolved `exec` refs whenever process-wide doctor leniency is set, and exposes a doctor-specific SDK parameter. Doctor tolerance should use an explicit inspect/read-only path so runtime and plugin callers still fail closed.
  - confidence: 0.82
