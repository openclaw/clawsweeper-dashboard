## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep strict SecretRef helpers strict:** `src/config/types.secrets.ts:143-148`
  - body: `doctorCommand` sets `OPENCLAW_ALLOW_EXEC_SECRETREF_FAILURES` for the whole process, and this exported strict helper then warns and returns for any exec SecretRef while that flag is set. Because `normalizeResolvedSecretInputString` is a plugin SDK helper used by provider/channel credential readers, doctor can now silently turn a fail-closed strict read into `undefined`; route doctor through the explicit read-only SecretRef resolution path instead of weakening the strict helper.
  - confidence: 0.88
- **[P2] Pass detected services to status cleanup hints:** `src/daemon/inspect.ts:28-30`
  - body: The new helper returns `[]` unless callers pass a service list, but `status.print.ts` still calls `renderGatewayServiceCleanupHints()` with no arguments after printing `extraServices`. That regresses `openclaw status --deep` by reporting extra gateway-like services without the cleanup hints users currently get.
  - confidence: 0.93
- **[P2] Keep hints for legacy cleanup skips:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every legacy service before rendering hints removes manual cleanup guidance when legacy cleanup is skipped by external policy, declined, fails, or is system-scoped. In those paths doctor still reports the legacy service but no longer gives the user an actionable command.
  - confidence: 0.88
- **[P2] Honor system service scope in cleanup hints:** `src/daemon/inspect.ts:38-46`
  - body: The helper receives `svc.scope`, but the macOS and Linux recipes always target the user service manager (`gui/$UID` and `systemctl --user`). For `doctor --deep` findings from `/Library/LaunchDaemons` or system systemd directories, the printed commands target the wrong manager and leave the detected service installed.
  - confidence: 0.9
