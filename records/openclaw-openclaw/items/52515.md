## Summary

Keep open for contributor revision. Current main still has the doctor/status cleanup-hint bug, but this PR has blocking correctness issues, lacks after-fix real behavior proof, is conflicting with the current doctor-health flow, and changes security-sensitive exported SecretRef behavior.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, downgrades unresolved exec SecretRefs during doctor to warnings, and updates generated plugin SDK API baselines.

## Best Possible Solution

Revise the fix so doctor and status render cleanup hints from actual detected service metadata, commands honor user/system scope, unremoved legacy services keep manual guidance, and doctor SecretRef warnings use an explicit read-only inspection path while strict runtime and plugin helpers remain strict.

## Reproduction Assessment

Yes, source-level. Current main can report a custom LaunchAgent that runs openclaw gateway as an extra service, and current doctor/status callers then render cleanup hints without passing that detected service context; I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

No. The direction is useful, but this patch misses the status caller, drops legacy fallback guidance, ignores service scope, targets the old inline doctor flow, and weakens strict SecretRef behavior through process-wide state.

Full review comments:

- **[P2] Rebase SecretRef tolerance onto doctor health flow:** `src/commands/doctor.ts:79-81`
  - body: Current main's `src/commands/doctor.ts` only lazy-loads `src/flows/doctor-health.ts`, but this patch wraps the old inline doctor body in a process env flag. The exec SecretRef warning path needs to be rebased into the current doctor-health flow or an explicit read-only inspection path.
  - confidence: 0.9
- **[P2] Keep status cleanup hints wired to detected services:** `src/daemon/inspect.ts:28-30`
  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when called without a service list, but `status.print.ts` still calls it with no arguments after reporting `extraServices`. That makes status output report extra gateway-like services without cleanup hints.
  - confidence: 0.93
- **[P2] Preserve manual hints for unremoved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every legacy service before rendering hints drops the fallback cleanup guidance when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. Doctor still reports the legacy service but no longer gives the user a manual removal command.
  - confidence: 0.88
- **[P2] Honor service scope in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The helper receives each service's `scope` but always emits user launchd/systemd commands. For deep-scan findings from LaunchDaemons or system systemd directories, those commands target the wrong service manager and leave the detected service installed.
  - confidence: 0.9
- **[P2] Keep strict SecretRef helpers fail-closed:** `src/config/types.secrets.ts:143-151`
  - body: This exported strict helper returns successfully for unresolved `exec` refs whenever process-wide doctor leniency is set, and the generated SDK baseline exposes that doctor-specific option. Doctor tolerance should use an explicit read-only inspection path so runtime and plugin callers still fail closed.
  - confidence: 0.82
