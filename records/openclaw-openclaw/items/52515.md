## Summary

Keep open. The linked doctor/gateway bug is real and still tracked by the open paired issue, but this PR is not merge-ready because it is stale against the current doctor-health flow, lacks real behavior proof, and introduces concrete cleanup-hint and SecretRef-boundary regressions.

## What This Changes

The PR changes doctor extra-gateway cleanup hints to render from detected service records and makes doctor tolerate unresolved exec SecretRefs by setting a process env flag and widening SecretRef helper parameters.

## Best Possible Solution

Revise the fix on current main so doctor and status render cleanup hints from actual detected service metadata, preserve actionable legacy fallback guidance, honor user/system scope, and keep SecretRef leniency in explicit read-only inspection paths rather than exported strict helpers.

## Reproduction Assessment

Yes, source-level: current main still has doctor and status cleanup-hint callers that render without detected service context, while the PR helper returns no hints for that unchanged call shape. I did not run a live macOS LaunchAgent scenario in this read-only review.

## Solution Assessment

No. The PR targets the right user problem, but the implementation is stale against current main and should use scope-aware service metadata plus command-local SecretRef inspection instead of process-wide strict-helper leniency.
