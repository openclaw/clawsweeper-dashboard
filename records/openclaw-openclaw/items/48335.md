## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the proposed loop-aware compaction guard hook, and the PR remains a draft stacked on open prerequisite PRs; the diff also has blocking prompt-instruction and documentation issues that need maintainer review.

## What This Changes

The branch adds `agents.defaults.compaction.guard` config scaffolding, transcript-tail detection and scoring, and a safeguard compaction hook that conditionally appends loop-aware compaction guidance for high-risk sessions.

## Best Possible Solution

Keep the stack open, rebase it onto current `src/agents/pi-hooks`, and land the runtime hook only after the guard directives are carried through a trusted instruction seam, transcript-derived signal details stay bounded/untrusted, and the config docs/help copy reflects that `guard.enabled` now affects runtime behavior.

## Reproduction Assessment

Yes. Read-only inspection of current main reproduces the missing behavior: there is no `compaction.guard` schema/type field, no `guardEnabled` runtime field, and no detector/scorer/augmentation call in `session_before_compact`; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as written. The proposed direction is plausible, but the current diff puts the new guard directives into the existing custom-instructions path that is wrapped as untrusted prompt data, and it leaves docs/help text saying the guard has no runtime effect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep guard directives out of the untrusted custom block:** `src/agents/pi-extensions/compaction-safeguard.ts:872-876`
  - body: The new guard block is appended to `customInstructions`, but the next step passes that value to `buildCompactionStructureInstructions`, whose current contract wraps custom instructions as untrusted prompt data. That means high-risk sessions can log `augmented=true` while the summarizer is told to treat the guard text as data, not operating instructions. Add a trusted guard-instruction parameter or compose the guard block into the structured instruction template while keeping transcript-derived signal details separately bounded and untrusted.
  - confidence: 0.84
- **[P2] Update guard docs once it affects runtime:** `docs/gateway/configuration-reference.md:1036`
  - body: This PR wires `guard.enabled` into the safeguard runtime, but the config reference still says the `guard` block currently does not change runtime behavior. If this branch lands, users will be told the setting is inert when it can now augment compaction instructions for high-risk sessions, so the docs/help copy needs to describe the new guarded-runtime behavior or stay out of this PR.
  - confidence: 0.9
