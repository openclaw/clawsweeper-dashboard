## Summary

Keep open: this draft, stacked feature is not implemented on current main and still has blocking review issues around trusted prompt boundaries, inaccurate docs/help, and missing real behavior proof.

## What This Changes

The branch adds stacked compaction-guard config, detector, and scorer pieces, wires `guardEnabled` into safeguard compaction runtime, and appends guard-specific compaction instructions for high-risk sessions.

## Best Possible Solution

Land or replace the prerequisite guard pieces first, then rebase onto the current `pi-hooks` compaction code with a trusted guard-instruction seam, data-only signal details, updated config docs/help, regression tests, and redacted real compaction-run proof.

## Reproduction Assessment

Yes for the PR defects by source review: PR head builds guard instructions, passes them as `customInstructions`, and current compaction wraps that route as untrusted prompt data; docs/help also say the guard has no runtime effect while runtime uses `guardEnabled`. This is not an end-to-end current-main reproduction of the broader feature.

## Solution Assessment

No, not as written. The feature direction is plausible, but fixed guard policy should use a trusted instruction path separate from transcript-derived signal data, and docs/help must match the new runtime behavior.
