## What This Changes

Adds a Control UI chat-run watchdog that probes `agent.wait` after idle to reconcile stale runs, reload history, flush queued sends, and reset on gateway/lifecycle events.

## Best Possible Solution

Land this focused UI-side watchdog only after real Control UI proof demonstrates missed-terminal recovery, history reload, and queued-send flush against a real gateway/browser flow.

## Reproduction Assessment

Yes, by source inspection: current main keeps chat busy while `chatRunId` is set and `flushChatQueue` returns while busy, so a missed terminal event can leave the queue blocked. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

Yes, likely: using the existing `agent.wait` contract from the Control UI is a bounded fix at the local state boundary instead of changing backend protocol or queue semantics. The remaining blocker is real behavior proof, not a clearer code location.
