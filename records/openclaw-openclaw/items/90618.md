## What This Changes

The PR adds a Control UI chat-run watchdog that probes `agent.wait` after idle, reconciles a terminal run, reloads history, flushes queued messages, and resets on gateway/lifecycle events.

## Best Possible Solution

## Reproduction Assessment

Yes, by source inspection: current main keeps chat busy while `chatRunId` is set and `flushChatQueue` returns while busy, so a missed terminal event can leave the queue blocked. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

Yes, likely: using the existing `agent.wait` contract from the Control UI is a bounded fix at the local state boundary instead of changing backend protocol or queue semantics. The remaining blocker is real behavior proof of the browser/gateway missed-terminal flow.
