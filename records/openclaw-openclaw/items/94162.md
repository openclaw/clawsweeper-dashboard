## Summary

Keep open: current main and the latest release still load configured bundle MCP servers sequentially on the reported `bundle-tools` path, and https://github.com/openclaw/openclaw/pull/94230 is an open closing PR rather than a merged fix.

## What This Changes

The issue reports 6-7 seconds of per-request `bundle-tools` latency when multiple MCP servers and skills are configured, and asks for lazy loading, caching, warm pooling, or a simple-chat fast path.

## Best Possible Solution

Keep this issue as the canonical tracker until https://github.com/openclaw/openclaw/pull/94230 lands or is replaced, then split any remaining lazy-loading or no-tools fast-path design into a narrower follow-up.

## Reproduction Assessment

Yes for the central source path: current main and v2026.6.8 await each configured MCP server connection and tools/list call inside one catalog loop. I did not run a live multi-server latency repro in this read-only review.

## Solution Assessment

Unclear for the full requested end state. Parallel catalog loading is the best bounded first fix for the proven sum-of-servers latency path, while true lazy loading or a simple-chat fast path needs maintainer product direction.
