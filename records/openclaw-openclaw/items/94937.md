## Summary

Keep open: current main still drops durable Telegram reasoning because runtime emits `isReasoning` block and final payloads, but the shared dispatcher suppresses those payloads before Telegram can persist them; no linked PR currently owns the exact fix.

## What This Changes

Request that Telegram persist model thinking as separate messages under `/reasoning on` instead of dropping `type:"thinking"` payloads while `/reasoning stream` already works.

## Best Possible Solution

Route durable `isReasoning` block and final payloads to Telegram only when the channel has explicit reasoning handling and `/reasoning on` is active, while preserving `/reasoning off`, `/reasoning stream`, and generic-channel suppression semantics.

## Reproduction Assessment

Yes, source-level: current main produces `isReasoning` block/final payloads, then `dispatch-from-config` returns before those payloads reach Telegram delivery. I did not run a live Telegram/DeepSeek session in this read-only pass.

## Solution Assessment

Yes. The best fix is a narrow dispatch or Telegram opt-in delivery repair for durable reasoning payloads, not a provider-specific DeepSeek path or a new config option.
