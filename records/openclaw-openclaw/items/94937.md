## Summary

Keep open: current main still drops Telegram durable `/reasoning on` thinking. The model/runtime produce `isReasoning` payloads, but the shared dispatcher suppresses block and final reasoning payloads before Telegram can persist them; `/reasoning stream` works because it uses the separate stream callback.

## What This Changes

The issue requests that Telegram `/reasoning on` persist model `type:"thinking"` content as separate `Thinking` messages instead of dropping it.

## Best Possible Solution

Route durable `isReasoning` payloads to Telegram when `/reasoning on` is active so they persist as separate `Thinking` messages, while preserving `/reasoning off`, `/reasoning stream`, and generic-channel suppression semantics.

## Reproduction Assessment

Yes, source-level. `/reasoning on` produces `isReasoning` block and final payloads, and current `dispatch-from-config` returns before those payloads reach Telegram delivery; I did not run a live Telegram/DeepSeek session in this read-only review.

## Solution Assessment

Yes. A narrow shared dispatch or Telegram-delivery repair is the best fit; a provider-specific DeepSeek path or new config option would duplicate the existing reasoning contract.
