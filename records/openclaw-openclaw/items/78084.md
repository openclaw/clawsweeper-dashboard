## Summary

Keep open: current main and v2026.6.6 still use exact `contracts.tools` matching, but this PR is not merge-ready because its wildcard selection can expose unrequested same-prefix tools and the external PR still lacks real behavior proof.

## What This Changes

The PR adds terminal wildcard matching for plugin `contracts.tools`, applies it to registration and plugin tool resolution, adds regression tests, and documents the manifest syntax.

## Best Possible Solution

Land a maintainer-approved bounded wildcard ownership contract that preserves manifest-first loading while filtering returned tools by concrete allowlists, with focused tests, docs, and real runtime proof.

No. The bounded wildcard direction is plausible, but this implementation must filter by concrete allowlists and still needs real behavior proof plus maintainer API approval.
