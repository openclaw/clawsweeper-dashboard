## Summary

Keep open: the PR addresses a real current-main limitation, but the proposed wildcard filtering can broaden explicit tool allowlists and the external-contributor real behavior proof gate is still unmet.

## What This Changes

The PR adds terminal wildcard matching for plugin `contracts.tools`, applies it to registration and plugin-tool resolution, adds regression tests, and documents the new manifest syntax.

## Best Possible Solution

Land a maintainer-approved bounded wildcard ownership contract that uses wildcards for ownership discovery while preserving exact explicit allowlist filtering and real plugin or Gateway proof.

## Reproduction Assessment

Yes, at source level. Current main requires exact `contracts.tools` declarations in registration and manifest allowlist resolution, so a wildcard-owned dynamic runtime tool cannot be selected without a contract change.

## Solution Assessment

No, not yet. The direction is plausible, but the current patch should filter factory-returned tools by the original explicit allowlist and still needs maintainer approval plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter wildcard results by the requested tool name:** `src/plugins/tools.ts:1114-1116`
  - body: When an explicit allowlist requests `dynamic_alpha`, the patch selects the `dynamic_*` contract so the plugin can load, but this condition then lets any factory-returned tool matching `dynamic_*` through, such as `dynamic_beta`, even though it was not allowlisted. Keep the wildcard for ownership/plugin selection, but filter returned tools against the original requested tool names unless the whole plugin/group/default set was selected.
  - confidence: 0.9
