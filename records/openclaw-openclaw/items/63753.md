## Summary

Keep open. Current main still has the macOS Talk routing gap: the app fetches Talk config and has a `talk.speak` helper, but OpenAI/custom providers still fall through to local system speech; the already-open #74645 is the right implementation candidate to review rather than closing this issue now.

## What This Changes

MacOS Talk mode should route non-local TTS providers through Gateway `talk.speak` instead of falling back to local AVSpeechSynthesizer.

## Best Possible Solution

Review and land #74645 or an equivalent macOS runtime fix that sends every non-local/non-MLX/non-system Talk provider through Gateway `talk.speak` as the primary playback path, keeps explicit local paths local, and adds an OpenAI/custom-provider regression test.

## Reproduction Assessment

Yes. With a Gateway config that resolves `talk.provider` to `openai`, current macOS parsing stores `activeTalkProvider = "openai"`, `playbackPlan` reaches the default branch, and playback uses `.systemVoiceOnly` with no expected `talk.speak` call.

## Solution Assessment

Yes for the runtime bug. The narrow maintainable fix belongs in `TalkModeRuntime.swift`: preserve MLX/system local behavior, keep any explicit ElevenLabs optimization, and route all other configured providers through Gateway `talk.speak`; the settings UI can be handled separately if still desired.
