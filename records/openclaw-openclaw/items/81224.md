## Summary

Keep open: this is a focused fix for a source-reproducible provider catalog bug, and the runtime approach looks correct. The remaining issue is a small docs gap because the provider pages still describe explicit vLLM/SGLang config as always disabling discovery.

## What This Changes

The branch changes shared self-hosted provider discovery so explicit provider config stays manual-only unless provider wildcard visibility is configured, then uses the configured base URL for discovery and adds focused tests plus a changelog entry.

## Best Possible Solution

Land the shared-helper fix with provider docs updated to explain that explicit vLLM/SGLang config remains manual by default, but `vllm/*` or `sglang/*` keeps discovery dynamic against the configured endpoint.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, explicit vLLM/SGLang provider config makes the shared helper return before the provider builder can query `/models`, and the related issue gives concrete wildcard config that triggers it.

## Solution Assessment

Mostly yes. The runtime fix is narrow and belongs in the shared self-hosted discovery helper, but the provider docs should be updated so the new wildcard exception is not documented as unsupported.
