## Summary

Keep open: this is a focused fix for a source-reproducible provider discovery bug, and the code path looks correct. The one concrete follow-up I found is that the vLLM/SGLang provider docs still describe explicit provider config as always disabling discovery, which would contradict the new wildcard exception after merge.

## What This Changes

The PR changes shared self-hosted provider discovery so explicit vLLM/SGLang provider endpoints remain manual-only for exact model entries but can still be used for `provider/*` wildcard discovery.

## Best Possible Solution

Land the shared helper fix with the new regression tests, keep exact configured model entries manual-only, and update the vLLM/SGLang provider docs to describe the `provider/*` wildcard exception.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: configure `models.providers.vllm` or `models.providers.sglang` plus `agents.defaults.models["provider/*"]`, and the shared helper returns before the provider builder can query `/models`.

## Solution Assessment

Yes for the code path: the PR fixes the shared discovery helper used by both affected providers instead of patching each plugin separately. The solution should also update the provider docs so the user-facing contract matches the new wildcard behavior.
