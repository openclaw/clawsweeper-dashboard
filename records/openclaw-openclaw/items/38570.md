## Summary

Keep open: current main and latest shipped release still build the A2A prompt contexts from requester session/channel only, while `identity.name` already exists in the config contract. The direct fix PR was closed unmerged, and the broader UI attribution trackers do not cover this model-facing requester-name prompt gap.

## What This Changes

Include the requester agent's configured `identity.name` in `sessions_send` A2A message, reply, and announce prompt contexts so receivers do not infer the sender from session keys.

## Best Possible Solution

Thread an optional requester identity name from existing agent config into the three A2A prompt builders, preserving current fallback behavior when no safe name is available.

## Reproduction Assessment

Yes, source-reproducible. Configure two agents with a requester `identity.name`, call `sessions_send`, and inspect the generated `extraSystemPrompt`; current main has no code path that can include that name.

## Solution Assessment

Yes. The narrow maintainable fix is prompt-context plumbing plus targeted tests; the broader UI sender-label/avatar contract should stay with the separate durable sender-identity tracker.
