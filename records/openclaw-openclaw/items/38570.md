## Summary

Keep this issue open: current main and the latest release still build the `sessions_send` A2A prompt contexts without the requester agent's configured identity name. The open closing PR is the right implementation path, but it has not merged and is currently blocked by conflict/proof state.

## What This Changes

The issue requests that inter-agent `sessions_send` prompt contexts include the requester agent's configured identity name so receivers do not infer the sender from the session key.

## Best Possible Solution

Refresh and land a narrow fix that resolves the requester agent from the existing session/config data, adds the optional identity-name line to all three shared A2A prompt contexts, and includes redacted real multi-agent proof before closing this issue.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.5` build the initial, reply, and announce A2A contexts without the requester’s configured identity name. I did not run a live LLM exchange reproducing the resulting misattribution.

## Solution Assessment

Yes, the narrow fix belongs at the `sessions_send` prompt owner boundary by resolving an existing configured identity and threading one optional value through the shared prompt builders. A workspace mapping workaround or new config surface would be less maintainable.
