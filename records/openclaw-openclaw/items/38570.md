## Summary

Keep this issue open: current main and the latest release still omit the configured requester identity from the shared sessions_send A2A prompt context, and the open linked PR is the active fix path.

## What This Changes

The issue asks for sessions_send agent-to-agent prompts to include the requester agent's configured identity.name so the receiver does not infer the sender from the session key.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/91271 so all three A2A prompt phases include a stable configured requester name while preserving the current placeholder session-key contract.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.8 build the initial, reply, and announce A2A prompt context without any configured requester identity.name field, though I did not run a live multi-agent exchange.

## Solution Assessment

Yes, the narrow maintainable direction is to resolve the requester agent identity from existing config at the sessions_send boundary and thread an optional stable name through the existing helper path. Changing routing, adding config, or relying on workspace memory workarounds would be broader than needed.
