## Summary

Keep open: current main and the latest release still omit the configured requester identity name from `sessions_send` A2A prompt context, and the open linked PR is the right implementation path to review rather than closing the issue early.

## What This Changes

The issue asks for `sessions_send` A2A initial, reply, and announce prompts to include the requester's configured `agents.list[].identity.name` so the receiving agent does not infer the sender from the session key.

## Best Possible Solution

Land or replace the linked requester-name PR after rebase and proof, adding the configured requester identity name to all three A2A prompt contexts while preserving prompt-cache placeholders.

## Reproduction Assessment

Yes at source level. Current main's initial, reply, and announce A2A prompt builders carry requester session/channel context but not the configured requester `identity.name`; I did not run a live multi-agent exchange.

## Solution Assessment

Yes, the requested fix shape is narrow. Resolve the requester agent identity at the `sessions_send` boundary and thread an optional stable name through the existing prompt builders, without changing routing, delivery, or adding config.
