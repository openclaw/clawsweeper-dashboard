## Summary

Current main still omits the requester identity name from all three agent-to-agent `sessions_send` prompt contexts, so the reported sender-attribution bug remains valid. The related direct fix PR is closed unmerged, and the broader UI sender-identity issues do not supersede this narrower prompt-context gap.

## What This Changes

The issue asks for `sessions_send` agent-to-agent prompt contexts to include the requester agent's configured `identity.name` so receiving agents can identify the actual sender.

## Best Possible Solution

Thread an optional requester identity name from existing agent config into the initial, reply, and announce A2A prompt builders, omitting the line when the requester session cannot be safely mapped or has no configured name.

## Reproduction Assessment

Yes. This is source-reproducible by configuring two agents with a requester `identity.name`, invoking `sessions_send`, and inspecting the generated A2A `extraSystemPrompt`; current main has no path that can include that name.

## Solution Assessment

Yes. The narrow maintainable solution is existing-config prompt-context plumbing plus regression coverage; broader durable UI sender-display work belongs with the separate multi-role Chat UI tracker.
