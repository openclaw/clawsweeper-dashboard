## Summary

Current main still omits the configured requester identity name from the `sessions_send` A2A prompt context, and the live linked PR has not merged, so this issue remains necessary.

## What This Changes

The issue asks for inter-agent `sessions_send` prompt contexts to include the requester agent's configured identity name so the receiver does not infer the wrong sender.

## Best Possible Solution

Land or replace the linked fix so the initial, reply, and announce A2A prompts include a stable configured requester name while preserving placeholder session IDs and existing routing provenance.

## Reproduction Assessment

Yes at source level. Current main’s initial, reply, and announce A2A prompt builders carry requester session/channel context but not the configured requester `identity.name`; I did not run a live multi-agent exchange.

## Solution Assessment

Yes, after rebase and proof. Resolving a configured requester name from the parsed requester agent id and threading that optional stable value into the existing helper path is narrower than changing routing, adding config, or relying on per-agent memory workarounds.
