## Summary

Keep open: current `main` and the latest release still omit the requester identity name from `sessions_send` A2A prompt context, and the open closing PR is not merged yet, so the issue is still necessary.

## What This Changes

The issue asks for `sessions_send` A2A prompts to include the requester agent's configured `identity.name` so receivers do not infer the wrong sender.

## Best Possible Solution

Thread the canonical configured requester identity name into the initial, reply, and announce A2A prompts while preserving placeholder session keys and avoiding routing or access-policy changes.

## Reproduction Assessment

Yes at source level. Current `main` builds the initial, reply, and announce A2A prompt contexts from requester session/channel and target session/channel only; I did not run a live multi-agent exchange because this review is read-only.

## Solution Assessment

Yes. Resolving the requester name from existing agent config and threading that stable value through the shared A2A prompt builders is narrower than adding config, changing routing, or relying on per-agent memory workarounds.
