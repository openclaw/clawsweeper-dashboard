## Summary

This should stay open: current main still builds the `sessions_send` agent-to-agent prompts from requester session/channel data only, while the configured requester `identity.name` is available elsewhere in config and is not threaded into the message, reply, or announce contexts.

## What This Changes

Request `sessions_send` A2A prompts include the requester agent's configured `identity.name` so receiver agents do not infer the sender from the session key.

## Best Possible Solution

Resolve the requester agent id from the requester session key, read its configured `identity.name`, and thread a trimmed requester-name hint through the initial, reply, and announce A2A prompt builders with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a requester with `agents.list[].identity.name` can call `sessions_send`, but the generated A2A `extraSystemPrompt` is built without any requester-name field.

## Solution Assessment

Yes. The narrow maintainable fix is to add the existing configured requester identity to the A2A prompt context without changing routing, delivery, session visibility, or UI sender display semantics.
