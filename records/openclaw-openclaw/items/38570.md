## Summary

Current main still omits the configured requester identity from all three `sessions_send` agent-to-agent prompt contexts, and the open linked PR is the active implementation path; keep this issue open until that PR is rebased, proven in a real multi-agent run, and merged.

## What This Changes

Add the requester agent’s configured identity name to the initial, reply, and announce prompt contexts used by inter-agent `sessions_send`.

## Best Possible Solution

Adapt the linked PR to current main’s shared context helper, resolve the requester name once from canonical config, preserve existing routing and delivery behavior, and cover the initial, ping-pong reply, and announce prompts with focused tests plus a redacted live multi-agent exchange.

## Reproduction Assessment

Yes at source level. Current main’s initial, reply, and announce prompt builders carry the requester session and channel but not the configured requester identity, directly matching the reported failure mechanism; a live failing exchange was not run during this read-only review.

## Solution Assessment

Yes, after rebasing the implementation. Adding an optional resolved requester name through the existing shared A2A context path is narrower and more maintainable than adding a new config, UI sender field, routing rule, or agent-memory workaround.
