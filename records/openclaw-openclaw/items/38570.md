## Summary

Keep open: current main and the latest release still omit the configured requester identity name from the `sessions_send` A2A prompt contexts, and an open closing PR now owns the fix path.

## What This Changes

The issue asks for `sessions_send` A2A contexts to include the requester agent's configured identity name so receivers do not infer the sender from the session key.

## Best Possible Solution

Ship a rebased fix that resolves the requester agent id from a parsed agent session key, threads the configured identity name through initial, reply, and announce A2A contexts, and preserves the placeholder session-key prompt-cache contract.

## Reproduction Assessment

Yes at source level. Current main builds all three A2A prompt contexts without the configured requester `identity.name`; I did not run a live multi-agent exchange.

## Solution Assessment

Yes, the requested direction is the narrow fix. The maintainable shape is to add stable configured requester identity metadata to the existing A2A prompt builders, not change routing, add a config knob, or rely on per-agent memory workarounds.
