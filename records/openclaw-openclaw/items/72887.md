## Summary

Keep open. Current main still has no review-queue, off, or alternate final-promotion destination for Dreaming deep promotion; both managed Dreaming and manual apply still append promoted candidates into `MEMORY.md`, so this remains a product/config contract request rather than cleanup-closeable stale work.

## What This Changes

Request a configurable review queue or write policy so Dreaming deep promotions can be staged before mutating `MEMORY.md`.

## Best Possible Solution

Add a documented, schema-backed final-promotion write policy in memory-core that preserves the current `MEMORY.md` default while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/config request. Source, docs, schema, and tests give high-confidence verification that the requested review queue or write policy does not exist on current main.

## Solution Assessment

Unclear until maintainers choose the public workflow contract. A separate final-promotion destination or approval flow looks safer than overloading `dreaming.storage.mode`, which current code and docs use for phase/report storage.
