## Summary

Keep open. Current main still routes deep Dreaming promotions and manual promotion apply through the direct `MEMORY.md` append path, while the existing storage setting and docs only cover phase/report artifacts, so the requested review queue remains a valid product/API gap.

## What This Changes

The issue requests a configurable deep-promotion review queue so eligible Dreaming candidates can be staged outside `MEMORY.md` before approval.

## Best Possible Solution

Add an additive final-promotion destination contract in memory-core, defaulting to current `MEMORY.md` behavior while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/design request. Current-main verification is high-confidence from source and focused tests: eligible managed Dreaming candidates and manual `memory promote --apply` candidates are written directly to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination setting or approval flow is safer than overloading `dreaming.storage.mode`, which currently means phase/report storage rather than canonical memory mutation.
