## Summary

Keep open. Current main still sends managed deep Dreaming promotions and manual `memory promote --apply` through the direct `MEMORY.md` append helper, and neither the config schema nor docs expose a final-promotion review queue.

## What This Changes

Request configurable Dreaming deep-promotion output so ranked candidates can be staged in a review queue before canonical `MEMORY.md` mutation.

## Best Possible Solution

Add an additive memory-core final-promotion destination contract that defaults to current `MEMORY.md` behavior while supporting structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/design request. Current-main verification is high-confidence from source, docs, and focused tests showing eligible promotions are still appended directly to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination or approval flow looks safer than overloading `dreaming.storage.mode`, which currently covers phase/report artifact placement rather than canonical memory mutation.
