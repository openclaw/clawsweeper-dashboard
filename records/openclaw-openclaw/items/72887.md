## Summary

Keep open. Current main still has no final-promotion review queue: managed deep dreaming ranks candidates, calls the direct MEMORY.md append helper, and only writes the deep report afterward. The schema, SDK config, CLI/docs, and linked related items do not provide the requested destination or approval contract.

## What This Changes

The issue requests a configurable deep-promotion review queue so Dreaming can stage promoted candidates before writing them to MEMORY.md.

## Best Possible Solution

Add an explicit additive final-promotion destination contract in memory-core that preserves current direct MEMORY.md writes as a compatible mode while allowing promoted candidates to be staged in a structured review artifact before canonical long-term memory is mutated; update the manifest, resolver, docs, and focused tests around that chosen contract.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior has a high-confidence source verification path: managed deep dreaming and manual `memory promote --apply` both call applyShortTermPromotions, which appends selected candidates to MEMORY.md.

## Solution Assessment

Unclear until maintainers choose the workflow contract. Directionally, a separate final-promotion destination or approval setting is cleaner than overloading dreaming.storage.mode, which currently describes phase/report artifact placement.
