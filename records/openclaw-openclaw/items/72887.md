## Summary

Keep open. Current main still sends managed deep Dreaming promotions through the direct MEMORY.md append path, and the schema, SDK resolver types, CLI/docs, and focused tests do not expose a final-promotion review queue or approval destination.

## What This Changes

Request support for staging deep Dreaming promotion output in a review queue before it mutates MEMORY.md.

## Best Possible Solution

Add an explicit, additive final-promotion destination contract in memory-core that preserves direct MEMORY.md append as the default but can stage selected deep candidates into a structured review artifact before canonical memory is changed.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior has a high-confidence source and focused-test path: run managed deep Dreaming with an eligible candidate and current main appends selected candidates into MEMORY.md.

## Solution Assessment

Unclear until maintainers choose the workflow contract. Directionally, a dedicated final-promotion write mode or approval destination is safer than reusing dreaming.storage.mode, which currently governs phase/report storage rather than canonical memory mutation.
