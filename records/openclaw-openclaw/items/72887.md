## Summary

Keep open. Current main still appends deep Dreaming promotions directly to MEMORY.md and exposes no final-promotion review destination, so this remains a valid memory-core product/API gap rather than closeable cleanup.

## What This Changes

The issue requests a configurable review queue for memory-core Dreaming deep promotions before they are appended to MEMORY.md.

## Best Possible Solution

Add an additive final-promotion destination contract in memory-core, defaulting to current MEMORY.md behavior while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/design request. Current-main verification is high-confidence from source and focused tests: eligible managed Dreaming candidates and manual memory promote --apply candidates are still written directly to MEMORY.md.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination setting or approval flow looks safer than overloading dreaming.storage.mode, which currently means phase/report storage rather than canonical memory mutation.
