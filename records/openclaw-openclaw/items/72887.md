## Summary

Keep open. Current `main` still routes managed deep Dreaming promotions through the direct `MEMORY.md` append helper, and the manifest schema, SDK config types, CLI/docs, and focused tests do not expose a final-promotion review queue or approval destination. The related storage-mode change separates phase/report artifacts, but it does not change where durable promotions are written.

## What This Changes

The issue requests a configurable memory-core deep Dreaming review/staging destination before promoted candidates are written into canonical `MEMORY.md`.

## Best Possible Solution

Add an explicit, additive final-promotion destination contract in memory-core that keeps direct `MEMORY.md` append as the default but can stage selected deep candidates into a structured review artifact before canonical memory is mutated; align manifest/schema, SDK types, CLI or Control UI behavior, docs, changelog, and focused tests around that contract.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior has a high-confidence source and test verification path: trigger managed deep Dreaming with an eligible candidate and current `main` appends the promoted snippet to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the workflow contract. Directionally, a separate final-promotion destination or approval setting is safer than overloading `dreaming.storage.mode`, which currently describes phase/report artifact placement rather than the durable write target.
