## Summary

Keep open. Current main still appends managed deep Dreaming and manual `memory promote --apply` output directly to `MEMORY.md`, and the schema/docs expose no final-promotion review queue or destination contract, so this remains a valid memory-core product/API request.

## What This Changes

Add a configurable review-queue destination for memory-core deep promotion before automatic writes to `MEMORY.md`.

## Best Possible Solution

Add an additive final-promotion destination contract in memory-core, defaulting to current `MEMORY.md` behavior while supporting structured review artifacts and an explicit approve/apply flow.

## Reproduction Assessment

Not applicable as a feature/config request. Current-main source, docs, and tests give a high-confidence behavior check that managed deep Dreaming and manual `--apply` still append to `MEMORY.md`, while `dreaming.storage.mode` only controls phase/report artifacts.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination setting or approval flow looks safer than overloading `dreaming.storage.mode`, which currently means phase/report storage rather than canonical memory mutation.
