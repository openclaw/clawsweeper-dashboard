## Summary

Keep open. Current main still routes managed deep-dreaming promotions directly through the `MEMORY.md` append helper, and the existing storage setting, manifest schema, docs, and tests only cover phase/report output rather than a final-promotion review queue or approval contract. The linked raw-snippet, metadata-contamination, storage-mode, and promote-then-rotate items are adjacent context, but they do not supersede this requested workflow decision.

## What This Changes

Add a configurable review or staging queue for memory-core deep dreaming promotions before promoted candidates are written into `MEMORY.md`.

## Best Possible Solution

Define an additive final-promotion destination contract in memory-core that preserves current direct `MEMORY.md` writes as the compatible mode while allowing promoted candidates to be staged in a structured review artifact before canonical long-term memory is mutated; update the manifest, resolver, docs, and focused tests around that contract.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior has a high-confidence source verification path: managed deep dreaming calls `applyShortTermPromotions`, and that helper appends selected candidates to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination or approval setting is likely cleaner than overloading `dreaming.storage.mode`, which currently describes phase/report artifact placement.
