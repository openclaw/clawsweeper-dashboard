## Summary

Keep open. Current main still sends managed Deep Dreaming promotion and manual `memory promote --apply` through the direct `MEMORY.md` append path, while the schema, SDK config types, CLI, docs, and tests still lack a review-queue or final-promotion destination policy.

## What This Changes

Request adding a configurable review queue or destination mode for Dreaming deep promotion before candidates are written to `MEMORY.md`.

## Best Possible Solution

Add an additive, schema-backed final-promotion write policy in memory-core that preserves current `MEMORY.md` behavior by default while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection on current main gives a high-confidence verification path: managed Deep and manual `--apply` still append to `MEMORY.md`, and no review destination exists.

## Solution Assessment

Unclear until maintainers choose the public workflow contract. A separate final-promotion destination or approval flow looks safer than overloading `dreaming.storage.mode`, which current code and docs use for phase/report storage.
