## Summary

Keep open. Current main and the latest release still route managed and manual Deep promotion through direct `MEMORY.md` writes, and no schema, SDK type, docs, or tests expose the requested review destination; this is a product/API config request needing maintainer design.

## What This Changes

Request a configurable Deep Dreaming promotion path that can stage promoted candidates in a review queue instead of always appending them to `MEMORY.md`.

## Best Possible Solution

Add an additive, schema-backed final-promotion destination policy in memory-core that preserves current `MEMORY.md` behavior by default while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/config request. Source, docs, schema, and release-tag inspection give high confidence that current behavior still writes promotions to `MEMORY.md` and lacks a review destination option.

## Solution Assessment

Unclear until maintainers choose the public workflow contract. A separate final-promotion destination or approval flow looks safer than overloading `dreaming.storage.mode`, which current code and docs use for phase/report storage.
