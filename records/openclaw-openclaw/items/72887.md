## Summary

Keep open. Current main still sends managed deep Dreaming promotions through the direct `MEMORY.md` append path, and the config schema, SDK config types, CLI/docs, and focused tests do not expose a final-promotion review queue or approval destination. The related storage-mode fix only separates phase/report artifacts; it does not change the durable promotion write target.

## What This Changes

Request configurable deep Dreaming promotion output so selected candidates can be staged in a review queue before any automatic append to `MEMORY.md`.

## Best Possible Solution

Add an explicit, additive final-promotion destination contract in memory-core that keeps direct `MEMORY.md` append as the default but can stage selected deep candidates into a structured review artifact before canonical memory is mutated; align manifest/schema, resolver types, CLI/UI behavior, docs, and focused tests around that contract.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior has a high-confidence source and test verification path: trigger managed deep Dreaming with an eligible candidate and current main appends the promoted snippet to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the workflow contract. Directionally, a separate final-promotion destination or approval setting is safer than overloading `dreaming.storage.mode`, which currently describes phase/report artifact placement rather than the durable write target.
