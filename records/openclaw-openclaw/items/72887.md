## Summary

Keep open. Current main still routes managed deep Dreaming promotions through the direct MEMORY.md append path, and the manifest schema, host SDK types, CLI/docs, and focused tests do not expose a final-promotion review queue or approval destination. The related storage-mode work only separates phase/report artifacts; it does not change the durable promotion write target.

## What This Changes

Request to add a configurable review or staging destination for memory-core deep Dreaming promotions before they are written to MEMORY.md.

## Best Possible Solution

Add an explicit, additive final-promotion destination contract in memory-core that preserves direct MEMORY.md append as the default but can stage selected deep candidates into a structured review artifact before canonical memory is mutated; align manifest/schema, resolver types, CLI/UI behavior, docs, and focused tests around that contract.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior has a high-confidence source and test verification path: trigger managed deep Dreaming with an eligible candidate and current main appends the promoted snippet to MEMORY.md.

## Solution Assessment

Unclear until maintainers choose the workflow contract. Directionally, a separate final-promotion write mode or approval destination is safer than reusing dreaming.storage.mode, because that setting already means phase/report storage rather than canonical memory mutation.
