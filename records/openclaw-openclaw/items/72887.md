## Summary

Keep open. Current main still sends managed deep Dreaming promotion and manual `memory promote --apply` through the direct `MEMORY.md` append path, while `dreaming.storage.mode` remains scoped to phase/report artifacts; the review queue is still a real memory-core product/API gap.

## What This Changes

The issue requests configurable deep-promotion output so candidates can be staged in a review queue before canonical `MEMORY.md` writes.

## Best Possible Solution

Add an additive final-promotion destination contract in memory-core, defaulting to current `MEMORY.md` behavior while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/API request. Current-main source, docs, and tests give high-confidence verification that eligible managed and manual promotions still write directly to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination or approval flow looks safer than overloading `dreaming.storage.mode`, which currently means phase/report storage.
