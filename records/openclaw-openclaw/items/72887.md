## Summary

Keep open. Current main still routes managed deep Dreaming promotion and manual `memory promote --apply` through the direct `MEMORY.md` append path, while the existing storage setting only controls phase/report artifacts; the requested review queue remains a valid memory-core product/API gap.

## What This Changes

The issue requests a configurable review-queue destination for deep Dreaming promotions before promoted candidates are written to `MEMORY.md`.

## Best Possible Solution

Add an additive final-promotion destination contract in memory-core, defaulting to current `MEMORY.md` behavior while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/API request. Current-main source, docs, and focused tests give high-confidence verification that eligible managed and manual promotions still write directly to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination setting or approval flow looks safer than overloading `dreaming.storage.mode`, which currently means phase/report storage rather than canonical memory mutation.
