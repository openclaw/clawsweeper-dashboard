## Summary

Keep open. Current main still routes managed Deep Dreaming promotion and manual `memory promote --apply` through the direct `MEMORY.md` append path, and the requested review queue/final-promotion destination policy is absent from schema, SDK types, docs, and tests; related contamination guards reduce bad snippets but do not solve this product/API gap.

## What This Changes

Request a configurable Deep Dreaming promotion destination so candidates can be staged in a review queue before any write to `MEMORY.md`.

## Best Possible Solution

Add an additive, schema-backed final-promotion destination policy in memory-core that preserves current `MEMORY.md` behavior by default while allowing structured review artifacts and an explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/config request. Current-main source, docs, and tests give high-confidence verification that managed and manual promotions still append directly to `MEMORY.md` and no review destination option exists.

## Solution Assessment

Unclear until maintainers choose the public workflow contract. A separate final-promotion destination or approval flow looks safer than overloading `dreaming.storage.mode`, which current code and docs use for phase/report storage.
