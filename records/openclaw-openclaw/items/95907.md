## Summary

Kept open because the report describes a real lifecycle failure class: an isolated cron run can be recorded as successful while the owned agent session appears to continue making provider calls. Current main has timeout cleanup and post-run context disposal, but I did not find source, release, or related-item evidence proving that success completion drains or terminates every owned execution path after `ok`.

## What This Changes

Reported isolated cron runs can mark `ok` while the underlying agent session continues making provider calls and consuming tokens until restart or timeout.

## Best Possible Solution

Make cron terminal success prove that the owned run and any descendant/background execution are settled, drained, or canceled before logging `ok`, or route the remaining work into the canonical session/transcript lifecycle owner if that is the true boundary.

## Reproduction Assessment

No high-confidence current-main reproduction is established. The reporter supplied detailed production telemetry, and source inspection shows the relevant success and timeout cleanup boundaries, but I did not replay the delayed provider-call loop on current main.

## Solution Assessment

Unclear for a single automatic fix. The likely best direction is the cron run lifecycle boundary, but maintainers need to identify whether the late calls are owned by the embedded runner, descendant run tracking, provider retry behavior, or session/transcript state.
