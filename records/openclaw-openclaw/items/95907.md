## Label Justifications

- `P1`: The report describes a scheduled agent lifecycle leak that can continue spending model tokens for hours after a cron run is recorded as successful.

## Summary

Keep this issue open: current main still gates isolated cron session cleanup on `deleteAfterRun`, recurring isolated create defaults do not set that flag, and the linked candidate PR is open rather than merged.

## What This Changes

The issue requests that successful isolated recurring cron runs terminate their underlying agent session promptly instead of continuing to consume model tokens after the cron run is marked ok.

## Best Possible Solution

Review and reconcile the linked candidate PR so isolated cron run contexts are bounded after success for both fresh and upgraded jobs, without breaking explicit retention choices or recurring cron APIs.

## Reproduction Assessment

No for the full live provider-token loop; the source path is reproducible because recurring isolated creates omit `deleteAfterRun` and cleanup returns early when that flag is absent. A real provider run would still be needed to prove the delayed billing cycle exactly.

## Solution Assessment

Unclear: fixing the lifecycle is necessary, but the best implementation still needs maintainer direction because open PRs propose conflicting contracts for recurring `deleteAfterRun` and existing persisted jobs.
