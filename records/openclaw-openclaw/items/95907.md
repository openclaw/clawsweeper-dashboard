## Label Justifications

- `P1`: The report describes a scheduled-agent lifecycle leak that can continue model calls and token spend after the cron run is recorded complete.
- `impact:session-state`: The affected behavior is an isolated cron session outliving and diverging from the parent run's terminal state.

## Summary

Current main still leaves recurring isolated cron jobs able to run with `deleteAfterRun` omitted, so the session cleanup path can skip deletion after the parent run is marked successful. This issue should stay open because the linked candidate fix is still open and has an unresolved upgrade/default decision.

## What This Changes

The issue requests that isolated cron agent sessions terminate or clean up when the cron run is recorded as successful, instead of continuing to consume model tokens afterward.

## Best Possible Solution

Review and land the linked PR, or a replacement, only once it handles both newly created and already persisted isolated cron jobs while preserving explicit `deleteAfterRun: false`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: create a recurring isolated cron job with omitted `deleteAfterRun`, and current main can leave the flag omitted while the terminal cleanup helper exits unless it is true. I did not establish a live provider-token reproduction.

## Solution Assessment

Is this the best way to solve the issue? The linked defaulting change is the right direction for fresh jobs, but the best solution also needs an explicit persisted-job upgrade decision before the issue is fully solved.
