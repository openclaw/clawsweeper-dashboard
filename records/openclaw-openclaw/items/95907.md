## Summary

Keep this issue open: current main still leaves recurring isolated cron jobs without an implicit `deleteAfterRun`, and the linked fix is an open PR rather than merged behavior. The current cleanup path is present for `deleteAfterRun: true`, but it is still gated by that flag.

## What This Changes

The issue reports recurring isolated cron runs leaving the underlying agent session alive after the cron run is recorded successful, causing continued provider token usage.

## Best Possible Solution

Merge a narrow cron lifecycle fix that defaults cleanup for implicit isolated cron jobs and explicitly handles or documents the upgrade behavior for existing persisted jobs, while preserving explicit `deleteAfterRun: false`.

## Reproduction Assessment

No, not with live-provider confidence. Source inspection does reproduce the gating problem: recurring isolated cron jobs currently do not get `deleteAfterRun`, and the terminal cleanup helper exits unless that flag is true.

## Solution Assessment

Unclear whether the current linked PR is the complete best fix. Its defaulting change is narrow and plausible, but maintainers should confirm upgrade behavior for existing implicit isolated cron jobs before calling the issue fully solved.
