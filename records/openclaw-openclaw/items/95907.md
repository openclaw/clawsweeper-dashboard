## Summary

Keep this issue open: current main still defaults `deleteAfterRun` only for one-shot jobs, so recurring isolated cron jobs can still miss the cleanup path unless the flag is explicit, and the direct fix is an open PR rather than merged behavior.

## What This Changes

The issue reports that a successful recurring isolated cron run can leave its underlying agent session alive, causing continued model-token consumption after the cron run is marked ok.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95950 or a replacement that defaults cleanup for recurring isolated cron jobs, preserves explicit `deleteAfterRun: false`, and handles existing persisted jobs deliberately.

## Reproduction Assessment

No, not with high-confidence live-provider reproduction. Source inspection does reproduce the likely cleanup gate: recurring isolated cron jobs can omit `deleteAfterRun`, and the terminal cleanup helper exits unless that flag is true.

## Solution Assessment

Unclear. The linked defaulting fix is narrow and plausible, but maintainers still need to decide upgrade behavior for existing persisted jobs and whether create-time normalization alone is enough.
