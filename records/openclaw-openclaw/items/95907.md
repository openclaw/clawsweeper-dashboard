## Summary

Keep this issue open: current main still lets recurring isolated cron jobs omit `deleteAfterRun`, and the open closing PR has not landed. Source inspection supports the cleanup-gating root cause, but the exact long-running provider-token loop still lacks a live reproduction on current main.

## What This Changes

The issue reports that an isolated recurring cron run can be marked `ok` while the underlying agent session keeps running and consuming model tokens.

## Best Possible Solution

Land one cron lifecycle fix, through https://github.com/openclaw/openclaw/pull/95950 or a replacement, that cleans up implicit isolated cron sessions for both fresh and upgraded jobs while preserving explicit opt-out.

## Reproduction Assessment

No, not as a high-confidence live repro on current main. Source inspection does reproduce the cleanup gate: recurring isolated cron jobs can omit `deleteAfterRun`, and terminal cleanup returns false for omitted values.

## Solution Assessment

Unclear. The linked PR is a plausible narrow fix for fresh job creation, but maintainers still need to decide or implement upgrade behavior for existing persisted jobs before this is the best complete fix.
