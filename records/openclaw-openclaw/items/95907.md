## Label Justifications

- `P1`: The report describes a scheduled-agent lifecycle leak that can keep spending model tokens for hours after a run is recorded successful.
- `impact:session-state`: The affected behavior is isolated cron session lifecycle and cleanup state after the parent cron run completes.

## Summary

Keep open: current main still lets recurring isolated cron jobs omit `deleteAfterRun`, and the session cleanup path only runs when that flag is truthy. The linked fix PR is still open, and a second open PR proposes the opposite product direction for recurring `deleteAfterRun`, so this issue remains necessary until maintainers choose and land one path.

## What This Changes

The issue asks for isolated cron sessions to stop or clean up immediately after a successful run is recorded, preventing post-completion token consumption.

## Best Possible Solution

Choose the cron lifecycle contract, then land one fix that handles fresh and existing isolated cron jobs while preserving explicit `deleteAfterRun: false` or intentionally rejecting the recurring combination with aligned callers.

## Reproduction Assessment

Yes for the cleanup/defaulting gap by source inspection: recurring isolated cron jobs can omit `deleteAfterRun`, and cleanup returns early for omitted values. No live-provider reproduction of the token loop was run in this review.

## Solution Assessment

Unclear. https://github.com/openclaw/openclaw/pull/95950 is a plausible fresh-create fix, but existing persisted jobs and the conflicting direction in https://github.com/openclaw/openclaw/pull/96045 need maintainer resolution before this issue is solved.
