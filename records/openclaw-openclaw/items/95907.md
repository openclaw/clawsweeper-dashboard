## Summary

Keep this issue open: current main still leaves recurring isolated cron jobs without an implicit cleanup flag, and the direct fixing PR is open rather than merged. The source path supports the reported lifecycle leak even though the exact long-running provider-token loop was not live-replayed.

## What This Changes

The issue reports isolated recurring cron jobs leaving the underlying agent session alive after the cron run is recorded as ok, causing continued model calls and token spend.

## Best Possible Solution

Land or replace the linked cron lifecycle fix with a maintainer-approved contract that prevents implicit isolated cron sessions from surviving terminal runs, covers existing persisted jobs or documents their migration path, and preserves any deliberate explicit opt-out behavior.

## Reproduction Assessment

Yes, source-level: recurring isolated jobs can omit deleteAfterRun, and the terminal cleanup helper returns false unless that field is truthy. No, this review did not live-replay the provider-side token cycle from the original WSL2 deployment.

Unclear: the linked defaulting PR is a plausible fresh-create fix, but the best solution needs maintainer agreement on persisted omitted jobs and the competing one-shot-only recurring deleteAfterRun contract.
