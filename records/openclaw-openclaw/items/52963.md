## Label Justifications

- `P2`: This is a valid Slack plugin feature request with clear code paths, but it is not an urgent regression in documented current behavior.
- `impact:message-loss`: User-level Slack DM events to the dedicated user are not routed into OpenClaw for this workflow.
- `impact:auth-provider`: The request depends on Slack bot/app/user token selection and xoxp reply routing.
- `impact:security`: Acting as a regular Slack user token changes identity, authorization, and audit boundaries.

## Summary

Keep open. Current main and the latest shipped release still model Slack Socket Mode as a bot-event, bot-token monitor path, so the requested user-level DM routing remains valid Slack plugin feature work that needs maintainer product/security approval.

## What This Changes

Support Slack user-level `message.im` events and xoxp reply routing so DMs to a dedicated Slack user can be handled by OpenClaw as that user.

## Best Possible Solution

Implement this in the bundled Slack plugin after approval: add `user_events` manifest/docs support, resolve the configured xoxp user identity, select the user reply token only on the approved opt-in path, prevent self-loops, and add focused unit plus live Slack Socket Mode proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still emits only Slack `bot_events`, builds monitor identity from the bot token, and posts normal monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The implementation boundary is the bundled Slack plugin, but maintainers should first decide the xoxp authorship policy and whether existing `userTokenReadOnly: false` is sufficient opt-in.
