## Label Justifications

- `P2`: This is a valid Slack plugin feature request with clear source paths and limited blast radius, but it is not an urgent regression in documented current behavior.
- `impact:security`: The request would allow OpenClaw to receive and reply as a regular Slack user identity rather than the app identity.
- `impact:message-loss`: User-level Slack DM events to the dedicated user are not routed into OpenClaw for this workflow.
- `impact:auth-provider`: The behavior depends on Slack app, bot, and user token selection plus per-inbound reply routing.

## Summary

Keep open. Current main and the latest shipped release still expose Slack Socket Mode through bot-event subscriptions, bot-token monitor identity, and bot-token monitor replies, so the requested user-level Slack DM routing is real Slack plugin feature work that needs product and security approval before implementation.

## What This Changes

Request support for Slack Socket Mode user-level `message.im` events so DMs to a configured xoxp user can be routed and replied to as that user.

## Best Possible Solution

Implement this in the bundled Slack plugin after approval: add user-event manifest/docs support, resolve the configured xoxp user identity, route inbound events by authorizing user context, select the reply token per inbound authorization, prevent self-loops, and prove it with focused tests plus live Slack Socket Mode validation.

## Reproduction Assessment

Yes for source-level reproduction: current main emits only Slack `bot_events`, builds monitor identity from the bot token, and posts normal monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The implementation boundary is the bundled Slack plugin, but maintainers should first decide the xoxp authorship policy and whether existing `userTokenReadOnly: false` is sufficient opt-in.
