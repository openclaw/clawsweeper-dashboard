## Label Justifications

- `P2`: This is a valid Slack plugin feature request with clear source paths and limited blast radius, but it is not an urgent regression in documented current behavior.
- `impact:security`: The requested behavior would let OpenClaw receive events and send replies as a regular Slack user identity rather than only the app identity.
- `impact:message-loss`: User-level Slack DM events to the dedicated user are not routed into OpenClaw for this workflow.
- `impact:auth-provider`: The behavior depends on Slack app subscriptions, bot token, user token, and per-inbound authorization routing.

## Summary

Keep open. Current main still models Slack Socket Mode around bot-event subscriptions, bot-token monitor identity, and bot-token reply delivery, so user-level DM routing remains valid Slack plugin feature work that needs product and security approval before implementation.

## What This Changes

Support Slack `message.im` events delivered on behalf of a configured user token and route replies through that user token for dedicated-user DM workflows.

## Best Possible Solution

Implement this inside the bundled Slack plugin after approval: add user-event manifest/docs support, resolve the configured xoxp user identity, route inbound events by Slack authorization, choose the reply token per inbound authorization, prevent self-loops, and prove it with focused tests plus live Slack validation.

## Reproduction Assessment

Yes for source-level reproduction: current main emits only Slack `bot_events`, builds monitor identity from the bot token, and posts normal monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The implementation boundary is the Slack plugin, but maintainers should first decide the xoxp authorship policy and whether existing `userTokenReadOnly: false` is sufficient opt-in.
