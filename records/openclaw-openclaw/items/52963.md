## Label Justifications

- `P2`: This is a meaningful Slack integration gap with a limited surface and an existing bot-DM workaround, not an emergency regression.
- `impact:security`: The request would let OpenClaw operate as a regular Slack user and needs explicit authorization and impersonation safeguards.
- `impact:message-loss`: The reported user-level `message.im` events for the configured Slack user are not routed into OpenClaw today.
- `impact:auth-provider`: The remaining design depends on choosing between Slack bot, app, and user tokens for inbound authorization and outbound replies.

## Summary

Keep open: current main and the latest release still use bot-token-centered Slack Socket Mode plus bot-preferred write routing, and no related merged PR implements user-authorized DM event routing. This is a security-sensitive Slack user-account feature, so it needs maintainer product/security design before a fix PR.

## What This Changes

Support Slack user-token Socket Mode DM events and route replies through the configured Slack user token so a dedicated Slack user account can act as the agent identity.

## Best Possible Solution

Design an explicit Slack plugin user-account DM mode that binds inbound user-authorized events to the configured user token, sends replies with the matching token, documents required scopes and APP-badge/security tradeoffs, and keeps the current bot/app behavior as the default.

## Reproduction Assessment

Partially: there is a high-confidence source-level reproduction because current main has no Slack authorization-envelope handling and still prefers bot-token writes when a bot token exists. I did not run a live Slack workspace with user-event subscriptions, so this is not a live reproduction.

## Solution Assessment

No for an automatic narrow bug fix: the requested behavior is plausible, but the safest solution is an explicit Slack user-auth mode with maintainer-approved security and scope semantics rather than inferring full user-account control from `userTokenReadOnly: false`.
