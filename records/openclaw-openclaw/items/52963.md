## Summary

Keep this open. Current main still treats Slack user tokens as supplemental read/fallback-write credentials, while the Slack monitor setup, inbound identity model, and reply dispatch remain bot-token centered; the requested user-level Socket Mode DM routing is not implemented.

## What This Changes

Request Slack support for user-level Socket Mode `message.im` events and opt-in xoxp replies so DMs to a regular Slack user can be handled without the Slack app badge.

## Best Possible Solution

Implement this in the bundled Slack plugin only after maintainer approval: generate and document the `user_events` setup, resolve the configured user identity at monitor startup, distinguish bot-level from user-level DM events, select the user token only for explicit opt-in replies, prevent self-loops, and add targeted unit plus live Slack validation.

## Reproduction Assessment

Yes for source-level reproduction: current main shows no `user_events` setup, no user-token identity in monitor context, and bot-token reply dispatch. A live Slack Socket Mode run would still be useful to verify the exact user-level event envelope.

## Solution Assessment

No automated fix path is best yet. The narrow implementation belongs in the Slack plugin, but user-token impersonation and inbound identity routing need maintainer approval and live validation first.
