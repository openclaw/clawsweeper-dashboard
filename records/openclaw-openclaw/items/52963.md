## Label Justifications

- `P2`: This is a valid Slack feature request with clear affected code paths, but it is not an urgent regression in documented current behavior.
- `impact:message-loss`: The reported user-level Slack DM events are not routed into OpenClaw, so inbound messages to the dedicated user are effectively dropped for this workflow.
- `impact:auth-provider`: The requested behavior depends on Slack bot/app/user token selection and xoxp reply routing.
- `impact:security`: Acting and replying as a Slack user token changes identity and audit boundaries and should receive explicit security review.

## Summary

Keep open. Current main and the latest shipped release still expose Slack Socket Mode as a bot-event, bot-token monitor path, so this remains valid bundled Slack plugin feature work that needs maintainer product/security approval before implementation.

## What This Changes

Request support for Slack user-level `message.im` events and xoxp replies so DMs to a dedicated Slack user can route through OpenClaw.

## Best Possible Solution

Implement this inside the bundled Slack plugin after approval: add `user_events` manifest/docs support, resolve the configured xoxp user identity, select the user reply token only on the approved opt-in path, prevent self-loops, and add focused unit plus live Slack Socket Mode proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still emits only Slack `bot_events`, builds monitor identity from the bot token, and posts normal monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The right implementation boundary is the bundled Slack plugin, but maintainers should first decide the user-token authorship policy and approved opt-in semantics.
