## Summary

Keep this issue open. Current main still does not implement the requested Slack user-level DM path: the bundled Slack plugin generates and documents bot-event subscriptions, resolves monitor identity from the bot token, and sends monitored replies through the bot token rather than selecting the configured xoxp user token for user-level DM conversations.

## What This Changes

The issue requests Slack user-level `message.im` event handling and xoxp reply routing so DMs sent to a configured Slack user account can be handled and answered as that user.

## Best Possible Solution

Implement an explicit, opt-in Slack plugin path for user-level DMs: generate and document the `user_events` setup, resolve the configured xoxp user's identity at monitor startup, distinguish bot-level and user-level DM events, select the user token only for approved user-level replies, prevent self-loops, and cover the behavior with focused unit tests plus live Slack validation.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: current main lacks generated `user_events`, lacks monitor user-token identity, and dispatches monitored replies through `ctx.botToken`; live Slack validation remains useful for payload details, not for proving the feature is still missing.

## Solution Assessment

Yes, the best boundary is a Slack-plugin implementation rather than a core change. The actual fix should wait for maintainer approval because the user-token impersonation and self-loop policy are product/security decisions, not a routine cleanup patch.
