## Summary

Keep this open. Current main still has bot-token-centered Slack Socket Mode setup and reply dispatch; configured Slack user tokens are present for supplemental reads and fallback writes, but there is no implemented user-level Socket Mode DM routing or per-event user-token reply selection.

## What This Changes

Request support in the bundled Slack plugin for `message.im` events delivered on behalf of a configured Slack user token, with replies sent through that user token.

## Best Possible Solution

Implement this in the bundled Slack plugin only after maintainer approval: add and document optional `user_events` setup, resolve the configured user identity at monitor startup, distinguish bot-level from user-level DM events, choose the user token only for explicit opt-in reply paths, prevent self-loops, and cover it with focused unit tests plus live Slack validation.

## Reproduction Assessment

Yes for source-level reproduction: current main has no user-events manifest/docs surface, no user-token identity in the monitor context, and normal monitor replies always pass `ctx.botToken`. Live Slack validation is still needed for the exact Slack payload shape.

## Solution Assessment

No automated implementation path is best yet. The narrow code location is the Slack plugin, but the authorship and impersonation semantics need maintainer product/security approval before a fix PR should be queued.
