## Label Justifications

- `P2`: This is valid Slack plugin feature work with clear source paths and limited blast radius, but it is not an urgent regression in documented current behavior.
- `impact:security`: The requested behavior would let OpenClaw receive and reply as a regular Slack user identity rather than only the app identity.
- `impact:message-loss`: User-level Slack DM events to the dedicated user are not routed into OpenClaw for this workflow.
- `impact:auth-provider`: The behavior depends on Slack app, bot, and user token selection plus per-inbound reply routing.

## Summary

Keep open. Current main and the latest shipped release still model Slack Socket Mode around bot-event subscriptions, bot-token monitor identity, and bot-token reply dispatch, so user-level Slack DM routing remains feature work that needs maintainer product and security approval.

## What This Changes

Support Slack user-level Socket Mode events so DMs to a configured Slack user token can be routed into OpenClaw and answered as that user.

## Best Possible Solution

Implement this in the bundled Slack plugin after approval: add user-event manifest/docs support, route inbound events by authorizing user context, choose reply tokens per inbound authorization, prevent self-loops, and prove it with focused tests plus live Slack validation.

## Reproduction Assessment

Yes for source-level reproduction: current main emits only Slack `bot_events`, builds monitor identity from the bot token, and posts normal monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The implementation boundary is the Slack plugin, but maintainers should first decide the xoxp authorship policy and whether existing `userTokenReadOnly: false` is sufficient opt-in.
