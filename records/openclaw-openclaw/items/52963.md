## Summary

Keep open: current main and v2026.6.8 still keep Slack Socket Mode and normal replies bot-token centered, do not model user-authorized event envelopes, and do not select a user token based on inbound authorization. The request is useful but security- and product-sensitive because it would let OpenClaw receive and send Slack DMs as a regular user identity.

## What This Changes

Requested Slack support for user-authorized `message.im` events so DMs to a dedicated Slack user can route into OpenClaw and reply through the configured user token.

## Best Possible Solution

Design an explicit Slack-plugin-owned user-account DM mode that maps inbound event authorizations to configured user tokens, sends replies with the matching token, documents scopes/audit tradeoffs, and keeps bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows the missing path on current main and v2026.6.8: there is no user-authorized envelope model and normal replies still send with the bot token.

## Solution Assessment

No, not as an automatic bug fix. The maintainable solution is an explicit Slack user-account mode with authorization, scope, token-selection, and audit semantics rather than inferring full user control from the existing read/write token setting.
