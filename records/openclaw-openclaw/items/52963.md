## Summary

Keep open: current `main` and the latest release still do not model Slack user-authorized event envelopes or select the matching user token for normal Slack replies when a bot token is also configured. The issue is clear and source-reproducible, but the requested human-user Slack identity mode needs maintainer product and security direction before implementation.

## What This Changes

Support Slack user-level direct-message events and reply routing through the configured Slack user token so a dedicated human Slack account can be controlled by OpenClaw.

## Best Possible Solution

Design an explicit Slack-plugin-owned user-account DM mode that maps inbound event authorizations to configured user tokens, sends replies with the matching token, documents scopes/audit tradeoffs, and keeps bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show the missing path: current main and `v2026.6.8` still require bot/app tokens for the monitor, lack a user-authorization envelope model, and prefer bot-token writes while a bot token exists.

## Solution Assessment

No, not as an automatic bug fix. The maintainable solution is an explicit Slack user-account mode with authorization, scope, token-selection, and audit semantics rather than inferring full user control from the existing read/write token setting.
