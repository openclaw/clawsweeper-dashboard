## Summary

Keep open: current main and the latest release still do not implement Slack user-authorized Socket Mode event routing or normal replies as the configured xoxp user. The request remains useful, but it is a feature/product and security-boundary decision rather than a narrow automatic bug fix.

## What This Changes

Request Slack to receive user-authorized `message.im` events for a configured xoxp user token and reply as that user instead of the app/bot identity.

## Best Possible Solution

Design an explicit Slack-plugin-owned user-account DM mode that maps inbound Slack authorizations to configured user tokens, deliberately selects the matching send token, documents scopes and audit tradeoffs, and keeps bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection supports the report because current main lacks user-authorized event-envelope routing and normal replies still send with the bot token.

## Solution Assessment

No, overloading `userTokenReadOnly: false` alone is not the best solution. The safer path is an explicit user-account mode with authorization mapping, token-selection, docs, and security guardrails.
