## Summary

Keep open: current main and v2026.6.10 still construct Slack Socket Mode around the bot token and send normal replies with the bot token, so Slack user-authorized DM routing and user-token replies are not implemented. The request is useful but needs product and security review because it would let OpenClaw receive and respond as a regular Slack user identity.

## What This Changes

Support Slack user-level `message.im` events and route replies through the configured user token for DMs sent to a regular Slack user.

## Best Possible Solution

Design a Slack-plugin-owned user-account DM mode that explicitly maps inbound Slack authorizations to configured user tokens, selects the matching send token, documents required scopes and audit tradeoffs, and keeps bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection supports the missing capability because current main has bot-token Slack app construction, bot-token normal replies, bot-event setup, and no authorizations-based user-token routing path.

## Solution Assessment

No, overloading `userTokenReadOnly: false` alone is not the best way to solve this. The safer route is an explicit user-account DM mode with authorization mapping, token selection, tests, docs, and security guardrails.
