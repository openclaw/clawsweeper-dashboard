## Summary

Keep open: current main and v2026.6.6 still lack an explicit Slack user-authorized event route and user-token reply selection, and the request needs product/security design rather than conservative cleanup closure.

## What This Changes

The issue requests support for Slack user-level `message.im` events and xoxp-based replies so DMs to a dedicated Slack user can be routed through OpenClaw.

## Best Possible Solution

Design an explicit Slack plugin user-account DM mode that binds inbound user-authorized events to the matching configured user token, uses that token for replies, documents required scopes and identity/security tradeoffs, and keeps current bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a clear current-main explanation: Slack user authorization envelopes are not modeled in the inbound path, and write routing still prefers the bot token when one exists.

## Solution Assessment

No for an automatic bug fix. The safer path is a maintainer-approved Slack user-account mode with explicit identity, scope, routing, and audit semantics rather than inferring full user control from the existing read/write token setting.
