- `impact:security`: The requested behavior would let OpenClaw receive events and send replies as a regular Slack user identity rather than only the app identity.

- `impact:auth-provider`: The behavior depends on Slack app subscriptions, bot token, user token, and per-inbound authorization routing.

## Summary

Keep open. Current main and the latest v2026.5.22 release still model Slack Socket Mode around bot-event setup, bot-token monitor identity, and bot-token reply delivery, so user-level DM routing remains valid Slack plugin feature/security work that needs maintainer product approval.

## What This Changes

Request support for Slack user-level Socket Mode `message.im` events and user-token replies so DMs to a dedicated Slack user account can route through OpenClaw.

## Best Possible Solution

Implement this inside the bundled Slack plugin after approval: add user-event manifest/docs support, resolve the configured xoxp user identity, route inbound events by Slack authorization, choose the reply token per inbound authorization, prevent self-loops, and prove it with focused tests plus live Slack validation.

## Reproduction Assessment

Yes for source-level reproduction: current main emits only Slack bot-event setup, builds monitor identity from the bot token, and posts normal monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The Slack plugin is the right boundary, but maintainers should first decide the xoxp authorship policy and whether existing `userTokenReadOnly: false` is sufficient opt-in or a new explicit mode is required.
