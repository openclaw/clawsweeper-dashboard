## Label Justifications

- `P2`: This is a clear Slack plugin feature request with source-level evidence, but it is not an urgent regression in documented current behavior.
- `impact:message-loss`: User-level Slack DMs to the dedicated user are not routed into OpenClaw for this workflow.
- `impact:auth-provider`: The request depends on Slack bot/app/user token selection and xoxp reply routing.
- `impact:security`: Acting as a regular Slack user token changes identity, authorization, and audit boundaries.

## Summary

Keep open. Current main still models Slack Socket Mode as a bot-event, bot-token monitor path, so user-level Slack DM routing remains valid Slack plugin feature work that needs maintainer product and security approval.

## What This Changes

Support Slack Socket Mode `message.im` events subscribed on behalf of a configured user token and route approved DM replies through that user token.

## Best Possible Solution

Implement this in the bundled Slack plugin after approval: add `user_events` manifest/docs support, route inbound envelopes by authorizing user, select the xoxp reply token only on the approved opt-in path, prevent self-loops, and prove it with focused tests plus live Slack Socket Mode validation.

## Reproduction Assessment

Yes for source-level reproduction: current main still emits only Slack `bot_events`, builds monitor identity from the bot token, and posts monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The implementation boundary is the Slack plugin, but maintainers should first decide the xoxp authorship policy and whether `userTokenReadOnly: false` is sufficient opt-in.
