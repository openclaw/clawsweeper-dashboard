## Summary

Keep open: current main and v2026.6.6 still center Slack Socket Mode and write routing on the bot token, and the generated Slack manifest/docs do not include user-event subscription support. The request remains a security-sensitive Slack user-account feature that needs maintainer product/security design before implementation.

## What This Changes

The issue requests Slack Socket Mode support for user-authorized DM events and user-token replies so a dedicated Slack user account can be agent-controlled without the app badge.

## Best Possible Solution

Design an explicit Slack plugin user-account DM mode that binds inbound user-authorized events to the matching configured user token, uses that token for replies, documents required scopes and APP-badge/security tradeoffs, and keeps current bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show the current Slack runtime does not route on Slack authorization envelopes and still prefers bot-token writes when a bot token exists.

## Solution Assessment

No for an automatic bug fix: the requested behavior is plausible, but the safer path is an explicit Slack user-account mode with maintainer-approved identity, scope, and audit semantics rather than inferring full user control from the existing read/write token setting.
