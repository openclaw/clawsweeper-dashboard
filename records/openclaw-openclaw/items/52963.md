## Label Justifications

- `P2`: This is a valid Slack plugin feature request with clear code paths and limited blast radius, not an urgent regression in documented current behavior.
- `impact:message-loss`: User-level Slack DM events to the dedicated user are not routed into OpenClaw for this workflow.
- `impact:auth-provider`: The request depends on Slack bot/app/user token selection and xoxp reply routing.
- `impact:security`: Acting as a regular Slack user token changes identity, authorization, and audit boundaries.

## Summary

Keep open. Current main and the latest shipped release still model Slack Socket Mode inbound handling around bot-event subscriptions, bot-token monitor identity, and bot-token reply delivery, so the requested user-level Slack DM routing remains real Slack plugin feature work that needs product and security approval.

## What This Changes

Support Slack Socket Mode user-level `message.im` events for DMs to a configured xoxp user and route approved replies through that user token.

## Best Possible Solution

Implement this in the bundled Slack plugin after approval: add user-event manifest/docs support, resolve the configured xoxp user identity, route inbound events by the authorizing user context, select the reply token per inbound authorization, prevent self-loops, and prove it with focused tests plus live Slack Socket Mode validation.

## Reproduction Assessment

Yes for source-level reproduction: current main still emits only Slack `bot_events`, builds monitor identity from the bot token, and posts normal monitor replies with `ctx.botToken`. A live Slack workspace is still needed for the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The implementation boundary is the bundled Slack plugin, but maintainers should first decide the xoxp authorship policy and whether existing `userTokenReadOnly: false` is sufficient opt-in.
