## Summary

Keep open: current main and v2026.6.8 still keep Slack event handling and normal replies centered on the bot token, while the issue asks for a separate user-authorized Socket Mode identity path. The request is useful but product- and security-sensitive because it would let OpenClaw receive and send Slack DMs as a regular user identity.

## What This Changes

The issue asks OpenClaw’s Slack channel to process user-authorized Socket Mode `message.im` events and reply through the configured Slack user token so DMs to a regular Slack user route into OpenClaw.

## Best Possible Solution

Design an explicit Slack-plugin-owned user-account DM mode that maps inbound Slack authorizations to configured user tokens, selects the matching send token deliberately, documents scopes and audit tradeoffs, and keeps bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a medium-confidence reproduction basis: current main and v2026.6.8 do not model a user-authorized Slack event envelope and normal replies still send with the bot token.

## Solution Assessment

No, not as an automatic bug fix using the existing read/write flag alone. The maintainable solution is an explicit Slack user-account mode with authorization, scope, token-selection, and audit semantics rather than inferring full user control from `userTokenReadOnly: false`.
