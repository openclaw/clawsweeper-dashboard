## Label Justifications

- `P2`: This is a valid Slack plugin feature request with a clear affected surface and limited blast radius, but it is not a documented-regression emergency.
- `impact:security`: The feature would allow OpenClaw to process inbound events and send replies as a regular Slack user identity.
- `impact:message-loss`: DMs sent to the dedicated Slack user are not routed into OpenClaw under the requested workflow.
- `impact:auth-provider`: The behavior depends on Slack bot/app/user tokens, OAuth scopes, event authorizations, and per-inbound send-token selection.

## Summary

Keep open. Current main and the latest checked release still model Slack inbound monitoring around bot-event subscriptions, bot-token monitor identity, and bot-token reply delivery, while the requested user-level DM flow needs explicit Slack plugin product and security approval.

## What This Changes

Request Slack plugin support for Socket Mode user-level `message.im` events and per-authorization user-token reply routing for dedicated-user DMs.

## Best Possible Solution

Implement this inside the bundled Slack plugin after approval: generate/document `user_events`, read event `authorizations`, resolve the configured xoxp user identity, route inbound DMs by authorizing user, select the reply token from that authorization, and prove loop prevention with tests plus live Slack validation.

## Reproduction Assessment

Yes for source-level reproduction: current main emits only Slack `bot_events`, derives monitor identity from the bot token, and posts monitor replies with `ctx.botToken`. I did not run a live Slack workspace reproduction of the exact user-level Socket Mode envelope.

## Solution Assessment

Unclear for automation. The Slack plugin is the right implementation boundary, but maintainers should decide the xoxp authorship policy and whether `userTokenReadOnly: false` is sufficient opt-in before code is queued.
