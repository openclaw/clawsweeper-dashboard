## Summary

Keep open: current `main` and v2026.6.6 still do not model Slack user-authorized event envelopes or route replies through the authorizing user token when a bot token is also configured, and the requested human-user Slack identity behavior needs explicit product and security design.

## What This Changes

Request Slack to process user-authorized `message.im` DM events and reply through the configured user token so a dedicated Slack user account can act as the agent identity.

## Best Possible Solution

Design an explicit Slack user-account DM mode that maps inbound authorizations to configured user tokens, sends replies with the matching token, documents required scopes and audit tradeoffs, and keeps bot/app behavior as the default.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows the current Slack path has no user-event authorization model and still prefers bot-token writes when both tokens exist.

## Solution Assessment

No for an automatic bug fix. The maintainable solution is an explicit Slack user-account mode with authorization, scope, token-selection, and audit semantics rather than inferring full user control from the existing read/write token setting.
