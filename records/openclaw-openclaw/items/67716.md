## Summary

Keep open. Current main still has the host scheduling gap: bootstrap/reconcile maintenance runs through the foreground path, while deferred compaction debt execution is only enabled for background maintenance that is currently scheduled for turn maintenance. The hot-cache part still looks partly plugin-owned, but the host-side bootstrap follow-up is narrow and actionable.

## What This Changes

The issue requests that context-engine bootstrap/reconcile and hot-cache maintenance stop leaving deferred compaction debt pending without a valid follow-up compaction path.

## Best Possible Solution

Add a generic host-level bootstrap/reconcile follow-up path for background-capable context engines so deferred compaction debt can run with allowDeferredCompactionExecution enabled. Keep lossless-claw-specific hot-cache policy out of core unless the plugin needs an additive API to report that policy intentionally superseded pending debt.

## Reproduction Assessment

Partly yes: the host scheduling gap is reproducible from current source with a fake context engine by calling bootstrapHarnessContextEngine and observing foreground maintenance without allowDeferredCompactionExecution or a queued background follow-up. The hot-cache sticky-debt case has concrete live traces but still needs a synthetic plugin-side repro.

## Solution Assessment

Yes for the host half: a narrow scheduling/contract fix in context-engine maintenance is the maintainable path. The hot-cache half should be solved in the plugin or through an additive generic signal rather than embedding lossless-claw policy in OpenClaw core.
