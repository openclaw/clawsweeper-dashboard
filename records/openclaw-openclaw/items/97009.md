## Summary

Keep open for maintainer merge review: the PR is not obsolete and the code change is a narrow, correct bug fix, but it intentionally changes provider retry/fallback behavior for overload-worded HTTP 429s.

## What This Changes

The PR reorders shared failover message classification, adds an overloaded-preserving HTTP 429 status branch, and adds regression coverage for overloaded, rate-limit, and billing 429 messages.

## Best Possible Solution

Land the narrow shared classifier fix once maintainers accept the overload retry/fallback semantics; a committed status-path regression would further pin the changed 429 branch.

## Reproduction Assessment

Yes, source-level: current main checks the broad 429 rate-limit matcher before overload matching and the explicit HTTP 429 branch returns `rate_limit` after billing checks. That is enough to reproduce the misclassification path without live provider access.

## Solution Assessment

Yes, the shared classifier is the right fix boundary because a provider-specific one-off would leave the same 429 overload precedence bug for other providers. The main remaining improvement is committed coverage for the explicit status branch, not a different implementation layer.
