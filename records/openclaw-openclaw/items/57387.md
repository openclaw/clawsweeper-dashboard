## Summary

Keep open. Current main has inter-session provenance marking, session visibility, agent-to-agent allowlist checks, HTTP denial for direct `sessions_send`, and plugin hook defenses, but it still has no cryptographic sender proof for `sessions_send`; the production-context comment reinforces that the structural auth gap remains.

## What This Changes

Add HMAC or token-based mutual authentication to `sessions_send` so receivers can verify claimed inter-agent sender provenance, ideally with per-agent policy.

## Best Possible Solution

Design a Gateway-managed signed provenance layer for inter-session messages, with replay protection and regression coverage, while keeping `tools.sessions.visibility` and `tools.agentToAgent` as authorization gates.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction of the missing capability: current main has no auth fields in the `sessions_send` schema or `InputProvenance` contract and no verification step before nested agent dispatch.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token signing is a plausible direction, but the fix should be designed as a core Gateway security/protocol feature rather than a narrow `sessions_send` parameter patch.
