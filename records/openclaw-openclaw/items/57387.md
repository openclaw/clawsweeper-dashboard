## Summary

Keep open. Current main still has `tools.agentToAgent` and session-visibility gates, and HTTP tool invocation hard-denies `sessions_send`, but the inter-session message path still carries unsigned provenance metadata with no HMAC, nonce, signature, or replay verification; the KinthAI comment describes an external mitigation, not an OpenClaw core fix.

## What This Changes

Request HMAC or token-based mutual authentication for `sessions_send` so inter-agent messages cannot forge their sender session.

## Best Possible Solution

Add a Gateway-managed signed provenance/auth layer for inter-session messages with replay protection, protocol/schema updates, tests, docs, and policy/config alignment while keeping visibility and A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction of the missing capability: `sessions_send`, `InputProvenance`, and the Gateway `agent` path carry plain source metadata with no auth material or verification step.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be a Gateway-managed signed provenance design with replay and key lifecycle rules rather than a model-controlled optional tool parameter.
