## Summary

Keep open. Current main still marks inter-session traffic and applies session visibility, `tools.agentToAgent`, and HTTP tool-invoke deny-list controls, but it does not provide cryptographic sender proof or replay protection for `sessions_send`; the production-context comment describes an external HMAC mitigation, not a core fix.

## What This Changes

Request HMAC/token-based mutual authentication for inter-agent `sessions_send` payloads, ideally configurable per agent or session.

## Best Possible Solution

Add a Gateway-managed signed provenance/auth layer for inter-session messages, with replay protection and tests, while preserving `tools.sessions.visibility` and `tools.agentToAgent` as authorization gates.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction of the missing capability: current main has no auth fields in the `sessions_send` schema or `InputProvenance` contract and no verification step before nested agent dispatch.

## Solution Assessment

Unclear at implementation-detail level. The HMAC/token direction is plausible, but the exact design should be reviewed as a core Gateway security/protocol change rather than treated as a narrow tool patch.
