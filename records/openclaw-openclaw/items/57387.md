## Summary

Keep open. Current main and the latest release still provide visibility gates and prompt/transcript provenance for `sessions_send`, but not a non-forgeable HMAC/token sender-authentication or replay-protection contract; the production KinthAI comment reinforces this as real multi-agent security hardening that needs security/product design.

## What This Changes

Request HMAC or token-based mutual authentication for inter-agent `sessions_send` messages so receivers can verify the claimed sender session.

## Best Possible Solution

Design a Gateway-managed signed inter-session provenance contract with replay protection, key lifecycle, capability-scoped sender policy, per-agent enforcement settings, docs, and regression coverage while keeping current visibility gates as defense in depth.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: the current `sessions_send` schema, A2A flow, `InputProvenance`, and Gateway protocol carry plain provenance metadata without HMAC, token, signature, nonce, timestamp, or replay verification. I did not run a live exploit because this review is read-only.

## Solution Assessment

Unclear at the exact implementation level. HMAC/token auth is a plausible direction, but the maintainable solution should be a Gateway-owned signed provenance and replay contract rather than exposing model-supplied secrets or ad hoc custom headers as the permanent API.
