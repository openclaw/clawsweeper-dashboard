## Summary

Keep open. Current main still treats `sessions_send` sender identity as access-gated metadata rather than non-forgeable proof: there is visibility/A2A policy and inter-session provenance labeling, but no HMAC/token/signature, nonce, timestamp, replay guard, or verification of the claimed source session.

## What This Changes

Request HMAC/token-based mutual authentication for inter-agent `sessions_send` so receiving agents can verify the claimed sender session, ideally with per-agent policy.

## Best Possible Solution

Design Gateway-managed signed inter-session provenance with replay protection, key lifecycle, protocol/schema updates, policy/config alignment, docs, and regression coverage while preserving existing visibility/A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_send`, A2A follow-ups, `InputProvenance`, and Gateway `agent` request handling carry plain source metadata without an auth material or verification step.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable solution should be Gateway-managed signed provenance with replay and key lifecycle rules rather than caller/model-supplied secrets in tool params.
