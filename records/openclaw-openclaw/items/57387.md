## Summary

Keep open. Current main still has provenance marking, session visibility, `tools.agentToAgent`, and HTTP tool-invoke denial as useful controls, but it does not cryptographically bind a `sessions_send` sender to the claimed session or provide replay protection; the production comment describes an external HMAC mitigation, not an OpenClaw fix.

## What This Changes

Request HMAC or token-based mutual authentication for inter-agent `sessions_send` traffic, ideally configurable per agent/session.

## Best Possible Solution

Add a Gateway-managed signed provenance/auth layer for inter-session messages, with replay protection, protocol/schema updates, regression tests, and docs, while preserving `tools.sessions.visibility` and `tools.agentToAgent` as authorization gates.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions_send`, `InputProvenance`, and the Gateway `agent` path carry plain source metadata with no auth material or verification step.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be a Gateway-managed signed provenance design with replay/key lifecycle rather than a narrow optional tool parameter.
