## Summary

Keep open. Current main still provides inter-session provenance metadata, session visibility, and `tools.agentToAgent` authorization gates, but it does not cryptographically bind a `sessions_send` sender to the claimed session or provide replay protection; the production discussion describes an external HMAC mitigation, not an OpenClaw core fix.

## What This Changes

The issue requests HMAC or token-based mutual authentication for inter-agent `sessions_send` traffic, ideally configurable per agent or session.

## Best Possible Solution

Add a Gateway-managed signed provenance/auth layer for inter-session messages with replay protection, protocol/schema updates, focused regression tests, docs, and policy/config alignment while keeping existing visibility and A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction of the missing capability: `sessions_send`, `InputProvenance`, and the Gateway `agent` path carry plain source metadata with no auth material or verification step.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be a Gateway-managed signed provenance design with replay and key lifecycle rules rather than a narrow optional tool parameter.
