## Summary

Keep open. Current main has inter-session prompt marking, session visibility checks, `tools.agentToAgent` allowlist checks, HTTP deny-listing for `sessions_send`, and hook-based defense in depth, but it still does not provide cryptographic sender proof for `sessions_send`; the later production-context comment reinforces that the structural auth gap remains.

## What This Changes

Request HMAC/token-based mutual authentication for inter-agent `sessions_send` payloads, ideally configurable per agent/session.

## Best Possible Solution

Add a Gateway-managed signed provenance/auth layer for inter-session messages while preserving `tools.sessions.visibility` and `tools.agentToAgent` as authorization gates.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction of the missing capability: current main has no auth fields in the sessions_send schema or InputProvenance contract and no verification step before nested agent dispatch. I did not run a live exploit because this was a read-only review.

## Solution Assessment

Unclear at implementation-detail level. The HMAC/token direction is plausible, but the exact design should be reviewed as a core Gateway security/protocol change rather than treated as a narrow tool patch.
