## Summary

Keep open. Current main and the latest release still carry inter-session provenance as plain metadata plus visibility/allowlist gates, not as a non-forgeable sender-authentication contract; the KinthAI production comment is relevant signal, but the fix needs security/product design before implementation.

## What This Changes

The issue requests per-session HMAC/token-based authentication and optional per-agent enforcement for inter-agent `sessions_send` messages.

## Best Possible Solution

Design Gateway-managed signed inter-session provenance with replay protection, key lifecycle, configurable enforcement policy, protocol/schema updates, docs, and regression coverage while retaining current visibility and A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_send`, A2A follow-ups, `InputProvenance`, and Gateway `agent` messages carry plain source metadata without HMAC, token, signature, nonce, timestamp, or replay verification.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be a Gateway-managed signed provenance contract with replay and key lifecycle rules rather than model-supplied secrets in tool parameters.
