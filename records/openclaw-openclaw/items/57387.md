## Summary

Keep open. Current main and v2026.6.8 have provenance envelopes, config-based A2A visibility, and broad authenticated `sessions.send`, but no Gateway/session HMAC, signature, nonce, replay check, verifier, or key lifecycle for `sessions_send`; the IdentyClaw/HOLA workaround and scoped-token/federation issues are useful related input, not a core replacement.

## What This Changes

Request HMAC or token-based mutual authentication so receiving sessions can cryptographically verify the claimed sender of inter-agent `sessions_send` payloads.

## Best Possible Solution

Design one Gateway-owned inter-session trust contract that composes message provenance, sender authorization, replay protection, key issuance/rotation/revocation, and scoped-token/federation work before implementation.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 expose provenance and config allowlists but no cryptographic sender verification or replay-proof payload for `sessions_send`. I did not run a live exploit reproduction.

## Solution Assessment

No implementation is proposed here, and the current provenance-only behavior is not the full requested security property. The best path needs maintainer security/product design before a fix PR.
