## Summary

Keep open. Current main has gateway/tool authorization, `tools.agentToAgent` allowlist checks, inter-session prompt marking, and plain provenance metadata, but it still lacks HMAC/token/asymmetric signing or any Gateway verification that the claimed `sessions_send` sender actually originated the message. The production-context comment describing an external HMAC gateway mitigation supports that this remains a real core security design gap, not cleanup-close material.

## What This Changes

The issue requests HMAC or token-based mutual authentication for inter-agent `sessions_send` messages so receiving sessions can verify the claimed sender session.

## Best Possible Solution

Add a core Gateway-managed, non-forgeable inter-session provenance mechanism: sign or otherwise bind sender session, target session, message/body hash, idempotency key, timestamp/nonce, and routing context, then verify it before accepting or displaying `inter_session` provenance. Keep `tools.agentToAgent`, session visibility, and hooks as authorization/defense-in-depth around the new proof rather than treating them as the proof itself.

## Reproduction Assessment

Yes for the missing protection. A static current-main check is high-confidence: `sessions_send` and Gateway schemas expose only plain message/provenance fields, and existing tests assert plain `inter_session` metadata rather than any HMAC/token verification path.

## Solution Assessment

Unclear as an implementation choice because there is no patch yet. HMAC tokens are a reasonable minimum, but the safest repository direction is a Gateway-owned authenticated provenance design with replay protection and backward-compatible protocol handling.
