## Summary

Keep open. Current `main` still uses plaintext inter-session provenance plus visibility/A2A gates; it does not implement HMAC/token signing, nonce/timestamp replay protection, or verification of the claimed sender session. The KinthAI comment is useful supporting context because it describes an external HMAC/gateway mitigation for the same attack surface, but that mitigation has not landed in OpenClaw core.

## What This Changes

The issue requests HMAC or token-based mutual authentication for `sessions_send` provenance in inter-agent messaging.

## Best Possible Solution

Add a Gateway-managed signed provenance layer for inter-session messages with replay protection, protocol/schema updates, policy/config alignment, docs, and regression coverage while keeping existing visibility and A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_send`, `runAgentStep`, `InputProvenance`, and the Gateway `agent` method carry plain source metadata with no auth material or verification step.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be Gateway-managed signed provenance with replay and key lifecycle rules rather than a caller-supplied optional tool parameter.
