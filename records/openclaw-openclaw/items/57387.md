## Summary

Keep open. Current main still carries inter-session provenance as plain metadata and routing context rather than a non-forgeable, replay-protected sender-authentication contract; the production KinthAI comment reinforces that this is a real multi-agent security concern, but the fix needs security/product design before implementation.

## What This Changes

Request HMAC or token-based mutual authentication so `sessions_send` recipients can verify that messages really came from the claimed sending agent/session.

## Best Possible Solution

Design Gateway-managed signed inter-session provenance with replay protection, key lifecycle, configurable enforcement policy, protocol/schema updates, docs, and regression coverage while retaining current visibility and A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_send`, A2A follow-ups, `InputProvenance`, and Gateway `agent` messages carry plain source metadata without HMAC, token, signature, nonce, timestamp, or replay verification.

## Solution Assessment

Unclear at implementation-detail level. HMAC or token auth is plausible, but the maintainable fix should be a Gateway-managed signed provenance contract with replay and key lifecycle rules rather than model-supplied secrets in tool parameters.
