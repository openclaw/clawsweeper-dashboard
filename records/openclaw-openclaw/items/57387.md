## Summary

Keep open. Current main still marks `sessions_send` provenance as plain inter-session metadata and gates access with visibility/A2A policy, but it does not add non-forgeable sender proof, replay protection, or verification of the claimed source session; the KinthAI comment is useful supporting evidence for the same attack surface, not an OpenClaw implementation.

## What This Changes

Request adding per-session HMAC/token-based mutual authentication for `sessions_send` so receivers can verify the claimed sender session.

## Best Possible Solution

Design and implement Gateway-managed signed inter-session provenance with replay protection, protocol/schema updates, policy/config alignment, docs, and regression coverage while preserving existing visibility/A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_send`, A2A follow-ups, `InputProvenance`, and the Gateway `agent` method carry plain source metadata without auth material or a verification step.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable solution should be Gateway-managed signed provenance with replay and key lifecycle rules rather than an optional caller-supplied tool parameter.
