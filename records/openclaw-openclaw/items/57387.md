## Summary

Keep open. Current main has session visibility gates and model-visible inter-session provenance, but it still does not provide HMAC/token/signature verification, nonce/timestamp replay material, or non-forgeable proof that the claimed source session actually originated a `sessions_send` message; the KinthAI comment adds concrete production context for why the gap matters.

## What This Changes

Request HMAC or token-based mutual authentication for inter-agent `sessions_send` messages so receivers can verify the claimed sender session.

## Best Possible Solution

Design Gateway-managed signed inter-session provenance with replay protection, key lifecycle, protocol/schema updates, policy/config alignment, docs, and regression coverage while preserving existing visibility/A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_send`, A2A follow-ups, `InputProvenance`, and Gateway `agent` handling carry plain source metadata without HMAC/token/signature verification or replay material.

## Solution Assessment

Unclear at implementation-detail level. HMAC or token auth is plausible, but the maintainable fix should be a Gateway-managed signed provenance contract with replay and key lifecycle rules rather than model-supplied secrets in tool parameters.
