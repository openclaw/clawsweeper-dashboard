## Summary

Keep open. Current main and v2026.6.8 still have metadata-only inter-session provenance plus gateway/operator authorization, but no Gateway-owned HMAC/token/signature, nonce, timestamp, replay check, or key lifecycle for `sessions_send`; the IdentyClaw ClawHub plugin and HOLA envelope workaround are useful defense-in-depth, not a core replacement.

## What This Changes

Requests per-session HMAC or token-based mutual authentication for inter-agent `sessions_send` messages, ideally configurable by agent role.

## Best Possible Solution

Design one Gateway-owned signed-provenance contract for `sessions_send`, with replay protection and target policy integrated with existing session visibility, A2A allowlists, and scoped device-token work.

## Reproduction Assessment

Yes. Source inspection shows `sessions_send` and Gateway `sessions.send` accept metadata-only provenance and no HMAC, token, signature, nonce, timestamp, or replay verification field; I did not run a live exploit because this is a design-sensitive security feature gap.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be a Gateway-managed signed-provenance contract with replay and key lifecycle rules rather than model-supplied secrets in tool parameters.
