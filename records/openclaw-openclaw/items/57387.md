## Summary

Keep open. Current main still relies on gateway/operator auth, session visibility, and A2A allowlists plus metadata-only provenance; the linked IdentyClaw ClawHub plugin/skill is a useful workaround, but it does not replace a Gateway-owned replay-protected sender-authentication contract for inter-agent sessions_send.

## What This Changes

Request Gateway-managed HMAC or token-based mutual authentication for inter-agent sessions_send messages, ideally enforceable per agent.

## Best Possible Solution

Design a Gateway-owned signed provenance/auth contract for sessions_send with replay protection and target policy, integrate it with existing visibility/A2A and device-token work, and document plugin envelope workarounds as optional defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: sessions_send and Gateway sessions.send accept metadata-only provenance and no HMAC, token, signature, nonce, timestamp, or replay verification field.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the best fix should be a Gateway-managed signed-provenance contract with replay and key lifecycle rules rather than model-supplied secrets in tool parameters.
