## Summary

Keep open. Current main and stable v2026.6.10 provide Gateway auth, `operator.write` scope, session visibility/A2A allowlists, and inter-session provenance, but not the requested per-message HMAC/token sender verification, replay protection, verifier, or key lifecycle; the IdentyClaw ClawHub plugin/skill is useful workaround evidence, not a core close reason.

## What This Changes

Issue #57387 requests HMAC/token-based mutual authentication for inter-agent `sessions_send` so receivers can verify the claimed sender session, preferably with per-agent configurability.

## Best Possible Solution

Define one maintainer-approved inter-session trust contract that composes provenance, sender authorization, replay protection, key lifecycle, scoped tokens, plugin workarounds, docs, and upgrade-safe defaults before implementation.

## Reproduction Assessment

Yes, source-level: current main and stable v2026.6.10 expose provenance and allowlists but no cryptographic sender verification or replay-proof `sessions_send` payload; no live exploit reproduction was run.

## Solution Assessment

Unclear; HMAC is plausible, but maintainers need to choose the security model, token/key lifecycle, sender graph semantics, and compatibility behavior before a fix PR is safe.
