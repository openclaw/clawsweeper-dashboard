## Summary

Keep open. Current main and the latest shipped tag still use visibility/allowlist gates plus metadata-only inter-session provenance; they do not provide a non-forgeable, replay-protected sender-authentication contract for sessions_send.

## What This Changes

The issue requests HMAC or token-based mutual authentication for inter-agent sessions_send messages so receivers can verify sender provenance.

## Best Possible Solution

Design Gateway-managed signed inter-session provenance with replay protection, key lifecycle, configurable enforcement policy, protocol/schema updates, docs, and regression coverage while retaining current visibility and A2A gates as defense in depth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: sessions_send, A2A follow-ups, InputProvenance, and Gateway schemas carry plain source metadata without HMAC, token, signature, nonce, timestamp, or replay verification.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be a Gateway-managed signed provenance contract with replay and key lifecycle rules rather than model-supplied secrets in tool parameters.
