## Summary

Keep open. Current main and latest v2026.6.10 still provide gateway auth, scoped operator methods, `sessions_send` visibility policy, and inter-session provenance, but not the requested cryptographic sender verification, replay protection, verifier, or key lifecycle for inter-agent messages.

## What This Changes

The issue requests HMAC or token-based mutual authentication for inter-agent `sessions_send` payloads.

## Best Possible Solution

Define one upgrade-safe inter-session trust contract that composes message provenance, sender authorization, replay protection, key issuance/rotation/revocation, scoped tokens, and plugin/ClawHub envelope workarounds before implementation.

## Reproduction Assessment

Yes, source-level only: current main and v2026.6.10 expose provenance and config allowlists but no cryptographic sender verification or replay-proof payload for `sessions_send`. I did not run a live exploit reproduction.

## Solution Assessment

Unclear; HMAC is plausible, but maintainers need to choose the security model, token lifecycle, sender graph semantics, and compatibility behavior before a fix PR is safe.
