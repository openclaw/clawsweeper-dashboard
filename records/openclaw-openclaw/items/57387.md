## Summary

Keep open. Current main has useful defense-in-depth for inter-session sends, but it still does not provide cryptographic sender proof for `sessions_send`, and the production-context comment reinforces that the remaining gap is security-sensitive.

## What This Changes

Add HMAC or token-based mutual authentication for inter-agent `sessions_send` messages so receivers can verify claimed sender sessions.

## Best Possible Solution

Add a Gateway-managed signed provenance/auth layer for inter-session messages while keeping `tools.sessions.visibility`, `tools.agentToAgent`, hooks, and HTTP tool denial as defense-in-depth.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction of the missing capability: current main has no auth fields in the `sessions_send` schema, `InputProvenance` contract, or Gateway verification path; I did not run a live exploit because this review is read-only.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token signing is a plausible direction, but the final design should be a core Gateway security/protocol decision with replay and compatibility rules, not a narrow `sessions_send` parameter patch.
