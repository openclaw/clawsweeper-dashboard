## Summary

Keep open. Current main has useful inter-session prompt marking, session visibility checks, and `tools.agentToAgent` authorization gates, but it still does not provide cryptographic sender proof for `sessions_send`; the production-context comment describes an external HMAC gateway workaround, which reinforces that the core capability remains missing.

## What This Changes

Request HMAC/token-based mutual authentication for inter-agent `sessions_send` traffic, ideally configurable per agent or session.

## Best Possible Solution

Add a Gateway-managed signed provenance/auth layer for inter-session messages while preserving `tools.sessions.visibility` and `tools.agentToAgent` as authorization gates.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction of the missing capability: current main has no auth fields in the `sessions_send` schema or `InputProvenance` contract and no verification step before nested dispatch.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token signing is a plausible direction, but the safest solution should be designed as a core Gateway security/protocol change rather than a narrow `sessions_send` parameter patch.
