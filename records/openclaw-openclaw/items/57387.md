## Summary

Keep open. Current main and the latest release still provide session visibility, A2A allowlists, gateway/operator auth, and metadata-only provenance, but no Gateway-owned HMAC/token/signature, nonce, timestamp, or replay verification for inter-agent `sessions_send`; the IdentyClaw ClawHub plugin/skill is useful defense in depth, not a core replacement.

## What This Changes

Request HMAC/token-based mutual authentication so receiving sessions can verify that inter-agent `sessions_send` messages came from the claimed sender session.

## Best Possible Solution

Design one Gateway-owned signed-provenance contract for `sessions_send`, with replay protection and target policy, integrated with existing session visibility, A2A allowlists, and scoped device-token work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_send` and Gateway `sessions.send` accept metadata-only provenance and no HMAC, token, signature, nonce, timestamp, or replay verification field; I did not run a live exploit because this is a design-sensitive feature gap.

## Solution Assessment

Unclear at implementation-detail level. HMAC/token auth is plausible, but the maintainable fix should be a Gateway-managed signed-provenance contract with replay and key lifecycle rules rather than model-supplied secrets in tool parameters.
