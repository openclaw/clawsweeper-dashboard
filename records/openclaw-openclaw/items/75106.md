## Summary

Keep open. Current main still has the reported long-running deep embedding probe path, and this PR is a relevant implementation candidate, but the patch is not merge-ready because the proposed budget does not cancel or avoid the inner 600-second batch timeout, verbose non-TTY progress remains hidden, and the changelog entry is missing required contributor credit.

## What This Changes

The PR changes the memory-core CLI deep status path to add an 8-second embedding-probe budget, render and document a distinct timeout state, expose `timedOut` on memory probe result types, add a probe-budget unit test, and update docs plus changelog.

## Best Possible Solution

Move the diagnostic budget into a status-specific or cancellable embedding probe path that does not create or leave behind the normal long batch timer, keep a distinct parseable timeout state for CLI/JSON output, use `log` for non-TTY progress visibility, and add the required changelog credit plus focused tests around the real manager timer behavior.

## Reproduction Assessment

Yes. #74544 gives user-level CLI steps, and source inspection gives a focused reproduction path: a cache-miss `memory status --deep` awaits the live embedding batch ping under the local 600-second batch timeout.

## Solution Assessment

No. A top-level `Promise.race` around `manager.probeEmbeddingAvailability()` is not the best fix because the inner batch timeout continues running; a manager-level short-budget/cancellable probe or status-only probe path is safer and actually bounds command lifetime.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid leaving the inner batch timeout alive:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: This races the public `manager.probeEmbeddingAvailability()` call, but on a cache miss that method has already entered `embedBatchWithRetry(["ping"])`, whose `withTimeout` creates the normal local 600s timer. Because this wrapper does not cancel or avoid that inner probe, the CLI can report the 8s timeout while the ref-counted inner timer still keeps the process open, so the command runtime is not actually bounded.
  - confidence: 0.92
- **[P2] Keep verbose non-TTY progress visible:** `extensions/memory-core/src/cli.runtime.ts:732`
  - body: When `--verbose` is set this chooses `fallback: "line"`, but `createCliProgress` only renders `line` on TTY streams and returns a no-op for non-TTY unless the fallback is `log`. As a result, `openclaw memory status --deep --verbose | cat` still suppresses the `Probing ...` labels during the diagnostic wait.
  - confidence: 0.9
- **[P2] Add the required changelog credit:** `CHANGELOG.md:9`
  - body: The repo changelog policy requires every added user-facing changelog entry to include at least one `Thanks @author` attribution. This new fix entry only has `Fixes #74544`, so it should credit the contributor before merge.
  - confidence: 0.97
