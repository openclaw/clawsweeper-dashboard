## Summary

Keep open. The PR targets a valid `memory status --deep` hang, but the proposed outer race likely leaves the underlying local batch probe's referenced 600s timeout alive, so it can still keep the CLI process open after the new timeout result is rendered.

## What This Changes

The PR adds an 8s `memory status --deep` embedding-probe budget, timeout status/type/docs, non-TTY progress fallback, tests, docs, and a changelog entry.

## Best Possible Solution

Move the diagnostic budget into a cancellable or event-loop-safe manager/provider probe path, preserve the current vector readiness split, render a distinct timeout state, and prove the real CLI process exits promptly.

## Reproduction Assessment

Yes for the source-level failure path: current main awaits `probeEmbeddingAvailability()`, which can enter `embedBatchWithRetry(["ping"])` under the 600s local batch timeout, and the linked issue includes matching 600s output. I did not run the reporter's Windows/local-provider setup in this read-only review.

## Solution Assessment

No. The direction is right, but this implementation races the manager probe from outside and does not cancel or neutralize the inner referenced batch timeout, so it is not the narrow reliable fix yet.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Cancel the inner embedding probe:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: This races an 8s timer against `manager.probeEmbeddingAvailability()`, but that method has already entered `embedBatchWithRetry(["ping"])`; the batch helper starts its own referenced 600s timeout for local providers. When this outer timeout wins, the inner probe and timer can keep the Node process alive, so callers may still wait for the original batch timeout even after seeing `Embeddings: timeout`. Put the diagnostic budget inside the probe/batch path, or make the underlying work abortable or otherwise unable to retain the event loop.
  - confidence: 0.88
