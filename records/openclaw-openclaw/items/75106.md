## Summary

Keep open. The PR targets a real `memory status --deep` hang and is correctly paired with open issue #74544, but the current patch should not merge as-is because the CLI-level budget can return while the inner 600-second manager batch timeout remains alive, verbose non-TTY progress still becomes a no-op, and the user-facing changelog entry lacks the required contributor credit.

## What This Changes

The PR adds an 8-second diagnostic budget and `timedOut` result for memory deep embedding probes, updates status rendering, docs, SDK type mirrors, tests, and the changelog.

## Best Possible Solution

Keep the contributor PR and linked issue open, but revise the fix so `memory status --deep` uses a short-budget manager/provider diagnostic probe that avoids or cancels the normal batch timeout, keeps non-TTY progress visible with `fallback: "log"`, preserves indexing's longer budget, renders and serializes a distinct timeout state, and carries focused tests, docs, and changelog credit.

## Reproduction Assessment

Yes. The related issue gives a user-level reproduction path, and source inspection gives the focused code path: deep memory status awaits `probeEmbeddingAvailability()`, whose cache-miss path runs `embedBatchWithRetry(["ping"])` under the local 600-second batch timeout.

## Solution Assessment

No. The proposed CLI-level `Promise.race` is not the best fix because it does not cancel or avoid the inner manager batch timeout; the safer maintainable solution belongs in a short-budget or cancellable diagnostic probe path closer to the manager/provider call.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid leaving the inner batch timeout alive:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: This races `manager.probeEmbeddingAvailability()` from the CLI, but on a cache miss that method has already entered `embedBatchWithRetry(["ping"])`, whose `withTimeout` creates the normal local 600-second timer. The command can therefore report the 8-second timeout while the process remains open until the inner probe settles; put the short budget inside a manager/provider probe path that avoids or cancels that timer.
  - confidence: 0.92
- **[P2] Keep verbose non-TTY progress visible:** `extensions/memory-core/src/cli.runtime.ts:732`
  - body: When `--verbose` is set this chooses `fallback: "line"`, but `createCliProgress` only renders `line` on TTY streams and returns a no-op for non-TTY unless the fallback is `log`. As a result, `openclaw memory status --deep --verbose | cat` still hides the probe labels during the diagnostic wait.
  - confidence: 0.9
- **[P2] Add the required changelog credit:** `CHANGELOG.md:9`
  - body: The added fix entry is user-facing and has an eligible contributor handle, but it does not include the required `Thanks @...` attribution. Add the contributor credit before merge.
  - confidence: 0.97
