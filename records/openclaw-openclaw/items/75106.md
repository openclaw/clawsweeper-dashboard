## Summary

Keep this PR open, but it is not merge-ready: the proposed budget races the outer await without cancelling the underlying embedding probe, so the diagnostic can still leave the provider request running and fail to be truly bounded.

## What This Changes

The PR adds an 8-second budget around `memory status --deep` embedding probes, a timeout result state, non-TTY progress fallback, SDK type updates, tests, docs, and a changelog entry.

## Best Possible Solution

Make the embedding readiness probe deadline-aware at the manager/provider layer, preserve current main's vector-store probing behavior, remove the changelog edit, and add real CLI proof from the patched command.

## Reproduction Assessment

No live current-main reproduction was run, but the source path is clear: deep status awaits `probeEmbeddingAvailability()`, which can call the local batch embedding path with a 600-second timeout. That makes the bug source-reproducible rather than fully reproduced here.

## Solution Assessment

No. The PR is pointed at the right symptom, but the best fix must cancel or deadline the underlying manager/provider operation, not only race the CLI await.
