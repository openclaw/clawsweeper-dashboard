## Summary

Keep open. The PR targets a valid diagnostic hang, but the proposed top-level budget does not actually bound the process lifetime and one non-TTY progress path remains silent, so it needs revision before merge.

## What This Changes

The PR adds an 8-second `memory status --deep` embedding-probe budget, a distinct timeout result/state, docs and changelog updates, and a focused probe-budget test.

## Best Possible Solution

Implement the diagnostic budget where the embedding probe is actually started, either through a short status-specific manager path or a cancellable provider probe, so `memory status --deep` cannot leave the normal 600-second batch timer alive. Keep indexing on the full batch timeout, render a distinct timeout state in text and JSON, preserve non-TTY progress with `log`, and update docs/changelog with contributor credit.

## Reproduction Assessment

Yes. The linked bug gives user-level steps, and source inspection gives the focused path: deep status awaits `probeEmbeddingAvailability()`, which calls `embedBatchWithRetry(["ping"])` and inherits the local 600-second batch timeout on cache miss.

## Solution Assessment

No. The current PR direction is useful, but the top-level `Promise.race` is not the best fix because it returns before the underlying manager probe and its ref-counted timeout are cancelled or avoided.
