## Summary

Keep open. The PR targets a valid memory CLI bug, but the current patch does not reliably bound the process because the slow provider probe is only raced, not cancelled, and it still lacks after-fix real behavior proof from a real `memory status --deep` run.

## What This Changes

The PR adds an 8-second `memory status --deep` embedding probe budget, timeout rendering/JSON/docs, non-TTY progress fallback wiring, tests, and a changelog entry.

## Best Possible Solution

Implement a status-only embedding probe that is truly bounded by cancelling or avoiding the long provider-init/batch path, keep the current readiness split, and prove it with real CLI output plus focused regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main's deep status path awaits `manager.probeEmbeddingAvailability()`, which can enter the local 600-second batch timeout on cache miss; I did not live-run the Windows provider setup.

## Solution Assessment

No. The direction is right, but the outer `Promise.race` and verbose fallback choice are not the narrow reliable fix because they can leave the old long-running probe path alive and silent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Cancel the probe work when the budget fires:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: `probeEmbeddingWithBudget()` returns after the budget, but it leaves `manager.probeEmbeddingAvailability()` running. On a cache miss that call can be in provider initialization or inside `embedBatchWithRetry(["ping"])`; close then waits for pending provider init, and the batch path's 600s timer is not cleared until the inner promise settles. The CLI can still stay alive until the old long timeout, so this needs a status-specific cancellable or short probe instead of only an outer race.
  - confidence: 0.91
- **[P2] Use log fallback for piped verbose runs:** `extensions/memory-core/src/cli.runtime.ts:732`
  - body: `createCliProgress()` only writes non-TTY progress when `fallback === "log"`. With `--verbose`, this sets `line`, which returns the no-op reporter for pipes and CI, so `openclaw memory status --deep --verbose | cat` can still show no `Probing...` labels while waiting. Keep non-TTY on `log` or choose the fallback from stream capability.
  - confidence: 0.94
