## Summary

Keep open. The PR targets a valid memory diagnostics bug, but the proposed outer race does not cancel the underlying embedding probe, so the reported 600-second local batch timer can still keep the CLI process alive after the new 8-second result is rendered; it also still needs real after-fix behavior proof.

## What This Changes

This PR adds an 8-second wrapper, timeout status/type/docs/tests, non-TTY progress fallback, and changelog entry for `memory status --deep` embedding probes.

## Best Possible Solution

Move the diagnostic budget into a cancellable status-only embedding probe or abortable provider call, keep the long indexing budget, render a distinct timeout state, and preserve current main's vector-store readiness split.

## Reproduction Assessment

Yes for the source-level bug: current main directly awaits the deep embedding probe, that probe can call `embedBatchWithRetry(["ping"])`, and the local batch path uses a 600-second timeout. I did not live-run the reporter's Windows/local-provider setup.

## Solution Assessment

No. The PR is in the right area, but an outer `Promise.race` is not enough because the underlying batch timeout/request can remain alive; the fix should cancel or avoid that underlying long-running operation when the status budget wins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Cancel the underlying embedding probe on timeout:** `extensions/memory-core/src/cli.runtime.ts:121`
  - body: `probeEmbeddingWithBudget()` only races the result of `manager.probeEmbeddingAvailability()`. On the reported local path that probe enters `embedBatchWithRetry()`, which installs the normal referenced 600s batch timeout; when this outer race wins, the original timer/request can still keep the CLI process alive until the batch timeout, so `memory status --deep` may still not return in the advertised 8s budget. Please make the status probe cancellable or move the short budget into the manager/provider path so the losing batch operation is actually stopped.
  - confidence: 0.9
- **[P2] Use a log fallback for non-TTY verbose probes:** `extensions/memory-core/src/cli.runtime.ts:734`
  - body: The new status progress fallback switches to `line` whenever `--verbose` is set, but `createCliProgress()` only emits output on non-TTY streams when the fallback is `log`; `line` returns the no-op reporter in CI, pipes, and diagnostic runners. That leaves `memory status --deep --verbose` silent in the same non-TTY scenario this PR is trying to fix, so choose `log` for non-TTY streams or avoid selecting `line` blindly from `opts.verbose`.
  - confidence: 0.86
