## Summary

Keep open. The PR targets a valid `memory status --deep` diagnostic hang, but the proposed CLI-level race does not actually bound the process because the inner embedding batch timeout can remain alive; it also leaves verbose non-TTY progress silent and misses required changelog credit.

## What This Changes

The branch adds an 8-second budget around memory-core deep embedding probes, exposes a `timedOut` probe result, renders a timeout state, adds a probe-budget test, and updates memory CLI docs, SDK types, and changelog.

## Best Possible Solution

Put the short diagnostic budget inside a manager/provider probe path that avoids or cancels the normal batch timeout, keep non-TTY status progress visible with `fallback: "log"`, preserve indexing's longer batch budget, and cover the real timer behavior plus timeout rendering in focused tests.

## Reproduction Assessment

Yes. The related issue gives user-level steps, and source inspection gives a high-confidence focused path: `memory status --deep` awaits a cache-miss embedding probe that runs `embedBatchWithRetry(["ping"])` under the local 600-second batch timeout.

## Solution Assessment

No. A top-level CLI `Promise.race` is not the best fix because it does not cancel or avoid the inner manager batch timeout; a manager-level short-budget or cancellable diagnostic probe is the safer maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid leaving the inner batch timeout alive:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: This races `manager.probeEmbeddingAvailability()` from the CLI, but on a cache miss that method has already entered `embedBatchWithRetry(["ping"])`, whose `withTimeout` creates the normal local 600-second timer. Since this does not cancel or avoid that inner probe, the CLI can report the 8-second timeout while the process remains open until the inner timer settles.
  - confidence: 0.92

  - body: When `--verbose` is set this chooses `fallback: "line"`, but `createCliProgress` only renders `line` on TTY streams and returns a no-op for non-TTY unless the fallback is `log`. As a result, `openclaw memory status --deep --verbose | cat` still hides the probe labels during the diagnostic wait.
  - confidence: 0.9

  - body: The repo changelog policy requires every added user-facing changelog entry to include at least one `Thanks @author` attribution. This new fix entry only has the issue reference, so it should credit the contributor before merge.
  - confidence: 0.97
