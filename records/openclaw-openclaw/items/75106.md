## Summary

Keep open. This is a plausible focused bug fix, but the proposed outer timeout can return while the inner embedding probe and its local 600s timer are still alive, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds an 8-second `memory status --deep` embedding probe budget, `timedOut` reporting, timeout rendering, non-TTY progress fallback, focused tests, docs, and a changelog entry.

## Best Possible Solution

Move the diagnostic budget into a cancellable or event-loop-safe manager/provider probe path, keep the current vector readiness split, render a distinct timeout state, and prove the real CLI exits promptly.

## Reproduction Assessment

Yes for the source-level failure path: current main awaits `probeEmbeddingAvailability()`, which can enter `embedBatchWithRetry(["ping"])` under the 600-second local batch timeout. I did not live-run the reporter's Windows/local-provider setup.

## Solution Assessment

No. The UX direction is right, but the PR wraps the probe from outside instead of cancelling or neutralizing the inner batch probe/timer, so it does not reliably bound the process yet.
