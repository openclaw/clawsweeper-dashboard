## Summary

Keep open. The PR targets a valid diagnostic hang from #74544, but the current head is not mergeable because the CLI-level race can return after 8 seconds while the underlying embedding batch timeout remains alive, verbose non-TTY progress is still suppressed, and the changelog entry misses required contributor credit.

## What This Changes

The branch adds an 8-second budget around memory-core `memory status --deep` embedding probes, renders a new timeout state, exposes `timedOut` in memory probe result types, and updates memory CLI docs, tests, and changelog.

## Best Possible Solution

Apply the short diagnostic budget inside a manager/provider probe path that avoids or cancels the normal batch timeout, keep non-TTY status progress visible with `fallback: "log"`, preserve indexing's longer budget, and keep the new timeout state documented and covered by tests that exercise the real timer behavior.

## Reproduction Assessment

Yes. The linked bug gives user-level steps, and current-main source inspection gives a focused reproduction path: `memory status --deep` awaits a cache-miss embedding probe that runs `embedBatchWithRetry(["ping"])` under the local 600-second batch timeout.

## Solution Assessment

No. The current PR direction is close, but a top-level CLI `Promise.race` is not the best fix because it does not cancel or avoid the inner manager batch timeout; a manager-level short-budget or cancellable diagnostic probe is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid leaving the inner batch timeout alive:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: This races `manager.probeEmbeddingAvailability()` from the CLI, but on a cache miss that method has already entered `embedBatchWithRetry(["ping"])`, whose `withTimeout` creates the normal local 600-second timer. Because the PR does not cancel or avoid that inner probe, the CLI can report the 8-second timeout while the ref-counted inner timer still keeps the process open, so the command runtime is not actually bounded.
  - confidence: 0.92
- **[P2] Keep verbose non-TTY progress visible:** `extensions/memory-core/src/cli.runtime.ts:732`
  - body: When `--verbose` is set this chooses `fallback: "line"`, but `createCliProgress` only renders `line` on TTY streams and returns a no-op for non-TTY unless the fallback is `log`. As a result, `openclaw memory status --deep --verbose | cat` still suppresses the `Probing ...` labels during the diagnostic wait.
  - confidence: 0.9
- **[P2] Add the required changelog credit:** `CHANGELOG.md:9`
  - body: The repo changelog policy requires every added user-facing changelog entry to include at least one `Thanks @author` attribution. This new fix entry only has `Fixes #74544`, so it should credit the contributor before merge.
  - confidence: 0.97
