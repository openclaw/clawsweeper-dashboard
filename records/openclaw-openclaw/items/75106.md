## Summary

Keep open. Current main still has the reported deep-status hang path, and this PR is useful but not merge-ready because the proposed CLI race can return while the underlying 600s embedding batch timeout remains alive, verbose non-TTY progress still goes quiet, and the changelog entry is missing contributor credit.

## What This Changes

The PR adds an 8-second budgeted embedding probe for `memory status --deep`, timeout rendering and JSON/type docs, focused tests, docs, and a changelog entry.

## Best Possible Solution

Put the short diagnostic budget in a manager/provider probe path that avoids or cancels the normal batch timeout, keep non-TTY status progress on `log`, preserve indexing's longer batch budget, and credit the contributor in the changelog.

## Reproduction Assessment

Yes. The linked bug report gives user-level steps, and current main shows `memory status --deep` awaiting an embedding probe that can run a batch ping through the 600s local timeout while non-TTY progress is a no-op.

## Solution Assessment

No. A top-level CLI `Promise.race` is not the best fix because it reports a timeout without stopping the inner manager batch timeout; a manager-level short-budget or cancellable diagnostic probe is the safer path.
