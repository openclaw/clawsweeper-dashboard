## Summary

Keep open. This is a maintainer-authored issue and current main still lacks the requested `tools.invoke` Gateway RPC: the SDK intentionally throws unsupported, the Gateway protocol only defines `tools.catalog` and `tools.effective`, and the existing direct invocation surface is HTTP-only. The remaining work crosses Gateway protocol, tool policy, approval, and SDK contract boundaries, so it needs maintainer/API review rather than conservative cleanup.

## What This Changes

Add an SDK-facing Gateway RPC method named `tools.invoke` that invokes tools through the normal policy and approval pipeline.

## Best Possible Solution

Add an additive, protocol-defined `tools.invoke` RPC with TypeBox schemas, exported protocol types, method-list and scope entries, a server handler that reuses the existing tool resolution and before-tool-call/plugin approval pipeline, typed refusal/approval-needed results, SDK wiring for `oc.tools.invoke()`, public docs, and targeted Gateway plus SDK tests. Preserve the existing HTTP `/tools/invoke` behavior unless maintainers explicitly choose to align both surfaces.
