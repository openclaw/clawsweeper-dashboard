## Summary

Keep open: the PR is aimed at a real doctor false-positive, but the current patch can suppress a valid root/default-account warning for account-capable non-Telegram channels and still lacks real `openclaw doctor` proof.

## What This Changes

The branch changes the shared doctor empty-allowlist scanner to skip a top-level `groupAllowFrom` warning when explicit enabled accounts have allowlists, threads a skip flag into the warning helper, and adds scanner regression tests.

## Best Possible Solution

Keep the intended false-positive fix, but make the suppression use channel/account semantics shared with account helpers so root/default account warnings are preserved unless the top-level record is proven to be only a fallback parent.

## Reproduction Assessment

Yes. Current main checks the top-level channel config before account records, and the PR body plus added tests show how that produces the reported false parent warning; I did not run tests because this review is read-only.

## Solution Assessment

No. The fix direction is right, but the implementation is too narrow because it encodes Telegram-only implicit-default detection inside a shared scanner that also serves sibling account-capable channels.
