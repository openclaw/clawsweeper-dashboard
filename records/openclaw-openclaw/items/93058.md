## Summary

Keep open: the latest head addresses the previous `allowFrom` fallback gap, but the suppression is still too broad because it treats any `accounts` map as proof that the top-level channel record is not a real account. That can hide a real top-level/default-account doctor warning when Telegram top-level credentials coexist with named accounts.

## What This Changes

The PR changes the shared doctor empty-allowlist scanner to skip the top-level group allowlist warning when account records appear covered by per-account allowlists, and adds regression tests for `groupAllowFrom` and `allowFrom` coverage.

## Best Possible Solution

Land one canonical doctor scanner fix that suppresses only parent-fallback false positives, uses enabled/runtime account semantics, and preserves warnings for active top-level default accounts.

## Reproduction Assessment

Yes. Current `main` source reproduces the original false parent warning path, and the remaining blocker is source-reproducible from the new `hasAccounts` branch plus Telegram's implicit default-account contract.

## Solution Assessment

No. The direction is useful, but the best fix must distinguish fallback-only parent config from active top-level/default accounts and must evaluate enabled/runtime account records rather than raw `accounts` entries.
