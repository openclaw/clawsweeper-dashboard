## What This Changes

The branch adds a 30-second command-queue starvation promotion constant, dequeue-time effective-priority selection, and a regression test for aged background work draining before fresh foreground work.

## Best Possible Solution

Choose one queue-policy owner, then land a sequence-stable starvation guard with current-head contention proof or fold this regression into the broader scheduler branch.

## Reproduction Assessment

Yes at source level: current main inserts by static priority and drains the first queued entry, so a continuous stream of foreground entries can keep older background work behind. I did not run a live contention scenario in this read-only review.

## Solution Assessment

Unclear as a standalone merge path: the guard is a plausible narrow fix, but the threshold and precedence tradeoff overlap an open scheduler branch and need maintainer policy choice plus real contention proof.
