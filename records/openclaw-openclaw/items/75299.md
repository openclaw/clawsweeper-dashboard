## What This Changes

This PR exports a 30-second starvation threshold, changes command-lane dequeue to scan for aged entries promoted above static priorities, and adds command-queue regression coverage.

## Best Possible Solution

Land a maintainer-approved fairness policy that bounds background starvation while preserving foreground/user precedence by default, or explicitly document and test the priority inversion if maintainers choose it.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the primitive-level guard is plausible, but unconditional promotion above foreground is not clearly the best policy for user/manual session work.
