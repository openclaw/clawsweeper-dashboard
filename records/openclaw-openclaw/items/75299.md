## Summary

Keep this PR open for human follow-up: the starvation problem is real and not fixed on current main, but the proposed comparator still lets aged foreground work outrank older low-priority work, so the guard does not guarantee eventual dequeue.

## What This Changes

The PR adds a 30-second starvation promotion constant, dequeue-time queue scanning, a starvation regression test, and a changelog entry for command-queue priority scheduling.

## Best Possible Solution

Use one canonical queue-aging comparator that makes sufficiently old lower-priority work eventually win, preserves existing priority/FIFO guarantees, and coordinates with the broader priority-queue follow-up.

## Reproduction Assessment

Yes, at source level: with an aged background entry and an aged foreground backlog, the proposed effective priority keeps aged foreground higher, so lower-priority work can still be skipped. I did not run tests because this review is read-only and the diff makes the path clear.

## Solution Assessment

No, not as-is. Dequeue-time aging is the right area, but the comparator should prefer aged or oldest work before static priority once the starvation threshold is crossed, and this PR still needs real behavior proof beyond unit tests.
