## Summary

Keep open: the starvation concern is valid on current main, but this PR does not yet guarantee starvation freedom and is stale against current queue diagnostics. It also still needs real runtime behavior proof beyond focused tests before merge.

## What This Changes

The PR adds a 30-second starvation promotion path to command-queue dequeue selection, with a focused regression test and changelog entry.

## Best Possible Solution

Implement dequeue-time aging so old lower-priority work eventually wins while preserving current queue diagnostics/tests, then coordinate that result with https://github.com/openclaw/openclaw/pull/80199.

## Reproduction Assessment

Yes, at source level: current main inserts by static priority and drains the head, while embedded user/manual work is foreground and maintenance triggers are background. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No, not as-is. Dequeue-time aging is the right surface, but the current comparator still lets aged foreground work outrank aged background work indefinitely and the branch must preserve current diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make aged low-priority work eventually win:** `src/process/command-queue.ts:262`
  - body: Adding 1_000 to the existing priority still orders aged foreground work above aged background work. In a sustained foreground backlog where those entries also cross the threshold, `pickNextIndex` can keep selecting foreground work while the background entry remains queued, so the guard does not deliver the claimed starvation protection.
  - confidence: 0.88

Rebase the starvation scan on top of the current diagnostic fields and coverage.
