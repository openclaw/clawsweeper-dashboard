## Summary

Keep open: the PR targets a real queue-starvation gap, but the proposed aging comparator still lets aged foreground backlog outrank older aged background work, and the supplied proof shows build/startup rather than the changed contention behavior.

## What This Changes

Adds a 30-second starvation-promotion threshold to command-lane dequeue selection and a regression test for aged background work beating fresh foreground work.

## Best Possible Solution

Define one canonical queue-aging policy where aged entries form a tier above non-aged entries and drain by enqueue age or FIFO, then reconcile that policy with the open scheduler PR before merge.

## Reproduction Assessment

Yes, at source level. On the PR head, aged background work computes to effective priority 999 while aged foreground work computes to 1001, so an older background entry can still lose behind any aged foreground backlog.

## Solution Assessment

No. The maintainable fix is to make starvation a separate tier above all non-aged work and order starved entries by enqueue age or FIFO, or to fold that policy into the broader scheduler PR.
