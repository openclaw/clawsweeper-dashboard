## Summary

Keep open. Current main still lacks priority-aware command queue scheduling, and the PR appears to be a relevant implementation candidate; the remaining question is maintainer acceptance of the shared scheduling API and exact-head validation, not conservative cleanup.

## What This Changes

The PR adds CommandPriority Low/Normal/High, in-place priority selection with starvation promotion, an optional enqueue priority option, and command-queue tests for ordering, starvation, wait callbacks, and timeouts.

## Best Possible Solution

Have maintainers decide whether this priority model belongs in the shared process command queue, then land this PR or an equivalent narrow patch after exact-head checks are clean and the targeted queue test coverage remains in place.

## Reproduction Assessment

Yes. Current main reproduces the missing capability by inspection: drainLane uses FIFO shift() and CommandQueueEnqueueOptions has no priority field; the PR’s added command-queue tests are the focused verification path.

## Solution Assessment

Yes for the implementation shape. In-place selection is the narrowest maintainable approach I found because it preserves enqueue timestamps, lane serialization, timeout accounting, and wait-callback timing while avoiding caller-specific lane proliferation; the API semantics still need maintainer approval.
