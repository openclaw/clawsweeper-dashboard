## Summary

Keep open: current main still lacks a starvation guard, but this PR is not merge-ready because the fixed priority offset can still starve aged lower-priority work when higher-priority backlog also ages, and the supplied proof is test-only.

## What This Changes

The PR adds a 30-second starvation-promotion constant, scans the command queue at dequeue time, adds one focused regression test, and adds a changelog entry.

## Best Possible Solution

Choose one queue-aging implementation that preserves current priority and diagnostics behavior, makes aged low-priority work eventually win, and includes real runtime proof before merge.

## Reproduction Assessment

Yes, at source level. Current main inserts foreground entries ahead of background entries and drains the head, and this PR’s comparator still lets aged foreground entries outrank aged background entries.

## Solution Assessment

No, not as-is. Dequeue-time aging is the right surface, but a fixed priority offset does not guarantee starvation freedom and the branch needs reconciliation with current queue diagnostics and the overlapping queue-priority work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make aged low-priority work outrank aged foreground backlog:** `src/process/command-queue.ts:262`
  - body: Adding 1_000 to the existing priority still leaves aged foreground entries above aged background entries. In a sustained foreground backlog where those entries also cross the threshold, `pickNextIndex` can keep selecting foreground work while the older background entry remains queued, so the guard does not guarantee the starvation protection it claims. Compare promoted/oldest status before static priority, or otherwise make aged low-priority work eventually win.
  - confidence: 0.86
