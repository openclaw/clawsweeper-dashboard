## Summary

Keep open: the starvation guard addresses a real primitive-level fairness gap, but it changes shipped foreground-first scheduling by letting aged background work outrank fresh user/manual work after 30 seconds, so this needs maintainer scheduling-policy approval or a safer default before merge.

## What This Changes

The branch adds a 30-second starvation-promotion threshold to the process command queue, switches dequeue selection from `shift()` to `pickNextIndex()`/`splice()`, exports the threshold, and adds focused queue tests.

## Best Possible Solution

Preserve shipped foreground/user precedence by default and add a maintainer-approved fairness mechanism that prevents indefinite lower-priority starvation without surprising interactive sessions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the queue primitive from source: current main drains the first pre-sorted entry with no age-based promotion, so continuous foreground insertion can keep background entries waiting; no live gateway/session reproduction was inspected.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the PR is a plausible primitive-level guard, but unconditional promotion above foreground is not clearly the best fix because current shipped behavior deliberately classifies user/manual runs as foreground and background triggers as background.
