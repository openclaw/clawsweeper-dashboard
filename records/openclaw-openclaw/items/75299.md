## Summary

Keep open: the starvation guard addresses a real queue fairness gap, but the current patch lets aged background or maintenance work outrank fresh foreground user/manual work after 30 seconds. That changes the shipped foreground-first scheduling behavior and needs either a maintainer scheduling-policy decision or a revision that preserves foreground priority.

## What This Changes

The PR adds a 30-second starvation-promotion threshold to command-lane dequeue selection and tests aged background work running before fresh foreground work.

## Best Possible Solution
