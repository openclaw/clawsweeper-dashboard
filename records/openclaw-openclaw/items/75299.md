## Summary

Keep open: the starvation-guard idea is useful, but the patch is not merge-ready because promoted entries still carry static priority, so an aged foreground backlog can continue to outrank older background work; the supplied proof shows build/startup, not contention behavior.

## What This Changes

The PR adds a 30-second starvation promotion constant, scans the command queue at dequeue time, and adds a regression test for an aged background task beating a fresh foreground task.

## Best Possible Solution

Implement starvation as an explicit aged tier where any starved entry beats non-starved entries and aged entries drain by oldest enqueue/FIFO, then reconcile that policy with https://github.com/openclaw/openclaw/pull/80199 before merge.

## Reproduction Assessment

Yes, at source level. The PR ranks starved entries as static priority plus 1000, so aged foreground work still outranks older aged background work; I did not run a live gateway contention repro in this read-only review.

## Solution Assessment

No, not as written. Dequeue-time aging is the right surface, but promoted entries need a separate fairness tier ordered by age or FIFO, or this should be folded into the broader scheduler policy branch.
