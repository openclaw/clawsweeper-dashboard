## Summary

Keep open: current main has priority insertion from the merged manual-session priority work, but it still lacks this PR’s starvation guard. The proposed patch needs changes before merge because adding a fixed offset to raw priority can still starve background work when a foreground backlog has also aged, and the supplied proof is focused tests/typecheck only.

## What This Changes

The PR adds a 30-second starvation promotion check at command-queue dequeue time, exports the threshold, adds a regression test, and updates the changelog.

## Best Possible Solution

Adopt one queue-aging comparator that guarantees old lower-priority work eventually wins, preserve legacy-priority normalization, cover aged high-priority backlog regression, and coordinate with the existing priority-queue follow-up.

## Reproduction Assessment

Yes, at source level. A queue with a background entry and a foreground backlog that also ages past the threshold will keep selecting aged foreground entries because their effective priority remains higher.

## Solution Assessment

No, not as-is. Dequeue-time aging is the right area, but the comparator should make sufficiently old entries win before static priority, and the PR still needs real behavior proof beyond tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make aging outrank aged foreground backlogs:** `src/process/command-queue.ts:279`
  - body: Adding 1000 to the existing priority leaves aged background at 999 and aged foreground at 1001. If the foreground backlog also crosses the threshold, pickNextIndex can keep selecting foreground entries while the background entry remains queued, so the starvation guard does not guarantee the behavior it claims. Compare aged/oldest status before static priority or otherwise make aged low-priority work eventually win.
  - confidence: 0.86
