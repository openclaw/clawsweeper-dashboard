## Summary

Keep open. Current main still lacks priority-aware queue scheduling, and the PR is a useful direction, but the exact head is not merge-ready because preserved queue entries created before the new `priority` field can compare as `undefined` and prevent a later high-priority entry from jumping ahead.

## What This Changes

This PR adds `CommandPriority` Low/Normal/High, priority-aware in-place command queue selection with starvation promotion, an optional enqueue priority option, and focused queue tests.

## Best Possible Solution

Normalize missing or invalid entry priorities to `CommandPriority.Normal`, cover preserved legacy singleton entries in tests, then land once maintainer API review and exact-head checks are clean.

## Reproduction Assessment

Yes. Current main reproduces the missing capability by source inspection, and the patch bug is reproducible by preserving a queued global singleton entry without `priority` before enqueueing a High entry behind it.

## Solution Assessment

No, not yet. The in-place scan is the right maintainable shape, but `effectivePriority()` must normalize legacy or malformed priority values before comparison.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Default legacy entries before comparing priority:** `src/process/command-queue.ts:206`
  - body: Queue state is preserved on `globalThis` across in-process restarts, so queued entries created by an older build can reach this code without a `priority` field. Returning `undefined` makes `pickNextIndex()` compare against `NaN`, so a new high-priority item behind a legacy normal item cannot jump ahead as promised; treat missing or invalid priorities as `CommandPriority.Normal` and add a regression test.
