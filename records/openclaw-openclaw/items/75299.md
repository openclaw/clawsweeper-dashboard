## Summary

Keep open: the starvation guard is a meaningful queue improvement idea, but the current implementation silently lets aged background work outrank fresh foreground/manual user turns already shipped in v2026.6.1, so it needs maintainer scheduling-policy approval or a narrower fix before merge.

## What This Changes

The PR adds a 30-second command-queue starvation promotion that selects aged queued entries ahead of all standard priorities and adds focused command-queue tests.

## Best Possible Solution

Land a scheduling policy that prevents background starvation while preserving user/manual foreground precedence by default, with caller-level tests for session/global lane ordering.

## Reproduction Assessment

Yes. Source inspection shows the patch promotes any aged queue entry above foreground, while current main and v2026.6.1 classify user/manual work as foreground and maintenance-style triggers as background.

## Solution Assessment

No. The guard is a plausible starvation mitigation, but the unconditional MAX_SAFE_INTEGER promotion is too broad because it collapses the shipped foreground/background policy instead of preserving user/manual precedence or documenting an explicit policy change.
