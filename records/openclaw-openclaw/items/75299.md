## Summary

Keep open: the starvation-guard idea is focused and useful, but the patch is not merge-ready because promoted entries still retain static priority inside the aged tier, the supplied proof only shows build/startup/compiled code, and the same scheduler policy overlaps with another open queue-priority PR.

## What This Changes

The PR exports a 30-second starvation threshold, changes command-queue dequeue to scan by effective priority, and adds a regression test for an aged background item beating a fresh foreground item.

## Best Possible Solution

Define one canonical queue-aging policy where aged entries form a distinct tier above non-aged entries and drain by enqueue age or FIFO, then reconcile that policy with the open queue-priority PR before merge.

## Reproduction Assessment

Yes, at source level. With an older background entry and an aged foreground backlog, `effectivePriority()` ranks aged foreground as 1001 and aged background as 999, so the background entry can still remain behind higher-priority aged work.

## Solution Assessment

No, not as written. Dequeue-time aging is the right surface, but the aged tier needs to compare by wait age or FIFO rather than preserving static priority, and maintainers should reconcile the policy with the related scheduler PR.
