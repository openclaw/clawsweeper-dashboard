## Summary

Keep open: current main still lacks command-queue priority scheduling, and this PR is meaningful but not merge-ready because it lacks real behavior proof, has a preserved-entry priority normalization bug, and overlaps with the broader open implementation at https://github.com/openclaw/openclaw/pull/80199.

## What This Changes

The branch adds Low/Normal/High priority support to the in-process command queue, priority-based dequeueing with starvation promotion, an optional enqueue priority field, and focused queue tests.

## Best Possible Solution

Adopt one canonical priority-queue implementation that normalizes legacy entries, wires foreground/background callers where the user-facing delay occurs, updates docs, and closes the linked issue only after real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows FIFO `state.queue.shift()` and no `priority` option, and PR-branch source shows preserved entries can compare with an undefined priority.

## Solution Assessment

No, not as-is. The in-place scan is a plausible direction, but this branch should normalize missing or malformed priorities and be reconciled with the broader implementation at https://github.com/openclaw/openclaw/pull/80199 before merge.
