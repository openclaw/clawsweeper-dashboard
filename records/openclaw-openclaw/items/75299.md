## Summary

Keep open: current main and the latest release have priority-aware enqueueing but still drain with `queue.shift()`, so the starvation guard is not implemented yet. The branch is focused, but it needs current-head contention proof and a maintainer choice against the overlapping scheduler work before merge.

## What This Changes

The branch adds a 30-second starvation promotion constant, dequeue-time effective-priority selection, and a regression test for aged background work draining before fresh foreground work.

## Best Possible Solution

Land one queue-policy path that adds starvation promotion with a sequence tie-breaker, focused regression coverage, and maintainer-approved ordering semantics, either in this branch or folded into the overlapping scheduler branch.

## Reproduction Assessment

Yes at source level: current main inserts by static priority and drains the first queued entry, so a continuous stream of foreground entries can keep older background work behind. I did not run a live contention scenario in this read-only review.

## Solution Assessment

Unclear: the standalone guard is plausible, but it changes queue ordering policy and overlaps https://github.com/openclaw/openclaw/pull/80199. The safer path is to choose one owner, add the sequence tie-breaker, and require current-head contention proof.
