## Summary

Keep open. The queue-priority direction is useful, but the PR still mishandles preserved legacy queue entries: entries created before the new field can lack `priority`, which breaks the comparison path and lets high-priority work remain blocked behind them.

## What This Changes

Adds `CommandPriority`, priority-aware command-queue selection with starvation promotion, an optional enqueue priority option, and focused command-queue tests.

## Best Possible Solution

Normalize missing or nonfinite queue-entry priorities to `CommandPriority.Normal`, add a regression test for preserved pre-priority singleton entries, then merge after exact-head validation is clean.

## Reproduction Assessment

Yes. Current main reproduces the missing capability by source inspection, and the patch bug is reproducible by preserving a queued singleton entry without `priority` ahead of a new high-priority entry.

## Solution Assessment

No, not yet. The in-place scan is a maintainable shape, but it needs default priority normalization before the new scheduling contract is safe across restart-preserved queue state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Default legacy priorities before comparing:** `src/process/command-queue.ts:206`
  - body: Queue state is preserved on `globalThis` across in-process restarts, so entries created by an older build can reach this function without a `priority` field. Returning `undefined` makes `pickNextIndex()` compare against `NaN`, so a high-priority item behind a legacy normal item cannot jump ahead as promised; treat missing or nonfinite priorities as `CommandPriority.Normal` and add a regression test.
  - confidence: 0.87
