## Summary

Keep this PR open: the starvation guard is not implemented on current main, and the patch is coherent and proof-backed, but it still needs maintainer scheduling-policy review because a fixed 30-second promotion can let aged background or normal work overtake fresh foreground/user work.

## What This Changes

Adds a 30-second starvation-promotion threshold to the process command queue dequeue path, with a regression test for aged background work draining ahead of fresh foreground work.

## Best Possible Solution

Land one documented queue scheduling policy: either accept this 30-second promotion with docs and maintainer sign-off, or move starvation handling into the broader priority-policy path and close/update the overlapping branch.

## Reproduction Assessment

Yes. Current main inserts by static priority and drains with queue.shift(), so a steady stream of foreground entries can keep older background work behind them; the PR adds a focused test for the aged-background-before-fresh-foreground case.

## Solution Assessment

Unclear as the best fix. Time-based promotion is a plausible fairness guard, but the 30-second threshold changes foreground/background ordering and overlaps https://github.com/openclaw/openclaw/pull/80199, so maintainer policy approval and docs should come before merge.
