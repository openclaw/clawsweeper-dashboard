## Summary

Keep open. The priority-queue direction is useful, but the patch has a compatibility bug for queue entries preserved across in-process restarts from older code because legacy entries have no `priority` field and compare as `undefined`.

## What This Changes

This PR adds CommandPriority Low/Normal/High, priority-aware in-place command queue selection with starvation promotion, an optional enqueue priority option, and focused queue tests.

## Best Possible Solution

Normalize missing or invalid entry priorities to `CommandPriority.Normal`, add a regression test for preserved legacy queue entries, then merge only after maintainer API approval and exact-head checks are clean.

## Reproduction Assessment

Yes. Current main reproduces the missing capability by source inspection, and the patch bug is reproducible by a preserved global singleton queue entry created before the new `priority` field existed.

## Solution Assessment

No, not yet. The in-place scan is the right shape, but it needs legacy/default priority normalization before it is a safe queue contract change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Default legacy entries before comparing priority:** `src/process/command-queue.ts:206`
  - body: Queue state is preserved on `globalThis` across in-process restarts, so queued entries created by an older build can reach this code without a `priority` field. Returning `undefined` makes `pickNextIndex` compare against `NaN`, so a new high-priority item behind a legacy normal item cannot jump ahead as promised. Treat missing or invalid priorities as `CommandPriority.Normal` and add a regression test for a preserved legacy entry.
  - confidence: 0.87
