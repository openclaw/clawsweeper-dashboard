## Summary

Keep open: the starvation guard idea is useful, but this branch is not merge-ready because its aged-tier comparator still lets an aged foreground backlog beat older background work, and the supplied proof does not exercise sustained contention. The queue policy also overlaps the open scheduler work in https://github.com/openclaw/openclaw/pull/80199.

## What This Changes

The PR adds a 30-second command-queue starvation threshold, selects queued entries by effective priority at dequeue time, and adds a regression test for aged background work beating fresh foreground work.

## Best Possible Solution

Define one canonical queue-aging policy where aged entries form a distinct tier above non-aged entries and drain by enqueue age or FIFO, then reconcile it with the open queue-priority PR before merge.

## Reproduction Assessment

Yes, at source level. With the PR, an aged background entry has effective priority 999 while an aged foreground entry has 1001, so an older background entry can still remain behind a sustained aged foreground backlog.

## Solution Assessment

No. The narrow maintainable fix is to make starvation a separate aged tier ordered by enqueue age or FIFO, or to fold that policy into the canonical queue-priority work.
