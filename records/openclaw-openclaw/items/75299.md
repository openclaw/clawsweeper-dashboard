## Summary

Keep open. Current main and the latest release still have static priority ordering with head-only dequeue, so the starvation guard is not already implemented; the PR is focused, but merge should wait for maintainer queue-policy ownership and real behavior proof.

## What This Changes

The PR adds a 30-second starvation promotion threshold to the process command queue and a regression test for aged background work outranking fresh foreground work.

## Best Possible Solution

Land one owner-approved command-lane starvation policy, with current-head real contention proof, either as this focused guard or folded into the broader scheduler branch.

## Reproduction Assessment

Yes, at source level: current main inserts queued work by static priority and then drains the queue head, so a steady stream of foreground entries can keep older background work behind. I did not establish a live wall-clock reproduction in this read-only review.

## Solution Assessment

Unclear: the focused guard is mechanically reasonable, but the 30-second promoted tier is a scheduler policy decision and overlaps https://github.com/openclaw/openclaw/pull/80199. The maintainable path is one accepted queue policy with proof, not two competing implementations.
