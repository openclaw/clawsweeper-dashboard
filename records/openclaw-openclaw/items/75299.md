## Summary

Keep open. Current main still has FIFO-only command queue draining and no enqueue priority option, while this PR adds the missing priority scheduling surface with focused tests; I did not find a concrete source-level correctness or security blocker, but it still needs maintainer acceptance of the shared queue API and exact-head check follow-up.

## What This Changes

The PR adds CommandPriority Low/Normal/High, priority-aware in-place dequeue selection with starvation promotion, an optional enqueue priority option, and command-queue tests for ordering, starvation, wait callbacks, and timeouts.

## Best Possible Solution

Have maintainers decide whether the shared process command queue should expose priority scheduling, then land this PR or an equivalent narrow patch after exact-head checks are clean, keeping the behavior centralized in the queue rather than adding caller-specific lanes.

## Reproduction Assessment

Yes. Current main reproduces the missing feature by inspection: drainLane uses FIFO shift() and the enqueue options type has no priority field; the PR’s added tests are the focused verification path.

## Solution Assessment

Yes, for the implementation shape. The in-place scan is a narrow maintainable approach because it preserves enqueue timestamps, lane serialization, timeout accounting, and onWait queue-depth behavior while adding priority ordering; final API exposure still needs maintainer approval.
