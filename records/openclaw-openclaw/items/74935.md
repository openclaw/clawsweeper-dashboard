## Summary

Keep open. The PR fixes a real Control UI problem, but the latest head still introduces a browser availability regression by parsing and pretty-printing arbitrary tool output JSON while rendering collapsed tool rows.

## What This Changes

The PR adds a tool-specific sidebar content kind, renders tool input/output in the sidebar as expanded blocks with raw-text fallback, routes non-canvas tool summary clicks to the sidebar, and updates UI tests plus the changelog.

## Best Possible Solution

Land the dedicated escaped tool-sidebar renderer and click-to-sidebar behavior after bounding or lazily deferring JSON parsing, preferably following the existing `MAX_JSON_AUTOPARSE_CHARS` pattern and keeping large raw payloads available without render-loop work.

## Reproduction Assessment

Yes. Current main source reproduces the original problem by sending tool sidebar content through markdown JSON fences, and the PR regression is source-reproducible because collapsed tool-card render constructs parsed sidebar content before user interaction.

## Solution Assessment

No as submitted. The dedicated `tool` sidebar content kind is the right maintainable direction, but the implementation should cap or defer JSON parse/pretty-print work before merge.

## Review Findings
