## Summary

Keep open: this is a plausible fix for a real Control UI tool-card/sidebar bug, and it usefully overlaps the open tool-bubble readability report, but the PR still has a P2 render-path availability regression and lacks after-fix real UI proof.

## What This Changes

The PR adds a dedicated Control UI `tool` sidebar content kind and routes tool-card input/output into expanded side-panel blocks with tests and a changelog entry.

## Best Possible Solution

Keep the dedicated escaped tool-sidebar design, but cap or defer JSON parsing/pretty-printing and add large-payload regression coverage before merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because tool sidebar content is built as markdown with JSON fences and rendered through the JSON-collapse markdown path; the PR regression is source-reproducible because collapsed non-canvas rows construct sidebar content and parse full `outputText` during render.

## Solution Assessment

No, not as submitted. The dedicated `tool` sidebar kind is the narrow maintainable direction, but the parse/pretty-print work needs a size cap or click-time deferral before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap JSON parsing before rendering tool rows:** `ui/src/ui/chat/tool-cards.ts:220-222`
  - body: `renderToolCard` now constructs `sidebarActionContent` for every collapsed non-canvas row, which calls this helper and parses plus pretty-prints the full `outputText` before the user opens the panel. Tool output can be multi-MB and model/tool-controlled, so this can stall the Control UI render path; cap it like the existing 20k guard or defer parsing until the sidebar opens.
  - confidence: 0.89
