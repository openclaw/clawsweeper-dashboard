## Summary

Keep open. The PR targets a real current-main UI bug and the dedicated tool sidebar design is the right direction, but the latest diff introduces an unbounded JSON parse/pretty-print path during tool-card rendering that should be fixed before merge.

## What This Changes

The PR adds a `tool` sidebar content kind, renders tool input/output as expanded sidebar blocks with raw-text fallback, routes non-canvas collapsed tool-card clicks to the sidebar, and adds focused UI tests plus a changelog entry.

## Best Possible Solution

Land a revised version that keeps the dedicated escaped tool sidebar renderer and non-canvas click-to-sidebar behavior, while bounding or lazily deferring JSON parsing/pretty-printing so large tool payloads cannot stall chat rendering.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction path: tool details are serialized as markdown, then JSON fenced blocks are rendered through the markdown `json-collapse` details path. The introduced performance regression is also source-reproducible because the PR builds parsed sidebar content during render for collapsed non-canvas tool cards.

## Solution Assessment

Not as submitted. A dedicated `tool` sidebar content kind is the narrow maintainable fix, but the implementation should cap or defer JSON parsing before merge rather than parsing arbitrary tool output in the render loop.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Cap JSON parsing before rendering tool cards:** `ui/src/ui/chat/tool-cards.ts:218-219`
  - body: `renderToolCard` now builds `sidebarActionContent` while rendering every collapsed non-canvas tool row, and that path calls `isJsonPayload`/`prettyPrintJsonOrSelf` on the full `outputText`.
