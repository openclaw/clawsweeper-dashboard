## Summary

Keep open: current main still routes tool-card side-panel details through markdown/JSON-collapse, while this PR implements the right dedicated sidebar direction but still has a bounded-parsing blocker and no after-fix real behavior proof.

## What This Changes

The PR adds a Control UI `tool` sidebar content kind, routes non-canvas tool-card clicks to that side panel, renders input/output as expanded blocks with raw fallback, and updates CSS, tests, and the changelog.

## Best Possible Solution

Keep the dedicated tool-sidebar design, but cap or defer JSON parsing, preserve existing large-payload guardrails, and merge only after real UI proof demonstrates the fixed side-panel behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible because tool sidebar content is built as markdown and the shared markdown renderer collapses multi-line JSON fences into a summary disclosure.

## Solution Assessment

No, not as submitted. A dedicated `tool` sidebar kind is the narrow maintainable solution, but the implementation needs capped or deferred JSON parsing before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap JSON parsing before rendering tool rows:** `ui/src/ui/chat/tool-cards.ts:220-222`
  - body: `renderToolCard` constructs sidebar content for collapsed non-canvas rows, and that helper parses plus pretty-prints the full `outputText` during render. Tool output is model/tool-controlled and can be multi-MB, so this can stall the Control UI before the user opens the side panel; cap it like the existing large-output guard or defer parsing until the sidebar opens.
