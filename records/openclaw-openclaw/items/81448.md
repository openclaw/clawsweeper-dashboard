Keep open: the reporter's `channels.openclaw-weixin.enabled true` path still maps to a current-main Gateway startup gap, and no merged fix proves that installed enabled external channel plugins are loaded from that enabled-only channel config shape. ## What This Changes

The issue reports a regression where an installed and enabled external Weixin channel plugin is visible to CLI inspection but not loaded by Gateway startup in OpenClaw 2026.5.7.

## Best Possible Solution

Align the configured-channel presence/startup planner so an explicitly enabled installed external channel can enter Gateway startup without weakening the non-bundled plugin trust gates, and expand or supersede the adjacent enabled-only config PR with Gateway startup coverage.

## Reproduction Assessment

Yes, from source inspection for the reported config shape: current main treats `channels.openclaw-weixin.enabled=true` as not meaningfully configured in the Gateway startup presence path, and Gateway then loads only the planned plugin ids. I did not run a live packaged Weixin Gateway smoke.

## Solution Assessment

No current-main solution is complete. The narrow fix should update the startup/channel-presence path and tests, or expand https://github.com/openclaw/openclaw/pull/81331, while preserving explicit trust for non-bundled external plugins.
