## Summary

This PR is a focused, plausible fix and should stay open for maintainer review; I did not find a discrete blocking correctness defect, but the doctor/gateway repair behavior is compatibility- and availability-sensitive.

## What This Changes

The PR teaches doctor gateway health checks to classify protocol mismatch separately, passes that state into daemon repair, skips dead-gateway repair on mismatch, adds loaded service-definition repair, and adds focused regression tests.

## Best Possible Solution

Land the narrow mismatch classification only after maintainers accept the daemon repair policy: protocol mismatch should avoid dead-gateway repair, while stale service-definition repair remains allowed and the broader Windows migration issue stays tracked separately.

## Reproduction Assessment

Yes. Current main source shows protocol mismatch is folded into the generic "gateway closed" path and then passed to daemon repair as healthOk=false; the PR also includes native Windows after-fix command output for the changed behavior.

## Solution Assessment

Mostly yes. Classifying protocol mismatch separately and skipping dead-gateway repair is the narrow fix, but the added stale service-definition repair before the skip is an upgrade-sensitive policy choice that maintainers should explicitly accept.
