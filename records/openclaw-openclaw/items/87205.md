## Summary

Keep open. The PR usefully distinguishes protocol mismatch from a dead gateway, but its early return skips the stale service-definition and restart repair path that the linked Windows fallback report needs, and the Windows proof is not yet independently inspectable.

## What This Changes

The PR detects gateway protocol mismatch in doctor health checks, passes that state into gateway daemon repair, skips daemon repair on mismatch, and adds focused regression tests plus service-definition repair plumbing.

## Best Possible Solution

Keep the protocol-mismatch messaging, but let doctor still run targeted stale service-definition and restart diagnostics before skipping dead-gateway install/start repair, then require redacted native Windows terminal proof before merge.

No live high-confidence reproduction was run in this review. Source inspection shows current main treats protocol-mismatch close errors as failed health and then enters daemon repair, while the PR body describes native Windows steps.

## Solution Assessment

No. Classifying protocol mismatch separately is the right direction, but returning before stale service-definition repair makes the fix too broad for the linked upgrade/fallback failure mode.
