## Summary

This PR should stay open because the protocol-mismatch classification is a useful fix, but the current branch returns before documented `doctor --deep` diagnostics that users need to identify stale clients during protocol mismatch recovery. The branch is also reported dirty against current `main`, so it needs a refresh before it can be landed.

## What This Changes

The PR classifies gateway protocol-mismatch health failures separately, passes that state into doctor daemon handling, skips daemon repair for mismatch cases, reuses service-definition repair helpers, and adds targeted regression tests.

## Best Possible Solution

Keep the protocol-mismatch health classification, but move the skip so it bypasses only mutating daemon install/start/restart repair while preserving non-mutating deep diagnostics and adding regression coverage for that ordering.

## Reproduction Assessment

Yes. Source inspection shows current main classifies a `gateway closed ... protocol mismatch` status failure as `Gateway not running`, and the PR supplies native Windows terminal output for the mismatch and `doctor --deep` completion; I did not rerun the Windows flow in this read-only review.

## Solution Assessment

No. The classification direction is right, but the early return should only skip mutating daemon repair, not the deep diagnostics that tell users which stale process to stop.
