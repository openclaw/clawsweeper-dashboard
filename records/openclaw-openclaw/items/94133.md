## Summary

This PR targets a real subagent availability problem, but the proposed heartbeat does not currently reach the idle-timeout listener: the listener is registered on the wrapper-created stream abort signal while the tool path notifies the run abort signal, so the WeakMap lookup cannot match and the watchdog is not reset. It also lacks after-fix real behavior proof from a real subagent run, so it should stay open for a focused repair.

## What This Changes

The PR adds a tool-activity heartbeat module, subscribes the LLM idle-timeout wrapper to it, and notifies after embedded-runner tool calls complete.

## Best Possible Solution

Keep the PR open, repair the heartbeat so the tool path and idle-timeout wrapper share the same activity key, and add focused timeout coverage plus redacted real subagent proof.

## Reproduction Assessment

Yes from source inspection: the existing idle wrapper waits on activity keyed by the provider stream signal, and this PR would notify a different run-level signal from the tool path. I did not run a live subagent repro in this read-only review.

## Solution Assessment

No. The owner boundary is plausible, but the implementation must share the activity signal between tool execution and streamWithIdleTimeout before it can solve the reported idle timeout.
