## Summary

Keep open. Current main still rejects Docker bind sources outside the runtime allowlist unless the dangerous external-source override is enabled, while the public sandboxing docs still show arbitrary host-path bind examples without clearly documenting that default constraint in the custom bind section. The linked #59718 is useful adjacent design context, but it does not supersede this narrower docs/runtime mismatch.

## What This Changes

The issue asks OpenClaw to reconcile `sandbox.docker.binds` behavior with the sandboxing docs because external host paths are rejected unless they are under the per-session sandbox or agent workspace roots.

## Best Possible Solution

Maintainers should decide the Docker bind contract explicitly, then either document the default sandbox/workspace-root allowlist plus `agents.defaults.sandbox.docker.dangerouslyAllowExternalBindSources` next to the custom bind examples, or intentionally relax the runtime policy for configured non-blocked external sources with security-focused regression coverage and coordinated least-privilege writable-bind semantics.

## Reproduction Assessment

Yes. Current code inspection gives a high-confidence reproduction path: configure a Docker bind source outside the sandbox/workspace roots, leave `dangerouslyAllowExternalBindSources` unset, and sandbox creation reaches `validateBindMounts` with bounded allowed roots and throws outside-allowed-roots. The inspected tests assert the same default failure and override success paths.

## Solution Assessment

Unclear. The current allowlist is a defensible safer default, but it conflicts with the bind examples; the narrowest maintainable fix is probably documentation/config clarification unless maintainers intentionally want configured external binds to be trusted by default.
