## Summary

Keep open. Current main still rejects Docker bind sources outside the runtime sandbox/workspace roots unless `dangerouslyAllowExternalBindSources` is enabled, while the custom-bind docs still show arbitrary host-path examples without stating the default roots or naming the override; https://github.com/openclaw/openclaw/issues/59718 is adjacent least-privilege design work, not a superseding tracker for this docs/runtime mismatch.

## What This Changes

The issue asks OpenClaw to align `docker.binds` behavior and docs so host paths outside the per-session sandbox or agent workspace roots are accepted or clearly documented with the dangerous override.

## Best Possible Solution

Maintainers should decide the Docker bind contract explicitly, then either document the default roots plus `dangerouslyAllowExternalBindSources` next to the examples or intentionally relax configured external binds with security-focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure a Docker bind whose source is outside `workspaceDir` and `agentWorkspaceDir`, leave `dangerouslyAllowExternalBindSources` unset, and current `buildSandboxCreateArgs` reaches `validateBindMounts` with bounded roots and throws `outside allowed roots`; the tests assert that path.

## Solution Assessment

Unclear. The current allowlist is a defensible safer default, so documentation clarification is likely the narrowest maintainable fix unless maintainers intentionally want configured external binds to be trusted by default.
