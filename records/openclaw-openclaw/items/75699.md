## Summary

Keep open. Current main has manual `/new` and `/reset`, lazy daily/idle reset, `/status` and `/context` visibility, and preemptive/overflow compaction, but it does not implement the requested config-backed percent-threshold session rollover with durable handoff and automatic fresh backing-session routing for Telegram/direct sessions. The request is valid but broad enough to need maintainer design before implementation.

## What This Changes

Request to add native config-backed context-pressure rollover for long-running direct sessions, including thresholds, durable handoff summaries, Telegram delivery preservation, new status/rollover commands, dry-run behavior, and tests.

## Best Possible Solution

Design this as an additive session lifecycle capability after maintainer review: typed config and threshold validation, a reliable prompt/context usage snapshot before model submission, redacted durable handoff artifacts, atomic rotation from one backing `sessionId` to another while keeping the logical session key and delivery context stable, handoff injection into the next turn, and generic direct-session plus fake Telegram-direct tests. If maintainers prefer plugin ownership, first add a generic SDK seam that safely supports this pre-dispatch lifecycle operation.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from source and docs inspection of session config, session initialization, command registration, context/status reporting, compaction paths, Telegram routing, and plugin/context-engine contracts.

## Solution Assessment

Unclear until maintainer design review. A native lifecycle policy or a new generic session-lifecycle SDK seam appears safer than a Telegram-only workaround because the feature must alter session-store state and delivery routing before the model call while preserving privacy and idempotency.
