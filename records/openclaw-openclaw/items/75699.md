## Summary

Keep open. Current main has manual reset, lazy daily/idle reset, context/status reporting, auto-compaction, compaction checkpoints, and optional successor transcript rotation, but it still does not provide the requested native `session.contextRollover` style policy with percent thresholds, durable handoff commands/files, dry-run behavior, and automatic fresh backing-session routing for Telegram/direct sessions.

## What This Changes

The issue requests native automatic context-pressure session rollover with durable handoff summaries, percent-based thresholds, preserved Telegram/direct delivery identity, and config-backed controls.

## Best Possible Solution

Design an additive session lifecycle policy or generic lifecycle API that validates typed rollover config, calculates pressure from fresh prompt/context snapshots, creates redacted durable handoff artifacts, atomically rotates the backing session while preserving the logical session key and delivery context, injects the handoff into the next turn, and reuses existing compaction checkpoint/successor-transcript pieces only where they fit the contract.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from current-main source/docs inspection and targeted searches across session config, inbound session initialization, command registry, compaction, Telegram routing, and plugin API surfaces.

## Solution Assessment

Unclear until maintainer design review. Existing compaction and successor-transcript behavior is useful adjacent infrastructure, but the requested behavior likely needs a native lifecycle policy or new generic API because the important operation is a delivery-preserving backing-session swap before the model call.
