## Summary

Keep open. Current main has manual session reset, lazy daily/idle reset, token/status visibility, and compaction, but it still lacks native config-backed percent-based context rollover with durable handoff creation and fresh backing-session routing for Telegram/direct sessions. The remaining work crosses config, session lifecycle, prompt/token accounting, durable storage, plugin/context-engine seams, Telegram delivery identity, and privacy behavior, so it needs maintainer design rather than cleanup closure.

## What This Changes

The issue requests native config-backed automatic context rollover that monitors token pressure, creates durable handoff summaries, and routes Telegram/direct sessions into fresh backing session ids without changing the logical channel identity.

## Best Possible Solution

Design an additive session lifecycle policy or generic lifecycle API under maintainer review: validate typed config, compute rollover from fresh prompt/context snapshots, create redacted durable handoff artifacts, atomically rotate the backing session id while preserving logical session key and delivery context, inject the handoff into the next turn, and cover generic direct sessions plus fake Telegram direct metadata in tests.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from current-main inspection of the strict session config schema, command registry, session initialization path, compaction/context docs, plugin hook contracts, and Telegram routing metadata.

## Solution Assessment

Unclear until maintainer design review. A native lifecycle policy or a new generic session-lifecycle API appears more maintainable than a ClawHub-only workaround with current hooks, because the feature must decide before the model call whether to swap the active backing session while preserving delivery identity.
