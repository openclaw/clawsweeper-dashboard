## Summary

Keep open. Current main still has manual reset, lazy daily/idle reset, context reporting, memory flush, and compaction, but not native percent-based session rollover with durable handoff and fresh backing-session routing for Telegram/direct sessions.

## What This Changes

The issue requests native config-backed percent-based context rollover that creates durable handoff summaries and routes Telegram/direct peers into a fresh backing session before context pressure becomes unsafe.

## Best Possible Solution

Design an additive generic session lifecycle policy or documented SDK seam that validates thresholds, computes fresh prompt pressure, writes redacted durable handoffs, atomically rotates the backing `sessionId`, preserves logical session and delivery identity, and injects handoff context before the next model turn.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from current-main inspection of strict session config, command registration, session initialization, compaction docs, Telegram routing metadata, and plugin hook contracts.

## Solution Assessment

Unclear until maintainer design review. A generic lifecycle policy or versioned plugin seam is safer than a Telegram-only workaround because the requested behavior crosses core session routing, transcript persistence, delivery identity, and security-sensitive handoff output.
