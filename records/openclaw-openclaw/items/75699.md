## Summary

Keep open. Current main has manual /new and /reset, lazy daily/idle reset, /status and /context reporting, preflight compaction, memory flush, and compaction-triggered sessionId rotation, but not native session.contextRollover thresholds, durable handoff commands/files, dry-run/status controls, or automatic percent-based backing-session rollover for Telegram/direct peers.

## What This Changes

The issue requests native config-backed automatic context rollover with durable handoff for long-running Telegram/direct sessions.

## Best Possible Solution

Design an additive generic session lifecycle policy or SDK-backed core seam that validates thresholds, computes fresh prompt pressure, writes redacted durable handoffs, atomically rotates the backing sessionId while preserving logical session/delivery identity, and injects the handoff before the next model turn.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from current-main inspection of session config, inbound session routing, compaction/memory-flush paths, command registration, Telegram-adjacent session helpers, and plugin hook contracts.

## Solution Assessment

Unclear until maintainer design review. A native lifecycle policy or new core plugin seam appears safer than a Telegram-only workaround because current hooks can inject context but do not own atomic pre-model backing-session rotation.
