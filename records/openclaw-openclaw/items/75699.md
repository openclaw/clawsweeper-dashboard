## Summary

Keep open. Current main has manual `/new`/`/reset`, daily/idle reset, `/compact`, context/status visibility, and token-gated memory/compaction helpers, but it does not implement native config-backed percent-based context rollover with durable handoff and automatic fresh-session routing for Telegram/direct sessions. The request is real but cross-cuts session lifecycle, config schema, context accounting, handoff persistence, and channel delivery identity, so it needs maintainer design review rather than conservative cleanup.

## What This Changes

Add native config-backed percent-based context rollover for Telegram/direct sessions, including durable handoff summaries, same logical session key with a fresh backing session id, status/rollover commands, dry-run, notifications, and delivery-preservation safeguards.

## Best Possible Solution

Design this as an additive native gateway session lifecycle policy if maintainers want it: add a typed config schema, validate threshold ordering, use fresh prompt-context snapshots for gating, create redacted durable handoff artifacts, atomically rotate the backing `sessionId` while preserving `deliveryContext`, inject the handoff into the next turn, and cover generic direct sessions plus fake Telegram direct metadata in tests. If maintainers prefer plugin ownership, first add a generic session-lifecycle hook/API that can safely perform pre-model-call rollover.

## Reproduction Assessment

Not applicable as a feature request. The current-main inspection gives a high-confidence absence check through strict config schema, command registry/docs, session lifecycle code, and exact symbol search rather than a runtime bug reproduction.

## Solution Assessment

Unclear until maintainer product/API review. A native lifecycle policy appears more appropriate than a ClawHub-only plugin with current APIs because the feature must switch session ids before the model call and preserve channel delivery identity, but the exact config and handoff command surface should be designed deliberately.
