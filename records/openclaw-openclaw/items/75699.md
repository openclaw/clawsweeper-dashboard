## Summary

Keep open. Current main has manual `/new` and `/reset`, lazy daily/idle session reset, `/compact`, context/status visibility, and token-pressure compaction helpers, but it still does not provide native config-backed percent-based context rollover with durable handoff summaries and automatic fresh backing-session routing for Telegram/direct sessions. The request is valid but broad and safety-sensitive, so it needs maintainer design rather than conservative cleanup.

## What This Changes

The issue requests native, config-backed context-pressure session rollover with durable handoff summaries and automatic fresh backing sessions for Telegram/direct sessions.

## Best Possible Solution

Design this as an additive session lifecycle policy only after maintainer review: validate a typed config schema, compute rollover from fresh prompt/context snapshots, create redacted durable handoff artifacts, atomically rotate the backing `sessionId` while preserving the logical session key and delivery context, inject the handoff into the next turn, and cover generic direct sessions plus fake Telegram direct metadata in tests. If maintainers prefer plugin ownership, first add a generic session-lifecycle hook/API that can safely perform pre-model-call rollover without breaking delivery identity.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from current-main inspection of strict session config schema, command registry/docs, session initialization, token/compaction helpers, Telegram routing, and plugin hook contracts.

## Solution Assessment

Unclear until maintainer design review. A native lifecycle policy or a new generic session-lifecycle API appears more maintainable than a ClawHub-only workaround with current hooks, because the feature must swap the active backing session before the model call while preserving channel delivery identity.
