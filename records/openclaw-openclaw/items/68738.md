## Summary

Keep open: current main and v2026.5.7 still perform unbounded memory-wiki page read fanout during compile, matching the reported iCloud/FileProvider trigger, and no related PR owns the fix.

## What This Changes

The issue asks for bounded filesystem-read concurrency in memory-wiki compile, especially `readPageSummaries`, to avoid iCloud/FileProvider deadlocks.

## Best Possible Solution

Add a plugin-local bounded read limit around `readPageSummaries`, preserve deterministic sorted output and fail-on-read-error behavior, and cover peak concurrent reads in `compile.test.ts`; leave a public config knob for later product discussion.

## Reproduction Assessment

Yes for the OpenClaw trigger: current main starts every page `fs.readFile` through one unbounded `Promise.all`. The live FileProvider EDEADLK symptom still needs a macOS iCloud-backed vault to reproduce end to end.

## Solution Assessment

Yes: a bounded worker in the memory-wiki plugin using the existing SDK concurrency helper is the narrowest maintainable fix. Adding a new dependency or user-facing config option is not required for the first repair.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/68738 by bounding memory-wiki compile page reads. In `extensions/memory-wiki/src/compile.ts`, replace the unbounded `Promise.all(filePaths.map(... fs.readFile ...))` in `readPageSummaries` with a plugin-local small fixed cap, such as 8, using `openclaw/plugin-sdk/concurrency-runtime` or an equivalent local worker. Preserve output ordering, the final sorted result, and fail the compile on any read error; do not add a new public config option or new dependency unless maintainers ask. Add or adjust `extensions/memory-wiki/src/compile.test.ts` to create more pages than the cap and assert peak concurrent read calls never exceed the cap. Add a `CHANGELOG.md` fix entry if this ships as a user-facing fix.
