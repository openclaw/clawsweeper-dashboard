## Summary

Keep open: current main and the latest release still start all memory-wiki compile page reads through an unbounded `Promise.all`, matching the reported iCloud/FileProvider failure trigger, and there is no linked closing PR.

## What This Changes

The issue requests bounded memory-wiki compile page-read concurrency to avoid EDEADLK, hangs, or partial reads on iCloud/FileProvider-backed vaults.

## Best Possible Solution

Add a bounded worker limit around `readPageSummaries` page reads, preserve deterministic sorted output, and cover peak concurrent reads in `compile.test.ts`; treat a config knob as a later product choice.

## Reproduction Assessment

Yes for the source-level trigger: current main fans out all page `fs.readFile` calls through one unbounded `Promise.all`. The FileProvider EDEADLK symptom itself still needs a macOS iCloud-backed vault for live reproduction.

## Solution Assessment

Yes: a plugin-owned bounded read worker using the existing SDK concurrency helper is the narrowest maintainable fix. Adding `p-limit` or a user-facing config option is not required for the first repair.
