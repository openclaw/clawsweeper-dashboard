## Summary

Keep open: current main and the latest release still start all memory-wiki page reads through an unbounded `Promise.all`, matching the reported FileProvider trigger, and the issue has no closing PR or canonical duplicate.

## What This Changes

Limit memory-wiki compile page reads so iCloud/FileProvider-backed vaults do not trigger EDEADLK under bursty filesystem read concurrency.

## Best Possible Solution

Add a plugin-local bounded read limit around `readPageSummaries`, preserve deterministic sorted output and existing error behavior, and cover peak concurrent reads in `compile.test.ts`; leave a user-facing config knob for a later product decision.

## Reproduction Assessment

Yes for the source-level trigger: current main starts every page `fs.readFile` from one unbounded `Promise.all`. Live reproduction of the macOS FileProvider EDEADLK symptom still needs an iCloud-backed vault.

## Solution Assessment

Yes: bounding the page reads inside the memory-wiki plugin with the existing SDK concurrency helper is the narrowest maintainable fix. Adding `p-limit` or a new public config option is not required for the first repair.
