## Summary

Keep open: current main still starts every memory-wiki page read through one unbounded `Promise.all`, the latest tagged release still has the same pattern, and live GitHub search did not show a closing PR or canonical duplicate owning the fix.

## What This Changes

The issue asks to cap memory-wiki compile markdown page reads so iCloud/FileProvider-backed vaults do not hang, fail with EDEADLK, or produce partial reads under an unbounded burst.

## Best Possible Solution

Add a bounded, plugin-local read worker in `readPageSummaries`, preserve deterministic sorted output and current read-error propagation, and cover peak concurrent reads in `compile.test.ts`.

## Reproduction Assessment

Yes for the source-level trigger: current main starts every page `fs.readFile` through one unbounded `Promise.all`. The full EDEADLK symptom still needs a macOS iCloud/FileProvider-backed vault for live reproduction.

## Solution Assessment
