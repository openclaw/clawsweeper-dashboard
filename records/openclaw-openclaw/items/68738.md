Add a plugin-local bounded read limit around `readPageSummaries`, preserve deterministic sorted output and existing error behavior, and cover peak concurrent reads in `compile.test.ts`; leave a user-facing config knob for later product discussion.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/68738. In the memory-wiki plugin, `readPageSummaries` currently starts every page `fs.readFile` at once via `Promise.all(filePaths.map(...))`, which can hang or fail on iCloud/FileProvider-backed vaults. Add a small bounded concurrency limit around page reads in `extensions/memory-wiki/src/compile.ts`, preferably reusing `runTasksWithConcurrency` from `openclaw/plugin-sdk/concurrency-runtime`. Preserve deterministic output ordering, the final title sort, and existing error behavior. Add focused regression coverage in `extensions/memory-wiki/src/compile.test.ts` proving compile page reads do not exceed the chosen cap. Do not add a new dependency or public config knob unless maintainers explicitly ask. Add a `CHANGELOG.md` fix entry because this is user-visible. Keep the fix scoped to memory-wiki compile reads.
