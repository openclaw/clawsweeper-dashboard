## Summary

Keep open: current main and v2026.5.18 still read every memory-wiki page through an unbounded `Promise.all` of `fs.readFile`, with no compile read concurrency limit or canonical fixing PR found.

## What This Changes

The issue requests bounding memory-wiki compile page reads so iCloud/FileProvider-backed vaults do not hang, fail, or corrupt wiki output under large read bursts.

## Best Possible Solution

Add a plugin-local bounded read limit around `readPageSummaries`, preserve deterministic output and current read-error behavior, and cover peak concurrent reads in `compile.test.ts`; treat a public config knob as later product work.

## Reproduction Assessment

Yes for the source-level trigger: current main maps every page to `fs.readFile` inside one unbounded `Promise.all`, and compile can call that path repeatedly. I did not run a live macOS iCloud/FileProvider reproduction in this Linux read-only review.

## Solution Assessment

Yes: bounding `readPageSummaries` inside the memory-wiki plugin with the existing SDK concurrency helper is the narrowest maintainable first fix. Adding a new dependency or public compile concurrency option is not required for the first repair.
