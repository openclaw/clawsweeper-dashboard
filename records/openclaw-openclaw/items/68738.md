## Summary

Keep open: current `main` and the latest release still read all memory-wiki compile pages through an unbounded `Promise.all`, matching the reported FileProvider trigger, and no duplicate or closing PR owns the fix.

## What This Changes

The issue requests a bounded concurrency limit for memory-wiki compile page reads so iCloud/FileProvider-backed vaults do not deadlock under large read bursts.

## Best Possible Solution

Add a plugin-local bounded read limit around `readPageSummaries`, preserve deterministic sorted output and existing error behavior, and cover peak concurrent reads in `compile.test.ts`; leave a user-facing config knob for later product discussion.

## Reproduction Assessment

Yes for the source-level trigger: current `main` starts all page `fs.readFile` calls from one unbounded `Promise.all`. Live reproduction of the FileProvider EDEADLK symptom still needs a macOS iCloud-backed vault.

## Solution Assessment

Yes: a bounded worker in the memory-wiki plugin using the existing SDK concurrency helper is the narrowest maintainable fix. Adding `p-limit` or a public config option is not required for the first repair.
