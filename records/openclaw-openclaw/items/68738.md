## Summary

Keep open: current main still has the reported unbounded memory-wiki compile read burst, the latest release tag shows the same pattern, and the report is concrete enough for a focused plugin-owned fix PR.

## What This Changes

The issue requests bounding memory-wiki compile page reads so iCloud/FileProvider-backed vaults are not hit with an unbounded fs.readFile burst.

## Best Possible Solution

Add a bounded worker limit inside the memory-wiki compile page read path using the existing plugin SDK concurrency helper, preserve deterministic output ordering, and add a focused regression test that proves peak concurrent reads stay under the cap. A config knob can wait unless maintainers explicitly want it in the first fix.

## Reproduction Assessment

Yes for the OpenClaw trigger path: a large memory-wiki vault followed by `openclaw wiki compile` reaches source that starts every page read through one unbounded Promise.all. The kernel/FileProvider EDEADLK symptom still requires a macOS iCloud-backed vault to reproduce live.

## Solution Assessment

Yes: a plugin-owned bounded read worker in `extensions/memory-wiki/src/compile.ts` is the narrowest maintainable fix because it addresses the pathological filesystem burst without changing core loader behavior or adding a new dependency.
