## Summary

Current main still matches the report: the memory-wiki compile path starts one `fs.readFile` per page inside an unbounded `Promise.all`, and the latest release still shipped that same pattern. This should stay open as a narrow plugin-owned reliability fix.

## What This Changes

The issue requests bounding `readPageSummaries` filesystem read concurrency in the memory-wiki compile path to avoid iCloud/FileProvider-backed vault deadlocks and partial reads.

## Best Possible Solution

Add a bounded worker limit inside the memory-wiki page-summary read path, using the existing plugin SDK concurrency helper or an equivalent local bounded loop, with a conservative fixed cap while preserving sorted deterministic output. Cover it with a focused regression test that proves peak concurrent reads stay under the cap.

## Reproduction Assessment

Yes for the OpenClaw trigger path: a large memory-wiki vault reaches `readPageSummaries`, which currently starts every page `fs.readFile` at once. The platform-specific iCloud/FileProvider EDEADLK symptom still needs macOS/iCloud live proof.

## Solution Assessment

Yes, a bounded worker in `extensions/memory-wiki/src/compile.ts` is the narrowest maintainable fix because this is plugin-owned behavior and does not require core loader changes. A user-facing config knob can be considered later if maintainers want tunability beyond a conservative default.
