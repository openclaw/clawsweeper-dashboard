## Summary

Keep open: current main and the latest stable release still use an unbounded page-read burst in memory-wiki compile, and no canonical duplicate or closing PR owns the fix.

## What This Changes

The issue requests bounded concurrency for memory-wiki page-summary reads so large iCloud/FileProvider-backed vaults do not hang, fail, or compile partial wiki data.

## Best Possible Solution

Add a plugin-local bounded read limit around `readPageSummaries`, preserve deterministic output and current read-error behavior, and cover peak concurrent reads in `compile.test.ts`; leave any public config knob for separate product review.

## Reproduction Assessment

Yes for the source-level trigger: current main starts one `fs.readFile` per page from an unbounded `Promise.all`, and compile can call that path repeatedly. I did not live-reproduce macOS FileProvider `EDEADLK` in this Linux read-only review.

## Solution Assessment

Yes: bounding `readPageSummaries` inside the memory-wiki plugin with the existing SDK concurrency helper is the narrowest maintainable first fix. Adding `p-limit` or a public compile concurrency setting is not required for the first repair.
