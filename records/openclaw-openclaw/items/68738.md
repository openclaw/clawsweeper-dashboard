## Summary

Keep open: current main and the latest release still have the unbounded memory-wiki compile read burst, and there is no paired open PR fixing this path.

## What This Changes

The issue requests bounding `readPageSummaries` file-read concurrency during memory-wiki compile to avoid iCloud/FileProvider EDEADLK failures.

## Best Possible Solution

Add a bounded worker limit inside the memory-wiki compile page-read path, preserve deterministic sorted output, and add a focused regression test for peak concurrent reads; a config knob can be considered later.

## Reproduction Assessment

Yes for the OpenClaw trigger path: a large memory-wiki vault reaches `readPageSummaries`, which currently starts every page read through one unbounded `Promise.all`. The kernel/FileProvider EDEADLK symptom still requires a macOS iCloud-backed vault for live reproduction.

## Solution Assessment

Yes: a plugin-owned bounded read worker using the existing SDK concurrency helper is the narrowest maintainable fix and avoids adding a new dependency or changing core loader behavior.
