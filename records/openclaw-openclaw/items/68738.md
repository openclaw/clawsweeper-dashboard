## Summary

Keep open: current main and the latest release still use an unbounded memory-wiki compile read burst, and the report is narrow, reproducible from source, and not protected or paired with a linked fixing PR in the provided context.

## What This Changes

The issue requests bounding memory-wiki compile page reads so iCloud/FileProvider-backed vaults are not hit with an unbounded `fs.readFile` burst.

## Best Possible Solution

Add a bounded worker limit inside the memory-wiki compile page-read path, preserve deterministic sorted output, and add a focused regression test for peak concurrent reads; a user config knob can be considered after the safe default lands.

## Reproduction Assessment

Yes for the OpenClaw trigger path: a large memory-wiki vault reaches `readPageSummaries`, which currently starts every page read through one unbounded `Promise.all`. The kernel/FileProvider `EDEADLK` symptom still requires a macOS iCloud-backed vault for live reproduction.

## Solution Assessment

Yes: a plugin-owned bounded read worker using the existing SDK concurrency helper is the narrowest maintainable fix and avoids core loader changes or a new dependency.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/68738. In the memory-wiki plugin, bound the filesystem read concurrency in `readPageSummaries` so `openclaw wiki compile` does not start one `fs.readFile` per markdown page at once on iCloud/FileProvider-backed vaults. Keep the fix plugin-owned in `extensions/memory-wiki`; do not add a new dependency and do not move behavior into core. Prefer `runTasksWithConcurrency` from `openclaw/plugin-sdk/concurrency-runtime` or the existing approved SDK surface, preserve the current deterministic title sort and filtering behavior, and keep errors visible rather than silently dropping failed reads. Add a focused regression test in `extensions/memory-wiki/src/compile.test.ts` that creates enough pages and stubs/delays reads or otherwise observes peak concurrent page reads to prove the cap is honored. Add a user-facing `CHANGELOG.md` fix entry because this affects `openclaw wiki compile`. Do not introduce a config migration unless necessary; a hard conservative default is acceptable for the first fix.
