## Summary

Current main still has the unbounded read burst in the memory-wiki compile path, the latest release contains the same implementation, and no closing PR or canonical duplicate was found, so this remains a valid narrow bug rather than cleanup-close material.

## What This Changes

The issue requests bounding memory-wiki compile page reads so iCloud/FileProvider-backed vaults are not hit with an unbounded fs.readFile burst.

## Best Possible Solution

Add a plugin-local bounded read limit around readPageSummaries, preserve deterministic output and current error behavior, and cover peak concurrent reads in compile.test.ts; defer any user-facing concurrency setting until maintainers decide it is needed.

## Reproduction Assessment

Yes for the source-level trigger: current main starts every page fs.readFile from one unbounded Promise.all in readPageSummaries. The live macOS FileProvider EDEADLK symptom still needs an iCloud-backed vault to reproduce directly.

## Solution Assessment

Yes: bounding readPageSummaries inside the memory-wiki plugin with the existing SDK concurrency helper is the narrowest maintainable fix. Adding p-limit or a new public config option is not required for the first repair.
