## Summary

Keep open: current main and the latest release still read every memory-wiki page through an unbounded `Promise.all` of `fs.readFile`, and targeted GitHub searches found no closing PR or canonical duplicate owning the fix.

## What This Changes

The issue asks to bound `readPageSummaries` filesystem read concurrency during memory-wiki compile so iCloud/FileProvider-backed vaults do not deadlock or produce partial reads.

## Best Possible Solution

Add a plugin-local bounded read limit around `readPageSummaries`, preserve deterministic output and current read-error behavior, and cover peak concurrent reads in `compile.test.ts`; leave a public config knob for later product review.

## Reproduction Assessment

Yes for the source-level trigger: current main starts one `fs.readFile` per page from an unbounded `Promise.all`, and compile can call that path repeatedly. I did not live-reproduce macOS FileProvider EDEADLK in this Linux read-only review.

## Solution Assessment

Yes: bounding `readPageSummaries` inside the memory-wiki plugin with the existing SDK concurrency helper is the narrowest maintainable first fix. Adding `p-limit` or a public compile concurrency setting is not required for the first repair.
