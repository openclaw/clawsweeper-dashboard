## Summary

Keep open: current main still has the reported unbounded memory-wiki compile read burst, the latest release tag has the same pattern, and the issue is concrete enough for a focused plugin-owned fix PR.

## What This Changes

The issue requests bounding memory-wiki compile page-read concurrency so iCloud/FileProvider-backed vaults do not receive an unbounded `fs.readFile` burst during `openclaw wiki compile` or ingest auto-compile.

## Best Possible Solution

Add a bounded worker limit inside the memory-wiki compile page-summary read path using the existing plugin SDK concurrency helper, keep deterministic output ordering and existing read-error semantics, and add a focused regression test proving peak concurrent page reads stay under the cap. A user-facing changelog entry should credit the reporter; a config knob can wait unless maintainers explicitly want it.

## Reproduction Assessment

Yes for the OpenClaw trigger path: a large memory-wiki vault followed by `openclaw wiki compile` reaches source that starts every page read through one unbounded `Promise.all`. The kernel/FileProvider EDEADLK symptom requires a macOS iCloud-backed vault to reproduce live.

## Solution Assessment

Yes: a plugin-owned bounded read worker in `extensions/memory-wiki/src/compile.ts` is the narrowest maintainable fix because it addresses the pathological filesystem burst without changing core loader behavior or adding a new dependency.
