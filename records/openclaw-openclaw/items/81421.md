## Summary

Keep open: the PR is a focused implementation with sufficient resolver proof and green CI, but it changes whole-agent runtime keys from legacy ignored values into active runtime policy while current main's docs, tests, and doctor migration contract still define those keys as ignored.

## What This Changes

The PR adds `agents.list[].agentRuntime` and `agents.defaults.agentRuntime` as fallback runtime policy sources and propagates `agent`/`defaults` metadata through harness and agent-list tests.

## Best Possible Solution

Either keep runtime policy provider/model-scoped and guide the linked issue to that config path, or make a maintainer-approved contract change that updates runtime selection, docs, doctor migration, and regression coverage together.

## Reproduction Assessment

Yes for the review finding: current main's source, docs, doctor migration rules, and tests consistently ignore whole-agent runtime keys, while the PR head makes those keys effective. I did not run tests because this was a read-only review.

## Solution Assessment

No. As written, the PR revives whole-agent runtime pins while the current public contract and doctor guidance still tell users those keys are ignored; the safer path is provider/model-scoped config or an explicit contract update across all surfaces.
