## Summary

Keep open: current main still has a credible source path where a saved `zai:default` API-key profile can be selected ahead of env/config fallback for embedded runs, and doctor auth health does not detect a present-but-bogus static API-key profile. The reporter’s follow-up narrows the problem to a bad saved Z.AI profile rather than a separate sub-agent auth path, but that repair gap is still real.

## What This Changes

The issue reports stale or bogus Z.AI auth profile state causing GLM embedded/sub-agent authentication failures after key rotation and doctor repair attempts.

## Best Possible Solution

Add a focused repair so bogus or stale default Z.AI API-key profiles are detected, refreshed, or clearly bypassed after key rotation without breaking explicit user-selected auth profiles.

## Reproduction Assessment

Yes for the source-level failure class, but not for a live Z.AI 401. Current main still has a path where a saved `zai:default` API-key profile is tried before env/config fallback, and doctor auth health treats present API-key profiles as static rather than detecting bogus static values.

## Solution Assessment

Unclear because no patch is proposed. The narrow maintainable direction is to harden Z.AI setup and/or generic doctor auth-profile repair for stale default API-key profiles while preserving explicit profile selection semantics.
