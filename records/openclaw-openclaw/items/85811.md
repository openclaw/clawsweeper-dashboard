## Summary

Keep open, but the patch should not merge as-is. The bug is plausible and the PR is focused, but routing mutable parent-only persona/user files through Codex collaboration-mode developer instructions is not a safe per-turn overlay and can leave stale profile blocks in Codex prompt history.

## What This Changes

The PR moves Codex workspace `TOOLS.md` into inherited thread developer instructions, sends `SOUL.md`, `IDENTITY.md`, and `USER.md` through turn-start collaboration developer instructions, updates prompt accounting/tests, and documents the Codex-native subagent boundary.

## Best Possible Solution

Keep the subagent boundary goal, but route mutable persona/user context through a truly per-turn non-persistent surface or through binding rotation/fingerprinting aligned with https://github.com/openclaw/openclaw/pull/83612, rather than using collaboration mode for profile files.

## Reproduction Assessment

Yes, at source/proof level: current main still places profile files in Codex thread developer instructions, and the PR body supplies before/after live Codex output for the native subagent boundary. I did not rerun the live Codex scenario in this read-only review.

## Solution Assessment

No. The narrow boundary goal is right, but collaboration-mode developer instructions are mode/history state in Codex, so they are not the best place for mutable persona and user-profile files.
