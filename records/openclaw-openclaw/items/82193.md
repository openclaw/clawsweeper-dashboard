## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block broadcast from default HTTP tool invoke:** `src/agents/tool-catalog.ts:157-163`
  - body: Registering `sessions_broadcast` as a core/profile tool without adding it to the Gateway HTTP default deny list makes a cross-session system-event injection primitive reachable through `/tools/invoke` when tool policy allows it. `sessions_send` is denied there by default for the same class of risk, so add `sessions_broadcast` to `DEFAULT_GATEWAY_HTTP_TOOL_DENY` with docs and tests.
  - confidence: 0.92
- **[P1] Enforce session visibility before delivery:** `src/agents/tools/sessions-broadcast-tool.ts:242-245`
  - body: The broadcast loop queues system events for resolved rows without the shared session visibility checker. Under default `tree` or stricter `self` visibility, rows returned by `sessions.list` can still receive injected events even though current session tools would deny that target.
  - confidence: 0.93
- **[P2] Classify rows before kind filtering:** `src/agents/tools/sessions-broadcast-tool.ts:200-202`
  - body: The PR documents filters such as `main`, `cron`, and `hook`, but compares requested values directly to raw `session.kind` and skips the filter when that field is absent. Current `sessions_list` first calls `classifySessionKind`, so this can skip intended sessions or let untyped rows through a requested kind filter.
  - confidence: 0.9
- **[P2] Register broadcasts as mutating tools:** `src/agents/openclaw-tools.ts:233-239`
  - body: After exposing `sessions_broadcast` in the runtime tool surface, mutation and replay classification still only name `sessions_send` and `sessions_spawn`, and the fallback heuristic does not match `broadcast`. Retry/recovery paths can therefore treat a successful broadcast as non-mutating even though it queued external session events.
  - confidence: 0.87
