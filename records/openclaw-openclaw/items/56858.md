## Summary

Keep open. The feature is still useful, but this PR is not mergeable as reviewed: the live patch targets the obsolete `pi-extensions` context-pruning path while current main runs `pi-hooks`, and the media cache still needs retention, atomic-write, prompt-gating, and changelog fixes before maintainer review.

## What This Changes

This PR adds an opt-in `agents.defaults.contextPruning.cacheMedia` setting that writes pruned media to a local disk cache, replaces stripped media with `[media cached: ...]` markers, and teaches image detection to hydrate those markers.

## Best Possible Solution

Port the feature to the active `src/agents/pi-hooks/context-pruning` implementation, keep it opt-in, reuse or match existing media-store cleanup and atomic-write patterns, gate prompt text on the effective config, update generated config docs plus `CHANGELOG.md`, and validate with focused pruning/media-hydration tests before merge.

## Reproduction Assessment

Yes. On current main, source and tests show image tool-result blocks are still converted to `[image removed during context pruning]`, and the proposed `cacheMedia` setting is not present in the active `pi-hooks` settings path.

## Solution Assessment

No, not as currently proposed. Disk caching may be a reasonable opt-in solution, but this patch targets the wrong runtime path and needs retention, atomic-write, marker-preservation, prompt-gating, and changelog work before it is the narrowest maintainable fix.

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch persists previously pruned media and exposes local cache paths without bounded cleanup or atomic writes.

Concerns:

- **[medium] Unbounded cached-media retention:** `src/media/media-cache.ts:45`
  - body: `cacheMediaToDisk` writes pruned media under the OpenClaw config directory and returns the path, but the live diff contains no TTL cleanup hook or size bound. That can retain sensitive media that pruning previously discarded.
  - confidence: 0.88
- **[medium] Non-atomic cache write race:** `src/media/media-cache.ts:38`
  - body: The cache path is checked with `access()` and then written directly. A concurrent caller can observe the destination as present before the write is complete and insert a marker pointing at a partially-written file.
  - confidence: 0.84
