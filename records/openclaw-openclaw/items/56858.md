## Summary

Keep open: the feature is plausible, but this PR still needs rework before merge because the implementation targets the removed `pi-extensions` path, lacks real behavior proof, and introduces persistent media retention without the current media-store lifecycle safeguards.

## What This Changes

The PR adds an opt-in `agents.defaults.contextPruning.cacheMedia` setting that writes pruned media to disk and replaces stripped media with cached-media markers for later image rehydration.

## Best Possible Solution

Rework the feature on the active `src/agents/pi-hooks/context-pruning` path, use existing media-store lifecycle primitives, gate prompt text on effective config, and require redacted live-session proof before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection shows current main loads `pi-hooks` while the PR implements under stale `pi-extensions`, and the cache helper writes directly to final paths without lifecycle bounds. I did not run a live session in this read-only review.

## Solution Assessment

No. Disk caching may be a reasonable opt-in direction, but this patch should be reworked around the active pruning implementation and current media-store safety patterns before it is maintainable.

- **[P2] Port caching to the active pruning hook:** `src/agents/pi-extensions/context-pruning/extension.ts:1-3`
  - body: Current main wires context pruning through `src/agents/pi-hooks/context-pruning`, and this checkout has no `src/agents/pi-extensions` directory. This PR adds the cache path under the removed `pi-extensions` tree with old `@mariozechner` imports, so after rebase the feature would not run on the active pruning path.
  - confidence: 0.96
- **[P2] Bound cached-media retention:** `src/media/media-cache.ts:46-47`
  - body: `cacheMediaToDisk` writes every pruned media block under the OpenClaw config directory without a max size, TTL cleanup call, or lifecycle owner. Since pruning previously discarded this media, long sessions can retain sensitive files indefinitely.
  - confidence: 0.92
- **[P2] Write cache files atomically:** `src/media/media-cache.ts:39-47`
  - body: The helper checks whether the destination exists and then writes directly to the final cache path. A concurrent prune or interrupted write can leave a marker pointing at a partially written cached media file; use the existing media store/temp-file pattern instead.
  - confidence: 0.9
- **[P2] Normalize MIME before deriving cached kind:** `src/media/media-cache.ts:67-68`
  - body: `mediaCacheKind` passes raw MIME text to `mediaKindFromMime`, whose checks are case-sensitive and do not strip parameters. Values such as `IMAGE/PNG` or `image/png; charset=binary` become `document`, so cached image hydration can skip valid images.
  - confidence: 0.87
- **[P2] Preserve only cached-media marker tokens:** `src/agents/pi-extensions/context-pruning/pruner.ts:16-30`
  - body: `extractCachedMediaMarkers` keeps the whole text block whenever it contains a cached-media marker. After a prior soft-trim plus cache pass, those blocks can still contain large retained payload text and trim notes, defeating the hard-clear reduction.
  - confidence: 0.83
- **[P3] Gate cached-media prompt text on the setting:** `src/agents/system-prompt.ts:457-458`
  - body: The new `## Cached Media` system-prompt section is appended for every agent even when `cacheMedia` is disabled. That adds misleading instructions and token overhead for the default behavior where no cached-media markers will exist.
  - confidence: 0.82
