## Summary

Keep open, but not merge-ready: the underlying context-pruning media-loss problem is valid, while this branch is stale against the active pruning hook, lacks real behavior proof, and introduces persistent media storage without enough lifecycle and security bounds.

## What This Changes

The PR adds an opt-in `agents.defaults.contextPruning.cacheMedia` setting, a disk media-cache helper, cached-media markers, image rehydration from those markers, prompt text, generated config baseline updates, and focused unit tests.

## Best Possible Solution

Rework this as a fresh current-main implementation in `src/agents/pi-hooks/context-pruning` that uses bounded atomic media-store lifecycle primitives, documents the opt-in persistence semantics, preserves current pruning behavior by default, and includes focused regression tests plus redacted live proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main loads `pi-hooks`, while the PR head implements the feature under stale `pi-extensions`, and the cache helper directly writes persistent files without bounded lifecycle primitives. I did not run a live session in this read-only review.

## Solution Assessment

No. The requested direction is useful, but this branch is not the best implementation path until it is ported to the active pruning hook and redesigned around bounded, atomic media storage with explicit opt-in retention semantics.
