## Summary

Keep this PR open but not mergeable as-is: the branch adds a useful opt-in idea, but the patch targets a stale context-pruning path, lacks real behavior proof, and introduces unresolved persistent-media storage risks.

## What This Changes

The PR adds an opt-in `cacheMedia` context-pruning setting, a disk media cache, cached-media transcript markers, image marker hydration, prompt guidance, config schema updates, and unit tests.

## Best Possible Solution

Rework the feature on the active `pi-hooks` pruning path, reuse or match the existing bounded atomic media-store contract, gate prompt text on effective config, update config docs/tests, and provide redacted live-session proof.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main loads `src/agents/pi-hooks/context-pruning`, while the PR implements the cache/write path under `src/agents/pi-extensions/context-pruning`, so enabling `cacheMedia` would not exercise the new path.

## Solution Assessment

No. Disk caching may be a reasonable opt-in direction, but this patch should be reworked around the active pruning implementation and existing media-store safety patterns before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Port caching to the active pruning hook:** `src/agents/pi-extensions/context-pruning/extension.ts:1-3`
  - body: Current main wires context pruning through `src/agents/pi-hooks/context-pruning` and `@earendil-works/*`, but this patch adds the cache path under stale `pi-extensions` files with old `@mariozechner/*` imports. The new option would not affect the active runtime path.
  - confidence: 0.96
- **[P2] Bound cached media retention:** `src/media/media-cache.ts:47`
  - body: `cacheMediaToDisk` writes every pruned media block under the OpenClaw config directory without a wired TTL cleanup or size bound. Long sessions can retain sensitive media indefinitely even though pruning previously discarded it.
  - confidence: 0.91
- **[P2] Write cached files atomically:** `src/media/media-cache.ts:39-47`
  - body: The code checks for an existing destination and then writes directly to the final cache path. A concurrent prune can observe that path as present and emit a marker to a partially written file.
  - confidence: 0.88
- **[P2] Normalize MIME before classifying cached media:** `src/media/media-cache.ts:67-68`
  - body: `mediaCacheKind` passes raw MIME text to `mediaKindFromMime`, whose checks are case-sensitive and do not strip parameters. Values such as `IMAGE/PNG` or `image/png; charset=binary` become `document`, so cached image hydration skips them.
  - confidence: 0.86
- **[P2] Preserve only cache marker tokens during hard clear:** `src/agents/pi-extensions/context-pruning/pruner.ts:23-28`
  - body: `extractCachedMediaMarkers` keeps the whole text block whenever it contains a cached-media marker. After a prior soft-trim+cache pass, hard-clear can retain large payload text and trim notes, defeating the intended context reduction.
  - confidence: 0.84
- **[P2] Track only surviving soft-trim marker refs:** `src/agents/pi-extensions/context-pruning/pruner.ts:474-480`
  - body: The soft-trim callback records every image before head/tail truncation, but replacement later only has marker slots that survived the trim. If a middle marker is elided, cache writes become unreachable or later surviving markers can be paired with the wrong ref.
  - confidence: 0.83
- **[P3] Gate cached-media prompt guidance:** `src/agents/system-prompt.ts:457-459`
  - body: The cached-media prompt section is emitted for every agent even though `cacheMedia` defaults off and the active pruning path cannot produce these markers. That adds token overhead and tells agents to look for files that do not exist.
  - confidence: 0.9
