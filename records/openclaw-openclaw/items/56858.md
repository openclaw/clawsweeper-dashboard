## Summary

Keep open: the PR still does not implement the feature on the active current-main pruning path, and the proposed persistent media cache has correctness and security-sensitive retention issues plus missing real behavior proof.

## What This Changes

The PR adds an opt-in `cacheMedia` context-pruning setting, disk media cache, cached-media transcript markers, image marker hydration, prompt text, config schema updates, and unit tests.

## Best Possible Solution

Port the feature to active `pi-hooks`, reuse or match the bounded atomic media-store contract, gate prompt guidance on effective config, update config docs/tests, and provide redacted live-session proof.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main loads `src/agents/pi-hooks/context-pruning`, while this PR implements the cache/write path under stale `src/agents/pi-extensions/context-pruning`, so enabling `cacheMedia` would not affect the active runtime path.

## Solution Assessment

No. Disk caching may be a reasonable opt-in direction, but this patch should be reworked around the active pruning hook and existing media-store safety patterns before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Port caching to the active pruning hook:** `src/agents/pi-extensions/context-pruning/extension.ts:1-3`
  - body: Current main wires context pruning through `src/agents/pi-hooks/context-pruning`, but this patch adds the async cache path under stale `pi-extensions` files with old Pi package imports. Enabling `cacheMedia` will not affect the active runtime path.
  - confidence: 0.96
- **[P2] Bound cached-media retention before writing files:** `src/media/media-cache.ts:47`
  - body: `cacheMediaToDisk` writes every pruned media block under the OpenClaw config directory, but the new cache has no wired TTL cleanup or size bound. Long sessions can retain sensitive media indefinitely even though pruning previously discarded it.
  - confidence: 0.9
- **[P2] Write cached media through an atomic temp rename:** `src/media/media-cache.ts:39-47`
  - body: The cache path is checked with `fs.access()` and then written directly to the final filename. A concurrent prune can observe the destination as present and emit a marker to a partially written file; use the existing sibling-temp/rename media-store pattern.
  - confidence: 0.88
- **[P2] Normalize MIME before deriving cached-media kind:** `src/media/media-cache.ts:67-68`
  - body: `mediaCacheKind` passes raw MIME text to `mediaKindFromMime`, whose checks are case-sensitive and do not strip parameters. Values such as `IMAGE/PNG` or `image/png; charset=binary` become `document`, so image hydration skips cached images.
  - confidence: 0.86
- **[P2] Keep only cached-media marker tokens through hard clear:** `src/agents/pi-extensions/context-pruning/pruner.ts:16-28`
  - body: `extractCachedMediaMarkers` pushes the whole text block whenever it contains a cached-media marker. After a prior soft-trim plus cache pass, hard-clear can keep large head/tail payload text and trim notes, defeating the intended context reduction.
  - confidence: 0.84
- **[P2] Preserve marker slots for soft-trim-only cached images:** `src/agents/pi-extensions/context-pruning/pruner.ts:251-253`
  - body: The soft-trim collection callback records every image before head/tail truncation, but replacement later only works where a placeholder marker survives in the trimmed text. If the image marker is elided and hard-clear does not run, files are cached without transcript references.
  - confidence: 0.82
- **[P3] Gate cached-media prompt guidance:** `src/agents/system-prompt.ts:457-458`
  - body: The cached-media prompt section is emitted for every agent even though `cacheMedia` defaults off and the active default pruning path cannot produce these markers. That adds token overhead and tells agents to look for files that do not exist.
  - confidence: 0.9
