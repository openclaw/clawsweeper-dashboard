## Summary

Keep open: the feature direction may be useful, but this PR is not mergeable as-is because it targets the stale pruning implementation, introduces persistent-media retention concerns, and still lacks real after-fix behavior proof.

## What This Changes

The branch adds an opt-in context-pruning media cache, cached-media markers, prompt guidance, image marker hydration, config schema/baseline updates, and unit tests.

## Best Possible Solution

Implement the opt-in feature in the active pi-hooks context-pruning path, reuse existing atomic media-store and cleanup patterns, gate prompt text on effective config, and prove cached image rehydration in a live session.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main loads pi-hooks context pruning and strips images, while this PR implements caching under pi-extensions with old imports, so the new option would not affect the active runtime path.

## Solution Assessment

No. Disk caching may be a reasonable opt-in direction, but the current patch should be ported to pi-hooks with bounded retention, atomic writes, prompt gating, and real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Port caching to the active pruning hook:** `src/agents/pi-extensions/context-pruning/extension.ts:1-3`
  - body: Current main loads context pruning from `src/agents/pi-hooks/context-pruning`, and the `pi-extensions` directory is gone. Adding the async cache/write path under `pi-extensions` with old package imports means enabling `cacheMedia` will not change the active runtime behavior.
  - confidence: 0.94
- **[P2] Add a bounded retention path before persisting media:** `src/media/media-cache.ts:46-47`
  - body: `cacheMediaToDisk` writes every pruned media block under the OpenClaw config directory, but this module has no wired TTL cleanup or size bound. Long sessions can retain sensitive media indefinitely even though pruning previously discarded it.
  - confidence: 0.88
- **[P2] Write cached media atomically:** `src/media/media-cache.ts:38-47`
  - body: The cache path is checked with `fs.access()` and then written directly to the final filename. A concurrent prune can observe the destination as present and emit a marker to a partially written file; use the existing sibling-temp/rename pattern instead.
  - confidence: 0.84
- **[P2] Normalize MIME before deriving cached-media kind:** `src/media/media-cache.ts:67-68`
  - body: `mediaCacheKind` passes raw MIME text to `mediaKindFromMime`, whose checks are case-sensitive and do not strip parameters. Values such as `IMAGE/PNG` or `image/png; charset=binary` become `document`, so image hydration will skip cached images.
  - confidence: 0.86
- **[P2] Keep only cached-marker tokens through hard clear:** `src/agents/pi-extensions/context-pruning/pruner.ts:16-28`
  - body: `extractCachedMediaMarkers` preserves the entire text block whenever it contains a cached-media marker. If the marker is embedded in a soft-trimmed block, hard-clear can keep head/tail payload and trim notes, defeating the intended context reduction.
  - confidence: 0.82
- **[P3] Gate cached-media prompt guidance:** `src/agents/system-prompt.ts:457-458`
  - body: The cached-media section is emitted for every agent even though `cacheMedia` defaults off and the default pruning path cannot produce these markers. That adds token overhead and tells agents to look for files that do not exist.
  - confidence: 0.9
