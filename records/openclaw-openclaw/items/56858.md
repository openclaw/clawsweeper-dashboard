## Summary

Keep this PR open, but do not merge it as-is. The feature is still absent from current main and the source work is useful, but the patch targets the stale `pi-extensions` context-pruning path while current main loads `pi-hooks`, and it still needs explicit retention, atomic-write, prompt-gating, docs/changelog, and security review before it is merge-ready.

## What This Changes

This PR adds an opt-in `agents.defaults.contextPruning.cacheMedia` setting that writes pruned media to a local cache, replaces stripped image placeholders with `[media cached: ...]` markers, teaches image reference detection to hydrate cached image paths, and updates config schema/generated baseline/tests.

## Best Possible Solution

Port the feature to `src/agents/pi-hooks/context-pruning`, keep it opt-in by default, add bounded cleanup and atomic cache writes before exposing paths, gate system-prompt guidance on the effective setting, update docs/generated config/changelog, and validate both pruning and cached-image hydration on the current runtime path.

## Reproduction Assessment

Yes. On current main, a prunable tool result containing an image is reduced to `[image removed during context pruning]`, and the active tests assert that behavior; `cacheMedia` is not accepted by the current context-pruning settings/schema.

## Solution Assessment

No, not as currently proposed. Disk caching may be a reasonable opt-in solution, but the branch needs a port to the active hook path plus retention, atomic-write, prompt-gating, docs, and changelog work before it is the narrowest maintainable fix.
