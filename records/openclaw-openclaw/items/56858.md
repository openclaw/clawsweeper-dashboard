## Summary

Keep open; the feature may be useful, but the current PR is not merge-ready because it targets the obsolete `pi-extensions` pruning path while current main loads `pi-hooks`, lacks real behavior proof, and still has media-retention and cache-write correctness issues.

## What This Changes

The branch adds an opt-in `agents.defaults.contextPruning.cacheMedia` setting, a disk media-cache module, cached-media markers, prompt guidance, image marker hydration, generated config docs, and tests.

## Best Possible Solution

Port the feature to the active `src/agents/pi-hooks/context-pruning` implementation, keep it opt-in, reuse existing media-store cleanup and atomic-write patterns, gate prompt text on effective config, and validate with focused pruning plus media-hydration proof.

## Reproduction Assessment

Yes. Source inspection of current main shows image tool-result blocks are still converted to `[image removed during context pruning]`, and the proposed `cacheMedia` setting is absent from the active `pi-hooks` settings path.

## Solution Assessment

No. Disk caching may be a reasonable opt-in direction, but this PR currently targets the wrong runtime path and needs retention, atomic-write, marker-preservation, prompt-gating, and real behavior proof before it is the narrowest maintainable solution.

  - body: The callback records every image before head/tail trimming, but hard-clear only re-emits placeholders by scanning the already-trimmed text. If an image marker fell into the elided middle, the file is written with no `[media cached:]` slot, making it unreachable.
  - confidence: 0.86

  - body: `extractCachedMediaMarkers` pushes the entire text block when it contains a cached-media marker. After a prior soft-trim pass, that can preserve large head/tail payload and trim notes through hard-clear, preventing the message from shrinking as intended.
  - confidence: 0.82
- **[P2] Bound cached media retention:** `src/media/media-cache.ts:46-47`
  - body: `cacheMediaToDisk` writes every pruned media file under the OpenClaw config directory, but the module has no TTL cleanup hook or size bound. Long sessions can retain sensitive media indefinitely even though pruning previously discarded it.
  - confidence: 0.88
- **[P2] Write cache files atomically:** `src/media/media-cache.ts:38-47`
  - body: The cache path is checked with `fs.access()` and then written directly to the final filename. A concurrent prune can observe the destination as present and emit a marker to a partially written file; write a sibling temp file and rename it into place.
  - confidence: 0.84
- **[P3] Gate cached-media prompt guidance:** `src/agents/system-prompt.ts:457-458`
  - body: This prompt section is emitted for every agent, but `cacheMedia` defaults off and the default pruning path cannot produce these markers. It adds token overhead and tells agents to look for files that do not exist.
