## Summary

Keep open: the feature idea is plausible, but the current PR is not mergeable because it implements the cache path on the stale `pi-extensions` pruning files instead of current main's active `pi-hooks` path, lacks real multi-turn proof, and introduces persistent media caching without a bounded atomic storage contract.

## What This Changes

The PR adds an opt-in `cacheMedia` context-pruning setting, disk media cache, cached-media transcript markers, image marker hydration, prompt guidance, config baseline updates, and unit tests.

## Best Possible Solution

Rework the feature on the active `pi-hooks` pruning path, reuse or match the bounded atomic media-store contract, gate prompt text on effective config, update config docs/tests, and provide redacted live-session proof.

## Reproduction Assessment

Yes for the PR defects. Source inspection shows current main loads `src/agents/pi-hooks/context-pruning`, while this PR implements the cache/write path under `src/agents/pi-extensions/context-pruning`, so enabling `cacheMedia` would not exercise the new path after rebasing.

## Solution Assessment

No. Disk caching may be a reasonable opt-in direction, but this patch should be reworked around the active pruning implementation and existing media-store safety patterns before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Port caching to the active pruning hook:** `src/agents/pi-extensions/context-pruning/extension.ts:1-3`
  - body: Current main wires context pruning through `src/agents/pi-hooks/context-pruning`, but this patch adds the cache/write path under stale `pi-extensions` files with old Pi package imports. The new option would not affect the active runtime path after rebasing.
  - confidence: 0.96
- **[P2] Bound cached media retention:** `src/media/media-cache.ts:44-47`
  - body: `cacheMediaToDisk` writes every pruned media block under the OpenClaw config directory without a wired TTL cleanup or size bound. Long sessions can retain sensitive media indefinitely even though pruning previously discarded it.
  - confidence: 0.91
- **[P2] Write cached files atomically:** `src/media/media-cache.ts:39-47`
  - body: The helper checks for an existing destination and then writes directly to the final cache path. A concurrent prune can observe that path as present and emit a marker to a partially written file.
  - confidence: 0.88
- **[P2] Normalize MIME before classifying cached media:** `src/media/media-cache.ts:64-65`
  - body: `mediaCacheKind` passes raw MIME text to `mediaKindFromMime`, whose checks are case-sensitive and do not strip parameters. Values such as `IMAGE/PNG` or `image/png; charset=binary` become `document`, so cached image hydration skips them.
  - confidence: 0.86
- **[P2] Track only surviving soft-trim marker refs:** `src/agents/pi-extensions/context-pruning/pruner.ts:458-465`
  - body: The soft-trim callback records every image before head/tail truncation, but replacement later only has marker slots that survived the trim. If a middle marker is elided, cache writes become unreachable or later surviving markers can be paired with the wrong ref.
  - confidence: 0.83
- **[P2] Preserve only cache marker tokens during hard clear:** `src/agents/pi-extensions/context-pruning/pruner.ts:21-29`
  - body: `extractCachedMediaMarkers` keeps the whole text block whenever it contains a cached-media marker. After a prior soft-trim plus cache pass, hard-clear can retain large payload text and trim notes, defeating the intended context reduction.
  - confidence: 0.84
