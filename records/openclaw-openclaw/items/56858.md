## Summary

Keep open: the feature is plausible, but this PR is not mergeable as-is because the cache path is implemented under a stale/non-current pruning location, persistent media retention needs security/product judgment, and the contributor has not provided real multi-turn behavior proof.

## What This Changes

Adds an opt-in `agents.defaults.contextPruning.cacheMedia` setting, disk media-cache helper, cached-media pruning markers, image rehydration, prompt text, config baseline updates, and unit tests.

## Best Possible Solution

Rework the feature on the active `src/agents/pi-hooks/context-pruning` path, use or match `src/media/store.ts` for bounded atomic storage and cleanup, gate prompt text on effective config, update docs/tests, and provide redacted live-session proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows current main loads `pi-hooks` while this PR implements the cache path under stale `pi-extensions`, and the cache helper writes directly to final paths. I did not run a live session in this read-only review.

## Solution Assessment

No. Disk caching may be a reasonable opt-in direction, but this patch should be reworked around the active pruning implementation and existing media-store safety patterns before it is maintainable.

Full review comments:

- **[P2] Port caching to the active pruning hook:** `src/agents/pi-extensions/context-pruning/extension.ts:1-3`
  - body: Current main wires context pruning through `src/agents/pi-hooks/context-pruning` and no `src/agents/pi-extensions` directory exists in the checkout. This PR adds the cache/write path under stale `pi-extensions` files with old Pi package imports, so after rebasing `cacheMedia` would not exercise the new behavior.
  - confidence: 0.97
- **[P2] Bound cached media retention:** `src/media/media-cache.ts:46-47`
  - body: `cacheMediaToDisk` writes every pruned media block under the OpenClaw config directory without a max size, TTL cleanup call, or lifecycle owner. Long sessions can retain sensitive media indefinitely even though pruning previously discarded it.
  - confidence: 0.92
- **[P2] Write cached files atomically:** `src/media/media-cache.ts:39-47`
  - body: The helper checks for an existing destination and then writes directly to the final cache path. A concurrent prune or interrupted write can leave a marker pointing at a partially written cached media file.
  - confidence: 0.9
- **[P2] Normalize MIME before classifying cached media:** `src/media/media-cache.ts:67-68`
  - body: `mediaCacheKind` passes raw MIME text to `mediaKindFromMime`, whose checks are case-sensitive and do not strip parameters. Values such as `IMAGE/PNG` or `image/png; charset=binary` become `document`, so cached image hydration skips them.
  - confidence: 0.87
- **[P2] Track only marker refs that survive soft trim:** `src/agents/pi-extensions/context-pruning/pruner.ts:471-480`
  - body: The soft-trim callback records every image before head/tail truncation is applied, but replacement later only has marker slots that survived the trim. If a middle marker is elided, cache writes become unreachable or later surviving markers can be paired with the wrong ref.
  - confidence: 0.84
- **[P2] Preserve only cached-media marker tokens:** `src/agents/pi-extensions/context-pruning/pruner.ts:16-27`
  - body: `extractCachedMediaMarkers` keeps the whole text block whenever it contains a cached-media marker. After a prior soft-trim plus cache pass, those blocks can still contain large head/tail payload text and trim notes, defeating the intended context reduction.
  - confidence: 0.83
- **[P3] Gate cached-media prompt text on the setting:** `src/agents/system-prompt.ts:457-458`
  - body: The new `## Cached Media` system-prompt section is appended for every agent even when `cacheMedia` is disabled. That adds misleading instructions and token overhead for the default behavior where no cached-media markers will exist.
  - confidence: 0.82
