## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested media-cache behavior, yet the patch is stale against the live context-pruning implementation and needs maintainer review for media retention, atomic writes, prompt gating, and release-note policy before it is safe to land.

## What This Changes

This PR adds an opt-in `agents.defaults.contextPruning.cacheMedia` setting that writes pruned media blocks to a local cache, replaces stripped image placeholders with `[media cached: ...]` markers, teaches image hydration to read those markers, and updates config schema/generated config/test coverage.

## Best Possible Solution

Port the feature onto the live `pi-hooks` context-pruning path, keep it opt-in by default, add bounded cache cleanup and atomic writes, gate system-prompt guidance on the effective config, update docs/generated config/changelog, and validate with focused context-pruning plus media-hydration tests before maintainer review.

## Reproduction Assessment

Yes. On current main, a context-pruned image tool result reproducibly becomes `[image removed during context pruning]`, and the current tests assert that behavior; enabling the proposed `cacheMedia` key is not possible because the schema/settings do not contain it.

## Solution Assessment

No, not as currently proposed against main. Disk caching may be a reasonable opt-in design, but this PR needs to be ported to the active hook path and tightened around retention, atomic writes, prompt gating, and changelog policy before it is the narrowest maintainable solution.

  - body: The PR says TTL cleanup mitigates disk accumulation, but the added cache module shown in the diff only writes content-addressed files and returns paths. With `cacheMedia` enabled, long sessions can keep every pruned media file under the cache directory indefinitely unless a cleanup policy and hook are added.
  - confidence: 0.82
- **[P2] Write cached files atomically before returning paths:** `src/media/media-cache.ts:40-47`
  - body: The cache path is checked with `access()` and then written directly. A concurrent pruning pass can observe the destination after it is created but before the write completes, return that path, and insert a marker pointing at a partially-written file; use a same-directory temp file plus rename or exclusive create before exposing the path.
  - confidence: 0.76

  - body: This section is added to every agent prompt even though `cacheMedia` defaults to false. That adds tokens and tells agents to look for markers that cannot appear under the default configuration; include it only when the effective pruning settings can emit cached-media markers.
