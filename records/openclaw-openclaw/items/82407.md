## Summary

Current main is not closeable: the existing WebControl compaction indicator is wired to streamed compaction events, but manual `/compact` runs as an awaited local RPC and no current UI state is set while that request is in flight.

## What This Changes

The issue requests a visible WebControl UI in-progress indicator while manual `/compact` compaction is running.

## Best Possible Solution

Reuse the existing `compactionStatus` indicator for manual `/compact`, either host-side around the awaited command or via a tightly scoped manual compaction event, without changing compaction semantics or harness behavior.

## Reproduction Assessment

Yes at source level: `/compact` awaits `sessions.compact`, and `sessions.compact` awaits manual compaction before responding, while no current-main code sets the existing UI compaction status for that awaited path. I did not run a live browser session.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing indicator state for manual compaction start, completion, skip, and failure, rather than adding a separate banner system.
