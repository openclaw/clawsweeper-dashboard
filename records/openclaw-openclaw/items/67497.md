## Summary

Keep open. Current main still routes Feishu setup/status through the probe path that constructs the default `@larksuiteoapi/node-sdk` HTTP client without a quiet logger, and the paired same-author PR #67734 remains the open implementation candidate with closing syntax for this issue.

## What This Changes

The issue reports that Feishu setup/status probing prints SDK `[info]: [ 'client ready' ]` lines in the interactive setup wizard and asks OpenClaw to silence SDK info logs while preserving useful Feishu status output.

## Best Possible Solution

Land or revise #67734, or an equivalent Feishu-owned fix, so setup/status probes create a quiet/custom SDK logger path without polluting the shared runtime client cache, preserve `Feishu: connected as ...`, and add focused regression coverage in the Feishu client/probe/setup-surface tests.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-backed: channel setup status calls Feishu `resolveStatusLines`, which calls `probeFeishu`, which constructs the SDK client whose pinned constructor logs `client ready` at info level by default.

## Solution Assessment

Yes. A narrow Feishu plugin fix that passes a quiet/custom SDK logger or lower logger level only for setup/status probing is the maintainable direction; suppressing all wizard output or changing core channel status rendering would be broader than necessary.
