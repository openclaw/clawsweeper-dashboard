## Summary

Keep open. Current main and the latest release still route Feishu setup/status probing through the default SDK client without logger suppression, and the same-author implementation PR at https://github.com/openclaw/openclaw/pull/67734 is still open with closing syntax for this issue.

## What This Changes

The issue requests silencing raw Feishu/Lark SDK info logs during setup/status probing while preserving the user-facing Feishu connection status line.

## Best Possible Solution

Implement or land a Feishu plugin probe path that uses a no-op SDK logger or equivalent quiet client for setup/status probes, preserves `Feishu: connected as ...`, and adds focused regression coverage.

## Reproduction Assessment

Yes. Current main's setup status invokes probeFeishu, probeFeishu creates the default SDK client, and @larksuiteoapi/node-sdk 1.64.0 defaults the client logger to info and emits `client ready`; I did not run a credentialed wizard session.

## Solution Assessment

Yes. A Feishu-owned probe-only quiet/custom SDK logger path is the narrow maintainable fix; suppressing core wizard output or changing global runtime Feishu logging would be broader than necessary.
