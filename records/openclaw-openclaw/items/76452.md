## Summary

Keep open. The underlying archive search bug is valid and the PR addresses the main indexing/stem-resolution gap, but the diff also changes the stored memory path for every agent-scoped live session transcript, which can leave duplicate or stale session hits after targeted sync.

## What This Changes

The PR indexes reset/deleted transcript archives, adds owner-aware archive hit resolution for `memory_search` visibility filtering, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Keep the archive indexing and archive-stem visibility fix, but preserve existing live transcript path identity or add an explicit index migration/full reindex before rebasing and validating the memory session lanes.

## Reproduction Assessment

Yes. Source inspection on current main shows reset/deleted archives are listed as usage-counted files but are emptied by `buildSessionEntry`, and archive hit paths are dropped by the current visibility stem extractor.

## Solution Assessment

No. The archive fix is directionally right, but changing the index path for all live agent session transcripts is not the narrowest maintainable solution because targeted sync does not clean the old path rows.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve live session index paths:** `packages/memory-host-sdk/src/host/session-files.ts:319-322`
  - body: `sessionPathForFile` now inserts the agent id for every transcript under `agents/<id>/sessions`, so an existing live transcript moves from `sessions/<basename>.jsonl` to `sessions/<agent>/<basename>.jsonl`. Targeted session sync does not prune unrelated session rows, so the first incremental update after this ships can leave the old live-session row searchable alongside the new one until a full reindex. Please keep the legacy path for non-archive transcripts, or add an explicit migration/full-reindex trigger for the path-format change.
  - confidence: 0.83
