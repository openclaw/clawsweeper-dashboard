## Summary

Keep open. Current main still has the reported failure shape for a fresh internal/local probe under `gateway.auth.mode="none"`: the auth layer accepts the connection as `method: "none"`, but that method is not treated as shared/internal auth, the probe intentionally sends no device identity unless one is already cached, and the non-Control-UI path can still close with `1008: device identity required`. The related #69066 RFC is relevant and still open rather than a shipped fix.

## What This Changes

The issue asks for Gateway-owned internal RPC probes and exec approval followup dispatches to keep working when `gateway.auth.mode` is `none`.

## Best Possible Solution

Add a narrowly scoped internal-service identity path, or an equivalently constrained loopback-only Gateway-owned caller exception, for internal probe/followup/backend RPC clients under `gateway.auth.mode="none"`. Keep device identity fail-closed for remote clients, browser-origin clients, node-role clients, and non-internal operator clients, with regression coverage for the probe and approval/followup paths.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="none"`, no token/password, and a fresh CLI with no cached operator device token follows a clear source path: local probe auth resolves empty, `probeGateway` disables device identity, and the non-Control-UI operator connection reaches the 1008 `device identity required` branch.

## Solution Assessment

No current fix is present on main. The token-mode workaround is operationally valid, but the maintainable fix should be an internal-only service identity or tightly scoped internal caller path rather than widening `auth.mode="none"` globally.
