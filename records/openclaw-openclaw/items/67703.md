## Summary

Keep open. The merged backend self-pairing fix covers the backend/followup part of the report, but current main still leaves first-time `openclaw-probe` clients outside the auth-none bypass, so the reported probe-side `1008: device identity required` path remains source-reproducible.

## What This Changes

The issue asks for `gateway.auth.mode=none` to keep internal RPC probes and exec-approval followup dispatches working without a device identity rejection.

## Best Possible Solution

Keep the issue open until a scoped probe/internal-service path covers auth-none probes and followups while preserving fail-closed behavior for browser-origin, remote, node-role, and ordinary device-less clients.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: auth-none probe auth resolves empty, first-time `probeGateway` sends no device identity, probe mode is outside the backend self-pairing bypass, and the missing-device branch closes with `1008`.

## Solution Assessment

No complete fix is on main. The backend PR is a good partial fix, but the remaining probe/internal caller path should be handled through a narrowly scoped auth policy rather than treating `method: "none"` as shared auth globally.
