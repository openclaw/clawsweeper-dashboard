## Summary

Keep open. The merged backend self-pairing fix covers the followup/backend part of this report, but current main still leaves first-time probe-mode Gateway clients outside the auth-none bypass path, so the reported probe side is not fully implemented.

## What This Changes

Allow `gateway.auth.mode="none"` internal Gateway RPC probe and followup clients to connect without `1008: device identity required`.

## Best Possible Solution

Keep the issue open until a scoped probe/internal-service path covers auth-none probes and followups while preserving fail-closed behavior for browser-origin, remote, node-role, and ordinary device-less clients.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: auth-none probe auth resolves empty, first-time `probeGateway` sends no device identity, probe mode is outside the backend self-pairing bypass, and the missing-device branch still closes with `1008`.

## Solution Assessment

No complete fix is on main. The merged backend self-pairing change is a good partial fix, but the remaining probe-mode path should be handled through a narrowly scoped internal/probe identity policy rather than treating `method: "none"` as shared auth globally.
