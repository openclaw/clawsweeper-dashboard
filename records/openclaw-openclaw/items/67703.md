## Summary

Keep open. Current main still has the reported Gateway auth gap for first-time local probe clients under `gateway.auth.mode="none"`: auth succeeds as `method: "none"`, but that is not treated as shared auth, the probe sends no device identity without an existing cached operator token, and the missing-device branch still closes with `1008: device identity required`.

## What This Changes

The issue requests that `gateway.auth.mode="none"` still allow Gateway-owned internal RPC probes and approval followup clients without external authentication.

## Best Possible Solution

Add a narrowly scoped internal service identity path, or an equivalently constrained loopback-only Gateway-owned caller exception, for probe/followup/backend RPC clients while keeping device identity fail-closed for ordinary remote, browser-origin, node-role, and operator CLI clients.

## Reproduction Assessment

Yes. A first-time local probe with `gateway.auth.mode="none"`, no token/password, and no cached operator device token follows a deterministic source path to `reject-device-required`; I did not run a live Gateway because this review was read-only.

## Solution Assessment

No current fix is present on main. The token-mode workaround is operationally valid, but the maintainable fix should be internal-only service identity or a tightly scoped Gateway-owned caller path rather than widening auth-none behavior globally.
