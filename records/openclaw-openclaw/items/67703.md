## Summary

Keep open. Current main still has the reported gateway auth gap: `auth.mode="none"` authenticates the WebSocket at the generic auth layer, but local probes/followup-style backend clients still get no shared credential, do not count as shared auth, and can still hit the `1008: device identity required` branch when no cached operator device identity exists. The linked #69066 RFC is useful design context, but it is open discussion, not a shipped fix.

## What This Changes

Make `gateway.auth.mode=none` allow Gateway-owned internal RPC probes and followup dispatches to work without requiring external shared-secret auth or a pre-paired device identity.

## Best Possible Solution

Add a narrowly scoped internal service identity path, or an equivalently constrained loopback-only Gateway-owned caller exception, for internal probe/followup/backend RPC clients under `gateway.auth.mode="none"`. Keep device identity fail-closed for remote clients, browser-origin clients, node-role clients, and ordinary operator CLI clients, with regression coverage for probe and approval/followup behavior.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="none"`, no token/password, and no cached operator device token follows a clear source path: local probe auth resolves empty, `probeGateway` sends no device identity, `method: "none"` is not `sharedAuthOk`, and the missing-device branch closes with `1008: device identity required`.

## Solution Assessment

No current fix is present on main. The token-mode workaround is operationally valid, but the maintainable fix should be an internal-only service identity or tightly scoped Gateway-owned caller path rather than widening auth-none behavior globally.
