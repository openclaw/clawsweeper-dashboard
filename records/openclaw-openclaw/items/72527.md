## Summary

Keep open. Current main still emits newer-config guidance only after successful config validation, while schema-invalid downgrade configs can fail through generic invalid-config or snapshot paths; the linked closing PR https://github.com/openclaw/openclaw/pull/72535 is still open, unmerged, and conflicting.

## What This Changes

The issue asks OpenClaw to warn or repair newer-version openclaw.json files during downgrade so older binaries do not fail with generic invalid-config errors.

## Best Possible Solution

Keep this as the canonical downgrade-config issue and revise, land, or replace https://github.com/openclaw/openclaw/pull/72535 so invalid-config and snapshot diagnostics share downgrade remediation with the valid newer-config warning path.

## Reproduction Assessment

Yes, by source inspection. A future-stamped config with unknown keys or changed value shapes fails validation in loadConfig or readConfigSnapshot before warnIfConfigFromFuture can emit newer-version guidance.

## Solution Assessment

No for closing as implemented. Updating only the valid-config warning path is too narrow; invalid-config and snapshot diagnostics need the same downgrade remediation, while doctor repair claims should stay precise about unknown-key stripping versus manual value-format fixes.
