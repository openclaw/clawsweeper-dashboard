## Summary

Keep this issue open. Current main still only provides partial downgrade protection: newer-version warnings are emitted after successful config validation, while the reported downgraded config can fail validation first, and the doctor repair path strips unknown keys without handling known keys whose value shape changed; PR #72535 remains the active open closing candidate.

## What This Changes

The issue requests downgrade-aware config detection and doctor recovery for `openclaw.json` files written by 2026.4.x that break when read by 2026.3.2.

## Best Possible Solution

Review and land, expand, or replace PR #72535 so newer-version guidance appears even for invalid downgraded configs and doctor either safely rewrites known downgrade-incompatible shapes or clearly enumerates the manual edits while preserving structured JSON serialization.

## Reproduction Assessment

Yes. The report gives concrete version steps and doctor errors, and current-main source inspection shows invalid downgraded config values can hit invalid-config handling before future-version warning logic runs.

## Solution Assessment

Unclear. A warning-copy-only fix would help, but the complete repair likely needs invalid-load/snapshot warning coverage and an explicit decision on which downgrade value-format rewrites doctor should perform.
