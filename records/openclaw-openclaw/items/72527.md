## Summary

Keep this issue open. The report is actionable, it is paired with open PR #72535 using closing syntax, and current main still only partially addresses the downgrade scenario: newer-version stamping exists, but schema-invalid downgrade configs can return or throw before the future-version warning/remediation path runs.

## What This Changes

The issue asks for downgrade-aware detection, warnings, and doctor/startup repair guidance when a 2026.4.x-written `openclaw.json` is read by OpenClaw 2026.3.2.

## Best Possible Solution

Keep this open until PR #72535 is reviewed, expanded, or replaced. The end state should surface future-version remediation even when config validation fails, keep JSON serialization structural, and make downgrade-incompatible value rewrites explicit enough that doctor does not silently discard user configuration.

## Reproduction Assessment

Yes. The issue gives concrete versions and validation errors, and current source inspection shows a deterministic path where a future-stamped config with schema-invalid downgrade values reaches invalid-config return/throw paths before `warnIfConfigFromFuture` runs.

## Solution Assessment

Unclear. The linked PR's warning-copy direction is useful, but the narrowest complete fix likely needs to attach future-version guidance to invalid config load/snapshot/preflight reporting or deliberately scope the PR to valid-config warnings only.
