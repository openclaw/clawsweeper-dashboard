## Summary

Keep this open. Current main still lets schema-invalid newer configs hit generic invalid-config handling before the newer-version warning/remediation path, and the linked closing PR is open, unmerged, and conflicting rather than a landed fix.

## What This Changes

The issue requests downgrade-aware diagnostics or repair when a newer `openclaw.json` contains keys or value shapes that an older OpenClaw cannot validate.

## Best Possible Solution

Keep this as the canonical downgrade-config issue and revise, land, or replace https://github.com/openclaw/openclaw/pull/72535 so invalid-config and snapshot diagnostics share downgrade remediation with the valid newer-config warning path.

## Reproduction Assessment

Yes by source inspection. A future-stamped config with unknown keys or changed value shapes fails validation in `loadConfig` or `readConfigSnapshot` before `warnIfConfigFromFuture` can emit newer-version guidance.

## Solution Assessment

No for the currently linked warning-only implementation path. The maintainable fix should surface the downgrade remediation from invalid-config and snapshot diagnostics too, while keeping doctor repair claims precise about unknown-key stripping versus manual value-format repair.
