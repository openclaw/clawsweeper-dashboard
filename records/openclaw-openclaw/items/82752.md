## Summary

Keep open: this is a focused, plausible fix for a real source-level plugin approval wait bug. Current main has the server/client approval-runtime contract, but the agent gateway helper does not pass the runtime token through to `GatewayClient`; the PR fills that missing path while preserving explicit gateway overrides, and I found no blocking correctness issue.

## What This Changes

The PR threads the operator approval runtime token through `callGatewayTool` and `callGateway` for default local `plugin.approval.*` calls and adds regression coverage for the default and explicit-gateway override paths.

## Best Possible Solution

Land a narrow fix that reuses the existing local approval-runtime token contract for default plugin approval gateway calls, preserves explicit gateway override isolation, and keeps the focused regression tests.

## Reproduction Assessment

Yes, with source-level confidence: current main records plugin approvals under one backend connection, but a later wait connection is not trusted as `internal.approvalRuntime` unless the runtime token reaches `GatewayClient`. I did not run a local repro in this read-only review.

## Solution Assessment

Yes, the proposed approach is the narrow maintainable fix because it threads an existing auth field through the missing helper layer and gates it to local `plugin.approval.*` calls with `operator.approvals` while excluding explicit gateway overrides.
