## Summary

Keep open: the branch addresses the cross-agent eviction path, but it also leaves retired auth/env-keyed app-server clients alive after rotation, which regresses the existing shared-client cleanup contract.

## What This Changes

The PR changes the Codex shared app-server client from one singleton to a Map keyed by start/auth/agent options, with a regression test and changelog entry.

## Best Possible Solution

Keep per-agent warm clients, but evict and close entries when secret-bearing auth/env/start options rotate so token reconnection and per-agent isolation both hold.

## Reproduction Assessment

Yes. Source inspection shows current main clears the singleton when agentDir changes because agentDir is part of the compatibility key, and the PR body supplies terminal proof for the intended two-agent reuse path.

## Solution Assessment

No. Pooling by the full compatibility key fixes agentDir churn, but it also turns auth/env/start-option rotations into long-lived pooled clients; the safer fix is to preserve the agent pool while actively closing stale secret/start generations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Close stale secret-keyed clients on rotation:** `extensions/codex/src/app-server/shared-client.ts:80-84`
  - body: This Map keeps one live entry for every full start-options key, including authToken and env changes. Current main closes the old shared client when that key changes, and the token-rotation test asserts the first client is destroyed; after this change, a rotated bearer token opens a new client but leaves the old websocket/process alive until global cleanup. Keep the per-agent pool, but evict or close incompatible secret/start generations.
  - confidence: 0.88
