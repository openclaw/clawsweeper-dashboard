## Summary

Keep this PR open, but it should not merge as-is because the keyed pool also keeps clients for retired auth/env keys alive, regressing the merged token-rotation cleanup behavior.

## What This Changes

The PR changes the Codex shared app-server client singleton into a keyed pool, adds a regression test and changelog entry.

## Best Possible Solution

Keep per-agent warm clients, but retain the existing cleanup semantics for auth/env/start-option rotation by splitting the pool key or evicting incompatible secret/start generations.

## Reproduction Assessment

Yes. Source inspection shows current main clears the singleton when `agentDir` changes because `agentDir` is part of `codexAppServerStartOptionsKey`, and the PR body supplies after-fix terminal proof for the intended pool behavior.

## Solution Assessment

No. Pooling by full compatibility key fixes the agent-dir eviction, but pooling secret/start-option rotations as long-lived entries is broader than the maintainable fix and regresses the existing auth-rotation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Close stale secret-keyed clients on rotation:** `extensions/codex/src/app-server/shared-client.ts:77-80`
  - body: This pools every full start-options key, including authToken and env changes. Current main intentionally tears down the old shared client when the bearer token changes; after this change, rotating `appServer.authToken` creates a new client but leaves the old websocket/process alive until global dispose, reopening the stale-token window and leaking one client per retired secret key. Keep the agent-dir pool, but evict/close entries when the secret-bearing or start-option generation changes.
  - confidence: 0.88
