## Summary

Keep open: the underlying Codex multi-agent shared-client bug is valid and the PR has sufficient real behavior proof, but the patch should not merge yet because it leaves retired auth/env-keyed app-server clients alive after token or start-option rotation.

## What This Changes

The PR replaces the Codex app-server singleton shared client with a start/auth/agent-keyed pool, adds alternating-agent regression coverage, changes the auth-token rotation expectation, and adds a changelog entry.

## Best Possible Solution

Keep keyed per-agent pooling, but close or evict stale secret/env/start-option generations so token rotation and per-agent isolation both hold.

## Reproduction Assessment

Yes, source-reproducible: current main deterministically clears the singleton when the compatibility key changes, and agentDir is part of that key. The PR body and linked issue comments add runtime proof for the two-agent reuse path, though I did not run a live OAuth/Codex overlap in this read-only review.

## Solution Assessment

No, not as written. A keyed pool is the right direction, but pooling by the full secret-bearing key without stale-entry cleanup regresses the existing token-rotation cleanup contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Close stale secret-keyed clients on rotation:** `extensions/codex/src/app-server/shared-client.ts:80-84`
  - body: This pool stores one live entry for every full compatibility key, including authToken and env changes. When a token rotates for the same websocket app-server, the patch opens a second client but leaves the old authenticated socket/process alive; current main closes that first client and has a regression test for it. Keep the per-agent pool, but evict and close stale secret/start generations.
  - confidence: 0.9
