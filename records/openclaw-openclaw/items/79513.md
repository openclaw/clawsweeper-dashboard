## Summary

Keep this PR open: it fixes a real Codex multi-agent shared-client problem, but the current head still has a blocking lifecycle/security regression because stale auth/env-keyed clients remain alive after rotation. It is also behind current main's native-auth `authProfileId: null` path, so the pool should be rebased over that behavior before merge.

## What This Changes

The PR changes the Codex shared app-server client from a single global slot to a keyed pool, adds shared-client regression coverage, and updates the changelog.

## Best Possible Solution

Land a rebased keyed-pool implementation that keeps per-agent warm clients, closes retired auth/env/start generations on rotation, and preserves current native-auth null behavior with regression tests.

## Reproduction Assessment

Yes. Current main source shows `agentDir` is part of the compatibility key and a key change clears the singleton; the PR body also provides terminal proof that the intended two-agent reuse path works after pooling.

## Solution Assessment

No. Keyed pooling is the right direction, but this branch must also evict stale secret/env/start generations and preserve current main's native-auth sentinel before it is the narrow maintainable fix.

## Review Findings

- **[P2] Close stale secret-keyed clients on rotation:** `extensions/codex/src/app-server/shared-client.ts:80-84`
  - body: This Map keeps one live entry for every full start-options key, including `authToken` and env changes. Current main closes the old shared client when that key changes, and the token-rotation test asserts the first client is destroyed; after this change, a rotated bearer token opens a new client but leaves the old websocket/process alive until global cleanup. Keep the per-agent pool, but evict or close incompatible secret/start generations.
  - confidence: 0.9

Rebase the pool over the null-sentinel path so those callers do not fail type checking or use the wrong auth context.
  - confidence: 0.84
