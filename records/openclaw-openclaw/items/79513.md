## Summary

Keep open. The reported bug is real and the PR has sufficient after-fix proof, but the current patch regresses the merged auth-rotation cleanup by retaining stale secret/start-option clients in the new pool.

## What This Changes

The branch converts the Codex shared app-server client singleton into a Map keyed by start/auth/agent options, updates shared-client tests, and adds a changelog entry.

## Best Possible Solution

Keep the keyed pool for per-agent warm clients, but evict and close stale entries when auth/env/start-option generations rotate, with tests for both agentDir reuse and token rotation cleanup.

## Reproduction Assessment

Yes, source-backed. Current main deterministically clears the singleton when agentDir changes because agentDir is part of the key, and the PR body supplies after-fix terminal proof; I did not run a live multi-agent gateway scenario.

## Solution Assessment

No. Pooling by the full key fixes agentDir eviction, but retaining secret/start-option rotations as long-lived entries is broader than the maintainable fix and weakens the existing auth-rotation cleanup contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Close stale clients when auth or start options rotate:** `extensions/codex/src/app-server/shared-client.ts:77-80`
  - body: The new Map is keyed by the full start-options key, so rotating appServer.authToken or env starts a new client but never closes the old one. Current main's tests and the merged auth-hardening PR require the old token client to be destroyed on rotation; otherwise stale bearer-token websocket/process state remains alive until global dispose.
