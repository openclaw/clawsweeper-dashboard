## Summary

Keep this PR open, but it is not merge-ready: the useful grep/no-match fix is real, yet the exit-code-only normalization would also mark Codex timeout or sandbox-denied command failures as completed and suppress channel-visible failure handling.

## What This Changes

The PR changes Codex app-server event projection so failed `commandExecution` items with numeric exit codes other than 126/127 become completed tool results, with focused tests for exit 1 and 127.

## Best Possible Solution

Land a narrower Codex-native status policy that suppresses expected negative checks without hiding timeout, sandbox, blocked, or shell-invocation failures, backed by focused projector tests and real channel proof.

## Reproduction Assessment

Yes, from source inspection: current main maps Codex `status: "failed"` command items to `lastToolError`, and the proposed diff suppresses that for numeric exits other than 126/127. I did not run a live Telegram or Discord reproduction.

## Solution Assessment

No, not as written: aligning Codex-native command status with OpenClaw exec semantics is the right direction, but the exit-code-only rule is too broad and needs a named failure taxonomy before merge.
