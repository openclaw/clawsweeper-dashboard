## Summary

Keep open. Current main and the latest release still only accept exact `gateway.controlUi.allowedOrigins` entries or the global `"*"`, and the open/closed related PRs do not supersede the requested `chrome-extension://*` policy decision.

## What This Changes

The issue requests opt-in `gateway.controlUi.allowedOrigins` support for scheme-level wildcard entries such as `chrome-extension://*` so Chrome extension WebSocket clients do not break when extension IDs change.

## Best Possible Solution

If maintainers approve the policy, add a constrained `chrome-extension://*` matcher with explicit docs plus positive and negative Gateway origin tests, while avoiding generic trailing-wildcard matching in the trust boundary.

## Reproduction Assessment

Yes, as a source-level reproduction: `Origin: chrome-extension://abcdefghijklmnop` with `allowedOrigins: ["chrome-extension://*"]` cannot pass the current exact `URL.origin` comparison. I did not run Vitest in this read-only review.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher because it is too broad for a Gateway trust boundary. A constrained Chrome-extension wildcard policy is the safer direction if maintainers approve it.
