## Summary

Keep open: current main and v2026.6.10 still only allow exact parsed origins or the global wildcard, while this issue asks for a new scheme-level wildcard policy on a Gateway browser-origin authorization boundary.

## What This Changes

The issue requests support for scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` so changing Chrome extension IDs do not require config edits.

## Best Possible Solution

Keep this open for maintainer security/product review; if approved, implement a constrained wildcard grammar centrally in `checkBrowserOrigin` with explicit allow/reject coverage and docs updates.

## Reproduction Assessment

Yes, at source level. Current main only checks the global `*` and exact parsed origins in `checkBrowserOrigin`, and Node serializes `chrome-extension://<id>` origins as `null`.

## Solution Assessment

Unclear. `checkBrowserOrigin` is the right implementation boundary if maintainers approve the behavior, but the wildcard policy is broader than the exact-origin fixes in related PRs and needs security review first.
