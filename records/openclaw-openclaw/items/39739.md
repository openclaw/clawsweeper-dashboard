## Summary

Keep open: current main and the latest release still only allow global `*` or exact parsed-origin matches, and the open related PRs address exact opaque-origin matching rather than the requested scheme-wildcard policy. This changes a Gateway browser-origin trust boundary, so it needs maintainer security/product approval before implementation.

## What This Changes

Request support for opt-in scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

If maintainers approve the policy, add a constrained `chrome-extension://*` matcher with negative tests, docs, and security-audit behavior aligned to the Gateway origin trust model.

## Reproduction Assessment

Yes, as a source-level check: current main compares the allowlist only to global `*` or `parsedOrigin.origin`, and Node reports Chrome-extension origins with `origin: "null"`. I did not run tests because this review is read-only.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher, because it is broader than the Chrome-extension use case at a Gateway browser-origin trust boundary. A constrained `chrome-extension://*` policy is the safer path if maintainers approve it.
