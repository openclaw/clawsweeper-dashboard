## Summary

Keep open: current main and v2026.5.22 still only support the global `*` wildcard or exact parsed origins, and scheme-level extension wildcards remain a Gateway browser-origin security/product decision.

## What This Changes

Request support for `chrome-extension://*`-style scheme-level wildcard entries in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Maintainers should explicitly approve or reject `chrome-extension://*`; if approved, implement a constrained extension-origin matcher while preserving exact/global behavior elsewhere with unit, handshake, docs, and security-audit coverage.

## Reproduction Assessment

Yes. A source-level check shows `checkBrowserOrigin` matches only global `*` or parsed `URL.origin`, and Node serializes `chrome-extension://abcdefghijklmnop` with origin `null`, so `chrome-extension://*` cannot match current main.

## Solution Assessment

No for the generic trailing-`*` matcher proposed in the issue body; it is too broad for this trust boundary. A Chrome-extension-scoped wildcard policy is the safer shape if maintainers approve the feature.
