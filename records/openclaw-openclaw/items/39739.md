## Summary

Keep open: current main and v2026.5.20 still do not support `chrome-extension://*` or other scheme-level wildcard allowlist entries. The related open PRs address exact opaque/custom origins, while the PR that specifically implemented this wildcard was closed unmerged, so the remaining work is a Gateway security/product policy decision.

## What This Changes

Support scheme-level wildcard values such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` so Chrome extension origin ID changes do not require repeated exact allowlist edits.

## Best Possible Solution

If maintainers approve the policy, add a constrained `chrome-extension://*` matcher, not generic prefix wildcards, while preserving exact allowlist semantics, rejecting literal `Origin: null`, and updating unit tests, Gateway handshake coverage, docs, and security-audit treatment.

## Reproduction Assessment

Yes, as a source-level check: current main accepts only the global `*` or exact parsed-origin matches, and there is no `chrome-extension://*` branch. The Node URL contract also shows Chrome-extension origins serialize to `origin: "null"`, which keeps the unsupported path visible.

## Solution Assessment

No, the generic trailing-`*` prefix matcher proposed in the issue body is not the best shape for this trust boundary. A narrower Chrome-extension-only wildcard policy is safer if maintainers approve the feature.
