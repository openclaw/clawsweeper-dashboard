## Summary

Keep open. Current main still only accepts global `*` or exact parsed origins, Chrome extension origins normalize to `origin: "null"` in Node, and the related PRs either cover exact custom-scheme matching or are closed unmerged wildcard attempts, so this remains a Gateway origin-trust policy change needing maintainer/security approval.

## What This Changes

Request support for opt-in scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` so Chrome extension clients do not need config updates when their extension ID changes.

## Best Possible Solution

If maintainers approve the policy, add a constrained Chrome-extension-only wildcard matcher with explicit docs, audit/security review, and negative tests while keeping exact custom-scheme normalization separate.

## Reproduction Assessment

Yes, as a source-level reproduction: `Origin: chrome-extension://abcdefghijklmnop` with `allowedOrigins: ["chrome-extension://*"]` cannot pass current `URL.origin` exact matching. I did not run Vitest in this read-only review.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher because it is too broad for a Gateway trust boundary and still depends on handling opaque URL origins. A constrained `chrome-extension://*` policy with focused docs and tests is the safer direction if maintainers approve it.
