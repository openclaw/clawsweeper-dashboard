## Summary

Keep open. Current main still only accepts the global `*` or exact normalized `URL.origin`, while the requested `chrome-extension://*` policy would broaden Gateway browser-origin trust semantics.

## What This Changes

Support scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Keep this open for Gateway security/product review; if approved, implement a scheme-specific `chrome-extension://*` matcher with docs and negative tests rather than generic prefix matching.

## Reproduction Assessment

Yes, as a source-level reproduction: `Origin: chrome-extension://abcdefghijklmnop` with `allowedOrigins: ["chrome-extension://*"]` cannot pass the current exact `URL.origin` comparison. I did not run Vitest in this read-only review.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher. A constrained Chrome-extension-only policy may be maintainable, but it needs explicit maintainer approval because this is a Gateway browser-origin trust decision.
