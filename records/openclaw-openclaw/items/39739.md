## Summary

Keep open: current main still has no `chrome-extension://*` or scheme-level wildcard support, and the remaining decision is a security-sensitive Gateway origin policy choice rather than a stale or already-implemented request. The related open PRs address exact non-standard origins, while the only PR that specifically implemented this wildcard is closed unmerged.

## What This Changes

The issue requests support for scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

If maintainers approve the policy, add a constrained `chrome-extension://*` matcher, not generic prefix wildcards, with unit tests, Gateway handshake coverage, docs, and security-audit treatment.

## Reproduction Assessment

Yes, as a source-level check: current main accepts only the global `*` or exact parsed-origin matches, and there is no `chrome-extension://*` branch. The Node URL contract also shows Chrome-extension origins serialize to `origin: "null"`, which keeps the unsupported path visible.

## Solution Assessment

No, the generic trailing-`*` prefix matcher proposed in the issue body is not the best shape for this trust boundary. A narrower Chrome-extension-only wildcard policy is safer if maintainers approve the feature.
