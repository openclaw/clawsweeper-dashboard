## Summary

Keep open. Current main still has no `chrome-extension://*` or scheme-wildcard allowlist path, and the remaining request is a Gateway origin-trust policy change that needs explicit security/product approval rather than stale cleanup.

## What This Changes

Support scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` for browser-origin Gateway connections.

## Best Possible Solution

If maintainers approve the policy, add a constrained Chrome-extension-only wildcard matcher with explicit docs and negative tests, while keeping exact opaque-origin fixes separate.

## Reproduction Assessment

Yes, as a source-level reproduction: `Origin: chrome-extension://abcdefghijklmnop` with `allowedOrigins: ["chrome-extension://*"]` cannot pass the current exact `URL.origin` comparison. I did not run Vitest in this read-only review.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher: it is too broad for a Gateway trust boundary and would not address Chrome extension origins without first handling `URL.origin === "null"`. A constrained `chrome-extension://*` policy with focused tests is the safer direction if maintainers approve it.
