## Summary

Keep open: current main still accepts only the global `*` wildcard or exact parsed origins for `gateway.controlUi.allowedOrigins`, while this issue asks for a new scheme-level wildcard policy. The related open PRs cover exact/custom-scheme origins, not the requested `chrome-extension://*` behavior, and the closed wildcard PR did not merge.

## What This Changes

Request support for scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` for Chrome extension Gateway clients with changing extension IDs.

## Best Possible Solution

If maintainers approve the policy, add a constrained `chrome-extension://*` matcher with negative tests, Gateway handshake coverage, docs, and security-audit treatment rather than a generic prefix wildcard.

## Reproduction Assessment

Yes, as a source-level check: current main accepts only global `*` or exact parsed-origin matches, and there is no `chrome-extension://*` branch. I did not run tests because this review is read-only and source inspection is sufficient for the unsupported policy.

## Solution Assessment

No, the generic trailing-`*` prefix matcher proposed in the issue body is not the best shape for this trust boundary. A narrower Chrome-extension-only wildcard policy is safer if maintainers approve the feature.
