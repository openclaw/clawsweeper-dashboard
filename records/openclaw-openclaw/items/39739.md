- `impact:security`: Scheme-level wildcards would authorize a larger class of browser applications to cross the gateway origin check.

## Summary

Current main still supports only exact normalized origins or the global `*`, and no merged PR implements scheme wildcards. Exact custom-scheme fixes remain open, but they deliberately do not resolve whether every origin under a scheme should be trusted, so this distinct security/product decision remains necessary.

## What This Changes

Add opt-in scheme-level wildcard matching to `gateway.controlUi.allowedOrigins`, primarily for Chrome extension origins.

## Best Possible Solution

Make an explicit maintainer security decision between retaining exact custom-scheme origins and supporting scheme wildcards; if accepted, implement a parsed scheme-only grammar with documented semantics and negative coverage across every shared matcher caller rather than generic string-prefix matching.

## Reproduction Assessment

Yes. Current main deterministically rejects `chrome-extension://*` as a scheme pattern because it recognizes only exact normalized origins and the standalone global wildcard.

## Solution Assessment

No, the proposed generic prefix matcher is not the best solution. If scheme wildcards are approved, a parsed and narrowly documented scheme-only rule is safer and avoids introducing arbitrary prefix patterns at a shared authorization boundary.

## Risks / Open Questions

- `chrome-extension://*` would trust every installed Chrome extension origin that can reach the gateway, not only the intended extension.
- The proposed generic trailing-`*` prefix loop would create an undocumented wildcard language that could also match malformed or unexpectedly broad patterns.
- Because `checkBrowserOrigin` is shared, an implementation could unintentionally broaden authenticated HTTP or MCP browser-origin access in addition to Control UI WebSockets.
