## Summary

Keep open: current main and the latest release still support only exact allowlist entries and the global wildcard for Gateway Control UI origins. The requested scheme-level wildcard would expand a browser-origin trust boundary, so it needs maintainer security/product approval before implementation.

## What This Changes

The issue requests support for allowlist patterns such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` so Chrome extension clients can connect without pinning a changing extension ID.

## Best Possible Solution

If maintainers approve the policy, add a constrained `chrome-extension://*` matcher with explicit negative tests, docs, and security-audit behavior aligned to the Gateway origin trust model.

## Reproduction Assessment

Yes, as a source-level check: current main only accepts global `*` or exact parsed-origin matches, and Node serializes `chrome-extension://<id>` with `origin: "null"`. I did not run tests because this review is read-only.

## Solution Assessment

No for the generic trailing-`*` prefix matcher in the issue body, because it would create broad wildcard semantics at a Gateway browser-origin trust boundary. A narrower `chrome-extension://*` policy with explicit tests and docs is the safer implementation if maintainers approve it.
