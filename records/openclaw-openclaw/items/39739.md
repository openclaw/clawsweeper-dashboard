## Summary

Keep open: current main and v2026.5.22 still support only the global `*` or exact parsed origins, while `chrome-extension://*` would be a new Gateway browser-origin policy decision. The related open PRs address exact non-standard origins, but the only linked PR that implemented this wildcard was closed unmerged.

## What This Changes

Request support for scheme-level wildcard allowlist values such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Approve or reject the wildcard policy explicitly; if approved, implement a constrained Chrome-extension-only matcher with focused unit, handshake, docs, and security-audit coverage.

## Reproduction Assessment

Yes, as a source-level check: current main accepts only the global `*` or exact parsed-origin matches, and Node serializes Chrome-extension origins with `origin=null`. There is no current `chrome-extension://*` branch to accept the requested pattern.

## Solution Assessment

No for the generic trailing-`*` matcher proposed in the body; that is too broad for this trust boundary. A narrower Chrome-extension-only wildcard policy is the safer shape if maintainers approve the feature.
