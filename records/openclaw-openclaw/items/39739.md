## Summary

Keep this open: current main and the latest release still accept only exact `gateway.controlUi.allowedOrigins` entries or global `*`, while this issue asks for new scheme-level wildcard semantics. The overlapping open PRs address exact opaque-origin matching, not the changing Chrome-extension-ID wildcard use case, so this still needs an explicit gateway security/product decision.

## What This Changes

Request support for scheme-level wildcard patterns such as `chrome-extension://*` in the Gateway Control UI/WebChat browser-origin allowlist.

## Best Possible Solution

Have maintainers decide the gateway origin policy first; if approved, implement a constrained `chrome-extension://*` matcher with canonical opaque-origin normalization, fail-closed validation for malformed/path/query/userinfo entries, and aligned tests, docs, and UI copy.

## Reproduction Assessment

Yes, at source level. `URL.origin` serializes `chrome-extension://<id>` as `null`, and current `checkBrowserOrigin` only compares exact parsed origins or global `*`, so `allowedOrigins: ["chrome-extension://*"]` cannot match today.

## Solution Assessment

No, not as sketched in the issue. A `startsWith` check against `parsedOrigin.origin` still sees `null`; if approved, the best fix is a constrained scheme wildcard matcher built on explicit opaque-origin normalization and fail-closed validation.
