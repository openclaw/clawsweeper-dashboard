## Summary

Keep open: current main still lacks scheme-level wildcard matching for `gateway.controlUi.allowedOrigins`, and the open exact-origin PRs do not cover the requested wildcard policy. Because this expands Gateway browser-origin allowlist semantics, maintainer security/product review is the right next step.

## What This Changes

Request support for scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` so Chrome extension origins can stay allowlisted across changing extension IDs.

## Best Possible Solution

If maintainers approve the policy, implement a constrained `chrome-extension://*` matcher with negative tests, docs, and security-audit behavior; avoid generic prefix wildcards at this trust boundary.

## Reproduction Assessment

Yes, as a source-level check: current main accepts only global `*` or exact parsed-origin matches, and there is no `chrome-extension://*` branch. I did not run tests because this is a read-only review.

## Solution Assessment

No, not the generic trailing-`*` prefix matcher from the issue body because it would create broad wildcard semantics at a Gateway browser-origin trust boundary. A narrower Chrome-extension-only policy is the safer implementation if maintainers approve it.
