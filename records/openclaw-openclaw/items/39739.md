## Summary

Keep open. Current main still lacks `chrome-extension://*` or scheme-wildcard allowlist semantics, and the remaining request is a Gateway browser-origin trust policy change rather than stale cleanup.

## What This Changes

Support opt-in `chrome-extension://*` scheme-level wildcard entries in `gateway.controlUi.allowedOrigins` so Chrome extension origins can stay allowed when extension IDs change.

## Best Possible Solution

If maintainers approve the policy, add a constrained Chrome-extension wildcard matcher with explicit docs plus positive and negative tests, while keeping generic trailing-wildcard matching out of the Gateway trust boundary.

## Reproduction Assessment

Yes, as a source-level reproduction: `Origin: chrome-extension://abcdefghijklmnop` with `allowedOrigins: ["chrome-extension://*"]` cannot pass the current exact `URL.origin` comparison. I did not run Vitest in this read-only review.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher: it is too broad for a Gateway trust boundary and does not address opaque extension origins without origin normalization. A constrained `chrome-extension://*` policy is the safer direction if maintainers approve it.
