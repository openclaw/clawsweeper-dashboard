## Summary

Keep open: current main and v2026.6.8 still only allow exact `gateway.controlUi.allowedOrigins` entries plus the global `*`, while this issue asks for a new scheme-level wildcard policy on a Gateway browser-origin authorization boundary.

## What This Changes

Support scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Approve a constrained scheme-wildcard grammar, then implement it in `src/gateway/origin-check.ts` with focused allow/reject tests for exact origins, `chrome-extension://*`, malformed inputs, literal `Origin: null`, empty-host schemes, and non-extension origins.

## Reproduction Assessment

Yes, at source level. Current main compares only `*` or exact `parsedOrigin.origin`, and a Node probe shows `chrome-extension://<id>` serializes to origin `null`, so `chrome-extension://*` cannot match today.

## Solution Assessment

No, not as sketched in the issue. A simple prefix check against `parsedOrigin.origin` would still see `null`; the maintainable path is an approved opaque-origin normalization plus constrained scheme-wildcard matcher.
