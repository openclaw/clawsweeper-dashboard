## Summary

Keep open: current main and the latest release still do not implement `chrome-extension://*` matching, and the related open PRs address exact custom-scheme origins rather than the requested wildcard policy.

## What This Changes

Requests support for `chrome-extension://*` scheme-level wildcard entries in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

If maintainers approve the policy, add a constrained `chrome-extension://*` matcher with docs plus positive and negative Gateway origin tests, while avoiding generic trailing-wildcard matching.

## Reproduction Assessment

Yes, as a source-level check: `checkBrowserOrigin` compares configured entries only to `parsedOrigin.origin`, and Node reports `chrome-extension://...` as opaque `origin: "null"`. I did not run tests in this read-only review.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher because it is too broad for a Gateway trust boundary. A constrained Chrome-extension wildcard policy is the safer direction if maintainers approve it.
