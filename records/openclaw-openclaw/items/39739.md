## Summary

Keep open. Current main still has only exact normalized-origin matching plus the global `*`, and the requested `chrome-extension://*` policy would broaden a security-sensitive Gateway browser-origin allowlist.

## What This Changes

Support scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Keep this open for explicit Gateway security/product review; if accepted, implement a constrained `chrome-extension://*` matcher with negative coverage and docs rather than a generic prefix wildcard.

## Reproduction Assessment

Yes, for the current unsupported behavior. Source inspection and Node URL behavior show Chrome extension origins normalize to `null`, while current main only checks global `*` or exact `parsedOrigin.origin`.

## Solution Assessment

No for the proposed generic trailing-`*` matcher. A scheme-specific `chrome-extension://*` path may be maintainable, but only after maintainers accept the origin-trust policy change.
