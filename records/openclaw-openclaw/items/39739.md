## Summary

Keep open: current main and the latest release still do not implement `chrome-extension://*` matching, and the linked open PRs solve adjacent exact custom-scheme origin matching rather than this wildcard policy. The requested behavior changes Gateway browser-origin trust semantics, so it needs maintainer approval before a repair PR is safe to queue.

## What This Changes

Support opt-in scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` for Chrome-extension Gateway clients.

## Best Possible Solution

If maintainers approve the policy, add a constrained Chrome-extension wildcard matcher with source-level validation, negative tests, docs, and Gateway config audit behavior aligned to the new trust model.

## Reproduction Assessment

Yes, as a source-level check: the current matcher only accepts global `*` or exact `parsedOrigin.origin`, so `chrome-extension://*` cannot match a Chrome-extension `Origin` header on current main. I did not run tests because this review is read-only.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher, because it is broader than the Chrome-extension use case at a Gateway browser-origin trust boundary. A constrained `chrome-extension://*` policy with explicit tests and docs is the safer maintainable path if maintainers approve it.
