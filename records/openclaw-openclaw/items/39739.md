## Summary

Keep open. Current `main` still accepts only the global `*` wildcard or an exact normalized origin for `gateway.controlUi.allowedOrigins`; `chrome-extension://*` is not implemented, #38290 is still an open exact-origin PR, and #39867 directly attempted this wildcard support but was closed unmerged.

## What This Changes

The issue requests scheme-level `gateway.controlUi.allowedOrigins` patterns such as `chrome-extension://*` so Chrome extension clients are not tied to a changing extension ID.

## Best Possible Solution

Keep the issue open for explicit gateway security/product review; if accepted, implement a constrained `chrome-extension://*` matcher rather than generic prefix wildcarding, preserve existing `Origin: null`, exact HTTP(S), global `*`, host-header fallback, and loopback behavior, then document the security tradeoff.

## Reproduction Assessment

Yes. Source inspection plus Node URL behavior show Chrome extension origins normalize to `"null"`, while current `checkBrowserOrigin` only matches global `*` or exact `parsedOrigin.origin`, so `chrome-extension://*` cannot be accepted on current `main`.

## Solution Assessment

No for the proposed generic trailing-`*` prefix matcher. A scheme-specific `chrome-extension://*` implementation may be the maintainable path, but only after maintainers accept the browser-origin trust broadening.
