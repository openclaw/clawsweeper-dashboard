## Summary

Keep open. Current main still only accepts the global `*` wildcard or exact normalized origins for `gateway.controlUi.allowedOrigins`; `chrome-extension://*` is not implemented. Related #38290 is still open and intentionally handles exact extension origins only, while #39867 directly attempted this wildcard support but was closed unmerged.

## What This Changes

The issue requests scheme-level wildcard support such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` so Chrome extension clients can connect despite changing extension IDs.

## Best Possible Solution

Keep the issue open for explicit gateway product/security judgment. If accepted, implement a constrained `chrome-extension://*` matcher in `src/gateway/origin-check.ts`, preserve literal `Origin: null` rejection, exact HTTP(S) matches, global `*`, host-header fallback, and local-loopback behavior, then add focused unit and WebSocket handshake tests plus docs/security guidance.

## Reproduction Assessment

Yes. Current main can be reproduced from source inspection plus Node URL behavior: `chrome-extension://...` normalizes to origin `"null"`, while `checkBrowserOrigin` only checks global `*` or exact `parsedOrigin.origin`.

## Solution Assessment

Unclear. A scheme-specific `chrome-extension://*` matcher would be safer than a generic trailing-asterisk prefix matcher, but maintainers should first decide whether this browser-origin broadening is acceptable; #38290's exact-origin approach is narrower and does not solve extension ID churn.
