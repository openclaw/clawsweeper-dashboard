## Summary

Keep this issue open. Current main and the latest release still only support the global `*` and exact parsed-origin matches, while the active related PRs intentionally leave `chrome-extension://*` as a separate Gateway security/product decision.

## What This Changes

Request support for scheme-level `gateway.controlUi.allowedOrigins` wildcard entries such as `chrome-extension://*` so changing Chrome extension IDs do not require allowlist edits.

## Best Possible Solution

Keep this as the canonical scheme-wildcard policy issue; if approved, implement a constrained Gateway origin matcher with explicit scheme scope, fail-closed tests, and docs for the new allowed-origin syntax.

## Reproduction Assessment

Yes, at source level. Current `checkBrowserOrigin()` only accepts the global `*` or exact parsed-origin matches, and Node serializes `chrome-extension://<id>` as `"null"`, so `chrome-extension://*` has no current-main match path.

## Solution Assessment

No, not as sketched. A raw `pattern.endsWith('*')` prefix check against `parsedOrigin.origin` would still see `"null"` today; the maintainable path is an explicit, constrained scheme-wildcard design after the security decision is made.
