## Summary

Keep open. Current main still only accepts the global `*` wildcard or an exact `URL.origin`, so `chrome-extension://*` is not implemented; #38290 covers exact extension origins only, and #39867 attempted this specific wildcard support but was closed unmerged.

## What This Changes

Support scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins` for browser-extension Gateway clients.

## Best Possible Solution

Keep this open for gateway security/product review; if accepted, implement a constrained `chrome-extension://*` matcher with negative tests and docs/security-audit updates rather than a generic prefix wildcard.

## Reproduction Assessment

Yes. Source inspection plus Node URL behavior show `chrome-extension://<id>` becomes `URL.origin === "null"`, while current main only matches the global `*` or exact `parsedOrigin.origin`, so `chrome-extension://*` cannot work today.

## Solution Assessment

No for the issue's generic trailing-`*` sketch. A narrowly scoped `chrome-extension://*` policy may be maintainable, but only after maintainers accept the security tradeoff and define the exact matching contract.
