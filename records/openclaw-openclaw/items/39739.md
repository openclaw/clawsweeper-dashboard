## Summary

Keep this open: current main and the latest release still only accept exact `gateway.controlUi.allowedOrigins` entries or global `*`, while this request asks for distinct scheme-level wildcard semantics. The related open PRs cover exact custom-origin matching, not the changing Chrome-extension-ID use case, and the wildcard needs an explicit gateway security/product decision.

## What This Changes

Add support for scheme-level wildcard entries such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Have maintainers decide the gateway security policy first; if approved, implement a narrow `chrome-extension://*` matcher using canonical opaque-origin normalization, reject path/query/fragment/credential-bearing allowlist entries, and update focused tests plus docs/UI wording.

## Reproduction Assessment

Yes, at source level. A focused check with `origin: "chrome-extension://<id>"` and `allowedOrigins: ["chrome-extension://*"]` cannot match today because `URL.origin` becomes `"null"` and the checker only accepts exact origins or global `*`.

## Solution Assessment

No, not as sketched in the issue. A `startsWith` check against `parsedOrigin.origin` would still see `"null"`; if maintainers approve the policy, the best fix is canonical opaque-origin normalization plus a scheme-restricted wildcard matcher and regression coverage.
