## Summary

Keep open: current main and the latest release still only support exact allowed origins plus the global `*`, and the related open PRs cover exact opaque-origin matching while intentionally leaving scheme-level wildcard policy as a separate security/product decision.

## What This Changes

The issue requests support for configured scheme-level origin patterns such as `chrome-extension://*` in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Make a maintainer security/product decision on constrained scheme wildcard semantics, then implement any approved matcher locally in `src/gateway/origin-check.ts` with regression tests for allowed, rejected, malformed, literal `null`, path/query, and non-extension cases.

## Reproduction Assessment

Yes, at source level. Current main compares only `*` or exact `parsedOrigin.origin`, and Node serializes `chrome-extension://<id>` as `null`, so a configured `chrome-extension://*` entry cannot match today.

## Solution Assessment

No, not as sketched in the issue. A prefix check against `parsedOrigin.origin` would still see `null`; the safer path is an explicit opaque-origin normalization plus a constrained, maintainer-approved scheme wildcard matcher.
