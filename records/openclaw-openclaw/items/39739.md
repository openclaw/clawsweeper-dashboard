## Summary

Keep open: current main and v2026.5.22 still only allow the global `*` or exact parsed origins, and `chrome-extension://*` remains a new Gateway browser-origin policy decision rather than implemented behavior.

## What This Changes

Request support for `chrome-extension://*`-style scheme-level wildcard patterns in `gateway.controlUi.allowedOrigins` so Chrome extension clients do not need config updates when their extension IDs change.

## Best Possible Solution

Maintainers should explicitly approve or reject `chrome-extension://*`; if approved, implement a constrained Chrome-extension-only matcher with exact semantics preserved elsewhere plus unit, handshake, docs, and security-audit coverage.

## Reproduction Assessment

Yes. As a source-level check, current main accepts only global `*` or exact parsed-origin matches, and Node serializes Chrome-extension origins with `origin=null`, so there is no current `chrome-extension://*` path to accept the requested pattern.

## Solution Assessment

No for the generic trailing-`*` matcher proposed in the issue body; it is too broad for this trust boundary. A Chrome-extension-scoped wildcard policy is the safer shape if maintainers approve the feature.
