## Summary

Keep open: current `main` and the latest release still only treat stale persistent ACP resume failures as recoverable when the message contains `resource not found` or `no matching session`, while this issue reports Kiro wrapping that stale-resume condition as `Internal error` / `SessionResumeRequiredError`.

## What This Changes

The issue asks OpenClaw to recover or discard stale persistent ACP session IDs for thread-bound Kiro/Claude ACP sessions instead of leaving the thread unable to produce replies.

## Best Possible Solution

Add a narrow ACP manager recovery classifier and regression test for persistent resume failures wrapped as `SessionResumeRequiredError` / `Internal error`, while keeping retries limited to first-attempt, pre-output persistent turns.

## Reproduction Assessment

No high-confidence live current-main Kiro reproduction was run. Source inspection gives a credible reproduction path by adapting the existing stale-session retry test to the reported `Internal error <- SessionResumeRequiredError` message, which current `main` does not match as recoverable.

## Solution Assessment

Yes, the best path is a narrow ACP manager recovery update plus a Kiro-shaped regression test. Broadly retrying all ACP `Internal error` turn failures would be too risky because unrelated runtime failures should still surface normally.
