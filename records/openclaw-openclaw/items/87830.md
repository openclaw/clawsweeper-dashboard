## Label Justifications

- `P1`: Affected thread-bound ACP conversations can stay alive but stop producing normal replies after backend resume invalidation.
- `impact:session-state`: The failure centers on stale persisted ACP session identity being reused after the backend no longer accepts it.
- `impact:message-loss`: The user-visible symptom is a sent message showing processing signals but never receiving a normal text response in the bound thread.

## Summary

Keep open: current main and the latest release still only retry stale persistent ACP turn failures whose message says `resource not found` or `no matching session`, so the reported Kiro `SessionResumeRequiredError` / `Internal error` resume wrapper remains a valid focused bug.

## What This Changes

Thread-bound ACP sessions should discard or refresh stale persistent resume IDs after backend invalidation instead of leaving Kiro threads unable to produce normal replies.

## Best Possible Solution

Add a narrow ACP manager recovery path for persistent `SessionResumeRequiredError` resume wrappers like the Kiro `Internal error` case, clear the stale identity, retry a fresh handle once before output, and cover that path in regression tests.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: adapt the existing stale persistent-session retry test to emit `Persistent ACP session ... could not be resumed: Internal error` from a `SessionResumeRequiredError` before output. I did not establish a live Kiro failing run in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow ACP manager matcher/retry update plus a Kiro-shaped regression test. Broadly retrying all generic ACP `Internal error` failures would be too risky because unrelated turn failures should still fail visibly.
