## Summary

Keep open: current main and the latest release still only recover stale persistent ACP resume failures when the error text says `resource not found` or `no matching session`, while the reported Kiro path is the acpx persistent-session `SessionResumeRequiredError` wrapper ending in `Internal error`.

## What This Changes

The issue asks OpenClaw to recover thread-bound persistent ACP sessions when a stored Kiro/acpx resume id becomes invalid instead of leaving the bound thread unable to reply.

## Best Possible Solution

Add a narrow ACP manager recovery path for persistent `SessionResumeRequiredError` resume wrappers like the Kiro `Internal error` case, clear the stale identity, retry a fresh handle once, and cover that exact path in regression tests.

## Reproduction Assessment

Yes for a source-level reproduction: adapt the existing stale persistent-session retry test to emit `Persistent ACP session ... could not be resumed: Internal error` from a `SessionResumeRequiredError` before output. I did not run a live Kiro harness in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow ACP manager matcher/retry update plus a Kiro-shaped regression test. Broadly retrying all generic ACP `Internal error` failures would be too risky because unrelated turn failures should still fail visibly.
