## Summary

Keep open: current `main` and latest release still use the chokidar-only skills watcher, so this PR is not obsolete. The updated PR body now supplies the macOS `lsof` and refresh proof requested by the prior review; I found no blocking code-review findings, with only residual real-Windows watcher proof left for maintainer judgment.

## What This Changes

The PR changes the skills runtime watcher to use native recursive `fs.watch` for existing skill roots on macOS/Windows, keeps chokidar as fallback and on Linux, and adds watcher lifecycle tests.

## Best Possible Solution

Keep this PR open for maintainer review and merge if maintainers accept macOS proof as sufficient for the reported leak, or ask for a Windows smoke run before merge if Windows parity must be proven first.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and submitted platform proof: current `main` and `v2026.6.1` watch `SKILL.md` through chokidar, and the PR body includes real macOS `lsof` output showing descriptor count scales before and drops after the native watcher path.

## Solution Assessment

Is this the best way to solve the issue? Yes: the native-recursive watcher shape preserves `SKILL.md` edit/add/remove refresh while avoiding the directory-only alternative that the PR evidence shows loses refresh events, and it matches the shipped sibling memory watcher pattern.
