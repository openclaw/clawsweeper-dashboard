- `impact:message-loss`: The reported failure suppresses expected heartbeat responses or notifications after stale pending delivery is replayed.

## Summary

Keep open: current main and v2026.5.18 still have a source-reproducible `target: "none"` heartbeat path where non-ack `pendingFinalDeliveryText` is replayed instead of calling the model, then the no-target branch returns without clearing the marker.

## What This Changes

The issue reports that non-isolated `target: "none"` heartbeat runs can stop making fresh model calls after a previous run leaves pending final-delivery state.

## Best Possible Solution

Clear or bypass heartbeat-owned stale pending-final-delivery state for `target: "none"` after the heartbeat runner accounts for it, while preserving durable recovery for real outbound failures and adding a focused regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: seed a non-isolated main heartbeat session with non-ack `pendingFinalDeliveryText`, run a `target: "none"` heartbeat, and the code returns the replay text before normal model work then leaves pending state uncleared. I did not execute that fixture or a live Docker/GLM run in this read-only review.

## Solution Assessment

Yes, the best fix is in the existing heartbeat/final-delivery state path, not a new option. The narrow repair is to make internal-only `target: "none"` completion clear or safely consume heartbeat pending-delivery state without weakening normal durable delivery retry semantics.
