## Label Justifications

- `P1`: This is a reported regression in a current release where scheduled heartbeat automation can silently stop making model calls or deliveries for real Docker and Telegram setups.
- `impact:message-loss`: The visible symptom is skipped or suppressed heartbeat output instead of expected heartbeat responses or notifications.
- `impact:session-state`: The root path involves persisted `pendingFinalDelivery` and `lastHeartbeatText` session fields driving later heartbeat behavior.

## Summary

Keep open: current main and the latest release still contain the heartbeat pending-final replay, duplicate-skip, and target-none swallow paths, and the linked fix PR is open while the reporter has added non-`target:none` Telegram evidence.

## What This Changes

The issue reports that scheduled heartbeats can stop making fresh model calls after stale `pendingFinalDelivery` state, first with `target: "none"` and later with Telegram delivery showing `reason: "duplicate"`.

## Best Possible Solution

Land or supersede the linked heartbeat pending-final fix so stale heartbeat-owned pending delivery cannot suppress fresh model calls for `target: "none"` or externally delivered heartbeats, coordinating with the broader pending-final cleanup already tracked in https://github.com/openclaw/openclaw/issues/83184.

## Reproduction Assessment

Yes, at source level: seeded stale `pendingFinalDeliveryText` can be returned before fresh heartbeat model work, then the runner can mark the run as `target-none` or `duplicate`. I did not run the live Docker/GLM or Telegram scenario in this read-only review.

## Solution Assessment

Yes in direction: the fix belongs in the existing heartbeat and final-delivery state path, not in a new option. The current linked PR should be reviewed or expanded so it covers the newly reported non-`target:none` duplicate/replay case or is coordinated with the broader pending-final cleanup PR.
