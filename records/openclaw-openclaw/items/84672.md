- `impact:message-loss`: Scheduled heartbeat output can be skipped or suppressed instead of reaching the configured heartbeat path.
- `impact:session-state`: The failure is driven by persisted `pendingFinalDelivery*` and `lastHeartbeat*` session state being replayed across runs.

## Summary

Keep open: current main still has source-reproducible heartbeat paths that can suppress later model calls, and the open closing PR has not merged; the reporter's Telegram `duplicate` follow-up also overlaps the broader open heartbeat pending-delivery cleanup tracked by https://github.com/openclaw/openclaw/issues/83184 and https://github.com/openclaw/openclaw/pull/83187.

## What This Changes

The issue reports scheduled heartbeat runs getting stuck after stale final-delivery state, causing later heartbeats to skip instead of calling the model.

## Best Possible Solution

Review and land a narrow heartbeat state fix that covers both stale `target:none` replay and successful-send pending cleanup, while preserving real duplicate suppression and failed-delivery recovery.

## Reproduction Assessment

Yes, at source level: seed a heartbeat session with non-ack `pendingFinalDeliveryText` and current main can return the replay text before a fresh model call, then skip through `target-none` or duplicate-suppression handling. I did not run a live Docker/GLM reproduction in this read-only review.

## Solution Assessment

Partially: the linked `target:none` PR addresses the original reproduction path, but the Telegram `duplicate` follow-up should be handled by the broader successful-send pending cleanup or an equivalent coordinated fix before this issue is closed.
