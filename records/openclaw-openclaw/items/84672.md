- `impact:message-loss`: The reported failure suppresses scheduled heartbeat responses instead of producing the expected heartbeat result.
- `impact:session-state`: The root path is persisted session `pendingFinalDelivery` state being replayed across heartbeat runs.

## Summary

Keep open: current main still has the source path that can replay stale heartbeat pending-final-delivery text and then swallow it through `target: "none"`; the linked fixing PR is open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Fix heartbeats configured with `target: "none"` and shared sessions so stale pending final delivery does not suppress future model calls.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84679 or an equivalent narrow fix that makes `target: "none"` heartbeat delivery state internal-only after the runner accounts for it, with regression coverage and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows that a non-ack `pendingFinalDeliveryText` in a shared heartbeat session can bypass the fresh model call, then the `target: "none"` path records a skipped event without clearing the marker; I did not run a live Docker/GLM repro.

## Solution Assessment

Yes. The right fix belongs in the existing heartbeat/final-delivery state path, not in a new option or user workaround; the linked PR is the active candidate pending proof and maintainer review.
