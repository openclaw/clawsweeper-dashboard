## Summary

Keep open: the report matches a current source path and the latest release behavior, where `target: "none"` heartbeats can keep replaying a stale non-ack `pendingFinalDeliveryText` instead of making a fresh model call.

## What This Changes

The issue reports that heartbeats with `isolatedSession: false` and `target: "none"` stop calling the LLM after a prior run leaves `pendingFinalDelivery` set.

## Best Possible Solution

Make `target: "none"` heartbeat delivery state internal-only once the heartbeat runner accounts for it, so stale heartbeat pending-final-delivery text cannot suppress future model calls, and add a focused regression test for non-ack pending final delivery.

## Reproduction Assessment

Yes, from source inspection and the reporter's stored-session evidence, though I did not run a live Docker/GLM repro. The source path is non-ack `pendingFinalDeliveryText` causing heartbeat reply replay, followed by `target-none` handling that does not clear the pending marker.

## Solution Assessment

Yes, this should be fixed in the existing heartbeat/final-delivery state path rather than by adding a new option. The safest repair is narrow regression coverage plus clearing or bypassing heartbeat-owned stale pending delivery for internal-only `target: "none"` runs without weakening normal durable delivery recovery.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84672. In heartbeats configured with `target: "none"` and `isolatedSession: false`, a prior heartbeat that leaves non-ack `pendingFinalDeliveryText` can cause later heartbeats to replay that stale text, emit `target-none`, and skip the model call indefinitely. Preserve the documented `target: "none"` contract: run the heartbeat internally but do not deliver externally. Do not weaken durable final-delivery recovery for normal user turns or externally delivered heartbeats. Prefer a narrow fix in the heartbeat/final-delivery state path, likely around `src/auto-reply/reply/get-reply.ts`, `src/infra/heartbeat-runner.ts`, or the final-delivery persistence in `src/auto-reply/reply/agent-runner.ts`. Add regression coverage for non-ack `pendingFinalDeliveryText` plus `target: "none"` proving the stale marker does not suppress the next model call, and keep existing ack/defer behavior covered. Add a user-facing changelog fix entry during the PR.
