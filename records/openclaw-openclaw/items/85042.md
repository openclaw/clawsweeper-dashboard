## Summary

Keep open: the standard Google omitted-API path and remediation hint have been fixed, but current main and the latest release still contain a source-visible path where endpoint-less configured provider model rows can pair a non-OpenAI provider credential with an OpenAI-family transport.

## What This Changes

The issue asks OpenClaw to fail locally when non-OpenAI provider models lack concrete provider/transport configuration instead of silently routing their credentials through an OpenAI endpoint.

## Best Possible Solution

Keep this as the canonical maintainer/security issue and implement a generic provider credential-to-transport coherence guard with focused regression and live proof while preserving explicit OpenAI-compatible custom-provider configs.

## Reproduction Assessment

Yes. Source inspection on current main shows endpoint-less configured provider model rows can resolve to openai-responses while credentials are still resolved by the non-OpenAI provider id; reporter comments provide prior live Google and opencode logs.

## Solution Assessment

Unclear until maintainers choose the compatibility boundary. A generic credential/transport coherence guard is the likely best shape, but it must not break explicit custom OpenAI-compatible provider setups without a deliberate migration or policy decision.
