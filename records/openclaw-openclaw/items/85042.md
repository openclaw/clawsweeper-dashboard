## Label Justifications

- `P1`: The issue reports a real provider-routing/auth workflow that can send a provider credential to the wrong endpoint and affects user model fallback runs.
- `impact:security`: The reported failure mode is credential disclosure to an unintended third-party model endpoint.

## Summary

Keep open: the Google omitted-api path, blank-baseUrl catalog path, and remediation hint were partially fixed, but current main and v2026.6.11 still contain the broader source-visible path where an endpoint-less configured provider model row can resolve to an OpenAI-family transport.

## What This Changes

The issue asks OpenClaw to stop routing Gemini or other provider-scoped model requests through OpenAI-family transports when provider config lacks a concrete matching transport, and to surface a local config error instead.

## Best Possible Solution

Add a provider/transport coherence guard before dispatch that rejects endpoint-less non-OpenAI provider rows unless a provider-owned/default transport resolves, while preserving documented official provider catalogs and handling any upgrade-sensitive custom-provider cases deliberately.

## Reproduction Assessment

Yes, source-level. The opencode-shaped case follows current configured fallback code from an endpoint-less provider model row to an openai-responses model with no baseUrl, and reporter logs provide live symptom evidence.

## Solution Assessment

Unclear until maintainers choose the compatibility boundary. A simple missing models.providers.google check is not the best fix because official provider catalogs and env-backed Google auth are valid; the better fix is provider/transport endpoint coherence before dispatch.
