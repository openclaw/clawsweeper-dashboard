## Summary

Keep open: current main and the latest release fixed the standard Google omitted-api route and the remediation hint, but current source still allows endpoint-less configured provider model rows to fall back to an OpenAI-family transport while credentials are selected by the non-OpenAI provider id.

## What This Changes

The issue reports that incomplete Google or custom provider model configuration can resolve to an OpenAI-family transport and send provider-specific API keys to the wrong endpoint.

## Best Possible Solution

Keep this as the canonical maintainer/security issue and implement a generic provider credential/transport coherence guard with regression and live proof, while preserving explicit OpenAI-compatible custom providers.

## Reproduction Assessment

Yes: source inspection on current main shows a configured provider model row can resolve to openai-responses without baseUrl, and reporter comments include prior live Google and opencode logs. I did not run a live current-main network repro.

## Solution Assessment

Unclear until maintainers choose the compatibility boundary; a generic transport and credential coherence guard is likely better than another Google-only patch, but it must preserve explicit custom OpenAI-compatible providers.
