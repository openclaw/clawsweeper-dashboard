## Label Justifications

- `P1`: The issue reports a real provider-routing credential misroute that can affect agent runs for users configuring Google or OpenAI-compatible provider overlays.
- `impact:security`: The failure can disclose a non-OpenAI provider API key to OpenAI's endpoint when transport metadata is incomplete.
- `impact:auth-provider`: The affected path is provider/model resolution, transport selection, and provider-id credential lookup.

## Summary

Keep open: current main still has a source-reproducible provider/transport mismatch path that can pair a non-OpenAI provider credential with an OpenAI-family transport default. Adjacent merged fixes improved the remediation hint and blank-baseUrl Google catalog handling, but they do not close the generic pre-dispatch safety boundary reported here.

## What This Changes

The issue requests a local provider-routing failure instead of sending Google or OpenCode provider credentials to OpenAI when the selected model lacks a concrete non-OpenAI transport.

## Best Possible Solution

Add a generic pre-dispatch/provider-resolution coherence guard that rejects non-OpenAI provider credentials paired with OpenAI-family default transports unless an explicit safe baseUrl or provider-owned transport contract exists, while preserving documented env-only Google catalog routes.

## Reproduction Assessment

Yes, source-reproducible: current main can resolve an endpoint-less configured provider model to an OpenAI-family API while the embedded stream wrapper injects credentials by the non-OpenAI provider id.

## Solution Assessment

Unclear until maintainers choose the compatibility boundary; a generic provider/transport coherence guard is safer than a Google-only patch, but it needs security and upgrade review.
