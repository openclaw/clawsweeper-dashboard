## Summary

Keep open: current main has partial fixes for the model hint and some Google catalog/baseUrl cases, but the broader provider credential-to-default-OpenAI transport bug remains supported by source inspection and the OpenCode repro comment.

## What This Changes

The issue asks OpenClaw to fail locally instead of silently routing Google or other provider model requests to OpenAI when provider transport config is missing or incomplete.

## Best Possible Solution

Add a provider/transport coherence guard before dispatch so provider credentials cannot use a default OpenAI endpoint unless that endpoint is explicitly configured for the selected provider/model.

## Reproduction Assessment

Yes for a source-level path: current fallback resolution can still pair a non-OpenAI provider credential with openai-responses when a configured custom/provider model lacks api/baseUrl, though no live current-main request was run.

## Solution Assessment

The best fix is a generic provider/transport coherence guard, not a Google-only requirement, because the OpenCode repro has a provider and model row but still lacks an explicit endpoint.
