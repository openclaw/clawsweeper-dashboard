## Summary

Keep open: current main and v2026.6.11 fixed several adjacent Google and remediation paths, but the broader source-visible bug remains where an endpoint-less configured non-OpenAI provider model can resolve to an OpenAI-family transport with that provider's credential.

## What This Changes

The issue asks OpenClaw to fail locally instead of routing Gemini or other non-OpenAI provider credentials to an OpenAI-family endpoint when provider transport config is missing or incomplete.

## Best Possible Solution

Add a provider/transport coherence guard before dispatch that rejects endpoint-less non-OpenAI provider rows unless a provider-owned/default transport supplies the endpoint, with focused regression coverage and an explicit upgrade path for custom providers.

## Reproduction Assessment

Yes, source-level. The opencode-shaped case follows current configured fallback code from an endpoint-less provider model row to an OpenAI-family transport, and reporter logs provide live symptom evidence.

## Solution Assessment

Unclear until maintainers choose the compatibility boundary. A simple missing models.providers.google check is not the best fix because official provider catalogs and env-backed Google auth are valid; the better fix is provider/transport endpoint coherence before dispatch.
