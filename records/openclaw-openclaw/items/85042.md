## What This Changes

The issue requests a local validation failure when a `google/*` Gemini model lacks a coherent Google provider/transport configuration, instead of letting a Google API key reach an OpenAI transport.

## Best Possible Solution

Add a narrow pre-dispatch invariant that `google/*` models resolve to a Google-owned transport, or fail locally with a clear configuration error, while preserving documented env-backed Gemini auth when transport resolution is correct.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection plus the reporter's log show a credible path class where Google provider credentials can be resolved separately from an OpenAI-family model transport.

## Solution Assessment

Unclear as a concrete implementation: fail-local behavior is the right direction, but requiring `models.providers.google` for every Gemini run is not necessarily the best fix. The safer solution is to reject provider/transport mismatches before dispatch while preserving documented env-backed Google auth.
