## What This Changes

The issue asks OpenClaw to fail locally when a `google/*` Gemini fallback lacks coherent Google provider/transport configuration instead of dispatching a Google credential through an OpenAI transport.

## Best Possible Solution

Add a narrow provider/transport coherence guard before dispatch so `google/*` models either resolve to a Google-owned transport or fail locally, while preserving documented env-backed Gemini auth.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run in this read-only review. Source inspection supports a plausible mismatch class where fallback model construction can default to `openai-responses`, provider credentials resolve separately from env/auth, and dispatch follows `model.api`.

## Solution Assessment

Unclear as a complete fix: failing locally before dispatch is the right direction, but requiring `models.providers.google` for all Gemini use would conflict with documented env-backed auth. The safer solution is a provider/transport coherence check that preserves valid Google env-auth flows.
