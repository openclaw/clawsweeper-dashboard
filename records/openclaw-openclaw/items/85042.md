## Summary

Keep open: the reporter added concrete runtime logs, and current main still has provider/model paths where a Google provider credential can be resolved while the selected transport is OpenAI-family rather than a Google-owned transport. I did not find a merged fix or canonical duplicate for this exact cross-provider credential-routing bug.

## What This Changes

The issue requests that Gemini fallback/model selections fail locally when Google provider routing is not correctly configured instead of sending a Google API key through an OpenAI endpoint.

## Best Possible Solution

Add a narrow pre-dispatch invariant so google/* models either resolve to a Google-owned transport or fail locally with a Google provider auth/config error, while preserving documented env-backed Google credentials.

## Reproduction Assessment

No live current-main repro was run in this read-only review. The reporter supplied concrete logs, and source inspection shows a credible provider/transport mismatch path where a Google provider credential can be paired with an OpenAI-family transport.

## Solution Assessment

Unclear as a direct config-validation fix: failing locally is the right outcome, but requiring models.providers.google for every Gemini path may break documented env-backed auth. The safer solution is a provider/transport consistency guard before credentials are dispatched.
