## Summary

Keep open: the report describes a plausible auth-provider security bug, and current source still has model paths where provider identity, transport API, and resolved credentials can diverge. I did not find a canonical duplicate or a current-main fix during this sweep.

## What This Changes

The issue asks OpenClaw to fail locally when a Gemini model is selected without a valid Google provider configuration instead of routing the request through OpenAI.

## Best Possible Solution

Add a narrow provider/transport validation guard so explicit google/* models either resolve to a Google-owned transport or fail with a local configuration error before any credential is dispatched.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only sweep. Source inspection does show a credible path class where a Google provider credential can be resolved separately from an OpenAI-family model transport.

## Solution Assessment

The requested fail-local behavior is directionally right, but the safest fix is not necessarily to require models.providers.google for every Gemini run. The maintainable solution is to reject provider/transport mismatches before dispatch while preserving documented env-backed Google auth where it resolves to google-generative-ai.
