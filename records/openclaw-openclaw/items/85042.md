## Label Justifications

- `P1`: The issue reports a real provider-routing credential misroute that can affect agent runs for users configuring Google or custom providers.
- `impact:security`: The reported failure can send a non-OpenAI provider API key to an OpenAI endpoint when endpoint metadata is incomplete.
- `impact:auth-provider`: The affected path is provider/model resolution and credential selection for configured model fallbacks.

## Summary

Keep open: current main has fixed adjacent remediation text and blank-baseUrl catalog loss, but the central provider/transport safety issue is still not proven fixed for endpoint-less configured provider models that can fall back to an OpenAI-family transport while resolving credentials by the non-OpenAI provider id.

## What This Changes

The issue asks OpenClaw to stop routing Google or other provider credentials to an OpenAI endpoint when provider/model configuration is missing or incomplete.

## Best Possible Solution

Define a generic pre-dispatch provider/transport coherence rule that fails locally for non-OpenAI provider credentials paired with an OpenAI-family transport default unless an explicit provider/model endpoint contract allows it, while preserving documented env-only Google catalog resolution.

## Reproduction Assessment

Yes from source, but not from a fresh live run: the current resolver can still default endpoint-less configured provider models to `openai-responses` while auth resolution uses the provider id.

## Solution Assessment

Unclear until maintainers choose the compatibility boundary; a generic coherence guard appears safer than a Google-only patch, but it needs security and upgrade review.
