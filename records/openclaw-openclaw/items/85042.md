## Summary

Keep open: current main and latest release fixed the direct Google defaulting path, but the broader reported provider/transport safety boundary remains source-reproducible for configured custom/provider model rows that lack an explicit `api` and `baseUrl`.

## What This Changes

The issue asks OpenClaw to fail locally instead of allowing a non-OpenAI provider credential to pair with an OpenAI-family transport when provider/model transport metadata is incomplete.

## Best Possible Solution

Add a generic pre-dispatch safety boundary for incomplete provider/model transport metadata, while preserving explicit custom OpenAI-compatible provider contracts and the already-fixed native Google path.

## Reproduction Assessment

Yes. Source inspection shows a current-main path where a configured provider model without explicit `api` or `baseUrl` can synthesize an `openai-responses` fallback model before dispatch.

## Solution Assessment

No current fix fully solves it. The best fix is a generic provider/endpoint validation boundary, not another one-provider special case, and maintainers need to choose the compatibility behavior for existing custom providers.
