## Summary

Keep open: the PR is still necessary because current main still falls back to raw `mcpConfigHash` when a stored strict `mcpResumeHash` exists but the current run lacks one. The patch looks correct, but merging it intentionally changes persisted CLI session reuse behavior and needs maintainer acceptance plus refreshed green checks.

## What This Changes

Adds an early MCP invalidation guard in `resolveCliSessionReuse` and a regression test for stored resume-hash bindings when the current run cannot provide a resume hash.

## Best Possible Solution

Land the narrow guard after maintainer acceptance of the strict resume-hash fail-closed behavior and after exact-head checks are refreshed green; keep the broader stale claude-cli lifecycle policy in https://github.com/openclaw/openclaw/issues/84593.

## Reproduction Assessment

Yes, at source level. Passing a binding with stored `mcpResumeHash`, matching raw `mcpConfigHash`, and no current `mcpResumeHash` falls through to reuse on current main, while the PR makes that case return `{ invalidatedReason: "mcp" }`.

## Solution Assessment

Yes for the narrow code shape, with maintainer acceptance still needed for the behavior change. The shared reuse helper is the right decision point, and the patch preserves legacy `mcpConfigHash` fallback only for bindings that never stored a strict resume hash.
