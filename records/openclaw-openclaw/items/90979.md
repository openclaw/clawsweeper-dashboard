## Summary

Keep open: the patch is a narrow, source-backed fix, but it intentionally changes persisted CLI session reuse to fail closed for a strict MCP resume-hash case, so maintainer acceptance and a green exact-head check refresh are still needed.

## What This Changes

The PR adds an MCP invalidation guard to `resolveCliSessionReuse` when a stored binding has `mcpResumeHash` but the current run cannot provide one, plus regression coverage.

## Best Possible Solution

Land the narrow guard after maintainers accept the strict resume-hash reset behavior and the PR head is refreshed or rerun green; keep broader CLI binding lifecycle policy in https://github.com/openclaw/openclaw/issues/84593.

## Reproduction Assessment

Yes, at source level. Passing a binding with stored `mcpResumeHash`, matching raw `mcpConfigHash`, and no current `mcpResumeHash` falls through to reuse on current main, while the PR makes that case return `{ invalidatedReason: "mcp" }`.

## Solution Assessment

Yes for the narrow code shape, with maintainer acceptance still needed for the behavior change. The shared reuse helper is the right decision point, and the patch preserves legacy `mcpConfigHash` fallback only for bindings that never stored a strict resume hash.
