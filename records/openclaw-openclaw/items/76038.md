## Summary

Keep open: current main has partially improved stuck-session recovery and session-prep cost, but it still preserves active reply work in the reported skip path and still has a no-op recovery outcome when no lane work is released; the Feishu/systemd upgrade claim remains under-specified.

## What This Changes

The issue asks OpenClaw to recover stuck processing sessions, avoid repeated multi-minute session preparation work, and preserve Feishu/systemd service configuration across upgrades.

## Best Possible Solution

Finish the diagnostics recovery path with targeted coverage for stale reply-run state and no-active-work recovery, keep the prep caching fixes, and split the Feishu/systemd override claim into a focused installer reproduction if it still fails.

## Reproduction Assessment

Partially yes. Source and tests show the active-reply skip and no-active-work no-op branches on current main, but the Feishu/systemd upgrade loss is not reproducible from the report without the exact unit/drop-in and upgrade sequence.

## Solution Assessment

No, closing or applying the one-line proposal directly is not yet the best way to solve the whole issue. The maintainable path is a focused diagnostics recovery change that distinguishes stale registry state from real active work, plus a separate installer/systemd repro if the env loss remains valid.
