## Summary

Keep open: current main has fixed part of the prep latency and added some stale/stalled recovery, but the `active_reply_work action=keep_lane` path remains in source and is covered by current tests as non-recovering; the Feishu/systemd upgrade claim still needs a separate focused reproduction.

## What This Changes

The issue reports stuck-session recovery paths that do not release blocked processing lanes, repeated slow session preparation, and possible Feishu systemd override loss after upgrade.

## Best Possible Solution

Keep the prep caching and extended stalled-run recovery already on main, then land a focused diagnostics recovery change for stale reply-run flags and split any confirmed Feishu/systemd override loss into its own installer repro and fix.

## Reproduction Assessment

Partially yes: current source and tests give a high-confidence path for the `active_reply_work` skip, and a user reproduced that path on `openclaw@2026.5.3`. The Feishu/systemd upgrade claim is not yet reproducible from the report alone.

## Solution Assessment

No broad combined fix is the best path. The maintainable solution is a focused recovery change for stale reply-run state, while keeping the already-landed prep improvements and handling the installer/systemd claim separately if reproduced.
