## Summary

Keep open: current main still hard-codes frame denial for Control UI responses and has no config/schema/docs escape hatch, but the requested opt-in is a security-sensitive new config mode that needs maintainer judgment.

## What This Changes

The issue requests a `gateway.controlUi` framing policy so operators can opt into same-origin or trusted-origin embedding while retaining the hardened default.

## Best Possible Solution

If maintainers accept the use case, add a narrowly validated `gateway.controlUi` frame policy with a default deny mode, documented CSP/X-Frame-Options semantics, schema/help/docs/audit coverage, changelog entry, and regression tests for DENY, SAMEORIGIN, and multi-origin CSP-only output.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: any Control UI HTTP response receives `X-Frame-Options: DENY` plus `frame-ancestors 'none'`, and the strict config schema has no override field. I did not launch a live browser because the checkout was read-only, but the code and existing test directly prove the reported header behavior on current main.

## Solution Assessment

Unclear as an implementation direction until maintainers decide the security policy. The requested `frameAncestors` knob is plausible, but the best fix should be deliberately scoped, validated, documented, and reviewed as a clickjacking-policy relaxation rather than treated as a mechanical bug fix.
