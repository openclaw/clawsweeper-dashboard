## Summary

Keep open: current main still hard-codes Control UI anti-framing headers with no config/schema/docs escape hatch, but solving this requires a security-sensitive public framing policy rather than a narrow bug-fix cleanup.

## What This Changes

Requests a `gateway.controlUi` config policy to allow same-origin or trusted-origin framing of the Control UI while keeping deny-by-default headers.

## Best Possible Solution

Keep the hardened default, and if maintainers accept the dashboard-embed use case, add a narrowly validated frame policy with documented CSP/X-Frame-Options semantics and regression coverage for deny, same-origin, and multi-origin modes.

## Reproduction Assessment

Yes. Source inspection and existing tests show any Control UI response receives `X-Frame-Options: DENY` plus `frame-ancestors 'none'`, while the strict config schema has no override field; I did not run a live browser iframe repro in the read-only checkout.

## Solution Assessment

Unclear until maintainers decide the security policy. The requested `frameAncestors` knob is plausible, but the maintainable fix should be deliberately scoped, validated, documented, and reviewed as a clickjacking-policy relaxation.
