## Summary

Keep open: current main still hard-codes Control UI anti-framing headers and has no config/schema/docs escape hatch. The requested fix is plausible, but it is a new public security policy mode that relaxes clickjacking defenses and needs maintainer judgment before implementation.

## What This Changes

Request to add a deny-by-default `gateway.controlUi` frame policy option that can allow same-origin or trusted-origin Control UI iframe embedding.

## Best Possible Solution

If maintainers accept the use case, add a narrowly validated deny-by-default `gateway.controlUi` frame policy with documented CSP/X-Frame-Options semantics, config schema/help/docs coverage, security-audit treatment, and regression tests for deny, same-origin, and multi-origin CSP-only modes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: every Control UI response receives `X-Frame-Options: DENY` plus `frame-ancestors 'none'`, and the strict config schema has no override field.

## Solution Assessment

Unclear until maintainers decide the security policy. The requested `frameAncestors` config is a plausible maintainable shape, but it should be reviewed as a deliberate clickjacking-policy relaxation rather than a mechanical bug fix.
