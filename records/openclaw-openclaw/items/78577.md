## Summary

Keep open: current main and the latest release still hard-code Control UI anti-framing headers with no config escape hatch, but the requested fix is a security-sensitive public framing policy and config addition that needs maintainer review.

## What This Changes

Request adds a gateway.controlUi opt-in to relax Control UI frame-ancestor and X-Frame-Options headers for same-origin or trusted iframe embedding while keeping deny-by-default behavior.

## Best Possible Solution

Keep the hardened default, and if maintainers accept dashboard embedding, add a narrowly validated frame policy with documented CSP/X-Frame-Options semantics and regression coverage for deny, same-origin, and multi-origin modes.

## Reproduction Assessment

Yes. Source inspection and existing tests show Control UI responses receive X-Frame-Options: DENY plus frame-ancestors 'none', while the strict config schema has no override field.

## Solution Assessment

Unclear until maintainers decide the security policy. The requested frameAncestors knob is plausible, but the maintainable fix should be deliberately scoped, validated, documented, and reviewed as a clickjacking-policy relaxation.
