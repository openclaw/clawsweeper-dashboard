Keep open: current main still has the documented inconsistency, and open fix PRs are already attached, including PRs with closing syntax. The issue should remain open until one fix lands, then the duplicate PRs can be resolved.

## What This Changes

Requests that group/channel failure-fallback replies honor the existing `agents.defaults.silentReply` and `surfaces.<id>.silentReply` policy instead of always becoming `NO_REPLY`.

## Best Possible Solution

Land one narrow resolver-based fix that threads the existing config/session/surface policy into the failure-fallback helper while preserving default group/channel silence.

## Reproduction Assessment

Yes. Source inspection on current main shows a group/channel generic runner failure goes through `resolveExternalRunFailureTextForConversation` and becomes `SILENT_REPLY_TOKEN` before the existing silent-reply policy can be consulted.

## Solution Assessment

Yes. Reusing `resolveSilentReplyPolicy` in the existing failure helper is the narrowest maintainable fix; adding another failure-specific knob would duplicate the current policy surface.
