## Summary

Keep open: current main still bypasses the documented silent-reply policy on the generic group/channel failure-fallback path, and this issue is now paired with open fix PRs such as https://github.com/openclaw/openclaw/pull/82086. The issue should close only after one linked fix lands or the competing fix PRs are explicitly superseded.

## What This Changes

Make runner failure-fallback replies respect `agents.defaults.silentReply` and `surfaces.<id>.silentReply` for group/channel conversations.

## Best Possible Solution

Land one narrow resolver-based fix that preserves default group/channel silence while honoring explicit `silentReply.group: "disallow"`, then close the issue after merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the generic group/channel failure helper returns `SILENT_REPLY_TOKEN` before any `resolveSilentReplyPolicy` call, while the docs and config tests define `silentReply.group: "disallow"` as an existing override.

## Solution Assessment

Yes. Reusing `resolveSilentReplyPolicy` in the failure-fallback helper is the narrowest maintainable fix; the remaining maintainer work is to select and validate one linked PR, not to invent a new config surface.
