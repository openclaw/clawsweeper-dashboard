## Summary

Keep open: current main intentionally silences generic pre-reply runner failures in group/channel contexts, and the requested failure-specific generic notice is still a product/config request rather than an already-fixed bug.

## What This Changes

The issue asks for an opt-in policy that lets group/channel chats receive curated generic fallback text after generic embedded LLM run failures instead of the current silent default.

## Best Possible Solution

Choose one coherent failure-notice policy that preserves the silent group/channel default while allowing explicitly configured curated generic failure text without reintroducing raw provider errors.

## Reproduction Assessment

Yes, by source inspection: generic pre-reply runner failures in group/channel contexts become `NO_REPLY`, and the default group silent-reply policy drops that final payload. I did not run the live Feishu timeout path.

## Solution Assessment

Unclear: the requested behavior is plausible, but adding `agents.defaults.runFailure.groupNotice` may duplicate existing silent-reply and channel error-policy surfaces. Maintainers should decide the config shape before implementation.
