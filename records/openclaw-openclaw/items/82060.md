## Summary

Keep open and queue a focused fix: current main still converts generic pre-reply runner failures in group/channel contexts to `NO_REPLY` before consulting the existing silent-reply policy surface. The default group/channel silence should be preserved, but the explicit `silentReply.group: "disallow"` override is a real existing config contract to honor.

## What This Changes

Align group/channel generic run-failure fallback delivery with the existing `agents.defaults.silentReply` and `surfaces.<id>.silentReply` policy.

## Best Possible Solution

Make the failure-fallback helper use the already resolved config/session/surface conversation policy, returning the compact sanitized failure copy only when that policy disallows silence while preserving current default group/channel quiet behavior.

## Reproduction Assessment

Yes, by source inspection: the generic group/channel failure helper unconditionally returns `SILENT_REPLY_TOKEN`, while the shared policy resolver and docs expose `silentReply` as the override surface. I did not run the live Feishu timeout path.

## Solution Assessment

Yes: reusing `resolveSilentReplyPolicy` with the existing run config and conversation context is the narrowest maintainable fix. Adding a new failure-specific config knob would duplicate the existing policy surface.
