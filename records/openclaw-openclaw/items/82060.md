## Summary

Keep open: current main still intentionally turns generic pre-reply runner failures in group/channel contexts into `NO_REPLY`, and the requested opt-in generic notice is a product/config change rather than an already-implemented cleanup close.

## What This Changes

Allow deployments to opt group/channel conversations into curated generic failure notices after generic embedded LLM run failures while preserving the current silent default.

## Best Possible Solution

Define one documented failure-notice policy, likely alongside existing silent-reply/surface or channel error-policy controls, that allows curated generic group notices without exposing raw provider details.

## Reproduction Assessment

Yes, at source level: generic pre-reply runner failures in group/channel contexts become `NO_REPLY`, and the default group silent-reply policy drops that final payload. I did not run the live Feishu timeout path.

## Solution Assessment

Unclear: the requested behavior is plausible, but the exact config shape should be chosen by maintainers because it overlaps existing silent-reply and channel error-policy surfaces.
