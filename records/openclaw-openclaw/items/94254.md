## Summary

Keep open: current main and the latest release still contain the broad app-password redaction gate that can mask benign four-part kebab identifiers, and the linked fix PR is open but not merged.

## What This Changes

The issue asks OpenClaw to stop redacting ordinary four-segment kebab identifiers as Apple app-specific passwords in persisted transcript/log content.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/94256; the desired end state is context-gated Apple app-password redaction that preserves ordinary transcript identifiers while still masking real app passwords.

## Reproduction Assessment

Yes, from source inspection: current main applies APP_SPECIFIC_PASSWORD_RE to generic text/error/message-style fields and persists redactSecrets output through transcript append. I did not execute the repo code because this review is read-only, but the source path and reporter's before/after command are specific and consistent.

## Solution Assessment

Yes, the narrow fix direction is to restrict Apple app-password masking to Apple/iCloud-specific fields or nearby Apple/iCloud context while preserving regression coverage for real app passwords. The linked PR is the active candidate for that fix and should be reviewed rather than duplicating work here.
