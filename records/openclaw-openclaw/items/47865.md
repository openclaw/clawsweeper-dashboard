## Summary

Keep open. Current main still routes main-session cron `systemEvent` jobs through heartbeat `target: "last"` without honoring the job's `delivery.mode: "none"`, and related open PRs only cover parts of the delivery/routing problem.

## What This Changes

The issue asks OpenClaw to prevent internal cron `systemEvent` and background exec system messages from being pushed into user chat when they are intended to be internal-only.

## Best Possible Solution

Define and implement one clear main-systemEvent delivery contract: explicit internal-only main cron jobs should not produce user-visible heartbeat replies, while chat-created reminders that should notify users should use an isolated announce path with preserved delivery context; keep the inherited-destination leak tracked with the linked narrower fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session `systemEvent` cron job with `delivery.mode: "none"` is accepted, enqueues a cron-tagged system event, and wakes heartbeat with `target: "last"`; if heartbeat resolves a chat target and the model emits visible text, the message is sent even though cron delivery status remains `not-requested`.

## Solution Assessment

Unclear. The narrowest maintainable fix is likely to split internal-only main system events from user-visible chat reminders, but maintainers should confirm whether `delivery.mode: "none"` must suppress main heartbeat replies or whether main/systemEvent jobs should be documented and migrated toward isolated announce jobs for reminders.
