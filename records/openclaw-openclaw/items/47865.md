## Summary

Keep open. Current main still routes main-session cron `systemEvent` jobs through the heartbeat path with `heartbeat.target = "last"`, while `delivery.mode: "none"` only records `lastDeliveryStatus: "not-requested"`; the related draft PR is still unmerged and the privacy semantics need maintainer review.

## What This Changes

Ensure cron `systemEvent` and related internal exec events configured with no delivery are handled internally instead of surfacing in the user chat UI.

## Best Possible Solution

Define and enforce one delivery policy for main-session `systemEvent` cron jobs: explicit no-delivery should carry an internal-only/no-user-delivery flag through enqueueing and heartbeat delivery, or the unsupported expectation should be rejected/documented with regression coverage for cron and exec events.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the reported job shape reaches `executeMainSessionCronJob`, records no requested delivery, then still wakes a `target: "last"` heartbeat that can relay to the last chat target.

## Solution Assessment

No for current main. The current implementation relies on heartbeat target resolution and prompt wording instead of carrying the cron no-delivery policy into the heartbeat delivery decision; the narrow maintainable fix should make that policy explicit in code and tests.
