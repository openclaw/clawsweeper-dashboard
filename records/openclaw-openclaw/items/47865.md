## Summary

Kept open because current main still has a source-visible path where explicit `sessionTarget: "main"` cron `systemEvent` runs wake the main heartbeat/session path and do not suppress the agent response based on `delivery.mode: "none"`; the related draft PR and newer routing issue cover adjacent fixes but are not merged.

## What This Changes

The issue asks for cron `systemEvent` jobs with `delivery: { mode: "none" }` to stay internal and not surface system/reminder output in the user chat UI.

## Best Possible Solution

Add a narrow cron/system-event delivery contract so explicit internal `main` system events with `delivery.mode: "none"` can wake internal state without forwarding the resulting reminder/log response to the user channel, while separately preserving the session-bound routing fix tracked for Telegram.

## Reproduction Assessment

Yes from source, but not from a live run in this read-only review: `executeSystemEvent` still wakes the main heartbeat/session path without consulting `delivery.mode`, while `lastDeliveryStatus: "not-requested"` only reflects the cron announce layer. A high-confidence live reproduction would need a configured WebChat cron job and agent response capture.

## Solution Assessment

Unclear. The best fix may be to treat `delivery.mode: "none"` as suppressing user-visible agent responses for explicit internal system events, but that is a contract decision because current tests also preserve `not-requested` for explicit main system events and related PR work focuses partly on chat-created reminder promotion.
