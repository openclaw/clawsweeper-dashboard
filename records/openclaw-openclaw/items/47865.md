## Summary

Keep open: current `main` still has a source-reproducible path where a main-session cron `systemEvent` ignores the job's `delivery.mode="none"` for the heartbeat response path, while the linked fix PR remains open, draft, and previously reviewed as needing changes.

## What This Changes

The issue reports that cron `systemEvent` payloads with `delivery: { mode: "none" }` still produce user-visible chat messages, including reminder text and internal exec-failure logs.

## Best Possible Solution

Preserve the documented main-session `delivery.mode="none"` contract by making cron-triggered heartbeat responses internal-only for those jobs, while keeping webhook, announce, and isolated-agent delivery semantics intact and adding focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session `systemEvent` job with `delivery.mode="none"` still enters `executeMainSessionCronJob`, which wakes heartbeat with `target: "last"`; if the session has a chat target, the heartbeat runner can relay the model response to the user.

## Solution Assessment

No current fix is landed. The narrow maintainable direction is to carry the cron delivery intent into the heartbeat wake path or otherwise force an internal-only heartbeat target for main `systemEvent` jobs whose delivery mode is `none`, with tests for reminder and exec-completion cases.
