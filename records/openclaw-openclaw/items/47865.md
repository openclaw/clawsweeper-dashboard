## Summary

Keep open. Current main still routes main-session cron `systemEvent` jobs through the heartbeat path without consulting `delivery.mode: "none"` before enqueueing the system event or waking `target: "last"`, and the related draft PR #58893 is unmerged while #73900 remains open for a related cross-context leak.

## What This Changes

The issue reports that internal cron system events and background exec failure events are being pushed into user chat surfaces despite `delivery: { mode: "none" }`.

## Best Possible Solution

Define one explicit delivery contract for main-session cron system events: either propagate `delivery.mode: "none"` as a no-user-delivery constraint through cron enqueueing, heartbeat prompt selection, and outbound delivery, or reject/clarify that this job shape is user-relay capable. Cover the main cron no-delivery case, the inherited heartbeat target case from #73900, and the exec completion path with regression tests.

## Reproduction Assessment

Yes. The issue's cron config maps directly to `executeMainSessionCronJob`: the system event is enqueued and heartbeat is woken with `target: "last"` even when the delivery plan would be `not-requested`.

## Solution Assessment

No for current main. The current behavior relies on heartbeat routing and prompt wording rather than enforcing the cron delivery policy at the cron-to-heartbeat boundary; a safer fix should make the delivery decision explicit in code and tests.
