## Summary

Keep open. Current main still has a source-traceable path where main-session cron `systemEvent` jobs ignore `delivery.mode: "none"`, enqueue a system event, and force a heartbeat delivery target of `last`; related fixes are open but unmerged.

## What This Changes

The issue reports cron `systemEvent` jobs with `delivery.mode: "none"` still producing user-visible chat messages and internal exec failure text.

## Best Possible Solution

Land or replace focused fixes that make explicit main-session `delivery.mode: "none"` system events non-user-visible and also cover the inherited `target: "last"` routing leak tracked by https://github.com/openclaw/openclaw/issues/73900.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: create a `sessionTarget: "main"` cron `systemEvent` with `delivery.mode: "none"`; the cron runner ignores that delivery mode, enqueues the event, and wakes heartbeat with `target: "last"`.

## Solution Assessment

Yes. The maintainable fix belongs in the cron/heartbeat delivery boundary, not in prompt wording: explicit no-delivery main cron events should suppress user-visible replies, and cron-forced `target: "last"` should not inherit explicit configured destinations.
