## Label Justifications

- `P2`: This is a real user-visible Control UI lag in external-channel Codex sessions, but it is scoped to transcript projection rather than core runtime availability.
- `impact:message-loss`: The accepted inbound channel message is effectively absent from the operator-visible transcript until the agent turn completes.
- `impact:session-state`: The reported failure is specifically durable session transcript state lagging behind the accepted channel turn.

## Summary

Close as duplicate: the same durable-transcript lag is already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/83528, and current main still shows the same Codex app-server final-mirror path that explains this Telegram report.

## What This Changes

This issue requests that Telegram-originated turns appear in Control UI immediately by persisting inbound user messages and streaming progress instead of committing the session transcript only at turn end.

## Best Possible Solution

Track and fix this through the canonical open issue by adding an early Codex prompt mirror using the existing transcript update and dedupe contract; split broader progress badges only if they still remain after that fix.

## Reproduction Assessment

Yes. Current main still mirrors the Codex app-server messages after turn completion while chat.history reads durable transcript rows, matching the delayed UI visibility described here.

## Solution Assessment

Yes for routing, not as a separate tracker. The maintainable fix should land through the canonical issue's early prompt-mirror path, with any broader assistant/tool streaming work split only if it remains independently necessary.
