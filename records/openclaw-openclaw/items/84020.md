## Label Justifications

- `P2`: The issue creates a real operator visibility gap during long turns, but it does not make the core runtime unusable.
- `impact:message-loss`: The inbound channel message is effectively absent from the operator-facing chat view until the transcript catches up.
- `impact:session-state`: The Control UI's visible session state can lag the live session/turn state for the duration of a long agent run.

## Summary

Keep open: the reported Control UI dark window is still supported by current-main source and the reporter's live release evidence. The narrow inbound-message visibility fix is actionable, but the broader assistant/tool progress streaming request crosses the existing external-channel visibility boundary and needs maintainer product/privacy judgment.

## What This Changes

The issue asks for inbound channel messages and in-progress agent turn activity to become visible in the Control UI before the full session transcript is committed at turn end.

## Best Possible Solution

Implement the narrow inbound path first by making the user message visible to the active Control UI session as soon as it is accepted, then handle external-channel assistant/tool progress through an explicit observer or visibility-policy design that keeps privacy and authorization boundaries clear.

## Reproduction Assessment

Yes, source-backed: current main still has `session.message` payload broadcast plumbing, but the Control UI reloads file-backed `chat.history` instead of applying the live payload, and the reporter supplied a live Telegram timing trace. I did not run a fresh live current-main Telegram reproduction in this read-only review.

## Solution Assessment

No, not as one undifferentiated change. The maintainable path is a narrow immediate-inbound fix plus a separate maintainer-approved decision for external-channel assistant/tool progress visibility.
