- `P2`: The issue describes a real operator visibility gap during long channel turns, but it does not make the core runtime unusable.
- `impact:message-loss`: The operator-facing chat view suppresses a received inbound message until much later, making the message appear missing during the run.

## Summary

Keep open: the latest reporter comment adds concrete live evidence for the delayed Control UI update, and current main still shows the same source-level gap. The inbound-message part has a narrow UI/session-event repair path, while assistant/tool progress for Telegram-originated turns crosses the existing external-channel visibility guard and needs maintainer scope judgment.

## What This Changes

The issue asks Control UI to show inbound channel messages immediately and stream assistant/tool progress during long channel-originated turns instead of waiting for the final transcript reload.

## Best Possible Solution

Split the work: merge safe inbound `session.message` user payloads directly into the selected Control UI transcript, then design a session-scoped external-channel observer stream for assistant/tool progress that preserves the current privacy guard.

## Reproduction Assessment

Yes at source level, and the reporter added a live v2026.5.12/v2026.5.18 timeline. I did not run a current-main Telegram live repro, but current main still broadcasts `session.message` payloads while the UI reloads transcript-backed `chat.history` instead of applying them.

## Solution Assessment

No, not as one undifferentiated fix. The inbound-message visibility has a narrow UI/session-event fix, while assistant/tool streaming for Telegram should be approved as a selected-session visibility policy change.
