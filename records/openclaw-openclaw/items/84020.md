## Label Justifications

- `P2`: The issue describes a real operator visibility gap during long turns, but it does not make the core runtime unusable.
- `impact:message-loss`: The reported symptom is that an inbound channel message is suppressed from the operator-facing chat view until much later.
- `impact:session-state`: The affected surface is session transcript projection and live session state in the Control UI.

## Summary

Keep open: current main has session transcript broadcast plumbing, but the Control UI still reloads file-backed history instead of applying the live `session.message` payload, so the reported delayed inbound-message visibility remains plausible. The broader request to stream assistant/tool progress for Telegram sessions crosses an intentional external-channel visibility guard and needs maintainer scope judgment.

## What This Changes

Investigate whether inbound channel messages and in-progress agent output can appear in the Control UI before the full agent turn is committed.

## Best Possible Solution

Split the fix: consume safe inbound `session.message` payloads directly in the Control UI for immediate transcript visibility, and handle external-channel assistant/tool progress as an explicit privacy/UX decision rather than widening the existing visibility guard incidentally.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The source path is reproducible: `session.message` contains a live payload, but the UI reloads `chat.history`, leaving display dependent on transcript-file freshness.

## Solution Assessment

No, not as one undifferentiated change. The inbound-message part has a narrow UI/session-event fix, while assistant/tool streaming for Telegram should be approved as a separate visibility policy change.
