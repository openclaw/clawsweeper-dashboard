## Summary

Keep open. Current main still has the unsafe Signal doctor repair path: shared Signal defaults can be promoted into `channels.signal.accounts.default`, and Signal does not provide a named-account promotion filter to prevent the reported phantom account.

## What This Changes

Prevent `openclaw doctor --fix` from moving shared Signal multi-account defaults into a phantom `channels.signal.accounts.default` entry.

## Best Possible Solution

Signal's doctor promotion contract should keep inheritable shared defaults at `channels.signal` and only migrate true legacy account identity or endpoint fields, with a regression test for the reported two-account config.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: the reported config shape gives `resolveSingleAccountKeysToMove` named accounts plus `cliPath`, `dmPolicy`, and `groupPolicy`, and the normalizer still writes those keys into `accounts.default`.

## Solution Assessment

No. The current generic promotion behavior is not the right solution for Signal multi-account configs because it turns shared defaults into a runnable account entry; a Signal-specific promotion filter or equivalent narrow guard is safer.
