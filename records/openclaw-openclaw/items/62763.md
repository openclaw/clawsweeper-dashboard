## Summary

Keep open. Current main still has the reported unsafe doctor compatibility path: named Signal accounts plus top-level shared defaults can be rewritten into `channels.signal.accounts.default`, and Signal runtime code still treats that generated entry as an enabled/configured account candidate.

## What This Changes

The issue reports that `openclaw doctor --fix` corrupts Signal multi-account configs by moving shared top-level defaults into a phantom `accounts.default` account.

## Best Possible Solution

Preserve shared Signal defaults at `channels.signal` for named-account configs, or apply a Signal-specific promotion filter that only moves true account identity fields, with regression coverage for doctor `--fix`.

## Reproduction Assessment

Yes at the source level. The reported config shape reaches the current doctor compatibility normalizer, which still moves `cliPath`, `dmPolicy`, and `groupPolicy` into `accounts.default` for Signal named-account configs.

## Solution Assessment

No for current main. The generic promotion behavior is not the right fix for Signal named-account shared defaults; a narrow Signal guard/filter plus regression tests is the safer maintainable path.
