## Summary

Keep open. Current main still has the unsafe Signal doctor repair path: a multi-account Signal config with top-level shared defaults can still be rewritten into `channels.signal.accounts.default`, and that new entry is visible to Signal runtime account enumeration.

## What This Changes

The issue asks `openclaw doctor --fix` to stop corrupting valid Signal multi-account configs by moving shared top-level defaults into a phantom `accounts.default` account.

## Best Possible Solution

Keep shared Signal defaults such as `cliPath`, `dmPolicy`, and `groupPolicy` at `channels.signal` for named-account configs, and add regression coverage for the reported two-account doctor repair case.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: the reported config has named accounts plus top-level `cliPath`, `dmPolicy`, and `groupPolicy`, and current main still moves those keys into `accounts.default` during doctor compatibility repair.

## Solution Assessment

No for current main. The generic promotion behavior is not the best solution for Signal because it turns inherited shared defaults into an account entry; a narrow Signal promotion filter or guard is safer.
