## Summary

Keep open. Current main still has the reported unsafe Signal doctor repair path: the generic legacy normalizer can move inherited Signal channel defaults into `accounts.default`, and Signal has not added a named-account promotion filter to keep shared defaults at the channel level.

## What This Changes

The issue asks OpenClaw to stop `openclaw doctor --fix` from creating a phantom `channels.signal.accounts.default` entry from shared top-level Signal settings in multi-account configs.

## Best Possible Solution

Keep the issue open and make Signal's doctor promotion distinguish real legacy account identity/transport fields from inherited channel defaults, with a regression test for the reported two-account shape so `cliPath`, `dmPolicy`, and `groupPolicy` remain at `channels.signal` instead of creating a runnable phantom default account.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path on current main: pass the reported config shape through the legacy doctor normalizer and the helper still selects the reported shared Signal keys for promotion into `accounts.default`.

## Solution Assessment

No. The current generic promotion behavior is not the best way to solve Signal multi-account migration because it converts inherited channel defaults into an account entry; a Signal-specific promotion filter or equivalent core contract is the safer narrow fix.

The observable bug is that `openclaw doctor --fix` can move inherited Signal multi-account defaults (`channels.signal.cliPath`, `dmPolicy`, `groupPolicy`) into a new `channels.signal.accounts.default` entry when named Signal accounts already exist, creating a runnable phantom account without an account number. Do not change unrelated channel migration semantics, do not add startup/load-time config migrations, and do not broaden core with Signal-specific IDs except through an existing plugin/setup contract.
