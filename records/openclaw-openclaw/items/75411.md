## Summary

Keep open: current `main` still preserves the reported Copilot routing gap. The guarded OpenAI Responses transport handles terminal 429s, but embedded runs only select it when the cached session stream is undefined or exactly `streamSimple`; a custom session stream still reports `session-custom` and returns the existing stream unchanged.

## What This Changes

The issue requests that GitHub Copilot `openai-responses` quota 429s fail fast even when the embedded runner uses the `session-custom` stream path.

## Best Possible Solution

Route every embedded Copilot `openai-responses` turn through `buildGuardedModelFetch` or an equivalent guarded provider-owned transport before idle-timeout handling, while preserving Copilot's header and payload wrapper behavior.

## Reproduction Assessment

Yes. The issue provides live `v2026.4.29` telemetry for the user-visible failure, and current `main` still has a source-level reproduction where a non-`streamSimple` session stream is classified as `session-custom` and bypasses the guarded Responses transport.

## Solution Assessment

Yes. The requested invariant is the maintainable target; the safest fix is a narrow stream-resolution/provider seam with regression tests, not a broad retry-policy change. PR #75272 shows an adjacent Codex-only pattern but does not cover Copilot `openai-responses`.
