## Summary

Keep open: current `main` still preserves the reported routing gap. The guarded OpenAI Responses transport exists, but embedded runs only select it when the cached session stream is undefined or exactly `streamSimple`; the locked Pi SDK creates a wrapper function around `streamSimple`, so GitHub Copilot `openai-responses` traffic can remain on the `session-custom` path described in the issue.

## What This Changes

The issue asks OpenClaw to fail fast on GitHub Copilot `openai-responses` HTTP 429s even when embedded runs use the session-custom stream path.

## Best Possible Solution

Route every embedded `openai-responses` Copilot turn through `buildGuardedModelFetch` or an equivalent guarded provider-owned transport before idle-timeout handling, while preserving Copilot's header and payload wrapper behavior.

## Reproduction Assessment

Yes. The issue provides live `v2026.4.29` telemetry for the user-visible failure, and current `main` plus the locked Pi SDK still show the same code path that classifies the Pi default stream wrapper as `session-custom`.

## Solution Assessment

Yes. The requested invariant is the right target; the maintainable fix should be a narrow generic embedded/provider stream-resolution change with regression tests, not a broad retry-policy change or a hard-coded Copilot-only branch unless the provider seam proves insufficient.
