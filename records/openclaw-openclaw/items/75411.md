## Summary

Keep open: current main still has a real routing gap for the reported Copilot Responses path. The guarded fetch handles terminal 429s when the boundary-aware transport is selected, but the embedded runner still preserves custom session streams, and the published Pi session default is itself an async wrapper around `streamSimple`, so it is not referentially equal to `streamSimple`.

## What This Changes

Ensure GitHub Copilot `openai-responses` quota 429s fail fast in embedded runner sessions instead of being misreported as idle timeouts on the session-custom stream path.

## Best Possible Solution

Make the generic embedded/provider stream boundary ensure every `openai-responses` Copilot embedded run reaches `buildGuardedModelFetch` or an equivalent guarded provider-owned transport before the idle watchdog, while preserving Copilot's header and payload wrapper behavior.

## Reproduction Assessment

Yes. The issue provides a concrete `v2026.4.29` live reproduction with immediate upstream 429 plus `session-custom` timeout telemetry, and current main plus the published Pi dependency source still show the same bypass route.

## Solution Assessment

Yes. The requested invariant is the right target; the safer implementation is a narrow generic fix in embedded stream resolution or provider stream registration for Responses transports, rather than a broad retry-policy change or a Copilot-only core branch unless tests show the generic seam is insufficient.

## Repair Work Prompt

Fix the Copilot `openai-responses` 429 fail-fast bypass tracked by https://github.com/openclaw/openclaw/issues/75411. Current main only selects `createBoundaryAwareStreamFnForModel` when the saved embedded base stream is undefined or exactly `streamSimple`, but `@mariozechner/pi-coding-agent@0.71.1` installs the default stream as an async wrapper around `streamSimple`, so Copilot's `wrapStreamFn` delegates into the dependency `openai-responses` client without OpenClaw's `buildGuardedModelFetch`. Implement a narrow generic repair so default Pi session streams for supported Responses APIs route through the boundary-aware transport or so provider stream registration returns the guarded Responses transport even without proxy/TLS overrides. Preserve Copilot header/payload wrapping, auth key injection, abort-signal forwarding, cache-boundary behavior, and explicit truly custom stream semantics unless the test describes why that stream is equivalent to the Pi default. Do not add a broad retry-policy rewrite, dependency override, or Copilot-only core branch unless a generic seam is not feasible. Add focused regression tests proving `github-copilot/gpt-5.4` with `api: "openai-responses"` no longer reports `session-custom` for the Pi default wrapper and reaches the guarded Responses fetch path; keep the existing custom-stream behavior covered for genuinely custom streams. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter, not bot or maintainer handles.
