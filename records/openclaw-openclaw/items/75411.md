## Summary

Keep open: current main still preserves the same session-custom bypass route for Copilot OpenAI Responses calls when the embedded session base stream is a Pi-managed wrapper rather than `streamSimple`, so the guarded 429 fail-fast transport is not guaranteed to run on the reported path.

## What This Changes

The issue asks OpenClaw to fail fast on GitHub Copilot OpenAI Responses HTTP 429 quota errors even when embedded runs use a `session-custom` stream instead of the boundary-aware transport.

## Best Possible Solution

Make embedded/provider stream resolution ensure Copilot `openai-responses` runs reach `buildGuardedModelFetch` or an equivalent provider-owned guarded transport before the idle watchdog, while keeping Copilot's header and payload wrapper behavior and preserving true custom stream overrides.

## Reproduction Assessment

Yes. The issue supplies a concrete `v2026.4.29` live reproduction, and current main still has a focused static reproduction path: pass a Pi-style non-`streamSimple` default wrapper into `describeEmbeddedAgentStreamStrategy`/`resolveEmbeddedAgentStreamFn` for `github-copilot/gpt-5.4` and observe `session-custom` plus unchanged stream selection.

## Solution Assessment

Yes. The requested invariant is the right target; the narrowest maintainable fix is in the generic embedded stream-resolution or provider stream-registration seam for Responses transports, not a broad retry-policy change or a Copilot-only branch unless tests prove the generic seam cannot distinguish the Pi default wrapper.
