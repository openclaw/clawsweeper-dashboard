## Summary

Keep open. Current main still leaves the reported Copilot runtime-auth path able to remain `session-custom`: the PI session default is an anonymous wrapper, Copilot stores the exchanged token in auth storage, and OpenClaw only switches non-default session streams to boundary-aware transport when `resolvedApiKey` is nonempty.

## What This Changes

The issue reports that GitHub Copilot OpenAI Responses 429s bypass guarded fail-fast handling when the embedded runner uses a session-custom stream path.

## Best Possible Solution

Carry an explicit default/prepared-stream marker, or otherwise recognize the PI-created default runtime-auth wrapper, so Copilot OpenAI Responses traffic reaches guarded transport while real custom streams still win.

## Reproduction Assessment

Yes, at source level: current main can receive a non-default PI session stream with Copilot runtime credentials only in authStorage and no `resolvedApiKey`, which still falls through as `session-custom`. I did not run the live exhausted-quota endpoint.

## Solution Assessment

Yes: routing Copilot OpenAI Responses through guarded transport or an equivalent provider-owned path is the narrow maintainable fix. Retrying, changing idle timeout defaults, or only improving error text would not preserve the fail-fast HTTP 429 behavior.
