## Summary

Current main still does not prove the reported GitHub Copilot path is fixed. The later runtime-auth boundary-aware fix covers providers where the credential is passed as `resolvedApiKey`, but Copilot stores its exchanged token in PI `authStorage`, so the anonymous PI session wrapper can still be treated as `session-custom`.

## What This Changes

The issue asks for GitHub Copilot OpenAI Responses HTTP 429s to fail fast in embedded session-custom streams instead of timing out as idle model silence.

## Best Possible Solution

Teach embedded stream resolution to recognize the PI SDK-created default session wrapper, or carry an explicit OpenClaw-prepared transport/default marker into the resolver, so Copilot OpenAI Responses traffic reaches guarded fetch handling while true custom streams still win.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can still pass an anonymous PI session stream with no `resolvedApiKey` for Copilot runtime auth, which the resolver classifies as `session-custom` instead of `boundary-aware:openai-responses`. I did not run the live exhausted-quota endpoint.

## Solution Assessment

Yes, the requested direction is still the right narrow fix: route the Copilot Responses call through guarded transport or an equivalent provider-owned path. Retry or idle-timeout tuning would mask the provider HTTP error instead of ensuring Responses traffic uses guarded 429 handling.
