## Summary

Keep open: current main still preserves the reported bypass. The guarded OpenAI Responses transport exists, but the embedded runner only selects it for an undefined or referentially identical `streamSimple` base; the locked Pi session factory supplies an async wrapper around `streamSimple`, so GitHub Copilot `openai-responses` traffic can still remain on the `session-custom` path.

## What This Changes

Ensure GitHub Copilot `openai-responses` embedded runs use guarded 429 fail-fast behavior even when the session stream strategy is `session-custom`.

## Best Possible Solution

Every embedded `openai-responses` turn should reach `buildGuardedModelFetch` or equivalent guarded provider transport before idle-timeout handling, while preserving Copilot's header and payload wrapper behavior and keeping true custom stream functions opt-in.

## Reproduction Assessment

Yes. The issue provides live `v2026.4.29` telemetry for the user-visible failure, and current main plus locked `@mariozechner/pi-coding-agent@0.71.1` statically reproduce the `session-custom` routing decision.

## Solution Assessment

Yes. The right fix is a narrow embedded/provider stream-resolution change with regression tests for the Pi default wrapper path, not a broader retry-policy change or a Copilot-only idle-timeout workaround unless the generic seam proves insufficient.
