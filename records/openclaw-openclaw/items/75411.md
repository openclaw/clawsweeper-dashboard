## Summary

Keep open: current main still has a real routing gap for the reported Copilot Responses path. The guarded fetch handles terminal 429s when the boundary-aware transport is selected, but embedded runs still preserve non-`streamSimple` session streams, and the Pi session default is an async wrapper around `streamSimple`, so this path can remain `session-custom` and miss the guarded transport.

## What This Changes

The issue requests that `github-copilot/gpt-5.4` `openai-responses` embedded runs fail fast on upstream HTTP 429 quota errors instead of surfacing them as idle timeouts through the `session-custom` stream path.

## Best Possible Solution

Make the embedded/provider stream boundary ensure every Copilot `openai-responses` embedded run reaches `buildGuardedModelFetch` or an equivalent guarded provider-owned transport before the idle watchdog, while preserving Copilot's header and payload wrapper behavior and avoiding broad retry-policy changes.

## Reproduction Assessment

Yes. The issue provides a concrete v2026.4.29 live reproduction with immediate upstream 429 and `session-custom` timeout telemetry, and current main plus the published Pi dependency source still show the same bypass route.

## Solution Assessment

Yes. The requested invariant is the right target; the safest implementation is a narrow generic fix in embedded stream resolution or provider stream registration for Responses transports, not a Copilot-only core branch unless tests show the generic seam is insufficient.
