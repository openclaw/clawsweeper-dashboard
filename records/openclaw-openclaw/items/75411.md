## Summary

Keep open. Current main still does not prove the reported GitHub Copilot runtime-auth path reaches the guarded OpenAI Responses transport: Copilot stores the exchanged token in PI auth storage, while the resolver only switches a non-default session stream to boundary-aware transport when `resolvedApiKey` is nonempty.

## What This Changes

The issue asks OpenClaw to fail fast on GitHub Copilot OpenAI Responses HTTP 429s even when embedded runs currently resolve to the `session-custom` stream path.

## Best Possible Solution

Carry an explicit default/prepared-stream marker or otherwise recognize the PI SDK-created default session wrapper so Copilot OpenAI Responses runtime-auth traffic reaches guarded transport while true custom streams still win.

## Reproduction Assessment

Yes, at source level: current main can pass an anonymous PI session stream with no `resolvedApiKey` for Copilot runtime auth, and the resolver classifies that as `session-custom`. I did not run the live exhausted-quota endpoint.

## Solution Assessment

Yes: routing Copilot OpenAI Responses traffic through guarded transport or an equivalent provider-owned path is the narrow maintainable fix. Retrying or tuning idle timeouts would mask the provider HTTP error instead of preserving fail-fast 429 handling.
