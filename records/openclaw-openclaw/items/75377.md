## Summary

Keep open. Current main and v2026.4.29 already contain the exact Telegram HTTP/1.1-only dispatcher and much of the multi-account polling hardening requested here, but the report says the observable Windows 4.29 stall and web_fetch timeout coupling still occur, so this is a remaining reliability investigation rather than an implemented cleanup close.

## What This Changes

The issue asks to disable HTTP/2 on Telegram polling dispatchers and consider per-bot undici pool isolation for Windows multi-account Telegram polling stalls and related web_fetch timeouts.

## Best Possible Solution

Treat the shipped HTTP/1.1 dispatcher change as already in place, then identify the still-failing path with fresh 4.29/current-main Windows diagnostics: Telegram `[telegram][diag]` liveness fields, `openclaw channels status --probe`, `openclaw doctor`, and representative web_fetch timeout traces. The eventual fix should be narrowly in the Telegram transport/watchdog path or the shared fetch/event-loop path identified by those diagnostics.

## Reproduction Assessment

Unclear. The issue gives plausible steps, but this review only verified source, docs, tests, release-tag behavior, and related report context; it did not run a live Windows multi-account Telegram gateway or reproduce the web_fetch timeout coupling.

## Solution Assessment

No. Applying allowH2: false again is not the best remaining fix because current main and v2026.4.29 already do that for Telegram dispatchers; the next fix should target the still-observed stall with fresh diagnostics rather than repeat an already-shipped change.
