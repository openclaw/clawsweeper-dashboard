## Summary

Keep open. Current main and the named v2026.4.29 release already force HTTP/1.1 for Telegram dispatchers and include multi-account polling hardening, but the reporter is describing a fresh Windows 4.29 reliability failure with web_fetch timeout coupling, so this should not be closed as implemented or duplicated without live diagnostics.

## What This Changes

The issue asks OpenClaw to fix Telegram long-polling stalls and coupled web_fetch timeouts on Windows 4.29, specifically by applying the existing HTTP/1.1-only dispatcher behavior to Telegram polling and considering per-bot pool isolation.

## Best Possible Solution

Keep the shipped HTTP/1.1-only Telegram dispatcher behavior, then investigate the remaining Windows 4.29 failure with fresh diagnostics from current main: Telegram liveness fields, `openclaw channels status --probe`, `openclaw doctor`, and representative web_fetch timeout traces. The eventual fix should land narrowly in the Telegram transport/watchdog path or the shared fetch/event-loop path that diagnostics identify, with #71066 and #75498 linked if they share the same root cause.
