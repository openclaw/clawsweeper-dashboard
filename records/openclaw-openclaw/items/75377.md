## Summary

Keep open. Current main and v2026.4.29 already contain the exact Telegram HTTP/1.1-only dispatcher and several multi-account polling hardening changes, but the report says the observable Windows 4.29 polling stall plus web_fetch timeout coupling still occurs. That makes this a remaining reliability investigation, not an implemented cleanup close.

## What This Changes

The issue asks to disable HTTP/2 on the Telegram polling dispatcher and consider per-bot undici pool isolation to stop Windows 4.29 Telegram polling stalls and web_fetch timeouts under multi-account long polling.

## Best Possible Solution

Keep the shipped HTTP/1.1 dispatcher change in place and investigate the still-failing path with fresh 4.29/current-main Windows diagnostics: Telegram [telegram][diag] liveness fields, openclaw channels status --probe, openclaw doctor, and representative web_fetch timeout traces. The eventual fix should be narrowly in the Telegram transport/watchdog path or the shared fetch/event-loop path proven by those diagnostics.

## Reproduction Assessment

Unclear. The issue gives plausible steps and a rollback comparison, but this review only verified source, release-tag source, tests, docs, and related context; it did not run a live Windows multi-account Telegram gateway or reproduce the web_fetch timeout coupling.

## Solution Assessment

No. Reapplying allowH2: false is not the best remaining fix because current main and v2026.4.29 already do that for Telegram and guarded web_fetch dispatchers; the next change should be diagnostic-driven against the remaining stall path.
