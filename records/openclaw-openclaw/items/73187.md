## Summary

Keep open. Current main still implements the compact Gateway startup log shape, and the requested auth/server/service-status startup summary is not emitted by the Gateway startup, post-attach, or scheduled-service paths.

## What This Changes

The issue requests restoring a structured Gateway startup summary that includes auth mode/source, server URL, agent model, loaded plugins, log file, health monitor, and heartbeat status instead of the current compact startup lines.

## Best Possible Solution

Add a redacted, deterministic Gateway startup summary through the existing startup logging/post-attach path, threading already-resolved auth, server URL, plugin, log-file, and scheduled-service state into that surface. Keep the durable file log JSONL-compatible and avoid adding a new config key unless maintainers explicitly ask for both formats.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main's startup helper and tests assert compact output, while the post-attach and scheduled-service paths do not emit the requested structured status summary.

## Solution Assessment

Yes. The narrowest maintainable path is to extend the Gateway-owned startup logging path and its focused tests; a broad logging subsystem rewrite or startup-time config migration is not needed.
