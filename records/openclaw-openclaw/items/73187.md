## Summary

Keep open. Current main still uses and tests the compact Gateway startup output, so the requested structured auth/server/plugin/log-file/service startup summary is not implemented or superseded.

## What This Changes

The issue requests restoring a structured Gateway startup log banner that includes auth mode/source, server URL, agent model, loaded plugins, log-file path, and health/heartbeat status.

## Best Possible Solution

Add a deterministic, redacted Gateway startup summary through the existing startup logging path, using already-resolved runtime state for auth, server, agent, plugins, log file, and service startup status. Prefer extending the Gateway-owned logging surface over adding a broad new configuration migration unless maintainers explicitly want both compact and structured modes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main's startup-log helper and focused tests still assert compact output, and the post-attach/scheduled-service paths do not emit the requested structured status summary.

## Solution Assessment

Yes for the requested direction, with one caveat: the narrow maintainable fix is to extend `logGatewayStartup` and its callers with redacted existing state, while leaving the exact banner style and configurability to maintainer UX judgment.
