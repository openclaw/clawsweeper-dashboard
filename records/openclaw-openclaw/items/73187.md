## Summary

Keep open. Current main still emits and tests the compact Gateway startup log shape, and no current source, docs, or related item provides the requested structured auth/server/model/plugins/log-file/service-status startup summary.

## What This Changes

The issue requests restoring or adding a structured Gateway startup log summary showing auth mode/source, server URL, agent model, loaded plugins, log file, health monitor, and heartbeat status.

## Best Possible Solution

Add a redacted, deterministic Gateway-owned startup summary on the existing startup logging path, fed by already-resolved auth, bind, model, plugin, log-file, health-monitor, and scheduled-service state. Keep file-log JSONL compatibility and avoid a new config switch unless maintainers explicitly want both output modes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current logGatewayStartup and its test assert compact output, while post-attach and scheduled-service startup paths do not emit the requested structured status summary; I did not run a live Gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable direction is to extend the existing Gateway startup logging surface and thread already-resolved runtime state into it; the exact old banner styling or a startupLogFormat option is a maintainer UX choice rather than a prerequisite.
