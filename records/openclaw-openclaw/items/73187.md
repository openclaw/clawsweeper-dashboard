## Summary

Keep open. Current main still emits the compact Gateway startup log shape and the focused tests assert that shape; the requested auth/server/service-status startup summary is not emitted by the startup, post-attach, or scheduled-service paths.

## What This Changes

Request to restore or make configurable a structured Gateway startup log that shows auth mode/source, server URL, agent model, loaded plugins, log file, and health-monitor/heartbeat status.

## Best Possible Solution

Add a redacted, deterministic Gateway startup summary through the existing startup logging/post-attach flow, using already-resolved auth, server, plugin, log-file, and scheduled-service state. Keep JSONL file logs compatible and avoid adding a new format config unless maintainers explicitly want dual output modes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main's startup helper and tests assert compact output, and the post-attach plus scheduled-service paths do not emit the requested structured status summary.

## Solution Assessment

Partly. Restoring the exact old emoji banner or adding a new config key is not necessarily required; the narrow maintainable fix is to extend the Gateway-owned startup summary with redacted status fields and focused tests.
