## What This Changes

The issue reports that memory-wiki bridge mode can recursively re-import generated files from its own wiki `sources/` directory when the vault is inside a workspace `memory/` directory.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95663 after real bridge-import proof, or replace it with the same owner-local self-source filter; avoid adding a new memory-search exclude config unless live evidence shows remaining overhead after the bridge loop is stopped.

## Reproduction Assessment

Yes from source, though I did not run a live reproduction. A focused current-main repro can publish a public artifact whose absolute path is under `config.vault.path/sources`; current bridge code feeds that artifact into import instead of filtering it.

## Solution Assessment

Yes for the central recursive bridge loop. Filtering self-originating artifacts in the memory-wiki bridge importer is the narrow plugin-owned fix and avoids adding a new config surface; any broader memory-indexer exclusion can be a follow-up if live proof shows remaining DB overhead.
