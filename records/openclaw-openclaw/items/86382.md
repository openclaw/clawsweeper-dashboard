## Label Justifications

- `P2`: This is a normal-priority backup/restore semantics gap with real disaster-recovery impact but no current core outage.
- `impact:data-loss`: Backup archives can omit durable chat transcript content that users may expect to recover after restore or migration.
- `impact:session-state`: The affected files are OpenClaw session transcripts and trajectory records tied to chat history state.

## Summary

Keep open: current main and v2026.6.6 still skip agent session transcript JSONL files during backup, so the chat-history restore contract remains unresolved. The broader native backup/restore umbrella at https://github.com/openclaw/openclaw/issues/13616 overlaps but does not replace this narrower transcript-preservation decision.

## What This Changes

Request to preserve or explicitly support agent session transcript JSONL files in `openclaw backup create` so chat history can be restored.

## Best Possible Solution

Choose a maintainer-approved backup contract for session history: a safe transcript snapshot path such as staged, quiesced, or inactive-only inclusion, or explicit docs that chat transcript restore remains outside `backup create` until native restore/export work lands.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: any state tree with `agents/<agentId>/sessions/*.jsonl` hits the volatile backup filter and is excluded from the archive; current tests assert that behavior.

## Solution Assessment

Unclear. The issue identifies a real restore-semantics gap, but simply reverting the skip is not the best fix because it would reopen the live tar EOF race; maintainers need to choose the snapshot and sensitive-data contract first.
