## Label Justifications

- `P2`: The report is a normal-priority backup/restore semantics gap affecting users who rely on backups for migration or disaster recovery.
- `impact:data-loss`: Backup archives can omit durable chat transcript content that users may expect to recover after restore.
- `impact:session-state`: The affected files are OpenClaw session transcripts and trajectory records tied to chat history state.

## Summary

Current main still omits session transcript JSONL files from backup archives, and current docs describe that omission as intentional; the issue should stay open because the requested chat-history restore contract is still unresolved and needs maintainer direction.

## What This Changes

The issue asks OpenClaw backup creation to preserve or explicitly support session transcript JSONL files so restored backups can recover chat history.

## Best Possible Solution

Add a maintainer-approved backup contract for session history: either a safe transcript snapshot path with focused tests and security notes, or explicit docs that chat transcript restore is outside `backup create` until a separate export/restore feature lands.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a state tree with `agents/<agentId>/sessions/*.jsonl` hits the volatile filter and is excluded from the tar archive.

## Solution Assessment

Unclear. The issue is right that restore completeness is unsolved, but simply reverting the filter would reintroduce live-backup EOF failures; a staged, quiesced, or opt-in transcript mode needs maintainer approval.
