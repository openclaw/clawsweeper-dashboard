## Label Justifications

- `P2`: This is a normal-priority backup/restore semantics gap with real disaster-recovery impact but no current core outage.
- `impact:data-loss`: Backup archives can omit durable chat transcript content that users may expect to recover after restore or migration.
- `impact:session-state`: The affected files are OpenClaw session transcripts and trajectory records tied to chat history state.

## Summary

Keep open: current main and v2026.6.8 still skip agent session transcript JSONL files during backup, while those JSONL files remain the durable chat-history source used by session and WebChat history. The broader native backup/restore umbrella at https://github.com/openclaw/openclaw/issues/13616 overlaps, but it does not replace this narrower transcript-preservation decision.

## What This Changes

The issue asks OpenClaw to preserve or explicitly support session transcript JSONL in backup archives so chat history can be restored, without reintroducing live-backup tar races.

## Best Possible Solution

Choose and land a maintainer-approved transcript backup contract: staged snapshot, quiesced backup, inactive-only inclusion, explicit export/restore support, or clear user-facing docs that chat transcript restore remains outside `openclaw backup create` for now.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: any state tree with `agents/<agentId>/sessions/*.jsonl` hits the volatile backup filter and is excluded from the archive, and current tests assert that omission.

## Solution Assessment

Unclear. The issue identifies a real restore-semantics gap, but reverting the skip is not the best fix because it would reopen the live tar EOF race; maintainers need to choose the snapshot and sensitive-data contract first.
