## Summary

Keep open. Current main still has the static Copilot catalog/default-limit behavior this PR is trying to replace, and the related #72824/#72825 context confirms a real user-facing capability gap, but the latest PR head still has merge-blocking correctness issues in auth scoping, cache scoping, and fallback thinking support.

## What This Changes

The PR adds runtime GitHub Copilot `/models` discovery with mapping, caching, catalog augmentation, dynamic-model preparation, thinking-profile tests, and small plugin-SDK/test cleanups.

## Best Possible Solution

Land a refined GitHub Copilot plugin implementation that fetches authenticated text-model capabilities through provider-owned hooks, preserves offline/timeout fallback behavior for known models, scopes cached capability data to the relevant credential/baseUrl/agent context, and includes focused tests for auth-profile-only discovery, multi-agent/profile switching, cache invalidation, and fallback thinking levels.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main returns an empty Copilot provider catalog with static 128K/8192 defaults, while PR head can be shown to drop `ctx.agentDir` in shared discovery, reuse a process-global cache across credential contexts, and omit `xhigh` in the no-API fallback.

## Solution Assessment

No, not as written. Dynamic `/models` discovery is the right direction, but the implementation should pass agent-scoped auth context into every discovery path, key or invalidate the cache by credential/baseUrl, and preserve current known-model thinking fallbacks when live discovery cannot run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass agentDir into model discovery:** `extensions/github-copilot/index.ts:54-57`
  - body: `fetchAndCacheModels` resolves the GitHub token without `ctx.agentDir`, even though the catalog, augmentation, and dynamic-model contexts carry it. A user authenticated only through an agent-local Copilot profile can pass the preflight path and still get an empty dynamic catalog because the shared fetch looks in the default auth store/env instead of the current agent store.
  - confidence: 0.89
- **[P2] Scope the model cache to the credential context:** `extensions/github-copilot/index.ts:44-45`
  - body: Once any request fills `cachedModelCapabilities`, every later request returns the same plan catalog without checking agent dir, auth profile, env token, config, or Copilot baseUrl. In a gateway serving multiple agents or after auth/profile changes, `/models list` and thinking caps can show stale capabilities from another Copilot account until restart.
  - confidence: 0.84
- **[P2] Keep xhigh fallback for known Copilot models:** `extensions/github-copilot/index.ts:463-464`
  - body: When the `/models` request fails, times out, or has not populated the cache, this fallback now returns the generic profile without `xhigh`. That regresses current support for known Copilot GPT/Codex models such as `gpt-5.4` and `gpt-5.2`, where main already exposes `xhigh` from the hardcoded allowlist.
  - confidence: 0.87
