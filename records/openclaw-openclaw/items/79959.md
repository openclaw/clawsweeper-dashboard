## Summary

Keep open: this is a useful Gateway attribution direction, but the PR is not mergeable as-is because protocol generation is stale, blank `originAgent` values can persist malformed provider metadata, current WebChat rendering does not consume `message.provider` for avatar/handle attribution, and external real behavior proof is missing.

## What This Changes

The branch adds an optional `originAgent` `chat.inject` parameter, threads it through the Gateway chat handler, and persists it as the injected assistant message `provider`.

## Best Possible Solution

Land the attribution feature only after the origin value is constrained and normalized, generated protocol clients are refreshed, focused Gateway/UI coverage is added, and redacted real `chat.inject` proof shows the intended persisted and visible behavior.

## Reproduction Assessment

Yes for the PR defects: CI logs reproduce the protocol-generation failure, and source inspection shows blank `originAgent` values pass validation and would be persisted. The feature itself still lacks real behavior proof from the contributor's setup.

## Solution Assessment

No: storing a provider override may be a reasonable primitive, but the current patch is incomplete without non-empty normalization, generated protocol output, UI attribution follow-through or corrected scope, and real behavior proof.

- **[P2] Regenerate protocol clients after schema changes:** `src/gateway/protocol/schema/logs-chat.ts:69`
  - body: Adding `originAgent` to `ChatInjectParamsSchema` changes the public Gateway protocol.   - confidence: 0.94

  - body: `originAgent` is accepted as a plain string and then persisted with `params.originAgent ?? "openclaw"`, so `""` or whitespace bypasses the documented default and creates an empty or malformed `provider`. Treat blank values as unset, or require a bounded non-empty normalized value before writing transcript metadata.
  - confidence: 0.9
- **[P2] Wire attribution into WebChat before relying on provider:** `src/gateway/server-methods/chat-transcript-inject.ts:97`
  - body: The PR says existing WebChat avatar/handle resolution will pick up the new provider, but current rendering groups assistant messages by role and passes only the session assistant avatar/name into `renderChatAvatar`; it does not read `message.provider`. If the user-facing fix is correct avatars/handles, this needs the UI attribution path or the claim should be narrowed and tested as transcript metadata only.
  - confidence: 0.8
