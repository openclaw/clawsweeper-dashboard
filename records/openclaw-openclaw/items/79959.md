## Summary

Keep open: the supplied terminal proof now covers the persistence/schema path, but the patch still overwrites a provider sentinel that current replay filtering and tests rely on to exclude transcript-only gateway-injected rows from later model context. Blank `originAgent` values are also accepted and persisted as attribution.

## What This Changes

The PR adds optional `originAgent` to `chat.inject`, threads it through the gateway handler, persists it as the injected assistant message `provider`, and adds two proof scripts.

## Best Possible Solution

Preserve `provider: "openclaw"` for gateway-injected transcript-only rows and add a separate validated display-attribution field or explicit transcript-only marker, with replay, schema, history/UI, and docs coverage updated together.

## Reproduction Assessment

Yes. Source inspection shows current main filters only `provider: "openclaw"` plus `model: "gateway-injected"` rows, while the PR persists `originAgent` into `provider`, so `chat.inject` with `originAgent` reproduces the introduced replay regression.

## Solution Assessment

No. Reusing `provider` is not the narrowest maintainable solution because that field is already part of the runtime replay contract; a separate display attribution field or transcript-only marker is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve the transcript-only provider sentinel:** `src/gateway/server-methods/chat-transcript-inject.ts:94`
  - body: Gateway-injected rows are filtered out of later model replay only when they keep `provider: "openclaw"` and `model: "gateway-injected"`. Persisting `originAgent` into `provider` makes injected UI notes miss that filter, so content intended for transcript display can be sent back to the next provider.
  - confidence: 0.96
- **[P2] Treat blank originAgent as absent:** `src/gateway/protocol/schema/logs-chat.ts:69`
  - body: `Type.String({ maxLength: 100 })` accepts `""` and whitespace, and the persistence path uses nullish coalescing, so blank attribution values can be stored instead of the documented `openclaw` default. Validate this as non-empty or normalize blank values before persistence.
  - confidence: 0.86
