## Summary

Keep open. Current main and the latest release have partial `descriptionLocalizations` plumbing for plugin/skill/native command paths, Discord serialization, and Telegram `language_code` sync, but the built-in `defineChatCommand` helper, Gateway command catalog, Web UI command palette, and Telegram custom-command config still expose only one fallback English description; the older Telegram-specific issue remains related but does not cover the broader built-in/API/UI scope here.

## What This Changes

The issue requests localized slash-command description fields so command names can stay stable while channel menus and UI can show translated descriptions.

## Best Possible Solution

Use one additive locale-map contract, preferably the existing `descriptionLocalizations` shape, then decide which built-in translations OpenClaw owns, how Gateway/Web UI clients choose locales, and whether Telegram custom-command config should get the same map.

## Reproduction Assessment

Yes. Current main and v2026.6.6 source show partial localization plumbing, but `DefineChatCommandInput`, `CommandEntrySchema`, Web UI command mapping, and Telegram custom-command config still expose only one description string.

## Solution Assessment

No, not exactly as proposed. The per-language field example is a useful feature sketch, but the better maintainable direction is to extend and document the existing `descriptionLocalizations` map and settle translation ownership before implementation.
