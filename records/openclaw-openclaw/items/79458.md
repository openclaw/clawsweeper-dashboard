## Best Possible Solution

Choose one canonical command-description localization contract, then wire built-in command definitions, Gateway `commands.list`, Web/TUI rendering, and native channel publication through it with focused regression coverage.

Yes for the current behavior: source inspection shows built-in command descriptions, Gateway `commands.list`, Web slash-command rendering, and TUI command lists all use a single English `description`. This is a feature gap rather than a failing-bug reproduction.

## Solution Assessment

Unclear until maintainers choose the API shape: the request is reasonable, but the maintainable fix should be a canonical localization contract rather than channel-specific overrides or isolated Chinese strings.
