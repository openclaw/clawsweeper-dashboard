## Summary

Keep open. Current main still misses the reported Feishu fallback shape: interactive-card extraction only reads flat card-template elements, while the rich-post fallback parser requires `content` or `post`/locale wrappers rather than top-level `title` plus nested `elements`. The linked same-author PR #60383 remains open and uses closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Fix Feishu interactive-card parsing when the API returns post-format fallback content with a top-level title and nested element arrays.

## Best Possible Solution

Land a Feishu-plugin-local parser and regression-test update that normalizes or recursively renders top-level fallback `elements` arrays, includes title/header text, and reuses the existing post renderer for `text`, `a`, and media placeholders where practical.

## Reproduction Assessment

Yes. A focused `getMessageFeishu` unit test can mock `client.im.message.get` with `msg_type: "interactive"` and the issue's top-level `{ title, elements: [[...]] }` body content; current source inspection shows that path returns `[Interactive Card]`.

## Solution Assessment

Yes. A narrow Feishu parser/test patch is the best way to solve this owner-specific parsing gap; no core API or cross-plugin change appears necessary.
