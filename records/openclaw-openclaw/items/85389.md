## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Resolve sender IDs before remapping chunk lines:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:717-718`
  - body: `remapChunkLines` rewrites chunk line numbers from flattened content positions to original JSONL line numbers, but this PR then uses those remapped values to index `messageSenderIds`. That can assign a chunk to the next user, the wrong user, or `undefined`, so mixed-user session memories can still be mis-scoped. Resolve sender IDs from the content-relative chunk lines before remapping, or carry a separate content-line range for attribution.
  - confidence: 0.94
- **[P1] Rebuild FTS tables instead of altering them:** `packages/memory-host-sdk/src/host/memory-schema.ts:79-82`
  - body: SQLite FTS5 virtual tables cannot be altered in place; the `ALTER TABLE ... ADD COLUMN sender_id` path throws on an existing `chunks_fts`, but the catch treats that as expected and leaves `ftsAvailable` true. Existing users then hit `no such column: sender_id` on FTS inserts/search after upgrade. Detect the old schema and rebuild/reindex the FTS table, or disable FTS with an actionable repair path.
  - confidence: 0.97
- **[P1] Do not leave legacy memories globally recallable:** `extensions/memory-core/src/memory/manager-sync-ops.ts:349`
  - body: The filter explicitly includes `OR sender_id IS NULL`, and the LanceDB path similarly keeps rows without a sender. For this privacy fix, pre-migration rows are the data most likely to contain the original leak, so they remain injectable to every sender unless the PR adds a tested migration/reindex/delete path or a maintainer-approved legacy policy.
  - confidence: 0.87
- **[P2] Use an unaliased sender filter in chunk fallback search:** `extensions/memory-core/src/memory/manager-search.ts:215`
  - body: The fallback call passes `sf` built for alias `c`, but `searchChunksByEmbedding` queries `FROM chunks` without that alias. When sqlite-vec is unavailable and `senderId` is set, the generated SQL references `c.sender_id` and the vector fallback fails instead of recalling scoped memories. Pass an unaliased filter for the fallback or alias the chunks table there.
  - confidence: 0.92
- **[P2] Scope LanceDB recall before applying the top-k limit:** `extensions/memory-lancedb/index.ts:282-305`
  - body: `vectorSearch(vector).limit(limit)` retrieves the global nearest memories first and only then filters by sender. If the nearest rows belong to another sender, the current sender can get no memories even when scoped matches exist just below the limit. The LanceDB backend needs to search within the sender scope, or otherwise fetch enough candidates to preserve the requested limit after filtering.
  - confidence: 0.82
