## Summary

Keep open. Current main still has the iMessage RPC stdin write path without a stream error listener, and the proposed fix is narrow, owner-local, and consistent with the existing Codex app-server handling for the same async EPIPE class. I found no blocking correctness or security findings; the remaining work is maintainer mergeability refresh and validation.

## What This Changes

The PR adds an iMessage RPC child stdin error listener routed through `failAll()`, adds a focused EPIPE regression test, and adds an Unreleased changelog fix entry.

## Best Possible Solution

Land a rebased version of this iMessage-local stream error handler after targeted iMessage tests and the changed gate pass; keep the global uncaught EPIPE handling as a backstop rather than relying on it for plugin RPC lifecycle cleanup.

## Reproduction Assessment

Yes. Current main registers a pending request and then writes to child stdin without a stream error listener; the PR’s mocked child-process test reproduces the failing path by emitting EPIPE on stdin while a request is pending.

## Solution Assessment

Yes. A child-stdin `error` listener routed through the existing pending-request cleanup is the narrowest maintainable fix inside the iMessage plugin. A per-write callback could be a later refinement, but it would not replace the need to consume async stream errors.
