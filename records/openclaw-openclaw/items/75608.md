## Summary

Keep open. Current main still has the iMessage RPC stdin write without a callback or stdin stream error listener, and the PR targets the same reliability gap described in #75438 with a narrow local listener, focused test, and changelog entry. I did not find a blocking correctness or security issue in the diff, but the exact branch still needs normal mergeability and CI validation before landing.

## What This Changes

The PR adds an iMessage RPC child stdin error listener, a unit test that emits EPIPE on the mocked stdin stream, and an Unreleased changelog fix entry.

## Best Possible Solution

Land a rebased version of this local iMessage RPC stdin error handling after targeted extension tests and the changed gate pass; keep the fix inside the iMessage plugin rather than relying on global uncaught-EPIPE suppression.

## Reproduction Assessment

Yes. Current main shows a pending request is registered before an unguarded child.stdin.write(), and Node writable streams emit an error event for write failures; the PR test provides a focused in-process reproduction by emitting EPIPE on the mocked stdin stream.

## Solution Assessment

Yes. A local child.stdin error listener routed through failAll() is a narrow maintainable fix for async stream write errors; a per-write callback could be a follow-up refinement, but it is not required to consume the uncaught stream error this PR targets.
