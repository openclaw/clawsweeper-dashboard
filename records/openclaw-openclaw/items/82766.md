## Summary

Keep open: current main still routes main-session cron work through the target human main-session lane, and an open closing PR is already attached. The protected `maintainer` label also means this needs explicit maintainer handling rather than cleanup closure.

## What This Changes

Route scheduled main-session cron wake work to a cron-owned non-interactive lane so human web-chat turns are not queued behind cron runs.

## Best Possible Solution

Review and land or reject the linked fix PR; the desired end state is for main-session cron wake work to run on a cron-owned lane while preserving the target main session's delivery context.

## Reproduction Assessment

Yes, from source inspection: current main enqueues and wakes `sessionTarget="main"` cron jobs with the job's main session key, and the heartbeat runner serializes by the resolved session lane. I did not run a live reproduction because this review was read-only.

## Solution Assessment

Yes: isolating cron execution while preserving the target main session delivery context is the narrowest maintainable fix. The open linked PR is the right path to review instead of closing this issue early.
