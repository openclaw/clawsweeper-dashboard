## Summary

Keep this PR open: current main still leaves Docker sandbox PID limits unset by default, but the proposed fallback needs a validated opt-out contract, docs/schema/test follow-through, maintainer security/product approval, and real Docker behavior proof before merge.

## What This Changes

Adds a `DEFAULT_SANDBOX_PIDS_LIMIT` constant of 1024 and uses it as the fallback in Docker sandbox config resolution when no agent or global `pidsLimit` is configured.

## Best Possible Solution

Approve the PID-cap default and opt-out semantics, then land this PR or a narrow replacement that updates resolver constants, schema/types, docs, focused tests, changelog, and real Docker proof while leaving broader CPU/memory defaults to https://github.com/openclaw/openclaw/issues/60827.

## Reproduction Assessment

Yes, at source level: resolve Docker sandbox config with no global or per-agent `pidsLimit`, then pass it to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`. I did not start a real Docker container in this read-only review.

## Solution Assessment

No as submitted. The resolver fallback is the right narrow implementation point, but the patch needs approved default/opt-out semantics plus schema, type/docs, tests, and real Docker proof before it is the best maintainable fix.
