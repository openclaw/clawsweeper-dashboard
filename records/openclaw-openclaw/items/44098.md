## Summary

The PR is a valid, narrow sandbox hardening proposal, but it should stay open because current main still leaves omitted Docker `pidsLimit` undefined and the patch changes the public opt-out/default contract without aligning schema, type docs, tests, or real Docker proof. The existing linked discussion is relevant and positive: it directly addresses https://github.com/openclaw/openclaw/issues/38604 and overlaps the broader resource-limit follow-up at https://github.com/openclaw/openclaw/issues/60827, but this PR is not obsolete or implemented on main.

## What This Changes

The branch adds a `DEFAULT_SANDBOX_PIDS_LIMIT = 1024` constant and uses it as the fallback Docker sandbox PID limit when neither agent nor global config sets `pidsLimit`.

## Best Possible Solution

Land a narrow sandbox hardening change after approving the PID-limit default and opt-out contract, then align schema, exported type/docs, focused tests, and real Docker proof while leaving broader CPU and memory defaults to https://github.com/openclaw/openclaw/issues/60827.

## Reproduction Assessment

Yes, at source level: resolve Docker sandbox config with no global or per-agent `pidsLimit`, then pass it to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`. I did not start a real Docker container in this read-only review.

## Solution Assessment

No as submitted. The resolver fallback is the right narrow implementation point, but the patch needs an approved opt-out contract plus schema, type/docs, tests, and real behavior proof before it is the best maintainable fix.
