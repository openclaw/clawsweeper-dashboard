## Summary

Keep this PR open. Current main and the latest release still leave Docker sandbox PID limits unset unless configured, so the underlying hardening gap is real; the PR is a narrow plausible fix, but it changes a security-sensitive default and needs maintainer review, docs/schema/test alignment, and green checks before landing.

## What This Changes

The PR adds `DEFAULT_SANDBOX_PIDS_LIMIT = 1024` and uses it as the fallback `pidsLimit` in Docker sandbox config resolution when neither agent nor global Docker config sets one.

## Best Possible Solution

Land a maintained sandbox hardening change that applies a chosen default PID limit in `resolveSandboxDockerConfig`, preserves explicit global and per-agent overrides, defines any unlimited opt-out in schema/types/docs, adds resolver and Docker-create coverage, and coordinates with #60827 for broader CPU and memory defaults.

## Reproduction Assessment

Yes. Resolve Docker sandbox config with no `pidsLimit` on current main and then build Docker create args; the resolved config leaves `pidsLimit` undefined and the args omit `--pids-limit`.

## Solution Assessment

No, not as submitted. The resolver fallback is the right narrow code boundary, but the PR should also align docs/schema/tests and opt-out semantics before it is the best maintainable fix.
