## Summary

Keep this PR open. Current main still leaves Docker sandbox PID limits unset, so the security hardening is useful, but the branch changes the default sandbox resource policy without aligning the public config contract, validated opt-out semantics, tests, docs, or changelog.

## What This Changes

The PR adds a `DEFAULT_SANDBOX_PIDS_LIMIT` constant set to `1024` and uses it as the fallback Docker sandbox `pidsLimit` when neither agent nor global config sets one.

## Best Possible Solution

Land a maintained sandbox hardening change that applies a chosen default PID limit in `resolveSandboxDockerConfig`, preserves explicit per-agent and global overrides, defines the supported unlimited/default opt-out in schema/types/docs/create-args, adds resolver and Docker-create coverage, includes a changelog entry crediting the contributor, and coordinates the remaining CPU/memory defaults with #60827.

## Reproduction Assessment

Yes. Source inspection on current main shows an unset `pidsLimit` resolves to `undefined`, and Docker create args only include `--pids-limit` for positive numeric values, so the reported missing default is reproducible without running a mutating container test.

## Solution Assessment

No, not as submitted. The resolver fallback is the right narrow implementation boundary, but the PR should align opt-out semantics, schema/types/docs, tests, and changelog before it is the best maintainable fix.
