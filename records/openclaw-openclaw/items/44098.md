## Summary

Keep open: the PID-limit hardening is still relevant on current main, but this PR is not merge-ready because it changes the sandbox default without aligning the validated opt-out contract, schema/docs/tests, or contributor real Docker behavior proof.

## What This Changes

The PR adds a `DEFAULT_SANDBOX_PIDS_LIMIT` constant set to 1024 and uses it as the fallback Docker sandbox `pidsLimit` when agent/global config omits the field.

## Best Possible Solution

Land this or a narrow replacement after approving the PID cap and opt-out contract, then update resolver, schema/types, docs, tests, and changelog while leaving broader resource budgets to the related default-limits issue.

## Reproduction Assessment

Yes. A safe source-level reproduction is to resolve Docker sandbox config with no global or per-agent `pidsLimit`, then pass that config to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`.

## Solution Assessment

No as submitted. The resolver fallback is the right narrow implementation point, but the patch needs an approved opt-out contract plus schema, docs, tests, and real behavior proof before it is the best maintainable fix.
