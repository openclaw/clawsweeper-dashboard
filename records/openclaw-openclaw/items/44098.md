## Summary

Keep open: current main still leaves sandbox PID limits unset, and this PR is useful security hardening but needs maintainer policy follow-through for the default cap, opt-out contract, docs/tests, and changelog before merge.

## What This Changes

The branch adds `DEFAULT_SANDBOX_PIDS_LIMIT = 1024` and falls back to it in `resolveSandboxDockerConfig` when neither agent nor global Docker config sets `pidsLimit`.

## Best Possible Solution

Land a maintained sandbox resource-default change that sets an approved PID cap, preserves explicit per-agent/global overrides, documents and validates the unlimited/default opt-out, adds resolver/create-args coverage, updates public docs/schema help, includes a changelog entry, and coordinates broader resource-limit policy with #60827.

## Reproduction Assessment

Yes. Source inspection on current main is enough: an unset `pidsLimit` resolves to `undefined`, and Docker create args omit `--pids-limit` unless the resolved value is a positive number.

## Solution Assessment

No, not as submitted. The resolver fallback is the right narrow implementation point, but the PR needs an approved opt-out contract plus docs/schema/tests/changelog follow-through before it is the best maintainable fix.
