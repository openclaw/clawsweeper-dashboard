## Summary

Keep open: current main still leaves sandbox Docker containers without a default PID cap, and this PR is a useful narrow start, but the default value and unlimited/default opt-out semantics need maintainer security-policy approval plus docs, schema, tests, and changelog follow-through before merge.

## What This Changes

The PR adds a `DEFAULT_SANDBOX_PIDS_LIMIT = 1024` constant and uses it as the fallback when sandbox Docker config has no agent or global `pidsLimit`.

## Best Possible Solution

Land an approved sandbox resource-default change that sets a documented PID cap, preserves explicit overrides, defines the opt-out behavior, updates schema/docs/tests/changelog, and coordinates with #60827 for broader resource limits.

## Reproduction Assessment

Yes. Source inspection on current main is enough: an unset `pidsLimit` resolves to `undefined`, and Docker create args omit `--pids-limit` unless the resolved value is positive.

## Solution Assessment

No, not as submitted. The resolver fallback is the right narrow implementation point, but the security default and opt-out contract need approval and public contract follow-through before this is the best maintainable fix.
