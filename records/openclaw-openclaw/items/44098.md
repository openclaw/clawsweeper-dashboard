## Summary

Keep this PR open. Current main still has no default Docker sandbox PID cap, so the hardening is not obsolete, but the patch needs maintainer/security agreement on the default and opt-out contract plus docs, tests, and changelog before merge.

## What This Changes

The PR adds a 1024 sandbox Docker PID-limit constant and uses it as the fallback in `resolveSandboxDockerConfig` when neither per-agent nor global config sets `pidsLimit`.

## Best Possible Solution

Land a maintained sandbox resource-default change that sets an agreed PID cap in the resolver, preserves explicit per-agent/global overrides, defines and validates the unlimited/default opt-out, updates public docs and schema help, adds resolver/create-args coverage, includes a changelog entry crediting the contributor, and coordinates broader CPU/memory defaults with #60827.

## Reproduction Assessment

Yes. Source inspection on current main is enough: an unset `pidsLimit` resolves to `undefined`, and Docker create args omit `--pids-limit` unless the value is a positive number.

## Solution Assessment

No, not as submitted. The resolver fallback is the narrow implementation boundary, but the patch needs schema/docs/tests/changelog follow-through and an owner-approved opt-out contract before it is the best maintainable fix.
