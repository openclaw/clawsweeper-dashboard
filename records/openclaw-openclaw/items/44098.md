## Summary

Keep open: current main still leaves Docker sandbox `pidsLimit` unset, so this PR addresses a real hardening gap, but the branch is not merge-ready because it changes the default without aligning the opt-out/schema/docs/test contract or providing real Docker behavior proof.

## What This Changes

The PR adds a `DEFAULT_SANDBOX_PIDS_LIMIT = 1024` constant and uses it as the fallback Docker sandbox `pidsLimit` when agent/global config omits the field.

## Best Possible Solution

Land this or a narrow replacement after maintainers approve the PID cap and opt-out contract, then update resolver, schema/types, docs, tests, and changelog together while leaving broader CPU/memory defaults to https://github.com/openclaw/openclaw/issues/60827.

## Reproduction Assessment

Yes. A safe source-level reproduction is to resolve Docker sandbox config with no agent/global `pidsLimit`, then pass that config to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`.

## Solution Assessment

No as submitted. The resolver fallback is the right narrow implementation point, but the patch needs an approved opt-out contract plus schema, docs, tests, and real behavior proof before it is the best maintainable fix.
