## Summary

Keep open: the PR addresses a real Docker sandbox PID-limit hardening gap, but it changes the default without a supported opt-out/config contract or real after-fix proof.

## What This Changes

This PR adds a 1024 PID-limit constant and uses it as the fallback `pidsLimit` in Docker sandbox config resolution.

## Best Possible Solution

Land a narrowed hardening patch that applies an approved PID default, preserves documented override/opt-out semantics through schema and types, adds focused resolver/create-args tests, updates user docs and changelog, and leaves CPU/memory defaults to the broader resource-limit issue.

## Reproduction Assessment

Yes, at source level. Resolving Docker sandbox config without `pidsLimit` leaves it undefined on current main, and Docker create args omit `--pids-limit` unless a positive value is present.

## Solution Assessment

No as submitted. The resolver fallback is the right implementation point, but the patch should also settle the default/opt-out contract and update schema, docs, and tests before merge.
