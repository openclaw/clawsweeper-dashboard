## Summary

Keep open: the hardening target is valid and current main still lacks a default sandbox Docker `pidsLimit`, but this PR should not merge as-is because it changes unset configs to a hard `1024` cap without an explicit valid Docker-default opt-out, docs/tests, or real Docker proof.

## What This Changes

The PR adds a `DEFAULT_SANDBOX_PIDS_LIMIT` constant and uses it as a fallback when sandbox Docker config does not specify `pidsLimit`.

## Best Possible Solution

Land a narrow PID-cap default only after maintainers approve the default value and opt-out semantics, then align schema/types/docs/tests and include redacted real Docker proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolveSandboxDockerConfig` leaves `pidsLimit` undefined when omitted and `buildSandboxCreateArgs` emits no `--pids-limit` unless the value is positive.

## Solution Assessment

No. The resolver fallback is a plausible implementation point, but the PR is not the best complete fix until it preserves an explicit opt-out contract, aligns schema/docs/tests, and provides real Docker proof.
