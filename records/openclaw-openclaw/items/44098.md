## Summary

Keep open: this PR targets a real, still-unfixed Docker sandbox PID-cap gap, but as submitted it changes upgrade behavior without a validated opt-out contract and lacks real Docker behavior proof plus schema/docs/test follow-through.

## What This Changes

Adds a `DEFAULT_SANDBOX_PIDS_LIMIT` constant set to 1024 and uses it as the fallback `pidsLimit` in Docker sandbox config resolution.

## Best Possible Solution

Approve the PID-cap default and opt-out semantics, then land this PR or a narrow replacement that updates resolver constants, schema/types, docs, focused tests, changelog, and real Docker proof while keeping broader CPU/memory defaults with the related resource-limits issue.

## Reproduction Assessment

Yes, at source level: resolve Docker sandbox config with no global or per-agent `pidsLimit`, then pass it to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`. I did not start a real Docker container in this read-only review.

## Solution Assessment

No as submitted. The resolver fallback is the right narrow implementation point, but the patch needs approved default/opt-out semantics plus schema, type/docs, tests, and real Docker proof before it is maintainable.
