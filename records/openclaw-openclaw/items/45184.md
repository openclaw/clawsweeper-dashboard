## Summary

Keep open: current main and the latest release still only evaluate whether the receiving Matrix account was mentioned, so a `requireMention: false` account can reach dispatch even when the message explicitly targets a sibling account. The remaining work is real, but it needs Matrix product/config design and coordination with the open participation PR and sibling outbound mention issue.

## What This Changes

Request an opt-in Matrix policy that suppresses dispatch for non-mentioned bot accounts when a room message explicitly mentions another configured Matrix account.

## Best Possible Solution

Add a Matrix-plugin-owned opt-in sibling-target suppression gate before typing and reply dispatch, with documented roster semantics, config/schema support, and regression coverage coordinated with the broader Matrix participation and outbound mention work.

## Reproduction Assessment

Yes, at source level. With `requireMention: false`, a room event whose Matrix mention targets another configured account is not dropped because current main only checks whether `selfUserId` was mentioned before continuing toward dispatch.

## Solution Assessment

Unclear. An early opt-in Matrix skip is plausible, but maintainers still need to choose the config shape, sibling roster source, alias behavior, and how this relates to broader Matrix participation work.
