## Summary

Current main still admits a Matrix room message for an account with `requireMention: false` even when validated mention metadata targets only a different configured account, and no merged canonical fix exists. The issue should remain open for a maintainer decision on cross-channel `ignoreOtherMentions` semantics; likely related people are @gumadeiras and @nickludlam with high confidence from Matrix mention history, plus @vincentkoc as a recent handler contributor with medium confidence.

## What This Changes

Add an opt-in Matrix admission rule that skips typing and model dispatch when a room message explicitly mentions a different configured bot account but not the current account.

## Best Possible Solution

Define one shared `ignoreOtherMentions`-style channel semantic, then implement Matrix admission deterministically from validated mention metadata and a prepared configured-account MXID set before typing or model dispatch; preserve untargeted room traffic and explicitly test mixed, human-only, multi-bot, command, and `@room` cases.

## Reproduction Assessment

Yes. On current main, configure two Matrix accounts in one room, set the non-target account to `requireMention: false`, and send a human event whose validated user mention targets only the sibling account; the non-target handler remains eligible because admission checks only whether the current account was mentioned when mention gating is enabled.

## Solution Assessment

No, the broad AI participation and freshness PR is not the best solution to this narrow issue. A small deterministic pre-dispatch policy aligned with the existing Discord concept and open Slack proposal is more maintainable, but maintainers must first define mixed-mention behavior and the shared config contract.

## Risks / Open Questions

- A generic “some other mention exists” rule would incorrectly suppress bots when users mention ordinary humans; target detection must intersect validated mentions with configured sibling Matrix user IDs.
- Mixed targeting is undefined: messages mentioning this bot plus a sibling, multiple siblings, or `@room` need an explicit product contract before implementation.
- Sibling-generated output may omit usable Matrix mention metadata as tracked in https://github.com/openclaw/openclaw/issues/80432, so text-only heuristics should not silently become the routing contract.
- Adding a Matrix-only config name before deciding the shared Discord/Slack/Matrix semantic could create channel drift and another compatibility-sensitive setting.
