## Summary

Keep this PR open: current main still has the zsh description escaping bug and the PR is a useful focused fix, but it is incomplete because subcommand descriptions still flow through the same generated zsh surface without `$` or backtick escaping.

## What This Changes

The PR adds a zsh option-description escaping helper for `$` and backticks and adds regression assertions in the completion CLI test.

## Best Possible Solution

Land one zsh escaping fix that applies expansion-safe description escaping to both option and subcommand descriptions, with focused `$` and backtick regression coverage in `src/cli/completion-cli.test.ts`.

## Reproduction Assessment

Yes. Current main source shows `$OPENCLAW_STATE_DIR` and backtick-containing descriptions emitted into zsh double-quoted completion specs without complete escaping, and the contributor’s terminal screenshots show the reported zsh completion error before and no error after the PR.

## Solution Assessment

No, not as submitted. Escaping option descriptions is the right direction, but the narrow complete fix must also protect subcommand descriptions generated by `generateZshSubcmdList`.
