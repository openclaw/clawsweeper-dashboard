## Summary

Keep open: this PR fixes zsh option-description escaping, but subcommand descriptions still flow through the same double-quoted zsh `_arguments` surface without escaping `$` or backticks.

## What This Changes

The PR adds a helper to escape `$` and backticks in generated zsh option descriptions and adds focused completion-output assertions for those characters.

## Best Possible Solution

Use one shared zsh description-escaping path for every description emitted into double-quoted `_arguments` specs, with regression coverage for option and subcommand descriptions containing `$` and backticks.

## Reproduction Assessment

Yes. Current main source shows `$` and backtick-bearing descriptions emitted into double-quoted zsh specs, and the zsh manual confirms those characters expand in that context; I did not run a live zsh repro in this read-only review.

## Solution Assessment

No. The helper is the right direction for option descriptions, but the patch leaves subcommand descriptions on the same unescaped double-quoted path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

Current main has backtick-bearing hooks install/update descriptions, so after this patch `openclaw hooks <TAB>` can still trigger command substitution instead of displaying literal text; route subcommand descriptions through the same helper and cover that path.
