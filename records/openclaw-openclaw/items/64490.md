## Summary

Keep this PR open: it addresses a real zsh completion escaping bug, but the patch is incomplete because the same generated zsh description surface still leaves subcommand descriptions vulnerable to `$` and backtick expansion.

## What This Changes

The branch adds a zsh option-description escaping helper and regression assertions for `$OPENCLAW_STATE_DIR` plus a backtick command example in completion output.

## Best Possible Solution

Land one zsh escaping fix that applies expansion-safe description escaping to both option and subcommand descriptions, with focused `$` and backtick coverage in `src/cli/completion-cli.test.ts`.

## Reproduction Assessment

Yes, at source level: current main emits `$OPENCLAW_STATE_DIR` and backtick-containing descriptions into generated zsh description paths without complete expansion escaping. I did not run an interactive zsh session in this read-only review.

## Solution Assessment

No, not as submitted. Escaping option descriptions is the right direction, but the narrow complete fix must also protect subcommand descriptions generated by `generateZshSubcmdList`.
