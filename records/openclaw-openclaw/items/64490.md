## Summary

Keep open: the PR is a useful focused bug fix with real zsh proof for the option-description path, but it still leaves subcommand descriptions exposed to the same `$` and backtick expansion surface.

## What This Changes

The PR adds a zsh description-escaping helper for option descriptions and regression assertions for `$OPENCLAW_STATE_DIR` and backtick-wrapped command text in generated completion output.

## Best Possible Solution

Use context-aware escaping for every zsh description emitted through double-quoted `_arguments` specs, while preserving the existing `_values` single-quote escaping needed for subcommand entries.

## Reproduction Assessment

Yes. Current main source shows `$` and backtick-bearing descriptions emitted into double-quoted zsh specs, the zsh manual confirms those expand there, and the contributor screenshots show the reported `openclaw models <TAB>` failure before and success after for the option path.

## Solution Assessment

No. The helper is the right direction for option descriptions, but the patch should also escape `$` and backticks for subcommand descriptions without dropping their existing single-quote handling.
