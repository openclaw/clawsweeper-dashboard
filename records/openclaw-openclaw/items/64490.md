## Summary

Keep this PR open: current main still emits unescaped `$` and backticks in zsh option-description specs, while the PR applies a focused generator-layer fix with regression coverage and direct terminal screenshot proof.

## What This Changes

The PR routes zsh option descriptions through a double-quoted-context escaper for `$` and backticks and adds regression assertions for those escaped descriptions.

## Best Possible Solution

Land the focused option-description escaping fix after normal maintainer validation; keep positional-argument completion work separate in the related zsh positional-completion PR.

## Reproduction Assessment

Yes, source-level: current main embeds `$OPENCLAW_STATE_DIR` and backtick-wrapped command examples into generated zsh option descriptions without escaping shell-substitution characters. I did not run a local zsh repro in this read-only review, but the PR screenshots show the before error and after completion menu.

## Solution Assessment

Yes for the reported option-description bug: escaping at the zsh generator layer is the narrow maintainable fix. The sibling subcommand path uses single-quoted entries, so it does not need the same double-quoted escaping rule for `$` and backticks.
