## Summary

Keep open: current main and the latest release still have the zsh description-expansion gap, and this PR targets the right option-description path, but the patch remains incomplete because subcommand descriptions use a separate unescaped path.

## What This Changes

The PR extracts zsh option-description escaping into a helper, escapes `$` and backticks for option descriptions, and adds focused completion CLI regression assertions.

## Best Possible Solution

Use one shared zsh double-quoted-description escaping helper for every option and subcommand description emitted into `_arguments`, with regression coverage for `$` and backticks in both paths and exact real zsh proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main emits `$` and backtick-bearing descriptions inside double-quoted zsh specs, and the upstream zsh grammar says parameter and command substitution occur inside double quotes; I did not run a live local zsh repro because zsh is not installed in this container.

## Solution Assessment

No, not yet. The option-description helper is the right direction, but the same escaping must cover subcommand descriptions before the zsh expansion bug is fully fixed.
