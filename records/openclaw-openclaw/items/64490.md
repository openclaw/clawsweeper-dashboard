## Summary

Keep open: current main still has the zsh description escaping gap, and this PR fixes option descriptions but remains incomplete because subcommand descriptions are emitted through the same generated zsh surface without `$` or backtick escaping. The contributor’s terminal screenshot is adequate real behavior proof for the option-path symptom, so the remaining blocker is a narrow code/test repair.

## What This Changes

The branch adds zsh option-description escaping for `$` and backticks and regression assertions for `$OPENCLAW_STATE_DIR` plus a backtick command example.

## Best Possible Solution

Land one zsh escaping fix that applies expansion-safe description escaping to both option and subcommand descriptions, with focused `$` and backtick coverage in `src/cli/completion-cli.test.ts`.

## Reproduction Assessment

Yes: current main has `$OPENCLAW_STATE_DIR` and backtick-containing descriptions flowing into generated zsh description paths without complete expansion escaping. I did not run a live zsh session, but the source path is direct and the contributor screenshot shows the option-path symptom fixed after the patch.

## Solution Assessment

No, not as-is. Escaping option descriptions is the right direction, but the narrow complete fix must also protect subcommand descriptions generated by `generateZshSubcmdList`.
