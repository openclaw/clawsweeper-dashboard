## Summary

Keep open. Current main still lacks the requested normal-chat configurable delivery policy: the code preserves mutating tool-warning visibility ahead of `messages.suppressToolErrors`, and the shipped changelog explicitly leaves this issue as the broader configurable-policy follow-up.

## What This Changes

Request a normal-chat config option or policy to suppress transient tool-error warning messages after successful agent recovery, such as `toolErrorDelivery: "all" | "final" | "off"` or a global `suppressToolErrorWarnings`.

## Best Possible Solution

Define one documented normal-chat tool-error delivery policy that preserves visible unresolved mutating failures by default, then align config types, schema/help, public docs, payload handling, and focused tests around that policy.

## Reproduction Assessment

Yes. The existing focused payload tests reproduce the current behavior: a success-looking assistant reply plus a mutating `lastToolError` still emits a warning payload, and `messages.suppressToolErrors` does not suppress mutating warnings.

## Solution Assessment

Unclear. The requested `all`/`final`/`off` shape is plausible, but the safest maintainable fix needs an explicit maintainer policy decision because simply moving `suppressToolErrors` ahead of the mutating guard would change the current safety contract.
