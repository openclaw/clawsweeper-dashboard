## Summary

Keep open: current main and the latest release include the broad `messages.suppressToolErrors` opt-out, but they still lack the requested normal-chat `all | final | off` delivery policy for recovered tool failures after success-looking replies.

## What This Changes

The issue requests a configurable normal-chat tool-error warning policy so hosted or team-chat deployments can suppress transient recovered tool failures from user-facing channels.

## Best Possible Solution

Keep this as the canonical product/config decision, then implement the chosen warning-delivery policy with schema/help/docs alignment and focused payload plus channel regression coverage.

## Reproduction Assessment

Yes, at source level. Current policy and tests show mutating tool warnings still appear after success-looking replies unless the broad suppression flag is enabled or the assistant explicitly acknowledges the failed action.

## Solution Assessment

Unclear until maintainers choose the product policy. The existing broad `messages.suppressToolErrors` flag covers an opt-in off case, but the requested `final` mode changes the safety boundary for failed mutating actions after optimistic assistant text.
