## Summary

Keep open. Current main still lacks the requested normal-chat delivery policy: `messages.suppressToolErrors` is only a partial warning switch, heartbeat suppression is run-scoped, and mutating `write`/`edit` failures can still emit warnings after success-looking assistant text.

## What This Changes

Request a normal-chat config option that suppresses or defers transient tool-error warnings when an agent later recovers successfully.

## Best Possible Solution

Define one documented normal-chat tool-error delivery policy that preserves unresolved mutating-failure safety while giving managed deployments a clear way to hide or reconcile recovered transient warnings, then align config types, schema/help, docs, tests, and changelog.

## Reproduction Assessment

Yes, source-level. Current tests show `messages.suppressToolErrors` suppresses non-mutating warnings but still expects mutating `write` failures, including after success-looking assistant text, to produce a warning payload.

## Solution Assessment

Unclear. The requested `all`/`final`/`off` style policy is plausible, but maintainers need to choose how it interacts with unresolved mutating failures before a safe implementation can be selected.
