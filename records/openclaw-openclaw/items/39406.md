## Summary

Keep open. Current main still lacks the requested normal-chat delivery policy: `messages.suppressToolErrors` is only partial, the exposed `suppressToolErrorWarnings` config remains heartbeat-scoped, and mutating `write`/`edit` failures can still produce visible warnings after success-looking assistant text.

## What This Changes

Request a config policy to hide or delay transient tool-error warnings after recovered or successful agent turns.

## Best Possible Solution

Define one documented normal-chat tool-error delivery policy that preserves unresolved mutating-failure safety while giving managed deployments a clear opt-in way to hide or reconcile recovered transient warnings.

## Reproduction Assessment

Yes, source-level. Current payload tests show `messages.suppressToolErrors` suppresses non-mutating warnings but still expects mutating `write` failures, including after success-looking assistant text, to produce a warning payload.

## Solution Assessment

Unclear. The requested `all`/`final`/`off` policy is plausible, but maintainers need to choose how it interacts with unresolved mutating failures before a safe implementation can be selected.
