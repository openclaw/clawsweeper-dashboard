## Summary

Keep open: current main and the latest release cover the explicit opt-out case with `messages.suppressToolErrors`, but the requested conditional `all | final | off` normal-chat policy remains unresolved and needs a maintainer product/safety decision.

## What This Changes

The issue requests a configurable normal-chat tool-error delivery policy so recovered transient tool failures can remain internal instead of producing confusing warning bubbles after a successful final reply.

## Best Possible Solution

Define one canonical `messages`-level delivery policy that keeps the conservative default, preserves or migrates the existing full-suppression switch, and tests recovered-success, ambiguous-success, acknowledged-failure, and unresolved-failure turns.

## Reproduction Assessment

Yes, at source level. Current payload tests construct a success-looking final reply plus a failed mutating tool and assert that a separate warning remains visible; I did not run tests because this review is read-only.

## Solution Assessment

Unclear as proposed. The existing `off` outcome is already available through `messages.suppressToolErrors`, but the `final` mode needs maintainer-defined safety semantics before it can be the narrowest maintainable fix.
