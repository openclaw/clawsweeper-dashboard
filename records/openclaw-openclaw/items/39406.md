## Summary

Keep open. Current main still lacks the requested configurable normal-chat delivery policy: heartbeat suppression is run-scoped, `messages.suppressToolErrors` remains a partial non-mutating warning switch, and mutating tool failures can still emit warnings after success-looking assistant text.

## What This Changes

Request a configurable normal-chat policy to suppress, defer, or disable transient tool-error warnings after successful agent recovery.

## Best Possible Solution

Define one documented normal-chat tool-error delivery policy that preserves unresolved mutating-failure safety while giving managed deployments a clear way to hide or reconcile recovered transient warnings, then align config types, schema/help, docs, tests, and changelog.

## Reproduction Assessment

Yes. Current main source and focused payload tests show `messages.suppressToolErrors` suppresses non-mutating warnings, while mutating `write`/`edit` failures can still emit warning payloads even after success-looking assistant text.

## Solution Assessment

Unclear. The requested `all`/`final`/`off` policy is plausible, but maintainers need to choose how it interacts with unresolved mutating failures before code can safely change.
