## Label Justifications

- `P1`: MiniMax Portal OAuth users can lose provider access after token expiry and must manually reauthenticate, breaking real model-provider workflows.
- `impact:auth-provider`: The issue is specifically about OAuth token refresh and provider credential resolution for `minimax-portal`.

## Summary

Keep open: current main and latest release v2026.6.6 still lack a MiniMax Portal `refreshOAuth` hook, so expired `minimax-portal` OAuth credentials can still fall through to a shared OAuth registry that does not include MiniMax. The direct refresh attempts in https://github.com/openclaw/openclaw/pull/77523 and https://github.com/openclaw/openclaw/pull/82056 are closed unmerged, while the remaining open MiniMax OAuth PR at https://github.com/openclaw/openclaw/pull/82587 covers a separate catalog-marker path.

## What This Changes

Implement MiniMax Portal OAuth auto-refresh so expired `minimax-portal` credentials use the stored refresh token instead of requiring manual reauthentication.

## Best Possible Solution

Keep this issue open until the MiniMax plugin owns a `refreshOAuth` implementation with legacy CN/global routing, focused regression coverage, and redacted live or equivalent refresh-path proof.

## Reproduction Assessment

Yes, at source level: expired `minimax-portal` OAuth credentials enter the OAuth refresh path, but current main has no MiniMax `refreshOAuth` hook and the shared OAuth registry does not include MiniMax. I did not perform a live expiry refresh because valid MiniMax OAuth credentials are required.

## Solution Assessment

Yes, the best fix direction is a provider-owned MiniMax `refreshOAuth` hook through the existing plugin seam, not a core fallback. Previous refresh PRs showed useful direction but did not land and still need legacy-region and refresh-path proof before merge.
