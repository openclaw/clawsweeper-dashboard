## Summary

Keep open: current main and the latest release still lack a MiniMax Portal `refreshOAuth` implementation, so expired `minimax-portal` OAuth credentials can still fall through to shared refresh code that does not include MiniMax. The direct refresh PRs were closed unmerged, and the remaining open MiniMax OAuth PR covers a separate catalog-marker failure.

## What This Changes

The issue requests MiniMax Portal OAuth auto-refresh so expired `minimax-portal` access tokens renew from the stored refresh token instead of requiring manual re-authentication.

## Best Possible Solution

Keep this issue open until the MiniMax plugin owns a `refreshOAuth` implementation with CN/global routing, focused regression coverage, and redacted live or equivalent refresh-path proof.

## Reproduction Assessment

Yes, at source level: expired `minimax-portal` OAuth credentials enter the shared OAuth refresh path, but current main has no MiniMax `refreshOAuth` hook and the shared OAuth registry does not include MiniMax. I did not run a live expiry refresh because valid MiniMax OAuth credentials are required.

## Solution Assessment

Yes, the best fix belongs in the provider-owned MiniMax `refreshOAuth` hook through the existing plugin seam, not in a core fallback. The implementation still needs owner or live validation for CN and global refresh behavior before it can safely land.
