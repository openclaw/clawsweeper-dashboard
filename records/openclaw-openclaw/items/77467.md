## Summary

Keep open: current main and the latest release still advertise MiniMax Portal OAuth login, but the MiniMax provider does not implement the provider-owned refresh hook, so expired `minimax-portal` OAuth credentials can still fall through to missing credentials. The report is source-reproducible and also has a second user confirmation, but fixing it should include provider-contract or live redacted MiniMax refresh proof.

## What This Changes

The issue requests MiniMax Portal OAuth access tokens to auto-refresh from stored refresh tokens instead of expiring into missing credentials.

## Best Possible Solution

Add a MiniMax-owned `refreshOAuth` implementation through the existing provider SDK hook, with CN/global unit coverage and one redacted live refresh proof before closing.

## Reproduction Assessment

Yes, from source inspection: expired MiniMax Portal OAuth credentials enter the generic refresh path, current MiniMax has no `refreshOAuth`, and built-in OAuth providers exclude `minimax-portal`. I did not run a live two-hour expiry cycle.

## Solution Assessment

Yes: a provider-owned `refreshOAuth` hook is the narrowest maintainable layer because the SDK already exposes that hook and core calls it before shared refreshers. The remaining gap is MiniMax endpoint/live-credential validation, not a new core API.
