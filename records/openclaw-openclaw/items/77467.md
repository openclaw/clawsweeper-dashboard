## Label Justifications

- `P1`: MiniMax Portal OAuth users can lose provider access after token expiry and must manually re-authenticate, which is a broken real agent/provider workflow.
- `impact:auth-provider`: The issue is specifically about OAuth token refresh and provider credential resolution for `minimax-portal`.

## Summary

Keep open: current main and v2026.6.6 still register MiniMax Portal OAuth login without a provider-owned refresh hook, so expired `minimax-portal` OAuth credentials can still fall through to the generic OAuth fallback and fail. Related PRs tried nearby fixes, but the direct refresh PRs are closed unmerged and the remaining open PR covers a separate catalog-marker path with unresolved review blockers.

## What This Changes

MiniMax Portal OAuth should auto-refresh stored OAuth credentials instead of requiring manual re-authentication after access-token expiry.

## Best Possible Solution

Implement a provider-owned `minimax-portal` `refreshOAuth` hook in the bundled MiniMax plugin, preserve CN/global endpoint selection for existing profiles, add focused regression coverage, and capture redacted live refresh or equivalent OAuthManager proof before landing.

## Reproduction Assessment

Yes, at source level: expired `minimax-portal` credentials call the provider refresh hook, but current main has no MiniMax `refreshOAuth` and the shared OAuth registry does not include MiniMax. I did not perform a live expiry refresh because valid MiniMax OAuth credentials are required.

## Solution Assessment

Yes, the best fix direction is a provider-owned MiniMax `refreshOAuth` implementation, not a core fallback. The prior PRs show the right seam but did not land and still left proof, region, or correctness concerns unresolved.
