## Summary

Keep open: current main and the latest release still lack a MiniMax Portal provider-owned `refreshOAuth` hook, so expired `minimax-portal` OAuth credentials still fall through to fallback refresh logic that does not include MiniMax. Direct refresh PR attempts are closed unmerged, and the remaining open MiniMax OAuth PR targets a separate catalog-marker bug rather than this expired-token refresh path.

## What This Changes

The issue asks OpenClaw to implement MiniMax Portal OAuth refresh so expired `minimax-portal` credentials can renew without manual re-authentication.

## Best Possible Solution

Implement provider-owned `minimax-portal` `refreshOAuth` through the existing plugin seam, with CN/global routing, regression coverage, and redacted real refresh-path proof before landing.

## Reproduction Assessment

Yes. Source inspection shows expired OAuth credentials call the provider `refreshOAuth` seam, while the current MiniMax Portal provider does not register that hook and the shared OAuth registry omits MiniMax; a live expiry repro was not run because valid MiniMax Portal OAuth credentials are required.

## Solution Assessment

Yes on direction, with provider-owner validation. The right seam is a provider-owned MiniMax refresh hook rather than a core fallback, but a safe implementation still needs CN/global routing, legacy-profile handling, and real refresh-path proof.
