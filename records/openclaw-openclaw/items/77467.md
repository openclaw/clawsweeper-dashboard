## Summary

Keep open: current main and v2026.5.7 still lack a MiniMax-owned OAuth refresh hook, so expired `minimax-portal` credentials can still fall through to the generic fallback and resolve as missing. The closed related PR at https://github.com/openclaw/openclaw/pull/77523 confirms the right direction but was not merged and left legacy-region and real-proof concerns unresolved.

## What This Changes

MiniMax Portal OAuth should refresh expired access tokens with stored refresh tokens instead of requiring manual re-authentication.

## Best Possible Solution

Add a MiniMax provider-owned `refreshOAuth` implementation that safely derives CN/global endpoint selection from credential or config state, persists rotated tokens, covers legacy profiles, and keeps MiniMax-specific refresh logic out of core auth.

## Reproduction Assessment

Yes, as a source-level reproduction. An expired `minimax-portal` OAuth credential reaches the plugin refresh seam, finds no MiniMax `refreshOAuth`, then cannot use the pi-ai fallback because MiniMax is not in that OAuth registry.

## Solution Assessment

Yes. The narrow maintainable fix is a MiniMax-local `refreshOAuth` hook through the existing provider contract; the closed PR showed the right direction but needs safer legacy-region handling and real refresh proof.
