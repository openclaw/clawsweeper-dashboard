## Label Justifications

- `P1`: Expired MiniMax Portal OAuth credentials can break real provider requests until users manually re-authenticate.
- `impact:auth-provider`: The issue is specifically about OAuth token refresh and credential resolution for the `minimax-portal` provider profile.

## Summary

Keep open: current main and the latest release still lack a MiniMax Portal provider-owned OAuth refresh hook, so expired `minimax-portal` credentials can still fall through to the generic OAuth fallback and fail. The direct refresh PRs are closed unmerged, while the remaining open MiniMax OAuth PR covers a separate catalog-marker path rather than this refresh hook.

## What This Changes

MiniMax Portal OAuth should refresh stored OAuth credentials automatically instead of requiring manual re-authentication after access token expiry.

## Best Possible Solution

Implement a provider-owned `minimax-portal` `refreshOAuth` hook in the bundled MiniMax plugin, preserve CN/global endpoint selection for existing profiles, add focused regression coverage, and capture redacted live refresh or equivalent OAuthManager proof before landing.

## Reproduction Assessment

Yes, at source level: expired `minimax-portal` credentials call the provider refresh seam, but current main has no MiniMax `refreshOAuth` and the shared OAuth registry does not include MiniMax. A live expiry refresh still needs valid MiniMax credentials.

## Solution Assessment

Yes: the best fix direction is a provider-owned MiniMax `refreshOAuth` implementation rather than a core fallback. The implementation should handle legacy region metadata and preserve provider refresh failure details.
