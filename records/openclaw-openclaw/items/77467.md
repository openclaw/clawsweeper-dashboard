## Summary

Keep open: current main and the latest release still lack a MiniMax-owned OAuth refresh hook, so the reported expired-token path can still fall through to the generic refresh fallback and return no credential. The closed related PR at https://github.com/openclaw/openclaw/pull/77523 confirms the right direction but was not merged and left concrete legacy-region and proof concerns unresolved.

## What This Changes

The issue reports that MiniMax Portal OAuth access tokens expire without automatic refresh because the MiniMax provider lacks a refreshOAuth implementation.

## Best Possible Solution

Add a provider-owned `refreshOAuth` implementation for `minimax-portal` that safely derives CN/global endpoint selection from credential/config state, persists rotated access and refresh tokens, covers legacy profiles, and keeps MiniMax-specific refresh logic out of core auth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an expired `minimax-portal` OAuth credential reaches plugin refresh, finds no MiniMax `refreshOAuth` hook, and then cannot use the pi-ai fallback because MiniMax is not a built-in OAuth provider.

## Solution Assessment

Yes. The narrow maintainable fix is a MiniMax provider-owned `refreshOAuth` hook through the existing plugin seam; adding MiniMax branches to core auth would violate the plugin ownership boundary.
