## Summary

Keep open: current `main` and `v2026.5.12` still lack a MiniMax-owned OAuth refresh hook, the only related implementation PR is closed unmerged, and the source path still lets expired `minimax-portal` credentials fall through to the generic OAuth fallback and resolve as missing.

## What This Changes

MiniMax Portal OAuth should refresh expired `minimax-portal` access tokens with the stored refresh token instead of returning `No credentials found`.

## Best Possible Solution

Add a MiniMax provider-owned `refreshOAuth` hook that uses the existing plugin contract, selects CN/Global refresh endpoints safely for both new and legacy credentials, persists rotated tokens, and keeps MiniMax-specific OAuth policy out of core auth.

## Reproduction Assessment

Yes, as a source-level reproduction. An expired `minimax-portal` OAuth credential reaches the core refresh path, finds no MiniMax `refreshOAuth` hook, cannot use the pi-ai fallback because MiniMax is not registered there, and then resolves as missing.

## Solution Assessment

Yes. The narrow maintainable solution is a MiniMax-local `refreshOAuth` implementation through the existing provider contract, with safe legacy region handling rather than a core auth fallback special case.
