## Summary

Keep open: current main still promises MiniMax OAuth auto-refresh, but the bundled MiniMax Portal provider only wires login and has no provider-owned refresh handler, so expired OAuth profiles can fall through to the generic refresh path without a MiniMax refresher.

## What This Changes

MiniMax Portal OAuth should refresh expired access tokens with the stored refresh token instead of requiring manual re-authentication.

## Best Possible Solution

Add a MiniMax-owned `refreshOAuth` implementation for `minimax-portal`, cover CN/global token endpoints, persist rotated credentials, and keep MiniMax-specific refresh logic out of core.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: an expired `minimax-portal` OAuth profile reaches plugin refresh, but the current provider has no `refreshOAuth` hook and the pi-ai fallback has no MiniMax provider. I did not perform a live wait-for-expiry run.

## Solution Assessment

Yes, the narrow maintainable fix is provider-owned refresh through the existing `refreshOAuth` seam. Adding MiniMax-specific branches to core auth would cross the plugin ownership boundary unnecessarily.
