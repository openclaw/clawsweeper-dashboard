## Label Justifications

- `P1`: MiniMax Portal OAuth users can lose provider access after token expiry and must manually re-authenticate, breaking real model-provider workflows.
- `impact:auth-provider`: The issue is specifically about OAuth token refresh and provider credential resolution for `minimax-portal`.

## Summary

Keep open: current main and the latest release still lack a MiniMax Portal `refreshOAuth` hook, so expired `minimax-portal` OAuth credentials can still fall through to the shared OAuth fallback that does not include MiniMax. Related direct refresh PRs are closed unmerged, and the remaining open MiniMax OAuth PR covers a separate catalog-marker path rather than this refresh hook.

## What This Changes

MiniMax Portal OAuth credentials should refresh automatically after access-token expiry instead of reporting missing credentials.

## Best Possible Solution

Implement provider-owned `minimax-portal` `refreshOAuth` in the bundled MiniMax plugin, preserve CN/global routing for existing and new profiles, add OAuthManager-path regression coverage, and require redacted live or equivalent refresh proof before landing.

## Reproduction Assessment

Yes, at source level: expired `minimax-portal` OAuth credentials call the provider refresh seam, but current main has no MiniMax `refreshOAuth` and the shared OAuth registry does not include MiniMax. I did not perform a live expiry refresh because valid MiniMax OAuth credentials are required.

## Solution Assessment

Yes, the best fix direction is provider-owned MiniMax `refreshOAuth` through the existing plugin seam, not a core fallback. The prior direct refresh PRs did not land, and the remaining open PR addresses a separate catalog-marker problem.
