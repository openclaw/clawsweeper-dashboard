## Summary

Keep open: current main still lacks a MiniMax Portal `refreshOAuth` hook, and the open PR #77523 explicitly targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for MiniMax Portal OAuth credentials to refresh automatically with the stored refresh token after access-token expiry.

## Best Possible Solution

Land PR #77523 or an equivalent narrow MiniMax-owned `refreshOAuth` implementation, keeping MiniMax-specific token refresh logic out of core auth.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: an expired `minimax-portal` OAuth profile reaches core OAuth refresh, but current main has no MiniMax provider `refreshOAuth` hook and no MiniMax fallback provider. I did not run a live two-hour token-expiry scenario.

## Solution Assessment

Yes, the best solution is provider-owned refresh through the existing `refreshOAuth` seam. Adding MiniMax-specific branches to core auth would cross the plugin ownership boundary unnecessarily.
