## Summary

Keep open: current main still advertises MiniMax OAuth auto-refresh, but the `minimax-portal` provider registers no `refreshOAuth` hook and the shared `pi-ai` fallback has no MiniMax OAuth provider.

## What This Changes

The issue asks OpenClaw to auto-refresh expired `minimax-portal` OAuth credentials instead of requiring manual re-authentication.

## Best Possible Solution

Add a MiniMax-owned `refreshOAuth` implementation through the existing provider plugin seam, with CN/global endpoint handling and regression coverage while keeping core auth generic.

## Reproduction Assessment

Yes, from source inspection: an expired `minimax-portal` OAuth profile reaches the provider refresh hook, but the provider has no hook and the shared `pi-ai` registry does not include MiniMax. I did not perform a live two-hour expiry run.

## Solution Assessment

Yes, the narrow maintainable fix is provider-owned refresh through the existing `refreshOAuth` seam. Adding MiniMax-specific refresh branches to core auth would cross the plugin ownership boundary unnecessarily.
