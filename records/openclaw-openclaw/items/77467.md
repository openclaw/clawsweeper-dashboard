## Summary

Keep open: current `main` still has MiniMax Portal OAuth login but no provider-owned refresh hook, so expired `minimax-portal` OAuth credentials still fall through the generic refresh path and can resolve as missing credentials.

## What This Changes

The issue reports that `minimax-portal` OAuth credentials expire and are not refreshed automatically because the MiniMax plugin lacks a provider-owned OAuth refresh hook.

## Best Possible Solution

Add a MiniMax-owned `refreshOAuth` implementation in `extensions/minimax`, wire it into the `minimax-portal` provider, and cover expired credential refresh for global/CN endpoint selection with focused tests plus live proof when credentials are available.

## Reproduction Assessment

Yes, source-reproducible: an expired `minimax-portal` OAuth credential enters the core refresh path, the MiniMax plugin has no `refreshOAuth` hook, and the built-in OAuth registry has no MiniMax provider. I did not run a live two-hour expiry test or a real refresh-token request in this read-only review.

## Solution Assessment

Yes: a provider-owned `refreshOAuth` hook is the narrowest maintainable fix because the SDK already defines that seam for custom refresh endpoints and core should not add MiniMax-specific refresh policy.
