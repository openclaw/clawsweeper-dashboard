## Summary

Keep open: current main still promises MiniMax OAuth auto-refresh, but `minimax-portal` only wires the login flow and has no provider-owned refresh handler, so expired OAuth profiles can fall through to a refresh path that has no MiniMax refresher.

## What This Changes

The issue asks for MiniMax Portal OAuth access tokens to refresh automatically from the stored refresh token instead of requiring manual re-authentication after expiry.

## Best Possible Solution

Implement MiniMax-owned `refreshOAuth` support for `minimax-portal`, preserving rotated access and refresh tokens while keeping MiniMax-specific refresh policy out of core auth.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an expired `minimax-portal` OAuth profile reaches the plugin refresh seam, but the provider has no `refreshOAuth` hook and the pi-ai fallback has no MiniMax provider.

## Solution Assessment

Yes. The narrow maintainable fix is a provider-owned `refreshOAuth` implementation through the existing plugin seam; adding MiniMax-specific refresh branches to core auth would cross the ownership boundary.
