## Summary

Keep open: the PR targets a real Feishu tool-gating gap, but it is not merge-ready because the Bitable/Base gate is only checked at aggregate registration time, so a disabled selected account can still execute Bitable tools when another account enables them. The supplied proof is also not enough for the changed runtime behavior because it does not show a patched install hiding or blocking disabled `feishu_bitable_*` tools.

## What This Changes

The PR adds Feishu `tools.bitable`/`tools.base` schema, types, defaults, docs, aggregate Bitable tool registration gating, account merge handling, and regression tests.

## Best Possible Solution

Enforce the Bitable/Base gate on the resolved selected account before each Bitable execution, keep aggregate registration only for model visibility when at least one account can use it, and add patched-runtime proof plus focused tests for disabled selected-account behavior.

## Reproduction Assessment

Yes for the review finding from source inspection: the PR gates registration with an OR across accounts, then execution resolves the selected account without checking that account's Bitable/Base gate. I did not run tests because this review is read-only.

## Solution Assessment

No; an aggregate registration gate is useful for tool visibility but is not the best complete fix once the PR documents per-account gates. The safer solution is selected-account enforcement plus explicit tests and runtime proof.
