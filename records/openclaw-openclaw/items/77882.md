## Summary

Keep open: this is a plausible Feishu config/security improvement, but the current PR head still gates Bitable registration with an any-enabled-account check, so a mixed-account setup can execute Bitable tools against a selected account whose merged Bitable/Base gate is disabled. The branch also changes public Feishu config semantics and is reported dirty against the base, so it needs maintainer review before merge.

## What This Changes

The PR adds Feishu `bitable`/`base` tool config keys, gates Bitable tool registration from merged account tools config, documents the new settings, and adds focused config/routing tests.

## Best Possible Solution

Land a refreshed branch that keeps Bitable tools hidden when globally disabled and also rejects or omits Bitable execution for any selected Feishu account whose merged Bitable/Base gate is disabled, with accepted alias semantics and proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure account `a` with `tools.bitable: false`, account `b` with `tools.bitable: true`, register Bitable tools, then execute `feishu_bitable_get_meta` with `agentAccountId` or explicit `accountId` set to `a`.

## Solution Assessment

No. Registration-time gating is necessary but not sufficient; the maintainable fix should also enforce the merged Bitable/Base gate on the resolved execution account and add regressions for disabled selected-account routing.
