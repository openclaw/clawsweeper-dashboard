## Summary

## Best Possible Solution

Land a version that preserves the documented canonical/alias config, blocks Bitable execution for the resolved disabled account, states the intended upgrade behavior for top-level hard gates, and includes redacted real disabled-path proof after a clean rebase.

## Reproduction Assessment

Yes, source inspection gives a high-confidence PR-head reproduction: configure account A with Bitable disabled and account B enabled, then execute a Bitable tool with `agentAccountId` or `accountId` set to A. The PR registers the family because B is enabled and resolves A without checking A's Bitable gate.

## Solution Assessment

No, registration-only gating is not the best final shape for the documented per-account gate. The maintainable fix is to enforce the resolved account's Bitable/Base gate at execution time or clearly change the docs/product contract to registration-only semantics.
