## Summary

Keep open: current main still lacks Feishu Bitable/Base tool gating, and this PR is the canonical implementation path, but the branch still has an alias-precedence bug and lacks required real behavior proof.

## What This Changes

The PR adds Feishu `tools.bitable`/`tools.base` config schema/type/default support, gates `feishu_bitable_*` registration, deep-merges per-account tools config, and adds Feishu regression tests.

## Best Possible Solution

Keep the Feishu plugin-local gate, fix alias precedence and public config docs, then merge only after redacted real tool-registration proof is posted.

## Reproduction Assessment

Yes, from source inspection: current main’s `registerFeishuBitableTools` registers after config/account checks without a Bitable/Base gate, while Drive and Wiki already use `resolveAnyEnabledFeishuToolsConfig`. I did not run a live Feishu gateway in this read-only review.

## Solution Assessment

No as written: the Feishu plugin is the right boundary, but the branch must preserve explicit `bitable` over the `base` alias in merged account config and add user-facing docs plus real behavior proof.
