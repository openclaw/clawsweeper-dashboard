## Summary

Keep this PR open: the Feishu Bitable/Base config gate is useful, but the patch still uses an aggregate registration check, so a disabled selected or explicit account can run Bitable tools when another account enables them. The branch is also reported dirty and the supplied proof does not show a patched runtime with Bitable/Base disabled.

## What This Changes

The PR adds Feishu `tools.bitable` and `tools.base` schema/type/docs support, merges account-level tools config, gates Bitable tool registration, and adds focused regression tests.

## Best Possible Solution

Land a rebased PR that adds the Bitable/Base gates, preserves Feishu upgrade behavior, enforces the effective gate for the resolved/default/explicit account, and includes focused tests plus redacted patched-runtime proof.

## Reproduction Assessment

Yes. The remaining defect is source-reproducible: configure account A with Bitable/Base disabled and account B enabled, then invoke a Bitable tool from context A or with `accountId: "a"`; the PR registers the tool because B is enabled and resolves A without a second gate.

## Solution Assessment

No. Aggregate registration is not the narrowest safe solution for a per-account tool gate; the resolved account needs an effective Bitable/Base check at visibility or execution time, with upgrade-sensitive tests and runtime proof.
