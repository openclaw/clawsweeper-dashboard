The branch adds Feishu `bitable`/`base` tool config keys, gates Bitable tool registration, changes Feishu account tool merging, documents the new settings, and adds regression tests.

## Best Possible Solution

Land a narrowed Feishu-owned fix that gates only Bitable/Base, preserves existing account `tools` override behavior unless maintainers explicitly approve and document a migration, and includes redacted patched-runtime proof of the disabled tool surface.

## Reproduction Assessment

Yes from source inspection: current main registers Bitable tools after only checking for enabled Feishu accounts, while sibling tool families already gate registration through `resolveAnyEnabledFeishuToolsConfig`. I did not run the repo tests because this is a read-only review.

No: the Bitable gate direction is right, but broad `tools` deep-merge is not the narrowest maintainable fix because it changes unrelated account-level tool semantics. A safer repair would normalize Bitable/Base per level and preserve existing override behavior for other tool families unless maintainers choose a documented migration.
