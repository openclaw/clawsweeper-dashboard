## Summary

Keep this PR open, but it is not merge-ready: the central Feishu Bitable gate is still global registration-only and does not enforce the disabled tool family on the selected account, and the supplied proof does not demonstrate a patched runtime hiding `feishu_bitable_*` under disabled config.

## What This Changes

This PR adds Feishu `tools.bitable`/`tools.base` schema, docs, defaults, account merge handling, Bitable registration gating, and focused regression tests.

## Best Possible Solution

Land a Feishu-owned gate that filters or rejects Bitable tools for the resolved execution account, preserves documented account-level alias semantics, and includes patched-runtime proof plus focused mixed-account regressions.

## Reproduction Assessment

Yes for the PR findings by source inspection: a mixed Feishu config with account A `tools.bitable: false` and account B `tools.bitable: true` still registers the Bitable factory globally and execution resolves the selected account through `createFeishuToolClient` without checking A's tool gate. I did not run tests because this review is read-only.

## Solution Assessment

No. The global no-any-account registration gate is useful, but the maintainable fix needs selected-account enforcement and explicit alias precedence coverage before this becomes a shipped config contract.

Full review comments:

- **[P1] Enforce Bitable/Base disables on the selected account:** `extensions/feishu/src/bitable.ts:589-592`
  - body: The new gate only checks whether any enabled account allows Bitable before registering `feishu_bitable_*`. In a mixed config where one account has `tools.bitable: false` and another has it enabled, an agent bound to the disabled account, or a call with `accountId` set to that account, still reaches `createFeishuToolClient`, so the account-level gate does not protect that account's Bitable data.
  - confidence: 0.9
- **[P2] Preserve account-level base aliases over inherited bitable:** `extensions/feishu/src/accounts.ts:195`
  - body: Deep-merging `tools` before alias resolution means an explicit top-level `tools.bitable: true` can make an account's legacy `tools.base: false` ineffective, because `resolveToolsConfig` sees both keys and always prefers `bitable`. That conflicts with the new per-account alias docs and the account-overrides-top-level merge contract.
  - confidence: 0.78
