## Summary

Keep open: current main still lacks a Feishu Bitable/Base tools gate, and this PR is the canonical remaining implementation, but the branch has an alias-precedence bug and still lacks required real behavior proof.

## What This Changes

The PR adds Feishu `tools.bitable`/`tools.base` config support, gates `feishu_bitable_*` registration with those settings, deep-merges account-level tools config, and adds Feishu regression tests.

## Best Possible Solution

Keep this PR open as the canonical Feishu Bitable tools-config path, fix the alias precedence in the account merge, document the new config keys, and require real behavior proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main's `registerFeishuBitableTools` reaches registration after config/account checks with no Bitable/Base gate, while Drive/Wiki/Perm already use `resolveAnyEnabledFeishuToolsConfig`. I did not run a live Feishu gateway reproduction in this read-only review.

## Solution Assessment

No for this branch as written: the Feishu plugin is the right boundary, but the account merge must honor explicit `bitable` over the `base` alias and the PR still needs user-facing config docs plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve explicit bitable over the base alias:** `extensions/feishu/src/accounts.ts:194`
  - body: The branch makes `resolveToolsConfig({ bitable: true, base: false })` resolve Bitable enabled, but `mergeFeishuAccountConfig` forces both values false whenever top-level `base` is false.   - confidence: 0.88

  - confidence: 0.76
