## Summary

## What This Changes

The PR adds Feishu `tools.bitable`/`tools.base` schema, type, docs, and tests, then gates `feishu_bitable_*` registration through the resolved Feishu tools config.

## Best Possible Solution

Enforce the Bitable/Base gate after resolving the target account, keep the config/docs/tests aligned with the approved alias semantics, rebase the branch, and add redacted patched-runtime proof showing disabled config hides or rejects Bitable tools.

## Reproduction Assessment

Yes, source-reproducible: with one Feishu account disabling Bitable and another enabling it, the PR registers the global Bitable tools and `createFeishuToolClient` resolves the selected disabled account without a tool-gate check.

## Solution Assessment

No. A registration-only aggregate gate is not sufficient; the maintainable fix must enforce the resolved account's Bitable/Base gate at execution and prove the disabled runtime behavior.
