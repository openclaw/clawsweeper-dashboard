- **[P1] Reject disabled Bitable accounts before execution:** `extensions/feishu/src/bitable.ts:583-591`
  - body: The new guard only checks whether any enabled account allows Bitable. In a mixed setup, the tool is registered because account b allows it, then `accountId: "a"` or agent context `a` still reaches `createFeishuToolClient` and runs Bitable against account a even if that account has `tools.bitable: false`; add a selected-account guard and negative tests for both selection paths.
  - confidence: 0.88
- **[P1] Preserve existing account tool override semantics:** `extensions/feishu/src/accounts.ts:183-189`
  - body: Adding `nestedObjectKeys: ["tools"]` changes every Feishu tool family, not just Bitable: existing account-level `tools` objects now inherit top-level disabled entries instead of replacing the top-level object. That can silently disable doc/wiki/drive/etc for upgraded multi-account configs, so either constrain this to the Bitable hard gate or add maintainer-approved upgrade coverage for the broader semantic change.
  - confidence: 0.78
