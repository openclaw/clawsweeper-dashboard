## Summary

Keep this PR open: the Feishu Bitable gating fix is useful, but the branch is not merge-ready because it documents account-level Bitable/Base gates while the runtime only checks whether any enabled account allows Bitable, and the supplied proof does not demonstrate the disabled-config behavior in a real OpenClaw run.

## What This Changes

The PR adds Feishu `tools.bitable`/`tools.base` config schema, types, docs, defaults, account tool merging, Bitable registration gating, and focused regression tests.

## Best Possible Solution

Keep Bitable enabled by default for compatibility, but make top-level disables global, make account-level disables apply to the selected account/context before tool exposure or execution, and document/prove that exact behavior.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the current-main gap: current Bitable registration only checks that Feishu has enabled accounts, while the existing Feishu tools config has no Bitable/Base gate. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. The top-level registration gate is the right direction, but the PR should either enforce the selected account's Bitable/Base gate or stop documenting account-level config as a per-account gate.

## Review Findings

Full review comments:

- **[P1] Enforce the selected account's Bitable gate:** `extensions/feishu/src/bitable.ts:583-584`
  - body: The new config/docs expose `channels.feishu.accounts.<id>.tools.bitable` as a per-account gate, but this check only tests whether any enabled account allows Bitable. In a mixed setup where account A has `bitable: false` and account B has `bitable: true`, the tool factory still returns `feishu_bitable_*` for an agent bound to A, and `getClient` will execute against A without rechecking A's merged tool config. Either suppress/deny Bitable for the selected account or narrow the docs/tests so account-level config is not presented as a gate.
  - confidence: 0.86
