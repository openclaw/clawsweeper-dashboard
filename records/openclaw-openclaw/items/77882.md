The PR adds Feishu Bitable/Base tool config keys, registration gating, docs, and regression tests for top-level and account-level tool config combinations.

## Best Possible Solution

Land a rebased version that globally hides Bitable tools when disabled, enforces the resolved account's effective Bitable/Base gate at execution time, and aligns docs, tests, and patched-runtime proof with that contract.

## Reproduction Assessment

Yes, from source inspection: with one account disabled for Bitable and another enabled, the aggregate registration gate keeps `feishu_bitable_*` registered and execution still resolves an explicit or contextual disabled account without checking that account's effective gate.

## Solution Assessment

No; the registration gate is the right direction, but the best fix also enforces the selected account's gate and resolves `base` alias precedence before merged inherited `bitable` values look explicit.
