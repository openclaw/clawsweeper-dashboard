## Label Justifications

- `P1`: The PR targets a user-facing session-history regression that can halt or corrupt agent tool workflows, but the current patch is not merge-ready.
- `merge-risk: 🚨 compatibility`: The diff can change appendMessage from a synchronous entry-id return into an asynchronous Promise for existing guarded session callers.

## Summary

This PR should stay open but is not merge-ready: the fix targets a real session-history symptom, but the branch changes the synchronous append contract, can create pending state for unwritten tool calls, is conflicting with current main, and still lacks real behavior proof.

## What This Changes

The PR adds a promise-chain mutex around session tool-result guard appends, moves pending tool-call tracking before append, and adds three concurrency regression tests.

## Best Possible Solution

Rebase and revise the guard so append side effects stay synchronous when the underlying SessionManager append is synchronous, pending tracking is transactional with accepted writes, and the race regression test uses a real yielding append path plus redacted real-session proof.

## Reproduction Assessment

Yes for the PR-introduced regressions by source inspection: the diff defers guarded append side effects and records pending IDs before write acceptance. I did not establish a live current-main reproduction of the original user race in this read-only review.

## Solution Assessment

No. The direction addresses a plausible race, but this implementation should preserve the append contract, keep pending state tied to accepted writes, and prove the real session behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:309-312`
  - body: `appendMutex.run(...)` always enters through `queue.then`, so even a synchronous `originalAppend` no longer mutates the session before `appendMessage` returns. The cast hides that runtime change from TypeScript, and callers that append then immediately read, flush, label, or inspect session state can now observe stale data or a Promise instead of the entry id.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:282-286`
  - body: This registers tool-call IDs before `beforeMessageWriteHook` and before `originalAppend`. If the hook blocks the assistant message or append throws, the guard can retain pending IDs for a tool call that never reached the transcript and later emit a synthetic result for an unwritten call.
  - confidence: 0.91
- **[P2] Make the race tests exercise async append interleaving:** `src/agents/session-tool-result-guard.test.ts:537-558`
  - body: These tests use `SessionManager.inMemory().appendMessage`, which is synchronous, and the first test queues the real `toolResult` microtask before the completion message. That does not create the awaited append window described in the PR, so the tests do not prove the proposed race fix.
  - confidence: 0.86
