## Summary

The PR should stay open: it targets a real session transcript race, but the branch still lacks contributor-side real behavior proof and introduces blocking session-state regressions around the append contract and pending tool-call tracking.

## What This Changes

The PR wraps `installSessionToolResultGuard` append handling in a promise-chain mutex, moves pending tool-call registration before the append path, and adds concurrent append regression tests.

## Best Possible Solution

Revise the fix so append side effects remain synchronous when the underlying append is synchronous, pending IDs are transactional with accepted writes, and the contributor supplies redacted real behavior proof for the affected session path.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the diff defers every guarded append through a promise chain and records pending IDs before write acceptance. I did not run a live current-main reproduction of the original user race in this read-only review.

## Solution Assessment

No. The direction addresses a real race, but the proposed mutex and pending-order change should be reworked to preserve existing append semantics and avoid stale pending state before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:309-312`
  - body: `appendMutex.run(...)` always enters through `queue.then`, so even a synchronous `originalAppend` no longer mutates the session before `appendMessage` returns. The cast hides that runtime change from TypeScript, and callers that append then immediately read, flush, or inspect pending state can now observe stale session state.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:282-286`
  - body: This registers tool-call IDs before `beforeMessageWriteHook` and `originalAppend`. If the hook blocks the assistant message or append throws, the guard can retain pending IDs for a tool call that never reached the transcript and later emit a synthetic result for an unwritten call.
  - confidence: 0.91
- **[P2] Make the race tests exercise async append interleaving:** `src/agents/session-tool-result-guard.test.ts:537-558`
  - body: These tests use `SessionManager.inMemory().appendMessage`, which is synchronous, and the first test queues the real `toolResult` before the completion message. That does not create the async write window described in the PR, so the tests do not prove the proposed race fix; use a raw append stub or hook that actually yields in the vulnerable interval.
  - confidence: 0.86
