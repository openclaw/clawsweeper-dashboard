## Summary

Keep this PR open, but do not merge it unchanged. The linked synthetic-result failures are credible and current main already has an idle-before-flush mitigation for the main observed cleanup race, while this branch introduces two blocking regressions: it silently makes a synchronous append contract asynchronous and records pending tool calls before the write path accepts the assistant message.

## What This Changes

The PR adds a promise-chain mutex around `installSessionToolResultGuard`, moves pending tool-call tracking before append persistence, and adds three concurrent append regression tests.

## Best Possible Solution

Keep the current idle-before-flush mitigation unless maintainers confirm a real asynchronous raw-append path. If append serialization is still needed, rework it so guarded appends preserve the synchronous `SessionManager.appendMessage` contract, record pending tool calls only after accepted persistence, and add a regression test that exercises the real async boundary rather than only microtask-scheduled synchronous appends.

## Reproduction Assessment

Unclear. The historical synthetic-result reports have credible coverage through the current wait-for-idle regression test, but this PR's exact concurrent `appendMessage` race assumes an async append boundary that is not present in the pinned `@mariozechner/pi-coding-agent` 0.71.1 `SessionManager.appendMessage` implementation.

## Solution Assessment

No. The proposed mutex is not the best fix as submitted because it changes the runtime return contract and can synthesize results for assistant tool calls that were blocked or failed before persistence; a safer fix must preserve append semantics and write-acceptance ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve synchronous appendMessage behavior:** `src/agents/session-tool-result-guard.ts:310-312`
  - body: `appendMutex.run(...)` always returns a Promise and defers the append body to a microtask, but the cast keeps TypeScript believing `appendMessage` still returns the synchronous entry id. Current `SessionManager.appendMessage` is declared and implemented as returning `string`, so callers can read stale transcript state or receive a Promise where an id is expected.
  - confidence: 0.94
- **[P1] Track pending calls only after write acceptance:** `src/agents/session-tool-result-guard.ts:282`
  - body: Recording `pendingState.trackToolCalls(toolCalls)` before `applyBeforeWriteHook(...)` and `originalAppend(...)` leaves pending ids behind when an assistant message is blocked or append fails. The next non-tool message can then synthesize tool results for tool calls that were never persisted, corrupting the transcript pairing the guard is meant to protect.
  - confidence: 0.92
