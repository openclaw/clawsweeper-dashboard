## Summary

Keep open: this targets a plausible session transcript bug, but the current PR is not mergeable as-is because it changes `appendMessage` timing, can leave pending IDs for blocked writes, lacks contributor real behavior proof, and is currently conflicting.

## What This Changes

The PR wraps `session-tool-result-guard` append handling in a promise-chain mutex, moves pending tool-call tracking before persistence, and adds concurrent append regression tests.

## Best Possible Solution

Preserve the synchronous `SessionManager.appendMessage` contract, serialize only the vulnerable guard state transition if needed, record pending IDs only after accepted writes or roll them back, and require real-session proof before merge.

## Reproduction Assessment

Yes for the PR regressions by source inspection: `appendMutex.run` defers append work through `Promise.then`, and pending tracking runs before the blockable write hook and persistence. I did not establish a live reproduction of the original concurrent append race on current main.

## Solution Assessment

No. The current PR solves a plausible race by changing a synchronous dependency contract and by recording pending IDs before write acceptance; a safer fix should preserve append timing and use accepted-write tracking or rollback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:310-312`
  - body: `appendMutex.run(...)` always schedules the guard body through a Promise continuation while the method is cast back to the original return type. Callers that append and immediately read, flush, or use the returned entry id can now observe stale session state even when the underlying `SessionManager.appendMessage` is synchronous.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:282-285`
  - body: This records assistant tool-call IDs before the blockable write hook and before persistence. If the hook blocks the assistant message or the append throws, the pending map can later flush synthetic results for tool calls that were never written to the transcript.
  - confidence: 0.9
