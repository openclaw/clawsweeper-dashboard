## Summary

Keep open: this is a plausible bug-fix PR, but the current patch introduces blocking regressions in the session append contract and still lacks real after-fix behavior proof.

## What This Changes

The branch adds a promise-chain mutex around session-tool-result-guard appends, moves pending tool-call tracking before persistence, and adds concurrent append regression tests.

## Best Possible Solution

Revise the fix to preserve SessionManager.appendMessage synchronous side effects and return value, track or roll back pending IDs only after accepted writes, and add redacted real-session proof for the race.

## Reproduction Assessment

Yes for the PR regressions by source inspection: appendMutex.run defers append side effects to a Promise microtask, and pre-write pending tracking can survive a blocked write. I did not establish a live reproduction of the original reported race.

## Solution Assessment

No. The current PR solves the suspected race by widening a synchronous dependency contract and moving pending tracking ahead of write acceptance; a safer fix should preserve the append contract and use accepted-write tracking or rollback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:310`
  - body: appendMutex.run(...) always starts the guard body through queue.then(...), so even an in-memory synchronous append now runs in a later microtask while the method is cast back to the old return type. Existing callers can append and immediately flush or read the session; with this change those operations can observe stale session state before the append has run.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:285`
  - body: This records assistant tool-call IDs before applyBeforeWriteHook(...) and originalAppend(...). If a before_message_write hook blocks the assistant message or the append fails, the pending map can contain IDs for tool calls that were never persisted, and the next flush can write synthetic tool results without a matching assistant tool-call message.
  - confidence: 0.9
