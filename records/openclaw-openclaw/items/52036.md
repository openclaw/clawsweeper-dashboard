## Summary

Keep this PR open for maintainer decision, but do not merge it as submitted. Current main already has the idle-before-flush mitigation for the historical synthetic-result symptom, while this PR introduces blocking append semantics regressions and does not establish the exact async raw-append race against the current dependency contract.

## What This Changes

The PR adds a promise-chain mutex around the session tool-result guard append path, moves pending tool-call tracking before append persistence, and adds concurrent append regression tests.

## Best Possible Solution

Keep the current idle-before-flush mitigation and only pursue append serialization if maintainers confirm a real async append boundary; any replacement should preserve synchronous `appendMessage` behavior and track pending tool calls only after write acceptance.

## Reproduction Assessment

Unclear. The historical synthetic-result symptom has credible coverage through the current idle-before-flush test, but this PR's exact concurrent append race assumes an async raw append boundary that is not present in the current pinned `SessionManager.appendMessage` contract.

## Solution Assessment

No. The submitted mutex is not the best fix because it changes the runtime return contract and records pending tool calls before the write path accepts the assistant message; a safer fix would preserve current append semantics and prove the actual async boundary first.
