## Summary

Keep open: the PR targets a plausible session-history race, but this branch introduces blocking regressions by making guarded appends microtask-delayed/asynchronous and by recording pending tool calls before write hooks can reject the assistant message; it also lacks after-fix real behavior proof.

## What This Changes

The PR wraps `installSessionToolResultGuard` append handling in a promise-chain mutex, moves pending tool-call tracking before persistence, and adds concurrent append regression tests.

## Best Possible Solution

Revise the fix so guarded appends preserve synchronous side effects and return values, track pending tool calls only after a message is accepted or roll back on failure, and include redacted real-session proof for the race.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows `appendMutex.run` defers all append side effects to a Promise microtask while upstream `appendMessage` is synchronous, and pre-write pending tracking can survive blocked writes. I did not establish a live reproduction of the original reported race.

## Solution Assessment

No. The fix should serialize the real race without widening `appendMessage` behind a cast or creating pending IDs for messages that hooks reject.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:310`
  - body: `appendMutex.run(...)` always starts the guard body through `queue.then(...)`, so even an in-memory synchronous append now runs in a later microtask while the method is cast back to the old return type. Existing guard callers can append and immediately flush/read the session; with this change `sm.appendMessage(toolCall); guard.flushPendingToolResults()` runs before the assistant message is recorded or pending is tracked, so the explicit flush becomes a no-op and session state is stale.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:285`
  - body: This records assistant tool-call IDs before `applyBeforeWriteHook(...)` can drop the message. If a `before_message_write` hook blocks the assistant message, the pending map still contains IDs for tool calls that were never persisted, and the next flush can write synthetic tool results without a matching assistant tool-call message.
  - confidence: 0.9
