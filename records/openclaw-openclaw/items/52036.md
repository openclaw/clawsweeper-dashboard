## Summary

Keep this PR open for maintainer handling, but do not merge it as-is. Current main preserves synchronous append semantics and already has the idle-before-flush mitigation for the historical synthetic-result symptom, while this PR introduces blocking regressions in the guarded append contract and pending-state ordering.

## What This Changes

This PR adds a promise-chain mutex around session tool-result guard appends, moves pending tool-call tracking before persistence, and adds concurrent append regression tests.

## Best Possible Solution

Keep the current idle-before-flush mitigation, and pursue append serialization only if maintainers confirm a real async append implementation with preserved return semantics and pending-state invariants.

## Reproduction Assessment

Unclear. The historical synthetic-result symptom is covered by current idle-before-flush tests, but this PR's exact race requires an async raw append boundary and the pinned `SessionManager.appendMessage` returns synchronously through `appendFileSync`.

## Solution Assessment

No. The submitted mutex changes the `appendMessage` runtime return contract and records pending ids before write acceptance; a safer path would preserve synchronous append behavior and prove the remaining async boundary first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve synchronous appendMessage behavior:** `src/agents/session-tool-result-guard.ts:310-312`
  - body: `appendMutex.run(...)` always returns a Promise and defers the append body to a microtask, but the cast keeps TypeScript believing `appendMessage` still returns the synchronous entry id. Current callers can read stale transcript state or receive a Promise where an id is expected.
  - confidence: 0.94
- **[P1] Track pending calls only after write acceptance:** `src/agents/session-tool-result-guard.ts:282-286`
  - body: Recording pending tool-call ids before the before-write hook and before `originalAppend` means a blocked or failed assistant write can leave ids pending for tool calls that were never persisted, allowing the next non-tool message to synthesize invalid results.
  - confidence: 0.92
