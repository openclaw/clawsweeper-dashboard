## Summary

Keep open, but do not merge as-is: the PR addresses a plausible synthetic tool-result race, yet the current diff introduces synchronous append contract regressions, tracks pending tool calls before write acceptance, and still lacks real behavior proof.

## What This Changes

The PR adds a promise-chain mutex around session-tool-result-guard append handling, moves pending tool-call tracking before persistence, and adds concurrent append regression tests.

## Best Possible Solution

Preserve the synchronous SessionManager.appendMessage contract, serialize only the vulnerable guard state transition if needed, record pending IDs only after accepted writes or roll them back, and require real-session proof before merge.

## Reproduction Assessment

Yes for the PR regressions by source inspection: appendMutex.run defers append work through Promise.then, and pending tracking runs before the blockable write hook and persistence. I did not establish a live reproduction of the original concurrent append race on current main.

## Solution Assessment

No. The current PR changes a synchronous dependency contract and records pending IDs before write acceptance; the safer fix should preserve append timing and use accepted-write tracking or rollback.
