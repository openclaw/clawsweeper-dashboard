## Summary

Keep open, but this PR is not merge-ready: the reported session transcript race is plausible and still related to open user reports, while the branch changes the synchronous append contract, records pending tool-call IDs before accepted writes, has a stale conflicting head, and lacks after-fix real behavior proof.

## What This Changes

The branch adds promise-chain serialization around guarded session appends, moves pending tool-call tracking before persistence, and adds concurrent append regression tests.

## Best Possible Solution

Revise the fix to preserve synchronous appendMessage timing, serialize or roll back only the vulnerable guard state transition, keep pending IDs tied to accepted writes, rebase onto current main, and add redacted real-session or terminal proof before merge.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the mutex defers guardedAppend through Promise.then and pending tracking runs before the blockable write hook. I did not establish a live current-main reproduction of the original concurrent append race during this read-only review.

## Solution Assessment

No. The proposed whole-append async mutex changes a synchronous dependency contract and moves pending state before write acceptance; a safer fix should preserve append timing and make pending tracking transactional with persisted messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:309-312`
  - body: `appendMutex.run(...)` always schedules the guard body through a Promise continuation, then casts that Promise back to `ReturnType<typeof originalAppend>`. Callers can append and immediately read, flush, or use the entry id while the message has not been appended yet, which breaks the synchronous SessionManager contract.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:282-285`
  - body: This records assistant tool-call IDs before `beforeMessageWriteHook` and before `originalAppend`. If the hook blocks the assistant message or the append throws, the pending map can later flush synthetic results for tool calls that were never written to the transcript.
  - confidence: 0.91
- **[P2] Make the race tests fail without the fix:** `src/agents/session-tool-result-guard.test.ts:537-558`
  - body: These tests use `SessionManager.inMemory().appendMessage`, which is synchronous, and the first test queues the real `toolResult` before the completion message. That does not create the async write window described in the PR, so the tests do not prove the proposed race fix; use a raw append stub that yields or otherwise force the failing interleaving.
  - confidence: 0.86
