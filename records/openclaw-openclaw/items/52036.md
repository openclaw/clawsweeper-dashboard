## Summary

Keep open: the PR targets a real unresolved session-transcript symptom cluster, but the current branch is conflicting, lacks real behavior proof, and introduces blocking session contract/state-ordering regressions.

## What This Changes

The PR adds a promise-chain mutex around guarded session appends, moves pending tool-call registration before persistence, and adds three concurrency tests for synthetic tool-result races.

## Best Possible Solution

Rebase and revise the guard so append side effects remain synchronous when the underlying append is synchronous, pending tracking is transactional with accepted writes, and the race regression test uses a real yielding append path plus redacted real-session proof.

## Reproduction Assessment

Yes for the PR-introduced regressions by source inspection: the diff defers guarded append side effects and records pending IDs before write acceptance. I did not establish a live current-main reproduction of the original user race in this read-only review.

## Solution Assessment

No. The direction addresses a plausible race, but this implementation should preserve the append contract, keep pending state tied to accepted writes, and prove the real session behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:309-312`
  - body: `appendMutex.run(...)` always enters through `queue.then`, so even a synchronous `originalAppend` no longer mutates the session before `appendMessage` returns. The cast hides that runtime change from TypeScript, and callers that append then immediately read, flush, or inspect session state can now observe stale data.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:282-286`
  - body: This registers tool-call IDs before `beforeMessageWriteHook` and before `originalAppend`. If the hook blocks the assistant message or append throws, the guard can retain pending IDs for a tool call that never reached the transcript and later emit a synthetic result for an unwritten call.
  - confidence: 0.91
- **[P2] Make the race tests exercise async append interleaving:** `src/agents/session-tool-result-guard.test.ts:537-558`
  - body: These tests use `SessionManager.inMemory().appendMessage`, which is synchronous, and the first test queues the real `toolResult` before the completion message. That does not create the async write window described in the PR, so the tests do not prove the proposed race fix.
  - confidence: 0.86
