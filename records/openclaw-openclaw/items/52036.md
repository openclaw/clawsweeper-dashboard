## Summary

Keep open: this is a plausible session-state bug-fix PR, but the current branch is conflicting, lacks real behavior proof, and introduces blocking append contract and pending-state regressions.

## What This Changes

The branch wraps the session tool-result guard append path in a promise-chain mutex, moves pending tool-call tracking before persistence, and adds three concurrency tests.

## Best Possible Solution

Keep the fix in the session guard and embedded write boundary, preserving synchronous append semantics while tying pending IDs to accepted writes and proving the real-session symptom after rebase.

## Reproduction Assessment

Yes for the PR-introduced regressions by source inspection: appendMutex.run defers guarded append side effects, and pending IDs are recorded before write acceptance. I did not run a live current-main reproduction of the original user race in this read-only review.

## Solution Assessment

No. The proposed mutex addresses a plausible race, but this implementation changes the append contract and makes pending state non-transactional; a safer fix must preserve sync semantics or explicitly migrate the API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:310-312`
  - body: `appendMutex.run(...)` always enters through `queue.then`, so even a synchronous `originalAppend` no longer mutates the session before `appendMessage` returns. The cast hides that runtime change from TypeScript, and callers that append then immediately read, flush, or inspect session state can now observe stale data.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:282-286`
  - body: This registers tool-call IDs before `beforeMessageWriteHook` and before `originalAppend`. If the hook blocks the assistant message or append throws, the guard can retain pending IDs for a tool call that never reached the transcript and later emit a synthetic result for an unwritten call.
  - confidence: 0.91
- **[P2] Make the race tests exercise async append interleaving:** `src/agents/session-tool-result-guard.test.ts:537-558`
  - body: These tests use `SessionManager.inMemory().appendMessage`, which is synchronous, and the first test queues the real `toolResult` before the completion message. That does not create the async write window described in the PR, so the tests do not prove the proposed race fix.
  - confidence: 0.86
