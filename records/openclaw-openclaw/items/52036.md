## Summary

Keep this PR open, but do not merge it unchanged. The historical synthetic-tool-result failures are credible and current main already has an idle-before-flush mitigation for one root cause, while this branch introduces two blocking transcript-integrity regressions around the synchronous append contract and pending-state ordering.

## What This Changes

The PR wraps `installSessionToolResultGuard` appends in a promise-chain mutex, moves pending tool-call tracking before persistence, and adds three concurrent append regression tests.

## Best Possible Solution

Do not merge this branch unchanged. The maintainable end state is either to close it after maintainer confirmation that current idle-flush handling covers the linked failures, or request a contract-preserving rework that keeps `appendMessage` synchronous, records pending tool calls only after accepted persistence, and tests a real supported async boundary.

## Reproduction Assessment

Unclear. The historical synthetic-result failures are credible and current main has a focused idle-before-flush regression test, but this PR's claimed concurrent `appendMessage` interleaving is not reproduced with high confidence because the pinned upstream append implementation is synchronous.

## Solution Assessment

No. The proposed mutex is not the best fix as submitted because it silently changes the append contract and can synthesize results for assistant tool calls that were blocked or failed to persist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve synchronous appendMessage behavior:** `src/agents/session-tool-result-guard.ts:310-312`
  - body: `appendMutex.run(...)` always returns a Promise and defers the append body to a microtask, but the cast leaves callers typed as receiving the synchronous entry id. The pinned `SessionManager.appendMessage` contract is a synchronous string-returning write, so immediate transcript reads, event broadcasts, or flushes can observe stale state.
  - confidence: 0.93
- **[P1] Track pending calls only after write acceptance:** `src/agents/session-tool-result-guard.ts:282`
  - body: Moving `pendingState.trackToolCalls(toolCalls)` before `applyBeforeWriteHook(...)` and `originalAppend(...)` leaves pending ids behind when an assistant write is blocked or fails. The next non-tool-result append can then synthesize results for tool calls that were never persisted.
  - confidence: 0.91
