## Summary

Keep open: the bug being targeted is plausible and still overlaps active missing-tool-result reports, but this PR introduces blocking append-contract regressions and has not supplied real behavior proof.

## What This Changes

The PR serializes `session-tool-result-guard` appends with a promise-chain mutex, moves pending tool-call registration before persistence, and adds concurrent append regression tests.

## Best Possible Solution

Preserve the synchronous `SessionManager.appendMessage` contract, serialize only the vulnerable guard state transition if needed, avoid pending IDs for blocked or failed writes, and prove the fix with redacted real-session output.

## Reproduction Assessment

Yes for the PR regressions by source inspection: `appendMutex.run` defers append side effects to a Promise microtask, and pre-write pending tracking can survive a blocked or failed write. I did not establish a live reproduction of the original reported race.

## Solution Assessment

No. The current PR changes a synchronous dependency contract and records pending tool calls before write acceptance; a safer fix should preserve append timing and use accepted-write tracking or rollback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve synchronous append side effects:** `src/agents/session-tool-result-guard.ts:310`
  - body: `appendMutex.run(...)` always starts the guard body through a Promise continuation, so even an in-memory synchronous append runs later while the method is cast back to the old return type. Callers that append and then immediately read, flush, or rely on the returned message id can now observe stale session state.
  - confidence: 0.94
- **[P1] Track pending IDs only after accepted writes:** `src/agents/session-tool-result-guard.ts:285`
  - body: This records assistant tool-call IDs before `applyBeforeWriteHook(...)` and `originalAppend(...)`. If a write hook blocks the assistant message or the append throws, the pending map can contain IDs for tool calls that were never persisted, letting the next flush create synthetic results without a matching assistant tool-call message.
  - confidence: 0.9
