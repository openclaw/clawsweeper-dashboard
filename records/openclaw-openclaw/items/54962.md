## Summary

Keep open, but not merge-ready: the bug is real against current `main`, yet the branch is stale relative to native Telegram callback routing and still lacks inspectable real Telegram proof.

## What This Changes

The branch changes the Telegram generic callback fallback to send `callback_data: <payload>` text to the agent instead of the raw callback payload.

## Best Possible Solution

Rebase onto current `main`, prefix only generic fallback callbacks after preserving native callback parsing, add focused generic/native callback assertions, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes for source-level reproduction: docs promise `callback_data: <value>`, while current `main` still sends raw `data` for non-native callback fallback messages. I did not run a live Telegram/webhook reproduction in this read-only review.

## Solution Assessment

No as currently proposed: the fix direction is right for generic callbacks, but the stale branch must be rebased so current native callback parsing remains intact. The mergeable shape is `nativeCallbackCommand ?? callback_data-prefixed generic text`, with tests and real Telegram proof.
