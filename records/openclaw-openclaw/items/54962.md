## Summary

Keep open: the Telegram callback bug is valid, but this PR is not mergeable as-is because it prefixes every fallback callback and would break existing slash-command callback routing such as `/models add <provider>`. Current main still needs a narrower Telegram fix.

## What This Changes

The PR changes the Telegram callback-query fallback to send `callback_data: ${data}` to the agent instead of the raw callback payload.

## Best Possible Solution

Update the Telegram fallback so only generic agent callbacks are prefixed, while native and slash-command callbacks keep their current routing, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes, at source level: current main forwards generic callback data raw while the Telegram docs require `callback_data: <value>`. I did not run a live Telegram webhook scenario in this read-only review.

## Solution Assessment

No. The intent matches the documented generic callback contract, but prefixing every fallback payload is too broad because slash-command callbacks must still start with `/`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve slash-command callbacks before prefixing:** `extensions/telegram/src/bot-handlers.runtime.ts:1570`

Keep native and slash-command callbacks raw, and prefix only generic user callback data.
