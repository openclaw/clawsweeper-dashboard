## Summary

Keep this PR open, but it still needs changes before merge. The linked Telegram callback bug is valid on current main, yet the submitted one-line prefix would now collide with command-style callback routing that current main added around native Telegram command callbacks.

## What This Changes

The PR changes the Telegram callback-query fallback to forward inline button callback data to the agent as `callback_data: ${data}` instead of the raw callback value.

## Best Possible Solution

Update the Telegram callback handler on current main so specialized handlers and command/native callback paths still run first, then send `callback_data: ${data}` only for generic user callback payloads, with a focused regression test and a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main sends non-native callback data to `processMessage` as raw `data`, while the linked report gives concrete Telegram inline-button steps that expect `callback_data: accept_BK-TEST-006`.

## Solution Assessment

No. The proposed one-line prefix is aimed at the right bug, but the maintainable fix must preserve native/command callback routing and prefix only the generic fallback payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Preserve command callback routing before prefixing:** `extensions/telegram/src/bot-handlers.runtime.ts:1567`

Keep command/native callback handling raw or decoded, and apply `callback_data:` only to generic user callbacks.
