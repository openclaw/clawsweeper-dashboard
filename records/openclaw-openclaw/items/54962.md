## Summary

Keep open: current main and the latest release still have the docs-backed Telegram callback text bug, but this PR is not merge-ready because it lacks inspectable real behavior proof and the stale one-line patch would break slash/native callback routing.

## What This Changes

The PR changes the Telegram callback-query fallback to forward `callback_data: ${data}` to the agent instead of the raw callback payload.

## Best Possible Solution

Land an equivalent targeted Telegram handler fix that prefixes only generic callback fallback text, preserves approval/plugin/select/model/native/slash callback routing, adds regression coverage, and collects real Telegram proof.

## Reproduction Assessment

Yes for the source-level routing contract: current main builds generic callback fallback messages from raw `data` while the docs promise `callback_data: <value>`. I did not run live Telegram/webhook proof in this read-only review.

## Solution Assessment

No. The prefix is directionally right, but the branch prefixes every fallback payload and needs a targeted rebase or replacement that preserves slash/native/internal callback routes plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve slash and native callback routing:** `extensions/telegram/src/bot-handlers.runtime.ts:1570`
  - body: This prefixes every fallback callback payload before command parsing. Telegram callbacks can carry slash commands such as `/models add <provider>`, and current main also preserves `tgcmd:` native callbacks; after this change those payloads no longer start with `/` or remain structured, so they are routed as ordinary chat text instead of command/native callback handling. Prefix only generic agent callback data and leave slash/native/internal callback paths raw or structured.
