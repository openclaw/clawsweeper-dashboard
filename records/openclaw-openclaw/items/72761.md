This PR routes drained runtime system events from reply prompt bodies into `extraSystemPrompt`, updates media-only prompt-placement tests, and also adds lazy plugin CLI memory-slot loading logic and tests.

## Best Possible Solution

Land a narrow system-events-only fix that keeps drained runtime-generated events out of `commandBody`, `followupRun.prompt`, `transcriptCommandBody`, and `followupRun.transcriptPrompt`, carries them through runtime system context, preserves heartbeat and ownership-downgrade semantics, adds a single-line changelog entry, and leaves plugin CLI activation behavior for a separate PR.

## Solution Assessment

No, not as submitted. Moving drained events into runtime system context is the narrow maintainable fix, but the live branch also changes plugin CLI loading and lacks release-note coverage; the safer solution is a cleaned branch containing only the system-event fix plus changelog and validation.
