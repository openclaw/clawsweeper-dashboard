## Summary

Keep open: this maintainer-labeled PR changes a prompt trust boundary, rewrites configured group system prompts at runtime, and still has only unit-test proof for an external contribution.

## What This Changes

The PR sanitizes finalized `GroupSystemPrompt` text, adds an optional `untrustedGroupSystemPrompt` supplemental fact routed into `UntrustedStructuredContext`, and updates inbound-context tests for Discord and generic channels.

## Best Possible Solution

Keep `GroupSystemPrompt` byte-preserving for trusted owner configuration, and only add a documented untrusted supplemental prompt-context field if maintainers approve that API and its runtime proof.

## Reproduction Assessment

Yes for the source-level behavior: current main sanitizes user-controlled body fields while leaving `GroupSystemPrompt` unchanged and later appends it to the system prompt. I did not run tests because this review is read-only.

## Solution Assessment

No as currently written: unconditional `GroupSystemPrompt` sanitization is not the safest fix because it changes trusted configured prompts for existing users. The safer path is to keep configured prompts intact and use a documented untrusted context field only for message-controlled prompt-like data.
