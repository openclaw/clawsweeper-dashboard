## Summary

Keep open. Current main has strong related fixes for sandboxed host skill paths, including the merged CLI startup fix in https://github.com/openclaw/openclaw/pull/92508, but this newer report is a distinct AlphaClaw managed-skill case where prompt-visible `~/.openclaw/skills/...` paths can still drift away from the live state root used by the `read` tool.

## What This Changes

Reports that managed skill paths in AlphaClaw Docker are advertised as `~/.openclaw/skills/...` and model `read` calls drift to stale host/workspace locations, causing Telegram skill-use loops.

## Best Possible Solution

Align managed skill prompt locations with the path space accepted by the `read` tool, ideally through the canonical managed skill snapshot/registry rather than broad host-path fallbacks, and cover AlphaClaw Docker plus the prior sandboxed startup path.

## Reproduction Assessment

Yes, partially at source level: current main still compacts prompt skill locations to `~` and the read path helper expands `~` without consulting the managed skill registry. I did not live-reproduce the exact AlphaClaw Telegram retry loop.

## Solution Assessment

Unclear. The narrow direction is to make prompt-advertised managed skill paths and `read` resolution share one canonical contract, but maintainers should choose whether that lives in prompt rendering, read-path resolution, or a managed-skill URI/registry path.
