## Label Justifications

- `P1`: A live AlphaClaw Docker Telegram skill workflow can get stuck in repeated failed reads even though the managed skill exists and is readable.
- `impact:session-state`: The failure is caused by prompt/session skill path state drifting away from the runtime path space accepted by the read tool.

## Summary

Keep open. Current main and the latest release still compact prompt-visible managed skill paths to `~` while the read path expands `~` with `os.homedir()`, and the linked open PR is the active implementation candidate rather than a merged fix.

## What This Changes

The issue reports that AlphaClaw Docker managed skill prompt paths can compact to `~/.openclaw/...`, resolve to stale host or workspace locations, and cause repeated `ENOENT` read loops.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/92894 or choose a different managed-skill URI/read-tool contract, then close this issue only after the fix is merged.

## Reproduction Assessment

Yes at source and reporter-proof level: the issue gives live AlphaClaw Docker logs and file-existence proof, and current main still has the compact-path/read-tool mismatch. I did not live-replay the full Telegram topic loop.

## Solution Assessment

Likely yes for the current narrow path: the open linked PR targets the prompt path contract and persisted snapshot reuse directly. Maintainers still need to accept that path contract or choose a managed URI/read-tool registry design instead.
