## Summary

Keep open: the OpenShell status bug is real and this PR repairs the backend-manager registration path, but the parent `sandbox` policy is too broad because it also makes config-only `sandbox explain` depend on all enabled plugins loading successfully.

## What This Changes

The PR adds a sandbox CLI command policy that loads plugins for sandbox commands and adds regression coverage for command policy resolution and plugin-owned sandbox backend status lookup.

## Best Possible Solution

Narrow plugin activation to the sandbox subcommands that inspect or remove backend-owned runtimes, while keeping config-only sandbox diagnostics available without loading plugins.

## Reproduction Assessment

Yes at source level: current main leaves sandbox commands on `loadPlugins: "never"`, OpenShell manager registration is gated on full plugin registration, and missing managers make registry entries report stopped.

## Solution Assessment

No as written: loading plugins for backend-touching sandbox subcommands is a good repair, but applying full activation to `sandbox explain` is broader than current source requires.

- **[P1] Keep plugin loading off sandbox explain:** `src/cli/command-catalog.ts:299`
  - body: This parent `sandbox` policy makes every sandbox subcommand load all enabled plugins before running. `sandbox explain` only resolves config/session/tool policy and does not call `listSandboxContainers()` or `getSandboxBackendManager()`, so an unrelated enabled plugin load failure would now break an existing config-only diagnostic command without helping the OpenShell status bug.
