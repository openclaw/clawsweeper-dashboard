## Summary

Keep this PR open: the source path supports the reported sandbox status regression and the patch is a plausible focused fix, but it still needs real OpenShell behavior proof and maintainer acceptance of loading plugin runtime for the whole `sandbox` CLI namespace before merge.

## What This Changes

The PR adds a `sandbox` CLI command policy that always loads plugins and adds unit coverage for sandbox command policy plus plugin-owned sandbox backend status handling.

## Best Possible Solution

Land a focused sandbox CLI policy and regression-test fix after live OpenShell proof, while keeping the plugin-loading scope acceptable for local sandbox commands.

## Reproduction Assessment

Yes, source-reproducible: current main defaults sandbox commands to `loadPlugins: "never"`, while OpenShell registers its manager only during full plugin registration and `listSandboxContainers` reports `running: false` when that manager is absent. I did not establish a live WSL/Ubuntu/OpenShell failing run in this read-only review.

## Solution Assessment

Yes, broadly: using the existing CLI command policy and bootstrap path is the narrow maintainable fix, and it avoids changing OpenShell's plugin registration contract. The remaining decision is whether `loadPlugins: "always"` should cover the whole `sandbox` namespace or be narrowed to status-touching subcommands.
