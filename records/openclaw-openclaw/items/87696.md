## Summary

Keep open: this PR addresses a real source-reproducible OpenShell sandbox status bug, but the current fix loads every effective enabled plugin before sandbox commands. Because CLI plugin bootstrap throws on any plugin load/register error, this can break existing Docker, SSH, and OpenShell sandbox status before command code runs.

## What This Changes

The PR adds a `sandbox` CLI catalog policy that always loads plugins and adds regression tests for OpenShell backend-manager status and missing-manager fallback behavior.

## Best Possible Solution

Use a sandbox-backend-scoped registration path, or get explicit maintainer acceptance that sandbox commands should activate every effective enabled plugin and fail when unrelated activation fails.

## Reproduction Assessment

Yes at source level: current main leaves sandbox commands on no-plugin startup, OpenShell registers its manager only during full plugin registration, and missing managers make `listSandboxContainers` report stopped entries. I did not run a live WSL plus OpenShell daemon reproduction in this read-only review.

## Solution Assessment

No as-is: the PR fixes the implicated path, but loading every enabled plugin is broader than the failing backend-manager registration path. A scoped backend registration path, or explicit maintainer acceptance of the availability tradeoff, is safer before merge.
