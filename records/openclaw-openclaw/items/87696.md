## Summary

Keep open: the PR targets the right source-level OpenShell sandbox status regression and includes useful runtime proof, but the current fix makes all sandbox CLI commands depend on successful activation of every enabled plugin, which is still a maintainer availability decision before merge.

## What This Changes

The PR adds a sandbox CLI catalog entry that loads plugins for sandbox commands and adds regression tests for OpenShell manager-backed status lookup and the missing-manager fallback.

## Best Possible Solution

Use a sandbox-backend-scoped registration path that installs backend managers without activating unrelated plugin runtime, or explicitly record maintainer acceptance that sandbox commands should fail when any enabled plugin fails to activate.

## Reproduction Assessment

Yes, at source level: current main leaves sandbox commands on the no-plugin-loading default, OpenShell registers its manager only during full plugin registration, and a missing manager makes listSandboxContainers report running: false. I did not run a live WSL+OpenShell daemon environment in this read-only review.

## Solution Assessment

No as-is: the fix is in the right area, but loading every enabled plugin is broader than the failing OpenShell backend-manager path. A scoped sandbox-backend registration path, or explicit maintainer acceptance of the availability tradeoff, is safer before merge.
