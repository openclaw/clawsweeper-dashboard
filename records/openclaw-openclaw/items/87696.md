## Summary

Keep open for normal PR review/merge: current main still has the sandbox CLI plugin-loading gap, and the patch is a focused fix with no blocking code findings. The main maintainer decision is whether to accept the availability tradeoff from loading all enabled plugins for the sandbox namespace.

## What This Changes

The PR adds a sandbox CLI startup policy that loads plugins for sandbox subcommands and adds regression tests for the policy and backend-manager status lookup.

## Best Possible Solution

Land this as the existing CLI-policy fix after maintainer acknowledgement of the all-plugin activation tradeoff, or pause for a separate sandbox-backend manifest scope if maintainers want docker/ssh inspection isolated from unrelated plugin failures.

## Reproduction Assessment

Yes. Current main's sandbox command path inherits `loadPlugins: "never"`, while OpenShell only registers `registerSandboxBackend("openshell", ...)` during full plugin registration, so the failure path is source-reproducible.

## Solution Assessment

Yes, with a maintainer risk decision. The patch uses the existing catalog/bootstrap path instead of hardcoding OpenShell, and the broader tradeoff is all-plugin activation because no sandbox-backend manifest scope exists today.
