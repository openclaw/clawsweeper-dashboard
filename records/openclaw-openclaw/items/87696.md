## Summary

Keep open: this PR targets the right OpenShell sandbox status regression and has useful proof, but the current patch makes sandbox commands depend on full enabled-plugin activation, which can turn unrelated plugin failures into `openclaw sandbox` availability failures.

## What This Changes

The PR adds a sandbox CLI catalog policy that loads plugins for sandbox commands and adds regression tests for OpenShell backend-manager status and missing-manager fallback behavior.

## Best Possible Solution

Use a sandbox-backend-scoped registration path that installs only the backend managers needed by sandbox registry entries, or explicitly record maintainer acceptance that sandbox commands should fail when any enabled plugin fails to activate.

## Reproduction Assessment

Yes, at source level: current main leaves sandbox commands on no-plugin startup, OpenShell registers its manager only during full plugin registration, and a missing manager reports registry entries as stopped. I did not run a live WSL+OpenShell daemon environment in this read-only review.

## Solution Assessment

No as-is: the PR fixes the implicated path, but loading every enabled plugin is broader than the failing backend-manager registration path. A scoped sandbox-backend registration path, or explicit maintainer acceptance of the availability tradeoff, is safer before merge.
