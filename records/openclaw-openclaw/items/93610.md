## Summary

Keep open. Current main and the latest release still lack an opencode-go stream-stall termination guard, and this issue is already the canonical report for multiple open candidate fixes, so it should stay open until a linked PR lands or is rejected.

## What This Changes

Fix intermittent opencode-go provider streams that deliver tokens but never send a final stream close, leaving cron runs stuck until diagnostic recovery aborts them.

## Best Possible Solution

Land a provider-owned opencode-go stream-stall fix with proof for both stalled and normal delayed completions, then close this issue from the merged PR; keep shared runtime/config compatibility changes out unless live evidence proves a reusable provider-family contract.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review because the provider failure is intermittent and needs opencode-go credentials. The source path is clear: current main waits for the upstream OpenAI-compatible iterator to close, and the issue trace shows tokens followed by silence until stuck-session abort.

## Solution Assessment

Yes for the current direction: a provider-owned, abortable opencode-go stream boundary matches the maintainer comments and avoids changing every OpenAI-compatible provider. The issue itself should remain open while the linked PRs are reviewed.
