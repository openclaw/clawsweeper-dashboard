## Summary

Keep open: current main still lacks an opencode-go stream-termination guard, the reported cron path can bypass the generic idle watchdog without an explicit timeout, and open linked PRs are still the active fix candidates.

## What This Changes

The issue reports intermittent opencode-go streamed model calls completing provider-side while OpenClaw never observes stream termination, causing cron runs to hang until stuck-session recovery aborts them.

## Best Possible Solution

Land one canonical provider-owned opencode-go stream-boundary fix, then decide whether the downstream stuck-recovery/failover classification PR remains separately useful.

## Reproduction Assessment

Unclear for a deterministic live repro: the provider failure is intermittent and was not forced here. The source path is reproducible by inspection because current cron-triggered runs can avoid the generic idle watchdog and the opencode-go plugin lacks the proposed stream-termination guard.

## Solution Assessment

Yes, the best direction appears to be a provider-owned opencode-go stream-boundary fix rather than a global OpenAI transport timeout. The issue should stay open until a canonical linked PR is merged or maintainers close that direction.
