## Summary

Keep open. Current main and the latest release still launch Codex ACP with an isolated `CODEX_HOME` that only materializes API-key auth, while Codex loads OAuth from the effective `CODEX_HOME` auth store; the reporter's OAuth-only failure is source-reproducible and still needs an auth-bridge decision.

## What This Changes

The issue requests that native ACP Codex sessions use the host Codex OAuth credential when spawning through OpenClaw's ACP route.

## Best Possible Solution

Choose a single ACPX-owned Codex OAuth bridge that keeps isolated safe config but makes host OAuth and refresh auth resolvable, with focused tests for OAuth-only and API-key hosts plus docs alignment.

## Reproduction Assessment

Yes, by source inspection rather than a live OAuth run: OpenClaw sets the ACP adapter's effective `CODEX_HOME` to an isolated state directory and does not copy host OAuth auth there, while Codex reads managed auth from the effective `codex_home` store. That matches the reported OAuth-only `Authentication required` failure before model selection.

## Solution Assessment

Unclear until maintainers choose the credential boundary. The reporter's copy/reference direction is plausible, but the best fix should be an explicit ACPX auth bridge that preserves the isolated config boundary instead of changing core `sessions_spawn` or adding another config knob.
