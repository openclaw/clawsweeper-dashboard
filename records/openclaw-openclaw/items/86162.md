## Summary

Keep this PR open, but it is not merge-ready: the current patch fast-acks raw Telegram callback payloads before plugin callback ownership can run, and the supplied live proof is from a related setup rather than the current PR head after the latest callback changes.

## What This Changes

The PR adds a pre-sequential Telegram callback acknowledgement path for selected final reply-button payloads, sends mapped visible acknowledgements after authorization, and adds callback-focused tests.

## Best Possible Solution

Keep the fast-ack idea, but gate it behind a collision-safe final-button contract or skip registered plugin namespaces, move visible acknowledgements after unhandled plugin callbacks, and prove the current branch in live Telegram.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the PR sends final-button acknowledgement text before plugin interactive dispatch, while plugin namespace matching can own the same raw values. I did not run a live repro because this review is read-only.

## Solution Assessment

No. The maintainable fix is to preserve plugin callback ownership first, either with a reserved final-button payload contract or by skipping registered plugin namespaces before any fast or visible acknowledgement is sent.
