## Summary

Keep open. Current main still has the OpenAI-bound tool schema normalization gap for object schemas whose `properties` key exists but is `undefined`, `null`, or otherwise not a usable object, and the linked closing PR is still open rather than merged.

## What This Changes

The issue reports a regression where `openai/gpt-5.4` rejects parameter-free MCP tools such as `flux-mcp__get_flux_instance` because their schemas reach OpenAI without a valid `properties: {}` object.

## Best Possible Solution

Keep this issue open until #75401 or an equivalent narrow fix merges. The durable fix belongs in the central OpenAI-bound tool-parameter normalizer, with regression coverage for missing, null, undefined, and non-object top-level `properties` values.

## Reproduction Assessment

Yes. Current main can be reasoned through with `{ type: "object", properties: undefined }`: the schema passes the top-level object guard unchanged, and the OpenAI Responses conversion path uses that normalizer before sending the function tool payload.

## Solution Assessment

Yes. The best direction is a narrow central-normalizer fix plus focused tests, because this avoids provider-specific call-site patches and covers MCP tools from any source before they reach OpenAI.
