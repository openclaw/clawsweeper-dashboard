## Summary

Keep open. Current main still has the reported normalization gap for object schemas whose `properties` key exists but is not a real properties object, and the issue is already paired with open closing PR #75401, so it should remain open until that PR or an equivalent fix merges.

## What This Changes

The issue reports a v2026.4.27 regression where `openai/gpt-5.4` rejects parameter-free MCP tool schemas, such as `flux-mcp__get_flux_instance`, because they are not normalized before OpenAI transport serialization.

## Best Possible Solution

Land PR #75401 or an equivalent narrow fix that treats absent, null, undefined, array, and non-object top-level `properties` values on object schemas as parameter-free schemas before OpenAI transport serialization, with regression coverage in the shared normalizer, provider OpenAI compat helper, and OpenAI transport contract paths.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path by passing `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` through the inspected normalizer and OpenAI transport conversion.

## Solution Assessment

Yes. A central schema-normalization fix is the narrow maintainable solution because it covers MCP tools and both OpenAI transport variants without adding a Flux-specific branch.
