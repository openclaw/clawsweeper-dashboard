## Summary

Keep open. Current main still has the reported normalization gap for object schemas whose `properties` key is present but not a real object, and PR #75401 is already the active closing implementation path.

## What This Changes

The issue requests a regression fix so parameter-free MCP tool schemas are normalized before OpenAI GPT-5.4 tool registration rejects them.

## Best Possible Solution

Keep the issue open until PR #75401 or an equivalent narrow fix lands, with the shared schema normalizer treating absent, null, undefined, array, and other non-object top-level `properties` values on object schemas as parameter-free schemas and covering that behavior in focused tests.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path because `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` can bypass normalization before OpenAI transport serialization.

## Solution Assessment

Yes in direction, but not complete on main. The narrow maintainable solution is to fix the shared schema normalizer and regression tests rather than adding Flux-specific or OpenAI-transport-only special cases.
