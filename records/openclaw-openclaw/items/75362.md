## Summary

Keep open. Current main still has the reported OpenAI-bound schema-normalization gap for object schemas whose `properties` key is present but not a real object, and #75401 is already an open unmerged closing PR for this regression.

## What This Changes

The issue reports that OpenClaw v2026.4.27 sends parameter-free MCP tool schemas to `openai/gpt-5.4` in a shape OpenAI rejects, and asks for OpenAI-bound normalization to restore `properties: {}` for those tools.

## Best Possible Solution

Review and land #75401, or an equivalent narrow patch, so OpenAI-bound schema normalization treats absent, `undefined`, `null`, and non-object top-level `properties` values on object schemas as parameter-free schemas, with regression coverage in the shared normalizer and OpenAI transport/provider paths.

## Reproduction Assessment

Yes. The issue provides a concrete v2026.4.27 OpenAI 400 log, and current main can be reproduced at the unit level by sending `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` through the inspected OpenAI-bound normalizer path.

## Solution Assessment

Yes. A narrow guard fix in the schema normalizer, backed by regression tests for malformed present `properties` values, is the most maintainable solution; this issue should stay open until the linked PR or an equivalent fix merges.
