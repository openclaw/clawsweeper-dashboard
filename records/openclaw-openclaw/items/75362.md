## Summary

Keep open. Current main still has the reported normalization gap for object schemas whose `properties` key is present but not a real object, and PR #75401 is already open as the closing fix for this regression.

## What This Changes

The issue reports a regression where OpenAI-bound parameter-free MCP tool schemas can reach `openai/gpt-5.4` without a valid top-level `properties` object and fail provider validation.

## Best Possible Solution

Land PR #75401 or an equivalent narrow fix that treats absent, null, undefined, array, and non-object top-level `properties` values on object schemas as parameter-free schemas before OpenAI transport serialization, with regression coverage in the shared normalizer and OpenAI transport paths.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path by passing `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` through the inspected normalizer.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable solution because it fixes the central schema guard instead of adding Flux-specific or transport-specific special cases; the issue should stay open until that fix or an equivalent one merges.
