## Summary

Keep open. Current main still has key-existence based schema guards that do not normalize typed object schemas when `properties` is present but not a real object, and the issue is already paired with open closing PR #75401; it should close only after that PR or an equivalent fix lands.

## What This Changes

The issue reports that OpenAI GPT-5.4 rejects parameter-free MCP tools such as `flux-mcp__get_flux_instance` after upgrading to v2026.4.27 because the outbound function schema lacks a valid `properties` object.

## Best Possible Solution

Land #75401 or an equivalent narrow patch in the shared tool-schema normalization path so typed object schemas with absent, null, undefined, or otherwise non-object `properties` become a valid empty object schema before OpenAI payloads are built, while preserving real existing property maps and adding regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an MCP tool schema like `{ type: "object", properties: null }` or an internal schema object with `properties: undefined`, which current main treats as already having `properties` instead of normalizing before the OpenAI tool payload path.

## Solution Assessment

Yes for the intended direction. The narrow maintainable fix is to tighten the shared schema guard and OpenAI strict compatibility helper rather than adding provider-specific special cases around `flux-mcp__get_flux_instance`; the issue should remain open until the linked PR or an equivalent patch is merged.
