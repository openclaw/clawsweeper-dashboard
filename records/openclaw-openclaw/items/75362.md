## Summary

Keep open. Current main still has a source-level path where object schemas with a present but invalid `properties` value bypass parameter-free normalization before OpenAI transport serialization, and the open closing PR is the correct place to resolve it before the issue is closed.

## What This Changes

The issue reports a regression where `openai/gpt-5.4` rejects Flux and similar parameter-free MCP tool schemas because OpenClaw does not always normalize empty object schemas before sending tools to OpenAI.

## Best Possible Solution

A narrow fix should treat top-level object schemas with absent, undefined, null, array, or other non-record `properties` values as parameter-free schemas at the OpenAI-family tool schema normalization boundary, with regression coverage for transport serialization and provider-hook normalization paths.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path: an object schema with an own `properties` key whose value is `undefined` or `null` bypasses normalization and is then serialized through the OpenAI Responses/WebSocket tool conversion paths.

## Solution Assessment

Yes in direction, but not complete on main. The narrow maintainable solution is to fix the schema normalizer and add regression tests, while reviewing PR #75401 to ensure it covers all current OpenAI-family normalization surfaces rather than only one call path.
