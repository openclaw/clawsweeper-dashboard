## Summary

Keep open. Current main still has the reported normalization gap for object schemas whose `properties` key is present but not a real object, and PR #75401 is already open as the closing implementation path.

## What This Changes

The issue requests restoring OpenAI-safe normalization for parameter-free MCP tool schemas so `openai/gpt-5.4` does not reject tools such as `flux-mcp__get_flux_instance` with `object schema missing properties`.

## Best Possible Solution

Keep this issue open until PR #75401 or an equivalent narrow fix lands, with the shared schema normalizer treating absent, null, undefined, array, and other non-object top-level `properties` values on object schemas as parameter-free schemas before OpenAI transport serialization.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path because `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` can pass the top-level object-schema guard without being rewritten to `properties: {}`.

## Solution Assessment

Yes in direction, but not complete on main. The narrow maintainable solution is to fix the shared normalizer and regression coverage rather than adding Flux-specific or OpenAI-transport-only special cases; the existing closing PR should be reviewed and fixed up if needed.
