## Summary

Keep open. Current main still has the OpenAI-bound schema-normalization gap described in the report, and the provided discussion shows an open closing PR already targeting this exact regression, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that OpenAI GPT-5.4 rejects parameter-free MCP tool schemas on v2026.4.27 when a tool reaches the OpenAI path without a valid top-level `properties` object.

## Best Possible Solution

Review and land #75401, or an equivalent narrow fix, so the central OpenAI-bound tool-schema normalization treats missing, null, undefined, array, and other non-object top-level `properties` values on object schemas as parameter-free schemas while preserving coverage for the shared normalizer and OpenAI transport paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` through the current normalizer path and it can return without inserting `properties: {}` because the guard checks key presence rather than value shape.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable solution: fix the central guard in `src/agents/pi-tools-parameter-schema.ts` and add focused regression tests for malformed `properties` values before the issue closes.
