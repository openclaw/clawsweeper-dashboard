## Summary

Keep open. Current main still has a source-level normalization gap for OpenAI/MCP object schemas whose top-level `properties` key exists but is `null`, `undefined`, an array, or another non-object value, and the linked PR #75401 is still the open closing fix candidate.

## What This Changes

The issue reports a regression where `openai/gpt-5.4` rejects parameter-free MCP tool schemas from `flux-operator-mcp` with `object schema missing properties` after upgrading to v2026.4.27.

## Best Possible Solution

Land PR #75401 or an equivalent narrow fix that treats absent, `null`, `undefined`, array, and non-object top-level `properties` values on object schemas as parameter-free before OpenAI transport serialization, with regression coverage in the shared normalizer, provider compat hook, and OpenAI transport paths.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path by passing an object schema with `properties: null` or `properties: undefined` through the inspected normalizer before the OpenAI transport conversion.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable solution because it fixes the central schema normalization path instead of adding Flux-specific or transport-specific special cases; the issue should stay open until that fix or an equivalent one merges.
