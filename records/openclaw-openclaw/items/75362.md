## Summary

Keep open. Current main still has the OpenAI-bound schema-normalization gap for object schemas whose `properties` key is present but not a real object, and the provided GitHub context shows an open closing PR targeting this exact regression; this issue should remain open until that PR or an equivalent fix merges.

## What This Changes

The issue reports a v2026.4.27 regression where OpenAI gpt-5.4 rejects parameter-free MCP tool schemas such as `flux-mcp__get_flux_instance` with `object schema missing properties`.

## Best Possible Solution

Review and land #75401, or an equivalent narrow fix, so the shared OpenAI-bound tool-schema normalizer treats absent, null, undefined, array, and other non-object top-level `properties` values on object schemas as parameter-free schemas where appropriate, with regression coverage in both the shared normalizer and OpenAI transport conversion paths.

## Reproduction Assessment

Yes. The issue includes a concrete v2026.4.27 OpenAI 400 log, and current main has a source-level reproduction path: `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` bypasses the missing-properties branch because the key exists.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix: tighten the object-schema guards in the central normalizer and add focused regression tests for malformed present `properties` values before the OpenAI payload is built.
