## Summary

Keep open. Current main still has the reported OpenAI/MCP schema-normalization gap for object schemas whose `properties` key is present but not a real object, and the provided GitHub context shows open PR #75401 already targets this exact regression with closing syntax.

## What This Changes

The issue requests restoring OpenAI GPT-5.4 compatibility for parameter-free MCP tools by normalizing empty or malformed object schemas before OpenAI transport serialization.

## Best Possible Solution

Land PR #75401 or an equivalent narrow fix that treats absent, null, undefined, array, and non-object top-level `properties` values on object tool schemas as parameter-free schemas before OpenAI transport serialization, with shared-normalizer and OpenAI transport coverage.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path by passing an object schema with present-but-invalid `properties` through the shared normalizer and OpenAI transport conversion.

## Solution Assessment

Yes. A central schema-normalization fix is the narrow maintainable path because it protects all affected MCP/OpenAI tool paths without adding Flux-specific or transport-specific special cases.
