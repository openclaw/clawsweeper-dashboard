## Summary

Keep open. Current main still has the schema-normalization gap for object schemas whose `properties` key exists but is `undefined`, `null`, or otherwise not a real object, and the provided GitHub context shows an open closing PR already targeting this exact regression.

## What This Changes

Restore OpenAI-bound MCP tool schema normalization so parameter-free object schemas with absent or malformed `properties` are sent with `properties: {}` instead of being rejected by `openai/gpt-5.4`.

## Best Possible Solution

Review and land #75401, or an equivalent narrow fix, so the central OpenAI-bound schema normalization treats absent, null, undefined, and non-object top-level `properties` values on object schemas as parameter-free schemas and preserves regression coverage for both the shared normalizer and OpenAI transport paths.

## Reproduction Assessment

Yes. The report includes a concrete v2026.4.27 OpenAI 400 log, and current main can be checked by passing `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` through the inspected normalizer path and observing that it does not insert `properties: {}`.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: tighten the object-schema guards and add regression tests for malformed `properties` values in the central OpenAI-bound normalization path. The issue should stay open until that PR or an equivalent fix merges.
