## Summary

Keep open. Current main still has the reported normalization gap for object schemas where the top-level `properties` key exists but is not a real properties object, and PR #75401 is already open with closing syntax for this issue.

## What This Changes

The issue requests that parameter-free MCP tool schemas be normalized before OpenAI GPT-5.4 transport serialization so OpenAI does not reject them with `object schema missing properties`.

## Best Possible Solution

Keep this issue open until PR #75401 or an equivalent narrow fix lands, with shared OpenAI/MCP schema normalization coercing absent, null, undefined, array, and other non-object top-level `properties` values on object schemas into an explicit empty properties object and covering both HTTP Responses and WebSocket paths.

## Reproduction Assessment

Yes. The issue includes a concrete OpenAI 400 log, and current main has a source-level reproduction path where `{ type: "object", properties: undefined }` or `{ type: "object", properties: null }` passes the complete-object guard without being normalized.

## Solution Assessment

Yes in direction, but it is not landed on main. The narrow maintainable fix belongs in the shared schema normalizer and regression tests, not in a Flux-specific or OpenAI-transport-only special case.
