## Summary

Keep open: current main still has both reported behaviors in source. The engine-owned compaction wrapper only closes the hook lifecycle when `result.compacted` is true, while direct compaction can return a successful no-op; the gateway validation formatter also still emits single-quoted additional-property names that are fed into cron tool errors.

## What This Changes

The issue asks OpenClaw to emit `after_compaction` for successful no-op compactions and make additional-property validation errors unambiguous for LLM tool callers.

## Best Possible Solution

Land a focused bug fix that closes successful no-op compaction cycles for observers and formats additional-property names in a JSON-unambiguous way, with targeted regression coverage for both paths.

## Reproduction Assessment

Yes at source level, but not with a live voice setup in this read-only review. Current main has an ok/no-op compaction return path that bypasses the queued wrapper's `after_compaction` gate, and the validator formatter still emits the single-quoted additional-property string asserted by tests.

## Solution Assessment

Yes for the narrow direction: emit an honest after-compaction event for successful no-op compactions and change the additional-property delimiter plus exact-string tests. A new recovery-strategy config should stay out of the first fix unless maintainers choose a broader product change.
