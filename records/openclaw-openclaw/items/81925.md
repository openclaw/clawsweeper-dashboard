## Summary

Keep open: current main still contains both reported source-level behaviors, and the adjacent closed reports or open PRs do not already land the exact fix. The compaction wrapper can emit `before_compaction` and skip the matching `after_compaction` on a successful no-op result, and the gateway protocol formatter still exposes additional-property names with single-quote delimiters in caller-visible validation errors.

## What This Changes

The issue requests a focused fix for no-op compaction hook completion and an unambiguous delimiter in additional-property validation errors.

## Best Possible Solution

Land a focused bug fix with regression coverage that emits an honest after-compaction event for successful no-op compactions and switches additional-property formatting to an unambiguous delimiter without adding a recovery-strategy config.

## Reproduction Assessment

Yes at source level: current main has a successful no-op compaction return path that bypasses the queued wrapper's `after_compaction` gate, and the validator formatter still emits the single-quoted additional-property text. I did not establish a live voice-channel reproduction in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to close the compaction hook lifecycle for `ok && !compacted` results and update the validation error string plus exact-string tests. The optional recovery-strategy config would be a product decision and should stay out of the first bug fix.
