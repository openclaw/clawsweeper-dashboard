## Summary

Close as not actionable in this repository: the relaunch loop came from a local out-of-band `launchctl submit` wrapper, while current main and v2026.5.22 do not ship, document, or generate that `ai.openclaw.oob-update.*` job family.

## What This Changes

Report that macOS one-shot update wrappers using `launchctl submit` can relaunch after clean exit and repeatedly restart the gateway when a no-op update path still reaches plugin sync and gateway restart.

## Best Possible Solution

Keep the current OpenClaw updater off `launchctl submit`; local wrappers should self-unload or use a generated one-shot LaunchAgent, and OpenClaw core should only add diagnostics for an officially produced or documented updater label family.

## Reproduction Assessment

No for an OpenClaw repo bug: the reported `launchctl submit` wrapper is plausible, but current main and v2026.5.22 have no source, docs, or helper path that produces that job family. The source-backed check instead shows an external wrapper incident plus a deliberately narrow legacy-label diagnostic.

## Solution Assessment

No, broadening core cleanup around an unowned local wrapper label is not the best fix without maintainer ownership approval. The safer solution is for the wrapper to use the validated one-shot LaunchAgent pattern outside OpenClaw, or for maintainers to first define an official handoff label/API that core can diagnose.
