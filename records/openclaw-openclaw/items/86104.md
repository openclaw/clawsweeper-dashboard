## Summary

Keep open. Current main and v2026.5.22 avoid `launchctl submit` in built-in update helpers, but status/doctor stale-updater diagnostics still only recognize legacy `ai.openclaw.update.*` labels and would miss the reported `ai.openclaw.oob-update.*` launchd loop; changing already-current update behavior also conflicts with current docs/tests and needs maintainer policy first.

## What This Changes

The issue requests safer macOS one-shot updater guidance plus broader stale launchd updater diagnostics for `ai.openclaw.oob-update.*` jobs and a no-op update path that exits before plugin sync or gateway restart.

## Best Possible Solution

Keep OpenClaw’s built-in updater off `launchctl submit`, then decide whether OpenClaw should officially recognize `ai.openclaw.oob-update.*`/manual update labels in status and doctor while leaving already-current package refresh semantics unchanged unless maintainers choose a new policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full macOS relaunch loop in this review, but source inspection confirms current main and v2026.5.22 ignore `ai.openclaw.oob-update.*` in stale updater diagnostics and intentionally keep already-current update finalization work.

## Solution Assessment

Is this the best way to solve the issue? Unclear; avoiding `launchctl submit` is the right one-shot pattern, but adopting an out-of-band update namespace and changing no-op update finalization require maintainer policy before a code change.
