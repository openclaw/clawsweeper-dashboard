## Label Justifications

- `impact:crash-loop`: The issue is about launchd repeatedly relaunching an updater and causing repeated gateway restarts despite successful exits.

## Summary

Keep open. Current main does not use `launchctl submit` for the built-in macOS update restart helper, but the existing stale-updater diagnostics still only recognize legacy `ai.openclaw.update.*` labels, so the reported `ai.openclaw.oob-update.*` restart loop is not fully covered by current code.

## What This Changes

The issue requests safer macOS one-shot updater guidance and broader stale launchd updater diagnostics after an out-of-band update wrapper repeatedly restarted the gateway despite a no-op package precheck.

## Best Possible Solution

Keep the built-in updater off `launchctl submit`, then decide whether to officially recognize OpenClaw-namespaced out-of-band update jobs in status/doctor guidance and document the supported one-shot LaunchAgent pattern.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live macOS reproduction was run, but source inspection confirms current diagnostics only match `ai.openclaw.update.*` and would miss the reported `ai.openclaw.oob-update.*` label.

## Solution Assessment

Is this the best way to solve the issue? Unclear; extending diagnostics looks narrow, but adopting external out-of-band updater labels and changing no-op update behavior need maintainer policy before an automated fix.
