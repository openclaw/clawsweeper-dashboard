## Label Justifications

- `P2`: This is a concrete provider-auth capability gap for Google Enterprise users, but it is a feature request rather than a confirmed regression in documented web_search behavior.
- `impact:auth-provider`: The issue is specifically about Google ADC credential resolution and auth routing for the web_search provider.

## Summary

Keep open: current main and v2026.6.10 still route the bundled Gemini web_search provider through API-key auth, while ADC bearer-token support is implemented only for the separate google-vertex model-provider path. This is a valid Google provider-auth feature request, but the permanent web_search API/config shape needs maintainer direction before implementation.

## What This Changes

Add Google Enterprise Application Default Credentials support to the Gemini web_search provider, similar to the existing google-vertex model-provider ADC path.

## Best Possible Solution

Implement accepted support inside the Google plugin by reusing the existing Vertex ADC helper or a narrow shared provider-auth contract, with focused docs and tests for both API-key and ADC web_search behavior.

## Reproduction Assessment

Yes, by source inspection: select the gemini web_search provider in an ADC-only Google environment with no API key and current main reaches missing_gemini_api_key before any ADC bearer-token path. I did not run a live Google Enterprise ADC web_search request.

## Solution Assessment

Unclear: Google-plugin-local support is the right boundary, but maintainers should choose the durable API shape before implementation. A direct webSearch.auth adc field is only one plausible option.
