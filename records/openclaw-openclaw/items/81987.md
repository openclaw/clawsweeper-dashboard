## Summary

Keep open: current main has manual abort primitives and model-auth status telemetry, but auth removal still only deletes saved provider profiles and does not stop or visibly flag active gateway model runs. This needs a maintainer decision on credential-removal semantics before automation changes gateway/UI behavior.

## What This Changes

The issue requests that removing a provider API key stop, warn about, or expose audit/stop controls for in-flight or scheduled LLM calls.

## Best Possible Solution

Define provider-auth removal as a gateway lifecycle event and connect it to active run tracking so affected provider runs are aborted or made clearly auditable with a documented stop path.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction against OpenRouter on current main. Source inspection does show auth removal has no active-run abort path while active attempts keep resolved runtime auth, so the issue is source-reproducible enough to keep open.

## Solution Assessment

Unclear until maintainers choose the intended credential-removal UX. Existing sessions.abort and chat.abort are the right lower-level primitives, but the missing piece is the provider-auth lifecycle policy that decides abort, warning, audit, or a combination.
