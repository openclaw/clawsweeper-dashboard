## Summary

Keep open: current main has manual abort primitives and model-call telemetry, but source inspection shows provider auth removal only deletes saved profiles and does not stop or visibly flag active model runs. The reporter's `@mapick/cost-firewall` context is useful mitigation evidence, but it does not replace a core credential-lifecycle decision.

## What This Changes

The issue requests that removing a model provider API key stop active or scheduled model calls, or at least expose a visible warning, audit, or stop path.

## Best Possible Solution

Define provider-auth removal as an active gateway lifecycle event, then wire it to existing run tracking so affected provider runs are aborted or visibly reported with a documented stop/audit path.

## Reproduction Assessment

No, not with high-confidence live proof. Source inspection shows saved-profile removal has no active-run abort path while active runs use abort controllers and resolved runtime auth, so the issue is source-reproducible but still needs a focused real or simulated provider reproduction.

## Solution Assessment

Unclear until maintainers choose the intended user experience. Existing `sessions.abort` and `/stop` are the right lower-level primitives, but the missing piece is the provider-auth lifecycle policy that decides whether key removal automatically aborts, warns, or exposes an audit/stop command.
