## Summary

Keep open: the report is plausible and source-backed, but current main does not show a credential-removal path that automatically aborts or visibly warns on existing model runs. The existing session abort/list APIs are useful mitigation, not a complete fix for removing a provider key while gateway work is already active.

## What This Changes

The issue asks the gateway to stop, warn, or expose audit/stop controls when model API credentials are removed while active or scheduled LLM work may continue.

## Best Possible Solution

Define a core credential-removal policy that invalidates or visibly warns on affected active/scheduled runs, using existing session/run tracking and abort primitives rather than relying on a separate cost plugin.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a plausible reproduction path: remove a model provider credential through config/auth state while an active run already has runtime auth, and current code does not connect that removal to the active-run abort path.

## Solution Assessment

Unclear until maintainers choose the product policy. The narrowest maintainable fix is likely to detect credential removal, identify affected active work, and either abort it with a visible event or surface an explicit warning plus a first-class stop/audit path.
