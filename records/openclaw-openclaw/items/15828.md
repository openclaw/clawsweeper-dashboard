## Summary

Keep open: current main and v2026.6.10 still leave bundled session-memory on the command-reset path, while the requested idle, shutdown, periodic checkpoint, and summary-output behavior remains missing and needs maintainer product/security decisions. Related PRs only cover pieces of the daily/idle surface and are not merged clean replacements for the broader request.

## What This Changes

Request automatic session-memory checkpoints on idle, shutdown, periodic message-count thresholds, and configurable LLM summary output instead of relying only on manual `/new` or `/reset`.

## Best Possible Solution

Keep this as the broad tracker, consolidate the daily/idle reset work with the active PR path, and define one bounded opt-in memory finalization contract covering shutdown, message-count checkpoints, summaries, dedupe, docs, tests, and privacy policy.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 keep session-memory registered and implemented for command:new/command:reset while idle rollover and shutdown use adjacent lifecycle paths that do not invoke it.

## Solution Assessment

Unclear as an implementation path. The requested behavior is useful, but the best fix needs maintainer decisions on trigger semantics, dedupe, timeout behavior, privacy, provider use, and overlap with existing daily/idle reset PRs.
