## Summary

Keep open. Current main and the latest release still implement bundled session-memory as a command-only hook for `/new` and `/reset`; daily/idle rollover and gateway shutdown lifecycle paths exist, but they do not currently make this hook save memory, and the related daily/idle PRs are open, conflicting, and only partial coverage for the broader request.

## What This Changes

The issue requests automatic session-memory checkpoints for idle, shutdown, message-count, and command triggers, plus an optional LLM-generated summary output mode.

## Best Possible Solution

Keep this as the umbrella session-memory automation tracker: choose one canonical daily/idle implementation, then separately scope shutdown, message-count checkpoints, and summary-output behavior with privacy guardrails, docs, and focused tests.

## Reproduction Assessment

Yes, from source inspection: session-memory accepts only command `new` and `reset` events, while daily/idle rollover and gateway shutdown use separate lifecycle paths. I did not run a live gateway reproduction because this is a read-only review and the source path is sufficient to prove the missing behavior.

## Solution Assessment

Unclear as a single implementation path. The direction is useful, but the best fix requires maintainer selection of the hook contract and config semantics before combining daily/idle rollover, shutdown checkpointing, message-count checkpoints, and summary output.
