- `P2`: The issue affects real session memory continuity, but it is bounded to memory/session lifecycle behavior and needs product/security design rather than emergency handling.

## Summary

Keep open: current main and v2026.6.10 still keep the bundled session-memory hook on the explicit command path, while idle, shutdown, periodic checkpoint, and summary-output behavior remain unresolved and product/security-sensitive.

## What This Changes

Request auto-triggered session-memory checkpoints for idle, shutdown, and message-count events, plus configurable summary-style memory output.

## Best Possible Solution

Keep this as the broad tracker, land or close the daily/idle slice through the existing PR path, then define one bounded opt-in memory finalization contract for shutdown, message-count checkpoints, summaries, dedupe, docs, tests, and privacy policy.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 keep session-memory registered and implemented for command:new/command:reset while idle rollover and shutdown use adjacent lifecycle paths that do not invoke it.

## Solution Assessment

Unclear as an implementation path. The requested behavior is useful, but the best fix needs maintainer decisions on trigger semantics, dedupe, timeout behavior, privacy, provider use, and overlap with existing daily/idle reset PRs.
