## Summary

Keep open. Current main still implements bundled session-memory as command-only persistence for `/new` and `/reset`; idle/daily rollover, shutdown checkpointing, message-count checkpointing, trigger configuration, and concise summary output remain unimplemented, while the related daily/idle PRs are still open partial implementations.

## What This Changes

Add configurable idle, shutdown, message-count, and command triggers plus optional LLM summary output to the bundled session-memory hook.

## Best Possible Solution

Keep this as the broader session-memory automation tracker and land a documented trigger surface that preserves `/new` and `/reset` while adding maintained daily/idle, shutdown-safe, message-count, and optional summary-output paths with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: enable session-memory and exercise stale daily/idle rollover, gateway shutdown, or normal message flow; current main only registers and accepts `command:new` and `command:reset`, and tests assert non-command events do not write memory.

## Solution Assessment

Unclear as a single one-shot change. The requested direction is valid, but the maintainable fix should first reconcile the overlapping daily/idle PRs, then separately design shutdown durability, message-count state, and summary-output behavior.
