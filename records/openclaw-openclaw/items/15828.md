## Summary

Keep open: current main and v2026.6.10 still keep bundled session-memory scoped to command-driven new/reset captures, while idle/daily rollover, shutdown, periodic checkpointing, trigger config, and summary-output behavior remain unresolved and product/security-sensitive.

## What This Changes

Requests configurable automatic session-memory captures on idle, shutdown, message-count checkpoints, and optional LLM summaries instead of only manual command-triggered memory saves.

## Best Possible Solution

Keep this as the broad tracker while maintainers consolidate the daily/idle path, then define one opt-in, bounded memory-finalization contract for shutdown, periodic checkpoints, and summary output with docs, privacy rules, dedupe behavior, and targeted tests.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 still register and handle bundled session-memory only for command new/reset, while idle/daily rollover and shutdown use adjacent lifecycle paths that do not invoke that handler.

## Solution Assessment

Unclear as a final implementation shape. The outcome is useful, but the exact trigger/config surface should be chosen with the overlapping daily/idle work and explicit privacy, timeout, dedupe, and summary semantics.
