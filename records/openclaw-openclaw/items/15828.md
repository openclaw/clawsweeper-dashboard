## Summary

Keep open: current main and v2026.6.6 still keep the bundled session-memory hook scoped to command-driven `/new` and `/reset`, while the requested idle, shutdown, periodic checkpoint, trigger-config, and summary-output behavior remains unimplemented and needs maintainer direction.

## What This Changes

The issue requests configurable session-memory auto-save triggers for idle, shutdown, message-count checkpoints, and optional LLM summaries instead of only command-triggered saves.

## Best Possible Solution

Keep this as the broader tracker and have maintainers choose one canonical memory-finalization contract before landing targeted work for daily/idle, shutdown, periodic checkpoints, and summary output.

## Reproduction Assessment

Yes, source-level with high confidence: current main and v2026.6.6 show session-memory only handling command new/reset events while stale rollovers and shutdown use adjacent lifecycle paths. I did not run a live gateway idle/shutdown scenario in this read-only review.

## Solution Assessment

Unclear as an implementation shape: the requested outcome is useful, but adding the exact proposed trigger/config surface wholesale is not proven to be the safest owner boundary. The best path is a maintainer-selected lifecycle contract that avoids duplicate command hooks and defines privacy/timeouts for automatic memory writes.
