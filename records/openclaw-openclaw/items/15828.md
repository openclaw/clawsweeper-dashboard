## Label Justifications

- `P2`: The issue addresses meaningful memory continuity but is a bounded enhancement requiring lifecycle and product design rather than emergency remediation.
- `impact:session-state`: The requested checkpoints determine whether session memory follows automatic rollover and gateway lifecycle transitions.
- `impact:data-loss`: The command-only writer can leave expected memory artifacts unwritten when a session ends automatically or the gateway shuts down.
- `impact:security`: Automatic transcript persistence and LLM summarization change how sensitive conversation content is retained and processed.

## Summary

Current main still leaves the bundled session-memory writer command-only, so the broader request remains necessary. Main now provides typed `session_end` lifecycle data for rollover and shutdown, making that the preferred integration point, while periodic checkpoints and generated summaries still require product and privacy decisions.

## What This Changes

Requests automatic session-memory checkpoints on idle, shutdown, and message-count thresholds, plus optional LLM-generated summaries.

## Best Possible Solution

Use the typed `session_end` plugin lifecycle as the single path for daily, idle, shutdown, and restart memory capture, preserve explicit command behavior through that same canonical owner where feasible, and design message-count snapshots plus generated summary content as separately approved opt-in capabilities with documented privacy and deduplication semantics.

## Reproduction Assessment

Yes at source level: current main clearly keeps bundled session-memory capture on `/new` and `/reset`, while daily, idle, shutdown, and restart endings flow through a separate typed lifecycle. This read-only review did not run a live gateway scenario.

## Solution Assessment

No, the proposed parallel internal-trigger/config design is not the best current solution because main now has a canonical typed `session_end` contract. The session-ending portion should integrate there, while periodic and LLM-summary behavior needs separate design approval.
