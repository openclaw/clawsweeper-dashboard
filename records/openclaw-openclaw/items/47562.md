## Summary

Keep open. This is a valid Telegram feature, but the PR still needs changes: the custom-only action path is validated after `emoji` is required, the branch is stale against current Telegram plugin file names, and the external PR has no real Telegram behavior proof.

## What This Changes

The PR adds Telegram custom emoji reaction support for inbound `message_reaction` updates and outbound reaction sends/actions, plus bot and send tests.

## Best Possible Solution

Port a narrow reaction-only implementation onto current main: expose the accepted custom emoji id input shape, send `ReactionTypeCustomEmoji`, emit inbound custom emoji reaction events after existing gates, document the reaction behavior, add focused schema/action/send/inbound tests, and collect real Telegram proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows a custom-only action call reaches `readReactionParams` before `customEmojiId` is read, and that helper requires `emoji`. The broader feature itself is not a bug reproduction; current-main source confirms it is still absent.

## Solution Assessment

No. The upstream Telegram/grammY contract supports the feature, but this PR is not the best current form because it targets stale files and leaves the custom-only action/schema path blocked; a current-main port with proof is safer.
