## What This Changes

The PR adds bidirectional Telegram custom emoji reaction handling for inbound `message_reaction` updates, outbound reaction sends/actions, and focused Telegram bot/send tests.

## Best Possible Solution

Port the reaction-only feature onto current Telegram plugin surfaces with `customEmojiId` schema/action support, `custom_emoji` send payloads, inbound events after existing gates, docs/tests, and redacted real Telegram proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the custom id is parsed after a required `emoji` parser, and the warning-result branch still keys custom-only metadata off `isEmpty`. The requested custom reaction support itself is a new Telegram feature rather than a broken current-main contract.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best merge path until it is ported to current runtime files, fixes custom-only action result paths, updates schema/docs/tests, and supplies real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Allow customEmojiId without an emoji:** `src/agents/tools/telegram-actions.ts:149`
  - body: The advertised custom-only flow still calls `readReactionParams` before reading `customEmojiId`, and that helper requires an `emoji` key. A call like `{ customEmojiId: "..." }` fails validation before `reactMessageTelegram` can send the custom reaction, so parse the custom id before deciding whether `emoji` is required.
  - confidence: 0.91
- **[P2] Report custom-only warnings as add attempts:** `src/agents/tools/telegram-actions.ts:182`
  - body: When `reactMessageTelegram` returns a warning for `emoji: ""` plus `customEmojiId`, this branch still reports `{ removed: true }` because `isEmpty` is true. That tells the agent a failed custom emoji add was a removal, so the warning payload should include custom add metadata instead.
  - confidence: 0.84
