## Summary

Keep open. Current main still only handles standard Telegram emoji reactions, while this PR is useful source work but is stale against the current Telegram plugin files and still has a custom-only action validation bug. Earlier review comments correctly identified the action-layer issue; the latest PR head fixes the success metadata case but does not remove the required `emoji` gate or update the current shared message-tool schema.

## What This Changes

The PR adds Telegram custom emoji reaction support for inbound `message_reaction` updates and outbound reaction sends/actions, with bot and send tests.

## Best Possible Solution

Port the feature into the current Telegram plugin surfaces: add `customEmojiId` to the shared message-tool schema and Telegram action runtime, send Telegram `custom_emoji` reaction payloads from `reactMessageTelegram`, emit inbound custom reaction system events only after the existing authorization and notification gates, update docs/changelog, and cover send/action/inbound/schema behavior with focused tests.

## Reproduction Assessment

Yes. Current-main inspection gives a high-confidence reproduction path: a custom-only reaction call has no schema/runtime parameter path, and a `message_reaction` update containing only `type: "custom_emoji"` is filtered out before enqueueing any system event.

## Solution Assessment

No. The feature direction is narrow and valid, but this PR is not the best current fix because it targets obsolete files and leaves the required-`emoji` validation/schema path in place; a current-main port or replacement PR is safer.
