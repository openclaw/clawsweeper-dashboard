## Summary

Keep open. Current main still lacks Telegram custom emoji reaction support, and this external PR remains unmergeable as-is because the advertised custom-only action still validates `emoji` before using `customEmojiId`, the branch edits pre-refactor files that are absent on current main, and it lacks real Telegram proof.

## What This Changes

The PR adds inbound Telegram `custom_emoji` reaction events and outbound `customEmojiId` reaction sending/action plumbing, plus bot and send tests.

## Best Possible Solution

Port the reaction-only feature into current Telegram plugin surfaces, add `customEmojiId` to the shared message-tool schema and action runtime, preserve existing reaction gates, update docs/tests, and require redacted real Telegram proof before merge.

## Reproduction Assessment

Yes, source-level for the PR defect. The PR reads `customEmojiId` after `readReactionParams`, and the current helper requires the `emoji` key before Telegram sending starts.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best mergeable solution until it is ported to current main's runtime/schema files, fixes custom-only validation, updates docs, and supplies real Telegram proof.
