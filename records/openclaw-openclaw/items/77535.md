## Summary

Close as implemented: current main already routes inbound Discord audio through the generic TTS pipeline and sends synthesized Discord replies as native voice messages, while this PR duplicates that with hard-coded channel IDs and a direct OpenAI call.

## What This Changes

The PR adds a Discord monitor TTS hook that turns final text replies to inbound audio messages into OpenAI-synthesized Discord voice messages for two hard-coded channel IDs, plus a preflight audio-origin flag.

## Best Possible Solution

Keep the shipped generic auto-TTS path and configure `messages.tts.auto: "inbound"` with the desired Discord/provider TTS settings.

## Reproduction Assessment

Not applicable. This is a feature PR, and the current-main verification is source/test/release proof that the equivalent Discord inbound-audio auto-TTS path already exists.

## Solution Assessment

No. The PR is not the best way to solve this because current main already uses the maintained provider-backed TTS pipeline, while this patch adds a second hard-coded OpenAI/channel-specific path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the configured TTS pipeline:** `extensions/discord/src/monitor/tts-response.ts:70-81`
  - body: This new hook posts every eligible final reply directly to OpenAI with `OPENAI_API_KEY`, `tts-1`, and `onyx`, bypassing `messages.tts`, provider selection, SecretInput/baseUrl handling, and account/voice overrides. Current main already has provider-backed auto-TTS for Discord voice-note delivery, so this would send operator-configured non-OpenAI deployments to OpenAI instead.
  - confidence: 0.93
- **[P2] Remove the hard-coded Discord channel IDs:** `extensions/discord/src/monitor/tts-response.ts:37`
  - body: Bundling two live Discord channel IDs makes the feature work only for that deployment and gives other operators no visible configuration or opt-out path. This should be driven by existing TTS/account configuration rather than committed deployment config.
  - confidence: 0.9
- **[P1] Avoid reintroducing the old Carbon REST type:** `extensions/discord/src/monitor/tts-response.ts:25`
  - body: The added file imports `RequestClient` from `@buape/carbon`, but current main has internalized the Discord client and `extensions/discord/package.json` no longer depends on Carbon. On a current-main rebase this new import should use the local `../internal/discord.js` type or the branch will fail to build without reintroducing that dependency.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a hidden outbound OpenAI data path and deployment-specific channel activation outside the configured TTS/security model.

Concerns:

- **[high] Bypasses configured TTS provider and secret policy:** `extensions/discord/src/monitor/tts-response.ts:70`
  - body: The hook sends final reply text to OpenAI directly with `process.env.OPENAI_API_KEY`, ignoring configured TTS providers, SecretInput resolution, base URLs, model/voice settings, and account overrides. That can disclose replies to OpenAI even when the operator configured a local or different TTS provider.
  - confidence: 0.93
- **[medium] Hard-codes live Discord channel configuration:** `extensions/discord/src/monitor/tts-response.ts:37`
  - body: The patch commits two Discord channel IDs as activation policy, creating hidden deployment-specific behavior in the bundled plugin instead of an explicit operator-controlled config surface.
  - confidence: 0.88
