## Summary

Current main already implements the maintainable version of this behavior: configurable auto-TTS can trigger on inbound audio, Discord advertises native voice-note delivery, and outbound Discord payloads marked as voice audio are sent through the existing voice-message path. The PR is redundant and should not merge because it adds a hardcoded, OpenAI-only side path.

## What This Changes

Adds a Discord monitor hook that converts final replies to OpenAI TTS voice messages for audio-origin messages in two hardcoded Discord channels.

## Best Possible Solution

Keep the shipped provider/capability-driven TTS and Discord voice-message pipeline, and do not merge this hardcoded OpenAI-specific hook.

## Reproduction Assessment

Not applicable; this is a feature PR, not a bug report. Current-main source, tests, and docs provide high-confidence proof that the intended voice-note TTS behavior already exists through the generic TTS pipeline.

## Solution Assessment

No; the PR is not the best way to solve this because it bypasses the shipped configurable TTS runtime and hardcodes Discord channel IDs. The safer solution is the existing `messages.tts.auto`/session TTS path plus Discord's `audioAsVoice` delivery.
