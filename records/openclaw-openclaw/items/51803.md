## Summary

Keep open: the reliability bug is real, but this PR's replay path still risks message loss, misrouting, and user-text authority elevation, and the branch is currently conflicting with main.

## What This Changes

This PR adds pending followup and drain-rejected message persistence with startup replay through system events, plus an unrelated Control UI `/model list/status` parsing fix and a committed runtime log.

## Best Possible Solution

Land one rebased maintainer-approved recovery path that persists accepted followups and drain-window arrivals as typed untrusted work, replays through normal queue or dispatch semantics with original routing/order/idempotency, and removes unrelated artifacts.

## Reproduction Assessment

Yes, source-level. Current main keeps followup queues in process memory and the PR itself demonstrates the risky replay path by consuming persisted data into ephemeral system events.

## Solution Assessment

No. The persistence goal is right, but system-event replay is not the best fix; the maintainable solution should replay typed untrusted work through the existing queue or inbound dispatch contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Replay persisted messages through typed dispatch:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: `enqueueSystemEvent()` only adds text for a later session turn; it does not preserve normal followup routing, does not start the target session, is capped by the system-event queue, and later renders the text as `System:` lines. Persisted followups and drain-window arrivals need to be rehydrated through the normal queue/dispatch path so replies go back to the original channel/thread in the right order.
  - confidence: 0.9
- **[P2] Preserve full inbound turn data before drain persistence:** `src/auto-reply/dispatch.ts:43-56`
  - body: `extractPersistedItem` stores only `Body`/routing fields, so media-only turns, command authorization/body semantics, `BodyForAgent`-specific adapters, attachments, and current inbound context cannot be faithfully replayed. Drain-window persistence should capture the typed context needed by the replay path, not only plain text.
  - confidence: 0.86
- **[P2] Keep persisted followups durable until replay drains:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` deletes the file before the messages are actually processed, and the PR then moves them into the non-persistent system-event queue. A second restart before the next session turn loses all consumed items, which is the failure mode this patch is meant to prevent.
  - confidence: 0.88
- **[P2] Persist queue overflow summary state:** `src/auto-reply/reply/queue/persist.ts:92-93`
  - body: The persist loop serializes only `state.items`, but current queue state also stores summarized overflow in `droppedCount`, `summaryLines`, and `summarySources`. Restarting a capped `summarize` queue would still discard the older summarized messages.
  - confidence: 0.82
- **[P2] Remove the committed runtime log:** `openclaw-2026-03-18.log:1`
  - body: This root log file is local runtime/debug output with embedded tool-call metadata and is unrelated to the product change. It should not be committed into the source tree or released with the package.
  - confidence: 0.95
