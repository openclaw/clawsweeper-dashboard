## Summary

Keep open. Current main still has the gateway restart message-loss gap, but this PR is not mergeable as proposed because replaying recovered user work as system events changes dispatch, trust, routing, batching, caps, and durability semantics, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds persisted followup/drain-window replay across gateway restart, persistence tests, a committed runtime log artifact, and an unrelated Control UI `/model list/status` change.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that replays structured followup and inbound work through typed untrusted dispatch/requeue contracts, preserves origin routing and overflow state, and consumes recovery data only after safe processing succeeds.

## Reproduction Assessment

Yes, at source level. Current main keeps accepted followups in process-local `FOLLOWUP_QUEUES`, rejects drain-window enqueues with `GatewayDrainingError`, and has no durable replay path in `src/auto-reply` or `src/gateway`.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR's system-event replay is not the narrow maintainable fix because it changes trust, routing, wakeup, batching, caps, and durability semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Replay recovered work through dispatch contracts:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: Recovered followups and drain-window messages are flattened into `enqueueSystemEvent()` calls. That bypasses the existing followup and inbound dispatch contracts, so recovered work can sit idle, batch differently, hit the system-event cap, or reply through the next unrelated surface instead of the original route.
  - confidence: 0.93
- **[P1] Preserve user trust during replay:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: The replay text says it is user-originated, but the code still calls `enqueueSystemEvent()` without `forceSenderIsOwnerFalse: true` or an equivalent untrusted path. Current system events drain as `System:` lines, so restart-window user text can gain higher prompt authority than normal inbound messages.
  - confidence: 0.96
- **[P2] Persist complete drain-rejected contexts:** `src/auto-reply/dispatch.ts:37-56`
  - body: The drain-window serializer keeps only text plus a few routing fields before replaying as a system event. Rich turns such as attachments, command authorization/body fields, structured context, and `BodyForAgent` versus envelope distinctions cannot be faithfully recovered.
  - confidence: 0.88
- **[P2] Keep recovery files durable until replay succeeds:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` deletes the recovery file before the recovered work is processed, then stores it only in the in-memory system-event queue. A second restart before the next turn drains those events loses the recovered messages.
  - confidence: 0.9
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:88-93`
  - body: The serializer only writes `state.items`, but capped queues can already have pending dropped-message state in `droppedCount` and `summaryLines`. Restart persistence would keep only the tail items and discard summarized overflow content.
  - confidence: 0.84
- **[P2] Remove the committed runtime log artifact:** `openclaw-2026-03-18.log:1`
  - body: `openclaw-2026-03-18.log` contains local runtime/debug and tool-call metadata unrelated to the fix. Keeping it in source adds avoidable information disclosure and release-artifact noise.
  - confidence: 0.95
