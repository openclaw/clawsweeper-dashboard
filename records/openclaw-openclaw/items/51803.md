## Summary

Keep this PR open for maintainer review: the linked restart message-loss bug is still source-reproducible on current main, but this branch is not merge-ready because replay goes through system events instead of typed dispatch/requeue contracts, real restart proof is missing, and the diff has prompt-trust plus committed-log security concerns.

## What This Changes

The PR persists followup queues and drain-window rejected messages across gateway restart, replays them as session system events on startup, adds persistence tests, changes Control UI `/model list/status` parsing, and adds a runtime log file.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that persists accepted followups and drain-window arrivals, replays them through typed untrusted dispatch or requeue contracts, preserves routing/overflow/idempotency, and consumes recovery data only after safe processing succeeds.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted followups live only in `FOLLOWUP_QUEUES`, drain-window enqueues reject with `GatewayDrainingError`, and there is no durable followup/drain replay path; I did not run a live gateway restart in this read-only pass.

## Solution Assessment

No. Persisting pending work is the right direction, but system-event replay is not the safest maintainable fix because it changes trust, routing, wakeup, batching, cap, and durability semantics.
