## Summary

Keep open: this PR targets a real gateway restart data-loss bug, but the current branch still replays recovered user messages through ephemeral system events, loses normal routing/wakeup/durability semantics, drops rich inbound context, and includes a committed runtime log artifact.

## What This Changes

The PR adds disk persistence and startup replay for pending followup queues and drain-window rejected inbound messages, plus an unrelated Control UI `/model list/status` fix and Vitest include update.

## Best Possible Solution

Rework the branch around durable structured replay through the existing followup and inbound dispatch paths, with explicit session wake/routing, acknowledgement-safe file retention, full context preservation, log-artifact removal, and a separate focused PR for the `/model` UI fix.

## Reproduction Assessment

Yes. Source inspection shows current main keeps followup queues in memory and rejects new queue enqueues once gateway draining is set; the PR path can also be traced from persisted item to `enqueueSystemEvent` instead of normal dispatch. No live restart test was run under the read-only review constraint.

## Solution Assessment

No. System-event replay is not the best way to solve the data-loss issue because it changes routing, wakeup, trust, batching, and durability semantics; structured re-enqueue or redispatch through the normal delivery machinery is the safer maintainable direction.

Overall confidence: 0.92

Full review comments:

- **[P1] Replay recovered items through the normal reply path:** `src/gateway/replay-persisted-followups.ts:35`
  - body: Recovered followups are flattened into `enqueueSystemEvent()` entries instead of being rebuilt as followup queue entries and drained through `runFollowup`. System events only appear on a later turn, do not wake the target session, and bypass the origin-routing logic used by normal queue drain, so restart recovery can still leave replies idle or delivered with unrelated traffic.
  - confidence: 0.93
- **[P1] Preserve replayed user text as untrusted input:** `src/gateway/replay-persisted-followups.ts:35`
  - body: This call does not pass `trusted: false`; current system events default to trusted and drain as `System:` lines. Adding prose that says the text is user-originated does not change the role boundary, so a queued user message can gain system-context authority after restart.
  - confidence: 0.96
- **[P2] Persist the full drain-rejected inbound context:** `src/auto-reply/dispatch.ts:43-56`
  - body: The drain-window serializer keeps only text and a few routing fields before replaying as a system event. Media-only messages, command authorization/body fields, and `BodyForAgent` versus envelope distinctions cannot be faithfully recovered after restart.
  - confidence: 0.88
- **[P2] Keep replay files until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:145`
  - body: `consumePersistedFollowups()` unlinks the pending file before replay has actually processed the items, and replay then stores them only in the in-memory system-event queue. A second restart before any turn drains those events permanently loses the recovered messages.
  - confidence: 0.9
- **[P2] Write pending followups with atomic replacement:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly, while parse failures on boot delete the file. If the process dies during truncate/write, the next boot can unlink the partial file and lose the queued followups this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Remove the committed runtime log:** `openclaw-2026-03-18.log:1`
  - body: `openclaw-2026-03-18.log` contains local runtime/debug and tool-call metadata unrelated to the fix. Keeping it in source leaks implementation details into clones and release artifacts.
