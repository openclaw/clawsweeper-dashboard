## Summary

Keep open: current main does not implement persisted followup/drain-window replay, but this PR is not merge-ready because the replay path still loses semantics and trust boundaries, and the branch includes an unrelated runtime log artifact.

## What This Changes

The PR adds followup and drain-rejected message persistence/replay through system events, shutdown/startup hooks, tests, an unrelated Control UI `/model` slash-command fix, and a committed runtime log file.

## Best Possible Solution

Use the existing durable session-delivery/restart-continuation pattern or normal inbound dispatch path to persist and replay user-originated work with route, trust, wake, retry, and queue-summary semantics intact.

## Reproduction Assessment

Yes from source inspection, though I did not run a live gateway restart. Current main keeps `FOLLOWUP_QUEUES` in memory and has no persistence hook before shutdown clears runtime state, so pending items can be lost on process restart.

## Solution Assessment

No. The PR’s system-event replay is not the best fix because it passively queues trusted text for a future turn rather than durably replaying user-originated inbound work through the dispatch/session-delivery path with routing and retry semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark replayed user text untrusted:** `src/gateway/replay-persisted-followups.ts:35`
  - body: Both replay paths enqueue persisted user prompts as system events without `trusted: false`. In current main, `enqueueSystemEvent` defaults to trusted and `drainFormattedSystemEvents` formats those as `System:` lines, so a user message received during restart can gain system-role authority despite the warning text embedded in the prompt.
  - confidence: 0.93
- **[P2] Replay through dispatch instead of passive system events:** `src/gateway/replay-persisted-followups.ts:35`
  - body: Putting persisted messages into the system-event queue does not start the target session, does not run the normal per-item followup drain, and does not preserve delivery routing. In per-sender or multi-channel sessions, the recovered message can sit until another turn and the eventual reply can follow the wrong trigger instead of the original sender/thread.
  - confidence: 0.88
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:94-98`
  - body: The persistence loop serializes only `state.items`, but `FollowupQueueState` also carries `droppedCount` and `summaryLines` for capped queues using `dropPolicy: summarize`. Restarting while a queue has summarized overflow will still discard that older user content instead of replaying the same summary the live drain would have produced.
  - confidence: 0.83
- **[P2] Remove the committed runtime log:** `openclaw-2026-03-18.log:1`
  - body: This PR adds `openclaw-2026-03-18.log`, which is local debug/runtime output with embedded tool-call metadata and is unrelated to the gateway replay fix. It should be removed from the branch rather than shipped in source or release artifacts.
  - confidence: 0.96
