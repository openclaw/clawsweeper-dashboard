## Summary

Keep open: current main still lacks followup/drain-window persistence, so the underlying bug is real, but the PR is not merge-ready because recovered user messages are replayed as trusted system events and bypass the normal queue, routing, wakeup, and durability semantics.

## What This Changes

The PR adds disk persistence and startup system-event replay for pending followup queues and drain-window rejected inbound messages, plus an unrelated Control UI `/model list/status` fix and Vitest include update.

## Best Possible Solution

Implement durable structured replay through the existing followup and inbound dispatch paths, with explicit wake/routing context, untrusted input handling, atomic and ack-safe retention, and a separate focused PR for the `/model` UI fix.

## Reproduction Assessment

Yes, source-level. Current main stores followup queues only in a process-global map and exposes `GatewayDrainingError` during gateway drain, while no persistence or replay path exists for either pending queue items or drain-rejected inbound messages.

## Solution Assessment

No. The PR's system-event replay changes trust, routing, wakeup, batching, and durability semantics; replaying structured work through the normal queue/dispatch machinery is the narrower maintainable fix.

Full review comments:

- **[P1] Replay recovered work through the normal reply path:** `src/gateway/replay-persisted-followups.ts:35`
  - body: Recovered followups and drain rejects are flattened into `enqueueSystemEvent()` instead of re-entering the followup/inbound dispatch paths. System events are capped, batch into a later `System:` block, carry no delivery context here, and do not start the target turn, so restart recovery can still leave replies idle or route them with unrelated traffic.
  - confidence: 0.93
- **[P1] Mark replayed user text untrusted:** `src/gateway/replay-persisted-followups.ts:35`
  - body: `enqueueSystemEvent()` defaults entries to trusted unless `{ trusted: false }` is passed. The added prose says the replayed text is user-originated, but it still drains as trusted `System:` content, so a user message received during restart can gain system-prompt authority.
  - confidence: 0.96
- **[P2] Persist enough context to redispatch drain rejects:** `src/auto-reply/dispatch.ts:41-56`
  - body: The drain-window serializer keeps only `Body` and a few routing fields. Normal finalized contexts can include `BodyForAgent`, command authorization/body fields, media fields, and other delivery metadata, so media-only or command turns become unrecoverable plain text after restart.
  - confidence: 0.88
- **[P2] Keep replay files until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` deletes the file before replay has actually been processed, and the replay then lives only in the in-memory system-event queue. A second restart before the next turn drains those events loses the recovered messages permanently.
  - confidence: 0.9
- **[P2] Write pending followups atomically:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly, while parse failures on boot delete the file. A termination during truncate/write can therefore turn a recoverable pending queue into a corrupt file that the next startup removes.
  - confidence: 0.86
