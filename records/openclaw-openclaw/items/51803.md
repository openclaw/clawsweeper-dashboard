## Summary

Keep open. Current main still has the underlying restart data-loss bug, but the latest PR head is not merge-ready: it replays persisted user messages through ephemeral system events instead of the normal followup/inbound dispatch paths, still treats replayed content as trusted system context, consumes persisted files before successful processing, loses rich inbound context, and commits a runtime log artifact.

## What This Changes

This PR adds disk persistence and startup replay for pending followup queues and drain-window rejected inbound messages, adds queue persistence tests, adjusts Control UI `/model list` and `/model status` handling, updates the Vitest UI test include list, and adds a runtime log file.

## Best Possible Solution

Replace or substantially rework the branch around a canonical fix for #51620: persist structured `FollowupRun` and inbound contexts, replay through the normal followup or inbound dispatch machinery, explicitly wake and route the target session/channel, keep persisted data durable until processing or acknowledgement succeeds, remove the runtime log artifact, and split the unrelated `/model` fix into its own focused change.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main stores followup queues in process memory, shutdown clears runtime state, and drain-window dispatch rejects with `GatewayDrainingError`; the PR head can be followed from persisted item to `enqueueSystemEvent`, which only drains on a later turn as `System:` prompt context. No live restart test was run under the read-only review constraint.

## Solution Assessment

No. The PR's system-event replay is not the best way to solve the data-loss bug because it changes dispatch, routing, trust, wakeup, and durability semantics; the safer path is structured persistence plus replay through the existing followup/inbound delivery paths.
