## Summary

Keep open, but this PR is not merge-ready: the linked restart message-loss bug still exists on current main, while this branch replays recovered user work through ephemeral system events, is conflict-marked against current main, lacks real restart/channel proof, and includes a committed runtime log artifact.

## What This Changes

This PR adds followup-queue and drain-rejected message persistence with startup replay, wires it into gateway shutdown/startup, adds persistence tests, changes Control UI `/model list/status` parsing, and adds a root runtime log file.

## Best Possible Solution

Land one maintainer-approved durable recovery path that persists accepted followups and drain-window arrivals as typed untrusted work, replays through normal queue or inbound dispatch semantics with original routing/order/idempotency, and removes unrelated artifacts.

## Reproduction Assessment

Yes, source-level: current main stores followup queues in process memory and rejects drain-window enqueues with GatewayDrainingError without a durable replay path. I did not run a live channel restart reproduction in this read-only review.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR's system-event replay is not the best fix because it changes routing, wakeup, batching, durability, and user-content authority semantics.

  - body: Recovered followups and drain-rejected messages are flattened into `enqueueSystemEvent()` instead of re-entering the followup or inbound dispatch path.   - confidence: 0.9

  - confidence: 0.86

  - confidence: 0.88
