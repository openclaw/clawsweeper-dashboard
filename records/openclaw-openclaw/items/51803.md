## Summary

Keep open: this PR targets a real gateway restart loss path, but the current branch replays persisted user work as system events, which bypasses normal dispatch/routing semantics, keeps replayed user text trusted as system content, drops durability before processing succeeds, and commits a runtime log artifact. It also still needs contributor real behavior proof before merge.

## What This Changes

The PR adds pending followup and drain-rejected message persistence with startup replay, plus an unrelated Control UI `/model list/status` parser change and a committed runtime log file.

## Best Possible Solution

Land one maintainer-approved durable recovery path for https://github.com/openclaw/openclaw/issues/51620 that replays through typed followup/inbound dispatch contracts, preserving trust, routing metadata, overflow summaries, and durable files until processing succeeds.

## Reproduction Assessment

Yes, at source level. Current main keeps FOLLOWUP_QUEUES in process memory and rejects new command-lane work once gateway draining is marked, so pending or drain-window messages can be lost across a gateway restart.

## Solution Assessment

No. This PR's system-event replay is not the narrow maintainable fix because it changes wakeup, routing, trust, batching, cap, and durability semantics; structured re-enqueue or redispatch through the existing contracts is safer.

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: Recovered followups and drain-window messages are flattened into `enqueueSystemEvent()` calls. That does not start the target session and bypasses the normal followup/inbound routing contracts, so recovered replies can sit idle or be delivered with unrelated later traffic.
  - confidence: 0.93
- **[P1] Preserve user trust on replay:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: These calls do not pass `trusted: false`, and current system events default to trusted before draining as `System:` prompt lines. The added prose that the text is user-originated does not change the role/trust flag, leaving a restart-window prompt-injection path.
  - confidence: 0.96
- **[P2] Persist the full drain-rejected inbound context:** `src/auto-reply/dispatch.ts:41-56`
  - body: The drain-window serializer keeps only text plus a few routing fields before replaying as a system event. Media-only turns, command authorization/body fields, structured context, and BodyForAgent versus envelope distinctions cannot be faithfully recovered after restart.
  - confidence: 0.88
- **[P2] Keep replay data durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` deletes the pending file before replay has actually processed the items, then replay stores them only in the in-memory system-event queue. A second restart before a turn drains those events permanently loses the recovered messages.
  - confidence: 0.9
- **[P2] Write pending followups with atomic replacement:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly, while parse failures on boot delete the file. If the process dies during truncate/write, the next boot can unlink the partial file and lose the queued followups this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:88-93`
  - body: This loop serializes only `state.items`, but capped queues can already have dropped older messages into `droppedCount` and `summaryLines`. Restart persistence would keep only tail items and discard the summarized overflow content.
  - confidence: 0.84
- **[P2] Remove the committed runtime log:** `openclaw-2026-03-18.log:1`
  - body: `openclaw-2026-03-18.log` contains local runtime/debug and tool-call metadata unrelated to the fix. Keeping it in source adds avoidable information disclosure and release-artifact noise.
