## Summary

Keep open. The PR targets a real gateway restart data-loss path, but the current patch is not mergeable because it converts recovered user messages into ephemeral system events, which changes delivery, wakeup, trust, and durability semantics.

## What This Changes

The branch adds disk persistence and startup replay for followup queues and drain-window rejected inbound messages, plus an unrelated Control UI `/model list/status` fix, Vitest include update, and a committed runtime log artifact.

## Best Possible Solution

Rework the gateway recovery as durable structured replay through the existing followup and inbound dispatch paths, coordinate it with #51620 and #51623, remove the log artifact, and split unrelated Control UI work into its own focused change if it is still needed.

## Reproduction Assessment

Yes, by source inspection. Current main keeps `FOLLOWUP_QUEUES` in memory and rejects new lane enqueues after `markGatewayDraining()`, while the PR replay path can be traced to system-event injection instead of normal queue or inbound dispatch.

## Solution Assessment

No. System-event replay is not the best way to solve this because it changes routing, wakeup, trust, batching, cap, and durability behavior; structured re-enqueue or redispatch is the safer maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Replay recovered items through the normal reply path:** `src/gateway/replay-persisted-followups.ts:33`
  - body: Recovered messages are flattened into `enqueueSystemEvent()` entries, so they do not wake the target session, do not run the normal followup drain, and do not preserve origin reply routing. A restart can still leave recovered replies idle or delivered with unrelated later traffic.
  - confidence: 0.93
- **[P1] Keep replayed user text untrusted:** `src/gateway/replay-persisted-followups.ts:33`
  - body: This call does not pass `trusted: false`; current system events default to trusted and later drain as `System:` lines. A user message received during a restart window can therefore gain system-context authority after reboot.
  - confidence: 0.96
- **[P2] Persist the full drain-rejected inbound context:** `src/auto-reply/dispatch.ts:43-56`
  - body: The drain-window serializer keeps only text and a few routing fields before replaying as a system event. Media-only messages, command authorization/body fields, and `BodyForAgent` versus envelope distinctions cannot be faithfully recovered after restart.
  - confidence: 0.88
- **[P2] Keep replay files durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:145`
  - body: `consumePersistedFollowups()` deletes the pending file before replay has actually processed the items, then replay stores them only in the in-memory system-event queue. A second restart before any turn drains those events permanently loses the recovered messages.
  - confidence: 0.9
- **[P2] Write pending followups with atomic replacement:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly, while parse failures on boot delete the file. If the process dies during truncate/write, the next boot can unlink the partial file and lose the queued followups this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:87-90`
  - body: This loop serializes only `state.items`, but capped queues can already have dropped older messages into `droppedCount` and `summaryLines`. Restart persistence would keep only tail items and discard the summarized overflow content.
  - confidence: 0.84
- **[P2] Remove the committed runtime log:** `openclaw-2026-03-18.log:1`
  - body: `openclaw-2026-03-18.log` contains local runtime/debug and tool-call metadata unrelated to the fix. Keeping it in source leaks implementation details into clones and release artifacts.
  - confidence: 0.95
