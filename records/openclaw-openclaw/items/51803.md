## Summary

Keep open: current main still has a source-level restart message-loss path, but this PR remains incorrect because system-event replay bypasses dispatch, routing, trust, and durability semantics, and the PR still lacks real restart proof.

## What This Changes

The PR adds persisted followup and drain-window message replay across gateway restarts, persistence tests, and an unrelated Control UI `/model list/status` fix.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that replays structured followup and inbound work through typed dispatch contracts with explicit untrusted semantics, origin routing, overflow state, and recovery files retained until safe processing succeeds.

## Reproduction Assessment

Yes, at source level. Current main keeps accepted followups in process-local `FOLLOWUP_QUEUES`, rejects drain-window enqueues with `GatewayDrainingError`, and has no durable pending-message replay path.

## Solution Assessment

No. Persisting pending work is directionally right, but this PR's system-event replay is not the narrow maintainable fix because it changes role/trust, routing, wakeup, batching, cap, and durability semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Replay recovered work through dispatch contracts:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: Recovered followups and drain-window messages are flattened into `enqueueSystemEvent()` calls. That bypasses the existing followup and inbound dispatch contracts, so recovered work can sit idle, batch differently, or reply through the next unrelated surface instead of the original route.
  - confidence: 0.93
- **[P1] Preserve user trust on replay:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: The latest patch adds prose saying the text is user-originated, but still calls `enqueueSystemEvent()` without `trusted: false`. System events default to trusted and drain as `System:` lines, so restart-window user text can gain higher prompt authority than the original inbound path.
  - confidence: 0.96
- **[P2] Persist the full drain-rejected inbound context:** `src/auto-reply/dispatch.ts:43-56`
  - body: The drain-window serializer keeps only text plus a few routing fields before replaying as a system event. Media-only turns, command authorization/body fields, structured context, and `BodyForAgent` versus envelope distinctions cannot be faithfully recovered.
  - confidence: 0.88
- **[P2] Keep replay data durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` deletes the recovery file before replayed work is processed, then stores it only in the in-memory system-event queue. A second restart before the next turn drains those events loses the recovered messages.
  - confidence: 0.9
- **[P2] Write pending followups with atomic replacement:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly. If the process dies during truncate/write, the next boot can treat the partial file as malformed and discard the queued followups this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:88-93`
  - body: This loop serializes only `state.items`, but capped queues can already have dropped older messages into `droppedCount` and `summaryLines`. Restart persistence would keep only the tail items and discard the summarized overflow content.
  - confidence: 0.84
- **[P2] Remove the committed runtime log artifact:** `openclaw-2026-03-18.log:1`
  - body: `openclaw-2026-03-18.log` contains local runtime/debug and tool-call metadata unrelated to the fix. Keeping it in source adds avoidable information disclosure and release-artifact noise.
  - confidence: 0.95
