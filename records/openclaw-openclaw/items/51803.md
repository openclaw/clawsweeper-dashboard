## Summary

Keep open. Current main still lacks durable followup-queue or drain-window inbound-message recovery, and this PR remains an implementation candidate for the open data-loss bug, but the branch is not safe to merge as-is because replay still flows through ephemeral system events, drops normal routing/context semantics, and includes an unrelated debug log artifact.

## What This Changes

This PR adds pending followup and drain-rejected message persistence/replay hooks for gateway restart, tests for the new persistence helpers, an unrelated Control UI `/model list` and `/model status` fix, and a committed runtime log file.

## Best Possible Solution

Keep the PR open for maintainer review or replacement, but revise the design to persist enough structured followup/inbound context and replay through the normal followup or dispatch machinery with durable acknowledgement, targeted wakeup, routing preservation, and restart-loop safety. Split the unrelated `/model` fix, remove the log artifact, and coordinate with #51620 and #51623 before choosing the canonical branch.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: on current main, followup queues are in-memory, gateway shutdown clears runtime state without persistence, and drain-window enqueues reject with `GatewayDrainingError` without a durable save path; I did not run live gateway tests because this was a read-only review.

## Solution Assessment

No. This PR is not the best way to solve the issue because system-event replay is an ephemeral prompt-injection mechanism, not the existing durable followup/dispatch contract; the safer fix is to preserve structured queue/inbound state and replay it through the normal owner-owned delivery path.

  - body: Queueing recovered messages as system events only makes them visible on some future turn for that session key; it does not trigger the target per-sender session or use the original followup routing path.   - confidence: 0.91
- **[P2] Persist the full drain-rejected inbound context:** `src/auto-reply/dispatch.ts:41-56`
  - body: The drain-window serializer keeps only text and a few routing fields, then replays that as a system event. Media-only messages, command authorization fields, and `BodyForAgent`/envelope distinctions are lost, so rich inbound turns cannot be faithfully recovered after restart.
  - confidence: 0.88
