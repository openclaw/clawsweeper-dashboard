## Summary

Keep this PR open, but it is not merge-ready: the restart-loss bug is valid, yet the latest branch still replays recovered user work through system events, loses normal routing/dispatch semantics, has durability gaps, includes a runtime log artifact, and lacks after-fix real behavior proof.

## What This Changes

This PR adds persisted followup/drain-rejected files and startup replay for gateway restarts, plus an unrelated Control UI `/model list/status` parsing change and a committed runtime log.

## Best Possible Solution

Rework recovery as durable structured replay through the existing followup queue and inbound dispatch paths, preserving user trust, routing metadata, batching/cap semantics, and replay files until processing succeeds.

## Reproduction Assessment

Yes, by source inspection: current main keeps `FOLLOWUP_QUEUES` in process memory and rejects `enqueueCommandInLane()` after gateway draining is marked, so a restart with queued or drain-window messages has a clear loss path. I did not run a live restart flow.

## Solution Assessment

No. The current PR is not the best fix because system-event replay changes wakeup, routing, trust, durability, batching, and cap behavior; structured re-enqueue or redispatch is the narrower maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: Recovered followups and drain-window messages are flattened into `enqueueSystemEvent()` entries. That does not start the target session, bypasses the normal followup/inbound dispatch contracts, and can leave recovered replies idle or delivered with unrelated later traffic.
  - confidence: 0.93
- **[P1] Preserve user-message trust when replaying:** `src/gateway/replay-persisted-followups.ts:35-48`
  - body: These calls do not pass `trusted: false`, so persisted user text drains as trusted `System:` content. The added prose saying the text is user-originated does not change the role/trust flag, leaving a restart-window prompt-injection path.
  - confidence: 0.96
- **[P2] Persist the full drain-rejected inbound context:** `src/auto-reply/dispatch.ts:41-56`
  - body: The drain-window serializer keeps only text plus a few routing fields before replaying as a system event. Media-only turns, command authorization/body fields, structured context, and `BodyForAgent` versus envelope distinctions cannot be faithfully recovered after restart.
  - confidence: 0.88
- **[P2] Keep replay files durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` deletes the pending file before replay has actually processed the items, then replay stores them only in the in-memory system-event queue. A second restart before a turn drains those events permanently loses the recovered messages.
  - confidence: 0.9
- **[P2] Write pending followups with atomic replacement:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly, while parse failures on boot delete the file. If the process dies during truncate/write, the next boot can unlink the partial file and lose the queued followups this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:88-93`
  - body: This loop serializes only `state.items`, but capped queues can already have dropped older messages into `droppedCount` and `summaryLines`. Restart persistence would keep only tail items and discard the summarized overflow content.
  - confidence: 0.84
- **[P2] Remove the committed runtime log:** `openclaw-2026-03-18.log:1`
  - body: `openclaw-2026-03-18.log` contains local runtime/debug and tool-call metadata unrelated to the fix. Keeping it in source leaks implementation details into clones and release artifacts.
  - confidence: 0.95
