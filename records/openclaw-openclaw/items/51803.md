## Summary

Keep this PR open, but it should not merge in its current shape. The branch addresses a real gateway restart data-loss bug, yet the recovery path still replays user messages as system-event text instead of durable structured followup/inbound work, and it also commits a runtime log artifact.

## What This Changes

The PR adds persisted followup/drain-rejected message files and startup replay for gateway restarts, plus an unrelated Control UI `/model list/status` parsing change.

## Best Possible Solution

Rework recovery as durable structured replay through the existing followup queue and inbound dispatch paths, preserve untrusted user semantics and routing metadata, remove unrelated/log artifacts, and coordinate the final design with the canonical restart-loss issue.

## Reproduction Assessment

Yes, by source inspection. Current main keeps followup queues in process memory and rejects new lane enqueues while the gateway is draining, so a restart with queued or drain-window inbound work has a clear loss path even though I did not run a live restart flow.

## Solution Assessment

No. The current PR's system-event replay is not the best fix because it changes wakeup, routing, trust, batching, caps, and durability semantics; structured re-enqueue or redispatch is the narrower maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/replay-persisted-followups.ts:35`
  - body: Recovered followups and drain-window messages are flattened into `enqueueSystemEvent()` entries, so they do not start the target session, do not run the normal followup drain, and do not preserve the original reply-routing contract. A restart can still leave recovered replies idle or delivered with unrelated later traffic.
  - confidence: 0.93
- **[P1] Keep replayed user text out of trusted system events:** `src/gateway/replay-persisted-followups.ts:35`
  - body: This call does not pass `trusted: false`; current system events default to trusted and later drain as `System:` lines. Adding prose that says the content is user-originated does not change the role/trust flag, so a restart-window user message can still gain system-event authority after reboot.
  - confidence: 0.96
- **[P2] Persist the full drain-rejected inbound context:** `src/auto-reply/dispatch.ts:41-56`
  - body: The drain-window serializer keeps only text plus a few routing fields before replaying as a system event. Media-only turns, command authorization/body fields, structured context, and `BodyForAgent` versus envelope distinctions cannot be faithfully recovered after restart.
  - confidence: 0.88
- **[P2] Keep replay files durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` deletes the pending file before replay has actually processed the items, then replay stores them only in the in-memory system-event queue. A second restart before any turn drains those events permanently loses the recovered messages.
  - confidence: 0.9
- **[P2] Write pending followups with atomic replacement:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly, while parse failures on boot delete the file. If the process dies during truncate/write, the next boot can unlink the partial file and lose the queued followups this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:88-93`
  - body: This loop serializes only `state.items`, but capped queues can already have dropped older messages into `droppedCount` and `summaryLines`. Restart persistence would keep only tail items and discard the summarized overflow content.
  - confidence: 0.84
- **[P2] Replay without the system-event queue cap:** `src/gateway/replay-persisted-followups.ts:35`
  - body: Each persisted item is replayed through `enqueueSystemEvent()`, whose queue keeps only 20 entries per session. A session with more recovered items, or recovered items plus other startup events, can silently drop the oldest messages during recovery.
  - confidence: 0.82
- **[P2] Remove the committed runtime log:** `openclaw-2026-03-18.log:1`
  - body: `openclaw-2026-03-18.log` contains local runtime/debug and tool-call metadata unrelated to the fix. Keeping it in source leaks implementation details into clones and release artifacts.
  - confidence: 0.95
