## Summary

Keep open. Current main still has the restart data-loss gap, but this PR is not merge-ready because its replay path converts recovered user messages into ephemeral system events, loses normal queue/dispatch semantics, consumes persisted data before processing is durable, adds a runtime log artifact, and is stale relative to current main’s dispatch/gateway code.

## What This Changes

This PR adds disk persistence for pending followup queues and drain-window rejected messages, replays them on gateway startup through system events, adds persistence tests, changes Control UI `/model list` and `/model status` handling, and adds a runtime log file.

## Best Possible Solution

Revise or replace the branch with a canonical fix for #51620 that persists structured `FollowupRun` and inbound contexts, replays them through the normal followup or inbound dispatch machinery, explicitly wakes and routes the target session/channel, keeps persisted data durable until processing or acknowledgement succeeds, splits the unrelated `/model` fix, and removes the log artifact.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: on current main followup queues are in-memory only, gateway shutdown clears runtime state without a persistence hook, and drain-window enqueues reject with `GatewayDrainingError` without a durable save path; no live restart test was run under the read-only review constraint.

## Solution Assessment

No. The current PR direction is not the best fix because system-event replay changes user messages into prompt context instead of preserving normal queue/dispatch semantics; the safer solution is structured persistence plus normal replay/wakeup/routing with durability until completion.

- **[P1] Replay through the followup dispatcher:** `src/gateway/replay-persisted-followups.ts:35`
  - body: Recovered followups are converted to `enqueueSystemEvent()` instead of being rebuilt as queue entries and drained through the normal followup path. System events only appear on a later turn and do not wake or route the original sender/session, so a restart can still leave replies idle or deliver them with unrelated traffic.
  - confidence: 0.93
- **[P1] Preserve untrusted user role on replay:** `src/gateway/replay-persisted-followups.ts:35`
  - body: This call does not pass `trusted: false`, while current system events default to trusted and drain as `System:` lines. A queued user message can gain system-role authority after restart, which is a prompt-trust regression.
  - confidence: 0.96
- **[P2] Persist complete drain-rejected contexts:** `src/auto-reply/dispatch.ts:41-56`
  - body: The drain-window serializer keeps only prompt text and a few routing fields before replaying as a system event. Media-only messages, command authorization, and `BodyForAgent`/envelope distinctions cannot be faithfully recovered after restart.
  - confidence: 0.88
- **[P2] Keep replay data durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:148`
  - body: `consumePersistedFollowups()` unlinks the pending file before replay has actually run, and replay then stores items only in the in-memory system-event queue. A second restart before any turn drains those events permanently loses the recovered messages.
  - confidence: 0.9
- **[P2] Write pending followups with atomic replacement:** `src/auto-reply/reply/queue/persist.ts:113`
  - body: The shutdown path writes `pending-followups.json` directly, while parse failures on boot cause the file to be deleted. If the process dies during truncate/write, the next boot can unlink the partial file and lose the queued followups.
  - confidence: 0.86
