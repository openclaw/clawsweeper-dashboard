## Summary

Keep this PR open for maintainer decision, but do not merge it as-is: current main routes late text+voice media through the lane-delivery path, while this patch adds handling in a fallback path that does not cover the stated TTS case and also conflicts with the current separate text + voice delivery direction.

## What This Changes

This PR attempts to delete a previously streamed Telegram text message and resend the final audioAsVoice payload with that text as the voice caption.

## Best Possible Solution

Decide the canonical Telegram TTS text/audio policy with https://github.com/openclaw/openclaw/issues/42539, then implement it in the lane-delivery path with focused tests for streamed text, voice fallback, buttons, and delete failures.

## Reproduction Assessment

Yes. Source and tests show current main keeps streamed Telegram text in place when late media arrives and sends voice media as a separate supplement; I did not run a fresh live Telegram session in this read-only review.

## Solution Assessment

No. The submitted patch is not the best fix because it targets the fallback send path rather than the current media lane, and the desired combined-caption behavior needs a maintainer product decision against the separate-delivery issue.
