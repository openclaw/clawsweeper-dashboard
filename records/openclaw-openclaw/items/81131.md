## Summary

Keep open, but do not merge as-is. The PR has useful Telegram screenshot proof for the desired happy path, but the patch conflicts with current released `spokenText`/TTS supplement behavior and can delete the already-visible streamed text before the replacement voice send succeeds.

## What This Changes

The PR changes Telegram final reply dispatch so a streamed text preview followed by an `audioAsVoice` final payload is deleted and resent as one voice note with a caption.

## Best Possible Solution

Resolve this as a product-approved Telegram TTS delivery contract: keep the current separate/fallback behavior by default, add a tested opt-in bundled-caption mode only if maintainers want it, and never delete a streamed preview until replacement delivery succeeds.

## Reproduction Assessment

Yes from source inspection: the PR sets `payload.text` from `spokenText` and starts deleting the streamed Telegram message before `sendPayload` succeeds. I did not run live Telegram because the source path and current released tests are enough to identify the merge blocker.

## Solution Assessment

No. Unconditional captioning and early deletion are not the safest fix; the maintainable path is a maintainer-approved delivery mode with success-gated cleanup and regression coverage for failure behavior.
