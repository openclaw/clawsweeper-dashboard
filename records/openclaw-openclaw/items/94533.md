## Label Justifications

- `P1`: A scheduled or isolated agent run can produce useful output but suppress delivery, breaking an active agent/channel workflow for real users.
- `impact:message-loss`: The reported failure mode drops or withholds produced cron and inter-agent reply messages at the delivery layer.
- `impact:session-state`: The bug couples terminal turn/run status to delivery state, causing completed work to be recorded as an error/not-delivered session outcome.

## Summary

Keep open: current main still lets a failed Codex turn or run-level/fatal signal make an isolated cron run end as `error`, and cron delivery-state persistence still records error runs as `not-delivered`, so the reported produced-output drop is not fixed by current source or by the still-open related PRs.

## What This Changes

The issue requests that cron/isolated-agent delivery preserve produced assistant output when an ancillary or background tool failure marks the final turn or run as failed.

## Best Possible Solution

Define one terminal outcome contract that separates deliverable assistant output from ancillary/background diagnostics, then update Codex projection, cron outcome classification, delivery persistence, and announce/inter-agent regression tests together.

## Reproduction Assessment

Yes from source inspection: current main still gates deliverable assistant output on completed turn status and persists error cron runs as not delivered, though I did not run a live cron reproduction.

## Solution Assessment

Unclear; the proposed per-job option should not be the first fix without a maintainer decision because core likely needs a single status/delivery contract that distinguishes fatal failures from ancillary diagnostics.
