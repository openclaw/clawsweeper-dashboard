## Label Justifications

- `P1`: The issue reports real scheduled and inter-agent workflows silently suppressing produced output, which is an urgent delivery regression for affected operators.
- `impact:message-loss`: The reported failure drops a produced cron deliverable or inter-agent reply before it reaches the intended recipient.
- `impact:session-state`: The persisted cron run state can record `error` and `not-delivered` even when the isolated session produced useful final output.

## Summary

Keep open: current main still lets cron finalization classify run-level or fatal tool/background failures as `error` despite final assistant output, and error cron status still records requested delivery as `not-delivered`; overlapping open PRs are not merged or safe canonical fixes yet.

## What This Changes

The issue asks OpenClaw to deliver produced cron or isolated-agent output even when only an ancillary background tool exits non-zero.

## Best Possible Solution

Define one cron terminal-outcome contract that delivers valid final assistant payloads after proven ancillary/background recovery while preserving true fatal run failures, denials, and explicit fail-closed behavior, then land one coordinated fix with cron and inter-agent regression proof.

## Reproduction Assessment

Yes, source-level: current main still routes run-level or fatal failure metadata through the cron outcome classifier into `status: error`, and error status forces requested primary delivery to `not-delivered`. I did not run a live scheduled Telegram cron with a killed background process.

## Solution Assessment

Unclear: the requested direction is valid, but a per-job option is not the best first step. Maintainers should settle the shared cron status/delivery contract and then choose one narrow implementation path rather than stacking overlapping PRs.
