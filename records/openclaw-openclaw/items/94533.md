## Label Justifications

- `P1`: The issue reports silently dropped scheduled and inter-agent deliverables in real workflows, which is an urgent agent/channel delivery regression for affected operators.
- `impact:message-loss`: The central symptom is produced Telegram or inter-agent output being withheld by the delivery gate.
- `impact:session-state`: Cron state can persist `error` and `not-delivered` despite useful assistant output being produced, creating run/session state drift.

## Summary

Keep open: current main and the latest release still map cron run-level or fatal failure-signal state to `error`, and persisted delivery state can still become `not-delivered` even when lower layers report delivery output; the exact ancillary background-process case still needs live maintainer repro and a terminal-status contract decision.

## What This Changes

The issue asks OpenClaw to deliver valid cron or isolated-agent output even when only an ancillary backgrounded tool exits non-zero, instead of marking the run failed and suppressing delivery.

## Best Possible Solution

Define one maintainer-approved isolated cron terminal-outcome contract that delivers valid final assistant payloads when only ancillary diagnostics failed, preserves true fatal failures, and applies consistently to cron announce and inter-agent reply delivery.

## Reproduction Assessment

Yes at source level, but not as a fresh live repro. Current main still routes run-level/failure-signal state through cron `error` status and `not-delivered` persistence, while the exact backgrounded-process scenario still needs live cron/Telegram or equivalent proof.

## Solution Assessment

Unclear as an implementation direction. The issue is a real bug, but the best fix is a maintainer-owned terminal-status and delivery contract rather than simply adding a per-job option or changing one classifier branch.
