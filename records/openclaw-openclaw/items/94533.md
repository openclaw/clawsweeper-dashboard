## Summary

Keep open: current main and the latest release still contain the cron outcome and delivery-state coupling that can suppress a produced deliverable after a fatal/run-level failure signal. The newer canonical issue covers a narrower recovered-tool-warning subset, but this issue also includes the ancillary background-process/final-turn semantics and an inter-agent reply facet that still need maintainer policy review.

## What This Changes

The issue asks OpenClaw to deliver valid cron or inter-agent assistant output even when an ancillary background/tool failure should only be diagnostic.

## Best Possible Solution

Land one reviewed narrow outcome contract that delivers valid final assistant payloads after recovered ancillary/background tool failures while preserving true fatal run failures, denials, and explicit fail-closed behavior.

## Reproduction Assessment

Yes, source-level: current main still feeds run-level and fatal error signals into cron outcome classification, then records requested delivery as not-delivered when the run status is error. I did not run a fresh live cron or inter-agent reproduction in this read-only review.

## Solution Assessment

Unclear as a final product shape: a per-job option is not the best first step, while a narrow core classifier/status contract is likely the maintainable fix. Maintainers need to decide which background and recovered tool failures are diagnostic rather than fatal before implementation is queued.
