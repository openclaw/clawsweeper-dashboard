## Label Justifications

- `P1`: The issue reports silently dropped cron and inter-agent deliverables for real scheduled workflows, which is an urgent delivery regression for affected operators.
- `impact:message-loss`: The core symptom is a produced Telegram or inter-agent reply being suppressed at the delivery layer.
- `impact:session-state`: The persisted cron state records an error and not-delivered result even when useful assistant output was produced.

## Summary

Keep open: current main still turns run-level errors or fatal failure signals into `error` cron outcomes and then records requested delivery as not delivered, so the reported produced-output/background-process failure path is not fixed by the existing recovered-tool-warning handling.

## What This Changes

The issue asks OpenClaw to deliver valid cron or inter-agent output when only an ancillary background process failed, instead of letting that diagnostic failure suppress the produced deliverable.

## Best Possible Solution

Define a structured terminal-outcome contract that preserves delivery of valid final assistant payloads when only ancillary background diagnostics failed, while keeping true fatal run failures, denials, and explicit fail-closed contracts fatal.

## Reproduction Assessment

Yes, source-level: current main still propagates run-level errors through the cron outcome classifier into an error status that the timer records as not delivered; I did not run a fresh live cron reproduction.

## Solution Assessment

Unclear: the per-job option is not the best first step; maintainers should first settle the core classifier/status contract for ancillary background failures versus true fatal failures and reconcile the open related PRs.
