## Label Justifications

- `P1`: The issue reports silently suppressed scheduled and inter-agent deliverables in real automation workflows.
- `impact:message-loss`: The central symptom is produced Telegram announce output or inter-agent reply output being gated away.
- `impact:session-state`: Cron run state can persist `error` and `not-delivered` despite useful assistant output and completed underlying work.

## Summary

Keep open: current main still couples cron/isolated-agent fatal turn state to delivery persistence, so a produced deliverable can still be recorded as error/not-delivered, and no merged PR owns the broader background-process plus inter-agent reply contract.

## What This Changes

The issue asks OpenClaw to deliver produced cron or inter-agent output even when an ancillary background process exits non-zero instead of letting that failure suppress delivery.

## Best Possible Solution

Define one cron and isolated-agent delivery contract that delivers valid final output after non-terminal ancillary failures while preserving true fatal errors and inter-agent reply semantics.

## Reproduction Assessment

Yes at source level, but not yet live: current main maps fatal run-level/classifier state to `error` and then to `not-delivered` even when final output or delivery evidence exists.

## Solution Assessment

Unclear as an implementation direction: the resolver/finalization path is implicated, but maintainers need to define typed non-terminal versus fatal signals rather than blindly letting final text win.
