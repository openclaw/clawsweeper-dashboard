## Summary

Keep open: current main still has a source-plausible path where a failed Codex turn with assistant text becomes a prompt/run error, and cron error state records announce delivery as not delivered. The issue is not superseded by the broader cron fail-closed or subagent-loss reports because this exact ancillary background-process failure mode remains unowned.

## What This Changes

Ensure cron/isolated announce delivery still sends a produced assistant deliverable when only an ancillary background process exits non-zero, without treating that process warning as the final turn failure.

## Best Possible Solution

Add a typed nonfatal diagnostic path for ancillary background-process failures so cron can deliver produced assistant payloads while preserving fatal behavior for material tool and run failures.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction shape: a Codex turn with assistant text plus `turn.status === "failed"` and a Process error becomes prompt/run failure, then cron error state records the announce as not delivered. I did not run a live Telegram cron reproduction in this read-only review.

## Solution Assessment

Unclear as proposed because adding a per-job option or delivering all failed-turn assistant text would be too broad. The best fix should distinguish nonterminal background-process diagnostics from material tool failures and keep the existing fail-closed safety path intact.
