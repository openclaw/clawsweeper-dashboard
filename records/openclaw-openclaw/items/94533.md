## Summary

Keep open: current main still lets run-level or fatal cron failure state override produced assistant output, and the cron service still persists requested delivery as not delivered when the final run status is error. The issue is recent, source-reproducible, and overlaps open cron-status policy work rather than being fixed or safely closeable.

## What This Changes

The issue asks OpenClaw to deliver a produced cron or inter-agent reply when only an ancillary background tool failure made the final run status non-completed.

## Best Possible Solution

Define a structured terminal-outcome contract that delivers valid final assistant payloads when only ancillary background diagnostics failed, while preserving true fatal run failures, denials, and explicit fail-closed contracts as fatal.

## Reproduction Assessment

Yes, source-level: current main still propagates run-level errors or fatal failure signals through the cron outcome classifier into error status, and the timer records requested error runs as not-delivered. I did not run a fresh live cron reproduction of the exact backgrounded process failure.

## Solution Assessment

Unclear: decoupling valid final output from ancillary diagnostics is the right direction, but adding a per-job option is not the best first step. Maintainers should settle the core terminal-status contract and reconcile the open related PRs before implementation.
