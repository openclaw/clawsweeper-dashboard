## Label Justifications

- `P1`: A scheduled agent/channel workflow can silently withhold produced cron or inter-agent output from the recipient, which is an urgent delivery regression class.
- `impact:message-loss`: The reported failure is about produced announce/reply content not reaching Telegram or the requesting agent.
- `impact:session-state`: The run's terminal status and delivery state diverge from the produced assistant output, leaving cron/session records in an error/not-delivered state.

## Summary

Keep open: current main still has the relevant cron coupling where run-level agent errors are treated as fatal and an error run cannot retain primary delivery as delivered, while the exact ancillary-background-exec trigger and inter-agent facet still need maintainer/live-repro confirmation before a safe fix shape is chosen.

## What This Changes

The issue reports that an isolated cron `agentTurn` can produce assistant output but lose announce/reply delivery when a non-final backgrounded exec-like process exits non-zero and flips the final run status to error.

## Best Possible Solution

Define one cron/isolated-agent terminal-outcome invariant: deliver produced final assistant output when only ancillary non-final background work failed, while recording that failure as diagnostics or a warning and applying the same rule to announce and inter-agent reply delivery.

## Reproduction Assessment

No, not at high confidence: the source shows the fatal run-level-error to not-delivered path, but I did not live-reproduce the exact ancillary backgrounded exec failure against current main.

## Solution Assessment

Unclear: the desired direction is likely to decouple deliverable output from non-final background failures, but maintainers need to choose the terminal-status semantics so real provider failures, denied execution, and ancillary warnings stay distinguishable.
