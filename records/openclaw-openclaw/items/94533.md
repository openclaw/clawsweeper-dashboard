## Label Justifications

- `P1`: The issue reports silently suppressed cron and inter-agent deliverables in real scheduled workflows, which is an urgent delivery regression for affected operators.
- `impact:message-loss`: The central failure is a produced Telegram or inter-agent reply being withheld at the delivery layer.

## Summary

Keep open: current main and the latest release still let run-level or fatal failure signals make an isolated cron run `error`, and cron persistence then records requested delivery as `not-delivered`. The open related PRs overlap the cron status policy but do not safely supersede this report.

## What This Changes

The issue asks OpenClaw to deliver produced cron and inter-agent outputs when only ancillary background tool/process failures occurred.

## Best Possible Solution

Define a structured cron terminal-outcome contract that still delivers valid final assistant payloads when only ancillary background diagnostics failed, while keeping true fatal run failures, denials, and explicit fail-closed contracts fatal.

## Reproduction Assessment

Yes, source-level. Current main still maps run-level errors and fatal failure signals to cron `error`, then records requested delivery as `not-delivered`; I did not run the live background-process cron in this read-only review.

## Solution Assessment

Unclear. Decoupling delivery from ancillary background failures is directionally right, but adding a per-job option should wait until maintainers define the core terminal-status semantics and reconcile the open cron-status PRs.
