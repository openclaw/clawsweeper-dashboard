## Summary

This is a valid focused exec-policy bug fix, but the PR still needs changes before merge because the new tests can touch real exec approval state and the model-facing `security` argument remains advertised as meaningful after the runtime starts ignoring it.

## What This Changes

This PR makes normal exec calls use configured `tools.exec.security` instead of model-supplied `security` arguments, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep configured exec security authoritative, isolate the regression tests' state directory, and align the model-facing schema/docs here or let the broader open schema-boundary PR own that cleanup.

## Reproduction Assessment

Yes. Source inspection of current main shows `params.security` flows through `normalizeExecSecurity()` and `minSecurity()`, so configured `full` plus model `allowlist` can become allowlist and hit the gateway allowlist-miss path; I did not run live commands in this read-only review.

## Solution Assessment

No for the PR as submitted. Using configured security is the narrow runtime fix, but the branch should isolate exec approval state in tests and update or remove the advertised normal `security` argument semantics before merge.
