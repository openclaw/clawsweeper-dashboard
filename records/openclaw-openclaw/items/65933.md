## Summary

Keep open: the runtime direction and after-fix proof are good, but the PR still introduces a real-state test side effect and leaves the model-facing exec `security` contract inconsistent with the new behavior.

## What This Changes

The PR makes normal exec calls ignore model-supplied `security` arguments, adds security-floor regression tests, and adds a changelog entry.

## Best Possible Solution

Keep configured exec security authoritative, isolate the regression tests from real approval state, and align the schema/docs here or reconcile that part with https://github.com/openclaw/openclaw/pull/76370.

## Reproduction Assessment

Yes. Source inspection of current main shows `params.security` flows through `normalizeExecSecurity()` and `minSecurity()`, so configured `full` plus model `allowlist` can become allowlist and reach the allowlist-miss path; I did not run live commands in this read-only review.

## Solution Assessment

No as submitted. Using configured security as the normal-call source is the narrow runtime fix, but the branch should isolate exec approval state in tests and update the advertised schema/docs semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Isolate exec approval state in the new tests:** `src/agents/bash-tools.exec.security-floor.test.ts:10`
  - body: These tests call `tool.execute`, which reaches `resolveExecApprovals()` and `ensureExecApprovals()`, saving `~/.openclaw/exec-approvals.json`. Since the setup only captures `SHELL`, running the test can create or rewrite a developer's real approval file; redirect HOME/USERPROFILE/OPENCLAW_STATE_DIR to a temp directory and restore them after each test.
  - confidence: 0.88

  - body: This change makes normal exec calls ignore `params.security`, but the tool schema and exec docs still present `security` as a per-call enforcement mode. Update or remove that model-visible wording, or fold this PR into the broader schema cleanup, so agents and users do not rely on a no-op policy override.
  - confidence: 0.76
