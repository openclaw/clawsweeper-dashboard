## Summary

Keep open: current main has a source-level path where a Codex app-server failed turn can propagate raw error text through `promptError` to the WebChat error surface, and duplicate searches found no canonical open item.

## What This Changes

The issue requests an actionable diagnostic when a run aborts with only `[max turns exceeded]`.

## Best Possible Solution

Add a narrow max-turn diagnostic formatter at the Codex harness or embedded-runner error boundary, with regression coverage for failed-turn and thrown-prompt-error paths and no new config option.

## Reproduction Assessment

Yes, source-level: a Codex app-server failed `turn/completed` notification with `error.message: "[max turns exceeded]"` would become `promptError`, then the embedded runner/WebChat path can surface `String(err)` without added context. I did not run a live Codex app-server scenario in this read-only review.

## Solution Assessment

No proposed patch is under review. The best repair is not a generic catch-all rewrite; it should normalize this known max-turn failure close to the Codex failed-turn or embedded-runner prompt-error boundary while preserving raw details in metadata/logs.
