## Summary

Keep open: current main and the latest shipped tag still queue short-term recall tracking from normal memory search even though the docs describe `dreaming.enabled` as the master dreaming switch. The fix looks narrow: gate the tool and CLI recall-tracking side effects on the resolved dreaming config while leaving ordinary search results untouched.

## What This Changes

The issue asks OpenClaw to stop writing `memory/.dreams` recall artifacts from normal memory search when `plugins.entries.memory-core.config.dreaming.enabled` is false.

## Best Possible Solution

Gate ordinary memory tool and CLI recall-tracking writes on the resolved `dreaming.enabled` value, while preserving normal search results and preserving recall tracking when dreaming is enabled.

## Reproduction Assessment

Yes, by source inspection: current main calls `recordShortTermRecalls(...)` from both the tool and CLI memory search paths without checking `dreaming.enabled`, and that helper writes the reported `.dreams` files. I did not run a live filesystem repro because this review is read-only.

## Solution Assessment

Yes, the best fix is a narrow call-site guard using the existing resolved dreaming config, plus tests that search still returns results and recall artifacts are not queued when dreaming is disabled.
