## Summary

Keep open: current main and the latest shipped tag still route ordinary memory search through short-term recall tracking even when `dreaming.enabled` is false, and the issue is now paired with open closing PRs that should be reviewed or superseded before closing. The strongest current fix path is https://github.com/openclaw/openclaw/pull/84444; https://github.com/openclaw/openclaw/pull/84445 is a second open attempt for the same issue.

## What This Changes

The issue asks OpenClaw to stop writing `.dreams` recall/event artifacts during ordinary memory search when `plugins.entries.memory-core.config.dreaming.enabled=false` while preserving normal memory search behavior.

## Best Possible Solution

Land one focused fix that gates tool and CLI recall-tracking writes on the resolved dreaming config, keeps ordinary search results unchanged, covers both disabled and enabled dreaming paths, and then closes this issue after merge.

## Reproduction Assessment

Yes, by source inspection. Current main calls `recordShortTermRecalls(...)` from both ordinary memory search paths without checking `dreaming.enabled`, and that helper writes the reported `.dreams` artifacts.

## Solution Assessment

Yes. The narrow maintainable fix is to gate only the recall-tracking side effect on the resolved dreaming switch while leaving normal memory search, `memory_get`, and transcript recall behavior intact.
