## Summary

Keep open: current main and the latest shipped tag still call the recall-tracking writer from normal memory search without checking `dreaming.enabled`, and two open closing PRs are already associated, so this issue should stay open until one fix lands.

## What This Changes

The issue requests that memory-core keep ordinary memory search working but skip `.dreams` recall and promotion artifact writes when `dreaming.enabled=false`.

## Best Possible Solution

Review and land one focused fix that gates only the tool and CLI recall-tracking side effects on the resolved dreaming config, with regression coverage for disabled and enabled dreaming paths.

## Reproduction Assessment

Yes, at source level. Current main calls `recordShortTermRecalls(...)` from both normal memory search paths without checking `dreaming.enabled`, and that helper writes the reported `.dreams` artifacts.

## Solution Assessment

Yes. The narrow maintainable fix is to gate the implicated recall-tracking calls on the resolved dreaming switch while preserving ordinary search results; the linked PRs are candidate implementations of that shape.
