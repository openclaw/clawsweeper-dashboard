## Summary

Keep open: current main still has a source-reproducible Telegram isolated-ingress path where a bot without `botInfo` drains spooled updates through `handleUpdate` before grammY has been initialized, matching the reporter's `Bot not initialized` logs.

## What This Changes

The issue asks OpenClaw to initialize Telegram polling bots before isolated-ingress spool draining calls `handleUpdate`.

## Best Possible Solution

Initialize the polling bot before isolated-ingress spool draining, use the existing recoverable setup-error retry behavior, and add a regression that proves an uninitialized bot drains successfully after `init()`.

## Reproduction Assessment

Yes, at source level: with isolated ingress enabled and `botInfo` absent, current main calls `handleUpdate` before any `bot.init()` path runs. I did not run live Telegram traffic in this read-only review.

## Solution Assessment

Yes, the right fix is narrow: call `await bot.init()` before isolated-ingress drain/worker processing, or guard with `bot.isInited()`, and use the existing recoverable setup retry path. Do not guard by reading `bot.botInfo`, because grammY's getter throws when the bot is uninitialized.
