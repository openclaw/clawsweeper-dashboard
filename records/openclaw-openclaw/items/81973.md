## Summary

Keep open. Current main still has the reported source path: isolated ingress is enabled by default, the polling bot is usually constructed without `botInfo`, and isolated spool drain calls `handleUpdate` without going through `run()` or `bot.init()` first; exact related searches did not find a separate canonical fix.

## What This Changes

The issue reports that Telegram isolated polling ingress drains spooled updates with `bot.handleUpdate(update)` before the grammY bot is initialized, causing repeated `Bot not initialized` failures.

## Best Possible Solution

Initialize the Telegram polling bot before isolated spool drain starts, using the existing recoverable setup retry path, and add regression coverage that fails when `handleUpdate()` runs before `init()`.

## Reproduction Assessment

Yes from source inspection, though not from a fresh live gateway run. Current main's isolated path starts spool drain without `bot.init()`, and grammY 1.42.0 throws from `handleUpdate()` until `init()` or constructor `botInfo` sets bot info.

## Solution Assessment

Yes, initializing before isolated drain is the narrow maintainable fix. Avoid the suggested `if (!bot.botInfo)` guard because grammY's `botInfo` getter itself throws before initialization; call idempotent `init()` or use `isInited()` if guarded.
