## Summary

Keep open: the Windows lock-contention crash is source-reproducible, but this PR no longer fixes the active code path because current main delegates session locking to fs-safe, and the external PR still lacks real Windows behavior proof.

## What This Changes

The PR adds Windows-only EPERM retry handling and mocked regression tests to the inline session write-lock acquisition loop.

## Best Possible Solution

Handle Windows lock-contention EPERM at the active fs-safe/file-lock boundary, preserve fast failure for real permission errors, add focused regression coverage, and collect redacted Windows runtime proof.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates session lock acquisition to SESSION_LOCKS.acquire, and @openclaw/fs-safe@0.2.4 rethrows EPERM because only EEXIST enters the retry path.

## Solution Assessment

No, not as-is. The retry direction is reasonable, but this branch patches an inactive inline fs.open path; the maintainable fix belongs in fs-safe or OpenClaw's file-lock wrapper.
