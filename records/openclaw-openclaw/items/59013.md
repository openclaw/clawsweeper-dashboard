## Summary

Keep open: the Windows EPERM crash is still source-reproducible on current main, but this PR should not merge as-is because it patches the old inline lock loop while current main delegates acquisition to the fs-safe lock manager. The branch also still lacks after-fix Windows behavior proof from a real setup.

## What This Changes

The branch adds Windows-specific EPERM retry handling to the legacy inline session write-lock loop and adds mocked EPERM regression tests.

## Best Possible Solution

Handle Windows lock-contention EPERM at the active fs-safe/file-lock boundary, preserve fast failure for real permission errors, and add focused regression coverage plus redacted Windows behavior proof.

## Reproduction Assessment

Yes, source-reproducible. Current main calls `SESSION_LOCKS.acquire`, and pinned `@openclaw/fs-safe@0.2.4` rethrows non-`EEXIST` exclusive-create errors, so a Windows EPERM would bubble out of session lock acquisition.

## Solution Assessment

No, not as-is. The retry direction is valid, but the PR patches obsolete inline code; the maintainable fix belongs at the active fs-safe/file-lock boundary with real Windows proof.
