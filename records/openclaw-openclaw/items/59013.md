## Summary

Keep open: the Windows `EPERM` bug is plausible and source-reproducible, but this PR now patches an obsolete inline lock loop while current main delegates acquisition to `@openclaw/fs-safe`, and the PR still lacks real Windows after-fix proof.

## What This Changes

The PR modifies the older direct session write-lock `fs.open("wx")` acquisition loop to retry Windows `EPERM` contention and adds mocked EPERM regression tests.

## Best Possible Solution

Implement Windows-only `EPERM` contention handling at the active fs-safe/file-lock boundary used by `SESSION_LOCKS`, preserve fast failure for real permission errors, add focused regression coverage, and require redacted Windows proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates to `SESSION_LOCKS.acquire`, `@openclaw/fs-safe@0.2.2` rethrows non-`EEXIST` errors from `fs.open(lockPath, "wx")`, and `acquireSessionWriteLock` rethrows non-file-lock errors, so a mocked Windows `EPERM` would still bubble out.

## Solution Assessment

No for this PR as-is. The Windows-only retry direction is narrow, but the implementation needs to move to the current fs-safe-backed lock boundary and be proven on real Windows.
