## Summary

Keep open: the Windows EPERM crash is source-reproducible, but this patch targets the old inline lock loop while current main delegates acquisition to fs-safe, so the proposed change would not fix the active path after rebase.

## What This Changes

The PR adds Windows EPERM retry handling to the legacy inline session write-lock `fs.open("wx")` loop and adds mocked tests.

## Best Possible Solution

Handle Windows lock-contention EPERM at the active fs-safe sidecar/file-lock boundary, preserving fast failure for real permission errors and adding focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates to fs-safe, and fs-safe 0.2.4 rethrows non-EEXIST `fs.open("wx")` errors, so a Windows EPERM from that path would bubble out of `acquireSessionWriteLock`.

## Solution Assessment

No, not as-is. The retry direction is reasonable, but the patch needs to move to the active fs-safe/file-lock path and include real Windows proof.
