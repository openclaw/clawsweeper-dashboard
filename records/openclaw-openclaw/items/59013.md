## What This Changes

The branch adds Windows EPERM retry handling and mocked regression tests around the session write-lock `fs.open("wx")` loop.

## Best Possible Solution

Handle Windows lock-contention `EPERM` at the active fs-safe/file-lock boundary or a narrow OpenClaw wrapper around it, preserve fast failure for real permission errors, add focused regression coverage, and collect redacted Windows runtime proof.

## Reproduction Assessment

Yes, source-reproducible but not live-verified: if fs-safe receives `EPERM` from its `fs.open(lockPath, "wx")` call, version 0.2.4 rethrows it because only `EEXIST` enters the retry path. No real Windows mandatory-locking repro artifact was posted.

## Solution Assessment

No, not as-is. The retry direction is reasonable, but the branch patches an inactive inline loop; the maintainable fix belongs at the fs-safe/file-lock boundary that current main actually uses.
