## Summary

Keep open: the Windows EPERM lock-contention crash remains source-reproducible, but this PR cannot merge as-is because it patches the old inline lock loop while current main delegates acquisition to fs-safe, and the contributor has not posted real Windows after-fix proof.

## What This Changes

The PR adds Windows-only EPERM retry handling and regression tests to the legacy inline session write-lock acquisition loop.

## Best Possible Solution

Handle Windows lock-contention EPERM at the active fs-safe sidecar/file-lock boundary, preserve fast failure for real permission errors, and consume the fixed dependency path in OpenClaw with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main calls `SESSION_LOCKS.acquire`, and the pinned fs-safe sidecar lock rethrows non-`EEXIST` `fs.open("wx")` errors, so a Windows EPERM would bubble out of session lock acquisition.

## Solution Assessment

No, not as-is. The retry direction is valid, but this branch patches stale inline code; the maintainable fix belongs at the fs-safe/file-lock boundary with real Windows proof.
