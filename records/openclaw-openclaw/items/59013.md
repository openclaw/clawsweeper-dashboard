## Summary

Keep open: the Windows EPERM crash remains a plausible source-reproducible bug on current main, but this PR currently edits the older inline lock loop instead of the active fs-safe-backed lock boundary and it still lacks real Windows after-fix proof.

## What This Changes

The PR adds Windows EPERM retry handling and regression tests around the older direct `acquireSessionWriteLock` `fs.open("wx")` lock loop.

## Best Possible Solution

Land a rebased fix at the active fs-safe/file-lock boundary, preserving fast failure for real permission errors while retrying Windows EPERM contention, with focused regression coverage and redacted real Windows proof.

## Reproduction Assessment

Yes, source-reproducible: current main delegates to `SESSION_LOCKS.acquire`, `@openclaw/fs-safe@0.2.2` rethrows non-`EEXIST` sidecar-lock open errors, and the OpenClaw wrapper rethrows non-file-lock errors. I did not run a live Windows repro in this read-only review.

## Solution Assessment

No for this PR as written. Windows-only EPERM retrying is the right direction, but the implementation must move to the current fs-safe/file-lock boundary and be backed by real Windows runtime proof.
