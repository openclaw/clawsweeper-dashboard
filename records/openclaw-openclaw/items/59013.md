## Summary

Keep open. Current main still appears to rethrow `EPERM` from the session lock create path, while this PR’s useful fix is not on main, is dirty against the current `@openclaw/fs-safe`-backed implementation, and lacks after-fix Windows runtime proof.

## What This Changes

The PR changes session write-lock acquisition to treat guarded Windows `EPERM` from exclusive lock-file creation as lock contention and adds EPERM retry tests.

## Best Possible Solution

Rebase or reimplement the Windows-only EPERM contention handling at the current file-lock boundary, preserve fast failure for real permission errors, add current regression coverage, and require redacted real Windows proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates to `@openclaw/fs-safe@0.2.2`, whose sidecar lock code rethrows non-`EEXIST` errors, and `acquireSessionWriteLock` rethrows non-file-lock errors, so a mocked Windows `EPERM` from `fs.open(lockPath, "wx")` would still escape.

## Solution Assessment

No for the current branch as-is. The intended Windows-only EPERM retry is the right narrow direction, but it must be reconciled with the current `@openclaw/fs-safe` lock path and backed by real Windows proof before merge.
