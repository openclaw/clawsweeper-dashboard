## Summary

Keep open: the Windows EPERM crash is source-reproducible, but this PR is not merge-ready because it patches the old inline lock loop while current main delegates lock acquisition to fs-safe, and the PR still lacks real Windows after-fix proof.

## What This Changes

The PR adds Windows-only EPERM retry handling to the old direct `fs.open("wx")` session write-lock loop and adds mocked regression tests.

## Best Possible Solution

Handle Windows lock-contention EPERM at the active fs-safe sidecar/file-lock boundary, while preserving fast failure for real permission errors and adding focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates to fs-safe, and the pinned fs-safe sidecar lock rethrows non-EEXIST `fs.open("wx")` errors, so a Windows EPERM from that open path would bubble out of `acquireSessionWriteLock`.

## Solution Assessment

No, not as-is. The narrow retry direction is reasonable, but the fix needs to move to the active fs-safe-backed lock boundary and be proven on real Windows.
