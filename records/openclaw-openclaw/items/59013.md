## Summary

Keep open: the reported Windows `EPERM` session-lock crash is still source-reproducible, but this branch now targets the old inline lock loop instead of current main’s fs-safe-backed lock boundary and still lacks real Windows after-fix proof.

## What This Changes

The PR adds Windows-only `EPERM` retry handling to the older `acquireSessionWriteLock` inline `fs.open` loop and adds mocked EPERM regression tests.

## Best Possible Solution

Implement Windows-only `EPERM` contention handling at the current file-lock boundary used by `SESSION_LOCKS`, preserve fast failure for real permission errors, add focused regression coverage, and require redacted real Windows proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates to `@openclaw/fs-safe@0.2.2`, whose sidecar lock code rethrows non-`EEXIST` errors, and `acquireSessionWriteLock` rethrows non-file-lock errors, so a mocked Windows `EPERM` from `fs.open(lockPath, "wx")` would still escape.

## Solution Assessment

No for the current branch as-is. The Windows-only retry direction is narrow, but the implementation must move to the current fs-safe-backed lock boundary and be backed by real Windows proof.
