## Summary

Keep open. Current main has overlapping first-class backup support, but it does not implement the macOS keep-awake helper or this Docker-specific restore workflow; the PR is useful prior work but is not merge-ready because the added backup/restore scripts handle secret-bearing archives and restore extraction too loosely.

## What This Changes

The PR adds macOS Docker keep-awake, backup, and restore helper scripts and documents them in the Docker install guide.

## Best Possible Solution

Keep the useful keep-awake/operator idea, but route backup semantics through the existing `openclaw backup create/verify` contract where possible and add a hardened restore path that validates tar member paths before extraction, defaults backup output outside the checkout or ignores it explicitly, documents secret-handling expectations, and has focused shell/CLI tests.

## Reproduction Assessment

Yes. The PR head shows the restore script extracting before member-path validation, and the backup script defaulting to an unignored repo-local output path while including `.env` and OpenClaw state; no mutating run is required to reproduce those code paths.

## Solution Assessment

No. The current PR is additive and addresses several prior comments, but the backup/restore design is not the safest maintainable path until it matches current main's archive validation contract and avoids placing secret-bearing backups in the source checkout by default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Validate tar entries before extracting backups:** `scripts/migrate/restore-openclaw.sh:102`
  - body: The restore path runs `tar -xzf "$ARCHIVE_PATH" -C "$tmpdir"` before inspecting any member names. A tarball whose companion `.sha256` was replaced with it can contain absolute or `../` entries and write outside `$tmpdir` during extraction; the later `SHA256SUMS` check happens after the overwrite. List and reject unsafe entries, or reuse the existing backup verifier's traversal checks, before extraction.
  - confidence: 0.88
- **[P1] Do not default secret backups inside the repo:** `scripts/migrate/backup-openclaw.sh:83`
  - body: With no `--output-dir`, the script writes archives to `$REPO_ROOT/backups`, while the archive includes `.env` and the OpenClaw config/workspace. The repo ignores `.env` but not `backups/` or these archive names, so following the docs leaves secret-bearing files as untracked checkout files that are easy to commit. Default outside the repo or add a matching ignore before documenting this path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces secret-bearing backup/restore scripts with concrete archive extraction and checkout-local backup exposure risks.

Concerns:

- **[high] Unsafe archive extraction before path validation:** `scripts/migrate/restore-openclaw.sh:102`
  - body: `restore-openclaw.sh` extracts the tarball before validating archive member paths, so a malicious or tampered archive plus matching companion checksum can write outside the temporary restore directory before the embedded manifest check runs.
  - confidence: 0.88
- **[high] Secret archives default to an unignored repo path:** `scripts/migrate/backup-openclaw.sh:83`
  - body: `backup-openclaw.sh` defaults to `$REPO_ROOT/backups` even though archives include `.env` and OpenClaw state; current ignore rules do not exclude that directory or archive suffixes, creating an avoidable credential commit risk.
  - confidence: 0.86
