## Summary

Keep open: the operator-helper idea is useful, but this PR is not merge-ready because the restore path still extracts a secret-bearing archive before enforcing a safe archive-entry/layout contract, the proof does not cover an actual restore or keep-awake lifecycle, and the keep-awake boundary overlaps https://github.com/openclaw/openclaw/pull/40846.

## What This Changes

This PR adds macOS Docker operator scripts for backup, restore, and manual keep-awake management, plus Docker install-guide documentation for those workflows.

## Best Possible Solution

Keep the operator migration direction, but make restore use a verifier-style archive preflight before extraction or live-state moves, decide the keep-awake ownership boundary, and require redacted macOS proof for backup, restore, and keep-awake behavior.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection of PR head shows `tar -xzf` runs before archive member/layout validation. I did not reproduce the full macOS Docker migration workflow locally.

## Solution Assessment

No, not as submitted. The maintainable path is to validate archive entries and expected payload layout before extraction or destructive restore steps, then settle whether keep-awake belongs in this Docker helper or the gateway lifecycle path.
