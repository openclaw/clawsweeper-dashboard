## Summary

Keep open. Current main still has the Gateway dedupe shrinking-loop bug, and the PR's production code is the right narrow fix, but the branch is not merge-ready because its regression test would still pass against the current bug and the required changelog entry is missing.

## What This Changes

The PR precomputes Gateway dedupe overflow, deletes a timestamp-sorted snapshot of oldest keys, and adds a maintenance test for reinserted older dedupe entries.

## Best Possible Solution

Land one canonical Gateway maintenance fix that snapshots the timestamp-sorted excess keys before deletion, adds a regression that fails on multi-entry overflow, records the user-facing fix in `CHANGELOG.md`, and then closes or supersedes the overlapping PR with a clear link.

## Reproduction Assessment

Yes. A current-main simulation with 1,010 dedupe entries deterministically leaves the map at 1,005 entries because the loop bound shrinks during deletion; the PR's current one-entry overflow test does not reproduce that failure.

## Solution Assessment

Yes for the production code: precomputing the excess count and deleting a timestamp-sorted snapshot is the narrowest maintainable fix. The PR is not complete as-is because the regression test should cover overflow by more than one entry and the changelog entry is missing.
