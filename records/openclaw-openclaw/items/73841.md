## What This Changes

## Best Possible Solution

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Seed the Gateway dedupe map above `DEDUPE_MAX` by more than one entry and advance the maintenance timer; current main under-evicts because the loop condition reads the map size while deleting from it.

## Solution Assessment

Is this the best way to solve the issue? Yes for the production code: precomputing the excess count and deleting a timestamp-sorted snapshot is the narrowest maintainable fix. The branch still needs a regression test that fails on current main and a changelog entry.
