## Summary

Keep open. Current main still has the shrinking dedupe cleanup loop, and the PR’s production change is the narrow right fix, but it still needs a stronger regression test, a changelog entry, and coordination with overlapping open PR #45979 before landing.

## What This Changes

The PR changes Gateway maintenance dedupe cleanup to precompute the overflow count, delete a timestamp-sorted snapshot of oldest keys, and add a reinsertion-oriented overflow regression test.

## Best Possible Solution

Land one canonical Gateway maintenance fix that snapshots timestamp-sorted excess keys before deletion, adds a multi-entry overflow regression that fails on current main, records the user-facing fix in the changelog, and then closes or supersedes the overlapping PR with a clear link.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The current-main loop can be simulated deterministically with 1,010 dedupe entries and leaves the map at 1,005, while the PR's current one-entry regression still passes against the buggy loop.

## Solution Assessment

Is this the best way to solve the issue? Yes for the production code: precomputing `excess` and deleting a timestamp-sorted key snapshot is the narrowest maintainable fix. The PR is not fully ready as-is because the regression should overflow by more than one entry and the changelog entry is missing.
