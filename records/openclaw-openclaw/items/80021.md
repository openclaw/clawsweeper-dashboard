## Summary

Keep open: this PR targets a real Telegram leak/dedupe boundary, but the current patch introduces delivery-state and progress-contract regressions and still lacks inspectable real behavior proof before merge.

## What This Changes

The PR adds a Telegram runtime communications guard at `sendMessageTelegram`, hard-codes a static progress draft, and adds send-boundary tests for technical text suppression plus progress and media dedupe.

## Best Possible Solution

Keep the guard Telegram-owned at the final delivery boundary, but use claim/commit/release or post-send recording, scope media dedupe to real report artifacts, preserve existing progress rendering/configuration, and add focused tests plus redacted live restart proof.

## Reproduction Assessment

Yes: source inspection is enough to reproduce the introduced failure modes because `checkAndRecord` persists before Telegram delivery, and current tests/docs define the progress rendering contract the PR bypasses. I did not run tests because this was a read-only review.

## Solution Assessment

No: moving the guard to the final send boundary is a reasonable direction, but this implementation is not the best fix because it commits dedupe before successful delivery, over-dedupes local media, and replaces configured progress previews with a hard-coded message.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record dedupe only after successful delivery:** `extensions/telegram/src/send.ts:619-627`
  - body: These guard checks run before media loading or Telegram API sends. Because `createPersistentDedupe.checkAndRecord` persists immediately, a failed media load or transient Telegram send error will still mark the progress/report as delivered and suppress retries for the guard TTL. Use a claim/commit/release flow or record only after the send succeeds.
  - confidence: 0.93
- **[P2] Preserve configured progress rendering:** `extensions/telegram/src/bot-message-dispatch.ts:555`
  - body: Hard-coding `Идет работа.` bypasses the existing progress renderer, including `streaming.progress.label`, max-line handling, and tool progress details. Current Telegram tests and docs expect configured labels and tool text in progress drafts, so this regresses released behavior unless the contract is intentionally changed.
  - confidence: 0.9
- **[P2] Scope media dedupe to report artifacts:** `extensions/telegram/src/runtime-comms-guard.ts:168-170`
  - body: This treats every local media URL as a report artifact. Telegram delivery supports local images, audio, video, voice, and documents, so sending the same local photo or voice note again in the same route would be suppressed for seven days even though the PR only intends report/artifact dedupe.
  - confidence: 0.86
