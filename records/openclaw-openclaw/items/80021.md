## Summary

Keep open; this draft PR has blocking delivery regressions in the proposed guard and still lacks inspectable real behavior proof.

## What This Changes

The PR adds Telegram runtime communication filtering/dedupe logic and changes progress preview rendering and outbound delivery tests.

## Best Possible Solution

Keep the guard Telegram-owned, but record dedupe only after confirmed sends, scope media dedupe to report artifacts, preserve existing progress rendering/configuration, and add focused tests plus redacted live restart proof.

## Reproduction Assessment

Yes: source inspection is enough to reproduce the introduced failure modes because `checkAndRecord` persists before delivery and the PR calls it before `send`. Current tests also define the progress preview behavior that the PR bypasses; no runtime test was run because this was a read-only review.

## Solution Assessment

No: the proposed path is not the best fix because it makes dedupe authoritative before delivery succeeds, over-broadly suppresses local media, and replaces the existing configurable progress contract with a hard-coded message. A narrower guard should filter sensitive text while keeping the established delivery and progress contracts intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record dedupe only after successful delivery:** `extensions/telegram/src/runtime-comms-guard.ts:158-161`
  - body: These helpers call `checkAndRecord` before Telegram delivery runs. Since `createPersistentDedupe` writes the key during that call, a transient Telegram send failure will still mark the progress/final/report as delivered, and a retry after restart will be suppressed for the guard TTL. Use a claim/commit style guard or record only after `send` succeeds.
  - confidence: 0.93
- **[P2] Keep progress previews on the configured renderer:** `extensions/telegram/src/bot-message-dispatch.ts:560`
  - body: Hard-coding `Идет работа.` bypasses the existing progress renderer, including `streaming.progress.label`, max-line handling, and tool progress lines. Current Telegram dispatch tests assert custom labels and tool text, so this regresses configured progress previews unless the contract, docs, and tests are intentionally changed.
  - confidence: 0.9
- **[P2] Restrict report dedupe to report-like files:** `extensions/telegram/src/runtime-comms-guard.ts:168-170`
  - body: This treats every local media URL as a report artifact. Telegram `sendMedia` supports local images, audio, video, and documents, so sending the same local photo or voice note again in the same route would be suppressed for seven days even though the PR only intends report/artifact-style local files to dedupe.
  - confidence: 0.86
