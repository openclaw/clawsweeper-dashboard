## Label Justifications

- `P1`: The PR addresses user-visible WhatsApp reply loss after a channel restart, which is a broken channel workflow affecting real users.
- `merge-risk: 🚨 message-delivery`: The proposed socket fallback can change which live WhatsApp socket sends a captured reply, read receipt, or group-metadata-backed send.
- `merge-risk: 🚨 auth-provider`: The fallback crosses from an old captured auth context to a successor controller without proving the auth directory or paired account is the same.

## Summary

Keep open: the PR targets a real WhatsApp restart message-loss path and the current source matches the reported stale-socket failure, but it should not merge until the successor socket is tied to the same auth state and the contributor supplies real after-fix WhatsApp proof.

## What This Changes

The PR exposes a WhatsApp controller `getCurrentSock()` handle and makes inbound reply/read/group metadata paths fall back to the registered successor controller socket when the original monitor socket ref is null, with a regression test.

## Best Possible Solution

Keep the socket handoff idea, but make the registry fallback session-safe by matching `authDir` or an equivalent session identity before using the successor socket, then attach redacted live WhatsApp proof for the health-monitor restart reply path.

## Reproduction Assessment

Yes via source inspection, though not by a live run in this read-only review. Shutdown clears the controller socket ref and stops retries, while the current send path throws when `getCurrentSock()` returns null; the PR body also includes redacted production logs for the same 408 restart pattern.

## Solution Assessment

No, not quite. The controller registry is the right seam for this repair, but the fallback needs an authDir/session match before it is the safest maintainable solution.
