## Summary

Keep open: the bug path is source-reproducible and the PR is focused, but v2 still needs after-fix real WhatsApp proof and a PN/LID-safe self-identity comparison before it is merge-ready.

## What This Changes

This PR adds WhatsApp controller socket/self-identity handles and lets stale inbound reply closures use a registered successor socket after restart, with regression coverage for matching and mismatched identities.

## Best Possible Solution

Keep the successor-socket handoff, but derive comparable self identities from auth state or carry e164/phoneNumber through the handle, cover PN-vs-LID success and mismatch cases, and attach redacted live restart proof.

## Reproduction Assessment

Yes, at source level: current main binds replies to the old monitor `socketRef`, shutdown clears that ref, and `sendTrackedMessage()` throws once reconnect retries stop. I did not run a live WhatsApp restart in this read-only review.

## Solution Assessment

No, not yet. The successor-controller fallback is the right shape, but the identity guard should reuse auth-state/e164 normalization and the PR still needs after-fix real WhatsApp proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize self identities before successor fallback:** `extensions/whatsapp/src/inbound/monitor.ts:241`
  - body: Baileys allows `Contact.id` to be either a PN JID or a LID and carries the PN separately as `phoneNumber`, but this fallback compares only raw `jid`/`lid` values. If the original socket reports the PN form and the successor reports the LID form for the same account, `identitiesOverlap()` returns false and the captured reply still throws instead of using the successor. Please derive a comparable identity from auth state, or carry e164/phoneNumber through the handle, and cover the PN-vs-LID same-account success case.
  - confidence: 0.87
