## Summary

This is a focused WhatsApp message-delivery bug fix, but it should stay open because the successor-socket handoff still has a PN/LID identity-normalization gap and the external PR has only test output, not after-fix real WhatsApp proof.

## What This Changes

The PR adds WhatsApp controller socket/self-identity handles so captured inbound reply closures can use a registered successor socket after a health-monitor restart, with regression tests for matching and mismatched identities.

## Best Possible Solution

Keep the successor-controller handoff, but derive comparable self identities from auth state or carry e164/phoneNumber through the handle, cover PN-vs-LID success plus mismatch cases, and attach redacted live restart proof.

## Reproduction Assessment

Yes, at source level: current main binds replies to the old monitor socketRef, shutdown clears that ref, and sendTrackedMessage() throws once reconnect retries stop. I did not run a live WhatsApp restart in this read-only review.

## Solution Assessment

No, not yet. The successor-controller fallback is the right shape, but the identity guard should reuse auth-state/e164 normalization and the PR still needs after-fix real WhatsApp proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize self identities before successor fallback:** `extensions/whatsapp/src/inbound/monitor.ts:242`
  - body: The fallback gates handoff on `identitiesOverlap(originalSelfIdentity, successorIdentity)`, but both identities are built from raw `sock.user.id/lid` and `identitiesOverlap()` resolves without the authDir/LID mapping used elsewhere.
