## Summary

Keep open: the lazy CLI startup optimization is plausible, but the latest PR head still bypasses finalized runtime config semantics and lacks after-fix real behavior proof. Current main also has overlapping CLI registration changes, so this should be repaired and rebased before merge.

## What This Changes

The branch adds metadata-only plugin CLI registration helpers, routes lazy core-command startup through them, reuses the validated config snapshot, and adds focused plugin CLI tests.

## Best Possible Solution

Preserve finalized runtime config semantics while retaining the metadata-only lazy startup path only where safe, then rebase against current main's CLI registration policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: exercise validated-config plugin CLI registration with a config finalization effect, then assert the plugin loader receives the finalized runtime config rather than the raw snapshot config.

## Solution Assessment

No. The metadata-only registration path is a reasonable narrow optimization, but reusing snapshot.config is not the best fix because it drops existing getRuntimeConfig finalization semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve runtime config finalization before plugin CLI loading:** `src/plugins/cli.ts:38`
  - body: `readConfigFileSnapshot()` returns the materialized snapshot before `finalizeLoadedRuntimeConfig()` applies env-derived config, shell-env fallback, duplicate agent-dir checks, generated owner display secret persistence, and overrides. Returning `snapshot.config` here means the later plugin loader receives a supplied `cfg`, so `resolvePluginRuntimeLoadContext()` will not call `getRuntimeConfig()` and plugin activation/registration can run against pre-finalized config.
  - confidence: 0.86
