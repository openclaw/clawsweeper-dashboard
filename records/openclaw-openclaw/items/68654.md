## Summary

Keep open. The performance direction is plausible and the author addressed the earlier plugin-command fallback concern, but the PR still bypasses current `main` runtime-config finalization before plugin CLI loading.

## What This Changes

The PR adds metadata-only plugin CLI registration entries, conditionally uses them for lazy built-in command startup, reuses the validated config snapshot, and adds focused plugin CLI tests.

## Best Possible Solution

Repair the branch so the metadata-only lazy startup optimization preserves finalized runtime config semantics, preferably by keeping `getRuntimeConfig()` or adding an explicitly finalized snapshot helper with regression coverage and a changelog entry if the user-facing perf change remains.

## Reproduction Assessment

Yes. A focused test can mock a valid snapshot plus a finalized `getRuntimeConfig()` result, call plugin CLI registration from validated config, and assert the plugin loader receives the finalized runtime config rather than raw `snapshot.config`.

## Solution Assessment

No, not as written. The metadata-entry optimization is a reasonable narrow direction, but reusing `snapshot.config` is not the best solution because it changes established runtime config semantics before plugin CLI registration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve runtime config finalization before plugin CLI loading:** `src/plugins/cli.ts:38`
  - body: `readConfigFileSnapshot()` exposes the materialized snapshot before runtime config finalization applies env-derived config, shell-env fallback, duplicate agent-dir checks, generated owner-display secret persistence, and overrides. Returning `snapshot.config` here means plugin CLI loading receives a supplied `cfg`, so `resolvePluginRuntimeLoadContext()` will not call `getRuntimeConfig()` and plugins can register against pre-finalized config. Keep the optimization only after preserving the current finalization semantics.
