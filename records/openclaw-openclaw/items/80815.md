## Summary

Keep open: the migration-time readiness gate is useful, but the patch currently routes source discovery through destination-agent auth, so it can evaluate and migrate based on the wrong Codex account; the protected `maintainer` label also requires explicit maintainer handling.

## What This Changes

The PR adds Codex migration-time app-readiness checks, a shared plugin app cache-key helper, target-only app inventory cache invalidation after plugin activation, tests, and docs.

## Best Possible Solution

Keep the source-readiness gate, but make source discovery use the source `CODEX_HOME` credentials without destination-agent auth, add a mixed source/target account regression test, and then provide redacted real migration proof.

## Reproduction Assessment

Yes for the blocking PR regression by source inspection: the PR passes destination `agentDir` and `config` into source app-server requests, while the shared client applies auth from those values before `plugin/list`, `plugin/read`, and `app/list`. I did not run a live two-account Codex migration.

## Solution Assessment

No. The readiness gate is a maintainable direction, but the implementation must preserve source and destination auth contexts instead of evaluating source app readiness under the target agent profile.
