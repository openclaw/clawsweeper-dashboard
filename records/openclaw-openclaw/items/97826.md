This issue should stay open because current main and the latest release still use the stdin-backed ffprobe path for Telegram video dimensions, and the direct closing PR remains open and unmerged.

## What This Changes

The issue requests fixing Telegram `sendVideo` aspect-ratio distortion for large videos by making `probeVideoDimensions` return width and height reliably instead of failing through `ffprobe pipe:0`.

## Best Possible Solution

Keep this issue as the canonical tracker until https://github.com/openclaw/openclaw/pull/97853 lands with real large-video proof or maintainers replace it with an equivalent fix to the shared video-dimension probe.

## Reproduction Assessment

Yes for the source path, but not for a full live reproduction here. Current main still pipes the video buffer into `ffprobe pipe:0`, and both Telegram send paths omit width and height when that probe returns `undefined`; live proof still needs a large faststart MP4 sent through Telegram.

## Solution Assessment

Yes, the seekable-temp-file approach in the linked PR is a plausible narrow fix because it preserves the existing helper contract and targets the non-seekable ffprobe input path. The issue should close only after that PR or an equivalent fix lands with real large-video proof.
