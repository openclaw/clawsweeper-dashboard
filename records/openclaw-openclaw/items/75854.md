## Summary

Keep open. The report is fresh, concrete, and targets a real documented contract: cron `payload.model` is supposed to use the selected allowed model, while general model selection documents alias-first resolution. Current source appears to route cron payload models through the shared allowed-model resolver, but there is no cron-specific regression test for configured aliases like `medium -> lmstudio/...`, and the reporter’s v2026.4.29 runtime evidence is specific enough that this should not be closed as already implemented or as a duplicate of the broader related model-allowlist reports.

## What This Changes

Cron `agentTurn` jobs should resolve configured `payload.model` aliases before `agents.defaults.models` allowlist validation, or reject unsupported aliases clearly at create/update time.

## Best Possible Solution

Add a focused cron regression test using the reporter’s shape: `agents.defaults.models` entries with aliases such as `medium -> lmstudio/qwen3.6-35b-a3b`, a cron `agentTurn` payload with `model: "medium"`, and an assertion that the resolved provider/model is the allowlisted target before validation. If that test fails, fix the narrow cron/shared resolver path rather than adding cron-specific alias logic.

Yes. The report gives concrete config, payload, command sequence, and runtime error strings; the high-confidence source-level reproduction is a focused isolated cron model-selection test using those aliases and allowlist entries.
