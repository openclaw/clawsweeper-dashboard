## Summary

Keep open. The report is concrete and recent, and it targets a real cron/runtime boundary: current source has shared alias-first model resolution, but the cron tests mostly mock that resolver and I did not find an exact regression test proving `payload.model: "medium"` resolves through `agents.defaults.models[*].alias` before allowlist validation in the isolated cron run path. Given the reporter's v2026.4.29 runtime evidence, this should get a narrow fix/test rather than cleanup closure.

## What This Changes

Cron jobs with `payload.model` set to a configured alias should resolve that alias before model allowlist validation.

## Best Possible Solution

Add an exact cron regression test with `agents.defaults.models` aliases like `medium -> lmstudio/...` and `payload.model: "medium"`, then make the isolated cron execution path resolve that alias to the full provider/model before allowlist validation if the test exposes a gap.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path: configure aliases in `agents.defaults.models`, create an isolated `agentTurn` cron with `payload.model` set to the alias, and run it; the expected failure is the reported allowlist rejection against a default-provider ref.

## Solution Assessment

Yes, resolving through the shared alias-aware model resolver before allowlist validation is the narrow maintainable direction. The safer implementation path is to first add the missing cron-level regression test, because the shared helper appears alias-aware but the reported runtime still fails.
