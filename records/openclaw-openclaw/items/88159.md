The PR removes the `logs --follow` force-journal latch, preserves journal cursor state, retries `logs.tail` after journal fallback, and adds CLI regression tests.

## Best Possible Solution

Keep journal fallback temporary, retry Gateway RPC after each fallback, and emit text plus JSON source-transition metadata whenever follow output changes source.

## Reproduction Assessment

Yes, source-level: current main sets `forceJournal` after a journal payload, and the next follow iteration reads journal before attempting `logs.tail`. I did not run a live Linux Gateway restart/systemd session in this read-only review.

## Solution Assessment

No, not yet: removing the permanent journal latch is the right owner-boundary fix, but the best fix also reports source transitions before recovered Gateway/file lines. A broader Gateway or logging backend rewrite is not needed.
