## Summary

Keep open: the PR fixes a real CLI follow-loop bug that still exists on current main, but current head also changes source-transition output and repeated JSON metadata without docs, and the supplied terminal proof is from an older head that does not show that new behavior.

## What This Changes

The PR removes the persistent journal fallback latch in `openclaw logs --follow`, retries `logs.tail` after temporary journal fallback, emits source metadata on source changes, and adds regression tests.

## Best Possible Solution

Keep the temporary journal bridge with independent Gateway/journal cursors, document repeated source metadata for follow output, and require current-head terminal proof before merge.

## Reproduction Assessment

Yes, at source level. Current main sets `forceJournal` after a journal payload and checks it before `logs.tail`, so recovered Gateway RPC is not retried until the journal path fails or the CLI restarts; I did not execute tests because this review is read-only.

## Solution Assessment

Yes for the implementation layer: removing the journal latch in the CLI follow loop is the narrow owner-boundary fix. It is not merge-complete until the changed JSON/text output contract is documented and current-head proof is refreshed.
