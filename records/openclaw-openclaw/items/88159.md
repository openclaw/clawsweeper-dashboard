## What This Changes

The PR changes `openclaw logs --follow` so systemd journal fallback remains temporary, Gateway `logs.tail` is retried after recovery, and source metadata is emitted when the log source changes.

## Best Possible Solution

Keep the temporary journal bridge fix, document repeated text/JSON source metadata for follow streams, and merge only after current-head redacted terminal proof shows the CLI returning from journal fallback to Gateway output in a disposable Gateway/journal setup.

## Reproduction Assessment

Yes at source level. Current main sets `forceJournal` after a journal payload and checks it before `logs.tail`, so recovered Gateway RPC is not probed on the next follow loop.

## Solution Assessment

Yes for the implementation layer: removing the CLI follow-loop latch is narrower than changing Gateway logging or the systemd journal reader. It is not merge-complete until the changed output contract is documented and current-head proof is stronger.
