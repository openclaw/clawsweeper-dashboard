## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rebase protocol constants onto v4:** `src/gateway/protocol/version.ts:1`
  - body: This branch adds `PROTOCOL_VERSION = 3`, but current main has protocol v4 constants and native generated models advertise v4. Rebase and regenerate the protocol surfaces so merging this branch cannot downgrade or conflict with the current Gateway/native client contract.
  - confidence: 0.9
- **[P1] Return to the preserved cadence after exhausted retries:** `src/cron/service/timer.ts:498`
  - body: After the last failed retry, this path uses `Math.max(effectiveNaturalNextRunAtMs, fallbackNextRunAtMs)`. For a short recurring schedule, the fallback backoff can be later than the preserved natural slot, so the job skips expected scheduled runs after retry exhaustion instead of returning to cadence.
  - confidence: 0.88
- **[P2] Validate retry fields as an all-or-nothing policy:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: `retryCount` and `retryDelayMs` are optional independently, but `applyJobResult` only retries when both are positive. A caller can therefore create or patch a job with a half policy that validates successfully and never retries at runtime.
  - confidence: 0.87
- **[P2] Reject one-shot retry flags or implement them:** `src/cli/cron-cli/register.cron-add.ts:90-91`
  - body: The CLI exposes retry flags for every schedule kind, including `--at`, but the runtime only consults per-job retry fields in the recurring branch. Either block these flags for one-shot jobs or implement the same per-job contract there so users do not get accepted no-op settings.
  - confidence: 0.82
