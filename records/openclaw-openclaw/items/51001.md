## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Rebase protocol constants and generated clients onto v4:** `src/gateway/protocol/version.ts:1`
  - body: This branch adds `PROTOCOL_VERSION = 3`, but current main has already moved the Gateway protocol constants and generated native client guards to version 4. Merging this stale protocol split would advertise the wrong protocol level and break current client compatibility.
  - confidence: 0.93
- **[P1] Return to the preserved cadence after retry exhaustion:** `src/cron/service/timer.ts:498`
  - body: When a recurring retry fails after the retry budget is exhausted, this line chooses the later of the preserved natural slot and the generic error backoff. For short recurring jobs, that can skip many natural runs, which breaks the PR's no-cadence-drift guarantee.
  - confidence: 0.9
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: `retryCount` and `retryDelayMs` are accepted independently, but runtime retry only happens when both are positive. A caller can successfully create or update a job with only one field and get no retries, so the schema should enforce an all-or-nothing policy or define defaults.
  - confidence: 0.89
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: These patch fields are stored, but the update flow only recomputes `nextRunAtMs` for schedule/enabled changes or missing nextRunAtMs. Editing retry settings onto an already failed recurring job can therefore leave the old natural next run untouched, making the retry edit a no-op for incident response.
  - confidence: 0.86
- **[P2] Block one-shot retry flags or honor them:** `src/cli/cron-cli/register.cron-add.ts:90-91`
  - body: The CLI accepts `--retry-count` and `--retry-delay` for every cron add, including `--at` jobs, but the one-shot runtime path still uses only the existing global transient retry config. Reject these flags for one-shot jobs or wire them into that path so users do not get a silent no-op.
  - confidence: 0.85
- **[P2] Clear stale retry state after recurring job edits:** `src/cron/service/timer.ts:519-525`
  - body: The success path restores `priorNaturalNextRunAtMs` whenever stale retry state is present, but schedule edits do not clear that state. Editing a job while a retry is pending can snap the next successful run back to the old cadence instead of the edited schedule.
  - confidence: 0.82
