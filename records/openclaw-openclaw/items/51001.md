- **[P1] Rebase protocol constants onto v4:** `src/gateway/protocol/version.ts:1`
  - body: This branch adds `src/gateway/protocol/version.ts` with only `PROTOCOL_VERSION = 3`, while current main has protocol v4 plus min-client/probe constants and matching native generated values. Merging this stale file would break the current Gateway/native protocol compatibility contract.
  - confidence: 0.94
- **[P1] Return to preserved cadence after retry exhaustion:** `src/cron/service/timer.ts:498`
  - body: When the final configured recurring retry fails, this branch falls through to `Math.max(effectiveNaturalNextRunAtMs, fallbackNextRunAtMs)`. For short schedules the fallback backoff can be later than the preserved natural slot, so an exhausted retry skips scheduled runs instead of returning to cadence.
  - confidence: 0.87
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:290-291`
  - body: `retryCount` and `retryDelayMs` are accepted independently, but the scheduler only retries when both are positive. A caller can successfully create or update a job with one field and get no retry behavior, so this needs all-or-nothing validation or documented defaults.
  - confidence: 0.9
- **[P2] Reject retry flags on one-shot jobs:** `src/cli/cron-cli/register.cron-add.ts:271-272`
  - body: `cron add --at ... --retry-*` sends retry fields even though the new per-job policy is only read in the recurring error path; one-shot jobs still use the existing global transient retry logic. Accepting these flags makes the CLI look successful while runtime behavior is unchanged.
  - confidence: 0.84
