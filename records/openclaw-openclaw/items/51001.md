## Summary

Keep open: current main still lacks per-job recurring cron retry support, and this PR is a plausible implementation for the open request at https://github.com/openclaw/openclaw/issues/49740, but the latest head still needs real behavior proof plus several blocking scheduler and contract repairs before merge.

## What This Changes

The PR adds fixed-delay retryCount/retryDelayMs support for recurring cron jobs across cron runtime state, persistence, gateway schemas, generated Swift models, CLI add/edit flags, and focused tests, plus small gateway protocol import-splitting changes.

## Best Possible Solution

Land a repaired version that defines a complete recurring-only retry contract, validates retry policy atomically, preserves natural cadence through retry success and exhaustion, recomputes failed-job scheduling after retry edits, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the PR defects: current main lacks the feature, and the PR head shows the cadence, edit-recompute, schema, and CLI contract gaps. I did not run tests or a live cron because this was a read-only review.

## Solution Assessment

No, not as-is. The feature direction matches the linked request, but the implementation needs tighter recurring-only semantics, atomic validation, scheduler state repair on edits, and real behavior proof before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve cadence after the final retry fails:** `src/cron/service/timer.ts:498`
  - body: When a configured recurring retry is no longer scheduled, this branch can choose the generic error backoff over the preserved natural slot. For short schedules, the last failed retry then skips the intended cadence slot, contradicting the no-drift behavior promised by the PR.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: `applyJobPatch()` stores the new retry policy, but retry-only updates do not recompute `nextRunAtMs`. Editing a failed recurring job to add retry settings can leave the existing later natural run in place, making the new settings ineffective for the incident-response case.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The schema accepts `retryCount` and `retryDelayMs` independently, while runtime only retries when both are positive. API callers can create or update a job that passes validation but never retries, so validate the pair as all-or-nothing or define a documented default.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot jobs:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: These fields are forwarded for any `cron add` schedule, but the per-job retry policy is only used in the recurring error branch. A one-shot job can appear to accept `--retry-*` while runtime behavior remains unchanged; reject those flags for one-shot jobs or implement matching one-shot semantics.
  - confidence: 0.82
