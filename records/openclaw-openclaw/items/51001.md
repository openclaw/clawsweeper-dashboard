## What This Changes

## Best Possible Solution

Land a repaired version that defines a complete recurring-only retry contract, validates policy atomically, preserves natural cadence through success and exhaustion, updates failed-job scheduling on retry edits, and keeps protocol/generated clients/docs aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve cadence after the final retry fails:** `src/cron/service/timer.ts:498`
  - body: When a configured recurring retry is no longer scheduled, this branch can choose the generic error backoff over the preserved natural slot. For short schedules, the last failed retry then skips the intended cadence slot, contradicting the no-drift behavior this PR promises.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: applyJobPatch() stores the new retry policy, but the update path only recomputes nextRunAtMs for schedule/enabled changes or missing next-run state. Editing a failed recurring job to add retry settings can leave the existing later natural run in place, making the new settings ineffective.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The schema accepts retryCount and retryDelayMs independently, but the runtime only retries when both are positive. API callers can create a job that passes validation while never retrying, so validate this as an all-or-nothing policy or define a documented default.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot jobs:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: These fields are forwarded for any cron add schedule, but the per-job retry policy is only used in the recurring error branch. A one-shot job can appear to accept --retry-* while runtime behavior remains unchanged; reject those flags for one-shot jobs or implement the same policy there.
  - confidence: 0.82
