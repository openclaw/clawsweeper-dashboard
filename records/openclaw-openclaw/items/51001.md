## Summary

Keep open. Current main still lacks configurable recurring cron retry support, and this open PR remains the active implementation candidate, but the latest head still has source-backed scheduler/API semantics defects plus missing real behavior proof before merge.

## What This Changes

The PR adds fixed-delay retry fields for recurring cron jobs across cron runtime state, job persistence, Gateway/Swift protocol models, CLI add/edit flags, and focused cron/CLI regression tests, plus small protocol import splits for startup memory.

## Best Possible Solution

Land a repaired additive implementation that validates retry policy atomically, defines recurring versus one-shot semantics explicitly, recomputes or documents retry edits for active failures, preserves cadence through retry success and exhaustion, updates docs/protocol clients, and includes real behavior proof.

## Reproduction Assessment

Yes for the PR review defects by source inspection: current main lacks the feature, and PR head shows concrete cadence, validation, edit, and one-shot contract gaps. I did not run live cron or tests because this was a read-only review.

## Solution Assessment

No, not as-is. The feature direction matches the canonical request, but the implementation needs tighter scheduler semantics, atomic validation, edit-state handling, one-shot policy clarity, and real behavior proof before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve cadence after the final retry fails:** `src/cron/service/timer.ts:498`
  - body: When no retry is scheduled, this branch can choose the generic error backoff over the preserved natural slot. For short recurring schedules, the last failed retry can skip intended cadence slots, contradicting the no-drift behavior promised by this PR.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: applyJobPatch() stores the new retry policy, but retry-only updates do not recompute nextRunAtMs. Editing a failed recurring job to add retry settings can leave the existing later natural run in place, so the new settings do not take effect for the current failure cycle.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The schema accepts retryCount and retryDelayMs independently, while runtime only retries when both are positive. API callers can create or update a job that validates successfully but never retries, so the pair should be all-or-nothing or have documented defaults.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot cron add:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: These fields are forwarded for every cron add schedule, but runtime only consults retryCount/retryDelayMs in the recurring error branch. A one-shot job can appear to accept --retry-* while behavior remains governed by the existing one-shot retry config.
  - confidence: 0.82
- **[P2] Reject retry edits on one-shot cron jobs:** `src/cli/cron-cli/register.cron-edit.ts:140-153`
  - body: cron edit accepts --retry-* and stores the fields without checking the existing job schedule, but one-shot error handling never reads the per-job retry policy. The edit flow should reject these flags for one-shot jobs or implement matching semantics.
  - confidence: 0.8
