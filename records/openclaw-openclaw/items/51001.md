## Summary

Keep open. Current main still lacks per-job cron retry controls, and this PR is a useful implementation candidate for the open feature request, but the PR head is not mergeable as-is because it still has scheduler semantics and validation bugs around cadence, edits, partial policies, and one-shot schedules.

## What This Changes

The PR adds fixed-delay retryCount/retryDelayMs support for recurring cron jobs across cron runtime state, gateway schemas, generated Swift clients, CLI add/edit options, and regression tests.

## Best Possible Solution

Keep this PR open as the implementation candidate for #49740, or replace it with a focused maintainer patch that adds per-job recurring retry policy end to end while preserving existing backoff protections, validating retry policy shape, defining one-shot behavior, clearing/recomputing retry state on edits, updating docs/generated clients, and covering the edge cases with regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve cadence after exhausted retries:** `src/cron/service/timer.ts:498`
  - body: When the last configured recurring retry fails, this branch uses the global error backoff as a lower bound. For a short schedule such as every 5s with one 1s retry, the 30s fallback can win over the preserved natural slot, skipping scheduled runs and breaking the advertised cadence-preserving behavior.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: `applyJobPatch()` stores the new retry policy, but `update()` only recomputes `nextRunAtMs` for schedule/enabled changes or missing next-run state. Editing a failed recurring job to add retry settings can leave the existing later natural run in place, making the new retry settings a no-op for the incident-response case.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The schema accepts `retryCount` and `retryDelayMs` independently, but the runtime only retries when both are positive. API callers can create or patch a job that passes validation while silently never retrying, so this should be all-or-nothing or have a documented default.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot schedules:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: These fields are forwarded for any `cron add` schedule, but the runtime only consults the per-job retry policy in the recurring branch. A one-shot `--at` job can appear to accept the retry policy while continuing to use only the existing global transient retry behavior.
  - confidence: 0.82
- **[P3] Validate retry count without partial parsing:** `src/cli/cron-cli/register.cron-edit.ts:140-145`
  - body: The edit path uses `Number.parseInt`, so a value like `2abc` is accepted as `2` even though the add path rejects non-integer input. Reuse the shared positive-integer parser so add and edit enforce the same CLI contract.
  - confidence: 0.78
