## What This Changes

The PR adds per-job fixed-delay retry fields for recurring cron jobs across cron runtime/state, CLI add/edit, Gateway schemas, generated Swift models, and regression tests, plus small Gateway protocol import splits for startup memory.

## Best Possible Solution

Land an additive recurring retry implementation only after retry policy validation, one-shot/edit semantics, cadence preservation, current protocol v4/client surfaces, focused tests, and redacted real CLI/gateway proof are all aligned.

## Reproduction Assessment

Yes. Source inspection of PR head shows the exhausted-retry Math.max path, independently optional retry fields, and update flow that stores retry edits without recomputing an already-failed job; no live cron run was executed in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this patch still has silent no-op configurations, stale protocol risk, and a cadence bug for short recurring schedules; the safer solution is to validate the retry contract and make edit/one-shot behavior explicit before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve cadence after exhausted retries:** `src/cron/service/timer.ts:498`
  - body: When an explicit recurring retry is exhausted, this branch still chooses the later of the preserved natural slot and the generic error backoff. For a short schedule such as every 5s with one 1s retry, the second failure jumps to the 30s/60s backoff instead of returning to the preserved cadence, contradicting the new retry-without-cadence-drift contract.
  - confidence: 0.93
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The gateway accepts retryCount and retryDelayMs independently, but applyJobResult only retries when both are positive. A caller can create or update a job with only one field and get a successful response with no retry behavior, so require the pair together or define a default for the missing side.
  - confidence: 0.88
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: Patching retryCount or retryDelayMs only stores the new fields; update() leaves nextRunAtMs unchanged unless the schedule/enabled flag changes or the value is missing. Editing a failed daily recurring job to add retries therefore keeps waiting for the existing natural run instead of scheduling the requested earlier retry.
  - confidence: 0.86
- **[P2] Gate retry flags to schedules that use them:** `src/cli/cron-cli/register.cron-add.ts:90-91`
  - body: These new flags are accepted for any cron add/edit flow, including one-shot jobs, but the one-shot error branch still uses only the global cron.retry policy and never reads the per-job retry fields. That makes a successful one-shot command look configured while the new settings are ignored at runtime.
  - confidence: 0.8
