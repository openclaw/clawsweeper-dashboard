## What This Changes

The PR adds fixed-delay recurring cron retry state and retryCount/retryDelayMs wiring through cron runtime, persistence, gateway schemas/generated Swift models, CLI add/edit, focused tests, and two gateway protocol import splits.

## Best Possible Solution

Land a repaired version that validates retry policy as a complete recurring-only contract, preserves natural cadence through success and exhaustion, updates failed-job scheduling when retry settings change, clears stale retry-cycle state on relevant edits, keeps generated protocol clients aligned, and adds the required changelog entry.

## Reproduction Assessment

Yes. Current main lacks per-job recurring retry controls, and the PR-specific defects are visible from the timer scheduling, jobs update, gateway schema, and CLI flag wiring paths without needing a live run.

## Solution Assessment

No, not as-is. The feature direction matches the linked cron retry request, but the patch needs tighter recurring-only validation and scheduler state handling before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve cadence after exhausted retries:** `src/cron/service/timer.ts:498`
  - body: When a configured recurring retry is no longer scheduled, this branch can choose the generic error backoff over the preserved natural slot. For short schedules, the final failed retry then skips the intended cadence slot, contradicting the no-drift behavior this PR promises.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: applyJobPatch() stores the new retry policy, but the update path only recomputes nextRunAtMs for schedule/enabled changes or missing next-run state. Editing a failed recurring job to add retry settings can leave the existing later natural run in place, making the new settings ineffective.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:290-294`
  - body: The gateway schema accepts retryCount and retryDelayMs independently, but the runtime only retries when both are positive. API callers can create a job that passes validation while never retrying, so validate this as an all-or-nothing policy or define a documented default.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot add:** `src/cli/cron-cli/register.cron-add.ts:271-275`
  - body: cron add forwards retryCount and retryDelayMs for any schedule, but the per-job retry policy is only used in the recurring error branch. A one-shot job can appear to accept --retry-* while runtime behavior remains unchanged.
  - confidence: 0.82
- **[P2] Reject retry edits for one-shot jobs:** `src/cli/cron-cli/register.cron-edit.ts:140-153`
  - body: cron edit accepts and forwards retry settings even for existing one-shot jobs, while one-shot runtime handling still uses only the global transient retry config. Guard this path or implement equivalent one-shot semantics.
  - confidence: 0.8
- **[P2] Clear stale retry-cycle state on schedule edits:** `src/cron/service/timer.ts:95-101`
  - body: The new recurring retry state can survive a schedule edit while a retry is pending. A later completion can then restore a stale natural slot from the old cadence, so edits that change schedule or retry policy need to clear or recompute the retry-cycle state.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/cli/cron-cli/register.cron-add.ts:87-88`
  - body: This PR adds user-facing cron CLI/API behavior, but the file list does not include CHANGELOG.md. Add a single-line Changes entry with appropriate contributor credit before merge.
  - confidence: 0.83
