## Summary

Keep open: current main still lacks per-job recurring cron retry support, and this PR is a useful implementation candidate but still has concrete scheduler, validation, CLI, and changelog blockers before merge.

## What This Changes

The PR adds fixed-delay retry fields for cron jobs across cron runtime state, gateway schemas, generated Swift protocol models, CLI add/edit, and targeted cron tests.

## Best Possible Solution

Land a repaired version that validates recurring retry policy as a complete, recurring-only contract, preserves natural cadence through success and exhaustion, updates failed-job scheduling when retry settings change, and records the user-facing change in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main has no per-job recurring retry controls, and the PR-specific defects are reproducible from the diff paths in timer scheduling, jobs patching, gateway validation, and CLI flag wiring.

## Solution Assessment

No, not as-is. The feature direction matches #49740, but the patch needs tighter recurring-only validation and scheduler state handling before it is the narrowest maintainable solution.

  - body: The gateway schema accepts retryCount and retryDelayMs independently, but the runtime only retries when both are positive. API callers can create a job that passes validation while never retrying, so validate this as an all-or-nothing policy or define a documented default.
  - confidence: 0.88
