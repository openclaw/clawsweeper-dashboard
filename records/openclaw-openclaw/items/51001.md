## Summary

Keep open: current main still lacks configurable recurring cron retry support, and this PR needs correctness fixes plus real behavior proof before it can merge.

## What This Changes

This PR adds fixed-delay recurring cron retry fields through cron runtime state, Gateway/Swift protocol models, CLI add/edit flags, persistence tests, and a small protocol import split for startup memory.

## Best Possible Solution

Land a repaired additive implementation that validates retry policy atomically, makes recurring versus one-shot semantics explicit, recomputes failed-job scheduling on retry edits, preserves cadence through success and exhaustion, updates protocol/docs, and includes real behavior proof.

## Solution Assessment

No, not as-is. The feature direction matches the canonical request, but the implementation needs tighter scheduler semantics, atomic validation, edit-state repair, and real behavior proof before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve cadence after the final retry fails:** `src/cron/service/timer.ts:498`
  - body: When a configured recurring retry is no longer scheduled, this branch can choose the generic error backoff over the preserved natural slot. For short schedules, the last failed retry can skip intended cadence slots, contradicting the no-drift behavior promised by this PR.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: applyJobPatch() stores the new retry policy, but retry-only updates do not recompute nextRunAtMs. Editing a failed recurring job to add retry settings can leave the existing later natural run in place, so the new settings do not take effect for the current failure cycle.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The schema accepts retryCount and retryDelayMs independently, while runtime only retries when both are positive. API callers can create or update a job that validates successfully but never retries, so the pair should be all-or-nothing or have documented defaults.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot cron add:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: These fields are forwarded for every cron add schedule, but runtime only consults retryCount/retryDelayMs in the recurring error branch. A one-shot job can appear to accept --retry-* while behavior remains governed by the existing one-shot retry config.
  - confidence: 0.82
- **[P2] Reject retry edits on one-shot cron jobs:** `src/cli/cron-cli/register.cron-edit.ts:140-153`
  - body: cron edit now accepts --retry-* and stores the fields without checking the existing job schedule, but one-shot error handling never reads the per-job retry policy. The edit flow should reject these flags for one-shot jobs or implement matching semantics.
  - confidence: 0.8
