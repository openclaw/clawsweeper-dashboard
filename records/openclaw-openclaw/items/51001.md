## Summary

Keep open. Current main still lacks per-job fixed-delay retry controls for recurring cron jobs, and this PR is a useful implementation candidate for the linked feature request, but the patch still has blocking scheduler and validation defects before it is mergeable.

## What This Changes

The PR adds `retryCount`/`retryDelayMs` support across cron job types/state, gateway cron schemas, generated Swift protocol models, CLI `cron add/edit`, runtime retry scheduling, and focused cron regression tests.

## Best Possible Solution

Keep this PR open and repair the recurring-only implementation so configured retries are validated as a complete policy, cannot be silently accepted for unsupported one-shot jobs, preserve natural cadence after success or exhaustion, recompute pending failed jobs when retry settings change, and include the required changelog entry before merge.

## Reproduction Assessment

Yes. Current main can be verified by code inspection: recurring failures only apply the existing exponential backoff floor and the CLI/schema have no per-job retry controls; the PR-specific defects are reproducible from the provided diff and review comments without a live gateway.

## Solution Assessment

No, not as-is. The direction is appropriate for the linked feature request, but the patch needs tighter scheduler semantics and validation before it is the narrowest maintainable solution.

Full review comments:

- **[P2] Preserve cadence after exhausted retries:** `src/cron/service/timer.ts:498`
  - body: When a configured recurring retry is no longer scheduled, this branch can pick the generic error backoff over the preserved natural slot. For short schedules, a final failed retry then skips the intended cadence slot, contradicting the PR's no-drift behavior.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: `applyJobPatch()` stores the new retry policy, but the update path only recomputes `nextRunAtMs` for schedule/enabled changes or missing next-run state. Editing a failed recurring job to add retry settings can leave the existing later natural run in place, making the new retry settings ineffective.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:290-294`
  - body: The gateway schema accepts `retryCount` and `retryDelayMs` independently, but the runtime only retries when both are positive. API callers can create a job that passes validation while never retrying, so validate this as an all-or-nothing policy or define a documented default.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot jobs:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: These fields are forwarded for any `cron add` schedule, but the per-job retry policy is only used in the recurring error branch. `cron add --at ... --retry-*` therefore appears to configure retries while runtime behavior remains unchanged; reject these flags for one-shot jobs or implement the same policy there.
  - confidence: 0.82
