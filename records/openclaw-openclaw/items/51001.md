## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve cadence after retries are exhausted:** `src/cron/service/timer.ts:498`
  - body: When the last retry fails, this path chooses `Math.max(effectiveNaturalNextRunAtMs, fallbackNextRunAtMs)`. For a short recurring job such as `--every 5s --retry-count 1 --retry-delay 1s`, the fallback backoff can push the next run far past the preserved natural slot, which breaks the PR's no-cadence-drift contract.
  - confidence: 0.91
- **[P2] Require complete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: `retryCount` and `retryDelayMs` are accepted independently, but runtime only retries when both are positive. A caller can successfully create or patch a job with one field and get no retry behavior, so the schema/service should reject partial policies or supply an explicit default.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: These setters persist retry edits, but `cron.update` only recomputes `nextRunAtMs` for schedule/enabled changes or missing values. Adding retry settings to an already-failed recurring job can therefore succeed while leaving the old natural next run in place instead of scheduling the requested retry wake.
  - confidence: 0.86
- **[P2] Reject retry flags for one-shot jobs:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: The add path sends retry settings for any schedule, including `--at`, but the one-shot error branch never reads `job.retryCount` or `job.retryDelayMs`. Until one-shot per-job retry is supported, these flags should fail closed for one-shot jobs instead of storing a policy that runtime ignores.
  - confidence: 0.84
- **[P1] Rebase the protocol split onto version 4:** `src/gateway/protocol/version.ts:1`
  - body: This branch adds `src/gateway/protocol/version.ts` with only `PROTOCOL_VERSION = 3`, while current main requires protocol v4 plus the min-client and probe constants. Resolve the stale branch by preserving current v4 exports before merging the import split.
  - confidence: 0.93
