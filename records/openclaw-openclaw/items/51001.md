## Summary

Keep open: this is a useful active feature PR, but it is not merge-ready. Current main still lacks per-job recurring cron retry support, while the PR head has blocking cadence, validation, and edit-semantics issues, and it lacks real behavior proof; https://github.com/openclaw/openclaw/issues/49740 remains open with this PR as the implementation candidate.

## What This Changes

The PR adds per-job fixed-delay recurring cron retry fields through CLI add/edit, gateway schemas, persisted cron job state, scheduler state, Swift protocol models, and regression tests, plus small gateway protocol import splits for startup memory.

## Best Possible Solution

Land a rebased additive implementation that validates a complete recurring retry policy, preserves natural cadence after retry success and exhaustion, handles edit and one-shot semantics explicitly, updates protocol/generated clients/docs, and includes focused tests plus redacted real CLI/gateway proof.

## Reproduction Assessment

Yes for the review findings. Source inspection of PR head shows the retry-exhaustion Math.max path, independently optional retry fields, and update flow that stores retry edits without recomputing an already-failed job's next wake; I did not run a live cron job in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this patch still has silent no-op configurations and a cadence bug for short recurring schedules; the safer solution is to validate the retry contract and make edit/one-shot behavior explicit before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve cadence after exhausted retries:** `src/cron/service/timer.ts:498`
  - body: When an explicit recurring retry is exhausted, this branch still chooses the later of the preserved natural slot and the generic error backoff. For a short schedule such as every 5s with one 1s retry, the second failure will jump to the 30s/60s backoff instead of returning to the preserved 5s cadence, contradicting the new retry-without-cadence-drift contract.
  - confidence: 0.93
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The gateway accepts retryCount and retryDelayMs independently, but applyJobResult only retries when both are positive. A caller can successfully create or update a job with only one field and then get no retry behavior at runtime, so the schema should require the pair together or define a default for the missing side.
  - confidence: 0.88
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: Patching retryCount or retryDelayMs only stores the new fields; update() leaves nextRunAtMs unchanged unless the schedule/enabled flag changes or the value is missing. Editing a failed daily recurring job to add retries therefore keeps waiting for the existing natural run instead of scheduling the requested earlier retry.
  - confidence: 0.86
- **[P2] Block retry flags on one-shot schedules:** `src/cli/cron-cli/register.cron-add.ts:90-91`
  - body: These flags are accepted for any cron add invocation, including --at jobs, but the one-shot error branch still uses only the global cron.retry policy and never reads the per-job retry fields. That makes a successful one-shot command look configured while the new settings are ignored at runtime.
  - confidence: 0.8
