Full review comments:

- **[P1] Rebase the protocol split onto version 4:** `src/gateway/protocol/version.ts:1`
  - body: This branch adds `PROTOCOL_VERSION = 3`, but current main has already moved the Gateway protocol constants and native generated models to version 4. Keeping this stale value during rebase would advertise an old protocol level and break current native/client compatibility, so the branch needs to be rebased and regenerated against protocol v4 before merge.
  - confidence: 0.93
- **[P1] Return to the preserved cadence after retry exhaustion:** `src/cron/service/timer.ts:498`
  - body: When a recurring retry fails after the retry budget is exhausted, this line chooses the later of the preserved natural slot and the generic error backoff. For a short recurring job, that can skip many natural runs, which breaks the PR's stated guarantee that retries do not affect schedule cadence.
  - confidence: 0.9
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: `retryCount` and `retryDelayMs` are accepted independently, but the runtime only retries when both are positive. A caller can successfully create or update a job with only one field and get no retries, so the schema should enforce an all-or-nothing policy or supply an explicit default.
  - confidence: 0.89
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: These patch fields are stored, but the update flow only recomputes `nextRunAtMs` for schedule/enabled changes or a missing next run. Editing retry settings onto an already failed recurring job can therefore leave the old natural next run untouched, making the incident-response edit appear to work while no retry is scheduled.
  - confidence: 0.86
- **[P2] Block one-shot retry flags or honor them:** `src/cli/cron-cli/register.cron-add.ts:90-91`
  - body: The CLI accepts `--retry-count` and `--retry-delay` for every cron add, including `--at` jobs, but the one-shot runtime path still uses only the existing global transient retry config. Either reject these flags for one-shot jobs or wire them into the one-shot retry path so users do not get a silent no-op.
  - confidence: 0.85
