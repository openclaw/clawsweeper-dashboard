## Summary

Keep open: current main and the latest release still map default/full-off Claude CLI runs to `bypassPermissions` with no root-aware branch; the `can_use_tool` dead-air stall is mostly addressed by https://github.com/openclaw/openclaw/issues/80819, but the root gateway permission-mode decision remains unresolved.

## What This Changes

The issue requests root-aware Claude CLI permission-mode handling, clearer root/systemd docs and diagnostics, and better surfacing when Claude permission handling stalls instead of waiting for the watchdog timeout.

## Best Possible Solution

Keep a maintainer-owned path that makes Claude CLI root behavior explicit: either fail fast with a clear diagnostic or route through a permission-control path, update root/systemd docs, and preserve intentional exec-policy security semantics.

## Reproduction Assessment

Not fully with live proof. Current main clearly reproduces the root-relevant source path that maps default YOLO exec policy to `bypassPermissions`, but the actual root/sudo Claude CLI rejection was not rerun in this read-only review.

## Solution Assessment

Unclear. Automatically switching root gateways to `acceptEdits` is only one possible fix; a fail-fast diagnostic or a live/ACP permission-control path may be safer because this is a security-sensitive provider boundary.
