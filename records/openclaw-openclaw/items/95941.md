## Summary

Keep open. Current main and the latest release support command cron jobs, but they still build one deliverable summary from stdout and stderr with no per-stream delivery filter, so the requested behavior remains a new config/API feature.

## What This Changes

Add a command-payload cron option such as `deliveryStreams` so announce/webhook delivery can include only selected stdout/stderr streams while diagnostics and watchdog monitoring keep both.

## Best Possible Solution

If maintainers accept the surface, add an additive default-compatible delivery stream selector for command payloads, preserving full diagnostics and watchdog monitoring while filtering only announce/webhook payload text.

## Reproduction Assessment

Yes. A command cron job that writes a clean stdout result and heartbeat/progress text to stderr follows the current `buildCommandSummary` path and will produce the documented combined stdout/stderr delivery block; I did not run it because this review is read-only.

## Solution Assessment

Unclear. The proposed stream list is plausible, but maintainers should first confirm whether the selector belongs on the command payload, delivery config, or a narrower command-output policy and how it should interact with failure alerts.
