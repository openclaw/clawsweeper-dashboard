## Summary

Keep open: current main and the latest release still have no command-payload `deliveryStreams` field, and the only exact companion implementation is closed unmerged. This remains a real feature/config-surface request that needs maintainer product review before a new cron payload and CLI contract is added.

## What This Changes

The issue requests a `deliveryStreams` option for command-payload cron jobs so announce/webhook delivery can filter stdout and stderr independently while preserving diagnostics.

## Best Possible Solution

Add one canonical command-payload stream filter only if maintainers accept the config surface; keep the default as both streams, preserve watchdog and diagnostics behavior, and align schema, runtime, storage, CLI, docs, and focused tests.

## Reproduction Assessment

Yes for the current source behavior: command summaries combine stdout and stderr when both are non-empty, and the schema/CLI expose no stream selector. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear pending maintainer decision: stream filtering belongs near the command summary/delivery contract, but the exact field shape and failure-alert semantics need owner approval because this adds public config surface.
