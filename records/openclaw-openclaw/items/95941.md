## Summary

Keep open: current main and the latest release still have no `deliveryStreams` command-payload field, and command cron delivery is still built from the combined stdout/stderr summary. The exact companion implementation at https://github.com/openclaw/openclaw/pull/95955 is closed unmerged, so the requested behavior still needs maintainer product approval rather than cleanup closure.

## What This Changes

The issue requests a `deliveryStreams` option for command-payload cron jobs so announce/webhook delivery can include stdout and/or stderr independently while run diagnostics and watchdog behavior keep observing both streams.

## Best Possible Solution

If maintainers accept the config surface, add one canonical command-payload stream filter with default both streams, apply it only to delivery summaries, preserve watchdog and diagnostics behavior, and update schema, runtime, CLI/UI, docs, and focused tests together.

## Reproduction Assessment

No failing bug reproduction is applicable because this is a feature request. Source inspection is high-confidence: current main combines stdout and stderr into one command summary and the protocol schema disallows unknown command-payload fields.

## Solution Assessment

Unclear pending maintainer decision. Stream filtering likely belongs near the command summary/delivery boundary, but the exact field shape, CLI/UI exposure, failure-alert semantics, and truncation order need owner approval.
