## Summary

Keep open: current main and v2026.6.9 still lack the requested heartbeat, and this branch is not merge-ready because the carried tool timestamp can still shorten later provider waits while the supplied proof does not cross the old idle window.

## What This Changes

The branch adds a runId-keyed tool-activity heartbeat, wraps embedded-runner tool execution to notify on completion, passes runId into streamWithIdleTimeout, clears heartbeat state on attempt teardown, and adds Vitest coverage.

## Best Possible Solution

Make the pre-stream tool timestamp single-use or clear it after the bridged wait, add a multi-chunk regression test, and require redacted long-running subagent proof before merge.

## Reproduction Assessment

Yes from source inspection, not from a live run in this review. Current main and v2026.6.9 have an LLM-only idle watchdog while agent-core executes tools between model turns, and the PR timer logic shows the carried-timestamp failure path.

## Solution Assessment

No as written. The run-scoped heartbeat boundary is plausible, but the carried timestamp must be consumed or cleared after the bridged wait, and the PR still needs proof that a real long-running subagent survives beyond the old idle window.
