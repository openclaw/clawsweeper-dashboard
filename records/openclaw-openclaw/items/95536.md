## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Notify the signal observed by the idle wrapper:** `src/agents/embedded-agent-runner/run/attempt.ts:3687`
  - body: `streamWithIdleTimeout` subscribes using the stream call's `options.signal`, which is the Agent active-run signal in this path, but this notification uses the embedded attempt's outer `runAbortController.signal`. Because the heartbeat map is keyed by `AbortSignal`, cataloged tool completions can miss the registered listener and fail to reset the idle timer.
  - confidence: 0.94
- **[P1] Record activity instead of retaining dead listeners:** `src/agents/embedded-agent-runner/run/llm-idle-timeout.ts:303`
  - body: Leaving `onToolActivity` subscribed after iterator cleanup does not carry progress into the next provider wait: cleanup has already set that stream closure's `waitingForProvider` to false, so a later tool completion calls `armTimer()` as a no-op and also accumulates stale closures. Store run-scoped activity, or otherwise make the next provider wait consume the completed-tool signal.
  - confidence: 0.9
- **[P1] Emit heartbeats from shared tool completion:** `src/agents/embedded-agent-runner/run/attempt.ts:3687`
  - body: This hook only runs inside the Tool Search catalog executor. Normal direct tools, including the reported `read` and `exec` style tools, execute through agent-core's shared `executePreparedToolCall` and `tool_execution_end` path, so those workflows can still hit the LLM idle timeout unless the shared completion boundary is wired or proved irrelevant.
  - confidence: 0.88
