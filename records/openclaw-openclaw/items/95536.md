## Summary

Keep this PR open, but it is not merge-ready: the heartbeat direction is useful, yet the patched idle timer can reuse a stale tool timestamp after fresh model output and abort an active stream too early. The PR also still lacks after-fix proof from a real long-running subagent run.

## What This Changes

The PR adds a run-scoped tool-activity heartbeat, wires tool completion notifications into embedded-runner tool execution, extends the LLM idle watchdog to observe that heartbeat, and adds focused Vitest coverage.

## Best Possible Solution

Keep the embedded-runner heartbeat approach, but consume the tool-activity offset only once when returning to the provider stream, add a regression test for stale timestamp reuse, and require redacted real subagent proof before merge.

## Reproduction Assessment

Yes, from source inspection of the patch. After a prior tool heartbeat, the patched wrapper resets first-arm accounting for each iterator.next(), so stale tool activity can shorten waits after fresh model output.

## Solution Assessment

No as patched. The heartbeat boundary is reasonable, but the timer offset should be consumed once after tool work or stream creation instead of being reapplied to every provider wait.

- **[P1] Consume Tool Heartbeats Only Once:** `src/agents/embedded-agent-runner/run/llm-idle-timeout.ts:334`
  - body: The new code sets `firstArmPending = true` every time the stream iterator waits for the next provider event. Keep the shortened timeout only for the first wait after tool work or stream creation, then let normal model progress reset subsequent waits.
