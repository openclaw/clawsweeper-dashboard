## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Notify the stream wrapper's active-run signal:** `src/agents/embedded-agent-runner/run/attempt.ts:3687`
  - body: `streamWithIdleTimeout` subscribes tool activity on the provider stream's `sourceSignal`, which comes from agent-core's active-run signal, but this notification uses the embedded attempt's outer `runAbortController.signal`. Since the heartbeat map is keyed by `AbortSignal` identity, cataloged tool completion can miss the registered listener and leave the provider wait timing out.
  - confidence: 0.94
- **[P1] Persist activity instead of stale callbacks:** `src/agents/embedded-agent-runner/run/llm-idle-timeout.ts:303`
  - body: Keeping old stream listeners registered does not persist tool progress: after a provider stream finishes, its listener has `waitingForProvider = false`, so a later tool completion calls `armTimer()` as a no-op. Store run-scoped last-tool activity, or otherwise carry the activity into the next provider wait, instead of relying on stale per-stream callbacks.
  - confidence: 0.9
- **[P1] Emit heartbeats from shared tool completion:** `src/agents/embedded-agent-runner/run/attempt.ts:3687`
  - body: This hook only runs for Tool Search catalog calls. The reported direct tools such as `read` and `exec` execute through agent-core's shared `executePreparedToolCall` path and emit `tool_execution_end` outside this executor, so those workflows can still hit the LLM idle timeout unless the shared completion path is wired or proved irrelevant.
  - confidence: 0.88
