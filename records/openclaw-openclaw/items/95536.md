## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Persist tool activity outside stream iterators:** `src/agents/embedded-agent-runner/run/llm-idle-timeout.ts:303`
  - body: This registers `armTimer` on `sourceSignal` from inside each provider stream iterator and then leaves that callback registered, but once a stream finishes `waitingForProvider` is false, so tool completion between LLM turns calls a stale no-op callback. Store run-scoped last-tool activity, or otherwise carry the activity into the next provider wait, instead of relying on per-stream callbacks.
  - confidence: 0.92
- **[P1] Notify the same signal the stream wrapper observes:** `src/agents/embedded-agent-runner/run/attempt.ts:3687`
  - body: `streamWithIdleTimeout` subscribes to tool activity on the stream call's `sourceSignal`, which is agent-core's active-run signal on the normal path, but this notification uses the embedded attempt's outer `runAbortController.signal`. Since the heartbeat map is keyed by AbortSignal identity, cataloged tool completion can miss the registered listener and leave the provider wait timing out.
  - confidence: 0.94
- **[P1] Emit heartbeats from shared tool completion:** `src/agents/embedded-agent-runner/run/attempt.ts:3687`
  - body: This hook only runs for Tool Search catalog calls. The reported direct tools such as `read` and `exec` execute through agent-core's shared `executePreparedToolCall` path and emit `tool_execution_end` outside this executor, so those workflows can still hit the idle-timeout behavior unless the shared completion path is wired or proved irrelevant.
  - confidence: 0.88
