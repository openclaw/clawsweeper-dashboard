## Summary

Keep open: the run-scoped heartbeat direction is useful, but the current head has a timer lifecycle bug that can make later provider waits inherit a stale tool timestamp, and the PR still has only test-output proof rather than a real long-running subagent run.

## What This Changes

The PR adds a run-scoped tool-activity heartbeat, wires tool completion notifications into embedded-runner tool execution, extends the LLM idle watchdog to observe that heartbeat, and adds focused Vitest coverage.

## Best Possible Solution

Keep the run-scoped heartbeat approach, fix the first-arm lifecycle so old tool activity affects only the first post-tool provider wait, add focused regression coverage, and require redacted real long-running subagent proof before merge.

## Reproduction Assessment

Yes for the PR defect from source inspection: PR head resets firstArmPending on every iterator wait, so stale tool activity can shorten later provider waits. No live subagent reproduction was run; the PR body confirms only focused tests were run.

## Solution Assessment

No. The run-scoped heartbeat boundary is the right direction, but the current timer lifecycle is not the best fix until the stored tool timestamp is consumed once and the real subagent path is proven.
