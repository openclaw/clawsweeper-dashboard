## Summary

Keep this PR open. Current main still lacks Android reverse-proxy transport auth, but the proposed patch is not merge-ready because proxy credentials can be sent outside the intended gateway origin and the TLS bypass needs explicit security/product handling.

## What This Changes

The PR adds Android manual/onboarding fields for reverse-proxy bearer/basic authentication and a TLS verification bypass, then sends those credentials through gateway and canvas network paths.

## Best Possible Solution

Revise the Android implementation so proxy credentials are persisted but only attached to the gateway WebSocket upgrade and strictly trusted same-origin canvas HTTP(S) requests, with the TLS bypass removed or split into a separately approved development-only flow.

## Reproduction Assessment

Yes. Configure bearer/basic proxy auth, then have the gateway invoke canvas.navigate with an external URL; the PR's top-level WebView.loadUrl path applies Authorization headers to that URL. The scheme-scoping bug is reproduced by the new test's expectation that both http and https with the same host:port match the same remoteAddress.

## Solution Assessment

No. The feature direction is valid and earlier persistence/duplicate-header comments were addressed, but this is not the best mergeable solution until credentials are scoped by full trusted origin and TLS validation bypass behavior receives explicit security/product treatment.
