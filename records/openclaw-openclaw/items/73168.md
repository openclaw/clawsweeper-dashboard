## Summary

Keep this PR open, but not merge-ready. The feature addresses the still-open Android reverse-proxy auth request, but the latest diff can leak proxy credentials from Android canvas loads and adds a persistent TLS verification bypass that needs maintainer/security review.

## What This Changes

This PR adds Android manual/onboarding fields for reverse-proxy bearer/basic auth and a TLS verification bypass, persists the credentials, and sends them through gateway WebSocket and canvas HTTP/WebView paths.

## Best Possible Solution

Revise the Android implementation so proxy credentials are persisted but only attached to the gateway WebSocket upgrade and strict same-origin gateway/canvas requests, with TLS bypass removed or split into a separately approved development-only flow.

## Reproduction Assessment

Yes. Configure bearer/basic proxy auth, then have the gateway invoke canvas.navigate with an external URL; the PR's top-level WebView.loadUrl path applies Authorization headers to that URL. The scheme-scoping bug is also visible from the added test expectation that http and https with the same host:port match the same remoteAddress.

## Solution Assessment

No. The feature direction is valid and addresses #51872, but the current implementation is not the narrowest maintainable solution because credential attachment is not fully origin-scoped and TLS verification bypass is persisted as normal connection state.
