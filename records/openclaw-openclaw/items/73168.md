## Summary

This PR should stay open: the reverse-proxy auth feature is useful and is the active implementation candidate for the related open request, but the current diff still exposes proxy credentials on canvas loads and adds a persisted TLS trust bypass that needs maintainer/security review. Earlier bot comments about persistence, duplicate headers, and StateFlow updates appear addressed in the latest head.

## What This Changes

The PR adds Android manual/onboarding fields for bearer/basic reverse-proxy auth, persists those values, sends Authorization headers on gateway WebSocket and canvas WebView requests, and adds a persisted disable-TLS-verification option.

## Best Possible Solution

Land a revised Android implementation that persists bearer/basic proxy auth and sends it only to the configured gateway/canvas origin, while removing the TLS bypass or splitting it into a separately approved development-only flow.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows canvas.navigate accepts a URL parameter and the PR passes proxy Authorization headers into loadUrl for any currentUrl, while the interceptor trusts only host:port. I did not need a live Android run to establish those code-path failures.

## Solution Assessment

No. The feature direction is valid, but the mergeable solution needs strict full-origin credential scoping and an explicit maintainer/security decision on whether any persisted TLS bypass belongs in the Android app.

## Security Review

Status: needs_attention

  - body: The PR applies bearer/basic Authorization headers to every WebView top-level canvas navigation, while canvas.navigate accepts gateway-provided URLs; this can send reverse-proxy credentials to a non-gateway origin.
