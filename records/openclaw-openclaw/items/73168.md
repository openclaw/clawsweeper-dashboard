## Summary

Keep open. The feature is a valid implementation candidate for the linked reverse-proxy auth request and the latest force-push appears to address earlier persistence and duplicate-header bot feedback, but the current diff still leaks proxy Authorization headers through Android canvas loading paths and adds a persistent TLS verification bypass that needs maintainer/security review.

## What This Changes

The PR adds Android manual/onboarding fields for reverse-proxy bearer/basic auth, persists those credentials, sends them on gateway and canvas requests, and adds a persisted disable-TLS-verification toggle.

## Best Possible Solution

Revise the Android implementation so proxy credentials are persisted but attached only to the gateway WebSocket upgrade and strictly trusted same-origin canvas HTTP(S) requests, with the TLS bypass removed or split into a separately approved development-only flow.

## Reproduction Assessment

Yes. Configure bearer/basic proxy auth, then have the gateway invoke canvas.navigate with an external URL; the PR's top-level WebView.loadUrl path applies Authorization headers to that URL. The scheme-scoping issue is also visible in the new test expectation that http and https with the same host:port match the same remoteAddress.

## Solution Assessment

No. The feature direction is useful, but the proposed implementation is not merge-ready until Authorization forwarding is scoped by full trusted origin and the TLS verification bypass receives explicit security/product treatment.
