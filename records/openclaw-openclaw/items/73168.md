## Summary

Keep this PR open: the Android reverse-proxy auth feature is useful and tied to the open proxy-auth request, but the latest branch still has credential-scoping security blockers and no contributor-provided real Android/proxy proof.

## What This Changes

The PR adds Android manual/onboarding inputs for bearer/basic reverse-proxy auth, persists them, sends Authorization headers on gateway WebSocket and canvas WebView requests, and adds a disable-TLS-verification setting.

## Best Possible Solution

Land a revised Android implementation that sends proxy Authorization only to the exact configured gateway/canvas origin and separates or removes the TLS bypass until explicitly approved.

## Reproduction Assessment

Yes for the PR blockers: source inspection on PR head shows canvas.navigate accepts gateway-provided URLs while CanvasController attaches proxy Authorization to any currentUrl, and CanvasScreen compares only host:port. A live Android run was not needed to establish those code-path failures.

## Solution Assessment

No. The feature direction is valid, but the current patch is not the best mergeable solution because credential forwarding needs exact-origin scoping and the TLS bypass needs a separate maintainer/security decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope canvas load auth to the gateway origin:** `apps/android/app/src/main/java/ai/openclaw/app/node/CanvasController.kt:166`
  - body: canvas.navigate accepts gateway-provided URLs, but this path passes proxy Authorization headers into every top-level loadUrl call. A navigation to an external URL would disclose reverse-proxy credentials, so attach these headers only after matching the configured gateway/canvas origin.
  - confidence: 0.94
- **[P2] Compare canvas request auth by full origin:** `apps/android/app/src/main/java/ai/openclaw/app/ui/CanvasScreen.kt:132-134`
  - body: The interceptor compares only host:port against remoteAddress, so http://host:port and https://host:port both qualify for proxy Authorization. Include the scheme in the trusted origin comparison before adding credentials.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch can disclose reverse-proxy credentials from Android canvas loads and adds a persisted TLS validation bypass.

Concerns:

- **[high] Top-level canvas navigation can leak credentials:** `apps/android/app/src/main/java/ai/openclaw/app/node/CanvasController.kt:166`
  - body: The PR applies bearer/basic Authorization headers to every WebView top-level canvas navigation, while canvas.navigate accepts gateway-provided URLs; this can send reverse-proxy credentials to a non-gateway origin.
  - confidence: 0.94
- **[medium] Canvas request scoping ignores URL scheme:** `apps/android/app/src/main/java/ai/openclaw/app/ui/CanvasScreen.kt:132`

  - confidence: 0.86
