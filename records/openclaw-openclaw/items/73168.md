## Summary

Keep this PR open. Current main still lacks Android reverse-proxy bearer/basic transport auth, and this PR is the active implementation candidate for the linked request, but the latest head still has security-sensitive credential scoping bugs in canvas loading plus an explicit persistent TLS verification bypass that needs maintainer/security review.

## What This Changes

The PR adds Android manual/onboarding fields for bearer/basic reverse-proxy auth and disabling TLS verification, persists the new settings, sends Authorization headers on gateway WebSocket and canvas paths, and adds focused Android tests.

## Best Possible Solution

Revise the Android implementation so reverse-proxy credentials are persisted but only attached to the gateway WebSocket upgrade and strictly trusted same-origin canvas HTTP(S) requests using a shared scheme/host/port comparison. Remove the TLS bypass or split it into a separately approved, warning-heavy development flow, then add changelog and targeted tests for credential scoping.

## Reproduction Assessment

Yes. Configure bearer/basic proxy auth, have the gateway invoke canvas.navigate with an external URL, and the PR's WebView.loadUrl(currentUrl, headers) path sends Authorization to that URL; for the scheme bug, an HTTPS gateway with an HTTP same-host:port canvas request passes the PR's authority-only check and is asserted as matching in the new test.

## Solution Assessment

No. The feature direction is valid and the latest head fixed earlier persistence/duplicate-header issues, but this is not the best mergeable shape until proxy credentials are scoped by full trusted origin and the TLS bypass receives explicit product/security treatment.
