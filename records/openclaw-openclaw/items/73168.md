## Summary

Keep open. The Android proxy-auth feature direction is valid, but this PR still has source-reproducible credential scoping bugs, a security-sensitive persisted TLS verification bypass, and no real Android/proxy behavior proof after the patch.

## What This Changes

The PR adds Android manual/onboarding fields for bearer/basic reverse-proxy auth, persists those credentials, forwards Authorization headers through gateway WebSocket and canvas WebView requests, and adds a disable-TLS-verification setting.

## Best Possible Solution

Land a revised Android implementation that sends proxy Authorization only to the exact configured gateway/canvas origin and separates or removes the TLS bypass until explicitly approved.

## Reproduction Assessment

Yes for the PR blockers: source inspection on PR head shows gateway-provided canvas URLs can receive proxy Authorization headers, and the interceptor scopes credentials by host:port instead of full origin. A live Android run is still needed for contributor behavior proof, but not to establish these code-path failures.

## Solution Assessment

No. The feature direction is reasonable, but the current patch is not the best mergeable solution because credential forwarding needs exact-origin scoping and the TLS bypass needs separate maintainer/security approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope canvas load auth to the gateway origin:** `apps/android/app/src/main/java/ai/openclaw/app/node/CanvasController.kt:166`
  - body: canvas.navigate accepts gateway-provided URLs, but this path passes proxy Authorization headers into every top-level loadUrl call. A navigation to an external URL would disclose reverse-proxy credentials, so attach these headers only after matching the configured gateway/canvas origin.
  - confidence: 0.94

  - body: The interceptor compares only host:port against remoteAddress, so http://host:port and https://host:port both qualify for proxy Authorization. Include the scheme in the trusted origin comparison before adding credentials.
  - confidence: 0.86
