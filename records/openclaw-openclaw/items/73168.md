## Summary

Keep this PR open. Current main still lacks Android transport-level proxy auth, but the latest PR head is not merge-ready because it can send reverse-proxy Authorization credentials to off-origin or scheme-mismatched canvas requests, and the persistent TLS-verification bypass needs maintainer/security approval.

## What This Changes

The PR adds Android manual/onboarding bearer and basic reverse-proxy auth fields, persists those values, threads Authorization headers through gateway WebSocket and canvas loading paths, adds a TLS-verification bypass toggle, and adds focused Android tests.

## Best Possible Solution

Land a narrowed Android proxy-auth implementation that persists bearer/basic values, sends Authorization only on the gateway WebSocket upgrade and strictly same-origin canvas HTTP(S) requests using shared scheme/host/port checks, and either removes the TLS bypass or ships it as a separately approved development-only flow with strong warnings, tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure bearer/basic proxy auth, invoke canvas.navigate with an external HTTPS URL, and observe the PR's wv.loadUrl(currentUrl, headers) path would attach Authorization; another is an HTTPS gateway authority with an http://same-host:same-port canvas request passing the authority-only interceptor check.

## Solution Assessment

No. The feature direction fits the Android side of the linked reverse-proxy request, and the latest head fixed earlier persistence and duplicate-header issues, but the current implementation does not safely scope credentials to the exact trusted origin and adds a persistent TLS trust bypass without a settled security design.

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff can leak reverse-proxy credentials from Android canvas loading and introduces a persistent TLS verification bypass.

Concerns:

- **[high] Top-level canvas navigation can leak credentials:** `apps/android/app/src/main/java/ai/openclaw/app/node/CanvasController.kt:166`
  - body: The PR applies bearer/basic `Authorization` headers to every top-level canvas `WebView.loadUrl` navigation, while canvas navigation accepts command-supplied URLs; this can disclose reverse-proxy credentials to a non-gateway origin.
  - confidence: 0.94
