## Summary

Keep open. Current main still lacks Android transport-level proxy auth, and this PR is the active implementation for the linked feature, but the latest diff is not merge-ready because it can expose proxy Authorization headers through canvas navigation, scopes canvas request auth by authority instead of full origin, and adds a persisted TLS verification bypass that needs security/product review.

## What This Changes

The PR adds Android manual/onboarding fields and persistence for reverse-proxy bearer/basic auth plus a TLS verification bypass, wiring those values into gateway WebSocket and canvas WebView requests with focused tests.

## Best Possible Solution

Revise the Android implementation so reverse-proxy credentials are stored but sent only on the gateway WebSocket upgrade and strictly trusted same-origin canvas HTTP(S) requests, with the TLS bypass removed or split into a separately approved development-only flow.

## Reproduction Assessment

Yes. Source inspection shows canvas.navigate accepts gateway-provided URLs, while the PR passes proxy Authorization headers to WebView.loadUrl for any currentUrl; the new test also expects http and https with the same host:port to both match.

## Solution Assessment

No. The feature direction is valid and earlier persistence/duplicate-header comments appear addressed, but the mergeable solution should scope credentials by full trusted origin and handle TLS bypass through explicit maintainer/security approval.
