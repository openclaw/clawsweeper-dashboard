## Summary

Keep this PR open. Current main still lacks Android transport-level bearer/basic auth for reverse-proxy-protected gateways, and this PR is the active implementation path for #51872, but the latest head is not merge-ready because it can leak proxy credentials through canvas navigation/subrequests and adds a persistent TLS verification bypass that needs maintainer/security approval.

## What This Changes

The PR adds Android manual/onboarding fields, SecurePrefs persistence, gateway WebSocket and canvas Authorization-header plumbing for bearer/basic reverse-proxy auth, a disable-TLS-verification option, TLS mismatch prompt changes, and focused Android tests.

## Best Possible Solution

Revise the Android implementation so reverse-proxy credentials are persisted but only attached to the gateway WebSocket upgrade and strictly same-origin canvas HTTP(S) requests using a shared scheme/host/port comparison. Remove the TLS bypass or split it into a separately approved, warning-heavy development flow, then add the required changelog entry and targeted tests for credential scoping.

## Reproduction Assessment

Yes. Configure bearer/basic proxy auth, have the gateway invoke canvas.navigate with an external URL, and the PR's WebView.loadUrl(currentUrl, headers) path sends Authorization to that URL; the scheme-scoping bug is also captured by the new test accepting both http and https for the same host:port.

## Solution Assessment

No. The feature direction is valid, but the current patch is not the narrowest maintainable solution until proxy credentials are scoped by full trusted origin and the TLS bypass is removed or explicitly approved as a development-only flow.
