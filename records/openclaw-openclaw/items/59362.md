## Summary

Keep open: current main and v2026.5.18 still route normal non-sandbox Windows host exec through a PowerShell `-Command` wrapper, so the reported console-flash path remains source-reproducible. The merged `windowsHide` fixes cover adjacent direct exec and shell-env probe paths, but not the normal exec tool wrapper contract described here.

## What This Changes

The issue requests a Windows exec behavior change so simple exec tool calls do not flash a console window when commands are spawned.

## Best Possible Solution

Define a Windows exec invocation contract that direct-spawns safely parsed simple executable calls with hidden windows while preserving PowerShell fallback for shell syntax, wrappers, PTY behavior, and approval policy, then prove both paths on Windows.

## Reproduction Assessment

Yes, source-reproducible: on Windows, `getShellConfig()` deterministically returns PowerShell `-Command`, and `runExecProcess()` always wraps normal non-sandbox host exec commands in that shell. I did not rerun live Windows visual proof in this read-only review.

## Solution Assessment

Unclear. Direct-spawning simple executable invocations is plausible, but the best fix needs a maintainer-approved boundary for shell features, wrappers, PTY fallback, and exec approval semantics.
