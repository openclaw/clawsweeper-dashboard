## Summary

Keep open: current main and the latest release still route normal Windows host exec through a PowerShell `-Command` wrapper, so the reported console-flash path remains source-reproducible; the related `windowsHide` shell-env fix addresses a different spawn path.

## What This Changes

The issue requests that Windows exec tool invocations avoid visible console window flashes when spawning commands.

## Best Possible Solution

Define a Windows exec invocation contract that direct-spawns safely parsed simple executable calls with hidden windows while preserving PowerShell fallback for shell syntax, builtins, wrappers, and existing approval behavior.

## Reproduction Assessment

Yes from source inspection: on Windows, normal host exec deterministically builds a PowerShell `-Command` child process while the child adapter hides only that wrapper. I did not run live Windows visual proof in this review.

## Solution Assessment

Unclear. Direct-spawning simple executable invocations is plausible, but the maintainable fix needs a maintainer-approved fallback boundary for shell features, wrappers, PowerShell cmdlets, and approval policy.
