## Summary

Keep open: current main and the latest release still route normal Windows gateway exec through a PowerShell `-Command` wrapper, while the already-merged `windowsHide` fixes cover adjacent spawn paths rather than the core exec-tool wrapper reported here.

## What This Changes

The issue requests that Windows exec-tool calls stop flashing a visible console window by avoiding the PowerShell wrapper when a direct hidden spawn is safe.

## Best Possible Solution

Design a Windows exec invocation contract that direct-spawns safely parsed simple executable calls with hidden windows, while falling back to the existing shell wrapper for shell syntax, wrappers, cmdlets, and approval-sensitive cases.

## Reproduction Assessment

Yes from source inspection: on Windows, a normal gateway host exec call still deterministically builds a PowerShell `-Command` child process. I did not run live Windows visual proof in this read-only review.

## Solution Assessment

Unclear. Direct-spawning simple executable invocations is plausible, but the maintainable fix needs an explicit fallback boundary for shell features, `.cmd` wrappers, PowerShell cmdlets, output capture, and approval policy.
