## Label Justifications

- `P1`: The report is a Windows regression affecting every normal exec tool invocation with visible UI disruption on real host workflows.

## Summary

Keep open: current main and the latest release still route normal Windows host exec through a PowerShell `-Command` wrapper, so the reported console-flash path remains source-reproducible even though adjacent `windowsHide` gaps were fixed.

## What This Changes

The issue asks OpenClaw to stop flashing a Windows console window during normal agent exec tool calls, likely by direct-spawning simple executable commands instead of always wrapping them in PowerShell.

## Best Possible Solution

Define a Windows exec invocation contract that direct-spawns safely parsed simple executable calls with hidden windows while preserving PowerShell fallback for shell syntax, builtins, wrappers, and existing approval behavior.

## Reproduction Assessment

Yes from source inspection: on Windows, normal host exec deterministically builds a PowerShell `-Command` child process while the child adapter only hides that wrapper. I did not run live Windows visual proof in this review.

## Solution Assessment

Unclear. Direct-spawning simple executable invocations is plausible, but the maintainable fix needs a maintainer-approved fallback boundary for shell features, wrappers, PowerShell cmdlets, and approval policy.
