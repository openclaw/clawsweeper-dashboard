## Summary

Keep open. Current main still routes non-sandbox host exec commands through PowerShell on Windows, and the existing `windowsHide: true` mitigation is applied to that wrapper process rather than avoiding it. The related focus-steal cleanup does not cover this more specific PowerShell-wrapper flash report.

## What This Changes

The issue asks OpenClaw to stop Windows exec tool calls from briefly showing a console window by spawning simple external commands directly instead of always wrapping them in PowerShell.

## Best Possible Solution

Keep the issue open until OpenClaw has a Windows exec implementation that avoids visible wrapper windows for simple external commands while falling back to the existing PowerShell path for shell features. The fix should include focused Windows-path unit coverage and, ideally, native Windows smoke proof for a direct executable invocation and a shell-feature invocation.

## Reproduction Assessment

Yes. The concrete reproduction is to run OpenClaw on Windows and trigger a normal host exec call; current main deterministically builds a PowerShell `-Command` child process for that path, which matches the reporter's root-cause example.

## Solution Assessment

Unclear. Direct-spawning simple commands is likely the right direction for the no-flash case, but the maintainable solution needs a carefully bounded design so command parsing, approvals, `.cmd` wrappers, and PowerShell-only shell syntax do not regress.
