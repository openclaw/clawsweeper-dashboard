## Summary

Keep open: current main and the latest release still route normal non-sandbox Windows host exec through a PowerShell `-Command` wrapper, so the reported console-flash path remains source-reproducible. The related shell-env probe flash was fixed separately, but that does not change the normal exec tool wrapper contract described here.

## What This Changes

The issue asks OpenClaw to stop flashing visible Windows console windows when the exec tool spawns commands.

## Best Possible Solution

Define a Windows exec invocation contract that direct-spawns safely parsed simple executable calls with hidden windows while preserving PowerShell fallback for shell syntax, wrappers, PTY behavior, and approval policy, then prove both paths on Windows.

## Reproduction Assessment

Yes, source-reproducible: on Windows, `getShellConfig()` deterministically returns PowerShell `-Command`, and `runExecProcess()` always wraps normal non-sandbox host exec commands in that shell. I did not run live Windows visual proof in this read-only review.

## Solution Assessment

Unclear. Direct-spawning simple executable invocations is plausible, but the best fix needs a maintainer-approved boundary for shell features, wrappers, PTY fallback, and exec approval semantics.
