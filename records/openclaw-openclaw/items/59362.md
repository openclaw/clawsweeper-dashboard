## Summary

Keep open: current main and the latest release still route normal Windows host exec through a PowerShell `-Command` wrapper; related `windowsHide` fixes covered separate spawn/helper paths but not the reported wrapper behavior.

## What This Changes

The issue requests that Windows exec tool calls stop flashing visible console windows, likely by direct-spawning simple executable commands instead of always using the PowerShell wrapper.

## Best Possible Solution

Design and land a Windows exec invocation contract that direct-spawns safely parsed simple executable calls with hidden windows while preserving a PowerShell fallback for shell syntax, wrappers, cmdlets, and approval semantics.

## Reproduction Assessment

Yes from source inspection: on Windows, normal non-sandbox host exec deterministically builds a PowerShell `-Command` child process for the command. I did not run live Windows visual proof, so this is source-reproducible rather than directly reproduced here.

## Solution Assessment

Unclear. Direct-spawning simple executable invocations is plausible, but the maintainable fix needs an explicit fallback boundary for shell features, `.cmd` wrappers, PowerShell cmdlets, and approval policy.
