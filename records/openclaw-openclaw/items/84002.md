## Summary

Keep open. Current main does not clearly cover the combined failure: Slack inbound routing still depends on the resolved session/thread context with no obvious cross-session plan recovery when the turn lands in an empty session, and OpenAI Responses replay still preserves encrypted reasoning items while `thinking_signature_invalid` is not classified as replay-invalid recovery state.

## What This Changes

The issue asks OpenClaw to preserve or recover Slack continuation context and recover from invalid OpenAI Responses encrypted reasoning replay instead of leaving the Slack user with a lost-context or stalled turn.

## Best Possible Solution

Split the work into targeted Slack session-recovery and OpenAI Responses replay-recovery fixes, each with regression coverage and a visible user recovery notice when prior state cannot be reused.

## Reproduction Assessment

No high-confidence live reproduction was established. The issue includes detailed sanitized telemetry, and current source inspection shows plausible paths for both the empty-session history result and the `thinking_signature_invalid` replay failure, but the redacted Slack transcript/proxy details are still needed for a precise repro.

## Solution Assessment

Unclear for one combined fix. The best maintainable direction is to split the two symptoms, then add focused recovery behavior for invalid Responses replay state and focused Slack routing/session recovery for continuation phrases that land in an empty session.
