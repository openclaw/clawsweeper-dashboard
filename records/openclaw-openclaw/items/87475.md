## Summary

Current main and the latest release already implement the useful policy split: detailed provider `invalid_request_error` text is still available to internal formatter/diagnostic paths, while user-facing assistant delivery and lifecycle surfaces use a generic schema-rejection message.

## What This Changes

The issue requests filtering noisy provider `invalid_request_error` internals, especially Anthropic thinking-signature path errors, out of user-facing assistant text without losing useful diagnostics elsewhere.

## Best Possible Solution

Keep the current split formatter behavior and route any remaining Anthropic signature creation or recovery failures through https://github.com/openclaw/openclaw/issues/92201.

## Reproduction Assessment

Yes, from source and tests: the old detailed formatter still shows how JSON `invalid_request_error` payloads became raw-ish diagnostics, while current user-facing callers now route through the safe wrapper. I did not run a live Telegram repro because current code and shipped tests already prove the behavior change.

## Solution Assessment

Yes. The current implementation is the maintainable direction because it preserves diagnostic detail for internal/failover paths while using stable generic copy for delivered assistant and lifecycle text.
