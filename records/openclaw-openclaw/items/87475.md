## Summary

Keep open: current main and the latest release still pass JSON-wrapped `invalid_request_error` text through to user-facing error replies, while the related Anthropic thinking recovery only reduces the loop/retry case and does not decide the formatter policy.

## What This Changes

The issue requests filtering noisy provider `invalid_request_error` messages, especially Anthropic thinking-signature API path diagnostics, out of delivered assistant text while preserving useful diagnostics.

## Best Possible Solution

Add a maintainer-approved formatter-level classifier for opaque replay/thinking-signature request rejections, with raw provider text preserved for logs and existing actionable diagnostics kept covered by tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pass a JSON `invalid_request_error` with a `messages.N.content.M` message into `formatAssistantErrorText` and it returns the raw `LLM request rejected: ...` copy. I did not run a live Anthropic/Telegram reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product boundary. A narrow formatter classifier for thinking-signature or replay-invalid API path messages looks maintainable, but broad path-based filtering needs tests to avoid hiding useful diagnostics.
