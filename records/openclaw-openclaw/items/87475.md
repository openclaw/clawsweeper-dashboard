## What This Changes

The issue asks OpenClaw to stop rendering API-internal `invalid_request_error` details such as Anthropic thinking-signature paths as assistant-visible chat text while preserving useful diagnostics elsewhere.

## Best Possible Solution

Keep the shipped split between diagnostic formatting and user-facing delivery, and leave the remaining invalid-signature creation investigation on the canonical signed-thinking issue.

## Reproduction Assessment

No failing current-main reproduction remains for the user-facing leak: current source and tests show structured `invalid_request_error` payloads are genericized before channel reply delivery. I did not run a live Telegram/Anthropic reproduction in this read-only pass.

## Solution Assessment

Yes. The implemented split is the right layer: internal formatting can keep provider diagnostics, while user-facing delivery uses generic copy and raw details remain available through structured observation/log fields.
