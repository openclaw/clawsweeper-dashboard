## Summary

Keep open. Current main still has the overflow diagnostic, bounded auto-compaction retries, memory-promotion preview/apply, and session cleanup primitives, but it does not connect repeated overflow pressure to a deduped promote-then-rotate lifecycle suggestion with dry-run candidates and operator confirmation.

## What This Changes

The issue requests a lifecycle suggestion that nudges operators to run memory promotion and rotate bloated sessions after repeated context overflow.

## Best Possible Solution

Approve a lifecycle design that tracks repeated per-session overflow pressure, emits one deduped structured suggestion with a stable id and candidate count, exposes it in CLI/UI surfaces, and requires explicit confirmation before promotion or rotation.

## Reproduction Assessment

Not applicable as a strict bug reproduction, but the current behavior gap is source-reproducible. The overflow-loop test simulates repeated context overflow, and current-main source inspection shows the requested promote/rotate suggestion is absent.

## Solution Assessment

Unclear until maintainers settle the lifecycle contract. A suggestion-and-confirmation flow is safer than unconditional promotion or rotation, but it should avoid hard-coding memory-core policy into the embedded runner.
