## Summary

Keep open. Current main has ACP runtime backends and tool sandbox backends, but it does not have per-spawn execution placement, resource profiles, Kubernetes worker lifecycle, or a worker protocol for spawned subagents/ACP sessions; this is a broad feature and architecture request rather than cleanup-close material.

## What This Changes

Request adding pluggable execution backend/profile selection for spawned subagents and ACP sessions, including Kubernetes/container/remote worker lifecycle and resource controls.

## Best Possible Solution

Run a maintainer design pass for a versioned execution-placement contract that reuses existing sandbox and ACP extension boundaries, then land a small first milestone behind explicit config and lifecycle/observability tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a report of broken existing behavior. Source inspection confirms current sessions_spawn and ACP paths lack the requested execution profile/worker placement surface.

## Solution Assessment

Unclear as an implementation plan: the direction is plausible, but Kubernetes/resource profiles should not be added as a one-off field without a maintainer-approved contract across subagents, ACP, sandboxing, config, cleanup, and observability.
