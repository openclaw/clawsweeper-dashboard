## Summary

Keep open because current main still has local subagent dispatch, ACP backend selection, and tool sandbox backends, but no per-spawn worker placement/profile/resource contract. The related items are narrower open follow-ups, not a merged canonical implementation.

## What This Changes

Add a generic execution backend/profile layer so `sessions_spawn` and ACP sessions can choose Kubernetes, container, local, or remote worker resources per spawn or config.

## Best Possible Solution

Keep this as an architecture umbrella until maintainers define a versioned execution placement contract, then split implementation across config/schema, `sessions_spawn`, ACP/runtime APIs, worker lifecycle/status, cleanup, docs, and tests.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would move agent execution into remote/containerized workers with workspace, secret, network, quota, and cleanup boundaries.

Concerns:

- **[high] Define remote execution trust boundary:**
  - body: A Kubernetes/container/remote-worker backend needs explicit rules for which session or channel may request a profile, what secrets and workspace data are mounted, and how network egress and cleanup are enforced.
  - confidence: 0.9
- **[medium] Keep worker protocol narrow and auditable:**
  - body: The proposed worker protocol would execute agent turns remotely, so status/log/event streaming, cancellation, and cleanup should be designed as constrained, authenticated, and observable operations before implementation.
  - confidence: 0.84
