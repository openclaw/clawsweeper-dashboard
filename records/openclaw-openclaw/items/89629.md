## Summary

Keep this PR open: the plugin API idea is useful, but the patch only supplies `usageState` on the in-memory dispatcher path while documented routed/durable `reply_payload_sending` executions still receive no usage snapshot. The related accessor PR at https://github.com/openclaw/openclaw/pull/89631 is complementary, not a replacement, and this PR also needs real after-fix behavior proof before merge.

## What This Changes

Adds an optional per-turn `usageState` field to `reply_payload_sending` so plugins can read provider, model, and usage metadata for a reply.

## Best Possible Solution

Thread the usage snapshot through every `reply_payload_sending` execution path, including routed durable delivery and replay, then add focused coverage and real proof from a plugin consuming the field.

## Reproduction Assessment

Yes. Source inspection shows routed replies persist and replay a `replyPayloadSendingHook` without any `usageState` field, while the PR only adds the field to the dispatcher before-deliver call.

## Solution Assessment

No. The added field is a plausible API shape, but the implementation is not the best fix until the same usage snapshot is carried through routed durable delivery or the API is explicitly narrowed.
