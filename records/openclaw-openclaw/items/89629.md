## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope OpenAI limits to the actual auth-profile credential:** `src/infra/provider-usage.limits.ts:31`
  - body: `auth-profile` is only the auth mechanism recorded in current runner metadata, not the credential type. Mapping it to OAuth here lets an OpenAI API-key profile turn resolve `openai` usage and then `loadProviderUsageSummary` may pick an unrelated stored OAuth/token credential, so the reply footer can show ChatGPT/OAuth windows for a turn that used an API key. Thread the selected profile's actual credential type or profile id into this resolver, and add the API-key-auth-profile-with-OAuth-present regression.
  - confidence: 0.91
- **[P2] Do not fall back usageState by session key for runless replies:** `src/auto-reply/dispatch.ts:343`
  - body: `consumeReplyUsageState` tries `runId` and then `sessionKey`, and it does not remove the entry after returning it. Since dispatch passes `opts?.runId` but that option is optional, a later command or non-agent reply on the same session can receive the previous agent turn's model/cost/limits snapshot for up to five minutes. Keep usageState tied to the exact live run, or destructively consume it so runless deliveries cannot inherit stale atoms.
  - confidence: 0.89
- **[P3] Document the new reply hook contract:** `src/plugins/hook-types.ts:546-556`
  - body: This adds plugin-facing `usageState` data to `reply_payload_sending`, including best-effort absence and provider-limit semantics, but `docs/plugins/hooks.md` still only documents the existing payload ordering/trust behavior. Please update the hook docs so plugin authors do not have to infer the new contract from TypeScript comments or the PR body.
  - confidence: 0.78
