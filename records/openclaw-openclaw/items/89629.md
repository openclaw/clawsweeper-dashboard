## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop falling back usageState by session key:** `src/auto-reply/dispatch.ts:343`
  - body: This call passes the session key into `consumeReplyUsageState`, and the new store returns a cached `sk:<sessionKey>` snapshot when `opts?.runId` is absent. Several channel paths can dispatch with reply options that do not carry a run id, so a later command/reply in the same session can inherit the previous turn's model, token, cost, and limit data for up to five minutes. Keep the handoff tied to the exact active run, or introduce a per-dispatch run id holder that is populated by the runner before delivery fires.
  - confidence: 0.9
- **[P1] Scope OpenAI limits to the actual credential profile:** `src/infra/provider-usage.limits.ts:31`
  - body: Mapping every `auth-profile` or missing auth signal to `oauth` still loses the actual profile type. Current OpenClaw can run an OpenAI Platform turn through an API-key auth profile while `requestShaping.authMode` is just `auth-profile`, so this resolver can fetch/cache OpenAI OAuth usage windows and expose them on an API-key turn. Thread the resolved credential mode/profile id, cache by that scope, and add a regression where an API-key OpenAI profile and an OAuth profile both exist.
  - confidence: 0.9
- **[P3] Document the new usageState hook contract:** `src/plugins/hook-types.ts:568-577`
  - body: `usageState` is a new plugin-facing hook payload with deliberately best-effort delivery semantics, but the public hook docs still describe only reply payload mutation and correlation fields. Add the contract and absence cases to the plugin hook docs so external consumers do not treat the field as always present or durable-replay safe.
  - confidence: 0.84
