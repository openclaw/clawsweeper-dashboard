## Summary

This PR should stay open: the new hook contract is useful, but current head still has merge-blocking correlation and auth-profile scoping bugs that can expose stale or wrong usageState atoms to hook consumers.

## What This Changes

The PR adds an optional per-turn usageState object to reply_payload_sending, records model/usage/context/identity/limit atoms from agent-runner, and introduces a cached provider-usage limit resolver with tests.

## Best Possible Solution

Expose usageState only when it is correlated to an exact turn and scope provider limit windows to the exact resolved credential profile/account, otherwise omit limits or usageState and document the optional live-only contract.

Yes for the review findings: source inspection shows sessionKey fallback consumption, authMode-only limit lookup, and ambient provider-auth resolution. I did not run a live replay, but the current head paths are clear enough for focused unit regressions.

## Solution Assessment

No, not yet. The hook boundary is plausible, but the implementation should use exact per-turn correlation and exact credential scoping, with usageState or limits omitted when those facts are unavailable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Stop falling back usageState by session key:** `src/auto-reply/dispatch.ts:364`
  - body: `consumeReplyUsageState` still probes the session key when `opts.runId` is absent. Since the store keeps each same-session snapshot for five minutes, a later delivery without a run id can receive the previous turn's model/auth/usage/limits data. Please correlate usageState only by an exact per-turn key and leave it absent when that key is unavailable.
  - confidence: 0.91
- **[P1] Scope limit windows to the turn's auth profile:** `src/auto-reply/reply/agent-runner.ts:1826-1828`
  - body: This passes only `authMode` into the limits resolver. For an auth-profile OpenAI turn, `authMode` is just the string `auth-profile`; `loadProviderUsageSummary` then resolves ambient configured auth and the new cache is keyed only by `openai`, so another profile/account's subscription windows can be attached to this turn. Pass the resolved auth profile/account into the usage fetch/cache, or omit `limits` when it cannot be scoped.
  - confidence: 0.88
- **[P3] Document the new usageState hook contract:** `src/plugins/hook-types.ts:576-585`
  - body: The PR adds a plugin-facing `reply_payload_sending` field, but public hook docs still describe only payload rewriting and generic correlation fields. Please update the plugin hooks docs with the optional, live-dispatcher-only `usageState` semantics so external plugin authors do not treat it as durable or always present.
  - confidence: 0.78
