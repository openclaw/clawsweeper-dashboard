## Summary

Keep open: the linked symlink write bug is real and the patch targets the right host write/edit path, but it should not merge as written because it canonicalizes the target path without canonicalizing the workspace root, which can break workspace-only writes for users whose workspace path itself is a symlink.

## What This Changes

The PR adds a workspace-only write/edit symlink-resolution helper in `src/agents/pi-tools.read.ts`, adds focused symlink write/edit tests, and updates the changelog for the linked bug.

## Best Possible Solution

Keep the symlink write fix, but compute relative paths against a consistently canonical workspace root and add regression coverage for symlinked workspace roots before merge.

## Reproduction Assessment

Yes, from source inspection: current main passes lexical workspace-relative write paths into `fs-safe`, while `fs-safe@0.2.7` rejects symlink directory components with `lstat()`. I did not run a live tool repro in this read-only review.

## Solution Assessment

No as written. Resolving symlinks before `fs-safe` sees the path is a reasonable direction, but the PR must canonicalize the workspace root and target consistently to avoid rejecting symlinked workspace roots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep root and target canonicalization aligned:** `src/agents/pi-tools.read.ts:895`
  - body: This resolves `absolutePath` through `fs.realpath(dirname(...))` but still compares it against the original lexical `root`. `resolveWorkspaceRoot()` only uses `path.resolve()`, and the upstream write/edit tools pass paths built from that lexical cwd, so a workspace opened through a symlink will now produce a real target path outside the lexical root and throw `Path escapes workspace root` for otherwise valid workspace-only writes/edits. Compute the relative path against the same canonical root used for the resolved target, and cover a symlinked workspace root.
  - confidence: 0.89
