## Summary

Keep open: the latest force-push covers the original host write-through-symlink case, but the edit path still compares a symlinked workspace-root candidate against the real workspace root before fs-safe can resolve it, so existing workspace-only edit setups can fail closed.

## What This Changes

The PR canonicalizes host workspace-only write/edit paths before fs-safe writes and adds symlink write tests plus a changelog entry.

## Best Possible Solution

Keep the symlink write workaround, but normalize root and candidate through the same canonical path for all workspace-only write/edit operations and add edit coverage for symlinked workspace roots.

## Reproduction Assessment

Yes. Source inspection shows current main passes lexical workspace-relative write paths into fs-safe 0.2.7, whose directory guard rejects symlink directory components with the reported error; I did not run a live OpenClaw session in this read-only review.

## Solution Assessment

No as written. Resolving symlinks before fs-safe sees the write path is reasonable, but edit access/readFile must canonicalize the candidate consistently with the root to avoid breaking symlinked workspace roots.
