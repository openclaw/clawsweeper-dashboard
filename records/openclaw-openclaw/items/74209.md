## Summary

Keep open. Current main has narrow Bonjour churn and runtime-dependency mitigations, but the broader default bundled-plugin/minimal-profile startup policy remains unresolved and was explicitly kept open in maintainer discussion.

## What This Changes

The issue asks for upgrades and Gateway startup to avoid default-loading large optional bundled plugin sets, with Bonjour/mDNS and runtime dependency staging kept non-blocking unless configured or explicitly enabled.

## Best Possible Solution

Define and document a durable Gateway startup and upgrade policy that loads only configured, explicitly enabled, or core-required plugins by default, preserves explicit disables and allowlists, keeps LAN discovery and other optional surfaces non-blocking or opt-in where appropriate, and provides a minimal-profile or safe-mode recovery path for operators.

## Reproduction Assessment

Unclear for the full live failure because the affected Debian/npm-global upgrade path was not replayed. The remaining policy surface is reproducible by inspection: current main still default-enables Bonjour and codifies default-enabled bundled startup/runtime-dependency inclusion in source, docs, and tests.

## Solution Assessment

No, the current main state is not the complete solution. The Bonjour restart cap and runtime-deps mitigations are useful narrow fixes, but the reported upgrade/startup problem needs a broader default bundled-plugin and recovery policy.
