## Summary

Keep open. Current main contains the narrow Bonjour flapping guardrail, but the broader reported startup/upgrade policy remains: Bonjour is still default-enabled at startup, bundled default startup sidecars are still admitted, and runtime-dependency planning still includes non-provider default-enabled bundled plugins.

## What This Changes

The issue asks OpenClaw to avoid enabling or initializing broad optional bundled plugins, especially Bonjour, during upgrades/startup when they are unused and can block or destabilize gateway readiness.

## Best Possible Solution

Define a durable startup and upgrade policy that loads only configured, explicitly enabled, or core-required plugins by default; keeps explicit disables, allowlists, and denylists authoritative; makes optional LAN discovery non-blocking or opt-in where appropriate; and documents a minimal-profile or safe-mode recovery path for operators.

## Reproduction Assessment

Unclear for the full live runtime failure because the Debian/npm-global upgrade path was not replayed. Yes for the remaining policy surface: current main source and tests show Bonjour and other default-enabled startup plugins still participate without explicit user configuration.

## Solution Assessment

No, the current narrow Bonjour watchdog mitigation is not the complete solution. The maintainable fix needs a broader startup/upgrade policy rather than another single-plugin loop guard.
