## Summary

Keep open. Current main has meaningful mitigations for Bonjour churn and discovery startup blocking, but it still does not resolve the broader default bundled-plugin/minimal-profile policy raised here; Bonjour remains default-on, empty-config Gateway startup still includes Bonjour and other default startup plugins, and maintainer discussion explicitly kept the broader issue open after the narrow Bonjour fix.

## What This Changes

The issue requests that upgrades and Gateway startup avoid enabling or initializing a broad set of optional bundled plugins, especially Bonjour, in a way that can block readiness or normal operation.

## Best Possible Solution

Define a maintainer-owned startup and upgrade policy that keeps explicitly configured or core-required plugins active, keeps optional discovery/provider/channel plugins non-blocking, preserves explicit disables and allowlists, and provides a documented minimal-profile or recovery mode for constrained hosts.

## Reproduction Assessment

Unclear for the full live failure: the Debian/npm-global upgrade path was not replayed. The remaining policy gap is reproducible by inspection because current main still default-enables Bonjour and still expects Bonjour plus other bundled defaults in the empty-config Gateway startup set.

## Solution Assessment

No, current main is not the complete solution for the reported class. The Bonjour sliding-window cap and discovery startup timeout are useful mitigations, but they do not settle the broader default bundled-plugin and minimal-profile upgrade behavior.
