## Summary

Keep open. Current main has a narrow Bonjour churn guardrail, but the issue’s broader default bundled-plugin/minimal-profile upgrade problem is still present in source, docs, and tests, and a maintainer explicitly kept the issue open for that remaining policy work.

## What This Changes

The issue asks OpenClaw to avoid enabling or initializing broad optional bundled plugins during upgrades in a way that can make gateway startup or normal operation unusable.

## Best Possible Solution

Define and document a durable startup/upgrade policy that loads only configured, explicitly enabled, or core-required plugins by default; keeps explicit disables, denylists, and allowlists authoritative; makes LAN discovery and other optional plugin surfaces non-blocking or opt-in where appropriate; and provides a minimal-profile or safe-mode recovery path for operators.

## Reproduction Assessment

Unclear for the full live failure because the affected Debian/npm-global upgrade path was not replayed. The remaining policy surface is reproducible by static inspection: current main still default-enables Bonjour and codifies default-enabled bundled startup/runtime-dependency inclusion in tests and docs.

## Solution Assessment

No, current main is not the complete solution. The Bonjour sliding-window cap is a useful narrow mitigation, but the reported upgrade/startup problem needs a broader default bundled-plugin and recovery policy.
