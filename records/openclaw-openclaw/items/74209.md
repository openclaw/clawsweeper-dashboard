## Summary

Keep open. Current main has narrow Bonjour churn and runtime-dependency mitigations, but the broader default bundled-plugin/minimal-profile upgrade policy remains unresolved and a maintainer explicitly kept this issue open for that remaining work.

## What This Changes

The issue requests that upgrades stop enabling or initializing a large unused bundled-plugin set, especially Bonjour, in a way that can block gateway startup or normal operation.

## Best Possible Solution

Define and document a durable startup and upgrade policy that loads only configured, explicitly enabled, or core-required plugins by default; keeps explicit disables, denylists, and allowlists authoritative; makes LAN discovery and other optional surfaces non-blocking or opt-in where appropriate; and provides a minimal-profile or safe-mode recovery path for operators.

## Reproduction Assessment

Unclear for the full live failure because the affected Debian/npm-global upgrade path was not replayed. The remaining policy surface is reproducible by inspection: current main still default-enables Bonjour and codifies default-enabled bundled startup/runtime-dependency inclusion in code, docs, and tests.

## Solution Assessment

No, current main is not the complete solution. The Bonjour sliding-window cap is a useful narrow mitigation, but the reported upgrade/startup problem needs a broader default bundled-plugin and recovery policy.
