## Summary

Keep open. Current main contains meaningful mitigations for the Bonjour churn and legacy runtime-dependency staging, but it still does not resolve the broader default bundled-plugin/minimal-profile upgrade policy that the maintainer explicitly left open.

## What This Changes

The issue requests that upgrades and gateway startup avoid enabling or initializing large unused bundled plugin sets, especially Bonjour, in a way that can make the gateway unusable and that OpenClaw provide a minimal/configured-only recovery path.

## Best Possible Solution

Define a maintainer-owned startup and upgrade policy that keeps explicitly configured or core-required plugins active, keeps optional discovery/provider/channel plugins non-blocking, preserves explicit disables and allowlists, and gives operators a documented minimal-profile or safe-mode recovery path for constrained hosts.

## Reproduction Assessment

Unclear for the full live failure: the exact Linux/npm-global upgrade path was not replayed. The remaining policy concern is reproducible by inspection because current main still default-starts Bonjour and still expects multiple bundled defaults in an empty-config Gateway startup plan.

## Solution Assessment

No, current main is not the complete solution for the reported class. The Bonjour watchdog cap, discovery timeout, and dependency-staging cleanup are useful mitigations, but the unresolved default bundled-plugin/minimal-profile behavior needs an explicit product policy decision.
