## Summary

Keep open. Current main has useful Bonjour-specific guardrails, but it still makes Bonjour a default-enabled startup plugin and still preserves default-enabled bundled startup/runtime-dependency policy. The issue discussion also contains maintainer and operator comments that the remaining problem is broader minimal-profile/default-plugin upgrade behavior, not only the Bonjour restart loop.

## What This Changes

The issue asks OpenClaw to prevent unused default bundled plugins, especially Bonjour, from blocking gateway startup after upgrades and to provide a smaller configured-only or safe recovery profile.

## Best Possible Solution

Define a durable upgrade/startup policy that loads only configured, explicitly enabled, or core-required plugins by default; keeps explicit disables, denylists, and allowlists authoritative; makes LAN discovery and optional provider/voice surfaces non-blocking or opt-in where appropriate; and documents a minimal-profile or safe-mode recovery path for operators.

## Reproduction Assessment

Unclear for the full live runtime failure because the Debian/npm-global upgrade path was not replayed. Yes for the remaining policy surface: current main statically shows Bonjour/default startup inclusion and default-enabled runtime-dependency behavior in source, docs, and tests.

## Solution Assessment

No, current main is not the complete solution. The Bonjour sliding-window cap is a good narrow mitigation, but the best fix is a broader startup/upgrade policy for default bundled plugins and recovery behavior.
