## Summary

Keep open. Current main has narrowed the worst Bonjour churn shape and adds discovery timeout handling, but the issue is intentionally broader than that: Bonjour is still default-enabled at startup, current startup planning still admits default-enabled bundled sidecars, and startup still pre-stages runtime dependencies for that startup set. The maintainer discussion explicitly says the broader default bundled-plugin/minimal-profile policy remains unresolved.

## What This Changes

The issue requests an upgrade/startup policy that avoids enabling or loading broad unused bundled plugins by default, keeps Bonjour/mDNS non-blocking, and gives constrained installs a minimal recovery profile.

## Best Possible Solution

Define and document a durable startup/upgrade policy that loads only configured, explicitly enabled, or core-required plugins by default; keeps explicit disables, denylists, and allowlists authoritative; treats LAN discovery and optional voice/provider surfaces as non-blocking or opt-in where appropriate; and provides a minimal-profile or safe-mode recovery path for operators.

## Reproduction Assessment

Unclear for the complete live failure: the Debian/npm-global upgrade path and constrained-host restart churn were not replayed. Yes for the remaining policy surface: current source and tests show default-enabled startup plugins and default-enabled runtime-dependency selection still exist.

## Solution Assessment

No, current main is not the complete solution. The Bonjour sliding-window cap and discovery timeout are good mitigations, but the best fix is a broader, maintainer-approved default bundled-plugin startup and recovery policy.
