## Summary

Keep open: current main and v2026.6.8 still use per-port Unix lsof in the relevant listener/status/restart-health paths, and the only focused linked mitigation is an open PR that explicitly leaves restart-health polling unresolved.

## What This Changes

The issue reports that macOS gateway port-health polling repeatedly spawns per-port lsof checks, which can contribute to launchservicesd saturation and a WindowServer watchdog reset on a busy host.

## Best Possible Solution

Keep this issue as the canonical macOS lsof-pressure follow-up, review/land the focused status batching PR if maintainers accept it, then address restart-health with a bounded batched or non-spawning probe design backed by macOS proof.

## Reproduction Assessment

Partially yes. Current main source-reproduces repeated per-port lsof in Unix listener inspection and restart-health/status callers, but this review did not live-reproduce the disruptive launchservicesd or WindowServer watchdog reset on macOS.

## Solution Assessment

No, this is not solved yet. The linked PR is a reasonable partial status-path mitigation, but the best full fix still needs restart-health lsof pressure reduced or replaced with a bounded non-spawning probe path.
