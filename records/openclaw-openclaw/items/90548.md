## Label Justifications

- `P2`: The report describes a serious macOS gateway availability hazard, but it is conditional on host load and not a broadly reproduced unusable-core-runtime emergency.
- `impact:crash-loop`: The issue concerns gateway health polling that can contribute to macOS watchdog teardown or process/session availability failure under load.

## Summary

Keep open: current `main` and the latest shipped tag still run Unix gateway port diagnostics through per-port `lsof`, and restart health loops can repeat that path while waiting for recovery. The macOS watchdog reboot chain was not locally reproduced, but the source-backed process-launch pressure is real enough to warrant a focused fix.

## What This Changes

The issue requests hardening macOS gateway port-health polling so it does not spawn per-port `lsof` checks on every poll cycle.

## Best Possible Solution

Refactor gateway restart/status port health so ordinary health waits use non-spawning socket reachability or bind probes first, and reserve `lsof` for conflict attribution or a single bounded diagnostic pass with focused macOS regression coverage.

## Reproduction Assessment

Yes for the source-level issue: current restart/status health paths call `inspectPortUsage`, and Unix `inspectPortUsage` invokes per-port `lsof`. No for the full macOS watchdog outcome: that remains reporter-provided and was not locally reproduced.

## Solution Assessment

Yes, the best fix direction is to remove `lsof` from the repeated health path and keep it only for bounded diagnostics; adding a new config knob or only lengthening the loop would leave the process-spawn pressure in place.
