## Summary

Keep open: current main and v2026.6.8 still route Unix listener diagnostics through per-port `lsof`, restart-health still calls that path, and the only focused mitigation is an open PR that explicitly leaves part of this issue unresolved.

## What This Changes

The issue reports that macOS gateway port-health polling repeatedly spawns per-port `lsof` processes, contributing to launchservicesd saturation and a WindowServer watchdog reset.

## Best Possible Solution

Review and land the focused status batching PR if maintainers accept that tradeoff, then keep this issue as the canonical follow-up to remove or tightly bound per-port `lsof` from restart-health polling with macOS proof.

## Reproduction Assessment

Partially yes. Current main source-reproduces repeated per-port `lsof` in Unix port inspection and restart-health callers, but this review did not reproduce the disruptive launchservicesd or WindowServer watchdog reset on macOS.

## Solution Assessment

No, this is not solved yet. The open PR is an acceptable partial mitigation for daemon status, but the best fix must also address the remaining restart-health/per-port lsof loop or replace it with a bounded non-spawning probe path.
