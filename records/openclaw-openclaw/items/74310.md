## Summary

Keep this open. Current main still normalizes `kimi-coding` to `kimi`, the latest release tag still contains that alias, and the supplied GitHub context shows open unmerged PR #74315 with closing syntax, so this issue should remain open until that PR or a replacement fix lands.

## What This Changes

The issue reports that provider normalization collapses the configured `kimi-coding` provider namespace into `kimi`, breaking model lookup for `kimi-coding/k2p5`.

## Best Possible Solution

Review and land PR #74315 if maintainers want `kimi-coding` to remain distinct everywhere, or replace it with an exact-configured-provider-first fix that preserves user-defined `kimi-coding` provider keys while keeping intended `kimi` and `kimi-code` compatibility. The final fix should cover model selection, catalog/status display, persisted overrides, and the no-fallback `/status` case from the report.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` still returns `kimi`, `normalizeModelRef` uses that helper before catalog lookup, and current tests assert the same collapse for normalization and persisted override paths.

## Solution Assessment

Unclear. The issue’s minimal alias removal may be correct if `kimi-coding` should always be distinct, but an exact configured provider lookup before alias fallback is the narrower maintainable option if backward-compatible aliasing is still required.
