## Summary

Keep this open. Current main still maps `kimi-coding` to `kimi`, the selected-model normalization path still uses that canonicalized provider, and open closing PR #74315 is the right place to review the intended Kimi provider contract before this issue is resolved.

## What This Changes

The issue requests preserving `kimi-coding` as a distinct provider namespace for explicit `models.providers` entries instead of normalizing it to `kimi`.

## Best Possible Solution

Review and land or revise PR #74315 so explicit configured provider ids are preserved while legacy `kimi-code` and documented `kimi/kimi-code` compatibility remain covered by focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is enough: `normalizeProviderId("kimi-coding")` returns `kimi`, and `normalizeModelRef("kimi-coding", "k2p5")` therefore produces provider `kimi`; the status selected-model path also lacks `models.providers` context.

## Solution Assessment

Unclear. PR #74315 is a plausible minimal fix, but maintainers should confirm whether `kimi-coding` is a stable user-configured provider namespace or whether a config-aware exact-provider-first resolver is the safer contract.
