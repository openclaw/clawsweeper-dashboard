## Summary

Keep this issue open. Current main still contains the reported `kimi-coding` to `kimi` normalization, the latest shipped tag still contains it, and the issue is already paired with open unmerged PR #74315 that uses closing syntax.

## What This Changes

The issue asks OpenClaw to preserve `kimi-coding` as a distinct configured provider namespace, or otherwise try exact configured provider keys before alias fallback, so `kimi-coding/k2p5` resolves directly without a fallback label.

## Best Possible Solution

Review and land PR #74315 if maintainers want `kimi-coding` to remain an exact configured provider id, or replace it with a narrow exact-key-first resolver that preserves legacy `kimi` and `kimi-code` compatibility while avoiding custom provider namespace collisions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` still returns `kimi`, and both model-ref normalization and configured catalog construction apply that rewrite before lookup.

## Solution Assessment

Unclear. Removing the alias is the minimal proposed fix, but exact configured-provider lookup before alias fallback may be safer unless maintainers confirm `kimi-coding` should be distinct everywhere.
