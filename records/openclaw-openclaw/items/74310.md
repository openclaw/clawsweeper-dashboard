## Summary

Keep this issue open. Current main still collapses `kimi-coding` to `kimi` in provider normalization, the latest shipped tag still contains that behavior, and the issue is already paired with open unmerged PR #74315 that is intended to close it.

## What This Changes

The issue requests that OpenClaw stop unconditionally normalizing the configured `kimi-coding` provider namespace to `kimi` during model resolution.

## Best Possible Solution

Review and land PR #74315 if `kimi-coding` should remain an exact configured provider id, or replace it with a narrow exact-key-first resolver that preserves legacy `kimi` and `kimi-code` compatibility while avoiding custom provider namespace collisions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` still returns `kimi`, and both model-ref parsing and configured catalog construction apply that rewrite before lookup.

## Solution Assessment

Unclear. Removing the alias is the smallest change and matches the report, but exact configured-provider lookup before alias fallback may be safer unless maintainers confirm `kimi-coding` should be distinct everywhere.
