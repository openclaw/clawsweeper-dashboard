## Summary

Keep this open. Current main still normalizes `kimi-coding` to `kimi`, the visible status path can still resolve the selected label without the configured provider catalog, and open PR #74315 is already the candidate implementation path for this issue.

## What This Changes

The issue requests preserving `kimi-coding` as a distinct configured provider namespace instead of rewriting it to `kimi` during model resolution and status display.

## Best Possible Solution

Review and land or replace PR #74315 after deciding the intended Kimi compatibility contract, with focused resolver and `/status` coverage for explicit `models.providers["kimi-coding"]` entries.

## Reproduction Assessment

Yes. Configure `agents.defaults.model.primary` as `kimi-coding/k2p5` with a `models.providers["kimi-coding"]` catalog entry; current source still has generic paths that rewrite the provider to `kimi` before status/catalog matching.

## Solution Assessment

Unclear. The one-line alias removal in PR #74315 is a plausible minimal fix, but an exact-provider-first resolver may be safer if maintainers want to preserve both legacy `kimi-code` aliases and explicit custom provider namespaces.
