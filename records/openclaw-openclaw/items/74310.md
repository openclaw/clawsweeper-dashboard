## Summary

Keep this issue open. Current main still normalizes `kimi-coding` to `kimi`, the configured model catalog still stores configured provider keys after that normalization, and an open unmerged closing PR (#74315) is already the implementation candidate.

## What This Changes

The issue requests that provider-namespaced model refs such as `kimi-coding/k2p5` resolve against a distinct configured `kimi-coding` provider instead of being collapsed to `kimi/k2p5`.

## Best Possible Solution

Land PR #74315 if maintainers want `kimi-coding` to remain a distinct configured provider id everywhere, or replace it with a config-aware resolver that tries exact configured provider keys before alias fallback while keeping intended `kimi` and `kimi-code` compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` still returns `kimi`, `normalizeModelRef` uses that normalized provider, and current tests assert the collapse.

## Solution Assessment

Unclear. The one-line alias removal in the linked PR may be the right minimal fix, but an exact-key-first resolver is safer if maintainers need to preserve legacy `kimi-code` alias behavior outside explicitly configured `kimi-coding` namespaces.
