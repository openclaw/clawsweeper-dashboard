## Summary

Keep this open. Current main still normalizes `kimi-coding` to `kimi`, current tests still encode that collapse, and the issue is paired with open candidate PR #74315 using closing syntax, so the issue should stay open until that PR or a replacement fix lands.

## What This Changes

The report asks OpenClaw to preserve `kimi-coding/...` as a distinct configured provider namespace instead of rewriting it to `kimi/...` during model resolution.

## Best Possible Solution

Review and land PR #74315 if the intended contract is to keep `kimi-coding` distinct, or replace it with an exact-configured-provider-first fix that preserves exact user provider keys while keeping intended `kimi` and `kimi-code` compatibility.

Yes. Source inspection against current main gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` still returns `kimi`, and `normalizeModelRef` plus runtime default handling still use that normalization.

## Solution Assessment

Unclear. Removing the alias is a narrow fix for the reported namespace break, but an exact configured-provider lookup before alias fallback may be safer if legacy `kimi-code` strings must continue to resolve independently of the outer provider key.
