## Summary

Keep open. Current main still collapses `kimi-coding` to `kimi`, the `/status` selected-model path still resolves from an agent-only config that cannot consult the configured `models.providers["kimi-coding"]` entry, and open PR #74315 is already a closing candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks OpenClaw model resolution to preserve a user-configured `kimi-coding/k2p5` provider namespace instead of normalizing it to `kimi/k2p5` and reporting a fallback.

## Best Possible Solution

Preserve exact configured provider ids when users define `models.providers["kimi-coding"]`, keep legacy `kimi-code` compatibility where intended, and land either PR #74315 or a narrow exact-configured-provider-first replacement with focused model-selection and `/status` coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure the default model as `kimi-coding/k2p5`; the current selected-model path can normalize it through `normalizeProviderId` without the full configured provider map, producing `kimi/k2p5` before status/fallback display.

## Solution Assessment

Unclear. The open PR is the minimal fix requested by the reporter, but the safer product contract may be to prefer exact configured provider keys before alias normalization so legacy `kimi-code` behavior and custom namespaces can coexist.
