## Summary

Keep open. Current main still collapses `kimi-coding` to `kimi`, the existing tests assert that behavior, and open closing PR #74315 is the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks for `kimi-coding/k2p5` to resolve against an explicitly configured `kimi-coding` provider instead of being normalized to `kimi/k2p5`.

## Best Possible Solution

Keep this issue open until #74315, or a config-aware replacement, preserves exact `models.providers["kimi-coding"]` lookup while retaining the intended legacy `kimi-code` compatibility and focused status/model-selection coverage.

## Reproduction Assessment

Yes. The source path is high-confidence: `parseModelRef` reaches `normalizeModelRef`, which calls `normalizeProviderId` and rewrites `kimi-coding` to `kimi`; current tests also assert that rewrite.

## Solution Assessment

Unclear. Removing `kimi-coding` from the alias chain is a plausible narrow fix, but an exact-provider-first lookup may be safer if maintainers want legacy `kimi-code` strings to keep mapping to `kimi` outside explicit `kimi-coding` configs.
