## Summary

Keep this open. Current main still collapses `kimi-coding` to `kimi`, and the `/status` selected-model path resolves from agent defaults without the full `models.providers` map, so the reported label/fallback behavior is still plausible on main. The issue is also already paired with open closing PR #74315, so the right cleanup is to review that PR or replace it with the preferred narrow fix, not close the issue first.

## What This Changes

The issue asks OpenClaw to stop collapsing the configured `kimi-coding` provider namespace into `kimi` so `kimi-coding/k2p5` resolves and displays as the exact configured provider/model ref.

## Best Possible Solution

Preserve exact configured provider ids when users explicitly define `models.providers["kimi-coding"]`, while keeping documented `kimi/kimi-code` and legacy `kimi-code` compatibility intact. The shipped behavior should have focused tests for `/status` selected labels and model resolution with a custom `kimi-coding` provider.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure an agent default as `kimi-coding/k2p5`, then `/status` resolves selected model from agent-only config and hits `normalizeProviderId`, which still maps `kimi-coding` to `kimi`.

## Solution Assessment

Unclear. PR #74315 is the minimal one-line fix, but an exact-configured-provider-first resolver may be safer unless maintainers confirm `kimi-coding` should be distinct everywhere despite docs currently presenting `kimi/...` as canonical.
