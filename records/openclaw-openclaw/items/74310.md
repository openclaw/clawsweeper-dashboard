## Summary

Keep this open. Current main still contains the `kimi-coding` to `kimi` alias and status selected-model resolution still lacks the configured provider catalog needed for the reporter's `/status` path; PR #74315 is the open closing candidate, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests preserving `kimi-coding` as a distinct provider namespace when resolving model refs such as `kimi-coding/k2p5`.

## Best Possible Solution

Preserve exact configured provider ids for explicit `models.providers["kimi-coding"]` entries, keep intended legacy `kimi-code` compatibility, and add focused resolver/status tests before closing this issue.

## Reproduction Assessment

Yes. Configure `agents.defaults.model.primary` as `kimi-coding/k2p5` with a `models.providers["kimi-coding"]` catalog entry; current source still has a selected-status path that resolves without that provider catalog and a fallback normalizer that rewrites `kimi-coding` to `kimi`.

## Solution Assessment

Unclear. PR #74315 is a plausible minimal fix, but maintainers should decide whether removing only the alias is enough or whether status/model resolution should consistently prefer exact configured provider ids before normalization.
