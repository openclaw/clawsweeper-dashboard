## Summary

Keep this issue open. Current main still collapses `kimi-coding` to `kimi`, the latest release tag still contains the same alias, and the issue is explicitly paired with open PR #74315, so it should remain open until that PR or a replacement fix lands.

## What This Changes

The issue asks OpenClaw to preserve `kimi-coding` as an exact configured provider namespace so `kimi-coding/k2p5` resolves directly instead of being normalized to `kimi/k2p5`.

## Best Possible Solution

Review and land PR #74315 if maintainers want `kimi-coding` to remain distinct globally, or replace it with an exact-configured-provider-first resolver that preserves explicit `models.providers["kimi-coding"]` while keeping intended `kimi` and `kimi-code` compatibility. The final fix should cover normalization, configured model lookup, persisted/session overrides, status display, and changelog expectations.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` still returns `kimi`, and current tests assert that persisted `overrideProvider: "kimi-coding"` cases resolve to provider `kimi`.

## Solution Assessment

Unclear. The issue’s minimal alias removal is plausible, but the safer maintainable solution may be exact configured-provider lookup before alias fallback so legacy `kimi-code` behavior and explicit `kimi-coding` provider namespaces can coexist deliberately.
