## Summary

Keep open. Current main has partial heartbeat and `NO_REPLY` suppression, but it still lacks the broader custom no-op sentinel or structured suppress-delivery contract requested here, and PR #73785 remains open and explicitly paired with this issue.

## What This Changes

Request runtime-level suppression for cron, heartbeat, and system-event no-op outputs so sentinel replies or structured no-reply results are not delivered to user-visible channels.

## Best Possible Solution

Keep this issue as the canonical tracker until delivery-boundary no-op classification is centralized for cron/system-event output, covering known exact sentinel outputs and leaving a clear path for structured or config-backed suppression when maintainers want more than fixed tokens.

## Reproduction Assessment

Yes for the remaining custom-sentinel gap: current source only recognizes `HEARTBEAT_OK` and `NO_REPLY`, so a delivered cron/system-event payload that is exactly `NO_NEW_AUDIO` or `SESSION_WATCHDOG_OK` has no current suppression contract. The duplicate-output example is unclear without timestamps and run IDs.

## Solution Assessment

Yes for the triage decision: keeping this open while PR #73785 or an equivalent fix is reviewed is the narrowest maintainable path. For the broader product shape, a structured no-reply result is safer long term than continually expanding hardcoded sentinel strings.
