## Summary

Keep open. Current main has several heartbeat and `NO_REPLY` suppression paths, but it still does not implement the broader cron/job-defined no-op contract requested here, and the open closing PR for the narrow sentinel fix is still unmerged and conflicted.

## What This Changes

The issue asks OpenClaw to suppress user-visible delivery for cron/heartbeat no-op results and provide stronger silent/no-reply semantics for background jobs.

## Best Possible Solution

Rebase and land https://github.com/openclaw/openclaw/pull/73785 or an equivalent narrow exact-sentinel fix, then decide separately whether cron should grow a structured `noReply`/`suppressDelivery` contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for exact custom sentinels and relayable exec `HEARTBEAT_OK` leakage; the duplicate-message subcase remains unclear without run identifiers and timestamps.

## Solution Assessment

Unclear for the full request. The open PR is the narrow best path for exact sentinel leaks, but a general cron `noReply`/`suppressDelivery` contract should be a deliberate API decision.
