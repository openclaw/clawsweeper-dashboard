## Summary

Keep open. Current main covers several heartbeat and `NO_REPLY` paths, but the broader cron/job-defined no-op contract and custom sentinel handling requested here are still missing, and #73785 remains the open closing PR for the narrow sentinel suppression work.

## What This Changes

Request stronger cron/heartbeat no-op delivery suppression so successful background jobs do not post visible sentinel or status messages.

## Best Possible Solution

Resolve #73785 or an equivalent narrow sentinel fix first, then track a separate explicit cron no-reply contract if maintainers want job-defined no-op semantics.

## Reproduction Assessment

Yes for the remaining custom sentinel/no structured contract path: current source only recognizes heartbeat acks and `NO_REPLY`, so `NO_NEW_AUDIO` or `SESSION_WATCHDOG_OK` would still be ordinary deliverable text. The duplicate-message subcase remains unclear without run identifiers and timestamps.

## Solution Assessment

Unclear for the final shape. `heartbeat_respond` is the right heartbeat-specific direction and #73785 is the narrow candidate for sentinel leaks, but a general cron `noReply`/`suppressDelivery` contract should be a deliberate API decision.
