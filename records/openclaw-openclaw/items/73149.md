## Summary

Keep open. Current main has narrow heartbeat `HEARTBEAT_OK` suppression and `NO_REPLY` handling, but it does not yet provide the broader no-op sentinel or structured suppress-delivery contract requested here. PR #73785 is open, unmerged, and explicitly paired with this issue, so this tracker should remain open until that PR or an equivalent fix lands.

## What This Changes

Cron and heartbeat no-op jobs should have runtime-level delivery suppression for silent results such as `HEARTBEAT_OK`, `NO_NEW_AUDIO`, and `SESSION_WATCHDOG_OK` instead of posting sentinel strings or status narration.

## Best Possible Solution

Use this issue as the canonical tracker until the delivery-boundary suppression work lands. The maintainable end state is centralized no-op sentinel handling for system-event delivery, with a structured no-reply job result contract considered separately if maintainers want a larger cron API.

## Reproduction Assessment

Yes for the primary heartbeat sentinel leak: the issue discussion gives a concrete delivery-path trace, and current source inspection shows only heartbeat-token and `NO_REPLY` suppression on main. The duplicate-message variant is unclear without timestamps and run IDs.
