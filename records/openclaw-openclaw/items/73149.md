## Summary

Keep open. Current main has narrow heartbeat and NO_REPLY suppression, but it does not yet implement the broader job-defined no-op sentinel or structured suppress-delivery contract requested here, and the provided GitHub context shows an open PR explicitly paired with this issue.

## What This Changes

The issue requests runtime-level suppression of cron/heartbeat no-op responses so sentinel outputs such as HEARTBEAT_OK, NO_NEW_AUDIO, or SESSION_WATCHDOG_OK are not delivered as visible channel messages.

## Best Possible Solution

Keep this issue as the canonical tracker until PR #73785 or an equivalent narrow fix lands. The maintainable end state is centralized delivery-boundary suppression for exact no-op sentinels, with a structured no-reply result flag handled as a follow-up API decision if maintainers want more than sentinel compatibility.
