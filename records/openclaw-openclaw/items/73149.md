## Summary

Keep open. Current main covers several heartbeat and `NO_REPLY` suppression paths, but #73785 is still the open closing PR for the narrow sentinel fix and current cron contracts still lack a general job-defined no-op or `suppressDelivery` result field.

## What This Changes

The issue requests runtime-level silent/no-op delivery semantics for cron and heartbeat jobs so successful no-op sentinel replies are not posted to user channels.

## Best Possible Solution

Review and land #73785 or an equivalent narrow sentinel suppression first, then decide separately whether cron needs a documented `noReply`/`suppressDelivery` contract for job-defined no-op results.

## Reproduction Assessment

Yes for the remaining custom-sentinel path by source inspection: current main only treats `HEARTBEAT_OK` and `NO_REPLY` as suppressible delivery controls, so `NO_NEW_AUDIO` or `SESSION_WATCHDOG_OK` would still be ordinary deliverable text. The duplicate-message subcase is unclear without run identifiers and timestamps.

## Solution Assessment

Unclear for the broader request. Heartbeat-specific `heartbeat_respond` plus #73785's narrow sentinel suppression is the maintainable immediate path, while a general cron no-reply API should be a deliberate contract decision.
