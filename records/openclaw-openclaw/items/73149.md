## Summary

Keep open. Current main has narrow `HEARTBEAT_OK` and `NO_REPLY` suppression paths, but the requested broader cron/system-event no-op contract is not fully implemented, and the provided GitHub context shows open PR #73785 explicitly paired with this issue via closing syntax.

## What This Changes

Request runtime-level silent/no-op delivery semantics for cron, heartbeat, and system-event jobs so no-op sentinels or structured no-reply results are not posted to user-visible channels.

## Best Possible Solution

Keep this issue as the canonical tracker until PR #73785 or an equivalent narrow implementation lands. The maintainable end state is centralized delivery-boundary suppression for exact no-op sentinels, with a structured cron/job no-reply result flag handled as a deliberate follow-up API decision if maintainers want that broader contract.
