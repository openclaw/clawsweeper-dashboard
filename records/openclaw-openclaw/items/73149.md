## Summary

Keep open. Current main has heartbeat-specific quiet handling and isolated-cron `NO_REPLY` suppression, but it still lacks the broader job-defined no-op/silent contract and custom sentinel handling requested here; #73785 remains the open paired implementation candidate with closing syntax.

## What This Changes

Cron and heartbeat no-op runs should suppress successful no-op outputs such as heartbeat acknowledgements and job-defined sentinel strings instead of posting them to user channels.

## Best Possible Solution

Keep this as the canonical tracker until #73785 or an equivalent focused fix lands, then decide whether a narrower follow-up is needed for generic job-defined no-op results.

## Reproduction Assessment

Yes for the primary suppression-gap class: current main still shows heartbeat-token and `NO_REPLY` suppression paths rather than a generic custom no-op result contract. The duplicate-message subcase remains unclear without run identifiers and timestamps.

## Solution Assessment
