## Summary

Keep open. Current main has heartbeat-specific `HEARTBEAT_OK` suppression and shared `NO_REPLY` suppression, but it does not yet implement the broader job-defined no-op sentinel or structured suppress-delivery contract requested here; the open paired PR #73785 is the right place to resolve or replace this work.

## What This Changes

The issue requests runtime-level suppression for cron/heartbeat no-op outputs so sentinel values such as `HEARTBEAT_OK`, `NO_NEW_AUDIO`, or `SESSION_WATCHDOG_OK` are not delivered as visible channel messages.

## Best Possible Solution

Keep this issue as the canonical tracker while PR #73785 is reviewed. The maintainable end state is centralized delivery-boundary suppression for exact no-op sentinels, with a structured no-reply cron/system-event result contract considered separately if maintainers want a broader API.

## Reproduction Assessment

Yes for the remaining sentinel gap by source inspection: `NO_NEW_AUDIO` and `SESSION_WATCHDOG_OK` are not recognized on current main, while `HEARTBEAT_OK` and `NO_REPLY` are. The duplicate-message variant remains unclear without run IDs or gateway logs.
