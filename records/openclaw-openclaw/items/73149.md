## Summary

Keep open. Current main has heartbeat-specific quiet handling and general `NO_REPLY` suppression, but it still lacks the broader cron/job-defined no-op contract requested here, and #73785 remains the open closing PR for this issue.

## What This Changes

Request a first-class cron/heartbeat silent no-op delivery mode so successful no-op sentinel results are not posted to user channels.

## Best Possible Solution

Review, land, replace, or close #73785 for the immediate sentinel leak, then track any remaining structured cron no-op contract separately if needed.

## Reproduction Assessment

Yes for the primary suppression-gap class: the issue comment gives a concrete delivery-path trace, and current main still shows heartbeat/`NO_REPLY` handling rather than job-defined no-op suppression. The duplicate-message subcase is unclear without run identifiers and timestamps.

## Solution Assessment

Unclear for the final implementation. #73785 is the narrow immediate fix path, while `heartbeat_respond` is a good heartbeat-specific seam and broader cron-level `noReply` semantics may need separate maintainer design.
