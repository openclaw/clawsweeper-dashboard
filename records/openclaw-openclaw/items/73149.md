## Summary

Keep open. Current main has heartbeat-specific and NO_REPLY suppression, but it still lacks the broader job-defined no-op sentinel or structured no-reply contract requested here, and PR #73785 is open and explicitly paired with this issue.

## What This Changes

Cron, heartbeat, and system-event runs should be able to suppress successful no-op replies such as HEARTBEAT_OK, NO_NEW_AUDIO, and SESSION_WATCHDOG_OK before channel delivery.

## Best Possible Solution

Keep this as the canonical tracker until PR #73785 or an equivalent focused fix lands, then consider a follow-up structured cron/job no-reply result contract if maintainers want broader custom no-op codes.
