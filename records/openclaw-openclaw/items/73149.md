## Summary

Keep open. Current main has narrow `HEARTBEAT_OK` and `NO_REPLY` suppression paths, but it still does not provide the broader job-defined no-op sentinel or structured suppress-delivery contract requested here, and PR #73785 is open and explicitly paired with this issue.

## What This Changes

The issue requests runtime-level silent/no-op delivery semantics for cron, heartbeat, and system-event jobs so successful no-op sentinel outputs are suppressed instead of posted to user channels.

## Best Possible Solution

Keep this issue as the canonical tracker until PR #73785 or an equivalent narrow fix lands. The maintainable end state is centralized delivery-boundary suppression for exact no-op sentinels, with a structured no-reply result flag treated as a follow-up only if maintainers want a larger cron/job API contract.
