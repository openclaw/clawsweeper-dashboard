## Summary

Keep open. Current main has heartbeat-specific quiet handling and structured `heartbeat_respond`, but it still lacks the broader cron/job-defined no-op contract and custom sentinel handling requested here; the paired implementation PR remains open with closing syntax.

## What This Changes

The issue requests runtime-level suppression for cron, heartbeat, and system-event no-op outputs so sentinel-only or successful no-op runs are not delivered to user channels.

## Best Possible Solution

Review and land, replace, or close PR #73785 first, then split any broader structured cron result contract into a focused follow-up if sentinel suppression is not enough.

## Reproduction Assessment

Yes for the primary suppression-gap class: the issue discussion gives a concrete delivery-path trace, and current main still shows heartbeat-specific rather than job-defined no-op handling. The duplicate-message subcase remains unclear without run identifiers and timestamps.

## Solution Assessment

Unclear for the final implementation: `heartbeat_respond` is a good heartbeat-specific seam, but it does not cover custom cron no-op sentinels. The narrow maintainable path is to review #73785 before opening a separate structured-result design.
