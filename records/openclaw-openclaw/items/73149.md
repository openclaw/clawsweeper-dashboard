## Summary

Keep open. Current main has partial heartbeat and `NO_REPLY` suppression, but it does not fully implement the broader no-op sentinel or structured suppress-delivery contract requested here, and PR #73785 is open with explicit closing syntax for this issue.

## What This Changes

The issue requests runtime-level silence/no-op delivery semantics for cron, heartbeat, and system-event runs so successful no-op sentinel outputs are not posted to user-visible channels.

## Best Possible Solution

Keep this as the canonical tracker until PR #73785 or an equivalent focused fix lands; the desired end state is centralized runtime no-op suppression for known system-event sentinels, with a structured cron/job no-reply result contract as the maintainable follow-up if maintainers want the broader API.
