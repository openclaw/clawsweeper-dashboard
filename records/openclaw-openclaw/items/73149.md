## Summary

Keep open. Current main has heartbeat-specific `HEARTBEAT_OK` suppression and generic `NO_REPLY` handling, but it still lacks the broader job-defined no-op sentinel or structured suppress-delivery contract requested here; PR #73785 remains open and explicitly paired with this issue, so the tracker should stay open until that PR or an equivalent fix lands.

## What This Changes

Add runtime-level suppression for cron/heartbeat no-op job results so sentinel replies such as `HEARTBEAT_OK`, `NO_NEW_AUDIO`, or `SESSION_WATCHDOG_OK` do not reach user-visible channels.

## Best Possible Solution

Keep this issue as the canonical tracker while PR #73785 is reviewed. The maintainable end state is centralized delivery-boundary suppression for exact no-op sentinels, with a structured no-reply job result contract considered separately if maintainers want a larger cron/system-event API.
