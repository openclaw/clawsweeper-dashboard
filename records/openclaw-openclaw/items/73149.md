## Summary

Keep open. Current main now has heartbeat-specific quiet handling and isolated-cron `NO_REPLY` suppression, but the broader cron/job-defined no-op contract requested here is still not present, and the provided context shows #73785 is an open unmerged PR with closing syntax for this issue.

## What This Changes

The issue requests a first-class cron/heartbeat silent no-op delivery contract so successful background runs do not post sentinel tokens or no-op summaries to user channels.

## Best Possible Solution

Keep this as the canonical tracker until #73785 or an equivalent focused fix lands, then split any remaining structured cron no-op contract into a narrower follow-up if needed.

## Reproduction Assessment

Yes for the primary suppression-gap class: the issue comment gives a concrete delivery-path trace, and current main still lacks handling for custom no-op sentinels such as `NO_NEW_AUDIO` and `SESSION_WATCHDOG_OK`. The duplicate-message subcase is unclear without run identifiers and timestamps.

## Solution Assessment

Unclear for the final implementation. `heartbeat_respond` is a good heartbeat-specific solution, while #73785 is the narrow candidate for known sentinel leaks; a broader per-job `noReply` or `suppressDelivery` contract may need separate maintainer design.
