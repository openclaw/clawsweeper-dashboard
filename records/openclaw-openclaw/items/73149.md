## Summary

Keep open. The report remains valid on current main: heartbeat/`NO_REPLY` paths have partial suppression, but channel-side heartbeat normalization still strips `HEARTBEAT_OK` in message mode rather than applying heartbeat ack suppression, custom no-op sentinels are not part of the runtime contract, and #73785 is still the open closing PR for the narrow fix.

## What This Changes

The issue requests runtime-level silent/no-op delivery for cron and heartbeat jobs so sentinel strings, no-op summaries, and duplicate quiet outputs are not posted to user channels.

## Best Possible Solution

Review and land #73785 or an equivalent narrow suppression fix first, then decide separately whether cron needs a documented structured no-reply/no-op result contract for job-defined sentinels.

## Reproduction Assessment

Yes. Current source shows a channel-side heartbeat/system-event reply like a short summary ending in `HEARTBEAT_OK` is normalized in message mode, while custom strings such as `NO_NEW_AUDIO` and `SESSION_WATCHDOG_OK` are not recognized by the cron silent-token path; the duplicate-delivery subcase remains unclear.

## Solution Assessment

Unclear for the full request. #73785 is the narrowest current fix for known sentinel leakage, but a general cron `noReply` or job-defined no-op contract should be a deliberate API decision rather than folded into cleanup automatically.
