## What This Changes

The PR derives and passes embedded `toolsAllow` from effective tool policies, preserves allowlisted plugin tools through profile filtering, adds focused tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the narrow allowlist propagation approach, normalize the group policy context consistently, and preserve existing allow-all semantics before merging.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level and log-backed: current main does not derive `toolsAllow` for embedded direct-lane runs, and profile filtering can remove a plugin tool before later explicit allow policy; the linked issue supplies matching Feishu runtime logs.

## Solution Assessment

Is this the best way to solve the issue? No, not fully: deriving `toolsAllow` from effective explicit policies and preserving plugin tools through profile stages is the right direction, but the branch should normalize `GroupSpace` and preserve allow-all policy semantics first.

## Review Findings

Overall correctness: patch is incorrect

  - body: This new preflight policy lookup passes `params.sessionCtx.GroupSpace` through raw, while the actual embedded run normalizes it before `runEmbeddedPiAgent`.
