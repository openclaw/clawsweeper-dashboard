## What This Changes

The PR derives explicit tool allowlists for embedded auto-reply runs, preserves allowlisted plugin tools through profile filtering, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the narrow allowlist propagation approach, fix the normalization and allow-all edge cases, then merge after focused tests and changed-surface checks pass.

## Reproduction Assessment

Yes, source-level and log-backed: current main does not derive toolsAllow for embedded direct-lane runs and can profile-filter a plugin tool before later explicit allow policy. I did not rerun the live Feishu setup in this read-only review.

## Solution Assessment

No, not fully: deriving toolsAllow from effective explicit policies and preserving plugin tools through profile stages is the right direction, but the branch should normalize GroupSpace and preserve allow-all semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Normalize GroupSpace before resolving toolsAllow:** `src/auto-reply/reply/agent-runner-utils.ts:344`
  - body: This preflight policy lookup passes `params.sessionCtx.GroupSpace` through raw, while the actual embedded run normalizes it before `runEmbeddedPiAgent`. A padded or whitespace-only group space can make group-scoped policy derivation differ from the run's final policy context, so normalize it before resolving `toolsAllow`.
  - confidence: 0.78
- **[P3] Skip preservation for allow-all policies:** `src/agents/tool-policy-pipeline.ts:176-190`
  - body: `allow: []` is an allow-all policy in `isToolAllowedByPolicyName`, but this helper appends preserved plugin names and turns it into a restrictive allowlist. Return early when `policy.allow.length === 0` so allow-all profile/provider policies keep their current semantics.
  - confidence: 0.72
