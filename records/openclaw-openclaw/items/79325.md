## Summary

Keep open: current main still has a source-visible alias resolution path that can rewrite the explicit OpenCode Go candidate to the OpenRouter alias before execution.

## What This Changes

The issue asks OpenClaw to preserve a fully-qualified `opencode-go/deepseek-v4-pro` selection when an OpenRouter DeepSeek V4 Pro route is also configured.

## Best Possible Solution

Use the already parsed explicit provider/model pair as the first primary fallback candidate, keep alias resolution for actual raw alias inputs, and add a regression test plus changelog entry.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure aliases for both DeepSeek refs, call `resolveFallbackCandidates` with provider `opencode-go` and model `deepseek-v4-pro`, and the current code can select the alias-backed OpenRouter ref first. A live provider run was not performed because the needed provider keys were not present.

## Solution Assessment

Yes, the narrow fix is to change model fallback candidate construction so explicit provider-qualified selections are not reinterpreted through a bare model alias. The safer alternative is not a new config knob; it is preserving explicit user intent while keeping alias behavior covered by existing tests.
