## Summary

Keep open: current main has partial protections for nested model IDs when the provider is stored separately, but a model-only persisted value like `deepseek/deepseek-v4-pro` is still parsed as provider `deepseek`, which matches the reported routing failure.

## What This Changes

Preserve OpenRouter as the provider when doctor or runtime session state handles nested native model IDs like `deepseek/deepseek-v4-pro`.

## Best Possible Solution

Keep provider and provider-native model ID separate through session and doctor repair paths, and resolve model-only nested IDs against configured aliases/catalog owners before falling back to first-slash provider parsing.

## Reproduction Assessment

Yes, at source level: a persisted entry with `model: "deepseek/deepseek-v4-pro"` and no `modelProvider` flows through the generic first-slash parser and resolves provider `deepseek`. I did not run the full CLI/OpenRouter live reproduction in this read-only review.

## Solution Assessment

Yes, the requested direction is the right fix: session/doctor model state should preserve the wrapper provider instead of requiring a provider alias workaround. The narrow implementation should be generic catalog or alias ownership resolution, not an OpenRouter-only special case.
