## Summary

Current main preserves OpenRouter as the runtime provider when the persisted or configured model id contains a provider-native slash path, and the fix/proof commit is on main after the latest release rather than shipped in v2026.5.12.

## What This Changes

The issue reports that doctor/session normalization can reparse `openrouter/deepseek/deepseek-v4-pro` as native `deepseek` instead of preserving `provider=openrouter` and `model=deepseek/deepseek-v4-pro`.

## Best Possible Solution

Keep the current main provider/model separation and ship it in the next release; reopen only with a fresh current-main repro showing doctor writes an incorrect provider field.

## Reproduction Assessment

No high-confidence current-main failure path remains: source inspection and current regression tests cover the reported OpenRouter-style provider/model split. I did not run a live OpenRouter credentialed request because this was a read-only review.

## Solution Assessment

Yes. Preserving provider and provider-native model as separate fields in the persisted/runtime selection path is the narrow maintainable fix, and current main does that without adding config or product policy.
