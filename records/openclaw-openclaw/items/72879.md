## Summary

Keep open. Current main still has a native Google replay path that can serialize a same-model Gemini `functionCall` without `thoughtSignature` when the stored tool-call block lacks one, and the issue discussion includes a recent native Gemini 3 thinking/tool-calling reproduction that matches Google's documented 400 behavior.

## What This Changes

The issue reports a native Google Gemini tool-calling regression where replayed `functionCall` parts can lose required `thoughtSignature` metadata after earlier fixes for adjacent signature problems.

## Best Possible Solution

Add a provider-scoped native Google replay repair that preserves real same-model signatures byte-for-byte, recovers signatures when the native response carried them, and handles unsigned transferred current-turn function-call history according to Google's documented Gemini 3 rules. Keep the existing cross-provider real-signature stripping behavior intact and add regression tests around native serialization, replay policy, and the cross-provider boundary.

## Reproduction Assessment

Yes. The issue includes a concrete native `google-generative-ai` Gemini 3 thinking/tool-calling reproduction, and current-main source inspection shows the outbound native serializer can still emit an unsigned `functionCall` when `block.thoughtSignature` is absent.

## Solution Assessment

Yes for keeping this open and routing a focused fix PR. The narrowest maintainable solution is a native Google replay/signature repair plus targeted regression coverage, not another broad transcript sanitizer or reuse of real signatures across provider boundaries.
