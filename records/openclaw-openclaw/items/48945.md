## Summary

Keep open: current main still lacks a Feishu thread-specific mention policy and this PR is a plausible implementation path, but PR head is not merge-ready because the broadcast path reads out-of-scope follow-up state and the public config/docs metadata are incomplete. The overlapping open `topicRequireMention` design also needs maintainer contract choice before either config surface lands.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option, in-memory active-topic tracking, and tests so topic-scoped group thread replies can bypass repeated bot @mentions.

## Best Possible Solution

Land one Feishu-owned topic reply mention policy that preserves group admission and top-level `requireMention`, carries follow-up state through normal and broadcast dispatch, and updates config metadata plus operator docs.

## Reproduction Assessment

Yes, at source level. Current main resolves `requireMention`, passes it into Feishu group sender activation ingress with `ctx.mentionedBot`, and returns before dispatch for unmentioned group messages, so topic-thread follow-ups still hit the same mention gate.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the branch needs the scoped-state fix, generated config metadata/docs, and a maintainer decision against the overlapping `topicRequireMention` design.
