## Summary

Keep this PR open: the Feishu feature is still useful and current main has not implemented `threadFollowUp`, but the current PR head is not mergeable because the broadcast path reads a block-scoped flag outside its scope and the new user-facing config is undocumented.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option, active-topic tracking, and tests so topic-scoped group thread replies can bypass repeated @mentions after the bot was invited.

## Best Possible Solution

Land a repaired Feishu-owned implementation that preserves group admission rules, scopes active topics per account/chat/topic, bounds the cache, carries the bypass state safely through broadcast dispatch, and documents the config surface.

## Reproduction Assessment

Yes, at source level. PR head declares `skipMentionForThread` inside the group block and reads it later in broadcast dispatch, while current main still drops unmentioned Feishu group messages before dispatch.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best mergeable form until the bypass state is hoisted, broadcast regression coverage is added, and the Feishu docs describe `threadFollowUp`.
