## Summary

Keep open: current main still drops unmentioned Feishu group thread replies when `requireMention` is active, and this PR remains the canonical implementation path for that user problem. The PR head is not merge-ready because it has a block-scoped `skipMentionForThread` reference, incomplete public config/discovery coverage for the new setting, and a default Feishu activation-policy change that needs maintainer approval.

## What This Changes

The PR adds Feishu `threadFollowUp` config, active topic tracking, mention-bypass handling for topic-scoped thread follow-ups, and focused Feishu bot tests.

## Best Possible Solution

Land a Feishu-owned implementation after policy approval, rebased onto current main’s ingress path, with the bypass flag scoped safely and public docs/generated config metadata/tests kept aligned.

## Reproduction Assessment

Yes. Current main can be source-reproduced with `groupSessionScope: "group_topic"`, `replyInThread: "enabled"`, and `requireMention: true`: an unmentioned thread reply has `mentionedBot=false`, reaches Feishu sender activation ingress, and returns before dispatch.

## Solution Assessment

No for the current patch. The `threadFollowUp` direction is plausible, but the PR head has a scope error, needs docs/generated metadata, and needs maintainer agreement on the default mention-bypass policy.
