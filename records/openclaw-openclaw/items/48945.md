## Summary

Keep open. Current main does not implement the Feishu thread follow-up bypass, and this PR addresses a real usability gap with live log proof, but the branch is not mergeable because the bypass flag is scoped inside the group block while broadcast dispatch uses it later, and the new config surface is missing generated metadata/docs.

## What This Changes

Adds Feishu `threadFollowUp` config, active topic tracking, mention-gate bypass logic, and bot tests so thread replies in topic-scoped group sessions can avoid repeated @mentions.

## Best Possible Solution

Land a repaired Feishu-owned mention-follow-up implementation that preserves group admission, scopes active topics per account/chat/topic, bounds the cache, carries the bypass state through broadcast dispatch, regenerates config metadata, and documents the operator-facing modes.

## Reproduction Assessment

Yes, at source level. Current main's Feishu group path still returns before dispatch for unmentioned group messages, and PR head has an out-of-scope `skipMentionForThread` read in broadcast active-agent selection.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the best mergeable form until the bypass state is hoisted, generated config metadata/docs are updated, and maintainers reconcile the overlap with https://github.com/openclaw/openclaw/pull/76020.
