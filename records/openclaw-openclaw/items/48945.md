## Summary

Keep open. The feature addresses a real Feishu thread follow-up gap and most earlier review feedback was addressed, but the latest PR head still has a blocking broadcast-path bug: `skipMentionForThread` is declared in the group block and then read later during broadcast dispatch.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option and active-topic tracking so topic-scoped group thread replies can bypass @-mention requirements after the bot has been invited.

## Best Possible Solution

Keep the feature direction, hoist the thread-follow-up bypass state into the outer `handleFeishuMessage` scope, and add focused broadcast coverage before normal maintainer review.

## Reproduction Assessment

Yes. The blocking PR bug is source-reproducible from the latest diff: the variable is block-scoped in the group gate and read later in broadcast dispatch; the original Feishu follow-up gap is also supported by the current mention-gate source path and the PR's live reproduction notes.

## Solution Assessment

No. The proposed feature design is plausible, but the current patch is not the best mergeable form until the bypass state is hoisted and broadcast behavior is covered by a targeted regression test.
