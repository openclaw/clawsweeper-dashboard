## Summary

Keep open. Current main still lacks the Feishu thread-follow-up mention bypass, but this PR head is not merge-ready because broadcast dispatch reads an out-of-scope `skipMentionForThread` binding and the new config surface is not fully documented/generated; a newer overlapping `topicRequireMention` PR also needs maintainer product judgment before choosing the final behavior.

## What This Changes

Adds a Feishu `threadFollowUp` config option, active-topic tracking, mention-bypass dispatch logic, and tests for unmentioned topic-thread follow-ups.

## Best Possible Solution

Choose one Feishu-owned mention-policy model, then land it with broadcast-safe state scope, generated config metadata, public docs, and focused tests for normal, topic, active, and broadcast flows.

## Reproduction Assessment

Yes, at source level. Current main still sends unmentioned Feishu group messages through the same mention gate and has no `threadFollowUp` key; I did not run a live Feishu flow in this read-only review.

## Solution Assessment

No, not as-is. The feature direction is plausible, but this branch needs the broadcast scope fix, generated/docs updates, and maintainer choice between this active-topic design and the newer opt-in `topicRequireMention` design.
