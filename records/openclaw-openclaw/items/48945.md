## Summary

Keep open: this PR targets a real Feishu topic-thread usability gap that is still absent from current main, but the current head has a blocking scoped-variable bug in broadcast dispatch.

## What This Changes

The PR adds Feishu `threadFollowUp` config modes, active-topic tracking, and tests so topic-scoped thread replies can bypass repeated @mentions after the bot is invited.

## Best Possible Solution

Land a Feishu-owned thread-follow-up policy that integrates with the current ingress/mention-gate path, preserves top-level group mention gating, covers broadcast follow-ups, and documents the new config.

## Reproduction Assessment

Yes by source inspection: current main has a Feishu group mention gate that returns before topic-thread follow-up routing, and no current `threadFollowUp` config exists. A live Feishu reproduction would use `groupSessionScope: "group_topic"`, `replyInThread: "enabled"`, and `requireMention: true`, then reply in the thread without @mention.

## Solution Assessment

Partly. A Feishu-owned config is the right boundary, but this PR is not mergeable until the scoped-variable bug is fixed and the implementation is rebased into current main's ingress structure.
