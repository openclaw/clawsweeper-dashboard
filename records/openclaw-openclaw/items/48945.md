## Summary

Keep this PR open: current main still has the Feishu repeated-mention behavior, and the PR has useful live proof and tests, but PR head is not build-safe and the new public config surface is not reflected in docs/generated config surfaces. The overlapping `topicRequireMention` PR is still open, so it is related context rather than a merged superseding fix.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option, in-memory active-topic tracking, and tests so topic-scoped group thread replies can bypass repeated bot @mentions.

## Best Possible Solution

Land one Feishu-owned topic reply mention policy that preserves group admission and top-level `requireMention`, carries follow-up state through normal and broadcast dispatch, and exposes the config consistently in docs and generated metadata.

## Reproduction Assessment

Yes, at source level. Current main resolves Feishu group `requireMention`, passes `ctx.mentionedBot` into ingress, and returns before dispatch for unmentioned group messages, so topic-thread follow-ups still hit the same mention gate.

## Solution Assessment

No, not as-is. The feature direction is plausible, but PR head has a scoped-variable build blocker, missing docs/generated config surfaces, and an unresolved product choice against the overlapping `topicRequireMention` design.
