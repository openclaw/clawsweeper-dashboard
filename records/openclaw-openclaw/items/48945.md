## Summary

Keep open. The user problem is real and current main still routes unmentioned Feishu topic replies through the normal mention gate, but this PR is not merge-ready because the branch has a broadcast-scope blocker, incomplete public config surfaces, and an unresolved policy/config-shape conflict with https://github.com/openclaw/openclaw/pull/76020 and https://github.com/openclaw/openclaw/issues/76010.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option, in-memory active-topic tracking, a mention-gate bypass for topic-scoped thread replies, and focused tests.

## Best Possible Solution

Pick one Feishu topic-thread mention policy, implement it through the existing policy/dispatch path including broadcast, regenerate config metadata and docs, and land the chosen branch with focused Feishu validation.

## Reproduction Assessment

Yes, at source level. Current main sends unmentioned Feishu group messages through the same `requireMention`/sender-activation path with no thread or topic policy input, so topic-thread follow-ups return before dispatch.

## Solution Assessment

No, not as-is. The active-topic approach could solve the usability gap, but the PR has a build-scope blocker, incomplete public config surfaces, and a competing open config design that needs maintainer choice first.
