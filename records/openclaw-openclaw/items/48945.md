## Summary

Keep open. Current main still lacks a thread-specific Feishu mention policy, and this PR contains relevant implementation work plus live verification, but the branch is not merge-ready because it has a build-scope blocker, incomplete public config surfaces, and a competing config design that needs maintainer choice.

## What This Changes

The branch adds a Feishu `threadFollowUp` config option, in-memory active-topic tracking, a mention-gate bypass for topic-scoped thread replies, and focused tests.

## Best Possible Solution

Pick one Feishu-owned topic-thread mention policy, implement it through the existing policy and dispatch path including broadcast, regenerate config metadata/docs, and land with focused Feishu validation.

## Reproduction Assessment

Yes, at source level. Current main parses `rootId`/`threadId` but sends group messages through the same `requireMention` gate without passing thread context, so unmentioned topic-thread follow-ups return before dispatch.

## Solution Assessment

No, not as-is. The active-topic approach could solve the usability gap, but this branch needs a scope fix, generated/docs config updates, and maintainer choice against the competing `topicRequireMention` design.
