## Summary

Keep open for maintainer review. The Feishu topic-thread mention gap is real on current main, but this PR is not mergeable as-is and now competes with a newer open policy/config design, so it needs a maintainer choice rather than cleanup closure.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option, in-memory active-topic tracking, a mention-gate bypass for topic-scoped thread replies, and focused tests.

## Best Possible Solution

Pick one Feishu topic-thread mention policy, implement it through the Feishu-owned policy and dispatch path including broadcast, update generated config metadata/docs/tests, and land either this branch or a replacement.

## Reproduction Assessment

Yes, at source level. Current main has thread/topic context before dispatch, but the Feishu group mention gate still resolves a single `requireMention` value and returns before dispatch for unmentioned topic-thread follow-ups.

## Solution Assessment

No, not as-is. The active-topic approach could solve the usability gap, but the PR has a broadcast-scope blocker, incomplete public config surfaces, and a competing open config design that needs maintainer choice first.
