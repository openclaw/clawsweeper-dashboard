## Summary

Keep this PR open: current main still has the Feishu thread-follow-up gap, and the branch is a plausible implementation path, but it has concrete merge-blocking review findings and overlaps with a separate unmerged config design.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option plus in-memory active-topic tracking and tests so topic-scoped group thread replies can bypass repeated bot @mentions.

## Best Possible Solution

Land one Feishu-owned topic follow-up mention policy that preserves group admission and top-level mention gating, works through current sender activation and broadcast dispatch, and ships schema, generated metadata, docs, and tests together.

## Reproduction Assessment

Yes, at source level. Current main still sends unmentioned Feishu group messages through the same requireMention activation gate and returns before dispatch, with no thread/topic follow-up exception in the policy input.

## Solution Assessment

No, not as-is. The `threadFollowUp` direction is plausible, but the branch needs the scoped-state bug fixed, generated/docs surfaces added, and reconciliation with the overlapping public config design before it is the best maintainable solution.
