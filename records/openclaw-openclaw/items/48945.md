## Summary

Keep open: the Feishu thread follow-up feature is still not on current main, but this branch is not merge-ready because the PR head has a broadcast-scope build bug, a native topic key mismatch, and missing generated/docs config surfaces.

## What This Changes

The PR adds a Feishu `threadFollowUp` config mode with active-topic tracking and tests so topic-scoped thread replies can skip repeated bot @mentions.

## Best Possible Solution

Land one Feishu-owned thread mention policy that preserves group admission and top-level `requireMention`, keys active follow-ups exactly like topic session routing, and updates generated metadata plus operator docs.

## Reproduction Assessment

Yes, at source level. Current main resolves group `requireMention`, passes it into Feishu sender activation ingress, and drops unmentioned group thread replies before dispatch.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the PR needs the scoped-state fix, native topic key alignment, generated metadata/docs, and a maintainer choice against the overlapping `topicRequireMention` design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Hoist follow-up state before broadcast dispatch:** `extensions/feishu/src/bot.ts:1738`
  - body: `skipMentionForThread` is declared inside the group-only block but read later in broadcast active-agent selection. That leaves this identifier out of scope at PR head, so the branch cannot compile once the broadcast path references it; keep the follow-up state in the outer handler scope before using it here.
  - confidence: 0.97
- **[P2] Use the native topic session key for active follow-ups:** `extensions/feishu/src/bot.ts:1157-1159`
  - body: For native Feishu topic groups, current session routing keys topics by `thread_id` before `root_id`, but this new follow-up check always prefers `rootId`. A topic starter can be marked under `omt_*` while its reply is looked up under the root message ID, causing the default `active` mode to keep dropping valid follow-ups in native topic groups.
  - confidence: 0.88
- **[P2] Regenerate config metadata for threadFollowUp:** `extensions/feishu/src/config-schema.ts:185`
  - body: This adds a public Feishu config key, but the generated bundled channel config metadata and config-doc baseline are not updated. As a result `config.schema.lookup`, Control UI, and config drift checks will not see the new field until `pnpm config:channels:gen` and `pnpm config:docs:gen` are reflected.
  - confidence: 0.86
- **[P3] Document the threadFollowUp modes:** `extensions/feishu/src/config-schema.ts:185`
  - body: The new top-level and per-group Feishu option is only described in comments and the PR body. Add it to the Feishu channel docs with the modes, default, topic-scope limits, and safe examples so operators can discover it after release.
  - confidence: 0.83
