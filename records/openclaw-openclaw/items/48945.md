## Summary

Keep this PR open. Current main still requires a bot mention before Feishu topic-thread routing can handle unmentioned group follow-ups, so the requested behavior is not implemented; however, the PR branch has a blocking scope/type defect in broadcast dispatch and the default mention-bypass policy needs maintainer approval plus docs/changelog coverage before merge.

## What This Changes

The PR adds a Feishu `threadFollowUp` config option, in-memory active-topic tracking, and tests so topic-scoped group thread replies can bypass the mention gate after the bot was invited to a topic.

## Best Possible Solution

Keep the PR open, fix the scoped broadcast reference, document and changelog the new Feishu config, and have a maintainer choose the canonical API/default for thread mention policy before landing.

## Reproduction Assessment

Yes. Current main can be checked with `groupSessionScope: "group_topic"`, `replyInThread: "enabled"`, and `requireMention: true`: an unmentioned thread reply reaches the existing mention gate and returns before topic routing can dispatch. The PR branch defect is also source-reproducible because `skipMentionForThread` is referenced outside its block scope.

## Solution Assessment

No for the current patch, though the active-topic model is a plausible product direction. The implementation must first fix the scope bug, then align docs/changelog and the default policy with maintainer intent; an opt-in default may be safer if maintainers want strict backward-compatible mention behavior.
