The PR adds half-block-art detection plus chat/sidebar CSS so UTF-8 QR/block-art code blocks render as tight black-and-white blocks in the web UI.

## Best Possible Solution

Land a retargeted Control UI fix that detects half-block art in the active markdown-it code renderers and shared raw-output preprocessing, with consistent chat/sidebar CSS and focused regression coverage.

Yes, source-level. Current main has no half-block handling, code blocks are emitted through active markdown-it renderer rules, and raw transcript/side-result text reaches the sanitizer without preprocessing; the PR screenshots show the visible before/after web UI behavior.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P2] Move half-block detection into active markdown renderers:** `ui/src/ui/markdown.ts:216-222`
  - body: This adds `.half-block-art` inside `htmlEscapeRenderer.code()`, but current main emits fenced and indented code through `md.renderer.rules.fence` and `md.renderer.rules.code_block`.   - confidence: 0.91

  - body: The new fencing only runs in `formatToolOutputForSidebar()`. Main assistant/tool transcript text and BTW side-result text still go straight to `toSanitizedMarkdownHtml()`, so an unfenced multiline QR reply remains paragraph text and never receives `.half-block-art`.
  - confidence: 0.88
- **[P3] Inherit the sidebar half-block code font size:** `ui/src/styles/chat/sidebar.css:128-134`
  - body: The sidebar override inherits color and line-height but leaves `.sidebar-markdown code { font-size: 0.88em; }` active on the inner `<code>`, shrinking the glyphs below the intended half-block `<pre>` size. Add `font-size: inherit` to keep sidebar rendering consistent.
  - confidence: 0.78
