Keep this PR open: the user-visible QR rendering problem is valid and the visual direction is supported by real screenshots, but the branch still misses current main's active markdown-it renderer and raw transcript paths, so it should not merge as-is.

## What This Changes

The PR adds half-block-art detection plus web chat/sidebar CSS so UTF-8 block-character QR code blocks render as tight black-and-white tiles.

## Best Possible Solution

Land a current-main UI fix that detects half-block art in markdown-it fence/code_block renderers and raw transcript/sidebar paths, preserves QR whitespace, keeps normal code blocks unchanged, and adds focused tests plus browser proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main lacks half-block detection/styling in active code-block renderers, and raw chat/sidebar text still flows through normal markdown sanitization. I did not run a browser repro in this read-only review.

## Solution Assessment

No. The intended visual fix is reasonable, but this branch hooks an obsolete renderer path and only partially handles raw output; the maintainable fix should be wired into current markdown-it renderers and the raw transcript/sidebar paths.

- **[P2] Move detection into the markdown-it code renderers:** `ui/src/ui/markdown.ts:215-219`
  - body: This adds `.half-block-art` from the old `htmlEscapeRenderer.code()` path, but current main renders fenced and indented code through markdown-it `fence` and `code_block` rules.   - confidence: 0.78
