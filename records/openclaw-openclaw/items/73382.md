## Summary

Keep open: current main still has the Bedrock Claude context-window gap, and this PR is a plausible narrow fix, but it should not merge as-is because the new family matcher misses real Claude 4 Bedrock ID shapes and external real behavior proof is absent.

## What This Changes

The PR backfills Bedrock Claude 3/3.5/Opus 4 context-window rows, adds a Claude family fallback in discovery, covers it with tests, and adds a changelog entry.

## Best Possible Solution

Repair the Bedrock plugin fallback to cover documented Claude 4 ID shapes with regression tests, or narrow the fallback claim to Claude 3.x, then add redacted real Bedrock proof before landing it as the canonical fix for https://github.com/openclaw/openclaw/issues/73328.

## Reproduction Assessment

Yes at source level. Current main lacks the reported known-window entries and falls back to the 32000 discovery default, and a read-only regex check shows this PR's Claude 4 fallback miss; I did not run a live Bedrock gateway.

## Solution Assessment

No, not as-is. The explicit table backfill is a narrow fit, but the advertised Claude 4 family fallback needs to match real Bedrock ID shapes or be removed from the claimed scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match real Claude 4 Bedrock IDs in the fallback:** `extensions/amazon-bedrock/discovery.ts:134`
  - body: The new fallback claims to cover Claude 4 variants, but `^anthropic\.claude-(?:3|4)` only matches IDs beginning `anthropic.claude-4`. Bedrock Claude 4 IDs use `anthropic.claude-sonnet-4...`, `anthropic.claude-opus-4...`, and `anthropic.claude-haiku-4...`, so an unlisted Claude 4 profile would still inherit the default context window. Widen the matcher and add a Claude 4 fallback test, or scope the fallback to Claude 3.x only.
  - confidence: 0.9
