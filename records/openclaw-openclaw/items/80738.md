## Summary

Keep open: the latest head still drops `forceRefresh` on the normal OAuth resolver path, so a persisted unexpired Codex OAuth credential can bypass the intended locked refresh path; the supplied proof is source-regex output rather than a real forced-refresh run.

## What This Changes

This PR removes the Codex app-server pre-refresh expired credential write, adds `forceRefresh` OAuth plumbing, and adds regression coverage for forced Codex ChatGPT token refresh.

## Best Possible Solution

Thread `forceRefresh` through the normal exported OAuth resolver call into the manager, keep focused regression coverage on the persisted unexpired Codex credential path, and add redacted runtime output from a real forced token refresh.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use a persisted `openai-codex` OAuth credential whose expiry is still in the future, invoke the Codex app-server forced refresh path, and verify whether the provider refresh actually runs.

## Solution Assessment

No. Removing the stale pre-refresh write is the right direction, but the PR must pass `forceRefresh` into the main `resolveApiKeyForProfile` OAuth-manager call, not only into fallback resolution.
