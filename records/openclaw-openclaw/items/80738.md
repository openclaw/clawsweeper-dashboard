## Summary

Keep open: the PR targets a real source-reproducible Codex auth bug, but the proposed in-memory expiry copy does not actually preserve forced refresh with the production OAuth resolver because that resolver reloads the persisted store under lock before refreshing. The PR also lacks after-fix real behavior proof.

## What This Changes

The PR changes Codex app-server forced OAuth refresh to pass an in-memory expired auth-profile copy instead of saving `expires: 0` before refresh, and adds a regression test.

## Best Possible Solution

Add a first-class locked force-refresh path through the auth profile resolver or OAuth manager so Codex app-server can refresh persisted OAuth without a temporary stale disk write, then cover the production reload behavior and provide redacted real-run proof.

## Reproduction Assessment

Yes. Current main source shows the forced Codex app-server refresh path writes `expires: 0` to `auth-profiles.json` before refresh, and the PR's proposed test shape can observe that disk write without needing broad setup.

## Solution Assessment

No. Avoiding the pre-refresh disk write is the right direction, but passing only an in-memory expired copy is not sufficient because the production OAuth refresh path reloads the persisted store under lock and can return the old usable credential.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve forced refresh after the store reload:** `extensions/codex/src/app-server/auth-bridge.ts:496`
  - body: Passing `refreshStore` here only expires the credential in memory. The real OAuth manager then enters the refresh path, reloads `auth-profiles.json` under the lock, and returns early if the persisted credential is still usable, so a forced app-server refresh with an unexpired stored Codex OAuth profile returns the old token and never calls the provider refresh. The test mock refreshes from `params.store`, so it misses this production behavior; this needs a real force-refresh path through the locked OAuth resolver instead.
  - confidence: 0.94
