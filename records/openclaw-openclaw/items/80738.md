## Summary

Keep open: the updated PR still has a blocking forced-refresh bug because the new flag is not forwarded through the normal exported OAuth resolution path, and the supplied proof is source-pattern output rather than a real forced Codex token refresh.

## What This Changes

The PR removes the Codex app-server pre-refresh expired credential write and adds a forceRefresh flag intended to force refresh through the locked OAuth manager.

## Best Possible Solution

Thread `forceRefresh` through the exported resolver's normal OAuth branch into the locked manager, add regression coverage that exercises the real manager path with an unexpired persisted Codex OAuth credential, and provide redacted runtime proof of a forced refresh.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use a persisted `openai-codex` OAuth credential whose expiry is more than five minutes away, then call the Codex app-server forced refresh path and observe whether the provider refresh is actually forced.

## Solution Assessment

No. The intended lock-path direction is right, but this patch forwards the flag only to a fallback helper while the normal OAuth branch used by the Codex bridge still drops it.
