## Summary

Keep open: current PR head still does not forward `forceRefresh` through the exported resolver's normal OAuth branch, so forced Codex refresh can still return an unexpired persisted credential; the supplied proof is also source inspection rather than a real refresh run.

## What This Changes

The PR removes the Codex app-server pre-refresh `expires: 0` write, threads a `forceRefresh` flag into OAuth refresh plumbing, and adds bridge regression coverage.

## Best Possible Solution

Thread `forceRefresh` through the exported resolver's normal OAuth-manager call, cover the real manager path with an unexpired persisted Codex OAuth credential, and add redacted runtime proof of a forced refresh.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use a persisted `openai-codex` OAuth credential whose expiry is still usable, then call the Codex app-server forced refresh path and observe whether the provider refresh is actually forced.

## Solution Assessment

No. The lock-path direction is appropriate, but the current patch still drops `forceRefresh` in the exported resolver path used by the Codex bridge, so it is not yet the narrow complete fix.
