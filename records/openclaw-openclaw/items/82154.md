## Summary

Keep open: the report is fresh and plausibly points at an auth/env propagation gap, but the current evidence does not isolate whether this is native subagent, ACP runtime, CLI backend, sandbox, or an unsupported provider env-var name.

## What This Changes

The issue reports that spawned subagents do not see environment variables available to the gateway process, causing provider API-key auth to fail in subagent runs.

## Best Possible Solution

First capture a focused repro that names the runtime and command path; if the bug is real, fix the narrow env/credential propagation path while preserving the existing secret-sanitization boundary.

## Reproduction Assessment

No; the issue describes an observed failure but lacks the exact current-main reproduction path and runtime details needed to distinguish env inheritance from provider env-name or backend clearing behavior.

## Solution Assessment

Unclear; broad env copying is not the safest default for secret-bearing gateway processes, so a narrow credential/profile or allowlisted env propagation fix would be preferable if reproduction confirms a real gap.
