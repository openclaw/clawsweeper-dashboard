## Summary

Keep open: the documented `/v1/responses` contract says a stable `user` value should share an agent session, and current main still only proves stable key derivation, not end-to-end transcript continuity. The open same-key race work at https://github.com/openclaw/openclaw/issues/84575 and https://github.com/openclaw/openclaw/pull/85404 may cover an in-flight subcase, but this report does not yet prove that is the whole failure.

## What This Changes

The issue reports that repeated `/v1/responses` calls with the same `user` value start independent conversations instead of retaining context.

## Best Possible Solution

Add a focused live or regression check for two `/v1/responses` calls using the same `user`; if the failure is the same in-flight race, land or adapt https://github.com/openclaw/openclaw/pull/85404, otherwise fix the narrower transcript-continuity boundary it reveals.

## Reproduction Assessment

No. The issue gives a plausible two-call HTTP path, but I did not run a live Gateway/model reproduction and current source inspection only proves stable session-key derivation, not the reported context loss.

## Solution Assessment

Unclear. The best fix depends on whether a failing current-main repro points to the open same-key new-session race, transcript persistence/session freshness, or another OpenResponses runtime replay boundary.
