## Summary

Keep open. Current main has meaningful WhatsApp reconnect and group recovery work, but the latest reporter update says v2026.4.29 still hits the same 408/428 reconnect cycle with severe liveness stalls, and current source still has observable no-listener windows plus incomplete coverage for the reported libsignal Bad MAC shape.

## What This Changes

The issue reports repeated WhatsApp Web listener disconnects or stalls that leave OpenClaw unable to receive or send WhatsApp messages, especially with `No active WhatsApp Web listener` errors after reconnect churn.

## Best Possible Solution

Keep this as the umbrella WhatsApp listener-stall regression until a maintainer validates current main with a linked WhatsApp account and a stability bundle, then split remaining behavior into narrow fixes for reconnect no-listener windows, libsignal Bad MAC recovery, group hydration loss, and any confirmed Gateway event-loop/provider-auth starvation.

Do we have a high-confidence way to reproduce the issue? Partially: source inspection reproduces a no-listener interval during reconnect backoff, and the issue has fresh real-world logs on v2026.4.29. The full live stall requires a linked WhatsApp account plus the reporter’s Windows/Z.AI or Linux/Docker setup against current main.

## Solution Assessment

Is this the best way to solve the issue? No for auto-close or a single autonomous repair. The merged WhatsApp fixes are the right narrow direction for known subproblems, but the newest logs add event-loop/provider timing evidence that needs live maintainer triage before choosing the next patch boundary.
