## Summary

Keep open. Current main and the latest shipped release contain useful WhatsApp reconnect, group recovery, and liveness improvements, but the issue discussion now includes a fresh 2026.4.29 reproduction and current source still has observable reconnect windows where outbound sends can see no active listener. The report is recent, cross-platform, and not proven resolved by the merged partial fixes.

## What This Changes

The issue reports a WhatsApp listener regression causing repeated disconnect/stall loops, missed inbound or group messages, and outbound `No active WhatsApp Web listener` failures on Windows with later Linux/Docker confirmation.

## Best Possible Solution

Keep this as the umbrella WhatsApp listener-stall regression until a maintainer validates current main with a real linked account, then split remaining confirmed behavior into focused WhatsApp or Gateway issues for outbound reconnect delivery, libsignal/Bad MAC recovery, group hydration/inbound loss, and event-loop saturation if those are distinct roots.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the no-listener outbound window is source-reproducible from current main, and the reporter reproduced similar 408/428 churn on 2026.4.29. The full live stall plus Bad MAC cycle still needs linked WhatsApp validation against current main.

## Solution Assessment

Is this the best way to solve the issue? No for closure: the merged fixes are good narrow repairs for related subproblems, but they do not prove the full umbrella report is resolved. The safer path is live validation followed by narrower fixes for any remaining behavior.
