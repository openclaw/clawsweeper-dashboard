## Summary

Keep open: current main and the latest release still have a no-entry/no-text fallback path for memory-core dreaming after session deletion, and the direct recovery PR remains open, conflicting, and not yet proof-ready.

## What This Changes

The issue asks OpenClaw to preserve the actual generated memory-core dream narrative text when subagent session cleanup archives/removes the session before the plugin reads messages.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land either a run/session-scoped recovery with archived-session regression proof or a core session-lifecycle contract that guarantees plugin readback after `waitForRun`.

## Reproduction Assessment

Yes, at source level. If `sessions.delete` removes the subagent session entry before memory-core calls `getSessionMessages`, current `sessions.get` has no sessionId and returns an empty message list, which drives the existing no-text fallback path.

## Solution Assessment

No, closing is not the best solution yet. Current main has only a short settled-read retry, while the remaining fix needs a maintainer-approved lifecycle or run/session-scoped recovery design plus proof.
