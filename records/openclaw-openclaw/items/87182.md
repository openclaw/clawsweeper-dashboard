## Summary

Keep open. Current main still has a path where memory-core treats an empty post-run session read as no generated narrative and writes only the fallback diary entry; #85821 improved the symptom but does not prove the model-generated assistant text is preserved when the session entry is gone.

## What This Changes

The issue reports that memory-core dreaming can lose generated narrative text when session cleanup archives the narrative subagent transcript before the host plugin reads it.

## Best Possible Solution

Preserve the documented subagent run -> wait -> getSessionMessages -> delete flow so generated assistant narrative remains readable until host extraction finishes, with memory-core regression coverage and fallback only after recoverable model output is exhausted.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The reporter provided a detailed timestamped trace, and current source shows that an emptied/deleted session read after waitForRun would take the fallback path and lose the generated text.

## Solution Assessment

Keeping this issue open is the right outcome. The best fix should preserve or recover the generated assistant text, not rely on another snippet fallback; a new plugin API option should be avoided unless maintainers choose that broader contract change.
