- `impact:data-loss`: The reported failure silently replaces generated narrative text with fallback diary content.

## Summary

Keep open: current main and the latest release only include the bounded post-run message-read retry, while the reported archived-session case can still return an empty message list and fall back to snippet diary text; the linked closing PR remains open and not land-ready.

## What This Changes

The issue reports memory-core losing generated dreaming narrative text when post-completion session cleanup archives the transcript before the plugin reads messages.

## Best Possible Solution

Preserve the documented subagent run, wait, read, delete flow by landing or replacing the linked PR with run/session-scoped recovery or a lifecycle fix that keeps generated narrative text readable until host extraction completes.

## Reproduction Assessment

No live current-main cron repro was run, but yes at source level: if the session entry has been deleted before memory-core calls getSessionMessages, current sessions.get returns [] and memory-core writes the no-text fallback.

## Solution Assessment

Unclear whether the current trajectory-recovery PR is the best final fix. The best solution should either preserve the documented session-read contract or use run/session-scoped recovery with regression proof, rather than guessing from stale session artifacts.
