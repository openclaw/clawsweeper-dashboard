## Summary

Keep open: current main and the latest release still only retry settled session-message reads before falling back, while the open closing PR is conflicting and already flagged as not land-ready; recent reporter comments also corroborate the archived/readback race on v2026.6.5.

## What This Changes

The issue reports that memory-core dreaming can lose generated narrative text when session cleanup archives the subagent transcript before the host plugin reads it.

## Best Possible Solution

Preserve the documented subagent run, wait, read, delete contract by landing or replacing the linked PR with run/session-scoped recovery or a lifecycle fix, plus archived-session regression coverage and redacted real cron proof.

## Reproduction Assessment

Yes at source level: if the session entry has been deleted before memory-core calls getSessionMessages, current sessions.get returns an empty message list and memory-core writes the no-text fallback. I did not run a live current-main dreaming cron reproduction.

## Solution Assessment

Unclear: trajectory recovery is a plausible mitigation, but the active PR is not yet the best fix until it uses run/session identity, preserves settled reads, and proves the exact branch; maintainers may also prefer a lifecycle contract fix.
