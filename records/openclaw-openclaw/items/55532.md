## What This Changes

The issue asks for Feishu WebSocket reconnects to use exponential backoff, avoid the PingInterval closed-connection crash/race, and optionally health-check before reconnecting after Feishu system-busy errors.

## Best Possible Solution

Resolve the canonical Feishu reconnect PR first, preferably with a Feishu-owned wrapper or supervisor that avoids global `WSClient` prototype mutation, cancels pending waits on shutdown, resets backoff after recovery, and has focused tests for SDK retry exhaustion and PingInterval handling. Split the token-cache poisoning concern into an upstream/dependency follow-up if it is not fixed locally.
