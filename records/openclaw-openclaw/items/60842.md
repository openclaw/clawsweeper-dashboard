## What This Changes

The PR forwards embedded-run `toolsAllow` into core tool construction as `forceAllowTools`, merges requested core tool names into restrictive allow policies, and adds regression coverage for re-exposure plus deny precedence.

## Best Possible Solution

Rebase or replace this with a current-main patch that preserves explicitly requested known core tools before profile/global/agent config allowlists drop them, while keeping explicit deny, sandbox, sender/group, subagent, owner-only, provider, and runtime gates authoritative.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes technically: a pre-pipeline force-allow for explicitly requested known core tools is the narrow fix, but it should be reviewed as a sensitive policy-precedence change and rebased onto current main.
