## Summary

Keep open; current main still has the source-reproducible bug and this PR is the linked implementation candidate, but it is conflicting against current main and lacks required real behavior proof for a sensitive tool-policy change.

## What This Changes

The PR threads embedded-run `toolsAllow` into tool construction as `forceAllowTools`, merges named core tools into restrictive allow policies, and adds regression coverage for re-exposure and deny precedence.

## Best Possible Solution

Land or replace this with a rebased patch that preserves explicitly requested known core tools before restrictive config allowlists remove them, while keeping explicit deny, sandbox, subagent, provider, owner-only, and runtime gates authoritative.

## Reproduction Assessment

Yes. Source inspection on current main shows `toolsAllow` is mapped to `runtimeToolAllowlist`, then config policies run through a downward-only pipeline before final runtime allow filtering, while `group:openclaw` excludes `exec`, `read`, and `write`.

## Solution Assessment

Mostly yes, but not merge-ready. A pre-policy force-allow for normalized known core tool ids is the narrow technical fix direction, but this branch needs rebase, current validation, real behavior proof, and maintainer approval for the security-sensitive precedence change.
