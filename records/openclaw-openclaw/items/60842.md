## Summary

Keep open. Current main still has the source-reproducible bug, and this PR remains the linked implementation candidate, but it needs contributor real behavior proof, current validation, and maintainer review of the security-sensitive tool-policy precedence change before merge.

## What This Changes

The PR forwards embedded-run `toolsAllow` into tool construction and adds a `forceAllowTools` policy merge so requested known core tools survive restrictive config allowlists while explicit deny remains authoritative.

## Best Possible Solution

Land or replace this with a current-main patch that preserves explicitly requested known core tools before restrictive config allowlists drop them, while keeping explicit deny, sandbox, group/sender, subagent, owner-only, provider, and runtime gates authoritative.

## Reproduction Assessment

Yes. Source inspection on current main shows `toolsAllow` becomes `runtimeToolAllowlist`, `group:openclaw` excludes `exec`/`read`/`write`, and the policy pipeline filters downward before the final runtime allow filter runs.

## Solution Assessment

Mostly yes technically, but not merge-ready. A pre-policy force-allow for normalized known core tool ids is the narrow fix direction, but the exact precedence change needs maintainer approval plus current validation and real behavior proof.
