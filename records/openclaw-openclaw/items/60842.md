## Summary

Keep this PR open. Current main still has the reported embedded-runner tool-policy gap, and the linked bug remains open; the proposed patch looks directionally correct, but it changes a command and filesystem tool exposure boundary and needs maintainer review on a rebased branch.

## What This Changes

The PR threads embedded-runner `toolsAllow` into `createOpenClawCodingTools` as `forceAllowTools`, merges known core tool IDs into restrictive allow policies while preserving deny semantics, and adds regression tests for re-exposure and deny precedence.

## Best Possible Solution

Rebase or rework the fix on current main so embedded runs can request known core tools through the policy pipeline before the final `toolsAllow` post-filter, while preserving deny, sandbox, group, subagent, owner-only, provider, and runtime capability gates. Land it with the linked bug and keep the focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: configure `tools.profile: "full"` with `tools.allow: ["group:openclaw"]`, run an embedded attempt with `toolsAllow: ["exec"]`, and current main can remove `exec` before the post-filter at `src/agents/pi-embedded-runner/run/attempt.ts:842`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer policy review: the PR is narrow and deny-preserving, but it deliberately changes a command/data-access policy boundary, so the safest path is a rebased version reviewed with that exception in mind.
