## Summary

Keep open. Current main still constructs core tools before applying the embedded-run `toolsAllow` post-filter, and this PR remains the open implementation candidate for the linked bug; it is not old enough for mostly-implemented cleanup and the central behavior is not on `main`.

## What This Changes

The PR forwards embedded-run `toolsAllow` into core tool construction as a constrained force-allow for known core tools and adds regression coverage for restrictive allowlists and deny precedence.

## Best Possible Solution

Land or replace a rebased version that preserves named runtime core tools before config allowlists drop them, while keeping explicit deny, sandbox, sender, group, subagent, owner-only, provider, and runtime gates authoritative.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: configure `tools.profile: "full"` with `tools.allow: ["group:openclaw"]`, run an embedded attempt with `toolsAllow: ["exec"]`, and current main can filter `exec` out before the final runtime allowlist.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer policy review; the patch is technically narrow, but it intentionally creates a per-run exception to restrictive allowlists for sensitive core tools.
