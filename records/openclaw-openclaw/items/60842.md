## What This Changes

The PR threads embedded-run `toolsAllow` into tool construction as `forceAllowTools`, merges explicitly requested known core tools into restrictive allow policies, and adds regression coverage.

## Best Possible Solution

Rebase or replace this with a current-main patch that preserves explicitly requested known core tools before config allowlists drop them, while keeping explicit deny, sandbox, sender/group, subagent, owner-only, provider, and runtime gates authoritative.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection on current main: `toolsAllow` becomes `runtimeToolAllowlist`, `group:openclaw` excludes `exec`/`read`/`write`, and the policy pipeline filters downward before the final allow filter runs.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes technically: a pre-policy force-allow for normalized known core tool ids is narrow, but the exact current-main patch needs maintainer policy review because it changes allowlist precedence for command/file tools.
