## Summary

Keep open: this targets a real MCP HTTP fetch hardening problem, but the patch is not correct. The proposed helper still fully materializes body-null responses through `arrayBuffer()` before enforcing the 16 MiB limit, and it also changes the existing text-only Response-like compatibility path to require `arrayBuffer()`.

## What This Changes

The PR replaces the MCP HTTP fetch text fallback with `readProviderTextResponse` and adds tests for body-null foreign responses and oversized fallback bodies.

## Best Possible Solution

Keep the PR open and replace the body-null fallback with a design that does not read the full response before enforcing the cap, while preserving or explicitly deciding the text-only MCP Response-like compatibility contract.

## Reproduction Assessment

Yes, from source inspection: on PR head the body-null fallback calls `readProviderTextResponse`, and the shared helper calls `res.arrayBuffer()` before checking the cap when no stream exists. I did not run tests because this was a read-only review.

## Solution Assessment

No. This is not the best fix because it moves the fallback to a helper whose body-null path still buffers the full response and changes the compatibility contract for text-only Response-like objects.
