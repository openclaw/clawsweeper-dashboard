## Summary

Keep open. The PR addresses a valid timeout/abort leakage path, but it should not merge as-is because it treats every aborted assistant turn as an error-surfaced timeout and would suppress partial output for user-initiated external aborts that current main explicitly preserves.

## What This Changes

The PR changes embedded run payload assembly to surface aborted assistant turns as formatted errors, suppress aborted answer/reasoning text, and add regression tests for timeout leakage of partial text, reply directives, and reasoning.

## Best Possible Solution

Keep the abort leak fix, but gate it on the timeout/provider-abort path rather than raw `stopReason: "aborted"`. Thread an explicit abort-source or timeout-only suppression flag into `buildEmbeddedRunPayloads`, preserve user-stop partial output, and add regression coverage for both timeout suppression and external abort preservation.

## Reproduction Assessment

Yes. Current main can be reproduced statically by calling `buildEmbeddedRunPayloads` with non-empty `assistantTexts`, `lastAssistant.stopReason: "aborted"`, and a timeout-like `errorMessage`; the PR's new regression tests cover that leak. The PR regression is also reproducible by combining the existing external-abort continue-normal contract with the patched payload builder, which has no `externalAbort` input and suppresses all aborted assistant text.

## Solution Assessment

No as written. The narrow maintainable fix is to suppress aborted assistant artifacts only when the runner knows the abort is a timeout/error delivery path, while leaving externally aborted user-stop turns on the existing partial-output path.
