## Summary

Keep open. Current main still handles only WhatsApp Web status 440 as non-retryable and rethrows opening-phase non-auth errors, while #75773 is an open closing PR for the same WhatsApp 428 fix.

## What This Changes

The issue reports a regression where the WhatsApp channel exits with HTTP 428 after upgrading to 2026.4.29 and needs channel-owned handling instead of a generic exit or retry loop.

## Best Possible Solution

Review and land or replace #75773 with WhatsApp-owned 428 terminal handling for both handshake and post-open close paths, preserving 401/440 behavior and adding regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: inject a Boom-like status 428 from `waitForWaConnection` or from the listener close path, and current main either rethrows it or treats it as retryable. The exact live scheduled recurrence was not reproduced here.

## Solution Assessment

Yes for the linked PR direction: the narrow maintainable fix belongs in the WhatsApp monitor/controller and should cover both handshake and listener-close 428s. Current main is not yet the solution because it still lacks that handling.
