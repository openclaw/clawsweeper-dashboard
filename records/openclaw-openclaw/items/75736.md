## Summary

Keep open. Current main still has a WhatsApp-specific reconnect gap: a Baileys/Boom 428 close during the initial socket-open wait is closed and rethrown before the monitor reaches the close-decision retry path, so it can surface as the generic gateway `channel exited` log described in the report.

## What This Changes

The issue reports that after upgrading to v2026.4.29 the WhatsApp channel exits around 04:00 UTC with HTTP 428 `Precondition Required` / `Connection Terminated` instead of recovering cleanly.

## Best Possible Solution

Fix the WhatsApp plugin so retryable Baileys close errors during the opening handshake are normalized through the same WhatsApp reconnect/status path used after a listener is active, while preserving terminal 401 logged-out and 440 conflict behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level failure path: mock `waitForWaConnection` to reject with a Boom-style `output.statusCode === 428` during `controller.openConnection` and observe that current `monitorWebChannel` rethrows it instead of scheduling a WhatsApp reconnect. The exact live 04:00 UTC cadence remains unverified.

## Solution Assessment

Is this the best way to solve the issue? No, the current behavior is not the best handling because opening-phase retryable closes bypass the WhatsApp reconnect loop. The narrow maintainable fix belongs in the WhatsApp monitor/controller path, not in generic gateway channel restarts or HTTP `If-Match`/ETag handling.
