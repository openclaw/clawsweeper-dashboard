## Summary

Keep open. The report matches a plausible WhatsApp reconnect gap on current main: established socket closes are handled by the WhatsApp monitor retry loop, but an initial/opening Baileys close with status 428 can escape that loop and be logged by the generic channel manager as a channel exit.

## What This Changes

The issue reports that after upgrading to 2026.4.29, the WhatsApp channel exits around 04:00 UTC with a Baileys/Boom HTTP 428 Precondition Required "Connection Terminated" error and leaves WhatsApp slow or unavailable.

## Best Possible Solution

Keep the issue open and fix the WhatsApp plugin so Baileys 428 connectionClosed during socket opening is handled by the same WhatsApp-specific reconnect/status path used for post-open closes, while leaving 401 logged-out and 440 conflict terminal behavior intact.

## Reproduction Assessment

Yes for the source-level path: a mocked waitForWaConnection rejection with Boom status 428 during controller.openConnection would currently be rethrown through monitorWebChannel and logged as a generic channel exit. The exact live 04:00 UTC recurrence still needs operator logs or a live WhatsApp session to confirm.

## Solution Assessment

No, current main does not appear to be the best handling for this failure because opening-phase 428 closes bypass the WhatsApp reconnect loop. The narrower maintainable fix is in the WhatsApp monitor/controller path, not in generic channel-manager restart policy or HTTP ETag headers.
