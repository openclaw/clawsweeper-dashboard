## Summary

Keep open. Current main still does not classify WhatsApp HTTP 428 as a terminal/relink condition, and open PR #75773 is already the paired implementation candidate with closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

After upgrading to v2026.4.29, WhatsApp Web intermittently exits with HTTP 428 / Connection Terminated and can leave outbound sends without an active listener.

## Best Possible Solution

Review and land or replace PR #75773 with WhatsApp-owned 428 handling for both opening and post-open close paths, plus regression tests that preserve existing 401 and 440 behavior.

## Reproduction Assessment

Yes for the source-level path: inject a Boom-like status 428 from waitForWaConnection/openConnection or from the listener close path and current main routes it through generic rethrow or retry handling. The exact scheduled live recurrence is supported by issue logs but was not locally reproduced in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable direction is to handle 428 inside the WhatsApp monitor/controller boundary and keep generic channel management unchanged. The open PR is the active candidate, but its non-retryable policy should be reviewed against the intermittent live logs before landing.
