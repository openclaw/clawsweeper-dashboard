## Summary

Keep open. Current main still has a plausible WhatsApp opening-phase reconnect gap: a Baileys close with HTTP 428 during `openConnection` is rethrown before the listener close-decision retry path is active, which matches the generic `channel exited` log shape in the report.

## What This Changes

The issue reports a WhatsApp channel regression after upgrading to v2026.4.29 where the channel exits with Boom HTTP 428 `Precondition Required` / `Connection Terminated` instead of recovering.

## Best Possible Solution

Keep the issue open and repair the WhatsApp plugin so opening-phase Baileys connectionClosed errors with retryable HTTP status codes, including 428, enter the same WhatsApp-specific reconnect/status path used after a listener is active, while preserving terminal handling for logged-out 401 and conflict 440.

## Solution Assessment

No, current main is not the best handling because it lets opening-phase 428 closes bypass the WhatsApp reconnect loop. The narrow maintainable solution belongs in the WhatsApp monitor/controller normalization path, not generic HTTP ETag handling or broad channel-manager restart policy.
