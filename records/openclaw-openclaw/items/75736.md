## Summary

Keep open. Current main still has a WhatsApp-owned 428 handling gap in both the opening and post-open reconnect paths, and the report is paired with open, unmerged PR #75773 using closing syntax, so this issue should stay open until that PR or a replacement fix lands.

## What This Changes

The report asks OpenClaw to handle recurrent WhatsApp channel exits with HTTP 428 Precondition Required after upgrading to v2026.4.29.

## Best Possible Solution

Land PR #75773 or a replacement WhatsApp plugin fix that handles 428 explicitly in both the handshake and post-open close paths, with tests preserving existing 401 logged-out and 440 conflict behavior and documenting the chosen retry-vs-relink policy.

## Reproduction Assessment

Yes for the source-level path: mock waitForWaConnection to reject with a Boom-like status 428 during controller.openConnection, and current main rethrows through monitorWebChannel to the generic channel manager. The live scheduled recurrence is supported by logs but was not locally reproduced.

## Solution Assessment

No for current main: it still lacks explicit 428 handling. The paired PR is the implementation candidate, but maintainers should review whether terminal relink guidance or bounded retry is the best behavior for intermittent live 428s.
