## Summary

Keep open. Current main still lacks WhatsApp-specific HTTP 428 handling in both the opening handshake path and the post-open close-decision path, and open PR #75773 is already the linked implementation candidate.

## What This Changes

The issue reports a WhatsApp Web regression where HTTP 428 `Precondition Required / Connection Terminated` exits the channel after upgrading to 2026.4.29.

## Best Possible Solution

Review and land or replace PR #75773 with WhatsApp-owned 428 handling for both handshake failures and listener close decisions, plus regression coverage that preserves existing 401 and 440 behavior.

## Reproduction Assessment

Yes for a source-level path: inject a Boom-like status 428 from `waitForWaConnection` or from the listener close path, and current main either rethrows it or treats it as retryable. The exact live 04:00 UTC recurrence was not reproduced here.

## Solution Assessment

No for current main; it does not yet implement the narrow WhatsApp 428 terminal/relink handling. The active PR direction is the maintainable boundary because it stays in the WhatsApp monitor/controller and adds focused regression tests.
