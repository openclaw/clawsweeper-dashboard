## Summary

Keep open. Current main still does not handle WhatsApp HTTP 428 in the WhatsApp reconnect boundary, and open PR #75773 is already the closing implementation candidate.

## What This Changes

The issue reports a regression where the WhatsApp Web channel exits with Boom HTTP 428 after upgrading to 2026.4.29, leaving the channel intermittently unavailable.

## Best Possible Solution

Review and land or replace PR #75773 with WhatsApp-owned 428 handling for both opening and post-open close paths, plus regression tests preserving existing 401 and 440 behavior.

## Reproduction Assessment

Yes for the source-level path: mock `waitForWaConnection` or the listener close path to produce a Boom-like status 428 and current main routes it through rethrow or generic retry handling. The exact scheduled live recurrence is supported by issue logs but was not locally reproduced.

## Solution Assessment

Yes, the narrow maintainable solution belongs in the WhatsApp monitor/controller boundary with targeted tests; changing the generic channel manager would be broader than needed.
