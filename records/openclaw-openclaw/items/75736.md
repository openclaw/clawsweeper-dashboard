## Summary

Keep open. Current main still lacks WhatsApp HTTP 428 handling in both the opening handshake and post-open close-decision paths, and open PR #75773 already tracks the narrow implementation with closing syntax.

## What This Changes

The issue reports a WhatsApp Web channel regression where HTTP 428 `Precondition Required / Connection Terminated` causes recurring channel exits after upgrading to 2026.4.29.

## Best Possible Solution

Review and land or replace PR #75773 with WhatsApp-owned 428 handling for both handshake failures and listener close decisions, plus regression coverage that preserves existing 401 and 440 behavior.

## Reproduction Assessment

Yes for a source-level path: inject a Boom-like status 428 from `waitForWaConnection` or from the listener close path, and current main either rethrows it or treats it as retryable. The exact live 04:00 UTC recurrence was not reproduced here.

## Solution Assessment

Yes for the linked PR direction, but no for current main. The maintainable fix belongs in the WhatsApp monitor/controller close classification with focused regression tests, without changing unrelated retry statuses.
