## Summary

Keep open: the fallback cleanup boundary is plausible, but the current PR head still strips a trailing assistant turn when any content block equals the stream-error sentinel, which can delete real mixed-content output before fallback/replay.

## What This Changes

The PR adds an in-memory transcript repair helper, wires it into embedded-runner fallback/replay cleanup, exports the zero-token usage predicate, and updates transcript hygiene docs, tests, and the changelog.

## Best Possible Solution

Keep the new in-memory fallback cleanup only after aligning its sentinel predicate with the current replay-history synthetic-tail contract and adding mixed-content preservation coverage.

## Reproduction Assessment

Yes for the PR-introduced defect: a trailing assistant content array containing the stream-error sentinel plus real text or a tool block reaches containsStreamErrorSentinel().some and is dropped at PR head. The original LiteLLM/Vertex fallback failure is source-supported by the related https://github.com/openclaw/openclaw/issues/77228 path, but I did not run a live provider fallback.

## Solution Assessment

No. The embedded-runner cleanup seam is reasonable, but the helper should mirror current main's narrower sentinel-only/provenance guard instead of stripping any assistant turn that merely contains the sentinel block.
