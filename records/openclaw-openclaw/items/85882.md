## What This Changes

The PR changes usage-cost transcript scanning so tokenized turns with unconfigured all-zero pricing are counted as missing-cost entries while explicitly configured zero-cost models remain complete zero-cost entries.

## Best Possible Solution

Land the missing-cost behavior only after the branch invalidates old usage-cost cache semantics and distinguishes generated unknown pricing from generated free pricing using the existing provider pricing contract or an explicit schema marker.

Full review comments:

- **[P1] Invalidate existing usage-cost caches:** `src/infra/session-cost-usage.ts:1138-1139`
  - body: The new missing-cost branch only runs when a transcript file is scanned, but existing `.usage-cost-cache.json` entries remain fresh because the cache version is still `3` and the pricing fingerprint is unchanged. A user with the old cached totals for the same unchanged session file will keep seeing `missingCostEntries: 0` after upgrade, and append-only refresh will preserve those old totals for the already-scanned bytes. Bump/include a semantics version in the cache freshness key and add a warm-cache regression test.
  - confidence: 0.91
- **[P1] Preserve generated free-pricing providers:** `src/infra/session-cost-usage.ts:1120-1125`
  - body: This guard treats every generated all-zero catalog row with no `models.providers` override as unknown, but official plugins already use generated zero pricing to mean free: Ollama declares `modelPricing.providers.ollama.external: false`, and its docs say auto-discovered model costs are `$0`. Those generated Ollama transcript rows will now get `entry.costTotal = undefined`, turning documented complete local/free usage into missing-cost entries. Use provider pricing provenance, or an explicit unknown/free catalog marker, before classifying generated zeros as missing.
  - confidence: 0.88
