## Summary

Keep open: the latest head fixes the stale usage-cost cache blocker and supplies useful live proof, but it still risks reclassifying generated local/free zero-cost provider rows as missing-cost entries. The branch needs a maintainer-confirmed pricing provenance contract or a narrow compatibility fix before merge.

## What This Changes

The PR changes usage-cost transcript scanning so token-bearing unconfigured all-zero model pricing is counted as missing-cost usage, adds regression coverage, and bumps the usage-cost cache version.

## Best Possible Solution

Land the missing-cost behavior with the cache-version bump, but classify all-zero pricing using source/provenance so unpriced external/API models become missing while local or self-hosted free providers remain complete zero-cost usage.

## Reproduction Assessment

Yes for source-level review: current main treats `cost.total: 0` as a complete cost and only increments `missingCostEntries` for undefined cost. The PR body also supplies before/after live CLI output for the gpt-5.5 case, but I did not run tests in this read-only review.

## Solution Assessment

No, not yet: the scan-time direction and cache-version bump are sound, but the guard currently treats generated all-zero catalog pricing as unknown unless it came from operator config. The safer fix must preserve generated local/free pricing using a maintained provenance contract.
