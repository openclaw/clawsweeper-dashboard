## Summary

Keep open: current main still has source-visible iOS Talk routing that can parse managed-room/non-realtime config and then force an active OpenAI provider back to realtime gateway-relay, and the reporter’s live update narrows the remaining session-close failure to the OpenAI Realtime path rather than pairing, permissions, or generic Gateway plumbing.

## What This Changes

The issue reports that iOS Talk starts an OpenAI realtime gateway-relay session, closes it before audio append, and ignores server STT/TTS routing configuration.

## Best Possible Solution

Repair iOS Talk routing so Gateway-default/server-selected STT/TTS or managed-room config is honored instead of being auto-forced to OpenAI relay, then validate the OpenAI realtime relay close path with a live iPhone/provider proof before changing Gateway lifecycle behavior.

## Reproduction Assessment

Yes for a source-level reproduction of the routing half: current iOS parses managed-room as native, then applyLoadedTalkConfig forces an active OpenAI provider back to realtime relay. The full immediate OpenAI provider-close symptom is supported by reporter live evidence but was not independently live-reproduced here.

## Solution Assessment

Unclear as a single fix: the requested behavior is valid, but the best repair likely splits iOS routing from Gateway/OpenAI relay lifecycle. Maintainers should first decide whether the canonical fix is to honor server routing by default, repair the OpenAI relay close, or do both in separate changes.
