## Summary

Keep open: current main and the latest release improve realtime-fallback diagnostics, but they still do not solve the central iOS Talk failure because OpenAI provider selection can force realtime relay after managed-room/non-relay routing is parsed, and the Gateway still forgets closed relay sessions before later audio appends.

## What This Changes

The issue reports iOS Talk OpenAI realtime sessions closing before audio append and server STT/TTS routing being ignored.

## Best Possible Solution

Honor configured non-relay or STT/TTS routing in iOS without forcing OpenAI relay, then prove and fix the OpenAI realtime relay close-before-ready path with live provider evidence and focused regression tests.

## Reproduction Assessment

Yes by source for the routing half: current main parses managed-room/non-relay as native, then applyLoadedTalkConfig can force activeProvider openai back to realtimeRelay. The reporter also supplied live same-device evidence that Microsoft RTC works while OpenAI Realtime fails, but I did not independently rerun a live iPhone/OpenAI provider repro.

## Solution Assessment

No current-main fix is sufficient: PR 91201 made fallback and structured realtime errors visible, but it did not remove the OpenAI relay override or prove the provider lifecycle is fixed. The narrow maintainable direction is to fix iOS routing ownership first and separately prove the OpenAI realtime close-before-ready path.
