## Summary

Keep open: current main still has a source-visible iOS routing path that can parse managed-room config and then force OpenAI back to realtime gateway-relay, while the reported post-close `Unknown Talk session` behavior matches the Gateway relay cleanup path.

## What This Changes

The issue reports that iOS Talk fails on OpenAI Realtime after the Gateway closes the relay session before audio append, and that iOS ignores server-configured STT/TTS managed-room routing.

## Best Possible Solution

Repair iOS Talk routing so server-selected managed-room/STT-TTS config is honored when appropriate, then validate the OpenAI realtime relay close path with a live iPhone and provider proof before changing Gateway lifecycle behavior.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction of the routing half: iOS parses managed-room as native, then the manager forces active OpenAI back to realtime relay. The immediate OpenAI provider-close symptom has strong reporter evidence but was not independently live-reproduced here.

## Solution Assessment

Unclear as a single fix: the requested behavior is valid, but the best repair likely splits iOS routing from Gateway/OpenAI relay lifecycle. Maintainers should decide whether to honor server routing by default, repair the OpenAI relay close, or do both in separate changes.
