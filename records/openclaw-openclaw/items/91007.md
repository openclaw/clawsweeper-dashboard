Keep open: current main still shows the iOS routing split and Gateway relay lifecycle path described in the report, and no merged PR or release evidence shows the OpenAI Realtime failure is fixed. The issue remains the canonical tracker for the iOS Talk/OpenAI Realtime session-close and server-routing behavior.

## What This Changes

Fix iOS Talk so OpenAI Realtime sessions remain usable for audio append or the client honors configured server STT/TTS routing instead of forcing realtime gateway relay.

## Best Possible Solution

Split the repair into a maintainer-approved iOS routing decision for server-selected non-relay/STT-TTS config and a live iPhone/OpenAI Realtime validation pass for the relay close lifecycle before changing Gateway behavior.

## Reproduction Assessment

Yes for the routing half by source inspection: iOS parses managed-room as native, then applyLoadedTalkConfig forces active OpenAI back to realtime relay. The immediate OpenAI provider-close symptom has strong reporter live evidence but was not independently live-reproduced here.

## Solution Assessment

Unclear as one fix: the report is valid, but the safest solution likely separates iOS routing semantics from Gateway/OpenAI relay lifecycle. Maintainers should decide whether server routing overrides iOS OpenAI relay selection, then require live provider proof for lifecycle changes.
