## Summary

Keep open: current main still has the source-visible iOS routing split where managed-room config parses as native but an active OpenAI provider is forced back to realtime gateway-relay, and the Gateway relay close path still forgets the session before later audio appends can hit `Unknown Talk session`. The reporter also added live same-device evidence that Microsoft RTC works while OpenAI Realtime fails, so this still needs maintainer-owned iOS/OpenAI validation and routing direction.

## What This Changes

The issue reports that iOS Talk opens an OpenAI realtime gateway-relay session that closes before audio append and ignores server-configured STT/TTS routing.

## Best Possible Solution

Repair iOS Talk routing so server-selected non-relay/STT-TTS config is honored when maintainers confirm that behavior, and separately validate the OpenAI realtime relay close path with a live iPhone/OpenAI provider proof before changing Gateway lifecycle behavior.

## Reproduction Assessment

Yes for a source-level reproduction of the routing half: current iOS parses managed-room as native, then the manager forces active OpenAI back to realtime relay. The immediate OpenAI provider-close symptom has strong reporter live evidence but was not independently live-reproduced here.

## Solution Assessment

Unclear as one fix: the report is valid, but the best repair likely splits iOS routing from Gateway/OpenAI relay lifecycle. Maintainers should decide whether server routing should override iOS OpenAI relay selection, then require live provider proof for any lifecycle change.
