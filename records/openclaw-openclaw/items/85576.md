## Summary

Keep open: this PR targets the right WebChat state-ordering bug and includes credible live-after-fix proof, but the new per-call suppression can still miss an already in-flight session refresh, leaving a source-visible path where `Done` publishes before deferred history replay.

## What This Changes

The PR delays WebChat terminal run-status publication after session reconciliation until deferred chat-history replay completes, with focused session/controller regression tests.

## Best Possible Solution

Keep the cleanup/status separation, but make terminal status suppression follow the pending deferred replay state rather than a single `loadSessions` call so in-flight and queued refreshes both wait for history replay.

## Reproduction Assessment

Yes, at source level: current main can publish terminal WebChat run status from session reconciliation before deferred history reload renders the final reply. I did not run a live browser reproduction in this read-only review, but the linked issue and PR proof provide real WebChat timing evidence.

## Solution Assessment

No, not yet: the direct session patch and owned refresh paths are handled, but the per-call override does not protect an already in-flight session refresh. The safer fix is to tie suppression to pending deferred replay state until `loadChatHistory` completes.
