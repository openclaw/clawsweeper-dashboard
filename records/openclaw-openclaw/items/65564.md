## Summary

Keep open: current main fixes the original stale sessionFile carryover, but the same-key isolated heartbeat rotation still overwrites the active store entry without reset-archiving the prior transcript, and an open focused fix PR already targets that remaining gap.

## What This Changes

The issue reports that heartbeat isolated-session rotation changes the stored sessionId while leaving prior heartbeat transcript files reused or orphaned, causing context and session-state residue to accumulate.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73809 or an equivalent narrow reset-archive change for the same-key heartbeat path, while keeping broader cron orphan cleanup and memory-ingestion hardening in separate follow-ups.

## Reproduction Assessment

Yes, at source level. Current main force-rotates isolated heartbeat sessions under the same key and overwrites the store entry while only archiving stale nested-key removals.

## Solution Assessment

Yes. Capturing the previous same-key entry before overwrite and archiving it with the existing store-directory-restricted reset helper is the narrowest maintainable fix for the remaining behavior.
