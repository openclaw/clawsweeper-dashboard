## Summary

Keep open: the stale-binding cleanup direction is valid, but this PR still retries recoverable `timeout`/`unknown` errors outside the CLI runner hook boundary and lacks after-fix real behavior proof.

## What This Changes

The PR widens CLI `FailoverError` handling in `attempt-execution.ts`, adds CLI attempt regression tests, and updates `CHANGELOG.md` for stale CLI session binding cleanup.

## Best Possible Solution

Land hook-aware stale-session recovery that clears stored bindings for all resumed-session `FailoverError`s, rethrows non-recoverable errors after cleanup, and retries recoverable errors before the final `agent_end`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main only clears stored CLI bindings for `session_expired`, while other `FailoverError` reasons with an active binding can leave the stale resume ID in place.

## Solution Assessment

No. Clearing stale bindings for all resumed-session `FailoverError`s is the right store boundary, but recoverable retry should live inside `cli-runner.ts` or an equivalent hook-aware seam so observers only see the final turn state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move recoverable retry inside the CLI runner hook boundary:** `src/agents/command/attempt-execution.ts:535-542`
  - body: The added `timeout`/`unknown` retry runs after `runPreparedCliAgent` has already emitted `agent_end` with `success: false` for those errors. If the fresh retry succeeds, hook consumers see a failed final state while the caller returns success, so keep the cleanup but move recoverable retry under the CLI runner lifecycle boundary or add equivalent hook-aware coverage.
