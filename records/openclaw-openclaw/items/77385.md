## Summary

Keep this PR open: the underlying stale CLI-session bug is valid on current main, but the patch retries recoverable timeout/unknown failures after `runCliAgent` has already emitted a failed `agent_end` hook, so it can report a failed run to hook consumers even when the retry succeeds.

## What This Changes

The PR widens CLI FailoverError handling in `runAgentAttempt` to clear stored CLI session bindings for any resumed-session failure, retry only recoverable reasons, adds an auth-failure regression test, and adds a changelog entry.

## Best Possible Solution

Coordinate stale-session clearing with the CLI retry before failure hooks are emitted, then land the fix with hook-aware regression coverage and a corrected changelog reference.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a stored CLI binding plus a non-session_expired `FailoverError` currently bypasses cleanup, and the PR's timeout/unknown retry path would run after `runCliAgent` has emitted a failed `agent_end` hook.

## Solution Assessment

No, not as written. Clearing in the caller is the right store boundary, but recoverable retry must happen before the CLI runner emits failure hooks, or hook consumers will observe a failed run that ultimately succeeded.
