## Summary

This PR targets a valid current-main stale CLI session bug, but the latest patch still retries recoverable `timeout`/`unknown` failures outside the CLI runner hook lifecycle and only provides mocked/unit-test proof.

## What This Changes

The PR widens CLI `FailoverError` session-store cleanup in `attempt-execution.ts`, adds CLI attempt regression tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the stale-session cleanup direction, but make recoverable retry hook-aware in `cli-runner.ts` or an equivalent runner seam, add runner lifecycle coverage, and require real CLI/backend proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the reported bug: current main only clears stored CLI bindings for `session_expired`, while CLI execution raises `timeout`, `unknown`, `auth`, `billing`, and `rate_limit` as `FailoverError` reasons. I did not run a live provider failure scenario in this read-only review.

## Solution Assessment

No. Clearing stale bindings for every resumed-session `FailoverError` is the right store boundary, but recoverable retry should stay inside the CLI runner lifecycle so hook observers see one final turn outcome.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep recoverable retry inside the CLI hook boundary:** `src/agents/command/attempt-execution.ts:535-542`
  - body: The new outer retry handles `timeout`/`unknown` after `runPreparedCliAgent` has already emitted `agent_end` with `success: false`. If the fresh retry succeeds, hook consumers see a failed final state while the caller returns success, so move the recoverable retry under the CLI runner lifecycle boundary or add an equivalent hook-aware seam.
