## Summary

Keep open: the cleanup direction is useful, but the patch should not merge as-is because the new timeout/unknown retry still happens after the CLI runner has emitted a failed `agent_end`, so a recovered turn can report failure to hook consumers while returning success to the caller.

## What This Changes

The PR clears persisted resumed CLI session bindings for every `FailoverError`, retries recoverable reasons with a fresh CLI session, rethrows non-recoverable reasons, adds CLI attempt tests, and updates the changelog.

## Best Possible Solution

Land a hook-aware stale-session recovery that clears persisted bindings for all resumed-session FailoverErrors, retries recoverable reasons before any failed `agent_end`, and keeps non-recoverable clear-and-rethrow behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main only clears stored CLI bindings for `session_expired`, and the PR’s timeout/unknown retry would run after `runPreparedCliAgent` emits a failed `agent_end`.

## Solution Assessment

No. Clearing stale bindings in the caller is a good store boundary for non-recoverable errors, but recoverable retry should move into `cli-runner.ts` or another hook-aware seam so observers only see the final turn result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move recoverable retry before failed agent_end:** `src/agents/command/attempt-execution.ts:535-542`
  - body: The patch retries `timeout` and `unknown` from `attempt-execution` after `runCliAgent` has already emitted `agent_end` with `success: false` for those FailoverErrors. If the fresh retry succeeds, hook consumers see a failed final state while the caller returns success; keep the cleanup, but move recoverable retry inside the CLI runner hook boundary or add equivalent hook-aware coverage.
  - confidence: 0.88
