## Label Justifications

- `P1`: A real subagent workflow can report children as accepted and running even though launch validation has already prevented execution and transcript production.
- `impact:session-state`: The child run, orchestrator, and external pipeline can disagree about whether execution is active or terminal.
- `impact:other`: Parallel orchestration reliability is materially affected because callers cannot reliably determine which accepted children actually launched.

## Summary

Keep this issue open: current main accepts and registers a subagent with an explicit thinking override, then can reject that override later during launch before attempt lifecycle setup, leaving orchestration state inconsistent. No merged fix or canonical replacement was found, although the proposed automatic-fallback and API-expansion work should be separated from the central lifecycle bug.

## What This Changes

Prevent programmatic subagent spawns from silently entering inconsistent states when the requested thinking level is unsupported by the selected CLI-backed model.

## Best Possible Solution

Preflight the resolved model and effective thinking before accepting the child, or immediately transition a rejected launch to the canonical terminal-error state; make fallback versus explicit rejection a deliberate spawn contract and handle acknowledgement/catalog enhancements separately.

## Reproduction Assessment

Yes—at source level with high confidence: `sessions_spawn` creates an explicit thinking override, and current launch validation throws for unsupported explicit levels before attempt lifecycle setup. The issue also supplies real 2026.6.5 task output, but this review did not run a fresh live fan-out reproduction.

## Solution Assessment

No, not as one bundled change: deterministic preflight or terminal failure propagation is the narrow bug fix, while automatic downgrade semantics, new acknowledgement fields, and CLI catalog exposure require separate product/API decisions.
