- `impact:message-loss`: The reported failure can prevent later chunks or final text from reaching the originating channel.

## Summary

This issue should stay open: current main and the latest release still create the foreground reply fence at inbound dispatch time, while https://github.com/openclaw/openclaw/pull/93977 is an open linked implementation candidate for the requested lazy-fence behavior.

## What This Changes

The issue requests deferring foreground reply fence creation from inbound dispatch to first visible delivery so same-session auto-replies do not truncate or interleave channel delivery.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/93977 as the canonical lazy-fence fix, or replace it with a maintainer-approved alternative that preserves same-session channel delivery order.

## Reproduction Assessment

Yes from source inspection: current main still creates the fence at inbound dispatch time and existing freshness tests encode same-session suppression. I did not run a live channel race in this read-only review.

## Solution Assessment

Yes, deferring fence creation to first visible delivery appears to target the root ordering problem rather than only the later cancellation condition. The best path is to review the linked fixing PR instead of opening a duplicate repair lane.
