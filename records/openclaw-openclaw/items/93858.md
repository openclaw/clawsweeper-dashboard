- `impact:message-loss`: The failure mode can suppress remaining reply text before it reaches the originating channel.
- `impact:session-state`: The race comes from foreground delivery generation state advancing before same-session turn admission state catches up.

## Summary

Keep open: current main and v2026.6.8 still create the foreground reply fence at buffered-dispatch entry, and live GitHub shows an open closing PR rather than a merged fix.

## What This Changes

Defer foreground reply fence creation from inbound dispatch entry to first actual delivery so same-session replies cannot truncate or interleave channel output.

## Best Possible Solution

Land a corrected lazy foreground-fence fix through https://github.com/openclaw/openclaw/pull/93977 or https://github.com/openclaw/openclaw/pull/91963 after preserving stale-suppression invariants, focused freshness tests, and real channel proof.

## Reproduction Assessment

Yes at source level: the buffered dispatch path still creates the foreground fence before reply admission, and the current freshness tests preserve the same stale-delivery suppression behavior under review. I did not run a live channel replay in this read-only pass.

## Solution Assessment

Yes in direction, but not yet as a completed repository fix. Delivery-time fence creation or equivalent ordering state is the right layer, while maintainers still need to review the open candidate PRs for stale-suppression invariants.
