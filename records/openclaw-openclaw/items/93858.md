## Label Justifications

- `impact:message-loss`: The reported symptom is visible assistant output failing to reach the originating plugin/channel delivery path.

## Summary

Keep open: current main and the latest release still create the foreground reply fence at buffered-dispatch entry, while the linked lazy-fence fix path is still open rather than merged.

## What This Changes

Defer foreground reply fence creation until first visible/source delivery so a queued same-session inbound cannot truncate or interleave an older in-flight source reply.

## Best Possible Solution

Land a corrected lazy foreground-fence fix, such as https://github.com/openclaw/openclaw/pull/91963 or https://github.com/openclaw/openclaw/pull/93977 after review, with tests preserving both queued-turn delivery and newer-visible stale suppression.

## Reproduction Assessment

Yes at source level: current main creates the foreground fence before same-session reply admission, and the guard can cancel older source delivery behind a newer generation. I did not run a live channel replay in this read-only review.

## Solution Assessment

Yes in direction: deferring foreground fence creation until visible/source delivery begins is the narrowest maintainable fix. The open PRs still need review because the final implementation must preserve stale-suppression invariants.
