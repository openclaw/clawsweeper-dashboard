## Label Justifications

- `P1`: The report describes a broken real channel workflow where same-session follow-up messages can truncate or reorder visible replies.
- `impact:message-loss`: The affected foreground delivery path can suppress later chunks or finals before they reach the originating channel/plugin.
- `impact:session-state`: The bug is an admission-versus-delivery state mismatch for same-session queued turns.

## Summary

Keep open: current main and v2026.6.8 still create the foreground reply fence at buffered dispatch entry, and the open clean PR https://github.com/openclaw/openclaw/pull/93977 is the direct implementation candidate that closes this issue.

## What This Changes

The issue asks OpenClaw to defer foreground reply fence creation until first visible delivery so queued same-session inbound messages cannot truncate or interleave an older in-flight reply.

## Best Possible Solution

Review and land a lazy-fence fix such as https://github.com/openclaw/openclaw/pull/93977, with focused regression coverage for current truncation and queued same-session delivery ordering.

## Reproduction Assessment

Yes, source-level: current main eagerly begins the foreground fence at buffered dispatch entry, and the existing freshness test expects the older final to be suppressed after a newer same-session inbound. The live interleaving proof is in reporter screenshots/comments, but I did not rerun that external-channel timing path here.

## Solution Assessment

Yes: deferring fence creation to the delivery boundary is the narrow owner-boundary fix because the current eager fence is created before same-session delivery/admission semantics line up. The active-generation guard in https://github.com/openclaw/openclaw/pull/93845 is only a partial overlap and does not by itself prove ordered delivery.
