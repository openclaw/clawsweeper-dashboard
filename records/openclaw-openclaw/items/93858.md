## Label Justifications

- `P1`: The issue describes a user-facing agent/channel workflow where concurrent same-session messages can lose or misorder visible replies.
- `impact:message-loss`: The reported behavior can truncate or suppress external-channel delivery while the reply continues in the session UI.
- `impact:session-state`: The bug is caused by foreground reply generation state advancing ahead of same-session turn admission and delivery ordering.

## Summary

Keep open: current main and the latest release still create the foreground reply fence at buffered-dispatch entry, while the direct lazy-fence fix remains an open PR rather than merged.

## What This Changes

The issue requests moving foreground reply fence creation from inbound dispatch entry to first delivery so same-session queued replies do not truncate or interleave.

## Best Possible Solution

Land a reviewed lazy-fence fix that defers generation creation for queued-before-visible turns, preserves freshness suppression for real newer visible winners, and includes focused regression plus live-channel proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 call `beginForegroundReplyFence` at buffered-dispatch entry before delivery, and the guard cancels once a newer visible generation wins. I did not run a live transport replay in this read-only review.

## Solution Assessment

Yes in direction: moving fence creation from inbound entry to first actual delivery is the right layer for queued-before-visible turns. The final PR still needs maintainer review to preserve existing freshness semantics and prove the real channel behavior.
