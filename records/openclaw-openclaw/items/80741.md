## Summary

Keep open: the fix direction is valid and source-reproducible, but the branch still leaves capability-driven Talk availability exposed for pending nodes and the external real behavior proof is still test/check summary only.

## What This Changes

The PR separates declared/effective node commands, blocks node command forwarding until node pairing approval, refreshes live command/remote-skill state on approval or removal, and adds Gateway/infra regression tests.

## Best Possible Solution

Gate every effective node availability surface that can enable command-like behavior, including capability-derived Talk availability, then land after maintainer review, CI, and redacted real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main returns declared commands for unpaired nodes and direct registry invokes lack a pairing check. I did not run the Gateway scenario in this read-only review.

## Solution Assessment

No, not as-is: the command-gating approach is the right narrow fix, but caps also need an approved/effective surface because Talk checks caps independently of commands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate Talk capabilities before registration:** `src/gateway/server/ws-connection/message-handler.ts:1342-1343`
  - body: This only swaps in the effective command list; `connectParams.caps` still carries values like `talk` for an unapproved node. `hasConnectedTalkNode()` treats that capability alone as enough to enable webchat `talk.mode`, so a pending Talk node can still expose a live capability before pairing approval. Clear or separately gate effective caps for pending nodes, or make Talk consult approved/effective node state.
  - confidence: 0.9
