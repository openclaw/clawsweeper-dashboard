## Label Justifications

- `P2`: The report describes a real repeated-query memory search failure with a clear source path, but it is scoped to the gateway memory tool rather than making the core runtime unusable.
- `impact:session-state`: The failure is caused by cached memory/index snapshot state losing metadata across calls, which is session/runtime state drift for the memory tool.

## Summary

The issue should stay open: current main still has a source-visible path where the gateway memory search tool can lose usable index metadata after a cached query snapshot is consumed, while the CLI path uses a separate manager flow that can continue to search the same index.

## What This Changes

The issue reports that the gateway `memory_search` tool succeeds once after cold start and then returns `index metadata is missing` on later calls despite a freshly built local memory index.

## Best Possible Solution

Keep the issue open and fix the gateway memory index cache so every cached query snapshot preserves or reloads metadata before `memory_search` runs repeated searches.

## Reproduction Assessment

Yes from source, but not from a live run: `getIndexedMemoryChunks` can return cached chunks without metadata after the cache is warm, and `searchIndexedMemory` then throws `index metadata is missing` on that snapshot path.

## Solution Assessment

Yes, the best fix is narrow: preserve metadata in the cached gateway query snapshot or reload it through the existing index metadata reader, then add a regression test for two consecutive gateway `memory_search` calls after cold start.
