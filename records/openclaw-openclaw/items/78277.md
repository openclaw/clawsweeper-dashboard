## Summary

Keep this PR open: the runtime direction matches the existing file-lock pattern for a real bootstrap-token race, but the added contention test does not actually simulate another process and the PR still lacks contributor-provided real behavior proof.

## What This Changes

The PR wraps bootstrap-token state mutations in the existing file-lock helper and adds device-bootstrap regression coverage for lock contention.

## Best Possible Solution

Keep the narrow file-lock fix approach, but require a true cross-process or otherwise non-reentrant lost-update regression test, a changelog entry, and contributor-supplied real behavior proof before merge.

## Reproduction Assessment

Yes from source inspection, not from a live run. Current main uses process-local locking plus whole-file bootstrap.json replacement, while the Gateway accepts verified bootstrap tokens and revokes them later, so the lost-update path described by the linked issue is credible.

## Solution Assessment

No as submitted. The runtime fix direction is the narrow existing pattern, but the regression test must use a real competing process or another non-reentrant stale-write harness, and the PR needs real behavior proof plus changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a non-reentrant contention test:** `src/infra/device-bootstrap.test.ts:99`
  - body: This test holds `acquireFileLock()` from the same process and manager that `issueDeviceBootstrapToken()` uses. The OpenClaw wrapper passes `allowReentrant: true`, and fs-safe re-enters a held same-process lock instead of waiting, so this does not simulate the external writer from the bug and can pass or fail based on async timing rather than cross-process serialization. Please spawn a child process or build a direct stale-snapshot/lost-update harness that cannot re-enter the lock.
  - confidence: 0.93
