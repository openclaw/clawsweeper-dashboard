## Summary

Keep open: this is a plausible security fix for a source-proven bootstrap-token race, but the PR still leaves a mutating bootstrap-state path outside the new lock and lacks after-fix real behavior proof.

## What This Changes

The PR adds a cross-process file-lock wrapper around several device bootstrap token state operations and adds one contention test for setup-code issuance.

## Best Possible Solution

Serialize every bootstrap-token read-modify-write path across processes, add non-reentrant child-process or stale-snapshot regression coverage, and require redacted after-fix setup-code proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: separate processes can each load devices/bootstrap.json and later replace it with stale whole-file state because current main uses only a module-local async lock.

## Solution Assessment

No. Cross-process locking is the right direction, but this patch is incomplete until every mutating bootstrap-state path is serialized and the regression test proves a non-reentrant process boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Lock the device-scoped revoke writer:** `src/infra/device-bootstrap.ts:221`
  - body: This patch moves several bootstrap-state writers to withBootstrapStateLock(), but revokeDeviceBootstrapTokensForDevice() still deletes records and persists devices/bootstrap.json under the process-local withLock. A rejected pending device can still race with issuance or verification and overwrite newer token state, so route this writer through the same cross-process lock.
  - confidence: 0.9
- **[P2] Replace the reentrant lock test:** `src/infra/device-bootstrap.test.ts:97-99`
  - body: The new contention test holds acquireFileLock() in the same process and manager used by withBootstrapStateLock(), while acquireFileLock() is documented/configured as reentrant. That does not exercise the separate-process lost-update case; use a child process or explicit stale-snapshot harness that cannot re-enter the lock.
  - confidence: 0.9
