## Summary

Keep open: current main and the latest release still use process-local bootstrap-token JSON writes, so the PR is still useful, but this branch is not merge-ready because it leaves one mutating writer outside the new cross-process lock, lacks real setup-code proof, and is currently dirty against main.

## What This Changes

The PR adds cross-process locking around device bootstrap-token state writes and a lock-contention regression test for setup-code lost-update races.

## Best Possible Solution

Finish cross-process serialization for every mutating bootstrap-token state path, add a non-reentrant stale-snapshot regression, refresh the branch, and require redacted real setup-code proof before merge.

## Reproduction Assessment

Yes. Source inspection shows independent setup-code and Gateway processes can load stale devices/bootstrap.json snapshots and later overwrite bootstrap-token binding or revocation because current main uses only process-local serialization.

## Solution Assessment

No. Cross-process file locking is the right narrow direction, but this branch leaves one reachable mutating writer on the old lock path and does not prove the actual stale-snapshot resurrection case.
