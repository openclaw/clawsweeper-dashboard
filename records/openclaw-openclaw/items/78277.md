## Summary

Keep this PR open: it targets a valid security bug, but the proposed patch still leaves a bootstrap-token state mutation outside the new cross-process lock, its regression test does not prove the cross-process lost-update case, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a cross-process file-lock wrapper around several device-bootstrap token state operations and adds a contention-oriented regression test.

## Best Possible Solution

Use one cross-process bootstrap state lock for every bootstrap-token read-modify-write path and cover the lost update with a non-reentrant child-process or explicit stale-snapshot regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: separate setup-code issuer and Gateway verifier/redeemer processes can each load devices/bootstrap.json and then replace it with stale whole-file state when only process-local locking is used.

## Solution Assessment

No, not as submitted. Cross-process serialization is the right narrow fix, but it must cover every mutating bootstrap state path and the test must prove cross-process or stale-snapshot contention.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Lock every bootstrap state mutation:** `src/infra/device-bootstrap.ts:292`
  - body: The branch switches several bootstrap-token writers to withBootstrapStateLock, but the potential merge result still leaves revokeDeviceBootstrapTokensForDevice() doing load/delete/persist under the old process-local withLock. That writer can race with issuance or verification and overwrite newer token state, so route it through the same cross-process lock.
  - confidence: 0.9
- **[P2] Use a non-reentrant contention test:** `src/infra/device-bootstrap.test.ts:97-99`
  - body: The new test holds acquireFileLock() from the same process and helper used by the implementation, but acquireFileLock() allows reentrant acquisition. This can pass without exercising a competing process or stale snapshot writer, so use a child process or an explicit stale-snapshot harness for the lost-update regression.
  - confidence: 0.93
