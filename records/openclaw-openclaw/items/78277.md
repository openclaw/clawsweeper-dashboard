## Summary

Keep open: the PR targets a real current-main auth-state race, but it is not merge-ready because one mutating bootstrap state path remains outside the new cross-process lock, the added test does not prove cross-process contention, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a cross-process file-lock wrapper around several device-bootstrap token state operations and adds a contention regression test.

## Best Possible Solution

Use one cross-process bootstrap state lock for every bootstrap-token read-modify-write path, add non-reentrant lost-update regression coverage, and require redacted real setup-code proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: separate Gateway and setup-code issuer processes can each load devices/bootstrap.json and then replace it with stale whole-file state because current main uses only a module-local lock.

## Solution Assessment

No, not as submitted. The cross-process lock direction is the right narrow fix, but it must cover every mutating bootstrap state path and the regression test must use a child process or explicit stale-snapshot harness that cannot re-enter the lock.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Lock every bootstrap state mutation:** `src/infra/device-bootstrap.ts:257`
  - body: The branch switches several bootstrap-token writers to `withBootstrapStateLock`, but `revokeDeviceBootstrapTokensForDevice()` still does a load/delete/persist under the old process-local `withLock`. That leaves a mutating writer that can race with issuance or verification and overwrite newer token state, so route this path through the same cross-process lock.
  - confidence: 0.9
- **[P2] Use a non-reentrant contention test:** `src/infra/device-bootstrap.test.ts:98`
  - body: The new test holds `acquireFileLock()` from the same process and wrapper used by the implementation, while `acquireFileLock()` allows reentrant acquisition. This can pass without exercising a competing process or stale snapshot writer, so use a child process or explicit stale-snapshot harness for the lost-update regression.
  - confidence: 0.93
