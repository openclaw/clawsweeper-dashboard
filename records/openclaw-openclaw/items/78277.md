## Summary

Keep open: this PR targets a real, source-reproducible bootstrap-token race, but the proposed merge result still leaves one mutating bootstrap-token state path outside the cross-process lock and the contributor has not provided required real behavior proof. A member comment also agrees the added regression test does not prove the cross-process case.

## What This Changes

The PR adds a cross-process file-lock wrapper around several device bootstrap token state operations and adds a contention test for setup-code issuance.

## Best Possible Solution

Serialize every bootstrap-token read-modify-write path across processes, add non-reentrant child-process or stale-snapshot regression coverage, and require redacted after-fix setup-code proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: separate processes can load and later replace devices/bootstrap.json under only a module-local lock while Gateway accepts verified bootstrap tokens for pairing auth; I did not run a live two-process Gateway repro in this read-only review.

## Solution Assessment

No. Cross-process locking is the right narrow direction, but this patch is incomplete because one mutating bootstrap-token cleanup path remains on the process-local lock and the regression test uses the same reentrant in-process helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Lock every bootstrap-state writer:** `src/infra/device-bootstrap.ts:316`
  - body: The proposed merge result still leaves `revokeDeviceBootstrapTokensForDevice()` under the old process-local `withLock` while it deletes records and persists `devices/bootstrap.json`. A rejected pending device can therefore race with issuance or verification and overwrite newer bootstrap-token state, so route this path through the same cross-process bootstrap lock.
  - confidence: 0.9
- **[P2] Use a non-reentrant race test:** `src/infra/device-bootstrap.test.ts:97-99`
  - body: The new contention test acquires `acquireFileLock()` from the same process and helper used by the implementation, but that wrapper is explicitly reentrant. This does not prove the separate-process lost-update case, so use a child process or an explicit stale-snapshot harness that cannot re-enter the lock.
  - confidence: 0.9
