## Summary

Keep this PR open: the reported bootstrap-token race is still real on current main, but this branch is not merge-ready because it is conflict-stale, leaves a mutating bootstrap-state writer outside the new cross-process lock, uses same-process reentrant lock coverage, and lacks inspectable real behavior proof.

## What This Changes

The PR adds a bootstrap-state file-lock wrapper around most device bootstrap token read-modify-write paths and adds a lock-contention regression test.

## Best Possible Solution

Finish cross-process serialization for every mutating bootstrap-token state path, replace the test with non-reentrant multi-process or explicit stale-snapshot coverage, rebase, and require redacted after-fix setup-code proof before merge.

## Reproduction Assessment

Yes from source inspection, though I did not run a live two-process Gateway repro. Current main still uses process-local bootstrap locking while the Gateway verifies, redeems, and revokes bootstrap tokens on the auth path, so stale whole-file writes can be reasoned through the code path.

## Solution Assessment

No. Cross-process locking is the right narrow direction, but this patch leaves one mutating writer process-local, uses a reentrant same-process test for a cross-process race, and lacks real after-fix setup-code proof.
