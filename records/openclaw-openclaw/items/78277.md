## Summary

Keep this PR open, but do not merge it as submitted. Current main still has the bootstrap-token lost-update shape, and this branch is the active linked fix path, but it lacks real behavior proof and leaves blocking coverage/implementation gaps around the cross-process race.

## What This Changes

The PR wraps most device-bootstrap token state mutations in a cross-process file lock and adds a setup-code issuance contention test.

## Best Possible Solution

Use one cross-process bootstrap state lock for every bootstrap-token read/modify/write path, add a non-reentrant or child-process stale-snapshot regression test, and require redacted real setup-code proof before merge.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Current main uses only a process-local lock around whole-file bootstrap.json updates while setup-code issuance and Gateway verification can happen in separate OpenClaw processes.

## Solution Assessment

No, not as submitted. The file-lock direction matches the existing pairing-store pattern, but the PR must cover every mutating bootstrap state path and replace the reentrant same-process test with a true cross-process or explicit stale-snapshot harness.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Lock every bootstrap state mutation:** `src/infra/device-bootstrap.ts:257`
  - body: The new helper is applied to most token mutations, but `revokeDeviceBootstrapTokensForDevice()` still loads and rewrites the same `devices/bootstrap.json` map under the old process-local `withLock`. That leaves a state writer that can still race with issuance or verification and overwrite newer bootstrap-token state, so please route this path through the same cross-process lock.
  - confidence: 0.9
- **[P2] Use a non-reentrant contention test:** `src/infra/device-bootstrap.test.ts:99`
  - body: This test holds `acquireFileLock()` from the same process and lock manager that the implementation uses, and the wrapper sets `allowReentrant: true`. That means the test does not model the external writer from the reported bug; please use a child process or explicit stale-snapshot harness that cannot re-enter the lock.
  - confidence: 0.93
