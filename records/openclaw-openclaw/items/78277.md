## Summary

Keep open: this targets a valid security-relevant bootstrap-token lost-update bug, but the live diff still leaves a mutating state path outside the cross-process lock, its regression test uses the reentrant in-process helper, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a cross-process file-lock wrapper around several device bootstrap token state operations and adds a bootstrap-token contention regression test.

## Best Possible Solution

Serialize every bootstrap-token read-modify-write path with one cross-process lock, add true child-process or stale-snapshot regression coverage, and require redacted setup-code proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows process-local serialization around whole-file devices/bootstrap.json writes, and Gateway accepts verified bootstrap tokens for pairing auth, so the two-process issuer/verifier interleaving is high confidence.

## Solution Assessment

No. Cross-process locking is the right narrow direction, but this PR does not lock every mutating bootstrap state path and its test does not exercise a non-reentrant cross-process race.

## Review Findings

- **[P2] Lock every bootstrap state mutation:** `src/infra/device-bootstrap.ts:316`
  - body: `revokeDeviceBootstrapTokensForDevice()` still deletes matching records and persists `devices/bootstrap.json` under the old process-local `withLock`. A rejected pending device or cleanup path can race with issuance or verification and overwrite newer token state, so route this mutating path through the cross-process bootstrap state lock too.
  - confidence: 0.9

It can pass without exercising the separate-process lost-update case, so use a child process or an explicit stale-snapshot harness that cannot re-enter the lock.
  - confidence: 0.92
