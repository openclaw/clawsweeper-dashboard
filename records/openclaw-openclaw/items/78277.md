## Summary

Keep open: this PR is the active fix path for a source-reproducible bootstrap-token race, but it still lacks real behavior proof and leaves concrete correctness/security gaps in the proposed merge result.

## What This Changes

The PR adds a cross-process file-lock wrapper around several device bootstrap token state operations and a contention test for setup-code issuance.

## Best Possible Solution

Serialize every bootstrap-token state read-modify-write path with one cross-process lock, add a true child-process or stale-snapshot regression test, and require redacted real setup-code proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows process-local locking around whole-file bootstrap state writes, and the Gateway accepts verified bootstrap tokens for pairing auth, so the linked two-process setup-code interleaving is high confidence.

## Solution Assessment

No. Cross-process locking is the right narrow direction, but this patch does not cover every mutating bootstrap state path and its regression test does not exercise a non-reentrant cross-process writer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Lock every bootstrap state mutation:** `src/infra/device-bootstrap.ts:316`
  - body: `revokeDeviceBootstrapTokensForDevice()` still deletes records and persists the bootstrap-token map under the old process-local `withLock`. A rejected pending device or cleanup path can therefore race with issuance or verification and overwrite newer token state, so route this path through `withBootstrapStateLock` too.
  - confidence: 0.9
- **[P2] Use non-reentrant contention coverage:** `src/infra/device-bootstrap.test.ts:97-99`
  - body: The new test acquires `acquireFileLock()` from the same process and helper used by the implementation, but that helper is reentrant. This does not prove the separate-process lost-update case, so use a child process or an explicit stale-snapshot harness that cannot re-enter the lock.
  - confidence: 0.92
