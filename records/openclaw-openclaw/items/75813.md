## Summary

Keep open: this PR is useful, but it is not merge-ready. The new permission helper does not match Discord's aggregate role-overwrite algorithm, so some valid channel-audience members can be denied, and the user-facing feature is missing the required changelog entry. The protected `maintainer` label also means this should not be auto-closed by cleanup.

## What This Changes

The PR adds top-level access group config for Discord channel-audience membership and uses those groups to authorize Discord DM allowlist entries that reference `accessGroup:<name>`.

## Best Possible Solution

Keep the feature in review, update the Discord permission helper to aggregate matching role overwrites before applying them, add focused coverage for conflicting role overwrites, add the Unreleased changelog entry, and then let the exact-head checks finish before maintainer merge.

## Reproduction Assessment

Yes. A focused unit test can mock a member with two roles whose channel overwrites conflict on `ViewChannel`, ordered allow then deny; Discord's aggregate role-overwrite algorithm should allow the member, while this PR's sequential loop returns false.

## Solution Assessment

No. The product direction is narrow and useful, but the implementation should compute effective Discord permissions with the documented aggregate role-overwrite semantics and include the required release note before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Aggregate role overwrites before applying them:** `extensions/discord/src/send.permissions.ts:108-112`
  - body: Discord's permission algorithm ORs all matching role overwrite denies and allows, then applies the aggregate deny mask followed by the aggregate allow mask. This loop applies each role overwrite in API order, so a user with one role allowing `ViewChannel` and a later role denying it will be rejected even though Discord's effective permission still allows the channel. That makes `accessGroup:<name>` fail for valid channel-audience members.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `docs/channels/discord.md:452-453`
  - body: This PR adds a user-facing Discord config and authorization feature, but the diff does not touch `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to add an Unreleased changelog entry, otherwise the new access-group configuration will be absent from release notes.
  - confidence: 0.99
