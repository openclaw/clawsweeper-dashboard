## Summary

Keep this PR open. Current main still lacks the requested reusable `accessGroups` and Discord channel-audience DM authorization surface, and the current PR head implements useful pieces of it, but the Discord permission helper still false-blocks guild owners because it omits the owner short circuit from Discord's permission semantics. The PR is also protected by the `maintainer` label and has failing checks, so it should stay in review rather than be closed or merged as-is.

## What This Changes

This PR adds top-level `accessGroups` config, generic message-sender access-group expansion helpers, Discord `discord.channelAudience` DM authorization, cross-channel integrations for sender groups, docs, config/API baselines, tests, and a changelog entry.

## Best Possible Solution

Keep the feature in review, add owner-aware Discord channel permission resolution using the guild owner field before role/admin/overwrite evaluation, add focused owner coverage, and rerun the targeted Discord/config/SDK checks plus the changed gate before merge.

## Reproduction Assessment

Yes. A focused test can mock a guild channel that denies `ViewChannel` to `@everyone`, a guild whose owner field matches the sender, and a member with no granting roles; the current PR helper has no owner branch, so it would return false even though Discord treats the owner as having all permissions.

## Solution Assessment

No. The feature direction is maintainable, but the current implementation is not the best fix until the Discord permission helper matches owner/admin/overwrite semantics; the narrow safer repair is an owner short circuit before role and overwrite calculations.
