## Summary

Keep this PR open. The earlier role-overwrite blocker appears addressed and the changelog entry is now present, but the new Discord channel-audience permission helper still misses Discord's guild-owner override, so valid server owners can be blocked from `accessGroup:<name>` DM authorization.

## What This Changes

The PR adds top-level `accessGroups` config, SDK helpers, docs/changelog, and channel integrations so reusable message-sender groups and Discord channel-audience groups can expand DM/group allowlists.

## Best Possible Solution

Keep the feature in review, make the Discord permission helper match Discord owner/admin/overwrite semantics, add focused coverage for the owner case, and preserve the current docs/changelog shape unless the public config wording changes.

## Reproduction Assessment

Yes. A focused unit test can mock a channel that denies `ViewChannel` to @everyone, a guild whose `owner_id` is the sender, and a member with no granting roles; Discord should allow the owner, while the PR helper returns false.

## Solution Assessment

No. The product direction is narrow and useful, but the permission implementation should include the guild-owner all-permissions rule before the PR is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor guild owners in channel audience checks:** `extensions/discord/src/send.permissions.ts:84-90`
  - body: Discord treats the guild owner as having all permissions before channel overwrites are applied. This helper only derives permissions from @everyone/member roles and Administrator, so a server owner with no granting role can be rejected from a `discord.channelAudience` access group even though they can view the channel. Include `owner_id` in the guild data and add a test for the owner path.
