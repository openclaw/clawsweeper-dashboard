## Summary

Keep open. Current main still lacks a provider-level `models.providers.<id>.enabled` contract, and the relevant planning, fallback, catalog, and status paths still treat every configured provider as active; the earlier implementation PR was closed unmerged, so this remains the active tracker.

## What This Changes

Add an optional provider-level `models.providers.<id>.enabled` flag that defaults to true and preserves disabled provider config while excluding it from model generation, fallback, catalogs, and status/probe output.

## Best Possible Solution

Add a documented `models.providers.<provider>.enabled?: boolean` config contract that defaults to enabled, preserves disabled config blocks, filters disabled providers from generated `models.json`, configured model catalogs, fallback candidates, list/status/probe paths, and exposes the disabled state in both JSON and rich `openclaw models status` output.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no `enabled` field in the provider type or strict schema, and the active planning/fallback/catalog/status code paths iterate every configured provider without a disabled gate.

## Solution Assessment

Yes, the requested provider-level flag is the narrowest maintainable solution if it is implemented consistently across config validation, model generation, fallback/catal og building, status output, docs, and tests. A safer implementation should avoid changing auth semantics or provider plugin discovery beyond filtering config-disabled providers where the operator expects them to be inactive.
