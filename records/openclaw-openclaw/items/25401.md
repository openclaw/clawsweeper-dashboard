## Summary

Keep open. Current main and the latest release still do not define or consistently honor a provider-level `models.providers.<id>.enabled` contract, and the two linked implementation PRs are closed unmerged, so this remains active product/config work.

## What This Changes

Request a default-true `models.providers.*.enabled` config flag that disables a provider without deleting its config and reflects that state in model resolution, fallback, catalog, and status output.

## Best Possible Solution

Approve and document a default-true provider enablement contract, then implement it consistently across config type/schema, models.json planning, configured catalogs, fallback/runtime resolution, status/probe output, docs, and regression tests.

## Reproduction Assessment

Yes for the source-level capability gap. Current main omits `enabled` from the provider type/schema and carries configured providers through planning, fallback, catalog, and status paths without a disabled-provider gate.

## Solution Assessment

Yes directionally, but not without maintainer approval. A provider-level default-true boolean is a maintainable solution only if every model/provider surface consumes the same state and the docs/tests define the contract.
