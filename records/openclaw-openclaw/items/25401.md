## Summary

Keep open. Current main still lacks a provider-level `models.providers.<id>.enabled` contract, the runtime paths still treat every configured provider as active, and the two implementation PRs found for this request are closed unmerged.

## What This Changes

The issue requests a default-enabled provider-level config flag so a configured model provider can be temporarily disabled without deleting its config block.

## Best Possible Solution

Approve a default-true provider enablement contract, then implement it consistently across config types/schema, models.json planning, fallback/catalog selection, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows `models.providers.<id>.enabled` is absent from the strict provider type/schema and the planning, fallback, catalog, docs, and status paths have no disabled-provider check.

## Solution Assessment

Yes directionally, but not as automated cleanup. A provider-level default-true boolean is maintainable only after maintainers approve the public config/status contract and all runtime surfaces honor it together.
