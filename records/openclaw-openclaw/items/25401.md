## Summary

Keep open. Current main and the latest tagged release still do not expose a provider-level enablement contract, and the two related implementation PRs are closed unmerged, so the remaining work is a product/config decision rather than cleanup.

## What This Changes

The issue requests a default-enabled `models.providers.<id>.enabled` boolean to disable a configured model provider without deleting its saved config.

## Best Possible Solution

Approve and document a default-true provider enablement contract, then implement it consistently across config types/schema, models.json planning, configured catalogs, fallback selection, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows the strict provider type/schema omit `enabled`, and planning, fallback, catalog, status, docs, and focused tests do not define a provider-config disabled state.

## Solution Assessment

Yes directionally, but not as automated cleanup. A provider-level default-true boolean is coherent only after maintainers approve the public config/status contract and all runtime surfaces honor it together.
