## Summary

Keep open. Current `main` and the latest release still lack a provider-level disable contract, the two related implementation PRs are closed unmerged, and the remaining work needs a maintainer decision on disabled-provider semantics before code should land.

## What This Changes

Request an optional `models.providers.<id>.enabled` config flag that disables a model provider while preserving its config block for later re-enable.

## Best Possible Solution

Approve one documented provider eligibility contract that defaults omitted values to enabled, preserves disabled config, and applies consistently across schema, models.json generation, runtime resolution, fallback, catalog/listing, auth/probe, status output, and docs.

## Reproduction Assessment

Yes, at source level. Current `main` lacks `models.providers.<id>.enabled` in the strict type/schema and the planning, fallback, catalog/listing, status, and probe paths have no disabled-provider eligibility check.

## Solution Assessment

Unclear pending maintainer product direction. A provider-level flag is the narrowest requested UX, but it should be implemented as one shared eligibility contract rather than scattered filters in each consumer.
