## Summary

Keep open. Current main still lacks a provider-level disabled-provider contract, the two implementation attempts linked from the discussion were closed unmerged, and no narrower open or merged item now owns the same requested behavior.

## What This Changes

The issue requests a default-enabled `models.providers.<id>.enabled` flag so users can temporarily disable a provider without deleting its config block.

## Best Possible Solution

Approve a default-true provider-enabled contract, then implement it through a shared helper across schema/types, models.json planning, selection/fallback, configured catalogs, embedded runner, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `enabled` is not a provider-level type/schema field and the central runtime paths still read every configured provider; no live command was run because this review is read-only.

## Solution Assessment

Yes in direction. A default-true provider-level boolean is the narrowest maintainable shape, but it needs to land as a complete public config/status contract rather than a partial type-only change.
