Keep open. Current main still lacks the requested provider-level enablement contract, both implementation PRs found in the related/search pass are closed unmerged, and the remaining work is a public config/status behavior decision rather than conservative cleanup.

## What This Changes

The issue requests a default-enabled `models.providers.<provider>.enabled` flag that preserves provider config while excluding disabled providers from model resolution, fallback, catalogs, and status output.

## Best Possible Solution

Approve and document a default-true provider enablement contract, then implement it consistently across config types/schema, models.json planning, configured catalogs, fallback selection, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows the strict provider type/schema omit `enabled`, and planning, fallback, catalog, status, docs, and focused tests have no provider-config disabled state.

## Solution Assessment

Yes directionally, but not as automated cleanup. A provider-level default-true boolean is coherent only after maintainers approve the public config/status contract and all runtime surfaces honor it together.
