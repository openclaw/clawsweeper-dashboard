## Summary

Keep open. Current main and the latest release still do not define or consume a provider-level `models.providers.<id>.enabled` contract, and both exact implementation attempts are closed unmerged, so this remains the canonical tracker pending maintainer product direction.

## What This Changes

Request a provider-level `models.providers.<id>.enabled?: boolean` setting that temporarily disables configured model providers without deleting their config.

## Best Possible Solution

Approve one shared provider eligibility contract that defaults omitted values to enabled, preserves disabled config blocks, and consistently gates schema, models.json planning, runtime model resolution, fallback, catalog/listing, auth/probe, status output, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main's strict provider config type/schema does not include `enabled`, and planning, fallback, catalog/listing, and status paths consume configured providers without a disabled-provider eligibility gate. I did not run a live config because this review is read-only.

## Solution Assessment

Unclear pending maintainer product direction. A provider-level flag is a plausible solution, but the maintainable fix should be a shared provider eligibility contract rather than scattered per-consumer filters.
