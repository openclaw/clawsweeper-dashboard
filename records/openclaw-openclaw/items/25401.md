## Summary

Keep open. Current main still lacks the requested provider-level `models.providers.<id>.enabled` contract, and the related implementation attempts are closed unmerged rather than shipped fixes.

## What This Changes

Add a provider-level `models.providers.<id>.enabled` boolean so users can temporarily disable a configured model provider without deleting its config block.

## Best Possible Solution

Add a documented provider-enabled contract that defaults to enabled, preserves disabled config blocks, excludes disabled providers from runtime selection and generated catalogs, and reports disabled state in status output.

## Reproduction Assessment

Yes. Source inspection at current main shows `enabled` is not accepted by the provider type or strict schema, and the planning, fallback, catalog, and status paths have no disabled-provider gate.

## Solution Assessment

Yes. A provider-level boolean defaulting to enabled is the narrowest maintainable direction, provided the implementation updates the full config/schema, runtime filtering, status/probe, docs, and regression-test contract.
