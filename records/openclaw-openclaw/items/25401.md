## Summary

Keep open. Current main and v2026.6.10 still lack a provider-level `models.providers.<id>.enabled` contract, both known implementation PRs are closed unmerged, and the remaining work needs maintainer approval for public config and fallback semantics.

## What This Changes

Request a provider-level `enabled: false` flag that disables a configured model provider without deleting its config block.

## Best Possible Solution

Approve one shared provider eligibility contract that defaults omitted values to enabled, preserves disabled config blocks, and consistently gates schema, planning, runtime resolution, fallback, catalog/listing, status/probe output, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current main's strict provider config type/schema does not include `enabled`, and the planning, fallback, catalog, and status paths consume configured providers without a disabled-provider eligibility gate. I did not run a live config because this review was read-only.

## Solution Assessment

Unclear pending maintainer product direction. A provider-level flag is a plausible narrow UX, but the best implementation should be a shared provider eligibility contract rather than scattered per-consumer filters.
