## Summary

Keep open. Current main and the latest release still do not define or honor `models.providers.<id>.enabled`, and the related implementation PRs remain closed unmerged, so the requested config/status behavior is still active product work.

## What This Changes

Add a default-true `models.providers.<id>.enabled` flag that preserves provider config while excluding disabled providers from model resolution, fallback, catalogs, and status as appropriate.

## Best Possible Solution

Approve and document a default-true provider enablement contract, then implement it consistently across config type/schema, models.json planning, configured catalogs, fallback/runtime resolution, status/probe output, docs, and regression tests.

## Reproduction Assessment

Yes for the source-level capability gap. Current main omits `enabled` from the provider type/schema and carries every configured provider through planning, fallback, catalog, and status paths without a disabled gate.

## Solution Assessment

Yes directionally, but it needs maintainer approval before implementation. A provider-level default-true boolean is maintainable only if all runtime and status surfaces honor the same disabled-provider state.
