Keep open. Current main and v2026.5.20 still lack a provider-level enablement contract, and both related implementation PRs are closed unmerged, so this remains an active product/config decision rather than conservative cleanup.

## What This Changes

Request a default-true `models.providers.<id>.enabled` config flag that can temporarily disable a model provider while preserving its config block.

## Best Possible Solution

Approve and document a default-true provider enablement contract, then implement it consistently across config type/schema, models.json planning, configured catalogs, fallback selection, runtime model resolution, status/probe output, docs, and focused tests.

## Reproduction Assessment

Yes for the source-level capability gap. Current main omits `enabled` from the provider type/schema and carries every configured provider through planning, fallback, catalog, and status paths.

## Solution Assessment

Yes directionally, but it needs maintainer approval before implementation. A provider-level default-true boolean is a coherent solution only if all runtime and status surfaces honor the same disabled-provider state.
