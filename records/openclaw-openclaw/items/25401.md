## Summary

Keep open. Current main and the latest release still do not define or honor a provider-level disable contract, and the two known implementation attempts are closed unmerged; the remaining blocker is maintainer approval for a new public config flag and its disabled-primary/fallback semantics.

## What This Changes

Issue requests a provider-level `models.providers.<id>.enabled` config flag to disable a configured model provider without deleting its config.

## Best Possible Solution

Approve one shared provider eligibility contract that defaults omitted values to enabled, preserves disabled config blocks, and consistently gates schema, planning, runtime resolution, fallback, catalog/listing, status/probe output, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current main's provider config type/schema lacks `enabled`, and the planning, fallback, catalog, auth/status, and release-tag checks show configured providers are consumed without a disabled-provider eligibility gate. I did not run a live config because this review was read-only.

## Solution Assessment

Unclear pending maintainer product direction. A provider-level flag is a plausible narrow UX, but the best implementation should be a shared provider eligibility contract rather than scattered per-consumer filters.
