## Summary

Keep open. Current main still lacks a provider-level `models.providers.<id>.enabled` contract, and the related implementation PR was closed unmerged while the type, strict schema, models.json planning, fallback, catalog/status paths, and docs still do not expose the requested disabled-provider behavior.

## What This Changes

Add an optional provider-level `models.providers.<id>.enabled` flag so configured providers can be temporarily disabled without deleting their config.

## Best Possible Solution

Add a documented config contract that defaults providers to enabled, preserves disabled config blocks, consistently excludes disabled providers from runtime selection surfaces, and reports disabled state in status output.

## Reproduction Assessment

Yes. Source inspection on current main shows `enabled: false` is not an accepted provider-level schema/type field and no current planning, fallback, catalog, or status path can honor it.

## Solution Assessment

Yes. A provider-level boolean defaulting to enabled is the narrowest maintainable direction, provided the implementation updates the full config, runtime filtering, status/probe, docs, and regression-test contract.
