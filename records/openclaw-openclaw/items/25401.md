## Summary

Keep open. Current main and the latest stable release still do not define or consume a provider-level `models.providers.<id>.enabled` contract, both exact implementation attempts are closed unmerged, and the remaining work needs a maintainer decision on public config semantics before implementation.

## What This Changes

Request a new provider-level `models.providers.<id>.enabled` boolean that preserves configured providers while excluding disabled providers from model resolution, fallback, `models.json`, and status output.

## Best Possible Solution

Approve one shared provider eligibility contract that defaults omitted values to enabled, preserves disabled config blocks, and consistently gates schema, planning, runtime resolution, fallback, catalog/listing, auth/probe, status output, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current main's strict provider config type/schema does not include `enabled`, and the planning, fallback, catalog, and status paths consume configured providers without a disabled-provider eligibility gate. I did not run a live config because this review was read-only.

## Solution Assessment

Unclear pending maintainer product direction. A provider-level flag is a plausible narrow UX, but the best implementation should be a shared provider eligibility contract rather than scattered per-consumer filters.
