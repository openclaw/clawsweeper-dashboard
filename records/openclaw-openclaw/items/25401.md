## Summary

Keep open. Current main and v2026.5.22 still lack a provider-level `enabled` contract, and the two implementation PRs that referenced this issue are closed unmerged, so the remaining work is an active product/config decision rather than conservative cleanup.

## What This Changes

Request a provider-level `models.providers.<id>.enabled` flag to disable a model provider while preserving its config block.

## Best Possible Solution

Approve and document a default-true provider enablement contract, then implement it consistently across config type/schema, models.json planning, configured catalogs, fallback/model resolution, status/probe output, docs, and focused tests.

## Reproduction Assessment

Yes for the source-level capability gap. Current main omits `enabled` from the provider type/schema and carries configured providers through planning, fallback, catalog, and status paths without a disabled-provider check.

## Solution Assessment

Yes directionally, but it needs maintainer approval before implementation. A provider-level default-true boolean is coherent only if every runtime and status surface honors the same disabled-provider state.
