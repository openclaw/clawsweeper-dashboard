## Summary

Keep open. Current main and v2026.6.6 still lack a provider-level disable contract, the two exact implementation attempts are closed unmerged, and the remaining work needs a maintainer decision on disabled-provider semantics before implementation.

## What This Changes

Request adding `models.providers.<id>.enabled?: boolean` so provider config can be preserved while disabled providers are excluded from model generation, fallback, catalog, and status paths.

## Best Possible Solution

Approve one shared provider eligibility contract that defaults omitted values to enabled, preserves disabled config blocks, and enforces disabled-provider behavior consistently across schema, planning, runtime resolution, fallback, catalog/listing, auth/probe, status output, and docs.

## Reproduction Assessment

Yes, at source level. Current main's strict provider config type/schema lacks `enabled`, and the planning, fallback, catalog/listing, runtime lookup, and status paths consume configured providers without a disabled-provider eligibility check.

## Solution Assessment

Unclear pending maintainer product direction. A provider-level flag is the narrowest requested UX, but it should be implemented as one shared eligibility contract rather than scattered filters in each consumer.
