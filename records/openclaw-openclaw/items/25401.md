## Summary

Keep open. Current main still lacks a provider-level `enabled` contract, and this is a new public config/status capability already marked for maintainer product review rather than conservative cleanup.

## What This Changes

Add an optional default-true `models.providers.<id>.enabled` flag that preserves provider config while excluding disabled providers from model generation, fallback/catalog selection, and status output.

## Best Possible Solution

Approve a default-true provider enablement contract, then implement it consistently across config types/schema, models.json planning, fallback/catalog selection, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes, for the reported gap. Source inspection on current main shows `models.providers.<id>.enabled` is absent from the provider type/schema and the central planning, fallback, catalog, and status paths have no disabled-provider check.

## Solution Assessment

Yes directionally, but not as an automated cleanup. A default-true provider-level boolean is the narrowest maintainable shape only after maintainers approve the new public config/status contract and all runtime surfaces honor it.
