## Summary

Keep open. Current main still lacks the requested provider-level disable contract, and the two implementation attempts linked from the discussion were both closed unmerged.

## What This Changes

The issue requests a default-true `models.providers.<id>.enabled` config flag that preserves provider config while excluding disabled providers from resolution, fallback, catalogs, and status output.

## Best Possible Solution

Add a documented default-true provider-enabled contract, define explicit-pin behavior, and apply it consistently across schema/types, models.json planning, fallback selection, configured catalogs, status/probe output, embedded runner resolution, docs, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the provider type/schema omit `enabled`, and the central planning, fallback, catalog, status, and embedded-runner paths still have no disabled-provider gate; no runtime command was run because this review is read-only.

## Solution Assessment

Yes in direction. A provider-level boolean defaulting to enabled is the narrowest maintainable shape, but it should be implemented as a complete public config contract rather than a type-only change.
