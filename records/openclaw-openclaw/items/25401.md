## What This Changes

Add an optional `models.providers.<id>.enabled` setting that preserves provider configuration while excluding disabled providers from model catalogs, routing, fallbacks, probes, and status output.

## Best Possible Solution

Approve one canonical provider-enabled contract, default omitted values to active, preserve disabled blocks, and enforce the state through a shared early provider-selection boundary used by catalog generation, runtime resolution, fallbacks, auth/probes, and status reporting.

## Reproduction Assessment

Yes, at source level: current main's strict provider type/schema has no `enabled` field, and provider planning and status enumeration consume all configured providers. No runtime command was executed because this review is read-only.

## Solution Assessment

Unclear pending the product decision. A provider-level flag is the narrowest user-facing shape proposed, but it is only the best fix if implemented through one shared eligibility contract rather than separate filters in each consumer.
