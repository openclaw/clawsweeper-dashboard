## Summary

Keep open. Current main and the latest release still lack a provider-level enablement contract, and the two implementation PRs that attempted this feature are closed unmerged, so this remains an active config/product decision rather than cleanup.

## What This Changes

The issue requests a `models.providers.<provider>.enabled?: boolean` config flag that preserves provider config while excluding disabled providers from model resolution, fallback lists, and status output.

## Best Possible Solution

Approve and document a default-true provider enablement contract, then implement it consistently across config type/schema, models.json planning, configured model catalogs, fallback/model resolution, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes for the source-level capability gap. Current main omits `enabled` from the provider type/schema and carries every configured provider through planning, fallback, catalog, and status paths.

## Solution Assessment

Yes directionally, but it needs maintainer approval before implementation. A provider-level default-true boolean is maintainable only if all runtime and status surfaces honor the same disabled-provider state.
