## Summary

Keep open. Current main still lacks a provider-level disabled-provider contract, both linked implementation attempts were closed unmerged, and no canonical replacement issue or merged PR now owns the remaining work.

## What This Changes

The issue requests a default-true `models.providers.<id>.enabled` boolean that preserves provider config while excluding disabled providers from model planning, fallback, catalogs, and status output.

## Best Possible Solution

Approve a default-true provider-enabled contract and implement it through a shared config helper across schema/types, models.json planning, model selection, fallback, configured catalogs, status/probe output, embedded runner resolution, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `enabled: false` is not a provider-level type/schema field and the central runtime paths still read every configured provider; no runtime command was run because this review is read-only.

## Solution Assessment

Yes in direction. A default-true provider-level boolean is the narrowest maintainable shape, but it should be landed as a complete public config/status contract rather than a type-only change.
