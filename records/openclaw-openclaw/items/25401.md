## Summary

Keep open. Current main still lacks a provider-level `models.providers.<id>.enabled` contract, and the core planning, fallback, catalog, and status paths continue to treat every configured provider as active; the related implementation PR was closed unmerged.

## What This Changes

Add an optional `models.providers.<provider>.enabled` flag so operators can disable a configured provider without deleting its config.

## Best Possible Solution

Add a documented provider-level enabled flag that defaults to active, preserves disabled config blocks, and consistently filters disabled providers from generation, selection, catalog, list/status/probe output, and fallback behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no provider-level `enabled` type/schema contract, and the relevant planning, fallback, catalog, and status code paths iterate configured providers without a disabled gate.

## Solution Assessment

Yes. The requested provider-level flag is the narrowest maintainable solution if implemented consistently across config validation, model generation, fallback/catalog selection, status/probe output, docs, and tests while leaving auth semantics unchanged.
