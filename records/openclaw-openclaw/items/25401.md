## Summary

Keep open. Current main still does not define or honor a provider-level `models.providers.<id>.enabled` contract, and both known implementation attempts are closed unmerged; this is a public config/status feature that still needs maintainer product approval.

## What This Changes

The issue requests a provider-level `models.providers.<id>.enabled` flag so configured providers can be disabled without deleting their configuration.

## Best Possible Solution

Approve a default-true provider enablement contract, then implement it consistently across config types/schema, models.json planning, fallback/catalog selection, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows `models.providers.<id>.enabled` is absent from the strict provider type/schema and the planning, fallback, catalog, docs, and status paths have no disabled-provider check.

## Solution Assessment

Yes directionally, but not as automated cleanup. A provider-level default-true boolean is a maintainable shape only after maintainers approve the public config/status contract and all runtime surfaces honor it together.
