## Summary

Keep open. Current main still lacks a provider-level `models.providers.<id>.enabled` contract, and this remains a public config/status feature that needs maintainer product approval rather than conservative cleanup.

## What This Changes

Add an optional default-true `models.providers.<id>.enabled` flag to disable a configured provider without deleting its config block.

## Best Possible Solution

Approve a default-true provider enablement contract, then implement it consistently across config types/schema, models.json planning, fallback/catalog selection, status/probe output, docs, and focused regression tests.

## Reproduction Assessment

Yes for the reported capability gap: source inspection on current main shows `models.providers.<id>.enabled` is absent from the provider type/schema and the central planning, fallback, catalog, docs, and status paths have no disabled-provider check. I did not run a runtime repro because this read-only review could prove the gap directly from source.

## Solution Assessment

Yes directionally, but not as automated cleanup. A default-true provider-level boolean is a maintainable shape only after maintainers approve the public config/status contract and all runtime surfaces honor it together.
