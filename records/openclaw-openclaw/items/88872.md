## What This Changes

The PR threads the resolved subagent target agent id through subagent task registration and makes task creation infer agent_id from childSessionKey before falling back to parent ownership.

## Best Possible Solution

Land the narrow attribution fix after maintainer review and check-gated merge handling, preserving parent owner/requester fields while storing the executing child agent in task_runs.agent_id for new rows.

## Reproduction Assessment

Yes. Current main builds cross-agent childSessionKey values from targetAgentId, but task creation resolves agentId from explicit agent id, ownerKey, or requesterSessionKey, so the source path reproduces the wrong parent attribution without running a write-producing repro.

## Solution Assessment

Yes. Passing targetAgentId through the existing subagent registration seam plus childSessionKey inference in task creation is the narrow maintainable fix; reader-side parsing of childSessionKey should remain a workaround, not the canonical persisted field.
