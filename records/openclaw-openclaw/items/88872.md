## Summary

Keep open: current main still derives task agent attribution from the parent owner/requester path, while this PR is a narrow implementation for the linked open task attribution issue and I found no blocking patch defect.

## What This Changes

The PR attributes spawned subagent task records to the executing child agent by passing `targetAgentId` through subagent registration and inferring from `childSessionKey` before parent ownership, with registry and SQLite regression tests.

## Best Possible Solution

Land this narrow attribution fix after required checks pass, preserving parent owner/requester fields while making `task_runs.agent_id` reflect the executing child agent.

## Reproduction Assessment

Yes, at source level: current main builds a child session key from the target agent but task creation resolves `agentId` from explicit, owner, then requester fields without considering the child key. I did not run a live `sessions_spawn` turn in this read-only review.

## Solution Assessment

Yes, this is the best narrow fix: task attribution belongs in the core task creation/registration path because `task_runs.agent_id` is already a persisted field and public task summary attribute. Reader-side parsing of `childSessionKey` remains a workaround, not the canonical source.
