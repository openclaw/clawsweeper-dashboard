Keep open for normal PR review: current main still derives task attribution from explicit agent, owner, then requester, while this patch routes spawned subagent task rows to the executing child agent and adds focused registry, SQLite, and spawn-seam coverage. I found no blocking patch findings.

## What This Changes

The branch passes targetAgentId through subagent run registration, infers task agentId from childSessionKey, and adds task registry, SQLite, and spawn seam regression tests.

## Best Possible Solution

Land the narrow attribution fix after the normal check gate, preserving parent owner/requester fields while persisting the executing child agent in task_runs.agent_id.

## Reproduction Assessment

Yes, source-reproducible: current main ignores childSessionKey in resolveTaskAgentId, so a parent-owned spawned task without explicit agentId resolves to the parent instead of the child. I did not run a live failing session because this was a read-only review.

## Solution Assessment

Yes: passing targetAgentId at the sessions_spawn registration seam and using childSessionKey as the generic fallback fixes the owner boundary without plugin label inference or a new config surface.
