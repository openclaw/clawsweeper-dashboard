## What This Changes

The PR threads the target child agent id through subagent run registration and task-record creation so spawned subagent task rows are attributed to the executing child agent.

## Best Possible Solution

Land the narrow attribution fix after maintainer review, preserving parent owner/requester fields while storing the executing child agent in task_runs.agent_id for new spawned task rows.

## Reproduction Assessment

Yes, at source level: current main builds childSessionKey from targetAgentId but task creation resolves agentId from explicit, owner, then requester fields only. I did not run a write-producing live repro because this cleanup review must keep the checkout read-only.

## Solution Assessment

Yes. Passing targetAgentId through the existing subagent registration seam and deriving from childSessionKey inside task creation is the narrow maintainable fix; plugin-side parsing of child_session_key should remain a workaround, not the persisted source of truth.
