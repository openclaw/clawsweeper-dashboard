## Summary

Keep open. Current main and the latest release still do not enforce the configured Feishu webhook path, and https://github.com/openclaw/openclaw/pull/54904 is open, unmerged, and explicitly tracks the fix.

## What This Changes

The issue asks Feishu webhook mode to reject requests whose URL path does not match configured `channels.feishu.webhookPath` before payload parsing or dispatch.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54904, or an equivalent Feishu plugin patch, with early path normalization/rejection and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the handler never compares the incoming URL path with `webhookPath`, so a correctly signed POST to another path can continue through body parsing, signature validation, and dispatch.

## Solution Assessment

Yes. The narrow maintainable solution is Feishu-owned route enforcement before generic guards, body parsing, signature validation, challenge handling, and dispatch; no broader SDK or product decision is needed.
