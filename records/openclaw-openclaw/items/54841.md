## Summary

Keep open. Current main still does not enforce the configured Feishu webhookPath, and PR #54904 remains open and unmerged with the targeted fix, so this issue should stay open until that PR or an equivalent Feishu-owned patch lands.

## What This Changes

The issue asks Feishu webhook mode to reject requests whose URL pathname does not match channels.feishu.webhookPath before body parsing, signature validation, challenge handling, or dispatch.

## Best Possible Solution

Land PR #54904, or an equivalent Feishu plugin patch, that normalizes the configured webhookPath, parses the incoming request pathname, returns 404 for mismatches before body parsing/signature validation/dispatch, preserves query-string acceptance on the configured path, and adds focused regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure Feishu webhook mode with a custom webhookPath and send a correctly signed POST to a different path on the same listener; current monitorWebhook has no path comparison before dispatch.

## Solution Assessment

Yes. Path enforcement belongs in the Feishu webhook ingress path before body parsing and signature work; the open PR direction described in the GitHub context is the narrow maintainable fix, while a broader SDK migration is not required for this bug.
