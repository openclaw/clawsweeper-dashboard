## Summary

Keep open: current main and v2026.5.18 still do not enforce the configured Feishu webhookPath before processing signed webhook requests, and an open unmerged PR already tracks the exact fix with closing syntax.

## What This Changes

The issue requests Feishu webhook mode to reject requests whose URL path does not match the configured channels.feishu.webhookPath before parsing or dispatching the webhook.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54904, or an equivalent Feishu plugin patch, with early path normalization/rejection before body parsing, signature validation, challenge handling, and dispatch plus focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows monitorWebhook never compares req.url or URL.pathname to webhookPath before body parsing, signature validation, challenge handling, and dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-owned early path normalization and mismatch rejection before the existing webhook pipeline; the linked open PR is the current implementation path to review.
