## Summary

Keep open: current main still lacks Feishu webhook path enforcement, and an open linked implementation PR already targets this exact bug, so the issue should wait for that PR or an equivalent fix.

## What This Changes

The issue asks Feishu webhook mode to reject requests whose URL path does not match channels.feishu.webhookPath before processing the webhook payload.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54904, or an equivalent Feishu plugin patch, that normalizes the configured path, rejects mismatches before body parsing/signature validation/dispatch, preserves query-string acceptance on the configured path, and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows monitorWebhook never compares req.url or URL.pathname to webhookPath before body parsing, signature validation, challenge handling, and dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-owned early path normalization and mismatch rejection before the existing webhook processing pipeline; the linked open PR is the current candidate to review.
