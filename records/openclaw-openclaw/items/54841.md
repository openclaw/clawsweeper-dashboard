## Summary

Keep open. Current `main` and the latest release still do not enforce Feishu `webhookPath`, and the open implementation PR that explicitly closes this issue has not merged.

## What This Changes

The issue asks Feishu webhook mode to reject requests whose URL path does not match the configured `channels.feishu.webhookPath`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54904, or an equivalent Feishu plugin patch, that normalizes the configured path, rejects mismatches before body parsing/signature validation/dispatch, preserves query-string acceptance, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `monitorWebhook` reads `webhookPath` but never compares the incoming request path before body parsing, signature validation, challenge handling, or dispatch, so a signed `POST /wrong-path` scenario is high-confidence.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-owned route enforcement at the webhook ingress boundary before body parsing and signature work; no broader SDK or product change is required.
