## Summary

Keep open for maintainer review: current `main` and latest release still run implicit provider discovery before honoring replace mode, and this PR is a narrow, well-proven candidate fix. The remaining blocker is compatibility acceptance for users who may have relied on discovered provider/plugin catalog rows despite the documented explicit-only replace-mode contract.

## What This Changes

The PR adds an early replace-mode return in agent provider planning and a focused Vitest mode matrix proving replace skips implicit provider discovery while merge/default still invoke it.

## Best Possible Solution

Land the early discovery guard with focused mode coverage once maintainers accept explicit-only replace-mode upgrade behavior and current merge validation is satisfactory.

## Reproduction Assessment

Yes. Source inspection on current `main` and `v2026.6.10` shows implicit discovery is awaited before replace-mode handling, and the PR proof shows the patched production resolver path performs zero discovery calls in replace mode.

## Solution Assessment

Yes as an implementation. Guarding before implicit discovery is the narrowest maintainable fix, avoids the startup work rather than filtering after it, and matches the model-picker precedent; the remaining question is maintainer acceptance of the compatibility effect.
