- `merge-risk: 🚨 compatibility`: Existing replace-mode users will intentionally stop receiving runtime-discovered provider entries after upgrade.

## Summary

This PR is still necessary: current main and the latest release still call implicit provider discovery before honoring `models.mode: "replace"`, while the branch adds a narrow early guard and mode-matrix coverage. I found no blocking patch defect, but the behavior change is compatibility-sensitive because replace-mode users will stop receiving runtime-discovered provider catalog entries.

## What This Changes

The PR adds an early return in agent model-provider planning so `models.mode: "replace"` skips implicit provider discovery, plus a new Vitest file covering replace, merge, and default modes.

## Best Possible Solution

Land the early discovery guard with the mode-matrix regression test after maintainers confirm the explicit-only replace-mode upgrade semantics and required checks are acceptable.

## Reproduction Assessment

Yes, from source with high confidence: current main awaits implicit provider discovery before any replace-mode provider planning can take effect. I did not run the full reported startup scenario, but the contributor supplied production-path output showing the resolver is skipped after the patch.

## Solution Assessment

Yes, this is the best fix shape: the early guard sits at the provider-gathering boundary, before the slow implicit discovery callee, while leaving explicit-provider normalization, secret handling, catalog splitting, and merge/default behavior on the existing path.
