## Summary

Keep open: current main and the latest release still lack the replace-mode discovery guard, and this PR is the focused active fix. I found no blocking patch defect, but maintainers still need to accept the compatibility change for users who accidentally depended on discovered providers in replace mode.

## What This Changes

The PR adds a replace-mode early return in model-provider planning and a colocated Vitest matrix ensuring implicit discovery is skipped only for `models.mode: "replace"`.

## Best Possible Solution

Land the early discovery guard with the mode-matrix regression coverage after maintainers accept the explicit-only replace-mode upgrade behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible because `resolveProvidersForModelsJsonWithDeps` awaits implicit discovery before mode handling, and the contributor supplied production-path live output showing the patched replace path performs zero discovery calls.

## Solution Assessment

Yes. Guarding at the provider-gathering boundary is the narrowest maintainable fix because later planner mode handling would still pay the discovery cost; the model picker already uses the same replace-mode short-circuit pattern.
