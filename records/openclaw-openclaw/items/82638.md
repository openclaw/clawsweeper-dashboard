## Summary

Keep open: the patch looks correct and proof-backed, but it intentionally changes upgrade behavior for replace-mode users, so maintainers should explicitly accept or reject that compatibility decision before merge.

## What This Changes

The branch adds an early replace-mode guard in model-provider planning so explicit providers are returned without running implicit provider discovery, plus a mode-matrix regression test.

## Best Possible Solution

Land the early discovery guard and mode-matrix coverage after maintainers accept explicit-only replace-mode upgrade behavior; otherwise close this PR and keep the canonical issue open for a different compatibility direction.

## Reproduction Assessment

Yes. Source inspection on current main shows resolveProvidersForModelsJsonWithDeps awaits implicit discovery before any replace-mode handling, and the contributor supplied production-path output showing the patched replace path performs zero discovery calls.

## Solution Assessment

Yes, subject to maintainer compatibility acceptance. Guarding at the provider-gathering boundary is the narrowest maintainable fix because later planner mode handling would still pay the discovery cost, and the model picker already uses the same replace-mode short-circuit pattern.
