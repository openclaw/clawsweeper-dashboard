## Summary

Keep open for maintainer review: current main and the latest release still run implicit provider discovery before replace-mode handling, while this PR adds a narrow guard with sufficient production-path proof and no blocking code-review findings. The remaining concern is an intentional compatibility change for users who were accidentally relying on discovered providers in replace mode.

## What This Changes

The PR short-circuits provider planning so `models.mode: "replace"` returns explicit providers without implicit discovery, and adds replace/merge/default-mode regression coverage.

## Best Possible Solution

Land the early discovery guard with the mode-matrix regression test if maintainers accept the documented explicit-only replace-mode semantics.

## Reproduction Assessment

Yes. Current main clearly awaits implicit provider discovery before any replace-mode branch, and the contributor supplied production-path output showing replace mode reaches zero discovery calls after the patch.

## Solution Assessment

Yes. The guard is at the provider-gathering boundary, so it avoids discovery in replace mode while preserving the canonical downstream normalization, secret-management, and catalog-writing pipeline.
