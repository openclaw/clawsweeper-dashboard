## Summary

Keep open. Current main only implements the concurrency part through `agents.defaults.memorySearch.remote.nonBatchConcurrency`; the retry max-attempt, base-delay, and max-delay values are still hardcoded in memory-core, and #68822 remains an open, unmerged implementation candidate that targets this issue.

## What This Changes

The issue asks for memory embedding indexing concurrency and retry backoff parameters to be configurable instead of hardcoded defaults.

## Best Possible Solution

Keep the issue open while #68822 or a successor is reviewed. The end state should extend the existing `agents.defaults.memorySearch` contract with validated retry tuning fields, wire them into the memory-core retry loop, preserve the shipped non-batch concurrency behavior, and update schema/help/docs plus focused tests before merge.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the missing configuration path: retry parameters are constants in memory-core, while schema/docs expose only concurrency and batch/timeout options. A live Gemini 429 is not needed to verify the requested retry knobs are unavailable.

## Solution Assessment

Unclear as proposed. Configurable retry behavior is the right direction, but the maintainable implementation should align with the current `agents.defaults.memorySearch` config shape rather than adding an unrelated top-level `memory.embedding` section without maintainer agreement.
