## Summary

Keep open. The linked reconnect problem is source-reproducible on current main, but this PR is not mergeable as written because the new pause payload violates the `GatewayReconnectPausedInfo` type and the hard stop would strand node-host clients after a long outage.

## What This Changes

The PR adds a hard-coded 30-attempt reconnect cap to `GatewayClient.scheduleReconnect`, resets the counter on stop and hello, and adds two GatewayClient tests.

## Best Possible Solution

Define a caller-aware reconnect exhaustion policy that preserves node-host recovery or supervisor restart, emits a typed terminal reason, and locks it with valid reconnect and reset tests plus changelog coverage.

## Reproduction Assessment

Yes for the unbounded reconnect path: source inspection shows ordinary close events repeatedly call `scheduleReconnect()` with no attempt limit. The separate zombie-process/listener accumulation impact is not independently reproduced against current main.

## Solution Assessment

No. A finite retry budget may be useful, but this diff is not the best current fix because it has a type error, can strand node-host processes, and does not test the reset path with a valid hello response.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Use a valid reconnect pause detail code:** `src/gateway/client.ts:937`
  - body: `GatewayReconnectPausedInfo.detailCode` is a required `string | null`, but this new branch passes `undefined`. With the current exported type, this should fail the GatewayClient typecheck; use `null` or add a typed terminal detail code and update callers accordingly.
  - confidence: 0.96
- **[P2] Preserve node-host recovery on retry exhaustion:** `src/gateway/client.ts:933-938`
  - body: The node host only exits for known auth `detailCode` values. This new max-attempt branch sends no detail code and also marks the client closed, so after a long gateway outage the node-host process logs that it is waiting for operator action and then stays alive without any future reconnect. The retry-exhausted path needs caller-aware handling, or node-host must explicitly exit/restart for it.
  - confidence: 0.88
- **[P3] Make the reset test complete a real hello:** `src/gateway/client.test.ts:1074-1086`
  - body: This test does not exercise the reset added in the implementation: it emits a `type: "connect"` message with a made-up id, while `GatewayClient` resolves hello only from a matching `type: "res"` response frame. It also performs only five post-success failures, so it would pass even if the counter never reset. Send a valid response for the actual connect id and drive enough closes to prove the reset.
  - confidence: 0.9
