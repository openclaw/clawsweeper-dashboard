## Summary

Keep open for maintainer review: current main still has the reported unbounded ordinary reconnect path, but this PR's blanket cap would stop node-host/backend clients after finite outages and the new reset test does not exercise a real hello-ok success path.

## What This Changes

The PR adds a private 30-attempt reconnect counter to GatewayClient, resets it on stop/hello, emits onReconnectPaused when exhausted, and adds two client tests.

## Best Possible Solution

GatewayClient should get caller-aware reconnect semantics that preserve node-host recovery while giving foreground callers a documented exhaustion path, focused regression coverage, and a changelog entry.

## Reproduction Assessment

Yes for the unbounded reconnect path: source inspection shows ordinary close events repeatedly reach scheduleReconnect(), which schedules start() again without any attempt budget. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. A static global cap is not the best fix because node-host explicitly relies on GatewayClient continuing to retry, so a caller-aware policy or explicit retry option is safer than changing every client mode at once.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve node-host recovery when attempts exhaust:** `src/gateway/client.ts:934-936`
  - body: This hard stop applies to every GatewayClient, including node-host clients whose current code says connect failures should keep retrying. After 30 ordinary closes, the PR sets closed and calls onReconnectPaused with detailCode undefined; node-host only exits for selected detail codes, so it would remain alive but permanently disconnected instead of reconnecting when the gateway returns. The retry budget needs to be caller-aware or opt-in before changing the shared scheduler.
  - confidence: 0.91
- **[P3] Send a real hello-ok payload in the reset test:** `src/gateway/client.test.ts:1075-1079`
  - body: GatewayClient resolves successful responses from parsed.payload, but this test sends the hello data under result. That makes helloOk undefined and the success handler can throw into the catch path, so the test can pass without actually proving reconnectAttempts resets after a successful handshake. Reuse the existing hello-ok helper or send a payload frame.
  - confidence: 0.86
