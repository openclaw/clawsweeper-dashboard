## Summary

Keep open. The PR addresses a real unbounded reconnect concern, but its shared hard cap changes `GatewayClient` semantics for long-lived node-host clients and the reset test does not exercise a valid hello-ok response.

## What This Changes

The PR adds a `GatewayClient` reconnect attempt counter capped at 30 attempts, pauses reconnect after exhaustion, resets the counter on stop/hello-ok, and adds two client tests.

## Best Possible Solution

Define and document a caller-aware reconnect policy that preserves node-host recovery while giving foreground or short-lived clients a finite exhausted path or threshold callback, with focused regression coverage.

## Reproduction Assessment

Yes for the PR regression: source inspection shows the proposed cap closes the shared `GatewayClient` after ordinary reconnects, and node-host does not exit for the new `detailCode: null` pause. The original zombie-process impact was not independently reproduced against current main.

## Solution Assessment

No. A shared hard stop in `GatewayClient` is not the narrowest maintainable solution because it changes all callers; a caller-aware retry policy or nonterminal threshold callback is safer.

## Review Findings
