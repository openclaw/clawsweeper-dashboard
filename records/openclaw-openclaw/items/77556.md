## Summary

Keep open. The PR targets a real unbounded reconnect path, but the shared hard stop applies to long-lived GatewayClient callers that currently rely on automatic recovery, so it needs maintainer reconnect-policy review before merge.

## What This Changes

The PR adds a shared GatewayClient reconnect attempt counter capped at 30 attempts, resets it on stop and hello-ok, and adds two client tests for exhaustion and reset behavior.

## Best Possible Solution

Define a caller-aware reconnect policy that preserves long-lived node-host recovery while giving foreground callers a finite terminal path or threshold callback, with focused regression coverage and docs if the contract changes.

## Reproduction Assessment

Yes, from source inspection: the PR closes the shared GatewayClient after 30 ordinary reconnects, and node-host does not exit for the new detailCode:null pause. I did not run tests because this review is read-only.

## Solution Assessment

No. A shared hard stop is not the narrowest maintainable fix because it changes all GatewayClient callers; a caller-aware retry policy or nonterminal threshold callback is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve node-host reconnect recovery:** `src/gateway/client.ts:931-941`
  - body: This max-attempt check applies to every GatewayClient, including runNodeHost. The node-host caller currently relies on GatewayClient to keep retrying, and its onReconnectPaused path only exits for auth detail codes; this new detailCode:null pause sets closed=true and leaves the process alive but unable to reconnect after a long gateway outage. Make the cap opt-in/caller-aware or keep node mode unbounded.
  - confidence: 0.9
- **[P3] Send a valid hello-ok in the reset test:** `src/gateway/client.test.ts:1072-1085`
  - body: The reset test never sends a frame that handleMessage accepts as the connect response; it emits type:"connect" instead of the existing type:"res" hello-ok response shape. As written, the test would still pass without the reset because it only drives five more failures, so it does not lock in the claimed behavior.
  - confidence: 0.86
