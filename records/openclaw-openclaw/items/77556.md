## Summary

Keep open. The PR addresses a real unbounded reconnect concern, but the current patch applies a hard stop to the shared GatewayClient path and would change node-host recovery semantics before maintainers have chosen that policy.

## What This Changes

The PR adds a GatewayClient reconnectAttempts counter capped at 30, pauses reconnects after exhaustion, resets the counter on stop/hello-ok, and adds two GatewayClient tests.

## Best Possible Solution

Define a caller-aware reconnect policy that preserves long-lived node-host recovery while giving foreground callers a finite terminal path or threshold callback, then document the contract and cover it with focused GatewayClient tests.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows ordinary reconnect exhaustion closes the shared GatewayClient, and current node-host pause handling returns for the new detailCode:null case; fake timers can drive the same path without a live gateway.

## Solution Assessment

No. A shared hard stop is not the narrowest maintainable solution because it changes node-host availability semantics; a caller-aware policy or nonterminal threshold callback is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve node-host reconnect recovery:** `src/gateway/client.ts:934-943`
  - body: This max-attempt check runs in the shared scheduleReconnect() path. runNodeHost currently relies on GatewayClient to keep retrying, and its reconnect-paused handler only exits for auth detail codes; this new detailCode:null pause leaves the process alive with a closed client after a long outage. Make the cap caller-aware or keep node mode unbounded.
  - confidence: 0.91
- **[P3] Send a real hello-ok response in the reset test:** `src/gateway/client.test.ts:1074-1086`

  - confidence: 0.86
