## Summary

Keep open. The latest PR head still applies one shared reconnect cap to all GatewayClient callers, including node-host recovery, and the reset test does not exercise the real hello-ok response path.

## What This Changes

The PR adds a shared GatewayClient reconnect attempt counter capped at 30 attempts, pauses reconnects after exhaustion, resets the counter on stop/hello-ok, and adds two client tests.

## Best Possible Solution

Define and document a caller-aware reconnect policy that preserves node-host recovery while giving foreground or short-lived clients a finite terminal path or threshold callback, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows the proposed cap closes the shared GatewayClient after 30 ordinary reconnects and node-host does not exit for the new detailCode:null pause; the reset test also does not send a valid response frame.

## Solution Assessment

No. A shared hard stop in GatewayClient is not the narrowest maintainable fix because it changes all callers; the safer path is caller-aware retry policy or a nonterminal threshold callback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve node-host reconnect recovery:** `src/gateway/client.ts:931-941`
  - body: This max-attempt check applies to every GatewayClient, including runNodeHost. The node-host caller currently relies on GatewayClient to keep retrying, and its onReconnectPaused path only exits for auth detail codes; this new detailCode:null pause sets closed=true and leaves the process alive but unable to reconnect after a long gateway outage. Make the cap opt-in/caller-aware or keep node mode unbounded.
  - confidence: 0.9
- **[P3] Send a valid hello-ok in the reset test:** `src/gateway/client.test.ts:1074-1087`
  - body: The reset test never sends a frame that handleMessage accepts as the connect response; it emits type:"connect" instead of the existing type:"res" hello-ok response shape. As written, the test would still pass without the reset because it only drives five more failures, so it does not lock in the claimed behavior.
  - confidence: 0.86
