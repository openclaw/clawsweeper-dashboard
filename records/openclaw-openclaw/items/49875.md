## Summary

Keep open. Current main still lacks the requested broadcast inbound_claim dispatch path, but the PR diff is not merge-ready because its insertion point can bypass fast-abort handling and plugin-bound fallback behavior, and the claimed early return is missing the same cleanup/diagnostic handling used by existing early returns.

## What This Changes

This PR adds a broadcast runInboundClaim call to dispatchReplyFromConfig before markProcessing and returns early when a plugin claims the inbound message.

## Best Possible Solution

Land the generic inbound_claim wiring only for ordinary non-plugin-bound dispatch, while preserving fast-abort precedence, plugin-owned binding fallback semantics, session diagnostics, inbound dedupe cleanup, and focused dispatch regression tests.

## Reproduction Assessment

Yes. Current main can be checked by the existing dispatch test that mocks runInboundClaim as handled and asserts it is not called, plus the /stop fast-abort tests that show the cancellation path the PR insertion would preempt.

## Solution Assessment

No. The requested wiring is the right product direction, but this PR’s exact placement is not the narrowest maintainable fix because it runs before fast-abort and after plugin-bound fallback, and it omits cleanup that existing early-return paths perform.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fast-abort before plugin claims:** `src/auto-reply/reply/dispatch-from-config.ts:419-423`
  - body: This generic claim runs before the existing tryFastAbortFromMessage path. A plugin that claims overlapping messages can also claim `/stop` or `abort`, so the function returns here and the documented cancellation path never runs.
  - confidence: 0.91
- **[P1] Skip generic claims after plugin fallback:** `src/auto-reply/reply/dispatch-from-config.ts:419-423`
  - body: When a plugin-bound conversation falls through missing_plugin or no_handler, dispatch has already chosen the OpenClaw fallback. Broadcasting here lets an unrelated plugin claim the same turn before replyResolver runs, breaking the fallback behavior covered by the existing tests.
  - confidence: 0.88
- **[P2] Clean up claimed-turn state before returning:** `src/auto-reply/reply/dispatch-from-config.ts:421-423`
  - body: The new handled return does not call markIdle or commitInboundDedupeIfClaimed. Other completed early returns do this, and skipping the dedupe commit leaves a claimed message key in the in-flight set instead of completing the turn cleanly.
  - confidence: 0.86
