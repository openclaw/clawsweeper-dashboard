## Summary

Keep open. Current main still lacks the generic inbound_claim broadcast that this PR is trying to add, but the proposed insertion point would regress cancellation, plugin-bound fallback, and inbound dedupe cleanup, so it should not merge as-is.

## What This Changes

The PR adds a broadcast runInboundClaim call to src/auto-reply/reply/dispatch-from-config.ts before agent dispatch and returns early when a plugin marks the inbound message as handled.

## Best Possible Solution

Land generic inbound_claim support for ordinary non-plugin-bound dispatch, while keeping fast-abort ahead of claim hooks, preserving plugin-owned fallback behavior, committing claimed inbound dedupe state, and updating focused dispatch plus hook-runner tests and changelog.

## Reproduction Assessment

Yes. Source inspection shows current main has no ordinary-path runInboundClaim call, and the PR regressions are reproducible from the existing dispatch ordering and tests without executing the suite.

## Solution Assessment

No. The requested wiring is the right direction, but this exact patch is not the best solution because it places the generic claim before fast-abort, after plugin-bound fallback decisions, and without committing claimed inbound dedupe state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fast-abort before plugin claims:** `src/auto-reply/reply/dispatch-from-config.ts:419-423`
  - body: This block runs before `tryFastAbortFromMessage()`. A global inbound-claim plugin that buffers overlapping messages can also claim `stop`, `abort`, or `/stop`, so dispatch returns here before the active run is cancelled.
  - confidence: 0.91
- **[P1] Skip generic claims after plugin fallback:** `src/auto-reply/reply/dispatch-from-config.ts:419-423`
  - body: When a plugin-bound conversation falls through `missing_plugin` or `no_handler`, dispatch has already chosen the OpenClaw fallback path. Broadcasting a generic claim here lets an unrelated plugin claim the turn before the promised fallback reply resolver runs.
  - confidence: 0.88
- **[P2] Commit claimed inbound dedupe before returning:** `src/auto-reply/reply/dispatch-from-config.ts:421-423`
  - body: `claimInboundDedupe()` has already put the message key in the in-flight set before this new early return. Returning without `commitInboundDedupeIfClaimed()` leaves that key in flight, so the same inbound message can be skipped indefinitely as `inflight`.
  - confidence: 0.89
- **[P2] Update dispatch tests for the new broadcast contract:** `src/auto-reply/reply/dispatch-from-config.ts:419-423`
  - body: The existing `does not broadcast inbound claims without a core-owned plugin binding` test configures `runInboundClaim` to return handled and asserts it is not called while the reply resolver runs. This PR changes that contract but does not update or replace the test, so the targeted dispatch suite will fail.
  - confidence: 0.9

## Security Review

Status: needs_attention
