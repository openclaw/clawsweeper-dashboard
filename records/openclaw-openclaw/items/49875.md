## Summary

Close this PR as superseded: current main does not add the exact generic `runInboundClaim` call, but the shipped `before_dispatch` hook covers the central global pre-model interception use case, and later broadcast work was closed after the author confirmed that hook covers the motivating listen-only/spam-filter/rate-limiter cases.

## What This Changes

The PR adds a broadcast `runInboundClaim()` call in `dispatchReplyFromConfig()` before agent dispatch so global `inbound_claim` handlers can claim ordinary channel messages.

## Best Possible Solution

Keep `inbound_claim` scoped to plugin-bound ownership, use shipped `before_dispatch` for global pre-model interception, and require a fresh hook-semantics proposal if exact broadcast claiming is still desired.

## Reproduction Assessment

Yes. Source inspection shows current main has no ordinary-path `runInboundClaim()` call, and the PR diff clearly inserts a generic claim before fast-abort and later dispatch guards.

## Solution Assessment

No. The branch is not the best solution because `before_dispatch` is already shipped for the central use case, while this insertion point creates ordering and fallback regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve fast-abort handling before claim hooks:** `src/auto-reply/reply/dispatch-from-config.ts:420`
  - body: This new broadcast claim runs before `tryFastAbortFromMessage()`. A plugin that claims overlapping messages can therefore swallow `stop`, `abort`, or `/stop` before the active run is cancelled, breaking the core escape hatch in the busy-agent path.
  - confidence: 0.9
- **[P1] Do not let unrelated claims swallow plugin fallback turns:** `src/auto-reply/reply/dispatch-from-config.ts:420`
  - body: After a plugin-owned binding falls through as missing or no-handler, the code has already promised normal OpenClaw fallback. This unguarded broadcast lets any other plugin claim the turn and return early, so the fallback reply resolver never runs.
  - confidence: 0.84
- **[P2] Mark the session idle before the handled return:** `src/auto-reply/reply/dispatch-from-config.ts:421-426`
  - body: The new handled return records completion but does not call `markIdle()`, unlike the existing plugin-bound handled path. For tracked sessions, diagnostics can be left in a processing state after a claimed message.
  - confidence: 0.78

## Security Review

Status: needs_attention
