## Summary

Keep open. The PR addresses the linked Discord bot-loop report with a plausible plugin-local guard and adequate terminal proof, but the new tracker runs before existing non-dispatch drop filters, so ignored bot messages can trip cooldown and suppress later legitimate bot traffic.

## What This Changes

The PR adds a Discord bot-pair sliding-window loop guard, botLoopProtection config/schema/docs/changelog entries, and focused tracker/preflight tests.

## Best Possible Solution

Keep the plugin-local tracker/config shape, but count only bot messages that survive preflight filters and would otherwise proceed toward dispatch, with a regression for ignored bot mentions.

## Reproduction Assessment

Yes. Source inspection of current main shows `allowBots: true` can let bot-authored Discord messages continue after the existing self and mention gates without any bot-pair cooldown, and the linked issue provides a concrete loop report.

## Solution Assessment

No, not quite. The tracker approach is the right narrow direction, but the current placement records messages before later drop filters, so it should move later or skip tracking for those non-dispatch paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Count only bot messages that survive preflight drops:** `extensions/discord/src/monitor/message-handler.preflight.ts:577-583`
  - body: The tracker is invoked before the existing `ignoreOtherMentions`, system-event, and empty-content drops below this block. With `allowBots: true` plus `ignoreOtherMentions`, repeated bot messages that current preflight would ignore can still fill the new window and put the pair in cooldown, causing a later legitimate bot message from the same pair to be dropped. Please record/suppress only after messages have survived the filters that would otherwise return `null` before dispatch, or explicitly skip tracking on those drop paths.
  - confidence: 0.82
