## Summary

Keep open. The PR is a plausible narrow Discord safety fix and includes usable terminal proof, but the current patch has blocking review findings around preflight accounting and stale generated config metadata.

## What This Changes

The PR adds a Discord bot-pair sliding-window cooldown guard, config schema/docs, changelog text, and focused tracker/preflight tests for the `allowBots` loop case.

## Best Possible Solution

Land a Discord-owned deterministic bot-pair guard after fixing the preflight accounting boundary and keeping the generated config schema surface in sync.

## Reproduction Assessment

Yes. Source inspection on current main shows `allowBots=true` accepts bot-authored messages after self-loop filtering with no bot-pair window/cooldown, so a focused preflight test can reproduce the loop acceptance path without live Discord.

## Solution Assessment

No in its current form. The sliding-window direction is a maintainable narrow fix, but the implementation should only count messages that survive preflight drops and should regenerate the bundled channel config metadata before merge.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0.88

Full review comments:

- **[P2] Move loop tracking after remaining preflight drops:** `extensions/discord/src/monitor/message-handler.preflight.ts:576-583`
  - body: The new guard records the bot pair before later drops such as `ignoreOtherMentions`, system events, and empty-content checks. With `allowBots: true` plus `ignoreOtherMentions`, repeated off-target bot messages can fill the cooldown for that bot/OpenClaw pair even though none reached dispatch, causing a later legitimate bot mention to be suppressed. Record the pair only once the message is known to be dispatchable, or split the tracker so dropped messages do not consume budget.
  - confidence: 0.86
- **[P2] Regenerate bundled Discord config metadata:** `src/config/zod-schema.providers-core.ts:617-623`
  - body: Adding `botLoopProtection` to `DiscordAccountSchema` changes the published Discord channel config schema, but `src/config/bundled-channel-config-metadata.generated.ts` is not updated and currently has no `botLoopProtection` entry. `buildConfigSchema()` serves bundled channel schemas from that generated metadata, so config/schema clients will not see the new option until the metadata is regenerated.
  - confidence: 0.9
