## Summary

Keep this PR open. The root Discord loop problem is source-reproducible and the pair-guard direction is reasonable, but the branch still has a Google Chat side-effect bug and the real behavior proof is stale against the current head.

## What This Changes

Adds shared bot-pair loop protection config/runtime/docs/tests and wires Discord, Slack, Matrix, and Google Chat accepted bot-authored messages into the guard.

## Best Possible Solution

Fix the Google Chat pre-admission side effect, refresh current-head real behavior proof, and then land the shared pair guard if maintainers approve the SDK/config defaults.

## Reproduction Assessment

Yes, from source inspection. Current main lets accepted Discord bot-authored messages reach dispatch when `allowBots` and mention/access gates pass, while the replay guard only dedupes the same Discord message id.

## Solution Assessment

Mostly, but not yet. A direction-independent pair guard before Discord dispatch is the right shape, but the PR currently leaves Google Chat typing messages ahead of suppression and needs refreshed current-head proof plus API/default approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate Google Chat before sending typing messages:** `extensions/googlechat/src/monitor.ts:360`
  - body: `botLoopProtection` is only passed into the core turn after Google Chat has already posted the typing indicator. When the core guard suppresses a bot-pair event, no delivery runs to update or delete that typing message, so every suppressed loop event can still emit a visible Google Chat message. Move the guard check ahead of the typing side effect, or defer creating the typing message until after turn admission is known.
  - confidence: 0.88
