## Summary

Keep this issue open. Current main still contains the shared fallback/default-account status path that can surface a synthetic Feishu `default` account, and same-author PR #64712 remains the active implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to stop creating or surfacing an unconfigured Feishu `default` account after gateway restarts when users have explicitly configured Feishu accounts.

## Best Possible Solution

Review and land #64712 or a replacement that distinguishes explicitly configured accounts from synthetic fallback ids across channel summary and gateway status, preserves diagnostics for a genuinely configured but broken default account, and adds focused Feishu/default-account regression coverage.

## Reproduction Assessment

Unclear for the exact multi-account gateway-restart persistence claim because this read-only review did not run a live Feishu gateway. The current source gives a high-confidence static reproduction path for the fallback/status half: an empty or synthetic account list can still produce a `default` status summary.

## Solution Assessment

Unclear whether the exact open PR is the final best implementation until it is reviewed. The best direction is a narrow shared status/account-enumeration fix, not a Feishu-only credential special case.
