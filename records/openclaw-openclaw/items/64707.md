## Summary

Keep this open because current main and the latest release still carry the shared fallback-account/status path that can surface a synthetic Feishu `default` account, and the same-author fix candidate at https://github.com/openclaw/openclaw/pull/64712 is still open rather than merged.

## What This Changes

The issue asks OpenClaw not to create or render a synthetic Feishu `default` account with `not configured` status after gateway restart when users only configured named Feishu accounts.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/64712 or a replacement that distinguishes real configured accounts from synthetic fallback ids, preserves diagnostics for genuinely unresolved configured accounts, and proves the Feishu restart/status path.

## Reproduction Assessment

Partially. The source-level fallback/status path can still synthesize and report `default`, but a full live gateway restart reproduction was not run, so persistence versus rendering remains unproven.

## Solution Assessment

No, closing is not the best way to solve the issue while the paired fix PR is still open and current main still has the implicated fallback behavior. The best fix should suppress synthetic phantom defaults without hiding real configured-account errors.
