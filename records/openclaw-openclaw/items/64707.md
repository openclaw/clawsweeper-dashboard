## Summary

Keep this issue open because current main still has the shared fallback/default-account status paths on the Feishu surface, lacks focused Feishu phantom-default restart/status coverage, and same-author PR #64712 remains the active implementation candidate rather than a merged fix.

## What This Changes

The issue reports that gateway restarts create or surface a phantom Feishu `default` account with `not configured` status when only named Feishu accounts are configured.

## Best Possible Solution

Review and land #64712, or a replacement, only after it proves whether `default` is persisted or rendered, distinguishes real configured Feishu accounts from synthetic fallback ids, and adds Feishu restart/status regression coverage.

## Reproduction Assessment

Partly: source inspection gives a clear current-main path for a synthetic default-account status through empty-account fallback handling. The exact live gateway-restart persistence claim for named Feishu accounts remains unverified without a matching configured Feishu gateway.

## Solution Assessment

Unclear whether the paired PR is the complete best fix because it targets empty-account fallback while the report describes named accounts after restart. The maintainable direction is still a shared status/account-enumeration fix with Feishu-focused regression coverage, not a Feishu credential special case.
