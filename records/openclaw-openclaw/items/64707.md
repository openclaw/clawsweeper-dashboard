## Summary

Keep this issue open because current main still has the shared fallback/default-account status path on the Feishu surface, and same-author PR #64712 remains the focused open implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to stop surfacing or recreating an unauthored Feishu `default` account with `not configured` status after gateway restarts when only named Feishu accounts are configured.

## Best Possible Solution

Land #64712 or a replacement that distinguishes real configured Feishu accounts from fallback ids across status surfaces while preserving diagnostics for explicitly configured but broken default accounts.

## Reproduction Assessment

Partly yes: current source and tests give a high-confidence path for reproducing the synthetic default-account status line through empty account-id fallback handling. The exact live gateway-restart persistence claim remains unclear without running a Feishu gateway with matching config and credentials.

## Solution Assessment

Unclear whether the linked PR is the final best patch until review, but the best direction is a shared status/account-enumeration fix with Feishu regression coverage rather than Feishu-only credential special cases.
