## Summary

Keep open: current main and the latest release still lack yielded-run input in isolated cron classification, and the PR is in the right layer, but the patch is too broad because a clean yield would now clear fatal classification for arbitrary prior error payloads; it also has only test proof, not real yielded-cron proof.

## What This Changes

The PR threads `meta.yielded` into isolated cron payload classification and adds helper tests for clean-yield recovery plus fatal-signal handling.

## Best Possible Solution

Narrow yield recovery to the proven benign warning or recovered-output case, keep fatal mutating and non-recoverable payloads fatal, then require real yielded-cron proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current `main` and `v2026.6.8` do not pass yielded metadata into the cron classifier, while the embedded runner emits `meta.yielded` for `sessions_yield`; the new blocking concern is also source-reproducible from the broader predicate and existing mutating-tool error payload tests.

## Solution Assessment

No, not yet. Threading `meta.yielded` into the classifier is the right layer, and changing UI would not address persisted cron status or delivery suppression, but the current recovery predicate is too broad because it bypasses fatal classification for arbitrary earlier error payloads.
