## Summary

Keep open: the PR targets a real cron false-fatal path, but the patch is too broad because any yielded run with an error payload and no run-level fatal signal can now be classified as ok, and the external PR still has only test proof rather than real yielded-cron proof.

## What This Changes

The PR threads `finalRunResult.meta.yielded` into isolated cron payload classification and adds helper tests so a yielded turn with an earlier tool warning is treated as nonfatal while fatal failure signals remain fatal.

## Best Possible Solution

Constrain yielded recovery to proven clean cases, add regression coverage for yielded only-error and non-recoverable error payloads, then require redacted real isolated-cron proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main does not pass yielded metadata into the cron classifier, and the PR's changed classifier path is directly exercised by helper tests; I did not run a live cron reproduction in this read-only review.

## Solution Assessment

No. Wiring the existing yielded signal into the existing classifier is the right layer, but the proposed condition is too broad because it clears fatal handling for any yielded error payload rather than only a proven clean yield.
