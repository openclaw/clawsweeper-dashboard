## Summary

Keep open. The bug and proposed behavior are well supported, and the PR includes useful real terminal proof, but the patch should not land as-is because it moves Anthropic token-class policy into core provider-usage fallback instead of the provider-owned usage auth boundary.

## What This Changes

The PR skips standard Anthropic API keys during provider usage auth while preserving setup-token-shaped Anthropic keys and adds provider-usage auth regression tests.

## Best Possible Solution

Use a provider-owned usage-auth or fallback-control contract so Anthropic can decline standard API keys for usage while setup-token usage and other providers' API-key usage paths stay covered by focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows Anthropic usage auth falls back from the plugin's OAuth-only resolver to any resolved API key, then sends that token to the OAuth usage endpoint.

## Solution Assessment

No. The intended behavior is right, but the implementation should keep Anthropic credential-class policy in the provider-owned usage auth path rather than adding a provider-id check to generic core fallback.
