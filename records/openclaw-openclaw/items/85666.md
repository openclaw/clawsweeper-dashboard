## Summary

Keep this PR open, but not merge-ready: it fixes the observed Anthropic status symptom, while the current implementation puts Anthropic credential-class policy into generic core usage fallback instead of the provider-owned usage auth path.

## What This Changes

The PR filters Anthropic API-key credentials out of provider usage fallback while preserving setup-token-shaped Anthropic tokens and updates provider usage auth tests.

## Best Possible Solution

Let provider usage auth own whether generic API-key fallback is allowed, then implement the Anthropic standard-key skip and setup-token allowance in the Anthropic provider with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows Anthropic usage auth can fall through from the plugin's OAuth-only resolver to generic API-key fallback, which can pass a standard Anthropic API key to the OAuth usage fetcher.

## Solution Assessment

No. The intended behavior is right, but the patch should keep Anthropic credential-class policy in the provider-owned usage auth path instead of adding a provider-id check to generic core fallback.
