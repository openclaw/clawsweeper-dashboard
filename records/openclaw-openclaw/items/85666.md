## Summary

Keep open: the PR targets the right Anthropic status bug and now has real status-path proof, but it still changes public plugin usage-auth fallback semantics through an empty-token sentinel that should be resolved before merge.

## What This Changes

The PR changes Anthropic usage-auth resolution and provider usage fallback so standard Anthropic API keys are skipped for usage status while setup tokens and other provider fallbacks stay covered by regression tests.

## Best Possible Solution

Keep the Anthropic API-key skip, but express no-fallback through an explicit typed usage-auth result or a documented maintainer-approved contract, with null/undefined preserving current fallback behavior.

## Reproduction Assessment

Yes, at source level: current main can resolve an Anthropic API-key-only profile through generic usage fallback and then send it as a bearer token to the Anthropic OAuth usage endpoint. I did not run a live current-main Anthropic status command in this read-only review.

## Solution Assessment

No as submitted: the behavior target is right, but an empty-string sentinel is not the best plugin SDK boundary. An explicit typed no-fallback result with docs or contract coverage is the safer maintainable path.
