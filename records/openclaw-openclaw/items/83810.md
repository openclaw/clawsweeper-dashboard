## What This Changes

Explicit agent runs should resolve configured model aliases before thinking/capability validation so an alias is validated against its resolved provider/model.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/84333 so bare explicit model overrides reuse the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation; keep `aliases[]` as a separate product/config follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an alias under `agents.defaults.models`, include a matching provider wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`; current main parses the alias as a default-provider model before validation.

## Solution Assessment

Yes for the central bug. Reusing the existing alias-aware resolver before capability validation is the narrow maintainable fix, while multiple aliases should remain a separate config/product decision.
