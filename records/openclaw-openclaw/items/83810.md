## Summary

Keep open: current main still parses an explicit providerless `--model` override as a literal model before using the configured alias resolver, so thinking validation can run against the wrong provider/model. The related preset request remains broader product work and does not supersede this narrow bug.

## What This Changes

The issue requests resolving explicit `--model` aliases before thinking and capability validation in agent runs.

## Best Possible Solution

Resolve explicit model overrides through the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation, and leave multi-alias config support to a separate product follow-up.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: configure an alias under `agents.defaults.models`, allow the default provider with a wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`; the explicit override path can validate as `openai/<alias>`.

## Solution Assessment

Yes for the central bug. The maintainable fix is to reuse `buildModelAliasIndex` and `resolveModelRefFromString` before capability validation, with focused regression coverage; the proposed multiple-alias config shape should be handled separately.
