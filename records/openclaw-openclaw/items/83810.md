## Summary

Keep open: current main still parses explicit `--model` overrides with the lower-level parser before consulting the configured alias resolver, so alias text can be validated as a literal default-provider model before thinking support is checked. The broader preset/config request at https://github.com/openclaw/openclaw/issues/83805 is related product UX work, but it does not supersede this narrow bug.

## What This Changes

The issue requests that explicit agent runs resolve configured model aliases before thinking and capability validation.

## Best Possible Solution

Resolve explicit `--model` overrides through the existing alias-aware model resolver before visibility, harness, auth-profile, and thinking validation, then add focused regression coverage; leave multi-alias schema support to a separate product decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure an alias under `agents.defaults.models`, include a matching provider wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`. I did not run it because this review is read-only.

## Solution Assessment

Yes for the central bug: reuse the existing alias-aware resolver before capability validation and add a regression test around provider wildcards plus explicit thinking. The multiple-alias config shape should stay separate.
