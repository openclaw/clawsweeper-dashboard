## What This Changes

The issue asks OpenClaw to resolve explicit agent-run model aliases before thinking/capability validation, with multi-alias config left as an adjacent improvement.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84333; the core fix should reuse the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation, while treating multi-alias config as a separate product follow-up.

Yes. Source inspection gives a high-confidence path: configure an alias under `agents.defaults.models`, include a matching provider wildcard such as `openai/*`, then run an explicit agent command with `--model <alias>` and a thinking level only supported by the resolved target.

## Solution Assessment

Yes for the central bug. Reusing the existing alias-aware resolver before validation is the narrowest maintainable fix; the proposed multiple-alias config shape should stay separate.
