## Summary

Keep open: current main and the latest release still parse a bare explicit `--model` override before the configured alias resolver, and an open closing PR now owns the fix path.

## What This Changes

Explicit agent runs should resolve configured model aliases before thinking/capability validation and use the resolved provider/model throughout the run metadata and validation path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84333 or an equivalent narrow change that reuses the existing alias-aware resolver before validation, leaving multi-alias config shape as a separate product follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an alias under `agents.defaults.models`, include a matching provider wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`; current main parses the alias as a literal default-provider model before thinking validation.

## Solution Assessment

Yes for the central bug. The best fix is to reuse the existing alias-aware resolver in the explicit-run branch and add a regression around provider wildcards plus explicit thinking; the proposed `aliases[]` shape should stay separate.
