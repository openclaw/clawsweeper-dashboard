## Summary

Keep open: current main still parses an explicit providerless `--model` value with `parseModelRef` before consulting the configured alias index, so thinking validation can run against the alias as a literal default-provider model. The broader preset request at https://github.com/openclaw/openclaw/issues/83805 is related product UX work and does not supersede this narrow bug.

## What This Changes

Explicit agent runs should resolve configured model aliases before thinking and capability validation.

## Best Possible Solution

Resolve explicit model overrides through the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation, while leaving multi-alias config support to a separate product decision.

## Reproduction Assessment

Yes, by source inspection: configure an alias under `agents.defaults.models`, allow the default provider via a wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`. I did not run the CLI path because this review is read-only.

## Solution Assessment

Yes for the central bug: reuse the existing alias-aware resolver before capability and thinking validation, and add a regression test around provider wildcards plus explicit thinking. The multiple-alias config shape should stay separate.
