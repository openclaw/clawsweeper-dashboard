## Summary

Keep open: current main still parses explicit `--model` overrides before the configured alias resolver, so alias-only runs can validate against the wrong provider/model. The related preset issue is broader UX work, and the open fix PR is not merged yet.

## What This Changes

Resolve explicit agent-run model aliases before thinking and capability validation.

## Best Possible Solution

Resolve explicit model-only overrides through the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation; keep `aliases[]` as a separate config/product follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an alias under `agents.defaults.models`, include a matching provider wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`; I did not execute it because this sweep is read-only.

## Solution Assessment

Yes for the central bug: reuse the existing alias-aware resolver before capability validation and add focused regression coverage. The multiple-alias config shape should be handled separately.
