## Summary

Keep open: current main still parses explicit providerless model overrides before the alias-aware resolver, so thinking validation can see a configured alias as a literal default-provider model. The broader preset request at https://github.com/openclaw/openclaw/issues/83805 does not supersede this narrow bug.

## What This Changes

The issue requests resolving explicit agent-run model aliases before thinking and capability validation.

## Best Possible Solution

Resolve explicit model overrides through the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation, and keep the multiple-alias config shape as a separate product follow-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure an alias in agents.defaults.models, include an allowed default-provider wildcard, then run an explicit --model alias with a thinking level only supported by the alias target. The current path parses the alias before using the alias index.

## Solution Assessment

Yes for the central bug: reuse the existing alias-aware model resolver before capability validation and add a regression test around provider wildcards plus explicit thinking. The proposed aliases[] config expansion should stay out of this bug fix unless maintainers choose it separately.
