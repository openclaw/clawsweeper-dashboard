## Summary

Keep open: current main still parses explicit providerless `--model` overrides as default-provider model refs before alias-aware resolution, so thinking validation can evaluate alias text like `code` as `openai/code` instead of the configured target.

## What This Changes

The issue asks OpenClaw to resolve explicit agent-run model aliases before capability and thinking-level validation.

## Best Possible Solution

Resolve explicit `--model` values through the existing alias-aware model resolver before visibility, harness, auth-profile, and thinking validation, with focused regression coverage for provider wildcards plus explicit thinking.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `agents.defaults.models` with an alias target such as `codex/gpt-5.5`, include a default-provider wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh` and the explicit override path can validate `openai/<alias>`.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to reuse the existing alias-aware resolver before capability validation; the multiple-alias config shape should remain a separate product follow-up.
