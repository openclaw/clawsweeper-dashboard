## Summary

Keep open: current main still parses a bare explicit `--model` override before using the configured alias resolver, then validates thinking support against that parsed provider/model. The open closing PR at https://github.com/openclaw/openclaw/pull/84333 is the active implementation path, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests resolving explicit model aliases before capability and thinking-level validation in `openclaw agent` runs.

## Best Possible Solution

Review and land or adapt https://github.com/openclaw/openclaw/pull/84333 so the explicit-run override path reuses the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a model alias under `agents.defaults.models`, include a matching provider wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`; the explicit override path can validate the default-provider literal instead of the alias target.

## Solution Assessment

Yes for the central bug. The maintainable fix is to reuse the existing alias-aware resolver before capability validation and keep the proposed multiple-alias config shape as separate product work.
