## Summary

Keep open: current main and the latest tagged release still parse a bare explicit `--model` value with `parseModelRef` before thinking validation, so configured aliases can still be treated as default-provider literals. The open linked fix PR is the right path to resolve this issue after maintainer review.

## What This Changes

The issue requests resolving explicit agent-run model aliases before thinking/capability validation so validation targets the configured provider/model rather than a literal default-provider model.

## Best Possible Solution

Review and land the focused open PR if maintainers agree aliases should win for bare explicit overrides, and keep multi-alias/preset UX work in the broader product follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure an alias under `agents.defaults.models` with a provider wildcard such as `openai/*`, then run `openclaw agent --model <alias> --thinking xhigh`; current main parses the alias as a default-provider literal before thinking validation.

## Solution Assessment

Yes for the central bug. The maintainable fix is to reuse the existing alias-aware resolver before visibility, harness, auth-profile, and thinking validation; the multiple-alias config idea should remain separate.
