## Summary

Keep open. Current main fixed the earlier channel-startup blocking path, but the remaining scheduled primary-model prewarm still falls back to a fixed 5000 ms timeout with no env/config override, and the Gateway config/schema/docs surfaces do not expose the requested knob.

## What This Changes

Request to make the Gateway startup primary-model prewarm timeout configurable via environment or Gateway config instead of always using the hardcoded 5000 ms fallback.

## Best Possible Solution

Add a validated, primary-model-specific Gateway startup prewarm timeout setting with env precedence, config fallback, and the existing 5000 ms default. Wire it only into the scheduled background prewarm path, keep channel startup detached, and update schema/help/labels/docs/tests/changelog together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: when scheduled startup prewarm runs without an explicit timeoutMs, any slow prewarm path uses the 5000 ms fallback and emits the warning.

## Solution Assessment

Yes. A narrow env/config resolver at the Gateway startup boundary is the maintainable fix; increasing the default globally or rejoining prewarm to channel readiness would risk reopening the earlier startup-blocking behavior.
