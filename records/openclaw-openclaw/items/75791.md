## Summary

Keep open. Current main fixed the earlier channel-startup blocking behavior, but this follow-up still identifies a real remaining gap: the scheduled primary-model prewarm path falls back to a hardcoded 5000 ms timeout and has no env/config override.

## What This Changes

The issue requests making the Gateway startup primary-model prewarm timeout configurable via environment variable or Gateway config instead of relying on the hardcoded 5000 ms default.

## Best Possible Solution

Add a validated Gateway startup timeout setting for primary-model prewarm, with `OPENCLAW_PRIMARY_MODEL_PREWARM_TIMEOUT_MS` taking precedence over config and the existing 5000 ms default. Wire it only into the scheduled background prewarm timeout, keep channel startup detached, and update schema/help/docs/tests/changelog together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the scheduled startup call omits `timeoutMs`, so any slow primary-model prewarm always uses the 5000 ms fallback and emits the timeout warning.

## Solution Assessment

Yes. A narrow env/config override at the Gateway startup boundary is the maintainable fix; increasing channel wait time or changing provider discovery behavior would reopen the earlier startup-readiness problem.
