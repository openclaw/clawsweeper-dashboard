## Summary

Keep open. Current main fixed the earlier channel-startup blocking behavior, but this follow-up still points at a real remaining configurability gap: the scheduled primary-model prewarm path still falls back to a hardcoded 5000 ms timeout and there is no Gateway config or env override for it.

## What This Changes

Request to make the primary model prewarm timeout configurable via environment or config instead of the current fixed startup budget.

## Best Possible Solution

Add a validated Gateway startup setting, preferably an explicit `gateway.primaryModelPrewarmTimeoutMs` or maintainer-approved equivalent, plus `OPENCLAW_PRIMARY_MODEL_PREWARM_TIMEOUT_MS` env precedence over config and the existing 5000 ms default. Wire it only into the scheduled primary-model prewarm timeout, keep the warmup detached from channel startup, and update schema/help/docs/tests/changelog together.

## Reproduction Assessment

Yes. Source inspection reproduces the issue: real startup calls the timeout wrapper without a `timeoutMs`, so it always uses the 5000 ms constant unless a test-only caller injects a value.

## Solution Assessment

Yes. A narrow env/config override at the Gateway startup boundary is the best fit; changing provider discovery or making channels wait longer would reopen the earlier startup-readiness problem.
