## Summary

Keep open. Current main still awaits embedded-run auth initialization before marking the `auth` startup stage, and stale OpenAI Codex OAuth credentials still flow through the locked provider refresh path; the reporter's follow-up ruled out only the outer `getApiKeyForModel` wrapper, not the inner refresh manager, lock, runtime import, or provider token call.

## What This Changes

The issue asks OpenClaw to reduce recurring cold OpenAI Codex OAuth auth-resolution latency on embedded dispatches and isolate the cold refresh subpath with timing data.

## Best Possible Solution

Add targeted auth substage timing around OAuth manager refresh, file locks, runtime import, and the OpenAI Codex token call, then implement a credential-safe cache or proactive refresh only for the confirmed hot path.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms that stale OpenAI Codex OAuth credentials still synchronously enter the locked refresh path inside the embedded-run auth stage, but the exact four-second timing needs current-main substep measurements.

## Solution Assessment

Unclear. The proposed cache may be the right fix, but the maintainable path is to measure the refresh manager and provider adapter first, then choose an owner-approved credential invalidation policy.
