## Summary

Keep open. Current main still routes stale OpenAI Codex OAuth credentials through the synchronous embedded-run auth path, and the reporter has added useful follow-up data ruling out `getApiKeyForModel`; the remaining refresh substeps still need live timing before choosing a cache or proactive-refresh policy.

## What This Changes

Investigate and reduce OpenAI Codex OAuth cold-path auth latency that recurs when stale access tokens refresh during dispatch.

## Best Possible Solution

Measure the refresh, store-load, and provider-runtime substeps first, then implement a narrowly scoped Codex OAuth refresh/cache policy with regression coverage if the refresh roundtrip is confirmed.

## Reproduction Assessment

No. Source inspection confirms the stale OpenAI Codex OAuth credential path runs synchronously inside the embedded-run auth stage, and the reporter provided field logs plus partial instrumentation, but there is not yet a current-main live reproduction with substep timing for the 4s bucket.

## Solution Assessment

Unclear. The proposed cache/proactive-refresh direction may be right, but targeted instrumentation should come first because this path is auth-sensitive and already has refresh-token reuse, locking, and cross-agent identity contracts.
