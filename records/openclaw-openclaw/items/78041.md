## Summary

Keep open. Current main still routes stale OpenAI Codex OAuth credentials through the embedded-run auth bucket and the locked provider refresh path; the reporter's follow-up narrowed the timing but did not yet identify the four-second substep.

## What This Changes

The issue asks OpenClaw to reduce recurring cold OpenAI Codex OAuth auth-resolution latency from about four seconds to near the warm dispatch path.

## Best Possible Solution

Add targeted auth substage timing around OAuth manager refresh, file locks, runtime import, and the OpenAI Codex token call, then optimize the confirmed hot path with credential-safe invalidation.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection confirms that stale OpenAI Codex OAuth credentials still synchronously enter the locked refresh path inside the embedded-run auth stage.

## Solution Assessment

Unclear. The proposed cache may be the right fix, but the maintainable path is to measure the refresh manager and provider adapter first, then choose an owner-approved credential invalidation policy.
