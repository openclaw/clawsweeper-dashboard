## Summary

Keep open: current main and the latest release still route OpenAI/Codex OAuth dispatch auth through embedded auth initialization, OAuth-manager refresh, and the OpenAI provider token refresh path, while the reporter’s follow-up only ruled out `getApiKeyForModel` as the dominant substep. ## What This Changes

Investigate and reduce intermittent multi-second cold-path auth latency for OpenAI/Codex OAuth dispatches while warm auth remains 2-4 ms.

## Best Possible Solution

Add auth substage timing around embedded auth initialization, OAuth access resolution, provider token refresh, and runtime auth preparation, then implement a measured cache or prewarm path that preserves OAuth locking, credential persistence, and provider runtime auth transforms.

## Reproduction Assessment

No, not from a high-confidence current-main live run. Source inspection and the reporter’s production timing show the cold path still exists, but live substage instrumentation is still needed to prove the dominant 4s component.

## Solution Assessment

Unclear. The proposed in-process cache is plausible, but the best fix depends on measured substep timings and must preserve OAuth refresh locking, persisted credential updates, and provider runtime-auth transforms.
