Keep open. Current main still performs stale openai-codex OAuth access-token refresh synchronously inside the embedded-run auth stage; the source matches the reported cold/warm split, but the exact 4s sub-step attribution still needs live timing before a token-cache or proactive-refresh design is chosen.

## What This Changes

The issue reports recurring multi-second cold auth-stage latency for openai-codex OAuth dispatches after the cached access token becomes stale.

## Best Possible Solution

Add targeted auth-stage instrumentation first, then implement an owner-approved proactive OAuth refresh or cache policy for openai-codex if measurements confirm refresh roundtrips dominate.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms that a stale openai-codex OAuth credential synchronously refreshes during the embedded-run auth stage, but the reported 4s timing still needs live sub-step instrumentation.

## Solution Assessment

Unclear. A proactive refresh/cache may be the right direction, but blindly extending token reuse is riskier than measuring refresh, store load, and runtime import timings and then choosing a safe invalidation policy.
