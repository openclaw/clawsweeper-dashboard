## Summary

Keep open. Current main still resolves normal embedded-run auth before the `auth` stage marker, and stale OpenAI Codex OAuth credentials still enter the synchronous locked refresh path; newer auth-store caching only addresses one suspected disk-read contributor and does not prove the reported cold-refresh latency is fixed.

## What This Changes

The issue asks to reduce recurring cold-path `openai-codex` OAuth auth-stage latency from about four seconds toward the warm 2-4 ms path.

## Best Possible Solution

Add targeted auth substage timing around OAuth manager refresh, file locking, Codex runtime import, and provider token refresh, then implement an owner-approved cache or proactive-refresh policy only for the confirmed hot path.

## Reproduction Assessment

No high-confidence live reproduction was run. The source path is clear for stale OpenAI Codex OAuth credentials synchronously refreshing inside the embedded-run auth stage, but the exact four-second timing still needs current-main substep measurements.

## Solution Assessment

Unclear. A token cache or proactive refresh may be the right fix, but the maintainable path is to measure the refresh manager and provider adapter first, then choose a credential-safe invalidation policy.
