## Summary

Keep open: current main still awaits the embedded Pi resource loader on the agent prep path, and the pinned upstream loader/package manager performs synchronous filesystem discovery under that call. The reporter's timings are plausible against the source, but I did not reproduce the 5-7s Windows stall locally.

## What This Changes

Replace the per-run synchronous session resource loading with a nonblocking or narrower embedded-run resource loader so agent startup does not monopolize the Gateway event loop.

## Best Possible Solution

Use a narrow embedded-run resource loader that preserves OpenClaw's inline compaction/pruning extensions and settings rehydration needs without scanning package, skill, prompt, theme, or context-file resources on every run.

## Reproduction Assessment

No, not at high confidence. Source inspection confirms synchronous filesystem work on the labeled prep stage, but I did not establish a current-main live Windows run showing the reported 5-7s stall.

## Solution Assessment

Unclear as written. Making the upstream loader fully async is directionally reasonable, but the narrower OpenClaw fix is likely to avoid dependency-wide discovery for embedded runs instead of parallelizing work that should not run there.
