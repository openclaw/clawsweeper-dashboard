## Summary

Keep open. The PR is a plausible new hook-secret capability with sufficient live proof, but it is not mergeable as-is because the new token-file resolver bypasses OpenClaw's shared secret-file hardening, and the new config surface still needs maintainer confirmation against the existing SecretRef policy.

## What This Changes

The PR adds a new `hooks.tokenFile` config field, routes gateway/Gmail hook auth through it, updates docs/schema/path normalization, and adds focused regression coverage.

## Best Possible Solution

Either route hook ingress secrets through the planned SecretRef credential path, or explicitly accept `hooks.tokenFile` as an interim surface implemented through the shared secret-file helper with matching schema, audit, docs, and tests.

## Reproduction Assessment

Not applicable as a bug repro. Source inspection shows current main only supports inline `hooks.token`, while the PR body demonstrates the new after-fix tokenFile behavior with a local live HTTP smoke.

## Solution Assessment

No, not as-is. The config direction needs maintainer confirmation, and any accepted file-backed hook token path should use `tryReadSecretFileSync` or the shared secret-file contract rather than raw `fs.readFileSync`.

## Review Findings

Overall correctness: patch is incorrect
