## Summary

Keep open: this PR adds a plausible new hook secret config surface and includes sufficient live smoke proof, but the patch is not merge-ready because the new token-file resolver bypasses OpenClaw's shared secret-file hardening and the interim `hooks.tokenFile` surface still needs maintainer acceptance against the SecretRef direction.

## What This Changes

The PR adds `hooks.tokenFile` across hook config/schema/docs/runtime paths, shares hook-token resolution with gateway and Gmail hook code, and adds tokenFile-focused regression tests.

## Best Possible Solution

Either accept `hooks.tokenFile` as an interim hook-secret config surface and implement it with the shared secret-file helpers, or defer hook secret files to the broader SecretRef credential surface instead of adding a parallel contract.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection shows current main only supports inline `hooks.token`, while the PR is adding a new file-backed secret capability with its own live smoke proof.

## Solution Assessment

No, not as-is. If maintainers accept the new config key, the narrow code fix is to route tokenFile reads through the shared secret-file helper and keep schema, docs, audit wording, and tests aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the shared secret-file reader for tokenFile:** `src/hooks/token.ts:19`
  - body: This new resolver reads the configured secret path with raw `fs.readFileSync` instead of OpenClaw's secret-file helper. Existing tokenFile credential paths use `tryReadSecretFileSync(..., { rejectSymlink: true })` and the shared helper enforces blank-path, max-size, regular-file, and symlink policy, so this can accept unsafe or misconfigured hook token files.
  - confidence: 0.91

## Security Review

Status: needs_attention
