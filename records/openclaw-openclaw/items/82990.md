## Summary

Keep open: the report is concrete, recent, and maps to current image-generation, async task wake, session recovery, and media-delivery code paths; current main has success-path media handling and an open dropped-media-warning PR, but no proof that safety-rejection stalls and stale planned media paths are fixed.

## What This Changes

The issue reports that an OpenAI Codex OAuth `image_generate` safety rejection leaves a tool call stalled, then session recovery preserves a planned-but-unwritten media path that fails Telegram delivery.

## Best Possible Solution

Treat this as a focused bug fix: make rejected async image-generation tasks terminal and visible to the agent/user without preserving or replaying any unconfirmed media path, then add regression coverage for the failure and recovery path.

## Reproduction Assessment

No high-confidence live repro was run, but the provided logs plus current source give a clear source-reproducible path: Codex image failure events throw, session-backed `image_generate` detaches, and missing local media paths fail delivery with `LocalMediaAccessError`. The exact stall/recovery handoff still needs a targeted regression or live proof.

## Solution Assessment

No, the current behavior is not the best solution because a provider safety rejection should become a terminal failed image-generation task without stale attachment state. The safer fix is in the generic image/media task and session-recovery path, not a Telegram-only workaround or a dropped-media warning alone.
