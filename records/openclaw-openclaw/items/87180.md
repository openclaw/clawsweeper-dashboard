## Summary

Keep open: current main and the latest release still treat unowned session-file advancement during the embedded prompt-lock release window as takeover/fatal cleanup, and the remaining fix needs a maintainer-approved recovery policy rather than cleanup closure.

## What This Changes

The issue requests recovery for embedded attempt session takeover during model API retry windows so generated replies are not silently lost.

## Best Possible Solution

Preserve takeover safety while making the original turn visibly recoverable: adopt a maintainer-approved retry, requeue, or fail-visible policy, add embedded-runner regression coverage, and fold the later duplicate report into this canonical issue.

## Reproduction Assessment

No high-confidence live reproduction yet; the report lacks logs, but current source and focused tests clearly reproduce the takeover-as-fatal path after prompt-lock release.

## Solution Assessment

No, not yet; a simple retry loop is not obviously safe because the current fence protects against unowned session writes, so maintainers need to choose recovery semantics that preserve ordering and delivery guarantees.
