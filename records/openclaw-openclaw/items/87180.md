## Summary

Keep open: current `main` still treats embedded session takeover after prompt-lock release as fatal/no-op cleanup, and the remaining fix needs a session-safe recovery decision rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to avoid silently losing replies when `EmbeddedAttemptSessionTakeoverError` occurs after the embedded session lock is released during provider retry or timeout windows.

## Best Possible Solution

Define and implement a session-aware takeover recovery contract that preserves stale-write safety while making channel failure visible or replaying the turn only when replay is proven safe.

## Reproduction Assessment

Yes, source-reproducible at medium confidence: current tests and code show external session edits during prompt-lock release produce `EmbeddedAttemptSessionTakeoverError`. A live Feishu/DeepSeek retry reproduction was not run in this read-only review.

## Solution Assessment

Unclear: retry/backoff alone is not obviously safe because the transcript may have advanced. The better fix likely needs maintainer-approved replay, queueing, or visible-failure semantics.
