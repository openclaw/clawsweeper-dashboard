## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested A2A ingress echo and relay surface, and the proposed branch remains draft with blocking correctness and security-sensitive ordering issues.

## What This Changes

The PR adds config-gated `sessions_send` ingress echo, nested relay guarding, dual-channel A2A relay mirroring, docs, and targeted tests across session tools, gateway loopback, config schema, and outbound message-action handling.

## Best Possible Solution

Keep the PR as a draft until the branch moves all guard checks before outbound side effects, makes strict relay delivery enforceable or removes that contract, uses turn-specific relay idempotency, accounts for provider account identity in bound-session protection, regenerates config artifacts, and adds the required changelog entry.

## Reproduction Assessment

Yes. The main blocker has a high-confidence static reproduction path: enable ingress echo, invoke `sessions_send` from a session whose latest user message has inter-session `sessions_send` provenance, and the PR path attempts a channel send before returning the nested-guard `forbidden` result.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best mergeable shape because it introduces externally visible sends before the guard and documents strict relay behavior that the implementation cannot enforce.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Move the nested guard before ingress echo:** `src/agents/tools/sessions-send-tool.ts:284-296`
  - body: The new ingress echo runs before the nested `sessions_send` guard. With ingress echo enabled, a recursive A2A call can still post to the target channel and only then return `forbidden`, which defeats the default guard's purpose. Move the guard before any echo or relay delivery side effect.
  - confidence: 0.91
- **[P2] Honor strict relay delivery failures:** `src/agents/tools/sessions-send-tool.a2a.ts:88-95`
  - body: `relay.requireDelivery` is exposed in config and docs, but relay sends always catch and log failures here. Since the A2A flow then continues, strict relay delivery can never block or abort as promised. Either propagate failures when `requireDelivery` is true or remove the strict config semantics.
  - confidence: 0.88
- **[P2] Include the relay turn in idempotency keys:** `src/agents/tools/sessions-send-tool.a2a.ts:84`
  - body: The relay idempotency key only includes run id, direction, and target index. In `mirrorTurns: "all"`, later turns with the same direction reuse the same key, so gateway idempotency can collapse distinct conversation turns. Add the turn number or another per-turn component.
  - confidence: 0.86
- **[P2] Commit generated config outputs:** `src/config/zod-schema.session.ts:62-80`
  - body: Adding these config fields changes the computed base config schema and config-doc baseline, but the PR does not update the generated schema/doc hash artifacts. Existing generated-schema or config-doc checks will fail, and the public config schema can stay stale.
  - confidence: 0.9
- **[P2] Compare account IDs before blocking bound sends:** `src/infra/outbound/message-action-runner.ts:148-153`
  - body: The new bound-session guard compares only channel and target. For multi-account providers such as Slack, the same target id can exist under different accounts or workspaces, so this can block a valid agent send because another agent is bound to a different account. Include the resolved account id in the guard comparison.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `docs/concepts/session-tool.md:130-156`
  - body: This is a user-facing feature/config change, but the PR does not include a `CHANGELOG.md` entry. Add a single-line entry under the active release section with valid contributor credit before this is merge-ready.
  - confidence: 0.82
