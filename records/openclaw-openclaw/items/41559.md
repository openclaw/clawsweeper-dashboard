## Summary

Keep open: this draft XL feature PR adds new A2A channel-visible delivery modes that are not on current main, and source review found blocking behavioral and guard-bypass issues before it can be considered mergeable.

## What This Changes

The PR adds config-gated A2A ingress echo, nested `sessions_send` guarding, dual-channel relay mirroring, docs, and targeted session/gateway/outbound tests.

## Best Possible Solution

Refresh the PR on current main, get maintainer sign-off on the new A2A visibility/config policy, and land a smaller implementation with regression coverage for guard-before-delivery, injected gateway callers, and strict relay failures.

## Reproduction Assessment

Yes, for the PR findings: source inspection of PR head shows ingress echo sends before the nested relay guard and uses the imported gateway caller. Relay delivery failures are also swallowed even when strict delivery is configured.

## Solution Assessment

No. The requested direction may be useful, but the current patch should move guard checks before any channel send, use the resolved `gatewayCall` consistently, and either implement or remove strict relay semantics before maintainer review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Run the nested relay guard before echoing:** `src/agents/tools/sessions-send-tool.ts:297`
  - body: The new ingress echo is sent before the nested `sessions_send` guard runs. With ingress echo enabled, a recursive send that should be rejected still posts the original payload to the target channel and only then returns `forbidden`, which bypasses the guard and can leak or duplicate messages. Move the guard ahead of any `send` side effect.
  - confidence: 0.91
- **[P2] Use the injected gateway caller for new calls:** `src/agents/tools/sessions-send-tool.ts:297`
  - body: This call uses the imported `callGateway` instead of the `gatewayCall` resolved from `opts.callGateway`. Callers that supply a custom gateway caller for loopback or isolated execution will still hit the default gateway for ingress echo, so the feature can fail or talk to the wrong gateway while the rest of `sessions_send` uses the injected caller.
  - confidence: 0.88
- **[P2] Honor strict relay delivery failures:** `src/agents/tools/sessions-send-tool.a2a.ts:88-95`
  - body: `relay.requireDelivery` is added to config/docs, but relay sends always catch and log failures. A user enabling strict relay delivery still gets a completed run with `relay.status: "pending"` instead of a blocked or failed result, contradicting the new contract.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or workflow changes, but the ingress echo can send channel-visible content before the new nested relay guard rejects the operation.

Concerns:

- **[medium] Ingress echo bypasses nested relay guard:** `src/agents/tools/sessions-send-tool.ts:297`
  - body: Because the echo `send` runs before the nested `sessions_send` guard, a forbidden recursive relay can still disclose the payload to the target channel. The guard should run before any outbound delivery side effect.
