## Summary

Keep open: current main does not implement the requested A2A ingress echo/dual-channel relay feature, and this draft branch still has correctness, security, config-contract, and proof blockers.

## What This Changes

This draft PR adds config-gated `sessions_send` ingress echo, nested relay guarding, dual-channel A2A relay mirroring, docs, tests, and an outbound bound-session guard.

## Best Possible Solution

Rebase the feature onto current main, preserve the existing thread/provenance/baseline/control-token protections, define strict relay/account semantics, update config metadata and docs, then attach redacted live proof for the A2A channel flow.

## Reproduction Assessment

Yes, for the patch defects by source inspection: enabling ingress echo can call gateway `send` before the nested `sessions_send` guard returns, and relay failures are caught despite strict delivery config. I did not execute the branch under the read-only review constraints.

## Solution Assessment

No. The feature direction may be useful, but this branch is not the narrowest mergeable solution until it preserves current main safety behavior and settles the new delivery/config semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move the nested guard before ingress echo:** `src/agents/tools/sessions-send-tool.ts:297-308`
  - body: The ingress echo gateway send runs before the nested `sessions_send` provenance guard. With ingress echo enabled, a recursive A2A call can still post to the target channel and only then return `forbidden`, which defeats the default loop guard.
  - confidence: 0.91
- **[P2] Preserve inter-session prompt annotation:** `src/agents/tools/sessions-send-tool.ts:393`
  - body: The target run receives the raw `message` instead of the annotated inter-session prompt text that current main uses. That drops the prompt marker telling the target agent this is tool-routed data rather than a direct user instruction.
  - confidence: 0.86
- **[P2] Keep rejecting thread session targets:** `src/agents/tools/sessions-send-tool.ts:267-278`
  - body: Current main rejects thread-scoped `sessions_send` targets before starting a run. This branch falls through into baseline capture, echo, and target-run setup, so a thread session key can be used for inter-agent coordination despite the current docs and guard.
  - confidence: 0.84
- **[P2] Honor strict relay delivery failures:** `src/agents/tools/sessions-send-tool.a2a.ts:73-95`
  - body: `relay.requireDelivery` is exposed in config and docs, but relay sends catch and log failures. Since the A2A flow then continues, strict relay delivery cannot block or abort as promised.
  - confidence: 0.88
- **[P2] Include the relay turn in idempotency keys:** `src/agents/tools/sessions-send-tool.a2a.ts:84`
  - body: The relay idempotency key only includes run id, direction, and target index. In `mirrorTurns: "all"`, later turns with the same direction reuse the same key, so gateway idempotency can collapse distinct conversation turns.
  - confidence: 0.86
- **[P2] Preserve baseline reply filtering:** `src/agents/tools/sessions-send-tool.a2a.ts:109-111`
  - body: Current main passes a baseline snapshot into the A2A wait path so delayed or fast target runs do not announce stale replies. This branch reads the latest assistant reply without that fingerprint check, reintroducing stale or duplicate A2A announce behavior.
  - confidence: 0.78
- **[P2] Suppress non-deliverable control tokens:** `src/agents/tools/sessions-send-tool.a2a.ts:119-130`
  - body: Current main filters `SILENT_REPLY`, `HEARTBEAT`, `REPLY_SKIP`, and `ANNOUNCE_SKIP` before A2A delivery. The relay path mirrors `latestReply` without that non-deliverable check, so control tokens can become visible channel messages.
  - confidence: 0.8
- **[P2] Regenerate config metadata for new keys:** `src/config/zod-schema.session.ts:65-80`
  - body: Adding these session config fields changes the public config schema/help metadata, but the PR only updates part of the schema surface. Generated config metadata can stay stale and generated-schema checks can fail.
  - confidence: 0.86
- **[P2] Compare account IDs before blocking bound sends:** `src/infra/outbound/message-action-runner.ts:148-153`
  - body: The new bound-session guard compares only channel and normalized target. For multi-account providers such as Slack, the same target id can exist under different accounts or workspaces, so this can block a valid agent send because another agent is bound to a different account.
  - confidence: 0.78
