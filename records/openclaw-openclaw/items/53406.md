## Summary

Keep open: current main has general active-run queue steering and ACP control commands, but ACP-bound chat replies and `/acp steer` still enter serialized ACP turns, while native sub-agent command steering is restart-based. The remaining work needs maintainer agreement on ACP live-input capability and sub-agent live-vs-restart semantics.

## What This Changes

Request to route messages sent in Discord or other chat threads into currently running ACP or sub-agent sessions as live input instead of waiting for the active turn to finish.

## Best Possible Solution

Define a capability-gated ACP live-input contract, route ACP-bound chat replies through it only when supported, preserve queued or restart fallbacks, and document native sub-agent steering boundaries.

## Reproduction Assessment

Yes, source-level for the remaining gap: start one ACP `runTurn` and submit a bound reply or `/acp steer` for the same session; current manager code and tests show same-session turns serialize until the active turn exits. Native embedded runs have newer queue steering, so the clearest remaining reproduction is the ACP-bound path.

## Solution Assessment

No current implementation is not the best final solution: `/acp steer` and bound follow-ups are useful but reuse serialized turn dispatch. The safer fix is a documented, backend-aware live-input capability with explicit fallback behavior rather than forcing all ACP runtimes into one assumed input model.
