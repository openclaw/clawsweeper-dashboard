## Summary

Keep this issue open. Current main still has ACP thread binding, output streaming, bound follow-up routing, `/acp steer`, and native sub-agent steering controls, but the requested live same-turn input path is still missing: ACP inbound chat and `/acp steer` enter the serialized `runTurn` path, and native sub-agent steering remains an abort-and-restart flow.

## What This Changes

The issue requests live inbound chat steering for already-running ACP and sub-agent sessions from Discord and other chat surfaces.

## Best Possible Solution

Define a documented active-turn input capability in the ACP runtime and manager contract, route bound chat replies to that path while a turn is running only when the backend advertises support, preserve queued-turn fallback otherwise, and make native sub-agent live-input versus restart behavior explicit in docs and tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start one ACP run and submit a second bound reply or `/acp steer` for the same session; current manager coverage shows the second `runTurn` is serialized until the first finishes, while native sub-agent steer aborts and restarts the run.

## Solution Assessment

Yes for the product direction, but not by bypassing the actor queue ad hoc. The maintainable solution is a typed backend-aware live-input capability with tests for supported and unsupported ACP backends, plus an explicit decision on whether native sub-agents keep restart semantics.
