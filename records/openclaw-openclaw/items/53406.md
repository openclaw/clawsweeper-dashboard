## Label Justifications

- `P2`: The issue affects important active agent steering workflows, but it is a bounded feature/product gap rather than an emergency outage.
- `impact:message-loss`: Messages intended to steer a running chat-surface session can be delayed until after the active turn rather than delivered to the running agent.
- `impact:session-state`: The requested behavior changes how input enters an already-running ACP or sub-agent session without losing that session’s progress.

## Summary

Keep open: current main and v2026.5.22 still do not provide a live input path for running ACP turns, and controlled sub-agent steering is implemented as abort-and-restart rather than same-turn input. The related main-session steering work is adjacent, but it does not settle the ACP runtime contract or sub-agent live-input semantics requested here.

## What This Changes

The issue requests live same-turn input routing from Discord and other chat surfaces into running ACP and sub-agent sessions.

## Best Possible Solution

Define a generic, permission-aware active-run input contract for ACP and controlled sub-agents, then wire channel ingress so authorized thread messages can reach that contract without bypassing queue policy, ownership checks, or session-state safety.

## Reproduction Assessment

Yes, by source inspection: Discord jobs are serialized before processing, ACP steer calls the serialized ACP turn path, and controlled sub-agent steering restarts the run. I did not run a live Discord or ACP harness scenario in this read-only review.

## Solution Assessment

Not applicable as a patch. The best fix is a maintainer-approved live-input contract across ACP manager, channel ingress, and sub-agent control, not a one-off Discord bypass.
