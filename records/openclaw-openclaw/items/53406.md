- `P2`: The issue affects a bounded but important interactive ACP/sub-agent chat workflow, with current source evidence and open candidate fixes but no merged implementation.
- `impact:session-state`: Guidance sent while an agent is active can miss the active session context and arrive only as a later turn.
- `impact:message-loss`: Messages are not necessarily dropped, but they are misdelivered relative to the intended active-run steering window.

## Summary

Keep open: current main and the latest release still serialize or reject the active-input path before a Discord ACP/sub-agent message can reliably steer a running session, and the related open PRs are candidate fix work rather than merged implementation.

## What This Changes

The issue requests live steering input for running ACP or sub-agent sessions from Discord threads and other chat surfaces, instead of waiting until the active turn finishes.

## Best Possible Solution

Keep this open as the Discord ACP/sub-agent surface while maintainers resolve the broader active-run steering cluster with a lifecycle-aware steering fix plus any required admission bypass, preserving non-steer serialization and delivery ordering.

## Reproduction Assessment

Yes, at source level: in steer mode, a same-session message sent while the run is active but not streaming still hits streaming-only gates or the per-session admission queue before it can steer the active runtime. I did not run a live Discord ACP session in this read-only review.

## Solution Assessment

Unclear as a final implementation path. Lifecycle-aware steering is the right core shape, but maintainers still need to decide how to combine it with same-session admission bypasses across Discord and other chat surfaces.
