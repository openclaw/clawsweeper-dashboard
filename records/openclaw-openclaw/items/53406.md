## Label Justifications

- `P2`: This is an important active-agent usability gap with bounded channel/session impact, but it is not an emergency outage or security bypass.
- `impact:message-loss`: Messages meant to steer a running chat-surface ACP or sub-agent session can be delayed until after the active turn instead of reaching the running agent.
- `impact:session-state`: The request changes how input is associated with an already-running ACP or sub-agent session without losing progress or corrupting ownership.

## Summary

Keep open: current main and v2026.5.22 still do not provide a live input path for running ACP turns or controlled sub-agent sessions from chat surfaces. The related main-session steer work and Discord ACP delivery PRs are adjacent, but they do not settle the ACP runtime contract or sub-agent live-input semantics requested here.

## What This Changes

The issue requests live chat-surface input for running ACP and sub-agent sessions so Discord thread messages can steer active work instead of waiting for a later turn.

## Best Possible Solution

Define a generic, permission-aware active-run input contract for ACP and controlled sub-agent sessions, then wire authorized channel/thread ingress through that contract without bypassing queue policy or session ownership checks.

## Reproduction Assessment

Yes, source-level: Discord ingress is still serialized by session-derived queue key, ACP steer enters the serialized ACP turn actor, and sub-agent steering restarts rather than injecting live input. I did not run a live Discord/ACP harness scenario in this read-only review.

## Solution Assessment

Not applicable as a patch. The requested behavior is valid, but the maintainable solution needs a maintainer-approved active-run input contract across ACP runtime adapters, channel ingress, and sub-agent control rather than a narrow Discord bypass.
