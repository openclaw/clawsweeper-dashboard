## Summary

Keep open: current main has ACP bindings, output streaming, `/acp steer`, and native active-run steering, but ACP-bound chat and `/acp steer` still enter serialized ACP turns, while native sub-agent steering remains a restart path. This needs maintainer agreement on live-input semantics rather than conservative cleanup closure.

## What This Changes

Requests live input for running ACP and sub-agent sessions so Discord/thread messages can correct or redirect the agent mid-turn instead of waiting or restarting.

## Best Possible Solution

Define a documented ACP live-input capability, route bound chat replies to it only when supported, keep queued/restart fallback behavior, and document whether native sub-agent steering is restart-based or live-input capable.

## Reproduction Assessment

Yes, source-level. Start one ACP `runTurn` and submit a bound reply or `/acp steer` for the same session; current manager code and tests show same-session turns are serialized until the active turn exits.

## Solution Assessment

No. The current `/acp steer` and bound follow-up paths are useful controls, but they reuse serialized turn dispatch; the safer solution is an explicit backend-aware live-input capability with documented fallback behavior.
