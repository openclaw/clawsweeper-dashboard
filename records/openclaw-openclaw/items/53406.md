## Summary

Keep this issue open: current main has useful ACP thread binding, output streaming, `/acp steer`, and general active-run steering for Pi/native Codex, but ACP chat input still goes through serialized `runTurn` calls and native sub-agent explicit steering remains a restart flow.

## What This Changes

Request live chat input so Discord and other thread-bound chat replies can steer running ACP and sub-agent sessions before the current turn finishes.

## Best Possible Solution

Define a documented ACP live-input capability, route bound chat replies to it only when supported, keep queued/restart fallback behavior, and document whether native sub-agent steering is restart-based or live-input capable.

## Reproduction Assessment

Yes, source-level. Start one ACP `runTurn` and submit a bound reply or `/acp steer` for the same session; current manager code and tests show same-session `runTurn` calls are serialized until the active turn exits.

## Solution Assessment

No, the existing `/acp steer` and bound follow-up paths are useful but incomplete for this request because they reuse serialized turn dispatch. The safer direction is an explicit backend-aware live-input capability with documented fallback behavior.
