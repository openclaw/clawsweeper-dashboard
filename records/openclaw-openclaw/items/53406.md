## Summary

Keep open: current main still routes ACP bound replies and `/acp steer` through serialized ACP turns, while native sub-agent steering is an abort-and-restart flow rather than same-turn input. Related open PRs address adjacent Discord queueing and sub-agent restart persistence, but neither completes the ACP live-input contract requested here.

## What This Changes

Request chat-surface routing so Discord and similar thread messages can steer active ACP and sub-agent runs before the current turn finishes.

## Best Possible Solution

Define a backend-advertised ACP live-input capability with explicit fallback behavior, then decide whether native sub-agent chat steering should become same-turn input or remain documented as restart-based.

## Reproduction Assessment

Yes, source-level: start one ACP `runTurn` and submit a bound reply or `/acp steer` for the same session; the manager serializes the second turn until the active one exits. Native sub-agent steering is also source-visible as abort-and-restart rather than same-turn input.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. The maintainable path is capability-gated ACP live input with documented fallback semantics, plus an explicit decision on sub-agent live input versus restart steering.
