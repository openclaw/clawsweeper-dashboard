## Summary

Keep open: current main still has ACP chat replies and `/acp steer` routed through serialized ACP turns, while native sub-agent steering is implemented as an abort-and-restart path rather than same-turn input. The remaining work needs a maintainer decision on a capability-gated ACP live-input contract and on whether sub-agent chat steering should become true live input or stay restart-based.

## What This Changes

The issue requests live chat input from Discord and other surfaces into running ACP and sub-agent sessions so users can redirect active work mid-turn.

## Best Possible Solution

Define a backend-advertised ACP live-input capability, route bound chat replies through it only when supported, keep queued or restart fallback behavior explicit, and document the native sub-agent live-vs-restart boundary.

## Reproduction Assessment

Yes, source-level: start one ACP `runTurn` and submit a bound reply or `/acp steer` for the same session; current manager code and tests show the second same-session turn waits until the active one exits. The native sub-agent side is also source-reproducible as restart-based steering rather than live input.

## Solution Assessment

No current implementation is not the final solution: existing `/acp steer` and bound follow-ups are useful but still reuse serialized turn dispatch. The maintainable path is a capability-gated live-input contract with explicit fallback behavior rather than assuming every ACP backend can accept same-turn input.
