## Label Justifications

- `P2`: This is a normal-priority interactive agent/channel workflow improvement with bounded blast radius, not a core outage or security bypass.
- `impact:message-loss`: The issue is about chat input not reaching the running ACP or sub-agent session at the time the user sends it.
- `impact:session-state`: The request centers on steering active ACP/sub-agent session state without killing and recreating the session.

## Summary

Keep open: current main has useful steering pieces, but ACP chat replies and `/acp steer` still go through serialized ACP turns, while native sub-agent steering is implemented as restart/remap rather than true same-turn input. The remaining work needs a maintainer decision on an ACP live-input capability contract and on the sub-agent live-vs-restart boundary.

## What This Changes

The issue requests same-turn interactive input for running ACP and sub-agent sessions from Discord and other chat surfaces.

## Best Possible Solution

Define a backend-advertised ACP live-input capability and explicit fallback behavior, then decide whether native sub-agent steering should gain a live-input path or remain documented restart/remap behavior.

## Reproduction Assessment

Yes, source-level: start one ACP `runTurn` and submit a bound reply or `/acp steer` for the same session; the second turn waits behind `withSessionActor`. Native sub-agent steering is also source-reproducible as restart/remap based through `steerControlledSubagentRun`.

## Solution Assessment

No current implementation is the final solution. The narrow maintainable path is a capability-gated live-input contract for ACP plus an explicit product decision for sub-agent live input versus restart/remap semantics.
