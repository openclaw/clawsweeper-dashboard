## Summary

Keep open. Current main and v2026.6.6 still only partially cover the request: embedded/main-session steering exists after a message reaches reply resolution, and explicit `/acp steer` exists, but Discord ingress still serializes same-key jobs and automatic ACP bound dispatch still submits follow-ups as normal prompts through a per-session ACP actor queue. Related work in https://github.com/openclaw/openclaw/issues/48003, https://github.com/openclaw/openclaw/pull/73365, and https://github.com/openclaw/openclaw/pull/89238 covers important adjacent main-session steering pieces, but it does not fully close the ACP thread-bound live-input loop requested here.

## What This Changes

The issue requests that messages sent in a Discord or other chat thread bound to a running ACP or sub-agent session be delivered to the running agent as live input instead of waiting for the current turn to finish.

## Best Possible Solution

Design one owner-approved live-input path that lets eligible same-thread messages bypass transport/admission serialization only when an active steer-capable session exists, and route ACP follow-ups as steer through the ACP manager/runtime without weakening ordering, command bypass, or fallback behavior.

## Reproduction Assessment

Yes, source-level. Discord messages still enter a same-key run queue before reply policy, automatic ACP bound dispatch still uses prompt mode, and ACP runTurn is serialized per session, so the requested automatic mid-run input path is not complete on current main.

## Solution Assessment

No, current main is only a partial solution. Explicit `/acp steer` and embedded-session steering exist, but completing the requested automatic chat input loop needs a coordinated channel/admission/ACP manager design rather than treating a single queue bypass as sufficient.
