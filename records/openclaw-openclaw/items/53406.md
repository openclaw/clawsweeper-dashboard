## Label Justifications

- `P2`: The issue affects an interactive ACP/sub-agent chat workflow with bounded channel and session blast radius.
- `impact:session-state`: User guidance can miss the active run's session context and arrive only after the run has completed.
- `impact:message-loss`: Messages are not permanently lost, but they miss the live-input delivery window and can become the wrong turn.

## Summary

Keep open: current main and latest release still serialize same-session Discord input before it can reach steering, and the core steering helpers still require an active stream. The broader active-run steering tracker and open PRs cover parts of this, but they do not yet fully close the Discord ACP/sub-agent thread input request.

## What This Changes

Issue asks for Discord and other chat messages sent during active ACP or sub-agent sessions to steer the running agent instead of waiting for the current turn to finish.

## Best Possible Solution

Resolve the active-run steering cluster by rebasing, reviewing, and landing a focused lifecycle-aware steering fix plus any needed same-session admission bypass, while preserving non-steer serialization, replay dedupe, typing cleanup, and delivery guarantees.

## Reproduction Assessment

Yes. Source inspection shows same-session Discord work is serialized by `job.queueKey` before reply processing, and current steering helpers still require `isStreaming` before accepting input; no live Discord run was executed in this read-only review.

## Solution Assessment

Unclear. The desired behavior is documented, but the best implementation still needs maintainer judgment on how to combine Discord/channel admission changes with the core lifecycle-aware steering fix.
