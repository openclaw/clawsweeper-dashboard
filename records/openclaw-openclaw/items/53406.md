- `impact:message-loss`: Messages meant to steer a running chat-surface ACP or subagent session are delayed into later turns or restart flows instead of reaching the active run.

## Summary

Keep open: current main and v2026.5.19 still lack an ACP active-turn input primitive, and controlled subagent steering is still abort-and-restart rather than same-turn input. The main-session steering work in https://github.com/openclaw/openclaw/issues/48003 and https://github.com/openclaw/openclaw/pull/73365 is adjacent, but it does not cover the ACP runtime contract or controlled subagent semantics requested here.

## What This Changes

Add live chat-surface input so messages in a bound Discord or other chat thread reach a running ACP or sub-agent session during the active turn instead of waiting or restarting.

## Best Possible Solution

Define a backend-advertised ACP live-input or turn-steer capability with documented fallback behavior, then explicitly decide whether controlled subagent steering should remain restart-based or become same-turn input.

## Reproduction Assessment

Yes, source-level. Start one ACP runTurn and submit a bound reply or /acp steer for the same session; current main serializes that second runTurn behind the active turn, while controlled subagent steer is implemented as abort-and-restart rather than same-turn input.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. The maintainable path is capability-gated ACP live input with documented fallback semantics, plus an explicit decision on controlled subagent live input versus restart steering.
