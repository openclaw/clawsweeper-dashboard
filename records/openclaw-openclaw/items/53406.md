## Summary

Keep open: current main still has ACP chat replies and `/acp steer` enter the serialized ACP turn path, while native sub-agent steering remains restart-based rather than live same-turn input. The adjacent open PRs address Discord queue bypass and sub-agent restart persistence, but they do not complete the requested ACP live-input contract.

## What This Changes

The issue asks for Discord and other chat-thread messages to be delivered as live input to running ACP and sub-agent sessions instead of waiting for the active turn to finish.

## Best Possible Solution

Define a backend-advertised ACP live-input capability with explicit fallback behavior, then decide whether native sub-agent chat steering should become same-turn input or remain restart-based and documented.

## Reproduction Assessment

Yes, source-level: start one ACP `runTurn` and send a bound reply or `/acp steer` for the same session; current main routes it through a second serialized `runTurn`. Native sub-agent steering is also source-visible as abort-and-restart rather than same-turn input.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. The maintainable path is capability-gated ACP live input with documented fallback semantics, plus an explicit decision on sub-agent live input versus restart steering.
