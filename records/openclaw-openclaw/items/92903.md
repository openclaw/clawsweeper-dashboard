## Label Justifications

- `P1`: Wrong workspace bootstrap for a bound multi-agent channel can break active agent workflows and give a routed agent the default agent’s instructions.
- `impact:session-state`: The issue is about agent/session context drift where workspace context and session-store selection can resolve under the wrong agent.
- `impact:security`: Loading the wrong AGENTS.md or SOUL.md can leave intended per-agent safety, identity, or policy instructions inactive.

## Summary

Keep open: current main and latest release still drop the bound route agent before shared inbound context reaches session and workspace resolution, so unscoped bound sessions can still inject the default agent workspace. The two same-issue fix PRs are closed unmerged.

## What This Changes

The issue reports that a non-default bound agent receives Project Context bootstrap files from the default workspace instead of its configured per-agent workspace.

## Best Possible Solution

Thread the routed agent through inbound context as a fallback below session-key precedence, align get-reply, dispatch, and session-store resolution, add focused regression tests, and make an explicit upgrade decision for previously mis-stored sessions.

## Reproduction Assessment

Yes. At source level, an unscoped configured binding can rewrite `route.agentId` to a non-default agent while `buildChannelInboundEventContext` drops that fact, causing `resolveSessionAgentId` to fall back to the default agent.

## Solution Assessment

Yes on fix shape, but it needs maintainer acceptance of the upgrade boundary. The narrow repair is fallback-only route-agent resolution below session-key precedence, not a broad bootstrap rewrite or explicit AgentId override.
