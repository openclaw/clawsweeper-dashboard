## Label Justifications

- `P1`: Wrong workspace bootstrap for a bound multi-agent channel can break active agent workflows and load the default agent's instructions.
- `impact:session-state`: The issue is about agent/session context drift where Project Context and session store selection can resolve under the wrong agent.
- `impact:security`: Loading the wrong AGENTS.md or SOUL.md can leave intended per-agent safety, identity, or policy instructions inactive.

## Summary

Keep open: current main and v2026.6.8 still omit the routed agent from inbound context, so an unscoped bound channel session can resolve workspace/bootstrap context under the default agent; both same-issue fix PRs are closed unmerged and the remaining repair needs maintainer handling for fallback precedence and session continuity.

## What This Changes

Ensure bound non-default channel agents load Project Context/bootstrap files from their configured per-agent workspace instead of the default workspace when the inbound session key is unscoped.

## Best Possible Solution

Thread the routed agent through inbound context as a fallback below session-key precedence, align get-reply, dispatch, and session-store resolution, and make an explicit maintainer decision on any mis-stored default-agent session migration.

## Reproduction Assessment

Yes, source-reproducible: configured binding routing can carry a non-default route agent with an unscoped session key, but buildChannelInboundEventContext drops that agent and downstream resolution falls back to the default agent. I did not run a live Feishu deployment in this read-only review.

## Solution Assessment

Yes on direction, with a maintainer decision needed: the narrow maintainable fix is fallback-only route-agent resolution below session-key precedence, not treating routed AgentId as an explicit override. Maintainers should decide whether affected default-agent session records need migration or can start fresh under the correct agent.
