## Label Justifications

- `P1`: Wrong workspace bootstrap for a bound multi-agent channel can break active agent workflows and give a routed agent the default agent's instructions.
- `impact:session-state`: The issue is about agent/session context drift where Project Context and session store selection can resolve under the wrong agent.

## Summary

Keep open: current main and v2026.6.8 still drop the bound route agent from shared inbound context, so an unscoped bound session can resolve to the default agent workspace; both same-issue fix PRs are closed unmerged.

## What This Changes

The issue reports that Feishu-bound non-default agents inject Project Context bootstrap files from the default workspace instead of the configured per-agent workspace.

## Best Possible Solution

Thread the routed agent through inbound context as a fallback below session-key precedence, align get-reply, dispatch, and session-store resolution, and add regression coverage for unscoped bound sessions plus scoped command-target sessions.

## Reproduction Assessment

Yes, at source level: an unscoped configured binding can rewrite route.agentId to a non-default agent while buildChannelInboundEventContext drops that fact, causing resolveSessionAgentId to fall back to the default agent. I did not run a live Feishu deployment in this read-only review.

## Solution Assessment

Yes on fix shape, but it needs maintainer acceptance of the upgrade boundary: the narrow repair is fallback-only route-agent resolution below session-key precedence, not a broad bootstrap rewrite or explicit AgentId override.
