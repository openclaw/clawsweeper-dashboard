## Label Justifications

- `impact:session-state`: The bug is agent/session context drift: an unscoped bound inbound turn can resolve workspace and session state under the default agent.

## Summary

Keep open: current main and v2026.6.9 still drop the bound route agent before workspace/session resolution, and https://github.com/openclaw/openclaw/pull/95118 is the open closing fix candidate that should be reviewed rather than closing this issue early.

## What This Changes

The issue reports that multi-agent channel turns routed to a non-default agent inject workspace bootstrap files from the default workspace instead of that agent's configured workspace.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95118 if maintainers accept the corrected session boundary, or require a focused migration if preserving previously mis-associated records is necessary.

## Reproduction Assessment

Yes, source-level: configured bindings can carry a non-default route.agentId while current main drops that value from inbound context, and unscoped session resolution then falls back to the default agent. I did not run a live Feishu deployment in this read-only review.

## Solution Assessment

Yes on fix shape: the narrow maintainable repair is to use the routed agent only as a fallback below explicit and session-key agent ids. The remaining maintainer question is upgrade policy for conversations previously stored under the wrong agent.
