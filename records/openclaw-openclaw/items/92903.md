## Label Justifications

- `P1`: The issue affects a real multi-agent channel workflow where a bound agent can load and persist under the wrong agent scope.
- `impact:session-state`: The bug is agent/session context drift: an unscoped bound inbound turn can resolve workspace and session state under the default agent.

## Summary

Keep open: current main and the latest release still lose the bound route agent before session/workspace resolution, so an unscoped bound inbound turn can still load the default agent workspace. The issue is now paired with https://github.com/openclaw/openclaw/pull/95118, which is the active fix candidate and should be reviewed rather than closing this report early.

## What This Changes

The issue reports that multi-agent inbound channel turns inject bootstrap files from the default workspace instead of the bound agent's configured workspace.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95118 if maintainers accept the corrected session boundary, or require a focused migration if preserving previously mis-associated default-agent records is necessary.

## Reproduction Assessment

Yes, source-level: configured bindings can carry a non-default route.agentId while current main drops that value from inbound context, and the unscoped session resolver then falls back to the default agent. I did not run a live Feishu deployment in this read-only review.

## Solution Assessment

Yes on fix shape, but not yet complete: the narrow maintainable repair is to use the routed agent only as a fallback below explicit and session-key agent ids. The remaining maintainer decision is whether fresh sessions for previously mis-stored unscoped conversations are acceptable or require migration.
