## Summary

Keep open: current main still appears to drop the bound route agent before the inbound session/workspace handoff, and the two PRs that explicitly proposed this fix are closed unmerged. The report is source-reproducible against the current channel inbound, session-agent, and Feishu paths, so this is still necessary work rather than conservative cleanup.

## What This Changes

The issue reports that channel-bound non-default agents inject bootstrap Project Context from the default workspace instead of the bound agent's own workspace.

## Best Possible Solution

Thread the configured route agent into `ctx.AgentId` and use it only as a fallback below explicit/session-key agent ids for reply, dispatch, and session-state ownership, with focused regression coverage for Feishu or generic configured bindings using unscoped session keys.

## Reproduction Assessment

Yes for a source-level reproduction, not a live Feishu run. Current main shows an unscoped configured-binding route can carry the bound agent to Feishu, but the shared inbound context builder does not expose that agent id to the session/workspace resolver.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve the route agent as a fallback, not to rewrite bootstrap loading. The core bootstrap loader already reads the supplied workspace, so the broken boundary is agent resolution before workspace selection.
