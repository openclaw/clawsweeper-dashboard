## Summary

Keep open: the reported WebChat auto-TTS bug is plausible, but this PR is not mergeable because it corrupts source encoding, changes final-mode block synthesis in a way that can duplicate audio, bypasses shared media normalization, and lacks real behavior proof. The same work is now better represented by https://github.com/openclaw/openclaw/issues/82628 and the narrower open fix PR at https://github.com/openclaw/openclaw/pull/82701.

## What This Changes

The PR changes speech-core auto-TTS, shared reply dispatch, and plugin-sdk reply payload files to try to deliver generated local TTS audio to WebChat.

## Best Possible Solution

Consolidate on the narrower accumulated-final-tail fix in https://github.com/openclaw/openclaw/pull/82701, keep `trustedLocalMedia` internal, preserve shared media normalization, and require real dispatch/WebChat proof before merge.

## Reproduction Assessment

Yes from source inspection: current main generates local TTS media, WebChat only embeds local audio when `trustedLocalMedia` is true, and the accumulated block TTS-only final payload currently drops that flag. I did not run a live WebChat or MiniMax repro in this read-only review.

## Solution Assessment

No. The PR touches the right area, but it corrupts source encoding, synthesizes block chunks in final mode, bypasses shared media normalization, and still misses the accumulated final-tail trust propagation needed for WebChat.
