## Summary

Keep this PR open because it targets a plausible current WebChat auto-TTS gap, but the proposed patch is not mergeable as-is: it introduces mojibake into user-visible/runtime text, rewrites a public SDK file as a binary/BOM-tainted change, and re-exposes an internal local-media trust flag that current main intentionally keeps out of the plugin SDK.

## What This Changes

The PR changes speech-core auto-TTS payload handling and dispatch media normalization so WebChat can receive generated TTS audio.

## Best Possible Solution

Keep the fix internal to the TTS/dispatch/WebChat pipeline, preserve the public SDK trust boundary, and add focused regression coverage for block-mode auto-TTS plus trusted local WebChat audio delivery.

## Reproduction Assessment

No high-confidence live reproduction was run, but source inspection gives a clear reproduction path: WebChat block replies reach auto-TTS as kind "block", current final-mode TTS skips them, and WebChat audio embedding requires trustedLocalMedia.

## Solution Assessment

No; the PR points at the right pipeline but is not the best safe implementation because it corrupts existing text and exposes an internal trust flag through the public plugin SDK.
