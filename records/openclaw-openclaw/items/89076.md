## What This Changes

The PR guards generic system prompt tool-report metadata reads so unreadable tool fields produce lossy report entries while healthy sibling tool entries remain reported.

## Best Possible Solution

Land this narrow guarded report-reader change, or the chosen canonical sibling, after maintainer review and normal validation while keeping lossy handling local to diagnostics.

## Reproduction Assessment

Yes. Current main directly reads report-only tool metadata in buildToolsEntries, so a throwing getter on name, description, label, or parameters can abort report generation before sibling entries are produced.

## Solution Assessment

Yes. Guarding only the diagnostic report-reader path is the narrowest maintainable fix because callers still pass normal runtime tools and the AgentTool execution contract is unchanged; shared safe-reader extraction can wait unless the parallel hardening keeps growing.
