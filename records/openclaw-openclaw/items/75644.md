## Summary

Keep open. Current main still lacks the `plugins.bundledMode` config key and still uses bundled runtime compat paths that can widen provider/extractor loading under restrictive `plugins.allow`; this PR is directionally correct, but it still needs public docs, a changelog entry, and green exact-head gates before merge.

## What This Changes

The PR adds `plugins.bundledMode`, wires `respect-allow` into bundled provider and extractor runtime compat paths, enables web-provider runtime caching in that mode, refreshes generated config metadata, and adds focused tests.

## Best Possible Solution

Land the default-preserving opt-in runtime mode after adding public docs and changelog coverage, then keep broader manifest-first runtime-loader cleanup as separate architecture work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main rejects `plugins.bundledMode` in the strict schema and still routes bundled provider/extractor paths through compat flags that can widen runtime plugin loading under a restrictive `plugins.allow`.

## Solution Assessment

Mostly yes. The opt-in config is the narrow maintainable fix because it preserves default compatibility while giving operators an allowlist-respecting mode; the safer completion path is docs/changelog coverage plus green exact-head gates.
