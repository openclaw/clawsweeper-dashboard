## Summary

Keep open. The closed Ollama `num_ctx` regression explains part of the symptom, but current main still has a clear fresh-onboarding/default-model path where OpenClaw records a large Ollama context window while native `/api/chat` leaves `options.num_ctx` unset, which matches the reported 4096-token Ollama load and persona/context loss.

## What This Changes

The issue reports a v2026.5.12 regression where Ollama/local-model agents stop following injected workspace persona files after update or fresh onboarding.

## Best Possible Solution

Decide the native Ollama policy for fresh setup, then either make onboarding explicitly set a safe `params.num_ctx` for selected local models or make status/docs avoid implying that discovered `contextWindow` is an active Ollama runtime context.

## Reproduction Assessment

Yes, at source level. Fresh Ollama setup records discovered `contextWindow` metadata without `params.num_ctx`, and current native `/api/chat` code/tests leave `options.num_ctx` unset in that case, matching the reported 4096-token Ollama load; I did not run live Ollama in this read-only review.

## Solution Assessment

Unclear. The doctor migration is a good compatibility fix for older explicit configs, but the fresh onboarding path needs an owner decision on whether setup should opt into `params.num_ctx` or preserve the current explicit-only runtime policy.
