## Summary

Keep this PR open because it is still the active fix path for a real core edit-tool corruption bug, but the current patch has a blocking offset-mapping defect in the new NFKC fast path.

## What This Changes

The PR preserves original LF-normalized content as the edit write/diff base, maps fuzzy-normalized match offsets back to original content before splicing, and adds edit-diff regression tests for unrelated-line and Unicode cases.

## Best Possible Solution

Land this PR only after fixing the NFKC fast path so offset mapping runs whenever NFKC changes the line content, with a regression test for length-neutral mixed expansion/composition before the target.

## Reproduction Assessment

Yes. Current-main source inspection shows the fuzzy path can normalize the whole file before write/diff, and the new PR-specific blocker is reproducible from string offsets where NFKC keeps total length equal but shifts a target position.

## Solution Assessment

No. The overall fix location is right, but the length-only NFKC fast path is not safe enough for user-file write ranges; it should compare string identity or always run the segment mapper when NFKC changes the line.
