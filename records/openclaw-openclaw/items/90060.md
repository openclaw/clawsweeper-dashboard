## Repair Work Prompt

Repair this PR's fuzzy edit offset mapping without changing the public edit tool contract. The current PR keeps original content as baseContent, but mapNormalizedOffsetToOriginal assumes same-line normalization is one-to-one; normalizeForFuzzyMatch calls NFKC, which can expand characters such as U+FB03 before the fuzzy target and make the splice hit the wrong original range. Implement an offset-aware mapping from fuzzy-normalized offsets to original LF-normalized UTF-16 offsets, or otherwise constrain normalization with tests proving correct offsets. Preserve the intended fix for https://github.com/openclaw/openclaw/issues/89994: fuzzy matching may locate text with smart quotes/dashes/spaces, but unrelated bytes must remain unchanged and the model/TUI diff must be against the original base. Add regression coverage in src/agents/sessions/tools/edit-diff.test.ts for a same-line NFKC-expanding character before the fuzzy match, plus the existing unrelated-line preservation cases.
