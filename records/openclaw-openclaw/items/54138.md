## Summary

Keep open. Current main still sends heartbeat reply payloads through the `isHeartbeat` branch unchanged before channel delivery, so the reported bracket-format tool blocks can still reach users. The issue is also paired with open closing-syntax PR #54165, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests preventing heartbeat-delivered reply payloads from exposing uppercase `[TOOL_CALL]` and `[TOOL_RESULT]` bracket-format blocks to Telegram, Discord, or other channel users.

## Best Possible Solution

Keep this issue open and review/land PR #54165, or land an equivalent focused fix in `src/auto-reply/reply/agent-runner-payloads.ts` that strips uppercase bracket tool-call/result blocks from heartbeat reply text before final channel delivery, with regression coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts`. Track the broader interactive/XML leak separately if it still reproduces.

## Reproduction Assessment

Yes. A unit-level call to `buildReplyPayloads` with `isHeartbeat: true` and payload text containing a same-line `[TOOL_CALL]...[/TOOL_CALL]` block would currently return that text unchanged, and the final runner passes heartbeat state into that builder.

## Solution Assessment

Yes for the narrow heartbeat bracket leak: sanitizing the heartbeat delivery branch plus focused regression tests is the smallest maintainable fix. Migrating heartbeat to isolated-session mode is broader architecture work and should not be required before blocking this user-visible leak.
