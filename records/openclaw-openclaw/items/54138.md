## Summary

Keep open. Current main and the latest release still pass heartbeat reply payload text through unchanged before channel delivery, and the open closing-syntax PR #54165 remains the right implementation candidate rather than a reason to close this issue early.

## What This Changes

The issue asks OpenClaw to prevent heartbeat replies from delivering uppercase bracket-format `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]` blocks to users.

## Best Possible Solution

Keep this issue open and review, land, or revise PR #54165, or land an equivalent focused fix in `src/auto-reply/reply/agent-runner-payloads.ts` with regression coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts`. Track the broader interactive/XML leak separately if it still reproduces after the heartbeat fix.

## Reproduction Assessment

Yes. A unit-level call to `buildReplyPayloads` with `isHeartbeat: true` and payload text containing a same-line `[TOOL_CALL]...[/TOOL_CALL]` block would currently return that text unchanged, and the final runner passes heartbeat state into that builder.

## Solution Assessment

Yes for the narrow reported leak: sanitize the heartbeat delivery branch before channel delivery and add focused regression tests. The larger isolated-session or stream-transition work is broader architecture and should not block the minimal leak fix.
