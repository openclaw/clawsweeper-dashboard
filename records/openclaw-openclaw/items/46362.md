## Summary

Keep open. The gateway-registry part has already been solved on current main by the pinned HTTP route registry, but the Discord voice escalation feature is still unique, stale against the current plugin layout, missing real behavior proof, and has blocking command-handling issues.

## What This Changes

The PR adds spoken Discord voice commands that persist session model overrides and also adds a gateway plugin-registry bridge with route-registration tests.

## Best Possible Solution

If accepted, port the voice model-switching design to `extensions/discord/src/voice/*`, use explicit and configurable model-switch commands, consume command-only utterances before agent routing, handle first-turn session creation, and keep current main's pinned HTTP route-registry implementation.

## Reproduction Assessment

Yes for the PR defects from source inspection: the diff matches common phrases like `think about this`, awaits `handleEscalationTrigger`, and then continues into agent routing, and it returns when the session entry is missing. Reproduction of the actual Discord voice behavior is not proven because the manual voice checks are unchecked.

## Solution Assessment

No. The gateway fix should stay on current main's pinned registry path, and the voice feature needs a fresh plugin-path implementation with explicit command semantics, first-turn persistence, and real Discord voice proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Port escalation to the active Discord plugin path:** `src/discord/voice/manager.ts:59`
  - body: Current main no longer uses `src/discord` for Discord voice; the active implementation lives under `extensions/discord/src/voice`. This branch needs to be rebased and ported there, otherwise the voice feature is not implemented against the current runtime surface.
  - confidence: 0.91
- **[P2] Consume model-switch utterances before agent routing:** `src/discord/voice/manager.ts:692-693`
  - body: After `handleEscalationTrigger` runs, execution still resolves speaker context and sends the same transcript to the agent. A command-only phrase like `go deep` will both change the model override and be answered as a normal user request.
  - confidence: 0.94
- **[P2] Require explicit medium-mode commands:** `src/discord/voice/manager.ts:84-85`
  - body: The `think about this/that/it` pattern is normal conversational speech, so ordinary prompts can silently persist a model override instead of being treated only as user content. Make the trigger an explicit mode-switch command.
  - confidence: 0.9
- **[P2] Persist overrides for first-turn voice sessions:** `src/discord/voice/manager.ts:781-784`
  - body: When `store[entry.route.sessionKey]` is absent, the handler returns without applying the override. A brand-new voice session can hit that path before the first agent turn creates the session entry, so the first spoken model-switch command is dropped.
  - confidence: 0.82
- **[P3] Allow registry cleanup without unsafe casts:** `src/plugins/runtime.ts:58-59`
  - body: The PR's getter returns `PluginRegistry | null`, but the setter only accepts `PluginRegistry`, forcing tests to clear the global with `null as never`. Either widen the setter or follow current main's pinned registry surface instead.
  - confidence: 0.8
