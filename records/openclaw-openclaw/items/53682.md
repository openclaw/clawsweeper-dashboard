## Summary

Keep open: the reported heredoc approval failure is plausible and not fixed on current main, but this PR currently weakens the approval boundary for timeout fallback approvals and still lacks after-fix real behavior proof.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals run the original command text instead of the rebuilt enforced allowlist command.

## Best Possible Solution

Land a narrow gateway approval fix that preserves raw heredoc text only after explicit `allow-once` or `allow-always`, keeps `enforcedCommand` for allowlist timeout fallback, and adds focused regression tests plus real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. Current main rebuilds allowlist-satisfied commands into `enforcedCommand` and uses that after approval, while heredoc analysis tests show heredoc bodies are parsed away from the executable argv; I did not run a live gateway/osascript repro in this read-only review.

## Solution Assessment

No. The PR keys raw execution off `requiresHeredocApproval`, but the same follow-up path is also used by allowlist timeout fallback; the fix should key raw execution off an explicit `allow-once` or `allow-always` decision.
