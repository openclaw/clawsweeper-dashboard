## Summary

Keep open: the PR targets a real gateway exec heredoc approval bug, but the proposed patch still has a security-boundary defect because it would run raw shell text for allowlist timeout fallback approvals and it lacks after-fix real behavior proof.

## What This Changes

The PR changes the gateway exec approval follow-up so heredoc-required approvals pass the original command text to execution instead of the rebuilt enforced allowlist command.

## Best Possible Solution

Land a narrow gateway approval fix that uses raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps timeout fallback on `enforcedCommand`, and adds focused tests plus redacted real gateway proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main requires approval for allowlist-satisfied heredocs but still passes the rebuilt `enforcedCommand` to execution. I did not run a live gateway approval flow in this read-only review.

## Solution Assessment

No. The proposed fix uses raw `params.command` for every heredoc-required approval, but the same follow-up path also handles allowlist timeout fallback where no explicit `allow-once` or `allow-always` decision occurred.
