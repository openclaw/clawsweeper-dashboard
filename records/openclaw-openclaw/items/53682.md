## Summary

Keep open: the PR targets a real gateway heredoc approval bug, but the patch is not mergeable because it would also run raw shell text for allowlist timeout fallback approvals without an explicit human allow decision.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute raw `params.command` instead of the rebuilt `enforcedCommand`.

## Best Possible Solution

Preserve raw heredoc text only for explicit `allow-once` or `allow-always` decisions, keep enforced execution for allowlist timeout fallback, and cover both branches with focused gateway approval tests.

## Reproduction Assessment

Yes, source-reproducible. A focused unit test can mock an allowlist-satisfied heredoc command, a timed-out approval decision, and `askFallback: "allowlist"`; this patch selects raw `params.command` while current main keeps `enforcedCommand`.

## Solution Assessment

No. The selector should key off an explicit `allow-once` or `allow-always` decision, not `requiresHeredocApproval`, so fallback approvals keep the enforced allowlist execution boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep enforced command for timeout fallback approvals:** `src/agents/bash-tools.exec-host-gateway.ts:285`
