## Summary

Keep open, but do not merge as written. Current main still has the heredoc approval execution problem this PR targets, while the PR broadens raw command execution into the timeout fallback path for allowlist approvals, which is a security-sensitive regression.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute the original `params.command` instead of the rebuilt allowlist `enforcedCommand`.

## Best Possible Solution

Land a narrow gateway approval fix that preserves raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps the rebuilt enforced command for allowlist timeout fallback approvals, and adds focused regression tests for both heredoc branches plus the non-heredoc path.

## Reproduction Assessment

Yes. A focused unit test can mock an allowlist-satisfied heredoc command, a timed-out approval decision, and `askFallback: "allowlist"`; the PR head would select raw `params.command` where current main keeps `enforcedCommand`.

## Solution Assessment

No. The proposed branch uses `requiresHeredocApproval` as the execution selector, but the safer selector is whether the approval decision was explicit rather than a timeout fallback.
