## Summary

Keep open: the heredoc approval bug is source-reproducible and current main has not absorbed the PR behavior, but the proposed patch broadens raw command execution to allowlist timeout fallback approvals and lacks required after-fix real behavior proof.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute the original command text instead of the rebuilt allowlist enforced command.

## Best Possible Solution

Preserve raw heredoc command text only after explicit `allow-once` or `allow-always`, keep allowlist timeout fallback on `enforcedCommand`, and add focused regression tests for explicit heredoc approval, fallback approval, and non-heredoc approval.

## Reproduction Assessment

Yes, source-reproducible: current main detects allowlist-satisfied heredoc segments and requires approval, then still passes the rebuilt `enforcedCommand` to execution. I did not run a live gateway approval flow in this read-only review.

## Solution Assessment

No. The proposed patch keys raw execution only on `requiresHeredocApproval`, but the same path is used for allowlist timeout fallback; the safer fix is to gate raw execution on explicit allow decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep enforced command for timeout fallback approvals:** `src/agents/bash-tools.exec-host-gateway.ts:288`
  - body: This uses `params.command` whenever `requiresHeredocApproval` is true, but timeout resolution with `askFallback === "allowlist"` reaches the same follow-up after setting `approvedByAsk = true`. That path has no explicit human allow decision, so it should keep `enforcedCommand` and reserve raw heredoc text for `allow-once`/`allow-always`.
  - confidence: 0.91
