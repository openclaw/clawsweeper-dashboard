## Summary

Keep open: current main still has the heredoc approval execution bug, but this PR changes a security-sensitive exec boundary too broadly and lacks after-fix real behavior proof.

## What This Changes

This PR changes gateway exec approval follow-up so heredoc-required approvals execute the original command text instead of the rebuilt enforced allowlist command.

## Best Possible Solution

Use raw heredoc command text only after explicit `allow-once` or `allow-always`, keep timeout fallback on `enforcedCommand`, and add focused tests plus redacted real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main requires approval for allowlist-satisfied heredocs, rebuilds an enforced command from argv, and then executes that enforced command after approval, which loses heredoc body text; I did not run a live gateway flow in this read-only review.

## Solution Assessment

No. The proposed fix should distinguish explicit `allow-once` or `allow-always` from timeout fallback; otherwise it trades the heredoc preservation bug for a security-boundary regression.
