## Summary

Keep open: the reported bug is source-reproducible and the direction is useful, but this PR still changes a host-exec approval boundary unsafely by applying raw heredoc execution to timeout fallback approvals, and it lacks after-fix real behavior proof.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute the original command text instead of the rebuilt enforced allowlist command.

## Best Possible Solution

Preserve raw heredoc text only after explicit `allow-once` or `allow-always`, keep `enforcedCommand` for allowlist timeout fallback, and add focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. Current main rebuilds allowlist-satisfied heredoc commands into `enforcedCommand` and uses that after approval, while analyzer tests show heredoc bodies are not part of the planned argv that the enforced command renders.

## Solution Assessment

No. The PR keys raw execution only on `requiresHeredocApproval`, but timeout fallback approvals share that follow-up path; the narrower solution is to use raw command text only for explicit allow decisions.
