## Summary

Keep open: the PR targets a real gateway heredoc approval bug, but the diff would run raw heredoc shell text for allowlist timeout fallback where no explicit human approval occurred, and it still lacks real behavior proof.

## What This Changes

The PR changes the async gateway exec approval follow-up so heredoc-required approvals pass the original command text to `runExecProcess` instead of the rebuilt enforced allowlist command.

## Best Possible Solution

Use raw heredoc command text only after explicit `allow-once` or `allow-always`, keep `askFallback: "allowlist"` timeout execution on `enforcedCommand`, and add focused tests plus redacted real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main requires approval for allowlist-satisfied heredocs, rebuilds an enforced command from argv, and then executes that enforced command after approval, which loses heredoc body text; I did not run a live gateway flow in this read-only review.

## Solution Assessment

No. The proposed fix should distinguish explicit `allow-once` or `allow-always` from timeout fallback; otherwise it trades the heredoc preservation bug for a security-boundary regression.
