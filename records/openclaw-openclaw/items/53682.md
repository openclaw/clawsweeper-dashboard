## Summary

Keep open. Current main still has the heredoc approval problem, but this branch is not merge-ready because it would run raw `params.command` for every heredoc approval follow-up, including the timeout `askFallback: "allowlist"` path where no explicit human allow decision occurred. The PR also lacks inspectable after-fix real behavior proof.

## What This Changes

The PR changes `src/agents/bash-tools.exec-host-gateway.ts` so heredoc-required gateway approval follow-ups execute the original command text instead of the rebuilt enforced allowlist command.

## Best Possible Solution

Land a narrow gateway approval fix that uses raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps timeout fallback on `enforcedCommand`, and adds focused approval tests plus redacted real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main requires approval for allowlist-satisfied heredocs, rebuilds an enforced command from argv, and then executes that enforced command after approval, which loses heredoc body text.

## Solution Assessment

No. The proposed fix should distinguish explicit `allow-once` or `allow-always` from timeout fallback; otherwise it trades the heredoc preservation bug for a security-boundary regression.
