## Summary

Current main still has a source-level gap that matches the report: QQ Bot group inbound handling constructs the correct QQ group target, but the prepared turn path does not persist that route into session delivery metadata, so follow-up delivery can fall back away from QQ. This should stay open for a focused channel delivery fix.

## What This Changes

The issue reports that QQ Bot group messages are received and processed, but replies and `sessions_send` announce delivery route to webchat instead of the originating QQ group.

## Best Possible Solution

Persist QQ Bot inbound group delivery metadata during turn/session recording and cover the QQ group announce path with focused regression tests, without changing generic webchat fallback semantics.

## Reproduction Assessment

No live QQ reproduction was run. Source inspection shows the QQ Bot group path builds `qqbot:group:<openid>` but does not persist that route through `updateLastRoute`, which matches the reported webchat fallback.

## Solution Assessment

Yes, the best fix is a narrow QQ Bot route-persistence repair with regression coverage. A core fallback change would be broader and more likely to mask channel-owned metadata bugs.
