## Summary

Keep open: this is a draft, broad mixed-surface PR, and current `main` does not fully implement its intended Discord/outbound delivery work. The proposed patch also still has concrete regressions in CLI banner config handling and gateway readiness proxy handling, so it is not merge-ready as-is.

## What This Changes

The branch combines session/model override recovery, Discord watchdog and disconnect hardening, outbound delivery fallback logic, agent directive defaults, gateway and macOS timing changes, README/package discoverability copy, and root/legal/launch documentation.

## Best Possible Solution

Split this into narrower maintainer-reviewable changes: keep any still-needed runtime hardening that fits current main's session and delivery APIs, preserve documented CLI and gateway proxy behavior, and handle legal/product launch documents as a separate owner-reviewed docs/legal change.

## Reproduction Assessment

Yes for the review findings: comparing the PR patch against current `main` shows the banner config regression and readiness proxy-settings regression without needing a live runtime. Full behavioral validation would require a rebased branch and targeted tests.

## Solution Assessment

No, this PR is not the best way to solve the underlying issues as submitted. A smaller set of rebased PRs would be safer because the branch mixes runtime fixes with product/legal docs and introduces at least one user-facing regression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Restore config-backed banner tagline mode:** `src/cli/banner-config-lite.ts:13`
  - body: The patch changes `readCliBannerTaglineMode` to read only `OPENCLAW_CLI_BANNER_TAGLINE_MODE`. The CLI schema and docs expose `cli.banner.taglineMode`, so users who set the banner mode in config would lose that behavior and still see the default/random banner. Keep the config read path or intentionally migrate the schema/docs/tests with the behavior change.
  - confidence: 0.87
- **[P3] Pass proxy trust settings into readiness probes:** `src/gateway/server-http.ts:786-788`
  - body: The patch handles gateway probes before loading config and passes empty trusted proxy settings into `handleGatewayProbeRequest`. That makes `/ready` ignore configured `gateway.trustedProxies` / `allowRealIpFallback` when deciding whether detailed readiness is safe to reveal behind a trusted reverse proxy. Keep readiness probes on the config-backed path or load those settings before calling the probe handler.
  - confidence: 0.74
