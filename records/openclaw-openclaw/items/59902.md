## Summary

Keep open. Current main and the latest release still make the `/cron` History button update a lower in-page Run history section without scrolling, focusing, or opening a dedicated surface, so the reported UX gap remains valid.

## What This Changes

The issue asks for clearer feedback when the `/cron` History button updates a lower run-history panel, such as scrolling/focusing that section or opening a dedicated history surface.

## Best Possible Solution

Keep the existing Run history card, but move attention to it with accessible focus and/or scroll behavior after an explicit History-button click, with focused UI coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: render `/cron` with enough scheduled jobs for Run history to sit below the fold, click a job History button, and current main only loads runs without moving viewport or focus.

## Solution Assessment

Yes. The scroll/focus variant is the narrowest maintainable solution for the reported confusion; a modal or drawer would be a larger UI redesign.
