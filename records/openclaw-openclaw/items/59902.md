## Summary

Keep open. Current main still makes the /cron History action an in-page run-history reload with no scroll, focus, modal, drawer, or visible attention shift, so the reporter’s confusion remains valid and localized enough for a focused UI repair.

## What This Changes

The issue requests clearer feedback when the /cron History button updates the lower run-history panel, such as scrolling/focusing that section or opening a dedicated history panel.

## Best Possible Solution

Keep the existing cron backend and Run history card, but make the explicit History button move attention to that card after selecting/loading a job, preferably with a stable section target plus scroll and focus behavior covered by a focused UI test. A modal or drawer should be reserved for a broader Control UI redesign.

## Reproduction Assessment

Yes. Render the /cron page with the Run history card below the fold, click a job’s History button, and current main only switches run scope and loads runs while leaving viewport and focus unchanged.

## Solution Assessment

Yes. The reporter’s scroll/focus option is the narrowest maintainable solution because the run-history surface and data flow already exist; adding a modal or drawer would expand the UI surface without being necessary for this confusion.
