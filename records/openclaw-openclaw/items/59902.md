## Summary

Keep open. Current main still makes the `/cron` History action reload the lower in-page Run history card without any scroll, focus, modal, drawer, or other attention shift, so the reported confusion remains valid and narrowly repairable.

## What This Changes

The issue requests clearer visible feedback when the `/cron` History button updates the lower run-history panel.

## Best Possible Solution

Keep the existing cron backend and Run history card, but make the explicit History button move attention to that card with a stable target plus scroll/focus behavior and focused UI coverage.

## Reproduction Assessment

Yes. Render the `/cron` page with the Run history card below the fold, click a job’s History button, and current main only switches the run scope and loads runs while leaving viewport and focus unchanged.

## Solution Assessment

Yes. The scroll/focus option is the narrowest maintainable fix because the run-history surface and data flow already exist; a modal or drawer would be broader UI redesign work.
