## Summary

Keep open. Current main still lets ANSI-bearing `logs.tail` text flow into the dedicated Control UI Logs tab without shared display normalization, while the Overview tail only strips ANSI locally; the duplicate report was closed in favor of this issue and the related fix PR was closed unmerged.

## What This Changes

The issue asks for the Control UI log panel to stop displaying raw ANSI escape codes and render or normalize styled gateway log text for readable browser display.

## Best Possible Solution

Land a focused Control UI logging fix that shares one ANSI display strategy across all `logs.tail` UI consumers, normalizes displayed message/subsystem/fallback text, preserves raw export semantics deliberately, and adds ANSI-bearing regression coverage.

## Reproduction Assessment

Yes, at source level: ANSI-styled gateway log text can flow from `readConfiguredLogTail()` through `parseLogLine()` into `renderLogs()` without display normalization. I did not run a live browser UI in this read-only review.

## Solution Assessment

Yes. UI-side display normalization is the narrowest maintainable fix because it improves browser readability without changing the `logs.tail` server contract or raw export behavior.
