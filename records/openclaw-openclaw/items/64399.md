## Summary

Keep open. Current main and latest release still preserve ANSI-bearing `logs.tail` text in the dedicated Control UI Logs tab and render it directly; only the Overview tail has a local display-only strip helper, and the focused prior fix PR was closed unmerged.

## What This Changes

The issue requests that the Control UI gateway log panel stop showing raw ANSI escape sequences and instead render or normalize them for browser display.

## Best Possible Solution

Add one shared Control UI ANSI display helper used by every `logs.tail` UI consumer, normalize displayed message/subsystem/fallback text, keep raw tail/export data deliberate, and add ANSI-bearing structured and plain-line regression coverage.

## Reproduction Assessment

Yes, at source level: feed an ANSI-bearing structured or plain line through `parseLogLine()` and render it through the dedicated Logs tab; current main preserves and displays the escape-coded text. I did not run a live browser UI in this read-only review.

## Solution Assessment

Yes. UI-side display normalization is the narrowest maintainable fix because it addresses browser readability without changing the server `logs.tail` contract, CLI fallback behavior, or raw log/export semantics.
