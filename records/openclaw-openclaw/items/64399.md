## Summary

Keep open. Current main still preserves ANSI-bearing `logs.tail` text in the dedicated Control UI Logs tab and renders it directly; only the Overview tail has a local strip helper. The prior direct fix PR is closed unmerged, so the bug remains a narrow, actionable UI display issue.

## What This Changes

The issue requests that the Control UI log panel stop displaying raw ANSI escape sequences and render or normalize styled log text for readability.

## Best Possible Solution

Add one shared Control UI ANSI display helper and apply it consistently to parsed `logs.tail` message, subsystem, and fallback display text. Keep raw log/export data unchanged, remove or reuse the duplicated Overview-only helper, and add regression coverage for ANSI-bearing structured and plain lines.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: feed an ANSI-bearing JSONL line into `parseLogLine()`, then render the resulting entry through `renderLogs()`; current main preserves the escape sequences in `message` and displays that value directly.

## Solution Assessment

Yes. The narrow maintainable fix is UI-side display normalization shared by all Control UI log consumers; changing server-side `logs.tail` would risk altering raw export and CLI behavior.
