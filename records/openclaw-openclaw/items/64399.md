## Summary

Keep open. Current main and latest `v2026.5.7` still preserve ANSI-bearing `logs.tail` text in the dedicated Control UI Logs tab and render it directly; the prior focused fix at https://github.com/openclaw/openclaw/pull/64431 was closed unmerged, and https://github.com/openclaw/openclaw/issues/45001 was already closed as a duplicate of this remaining bug.

## What This Changes

The issue requests that the Control UI gateway log panel stop displaying raw ANSI escape sequences and instead render or normalize them for readable browser display.

## Best Possible Solution

Land a focused Control UI logging fix that shares one ANSI display strategy across every `logs.tail` UI consumer, normalizes displayed message/subsystem/fallback text, keeps raw export behavior deliberate, and adds ANSI-bearing regression coverage.

## Reproduction Assessment

Yes, at source level: feed an ANSI-bearing structured or plain line through `parseLogLine()` and the dedicated Logs tab will render the preserved `message` or `raw` text directly. I did not run a live browser UI in this read-only review.

## Solution Assessment

Yes. UI-side display normalization is the narrowest maintainable fix because it improves browser readability without changing the server `logs.tail` contract, CLI behavior, or raw export semantics.
