## Summary

Keep open: the identity-threading shape is directionally right, but the PR introduces an unsupported Slack `icon_emoji` path for raw Unicode and still lacks real Slack behavior proof, so it is not merge-ready.

## What This Changes

The PR forwards outbound identity through the legacy runtime send factory and heartbeat sends, changes Slack outbound emoji mapping, and adds focused regression tests plus a changelog entry.

## Best Possible Solution

Keep the identity propagation fix, but make Slack emoji handling conform to Slack’s documented `icon_emoji` contract before requiring a redacted live Slack proof of delivery and rendering.

## Reproduction Assessment

Yes at source level: current main drops identity in the legacy runtime sender and heartbeat send calls, while Slack can only apply persona metadata when `context.identity` reaches it. I did not establish a live Slack failing run in this read-only review.

## Solution Assessment

No, not as currently written: threading `OutboundIdentity` is the right boundary, but forwarding raw Unicode directly to Slack `icon_emoji` conflicts with the documented Slack API shape and needs to be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not send raw Unicode as Slack icon_emoji:** `extensions/slack/src/outbound-adapter.ts:72`
  - body: This removes the shortcode guard and sends values like `📟` directly as `icon_emoji`. Slack documents `icon_emoji` in colon-shortcode form, and `postSlackMessageBestEffort` only retries without identity for `missing_scope`; an invalid emoji payload would be rethrown and can fail heartbeat/runtimeSend delivery for existing agents that currently configure raw emoji. Keep shortcode-only behavior or add a real Slack-backed Unicode-to-shortcode normalization path with fallback coverage.
  - confidence: 0.87
