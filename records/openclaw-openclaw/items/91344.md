## Summary

Keep open: the branch fixes the `/reset soft` args forwarding path, but its reset detector change would let non-soft hard-reset tails bypass destructive-reset confirmation, and the external PR still lacks real Control UI plus Gateway proof.

## What This Changes

The PR updates Control UI reset slash-command dispatch to pass `/reset` args through, bypasses confirmation for `/reset soft`, and adds app-chat regression tests.

## Best Possible Solution

Keep the args-preserving dispatch, but make the no-confirm exception match only exact `/reset soft` or `/reset soft <tail>` and keep all other `/reset` tails classified as destructive before landing with redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: the proposed detector treats any `/reset soft*` prefix as non-destructive while the gateway only accepts `soft` as a bounded token, and `/reset <non-soft tail>` still falls through to hard reset handling. I did not run tests because this review is read-only.

## Solution Assessment

No. Passing reset args through is the right fix, but the detector should share the gateway soft-reset boundary and retain hard-reset classification for every other `/reset` tail.
