## Summary

Keep open. Current main still does not provide the requested per-run introspection: scheduled deep dreaming reports contain aggregate counts, verbose scoring remains logger-only, rejected candidates are filtered before callers can report rejection reasons, and the memory CLI has single-candidate `promote-explain` plus status audit but no aggregate dreaming doctor diagnostic.

## What This Changes

The issue requests memory-core dreaming diagnostics that expose ranked/rejected candidate details in deep reports, write structured verbose scoring artifacts, and add a dreaming doctor command for aggregate gate reachability.

## Best Possible Solution

Add one shared deep-promotion evaluation surface in memory-core that returns bounded considered candidates with score components, thresholds, pass/fail state, promotion/apply status, and rejection reasons, then reuse it for scheduled deep reports, optional structured verbose artifacts, and an aggregate CLI doctor diagnostic without changing promotion policy or defaults.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an additive diagnostics request. Current-main source inspection is a high-confidence verification path because the report, logging, ranking, and CLI registration paths are all visible and lack the requested artifacts/command.

## Solution Assessment

Yes, the requested direction is best solved by sharing the actual promotion evaluation logic across report, JSONL, and doctor outputs. Duplicating scoring or gate logic in separate callers would risk drift from the real promotion behavior.
