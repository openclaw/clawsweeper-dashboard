## Summary

Keep open. Current main has adjacent memory dreaming diagnostics, but it still lacks the requested user-visible introspection: deep reports are aggregate-only, verbose scoring stays in logger output, rejected candidates do not carry gate reasons, and the memory CLI has single-candidate `promote-explain` rather than an aggregate dreaming doctor.

## What This Changes

The issue requests richer memory deep-dreaming introspection: per-candidate report details, file-backed verbose scoring, and an aggregate dreaming doctor command.

## Best Possible Solution

Add a shared deep-promotion evaluation surface that returns bounded considered candidates with score components, thresholds, pass/fail state, promotion status, and rejection reasons, then reuse it for the markdown report, verbose JSONL, and a CLI diagnostic without changing promotion policy.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an additive diagnostics request. Current-main source inspection is a high-confidence verification path because the report, logging, ranking, and CLI registration paths are visible and lack the requested surfaces.

## Solution Assessment

Yes, the requested direction is best solved by sharing the actual promotion evaluation logic across report, JSONL, and doctor outputs. Duplicating scoring or gate logic in separate callers would risk drift from real promotion behavior.

## Repair Work Prompt

Implement issue #70072 as a focused memory-core diagnostics PR. Add a shared deep-promotion evaluation/explain helper that can inspect short-term recall entries and return considered candidates, score components, threshold values, pass/fail booleans, promotion state, and concise rejection reasons without changing scoring weights, thresholds, ranking order, or promotion policy. Use that helper to enrich `memory/dreaming/deep/YYYY-MM-DD.md` with a bounded ranked-candidates section and thresholds, write structured verbose scoring output to a workspace artifact such as `memory/.dreams/deep/YYYY-MM-DD-verbose.jsonl` when `dreaming.verboseLogging` is enabled, and add an aggregate CLI diagnostic command for the `openclaw memory dreaming doctor` use case. Update CLI types/runtime/registration, focused tests, docs, and CHANGELOG for the user-facing diagnostic surface. Do not move memory-core behavior into core except through existing plugin/SDK seams, and do not change the default promotion gates or the fix already made for #68882.
