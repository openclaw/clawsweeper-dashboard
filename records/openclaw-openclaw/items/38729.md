## Summary

Keep open: the PR's legacy `supportsXHighThinking` hook is still bypassed by Anthropic's non-empty `resolveThinkingProfile`, so it would not expose `xhigh`/`max` for Claude Opus 4.6. Current docs and tests encode Claude 4.6 as adaptive-only, while the related #62950 directive-preservation PR is separate input-routing work.

## What This Changes

The PR adds an Anthropic provider `supportsXHighThinking` callback for Claude Opus 4.6 model IDs in `extensions/anthropic/register.runtime.ts`.

## Best Possible Solution

Keep Claude thinking support defined through `resolveClaudeThinkingProfile`; if maintainers approve Opus 4.6 max effort, update that profile plus Anthropic tests and thinking docs, otherwise revise or close this PR as adaptive-only.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: resolve Anthropic thinking levels for `claude-opus-4-6` on current main and observe the non-empty Anthropic provider profile returns before the legacy `supportsXHighThinking` hook can add `xhigh`.

## Solution Assessment

No. Adding `supportsXHighThinking` is not the best fix because Anthropic already uses the profile-driven path; the maintainable fix, if approved, is to update the shared Claude profile source of truth and matching tests/docs.
