## Summary

Keep open. Current main's profile-driven thinking path makes this patch ineffective: Anthropic already returns a non-empty `resolveThinkingProfile`, which bypasses the legacy `supportsXHighThinking` hook, and current tests/docs advertise Claude Opus 4.6 as adaptive-only rather than `xhigh`/`max`.

## What This Changes

The branch adds an Anthropic `supportsXHighThinking` callback in `extensions/anthropic/register.runtime.ts` to advertise Claude Opus 4.6 xhigh/max thinking.

## Best Possible Solution

Keep Claude thinking support defined through `resolveClaudeThinkingProfile`; if maintainers approve Opus 4.6 max effort as user-facing, update that profile with matching Anthropic tests/docs and require real behavior proof.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection on current main shows Anthropic's non-empty provider profile returns before the legacy `supportsXHighThinking` hook is consulted. The requested Opus 4.6 `xhigh` exposure itself is a feature/capability decision, not a strict bug reproduction.

## Solution Assessment

No. Adding a legacy compatibility hook is not the maintainable path on current main; if approved, the implementation belongs in `resolveClaudeThinkingProfile` with matching tests and docs.
