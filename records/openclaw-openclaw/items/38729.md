## Summary

Keep open: the PR is a focused Anthropic thinking-capability change, but the current diff is not correct on current main because Anthropic thinking support now comes from `resolveThinkingProfile`; adding the deprecated `supportsXHighThinking` hook is bypassed before it can expose `xhigh`. Current docs and tests still keep Claude Opus 4.6 adaptive-only, so advertising `xhigh`/`max` also needs a maintainer capability decision and real behavior proof.

## What This Changes

The PR adds an Anthropic provider `supportsXHighThinking` callback for Claude Opus 4.6 model IDs.

## Best Possible Solution

Keep Claude thinking support defined through `resolveClaudeThinkingProfile`; if maintainers approve Opus 4.6 max effort as user-facing, update that profile with focused Anthropic tests/docs and runtime proof, otherwise close this PR as adaptive-only.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows current main returns Anthropic's non-empty provider profile before consulting the legacy xhigh hook. The broader Opus 4.6 `xhigh` request is a provider-capability feature decision rather than a strict bug reproduction.

## Solution Assessment

No. Adding a deprecated compatibility hook is not the maintainable path on current main; the approved implementation, if any, belongs in `resolveClaudeThinkingProfile` with matching tests/docs and proof.
