## Summary

Current main still lacks a review-queue or off mode for Dreams Deep promotion; source and docs confirm Deep promotion is designed to write `MEMORY.md`. Keep this open because it is a valid feature/config-policy request that needs maintainer design alignment, not a cleanup close.

## What This Changes

Request a Dreams Deep promotion review-queue/off mode plus related source gating, phase scheduling, and write-surface documentation so Dreams can run without autonomous `MEMORY.md` edits.

## Best Possible Solution

Add a documented, schema-backed Dreams promotion policy that stages proposals without editing `MEMORY.md`, then align phase source gates, scheduling behavior, docs, and UI/status around that contract.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main has no review-queue mode, and the Deep promotion path still calls the writer that appends to `MEMORY.md`. The related Light `sources` concern is also visible because daily and session ingestion run unconditionally.

## Solution Assessment

No, current main is not the best end state for policy-conscious deployments because it only offers all-or-nothing dreaming enablement around automatic Deep promotion. The safer path is a maintainer-approved promotion policy contract before implementation.
