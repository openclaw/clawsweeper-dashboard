## Summary

Current main still lacks a Dreams review-queue/off promotion policy, and both source and docs show Deep promotion is intentionally wired to append durable entries to `MEMORY.md`. This should stay open as a feature/config-policy request that needs maintainer design alignment rather than cleanup closure.

## What This Changes

Request a Dreams review-queue/off Deep promotion mode, source gating, independent scheduling, and write-surface docs so Dreams can run without autonomous `MEMORY.md` writes.

## Best Possible Solution

Add a documented, schema-backed Dreams promotion policy that can stage Deep proposals without editing `MEMORY.md`, then align source gates, scheduling semantics, status/UI copy, and docs around that contract.

## Reproduction Assessment

Yes, from source inspection on current main: the Deep-triggered sweep still calls the promotion writer that appends to durable memory, and no review-queue/off promotion mode exists in the resolver or manifest schema. I did not run a live cron because the read-only code path is direct enough to establish the current behavior.

## Solution Assessment

Unclear until maintainers choose the public config contract. The proposed direction is reasonable, but the maintainable fix should first settle defaults, migration behavior, and whether Light/REM scheduling should remain one sweep or become independently managed.
