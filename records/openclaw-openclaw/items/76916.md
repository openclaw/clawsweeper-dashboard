## Summary

Keep open: current main still lacks a first-class Dreams review-queue/off promotion policy. Some docs now explicitly describe the current write surface, but the managed Deep path still appends promotions to `MEMORY.md`, so the central request remains valid and needs maintainer design alignment.

## What This Changes

The issue asks for a Dreams Deep promotion review-queue/off mode so analysis can run without autonomously editing curated memory files like `MEMORY.md`.

## Best Possible Solution

Add a documented, schema-backed Dreams promotion policy that can stage proposed durable-memory changes without directly editing `MEMORY.md`, with source gates, scheduling, tests, status/UI, and docs aligned around the chosen contract.

## Reproduction Assessment

Yes. Source inspection on current main shows no review-queue/off promotion policy in the SDK/schema/docs, the managed cron config is still derived from Deep enablement, and the Deep path calls the `MEMORY.md` promotion writer.

## Solution Assessment

Unclear until maintainers choose the public contract. A review-queue/off policy is a plausible direction, but the exact schema, defaults, phase scheduling semantics, UI/status behavior, and docs need design approval before implementation.
