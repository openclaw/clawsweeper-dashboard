## Summary

Keep open: current main still lacks a first-class Dreams promotion policy that can stage Deep proposals instead of writing durable memory, while source and docs confirm Deep remains the phase that appends to `MEMORY.md`; this needs maintainer config/product design rather than cleanup closure.

## What This Changes

Request a Dreams Deep promotion mode that stages proposals for review instead of autonomously writing durable promotions to `MEMORY.md`.

## Best Possible Solution

Add a documented, schema-backed Dreams promotion policy that can stage Deep proposals without editing `MEMORY.md`, then align scheduling, source gates, CLI/UI status, tests, and docs around that contract.

## Reproduction Assessment

Yes. Source inspection on current main is enough: there is no review-queue/off promotion mode in the SDK type or manifest schema, and the managed Deep path still calls the writer that appends to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers choose the public contract. A review-queue/off policy is a plausible direction, but implementing an ad hoc proposal file without schema, defaults, UI/status, and docs alignment would be incomplete.
