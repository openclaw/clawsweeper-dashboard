## Summary

Keep open. Current main only partially addresses the LaunchAgent downtime report: restart now prefers `kickstart -k` with re-bootstrap fallback, but generated plists still use a one-second throttle, omit `ExitTimeOut`, and package/update service flows can still route through `stopLaunchAgent` paths that fall back to `bootout` and leave the service unloaded.

## What This Changes

The issue asks OpenClaw to harden macOS LaunchAgent auto-update and service lifecycle behavior by changing throttle/shutdown policy and avoiding non-uninstall `bootout` paths that can leave the gateway down.

## Best Possible Solution

Land a focused macOS LaunchAgent lifecycle fix that codifies the chosen launchd restart/backoff policy, adds an explicit shutdown timeout to the generated plist and tests, and makes non-uninstall bootout paths unnecessary or reliably recoverable across install, update, stop, restart, and failed-update recovery.

## Reproduction Assessment

Yes. The issue gives a concrete macOS LaunchAgent failure path, and current source still supports the unresolved mechanism: generated plists contain `ThrottleInterval=1` with no `ExitTimeOut`, and package updates still call the macOS stop path that can fall back to `bootout`.

## Solution Assessment

No. Current main is only a partial repair, and the adjacent `SuccessfulExit=false` PR is not the best complete solution as-is because the restart/update paths still rely on clean relaunch behavior and the issue also needs plist timeout and update/bootout lifecycle coverage.
