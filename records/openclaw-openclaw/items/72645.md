## Summary

Keep open: the feature direction is plausible, but this PR still puts high-cardinality and sensitive correlation identifiers into OTel metric labels, has a smaller skill-scope labeling bug, and lacks live OTLP/Jaeger/Prometheus proof for the new exported behavior.

## What This Changes

The PR adds message-processing and skill-execution diagnostic events, exports corresponding diagnostics-otel spans and metrics, adds a skill observability doc, and includes several test/script/package stabilizations.

## Best Possible Solution

Land a narrower telemetry patch that keeps correlation IDs on spans/logs only, keeps metrics low-cardinality, fixes effective skill-scope labeling, and includes redacted live collector proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows identifier-heavy attrs flowing directly into OTel metric add/record calls and skillScope using the raw snapshot instead of the effective snapshot. For the feature itself, this is not a current-main bug reproduction path; the missing proof is a live OTLP/Jaeger/Prometheus run.

## Solution Assessment

No. First-class diagnostics may be the right direction, but this implementation should separate trace correlation attrs from metric labels, fix the sandbox skill-scope label, and prove the exported behavior in a real collector before merge.

Full review comments:

- **[P1] Keep run and session IDs off message.processed metrics:** `extensions/diagnostics-otel/src/service.ts:1410-1412`
  - body: The new run_id/session fields are added to the attrs object that is passed to messageProcessedCounter.add and messageDurationHistogram.record. That turns an existing low-cardinality metric into per-run/session time series; keep these IDs on spans/logs rather than metric labels.
  - confidence: 0.9
- **[P1] Split message correlation IDs from metric labels:** `extensions/diagnostics-otel/src/service.ts:1436-1441`
  - body: messageProcessingBaseAttrs includes run_id, messageId, chatId, session_key, and session_id, then the same attrs are recorded on counters and histograms. Each message/chat/session becomes a new metric series and may expose identifiers to metric backends; build separate low-cardinality metric attrs and keep these IDs trace-only.
  - confidence: 0.92
- **[P1] Split run and session IDs from skill metric labels:** `extensions/diagnostics-otel/src/service.ts:2321-2323`
  - body: skillExecutionBaseAttrs includes run_id, session_key, and session_id before the attrs are used for skill counters and histograms. Skill metrics should be bounded labels such as phase, outcome, and scope, while correlation IDs stay on spans.
  - confidence: 0.9
- **[P3] Base skill scope on the effective snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:965-967`
  - body: skillsSnapshotForRun is cleared when a sandboxed run cannot use the snapshot, but skillScope still checks params.skillsSnapshot. Those runs will be labeled as snapshot even though skill resolution uses the workspace path; derive the label from skillsSnapshotForRun.
  - confidence: 0.78
