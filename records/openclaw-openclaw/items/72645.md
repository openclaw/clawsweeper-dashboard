Keep open. The PR adds a plausible observability feature, but current PR-head source still exports raw run/session/message identifiers into OTel spans and metric labels, and the supplied proof does not show the actual OTLP/Jaeger/Prometheus behavior it changes.

The PR adds `message.processing.*` and `skill.execution.*` diagnostics events from embedded runs, exports them through `diagnostics-otel` spans/metrics, and adds related docs/tests plus adjacent CI/script/dependency changes.

## Best Possible Solution

Land a narrowed telemetry contract with privacy-preserving spans, bounded metric labels, canonical docs coverage, focused scope, and redacted live OTel export proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows raw identifier attrs on spans/metrics. It is not a current-main bug reproduction because this PR adds a new observability feature.

## Solution Assessment

No. The maintainable path is to keep external telemetry privacy-preserving and low-cardinality, then prove the new OTel spans/metrics against a real collector before merge.
