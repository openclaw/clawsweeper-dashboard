## Summary

Keep open: the latest PR head still violates the current OTel privacy/cardinality contract, has a TypeScript assignability blocker, lacks live collector proof for the new exporter behavior, and still includes unrelated dependency, macOS, script, and test-runner changes.

## What This Changes

The PR adds `message.processing.*` and `skill.execution.*` diagnostics/OTel events with docs and tests, while also changing qa-lab dependency metadata, macOS formatting, and test tooling.

## Best Possible Solution

Land a narrower telemetry design that keeps metric labels bounded, keeps raw identifiers out of default export unless explicitly approved, adds exporter coverage/docs in the canonical OTel catalog, and includes redacted live collector proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows raw identifiers flowing into span and metric attributes, plus a still-present TypeScript assignability error. The feature itself still lacks a live collector proof path.

## Solution Assessment

No. First-class message and skill telemetry may be useful, but this implementation should separate trace correlation from metric labels and stay within the existing OTel privacy contract unless maintainers approve a new policy.

  - confidence: 0.92

  - body: Adding run and session IDs to `openclaw.message.processed` and `openclaw.message.duration_ms` turns an existing bounded metric into per-run/session series and conflicts with the documented channel/outcome label shape. Leave these metrics low-cardinality and keep correlation out of metric attrs.

  - body: `messageProcessingBaseAttrs` includes run, message, chat, session key, and session id, and the same attrs are used for counters and histograms. That creates per-message/chat/session metric series and exports identifiers to metric backends; build separate low-cardinality metric attrs.
  - confidence: 0.94

  - body: `skillExecutionBaseAttrs` includes run and session identifiers before the attrs are used for skill counters and histograms. Skill metrics should stay bounded labels such as phase, outcome, and scope while any approved correlation data stays off metric labels.
  - confidence: 0.92
