## Summary

Keep open: the latest PR head still changes the OTel privacy and metric-cardinality contract, has a likely TypeScript build blocker, and lacks live proof that the new spans/metrics export correctly.

## What This Changes

The PR adds `message.processing.*` and `skill.execution.*` diagnostics events, OTel spans/metrics/docs, and also touches macOS formatting, qa-lab dependencies, and test tooling.

## Best Possible Solution

Land a narrower telemetry design that preserves bounded metric labels, keeps raw identifiers out of default OTel export unless explicitly approved, adds exporter tests/docs, and includes redacted live collector proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows raw identifier attrs flowing into span and metric attributes. The feature itself still lacks live collector proof showing the exported spans and metrics.

## Solution Assessment

No. First-class message and skill telemetry may be useful, but this implementation should separate correlation from metric labels and stay within the existing OTel privacy contract unless maintainers approve a new policy.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Widen the skill attrs type before spreading it:** `extensions/diagnostics-otel/src/service.ts:2354-2355`
  - body: `skillExecutionBaseAttrs` returns `Record<string, string | number | boolean>`, but the completed and error handlers assign its spread to `Record<string, string | number>`. That index-signature mismatch is still present and should fail TypeScript; narrow the base return type or widen both locals.
  - confidence: 0.9
- **[P1] Keep raw run and session IDs out of spans:** `extensions/diagnostics-otel/src/service.ts:1205-1211`
  - body: These new snake_case keys are not in `DROPPED_OTEL_ATTRIBUTE_KEYS`, so default run, harness, tool, and related spans can export raw run/session identifiers despite the current privacy contract saying session keys are not exported. Keep correlation in trace context, use bounded hashes, or gate this behind an explicit reviewed opt-in.
  - confidence: 0.92
- **[P1] Keep IDs off existing message metrics:** `extensions/diagnostics-otel/src/service.ts:1410-1412`
  - body: Adding run and session IDs to `openclaw.message.processed` and `openclaw.message.duration_ms` turns an existing bounded metric into per-run/session series and conflicts with the documented channel/outcome label shape. Leave these metrics low-cardinality and keep correlation out of metric attrs.
  - confidence: 0.93
- **[P1] Split message identifiers from message-processing metrics:** `extensions/diagnostics-otel/src/service.ts:1436-1441`
  - body: `messageProcessingBaseAttrs` includes run, message, chat, session key, and session id, and the same attrs are used for counters and histograms. That creates per-message/chat/session metric series and exports identifiers to metric backends; build separate low-cardinality metric attrs.
  - confidence: 0.94
- **[P1] Split run and session IDs from skill metrics:** `extensions/diagnostics-otel/src/service.ts:2321-2323`
  - body: `skillExecutionBaseAttrs` includes run and session identifiers before the attrs are used for skill counters and histograms. Skill metrics should stay bounded labels such as phase, outcome, and scope while any approved correlation data stays off metric labels.
  - confidence: 0.92
- **[P3] Base skill scope on the effective snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1319-1321`
  - body: Read-only sandbox runs clear `skillsSnapshotForRun`, but `skillScope` still checks `params.skillsSnapshot`. Those runs will be reported as `snapshot` even though skill resolution uses workspace mode; derive the label from `skillsSnapshotForRun`.
  - confidence: 0.78
