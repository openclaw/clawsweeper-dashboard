## Summary

Keep open. The PR is a plausible observability feature, but latest head still puts per-message/run/session identifiers into OTel metric attributes, mislabels sandboxed skill sources, and lacks live OTLP/Jaeger/Prometheus proof for the new exported behavior.

## What This Changes

The PR adds message.processing and skill.execution diagnostic events, OTel spans/metrics/docs, and related test/script/package adjustments.

## Best Possible Solution

Keep correlation identifiers trace-only, use bounded metric labels, derive skill scope from the effective snapshot, trim unrelated churn where possible, and land only with redacted live collector proof.

## Reproduction Assessment

Yes for the review findings: source inspection of latest PR head shows ID-bearing attr objects are passed directly to OTel metric add/record calls and skillScope reads raw params.skillsSnapshot. For the feature itself, live OTLP/Jaeger/Prometheus reproduction is still missing.

## Solution Assessment

No. First-class diagnostics may be the right direction, but this patch should separate trace correlation attributes from metric labels, fix the skill scope source, and prove the OTel output in a real collector before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep message metrics free of correlation IDs:** `extensions/diagnostics-otel/src/service.ts:1410-1412`
  - body: `attrs` now includes per-run/session IDs before it is passed to `messageProcessedCounter.add` and `messageDurationHistogram.record`. That creates a new metric series per run/session and exposes correlation identifiers through metrics; keep those IDs on spans/traces only and use bounded metric labels such as channel and outcome.
  - confidence: 0.91
- **[P1] Separate trace IDs from processing metric labels:** `extensions/diagnostics-otel/src/service.ts:1436-1441`
  - body: `messageProcessingBaseAttrs` collects run, message, chat, and session IDs, and that same object is used for the processing counter and duration histogram. Build a separate low-cardinality metric attr object for channel, phase, and outcome, and keep these identifiers trace-only.
  - confidence: 0.92
- **[P1] Separate run and session IDs from skill metric labels:** `extensions/diagnostics-otel/src/service.ts:2321-2323`
  - body: The skill lifecycle base attrs include `openclaw.run_id`, `openclaw.session_key`, and `openclaw.session_id`, then those attrs are recorded on skill counters and histograms. Skill metrics should use bounded labels such as phase, outcome, and scope; keep run/session correlation on spans.
  - confidence: 0.9
- **[P3] Label skill scope from the effective snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:965-967`
  - body: `skillsSnapshotForRun` is cleared when the sandbox cannot use the supplied snapshot, but `skillScope` still checks `params.skillsSnapshot`. Those runs will be reported as `snapshot` even though skill resolution uses workspace skills; base the label on `skillsSnapshotForRun`.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Latest head exports run/message/chat/session identifiers as metric attributes, creating sensitive high-cardinality telemetry dimensions.

Concerns:

- **[medium] Unbounded identifiers in OTel metric labels:** `extensions/diagnostics-otel/src/service.ts:1410`
  - body: The new and modified metric attrs include run_id, messageId, chatId, session_key, and session_id before counter/histogram add/record calls. Metric backends often store labels as series dimensions, so this can expose correlation IDs and create unbounded time-series cardinality.
  - confidence: 0.9
- **[medium] New snake_case IDs bypass the existing drop list:** `extensions/diagnostics-otel/src/service.ts:1436`
  - body: The current OTEL attribute drop list covers camelCase identifier keys, while the patch exports snake_case keys such as `openclaw.session_key` and `openclaw.run_id` through span attrs. That conflicts with the current docs promise that session keys are not exported by default.
  - confidence: 0.84
