## Summary

Keep open: the telemetry feature is plausible, but the current PR still exports raw run/session/message/chat identifiers into OTel spans and metrics, conflicts with the existing privacy/cardinality contract, and lacks live OTLP/Jaeger/Prometheus proof for the new behavior.

## What This Changes

The PR adds `message.processing.*` and `skill.execution.*` diagnostics events plus diagnostics-otel spans/metrics/docs, while also carrying macOS, gateway, qa-lab dependency, and test tooling changes.

## Best Possible Solution

Land a narrower telemetry change that keeps metric labels bounded, keeps raw identifiers out of default OTel export, adds focused exporter tests/docs, and includes redacted live collector proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows raw identifier attrs flowing into metric add/record calls and span attrs. The feature itself still lacks live collector proof showing the exported spans and metrics.

## Solution Assessment

No. First-class message and skill telemetry may be the right direction, but this implementation should separate correlation from metric labels and stay within the existing OTel privacy contract unless maintainers approve a new explicit policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep raw run and session IDs out of spans:** `extensions/diagnostics-otel/src/service.ts:1205-1211`
  - body: The current OTel privacy contract and tests strip diagnostic identifiers, but these new snake_case attrs are not in `DROPPED_OTEL_ATTRIBUTE_KEYS`. This makes run/model/tool/harness spans export raw run and session identifiers by default; keep correlation in trace context, use bounded hashes, or add an explicit reviewed opt-in.
  - confidence: 0.91
- **[P1] Keep IDs off existing message metrics:** `extensions/diagnostics-otel/src/service.ts:1410-1412`
  - body: Adding run and session IDs to `openclaw.message.processed` and `openclaw.message.duration_ms` turns an existing bounded metric into per-run/session series and violates the documented label policy. Leave these metrics at channel/outcome-level and keep correlation out of metric attrs.
  - confidence: 0.93
- **[P1] Split message IDs from message-processing metrics:** `extensions/diagnostics-otel/src/service.ts:1436-1441`
  - body: `messageProcessingBaseAttrs` includes run, message, chat, session key, and session id, then the same attrs are used for counters and histograms. That creates per-message/chat/session metric series and exports identifiers to metric backends; build separate low-cardinality metric attrs.
  - confidence: 0.94
- **[P1] Split run and session IDs from skill metrics:** `extensions/diagnostics-otel/src/service.ts:2321-2323`
  - body: `skillExecutionBaseAttrs` includes run and session identifiers before the attrs are used for skill counters and histograms. Skill metrics should stay bounded labels such as phase, outcome, and scope while any approved correlation data stays off metric labels.
  - confidence: 0.92
- **[P3] Base skill scope on the effective snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1047-1049`
  - body: Read-only sandbox runs clear `skillsSnapshotForRun`, but `skillScope` still checks `params.skillsSnapshot`. Those runs will be reported as `snapshot` even though skill resolution uses workspace mode; derive the label from `skillsSnapshotForRun`.
  - confidence: 0.78
