## Summary

Keep open: the feature is useful, but the current PR head still exports raw session/correlation identifiers into OTel attributes and metric labels, and the supplied proof does not show the changed OTLP collector path.

## What This Changes

The PR adds message-processing and skill-execution diagnostic events, emits them from embedded runs, maps them to diagnostics-otel spans/metrics, adds an OTel skill-observability doc, and adjusts qa-lab/test/cron support files.

## Best Possible Solution

Land a privacy-preserving message/skill lifecycle telemetry contract with bounded metric labels, no raw session keys, canonical OTel docs, and real collector proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new observability feature. The blocking findings are source-backed against the PR head, and the actual collector export path still lacks real proof.

## Solution Assessment

No. The safer maintainable path is to keep raw session/message correlation out of exported labels, use trace context or approved bounded hashes, update the canonical OTel docs, and prove the OTLP path in a real setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove raw session IDs from exported OTel attributes:** `extensions/diagnostics-otel/src/service.ts:1193-1198`
  - body: `addRunAttrs` writes `openclaw.session_key` and `openclaw.session_id`, but the redaction drop list only blocks the camelCase keys. These values can leave the host through OTLP and conflict with the documented promise that session keys are not exported.
  - confidence: 0.93
- **[P2] Keep message identifiers out of metric labels:** `extensions/diagnostics-otel/src/service.ts:1422-1427`
  - body: `messageProcessingBaseAttrs` is passed to the message-processing counter and duration histogram while carrying run, message, chat, and session identifiers. That creates a metric series per message/session and leaks routing metadata; keep those identifiers on spans or events, not metric labels.
  - confidence: 0.9
- **[P2] Keep run and session identifiers out of skill metrics:** `extensions/diagnostics-otel/src/service.ts:2241-2243`
  - body: `skillExecutionBaseAttrs` includes `openclaw.run_id` and session identifiers, and the skill counter/histograms record with that same attrs object. This makes skill metrics high-cardinality per run/session; use bounded labels such as phase, scope, and outcome instead.
  - confidence: 0.88
- **[P3] Publish the new OTel docs through the canonical page:** `docs/diagnostics/otel-skill-observability.md:1`
  - body: The added diagnostics page has no front matter and is not wired into docs navigation, while `/gateway/opentelemetry` is the published OTel reference. Fold this content into the canonical page or add the required navigation/front matter so users can find it.
  - confidence: 0.78
