## Summary

Keep open. The PR is a plausible observability feature, but current head still has blocking type/source issues, exports raw session/message identifiers in ways that conflict with the existing OTel privacy and metric-cardinality contract, and does not yet prove the new OTLP behavior in a real collector path.

## What This Changes

The PR adds message-processing and skill-execution diagnostic events, emits them from embedded runs, maps them to diagnostics-otel spans/metrics, adds a skill observability doc, and touches qa-lab dependency/test fixtures.

## Best Possible Solution

Land a bounded, privacy-preserving telemetry contract that adds message and skill lifecycle spans/metrics without raw session keys or unbounded metric labels, with type-safe implementation, canonical OTel docs, and real OTLP smoke proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new observability feature. The blocking review findings are source-backed from the PR head and existing docs, with typecheck and OTel smoke validation still needed in a writable lane.

## Solution Assessment

No. The maintainable direction is to keep correlation on trace/span context or bounded hashed attributes, avoid raw session/message labels in metrics, fix the TypeScript mismatch, and prove the actual OTLP export path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep raw session identifiers out of OTel attributes:** `extensions/diagnostics-otel/src/service.ts:1193-1198`
  - body: `addRunAttrs` now copies `sessionKey` and `sessionId` into spans as `openclaw.session_key` and `openclaw.session_id`. The current OTel privacy docs explicitly say exported spans never include session keys, and these attributes can leave the process for third-party collectors. Please remove raw session identifiers from exported attributes or replace them with an approved bounded/hash-only correlation scheme.
  - confidence: 0.92
- **[P2] Keep per-message identifiers out of metric labels:** `extensions/diagnostics-otel/src/service.ts:1422-1427`
  - body: `messageProcessingBaseAttrs` includes `runId`, `messageId`, `chatId`, `sessionKey`, and `sessionId`, and the same object is passed to `messageProcessingCounter` and duration histograms. That creates a new metric series per run/message/chat/session, which breaks the bounded-label pattern used by the existing OTel metrics. Keep high-cardinality correlation on spans or events, not metric attributes.
  - confidence: 0.89
- **[P2] Align the skill attrs value type:** `extensions/diagnostics-otel/src/service.ts:2274-2275`
  - body: `skillExecutionBaseAttrs` is typed to return values that may be boolean, but the completed and error paths spread it into `Record<string, string | number>`. TypeScript rejects that index-signature narrowing; either narrow the helper return type or widen both `attrs` declarations consistently.
  - confidence: 0.87
- **[P3] Put the new OTel docs on the published path:** `docs/diagnostics/otel-skill-observability.md:1`
  - body: This adds a standalone diagnostics doc without front matter or docs.json navigation, while the existing `/gateway/opentelemetry` page is the canonical published OTel reference and still omits the new message/skill signals. Please fold the new content into the canonical page or add the required front matter/navigation so users can find it.
  - confidence: 0.78
