## Summary

Keep open. The PR is active and adds a plausible diagnostics feature, but it should not merge yet because the new lifecycle metrics use sensitive, unbounded correlation identifiers as metric labels and the supplied proof does not show the new OTLP/Jaeger/Prometheus path running. The related skill lifecycle request at https://github.com/openclaw/openclaw/issues/46780 also remains open as API/product context.

## What This Changes

The PR adds message-processing and skill-execution diagnostic events, exports them through diagnostics-otel spans/metrics, and includes related docs plus test/script/package adjustments.

## Best Possible Solution

Define the approved trace-only correlation attributes, keep metric labels low-cardinality, use the effective skill source in telemetry, trim unrelated churn, and land with redacted live OTLP collector proof.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows ID-heavy attrs flowing directly into OTel metric add/record calls and the skillScope value using the wrong source variable. For the feature itself, this is not a bug-style reproduction path; the missing proof is a live OTLP/Jaeger or Prometheus run.

## Solution Assessment

No. First-class diagnostics may be the right direction, but this implementation should separate trace correlation attrs from metric labels, fix the sandbox skill-scope label, and prove the exported behavior in a real collector before merge.

Overall correctness: patch is incorrect

Full review comments:

  - body: `messageProcessingBaseAttrs` includes run, message, chat, and session identifiers, and the same object is recorded as counter/histogram labels. This makes every run/session a new metric series; skill metrics should use bounded labels such as phase/outcome/scope and keep correlation IDs off metric records.
  - confidence: 0.88
- **[P3] Use the effective skills snapshot for scope labels:** `src/agents/pi-embedded-runner/run/attempt.ts:968-970`
  - body: `skillsSnapshotForRun` is cleared when a sandboxed run cannot use the snapshot, but `skillScope` still checks raw `params.skillsSnapshot`. Those runs will be labeled as `snapshot` even though the code actually resolves skills without that snapshot; base the label on `skillsSnapshotForRun` so the telemetry matches runtime behavior.
  - confidence: 0.78
