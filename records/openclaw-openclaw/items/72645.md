## Summary

This PR should stay open but is not merge-ready: current main does not already implement the requested per-message/skill OTel lifecycle telemetry, and the latest PR head still has blocking privacy, cardinality, type, docs, and changelog issues. The earlier duplicate-import finding appears fixed at the latest head, but the Greptile/ClawSweeper concerns around exported identifiers and skill attr typing still apply.

## What This Changes

The PR adds message.processing and skill.execution diagnostic events, emits them from embedded runs, exports matching diagnostics-otel spans/metrics, adds a skill observability doc, and adjusts qa-lab/test metadata.

## Best Possible Solution

Define a bounded, privacy-safe telemetry contract first, then merge a narrower implementation that compiles, keeps raw IDs off spans/metrics or hashes them under an approved contract, wires docs and changelog, and adds focused producer/exporter tests.

## Security Review

Status: needs_attention

Summary: The diff introduces telemetry privacy and supply-cost concerns by exporting raw session/message identifiers and unbounded labels through external OTLP paths.

Concerns:

- **[medium] Raw session identifiers exported to OTLP:** `extensions/diagnostics-otel/src/service.ts:1168`
  - body: The PR adds `openclaw.session_key` and `openclaw.session_id` to common span attributes even though the current public OTel contract says spans never include session keys. External collectors may retain these routing identifiers outside the OpenClaw trust boundary.
  - confidence: 0.94
- **[medium] Unbounded IDs added as metric attributes:** `extensions/diagnostics-otel/src/service.ts:1397`
  - body: The new message and skill metric attrs include per-run, per-message, per-chat, or per-session values. Prometheus-compatible collectors can turn those into one series per message/session, increasing retention cost and exposing routing identifiers.
  - confidence: 0.92
