## Summary

Keep open. The telemetry direction is plausible, but PR head still sends run/message/chat/session identifiers into OTel metric attributes, needs live collector proof for the new exported behavior, and overlaps an open skill-lifecycle API request without resolving that product design.

## What This Changes

The branch adds message-processing and skill-execution diagnostic events, diagnostics-otel spans and metrics, a skill observability doc, and several supporting test/script/package stabilizations.

## Best Possible Solution

Land a narrower telemetry patch that keeps correlation IDs on spans/logs, keeps metrics bounded, derives skill scope from the effective snapshot, and includes redacted live collector proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows identifier-heavy attrs flowing directly into OTel metric add/record calls and skillScope using the raw snapshot instead of the effective snapshot. For the feature itself, this is not a current-main bug reproduction path; the missing proof is a live collector run.

## Solution Assessment

No. First-class diagnostics may be the right direction, but this implementation should separate trace correlation attrs from metric labels, fix the sandbox skill-scope label, and prove the exported behavior in a real collector before merge.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep run and session IDs off processed metrics:** `extensions/diagnostics-otel/src/service.ts:1410-1412`
  - body: These new attrs are passed to messageProcessedCounter.add and messageDurationHistogram.record, turning the existing bounded message metrics into per-run/session series and exporting identifiers to metric backends. Keep these IDs on spans or logs and leave the metric attrs low-cardinality.
  - confidence: 0.9
- **[P1] Split message correlation IDs from metric attrs:** `extensions/diagnostics-otel/src/service.ts:1436-1441`
  - body: messageProcessingBaseAttrs includes run_id, messageId, chatId, session_key, and session_id, and the same attrs are recorded on counters and histograms. That creates per-message/chat/session metric series; build separate low-cardinality metric attrs and keep correlation IDs trace-only.
  - confidence: 0.92
- **[P1] Split run and session IDs from skill metric attrs:** `extensions/diagnostics-otel/src/service.ts:2321-2323`
  - body: skillExecutionBaseAttrs includes run_id, session_key, and session_id before the attrs are used for skill counters and histograms. Skill metrics should stay bounded labels such as phase, outcome, and scope while correlation IDs stay on spans.
  - confidence: 0.9
- **[P3] Base skill scope on the effective snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:965-967`
  - body: skillsSnapshotForRun is cleared when a sandboxed run cannot use the snapshot, but skillScope still checks params.skillsSnapshot. Those runs will be labeled as snapshot even though skill resolution uses the workspace path; derive the label from skillsSnapshotForRun.
  - confidence: 0.78
