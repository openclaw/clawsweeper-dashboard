## Summary

Keep open: the feature is plausibly useful and related to the open skill lifecycle observability request, but this branch still has blocking diagnostics-otel issues and the supplied proof does not exercise the OTLP export path.

## What This Changes

The PR adds message-processing and skill-execution diagnostic events, emits them around embedded-run stages, maps them to diagnostics-otel spans/metrics, adds observability docs, and carries several test/script/package adjustments.

## Best Possible Solution

Narrow the branch to the diagnostics feature, keep metrics low-cardinality, use an explicitly approved trace-correlation contract, align docs/tests with that contract, and add live OTLP collector proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows the high-cardinality attrs are passed directly to metric record calls, and the skill attrs type mismatch remains in the final diff. For the feature itself, reproduction is not a bug-style path; the missing proof is a live OTLP/Jaeger or Prometheus run.

## Solution Assessment

No. First-class events are a reasonable direction, but this implementation should split metric labels from trace-only correlation attrs, resolve the strict typing issue, and prove the exported behavior in a real collector before merge.
