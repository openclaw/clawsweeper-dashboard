## Summary

Keep open: the telemetry goal is plausible, but the current PR still conflicts with the existing OTel/Prometheus privacy and low-cardinality contract and does not include live exporter proof for the new spans/metrics.

## What This Changes

The PR adds `message.processing.*` and `skill.execution.*` diagnostics events with OTel spans/metrics/docs, plus qa-lab dependency metadata and some macOS/test tooling changes.

## Best Possible Solution

Land a narrower telemetry design that keeps metric labels bounded, keeps identifier correlation in approved trace context or redacted/hash form, updates the canonical OTel docs/tests, and includes redacted live collector proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows raw run/session/message/chat identifiers flowing into OTel span and metric attributes. The requested feature itself still lacks live collector proof.

## Solution Assessment

No. First-class message and skill telemetry may be useful, but this implementation should separate trace correlation from metric labels and stay within the existing telemetry privacy contract unless maintainers approve a policy change.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep raw run and session IDs out of spans:** `extensions/diagnostics-otel/src/service.ts:1205-1211`
  - body: These new snake_case keys are not in `DROPPED_OTEL_ATTRIBUTE_KEYS`, so default run, harness, tool, and skill spans can export raw run/session identifiers despite the current redaction contract. Use approved trace context, bounded hashes, or explicit maintainer-approved config instead.
  - confidence: 0.92
- **[P1] Keep IDs off existing message metrics:** `extensions/diagnostics-otel/src/service.ts:1410-1412`
  - body: Adding run and session IDs to `openclaw.message.processed` and `openclaw.message.duration_ms` changes an existing bounded metric into per-run/session series and conflicts with the documented channel/outcome label shape. Keep these metrics low-cardinality.
  - confidence: 0.93
- **[P1] Split identifiers from message-processing metric attrs:** `extensions/diagnostics-otel/src/service.ts:1436-1441`
  - body: `messageProcessingBaseAttrs` includes run, message, chat, and session identifiers, and the same attrs are used for counters and histograms. That exports identifiers to metric backends and creates per-message/chat/session series; build separate low-cardinality metric attrs.
  - confidence: 0.94
- **[P1] Split run and session IDs from skill metrics:** `extensions/diagnostics-otel/src/service.ts:2321-2323`
  - body: `skillExecutionBaseAttrs` includes run and session identifiers before the attrs are recorded on skill counters and histograms. Skill metrics should stay bounded labels such as phase, outcome, and scope while approved correlation data stays off metric labels.
  - confidence: 0.92
- **[P3] Base skill scope on the effective snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1319-1321`
  - body: Read-only sandbox runs clear `skillsSnapshotForRun`, but `skillScope` still checks `params.skillsSnapshot`. Those runs will be reported as `snapshot` even though skill resolution uses workspace mode; derive the label from `skillsSnapshotForRun`.
  - confidence: 0.78
