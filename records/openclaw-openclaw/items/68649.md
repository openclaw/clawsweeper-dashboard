## Summary

Keep open. Current main still leaves the remote PDF load path without an idle body-read timeout, while the lower media reader already supports one; the broader stuck-session recovery work does not make blocked native tool calls recoverable enough to replace the PDF-specific fix.

## What This Changes

The issue reports that stalled remote PDF fetches can wedge an agent session and asks for PDF fetch timeouts plus recovery for queued commands.

## Best Possible Solution

Thread a conservative readIdleTimeoutMs through the PDF remote-load path and add regression coverage, while leaving broader stuck-session abort policy to https://github.com/openclaw/openclaw/issues/71127.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists because the PDF tool reaches a body reader that only enforces idle timeouts when the caller supplies readIdleTimeoutMs, and current main does not supply it.

## Solution Assessment

Yes for the PDF-specific defect. The narrow maintainable fix is to pass readIdleTimeoutMs from the PDF tool through WebMediaOptions into readRemoteMediaBuffer, not to solve slash-command interruption in the same patch.
