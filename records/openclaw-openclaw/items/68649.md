- `impact:crash-loop`: The affected session can hang indefinitely and require process-level operator recovery.
- `impact:message-loss`: Subsequent user messages and slash commands queue behind the stuck tool call and do not get processed while the session is wedged.
- `impact:session-state`: The session appears active but is effectively zombie-stuck in an in-flight tool call state.

## Summary

Keep open. Current main has general stuck-session recovery as a partial backstop, but the PDF remote-load path still does not pass an idle body-read timeout to the media reader, so a stalled remote PDF can still block until the broader watchdog rather than returning a timely PDF tool error.

## What This Changes

The issue reports that remote PDF processing can hang indefinitely and wedge a session so queued messages and slash-command recovery cannot run.

## Best Possible Solution

Keep this open until the PDF remote-load path enforces a conservative idle body-read timeout at the PDF/WebMedia seam with focused regression coverage, while leaving broader session recovery behavior separate.

## Reproduction Assessment

Yes, at source level with high confidence. The PDF tool reaches `readResponseWithLimit` through `loadWebMediaRaw` without supplying `readIdleTimeoutMs`, and that reader waits directly on `reader.read()` when no chunk timeout is present.

## Solution Assessment

Yes for the PDF-specific defect. The narrow maintainable fix is to pass a conservative `readIdleTimeoutMs` through `WebMediaOptions` into `readRemoteMediaBuffer` and test that seam, not to change slash-command or diagnostic recovery policy in the same patch.
