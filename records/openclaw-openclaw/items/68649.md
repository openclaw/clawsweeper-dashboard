## Summary

Keep open. Current main still leaves the remote PDF load path without a body-read idle timeout, while the lower media reader already has a timeout contract that the PDF wrapper does not expose or pass through; the broader stuck-session recovery part is separately tracked in the open core recovery issue.

## What This Changes

The issue asks for stalled remote PDF fetches to return an error instead of wedging the session, plus broader recovery for queued messages and slash commands.

## Best Possible Solution

Add `readIdleTimeoutMs` to `WebMediaOptions`, forward it to `readRemoteMediaBuffer`, set a conservative default for remote PDF loads, and cover the PDF seam with focused tests while leaving general stuck-session recovery to https://github.com/openclaw/openclaw/issues/71127.

## Reproduction Assessment

Yes, source-level with high confidence: remote PDF inputs call `loadWebMediaRaw` without `readIdleTimeoutMs`, and the lower reader only aborts stalled body reads when `chunkTimeoutMs` is supplied. I did not perform a live stalled-CDN replay.

## Solution Assessment

Yes for the PDF-specific defect: the narrow fix is to plumb the existing idle-timeout option through web-media and set it from the PDF remote-load path. No for bundling slash-command bypass or watchdog policy into the same repair; that belongs with https://github.com/openclaw/openclaw/issues/71127.
