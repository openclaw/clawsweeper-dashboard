## Summary

Keep open. Current main still leaves the remote PDF load path without an idle body-read timeout, and the newer stuck-session recovery path intentionally does not abort active tool calls; the closed unmerged PDF timeout PR confirms the same narrow fix shape.

## What This Changes

The issue reports that a stalled remote PDF fetch can wedge the agent session queue, leaving later messages and recovery commands queued behind the stuck tool call.

## Best Possible Solution

Add internal web-media idle-timeout plumbing, pass a conservative default for remote PDF loads, and cover the PDF seam with focused regression tests while leaving broad session recovery policy to https://github.com/openclaw/openclaw/issues/71127.

## Reproduction Assessment

Yes, at source level with high confidence: remote PDF loads still do not supply `readIdleTimeoutMs`, and the lower reader only aborts stalled chunks when a caller supplies `chunkTimeoutMs`. I did not perform a live stalled-CDN replay.

## Solution Assessment

Yes for the PDF-specific defect: the narrow maintainable fix is to plumb `readIdleTimeoutMs` through `WebMediaOptions` and set it from the PDF remote-load path. Broad `/new`, `/restart`, and active tool/model recovery policy should remain with https://github.com/openclaw/openclaw/issues/71127.
