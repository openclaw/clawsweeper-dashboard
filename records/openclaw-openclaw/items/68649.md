## Summary

Keep open. Current main still has the PDF-specific stalled remote-fetch gap: the PDF tool reaches the shared web-media loader without an idle-read timeout, while the lower fetch layer only aborts stalled body reads when callers supply one; the focused PR that proposed this plumbing was closed unmerged.

## What This Changes

The issue reports that stalled remote PDF processing can wedge an agent session and leave follow-up messages plus `/new` and `/restart` queued behind the stuck run.

## Best Possible Solution

Add optional web-media idle-read timeout plumbing, set a conservative default for remote PDF loads, and cover the PDF call seam with regression tests while leaving slash-command bypass and session-watchdog behavior to a separate scheduler decision.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: remote PDF URLs reach `loadWebMediaRaw` without an idle-read timeout, and the lower fetch layer only aborts stalled body reads when `readIdleTimeoutMs` is supplied.

## Solution Assessment

Yes for the PDF-specific defect. The narrow timeout plumbing is the maintainable fix; mixing in `/new`, `/restart`, or session watchdog changes would expand the PR beyond the proven root cause.
