## Summary

Keep open. Current main still leaves the PDF remote-download path without an idle body-read timeout: the PDF tool passes only size/local-root/SSRF options into the shared media loader, the loader does not expose or forward `readIdleTimeoutMs`, and the lower fetch layer only aborts stalled body reads when that option is supplied. The focused linked PR that proposed this plumbing is closed unmerged.

## What This Changes

The issue asks for stalled remote PDF fetches to time out and for stuck agent sessions to recover or be interruptible without a gateway restart.

## Best Possible Solution

Add optional `readIdleTimeoutMs` plumbing through the web-media loader, set a conservative default for remote PDF loads, cover the PDF call seam with tests, and split slash-command bypass or watchdog behavior into a separate scheduler follow-up if still needed.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: invoke the `pdf` tool with an `http(s)` PDF whose response body stalls, and current main reaches `fetchRemoteMedia` without `readIdleTimeoutMs`, so the reader has no idle abort.

## Solution Assessment

Yes for the PDF-specific defect: reusing the existing `fetchRemoteMedia` idle-timeout seam is the narrowest maintainable fix. Mixing `/new`, `/restart`, or session-watchdog changes into the same patch would expand beyond the proven root cause.
