## Summary

Keep open. Current main still leaves the reported PDF-specific remote fetch path without an idle body-read timeout, even though the lower fetch layer already supports one and other media download paths use the same guard.

## What This Changes

The issue requests bounded PDF tool execution for stalled large remote PDFs plus broader session recovery for queued slash commands after a stuck tool call.

## Best Possible Solution

Add timeout plumbing for remote PDF downloads through `loadWebMediaRaw`/`fetchRemoteMedia`, with focused regression tests, while leaving broad scheduler and slash-command recovery to a separate maintainer-scoped item.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a remote `http(s)` PDF reaches `fetchRemoteMedia` without `readIdleTimeoutMs`, so a response body that stops yielding chunks has no idle abort at this call path.

## Solution Assessment

Yes for the PDF-specific defect. The narrow maintainable fix is to forward a conservative idle-read timeout from the PDF tool through web-media; broader `/new`, `/restart`, and session watchdog behavior should not be bundled into that repair without maintainer direction.
