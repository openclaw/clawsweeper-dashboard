## Summary

Keep open. Current main still leaves the remote PDF download path without a PDF-specific idle-read timeout, so a stalled response body can keep the tool call stuck long before any useful session recovery boundary.

## What This Changes

The issue asks for remote PDF tool fetches to time out and for stuck tool executions not to block session recovery commands.

## Best Possible Solution

Add an optional web-media idle-read timeout, forward it to fetchRemoteMedia, set a conservative default for remote PDF loads, and cover the PDF call seam with regression tests while leaving broader scheduler recovery separate.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a remote HTTP PDF reaches loadWebMediaRaw without readIdleTimeoutMs, and the lower fetch layer only aborts stalled body reads when that option is supplied.

## Solution Assessment

Yes for the PDF-specific defect. The narrow timeout plumbing is the maintainable fix; slash-command bypass and session watchdog behavior should not be mixed into the same repair without maintainer direction.
