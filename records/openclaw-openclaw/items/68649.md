## Summary

Keep open. Current main still has the PDF-specific stalled remote-fetch gap: the PDF tool reaches the shared web-media loader without an idle-read timeout, while the lower fetch layer only aborts stalled body reads when callers supply `readIdleTimeoutMs`; the focused PR that proposed this plumbing is closed unmerged.

## What This Changes

The issue requests bounded recovery for stalled remote PDF fetches so a hung PDF tool call does not wedge the agent session or block later recovery commands.

## Best Possible Solution

Add optional web-media idle-read timeout plumbing, set a conservative default for remote PDF loads, and cover the PDF call seam with regression tests while leaving generic stuck-session recovery to the canonical diagnostics issue.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: remote PDF URLs reach `loadWebMediaRaw` without an idle-read timeout, and the lower fetch layer only aborts stalled body reads when `readIdleTimeoutMs` is supplied.

## Solution Assessment

Yes for the PDF-specific defect. The narrow timeout plumbing is the maintainable fix; mixing in `/new`, `/restart`, or session watchdog changes would expand the repair beyond the proven root cause and overlap `https://github.com/openclaw/openclaw/issues/71127`.
