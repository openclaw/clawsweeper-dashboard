## Label Justifications

- `impact:crash-loop`: The report is about a hanging tool call that can leave an agent session unavailable and previously required gateway restart.

- `impact:session-state`: The affected behavior leaves session processing state wedged behind an in-flight tool call.

## Summary

Keep open. Current main still does not put an idle body-read timeout on the remote PDF load path; the newer stuck-session recovery is a last-resort run abort path, not a bounded PDF fetch timeout, and the PDF tool does not consume the abort signal passed to tool execution.

## What This Changes

The issue requests a timeout and recovery path for stalled remote PDF downloads so a hanging `pdf` tool call cannot wedge an agent session and queued commands.

## Best Possible Solution

Thread a conservative `readIdleTimeoutMs` through the PDF remote-load path and `WebMediaOptions`, with regression coverage at the PDF seam and no change to broader stuck-session policy.

## Reproduction Assessment

Yes, at source level. The PDF tool reaches a body reader that only enforces idle timeouts when `readIdleTimeoutMs`/`chunkTimeoutMs` is supplied, and current main does not supply it on the PDF remote-load path.

## Solution Assessment

Yes for the PDF-specific defect. The maintainable fix is to pass an idle body-read timeout through `pdf-tool.ts` and `web-media.ts`, while leaving slash-command and watchdog recovery policy to the existing diagnostics recovery surface.
