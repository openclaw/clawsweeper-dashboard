## Summary

Keep open. Current main still has the PDF-specific remote-fetch timeout gap: the lower fetch helper can abort stalled body reads, but the PDF tool calls the web-media wrapper without any idle timeout and the wrapper cannot forward one. The related focused PR is closed unmerged, while the broader lane-stall issue does not remove this PDF path defect.

## What This Changes

The issue requests a timeout for stalled remote PDF fetches plus recovery behavior so stuck tool calls do not wedge the session or block `/new` and `/restart`.

## Best Possible Solution

Land a narrow PDF/media fix that adds an optional read-idle timeout to the web-media options, forwards it to `fetchRemoteMedia`, sets a conservative default for remote PDF loads, and covers the PDF call seam with a regression test. Keep slash-command bypass and generalized watchdog recovery tracked separately from the PDF-specific repair.

## Reproduction Assessment

Yes for the narrow PDF-fetch hang. Current main has a source-level reproduction path where a remote PDF reaches `loadWebMediaRaw` without `readIdleTimeoutMs`, while `fetchRemoteMedia` only aborts stalled body reads when that option is supplied; the reporter also provided a transcript ending at a `pdf` tool call with no result.

## Solution Assessment

Yes, the focused timeout plumbing proposed by #68747 is the best maintainable fix for the PDF-specific failure because the lower fetch layer already has the contract. The `/new`, `/restart`, and watchdog portions should stay out of that patch unless maintainers intentionally broaden the work.
