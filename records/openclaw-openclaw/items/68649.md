## Summary

Keep open. Current main still has the PDF-specific remote-fetch timeout gap: the PDF tool reaches the shared web-media loader without an idle-read timeout, and the wrapper still cannot forward one to the lower fetch layer.

## What This Changes

The issue requests bounded remote-PDF fetching and recovery behavior so a stalled `pdf` tool call does not wedge a WhatsApp agent session.

## Best Possible Solution

Add an optional web-media idle-read timeout, forward it to `fetchRemoteMedia`, set a conservative default for remote PDF loads, and cover the PDF call seam with regression tests while leaving broader scheduler recovery separate.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a remote HTTP PDF reaches `loadWebMediaRaw` without `readIdleTimeoutMs`, and the lower fetch layer only aborts stalled body reads when that option is supplied.

## Solution Assessment

Yes for the PDF-specific defect. The narrow timeout plumbing is the maintainable fix; `/new`, `/restart`, and session watchdog changes should not be mixed into this repair without explicit maintainer direction.
