## Summary

Keep open. Current main still leaves the remote PDF load path without an idle body-read timeout, and the closed reference PR that targeted this exact gap was not merged; the broader stuck-session recovery work is related but does not replace the PDF-specific timeout fix.

## What This Changes

Requests timeout and recovery behavior so a stalled remote PDF fetch cannot wedge an agent session and queue later WhatsApp messages or slash commands behind it.

## Best Possible Solution

Add internal web-media idle-timeout plumbing for remote PDF loads with focused regression coverage, while leaving broad stuck-session abort policy to the canonical recovery thread.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the PDF path reaches a lower reader that only times out stalled chunks when the caller supplies readIdleTimeoutMs, and this caller does not supply it. I did not replay a live stalled CDN response.

## Solution Assessment

Yes for the PDF-specific defect: the narrow fix is to pass a conservative readIdleTimeoutMs through WebMediaOptions from the PDF tool into readRemoteMediaBuffer. The slash-command and session-wide abort behavior belongs to the broader recovery thread.
