## Summary

Keep open. Current main still leaves the remote PDF load path without an idle-read timeout, even though the lower fetch/reader stack only aborts stalled body reads when a caller supplies `readIdleTimeoutMs`; the broader stuck-session recovery policy remains separately tracked.

## What This Changes

The issue requests timeout and recovery behavior for hung remote PDF tool calls so a stalled PDF fetch cannot wedge a session or block `/new` and `/restart`.

## Best Possible Solution

Add optional web-media idle-timeout plumbing, pass a conservative default for remote PDF loads, and cover the PDF seam with regression tests while leaving general stuck-session recovery to https://github.com/openclaw/openclaw/issues/71127.

## Reproduction Assessment

Yes, source-level with high confidence: remote PDF inputs call `loadWebMediaRaw` without `readIdleTimeoutMs`, and the lower reader only times out when `chunkTimeoutMs` is present. I did not perform a live stalled-CDN replay.

## Solution Assessment

Yes for the PDF-specific defect: plumb `readIdleTimeoutMs` through `WebMediaOptions` and set it from the PDF remote-load path with focused tests. No for broad `/new`, `/restart`, or watchdog policy changes in this repair; those belong with https://github.com/openclaw/openclaw/issues/71127.
