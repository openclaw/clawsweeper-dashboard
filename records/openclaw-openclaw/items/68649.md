## Summary

Keep open. Current main still leaves the remote PDF load path without an idle body-read timeout, and the broader stuck-session recovery path intentionally does not recover blocked native tool calls, so the PDF-specific hang remains actionable.

## What This Changes

The issue reports that remote PDF processing can hang indefinitely and leave the session queue unable to process messages or recovery slash commands.

## Best Possible Solution

Add internal web-media idle-timeout plumbing for remote PDF loads, with focused regression coverage, while leaving broad stuck-session abort policy to the separate recovery issue.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the PDF path reaches a lower reader that only times out stalled chunks when the caller supplies readIdleTimeoutMs, and this caller does not supply it. I did not replay a live stalled CDN response.

## Solution Assessment

Yes for the PDF-specific defect: the narrow solution is to pass a conservative readIdleTimeoutMs through WebMediaOptions from the PDF tool into readRemoteMediaBuffer. The slash-command and session-wide abort behavior belongs to the broader recovery thread.
