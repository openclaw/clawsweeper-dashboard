## Summary

Keep open. Current main still preloads LM Studio models through the native load endpoint before inference, and LM Studio's own JIT/TTL docs make the reported unload behavior plausible and actionable.

## What This Changes

The issue asks for LM Studio JIT auto-unload behavior to work when OpenClaw switches between LM Studio models, likely by making the provider's manual load call optional.

## Best Possible Solution

Add a provider-owned, documented LM Studio preload/load opt-out that defaults to the current behavior and skips the native load call when users want LM Studio JIT to own model loading and eviction.

## Reproduction Assessment

Yes. The issue gives a detailed LM Studio JIT/TTL scenario, and current source deterministically calls `/api/v1/models/load` before LM Studio inference, which is the behavior the report says prevents JIT unload semantics.

## Solution Assessment

Yes, the narrowest maintainable direction is a LM Studio provider-local opt-out with the existing preload behavior preserved by default; changing the default to never preload would risk users with JIT disabled.

## Repair Work Prompt

Fix OpenClaw issue #75921: LM Studio JIT auto-unload is bypassed because the bundled `lmstudio` provider calls `/api/v1/models/load` before inference. Add a documented provider-owned opt-out for the manual preload/load API that preserves current behavior by default. When the opt-out is disabled, `wrapLmstudioInferencePreload` should skip `ensureLmstudioModelLoaded` for chat inference and let LM Studio JIT load the model on the first inference request. Consider whether the same option should apply to embeddings warmup or document the boundary clearly. Keep the change inside the LM Studio plugin/config contract; do not change generic OpenAI-compatible provider behavior or global provider defaults. Update strict config types/schema/help/generated artifacts as required, LM Studio docs, model-provider docs if needed, focused tests, and CHANGELOG.md because this is user-facing.
