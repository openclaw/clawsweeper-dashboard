## Summary

Keep this PR open. Current main still has the textless ACP status crash path, while the PR head guards both the original `hashText(event.text)` crash and the downstream `emitSystemStatus(event.text)` crash noted in review; it remains a normal maintainer-review candidate rather than something to close as implemented.

## What This Changes

The PR updates `src/auto-reply/reply/acp-projector.ts` so missing status text is treated as an empty string when computing usage-update hashes and emitting system status during Telegram/ACP dispatch.

## Best Possible Solution

Land this branch or a superseding patch that keeps both projector guards, adds a focused `acp-projector` or `dispatch-acp` regression test for a visible textless `usage_update`, and resolves the ACP status-text contract at the runtime boundary or with documented normalization.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: feed the projector a visible `usage_update` status event whose runtime `text` is missing, and the current code reaches either `hashText(event.text).trim()` or `emitSystemStatus(event.text).trim()`.

## Solution Assessment

Yes for the narrow crash fix: guarding both the hash key and status emission call is the smallest defensive change at the active projector boundary. The safer merge-ready version should add regression coverage and separately clean up the ACP event type mismatch.
