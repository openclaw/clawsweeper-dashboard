## Summary

Keep open. Current `main` still has the reported ACP/Telegram textless-status crash paths, while this PR guards both known trim sites after the Greptile review; it is useful source work but still needs merge review and the required release-note/test polish.

## What This Changes

The PR changes `src/auto-reply/reply/acp-projector.ts` so missing ACP status text is treated as an empty string when hashing usage updates and emitting system status.

## Best Possible Solution

Land this branch or a superseding patch that keeps both projector guards, adds a focused regression test for textless visible `usage_update`, adds the required changelog entry, and then separately aligns the ACP status-event type or normalizes status text at the runtime boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current `main`: feed the ACP reply projector a visible `usage_update` status event with missing `text`; missing numeric usage fields reach `hashText(event.text).trim()`, and numeric textless usage updates can still reach `emitSystemStatus(event.text).trim()` when summaries are enabled.

## Solution Assessment

Yes for the narrow code fix: guarding both the hash key and status emission call is the smallest maintainable crash fix at the active projector boundary. The merge-ready path should add regression coverage, a changelog entry, and decide whether to make status text optional in the ACP runtime contract.
