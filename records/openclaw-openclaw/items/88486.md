- `merge-risk: 🚨 session-state`: The patch can mis-associate diagnostic events by pairing a source `sessionKey` with a target transcript UUID on native command redirects.

## Summary

Keep open: the normal interactive diagnostic UUID fix is valid and has real runtime proof, but the patch still has a blocking session-state defect for native command redirects because it can pair the source conversation `sessionKey` with the target transcript `sessionId`.

## What This Changes

The PR hoists the interactive session-store lookup, passes its entry session UUID into the diagnostic message lifecycle, and adds a Slack dispatch regression test for diagnostic session IDs.

## Best Possible Solution

Keep the UUID population, but compute one resolved diagnostic session ref before lifecycle construction so `sessionKey` and `sessionId` always come from the same source or target session, with regression coverage for native command redirects.

## Reproduction Assessment

Yes, by source inspection on the PR head: native commands can resolve the store entry from `CommandTargetSessionKey` while the lifecycle default `sessionKey` still comes from `ctx.SessionKey`. I did not run the Vitest suite because this review is read-only.

## Solution Assessment

No, not as submitted: carrying the UUID is the right direction, but the implementation needs to align both diagnostic ref fields from one resolved session before lifecycle creation. The safer fix is a small ref normalization plus a native command-target regression test.
