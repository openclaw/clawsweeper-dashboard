## Summary

Keep open: current main still omits the session UUID on interactive diagnostic lifecycle events, so the PR addresses a real source-visible gap, but it is not merge-ready until it keeps `sessionKey` and `sessionId` coherent for native command target turns and supplies real behavior proof.

## What This Changes

The PR hoists the interactive session-store lookup, passes its `sessionId` into `createDiagnosticMessageLifecycle`, and adds a regression test for Slack interactive diagnostic events.

## Best Possible Solution

Populate interactive diagnostic events from one coherent resolved session reference, cover both ordinary and native command target cases, then merge after redacted real plugin/exporter proof.

## Reproduction Assessment

Yes from source inspection: current main creates the interactive diagnostic lifecycle with `sessionKey` only while the session-store UUID is available later in the same function, and the cron path already passes both fields. I did not run the test suite because this review was explicitly read-only.

## Solution Assessment

No, not yet: adding the UUID from the store is the right direction for ordinary sessions, but the PR must keep `sessionKey` and `sessionId` from the same resolved session for native command targets.
