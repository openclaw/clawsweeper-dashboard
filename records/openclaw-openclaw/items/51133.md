## Summary

Keep open. Current main still leaves the reported Feishu/MiniMax loop path plausible: `sessions_list` can return an empty filtered list without clarifying the caller's current-session state, loop detection is disabled by default, and direct `message(action="send")` calls still flow through Feishu send handling without a per-turn exact-duplicate guard.

## What This Changes

The issue reports a Feishu and MiniMax-M2.5 loop where the agent repeatedly calls `message` and `sessions_list`, sending duplicate outbound messages after `sessions_list` returns zero rows.

## Best Possible Solution

Make current-session state unambiguous to the model and prevent one agent turn from delivering the exact same Feishu outbound send repeatedly. The most maintainable direction is to steer self-session checks to `session_status(sessionKey="current")` or annotate empty `sessions_list` results, plus add a narrow exact-duplicate send guard with regression coverage for the reported `sessions_list` zero-row sequence.

## Reproduction Assessment

No high-confidence live reproduction is available from the issue alone: it gives a concrete sequence but not a transcript, credentials, or deterministic MiniMax/Feishu fixture. Static inspection does verify the code-level path where `sessions_list` can return `count: 0` and repeated identical `message` sends can still be dispatched.

## Solution Assessment

Yes, the best fix direction is a narrow source-level repair, not closing as solved and not broadly enabling all loop detection by default. A safer alternative to broad model-policy changes is an additive current-session clarification plus exact same-turn duplicate-send suppression.
