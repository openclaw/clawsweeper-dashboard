## Summary

Keep open. Current main has prompt guidance that reduces `sessions_list` polling, but the reported MiniMax/Feishu duplicate-message loop is not proven fixed because `sessions_list` can still return an empty list without explaining current-session semantics, and repeated successful message-tool sends are recorded for downstream reply dedupe rather than blocked at the send point.

## What This Changes

The issue reports a MiniMax/Feishu loop where the agent repeatedly calls `message` and `sessions_list`, producing duplicate visible replies after `sessions_list` returns zero sessions.

## Best Possible Solution

Clarify the `sessions_list` empty-result contract in the tool result and prompt/tests, then separately decide whether same-turn duplicate visible sends should have runtime enforcement.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection still shows the ambiguous empty `sessions_list` path and no hard duplicate message-tool send guard, but the exact MiniMax/Feishu loop was not reproduced on current main.

## Solution Assessment

Unclear. A narrow `sessions_list` contract clarification is likely safer than changing session visibility or blocking repeated sends, while runtime duplicate-send enforcement needs maintainer policy judgment.
