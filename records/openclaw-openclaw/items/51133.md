## Summary

Keep open. Current main has prompt guidance that reduces `sessions_list` polling, but the reported MiniMax/Feishu duplicate-message loop is not proven fixed, and the remaining fix direction needs maintainer choice between tool-result guidance, self-session visibility semantics, or duplicate-send guardrails.

## What This Changes

The issue reports an agent loop where MiniMax-M2.5 on Feishu repeatedly sends duplicate messages after `sessions_list` sometimes returns `count:0` for the current session.

## Best Possible Solution

Clarify `sessions_list` empty-result semantics in the tool result and prompt/tests first, then consider a separate, explicit policy for same-turn duplicate visible sends only if maintainers want runtime enforcement.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection shows a plausible current path where `sessions_list` can return `{count:0}` without explaining that the current session is not lost, but I did not reproduce the MiniMax/Feishu loop on current main.

## Solution Assessment

Unclear. A narrow tool-result/prompt clarification looks safer than changing session visibility or blocking duplicate sends, while a hard runtime duplicate guard needs maintainer policy judgment.
