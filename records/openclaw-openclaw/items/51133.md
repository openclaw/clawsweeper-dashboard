## Summary

Keep open. Current main still leaves the reported loop path plausible: `sessions_list` can return an empty filtered list without explaining current-session state, repeated `message(action="send")` calls still dispatch to Feishu, and tool-loop detection remains disabled by default.

## What This Changes

The issue reports a MiniMax/Feishu run where the model repeatedly calls `message` and `sessions_list` after `sessions_list` sometimes returns `count: 0`, producing duplicate outbound messages.

## Best Possible Solution

Make current-session state unambiguous when `sessions_list` is empty and add narrow same-turn duplicate-send suppression for identical Feishu/message-tool sends, without broadly enabling loop detection or changing unrelated channel semantics.

## Reproduction Assessment

No high-confidence live reproduction is available from the issue alone because it lacks a transcript, credentials, and a deterministic MiniMax/Feishu fixture. Source inspection does verify the current-main path where `sessions_list` can return `count: 0` and repeated identical `message` sends can still dispatch.

## Solution Assessment

Yes, a narrow source-level repair is the best direction: clarify `sessions_list` current-session state and suppress exact duplicate same-turn outbound sends. Broad default loop detection or model/provider-specific policy would be a riskier and less targeted fix.
