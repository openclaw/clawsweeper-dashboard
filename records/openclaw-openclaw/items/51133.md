## Summary

Keep open. Current main has prompt guidance and opt-in loop detection, but the default `sessions_list` result can still look like an empty complete answer without current-scope semantics, and message-tool sends are still dispatched rather than blocked as same-turn duplicates; the reported MiniMax/Feishu duplicate-message loop is not proven fixed.

## What This Changes

The issue reports a MiniMax/Feishu loop where the agent repeatedly calls `message` and `sessions_list`, causing duplicate visible replies.

## Best Possible Solution

Add additive `sessions_list` scope/current-session metadata or note, likely through https://github.com/openclaw/openclaw/issues/50646, and separately make an explicit maintainer decision on runtime duplicate-send enforcement.

## Reproduction Assessment

No high-confidence live Feishu/MiniMax reproduction is available. Source inspection still shows the ambiguous `sessions_list` empty-result path, opt-in loop detection, and no hard duplicate-send guard.

## Solution Assessment

Unclear. A narrow `sessions_list` contract clarification is the best first fix, while runtime duplicate visible-send suppression needs maintainer policy judgment.
