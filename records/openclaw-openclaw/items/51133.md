## Summary

Keep open. Current main still leaves the reported Feishu/MiniMax loop path plausible: `sessions_list` can return an empty filtered list without clarifying current-session state, direct `message(action="send")` still reaches Feishu sending, and repetitive tool-loop blocking remains disabled by default.

## What This Changes

The issue reports a MiniMax/Feishu agent loop where `sessions_list` returns zero rows and the model repeatedly sends duplicate messages while trying to recover the session.

## Best Possible Solution

Make current-session state unambiguous to the model and prevent one agent turn from delivering the exact same Feishu outbound send repeatedly, without broadly enabling loop detection or changing unrelated messaging semantics.

## Reproduction Assessment

No high-confidence live reproduction is available from the issue alone because it lacks a transcript, credentials, and deterministic MiniMax/Feishu fixture. Source inspection does verify the code-level path where `sessions_list` can return `count: 0` and repeated identical `message` sends can still be dispatched.

## Solution Assessment

Yes, a narrow source-level repair is the best direction: clarify current-session state around empty `sessions_list` results and add exact same-turn duplicate-send suppression. Broadly changing model policy or enabling all loop detection by default would be a riskier fix.
