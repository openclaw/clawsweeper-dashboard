Keep open: the report describes a plausible user-visible runaway loop, and current main still leaves rolling tool-loop detection opt-in while overlapping loop-default PRs remain unresolved.

## What This Changes

## Best Possible Solution

Choose one focused maintainer-approved loop-breaker policy for small/chat-channel runs, preserve run-scoped no-progress semantics, and add regression coverage for the reported `message` plus `sessions_list` loop before closing.

## Solution Assessment

Unclear as a direct fix path. The issue points at a real guardrail gap, but the best solution is a focused maintainer-chosen loop-detection/default policy rather than treating `sessions_list` count 0 alone as the whole root cause.
