## Summary

Keep open: current main has useful prompt guidance against polling loops, but `sessions_list` still depends entirely on gateway/store rows and can return `count: 0`; that leaves the reported current-session-missing failure mode plausible rather than implemented away.

## What This Changes

The issue asks OpenClaw to prevent MiniMax/Feishu turns from looping through duplicate `message` sends and `sessions_list` calls when the current session appears missing.

## Best Possible Solution

Make `sessions_list` self/tree results unable to imply that the caller's current session disappeared, ideally by returning a minimal current-session row or explicit current-session metadata when the gateway store list is empty, with regression coverage for the empty-list case.

## Reproduction Assessment

Yes, at source level: if gateway `sessions.list` returns an empty `sessions` array for the current session, `sessions_list` returns `count: 0` because it only forwards filtered gateway rows. I did not run a live Feishu/MiniMax reproduction.

## Solution Assessment

No, the current prompt-only anti-polling guidance is not the best complete fix. The narrower maintainable repair is to make the tool result itself consistent with the documented current-session visibility contract and then add targeted regression tests.

## Repair Work Prompt

Fix the current-session-missing `sessions_list` behavior from https://github.com/openclaw/openclaw/issues/51133. In `src/agents/tools/sessions-list-tool.ts` and shared helpers as needed, ensure the model-facing `sessions_list` result for self/tree visibility cannot return `count: 0` merely because the caller's current session row is absent from gateway `sessions.list`; synthesize or otherwise expose a minimal current-session row/metadata only when appropriate and without broadening cross-agent visibility. Add regression coverage for an empty gateway `sessions.list` response with `agentSessionKey` set, plus a normal non-empty list case to avoid duplicates. Do not enable `tools.loopDetection` by default and do not change message delivery semantics beyond this contract. Add a user-facing changelog fix entry if behavior changes.
