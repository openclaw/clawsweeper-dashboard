## Summary

Keep open: the updated PR now delays the outer timeout rejection, but the real native compaction caller still ignores the wrapper-provided abort signal and cancels through `abortCompaction()`, so the central 180s native timeout loop is not fixed. The PR body proof is still unit/fake-timer output rather than a real long compaction or gateway run.

## What This Changes

The PR changes compaction timeout handling to preserve partial summaries, updates preflight timeout guidance to mention the existing timeout setting, adds regression tests, and removes an unused timeout helper.

## Best Possible Solution

Keep the timeout guidance, but implement recovery at the native session compaction boundary so completed summary progress survives the real abortCompaction timeout path, with regression coverage through that call path and redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current main wraps native compaction in one 180s operation timeout, but the native caller ignores the wrapper signal and cancels through abortCompaction, which does not return partial progress. I did not run a live slow-provider compaction in this read-only review.

## Solution Assessment

No. The preflight copy and inner timeout pieces are plausible, but the outer safety-timeout test uses a synthetic callback that observes the wrapper signal rather than the real native activeSession.abortCompaction path.
