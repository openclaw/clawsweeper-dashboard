## Summary

Keep this PR open: it addresses the right Zalo owner surface, but the new predicate ignores the normalized target value that core passes for provider-prefixed targets, so the fix is still incomplete and needs real CLI behavior proof before merge.

## What This Changes

The PR replaces Zalo's numeric-only target resolver with a Zalo-local numeric-or-hex chat_id predicate and adds unit coverage for that predicate.

## Best Possible Solution

Keep the Zalo-specific predicate in the plugin, make it validate the normalized target value as well as bare input, cover bare and provider-prefixed chat IDs, and add real CLI proof for the fixed path.

## Reproduction Assessment

Yes. Current main's Zalo plugin uses the generic numeric-only resolver before the outbound resolver's normalized fallback, and the linked issue gives concrete hex chat_id examples that hit that path.

## Solution Assessment

No. The plugin-local fix is the right layer, but it should honor the normalized argument that core passes so advertised Zalo provider prefixes work for the same hex chat IDs.
