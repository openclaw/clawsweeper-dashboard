## Summary

Keep open: current main still has the Gateway-first fallback behavior this PR is trying to change, but the draft patch is not safe to merge because the new local-first path bypasses Gateway authorization and notification side effects.

## What This Changes

The branch makes `openclaw devices approve` try local loopback pairing-file approval before Gateway approval, suppresses local fallback when explicit auth flags are present, and updates the CLI fallback tests.

## Best Possible Solution

Preserve the user-facing goal, but keep approval behind Gateway authorization or mirror the Gateway authz and `device.pair.resolved` side effects in a tightly bounded local-loopback path with tests, docs, and changelog coverage.

## Reproduction Assessment

Yes for review purposes: source inspection shows the PR approves locally before the Gateway path, and existing gateway tests exercise the non-admin cross-device approval denial this would bypass. A live runtime repro was not run under the read-only review constraint.

## Solution Assessment

No. The local-first approach is not the narrowest safe fix because it changes the approval authority and side-effect contract; a safer fix should preserve Gateway authorization or explicitly reproduce the same checks and notifications.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve Gateway authorization before local approval:** `src/cli/devices-cli.ts:168-172`
  - body: The new local-first branch approves from pairing files before `device.pair.approve` can enforce caller-device ownership. Because `approveLocalPairingRequest` supplies `operator.admin`, an implicit local device-token session can approve a different pending device that the Gateway path would reject.
  - confidence: 0.87
- **[P2] Preserve approval resolution broadcasts:** `src/cli/devices-cli.ts:168-172`
  - body: Returning immediately after local approval skips the Gateway success path that logs approval and broadcasts `device.pair.resolved`. Existing UI or Gateway subscribers that rely on that event can remain stale after a CLI local-first approval.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/cli/devices-cli.ts:168-172`
  - body: This is a user-facing `openclaw devices approve` fix, but the branch does not update `CHANGELOG.md`. Repo policy requires a one-line changelog entry for user-facing fixes.
  - confidence: 0.79
