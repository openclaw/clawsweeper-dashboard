## What This Changes

The PR updates the devices CLI fallback resolver so explicit `--token` or `--password` auth cannot fall back to local pairing files, and adds focused CLI tests plus devices docs for that rule.

## Best Possible Solution

Land the narrow fallback guard with tests/docs after maintainer review, keeping Gateway approval authoritative and allowing local fallback only for implicit local loopback credentials.

## Reproduction Assessment

Yes. Source inspection on current main shows the fallback resolver only blocks explicit `--url`, not explicit `--token` or `--password`, and the PR body supplies a real loopback Gateway after-fix run showing explicit-token auth stays on the Gateway path.

## Solution Assessment

Yes. Placing the guard in the shared fallback resolver covers both list and approve while preserving Gateway-first approval side effects; I did not find a narrower safer fix.
