## Summary

Keep open: current main still uses the Gateway-first approval path, and the PR branch is draft/dirty with blocking authorization and notification side-effect concerns in a device approval flow.

## What This Changes

The PR changes `openclaw devices approve` to try local loopback approval from pairing files before Gateway approval, suppress local fallback when token/password auth is explicit, and update CLI tests for local-first plus unknown-request fallback behavior.

## Best Possible Solution

Keep the user-facing goal, but implement it with the Gateway authorization and resolution contract intact: either keep approval behind the Gateway decision or mirror caller-device authz plus `device.pair.resolved` side effects in a narrowly documented local-loopback path, with docs and tests for explicit auth, unknown request IDs, device-token ownership, and UI refresh behavior.

## Reproduction Assessment

Yes for review purposes: the source path shows the PR approves locally before Gateway auth, and `src/gateway/server-methods/devices.test.ts` already gives a focused unit pattern for the cross-device non-admin denial that this bypasses. A live runtime repro was not attempted under the read-only constraint.

## Solution Assessment

No. The proposed local-first approach is not the safest fix because it bypasses Gateway authorization and notification side effects; a maintainable solution should preserve or explicitly reproduce those contracts before approving from local state.
