## Summary

Keep this open. Current main still has exec policy, allowlist, durable approvals, strict inline-eval checks, and a narrow stdin-only safe-bin path, but not the requested built-in L1-L4 shell-command risk classifier or dangerous-command auto-deny path; the new discussion reinforces that this is a security-boundary design issue, not conservative cleanup.

## What This Changes

Request to add a built-in exec approval risk classifier that auto-approves safe commands, prompts for intermediate commands, and auto-denies dangerous shell commands.

## Best Possible Solution

Define the exec approval threat model first, then choose either a narrowly opt-in core policy layer based on parsed argv/shell structure or an explicit plugin policy seam that can return allow, require approval, or deny for well-defined command shapes.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main source and docs inspection, which shows allowlist/safe-bin/strict-inline-eval behavior rather than a built-in L1-L4 classifier.

## Solution Assessment

Unclear. The requested static L1-L4 classifier is not proven to be the best design because auto-allow and auto-deny decisions change a host exec security boundary; a parsed, opt-in policy contract is safer than command-name classification.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes changing automatic allow/deny decisions for host exec commands.

Concerns:

- **[medium] Avoid static command-name auto-approval:**
  - body: Classifying broad commands such as `cat`, `grep`, `find`, `curl`, `sed`, `awk`, `xargs`, or `git` from command text alone can miss shell composition, redirects, file operands, PATH hijacking, wrappers, and platform parsing differences, potentially weakening exec approvals.
