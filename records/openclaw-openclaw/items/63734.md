## Summary

Keep this open. Current main still has policy modes, explicit allowlists, durable approvals, a narrow stdin-only safe-bin path, and strict inline-eval checks, but it does not implement the requested built-in L1-L4 exec risk classifier or dangerous-command auto-deny path; the later comments reinforce that this is a security-boundary design issue, not conservative cleanup.

## What This Changes

Request adding a built-in L1-L4 command risk classifier to exec approvals so safe commands can auto-run and dangerous commands can be denied.

## Best Possible Solution

Keep this as an explicit maintainer/security design issue: define the threat model first, then implement a parsed, opt-in exec policy layer or documented plugin policy seam with docs and regression coverage across gateway and node hosts.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source/docs/test inspection on current main, which shows allowlist, safe-bin, durable-approval, and strict-inline-eval behavior rather than a built-in L1-L4 classifier.

## Solution Assessment

Unclear. The requested static L1-L4 classifier is not proven to be the best design because auto-approval and auto-denial change a security boundary; a parsed, opt-in policy layer or reviewed plugin policy seam is safer than command-name classification.
