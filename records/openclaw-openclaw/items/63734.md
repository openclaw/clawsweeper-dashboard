## Summary

Keep this open. Current main still does not have a built-in L1-L4 exec command risk classifier or a dangerous-command auto-deny path; the shipped behavior remains policy, allowlist, durable approvals, strict inline-eval checks, and a narrow stdin-only safe-bin fast path. The later discussion is useful: one commenter offered an L4 prototype, and another correctly called out argv and shell-composition context risks, which reinforces that this needs maintainer/security design rather than automatic cleanup.

## What This Changes

Add a built-in exec command risk classifier that can auto-approve safe command shapes, apply stronger approval for risky commands, and auto-deny dangerous commands.

## Best Possible Solution

Keep this issue open for maintainer and security design. If core accepts the feature, implement it as an explicit opt-in exec policy layer based on parsed argv/shell structure, documented threat model, UI/config/docs updates, and regression tests across gateway and node hosts; if plugin-first is preferred, add or document an exec-specific plugin policy API that can safely block, require approval, or allow only well-defined command shapes.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main: the exec approval contract, gateway path, node path, docs, and tests show allowlist/safe-bin/strict-inline-eval behavior rather than a built-in L1-L4 classifier.

## Solution Assessment

Unclear; the requested broad static classifier is not proven to be the best design because it changes the host-exec security boundary. A narrower maintainable direction is a parsed, opt-in policy layer or a reviewed plugin policy seam, with L4 detection treated as one component rather than an ad hoc command-text preflight.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes changing when host exec commands are automatically allowed or denied.

Concerns:
