## Summary

Keep open. Current main still implements deterministic exec approval modes, allowlists, safeBins, durable approvals, strict inline-eval gates, and command highlighting rather than the requested built-in L1-L4 command risk classifier; the remaining work is a security/product design decision, not conservative cleanup.

## What This Changes

Request to add a built-in L1-L4 command risk classifier that auto-approves safe exec commands, escalates ambiguous commands, and hard-denies dangerous commands.

## Best Possible Solution

Keep this as the canonical adaptive exec-risk approval design item, coordinated with the denylist trackers, and require parsed argv/shell-context semantics plus opt-in config/docs/tests before changing execution policy.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, docs, focused tests, live issue context, related trackers, and latest-release inspection show the requested classifier is absent from current main and not shipped as a policy mode.

## Solution Assessment

Unclear. The requested L1-L4 direction addresses real approval friction, but a static built-in classifier is not proven to be the safest maintainable design; parsed, opt-in deterministic policy coordinated with denylist work is the safer path.
