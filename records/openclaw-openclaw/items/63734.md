## Summary

Keep open. Current main and the latest shipped release still use deterministic exec policy, allowlists, safeBins, durable approvals, and strict inline-eval gates rather than the requested built-in L1-L4 command risk classifier; the discussion and related trackers make this a security/product design item, not conservative cleanup.

## What This Changes

Request adds built-in L1-L4 command risk classification to exec approvals, auto-approving safe commands and auto-denying dangerous ones.

## Best Possible Solution

Keep this as the canonical maintainer/security design item for an additive, parsed, opt-in exec risk policy coordinated with the denylist tracker and backed by docs plus gateway/node regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, docs, tests, shipped-release inspection, and related issue history provide a high-confidence verification path showing the requested classifier is absent from current main.

## Solution Assessment

Unclear. A static L1-L4 classifier is not proven to be the safest maintainable design; a parsed, opt-in policy layer coordinated with native deny-pattern work is the safer direction.

## Security Review

Status: needs_attention

Summary: This request is security-sensitive because it would change when host exec commands are automatically allowed, prompted, or blocked.
