## Summary

Keep this open. Current main still has deterministic exec security/ask modes, explicit allowlists, narrow safeBins, durable approvals, and strict inline-eval gates, but not the requested built-in L1-L4 command risk classifier or dangerous-command auto-deny path; the related discussion confirms this is a security-boundary design issue rather than conservative cleanup.

## What This Changes

Add a built-in L1-L4 command risk classifier for exec approvals that can auto-allow safe commands and auto-deny dangerous ones.

## Best Possible Solution

Keep this as a maintainer/security design issue: define a parsed, opt-in exec policy/risk layer with deterministic hard guards, docs, regression coverage, and coordination with the open denylist trackers.

## Reproduction Assessment

Not applicable as a bug reproduction. The high-confidence verification path is source, docs, tests, and issue-history inspection on current main, which shows deterministic exec policy, allowlists, safeBins, durable approvals, and strictInlineEval rather than a built-in L1-L4 classifier.

## Solution Assessment

Unclear. The requested static L1-L4 classifier is not proven to be the safest maintainable design; a parsed, opt-in policy layer coordinated with denylist work is safer than broad command-name classification.
