## Summary

Keep open. Current main still has deterministic exec security/ask modes, allowlists, safe-bin fast paths, durable approvals, strict inline-eval handling, and display/risk hints, but not the requested built-in L1-L4 exec risk classifier; the adjacent tirith scanner PR at https://github.com/openclaw/openclaw/pull/41320 is open, not linked as a closing fix, and has unresolved security/default-policy review blockers.

## What This Changes

The issue requests a built-in L1-L4 command risk classifier for exec approvals, including auto-allowing safe read-only commands and auto-denying dangerous commands.

## Best Possible Solution

Design an opt-in, parser-backed exec policy surface that coordinates argv-shape auto-allow and hard-deny rules with existing allowlist, safeBins, durable approvals, strictInlineEval, UI, audit, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection give high-confidence evidence that current main lacks the requested risk-classifier policy surface, while the discussion frames this as a new security/product capability.

## Solution Assessment

Unclear as proposed. The L1-L4 goal addresses real approval fatigue, but a command-text or binary-name classifier is not the safest maintainable design; parsed argv-shape policy with opt-in semantics is safer.
