## Summary

Keep open. Current main and v2026.5.18 still provide deterministic exec security/ask modes, allowlists, narrow safeBins, strict inline-eval checks, and display-only command highlighting, but not the requested built-in L1-L4 exec risk classifier; related denylist/tool-classification issues are adjacent rather than a merged or clearly superseding fix.

## What This Changes

The issue requests a built-in exec command risk classifier that can auto-allow safe commands, escalate ambiguous commands, and auto-deny dangerous commands.

## Best Possible Solution

Design an opt-in, parser-backed exec policy surface that coordinates argv-shape auto-allow and hard-deny rules with existing allowlist, safeBins, durable approvals, strictInlineEval, docs, UI, audit, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection give high-confidence evidence that current main lacks the requested risk-classifier policy surface, while the discussion frames this as a new security/product capability.

## Solution Assessment

Unclear as proposed. The L1-L4 goal addresses real approval fatigue, but a command-text or binary-name classifier is not the safest maintainable design; parsed argv-shape policy with opt-in semantics is safer.
