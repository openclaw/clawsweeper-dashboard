- `P2`: This is a concrete security and approval-ergonomics improvement, but it is a new policy capability requiring product/security design rather than an urgent regression fix.
- `impact:data-loss`: The requested hard-deny class targets destructive commands such as rm -rf, mkfs, dd, and fork bombs that can cause irreversible host damage.
- `impact:security`: The request changes the host exec approval boundary, including when commands are auto-allowed, prompted, escalated, or blocked.

## Summary

Keep open. Current main still has deterministic exec security/ask modes, allowlists, safeBins, durable approvals, strict inline-eval checks, and display-only command highlighting, but not the requested built-in L1-L4 exec command risk classifier; the older LLM-assisted approval issue was closed as duplicate to this canonical tracker, while broader tool-confirmation/safe-mode issues do not supersede this narrower exec approval design.

## What This Changes

Add a built-in L1-L4 command risk classifier to exec approvals that auto-allows safe commands, prompts or escalates ambiguous commands, and hard-denies dangerous commands.

## Best Possible Solution

Design an opt-in, parser-backed exec policy surface that coordinates argv-shape auto-allow and hard-deny rules with existing allowlist, safeBins, durable approvals, strictInlineEval, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection give high-confidence evidence that current main lacks the requested risk-classifier policy surface, while related comments show this is a new security/product capability.

## Solution Assessment

Unclear as proposed. The L1-L4 goal addresses real approval fatigue, but a command-text or binary-name classifier is not the safest maintainable design; parsed argv-shape policy with opt-in semantics is safer.
