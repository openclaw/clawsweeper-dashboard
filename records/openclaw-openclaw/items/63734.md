## Summary

Keep this open. Current main still does not implement the requested built-in L1-L4 exec risk classifier or a dangerous-command auto-deny path; exec approval behavior remains policy, allowlist, durable approvals, strict inline-eval checks, and a narrow stdin-only safe-bin fast path. The later discussion is useful but reinforces that this is a security-boundary design question rather than conservative cleanup.

## What This Changes

Request to add a built-in L1-L4 command risk classifier to exec approvals, auto-allowing safe commands and auto-denying dangerous commands.

## Best Possible Solution

Keep this as an explicit maintainer/security design issue. A maintainable end state would define the threat model first, then implement either a core opt-in exec policy layer based on parsed argv/shell structure or a documented plugin policy seam that can return allow, require approval, or deny for well-defined command shapes, with docs and regression tests for gateway and node hosts.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source/docs/test inspection on current main, which shows allowlist/safe-bin/strict-inline-eval behavior rather than a built-in L1-L4 classifier.

## Solution Assessment

Unclear. The requested static L1-L4 classifier is not proven to be the best design because auto-approval and auto-denial change a security boundary; a parsed, opt-in policy layer or reviewed plugin policy seam is safer than command-name classification.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes changing when host exec commands are automatically allowed or denied.

Concerns:

- **[medium] Heuristic auto-approval can weaken exec boundaries:**
  - body: Classifying broad commands such as cat, grep, curl, sed, awk, xargs, find, or git from command text alone can miss shell composition, file operands, redirects, wrappers, PATH hijacking, aliases, and platform parsing differences. Maintainer/security design should come before implementation.
