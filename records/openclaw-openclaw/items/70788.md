## Summary

Keep open. Current main still generates the Windows Startup-folder launcher as a minimized `cmd.exe` wrapper, writes only the `.cmd` login item, and keeps focused tests asserting that shape; the related Microsoft tracker is an open checklist, not an implementation.

## What This Changes

Request to suppress the Windows Startup-folder gateway login-item console flash by delegating the generated launcher through a hidden no-window mechanism such as a `wscript.exe`/VBScript sidecar and cleaning it up on uninstall.

## Best Possible Solution

Implement the fix in the Windows daemon Startup-folder fallback: preserve the existing `.cmd` login-item path for service detection, route login startup through a hidden no-window launcher with robust quoting, remove any sidecar on uninstall, and update focused tests plus the user-facing changelog.

Yes. A high-confidence source-level reproduction exists by forcing the `schtasks` fallback path and inspecting the generated Startup-folder script; current focused tests already assert the minimized `cmd.exe` launcher that the report identifies as flashing.
