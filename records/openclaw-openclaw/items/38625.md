## Summary

Keep open. Current main still does not provide the requested Crab-facing HTTP API: health probes remain shallow liveness/readiness JSON without a version field, and update orchestration is only exposed through the admin-scoped WebSocket RPC surface.

## What This Changes

The issue requests a `version` field in gateway HTTP health responses and an authenticated `POST /admin/update` endpoint for fleet update orchestration.

## Best Possible Solution

Add a config-independent `version` field to `/health` and `/healthz` using the existing runtime version resolver, then make an explicit product decision for updates: either document `update.run`/`update.status` as the supported fleet integration path or add a narrowly authenticated, rate-limited HTTP wrapper that reuses the shared update-runner/protocol schema. If version pinning is accepted, extend the shared update contract rather than making it HTTP-only.

## Reproduction Assessment

Yes. Source inspection and current tests reproduce the missing health version field, and route/schema searches reproduce the absence of an HTTP admin update endpoint or target-version parameter.

## Solution Assessment

Unclear. The health `version` field is a narrow, maintainable addition, but the proposed `POST /admin/update` shape is not clearly the best solution until maintainers decide whether REST should wrap the existing WS update RPC and how target-version pinning should work.
