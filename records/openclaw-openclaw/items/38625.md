## Summary

Keep open. Current main still only partially overlaps the fleet-management request: version is available through authenticated status/RPC surfaces, but the shallow `GET /health` payload still has no version, Admin HTTP RPC exposes `update.status` but not `update.run`, and the requested target-version HTTP update contract does not exist.

## What This Changes

Request adding a `version` field to `GET /health` and a gateway-token-authenticated `POST /admin/update` endpoint for fleet update orchestration.

## Best Possible Solution

Design an additive fleet/admin API that separates low-risk version discovery from privileged update triggering, reuses existing status/Admin HTTP RPC/update primitives, and defines auth scope, target-version semantics, restart behavior, audit/rate limits, response redaction, and interaction with the external-updater direction.

## Solution Assessment

No: adding `POST /admin/update` exactly as requested is not yet the best conservative fix. The safer path is a maintainer-designed fleet/admin contract that defines the privileged update boundary before exposing update triggering through HTTP.
