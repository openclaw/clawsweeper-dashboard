## Summary

Keep open. Current main only partially overlaps the request: runtime version is available on status and update.run exists on the admin-scoped Gateway control plane, but HTTP /health still returns shallow liveness, admin HTTP RPC only exposes update.status, and target_version is not part of the update contract.

## What This Changes

Request to expose the runtime version in GET /health and add an authenticated POST /admin/update endpoint with optional target_version for fleet updates.

## Best Possible Solution

Design an additive Gateway fleet/admin API that reuses existing status, update.run, and admin HTTP RPC primitives with explicit auth scope, rate/audit behavior, target-version validation, restart coalescing, and response redaction before adding routes.

## Reproduction Assessment

Not applicable: this is a new Gateway API feature request, not broken existing behavior. Source inspection shows the requested HTTP /health version field, HTTP update trigger, and target_version contract are still absent on current main.

## Solution Assessment

No: adding POST /admin/update exactly as requested is not the best conservative fix yet. The maintainable path is a maintainer-designed admin/fleet API that deliberately widens privileged update access rather than exposing update.run through a one-off route.

## Security Review

Status: needs_attention

Summary: The requested HTTP update trigger is security-sensitive because it would expose package-manager update and restart control through a network admin surface.
