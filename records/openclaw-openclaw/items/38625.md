## Summary

Keep open. Current main only partially overlaps the fleet-management request: shallow `GET /health` still has no version field, the bundled Admin HTTP RPC plugin exposes `POST /api/v1/admin/rpc` with `update.status` but not update execution, and `update.run` has no `target_version` contract.

## What This Changes

Request adding a runtime version field to gateway health responses and a token-authenticated HTTP admin update endpoint with optional target-version support for fleet orchestration.

## Best Possible Solution

Design an additive fleet/admin API that separates low-risk version discovery from privileged update triggering, reuses existing status/Admin HTTP RPC/update primitives, and defines auth scope, target-version semantics, restart behavior, audit/rate limits, response redaction, and external-updater alignment.

## Reproduction Assessment

Not applicable: this is a new gateway/admin API feature request, not a broken existing behavior report. Source inspection gives high confidence that current main lacks the `/health` version field, `/admin/update` route, and `target_version` update contract.

## Solution Assessment

No: adding `POST /admin/update` exactly as requested is not yet the best conservative fix. The safer path is a maintainer-designed fleet/admin contract that decides the privileged HTTP update boundary before exposing update execution.
