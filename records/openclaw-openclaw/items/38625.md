## Summary

Keep open. Current main still lacks both requested Crab-facing HTTP additions: shallow health probes do not expose a version field, and update orchestration exists on the admin-scoped WebSocket RPC surface rather than `POST /admin/update`.

## What This Changes

Request to add gateway version metadata to `GET /health` and a token-authenticated `POST /admin/update` endpoint with optional `target_version` for Crab fleet management.

## Best Possible Solution

Split the request: consider adding version to the shallow health probe, then decide whether Crab should use documented `update.run` / `update.status` or a REST wrapper with equivalent admin scoping, rate limiting, audit logging, restart safeguards, and strict version validation.

## Reproduction Assessment

Yes. Source inspection on current main shows `/health` and `/healthz` serialize only `{ ok, status }`, there is no `POST /admin/update` route, and `UpdateRunParamsSchema` rejects `target_version`.

## Solution Assessment

Unclear. The health version field is a narrow additive contract, but the REST update endpoint is not obviously the best path because the supported update contract is currently WebSocket RPC and any HTTP wrapper needs security/API approval.
