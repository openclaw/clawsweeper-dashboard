## Summary

Keep open: current main and v2026.6.10 still return shallow health JSON without version metadata, and the default-off Admin HTTP RPC plugin exposes `update.status` but not `update.run`. The version-only half overlaps a narrower open issue, while HTTP-triggered update authority remains a security-sensitive product/API decision.

## What This Changes

Request adds runtime version metadata to the Gateway health response and an authenticated HTTP admin update trigger for fleet tooling.

## Best Possible Solution

Let https://github.com/openclaw/openclaw/issues/91795 carry the narrow version metadata contract, and separately decide whether the default-off Admin HTTP RPC plugin or another authenticated HTTP surface may expose `update.run` with explicit security docs and tests.

## Reproduction Assessment

Yes, at source level. Current main’s `/health` and `/healthz` serializer returns only `{ ok: true, status }`, and Admin HTTP RPC allowlists `update.status` while omitting the existing `update.run` RPC.

## Solution Assessment

No, not as one automatic combined change. Version reporting is a narrow additive API contract, but HTTP-triggered updates should reuse or wrap `update.run` only after maintainers approve the security boundary and public API shape.
