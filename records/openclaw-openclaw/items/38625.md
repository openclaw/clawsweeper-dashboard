## Summary

Keep open: the public health version field is still missing and partly overlaps the narrower open version-metadata issue, while HTTP-triggered update authority remains a security-sensitive Gateway product decision rather than a safe automatic fix.

## What This Changes

The issue requests two Gateway API additions: version metadata in the public health response and an authenticated HTTP update endpoint for fleet tooling.

## Best Possible Solution

Keep this combined tracker open for maintainer direction: let the narrower version issue carry the public health/version field decision, then decide whether HTTP update orchestration belongs in Admin HTTP RPC or a dedicated endpoint with explicit security docs and tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 still serialize public health as only `{ ok: true, status }`, and Admin HTTP RPC still allowlists `update.status` but not `update.run`. I did not start a live gateway in this read-only review.

## Solution Assessment

No, not as an automatic combined change or a new ad hoc `/admin/update` endpoint. The maintainable path is to reuse the existing version resolver and `update.run` control-plane semantics only after maintainers choose the public HTTP contract and security boundary.
