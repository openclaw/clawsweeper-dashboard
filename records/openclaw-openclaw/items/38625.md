## Summary

Keep open. Current main still does not provide the requested Crab-facing HTTP API: `/health` and `/healthz` return shallow liveness JSON without `version`, and there is no `POST /admin/update` route. The existing admin-scoped WS `update.run`/`update.status` path is relevant partial overlap, but it is not the requested REST surface and its schema still rejects target-version fields.

## What This Changes

Request adding a `version` field to `GET /health` and an authenticated `POST /admin/update` endpoint for Crab fleet monitoring and update orchestration.

## Best Possible Solution

Add a config-independent runtime `version` field to `/health` and `/healthz` if maintainers accept that public liveness contract change. For updates, either document `update.run`/`update.status` as the supported fleet integration path or add an authenticated admin HTTP wrapper that reuses the shared update runner and protocol schema; target-version pinning should be added to the shared update contract if accepted, not only to HTTP.
