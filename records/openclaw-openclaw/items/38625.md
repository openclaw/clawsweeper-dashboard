## Summary

Keep open. Current main still only partially overlaps the request: runtime version is available through status-oriented surfaces and Admin HTTP RPC exists, but the HTTP health probe remains shallow, update execution is not exposed through Admin HTTP RPC, and target-version update semantics are absent.

## What This Changes

Add a version field to the HTTP health response and add an authenticated admin update endpoint with optional target-version pinning for fleet tooling.

## Best Possible Solution

Design an additive fleet/admin API that separates low-risk version discovery from privileged update triggering, reuses existing status, Admin HTTP RPC, and update primitives, and defines auth scope, target-version semantics, restart behavior, audit/rate limits, response redaction, and interaction with the external-updater direction.

Not applicable: this is a new gateway/admin API feature request, not a broken existing behavior report. Source inspection shows the requested `/health` version field, `/admin/update` route, and `target_version` update contract are still absent on current main.

## Solution Assessment

No: adding `POST /admin/update` exactly as requested is not yet the best conservative fix. The safer path is a maintainer-designed fleet/admin contract that defines the privileged update boundary before exposing update triggering through HTTP.
