## Summary

Keep open. Current main only partially overlaps the request: version data exists on gateway status and an opt-in admin HTTP RPC plugin exists, but HTTP /health still returns shallow liveness, the admin HTTP allowlist excludes update.run, and target_version is not part of the update contract.

## What This Changes

Request to add a gateway version field to GET /health and an authenticated POST /admin/update endpoint with optional target-version update orchestration.

## Best Possible Solution

Design an additive gateway fleet/admin API that reuses the existing status, admin HTTP RPC, and update.run primitives with explicit authorization, target-version semantics, restart handling, audit/rate behavior, and redacted response shape before exposing update triggering over HTTP.

## Reproduction Assessment

Not applicable: this is a new gateway API feature request, not broken existing behavior. Source inspection shows the requested HTTP health version field, HTTP update trigger, and target_version contract are still absent on current main.

## Solution Assessment

No: adding POST /admin/update exactly as requested is not yet the best conservative change. The safer path is a maintainer-designed admin/fleet API that deliberately defines the privileged update boundary before exposing update.run through HTTP.
