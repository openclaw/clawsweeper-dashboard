## Summary

Keep open. Current main still lacks both requested Crab-facing HTTP additions: shallow gateway health probes return only `ok` and `status`, while update orchestration remains on the admin-scoped WebSocket RPC surface rather than a `POST /admin/update` route.

## What This Changes

Request to expose the OpenClaw version in `GET /health` and add an authenticated `POST /admin/update` REST endpoint with optional `target_version` for Crab fleet updates.

## Best Possible Solution

Split the request: add the health version as an additive probe contract if maintainers approve it, and separately decide whether Crab should use documented `update.run`/`update.status` or a REST wrapper with equivalent admin scoping and safety controls.

## Reproduction Assessment

Yes, for source-level verification. Current main shows `/health` and `/healthz` serialize only `{ ok, status }`, lacks a `POST /admin/update` route, and rejects `target_version` in the existing update RPC schema.

## Solution Assessment

Unclear. Adding a version field to the shallow health probe is narrow and additive, but adding a REST update endpoint is security-sensitive and may not be the best path while the supported update contract is WebSocket RPC.
