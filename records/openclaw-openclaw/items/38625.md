## Summary

Keep open. Current main still lacks both requested Crab-facing HTTP additions: the shallow health probe has no version field, and update orchestration exists only on the admin-scoped WebSocket RPC surface.

## What This Changes

The issue requests adding `version` to the gateway health response and adding an authenticated `POST /admin/update` endpoint with optional version targeting.

## Best Possible Solution

Expose the gateway version on the shallow health probe if maintainers accept that additive contract, then decide whether Crab should use documented `update.run` / `update.status` or a REST wrapper with equivalent authorization, rate limiting, audit logging, restart safeguards, and target-version validation.

## Reproduction Assessment

Yes. Source inspection on current `origin/main` shows `/health` and `/healthz` return only `{ ok, status }`, there is no `POST /admin/update` route, and the update schema rejects `target_version`.

## Solution Assessment

Unclear. The health version field is narrow, but the update endpoint needs maintainer API/security direction because the existing documented update path is WebSocket RPC rather than REST.
