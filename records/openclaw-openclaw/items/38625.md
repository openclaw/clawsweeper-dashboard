## Summary

Keep open. Current main and the latest release still lack both requested Crab-facing HTTP additions: shallow gateway health probes do not expose a version field, and update orchestration remains on the admin WebSocket RPC surface rather than `POST /admin/update`.

## What This Changes

The issue asks to add an OpenClaw version field to `GET /health` and add a token-authenticated HTTP `POST /admin/update` endpoint for Crab fleet update orchestration.

## Best Possible Solution

Split the request: add the health version as an additive probe contract if maintainers approve it, and separately decide whether Crab should use documented `update.run` / `update.status` or a REST wrapper with equivalent admin scoping, rate limiting, audit logging, restart safeguards, and strict target-version validation.

## Reproduction Assessment

Yes for source-level verification. Current main shows `/health` and `/healthz` serialize only `{ ok, status }`, lacks a `POST /admin/update` route, and rejects `target_version` in the existing update RPC schema.

## Solution Assessment

Unclear. Adding a version field to the shallow health probe is a narrow additive change, but adding a REST update endpoint is security-sensitive and may not be the best path while the supported update contract is WebSocket RPC.
