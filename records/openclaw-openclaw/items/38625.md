## Summary

Keep open: current main still lacks version metadata on the HTTP and WS health payloads, while update execution already exists as `update.run` and is not exposed through the Admin HTTP RPC allowlist. The combined request remains useful, but it needs maintainer security/product direction before the update write crosses the HTTP boundary.

## What This Changes

The issue requests version metadata in gateway health responses plus an authenticated HTTP admin endpoint for triggering OpenClaw updates.

## Best Possible Solution

Split the scope: add a documented running-version health surface after maintainers choose the public field set, and decide separately whether Admin HTTP RPC may expose the existing `update.run` control-plane write with explicit security tests and policy.

## Reproduction Assessment

Yes at source level: current `/health` and `/healthz` return only shallow liveness data, the WS `HealthSummary` lacks version, and Admin HTTP RPC excludes `update.run`. I did not run a live gateway because the current source, docs, tests, and latest release tag establish the capability gap directly.

## Solution Assessment

No, not as one combined endpoint change. Version reporting is a narrow additive contract, but update execution should reuse canonical `update.run` and needs a separate maintainer-approved HTTP security-boundary decision.
