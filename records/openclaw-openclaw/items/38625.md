## Summary

Keep open. Current main still lacks the requested Crab-facing REST surface: the HTTP health probe has no version field, and gateway self-update is available only through the admin-scoped WebSocket `update.run`/`update.status` contract.

## What This Changes

Request adding an OpenClaw gateway version field to HTTP health responses and an authenticated `POST /admin/update` endpoint for gateway self-updates.

## Best Possible Solution

Expose a version field on the lightweight health probe only if maintainers accept that public API shape, then decide whether fleet updates should use the documented WebSocket RPC or a narrowly authenticated REST wrapper over the same update runner and safeguards.

## Reproduction Assessment

Yes for the observable API gap. Source inspection shows current main serializes `/health` and `/healthz` without `version`, has no `POST /admin/update` route, and rejects `target_version` through the current update schema.

## Solution Assessment

Unclear. The health field is a narrow additive change, but the update endpoint is not clearly the best solution until maintainers choose between documenting the existing WebSocket update RPC and adding an HTTP wrapper with equivalent safeguards.
