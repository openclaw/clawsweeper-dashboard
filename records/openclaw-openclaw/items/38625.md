## Summary

Keep open. Current main still lacks the requested Crab-facing HTTP additions: the health probe has no version field, and update orchestration is only exposed through the admin-scoped WebSocket RPC surface.

## What This Changes

Request to add a `version` field to gateway HTTP health responses and an authenticated `POST /admin/update` REST endpoint for fleet update orchestration.

## Best Possible Solution

Expose the gateway version on the shallow HTTP health probe if maintainers accept that compatibility change, then decide whether Crab should use documented WS `update.run`/`update.status` or a REST wrapper with the same admin, audit, and rate-limit safeguards.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence verification path: current main serializes `/health` and `/healthz` without `version`, has no `POST /admin/update` route, and rejects `target_version` through the current update schema.

## Solution Assessment

Unclear. The health version field is narrow, but the REST update endpoint is not clearly the best API until maintainers choose between documenting the existing WS RPC for Crab or adding an HTTP wrapper with equivalent safeguards.
