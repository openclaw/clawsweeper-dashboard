## Summary

Keep open: current main still lacks both requested Crab-facing HTTP additions, and the update endpoint half needs maintainer API/security direction before implementation.

## What This Changes

The issue requests adding the gateway version to the HTTP health response and adding an authenticated `POST /admin/update` endpoint with optional target-version selection.

## Best Possible Solution

Expose the gateway version on the shallow health probe if maintainers accept that API change, then either document Crab against the existing WS `update.run`/`update.status` surface or add a REST wrapper with equivalent safeguards.

## Reproduction Assessment

Yes. Source inspection on current main shows `/health` and `/healthz` return only `{ ok, status }`, there is no `POST /admin/update` route, and the current update-run schema has no `target_version` field.

## Solution Assessment

Unclear. The version field is a narrow additive health-probe change, but the update endpoint is not clearly the best API until maintainers choose between documenting the existing WS RPC for Crab or approving a REST wrapper with matching safeguards.
