## Summary

Keep open. Current main only partially overlaps the fleet-management request: version is available through authenticated status/RPC surfaces, but the shallow health probe still has no version field, Admin HTTP RPC exposes update status but not update execution, and the requested target-version HTTP update contract is absent.

## What This Changes

The issue requests adding a version field to the gateway health response and adding an authenticated HTTP admin update endpoint with optional target-version selection.

## Best Possible Solution

Design an additive fleet/admin API that separates low-risk version discovery from privileged update triggering, reuses existing status/Admin HTTP RPC/update primitives, and defines auth scope, target-version semantics, restart behavior, audit/rate limits, response redaction, and interaction with the external-updater direction.

## Reproduction Assessment

Not applicable: this is a new gateway/admin API feature request, not a broken existing behavior report. Source inspection shows the requested health version field, admin update route, and target-version update contract are still absent on current main.

## Solution Assessment

No: adding POST /admin/update exactly as requested is not yet the best conservative fix. The safer path is a maintainer-designed fleet/admin contract that defines the privileged update boundary before exposing update triggering through HTTP.
