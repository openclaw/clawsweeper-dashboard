## Summary

Keep open. Current main still lacks the requested Crab-facing HTTP API: `/health`/`/healthz` live probes do not include a version field, and update orchestration exists only as the admin-scoped WebSocket `update.run`/`update.status` surface.

## What This Changes

Request to expose the gateway runtime version in `GET /health` and add an authenticated `POST /admin/update` endpoint for fleet update orchestration.

## Best Possible Solution

Add a config-independent version field to the HTTP health probe, then make an explicit maintainer decision on whether fleet updates should use documented WS `update.run`/`update.status` or a narrowly authenticated HTTP wrapper that reuses the shared update schema and control-plane safeguards.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main serializes `/health` and `/healthz` without a version field, has no `POST /admin/update` route, and rejects `target_version` through the current update schema.

## Solution Assessment

Unclear. The health version field is a narrow maintainable addition, but the proposed REST update endpoint is not clearly the best solution until maintainers choose between documenting the existing WS RPC and adding an HTTP wrapper with equivalent safeguards.
