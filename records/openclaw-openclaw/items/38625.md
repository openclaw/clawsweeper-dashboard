## Summary

Keep open. Current main still lacks the requested `/health` version field and an HTTP update trigger with a `target_version` contract; the merged default-off admin HTTP RPC plugin only partially overlaps by exposing `health`, `status`, and `update.status`, while update triggering over HTTP remains a privileged API/security design decision.

## What This Changes

The issue requests adding runtime version to `GET /health` and adding an authenticated HTTP admin update endpoint with optional `target_version` for fleet orchestration.

## Best Possible Solution

Design a Gateway fleet-management/admin HTTP contract around existing update, status, and version primitives, specifying version exposure, update triggering, target-version policy, auth, audit, restart, and redaction semantics before implementation.

## Reproduction Assessment

Not applicable: this is a new Gateway API feature request, not broken existing behavior. Source inspection on current main shows the requested `/health` version field and HTTP update trigger are still absent.

## Solution Assessment

No, not as a direct one-off `POST /admin/update` route exactly as requested. The safer path is a maintainer-designed admin/fleet API that reuses existing updater logic and the default-off admin HTTP RPC security model without accidentally widening privileged access.
