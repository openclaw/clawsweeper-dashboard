## Summary

Keep open. Current main still only partially overlaps the request: version is available on status surfaces, and an opt-in admin HTTP RPC plugin exists, but HTTP `/health` remains shallow liveness, `update.run` is not exposed through the admin HTTP allowlist, and the requested `target_version` update contract is absent.

## What This Changes

Request to add a version field to `GET /health` and a token-authenticated HTTP admin update endpoint with optional target-version pinning.

## Best Possible Solution

Design an additive fleet/admin API that deliberately separates safe version discovery from privileged update triggering, reusing existing status, admin HTTP RPC, and update primitives with explicit authorization, target-version semantics, restart behavior, audit/rate limits, and redacted responses.

## Reproduction Assessment

Not applicable: this is a new gateway API feature request, not a broken existing behavior report. Source inspection shows the requested `/health` version field, `/admin/update` route, and `target_version` gateway update contract are still absent on current main.

## Solution Assessment

No: adding `POST /admin/update` exactly as requested is not yet the best conservative fix. The safer path is a maintainer-designed fleet/admin contract that defines the privileged update boundary before exposing `update.run` through HTTP.
