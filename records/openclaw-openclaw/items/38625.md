## Label Justifications

- `P2`: The health metadata is a bounded monitoring improvement, while HTTP update execution warrants normal-priority product and security review rather than emergency handling.
- `impact:security`: The requested HTTP update operation would grant a network credential authority to install executable code and restart the gateway service.
- `impact:other`: The absent health version field limits fleet monitoring and upgrade coordination.

## Summary

This remains useful, but the two requests need separate treatment: health still lacks a version field, while update execution already exists as the hardened `update.run` control-plane RPC and is deliberately excluded from the Admin HTTP allowlist. The HTTP mutation therefore needs an explicit maintainer security/product decision rather than a parallel updater implementation.

## What This Changes

Add the running OpenClaw version to gateway health output and allow fleet tools to initiate authenticated updates without SSH.

## Best Possible Solution

Split the scope: add a canonical running-version field to health if maintainers want that monitoring contract, and consider HTTP update execution separately by exposing the existing `update.run` only after an approved threat model, authorization policy, rate limits, audit behavior, and focused security tests.

## Reproduction Assessment

Yes at source level: current health construction demonstrably lacks version, and the Admin HTTP allowlist demonstrably excludes update execution. This is a clearly located feature gap rather than a regression in an existing contract.

## Solution Assessment

No, not as one combined endpoint change. Version reporting is a narrow additive contract, but update execution should remain on the existing canonical control plane and requires a separate approval before HTTP exposure.
