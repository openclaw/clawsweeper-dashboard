## Summary

Keep open. Current main only partially overlaps the request: gateway status can expose runtime version and the opt-in Admin HTTP RPC plugin exists, but `/health` remains a shallow liveness response, `update.run` is not allowlisted over Admin HTTP RPC, and the requested `target_version` update contract is absent.

## What This Changes

Request to add a gateway version field to `GET /health` and an authenticated HTTP update endpoint for fleet update orchestration.

## Best Possible Solution

Design an additive fleet/admin API that separates low-risk version discovery from privileged update triggering, reusing existing status, Admin HTTP RPC, and update primitives with explicit auth scope, target-version semantics, restart behavior, audit/rate limits, and redacted responses.

## Reproduction Assessment

Not applicable: this is a new gateway/admin API feature request, not a broken existing behavior report. Source inspection shows the requested `/health` version field, `/admin/update` route, and `target_version` gateway update contract are still absent on current main.

## Solution Assessment

No: adding `POST /admin/update` exactly as requested is not yet the best conservative fix. The safer path is a maintainer-designed fleet/admin contract that defines the privileged update boundary before exposing update triggering through HTTP.
