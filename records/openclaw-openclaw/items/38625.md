## Summary

Keep open. Current main partially overlaps the request because runtime version exists on status and update.run exists on the admin Gateway control plane, but /health still has shallow liveness output, HTTP admin RPC only allowlists update.status, and target_version is not in the update contract.

## What This Changes

The issue requests adding a version field to Gateway health responses and adding an authenticated POST /admin/update API for fleet update orchestration.

## Best Possible Solution

Design an additive Gateway fleet/admin API that reuses existing status, update.run, and admin HTTP RPC primitives with explicit auth scope, rate/audit behavior, target-version validation, restart coalescing, and response redaction before adding routes.

## Reproduction Assessment

Not applicable: this is a new Gateway API feature request, not broken existing behavior. Source inspection shows the requested HTTP /health version field, HTTP update trigger, and target_version contract are still absent on current main.

## Solution Assessment

No: adding POST /admin/update exactly as requested is not the best conservative fix yet. The maintainable path is a maintainer-designed admin/fleet API that deliberately widens privileged update access instead of exposing update.run through a one-off route.

Status: needs_attention

Summary: The requested HTTP update trigger is security-sensitive because it would expose package-manager update and restart control through a network admin surface.

  - confidence: 0.93
