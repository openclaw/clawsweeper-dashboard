## Summary

Keep open. Current main and the latest release still route unavailable exec-approval replies through generic Web UI / terminal UI copy, and the linked implementation PR was closed unmerged. The request remains narrow and actionable because the dashboard and approvals CLI surfaces already exist.

## What This Changes

The issue requests that exec approval timeout or unavailable messages include concrete Control UI, approval inspection, and channel setup guidance.

## Best Possible Solution

Update the shared exec approval unavailable reply so no-route, disabled, and unsupported paths give concrete recovery steps: open the dashboard through `openclaw dashboard --no-open`, inspect effective policy with `openclaw approvals get` or its alias, and enable the appropriate channel/account-scoped `execApprovals.enabled` path only when that channel supports native approvals.

## Reproduction Assessment

Yes. A source-level reproduction is to call `buildExecApprovalUnavailableReplyPayload()` for `no-approval-route` or `initiating-platform-disabled`; current main returns generic Web UI / terminal UI text and the focused tests cover those branches without the requested recovery hints.

## Solution Assessment

Yes. The narrow maintainable solution is a shared helper and test update, preserving channel-specific setup guidance while adding token-safe dashboard and approvals CLI recovery steps.
