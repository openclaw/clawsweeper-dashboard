## Summary

Keep open. Current main and the latest release have opt-in rolling tool-loop detection plus a default post-compaction guard, but they still do not provide the requested default same-turn, first-class `read` path-set/reread-ratio nudge before a run burns tokens and stalls.

## What This Changes

The issue proposes a warning-only same-turn reread loop guard for the first-class `read` tool to reduce repeated file reads and token burn.

## Best Possible Solution

Extend the existing tool-loop detection path with a lightweight read-aware default warning for one run/turn that recognizes repeated paths or small path sets without nearby edit/write/test progress, while keeping hard blocking and broader heuristics behind explicit maintainer policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live replay, but source inspection confirms current main lacks a default first-class `read` reread nudge; the only rolling detector that catches same-args `read` repeats is disabled by default.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide the policy, but the maintainable direction is to reuse the existing loop-detection and before-tool-call infrastructure rather than add a separate read-tool-only subsystem.
