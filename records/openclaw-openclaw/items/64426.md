## Summary

Keep open. Current main has a generic opt-in tool-loop detector for first-class tools, but it is disabled by default and keys generic read repeats by identical tool+params; warning-level hits are diagnostic/log events rather than the requested in-turn read nudge, so the central read path-set/reread-ratio guard still needs maintainer product review.

## What This Changes

The issue proposes a warning-first same-turn guard for repeated first-class `read` calls to reduce token burn and stalled runs.

## Best Possible Solution

Extend the existing tool-loop detector and before-tool hook with a narrow, warning-first same-run read-repeat signal that uses existing config and diagnostic surfaces, with focused tests for legitimate rereads and loop escapes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live transcript replay, but source inspection shows current main would not intervene for the reported no-config first-class read loop because rolling detection is disabled by default and the generic detector only matches exact params.

## Solution Assessment

Is this the best way to solve the issue? The requested direction is plausible, but the maintainable path is to extend the existing loop-detection hook rather than add a parallel read-only guard; thresholds, default behavior, and warning visibility need maintainer decision first.
