## Summary

Keep open. Current main has adjacent generic tool-loop protection, but it is opt-in for the rolling detector and still does not cover the requested first-class read-tool small-file-set/reread-ratio guard with no edit/test progress.

## What This Changes

The issue proposes a same-turn soft guard for repeated first-class `read` tool rereads that burn tokens without edit/test progress.

## Best Possible Solution

Define a maintainer-approved read-loop policy, then extend the existing detector to recognize first-class `read` reread or small-file-set patterns with focused tests and docs while coordinating semantics with https://github.com/openclaw/openclaw/issues/63352.

## Reproduction Assessment

Yes, at source level. Current main still leaves rolling loop detection off by default and only covers exact repeated `read` arguments or two-pattern ping-pong when enabled, not the reported small-file-set reread ratio with no edit/test progress.

## Solution Assessment

Unclear. A read-specific soft guard is plausible, but the best implementation depends on maintainer choices about default behavior, tuning, warning delivery, and alignment with the exec-side proposal.
