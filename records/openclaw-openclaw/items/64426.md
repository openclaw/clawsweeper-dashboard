## Summary

Keep open: current main has a generic opt-in tool-loop detector that can observe repeated `read` calls, but it does not provide the requested default same-turn first-class read reread nudge, and this issue remains the canonical read-tool proposal after related duplicate cleanup.

## What This Changes

The issue proposes a warning-only guard for repeated same-turn first-class `read` tool calls that burn tokens without progress.

## Best Possible Solution

Extend the existing before-tool-call loop-detection path with a narrowly scoped, low-friction first-class `read` reread nudge only after maintainers choose the default behavior and thresholds.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection shows current main can execute repeated `read` calls and only warns through opt-in generic loop detection, but I did not replay the exact webchat transcript loop.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide the UX policy; the existing before-tool-call loop detector is the right technical seam, but read-specific defaults and thresholds should be chosen deliberately.
