## Summary

Keep open: current main and the latest release have a generic tool-loop detector that can observe repeated `read` calls only when rolling loop detection is enabled, but they still do not provide the requested default same-turn first-class read reread nudge. This remains the canonical read-tool proposal after the broader duplicate was closed into this issue and the exec-reread sibling closed stale.

## What This Changes

The issue requests a warning-only same-turn reread-loop guard for the first-class `read` tool to reduce token and quota burn.

## Best Possible Solution

Extend the existing before-tool-call loop-detection path with a narrowly scoped, low-friction first-class `read` reread nudge after maintainers choose the default behavior and thresholds.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection does show current main can execute repeated first-class `read` calls without a default read-specific guard, but the exact webchat loop was not replayed.

## Solution Assessment

Unclear until maintainers decide the UX policy. The existing before-tool-call detector is the right technical seam, but read-specific defaults, warning text, and thresholds should be chosen deliberately.
