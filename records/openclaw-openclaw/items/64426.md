## Summary

Keep open. Current main has a generic opt-in tool-loop detector, but it does not yet provide the requested first-class `read` reread guard for the reported pattern: the guard is disabled by default, warning outcomes are diagnostic-only, and the generic detector only counts identical tool+args or narrow ping-pong patterns rather than reread ratio across a small file set with no edit/test progress.

## What This Changes

Request to add a warning-only same-turn reread loop guard for first-class `read` tool calls to reduce token burn and stalls.

## Best Possible Solution

Add a shared loop-safety design that covers both first-class `read` calls and exec-based readbacks, with read-aware path-set/reread-ratio heuristics, a transcript-visible soft nudge or synthetic warning result, focused tests, and updated loop-detection docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with default config, loop detection is off, and with it enabled the current read coverage only catches identical-args thresholds or generic ping-pong cases, not the reported multi-file reread loop before the turn burns context.

## Solution Assessment

No. The current generic loop detector is useful adjacent infrastructure, but it is not the best complete solution for this issue because it is opt-in, diagnostics-only at warning level, and not specific to read reread patterns or nearby progress signals.
