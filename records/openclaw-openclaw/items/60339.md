## Summary

Keep open. Current main still drops `parsed.offloadedRefs` on the iOS share/node `agent.request` path before transcript media persistence, and the direct closing PR is closed unmerged.

## What This Changes

The issue reports that large-image `offloadedRefs` from iOS share or node ingress are not written into transcript `MediaPath`/`MediaPaths`, making attachments unavailable on history replay.

## Best Possible Solution

Share the existing `chat.send` media persistence/transcript rewrite contract with the node `agent.request` path and add focused node-events regression coverage for offloaded refs becoming `MediaPath`/`MediaPaths`.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to follow an iOS share or node `agent.request` with a large image through `parseMessageWithAttachments`: current main returns `offloadedRefs`, then `server-node-events.ts` drops them before any transcript media rewrite.

## Solution Assessment

Yes. Reusing the existing `chat.send` persistence and transcript rewrite helpers is the narrow maintainable fix; duplicating a node-only transcript writer would add risk around the session transcript contract.
