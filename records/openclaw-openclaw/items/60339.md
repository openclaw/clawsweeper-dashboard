## Label Justifications

- `P1`: The issue reports a current regression where iOS/node attachment media is lost from session history, breaking follow-up agent context for affected users.
- `impact:data-loss`: The transcript loses persisted media reference metadata even though the attachment was saved, making history replay lose user-provided media.
- `impact:session-state`: The bug corrupts the session transcript state by omitting media fields needed for multi-turn context.

## Summary

Keep open. Current main still has the reported gateway transcript bug: the iOS share/node `agent.request` path preserves `message`, `images`, and `imageOrder` after attachment parsing but does not carry `offloadedRefs` into transcript media persistence, and the direct fix PR is closed unmerged.

## What This Changes

Persist offloaded attachment references from iOS share/node `agent.request` ingress into transcript `MediaPath`/`MediaPaths` fields.

## Best Possible Solution

Land a narrow gateway fix that carries parsed offloaded image refs through the node `agent.request` path using the existing `chat.send` media persistence/transcript rewrite contract, with regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction follows iOS Share or node ingress into `node.event`/`agent.request`, where `parseMessageWithAttachments` returns `offloadedRefs` but `server-node-events.ts` drops them before transcript media persistence.

## Solution Assessment

Yes. Reusing or extracting the existing `chat.send` media persistence and transcript rewrite helpers is the narrow maintainable fix; raw transcript appends would be riskier for the session transcript contract.
