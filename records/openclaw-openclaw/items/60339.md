- `impact:data-loss`: The persisted transcript can lose the media metadata needed to recover an offloaded attachment later.
- `impact:message-loss`: Later agent turns may not receive the media references from earlier node/iOS-share messages.

## Summary

Keep open. Current main still drops `parsed.offloadedRefs` on the iOS share/node `agent.request` path before transcript `MediaPath`/`MediaPaths` can be written, and the direct fix PR is closed unmerged.

## What This Changes

The issue asks for offloaded attachment refs from iOS share/node ingress to be preserved in JSONL transcript media fields.

## Best Possible Solution

Reuse or extract the existing `chat.send` media persistence and transcript rewrite contract for node `agent.request`, avoiding raw JSONL appends and locking the behavior with a focused regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send an offloaded image through `node.event`/`agent.request`, and `parseMessageWithAttachments` returns `offloadedRefs` that `server-node-events.ts` never carries into transcript media persistence. I did not run a live iOS/WSL transcript flow in this read-only review.

## Solution Assessment

Yes, the maintainable fix is to share the existing `chat.send` saved-media and transcript rewrite contract with the node ingress path, plus regression coverage. The earlier unmerged PR is directionally useful but should be rebased against current main rather than applied blindly.
