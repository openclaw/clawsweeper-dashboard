## Label Justifications

- `P1`: The bug drops attachment metadata from persisted history for a real iOS/node ingress workflow, breaking later agent access to previously sent media.
- `impact:data-loss`: Offloaded attachment references are lost from the JSONL transcript even though the media was saved.
- `impact:message-loss`: The user’s media payload is not recoverable from later message history on the affected path.
- `impact:session-state`: Session transcript state drifts from the actual user turn, so multi-turn replay cannot reconstruct the image context.

## Summary

Keep open. Current main and the latest release still parse and offload `agent.request` attachments but only carry `message`, `images`, and `imageOrder` into dispatch, so `offloadedRefs` never reach the transcript media rewrite; the direct fix PR is closed unmerged.

## What This Changes

The issue asks for offloaded media refs from iOS share/node `agent.request` ingress to persist into transcript `MediaPath`/`MediaPaths` fields.

## Best Possible Solution

Share or extract the existing `chat.send` media persistence/transcript rewrite contract for the node `agent.request` path and add focused regression coverage for offloaded refs becoming transcript media fields.

## Reproduction Assessment

Yes, at source level: follow iOS share or node `agent.request` with a large image through `parseMessageWithAttachments`; current main returns `offloadedRefs`, then `server-node-events.ts` drops them before transcript media persistence. I did not run a live iOS/WSL transcript scenario in this read-only review.

## Solution Assessment

Yes. Reusing or extracting the existing `chat.send` media persistence and transcript rewrite helpers is the narrow maintainable fix; duplicating raw transcript writes would create a second contract.
