## Summary

Keep open; the Discord proxy alignment is useful, and the earlier NO_PROXY WebSocket review gap appears addressed, but the latest voice-upload follow-up bypasses the existing SSRF-guarded upload URL request while sending the bot token.

## What This Changes

The PR adds a Discord undici transport, wires REST, webhook, monitor, gateway, and voice paths into explicit/env/OPENCLAW proxy handling, allows remote Discord proxies, updates tests, and adds a changelog entry.

## Best Possible Solution

Keep the Discord-owned transport fix, but route voice upload URL requests through the guarded fetch path with the selected dispatcher policy before landing with focused Discord proxy and voice tests plus changed-gate proof.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original proxy bug via remote Discord proxy rejection, and the PR regression is source-reproducible because sendVoiceMessageDiscord creates a transport unconditionally before requestVoiceUploadUrl calls transport.fetch directly.

## Solution Assessment

No, not yet. The Discord-local transport approach is the right shape and the NO_PROXY follow-up is covered, but the voice upload URL request must preserve SSRF guard and redirect/header safety while using proxy transport.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep the voice upload URL guard in the proxy path:** `extensions/discord/src/voice-message.ts:314-315`
  - body: sendVoiceMessageDiscord now always constructs a DiscordTransport, so this branch is taken even when no proxy is configured. That skips the existing fetchWithSsrFGuard checks around rest.options.baseUrl while sending Authorization; an injected or custom rest base URL can receive the bot token or reach internal hosts. Use the guarded fetch path with the chosen dispatcher policy/fetch implementation instead of calling transport.fetch directly.
  - confidence: 0.84
