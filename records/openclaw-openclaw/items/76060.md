## Summary

Keep open. The PR targets a real Discord proxy gap on current main, but it is not merge-ready because the new env/OPENCLAW_PROXY_URL transport is tested only through the new helper and remains bypassed by the production Discord fetch call sites; it also lacks the required changelog entry.

## What This Changes

The PR adds a Discord-specific fetch transport that uses undici dispatchers for direct, environment, and explicit proxy cases, removes the loopback-only Discord proxy restriction, and updates Discord proxy tests.

## Best Possible Solution

Land a revised Discord-owned transport that routes explicit, env, and managed proxy settings through the actual REST, webhook, and gateway call sites while preserving custom fetches and documenting the user-facing fix.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on main, remote Discord proxy URLs fail `validateDiscordProxyUrl`, and on the PR head the managed proxy path is only exercised by direct helper tests rather than the production account/rest fetch resolution path.

## Solution Assessment

No. The dispatcher helper direction matches Telegram, but the PR should first wire it into production Discord fetch resolution and add the required changelog entry.
