## Summary

This should stay open: current main narrows `compat.thinkingFormat` but has no doctor migration for the retired Qwen-style values, so an otherwise repairable config can still fail validation during `doctor --fix`.

## What This Changes

Request `doctor --fix` to migrate invalid legacy `models.providers.<id>.models[].compat.thinkingFormat` values after the allowed enum was narrowed.

## Best Possible Solution

Add a focused doctor config normalizer that removes or maps retired Qwen-style `compat.thinkingFormat` values across configured provider model entries, with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a config containing `models.providers.<id>.models[].compat.thinkingFormat` set to a retired value such as `qwen` or `qwen-chat-template` is rejected by the current schema, and doctor has no migration for that field before write-time validation.

## Solution Assessment

Yes. A narrow doctor migration in the existing legacy compatibility normalizer is the maintainable fix; widening the schema to keep retired values valid would preserve stale runtime config instead of migrating it.
