## Summary

Keep open. Current main still has the reported behavior shape: queue-mode steering is attempted only through a streaming-only gate, active embedded Pi handles reject non-streaming delivery, active `steer` falls back to followup, and Discord same-session messages are serialized before the reply runner can attempt steering.

## What This Changes

The issue asks OpenClaw to make `steer` queue mode deliver inbound Discord messages into active runs instead of silently falling back to followup, with diagnostics when steering is rejected.

## Best Possible Solution

Review and land #73365, #75226, or a focused replacement that lets eligible same-session messages reach active-run steering before the next model boundary, keeps intentional followup fallback, and emits clear diagnostics when steering is rejected.

## Reproduction Assessment

Yes. The source-level path is high confidence: Discord same-session work is serialized, `runReplyAgent` only attempts steering while `isStreaming` is true, and active `steer` falls through to the followup queue when injection is not accepted.

## Solution Assessment

Yes. The narrow maintainable fix is lifecycle-aware active-run steering plus rejection diagnostics, aligned to the current model-boundary docs rather than the older skipped-tool-call wording.
