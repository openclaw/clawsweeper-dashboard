## Summary

Keep open. Current main still has the reported behavior shape for Discord/main-session steering: Discord same-session inbound work is serialized before reply handling, the reply runner only attempts Pi steering when the active run is reported streaming, the embedded helper still rejects non-streaming active handles, and active `steer` still falls back to a followup queue entry when steering is not accepted. Related #48003 is the older root-cause report, but this issue keeps distinct Discord and diagnostic-fallback detail while #73365 and #75226 remain open.

## What This Changes

The issue requests that `messages.queue.mode: "steer"` deliver eligible inbound Discord/main-session messages into the active run instead of silently falling back to followup behavior, with diagnostics when steering is rejected.

## Best Possible Solution

Implement lifecycle-aware active-run steering so eligible same-session inbound messages can reach the embedded steering queue while the run is active, including the Discord session queue path; emit actionable diagnostics when steering is skipped or rejected; keep deliberate followup fallback; and update focused regression tests plus queue docs to match the final boundary semantics.

## Reproduction Assessment

Yes. The issue has concrete Discord/config steps, and current main provides a source-level reproduction path: same-session Discord messages are serialized before reply handling, while active `steer` fallback is still `enqueue-followup` when steering is not delivered.

## Solution Assessment

Yes, the narrow maintainable direction is a focused steering delivery fix plus diagnostics rather than a broad queue rewrite. The implementation should cover both the pre-runner Discord/session queue bottleneck and the runner/helper rejection paths.
