## Summary

Keep open. Current main still has the reported steer-to-followup shape in the active-run path, and the focused implementation candidate, #73365, is open, unmerged, and maintainer-labeled. The issue should wait for maintainer review or an equivalent merged fix rather than be closed as implemented or stale.

## What This Changes

The issue asks for `messages.queue.mode: "steer"` to inject inbound messages into active runs instead of silently degrading to followup, with diagnostics when steering is rejected.

## Best Possible Solution

Land #73365 or an equivalent narrow replacement that makes active-run steering lifecycle-aware, preserves deliberate followup fallback, emits useful rejection diagnostics, and adds regression coverage for the same-session channel queue path without broad queue rewrites.

## Reproduction Assessment

Yes. The issue gives concrete Discord/config steps, and current main provides a source-level reproduction path: same-session Discord work is serialized per session key, and any active non-streaming steer attempt that reaches reply handling is gated away from `queueEmbeddedPiMessage` and falls through to followup behavior.

## Solution Assessment

Yes. Reviewing and landing #73365 or a narrow equivalent is the best current path because it targets the active/non-streaming lifecycle gate and diagnostics directly; broader tool-boundary interruption semantics should be a separate maintainer decision if they differ from the current docs.
