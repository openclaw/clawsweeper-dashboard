## Summary

Keep open. Current main still has the reported behavior shape: the reply runner only attempts embedded Pi steering while `isStreaming` is true, the embedded queue helper rejects active non-streaming handles, active `steer` still maps to followup when injection is not accepted, and Discord same-session work is serialized through the channel run queue. Related #48003 is the older canonical report, but this issue also preserves useful Discord and diagnostics detail while open PRs #73365 and #75226 remain unmerged.

## What This Changes

Report that Discord/main-session `messages.queue.mode: "steer"` waits for a later followup instead of injecting into an active embedded Pi run, with missing diagnostics when steering falls back.

## Best Possible Solution

Implement lifecycle-aware active-run steering so eligible same-session messages can reach the embedded steering queue while the run is active, including non-streaming between-stream phases; emit actionable diagnostics when steering is rejected or falls back; preserve intentional followup fallback; and cover the Discord/session queue path with focused regression tests.

## Reproduction Assessment

Yes. The issue has concrete Discord/config steps, and current main provides a source-level reproduction path: active `steer` work that is non-streaming or serialized by the same session key falls through to followup instead of successful steering.

## Solution Assessment

Yes. The narrow maintainable direction is to review and land #73365, #75226, or an equivalent focused replacement that changes active-run steering eligibility and diagnostics without broad queue churn.
