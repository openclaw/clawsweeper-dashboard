## Summary

Keep this PR open, but it has a blocking security/correctness issue: the branch enables Codex workspace-write network access whenever OpenClaw reports a sandboxed session, even though the normal stdio Codex app-server is still spawned by the gateway process rather than run through the sandbox backend.

## What This Changes

The PR threads an internal Codex app-server flag so OpenClaw-sandbox-clamped turns send `workspaceWrite.networkAccess=true`, with focused tests for the new policy path.

## Best Possible Solution

Keep default `workspace-write` network denied, and only enable network through a policy or marker that proves Codex native execution is actually delegated to an OpenClaw sandbox backend such as Docker, SSH, or the Codex `externalSandbox` contract.

## Reproduction Assessment

Yes, by source inspection for the PR-introduced regression: the branch would set the network opt-in from any enabled sandbox context, while current main starts the stdio app-server through host `child_process.spawn`. I did not run the full production callback scenario locally.

## Solution Assessment

No. The requested behavior is valid, but `workspaceWrite.networkAccess=true` is too broad unless it is gated on a real external sandbox execution path or replaced with a Codex external-sandbox policy.
