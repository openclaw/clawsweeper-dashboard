## Summary

Keep open: current main and the latest release still have a source-backed path where a successful generated-media job is converted to a failed task when completion delivery evidence is false. The report is actionable, but the proposed blanket warning would weaken the documented message-tool delivery contract and should be replaced with a narrower delivery-evidence or direct-fallback fix.

## What This Changes

The issue asks OpenClaw to avoid marking a successfully generated background media task as failed when fallback textual completion delivery has already sent the media to the user.

## Best Possible Solution

Make generated-media completion delivery evidence match actual user delivery, including the direct fallback path for missing media, while preserving failure when no real delivery can be proven.

## Reproduction Assessment

No live end-to-end Telegram reproduction was run, but the source path is clear: current tests and code already show a successful generation becomes a failed task when completion delivery returns false. The exact plain-text media URL delivery claim still needs focused regression coverage or live proof.

## Solution Assessment

No, the suggested blanket warning is not the best fix because it would also mark genuinely undelivered generated media as successful. The safer fix is to recognize actual media delivery or perform the existing idempotent direct fallback, then complete only when delivery is proven.
