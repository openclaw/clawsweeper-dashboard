## Label Justifications

- `P1`: The report blocks normal subagent orchestration for supported non-off thinking on shipped OpenAI/Codex GPT-5 routes.
- `impact:session-state`: The failing value is per-session child-agent thinking state applied during subagent creation and run startup.

## Summary

Keep open: current main and v2026.5.20 still have a catalog-first off-only resolver path for `reasoning:false`, the reported `sessions_spawn` rejection flows through that shared validation path, and an open fixing PR already references this issue with closing syntax.

## What This Changes

The issue asks OpenClaw to accept supported non-off thinking levels for OpenAI/Codex GPT-5 subagent spawns instead of rejecting them through stale off-only catalog metadata.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84902 or a successor with focused resolver coverage plus live or runtime-probe proof, then close this issue with the merged commit and release provenance.

## Reproduction Assessment

Yes at source level, but I did not run a live subagent. A `reasoning:false` catalog entry reaches `resolveThinkingProfile`, returns off-only before provider policy, and `get-reply-run` emits the reported unsupported-thinking message while `subagent-spawn` forwards the explicit thinking override.

## Solution Assessment

Yes, the best direction is a shared resolver fix that lets explicitly authoritative provider thinking profiles override stale catalog false for OpenAI/Codex while preserving off-only behavior for non-reasoning models/providers. Closing this against the doctor migration would be incorrect because that migration only repaired persisted config validation.
