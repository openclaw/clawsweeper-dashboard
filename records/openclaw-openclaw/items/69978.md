- `P2`: The issue affects real TUI users running long sparse-output model work, but the remaining work is a bounded config/UX improvement rather than a core outage.
- `impact:session-state`: The watchdog path can clear TUI active-run state and show idle during quiet runs, which is session-state drift even when backend work may still be active.

## Summary

Keep open. Current main and the latest shipped tag still lack a TUI `/watchdog` command or public/persisted watchdog setting, the repeated notice path remains source-reproducible, and related open work covers config and notice pieces without fully replacing the requested command UX.

## What This Changes

The issue requests a TUI `/watchdog` command, persisted watchdog tuning or disablement, and per-run duplicate warning suppression for the streaming watchdog.

## Best Possible Solution

Choose one canonical watchdog UX, then land a focused TUI change that preserves reconnect recovery while adding the selected off/duration control, run-scoped duplicate-warning handling, docs, and fake-timer coverage coordinated with https://github.com/openclaw/openclaw/issues/68596, https://github.com/openclaw/openclaw/pull/77375, and https://github.com/openclaw/openclaw/pull/79997.

## Reproduction Assessment

Yes for source-level reproduction: current main has no `/watchdog` command or persisted setting, and the handler can clear `activeChatRunId`, re-adopt the same unfinished run on a later delta, and arm another timeout without a warned-run guard. I did not run a live long-reasoning TUI session during this read-only review.

## Solution Assessment

Unclear. Per-run warning suppression is narrow, but the command/config/persistence surface overlaps active related work and needs a maintainer UX decision before implementation is queued.
