## Summary

Keep open. Current main still has generic browser Talk in `/chat`, but the requested xAI/Grok realtime voice path is not registered or exposed on the Talk provider surface, and the linked xAI work targets a separate `/voiceclaw/realtime` gateway surface.

## What This Changes

The issue asks for a Grok/xAI live voice loop in the `/chat` UI with browser speech input, spoken agent replies, and a continuous provider-backed conversation.

## Best Possible Solution

Keep this as the tracker for a maintainer-approved xAI `RealtimeVoiceProviderPlugin` for `/chat` Talk, with matching xAI manifest/docs/UI/tests and a clear routing decision for any shared code with the adjacent `/voiceclaw/realtime` work.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence current main cannot satisfy xAI/Grok `/chat` Talk because xAI is not registered as a realtime voice provider and docs still mark realtime voice unexposed.

## Solution Assessment

Yes for keeping it open. The maintainable path is to use the existing Talk realtime voice provider registry, with maintainer judgment on xAI transport shape and bundled-versus-external plugin scope before implementation.
