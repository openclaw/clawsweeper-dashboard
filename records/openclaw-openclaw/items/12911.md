## Summary

Current main provides a generic browser Talk experience, but it still does not register or expose an xAI/Grok realtime voice provider; the linked open work explicitly targets the separate `/voiceclaw/realtime` surface, so the requested `/chat` capability remains necessary and needs a maintainer product decision.

## What This Changes

Add xAI/Grok realtime voice as a selectable provider for the web chat Talk experience, including browser microphone input and spoken agent responses.

## Best Possible Solution

Keep one provider-agnostic Talk flow and add an xAI `RealtimeVoiceProviderPlugin` in the xAI plugin, using the existing direct-browser or gateway-relay contracts where possible, with a single documented provider-selection setting and focused browser, relay, credential-redaction, transcript, tool-call, and fallback coverage.

## Reproduction Assessment

Yes at the source level: current main has a concrete realtime Talk provider registry, the xAI plugin is absent from it, and the xAI docs explicitly mark realtime voice unavailable. This is a feature-gap verification rather than a failing bug reproduction.

## Solution Assessment

Yes regarding ownership: extending the existing realtime voice provider plugin seam is the narrowest maintainable solution and avoids duplicating the Talk loop. The exact xAI transport reuse and provider-selection/configuration contract remain maintainer decisions.
