## Summary

Keep open. Current main has generic browser Talk in `/chat`, but xAI/Grok realtime voice is still not registered on the Talk realtime provider surface and the adjacent xAI work tracks other voice surfaces rather than replacing this request.

## What This Changes

Request to add an xAI/Grok-powered Live voice loop in the `/chat` UI with browser microphone input, in-browser assistant audio, transcription, and continuous spoken turns.

## Best Possible Solution

Implement xAI/Grok as a provider-owned `RealtimeVoiceProviderPlugin` for `/chat` Talk, sharing protocol code with adjacent voice-call or `/voiceclaw/realtime` work only after the transport boundary is settled.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug. Source inspection gives high confidence that current main cannot satisfy xAI/Grok `/chat` Talk because xAI is not registered as a realtime voice provider.

## Solution Assessment

Yes, keeping it open is the best current outcome. The maintainable path is an xAI realtime voice provider on the existing Talk provider-plugin surface, with an additive transport seam only if xAI cannot fit the current browser-session or Gateway relay contracts.
