## Summary

The feature remains missing on current main: `/chat` Talk is provider-selectable, but xAI is still neither registered nor documented as exposed. The issue should stay open for an explicit maintainer product decision because the existing provider seam can support the official xAI browser/WebSocket flow, while the linked `/voiceclaw/realtime` work is a separate surface.

## What This Changes

Add Grok/xAI as a live realtime voice provider for the `/chat` Talk UI so users can converse by browser microphone and hear spoken agent responses.

## Best Possible Solution

Add an xAI `RealtimeVoiceProviderPlugin` under `extensions/xai`, use official ephemeral client secrets for browser WebSocket sessions or the existing gateway relay, register the capability in the plugin manifest, and validate the real `/chat` Talk loop without adding a parallel UI or gateway voice architecture.

## Reproduction Assessment

Yes for the capability gap. Current source and docs deterministically show that `/chat` Talk supports registered realtime providers while xAI realtime voice is still unregistered and unexposed.

## Solution Assessment

Yes at the architectural level. The existing realtime voice provider and browser-session/relay contracts are the narrowest maintainable solution; the separate `/voiceclaw/realtime` adapter is not the correct owner for this UI path.
