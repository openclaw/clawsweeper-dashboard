## Summary

Current main provides a generic browser Talk experience, but xAI realtime voice remains explicitly unexposed. The linked open PR implements the separate `/voiceclaw/realtime` gateway surface and expressly does not implement the provider-plugin surface used by `/chat`, so the requested capability remains necessary.

## What This Changes

Add xAI/Grok-backed live browser voice conversations to the `/chat` Talk experience.

## Best Possible Solution

Implement xAI as a provider-owned realtime voice plugin for the existing Talk surface, extending the generic transport contract only where xAI requires it and documenting fresh setup, provider selection, credentials, and fallback behavior.

## Reproduction Assessment

Yes at source level: current main exposes browser Talk while the xAI provider documentation and registration surface still omit realtime voice. This is a clearly inspectable feature gap rather than a failure of an existing contract.

## Solution Assessment

Yes on the architectural direction: the existing provider-plugin seam is the narrowest maintainable solution. Maintainers must still choose the xAI transport, credential, and provider-selection contract before implementation.
