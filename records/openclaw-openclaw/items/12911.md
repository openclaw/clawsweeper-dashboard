## Summary

Keep this issue open: current main provides browser Talk through registered OpenAI and Google realtime voice providers, but xAI/Grok is still absent from that provider surface. The linked xAI work remains open and explicitly targets the separate `/voiceclaw/realtime` gateway interface, while maintainer discussion says this browser Talk request remains distinct.

## What This Changes

Add Grok/xAI-backed live voice conversation to the web `/chat` Talk experience.

## Best Possible Solution

Add an xAI-owned realtime voice provider through the existing Talk plugin seam, reuse Gateway relay where the upstream protocol permits, and extend the generic provider contract only when a live xAI API test proves an additional transport capability is necessary.

## Reproduction Assessment

Yes at source level: current main deterministically registers browser Talk providers for OpenAI and Google but not xAI. A live xAI session is still required to validate the eventual transport implementation, not to establish the missing capability.

## Solution Assessment

The plugin-owned realtime voice provider seam is the best maintainable boundary, but the exact browser-session versus Gateway-relay implementation remains a product and upstream-contract decision.
