## Summary

Current main and the latest release still match the reported failure path: Podman containers run with HOME=/home/node, Claude CLI setup reads the container home for credentials, and neither the setup run nor Quadlet mounts Claude CLI state. This should stay open as a valid, narrow Podman/Claude CLI auth regression.

## What This Changes

The issue asks OpenClaw to make Podman setup recognize an existing host Claude CLI login by exposing the needed Claude credentials inside the onboarding container.

## Best Possible Solution

Update the Podman launcher and Quadlet flow so Claude CLI reuse works in containers through a narrow, validated credential mount or equivalent auth handoff that covers both onboarding and runtime.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the Podman setup container has HOME=/home/node but no /home/node/.claude state, and the Anthropic CLI setup path throws the reported error when its credential reader returns null. I did not run a live Fedora/Podman reproduction in this read-only pass.

## Solution Assessment

Yes, with one scope correction: the report's mount-based direction is likely the narrowest fix, but it should cover the setup container, the gateway runtime container, and the Quadlet template rather than only the setup run.
