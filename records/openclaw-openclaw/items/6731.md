## Summary

Keep open. Current main and v2026.5.12 still preserve trusted-operator host exec as no-approval by default unless the operator opts into stricter policy, and the requested hard/default dangerous-operation gate remains a core product and security decision.

## What This Changes

The issue requests a safe/unsafe ClawdBot mode or hard dangerous-operation gate for host exec, especially destructive PowerShell commands and prompt-injection risk.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement a native/default dangerous-operation gate with Windows classifications, approval UX, sandbox interaction, docs, and regression tests, or explicitly document the current opt-in model as final.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.12 show host exec can still default to security="full" and ask="off", while requiresExecApproval does not classify destructive commands under that posture. I did not run a destructive Windows command.

## Solution Assessment

No as originally proposed: rewriting in Rust or adding a broad safe/unsafe mode is not the narrowest maintainable fix. A focused product/security decision around core exec/tool policy is the safer path before implementation.
