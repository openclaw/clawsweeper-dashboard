## Summary

Keep open. Current main and v2026.5.12 still preserve trusted-operator host exec as no-approval by default unless stricter policy is configured, while the requested hard/default destructive-operation gate remains a product/security decision rather than cleanup-closeable work.

## What This Changes

The issue requests a safe/unsafe mode or hard safety gate for OpenClaw host execution, especially destructive commands, sandbox posture, and prompt-injection risk.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement a native/default destructive-operation gate with Windows classifications, approval UX, sandbox interaction, docs, and regression tests, or explicitly document the current opt-in model as final.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.12 show host exec can still default to security="full" and ask="off", while requiresExecApproval does not classify destructive commands under that posture. I did not run a destructive Windows command.

## Solution Assessment

No as originally proposed: rewriting in Rust or adding a broad safe/unsafe mode is not the narrowest maintainable fix. A focused product/security decision around core exec/tool policy is the safer path before implementation.
