## Summary

Keep open. Current main still implements hard exec controls only through configured policy, allowlists, approvals, sandboxing, and tool restrictions; it does not provide the native default dangerous-operation gate requested in the discussion, and a maintainer comment already kept this item open for that reason.

## What This Changes

The issue requests a safe/unsafe mode or hard tool-layer safety gate for ClawdBot/OpenClaw destructive operations and prompt-injection risk.

## Best Possible Solution

Keep this open until maintainers either implement a native default hard gate for destructive host operations, with Windows PowerShell coverage and approval UX/docs/tests, or explicitly decide that the current opt-in exec approvals, sandboxing, and tool-policy model is the final product stance.

## Reproduction Assessment

Yes for the remaining policy gap: no destructive live run is needed because current main source and docs show default host exec remains `security="full"` with `ask="off"`, and the approval predicate has no dangerous-operation classifier. The original Windows `Remove-Item` incident was not safely reproduced directly.

## Solution Assessment

No, the original Rust rewrite framing is not the narrow maintainable fix, and current opt-in controls do not fully satisfy the requested default hard gate. The better solution is a targeted core exec/tool-policy design decision covering default safe mode, dangerous-command classification, approval behavior, sandbox interaction, and regression coverage.
