## Summary

Keep open. Current main and the latest shipped release still document and implement trusted-operator host exec as `security="full"` and `ask="off"` by default when no sandbox is active, and the requested hard/default destructive-operation gate remains a product/security decision rather than cleanup-closeable.

## What This Changes

Request a safe/unsafe ClawdBot mode or hard tool-layer safety gate for destructive agent operations such as Windows PowerShell removals, credential revocation, and config overwrites.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement a native/default dangerous-operation gate with Windows classifications, approval UX, sandbox interaction, docs, and regression tests, or explicitly document the current opt-in model as final.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.12 show host exec can still default to `security="full"` and `ask="off"`, while `requiresExecApproval` does not classify destructive commands under that posture. I did not run a destructive Windows command.

## Solution Assessment

Unclear as originally proposed: rewriting in Rust or adding broad safe/unsafe mode is not a narrow maintainable fix. The safer path is a focused product/security decision around core exec/tool policy, then bounded implementation if maintainers choose to change the current stance.
