## Summary

Keep open. Current main and v2026.5.22 still preserve trusted-operator host exec as `security="full"` and `ask="off"` by default, while this issue asks for a hard/default destructive-operation gate or explicit safe/unsafe posture that needs product and security review.

## What This Changes

The issue requests a safe/unsafe execution posture or hard gate for dangerous ClawdBot operations, especially destructive host exec commands and prompt-injection-sensitive workflows.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement a focused native/default destructive-operation gate with Windows PowerShell classifications, compatibility behavior, docs, and regression tests, or explicitly document the current opt-in model as final.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.22 still resolve non-sandbox host exec to `security="full"` and `ask="off"`, while `requiresExecApproval` does not classify destructive commands under that posture. I did not run `Remove-Item` or another destructive command.

## Solution Assessment

No as originally proposed: a Rust rewrite or broad safe/unsafe mode is not the narrowest maintainable fix. A focused product/security decision around core exec/tool policy, default posture, Windows destructive-command handling, compatibility, and docs/tests is the safer path.
