## Summary

Keep open: current main and the latest release have stronger opt-in exec controls, but they still intentionally preserve trusted-operator no-approval host exec and do not provide the default hard destructive-operation gate requested in the discussion.

## What This Changes

The issue requests a safe/unsafe ClawdBot mode with hard sandbox or destructive-operation protection for prompt-injection and elevated host-exec risk.

## Best Possible Solution

Define a maintainer-approved safe default using the existing mode and sandbox abstractions, keep `full` as explicit break-glass, and document/test fresh-install and upgrade behavior for gateway, node, Windows PowerShell, and unattended sessions.

## Reproduction Assessment

Yes, at high confidence from source and documented behavior: omitted host exec policy resolves to `security="full"` with `ask="off"`, and approval logic has no generic destructive-operation predicate. I did not run a destructive Windows command because the source-level policy gap is enough and a live destructive repro would be unsafe.

## Solution Assessment

No, the literal Rust rewrite or small `dangerousOps` list is not the best fix. The maintainable direction is a product/security decision that uses the existing closed exec-mode, sandbox, allowlist, and approval boundaries instead of a brittle command-name blocklist.
