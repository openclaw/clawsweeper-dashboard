## Summary

Current main now has normalized `deny`/`allowlist`/`ask`/`auto`/`full` exec modes and recommends guarded `auto`, but omitted configuration still resolves to unrestricted `full` with prompts off, and YOLO mode intentionally has no destructive-command classifier; the requested safe-by-default hard gate therefore remains a product/security decision.

## What This Changes

Requests a native safe/unsafe operating mode with hard sandboxing or confirmation gates for destructive agent operations.

## Best Possible Solution

Define a maintainer-approved safe default using the existing mode abstraction—prefer sandbox plus deterministic allowlist or human approval—while keeping `full` as an explicit break-glass choice and documenting tested fresh-install and upgrade behavior across gateway, node, Windows PowerShell, and unattended sessions.

## Reproduction Assessment

Yes at high confidence from source and documented behavior: omit `tools.exec.mode` and host exec resolves to `security=full` with `ask=off`, while `requiresExecApproval` has no generic destructive-operation predicate. A destructive live command is unnecessary and unsafe for this verification.

## Solution Assessment

No, a command-name `dangerousOps` list or Rust rewrite is not the best solution; the maintainable path is to use the existing closed exec-mode policy and sandbox/approval boundaries, then make an explicit default and migration decision.
