## Summary

Keep open. Current main still documents and implements no-approval host exec as the trusted-operator default, and the requested native/default dangerous-operation gate remains a core security/product decision rather than a cleanup-closeable item.

## What This Changes

The issue asks for a safe/unsafe ClawdBot mode with stronger hard controls around sandboxing, root access, prompt injection, and destructive host operations.

## Best Possible Solution

Keep this open until maintainers either implement a native default hard gate for destructive host operations with Windows PowerShell coverage, approval UX, docs, and tests, or explicitly decide that the current opt-in exec approvals, sandboxing, and tool-policy model is the final stance.

## Reproduction Assessment

Yes for the source-level policy gap: current main still defaults host exec to security="full" with ask="off", and the approval predicate has no destructive-operation classifier. The original Windows Remove-Item incident was not safely reproduced directly.

## Solution Assessment

No, the Rust rewrite and broad safe/unsafe framing are not the narrow maintainable path. The better solution is a targeted core exec/tool-policy decision covering safe defaults, destructive-command classification, approvals, sandbox interaction, and regression coverage.
