## Summary

The requested hard safety boundary remains absent from current main and v2026.6.5: host exec can intentionally use full security with approvals off, and approval decisions do not separately classify destructive operations. This remains core security and product-policy work with upgrade consequences, so it is not cleanup-closeable.

## What This Changes

Requests a safe/unsafe operating mode with hard enforcement or approvals for destructive agent operations and prompt-injection risk.

## Best Possible Solution

Specify a core exec safety contract that makes the conservative path easy and explicit, preserves an intentional full-power opt-in, defines cross-platform approval and sandbox behavior, and includes fresh-install and upgrade coverage rather than relying on a command-name denylist.

## Reproduction Assessment

Yes at the source-contract level: current main and v2026.6.5 expose full+ask-off host execution, and the approval predicate has no destructive-operation classifier. This read-only review did not execute a destructive command, so it proves the missing gate rather than an actual damage incident.

## Solution Assessment

No, a simple dangerous-command allowlist or denylist is not the best solution because shell semantics make it incomplete and brittle. The safer direction is a maintainer-defined exec policy integrating approvals, sandboxing, explicit full-power opt-in, Windows behavior, and upgrade compatibility.
