## Summary

This issue remains necessary: current main provides opt-in restrictive exec modes, sandboxing, and auto-review, but unrestricted host exec still intentionally lacks a separate destructive-operation gate. Prior blocklist and high-risk-confirmation attempts closed unmerged after exposing bypass and compatibility concerns, so this requires an explicit product and security design rather than cleanup closure.

## What This Changes

Requests a safe mode with hard enforcement for destructive agent operations, contrasted with an explicitly trusted unrestricted mode.

## Best Possible Solution

Adopt an explicit core exec-policy design with a conservative safe profile and a clearly named trusted unrestricted profile, enforcing structured approval or sandbox policy at the execution boundary rather than relying on a brittle command-name denylist, with Windows and POSIX coverage plus documented upgrade behavior.

## Reproduction Assessment

Yes at the source-contract level: selecting or inheriting full/off host exec permits commands without a separate destructive-operation classifier, exactly as current docs state. Running a destructive live command would add risk without improving confidence.

## Solution Assessment

Unclear for the issue's proposed shape: safe versus trusted profiles are directionally appropriate, but a Rust rewrite or configurable command blocklist is not the narrowest reliable solution; maintainers must first define enforcement semantics and the compatibility contract.
