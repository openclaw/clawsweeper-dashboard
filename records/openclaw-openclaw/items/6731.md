## Summary

Keep open. Current main and the latest shipped release still document and implement trusted-operator host exec with `security="full"` and `ask="off"` by default, and there is still no native/default destructive-command gate for cases like Windows PowerShell `Remove-Item`; this needs product and security review rather than conservative cleanup closure.

## What This Changes

Request a safe/unsafe mode or hard default gate for agent exec so destructive operations and prompt-injection-driven tool use require stronger sandboxing or approval.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement a default/native dangerous-operation gate with Windows classifications, approval UX, sandbox interaction, docs, and regression tests, or explicitly document the current opt-in model as final.

## Reproduction Assessment

Yes, for the central current behavior at source level: docs and code show host exec can run with `security="full"` and `ask="off"`, and `requiresExecApproval` does not classify destructive commands under that posture. I did not run a destructive live Windows command.

## Solution Assessment

Unclear as proposed: the broad safe/unsafe and Rust rewrite framing is not a narrow maintainable fix. The maintainable path is a focused product/security decision followed by core exec/tool-policy changes if maintainers choose to change the current stance.
