## Summary

Keep open. Current main and v2026.5.12 still preserve trusted-operator host exec as no-approval by default unless stricter policy is configured, and the requested native/default destructive-operation gate remains a product/security decision rather than conservative cleanup.

## What This Changes

The issue requests a safe/unsafe ClawdBot mode or hard platform safety gate that prevents destructive agent operations and prompt-injection-driven host actions by default.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement a native/default destructive-operation gate with Windows PowerShell classifications, approval UX, sandbox interaction, docs, and regression tests, or explicitly document the current opt-in model as final while linking narrower follow-up items.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.12 show host exec can still default to security="full" and ask="off", while requiresExecApproval does not classify destructive commands under that posture. I did not run Remove-Item or another destructive Windows command.

## Solution Assessment

No as originally proposed: a Rust rewrite or broad safe/unsafe mode is not the narrowest maintainable fix. A focused product/security decision around core exec/tool policy, default posture, Windows destructive-command handling, and compatibility is the safer path.
