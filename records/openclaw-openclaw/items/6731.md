## Summary

Keep open. Current main and the latest shipped release still preserve trusted-operator host exec as full/off by default, while the issue asks for a native hard/default destructive-operation gate or safe/unsafe posture that needs explicit product and security judgment.

## What This Changes

The issue requests a safe/unsafe security posture or hard destructive-operation gate for ClawdBot/OpenClaw host execution, motivated by prompt-injection and data-loss risk.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement focused destructive-operation gating with Windows PowerShell coverage, compatibility behavior, docs, and tests, or document that opt-in approvals/sandboxing are the final model.

No as originally proposed: a Rust rewrite or broad safe/unsafe mode is not the narrowest maintainable fix. A focused product/security decision around core exec/tool policy, default posture, Windows destructive-command handling, compatibility, and docs/tests is the safer path.
