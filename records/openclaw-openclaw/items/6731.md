## Summary

Keep open. Current main still implements and documents no-approval host exec as the trusted-operator default unless the operator opts into stricter policy, and I did not find the native/default dangerous-operation gate requested for cases like Windows PowerShell `Remove-Item`; the remaining work needs product and security review, not cleanup closure.

## What This Changes

Request a safe/unsafe OpenClaw mode or hard default safety gate for sandboxing, root access, prompt-injection risk, and destructive host-exec operations.

## Best Possible Solution

Have maintainers choose the core exec/tool-policy stance, then either implement a default/native dangerous-operation gate with Windows classifications, approval UX, sandbox interaction, docs, and regression tests, or explicitly document the current opt-in model as final.

## Reproduction Assessment

Yes for the central current behavior: source and docs show host exec can run with `security="full"` and `ask="off"`, and `requiresExecApproval` does not classify destructive commands under that posture. I did not run a destructive live Windows command.

## Solution Assessment

Unclear as an implementation shape: the broad safe/unsafe/Rust rewrite framing is not a narrow maintainable fix. The maintainable path is a focused product/security decision followed by core exec/tool-policy changes if maintainers choose to change the current stance.
