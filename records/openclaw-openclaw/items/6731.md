## Summary

Keep open. Current main still documents and implements no-approval host exec as the trusted-operator default, while the requested native dangerous-operation gate remains unresolved and security/product-scoped.

## What This Changes

Request a safe/unsafe mode or hard tool-layer gate that prevents destructive ClawdBot/exec actions unless explicitly approved, with sandboxed safe defaults as part of the direction.

## Best Possible Solution

Keep this open until maintainers either implement a native default hard gate for destructive host operations with Windows PowerShell coverage, approval UX, docs, and tests, or explicitly decide that opt-in exec approvals, sandboxing, and tool policy are the final stance.

## Reproduction Assessment

Yes for the source-level policy gap: current main defaults host exec to security="full" with ask="off", and the approval predicate has no destructive-operation classifier. I did not run an unsafe Windows Remove-Item reproduction.

## Solution Assessment

No, the Rust rewrite and broad safe/unsafe framing are not the narrow maintainable solution. The better path is a targeted core exec/tool-policy decision covering safe defaults, destructive-command classification, approvals, sandbox interaction, and regression coverage.
