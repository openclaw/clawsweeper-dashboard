## Summary

Keep open. Current main still has configurable safeguards, but host exec remains no-approval by default for trusted gateway/node operators and there is no native destructive-command classifier; prior maintainer-facing discussion also explicitly kept the item open for product/security follow-up.

## What This Changes

The issue requests a safe/unsafe ClawdBot mode with hard platform gates for destructive or elevated host-exec operations and prompt-injection risk.

## Best Possible Solution

Keep this as a core exec/tool-policy security item until maintainers either implement a hard/default dangerous-operation gate with docs and tests, or explicitly decide the current opt-in controls are the final product stance.

## Reproduction Assessment

Yes, as a source-level repro. With gateway/node host exec left at the documented defaults, the code resolves security to full and ask to off, and the approval predicate has no separate destructive-operation classifier for commands such as Windows PowerShell Remove-Item.

## Solution Assessment

No clear implementation is ready to automate. The maintainable path is a narrower maintainer security/product decision around default safe mode, destructive-command classification, approval UX, sandbox interaction, and regression coverage, not a Rust rewrite or an ad hoc local patch.
