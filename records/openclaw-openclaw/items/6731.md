## Summary

Keep open. Current main still documents and implements no-approval host exec as the trusted-operator default, and prior maintainer/bot comments explicitly kept this open because a native/default dangerous-operation gate remains unresolved.

## What This Changes

Request a safe/unsafe operating mode or hard tool-layer gate that prevents destructive ClawdBot host actions and prompt-injection-driven operations by default.

## Best Possible Solution

Keep this open until maintainers either implement a native default hard gate for destructive host operations, with Windows PowerShell coverage and approval UX/docs/tests, or explicitly decide that the current opt-in exec approvals, sandboxing, and tool-policy model is the final stance.

## Reproduction Assessment

Yes for the source-level policy gap: current main still defaults host exec to security="full" with ask="off", and the approval predicate has no destructive-operation classifier. The original Windows Remove-Item incident was not safely reproduced directly.

## Solution Assessment

No, the Rust rewrite and broad safe/unsafe framing are not the narrow maintainable path. The better solution is a targeted core exec/tool-policy decision covering safe defaults, dangerous-command classification, approvals, sandbox interaction, and regression coverage.
