## Summary

Keep open. Current main still documents and implements no-approval host exec as the trusted-operator default, with hard controls available only through configured policy, allowlists, approvals, sandboxing, and tool restrictions; that does not satisfy the requested native/default dangerous-operation gate.

## What This Changes

The issue requests a safe/default hard gate for dangerous ClawdBot or host exec operations, with unsafe elevated operation left as an explicit risk mode.

## Best Possible Solution

Keep this open until maintainers either implement a native default hard gate for destructive host operations, with Windows PowerShell coverage and approval UX/docs/tests, or explicitly decide that the current opt-in exec approvals, sandboxing, and tool-policy model is the final product stance.

## Reproduction Assessment

Yes for the source-level policy gap: current main shows default host exec remains security="full" with ask="off", and the approval predicate has no dangerous-operation classifier. The original Windows Remove-Item incident was not safely reproduced directly.

## Solution Assessment

No, the Rust rewrite framing is not the narrow maintainable fix, and the current opt-in controls do not fully satisfy the requested default hard gate. The better path is a targeted core exec/tool-policy product decision covering safe defaults, dangerous-command classification, approvals, sandbox interaction, and regression coverage.
