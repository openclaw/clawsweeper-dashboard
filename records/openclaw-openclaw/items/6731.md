## Summary

Current main offers configurable sandboxing, allowlists, approval modes, auto-review, and strict inline-eval checks, but still defaults host exec to full access with approvals off and does not provide the requested hard/default destructive-operation gate. This remains a core security and product-policy decision rather than cleanup-closeable work.

## What This Changes

Requests a native safe/unsafe operating mode with hard enforcement for destructive agent operations and prompt-injection risk.

## Best Possible Solution

Adopt a core policy-driven safe execution profile that defaults untrusted or autonomous agents to sandboxed or approval-gated host actions, retains an explicit trusted full-access profile, and covers Windows PowerShell plus indirect execution paths without relying only on command-name heuristics.

## Reproduction Assessment

Yes. Current-main source establishes the policy gap without executing a destructive command: host exec falls back to full/off, and approval is based on configured policy or allowlist misses rather than destructive-operation classification.

## Solution Assessment

No, not as a complete Rust rewrite or a simple dangerous-command string list. The best path is a focused policy/profile design at the existing exec, approval, sandbox, and tool-policy boundary with explicit trusted-mode compatibility.
