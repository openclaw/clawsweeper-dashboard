## Summary

Keep open. Current main still has no account-level Slack native-command naming surface, both linked implementation PRs are closed unmerged, and the remaining work is a public Slack config/API decision rather than a cleanup-closeable duplicate.

## What This Changes

The issue requests a Slack account-level native slash-command naming option so multiple Slack apps can register distinct commands for the same internal actions.

## Best Possible Solution

Add a validated Slack account-level native command naming contract and thread it through native spec generation, registration, dispatch lookup, argument-menu callbacks, docs, and focused regression tests while preserving /agentstatus compatibility.

## Reproduction Assessment

Yes for the currently unsupported path: strict Slack config has no account-level native naming field, and native registration/lookup are provider-scoped. I did not perform a live multi-app Slack workspace test in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config shape. Account-level native naming is the right boundary, but the closed PR history shows nativePrefix versus nativeNames remains an API/product decision.
