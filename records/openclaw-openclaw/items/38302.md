## Label Justifications

- `P2`: The issue affects a real multi-agent Slack workflow but is bounded to Slack native command naming and needs product review before implementation.
- `impact:message-loss`: Duplicate Slack native command names can route commands to the wrong app or make the intended agent command unreachable.
- `impact:session-state`: Commands such as reset, compact, stop, model, and status can affect or report the wrong agent session in shared multi-agent Slack setups.

## Summary

Keep open. Current main and the latest release still expose only provider-scoped Slack native command names, the two linked implementation PRs closed unmerged, and the same Slack multi-app symptom was already closed elsewhere as a duplicate of this canonical item.

## What This Changes

The issue requests account-scoped Slack native slash command naming so multiple Slack apps can target different agents without duplicate command-name collisions.

## Best Possible Solution

Keep this as the canonical Slack multi-app native-command item and implement a maintainer-approved account-scoped naming contract with validation, collision handling, docs, and tests, or explicitly document duplicate Slack native names as unsupported.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main still generates and looks up Slack native commands by provider-level names only, while Slack account config has no account-scoped native naming field; no live Slack workspace run was performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? The need is valid, but the exact prefix proposal is not yet clearly the best fix because later review feedback favored an explicit nativeNames-style mapping with validation and collision checks.
