## Summary

Keep open. Current main still has provider-scoped Slack native command names, the latest shipped release has the same missing surface, and both linked implementation attempts were closed unmerged, so this remains a real Slack config/API product decision rather than a cleanup-close candidate.

## What This Changes

The issue requests account-level Slack native slash command naming so multiple Slack apps can register distinct commands for different agents.

## Best Possible Solution

Ship a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration/dispatch, argument-menu callbacks, docs, and regression tests while preserving /agentstatus.

## Reproduction Assessment

Yes, at source level: configure multiple Slack app accounts with the same native command names and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior before implementation.
