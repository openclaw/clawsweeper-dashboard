## Label Justifications

- `P2`: This is a real multi-agent Slack workflow gap with meaningful impact, but it is limited to native Slack command deployments and needs product direction before repair.
- `impact:message-loss`: Colliding Slack native command names can send a user command to the wrong Slack app/account or leave the intended app unreachable.
- `impact:session-state`: Commands such as reset, compact, stop, and status can affect or report the wrong agent/session when native command names collide.

## Summary

Keep open: current main and v2026.6.6 still lack account-scoped Slack native command names, and the closed PR attempts did not merge. The remaining work needs a maintainer product decision on the permanent config contract.

## What This Changes

Request account-level Slack native command naming so multiple Slack app accounts can register distinct command variants without collisions.

## Best Possible Solution

Choose the supported Slack account-scoped native command contract, then implement it through config schema, spec generation, lookup, arg-menu roundtrip, docs, and regression tests.

## Reproduction Assessment

Yes at source level: current Slack native command registration and lookup carry provider context but no account-specific command-name data, and the config schema has no nativePrefix/nativeNames field. I did not run a live multi-app Slack workspace repro.

## Solution Assessment

Unclear: the proposed prefix is plausible, but earlier PR discussion moved toward explicit nativeNames, so a maintainer should choose the permanent contract before implementation.
