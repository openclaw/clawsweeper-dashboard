## Label Justifications

- `P2`: This is a real multi-agent Slack limitation with channel-specific blast radius and normal maintainer priority.
- `impact:message-loss`: Duplicate Slack native command names can make commands unavailable, misrouted, or handled by the wrong app or agent.
- `impact:session-state`: Commands such as reset, compact, stop, and status affect agent/session state and cannot be targeted reliably in the reported setup.

## Summary

Keep open: current main and v2026.5.20 still only support provider-scoped Slack native command names, while Slack's non-namespaced command model and two user reports keep the multi-app collision real. The remaining work is a new account-level Slack native naming contract, and that needs maintainer product/API choice rather than cleanup closure.

## What This Changes

Request account-level Slack native command naming so multiple Slack app accounts can register distinct slash commands for reset, compact, stop, status, and related native commands.

## Best Possible Solution

Define and ship a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration, interactive menu callbacks, docs, and regression tests while preserving the /agentstatus exception.

## Reproduction Assessment

Yes, at source level: configure multiple Slack accounts/apps with native commands enabled and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior.
