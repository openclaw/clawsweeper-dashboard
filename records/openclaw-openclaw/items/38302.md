## Summary

Keep open. Current main still lacks the requested account-level Slack native command naming: the Slack config type/schema/metadata expose only enabled/name/sessionPrefix/ephemeral, native command generation and lookup are provider-scoped, and Slack registration/argument-menu dispatch still resolve names only for provider "slack". The two linked implementation attempts, #38326 and #38391, are closed and unmerged, while #74343 adds fresh Slack multi-app evidence that appears compatible with the same fix cluster.

## What This Changes

Add per-account Slack native slash-command naming so multiple Slack apps can register distinct commands such as prefixed reset/compact/stop commands for separate agents.

## Best Possible Solution

Implement a validated account-level Slack native naming surface in the bundled Slack plugin and shared native-command contract, preserving current unprefixed behavior by default and the existing /status -> /agentstatus provider override. The fix should thread the account-specific names through native spec generation, Slack command registration, native dispatch lookup, and argument-menu callbacks, then update generated config metadata, Slack/slash-command docs, and focused multi-account tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the proposed `channels.slack.accounts.<id>.slashCommand.nativePrefix` field is absent from the type and rejected by the strict schema, and current native lookup cannot resolve a prefixed name such as `/oz-reset` back to the `reset` command.

## Solution Assessment

Yes, the best repair belongs in the Slack account config plus the shared native-command lookup seam. A Slack-only registration change would not be enough because command dispatch and interactive argument-menu callbacks also currently use provider-only name resolution.
