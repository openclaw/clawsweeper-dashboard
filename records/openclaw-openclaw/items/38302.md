## Summary

Keep open. Current main still lacks account-level Slack native-command naming, and the two linked implementation attempts are closed unmerged; resolving this needs a maintainer API/product decision rather than cleanup closure.

## What This Changes

Request account-level Slack slash-command native naming so multiple Slack apps or agents can register distinct native commands in one workspace.

## Best Possible Solution

Ship a validated Slack account-level native command naming contract and thread it through config schema/types, native spec generation, reverse lookup, arg-menu payloads, docs, and regression tests while preserving `/agentstatus`.

## Reproduction Assessment

Yes at source and dependency-contract level: configure multiple Slack apps in one workspace with the same native command names, and current OpenClaw generates and resolves Slack native command names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the likely boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior before implementation.
