## Summary

Keep open. Current main and the latest shipped release still lack account-level Slack native-command naming, and the two linked implementation PRs were closed unmerged, so this remains a product/API decision rather than a cleanup-close candidate.

## What This Changes

Request account-specific Slack native slash-command names so multiple Slack apps in one workspace can avoid command-name collisions.

## Best Possible Solution

Ship a validated Slack account-level native command naming contract through config schema/types, native spec generation, reverse lookup, Slack registration/dispatch, argument-menu callbacks, docs, and regression tests while preserving /agentstatus.

## Reproduction Assessment

Yes, at source level: configure multiple Slack app accounts with the same native command names and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior before implementation.
