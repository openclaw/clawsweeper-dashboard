## Label Justifications

- `P2`: This is a real multi-agent Slack limitation with channel-specific blast radius and normal maintainer priority.
- `impact:message-loss`: Duplicate Slack native command names can make slash commands unavailable, misrouted, or handled by the wrong app/agent.
- `impact:session-state`: The affected commands include reset, compact, stop, and status, which act on agent/session state and cannot be reliably targeted in the reported setup.

## Summary

Keep open: current main still has provider-scoped Slack native command names and strict Slack slash-command config without account-level naming. The related implementation attempts were closed unmerged after discussion moved from prefix to explicit mapping and left validation, collision, and menu semantics unresolved.

## What This Changes

The issue requests account-level Slack native slash-command naming so each Slack app/agent can register unique commands such as `/oz-reset` and avoid multi-app command collisions.

## Best Possible Solution

Ship a validated account-level Slack native-command naming contract across config types/schema, native spec generation, reverse lookup, Slack registration/dispatch, argument-menu callbacks, docs, and regression tests while preserving /agentstatus.

## Reproduction Assessment

Yes, source-level: with multiple Slack accounts and native commands enabled, current main generates Slack native names from config plus provider only and resolves incoming names by provider only, so all accounts expose the same names. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers must choose nativePrefix versus explicit nativeNames mapping plus validation, collision, and argument-menu semantics before implementation.
