## Label Justifications

- `P2`: This is a real multi-agent Slack native-command workflow gap with meaningful impact, but it is bounded to Slack native command deployments and needs product direction before repair.
- `impact:message-loss`: Colliding Slack native command names can route a command to the wrong Slack app/account or leave the intended app unreachable.
- `impact:session-state`: Commands such as reset, compact, stop, and status can affect or report the wrong agent/session when native Slack command names collide.

## Summary

Keep open: current main and v2026.6.10 still lack account-scoped Slack native command names, and both direct implementation attempts were closed unmerged. The remaining work is valid, but it needs a maintainer product decision on prefix versus explicit mapping before code should be queued.

## What This Changes

The issue requests account-scoped Slack native slash command names so multiple Slack app identities can register and target `/reset`, `/compact`, `/stop`, and related commands without command-name collisions.

## Best Possible Solution

Define one supported Slack account-scoped native command naming contract, then implement it across config schema, native spec generation, Slack registration and dispatch lookup, arg-menu roundtrip, docs, and regression tests.

## Reproduction Assessment

Yes, source-level: current main has provider-scoped Slack native command generation and strict Slack slash config with no account-level native naming option. I did not run a live multi-app Slack workspace reproduction.

## Solution Assessment

Unclear: the prefix proposal is plausible, but the closed successor PR moved toward explicit nativeNames, and maintainers still need to choose the permanent config contract before implementation.
