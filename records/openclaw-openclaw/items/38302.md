## Label Justifications

- `P2`: This is a real multi-agent Slack limitation with a channel-specific blast radius and normal maintainer priority.
- `impact:message-loss`: Duplicate Slack native command names can make commands unavailable, misrouted, or handled by the wrong app or agent.
- `impact:session-state`: Commands such as reset, compact, stop, and status affect agent/session state and cannot be targeted reliably in the reported setup.

## Summary

Keep open: current main and v2026.5.22 still lack an account-level Slack native command naming contract, the two linked implementation PRs closed unmerged, and this issue remains the canonical lane for the multi-app Slack slash-command collision problem.

## What This Changes

Request account-level Slack native slash-command naming so separate Slack apps in a multi-agent workspace can register unique commands like prefixed or mapped reset, compact, stop, and status commands.

## Best Possible Solution

Define a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration, interactive menu callbacks, docs, and focused regression tests while preserving the /agentstatus exception.

## Reproduction Assessment

Yes, at source level: configure multiple Slack accounts/apps with native commands enabled and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and specify validation, collision, fallback, and interactive-menu behavior.
