## Summary

Keep open. Current main still has provider-scoped Slack native command names and strict Slack slash-command config, so the requested account-level prefix or mapping remains unimplemented and needs an API/product decision rather than cleanup closure.

## What This Changes

The issue requests account-level Slack native slash-command naming so multiple Slack apps can register distinct commands for different OpenClaw agents.

## Best Possible Solution

Ship a validated Slack account-level native-command naming contract and thread it through config schema/types, native spec generation, reverse lookup, Slack arg-menu payloads, docs, and regression tests while preserving `/agentstatus`.

## Reproduction Assessment

Yes at source and dependency-contract level: current main only derives Slack native names from provider-wide command metadata, and Slack command names are not namespaced. I did not run a live multi-app Slack workspace test.

## Solution Assessment

Unclear. Account-level native naming is the likely boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior.
