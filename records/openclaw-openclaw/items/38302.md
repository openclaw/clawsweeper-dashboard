## Label Justifications

- `P2`: This is a real multi-agent Slack native-command workflow gap with meaningful but bounded production impact and a product-direction dependency.
- `impact:message-loss`: Colliding Slack native command names can cause commands to be handled by the wrong Slack app/account or leave the intended app unreachable.
- `impact:session-state`: Commands such as reset, compact, stop, and status can affect or report the wrong agent/session when native Slack command names collide.

## Summary

Keep open: current main and latest release still lack an account-scoped Slack native-command naming contract, while both linked implementation PRs were closed unmerged. The remaining work is valid but needs maintainer direction on prefix versus explicit mapping, validation, docs, and Slack app setup expectations.

## What This Changes

Request account-level Slack native slash command naming so multi-agent Slack apps can register distinct commands such as `/oz-reset` and avoid shared-name collisions.

## Best Possible Solution

Define one supported account-scoped Slack native command naming contract, validate it, thread it through registration/lookup/arg-menu callbacks, and document manual Slack manifest setup.

## Reproduction Assessment

Yes, at source level: current Slack config/schema and native command registry have no account-scoped native command naming path. I did not run a live multi-app Slack workspace reproduction.

## Solution Assessment

Unclear until maintainers choose the contract; account-scoped native naming is the right problem boundary, but prefix-only versus explicit nativeNames remains a product/API decision.
