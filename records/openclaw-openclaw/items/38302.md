## Summary

Keep open. Current main still lacks an account-level Slack native-command naming surface, the linked implementation attempts are closed unmerged, and the remaining work is a new public Slack config/API decision rather than a cleanup-closeable duplicate.

## What This Changes

Request an account-level Slack slash-command native naming or prefix option so multiple Slack apps/agents can register distinct native commands in the same workspace.

## Best Possible Solution

Add a validated Slack account-level native command naming contract and thread it through native spec generation, registration, dispatch lookup, argument-menu callbacks, docs, and focused regression tests while preserving the existing `/status` to `/agentstatus` behavior.

## Reproduction Assessment

Yes for the current unsupported path: source inspection shows the proposed account-level naming field is absent from strict Slack config validation, and native lookup/registration are provider-scoped. I did not perform a live multi-Slack-app workspace test in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config shape. Account-level native naming is the right boundary, but the earlier PR history shows `nativePrefix` versus `nativeNames` is still a product/API decision.
