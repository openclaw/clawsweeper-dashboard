- `impact:message-loss`: Colliding Slack native command names can route commands to the wrong Slack app/account or leave the intended app unreachable.

## Summary

Keep open: current main and latest release still lack account-scoped Slack native command names, and both direct implementation PRs were closed unmerged. The remaining work is valid, but it needs a maintainer product decision on prefix versus explicit mapping before a fix should be queued.

## What This Changes

The issue requests account-scoped Slack native slash command names so multiple Slack apps in one workspace can avoid collisions for commands such as reset, compact, and stop.

## Best Possible Solution

Define one supported Slack account-scoped native command naming contract, then implement it across config schema, native spec generation, Slack registration and dispatch lookup, arg-menu roundtrip, docs, and regression tests.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 have provider-scoped Slack native command generation and strict Slack slash config with no account-level native naming option. I did not run a live multi-app Slack workspace reproduction.

## Solution Assessment

Unclear: the prefix proposal is plausible, but the closed successor PR moved toward explicit nativeNames, and maintainers still need to choose the permanent config contract before implementation.
