## Label Justifications

- `P2`: This is a real multi-agent Slack limitation with channel-specific blast radius and normal maintainer priority.
- `impact:message-loss`: Duplicate Slack native command names can make commands unavailable, misrouted, or handled by the wrong app or agent.
- `impact:session-state`: Commands such as reset, compact, stop, and status affect agent/session state and cannot be targeted reliably in the reported setup.

## Summary

Keep open: current main and v2026.5.20 still lack any account-level Slack native command naming contract, while Slack's non-namespaced slash-command behavior and multiple user reports keep the multi-app collision real. This is useful work, but it needs maintainer product/API choice rather than cleanup closure.

## What This Changes

The issue requests per-account Slack native slash command names so multiple OpenClaw agents/apps can register independently targetable commands in the same Slack workspace.

## Best Possible Solution

Define and ship a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration, interactive menu callbacks, docs, and focused regression tests while preserving the /agentstatus exception.

## Reproduction Assessment

Yes, at source level: configure multiple Slack accounts/apps with native commands enabled and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior.
