## Label Justifications

- `P2`: The request addresses a real multi-agent Slack limitation with channel-specific blast radius and normal maintainer priority.
- `impact:message-loss`: Duplicate Slack native command names can make commands unavailable, misrouted, or handled by the wrong app or agent.
- `impact:session-state`: Commands such as reset, compact, stop, and status operate on agent or session state and cannot be targeted reliably in the reported setup.

## Summary

Keep open: current main and v2026.5.20 still only support provider-wide Slack native command names, while confirmed multi-app Slack setups need an account-level naming contract. This is a real feature/API decision rather than a safe automatic bug fix.

## What This Changes

Request adds account-level Slack slash-command native naming, such as prefixes or explicit mappings, so each Slack app/agent can register distinct reset, compact, stop, and status commands.

## Best Possible Solution

Define and ship a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration/dispatch, argument-menu callbacks, docs, and regression tests while preserving the provider-wide /agentstatus alias.

## Reproduction Assessment

Yes, at source level: configure multiple Slack accounts/apps with native commands enabled and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level naming is likely the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, manifest, and interactive-menu behavior.
