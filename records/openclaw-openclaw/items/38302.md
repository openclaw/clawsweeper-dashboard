## Summary

Keep open. Current main and v2026.5.12 still expose only provider-scoped Slack native slash-command names, while the user confirmation and two closed unmerged implementation attempts show the remaining work is a deliberate account-level config/API decision.

## What This Changes

Request account-level Slack native slash-command naming so multi-agent Slack apps can register non-colliding commands.

## Best Possible Solution

Ship a validated Slack account-level native command naming contract and thread it through config schema/types, native spec generation, reverse lookup, arg-menu payloads, docs, and regression tests while preserving the existing /agentstatus override.

## Reproduction Assessment

Yes at source and dependency-contract level: Slack command names are not namespaced, and current OpenClaw only derives provider-scoped Slack native names with no account-level naming field. I did not run a live multi-app Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the likely boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior before implementation.
