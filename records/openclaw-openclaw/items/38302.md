## Summary

Keep open. Current main and the latest shipped release still lack account-level Slack native-command naming, the two implementation attempts were closed unmerged, and the thread has an independent user confirming the same multi-app Slack problem.

## What This Changes

Request account-level Slack native slash-command names or prefixes so multiple Slack app accounts can register distinct commands for different agents.

## Best Possible Solution

Ship a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration/dispatch, argument-menu callbacks, docs, and regression tests while preserving /agentstatus.

## Reproduction Assessment

Yes, at source level: configure multiple Slack accounts with native commands enabled and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior.
