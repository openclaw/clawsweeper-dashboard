## Summary

Keep open. Current main still has provider-scoped Slack native command names, the account-level naming config requested here is absent, and the linked implementation attempts are closed without merge; this needs a maintainer API choice rather than cleanup closure.

## What This Changes

The issue requests an account-level Slack native slash-command naming option so multiple Slack apps can register distinct command names for different OpenClaw agents.

## Best Possible Solution

Add a validated Slack account-level native command naming contract and thread it through native spec generation, registration, dispatch lookup, argument-menu callbacks, docs, and focused regression tests while preserving the existing `/status` to `/agentstatus` behavior.

## Reproduction Assessment

Yes for the current unsupported path: source inspection shows no account-level native naming field in strict Slack config, and native registration/lookup are provider-scoped. I did not perform a live multi-app Slack workspace test in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. Account-level native naming is the right boundary, but `nativePrefix` versus explicit `nativeNames` is still a product/config decision.
