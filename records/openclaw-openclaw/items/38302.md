## Label Justifications

- `P2`: This is a real multi-agent Slack native-command workflow gap with meaningful but bounded production impact and product-direction dependency.
- `impact:message-loss`: Colliding Slack native command names can cause commands to be handled by the wrong Slack app/account or leave the intended app unreachable.
- `impact:session-state`: Commands such as reset, compact, stop, and status can affect or report the wrong agent/session when native Slack command names collide.

## Summary

Keep open: current main and latest release still lack account-scoped Slack native command names, and both direct implementation PRs were closed unmerged. The remaining work is valid but needs maintainer direction on prefix versus explicit mapping, validation rules, docs, and manual Slack setup expectations.

## What This Changes

Requested account-scoped Slack native slash command naming so multiple Slack apps can register distinct command names for multi-agent deployments.

## Best Possible Solution

Choose the supported Slack account-scoped native command contract, then implement config schema, native spec generation, reverse lookup, arg-menu roundtrip, docs, and regression tests together.

## Reproduction Assessment

Yes, source-level: current Slack config/schema and native command registry have no account-scoped native command naming path. I did not run a live multi-app Slack workspace reproduction.

## Solution Assessment

Unclear until maintainers choose the contract; account-scoped native naming is the right problem boundary, but prefix-only versus explicit nativeNames is still a product/API decision.
