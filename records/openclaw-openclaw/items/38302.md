## Summary

Keep open. Current main still has provider-scoped Slack native slash-command names with no account-level naming contract, and the two linked implementation attempts were closed unmerged after design and validation concerns rather than superseded by a shipped fix.

## What This Changes

The issue asks for account-level Slack native slash-command naming, such as a prefix or explicit mapping, so multiple Slack apps can register distinct commands for different agents.

## Best Possible Solution

Ship a validated Slack account-level native command naming contract and thread it through config schema/types, native spec generation, reverse lookup, arg-menu payloads, docs, and regression tests while preserving the existing /agentstatus override.

## Reproduction Assessment

Yes at source and dependency-contract level: Slack slash command names are not namespaced, and current OpenClaw only generates and resolves Slack native command names by provider, not account. I did not run a live multi-app Slack workspace test. ([docs.slack.dev](https://docs.slack.dev/interactivity/implementing-slash-commands/))

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but the prior PR history shows maintainers still need to choose prefix versus explicit mapping and define validation, collision, and arg-menu semantics before implementation.
