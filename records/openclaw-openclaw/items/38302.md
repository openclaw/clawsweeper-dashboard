## Label Justifications

- `P2`: This is a valid multi-agent Slack limitation with meaningful impact but a bounded deployment-specific blast radius.
- `impact:message-loss`: Colliding native command names can route a user's command intent through an unintended Slack app or account.
- `impact:session-state`: Commands such as reset, compact, stop, and status can operate on the wrong bound agent or session.

## Summary

Current main still lacks an account-aware native-command naming seam, and both linked implementation attempts closed unmerged; the request remains valid but needs a maintainer product decision because it adds a public Slack account config contract and the prefix-only shape is not clearly the best long-term API.

## What This Changes

Add per-account Slack native command names so multiple Slack apps in one workspace can expose distinct slash commands for their bound agents.

## Best Possible Solution

Define one validated per-account Slack `nativeNames` mapping used symmetrically by registration and inbound lookup, reject collisions and invalid Slack names, preserve current names when unset, and document the required manual Slack manifest registration.

## Reproduction Assessment

Yes at source level: current native-name generation and lookup have provider context but no account context, so separate Slack accounts cannot configure distinct names. This read-only review did not exercise a live multi-app Slack workspace.

## Solution Assessment

No, the proposed prefix-only option is not yet proven to be the best API. The narrower maintainable direction is a single validated explicit mapping that covers exact Slack manifest names, reserved-name overrides, collision detection, registration, and reverse lookup.

## Risks / Open Questions

- A new account-level naming field becomes a compatibility-sensitive public configuration contract that must align types, schema, config metadata, docs, and validation.
- Registration and inbound reverse lookup must consume the same normalized mapping and reject collisions; otherwise a command can be duplicated, unreachable, or resolved to the wrong internal command.
- A prefix-only API is rigid around provider-reserved names and existing manually registered Slack manifests, while a fully free-form mapping needs careful validation and discoverability.
- The related Discord visibility and agent-targeting reports describe adjacent but distinct problems; broadening this Slack naming change to solve the whole cluster would increase scope and product ambiguity.
