## Label Justifications

- `P2`: The request addresses a real multi-agent Slack limitation with limited channel-specific blast radius and normal maintainer priority.
- `impact:message-loss`: Duplicate Slack native command names can make slash commands unavailable, misrouted, or handled by the wrong app/agent.
- `impact:session-state`: Commands such as reset, compact, stop, and status operate on agent/session state and cannot be targeted reliably in the reported setup.

## Summary

Keep open. Current main and v2026.5.19 still lack any account-level Slack native command naming surface, and the issue has independent confirmation plus two closed unmerged implementation attempts.

## What This Changes

Requests an account-level Slack native command prefix or mapping so multiple Slack apps can register distinct native slash commands for different agents.

## Best Possible Solution

Define and ship a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration/dispatch, argument-menu callbacks, docs, and regression tests while preserving /agentstatus.

## Reproduction Assessment

Yes, at source level: configure multiple Slack accounts/apps with native commands enabled and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior.
