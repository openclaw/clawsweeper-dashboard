## Label Justifications

- `P2`: This is a real multi-agent Slack native-command workflow gap with meaningful impact, but it is bounded to Slack native command deployments and needs product direction before repair.
- `impact:message-loss`: Colliding Slack native command names can cause commands to be handled by the wrong Slack app/account or leave the intended app unreachable.
- `impact:session-state`: Commands such as reset, compact, stop, and status can affect or report the wrong agent/session when native Slack command names collide.

## Summary

Keep open: current main and v2026.6.6 still lack account-scoped Slack native command names, and the two direct implementation attempts were closed unmerged. The remaining work is valid but needs a maintainer product decision on the permanent Slack config contract before a repair PR is safe.

## What This Changes

Request account-level Slack native command prefixes or custom native names so multiple Slack apps/agents can register distinct slash commands in the same workspace.

## Best Possible Solution

Choose the supported Slack account-scoped native command contract, then implement it through config schema, native spec generation, lookup, arg-menu roundtrip, docs, and regression tests.

## Reproduction Assessment

Yes at source level: current Slack native command registration and lookup carry provider context but no account-specific command-name data, and the config schema has no nativePrefix or nativeNames field. I did not run a live multi-app Slack workspace repro.

## Solution Assessment

Unclear: the proposed prefix is plausible, but one closed attempt moved toward explicit nativeNames and neither approach merged. Maintainers should choose the permanent config contract before implementation.
