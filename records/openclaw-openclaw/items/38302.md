## Summary

Keep open. Current main still has only provider-scoped Slack native slash-command names, while the issue discussion and two closed unmerged PR attempts show the remaining work is a deliberate account-level config/API design rather than a stale or already-fixed report.

## What This Changes

The issue requests an account-level Slack native slash-command naming option so multiple OpenClaw agents in one Slack workspace can avoid native command collisions.

## Best Possible Solution

Ship a validated Slack account-level native command naming contract and thread it through config schema/types, native spec generation, reverse lookup, arg-menu payloads, docs, and regression tests while preserving the existing `/agentstatus` override.

## Reproduction Assessment

Yes at source and dependency-contract level: Slack commands are not namespaced, and current OpenClaw only derives provider-scoped Slack native names with no account-level naming field. I did not run a live multi-app Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the likely boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior before implementation.
