## Label Justifications

- `P2`: This is a normal-priority multi-agent Slack improvement with concrete impact on affected deployments but a limited overall blast radius.
- `impact:message-loss`: A native slash invocation can be routed to a different Slack app than the user intended, suppressing the intended agent action.
- `impact:session-state`: Commands such as reset, compact, and stop can mutate the wrong agent session when command ownership collides.

## Summary

Keep open: current main and v2026.6.5 still use provider-wide native command names, both linked implementation PRs closed unmerged, and the remaining work requires a maintainer decision between prefixing, explicit mappings, or another canonical account-aware naming contract.

## What This Changes

Provide account-specific Slack native slash-command names so multiple Slack apps can avoid collisions and route commands to the intended agent.

## Best Possible Solution

Choose one canonical account-level native-name contract, validate Slack-safe unique names, preserve provider-wide reserved-name handling, carry the receiving account through registration and lookup, and document the required Slack manifest setup.

## Reproduction Assessment

Yes at source level: native names are resolved only by provider, Slack registration and dispatch use that shared registry, and no account-level naming field exists. This review did not rerun the scenario in a live multi-app Slack workspace.

## Solution Assessment

Unclear: account-aware naming is the right boundary, but the two closed proposals demonstrate that prefixing versus explicit mappings and their validation semantics require maintainer product judgment before implementation.
