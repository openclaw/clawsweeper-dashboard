## Summary

Keep open: current main still lacks an account-level Slack native command naming contract, the linked implementation PRs are closed unmerged, and the remaining decision affects public Slack config plus multi-agent command routing.

## What This Changes

The issue requests account-level Slack native slash-command naming so separate Slack apps in a multi-agent setup can register distinct commands and avoid Slack command ownership collisions.

## Best Possible Solution

Choose a maintainer-approved Slack account-level native naming contract, likely explicit nativeNames mapping if prior feedback still holds, then implement it across schema/types, registry generation, reverse lookup, Slack registration, arg-menu callbacks, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: current Slack native command generation and lookup are provider-scoped, and Slack account config has no nativePrefix/nativeNames field to distinguish multiple apps. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Unclear: account-level native naming is the right boundary, but the prefix proposal is not clearly the best API because the linked PR discussion shifted toward explicit nativeNames mapping and raised validation/collision issues.
