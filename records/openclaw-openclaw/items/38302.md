## Label Justifications

- `P2`: The request addresses a real multi-agent Slack limitation with channel-specific blast radius and normal maintainer priority.
- `impact:message-loss`: Duplicate Slack native command names can make commands unavailable, misrouted, or handled by the wrong app or agent.
- `impact:session-state`: Commands such as reset, compact, stop, and status operate on agent or session state and cannot be targeted reliably in the reported setup.

## Summary

Keep open because current main and the latest release still expose only provider-scoped Slack native command names, while this issue needs a new account-level Slack native-command naming contract.

## What This Changes

The issue requests account-level Slack native slash-command names or prefixes so multiple OpenClaw Slack apps can avoid command-name collisions in one workspace.

## Best Possible Solution

Define and ship a validated Slack account-level native command naming contract across config schema/types, native spec generation, reverse lookup, Slack registration/dispatch, argument-menu callbacks, docs, and regression tests while preserving /agentstatus.

## Reproduction Assessment

Yes, at source level: configure multiple Slack accounts/apps with native commands enabled and current OpenClaw still generates and resolves Slack native names only by provider. I did not run a live Slack workspace test in this read-only review.

## Solution Assessment

Unclear. Account-level native naming is the right boundary, but maintainers still need to choose prefix versus explicit mapping and define validation, collision, and interactive-menu behavior.
