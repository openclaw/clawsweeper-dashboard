## Summary

Keep open. Current main still builds a flat `<available_skills>` prompt catalog and mitigates size with static allowlists, prompt budgets, compact fallback, and truncation notices rather than hierarchical, tag-based, or lazy skill discovery. The related hook-narrowing PR and skill-graph issue are relevant follow-ups, but they are not merged/shipped and do not fully supersede this detailed tracker.

## What This Changes

Request to reduce skills prompt overhead by replacing or supplementing flat all-skills XML injection with hierarchical, tag-based, or lazy skill discovery/narrowing.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker until maintainers choose an additive contract. A maintainable end state should narrow or discover relevant skills before prompt injection, while preserving deterministic ordering, existing static allowlists, prompt budgets, plugin-bundled skill eligibility, and on-demand `SKILL.md` loading.

## Reproduction Assessment

Yes for the structural behavior: current main source, tests, and docs show a flat `<available_skills>` catalog with budget/compact fallback rather than hierarchical or lazy discovery. No for the reporter’s exact token measurements because no concrete 80-plus-skill fixture or transcript was supplied.

## Solution Assessment

Unclear. The issue identifies a real scaling limitation, but the best fix requires maintainer product/API judgment across hook-based narrowing, tags/categories, and the open skill-graph proposal rather than a narrow autonomous patch.
