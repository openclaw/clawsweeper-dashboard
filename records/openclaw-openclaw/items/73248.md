## Summary

Keep open. Current main still exposes skills through a flat `<available_skills>` prompt catalog, static allowlists, and prompt budget controls; it does not implement hierarchical, tag-based, or lazy skill discovery. The linked hook-narrowing PR #68734 remains open and unmerged, and #74100/#52861 cover overlapping broader prompt-overhead designs rather than a shipped fix.

## What This Changes

The issue requests a scalable skills discovery and prompt-injection model so large skill sets do not inject every skill description into the initial system prompt.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A maintainable end state should add an additive, documented narrowing or discovery contract, likely through hook-driven `availableSkills`/`skillsAllow`, categories/tags, or a skill graph, while preserving deterministic skill ordering, explicit allowlists, prompt budgets, bundled plugin skills, and on-demand SKILL.md loading.

## Reproduction Assessment

Yes for the structural behavior: current main source and docs show skills still flow through a flat `<available_skills>` catalog with budgeted truncation. No for the reporter's exact token measurements because the issue does not include a runnable 80-plus-skill fixture, transcript, or provider usage logs.

## Solution Assessment

Unclear. The issue identifies a real scaling limitation, but the best implementation needs maintainer API/product judgment across hook narrowing, tag/category config, lazy retrieval, and the related skill-graph proposal rather than a narrow autonomous patch.
