## Summary

Keep open. Current main still exposes skills through a flat `<available_skills>` catalog and static allowlist/budget controls; the related hook-narrowing PR #68734 is still open, so there is no shipped implementation that resolves the requested hierarchical, tag-based, or lazy skill discovery behavior.

## What This Changes

The issue requests a hierarchical, categorized, tag-based, or lazy-loading skills discovery mechanism to reduce the flat skills prompt overhead for large skill installations.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A maintainable end state should add an additive, documented narrowing or discovery contract such as categories, tags, hook-driven `skillsAllow`, or lazy retrieval while preserving deterministic ordering, explicit allowlists, prompt budgets, plugin-bundled skills, and on-demand SKILL.md loading.

## Reproduction Assessment

Yes for the structural behavior: current main source and docs show skills still flow through a flat `<available_skills>` prompt catalog. No for the reporter's exact token measurements because no runnable 80-plus-skill fixture or logs were provided.

## Solution Assessment

Unclear at the design level: the issue identifies a real limitation, but maintainers still need to choose between tag/category config, hook-based narrowing, lazy retrieval, or a skill graph. The safest path is an additive, documented contract with focused prompt-builder, hook, and config tests rather than a broad rewrite of skill loading.
