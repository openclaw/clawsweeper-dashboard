## Summary

Keep open: current main still routes Google Chat `app-url` webhook verification through the plugin-owned OAuth2/Gaxios transport path, and no exact duplicate or merged fix was found. The reporter's log matches the pinned google-auth/gaxios certificate-retrieval path, while current tests mock that dependency path instead of covering the live header contract.

## What This Changes

Google Chat inbound webhook authentication regressed after the 2026-04-29 working build and now rejects requests with `Failed to retrieve verification certificates: config.headers.has is not a function`.

## Best Possible Solution

Patch the Google Chat plugin auth runtime so google-auth/gaxios always receive compatible `Headers` through certificate retrieval while preserving SSRF guard, proxy, and mTLS behavior, then add focused regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection plus the reporter's concrete log gives a medium-confidence path through `verifyIdToken` -> google-auth certificate retrieval -> the dependency interceptor expecting `config.headers.has`.

## Solution Assessment

Yes: the narrow maintainable fix belongs in `extensions/googlechat` around the custom google-auth/Gaxios transport or header normalization. Core gateway routing, a new config option, or broad dependency changes would be a less targeted solution.
