## Summary

Keep open: current main still routes Google Chat webhook bearer verification through the same plugin-owned OAuth2Client/Gaxios transport path, and #76791 is an open closing PR for this regression. The issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Google Chat inbound webhook authentication regressed with `config.headers.has is not a function` during Google verification certificate retrieval.

## Best Possible Solution

Review and land #76791, or an equivalent plugin-local fix, so the Google Chat auth transport normalizes plain-object headers to `Headers` before google-auth/gaxios interceptors while preserving the SSRF guard and adding regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The issue is source-reproducible from the documented webhook auth flow through `verifyIdToken`, the current shared Gaxios transport, and the reporter's exact auth-rejection log.

## Solution Assessment
