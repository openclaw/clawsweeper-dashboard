## Summary

Keep open: current main still gates the stale CLI binding clear-and-fresh-retry path on `FailoverError.reason === "session_expired"`, while timeout and unknown FailoverError values from a resumed CLI session still fall through to model fallback.

## What This Changes

The issue asks OpenClaw to clear a persisted CLI session binding and retry once fresh for any FailoverError from an active resumed CLI session, not only `session_expired`.

## Best Possible Solution

Broaden the existing stale-binding recovery branch to clear and retry once fresh for FailoverError from an active persisted CLI binding, while preserving the guard that fresh-session failures do not create a retry loop.

## Reproduction Assessment

Yes by source inspection: mock runCliAgent to reject with FailoverError reason timeout or unknown while activeCliSessionBinding has a sessionId, and current main bypasses the clear-and-fresh-retry branch. I did not run the test because this review was required to keep the checkout read-only.

## Solution Assessment

Yes: reusing clearCliSessionInStore and the existing runCliWithSession(undefined) fresh retry is the narrowest maintainable fix. The implementation should keep the active persisted binding and store-path guards so non-resume failures still fall through normally.
