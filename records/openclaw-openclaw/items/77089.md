## Summary

Keep open: current main and the latest release clear a stale Claude CLI binding after non-expired `FailoverError`s, but they still do not retry a fresh CLI session for `timeout` or `unknown`; the same-turn recovery request remains, and the related fix PRs are closed unmerged.

## What This Changes

Request to make resumed CLI backend failures clear the stale session binding and retry once with a fresh CLI session for more `FailoverError` reasons than `session_expired`.

## Best Possible Solution

Add a narrow recoverable-error fresh retry in `attempt-execution` for resumed Claude CLI timeouts/unknown-style failures, while preserving clear-and-rethrow behavior for nonrecoverable provider/auth errors.

## Reproduction Assessment

Yes from source inspection, but I did not execute a live run in this read-only review. The current test fixture already models a resumed Claude CLI `timeout` and asserts one failed call with no fresh retry.

## Solution Assessment

Yes, the best fix is a bounded change in the existing catch path rather than a new fallback system. Retrying only recoverable stale-session reasons keeps operator-visible auth/billing/config failures from being masked.
