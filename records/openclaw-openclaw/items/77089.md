## Summary

Keep open: current main still gates stale CLI session clear-and-retry on `session_expired`, while CLI timeout and unclassified crash paths are thrown as `FailoverError` reasons that bypass that recovery path.

## What This Changes

Request that the CLI backend clear a persisted resumed-session binding and retry once fresh after a resumed CLI `FailoverError`, not only after `session_expired`.

## Best Possible Solution

Reuse the existing provider-scoped stale-binding clear path and one-time fresh retry for failed resumed CLI sessions, with regression coverage for timeout and unknown reasons while preserving normal fresh-session failure behavior.

## Reproduction Assessment

Yes by source inspection: mock `runCliAgent` to reject with `FailoverError` reason `timeout` or `unknown` while an active CLI session binding exists, and the current branch bypasses recovery and rethrows. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

Yes: reusing `clearCliSessionInStore` plus the existing fresh retry branch is the narrowest maintainable repair. The fix should keep the guard on an active persisted binding so fresh-session failures do not gain a retry loop.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77089. In `runAgentAttempt`, when a CLI run using an active persisted `cliSessionBinding` fails with a `FailoverError`, clear that provider's stale session binding and retry once with a fresh CLI session before falling through to model fallback; do not limit recovery to `session_expired`. Preserve the existing guard requiring an active binding plus `sessionKey`, `sessionStore`, and `storePath`; do not add retries for fresh-session failures or for the fresh retry failing. Reuse `clearCliSessionInStore` and the existing post-retry binding persistence path. Add regression coverage in `src/agents/command/attempt-execution.cli.test.ts` for `timeout` and `unknown` FailoverError reasons from a resumed binding, asserting first call uses the stale session, second call is fresh, and session store/persisted state is cleared/replaced as appropriate. Include a user-facing `CHANGELOG.md` Fixes entry. Do not change CLI watchdog defaults or add a new config option.
