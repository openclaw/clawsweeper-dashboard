## Summary

Keep open: current main still clears and retries a persisted CLI session binding only for `session_expired`, while `timeout` and `unknown` `FailoverError` paths from a resumed binding still rethrow into fallback without clearing the stale binding.

## What This Changes

Request to make CLI backend recovery clear a stale resumed session binding and retry once fresh for `FailoverError` reasons beyond `session_expired`.

## Best Possible Solution

Broaden the stale-binding recovery branch for resumed CLI sessions, add timeout and unknown regression coverage, and keep watchdog default tuning as a separate product/docs decision.

## Reproduction Assessment

Yes by source inspection: mock `runCliAgent` to reject with `new FailoverError(..., { reason: "timeout" })` or `reason: "unknown"` while `activeCliSessionBinding.sessionId` exists, and the current `session_expired` guard rethrows instead of clearing and retrying fresh.

## Solution Assessment

Yes: reusing the existing `clearCliSessionInStore` plus the already implemented one-shot fresh retry is the narrowest maintainable fix, provided the guard remains tied to a reused persisted binding so fresh-session failures do not loop.
