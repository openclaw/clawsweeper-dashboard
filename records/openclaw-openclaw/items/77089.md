## Summary

Keep open: current main still only clears and retries a persisted CLI session binding for `session_expired`, while `timeout` and `unknown` `FailoverError` paths can still rethrow from a resumed binding; open PR #77141 is the active closing implementation candidate.

## What This Changes

The issue asks to clear a persisted CLI session binding and retry once with a fresh session for any `FailoverError` from a resumed CLI backend, not only `session_expired`.

## Best Possible Solution

Land or revise #77141 so active CLI session bindings clear and retry once fresh for timeout and unknown `FailoverError` reasons, with focused regression tests and no new retry loop for fresh-session failures.

## Reproduction Assessment

Yes. By source inspection, a resumed CLI run with an active binding that throws `FailoverError` reason `timeout` or `unknown` bypasses the current `session_expired` branch and rethrows instead of clearing and retrying fresh.

## Solution Assessment

Yes. Broadening the existing guarded stale-binding recovery path is the narrowest maintainable fix, and the issue should stay open while the already-linked closing PR is reviewed or replaced.
