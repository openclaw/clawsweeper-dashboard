## Summary

Keep open: current main still only clears and retries a persisted CLI session binding for session_expired, while timeout and unknown CLI FailoverError paths can still rethrow from a resumed binding into model fallback.

## What This Changes

The issue requests clearing a stale CLI backend session binding and retrying once with a fresh session for any FailoverError raised while resuming that binding.

## Best Possible Solution

Broaden the stale-binding recovery branch in runAgentAttempt to clear and retry once fresh for FailoverError from an active reused CLI binding, with regression coverage for timeout and unknown reasons.

## Reproduction Assessment

Yes by source inspection: a mocked runCliAgent rejection with FailoverError reason timeout or unknown while an active cliSessionBinding exists will bypass the current session_expired-only branch and rethrow. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes: reusing the existing clearCliSessionInStore plus the already implemented fresh retry is the narrowest maintainable fix. The patch should keep the guard on active persisted bindings so fresh-session failures do not gain a new retry loop.
