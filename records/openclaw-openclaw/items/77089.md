## Summary

Keep open: current main and v2026.5.3 still gate stale CLI session clearing/retry on `session_expired`, while `timeout` and `unknown` are real `FailoverError` reasons; open PR #77385 is the active closing implementation candidate.

## What This Changes

Retry resumed CLI backend sessions once with a fresh session after any active-binding `FailoverError`, instead of only `session_expired`.

## Best Possible Solution

Land or revise #77385 so active CLI session bindings are cleared for all `FailoverError` reasons, while fresh-session retry remains bounded to recoverable reasons and covered by focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows a resumed CLI run with an active binding only enters the clear/retry branch for `session_expired`; `timeout` and `unknown` `FailoverError` values from the executor fall through to `throw err`.

## Solution Assessment

Yes. Broadening the existing guarded stale-binding recovery path is the narrowest maintainable fix, and the issue should stay open while the already-linked closing PR is reviewed or replaced.
