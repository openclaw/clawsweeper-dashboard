## Summary

Keep open: current main still has the reported `session_expired`-only recovery gate, and open PR #77141 uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and merged.

## What This Changes

The issue requests that resumed CLI backend sessions clear their stale binding and retry fresh for any `FailoverError`, not only `session_expired`.

## Best Possible Solution

Review and land #77141 or an equivalent narrow patch that clears and retries once fresh for `FailoverError` from an active reused CLI binding, with timeout and unknown regression coverage.

## Reproduction Assessment

Yes, by source inspection: a resumed CLI binding that throws `FailoverError` with reason `timeout` or `unknown` bypasses the current `session_expired` guards and rethrows. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes: broadening the existing clear-and-retry path for an active reused binding is the narrowest maintainable fix. The guard should stay tied to an existing binding so fresh-session failures do not gain an unintended retry loop.
