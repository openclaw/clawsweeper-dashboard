## Summary

Keep open: current main still only retries a stale CLI session fresh for `session_expired`, while `timeout` and `unknown` `FailoverError` paths can still rethrow from a resumed binding into fallback. The open #77141 PR uses closing syntax for this issue, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue asks the CLI backend to clear a persisted resumed-session binding and retry once fresh for any `FailoverError`, not only `session_expired`.

## Best Possible Solution

Review and land #77141 or an equivalent narrow patch that removes the reason guard while preserving the active resumed-binding guard and adding timeout/unknown regression coverage.

## Reproduction Assessment

Yes, by source inspection: a mocked CLI run that rejects with `FailoverError` reason `timeout` or `unknown` while a reusable session binding exists bypasses the current fresh-retry branch and rethrows. I did not execute it because this was a read-only review.

## Solution Assessment

Yes: reusing the existing clear-and-retry-fresh path for any `FailoverError` from an active resumed binding is the narrowest maintainable fix. The guard should stay limited to persisted/resumed sessions so fresh-session failures do not gain an extra retry loop.
