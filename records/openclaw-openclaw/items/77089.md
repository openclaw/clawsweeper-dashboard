## Summary

Keep open: current main still has the session_expired-only retry gate, and #77141 is an open closing PR for the same stale CLI session recovery work, so closing before merge would be premature.

## What This Changes

Request that CLI-backed sessions clear a stale resumed binding and retry once with a fresh session for FailoverError cases beyond session_expired.

## Best Possible Solution

Review and land or amend #77141 so an active persisted CLI binding is cleared and retried fresh for non-session_expired FailoverError paths, with timeout and unknown regression coverage.

## Reproduction Assessment

Yes by source inspection: a resumed CLI run that throws FailoverError with reason timeout or unknown while a binding exists bypasses the current retry branch and rethrows. I did not run tests because this review was read-only.

## Solution Assessment

Yes: reusing the existing guarded clear-and-fresh-retry path is the narrow maintainable fix, as long as it remains scoped to an active persisted CLI binding and includes regression coverage for timeout and unknown reasons.
