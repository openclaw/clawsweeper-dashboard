## Summary

Keep open: current main still has the reported `session_expired`-only stale CLI session recovery path, and an open closing PR already targets the exact fix, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests broadening CLI stale-session recovery so any `FailoverError` from a persisted resumed CLI binding clears the binding and retries once fresh.

## Best Possible Solution

Review and land, revise, or replace #77141; after a merged fix, this issue can close through the linked PR with timeout and unknown retry regression coverage in place.

## Reproduction Assessment

Yes. By source inspection, a mocked resumed CLI run that throws `FailoverError` with reason `timeout` or `unknown` while an active persisted binding exists will bypass the current `session_expired` branch and rethrow into fallback.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing clear-and-retry-fresh path for any `FailoverError` from an active reused binding while keeping the session-store guards and adding timeout/unknown regression tests.
