## Summary

Keep open: the PR targets a real source-reproducible provider failover gap, but it hard-codes provider-native mappings in the core classifier even though structured provider signals already have a provider-owned hook path. It also still needs after-fix real behavior proof beyond unit/type/static checks.

## What This Changes

The branch extends agent failover error classification and tests so code-only OpenAI, Google-family, and Anthropic structured errors map to failover reasons and no longer count as empty responses.

## Best Possible Solution

Keep the useful failover behavior, but implement provider-native code mappings in the OpenAI, Google, and Anthropic provider `classifyFailoverReason` hooks, leave the core code classifier generic, and add after-fix runtime proof.

## Reproduction Assessment

Yes from source inspection, not from a local test run because this review is read-only. Current main’s code-only classifier lacks these provider-native mappings, and the runtime-kind classifier treats message-less/status-less structured code signals as `empty_response`.

## Solution Assessment

No: this is a plausible fix to the observed bug, but not the best boundary. The existing provider `classifyFailoverReason` hook is the narrower maintainable place for provider-native OpenAI, Google, and Anthropic code mappings.
