## Summary

Keep open. Current main still lacks a native outbound LLM privacy filter, so the PR is not obsolete, but the latest PR head remains unsafe to merge because a tool-result replay shape can bypass scrubbing and the TTS summary path can silently send raw text when privacy setup fails.

## What This Changes

The PR adds a default-on privacy config, detector/replacer/mapping-store modules, outbound LLM scrubbing with inbound restoration, custom JSON5 rules, log-redaction integration, docs, and unit tests across embedded runs, BTW, and TTS summary paths.

## Best Possible Solution

Keep this work open, but require a maintainer security/architecture pass before merge. The desired end state is a documented LLM-boundary privacy contract that covers all provider replay text shapes, TTS/BTW/cache/log paths, and explicitly defines fail-open versus fail-closed behavior with focused tests for each supported message shape.

## Reproduction Assessment

Yes for the concrete PR bugs: a static read of PR head shows toolResult array blocks are filtered only for type "text", while current main provider replay consumes input_text/output_text as text; the TTS path also visibly falls back to the original text on privacy setup failure. No runtime repro was run because the checkout had to remain read-only.

## Solution Assessment

No as-is. The direction addresses a real missing privacy capability, but the current implementation still has egress-safety gaps; the safer path is to rebase, fix the remaining bypass/fail-open semantics, and have maintainers approve the product/security boundary before merge.
