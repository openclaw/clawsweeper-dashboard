## Summary

## What This Changes

Adds a default-on privacy config, detector/replacer/mapping-store pipeline, stream filtering/restoration, custom JSON5 rules, and log/TTS/BTW privacy filtering for LLM traffic.

## Best Possible Solution

Choose one secops-approved privacy boundary and land a rebased implementation with fail-closed provider paths, aligned config/schema/docs/tests, and redacted live provider/channel proof.

## Reproduction Assessment

Yes for the PR defects: PR-head source shows TTS privacy failures falling back to raw text, and toolResult input_text/output_text blocks being skipped even though OpenAI replay treats them as provider text. For the broader feature, current main lacks the requested provider-bound privacy filter.

## Solution Assessment

No as-is. The direction addresses a real privacy gap, but this implementation needs fail-closed behavior, conflict resolution, maintainer/secops architecture selection, and real runtime proof before it is a maintainable merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fail closed when TTS privacy setup fails:** `src/tts/tts-core.ts:478-480`
  - body: If privacy context setup or `filterText` throws, this catch restores the original transcript into `summaryPromptText`, and the later `completeSimple` call still sends it to the model provider. With privacy enabled, this path should skip the provider call or otherwise fail closed instead of leaking the transcript in the degraded case.
  - confidence: 0.9
- **[P1] Filter OpenAI toolResult text block shapes:** `src/privacy/stream-wrapper.ts:535-536`
  - body: `filterToolResultMessage` only rewrites `text` blocks, but OpenAI replay treats `input_text` and `output_text` blocks as provider text too. Tool outputs stored in those shapes can still carry raw secrets to the next model call, so toolResult filtering should mirror the user-message block handling.
  - confidence: 0.88
