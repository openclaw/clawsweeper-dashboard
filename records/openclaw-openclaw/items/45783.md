## Summary

Keep open: current main still lacks provider-bound privacy filtering, and this PR remains a security-sensitive feature with fail-open provider paths, overlapping active privacy work, and no real after-fix provider/channel proof.

## What This Changes

The PR adds a default-on privacy detector/replacer/mapping-store pipeline with config, custom rules, log redaction, stream wrapping, BTW, and TTS integration for LLM traffic.

## Best Possible Solution

Choose one privacy architecture, keep provider-bound privacy fail-closed or in-memory-safe on persistence failures, align config/docs/tests, and require redacted live provider/channel proof before merge.

## Reproduction Assessment

Yes for the PR defects: PR-head source shows privacy setup failures can skip filtering in provider-bound paths and TTS can fall back to raw summary text before `completeSimple`. For the broader feature, reproduction is not applicable because current main does not ship this provider-bound filtering surface.

## Solution Assessment

No as-is. The direction addresses a real privacy gap, but a default-on LLM-bound privacy layer needs maintainer architecture selection, fail-closed behavior, config/docs alignment, and live runtime proof before it is a maintainable merge path.
