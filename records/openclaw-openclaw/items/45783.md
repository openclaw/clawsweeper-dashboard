## Summary

Keep this PR open. Current main still does not provide the proposed LLM-boundary privacy pipeline, so the contribution is not obsolete, but the PR is not merge-ready because the proposed stream wrapper still leaves at least one replay path unfiltered and the fail-open behavior needs explicit maintainer security judgment.

## What This Changes

The PR adds a privacy config section, detector/replacer/mapping-store modules, outbound LLM scrub and inbound restore wrappers for embedded runs plus BTW/TTS paths, log-redaction integration, docs, and unit tests.

## Best Possible Solution

Keep the work open, but require maintainer architecture and security review before merge. The desired end state is a first-class, documented LLM-boundary privacy contract with clear config/schema semantics, explicit fail-open versus fail-closed behavior, complete coverage for provider replay block shapes, cache/log/BTW/TTS paths, and focused tests for each supported message shape.

## Reproduction Assessment

Yes for the concrete review finding: at PR head, a static read of src/privacy/stream-wrapper.ts shows toolResult array blocks are filtered only when block.type is "text", while the PR separately acknowledges OpenAI-style input_text/output_text blocks for user messages. For the broader feature request, reproduction is not a failing current-main test but current-main source inspection confirms the native privacy pipeline is absent.

## Solution Assessment

No as-is. The proposed direction addresses a real missing privacy capability, but the current branch is too security-sensitive to merge with an unfiltered tool-result replay shape and unresolved fail-open policy; a safer path is to fix those gaps or split the feature into a smaller maintainer-approved privacy boundary contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Filter OpenAI text blocks in tool results:** `src/privacy/stream-wrapper.ts:535-536`
  - body: `filterToolResultMessage` only rewrites blocks whose type is `text`, while the same wrapper handles `input_text`/`output_text` for user content. Tool results can be stored and replayed in those OpenAI-style text block shapes, so a tool-returned secret in such a block will be sent back to the provider unmasked on the next turn.
  - confidence: 0.87
- **[P2] Avoid raw TTS egress when privacy setup fails:** `src/tts/tts-core.ts:478-480`
  - body: When `createPrivacyFilterContext` throws, this catch resets `summaryPromptText` to the original text and continues to `completeSimple`. That means a locked or unwritable mapping store can silently send the unsanitized transcript to the summary model even though privacy filtering is enabled.
  - confidence: 0.75

## Security Review

Status: needs_attention

Summary: The PR is a security/privacy feature and still has concrete egress-safety concerns before merge.

Concerns:

- **[high] Tool-result text blocks can bypass scrubbing:** `src/privacy/stream-wrapper.ts:535`
  - body: The proposed toolResult filtering handles only `type: "text"` blocks, leaving OpenAI-style `input_text`/`output_text` blocks unchanged; secrets returned by tools in those shapes can be replayed to providers unfiltered.
  - confidence: 0.87
- **[medium] Privacy setup failures can fail open to provider calls:** `src/tts/tts-core.ts:478`
  - body: The PR catches privacy context initialization failures and continues with the original text in provider paths such as TTS summarization; this may be acceptable only if maintainers explicitly choose fail-open semantics for a default-on privacy boundary.
  - confidence: 0.74
