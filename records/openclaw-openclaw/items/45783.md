## Summary

Keep open. Current main does not implement LLM-boundary privacy filtering, and this PR head still has egress-safety defects in tool-result replay and TTS fail-open handling; because the change is default-on and security-sensitive, it needs maintainer security/architecture review before merge.

## What This Changes

The PR adds a default-on privacy config, detector/replacer/mapping store, outbound LLM scrubbing with inbound restoration, custom JSON5 rules, log redaction integration, docs, and tests.

## Best Possible Solution

Keep the PR open for maintainer security/architecture review, then either fix the remaining bypasses on this branch or consolidate the work with the canonical privacy implementation path.

## Reproduction Assessment

Yes for the PR findings: source inspection at PR head shows toolResult input_text/output_text bypass and TTS fail-open behavior, and current main shows those payloads flow to provider calls. No runtime test was run because this review was read-only.

## Solution Assessment

No as-is. The direction addresses a real privacy gap, but the implementation needs the remaining egress/fail-open defects fixed and maintainer approval of the default-on security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Filter OpenAI tool-result text blocks:** `src/privacy/stream-wrapper.ts:535-536`
  - body: `filterToolResultMessage` leaves every non-`text` block unchanged, but OpenAI replay converts `input_text` and `output_text` tool-result blocks into provider text. A tool-returned secret stored in those block shapes will be sent upstream raw on the next turn.
  - confidence: 0.89
- **[P2] Keep TTS summaries masked on privacy setup failure:** `src/tts/tts-core.ts:478-480`
  - body: When privacy context setup or filtering throws, this catch restores `summaryPromptText` to the original text and still calls the summary model. A busy or unwritable mapping store can therefore silently send the unfiltered transcript even though privacy filtering is enabled.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch still has concrete privacy-egress bypasses before it can merge.

Concerns:

- **[high] Tool-result text blocks can bypass scrubbing:** `src/privacy/stream-wrapper.ts:535`
  - body: The proposed toolResult filtering handles only `type: "text"` blocks, leaving OpenAI-style `input_text` and `output_text` blocks unchanged even though current replay converts them to provider text.
  - confidence: 0.89
- **[medium] TTS privacy setup failures can fail open:** `src/tts/tts-core.ts:478`
  - body: The TTS summary path catches privacy setup/filtering failures and continues with the original text, which can leak the unfiltered transcript to the configured summary model.
  - confidence: 0.78
