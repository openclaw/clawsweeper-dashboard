## Summary

Keep open: the feature is useful and the supplied Telegram screenshots are credible, but the current patch still has a blocking routing bug where allowed guest turns can bypass configured Telegram account/channel agent bindings and run under the default agent/session.

## What This Changes

The PR adds opt-in Telegram Guest Mode support through `channels.telegram.guest.enabled`, `guest_message` ingestion, isolated guest sessions, `answerGuestQuery` delivery, Telegram docs/schema updates, and tests.

## Best Possible Solution

Preserve the opt-in Guest Mode design, but resolve guest turns through the existing Telegram account/channel route binding path using a direct guest peer, then apply only the isolated guest session key for the resolved agent.

## Reproduction Assessment

Yes for the blocker: source inspection traces `bot-guest.ts` default-agent selection into `bot-message-context.ts` binding skipping. A focused regression test can configure a Telegram account/channel binding and assert the guest payload uses that bound agent/session.

## Solution Assessment

No: the Telegram plugin and `answerGuestQuery` are the right ownership/API direction, but the route-selection path is too broad because it bypasses existing account/channel bindings. The safer fix is to resolve the account/channel route first, then isolate the guest session for that resolved route.
