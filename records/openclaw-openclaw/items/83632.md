## What This Changes

## Best Possible Solution

Keep the opt-in Guest Mode direction, but resolve guest turns through existing Telegram account/channel routing before applying the isolated guest session and answerGuestQuery delivery.

## Reproduction Assessment

Yes for the blocker: source inspection traces bot-guest.ts default-agent selection into bot-message-context.ts binding skipping. A focused regression test can configure a Telegram account/channel binding and assert the guest payload uses that bound agent while keeping the guest session isolated.

## Solution Assessment

No: the Telegram plugin and answerGuestQuery owner boundary are right, but the current route-selection path is too broad because it bypasses existing account/channel bindings. The safer fix is to resolve the account/channel route first, then apply an isolated guest peer session for that resolved route.
