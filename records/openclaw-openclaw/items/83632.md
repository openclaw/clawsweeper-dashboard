## Summary

Keep open: current main still does not implement Telegram Guest Mode, and this PR contains useful live-proven feature work, but the patch is not merge-ready because guest turns still force the default agent path instead of preserving configured Telegram routing before only overriding the guest session identity.

## What This Changes

The PR adds opt-in Telegram Guest Mode handling with `guest_message` allowed-update control, isolated guest sessions, `answerGuestQuery` delivery, media-context handling, config/schema/docs updates, and Telegram tests.

## Best Possible Solution

Preserve configured Telegram route selection first, keep the guest session and one-shot delivery isolated, and then let maintainers decide whether the opt-in Guest Mode contract belongs in the bundled Telegram plugin now or should wait for the raw-ingress seam.

## Reproduction Assessment

Yes. The merge-blocking issue is source-reproducible on the PR head: `bot-guest.ts` chooses the default agent and `bot-message-context.ts` skips bindings when a session override is present, so a non-default Telegram account/channel route is bypassed.

## Solution Assessment

No. Implementing Guest Mode in the Telegram plugin is a plausible owner boundary, but this patch is not the best fix until it preserves configured route selection and isolates only the guest session/delivery path.
