## Summary

Keep open: the feature is useful and has live Telegram proof, but the current PR still has a blocking routing/security defect because guest turns are forced through the default agent instead of preserving configured account/channel routing.

## What This Changes

The PR adds opt-in Telegram Guest Mode config, allowed update/webhook/polling support, isolated guest sessions, answerGuestQuery delivery, docs, and Telegram tests.

## Best Possible Solution

Keep explicit opt-in and account-level allowFrom, but resolve guest turns through existing Telegram account/channel routing while isolating the guest session from group conversation bindings.

## Reproduction Assessment

Not applicable as a current-main bug, but the PR defect is source-reproducible: the guest handler forces `resolveDefaultAgentId` before building the route context. A focused regression would configure a Telegram account/channel binding to a non-main agent and send a guest update.

## Solution Assessment

No: the implementation should not force every guest turn to the default agent. The safer path is normal account/channel routing with guest-specific session isolation and group binding bypass only where intended.
