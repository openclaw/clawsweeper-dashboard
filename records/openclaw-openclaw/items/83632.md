## What This Changes

The PR adds opt-in Telegram Guest Mode support with `guest_message` ingress, isolated guest sessions, `answerGuestQuery` delivery, config/schema/docs updates, media context handling, and Telegram tests.

## Best Possible Solution

Repair guest routing so the Telegram account/channel route is resolved first, then build an isolated guest session from that route while keeping source-group bindings out of scope and refreshing the branch conflict.

## Reproduction Assessment

Yes for the blocking review finding: a config with a Telegram account/channel binding and `guest.enabled` will still choose `resolveDefaultAgentId` in the guest handler before constructing the guest session. The general Guest Mode feature itself is new behavior rather than a bug repro.

## Solution Assessment

No: the PR is a plausible implementation, but the best fix must preserve account/channel route selection before isolating the guest session. A safer shape is to derive the route through the existing Telegram routing path, then override only the guest-specific session/conversation identity.
