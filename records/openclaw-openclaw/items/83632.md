- `P2`: This is a meaningful Telegram feature and safety boundary with limited blast radius, not a shipped core-runtime outage.

## Summary

Keep open: the feature and live proof are meaningful, but the PR still has a blocking routing defect. Allowed guest turns force the default agent while normal Telegram DMs can honor account/channel bindings, so multi-agent operators can cross model, tool, policy, and session boundaries.

## What This Changes

The PR adds opt-in Telegram Guest Mode handling with guest_message admission, isolated guest sessions, answerGuestQuery delivery, config/schema/docs updates, media context handling, and tests.

## Best Possible Solution

Keep the opt-in Guest Mode design, but resolve the existing Telegram account/channel route first, then replace only the session peer with a guest-scoped key and cover that with an account-bound guest regression test.

## Reproduction Assessment

Yes, for the merge blocker: source inspection shows current Telegram DMs can route by account binding, while the PR guest handler forces `resolveDefaultAgentId` and skips bindings before dispatch. I did not run a live account-bound Telegram guest setup.

## Solution Assessment

No. The PR is a plausible feature implementation, but the best fix is to preserve account/channel route resolution and isolate only the guest session key, rather than bypassing the route that selects the agent.
