## Summary

Keep open: this is a useful Telegram Guest Mode PR with real proof, but it is not merge-ready because guest turns still force the default agent before applying a guest session override, which bypasses configured Telegram account/channel route bindings.

## What This Changes

Adds opt-in Telegram Guest Mode support with guest_message ingestion, answerGuestQuery delivery, isolated guest sessions, media context handling, config/schema/docs updates, and Telegram tests.

## Best Possible Solution

Preserve the opt-in Guest Mode design, but resolve Telegram account/channel route bindings first and then apply a guest-specific isolated session key for the resolved agent.

## Reproduction Assessment

Yes for the blocker at source level: bot-guest.ts fixes routeAgentId to the default agent, then bot-message-context.ts skips bindings when the guest session override is present. I did not run the added tests or a live account-bound Telegram setup in this read-only review.

## Solution Assessment

No: the Telegram plugin and answerGuestQuery path are the right ownership direction, but the current route-selection shape is too broad. The safer solution is to resolve existing account/channel route bindings first, then isolate only the guest session identity for that resolved route.
