## Summary

Keep this PR open. Current main still uses a fixed two-hour maintenance cleanup threshold, so the intended per-job stale-lock fix is not already implemented, but the PR head should not merge as-is because it caps every finite job timeout at two hours and can clear locks before valid long-running cron jobs are allowed to finish.

## What This Changes

The PR changes cron maintenance to derive stale `runningAtMs` cleanup from `resolveCronJobTimeoutMs(job) + 5 minutes`, capped by a two-hour fallback, and adds or updates cron regression tests.

## Best Possible Solution

Keep the per-job stale-lock direction, but compute the maintenance threshold so no-timeout jobs retain the existing two-hour fallback, finite timeouts within Node's timer ceiling are respected as `timeout + buffer`, and only overflow-sized values are rejected, normalized, or capped by an explicit overflow policy. Add focused tests for short custom timeouts, no-timeout fallback, valid long finite timeouts, and overflow-sized `timeoutSeconds`, plus the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is an isolated `agentTurn` cron job with `timeoutSeconds` set to three hours and `runningAtMs` just over two hours old; the PR clears the lock at the two-hour cap even though runtime accepts a three-hour `setTimeout`.

## Solution Assessment

No. The per-job threshold is the right narrow direction, but clamping all finite values to the two-hour fallback is too broad; the safer fix is to handle only values above Node's timer ceiling while preserving valid configured long timeouts.
