## Summary

Keep open: current main still has the fixed two-hour cron stuck-lock cleanup, and this PR is a focused bug fix, but the current PR head can clear valid long-running jobs too early and lacks real behavior proof from a real setup.

## What This Changes

The PR changes cron stuck-running cleanup to use each job's resolved timeout plus a buffer/fallback and adds focused regression coverage.

## Best Possible Solution

Keep the per-job cleanup direction, but preserve timeout + buffer for valid finite timeouts and use the two-hour fallback only for no-timeout or runtime-overflow-sized values, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main keeps a 30-minute job locked until the fixed two-hour threshold, while the PR head incorrectly clears a three-hour job at the two-hour clamp even though runtime accepts that timeout.

## Solution Assessment

No. The per-job threshold is the right narrow fix, but the current clamp is too broad; only no-timeout or overflow-sized values should fall back to the two-hour threshold.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve locks until valid long timeouts expire:** `src/cron/service/jobs.ts:393`
  - body: With timeoutSeconds above two hours but below Node's timer ceiling, runtime still allows the job to run for that configured duration. This clamp clears runningAtMs at the two-hour fallback, so a later maintenance tick can make the same job runnable while the original execution is still active; cap only overflow-sized values without shortening valid long timeouts.
