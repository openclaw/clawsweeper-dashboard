## Summary

Keep open. The underlying cron stale-lock bug is source-reproducible and the PR has useful threshold logic, but the branch is stale against current main, risks dropping newer post-clear skip behavior, and lacks required real behavior proof.

## What This Changes

The PR replaces the fixed two-hour cron stale-running marker threshold with a per-job threshold based on `resolveCronJobTimeoutMs()` plus a buffer, with focused regression tests.

## Best Possible Solution

Rebase the per-job threshold change onto current main, preserve the post-clear skip decision, add regression coverage for that combined behavior, and provide real cron/gateway recovery proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main still uses a fixed two-hour `STUCK_RUN_MS`, so a 30-minute agentTurn job with a 40-minute-old `runningAtMs` lock remains blocked even though the timeout policy can resolve the shorter job timeout.

## Solution Assessment

No, not as currently branched. The per-job threshold direction is appropriate, but the PR must be rebased into current main without dropping the newer post-clear skip semantics.
