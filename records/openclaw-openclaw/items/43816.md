## Summary

Keep open: the focused cron recovery fix is still useful, but the current PR head has a blocking scheduler regression for valid long timeouts and lacks required real behavior proof from the external contributor.

## What This Changes

The PR changes cron stuck-lock cleanup to derive the stale-running threshold from each job timeout plus a buffer/fallback and adds focused cron regression tests.

## Best Possible Solution

Keep the per-job cleanup direction, but preserve timeout plus buffer for valid finite timeouts and use the fallback or cap only for no-timeout and runtime-overflow-sized values, with focused regression coverage and real cron proof.

## Reproduction Assessment

Yes. A high-confidence source-level path is an agentTurn cron job with timeoutSeconds=10800 and a runningAtMs older than two hours but younger than the configured timeout plus buffer; the PR clears the lock while the runtime still accepts that timeout.

## Solution Assessment

No. The per-job threshold is the right narrow fix, but the current clamp is too broad; only undefined or Node-overflow-sized values should use the two-hour fallback rather than shortening valid long finite timeouts.

## Review Findings
