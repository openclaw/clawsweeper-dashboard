## Summary

Keep open: current main still has the fixed two-hour cron stale-marker cleanup, but this branch is not merge-ready because it loses current main's post-clear skip decision and caps long finite job timeouts at two hours before their configured execution window expires.

## What This Changes

The PR changes cron maintenance to clear stale `runningAtMs` markers using each job's resolved timeout plus a buffer, with regression tests for shorter timeout thresholds.

## Best Possible Solution

Land one canonical timeout-aware stale-marker cleanup that reuses `resolveCronJobTimeoutMs`, preserves post-clear skip behavior, avoids clearing finite long-timeout jobs early, and includes focused regression tests plus redacted real cron/gateway recovery proof.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: an `agentTurn` cron job with `timeoutSeconds: 1800` and a 40-minute-old persisted `runningAtMs` marker remains blocked because cleanup still waits for the fixed two-hour threshold.

## Solution Assessment

No, not as currently branched. The per-job threshold direction is appropriate, but the implementation must preserve current main's due-slot protection and avoid capping ordinary finite explicit timeouts at the two-hour fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the post-clear skip decision:** `src/cron/service/jobs.ts:400-401`
  - body: Current main returns from `normalizeJobTickState` right after clearing a stale marker unless the slot is known to have already executed. This branch clears `runningAtMs` and falls through with `skip: false`, so after rebase a recompute-expired maintenance pass can advance a never-executed overdue `nextRunAtMs` instead of leaving it runnable.
  - confidence: 0.9
- **[P2] Do not cap finite job timeouts at two hours:** `src/cron/service/jobs.ts:393`
  - body: The new `Math.min(..., STUCK_RUN_FALLBACK_MS)` applies to every finite timeout above the fallback ceiling, not just values that overflow `setTimeout`. A job configured to run for longer than two hours can have its lock cleared while the original execution is still within its configured timeout, allowing an overlapping run.
  - confidence: 0.87
