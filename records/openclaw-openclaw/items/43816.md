## Summary

Keep open: the bug is still source-reproducible on current main and the per-job threshold direction is useful, but this PR branch is not currently mergeable as-is because it predates newer cron maintenance semantics and lacks required real behavior proof.

## What This Changes

The PR changes cron stale-running-marker cleanup from a fixed two-hour threshold to a threshold derived from each job's resolved timeout plus a buffer, with focused regression tests.

## Best Possible Solution

Rebase the per-job threshold change onto current main, preserve the current post-clear skip contract, add/adjust regression coverage for the combined behavior, and require redacted real cron recovery proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main still uses a fixed two-hour `STUCK_RUN_MS`, so an agent-turn job with a 30-minute timeout and a 40-minute-old stale `runningAtMs` marker remains blocked even though the timeout policy can resolve a shorter execution window.

## Solution Assessment

No, not as currently branched. The per-job threshold direction is appropriate, but the PR must be rebased into current main while preserving the newer post-clear skip behavior and backed by real behavior proof.
