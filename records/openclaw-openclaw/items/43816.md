## Summary

Keep open: current main still uses the fixed two-hour stale-lock cleanup, so the PR addresses a real remaining cron behavior, but the current head still shortens valid finite timeouts above two hours and lacks the required changelog entry.

## What This Changes

The PR changes cron maintenance to derive stale `runningAtMs` cleanup from each job's resolved timeout plus a buffer/fallback and adds cron regression coverage.

## Best Possible Solution

Keep the per-job stale-lock cleanup direction, preserve finite configured timeouts through `timeout + buffer`, use the two-hour fallback only for no-timeout or overflow-sized values, and document the user-facing fix.

## Reproduction Assessment

Yes. A source-level reproduction is an `agentTurn` cron job with `timeoutSeconds` set to three hours and `runningAtMs` just over two hours old; the PR clears the lock at the two-hour cap even though runtime accepts a three-hour `setTimeout`.

## Solution Assessment

No. The per-job threshold is the right narrow fix, but clamping all finite values to the fallback is too broad; only no-timeout or overflow-sized values should fall back, and the fix needs changelog coverage.
