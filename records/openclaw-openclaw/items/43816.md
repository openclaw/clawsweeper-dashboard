## What This Changes

This PR changes cron stale `runningAtMs` cleanup from a fixed two-hour threshold to a per-job timeout-plus-buffer threshold and adds related regression tests/comment updates.

## Best Possible Solution

Review and land one canonical timeout-aware stale-marker implementation, likely https://github.com/openclaw/openclaw/pull/51024 after rebase and proof, while preserving post-clear skip behavior and long finite timeout budgets.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: an `agentTurn` cron job with `timeoutSeconds: 1800` and a 40-minute-old persisted `runningAtMs` marker remains blocked because cleanup still waits for the fixed two-hour threshold.

## Solution Assessment

No. Timeout-aware cleanup is the right direction, but this branch is not the best merge target because it caps ordinary finite long timeouts and drops current main's post-clear skip guard; the newer overlapping PR is the better canonical path after rebase/proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the post-clear skip decision:** `src/cron/service/jobs.ts:400-401`
  - body: Current main returns immediately after clearing a stale marker unless the slot is known to have already executed. This branch clears `runningAtMs` and falls through with `skip: false`, so a rebase over current main can advance a never-executed overdue `nextRunAtMs` instead of leaving it runnable.
  - confidence: 0.9
- **[P2] Do not cap finite job timeouts at two hours:** `src/cron/service/jobs.ts:393`
  - body: `Math.min(jobTimeoutMs + STUCK_RUN_BUFFER_MS, STUCK_RUN_FALLBACK_MS)` applies the fallback ceiling to every finite timeout above two hours. A job configured to run longer than two hours can have its lock cleared while the original execution is still within its configured timeout, allowing an overlapping run.
  - confidence: 0.87
