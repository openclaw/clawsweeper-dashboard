## Summary

Keep open: the per-job stale-lock direction is useful, but the current PR should not merge because it shortens valid long finite timeouts to the two-hour fallback and still lacks external real behavior proof.

## What This Changes

The PR replaces cron's fixed two-hour stale `runningAtMs` cleanup threshold with timeout-derived per-job thresholds and adds scheduler regression tests.

## Best Possible Solution

Keep the per-job stale-lock cleanup direction, but preserve `timeout + buffer` for valid finite timeouts and reserve the fallback/cap for no-timeout or Node-overflow-sized values, with focused regression coverage and real cron proof.

## Reproduction Assessment

Yes. A high-confidence source-level path is an `agentTurn` cron job with `timeoutSeconds=10800` and a `runningAtMs` older than two hours but younger than the configured timeout plus buffer; the PR clears the lock while runtime still accepts that timeout.

## Solution Assessment

No. The per-job threshold is the right narrow fix, but the current clamp is too broad; only undefined or Node-overflow-sized values should fall back instead of shortening valid finite timeouts.
