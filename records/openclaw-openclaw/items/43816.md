## Summary

Keep this PR open: current main still uses a fixed two-hour cron stuck-lock threshold, but the PR head still shortens valid finite timeouts above two hours and lacks the required user-facing changelog entry.

## What This Changes

This PR changes cron maintenance to clear stale `runningAtMs` markers from each job's resolved timeout plus a five-minute buffer, with a two-hour fallback/cap, and adds cron regression coverage plus a test comment update.

## Best Possible Solution

Keep the per-job stale-lock direction, but let finite timeouts within the runtime timer ceiling use `timeout + buffer`, keep the two-hour fallback for no-timeout jobs, and handle overflow-sized values with an explicit cap/rejection policy plus focused tests and changelog coverage.

## Reproduction Assessment

Yes. A source-level reproduction is an `agentTurn` cron job with `timeoutSeconds` set to three hours and `runningAtMs` just over two hours old; the PR clears the lock at the two-hour cap even though runtime accepts a three-hour `setTimeout`.

## Solution Assessment

No. The per-job threshold is the right narrow fix, but clamping all finite values to the fallback is too broad; only overflow-sized values should be capped or rejected.
