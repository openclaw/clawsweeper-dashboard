## Summary

Keep open: current main still has the fixed two-hour cron stuck-lock threshold, so the bug remains relevant, but this PR head is not mergeable as-is because it drops newer current-main post-clear skip semantics and lacks after-fix real behavior proof.

## What This Changes

The PR changes cron maintenance to clear stale `runningAtMs` markers using `resolveCronJobTimeoutMs(job)` plus a five-minute buffer, with regression tests for shorter per-job thresholds.

## Best Possible Solution

Land the per-job stuck-lock threshold on top of current main while preserving due-slot protection, with focused regression coverage and one redacted real cron/gateway recovery proof artifact.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a 30-minute `agentTurn` job with a 40-minute-old `runningAtMs` marker is still below the fixed two-hour clearing threshold even though the timeout policy can resolve the shorter job timeout.

## Solution Assessment

No, not as currently branched. The per-job threshold direction is appropriate, but the PR must be rebased so clearing a stale marker still preserves current main's post-clear skip decision for never-executed due slots.
