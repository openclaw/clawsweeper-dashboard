## Summary

Keep open: the OOM hardening class is valid and current main still has unbounded provider success reads, but this branch is not merge-ready because it duplicates the shared provider JSON reader, uses fixed 1 MiB caps on compatibility-sensitive paths, overlaps merged and narrower open fixes, and has only mock/test proof.

## What This Changes

The PR replaces selected provider success-path `response.json()` calls with bounded `readResponseWithLimit` parsing across Vydra, xAI, Runway, PixVerse, DeepInfra, Together, ElevenLabs, Microsoft Foundry, and Microsoft speech paths, plus two test mocks.

## Best Possible Solution

Rebase or split this batch into narrow provider fixes that reuse `readProviderJsonResponse` or a payload-specific shared wrapper, omit already-fixed overlap, and add real streamed `Response` tests plus redacted terminal proof.

## Reproduction Assessment

Yes, at source level: current main still has direct successful `response.json()` reads in several targeted external-provider paths. A streamed oversized `Response` harness would reproduce the unbounded-buffering class, but this read-only review did not execute it.

## Solution Assessment

No. Bounded reads are the right fix class, but this PR should use the shared provider JSON reader, choose payload-appropriate caps, preserve current-main overlap, and add real behavior proof before it is the best solution.
