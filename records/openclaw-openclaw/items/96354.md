## Summary

Keep open: the hardening goal is valid, but this broad branch is not merge-ready because it reimplements provider JSON parsing with local 1 MiB caps, overlaps merged and narrower active fixes, and still lacks proof through the changed provider/auth/image/video paths.

## What This Changes

The PR replaces selected provider success JSON parses in Vydra, xAI, Runway, PixVerse, DeepInfra, Together, ElevenLabs, Microsoft speech, and Microsoft Foundry with local 1 MiB `readResponseWithLimit` reads and adds generic bounded-reader proof/tests.

## Best Possible Solution

Split or rebase into narrow provider-family fixes that preserve merged/open overlaps, use `readProviderJsonResponse` or payload-specific shared caps, and include provider-path streamed-response proof.

## Reproduction Assessment

Yes, source-reproducible: current main still has direct successful `response.json()` reads in several targeted external-provider paths, and the PR body demonstrates generic bounded-reader cancellation versus unbounded buffering. I did not execute a live OOM reproduction in this read-only review.

## Solution Assessment

No. Bounded reads are the right fix class, but this PR should reuse the shared provider JSON reader, choose payload-appropriate caps, preserve current-main overlap, and provide representative provider-path proof.
