## Summary

Keep open: the branch still targets real unbounded JSON reads on current main, but it is not a clean landing path because it duplicates the shared provider JSON reader, applies fixed 1 MiB caps to compatibility-sensitive provider/auth paths, overlaps narrower proof-positive PRs, and its new proof harness is not representative enough.

## What This Changes

The PR replaces selected provider success-path `response.json()` calls with `readResponseWithLimit` parsing across Vydra, xAI, Runway, PixVerse, DeepInfra, Together, ElevenLabs, Microsoft Foundry, and Microsoft speech paths, plus two test mocks and a generic HTTP proof test.

## Best Possible Solution

Split or rebase this into narrow provider fixes that reuse `readProviderJsonResponse` or payload-specific shared helpers, preserve already-fixed overlap, and add representative streamed `Response` proof for each risk surface.

Yes, at source level: current main still has direct successful `response.json()` reads in several targeted external-provider paths. I did not execute a failing current-main OOM repro in this read-only review.

## Solution Assessment

No. Bounded reads are the right fix class, but this PR should reuse the shared provider JSON reader, choose payload-appropriate caps, preserve current-main overlap, and provide representative provider-path proof.
