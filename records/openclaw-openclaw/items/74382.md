## Summary

Keep open. Current main still has the reported onboarding UX problem: the skill install picker can show the keyless local `openai-whisper` skill, then the same `setupSkills` pass prompts for every missing `primaryEnv` skill without tying credential prompts to the selected install choices or adding clarifying transition copy. The issue also has open closing PRs #74407 and #74891, so it should remain open until one lands or is replaced.

## What This Changes

Fix the skills onboarding flow so keyless local OpenAI Whisper setup is not followed by misleading OpenAI or ElevenLabs API-key prompts unless the keyed skills are actually being configured.

## Best Possible Solution

Keep this issue open until a linked PR lands. The desired end state is a narrow `setupSkills` fix with regression coverage: local `openai-whisper` remains keyless, `OPENAI_API_KEY` and `ELEVENLABS_API_KEY` prompts are clearly scoped to the actual env-gated skills, and skipped binary-backed skills do not receive surprise credential prompts unless that is an intentional product decision.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: make skill status contain missing keyless `openai-whisper` plus missing env-gated `openai-whisper-api` or `sag`; current `setupSkills` can show the install list first and then prompt for every missing `primaryEnv` skill.

## Solution Assessment

Unclear until a linked PR is reviewed. The narrowest maintainable solution appears to be selected-skill gating plus clear credential-phase copy in `src/commands/onboard-skills.ts`, with focused tests in `src/commands/onboard-skills.test.ts`; copy-only clarification may not cover the skipped-install case described by #74891.
