## Summary

Keep open. Current main still has the reported onboarding mismatch: the dependency picker can show the keyless local `openai-whisper` skill, then the credential loop prompts for every missing env-gated skill, including separate `openai-whisper-api` and `sag` skills. The issue also has open closing PRs #74407 and #74891, so it should remain open until one lands or is replaced.

## What This Changes

The issue asks onboarding to stop appearing to request OpenAI or ElevenLabs API keys for the keyless local `openai-whisper` skill after the install list says no API key is needed.

## Best Possible Solution

Keep the issue open until a linked fix lands. The desired end state is a narrow onboarding change with regression coverage: local `openai-whisper` remains visibly keyless, credential prompts are scoped to actual keyed skills being configured, and the UI copy makes `openai-whisper-api` and `sag` distinct from the local Whisper CLI skill.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: make `openai-whisper` missing by binary, leave `OPENAI_API_KEY` or `ELEVENLABS_API_KEY` unset for the separate keyed skills, run the skills onboarding path, and observe that the install picker and all-missing credential loop are independent.

## Solution Assessment

No for current main. The narrow maintainable solution is to gate credential prompts by the selected install/configuration path for bin-backed skills, with explanatory copy for truly separate env-only or keyed skills; PR #74891 appears closer to that boundary than a clarification-only change, but maintainers should choose the canonical implementation.
