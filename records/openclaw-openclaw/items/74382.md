## Summary

Keep open. Current main still has the reported onboarding UX problem: the skill install picker can list the keyless local `openai-whisper` skill, then the same `setupSkills` pass prompts for missing `primaryEnv` credentials on other missing skills such as `openai-whisper-api` and `sag`. The issue is also already paired with open closing PRs #74407 and #74891, so it should stay open until one lands or is replaced.

## What This Changes

The report asks onboarding not to prompt for OpenAI Whisper or ElevenLabs API keys immediately after listing a keyless local `openai-whisper` skill as not requiring an API key.

## Best Possible Solution

Keep this issue open until a linked PR lands. The desired end state is a narrow `setupSkills` fix with regression coverage: local `openai-whisper` remains clearly keyless, credential prompts are scoped to the actual env-gated skills the user is configuring, and skipped binary-backed skills do not receive surprise API-key prompts unless that is an explicit product choice.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: make skill status contain missing keyless `openai-whisper` plus missing env-gated `openai-whisper-api` or `sag`; `setupSkills` can show the install list first and then prompt for every missing `primaryEnv` skill.

## Solution Assessment

Unclear which open PR is the best final patch until maintainers review them. Current main is not the best solution because it prompts from the full missing set; the narrow maintainable fix should gate credential prompts where appropriate and add focused tests for the mixed keyless/keyed onboarding sequence.
