## Summary

Keep open. Current main still has the reported onboarding UX path: the install step can advertise keyless local Whisper setup, then the credential loop prompts for every missing env-gated skill, including separate OpenAI Whisper API and ElevenLabs skills. Open PRs #74407 and #74891 both target this issue, so it should stay open until one lands or is replaced.

## What This Changes

The issue requests that `openclaw onboard` stop appearing to ask for OpenAI or ElevenLabs API keys after advertising keyless local OpenAI Whisper setup, or clearly scope those prompts to the actual keyed skills.

## Best Possible Solution

The end state should be a narrow `setupSkills` change with regression coverage: local `openai-whisper` remains keyless, credential prompts are clearly scoped to env-gated skills, and skipped binary-backed keyed skills do not prompt for secrets unless the user opted into configuring them.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: make skill status contain missing keyless `openai-whisper` plus missing env-gated `openai-whisper-api` or `sag`, then run `setupSkills`; current main can show the install list first and later prompt for every missing `primaryEnv` skill.

## Solution Assessment

No, current main is not the best solution because it still conflates the install and credential phases. The safer fix is selected-skill gating plus clear credential-phase copy in `src/commands/onboard-skills.ts`, with focused tests in `src/commands/onboard-skills.test.ts`.
