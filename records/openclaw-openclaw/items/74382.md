## Summary

Keep open. Current `main` still has the reported onboarding UX problem: skill dependency installation is presented first, then the same `setupSkills` flow prompts for every missing `primaryEnv` skill without a selected-skill gate or transition copy. The issue is also paired with open implementation PRs #74407 and #74891, so it should not be closed before one lands or is replaced.

## What This Changes

The issue reports that `openclaw onboard` lists the local keyless `openai-whisper` skill but later prompts for OpenAI/ElevenLabs API keys, making the setup flow appear contradictory.

## Best Possible Solution

Keep this issue open until a linked PR lands. The desired end state is a narrow `setupSkills` fix with regression coverage: local `openai-whisper` remains keyless, prompts for `OPENAI_API_KEY` and `ELEVENLABS_API_KEY` are clearly scoped to the actual env-gated skills, and skipped binary-backed skills do not receive surprise credential prompts unless that is an intentional product choice.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: make skill status contain missing keyless `openai-whisper` plus missing env-gated `openai-whisper-api` or `sag`; current `setupSkills` can show the install list first and then prompt for every missing `primaryEnv` skill.

## Solution Assessment

Unclear until PR review. The narrowest maintainable fix appears to be selected-skill gating plus clear credential-phase copy in `src/commands/onboard-skills.ts`, with focused tests in `src/commands/onboard-skills.test.ts`; copy-only clarification may not cover the skipped-install variant described by #74891.
