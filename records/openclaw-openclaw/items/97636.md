## Summary

Keep open: current main still has the reported Control UI storage path where a missing scoped settings record can fall back to the legacy global settings object, and that `gatewayUrl` is then used for the WebSocket connection. The issue is narrow, source-reproducible, and not owned by any open merged or viable replacement PR.

## What This Changes

The issue reports that Control UI can reuse a legacy saved gateway URL from one base-path deployment when opening another deployment on the same origin, causing the WebSocket to connect to the wrong Gateway.

## Best Possible Solution

Fix `ui/src/ui/storage.ts` so stale same-origin, different-base-path legacy fallback data cannot override the page-derived Gateway URL, while preserving deliberate custom Gateway URLs and adding focused storage regression coverage.

## Reproduction Assessment

Yes by source inspection: with no scoped settings for the current derived URL, `loadSettings()` can read the legacy global settings object and carry its `gatewayUrl` into the WebSocket client. I did not run a live reverse-proxy setup in this read-only review.

## Solution Assessment

Yes, the best repair is in the Control UI storage loader/persistence boundary, not a new config option or Gateway routing change. The fix should be regression-tested with path A saved first, then path B loaded without a path-B scoped record.
