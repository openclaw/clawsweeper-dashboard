## Summary

Keep open. Current main and v2026.6.10 still let missing path-scoped Control UI settings fall through to the shared legacy localStorage value, and the WebSocket client uses that stored `gatewayUrl`, so the wrong-gateway connection remains source-reproducible.

## What This Changes

Fix Control UI settings loading so path-based deployments do not reuse a stale saved WebSocket gateway URL from a different base path.

## Best Possible Solution

Teach Control UI storage to distinguish page-derived gateway URLs from explicit user overrides, stop a shared legacy record from selecting a sibling base-path backend, and add a regression test for visiting path A then path B with no path-B scoped settings.

## Reproduction Assessment

Yes, source-reproducible. On a `/gateway-b` page with no scoped settings, `loadSettings()` can fall through to the unscoped legacy record from `/gateway-a`, and the Gateway client then opens its WebSocket to that stored URL.

## Solution Assessment

Yes, the narrow storage-layer fix is the best direction. The Gateway server and WebSocket client already honor the selected URL, so the repair should stay in `loadSettings()`/`persistSettings()` plus regression coverage rather than adding config or runtime routing policy.
