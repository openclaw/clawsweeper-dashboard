- `impact:message-loss`: A user can view or send chat through the wrong Gateway when the browser reuses another base path's saved WebSocket URL.

## Summary

Keep open: current main and v2026.6.10 still let a missing current scoped settings record fall back to the legacy global settings entry, and the loaded `gatewayUrl` is passed directly to the Control UI WebSocket client. The related merged base-path fixes cover default URL derivation and scoped settings keys, but not this residual stale legacy fallback.

## What This Changes

The issue requests a fix so Control UI pages served from different base paths on the same origin do not reuse another Gateway's saved WebSocket URL.

## Best Possible Solution

Fix `ui/src/ui/storage.ts` so stale same-origin, different-base-path legacy fallback data cannot override the page-derived Gateway URL, while preserving intentional custom Gateway URLs and adding focused storage regression coverage.

## Reproduction Assessment

Yes by source inspection: seed the legacy global settings key with Gateway A's `gatewayUrl`, load Gateway B with no B-scoped settings key, and `loadSettings()` can return Gateway A's URL. I did not run the live reverse-proxy setup in this read-only review.

## Solution Assessment

Yes. The best fix belongs at the Control UI storage load/persistence boundary, not in Gateway routing or a new config option, with a regression test for path A saved first and path B loaded without its own scoped record.
