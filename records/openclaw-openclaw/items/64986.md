## Summary

Keep open. Current main still leaves the built-in macOS Test Voice Wake path mismatched with its UI contract: the UI asks users to say only a trigger word, but the tester reports success only when it extracts non-empty command text after the trigger. The related Mac mini report is already closed as a duplicate of this issue, so this remains the canonical source-backed Voice Wake cleanup item.

## What This Changes

The issue reports that macOS companion Voice Wake and the built-in Test Voice Wake path fail on version 2026.4.10 despite permissions, enabled Voice Wake settings, trigger words, selected microphone, and explicit en-US locale.

## Best Possible Solution

Fix the macOS companion owner path by making VoiceWakeTester follow the same trigger-only pause semantics as VoiceWakeRuntime, or deliberately update the UI contract if maintainers require a command after the trigger. Add focused Swift coverage for trigger-only tester success and ordinary trailing-mention rejection, then live-retake the Voice Wake and Trigger Talk Mode flows on macOS.

## Reproduction Assessment

Yes for the built-in tester path: current source shows that saying only a configured trigger word can reach final recognition but still fail because the tester requires non-empty command text. Live reproduction of the always-on runtime path is still unclear without macOS microphone/TCC hardware validation.

## Solution Assessment

Yes, the narrowest maintainable solution is to align the tester’s detection semantics with the documented/runtime trigger-only behavior and cover that path with Swift tests. Changing only docs or asking for more logs would leave the source-backed tester mismatch in place.

## Repair Work Prompt

Fix the macOS companion Voice Wake built-in tester so it accepts the trigger-only flow promised by the UI and documented/runtime behavior. In current main, VoiceWakeTestCard tells users to say a trigger word, but VoiceWakeTester only reports .detected when match.command is non-empty, and the fallback path uses minCommandLength: 1, so bare trigger speech can fail as No trigger heard. Align VoiceWakeTester and its timeout/silence fallback with VoiceWakeRuntime trigger-only pause semantics, preserving false-positive protection for ordinary speech that merely mentions a trigger. Add focused Swift tests proving trigger-only tester success for configured triggers and rejection for trailing ordinary mentions. Do not change provider/TTS behavior, Talk Mode playback, global routing semantics, or bundled trigger defaults. Update docs only if behavior text changes, and add a user-facing CHANGELOG.md fix entry crediting Thanks @zoiks65; mention #59767 as duplicate context if useful. Validate with targeted macOS Swift tests and Swift formatting; run pnpm check:changed in Testbox before handoff if macOS app/source changes are made.
