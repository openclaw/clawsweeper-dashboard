## Summary

Keep open: current main still leaves the commitments store write paths vulnerable to lost updates, and there is an open fix PR that explicitly closes this issue, so the right cleanup path is to review or land that PR rather than closing this issue early.

## What This Changes

The issue reports that concurrent commitments-store load-modify-save mutations can overwrite each other and lose dismissed, sent, or attempted state.

## Best Possible Solution

Land a commitments-store serialization fix that protects status, attempt, inferred-upsert, expiry, and legacy rewrite mutations with same-process queueing and cross-process locking, then close this issue through the merged PR.

## Reproduction Assessment

Yes. The issue gives a deterministic `Promise.all` repro, and current main source still shows both concurrent callers can load the same snapshot and save stale whole-store state; I did not run it because this pass is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to serialize commitments-store mutations around the existing load-modify-save critical sections, with cross-process locking because CLI and gateway writers can be separate processes.
