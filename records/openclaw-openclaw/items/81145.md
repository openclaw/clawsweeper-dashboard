## Summary

Keep open: current main still has the unsynchronized commitments store load-modify-save paths, and the open linked fix PR has not merged yet. This is a valid narrow data-loss bug, but the issue should remain open until https://github.com/openclaw/openclaw/pull/81153 or an equivalent fix lands.

## What This Changes

The issue reports lost commitments store updates when concurrent status, attempt, or upsert writes race through whole-file load-modify-save paths.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81153, or an equivalent commitments-local writer queue plus cross-process file lock with regression tests, then close this issue after merge.

## Reproduction Assessment

Yes. The issue includes a deterministic `Promise.all` repro against exported store functions, and current main source still shows independent whole-file load/mutate/save paths without serialization; I did not run it in this read-only review.

## Solution Assessment

Yes. Serializing commitments-store mutations with a commitments-specific queue plus `withFileLock` is the narrowest maintainable fix because it covers both same-process races and CLI-versus-gateway writers without changing public mutator signatures.
