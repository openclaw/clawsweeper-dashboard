## Summary

Current main still has the unsynchronized commitments-store mutation paths, so this is a valid data-loss bug. The issue should stay open because an open PR already references it with closing syntax and is the active fix path.

## What This Changes

The issue reports lost updates in `~/.openclaw/commitments/commitments.json` when concurrent commitments mutators perform unsynchronized whole-file load-modify-save writes.

## Best Possible Solution

Land a narrow commitments-store serialization fix that protects all load-modify-save mutations with same-process queuing and cross-process file locking, then close this issue with the merged commit and release provenance.

## Reproduction Assessment

Yes. The issue provides a deterministic `Promise.all` repro against exported store functions, and current main source still shows both concurrent calls can read the same store snapshot and later save stale whole-file state.

## Solution Assessment

Yes. The best fix is the narrow storage-layer serialization already proposed by the open PR: protect commitments-store mutation helpers rather than changing heartbeat delivery policy or adding config.
