## Summary

Keep open: current main still has unsynchronized commitment-store load-modify-save paths, the documented behavior requires dismissals and delivery limits to persist, and no duplicate or merged fix surfaced in the canonical search.

## What This Changes

The issue asks OpenClaw to serialize commitment-store writes so concurrent status, attempts, and inferred-upsert mutations cannot overwrite each other.

## Best Possible Solution

Add a narrow commitment-store write helper that serializes each resolved store path and protects the load-mutate-save critical section, with focused regression coverage for concurrent status and attempts updates.

## Reproduction Assessment

Yes. The issue gives a deterministic Promise.all repro, and current source inspection confirms both writers can read the same snapshot and then write full-file replacements; I did not run it because this review is read-only.

## Solution Assessment

Yes. The maintainable fix is to serialize commitment-store mutations at the store helper boundary, preferably combining an in-process per-path queue with the existing file-lock primitive rather than changing heartbeat, CLI, or commitment product semantics.
