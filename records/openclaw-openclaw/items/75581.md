## Summary

Keep open: the proposed SDK seam is useful, but the current PR head still drops the live config before outbound attachment delivery, and the supplied terminal proof does not exercise a bundled plugin calling the public `api.sendSessionAttachment` loader path.

## What This Changes

Adds `OpenClawPluginApi.sendSessionAttachment`, host-side attachment validation/routing, delivery hints, Telegram formatting exports, Gateway send options, docs, changelog, and regression coverage.

## Best Possible Solution

Keep the bundled-only seam, pass the live config through route lookup, file validation, and outbound delivery, then add public SDK-path real behavior proof before final maintainer review.

## Reproduction Assessment

Not applicable for a current-main bug because this is an additive Plugin SDK feature. The blocking PR regression is source-reproducible by following `sendPluginSessionAttachment` into a `sendMessage` call that omits `cfg`.

## Solution Assessment

No, not merge-ready as written: the seam shape is plausible, but outbound delivery must reuse the same live config chosen for route lookup and path validation, and proof should exercise the public SDK loader path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass the live config into attachment delivery:** `src/plugins/host-hook-workflow.ts:278-292`
  - body: `sendPluginSessionAttachment` resolves the session route and attachment paths with `params.config`, but then calls `sendMessage` without `cfg`. `sendMessage` falls back to the process runtime config when `cfg` is absent, so after a config reload or per-agent config change this can validate one route and deliver with stale channel/account/media policy. Pass `cfg: params.config` into this call and cover the handoff.
  - confidence: 0.88
