## Summary

Keep open. Current main still lacks the requested `sendSessionAttachment` SDK surface, and this PR is the active implementation slice. The latest head fixes the earlier MarkdownV2 contract mismatch by narrowing the public hint to HTML and adds route-lookup hardening, but the new candidate ranking can still select a newer non-routable alias over a routable canonical entry, so attachment delivery can fail even when a valid route exists.

## What This Changes

The PR adds a bundled-plugin `sendSessionAttachment` API, host-side file validation and delivery through the active session route, Telegram/Slack attachment hints, outbound parse-mode plumbing, Telegram helper re-exports, tests, generated SDK baseline hashes, and a changelog entry.

## Best Possible Solution

Keep the additive bundled-plugin attachment seam, but change delivery lookup to prefer any routable candidate over non-routable aliases before using freshness as a tie-breaker. Add a regression where a newer alias lacks route metadata while the canonical entry is routable, then rerun the focused plugin/session/Telegram checks and the failing exact-head gates.

## Reproduction Assessment

Yes. On the latest PR head, create a session store with a newer mixed-case alias entry that has no routable route and a canonical lower-case entry with `lastChannel`/`lastTo`; `extractDeliveryInfo(aliasKey)` selects the newer non-routable entry and attachment delivery reports no active route.

## Solution Assessment

No, not as currently written. The seam direction is maintainable and the earlier MarkdownV2 mismatch is addressed, but route candidate selection should rank routability first and freshness second so stale or partial aliases cannot mask a valid canonical route.
