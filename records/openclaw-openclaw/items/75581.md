## Summary

Keep open: current main does not already implement this SDK seam, and the PR is still a useful active implementation, but the latest head still drops the live config before outbound delivery and the supplied proof does not exercise the public plugin API path.

## What This Changes

The PR adds bundled-plugin `OpenClawPluginApi.sendSessionAttachment`, host-side attachment validation and route lookup, channel delivery hints, Telegram formatting exports, gateway send options, docs, changelog, and regression coverage.

## Best Possible Solution

Keep the bundled-only host-mediated seam, pass the live config through the entire delivery path, add public SDK-path proof, then let maintainers review the additive Plugin SDK contract.

## Reproduction Assessment

Not applicable for the feature request itself; there is no current-main bug to reproduce. The blocking config handoff is source-reproducible on the PR head by following `sendPluginSessionAttachment` from config-based route/path resolution to the `sendMessage` call that omits `cfg`.

## Solution Assessment

No, not merge-ready as written: the seam shape is plausible, but downstream delivery should reuse the same live config and proof should exercise the public SDK API rather than only the internal helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Pass the live config into attachment sends:** `src/plugins/host-hook-workflow.ts:278-292`
  - body: `sendPluginSessionAttachment` uses `params.config` to find the session route and resolve attachment paths, but then calls `sendMessage` without `cfg`. When the plugin runtime supplies a live config that differs from the process snapshot, outbound target/account resolution, channel config, and media policy can use stale/default settings after route lookup, so the same attachment may fail or send with the wrong channel settings. Pass `cfg: params.config` into this call and cover that handoff.
  - confidence: 0.82
