## Summary

Keep open: this is a recent external feature PR adding a public Plugin SDK/Gateway seam that is not on current main. Source review found one config propagation bug, and the supplied proof still bypasses the public plugin API path.

## What This Changes

The PR adds bundled-only `OpenClawPluginApi.sendSessionAttachment`, host-side file/session-route validation, outbound delivery hints, Gateway send options, Telegram helper exports, docs, changelog, and contract coverage.

## Best Possible Solution

Keep the bundled-only host-mediated seam, pass the same live config through the whole delivery path, and require real proof from a bundled plugin using the public SDK API before maintainer API review and merge.

## Reproduction Assessment

Not applicable for the feature request itself: there is no current-main bug to reproduce. The config propagation finding is source-reproducible by following `sendPluginSessionAttachment` from config-based route lookup to a `sendMessage` call that omits `cfg`.

## Solution Assessment

No, not merge-ready as written: the seam shape is plausible, but downstream delivery should reuse the live config and proof should exercise the public SDK API rather than only the internal helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Pass the live config into attachment sends:** `src/plugins/host-hook-workflow.ts:278`
  - body: At this point the helper has already resolved the session route and attachment paths with `params.config`, but the downstream `sendMessage` call omits `cfg`. When the plugin runtime supplies a live config that differs from the process snapshot, outbound target/account resolution and media policy can use stale/default config after route lookup, so the same attachment may fail or send with the wrong channel settings. Pass `cfg: params.config` into `sendMessage` and cover that handoff.
  - confidence: 0.79
