## Summary

Keep open: current main lacks the requested SDK seam, and this active PR still has a source-proven live-config handoff bug plus insufficient public SDK-path proof.

## What This Changes

The PR adds a bundled-only `OpenClawPluginApi.sendSessionAttachment` seam with host-side file validation/session-route delivery, channel attachment hints, Telegram formatting exports, Gateway send options, docs, changelog, and contract tests.

## Best Possible Solution

Keep the additive bundled-only SDK seam, pass the selected live config through outbound delivery, and require redacted public SDK/loader-path runtime proof before maintainer merge review.

## Reproduction Assessment

Not applicable for the feature request itself because this is new SDK surface, not a current-main bug. The blocking PR regression is source-reproducible by tracing `sendSessionAttachment` from the registry into `sendPluginSessionAttachment`, then into a `sendMessage` call that omits `cfg`.

## Solution Assessment

No, not merge-ready as written. The seam direction is plausible, but the delivery call should reuse the live config already selected by the registry and proof should exercise the public Plugin SDK/loader path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass the live config into attachment sends:** `src/plugins/host-hook-workflow.ts:278-292`
  - body: `sendPluginSessionAttachment` receives the live config and uses it for session-route lookup and workspace-relative file validation, but then calls `sendMessage` without `cfg`. `sendMessage` falls back to the process runtime config when `cfg` is omitted, so after a config reload the attachment can be routed and validated against one snapshot, then delivered with stale channel/account/media policy. Pass `cfg: params.config` into this call and cover the handoff.
  - confidence: 0.88
