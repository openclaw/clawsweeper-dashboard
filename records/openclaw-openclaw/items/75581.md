## Summary

Keep this PR open: current main does not have the new session-attachment API, but the PR is not merge-ready because the attachment path still drops the live config before outbound delivery and the supplied real-behavior proof does not exercise the public plugin API path.

## What This Changes

The PR adds bundled-plugin `api.sendSessionAttachment`, host-side attachment validation and session-route delivery, channel hints, Telegram formatting exports, gateway send options, docs, changelog, and regression coverage.

## Best Possible Solution

Keep the bundled-only seam, forward the live config into outbound delivery, cover that handoff, and require public `api.sendSessionAttachment` real-behavior proof before SDK-surface maintainer approval.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR head uses `params.config` for route/path selection but calls `sendMessage` without `cfg`, while `sendMessage` accepts and consumes `cfg`. Reproduction is not applicable to current main as a bug because the requested API is not on main.

## Solution Assessment

No, not merge-ready as written: the seam direction is plausible, but downstream delivery should reuse the same live config and the real-behavior proof should exercise the public SDK API rather than only the internal helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass the live config into attachment sends:** `src/plugins/host-hook-workflow.ts:278-292`
  - body: `sendPluginSessionAttachment` uses `params.config` to find the session route and resolve attachment paths, but this `sendMessage` call omits `cfg`. Since `sendMessage` falls back to runtime config when `cfg` is absent, a hot-reloaded or session-specific config can select one store/root for validation and a different outbound target, account, or media policy for delivery. Pass `cfg: params.config` into this call and cover that handoff.
  - confidence: 0.86
