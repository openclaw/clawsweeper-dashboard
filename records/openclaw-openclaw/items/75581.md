## Summary

Keep open: this is an active additive Plugin SDK PR that is not on main, but the current head is not merge-ready because attachment dispatch still drops the live config before outbound send and the supplied proof does not exercise the public `api.sendSessionAttachment` path.

## What This Changes

The PR adds a bundled-only `OpenClawPluginApi.sendSessionAttachment` seam, host-side attachment validation and routing, outbound delivery hints, Telegram formatting exports, Gateway send options, docs, changelog, and regression coverage.

## Best Possible Solution

Keep the bundled-only seam, preserve the same live config through route lookup, file validation, and outbound delivery, add public SDK-path real behavior proof, then let maintainers review the additive SDK contract.

## Reproduction Assessment

Not applicable for the feature request itself; there is no current-main bug to reproduce. The blocking PR regression is source-reproducible on the PR head by following `sendPluginSessionAttachment` from config-based route/path resolution into a `sendMessage` call that omits `cfg`.

## Solution Assessment

No, not merge-ready as written: the seam shape is plausible, but downstream delivery must reuse the live config and proof should exercise `api.sendSessionAttachment` through the public Plugin SDK/loader path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass the live config into attachment sends:** `src/plugins/host-hook-workflow.ts:278-292`
  - body: `sendPluginSessionAttachment` uses `params.config` to find the session route and resolve attachment paths, but then calls `sendMessage` without `cfg`. When the plugin runtime supplies a live config that differs from the process snapshot, outbound target/account resolution and delivery policy can use stale settings after route lookup, so the attachment may fail or send with the wrong channel settings. Pass `cfg: params.config` into this call and cover that handoff.
  - confidence: 0.86
