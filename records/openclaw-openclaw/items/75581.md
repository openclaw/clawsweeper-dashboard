## Summary

Keep open: current main still lacks the host-mediated attachment SDK seam, and this PR is the active implementation path, but the patch has a concrete Telegram correctness bug where document-mode attachment hints are dropped on the full plugin media-send path.

## What This Changes

The PR adds bundled-plugin `sendSessionAttachment` support with host-side file validation/routing, Gateway send option plumbing, Telegram delivery hints/exports, supporting Matrix/Google Chat fixes, generated baselines, tests, and a changelog entry.

## Best Possible Solution

Fix the Telegram media option propagation, then have maintainers review and land the split SDK seam with the related fresh-main PR stack.

## Reproduction Assessment

Yes for the PR finding: source inspection shows `sendSessionAttachment` passes `forceDocument`, while Telegram's full-plugin `sendMedia` branch drops it before calling `sendTelegramOutbound`. The original item is a feature PR, so there is no shipped bug reproduction path on current main.

## Solution Assessment

No, not as-is. The host-mediated design is narrow and maintainable, but the Telegram document-mode path must be wired through the real full-plugin media sender before this can be considered correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward forceDocument through Telegram media sends:** `extensions/telegram/src/channel.ts:1133-1148`
  - body: `sendSessionAttachment` forces `forceDocument: true` when `forceDocumentMime` is present, but ordinary attachment batches go through `sendMessage` -> `deliverOutboundPayloads` -> Telegram `attachedResults.sendMedia`, not `sendPayload`. This branch now forwards `parseMode` but still drops `forceDocument`, so MIME-validated Telegram attachments can still be sent as normal media instead of documents. Please thread `forceDocument` through `sendTelegramOutbound` and cover the full plugin path, not only the mocked `sendMessage` call.
  - confidence: 0.88
