## Summary

Keep open. Current main still lacks the host-mediated session attachment API, and this PR is the active implementation slice, but the latest head still has a concrete routing bug in normalized session lookup: a newer non-routable alias can hide an older routable alias with the same normalized key, causing attachment delivery to report no active route even when one exists.

## What This Changes

The PR adds bundled-plugin `sendSessionAttachment` support with host-side file validation and session-route delivery, channel delivery hints, Telegram formatting exports, Gateway send option plumbing, tests, API baseline hash updates, and a changelog entry.

## Best Possible Solution

Keep the additive bundled-plugin attachment seam, but update normalized delivery-route lookup so routable aliases always outrank non-routable freshness, add the missing inverse mixed-case regression, and then rerun the targeted SDK/session/Telegram/lint/API checks before normal maintainer review.

## Reproduction Assessment

Yes. On the PR head, create a session store where the queried/canonical direct key is absent or non-routable, a newer lowercase normalized alias is non-routable, and an older mixed-case alias with the same normalized key has `deliveryContext`; the normalized fallback can return the non-routable record and attachment delivery reports no active route.

## Solution Assessment

No. The proposed host-mediated attachment seam is the right narrow direction, but the current normalized index implementation is not the best fix until it prefers routable entries before comparing `updatedAt`.
