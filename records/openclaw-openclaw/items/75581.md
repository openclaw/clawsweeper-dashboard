## Summary

Keep open. Current main still lacks the requested host-mediated session attachment API, and this PR is the active implementation slice, but the latest head still has a concrete gateway-mode routing blocker that should be fixed before merge.

## What This Changes

The PR adds a bundled-plugin `sendSessionAttachment` Plugin SDK API with host-side file validation/routing, attachment delivery hints, Gateway send options, Telegram formatting exports, tests, generated API/protocol updates, Matrix/Google Chat follow-up fixes, and a changelog entry.

## Best Possible Solution

Keep the additive bundled-only attachment seam, but make the direct-delivery check match the outbound resolver's actual loaded/active/bundled channel resolution before validating and sending host-local files, with a regression for an unloaded bundled gateway-mode channel.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a session route whose delivery channel is a gateway-mode bundled channel such as WhatsApp while that channel is not loaded in the active registry; the PR guard skips rejection and `sendMessage` later resolves the bundled gateway plugin.

## Solution Assessment

No, not as submitted. The API direction is maintainable, but the direct-delivery guard is narrower than the actual outbound resolution path; the safer fix is a focused resolver/metadata check plus a regression test rather than changing the public API shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject gateway-mode channels before bundled fallback:** `src/plugins/host-hook-workflow.ts:211-217`
  - body: The guard checks only `getLoadedChannelPluginById`, so an unloaded bundled gateway channel such as WhatsApp/Synology skips the rejection and then `sendMessage` resolves the bundled plugin and sends host-local media paths through gateway RPC. That path cannot read the files this API just validated, so session attachments fail for gateway-mode routes unless the channel was already loaded; use the same outbound/plugin delivery-mode resolver or a light metadata equivalent before calling `sendMessage`.
  - confidence: 0.94

## Repair Work Prompt

Repair the latest head of https://github.com/openclaw/openclaw/pull/75581. The observable bug is that `sendPluginSessionAttachment` rejects gateway-mode channels only when the channel plugin is already loaded; unloaded bundled gateway-mode channels can still be resolved by `sendMessage` and receive host-local media paths over gateway RPC. Fix the direct-delivery guard so it uses the same effective loaded/active/bundled channel delivery-mode resolution as outbound send, or a lightweight equivalent that does not reintroduce broad startup work unnecessarily. Add a regression where the active registry does not contain the channel but the bundled channel is gateway-mode, and assert `sendMessage` is not called. Do not broaden `sendSessionAttachment` beyond bundled plugins, do not stage/upload files as part of this repair, and do not change unrelated workflow seam APIs. Keep the existing changelog entry; no new user-facing changelog line is needed for this PR-internal fix unless the final diff changes the public behavior description.
