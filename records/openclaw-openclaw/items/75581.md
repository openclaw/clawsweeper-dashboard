## Summary

Keep open. Current main still lacks the requested `sendSessionAttachment` SDK surface, and this PR is the active implementation slice; the latest head addresses several earlier routing and Telegram-caption concerns but still has a concrete result-contract bug in forced-MIME validation that should be fixed before merge.

## What This Changes

The PR adds a bundled-plugin `sendSessionAttachment` API with host-side attachment validation, session delivery-route lookup, Telegram/Slack delivery hints, Telegram formatting exports, tests, API baseline updates, and a changelog entry.

## Best Possible Solution

Keep the additive bundled-plugin attachment seam, but wrap validation and MIME probing failures into the same result-based error contract, add a regression for a post-`lstat` read failure, then rerun the focused plugin SDK, session delivery, Telegram, lint, API, and diff checks before normal maintainer review.

## Reproduction Assessment

Yes. On the PR head, call `sendPluginSessionAttachment` with a bundled origin, a routable session, `channelHints.telegram.forceDocumentMime`, and a file that passes `lstat` but fails `readFile` before MIME detection; the promise rejects instead of resolving `{ ok: false, error }`.

## Solution Assessment

No. The overall host-mediated attachment seam is the right narrow direction, but the current implementation is not the best fix until all local-file validation errors preserve the SDK’s result-based failure contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle MIME read failures as result errors:** `src/plugins/host-hook-workflow.ts:113-114`
  - body: `validateAttachmentFiles` performs the forced-MIME `readFile` before `sendPluginSessionAttachment` enters its `try`/`catch`. Wrap validation/MIME probing errors or move this validation under the existing catch, and add a regression for the read-failure case.
