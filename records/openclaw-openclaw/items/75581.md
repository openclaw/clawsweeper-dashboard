## Summary

Keep this PR open: the host-mediated attachment seam is not implemented on current main and the PR has been heavily hardened, but the current head still regresses Telegram default Markdown chunk planning by advertising the base adapter as text-mode.

## What This Changes

The PR adds bundled-only Plugin SDK `sendSessionAttachment` support with host-side file validation, active-session delivery routing, Telegram/Slack delivery hints, gateway send-option plumbing, docs, generated Swift models, and regression tests.

## Best Possible Solution

Land the attachment seam after preserving Telegram's existing Markdown-aware default chunk planning while selecting HTML-safe chunking only for explicit HTML formatting.

## Reproduction Assessment

Yes for the review finding: source inspection of PR head plus current main's outbound planner shows that `chunkerMode: "text"` routes newline chunking through plain `chunkByParagraph` before the Telegram Markdown chunker sees default Markdown text.

## Solution Assessment

No: the host-mediated attachment seam is directionally sound, but changing Telegram's adapter-wide chunker mode is not the narrowest safe way to support HTML captions because it weakens the existing Markdown path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve Markdown-aware newline chunking:** `extensions/telegram/src/outbound-base.ts:13`
  - body: This advertises Telegram as `chunkerMode: "text"` for every send. In newline mode, `planOutboundTextMessageUnits` only uses `chunkMarkdownTextWithMode` when the adapter mode is `markdown`; text mode pre-splits with `chunkByParagraph`, whose long-paragraph fallback uses plain `chunkText`. Default Telegram Markdown messages can therefore be split before the Markdown-aware chunker sees the whole block. Please keep Markdown-aware planning for normal sends and make the HTML behavior context-specific.
  - confidence: 0.86
