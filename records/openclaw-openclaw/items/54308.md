## Summary

Keep open. The Greptile summary correctly identifies useful source work around Discord teardown draining, but current main still does not implement that idle-drain behavior, and this PR is not merge-ready: the diff weakens current fail-closed Discord bot identity startup behavior and is stale against the current createChannelRunQueue/message-run-queue architecture. Prior ClawSweeper review comments flagged the same P2 issue, and the current main/source checks confirm it still applies to the PR diff.

## What This Changes

This PR adds Discord managed bot identity tracking and idle-drain teardown wiring, propagates SenderManagedAccountId into Discord message context, updates queue/provider tests, and includes smaller Telegram message schema and plugin type changes.

## Best Possible Solution

Rebase and narrow the work to a Discord-owned teardown fix that preserves fetchDiscordBotIdentity fail-closed startup semantics, adds a minimal idle-drain primitive to the current createChannelRunQueue/message-run-queue cleanup path, and splits Telegram or plugin contract edits into separate PRs if still needed.

Yes. For the blocking PR defect, make client.fetchUser('@me') reject in the provider test path: current main rejects before handler creation, while the PR diff catches the error and can continue with botUserId undefined.

## Solution Assessment

No. The idle-drain direction is plausible, but this patch is not the best current solution because it weakens fatal bot identity startup semantics, is stale against the current queue/cleanup shape, and bundles unrelated surfaces.
