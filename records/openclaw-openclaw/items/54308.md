## Summary

Keep open: current main still lacks the PR's Discord idle-drain/managed-identity teardown path, but the proposed patch is not merge-ready because it can fail open when Discord bot identity lookup fails and it spans stale mixed-scope changes.

## What This Changes

This PR adds Discord managed bot identity tracking, idle-drain teardown plumbing, SenderManagedAccountId context, queue/provider tests, and smaller Telegram/plugin contract type changes.

## Best Possible Solution

Rebase and narrow the patch to a Discord-owned teardown fix that preserves fatal bot identity startup failures and adds an idle-drain seam only where the current lifecycle architecture needs it.

## Reproduction Assessment

Yes for the PR blocker by source inspection: make client.fetchUser('@me') reject and current main rejects before handler creation, while the PR catch path can continue with botUserId undefined. The original teardown race remains source-plausible from the missing waitForIdle/managed-identity cleanup seam, but I did not live-reproduce it.

## Solution Assessment

No. The idle-drain direction is plausible, but this patch is not the best current solution because it swallows bot identity failures and combines stale Discord lifecycle changes with unrelated Telegram/plugin contract edits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve bot identity fetch failures:** `extensions/discord/src/monitor/provider.ts:1040-1051`
  - body: The patch catches fetchUser('@me') failures and continues startup with botUserId left undefined. Current Discord startup treats a missing bot identity as fatal, and the handler uses botUserId to suppress the bot's own messages before debounce/preflight. Starting without it can let the monitor process its own messages instead of surfacing the startup failure, so this path should rethrow or keep using fetchDiscordBotIdentity.
  - confidence: 0.88
