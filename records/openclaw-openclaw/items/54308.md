## Summary

Keep open. Current main still does not provide the PR's intended Discord idle-drain teardown path, so the source work is not obsolete, but the branch is not mergeable as-is because it can start Discord without a resolved bot identity and also mixes unrelated Telegram/plugin contract changes into the teardown fix.

## What This Changes

This PR adds Discord managed bot identity tracking, threads an idle-drain lifecycle through inbound queue/provider teardown, adds SenderManagedAccountId context, updates queue/provider tests, and includes small Telegram message schema and plugin type changes.

## Best Possible Solution

Rebase and narrow the work to a Discord-owned teardown fix that fits the current createChannelRunQueue/provider.cleanup architecture, preserves fatal bot identity startup failures, adds the smallest idle-drain primitive needed for in-flight Discord messages, and splits Telegram/schema or plugin contract changes into separate PRs if still wanted.

## Reproduction Assessment

Yes. The blocking PR defect can be reproduced by making client.fetchUser('@me') reject: current main rejects before handler creation, while the PR's catch path can continue with botUserId undefined. The original teardown race has a plausible code-level path but no live reproduction in the PR discussion.

## Solution Assessment

No. The idle-drain direction is plausible, but this patch is not the best current solution because it weakens fail-closed bot identity semantics, is stale against the current queue/cleanup architecture, and bundles unrelated Telegram/plugin contract edits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve bot identity fetch failures:** `extensions/discord/src/monitor/provider.ts:1040-1051`
  - body: The patch catches fetchUser('@me') failures and continues startup with botUserId left undefined. Current Discord startup treats a missing bot identity as fatal, and the handler uses botUserId to suppress the bot's own messages before debounce/preflight. Starting without it can let the monitor process its own messages instead of surfacing the startup failure, so this path should rethrow or keep using fetchDiscordBotIdentity.
  - confidence: 0.88
