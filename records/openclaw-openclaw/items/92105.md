## Summary

The requested configurability is not implemented on current main and is already represented by an open same-author implementation PR, so this issue should remain open while maintainers decide the configuration contract and review that PR. Recursive scanning is a separable existing-behavior defect tracked by https://github.com/openclaw/openclaw/issues/90869 and https://github.com/openclaw/openclaw/pull/90870, while custom page groups remain a broader product and compatibility decision.

## What This Changes

Add configurable memory-wiki page-kind directories and headings, optional vault-local overrides, and recursive Markdown discovery for nested vault layouts.

## Best Possible Solution

Keep recursive scanning as the narrow bug fix, then define one coherent plugin-owned page-layout contract that uses canonical configuration, distinguishes searchable directories from the single write destination for each kind, preserves existing defaults, and updates every read/write/report path plus docs and upgrade tests together.

## Reproduction Assessment

Yes for the stated capability gap: current source and docs conclusively show five fixed directories and no page-group configuration. This is not a reproduction of broken documented behavior; it is source proof that the requested feature is absent.

## Solution Assessment

No, the exact proposed design is not yet the best solution. The customization goal is valid, but dual configuration ownership and ambiguous multi-directory write routing should be resolved before accepting the implementation shape.

## Risks / Open Questions

- The proposed `.wiki-page-groups.json` creates a second steady-state configuration source whose precedence, validation, reload lifecycle, and diagnostics could drift from canonical OpenClaw plugin config.
- Allowing multiple directories for one kind makes reads straightforward but leaves write routing ambiguous unless one directory is explicitly designated as the canonical creation target.
- The fixed directory names currently carry semantics across query, apply, compile, lint, status, reports, initialization, docs, and tests; changing only scanners or index headings would produce inconsistent behavior.
