## Summary

Keep open for maintainer product review: current main still uses a fixed memory-wiki layout, the unique configurable-layout capability is not implemented elsewhere, and the prior feature PR was closed because its branch no longer contained the feature—not because the idea was rejected. Recursive subfolder discovery is already separated into https://github.com/openclaw/openclaw/issues/90869 and https://github.com/openclaw/openclaw/pull/90870, so this issue should focus only on whether OpenClaw should support custom page-group layouts and through which single configuration authority.

## What This Changes

Add configurable memory-wiki page groups and custom index directories, including optional vault-local overrides, while supporting nested Markdown files.

## Best Possible Solution

Keep recursive discovery in its focused bug fix, then—if maintainers approve custom layouts—define one canonical plugin-owned page-layout model and migrate every memory-wiki read, write, initialization, report, link, and validation path together, with an unambiguous primary write directory per kind.

## Reproduction Assessment

Not applicable: the configurable-layout portion requests new behavior rather than reporting a failure of the current fixed-layout contract. The recursive subfolder bug is separately reproducible and tracked at https://github.com/openclaw/openclaw/issues/90869.

## Solution Assessment

No, not in the combined form proposed. Recursive discovery belongs in the existing focused fix, while configurable layouts need a narrower owner-approved design with one canonical configuration authority and complete read/write routing semantics.

## Risks / Open Questions

- The proposed `.wiki-page-groups.json` would make vault content a second configuration authority alongside canonical OpenClaw plugin config, requiring explicit precedence, diagnostics, and ownership semantics.
- Custom directories change a vault-wide invariant shared by initialization, creation routing, compile indexes, reports, lint, status, search, imports, and links; a partial implementation could silently read and write different layouts.
- Supporting multiple directories per kind needs a clear rule for where newly created pages and generated reports are written.
