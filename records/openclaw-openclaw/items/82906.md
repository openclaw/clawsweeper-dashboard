## Summary

Keep open: the approval gate is aimed at the right Codex/Gateway boundary, and the prior allow-always server-side mismatch is mostly addressed, but the patch still leaves a Web UI decision mismatch and the current PR proof is mock-only.

## What This Changes

The PR gates Codex CLI session resume behind plugin approval, forwards restricted plugin approval decisions through node-invoke policy requests, and adds focused resume approval tests.

## Best Possible Solution

Land this after the Control UI respects plugin approval allowed decisions and the contributor adds redacted real behavior proof for the Gateway/headless-node approval path.

## Reproduction Assessment

Yes. Current main source shows the dangerous Codex resume policy dispatching directly to the node, and the node-host resume handler can run `codex exec resume`; I did not run a live repro in this read-only review.

## Solution Assessment

No, not yet. Gating at the plugin node-invoke policy with `allow-once`/`deny` is the right narrow fix, but the Web UI must also honor those allowed decisions before the visible approval path matches the policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Hide disallowed approval decisions in the UI:** `src/gateway/node-invoke-plugin-policy.ts:68-70`
  - body: Forwarding `allowedDecisions` here gives the Codex resume prompt a restricted decision set, but the Control UI parser and modal never read that field and always render `Always allow` for plugin approvals. Please carry the field through `parsePluginApprovalRequested` and render only permitted buttons, with a UI regression test.
  - confidence: 0.82
