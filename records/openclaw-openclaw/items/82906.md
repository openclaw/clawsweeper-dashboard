## Summary

Keep this PR open: source inspection supports the approval-gate bug it is trying to fix, but the patch still needs a real behavior proof and one approval-semantics fix before merge. Likely related people for routing are Vincent Koc for the current Codex node-session implementation and mbelinky for the merged Codex CLI node-session feature history.

## What This Changes

The PR adds plugin approval gating before Codex CLI session resume dispatch and adds policy-level tests for deny, missing-approval-runtime, and allow-once paths.

## Best Possible Solution

Keep the gateway-side approval gate for Codex CLI resume, but make the approval decisions match the semantics the product can actually honor and require a live gateway-to-node approval proof before merge.

## Reproduction Assessment

Yes. Current main's Codex CLI resume policy calls ctx.invokeNode() directly while the node-host handler runs a new codex exec resume turn, so the missing approval gate is source-reproducible without running the full live flow.

## Solution Assessment

No. Gating before ctx.invokeNode() is the right boundary, but the patch should not offer or accept allow-always unless it implements durable semantics, and it still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Restrict the resume approval decisions:** `extensions/codex/src/node-cli-sessions.ts:110`
  - body: This request does not set allowedDecisions, so the plugin approval UI falls back to allow-once|allow-always|deny. The handler then accepts allow-always, but the Codex resume policy never persists or reuses that decision, so choosing allow-always behaves like a one-shot approval. Please either restrict this approval to allow-once/deny or implement durable allow-always semantics before accepting it.
  - confidence: 0.84
