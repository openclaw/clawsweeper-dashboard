## Summary

Keep open: the approval bypass is real and the updated PR body now has enough live Gateway/node proof, but the branch still accepts an unsupported `allow-always` decision that this resume path does not persist or reuse.

## What This Changes

The PR wraps Codex CLI session resume node dispatch in a plugin approval request and adds policy tests for denied, unavailable, and allow-once approval paths.

## Best Possible Solution

Land the approval gate after the Codex resume approval either only offers `allow-once`/`deny` through the node-invoke approval runtime or implements durable `allow-always` behavior with tests.

## Reproduction Assessment

Yes. Current main source shows the dangerous Codex resume node-invoke policy dispatching directly to the node, and the node handler can run a new `codex exec resume` turn.

## Solution Assessment

No. Gating before `ctx.invokeNode()` is the right boundary, but the current patch should not accept `allow-always` unless it can honor durable semantics; forwarding `allowedDecisions` and requesting only `allow-once`/`deny` is the narrower fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject unsupported allow-always approvals:** `extensions/codex/src/node-cli-sessions.ts:110`
  - body: Node-invoke policy approvals currently cannot request a smaller decision set: the runtime builds `PluginApprovalRequestPayload` without `allowedDecisions`, while plugin approvals default to `allow-once|allow-always|deny`. This handler then accepts `allow-always`, but it never persists or reuses that decision for future Codex resumes, so the UI can offer a durable approval that only behaves as a one-shot. Please either pass `allowedDecisions` through and request only `allow-once`/`deny` here, or implement durable `allow-always` semantics before accepting it.
  - confidence: 0.87
