## Summary

Keep open: the approval gate belongs at this boundary, but the external PR only provides unit-test proof and currently accepts allow-always without implementing durable semantics.

## What This Changes

The PR changes the Codex CLI session resume node-invoke policy to request plugin approval before dispatch and adds policy tests for deny, missing approval runtime, and allow-once.

## Best Possible Solution

Keep the gateway-side approval gate, restrict or correctly honor approval decisions, and land only after real gateway-to-node approval proof.

## Reproduction Assessment

Yes. Current main's resume policy calls ctx.invokeNode() directly while the node-host handler runs codex exec resume, so the missing approval gate is source-reproducible without a live run.

## Solution Assessment

No. Gating before ctx.invokeNode() is the right boundary, but the patch should not offer or accept allow-always unless it implements durable semantics and includes live approval-path proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Restrict the resume approval decisions:** `extensions/codex/src/node-cli-sessions.ts:110`
  - body: Because the node-invoke approval runtime does not pass allowedDecisions and plugin approvals default to allow-once|allow-always|deny, this handler lets an operator choose allow-always but then treats it like a one-shot approval. Either restrict the request to allow-once/deny or persist durable allow-always semantics before accepting it.
