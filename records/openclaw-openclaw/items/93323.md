Review or repair the existing pending-handoff fix path so non-terminal direct announces do not exhaust retries and drop completions, with consumption-backed delivery or requester wake coverage plus real gateway proof.

Yes at source level, but not as a live current-main run in this review. Current main still maps a non-terminal direct completion handoff to `completion_handoff_pending`, and the registry retry cap can then give up after three failed delivery attempts.

## Solution Assessment

Unclear as a final implementation until maintainers choose the landing shape. The fix belongs in subagent announce delivery/retry handling, but the current open PRs need durability, requester wake, SDK compatibility, and real-proof issues resolved before they are the best fix.
