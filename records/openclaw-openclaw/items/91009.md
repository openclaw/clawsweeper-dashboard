## Summary

Keep open: current `main` and the latest `v2026.6.8` release still install synchronous Codex native hook command relays on the PreToolUse path, field comments report continued relay CPU/RSS pileups after the lifetime fix, and the only closing PR is still open.

## What This Changes

The issue reports Codex PreToolUse native hook relay fan-out spawning heavy `openclaw-hooks` subprocesses that can stall gateway RPC and exhaust host resources.

## Best Possible Solution

Keep this as the canonical P1 tracker while maintainers review and scope https://github.com/openclaw/openclaw/pull/94515 or choose a broader owner-approved fix such as a lightweight relay entrypoint, pre-exec concurrency gate, or in-process/resident loop-detection path that preserves real hook enforcement.

## Reproduction Assessment

Yes for source-level reproduction: current main still builds synchronous Codex command hooks that execute `openclaw hooks relay` before tools when PreToolUse is selected, and multiple field reports reproduce CPU/RSS waves. I did not run a live multi-agent load test in this read-only review.

## Solution Assessment

Unclear. Omitting idle relays is a plausible narrow mitigation, but maintainers still need to decide whether explicit loop detection should stay on the per-tool subprocess path or move to a lightweight, resident, or concurrency-bounded design.
