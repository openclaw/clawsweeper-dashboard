## Summary

Keep open: current main still installs synchronous Codex native PreToolUse command hooks and builds `openclaw hooks relay` CLI commands for the loop-detection path, while the merged lifetime fix only bounds stuck relay processes and does not remove the per-tool-call process fan-out reported here. The latest comments add live host/OOM evidence, so this remains a core Codex relay availability bug rather than stale or superseded cleanup.

## What This Changes

The issue requests fixing Codex native hook relay fan-out that spawns heavy `openclaw-hooks` CLI processes during tool bursts and can stall or OOM the gateway.

## Best Possible Solution

Keep this as a core Codex native-hook relay stability bug and make loop-detection-only native tool events cheap, resident, or otherwise bounded without losing native hook safety semantics; add load regression proof for rapid Codex tool bursts.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction path: current main enables Codex native relay, keeps PreToolUse active when loop detection can run, and Codex executes configured command hooks as child processes. I did not run the full live multi-agent OOM scenario in this read-only review.

## Solution Assessment

No: disabling native relay is only a workaround because it drops useful hook and safety behavior. The best fix should reduce or eliminate per-tool full-CLI relay fan-out while preserving native PreToolUse policy and diagnostics.
