## Summary

Keep open: current main and the latest release still install synchronous Codex PreToolUse command-hook relays when loop-detection work may run, and Codex executes those hooks as subprocesses before tool execution. The merged lifetime fix for leaked relays helps the stuck-process class, but it does not remove the CPU-bound per-tool-call fan-out reported here.

## What This Changes

The issue reports CPU-bound Codex native PreToolUse hook relay subprocess fan-out that stalls gateway RPC and can drive hosts into OOM or restart-loop conditions during Codex tool-call bursts.

## Best Possible Solution

Keep this issue open as the specific P1 tracking item for Codex PreToolUse relay fan-out; the durable fix should avoid heavyweight per-tool CLI subprocesses for loop-detection-only work while preserving real before-tool policy enforcement and adding live burst-load regression proof.

## Reproduction Assessment

Yes, from source inspection: current main still builds synchronous Codex PreToolUse command hooks that execute `openclaw hooks relay` subprocesses before tool calls when loop detection may run. I did not run the live multi-agent OOM scenario, so the host-level severity remains live-proof pending.

## Solution Assessment

Unclear. A narrow code fix is plausible, but the best solution needs maintainer choice between moving loop detection in-process/resident, changing when PreToolUse registers, or preserving relay behavior for safety-critical policy paths.
