## Summary

Keep open: current main and the latest release still have the synchronous Codex native PreToolUse relay path that can spawn `openclaw hooks relay` subprocesses, multiple field reports confirm CPU/RSS fan-out after the lifetime fix, and the linked closing PR is still open.

## What This Changes

Fix Codex native PreToolUse relay fan-out so no-op or loop-detection-only tool-call bursts do not spawn heavyweight `openclaw-hooks` processes that stall gateway RPC.

## Best Possible Solution

Keep this as the canonical P1 tracker until maintainers either land https://github.com/openclaw/openclaw/pull/94515 with an explicit scope decision or choose a broader relay design that preserves hook enforcement without heavyweight per-tool fan-out.

## Reproduction Assessment

Yes for source-level reproduction: current main and `v2026.6.8` still build synchronous Codex PreToolUse command hooks that execute `openclaw hooks relay`, and upstream Codex runs matched command hooks as child processes. I did not run a live multi-agent load test in this read-only review.

## Solution Assessment

Unclear as a complete fix: the linked PR is a plausible narrow mitigation for idle/default PreToolUse relays, but maintainers still need to decide whether explicit loop detection, concurrency caps, or a lightweight relay path are required before closing the canonical P1.
