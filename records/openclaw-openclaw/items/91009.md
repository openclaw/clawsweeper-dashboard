## Label Justifications

- `P1`: The report has multiple field confirmations of Gateway RPC stalls and host OOM/restart behavior during normal Codex tool bursts.
- `impact:crash-loop`: The issue is about process-level availability: CPU-bound relay child fan-out can make Gateway RPC time out and, on small hosts, repeatedly OOM-kill the gateway.

## Summary

Keep open: current main still enables synchronous Codex native PreToolUse command relay when loop detection is not explicitly disabled, and Codex runs those hooks as child processes; the merged lifetime fix only bounds leaked children and does not remove the CPU/RSS fan-out reported here.

## What This Changes

The issue asks OpenClaw to stop Codex native PreToolUse hook relay from spawning heavyweight `openclaw-hooks` subprocesses per tool call and degrading Gateway availability.

## Best Possible Solution

Keep this as a core Codex native-hook relay stability bug and make the loop-detection/native PreToolUse path cheap, resident, or concurrency-bounded without dropping fail-closed safety semantics.

## Reproduction Assessment

Yes for a high-confidence source-level path and released-build field proof: current main still registers synchronous Codex PreToolUse command relay when loop detection is not disabled, and Codex executes command hooks as child processes. I did not run the full current-main multi-agent load reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice: disabling the relay is an effective workaround but drops useful safety behavior. The maintainable fix should reduce or bound per-tool CLI fan-out while preserving native hook policy and approval semantics.
