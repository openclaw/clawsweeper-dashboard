## Summary

Keep open: current main and the latest release still make Codex PreToolUse relay active when loop detection is enabled by default, and Codex runs matching hooks as synchronous command subprocesses. The merged lifetime fix for https://github.com/openclaw/openclaw/pull/91550 addresses stuck relays, not this short-lived CPU-bound fan-out/OOM mode, and the related open PRs are not merged canonical fixes for this report.

## What This Changes

The issue reports CPU and memory saturation from Codex PreToolUse native hook relay subprocess fan-out during tool-heavy Codex app-server turns.

## Best Possible Solution

Keep the issue open and design a bounded Codex native PreToolUse path that avoids full CLI subprocess fan-out for loop-detection-only work while preserving policy-active before_tool_call enforcement.

## Reproduction Assessment

Yes for a source-level reproduction path: current main enables Codex native relay, loop detection keeps PreToolUse active by default, and upstream Codex executes selected command hooks as subprocesses. A full live gateway-starvation reproduction still needs the reporter-style multi-agent/tool-burst load run.

## Solution Assessment

Unclear for the exact implementation because the safest fix changes native PreToolUse loop-detection and policy behavior. Disabling native relay globally is not the best fix; the maintainable direction is a resident/in-process or bounded relay path that keeps security policy enforcement intact.
