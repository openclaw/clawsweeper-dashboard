## Label Justifications

- `P1`: The report is against the latest release and describes real Gateway health/status timeouts with event-loop degradation during normal Codex tool bursts.
- `impact:crash-loop`: The issue is about Gateway process availability: local RPCs time out and liveness reports event-loop and CPU degradation even though the service stays alive.

## Summary

Keep open: current main and the latest release still install synchronous Codex PreToolUse command hooks when loop detection can run, and Codex executes those hooks as subprocesses; the related native-relay reports do not fully cover this CPU-bound relay fan-out plus Gateway timeout mode.

## What This Changes

The issue asks OpenClaw to stop Codex PreToolUse native hook relay from spawning CPU-heavy `openclaw hooks relay` processes per tool call and stalling Gateway health/status RPCs during tool bursts.

## Best Possible Solution

Keep this as a core Codex native-hook relay stability bug and make the PreToolUse loop-detection path cheap, bounded, or resident while preserving native hook safety semantics.

## Reproduction Assessment

Yes for a source-level reproduction path: current main enables Codex native relay, loop detection keeps PreToolUse active, and Codex command hooks spawn child processes per selected hook. A full live Gateway starvation reproduction still needs the reporter's 20-30 rapid tool-call scenario or equivalent load proof.

## Solution Assessment

Unclear as an implementation direction: disabling native relay is an effective workaround but is not the best fix because it drops useful hook behavior. The maintainable fix should reduce or bound per-tool CLI fan-out for loop-detection-only PreToolUse while preserving security and plugin hook semantics.
