## Summary

Keep open: current main still installs Codex native relay as synchronous command hooks, and the merged sibling fix only bounds stuck relay lifetime rather than removing the per-tool subprocess fan-out. The new issue comments add stronger live load/OOM evidence for the same remaining failure mode.

## What This Changes

The issue reports CPU and memory pressure from Codex native PreToolUse hook relay subprocess fan-out during Codex app-server tool-call bursts.

## Best Possible Solution

Keep this as a core Codex relay stability bug and move loop-detection or no-policy native hook handling away from full CLI subprocesses, with burst-load proof that Gateway RPCs stay responsive and native hook safety semantics are preserved.

## Reproduction Assessment

Yes for a source-level reproduction path: current main still installs synchronous Codex relay command hooks and keeps PreToolUse active when loop detection may run. The issue comments add live process/OOM evidence, but this review did not run the multi-agent load scenario locally.

## Solution Assessment

No current fix fully solves it: disabling native relay is only a workaround, and the merged lifetime fix addresses stuck children rather than per-event subprocess cost. The best fix needs a resident, slimmer, or selectively skipped relay path that keeps policy safety intact.
