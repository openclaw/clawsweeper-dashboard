## Summary

Keep open: current main and the latest stable release still contain the synchronous Codex native PreToolUse relay path, multiple field reports confirm CPU/RSS fan-out after the lifetime fix, and the linked closing PR remains open/unmerged.

## What This Changes

The issue requests fixing Codex native hook relay fan-out where `PreToolUse` tool bursts spawn many heavy `openclaw-hooks` processes and can stall gateway RPC or exhaust host memory.

## Best Possible Solution

Keep #91009 as the canonical P1 tracker until maintainers either land #94515 with an explicit scope decision or choose a broader relay design that preserves hook safety without heavyweight per-tool fan-out.

## Reproduction Assessment

Yes for source-level reproduction: current main and `v2026.6.9` still build synchronous Codex PreToolUse command hooks backed by `openclaw hooks relay`, and upstream Codex spawns those configured hooks as child processes; no live load test was run in this read-only review.

## Solution Assessment

Unclear as a complete fix: #94515 is a plausible narrow mitigation for idle/default PreToolUse relays, but maintainers still need to decide whether explicit loop detection, concurrency caps, or a lightweight relay path are required before closure.
