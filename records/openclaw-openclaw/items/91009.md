## Summary

Keep open: current main and v2026.6.8 still install synchronous Codex native hook command relays on the PreToolUse path, field comments report CPU/RSS fan-out after the lifetime fix, and the open closing PR has not merged.

## What This Changes

The issue asks OpenClaw to stop or bound Codex native PreToolUse hook relay subprocess fan-out so gateway health/status RPCs remain responsive under tool-call bursts.

## Best Possible Solution

Keep this as the canonical P1 tracker while maintainers review and either land https://github.com/openclaw/openclaw/pull/94515 or choose a broader owner-approved relay design that preserves real hook enforcement without per-tool heavyweight fan-out.

Yes for source-level reproduction: current main still builds synchronous Codex command hooks that execute `openclaw hooks relay` before tools when PreToolUse is selected, and multiple field reports reproduce CPU/RSS waves. I did not run a live multi-agent load test in this read-only review.

## Solution Assessment

Unclear. Omitting idle relays in the linked PR is a plausible narrow mitigation, but maintainers still need to decide whether explicit loop detection should stay on the per-tool subprocess path or move to a lightweight, resident, or concurrency-bounded design.
