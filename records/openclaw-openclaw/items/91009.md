## Summary

Keep open: current main and the latest release still have the reported fan-out shape for Codex PreToolUse when loop-detection work is active. The merged lifetime fix for related stuck relay processes bounds child lifetime, but it does not remove the per-tool synchronous command-hook spawn cost that the newer comments tie to Gateway stalls and OOM restarts.

## What This Changes

The issue reports that Codex native PreToolUse relay invokes heavy `openclaw-hooks` CLI subprocesses during tool bursts, starving Gateway RPC and sometimes causing OOM/restart loops.

## Best Possible Solution

Implement an owner-approved relay design that avoids per-tool full CLI bundle spawns for loop-detection-only PreToolUse while preserving `before_tool_call`, trusted policy, loop-detection, and diagnostics semantics.

## Reproduction Assessment

Yes for a source-level path: current main and the latest release still enable Codex native relay and can install synchronous PreToolUse command hooks when loop detection is active. The comments add live load/OOM proof, but I did not run a fresh current-main multi-agent Gateway reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation direction: disabling native relay is an effective workaround but not the best fix because it drops useful hook safety behavior. The maintainable fix should reduce or eliminate per-tool CLI fan-out for loop-detection-only PreToolUse while preserving policy enforcement.
