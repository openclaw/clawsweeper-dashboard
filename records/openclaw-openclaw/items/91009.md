## Summary

Keep open: current main and the latest v2026.6.8 release still have the synchronous Codex native-hook command path that can spawn `openclaw hooks relay` subprocesses for PreToolUse, and post-fix field comments report continued relay pileups after the lifetime fix. The linked closing PR is still open and appears to need maintainer scope review rather than cleanup closure.

## What This Changes

The issue reports Codex native PreToolUse relay subprocess fan-out causing gateway RPC stalls, high CPU/RSS, and host crash-loop behavior during tool-heavy Codex turns.

## Best Possible Solution

Keep this as the canonical P1 tracker while maintainers review and scope https://github.com/openclaw/openclaw/pull/94515 or choose a broader owner-approved fix such as a lightweight relay entrypoint, pre-exec concurrency gate, or in-process/resident loop-detection path that preserves real hook enforcement.

## Reproduction Assessment

Yes. The source path still builds synchronous Codex command hooks that execute `openclaw hooks relay` before tools when PreToolUse is selected, and multiple field reports reproduce CPU/RSS waves; I did not run a live multi-agent load test in this read-only review.

## Solution Assessment

Unclear. Disabling native relay entirely is a proven workaround but not the best fix because it drops useful hook behavior; the maintainable solution needs owner choice between omitting idle relays, making the relay lightweight, bounding subprocess concurrency before Codex execs it, or moving loop detection out of the per-tool subprocess path.
