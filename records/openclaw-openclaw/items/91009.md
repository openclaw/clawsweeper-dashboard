## Label Justifications

- `P1`: The issue has multiple user reports of gateway RPC stalls and host OOM/restart loops under normal Codex app-server tool-call bursts.
- `impact:crash-loop`: The reported impact is process-level availability: `openclaw-hooks` swarms consume enough CPU/RSS to stall the gateway and trigger repeated OOM kills.

## Summary

Keep open: current main still routes selected Codex native hook events through synchronous command hooks that spawn `openclaw hooks relay` processes, and the merged lifetime fix only addresses stuck relays, not the reported CPU/RSS fan-out under multi-agent tool bursts.

## What This Changes

The issue reports that Codex PreToolUse native hook relay spawns heavyweight per-tool-call `openclaw-hooks` CLI processes that can CPU-starve or OOM the gateway under tool-call bursts.

## Best Possible Solution

Keep this as a core Codex native-hook relay stability bug and make the PreToolUse/PostToolUse relay path cheap, resident, or in-process for loop-detection and no-op cases while preserving native tool policy semantics.

## Reproduction Assessment

Yes for a source-level reproduction path: current main still installs synchronous Codex command hooks that spawn `openclaw hooks relay` processes for selected native events. I did not run the multi-agent load scenario, so the exact OOM threshold is not personally reproduced here.

## Solution Assessment

No, disabling native hook relay is only a blunt workaround because it drops useful safety and plugin behavior. The maintainable fix needs a lower-cost relay path or narrower event registration backed by load proof and maintainer approval of the safety tradeoff.
