## Label Justifications

- `P1`: The issue has current source support plus multiple production reports of gateway RPC stalls and OOM/restart loops during normal Codex tool bursts.
- `impact:crash-loop`: The report and comments describe repeated process-level availability failures, including gateway timeouts, kernel OOM kills, and service restarts caused by relay process fan-out.

## Summary

Keep open: current main and v2026.6.6 still enable Codex native PreToolUse command hooks, and the loop-detection path still makes selected tool-call events spawn `openclaw hooks relay` subprocesses. PR https://github.com/openclaw/openclaw/pull/91550 fixed the stuck-process lifetime class, but it does not solve the short-lived CPU/RSS fan-out described here, and the broader https://github.com/openclaw/openclaw/issues/84037 does not replace this P1 crash-loop report.

## What This Changes

The issue asks OpenClaw to prevent Codex native PreToolUse relay process fan-out from starving gateway RPCs and crashing small hosts under tool-call bursts.

## Best Possible Solution

Keep this as a specific P1 core Codex relay stability bug and design a cheap, bounded, or resident path for loop-detection/native-hook work that preserves PreToolUse policy semantics and has load proof for rapid Codex tool bursts.

## Reproduction Assessment

Yes for a source-level reproduction path: current main installs Codex native command hooks, keeps PreToolUse active when loop detection can run, and Codex spawns command-hook child processes. I did not run the full live 20-30 tool-call gateway starvation scenario in this read-only review, but the issue comments add concrete process and OOM observations.

## Solution Assessment

Unclear as an implementation direction: disabling native relay is a useful workaround but not the best fix because it drops native hook safety behavior. The maintainable solution should reduce or bound the per-tool CLI process cost while keeping before-tool policy, loop detection, and Codex hook semantics intact.
