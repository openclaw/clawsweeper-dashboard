## Label Justifications

- `P1`: The current shipped path can make gateways unresponsive and multiple independent users report severe CPU pressure or host OOM during normal Codex tool activity.
- `impact:crash-loop`: The discussion includes repeated kernel OOM kills followed by gateway service restarts under concurrent relay-process fan-out.

## Summary

Keep open: current main and v2026.6.5 still install a synchronous Codex PreToolUse command hook whenever default loop detection may run, and upstream Codex starts a fresh subprocess for every matching hook invocation. The merged lifetime fix prevents indefinitely stuck relay children but does not address the CPU and memory fan-out reproduced by several users.

## What This Changes

Prevent per-tool Codex native hook relay process fan-out from exhausting CPU and memory or stalling gateway RPC while preserving pre-tool safety behavior.

## Best Possible Solution

Preserve PreToolUse policy semantics while replacing full OpenClaw CLI startup with a slim or resident relay path, enforce bounded per-session concurrency, and prove the result under overlapping multi-agent tool bursts using process-count, RSS, event-loop, and gateway RPC measurements.

## Reproduction Assessment

Yes for the underlying mechanism: current OpenClaw activates a synchronous PreToolUse command hook under default loop detection, and upstream Codex spawns a child for every invocation. This read-only review did not execute the full multi-agent load scenario on current main.

## Solution Assessment

No, globally disabling native relay is not the best solution because it also removes safety behavior. The narrow maintainable direction is to retain policy enforcement while eliminating heavyweight per-event CLI startup and bounding concurrency.
