- `impact:crash-loop`: The reported failure mode is process-level availability degradation: gateway health/status timeouts, event-loop starvation, and repeated OOM kills on constrained hosts.

## Summary

Keep open: current main and v2026.6.6 still route Codex native PreToolUse through synchronous command hooks that launch `openclaw hooks relay`, while the merged lifetime fix only bounds stuck relay children and does not address the CPU/RSS fan-out reported here.

## What This Changes

The issue asks OpenClaw to prevent Codex native PreToolUse relay bursts from spawning heavyweight `openclaw-hooks` processes that starve gateway RPC and can OOM small hosts.

## Best Possible Solution

Keep this open as a core Codex native-hook relay availability bug and design a cheaper, bounded, or resident PreToolUse path that avoids per-tool full CLI startup while preserving native tool policy enforcement.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction path: current main still enables Codex native relay, selects PreToolUse when loop detection may run, and Codex executes the selected hook as a command subprocess. I did not run the full live multi-agent burst locally, but the issue now has independent live process/OOM reports.

## Solution Assessment

Unclear as a concrete implementation: disabling native relay is not the best fix because it drops policy and plugin behavior. The best direction is to remove or bound the heavyweight per-event CLI path for loop-detection/no-op PreToolUse while keeping security-critical PreToolUse delivery intact.
