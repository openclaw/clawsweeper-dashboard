## Summary

Keep open. Current main still stores cron-friendly labels, but the CLI/TUI status summary contract and renderers do not carry or display `label`/`displayName`; the Control UI dropdown part is already covered, and open PR #51257 plus the fork PR remain the implementation candidates.

## What This Changes

Request to show stored session labels, display names, or cron job names in `/status`, the TUI statusline/footer, and the session picker instead of UUID-heavy cron run keys.

## Best Possible Solution

Carry `label` and `displayName` through `SessionStatus` and `GatewayStatusSummary`, populate them from `SessionEntry`, render a sanitized friendly fallback in CLI status, TUI `/status`, the TUI footer/statusline, and the session picker, then land the focused implementation with tests for cron labels, subagent labels, key fallback, and terminal sanitization.

## Reproduction Assessment

Yes. A cron-labeled session entry can be followed through current source: the label is written in the cron path, but the status summary omits it and both CLI/TUI status renderers print the raw key.

## Solution Assessment

Yes, the issue's proposed direction is the narrow maintainable fix: thread existing metadata through the status contracts and renderers rather than inventing new cron-only state. The linked PR direction is appropriate, but it should be rebased and expanded to cover the current CLI/TUI paths and tests.
