## Summary

Keep open: current main and the latest release still do not carry session `label`/`displayName` through the status summary, so `/status` and the TUI status overlay render raw session keys. The Control UI dropdown slice is already fixed, but the remaining CLI/TUI status surfaces still need a maintainer-approved display contract and a current implementation path.

## What This Changes

Request to show cron job names or stored session labels in `/status`, the TUI statusline, and session pickers instead of UUID-heavy session keys.

## Best Possible Solution

Define one canonical session display fallback for status surfaces, keep raw keys available, then land or replace the linked implementation with tests for CLI `/status`, TUI `/status`, the footer/statusline, and cron/subagent labels.

## Reproduction Assessment

Yes. Source inspection on current main shows the status summary omits `label`/`displayName`, while CLI and TUI status renderers still use the raw session key; I did not run a live TUI because this review is read-only.

## Solution Assessment

No, not yet. The linked old patch is a plausible starting point, but the best fix should update the current status section split, include the TUI footer/statusline behavior, preserve raw-key fallback, and add focused coverage before merge.
