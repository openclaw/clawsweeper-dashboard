## Summary

Keep open. The PR targets a real transcript-growth bug from #66443, but the proposed replay dedupe key is broader than the transcript contract allows: message entries are keyed only by role and content, so the skip path can delete legitimate repeated messages during any shared transcript rewrite.

## What This Changes

The PR adds SHA-256 identity tracking to transcript rewrite suffix replay and regression tests for duplicate user messages, compaction entries, bootstrap custom entries, and file-level replay.

## Best Possible Solution

Revise the replay idempotency rule so it suppresses proven replay pollution without collapsing distinct entries. A maintainable fix should use role-aware stable metadata or a narrower recovery-specific guard, with regression coverage for repeated user prompts, same-text tool results with different toolCallId/toolName, bashExecution entries with different command/output, and the #66443 overflow pattern.

## Reproduction Assessment

Yes. A focused in-memory SessionManager setup can append a rewrite target followed by two same-key message entries; with this PR's role+content key, the later entry is skipped during suffix replay. The original bug also has a concrete #66443 overflow-recovery capture with repeated role=user content hashes.

## Solution Assessment

No. The helper is a reasonable place to address suffix replay pollution, but role-plus-content identity for all message entries is not the narrowest maintainable fix; the safer path is role-aware metadata identity or a more targeted pollution guard.
