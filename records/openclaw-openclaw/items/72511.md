## Summary

Keep this PR open. It targets a real overflow-recovery transcript growth bug, but the proposed message dedupe identity is too broad for a shared branch-rewrite helper and can silently drop legitimate repeated messages during ordinary rewrites.

## What This Changes

The PR adds SHA-256 identity tracking to transcript rewrite suffix replay and regression tests for duplicate user, compaction, bootstrap custom, and file-level replay cases.

## Best Possible Solution

Revise the replay idempotency rule so it only suppresses proven replay pollution while preserving legitimate repeated transcript entries. A maintainable end state would use metadata-aware message identity or a narrower pollution-specific guard, with regression coverage for repeated user prompts, same-text tool results with different toolCallId/toolName, and the #66443 overflow pattern.

## Reproduction Assessment

Yes. A focused SessionManager.inMemory setup can append two same-role/same-content suffix messages after a rewrite target; with this PR's key, the later entry is skipped during suffix replay. The original bug also has a concrete #66443 overflow-recovery capture with repeated role=user content hashes.

## Solution Assessment

No. The helper is a reasonable place to address suffix replay pollution, but role-plus-content dedupe for all message entries is broader than the safest fix; the safer path is metadata-aware identity or a narrower suppression rule tied to replay pollution.
