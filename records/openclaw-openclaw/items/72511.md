## Summary

Keep open. The PR addresses a real transcript-growth bug, but the proposed patch is not merge-ready because the message identity key can delete legitimate repeated transcript entries, and the file-state rewrite path used by session-file rewrites is not covered by the new dedupe logic.

## What This Changes

The PR adds SHA-256 identity tracking to transcript rewrite suffix replay plus regression tests for duplicate user messages, compaction entries, bootstrap custom entries, and file-level replay.

## Best Possible Solution

Revise the replay idempotency rule so it suppresses proven replay pollution without collapsing distinct entries, then share that logic across both SessionManager and TranscriptFileState rewrites with regression coverage for repeated prompts, same-text tool results, and the #66443 overflow pattern.

## Reproduction Assessment

Yes. A focused in-memory SessionManager setup can append a rewrite target followed by two same-role/same-content suffix messages; with this PR's role+content key, the later entry is skipped during suffix replay. The file-path gap is also reproducible from current code because rewriteTranscriptEntriesInSessionFile calls rewriteTranscriptEntriesInState, which the PR does not change.

## Solution Assessment

No. The helper is a reasonable place to address suffix replay pollution, but role-plus-content identity for all message entries is broader than the safest fix, and the current patch does not apply the invariant to the file-state rewrite path.
