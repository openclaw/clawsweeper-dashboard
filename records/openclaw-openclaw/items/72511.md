## Summary

The PR should stay open because the replay hardening is aimed at a real overflow-rewrite bug, but the proposed message identity can delete legitimate repeated turns and the branch does not cover the current file-state rewrite path on main.

## What This Changes

The PR adds SHA-256-based duplicate suppression to `rewriteTranscriptEntriesInSessionManager` suffix replay and adds regression tests for duplicate user, compaction, bootstrap custom, clean-session, ordering, and file-level rewrite cases.

## Best Possible Solution

Implement one shared replay-idempotency helper for both SessionManager and TranscriptFileState rewrites that suppresses replay-proven duplicates without collapsing distinct turns, with persisted JSONL regression coverage for #66443.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR-branch reproduction: put two legitimate same-role, same-content suffix messages after the rewrite point, and the PR's role-plus-content key skips the later entry.

## Solution Assessment

No. The rewrite boundary is reasonable, but this implementation's identity key is too broad and the guard needs to be shared with the current file-state rewrite path.
