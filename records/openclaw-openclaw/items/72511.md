## Summary

Keep this PR open: it targets the right transcript-rewrite boundary for the valid #66443 report, but the latest patch still blocks merge because the proposed message identity is too broad and can delete distinct replayed transcript entries.

## What This Changes

The PR adds SHA-256 identity tracking to transcript rewrite suffix replay and regression tests for duplicate user, compaction, custom, and custom_message entries.

## Best Possible Solution

Revise the replay idempotency rule so it fixes the polluted-suffix cases from #66443 without changing ordinary transcript rewrite losslessness. The maintainable end state is a metadata-aware or narrowly scoped duplicate identity, with regression coverage for legitimate repeated user text, same-text tool results with different tool ids, assistant messages with distinct response ids, and contentless message roles.

## Reproduction Assessment

Yes. A focused SessionManager.inMemory branch with a replacement target followed by two distinct same-role/same-content messages would make this PR skip the later suffix entry; #66443 also provides a concrete overflow-recovery capture for the original duplicate-user symptom.

## Solution Assessment

No. The PR is aimed at the right helper, but role-plus-content dedupe for every message entry is broader than the safest fix; include stable message metadata or restrict content-only suppression to entries proven to be replay pollution.
