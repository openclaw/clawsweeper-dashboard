## Summary

Keep this PR open but not merge-ready: current main still needs the duplicate-replay fix, but this patch can drop legitimate repeated messages and does not cover the current file-state rewrite path. The linked #66443 and #69208 context supports the problem, not the exact identity rule in this branch.

## What This Changes

The PR adds SHA-256 identity deduplication inside transcript rewrite suffix replay plus regression tests for duplicate user messages, compaction entries, bootstrap custom entries, clean sessions, ordering, and file-level replay.

## Best Possible Solution

Implement a shared replay-idempotency helper for both SessionManager and TranscriptFileState rewrites that suppresses replay-proven duplicates without collapsing distinct turns, with real JSONL coverage for #66443 and repeated legitimate entries.

## Reproduction Assessment

Yes. Source inspection gives a focused reproduction: append a rewrite target followed by two same-role, same-content but logically distinct entries, and the PR's role-plus-content key causes the later entry to hit seenIdentities and be skipped.

## Solution Assessment

No. The rewrite boundary is a reasonable place to harden replay, but the current key is too broad and the guard needs to be shared with the file-state rewrite path now used on main.
