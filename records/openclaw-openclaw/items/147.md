## Summary

Keep open. Current main has generic Gateway, node, Voice Wake, plugin, and OpenAI-compatible HTTP primitives that make an external Raspberry Pi voice bridge feasible, and the discussion includes a working external implementation, but OpenClaw still does not ship or document a first-class Brabble/Raspberry Pi distributed voice-wake node or a minimal supported external voice-node contract. The remaining work is a product and security boundary decision, not conservative cleanup.

## What This Changes

Add Brabble or an equivalent Raspberry Pi voice-wake pipeline as a distributed OpenClaw node that forwards wake-triggered speech to the Gateway and returns TTS responses.

## Best Possible Solution

Maintainers should choose the supported boundary first: document a minimal external voice-node contract and promote or link a companion through ClawHub/docs, or define a small generic node/plugin API only if current Gateway surfaces are insufficient.

## Reproduction Assessment

Not applicable as a feature/product request rather than a bug. The verification path is high-confidence: current-main source and docs show generic voice/node/Gateway primitives but no first-class Brabble or Raspberry Pi distributed voice-wake node.

## Solution Assessment

Unclear. Direct native Brabble integration is not proven to be the best path; current evidence favors first defining the supported external-node contract and trust boundary, then adding core or plugin API only if there is a real gap.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because an official distributed voice node would combine always-on microphone capture with credentials that can reach privileged Gateway agent surfaces.
