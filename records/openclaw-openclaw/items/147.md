## Summary

Keep open. Current main has generic Gateway, node-event, Voice Wake, and OpenAI-compatible HTTP primitives, and the discussion includes a working external Raspberry Pi companion, but OpenClaw still has no supported Brabble/Raspberry Pi distributed voice-wake node or documented minimal external voice-node contract.

## What This Changes

Request Brabble/Raspberry Pi distributed voice-wake nodes that detect a wake word, send audio/text to the Gateway, and play TTS responses.

## Best Possible Solution

Maintainers should choose the supported boundary first: document a minimal external voice-node contract and promote/link the Raspberry Pi companion through ClawHub/docs, or add a small generic node/plugin API only if current Gateway surfaces are insufficient.

## Reproduction Assessment

Not applicable; this is a feature/product request rather than a bug. The verification path is high-confidence: current-main source and docs show generic voice/node/Gateway primitives but no first-class Brabble or Raspberry Pi distributed voice-wake node.

## Solution Assessment

Unclear. Direct native Brabble integration is not proven to be the best path; the safer maintainable path is to define the external-node contract and trust boundary before adding core code.

## Security Review

Status: needs_attention

Summary: The requested official distributed voice node is security-sensitive because it combines always-on microphone capture with privileged Gateway credentials.

Concerns:

- **[medium] Define the voice-node trust boundary:** `docs/gateway/openai-http-api.md:49`
  - body: Official Brabble/Pi guidance would likely call /v1/chat/completions from an always-on microphone device, and the docs treat bearer access there as full operator access. Credential storage, pairing, revocation, scope, and microphone privacy need explicit design before core docs or code promote this path.
  - confidence: 0.9
