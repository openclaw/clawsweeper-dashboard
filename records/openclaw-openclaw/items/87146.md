## Summary

Keep open. The code change is a narrow gateway authorization hardening and source review supports the reported gap, but it is not merge-ready because it changes a shipped approval boundary without aligned docs and the provided proof is only focused tests/mocks.

## What This Changes

The PR tightens `device.pair.approve` so every non-admin caller checks the pending request before approving non-operator device roles, with server-method regressions for shared-auth operator and node approvals.

## Best Possible Solution

Land the stricter gateway authorization only after docs, upgrade-visible release-note context, and redacted real gateway proof show the pairing-only denial and admin success path.

## Reproduction Assessment

Yes from source, with high confidence but without a live run. Current main sets `callerDeviceId` only for device-token auth and skips the pending role gate for shared-auth sessions, allowing the path to fall through to `approveDevicePairing`.

## Solution Assessment

Mostly yes. Moving the pending-request role gate outside the caller-device-id check is the narrow handler fix, but the PR still needs docs alignment and real gateway proof before merge.
