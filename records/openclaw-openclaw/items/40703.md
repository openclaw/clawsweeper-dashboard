## Summary

Keep open: the linked bug is still source-reproducible on current main, but this PR is not mergeable as-is because it targets a removed startup shim, uses a narrower auth check than runtime behavior, and lacks real behavior proof.

## What This Changes

The PR adds a gateway startup guard and tests that throw when the resolved default model provider has no configured auth before channels start.

## Best Possible Solution

Implement auth visibility in the active startup path using runtime-equivalent provider auth resolution, with explicit maintainer policy on fail-fast versus warning and coverage for env, config, profile, AWS, local synthetic, plugin auth, and skip flags.

## Reproduction Assessment

Yes. The linked bug gives a concrete fresh-profile Anthropic-default/no-auth path, and current main source still starts channels without a default-provider auth gate or warning; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The direction is plausible, but this patch is not the best current solution because it targets a removed module, can false-block valid runtime auth configurations, and hardcodes circular remediation text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but runtime auth also accepts config-backed provider keys, AWS/Bedrock auth, workspace-aware env lookup, and synthetic local/plugin auth. A fail-fast startup guard using this narrower helper can reject configurations that would work at request time.
  - confidence: 0.93
- **[P2] Port the guard to the active startup path:** `src/gateway/server-startup.ts:92-104`
  - body: Current main no longer has `src/gateway/server-startup.ts`; channel startup now runs through `src/gateway/server-startup-post-attach.ts`. As written, this branch needs to be ported to the current startup split before it can affect real gateway startup.
  - confidence: 0.92
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that is already the failing default model, the remediation points back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
