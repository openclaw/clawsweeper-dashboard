## Summary

Keep open, but this branch should not merge as-is: current main still lacks a default-provider startup auth guard/warning, while the PR targets a removed startup module, uses a narrower auth helper than runtime, and has no real behavior proof.

## What This Changes

The branch adds a gateway startup guard and tests that throw before channel startup when the resolved default model provider lacks configured auth.

## Best Possible Solution

Implement auth visibility on the active startup path with the runtime-aligned provider auth contract, deliberate fail-fast-or-warning policy, and coverage for skip flags, config/env/profile, AWS, local synthetic, and plugin auth cases.

## Reproduction Assessment

Yes, from source inspection and the linked report. Current main starts channels without a default-provider auth gate, and the linked bug gives a fresh-profile Anthropic-default/no-auth path; I did not run a live gateway, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The direction is plausible, but this patch is not the best current solution because it targets a removed file, can false-block valid runtime auth, and hardcodes circular remediation text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This gate calls hasAuthForProvider with only provider and agentDir, but runtime auth also accepts config-backed provider keys, workspace-aware env lookup, AWS SDK/Bedrock auth, local synthetic auth, and plugin synthetic auth. A fail-fast startup guard using this narrower helper can reject configurations that would work at request time.
  - confidence: 0.93
- **[P2] Port the guard to the active startup path:** `src/gateway/server-startup.ts:100-105`
  - body: Current main no longer has src/gateway/server-startup.ts; channel startup now runs through src/gateway/server-startup-post-attach.ts. As written, this branch needs to be rebased and ported before it can affect real gateway startup.
  - confidence: 0.92
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that is already the failing default model, the remediation points back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
