## Summary

Keep open: the PR targets a real gateway startup failure mode, but the current patch is not mergeable as-is because its auth guard is narrower than runtime auth and would block valid provider configurations.

## What This Changes

The PR adds a gateway startup auth guard and tests so missing auth for the resolved default model provider fails before channels start.

## Best Possible Solution

Rework the fix onto current `startGatewaySidecars` with a runtime-equivalent bounded auth check, preserving skip-channel behavior and valid non-API-key auth modes.

## Reproduction Assessment

Yes, from source inspection: a config using `amazon-bedrock` aws-sdk auth or local/synthetic provider auth can be accepted by runtime auth while the PR's `hasAuthForProvider` guard returns false. I did not run a live gateway because this review was read-only.

## Solution Assessment

No. The fail-fast direction is plausible, but this patch uses the wrong auth predicate and targets an outdated startup file; the safer fix is to share or mirror runtime auth availability without importing heavy provider runtime during startup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Use runtime auth resolution before blocking startup:** `src/gateway/server-startup.ts:48-50`
  - body: The new guard calls `hasAuthForProvider`, which only checks env keys and auth-profile entries. Runtime auth also accepts config-backed API keys, synthetic local/Ollama auth, plugin synthetic auth, and Amazon Bedrock `aws-sdk`, so valid gateways can now fail before channels start.
  - confidence: 0.9
- **[P3] Avoid repeating the failing model in the example:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests `openai-codex/gpt-5.4`. When that is already the failing default, the remediation tells users to keep the same broken selection, so the example should be generic or choose a different provider and have coverage for that edge case.
  - confidence: 0.86
