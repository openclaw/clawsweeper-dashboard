## Summary

Keep open: the linked bug is plausible and current main still lacks a startup auth gate or clear missing-auth warning, but this branch should not merge as-is because it targets a removed startup module, uses a narrower auth check than runtime execution, lacks real behavior proof, and needs a maintainer decision on fail-fast versus warning/prompt behavior.

## What This Changes

The PR adds a gateway startup auth guard and focused tests to throw before channel startup when the resolved default model provider lacks configured auth.

## Best Possible Solution

Implement default-provider auth visibility in the active post-attach startup or startup-log path using runtime-aligned auth checks, provider-specific remediation, skip-channel coverage, and an explicit warning-versus-fail-fast decision.

## Reproduction Assessment

Yes, at source level: current main starts channels without a default-provider auth gate, and the linked bug includes startup logs plus a working model-switch workaround. I did not run a live macOS/Telegram reproduction in this read-only review.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it edits an obsolete startup module and uses an auth helper that is narrower than runtime provider auth resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This gate calls `hasAuthForProvider` with only provider and agentDir, but runtime auth can come from config/workspace env, AWS/Bedrock, custom local providers, and plugin synthetic auth. As a fail-fast startup blocker, this can reject valid existing configurations before channels start.
  - confidence: 0.93
- **[P1] Port the guard to the active startup path:** `src/gateway/server-startup.ts:100-105`
  - body: Current main no longer uses `src/gateway/server-startup.ts` for sidecar/channel startup; active startup now lives in `src/gateway/server-startup-post-attach.ts`. As written, this branch conflicts and the guard must be ported before it can affect gateway startup.
  - confidence: 0.94
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests `openclaw models set "openai-codex/gpt-5.4"`. If that is already the failing default model, the remediation points users back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
