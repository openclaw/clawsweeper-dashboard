## Summary

Keep open: the startup missing-auth problem is still valid, but this PR cannot merge as-is because it targets a removed startup module, uses a narrower auth check than runtime, lacks real behavior proof, and still needs maintainer policy on fail-fast versus warning behavior.

## What This Changes

The branch adds a gateway startup guard and focused tests that throw before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Implement missing default-provider auth visibility in the active startup path with runtime-aligned auth checks, provider-specific remediation, skip-flag coverage, and explicit maintainer policy on fail-fast versus warning behavior.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main starts channels in src/gateway/server-startup-post-attach.ts without checking default-provider auth first, and the linked bug includes concrete startup logs and workaround. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The PR direction is useful, but this patch is not the best current solution because it edits a removed startup file, uses a narrower helper than runtime auth resolution, and hardcodes remediation that can point back to the failing model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This startup gate calls hasAuthForProvider with only provider and agentDir, so it misses config-backed API keys, workspace env lookup, AWS/Bedrock auth, local synthetic auth, and plugin synthetic auth that runtime accepts. As a fail-fast blocker, that can reject valid configurations before channels start.
  - confidence: 0.93
- **[P1] Port the guard to the active startup path:** `src/gateway/server-startup.ts:92-104`
  - body: Current main no longer uses this startup module for sidecars and channel startup; the active path is src/gateway/server-startup-post-attach.ts. As written, the guard needs a real rebase and port before it can affect current gateway startup.
  - confidence: 0.94
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests openclaw models set "openai-codex/gpt-5.4". If that is already the failing default model, the remediation sends users back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
