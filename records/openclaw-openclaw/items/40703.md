## Summary

Keep this PR open, but it is not mergeable as-is: the underlying startup-auth bug still appears present on current main, while the proposed guard uses a narrower auth check that can reject valid runtime configurations and the contributor has not provided real after-fix startup proof.

## What This Changes

The PR adds a gateway startup guard that throws when the resolved default model provider has no auth, plus focused tests for missing auth, existing auth, and skipped channel startup.

## Best Possible Solution

Add the guard in the current post-attach startup path, use hasAuthForModelProvider or an equivalent runtime-aligned check, avoid misleading remediation text, and include focused regression coverage plus redacted real startup proof.

## Reproduction Assessment

Yes. Source inspection shows current main still starts channels after scheduling non-blocking model prewarm without a default-provider auth gate, and the PR's false-positive path is visible from its narrower helper call.

## Solution Assessment

No. The fail-fast direction is plausible, but this patch should be ported to src/gateway/server-startup-post-attach.ts and use runtime-equivalent auth resolution before blocking startup.
