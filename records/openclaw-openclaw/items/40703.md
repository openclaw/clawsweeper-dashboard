## Label Justifications

- `P1`: The PR targets a broken gateway/channel workflow where a bot can appear online but fail to reply when the default model provider lacks auth.
- `merge-risk: 🚨 compatibility`: The proposed hard startup guard can turn existing upgrade setups from request-time failure or warning into gateway startup failure.
- `merge-risk: 🚨 auth-provider`: The diff changes provider credential detection before the runtime model-auth contract runs.
- `merge-risk: 🚨 availability`: A false-negative auth check would prevent channel startup and make the gateway unavailable despite valid auth modes.

## Summary

Keep open, but not merge-ready: the bug signal is valid, while the branch is now conflicting, targets the old startup path, uses a narrower auth check than runtime provider auth, and still lacks real behavior proof.

## What This Changes

The branch adds a gateway startup guard and tests that throw before channel startup when the resolved default model provider lacks detected auth.

## Best Possible Solution

Implement default-provider auth visibility in the current startup/log or post-attach path using runtime-aligned auth evidence, provider-specific remediation, skip-channel coverage, and an explicit warning-versus-fail-fast decision.

## Reproduction Assessment

Yes, at source level. The linked report describes the silent gateway/channel failure, and current main still logs the selected model and starts sidecars without a default-provider missing-auth guard; I did not run live macOS or Telegram proof in this read-only review.

## Solution Assessment

No. The goal is valid, but this branch targets a stale/conflicting startup file and checks less auth evidence than runtime uses; the safer solution is a current-main startup/log change using runtime-aligned auth and an explicit fail-fast policy choice.
