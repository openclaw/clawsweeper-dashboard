## Summary

Keep open, but do not merge as-is. The reported silent startup failure is still relevant on current main, but this branch targets a stale startup module, uses a narrower auth helper that can reject valid runtime auth configurations, has a misleading remediation example, and still lacks real after-fix behavior proof.

## What This Changes

The branch adds a gateway startup auth guard in `src/gateway/server-startup.ts`, throws before channel startup when the default model provider lacks auth, and adds unit tests for missing auth, auth present, and skipped channels.

## Best Possible Solution

Port the guard to the current post-attach channel startup path, use the runtime-aligned provider auth contract, avoid circular remediation text, settle fail-fast versus warning behavior explicitly, and add focused tests plus redacted real startup proof.

## Reproduction Assessment

Yes. Source inspection shows current main starts channels without a default-provider auth gate, and the linked issue provides a concrete new-profile symptom and workaround; I did not run a live gateway startup in this read-only sweep.

## Solution Assessment

No. The fail-fast direction is plausible, but this implementation is not the best fix because it is on the stale startup path, uses a narrower helper than runtime auth, and can produce a circular remediation command.
