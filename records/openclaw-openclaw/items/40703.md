## Summary

Keep open: the PR addresses a real startup-auth problem, but the branch is conflicting, targets an obsolete startup module, lacks real behavior proof, and can reject runtime-valid auth configurations.

## What This Changes

The PR adds a gateway startup auth guard and tests that throw before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Implement startup auth visibility in the active post-attach startup path using the runtime-aligned auth contract, with provider-specific remediation and coverage for env, config, profile, AWS, local synthetic, plugin synthetic, and skip-flag cases.

## Reproduction Assessment

Yes, from source and the linked bug path. Current main still reaches channel startup without a default-model auth guard or missing-auth warning, but I did not run a live gateway in this read-only review.

## Solution Assessment

No. The direction is plausible, but this patch is not the best current fix because it targets an obsolete startup module, uses narrower auth resolution than runtime, and hardcodes circular remediation text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but runtime auth also accepts config-backed provider keys, workspace-aware env lookup, AWS/Bedrock auth, and synthetic local/plugin auth. A fail-fast startup guard using this narrower helper can reject configurations that would work at request time.
  - confidence: 0.93
- **[P2] Port the guard to the active startup path:** `src/gateway/server-startup.ts:92-104`
  - body: Current main starts sidecars through `src/gateway/server-startup-post-attach.ts`, while this patch modifies `src/gateway/server-startup.ts`. The guard needs to be rebased and ported before it can affect real gateway startup.
  - confidence: 0.92
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that is already the failing default model, the remediation points back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
