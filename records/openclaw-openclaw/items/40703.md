## Summary

Keep open. Current main still lacks the requested fail-fast default-model auth guard, and this PR remains a relevant implementation candidate, but the proposed patch is not merge-ready because it targets the old startup surface and would reject several auth modes that runtime currently accepts.

## What This Changes

The PR adds a gateway startup auth guard and focused tests that throw before channel startup when the resolved default model provider lacks configured auth.

## Best Possible Solution

If maintainers want this fail-fast behavior, add the guard in the current post-attach sidecar startup path immediately before channel startup, honor the existing channel/provider skip flags, use the runtime-aligned provider auth checker with cfg/workspace/env/agent-dir context, avoid circular remediation text, and cover env, profile, config, AWS, local synthetic, plugin synthetic, and skip-flag cases.

## Reproduction Assessment

Yes for the PR defects. Static inspection shows a valid runtime-auth provider can be rejected by the PR's narrower helper, and the hardcoded remediation becomes circular when the failing default is already openai-codex/gpt-5.4.

## Solution Assessment

No. The target behavior is plausible, but this patch is not the best solution because it targets the old startup surface and does not reuse the runtime auth availability seam that current main already exposes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use runtime auth availability before blocking startup:** `src/gateway/server-startup.ts:47-50`
  - body: The new startup gate calls hasAuthForProvider with only provider and agentDir, but that helper only checks env keys and stored profiles. Valid runtime auth through config/workspace keys, AWS SDK Bedrock auth, local synthetic auth, or plugin synthetic auth can be rejected before channels start.
  - confidence: 0.9
- **[P3] Avoid suggesting the failing default model:** `src/gateway/server-startup.ts:61`
  - body: The error text always suggests openclaw models set "openai-codex/gpt-5.4". When that is already the default model whose provider lacks auth, the remediation points users back to the same broken configuration.
  - confidence: 0.82
