## Summary

Keep open: the linked startup-auth problem is still not implemented on current main, but this PR cannot merge as-is because it targets an obsolete startup file, uses a narrower auth helper than runtime execution, hardcodes circular remediation, and lacks real behavior proof.

## What This Changes

This PR adds a gateway startup guard that throws before channel startup when the resolved default model provider lacks auth, with focused unit tests.

## Best Possible Solution

Implement missing default-provider auth visibility in the active startup path using runtime-aligned auth checks, provider-specific remediation, skip-channel coverage, and an explicit maintainer choice on warning versus fail-fast behavior.

## Reproduction Assessment

Yes. This is source-reproducible: current main starts channels from startGatewaySidecars without checking default-provider auth first, and the linked issue includes concrete startup logs plus a model-switch workaround.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it edits stale startup code, uses a narrower helper than runtime auth resolution, and can point users back to the same failing model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This gate calls hasAuthForProvider with only provider and agentDir, but runtime auth can come from config/workspace env, AWS/Bedrock, custom local, and plugin synthetic auth. As a fail-fast startup blocker, that can reject valid configurations before channels start.
  - confidence: 0.93
- **[P1] Port the guard to the active startup path:** `src/gateway/server-startup.ts:92-104`
  - body: Current main no longer tracks src/gateway/server-startup.ts; sidecars and channel startup now live in src/gateway/server-startup-post-attach.ts. As written, this branch needs a rebase and port before the guard can affect current gateway startup.
  - confidence: 0.94
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests openclaw models set "openai-codex/gpt-5.4". If that is already the failing default model, the remediation sends users back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
