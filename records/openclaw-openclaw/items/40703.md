## Summary

Keep open: the underlying startup-auth problem is still plausible, but this branch is not merge-ready because it targets an obsolete startup module, uses a narrower auth check than runtime resolution, and still lacks real startup proof.

## What This Changes

The PR adds a gateway startup guard and focused tests that throw before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Implement the auth visibility in the current startup path using the runtime-aligned provider auth contract, choose fail-fast versus warning deliberately, avoid circular remediation text, and prove it with focused tests plus redacted live startup logs.

## Reproduction Assessment

Yes. The linked issue gives a concrete new-profile startup and Telegram-silence path, and source inspection on current main shows channels start without a default-provider auth gate; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The fail-fast direction is plausible, but this branch is not the best current fix because it targets a removed startup module, uses a narrower helper than runtime auth resolution, and can suggest the same failing model.
