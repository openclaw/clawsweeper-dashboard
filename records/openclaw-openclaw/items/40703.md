## Label Justifications

- `P1`: The PR affects gateway startup behavior for real channel workflows that can otherwise appear online but fail to reply.
- `merge-risk: 🚨 compatibility`: The proposed fail-fast guard can turn existing upgrade setups from request-time failure or warning into gateway startup failure.

- `merge-risk: 🚨 availability`: False-negative auth detection would prevent channel startup and make the gateway unavailable for users with valid alternative auth modes.

## Summary

Keep open: the user-facing goal is still relevant, but this PR is not merge-ready because it conflicts with current main, targets the old startup module, uses auth detection narrower than the runtime contract, and still lacks real behavior proof.

## What This Changes

The PR adds a gateway startup guard that fails before channel startup when the resolved default model provider lacks configured auth, plus focused tests.

## Best Possible Solution

Implement default-provider auth visibility in the active post-attach or startup-log path using runtime-aligned provider auth evidence, provider-specific remediation, skip-channel coverage, and an explicit warning-versus-fail-fast policy choice.

## Reproduction Assessment

Yes at source level, but not with live macOS or Telegram proof in this read-only pass. The linked report shows the gateway starting with an unauthenticated Anthropic default, and current main still lacks a default-provider auth startup guard.

## Solution Assessment

No. The user-facing goal is valid, but this patch targets a stale startup module and uses an auth helper that is narrower than runtime provider auth resolution; the safer path is a current-main implementation with an explicit warning-versus-fail-fast decision.
