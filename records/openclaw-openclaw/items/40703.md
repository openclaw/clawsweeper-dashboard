## Summary

Keep open: current main does not implement this fail-fast default-model auth guard, and the PR targets a real bug, but the proposed guard would reject valid runtime auth configurations by using a narrower provider-auth helper before channel startup.

## What This Changes

The PR adds a gateway startup guard that errors before channel startup when the resolved default model provider lacks configured auth, plus focused tests.

## Best Possible Solution

Port the guard to the current gateway sidecar startup path and base it on the runtime-equivalent provider auth availability contract, with regression coverage for missing auth and valid config-backed auth modes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a default provider with valid `models.providers.<provider>.apiKey`, Bedrock AWS SDK auth, or synthetic local provider auth can be accepted by runtime auth resolution but rejected by the PR’s `hasAuthForProvider` startup gate when no env/profile entry exists.

## Solution Assessment

No. The fail-fast direction is a reasonable fix for the linked startup bug, but the implementation should use `hasAuthForModelProvider` or an equivalent runtime-aligned check and avoid circular example guidance.
