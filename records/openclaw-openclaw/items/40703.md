## Label Justifications

- `P1`: The linked behavior can leave channel users with an online-looking gateway that does not respond when the default model provider lacks usable auth.
- `merge-risk: 🚨 compatibility`: The PR can turn previously working auth configurations into startup failures without an upgrade-safe migration or policy decision.
- `merge-risk: 🚨 auth-provider`: The proposed guard uses a narrower auth check than runtime provider auth resolution.
- `merge-risk: 🚨 availability`: The PR intentionally blocks gateway/channel startup and can do so for false-negative auth states.

## Summary

Keep open: the underlying startup/auth problem is still plausible on current main, but this branch is not mergeable as-is because it targets a stale startup module, can reject runtime-valid auth, and lacks real behavior proof.

## What This Changes

The PR adds a gateway startup guard and tests to fail before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Implement default-provider auth visibility in the active post-attach startup or startup-log path using runtime-aligned auth checks, provider-specific remediation, skip-channel coverage, and an explicit warning-versus-fail-fast decision.

## Reproduction Assessment

Yes, at source level: current main starts channels without a default-provider auth visibility gate, and the linked bug includes startup logs plus a working model-switch workaround. I did not run a live macOS/Telegram reproduction in this read-only review.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it edits an obsolete startup module and uses an auth helper that is narrower than runtime provider auth resolution.

  - body: The error always suggests openclaw models set "openai-codex/gpt-5.4". If that is already the failing default model, the remediation points users back to the same broken setup instead of giving an actionable alternative.
