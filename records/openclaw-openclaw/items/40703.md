## Summary

Keep open for maintainer/contributor follow-up: the requested bug fix is still relevant, but this PR is outdated, targets a startup module that no longer exists on current main, uses a narrower auth check than runtime, and lacks required real behavior proof.

## What This Changes

The PR adds a gateway startup guard that throws before channel startup when the resolved default model provider has no configured auth, plus focused tests for that helper.

## Best Possible Solution

Implement missing default-provider auth visibility in the active post-attach startup path using the runtime-aligned auth contract, with maintainer-approved fail-fast or warning behavior and provider-specific remediation.

## Reproduction Assessment

Yes, from source and linked bug context. Current main resolves/logs the default model and starts channels in src/gateway/server-startup-post-attach.ts without checking default-provider auth first; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The PR direction is plausible, but this patch is not the best current solution because it edits a removed startup file, uses a narrower auth helper than runtime, and hardcodes circular remediation text.

  - body: This guard calls hasAuthForProvider with only provider and agentDir, but runtime accepts config-backed API keys, workspace-aware env lookup, AWS/Bedrock auth, local synthetic auth, and plugin synthetic auth.   - confidence: 0.93

  - confidence: 0.94

  - body: The error always suggests openclaw models set "openai-codex/gpt-5.4". If that is already the failing default model, the remediation points users back to the same broken setup instead of giving an actionable alternative.
