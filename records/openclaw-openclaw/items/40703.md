## Label Justifications

- `P1`: The PR affects gateway startup for channel workflows that can appear online but fail to reply when the default model provider lacks auth.
- `merge-risk: 🚨 compatibility`: The proposed hard startup guard can turn existing upgrade setups from request-time failure or warning into gateway startup failure.
- `merge-risk: 🚨 auth-provider`: The diff changes provider credential detection before the runtime model-auth contract runs.
- `merge-risk: 🚨 availability`: A false-negative auth check would prevent channel startup and make the gateway unavailable despite valid auth modes.

## Summary

Keep open for maintainer handling: the user-facing problem is real, but this stale PR is conflicting, targets an old startup module, lacks real behavior proof, and would turn missing default-model auth into a hard startup failure without runtime-equivalent auth checks or an explicit upgrade decision.

## What This Changes

The PR adds a gateway startup guard and tests that throw before channel startup when the resolved default model provider lacks detected auth.

## Best Possible Solution

Implement default-provider auth visibility in the current post-attach or startup-log path with runtime-aligned auth evidence, provider-specific remediation, skip-channel coverage, and an explicit warning-versus-fail-fast policy choice.

## Reproduction Assessment

Yes at source level. The linked report describes a gateway that starts with an unauthenticated default provider, and current main still logs the selected model and starts channels without a default-provider auth guard; I did not run live macOS or Telegram proof in this read-only review.

## Solution Assessment

No. The user-facing goal is valid, but this PR is no longer the best patch because it targets a stale startup module and uses a narrower auth helper than runtime provider auth resolution.

Overall correctness: patch is incorrect

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This fail-fast gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but runtime auth can come from config/workspace env, AWS/Bedrock, custom local providers, and plugin synthetic auth. As a startup blocker, it can reject valid existing configurations before channels start.
  - confidence: 0.94
- **[P1] Port the guard to the active startup path:** `src/gateway/server-startup.ts:97-103`

  - confidence: 0.95
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests `openclaw models set "openai-codex/gpt-5.4"`. If that is already the failing default model, the remediation points users back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
