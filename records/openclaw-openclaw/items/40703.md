## Summary

Keep this PR open, but it is not merge-ready: the goal is still useful, while the branch is now conflicting, targets the old startup module, uses a narrower auth check than current runtime auth, and still lacks real after-fix behavior proof.

## What This Changes

The PR adds a gateway startup guard and focused tests that throw before channel startup when the resolved default model provider lacks configured auth.

## Best Possible Solution

Implement default-provider auth visibility in the active post-attach or startup-log path using runtime-aligned provider auth evidence, provider-specific remediation, skip-channel coverage, and an explicit warning-versus-fail-fast decision.

## Reproduction Assessment

Yes at source level: current main starts channels from `src/gateway/server-startup-post-attach.ts` without a default-provider auth guard, and the linked report includes startup logs plus a model-switch workaround. I did not run a live macOS or Telegram reproduction in this read-only review.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it edits a stale startup module and uses an auth helper that is narrower than runtime provider auth resolution.
