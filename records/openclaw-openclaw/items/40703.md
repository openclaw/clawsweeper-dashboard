## Summary

Keep open. The reported startup-auth problem is still plausible, but this PR is not merge-ready because it conflicts with current main, edits the old startup module, blocks startup with an auth helper narrower than runtime auth resolution, and has no real behavior proof.

## What This Changes

The PR adds a gateway startup guard and tests that throw when the resolved default model provider appears to lack configured auth.

## Best Possible Solution

Implement default-provider auth visibility in the active post-attach or startup-log path with runtime-aligned provider auth evidence, provider-specific remediation, skip-channel coverage, and an explicit warning-versus-fail-fast decision.

## Reproduction Assessment

Yes at source level, but not with a live macOS/Telegram run here. Current main starts channels from the post-attach startup path without a default-provider auth guard, and the linked report includes startup logs plus a model-switch workaround.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it edits a stale startup module and uses an auth helper that is narrower than runtime provider auth resolution.
