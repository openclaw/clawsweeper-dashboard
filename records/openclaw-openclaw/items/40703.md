## Summary

Keep open. The underlying bug remains plausible, but this PR is not mergeable on current main: it edits a removed startup module, uses a narrower auth helper than runtime resolution, hardcodes circular remediation, and lacks real behavior proof.

## What This Changes

The branch adds a gateway startup guard and tests that throw before channel startup when the resolved default model provider lacks configured auth.

## Best Possible Solution

Implement auth visibility in the active startup path using runtime-equivalent provider auth resolution, with maintainer policy on fail-fast versus warning and coverage for env, config, profile, AWS, local synthetic, plugin auth, and skip flags.

## Reproduction Assessment

Yes. The linked bug gives a concrete fresh-profile Anthropic-default/no-auth path, and current main source still reaches channel startup without a default-model auth guard or startup warning; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The direction is plausible, but this patch targets a removed module, uses narrower auth resolution than runtime, and hardcodes circular remediation text; a ported warning or fail-fast guard needs maintainer policy first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use runtime-equivalent auth before blocking startup:** `src/gateway/server-startup.ts:49-52`
  - body: This gate calls hasAuthForProvider with only provider and agentDir, but runtime auth also accepts config-backed provider keys, AWS/Bedrock auth, workspace-aware env lookup, and synthetic local/plugin auth. A fail-fast startup guard using this narrower helper can reject configurations that would work at request time.
  - confidence: 0.93
- **[P2] Port the guard to the active startup path:** `src/gateway/server-startup.ts:92-104`
  - body: Current main no longer has src/gateway/server-startup.ts; channel startup now runs through src/gateway/server-startup-post-attach.ts. As written, this branch needs to be ported before it can affect real gateway startup.
  - confidence: 0.92
- **[P3] Avoid suggesting the same failing model:** `src/gateway/server-startup.ts:61`
  - body: The error always suggests openclaw models set "openai-codex/gpt-5.4". When that is already the failing default model, the remediation points back to the same broken setup instead of giving an actionable alternative.
  - confidence: 0.87
