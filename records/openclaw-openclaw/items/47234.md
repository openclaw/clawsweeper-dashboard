## Summary

This PR should stay open because current main still lacks a group/channel-to-main route binding surface, but the proposed patch is not ready to merge: it introduces a privacy-sensitive session-boundary feature, has unresolved routing gaps, includes a bundled IRC active-client change better handled in its existing companion PR, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds a per-binding `groupScope` option for routing matched group/channel traffic to an agent main session and also bundles an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive binding session target under a consistent `bindings[].session` contract, keep group/channel isolation as the default, wire inbound/outbound/Telegram-topic session derivation, add regression coverage and docs for the privacy implications, and handle IRC proactive delivery through the separate IRC PR or a gateway-owned design.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection confirms the current main contract has no group-to-main binding option and still isolates non-direct peers.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but the patch needs a settled config contract, docs for the session-boundary change, outbound and Telegram topic propagation, focused tests, and separation from the IRC active-client work before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new option is only threaded through inbound route resolution. Outbound helpers that infer a target still build fallback session keys from global `dmScope`, so sends without an explicit `sessionKey` can mirror the same merged group/channel conversation into a per-channel session instead of the configured main session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and only pass `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through a real gateway-owned path:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry added here is module-local. CLI proactive sends and other processes outside the gateway monitor will see an empty registry and fall back to the transient unjoined IRC send path that this PR is trying to avoid.
  - confidence: 0.86
- **[P2] Wait for IRC joins before publishing monitor clients:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the client immediately after `connectIrcClient()` resolves, but that readiness is based on connection registration, not confirmed channel membership. Startup proactive sends can therefore use a socket that has not joined the target channel yet.
  - confidence: 0.78
- **[P3] Fall back when active IRC sends throw:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
