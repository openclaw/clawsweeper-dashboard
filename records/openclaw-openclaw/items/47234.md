## Summary

Keep open. Current main still lacks a group-to-main binding option, and this PR addresses a real user-requested feature, but the patch is not mergeable as-is because it leaves outbound, Telegram topic, IRC delivery, privacy docs, and real behavior proof unresolved.

## What This Changes

The branch adds a per-binding `groupScope` route option for group/channel peers to use an agent's main session and bundles an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract that preserves default group/channel isolation, supports selected main or shared-session routing, wires inbound, outbound, Telegram topic, queue/runtime-policy, docs, and tests, and handles IRC proactive delivery through its own gateway-owned design.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection gives high confidence that current main still has only DM-specific binding session scope and keeps non-direct peers isolated.

## Solution Assessment

No. The per-binding opt-in direction is plausible and has user support, but this patch is not the best mergeable solution until outbound routing, Telegram topic routing, IRC delivery, privacy docs, and real proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound `resolveAgentRoute()`. Outbound helpers that infer a target still build group/channel session keys from global `dmScope` only, so sends without an explicit `sessionKey` can fork a configured main-session group back into a per-channel session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and pass only `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through a gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other processes outside the gateway monitor will see an empty registry and fall back to the transient unjoined IRC send path this PR is trying to avoid.
  - confidence: 0.86
- **[P2] Publish IRC clients only after joined membership is known:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the client immediately after `connectIrcClient()` resolves, but that readiness is based on connection registration rather than confirmed channel membership. Startup sends can therefore use a socket that has not joined the target channel yet.
  - confidence: 0.78
- **[P3] Fall back when active IRC sends throw:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
