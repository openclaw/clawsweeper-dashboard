## Summary

Keep open. Current main still lacks the requested group/channel-to-main session contract, the PR has visible user demand and a plausible per-binding direction, but the branch is not mergeable as-is because it drops the new scope in outbound/Telegram paths, mixes in an incomplete IRC delivery fix, changes a privacy-sensitive boundary without docs, and lacks inspectable real behavior proof.

## What This Changes

The PR adds a per-binding `groupScope` option for group/channel peers to route into an agent main session and also adds an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract that preserves default group/channel isolation, supports selected main or shared-session routing, carries the setting through inbound, outbound, Telegram topic, queue/runtime-policy, docs, and tests, and handles IRC proactive delivery through a gateway-owned joined connection.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection on current main confirms group/channel peers still resolve to isolated keys and the config contract has no `groupScope` surface.

## Solution Assessment

No. The per-binding opt-in direction is plausible and supported by user comments, but this patch is not the best mergeable solution until outbound routing, Telegram topic routing, IRC delivery design, privacy docs, and real proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Carry groupScope through outbound session routing:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound `resolveAgentRoute()`. Outbound helpers that infer a target still build group/channel session keys from global `dmScope` only, so sends without an explicit `sessionKey` can fork a configured main-session group back into a per-channel session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and pass only `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through a gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other process-separated send paths will not see the gateway monitor's map, so they can fall back to the transient unjoined IRC send path this PR is trying to avoid.
  - confidence: 0.9
- **[P2] Publish IRC clients only after joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the active client immediately after `connectIrcClient()` resolves, but readiness is based on registration before confirmed channel membership. Startup sends can therefore use a socket that has not joined the target channel yet.
  - confidence: 0.82
- **[P3] Fall back when active IRC sends fail:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
