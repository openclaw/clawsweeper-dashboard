## Summary

Keep open. Current main still lacks a group/channel-to-main session scope, and the PR is a plausible implementation direction, but it is not merge-ready because the proposed scope is not propagated through outbound and Telegram topic routing, lacks user-facing privacy documentation, and bundles IRC active-client work that is already tracked separately in #45662.

## What This Changes

This PR adds a per-binding `groupScope` route option for group/channel sessions and includes an IRC active-client registry so sends can reuse the monitor connection.

## Best Possible Solution

Ship this as a first-class, documented binding session-scope feature: settle the config shape alongside existing binding session overrides, propagate the scope through inbound routing, outbound session derivation, Telegram topic overrides, generated config help, docs, changelog, and focused regression tests, then keep the IRC active-client fix on its own PR path.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main rejects or omits `groupScope`, current non-direct peers resolve to isolated group/channel keys, and the PR diff leaves outbound and Telegram recomputation paths without the new scope. No tests were run because this review was read-only.

## Solution Assessment

No. The per-binding opt-in idea is plausible, but this patch is not the best implementation yet because it misses supported session derivation paths, lacks privacy-facing documentation, and mixes in unrelated IRC work already tracked by #45662.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: When a binding sets `groupScope: "main"`, inbound routing can collapse group/channel traffic onto the main session, but outbound helpers still derive group targets from only global `dmScope`. Sends that omit an explicit `sessionKey` can therefore fork the same configured conversation back into a per-channel session instead of the main session.
  - confidence: 0.89
- **[P2] Preserve groupScope in Telegram topic overrides:** `src/routing/resolve-route.ts:100`
  - body: The new scope is not carried into Telegram forum-topic `topicAgentId` overrides. That path recomputes `route.sessionKey` with `buildAgentSessionKey()` and only global `dmScope`, so a topic matching `groupScope: "main"` would be overwritten back to a per-topic group session.
  - confidence: 0.84
- **[P3] Fall back when active IRC sends fail:** `extensions/irc/src/send.ts:72-74`
  - body: When an active monitor client is found, the new IRC path calls `active.sendPrivmsg()` and returns. If the monitor connection drops after `isReady()` but before the write, proactive sends fail instead of trying the transient fallback preserved for missing active clients.
  - confidence: 0.72
