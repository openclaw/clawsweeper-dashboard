## Summary

Keep open: the PR targets a valid feature gap, but it is not merge-ready because it adds a privacy-sensitive session-sharing contract without complete routing coverage, real behavior proof, or a settled public config shape.

## What This Changes

The PR adds a per-binding `groupScope` option for routing matched group/channel peers into an agent main session and also bundles an IRC active-client reuse change for proactive sends.

## Best Possible Solution

Define a documented additive binding-session target under the existing `bindings[].session` contract, preserve default group/channel isolation, thread the chosen target through inbound, outbound, Telegram topic routing, docs, tests, and security semantics, and keep the IRC proactive-send fix separately reviewed or redesigned.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection gives high confidence that current main still isolates non-direct peers and exposes no group-to-main option.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch should not merge until it aligns with the nested binding session contract, covers outbound and Telegram topic routing, separates or fixes IRC delivery, documents security semantics, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Carry groupScope through outbound session routing:** `src/routing/resolve-route.ts:673`
  - body: This only threads the new scope through inbound route resolution. Sends that omit an explicit `sessionKey` still derive their base session through `buildOutboundBaseSessionKey()`, which reads only global `dmScope`, so a configured main-scoped group can fork history into the old per-channel session.
  - confidence: 0.91
- **[P2] Keep groupScope inside binding session config:** `src/config/types.agents.ts:43`
  - body: Current main already exposes route-level session policy as `bindings[].session.dmScope` with a strict `BindingSessionSchema`. Adding `bindings[].groupScope` as a top-level sibling creates a competing public config shape for the same concept and makes future session-target policy harder to evolve consistently.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `sessionKey` after `resolveAgentRoute()` and pass only `dmScope`/`identityLinks`. A topic that matches a group-to-main binding will be rewritten back to a per-topic group session instead of the configured main session.
  - confidence: 0.86
- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: The new active-client lookup is a module-local map. `openclaw message send` runs through the CLI send path while IRC outbound remains direct, so CLI or cross-process proactive sends can still fall back to the same transient unjoined client that this change is trying to avoid.
  - confidence: 0.9
- **[P2] Publish IRC clients only after confirmed JOIN membership:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` marks the client ready on numeric `001` and sends JOIN commands afterward. Registering the monitor client immediately after connect resolves can still allow channel sends before the server has accepted channel membership.
  - confidence: 0.84
