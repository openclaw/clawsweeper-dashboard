## Summary

## What This Changes

This PR adds a top-level per-binding `groupScope` option for routing matched group/channel peers into the agent main session and bundles an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract under the current route-binding `session` config, preserve default group/channel isolation, carry the chosen target through inbound, outbound, Telegram topics, runtime policy, docs, and tests, and keep the IRC delivery fix either separately reviewed or redesigned around a gateway-owned joined connection.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection gives high confidence that current main still isolates non-direct peers and exposes no group-to-main option.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch is not the best mergeable solution until it aligns with the current binding session contract, carries scope through outbound and Telegram topic routing, separates or redesigns IRC delivery, documents security semantics, and adds real behavior proof.

Full review comments:

- **[P2] Nest groupScope under the binding session contract:** `src/config/types.agents.ts:43`
  - body: Current main already puts per-binding session routing under `bindings[].session.dmScope`. Adding `groupScope` at the top level would create a second public shape for the same routing concern, so the PR should rebase and put the group/channel session target under the chosen `session` contract before this becomes config API.
  - confidence: 0.86
- **[P1] Carry groupScope through outbound session routing:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound `resolveAgentRoute()`. Outbound helpers that infer a target still build group/channel session keys from global `dmScope` only, so sends without an explicit `sessionKey` can fork a configured main-session group back into a per-channel session.
  - confidence: 0.9
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and pass only `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.86
- **[P2] Route IRC active sends through a gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other process-separated send paths will not see the gateway monitor's map, so they can still fall back to the transient unjoined IRC send path this PR is trying to avoid.
  - confidence: 0.88
- **[P2] Publish IRC clients only after joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the active client immediately after `connectIrcClient()` resolves, but readiness is set on IRC login before confirmed channel membership. Startup sends can therefore use a socket that has not joined the target channel yet.
  - confidence: 0.82
- **[P3] Fall back when active IRC sends fail:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
