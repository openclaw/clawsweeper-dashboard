## Summary

Keep open. Current main still lacks the requested group/channel-to-main binding contract, and this PR is a privacy-sensitive feature with unresolved correctness gaps, missing real behavior proof, and an IRC delivery fix bundled into the same branch.

## What This Changes

The PR adds a binding-level `groupScope` option for routing matched group/channel peers to an agent main session and also adds an IRC active-client registry for proactive sends.

## Best Possible Solution

Ship a documented additive binding session-scope or typed session-target contract aligned with `bindings[].session`, preserving default group isolation and wiring inbound, outbound, Telegram topic overrides, generated config metadata, docs, privacy guidance, and focused tests; split or rebase the IRC delivery fix.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection gives high confidence current main still lacks group/channel-to-main binding routing, and the proposed branch can be reviewed through routing, outbound, Telegram, and IRC source paths.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch is incomplete until the API shape, privacy docs, outbound and Telegram propagation, tests, and separate IRC delivery design are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new option is only threaded through inbound route resolution. Outbound helpers that infer a target still build the base session from global `dmScope`, so sends without an explicit `sessionKey` can mirror a merged group/channel back into a per-channel session instead of the configured main session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and only pass `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through the gateway connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends run outside the gateway monitor process, so the lookup is empty and the code falls back to the transient unjoined IRC send path that this PR is trying to avoid.
  - confidence: 0.87
- **[P2] Wait for IRC joins before publishing monitor clients:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` marks readiness on numeric `001`, sends JOIN commands, and resolves without waiting for confirmed channel membership. Registering the client immediately can route startup proactive sends through a socket that has not joined the target channel.
  - confidence: 0.82
- **[P3] Fall back when active IRC sends throw:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
