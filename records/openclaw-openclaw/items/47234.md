## Summary

Keep open. The requested group/channel-to-main session feature is still absent from current main, but this PR is incomplete: it adds a privacy-sensitive config surface without docs/policy coverage, misses outbound and Telegram routing paths, and bundles IRC active-client work that still has runtime-boundary issues.

## What This Changes

Adds a top-level `bindings[].groupScope` route option for group/channel-to-main session keys and bundles an IRC active-client registry to reuse monitor connections for sends.

## Best Possible Solution

Define a documented additive binding session target aligned with `bindings[].session`, preserve default group/channel isolation, wire inbound, outbound, Telegram topic overrides, generated metadata, docs, and tests, and handle IRC proactive delivery through a separate gateway-owned design.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection shows current main still isolates non-direct peers and lacks a group-to-main binding surface.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch is incomplete until the config contract, privacy docs, outbound inference, Telegram topic routing, tests, and separate IRC delivery design are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new option is only threaded through inbound route resolution. Outbound helpers that infer a target still build fallback session keys from global `dmScope`, so sends without an explicit `sessionKey` can mirror the same merged group/channel conversation into a per-channel session instead of the configured main session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and only pass `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through the gateway connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends run outside the gateway monitor process, so the lookup is empty and the code falls back to the transient unjoined IRC send path that this PR is trying to avoid.
  - confidence: 0.87
- **[P2] Wait for IRC joins before publishing monitor clients:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` marks readiness on numeric `001`, sends JOIN commands, and resolves without waiting for confirmed channel membership. Registering the client immediately can route startup proactive sends through a socket that has not joined the target channel.
  - confidence: 0.82
- **[P3] Fall back when active IRC sends throw:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
