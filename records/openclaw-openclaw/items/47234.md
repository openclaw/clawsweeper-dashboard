## Summary

Keep open. The feature is still absent from current main and the PR addresses a real routing gap, but the patch is not merge-ready because the new scope is not propagated through all session derivation paths, lacks user-facing privacy documentation, and is mixed with the separate IRC active-client work from #45662.

## What This Changes

This PR adds an opt-in route-binding groupScope option for group/channel peers and also adds an IRC active-client registry for proactive IRC sends.

## Best Possible Solution

Ship group session scope as a documented binding session-scope feature, likely aligned with bindings[].session, with inbound routing, outbound derivation, Telegram topic overrides, generated config help, docs, changelog, and focused regression tests all updated while keeping the IRC active-client fix on its own path or rebased after #45662.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main has no groupScope, group/channel peers derive isolated keys, and the PR diff would leave outbound and Telegram topic recomputation paths using only dmScope.

## Solution Assessment

No. The per-binding opt-in direction is plausible and supported by #7524 plus the recent Discord-use comment, but this implementation is not the best form until it covers all session-key paths, documents the privacy boundary, and resolves the config-shape and #45662 split decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new binding scope is only threaded through inbound route resolution. Outbound fallback sends still derive group targets from `buildOutboundBaseSessionKey()`, which passes only global `dmScope`, so sends without an explicit `sessionKey` can fork a configured group-to-main conversation back into a per-channel session.
  - confidence: 0.89
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:65-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and only global `dmScope`. A topic that matches a group-to-main binding can therefore be overwritten back to a per-topic group session instead of staying on the main session.
  - confidence: 0.84
- **[P3] Fall back when active IRC sends fail:** `extensions/irc/src/send.ts:72-74`
  - body: When the new active monitor client path is used, `active.sendPrivmsg()` returns immediately without a transient fallback. If the persistent client drops after the readiness check but before the write, proactive IRC sends fail even though the old transient path could still establish a fresh connection.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The patch does not add dependency, workflow, or supply-chain risk, but it exposes an opt-in group-to-main privacy-boundary collapse without generated config help or public docs.

Concerns:

- **[medium] Document the group-to-main privacy boundary:** `src/config/zod-schema.agents.ts:40`
  - body: `groupScope: "main"` makes messages from a matched group/channel part of the agent main-session context for other interactions. The PR body acknowledges this, but the diff does not update generated config help or public docs, so users could enable the sharing mode without clear product guidance.
