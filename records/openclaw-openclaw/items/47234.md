## Summary

Keep open: current main still lacks the requested group-to-main routing surface, and this PR is useful but not mergeable yet because the new scope is lost in outbound and Telegram topic paths, the IRC active-client change overlaps a separate open PR, and contributor real behavior proof is still missing.

## What This Changes

The branch adds a top-level per-binding `groupScope` route option to collapse matched group/channel peers into an agent main session, and also adds an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract under binding session config, preserve default group/channel isolation, carry it through inbound, outbound, Telegram topics, runtime policy, docs, and tests, and land IRC proactive delivery through one canonical gateway-owned connection design.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection gives high confidence that current main still lacks a group-to-main option and keeps non-direct peers isolated.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch is not the best mergeable solution until outbound routing, Telegram topic routing, IRC delivery design, privacy docs/runtime semantics, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Carry groupScope through outbound session routing:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound `resolveAgentRoute()`. Outbound helpers that infer a target still build group/channel session keys from global `dmScope` only, so sends without an explicit `sessionKey` can fork a configured main-session group back into a per-channel session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and pass only `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through a gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other process-separated send paths will not see the gateway monitor's map, so they can fall back to the transient unjoined IRC send path this PR is trying to avoid.
  - confidence: 0.9
- **[P2] Publish IRC clients only after joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the active client immediately after `connectIrcClient()` resolves, but readiness is based on login before confirmed channel membership. Startup sends can therefore use a socket that has not joined the target channel yet.
  - confidence: 0.82
- **[P3] Fall back when active IRC sends fail:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
