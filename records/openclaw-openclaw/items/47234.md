## Summary

Keep open: the feature is still useful and current main does not implement group-to-main routing, but this PR is not merge-ready because it lacks inspectable real behavior proof, conflicts with the current nested binding-session contract, misses outbound and Telegram topic routing paths, and bundles an IRC delivery change that is still separately under review.

## What This Changes

The PR adds a top-level per-route-binding `groupScope` option for group/channel peers to use the agent main session and also adds an IRC monitor active-client registry for proactive sends.

## Best Possible Solution

Define an additive binding-session target under the existing `bindings[].session` contract, preserve default group/channel isolation, thread the chosen target through inbound, outbound, Telegram topic routing, docs, tests, and security semantics, and keep the IRC proactive-send fix separately reviewed or redesigned.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source, docs, tests, and related issue review give high confidence that current main still isolates non-direct peers and exposes no group-to-main option.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch should align with the nested binding session contract, cover outbound and Telegram topic routing, split or harden IRC delivery, document security semantics, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Carry group scope through outbound session routing:** `src/routing/resolve-route.ts:676`
  - body: This only threads the new scope through inbound route resolution. Sends that omit an explicit `sessionKey` still derive their base session through `buildOutboundBaseSessionKey()`, which reads only global `dmScope`, so a configured main-scoped group can fork history into the old per-channel session.
  - confidence: 0.91
- **[P2] Put group scope under the binding session contract:** `src/config/types.agents.ts:45`
  - body: Current main already exposes route-level session policy as `bindings[].session.dmScope` with a strict `BindingSessionSchema`. Adding `bindings[].groupScope` as a top-level sibling creates a competing public config shape for the same concept and makes future session-target policy harder to evolve consistently.
  - confidence: 0.88
- **[P2] Preserve group scope in Telegram topic overrides:** `src/routing/session-key.ts:138`
  - body: Telegram topic-agent overrides rebuild `sessionKey` after `resolveAgentRoute()` and pass only `dmScope` and `identityLinks`. A topic that matches a group-to-main binding will be rewritten back to a per-topic group session instead of the configured main session.
  - confidence: 0.86
- **[P2] Split or harden the IRC active-client send path:** `extensions/irc/src/send.ts:72-74`
  - body: The active-client lookup is a module-local map and does not help CLI or cross-process proactive sends; it also sends through the active client with no fallback if the socket drops between `isReady()` and `sendPrivmsg()`. Keep this out of the routing PR or cover the delivery semantics in the separate IRC PR.
  - confidence: 0.89
- **[P2] Publish IRC clients only after channel membership is confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` resolves after sending JOIN commands, not after receiving server confirmation that the client is in those channels. Registering the monitor client immediately can still allow a proactive channel send before membership is accepted.
  - confidence: 0.83
