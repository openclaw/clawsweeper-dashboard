## Summary

Keep open. The requested group/channel-to-main routing feature is still absent on current main and has clear user demand, but this PR is not mergeable as-is because it lacks inspectable real behavior proof, leaves routing paths incomplete, changes a privacy-sensitive session boundary without settled docs/API semantics, and bundles separate IRC delivery work.

## What This Changes

This PR adds a per-binding `groupScope` route option to collapse matching group/channel sessions into the agent main session and bundles an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract that preserves group/channel isolation by default, wire it through inbound, outbound, Telegram topic, queue/runtime-policy paths, and handle IRC proactive delivery separately through a gateway-owned design with focused tests and real proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection gives high confidence that current main still has no group-to-main binding option and keeps non-direct peers isolated.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch is not the best maintainable solution until the config contract, privacy docs, outbound routing, Telegram topic handling, and separate IRC delivery design are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new option is only threaded through inbound route resolution. Outbound helpers that infer a target still build fallback session keys from global `dmScope`, so sends without an explicit `sessionKey` can fork the same merged group/channel conversation into a per-channel session instead of the configured main session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and only pass `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through a gateway-owned path:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other processes outside the gateway monitor will see an empty registry and fall back to the transient unjoined IRC send path that this PR is trying to avoid.
  - confidence: 0.86
- **[P2] Wait for IRC joins before publishing monitor clients:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the client immediately after `connectIrcClient()` resolves, but readiness is based on connection registration, not confirmed channel membership. Startup proactive sends can therefore use a socket that has not joined the target channel yet.
  - confidence: 0.78
- **[P3] Fall back when active IRC sends throw:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
