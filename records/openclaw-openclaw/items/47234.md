## Summary

Keep open: the feature request is valid and current main still lacks group/channel-to-main routing, but this PR is not mergeable yet because the patch misses outbound and Telegram routing paths, bundles an incomplete IRC delivery design, changes a privacy-sensitive session boundary, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds a per-binding `groupScope` route option for group/channel peers and bundles an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract that preserves default group/channel isolation, carries selected main/shared routing through inbound, outbound, Telegram topics, queue/runtime policy, docs, schema/help metadata, and tests, and handle IRC proactive delivery through a gateway-owned joined connection or a separate focused fix.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection confirms current main and v2026.5.12 still isolate non-direct peers and expose only DM-specific binding session overrides.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but the current patch is not the best mergeable solution until outbound routing, Telegram topic routing, IRC delivery design, privacy/runtime semantics, generated config metadata, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Carry groupScope through outbound session routing:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound `resolveAgentRoute()`. Outbound helpers that infer a target still build group/channel session keys from global `dmScope` only, so sends without an explicit `sessionKey` can fork a configured main-session group back into a per-channel session.
  - confidence: 0.89
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and pass only `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through a gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other process-separated send paths will not see the gateway monitor's map, so they can fall back to the transient unjoined IRC send path this PR is trying to avoid.
  - confidence: 0.91
- **[P2] Publish IRC clients only after joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the active client immediately after `connectIrcClient()` resolves, but readiness is based on login before confirmed channel membership. Startup sends can therefore use a socket that has not joined the target channel yet.
  - confidence: 0.84
- **[P2] Align groupScope with the binding session contract:** `src/config/zod-schema.agents.ts:40`
  - body: Current main's route-binding session controls live under `bindings[].session`, but this PR adds a top-level `groupScope` and does not update generated schema/help metadata. That creates two competing locations for session-routing options and leaves the public config contract incomplete.
  - confidence: 0.78
- **[P3] Fall back when active IRC sends fail:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
