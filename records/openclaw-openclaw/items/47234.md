## Summary

Keep open. Current main still lacks the requested group/channel-to-main session feature, but this patch is incomplete because it misses outbound and Telegram-derived session paths, bundles unresolved IRC active-client work, and needs privacy documentation plus observable real behavior proof.

## What This Changes

The branch adds a top-level `bindings[].groupScope` route-binding option for group/channel-to-main session routing and adds an IRC active-client registry used by outbound sends.

## Best Possible Solution

Ship a documented, additive binding session-scope or session-target contract aligned with `bindings[].session`, preserve default isolation, propagate it through inbound, outbound, Telegram topic overrides, docs, generated config surfaces, and tests, and split or rebase the IRC fix.

## Reproduction Assessment

Yes. Source inspection on current main shows strict schemas expose only `dmScope`, non-direct peers still derive isolated keys, and the PR can be checked against outbound, Telegram, and IRC paths without running a live channel setup.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch is not the best complete solution until the API shape, privacy docs, outbound/Telegram propagation, and separate IRC delivery design are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The PR only threads `groupScope` through inbound route resolution. Outbound helpers that infer a target still build the base session from global `dmScope`, so sends without an explicit `sessionKey` can mirror a merged group/channel back into a per-channel session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and only pass `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session.
  - confidence: 0.84
- **[P2] Route IRC active sends through the gateway connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends run outside the gateway monitor process, so the lookup is empty and the code falls back to the transient unjoined IRC send path that this PR is trying to avoid.
  - confidence: 0.87
- **[P2] Wait for IRC joins before publishing monitor clients:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` resolves readiness on numeric `001` and sends JOINs afterward. Registering the client immediately can route startup proactive sends through a socket that has not confirmed membership in the target channel.
  - confidence: 0.82
- **[P3] Fall back when active IRC sends throw:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
