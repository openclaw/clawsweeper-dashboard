## Summary

Keep open. Current main still lacks any group-to-main route-binding option, and this PR remains a plausible feature direction with user support, but the branch is not mergeable as-is because it misses related routing paths, lacks inspectable real behavior proof, and changes a privacy-sensitive session boundary without settled docs or policy semantics.

## What This Changes

The PR adds an optional per-route-binding `groupScope` setting for group/channel peers to route to the agent main session and also bundles an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract, likely aligned with the existing binding `session` block, that preserves default group isolation, supports selected main or shared-session routing, and threads through inbound, outbound, Telegram topic, queue/runtime-policy, docs, and tests while handling IRC proactive delivery separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection on current main gives high confidence that group/channel peers still isolate and no group-to-main config surface exists.

## Solution Assessment

No. The per-binding opt-in direction is plausible and has supporting user comments, but this patch is not the best mergeable solution until outbound routing, Telegram topic routing, IRC delivery design, privacy docs, and real proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound `resolveAgentRoute()`. Outbound helpers that infer a target still build group/channel session keys from global `dmScope` only, so sends without an explicit `sessionKey` can fork a configured main-session group back into a per-channel session.
  - confidence: 0.88
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild `route.sessionKey` with `buildAgentSessionKey()` and pass only `dmScope`. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Route IRC active sends through a gateway-owned path:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other processes outside the gateway monitor will see an empty registry and fall back to the transient unjoined IRC send path this PR is trying to avoid.
  - confidence: 0.8
- **[P3] Fall back when active IRC sends throw:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
