## Summary

Keep this PR open, but do not merge it as-is: the requested per-binding group-to-main routing is still absent from current main and has real user support, while this branch remains incomplete across outbound routing, Telegram topic routing, IRC delivery design, privacy documentation, and contributor proof.

## What This Changes

The PR adds a per-binding groupScope option for selected group/channel peers to route into an agent main session and bundles an IRC active-client registry for proactive sends.

## Best Possible Solution

Define a documented additive session-target contract that preserves default group/channel isolation, supports selected main or shared-session routing, and wires it through inbound, outbound, Telegram topic, queue/runtime-policy, docs, and tests while handling IRC proactive delivery separately.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source inspection confirms current main still has no group-to-main binding option and keeps non-direct peers isolated.

## Solution Assessment

No. The per-binding opt-in direction is plausible and has user support, but this patch is not the best mergeable solution until outbound routing, Telegram topic routing, IRC delivery design, privacy docs, API placement, and real proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Propagate groupScope into outbound session routing:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound route resolution. Outbound fallback routing still derives group/channel session keys from global dmScope only, so sends without an explicit sessionKey can fork a configured main-session group back into a per-channel session.
  - confidence: 0.88
- **[P2] Preserve groupScope for Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild route.sessionKey with buildAgentSessionKey() and pass only dmScope. A topic matching a group-to-main binding would be rewritten back to a per-topic group session, so that config combination would not work.
  - confidence: 0.84
- **[P2] Keep IRC channel sends on a gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The active-client registry is module-local. CLI proactive sends and other processes outside the gateway monitor will see an empty registry and fall back to the transient unjoined IRC send path this PR is trying to avoid.
  - confidence: 0.86
- **[P2] Publish IRC clients only after joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: The monitor registers the client immediately after connectIrcClient() resolves, but readiness is based on registration and sent JOIN commands rather than confirmed channel membership. Startup sends can use a socket that has not joined the target channel yet.
  - confidence: 0.78
- **[P3] Fall back when active IRC writes fail:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between isReady() and sendPrivmsg, the transient fallback is skipped.
  - confidence: 0.72
