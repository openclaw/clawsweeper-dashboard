## Summary

Keep open: the feature is still useful and not implemented on main, but this PR is not merge-ready because it adds an unsettled session/config contract, misses outbound and Telegram topic routing paths, bundles a separate IRC delivery change, and lacks inspectable real behavior proof.

## What This Changes

This PR adds a per-route-binding `groupScope` config for routing matched group/channel peers to the agent main session and also reuses IRC monitor clients for proactive sends.

## Best Possible Solution

Define a documented additive binding-session target under the existing `bindings[].session` contract, preserve default group/channel isolation, thread the chosen target through inbound, outbound, Telegram topic routing, docs, tests, and security semantics, and keep the IRC proactive-send fix separately reviewed or redesigned.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Source and release inspection give high confidence that current main and v2026.5.18 still isolate non-direct peers and expose no group-to-main option.

## Solution Assessment

No. The per-binding opt-in direction is plausible, but this patch should not merge until it aligns with the nested binding session contract, covers outbound and Telegram topic routing, separates or fixes IRC delivery, documents security semantics, and includes real behavior proof.

Full review comments:

- **[P1] Carry groupScope through outbound session routing:** `src/routing/resolve-route.ts:673`
  - body: The new option only flows through inbound route resolution. Outbound helpers still derive group/channel targets with global `dmScope` only, so sends without an explicit `sessionKey` can fork a configured main-session group back into a per-channel session.
  - confidence: 0.9
- **[P2] Nest groupScope under the binding session contract:** `src/config/types.agents.ts:43`
  - body: Current main already exposes per-binding session routing as `bindings[].session.dmScope`. Adding `groupScope` at the binding top level creates a competing public config shape for the same routing concern, so this should live under the chosen `session` contract before it becomes API.
  - confidence: 0.86
- **[P2] Preserve groupScope in Telegram topic overrides:** `extensions/telegram/src/conversation-route.ts:70-72`
  - body: Telegram topic-agent overrides rebuild the route session key with `buildAgentSessionKey()` and pass only `dmScope`. A topic matching a group-to-main binding would therefore be rewritten back to a per-topic group session.
  - confidence: 0.86
- **[P2] Publish IRC clients only after joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: The active-client registry is populated as soon as `connectIrcClient()` resolves, but the IRC client resolves on login before channel JOIN confirmation. Startup sends can use a socket that is ready but not yet joined to the target channel.
  - confidence: 0.82
- **[P3] Fall back when active IRC sends fail:** `extensions/irc/src/send.ts:72-74`
  - body: After finding a ready active client, the new path sends and returns without catching write errors. If the connection drops between `isReady()` and `sendPrivmsg`, the transient fallback is skipped.
  - confidence: 0.72
