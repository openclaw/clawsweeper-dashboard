## Summary

Keep this issue open. Current main still has the Feishu inbound debounce machinery that should batch same-key text events, but the reporter reproduced the Feishu DM idle-burst path after switching to `messages.inbound.debounceMs`, and the current regression coverage still exercises group-shaped Feishu events rather than the exact p2p/DM same-sender scenario.

## What This Changes

The issue reports that rapid Feishu messages configured for collect/inbound debounce batching are processed as separate replies instead of one batched agent turn.

## Best Possible Solution

Add direct Feishu p2p/DM idle-burst coverage first, then either fix the Feishu debounce key/text/effective-window path if the test fails or add targeted diagnostics/docs if the implementation already passes. Keep active-run queue serialization coordinated with #54409 instead of changing global queue semantics speculatively.
