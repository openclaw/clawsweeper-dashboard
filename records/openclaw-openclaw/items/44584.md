## Summary

Keep open. Current main has fixed the repeated Discord thread-starter injection, but the CLI still selects the agent session from raw `--to` before Discord target normalization, channel, or account delivery context can affect session selection.

## What This Changes

The issue reports Discord thread metadata being re-injected on every turn and `openclaw agent --channel discord --to ...` routing to `agent:*:main` instead of the live thread-bound session.

## Best Possible Solution

Keep the shipped Discord thread-starter guard and add a source-level Discord-aware delivery-context lookup for CLI agent runs before falling back to main-session derivation.

## Reproduction Assessment

Yes, from source inspection. With an existing `agent:main:discord:channel:<id>` session, `openclaw agent --channel discord --to <id>` still computes the session key from raw `to` before Discord normalization or account delivery context can participate.

## Solution Assessment

Yes. The maintainable fix is a narrow source change in the CLI/session resolver that normalizes the Discord delivery target and reuses matching stored session entries; compiled `dist` patching is not a durable solution.
