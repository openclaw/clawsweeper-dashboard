## Summary

Keep open. Current main has fixed the Discord thread-starter repeat for existing thread sessions, but the CLI still resolves `sessionKey` before Discord channel/account delivery context is available, and per-turn user metadata still includes volatile IDs/timestamps/history blocks.

## What This Changes

The issue asks for source fixes so Discord thread metadata is not repeatedly injected and `openclaw agent --channel discord --to ...` resumes the live thread-bound session instead of the main session.

## Best Possible Solution

Keep the shipped first-turn thread-starter guard, add delivery-context session reuse for Discord CLI agent runs, and tighten follow-up Discord thread prompt metadata without editing compiled `dist` bundles.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw agent --channel discord --to channel:<thread-id>` computes the session key before `channel` or `accountId` are considered, so it can target the main session instead of the recorded Discord thread session.

## Solution Assessment

Yes. A focused source fix in the CLI/session resolver and gateway agent path is the narrow maintainable solution; the manual `dist` bundle patch from the report is useful evidence but should not be preserved as the implementation.
