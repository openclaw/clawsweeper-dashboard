## Summary

Keep open. Current main shipped the first-turn Discord thread-starter guard, but CLI Discord target routing still resolves the session key before channel/account delivery context is available, and follow-up prompt construction can still include volatile per-turn metadata/history for thread turns.

## What This Changes

The issue requests a source-level fix for repeated volatile Discord thread metadata and `openclaw agent --channel discord --to ...` resolving to the wrong session.

## Best Possible Solution

Keep the shipped first-turn thread-starter guard, add delivery-context session reuse for Discord CLI agent runs, and tighten Discord thread follow-up metadata without editing compiled `dist` bundles.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw agent --channel discord --to channel:<thread-id>` computes `sessionKey` before channel/account context is considered, so it can target the main session instead of the recorded Discord thread session.

## Solution Assessment

Yes. A focused source fix in the CLI/session resolver and gateway agent path is the narrow maintainable solution; the reporter's manual compiled-bundle patch is useful evidence but should not be preserved as the implementation.
