## Summary

Keep open. Current main partially stabilizes inbound metadata by keeping volatile fields out of the system prompt/static hash, but it still preserves the reported Discord follow-up context injection in prompt/user context and still resolves CLI agent sessions before Discord channel/account delivery context is available.

## What This Changes

Fix Discord thread prompt metadata repetition and make `openclaw agent --channel discord --to ...` reuse the live Discord thread-bound session instead of falling back to the agent main session.

## Best Possible Solution

Implement the fix in source, not in compiled `dist` bundles: add delivery-context session reuse for CLI/gateway agent runs using normalized `{channel, to, accountId}` while preserving explicit session selectors, and limit Discord thread starter/history plus volatile per-turn metadata to the turns where that context is actually needed.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `openclaw agent --channel discord --to channel:<thread-id>` resolves its session before `channel`/`accountId` are considered, and existing tests still expect follow-up thread history/starter prompt injection.

## Solution Assessment

Yes, a focused source fix is the right direction. The manual bundle patch described in the issue is useful evidence but is not maintainable; the safer implementation belongs in the CLI/session resolver, gateway agent handling, Discord thread context path, and targeted regression tests.
