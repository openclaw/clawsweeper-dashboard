## Summary

Keep open. The Discord thread-starter repeat was fixed on current main, but the CLI still resolves the agent session before Discord channel/account delivery context or target normalization can participate, and a same-author open tracker for that routing gap is still unresolved.

## What This Changes

The issue requests source fixes for repeated volatile Discord thread context and for `openclaw agent --channel discord --to ...` to reuse the live Discord thread session instead of the generic agent main session.

## Best Possible Solution

Handle this as one clustered source fix: preserve the shipped Discord starter guard, add Discord-aware delivery-context session reuse for CLI agent runs, and close both this issue and the narrower routing tracker after the fix lands.

## Reproduction Assessment

Yes, from source inspection. With an existing Discord thread session, `openclaw agent --channel discord --to <thread-id>` still computes `sessionKey` from raw `to` before Discord channel/account context can affect session selection.

## Solution Assessment

Yes. The best fix is a narrow source change in the CLI/session resolver that normalizes Discord delivery targets and reuses matching stored session entries before falling back to main-session derivation; compiled `dist` patches are not maintainable.
