## Summary

Keep open. Current main has the related Discord first-turn `ThreadStarterBody` guard, but the CLI still computes the agent session key before using Discord channel/account delivery context, and follow-up user-context metadata/history blocks can still be prompt-visible every turn.

## What This Changes

The issue asks to stop repeated Discord thread metadata/history injection and make `openclaw agent --channel discord --to ...` reuse the live thread-bound session.

## Best Possible Solution

Keep the shipped first-turn starter guard, add delivery-context session reuse for Discord CLI agent runs, and narrowly suppress volatile Discord thread follow-up metadata without editing compiled `dist` bundles.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw agent --channel discord --to channel:<thread-id>` computes `sessionKey` before `channel` or account context can affect resolution, so it can target the agent main session instead of the existing Discord thread session.

## Solution Assessment

Yes. The maintainable fix belongs in source: teach the agent CLI/session resolver to look up by normalized delivery context and tighten Discord thread follow-up metadata, while preserving the already-shipped first-turn starter behavior.
