## Summary

Current main still does not fully resolve the report. Discord thread starter/history and volatile user-role metadata can still be injected on follow-up thread turns, and `openclaw agent --channel discord --to ...` still resolves the session before using channel/account delivery context. The metadata half overlaps open #41355, but the CLI session-routing half is not superseded by that narrower issue.

## Best Possible Solution

Keep this open. Implement a focused source fix that adds delivery-context session reuse for CLI/gateway agent runs by normalized {channel, to, accountId}, including Discord bare-ID and channel:<id> equivalence, while preserving explicit session selectors. In the same PR or a coordinated companion to #41355, stop Discord thread starter/history and volatile metadata from being re-injected on follow-up Discord thread turns, with regression tests and docs updates if CLI semantics change.
