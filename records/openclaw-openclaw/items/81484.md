## Summary

Keep open: this is a current Discord guild/server regression report with live reporter evidence, and current main still has a plausible malformed-send path where configured Discord accounts expose broad message-tool fields while runtime send rejects payloadless sends. I found no merged fix or canonical open replacement that covers the malformed payload plus repeated outbound loop behavior.

## What This Changes

The issue asks OpenClaw to make Discord guild/server prompts produce one valid visible reply and stop without malformed sends, duplicate replies, or outbound loops.

## Best Possible Solution

Tighten the Discord guild visible-reply message-tool contract so action="send" is payload-focused and replay-safe, while preserving explicit non-send Discord actions and the intentional private-by-default group/channel reply policy.

## Reproduction Assessment

No. The reporter provided live logs, screenshots, and a follow-up confirmation, and source inspection shows a matching malformed-send path, but I did not run a live Discord guild against current main to prove the duplicate loop.

## Solution Assessment

Unclear because there is no patch yet. The best direction appears to be a narrow action-specific schema/validation and replay-safety fix in the message-tool/Discord action path, not a prompt workaround or a change to group reply visibility defaults.
