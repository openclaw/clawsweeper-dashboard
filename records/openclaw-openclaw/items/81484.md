## Repair Work Prompt

Fix the Discord guild/server reply regression reported in https://github.com/openclaw/openclaw/issues/81484. Current main still lets configured Discord accounts expose broad message-tool action fields while guild visible replies require `action="send"` with real payload, and runtime send validation rejects empty send payloads. Keep the change narrow: tighten the message-tool contract or validation for Discord guild visible replies so `action="send"` is payload-focused and malformed mixed-schema sends fail earlier or are prevented, and ensure one inbound guild prompt cannot produce repeated outbound visible replies from the same run. Preserve non-send Discord actions such as read/react/poll/thread/admin through their explicit actions; do not add new config or change the default visible-reply policy. Account for existing main-only generic bot-pair loop protection, but do not treat it as sufficient for same-run duplicate sends. Add focused regression coverage around the likely files. Validate with targeted tests and, if credentials are available, a Discord guild smoke that sends one @mention prompt and verifies exactly one final visible reply. Add a changelog entry because this is user-visible.
