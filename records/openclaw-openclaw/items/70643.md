## Summary

Keep this open. Current main still lacks a shared transaction lock around config read/merge/validate/write/refresh, the direct fix PR for this issue was closed unmerged, and the newer open PRs are overlapping but not yet landed canonical fixes for this concatenated-JSON report.

## What This Changes

Ensure concurrent OpenClaw config writers cannot corrupt or invalidate openclaw.json before the gateway reloads it.

## Best Possible Solution

Land one canonical core config mutation fix that serializes each OpenClaw-owned config transaction, including include writes and runtime refresh/notification ordering, while preserving existing validation, rejection, and last-good recovery behavior.

Unclear. The reporter supplied credible live audit evidence and current main still lacks transaction-level config serialization, but I did not run a race harness and the exact concatenated-JSON interleaving is not proven from source alone.

## Solution Assessment

Yes. A transaction-level lock or fresh-state transform around the full mutation is the maintainable direction; locking only the final rename or adding a narrow agents-add retry would not cover every config writer described here.
