## Summary

Keep this issue open. Current `main` still has validation, atomic temp-file replacement, rejected/clobbered artifacts, and last-good recovery, but I did not find serialization around the full OpenClaw-owned config read/merge/validate/write/runtime-refresh transaction. The related lock PR is closed unmerged, and the related open issue covers a distinct partial-snapshot writer problem rather than superseding this concatenated-JSON race.

## What This Changes

The issue requests serialized, atomic OpenClaw-owned writes to `openclaw.json` so concurrent config mutations cannot concatenate or clobber JSON and cause gateway reload failures.

## Best Possible Solution

Land a focused core config fix that serializes OpenClaw-owned config mutations across the complete transaction, including include writes and runtime refresh/notification ordering, while preserving existing validation and recovery semantics.

## Reproduction Assessment

Unclear: the reporter supplied credible live audit evidence and quarantined concatenated snapshots, and current `main` still has the unsynchronized transaction shape. This read-only review did not run a race harness, and I did not find an existing deterministic regression test for concurrent config writers.

## Solution Assessment

Yes: transaction-level serialization is the narrow maintainable direction. Locking only the final temp-file replacement would not cover stale read/merge snapshots or the include fast path, so the repair should cover the whole mutation path.
