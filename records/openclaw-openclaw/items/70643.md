## Summary

Keep this issue open. Current main still validates and atomically replaces config files, preserves rejected/clobbered payloads, and restores last-known-good config, but I did not find serialization around the full OpenClaw-owned read/merge/validate/write/runtime-refresh transaction. The related lock PR was closed unmerged, and the newer related issue tracks a distinct partial-snapshot writer problem rather than this concatenated-JSON race.

## What This Changes

The issue requests serialized, atomic OpenClaw-owned writes to `openclaw.json` so concurrent config mutations cannot concatenate or clobber JSON and cause gateway reload failures.

## Best Possible Solution

Land a focused core config fix that serializes OpenClaw-owned config mutations across the full read/merge/validate/write/runtime-refresh transaction, likely using the existing file-lock primitive plus a per-config-path in-process queue, while preserving current validation, include, SecretRef/env, backup, rejected/clobbered, last-good, and notification semantics.

## Reproduction Assessment

Unclear: the reporter supplied credible live audit evidence and quarantined concatenated snapshots, and current main still has an unsynchronized transaction shape. This read-only review did not run a race harness, and I did not find an existing deterministic regression test for concurrent config writers.

## Solution Assessment

Yes: transaction-level serialization is the narrow maintainable direction. Locking only the final temp-file replacement would not cover stale read/merge snapshots or the include fast path, so the repair should cover the whole mutation path and its runtime refresh/notification ordering.
