## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require real credentials before Activity auth bypass:** `src/gateway/a2ui-activity.ts:151-154`
  - body: `isCanvasActivityAccessAllowed` returns true when Activity mode is enabled and no token is configured, after only checking query/cookie/referer launch markers. Because `authorizeCanvasRequest` uses this before bearer or node-capability auth, a forged marker can expose Canvas/A2UI without the normal Gateway credential.
  - confidence: 0.95
- **[P1] Move Activity config under the Canvas plugin owner:** `src/config/zod-schema.ts:654-668`
  - body: This adds `canvasHost.activity` under the root config schema, but current main has moved Canvas host configuration to `plugins.entries.canvas.config.host` and migrates legacy root `canvasHost` away. Porting this as root config would create a competing public config source and break the current upgrade model.
  - confidence: 0.93
- **[P1] Bundle or pin the Discord SDK before injecting it:** `src/canvas-host/a2ui.ts:171-176`
  - body: The injected Canvas helper dynamically imports executable JavaScript from jsdelivr with an esm.sh fallback. That makes every Activity page depend on mutable third-party runtime code in the Canvas origin; this needs a bundled/pinned dependency or an explicitly approved loading policy before merge.
  - confidence: 0.9
- **[P1] Stop sending Activity tokens in WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The browser bridge copies `activityToken` from the page URL into the A2UI WebSocket query string and uses `ws://` on HTTP pages. Tokens in URLs leak through logs/history/referers, and cleartext ws exposes them on misconfigured HTTP deployments.
  - confidence: 0.92
- **[P2] Bound Activity broadcasts and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every connected WebSocket without a byte cap or bufferedAmount/backpressure handling. A large A2UI payload or slow clients can grow memory and stall the event loop, so the hub needs size limits and drop/close behavior.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff changes auth and browser execution paths and has concrete security-boundary concerns before merge.

Concerns:

- **[high] Activity marker auth bypass:** `src/gateway/a2ui-activity.ts:151`
  - body: Activity access can succeed without a configured token when ordinary launch marker query/cookie/referer data is present, before existing Gateway bearer or node-capability checks run.
  - confidence: 0.95
- **[high] Mutable CDN code execution:** `src/canvas-host/a2ui.ts:173`
  - body: The injected helper imports the Discord SDK from jsdelivr or esm.sh at runtime inside the Canvas origin, creating a supply-chain execution dependency that is not pinned or bundled.
  - confidence: 0.9
- **[medium] Activity token in URL and cleartext WebSocket:** `src/canvas-host/a2ui/index.html:327`
  - body: The A2UI bridge places the Activity token in the WebSocket URL and downgrades to ws:// on HTTP pages, which can expose the token in logs or on the network.
  - confidence: 0.92
- **[medium] Unbounded WebSocket fanout:** `src/gateway/a2ui-activity.ts:240`
  - body: Activity broadcasts retain arbitrary payloads and send to all clients without payload limits or backpressure checks, which can be abused for memory or CPU exhaustion.
  - confidence: 0.88
