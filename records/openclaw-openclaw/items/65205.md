## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Move Activity config under the Canvas plugin:** `src/config/types.gateway.ts:41-59`
  - body: Current main migrates root `canvasHost` into `plugins.entries.canvas.config.host` and deletes the root key. Adding `canvasHost.activity` here creates a second steady-state Canvas config surface and will drift from the plugin-owned schema/docs during upgrades.
  - confidence: 0.94
- **[P1] Require a real Activity credential before bypassing auth:** `src/gateway/a2ui-activity.ts:150-154`
  - body: With Activity mode enabled and no token configured, this returns `true`, and `authorizeCanvasRequest` accepts the Canvas/A2UI request before normal Gateway bearer or capability auth. Launch-context query/cookie/referer markers are not a credential, so public deployments can expose Canvas/A2UI unauthenticated.
  - confidence: 0.96
- **[P1] Bundle the Discord SDK instead of importing CDNs:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected helper executes `@discord/embedded-app-sdk` from mutable third-party CDN URLs inside the Canvas origin. This should be a reviewed pinned dependency or local asset before exposing Activity pages.
  - confidence: 0.92
- **[P1] Keep Activity tokens out of WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The bridge copies `activityToken` from the page URL into the WebSocket query string and falls back to `ws://` for HTTP pages, exposing credentials to logs, browser history, intermediaries, or cleartext transport. Use a safer cookie/exchange/protocol path and fail closed for insecure transport.
  - confidence: 0.9
- **[P2] Bound Activity payload fanout and slow clients:** `src/gateway/a2ui-activity.ts:220-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every Activity WebSocket without byte limits or `bufferedAmount` handling. A large or slow-client workload can grow memory and stall the gateway event loop.
  - confidence: 0.91
- **[P2] Clear failed Discord SDK load promises:** `src/canvas-host/a2ui.ts:202-208`
  - body: When `OpenClaw.discord.load()` fails once, the rejected `_loadPromise` is retained and all later calls await the same rejection. Clear the promise on failure so transient CDN or ready-timeout failures can recover without a page reload.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces concrete Activity auth, token transport, remote script execution, and WebSocket fanout concerns that need security and owner review before merge.

Concerns:

- **[high] Activity mode can bypass Gateway auth without a credential:** `src/gateway/a2ui-activity.ts:151`
  - body: `isCanvasActivityAccessAllowed` returns true when Activity mode is enabled without a token, and the HTTP/WS authorization path accepts that before normal Gateway auth.
  - confidence: 0.96
- **[high] Canvas helper executes mutable third-party CDN JavaScript:** `src/canvas-host/a2ui.ts:173`
  - body: The injected Activity helper imports the Discord SDK from public CDN URLs at runtime inside the Canvas page origin, creating supply-chain script execution risk.
  - confidence: 0.92
- **[medium] Activity token is copied into URL query strings:** `src/canvas-host/a2ui/index.html:325`
  - body: The A2UI bridge reads the Activity token from the page URL and forwards it in the WebSocket URL, which can leak through logs, intermediaries, browser history, or HTTP transport.
  - confidence: 0.9
- **[medium] A2UI Activity fanout is unbounded:** `src/gateway/a2ui-activity.ts:240`
  - body: The Activity hub retains the last JSONL payload and sends each payload to all clients without size limits or backpressure checks, creating gateway memory and event-loop risk.
  - confidence: 0.91
