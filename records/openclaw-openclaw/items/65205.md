## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Rebase Activity hosting onto the Canvas plugin:** `src/config/types.gateway.ts:41-47`
  - body: Current main and v2026.6.6 route Canvas/A2UI through the Canvas plugin and keep root `canvasHost` only as legacy migration input. Adding Activity config back under root `CanvasHostConfig` reopens a retired core-owned surface and will drift from the shipped plugin route/auth owner.
  - confidence: 0.94
- **[P1] Require real Activity auth before bypassing gateway auth:** `src/gateway/a2ui-activity.ts:150-154`
  - body: When Activity mode is enabled without a configured token, any request with `instance_id`, `frame_id`, `platform`, or the matching cookie/referer marker is accepted before normal Gateway auth. Those markers are caller-controlled, so public Canvas/A2UI routes can be opened without a real secret or signed Discord assertion.
  - confidence: 0.96
- **[P1] Do not runtime-import the Discord SDK from CDNs:** `src/canvas-host/a2ui.ts:171-175`
  - body: The injected helper executes `@discord/embedded-app-sdk` from jsDelivr or esm.sh inside the Canvas origin. That adds unpinned third-party script execution to a Gateway-served page; the SDK should be bundled or otherwise pinned and reviewed through the owning plugin/package.
  - confidence: 0.93
- **[P2] Bound A2UI Activity payloads and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores the entire JSONL payload and sends it to every WebSocket client without a maximum size, queue limit, or `bufferedAmount`/backpressure check. One large push or slow client set can grow memory and stall the Gateway event loop.
  - confidence: 0.9
- **[P2] Keep Activity tokens out of WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The browser bridge copies `activityToken`/`a2uiToken` into the WebSocket query string and uses `ws://` when the page is served over HTTP. That leaks a bearer-style token through URLs and non-TLS transport; use a non-URL auth mechanism such as a secure HttpOnly cookie or an explicit exchange.
  - confidence: 0.88
- **[P2] Allow SDK load retries after transient failures:** `src/canvas-host/a2ui.ts:202-207`
  - body: If the SDK import or `ready()` call fails once, the rejected `_loadPromise` stays cached and all later `OpenClaw.discord.load()` calls await the same failure. Clear the cached promise on failure so transient CDN/network/ready timeouts can recover without a full page reload.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security-boundary concerns around unauthenticated Canvas/A2UI access, unpinned third-party script execution, and Activity token transport.

Concerns:

- **[high] Spoofable markers can bypass Gateway auth:** `src/gateway/a2ui-activity.ts:150`
  - body: Activity access returns true without a configured token after checking only caller-controlled query/cookie/referer launch markers, and Gateway auth accepts that before normal bearer/capability checks.
  - confidence: 0.96
- **[high] Injected helper executes CDN JavaScript:** `src/canvas-host/a2ui.ts:171`
  - body: The proposed helper imports the Discord embedded SDK from jsDelivr with an esm.sh fallback at runtime, adding unpinned third-party code execution to the Canvas origin.
  - confidence: 0.93
- **[medium] Activity token travels in URL query strings:** `src/canvas-host/a2ui/index.html:325`
  - body: The A2UI bridge puts the Activity token into the WebSocket query string and can use a non-TLS ws:// URL when the page is served over HTTP.
  - confidence: 0.88
- **[medium] A2UI hub has unbounded retained and broadcast payloads:** `src/gateway/a2ui-activity.ts:240`
  - body: The hub retains the last JSONL payload and sends payloads to every client without size or slow-client controls, which can be abused for memory or event-loop pressure.
  - confidence: 0.9
