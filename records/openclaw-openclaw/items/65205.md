## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require tokened Activity auth before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:150-154`
  - body: `isCanvasActivityAccessAllowed` returns true when no activity token is configured, so any request that can satisfy the caller-controlled launch-context check reaches Canvas/A2UI before normal bearer or node-capability auth. Current main explicitly denies Canvas, A2UI, and Canvas WS without bearer/capability; make Activity access fail closed without a token or use a verifiable Discord assertion before bypassing that boundary.
  - confidence: 0.96
- **[P1] Move Activity config to the Canvas plugin owner:** `src/config/zod-schema.ts:660-665`
  - body: This adds `canvasHost.activity` under the legacy root Canvas config, but current main has moved Canvas hosting into the bundled Canvas plugin and migrates root `canvasHost` into `plugins.entries.canvas.config.host`. Adding new public docs and schema under the legacy root shape would send users and runtime code to a competing config owner after upgrade.
  - confidence: 0.94
- **[P1] Bundle or pin the Discord Activity SDK:** `src/canvas-host/a2ui.ts:172-176`
  - body: The injected helper imports executable SDK code from jsDelivr and then esm.sh at runtime inside the Canvas origin. For this security-sensitive iframe/auth surface, use a bundled/pinned dependency or an explicitly approved local loader so a mutable CDN response cannot change OpenClaw's Canvas behavior after release.
  - confidence: 0.92
- **[P1] Bound Activity A2UI broadcasts before retaining payloads:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every connected Activity WebSocket without byte limits or `bufferedAmount`/slow-client handling. A large `canvas.a2ui.pushJSONL` or `canvas.a2ui.push` result can retain memory and fan out stalled sends across the gateway process, so add payload limits and backpressure handling before merge.
  - confidence: 0.91
- **[P2] Clear failed SDK loads before retrying:** `src/canvas-host/a2ui.ts:202-208`
  - body: When `OpenClaw.discord.load()` fails once, the rejected promise remains in `state._loadPromise`, so later calls re-await the same failure and cannot recover from transient CDN or `ready()` errors without a full page reload. Clear the cached promise in the catch path or use a retryable loader state.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces concrete Activity auth, third-party script execution, token transport, and unbounded broadcast concerns that need owner/security review before merge.

Concerns:

- **[high] Activity auth can bypass Gateway auth without a token:** `src/gateway/a2ui-activity.ts:151`
  - body: Activity-mode Canvas/A2UI access can return authorized before normal bearer or node-capability auth, and the no-token path relies on spoofable launch markers from URL, cookie, or referer state.
  - confidence: 0.96
- **[high] Injected helper executes mutable CDN JavaScript:** `src/canvas-host/a2ui.ts:173`
  - body: The Canvas origin would dynamically import the Discord SDK from third-party CDNs at runtime, creating a supply-chain execution path outside normal dependency pinning and review.
  - confidence: 0.92
- **[medium] A2UI Activity fanout is unbounded:** `src/gateway/a2ui-activity.ts:240`
  - body: The hub retains the last JSONL payload and broadcasts to every WebSocket client without size or backpressure checks, creating a gateway availability risk from oversized or slow-client traffic.
  - confidence: 0.91
- **[medium] Activity token is placed in a WebSocket URL query:** `src/canvas-host/a2ui/index.html:327`
  - body: The A2UI Activity bridge copies `activityToken` into the WebSocket query string and will use `ws://` when the page is served over HTTP, increasing leakage risk through logs, intermediaries, or cleartext local/proxy traffic.
  - confidence: 0.82
