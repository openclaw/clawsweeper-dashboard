## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move Activity config under the Canvas plugin owner:** `src/config/zod-schema.ts:660-667`
  - body: Current main treats root `canvasHost` as legacy migration input and resolves active host config from the Canvas plugin. Adding `canvasHost.activity` here creates a new public config surface in the retired root location, so Activity settings should live under the Canvas plugin config or have an explicit migration decision.
  - confidence: 0.93
- **[P1] Require real Activity credentials before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:150-154`
  - body: When Activity mode is enabled without a token, this returns true after only marker-based launch-context checks, and `authorizeCanvasRequest` accepts it before bearer or node-capability auth. Public Canvas/A2UI access needs a mandatory secret or verifiable signed Discord launch assertion before this bypass is allowed.
  - confidence: 0.95
- **[P1] Bundle or pin the Discord SDK before injecting it:** `src/canvas-host/a2ui.ts:171-176`
  - body: The injected Canvas helper runtime-imports executable SDK code from mutable CDN URLs in the page origin. That adds an unpinned supply-chain execution path to hosted Activity pages; use a bundled/pinned SDK path or an owner-approved constrained loading model before merge.
  - confidence: 0.9
- **[P1] Stop carrying Activity tokens in WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The bridge copies `activityToken` from the page URL into the WebSocket query string and falls back to `ws://` on HTTP pages. That leaks tokens through URLs/logs and can send them in cleartext on misconfigured deployments; use a secure cookie or other non-URL credential path and fail closed without WSS.
  - confidence: 0.88
- **[P2] Bound Activity broadcasts and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every Activity WebSocket client without byte limits or `bufferedAmount` handling, while `node.invoke` can stringify arbitrary objects into that path. Add payload caps and slow-client behavior before exposing this bridge.
  - confidence: 0.9
- **[P2] Reset failed Discord SDK loads so helpers can recover:** `src/canvas-host/a2ui.ts:166-208`
  - body: If the SDK import or `ready()` call fails once, the rejected `_loadPromise` is kept forever. Later calls re-await the same rejected promise, so a transient CDN or readiness failure leaves `OpenClaw.discord.load()` broken until a full page reload.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces concrete Activity auth, token-transport, browser supply-chain, and WebSocket fanout risks that need owner-approved fixes before merge.

Concerns:

- **[high] Marker-only Activity auth can bypass Gateway auth:** `src/gateway/a2ui-activity.ts:150`
  - body: Activity requests can be accepted before normal Gateway auth when the token is absent and ordinary launch-context markers are present, which are not credentials.
  - confidence: 0.95
- **[high] Injected Canvas helper executes mutable CDN code:** `src/canvas-host/a2ui.ts:173`
  - body: The helper imports @discord/embedded-app-sdk from public CDN URLs at runtime in the Canvas page origin without bundling or pinning.
  - confidence: 0.9
- **[medium] Activity token is placed in WebSocket query URLs:** `src/canvas-host/a2ui/index.html:325`
  - body: The A2UI bridge forwards activityToken/a2uiToken in the WebSocket URL and allows ws:// on HTTP pages, increasing leakage and cleartext transport risk.
  - confidence: 0.88
- **[medium] A2UI Activity broadcast has no payload or backpressure bounds:** `src/gateway/a2ui-activity.ts:240`
  - body: The hub retains and sends arbitrary JSONL to every connected client without byte limits or slow-client handling, creating an avoidable denial-of-service surface.
  - confidence: 0.9
