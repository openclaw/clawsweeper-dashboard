## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move Activity config back under the Canvas plugin boundary:** `src/config/zod-schema.ts:658-665`
  - body: Current main treats root `canvasHost` as a legacy migration input and registers Canvas/A2UI through the Canvas plugin. Adding `canvasHost.activity` here creates a second root/core Canvas surface and conflicts with the plugin-owned route/config model, so the Activity config and route/auth handling need to be rebased into the Canvas plugin boundary instead.
  - confidence: 0.93
- **[P1] Require real Activity credentials before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:146-154`
  - body: With `activity.enabled` and no configured token, this path returns true after launch-context checks based on request-controlled query/cookie/referer markers, and `authorizeCanvasRequest` accepts that before bearer or node-capability auth. A public caller can spoof those markers and reach Canvas/A2UI without the normal Gateway proof, so Activity access needs fail-closed credentialed or verifiable authorization.
  - confidence: 0.94
- **[P1] Bundle or pin the Discord SDK before injecting it:** `src/canvas-host/a2ui.ts:169-173`
  - body: The injected helper imports executable code from jsDelivr and then esm.sh at runtime inside the Canvas origin. That creates a mutable third-party code execution path for a security-sensitive public Activity surface; use a pinned bundled dependency or an explicitly approved, constrained loading design before merge.
  - confidence: 0.9
- **[P1] Stop putting Activity tokens in WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-330`
  - body: The A2UI bridge appends `activityToken` to the WebSocket query string and downgrades to `ws://` on HTTP pages. That leaks the token through URLs/logs and can transmit it in cleartext on misconfigured deployments, so authenticated Activity traffic should require WSS and use a safer server-set credential channel.
  - confidence: 0.88
- **[P2] Bound Activity broadcasts and slow clients:** `src/gateway/a2ui-activity.ts:238-244`
  - body: `broadcastPush` stores the full JSONL payload and sends it to every connected client without any size limit, rate limit, or `bufferedAmount` handling. A large A2UI update or slow clients can pin memory and stall the event loop, so the hub needs payload caps and backpressure/drop behavior.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces concrete Activity auth, token transport, browser code loading, and WebSocket broadcast risks that need maintainer security review before merge.

Concerns:

- **[high] Spoofable Activity auth bypass:** `src/gateway/a2ui-activity.ts:146`
  - body: Activity access can be granted before normal Gateway auth when launch context is inferred from request-controlled markers, and no configured token is accepted as authorization.
  - confidence: 0.94
- **[high] Mutable remote SDK execution:** `src/canvas-host/a2ui.ts:169`
  - body: The Canvas helper dynamically imports the Discord SDK from CDN hosts at runtime, executing third-party code in the Canvas origin without a pinned local dependency or integrity boundary.
  - confidence: 0.9
- **[medium] Activity token in URL and possible cleartext WS:** `src/canvas-host/a2ui/index.html:325`
  - body: The A2UI bridge carries Activity tokens in the WebSocket URL and chooses `ws://` when served over HTTP, making token leakage likely in logs or misconfigured networks.
  - confidence: 0.88
- **[medium] Unbounded Activity WebSocket broadcast:** `src/gateway/a2ui-activity.ts:238`
  - body: The hub retains and broadcasts arbitrary A2UI payloads to all clients without size or backpressure limits, creating a denial-of-service risk.
  - confidence: 0.86
