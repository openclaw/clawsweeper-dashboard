## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Rebase Activity hosting onto the Canvas plugin:** `src/config/types.gateway.ts:38-46`
  - body: Current main and v2026.6.6 have moved Canvas/A2UI hosting and config to `extensions/canvas`, with root `canvasHost` kept only as a hidden legacy migration surface. Adding `CanvasHostActivityConfig` under root gateway config reopens the retired core Canvas path and will drift from the only shipped Canvas route/auth owner.
  - confidence: 0.93
- **[P1] Require real Activity auth before bypassing gateway auth:** `src/gateway/a2ui-activity.ts:150-154`
  - body: When Activity mode is enabled without a configured token, any caller that supplies `instance_id`, `frame_id`, `platform`, or the equivalent cookie/referer marker is accepted before normal gateway auth. Those markers are attacker-controlled, so public Canvas/A2UI routes can be opened without a real secret or signed Discord assertion.
  - confidence: 0.95
- **[P1] Do not runtime-import the Discord SDK from CDNs:** `src/canvas-host/a2ui.ts:169-173`
  - body: The injected Canvas helper executes `@discord/embedded-app-sdk` from jsDelivr or esm.sh inside the Canvas origin. That adds unpinned third-party script execution to a gateway-served page; bundle and pin the SDK through the owning plugin/package instead.
  - confidence: 0.92
- **[P2] Bound A2UI Activity payloads and slow clients:** `src/gateway/a2ui-activity.ts:238-240`
  - body: `broadcastPush` stores the entire JSONL string and sends it to every WebSocket client without a maximum size, queue limit, or buffered-amount/backpressure check. One large push or slow client set can grow memory and stall the gateway event loop.
  - confidence: 0.9
- **[P2] Keep Activity tokens out of WebSocket query URLs:** `src/canvas-host/a2ui/index.html:324-326`
  - body: The browser bridge copies `activityToken`/`a2uiToken` into the WebSocket URL query string and will use `ws://` when the page is served over HTTP. That leaks a bearer-style token through URLs and non-TLS transport; use an HttpOnly/Secure cookie or another non-URL auth mechanism for the socket.
  - confidence: 0.88
- **[P2] Allow Discord SDK load retries after transient failures:** `src/canvas-host/a2ui.ts:194-199`
  - body: If the SDK import or `ready()` call fails once, the rejected `_loadPromise` stays cached and every later `OpenClaw.discord.load()` awaits the same failure. Clear the cached promise on failure so a transient CDN/network/ready timeout can recover without reloading the entire page.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security-boundary concerns around unauthenticated Canvas/A2UI access, unpinned third-party script execution, and token transport.

Concerns:

- **[high] Spoofable markers can bypass Gateway auth:** `src/gateway/a2ui-activity.ts:150`
  - body: Activity access returns true without a configured token after checking only caller-controlled query/cookie/referer launch markers, and Gateway auth accepts that before normal bearer/capability checks.

  - body: The proposed helper imports the Discord embedded SDK from jsDelivr with an esm.sh fallback at runtime, adding unpinned third-party code execution to the Canvas origin.
  - confidence: 0.92
- **[medium] Activity token travels in URL query strings:** `src/canvas-host/a2ui/index.html:324`
  - body: The A2UI bridge puts the Activity token into the WebSocket query string and can use a non-TLS ws:// URL when the page is served over HTTP.
  - confidence: 0.88

  - body: The hub retains the last JSONL payload and sends payloads to every client without size or slow-client controls, which can be abused for memory or event-loop pressure.
  - confidence: 0.9
