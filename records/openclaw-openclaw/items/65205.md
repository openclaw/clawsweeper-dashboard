## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Require real activity auth before bypassing gateway auth:** `src/gateway/server/http-auth.ts:86-87`
  - body: With `canvasHost.activity.enabled=true` and no `activity.token`, this early return accepts activity Canvas requests before normal gateway bearer/capability auth. Since the downstream launch-context check accepts spoofable query/cookie/referer markers, an unauthenticated caller can add `instance_id`, `frame_id`, or `platform` and open Canvas/A2UI routes. Make the activity path fail closed without a configured secret or verifiable Discord assertion.
  - confidence: 0.95
- **[P1] Bundle or explicitly pin the Discord SDK:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected Canvas helper imports executable SDK code from jsDelivr, falling back to esm.sh, at runtime in the Canvas page origin. That makes Activity pages depend on unpinned third-party CDN code for a security-sensitive auth/helper surface; use a bundled pinned dependency or an explicitly approved loader/CSP strategy instead.
  - confidence: 0.88
- **[P2] Bound activity broadcasts before storing or sending them:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores the full JSONL payload and sends it to every connected WebSocket without size limits, rate limits, or `bufferedAmount` backpressure checks. A large A2UI push can therefore retain memory and fan out work to all clients; enforce payload limits and drop or close slow clients before retaining/sending.
  - confidence: 0.88
- **[P2] Avoid putting activity tokens in WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The A2UI bridge copies `activityToken`/`a2uiToken` from the page URL into the WebSocket query string and allows `ws://` when the page is served over HTTP. Tokens in URLs leak through logs and are cleartext on HTTP; rely on the secure server-set cookie or another non-URL auth channel and fail closed for token-authenticated activity bridge traffic without HTTPS.
  - confidence: 0.82
- **[P2] Reset failed SDK loads so the helper can recover:** `src/canvas-host/a2ui.ts:166-210`
  - body: After a failed `OpenClaw.discord.load()` call, `_loadPromise` keeps the rejected promise forever. Later calls re-await the same rejection even if the CDN/network or Discord ready path has recovered, so the user must reload the page; clear `_loadPromise` on failure before rethrowing.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces concrete authorization and supply-chain concerns in the new Discord Activity Canvas path.

Concerns:

- **[high] Spoofable activity markers bypass gateway auth:** `src/gateway/server/http-auth.ts:86`
  - body: Activity Canvas requests are allowed before bearer/capability auth, and the activity check returns true without a token after trusting query/cookie/referer launch markers that external callers can spoof.
  - confidence: 0.95
- **[high] Runtime CDN SDK execution in Canvas origin:** `src/canvas-host/a2ui.ts:173`
  - body: The helper dynamically imports `@discord/embedded-app-sdk` from third-party CDNs at runtime, executing unpinned remote code in the hosted Canvas page origin.
  - confidence: 0.88
- **[medium] Token is propagated through URL-based WebSocket auth:** `src/canvas-host/a2ui/index.html:327`
  - body: The A2UI bridge sends the activity token in the WebSocket query string and downgrades to `ws://` on HTTP, which can expose the token through logs or cleartext transport if the page is not served over HTTPS.
  - confidence: 0.82
- **[medium] Unbounded broadcast can exhaust memory or stall clients:** `src/gateway/a2ui-activity.ts:240`
  - body: The activity hub retains and broadcasts unbounded JSONL payloads to every connected client without per-client backpressure checks.
  - confidence: 0.88
