## Best Possible Solution

## Solution Assessment

No; the PR is a plausible feature path, but it is not the best current fix because it reopens retired root Canvas config, uses spoofable Activity auth, and loads an SDK from public CDNs. The safer direction is a current-main rebase that extends the Canvas/Discord plugin seams with fail-closed auth and bounded runtime behavior.

Full review comments:

- **[P1] Require non-spoofable Activity auth before bypass:** `src/gateway/a2ui-activity.ts:151-152`
  - body: When Activity mode is enabled without a token, this returns true after only checking caller-controlled launch markers such as query params, cookies, or Referer. Because authorizeCanvasRequest runs this before bearer/capability auth, an external client can append a fake Activity marker and reach Canvas/A2UI routes without normal gateway auth.
  - confidence: 0.94
- **[P1] Move Activity config under the Canvas plugin:** `src/config/zod-schema.ts:660-667`
  - body: Current main moved Canvas host config to plugins.entries.canvas.config.host and keeps root canvasHost only as legacy migration input. Adding canvasHost.activity here reopens the retired root config surface and splits Canvas ownership between core and the plugin, so upgrades and docs will drift unless Activity config is added through the Canvas/plugin-owned contract.
  - confidence: 0.9
- **[P1] Bundle or pin the Discord Activity SDK:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected Canvas helper imports executable JavaScript from jsDelivr and then esm.sh at runtime inside the Canvas origin. That creates a mutable third-party code execution path for Activity pages; ship the SDK as a normal pinned dependency or another maintainer-approved local asset path instead.
  - confidence: 0.88
- **[P2] Bound Activity A2UI broadcasts:** `src/gateway/a2ui-activity.ts:240-243`
  - body: broadcastPush keeps the full JSONL payload in memory and sends it to every connected Activity WebSocket without size limits, client limits, or backpressure handling. A large A2UI update or slow clients can turn a node push into gateway memory/CPU pressure, so this needs bounded payloads and send policy before merge.
  - confidence: 0.86
