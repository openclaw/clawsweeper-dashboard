Full review comments:

- **[P1] Move Activity config under the Canvas plugin:** `src/config/types.gateway.ts:41-59`
  - body: Current main treats root `canvasHost` as a legacy migration-only key and serves Canvas/A2UI through the bundled Canvas plugin. Adding `canvasHost.activity` creates a competing config owner and upgrade path; put Activity settings under `plugins.entries.canvas.config.host` or get an explicit migration/product decision.
  - confidence: 0.95
- **[P1] Require credentials before Activity auth bypass:** `src/gateway/a2ui-activity.ts:151-154`
  - body: With Activity enabled and no token, this returns `true`, and `authorizeCanvasRequest` accepts the Activity path before normal Gateway bearer or node-capability auth. Activity access needs a real token or verifiable Discord assertion before public Canvas/A2UI routes are exposed.
  - confidence: 0.97
- **[P1] Bundle or pin the Discord SDK:** `src/canvas-host/a2ui.ts:172-175`
  - body: The injected Canvas helper imports executable SDK code from cdn.jsdelivr.net and then esm.sh at runtime, with no pinned lockfile dependency or CSP/SRI control. That makes the Canvas origin trust mutable third-party script responses; bundle or otherwise pin and constrain the SDK before shipping this helper.
  - confidence: 0.92
- **[P1] Keep Activity tokens out of WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The bridge reads `activityToken` from the page URL, forwards it in the WebSocket query string, and falls back to `ws://` on HTTP pages. Use a server-set secure credential path or token exchange and fail closed on insecure transport.
  - confidence: 0.93
- **[P2] Bound Activity broadcasts and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every Activity WebSocket without byte limits or bufferedAmount/backpressure handling. Add payload caps and slow-client behavior before exposing this bridge publicly.
  - confidence: 0.91
- **[P2] Reset failed SDK loads so Activity pages can recover:** `src/canvas-host/a2ui.ts:202-208`
  - body: If `load()` fails once, the rejected promise stays in `state._loadPromise`, so later calls re-await the same failed promise until a full page reload. Clear the cached promise in the catch path so transient CDN/ready failures can be retried.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff changes Activity auth, browser script loading, token transport, and WebSocket fanout in ways that need security hardening before merge.

Concerns:

- **[high] Activity mode can bypass Gateway auth without credentials:** `src/gateway/a2ui-activity.ts:151`
  - body: When Activity is enabled with no configured token, the PR allows Canvas/A2UI requests before bearer or node-capability authorization, and the launch-context markers are spoofable request/query data.
  - confidence: 0.97
- **[high] Canvas helper executes remote CDN code:** `src/canvas-host/a2ui.ts:173`
  - body: The injected helper imports the Discord SDK from third-party CDNs at runtime inside the Canvas origin, broadening supply-chain script execution without a pinned package or CSP-backed policy.
  - confidence: 0.92
- **[medium] Activity token leaks through URL/WebSocket query strings:** `src/canvas-host/a2ui/index.html:325`
  - body: The browser bridge reads tokens from the page URL and forwards them in the WebSocket URL, which can leak through logs, intermediaries, browser history, and cleartext ws:// deployments.
  - confidence: 0.93
- **[medium] A2UI Activity fanout has no payload or backpressure limits:** `src/gateway/a2ui-activity.ts:240`
  - body: The hub retains arbitrary JSONL and sends it to every connected client without byte caps or slow-client handling, creating memory and event-loop availability risk.
  - confidence: 0.91
