Full review comments:

- **[P1] Port Activity config to the Canvas plugin boundary:** `src/config/types.gateway.ts:41-59`
  - body: Current main treats root `canvasHost` as legacy migration input and hides it from the public schema, while Canvas host settings now live under the bundled Canvas plugin. Adding `canvasHost.activity` creates a new root config contract that bypasses that owner boundary; this should move to the Canvas plugin config/migration path or get explicit maintainer approval for reopening root config.
  - confidence: 0.91
- **[P1] Require real Activity credentials before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:151-154`
  - body: When Activity mode is enabled without `activity.token`, any request with spoofable launch markers can pass `authorizeCanvasRequest` before bearer or capability auth. Require a token or verifiable Discord assertion before granting Activity-scoped access, and keep launch context as an additional check rather than the credential.
  - confidence: 0.93
- **[P1] Bundle the Discord SDK instead of importing mutable CDNs:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected Canvas helper executes `@discord/embedded-app-sdk` from jsDelivr and then esm.sh at runtime inside the Canvas origin. That bypasses the lockfile and normal dependency review path; ship a pinned local dependency or controlled served asset instead.
  - confidence: 0.88
- **[P1] Bound A2UI Activity broadcasts and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores an arbitrary JSONL string and sends it to every connected client without a byte limit or backpressure check, and `node.invoke` can feed it arbitrary stringified payloads. Add size/rate limits and slow-client handling before exposing this public Activity fanout path.
  - confidence: 0.86
