Full review comments:

- **[P1] Move Activity config under the Canvas plugin owner:** `src/config/zod-schema.ts:660-667`
  - body: Current main migrates legacy root `canvasHost` into `plugins.entries.canvas.config.host` and hides the root key from the public schema. Adding `canvasHost.activity` here creates a second runtime Canvas config source; put the settings under the Canvas plugin config or get an explicit migration/product decision.
  - confidence: 0.94
- **[P1] Require credentials before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:151-152`
  - body: With Activity enabled and no token, this returns `true` after launch-context checks based on query/cookie/referer data, and the HTTP auth path accepts it before normal bearer or capability auth. Public Activity access needs a real token or verifiable Discord assertion before exposing Canvas/A2UI.
  - confidence: 0.96
- **[P1] Bundle or pin the Discord SDK before injecting it:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected helper imports executable code from jsDelivr with an esm.sh fallback at runtime inside the Canvas origin. That creates a mutable third-party code execution path for a public Activity surface; use a pinned bundled dependency or an explicitly approved constrained loader.
  - confidence: 0.91
- **[P1] Stop putting Activity tokens in WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The bridge reads `activityToken` from the page URL, forwards it in the WebSocket query string, and falls back to `ws://` on HTTP pages. Use WSS plus a safer server-set credential channel or token exchange so Activity credentials do not leak through URLs, logs, or cleartext transport.
  - confidence: 0.9
- **[P2] Bound Activity broadcasts and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every connected Activity client without byte limits, rate limits, or `bufferedAmount` handling. Add payload caps and slow-client drop or terminate behavior before exposing this hub.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces concrete Activity auth, token transport, browser code loading, and WebSocket fanout risks that need security and owner review before merge.

Concerns:

- **[high] Spoofable Activity auth bypass:** `src/gateway/a2ui-activity.ts:151`
  - body: Activity access can be granted before normal Gateway auth when launch context is inferred from request-controlled markers, and no configured token is accepted as authorization.
  - confidence: 0.96
- **[high] Mutable remote SDK execution:** `src/canvas-host/a2ui.ts:173`
  - body: The Canvas helper dynamically imports the Discord SDK from CDN hosts at runtime, executing third-party code in the Canvas origin without a pinned local dependency or integrity boundary.
  - confidence: 0.91
- **[medium] Activity token in URL and possible cleartext WS:** `src/canvas-host/a2ui/index.html:325`
  - body: The A2UI bridge carries Activity tokens in the WebSocket URL and chooses ws:// when served over HTTP, making token leakage likely in logs or misconfigured networks.
  - confidence: 0.9
- **[medium] Unbounded Activity WebSocket broadcast:** `src/gateway/a2ui-activity.ts:240`
  - body: The Activity hub retains and broadcasts arbitrary A2UI payloads to all clients without size or backpressure limits, creating a denial-of-service risk.
  - confidence: 0.88
