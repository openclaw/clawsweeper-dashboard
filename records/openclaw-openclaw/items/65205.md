## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Require tokened Activity auth before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:151-154`
  - body: When Activity mode is enabled without `activity.token`, this returns `true` after only query/cookie/referer launch markers, and `authorizeCanvasRequest` then skips normal Gateway bearer/capability auth. Those markers are not a verifier, so a public Canvas/A2UI route can be opened by a spoofed request.
  - confidence: 0.94
- **[P1] Move Activity config to the Canvas plugin owner:** `src/config/zod-schema.ts:660-667`
  - body: This adds `canvasHost.activity` under the root legacy Canvas host config, but current main moved Canvas host config to `plugins.entries.canvas.config.host` and keeps root `canvasHost` only as doctor-migration input. Adding new runtime behavior here creates a second Canvas config source and bypasses the plugin owner boundary.
  - confidence: 0.91
- **[P1] Bundle or pin the Discord Activity SDK:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected helper imports executable SDK code from jsDelivr with an esm.sh fallback at runtime inside the Canvas origin. A production Canvas page should not depend on mutable third-party CDN JavaScript for this privileged helper; bundle/pin the SDK through the owning package or route it through an explicitly approved asset path.
  - confidence: 0.9
- **[P1] Bound Activity A2UI payload fanout:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores the full JSONL payload and sends it to every connected Activity WebSocket with no byte cap or `bufferedAmount` slow-consumer handling. Large node-supplied A2UI payloads can therefore be retained and fanned out until the gateway stalls or memory grows.
  - confidence: 0.88
- **[P2] Clear failed SDK loads before retrying:** `src/canvas-host/a2ui.ts:202-208`
  - body: If `load()` fails once, the rejected promise remains in `state._loadPromise`, so later calls keep re-awaiting the same failure even after a transient CDN or `ready()` issue has cleared. Clear the promise in the catch path so users can retry without a full page reload.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch changes Gateway auth and Canvas script execution in ways that need security rework before merge.

Concerns:

- **[high] Spoofable Activity markers bypass Gateway auth:** `src/gateway/a2ui-activity.ts:151`
  - body: Activity mode can authorize Canvas/A2UI requests without a configured token when query, cookie, or referer launch markers are present; those markers are not authentication.
  - confidence: 0.94
- **[high] Mutable CDN SDK runs inside Canvas origin:** `src/canvas-host/a2ui.ts:173`
  - body: The injected helper imports executable Discord SDK code from public CDNs at runtime, expanding the Canvas origin trust boundary to unpinned third-party responses.
  - confidence: 0.9
- **[medium] Unbounded A2UI Activity fanout can stall Gateway:** `src/gateway/a2ui-activity.ts:240`
  - body: The new WebSocket hub retains and broadcasts arbitrary JSONL to every client without payload or backpressure limits.
  - confidence: 0.88
- **[medium] Activity token is forwarded in WebSocket URL on HTTP:** `src/canvas-host/a2ui/index.html:327`
  - body: The client script builds `ws://.../__openclaw__/a2ui/ws?activityToken=...` when served over HTTP, which can expose the token through cleartext transport and URL logs in misconfigured deployments.
  - confidence: 0.8
