Full review comments:

- **[P1] Move Activity config under the Canvas plugin owner:** `src/config/types.gateway.ts:41-59`
  - body: This adds `canvasHost.activity` to root Gateway config, but current main treats root `canvasHost` as a legacy migration source and hides it from public schema. Adding a new subkey there creates a competing config surface and bypasses the Canvas plugin owner.
  - confidence: 0.94
- **[P1] Require credentials before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:150-154`
  - body: With Activity enabled and no token, this returns `true` after checking request-controlled query/cookie/referer launch markers, and `authorizeCanvasRequest` accepts Activity routes before bearer or capability auth. Public Canvas/A2UI access needs a real token or verifiable Discord assertion.
  - confidence: 0.96
- **[P1] Bundle or pin the Discord SDK before injecting it:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected helper imports executable code from jsDelivr with an esm.sh fallback at runtime inside the Canvas origin. That creates a mutable third-party code execution path for a public Activity surface; use a pinned bundled dependency or an explicitly approved constrained loader.
  - confidence: 0.91
- **[P1] Stop putting Activity tokens in WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The bridge reads `activityToken` from the page URL, forwards it in the WebSocket query string, and falls back to `ws://` on HTTP pages. Use a safer server-set credential or token exchange and fail closed on insecure transport.
  - confidence: 0.9
- **[P2] Bound Activity broadcasts and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every Activity WebSocket without byte limits, rate limits, or `bufferedAmount` handling. Add payload caps and slow-client drop/terminate behavior before exposing this hub.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces Activity auth, token transport, remote script execution, and WebSocket fanout concerns that need security and owner review before merge.

Concerns:

- **[high] Spoofable Activity auth bypass:** `src/gateway/a2ui-activity.ts:150`
  - body: Activity access can be granted before normal Gateway auth when launch context is inferred from request-controlled markers and no configured token is accepted as authorization.
  - confidence: 0.96
- **[high] Mutable remote SDK execution:** `src/canvas-host/a2ui.ts:173`
  - body: The Canvas helper dynamically imports the Discord SDK from CDN hosts at runtime, executing third-party code in the Canvas origin without a pinned local dependency or integrity boundary.
  - confidence: 0.91

  - body: The A2UI bridge carries Activity tokens in the WebSocket URL and chooses `ws://` when served over HTTP, making token leakage likely in logs or misconfigured networks.

  - body: The Activity hub retains and broadcasts arbitrary A2UI payloads to all clients without size or backpressure limits, creating a denial-of-service risk.
