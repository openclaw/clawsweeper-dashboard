## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Require real activity auth before bypassing gateway auth:** `src/gateway/server/http-auth.ts:86-87`
  - body: `authorizeCanvasRequest` accepts Activity Canvas/A2UI requests before bearer or capability auth, but `isCanvasActivityAccessAllowed` returns true without a configured token after only checking query/cookie/referer launch markers. An external caller can spoof those markers and reach Canvas routes when Activity mode is enabled, so this needs a real shared secret or signed Discord assertion and fail-closed behavior.
  - confidence: 0.95
- **[P1] Do not runtime-import the Discord SDK from CDNs:** `src/canvas-host/a2ui.ts:173-175`
  - body: The injected Canvas helper falls back to importing executable SDK code from jsDelivr or esm.sh at runtime. That gives third-party CDN responses script execution in the Canvas origin; bundle and pin the SDK as an OpenClaw/plugin asset instead.
  - confidence: 0.92
- **[P1] Rebase Activity hosting onto the Canvas plugin:** `src/gateway/server-runtime-state.ts:3-4`
  - body: This branch imports and starts `src/canvas-host` directly from gateway runtime, but current main moved Canvas hosting behind `extensions/canvas` plugin routes and plugin config. Merging this shape would recreate a competing core Canvas path and bypass the current Canvas config/migration owner.
  - confidence: 0.9
- **[P2] Bound A2UI activity payloads and slow clients:** `src/gateway/a2ui-activity.ts:240-243`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every connected WebSocket without size or backpressure checks, while `node.invoke` can feed this path after stringifying an arbitrary object. Add payload limits and slow-client handling before exposing this as an Activity transport.
  - confidence: 0.9
- **[P2] Avoid putting Activity tokens on insecure WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-329`
  - body: The browser bridge copies `activityToken` from the page query into the WebSocket query and uses `ws://` when the page is served over HTTP. Authenticated Activity traffic should require WSS/secure cookies or another protected token channel instead of leaking secrets through URLs and cleartext WebSockets.
  - confidence: 0.84
