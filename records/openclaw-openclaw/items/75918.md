## Summary

Keep this PR open, but it is not merge-ready: the persistent hook-session idea is useful and the live proof is strong, yet the patch still lets direct hook callers force persistent reuse through the default hook session key and also revives implicit legacy Codex auth-profile selection.

## What This Changes

The PR adds opt-in hook `sessionMode` support for persistent webhook agent sessions and also changes hook/config/docs tests plus agent, plugin-registry, and Codex app-server auth/runtime routing paths.

## Best Possible Solution

Land a narrower hook-session change that keeps default isolation, allows persistent mode only with an explicit trusted scoped session key or trusted mapping, and leaves legacy Codex auth cleanup on the canonical `openai`/doctor path or in a separately approved PR.

## Solution Assessment

No, not yet; persistent hook sessions are the right product direction, but this patch's direct-request gating and legacy auth fallback are not the safest merge shape. A narrower hook-only fix with explicit scoped-key gating, plus separate maintainer-approved auth/runtime work if needed, is the better solution.

  - body: `normalizeAgentPayload` now accepts `sessionMode: "persistent"` from any direct `/hooks/agent` body. Reject persistent direct requests without an explicit allowed scoped `sessionKey`, or only allow persistent mode from trusted mappings with configured scoped keys.

Current docs and doctor warnings say legacy Codex auth profile ids/order are doctor-only legacy state that should migrate to `openai`, while the Codex app-server only needs externally supplied `chatgptAuthTokens` once a profile is explicitly chosen. Keep explicit legacy handling if needed, but do not add this provider to implicit selection.
