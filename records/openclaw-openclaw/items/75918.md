## What This Changes

The PR adds an opt-in persistent hook session mode for webhook agent runs and also changes agent harness/runtime, Codex auth, plugin registry, config, docs, and test surfaces.

## Best Possible Solution

Land a focused current-main explicit `sessionMode` hook implementation with isolated default and key-prefix policy intact, and remove or split the legacy Codex auth-provider runtime changes unless maintainers explicitly approve that direction.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 still route hook-agent requests through isolated cron session creation with `forceNew`, while session reuse is available only when `forceNew` is false.

## Solution Assessment

Mostly yes for the hook issue: explicit opt-in `sessionMode` is safer than automatic reuse because it preserves default isolation, but this PR is not the best merge shape while it includes the legacy Codex auth-provider runtime route and lacks branch-head proof for the final gate.
