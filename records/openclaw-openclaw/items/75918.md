## Summary

Keep open. Current main still forces hook-agent dispatches into fresh isolated sessions, and the PR is a narrow implementation candidate for #11665 with prior art from the closed #27193; the only concrete review blocker I found is the missing required changelog entry.

## What This Changes

Adds `sessionMode: "isolated" | "persistent"` for direct and mapped hook-agent runs, wiring it through hook config/schema/docs, replay scope, dispatch, and cron session reuse tests.

## Best Possible Solution

Land the opt-in persistent hook session mode after adding changelog coverage and normal CI/rebase proof, while keeping isolated mode as the default and preserving session-key prefix policy.

## Reproduction Assessment

Yes. On current main, a direct or mapped hook-agent request with the same `sessionKey` still goes through an isolated cron job with `forceNew: true`, and `resolveCronSession` only reuses stored sessions when `forceNew` is false.

## Solution Assessment

Yes, with one process gap. An explicit `sessionMode` defaulting to isolated is the narrowest backward-compatible fix for trusted webhook threads, but the user-facing feature still needs a changelog entry before merge.
