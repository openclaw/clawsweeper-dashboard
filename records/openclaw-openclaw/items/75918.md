## What This Changes

The branch adds an optional `sessionMode` hook setting/payload field, carries it through direct and mapped hook dispatch into the isolated cron runner, updates generated config docs/metadata, and adds focused hook/config/cron tests.

## Best Possible Solution

Merge the explicit opt-in persistent hook session feature after the changelog and mergeability cleanup, preserving isolated as the default.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: on current main, send repeated `/hooks/agent` or mapped hook requests with the same allowed `sessionKey`; the gateway creates an isolated job and the cron runner passes `forceNew: true`, so the stored session id is not reused.

## Solution Assessment

Is this the best way to solve the issue? Yes, the explicit `sessionMode: "persistent"` opt-in is the narrowest maintainable fix because it keeps existing webhook isolation as the default and reuses sessions only when a trusted integration requests it.
