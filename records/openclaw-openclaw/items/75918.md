## Summary

Keep open. Current main still forces hook-agent dispatch through fresh isolated sessions, and this PR is the relevant implementation for #11665 and the earlier #27193, but it still lacks the required user-facing changelog entry before merge.

## What This Changes

The branch adds opt-in hook `sessionMode` support for direct `/hooks/agent` calls, hook mappings/transforms, replay scoping, cron session reuse, generated config metadata, docs, and focused tests.

## Best Possible Solution

Merge the explicit opt-in persistent hook session feature after adding an active-version changelog entry and preserving isolated mode as the default.

## Reproduction Assessment

Yes. On current main, repeated direct or mapped hook-agent requests with the same allowed `sessionKey` still enter an isolated cron job and reach `resolveCronSession` with `forceNew: true`, so code inspection gives a high-confidence reproduction path for fresh session IDs each time.

## Solution Assessment

Yes. An explicit `sessionMode: "persistent"` opt-in is the narrowest maintainable fix because it avoids automatic webhook history reuse and keeps the existing hook token plus session-key prefix gates in place.
