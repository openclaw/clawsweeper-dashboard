Keep open: current main and v2026.6.8 still lack persistent hook-session mode, and this PR is the active implementation candidate for the linked webhook session-continuity problem, but it is not merge-ready while it still expands legacy Codex auth routing and lacks current-head real proof for the final direct-request rejection gate.

## What This Changes

The branch adds hook `sessionMode` support for direct and mapped hook-agent runs, updates config/docs/tests, and also changes agent harness fallback, plugin registry reload, and Codex app-server auth behavior.

## Best Possible Solution

Land a focused current-main hook `sessionMode` implementation with isolated default and key policy intact, and remove or split the legacy Codex auth/runtime changes unless maintainers explicitly approve that auth direction with upgrade proof.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests as isolated cron jobs, passes `forceNew: true`, and only reuses stored sessions when `forceNew` is false; I did not run a live webhook in this read-only review.

## Solution Assessment

Mostly yes for the hook problem: explicit opt-in persistence is safer than automatic reuse because it preserves default isolation. The current PR is not the best merge shape while it includes legacy Codex auth routing and lacks latest-head real proof.
