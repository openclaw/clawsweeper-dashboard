## What This Changes

This PR adds a `sessionMode: "isolated" | "persistent"` hook-agent option across hook mapping config, direct `/hooks/agent` payloads, transform results, idempotency scope, docs, and the isolated cron runner, plus related harness-runtime forwarding fixes.

## Best Possible Solution

Land the explicit opt-in persistent hook mode only after live two-invocation hook proof, default-isolation proof, and maintainer acceptance of the scoped-key security model.

Yes, from source inspection: current main routes hook-agent jobs through the isolated cron runner with `forceNew: true`, so an existing fresh session for the same hook key is not reused. I did not run a live webhook reproduction in this read-only review.

## Solution Assessment

Yes, the explicit opt-in mode is a safer fix shape than automatic session reuse because default isolation stays intact. It still needs real runtime proof and maintainer security/product acceptance before merge.
