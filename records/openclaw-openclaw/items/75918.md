## Summary

## What This Changes

Adds an opt-in `sessionMode: "isolated" | "persistent"` hook-agent option across hook config/schema/docs, direct `/hooks/agent` payloads, mapping transforms, replay scope, and isolated cron session resolution.

## Best Possible Solution

Land the opt-in persistent hook session mode only after redacted runtime proof and owner acceptance, while preserving isolated mode as the default and keeping existing session-key prefix controls.

## Reproduction Assessment

Yes, source-reproducible: current main dispatches hook agents through an isolated cron job and passes `forceNew: true`, so a repeated hook `sessionKey` cannot reuse the prior transcript. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Likely yes: an explicit `sessionMode` that preserves isolated default behavior is the narrowest maintainable shape for trusted webhook threads. It still needs runtime proof and maintainer acceptance because it creates a public session/security contract.
