## Summary

Keep open. Current main still lacks persistent hook sessions, and this PR is a useful refresh of the closed #27193 work for the open #11665 request, but the head is dirty against main and still lacks the required active-version changelog entry.

## What This Changes

The PR adds an opt-in `sessionMode: "isolated" | "persistent"` hook-agent mode through hook config/schema/docs, direct payloads, mappings/transforms, replay scope, cron session resolution, and focused tests.

## Best Possible Solution

Merge the explicit persistent hook-session mode after adding an active-version changelog entry, refreshing it onto current main, and preserving isolated mode as the default.

## Reproduction Assessment

Yes. Source inspection on current main shows hook-agent dispatch reaches the isolated cron runner without a lifecycle mode and `prepareCronRunContext` passes `forceNew: true` for isolated jobs, so repeated stable `sessionKey` hook calls cannot reuse the stored fresh session.

## Solution Assessment

Yes. The explicit `sessionMode: "persistent"` opt-in is the narrowest maintainable fix because it keeps default isolation and leaves the existing request `sessionKey` allow/prefix gates in place.
