## Summary

Keep open: the hardening direction is useful, but the final head still re-reads the original source tools during metadata preservation after filtering, so an unreadable tool entry or name can still abort runtime normalization before the quarantine fully protects the turn.

## What This Changes

The PR filters/quarantines unreadable or malformed agent runtime tool schemas before normalization, logs runtime diagnostics in attempt and compaction paths, adds schema projection/quarantine coverage, and includes small lint fixes.

## Best Possible Solution

Land a version that never re-reads quarantined tool entries during runtime normalization, keeps attempt/compaction diagnostics, and lets the stacked doctor follow-up own broader CLI reporting.

## Reproduction Assessment

Yes. Source inspection shows filterProviderNormalizableTools can quarantine unreadable entries, but normalizeAgentRuntimeTools then passes the original params.tools into preserveRuntimeToolMetadata, whose source iteration/name reads can still throw; no tests were run because this review is read-only.

## Solution Assessment

No. The best fix is to carry only filtered readable source tools, or a safe source metadata map, into metadata preservation and add a regression for unreadable entry/name cases before merge.
