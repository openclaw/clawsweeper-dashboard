## What This Changes

The PR adds a dispatcher-idle wait after successful block reply sends in generic and ACP delivery paths, introduces an abort-aware idle-wait helper, and adds focused regression tests.

## Best Possible Solution

Land the narrow delivery barrier if maintainers accept the ordering-versus-latency tradeoff, with live transport proof or an ACP abort signal if they want extra safety before merge.

## Reproduction Assessment

Yes. Current main source shows same-channel block replies queue through a boolean dispatcher path while tool start waits only for the block callback chain, and the PR body supplies before/after diagnostic runtime output for that path.

## Solution Assessment

Yes, mostly. Waiting for dispatcher idle after an accepted block send is the narrowest fix that preserves the current boolean dispatcher contract; adding cancellation to the ACP wait would be the safer variant if maintainers want to reduce availability risk before merge.
