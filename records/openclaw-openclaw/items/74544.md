## Summary

Keep open. Current main still lets `memory status --deep` wait directly on the live embedding probe, that probe can inherit the 600-second local batch timeout, and open PR #75106 is already linked with closing syntax for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to stop waiting silently on a long live embedding probe, emit progress, return a parseable timeout/error, and separately investigate the Windows local embedding timeout.

## Best Possible Solution

Review and land PR #75106 or an equivalent focused memory-core change that bounds only the diagnostic embedding probe, emits non-TTY progress, and distinguishes timeout from provider/index unavailability; handle any remaining Windows local-provider failure as a separate follow-up if it still reproduces.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main directly awaits the live embedding probe, and that probe can call a local embedding batch with a 600-second timeout. The Windows/model-specific embedding failure still needs reporter logs or live provider reproduction.

## Solution Assessment

Yes for the linked fix direction: a narrow diagnostic budget plus visible progress is the maintainable fix for `memory status --deep`, while preserving longer indexing timeouts for real embedding work.
