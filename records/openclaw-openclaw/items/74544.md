## Summary

Keep open. Current main still allows `openclaw memory status --deep` to wait on the live embedding probe with no diagnostic budget, and the issue is already paired with open PR #75106, which specifically targets this behavior with closing syntax.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid a silent multi-minute embedding-probe wait, emit progress or a parseable timeout, and leave the Windows local-embedding batch timeout for follow-up investigation.

## Best Possible Solution

Review and land PR #75106 or an equivalent focused memory-core change that bounds only the diagnostic embedding probe, emits non-TTY progress, and distinguishes timeout from provider/index unavailability; handle any remaining Windows local-provider failure as a separate follow-up, with #47235 left to track the broader probe/state-reporting concerns if needed.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main directly awaits the live embedding probe, and that probe can run a local embedding batch with a 600-second timeout. The Windows/model-specific local embedding failure still needs reporter logs or live provider reproduction.

## Solution Assessment

Yes for the linked fix direction: a narrow diagnostic budget plus visible progress is the maintainable fix for `memory status --deep`, while preserving longer indexing timeouts for real embedding work.
