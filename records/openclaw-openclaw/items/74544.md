## Summary

Keep open. Current main still lets `openclaw memory status --deep` await the live embedding batch probe without a status-specific timeout or non-TTY progress fallback, and the linked implementation PR is still open rather than merged.

## What This Changes

The issue reports that `openclaw memory status --deep` can sit silently on the live embedding probe until the 600-second batch timeout even though plain status reports vector and FTS ready.

## Best Possible Solution

Land a revised linked PR or replacement that uses a short cancellable status-only embedding probe, emits non-TTY progress, renders a distinct timeout state, preserves the long indexing budget, and leaves broader local GGUF isolation to its own issue.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits `probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` under the 600-second local batch timeout. The exact Windows backend timing was not live-reproduced in this review.

## Solution Assessment

No for closing this issue now. Current main still lacks the bounded, visible, parseable status-only probe, and the open linked PR is the implementation candidate rather than a merged fix.
