## Summary

Keep open. Current main still lets `memory status --deep` wait on the live embedding probe, that probe can inherit the 600-second local embedding batch timeout on a cache miss, and open PR #75106 is already the linked implementation candidate for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to emit progress or a parseable timeout instead of silently waiting through a long local embedding probe, plus follow-up investigation of the Windows local embedding timeout.

## Best Possible Solution

Review and land or close PR #75106 first; after that, track the Windows local embedding provider timeout separately only if it still reproduces.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits `probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` and inherit the 600-second local batch timeout. The exact Windows provider/model failure still needs reporter debug logs or live reproduction.

## Solution Assessment

Yes for the active fix direction: a bounded diagnostic probe with visible non-TTY progress and a distinct timeout result is the narrow maintainable fix for `memory status --deep`. The local embedding backend timeout itself should remain separate unless the same patch can prove and fix that root cause safely.
