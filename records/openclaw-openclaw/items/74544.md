## Summary

Keep open. Current main still lets the deep memory status path wait directly on the live embedding probe, and the provided GitHub context shows open PR #75106 targets this issue with closing syntax, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid silently waiting on the 600s local embedding batch timeout and to clarify the underlying Windows local-embedding timeout.

## Best Possible Solution

Land PR #75106 or an equivalent focused memory-core fix that bounds only the diagnostic embedding probe, emits visible non-TTY progress, and distinguishes a timeout result from provider or index unavailability; investigate the Windows local-provider timeout separately if it remains after the diagnostic path is bounded.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main directly awaits the live embedding probe, and that probe can call a local embedding batch with a 600-second timeout. The Windows/model-specific embedding failure still needs reporter logs or live provider reproduction.

## Solution Assessment

Yes, the best fix direction is a narrow diagnostic budget and visible progress for `memory status --deep`, without shortening the longer indexing batch timeout used for real embedding work.
