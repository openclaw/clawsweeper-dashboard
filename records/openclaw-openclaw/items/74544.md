## Summary

Keep open. Current main still lets `openclaw memory status --deep` await the live embedding probe, which can enter the 600-second local batch timeout path with no non-TTY progress fallback and no distinct timeout result; the linked implementation PR is open and unmerged.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid silently waiting on a long local embedding probe timeout and to surface a bounded, parseable diagnostic result.

## Best Possible Solution

Land a revised linked PR or replacement that uses a cancellable or event-loop-safe status-only embedding probe, emits non-TTY progress, renders a distinct timeout state, and preserves the longer indexing budget.

## Reproduction Assessment

Yes for the source-level failure path: current main awaits `probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` under the 600-second local batch timeout. The exact Windows backend timing was not live-reproduced here.

## Solution Assessment

Yes, the narrow maintainable direction is a bounded status-only probe with visible progress and a distinct timeout result. This issue should stay open until the linked PR or a replacement is revised, proven, and merged.
