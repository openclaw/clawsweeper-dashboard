## Summary

Keep open. Current main still lets `memory status --deep` await the live embedding probe directly, and that probe can inherit the 600-second local batch timeout while non-TTY callers receive no progress output. The provided context also shows open PR #75106 targets this issue with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw to avoid silent long waits in `openclaw memory status --deep` and to investigate the Windows/local embeddings timeout behind the failed `main` store probe.

## Best Possible Solution

Land PR #75106 or an equivalent focused memory-core change that bounds only the diagnostic embedding probe, emits visible non-TTY progress, and represents a probe timeout separately from provider/index unavailability; investigate the Windows local-provider timeout separately if it remains after the diagnostic path is bounded.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main directly awaits the live embedding probe, which can call a local embedding batch with a 600-second timeout and no non-TTY progress fallback. The provider-specific Windows timeout still needs reporter logs or live environment reproduction.

## Solution Assessment

Yes for the immediate CLI problem: a narrow diagnostic budget plus visible progress is the maintainable fix and should not shorten the longer timeout used for real indexing work. The underlying local embeddings timeout may need a separate provider/runtime investigation after the diagnostic command is bounded.
