## Summary

Keep open. Current `main` still lets `memory status --deep` wait on the live embedding probe without a separate diagnostic budget or non-TTY progress fallback, and open PR #75106 already targets this issue with closing syntax.

## What This Changes

The issue asks for `openclaw memory status --deep` to complete or emit visible, parseable progress/error instead of silently waiting through the local embedding batch timeout.

## Best Possible Solution

Review and land or close PR #75106 first; if it lands, track any remaining Windows/local-provider embedding failure separately or under the broader probe-semantics issue #47235.

## Reproduction Assessment

Yes for the CLI control-flow problem: current `main` directly awaits the live embedding probe, and that probe can call a local embedding batch with a 600-second timeout. The provider/model-specific Windows failure still needs reporter logs or live reproduction.

## Solution Assessment

Yes for the active fix direction: a bounded, visible diagnostic probe is the narrow maintainable fix for `memory status --deep` while preserving longer timeouts for real indexing work. The underlying embedding-provider timeout may be a separate follow-up rather than part of this status-budget fix.
