## Summary

Keep open. Current main still awaits the live memory embedding probe without a separate diagnostic budget, and the deep status progress call still has no non-TTY fallback, so the reported silent 600-second wait remains plausible. Open PR #75106 explicitly targets this issue with closing syntax and is still the right implementation candidate to review before closing this issue.

## What This Changes

Bound `openclaw memory status --deep` so its live embedding probe cannot sit silently on the full local embedding batch timeout, while preserving a path to investigate the underlying Windows/local-model embedding failure.

## Best Possible Solution

Review and land PR #75106 or a narrow replacement that adds a bounded diagnostic embedding probe, visible non-TTY progress, and a parseable timeout state for `memory status --deep`; then keep any remaining provider/model timeout investigation tracked separately, likely alongside #47235 if it proves to be the same probe-state problem.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main awaits the live embedding probe directly, and that probe can inherit the 600-second local batch timeout. The exact Windows/local-model timeout still needs live logs or a matching provider reproduction.

## Solution Assessment

Yes. A bounded diagnostic probe with progress output is the narrowest maintainable fix for the reported CLI behavior because it does not shorten real indexing work, while the underlying provider timeout can be investigated after the status command stops looking hung.
