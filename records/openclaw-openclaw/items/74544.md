## Summary

Keep open. Current main still has the deep status command awaiting the live embedding probe directly, and the provided GitHub context shows open PR #75106 targets this issue with closing syntax, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests that `openclaw memory status --deep` avoid a long silent wait on the live embedding probe and distinguish probe timeout from underlying memory index readiness.

## Best Possible Solution

Review and land PR #75106, or an equivalent narrow memory-core fix, that bounds only the diagnostic embedding probe, emits visible non-TTY progress, and surfaces a parseable timeout state while preserving the longer indexing batch timeout for real embedding work.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main directly awaits a live embedding batch probe from `memory status --deep`, and that probe can inherit the 600-second local batch timeout. The Windows/model-specific embedding failure still needs reporter logs or live provider reproduction.

## Solution Assessment

Yes, the best fix direction is a bounded diagnostic probe with visible progress and a timeout result distinct from provider/index unavailability. Any persistent embeddingsgemma local-provider timeout should be handled as a separate provider/indexing follow-up if it remains after the diagnostic path is bounded.
