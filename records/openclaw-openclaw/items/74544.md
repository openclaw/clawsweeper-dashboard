## Summary

Keep open. Current main still has the reported diagnostic failure mode, and open PR #75106 is already the linked implementation candidate with closing syntax for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to stop waiting silently on a long embedding probe timeout and to expose the underlying embedding failure clearly.

## Best Possible Solution

Review and land #75106 or an equivalent narrow fix that bounds only the status-time embedding probe, emits visible non-TTY progress, and preserves the longer indexing batch timeout.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits the embedding probe, and that probe can enter a 600-second local batch timeout path. I did not live-reproduce the reporter's Windows provider setup or dataset-specific timeout.

## Solution Assessment

Yes for the proposed direction: a bounded status-only probe with visible progress and a distinct timeout state is the narrow maintainable fix. The open linked PR should be reviewed first instead of opening a duplicate repair branch.
