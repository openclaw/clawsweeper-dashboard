## Summary

Keep open. Current main and latest v2026.5.2 still let `memory status --deep` await the live embedding batch probe directly, and open PR #75106 is already the linked, unmerged implementation candidate for the bounded diagnostic path.

## What This Changes

Request to make `openclaw memory status --deep` avoid a silent long wait on the live embedding probe and to investigate the reported Windows local-store embedding timeout.

## Best Possible Solution

Review and land a #75106-style fix that bounds only the status diagnostic probe, emits visible non-TTY progress, reports a distinct timeout state, and leaves the longer indexing batch budget intact.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits `probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` and inherit the 600-second local batch timeout. The exact Windows provider/store timeout still needs live logs or reporter reproduction.

## Solution Assessment

Yes for the fix direction: a bounded status-only diagnostic probe with visible progress and a distinct timeout result is the narrow maintainable solution. The existing linked PR should be reviewed first rather than creating a second repair branch.
