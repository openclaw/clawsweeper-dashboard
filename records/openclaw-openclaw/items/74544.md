## Summary

Keep open. Current main still has the reported diagnostic behavior: the deep status path awaits the live embedding probe without its own budget and can be silent for non-TTY callers, while open PR #75106 directly targets this issue with closing syntax and is still unmerged in the provided GitHub context.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid a long silent wait on the embedding probe and to distinguish probe timeout from actual vector/FTS readiness.

## Best Possible Solution

Review and land PR #75106 or an equivalent narrow memory-core change that bounds only the diagnostic embedding probe, emits visible non-TTY progress, and preserves the longer indexing batch timeout; keep the broader probe accuracy/flapping concern in #47235 if it remains distinct.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main directly awaits a live `embedBatchWithRetry(["ping"])` probe that can use the 600s local batch timeout, and the progress wrapper is silent for non-TTY output without a fallback. The deeper Windows/model-specific provider timeout still needs live logs or a maintainer reproduction.

## Solution Assessment

Yes, a bounded diagnostic probe with explicit timeout state and non-TTY progress is the narrowest maintainable first fix. The safer boundary is to change `memory status --deep`, not the real indexing batch timeout or provider runtime defaults.
