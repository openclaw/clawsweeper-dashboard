## Summary

Keep open. Current main still lets the deep memory status path wait on the live embedding probe without a status-specific budget, and open PR #75106 is already the linked implementation candidate for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid a silent 600-second embedding probe wait and to make the Windows/main-store embedding failure diagnosable.

## Best Possible Solution

Review and land or rework the #75106-style fix: bound the status-only embedding probe, show non-TTY progress, return a distinct timeout state, and preserve the longer indexing batch timeout.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main awaits `manager.probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` and inherit the 600-second local batch timeout. The Windows-specific provider failure still needs reporter logs or a matching live environment.

## Solution Assessment

Yes for the fix direction: a bounded status-only probe with visible non-TTY progress and a distinct timeout state is the narrow maintainable solution. Because #75106 is open and linked, this issue should stay open until that PR lands or is replaced.
