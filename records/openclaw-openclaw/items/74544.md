## Summary

Keep open. Current main still allows `memory status --deep` to wait directly on the live embedding batch probe without a diagnostic budget or non-TTY progress fallback, and the provided GitHub context shows PR #75106 is an open, unmerged implementation candidate with closing syntax for this issue.

## What This Changes

The issue requests that `openclaw memory status --deep` stop appearing hung while it waits on a 600-second local embedding batch timeout, and that the Windows local embedding timeout for the main agent be investigated.

## Best Possible Solution

Review and land PR #75106 or an equivalent focused memory-core change that bounds only the diagnostic embedding probe, emits visible non-TTY progress, and reports timeout distinctly from unavailable; keep the longer indexing batch timeout intact and investigate the Windows local-provider failure separately if it remains.

## Reproduction Assessment

Yes for the CLI control-flow bug: current main directly awaits the live embedding probe, and that probe can call a local embedding batch with a 600-second timeout. The Windows/provider-specific failure still needs reporter logs or a matching live environment to reproduce fully.

## Solution Assessment

Yes, the best fix direction is a narrow diagnostic timeout and progress fallback in the memory-core CLI rather than shortening the real indexing batch timeout. The existing open PR appears to target that boundary.
