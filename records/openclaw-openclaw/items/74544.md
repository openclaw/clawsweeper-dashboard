## What This Changes

The issue asks for `openclaw memory status --deep` to stop silently waiting on a long embeddings probe, surface progress or timeout output, and investigate the Windows local-store embedding batch timeout.

## Best Possible Solution

Review and land a narrow memory-core fix that bounds only the diagnostic embedding probe, emits visible non-TTY progress and a parseable timeout state, and leaves real indexing batch timeouts intact; track any persistent provider/model failure separately if it remains after the diagnostic path is fixed.
