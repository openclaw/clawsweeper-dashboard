## Summary

Keep open. Current main still has the silent, unbounded `memory status --deep` embedding probe path, and open PR #75106 is already the explicit implementation candidate for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid a silent multi-minute wait on the live embedding probe and to surface the underlying Windows local-embedding timeout clearly.

## Best Possible Solution

Review and land or close PR #75106 first; if the Windows local embedding batch still times out after that, track the backend/provider failure separately with debug logs.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits `probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` and inherit the 600-second local batch timeout. The exact Windows provider/model failure still needs reporter debug logs or live reproduction.

## Solution Assessment

Yes for the active fix direction: a bounded diagnostic probe with visible non-TTY progress and a distinct timeout state is the narrow maintainable fix for the status command. The local embedding backend timeout itself should stay separate unless the same patch can prove and fix it safely.
