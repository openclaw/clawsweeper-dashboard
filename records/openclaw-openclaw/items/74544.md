## Summary

Keep open. Current main still lets `memory status --deep` await the live embedding probe directly, and open PR #75106 is already the explicit implementation candidate for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid silently waiting on a 600-second embedding batch timeout and to expose bounded, parseable diagnostic output.

## Best Possible Solution

Review and land a corrected #75106-style fix that bounds the status-only diagnostic probe, emits visible non-TTY progress, reports a distinct timeout state, and leaves the longer indexing batch budget intact.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits `probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` and inherit the 600-second local batch timeout. The exact Windows provider failure still needs live logs or reporter reproduction.

## Solution Assessment

Yes for the fix direction: a bounded status-only probe with visible progress and a distinct timeout state is the narrow maintainable solution. The linked PR should be reviewed first rather than duplicating the work.
