## Summary

Keep open. Current main still lets `memory status --deep` wait directly on the live embedding probe, that probe can inherit the 600-second local batch timeout, and open PR #75106 already targets this issue with closing syntax.

## What This Changes

The issue asks for `openclaw memory status --deep` to stop waiting silently on a long embedding probe and to report bounded, parseable diagnostic status.

## Best Possible Solution

Memory-core should expose a bounded, visible deep-status embedding diagnostic that distinguishes timeout from provider unavailability, with any remaining Windows local-provider failure tracked separately if it still reproduces.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits the live embedding probe, and that probe can call a local embedding batch with a 600-second timeout. The provider/model-specific Windows root cause still needs reporter logs or live reproduction.

## Solution Assessment

Yes for the linked fix direction: adding a narrow diagnostic budget plus non-TTY progress is the maintainable fix for `memory status --deep` while preserving longer timeouts for real indexing work.
