## Summary

Keep open. Current main still allows `memory status --deep` to wait on the live embedding probe, that probe can inherit the 600-second local batch timeout, and open PR #75106 is already the linked implementation candidate for this issue.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid a silent long embedding-probe wait and to expose progress or a bounded timeout for the underlying Windows local-embedding failure.

## Best Possible Solution

Review and land or close PR #75106 first; after that, track the Windows-specific local embedding timeout separately only if it still reproduces.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits `manager.probeEmbeddingAvailability()`, which calls `embedBatchWithRetry(["ping"])` and can inherit the 600-second local batch timeout. The exact reporter machine/provider failure still needs live logs or reproduction.

## Solution Assessment

Yes for the active fix direction: a bounded diagnostic probe with visible non-TTY progress and a distinct timeout state is the narrow maintainable fix for the silent `--deep` wait. The underlying provider timeout should remain separate unless the same patch proves and fixes it safely.
