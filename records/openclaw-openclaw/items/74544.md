## Summary

Keep open. Current main still lets `memory status --deep` await the live embedding probe directly, that probe can inherit the 600-second local batch timeout, and open PR #75106 already targets this issue with closing syntax.

## What This Changes

`openclaw memory status --deep` should avoid a silent long wait on the embedding readiness probe and should distinguish probe timeouts from memory index health.

## Best Possible Solution

Review and land or close PR #75106 first; the desired end state is a bounded, visible deep-status embedding diagnostic with a distinct timeout state, plus a separate follow-up only if the Windows local embedding failure persists.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits a live embedding probe, and that probe can call the local embedding batch path with a 600-second timeout. The provider-specific Windows failure still needs reporter logs or a live environment to reproduce exactly.

## Solution Assessment

Yes for the linked fix direction: a narrow memory-core diagnostic budget plus non-TTY progress output is the maintainable way to make `memory status --deep` usable without shortening real indexing timeouts.
