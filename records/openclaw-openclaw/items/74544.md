## Summary

Keep open. Current main still directly awaits the live memory embedding probe during `memory status --deep`, that probe can inherit the 600-second local embedding batch timeout, and open PR #75106 already targets this issue with closing syntax.

## What This Changes

The issue reports that `openclaw memory status --deep` can stay silent for minutes because its live embeddings probe waits on a long local batch timeout while basic status reports the index ready.

## Best Possible Solution

Review and land or close PR #75106 first; if it lands, track any remaining Windows/local-provider embedding failure separately only if it still reproduces.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits the live embedding probe, and that probe can call a local embedding batch with a 600-second timeout. The provider/model-specific Windows failure still needs reporter logs or live reproduction.

## Solution Assessment

Yes for the active fix direction: a bounded, visible diagnostic probe is the narrow maintainable fix for `memory status --deep` while preserving longer timeouts for real indexing work. The underlying local provider timeout may be a follow-up rather than part of the status-budget fix.
