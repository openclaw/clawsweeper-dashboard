## Summary

Keep open. Current main still lets `memory status --deep` await the live embedding probe directly, and open PR #75106 is already the explicit closing implementation candidate for the bounded status-probe behavior.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid silently waiting on a 600-second embedding batch timeout and to investigate the Windows local embedding timeout for the main memory store.

## Best Possible Solution

Review and land or close PR #75106 for the bounded CLI probe/progress behavior; if the Windows local embedding batch still times out after that, track the provider/store failure separately with debug logs.

## Reproduction Assessment

Yes for the CLI control-flow problem: current main directly awaits `probeEmbeddingAvailability()`, which can call `embedBatchWithRetry(["ping"])` and inherit the 600-second local batch timeout. The exact Windows provider failure still needs live logs or reporter reproduction.

## Solution Assessment

Yes for the linked fix direction: a bounded status-only probe with visible progress and a distinct timeout state is the narrow maintainable fix. The global embedding batch timeout should stay unchanged unless the provider failure is proven separately.
