## Summary

Keep this PR open, but it is not merge-ready: the main fix direction is plausible, yet the patch leaves the daily cost chart/export scope inconsistent and still lacks real multi-agent Control UI plus gateway proof. A newer draft PR covers the cost-scoping idea, but it is draft/mock-only and not a safe superseding close target.

## What This Changes

The branch derives a single exact `agent:<id>` Usage query, sends it to `sessions.usage`, seeds Usage agent filter options from `agents.list`, reloads when the applied agent scope changes, and adds a legacy-gateway fallback for `agentId` rejection.

## Best Possible Solution

Land one coherent fix that scopes both `sessions.usage` and `usage.cost` from the same exact normalized `agent:` query, preserves legacy-gateway behavior, and includes redacted multi-agent UI/gateway proof.

## Reproduction Assessment

Yes, source-level: current main sends no `agentId` from `loadUsage`, while the gateway defaults omitted `sessions.usage.agentId` to the default agent. I did not run a live multi-agent browser reproduction in this read-only review.

## Solution Assessment

No, not yet: translating the selected `agent:` query into `sessions.usage.agentId` is the right direction, but the fix also needs cost/daily data in the same agent scope plus real UI/gateway proof.
