## Summary

Keep open: the PR addresses a valid source-proven Control UI regression, but the patch introduces an upgrade-compatibility blocker because it sends a newer `sessions.usage.agentId` parameter without the legacy-gateway fallback pattern used by the same controller. The supplied proof is useful but not yet enough for the external-PR real behavior gate because it only exercises the resolver helper, not an after-fix multi-agent UI/gateway request.

## What This Changes

The PR derives a single `agent:<id>` Usage query token, passes it as `agentId` to `sessions.usage`, seeds Usage filter options from `agents.list`, reloads on server-side agent-scope changes, and adds focused UI/controller tests.

## Best Possible Solution

Land the UI agent-scope fix after adding legacy-gateway retry coverage for unsupported `agentId` and attaching redacted after-fix proof from a real multi-agent Usage page request.

## Reproduction Assessment

Yes. The linked issue gives a concrete multi-agent flow, and current source shows the UI omits `agentId` while the gateway scopes omitted requests to the default agent; I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No as submitted. Translating the existing agent filter into `sessions.usage.agentId` is the right narrow direction, but the patch should preserve the controller's existing old-gateway compatibility pattern and prove the real multi-agent UI path.
