## Summary

Keep open: this PR is pointed at the right Usage-page regression, but it is not complete because current main's `usage.cost` handler ignores `agentId`, so the daily cost data can remain cross-agent while session rows are scoped. It also still has only mocked/unit proof, not a real multi-agent Control UI and gateway run.

## What This Changes

The branch derives a single exact `agent:<id>` Usage query token, sends it as `agentId` to usage RPCs with a legacy retry, seeds agent filter options from `agents.list`, and reloads usage when the server-side agent scope changes.

## Best Possible Solution

Land an end-to-end fix that scopes both `sessions.usage` and `usage.cost`, preserves older-gateway fallback behavior, and includes focused regression tests plus redacted live multi-agent Control UI/gateway proof.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the PR sends `agentId` to `usage.cost`, while current main's `usage.cost` handler ignores that field and calls the cache loader without agent scope. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

No. The UI-side query-to-`agentId` mapping is the right direction, but the fix is incomplete until the gateway cost path consumes the same scope and the contributor supplies real multi-agent behavior proof.
