## Summary

Keep this PR open. The reported cron fallback budget bug is real on current main, but the latest patch still has merge-blocking correctness issues and lacks contributor-owned after-fix real behavior proof.

## What This Changes

The PR adds a model-fallback `beforeAttempt` stop hook, threads a cron deadline into isolated cron execution, skips low-budget fallback candidates, adds regression coverage, and updates the changelog.

## Best Possible Solution

Land a rebased patch that keeps deferred isolated-timeout semantics, derives the fallback deadline from the same execution-start clock as the cron abort timer, preserves explicit stop reasons, fixes the test import, and includes real cron-run proof plus focused regression tests.

## Reproduction Assessment

Yes. The source path is high-confidence: current main has no pre-attempt fallback guard, and the PR body includes production logs showing a fallback attempt receiving effectively zero budget; I did not run a live cron reproduction in this read-only review.

## Solution Assessment

No. The `beforeAttempt` seam and per-attempt budget threshold are reasonable, but the latest patch is not mergeable until the deadline uses the deferred execution-start clock, the duplicate import is fixed, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Remove the duplicate isolated runner import:** `src/cron/service.issue-regressions.test.ts:22`
  - body: `createDefaultIsolatedRunner` is imported from `service.issue-regressions.test-helpers.js` and again here from `service.regression-fixtures.js`. That redeclares the same imported binding in one module, so the test file cannot parse/typecheck and the latest PR checks fail; keep only one import source.
  - confidence: 0.98
- **[P2] Anchor fallback deadlines to the deferred timeout start:** `src/cron/service/timer.ts:117`
  - body: For detached agent-turn jobs, `deadlineAtMs` is computed before `executeJobCore(...)`, but the real abort timer is intentionally started later from `onExecutionStarted`. Queue or setup time is therefore counted against the fallback guard but not against the actual cron timeout, so a fallback can be skipped while the real execution budget is still fresh.
  - confidence: 0.91
- **[P2] Throw the stop error when no candidate failed:** `src/agents/model-fallback.ts:1114-1115`
  - body: When `beforeAttempt` returns `stop` after earlier candidates were only skipped, `lastError` is still undefined, so this path falls through to the generic fallback summary and drops the hook's explicit `Skipping fallback...` error. Preserve that stop error so callers see why the chain stopped.
  - confidence: 0.85
