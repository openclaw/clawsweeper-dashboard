## Summary

Keep open. The PR is a useful bug fix, but the latest head still needs changes: the current proof predates the latest runtime force-push, the stop hook can still lose its explicit stop error when no candidate has actually failed, and the hook can consume cooldown probe throttle before an attempt is made.

## What This Changes

This PR adds an optional model-fallback `beforeAttempt` hook, threads cron deadline access into isolated cron execution, skips fallback attempts when remaining cron budget is too small, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a repaired version that preserves explicit stop errors, avoids mutating cooldown probe state until an attempt will actually run, and refreshes real behavior proof against the latest head.

## Reproduction Assessment

Yes. Source inspection confirms current main has no pre-attempt cron fallback veto, and the PR body includes production logs plus terminal output demonstrating the intended low-budget fallback skip; I did not run a live provider repro.

## Solution Assessment

No. The generic hook and deferred deadline getter are the right direction, but the current patch is not the best mergeable form until the stop-error and cooldown-probe edge cases are fixed and proof is refreshed for the latest head.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve stop errors when no candidate has failed:** `src/agents/model-fallback.ts:1120-1121`
  - body: When `beforeAttempt` stops after earlier candidates were only skipped for auth or cooldown, `lastError` is still undefined. This branch logs the hook's explicit error but then falls into the generic fallback summary, so callers can still miss the clear `Skipping fallback...` reason. Store or throw the stop error when there is no prior real failure.
  - confidence: 0.86
- **[P3] Defer cooldown probe marking until after stop checks:** `src/agents/model-fallback.ts:1092-1101`
  - body: The new stop hook runs after cooldown-probe bookkeeping. If a near-deadline cron run stops a cooldown probe here, `markProbeAttempt` has already consumed the 30s provider probe slot even though no provider call was made, so the next run can skip the real recovery probe. Run the stop check before marking or mark only after the attempt proceeds.
  - confidence: 0.78
