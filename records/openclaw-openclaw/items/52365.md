## Summary

Keep open: the PR addresses a real cron fallback timeout problem and has useful proof, but the latest head still has semantic blockers around the deadline clock origin and stop-error propagation.

## What This Changes

The PR adds an optional model-fallback pre-attempt hook, threads cron deadline data into isolated cron execution, skips low-budget fallback attempts, adds focused cron/model-fallback tests, adjusts cron test routing, and updates the changelog.

## Best Possible Solution

Land a repaired version that keeps the generic hook, derives the cron deadline from the same moment the deferred timeout is armed, preserves explicit stop errors, and retains focused regression coverage.

## Reproduction Assessment

Yes. The issue is source-reproducible: current main has no pre-attempt cron fallback veto, and the PR body includes production logs plus terminal proof for the intended budget gate.

## Solution Assessment

No. The approach is directionally good, but the current patch is not the best mergeable form until the deadline clock matches deferred execution start and stop-only paths preserve the hook's explicit error.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Anchor fallback deadlines to execution start:** `src/cron/service/timer.ts:121`
  - body: For detached `agentTurn` jobs, current cron timeout accounting starts when `onExecutionStarted` arms `startTimeout()`. This new `deadlineAtMs` is captured earlier, before setup and queue work, so slow setup can make the fallback guard skip a valid fallback even though the real abort timer still has a full execution budget. Compute or pass the deadline from the same moment the deferred timeout is armed.
  - confidence: 0.93
- **[P2] Preserve stop errors without prior failures:** `src/agents/model-fallback.ts:1120-1121`
  - body: When `beforeAttempt` stops after earlier candidates were only skipped for auth/cooldown, `lastError` is still undefined. The loop then falls into the generic fallback summary and drops the hook's explicit `Skipping fallback...` message, so the low-budget stop path can still report a misleading generic failure. Store/throw the stop error or record it in the attempts list.
  - confidence: 0.86
