## Summary

Keep open. Current main still lacks the proactive pre-attempt cron budget guard, and this PR contains useful implementation work, but the latest head is not mergeable as-is because it starts propagated deadline accounting before current main's execution-start timeout boundary and can still drop the explicit stop reason after skip-only attempts.

## What This Changes

The PR adds a generic model-fallback pre-attempt stop hook, threads a cron deadline into isolated cron execution, skips fallback candidates when remaining budget is too low, and adds model-fallback and cron regression tests.

## Best Possible Solution

Rebase the PR onto current main and adapt the deadline propagation to the existing `onExecutionStarted` timeout seam: compute the propagated deadline when the outer timeout is actually armed, preserve timed-out-run cleanup, keep the fallback threshold based on `prepared.context.timeoutMs`, and make `beforeAttempt` stops surface their explicit error when there is no prior real candidate failure.

## Reproduction Assessment

Yes. The PR body gives a concrete production log where the primary consumed nearly all cron budget and the fallback was killed at 0 ms, and current-main source inspection confirms there is no pre-attempt guard before fallback candidate execution.

## Solution Assessment

No. The proposed direction is maintainable, but the latest patch is not the best mergeable solution until it preserves current main's execution-start timeout boundary and propagates explicit stop errors in skip-only fallback chains.
