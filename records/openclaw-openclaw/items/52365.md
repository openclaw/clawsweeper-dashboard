## Best Possible Solution

Land a rebased patch that keeps deferred isolated-timeout semantics, uses an execution-start-aligned fallback deadline, preserves explicit stop errors, and includes real behavior proof plus focused tests.

## Reproduction Assessment

Yes. The PR body includes production logs, and current-main source inspection shows no pre-attempt fallback guard before cron-selected fallback candidates run.

## Solution Assessment

No. The approach is maintainable, but this PR is not the best mergeable implementation until the propagated deadline uses the same clock origin as the deferred cron timeout and stop-only failures preserve the hook's explicit error.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Derive fallback deadlines from the execution-start timeout:** `src/cron/service/timer.ts:117`
  - body: For detached agent-turn jobs, this captures `deadlineAtMs` before setup even though current main intentionally starts the actual abort timer only when `onExecutionStarted` fires.   - confidence: 0.91

  - confidence: 0.87
