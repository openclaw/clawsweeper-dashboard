## Summary

Keep open: current main has partial sessions.list hot-path mitigations, but bare RPC calls and several Control UI chat/session-selector paths can still request unbounded lists, and there is no current-main Slack-heavy replay proving the gateway starvation is gone.

## What This Changes

Bound or otherwise make sessions.list/dashboard session loading cheap enough that large Slack-originated session stores do not starve the gateway or Slack Socket Mode.

## Best Possible Solution

Keep the issue open until maintainers choose an API-compatible bounded or paginated session-list contract and verify it against a large Slack-heavy store with concurrent channel activity.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this review; I did not run a large Slack-heavy current-main store with live Socket Mode. Source inspection does show that bare sessions.list and several Control UI refresh paths can still enumerate all selected rows when no limit is sent.

## Solution Assessment

Unclear: the current lightweight async path is a good partial mitigation, but it does not settle whether bare sessions.list and chat/session-selector refreshes should be capped, paginated, cached, or isolated by default.
