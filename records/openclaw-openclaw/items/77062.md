## Summary

Keep open: current main has several `sessions.list` hot-path mitigations, including a main-only thinking-enrichment fix, but the bare RPC still has no default cap and I did not find current-main proof that a large Slack-heavy store no longer starves the gateway.

## What This Changes

The issue asks for `sessions.list` and dashboard session loading to stay bounded or cheap so large Slack-originated session stores cannot saturate the gateway and degrade Slack Socket Mode responsiveness.

## Best Possible Solution

Keep the issue open until maintainers either verify the main fixes on a large Slack-heavy store or land an API-compatible bounded, paginated, cached, or isolated session-list path that protects channel responsiveness.

## Reproduction Assessment

No for the full Slack starvation path: source inspection shows bare `sessions.list` can still enumerate every selected row, but I did not establish a current-main large Slack-heavy store replay with CPU and Socket Mode degradation. The partial hot-path mitigations are source-verifiable and covered by targeted tests.

## Solution Assessment

Unclear: async lightweight rows and bounded Control UI defaults are maintainable mitigations, but the remaining choice between default caps, pagination, caching, or process isolation is API/product-sensitive.
