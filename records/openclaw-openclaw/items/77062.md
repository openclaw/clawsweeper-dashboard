## Summary

Keep open: current main has main-only `sessions.list` hot-path mitigations, but bare `sessions.list` still has no default cap and I did not find a current-main large Slack-store replay proving the gateway starvation is gone.

## What This Changes

The issue asks for `sessions.list` and dashboard session loading to stay cheap enough that large Slack-originated session stores cannot saturate the gateway or degrade Slack responsiveness.

## Best Possible Solution

Keep the issue open until maintainers either verify the unreleased main fixes on a large Slack-heavy store or land an API-compatible bounded/paginated session-list path that protects channel responsiveness.

## Reproduction Assessment

Unclear for the full Slack starvation: source inspection confirms the bare RPC can still enumerate all selected rows, but I did not measure a large Slack-heavy store on current main. The expensive enrichment path is partly fixed in source and covered by targeted tests.

## Solution Assessment

Unclear: the current main changes are the right direction and cover several reported hot spots, but they do not settle the API/product question of whether bare `sessions.list` should be capped, paginated, cached, or isolated by default.
