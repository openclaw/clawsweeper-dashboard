## Summary

Keep open. Current main still lacks the requested WhatsApp denylist, and this PR is the active implementation, but it is not mergeable as-is because the branch includes private deployment material and the deny matcher is not forwarded through the command-gated access path. The earlier self-chat bot finding appears addressed by the branch’s dedicated `isSenderDenied` callback, so the remaining review should focus on cleanup and the command-gate propagation bug.

## What This Changes

This PR adds optional WhatsApp `denyFrom` config through account resolution, config schema/metadata, and shared DM/group access logic, while also adding two deployment-specific files from the author’s fork environment.

## Best Possible Solution

Keep the feature under maintainer review, but reduce the branch to the WhatsApp denylist contract only: remove the deployment files, wire `isSenderDenied` through every shared helper path, add unit coverage for direct, self-chat, pairing, and command-gate behavior, then update WhatsApp docs and the active changelog before merge.

## Reproduction Assessment

Yes. The functional bug is reproducible from the PR diff by exercising `resolveDmGroupAccessWithCommandGate` with `denyFrom` and `isSenderDenied`; the helper forwards the list but drops the matcher, so the deny branch cannot run in that path. The security finding is directly reproducible by inspecting the added deployment files in the PR patch.

## Solution Assessment

No. A WhatsApp denylist can be a reasonable narrow feature, but this branch is not the best form because it mixes unrelated private deployment artifacts with the config-contract change and leaves one shared access path inconsistent. The safer path is a cleaned feature-only diff with complete matcher propagation and tests.
