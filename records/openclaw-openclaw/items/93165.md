## Summary

Close as ClawHub/plugin-owned: the reported failure is source-reproducible, but it is in the deprecated external BlueBubbles package calling a removed channel-runtime alias; current OpenClaw supports iMessage through the bundled imsg path, and any maintained BlueBubbles continuation should ship outside core with accurate compatibility metadata.

## What This Changes

The issue requests a 2026.6-compatible BlueBubbles package and core startup diagnostics for stale channel plugin API compatibility after inbound BlueBubbles messages fail on OpenClaw 2026.6.6.

## Best Possible Solution

Keep BlueBubbles out of core, migrate affected users to the supported imsg-backed iMessage plugin, and publish any maintained BlueBubbles continuation through ClawHub with current compatibility metadata.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: the published 2026.5.7 BlueBubbles package calls core.channel.turn.run, while current main exposes channel.inbound.run and tests assert the old turn alias is absent. I did not run a live BlueBubbles server, but the package/runtime mismatch directly matches the reported TypeError.

## Solution Assessment

No, resurrecting or patching BlueBubbles in core is not the best repository-level solution. The narrow maintainable path is the supported imsg migration or an external BlueBubbles plugin release with accurate compatibility metadata; any broader startup diagnostic policy should be a separate maintainer-approved core discussion.
