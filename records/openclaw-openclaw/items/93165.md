## Label Justifications

## Summary

Close as ClawHub/plugin-owned: the reported crash is real from source inspection, but it comes from the deprecated external BlueBubbles package calling a removed channel runtime alias; current OpenClaw supports iMessage through `imsg`, and a compatible BlueBubbles build should live outside core.

## What This Changes

Request a compatible 2026.6.x BlueBubbles plugin build and a startup diagnostic after `@openclaw/bluebubbles@2026.5.7` breaks inbound messages on OpenClaw 2026.6.6.

## Best Possible Solution

Do not resurrect BlueBubbles in core; migrate affected users to the supported iMessage plugin or publish a compatible external BlueBubbles plugin through ClawHub with accurate compatibility metadata.

## Reproduction Assessment

Yes, from source inspection rather than a live BlueBubbles setup: the 2026.5.7 package calls `core.channel.turn.run`, while current main exposes `runtime.channel.inbound.*` and tests assert the old `turn` alias is absent. That matches the reported TypeError reading `run`.

## Solution Assessment

No, a new bundled/core BlueBubbles fix is not the best repository-level solution because current docs define iMessage via `imsg` as the supported path. The maintainable path is an external plugin update with accurate compatibility metadata, or a separate maintainer-approved core diagnostic if broader plugin metadata policy changes are desired.
