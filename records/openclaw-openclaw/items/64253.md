## Summary

Keep open. Current main still preserves the reported failure shape for completion announcements: completion-mode dispatch remains direct-first, dormant or non-wakeable requesters still use a long Gateway `agent` RPC with `expectFinal: true`, and `gateway timeout` is still treated as transient and retried. Recent wake and direct-send fallback paths reduce some delivery losses, but they do not prove the Windows alive-but-unresponsive Gateway condition is fixed.

## What This Changes

The issue reports that sustained Windows subagent workloads leave the Gateway alive but unresponsive, with completion announcement calls timing out and channels going silent until restart.

## Best Possible Solution

Keep this as the focused Windows subagent completion-timeout report. The best end state is a stress regression or live Windows proof for concurrent completion announcements against saturated Gateway `agent` handling, followed by the narrow resilience change that proof supports: a bounded direct timeout, earlier fallback, isolated announce delivery, fanout/admission control, or an unresponsive-service recovery probe.

## Reproduction Assessment

Unclear: the issue provides concrete logs and a plausible Windows Scheduled Task workload, and current tests preserve the `gateway timeout after 120000ms` retry behavior. A high-confidence reproduction still needs a current-main Windows stress run or an equivalent regression test that makes the Gateway lane saturation observable.

## Solution Assessment

No: current main has partial mitigations but is not enough to call this solved. The safest fix direction depends on proving whether the hang is caused by the direct announce timeout policy, lane saturation, provider stalls, Gateway event-loop blocking, or Windows service recovery behavior.
