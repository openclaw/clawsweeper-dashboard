## Summary

Keep open. Current main still preserves the reported failure shape: completion announces are direct-first, dormant requesters still use long `agent` RPCs with `expectFinal: true`, and `gateway timeout` remains a transient retry signal; newer wake and direct-send fallbacks reduce delivery loss but do not prove the Windows alive-but-unresponsive Gateway condition is fixed.

## What This Changes

Investigate and fix Windows Gateway unresponsiveness under sustained subagent completion-announcement load, where completion delivery times out and channels stop responding until restart.

## Best Possible Solution

Keep this as the focused Windows subagent completion-timeout report, add a stress reproduction for concurrent completion announces under saturated Gateway `agent` handling, then make the smallest proven resilience change such as bounded direct timeout, earlier fallback, announce isolation, admission control, or an unresponsive-service recovery probe.

## Reproduction Assessment

Unclear: the issue provides concrete logs and a plausible Windows Scheduled Task workload, and current tests preserve the timeout/retry shape. A high-confidence reproduction still needs a current-main Windows stress run or an executable regression that makes Gateway lane saturation observable.

## Solution Assessment

No: current main has partial wake/fallback mitigations, but dormant completion requesters can still spend long periods in direct `agent` announce retries. The safer path is to prove the root cause before choosing the fix boundary.
