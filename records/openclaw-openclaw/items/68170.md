## Summary

Keep open. Current main and the latest release still have the broad sibling-home Doctor scan and still emit discovered sibling `.openclaw` paths; related #68752 corroborates the multi-user-host noise but is closed and does not supersede this still-valid privacy/noise report.

## What This Changes

The issue asks Doctor state-integrity checks to stop enumerating sibling `/home` or `/Users` state directories by default, or to add opt-out/redaction/filtering so cross-user paths and backup directories are not reported as split-history warnings.

## Best Possible Solution

Keep this issue open and change `src/commands/doctor-state-integrity.ts` so Doctor no longer enumerates sibling home directories by default. The maintainable end state is a state-dir diagnostic scoped to the current effective home plus configured/default state paths, with no verbatim non-current-user paths in output; add regression tests for `/home/alice` versus `/home/bob` and backup-sibling names, then update the Doctor docs if the semantics change.

## Reproduction Assessment

Yes. On Linux or macOS, any readable sibling directory under `/home` or `/Users` with a `.openclaw` subdirectory is collected by `findOtherStateDirs` and printed by `noteStateIntegrity`; the current source and release tag are sufficient to reproduce the behavior without modifying this checkout.

## Solution Assessment

No current solution is in place. Among the issue's proposed directions, scoping or removing broad sibling-home enumeration by default is safer than only adding an opt-out or backup-name filter because it addresses both path disclosure and false positives.
