## Summary

This issue should stay open: current main and the latest release still enumerate sibling home directories in Doctor, and the focused PR that closes this issue is still open rather than merged.

## What This Changes

`openclaw doctor` should stop scanning sibling OS account homes for extra `.openclaw` state directories to avoid cross-user path disclosure and backup-home false positives.

## Best Possible Solution

Review and land or explicitly reject the linked scoped-scan PR, including the matching Doctor docs update so the documented check no longer promises cross-home enumeration.

## Reproduction Assessment

Yes from source: current main enumerates `/home` or `/Users` siblings and prints every discovered `.openclaw` path in the Doctor warning. I did not create real OS users in this read-only review.

## Solution Assessment

Yes, the best direction is to scope the diagnostic to the current account/default state directory rather than adding a new env/config opt-out or backup-name heuristic as the primary fix.
