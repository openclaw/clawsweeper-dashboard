## Summary

Keep open. Current main still performs the reported broad sibling-home scan and still emits discovered sibling `.openclaw` paths in Doctor output; the latest release tag also contains the behavior. The related #68752 corroborates the multi-user-host noise but is closed and does not supersede this remaining privacy and false-positive issue.

## What This Changes

Request that `openclaw doctor` stop scanning sibling `/home/*` or `/Users/*` state directories by default, or provide scoped, opt-out, or redacted behavior to avoid cross-user disclosure and backup false positives.

## Best Possible Solution

Patch Doctor state integrity so default diagnostics only inspect the current effective home and explicitly configured/current state paths, avoid verbatim non-current-user paths in output, add focused regression tests for sibling users and backup-style directories, and update the public Doctor docs and changelog for the changed warning semantics.

## Reproduction Assessment

Yes. On Linux or macOS, any readable sibling directory under `/home` or `/Users` containing `.openclaw` is deterministically collected by `findOtherStateDirs` and printed by `noteStateIntegrity`; the source path is enough to reproduce without mutating this checkout.

## Solution Assessment

No current solution is in place. A default-scoped or redacted diagnostic is safer than only adding an opt-out or backup-name filter because it addresses both cross-user path disclosure and backup false positives.
