## Summary

Keep open: current main has package-update mitigations, but shutdown and invalid-config reload still resolve replaceable lazy chunks after startup, matching the reported missing-hash failure mode.

## What This Changes

The issue reports that npm package replacement can remove hashed gateway chunks while the old gateway process still needs them for config reload and graceful shutdown.

## Best Possible Solution

Make shutdown/restart recovery and the relevant config reload failure path resident or stable across package replacement, while keeping the existing preferred `openclaw update` service-stop flow.

## Reproduction Assessment

Yes, source-reproducible with high confidence: run a gateway from an old dist, replace the package dist so old hashed chunks disappear, then trigger config reload or forced restart. Current main still resolves `server-close` and `legacy-config-issues` dynamically after startup.

## Solution Assessment

Yes, the narrow maintainable fix is to make shutdown/restart recovery resident or use stable dist boundaries for required post-upgrade cleanup. A docs-only workaround or broad package-manager redesign would leave the old process failure mode too exposed.
