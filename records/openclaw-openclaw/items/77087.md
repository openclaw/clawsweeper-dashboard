## Summary

Keep open; current main still lazy-loads shutdown and config-diagnostic chunks after gateway startup, while the update docs already recognize that replaced package trees can strand old in-memory processes.

## What This Changes

Ensure a running Gateway can restart or recover when an in-place npm upgrade removes hash-suffixed dist chunks referenced by the old process.

## Best Possible Solution

Make the gateway shutdown/restart recovery path resident across package tree swaps and treat stale chunk import failures during reload as an upgrade recovery condition instead of invalid-config flapping.

## Reproduction Assessment

Yes for a source-level reproduction path: start the gateway, replace the installed dist tree so old hashed chunks disappear, then trigger reload or restart. I did not run the live macOS npm upgrade, but current source still has the lazy imports needed for the reported ERR_MODULE_NOT_FOUND failures.

## Solution Assessment

No current solution is present on main. A narrow code fix should eagerly load or inline shutdown-critical modules and add stale-chunk recovery on reload; docs-only guidance would leave the bricked restart path intact.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77087. Running Gateway must recover from npm global in-place upgrades that remove old hash-suffixed dist chunks. Keep the fix in gateway/update lifecycle code, not a new config option. Make shutdown/restart-critical code resident before package files can be replaced: eliminate or eagerly resolve the dynamic import of `./server-close.js` from `src/gateway/server.impl.ts`, and audit shutdown prelude/global hook imports so `server.close()` cannot fail with `ERR_MODULE_NOT_FOUND` for deleted old chunks. Also handle config reload `ERR_MODULE_NOT_FOUND` from stale dist chunk imports by logging an upgrade/stale-code diagnostic and forcing a fresh process restart or other safe recovery instead of treating it as invalid config and flapping.
