## Summary

Keep open: current main still has a shutdown path that first lazy-loads `server-close` during gateway close, while the docs still list manual package-manager updates followed by `openclaw gateway restart`; the issue logs match that replaceable hashed-chunk failure mode.

## What This Changes

The issue asks OpenClaw to recover cleanly when a running gateway's packaged `dist` chunks are replaced during an in-place npm upgrade.

## Best Possible Solution

Make the minimal gateway shutdown/restart path independent of replaceable hashed chunks, or give every required post-upgrade shutdown dependency a stable resident boundary, then cover the package replacement recovery path.

## Reproduction Assessment

Yes, with source-backed confidence but not a live run: replace the package `dist` while the gateway is alive, then trigger config reload or shutdown so current code resolves a lazy chunk after the old file is gone. The issue logs supply the concrete failing module names and current main still has the lazy shutdown import.

## Solution Assessment

No. The current `openclaw update` mitigations reduce the preferred-path risk, but the narrower maintainable fix is to make shutdown/restart recovery resident or stable even after manual package replacement.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77087. The observable bug is that a running packaged gateway can be stranded after `npm install -g openclaw@<newver>` replaces hash-suffixed `dist` chunks; config reload and especially shutdown can then hit `ERR_MODULE_NOT_FOUND` for old chunks such as `legacy-config-issues-*` or `server-close-*`. Keep the fix in generic core/package-upgrade shutdown mechanics, not version-specific filenames. Make the minimal shutdown/restart path independent of deleted hashed chunks: either eagerly import/cache the shutdown close/prelude path at gateway startup, move the required close/restart skeleton behind stable dist entries, or otherwise make missing lazy shutdown chunks fall through to a fresh supervised process restart without hanging. Audit adjacent close-time dynamic imports such as `server-close`, gateway stop hooks, lifecycle/runtime restart helpers, and config reload restart scheduling. Add regression coverage for the chosen boundary: unit/source tests around `src/gateway/server.impl.ts`/`src/cli/gateway-cli/run-loop.ts`, build/import guard coverage in `scripts/check-cli-bootstrap-imports.mjs` if stable-entry guarantees are used, and an upgrade-survivor/package scenario when feasible. Update `docs/install/updating.md` only if the user-facing manual npm recovery guidance changes, and add a changelog fix entry because this is user-facing. Do not change plugin ownership boundaries or add a config option.
