## Summary

Close as implemented on main: current code routes gateway shutdown through a stable runtime alias, emits compatibility aliases for the reported `server-close-*` hashes, and the update CLI stops a managed Gateway before package replacement or blocks unsafe in-service updates.

## What This Changes

The issue asks for the Gateway update/restart path to recover after npm replaces hashed `dist` chunks while a Gateway process is still running, especially by keeping shutdown independent of deleted chunks.

## Best Possible Solution

Keep the current main implementation: stable shutdown runtime aliases, compatibility aliases for recent shutdown chunk names, and supervised package-swap coordination for managed Gateway updates.

## Reproduction Assessment

Yes, source-backed but not live: the original path is manual package-tree replacement while a pre-fix Gateway later lazy-imports a deleted `server-close` chunk. Current main's alias rewrite and compatibility tests remove that shutdown failure path.

## Solution Assessment

Yes. The current main fix is narrower than forcing every Gateway dependency eager: it stabilizes the shutdown recovery boundary and coordinates managed package replacement without changing the broader plugin/runtime loading model.
