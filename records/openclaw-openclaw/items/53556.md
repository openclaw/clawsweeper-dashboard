## Summary

Keep open. Current main still only partially covers the reported Feishu compatibility break: legacy `groupPolicy: "allowall"` and `topicSessionMode` are handled, but legacy `botId` is still not accepted, normalized, surfaced in setup/status, migrated by doctor, or documented. The linked #52885 is closed and tracks Weixin/plugin-SDK compatibility, not this Feishu config migration.

## What This Changes

The issue requests backward-compatible channel config upgrade handling, especially for Feishu `botId` to `appId`, `topicSessionMode` to `groupSessionScope`, and `groupPolicy: "allowall"` to `"open"` migrations.

## Best Possible Solution

Add a Feishu-owned compatibility path that accepts legacy top-level and per-account `botId`, normalizes it to canonical `appId`, exposes an `openclaw doctor --fix` migration for stored configs, keeps the existing `allowall` and `topicSessionMode` compatibility intact, and documents the upgrade path without adding startup-time core migrations.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: validate or inspect a `channels.feishu` config with `botId` plus `appSecret`, and current schema/runtime/setup paths still look only for `appId`; no live Feishu account is needed to confirm that compatibility gap.

## Solution Assessment

Yes. The narrowest maintainable fix is Feishu-owned schema/resolver/setup/doc/doctor compatibility for `botId`, using the existing bundled channel doctor contract pattern; a broad core startup migration would violate the repo’s owner-boundary guidance.
