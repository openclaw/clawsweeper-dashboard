## Summary

Keep this open: current main still covers `allowall` and `topicSessionMode`, but it does not accept, read, document, or migrate legacy Feishu `botId` to canonical `appId`, so the concrete compatibility break remains repo-owned.

## What This Changes

The issue requests backward-compatible channel config migration, especially Feishu legacy keys such as `botId`, `topicSessionMode`, and `allowall`.

## Best Possible Solution

Add a Feishu-owned compatibility path that accepts legacy top-level and per-account `botId`, normalizes it to `appId`, exposes an idempotent doctor migration, and documents the upgrade behavior without adding startup-time core migrations.

## Reproduction Assessment

Yes. A Feishu config with `channels.feishu.botId` and `appSecret` but no `appId` follows the reporter's legacy mapping and current source shows schema validation, credential resolution, and setup readiness all ignore `botId`.

## Solution Assessment

Yes. The narrow maintainable fix is in the Feishu plugin plus its doctor compatibility contract; a broad startup migration system would conflict with the repo's existing doctor-owned legacy config policy.
