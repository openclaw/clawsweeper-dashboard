## Summary

Keep open: current main covers the reported `allowall` and `topicSessionMode` compatibility cases, but the Feishu legacy `botId` to `appId` upgrade path still appears missing in schema, credential resolution, setup readiness, and doctor migration.

## What This Changes

The issue asks for backward-compatible channel config handling, especially Feishu legacy `botId`, `topicSessionMode`, and `groupPolicy: "allowall"` mappings after upgrades.

## Best Possible Solution

Add a Feishu-owned compatibility path that accepts legacy top-level and per-account `botId`, normalizes it to `appId` with current `appId` taking precedence, exposes an idempotent `openclaw doctor --fix` migration, and documents the upgrade behavior.

## Reproduction Assessment

Yes, source-backed. A Feishu config with `channels.feishu.botId` and `appSecret` but no `appId` follows the reporter's legacy mapping, while current schema validation, credential resolution, setup readiness, and doctor migration all ignore `botId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin and its doctor compatibility contract; a broad startup migration system would conflict with the repo policy that legacy config repair belongs in `openclaw doctor --fix`.

Add Feishu-owned support for legacy top-level and per-account `botId` as an alias for `appId`, with explicit `appId` taking precedence when both are present. Update Feishu schema/account/setup readiness behavior and focused docs so upgraded users know to run doctor.
