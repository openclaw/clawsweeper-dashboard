## Summary

Keep open: current main already covers the reported `allowall` and legacy `topicSessionMode` cases, but Feishu still only accepts and resolves `appId`; legacy `botId` is not in the strict schema, credential resolver, setup status, docs, or doctor migration hook.

## What This Changes

The issue requests backward-compatible channel configuration handling for legacy Feishu fields such as `botId`, `topicSessionMode`, and `allowall` during upgrades.

## Best Possible Solution

Add Feishu-owned compatibility for legacy `botId` by normalizing it to `appId` in schema/runtime/setup status and through an idempotent `openclaw doctor --fix` migration, while preserving explicit `appId` precedence.

## Reproduction Assessment

Yes, source-backed. A Feishu config with `channels.feishu.botId` and `appSecret` but no `appId` follows the reporter's legacy mapping, while current strict schema, credential resolution, and setup readiness only read `appId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin and its doctor compatibility hook; a broad startup-time config migration/versioning system is not the best fit for the repo policy.
