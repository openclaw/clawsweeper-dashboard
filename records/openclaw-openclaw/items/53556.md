## Summary

Keep open: current main still covers the reported `allowall` and `topicSessionMode` compatibility cases, but legacy Feishu `botId` remains unsupported by schema, credential resolution, setup readiness, and doctor migration paths.

## What This Changes

Requests backward-compatible channel config handling for legacy Feishu fields such as `botId`, `topicSessionMode`, and `groupPolicy: "allowall"` after upgrades.

## Best Possible Solution

Add a Feishu-owned compatibility path that accepts legacy top-level and per-account `botId`, normalizes it to `appId`, exposes an idempotent `openclaw doctor --fix` migration, and documents the upgrade behavior without adding startup-time core migrations.

## Reproduction Assessment

Yes, source-level. A Feishu config with `channels.feishu.botId` and `appSecret` but no `appId` follows the reporter's mapping, and current source shows schema, credential resolution, setup readiness, and doctor migration paths do not consume `botId`.

## Solution Assessment

Yes for the narrow compatibility part. The maintainable path is a Feishu plugin plus doctor migration repair; the requested broad startup migration/versioning system is not the best fit for OpenClaw's config ownership policy.
