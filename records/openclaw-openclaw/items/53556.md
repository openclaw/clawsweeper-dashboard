## Summary

Keep open: current main still handles the reported `allowall` and `topicSessionMode` compatibility cases, but legacy Feishu `botId` is still neither accepted by the schema nor read by credential/setup paths, so the concrete upgrade break remains actionable.

## What This Changes

The issue requests backward-compatible handling or migration for legacy channel configuration fields, especially Feishu `botId`, `topicSessionMode`, and `groupPolicy: "allowall"`.

## Best Possible Solution

Add a Feishu-owned compatibility path that accepts legacy top-level and per-account `botId`, normalizes it to `appId`, exposes an idempotent doctor migration, and documents the upgrade behavior without adding startup-time core migrations.

## Reproduction Assessment

Yes. A Feishu config with `channels.feishu.botId` and `appSecret` but no `appId` follows the reporter's legacy mapping, and current source shows schema validation, credential resolution, and setup readiness all ignore `botId`.

## Solution Assessment

Yes. The narrow maintainable fix is in the Feishu plugin plus its doctor compatibility contract; a broad startup migration system would conflict with the repo's doctor-owned legacy config policy.
