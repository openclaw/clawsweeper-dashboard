## Summary

Current main still does not fully handle the Feishu legacy configuration break described here. `groupPolicy: "allowall"` and legacy `topicSessionMode` are covered, but legacy `botId` is still absent from the Feishu schema, credential resolver, setup status detection, manifest metadata, docs, and doctor compatibility migration path, so there is still a concrete repo-owned fix to make.

## What This Changes

The issue requests backward-compatible channel configuration migration for renamed Feishu fields and enum values, especially `botId` to `appId`, `topicSessionMode` to `groupSessionScope`, and `groupPolicy: "allowall"` to `"open"`.

## Best Possible Solution

Add a Feishu-owned compatibility path that accepts legacy top-level and per-account `botId`, normalizes it to canonical `appId`, exposes an `openclaw doctor --fix` migration for stored configs, preserves the existing `allowall` and `topicSessionMode` behavior, and documents the upgrade path without adding startup-time core migrations.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a Feishu config with `channels.feishu.botId` and no `appId`: current schema, setup, and credential-resolution code do not accept or read that key, while related legacy fields already have targeted compatibility coverage.

## Solution Assessment

Yes. The best fix is a narrow Feishu plugin plus doctor compatibility migration, because the missing behavior is owner-specific and does not require a new core/plugin API or a broad startup migration system.
