## Summary

Keep open: current main and the latest release already cover the reported `allowall` and `topicSessionMode` compatibility cases, but legacy Feishu `botId` is still not accepted, read, or migrated to `appId`, so the concrete upgrade break remains actionable.

## What This Changes

The issue requests backward-compatible channel configuration handling for legacy Feishu fields such as `botId`, `topicSessionMode`, and `groupPolicy: "allowall"`.

## Best Possible Solution

Add a Feishu-owned compatibility repair that maps top-level and per-account legacy `botId` to canonical `appId` through the existing doctor/plugin contract, preserves explicit `appId` precedence, and documents the upgrade path without adding core startup migrations.

## Reproduction Assessment

Yes, source-backed. A Feishu config with `channels.feishu.botId` and `appSecret` but no `appId` follows the reporter's legacy mapping, while current schema validation, credential resolution, setup readiness, and doctor migration all ignore `botId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin and its doctor compatibility contract; a broad startup migration system would conflict with the repository policy that legacy config repair belongs in `openclaw doctor --fix`.
