## Summary

Keep open: current main and the latest release cover the reported `allowall` and `topicSessionMode` cases, but Feishu legacy `botId` is still ignored by schema, credential resolution, and setup readiness, so the upgrade compatibility gap remains actionable.

## What This Changes

The issue requests backward-compatible channel configuration handling for renamed or changed Feishu fields such as `botId`, `topicSessionMode`, and `groupPolicy: "allowall"`.

## Best Possible Solution

Add Feishu-owned compatibility for legacy top-level and per-account `botId`, with current `appId` taking precedence, an idempotent `openclaw doctor --fix` migration, documentation, and focused regression coverage.

Yes for the narrow remaining gap: a Feishu plugin/doctor compatibility migration is the maintainable path. A broad startup-time migration layer would conflict with the repository policy that legacy config repair belongs in `openclaw doctor --fix`.

## Repair Work Prompt

Fix the remaining Feishu legacy config compatibility gap from https://github.com/openclaw/openclaw/issues/53556. Current main already normalizes `groupPolicy: "allowall"` to `"open"` and maps legacy `topicSessionMode` to group-topic routing; keep those behaviors. Add a bounded Feishu-owned repair path for legacy `channels.feishu.botId` and account-level `botId` so old config can be migrated or interpreted as `appId` in the supported repair/config path without introducing startup-time core migrations. Prefer `openclaw doctor --fix` / plugin compatibility normalization over runtime hot-path shims per AGENTS.md. Preserve `appId` precedence when both keys are present, do not change Feishu connection defaults or group security policy, add focused regression tests for top-level and account-level legacy `botId`, and include a user-facing changelog fix entry if this becomes a config repair shipped to users.
