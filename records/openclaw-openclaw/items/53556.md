## Summary

Keep open: current main and the latest release still cover the reported `allowall` and `topicSessionMode` compatibility cases, but legacy Feishu `botId` is still not accepted, read, or migrated, so the concrete upgrade break remains actionable.

## What This Changes

The issue requests backward-compatible channel config handling for legacy Feishu fields such as `botId`, `topicSessionMode`, and `groupPolicy: "allowall"` after OpenClaw upgrades.

## Best Possible Solution

Add a Feishu-owned compatibility path that accepts legacy top-level and per-account `botId`, normalizes it to `appId`, exposes an idempotent doctor migration, and documents the upgrade behavior without adding startup-time core migrations.

## Reproduction Assessment

Yes, source-backed. A Feishu config with `channels.feishu.botId` and `appSecret` but no `appId` follows the reporter's legacy mapping, while current schema validation, credential resolution, setup readiness, and doctor migration all ignore `botId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin and its doctor compatibility contract; a broad startup migration system would conflict with the repo policy that legacy config repair belongs in `openclaw doctor --fix`.
