## Summary

Current main and shipped releases already cover the central restart-continuity request: interrupted main sessions and subagent work are detected from persisted state and resumed, with durable restart handoff for OpenClaw-initiated restarts; narrower remaining failure modes are already tracked separately.

## What This Changes

Request durable session checkpoint/resume behavior so in-flight agent work can survive gateway restarts without losing task context.

## Best Possible Solution

Keep the shipped restart-recovery implementation as the core behavior, and use narrower follow-up issues for residual delivery retry, context-overflow, or explicit checkpoint-command semantics.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source and release history show current main already has restart recovery for the central behavior rather than a failing current-main path.

## Solution Assessment

Yes for the central user problem: resuming from persisted transcripts, `abortedLastRun`, and durable restart continuations is safer than blindly re-executing the last tool call. A user-confirmed `/resume` checkpoint format would need a separate product design because replaying tool calls can repeat side effects.
