## Label Justifications

- `P3`: The transcript data is preserved and a full-message reader exists, but the main WebChat history bubble remains capped without a user config path.
- `impact:message-loss`: Long assistant replies are visibly truncated after WebChat history reload, which users experience as missing message text even though the raw transcript remains available.

## Summary

Keep open: current main and v2026.6.9 still use the built-in 8KB WebChat history display cap with no persisted config key, and https://github.com/openclaw/openclaw/pull/95321 is open and unmerged as the active candidate fix.

## What This Changes

The issue asks for a supported configuration setting to raise the WebChat/Control UI chat history text projection cap above the hardcoded 8KB default.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/95321 for product direction and, if accepted, land a bounded config contract with strict schema, docs, tests, and real WebChat reload proof; otherwise keep the current full-message reader design and close the paired work together.

## Reproduction Assessment

Yes by source inspection: current main uses the 8KB default unless a request supplies `maxChars`, and the ordinary Control UI history path does not expose that request override to users. I did not run a live WebChat repro in this read-only review.

## Solution Assessment

Unclear as a product decision. A bounded config key would solve the reporter's requested workflow, but current docs and repository policy make new persisted config surface maintainer-sensitive.
