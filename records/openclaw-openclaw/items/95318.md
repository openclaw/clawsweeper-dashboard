## Label Justifications

- `P3`: The request is a low-risk WebChat ergonomics/config improvement because full transcript data exists and current UI has a side-reader path, but the main history bubble remains capped.
- `impact:message-loss`: The issue is about long assistant message content being visibly truncated in WebChat history reloads, which users experience as missing message text.

## Summary

Keep open: current main and v2026.6.8 still use the hardcoded 8KB history projection cap and document that WebChat has no persisted config section; the linked fixing PR is open and unmerged, so this issue should stay open until that PR lands, is replaced, or maintainers reject the config direction.

## What This Changes

Request a supported OpenClaw config setting to raise the WebChat/chat.history display projection cap beyond the built-in 8KB limit for long assistant replies.

## Best Possible Solution

Either keep the current bounded history plus full-message reader path and explicitly reject the config knob, or land a maintainer-sponsored config contract with schema, docs, bounds, tests, and real WebChat reload proof.

## Reproduction Assessment

Yes, from source inspection: current main still defaults WebChat history projection to 8KB, ignores config in the resolver, and documents no persisted WebChat config section. I did not run a live browser reload in this read-only review.

## Solution Assessment

Unclear as a product direction. A config knob would address the reporter's requested workflow, but current docs and prior cleanup intentionally prefer bounded history plus the `chat.message.get` full-message reader, so maintainers need to choose the supported path.
