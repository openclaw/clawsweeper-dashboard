- `P3`: The issue affects WebChat history readability for long messages but does not report data loss, a crash, or an unusable core runtime.

## Summary

Keep open: current main and the latest release still use the 8,000-character WebChat history projection default without a persisted config key, and the linked fixing PR is open but unmerged.

## What This Changes

The issue requests a supported config option to raise the WebChat/Control UI chat-history text projection cap above the hardcoded 8KB default.

## Best Possible Solution

Keep this issue open until the linked PR lands, is replaced by a maintainer-sponsored config-contract PR, or maintainers explicitly decide to keep bounded history plus the full-message reader without a new config key.

## Reproduction Assessment

Yes. Source inspection shows current main still projects history with an 8,000-character default unless an RPC maxChars override is supplied, and ordinary WebChat history/startup reload has no persisted config key for raising it.

## Solution Assessment

Unclear as product direction. A config knob would solve the reporter's requested workflow, but maintainers need to choose it over the current bounded-history plus chat.message.get full-message-reader design and require schema, docs, bounds, tests, and WebChat proof if they do.
