## Summary

Keep open: current main and v2026.6.10 still use the hardcoded 8KB WebChat history projection cap with no supported config key, and the open linked PR remains the active candidate rather than a merged fix.

## What This Changes

The issue requests a supported configuration path for increasing the WebChat history text projection cap beyond the hardcoded 8KB default.

## Best Possible Solution

Keep this issue open and review the linked implementation PR as the decision point for whether WebChat should add a configurable history cap, rely on the existing full-message reader, or support both.

## Reproduction Assessment

Yes from source inspection: current main still caps projected history text at 8,000 characters unless a request-level maxChars is supplied, and the resolver ignores config. I did not run a live WebChat browser reload, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

Unclear. Adding a config key is plausible, but current main also has an on-demand full-message reader that may be the preferred maintainable solution for long history without increasing default payloads.
