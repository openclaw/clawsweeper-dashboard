- `impact:message-loss`: The issue is about long assistant text being omitted from the visible WebChat history bubble behind a truncation marker, even though canonical transcript data remains available.

## Summary

Close as duplicate/superseded: the same WebChat truncation and configurable-cap direction was already decided in the related closed reports, and current main plus v2026.6.8 intentionally use bounded `chat.history`, per-request `maxChars` for API callers, and `chat.message.get` for full-message side reading instead of a persisted WebChat config knob.

## What This Changes

The issue requests a supported `openclaw.json` setting to raise the WebChat/chat.history text projection cap above the built-in 8,000-character default.

## Best Possible Solution

Keep the current supported contract: bounded WebChat history previews, per-request `chat.history.maxChars` for API clients, and the `chat.message.get` sidebar path for full text; reopen only if maintainers want to sponsor a new persisted config surface.

## Reproduction Assessment

Yes, source inspection confirms the reported visible behavior: Control UI history reloads do not pass a larger `maxChars`, so Gateway applies the 8,000-character default and appends the truncation marker. I did not run a live browser repro in this read-only cleanup review.

## Solution Assessment

No, adding a persisted WebChat config knob is not the current best solution because current docs and maintainer discussion retired that config surface. The maintained direction is bounded history plus an on-demand full-message reader.
