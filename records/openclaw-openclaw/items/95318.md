- `impact:message-loss`: Long assistant history text is visibly truncated after reload, which users experience as missing message content even though the transcript remains stored.

## Summary

Keep open: current main and v2026.6.8 still do not provide a supported WebChat history-cap config setting, and the open linked PR is the active candidate for the proposed fix. The remaining decision is whether maintainers want to add a new persistent gateway config surface or keep the existing bounded `chat.history` plus full-message reader design.

## What This Changes

The issue asks for a supported `openclaw.json` setting to raise WebChat chat-history text projection above the built-in 8KB default.

## Best Possible Solution

Either explicitly keep bounded `chat.history` plus `chat.message.get` as the supported design, or land a maintainer-sponsored config contract with schema, docs, numeric bounds, tests, and real WebChat reload proof.

## Reproduction Assessment

Yes. Source inspection shows the normal WebChat history reload path uses the gateway's 8KB default and current schema/docs expose no supported config key; I did not run a live browser reload.

## Solution Assessment

Unclear as a product direction. A config knob would solve the reporter's requested workflow, but current code and docs intentionally favor bounded history plus the `chat.message.get` full-message reader, so maintainers need to choose the supported path.
