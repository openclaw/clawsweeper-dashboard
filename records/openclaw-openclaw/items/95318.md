## Summary

Keep open: current main and the latest release still have a source-backed 8KB WebChat history projection cap with no persisted config path, but the requested fix is a new config/product direction and is already paired with an open closing PR.

## What This Changes

Request a supported configuration setting to raise the WebChat/chat.history display projection cap above the hardcoded 8KB default.

## Best Possible Solution

Maintainers should decide whether to sponsor a documented, schema-backed history-cap config knob; if not, keep the current bounded history plus `chat.message.get` design explicit in docs.

## Reproduction Assessment

Yes, as a source reproduction: current main uses an 8,000-character default when no RPC `maxChars` is supplied, Control UI does not pass a larger cap on normal reload, and the schema/docs expose no persisted setting. I did not run a live WebChat browser reload in this read-only review.

## Solution Assessment

Unclear. A config knob is a plausible narrow answer for local deployments, but current docs and history intentionally prefer bounded `chat.history` plus `chat.message.get`, so maintainer direction is the safer next step.
