## Label Justifications

- `P3`: The request is a low-risk WebChat ergonomics/config improvement because the full transcript remains stored and an adjacent full-message reader exists, but the main history bubble remains capped.
- `impact:message-loss`: Long assistant content is visibly truncated after WebChat history reloads, which users experience as missing message text in the chat view.

## Summary

Keep open: current main and the latest release still use a hardcoded 8KB WebChat history projection cap with no supported persisted config key, and the linked fix candidate remains open and unmerged.

## What This Changes

The issue requests a supported WebChat/Gateway configuration setting to raise the chat history text projection cap above the hardcoded 8KB default.

## Best Possible Solution

Either keep bounded history plus `chat.message.get` and explicitly reject a cap knob, or land a maintainer-sponsored bounded config contract with schema, docs, tests, and real WebChat reload proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: create a long assistant message, reload WebChat history without a per-request override, and the Gateway projection applies the 8KB default and appends the truncation marker. I did not run a live browser reload in this read-only review.

## Solution Assessment

Unclear as product direction. A config knob would address the reporter's requested workflow, but current docs and prior cleanup intentionally prefer bounded history plus `chat.message.get`, so maintainers need to choose the supported contract.
