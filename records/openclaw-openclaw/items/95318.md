## Label Justifications

- `P3`: The issue is a low-risk WebChat ergonomics/config request with preserved transcript data and an existing full-message reader, but the main history bubble remains capped.
- `impact:message-loss`: Long assistant content is visibly truncated after WebChat history reloads, which users experience as missing message text even though the raw transcript still exists.

## Summary

Keep open: current main and v2026.6.9 still use the 8KB WebChat history projection default without a persisted config key, and the linked fixing pull request is open rather than merged.

## What This Changes

The issue requests a supported configuration setting to raise the WebChat/chat-history display projection cap above the current hardcoded 8KB behavior.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/95321 lands as a complete bounded config contract, is replaced by a maintainer-sponsored approach, or maintainers explicitly reaffirm the bounded-history plus full-message-reader design.

## Reproduction Assessment

Yes at source level. Current main reloads WebChat history without a per-request `maxChars`, the resolver ignores config and falls back to 8,000 characters, and projection appends `...(truncated)...`; I did not run a live WebChat browser repro in this read-only review.

## Solution Assessment

Unclear as product direction. A config knob would solve the reporter's requested workflow, but maintainers need to choose it over the current bounded-history plus `chat.message.get` reader design and require schema, docs, bounds, tests, and live WebChat proof if they do.
