## Summary

Keep open: current main and the latest release still use the hardcoded 8KB WebChat history projection cap with no persisted config key, and the linked fixing PR is open rather than merged.

## What This Changes

The issue requests a supported WebChat/Gateway config setting to raise the chat-history display projection cap for long assistant replies after reload.

## Best Possible Solution

Keep this issue open until the linked PR lands as a complete bounded config contract, is replaced by a maintainer-sponsored alternative, or maintainers explicitly keep the bounded-history plus full-message-reader design.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main projects `chat.history` with an 8KB default unless a per-request `maxChars` is supplied, and no persisted Gateway/WebChat config key is accepted for ordinary reloads. I did not run a live WebChat session in this read-only review.

Unclear. A config knob would satisfy the reporter's requested workflow, but maintainers need to choose it over the current bounded-history plus `chat.message.get` full-message-reader design and require schema, docs, bounds, tests, and live WebChat proof if they do.
