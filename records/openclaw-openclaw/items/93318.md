## Summary

Keep open: the bridge-to-dylib mapping fixes the prior bridge-token defect, but this PR still breaks configured AppleScript voice/threaded attachment sends and lacks real iMessage proof.

## Best Possible Solution

## Reproduction Assessment

Yes for source-level reproduction: current main resolves `sendTransport` for RPC text sends but hardcodes `--transport auto` for media `send-attachment`. PR-head source also shows the remaining AppleScript voice/threaded attachment conflict against the upstream CLI contract.

## Solution Assessment

No. The bridge-to-dylib mapping is the right boundary fix, but the branch should also preserve existing fallback behavior for AppleScript-incompatible attachment modes and provide real transport proof.
