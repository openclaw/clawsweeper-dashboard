## Summary

Keep open: the code direction looks correct and still useful, but this external PR is not merge-ready because its after-fix proof is still a stubbed argv capture rather than a real macOS `imsg`/Messages attachment send.

## What This Changes

This PR threads the existing iMessage `sendTransport` setting into `send-attachment` media sends, maps `bridge` to `dylib`, falls back for AppleScript-incompatible audio/reply modes, and adds regression tests.

## Best Possible Solution

Land the narrow sender-boundary fix after redacted real macOS proof shows configured attachment sends use a supported transport without regressing default auto, bridge, AppleScript image, voice, or threaded-reply behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main forwards `sendTransport` on RPC text sends but hardcodes `--transport auto` for `send-attachment`. I did not establish a live macOS Messages repro in this read-only review.

## Solution Assessment

Yes for the code direction: the patch fixes the existing sender boundary and matches the checked upstream CLI contract. It is not merge-ready until real macOS attachment-send proof exists or maintainers explicitly override that gate.
