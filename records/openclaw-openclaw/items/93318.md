## Label Justifications

- `P2`: The PR targets a concrete iMessage attachment delivery bug with a channel-limited blast radius.

## Summary

Keep open: the underlying iMessage attachment transport bug is valid, but the PR still forwards OpenClaw's `bridge` token directly to an `imsg send-attachment` CLI that rejects it, and the added proof remains a stubbed argv capture rather than real macOS delivery proof.

## What This Changes

The PR adds an iMessage attachment helper transport parameter, threads the resolved `sendTransport` into `send-attachment` calls, and adds an `applescript` argv regression test.

## Best Possible Solution

Normalize OpenClaw transport values at the attachment boundary, including `bridge` to `dylib` and unsupported AppleScript flag combinations, then require redacted real macOS attachment-send proof.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves `sendTransport` for RPC text sends but hardcodes `--transport auto` for media `send-attachment`. The PR's bridge path is also source-reproducible because upstream `send-attachment` rejects `bridge`.

## Solution Assessment

No. Forwarding the existing config token literally is too narrow because `bridge` must be translated to `dylib` for `send-attachment`, and unsupported AppleScript attachment flag combinations need an explicit route.
