## Summary

Keep open: the intended iMessage attachment transport fix is useful, but the patch is not merge-ready because it forwards OpenClaw's `bridge` config token directly to `imsg send-attachment`, whose CLI contract expects `dylib`; the external PR also has mock-only proof.

## What This Changes

The PR threads the resolved iMessage `sendTransport` into media `send-attachment` calls and adds an argv regression test for an `applescript` account override.

## Best Possible Solution

Normalize the existing OpenClaw transport contract at the attachment boundary, including `bridge` to `dylib` and unsupported AppleScript attachment flag combinations, then add focused coverage and redacted real macOS proof.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves `sendTransport` for RPC text sends but hardcodes `--transport auto` for media `send-attachment`. I did not run a live macOS attachment send in this read-only review.

## Solution Assessment

No. Forwarding the value literally is not the best fix because OpenClaw's RPC token `bridge` differs from the attachment CLI's `dylib` token; a narrow adapter at the attachment boundary is safer.
