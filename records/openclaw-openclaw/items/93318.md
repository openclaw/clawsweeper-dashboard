## Summary

Keep open: the underlying iMessage attachment transport bug is real, but this patch is not merge-ready because it passes OpenClaw's raw `bridge` config value to `imsg send-attachment`, whose released/current CLI expects `dylib`, and the PR only supplies mock-level argv proof.

## What This Changes

The PR adds a `transport` parameter to the iMessage attachment helper, passes the resolved account `sendTransport` into it, and adds a regression test for an applescript-configured media send.

## Best Possible Solution

Keep the PR open, translate the existing OpenClaw transport enum for `send-attachment` before merge, cover the bridge and unsupported applescript cases, then require redacted live macOS iMessage attachment proof.

## Reproduction Assessment

Yes. Current main hardcodes attachment sends to `--transport auto`, while text sends already use the configured transport; the proposed raw bridge path is also source-reproducibly invalid against the current `imsg` CLI contract.

## Solution Assessment

No. Threading the existing setting is the right owner boundary, but the attachment CLI needs an explicit transport mapping and unsupported-combination coverage before this is the best fix.
