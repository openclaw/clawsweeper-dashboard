## Label Justifications

- `P2`: The PR targets a real iMessage attachment delivery bug with channel-limited blast radius, but it is not an emergency.
- `merge-risk: 🚨 compatibility`: The diff can break existing `sendTransport: "bridge"` configs by passing a supported OpenClaw value to an upstream CLI that rejects it.

## Summary

Keep open: the current-main bug is real, but this PR is not merge-ready because it forwards OpenClaw's `bridge` config token directly to a CLI that expects `dylib`, and the proof still exercises a stubbed subprocess boundary rather than a real macOS/iMessage attachment send.

## What This Changes

The PR threads the resolved iMessage `sendTransport` into `trySendAttachmentForTarget` and adds an argv regression test for an `applescript` attachment send.

## Best Possible Solution

Normalize transport values at the attachment boundary, cover bridge and AppleScript-incompatible attachment cases with focused tests, then add redacted real macOS attachment-send proof.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves `sendTransport` for RPC text sends but hardcodes `--transport auto` for media `send-attachment`. The branch's remaining defect is also source-reproducible because upstream rejects `bridge` and AppleScript voice/reply flag combinations.

## Solution Assessment

No. Forwarding the existing config token literally is too narrow; the attachment boundary needs an adapter from OpenClaw transport names to the upstream CLI contract plus tests for the non-default paths.
