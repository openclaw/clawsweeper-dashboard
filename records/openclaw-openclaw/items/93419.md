## Label Justifications

- `P1`: The issue blocks plugin-triggered embedded agent turns for users whose authenticated default provider is not OpenAI.
- `impact:auth-provider`: The failure is a provider/model routing mismatch that sends embedded runs to OpenAI and then fails OpenAI auth.

## Summary

Keep open: current main still initializes omitted `runEmbeddedAgent` provider/model values from the global OpenAI defaults, and the latest release tag has the same fallback. That contradicts the public SDK runtime contract and the gateway's normal configured-model resolver, so this is a valid narrow provider-routing bug with no exact canonical duplicate or merged fix found.

## What This Changes

Request that `runEmbeddedAgent` inherit the configured default model when callers omit `provider` and `model`, instead of falling back to `openai/gpt-5.5`.

## Best Possible Solution

Resolve the embedded runner's initial provider/model through the existing configured default-model helper, preserve explicit per-call overrides, then add a regression covering a non-OpenAI default with omitted `provider` and `model`.

## Reproduction Assessment

Yes from source inspection, but not by executing a live plugin repro. The no-provider/model path in `runEmbeddedAgent` selects `openai/gpt-5.5` from constants, while the docs and gateway resolver show the expected configured-default path.

## Solution Assessment

Yes, the best fix is to reuse the existing configured model-resolution path inside the embedded runner rather than adding a new option or asking every plugin to pass provider/model explicitly. The repair should keep explicit call overrides authoritative and normalize canonical `provider/model` refs safely.
