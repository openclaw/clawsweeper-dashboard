## Summary

Keep open: current main and v2026.6.6 still only supervise respawn children through catchable signal forwarding plus systemd control-group cleanup, while the requested parent-death guard for wrapper SIGKILL/crash/orphan cases is not present. The focused fixing PR was closed unmerged for queue cleanup, and no open replacement now owns the remaining launcher/systemd orphan behavior.

## What This Changes

The issue reports that an `openclaw.mjs gateway` respawn child can outlive its wrapper, keep the gateway port bound, and make systemd restarts fail with `EADDRINUSE`.

## Best Possible Solution

Land an equivalent parent-death guard in the launcher/shared respawn supervision path, with regression coverage for SIGKILL/crash-style wrapper death and without adding service-specific fuser policy as the primary fix.

## Reproduction Assessment

No live systemd reproduction was run in this read-only review. The source-level path is clear for wrapper death that bypasses signal handlers because current respawn supervision only reacts when the parent receives a catchable signal.

## Solution Assessment

Yes. A parent-death guard in the launcher/shared respawn path is the maintainable fix; the systemd `fuser` workaround is defensive cleanup and does not solve manually launched or out-of-cgroup orphan children.
