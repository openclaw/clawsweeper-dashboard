## Summary

Keep open: current main and v2026.6.8 still rely on catchable signal forwarding plus service-mode stale-port cleanup, but they do not guarantee a respawned gateway child exits when the launcher parent disappears without running handlers. The focused fixing PR was closed unmerged, and the related systemd report was already closed into this issue as the canonical remaining work.

## What This Changes

The issue asks OpenClaw to prevent launcher-respawned gateway children from outliving the wrapper and holding the gateway port during systemd restarts.

## Best Possible Solution

Land an equivalent parent-death guard in the launcher/shared respawn supervision path, with SIGKILL/crash-style regression coverage and Linux systemd restart proof; keep stale-port cleanup as secondary defense.

## Reproduction Assessment

No live current-main systemd reproduction was run in this read-only review. The source path is clear: current respawn cleanup depends on parent-side signal handlers, while the reported SIGKILL/crash-style parent death cannot run that cleanup path.

## Solution Assessment

Yes. A parent-death guard in the launcher/shared respawn path is the narrow maintainable fix; systemd `fuser`, `KillMode`, and stale-port cleanup are useful mitigations but do not solve manually launched or different-owner orphan children.
