Keep open. Current main does not support the specific in-memory store-load theory, but the reported Feishu restart recovery failure remains plausible because recovery is still a fixed-delay, non-readiness-aware path for a push channel.

## What This Changes

The issue requests reliable recovery and response delivery for interrupted Feishu DM main sessions after a gateway restart.

## Best Possible Solution

Keep this open and make restart-aborted main-session recovery durable and readiness-aware for push channels, without adding Feishu-specific core policy or a delay config knob as the primary fix.

No high-confidence live reproduction was established. Source inspection does show a plausible current-main timing path: fixed-delay recovery is independent of Feishu WebSocket readiness, while the store-load theory is not supported by the disk-backed store code.

## Solution Assessment

No, a configurable delay alone is not the best fix. The maintainable path is a generic readiness/backoff or deferred-recovery mechanism that preserves retry state until the recovered turn can be queued or delivered.

## Repair Work Prompt

Fix the Feishu/push-channel restart recovery timing bug reported in https://github.com/openclaw/openclaw/issues/82974. On current main, lock marking already uses disk-backed updateSessionStore, so do not add a store-loading migration as the main fix. Keep core plugin-agnostic: make restart-aborted main-session recovery readiness/backoff-aware for push-channel delivery and avoid clearing abortedLastRun or suppressing future recovery until the resumed turn is actually accepted for recovery and final delivery can be queued or sent. Avoid adding a public recoveryDelayMs config option as the primary fix unless maintainers explicitly ask for that product surface. Add focused regression coverage for the recovery scheduler and any gateway/Feishu readiness contract touched.
