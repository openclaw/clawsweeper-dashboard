## Label Justifications

- `merge-risk: 🚨 compatibility`: The branch deliberately changes multi-bot shared-thread behavior by suppressing non-owning bot routing for active bound-session threads.

## Summary

Keep this PR open, but it is not merge-ready: current main still has the account-scoped Discord thread-binding gap, while the PR branch is dirty against main, lacks real two-bot Discord proof, and still has a P1 stale-config message-drop risk.

## What This Changes

The PR adds a cross-account active Discord thread-binding lookup, suppresses generic routing for non-owning bot accounts, exports the helper, and adds preflight regression tests.

## Best Possible Solution

Rebase the Discord fix onto current main, keep exclusive bound-thread ownership only for active thread bindings, use the live runtime config snapshot for stale foreign-owner decisions, and require focused regressions plus redacted two-bot Discord proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main resolves Discord bindings through the inbound account id and the thread manager keys getByThreadId by account id, so another bot cannot see a binding owned by the first account. I did not run a live two-bot guild reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed: the direction is plausible, but the branch needs to be rebased onto current main and the foreign-owner stale-binding check should use the live runtime config snapshot. The safer end state is an owner-bound cross-account lookup with explicit tests for active owners, stopped managers, disabled accounts, and hot config changes.
