## Summary

Keep open. Current main still has an account-scoped Discord thread-binding lookup, so the linked multi-bot isolation bug remains source-reproducible; this PR targets the right owner boundary but is not ready because its stale-owner check uses the preflight config and the branch is dirty against current main.

## What This Changes

The PR adds a Discord cross-account active thread-binding lookup in preflight, exports that helper, and adds regressions for foreign-owner suppression plus stale, removed, and disabled owner bindings.

## Best Possible Solution

Adapt the fix onto the current Discord preflight/routing layout with an active cross-account owner lookup that ignores disabled or removed owners, preserves same-account behavior, and includes focused regressions plus a changelog entry.

## Reproduction Assessment

Yes, at source level. In a two-Discord-account setup sharing a thread, bind the thread through bot A with `sessions_spawn(thread=true, mode="session")`, then send a follow-up visible to bot B; current main only checks bot B's account-scoped binding adapter and can fall through to generic routing.

## Solution Assessment

No, not as written. The owner-boundary direction is right, but stale-owner filtering should use active runtime config, and the branch also needs a rebase/adaptation plus changelog coverage before merge.
