## Summary

Keep this PR open. Current main still resolves Discord thread bindings through the current Discord account only and does not implement the proposed cross-account owner suppression; the PR targets a real remaining routing isolation gap, but the branch needs maintainer review because the new suppression path can still drop valid messages when it checks a stale startup config instead of the active runtime config.

## What This Changes

The PR adds Discord preflight logic and thread-binding manager lookup support to suppress generic routing for a non-owning bot account when another active Discord account owns the thread binding, plus focused regression tests for stale and disabled foreign bindings.

## Best Possible Solution

Keep the PR open as the implementation candidate, but adapt it onto current main as a narrow Discord fix: add a cross-account active thread-owner lookup in the Discord thread-binding surface, run it before configured/generic routing for thread messages with no local binding, use the active runtime config to ignore stale or disabled foreign owners, preserve same-account behavior, and land focused tests for active foreign-owner suppression plus stale/removed/disabled owner cases.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: run two Discord bot accounts that can see the same thread, bind the thread under bot A via `sessions_spawn(thread=true, mode="session")`, then send a follow-up visible to bot B; current main resolves only bot B's account-scoped adapter and can fall through to generic routing.

## Solution Assessment

No, not as written. The PR is pointed at the right boundary, but the suppression check should use the active runtime config and be rebased into current main's split routing code so stale startup config cannot falsely drop valid messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Use active config for foreign-owner checks:** `extensions/discord/src/monitor/message-handler.preflight.ts:365`
  - body: The new suppression call passes `params.cfg`, but Discord monitors keep that startup config while thread-binding manager operations already consult the runtime snapshot. If a foreign account is removed, disabled, or turns off thread bindings in the active config after startup and its manager is stopped, this stale config can still treat the old binding as an active owner and drop valid messages for another bot until expiry. Resolve or pass the active runtime config for this check.
  - confidence: 0.83
