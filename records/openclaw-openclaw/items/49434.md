## Summary

Keep this PR open: the underlying Discord multi-bot thread ownership bug is still source-reproducible on current main, but the branch needs at least one correctness fix and real Discord multi-bot proof before merge.

## What This Changes

The PR adds cross-account Discord thread-binding lookup and preflight suppression so a non-owning Discord bot does not route messages from another account’s active bound thread, with tests for stale or disabled foreign bindings.

## Best Possible Solution

Keep the focused cross-account isolation fix, adapt it to the current Discord preflight routing structure, use the correct current config source for stale-owner filtering, and require redacted live Discord multi-bot proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main resolves Discord thread bindings by the inbound account, so a second configured bot that sees the same thread cannot resolve the first account’s active binding before falling through to normal routing.

## Solution Assessment

No. The direction is right, but the submitted branch still has a stale-config false-drop case and needs to be updated against the current preflight routing structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Use fresh config for foreign ownership checks:** `extensions/discord/src/monitor/message-handler.preflight.ts:368`
  - body: The function loads `freshCfg` for binding-aware route resolution, but the new foreign-owner suppression passes the stale handler snapshot (`params.cfg`). If a foreign Discord account is removed, disabled, or has thread bindings turned off at runtime and its manager has stopped, this still treats its old binding as active and drops valid messages for the current bot until the binding expires. Pass the same fresh config used for route/binding lookup into `findActiveThreadBindingByThreadId` and update the regression that currently asserts the stale snapshot behavior.
