## Label Justifications

- `P1`: The PR targets a real Discord multi-bot thread/session isolation failure that can route follow-up messages to the wrong bot/session.
- `merge-risk: 🚨 compatibility`: The patch deliberately changes multi-bot shared-thread behavior by suppressing non-owning bot routing for active bound-session threads.

## Summary

Keep open: current main still has account-scoped Discord thread-binding resolution, but this PR is not merge-ready because it conflicts with main, lacks live two-bot Discord proof, and still has a stale-config message-drop defect.

## What This Changes

The PR adds a cross-account Discord thread-binding lookup, suppresses generic routing for non-owning bot accounts in bound threads, exports the helper, and adds Discord preflight regression tests.

## Best Possible Solution

Rebase onto current main's routing-preflight split, evaluate foreign binding ownership against active runtime config, and merge only after focused regressions plus redacted live two-bot Discord proof confirm exclusive bound-thread routing.

## Reproduction Assessment

Yes, source-reproducible: current main resolves Discord thread bindings through the inbound account id, and the thread manager lookup is keyed by that account id. I did not run a live two-bot Discord guild reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed: the cross-account lookup direction is plausible, but the PR needs a current-main rebase and live-config handling for stale foreign owners. The safer solution is an owner-bound lookup with regressions for active owners, stopped managers, disabled accounts, and hot config changes.
