## Summary

Keep this PR open: the underlying Discord thread-isolation bug is valid and still not implemented on current main, but the latest head still has a config-source bug that can turn stale bindings into dropped messages, and the PR body provides only test-level proof.

## What This Changes

The PR adds a cross-account Discord thread-binding lookup in preflight, exports it from the thread-bindings barrel, and adds regressions for non-owning bot suppression plus stale/disabled foreign bindings.

## Best Possible Solution

Land a current-main adaptation that preserves exclusive active thread ownership across Discord accounts, consults current runtime config when filtering stopped/stale foreign owners, keeps same-account routing unchanged, and includes focused regressions plus redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level. Current main resolves Discord runtime bindings with the current accountId and the thread manager lookup is account-scoped, so a second bot can miss another account's active binding before generic routing continues.

## Solution Assessment

No, not as written. Exclusive cross-account suppression is the right direction, but stopped foreign-owner filtering should use current runtime config rather than the handler's startup preflight snapshot.
