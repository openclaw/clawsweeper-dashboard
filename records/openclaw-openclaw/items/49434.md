## Summary

Keep open. Current main still lacks cross-account Discord thread-owner suppression, so the PR is not obsolete, but the proposed patch is not ready to merge because it uses the preflight/startup config for the new foreign-owner check and omits the required changelog entry for a user-facing Discord fix.

## What This Changes

The PR adds Discord preflight logic and thread-binding manager lookup support to drop thread messages for a non-owning bot account when another Discord account owns an active thread binding, plus focused regression tests for stale and disabled foreign bindings.

## Best Possible Solution

Adapt the PR onto current main as a narrow Discord fix: add a cross-account active thread-owner lookup in the Discord thread-binding surface, run it before configured/generic routing for thread messages with no local binding, use the active runtime config snapshot to ignore stale/removed/disabled foreign owners, preserve same-account behavior, and add the missing changelog entry plus focused tests for active, stale, removed, and disabled-owner cases.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: run two Discord bot accounts that can see the same thread, bind the thread under bot A with `sessions_spawn(thread=true, mode="session")`, then send a follow-up visible to bot B; current main only checks bot B's account-scoped binding adapter and can fall through to generic routing.

## Solution Assessment

No, not as written. The PR targets the right owner boundary, but it should not lock the foreign-owner stale-binding decision to `params.cfg`; using the active runtime config via the runtime snapshot/config selector is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Use active config for foreign-owner checks:** `extensions/discord/src/monitor/message-handler.preflight.ts:365`

Use the active runtime config for this check.
