## Summary

Keep open. The PR targets a real closed-stdin watchdog false positive, but it is not merge-ready because the guard uses payload truthiness and must be rebased around current Claude live-session watchdog behavior.

## What This Changes

The PR changes `src/agents/cli-runner/execute.ts` to skip `noOutputTimeoutMs` when `stdinPayload` is truthy and to handle the now-optional watchdog value in timeout messages.

## Best Possible Solution

Land a rebased or replacement fix that disables the supervisor no-output watchdog only for normal closed-stdin child spawns, keeps Claude live-session watchdog timers numeric, avoids `0s` timeout messages, and adds focused tests plus the required changelog entry.

## Reproduction Assessment

Yes. A code-level reproduction is clear: run a stdin-mode or long-prompt fallback CLI backend that stays silent longer than `noOutputTimeoutMs`; current main passes both `input` and `noOutputTimeoutMs` to the child supervisor even though the child adapter closes stdin.

## Solution Assessment

No, not as submitted. The direction is right, but the guard should use raw stdin presence or explicit closed-stdin state, be scoped to non-live child spawns, and avoid misleading disabled-watchdog timeout text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Scope the watchdog skip to closed-stdin child runs:** `src/agents/cli-runner/execute.ts:169-175`
  - body: The new guard is computed before the live-session branch, but current main uses stdin-mode Claude live sessions with `stdinMode: "pipe-open"` and numeric turn watchdogs. A rebase that makes this shared value `undefined` for stdin backends would disable or break Claude live-session no-output protection, so keep the skip in the normal closed-stdin child path.
  - confidence: 0.86
- **[P2] Use stdin presence for the watchdog guard:** `src/agents/cli-runner/execute.ts:169-175`
  - body: `stdinPayload` is `stdin ?? ""`, so an empty stdin-mode prompt still closes stdin but leaves this truthiness guard false. Guard on raw stdin presence or an explicit closed-stdin state so empty stdin gets the same watchdog behavior as non-empty stdin.
  - confidence: 0.9
- **[P3] Avoid reporting a 0s no-output timeout:** `src/agents/cli-runner/execute.ts:219-225`
  - body: When the watchdog is disabled, `noOutputTimeoutMs ?? 0` can produce a user-facing `0s` timeout if an unexpected no-output result reaches this branch. Build separate enabled and disabled watchdog messages instead of inventing a zero-second duration.
  - confidence: 0.78
