## Summary

Keep open: current main still applies the CLI no-output watchdog to closed-stdin child spawns, but this PR is now conflicting, lacks real behavior proof, and needs rework so it does not weaken Claude live-session watchdogs.

## What This Changes

The PR changes `src/agents/cli-runner/execute.ts` to make the CLI no-output watchdog optional when a stdin payload is present and adjusts no-output timeout diagnostics for an optional value.

## Best Possible Solution

Rebase against current main and disable the supervisor no-output watchdog only for normal child spawns with defined closed stdin, while preserving numeric Claude live-session per-turn watchdogs and overall timeouts.

## Reproduction Assessment

Yes, at source level. Current main passes `input: stdinPayload` and a numeric `noOutputTimeoutMs` to the child supervisor, while the child adapter closes stdin whenever `input` is defined; I did not run a live long-silent CLI process in this read-only review.

## Solution Assessment

No. The intended fix is plausible, but this branch should be reworked so only the normal closed-stdin child-spawn path disables the supervisor watchdog while Claude live-session turn watchdogs remain numeric.

  - body: On current main the same `noOutputTimeoutMs` value is also passed into `runClaudeLiveSessionTurn`, whose contract requires a number and keeps stdin open.   - confidence: 0.86

  - confidence: 0.82
- **[P3] Avoid zero-second watchdog messages:** `src/agents/cli-runner/execute.ts:219-225`
  - body: If a no-output result reaches this branch while the watchdog value is undefined, the new fallback reports `0s`, which points users at a bogus timeout. Build a generic message for the undefined case instead of formatting zero seconds.
  - confidence: 0.78
