## What This Changes

The PR changes `src/agents/cli-runner/execute.ts` to skip the CLI no-output watchdog when stdin input is present and adjusts no-output timeout diagnostics for an optional watchdog value.

## Best Possible Solution

Rebase against current main and disable the supervisor no-output watchdog only for normal child spawns with defined closed stdin, while preserving numeric Claude live-session watchdogs and overall timeouts.

## Reproduction Assessment

Yes, at source level. Current main passes defined stdin input to the child adapter, which closes stdin immediately, while still arming a numeric no-output watchdog; I did not run a live long-silent CLI process in this read-only review.

## Solution Assessment

No. The intended fix is plausible, but this branch should be reworked so only the normal closed-stdin child-spawn path disables the supervisor watchdog while Claude live-session turn watchdogs remain numeric.

Full review comments:

- **[P2] Keep live-session watchdogs numeric:** `src/agents/cli-runner/execute.ts:169-175`
  - body: On current main the same `noOutputTimeoutMs` value is also passed into `runClaudeLiveSessionTurn`, whose contract requires a number and keeps stdin open.   - confidence: 0.86
- **[P2] Check stdin presence, not payload truthiness:** `src/agents/cli-runner/execute.ts:169-175`
  - body: `stdinPayload` is `stdin ?? ""`, so an stdin-mode backend with an empty prompt still closes stdin in the child adapter but keeps the no-output watchdog enabled. Use `stdin != null` or an explicit closed-stdin boolean for the normal child path.
  - confidence: 0.82
- **[P3] Avoid zero-second watchdog messages:** `src/agents/cli-runner/execute.ts:219-225`
  - body: If a no-output result reaches this branch while the watchdog value is undefined, the new fallback reports `0s`, which points users at a bogus timeout. Build a generic message for the undefined case instead of formatting zero seconds.
  - confidence: 0.78
