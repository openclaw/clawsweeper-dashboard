## Summary

Keep open: the underlying closed-stdin watchdog bug still exists on current main, but this stale PR is conflicting, lacks real behavior proof, and needs rework to avoid weakening current Claude live-session watchdogs.

## What This Changes

The PR changes `src/agents/cli-runner/execute.ts` to skip the CLI no-output watchdog when stdin payload is present and adjusts watchdog timeout diagnostics for an optional timeout value.

## Best Possible Solution

Rebase against current main and disable the supervisor no-output watchdog only for normal child spawns with defined closed stdin, while preserving numeric Claude live-session turn watchdogs and overall timeouts.

## Reproduction Assessment

Yes, from source inspection. Current main passes `input: stdinPayload` and a numeric `noOutputTimeoutMs` to the child supervisor, while the child adapter closes stdin whenever `input` is defined; I did not run a live long-silent CLI process in this read-only review.

## Solution Assessment

No. The intended fix is plausible, but this branch is stale and should be reworked so the closed-stdin child-spawn path changes without changing the current Claude live-session watchdog contract.

- **[P2] Keep live-session watchdogs numeric:** `src/agents/cli-runner/execute.ts:169-175`
  - body: The stale patch makes `noOutputTimeoutMs` optional at the shared runner boundary. On current main that value is also passed into `runClaudeLiveSessionTurn`, whose stdin stays open with `stdinMode: "pipe-open"` and whose timers require a number, so resolve the conflict by disabling the supervisor watchdog only on the normal closed-stdin child spawn.
  - confidence: 0.86
- **[P2] Check stdin presence, not payload truthiness:** `src/agents/cli-runner/execute.ts:169-175`
  - body: `stdinPayload` is `stdin ?? ""`, so an stdin-mode backend with an empty prompt still closes stdin in the child adapter but keeps the no-output watchdog enabled. Use `stdin != null` or an explicit closed-stdin boolean for the normal child path.
  - confidence: 0.82
- **[P3] Avoid zero-second watchdog messages:** `src/agents/cli-runner/execute.ts:219-225`
  - body: If a no-output result reaches this branch while the watchdog value is undefined, the new fallback reports `0s`, which points users at a bogus timeout. Build a generic message for the undefined case instead of formatting zero seconds.
  - confidence: 0.78
