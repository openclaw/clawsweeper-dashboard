## Summary

Keep open: current main has not implemented this fallback change, but the branch is not mergeable as-is because it routes a secret-sensitive writer through resolved runtime config, the added test fails against current main, and real behavior proof is missing.

## What This Changes

The PR changes omitted-config models.json planning to use the loaded runtime config for `config` while retaining `runtimeSource` for secret markers, and adds a regression test for the undefined-config path.

## Best Possible Solution

Revise the fallback to stay source-authoritative for serialization and preserve SecretRef markers across provider request fields, or address the provider-discovery startup bloat path if that is the actual remaining bug.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection shows `loaded` can be a resolved runtime snapshot, nested provider request SecretRefs resolve to plaintext, and marker restoration only covers provider `apiKey` and top-level `headers`. The added test failure is also visible in the PR merge ref against current main.

## Solution Assessment

No. Using `loaded` for the serialized config is not the safest fix because it can persist resolved request secrets and does not address the explicit-config provider-discovery path described by the linked report.
