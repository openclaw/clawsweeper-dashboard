## Summary

Keep open: current main has not implemented this PR's fallback change, but the branch should not merge as-is because it can serialize resolved provider request secrets, the added test is broken, and the PR still lacks real after-fix behavior proof.

## What This Changes

The PR changes omitted-config models.json planning to use the loaded config instead of runtimeSource and adds a regression test for that branch.

## Best Possible Solution

Revise the branch into a source-authoritative fallback that preserves SecretRef markers across provider request.* fields, or close it in favor of the provider-discovery startup bloat fix if the fallback path is not needed.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection traces loaded to the resolved runtime snapshot, shows provider request.* SecretRefs resolving to plaintext, and shows marker restoration only covers apiKey/top-level headers. The added test's missing helper import and wrong helper arity are also directly visible from the diff and current test file.

## Solution Assessment

No. Setting config to loaded is not the safest fix because it can serialize resolved request secrets and does not address the explicit-config provider-discovery bloat path described by the linked report.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep omitted config from writing resolved request secrets:** `src/agents/models-config.ts:105`
  - body: When this branch sets the planned `config` to `loaded`, omitted-config calls with an active secrets runtime can serialize the resolved runtime snapshot. Runtime resolution turns `models.providers.*.request` SecretRefs into plaintext strings, while `models-config` only restores `apiKey` and top-level `headers` markers before writing `models.json`, so request auth/header/TLS secrets can be persisted.
  - confidence: 0.9

  - confidence: 0.94
