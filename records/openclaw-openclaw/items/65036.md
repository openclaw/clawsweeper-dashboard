## Summary

Keep open; the PR is not implemented on current main and remains a plausible bug-fix direction, but the proposed line change is not safe to merge because it can serialize resolved request secrets and the added regression test does not fit the current harness.

## What This Changes

The PR changes omitted-config models.json planning to prefer the loaded active config over the runtime source snapshot and adds a regression test for that branch.

## Best Possible Solution

Revise models.json generation to stay source-authoritative for serialization and SecretRef markers, then address the provider-discovery bloat path through a scoped behavior decision or canonical follow-up.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: active secrets runtime keeps resolved and source configs separately, provider request SecretRefs resolve to plaintext, and marker restoration only covers provider `apiKey` plus top-level `headers`. I did not run tests in this read-only review.

## Solution Assessment

No. Using the loaded runtime config as the serialized config is not the safest fix because it can persist resolved request secrets, and the linked bloat report points to explicit-config plugin discovery as the observed production path.
