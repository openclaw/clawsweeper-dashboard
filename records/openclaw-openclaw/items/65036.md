## Summary

Keep open: current main has not implemented this PR's central change, but the proposed fallback is unsafe because it can feed a resolved runtime config into models.json serialization. The linked bloat report also says the observed production startup path uses explicit config and provider discovery, not this omitted-config branch.

## What This Changes

The PR changes omitted-config models.json planning to use the loaded runtime config for `config`, keeps `runtimeSource` for secret markers, and adds an undefined-config regression test.

## Best Possible Solution

Revise the models.json path to stay source-authoritative for serialization, preserve SecretRef markers across nested provider request fields, and handle the provider-discovery bloat path if that remains the live root cause.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: secrets runtime stores resolved config separately from source config, provider request SecretRefs resolve to plaintext, and marker restoration only covers provider `apiKey` plus top-level `headers`. I did not run tests in this read-only review.

## Solution Assessment

No. Using `loaded` for serialized config is not the safest fix because it can persist resolved request secrets, and the linked report indicates the observed bloat likely needs a provider-discovery behavior decision.
