## Summary

Keep open: current main still has no `createOnMiss` or equivalent strict existing-session policy, and the remaining work is a compatibility-sensitive config/API feature rather than conservative cleanup. Source inspection shows the current model-facing send path can still normalize key-shaped targets and flow into agent-session creation, while related searches found no merged or open PR that owns this exact strict-mode request.

## What This Changes

Request an opt-in strict session mode for `sessions_send`, `sessions_spawn`, and session-key resolution so unknown session keys fail fast instead of creating new sessions.

## Best Possible Solution

Add an explicit opt-in strict existing-session policy with default-compatible behavior, canonical config/schema/docs, and focused tests for both current create-on-miss behavior and fail-closed unknown-session behavior.

## Reproduction Assessment

Yes, at source level: key-shaped session targets can pass `resolveSessionReference`, and the agent handler can create a new session when the explicit requested key has no entry. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API review. The default-true opt-in strict mode is plausible and compatibility-aware, but the permanent config location and exact affected surfaces should be decided before implementation.
