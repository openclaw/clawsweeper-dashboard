## Label Justifications

- `P2`: This is a normal-priority gateway/session safety improvement with bounded but real routing impact, not an outage or emergency.
- `impact:session-state`: The reported failure mode splits conversation state by creating or using an unintended phantom session key.
- `impact:message-loss`: Messages addressed to a malformed key can be routed into an unintended session where the expected peer never sees them.

## Summary

Current main still does not implement the requested `createOnMiss=false` or equivalent strict session policy. The protocol `sessions.send` path has a partial not-found guard, but the model-facing `sessions_send` path can still resolve arbitrary non-empty keys into agent dispatch, and adding fail-closed behavior is a compatibility-sensitive config/API decision.

## What This Changes

Request a backwards-compatible strict session mode so `sessions_send` and related session dispatch paths fail on unknown session keys instead of implicitly creating or adopting sessions.

## Best Possible Solution

Add a default-compatible, opt-in strict existing-session policy with schema/docs and focused tests once maintainers confirm the permanent config or request-level API shape.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: `sessions_send` can resolve a non-empty key without an existence check and pass it to agent dispatch, where a missing entry is treated as a new session.

## Solution Assessment

Is this the best way to solve the issue? Unclear pending maintainer API review; the requested strict mode is plausible and aligned, but the exact config/request surface and included session-creation paths need a product decision.
