## Summary

Keep open: current main still lacks an opt-in strict create-on-miss policy for agent session sends/spawns, and the source path can still accept key-shaped targets that later create a missing agent session.

## What This Changes

The issue requests a default-compatible strict mode so `sessions_send`, `sessions_spawn`, or session-key resolution can fail fast on unknown session keys instead of implicitly creating a new session.

## Best Possible Solution

Add an opt-in, default-compatible strict existing-session policy with schema/protocol/docs and focused coverage after maintainers settle the name and affected session surfaces.

## Reproduction Assessment

Yes, at source level: key-shaped targets can pass `resolveSessionReference`, then the agent handler can create and persist a new requested-key session when no entry exists. I did not run a live Gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API review. The requested default-compatible strict mode is plausible, but the permanent config location and exact session surfaces should be settled before implementation.
