## Summary

Keep open: current main still lacks a strict create-on-miss policy, and the source path still lets key-shaped `sessions_send` targets reach agent dispatch where a missing requested key can be persisted as a new session. The request is valid, but it adds config surface and changes session-creation semantics, so it needs maintainer product/API review before implementation.

## What This Changes

Request an opt-in strict mode for session-key resolution so unknown `sessions_send` and related session targets fail with a structured not-found error instead of implicitly creating a new session.

## Best Possible Solution

Add an opt-in strict existing-session policy only after maintainers decide the config namespace and affected surfaces, preserving default create/adopt behavior and adding focused tests plus docs.

## Reproduction Assessment

Yes, at source level: key-shaped targets can pass `resolveSessionReference`, and the agent path can persist a new requested-key session when no entry exists. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API review. A default-compatible strict mode is plausible, but the permanent config location and exact session-creation surfaces should be settled before implementation.
