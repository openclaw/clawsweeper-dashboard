## Summary

Keep open: current main still lacks `createOnMiss` or an equivalent strict existing-session policy, and source inspection still shows permissive key-shaped resolution flowing into session creation. This is valid work, but it needs maintainer API/product review because it adds a new config surface and opt-in fail-closed session semantics.

## What This Changes

Request an opt-in strict policy for `sessions_send`, `sessions_spawn`, and session-key resolution that rejects unknown session keys instead of implicitly creating a new session.

## Best Possible Solution

Add a default-compatible, opt-in strict existing-session policy with canonical config/schema/docs and focused tests for both current create-on-miss behavior and fail-closed unknown-session behavior.

Yes, at source level: a key-shaped target can resolve as accepted without requiring an existing entry, and the agent path can create a new requested-key session. I did not run a live gateway reproduction in this read-only review.
