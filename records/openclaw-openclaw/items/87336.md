## Summary

Keep open: current main and the latest release still do not provide a `createOnMiss` or equivalent strict session policy, and the remaining work is a compatibility-sensitive config/API decision rather than cleanup.

## What This Changes

Request a strict session resolution mode so gateway `sessions_send` and `sessions_spawn` fail on unknown session keys instead of implicitly creating a new session.

## Best Possible Solution

Add a maintainer-approved, backward-compatible strict mode at the session-reference/tool boundary if this product direction is accepted, with docs and tests proving both current default behavior and opt-in fail-closed behavior.

## Reproduction Assessment

Yes, as source-reproducible: Gateway key resolution rejects a missing key, but the agent-tool resolver falls back to returning the raw canonical-looking key as OK before `sessions_send` dispatches it. I did not run a live repro because this read-only review is evaluating a requested opt-in strict behavior.

## Solution Assessment

Unclear pending maintainer API review. The default-preserving strict mode is plausible, but the permanent config/protocol shape and affected session surfaces need a product decision first.
