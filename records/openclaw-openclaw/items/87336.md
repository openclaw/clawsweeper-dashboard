## Summary

Keep open: current main still lacks a `createOnMiss` or equivalent strict existing-session policy, and the source path still permits key-shaped targets to reach agent dispatch where a fresh session can be created.

## What This Changes

Request an opt-in strict create-on-miss policy so unknown session-key targets for session send/spawn flows fail instead of creating a phantom session.

## Best Possible Solution

Add a default-compatible opt-in strict existing-session policy with schema/docs and focused coverage for unknown-key sends plus any intentionally excluded session-creation paths.

## Reproduction Assessment

Yes, at source level: key-shaped targets can pass `resolveSessionReference`, and the agent handler can persist a new requested-key session when no entry exists. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API review. The requested default-compatible strict mode is plausible, but the permanent config location and exact affected surfaces should be settled before implementation.
