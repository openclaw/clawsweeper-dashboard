## Summary

Keep open. Current main still leaves LiteLLM interactive onboarding on the generic API-key path: it prompts for only the key, preserves or defaults the proxy URL to localhost, and exposes a one-model static catalog instead of prompting for the LiteLLM URL and discovering proxy models. The related PR in the provided context is closed unmerged and targets `tools.media` alias validation, not this onboarding regression.

## What This Changes

The issue asks for LiteLLM onboarding to prompt for the proxy server URL and show discovered LiteLLM models for default-model selection instead of skipping URL setup and showing one model.

## Best Possible Solution

Fix this in the bundled LiteLLM provider plugin: add provider-owned interactive setup that prompts for the LiteLLM proxy base URL, collects the API key, discovers selectable models from the configured OpenAI-compatible `/models` endpoint when reachable, falls back to manual model entry, and updates LiteLLM tests, docs, and the changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a fresh `openclaw onboard --auth-choice litellm-api-key` path: the registered LiteLLM auth method prompts only for the API key, applies localhost or existing base URL config, and the provider catalog contains only the static default model.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/litellm` using the existing provider setup/discovery seams; moving this into core would violate the provider-owned onboarding boundary and is not needed for the reported behavior.

## Repair Work Prompt

Fix the LiteLLM onboarding regression tracked by https://github.com/openclaw/openclaw/issues/44789. Current `openclaw onboard --auth-choice litellm-api-key` only prompts for the LiteLLM API key, keeps the proxy URL as an existing config value or `http://localhost:4000`, and leaves users with the one static `litellm/claude-opus-4-6` catalog entry. Implement the fix inside the bundled LiteLLM provider plugin, not core, unless a small reusable SDK helper is strictly necessary.

Expected behavior: interactive LiteLLM setup prompts for the LiteLLM proxy base URL using the existing localhost default, collects/stores the API key with existing secret-ref/profile semantics, discovers selectable models from the configured OpenAI-compatible `/models` endpoint when reachable, and falls back to manual model entry when discovery fails or returns nothing. Preserve current non-interactive behavior, especially `--custom-base-url`, existing `models.providers.litellm.baseUrl` configs, and current image-generation provider behavior. Do not touch the unrelated `tools.media` LiteLLM alias validation topic from #54872.

Likely implementation files: `extensions/litellm/index.ts`, `extensions/litellm/onboard.ts`, `extensions/litellm/provider-catalog.ts`, and LiteLLM tests. Consider reusing `openclaw/plugin-sdk/provider-setup` helpers such as guarded OpenAI-compatible discovery, but keep LiteLLM-specific defaults and policy in the plugin. Update `docs/providers/litellm.md` and add a changelog entry because this changes user-facing onboarding behavior.

Validation: run targeted LiteLLM tests, targeted formatting on touched files, and the changed gate before handoff. Use Testbox for broad `pnpm check:changed` if it fans out beyond the targeted loop.
