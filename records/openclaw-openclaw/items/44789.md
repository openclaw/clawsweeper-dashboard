## Summary

Keep open: current main and v2026.6.6 still route named LiteLLM onboarding through API-key capture plus static localhost/default-model setup, while URL/model prompting exists only in the generic custom-provider path or the non-interactive LiteLLM flag path.

## What This Changes

The issue reports that LiteLLM onboarding in OpenClaw 2026.03.11 skipped server URL configuration and only offered one model during model selection.

## Best Possible Solution

Maintainers should choose a provider-owned LiteLLM interactive setup design that handles API key, base URL, and reachable/unreachable model discovery without duplicating the generic custom-provider flow.

## Reproduction Assessment

No high-confidence live reproduction was performed. Source inspection shows the named LiteLLM flow still applies a static localhost/default-model provider config, while custom base URL prompting is not part of that named interactive path.

## Solution Assessment

Unclear as a product decision. A LiteLLM-owned interactive URL/model-discovery wizard may be the right solution, but it should be designed against the existing custom-provider path instead of patched ad hoc.
