## Summary

Keep open. Current main and the latest release still do not provide an interactive LiteLLM server-URL prompt or dynamic model selection; the narrower non-interactive `--custom-base-url` bug was carried forward separately, but the reported interactive setup behavior still needs maintainer product/security direction.

## What This Changes

The issue requests that LiteLLM onboarding prompt for an API key and server URL, then discover and offer the proxy's models for default-model selection.

## Best Possible Solution

Maintainers should choose the provider-owned UX: either add guarded interactive LiteLLM remote-proxy URL capture and model discovery, or explicitly route remote-proxy users to Custom Provider, non-interactive setup, and manual config docs.

## Reproduction Assessment

Yes, source-level. Selecting the LiteLLM API-key auth method in current main prompts only for an API key and applies one static default model; I did not run the wizard because this review is read-only and cannot write credentials/config.

## Solution Assessment

Unclear. Restoring or adding an interactive LiteLLM URL/model-discovery flow is plausible, but the safer direction needs maintainer choice because the alternative is to keep remote proxy setup non-interactive/manual and document that boundary clearly.
