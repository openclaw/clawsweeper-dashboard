## Summary

Keep open. Current main still has LiteLLM interactive onboarding as an API-key-only provider path with one static default model; the separate non-interactive `--custom-base-url` fix landed, but the reported interactive proxy URL and model-discovery flow remains unresolved and needs a provider UX decision.

## What This Changes

The issue asks OpenClaw interactive onboarding to let LiteLLM users enter a proxy URL and discover/select models instead of defaulting to localhost with one static model.

## Best Possible Solution

Choose whether LiteLLM should use a provider-owned self-hosted setup/discovery flow or wait for the broader dynamic catalog design, then add guarded credentialed `/models` discovery with focused onboarding tests and docs.
