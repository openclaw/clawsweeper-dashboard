## Summary

Keep open. Current main still leaves the interactive LiteLLM onboarding path without a proxy URL prompt and seeds only the fallback model; the existing non-interactive `--custom-base-url` fix and broader proxy/dynamic-catalog threads do not fully resolve this report.

## What This Changes

Restore LiteLLM onboarding so interactive setup asks for the proxy URL/API key and offers configured or discovered LiteLLM models instead of defaulting to localhost with one model.

## Best Possible Solution

Add a provider-owned LiteLLM interactive setup path that prompts for the proxy base URL, preserves non-interactive `--custom-base-url`, and feeds the configured URL/API key into the provider catalog or allowlist flow with a safe fallback to the existing default model.

## Solution Assessment
