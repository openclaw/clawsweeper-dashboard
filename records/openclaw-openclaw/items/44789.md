## Summary

Keep open. Current main still only supports LiteLLM proxy URL injection through the non-interactive `--custom-base-url` path, while the interactive LiteLLM onboarding choice prompts for an API key and applies a static single-model preset; the related dynamic catalog work remains open and does not cover the LiteLLM interactive URL prompt by itself.

## What This Changes

The issue asks OpenClaw onboarding to restore LiteLLM setup that prompts for server URL/API key and discovers the proxy model list instead of falling back to one default model.

## Best Possible Solution

Maintainers should choose whether LiteLLM gets a provider-owned interactive remote proxy setup flow, then implement guarded URL capture and model discovery or explicitly direct remote-proxy users to Custom Provider, non-interactive setup, and manual config.

## Reproduction Assessment

Yes, source-level. Selecting the LiteLLM API-key auth method in current main prompts only for the API key and applies a static default LiteLLM config, while URL handling exists only in the non-interactive override.

## Solution Assessment

Unclear. Restoring an interactive URL/model discovery flow is plausible, but the maintainable fix needs provider/product/security direction because credentialed requests to user-configured `/models` endpoints overlap the dynamic catalog design.
