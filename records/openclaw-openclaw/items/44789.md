## Summary

Keep open: current main and the latest release still do not provide first-class LiteLLM interactive proxy URL/model discovery; they support generic custom-provider setup, manual config, and non-interactive LiteLLM `--custom-base-url` instead.

## What This Changes

The issue asks for LiteLLM interactive onboarding to prompt for proxy URL/API key and list discovered models instead of falling back to one default LiteLLM model.

## Best Possible Solution

Decide the LiteLLM interactive setup contract; if maintainers want first-class proxy onboarding, implement it in the LiteLLM-owned setup path using the existing `baseUrl` config shape and focused regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the current interactive LiteLLM path prompts for API key only, while base URL handling exists through manual config, Custom Provider, or non-interactive `--custom-base-url`.

## Solution Assessment

Unclear as a product choice. First-class LiteLLM interactive URL/model discovery may be the best user experience, but current supported paths already cover generic custom endpoints and non-interactive proxy setup.
