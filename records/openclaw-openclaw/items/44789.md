## Summary

Keep open. Current main has fixed the non-interactive LiteLLM `--custom-base-url` path, but the interactive LiteLLM onboarding choice still only runs the generic API-key prompt and then falls back to the default LiteLLM provider/model config; the requested model-listing behavior also overlaps an open dynamic-catalog design issue.

## What This Changes

The issue asks for LiteLLM onboarding to prompt for API key and server URL, then list LiteLLM proxy models for default-model selection.

## Best Possible Solution

Keep this open for maintainer direction: either add a LiteLLM provider-owned interactive base URL/key/model setup now and leave dynamic `/v1/models` discovery to https://github.com/openclaw/openclaw/issues/74481, or explicitly fold LiteLLM discovery into that catalog design.

Yes, source-level. Interactive LiteLLM setup still routes through the generic API-key auth method and default config applier, while only the non-interactive path reads `customBaseUrl`; I did not run the wizard because this review is read-only.

No close or automated fix yet. The URL prompt is provider-local and likely narrow, but the requested automatic model listing is a broader dynamic-catalog/product-policy decision already tracked in a separate open issue.
