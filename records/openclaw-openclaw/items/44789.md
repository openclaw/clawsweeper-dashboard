## Summary

Keep open. The shipped code now handles the narrower non-interactive `--custom-base-url` case, but current main and the latest release still do not provide an interactive LiteLLM server-URL prompt or dynamic proxy model selection, so the remaining work needs maintainer product/security direction rather than automatic cleanup.

## What This Changes

The issue requests that interactive LiteLLM onboarding ask for a server URL and discover/select proxy models instead of applying the default localhost/static model setup.

## Best Possible Solution

Maintainers should choose the provider-owned UX: either add guarded interactive LiteLLM URL capture and proxy model discovery, or make the wizard/docs explicitly route remote-proxy users to non-interactive `--custom-base-url`, Custom Provider, and manual config.

## Reproduction Assessment

Yes, source-level. Selecting the LiteLLM API-key path prompts only for an API key and the LiteLLM catalog exposes one static default model; I did not run the wizard because this review must not write config or credentials.

## Solution Assessment

Unclear. Restoring or adding interactive LiteLLM URL/model discovery is plausible, but the maintainable path needs maintainer choice because the alternative is to keep remote proxy setup non-interactive/manual and document that boundary clearly.
