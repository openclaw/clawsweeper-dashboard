## Summary

Keep open: current main and the latest release still route the named LiteLLM onboarding choice through API-key capture plus static localhost/default-model config, while the URL/model prompt path remains limited to the generic custom-provider flow or non-interactive LiteLLM flags. Related provider-catalog issues overlap model discovery, but none fully owns the LiteLLM interactive URL prompt gap.

## What This Changes

The issue reports that `openclaw onboard` no longer prompts for LiteLLM server URL configuration and only offers one model choice after selecting LiteLLM.

## Best Possible Solution

Maintainers should choose the intended named LiteLLM onboarding UX, then either add a LiteLLM-owned interactive base-URL/model setup path using existing custom-provider pieces where appropriate or explicitly document the current localhost/default-model shortcut as intentional.

## Reproduction Assessment

Yes at source level: current main routes only `custom-api-key` into URL/model prompting, while the LiteLLM choice runs shared API-key auth and static LiteLLM config. No live LiteLLM proxy flow was run, so this is not a full runtime reproduction.

## Solution Assessment

Unclear as a product decision. A provider-owned LiteLLM interactive wizard may be the right fix, but maintainers should choose that against the existing custom-provider and dynamic-catalog paths rather than patching a one-off prompt.
