## Summary

Keep open. Current main only covers the narrower non-interactive LiteLLM `--custom-base-url` path; the interactive LiteLLM choice still remains API-key/default-model oriented, so the requested URL prompt and model-discovery flow should not be swept closed as already implemented.

## What This Changes

Request interactive LiteLLM onboarding to collect the API key and server URL, then discover and offer LiteLLM proxy models for default model selection.

## Best Possible Solution

Decide whether LiteLLM should adopt the existing self-hosted provider setup pattern for interactive onboarding, or keep LiteLLM API-key-only and explicitly steer remote proxy users to Custom Provider or non-interactive/manual config.

## Reproduction Assessment

Yes from source for the missing interactive URL prompt, but not from a live wizard run. The current LiteLLM interactive method prompts for the API key only, while `customBaseUrl` is consumed only in the non-interactive override.

## Solution Assessment

Unclear. Reusing the self-hosted provider setup helper is the likely maintainable direction, but maintainers should first decide whether LiteLLM should gain a dedicated interactive URL/model flow or continue relying on Custom Provider/non-interactive setup for remote proxies.
