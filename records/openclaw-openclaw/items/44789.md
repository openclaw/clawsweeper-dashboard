## Summary

Keep open. Current main and the latest release handle the narrower non-interactive LiteLLM `--custom-base-url` path, but interactive LiteLLM onboarding still captures only an API key and exposes a static single-model catalog, so the remaining work needs maintainer product/security direction rather than cleanup.

## What This Changes

The issue asks for LiteLLM onboarding to prompt for the proxy URL/API key and offer models discovered from the configured LiteLLM server.

## Best Possible Solution

Maintainers should choose the provider-owned UX: add guarded interactive LiteLLM URL capture and proxy model discovery, or make the wizard/docs explicitly route remote-proxy users to non-interactive `--custom-base-url`, Custom Provider, and manual config.
