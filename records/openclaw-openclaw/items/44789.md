## Label Justifications

- `P2`: Remote LiteLLM users can be silently routed to localhost during onboarding, but the failure is provider-scoped and has direct-config, non-interactive, and Custom Provider workarounds.
- `impact:auth-provider`: The missing URL step affects provider endpoint selection, model discovery, and default-model routing.

## Summary

This issue remains necessary: current main supports remote LiteLLM URLs through direct config, non-interactive `--custom-base-url`, or the generic Custom Provider wizard, but the dedicated interactive LiteLLM choice still prompts only for an API key and otherwise defaults to localhost with one preset model. The supported Custom Provider path is a useful workaround, not equivalent to the reported LiteLLM-specific workflow.

## What This Changes

Restore proxy URL entry and endpoint-backed model discovery in the dedicated interactive LiteLLM onboarding flow.

## Best Possible Solution

Keep LiteLLM-specific setup provider-owned and add an interactive step that confirms the localhost default or captures a remote proxy URL before credentials and catalog discovery, preserving existing configured URLs and adding focused interactive regression coverage.

## Reproduction Assessment

Yes at source level: selecting the dedicated LiteLLM interactive auth flow captures an API key but no proxy URL, after which an unconfigured provider defaults to `http://localhost:4000` and one preset model. A live CLI transcript was not produced during this read-only review.

## Solution Assessment

The best technical boundary is a provider-local interactive endpoint step, but maintainers must choose whether it should confirm localhost by default or steer remote proxy users toward the generic Custom Provider flow.
