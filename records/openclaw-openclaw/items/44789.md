## Summary

Keep open. Current main and the latest release only cover the narrower non-interactive LiteLLM custom-base-url path; interactive LiteLLM onboarding still prompts only for an API key and still exposes a static one-model catalog, so the reported setup/model-selection gap remains unresolved and needs a maintainer UX/security decision.

## What This Changes

The issue asks to restore LiteLLM interactive onboarding so users can enter the proxy URL and API key, then choose from models discovered from that proxy instead of a single static default.

## Best Possible Solution

Choose a LiteLLM-owned self-hosted setup/discovery flow or the broader dynamic catalog seam, then implement guarded credentialed `/models` discovery and interactive URL handling in the LiteLLM plugin with focused onboarding tests and docs.

## Reproduction Assessment

Yes, source-level. Current LiteLLM interactive auth prompts only for an API key while the onboarding preset and catalog expose one static default model; I did not run the wizard live.

## Solution Assessment

Unclear. The requested behavior is plausible, but maintainers need to choose whether it belongs in a LiteLLM-specific self-hosted setup flow or the broader dynamic catalog design tracked at https://github.com/openclaw/openclaw/issues/74481.
