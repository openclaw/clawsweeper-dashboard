## Summary

Keep open. Current main still has a source-level LiteLLM onboarding gap: the interactive LiteLLM auth path collects only the API key, defaults the provider URL to localhost when no prior config exists, and exposes the static one-model LiteLLM catalog unless a base URL was already configured or passed through the non-interactive path.

## What This Changes

Interactive LiteLLM onboarding should prompt for the LiteLLM proxy URL/API key and present discovered or configured models instead of defaulting to localhost with a single model.

## Best Possible Solution

Keep the fix in the LiteLLM provider plugin: collect or retain the proxy base URL during interactive onboarding, use the configured endpoint for model selection, and fall back to manual model entry without adding core LiteLLM special cases.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the interactive LiteLLM path only prompts for the API key, while base URL override is wired only through non-interactive setup. I did not run `openclaw onboard` because this review is read-only and onboarding writes config.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-owned LiteLLM interactive setup using the existing self-hosted/OpenAI-compatible provider seams, not a core special case.
