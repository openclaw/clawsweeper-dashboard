## Summary

Keep open. Current main and the latest release still route interactive LiteLLM onboarding through an API-key-only prompt with a static default model, while remote proxy URL handling is only documented and wired for non-interactive/manual setup.

## What This Changes

The issue asks to restore LiteLLM interactive onboarding so users can enter a proxy URL/API key and choose from the proxy model list instead of getting one static model.

## Best Possible Solution

Maintainers should decide whether LiteLLM owns an interactive remote-proxy setup flow, then either implement guarded URL capture and model discovery there or explicitly route remote-proxy users to Custom Provider, non-interactive setup, and manual config.

## Reproduction Assessment

Yes, source-level. Selecting the LiteLLM API-key auth method in current main prompts only for the API key and applies a static default model preset; base URL injection is wired only through the non-interactive override.

## Solution Assessment

Unclear. Restoring an interactive LiteLLM URL/model discovery flow is plausible, but the maintainable fix needs provider/product/security direction because it would introduce credentialed requests to user-configured proxy endpoints.
