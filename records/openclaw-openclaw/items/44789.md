## Summary

Keep open. Current main still routes LiteLLM interactive onboarding through the generic API-key-only path, preserves or defaults the proxy URL to localhost, and exposes a single static LiteLLM model rather than prompting for the proxy URL and discovering proxy models.

## What This Changes

The issue asks for LiteLLM onboarding to prompt for API key and server URL, then list available LiteLLM proxy models for default selection.

## Best Possible Solution

Fix this in the bundled LiteLLM provider plugin by adding a provider-owned setup path that prompts for proxy base URL, captures the API key, discovers OpenAI-compatible proxy models when reachable, and falls back to manual model entry without moving LiteLLM policy into core.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the fresh interactive LiteLLM onboard path uses the generic API-key method, has no base URL prompt, and seeds localhost plus one static model. I did not run the live CLI because this review is read-only and pnpm is unavailable in the checkout.

## Solution Assessment

Yes. The narrow maintainable fix is to wire LiteLLM into the existing provider-owned self-hosted setup and discovery seams, matching vLLM/SGLang patterns while retaining LiteLLM-specific defaults and guardrails.
