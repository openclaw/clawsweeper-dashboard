Keep open: current main covers a related non-interactive LiteLLM `--custom-base-url` path, but the interactive LiteLLM onboarding path still only prompts for the API key and then applies the default localhost/single-model config, so the reported URL/model-selection gap remains plausible.

## What This Changes

The issue requests LiteLLM onboarding to prompt for API key and proxy URL, then present discovered LiteLLM models instead of falling back to one model.

## Best Possible Solution

Add provider-owned LiteLLM interactive setup that asks for the proxy URL and API key, discovers `/v1/models` when safely reachable, falls back to manual/default model entry when needed, and preserves existing config.

## Solution Assessment
