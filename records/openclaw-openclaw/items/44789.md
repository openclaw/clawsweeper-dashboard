## Summary

Keep this open for maintainer judgment: current main still does not prompt for a LiteLLM base URL in the interactive LiteLLM auth choice, but the shipped docs and release history I checked do not prove that exact prompt was an established contract rather than a requested provider-onboarding improvement.

## What This Changes

The issue asks for LiteLLM onboarding to prompt for API key and server URL, then discover and show the proxy’s available models instead of falling back to one default model.

## Best Possible Solution

Maintainers should decide whether the bundled LiteLLM auth choice should use the self-hosted provider setup pattern; if yes, implement it inside the LiteLLM plugin with focused tests for interactive base URL, API key, model discovery, and the existing non-interactive path.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence path that the interactive LiteLLM auth choice prompts only for an API key and then falls back to the default provider config; I did not execute a live wizard session or LiteLLM proxy.

## Solution Assessment

Unclear. Reusing the provider-owned self-hosted setup helpers looks like the narrowest code direction, but maintainers should first decide whether the LiteLLM-specific wizard should grow that prompt or keep remote proxy setup in Custom Provider/non-interactive docs.
