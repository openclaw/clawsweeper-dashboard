## Summary

Keep open. Current main and the latest release still route interactive LiteLLM onboarding through an API-key-only prompt and static default model config; the only bundled LiteLLM base-URL injection is the non-interactive `--custom-base-url` path, while the broader dynamic catalog work remains open and does not fully cover this interactive LiteLLM setup report.

## What This Changes

The issue asks OpenClaw to restore interactive LiteLLM onboarding that prompts for the proxy URL and discovers selectable models instead of defaulting to a single static LiteLLM model.

## Best Possible Solution

Maintainers should decide whether LiteLLM owns an interactive remote-proxy setup flow, then either implement guarded URL capture and model discovery there or explicitly route remote-proxy users to Custom Provider, non-interactive setup, and manual config.

## Reproduction Assessment

Yes, source-level. Selecting the LiteLLM API-key auth method in current main prompts only for the API key and applies a static default config; base URL handling is wired only into the non-interactive override.

## Solution Assessment

Unclear. Restoring an interactive LiteLLM URL/model discovery flow is plausible, but the maintainable fix needs provider/product/security direction because it would introduce credentialed requests to user-configured proxy endpoints.
