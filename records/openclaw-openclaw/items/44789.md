## Summary

Keep open: current main and v2026.6.6 still route named LiteLLM onboarding through API-key capture plus static localhost/default-model setup, while URL/model prompting exists only in the generic custom-provider path or non-interactive LiteLLM flag path. The remaining question is a maintainer UX/product choice, not a cleanup close.

## What This Changes

The issue reports that LiteLLM onboarding in OpenClaw 2026.03.11 skipped server URL/API-key setup and only offered one model instead of discovering LiteLLM models.

## Best Possible Solution

Maintainers should choose a provider-owned LiteLLM interactive setup design, then cover API key, base URL, and reachable/unreachable model-discovery behavior without duplicating the custom-provider flow.

## Reproduction Assessment

No high-confidence live reproduction was performed. Source inspection shows the named LiteLLM flow only captures an API key and applies a localhost/static-model default, while URL/model prompting lives in another path.

## Solution Assessment

Unclear as a product decision. A LiteLLM-owned interactive URL/model-discovery wizard may be the right UX, but it should be designed against the existing custom-provider setup path instead of patched ad hoc.
