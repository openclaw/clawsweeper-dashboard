## Summary

Keep open. Current main still sends interactive LiteLLM onboarding through a generic API-key-only path, while URL handling is non-interactive-only and the catalog still exposes one static model.

## What This Changes

LiteLLM interactive onboarding should ask for the proxy URL and API key, then offer discovered proxy models instead of falling back to localhost plus one static model.

## Best Possible Solution

Fix this inside the bundled LiteLLM provider by giving it a provider-owned interactive setup flow for base URL, API key, and model selection while keeping generic core/plugin SDK seams reusable.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: selecting the LiteLLM wizard path runs an API-key-only auth method, while URL handling is non-interactive-only and the catalog contains one static model. I did not run the live CLI.

## Solution Assessment

Yes. The narrow maintainable fix is provider-owned LiteLLM setup/discovery, likely reusing the self-hosted OpenAI-compatible helper patterns without moving LiteLLM policy into core.
