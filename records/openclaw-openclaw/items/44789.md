## Summary

Keep open: current main and v2026.6.6 still split LiteLLM setup between API-key-only interactive onboarding, non-interactive `--custom-base-url`, manual config, and generic Custom Provider; first-class interactive LiteLLM URL/model discovery remains a maintainer product decision.

## What This Changes

The issue asks for `openclaw onboard` LiteLLM setup to prompt for server URL/API key and list proxy-discovered models instead of falling back to one default model.

## Best Possible Solution

Keep the current documented split unless maintainers intentionally add a LiteLLM-owned interactive setup flow using the existing `models.providers.litellm.baseUrl` shape and focused regression coverage.

## Reproduction Assessment

No live reproduction was run. Source inspection gives a medium-confidence source repro: interactive LiteLLM auth only prompts for API key, while fresh config falls back to localhost plus one default model unless base URL is supplied through another path.

## Solution Assessment

Unclear as a product choice. First-class LiteLLM interactive URL/model discovery may be the best UX, but current supported paths already cover generic custom endpoints, manual config, and non-interactive proxy setup.
