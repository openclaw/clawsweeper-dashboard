## Summary

Keep open. Current main still has the reported LiteLLM onboarding gap: the interactive LiteLLM path captures only an API key, remote URL setup is routed through non-interactive/manual config, and the provider catalog remains seeded from one static default model rather than a LiteLLM proxy model list.

## What This Changes

The issue asks for LiteLLM onboarding to prompt for proxy URL/API key and show available proxy models instead of defaulting to localhost with one static model option.

## Best Possible Solution

Maintainers should decide whether LiteLLM gets a provider-owned interactive proxy URL and model discovery flow now, or whether this should wait for the broader configured-provider dynamic catalog design.

## Reproduction Assessment

Yes, source-level. Current LiteLLM interactive auth prompts only for an API key, while the onboarding preset and provider catalog expose one static default model; I did not run the wizard in this read-only review.

## Solution Assessment

Unclear. Interactive LiteLLM URL/model discovery is plausible, but the maintainable solution depends on whether maintainers want a LiteLLM-specific UX or the broader dynamic catalog path tracked in https://github.com/openclaw/openclaw/issues/74481.
