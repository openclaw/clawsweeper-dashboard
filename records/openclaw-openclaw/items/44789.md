## Summary

Keep open: source inspection on current main and v2026.6.10 shows the named LiteLLM auth path still prompts only for an API key, applies localhost plus one default model when no base URL already exists, and leaves URL/model setup to Custom Provider or non-interactive flags.

## What This Changes

The issue requests restoring interactive LiteLLM onboarding so users can enter a proxy URL/API key and choose discovered models from that LiteLLM endpoint.

## Best Possible Solution

Restore a safe LiteLLM interactive setup path that captures the proxy URL before model selection, preserves existing base URLs, and has focused coverage for localhost default, remote proxy, and model-picker behavior.

## Reproduction Assessment

Yes, at source level: selecting litellm-api-key enters the generic API-key path with no base-URL prompt, then LiteLLM config preserves an existing URL or falls back to localhost and one default model. I did not run a live interactive TTY/LiteLLM proxy session.

## Solution Assessment

Unclear. A LiteLLM-specific URL prompt may be the right fix, but maintainers should decide whether the durable shape is provider-specific prompt metadata, reuse of the Custom Provider flow, or an explicit live-discovery policy.
