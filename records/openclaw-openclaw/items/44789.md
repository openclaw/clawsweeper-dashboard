## Summary

Keep open. Current main and the latest release still do not provide an interactive LiteLLM base URL prompt or dynamic LiteLLM proxy model discovery; the remaining model-list portion overlaps the open dynamic-catalog design item at https://github.com/openclaw/openclaw/issues/74481, but that does not cover the LiteLLM-specific onboarding UX decision by itself.

## What This Changes

The issue requests interactive LiteLLM onboarding that captures API key and server URL, then lists models from the configured LiteLLM proxy for default-model selection.

## Best Possible Solution

Maintainers should choose a provider-owned UX: either add guarded interactive LiteLLM URL capture and proxy model discovery aligned with the dynamic catalog design, or explicitly keep remote proxy setup on Custom Provider, non-interactive, and manual config paths.

## Reproduction Assessment

Yes, source-level. Current LiteLLM interactive auth prompts only for an API key, while the LiteLLM provider catalog and onboarding preset expose one static default model; I did not run the wizard because this review is read-only.

## Solution Assessment

Unclear. Interactive LiteLLM URL/model discovery is plausible, but the maintainable solution depends on whether maintainers want a LiteLLM-specific UX or the broader configured-provider dynamic catalog path in https://github.com/openclaw/openclaw/issues/74481.
