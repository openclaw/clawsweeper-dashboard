Keep open. Current main and the latest release only cover the narrower non-interactive LiteLLM `--custom-base-url` path; the interactive LiteLLM onboarding choice still prompts for an API key and applies the static default LiteLLM model/config, so this is not closeable as implemented.

## What This Changes

The issue asks for interactive LiteLLM onboarding to collect API key, server URL, and usable model choices instead of skipping URL configuration and showing only one model.

## Best Possible Solution

Maintainers should decide whether LiteLLM should gain a provider-owned self-hosted interactive setup flow, or whether remote proxy users should be directed to Custom Provider, non-interactive `--custom-base-url`, or manual config.

## Reproduction Assessment

Yes, source-level. The current LiteLLM interactive auth method prompts only for the API key and then applies static LiteLLM config; the URL injection exists only in the non-interactive override.

## Solution Assessment

Unclear. Reusing the shared self-hosted provider setup pattern is plausible, but maintainers should first choose whether LiteLLM interactive onboarding should own URL/model discovery or stay API-key-only with Custom Provider/non-interactive/manual setup as the supported remote-proxy paths.
