## Summary

Keep open. Current main and the latest release fixed the narrower non-interactive `--custom-base-url` path, but interactive LiteLLM onboarding still prompts only for the API key and exposes a static one-model catalog, so the reported LiteLLM setup/model-selection gap remains a maintainer UX decision.

## What This Changes

Interactive LiteLLM onboarding should collect the proxy URL and present models discovered from the LiteLLM proxy instead of falling back to one static model.

## Best Possible Solution

Choose either a LiteLLM-owned self-hosted setup/discovery flow or the broader dynamic catalog seam, then implement guarded credentialed `/models` discovery and interactive URL handling in the LiteLLM plugin with focused onboarding tests.

## Reproduction Assessment

Yes, source-level. Current LiteLLM interactive auth prompts only for an API key while the onboarding preset and catalog expose one static default model; I did not run the wizard live.

## Solution Assessment

Unclear. The requested behavior is plausible, but maintainers need to choose whether it belongs in a LiteLLM-specific self-hosted setup flow or the broader dynamic catalog design tracked at https://github.com/openclaw/openclaw/issues/74481.
