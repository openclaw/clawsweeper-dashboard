## Summary

Keep open. Current main still has a LiteLLM interactive onboarding path that prompts for the API key, applies the localhost default base URL, and seeds one default model; the existing non-interactive `--custom-base-url` support and the broader closed proxy issue do not resolve this interactive URL/model-list report.

## What This Changes

The issue requests LiteLLM onboarding to prompt for API key and server URL, then use that endpoint's model list instead of falling back to one default model.

## Best Possible Solution

Add provider-owned interactive LiteLLM setup that asks for the proxy base URL with the localhost default, preserves existing and non-interactive behavior, and feeds that URL into the model catalog/allowlist step.

## Reproduction Assessment

Yes, from source inspection: a fresh interactive LiteLLM auth choice runs only the API-key auth method, then applies `http://localhost:4000` and one default model. I did not execute the wizard, so this is source-reproducible rather than an observed terminal repro.

## Solution Assessment

Yes, the narrowest maintainable fix is in the LiteLLM provider setup path: prompt for base URL before applying `applyLitellmConfig`, then let the existing provider catalog/allowlist flow use that config. Closing as duplicate of the broader proxy issue would lose the specific interactive LiteLLM gap.

## Repair Work Prompt

Fix the LiteLLM interactive onboarding regression reported in https://github.com/openclaw/openclaw/issues/44789. In the LiteLLM provider plugin, prompt for the LiteLLM proxy base URL during interactive `litellm-api-key` setup, defaulting to `http://localhost:4000` and preserving an existing configured `models.providers.litellm.baseUrl`. Feed that base URL into `applyLitellmConfig` before model selection so the provider catalog/allowlist can use the selected endpoint instead of silently using localhost and the single default model. Preserve the existing non-interactive `--custom-base-url` behavior and do not change generic custom-provider onboarding. Add focused tests for interactive base URL handling plus the existing non-interactive case; add a single-line `CHANGELOG.md` fix entry because this is user-visible.
