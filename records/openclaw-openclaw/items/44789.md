## Summary

Keep open. Current main still routes interactive LiteLLM onboarding through an API-key-only provider path, so the reported skipped proxy URL/model discovery behavior remains valid and narrow.

## What This Changes

The issue asks for LiteLLM onboarding to prompt for proxy URL/API key and offer discovered proxy models instead of defaulting to localhost with one static model.

## Best Possible Solution

Fix this in the bundled LiteLLM provider plugin by adding a provider-owned interactive setup path that prompts for proxy base URL, captures the API key, discovers OpenAI-compatible proxy models when reachable, and falls back to manual model entry without moving LiteLLM policy into core.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the fresh interactive LiteLLM auth path only prompts for an API key, while base URL handling is non-interactive-only and the catalog exposes one static model. I did not run the live CLI in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to wire LiteLLM into provider-owned self-hosted setup/discovery behavior, similar to vLLM/SGLang patterns, while keeping shared core seams generic.
