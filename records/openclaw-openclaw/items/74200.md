## Summary

Keep this PR open. Current main still has the destructive sandbox skill sync behavior the PR is trying to fix, but the proposed replacement introduces stale sandbox skill state: previously synced skills can survive after they fall out of the effective manifest, and deleted files inside a source skill can persist because the skip check only compares newest mtimes.

## What This Changes

The PR changes `syncSkillsToWorkspace` from deleting and recopying the whole target `skills` directory to an incremental per-skill sync with newest-mtime skips, debug counts, and regression tests for preserving and recopying skill directories.

## Best Possible Solution

Keep the data-loss fix intent, but use an authoritative owned-sync strategy: distinguish OpenClaw-owned copied skill directories from user-created target content, prune owned dirs that disappear from a successfully built manifest, copy each owned skill with full tree semantics or a reliable content manifest, and add regression coverage for allowlist removal, eligibility changes, source file deletion, and failed copy cleanup.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to sync a skill once, then either remove it from the effective allowlist or delete the newest source file and sync again; the proposed branch can leave the old skill directory or deleted file in the target. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR removes the whole-directory wipe, but preserving every non-manifest target directory and skipping by newest mtime is not the narrowest maintainable fix for sandbox skill staging; an owned-manifest sync with full-tree replacement semantics is safer.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Prune stale synced skills when the manifest changes:** `src/agents/skills/workspace.ts:968-976`
  - body: Replacing the target wipe with `mkdir` and leaving all non-manifest directories untouched means a sandbox can keep an OpenClaw-synced skill after allowlists, config gates, plugin enablement, or remote eligibility remove it. The sync needs to distinguish user-created target content from owned synced skill dirs and prune the owned dirs that are no longer in a successful manifest.
  - confidence: 0.87
- **[P2] Avoid mtime-only skips for deleted source files:** `src/agents/skills/workspace.ts:1017-1019`
  - body: The `destMtime >= srcMtime` shortcut misses source deletions: if the source skill deletes its newest file, the destination still has that newer file, so the next sync skips and the stale file remains. Use a content/tree manifest or recopy the owned skill directory when tree shape changes.
  - confidence: 0.92
