## Summary

Keep open. The PR addresses a real destructive sync path, but the proposed incremental sync leaves stale owned skills and deleted source files in sandbox targets, and it still lacks external after-fix real behavior proof.

## What This Changes

The PR changes `syncSkillsToWorkspace` from a whole-target wipe and full recopy to an incremental per-skill sync with newest-mtime skips and adds regression tests for preserved target skills, skipped unchanged copies, and modified-source recopying.

## Best Possible Solution

Keep the non-destructive data-loss fix direction, but implement an owned-manifest sync that preserves unknown user-created target content while pruning OpenClaw-owned stale skills and detecting full tree changes, coordinated with the open failed-copy cleanup PR.

## Reproduction Assessment

Yes, by source inspection of the proposed branch. Sync a skill once, then remove it from the effective allowlist or delete the newest source file; the next sync can leave the old directory or deleted file in the target.

## Solution Assessment

No. Avoiding the whole-target wipe is the right direction, but preserving every non-manifest directory and using newest mtime as the freshness contract is not the narrowest maintainable fix.
