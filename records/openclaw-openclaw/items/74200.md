## Summary

Keep open. The data-loss goal is valid, but this PR is not safe to merge because it can preserve no-longer-eligible sandbox skills and the newest-mtime shortcut misses source deletions; both issues were also called out in the existing bot review discussion.

## What This Changes

The PR changes `syncSkillsToWorkspace` from whole-directory deletion and recopy to non-destructive per-skill mtime-based syncing, with regression tests for preserved, skipped, and updated skill directories.

## Best Possible Solution

Keep the non-destructive intent, but implement an owned-manifest sync that preserves true user-created target content while pruning stale OpenClaw-owned skill dirs and using full-tree replacement or content comparison for changed skills.

## Reproduction Assessment

Yes. Sync a skill once, then remove it from the effective allowlist or delete the newest file from the source skill and sync again; the proposed branch can leave the old skill directory or deleted file in the target.

## Solution Assessment

No. Avoiding the whole-target wipe is directionally right, but preserving every non-manifest directory and relying on newest mtime is not the safest maintainable sandbox sync contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prune stale owned skills when the manifest changes:** `src/agents/skills/workspace.ts:968-976`
  - body: Replacing the target wipe with `mkdir` leaves directories outside the current manifest untouched. Since sandbox sync is filtered by allowlists, config gates, plugin enablement, and remote eligibility, a previously copied skill can remain available after it is no longer eligible. Track and prune OpenClaw-owned dirs while preserving unknown user content.
  - confidence: 0.88
- **[P2] Avoid newest-mtime skips for tree deletions:** `src/agents/skills/workspace.ts:1017-1019`
  - body: The `destMtime >= srcMtime` shortcut misses source deletions: if the newest source file is removed, the destination still has that file and a newer mtime, so the next sync skips and the stale file remains indefinitely. Use a content/tree manifest or recopy owned skill directories when tree shape changes.
  - confidence: 0.92
