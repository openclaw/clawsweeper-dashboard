## Summary

Keep open: the underlying skill-sync data-loss problem is still present on current main, but this branch is not merge-ready because it targets a removed runtime path, changes sandbox skill cleanup semantics in a way that can preserve stale or ineligible skills, and lacks real behavior proof.

## What This Changes

The branch changes the old workspace skill sync helper from whole-directory deletion to incremental per-skill copying with mtime-based skips and adds regression tests for preserving target skill directories.

## Best Possible Solution

Rework the active src/skills/loading sync path so genuine user-owned skill data is protected while stale managed, removed, disabled, or ineligible materialized skill copies are pruned, then prove deletion, eligibility, and real sync behavior.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main still loads skill entries and deletes target skill children before copying the current manifest, which matches the data-loss mechanism when the loaded entry set is empty or partial.

## Solution Assessment

No as submitted. The non-destructive direction is plausible, but the best fix must land on src/skills/loading/workspace.ts and distinguish user-owned target data from stale managed or ineligible skill copies.
