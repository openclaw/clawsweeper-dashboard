## Summary

Keep open. The reported destructive sync behavior is real on current main, but this PR’s replacement sync can leave stale owned skill content in sandbox targets and can miss source-tree deletions, so it is not safe to merge as-is.

## What This Changes

The PR changes `syncSkillsToWorkspace` to preserve the target skills directory, sync per skill with newest-mtime skips, add debug counters, and add regression tests for preserved, skipped, and updated skill directories.

## Best Possible Solution

Keep the non-destructive data-loss direction, but implement an owned-manifest sync that preserves unknown user-created target directories while pruning OpenClaw-owned stale dirs and detecting file-set deletions before skipping copies.

## Reproduction Assessment

Yes, by source inspection. On the PR branch, sync a skill once and then either remove it from the effective manifest or delete the newest source file; the next sync can leave stale owned content in the target.

## Solution Assessment

No. Avoiding a whole-directory wipe is the right direction, but preserving every non-manifest directory and using newest mtime as the freshness contract is not the narrowest maintainable fix for sandbox skill sync.
