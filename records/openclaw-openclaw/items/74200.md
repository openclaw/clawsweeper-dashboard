## Summary

Keep open. The PR addresses a real destructive-sync problem on current main, but the proposed replacement introduces stale sandbox skill retention and an incomplete mtime skip heuristic that should block merge.

## What This Changes

The PR replaces whole-directory skill target deletion with per-skill incremental copying and adds regression tests for preserving existing target skills, skipping unchanged copies, and recopying modified sources.

## Best Possible Solution

Keep the no-wholesale-wipe direction, but implement an owned-manifest sync that preserves true user-created target content while pruning stale OpenClaw-owned skill copies and replacing or fingerprinting full skill trees when their contents change.

## Reproduction Assessment

Yes. Sync a skill once, then remove it from the effective source manifest through plugin/config/eligibility changes and sync again; the proposed branch leaves the old target directory. A second reproduction is to sync a skill, delete the newest source file, and sync again; the mtime skip can leave the deleted file in the destination.

## Solution Assessment

No. Avoiding the whole-directory wipe is directionally right, but preserving every non-manifest directory and relying on newest mtime is not the narrowest maintainable sync contract for sandbox skills.

## Review Findings

- **[P1] Prune stale OpenClaw-owned skill dirs:** `src/agents/skills/workspace.ts:968-976`

A skill copied by a previous run can remain after plugin enablement, remote eligibility, or the effective skill set removes it, and the sandbox can later see that stale copy as a workspace skill.
