## Summary

Keep this PR open: it addresses a real destructive skill-sync problem in current main, but the proposed branch is not safe to merge because it can leave previously eligible skills and deleted source files available in sandbox targets after the effective skill manifest changes. Current docs and tests treat agent allowlists, config gates, and eligibility filters as authoritative, and sandbox setup uses the synced target as part of the sandbox tool workspace surface.

## What This Changes

The PR changes `syncSkillsToWorkspace` from deleting and recopying the whole target `skills` directory to an incremental per-skill sync with newest-mtime skips, debug counts, and regression tests for preserving and recopying skill directories.

## Best Possible Solution

Keep the data-loss fix intent, but implement an authoritative owned-sync strategy: build a successful eligible manifest, distinguish OpenClaw-owned synced skill directories from user-created target content, prune previously synced owned skills that disappear from the manifest, replace each copied skill directory atomically or after a full tree comparison, and add regression coverage for allowlist removal, eligibility changes, source file deletion, and failed copy cleanup.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to sync a skill once, then either remove it from the effective allowlist or delete a newer file from the source skill and sync again; the proposed branch can leave the old skill directory or deleted file in the target. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR fixes the most obvious whole-directory wipe, but preserving all non-manifest target directories and skipping by newest mtime is not the narrowest maintainable solution for sandbox skill staging; an owned-manifest sync with full-tree semantics is safer.

  - body: The `destMtime >= srcMtime` shortcut misses source deletions: if a newer file is deleted from the source, the destination still has that newer file, so the next sync skips and the stale file remains indefinitely. Use a content/tree manifest or recopy the owned skill directory when the tree shape changes.
