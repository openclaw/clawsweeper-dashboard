## Summary

Keep this PR open: it targets a real destructive-sync risk in `syncSkillsToWorkspace`, but the proposed implementation is not safe to merge because it can preserve stale or no-longer-eligible skill content in sandbox targets. The same concerns were already raised in the PR discussion, and current `main` still uses the destructive authoritative sync path, so this is neither obsolete nor implemented on main.

## What This Changes

The PR changes `syncSkillsToWorkspace` from wholesale target directory deletion to per-skill incremental syncing with mtime-based skips, and adds tests for preserving target skill directories and recopying modified skills.

## Best Possible Solution

Replace the wholesale wipe with an owned, authoritative sync strategy: only update the directories OpenClaw owns, prune previously synced owned skills that are absent from a successfully built eligible manifest, preserve user-owned target directories only under an explicit documented contract, and use full-tree copy or content-manifest comparison instead of newest-mtime alone.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to sync a skill with two files, delete the newer source file, and sync again; the PR's `destMtime >= srcMtime` skip leaves the deleted file in the target. A second reproduction is to sync an allowed skill, remove it from the effective allowlist or eligibility, and observe that the PR preserves the old target directory.

## Solution Assessment

No. The PR addresses a real data-loss class, but preserving all non-manifest target directories and relying on newest file mtime is not the narrowest maintainable solution for sandbox skill staging.

- **[P1] Prune stale skills when the manifest changes:** `src/agents/skills/workspace.ts:968-976`
  - body: Replacing the target wipe with `mkdir` and never removing non-manifest directories means a sandbox can keep a skill after agent allowlists, plugin config, or remote eligibility remove it. The sync path needs an owned-manifest cleanup strategy so stale skills do not remain available.
  - confidence: 0.87

  - body: The `destMtime >= srcMtime` shortcut misses deletions: after copying files A and B, deleting newer source file B lowers the source newest mtime while the destination still has B, so the next sync skips and leaves B in the sandbox copy.
  - confidence: 0.92
