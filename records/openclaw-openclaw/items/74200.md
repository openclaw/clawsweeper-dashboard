## Summary

Keep open. The PR fixes a real destructive-sync risk directionally, but the proposed incremental sync should not merge as-is because it can retain stale or no-longer-eligible sandbox skills and can miss source file deletions.

## What This Changes

The PR changes `syncSkillsToWorkspace` to a non-destructive per-skill incremental sync with newest-mtime skips, debug counts, and new regression tests for preserved, skipped, and updated skill directories.

## Best Possible Solution

Keep the non-destructive intent, but implement an owned-manifest sync that preserves unknown user-created target content while pruning OpenClaw-owned stale skill dirs and replacing or comparing full trees when file sets change, coordinated with #72254.

## Reproduction Assessment

Yes, by source inspection. In the proposed branch, sync a skill once, then remove it from the effective allowlist or delete the newest source file; the next sync can leave the old directory or deleted file in the target.

## Solution Assessment

No. Avoiding the whole-target wipe is the right direction, but preserving every non-manifest directory and using newest mtime as the freshness contract is not the narrowest maintainable fix.
