## Summary

Keep open: this is a useful fix direction for a real Telegram reconnect-drain bug, but the PR is not merge-ready because the 30s cooldown may expire by the next normal long-poll success, the public Plugin SDK helper return type changes without a settled compatibility contract, and the supplied proof is build/grep output rather than runtime evidence.

## What This Changes

The PR changes reconnect delivery draining to return matched/drained/skipped counts and makes Telegram polling cooldown reconnect drains when every matched delivery is already in flight.

## Best Possible Solution

Land a Telegram drain suppression that spans at least one full long-poll cycle or tracks the claimed entry directly, while preserving or explicitly documenting the Plugin SDK contract and adding redacted runtime proof that the repeated drain log stops.

## Reproduction Assessment

Yes. The linked issue has concrete runtime logs, and current source shows poll-success invokes reconnect drain while the delivery queue skips entries held by withActiveDeliveryClaim, which explains the repeated 'already being recovered' loop without needing a speculative path.

## Solution Assessment

No. The approach is plausible, but as written the cooldown matches the normal long-poll interval and the SDK return-type expansion needs an explicit compatibility path or a narrower internal seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use a cooldown that survives the next long poll:** `extensions/telegram/src/polling-session.ts:125`
  - body: This sets the reconnect-drain cooldown to exactly 30,000 ms, but poll-success is emitted after each getUpdates success and Telegram's default long-poll timeout is 30 seconds. The next empty poll will usually finish at or after the cooldown boundary, so the same in-flight delivery can be drained and logged again on the next poll, leaving the reported every-long-poll loop mostly intact. Make the suppression last beyond at least one full long-poll cycle or tie it to the in-flight entry/claim release, and add a regression that advances past one normal poll interval.
  - confidence: 0.87
- **[P1] Preserve the public drainPendingDeliveries contract:** `src/plugin-sdk/delivery-queue-runtime.ts:28`
  - body: This public SDK subpath currently resolves a Promise<void> and is exported/documented for plugins, but the patch changes it to Promise<ReconnectDrainResult> to serve a Telegram-specific signal. That makes a local fix part of the third-party SDK contract without docs, API baseline, or maintainer-visible migration discussion. Keep the SDK return type compatible and expose the result through a narrower compatible seam, or intentionally update the public SDK contract artifacts and migration guidance.
  - confidence: 0.84
