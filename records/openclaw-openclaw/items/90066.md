## Summary

Keep open: the PR targets a real Telegram reconnect-drain loop, but the current patch still has a blocking timing issue, changes a public Plugin SDK return contract without compatibility handling, and does not yet include real Telegram/log proof of the behavior stopping.

## What This Changes

The PR makes reconnect drains return matched/drained/skipped counts and uses those counts in Telegram polling to add a cooldown when all matched deliveries are already in flight.

## Best Possible Solution

Keep the drain-result idea, but make suppression cover the real poll cadence or in-flight entry state, preserve or explicitly approve the Plugin SDK contract change, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: poll-success calls the reconnect drain, the drain skips entries held by withActiveDeliveryClaim, and the session guard clears immediately after that skipped drain returns. The linked report adds runtime logs showing the same entry skipped on later poll successes.

## Solution Assessment

No. Returning drain counts is a plausible owner-boundary fix, but a 30-second cooldown does not reliably cover the 30-second long-poll cadence, and the public SDK return type needs compatibility handling or maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Cover the next long-poll success before clearing the cooldown:** `extensions/telegram/src/polling-session.ts:125`
  - body: The reported loop fires on poll-success about every 30 seconds, and Telegram's default long-poll timeout is also 30 seconds. With `DELIVERY_DRAIN_COOLDOWN_MS` set to exactly `30_000`, the next successful long poll can arrive after the cooldown expires and call `drainPendingDeliveries` again for the same in-flight entry, so the repeated `already being recovered` drain can continue despite the immediate re-entry test. Use a longer or entry-aware suppression and add a delayed poll-success regression.
  - confidence: 0.88
- **[P1] Preserve the published drainPendingDeliveries return contract:** `src/plugin-sdk/delivery-queue-runtime.ts:28-30`
  - body: `openclaw/plugin-sdk/delivery-queue-runtime` is a published Plugin SDK subpath, and the current helper contract returns `Promise<void>`. Changing it to `Promise<ReconnectDrainResult>` can break external plugins that pin or implement the existing type, while the Telegram fix only needs result data for this recovery path. Preserve the old export or add an explicitly approved additive result API with docs/contract coverage.
  - confidence: 0.8
