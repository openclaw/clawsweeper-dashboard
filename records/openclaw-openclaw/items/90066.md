## Summary

Keep open: the PR targets the right Telegram reconnect-drain path and now preserves the old SDK void wrapper, but the fixed 60s cooldown can still re-enter during the reported 129s in-flight send, the new SDK export needs maintainer-visible contract alignment, and the supplied proof is build/test output rather than live Telegram runtime evidence.

## What This Changes

This branch returns reconnect drain counts from core, adds an SDK result-bearing drain helper while keeping the old void wrapper, cooldowns Telegram reconnect drains after all matched entries are in flight, adds regression tests, and edits CHANGELOG.md.

## Best Possible Solution

Tie suppression to the active in-flight delivery state or otherwise prove the cooldown fully covers the reported stall, keep the SDK void wrapper while documenting any new public result API, remove the changelog edit, and add redacted live Telegram runtime proof before merge.
