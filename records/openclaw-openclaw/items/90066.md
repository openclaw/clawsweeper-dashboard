## Summary

Keep open: this is a valid Telegram delivery bug PR, but the latest patch is not merge-ready because the fixed 60s cooldown can expire during the reported 129s in-flight send, the new public SDK helper needs contract alignment, and the supplied proof still does not show the real Telegram stall path.

## What This Changes

The PR changes Telegram reconnect drain to return drain counts, adds a result-bearing Plugin SDK helper, applies a 60s Telegram drain cooldown for in-progress entries, and adds regression tests.

## Best Possible Solution

Land a narrow Telegram/outbound fix that suppresses re-entry for the lifetime of the claimed delivery, preserves or explicitly updates the Plugin SDK contract, and includes live Telegram proof.

## Reproduction Assessment

Yes from source inspection, though not live reproduced here. Current main drains on poll-success, live sends hold withActiveDeliveryClaim, and the linked report provides a 129484ms trace of repeated already-being-recovered logs.

## Solution Assessment

No. The fixed 60s cooldown is a plausible throttle, but it can expire before the observed in-flight claim ends; the better fix ties suppression to the active claim lifecycle and aligns or avoids the public SDK API expansion.
