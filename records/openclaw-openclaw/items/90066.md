## Summary

This PR targets a real Telegram reconnect-drain bug, but it is not merge-ready: the fixed 60s cooldown can expire while the same observed 129s live send still holds the delivery claim, the new public Plugin SDK helper is not aligned with the SDK docs/API baseline, and the supplied proof is terminal build/grep/test output rather than live Telegram behavior.

## What This Changes

The PR changes outbound reconnect drain to return result counts, adds a result-bearing Plugin SDK helper, applies a 60s Telegram reconnect-drain cooldown when all matched entries are already in progress, and adds regression tests.

## Best Possible Solution

Tie reconnect-drain suppression to the active delivery claim lifecycle or otherwise prove it covers stalls longer than the observed 129s, keep the SDK contract aligned, and require redacted live Telegram evidence before merge.

## Reproduction Assessment

Yes for the source path: current main triggers reconnect drain on poll-success while live sends hold withActiveDeliveryClaim, and drain skips entries already in entriesInProgress. I did not run a live Telegram stall, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No. The PR is a plausible mitigation, but the fixed 60s cooldown does not cover the reported 129s in-flight claim and the new public SDK helper needs contract alignment before it is the best fix.
