## Best Possible Solution

Fix the alias-aware explicit-config guard, add real behavior proof for the runtime paths beyond normal OpenAI CLI conversion, then have TTS/provider owners decide whether this shared speech-core policy should land.

## Reproduction Assessment

Yes for the review finding by source inspection: Azure Speech accepts the `azure` alias, but the auto-emotion guard checks raw config only under the canonical provider id before adding inferred prosody. For the feature itself, reproduction is not applicable on current main because the config/runtime surface is new.

## Solution Assessment

No, not yet. The patch should preserve alias-keyed effective provider config before adding inferred provider overrides, and the shared speech-core policy still needs TTS/provider owner review.
