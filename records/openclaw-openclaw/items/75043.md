## Summary

Keep open: this is a recent feature PR for a new public TTS config surface, current main does not implement it, and the thread already identifies an unresolved TTS/provider product-boundary decision. The latest patch also has a concrete correctness gap in the streaming TTS path and still needs current-head real behavior proof before merge.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` config, speech-core emotion inference, provider-specific TTS mappings, provider adapter plumbing, schema/help/docs, tests, and a changelog entry.

## Best Possible Solution

Have TTS/provider owners decide the shared `autoEmotion` product boundary; if approved, make synthesis, streaming, and telephony apply the same precedence model and land with current-head real behavior proof.

## Reproduction Assessment

Not applicable for the feature request itself, but the review finding is source-reproducible: PR head applies auto emotion to normal and telephony synthesis while the stream path passes only raw request overrides. I did not run runtime tests because this was a read-only review.

## Solution Assessment

No, not as currently written. The overall direction may be acceptable after owner approval, but the implementation should apply the same auto-emotion override logic to streaming synthesis before it can be considered complete.
