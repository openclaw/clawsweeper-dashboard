## Summary

Keep open. This is an active feature PR, not implemented on main or superseded; the PR head adds the requested config/runtime surface, but the streaming TTS path still bypasses autoEmotion and the supplied proof is stale after later head updates.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` config, speech-core emotion inference, provider-native TTS mappings, docs/schema/help updates, and focused tests.

## Best Possible Solution

Land the TTS feature only if owners accept the shared speech-core policy, after applying autoEmotion to streaming synthesis, adding a streaming regression test, and refreshing latest-head real behavior proof.

## Reproduction Assessment

Not applicable for the feature request itself. Source inspection shows current main has no `autoEmotion` config/runtime surface, while the PR adds a new optional behavior rather than fixing a defined broken contract.

## Solution Assessment

No, not yet. The implementation should either consistently apply autoEmotion across normal, telephony, and streaming TTS paths or explicitly exclude streaming in the product contract; the current diff silently skips streaming synthesis.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Apply autoEmotion in the streaming TTS path:** `extensions/speech-core/src/tts.ts:1728`
