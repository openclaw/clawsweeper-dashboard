## Summary

Keep open. Current main still lacks `messages.tts.autoEmotion`, and this PR is useful but not merge-ready because its explicit-setting guards can suppress or override auto-emotion contrary to the new docs, it is missing the required user-facing changelog entry, and the author has asked maintainers to decide the shared speech-core product boundary before merge.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` config, shared speech-core emotion inference, provider-specific TTS mappings, OpenAI/Azure/Microsoft adapter support, docs, generated config schema/help/labels, and focused tests.

## Best Possible Solution

Revise the PR so explicit-setting detection matches the documented provider fields, respects provider aliases and legacy config keys before applying auto-emotion, adds the required Unreleased changelog entry, and then have the requested maintainers decide whether shared speech-core is the right long-term boundary for this feature.

## Reproduction Assessment

Yes. On the PR head, configure ElevenLabs with `autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, or configure Azure/Microsoft prosody through a supported alias such as `azure`/`edge`; code inspection shows the first case suppresses auto-emotion unexpectedly and the second can overwrite explicit alias config.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be maintainable after product-boundary approval, but the current override detection is broader than the ElevenLabs fields the PR owns, narrower than the provider alias surface the repo supports, and missing the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Don't treat similarityBoost as an emotion override:** `extensions/speech-core/src/tts.ts:392-394`
  - body: With ElevenLabs, `hasElevenLabsVoiceSettings` returns true when the effective config only sets `voiceSettings.similarityBoost`. This PR only auto-writes `voiceSettings.style` and `voiceSettings.stability`, and the docs say only explicit style or stability should opt out, so a user who pins similarity loses auto emotion unexpectedly.
  - confidence: 0.9
- **[P2] Check alias configs before applying auto emotion:** `extensions/speech-core/src/tts.ts:1300-1301`
  - body: The explicit-setting check reads only `resolveRawProviderConfig(..., params.providerId)`, so canonical providers miss supported aliases such as Azure's `azure` config and Microsoft's legacy `edge` config. Those explicit prosody settings still feed the providers, but auto-emotion can overwrite them because this guard never sees them.
  - confidence: 0.82
- **[P2] Add the required TTS changelog entry:** `CHANGELOG.md:5-7`
  - body: This PR adds a user-facing `messages.tts.autoEmotion` config and provider behavior, but the PR head leaves `CHANGELOG.md` without an Unreleased Changes entry. Repo policy requires user-facing feat/fix/perf changes to include a single-line changelog entry with contributor attribution.
  - confidence: 0.93
