## Summary

Keep open. The feature is not on current main and the PR still has a concrete correctness gap: telephony now receives auto-emotion overrides, but the real OpenAI and Azure telephony adapters drop the generated controls before calling their transports.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` config, shared speech-core emotion inference, provider-specific TTS mappings, docs/config schema/help/labels, changelog, and focused tests.

## Best Possible Solution

Wire generated auto-emotion controls through the real OpenAI and Azure telephony adapters, add provider-level telephony regression coverage, then have owners decide the shared speech-core policy boundary before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `textToSpeechTelephony` passes auto-emotion provider overrides onward, while the real OpenAI and Azure telephony methods omit those generated instructions/prosody before transport calls.

## Solution Assessment

No, not as-is. The shared mapping may be maintainable if owners approve the product boundary, but the implementation must apply generated controls in real telephony providers or explicitly exclude telephony from the feature.
