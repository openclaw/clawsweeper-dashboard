## Summary

Keep open: current main still lacks the proposed `messages.tts.autoEmotion` surface, and this active PR is the implementation path for a new TTS feature rather than cleanup that can be closed. The patch still has an alias-precedence correctness issue, and the linked tracker at https://github.com/openclaw/openclaw/issues/82109 leaves the shared speech-core product boundary open for owner review.

## What This Changes

Adds opt-in `messages.tts.autoEmotion` config, shared TTS emotion inference/mapping, provider adapter support for native speech controls, docs/schema/help/changelog updates, and focused tests.

## Best Possible Solution

Fix alias-aware precedence, get TTS/provider owner agreement on shared speech-core `autoEmotion` versus provider-owned hints, and require current real behavior proof for normal, streaming, and telephony paths before merge.

## Reproduction Assessment

Yes for the review finding by source inspection: configure Azure Speech with explicit prosody under `messages.tts.providers.azure` and enable `autoEmotion`; the PR guard can miss that alias before adding inferred prosody. For the feature itself, current main has no `autoEmotion` surface, so a current-main bug repro is not applicable.

## Solution Assessment

No, not yet. The patch should honor the already resolved/effective provider config or otherwise check aliases before applying inferred provider overrides, and the shared speech-core policy still needs owner review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor alias-keyed Azure prosody before auto emotion:** `extensions/speech-core/src/tts.ts:1299-1301`
  - body: The explicit-setting guard rebuilds provider config from raw config under the canonical id, so an Azure user with `messages.tts.providers.azure.rate` / `pitch` / `volume` is not seen even though the provider resolver accepts that alias.
