## Summary

Keep open. The latest PR head still has a concrete ElevenLabs precedence bug and lacks the required changelog entry; current main does not already implement `autoEmotion`, and the shared speech-core/provider-boundary decision still needs maintainer review.

## What This Changes

This PR adds opt-in `messages.tts.autoEmotion` inference in speech-core, maps inferred emotions to provider-native TTS controls, and updates provider adapters, config schema/help/docs, generated schema, and tests.

## Best Possible Solution

Revise the ElevenLabs explicit-setting detection and regression coverage, add the changelog entry, then get maintainer approval on the shared speech-core mapping boundary before merge.

## Reproduction Assessment

Yes. On the PR head, enable `messages.tts.autoEmotion` for ElevenLabs and set only `voiceSettings.similarityBoost`; `hasElevenLabsVoiceSettings` returns true and prevents generated `style`/`stability` for happy text.

## Solution Assessment

No, not as-is. The opt-in mapping may be maintainable if the boundary is approved, but the ElevenLabs guard should only treat `style` or `stability` as emotion-equivalent for this auto-emotion opt-out, and the user-facing feature needs a changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Stop treating similarityBoost as an emotion override:** `extensions/speech-core/src/tts.ts:392-394`
