## Summary

Keep open: current main does not implement `autoEmotion`, and this PR still addresses a real TTS feature gap, but the patch is not ready because the ElevenLabs explicit-setting guard still suppresses auto-emotion for `similarityBoost`-only configs and the user-facing feature lacks a changelog entry.

## What This Changes

This PR adds opt-in `messages.tts.autoEmotion` config, shared speech-core emotion inference and provider-specific TTS overrides, OpenAI/Microsoft/Azure adapter support, docs/schema/help updates, and focused tests.

## Best Possible Solution

Revise ElevenLabs explicit-setting detection so only `voiceSettings.style` and `voiceSettings.stability` suppress auto-emotion, add a regression test for similarityBoost-only config or request overrides, add the required `CHANGELOG.md` entry with contributor credit, and then have the requested maintainers decide the shared speech-core/provider-boundary design.

## Reproduction Assessment

Yes. On the PR head, configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, then synthesize happy text; the predicate at `extensions/speech-core/src/tts.ts:392` returns true before auto `style`/`stability` can be built.

## Solution Assessment

No, not as-is. The opt-in provider mapping approach may be acceptable after owner review, but the current ElevenLabs guard is broader than the fields the PR maps and broader than the docs describe.
