## Summary

Keep open. Current main does not implement `messages.tts.autoEmotion`, so this PR is not obsolete, but the PR head still has a concrete ElevenLabs override bug, lacks the required changelog entry, and has an explicit maintainer product-boundary review request.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference in speech-core, maps inferred emotions to provider-native TTS controls, and updates provider adapters, config schema/help/docs, generated schema, and tests.

## Best Possible Solution

Revise the ElevenLabs explicit-setting detection and regression coverage, add the changelog entry, preserve current-main OpenAI TTS behavior during branch update, then have the relevant maintainers approve or redirect the shared speech-core mapping design.

## Reproduction Assessment

Yes. On the PR head, enable `messages.tts.autoEmotion`, choose ElevenLabs, and provide only `voiceSettings.similarityBoost`; `hasElevenLabsVoiceSettings` returns true and prevents generated `style`/`stability` for happy text.

## Solution Assessment

No, not as-is. The opt-in provider-aware mapping may be maintainable if the boundary is approved, but the current ElevenLabs guard is broader than the fields the PR maps and broader than the docs describe.
