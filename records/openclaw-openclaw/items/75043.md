## Summary

Keep open. Current main does not implement `messages.tts.autoEmotion`, and the PR still contains a concrete ElevenLabs precedence bug plus a missing required changelog entry, while the author has explicitly asked maintainers to review the shared speech-core/provider-boundary direction.

## What This Changes

The PR adds opt-in TTS auto-emotion inference in speech-core, maps inferred emotions into provider-native controls, updates OpenAI/Microsoft/Azure provider adapters, exposes config schema/help/docs, and adds focused tests.

## Best Possible Solution

Revise the ElevenLabs explicit-setting detection so only `voiceSettings.style` and `voiceSettings.stability` suppress auto-emotion, add a regression test for similarity-only config or overrides, add the required changelog entry with contributor credit, then have the relevant maintainers decide the shared speech-core/provider-boundary design before merge.

## Reproduction Assessment

Yes. On the PR head, configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, then synthesize happy text; `hasElevenLabsVoiceSettings` returns true before auto `style`/`stability` can be built.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be acceptable after maintainer boundary review, but the current ElevenLabs guard is broader than the fields the PR maps and broader than the docs describe.
