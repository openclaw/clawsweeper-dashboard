## Summary

Keep open. Current main does not implement this feature, and the PR head still has a blocking ElevenLabs override bug plus a missing changelog entry; the author also requested maintainer product-boundary review before merge.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference in speech-core, maps emotions into provider-specific TTS controls, and updates provider adapters, config schema/help, docs, and tests.

## Best Possible Solution

Revise the ElevenLabs explicit-setting detection so only `voiceSettings.style` and `voiceSettings.stability` suppress auto-emotion, add a regression test for similarity-only config or overrides, add the required changelog entry with contributor credit, then have the relevant maintainers decide the shared speech-core/provider-boundary design before merge.

## Reproduction Assessment

Yes. On the PR head, configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, then synthesize happy text; `hasElevenLabsVoiceSettings` returns true before auto `style`/`stability` can be built.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be acceptable after maintainer boundary review, but the current ElevenLabs guard is broader than both the mapped fields and the docs, and the user-facing feature still needs a changelog entry.
