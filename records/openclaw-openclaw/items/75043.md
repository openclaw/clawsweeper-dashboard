## Summary

Keep open. Current main does not implement this TTS auto-emotion surface, and the PR head still has a blocking ElevenLabs precedence bug plus a missing changelog entry; the author also asked maintainers to resolve the shared speech-core/provider-boundary question before merge.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference in speech-core, maps inferred emotion to provider-specific TTS controls, updates OpenAI/Microsoft/Azure adapter handling, and adds config schema/help/docs/tests.

## Best Possible Solution

Revise the ElevenLabs explicit override detection so only `voiceSettings.style` and `voiceSettings.stability` suppress auto-emotion, add regression coverage for similarity-only config or request overrides, add the required changelog entry with contributor credit, then have the relevant TTS maintainers decide whether the shared speech-core/provider-boundary design should land.

## Reproduction Assessment

Yes. On the PR head, configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, then synthesize happy text; `hasElevenLabsVoiceSettings` returns true before auto `style`/`stability` can be built.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be acceptable after maintainer boundary review, but the current ElevenLabs guard is broader than the fields the PR maps and broader than the docs describe.
