## Summary

Keep open. Current main still lacks the requested auto-emotion feature, and the PR head still drops generated auto-emotion controls in the real OpenAI and Azure telephony provider calls, so the patch remains incorrect pending a narrow repair and owner product-boundary review.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference, provider-specific TTS mappings, config/schema/docs/help updates, a changelog entry, and focused tests.

## Best Possible Solution

Wire generated auto-emotion controls through the real OpenAI and Azure telephony adapters with provider-level regression tests, then have TTS/provider owners decide whether the shared speech-core policy and public config should ship.

## Reproduction Assessment

Yes. Source inspection of the PR head shows speech-core now passes generated telephony provider overrides onward, while the real OpenAI and Azure telephony methods omit those generated instructions/prosody before transport calls.

## Solution Assessment

No, not as-is. The shared mapping may be maintainable if owners approve the product boundary, but the implementation must apply generated controls in real telephony providers or explicitly exclude telephony from the feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Pass OpenAI auto emotion into telephony TTS:** `extensions/openai/speech-provider.ts:354`
  - body: The PR teaches speech-core to pass auto-emotion overrides into `synthesizeTelephony`, but OpenAI still sends `instructions: config.instructions` in the telephony request. Enabled auto emotion therefore works for normal OpenAI TTS but is silently ignored for voice-call TTS; use the same merge policy as normal synthesis and add a telephony regression test.
  - confidence: 0.93
- **[P2] Pass Azure prosody into telephony TTS:** `extensions/azure-speech/speech-provider.ts:333-342`
  - body: Azure normal synthesis forwards override `rate`, `pitch`, and `volume`, but the telephony path omits those fields when calling `azureSpeechTTS`. Auto-emotion prosody is generated for Azure telephony and then dropped, so wire those values through and cover the real Azure telephony provider path.
  - confidence: 0.92
