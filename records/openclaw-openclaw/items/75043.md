## Summary

Keep open. Current main does not implement `messages.tts.autoEmotion`, and the PR is still useful, but the current head has a concrete ElevenLabs precedence bug and is missing the required changelog entry. The author also explicitly asked for maintainer product-boundary review on whether shared speech-core should own provider-aware emotion mapping.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference, maps inferred emotions to provider-native TTS controls, updates OpenAI/Microsoft/Azure speech providers, exposes config schema/help/labels/docs, and adds focused tests.

## Best Possible Solution

Revise the ElevenLabs explicit-setting detection so only `voiceSettings.style` and `voiceSettings.stability` suppress auto-emotion, add a regression test for similarityBoost-only config or overrides, add the required `CHANGELOG.md` entry with contributor credit, then have the requested maintainers decide the shared speech-core/provider-boundary design.

## Reproduction Assessment

Yes. On the PR head, configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, then synthesize happy text; `hasElevenLabsVoiceSettings` returns true before auto `style`/`stability` can be built.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be acceptable after maintainer boundary review, but the current ElevenLabs guard is broader than the fields the PR maps and broader than the docs describe.
