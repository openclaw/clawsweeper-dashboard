## Summary

Keep open: the PR now applies autoEmotion across normal, telephony, and streaming TTS paths, but it still has a correctness issue where accepted provider aliases can bypass the explicit-setting precedence guard. It also still needs TTS/provider owner approval for the shared speech-core product boundary.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference and maps inferred emotion into provider-native TTS controls across speech-core, provider adapters, config schema/help, docs, tests, and changelog.

## Best Possible Solution

Fix the alias-aware explicit-setting guard, keep the normal/telephony/streaming coverage, and land only after TTS/provider owners accept the shared `autoEmotion` policy; otherwise use https://github.com/openclaw/openclaw/issues/82109 and https://github.com/openclaw/openclaw/issues/67539 to reshape the direction.

## Reproduction Assessment

Not applicable. This PR adds a new optional config/runtime capability, and current-main source inspection confirms the `autoEmotion` surface is absent rather than broken.

## Solution Assessment

No, not yet. The overall design is coherent after the streaming-path fix, but the implementation must preserve explicit alias-backed provider config before autoEmotion and still needs owner product-boundary approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Honor provider aliases before applying autoEmotion:** `extensions/speech-core/src/tts.ts:1298-1299`
  - body: `resolveRawProviderConfig` only checks the canonical provider id, so accepted alias config such as `messages.tts.providers.azure.rate` is missed by the explicit-setting guard. Azure then applies generated `providerOverrides.rate/pitch/volume` before `config.rate/pitch/volume`, so enabling autoEmotion can override an explicit user prosody setting that the provider otherwise accepts.
  - confidence: 0.85
