## Summary

Keep open: this is a new provider-boundary feature that is not on current main, and the latest PR head still has a concrete Azure alias precedence bug plus an open product-boundary review question.

## What This Changes

Adds opt-in `messages.tts.autoEmotion` config, shared speech-core emotion inference, provider-native TTS override mapping, config/schema/docs updates, and focused tests.

## Best Possible Solution

Fix the alias-aware explicit-config check, then land only if TTS/provider owners accept the shared opt-in policy and the PR body includes real proof for each runtime path changed.

## Reproduction Assessment

Yes for the review finding by source inspection: configure Azure Speech under `messages.tts.providers.azure` with explicit prosody and enable auto emotion, and the guard can miss that alias-keyed config. For the feature itself, reproduction is not applicable on current main because the config/runtime surface does not exist there.

## Solution Assessment

No, not yet. The patch should preserve alias-keyed effective provider config before adding inferred provider overrides, and the shared speech-core policy still needs owner product review.
