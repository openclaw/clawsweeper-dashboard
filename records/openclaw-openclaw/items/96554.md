- **[P1] Register workboard_move in the plugin contract:** `extensions/workboard/src/tools.ts:1003`
  - body: This adds `workboard_move` to the factory, but Workboard exposes tools through the explicit names list in `extensions/workboard/index.ts` and the manifest `contracts.tools` list. `resolvePluginTools` rejects undeclared returned tools, so agents will not actually receive this new tool until those contracts are updated and covered through the runtime registration path.
  - confidence: 0.93
- **[P2] Enforce claim scope inside the move mutation:** `extensions/workboard/src/tools.ts:1021`
  - body: `readScopedCardToolParams` checks the claim before calling `store.move`, but `store.move` goes through an unscoped `update()` and does not re-check ownership inside the mutation. A concurrent claim between the precheck and the update can still be overwritten, unlike sibling scoped mutations that pass scope into the store method.
  - confidence: 0.78
- **[P2] Reuse the canonical status list in the CLI:** `extensions/workboard/src/cli.ts:172-183`
  - body: The CLI command introduces a second hard-coded status list even though Workboard already exports `WORKBOARD_STATUSES` and the store validates status with it. If the canonical status set changes, the CLI can reject a valid store status before the request reaches the store, so this should import the shared constant or rely on store validation.
  - confidence: 0.86
- **[P3] Document the new Workboard command and tool:** `extensions/workboard/src/cli.ts:156`
  - body: This PR adds a public CLI command and agent tool, but the Workboard CLI reference still lists only list/create/show/dispatch and the plugin docs still omit `workboard_move`. Please update the relevant docs so users and agents see the new supported surface.
  - confidence: 0.82
