## Summary

Keep open. Current main has useful Google Meet hardening, including the merged #73256 observe-mode/status work, but it still does not deliver the central requested outcome: a supported, reliable headless/listen-first Meet participant with documented identity semantics and end-to-end live audio or transcription health proof.

## What This Changes

The issue requests reliable headless Google Meet agent joining with identity/entry modes, listen/transcribe as the minimum useful mode, bidirectional voice as the target, and explicit audio/transcription health checks.

## Best Possible Solution

Keep this as Google Meet plugin product work: define the identity and meeting-entry model, choose an official Media API versus browser-participant strategy, expose status that separates browser/lobby/admission/provider/audio/transcript state, add preflight checks that fail on missing real audio or transcript movement, and validate with live Meet proof.

## Reproduction Assessment

Unclear. The issue includes concrete 2026.4.24 live failure data and current main still exposes the relevant join/status/test-speech surfaces, but a high-confidence reproduction now requires a real Meet, signed-in Chrome profile or paired node, realtime provider, and audio route that were not run in this read-only review.

## Solution Assessment

No. Current main is a useful partial hardening path, but it is not the complete or narrowest maintainable solution because the realtime path still depends on Chrome/profile/virtual-audio behavior and observe-only mode does not prove a live transcript stream.
