## Summary

Keep this PR open: the mobile drawer normal-property fallbacks are plausible, but the focus-ring fallback still does not fix the stated older-browser a11y bug because the winning custom property still contains color-mix(), and the external proof gate is still only a written claim.

## What This Changes

The PR adds CSS fallback declarations for Control UI focus rings, the mobile navigation drawer shadow/backdrop, and the collapsed sidebar active accent when color-mix() is unsupported.

## Best Possible Solution

Guard each color-mix focus-ring custom-property value with @supports or move the fallback to the actual box-shadow use sites, keep the ordinary-property mobile drawer fallbacks, and require real older-browser proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main puts color-mix() in --focus-ring and mobile drawer declarations used by focus-visible and drawer UI paths. I did not run Firefox 112 or old Safari in this read-only review.

## Solution Assessment

No, not yet: ordinary-property fallbacks in layout.mobile.css are the right shape, but base.css needs @supports-guarded custom-property values or use-site fallbacks so the solid ring actually wins in no-color-mix browsers.
