## Summary

Keep open: the mobile drawer fallbacks are in the right shape, but the focus-ring custom property still gets overwritten by a later color-mix() value in no-color-mix browsers, and the PR has no inspectable real-behavior proof.

## What This Changes

The PR adds CSS fallback declarations for Control UI focus rings, mobile drawer shadow/backdrop, and collapsed sidebar active accents when color-mix() is unsupported.

## Best Possible Solution

Keep the solid focus-ring value as the default, move the color-mix() focus-ring values behind @supports or to use-site fallbacks, keep the ordinary-property drawer fallbacks, and require inspectable older-browser proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main puts color-mix() in --focus-ring and mobile drawer declarations used by focus-visible and drawer UI paths, though I did not run Firefox 112 or old Safari in this read-only review.

## Solution Assessment

No. layout.mobile.css uses the right ordinary-property fallback shape, but base.css needs @supports-guarded custom-property values or use-site fallbacks so the solid ring actually wins in no-color-mix browsers.
