## Summary

Keep open: this is a focused Telegram regression PR, but it is not merge-ready because current main already gates Bot API 10.1 rich sends behind `richMessages`, while this formatter change introduces malformed expandable blockquotes and an unsupported legacy HTML tag with only simulation proof.

## What This Changes

The PR adds a `richMode` option to `markdownToTelegramHtml`, downgrades blockquote and heading tags to bold by default, and adds `br` to the legacy Telegram HTML allowlist.

## Best Possible Solution

Keep the current `richMessages` opt-in boundary as the baseline, and only change the legacy formatter if live Telegram Web proof shows `parse_mode=HTML` blockquotes still fail, with tests preserving expandable blockquotes and the documented tag contract.

## Reproduction Assessment

The original Telegram Web symptom has user reports and screenshots, but this review did not run a live Web client repro.

## Solution Assessment

No. Current main already solves the Bot API 10.1 rich-message default through `richMessages`, and Telegram's documented standard HTML path supports blockquotes; a legacy formatter downgrade needs live Web proof and narrower handling.
