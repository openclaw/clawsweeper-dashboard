## Summary

Keep open: this PR targets a real Telegram readability regression, but the patch is not merge-ready because it changes only the direct formatter helper while the normal final-reply chunk renderer still emits the same tags, and the supplied proof is only a synthetic simulation rather than live Telegram Web or live transport proof.

## What This Changes

The branch adds a `richMode` option to `markdownToTelegramHtml`, downgrades blockquote and heading tags to `<b>` on the default path, and allows `<br>` in the legacy Telegram HTML tag set.

## Best Possible Solution

Use one maintainer-approved Telegram compatibility policy: keep rich messages opt-in for default users, add fallback for opt-in rich sends where needed, and only add legacy parse-mode tag downgrades through a shared direct and chunked renderer with tests and live Telegram proof.

## Reproduction Assessment

Yes, from source and linked issue context: v2026.6.8 sends rich Telegram messages in the reported area, while current non-rich chunked rendering can still emit blockquote HTML through the standard delivery path. I did not run a live Telegram Web repro in this read-only review.

## Solution Assessment
