## Summary

Keep open: current main and v2026.6.10 still hardcode Twilio voice-call REST routing to US1, while the open linked PR is the active implementation candidate and should close this issue only after merge.

## What This Changes

Request support for a closed `twilio.region` voice-call setting that derives Twilio regional REST hostnames and SSRF allowlists while preserving the US1 default.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/95832, keeping this issue open as the product record until a region-only Twilio routing implementation is merged.

## Reproduction Assessment

Yes, at source level: current main constructs Twilio REST URLs from `api.twilio.com`, both REST allowlists only permit `api.twilio.com`, and the config schema has no `twilio.region`. I did not run a credentialed non-US current-main failure path.

## Solution Assessment

Yes, the region-only field with internally derived exact hostnames is the narrow maintainable path because it avoids arbitrary base URLs and edge/region drift. This issue should remain open until the linked PR merges or maintainers choose a different contract.
