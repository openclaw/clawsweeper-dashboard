## Label Justifications

- `impact:auth-provider`: The issue concerns provider routing and Region-specific Twilio credentials for the voice-call plugin.

## Summary

Keep open. Current main and the latest release still hardcode the voice-call Twilio REST path and SSRF allowlists to `api.twilio.com`, and the plugin config schema has no `twilio.edge` or `twilio.region` fields; the earlier similar pull request is useful implementation context but was closed unmerged.

## What This Changes

Request optional voice-call Twilio Edge and Region configuration that derives exact regional API hostnames and SSRF allowlists while preserving the current US1 default.

## Best Possible Solution

Add plugin-owned structured `twilio.edge` and `twilio.region` support with shared hostname derivation for POST and status requests, exact SSRF allowlisting, docs, UI hints, and focused default/regional/partial-config tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current config has no `twilio.edge` or `twilio.region`, and all Twilio REST requests are built or allowlisted for `api.twilio.com`. I did not run a live Twilio regional call.

## Solution Assessment

Yes as a direction: structured Edge and Region fields are safer than a raw `baseUrl` and match Twilio's current regional REST contract. The exact validation, edge inference, and env-var surface still need maintainer approval because they add plugin config and touch an SSRF boundary.
