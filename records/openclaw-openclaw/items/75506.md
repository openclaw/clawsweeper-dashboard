## Summary

Keep open. Current main still has a credible packaged-CLI warning path: configured-channel doctor/control-plane lookups can full-load the bundled Telegram channel with runtime dependency staging disabled, while the Telegram full plugin graph still statically reaches `grammy`. The reporter's Linux evidence and the follow-up macOS confirmation both point at the same package-root versus staged-runtime-deps mismatch; related open reports cover overlapping packaged dependency symptoms, but no merged fix or direct implementation PR has resolved this exact CLI warning path yet.

## What This Changes

The issue requests that package-installed CLI commands avoid noisy missing-`grammy` warnings when Telegram runtime dependencies are staged outside the OpenClaw package root.

## Best Possible Solution

Keep metadata, doctor, cron, completion, and other control-plane paths on manifest/public-artifact or setup-only surfaces, and keep Telegram runtime-only modules lazy or loaded through the prepared bundled runtime root only when actual Telegram execution needs them. The end state should make a packaged install with externally staged Telegram deps quiet for unrelated CLI commands without requiring `grammy` to be copied into the core package root.

## Reproduction Assessment

Yes. A high-confidence reproduction is a packaged install where `grammy` is absent from the OpenClaw package root but present under the staged plugin-runtime-deps root, then running `openclaw doctor`, `openclaw cron list`, or `openclaw completion --help` with Telegram configured. Current main source still contains the full-load path and static `grammy` imports needed for the warning.

## Solution Assessment

Yes, the issue's direction is the maintainable fix boundary: avoid importing runtime-heavy Telegram modules from unrelated CLI/control-plane commands or load them through the prepared runtime-deps root only for real runtime work. Adding `grammy` to core `package.json` may hide the symptom for one channel, but it conflicts with the staged bundled-plugin dependency model and would not address the generic loader boundary.
