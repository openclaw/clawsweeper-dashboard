## Summary

Keep open. Current main still has a credible packaged-install warning path: doctor/channel metadata can fall through to full bundled Telegram plugin loading with runtime dependency staging disabled, while the full Telegram channel graph still has static value imports from `grammy`. The prior ClawSweeper comment is consistent with current source inspection; this is not superseded by the broader runtime-deps cluster because this issue has a narrow CLI warning surface and concrete repair boundary.

## What This Changes

The issue requests that CLI commands stop warning about missing Telegram runtime dependency `grammy` when those dependencies are already staged for plugin runtime use.

## Best Possible Solution

Keep diagnostic and metadata CLI paths on lightweight manifest, setup-entry, or doctor-contract artifacts, and only load Telegram runtime modules through the prepared staged runtime root when real Telegram execution is needed.

## Reproduction Assessment

Yes. Use a package-style install where Telegram dependencies are staged under the plugin runtime-deps root but absent from the package root, configure Telegram, then run `openclaw doctor` or another metadata-only CLI command that consults channel doctor/runtime metadata; current source still contains the full-load path needed for the warning.

## Solution Assessment

Yes. The maintainable direction is to stop metadata/doctor CLI code from importing the full Telegram runtime graph, rather than copying `grammy` into the global package root or making unrelated commands perform runtime-dependency repair.
