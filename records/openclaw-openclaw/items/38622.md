## Label Justifications

- `P1`: Affected agents can silently start without expected persona, identity, memory, or operating instructions when bootstrap files are symlinked.
- `impact:session-state`: Rejected bootstrap files remove expected session context and can change agent behavior across every affected session.
- `impact:security`: Any fix must decide which symlink targets outside the workspace may be trusted and read into model context.

## Summary

Current main and the latest release still render out-of-workspace symlinked bootstrap files as missing, and an open maintainer PR already owns the security-sensitive fix direction, so this issue should stay open.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files when the workspace entry is a symlink to a shared file outside the workspace instead of rendering it as missing.

## Best Possible Solution

Refresh and land the focused workspace-bootstrap reader fix in https://github.com/openclaw/openclaw/pull/73401 with an explicit trust policy and regression tests for allowed symlink hops plus rejected direct, hardlinked, dangling, non-regular, sensitive, and oversized targets.

## Reproduction Assessment

Yes, at source level: create a workspace with `SOUL.md` symlinked to a file outside the workspace, and the current `openRootFile` boundary rejects that target before `loadWorkspaceBootstrapFiles` maps the read failure to `missing: true`. I did not run a live filesystem repro in this read-only review.

## Solution Assessment

No, not as blanket symlink following. The maintainable path is a fixed-bootstrap-only policy at the active fs-safe/OpenClaw wrapper boundary, with explicit sensitive-path and non-bootstrap rejection tests.
