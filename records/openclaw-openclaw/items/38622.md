## Summary

Keep open: current main and the latest release still route fixed workspace bootstrap files through a workspace-root guarded reader, so external symlink targets still become missing Project Context entries. The issue remains the canonical report, with an open replacement PR awaiting the external-bootstrap trust-boundary decision.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files such as SOUL.md and AGENTS.md when those files are symlinks to shared files outside the workspace directory.

## Best Possible Solution

Keep this as the canonical report until the scoped fixed-bootstrap symlink policy in https://github.com/openclaw/openclaw/pull/73401 or an equivalent implementation lands with regression tests and security review, or until maintainers document external bootstrap symlinks as unsupported with a clear blocked diagnostic.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a fixed bootstrap symlink is passed to openRootFile with the workspace as root, fs-safe rejects an external target, and loadWorkspaceBootstrapFiles records the entry as missing; I did not run a live session in this read-only review.

## Solution Assessment

Unclear until maintainers settle the trust model. A scoped fixed-bootstrap reader exception is safer than relaxing fs-safe globally, while an explicit opt-in or unsupported-policy diagnostic may be preferable if external prompt-context reads must stay explicit.
