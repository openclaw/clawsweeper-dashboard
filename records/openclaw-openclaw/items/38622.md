## Summary

Keep open: current main and the latest release still reject external symlinked workspace bootstrap files through the workspace-root boundary reader, collapse the failure into a missing bootstrap entry, and render the reported `[MISSING]` marker. Related open PRs now own alternative fix paths, but the trust-boundary decision is still unresolved.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files such as `SOUL.md` when they are symlinks to readable files outside the workspace instead of injecting a `[MISSING]` marker.

## Best Possible Solution

Land or explicitly reject a scoped bootstrap-symlink policy that preserves boundary guards and surfaces blocked-symlink diagnostics, then consolidate related reports once shipped.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an external symlinked bootstrap file reaches `openRootFile` under the workspace root, fails the fs-safe boundary contract, is stored as `missing: true`, and renders as `[MISSING] Expected at: ...`.

## Solution Assessment

Unclear until maintainers settle the trust model. The narrower fixed-bootstrap approach in https://github.com/openclaw/openclaw/pull/73401 has a smaller blast radius than the broader configurable allowlist in https://github.com/openclaw/openclaw/pull/38981, but either path must keep security guards and diagnostics explicit.
