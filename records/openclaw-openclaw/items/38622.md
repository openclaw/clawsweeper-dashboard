## Summary

Keep open: current main still rejects external workspace-bootstrap symlink targets and renders them as missing, while the active maintainer PR is open and the security policy is not settled.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files such as SOUL.md when they are symlinks to shared files outside the workspace instead of injecting `[MISSING]`.

## Best Possible Solution

Land or replace the scoped fixed-bootstrap symlink implementation with explicit security policy, preserving hardlink, regular-file, size, extra-bootstrap, and sensitive-state protections.

## Reproduction Assessment

Yes, at source level: fixed bootstrap files are opened under the workspace root and any boundary failure is rendered as `[MISSING]`. The issue and follow-up Linux comment provide deterministic symlink layouts, but this read-only review did not run a live filesystem reproduction.

## Solution Assessment

No blanket symlink-following policy is not the best fix. The maintainable path is a narrow fixed-bootstrap policy at the workspace/fs-safe boundary with explicit security tests and maintainer approval.
