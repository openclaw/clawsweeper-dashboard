## Summary

Keep open: current main and the latest release still route fixed workspace bootstrap files through a workspace-root guarded reader, so external symlink targets are still collapsed into missing Project Context entries. The linked implementation PRs cover plausible fixes, but the remaining decision is the security/product boundary for external local files entering prompt context.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files such as SOUL.md when they are symlinks to shared files outside the workspace root.

## Best Possible Solution

Keep this as the canonical user-facing report until the scoped fixed-bootstrap symlink path in https://github.com/openclaw/openclaw/pull/73401 or an equivalent maintainer-approved policy lands, or until maintainers explicitly document external bootstrap symlinks as unsupported.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an external fixed-bootstrap symlink reaches openRootFile under the workspace root, fails the fs-safe boundary contract, is recorded as missing, and renders as [MISSING].

## Solution Assessment

Unclear until maintainers settle the trust model. A scoped fixed-bootstrap reader exception is narrower than relaxing shared boundary checks, while an explicit opt-in or unsupported-policy decision may be preferable if external prompt-context reads must stay explicit.
