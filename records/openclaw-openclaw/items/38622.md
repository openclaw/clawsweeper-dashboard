## Label Justifications

- `P1`: Affected agents can silently start without expected persona, identity, memory, or operating instructions when their workspace bootstrap files are symlinked.
- `impact:session-state`: Rejected bootstrap files remove session context and can change agent behavior across every affected session.
- `impact:security`: Any fix must decide which symlink targets outside the workspace may be trusted and injected into model context.

## Summary

Current main and the latest release still route fixed workspace bootstrap files through a workspace-root bounded read and render any rejected read as missing, so the reported symlink behavior remains unresolved. The issue should stay open because an active maintainer PR is already carrying the focused implementation, but that PR is still unmerged and security-policy-sensitive.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files such as SOUL.md when they are symlinks to readable files outside the workspace instead of injecting them as missing.

## Best Possible Solution

Refresh and land the focused workspace-bootstrap reader fix in https://github.com/openclaw/openclaw/pull/73401 with an explicit security policy and regression tests for allowed symlink hops and rejected sensitive, direct, hardlinked, dangling, non-regular, and oversized targets.

## Reproduction Assessment

Yes at source level: fixed bootstrap files are opened under the workspace root, and any failed guarded read is rendered as missing. I did not run a live filesystem reproduction in this read-only review, but the issue also includes a Linux release reproduction with the same code path.

## Solution Assessment

No final solution has landed yet. The workspace-reader layer is the right likely boundary, but maintainers still need to approve the external symlink trust policy and refresh the open implementation before merge.
