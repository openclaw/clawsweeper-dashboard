## Summary

Keep open: current main still marks external symlinked workspace bootstrap files as missing, and the related open implementation PRs need maintainer trust-boundary review rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to follow symlinks for workspace bootstrap files such as `SOUL.md` instead of injecting them as `[MISSING]`.

## Best Possible Solution

Land or explicitly supersede a scoped fixed-bootstrap symlink reader with clear blocked-symlink diagnostics and regression coverage, while preserving regular-file, size, hardlink, non-regular-file, dangling-link, private-state, and extra-bootstrap protections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an external symlinked `SOUL.md` reaches `openRootFile` under the workspace root, fails the root-file symlink boundary, is recorded as missing, and is rendered as `[MISSING] Expected at: ...`; a fresh Linux reproduction in the issue discussion matches that path.

## Solution Assessment

Unclear until maintainers settle the trust model. The scoped fixed-bootstrap approach in https://github.com/openclaw/openclaw/pull/73401 is narrower than the broader allowlist in https://github.com/openclaw/openclaw/pull/38981, but both change what external local files may feed prompt bootstrap content.
