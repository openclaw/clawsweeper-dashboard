## Summary

Keep open: current main and the latest release still route fixed workspace bootstrap files through a workspace-root guarded reader, so external symlink targets still become missing Project Context entries. The issue is canonical for this user-facing behavior, and the remaining path is maintainer/security review of the scoped fix in https://github.com/openclaw/openclaw/pull/73401 or an equivalent policy decision.

## What This Changes

The issue requests that workspace bootstrap file injection load symlink targets instead of rendering symlinked files as `[MISSING]`.

## Best Possible Solution

Keep this as the canonical report until https://github.com/openclaw/openclaw/pull/73401 or an equivalent scoped fixed-bootstrap symlink policy lands with regression tests and security review, or until maintainers explicitly document external bootstrap symlinks as unsupported with a clear blocked diagnostic.

Yes. Source inspection gives a high-confidence path: an external fixed-bootstrap symlink reaches openRootFile under the workspace root, fails the fs-safe boundary contract, is recorded as missing, and renders as `[MISSING]`.

## Solution Assessment

Unclear until maintainers settle the trust model. A scoped fixed-bootstrap reader exception is narrower than relaxing shared boundary checks, while an explicit opt-in or unsupported-policy decision may be preferable if external prompt-context reads must stay explicit.
