## Summary

Keep open: current main and v2026.5.12 still route fixed workspace bootstrap files through the workspace-root guarded reader, so a symlinked SOUL.md pointing outside the workspace is still recorded and rendered as missing. This is the canonical open report after the duplicate at https://github.com/openclaw/openclaw/issues/40210 was closed, and the remaining implementation path is active but unresolved in open PRs such as https://github.com/openclaw/openclaw/pull/73401, https://github.com/openclaw/openclaw/pull/38981, and https://github.com/openclaw/openclaw/pull/64473.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files such as SOUL.md when they are symlinks to readable files outside the workspace instead of rendering them as missing.

## Best Possible Solution

Land or explicitly supersede a scoped fixed-bootstrap symlink implementation with negative security coverage, while avoiding a global relaxation of shared fs-safe boundary primitives.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a sibling-workspace symlinked SOUL.md reaches openRootFile under the workspace root, fails the root boundary contract, is marked missing by loadWorkspaceBootstrapFiles, and is rendered as `[MISSING] Expected at: ...`; I did not create a filesystem fixture because this review is read-only.

## Solution Assessment

Unclear until maintainers settle the trust model. A scoped fixed-bootstrap-reader exception looks safer than a broad allowedExternalPaths feature or a generic ~/.openclaw boundary fallback, but the open implementation still needs maintainer security review.
