## Summary

Keep open: current main still routes workspace bootstrap files through a workspace-root guarded reader, so an external symlinked SOUL.md can still be marked missing. The recent Linux reproduction and open implementation PRs show this remains active; the blocker is maintainer trust-boundary review, not stale cleanup.

## What This Changes

The issue requests that workspace bootstrap files such as SOUL.md load normally when they are symlinks to shared files outside the workspace directory instead of rendering as [MISSING].

## Best Possible Solution

Land or explicitly supersede the scoped bootstrap-reader fix that preserves the existing regular-file, size, hardlink, dangling-link, non-regular-file, private-state, and extra-bootstrap protections while improving diagnostics for boundary-blocked symlinks.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a sibling-workspace symlinked SOUL.md reaches openRootFile under the workspace root, fails the fs-safe boundary contract, is marked missing by loadWorkspaceBootstrapFiles, and is rendered as [MISSING] Expected at: <path>.

## Solution Assessment

Unclear until maintainers settle the trust model. The scoped fixed-bootstrap-reader approach in https://github.com/openclaw/openclaw/pull/73401 is narrower than the broader allowedExternalPaths proposal, but it still changes which external files may feed prompt bootstrap content.
