## Summary

Keep open: current main and latest v2026.5.19 still route fixed workspace bootstrap files through the workspace-root guarded reader, so symlinks whose targets resolve outside the workspace become missing Project Context entries. Related open PRs now cover plausible fixes, especially the scoped maintainer-labeled replacement PR, but the remaining decision is the security/product boundary for external local files entering prompt context.

## What This Changes

Allow workspace bootstrap files such as SOUL.md and AGENTS.md that are symlinks to shared files outside the workspace to load into Project Context instead of rendering as missing.

## Best Possible Solution

Keep the shared fs-safe/root-boundary helpers strict by default, then land or explicitly reject a scoped fixed-bootstrap symlink policy with clear blocked-symlink diagnostics and preserved hardlink, dangling-link, non-regular, oversized, private-state, and extra-bootstrap protections.

## Reproduction Assessment

Yes. Source inspection plus the fs-safe 0.2.7 contract gives a high-confidence path: an external symlinked bootstrap file fails the workspace-root boundary read, is stored as missing, and then renders as [MISSING] Expected at: <path>.

## Solution Assessment

Unclear until maintainers settle the trust model. The scoped fixed-bootstrap approach in https://github.com/openclaw/openclaw/pull/73401 is narrower than broad boundary relaxation or a global allowlist, but any accepted path needs explicit security boundaries and diagnostics.
