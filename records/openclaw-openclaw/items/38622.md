## Summary

Keep open: current main and v2026.5.12 still route fixed workspace bootstrap files through the workspace-root guarded reader, so a symlinked SOUL.md resolving outside the workspace is still treated as missing. The issue is already paired with open implementation work, so the useful path is maintainer review of the scoped bootstrap-reader fix and its trust-boundary implications rather than cleanup closure.

## What This Changes

Symlinked workspace bootstrap files such as SOUL.md should be followed and injected instead of being rendered as missing when their target is outside the workspace directory.

## Best Possible Solution

Land or explicitly supersede a scoped fixed-bootstrap reader change that follows intentional symlinks only for fixed bootstrap files while preserving hardlink, dangling-link, non-regular-file, oversized-file, private-state, and extra-bootstrap protections.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a sibling-workspace symlinked SOUL.md reaches openRootFile under the workspace root, fails the fs-safe boundary contract, is marked missing by loadWorkspaceBootstrapFiles, and is rendered as [MISSING] Expected at: ... .

## Solution Assessment

Unclear until maintainers settle the trust model. The scoped fixed-bootstrap-reader approach in https://github.com/openclaw/openclaw/pull/73401 is narrower than a broad allowedExternalPaths configuration, but it still changes which external files may feed prompt bootstrap content.
