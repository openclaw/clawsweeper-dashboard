## Summary

Keep open: current main and the latest release still route fixed workspace bootstrap files through the workspace-root boundary reader, so external symlink targets collapse into missing prompt context. Related open PRs cover fix paths, but the remaining decision is the security/product policy for which external files may enter model context.

## What This Changes

The issue asks OpenClaw to follow symlinked workspace bootstrap files such as SOUL.md instead of injecting them as missing context.

## Best Possible Solution

Land or explicitly reject a scoped bootstrap-symlink policy with clear blocked-symlink diagnostics, while preserving hardlink, size, non-regular-file, dangling-link, private-state, and extra-bootstrap boundary protections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an external symlinked bootstrap file fails the workspace-root boundary reader, is stored as missing, and renders as `[MISSING] Expected at: ...`.

## Solution Assessment

Unclear until maintainers settle the trust model. The scoped fixed-bootstrap approach in https://github.com/openclaw/openclaw/pull/73401 has a smaller blast radius than the broader configurable allowlist in https://github.com/openclaw/openclaw/pull/38981, but either path needs explicit security boundaries and diagnostics.
