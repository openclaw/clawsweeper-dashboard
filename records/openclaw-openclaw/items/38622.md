## Summary

This should stay open because current `main` and the latest release still treat external symlinked bootstrap files as failed guarded reads, while the active fix path needs maintainer/security review of the workspace trust boundary.

## What This Changes

Workspace bootstrap files such as `SOUL.md` should load through symlinks instead of rendering as `[MISSING]` when the target is a readable shared file.

## Best Possible Solution

Keep this issue open until maintainers either land a scoped, security-reviewed bootstrap symlink reader or document external bootstrap symlinks as unsupported.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an external bootstrap symlink reaches `openRootFile`, fs-safe rejects the symlink escape, and the bootstrap loader renders the failed read as `[MISSING]`; I did not run a live filesystem repro in this read-only review.

## Solution Assessment

Unclear as a final policy decision. A scoped fixed-bootstrap fallback appears safer than broadening the shared boundary resolver, but maintainers still need to decide the allowed external target model and private-state exclusions.
