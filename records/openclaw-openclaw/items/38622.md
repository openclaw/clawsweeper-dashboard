## Summary

Keep open: current main still collapses external symlinked bootstrap reads to missing, and open implementation PRs still own the trust-boundary decision. This is a real session-state bug, but the fix changes what local files can enter prompt bootstrap context, so it needs maintainer/security review rather than stale cleanup.

## What This Changes

The issue requests that symlinked workspace bootstrap files such as `SOUL.md` be followed and injected instead of rendered as missing.

## Best Possible Solution

Land or explicitly supersede a scoped bootstrap-symlink fix with clear blocked-symlink diagnostics and regression coverage while preserving size, hardlink, non-regular-file, dangling-link, private-state, and extra-bootstrap protections.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path from an external symlinked bootstrap file through `openRootFile` boundary rejection to a missing entry and `[MISSING]` render; the issue discussion also has a recent packaged Linux reproduction.

## Solution Assessment

Unclear until maintainers settle the trust model. The narrow fixed-bootstrap approach in https://github.com/openclaw/openclaw/pull/73401 has a smaller blast radius than the broader allowlist in https://github.com/openclaw/openclaw/pull/38981, but both change what external local files can feed prompt bootstrap content.
