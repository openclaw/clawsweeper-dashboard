## Summary

Keep open: current `main` and the latest release still route workspace bootstrap reads through a root-bounded fs-safe open, so external symlink targets remain rejected and rendered as missing. The active replacement PR is open but conflicting and security-sensitive, so it is not a merged implementation or safe close target yet.

## What This Changes

The issue requests that symlinked workspace bootstrap files load their target contents instead of appearing as missing in injected project context.

## Best Possible Solution

Land or replace the scoped bootstrap-reader fix after maintainers choose the exact symlink trust policy, keeping arbitrary direct external paths, extra/glob bootstrap paths, hardlinks, dangling links, non-regular files, and oversized targets rejected.

## Reproduction Assessment

Yes, at source level. Current `main` opens bootstrap files under the workspace root and renders any boundary validation failure as missing; the issue and later Linux comment provide deterministic symlink layouts, but this read-only review did not run a live filesystem repro.

## Solution Assessment

Unclear. The issue is valid, but the best fix needs maintainer security judgment between a scoped fixed-bootstrap symlink reader and clearer blocked-by-boundary diagnostics; a broad external-path allowance would be too risky.
