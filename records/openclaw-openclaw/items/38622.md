## Summary

Keep open: current main and v2026.5.12 still route workspace bootstrap files through a workspace-root guarded reader, so an external symlinked SOUL.md is still marked missing. Prior discussion and the open implementation PRs identify a real fix path, but the remaining decision is the trust boundary for external prompt bootstrap content.

## What This Changes

Allow symlinked workspace bootstrap files such as SOUL.md to load normally instead of appearing as [MISSING].

## Best Possible Solution

Land or explicitly supersede a scoped fixed-bootstrap symlink reader that preserves regular-file, size, hardlink, dangling-link, non-regular-file, private-state, and extra-bootstrap protections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a sibling-workspace symlinked SOUL.md reaches openRootFile under the workspace root, fails the fs-safe boundary contract, is marked missing by loadWorkspaceBootstrapFiles, and is rendered as [MISSING] Expected at: ... .

## Solution Assessment

Unclear until maintainers settle the trust model. The scoped fixed-bootstrap-reader approach in https://github.com/openclaw/openclaw/pull/73401 is narrower than the broad allowedExternalPaths and boundary-path alternatives, but it still changes which external files may feed prompt bootstrap content.
