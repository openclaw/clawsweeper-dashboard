## Summary

Keep open: current main still routes workspace bootstrap reads through the root-boundary file opener and collapses external-symlink rejection to a missing bootstrap entry. The issue is already paired with open implementation/product-decision PRs, so this is not a stale cleanup close.

## What This Changes

The issue asks OpenClaw to load symlinked workspace bootstrap files such as SOUL.md instead of injecting them as missing.

## Best Possible Solution

Land or explicitly supersede a scoped bootstrap-symlink fix with clear blocked-symlink diagnostics and regression coverage while preserving size, hardlink, non-regular-file, dangling-link, private-state, and extra-bootstrap protections.

## Reproduction Assessment

Yes. Source inspection on current main traces external symlinked bootstrap files through `openRootFile` boundary rejection, `loadWorkspaceBootstrapFiles` missing entries, and `[MISSING]` rendering; the issue discussion also includes a recent packaged Linux reproduction.

## Solution Assessment

Unclear until maintainers settle the trust model. The narrow fixed-bootstrap approach in https://github.com/openclaw/openclaw/pull/73401 has a smaller blast radius than the broader allowlist in https://github.com/openclaw/openclaw/pull/38981, but both change what external local files can feed prompt bootstrap content.
