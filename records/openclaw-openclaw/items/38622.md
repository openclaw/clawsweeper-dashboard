## Summary

Keep open: current main still rejects external symlinked workspace bootstrap files through the boundary-safe reader, collapses that rejection into a missing bootstrap entry, and renders the reported `[MISSING]` marker. The issue also has recent packaged reproduction context and open fix/product-decision PRs, so closing it would preempt the active resolution path.

## What This Changes

The issue asks OpenClaw to load, or at least clearly diagnose, symlinked workspace bootstrap files instead of injecting `[MISSING]` when the symlink target lives outside the workspace root.

## Best Possible Solution

Land or explicitly supersede a scoped bootstrap-symlink fix with blocked-symlink diagnostics and regression coverage while preserving size, hardlink, non-regular-file, dangling-link, private-state, and extra-bootstrap protections.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: an external symlinked bootstrap file goes through `openRootFile`, is rejected by the root boundary, then becomes `missing: true` and renders `[MISSING]`. A recent issue comment also reports the same behavior on packaged Linux v2026.5.7.

## Solution Assessment

Unclear until maintainers settle the trust model. The narrower fixed-bootstrap approach in https://github.com/openclaw/openclaw/pull/73401 has a smaller blast radius than the broader configurable allowlist in https://github.com/openclaw/openclaw/pull/38981, but either path must keep the security guards explicit.
