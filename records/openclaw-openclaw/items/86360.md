## Summary

This PR is still necessary because current main does not persist or reuse the bound Codex app-server agent id for no-session bound turns. The fresh-binding fix is in the right area, but the branch still intentionally leaves existing saved bindings without `data.agentId` on the old fallback path, so merge needs a maintainer security/upgrade decision.

## What This Changes

The PR stores optional Codex app-server binding `agentId`, passes `/codex bind` session identity into new bindings, threads that identity through native execution/runtime policy resolution, and adds focused regression tests.

## Best Possible Solution

Merge only after legacy saved bindings either recover the bound agent deterministically, fail closed with rebind guidance, or have explicit maintainer acceptance for fresh-binding-only enforcement.

## Reproduction Assessment

Yes from source inspection: current main stores no bound agent id, and app-server inbound guard/runtime policy use session-key/default-agent inference. A no-session bound turn for a non-default agent can therefore miss that agent's `tools.exec.host=node` policy.

## Solution Assessment

No, not merge-ready as proposed. Passing the bound agent id is the right fresh-binding fix, but the best fix must handle old binding records or explicitly fail closed/accept the upgrade gap.
