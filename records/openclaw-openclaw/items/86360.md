## Summary

Keep this protected maintainer PR open: the fix direction is right, but the patch still leaves pre-existing persisted Codex app-server bindings without an agent id on the old path, and it appears to make existing `/codex bind` command tests stale.

## What This Changes

The PR stores a Codex app-server conversation binding agent id and passes it into the native execution guard so bound turns can respect per-agent `tools.exec.host=node` policy.

## Best Possible Solution

Carry the binding agent id for new records, resolve or fail closed for existing v1 bindings that only have `agentDir`, and update the broader Codex command tests so the full suite matches the new contract.

## Reproduction Assessment

Yes, from source inspection: current main can block node exec when an agent id is known, but persisted app-server binding data lacks that id and no-session inbound claims pass only the session key into the guard. The untested legacy-record path remains source-reproducible on the PR because old v1 binding data still reads with `agentId` undefined.

## Solution Assessment

No, not yet. Persisting `agentId` is the right narrow fix for new bindings, but the best solution also handles existing v1 bindings or fails closed when the bound agent cannot be resolved, and updates the stale command tests.
