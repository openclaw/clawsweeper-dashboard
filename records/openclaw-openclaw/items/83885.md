## Summary

The current Twitch client path still caches the `ChatClient` immediately and logs success before Twurple has emitted a connect/auth event. The reported `await client.connect()` fix is not compatible with Twurple 8.1.4 because `connect()` returns `void`, so this should stay open as a real but narrowly scoped lifecycle cleanup that follows Twurple event semantics instead of awaiting a promise.

## What This Changes

The issue asks OpenClaw to avoid caching or reporting a Twitch chat client as connected before the initial connection/auth lifecycle is actually known.

## Best Possible Solution

Update the Twitch client manager to model Twurple's event lifecycle explicitly: log connected only from `onConnect`, keep or remove cached clients based on concrete disconnect/auth terminal states, and add focused tests for authentication failure and retry/cache behavior.

## Reproduction Assessment

Yes for the source-level lifecycle problem: current `main` starts `client.connect()`, caches the client, and logs success before any Twurple connect/auth event. No for the full invalid-credential runtime claim, because I did not run a live Twitch credential failure path and Twurple's contract uses events rather than a rejected `connect()` promise.

## Solution Assessment

No, not as written: awaiting `client.connect()` is not the best fix because Twurple 8.1.4 returns `void`. The maintainable fix should register Twurple lifecycle handlers and add tests that prove cache/log behavior across connect, auth failure, disconnect, and retry states.

## Repair Work Prompt

Fix the Twitch client manager lifecycle for https://github.com/openclaw/openclaw/issues/83885. In `extensions/twitch/src/twitch-client.ts`, do not treat `ChatClient.connect()` as awaitable; Twurple 8.1.4 declares `connect(): void` and reports connection/auth state through events. Register lifecycle handlers before starting the connection so `Connected to Twitch as ...` is logged only after `onConnect`, authentication failures and disconnects are logged with useful context, and the `clients` map does not retain a terminally unusable client in a way that prevents a clean later setup. Be careful not to fight Twurple's built-in auth retry loop by spawning duplicate reconnecting clients. Add focused regression tests in `extensions/twitch/src/twitch-client.test.ts` by extending the ChatClient mock with `onConnect`, `onAuthenticationFailure`, and/or `onDisconnect` hooks; cover the existing reuse path plus failure/cache cleanup or retry-safe behavior. Do not add a new config option or move Twitch ownership into core. Changelog is likely needed only if the user-visible Twitch connection behavior changes.
