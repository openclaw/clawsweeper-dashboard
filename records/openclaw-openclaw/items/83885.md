## Summary

Keep open: current main and the latest release still cache the Twitch `ChatClient` immediately after `connect()` and log success before Twurple emits connect/auth lifecycle events. The report is valid, but the suggested `await client.connect()` shape is not the right fix because Twurple 8.1.4 exposes `connect(): void`.

## What This Changes

The issue reports that Twitch `ChatClient.connect()` is started without lifecycle-aware cache cleanup, allowing a failed connection to remain in the client cache.

## Best Possible Solution

Update the Twitch client manager to model Twurple lifecycle events explicitly: log connected only from `onConnect`, clean up or refresh cached clients on terminal auth/disconnect states, preserve Twurple reconnect semantics, and add focused regression tests for retry/cache behavior.

## Reproduction Assessment

Yes for the source-level lifecycle problem: current main starts `client.connect()`, caches the client, and logs success before any Twurple connect/auth event. I did not live-reproduce the full invalid-credential Twitch failure path.

## Solution Assessment

No, not as written: awaiting `client.connect()` is not the best fix because Twurple 8.1.4 returns `void`. The maintainable fix should use Twurple lifecycle handlers and prove cache/log behavior across connect, auth failure, disconnect, and retry states.
