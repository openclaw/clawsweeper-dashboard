## Summary

This should stay open: current main still caches the Twitch `ChatClient` and logs success immediately after starting Twurple connection setup, before any connect/auth lifecycle event confirms success. The requested await-based fix is not compatible with Twurple 8.1.4, so the repair should use Twurple event handlers instead.

## What This Changes

The issue asks OpenClaw to avoid storing or reporting a Twitch chat client as connected until the underlying Twurple connection/auth lifecycle has succeeded.

## Best Possible Solution

Update the Twitch client manager to model Twurple lifecycle events explicitly: log connected from a real connection event, invalidate cached clients on terminal auth/disconnect states, and cover retry/cache behavior with focused tests.

## Reproduction Assessment

Yes for the source-level lifecycle problem: current main starts `client.connect()`, caches the client, and logs success before any Twurple connect/auth event. The live invalid-credential path was not run, so that runtime claim remains medium confidence.

## Solution Assessment

No, not as written: awaiting `client.connect()` is not the best fix because Twurple 8.1.4 declares `connect(): void`. The maintainable solution is to use Twurple lifecycle events and add regression coverage for auth failure, disconnect cleanup, and retry behavior.
