## Summary

Keep open: current main still starts the Twitch `ChatClient`, stores it in the manager cache, and logs success before Twurple emits any connect/auth lifecycle event. The report points at a real lifecycle bug, but the suggested `await client.connect()` shape is not valid for Twurple 8.1.4 because `connect()` returns `void`.

## What This Changes

The issue reports that Twitch `getClient` caches and logs a `ChatClient` immediately after starting connection, so auth or connection failure can leave a broken client reused from the clients map.

## Best Possible Solution

Model Twurple lifecycle explicitly in the Twitch client manager: log connected only from `onConnect`, handle terminal auth/disconnect states without duplicating Twurple retry behavior, and add focused cache/retry tests.

## Reproduction Assessment

Yes for the source-level lifecycle problem: current main calls `connect()`, then caches and logs success before any Twurple connect/auth event. No for the full live invalid-credential failure path, because that needs real Twitch credential proof.

## Solution Assessment

No to the issue's await-based suggestion, because Twurple 8.1.4 declares `connect(): void`. The maintainable fix should use Twurple lifecycle events and regression tests for auth failure, disconnect, retry, and cache reuse.
