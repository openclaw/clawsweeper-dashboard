## Summary

Keep open: current main still caches and logs the Twitch ChatClient immediately after fire-and-forget connect, before Twurple emits a connect or auth event; the issue is valid, but the await-based wording needs an event-driven fix.

## What This Changes

The issue asks for Twitch getClient to avoid storing or logging a connected ChatClient before the underlying Twurple connection/auth lifecycle has actually succeeded.

## Best Possible Solution

Update TwitchClientManager to track Twurple lifecycle events explicitly, log connected only from onConnect, clean up unusable cached clients on terminal auth/connect failure, and add focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls client.connect(), then caches and logs the client before any Twurple onConnect or auth event. I did not run live invalid Twitch credentials, so the exact runtime auth-failure symptom should be proven in the fix PR if feasible.

## Solution Assessment

No, not the suggested await shape: @twurple/chat 8.1.4 declares connect(): void. The maintainable fix should use Twurple lifecycle handlers for logging and cache cleanup, with tests for auth/connect failure and retry behavior.
