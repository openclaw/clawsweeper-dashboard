## Summary

## What This Changes

The issue asks OpenClaw to stop caching or logging a Twitch chat client as connected before the initial Twurple connection/auth lifecycle has actually succeeded.

## Best Possible Solution

Update `TwitchClientManager` to model Twurple lifecycle events explicitly: log successful connection from `onConnect`, clear the cache for the same client/key on terminal auth/disconnect failure, and cover retry/cache behavior with focused tests.

## Solution Assessment

No, not as suggested with `await client.connect()`: Twurple 8.1.4 exposes a void `connect()` and reports connection/auth state through events. The maintainable fix is event-driven cache/log handling with regression tests.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83885 in the Twitch plugin only. `TwitchClientManager.getClient` currently calls `client.connect()`, immediately stores the client in `this.clients`, and logs success before Twurple emits any connect/auth lifecycle event. Do not change core channel policy and do not implement `await client.connect()` because `@twurple/chat` 8.1.4 declares `connect(): void`. Register Twurple lifecycle handlers before `connect()`: log success from `onConnect`, clear the cached entry only when the event corresponds to the same client/key on authentication failure or terminal disconnect, and avoid fighting Twurple's built-in reconnect/auth retry behavior. Add focused tests in `extensions/twitch/src/twitch-client.test.ts` with mocks for `onConnect`, `onAuthenticationFailure`, and `onDisconnect` proving no success log before `onConnect`, cache cleanup on failure/disconnect, and fresh setup on a later `getClient`. Preserve existing send/message behavior. Validation: `node scripts/run-vitest.mjs extensions/twitch/src/twitch-client.test.ts` and `git diff --check`. User-facing fix should receive a changelog entry during maintainer landing if this becomes a shipped repair.
