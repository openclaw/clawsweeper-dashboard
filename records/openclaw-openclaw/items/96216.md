## Summary

Keep this PR open: it is a focused security hardening change, but the latest patch still applies version-suffix stripping to every interpreter family, which misclassifies unrelated commands such as `r2 -e` as R inline eval. The external PR also still has mock/test-only behavior proof, so it needs a code correction, maintainer/security review, and real redacted runtime proof before merge.

## What This Changes

The PR expands strict inline-eval detection to versioned interpreter executables, PHP/R inline flags, and matching allowlist-pattern suppression with detector and node-host tests.

## Best Possible Solution

Land a narrow detector hardening that only strips version suffixes for known versioned interpreter families or otherwise avoids one-character collisions, keeps contract-backed PHP/R coverage, adds a negative `r2 -e` regression test, and includes real node-host proof before maintainer merge.

## Reproduction Assessment

Yes at source level: current main exact-matches interpreter names and the strictInlineEval path consumes that detector, while the PR's global suffix stripping makes `r2 -e` match the new R spec. I did not execute a live node-host run in this read-only review.

## Solution Assessment

No as written: centralizing interpreter matching is the right boundary, but applying suffix stripping to every interpreter family is too broad. Limit version stripping to known versioned families or avoid one-character bases, then re-prove the runtime path.
