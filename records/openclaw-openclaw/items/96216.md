## Summary

Keep open: this is useful exec approval hardening and the unsupported R `--exec` case was removed, but the current version-suffix matcher can still misclassify unrelated `r2`/radare2 commands as R and the contributor proof is still Vitest/handler-test output rather than a real setup.

## What This Changes

The PR expands strict inline-eval detection for version-suffixed interpreter names plus PHP/R inline-code flags and adds detector and node-host regression coverage.

## Best Possible Solution

Land a narrow detector hardening that only strips version suffixes for known versioned interpreter families, keeps PHP/R flag coverage contract-backed, and includes real node-host proof plus maintainer acceptance of the stricter prompts.

## Reproduction Assessment

Yes at source level: current main exact-matches interpreter names and the strictInlineEval path consumes that detector. I did not execute a live node-host run in this read-only review.

## Solution Assessment

No as written: centralizing interpreter matching is the right boundary, but applying suffix stripping to every interpreter name misclassifies `r2`/radare2 as R. Limit version stripping to known versioned families or avoid one-character bases, then re-prove the runtime path.
