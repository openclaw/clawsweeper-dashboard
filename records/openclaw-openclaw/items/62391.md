## Summary

Keep open. Current main still drops Slack Web API structured fields in the recovery path: delivery queue recovery formats thrown errors through the generic formatter, and that formatter keeps only `Error.message` plus causes for `Error` instances. Slack Web API platform errors attach the useful `missing_scope` details under `error.data`, so the queue `lastError` and recovery logs can still lose `needed` and granted-scope data. The related open Slack logging PR improves other Slack logging sites but does not touch the outbound send/delivery-queue path.

## What This Changes

The issue asks OpenClaw to preserve Slack Web API `missing_scope` details such as `needed` and granted scopes in delivery queue `lastError` values and recovery logs.

## Best Possible Solution

Add a Slack-owned, redaction-safe API error formatter at the outbound send boundary, rethrowing or wrapping Slack Web API failures with an enriched message before they cross into core delivery queue code. Keep core delivery recovery generic, add focused tests for `missing_scope` `needed` and `response_metadata.scopes`/`acceptedScopes`, and add a user-facing changelog entry crediting the reporter.

## Reproduction Assessment

Yes. A deterministic unit reproduction is enough: make Slack outbound delivery throw an `Error` with Slack-style `data.needed` and `data.response_metadata.scopes`, then run the delivery queue failure/recovery path; current main will persist only `err.message` because `formatErrorMessage` ignores `err.data` for `Error` instances.

## Solution Assessment

No for current main: the implementation still relies on the generic `Error.message` path. The narrow maintainable fix is to format Slack SDK errors inside the Slack plugin's outbound send path, rather than teaching core delivery recovery about Slack-specific fields.
