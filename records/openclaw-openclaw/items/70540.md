## Summary

Keep open: current main still lacks a core `memory_add` write primitive, but this PR is draft, conflicting, missing real behavior proof, and has correctness/security blockers around plugin contracts, block boundaries, and write-tool permissions.

## What This Changes

The PR branch adds a memory-core `memory_add` tool, daily Markdown memory block parsing, gateway invocation support, tool catalog/display metadata, docs, and tests.

## Best Possible Solution

Land a reworked memory-core write slice only after write authority is explicit, block-boundary appends are robust, manifest/docs/gateway surfaces match current main, and real behavior proof is provided.

## Reproduction Assessment

Yes. Source inspection is enough for this review: current main exposes only `memory_search` and `memory_get`, and the PR head shows the proposed write path plus manifest, policy, docs, and append-boundary blockers.

## Solution Assessment

No. A memory-core write primitive is plausible, but this patch is not yet the narrowest maintainable solution because it broadens existing memory policy, misses current plugin manifest contracts, mishandles separator-free daily files, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Declare memory_add in the plugin manifest:** `extensions/memory-core/index.ts:60-66`
  - body: Current main rejects plugin tool registrations that are not declared in `contracts.tools`. This PR registers `memory_add` but does not update memory-core's manifest contract, so the new tool can be dropped with a plugin diagnostic after rebase. Add `memory_add` to the manifest contract alongside the registration.
  - confidence: 0.88
- **[P2] Separate new blocks after existing raw memory:** `extensions/memory-core/src/memory/add-memory-block.ts:47-52`
  - body: When today's memory file already exists without a trailing `----` separator, this only inserts blank-line spacing before the new formatted block. `parseMemoryBlocks` will then treat the prior raw content and the newly added text as one semantic block until the new trailing separator, so the added memory is not indexed as its own unit.
  - confidence: 0.91
- **[P2] Keep memory writes out of existing memory groups:** `src/agents/tool-catalog.ts:145-150`
  - body: Putting `memory_add` in section `memory` and profile `coding` makes existing `group:memory`, `group:openclaw`, and coding-profile grants include a persistent write tool. Before this change that group was recall-only, so gate the write primitive behind an explicit tool name or a separately reviewed write group/profile.
  - confidence: 0.9
- **[P3] Keep public tool-policy docs aligned:** `docs/gateway/configuration-reference.md:2222`
  - body: This updates some policy docs to say `group:memory` includes `memory_add`, but the current primary tools/custom providers page still lists only `memory_search` and `memory_get`. If the tool remains in this group after permission review, update the current docs surface too so users do not get conflicting policy guidance.
  - confidence: 0.8
