## Summary

Keep open. The proposed `memory_add` slice is still useful and not implemented on current main, but the PR has blocking correctness, security, and real-behavior-proof gaps before it can merge.

## What This Changes

The PR adds a memory-core `memory_add` tool that appends Markdown memory blocks to daily files, makes daily-memory indexing block-aware, and updates tool policy, display metadata, docs, tests, and gateway invoke handling.

## Best Possible Solution

Land a reworked memory-core write slice only after write-tool authority is explicit, block-boundary appends are robust, plugin manifest/docs/gateway surfaces are aligned with current main, and real behavior proof is provided.

## Reproduction Assessment

Yes. Static source inspection is enough: current main has no `memory_add`, and the PR diff shows the proposed write path, policy broadening, and manifest/docs mismatches without needing to run tests in this read-only review.

## Solution Assessment

No. A memory-core write primitive is plausible, but this patch is not yet the safest maintainable solution because it broadens existing memory policy, misses current plugin manifest contracts, mishandles separator-free daily files, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Declare memory_add in the plugin manifest:** `extensions/memory-core/index.ts:60-66`
  - body: Current main rejects plugin tool registrations that are not declared in `contracts.tools`. This PR registers `memory_add` but leaves memory-core's manifest without that tool contract, so the new tool can be dropped with a plugin diagnostic after rebase. Add `memory_add` to the manifest contract alongside the registration.
  - confidence: 0.86
- **[P2] Separate new blocks after existing raw memory:** `extensions/memory-core/src/memory/add-memory-block.ts:47-52`
  - body: When today's memory file already exists without a trailing `----` separator, this only inserts blank-line spacing before the new formatted block. `parseMemoryBlocks` will then treat the prior raw content and the newly added text as one semantic block until the new trailing separator, so the added memory is not indexed as its own unit. Close or otherwise separate any existing raw tail before appending the new block.
  - confidence: 0.91
- **[P2] Keep memory writes out of existing memory groups:** `src/agents/tool-catalog.ts:145-150`
  - body: Putting `memory_add` in section `memory` and profile `coding` makes existing `group:memory`, `group:openclaw`, and coding-profile grants include a persistent write tool. Before this change that group was recall-only, so gate the write primitive behind an explicit tool name or a separately reviewed write group/profile instead of widening existing configs on upgrade.
  - confidence: 0.9
- **[P3] Keep public tool-policy docs aligned:** `docs/gateway/configuration-reference.md:2222`
  - body: This updates policy docs to say `group:memory` includes `memory_add`, but the primary Tools and custom providers page still lists only `memory_search` and `memory_get`. If the tool remains in this group after permission review, update `docs/gateway/config-tools.md` too so users do not get conflicting policy guidance.
  - confidence: 0.83
