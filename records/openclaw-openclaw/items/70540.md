## Summary

Keep open. Current main still lacks the requested memory write primitive, and this draft PR remains useful source work, but it has concrete blockers around markdown block isolation and permission broadening for existing memory tool groups. It also needs a rebase pass because current main moved direct tool invocation into the shared gateway helper.

## What This Changes

The PR adds a memory-core `memory_add` tool that appends Markdown memory blocks to daily memory files, makes daily-memory indexing split on `----`, and updates catalog, gateway invoke handling, display metadata, docs, and tests.

## Best Possible Solution

Land a focused memory-core write slice after explicit approval for write-tool authority, with robust block-boundary handling for existing daily files, current-main gateway invoke wiring, aligned public docs, changelog coverage, and tests for policy and indexing behavior. Broader bounds, duplicate handling, list, replace, and remove flows can remain tracked in the related memory-management issues.

## Reproduction Assessment

Yes. Static inspection is enough to reproduce the current missing feature and the proposed regressions: main has no `memory_add`, while the PR append code can collapse a separator-free daily file with the new block and `CORE_TOOL_GROUPS` expands `memory_add` through `sectionId: "memory"`.

## Solution Assessment

No. Adding a memory-core write primitive is a plausible direction, but this patch is not yet the safest maintainable solution because it silently changes existing memory-group authority, does not isolate appends after raw daily content, misses a duplicated docs page/changelog entry, and needs rebase onto the shared gateway invoke path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Separate new blocks after existing raw memory:** `extensions/memory-core/src/memory/add-memory-block.ts:50-53`
  - body: When today's memory file already exists without a trailing `----` separator, this appends only whitespace before the formatted block. `parseMemoryBlocks` will then treat the prior raw content and the newly added text as one block until the new trailing separator, so the new memory is not indexed as its own semantic unit. Close or separate any existing raw tail before appending the new block.
  - confidence: 0.91
- **[P2] Keep memory writes out of existing memory groups:** `src/agents/tool-catalog.ts:149-150`
  - body: Adding `memory_add` to the `memory` section and `coding` profile makes existing `group:memory` policies grant a persistent write tool; before this change that group was read-only. Gate the write primitive behind an explicit tool name or a separately reviewed write group/profile so existing configs do not gain write authority on upgrade.
  - confidence: 0.89
- **[P3] Update the other gateway tools reference:** `docs/tools/index.md:190`
  - body: This docs table now says `group:memory` includes `memory_add`, but `docs/gateway/config-tools.md` still documents the group as only `memory_search` and `memory_get`. Update the duplicate public config-tools page too so users do not get conflicting policy guidance.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `extensions/memory-core/index.ts:57-65`
  - body: This is a user-facing `feat` that adds a new agent tool, but the PR does not include `CHANGELOG.md`. Repo policy requires user-facing feature changes to add a single-line changelog entry with contributor credit before merge.
  - confidence: 0.86
