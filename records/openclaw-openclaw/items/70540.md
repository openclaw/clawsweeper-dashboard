## Summary

Keep open. Current main still lacks `memory_add`, and this draft PR is the active implementation path, but the proposed patch has blocking correctness and security-policy concerns around daily-file block isolation and existing `group:memory` permission semantics.

## What This Changes

The PR adds a memory-core `memory_add` tool, daily Markdown memory-block parsing/indexing, gateway/catalog/display/docs exposure, and targeted tests for the new write primitive.

## Best Possible Solution

Land a focused memory-core write slice after the permission model explicitly preserves read-only memory group semantics or receives maintainer approval to change them, the append path safely separates existing raw daily content from new blocks, and all duplicated public tool-policy docs are updated with targeted tests and changed-gate proof.

## Reproduction Assessment

Yes. Static inspection is enough here: current main has only `memory_search`/`memory_get`, while the PR patch shows both the raw-tail append behavior and the `sectionId: "memory"` catalog addition that expands existing group policies.

## Solution Assessment

No. The implementation direction is plausible, but this patch is not yet the safest maintainable solution because it couples the write primitive to existing read-oriented policy groups and does not robustly isolate appended blocks in existing daily files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Separate new blocks after existing raw memory:** `extensions/memory-core/src/memory/add-memory-block.ts:50-53`
  - body: When today's memory file already has separator-free content, appending only whitespace before the formatted block makes `parseMemoryBlocks` treat the existing raw tail and the new memory as one block until the new trailing separator. Close or separate the existing raw tail before appending so the new memory indexes as its own semantic unit.
  - confidence: 0.91
- **[P2] Keep memory writes out of existing memory groups:** `src/agents/tool-catalog.ts:148-149`
  - body: Adding `memory_add` to the `memory` section and `coding` profile makes existing `group:memory` and default coding policies grant a persistent write tool. Gate this behind an explicit tool name or separately reviewed write group/profile so read-only memory configs do not gain write authority on upgrade.
  - confidence: 0.88
- **[P3] Update the other gateway tools reference:** `docs/tools/index.md:190`
  - body: This docs table now says `group:memory` includes `memory_add`, but `docs/gateway/config-tools.md` still documents the group as only `memory_search` and `memory_get`. Update the duplicate public config-tools page too so users do not get conflicting policy guidance.
  - confidence: 0.82
