## Summary

Keep open for maintainer review. Current main still lacks `memory_add`, so the PR is not obsolete, but the patch adds a persistent write-capable memory tool and has concrete correctness, docs, and permission-model blockers before merge.

## What This Changes

The PR adds a memory-core `memory_add` tool that appends Markdown blocks to daily memory files, changes daily-memory indexing to split on `----`, and updates tool policy, display metadata, docs, tests, and gateway invoke handling.

## Best Possible Solution

Land a reworked memory-core write slice only after explicit write-tool authority is approved, block-boundary appends are robust, current shared gateway invoke/docs are aligned, and after-fix real behavior proof is provided.

## Reproduction Assessment

Yes. Static inspection is enough to reproduce the current missing feature and the proposed regressions: main has no `memory_add`, while the PR append code can merge an existing raw daily-file tail with the new block and its catalog entry broadens existing memory policies.

## Solution Assessment

No. A memory-core write primitive is plausible, but this patch is not yet the safest maintainable solution because it changes write authority through existing groups/profiles, mishandles separator-free daily files, needs current-main gateway/docs rebase work, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Separate new blocks after existing raw memory:** `extensions/memory-core/src/memory/add-memory-block.ts:50-53`
  - body: When today's memory file already exists without a trailing `----` separator, this only inserts blank-line spacing before the new formatted block. `parseMemoryBlocks` will then treat the prior raw content and the newly added text as one semantic block until the new trailing separator, so the added memory is not indexed as its own unit. Close or otherwise separate any existing raw tail before appending the new block.
  - confidence: 0.91
- **[P2] Keep memory writes out of existing memory groups:** `src/agents/tool-catalog.ts:149-150`
  - body: Putting `memory_add` in section `memory` and profile `coding` makes existing `group:memory` and coding-profile grants include a persistent write tool. Before this change that group was recall-only, so gate the write primitive behind an explicit tool name or a separately reviewed write group/profile instead of widening existing configs on upgrade.
  - confidence: 0.89
- **[P3] Keep public tool-policy docs aligned:** `docs/gateway/configuration-reference.md:2222`
  - body: This updates docs that say `group:memory` includes `memory_add`, but the current public Tools and custom providers page still lists only `memory_search` and `memory_get`. If the tool remains in this group after the permission review, update `docs/gateway/config-tools.md` too so users do not get conflicting policy guidance.
  - confidence: 0.82
