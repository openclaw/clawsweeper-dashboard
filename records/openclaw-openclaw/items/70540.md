## What This Changes

The PR adds a memory-core memory_add tool that writes daily Markdown memory blocks, updates daily-memory indexing, and extends tool policy/docs/display metadata for the new write primitive.

## Best Possible Solution

Land a reworked memory-core write slice only after write-tool authority is explicit, block-boundary appends are robust, shared gateway/docs surfaces are rebased and aligned, and real setup proof is attached.

## Reproduction Assessment

Yes. Source inspection on current main shows there is no memory_add registration, while the PR diff is enough to reproduce the append-boundary and permission-broadening issues without running the branch.

## Solution Assessment

No. A memory-core write primitive is plausible, but this patch is not yet the safest maintainable solution because it widens existing memory grants, mishandles separator-free daily files, leaves one docs page inconsistent, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Separate new blocks after existing raw memory:** `extensions/memory-core/src/memory/add-memory-block.ts:50-53`
  - body: When today's daily file already exists without a trailing `----` separator, this only inserts blank-line spacing before the new formatted block. `parseMemoryBlocks` will keep the old raw tail and the newly added text in one semantic block until the new trailing separator, so `memory_add` may not create a separate searchable unit.
  - confidence: 0.91
- **[P2] Do not put memory_add in existing memory grants:** `src/agents/tool-catalog.ts:149-150`
  - body: Adding `memory_add` to section `memory` and profile `coding` makes existing `group:memory` and coding-profile grants include persistent writes. Keep the write primitive behind an explicit tool name or separately approved write group/profile so read-only memory configs are not widened on upgrade.
  - confidence: 0.89
- **[P3] Update the remaining tool-policy page:** `docs/gateway/configuration-reference.md:2222`
  - body: This updates some group tables to include `memory_add`, but the current tools/custom-providers page still lists `group:memory` as `memory_search` and `memory_get`. If the tool remains in that group after permission review, update that page too so policy docs agree.
  - confidence: 0.82
