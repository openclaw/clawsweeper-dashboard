## Summary

Keep open. Current main still has the Discord-visible failure path: background exec failures enqueue raw output tails as untrusted system events, heartbeat can turn those events into user-relay replies, and the Discord outbound sanitizer has no exec/system-event-specific suppression or rewrite. The related open PRs are adjacent UI/audience work, not a merged fix for this Discord behavior.

## What This Changes

The issue asks OpenClaw to stop posting raw exec failure/system-event text as a separate visible Discord message after exec cards.

## Best Possible Solution

Add a scoped Discord-visible exec completion policy that suppresses or rewrites raw `System (untrusted)` / `Exec failed (...) :: ...` follow-ups while preserving exec cards and intentional sanitized async completion summaries where they are useful.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A source-level reproduction is clear on current main: enqueue a failed background exec notify-on-exit event with Discord delivery context, let the exec-event heartbeat build a user-relay prompt, and observe that Discord outbound has no exec-failure-specific sanitizer before sending the model reply.

## Solution Assessment

Is this the best way to solve the issue? No for current main. The narrow maintainable fix is a scoped heartbeat/outbound policy for Discord-visible exec failures, not the existing global `tools.exec.notifyOnExit=false` knob or a broad system-event redesign as a prerequisite.
