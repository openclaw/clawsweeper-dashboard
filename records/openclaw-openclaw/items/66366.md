## Summary

Keep open. Current main still has the reported Discord-visible path: background exec failures enqueue raw command-tail text as untrusted system events, exec-event heartbeats can ask the model to relay those details, and Discord outbound delivery still lacks a raw exec/system-event suppression or sanitizer policy. Related #72201 is broader/open audience work and #68045 is UI/transcript cleanup, so neither has implemented this specific Discord behavior on main.

## What This Changes

The issue asks OpenClaw to prevent Discord from posting raw exec failure/system-event text as a separate visible chat message after exec cards.

## Best Possible Solution

Add a scoped Discord-visible exec completion policy that suppresses or rewrites raw `System (untrusted)` / `Exec failed (...) :: ...` follow-ups while preserving exec cards and intentional sanitized async completion summaries where they are useful.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A source-level reproduction is clear on current main: enqueue a failed background exec notify-on-exit event with Discord delivery context, let the exec-event heartbeat build a relay prompt, and observe that Discord outbound has no exec-failure-specific sanitizer before sending the model reply.

## Solution Assessment

Is this the best way to solve the issue? No for current main. The narrow maintainable fix is a scoped heartbeat/outbound policy for Discord-visible exec failures, not the existing global `tools.exec.notifyOnExit=false` knob or a broad system-event redesign as a prerequisite.
