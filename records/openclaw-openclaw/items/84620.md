## Summary

Keep this PR open: it targets a real outbound delivery regression, but the current patch is not merge-ready because it makes pinned setup-only channel entries terminal and can break the existing bootstrap path that loads a runtime outbound adapter after the first lookup fails.

## What This Changes

The PR changes outbound adapter resolution to use a pinned channel entry without consulting the active registry, while still falling back to the active registry for channels missing from the pinned registry, and updates channel pinning tests.

## Best Possible Solution

Keep the pinned registry as the stable request-time source, but update outbound bootstrap to repin or explicitly route through the runtime registry it loads, then cover setup-shell bootstrap, registry replacement, and dynamic-channel lookup with tests plus redacted real Discord proof.

## Reproduction Assessment

Yes from source inspection: pin a setup-only channel entry with no outbound, set the active registry to a runtime entry with outbound, and call outbound adapter resolution. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as-is. Removing unstable request-time fallback is directionally useful, but the patch also needs to preserve the existing on-demand bootstrap path by repinning or returning the loaded runtime registry explicitly.
