## Summary

Keep this PR open: it targets a real Discord outbound regression, but the patch removes the active-registry fallback without preserving the existing setup-shell bootstrap path, so it can still leave outbound sends unable to resolve an adapter after on-demand runtime loading.

## What This Changes

The PR removes `getActivePluginRegistry()` fallback resolution from `createChannelRegistryLoader` and changes the channel pinning test to expect outbound lookup to use only the pinned channel registry.

## Best Possible Solution

Keep pinned registry as the stable outbound source, but make on-demand outbound bootstrap repin or explicitly return the loaded runtime registry before removing the active fallback; cover both registry replacement and setup-shell bootstrap cases with tests plus redacted real Discord proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: use a pinned setup-only channel entry with no outbound, set the active registry to a runtime entry with outbound, then call outbound adapter resolution. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as-is. Removing the unstable active fallback is directionally right for the reported regression, but the maintainable fix also has to preserve the existing on-demand bootstrap path by repinning or explicitly returning the runtime registry it loads.
