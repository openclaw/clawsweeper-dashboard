## What This Changes

The PR wraps prompt-release fence bookkeeping in a try/finally so the underlying session write lock is released when fence reads throw, and adds a regression test.

## Best Possible Solution

Land the focused finally-based guard after the contributor adds redacted real behavior proof, keeping the regression test around the lock-release invariant.

## Reproduction Assessment

Yes, source-reproducible: current main clears heldLock before filesystem fence reads that can throw, and the PR adds a targeted fs.stat EIO test for that path. I did not execute the test in this read-only review.

## Solution Assessment

Yes: releasing the copied lock in a finally at the exact point heldLock is cleared is the narrowest maintainable fix. Moving the clear later or swallowing fence errors would either keep the lock longer on failure or hide the real I/O problem.
