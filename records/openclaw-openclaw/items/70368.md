## Summary

Keep open: current main still has the old loopback warning behavior, but this PR head remains incorrect because the audit classifier can suppress exposed-gateway findings from process/container state and config-only custom loopback checks.

## What This Changes

The branch changes gateway security-audit bind classification, adds an `isLoopbackEquivalentBind` helper and tests, and adds redacted audit evidence fixtures under `tools/audit-evidence`.

## Best Possible Solution

Classify gateway audit exposure from the same effective bind host used by runtime, or fail closed for ambiguous/fallback cases, while keeping non-loopback missing-auth/origin/trusted-proxy findings loud.

## Reproduction Assessment

Yes. Source inspection of PR head shows `auto` and custom loopback classification can diverge from runtime bind resolution, and current main still demonstrates the original warning behavior; I did not run tests because this review is read-only.

## Solution Assessment

No. The severity split may be reasonable, but this implementation should use the effective runtime bind host or fail closed before suppressing or downgrading gateway exposure findings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Classify auto from the effective bind host:** `src/config/gateway-bind.ts:41-42`
  - body: `bind="auto"` is still treated as loopback whenever the audit process is not detected as a container. Runtime can resolve the same mode to `0.0.0.0` inside containers or when the `127.0.0.1` bind probe fails, and this predicate now suppresses or downgrades gateway exposure findings.
  - confidence: 0.95
- **[P2] Check custom bind reachability before downgrading:** `src/config/gateway-bind.ts:34-39`
  - body: `customBindHost="127.0.0.1"` is marked loopback from config alone, but runtime falls back to `0.0.0.0` when that host cannot be bound. The audit should use the effective runtime host or fail closed before suppressing non-loopback exposure findings.
  - confidence: 0.9
- **[P3] Force the non-container auto evidence path:** `tools/audit-evidence/capture.sh:105`
  - body: The `auto-bind-non-container` fixture is generated without forcing non-container detection or controlling the bind probe, so on containerized runners it can record the exposed branch instead of proving a bare-metal auto case.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff changes security-audit suppression gates in ways that can hide exposed gateway bind findings.

Concerns:

- **[medium] Auto bind can be misclassified as loopback:** `src/config/gateway-bind.ts:41`
  - body: The classifier depends on the audit process environment for `auto`, while runtime bind resolution can return `0.0.0.0`; using it to suppress or downgrade gateway exposure findings can create audit false negatives.
  - confidence: 0.95
- **[medium] Custom bind fallback is not reflected in audit severity:** `src/config/gateway-bind.ts:34`
  - body: The classifier treats loopback custom hosts as safe without the runtime bind probe, even though runtime falls back to `0.0.0.0` when that host cannot be bound.
  - confidence: 0.9
