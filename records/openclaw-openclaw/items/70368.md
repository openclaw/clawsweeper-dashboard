## Summary

Keep open: the central change is not on current main, and this PR still has a blocking security-audit false-negative risk. The latest commit message and comment say the classifier was changed to use an effective bind host, but the actual head still classifies `bind="auto"` from the audit process environment and the latest commit only added audit evidence files.

## What This Changes

The branch adds a gateway bind classifier, uses it to tier gateway security-audit findings and sibling gates, updates audit tests, and adds redacted audit evidence fixtures.

## Best Possible Solution

Use the same effective bind host that gateway startup resolves, or fail closed for ambiguous `auto` and failed custom binds, then regenerate real audit proof for loopback and exposed-auto scenarios.

## Reproduction Assessment

Yes. Source inspection is enough: current main still has the original loopback warning, and PR head still classifies `auto` with `isContainerEnvironment()` while runtime bind resolution can produce `0.0.0.0`.

## Solution Assessment

No. The useful direction is a context-aware audit severity split, but the implementation should classify from the effective runtime bind host or fail closed before security-audit gates depend on the result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Fail closed for auto bind classification:** `src/config/gateway-bind.ts:41-42`
  - body: `bind="auto"` still returns loopback on any non-container audit process, but `resolveGatewayBindHost` can return `0.0.0.0` when localhost binding fails or when the same config later runs in a container. Since this predicate now controls `bind_no_auth`, allowed-origins, and trusted-proxy severity, exposed deployments can lose critical/warn audit findings; pass the effective bind host into the classifier or keep `auto` non-loopback unless locality is proven.
  - confidence: 0.94
- **[P3] Make the non-container audit evidence force that path:** `tools/audit-evidence/capture.sh:104-106`
  - body: The `auto-bind-non-container` fixture is generated without forcing container detection false, so when the script runs in a container it records the same exposed result as the simulated-container case. That does not prove the bare-metal `auto` behavior that the classifier uses to downgrade security findings.
  - confidence: 0.89
