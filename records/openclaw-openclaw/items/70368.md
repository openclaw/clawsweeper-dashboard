## Summary

Keep this PR open: current main still has the reported loopback warning behavior, but the live PR head still has the security blocker from the earlier review. The latest author comment says the classifier now uses an effective bind host, but the actual head source still classifies `auto` from the audit process environment and `audit.ts` does not pass an effective host into the gateway audit.

## What This Changes

The PR adds a loopback-equivalent gateway-bind classifier, uses it in gateway audit findings to split trusted-proxies severity and sibling exposure checks, expands tests, and adds redacted audit-evidence fixtures.

## Best Possible Solution

Drive gateway audit classification from the same effective bind host used by gateway startup, or fail closed for ambiguous `auto`/fallback cases, then keep non-loopback exposure findings loud and attach redacted live audit output for loopback, bare-metal auto, and exposed-auto cases.

## Reproduction Assessment

Yes. Source inspection is enough for the blocking PR regression: PR head classifies `auto` using `isContainerEnvironment()`, while runtime `auto` can resolve to `0.0.0.0`; I did not run tests because this is a read-only review.

## Solution Assessment

No. The severity split is reasonable, but this implementation should classify from the effective runtime bind host or fail closed for ambiguous binds before security-audit gates depend on it.
