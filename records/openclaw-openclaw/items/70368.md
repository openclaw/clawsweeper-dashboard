## Summary

Keep this PR open for author or maintainer revision. Current main still has the reported loopback warning behavior, so the branch is not obsolete, but the current patch has blocking security-audit false-negative risks because it suppresses critical exposure findings from a classifier that can disagree with runtime bind resolution.

## What This Changes

The PR adds a gateway bind classifier, uses it to tier several gateway security-audit findings for loopback-equivalent versus non-loopback binds, and extends gateway audit tests.

## Best Possible Solution

Revise the branch toward a conservative effective-bind audit model: downgrade only definitely local-only exposure, fail closed for ambiguous auto and fallback-prone custom binds, keep sibling auth/origin/trusted-proxy findings coherent, and update docs plus changelog for the user-visible audit output change.

## Reproduction Assessment

Yes. Current-main behavior is reproducible from the loopback audit source/test path, and the PR regression is reproducible by comparing isLoopbackEquivalentBind for auto/custom against resolveGatewayBindHost fallback paths that can expose 0.0.0.0.

## Solution Assessment

No. The current PR is directionally useful, but classifying from configured bind strings and auditor-process container state is not the safest solution; the safer fix is to classify from an explicit or probed effective bind host, or fail closed when that cannot be proven.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fail closed for auto bind classification:** `src/config/gateway-bind.ts:40`
  - body: `bind="auto"` is classified as loopback whenever the audit process is not in a container, but runtime can still resolve `auto` to `0.0.0.0` if the actual gateway runs in a container or cannot bind `127.0.0.1`. Since this predicate now suppresses `gateway.bind_no_auth` and `gateway.control_ui.allowed_origins_required`, exposed deployments can lose critical findings.
  - confidence: 0.9
- **[P2] Probe custom loopback binds before suppressing checks:** `src/config/gateway-bind.ts:33-38`
  - body: A loopback-looking `customBindHost` is treated as safe from the configured string alone, but `resolveGatewayBindHost` falls back to `0.0.0.0` when that host cannot be bound. In that case the audit suppresses non-loopback auth and allowed-origin criticals for a listener that is actually exposed.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/security/audit-gateway-config.ts:143-189`
  - body: This changes user-visible `openclaw security audit` severity and adds a new non-loopback warning path, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `fix` changes to add a single-line changelog entry with contributor credit.
  - confidence: 0.86

## Security Review

Concerns:
