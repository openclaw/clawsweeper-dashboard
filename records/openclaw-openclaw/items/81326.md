## Summary

Keep open: current main still has a source-level path where a Codex-selected run reaches harness lookup without a registered `codex` harness, and the non-mutating diagnostics do not yet clearly report the missing Codex owner plugin. The remaining work is narrow because the issue discussion already points away from gateway startup failure and toward a doctor diagnostic/repair path.

## What This Changes

Diagnose and repair configs that select the Codex agent runtime when the `@openclaw/codex` plugin is absent or disabled.

## Best Possible Solution

Add a non-mutating `openclaw doctor` diagnostic, with matching `doctor --fix` repair coverage, for an active Codex runtime whose owner plugin is absent; keep gateway startup permissive and avoid startup-time auto-install side effects.

## Reproduction Assessment

Yes, from source and reporter logs, but not from a live command in this read-only review. A Codex-selected OpenAI route without a registered Codex plugin reaches `selectAgentHarness` and throws, while `plugins doctor` only reports diagnostics for plugins it can see.

## Solution Assessment

Yes, the best current fix direction is a doctor diagnostic/repair path rather than gateway startup failure. That matches the maintainer comment and avoids network/install side effects during gateway startup.
