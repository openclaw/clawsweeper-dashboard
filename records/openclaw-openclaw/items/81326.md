## Summary

Keep open. Current main still has a narrow diagnostic gap: Codex can be the selected OpenAI agent runtime while the owner plugin is absent, and `plugins doctor` only reports load/registry problems for plugins it already sees; maintainer discussion also points toward surfacing this in doctor rather than failing gateway startup.

## What This Changes

The issue asks OpenClaw to diagnose or repair configs that select the Codex agent runtime when the `@openclaw/codex` plugin is not installed or enabled.

## Best Possible Solution

Add an actionable doctor diagnostic, and repair in `doctor --fix` where appropriate, for active Codex runtime policy whose owner plugin is absent or disabled; do not add gateway startup fail-fast or startup network installs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: select an OpenAI/Codex runtime with no registered `codex` harness and the harness selector throws before reply. The missing diagnostic portion is also visible because `plugins doctor` only checks current plugin diagnostics, not config-only runtime owner requirements.

## Solution Assessment

Yes for the doctor-surfacing part, with a narrower boundary than the full issue alternatives. Gateway startup fail-fast and startup auto-install are not the best first fix because the maintainer comment notes fallback/default-agent cases and prefers doctor visibility.
