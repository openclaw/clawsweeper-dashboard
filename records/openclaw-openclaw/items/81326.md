## Summary

Keep open: current main has pieces of the repair path, especially `openclaw doctor --fix`, but the reported pre-traffic diagnostic gap is not fully closed for `plugins doctor` or gateway startup, and an open related PR only covers the update-time repair slice.

## What This Changes

The issue asks OpenClaw to diagnose, block, or repair configurations that select the Codex runtime when the Codex plugin is not installed or enabled.

## Best Possible Solution

Decide the owner boundary first, then add a focused pre-traffic diagnostic path: either make `plugins doctor`/`openclaw doctor` report the missing `@openclaw/codex` owner plugin clearly, fail fast at gateway startup without package-manager side effects, or land an explicit update/doctor repair policy that maintainers accept.

## Reproduction Assessment

Yes from source inspection: a selected `codex` runtime reaches a fail-closed harness lookup when no Codex plugin has registered the harness, and `plugins doctor` does not run the missing configured plugin repair path. I did not execute a live install-state repro because this review is read-only.

## Solution Assessment

Unclear. `openclaw doctor --fix` already has a repair path, but the best fix still needs maintainer choice between `plugins doctor` diagnostics, gateway fail-fast validation, and package-install repair during startup/update flows.
