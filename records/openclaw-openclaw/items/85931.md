## Summary

Keep open for normal PR review/landing. Current main still only serializes QMD update work inside one process, while this PR adds a narrow per-agent cross-process lock matching the existing embed-lock pattern; I found no blocking correctness issue in the diff.

## What This Changes

The PR adds a per-agent cross-process `qmd-update.lock` around QMD update work and adds regression coverage for the lock path and options.

## Best Possible Solution

Land the narrow update-lock fix after maintainers accept or verify the bounded cross-process wait behavior with real QMD/gateway-plus-CLI proof.

Yes, from source and supplied terminal proof, though I did not rerun it locally in this read-only review. ## Solution Assessment

Yes. A per-agent cross-process file lock inside the existing QMD update queue is the narrowest maintainable fix and reuses the existing embed-lock contract without adding config or changing search results.
