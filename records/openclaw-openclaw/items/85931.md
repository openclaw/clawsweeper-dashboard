## Summary

## What This Changes

The PR wraps QMD update and embed writes with a shared per-store cross-process lock while keeping the existing global embed lock, plus adds memory-core regression coverage for that lock behavior.

## Best Possible Solution

Merge the shared per-store QMD write lock once maintainers accept the availability tradeoff and proof depth, or ask for one real-`qmd` gateway-plus-CLI concurrency smoke before landing.

## Reproduction Assessment

Yes for the modeled failure: current main lacks a shared cross-process lock between QMD update and embed, and the PR body supplies terminal proof reproducing `SQLITE_BUSY` without the lock and success with it. I did not run a full real-`qmd` gateway-plus-CLI repro in this read-only review.

## Solution Assessment

Yes, the proposed per-store file lock is the narrowest maintainable fix for writer-vs-writer contention on the same QMD `index.sqlite` while preserving config and search behavior. The only remaining decision is whether maintainers require full real-`qmd` proof before merge.
