## Summary

Keep open for maintainer review: the patch is narrow and appears correct, with good terminal proof, but it intentionally changes QMD contention from `SQLITE_BUSY` failures into bounded cross-process waits that maintainers should accept or verify with real QMD before merge.

## What This Changes

The PR adds a per-agent `qmd-update.lock` around QMD update work and adds regression coverage for the new lock path and options.

## Best Possible Solution

Land the narrow update-lock fix after maintainers accept or verify the bounded cross-process wait behavior with real QMD gateway-plus-CLI proof.

## Reproduction Assessment

Yes, with high confidence from source and supplied terminal proof, though I did not rerun it locally in this read-only review. Current main has only process-local update serialization, and the PR body shows the two-process SQLite writer collision turning from `SQLITE_BUSY` into a successful wait when wrapped with the same file-lock primitive.

## Solution Assessment

Yes. A per-agent cross-process file lock inside the existing QMD update queue is the narrowest maintainable fix and reuses the existing embed-lock contract without adding config or changing search results.
