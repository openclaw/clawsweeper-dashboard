## Summary

Keep this PR open, but do not merge it as-is: it targets a real skills DX problem, yet the patch only handles `skills list`, leaves `skills info` and `skills check` misleading for the new parse-error state, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a `parseError` field to skill entries/status, marks parse-error skills ineligible, and displays the error in `openclaw skills list` text and JSON output.

## Best Possible Solution

Keep the tolerant frontmatter parser, and if diagnostics are still desired, collect load/parse failures at the loader boundary and surface the same parse-error state consistently in list, info, check, and JSON outputs with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the PR defect: once a skill has `parseError` and `eligible: false`, `formatSkillInfo` ignores `parseError` and `formatSkillsCheck` classifies it as missing requirements; no live CLI run was performed because this review is read-only.

## Solution Assessment

No. The patch is not the best complete fix as-is because it only surfaces the new error state in `skills list`; the maintainable fix should capture diagnostics before skills disappear and route that state through every CLI view that reports skill readiness.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Surface parse errors in skill info:** `src/cli/skills-cli.format.ts:199-207`
  - body: The PR introduces `parseError` as a status state, but `openclaw skills info <name>` still chooses its text status only from disabled/allowlist/filter/eligible and never prints the parse message. A malformed skill can therefore show a parse error in `skills list` but still appear as generic setup state in the detailed view, which leaves the new diagnostic hidden in a primary CLI path.
  - confidence: 0.9
- **[P2] Separate parse errors from missing requirements:** `src/cli/skills-cli.format.ts:323-325`
  - body: After this PR marks parse-error skills `eligible: false`, the existing `missingReqs` filter in `skills check` will include them as missing requirements even when `missing` is empty. Add a parse-error bucket/section and exclude those entries from missing requirements in both text and JSON check output.
  - confidence: 0.9
