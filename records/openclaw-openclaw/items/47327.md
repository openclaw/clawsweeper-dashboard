## Summary

This PR should stay open because it targets a real skills CLI bug, but the patch is incomplete: failed local skill candidates can still be filtered out before diagnostics exist, and two documented diagnostic views would still present parse-error skills incorrectly.

## What This Changes

Adds a `parseError` field to skill/status records and renders SKILL.md parse failures in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve read/parse diagnostics at the local discovery boundary and render them consistently across `list`, `info`, and `check` text/JSON output, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a skill with malformed SKILL.md frontmatter and the local loader catches the parse error, returns null, and filters the candidate before status generation; on the PR branch, entries that do carry `parseError` would still be mishandled by `info` and `check`.

## Solution Assessment

No. The branch adds useful plumbing, but it starts after some failed candidates have already been dropped and only updates the list view; the narrower maintainable fix is loader diagnostics plus all CLI diagnostic surfaces.
