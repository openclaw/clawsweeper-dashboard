## Summary

Keep open: current main still has the malformed-skill silent-drop behavior, but this PR is not mergeable as-is because it only attaches parse errors after a skill has already loaded, leaves `skills info` and `skills check` inconsistent, and lacks after-fix real CLI proof.

## What This Changes

Adds `parseError` plumbing for skill entries/status, captures frontmatter read/parse failures while building workspace skill entries, and shows parse errors in `openclaw skills list` text/JSON output.

## Best Possible Solution

Retarget the fix to the current local-loader/status contract, preserve diagnostics for candidates that fail before `SkillEntry` creation, and surface one parse-error state consistently in list, info, check, JSON output, tests, and redacted real CLI proof.

## Reproduction Assessment

Yes. Source inspection plus the upstream loader contract show a malformed `SKILL.md` parse failure returns no skill entry on current main; I did not run the CLI because this review is read-only.

## Solution Assessment

No. The PR moves toward the right user-facing diagnostic, but it is not the best mergeable solution because it misses failures dropped before entry creation and does not update all CLI views.

- **[P2] Preserve loader diagnostics for dropped skills:** `src/agents/skills/workspace.ts:515-518`
  - body: The new catch runs only after `loadSkillsFromDir` has returned a `Skill` and `merged` has been built.   - confidence: 0.95

A malformed skill can therefore show an error in list output while the detailed view reports a generic missing-requirements state.
  - confidence: 0.9

  - body: After the PR marks parse-error skills ineligible, `formatSkillsCheck` includes them in `missingReqs` because the filter ignores `parseError`. Add a parse-error bucket and exclude those entries from missing requirements in both text and JSON check output.
  - confidence: 0.9
