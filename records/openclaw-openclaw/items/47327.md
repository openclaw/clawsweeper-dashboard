## Summary

Keep open: the branch is still incomplete because malformed local `SKILL.md` candidates can be filtered out before any status entry exists, `info` and `check` remain misleading for parse-error skills, and the external contributor has not supplied real behavior proof.

## What This Changes

The PR adds `parseError` plumbing through skill entries/status and renders parse errors in `openclaw skills list` text/JSON while marking those skills ineligible.

## Best Possible Solution

Preserve failed skill discovery diagnostics before filtering, pass them through status, render parse/read errors consistently in `list`, `info`, and `check` text/JSON output, and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a local skill with malformed YAML frontmatter such as an unquoted `: ` in `description`, then inspect `openclaw skills list`, `info`, and `check`; I did not run it because this review is read-only.

## Solution Assessment

No. The branch is pointed in the right direction, but the narrow maintainable fix needs to preserve failed candidates from the loader and update every documented CLI diagnostic surface, not only `list`.
