## Summary

Keep open: the PR is useful but not mergeable as-is. Current main has since made the reported colon-in-description case more tolerant, but it still does not implement the PR's intended parse-error diagnostic surface, and the branch only handles `skills list` after a skill has already survived discovery.

## What This Changes

The PR adds a `parseError` field to skill/status records and renders it in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve discovery diagnostics before filtering and render parse/read errors consistently across list/info/check text and JSON, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: failed local skill reads/parses can be filtered out before status generation, while entries that do carry `parseError` would still be ignored by info/check in this branch.

## Solution Assessment

No. The proposed list-only plumbing records diagnostics too late for dropped candidates and leaves two documented CLI diagnostic views misleading.
