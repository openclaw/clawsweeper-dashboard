## Review Findings

Full review comments:

- **[P2] Preserve diagnostics before entries are built:** `src/agents/skills/workspace.ts:511-518`
  - body: The patch records `parseError` only after a `Skill` already exists in `merged`. The silent-drop case is a malformed `SKILL.md` that fails before a `SkillEntry` can be created, so there is no entry to annotate and the skill can still be absent from CLI output on this branch.
  - confidence: 0.93
- **[P2] Surface parse errors in skill info:** `src/cli/skills-cli.format.ts:195-201`
  - body: `openclaw skills info` still derives status only from eligibility/disabled/allowlist state and never prints `skill.parseError`. A parse-error entry would therefore show a generic readiness or missing-requirements state instead of the actual parse failure.
  - confidence: 0.9
- **[P2] Separate parse errors from missing requirements:** `src/cli/skills-cli.format.ts:281-283`
  - body: After the patch marks parse-error skills ineligible, `formatSkillsCheck` includes them in `missingReqs` because this filter ignores `parseError`. That makes check output and JSON summaries categorize parser failures as missing setup requirements.
  - confidence: 0.88
