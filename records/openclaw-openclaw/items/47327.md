## Summary

Keep open: the PR points at a real skills bug, but the current head still records diagnostics too late for skills dropped by the loader and only updates `list`, leaving `info` and `check` misleading. It also lacks external after-fix real behavior proof.

## What This Changes

The PR adds `parseError` fields to skill/status entries, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve failed skill discovery diagnostics before filtering, pass them through status, and render parse/read errors consistently across `list`, `info`, and `check` text/JSON output with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows malformed or unreadable local `SKILL.md` files return `null` before status generation, and the PR's `parseError` path would still leave `info` and `check` misleading.

## Solution Assessment

No. The branch is the right general direction, but it records diagnostics too late for dropped candidates and only surfaces them in one documented CLI view.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve failed skill diagnostics before filtering:** `src/agents/skills/workspace.ts:513-518`
  - body: The new `parseError` is assigned only while mapping skills that were already returned by the loader. A `SKILL.md` read or parse failure still returns `null` in the local loader and is filtered out before a status row exists, so the absent-from-list case can remain silent.
  - confidence: 0.88
- **[P2] Show parse errors in skill info:** `src/cli/skills-cli.format.ts:199-207`
  - body: This branch threads `parseError` into status records, but `formatSkillInfo` still chooses Ready/Needs setup from eligibility and never prints the diagnostic. Users checking a specific malformed skill would still get a misleading info view.
  - confidence: 0.9
- **[P2] Separate parse errors in skills check:** `src/cli/skills-cli.format.ts:323-325`
  - body: `formatSkillsCheck` still buckets every active non-eligible skill as missing requirements and emits no parse-error summary or JSON section. A parse-error status row would show up as an empty missing-requirements entry instead of the actual diagnostic.
  - confidence: 0.9
