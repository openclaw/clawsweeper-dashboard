## Summary

Keep open. This is a real bugfix attempt for a concrete malformed `SKILL.md` discovery bug, but the current PR is incomplete and conflict-marked: it surfaces parse errors in list output while `skills info` and `skills check` still report misleading states, and no after-fix real behavior proof is present.

## What This Changes

The branch adds `parseError` plumbing for skill entries/status, captures `SKILL.md` read or parse failures in the workspace loader, excludes those skills from eligibility, and shows the error in `skills list` text/JSON output.

## Best Possible Solution

Implement one parse-error/diagnostic state in the current local-loader and status contracts, then surface it consistently through `skills list`, `skills info`, `skills check`, JSON output, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main returns `null` on local `parseFrontmatter` failures, and the linked issue gives a concrete malformed-frontmatter `SKILL.md` flow; I did not run the CLI because this review is read-only.

## Solution Assessment

No. The PR moves in the right direction, but it is not the best mergeable form because it only updates list output, leaves info/check misleading, and needs to be rebased onto the current local-loader path.
