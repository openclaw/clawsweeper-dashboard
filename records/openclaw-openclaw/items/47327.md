## Summary

Keep open: the PR is a plausible skills diagnostics fix, but it is incomplete because two CLI views still ignore the new parse-error state, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

Adds a parseError field to skill entries/status, captures SKILL.md read/parse errors in the skill loader, marks those skills ineligible, and displays the error in skills list text/JSON output.

## Best Possible Solution

Rebase onto the current local skill loader and surface one parse-error state consistently through list, info, check, and JSON outputs with focused regression coverage and real CLI proof.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough: parseError makes a skill ineligible, while formatSkillInfo ignores parseError and formatSkillsCheck classifies all ineligible parse-error skills as missing requirements.

## Solution Assessment

No. The patch is not complete as-is because it only updates skills list; the safer fix is to collect diagnostics at the current loader boundary and route the same state through every skills readiness view.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

A malformed skill can therefore show the error in skills list while the detailed view reports generic readiness or setup state.
  - confidence: 0.9
