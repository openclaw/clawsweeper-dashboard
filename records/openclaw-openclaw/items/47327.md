## Summary

Keep open. The requested diagnostic behavior is still useful, but this branch is incomplete: malformed local skills can still disappear before a status record exists, and the proposed formatter changes do not cover the `skills info` and `skills check` surfaces called out in review.

## What This Changes

The PR adds a `parseError` field to skill/status records, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Keep the tolerant frontmatter parser. If maintainers still want explicit diagnostics for unrecoverable local skill read/parse failures, add a discovery-level diagnostic record before failed skills are filtered out, plumb it through the status contract, and render it consistently in `skills list`, `skills info`, and `skills check` text and JSON with focused regression tests.

## Reproduction Assessment

Yes, for the PR defects. Code inspection gives a high-confidence path: malformed or unreadable `SKILL.md` files can return `null` before status records exist, and even a synthetic parse-error status would still be ignored or misbucketed by `skills info` and `skills check`.

## Solution Assessment

No. The submitted approach records diagnostics too late in the loader pipeline and updates only the list formatter; the maintainable fix needs a discovery diagnostic contract plus complete list/info/check handling.
