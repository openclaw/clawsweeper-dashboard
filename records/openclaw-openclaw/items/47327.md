## Summary

Keep open. Current main fixes the original colon-in-description frontmatter case, but it still has no discovery/status diagnostic for genuinely unreadable or unparseable SKILL.md files, and this PR does not fully solve that remaining behavior because it records diagnostics after failed skills can already be filtered out and only updates the list formatter.

## What This Changes

The PR adds parseError plumbing to skill entries and status records, marks parse-error skills ineligible, and renders that diagnostic in openclaw skills list text and JSON output.

## Best Possible Solution

Keep the tolerant frontmatter parser, then add a discovery-level diagnostic record for unreadable or unrecoverably invalid local SKILL.md files before failed candidates are filtered out. Plumb that diagnostic through the status contract and render it consistently in skills list, skills info, and skills check text/JSON with focused regression coverage.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: a skill whose SKILL.md cannot be read or whose frontmatter parsing fails can return null in loadSingleSkillDirectory before any status row exists; a synthetic parseError status would also be ignored by skills info and misbucketed by skills check in the current formatter.

## Solution Assessment

No. The proposed patch records parseError too late in the loader pipeline and updates only the list formatter, so it is not the narrowest complete fix for the stated user-visible diagnostics.
