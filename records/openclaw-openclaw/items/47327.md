## Summary

Keep open: the PR addresses a real skills diagnostics gap, but the patch is incomplete because failed local skill candidates can still disappear before a status row exists and parse errors are not handled consistently across documented CLI views.

## What This Changes

The branch adds `parseError` plumbing to skill entries/status records and renders parse-error diagnostics in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve discovery diagnostics before local-loader filtering, carry them through the status contract, and render them consistently in skills list/info/check text and JSON with regression tests and a changelog entry.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: failed local skill reads or unrecoverable parsing return null before status generation, while a status entry with `parseError` would still be ignored by info/check in the current formatter.

## Solution Assessment

No. The proposed patch records diagnostics too late for dropped local skills and updates only the list formatter, so it is not the narrowest complete fix for the stated user-visible behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve failed skill diagnostics before filtering:** `src/agents/skills/workspace.ts:513-518`
  - body: The patch sets `parseError` only while mapping already-loaded skill records, but local `SKILL.md` read/parse failures can still return null before any `SkillEntry` exists. That leaves the dropped-skill case with no row for list/info/check to display.
  - confidence: 0.88
- **[P2] Show parse errors in skill info:** `src/cli/skills-cli.format.ts:199-207`
  - body: `openclaw skills info` still derives status from disabled/allowlist/filter/eligible state and never prints `skill.parseError`. A parse-error skill that reaches status can therefore appear ready or needing setup without the actual diagnostic.
  - confidence: 0.9
- **[P2] Separate parse errors in skills check:** `src/cli/skills-cli.format.ts:323-325`
  - body: `skills check` still puts every active non-eligible skill under missing requirements. Parse-error skills need their own summary and text/JSON section, otherwise users get an empty or misleading missing-requirements entry.
  - confidence: 0.9

  - body: This is a user-facing skills CLI fix, but the branch does not add an Unreleased Fixes entry. Repo policy requires a changelog entry for this kind of change.
  - confidence: 0.86
