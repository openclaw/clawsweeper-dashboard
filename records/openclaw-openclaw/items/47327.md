## Summary

Keep open. The PR is aimed at a real skills CLI bug, but the branch is incomplete: it records diagnostics only after failed candidates can already be dropped, updates `list` without `info`/`check`, and lacks after-fix real behavior proof from the contributor.

## What This Changes

The PR adds a `parseError` field to skill/status records and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve failed skill discovery diagnostics before filtering and render parse/read errors consistently across `list`, `info`, and `check` text/JSON output, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: malformed or unreadable local `SKILL.md` files can return `null` before status generation, and parse-error status entries would still be misleading in `info` and `check`.

## Solution Assessment

No. The proposed branch is the right general direction, but it records diagnostics too late for dropped candidates and only surfaces them in one of the documented CLI diagnostic views.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve failed skill diagnostics before filtering:** `src/agents/skills/workspace.ts:513-518`
  - body: The new `parseError` is assigned only while mapping skills that were already returned by `loadSkillsFromDirSafe`. A `SKILL.md` read/parse failure still returns `null` in the local loader and is filtered out before a status row exists, so the reported absent-from-list case remains silent.
  - confidence: 0.88
- **[P2] Show parse errors in skill info:** `src/cli/skills-cli.format.ts:199-207`
  - body: This branch threads `parseError` into status records, but `formatSkillInfo` still chooses Ready/Needs setup from eligibility and never prints the diagnostic. Users checking a specific malformed skill would still get a misleading info view.
  - confidence: 0.9
- **[P2] Separate parse errors in skills check:** `src/cli/skills-cli.format.ts:323-325`
  - body: `formatSkillsCheck` still buckets every active non-eligible skill as missing requirements and emits no parse-error summary or JSON section. A parse-error status row would show up as an empty missing-requirements entry instead of the actual diagnostic.
  - confidence: 0.9
