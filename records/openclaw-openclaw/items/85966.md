Make Android drive its connect request from the received `connect.challenge` with a single guarded connect-start path, preserve intentional loopback fallback behavior, and add focused Android regression coverage for the no-connect-before-close case.

## Repair Work Prompt

Fix the Android GatewaySession WebSocket connect race reported in https://github.com/openclaw/openclaw/issues/85966. Current Android code waits for connect.challenge via connectNonceDeferred from onOpen and calls closeQuietly() with code 1000 reason bye on failure; a prior unmerged related PR at https://github.com/openclaw/openclaw/pull/5441 diagnosed the same bye-before-connect pattern. Implement a narrow Android-side handshake state repair so connect.challenge directly drives exactly one sendConnect path, while preserving any intended loopback/local fallback without duplicate connect requests. Do not broaden bootstrap/operator scopes or gateway auth policy unless source inspection proves that is still required; current main already requests/persists operator.pairing for Android operator handoff. Add focused Android unit coverage around challenge-driven connect and no close-before-connect behavior. Changelog is expected only if the resulting fix is user-visible per maintainer policy.
