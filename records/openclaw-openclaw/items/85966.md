## Summary

Keep open: current main and v2026.5.22 still show an Android/server bootstrap-scope mismatch, with Android requesting `operator.pairing` while QR/setup-code handoff deliberately grants only `operator.approvals`, `operator.read`, and `operator.write`. The reporter's latest logs match that `scope_mismatch` path on `acf265d`; the separate `code=1000 bye` pending-handshake variant still needs a current-main live rerun after the scope path is aligned.

## What This Changes

The issue asks for Android operator WebSocket pairing to reach Ready after node pairing over Tailscale Serve, with clearer diagnostics for scope, handshake, persisted-token, and backup-reset behavior.

## Best Possible Solution

Keep QR/setup-code operator handoff bounded, align Android's default operator reconnect with the granted bootstrap scopes, and add focused diagnostics/tests for both scope mismatch and pre-connect close behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? ## Solution Assessment

Is this the best way to solve the issue? Yes for the scope slice: preserve the documented bounded handoff and fix Android to request only scopes the QR operator token actually grants, with a separate approved path for pairing/admin needs. The pending-bye slice needs added Android logging or a focused reconnect test before choosing a broader handshake repair.
