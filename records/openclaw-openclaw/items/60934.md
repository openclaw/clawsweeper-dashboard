## Summary

Keep this PR open, but it needs changes before merge. Current main still does not apply provider-level `models.providers.*.auth` token/oauth selection to auth-profile candidates, while this patch can silently fall back to mismatched profiles and leaves availability checks out of sync.

## What This Changes

This PR changes `src/agents/model-auth.ts` so `resolveApiKeyForProvider` filters auth-profile candidates by provider-level `auth: "token"` or `auth: "oauth"` before resolving credentials.

## Best Possible Solution

Centralize provider-level token/oauth profile eligibility, reuse it from runtime resolution and availability checks, treat no matching profile type as no matching profile instead of falling back, and add focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A focused unit case can configure `models.providers.anthropic.auth: "token"` with OAuth and token profiles, then verify current main walks OAuth first and the PR still falls back to OAuth when no matching token profile exists.

## Solution Assessment

No. Filtering only `resolveApiKeyForProvider` and falling back to the original order is not the best fix; the safer solution is a shared strict eligibility helper used by both resolver and availability paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: With `models.providers.<id>.auth: "token"` and only OAuth profiles available, `filteredOrder` is empty so `finalOrder` reuses the original `order` and can select OAuth anyway. That preserves the auth-routing bug this patch is trying to fix; the override should report no matching credential instead of silently trying the mismatched type.
  - confidence: 0.91
- **[P2] Filter availability checks with the same auth mode:** `src/agents/model-auth.ts:350-357`
  - body: The resolver now skips profiles outside the provider auth override, but `hasAvailableAuthForProvider` still scans the unfiltered order. With token auth, an expired token profile plus a valid OAuth profile can make readiness return true while runtime resolution fails after skipping OAuth, so both paths should share the same eligibility helper.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/agents/model-auth.ts:350`
  - body: This is a user-facing auth-routing fix and the PR does not touch `CHANGELOG.md`. OpenClaw policy requires a single-line changelog entry for user-visible fixes before merge.
  - confidence: 0.86
