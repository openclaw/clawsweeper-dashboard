## Summary

Keep open: the auth-routing problem is source-reproducible, but this PR is not correct as-is because the new resolver-only filter can still fall back to mismatched profiles, availability checks remain on the old unfiltered path, and no real behavior proof is present.

## What This Changes

The PR filters `resolveApiKeyForProvider` auth-profile selection by provider-level token/oauth auth override before resolving a profile.

## Best Possible Solution

Centralize provider-level token/oauth profile eligibility, reuse it from resolver and availability paths, add focused regression tests, then verify with redacted real auth or Telegram-equivalent runtime proof.

## Reproduction Assessment

Yes, at source level: `models.providers.*.auth` accepts `token`/`oauth`, while current resolver and availability paths still scan the auth-profile order without a provider-level token/oauth filter. I did not run a live Telegram or billing-flow reproduction in this read-only review.

## Solution Assessment

No. The PR is in the right area, but the fallback preserves mismatched-profile selection and the availability path needs the same shared eligibility helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: When `authOverride` is `"token"` or `"oauth"` and no profile of that type exists, this fallback reuses the full profile order, so the configured auth mode can still be ignored. Treat the empty filtered set as no eligible profile instead of silently trying the other mode.
  - confidence: 0.92
- **[P2] Apply auth-mode filtering to availability checks:** `src/agents/model-auth.ts:350-357`
  - body: The new eligibility rule only changes `resolveApiKeyForProvider`; `hasAvailableAuthForProvider` still reports readiness from the unfiltered profile order. Share the token/oauth eligibility helper so availability and execution cannot disagree.
