## Summary

Keep open: the PR targets an existing provider-auth contract that current main still does not enforce for token/oauth profile scanning, but the proposed patch is incomplete and lacks required real behavior proof.

## What This Changes

The PR filters `resolveApiKeyForProvider` auth-profile candidates by provider-level `auth: "token"` or `auth: "oauth"`.

## Best Possible Solution

Centralize provider-level token/oauth profile eligibility and reuse it from resolver and availability paths, with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main parses `models.providers.<id>.auth` but both resolver and availability checks still scan the unfiltered auth-profile order, so a focused unit case can cover OAuth-first or no-token-profile scenarios without a live Telegram run.

## Solution Assessment

No. The proposed inline filter with fallback is not the best fix because it can still use mismatched profiles and leaves availability checks on different semantics; a shared strict eligibility helper is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: With `models.providers.<id>.auth: "token"` and no matching token profile, `filteredOrder` is empty and `finalOrder` reuses the full order, so an OAuth profile can still be selected despite the override. Treat no matching profile type as no matching profile instead of silently trying a different auth mode.
  - confidence: 0.91
- **[P2] Apply the same auth filter to availability checks:** `src/agents/model-auth.ts:350-357`
  - body: The new filtering only changes `resolveApiKeyForProvider`; `hasAvailableAuthForProvider` still scans the unfiltered profile order. With a token override, an unusable token profile plus a usable OAuth profile can report available and then fail at runtime once strict resolution is applied, so both paths need the same eligibility helper.
  - confidence: 0.86
