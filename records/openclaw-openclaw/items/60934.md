## Summary

Keep this PR open: it targets a real, source-reproducible provider-auth selection bug, but the branch is not merge-ready because the new filter can still fall back to mismatched profiles, leaves availability checks on different semantics, and has no after-fix real behavior proof.

## What This Changes

The branch adds inline token/oauth filtering to `resolveApiKeyForProvider` based on `models.providers.<id>.auth`, then falls back to the original unfiltered auth-profile order when no matching profile type exists.

## Best Possible Solution

Centralize provider-level token/oauth auth-profile eligibility and reuse it from both resolver and availability paths, with focused regression coverage and redacted real behavior proof from an affected setup.

## Reproduction Assessment

Yes. Source inspection shows current main parses provider-level `auth` values, while resolver and availability paths still scan the resolved profile order without provider-level token/oauth filtering; a focused unit case can cover OAuth-first and no-token-profile configurations.

## Solution Assessment

No. The inline resolver-only filter is not the best fix because it can silently fall back to mismatched auth profiles and leaves readiness checks using different eligibility semantics; a shared strict helper is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: When `models.providers.<id>.auth` is set to `"token"` and no token profiles exist, `filteredOrder` is empty and `finalOrder` reuses the unfiltered order, so OAuth can still be selected despite the override. Treat no matching profile type as no matching profile instead of silently trying the other auth mode.
  - confidence: 0.92

  - body: This filter only changes `resolveApiKeyForProvider`; `hasAvailableAuthForProvider` still scans the original profile order. Callers can therefore report provider auth as available for a profile the corrected resolver should reject, so both paths need the same eligibility logic.
  - confidence: 0.87
