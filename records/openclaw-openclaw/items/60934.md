## Summary

Keep open: this is a plausible bug-fix PR against an existing auth strategy surface, but the patch is not correct enough to merge and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds token/oauth filtering with a fallback in `src/agents/model-auth.ts` when resolving auth profiles for a provider-level auth override.

## Best Possible Solution

Centralize provider-level token/oauth profile eligibility, reuse it from resolver and availability checks, add focused regression tests, then verify with redacted real auth or Telegram-equivalent runtime proof.

## Reproduction Assessment

Yes, at source level: `models.providers.*.auth` accepts `token`/`oauth`, while current resolver and availability paths still scan the auth-profile order without provider-level token/oauth filtering. I did not run a live Telegram or billing-flow reproduction in this read-only review.

## Solution Assessment

No. The PR targets the right resolver area, but the fallback preserves mismatched-profile selection and the availability path needs the same shared eligibility rule.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: When `authOverride` is `"token"` or `"oauth"` and no profile of that type exists, this fallback reuses the full profile order, so the configured auth mode can still be ignored. Treat the empty filtered set as no eligible profile instead of silently trying the other mode.
  - confidence: 0.92
- **[P2] Use the same eligibility in availability checks:** `src/agents/model-auth.ts:350-357`
  - body: The new eligibility rule only changes `resolveApiKeyForProvider`; `hasAvailableAuthForProvider` still reports readiness from the unfiltered profile order. Share the token/oauth eligibility helper so availability and execution cannot disagree.
  - confidence: 0.87
