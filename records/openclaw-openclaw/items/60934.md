## Summary

Keep open: the PR targets a real auth-provider contract, but it is not merge-ready because the branch still allows mismatched profile fallback, leaves availability checks on a separate path, and lacks contributor real-behavior proof.

## What This Changes

The PR changes `src/agents/model-auth.ts` so provider-level `token` or `oauth` auth overrides filter candidate auth profiles before credential resolution.

## Best Possible Solution

Centralize provider auth-mode profile eligibility, reuse it in resolver and availability checks, and make the strict-versus-compatibility fallback behavior explicit with tests and upgrade proof.

## Reproduction Assessment

Yes, at source level: current main does not apply provider-level `token`/`oauth` auth overrides to profile iteration, and the PR's fallback reuses unfiltered profiles when no matching profile exists. I did not run a live Telegram or billing-flow reproduction in this read-only review.

## Solution Assessment

No. The PR touches the right auth resolver surface, but the fallback preserves a mismatched-profile path and the availability check needs the same shared eligibility rule before this is the narrow maintainable fix.
