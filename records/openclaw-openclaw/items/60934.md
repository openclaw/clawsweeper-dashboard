## Summary

Keep open: the PR addresses a real provider-auth routing gap that is still visible on current main, but the proposed patch is incomplete because it falls back to mismatched profiles and leaves availability checks unfiltered.

## What This Changes

The PR changes `src/agents/model-auth.ts` so provider-level token/oauth auth overrides filter auth-profile candidates before credential resolution.

## Best Possible Solution

Centralize provider-level token/oauth profile eligibility and reuse it from runtime resolution and availability checks, treating no matching profile type as no matching profile and adding focused regression coverage.

## Reproduction Assessment

Yes. A focused unit case can configure `models.providers.anthropic.auth: "token"` with OAuth and token profiles and show that current main still walks the unfiltered profile order; I did not run tests because this review is read-only.

## Solution Assessment

No. Filtering only `resolveApiKeyForProvider` and falling back to the original order is not the best fix; a shared strict eligibility helper should drive both resolver and availability paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: With `models.providers.<id>.auth: "token"` and no token profiles, `filteredOrder` is empty and `finalOrder` reuses the full `order`, so an OAuth profile can still be selected. The override should make the profile scan find no matching token credential instead of silently trying a mismatched profile type.
  - confidence: 0.91
- **[P2] Share auth-mode filtering with availability checks:** `src/agents/model-auth.ts:354`
  - body: The new filtering is inline in `resolveApiKeyForProvider`, while `hasAvailableAuthForProvider` still scans `resolveAuthProfileOrder` without the provider auth mode. That can make readiness report true for credentials the runtime resolver should reject under token/oauth override, so both paths need the same eligibility helper.
  - confidence: 0.86
