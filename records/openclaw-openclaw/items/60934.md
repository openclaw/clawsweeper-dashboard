## Summary

This PR should stay open because the target behavior is still relevant on current main, but the proposed patch is incomplete and lacks required after-fix real behavior proof. The diff only filters the resolver path, falls back to mismatched profiles when no requested auth type exists, and leaves availability checks using unfiltered auth-profile order.

## What This Changes

The PR changes `src/agents/model-auth.ts` so `resolveApiKeyForProvider` filters candidate auth profiles by provider-level `auth: "token"` or `auth: "oauth"` before resolving a profile.

## Best Possible Solution

Centralize provider-level auth-profile eligibility for token/oauth modes and reuse it from both resolver and availability paths, with focused regression coverage and redacted real behavior proof from an affected setup.

## Reproduction Assessment

Yes, source-reproducible. Current main parses provider-level `auth` values but the resolver and availability paths still scan unfiltered auth-profile order, so a focused unit case can cover OAuth-first and no-token-profile scenarios without a live Telegram run.

## Solution Assessment

No. The proposed inline filter is not the best fix because it can still fall back to mismatched profiles and leaves availability checks on different semantics; a shared strict eligibility helper is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: When `models.providers.<id>.auth` is set to `"token"` and there are no token profiles, `filteredOrder` is empty and `finalOrder` reuses the full unfiltered order, so OAuth can still be selected despite the override. Treat no matching profile type as no matching auth instead of silently trying the other auth mode.
  - confidence: 0.91
- **[P2] Apply the auth filter to availability checks:** `src/agents/model-auth.ts:350-357`
  - body: Only `resolveApiKeyForProvider` gets the new token/oauth filtering; `hasAvailableAuthForProvider` still scans the unfiltered profile order. That can make provider readiness report success for profiles the strict resolver should reject, so both paths need the same eligibility logic.
  - confidence: 0.86
