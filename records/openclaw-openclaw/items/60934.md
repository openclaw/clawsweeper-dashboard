## Summary

Keep this PR open, but do not merge it as-is. The targeted provider-auth routing gap still exists on current main, while the proposed patch keeps a mismatched-profile fallback, leaves availability checks on the old unfiltered path, and lacks the required user-facing changelog entry.

## What This Changes

The PR changes `src/agents/model-auth.ts` so `resolveApiKeyForProvider` filters auth-profile candidates by provider-level `auth: "token"` or `auth: "oauth"` before resolving credentials.

## Best Possible Solution

Centralize provider-level token/oauth profile eligibility, reuse it from both runtime resolution and availability checks, never silently fall back to a mismatched profile type, and add focused regressions for mixed profiles, no matching profile type, and unresolved matching profiles.

## Reproduction Assessment

Yes. A focused unit case can configure `models.providers.anthropic.auth: "token"` with OAuth and token profiles; current main walks the unfiltered order and the default sorter prefers OAuth before token.

## Solution Assessment

No. Filtering only the resolver and then falling back to the original order is not the best fix; the safer path is a shared eligibility helper used by resolver and availability checks with explicit no-match behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: With `models.providers.<id>.auth: "token"` and only OAuth profiles available, `filteredOrder` is empty so `finalOrder` reuses the original `order` and can select OAuth anyway. That preserves the auth-routing bug this PR is trying to fix; the override should fail or report no matching credential instead of silently trying the mismatched type.
  - confidence: 0.91
- **[P2] Filter availability checks with the same auth mode:** `src/agents/model-auth.ts:350-357`
  - body: The new filtering only applies in `resolveApiKeyForProvider`. If a matching token profile is present but cannot resolve, `hasAvailableAuthForProvider` can still return true by seeing an OAuth profile that runtime resolution will skip, producing false provider readiness. Share the same auth-mode eligibility helper with the availability path.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/agents/model-auth.ts:350`
  - body: This is a user-facing `fix` change and the PR does not touch `CHANGELOG.md`. OpenClaw policy requires a single-line changelog entry with non-bot attribution for user-visible fixes before merge.
  - confidence: 0.86
