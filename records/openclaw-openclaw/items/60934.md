## Summary

Keep open: the auth-routing problem is plausible against the existing provider auth contract, but this branch should not merge as-is because it can still select mismatched profiles, leaves readiness checks on different eligibility, lacks real behavior proof, and is currently conflicting.

## What This Changes

The branch filters `resolveApiKeyForProvider` auth-profile iteration by provider-level `token` or `oauth` auth overrides in `src/agents/model-auth.ts`.

## Best Possible Solution

Centralize provider auth-mode profile eligibility, reuse it in resolver and availability checks, add focused regression coverage for matching and missing profile types, and require redacted real runtime proof before merge.

## Reproduction Assessment

Yes, at source level: the provider auth contract includes `token` and `oauth`, while this PR's resolver-only filter and current availability path show how mismatched profile selection can still occur. I did not run a live Telegram or billing-flow reproduction in this read-only review.

## Solution Assessment

No. The PR targets the right auth resolver area, but the fallback preserves the mismatched-profile bug and the availability path needs the same shared eligibility rule before this is a maintainable fix.
