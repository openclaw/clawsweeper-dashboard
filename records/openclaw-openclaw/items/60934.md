## Summary

Keep open: the PR targets a real provider-auth routing contract, but the branch is not merge-ready because the new resolver filter can silently fall back to the wrong auth mode, availability checks do not share the same eligibility rule, the head is merge-conflicting, and real behavior proof is still missing.

## What This Changes

The PR changes `src/agents/model-auth.ts` so `resolveApiKeyForProvider` filters auth-profile iteration by a provider-level `token` or `oauth` auth override before selecting a profile.

## Best Possible Solution

Centralize provider auth-mode profile eligibility, reuse it in resolver and availability checks, decide the fail-closed versus documented compatibility fallback behavior explicitly, and cover both fresh and upgrade cases with focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes, at source level: with `models.providers.<id>.auth: "token"` and only OAuth profiles, the PR's `filteredOrder` becomes empty and falls back to the unfiltered order. I did not run a live Telegram or billing-flow reproduction in this read-only review.

## Solution Assessment

No. The PR touches the right auth resolver surface, but the fallback preserves a mismatched-profile path and availability checks need the same shared eligibility rule before this is the narrow maintainable fix.
