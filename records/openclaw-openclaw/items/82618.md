## Summary

Keep open: the PR targets a real adjacent gap in the edit recovery path, but it lacks real behavior proof and the patch adds an unbounded diff implementation while missing the pinned upstream edit success shape.

## What This Changes

The PR adds a custom unified-diff builder, threads generated edit diffs into the host edit wrapper, and adds focused diff/wrapper tests.

## Best Possible Solution

Reuse the upstream edit result diff on normal success, add a bounded recovery-only fallback if still needed, and validate the selected UX with a real exported-session artifact.

## Reproduction Assessment

Partly: source inspection shows the recovery result has an empty diff and the HTML renderer can consume `details.diff`. The broad normal-success claim is not fully reproduced because the pinned upstream edit tool already returns `details.diff`.

## Solution Assessment

No: duplicating diff generation with an unbounded LCS is riskier than reusing the upstream edit diff contract and adding a size-bounded recovery fallback after maintainer UX direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Bound diff generation before allocating LCS:** `src/agents/pi-tools.host-edit.ts:107`
  - body: The safety cap is applied only after `buildLineOps` has already allocated an `(oldLines + 1) * (newLines + 1)` DP matrix. A large edit or recovery readback can therefore allocate millions of entries and hang or OOM after the file was written, so the edit tool may fail instead of returning the intended best-effort success result.
  - confidence: 0.86
- **[P3] Match the real edit success shape:** `src/agents/pi-tools.host-edit.ts:338`
  - body: The pinned upstream `createEditTool` success result contains `content` and `details` but no `isError: false`, while this guard only accepts explicit `false`. That skips the new happy-path attachment branch for real wrapped edit results when a result has missing or empty diff.
  - confidence: 0.78
