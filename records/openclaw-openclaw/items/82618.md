## Summary

Keep open: the PR is aimed at a real adjacent gap, but it introduces an unbounded hand-rolled diff path and still lacks after-fix real behavior proof for the HTML export flow. Source inspection also shows normal upstream edit success already returns `details.diff`, so the safest path is to narrow this to the missing recovery/display behavior instead of duplicating diff generation broadly.

## What This Changes

The PR adds a hand-rolled edit diff builder, threads generated diffs into the host edit wrapper on success and recovery paths, and adds tests for diff generation and attachment.

## Best Possible Solution

Reuse the upstream edit result diff where it already exists, add a bounded recovery-only fallback if still needed, and land only after a real exported-session artifact proves the rendered behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection clearly reproduces the empty-diff recovery result and the HTML renderer contract, but the broad claim that every normal successful edit lacks `details.diff` is not reproduced because the pinned upstream edit tool already returns it.

## Solution Assessment

Is this the best way to solve the issue? No: duplicating diff generation with an unbounded LCS is riskier than reusing the upstream edit diff contract or adding a size-bounded recovery fallback after maintainer UX direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Bound diff generation before building the LCS matrix:** `src/agents/pi-tools.host-edit.ts:107`
  - body: The safety cap is applied only after `buildLineOps` has already allocated an `(oldLines + 1) * (newLines + 1)` DP matrix. A large edit or recovery readback can therefore allocate millions of entries and hang or OOM after the file was written, so the edit tool may fail instead of returning the intended best-effort success result. Put a byte/line guard before LCS or reuse the upstream edit diff helper/result.
  - confidence: 0.86
- **[P3] Match the real edit success shape:** `src/agents/pi-tools.host-edit.ts:338`
  - body: The actual `createEditTool` success result from `@earendil-works/pi-coding-agent@0.74.0` contains `content` and `details` but no `isError: false`, while this guard only accepts `r.isError === false`. That means the new happy-path attachment branch tested here is skipped for the real wrapped edit tool when a result has missing or empty diff; use an upstream-shaped test and treat absent `isError` as non-error if this fallback is needed.
  - confidence: 0.78
