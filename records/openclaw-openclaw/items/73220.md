## Summary

Keep open. The PR targets the right config validation mapper and addresses a real operator-diagnostics gap, but the submitted helper would add numeric-value wording to non-numeric Zod size/length issues and would misstate exclusive lower bounds already used by current positive-number config fields; it also needs the required changelog entry before merge.

## What This Changes

This PR adds mapped Zod min/max bound hints to config validation errors and regression coverage for `session.agentToAgent.maxPingPongTurns` rejecting values above 5.

## Best Possible Solution

Land a narrow mapper-layer fix that appends bound hints only for numeric Zod origins, distinguishes inclusive from exclusive bounds, preserves allowed-values behavior, adds focused regression coverage for ceiling, exclusive lower-bound, and non-numeric length cases, and records the user-facing diagnostics fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main has a source-level reproduction with `validateConfigObjectRaw({ session: { agentToAgent: { maxPingPongTurns: 10 }}})`, which hits `.max(5)` while the mapper returns the raw Zod message without a maximum-bound hint; the PR regressions are reproducible through `__testing.mapZodIssueToConfigIssue` with `z.string().min(1)` and `z.number().positive()`.

## Solution Assessment

No, not as submitted. `mapZodIssueToConfigIssue` is the right narrow surface, but the helper must check numeric `origin`, respect `inclusive === false`, and add the required changelog entry before this is the maintainable fix.
