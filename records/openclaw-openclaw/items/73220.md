## Summary

Keep open. The PR targets the right config-validation mapper, but as submitted it would add numeric-value wording to length/size errors, misstate exclusive bounds, and omit the required changelog entry for user-facing diagnostics.

## What This Changes

The PR adds a config-validation helper that appends min/max hints to Zod `too_big`/`too_small` messages and adds a `maxPingPongTurns` regression test.

## Best Possible Solution

Land a focused mapper update that enriches only number/int bound issues, respects inclusive and exclusive constraints, preserves allowed-values metadata, covers max/min/exclusive/non-numeric cases, and records the diagnostics fix in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main validates `session.agentToAgent.maxPingPongTurns` with `.min(0).max(5)` while `mapZodIssueToConfigIssue` returns the raw Zod message without a bound hint; no tests were run because this was a read-only review.

## Solution Assessment

No, not as submitted. `mapZodIssueToConfigIssue` is the right narrow surface, but the helper needs to check Zod issue origin and `inclusive` semantics before appending value-bound wording.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate bound hints to numeric origins:** `src/config/validation.ts:490-505`
  - body: Zod also emits `too_big`/`too_small` with numeric `maximum`/`minimum` for string length and array/set/file size checks. Since this helper only checks the issue code and bound type, non-numeric config errors can get wording like `maximum allowed value is 50`; gate this on numeric origins such as `number` and `int` before appending value-bound guidance.
  - confidence: 0.9
- **[P2] Respect exclusive numeric bounds:** `src/config/validation.ts:495-505`
  - body: Zod emits `inclusive: false` for strict bounds such as `.gt()`/`.lt()`, and current numeric config schemas use `.positive()`. As written, a rejected boundary value can be reported as `minimum allowed value is 0`, which makes the invalid value look allowed; branch on `record.inclusive === false` and cover that case.
  - confidence: 0.9
- **[P3] Add the changelog entry:** `src/config/validation.ts:512-513`
  - body: This changes user-facing config validation output, but the PR does not touch `CHANGELOG.md`. Add a single-line Unreleased Fixes entry with appropriate human credit so the diagnostics improvement is included in release notes.
  - confidence: 0.88
