## Summary

Keep open. The PR addresses a valid config diagnostic gap, but the submitted helper is too broad for Zod's issue contract, would misstate exclusive bounds used by current config schemas, and is missing the required user-facing changelog entry.

## What This Changes

The PR adds a config validation helper that appends min/max bound text to Zod `too_big`/`too_small` messages and adds a regression test for `session.agentToAgent.maxPingPongTurns`.

## Best Possible Solution

Land a narrow mapper change that enriches only numeric-origin bounds, handles inclusive and exclusive constraints accurately, preserves allowed-values behavior, covers max/min/exclusive/nonnumeric cases, and records the diagnostics fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes by source inspection. `session.agentToAgent.maxPingPongTurns: 10` reaches a `.max(5)` Zod schema while current main returns the raw mapped Zod message without a maximum-bound hint.

## Solution Assessment

No, not as submitted. `mapZodIssueToConfigIssue` is the right surface, but the helper needs to gate on numeric Zod origins, respect `inclusive === false`, and add focused tests plus the changelog entry.

  - body: Zod emits `too_big` and `too_small` with numeric bounds for string length, array/set size, and file size checks too.
