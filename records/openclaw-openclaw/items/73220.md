## Summary

Keep open. The PR targets the right config-validation mapper for the linked operator-diagnostics bug, but the submitted helper is not merge-ready because it would add numeric-value wording to non-numeric Zod length/size issues, misstate exclusive numeric bounds already used by `.positive()` config fields, and lacks the required changelog entry.

## What This Changes

The PR adds numeric min/max hints to mapped Zod config validation errors and adds regression coverage for the `session.agentToAgent.maxPingPongTurns` ceiling case.

## Best Possible Solution

Land a narrow config-layer mapper change that enriches only numeric-value Zod bounds, handles inclusive and exclusive constraints accurately, preserves existing allowed-values behavior, adds focused mapper coverage for numeric max/min plus non-numeric length and exclusive/positive cases, and records the user-facing diagnostics fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main can be reproduced conceptually by validating `session.agentToAgent.maxPingPongTurns: 10`, which reaches the `.max(5)` schema but maps the raw Zod message without the maximum-bound hint; the PR-introduced regressions are reproducible from Zod v4.4.1 source and current `.positive()`/string-min config schemas.

## Solution Assessment

No, not as submitted. `mapZodIssueToConfigIssue` is the right narrow surface, but the helper needs to check numeric origins and `inclusive === false`, then add the missing changelog line before this is the maintainable fix.

  - body: Zod also emits `too_big`/`too_small` with numeric `maximum`/`minimum` for string length and array/set/file size checks.
