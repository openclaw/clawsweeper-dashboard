## Summary

Keep open: the patch targets the right config mapper, but it would produce misleading bound hints for non-numeric and exclusive-bound Zod issues, and it is missing the required changelog entry.

## What This Changes

The PR adds a helper that appends numeric min/max text to mapped Zod config validation errors plus a regression test for `session.agentToAgent.maxPingPongTurns`.

## Best Possible Solution

Land a narrow config-layer mapper change that enriches numeric value bounds only for numeric origins, handles exclusive and inclusive bounds accurately, preserves allowed-values behavior, covers the mapper cases, and records the user-facing diagnostics fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. A source-level reproduction is available by validating `session.agentToAgent.maxPingPongTurns: 10` on current main: the schema has `.max(5)`, while the mapper currently returns the raw Zod message without a maximum-bound hint.

## Solution Assessment

No, not as submitted. `mapZodIssueToConfigIssue` is the right narrow surface, but the helper needs numeric-origin gating, `inclusive === false` handling, focused regression coverage, and the changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restrict numeric hints to numeric issues:** `src/config/validation.ts:490-505`
  - body: Zod also emits `too_big` and `too_small` with numeric `maximum`/`minimum` for string length and array/set/file size checks. Because this helper only checks the issue code and bound type, fields such as `z.string().max(50)` can be reported as `maximum allowed value is 50`; gate the hint on numeric origins before appending value-bound wording.
  - confidence: 0.9
- **[P2] Respect exclusive numeric bounds:** `src/config/validation.ts:496-505`
  - body: Zod implements `.positive()` as `gt(0)` and emits `inclusive: false`; current config schemas already use positive numeric fields. As written, a rejected zero value would say `minimum allowed value is 0`, which makes the boundary look valid, so branch on `record.inclusive === false` and cover it in the mapper tests.
  - confidence: 0.9
- **[P3] Add the changelog entry:** `src/config/validation.ts:512-513`
  - body: This changes user-facing config validation output, but the PR does not update `CHANGELOG.md`. Add a single-line Unreleased Fixes entry with contributor credit so the diagnostics improvement is included in release notes.
  - confidence: 0.88
