## Summary

This should close as a duplicate of the older open skill-prompt token work. Current main already snapshots the eligible skill catalog instead of doing a fresh full filesystem scan on every reply, but it still injects the prompt-facing available-skills catalog when skills are eligible; that remaining lazy/names-only/dynamic matching problem is already tracked by #64760, with related broader coverage in #73248 and an open implementation direction in #68734.

## What This Changes

The issue requests on-demand skill loading so ordinary chat turns do not repeatedly scan or inject the full available-skills list.

## Best Possible Solution

Keep #64760 as the canonical skill-specific thread and evaluate the dynamic narrowing design there, alongside the broader hierarchical discovery discussion in #73248 and the hook-based approach in #68734.

## Reproduction Assessment

Yes for the remaining prompt-overhead behavior: source inspection shows eligible skills still render into the system prompt and are scanned by instruction before replies. The literal full filesystem scan on every reply is not reproduced on current main because sessions persist and reuse a skills snapshot unless it is missing or stale.

## Solution Assessment

Yes, duplicate closure is the best cleanup path here because the older open issue already captures the same lazy-loading and matching options with more detail. A code fix should not be split into another issue thread before maintainers settle the product/API direction.
