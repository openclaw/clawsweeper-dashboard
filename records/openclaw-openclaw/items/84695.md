## Summary

Keep open: current main and the latest release still run preflight compaction before the automatic memory-flush turn, so the reported daily-memory loss remains source-reproducible. The right next step is to review and land or intentionally reject the open linked fix at https://github.com/openclaw/openclaw/pull/84792 rather than close this issue early.

## What This Changes

Restore automatic compaction memory flush so daily memory is written before compaction reduces the active transcript.

## Best Possible Solution

Land or intentionally reject the linked direct-run ordering fix, then close this issue after merge while keeping reset-flush and bounded-append follow-ups separate.

## Reproduction Assessment

Yes, from source inspection: current main enters preflight compaction before memory flush, and post-compaction token state can make the flush gate return without writing. I did not run a live long-context Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the linked PR’s direction is the narrow maintainable fix for this issue: move the existing direct-run memory flush before existing preflight compaction. Broader reset flushing, in-turn compaction catch-up, and append validation are separate issues.
