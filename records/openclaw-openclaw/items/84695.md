## Summary

Keep open: current main still has a source-level path where preflight compaction runs before memory flush, then records post-compaction tokens as fresh state before the flush gate evaluates, matching the reported silent no-write. The nearby open issues for in-turn compaction and MiniMax missing-usage accounting are related but do not cover this specific preflight ordering regression.

## What This Changes

The issue reports that automatic Telegram compaction shows the compaction notice but does not append the configured memory-flush summary to memory/YYYY-MM-DD.md.

## Best Possible Solution

Restore the automatic memory flush so it evaluates and can run against the pre-compaction transcript before preflight compaction rotates or lowers active token state, with focused regression coverage for the ordering.

## Reproduction Assessment

Yes at source level, but not by live transport run. Current main compacts first, stores post-compaction tokens as fresh state, and then can skip runMemoryFlushIfNeeded because the token gate now sees the lower compacted token count.

## Solution Assessment

Yes. The narrowest maintainable fix is to repair ordering or state handoff in the existing compaction memory-flush path, then add regression coverage proving flush evaluation happens before preflight compaction can reset token pressure.
