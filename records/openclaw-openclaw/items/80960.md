## Summary

Keep open: current main still retains a fresh `.bak-*` file after each successful session-file repair, the tests and docs still encode retained backups, and the live fix PR with closing syntax is open rather than merged.

## What This Changes

The issue asks OpenClaw to stop session-file repair from accumulating unbounded retained `.bak-*` backups across repeated successful repairs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80969 or an equivalent focused cleanup that deletes success-path transient backups while preserving/reporting retained backups on cleanup or repair failure.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: repeatedly repair a session file that keeps needing cleanup, and current main writes a unique retained `.bak-*` on each successful repair; the linked PR body also supplies real filesystem before/after proof.

## Solution Assessment

Yes. Deleting the backup only after `replaceFileAtomic` succeeds is the narrowest maintainable fix for the success-path disk leak; rotation or a config knob would be broader than this bug requires.
