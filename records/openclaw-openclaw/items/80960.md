## Summary

Keep open: current main and the latest release still retain a fresh `.bak-*` file after every successful session-file repair, and this issue is paired with an open fixing PR that uses closing syntax. The issue should close only after that PR, or an equivalent fix, lands.

## What This Changes

The issue reports that successful `repairSessionFileIfNeeded` calls retain a full `.bak-*` sibling on every repair, allowing repeated repair loops to grow session directories without bound.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80969 or an equivalent focused session-file-repair fix, then track stale ended-session reachability separately if it still reproduces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: repeatedly trigger successful repair on a session file that keeps needing repair, and current main writes a unique retained `.bak-*` on each successful pass.

## Solution Assessment

Yes. The narrow maintainable solution is to remove or bound success-path backups in `session-file-repair.ts`; the open PR's delete-after-success approach is smaller than adding TTL/config policy for this leak.
