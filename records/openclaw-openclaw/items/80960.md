## Summary

Keep open: current main still retains a fresh full `.bak-*` backup on each successful repair, and the live fixing PR is open with closing syntax for this issue rather than merged.

## What This Changes

The issue asks OpenClaw to stop successful session-file repair from leaving unlimited `.bak-*` snapshots on repeated repairs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80969, or close/replace it with an equivalent focused fix that cleans successful repair backups and updates the tests and transcript hygiene docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: repeatedly invoke `repairSessionFileIfNeeded` on a session that needs repair and current main writes a unique retained `.bak-*` file each time; I did not run tests because this was a read-only checkout review.

## Solution Assessment

Yes. Deleting the transient backup after successful `replaceFileAtomic` or bounding per-session backups is the narrow maintainable fix; the open PR's delete-after-success approach is narrower than adding a new setting or broader storage policy.
