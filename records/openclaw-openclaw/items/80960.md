## Summary

Keep open: current main still retains every successful session-file repair backup, and an open linked PR already targets the narrow fix with closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Prevent `repairSessionFileIfNeeded` from accumulating unbounded `.bak-<pid>-<timestamp>` session snapshots across repeated repair invocations.

## Best Possible Solution

Land the focused cleanup in https://github.com/openclaw/openclaw/pull/80969 or an equivalent replacement, then close this issue from the merged fix and track ended-session reachability separately if it still reproduces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: repeatedly repair a session file that keeps needing repair, and current main writes a fresh retained `.bak-*` on each successful repair; I did not execute it because this review was read-only.

## Solution Assessment

Yes. Deleting the backup only after successful atomic replacement, while retaining/reporting it on cleanup failure, is the narrowest maintainable fix for the success-path disk-growth bug.
