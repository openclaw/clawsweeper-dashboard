## Summary

Keep open. Current main still has the terminal lifecycle branch that returns after a truthy chat registry `peek()` but undefined `shift()`, so the reported final chat event suppression remains a valid narrow gateway bug; the only linked implementation PR is closed unmerged.

## What This Changes

Terminal gateway lifecycle handling should emit a final chat event even when a chat run registry entry disappears between `peek()` and `shift()`.

## Best Possible Solution

Patch the gateway finalizer to fall back to the already resolved session/run identifiers on a shift miss, preserve cleanup, add a focused regression test, and record the user-facing fix in the changelog.

## Reproduction Assessment

Yes for a high-confidence source-level path: force a truthy registry `peek()` and an undefined `shift()` in `createAgentEventHandler`, emit a terminal lifecycle event, and observe that current main skips the final chat payload. I did not run a live gateway timing reproduction for the intermittent drift trigger.

## Solution Assessment

Yes. Fixing `finalizeLifecycleEvent` is the narrowest maintainable solution because the client already retires visible runs when it receives the terminal chat payload.

Fix the current gateway chat finalization bug. When `chatRunState.registry.peek(evt.runId)` finds a chat link but `chatRunState.registry.shift(evt.runId)` returns undefined, do not return before finalizing; emit the same terminal chat final/error payload using the already resolved `sessionKey` and `eventRunId` fallback, preserve cleanup, and avoid duplicate finals. Add a focused regression test in `src/gateway/server-chat.agent-events.test.ts` that simulates the peek/shift drift. Add a single-line user-facing fix entry to `CHANGELOG.md`. Do not change provider behavior or broader lifecycle policy.
