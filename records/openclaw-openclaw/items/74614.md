## Summary

Keep open. Current `main` still has the reported gateway terminal-lifecycle branch where a truthy chat registry peek followed by a missing shift returns before broadcasting the final `chat` event, and the linked implementation PR is closed unmerged.

## What This Changes

Ensure gateway terminal lifecycle events still emit a final chat event using resolved session/run fallbacks when `registry.peek()` succeeds but `registry.shift()` returns `undefined` after streamed output.

## Best Possible Solution

Fix the gateway producer path so a shift miss after a resolved chat link falls back to the already computed `sessionKey` and client run id, then add a focused regression test that forces `peek()` to return a link while `shift()` returns `undefined` in the same terminal lifecycle handler call.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a linked chat run, force `chatRunState.registry.shift(evt.runId)` to return `undefined` after `peek()` has returned a link, then emit a terminal lifecycle `end` or `error`; current `main` returns before broadcasting the final `chat` event.

## Solution Assessment

Yes. The narrow gateway fallback is the best fix because it preserves the producer-side terminal event contract instead of adding a UI/TUI recovery path after the final event has already been suppressed.
