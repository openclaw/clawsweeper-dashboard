## Summary

Keep open. Current main still returns before `emitChatFinal(...)` when `peek()` finds a chat link but `shift()` misses, so the reported stuck-streaming finalization gap remains a narrow valid gateway bug; the only linked implementation PR was closed without merge.

## What This Changes

A terminal gateway chat lifecycle event should still emit the final chat payload when the chat-run registry entry disappears between `peek()` and `shift()`.

## Best Possible Solution

Patch the gateway finalizer to use the already resolved fallback session/run identifiers on a shift miss, preserve terminal cleanup, add a focused regression test, and record the user-facing fix in the changelog.

## Reproduction Assessment

Yes for a source-level unit reproduction: add a chat registry link, make `registry.shift(evt.runId)` return `undefined` after `peek()` succeeds, emit a terminal lifecycle event, and observe current main skipping the final chat payload. I did not live-reproduce the intermittent timing drift trigger.

## Solution Assessment

Yes. Fixing `finalizeLifecycleEvent` is the narrowest maintainable solution because it already resolves fallback identifiers before shifting and clients already retire visible runs from the terminal chat payload.
