## Summary

Keep open. Current main still has the reported gateway lifecycle branch that clears context and returns when `peek()` finds a chat link but `shift()` misses, before emitting the terminal final chat event. The issue is recent, source-backed, and the related focused fix PR is closed without merge, so no close reason applies.

## What This Changes

The issue requests a gateway chat lifecycle fix so terminal `end` or `error` events still emit a final chat event when the run registry entry drifts between `peek()` and `shift()`.

## Best Possible Solution

Update `src/gateway/server-chat.ts` so terminal lifecycle handling emits exactly one final chat event using the resolved session/run fallback when `registry.shift(evt.runId)` misses, then add a focused regression test and a user-facing changelog fix entry.

## Reproduction Assessment

Yes. A focused unit test can arrange for `chatRunState.registry.peek(evt.runId)` to return a link and the subsequent `shift(evt.runId)` to return `undefined` during a terminal lifecycle event; current main then returns before broadcasting a final `chat` event.

## Solution Assessment

Yes. The producer-side fallback is the narrowest maintainable fix because it repairs the dropped terminal event at the gateway rather than adding another downstream UI recovery path after the event has already been suppressed.
