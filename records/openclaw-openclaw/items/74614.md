## Summary

Keep open. Current main still has the reported terminal lifecycle branch that clears run context and returns when `peek()` found a chat link but `shift()` misses, so the fallback `emitChatFinal(...)` path is still skipped; the only linked focused PR is closed without merge.

## What This Changes

The issue asks the gateway chat lifecycle finalizer to emit a terminal chat final event when a run registry entry disappears between `peek()` and `shift()` after streamed assistant output.

## Best Possible Solution

Update `finalizeLifecycleEvent` so terminal `end` and eligible `error` lifecycle events emit exactly one final chat event whenever resolved session/run context exists, falling back to the already computed `sessionKey` and client run id when `shift()` misses, then cover that branch with a focused gateway agent-event regression test.

## Reproduction Assessment

Yes. A focused unit reproduction can arrange a chat registry `peek()` hit followed by a `shift()` miss during a terminal lifecycle `end` or eligible `error` event, then assert that current main returns without broadcasting a final `chat` event.

## Solution Assessment

Yes. The narrow producer-side fallback in the gateway lifecycle finalizer is the best repair because it fixes the dropped terminal event at the source; adding more Control UI/TUI recovery would only mask one symptom after the final event has already been suppressed.
