## Summary

Keep open. Current main still has the reported terminal lifecycle branch that returns after a `peek()` hit but before `emitChatFinal(...)` when `shift()` misses, the registry API still permits a miss, Control UI state still depends on receiving the final chat event, and the linked focused fix PR is closed without merge.

## What This Changes

Ensure gateway terminal lifecycle handling still emits a final chat event when the chat-run registry entry is unavailable at `shift()` time after streamed output was already delivered.

## Best Possible Solution

Update the gateway producer path so a terminal lifecycle event with resolved session/run context always emits the final chat event once, falling back to the resolved `sessionKey` and client run id when `shift()` misses, and cover that branch with a focused regression test.

## Reproduction Assessment

Yes. A focused unit reproduction can force `chatRunState.registry.peek(evt.runId)` to return a link and the subsequent `shift(evt.runId)` to return `undefined` during a terminal lifecycle `end` or `error`; current main then returns before broadcasting a final `chat` event.

## Solution Assessment

Yes. The narrow gateway fallback is the best repair because it fixes the producer-side terminal event drop instead of adding another Control UI/TUI recovery path after the final event has already been suppressed.
