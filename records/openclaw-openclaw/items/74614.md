## Summary

Keep open. Current main still contains the reported early return before terminal chat finalization, the registry API still allows a shift miss, clients depend on the final chat event to leave streaming/running state, and the linked focused fix PR is closed without merge.

## What This Changes

The issue requests that gateway terminal lifecycle handling emit a final chat event even when `registry.peek()` finds a chat link but the subsequent `registry.shift()` misses after streamed output.

## Best Possible Solution

Repair the gateway producer path so the terminal lifecycle handler falls back to the already-resolved `sessionKey` and client run id when `shift()` misses, then cover that branch with a focused gateway regression test.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a linked chat run, force `chatRunState.registry.shift(evt.runId)` to return `undefined` after `peek()` returned a link, then emit a lifecycle `end` or `error`; current main returns before broadcasting a final chat event.

## Solution Assessment

Yes. The narrow gateway fallback is the best fix because it preserves the producer-side terminal event contract instead of adding another UI/TUI recovery path after the final event has already been dropped.
