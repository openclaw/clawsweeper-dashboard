## Summary

Keep open: current main still treats `env` block values as literal strings, but the reported `file:` shorthand is not a documented config/env contract and overlaps with the existing structured SecretRef mechanism, so resolving it needs maintainer product/security direction rather than conservative cleanup.

## What This Changes

The issue asks for `file:...` values in `openclaw.json` `env` entries to be expanded from external files before provider auth uses them.

## Best Possible Solution

Keep structured SecretRef objects as the canonical file-backed secret path unless maintainers explicitly approve a new `file:` env shorthand; if approved, implement it with the same file-provider safety rules, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: `applyConfigEnvVars` copies direct `env` string values into the runtime environment and only special-cases `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` would remain literal. I did not run a live xAI request because the code path is enough to establish current-main behavior.

## Solution Assessment

No clear best fix yet: the maintainable existing path is a structured SecretRef on `models.providers.*.apiKey` or `plugins.entries.xai.config.webSearch.apiKey`, while adding `file:` under `env` would be new config syntax that needs maintainer approval and security semantics.
