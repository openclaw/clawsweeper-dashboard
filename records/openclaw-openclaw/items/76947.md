## Summary

Keep open: current main source-reproduces the literal `file:` behavior, but `file:` env shorthand is not a documented OpenClaw config contract and overlaps the structured SecretRef file-provider path, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

The issue asks for `file:` values in the `openclaw.json` `env` block to be resolved from files before provider authentication.

## Best Possible Solution

Keep structured SecretRef objects as the canonical file-backed secret path unless maintainers explicitly approve a new `file:` env shorthand; if approved, implement it through the existing file-provider safety model with docs and regression coverage.

## Reproduction Assessment

Yes, at source level: a config `env` value of `file:secrets/xai-api-key.txt` flows through `applyConfigEnvVars` as a literal string. I did not run a live xAI request because the current-main source path is enough to establish that behavior.

## Solution Assessment

No clear best fix yet: resolving `file:` inside `env` would create a new config-secret shorthand, while the documented maintainable route is structured SecretRef on supported credential fields. Maintainers should decide whether to add the shorthand, reject it with clearer diagnostics/docs, or improve the SecretRef migration path.
