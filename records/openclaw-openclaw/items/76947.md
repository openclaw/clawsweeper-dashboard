## Summary

Keep open: current main still treats config `env` values as literal strings, and the linked implementation candidate remains open, so this issue should not be cleanup-closed before maintainers decide the config-secret contract.

## What This Changes

The issue requests `file:` values in the `openclaw.json` `env` block to be resolved from files before provider authentication uses them.

## Best Possible Solution

Maintainers should decide whether `file:` under `env` is a supported public config contract; if approved, implement it through SecretRef-grade file resolution with docs and regression coverage, otherwise steer users to structured SecretRefs or process env.

## Reproduction Assessment

Yes, at source level: `applyConfigEnvVars` currently copies direct config env strings literally after only skipping existing env values and `${VAR}` placeholders. I did not run a live xAI request because the config path itself establishes the reported literal-value behavior.

## Solution Assessment

Unclear: resolving `file:` under `env` may match the requested behavior, but it is not currently documented and overlaps the existing SecretRef file-provider model. The maintainable solution is either to reject the shorthand clearly or implement it using the same guarded secret resolver semantics.
