## Summary

Keep open: current main still treats config `env` values as literal strings, and open PR #77050 explicitly proposes the implementation for this issue, so the issue should remain open until that PR is reviewed and either merged or closed.

## What This Changes

The issue requests resolving `file:<relative path>` values in the `openclaw.json` `env` block into file contents before provider authentication uses the env var.

## Best Possible Solution

Review PR #77050 against the existing SecretRef model; if maintainers accept the shorthand, land it with regression tests, docs, and changelog coverage, otherwise document the supported SecretRef/file-provider path more clearly.

## Reproduction Assessment

Yes, at source level: current main collects string env entries and assigns them directly unless they contain `${VAR}`, so `file:secrets/xai-api-key.txt` remains literal during gateway config finalization. I did not run a live xAI request because the source path establishes the reported behavior.

## Solution Assessment

Unclear: adding `file:` under `env` may be useful, but it is new config-secret syntax while the documented maintainable path is structured SecretRef objects. The proposed PR should be reviewed for product fit and SecretRef-equivalent safety before merge.
