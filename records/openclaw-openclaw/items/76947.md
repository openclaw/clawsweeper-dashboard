## Summary

Keep open: current main still copies `file:` config env values literally, and #77050 is an open closing PR for the same behavior, so this issue should remain open until that PR lands or maintainers choose a different secret-syntax direction.

## What This Changes

The issue requests resolving `file:<path>` values in `openclaw.json` `env` entries before provider authentication uses those environment variables.

## Best Possible Solution

Review #77050 against the existing SecretRef security model, then either land a documented `file:` env shorthand with regression coverage or explicitly steer users to structured file-backed SecretRefs.

## Reproduction Assessment

Yes, at source level: current main’s `applyConfigEnvVars` copies `file:secrets/xai-api-key.txt` as a literal env value, and the gateway calls that path during config activation. I did not run a live xAI request because the read-only source path establishes the reported literal-value behavior.

## Solution Assessment

Unclear pending maintainer decision: #77050 is the right current implementation candidate, but accepting `file:` inside `env` is new config-secret syntax that should be checked against the structured SecretRef file-provider contract.
