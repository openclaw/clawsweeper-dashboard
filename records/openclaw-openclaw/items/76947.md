## Summary

Keep open: current main still passes `env` values through as literal strings, while the documented file-backed secret path is the structured SecretRef provider model. This is valid source-level behavior to discuss, but supporting `file:` inside `env` needs maintainer product/security direction rather than cleanup closure.

## What This Changes

The issue asks for `file:...` values in `openclaw.json` `env` entries to be expanded from files before xAI/provider authentication.

## Best Possible Solution

Keep structured SecretRef objects as the canonical file-backed secret path unless maintainers approve a `file:` env shorthand; if approved, implement it through the existing SecretRef/file-provider safety model with docs and regression coverage.

## Reproduction Assessment

Yes, at source level: set `env.XAI_API_KEY` to `file:secrets/xai-api-key.txt` with no process env override and `applyConfigEnvVars` will assign that literal string. I did not run a live xAI request because the read-only source path is enough to establish current-main behavior.

## Solution Assessment

Unclear/no: blindly reading `file:` inside `env` is not yet the best proven fix because existing SecretRefs already own file-secret safety and activation semantics. The maintainable path is a maintainer decision between adding the shorthand via that resolver, rejecting it, or improving diagnostics/docs.
