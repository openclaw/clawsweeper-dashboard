## Summary

Keep open: current main still treats config `env` values as literal strings, and an open PR already references this issue with closing syntax, so cleanup should wait for maintainers to accept or reject that fix path.

## What This Changes

The issue requests resolving `file:` references in `openclaw.json` `env` entries before those values are used as provider environment variables.

## Best Possible Solution

Review the linked PR and decide the public config contract: either implement `file:` env shorthand through SecretRef-grade guarded file resolution with docs and regression coverage, or document that users should use structured SecretRefs or process env instead.

## Reproduction Assessment

Yes, at source level: `applyConfigEnvVars` copies config env strings literally after only skipping existing env values and `${VAR}` placeholders. A live xAI request is not needed to prove the config-path behavior, though it would prove the provider symptom end to end.

## Solution Assessment

Unclear: resolving `file:` under `env` matches the requested behavior, but it adds a new secret-loading config scheme. The maintainable solution should either reuse the existing SecretRef file-provider safety model or explicitly reject the shorthand with docs.
