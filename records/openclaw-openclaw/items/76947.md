## Summary

Keep open: current main still copies `file:` config env values literally, and an open closing PR already targets this issue, so cleanup should wait for maintainer review of that implementation and the secret-handling contract.

## What This Changes

The issue requests resolving `file:<path>` values in the `openclaw.json` `env` block to file contents before provider authentication uses the env var.

## Best Possible Solution

Choose one canonical contract: either land a reviewed `file:` env shorthand with SecretRef-equivalent file safety and docs, or reject the shorthand and improve diagnostics/docs so users use supported SecretRef credential fields.

## Reproduction Assessment

Yes, at source level: configure `env.XAI_API_KEY` as `file:secrets/xai-api-key.txt` with no process env override, and the current loader assigns that exact string. I did not run a live xAI request because the source path establishes the literal env value.

## Solution Assessment

Unclear: PR #77050 is the active implementation candidate, but the underlying decision is whether OpenClaw should support a new env shorthand or direct users to existing SecretRef file providers. A maintainable fix should reuse the existing secret safety semantics and include docs plus regression coverage.
