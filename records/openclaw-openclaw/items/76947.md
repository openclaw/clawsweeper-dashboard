## Summary

Keep open: current main still treats config `env` values as literal strings, while `file:` env shorthand is not a documented contract and overlaps the structured SecretRef file-provider path, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

The issue requests expansion of `file:` values in the `openclaw.json` `env` block so xAI API keys can be read from external files before provider authentication.

## Best Possible Solution

Keep structured SecretRef objects as the canonical file-backed secret path unless maintainers explicitly approve a new `file:` env shorthand; if approved, implement it through the existing file-provider safety model with docs and regression coverage.

## Reproduction Assessment

Yes, at source level: `applyConfigEnvVars` copies direct config `env` strings into the runtime environment and only skips `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. I did not run a live xAI request because the read-only source path is enough to establish current-main behavior.

## Solution Assessment

No clear best fix yet: adding `file:` under `env` would be new config-secret syntax, while the existing maintainable route is structured SecretRef on supported credential fields. Maintainers should decide whether to add the shorthand, reject it, or improve diagnostics/docs for the SecretRef path.
