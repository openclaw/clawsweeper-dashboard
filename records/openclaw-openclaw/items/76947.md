## Summary

Keep open: current main still has the reported source-level behavior, and the prior ClawSweeper comment matches the evidence, but `file:` values in `env` are not a documented OpenClaw config contract and overlap with structured SecretRefs, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

Support or diagnose `file:...` values in the `env` block of `openclaw.json` so API-key env vars can be backed by files instead of inline strings.

## Best Possible Solution

Keep structured SecretRef file providers as the canonical file-backed secret path unless maintainers approve a new `file:` env shorthand; if approved, implement it through the same resolver, safety checks, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: setting `env.XAI_API_KEY` or `env.vars.XAI_API_KEY` to `file:secrets/xai-api-key.txt` would be copied as that literal string by `applyConfigEnvVars`. I did not run a live xAI request because the config/env path is clear from current source.

## Solution Assessment

Unclear/no: adding `file:` to `env` is not clearly the best fix because docs already define file-backed secrets as structured SecretRefs on supported credential paths. Maintainers should choose between documenting/rejecting the shorthand, improving diagnostics, or adding it with full SecretRef-equivalent safety semantics.
