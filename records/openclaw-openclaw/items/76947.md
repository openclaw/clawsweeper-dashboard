## Summary

Keep open: current main still treats config `env` values as literal strings, while `file:` env shorthand is not a documented OpenClaw contract and overlaps the structured SecretRef file-provider path, so this needs maintainer product/security direction.

## What This Changes

The issue asks OpenClaw to resolve `file:...` values in `openclaw.json` `env` entries before provider authentication.

## Best Possible Solution

Keep structured SecretRef objects as the canonical file-backed secret path unless maintainers explicitly approve a compatible `file:` shorthand; any implementation should reuse SecretRef validation, permission checks, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: `applyConfigEnvVars` copies direct config `env` strings into the runtime environment and only skips `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. I did not run a live xAI request because the read-only source path is enough to establish current-main behavior.

## Solution Assessment

Unclear: adding `file:` under `env` would create new config-secret syntax, while the existing maintainable route is structured SecretRef on supported credential fields. Maintainers should decide whether to add the shorthand through the SecretRef safety model or improve diagnostics/docs for unsupported shorthand.
