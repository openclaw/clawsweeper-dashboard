## Summary

Keep open: current main still treats config `env` values as literal strings, and an open closing PR already tracks the proposed `file:` resolver, so this issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests resolving `file:<path>` values in `openclaw.json` `env` entries into file contents before provider auth uses those env vars.

## Best Possible Solution

Review #77050 against the existing SecretRef file-provider safety model, then either land a documented resolver with regression coverage or reject the shorthand with clearer diagnostics/docs pointing users to structured SecretRefs.

## Reproduction Assessment

Yes, at source level: with no existing `XAI_API_KEY`, `env.XAI_API_KEY: "file:secrets/xai-api-key.txt"` is collected and assigned literally by `applyConfigEnvVars`. I did not run a live xAI call because the read-only source path is enough to establish current-main behavior.

## Solution Assessment

Unclear: adding `file:` under `env` may solve the report, but it expands the config-secret contract and should preserve the structured SecretRef file-provider safety rules. The narrow maintainable path is to review the linked PR with that security model in mind.
