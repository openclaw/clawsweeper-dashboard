## Summary

Keep open: current main still passes `file:` env values through literally, and #77050 is an open closing PR for the same path, so this should wait for maintainer review/merge or an explicit product decision.

## What This Changes

Request that config env values such as `env.XAI_API_KEY: "file:secrets/xai-api-key.txt"` be resolved from files instead of passed to providers literally.

## Best Possible Solution

Review #77050 against the SecretRef security model, then either land a documented file-backed env shorthand with regression coverage or document structured SecretRef as the supported path.

## Reproduction Assessment

Yes, at source level. `applyConfigEnvVars` copies string env entries into runtime env after only blocking existing env values and `${VAR}` references, so a `file:...` string remains literal.

## Solution Assessment

Unclear. Resolving `file:` under `env` is a new shorthand beside structured SecretRef file providers, so the maintainable fix is to review #77050 for path-security, docs, and tests or reject it in favor of SecretRef guidance.
