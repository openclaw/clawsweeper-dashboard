## Summary

Keep open: current main still passes `file:` config env values literally, and #77050 is an open unmerged closing PR for the same behavior, so this issue should remain open for PR and maintainer review rather than cleanup closure.

## What This Changes

Requests support for resolving `file:` values in `openclaw.json` `env` entries so API keys such as `XAI_API_KEY` read from files instead of being passed literally.

## Best Possible Solution

Review #77050 against the existing SecretRef model; either land a documented, tested `file:` env implementation with equivalent safety checks or keep structured SecretRef objects as the supported file-backed secret path.

## Reproduction Assessment

Yes, at source level: configuring `env.XAI_API_KEY` as `file:secrets/xai-api-key.txt` reaches `applyConfigEnvVars` and is assigned literally when the process env lacks `XAI_API_KEY`. I did not run a live xAI request because the current-main source path establishes the reported behavior.

## Solution Assessment

Unclear: resolving `file:` in `env` may solve the user-visible failure, but it is a new shorthand beside the structured SecretRef file-provider contract. The maintainable solution should be decided while reviewing #77050 rather than closing this issue early.
