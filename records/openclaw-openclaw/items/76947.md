## Summary

Keep open: current main still passes config `env` values through as literal strings, while the documented contract treats `env` as inline strings and file-backed secrets as structured SecretRefs, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

The issue requests resolving `file:` values in the config `env` block before provider auth uses those env vars.

## Best Possible Solution

Keep structured SecretRef objects as the canonical file-backed secret path unless maintainers explicitly approve an `env` shorthand, then implement it through the existing secret safety model with docs and regression coverage.

## Reproduction Assessment

Yes, at source level: the config env loader copies string values into the runtime env and only special-cases `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. I did not run a live xAI request because the current-main source path is enough to establish the reported behavior.

## Solution Assessment

Unclear: resolving `file:` under `env` is not currently the documented SecretRef contract and would need product/security approval. The safer existing direction is to use structured SecretRefs on supported credential fields or improve diagnostics/docs if maintainers do not want the shorthand.
