## Summary

Keep open: current main still has source-level literal handling for config `env` values, and open PR #77050 already references this issue with closing syntax, so the issue should remain open until that PR is reviewed, landed, or rejected.

## What This Changes

The issue asks for `openclaw.json` env entries such as `XAI_API_KEY: "file:secrets/..."` to be resolved to file contents instead of passed literally to providers.

## Best Possible Solution

Review PR #77050 against the existing SecretRef file-provider contract; either land a documented, tested shorthand that preserves those safety semantics, or reject/document the shorthand in favor of structured SecretRef objects.

## Reproduction Assessment

Yes, source-level: with no existing `XAI_API_KEY`, `applyConfigEnvVars` copies `env.XAI_API_KEY: "file:secrets/..."` into the runtime env as the same literal string. A live xAI call is not needed to establish that current-main behavior.

## Solution Assessment

Unclear: adding `file:` to `env` is new config-secret syntax, while OpenClaw already has a structured file SecretRef path. The paired PR is the right place to decide whether that shorthand is maintainable and safe.
