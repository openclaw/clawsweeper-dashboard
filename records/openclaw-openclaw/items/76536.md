## What This Changes

The PR updates capability-provider runtime fallback to cold-load scoped manifest-contract providers when absent from the startup registry, adds a regression test, and records the fix in CHANGELOG.

## Best Possible Solution

## Reproduction Assessment

Yes. Source inspection on current main shows manifest contracts can identify enabled external provider plugin ids, but the runtime fallback only consults the compatible loaded registry; if the scoped provider plugin was not startup-loaded, provider entries remain empty and the TTS path can report `no provider registered`.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix because it preserves startup-registry reuse and only invokes the existing scoped runtime loader after a loaded-registry miss; the remaining blocker is branch freshness and validation, not design direction.
