Keep this PR open: current main still has the reported manifest-contract provider cold-load regression, and the PR is a narrow source-level fix with focused regression coverage. The only concrete merge blocker I found is the required user-facing changelog entry.

## What This Changes

The PR restores scoped cold runtime loading for enabled manifest-declared capability providers when the compatible loaded startup registry does not contain the requested provider.

## Best Possible Solution

Keep the scoped cold-load fallback, add one Unreleased Fixes changelog bullet, then merge after targeted capability-provider validation and the changed gate are green.

## Reproduction Assessment

Yes. Source inspection on current main shows manifest contracts can select an enabled external provider, but the fallback only consults a compatible loaded registry and returns no entries when that provider was not startup-loaded.

## Solution Assessment

Yes. The PR is the narrowest maintainable repair because it preserves startup-registry reuse and only cold-loads the scoped manifest-derived plugin ids on a miss; the remaining change is documentation/changelog hygiene, not a runtime redesign.
