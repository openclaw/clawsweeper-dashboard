Before merge, it needs real CLI behavior proof from the contributor and a small cleanup so the parse reason is not repeated through the error cause chain.

## What This Changes

The PR wraps `secrets apply --from` plan JSON parsing with a path-aware error and adds a malformed-plan regression test.

## Best Possible Solution

Merge a focused path-aware parse error fix after the contributor posts redacted real CLI output and the wrapper formats the parse reason exactly once.

## Reproduction Assessment

Yes, from source inspection: current main parses the plan JSON directly in `readPlanFile`, then the outer apply handler formats the resulting syntax error without file-path context. I did not run the repro because this review is read-only.

## Solution Assessment

Mostly yes: fixing `readPlanFile` is the narrow owner-boundary change because that helper has both the path and parse operation. The implementation should avoid combining an embedded formatted cause with `{ cause: err }`, since the outer formatter expands causes again.
