## What This Changes

The PR wraps `secrets apply --from` plan JSON parsing to include the plan file path in malformed JSON errors and adds a focused CLI regression test.

## Best Possible Solution

Merge a narrow path-aware parse error fix once real CLI output proves the user-facing command and the wrapper formats the parse reason exactly once.

## Reproduction Assessment

Yes, from source inspection: current main parses the plan JSON directly in `readPlanFile`, then the outer apply handler formats the resulting syntax error without file-path context. I did not run the full CLI path because this review is read-only.

## Solution Assessment

Mostly yes: `readPlanFile` is the narrow fix point because it has both the path and parse operation. The implementation should avoid combining an embedded formatted cause with `{ cause: err }`, since the outer formatter expands causes again.
