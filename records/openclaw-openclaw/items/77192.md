## Summary

Keep this PR open because it fixes the misleading alias wording in the right area, but the replacement text introduces an overbroad claim that `/new` always leaves the previous session accessible.

## What This Changes

This PR updates `docs/tools/slash-commands.md` to describe `/new` and `/reset` separately instead of calling `/reset` the reset alias for `/new`.

## Best Possible Solution

Revise the docs to remove the alias claim while scoping the preserved-session language to Control UI, and describe Gateway `/new` and `/reset` as explicit reset triggers with distinct intent/reason wording.

## Reproduction Assessment

Yes. Current main contains the misleading alias wording, and source inspection shows Gateway `/new` and `/reset` share reset-trigger handling while Control UI intercepts only `/new` into a separate dashboard-session action.

## Solution Assessment

No. Removing the alias wording is the right direction, but the proposed wording should not promise previous-session access for every core `/new` use.
