## Summary

Keep open: the PR addresses a real docs problem, but the replacement sentence overstates `/new` by promising accessible previous sessions in the core slash-command docs. Current source shows the dashboard-session behavior is a Control UI intercept, while Gateway slash handling treats `/new` and `/reset` as reset triggers on the resolved session key.

## What This Changes

This PR updates `docs/tools/slash-commands.md` to describe `/new` and `/reset` separately instead of presenting `/reset` as `/new`'s reset alias.

## Best Possible Solution

Keep the alias removal, but revise the replacement wording so it distinguishes Gateway reset semantics from the Control UI’s dashboard-session intercept without promising previous-session access for all `/new` uses.

## Reproduction Assessment

Yes. Current main still contains the misleading alias wording, and source inspection shows the behavior split: Gateway treats `/new` and `/reset` as reset triggers, while Control UI intercepts `/new` into a separate new-session action.

## Solution Assessment

No. Removing the alias wording is the right direction, but the proposed replacement should avoid the broad claim that `/new` always preserves an accessible previous session.
