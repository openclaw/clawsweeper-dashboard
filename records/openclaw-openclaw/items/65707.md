## Summary

Keep open. Current main still leaves cache-ttl pruning at the default three assistant turns, and the PR is useful source work, but the latest branch still passes the raw prompt mode from the normal run path. That misses runs where the runner forces the actual system prompt to minimal through `toolsAllow`, so the patch needs one focused update before merge.

## What This Changes

The PR adds prompt-mode plumbing to embedded extension setup, caps cache-ttl context-pruning `keepLastAssistants` at 1 in minimal prompt mode, passes that mode from run and compaction paths, and adds extension-factory tests for minimal versus full prompt mode.

## Best Possible Solution

Land an updated version that passes the effective prompt mode into `buildEmbeddedExtensionFactories` for the run path, keeps the compaction path's minimal/full signal, preserves `keepLastAssistants: 0`, and covers full chat, subagent/cron minimal, and tools-allow minimal behavior in focused tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a normal-session run with a non-empty `toolsAllow` renders the system prompt with `effectivePromptMode: "minimal"`, while the PR's run-path factory argument remains the raw `promptMode`, so pruning keeps the default three assistant turns.

## Solution Assessment

No. The branch direction is narrow and maintainable, but the run path should pass `effectivePromptMode` or compute the same value before pruning setup so context pruning matches the prompt actually sent to the model.
