## Summary

Keep this PR open: the optimization is focused and still useful, but the branch has a correctness blocker around raw versus effective prompt mode and lacks after-fix real behavior proof from the contributor.

## What This Changes

The PR adds prompt-mode plumbing to embedded context-pruning setup so cache-ttl pruning caps `keepLastAssistants` at 1 for minimal-prompt runs, with tests and run/compaction call-site changes.

## Best Possible Solution

Land a rebased branch that derives pruning policy from the same effective prompt mode used for prompt construction, preserves explicit `keepLastAssistants: 0`, covers subagent, cron, and `toolsAllow` minimal paths, and includes redacted real-run proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: the branch passes raw `promptMode` while the runner separately forces `effectivePromptMode` to `minimal` for non-empty `toolsAllow`. I did not run tests because this review was read-only.

## Solution Assessment

No. The direction is narrow, but the branch should pass `effectivePromptMode` or an equivalent prepared signal into pruning setup and add coverage for the `toolsAllow` path before merge.
