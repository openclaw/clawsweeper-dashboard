## Summary

Keep this PR open: the optimization is plausible, but the latest branch still passes the raw prompt mode into pruning setup, so `toolsAllow` runs that actually render a minimal prompt would keep the default three assistant turns.

## What This Changes

The PR caps cache-ttl context-pruning `keepLastAssistants` at 1 when embedded runner prompt mode is minimal, wires that prompt-mode argument through run and compaction setup, and adds focused extension-factory tests.

## Best Possible Solution

Land an updated version that passes the effective prompt mode into pruning setup, keeps the compaction path’s minimal/full signal, preserves `keepLastAssistants: 0`, and covers full chat, subagent/cron minimal, and `toolsAllow` minimal behavior.

## Reproduction Assessment

Yes. Source inspection is enough: a run with non-empty `toolsAllow` renders with `effectivePromptMode: "minimal"`, while the PR branch passes raw `promptMode` to pruning setup, so the new cap does not apply on that supported path.

## Solution Assessment

No. The direction is narrow, but the run path should pass `effectivePromptMode` or compute the same effective signal before building extension factories, then add a regression test for the `toolsAllow` case.
