## Summary

Keep this PR open. Current main still lacks the requested minimal-prompt context-pruning cap, and the PR remains a useful narrow implementation candidate after switching away from the unreliable `skillsSnapshot` signal, but the run-path plumbing should use the effective prompt mode so all runs that actually render a minimal prompt get the same pruning behavior.

## What This Changes

The PR passes prompt mode into embedded extension setup, caps context-pruning `keepLastAssistants` to 1 for minimal prompt mode, forwards that mode from run and compaction paths, and adds focused extension-factory tests.

## Best Possible Solution

Land this PR or an equivalent patch after routing the effective prompt mode into pruning setup, preserving full-chat retention at 3 and explicit `keepLastAssistants: 0`, with focused tests for minimal, full, and tools-allow minimal behavior.

## Reproduction Assessment

Yes. A static reproduction is clear: current main forces `effectivePromptMode` to `minimal` when `toolsAllow` is present, while the PR diff passes the raw `promptMode` into pruning setup; a regular-session tools-allow test would show `keepLastAssistants` remains 3 instead of 1.

## Solution Assessment

No. The branch direction is right, but the run path should pass `effectivePromptMode` or equivalent into `buildEmbeddedExtensionFactories` so the pruning cap follows the prompt actually sent to the model.
