## Summary

Keep open: this is a valid documented cron retry bug and the PR covers the no-status structured payload path with sufficient terminal proof, but current head still lets explicit 5xx status classification override the new `server_error` reason.

## What This Changes

The PR adds `server_error` to failover/runtime/cron protocol types, classifies structured OpenAI-compatible `server_error` payloads, makes cron retry consult the failover classifier, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Keep the fix in the failover and cron ownership boundary, but adjust classifier precedence so structured `server_error` is honored for no-status and explicit 5xx payloads while generic status-only 5xx failures remain `timeout`.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current docs advertise `retryOn: ["server_error"]`, while current main only raw-regexes 5xx text and omits `server_error` from the typed failover/protocol path. The PR body's terminal proof covers the real CronService no-status structured payload path after the patch.

## Solution Assessment

No, not yet. The PR direction is the narrow maintainable fix, but it should let `messageReason === "server_error"` win for explicit 5xx statuses while preserving the added 4xx auth/billing/format precedence and existing generic 5xx timeout behavior.
