## Summary

Keep open. The PR fixes the right cron/failover boundary, but the latest head still lets explicit 500/502/504 status classification collapse structured `server_error` payloads back to `timeout`, so a merge would leave part of the documented retry behavior broken.

## What This Changes

The PR adds a structured `server_error` failover reason, wires it into cron retry/state/protocol handling, adds regression coverage, updates the changelog, and includes a small version-test env cleanup.

## Best Possible Solution

Update the classifier so structured `server_error` wins for explicit 5xx responses while explicit 4xx statuses still win as auth, billing, or format failures, then land this narrow fix with the focused regressions.

## Reproduction Assessment

Yes. This is source-reproducible: current docs accept `retryOn: ["server_error"]`, current main only retries raw 5xx text and classifies the structured Codex/OpenAI payload as `timeout`, and the PR adds the right focused cron regression for the string-only path.

## Solution Assessment

No, not yet. The direction is the narrow maintainable fix, but the status classifier still returns `timeout` first for explicit 500/502/504 structured payloads; letting `server_error` win only on 5xx statuses is the safer completion.
