## Summary

Keep open. Current main still lacks first-class cron `server_error` retry handling, but this PR should not merge as-is because the latest diff still needs a safer no-status/string-error classifier path and a required changelog entry.

## What This Changes

This PR adds `server_error` as a failover and cron protocol reason, maps it to status 500, excludes it from auth-profile failure state, makes cron retry matching consult failover classification, and adds focused regression coverage.

## Best Possible Solution

Land a rebased or replacement patch that makes `server_error` a first-class failover and cron protocol reason while preserving parsed/explicit 4xx classification ahead of any structured server-error shortcut.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `retryOn: ["server_error"]`, return a structured Codex/OpenAI `server_error` string without numeric 5xx text, and current cron retry matching will not treat it as a `server_error` retry.

## Solution Assessment

No, not as-is. The PR is the right narrow direction, but it should be rebased onto current failover classification and must let parsed or explicit 4xx status handling win before `server_error`; it also needs the required changelog entry.
