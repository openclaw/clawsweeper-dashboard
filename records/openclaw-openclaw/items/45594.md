## Summary

Keep open. Current main still exposes `cron.retry.retryOn: ["server_error"]` without first-class `server_error` failover/schema support, so the PR addresses a real contract gap; however the PR branch is dirty against current main and the proposed classifier shortcut still has a blocking 4xx string-precedence bug.

## What This Changes

The PR adds `server_error` to failover and cron protocol contracts, maps it to status 500, excludes it from auth-profile failure state, makes cron retry matching consult failover classification, and adds regression coverage for structured OpenAI/Codex `server_error` payloads.

## Best Possible Solution

Rebase the change onto the current failover-classification structure, add `server_error` as a first-class failover and cron protocol reason, classify structured OpenAI/Codex server failures without overriding explicit or parsed 4xx auth/billing/format precedence, and keep focused cron retry plus protocol conformance coverage with the patch.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: configure `retryOn: ["server_error"]`, return a structured Codex/OpenAI `server_error` string without a 5xx status, and cron will not match it because the classifier returns `timeout` and the retry regex only sees raw 5xx text.

## Solution Assessment

No, not as-is. The PR direction is the right narrow product fix, but the current branch needs a rebase and the classifier shortcut should be gated by explicit or inferred 5xx status after 4xx parsing so permanent auth/billing/format failures do not become retryable.
