## Summary

Keep open: current `main` still has the documented `server_error` cron retry gap, and this PR is useful source work, but the branch is not ready as-is because it can misclassify status-less string 4xx errors and appears stale against the current failover classifier refactor.

## What This Changes

The PR adds a `server_error` failover reason, maps it to status 500, exposes it in cron protocol state, and routes cron retry checks through failover classification before regex fallback.

## Best Possible Solution

Land a rebased fix that carries a distinct `server_error` reason through the current `FailoverClassification` flow while preserving 4xx auth/billing/format precedence and adding focused cron/failover regression coverage.

## Reproduction Assessment

Yes, from source inspection: current `main` documents `server_error` as a cron retry selector, but structured Codex/OpenAI `server_error` payloads classify as `timeout` and cron only matches `server_error` with a raw 5xx regex. I did not run a live failing test because this is a read-only review and `pnpm` is unavailable.

## Solution Assessment

No as currently written. The intended fix direction is right, but the PR should be adapted to current `main` and must not let status-less strings with leading 4xx/billing text be classified as retryable `server_error` solely because the body contains a structured marker.
