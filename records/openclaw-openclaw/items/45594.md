## Summary

Keep open: the PR addresses a documented cron retry bug and includes real terminal proof, but the patch still drops the new `server_error` reason when the structured payload also carries an explicit HTTP 5xx status.

## What This Changes

The PR adds a `server_error` failover reason, wires it into cron retry/state/protocol handling, adds regression coverage, updates the changelog, and includes a small version-test env cleanup.

## Best Possible Solution

Keep the PR open and adjust failover classification so structured `server_error` is preserved for explicit 5xx statuses while explicit 4xx statuses still win as auth, billing, or format failures.

## Reproduction Assessment

Yes, source-reproducible. Current docs/config accept `retryOn: ["server_error"]`, while current main only matches raw 5xx text and classifies the structured Codex/OpenAI payload as `timeout`; the PR's new regression test is the right focused reproduction path.

## Solution Assessment

No, not quite. The fix direction is narrow and maintainable, but the classifier should also preserve `server_error` when the structured payload has an explicit 500/502/504 status instead of letting the existing status branch collapse it back to `timeout`.
