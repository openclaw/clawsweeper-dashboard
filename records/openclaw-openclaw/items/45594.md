## Summary

Keep open: current main still has the documented `cron.retry.retryOn: ["server_error"]` gap, and the PR is useful, but it needs a current-main repair before merge because the structured marker can still shadow string-only 4xx errors and the new failover reason is not aligned with current auth-profile failure policy.

## What This Changes

The PR adds a `server_error` failover reason for structured OpenAI/Codex errors and wires cron retry/state/schema handling so one-shot jobs with `retryOn: ["server_error"]` can retry those payloads.

## Best Possible Solution

Land a rebased narrow fix that treats structured provider `server_error` as a transient, non-auth-health failover reason for cron retry/state/schema, while preserving 4xx/auth/billing precedence and adding regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes, source-level. Current main accepts and documents `retryOn: ["server_error"]`, but cron only retries that key when the raw error contains a `5xx` token, so the provided structured OpenAI/Codex payload without a status code will not retry; I did not execute Vitest in this read-only pass.

## Solution Assessment

No, not as posted. The implementation direction is right, but the PR should be rebased to current main, keep string-only 4xx/auth/billing signals ahead of structured `server_error`, update the extracted auth-profile failure policy, and add the required changelog entry.
