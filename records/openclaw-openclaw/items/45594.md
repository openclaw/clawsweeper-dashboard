## Summary

Keep open. Current main still has the reported cron/failover mismatch, and the PR is directionally useful, but the proposed patch still needs a classifier-ordering repair before merge so parsed 4xx failures cannot become retryable server_error failures.

## What This Changes

The PR adds server_error to failover and cron protocol contracts, maps it to status 500, excludes it from auth-profile failure state, makes cron retry matching consult failover classification, and adds regression coverage for structured OpenAI/Codex server_error payloads.

## Best Possible Solution

Land a rebased or replacement patch that adds server_error as a first-class failover and cron protocol reason, classifies structured OpenAI/Codex server failures only after explicit or parsed 4xx status handling wins, keeps server_error out of auth-profile failure state, and preserves focused cron retry plus protocol conformance coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: configure retryOn: ["server_error"], return a structured Codex/OpenAI server_error string without 5xx digits, and cron will not retry because the classifier returns timeout while cron's server_error retry gate only matches raw 5xx text.

## Solution Assessment

No, not as-is. The PR is the right narrow product direction, but the implementation should be rebased onto current failover classification and must honor parsed 4xx status before the server_error shortcut; it also needs the required changelog entry.
