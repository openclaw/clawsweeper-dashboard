## Summary

Keep open: this is a real documented cron retry bug with sufficient terminal proof and green relevant CI, but the latest head still leaves structured `server_error` payloads with explicit 5xx status classified as `timeout`.

## What This Changes

The PR adds a `server_error` failover reason across agent/runtime/cron protocol surfaces, uses failover classification for cron retry matching, adds regression coverage and a changelog entry, and cleans up one version-test env restore helper.

## Best Possible Solution

Keep the PR open and make the classifier preserve message-derived `server_error` for explicit 5xx statuses while retaining the new 4xx precedence tests and existing status-only fallback behavior.

## Reproduction Assessment

Yes. The original bug is source-reproducible from current main because docs accept `retryOn: ["server_error"]` while current retry matching only sees raw `5xx` text; the remaining PR bug is source-reproducible at head with an error shaped like `{ status: 500, message: <structured server_error payload> }`.

## Solution Assessment

No, not yet. The fix direction is narrow and maintainable, but it should also let structured `server_error` win for explicit 5xx statuses so the new reason is not collapsed back to `timeout`.
