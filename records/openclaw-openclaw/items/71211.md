## Summary

Keep this open. Current main still constructs foreground, background process, and node-host exec tool results from stdout/stderr-derived text before an exec-owned redaction boundary; recent merged redaction work improves UI, transcript, logging, and payment-field coverage but does not close the reported agent/tool-result path.

## What This Changes

The issue asks OpenClaw to redact or gate secret-bearing stdout/stderr before exec tool output is returned to the agent.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary that sanitizes stdout/stderr-derived content and structured details before foreground exec, background process poll/log, node-host exec, middleware handoff, live model context, UI events, and transcript persistence see the result, while keeping existing downstream logging/UI/transcript redaction as defense in depth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: any command that prints an env-style secret can place that text in outcome/session aggregates that are copied into exec result content/details before an exec-owned redaction call.

## Solution Assessment

Yes. Redacting stdout/stderr-derived output before tool result construction, or at a shared normalized tool-result boundary before model context, is the narrow maintainable fix; blocking specific file reads alone would not cover generic secret-bearing command output.
