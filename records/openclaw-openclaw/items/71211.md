## Summary

Keep this open. Current main has useful downstream redaction for UI/tool events and transcript sinks, but the exec-owned tool-result construction paths still assemble stdout/stderr-derived text and details from raw aggregates before a dedicated redaction boundary; the merged related redaction work narrows adjacent logging/UI exposure but does not fully close this issue.

## What This Changes

The issue asks OpenClaw to prevent secret-bearing stdout/stderr from exec-style tools from entering agent/tool-result context without redaction.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary that sanitizes stdout/stderr-derived content and structured details before foreground exec, background process poll/log, node-host exec, approval follow-up summaries, middleware handoff, UI events, and transcript persistence see the result; any raw reveal behavior should be explicit and maintainer-approved.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints `OPENROUTER_API_KEY=sk-or-v1-...` flows through `session.aggregated` and `outcome.aggregated` into foreground exec content/details before an exec-owned redaction call.

## Solution Assessment

Yes, the issue's proposed direction is the right boundary, but a foreground-only patch would be incomplete. The maintainable fix should cover foreground, background process, node-host, follow-up, middleware, event, and transcript paths with one shared redaction helper or contract.
