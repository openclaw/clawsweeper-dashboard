## Summary

Keep open. Current main still passes stdout/stderr-derived aggregates into exec, process, and node-host tool-result content/details before an exec-owned redaction boundary; adjacent UI/logging redaction PRs help other surfaces but do not resolve the agent-facing raw result path.

## What This Changes

The issue asks OpenClaw to redact or otherwise protect secret-bearing stdout/stderr before exec tool results are delivered to the agent.

## Best Possible Solution

Add a shared, tested redaction boundary for exec/process/node-host stdout/stderr-derived result content and details, reusing the existing redaction helpers where possible while keeping UI, transcript, and logging sanitizers as defense in depth.

## Reproduction Assessment

Yes, at source level. Current main copies stdout/stderr-derived aggregate text into exec/process/node-host returned tool-result content and details before any local exec-owned redaction call, and the reporter also confirmed a fake canary on v2026.5.7.

## Solution Assessment

Yes, directionally. Redacting stdout/stderr-derived tool-result content/details before agent delivery is the narrowest durable fix; hard-blocking secret-file reads alone would miss generic command output that contains secrets.
