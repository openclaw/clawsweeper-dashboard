## Summary

Keep this open. Current main has downstream redaction for UI/tool-event and transcript/logging surfaces, but foreground exec, background process output, and node-host exec still construct tool-result content/details from stdout/stderr-derived aggregates before an exec-owned redaction boundary.

## What This Changes

The issue asks OpenClaw to prevent secrets printed by exec stdout/stderr from entering agent tool-result context by redacting, blocking, or requiring explicit opt-in for sensitive command output.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary for stdout/stderr-derived text and structured details before the result is handed back to agent runtimes, with downstream UI/log/transcript redaction kept as defense in depth and tests covering foreground, background, node-host, and harness middleware paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: any command that prints an env-style secret flows through raw outcome/session aggregates into exec result content/details before an exec-owned redaction call.

## Solution Assessment

Yes. Redacting stdout/stderr-derived output before tool-result construction is the narrowest maintainable direction; blocking specific secret-file reads can supplement it but would not cover generic secret-bearing command output.
