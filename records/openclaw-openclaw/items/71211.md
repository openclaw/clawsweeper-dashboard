## Summary

Keep this open. Current main still builds foreground, background, and node-host exec tool results from raw stdout/stderr-derived aggregates before an exec-owned redaction boundary; the related merged redaction work improves UI, transcript, and logging sinks but does not fully resolve the reported agent/tool-result path.

## What This Changes

The issue asks OpenClaw to prevent secret-bearing stdout/stderr from exec-style tools from entering agent/tool-result context without redaction.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary that sanitizes stdout/stderr-derived content and structured details before foreground exec, background process poll/log, node-host exec, middleware handoff, UI events, and transcript persistence see the result, while keeping downstream logging/UI redaction as defense in depth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints an env-style secret flows through outcome/session aggregates into exec result content/details before an exec-owned redaction call.

## Solution Assessment

Yes. Redacting stdout/stderr-derived output before tool result construction is the narrowest maintainable direction; blocking specific file reads could supplement it but would not cover generic secret-bearing command output.
