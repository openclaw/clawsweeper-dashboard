## Summary

Keep this open. Current main still has raw stdout/stderr-derived exec output crossing the exec tool-result boundary before an exec-owned redaction pass; downstream event/transcript sanitizers are useful defense in depth but do not fully resolve the reported raw result construction path.

## What This Changes

The issue requests secret redaction for stdout/stderr-derived exec tool results before those results are returned to agent/runtime context.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary that sanitizes stdout/stderr-derived text and structured details before foreground exec, background process poll/log, node-host exec, and approval follow-up summaries reach agent context, while keeping UI/event/transcript sanitization as defense in depth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: an exec command that prints an env-style secret such as `OPENROUTER_API_KEY=...` flows through `outcome.aggregated` into foreground, process, or node-host tool result content/details before an exec-owned redaction call.

## Solution Assessment

Yes. The issue's proposed direction, redacting stdout/stderr-derived output before tool result construction, is the narrowest maintainable fix; blocking `.env` reads can supplement it but would not protect generic secret-bearing command output.
