## Summary

Keep this open. Current main still builds foreground, background, and node-host exec tool results from raw stdout/stderr-derived aggregates before an exec-owned redaction boundary; downstream UI/event/transcript redaction exists, but it does not fully resolve the internal tool-result exposure described here.

## What This Changes

The issue asks OpenClaw to redact or otherwise guard secret-bearing stdout/stderr before exec and process tool results are returned to agent runtimes.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary for stdout/stderr-derived content and structured details before results are handed back to agent runtimes, then keep the existing UI/event/transcript sanitizers as defense in depth with tests across foreground exec, background process, node-host exec, and embedded event emission.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints an env-style token flows through raw outcome/session aggregates into exec or process result content/details before an exec-owned redaction call.

## Solution Assessment

Yes. Redacting stdout/stderr-derived output before tool-result construction is the narrowest maintainable direction; blocking specific file reads can supplement it but would not cover generic command output that contains secrets.
