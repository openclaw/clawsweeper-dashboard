## Summary

Keep open. Current main still hardcodes the Dream Diary narrative system prompt and prompt builder, and the memory-core schema, resolver, docs, and Dreams UI expose only enablement/cadence/model/status-style controls rather than a user prompt or template contract.

## What This Changes

The issue requests a user-configurable Dream Diary prompt/template under Dreaming config, exposed in Settings/Dreaming UI with placeholder variables such as date, time, count, and theme.

## Best Possible Solution

Add a memory-core-owned Dream Diary prompt/template contract under plugins.entries.memory-core.config.dreaming, preserve the current behavior as the default, support a small documented placeholder set, thread the resolved config through light, REM, and deep narrative generation, expose it in the existing Dreams/settings surface, and update docs plus focused backend/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. The current gap is high-confidence from source inspection because the schema rejects prompt/template keys and the narrative generation path never reads them.

## Solution Assessment

Yes, with scope adjustment. The requested direction is maintainable if implemented as a memory-core config contract rather than a top-level core shortcut, with defaults preserved and placeholders explicitly documented.
