## Summary

Keep open. Current main and the latest release still mark Telegram preview-finalized delivery as complete before all visible follow-up sends have settled, and the open closing PR is unmerged, conflicting, and still needs real Telegram proof.

## What This Changes

The issue requests clearer Telegram partial-stream final delivery observability so operators can distinguish fresh sends, preview finalization, retained previews, hook cancellation, transport failure, and intentional skips.

## Best Possible Solution

Telegram partial-stream finals should only mark delivery complete after the platform-visible preview edit and any suffix sends settle, while hooks/logs distinguish preview finalization from fresh sends and failures.

## Reproduction Assessment

Yes at a source level: current Telegram lane delivery marks a preview-finalized result as delivered before remaining suffix chunks are sent, matching the reported ambiguity between session state and actual Telegram delivery. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, this is not solved on main yet. The best current path is to resolve and review the linked Telegram delivery PR, or replace it with the same delivery-confirmation boundary plus real Telegram proof if maintainers choose a different shape.
