## Summary

This issue should stay open because current main still has source-level delivery-state ambiguity in Telegram partial-stream finals, and the directly linked closing PR is open, conflicting, and not yet live-proven.

## What This Changes

The issue asks for clearer Telegram partial-stream final delivery observability across sent, preview-finalized, preview-retained, canceled, failed, and skipped outcomes.

## Best Possible Solution

Land the linked Telegram fix or a current-main successor that delays delivered-state success until all final chunks settle, emits clear delivery success/failure signals, and preserves the one-preview-message behavior unless maintainers explicitly choose a new strict mode.

## Reproduction Assessment

Yes at source level: current main marks Telegram preview-finalized delivery before all follow-up final chunks settle, and fallback logic then trusts that delivered flag. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes for the linked PR direction, but not yet as a merge artifact. Delaying delivered-state success until all final chunks succeed is the narrow maintainable fix; delivery-mode taxonomy or strict fresh-final mode should remain a maintainer product decision.
