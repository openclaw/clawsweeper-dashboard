- `P2`: The issue affects Telegram delivery debugging and operator confidence, but it is scoped to observability rather than a confirmed crash or total channel outage.
- `impact:message-loss`: The report is about distinguishing delivered Telegram replies from canceled, failed, skipped, or preview-only outcomes when users report missing messages.
- `impact:session-state`: The discussion explicitly describes transcript/session state showing an assistant answer while Telegram delivery state remains ambiguous.

## Summary

Keep open. Current main and the latest release still distinguish Telegram preview outcomes internally, but only `preview-finalized` gets a narrow internal hook/session mirror and the public hook/docs surface has no delivery-mode field for operators to separate sent, preview-finalized, retained, canceled, failed, or skipped outcomes.

## What This Changes

Request clearer operator-visible Telegram partial-stream final-delivery outcomes for sent, preview-finalized, preview-retained, canceled, failed, and skipped replies.

## Best Possible Solution

Define additive Telegram final-delivery outcome metadata for hooks and logs, document it, and preserve one-message preview finalization as the default behavior.

## Reproduction Assessment

Yes, at source level but not with a live Telegram run. Current main’s partial-stream path still returns preview-finalized/preview-retained without a public delivery outcome field, and the latest shipped tag has the same hook shape.

## Solution Assessment

Unclear until maintainers choose the additive contract. The safer direction is outcome metadata in hooks/logs for existing delivery paths, not forcing fresh final messages or adding a strict mode as the first fix.
