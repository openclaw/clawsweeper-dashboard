## Label Justifications

- `P1`: The issue affects real Telegram partial-stream deployments where final replies can be treated as delivered while users may not receive the final text.
- `impact:message-loss`: The reported failure mode is that final Telegram replies can be suppressed, truncated, or indistinguishable from failed delivery.
- `impact:session-state`: The session transcript and delivery state can indicate an assistant final answer even when Telegram delivery semantics remain ambiguous.

## Summary

Keep this open: current main and the latest release still mark Telegram partial-stream finals as delivered before follow-up chunks settle, the fallback gate trusts that delivered state, and the linked closing PR is open, conflicting, and still missing live Telegram proof.

## What This Changes

The issue requests clearer Telegram partial-stream final-delivery observability and delivery-state parity for sent, preview-finalized, preview-retained, canceled, failed, and skipped outcomes.

## Best Possible Solution

Land or replace the Telegram-owned fix so delivered state and operator-visible final-delivery outcome are recorded only after preview edits and follow-up chunks actually succeed, with the strict-mode/config part handled only if maintainers explicitly want that product surface.

## Reproduction Assessment

Yes at source level: current main marks Telegram preview-finalized finals as delivered before follow-up chunks complete, and fallback logic trusts that delivered state. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes for the reliability direction: the fix belongs in the Telegram lane-delivery and dispatch boundary, not in generic session state. The logs/events and optional strict mode still need maintainer product judgment before being folded into the final scope.
