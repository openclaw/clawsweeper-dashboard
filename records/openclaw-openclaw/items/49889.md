## What This Changes

The issue requests clearer Telegram partial-stream final delivery observability so operators can distinguish fresh sends, preview finalization, retained previews, cancellation, transport failure, and intentional skips.

## Best Possible Solution

Land the linked Telegram fix or a current-main successor that delays delivered-state success until all final chunks settle, emits clear delivery success/failure signals, and leaves strict fresh-final behavior behind explicit maintainer product approval.

## Reproduction Assessment

Yes at source level: current main marks Telegram preview-finalized delivery before all follow-up final chunks settle, and fallback logic then trusts that delivered flag. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes for the linked PR direction, but not yet as a merge artifact. Delaying delivered-state success until all final chunks succeed is the narrow maintainable fix; delivery-mode taxonomy or strict fresh-final mode should remain a maintainer product decision.
