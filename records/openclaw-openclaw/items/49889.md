- `P2`: The issue affects Telegram operator debugging and delivery confidence for partial streaming, but it is scoped to observability rather than a confirmed crash or total channel outage.
- `impact:message-loss`: The report is about distinguishing successful Telegram delivery from canceled, failed, skipped, or preview-only outcomes when users say messages did not arrive.
- `impact:session-state`: The discussion specifically calls out transcript state showing an assistant answer while Telegram delivery state is ambiguous.

## Summary

Keep open. Current main has a shipped partial fix for finalized previews, but the broader operator-visible outcome gap remains: preview-finalized only emits an internal hook, preview-retained is still collapsed into delivered state, and public hook/docs have no delivery outcome field.

## What This Changes

Request clearer Telegram partial-stream final-reply outcome observability for sent, preview-finalized, preview-retained, hook-canceled, transport-failed, and skipped deliveries.

## Best Possible Solution

Add additive Telegram final-delivery outcome metadata to hooks and logs while preserving in-place preview finalization as the default behavior.

## Reproduction Assessment

No live high-confidence Telegram run was performed. Source inspection is high-confidence: current main still lacks a public delivery outcome field and only mirrors preview-finalized through the internal hook path.

## Solution Assessment

Unclear until maintainers choose the additive contract. The safer direction is outcome metadata in hooks/logs for existing delivery paths, not forcing fresh final messages or adding a strict mode as the first fix.
