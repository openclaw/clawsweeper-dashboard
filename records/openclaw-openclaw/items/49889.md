## Summary

This issue is still necessary: current main and the latest release still have Telegram partial-stream final paths that can mark delivery before all final follow-up chunks settle, and the linked repair PR is open, conflicting, and not yet proven live.

## What This Changes

The issue requests clear Telegram partial-stream final delivery observability for sent, preview-finalized, preview-retained, canceled, failed, and skipped outcomes.

## Best Possible Solution

Keep this issue open while maintainers review or replace the linked Telegram PR, then land a Telegram-plugin fix that delays delivered state until final delivery really completes and exposes enough outcome detail for operators without adding a new config mode by default.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 mark Telegram preview-finalized delivery before final follow-up chunks settle, and fallback logic then trusts delivered state. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Unclear as a final product shape: delaying delivered state is the right narrow bug-fix direction, but adding delivery-mode logs, hook fields, session confirmation, or a strict final-send mode needs maintainer direction.
