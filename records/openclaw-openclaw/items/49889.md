## Summary

Keep open: current main and the latest release still have source-level evidence that Telegram partial-stream final delivery can be marked delivered before follow-up chunks finish, and the issue has an open linked fixing PR that has not merged yet.

## What This Changes

The issue asks for Telegram partial-stream final replies to expose clear delivery outcome observability across sent, preview-finalized or retained, canceled, failed, and skipped paths.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84563 or a current-main successor that delays delivered-state until Telegram final chunks actually settle, then decide any extra delivery-mode logging or strict-mode surface as a maintainer-scoped follow-up.

## Reproduction Assessment

Yes at source level: current main marks Telegram partial-stream finals delivered before long-final follow-up chunks finish, and the linked PR plus June production logs corroborate the same failure mode. I did not run a live Telegram reproduction in this read-only issue pass.

## Solution Assessment

Unclear as a final product shape: delaying delivered-state until all final chunks settle is the right narrow fix direction, but maintainers still need to decide whether explicit outcome logs/events or a strict fresh-send mode are part of this issue or a follow-up.
