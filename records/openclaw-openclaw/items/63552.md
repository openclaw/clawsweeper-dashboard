## Summary

Keep this PR open. Current main has some newer scoped-discovery plumbing, but it still does not implement the PR's central explicit-provider optimization for ordinary `models.json` generation: non-live discovery only scopes when a caller supplies `providerDiscoveryProviderIds`, and the production filter path still does not use the already computed `explicitProviders`. The overlapping #72352 is also open and not a merged or selected replacement, so it is not a safe superseding close reason.

## What This Changes

The PR adds an explicit-provider discovery filter to models-config provider discovery and regression tests for non-live explicit-provider scoping, unknown-owner fallback, and live `all`/mixed-filter behavior.

## Best Possible Solution

Adapt one focused implementation onto current main that scopes ordinary non-live `models.json` provider discovery from configured explicit providers, reuses the existing provider-owner/metadata seams, preserves live `all` and mixed live-filter semantics, and keeps separate replace-mode semantics tracked by #66957 unless the chosen patch explicitly covers them.

## Reproduction Assessment

Yes. The static current-main path is enough to reproduce the remaining gap: call `ensureOpenClawModelsJson(cfg)` with `cfg.models.providers` set and no `providerDiscoveryProviderIds`; `resolveImplicitProviders` reaches the filter without the explicit provider set, and non-live discovery remains broad.

## Solution Assessment

Yes for the explicit-provider slice, with a rebase caveat. The narrow owner-filter fallback is maintainable, but the final version should be reconciled with current main's newer `providerDiscoveryProviderIds`, metadata-owner helpers, and #72352 overlap rather than landing the stale branch verbatim.
