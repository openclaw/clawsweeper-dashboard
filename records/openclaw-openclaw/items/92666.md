## Summary

Keep open. Current main and v2026.6.6 still have requestId-only device-pair approvals and Telegram-only text notifications, while this request needs a broader origin-aware, channel-bound approval contract for security-sensitive scope upgrades.

## What This Changes

Request channel-origin metadata on device scope-upgrade requests and channel-specific approval cards that return to the triggering chat session.

## Best Possible Solution

Design a generic, replay-safe device-pair approval origin/callback contract, keep pending device pairing as the durable source of truth, and let channel plugins render/authorize native approval cards through that contract.

## Reproduction Assessment

Yes, source-level. Current main shows pending device requests lack channel-origin metadata and approval remains requestId/manual-command based; I did not run a live QQ, Feishu, or WebChat setup.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix first needs a generic security/API contract for origin-bound device-pair approvals rather than a channel-specific shortcut.
