## Summary

Keep open. Current main and v2026.6.8 still use requestId/manual-command device scope-upgrade approval, while the requested channel-origin approval card path is security-sensitive and needs maintainer product/API design.

## What This Changes

The issue requests channel-origin metadata and in-channel approval cards for device scope upgrade approvals triggered from chat sessions.

## Best Possible Solution

Define a generic, replay-safe device-pair approval delivery contract that can reuse existing channel approval SDK patterns, keep pending device pairing as the source of truth, and then add channel-specific renderers behind that contract.

## Reproduction Assessment

Yes, at source level: current pending requests and protocol events have no channel-origin/callback-routing fields, and the current approval path remains requestId/manual-command based. I did not run a live QQ, Feishu, Discord, Slack, or WebChat scenario.

## Solution Assessment

Unclear as a ready implementation: the proposed UX is plausible, but the best fix should first define a generic approval-origin/callback contract instead of adding one-off channel-specific gateway behavior.
