## Summary

Keep open. Current main and v2026.6.8 still have requestId-only device scope-upgrade approval with no channel-origin metadata or approval-card callback path, and related items cover narrower inline-button or fallback-guidance surfaces rather than superseding this security-sensitive design work.

## What This Changes

The issue requests channel-aware in-chat approval cards for device scope-upgrade requests that currently require terminal/control-plane approval.

## Best Possible Solution

Define a generic, replay-safe device-pair approval origin/callback contract, keep pending device pairing as the source of truth, and implement channel-specific renderers behind that contract.

## Reproduction Assessment

Yes, as a source-level missing capability: current main shows pending device requests do not carry channel origin metadata and approval remains requestId/manual-command based. I did not run a live QQ, Feishu, WebChat, Discord, or Slack setup.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix must first define a generic approval origin/callback contract that keeps channel-specific rendering out of core policy and preserves the existing authorization boundary.
