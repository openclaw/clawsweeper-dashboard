## Summary

Keep open. Current main and v2026.6.8 still use requestId/manual device-scope approval without channel-origin metadata or channel-bound approval cards, and the related open items only cover narrower Telegram pairing buttons or Feishu fallback clarity.

## What This Changes

The issue requests channel-origin metadata and in-channel approval cards for device scope upgrade requests triggered from chat sessions.

## Best Possible Solution

Define a generic, replay-safe device-pair approval origin/callback contract, keep pending device pairing as the source of truth, and let channel plugins render cards through the existing approval capability model where it fits.

## Reproduction Assessment

Yes, at source level: current pending records and protocol events still have no channel origin metadata, and current docs/handlers still route scope-upgrade approval through requestId/manual device approval. I did not run a live QQ, Feishu, Discord, Slack, or WebChat setup.

## Solution Assessment

Unclear. Channel-aware approval is a plausible direction, but the best fix should first define a generic replay-safe approval origin/callback contract instead of adding one-off channel policy in core.
