## Summary

Keep open. Current main and v2026.6.9 still use requestId-only device scope-upgrade approval with no channel-origin metadata or approval-card callback path, and the related issues cover narrower WebChat, Telegram, or Feishu fallback surfaces rather than replacing this security-sensitive design item.

## What This Changes

The issue requests channel-origin metadata and in-channel approval cards for pending device scope-upgrade approvals triggered from chat sessions.

## Best Possible Solution

Define a generic, replay-safe device scope-upgrade approval origin/callback contract, keep device pairing state as the source of truth, and let channel plugins render approval cards behind that contract after maintainer security/API direction.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.9 create requestId-only pending device scope-upgrade approvals with no channel origin metadata or approval-card dispatch path; I did not run a live QQ, Feishu, WebChat, Discord, or Slack setup.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix must first define a generic approval origin/callback contract that keeps channel-specific rendering out of core policy and preserves existing approval authorization checks.
