## Summary

Keep open: current main and v2026.6.6 still model device scope upgrades as pending device-pairing records approved through the Gateway/CLI, with no persisted channel-origin envelope or in-channel approval-card callback. The request is useful, but it crosses operator authorization, persisted device state, Gateway protocol, channel callbacks, and plugin boundaries, so it needs maintainer product/API design rather than conservative cleanup closure.

## What This Changes

The issue requests channel-origin metadata and in-channel approval cards for device scope upgrade requests triggered from chat sessions.

## Best Possible Solution

Design a generic origin-aware device approval flow that keeps `pending.json`/device pairing as the authorization source of truth, uses typed presentation/actions for channel UI, preserves `operator.admin`/owner checks, and adds focused Gateway, plugin, channel, docs, and upgrade tests after the boundary is agreed.

## Reproduction Assessment

Yes, from source and docs rather than a live chat setup: current main creates pending scope-upgrade records and documents terminal `openclaw devices approve <requestId>`, while the pending schema and approval RPC lack channel-origin or in-channel callback fields. This is a feature gap, not a broken documented in-channel approval contract.

## Solution Assessment

No, not as the issue's concrete per-channel implementation plan. The desired user outcome is valid, but the maintainable fix should first choose a generic core/plugin approval seam and reuse typed channel presentation/action contracts instead of wiring product-specific approval cards separately into each channel.
