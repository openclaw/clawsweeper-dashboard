## Summary

Keep open: current main and v2026.6.6 still lack channel-origin metadata and channel-bound approval cards for device scope upgrades, while https://github.com/openclaw/openclaw/issues/87283 was closed as a duplicate of this broader remaining design and https://github.com/openclaw/openclaw/issues/45839 remains narrower Telegram pairing-button work.

## What This Changes

The issue requests channel-origin metadata and in-channel approval cards for device scope upgrade requests so users can approve from the triggering chat instead of SSH/control-plane CLI.

## Best Possible Solution

Design a generic, replay-safe device-pair approval origin/callback contract, keep pending device pairing as the source of truth, reuse typed native approval delivery where possible, and implement channel-specific renderers behind that contract.

## Reproduction Assessment

Yes, at source level: current main creates request-id-based pending device scope upgrades, but the pending/event schemas do not carry channel origin metadata and the approval path remains manual/requestId based. I did not run a live QQ, Feishu, Discord, Slack, or WebChat setup.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix should be a generic approval origin/callback contract with explicit auth, expiry, and replay semantics rather than ad hoc per-channel cards wired directly into device pairing.
