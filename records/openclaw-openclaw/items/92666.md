## Summary

Keep open. Current main still lacks channel-origin metadata and channel-bound approval cards for device scope upgrades, and related open issues cover narrower Telegram buttons or browser approval UX rather than this broader cross-channel device-pair approval flow.

## What This Changes

The issue requests storing the triggering chat context on device scope-upgrade requests and sending an in-channel approval card that resolves the pending device-pair approval.

## Best Possible Solution

Design a generic, replay-safe device-pair approval origin/callback contract, keep pending device pairing as the source of truth, and implement channel-specific approval renderers behind that contract.

## Reproduction Assessment

Yes, for the missing capability: current source shows pending requests do not carry channel origin metadata and device-pair approval remains requestId/manual-command based. I did not run a live QQ, Feishu, or WebChat setup.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix must first define a generic approval origin/callback contract and keep channel-specific rendering out of core policy.
