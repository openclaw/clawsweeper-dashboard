## Summary

Keep open: current main still has the reported response-vs-disk mismatch, and two open PRs already reference this issue as their closing target.

## What This Changes

The issue requests that `config.set`, `config.patch`, and `config.apply` return the canonical config persisted to disk instead of the pre-write in-memory config.

## Best Possible Solution

Thread the `persistedConfig` returned by `writeConfigFile` through `replaceConfigFile` and `commitGatewayConfigWrite`, return that redacted value from all three gateway write RPCs, and land one of the linked PRs with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the write RPC handlers respond with `parsed.config` or `validated.config` while `writeConfigFile` can return a transformed `persistedConfig`.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate the existing persisted write result through the shared config write abstraction, rather than patching each handler independently or adding a new integrity field first.
