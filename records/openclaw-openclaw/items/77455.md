## Summary

Keep open: current main still has the reported write-response mismatch. The config mutation handlers respond with the pre-write validated config while the lower write path can transform, stamp, and preserve source-authored values before persisting.

## What This Changes

The issue requests that `config.set`, `config.patch`, and `config.apply` responses return the canonical persisted config state rather than the in-memory validated input.

## Best Possible Solution

Return a redacted canonical post-write config from the shared config write abstraction, preferably with regression coverage that proves write RPC responses match the persisted/readback state for transformed writes.

## Reproduction Assessment

Yes. Current main source shows `config.patch` responds with `validated.config` after the write, while the write path can transform and stamp a different persisted payload before disk commit.

## Solution Assessment

Yes. The issue's suggested direction is the right boundary if implemented through `commitGatewayConfigWrite`/`replaceConfigFile` so all write RPCs share the canonical persisted response instead of patching handlers independently.

Prefer updating the shared write abstraction (`commitGatewayConfigWrite` / `replaceConfigFile`) so the normal root writer and `$include` fast path both expose the persisted or refreshed source config plus path/hash as needed.
