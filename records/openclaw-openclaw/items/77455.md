## What This Changes

The issue requests that config.set, config.apply, and config.patch return the canonical persisted config state rather than the pre-write in-memory config.

## Best Possible Solution

Land one canonical persisted-response fix that carries persistedConfig from writeConfigFile through replaceConfigFile and commitGatewayConfigWrite into redacted write responses, with regression coverage for response/readback equality.

## Reproduction Assessment

Yes, source-reproducible. Current main config.set, config.patch, and config.apply write through a path that drops the persisted write result and then responds with parsed.config or validated.config, while the lower writer can return a different stamped persistedConfig.

## Solution Assessment

Yes. The narrow maintainable fix is to return the persisted write result through the shared config write abstractions and use that value in all write RPC responses; the active PR cluster is already pursuing that direction.
