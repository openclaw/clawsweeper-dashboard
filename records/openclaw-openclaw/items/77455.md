## Summary

Keep open: current main still returns the submitted/validated config from `config.set`, `config.patch`, and `config.apply` after writing, while the lower writer can persist a transformed disk payload. The issue already has open closing PRs #77462 and #77486, so the right cleanup is to review and land one canonical fix rather than close this issue now.

## What This Changes

Fix config write RPC responses so their `config` payload matches the canonical persisted disk state after write-time transformations.

## Best Possible Solution

A single merged fix should return a redacted canonical post-write config from the shared write path and add regression coverage for response-disk equality across `config.set`, `config.patch`, and `config.apply`.

## Reproduction Assessment

Yes, by source inspection on current main: the handlers deterministically respond with pre-write config while `writeConfigFile` can persist a transformed payload. I did not run a live RPC repro in this read-only review, so the structured status is `source_reproducible` rather than `reproduced`.

## Solution Assessment

Yes. Threading the writer's persisted result through `replaceConfigFile` and `commitGatewayConfigWrite` is the narrow maintainable boundary, and one of the open closing PRs should become the canonical implementation.
