## Best Possible Solution

Route TTS through `routeReply` using the same active source target semantics as sibling delivery-aware tools, cover the `currentChannelId`-present and `agentTo`-absent case, then land with focused tests plus maintainer-accepted WhatsApp proof expectations.

## Reproduction Assessment

No live WhatsApp reproduction was run, but the failure mode is source-reproducible: ordinary source-channel runs pass the destination through `currentChannelId`, while the PR's inline TTS send only checks `agentTo`.

## Solution Assessment

No: routing through `routeReply` is the right boundary, but this implementation is not yet the best fix because it misses the established `currentChannelId ?? agentTo` target semantics used by sibling delivery paths.
