## Summary

Keep open. Current main still has the documented full-backup path that includes workspaces by default, and there is no merged fix or canonical duplicate for the Windows long-path failure; the exact root cause still needs Windows proof because current code uses npm `tar`, not Windows `tar.exe`.

## What This Changes

The issue reports that full `openclaw backup create` fails or hangs on Windows with long workspace paths while `--only-config` succeeds.

## Best Possible Solution

Reproduce on Windows current main with a long-path workspace, then fix the archive path handling or diagnostics in `backup create` while keeping backup scope consistent with the documented contract unless owners choose an exclude-policy change.

## Reproduction Assessment

No. Source inspection confirms the full-backup versus config-only path split, but I did not establish a Windows current-main failure with a long workspace path, and the stated BSD tar cause conflicts with current npm `tar` usage.

## Solution Assessment

Unclear. Replacing the backend with 7-Zip or adding broad workspace exclusions is too wide until the npm tar failure is reproduced; a narrower path-handling or diagnostic fix may be better.
