## Summary

Keep open. The report is recent and describes a documented backup flow failing on Windows; current main still backs up workspaces by default, while the reported BSD tar root cause is not proven because the current implementation uses the npm `tar` library rather than shelling out to Windows `tar.exe`.

## What This Changes

The issue reports that `openclaw backup create` fails or hangs on Windows full backups with long workspace paths while `--only-config` succeeds.

## Best Possible Solution

Reproduce on Windows current main with a workspace path longer than 260 characters, then fix the existing backup archiver or diagnostics without changing backup payload scope unless maintainers choose that policy.

## Reproduction Assessment

No high-confidence current-main reproduction is available. Source inspection confirms the full-backup versus config-only path split, but it does not prove the Windows long-path hang or exit 255.

## Solution Assessment

Unclear. Replacing the backend with 7-Zip or adding broad exclusions is not proven to be the best fix until the current npm `tar` Windows failure is reproduced and scoped.
