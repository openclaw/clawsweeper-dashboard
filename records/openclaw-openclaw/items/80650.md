## Summary

Keep open. The report describes a documented backup flow failing on Windows, current main still includes workspace directories in full backups, and no current-main Windows long-path reproduction or merged fix was found; the stated BSD tar root cause is also not proven because the current implementation uses the npm `tar` library rather than shelling out to Windows `tar.exe`.

## What This Changes

The issue requests that full `openclaw backup create` succeed on Windows when workspace files include paths longer than 260 characters.

## Best Possible Solution

Reproduce on Windows current main with a workspace path longer than 260 characters, then fix the existing backup archiver or diagnostics while preserving documented workspace backup scope unless maintainers deliberately change that policy.

## Reproduction Assessment

No high-confidence current-main reproduction is available. Source inspection confirms the same full-backup versus `--only-config` path split, but the Windows long-path hang or exit 255 has not been reproduced and the BSD tar explanation does not match current node-tar usage.

## Solution Assessment

Unclear. Replacing the backend with 7-Zip or adding broad exclusions is not yet the best proven fix; a Windows repro should first identify whether the narrow repair belongs in path handling, tar error reporting, retry behavior, or documented backup scope.
