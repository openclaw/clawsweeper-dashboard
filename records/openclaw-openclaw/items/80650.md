## Summary

Keep open. The report is recent and describes a documented backup flow failing on Windows full backups; current main still backs up workspace paths by default, and the reported BSD tar root cause is not proven because OpenClaw uses the npm `tar` library rather than shelling out to Windows `tar.exe`.

## What This Changes

Reports `openclaw backup create` failing on Windows full backups with long workspace paths while `--only-config` succeeds.

## Best Possible Solution

Reproduce on Windows current main with a long workspace path, identify the actual failing layer, then fix the existing archiver or error reporting without changing backup scope unless maintainers choose a documented exclude/backend policy.

## Reproduction Assessment

No. The source confirms the full-backup versus config-only path split, but there is no high-confidence Windows current-main reproduction and the reported BSD tar cause conflicts with current node-tar usage.

## Solution Assessment

Unclear. Replacing the backend with 7-Zip or adding broad exclusions may be useful, but a narrower node-tar Windows path-handling or diagnostic fix may be better once the failure is reproduced.
