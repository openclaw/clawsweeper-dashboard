## Summary

Keep open. The report is recent and describes a documented backup flow failing on Windows; current main still includes workspace directories by default, and the tests do not prove that long Windows workspace paths are handled. The BSD tar diagnosis conflicts with current source because OpenClaw uses the npm `tar` library, so this needs Windows repro and root-cause confirmation rather than cleanup closure.

## What This Changes

Report asks for `openclaw backup create` to succeed on Windows when backing up workspaces containing long paths instead of hanging or exiting 255.

## Best Possible Solution

Reproduce on Windows current main with a workspace path longer than 260 characters, then fix the backup archiver or error reporting in the existing backup path without changing workspace payload scope unless maintainers explicitly choose that policy.

## Reproduction Assessment

No high-confidence current-main reproduction is available. Source inspection confirms the full-backup versus config-only path split, but the Windows long-path hang/exit was not reproduced and the stated BSD tar cause conflicts with current node-tar usage.

## Solution Assessment

Unclear. Replacing the backend with 7-Zip or adding broad exclusions is not clearly the best fix until the current node-tar Windows failure is reproduced; a narrower path-handling or diagnostic fix may be safer.
