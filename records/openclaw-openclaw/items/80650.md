## Summary

Keep open. The report is recent and describes a documented full-backup flow failing on Windows, while current main still includes workspace directories by default; however, the claimed BSD tar root cause conflicts with current source because OpenClaw uses the npm `tar` library rather than shelling out to Windows `tar.exe`.

## What This Changes

The issue reports that full `openclaw backup create` fails or hangs on Windows with long workspace paths while `--only-config` succeeds.

## Best Possible Solution

Reproduce on Windows current main with a workspace path longer than 260 characters, then fix the existing node-tar backup path or diagnostics without changing backup payload scope unless maintainers choose that policy.

## Reproduction Assessment

No, we do not have a high-confidence current-main reproduction yet. Source inspection confirms the full-backup versus config-only path split, but the Windows long-path hang/exit was not reproduced and the stated BSD tar cause conflicts with the current node-tar implementation.

## Solution Assessment

Unclear. Replacing the backend with 7-Zip or adding broad exclusions is not clearly the best fix until the current node-tar Windows failure is reproduced; a narrower path-handling or diagnostic fix may be safer.
