## Summary

Keep open. The report describes a documented backup flow failing on Windows, and current `main` still archives workspace directories by default; however, the stated BSD `tar.exe` root cause does not match current source, which uses the npm `tar` package, so the remaining work is Windows reproduction and root-cause confirmation rather than cleanup closure.

## What This Changes

The issue reports that full `openclaw backup create` fails or hangs on Windows with long workspace paths while `--only-config` succeeds.

## Best Possible Solution

Reproduce on Windows 10/11 against current `main` or the latest release with a workspace path over 260 characters, then fix the existing backup archiver or diagnostics without changing workspace payload scope unless maintainers choose that policy.

## Reproduction Assessment

No high-confidence current-main reproduction is available. Source inspection confirms the full-backup versus config-only path split and the node-tar archive path, but the Windows long-path hang or exit 255 was not reproduced.

## Solution Assessment

Unclear. A 7-Zip backend, broad exclusions, or new exclude flags may be useful, but they are not proven to be the narrowest fix until the current node-tar Windows failure is reproduced.
