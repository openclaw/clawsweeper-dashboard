## Summary

Keep open. This is a recent, concrete report against documented backup behavior; current main still archives workspace directories by default, and the source/tests do not prove that long Windows workspace paths are handled. The BSD tar diagnosis conflicts with current source because backup uses the npm `tar` library rather than shelling out to Windows `tar.exe`, so this needs Windows reproduction before choosing a fix.

## What This Changes

The issue asks for `openclaw backup create` to succeed on Windows when full workspace backup includes paths longer than 260 characters.

## Best Possible Solution

Reproduce current main on Windows with a workspace path longer than 260 characters, then fix the backup archiver or diagnostics while preserving the documented workspace backup scope unless maintainers choose an exclusion/backend policy.

## Reproduction Assessment

No. The issue gives concrete Windows steps, but I did not establish the failure on current main; source only confirms that full backup includes workspace paths while `--only-config` skips them.

## Solution Assessment

Unclear. Replacing the backend with 7-Zip or adding broad exclusions is not proven best while current source uses npm `tar`; the safer path is a targeted Windows repro followed by a narrow archiver or error-reporting fix.
