## Summary

Keep open. Current main still treats full backups as including workspace directories by default and only skips installed plugin dependency trees under the state extensions directory, while the reported Windows long-path failure has not been reproduced or fixed on main. The report's BSD tar root-cause does not match current source, which uses the npm `tar` library, so maintainers need Windows proof before choosing an archive-backend or backup-scope change.

## What This Changes

The issue reports that `openclaw backup create` fails on Windows full backups when workspace paths exceed 260 characters, while `--only-config` succeeds.

## Best Possible Solution

Keep the issue open until maintainers get a Windows long-path reproduction against current main and choose a narrow backup behavior, likely within the existing `backup-create` pipeline rather than an unconditional external 7-Zip dependency.

## Reproduction Assessment

No high-confidence current-main Windows reproduction is available. Source inspection shows full backup still walks workspace directories by default and does not skip arbitrary workspace `node_modules`, but the exact long-path failure needs a real Windows run.

## Solution Assessment

Unclear as proposed. Replacing the archive path with 7-Zip would add an external dependency; the better first step is to prove the current npm `tar` failure on Windows and then decide whether to fix long-path handling, default scope, or documented exclusions.
