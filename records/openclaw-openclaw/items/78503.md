## Summary

Keep this PR open: it addresses a valid status-command performance bug and I did not find a concrete blocking code defect, but the default JSON output intentionally becomes leaner, so maintainers should accept that compatibility tradeoff before merge.

## What This Changes

The PR adds status-scan policy switches so default `status --json` skips update checks, task summaries, memory inspection, and local status-RPC fallback work while `--all` keeps the deeper diagnostics.

## Best Possible Solution

Land the focused gating after maintainer acceptance of the default JSON health-probe contract, keeping update/task/memory/RPC diagnostics under `--all` with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main still starts update/gateway work and task registry summaries on the default JSON path outside cold-start skipping. I did not run a timing repro in this read-only review.

## Solution Assessment

Mostly yes: using the existing `--all` boundary is the narrow maintainable fix for keeping default JSON status useful as a health probe. The main alternative would be preserving full diagnostics by default and adding an explicit lean mode, but that would not solve the reported default health-probe latency.
