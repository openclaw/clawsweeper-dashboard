## Summary

Keep open for maintainer review: the patch is a focused status CLI performance fix with sufficient terminal proof and no blocking code findings, but it changes the default JSON diagnostics contract, so maintainers should explicitly accept the compatibility tradeoff.

## What This Changes

The PR moves update checks, task summaries, memory inspection, and local status-RPC fallback for `status --json` behind `--all`, caps the default gateway probe, and extracts a cron timeout helper/test.

## Best Possible Solution

Land the focused gating if maintainers accept `status --json` as a lean health probe, with full diagnostics remaining available through `status --json --all`.

## Reproduction Assessment

Yes, at source level: current main still runs update checks, task registry summary inspection, and local status-RPC fallback work on default JSON outside cold-start skipping. I did not rerun the Pi-class timing repro in this read-only review.

## Solution Assessment

Yes, mostly: using the existing `--all` boundary is the narrow maintainable way to make default JSON status usable as a health probe. The remaining question is maintainer acceptance of the compatibility tradeoff for existing JSON consumers.
