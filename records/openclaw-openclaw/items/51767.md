## Summary

Keep open. Current main and the latest shipped tag still implement `cron.sessionRetention` as cron-run-only, while the generic `session.maintenance` path is a broader age/count/disk-budget cleanup and not the requested lifecycle-aware subagent/ACP retention policy. The direct fix PR was closed unmerged and related reports have been closed into this tracker, so this remains the canonical item.

## What This Changes

Request a restart-safe retention policy so completed subagent and ACP session rows are pruned instead of accumulating in `sessions.json` outside cron-run retention.

## Best Possible Solution

Keep this as the canonical tracker for a documented, lifecycle-aware, restart-safe retention policy that prunes completed subagent and ACP session-store rows while preserving active, parent-owned, thread-bound, and intentionally persistent sessions.

## Reproduction Assessment

Yes for source-level reproduction: current main still has a cron-key-only retention guard, default subagent cleanup is `keep`, and keep/session-mode subagent sweeps do not delete session-store rows. I did not live-reproduce RSS growth in this read-only review.

## Solution Assessment

No, the issue's simple guard expansion is not clearly the best standalone fix because cron runs, keep-mode subagents, session-mode subagents, and ACP tasks have different lifecycle rules. A durable lifecycle-aware sweep or new documented retention policy is safer.
