## Summary

Keep open. Current main and v2026.5.7 still leave `cron.sessionRetention` cron-only, while the newer generic `session.maintenance` path is a different age/count/disk-budget cleanup rather than a terminal-state subagent/ACP retention policy. The direct fix PR was closed unmerged and the related reports are closed into this tracker, so this remains the canonical item.

## What This Changes

Request that completed subagent and ACP session entries be covered by restart-safe retention instead of leaving `cron.sessionRetention` limited to cron-run sessions.

## Best Possible Solution

Keep this as the canonical tracker for a lifecycle-aware, restart-safe retention design that prunes completed subagent and ACP session-store rows while preserving active, parent-owned, and intentionally persistent sessions.

## Reproduction Assessment

Yes for source-level reproduction: current main still has a cron-key-only retention guard, default subagent cleanup is `keep`, and keep/session-mode subagent sweeps do not delete session-store rows. I did not live-reproduce RSS growth in this read-only review.

## Solution Assessment

No, the issue's simple guard expansion is not clearly the best standalone fix because cron, subagent keep/session modes, and ACP task ownership have different lifecycle rules. The safer solution is a durable lifecycle-aware sweep or documented policy that has explicit terminal-state, transcript, and persistence semantics.
