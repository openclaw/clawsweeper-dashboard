## Summary

Keep open. Current main still leaves `cron.sessionRetention` scoped to cron-run keys, while completed keep-mode subagent sessions and ACP/session cleanup follow separate lifecycle paths; the closed direct PR and duplicate reports make this the canonical tracker rather than a resolved item.

## What This Changes

Apply retention cleanup to completed subagent and ACP session-store entries instead of only cron-run sessions.

## Best Possible Solution

Keep this as the canonical tracker for a lifecycle-aware, restart-safe retention path that prunes completed subagent and ACP session-store rows with explicit terminal-state and transcript rules.

## Reproduction Assessment

Yes for source-level reproduction: current main's cron reaper still has a cron-key-only guard, default subagent cleanup keeps session rows, and the delete path only runs for archive-timed delete-mode records. I did not live-reproduce RSS growth because this review was read-only.

## Solution Assessment

No, the simple guard expansion suggested in the issue is not clearly the safest fix because cron runs, keep/session-mode subagents, and ACP task sessions have different lifecycle ownership. The maintainable path is a durable retention sweep with terminal-state checks and regression coverage.
