## Summary

Keep open. Current main still scopes `cron.sessionRetention` to cron run keys, while default native subagent runs are kept in the session store and ACP cleanup is handled through narrower task-maintenance rules rather than an equivalent durable short-retention sweep. Related reports have been folded into this tracker and the direct fix PR was closed unmerged, so this remains the canonical item.

## What This Changes

Request durable retention cleanup for completed subagent and ACP session-store entries and transcripts, rather than pruning only isolated cron run sessions.

## Best Possible Solution

Keep this as the canonical tracker and design a lifecycle-aware, restart-safe retention path for completed native subagent and ACP sessions. The end state should preserve active and intentionally persistent sessions, use existing session deletion and transcript archival helpers, cover startup/store-write catch-up, and document the chosen retention knob.

## Reproduction Assessment

Yes for the code-path reproduction: current main still has a cron reaper guard that only accepts `isCronRunSessionKey`, and current subagent tests explicitly keep default keep-mode runs off the delete sweep path. I did not live-reproduce the reported RSS growth, but the cleanup exclusion is directly visible in source and tests.

## Solution Assessment

Unclear/no for the suggested one-line guard expansion: it would address the narrow exclusion but is not clearly the safest product fix across keep-mode subagents, thread-bound sessions, parent-owned ACP sessions, and transcript cleanup. A durable lifecycle-aware sweep with targeted tests is the safer direction.
