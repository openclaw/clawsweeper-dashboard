## Summary

Keep this open. Current main still makes `cron.sessionRetention` a cron-run-only reaper, while extending retention to completed subagent/ACP sessions would change public cleanup semantics because subagent `cleanup` defaults to `keep` and the docs frame subagent auto-archive as a separate best-effort lifecycle.

## What This Changes

Apply persistent retention cleanup to completed subagent and ACP sessions rather than only pruning isolated cron-run sessions.

## Best Possible Solution

Define an explicit retention policy for completed ephemeral subagent/ACP sessions, likely through session maintenance rules or a dedicated subagent/ACP retention setting, then implement a restart-safe sweep with transcript handling and docs/tests aligned.

## Reproduction Assessment

Yes for the narrow reported behavior: current source shows `cron.sessionRetention` only visits cron-run keys, and current tests do not cover pruning plain subagent/ACP keys in that reaper. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear. Extending the existing reaper is mechanically small, but repurposing `cron.sessionRetention` for subagent/ACP cleanup is not clearly the best public contract given current docs and `cleanup: "keep"` semantics.
