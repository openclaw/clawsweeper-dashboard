## Summary

Keep open. Current main and the latest release still have a cron-only retention sweep, and the related duplicate reports were closed back to this issue as the canonical tracker while the matching fix PR was closed unmerged. The remaining work needs maintainer policy on whether to extend `cron.sessionRetention` beyond cron or add a dedicated subagent/ACP retention surface.

## What This Changes

The issue requests restart-safe retention cleanup for completed subagent and ACP sessions instead of sweeping only cron-run session rows.

## Best Possible Solution

Keep this issue as the canonical tracker and design one lifecycle-aware, restart-safe retention policy for completed subagent and ACP session rows, with schema/docs alignment and tests that preserve active, persistent, thread-bound, and main sessions.

## Reproduction Assessment

Yes, at source level: an expired `agent:*:subagent:*` or ACP session-store key reaches `sweepCronRunSessions`, then the current `isCronRunSessionKey` guard skips it before deletion. I did not run the live gateway path in this read-only review.

## Solution Assessment

Unclear as proposed: widening the cron guard is directionally relevant, but the maintainable fix needs a policy decision and lifecycle checks for keep/session modes, ACP bindings, completion markers, and transcript archival. A documented subagent/ACP retention contract is safer than a bare cron reaper edit.
