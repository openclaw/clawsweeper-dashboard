## Summary

Keep open: current main still routes `cron.sessionRetention` through a cron-only reaper, while completed default subagent/ACP sessions can remain outside that path; the prior fix PR is closed unmerged and the remaining design needs maintainer policy direction.

## What This Changes

Request persistent retention cleanup for completed subagent and ACP sessions analogous to cron run-session retention.

## Best Possible Solution

Define a restart-safe retention policy for completed non-main sessions, likely through a dedicated subagent/ACP retention surface wired into session-store maintenance, with coverage for completed, active, thread-bound, restart, and transcript archival cases.

## Reproduction Assessment

Yes, from source inspection: the current reaper skips every key that is not `isCronRunSessionKey`, while default subagent/ACP tracking can leave completed sessions outside that path. I did not mutate the checkout to add a failing test, but the current-main path is direct and high-confidence.

## Solution Assessment

Unclear. A simple guard widening in the cron reaper would address part of the leak, but the maintainable fix needs a product decision on whether to overload `cron.sessionRetention` or add a dedicated retention policy for subagent/ACP sessions.
