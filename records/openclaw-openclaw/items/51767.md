## Summary

Keep open. Current main still has the reported retention gap: cron retention only sweeps cron-run session keys, while default keep-mode subagents and ACP/session task cleanup use separate lifecycle paths that do not provide durable `cron.sessionRetention`-style pruning for completed subagent/ACP rows. The direct fix PR was closed unmerged, and the later duplicate reports are closed into this tracker, so this remains the canonical issue.

## What This Changes

Request that `cron.sessionRetention` or an equivalent durable retention path prune completed subagent and ACP sessions instead of only cron-run sessions.

## Best Possible Solution

Keep this as the canonical tracker for a lifecycle-aware, restart-safe retention implementation that prunes completed subagent and ACP session-store rows while preserving active, parent-owned, and intentionally persistent sessions.

## Reproduction Assessment

Yes for source-level reproduction: current main still has a cron-key-only retention guard, default keep-mode subagents avoid archive/delete cleanup, and ACP cleanup is handled by separate task/session ownership paths. I did not live-reproduce RSS growth because this review was read-only.

## Solution Assessment

No, the issue's simple guard expansion is not clearly the best standalone fix because it may conflate cron, subagent keep/session modes, and ACP task lifecycles. The safer solution is a durable lifecycle-aware sweep with explicit terminal-state and transcript behavior plus regression tests.
