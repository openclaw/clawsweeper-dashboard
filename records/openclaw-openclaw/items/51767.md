## Summary

Keep open. Current `main` still has a cron-only retention reaper, default run-mode subagents still resolve to keep-mode with no archive timestamp, and the keep-mode subagent sweeper removes only registry records rather than session-store rows or transcripts. ACP has newer parent-owned task cleanup, but that is a separate lifecycle path and does not make `cron.sessionRetention` a generic subagent/ACP retention policy. The duplicate reports are closed into this tracker and the direct fix PR was closed unmerged, so this remains the canonical item.

## What This Changes

The issue requests that `cron.sessionRetention` or an equivalent retention policy prune completed subagent and ACP session-store rows instead of limiting cleanup to cron-run sessions.

## Best Possible Solution

Keep this as the canonical tracker and design a restart-safe retention path for completed native subagent and ACP sessions. The implementation should preserve active and intentionally persistent/thread-bound sessions, use `sessions.delete` or existing transcript archive helpers for store/transcript cleanup, define the retention config contract in schema/docs, and add tests for completed keep-mode subagents, active preservation, parent-owned ACP cleanup, and transcript handling.

## Reproduction Assessment

Yes. The code-path reproduction is high-confidence: current `main` still has a cron-only reaper guard, default run-mode subagents resolve to `cleanup: "keep"`, and keep-mode subagent cleanup removes registry records without deleting session-store entries. I did not live-reproduce RSS growth because the review was required to stay read-only.

## Solution Assessment

No. The issue’s simple suggested guard expansion is not clearly the best fix because subagent keep/session modes, ACP task ownership, active bindings, and transcript retention have separate lifecycle rules. A safer fix is a dedicated lifecycle-aware retention sweep or documented config contract with tests for the preservation cases.
