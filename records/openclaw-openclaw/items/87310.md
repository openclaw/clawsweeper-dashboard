## Summary

Keep open: current main still has the source-level ownerless diagnostic activity gap, and https://github.com/openclaw/openclaw/pull/90750 is an open closing PR that appears to own the narrow repair path. This issue should remain open until that PR, or an equivalent focused fix, lands.

## What This Changes

The issue asks OpenClaw to clear or quarantine stale diagnostic native tool-call activity during recovery, reset, or session replacement so later work on the same session key is not misclassified as `blocked_tool_call`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90750 with focused ownerless diagnostic activity eviction and regression coverage, then close this issue from the merged fix.

## Reproduction Assessment

Yes. Source inspection shows the current-main path: native tool start creates `activeTools`, reply-run teardown can end embedded work with `clearRunActivity: false`, and a later stale `tool_call` snapshot still classifies as `blocked_tool_call`; I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes. The best solution is a narrow diagnostic activity reconciliation/eviction path when the embedded owner is gone, with tests preserving genuine active tool blocking; a new config knob or blanket clear would be less maintainable.
