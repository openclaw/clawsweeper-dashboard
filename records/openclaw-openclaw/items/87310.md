## Summary

Keep open: current main and latest release still contain the source-visible reply-run teardown path that can leave diagnostic tool/model markers behind, and the open linked PR is the focused candidate fix. No duplicate repair job should be queued while that PR is open.

## What This Changes

The issue asks OpenClaw to clear or quarantine stale diagnostic native tool-call activity after recovery/reset so later work on the same session key is not misclassified as blocked_tool_call.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90750 if maintainers accept its owner-less marker eviction boundary, then close this issue from the merged fix while tracking broader lane/write-lock cleanup separately.

## Reproduction Assessment

Yes. Source inspection shows a native tool start can be recorded, reply-run teardown calls embedded-run end with clearRunActivity:false, and stale tool_call activity still maps to blocked_tool_call.

## Solution Assessment

Yes. The focused fix belongs in the diagnostic activity owner/reply-run teardown path; recovery-only or channel-specific cleanup would miss the clean owner-less teardown case.
