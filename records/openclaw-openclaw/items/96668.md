## Summary

Keep open: the latest head fixes the previous OpenAI Responses payload-budgeting gap, but the boosted empty-history lightweight path still uses raw tool-schema budgeting for OpenAI-compatible Completions even though those providers wrap tools before sending them to the model.

## What This Changes

The PR scopes a larger small-context prompt budget to history-free lightweight pre-prompt runs, subtracts tool-schema pressure, threads context/tools into the runner precheck, and adds regression plus SDK compatibility tests.

## Best Possible Solution

Extend provider-shaped tool budgeting to OpenAI-compatible Completions or keep the shared floor when provider conversion can expand tools, while preserving the PR's empty-history lightweight scope and optional SDK diagnostic field.

## Reproduction Assessment

Yes. Current main's source leaves 2,048 prompt tokens for a 4,096-token window, while the canonical issue and PR proof use a 3,544-token lightweight cron prompt that cannot fit that budget.

## Solution Assessment

No, not yet. The scoped no-history lightweight fix is the right layer, but it is incomplete until provider-shaped budgeting also covers OpenAI-compatible Completions tool payloads.
