## Summary

Keep open for maintainer review: the patch is a scoped, proof-backed bug mitigation with no new blocking code finding, but it intentionally changes small-context prompt/reserve behavior and only covers the no-active-tools lightweight path.

## What This Changes

The PR raises pre-prompt budget only for history-free lightweight embedded runs with no prompt images or active tools, and treats runtime `toolsAllow`, including `[]`, as a minimal prompt surface.

## Best Possible Solution

Land this only as a scoped no-active-tools mitigation if maintainers accept the reserve-headroom tradeoff, and keep the canonical lightweight-cron issue open for remaining tool-enabled/default coverage.

## Solution Assessment

This is an acceptable partial mitigation, not a complete fix for the canonical issue. The no-active-tools gate is safer than a global floor change, but maintainers still need to decide whether the partial scope and 512-token reserve headroom are acceptable.
