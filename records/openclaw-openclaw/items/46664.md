## Summary

Keep open. Current main has partial mobile queue primitives, but not the requested durable cross-platform offline command outbox with general text/voice capture, replay metadata, queue management, and persistence.

## What This Changes

The issue requests an offline, persistent mobile command queue for iOS and Android that captures user intents while disconnected and syncs them to the gateway when connectivity returns.

## Best Possible Solution

Define a staged mobile outbox design: persist existing Android voice queue behavior first, add a shared queued-intent payload with idempotency and captured-time metadata, then extend iOS and general text/voice queue management behind deliberate UX and privacy choices.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection establishes current partial behavior plus missing durable iOS/Android general command outbox support rather than a failing bug path.

## Solution Assessment

No, not as one broad implementation. The better maintainable path is a staged design that reuses existing mobile queues and gateway idempotency while separately deciding persistence, captured-time semantics, background sync expectations, and queue-management UX.

## Risks / Open Questions

- A durable mobile outbox needs explicit delivery semantics to avoid duplicate command execution after crashes or reconnect races.
- Storing offline text or audio locally has privacy and retention implications that should be decided before implementation.
- Background sync behavior differs across iOS and Android, so promising immediate automatic flush may overstate what the OS allows.
