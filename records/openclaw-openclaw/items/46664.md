## Risks / Open Questions

- Durable mobile storage would hold sensitive text commands and possibly audio or location context, so encryption, retention, deletion, redaction, and size limits need security design before implementation.
- At-least-once reconnect replay needs client-generated ids, captured-time metadata, Gateway acknowledgements, and dedupe behavior before the mobile apps can safely flush delayed commands.
- Mobile OS background execution limits mean the product promise must distinguish foreground reconnect, best-effort background sync, and user-visible review-before-send behavior.
- A full cross-platform outbox, queue-management UI, protocol metadata, and background-sync implementation is broad enough that a single automated fix PR would likely choose product/security policy prematurely.
