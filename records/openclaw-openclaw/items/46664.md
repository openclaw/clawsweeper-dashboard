## Label Justifications

- `P2`: This is a meaningful mobile reliability feature with limited immediate blast radius but non-trivial product, protocol, and privacy scope.
- `impact:message-loss`: The request is specifically about preventing offline user commands from being lost before they reach the gateway.

- `impact:security`: The proposed design may store sensitive text and audio on-device, so storage, deletion, and redaction policy are security-relevant.

## Summary

Keep open. Current main still has only narrow, mostly in-memory mobile queue paths plus gateway-side pending-node work, not the requested durable cross-platform text/voice mobile command outbox; the remaining work needs product, replay, storage, and privacy decisions.

## What This Changes

Add a durable offline queue to the iOS and Android node apps so text and voice commands captured while disconnected sync to the gateway when connectivity returns.

## Best Possible Solution

Define a staged mobile outbox contract first, then add durable encrypted local storage and idempotent reconnect replay for text/voice intents before broader queue-management UI.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source inspection gives a high-confidence current-state check: existing mobile queues are narrow and not durable general command outboxes.

## Solution Assessment

No for implementing the full SQLite/background-sync/queue-management proposal as one sweep. The maintainable path is a staged payload, persistence, privacy, and replay contract that builds on the existing narrow mobile queue patterns.

## Risks / Open Questions

- Persisting raw text commands or audio blobs without encryption, retention limits, delete controls, and log redaction would expose sensitive user data on mobile devices.
- At-least-once reconnect replay without a mobile-generated idempotency key and gateway dedupe contract could duplicate user commands after crashes or partial syncs.
- Mobile OS background execution limits mean “sync automatically when connectivity returns” may need explicit foreground/background product expectations rather than a guaranteed immediate flush.
