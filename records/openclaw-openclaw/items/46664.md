## Summary

Keep open. Current main and v2026.6.10 have partial queue primitives, but they still do not provide the requested durable iOS/Android phone outbox for arbitrary text and voice intents; the issue discussion correctly narrows the remaining work to persistence, idempotent replay, privacy, and product semantics.

## What This Changes

Request adds a durable offline command outbox for iOS/Android node apps that stores text and voice intents locally and replays them to the Gateway on reconnect with status and queue management.

## Best Possible Solution

Define a staged mobile outbox contract first, then implement encrypted durable storage and idempotent reconnect replay for text and voice before expanding queue-management UI.

## Risks / Open Questions

- Durable mobile storage would hold sensitive text commands and possibly audio or location context, so encryption, retention, deletion, redacted diagnostics, and size limits need security design before implementation.
- At-least-once reconnect replay needs client-generated ids, captured-time metadata, Gateway acknowledgements, and dedupe behavior before mobile apps can safely flush delayed commands.
- Mobile OS background execution limits mean the product promise must distinguish foreground reconnect, best-effort background sync, and review-before-send behavior.
- A full cross-platform outbox, queue-management UI, protocol metadata, and background-sync implementation is too broad for a single automated fix PR without maintainer product decisions.
