## Summary

Keep this PR open: the occupied-port preflight is the right direction, but current head drops the authorization URL on the busy-port path, so the immediate manual prompt can leave users with no URL to open. The regression test and terminal proof cover suppressing the opener and prompting immediately, but not preserving a usable manual OAuth URL.

## What This Changes

Adds a localhost:1455 availability preflight to the OpenAI Codex OAuth core wrapper and bundled OpenAI runtime copy, with a regression test and changelog entry.

## Best Possible Solution

Keep the port-conflict preflight, suppress automatic browser opening, and show or log the authorize URL with manual-open instructions in both OpenAI OAuth wrappers, with regression coverage for URL availability.

## Reproduction Assessment

Yes, by source inspection. Occupying 127.0.0.1:1455 makes the upstream OAuth helper fall back while still calling onAuth with the authorize URL; the PR’s busy-port branch returns before that URL is exposed.

## Solution Assessment

No, not as written. The narrow preflight is maintainable, but the fix must preserve a manual authorization URL while still avoiding openUrl on the occupied callback port.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve the manual OAuth URL when skipping the opener:** `src/plugins/provider-openai-codex-oauth.ts:243-245`
  - body: When localhost:1455 is busy, this return drops the `event.url` supplied by `loginOpenAICodex`.   - confidence: 0.93

  - body: The bundled OpenAI runtime copy has the same early return before the SDK OAuth handler can expose the authorization URL. Users going through the bundled plugin would reach the manual prompt with no URL to open, so this copy needs the same manual-URL behavior as the core wrapper.
  - confidence: 0.9
