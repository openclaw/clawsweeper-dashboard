## Summary

Keep open. Current main includes the merged structured OpenAI/Codex `server_error` cron retry fix, but the reported Anthropic/pi-ai `api_error: Internal server error` path still lands in a timeout/server-error lane that does not persist auth-profile cooldown or cap same-provider retries, so the original sustained-outage fast-fail problem remains unresolved.

## What This Changes

The issue requests that cron agent runs detect sustained LLM provider 500/api_error outages and fail or retry promptly instead of consuming the full timeout window.

## Best Possible Solution

Decide the provider-outage policy first, then implement a targeted fix that bounds same-provider retries for Anthropic/pi-ai internal-server outages without broadly poisoning generic 5xx/server-error cooldowns; keep cron catch-up, gap detection, and delivery notifications as separate scoped work unless maintainers expand this issue.

## Reproduction Assessment

Yes, at source level: current main classifies internal-server provider text into the timeout/server-error path, and the auth-profile policy returns null for both `timeout` and `server_error`. I did not run a live provider-outage replay.

## Solution Assessment

Unclear. The merged `server_error` cron retry fix is a good narrow solution for structured OpenAI/Codex retry state, but it does not solve the Anthropic/pi-ai same-provider fast-fail behavior and current cooldown policy makes the remaining fix a maintainer policy decision.
