## Summary

Keep open. Current main and the latest release include the merged structured `server_error` cron retry work, but the reported provider-outage path still has unresolved policy and behavior: internal-server/API failures remain timeout/server-error cases, shared provider cooldown is intentionally skipped, and recurring cron jobs do not implement provider-outage fast-fail catch-up or skipped-run delivery semantics.

## What This Changes

The issue requests fast-fail, retry/backoff, and delivery surfacing for cron agent jobs during sustained LLM provider API outages.

## Best Possible Solution

Keep the structured `server_error` cron retry support, then define and implement an owner-reviewed provider-outage fast-fail policy covering Anthropic-style errors, cooldown/skipping semantics, recurring cron catch-up, and failure notification with live proof.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a credible current-main path: cron still applies one job-level timeout, API/internal-server failures do not mark shared cooldown, and recurring cron error backoff does not implement provider-outage catch-up.

## Solution Assessment

Unclear. A narrow classifier change could improve the reported Anthropic case, but the maintainable fix needs a product/owner decision because provider 5xx cooldown, uniform-outage detection, skipped-run semantics, and notifications can affect shared routing behavior.
