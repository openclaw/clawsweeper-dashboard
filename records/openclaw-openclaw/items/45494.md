## Summary

Keep open. Current main partially handles structured `server_error` cron retry/state, but the reported provider-outage fast-fail path is still unresolved because timeout/server-error reasons deliberately do not create auth-profile cooldown, so same-provider fallback attempts are not skipped unless another cooldown signal already exists.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry/back off, and notify when sustained LLM provider 500 errors consume the full cron timeout window.

## Best Possible Solution

Keep the merged structured `server_error` retry behavior, then make an owner-reviewed provider-outage fast-fail policy with regression coverage for Anthropic SDK `api_error: Internal server error`, same-provider skip behavior, and cron failure notification/backoff state.

## Reproduction Assessment

No, not as a high-confidence live outage run. Source inspection gives a credible current-main path: provider server errors classify into timeout/server-error handling, those reasons do not mark auth-profile cooldown, and fallback only skips candidates that already have cooldown state.

## Solution Assessment

Unclear. A narrow targeted fix may be possible, but the best policy choice needs maintainer review because treating provider 5xx as cooldown-worthy can either solve fast-fail or incorrectly suppress healthy shared profiles.
