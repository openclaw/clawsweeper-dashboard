## Summary

Keep open. Current main still routes provider-side `api_error: Internal server error` text through the timeout/server-error classifier, and timeout failover reasons intentionally skip auth-profile cooldown, so the reported cron stall remains plausible and the closed unmerged fix PR is useful context rather than proof of a fix.

## What This Changes

The issue asks for cron agent jobs to fail promptly, retry with backoff, and notify on sustained LLM provider HTTP 500 `api_error` outages instead of consuming the full cron timeout window.

## Best Possible Solution

Classify Anthropic/Pi SDK internal-server `api_error` outage signals as provider-busy/overloaded or an equivalent cooldown-eligible reason, prove same-provider profile retries are capped and cross-provider fallback proceeds, and leave cron catch-up, gap detection, and notification redesign as separate follow-up work if still needed.

## Reproduction Assessment

Yes for the source-level failure path: current main classifies the reported internal-server `api_error` shape as `timeout`, and timeout reasons skip shared auth-profile cooldown. The full 180-second cron timing would need either a live/provider fixture or a focused runner test to reproduce end to end.

## Solution Assessment

Yes, the narrow classifier/cooldown repair is the best first fix because it uses existing overload rotation, cooldown, and fallback behavior instead of adding cron-specific provider policy. The safer variant should target Anthropic-style `api_error` payloads and preserve generic raw 500s as timeout unless maintainers choose otherwise.
