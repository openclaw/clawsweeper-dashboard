## Summary

Keep open: current main and the latest release have partial cron retry improvements for structured `server_error`, but they do not close the reported plain `api_error: Internal server error` outage path or the requested fast-fail, retry/backoff, and alert semantics.

## What This Changes

The issue asks OpenClaw cron agent jobs to fast-fail, retry/backoff, and notify on sustained LLM provider 500/outage failures instead of burning the full cron timeout window.

## Best Possible Solution

Keep this as the canonical maintainer-tracked provider-500 cron outage issue, then implement an explicit classifier/cooldown policy plus focused cron/fallback fast-fail, retry/backoff, and failure-alert coverage without merging the separate stream-stall and wall-clock timeout problems.

## Reproduction Assessment

Yes for a source-level path: current main recognizes transient internal-server/api-error shapes but classifies them as timeout-shaped failover, and cron retry only runs after the error result is returned. No live sustained provider-outage cron run was reproduced in this read-only review.

## Solution Assessment

No current complete solution exists on main. The best fix is not a broad one-line classifier change; maintainers should choose the provider-outage class and shared-cooldown policy, while tracking stream-stall and wall-clock timeout behavior as separate repairs.
