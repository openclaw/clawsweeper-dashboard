## Summary

Keep open: current main has partial structured `server_error` retry and cron failure-notification machinery, but it does not prove the central fast-fail behavior for the reported plain `api_error: Internal server error` cron outage path. ## What This Changes

The issue reports cron agent runs exhausting the full timeout during sustained LLM provider failures instead of classifying the provider outage, retrying promptly, and surfacing the skipped or failed run.

## Best Possible Solution

Keep this as the canonical cron provider-outage issue, choose the provider-error classification policy first, then land a focused fix with tests for plain `LLM error api_error: Internal server error`, cron retry timing, and failure notification behavior.

## Reproduction Assessment
