## Summary

Keep open. Current main and the latest release include the merged structured `server_error` cron retry work, but the reported Anthropic/Pi SDK `api_error: Internal server error` path still resolves to timeout/server-error behavior that does not mark auth-profile cooldown or bound same-provider retry time, so the sustained-outage fast-fail problem remains unresolved.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry with backoff, and notify delivery when sustained LLM provider internal-server errors would otherwise consume the full cron timeout window.

## Best Possible Solution

Define the provider-outage policy, then implement a targeted bound for Anthropic/Pi internal-server errors or same-provider retry exhaustion without broadly converting all generic 5xx/server-error signals into shared auth-profile cooldowns.

## Reproduction Assessment

Yes, at source level: current main and the latest release classify internal-server provider text into timeout/server-error paths, and the auth-profile policy returns `null` for those reasons. I did not run a live Anthropic outage replay.

## Solution Assessment

Unclear. The merged structured `server_error` cron retry fix is a good narrow repair for OpenAI/Codex retry state, but it does not solve the Anthropic/Pi same-provider fast-fail behavior; the remaining fix should be scoped by maintainers before changing cooldown semantics.
