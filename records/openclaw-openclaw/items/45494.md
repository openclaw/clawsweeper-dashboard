## Label Justifications

- `P1`: The issue reports scheduled agent jobs missing real data-collection windows during sustained provider outages, a broken cron workflow affecting users.
- `impact:data-loss`: The reported timeout behavior can leave gaps in expected cron data collection or backup history.
- `impact:auth-provider`: The unresolved behavior is in model-provider error classification, fallback routing, and auth-profile/cooldown policy.

## Summary

Keep open: current main and v2026.6.10 still route plain `api_error: Internal server error` through timeout-style classification, and cron retry/failure alerts happen after the agent run returns rather than proving provider-outage fast-fail before the cron window is consumed.

## What This Changes

The issue requests cron agent jobs to fast-fail, retry with backoff, and notify delivery when sustained LLM provider API errors would otherwise consume the full cron timeout window.

## Best Possible Solution

Keep this as the canonical HTTP 500 cron provider-outage issue, have maintainers choose the provider-outage classification policy, then land focused coverage for plain `LLM error api_error: Internal server error`, fast-fail timing, retry backoff, and failure notification behavior.

## Reproduction Assessment

Yes. Source inspection shows the reported `api_error: Internal server error` path is classified through timeout-style handling rather than a distinct provider-outage class; I did not run a live upstream outage reproduction.

## Solution Assessment

Unclear. A narrow classifier patch is plausible, but maintainers still need to choose whether this should be `server_error`, `overloaded`, or cron-local transient because each option changes cooldown, fallback, retry, and alert behavior.
