## Summary

Keep open. Current main has the merged structured `server_error` cron retry work, but the reported provider-outage fast-fail path is still not implemented because server-error/timeout classifications intentionally do not create auth-profile cooldown and fallback only skips candidates when cooldown already exists.

## What This Changes

Request that cron agent runs detect sustained LLM provider server errors across fallback candidates, fail early, reschedule or retry with backoff, and notify configured delivery instead of consuming the full run timeout.

## Best Possible Solution

Keep the structured `server_error` retry behavior, then define an owner-reviewed provider-outage fast-fail policy with regression coverage for Anthropic SDK `api_error: Internal server error`, same-provider skip behavior, cron backoff state, and configured failure notification.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a credible current-main path: server errors classify as timeout/server_error, those reasons do not mark auth-profile cooldown, and fallback skips same-provider candidates only when cooldown already exists.

## Solution Assessment

Unclear. A narrow fix may be possible, but the maintainable solution needs maintainer policy review because provider 5xx cooldown can improve cron fast-fail while also risking shared-profile suppression.
