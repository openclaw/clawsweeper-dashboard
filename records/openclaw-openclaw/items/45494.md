## Summary

Keep open. Current main and the latest release still do not fully solve the reported cron provider-outage path: structured server_error one-shot retry is partially fixed, but Anthropic-style API/internal-server failures still flow through generic timeout/server-error handling, the cron watchdog remains whole-run scoped, and recurring catch-up/notification semantics require maintainer policy.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry or catch up, and notify when sustained LLM provider API/internal-server errors make a fallback chain consume the full cron timeout.

## Best Possible Solution

Keep this concrete bug open while maintainers choose a coordinated provider-outage policy that works with the related recurring retry and reason-aware guardrail work, then implement fast failure, safe fallback/cooldown behavior, catch-up, and configured failure delivery together.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a credible current-main path: cron uses one whole-run timeout, API/internal-server errors are handled as generic timeout/server-error classes, and recurring failures do not catch up before the natural schedule without separate policy work.

Unclear.
