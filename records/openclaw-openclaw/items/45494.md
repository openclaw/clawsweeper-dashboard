## Summary

Keep open. Current main and v2026.5.22 only cover structured `server_error` one-shot retry; the reported Anthropic-style API/internal-server outage path still maps through timeout handling, and recurring catch-up/fast-fail policy remains unsettled.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry with backoff, and notify delivery targets when sustained LLM provider 500/API errors make the fallback chain unable to produce a run.

## Best Possible Solution

Keep the structured `server_error` support, then define an owner-reviewed provider-outage policy for Anthropic-style API errors, per-provider attempt bounds or uniform-outage detection, safe cooldown semantics, recurring catch-up, and configured failure delivery.

## Reproduction Assessment

Yes at source level, but not as a live provider-outage run: current main still has one cron job timeout, classifies plain API/internal-server text through timeout handling, and only documents one-shot retry. The exact sustained Anthropic 500 timing needs live or Crabbox proof before an automated bug-fix lane should run.

## Solution Assessment

Unclear. A narrow classifier change could improve the reported Anthropic case, but the maintainable fix needs maintainer review because provider 5xx cooldown, uniform-outage detection, recurring catch-up, and notification behavior affect shared routing and scheduler semantics.
