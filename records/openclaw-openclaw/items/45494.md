## Summary

Keep open. Current main has the merged structured `server_error` cron retry/state work, but the reported Anthropic-style plain `api_error: Internal server error` fast-fail/cooldown path is not fully resolved and still needs an owner decision around provider outage semantics.

## What This Changes

The issue requests cron agent runs to fail fast, retry/back off, and surface notifications when sustained LLM provider 500/API errors would otherwise consume the whole cron timeout window.

## Best Possible Solution

Keep the merged structured `server_error` retry behavior, then make an owner-reviewed provider-outage policy decision with regression coverage for the plain Anthropic `api_error: Internal server error` path, cooldown/skipping behavior, cron retry state, and failure alert behavior.

## Reproduction Assessment

No, not as a full live outage reproduction. Source inspection gives a credible current-main path for the remaining gap: the reported plain API error still lands on a non-cooldown timeout/server-error path, while fallback skipping depends on cooldown state.

## Solution Assessment

Unclear. A targeted Anthropic `api_error` reclassification is plausible, but maintainers need to choose between that, per-attempt timeout policy, or uniform-outage detection without broadly poisoning generic 5xx provider health.
