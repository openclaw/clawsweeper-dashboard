## Summary

Keep open. Current main has a partial structured `server_error` cron retry fix, but the reported provider-outage fast-fail path is still unresolved because generic/plain internal-server failures remain timeout/server-error signals that do not mark shared auth-profile cooldown, and fallback skipping only happens after cooldown already exists.

## What This Changes

Cron isolated agent jobs should fast-fail, retry with backoff, and notify on sustained LLM provider server errors instead of consuming the full `timeoutSeconds` window.

## Best Possible Solution

Keep the structured `server_error` cron retry behavior, then define an owner-reviewed provider-outage fast-fail policy with live proof and regression coverage for Anthropic-style server errors, same-provider skip behavior, recurring cron backoff, and failure notification.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a credible current-main path: cron timeout wraps the whole job, server/internal-server errors do not create shared cooldown, and fallback skipping requires cooldown that fresh server errors do not create.

## Solution Assessment

Unclear. A narrow classifier change could improve the reported case, but the maintainable fix needs maintainer policy review because provider 5xx cooldown and uniform-outage skip semantics can affect shared auth/provider routing.
