## Summary

Keep open: current main and v2026.6.8 contain structured `server_error` cron retry and failure-alert support, but the reported plain `api_error: Internal server error` cron outage path is still routed through timeout-style behavior and has no proof that uniform provider outages fast-fail before the cron window is consumed.

## What This Changes

The issue asks cron agent jobs to fast-fail sustained LLM provider API outages, retry with backoff, and notify configured delivery targets instead of burning the full cron timeout silently.

## Best Possible Solution

Keep this as the canonical HTTP 500 cron provider-outage issue, decide the provider-outage classification policy first, then land focused coverage for plain `LLM error api_error: Internal server error`, cron fast-fail timing, retry backoff, and failure notification behavior.

## Reproduction Assessment

Yes. ## Solution Assessment

Unclear. A narrow classifier patch is plausible, but maintainers still need to choose whether this belongs as `server_error`, `overloaded`, or a cron-local transient and how it should interact with stream-stall and outer cron timeout budgeting.
