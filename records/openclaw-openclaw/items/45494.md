## Summary

Keep open: current main and v2026.6.6 partially cover structured `server_error` cron retry handling, but the reported plain Anthropic/Pi-style `api_error: Internal server error` still flows through the generic timeout/model-scoped path rather than a provider-outage fast-fail path, and the requested skip/retry/delivery semantics need maintainer direction.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry, and notify when sustained LLM provider 500s would otherwise burn the full cron timeout window.

## Best Possible Solution

Make a maintainer-approved failover classification policy for plain provider 500s, then add focused regression coverage proving cron reaches cross-provider fallback or schedules a clear transient retry before the run timeout is exhausted.

## Reproduction Assessment

Yes, at source level: current main classifies the reported plain internal-server error text as `timeout`, and timeout cooldowns are model-scoped rather than provider-outage fast-fail signals. I did not force a live Anthropic outage in this read-only review.

## Solution Assessment

No, current main is only a partial solution; structured `server_error` retry support landed, but the plain provider-500 path and provider-wide fast-fail semantics remain unresolved. A safer fix needs maintainer choice on whether this maps to `overloaded`, `server_error`, or a distinct outage reason.
