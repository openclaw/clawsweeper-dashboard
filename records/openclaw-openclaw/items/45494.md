## Summary

Keep open. Current main still routes Anthropic/pi-ai `api_error: Internal server error` and structured internal-server payloads into the `timeout` failover bucket, and timeout reasons are explicitly excluded from auth-profile cooldown, so the reported cron stall remains source-reproducible; open #45594 overlaps with cron `server_error` retry state but does not cover this same-provider cooldown gap.

## What This Changes

Cron agent jobs with sustained LLM provider HTTP 500/internal-server failures should fail and retry promptly instead of spending the full cron timeout on same-provider fallback attempts.

## Best Possible Solution

Classify Anthropic/pi-ai internal-server `api_error` signals as cooldown-eligible provider-outage failures and prove same-provider retries are capped while cross-provider fallback advances; keep cron catch-up, gap detection, and delivery notifications as separate product work unless maintainers expand scope.

## Reproduction Assessment

Yes, at source level: current main classifies `api_error` internal-server text as `timeout`, then skips auth-profile cooldown for timeout reasons. I did not run a live 180-second cron outage replay in this read-only review.

## Solution Assessment

Yes, a narrow classifier/cooldown repair is the best first fix because it reuses existing overloaded/cooldown/fallback mechanics. If #45594 lands first, the repair should integrate with its `server_error` contract instead of creating a second overlapping path.
