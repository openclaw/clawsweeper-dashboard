- `P1`: The issue reports missed scheduled cron data-collection windows during real provider outages, which is an urgent agent workflow reliability failure.
- `impact:data-loss`: The reported impact is missing scheduled data collection or backup snapshots when cron runs burn their full timeout and skip the window.

## Summary

Keep open: current main partially handles structured OpenAI/Codex `server_error` cron retries, but the reported plain Anthropic/Pi-style `api_error: Internal server error` path still lands in generic timeout handling, and the requested fast-fail/skip/notification semantics need maintainer direction.

## What This Changes

Cron agent jobs should fast-fail, retry/back off, and surface provider-outage failures instead of exhausting `timeoutSeconds` during sustained LLM API 500 outages.

## Best Possible Solution

Add a maintainer-approved provider-outage classification path that handles plain `api_error` internal-server failures, preserves safe fallback/cooldown boundaries, and wires cron retry or failure-alert behavior without poisoning shared auth state for unrelated sessions.

## Reproduction Assessment

Yes at source level, but not as a live outage: a focused check can pass plain `LLM error api_error: Internal server error` through the current classifier and observe generic timeout behavior rather than the requested outage/cooldown classification.

## Solution Assessment

Unclear. The requested fast-fail direction is right, but a classifier-only patch is only partial until maintainers choose the permanent failure reason and cron skip/retry/notification semantics.
