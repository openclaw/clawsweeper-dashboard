## Summary

Keep open. Current main contains the merged structured `server_error` cron retry work, but the reported cron provider-outage path is still unresolved: API/internal-server failures remain conservative timeout/server-error cases, shared provider cooldown is not applied, and recurring cron backoff still does not implement fast-fail catch-up for outage windows.

## What This Changes

Cron agent jobs should fast-fail, retry/back off, and optionally notify when sustained LLM provider 500 errors would otherwise consume the whole job timeout.

## Best Possible Solution

Keep the shipped structured `server_error` cron retry support, then define an owner-reviewed provider-outage fast-fail policy with live proof and regression coverage for Anthropic-style errors, shared cooldown/skipping semantics, recurring cron catch-up, and failure notification.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a credible current-main path: cron still applies one job-level timeout, API/internal-server failures do not mark shared cooldown, and recurring cron error backoff does not implement provider-outage catch-up.

## Solution Assessment

Unclear. A narrow classifier change could improve the reported Anthropic case, but the maintainable fix needs a product/owner decision because provider 5xx cooldown, uniform-outage detection, skipped-run semantics, and notifications can affect shared routing behavior.
