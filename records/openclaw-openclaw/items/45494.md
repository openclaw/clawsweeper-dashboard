## Summary

Keep open: current main has partial cron retry improvements for structured `server_error`, but the reported plain `api_error: Internal server error` outage path still maps through timeout-shaped fallback handling and the requested fast-fail, retry, and alert semantics remain a maintainer/product decision.

The issue requests cron agent runs to fail, retry, and notify quickly during sustained LLM provider 500/outage failures instead of consuming the full `timeoutSeconds` window and missing scheduled work.

## Best Possible Solution

Keep this open for a maintainer-scoped design that separates plain provider 500 classification from stream-stall and outer-wall-clock timeout issues, then adds focused classifier, cron/fallback fast-fail, retry/backoff, and failure-alert coverage without poisoning shared auth-profile health.
