## Summary

Keep open. Current main has shipped recurring transient backoff, structured `server_error` retry handling, and configurable cron failure alerts, but the central fast-fail problem remains: the exact Anthropic/Pi SDK text `LLM error api_error: Internal server error` bypasses the structured `api_error` detector and is classified as `timeout`, while timeout failures intentionally do not establish the same-provider cooldown needed to skip repeated full SDK retry cycles. The unmerged focused fix at https://github.com/openclaw/openclaw/pull/45625 identified this same gap, and the newer May datapoint shows long generic failover timeouts still occur after the partial retry work shipped.

## What This Changes

Make cron agent runs promptly escape sustained provider 500 failures, retry with backoff, and surface configured failure notifications instead of consuming the full run timeout and missing scheduled work.

## Best Possible Solution

At the shared agent failover boundary, recognize the exact SDK-emitted transient `api_error` form and map it to an explicitly approved cooldown-capable reason, with regression coverage proving prompt same-provider skipping and cross-provider fallback; retain the already-shipped cron recurring backoff and failure-alert paths rather than adding cron-specific outage deduplication.

## Reproduction Assessment

Yes at source level: the exact reported plain-text error bypasses the structured `api_error` detector and follows timeout handling, while the cron watchdog bounds the complete agent run. A live current-main provider outage has not been forced, so timing improvement still needs real-runtime proof.

## Solution Assessment

The shared failover classifier is the narrowest maintainable fix boundary, not cron-specific uniform-error detection. The remaining judgment is which cooldown-capable failover reason correctly represents provider HTTP 500 without poisoning healthy profiles or changing unrelated provider behavior.
