## Summary

Keep open. Current main and the latest release only partially address this: structured `server_error` one-shot cron retry is shipped, but the reported sustained provider-outage path still lacks a settled fast-fail/cooldown policy, the cron watchdog remains whole-run scoped, and recurring catch-up plus configured failure notification semantics are not fully implemented.

## What This Changes

The issue asks cron isolated agent jobs to fast-fail during sustained LLM provider 5xx outages, retry or catch up with backoff, and notify configured delivery targets instead of consuming the full job timeout silently.

## Best Possible Solution

Define a coordinated provider-outage policy that adds bounded fast-fail behavior, reason-aware cooldown or uniform-outage handling, recurring catch-up, and configured failure delivery without poisoning healthy profiles.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a credible current-main path: cron uses one whole-run timeout, plain API/internal-server errors flow through generic timeout handling, and recurring failures do not catch up before the natural schedule without separate policy work.

## Solution Assessment

Unclear. A narrow classifier tweak could improve the exact Anthropic payload, but the maintainable solution needs maintainer review because provider 5xx cooldown, uniform-outage detection, recurring catch-up, skipped-run semantics, and delivery notifications affect shared routing and scheduler behavior.
