## Summary

Keep open: current main has partial fixes for structured `server_error` cron retry and failure visibility, but the exact plain-text Anthropic/Pi-style `api_error: Internal server error` path still does not become a cooldown-capable provider-outage signal, and the remaining reason semantics need maintainer approval.

## What This Changes

The issue requests fast-fail classification, retry/backoff, and delivery visibility for cron agent runs that can otherwise burn their full timeout during sustained LLM provider failures.

## Best Possible Solution

At the shared agent failover boundary, recognize the exact SDK-emitted transient `api_error` form and map it to a maintainer-approved cooldown-capable reason, with regression coverage proving same-provider skipping and cross-provider fallback before the cron watchdog expires.

## Reproduction Assessment

Yes at source level: the exact reported plain-text provider error is absent from regression coverage, and current classifier/cooldown code shows it does not become a profile-wide cooldown-capable provider-outage reason. A live sustained provider outage was not forced in this read-only review.

## Solution Assessment
