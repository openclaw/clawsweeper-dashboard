## Summary

Keep open. Current main includes the merged structured `server_error` cron retry work, but the reported provider-outage fast-fail path still needs policy review because server-error and timeout classifications intentionally avoid shared auth-profile cooldown, and fallback only skips same-provider candidates after cooldown already exists.

## What This Changes

The issue asks cron isolated agent jobs to fast-fail, retry/back off, and notify configured delivery when sustained LLM API 500s make the fallback chain fail.

## Best Possible Solution

Keep the structured `server_error` retry behavior, then define an owner-reviewed provider-outage fast-fail policy with regression coverage for Anthropic SDK `api_error: Internal server error`, same-provider skip behavior, recurring cron backoff, and configured failure notification.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a credible current-main path: server errors classify as timeout/server_error, those reasons do not mark auth-profile cooldown, and fallback skips same-provider candidates only when cooldown already exists.

## Solution Assessment

Unclear. A narrow cooldown classification change may improve this case, but the maintainable solution needs maintainer policy review because provider 5xx cooldown can also suppress shared profiles that might otherwise recover.
