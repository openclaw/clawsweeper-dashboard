## Summary

Keep open. Current main partially addresses cron `server_error` retry/state through the merged server-error retry work, but the original fast-fail problem is not fully resolved: timeout/server-error classifications still do not mark auth-profile cooldown, so same-provider fallback candidates can still be attempted instead of skipped during a provider outage.

## What This Changes

Make cron agent runs fast-fail, back off, and surface failures when model fallbacks hit sustained provider server errors instead of burning the full `timeoutSeconds` window.

## Best Possible Solution

Keep the merged structured `server_error` retry behavior, then make an owner-reviewed decision on a narrow provider-outage fast-fail policy with regression coverage for Anthropic `api_error: Internal server error`, fallback cooldown/skipping, and cron retry/alert state.

## Reproduction Assessment

No, not as a full live outage reproduction. Source inspection gives a credible current-main path for the remaining gap: `timeout`/`server_error` do not mark auth-profile cooldown, while fallback skips same-provider candidates only when cooldown state exists.

## Solution Assessment

Unclear. The merged structured `server_error` retry fix is good, but the remaining fast-fail behavior needs a maintainer choice between targeted Anthropic overload classification, per-attempt timeout policy, or uniform-outage detection without poisoning generic 5xx provider health.
