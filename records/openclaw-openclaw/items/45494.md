## Summary

Keep open. Current main has structured `server_error` one-shot retry and failure notification plumbing, but the reported provider-outage path still lacks a clear fast-fail policy: Anthropic-style internal-server/API errors remain timeout/server-error cases that do not mark shared provider cooldown, and recurring cron catch-up is still separate open product work.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry, and notify when sustained LLM provider 500/API errors exhaust or stall the fallback chain.

## Best Possible Solution

Keep the merged structured `server_error` retry support, then make an owner-reviewed provider-outage policy for Anthropic-style API errors, cooldown/skip semantics, recurring catch-up, and failure notification, while leaving generic recurring retry to the existing canonical issue and PR.

## Reproduction Assessment

No high-confidence live reproduction is available in this review. Source inspection gives a credible current-main path: cron still applies one job-level timeout, internal-server/API errors do not mark shared cooldown, and recurring failures do not catch up before the natural next run.

## Solution Assessment

Unclear. A narrow classifier change could improve the reported Anthropic case, but the maintainable fix needs product and owner review because provider 5xx cooldown, uniform-outage detection, skipped-run semantics, and notification behavior affect shared routing.
