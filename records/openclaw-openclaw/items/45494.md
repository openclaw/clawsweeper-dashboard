## Summary

Keep open. Current main partially handles structured `server_error` cron retry, but the reported provider-outage fast-fail behavior is still not implemented: transient provider 5xx/internal-server paths are classified as `timeout` or `server_error`, and both are intentionally excluded from auth-profile cooldown, so same-provider candidates are only skipped when separate cooldown state already exists.

## What This Changes

The issue reports cron agent jobs waiting through the full run timeout during sustained LLM provider 500 errors instead of failing fast, retrying with backoff, and notifying configured delivery targets.

## Best Possible Solution

Keep the merged structured `server_error` retry behavior, then make an owner-reviewed provider-outage policy for Anthropic `api_error: Internal server error` with focused coverage for classification, cooldown/skipping, cron retry state, and failure notification semantics.

## Reproduction Assessment

No, not as a full live provider-outage reproduction. Source inspection gives a credible current-main path for the remaining gap: the relevant server-error classes do not create auth-profile cooldown, while fallback skipping depends on cooldown state.

## Solution Assessment

Unclear. The merged structured `server_error` retry fix is a good narrow improvement, but the requested fast-fail behavior needs a maintainer choice between targeted Anthropic overload classification, uniform-outage detection, or per-attempt timeout budgeting.
