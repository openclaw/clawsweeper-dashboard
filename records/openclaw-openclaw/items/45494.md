## Summary

Keep open. Current main includes the merged structured `server_error` cron retry work, but the reported provider-outage path is still unresolved: API/internal-server failures remain conservative timeout/server-error cases, shared provider cooldown is not applied, and recurring cron backoff does not implement fast-fail catch-up for outage windows.

## What This Changes

Issue requests cron agent jobs to fast-fail, retry/back off, and notify delivery during sustained LLM provider 500/API-error outages instead of consuming the full job timeout.

## Best Possible Solution

Keep the shipped structured `server_error` cron retry support, then define an owner-reviewed provider-outage fast-fail policy with live proof and regression coverage for Anthropic-style errors, shared cooldown/skipping semantics, recurring cron catch-up, and failure notification.

## Solution Assessment

Unclear. A narrow classifier change could improve the reported Anthropic case, but the maintainable fix needs a product/owner decision because provider 5xx cooldown, uniform-outage detection, skipped-run semantics, and notifications can affect shared routing behavior.
