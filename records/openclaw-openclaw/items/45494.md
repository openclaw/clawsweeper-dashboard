## Summary

Keep open. Current main partially fixed structured `server_error` cron retry, but the reported provider-outage path still lacks a settled fast-fail/cooldown policy for Anthropic-style API/internal-server errors and recurring cron catch-up remains separately open.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry/back off, and notify delivery instead of consuming the full `timeoutSeconds` window during sustained LLM provider 500/API errors.

## Best Possible Solution

Keep the merged structured `server_error` support, then make an owner-reviewed provider-outage policy covering Anthropic-style API errors, per-provider attempt bounds or uniform-outage detection, safe cooldown semantics, recurring catch-up, and configured failure delivery.

No high-confidence live reproduction was run in this review. Source inspection gives a credible current-main path: cron still has one job-level timeout, API/internal-server errors avoid shared cooldown, and recurring failures do not retry before the natural next schedule.
