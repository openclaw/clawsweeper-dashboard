## Summary

Keep open: current main still treats plain Anthropic/Pi-style `api_error: Internal server error` as generic timeout, while the merged structured `server_error` retry fix and current cron-local cooldown policy do not implement the requested fast-fail, skip, and notification behavior.

## What This Changes

The issue asks cron isolated agent jobs to fast-fail, retry with backoff, and surface delivery notifications during sustained LLM provider 500/api_error outages instead of consuming the full job timeout.

## Best Possible Solution

Have maintainers choose the permanent provider-outage classification and cron semantics, then implement a focused fix across the failover classifier, cron retry/failure-alert behavior, and regression/live proof without broadening shared auth cooldown unexpectedly.

## Reproduction Assessment

Yes at source level, but not as a live outage. A focused check can pass plain `LLM error api_error: Internal server error` through the current classifier and observe generic timeout handling rather than the requested outage classification.

## Solution Assessment

Unclear. A classifier-only `overloaded` change is not necessarily the best fix after current cron-local transient cooldown isolation; maintainers need to choose the failure reason, retry behavior, and alert semantics together.
