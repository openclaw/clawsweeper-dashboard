## Summary

Keep open. Current main still classifies Anthropic/Pi SDK `api_error: Internal server error` and structured `server_error` payloads as `timeout`, while timeout failover reasons skip auth-profile cooldown, so the reported cron stall remains source-reproducible; open #45594 overlaps with cron `server_error` retry state but does not cover this cooldown gap.

## What This Changes

The issue requests that cron agent jobs fail or retry promptly during sustained provider 500/api_error outages instead of consuming the full cron timeout window.

## Best Possible Solution

Classify Anthropic-style internal-server `api_error` signals as cooldown-eligible provider outage failures and prove same-provider profile retries are capped while cross-provider fallback advances; keep cron catch-up, gap detection, and notification redesign separate unless maintainers explicitly expand scope.

## Reproduction Assessment

Yes for the source-level path: current main classifies `LLM error api_error: Internal server error` and structured internal-server payloads as `timeout`, then skips auth-profile cooldown for timeout reasons. A full live 180-second cron/provider outage replay was not run in this read-only review.

## Solution Assessment

Yes, a narrow classifier/cooldown repair is the best first fix because it reuses the existing overloaded/cooldown/fallback mechanics. If #45594 lands first, the repair should integrate with its `server_error` contract rather than adding a second overlapping reason path.
