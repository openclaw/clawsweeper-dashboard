## Label Justifications

- `P1`: The issue reports scheduled agent jobs missing real data-collection windows during sustained provider outages, which is a broken cron workflow affecting users.
- `impact:data-loss`: The reported timeout behavior can leave gaps in expected cron data collection or backup history.

## Summary

Keep open: current main and v2026.6.10 still route the reported plain `api_error: Internal server error` shape through timeout-style handling, and cron retry or alert behavior happens after the agent run returns rather than proving provider-outage fast-fail before the cron window is consumed.

## What This Changes

The issue asks OpenClaw cron agent jobs to fast-fail, retry with backoff, and notify on sustained definitive LLM provider 500 errors instead of consuming the full cron timeout window.

## Best Possible Solution

Keep this as the canonical HTTP 500 cron provider-outage issue, choose the provider-outage classification policy, then land focused coverage for plain `LLM error api_error: Internal server error`, fast-fail timing, retry backoff, and failure notification behavior.

## Reproduction Assessment

Yes. Source inspection shows the reported plain `api_error: Internal server error` path does not hit the structured `server_error` path and is still handled through timeout-style classification; I did not run a live upstream outage reproduction.

## Solution Assessment

Unclear. A narrow classifier patch is plausible, but maintainers still need to choose whether this should be `server_error`, `overloaded`, or cron-local transient because each option changes cooldown, fallback, retry, and alert behavior.
