## Summary

Keep open: current main and the latest release have partial cron retry improvements for structured `server_error`, but the reported plain Anthropic-style `api_error: Internal server error` outage path still goes through timeout-shaped failover behavior, and the requested fast-fail, retry/backoff, skipped-run semantics, and alerting policy need maintainer choice.

## What This Changes

The issue requests that cron agent jobs fast-fail on sustained provider API failures, retry or back off without burning the full cron timeout, and surface missed-run notifications.

## Best Possible Solution
