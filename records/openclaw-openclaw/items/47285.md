## Summary

Keep open: this is a useful feature request, but the PR still lacks live Azure embedding proof and the request construction does not yet match Azure embedding routing, auth, or API-version ownership.

## What This Changes

The branch adds Azure OpenAI detection to memory-lancedb's direct OpenAI embedding client, injects `api-key` and `api-version` defaults, adds mocked coverage, and adds a changelog entry.

## Best Possible Solution

Implement Azure-aware memory-lancedb embeddings with parsed Azure host suffixes, `AzureOpenAI` or explicit deployment-scoped request routing, no bearer auth on Azure API-key requests, a configurable API-version contract, focused request-shape tests, and redacted live Azure proof.

## Reproduction Assessment

Yes, source-reproducible. PR head still builds a generic OpenAI client and posts to `/embeddings`, while the Azure SDK and Microsoft REST contracts require Azure-specific API-version, auth, and embedding routing behavior.

## Solution Assessment

No. Constructor defaults on the generic OpenAI client are not the clean native Azure support path; the maintainable fix should own Azure routing, auth, host parsing, and API-version configuration together.

Full review comments:

- **[P2] Build deployment-scoped Azure embedding requests:** `extensions/memory-lancedb/index.ts:357-364`
  - body: The patch only adds Azure defaults to the generic OpenAI client, but `embed()` still posts to `/embeddings`. Azure's REST contract and the OpenAI SDK Azure client route embedding calls through Azure-specific deployment/API-version handling, so resource-root Azure URLs can still fail after this change. ([raw.githubusercontent.com](https://raw.githubusercontent.com/openai/openai-node/v6.37.0/src/azure.ts))
  - confidence: 0.92
- **[P2] Switch Azure credentials away from bearer auth:** `extensions/memory-lancedb/index.ts:357-363`
  - body: Passing `apiKey` to the generic `OpenAI` client still creates bearer auth, so adding `defaultHeaders: { "api-key": apiKey }` does not cleanly switch the request to Azure API-key auth. Use `AzureOpenAI` or an explicit Azure request path that does not send bearer auth for API-key requests. ([raw.githubusercontent.com](https://raw.githubusercontent.com/openai/openai-node/v6.37.0/src/azure.ts))
  - confidence: 0.9
- **[P2] Expose the Azure API version in config:** `extensions/memory-lancedb/index.ts:363`
  - body: The PR hard-codes `api-version=2024-02-01`, while memory-lancedb config and manifest do not accept an API-version setting. Users needing a different supported Azure version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure hosts before changing auth:** `extensions/memory-lancedb/index.ts:354`
  - body: Substring matching the entire `baseUrl` can trigger on non-Azure URLs whose path or query contains `.openai.azure.com`, and it misses Azure host families OpenClaw already recognizes such as `.services.ai.azure.com` and `.cognitiveservices.azure.com`. Parse `URL.hostname` and suffix-check known Azure hosts before changing request behavior.
  - confidence: 0.84
