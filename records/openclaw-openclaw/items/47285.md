## Summary

Keep open. Current main still lacks native Azure handling for memory-lancedb embeddings, but this PR needs real Azure proof and an Azure-aware request contract before it is safe to merge.

## What This Changes

The branch adds a memory-lancedb Azure endpoint substring check, Azure `api-key` and hard-coded `api-version` OpenAI client defaults, mocked constructor coverage, and a changelog entry.

## Best Possible Solution

Implement Azure-aware memory-lancedb embeddings with parsed Azure host detection, deployment-scoped routing through `AzureOpenAI` or equivalent explicit requests, Azure API-key auth, configurable API versioning, focused tests, and redacted live Azure proof.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the review blockers: PR head still constructs the generic `OpenAI` client, uses a one-suffix substring check, posts through the existing `/embeddings` path, and hard-codes `api-version`.

## Solution Assessment

No. Constructor defaults on the generic client are not enough for native Azure support; the maintainable path should own deployment routing, Azure auth, API-version configuration, and parsed host detection together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Route Azure embeddings through deployments:** `extensions/memory-lancedb/index.ts:360-363`
  - body: The patch only adds Azure defaults to a generic OpenAI client. When a user points `baseUrl` at an Azure resource root, `/openai/v1`, or a Foundry endpoint, memory-lancedb can still post to the non-deployment `/embeddings` shape while the config suggests Azure is supported.
  - confidence: 0.92
- **[P2] Use Azure auth instead of layered bearer auth:** `extensions/memory-lancedb/index.ts:356`
  - body: Passing `apiKey` into `new OpenAI` keeps the SDK's bearer-auth path; adding `api-key` as a default header does not cleanly switch the request to Azure auth. Use `AzureOpenAI` or an explicit request path that sends only the intended Azure API-key auth.
  - confidence: 0.9
- **[P2] Make the Azure API version configurable:** `extensions/memory-lancedb/index.ts:361`
  - body: The PR pins `api-version` to `2024-02-01`, but memory-lancedb config and manifest reject unknown embedding keys, so users cannot select the Azure API version their deployment requires.
  - confidence: 0.91
- **[P2] Parse supported Azure hostnames:** `extensions/memory-lancedb/index.ts:352`
  - body: Checking `baseUrl?.includes(".openai.azure.com")` treats the whole URL as a string and misses Azure AI Foundry or Cognitive Services hosts that current OpenClaw Azure handling recognizes by parsed hostname.
  - confidence: 0.86
