## Label Justifications

- `P2`: This is a scoped memory plugin enhancement with limited blast radius but meaningful Azure embedding usability impact.
- `merge-risk: 🚨 compatibility`: The PR hard-codes Azure request behavior and host detection that can fail for supported Azure endpoint variants or deployments needing another API version.
- `merge-risk: 🚨 auth-provider`: The PR changes credential header behavior for memory embedding requests while still using the generic OpenAI API-key client path.

## Summary

Keep open: the PR has useful Azure embedding intent, but the patch still shapes Azure requests through the generic OpenAI client, lacks configurable API versioning and parsed Azure host support, and has no live Azure behavior proof.

## What This Changes

The PR adds `.openai.azure.com` detection to memory-lancedb's direct OpenAI embedding client, injects an `api-key` header and hard-coded `api-version`, adds mocked coverage, and updates the changelog.

## Best Possible Solution

Implement Azure-aware memory-lancedb embeddings with parsed host detection, deployment-scoped routing through `AzureOpenAI` or an equivalent explicit request path, non-bearer API-key auth, configurable API versioning in config/manifest, focused tests, and redacted live Azure proof.

## Reproduction Assessment

Yes, source-reproducible. PR head still constructs the generic `OpenAI` client and posts to `/embeddings`, while the OpenAI SDK Azure source shows API-version query, deployment rewrite, and `api-key` auth behavior in `AzureOpenAI`.

## Solution Assessment

No. Constructor defaults on the generic client are not enough for native Azure support; the maintainable path should own deployment routing, Azure auth, API-version configuration, and parsed host detection together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Build deployment-scoped Azure embedding requests:** `extensions/memory-lancedb/index.ts:360-363`
  - body: The patch only adds Azure defaults to a generic OpenAI client. When a user points `baseUrl` at an Azure resource root, memory-lancedb will still call `/embeddings` instead of the deployment-scoped Azure path, so real Azure embedding calls can fail while the config suggests Azure is supported.
  - confidence: 0.92
- **[P2] Stop sending bearer auth on Azure embedding calls:** `extensions/memory-lancedb/index.ts:358`
  - body: Passing `apiKey` into `new OpenAI` continues the SDK's bearer auth path; adding `api-key` as a default header does not cleanly remove `Authorization: Bearer` for Azure. Use `AzureOpenAI` or an explicit Azure request path that sends only the intended Azure API-key auth.
  - confidence: 0.9
- **[P2] Make the Azure API version configurable:** `extensions/memory-lancedb/index.ts:363`
  - body: The PR pins `api-version` to `2024-02-01`, but memory-lancedb config and manifest reject unknown embedding keys, so users cannot select the Azure API version their deployment requires.
  - confidence: 0.9
- **[P2] Parse supported Azure hostnames before switching auth:** `extensions/memory-lancedb/index.ts:354`
  - body: Checking `baseUrl?.includes('.openai.azure.com')` treats the whole URL as a string and misses Azure AI Foundry hosts that OpenClaw already classifies as Azure, such as `*.services.ai.azure.com`. Parse `URL.hostname` and suffix-check the supported host families before changing auth.
  - confidence: 0.84
