## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Use deployment-scoped Azure embedding routing:** `extensions/memory-lancedb/index.ts:360-363`
  - body: The patch only adds Azure defaults to a generic OpenAI client. When a user points `baseUrl` at an Azure resource root, `/openai/v1`, or a Foundry endpoint, memory-lancedb will still post to `/embeddings` instead of the deployment-scoped Azure embeddings path, so real Azure calls can fail while the config suggests Azure is supported.
  - confidence: 0.92
- **[P2] Avoid bearer auth on Azure embedding requests:** `extensions/memory-lancedb/index.ts:358`
  - body: Passing `apiKey` into `new OpenAI` keeps the SDK's bearer auth path; adding `api-key` as a default header does not cleanly remove `Authorization: Bearer` for Azure. Use `AzureOpenAI` or an explicit Azure request path that sends only the intended Azure API-key auth.
  - confidence: 0.9
- **[P2] Expose the Azure API version in config:** `extensions/memory-lancedb/index.ts:363`
  - body: The PR pins `api-version` to `2024-02-01`, but memory-lancedb config and manifest reject unknown embedding keys, so users cannot select the Azure API version their deployment requires.
  - confidence: 0.9
- **[P2] Parse Azure host suffixes before switching auth:** `extensions/memory-lancedb/index.ts:354`
  - body: Checking `baseUrl?.includes('.openai.azure.com')` treats the whole URL as a string and misses Azure AI Foundry hosts that OpenClaw already classifies as Azure, such as `*.services.ai.azure.com`. Parse `URL.hostname` and suffix-check the supported host families before changing auth.
  - confidence: 0.84
