## Summary

Keep open: this is useful feature work, but the PR still has native Azure request-shaping defects and lacks contributor-supplied real Azure behavior proof, so it should not merge as-is.

## What This Changes

The PR adds Azure OpenAI detection to memory-lancedb's direct OpenAI-compatible embedding client, injects api-key/api-version constructor options, adds a mocked constructor test, and updates the changelog.

## Best Possible Solution

Implement Azure-aware memory-lancedb request shaping with validated hostname parsing, deployment-scoped embeddings URLs or AzureOpenAI, configurable apiVersion in config/manifest/docs, focused tests, and redacted live Azure proof while preserving the generic OpenAI-compatible path.

## Reproduction Assessment

Yes for the PR defects by source inspection, but not by live Azure execution. Configure memory-lancedb with embedding.provider openai, an Azure resource baseUrl, an Azure key, and a deployment/model; the proposed path still uses the generic OpenAI client and /embeddings transport.

## Solution Assessment

No. The constructor-only patch is not the best solution because native Azure support needs endpoint normalization or AzureOpenAI, Azure auth semantics, configurable API versioning, hostname parsing, focused tests, and runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding URLs:** `extensions/memory-lancedb/index.ts:357-361`
  - body: For the resource-root workflow described by the PR, this still sets baseURL directly and embed() posts to /embeddings. Azure embeddings require /openai/deployments/{deployment}/embeddings?api-version=..., so a base URL like https://resource.openai.azure.com will miss the deployment path.
  - confidence: 0.92
- **[P2] Avoid bearer auth on Azure API-key requests:** `extensions/memory-lancedb/index.ts:356-360`
  - body: Passing apiKey to the generic OpenAI client still makes it add Authorization: Bearer; adding defaultHeaders: {"api-key": apiKey} does not suppress that. Use AzureOpenAI or explicitly remove bearer auth so Azure API-key calls do not carry conflicting auth.
  - confidence: 0.86
- **[P2] Expose the Azure API version in plugin config:** `extensions/memory-lancedb/index.ts:361`
  - body: The PR hard-codes api-version=2024-02-01, while the memory-lancedb config/schema/manifest do not accept an apiVersion setting. Users who need another supported Azure API version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure hosts by hostname suffix:** `extensions/memory-lancedb/index.ts:352`
  - body: baseUrl?.includes(".openai.azure.com") can match path/query text on non-Azure hosts and misses the .services.ai.azure.com and .cognitiveservices.azure.com host families used elsewhere in OpenClaw. Parse URL.hostname and check the known suffixes before changing auth behavior.
  - confidence: 0.84
