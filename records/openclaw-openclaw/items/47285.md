## Summary

Keep this PR open, but the current patch is not mergeable: native Azure OpenAI embeddings are useful, yet current main still only has the generic memory-lancedb OpenAI-compatible path and this diff does not build Azure deployment URLs, fully separate Azure auth shape, or expose API-version configuration.

## What This Changes

The PR changes the memory-lancedb direct OpenAI-compatible embedding client to detect `.openai.azure.com` base URLs and add Azure `api-key` plus `api-version=2024-02-01` OpenAI SDK defaults.

## Best Possible Solution

Add Azure-aware memory-lancedb request shaping with parsed hostname detection, deployment-scoped embedding URLs, explicit API-version configuration, docs, changelog, and targeted tests while preserving the generic OpenAI-compatible proxy path for non-Azure endpoints.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure memory-lancedb with `embedding.provider: "openai"`, an `embedding.apiKey`, an Azure resource `baseUrl`, and a deployment name as `model`; current main posts to `/embeddings`, and this PR would still not build `/openai/deployments/{deployment}/embeddings` or expose API-version configuration.

## Solution Assessment

No. The constructor-only patch is not the best fix because Azure support needs endpoint normalization, auth-header selection, API-version configuration, tests, and docs rather than hard-coded defaults on the generic OpenAI client.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build Azure deployment URLs for embeddings:** `extensions/memory-lancedb/index.ts:176`
  - body: The Azure branch leaves `baseURL` as the configured resource URL while `embed()` still posts to `/embeddings`. For the resource-root workflow described by this PR, Azure embeddings use `/openai/deployments/{deployment-id}/embeddings?api-version=...`, so requests will go to the wrong endpoint. ([learn.microsoft.com](https://learn.microsoft.com/en-us/azure/ai-services/openai/reference?utm_source=openai))
  - confidence: 0.92
- **[P2] Avoid generic bearer auth for Azure requests:** `extensions/memory-lancedb/index.ts:174-178`
  - body: This still constructs the generic `OpenAI` client with `apiKey`. OpenAI SDK v6.35.0 adds `Authorization: Bearer <apiKey>` before default headers, so adding `api-key` does not make the request Azure-specific and can send an invalid bearer credential alongside the Azure key header. ([raw.githubusercontent.com](https://raw.githubusercontent.com/openai/openai-node/v6.35.0/src/client.ts))
  - confidence: 0.88
- **[P2] Expose Azure API version in config:** `extensions/memory-lancedb/index.ts:179-180`
  - body: The patch hard-codes `api-version=2024-02-01`, but memory-lancedb config and manifest schema do not accept an API-version field. Users who need a tenant-required or newer Azure version would have no supported override path.
  - confidence: 0.9
- **[P2] Parse Azure hosts from the URL hostname:** `extensions/memory-lancedb/index.ts:173`
  - body: Checking `baseUrl?.includes(".openai.azure.com")` can match path/query text and misses Azure host families that OpenClaw already handles elsewhere, including `*.services.ai.azure.com` and `*.cognitiveservices.azure.com`. Use parsed hostname suffix checks instead.
  - confidence: 0.84
