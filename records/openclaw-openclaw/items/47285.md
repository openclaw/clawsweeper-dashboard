## Summary

Keep open: the feature request is still relevant, but this patch does not yet match Azure OpenAI embedding request semantics and still lacks contributor-supplied live Azure proof. Current main has not already implemented native Azure memory-lancedb embeddings.

## What This Changes

The PR adds Azure OpenAI detection to memory-lancedb’s OpenAI-compatible embedding client, plus a mocked constructor-options test and a changelog entry.

## Best Possible Solution

Implement Azure-aware memory-lancedb embeddings with parsed host suffixes, deployment-scoped request routing or AzureOpenAI, configurable API version metadata, focused regression tests, and redacted live Azure proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the patched path still uses generic OpenAI with post("/embeddings"), so Azure resource-root endpoints will not get deployment-scoped embedding URLs and generic apiKey still produces bearer auth. No live Azure reproduction proof is present.

## Solution Assessment

No. The constructor-only patch is not the best solution because native Azure support needs Azure request-shaping semantics, hostname parsing, configurable API versioning, and real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding URLs:** `extensions/memory-lancedb/index.ts:357-361`
  - body: For the resource-root workflow described by this PR, setting `baseURL` still leaves `embed()` posting to `/embeddings`. Azure embeddings need a deployment-scoped path such as `/openai/deployments/{deployment}/embeddings?api-version=...`, so `https://resource.openai.azure.com` will hit the wrong route.
  - confidence: 0.92
- **[P2] Avoid bearer auth on Azure API-key requests:** `extensions/memory-lancedb/index.ts:356-360`
  - body: Passing `apiKey` to the generic OpenAI client still makes the SDK add `Authorization: Bearer ...`; adding `defaultHeaders: { "api-key": apiKey }` does not suppress that header. Use `AzureOpenAI` or an Azure-specific transport path that sends API-key auth without the conflicting bearer header.
  - confidence: 0.89
- **[P2] Expose the Azure API version in plugin config:** `extensions/memory-lancedb/index.ts:361`
  - body: The PR hard-codes `api-version=2024-02-01`, while the memory-lancedb config parser and manifest do not accept an `apiVersion` setting. Users who need a different supported Azure API version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure hosts by hostname suffix:** `extensions/memory-lancedb/index.ts:352`
  - body: `baseUrl?.includes(".openai.azure.com")` can match path/query text on non-Azure hosts and misses Azure host families that OpenClaw already recognizes elsewhere, including `.services.ai.azure.com` and `.cognitiveservices.azure.com`. Parse `URL.hostname` and check known suffixes before switching auth/request behavior.
  - confidence: 0.84
