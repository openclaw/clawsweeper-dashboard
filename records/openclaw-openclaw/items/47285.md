## Summary

Keep open: this is a useful feature, but the patch is not mergeable as-is because it only adds generic OpenAI client defaults while Azure embeddings need Azure-specific routing/auth semantics, configurable API versioning, and real Azure runtime proof.

## What This Changes

The PR adds Azure endpoint detection to memory-lancedb’s OpenAI-compatible embedding client, injects api-key/api-version defaults for matching base URLs, adds a mocked constructor-options test, and adds a changelog entry.

## Best Possible Solution

Implement Azure-aware memory-lancedb embeddings with hostname-suffix detection, deployment-scoped routing or AzureOpenAI, configurable API versioning, focused request-shape tests, and redacted live Azure proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current memory-lancedb posts through the generic OpenAI client to /embeddings, while the OpenAI SDK AzureOpenAI path adds api-version, api-key auth, and deployment endpoint rewriting. There is no live after-fix Azure proof.

## Solution Assessment

No. Constructor-only Azure detection is not the narrow maintainable solution; the patch needs Azure-aware request shaping, hostname parsing, API-version configuration, and runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding requests:** `extensions/memory-lancedb/index.ts:357-360`
  - body: For the resource-root workflow described by this PR, setting constructor defaults still leaves embed() posting to `/embeddings`. Azure embeddings need deployment-scoped routing such as `/openai/deployments/{deployment}/embeddings?api-version=...`, so a plain Azure resource base URL will hit the wrong route.
  - confidence: 0.92
- **[P2] Avoid bearer auth on Azure API-key requests:** `extensions/memory-lancedb/index.ts:357-363`
  - body: Passing `apiKey` to the generic OpenAI client still makes the SDK add `Authorization: Bearer ...`; adding `defaultHeaders: { "api-key": apiKey }` does not suppress that credential. Use `AzureOpenAI` or an Azure-specific transport path that sends API-key auth without the generic bearer header.
  - confidence: 0.89
- **[P2] Expose the Azure API version in plugin config:** `extensions/memory-lancedb/index.ts:363`
  - body: The PR hard-codes `api-version=2024-02-01`, while the memory-lancedb config parser and manifest do not accept an `apiVersion` setting. Users who need a different supported Azure API version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure hosts by hostname suffix:** `extensions/memory-lancedb/index.ts:354`
  - body: `baseUrl?.includes(".openai.azure.com")` can match path/query text on non-Azure hosts and misses Azure host families OpenClaw already recognizes elsewhere, including `.services.ai.azure.com` and `.cognitiveservices.azure.com`. Parse `URL.hostname` and check known suffixes before switching auth/request behavior.
  - confidence: 0.84
