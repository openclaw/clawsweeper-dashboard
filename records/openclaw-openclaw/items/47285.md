## Summary

Keep open: native Azure embeddings are useful, but this PR is not merge-ready because it only adjusts generic OpenAI constructor options while Azure embedding calls need deployment-scoped request shaping, Azure auth semantics, configurable API versioning, and real Azure proof.

## What This Changes

The PR adds Azure endpoint detection to memory-lancedb's OpenAI-compatible embedding client, injects Azure `api-key` and `api-version` options, adds mocked coverage, and adds a changelog entry.

## Best Possible Solution

Implement Azure-aware memory-lancedb request shaping with parsed host suffixes, deployment-scoped embedding URLs or `AzureOpenAI`, config/manifest/docs support for `apiVersion`, focused tests, and redacted live Azure proof.

## Reproduction Assessment

Yes for the PR defects by source inspection, not for live Azure execution. Configure memory-lancedb with `embedding.provider: "openai"`, an Azure resource `baseUrl`, an Azure key, and a deployment/model; the proposed path still uses generic `OpenAI` and `/embeddings` transport.

## Solution Assessment

No. The constructor-only patch is not the best solution because native Azure support needs Azure request-shaping semantics, configurable API versioning, hostname parsing, focused tests, and runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding URLs:** `extensions/memory-lancedb/index.ts:357-361`
  - body: For the resource-root workflow described by the PR, this still sets `baseURL` directly and `embed()` posts to `/embeddings`. Azure embeddings need a deployment-scoped endpoint such as `/openai/deployments/{deployment}/embeddings?api-version=...`, so `https://resource.openai.azure.com` will hit the wrong path.
  - confidence: 0.92
- **[P2] Avoid bearer auth on Azure API-key requests:** `extensions/memory-lancedb/index.ts:356-360`
  - body: Passing `apiKey` to the generic `OpenAI` client still makes the SDK add `Authorization: Bearer ...`; adding `defaultHeaders: { "api-key": apiKey }` does not suppress that header. Use `AzureOpenAI` or a transport path that sends Azure API-key auth without conflicting bearer auth.
  - confidence: 0.86
- **[P2] Expose the Azure API version in plugin config:** `extensions/memory-lancedb/index.ts:361`
  - body: The PR hard-codes `api-version=2024-02-01`, while the memory-lancedb config parser and manifest do not accept an `apiVersion` setting. Users who need another supported Azure API version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure hosts by hostname suffix:** `extensions/memory-lancedb/index.ts:352`
  - body: `baseUrl?.includes(".openai.azure.com")` can match path/query text on non-Azure hosts and misses `.services.ai.azure.com` and `.cognitiveservices.azure.com`, which OpenClaw already treats as Azure host families elsewhere. Parse `URL.hostname` and check the known suffix list before changing auth behavior.
  - confidence: 0.84
