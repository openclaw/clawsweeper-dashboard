## Summary

Keep open: the PR is a useful feature direction, but the current patch still shapes Azure embedding requests through the generic OpenAI client, lacks configurable API versioning, and has only mocked/local verification rather than real Azure memory-lancedb proof.

## What This Changes

The PR adds `.openai.azure.com` detection to memory-lancedb’s OpenAI-compatible embedding client, injects `api-key` and a fixed `api-version`, adds a mocked constructor-options test, and updates the changelog.

## Best Possible Solution

Implement Azure-aware memory-lancedb embeddings with hostname-suffix detection, deployment-scoped routing or `AzureOpenAI`, configurable API versioning, focused request-shape tests, docs/manifest follow-through, and redacted live Azure proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current memory-lancedb posts through the generic OpenAI client to `/embeddings`, while the OpenAI SDK Azure path adds `api-version`, `api-key` auth, and deployment endpoint rewriting. There is no live after-fix Azure proof.

## Solution Assessment

No. Constructor-only Azure detection on the generic client is not the narrow maintainable solution; the patch needs Azure-aware request shaping, hostname parsing, API-version configuration, and runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding requests:** `extensions/memory-lancedb/index.ts:358-363`
  - body: For the resource-root workflow described by this PR, setting constructor defaults still leaves `embed()` posting to `/embeddings`. Azure embeddings need deployment-scoped routing such as `/openai/deployments/{deployment}/embeddings?api-version=...`, so a plain Azure resource base URL will hit the wrong route.
  - confidence: 0.92
- **[P2] Omit bearer auth on Azure API-key requests:** `extensions/memory-lancedb/index.ts:360`
  - body: Passing `apiKey` to the generic OpenAI client still makes the SDK add `Authorization: Bearer ...`; adding `defaultHeaders: { "api-key": apiKey }` does not remove that credential. Use `AzureOpenAI` or an Azure-specific transport path that sends API-key auth without the generic bearer header.
  - confidence: 0.9
- **[P2] Expose the Azure API version in plugin config:** `extensions/memory-lancedb/index.ts:361`
  - body: The PR hard-codes `api-version=2024-02-01`, while the memory-lancedb parser and manifest do not accept an API-version setting. Users who need a different supported Azure API version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure endpoints by hostname suffix:** `extensions/memory-lancedb/index.ts:352`
  - body: `baseUrl?.includes(".openai.azure.com")` can match path or query text on non-Azure hosts and misses Azure host families OpenClaw already recognizes elsewhere, including `.services.ai.azure.com` and `.cognitiveservices.azure.com`. Parse `URL.hostname` and check known suffixes before switching auth/request behavior.
  - confidence: 0.84
