## Summary

Keep this PR open for maintainer review: the Azure memory-lancedb feature is useful, but the current patch is still incomplete and has no contributor-supplied live Azure proof. Current main does not already implement the central change, and an open Microsoft tracker still lists this PR as remaining Azure work.

## What This Changes

The PR adds Azure endpoint detection to memory-lancedb’s OpenAI-compatible embeddings client, injects `api-key` and `api-version` options, adds mocked coverage, and adds a changelog entry.

## Best Possible Solution

Implement Azure-aware memory-lancedb request shaping with validated hostname parsing, deployment-scoped embeddings URLs or `AzureOpenAI`, configurable `apiVersion` in config/manifest/docs, focused tests, and redacted live Azure proof while preserving generic OpenAI-compatible providers.

## Reproduction Assessment

Yes for the PR defects by source inspection, but not by live Azure execution. Configure memory-lancedb with `embedding.provider: "openai"`, an Azure resource `baseUrl`, an Azure key, and a deployment/model; the proposed path still uses the generic OpenAI client and `/embeddings` transport.

## Solution Assessment

No. The constructor-only patch is not the best solution because native Azure support needs endpoint normalization or `AzureOpenAI`, Azure auth semantics, configurable API versioning, hostname parsing, focused tests, and runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding URLs:** `extensions/memory-lancedb/index.ts:357-361`
  - body: For the resource-root workflow described by the PR, this still sets `baseURL` directly and `embed()` posts to `/embeddings`. Azure embeddings need a deployment-scoped endpoint such as `/openai/deployments/{deployment}/embeddings?api-version=...`, so a base URL like `https://resource.openai.azure.com` will miss the deployment path.
  - confidence: 0.92
- **[P2] Avoid bearer auth on Azure API-key requests:** `extensions/memory-lancedb/index.ts:356-360`
  - body: Passing `apiKey` to the generic `OpenAI` client still makes the SDK add `Authorization: Bearer`; adding `defaultHeaders: { "api-key": apiKey }` does not suppress that. Use `AzureOpenAI` or otherwise ensure Azure API-key calls do not carry conflicting bearer auth.
  - confidence: 0.86
- **[P2] Expose the Azure API version in plugin config:** `extensions/memory-lancedb/index.ts:361`
  - body: The PR hard-codes `api-version=2024-02-01`, while the memory-lancedb config/schema/manifest do not accept an `apiVersion` setting. Users who need another supported Azure API version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure hosts by hostname suffix:** `extensions/memory-lancedb/index.ts:352`
  - body: `baseUrl?.includes(".openai.azure.com")` can match path or query text on non-Azure hosts and misses the `.services.ai.azure.com` and `.cognitiveservices.azure.com` host families used elsewhere in OpenClaw. Parse `URL.hostname` and check known suffixes before changing auth behavior.
  - confidence: 0.84
