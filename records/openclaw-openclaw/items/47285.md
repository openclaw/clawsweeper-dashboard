## Summary

Keep this PR open, but do not merge it as-is: the feature target is real, yet the patch only adds Azure headers/query defaults and does not make memory-lancedb build Azure deployment-scoped embedding requests or expose the required API-version/config surface.

## What This Changes

The PR changes the memory-lancedb direct OpenAI-compatible embedding client to detect `.openai.azure.com` base URLs and add Azure `api-key` plus `api-version=2024-02-01` OpenAI SDK defaults.

## Best Possible Solution

Add Azure-aware request shaping in memory-lancedb with parsed hostname detection, deployment-scoped embedding URLs, explicit API-version configuration, docs, and targeted tests while preserving the generic OpenAI-compatible proxy path for non-Azure endpoints.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure memory-lancedb with `embedding.provider: "openai"`, an `embedding.apiKey`, and an Azure resource `baseUrl`; current main posts to `/embeddings`, and this PR would still not build Azure's deployment-scoped URL or expose API-version configuration.

## Solution Assessment

No. The constructor-only patch is not the narrowest maintainable fix because Azure support needs endpoint normalization, auth-header selection, API-version config, and tests rather than hard-coded defaults on the generic OpenAI client.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build Azure deployment URLs for embeddings:** `extensions/memory-lancedb/index.ts:176-177`
  - body: The Azure branch leaves `baseURL` as the configured resource URL while `embed()` still posts to `/embeddings`. For the resource-root workflow described by this PR, Azure expects `/openai/deployments/{deployment}/embeddings`, so requests will go to the wrong endpoint at runtime.
  - confidence: 0.91
- **[P2] Avoid sending OpenAI bearer auth to Azure:** `extensions/memory-lancedb/index.ts:174-178`
  - body: This still constructs the generic `OpenAI` client with `apiKey`. In the locked OpenAI SDK, that client adds `Authorization: Bearer <key>` before merging defaults, so adding `api-key` leaves Azure requests with the wrong auth shape unless the bearer header is cleared or an Azure-specific request path is used.
  - confidence: 0.88
- **[P2] Expose the Azure API version in config:** `extensions/memory-lancedb/index.ts:179-180`
  - body: The patch hard-codes `api-version=2024-02-01`, but memory-lancedb config and manifest schema do not accept an API-version field. Users who need a newer or tenant-required Azure version would have no supported override path.
  - confidence: 0.9
- **[P2] Parse Azure hostnames consistently:** `extensions/memory-lancedb/index.ts:173`
  - body: Checking `baseUrl?.includes(".openai.azure.com")` misses Azure AI Foundry hosts already handled elsewhere in OpenClaw, and it can match path or query text instead of the parsed hostname.
  - confidence: 0.84
