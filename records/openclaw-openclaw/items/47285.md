## Summary

Keep this PR open. Current main still does not provide the native Azure OpenAI memory-lancedb embedding path the PR is trying to add, but the proposed patch is not merge-ready because it only adds constructor defaults and leaves Azure URL shaping, auth suppression, API-version configurability, and host detection incomplete.

## What This Changes

The PR changes the memory-lancedb direct OpenAI-compatible embedding client to detect `.openai.azure.com` base URLs and add Azure `api-key` plus `api-version=2024-02-01` OpenAI SDK defaults.

## Best Possible Solution

Add Azure-aware request shaping in the memory-lancedb embedding path: parse recognized Azure hostnames, normalize resource-root and `/openai/v1` inputs, send only Azure `api-key` auth, build `/openai/deployments/{deployment}/embeddings` requests, expose API-version configuration through config/manifest/docs/tests, and preserve the generic OpenAI-compatible proxy path for non-Azure endpoints.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure memory-lancedb with `embedding.provider: "openai"`, `embedding.apiKey`, and an Azure resource `baseUrl`; current main posts to `/embeddings`, and this PR would still not build Azure's deployment-scoped URL or expose API-version configuration.

## Solution Assessment

No. The constructor-only patch is a useful start, but the maintainable fix should follow the existing parsed-host Azure pattern or use the SDK's Azure client behavior, with explicit config/docs/tests rather than hard-coded defaults on the generic OpenAI client.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build Azure deployment URLs for embeddings:** `extensions/memory-lancedb/index.ts:176-177`
  - body: The Azure branch leaves `baseURL` as the configured resource URL while `embed()` still posts to `/embeddings`. For the resource-root workflow described by this PR, Azure expects `/openai/deployments/{deployment}/embeddings`, so requests will go to the wrong endpoint at runtime.
  - confidence: 0.91
- **[P2] Avoid sending OpenAI bearer auth to Azure:** `extensions/memory-lancedb/index.ts:174-178`
  - body: This still constructs the generic `OpenAI` client with `apiKey`. In the locked OpenAI SDK, that client adds `Authorization: Bearer <key>` before merging `defaultHeaders`, so adding `api-key` leaves Azure requests with the wrong auth shape unless the bearer header is cleared or the Azure client/request path is used.
  - confidence: 0.88
- **[P2] Expose the Azure API version in config:** `extensions/memory-lancedb/index.ts:179-180`
  - body: The patch hard-codes `api-version=2024-02-01`, but memory-lancedb config and manifest schema do not accept an API-version field. Users who need a newer or tenant-required Azure version would have no supported override path.
  - confidence: 0.9
- **[P2] Parse Azure hostnames consistently:** `extensions/memory-lancedb/index.ts:173`
  - body: Checking `baseUrl?.includes(".openai.azure.com")` misses Azure AI Foundry and cognitive-services hosts already handled elsewhere in OpenClaw, and it can match path or query text instead of the parsed hostname.
  - confidence: 0.84
