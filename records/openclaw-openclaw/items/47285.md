## Summary

Keep open: current main still lacks native Azure OpenAI embedding support for memory-lancedb, and this PR contains useful feature work, but the branch is not mergeable as-is because the Azure path still uses the generic OpenAI client shape, hard-codes API versioning, misses host parsing coverage, and lacks real Azure behavior proof.

## What This Changes

The PR adds Azure endpoint detection to memory-lancedb OpenAI embeddings, injects `api-key` and `api-version` constructor defaults, adds mocked test coverage, and adds a changelog entry.

## Best Possible Solution

Implement Azure-aware memory-lancedb embeddings with parsed Azure host suffixes, `AzureOpenAI` or explicit deployment-scoped requests, configurable API versioning in config/manifest/docs, request-shape tests, and redacted live Azure proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: PR head still constructs the generic OpenAI client and posts `"/embeddings"`, while the OpenAI SDK Azure client rewrites deployment endpoints and emits `api-key` auth. No live Azure behavior proof is present for the contributor's environment.

## Solution Assessment

No. Constructor defaults on the generic client are not enough; the maintainable fix should use Azure-aware request shaping and configurable API-version ownership rather than hard-coded constructor options.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Build deployment-scoped Azure embedding requests:** `extensions/memory-lancedb/index.ts:357-363`
  - body: The Azure branch still instantiates the generic OpenAI client, while `embed()` continues to post to `"/embeddings"`. For the resource-root workflow described by this PR, Azure needs `/openai/deployments/{deployment}/embeddings?api-version=...` or the SDK `AzureOpenAI` deployment rewrite, so plain Azure resource URLs can still route incorrectly.
  - confidence: 0.92
- **[P2] Switch Azure requests away from bearer auth:** `extensions/memory-lancedb/index.ts:357-362`
  - body: Passing `apiKey` to the generic `OpenAI` client still creates `Authorization: Bearer ...`; adding `defaultHeaders: { "api-key": apiKey }` adds a second credential form rather than switching auth modes. Use `AzureOpenAI` or explicit Azure request construction so Azure calls send the intended auth shape.
  - confidence: 0.9
- **[P2] Expose Azure API version through config:** `extensions/memory-lancedb/index.ts:363`
  - body: The patch hard-codes `api-version=2024-02-01`, but memory-lancedb config and the manifest do not accept an API-version setting. Users needing a different supported Azure API version would have to fork the plugin.
  - confidence: 0.9
- **[P2] Parse Azure hosts from the URL hostname:** `extensions/memory-lancedb/index.ts:354`
  - body: Substring matching the full `baseUrl` can enable Azure behavior for non-Azure URLs whose path or query contains `.openai.azure.com`, and it misses Azure host families OpenClaw already recognizes such as `.services.ai.azure.com` and `.cognitiveservices.azure.com`. Parse `URL.hostname` and suffix-check the known Azure hosts before changing request behavior.
  - confidence: 0.84
