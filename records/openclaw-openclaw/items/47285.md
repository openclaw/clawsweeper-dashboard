## Summary

Keep open: native Azure OpenAI embeddings support is not implemented on current main, and this PR is still incomplete because it only adjusts constructor options, lacks deployment-scoped request shaping/configurable API version handling, and has no real Azure run proof.

## What This Changes

This PR adds Azure OpenAI detection to memory-lancedb's OpenAI-compatible embedding client, injects api-key/api-version request options, adds a constructor-focused test, and updates the changelog.

## Best Possible Solution

Implement Azure-aware memory-lancedb request shaping with validated Azure hostname parsing, deployment-scoped embeddings URLs or AzureOpenAI, configurable apiVersion in config/manifest/docs, focused tests, and redacted live Azure proof while preserving the generic OpenAI-compatible path.

## Reproduction Assessment

Yes for the PR defects by source inspection, but not by live Azure execution. Configure memory-lancedb with embedding.provider openai, an Azure resource baseUrl, an Azure key, and a deployment/model; the proposed constructor path still uses the generic OpenAI client and /embeddings transport.

## Solution Assessment

No. The current constructor-only patch is not the best way to solve the issue because native Azure support needs endpoint normalization, Azure auth semantics, configurable API versioning, hostname parsing, tests/docs, and runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding URLs:** `extensions/memory-lancedb/index.ts:357-361`
  - body: For the resource-root workflow described here, this still constructs the normal OpenAI client with baseURL set directly and embed() posts to /embeddings. Azure embeddings use /openai/deployments/{deployment-id}/embeddings?api-version=..., so a base URL like https://resource.openai.azure.com will miss the deployment path; normalize the endpoint or use AzureOpenAI.
  - confidence: 0.92
- **[P2] Avoid bearer auth on Azure API-key requests:** `extensions/memory-lancedb/index.ts:356-360`
  - body: new OpenAI({ apiKey, defaultHeaders: {"api-key": apiKey} }) still lets the generic client add Authorization: Bearer, while the SDK's AzureOpenAI override replaces bearer auth with api-key. Suppress Authorization or use AzureOpenAI so Azure API-key calls do not carry conflicting auth.
  - confidence: 0.86
- **[P2] Expose the Azure API version in plugin config:** `extensions/memory-lancedb/index.ts:361`
  - body: The PR hard-codes api-version=2024-02-01 on every Azure request, but the memory config/schema only allow provider, apiKey, model, baseUrl, and dimensions. Azure API version is a required query parameter, so users who need another supported version cannot configure this.
  - confidence: 0.9
- **[P2] Parse Azure hosts by hostname suffix:** `extensions/memory-lancedb/index.ts:352`
  - body: baseUrl?.includes(".openai.azure.com") can match path/query text on non-Azure hosts and misses Azure AI Foundry/cognitiveservices host families already handled elsewhere. Parse new URL(baseUrl).hostname and check the known suffix list before changing auth behavior.
  - confidence: 0.84
