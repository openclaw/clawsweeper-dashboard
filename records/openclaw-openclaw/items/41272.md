## Summary

Keep open. Current main still rejects `timeoutSeconds: 0` in the Control UI cron form and drops explicit zero values from the UI payload path even though the cron API/runtime already accepts zero as no-timeout; because #41275 is an open same-author PR with closing syntax for this issue, the issue should stay open until that PR lands or is closed.

## What This Changes

Allow the Control UI cron form to accept `timeoutSeconds: 0` as no-timeout for agent-turn cron jobs.

## Best Possible Solution

Review and land or replace #41275 with a focused UI fix that accepts zero, preserves explicit zero in the submitted payload, updates controller tests, and aligns validation copy if needed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `validateCronForm` rejects `timeoutSeconds: "0"`, and the existing controller test currently expects that error. I did not run the UI test because this was a read-only review and `pnpm` is unavailable in PATH, but the code path is direct.

## Solution Assessment

Yes, the narrow maintainable fix is in the Control UI controller and tests: allow zero during validation and include explicit zero in the cron payload. The already-open #41275 is the right implementation path to review rather than spawning a separate repair lane.
