## Summary

Keep open. Current main still has the reported Control UI mismatch: the form rejects `timeoutSeconds: 0`, the submit payload omits an explicit zero, and the UI test still asserts zero is invalid while the protocol/runtime paths already accept zero as no-timeout. The issue also has open PR #41275 with closing syntax, so it should remain open until that or an equivalent fix lands.

## What This Changes

The issue requests that the cron job Control UI accept `timeoutSeconds: 0` as no-timeout mode instead of treating it as a validation error.

## Best Possible Solution

Keep the issue open and review/land PR #41275 or a narrow replacement that updates the Control UI validation, preserves an explicitly entered zero in the payload, and adjusts cron controller tests for zero, positive, negative, malformed, and empty timeout inputs while leaving the existing backend/protocol contract intact.

## Reproduction Assessment

Yes. The current source provides a high-confidence reproduction path: entering `"0"` in the agent-turn cron form reaches `validateCronForm`, parses to `0`, and triggers `cron.errors.timeoutInvalid`; the current UI test asserts that behavior.

## Solution Assessment

No, the issue’s one-line proposed fix alone is not the best complete solution. The narrow maintainable fix should also preserve explicit zero in `buildCronPayload` and avoid accepting malformed strings that currently fall back to parsed zero.
