## What This Changes

The issue asks for the cron job UI to accept `timeoutSeconds: 0` as a no-timeout value instead of rejecting it during form validation.

## Best Possible Solution

Land a narrow UI fix that accepts zero in validation, preserves `timeoutSeconds: 0` in the submitted payload, and updates the adjacent controller tests so runtime and UI semantics match.

## Reproduction Assessment

Yes, from source inspection: `validateCronForm` rejects a non-empty timeout value when the parsed number is `<= 0`, and the current controller test explicitly expects `timeoutSeconds: "0"` to be invalid. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the narrow UI alignment is the best fix. Runtime normalization and timeout policy already support zero as no-timeout, so the maintainable path is to fix the UI validation and payload construction plus regression coverage.
