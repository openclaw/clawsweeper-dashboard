## Summary

Keep open: current main and the latest release still reject explicit cron `timeoutSeconds: 0` in the Control UI and omit zero from the submitted payload, while the schema/runtime contract accepts zero as no-timeout. The issue has an open linked implementation PR with closing syntax, so it should remain the tracker until that PR lands or is replaced.

## What This Changes

The issue asks for the cron Control UI to accept explicit `timeoutSeconds: 0` as no-timeout mode for agent-turn cron jobs.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/41275 with a complete controller fix that accepts and preserves explicit zero while keeping invalid timeout input rejected.

## Reproduction Assessment

Yes. Current main sends `timeoutSeconds: "0"` through the `timeout <= 0` validation error path and drops explicit zero from payload construction, while schema and runtime policy already accept zero as no-timeout.

## Solution Assessment

No, not as the issue's validator-only sketch. The controller is the right boundary, but the maintainable fix must also preserve explicit zero in the payload, keep malformed input invalid, and update focused tests plus validation copy.
