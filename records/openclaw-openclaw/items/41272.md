## Summary

Keep open: current main still rejects explicit cron `timeoutSeconds: 0` in the Control UI and drops zero from the submitted payload, while the gateway/runtime contract already accepts zero as no-timeout. The issue is also paired with an open closing PR, so it should stay open until that fix is landed or replaced.

## What This Changes

The issue asks for the cron Control UI to accept `timeoutSeconds: 0` as no-timeout mode and preserve that value when submitting a cron job.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/41275 with a complete controller fix that accepts and preserves explicit zero, rejects malformed and negative timeout input, and updates focused tests plus validation copy/locales.

## Reproduction Assessment

Yes. Source inspection on current main shows `timeoutSeconds: "0"` reaches the `timeout <= 0` validation error path and payload construction only submits positive timeout values, while schema and runtime policy accept zero as no-timeout.

## Solution Assessment
