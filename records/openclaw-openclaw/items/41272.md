## Summary

Keep open: current main and v2026.5.12 still reject or drop explicit `timeoutSeconds: 0` in the Control UI, while the gateway/runtime contract accepts zero as no-timeout. The linked open PR at https://github.com/openclaw/openclaw/pull/41275 is the active closing path, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Cron job forms should accept `timeoutSeconds: 0` as no-timeout and submit that explicit value.

## Best Possible Solution

The end state should be a Control UI fix that accepts and submits explicit zero while keeping malformed and negative timeout input invalid, with aligned tests and i18n copy.

## Reproduction Assessment

Yes. Current main sends entered `0` through the `timeout <= 0` validation error path and only submits positive timeout values, while the gateway schema and runtime timeout policy already accept zero.

## Solution Assessment

No. The issue's validator-only proposed fix is incomplete; the maintainable fix also needs to preserve explicit zero in the payload and keep malformed timeout text invalid, with focused coverage and copy updates.
