## Summary

Keep open: current main and the latest release still reject `timeoutSeconds: 0` in the Control UI, while the gateway protocol/runtime already allow zero as no-timeout. The open linked PR is the current fix candidate, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that the cron job UI rejects `timeoutSeconds: 0`, which should mean no timeout.

## Best Possible Solution

Change the Control UI cron form to accept finite non-negative timeout values and serialize an explicitly entered `0`, with focused controller coverage; keep the existing gateway/runtime no-timeout contract unchanged.

## Reproduction Assessment

Yes. Source inspection shows `validateCronForm` rejects `timeoutSeconds: "0"`, and the adjacent controller test currently expects that error; I did not run a browser/manual repro in this read-only pass.

## Solution Assessment

Yes. Since the shared protocol and runtime already support zero as no-timeout, the narrow maintainable fix belongs in the Control UI validation and payload serialization path rather than in cron runtime policy.
