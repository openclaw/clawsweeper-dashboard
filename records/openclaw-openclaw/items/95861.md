## Summary

The report should stay open: shipped OpenClaw v2026.6.9 promotes Windows Companion v0.6.3, whose browser.proxy implementation derives 127.0.0.1 browser-control from the gateway URL port, while the matching Windows-node fix is merged only in the companion repo and has not been promoted to an OpenClaw stable release asset yet.

## What This Changes

The issue asks OpenClaw to stop Windows Companion browser control from deriving 127.0.0.1:445 from an HTTPS/Tailscale gateway URL and instead use the real browser-control endpoint for the active gateway.

## Best Possible Solution

Promote a stable Windows Companion release containing the BrowserControlPort endpoint resolver, then update any Windows/Tailscale docs needed to explain the override for split or reverse-proxy topologies.

## Reproduction Assessment

Yes from source, though not from a live Windows/Tailscale run here. The shipped v0.6.3 Windows Companion code derives browser-control as gateway URL port plus two and emits the same unreachable-host message reported for 127.0.0.1:445.

## Solution Assessment

Yes, the best owner-boundary fix is in the Windows Companion endpoint resolver, not OpenClaw core's node proxy dispatch. The merged companion PR's explicit BrowserControlPort/managed-tunnel resolver is the right direction, but this issue should stay open until that fix is shipped through OpenClaw's Windows assets.
