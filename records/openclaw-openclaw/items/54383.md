## Summary

Keep open. Current main still forwards the configured compaction reserve budget into Pi summarization, the pinned Pi package still turns that into `Math.floor(0.8 * reserveTokens)`, and the Anthropic Messages transport preserves a positive requested `maxTokens` instead of clamping it to the model cap. The linked same-author PR #54392 remains open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Clamp built-in compaction summarization output budgeting so Anthropic Claude 4.6 models with 1M context do not request `max_tokens` above their 128000 output limit.

## Best Possible Solution

Land PR #54392 or an equivalent maintainer-approved patch that clamps the built-in LLM compaction reserve/output budget against the resolved summarization model output cap before Pi `generateSummary` or Anthropic transport can send `max_tokens`, while preserving provider/plugin compaction behavior and `context1m` header handling. Add regression coverage for an Anthropic Claude 4.6 1M-context setup with `reserveTokensFloor: 300000` proving the generated request cannot exceed 128000 output tokens.

## Reproduction Assessment

Yes. The source-level path is high-confidence: `reserveTokensFloor: 300000` flows into Pi as `reserveTokens`, Pi 0.71.1 computes `maxTokens = Math.floor(0.8 * reserveTokens) = 240000`, and Anthropic transport sends that positive request without applying the 128000 model cap.

## Solution Assessment

Yes. The narrow maintainable fix is to clamp the built-in compaction output budget before it reaches Pi, or equivalently before Anthropic transport sends the request; the existing open PR is the right implementation candidate to review or replace rather than starting a duplicate lane.
