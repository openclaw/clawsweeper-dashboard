## Summary

Keep open. Current main still has the reported built-in compaction path: OpenClaw forwards the configured compaction reserve token budget into Pi summarization, the pinned Pi 0.71.1 package still derives summarization maxTokens as 80% of that reserve, and direct Anthropic Messages preserves any positive requested maxTokens rather than clamping it to the model cap. The same-author closing PR #54392 is still open and unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Anthropic Claude 1M-context compaction can request a summarization max_tokens value above the model output cap and asks OpenClaw to clamp or otherwise respect the model's maxTokens limit.

## Best Possible Solution

Land PR #54392 or an equivalent maintainer-approved patch that clamps the built-in LLM compaction reserve/output budget against the resolved summarization model output cap before Pi generateSummary can request max_tokens, while preserving provider/plugin compaction behavior and Anthropic context1m header handling. Add regression coverage for an Anthropic Claude 4.6 1M-context setup with reserveTokensFloor: 300000 proving the generated request cannot exceed 128000 output tokens.

## Reproduction Assessment

Yes. The source-level path is high-confidence: with reserveTokensFloor/reserveTokens at 300000, current OpenClaw forwards that reserve into Pi, Pi 0.71.1 computes maxTokens as floor(0.8 * reserveTokens), and direct Anthropic Messages sends a positive requested max_tokens without applying the model ceiling.

## Solution Assessment

Yes, the narrow maintainable fix is to clamp the built-in LLM compaction budget before it reaches Pi or before Anthropic transport sends max_tokens. The existing open PR is the right implementation candidate to review or replace rather than starting a duplicate fix lane.
