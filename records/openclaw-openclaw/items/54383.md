## Summary

Keep open. Current main still forwards the large compaction reserve budget into Pi for direct Anthropic summarization, Pi 0.71.1 still turns that into 80% `maxTokens`, and the direct Anthropic Messages transport still sends a positive requested value without clamping it to the model cap; the open same-author PR #54392 remains the implementation candidate.

## What This Changes

The issue requests clamping compaction summarization output tokens for Anthropic 1M-context models so `/compact` does not request `max_tokens` above the model's 128000-token output limit.

## Best Possible Solution

Land PR #54392 or an equivalent maintainer-approved patch that clamps the built-in LLM compaction output budget against the resolved summarization model output cap before Pi or the direct Anthropic transport can send `max_tokens`, with regression coverage for an Anthropic Claude 4.6 1M-context setup using `reserveTokensFloor: 300000`.

## Reproduction Assessment

Yes. The source-level path is high-confidence: `reserveTokensFloor: 300000` remains a 300000 reserve for a 1M context budget, Pi 0.71.1 computes `maxTokens = floor(0.8 * reserveTokens) = 240000`, and direct Anthropic Messages sends that positive request as `max_tokens`.

## Solution Assessment

Yes. The narrow maintainable fix is to clamp the compaction summary budget to the selected model's output cap before the request reaches Pi or before direct Anthropic sends `max_tokens`; broader context-token resolution work such as #71136 does not replace this specific compaction fix.
