## Summary

Keep open. Current main still lacks previous-version rollback after post-update doctor failure, and this PR is useful source work but not mergeable as-is because the rollback is wired into an outdated CLI-only flow and the added guard can run even when the package update itself failed.

## What This Changes

This PR modifies the CLI package-install update path to reinstall `packageName@beforeVersion` after a failing post-update doctor step and rerun doctor on the rolled-back entrypoint.

## Best Possible Solution

Land a current-main replacement that implements previous-version rollback in `runGlobalPackageUpdateSteps` or a narrow shared callback seam so CLI and Gateway package updates behave consistently, gated on a known previous version plus a successful package replacement followed by post-verify doctor failure, with tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. The PR diff gives a high-confidence static reproduction: with `beforeVersion` set, an update step failure, and a later doctor failure, the rollback branch still runs because it only checks `doctorStep.exitCode !== 0 && beforeVersion`.

## Solution Assessment

No. The proposed safety net is directionally useful, but the maintainable fix should live in the shared package-update path and should require a successful update/swap before attempting rollback.
