## Summary

Keep open: the rollback idea is still not implemented on current main, but this PR is not ready because it patches an obsolete CLI-local boundary, misses the shared gateway/Control UI update path, and lacks required real behavior proof.

## What This Changes

The PR adds a rollback attempt to the CLI global package update path when post-update doctor fails and a previous version is known.

## Best Possible Solution

If maintainers want automatic doctor-failure rollback, implement it in `runGlobalPackageUpdateSteps` after confirmed package replacement, cover CLI and gateway callers with regression tests, and keep broader backup/restore policy coordinated with https://github.com/openclaw/openclaw/issues/14526.

## Reproduction Assessment

Yes for source-level review: current main can be traced through `runPackageInstallUpdate` and `runGatewayUpdate` into `runGlobalPackageUpdateSteps`, where a failed post-verify doctor becomes `failedStep` without reinstalling the previous version. I did not run a live global package update in this read-only review.

## Solution Assessment

No. The rollback direction may be useful, but this patch puts it in the wrong boundary and should gate rollback on a successful package replacement before doctor failure.
