## Summary

Keep open: the requested rollback is not implemented on current main and the PR still targets a real update-safety gap, but the branch is not mergeable as-is because it patches the old CLI-local path, misses the gateway/shared-helper path, lacks successful-replacement gating, and has no real behavior proof.

## What This Changes

The PR adds a previous-version global package reinstall and second doctor run after post-update doctor failure in the CLI package-update path.

## Best Possible Solution

Implement any doctor-failure rollback in `runGlobalPackageUpdateSteps` after a successful package replacement, with CLI and gateway regression coverage, after maintainer direction on automatic rollback scope.

## Reproduction Assessment

Yes for source-level review: current main’s shared helper runs the post-verify doctor and returns the failed doctor step without rollback, and `update-runner.test.ts` covers that behavior. I did not run a live global package update in this read-only review.

## Solution Assessment

No. The rollback direction may be useful, but this patch should move to the shared package-update helper, trigger only after successful package replacement followed by failed post-verification, and align with the open safe-update policy discussion.
