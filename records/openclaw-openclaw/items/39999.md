## Summary

Keep open for contributor and maintainer follow-up. Current main still reports a failed post-update doctor without automatic rollback, but this branch patches an older CLI-local boundary, misses the shared package-update helper used by gateway updates, and lacks real behavior proof.

## What This Changes

The PR adds a CLI package-update rollback branch that reinstalls the previous global package version and reruns doctor when post-update doctor fails.

## Best Possible Solution

If maintainers want automatic doctor-failure rollback, implement it in `runGlobalPackageUpdateSteps` after a confirmed successful package replacement, cover both CLI and gateway callers, and coordinate the policy with the safe-update design.

## Reproduction Assessment

Yes for source-level review: current main flows package updates through `runGlobalPackageUpdateSteps`, where a failed post-verify doctor becomes `failedStep` without reinstalling the previous version. I did not run a live global package update in this read-only review.

## Solution Assessment

No. The rollback idea may be useful, but this patch puts it in the wrong boundary and should also gate rollback on a successful package replacement before doctor failure.
