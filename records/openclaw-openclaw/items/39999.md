## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks an automatic rollback after a post-update doctor failure, while the proposed patch has a concrete guard bug and is stale against the newer shared package-update flow.

## What This Changes

This PR adds a rollback attempt to the CLI package-install update path that reinstalls `packageName@beforeVersion` after a failing post-update doctor step and reruns doctor against the rolled-back entrypoint.

## Best Possible Solution

Land a current-main replacement that adds previous-version rollback through `runGlobalPackageUpdateSteps` or a narrow shared callback seam, only after a known previous version plus a successful package replacement/swap followed by post-verify doctor failure. It should rerun doctor with the correct managed-service environment, keep the update result failed for visibility, add focused tests and a changelog entry, and leave the broader backup/restore workflow to #14526.

## Reproduction Assessment

Yes. The proposed diff gives a high-confidence static reproduction: with `beforeVersion` set, an update command failure, and a later doctor failure, the rollback branch still runs because it does not require the package update or swap to have succeeded.

## Solution Assessment

No. The safety net is directionally useful, but the proposed CLI-local patch is not the best current-main solution; the safer path is a shared package-update rollback seam gated on a successful replacement and preserving the newer doctor/service context.
