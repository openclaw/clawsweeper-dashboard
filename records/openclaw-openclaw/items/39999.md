## Summary

Keep open. Current main still lacks automatic rollback after a failed post-update doctor, but this external PR cannot merge as-is because it patches a stale CLI-local path, runs rollback too broadly, and lacks real behavior proof.

## What This Changes

The PR adds a previous-version reinstall rollback attempt when the post-update doctor step fails during global package updates.

## Best Possible Solution

Implement targeted rollback in `runGlobalPackageUpdateSteps` after a successful package replacement followed by failed post-verify doctor, with focused CLI and gateway regression coverage and coordination with the broader safe-update proposal.

## Reproduction Assessment

Yes for source-level review: current main can be traced through `runGlobalPackageUpdateSteps`, which reports a failed post-verify doctor step as an error without rollback. I did not run a live global package update in this read-only review.

## Solution Assessment

No. The rollback direction is reasonable, but the current patch targets a stale CLI-local boundary and should instead live in the shared helper with rollback gated to a successful package replacement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Move rollback into the shared package-update helper:** `src/cli/update-cli/update-command.ts:312`
  - body: Current main routes both CLI and gateway global package updates through `runGlobalPackageUpdateSteps`. Adding rollback only in the CLI command body misses the gateway/Control UI update path and puts the behavior outside the maintained package-update boundary.
  - confidence: 0.88
