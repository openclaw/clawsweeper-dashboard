## Summary

Keep open: current main still lacks automatic previous-version rollback after a failing post-update doctor step, and the PR intent remains useful, but this patch is against an older CLI-local update path and needs a current-main shared-helper repair before merge.

## What This Changes

The PR adds a CLI package-update rollback branch that reinstalls `packageName@beforeVersion` after a failing post-update doctor step and reruns doctor on the rolled-back entrypoint.

## Best Possible Solution

Implement rollback at the shared package-update boundary after a successful replacement/swap plus failing post-verify doctor, cover CLI and gateway paths, add regression tests and changelog, and leave the broader backup/snapshot workflow to #14526.

## Reproduction Assessment

Yes for the review findings: the PR diff statically shows rollback is gated only on doctor failure plus `beforeVersion` and is added to the older CLI-local body. For the broader behavior, current main source and tests show doctor failure returns an error without automatic previous-version rollback, though I did not run a live global update.

## Solution Assessment

No. The rollback safety net is useful, but this patch is not the best mergeable form on current main because the maintained boundary is `runGlobalPackageUpdateSteps`, and rollback should only run after a successful package replacement/swap followed by post-verify doctor failure.
