## Summary

Keep open: current main still lacks automatic rollback after a post-update doctor failure, and this PR is directionally useful, but the branch needs a current-main port plus a tighter rollback guard before it can merge.

## What This Changes

This PR adds a CLI package-install rollback attempt that reinstalls `packageName@beforeVersion` after a failing post-update doctor step and then reruns doctor on the rolled-back entrypoint.

## Best Possible Solution

Land a current-main replacement that adds previous-version rollback at the shared package-update boundary, only after a known previous version and successful package replacement/swap followed by post-verify doctor failure; include rollback doctor context, focused tests, and a changelog entry while leaving the broader backup/restore workflow to #14526.

## Reproduction Assessment

Yes for the review finding: the PR diff statically shows rollback is gated only on doctor failure plus `beforeVersion`, so it can run after a failed update command. For the broader behavior, current main source shows doctor failure returns an error without an automatic previous-version rollback, but I did not run a live global package update in this read-only review.

## Solution Assessment

No. The rollback safety net is useful, but the current PR is not the best mergeable form because it is CLI-local against an older update flow and lacks the successful-replacement guard; a shared helper seam is the narrower maintainable path on current main.
