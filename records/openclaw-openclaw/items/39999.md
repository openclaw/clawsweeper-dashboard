## Summary

Keep this PR open. Current main still does not implement automatic previous-version reinstall rollback after a package-update doctor failure, and the submitted patch is useful but not merge-ready because it is now behind the shared package-update helper refactor and has a concrete rollback guard bug.

## What This Changes

This PR adds a previous-version global package reinstall rollback in the CLI package-update path when post-update doctor fails, then reruns doctor on the restored install when the entrypoint exists.

## Best Possible Solution

Implement previous-version rollback in the shared package-update helper or a narrow shared callback seam so CLI and Gateway package updates behave consistently, with rollback gated on a known previous version and a successful live replacement followed by post-verify doctor failure.

## Reproduction Assessment

Yes. The PR diff gives a high-confidence static reproduction: with beforeVersion set, updateStep failing, and doctorStep failing, the new rollback branch still runs because it ignores updateStep.exitCode.

## Solution Assessment

No. The proposed direction is useful, but the patch should move into the shared package update path and gate rollback on a successful update/swap before doctor failure rather than patching only the older CLI flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate rollback behind a successful update:** `src/cli/update-cli/update-command.ts:312`
  - body: The rollback branch currently runs whenever doctor fails and beforeVersion is known, even if the update command itself failed and the install was never replaced. That can trigger an unnecessary package-manager reinstall of the already-installed version and obscure the real failure; require the update/replacement step to have succeeded before attempting rollback.
  - confidence: 0.89
