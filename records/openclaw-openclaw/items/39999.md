## Summary

Keep open for maintainer follow-up. Current main still lacks doctor-failure rollback, so this is not implemented on main, but the PR is not mergeable as-is because it targets an obsolete CLI-local boundary, misses the shared gateway update path, lacks real behavior proof, and overlaps a broader rollback-policy discussion.

## What This Changes

The PR adds CLI package-update rollback that reinstalls the previous global package version and reruns doctor when the post-update doctor step fails.

## Best Possible Solution

If maintainers want automatic doctor-failure rollback, implement it in runGlobalPackageUpdateSteps after a successful package replacement, cover both CLI and gateway callers, and align the policy with the broader safe-update design.

## Reproduction Assessment

Yes for source-level review: current main can be traced through runPackageInstallUpdate/runGatewayUpdate into runGlobalPackageUpdateSteps, where a failed post-verify doctor becomes failedStep with no previous-version reinstall. I did not run a live global package update in this read-only review.

## Solution Assessment

No. The rollback direction may be useful, but this patch puts it in the wrong boundary and should also gate rollback on a successful package replacement before doctor failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
