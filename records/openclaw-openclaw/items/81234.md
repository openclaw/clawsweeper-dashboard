## Summary

Keep open: current main fixed parts of the cron timeout and cron-run session reuse story, but source still preserves and uses stored channel-shaped `sessionKey` values for isolated cron delivery paths, which conflicts with the documented fresh-session contract and needs maintainer judgment before changing compatibility behavior.

## What This Changes

The issue reports beta cron jobs timing out after `turn-accepted` and Discord DM replies being blocked when isolated cron jobs retain stale direct-message `sessionKey` values.

## Best Possible Solution

Keep the issue open for an owner-approved cron/doctor repair that clears or loudly reports channel-shaped stored keys for isolated jobs while preserving intentional explicit delivery targets and separately triaging any remaining Codex provider timeout symptoms.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full live Discord blockage because no beta host reproduction was run; source inspection does provide a reliable current-main path showing isolated jobs can preserve and use stored channel-shaped `sessionKey` values.

## Solution Assessment

Is this the best way to solve the issue? Unclear: clearing stale keys matches the isolated-session docs, but current tests intentionally preserve creator-session delivery fallback, so maintainers should choose the compatibility boundary before implementation.
