## Summary

Keep open: current main still has a source-level path where an isolated cron job can preserve and later use a stored channel-shaped sessionKey for delivery or failure notification, while the OpenAI/Codex timeout portion is only partly covered by a separate live follow-up.

## What This Changes

The issue reports beta cron timeouts after turn acceptance and Discord DM blockage when isolated cron jobs retain stale direct-message session keys.

## Best Possible Solution

Split the fix into a narrow isolated-cron sessionKey normalization/doctor path for stale channel keys, plus a separate provider/runtime diagnostics follow-up for OpenAI/Codex timeouts.

## Reproduction Assessment

No for the full incident: I did not reproduce a live Discord DM blockage or beta host timeout. The stale sessionKey half is source-reproducible because current main still preserves and uses job.sessionKey in isolated cron delivery paths.

## Solution Assessment

Unclear: clearing or warning on stale channel-shaped keys is likely the right direction, but the exact boundary needs maintainer judgment because current tests intentionally allow creator sessionKey fallback for isolated delivery.
