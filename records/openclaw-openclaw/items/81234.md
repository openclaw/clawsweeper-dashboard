## Summary

Keep open: current main still has source-level paths where an isolated cron job can preserve and use a stored channel-shaped `sessionKey`, while the OpenAI/Codex `turn-accepted` timeout portion is not proven fixed or owned by a narrower merged change.

## What This Changes

The issue asks OpenClaw to prevent isolated cron jobs from reusing stale Discord DM session keys and improve cron timeout/status diagnostics for OpenAI/Codex-backed scheduled jobs.

## Best Possible Solution

Split this into a focused cron isolation repair for stale channel-shaped `sessionKey` values plus doctor/startup detection, and track OpenAI/Codex timeout diagnostics as a separate runtime investigation if it still reproduces.

## Reproduction Assessment

Yes, at source level: current docs promise fresh isolated cron sessions, while current code preserves and reuses stored `job.sessionKey` in delivery/notification paths. No live Discord beta reproduction was run, so the full lane-blocking incident remains unproven end to end.

## Solution Assessment

Unclear: clearing or warning on stale channel-shaped `sessionKey` values is plausible, but the maintainable fix must distinguish intentional current-session delivery context from execution/session ownership. Provider timeout diagnostics should be solved separately from the cron isolation bug.
