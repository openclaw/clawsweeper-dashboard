## Summary

Keep open: current main has addressed parts of the cron/Codex timeout story, but source still allows an isolated cron job to preserve and use a channel-shaped stored `sessionKey` in delivery and notification paths, which conflicts with the documented isolated-session contract.

## What This Changes

The issue asks OpenClaw to prevent isolated cron jobs from retaining live Discord DM session keys and to improve diagnostics/routing for OpenAI/Codex cron timeouts.

## Best Possible Solution

Keep the run-scoped cron execution behavior, then make an owner-approved cron/doctor repair that clears or loudly reports channel-shaped stored keys for isolated jobs without breaking intentional explicit delivery targets.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Discord repro was run; source inspection does give a reliable path showing stored `sessionKey` preservation and delivery use on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: clearing stale keys matches the isolated cron docs, but current tests intentionally preserve creator-session delivery fallback, so maintainers should choose the compatibility boundary before implementation.
