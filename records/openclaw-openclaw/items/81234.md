## Summary

Keep open: current main still has source-level paths where isolated cron jobs can preserve and consume a stored channel-shaped `sessionKey` for delivery, preview, notification, and task ownership, while the OpenAI/Codex timeout portion overlaps a separate open runtime investigation.

## What This Changes

The issue reports beta cron jobs retaining Discord DM-shaped session keys and timing out after `turn-accepted`, causing stale cron state to affect Discord DM responsiveness and cron reliability.

## Best Possible Solution

Split this into a narrow cron-session repair for isolated jobs, a doctor/startup warning or migration for legacy channel-shaped `sessionKey` values, and a separate provider/runtime diagnostics path for OpenAI/Codex `turn-accepted` timeouts.

## Reproduction Assessment

No, not for the full reported Discord blockage: I did not run a live beta host or Discord DM path. Source inspection does provide a credible current-main path where isolated jobs preserve and consume stored channel-shaped `sessionKey` values outside the primary model-run key.

## Solution Assessment

Unclear: the reporter's suggested clearing/warning direction is plausible, but the best maintainable fix should preserve intentional current-chat delivery fallback while blocking stale direct-message lane contamination.
