## Summary

Keep open: current main has fixed parts of the cron/Codex timeout path, but isolated cron jobs can still preserve and use a channel-shaped stored `sessionKey` for delivery and notification context, while doctor normalization currently preserves that stale key instead of clearing or warning on it.

## What This Changes

The issue asks OpenClaw to prevent isolated cron jobs from retaining live channel session keys, improve `turn-accepted` timeout diagnostics, clarify Codex/OpenAI model routing, and distinguish stale cron failures from active ones.

## Best Possible Solution

Keep the scoped cron execution behavior, then make an owner-approved cron/doctor repair that clears or loudly reports channel-shaped stored keys for isolated jobs while preserving any intentionally explicit delivery target semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not end to end: no live Discord DM repro was run, but current source gives a clear path where isolated cron delivery and doctor migration preserve and consume a channel-shaped stored `sessionKey`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide the compatibility boundary, because clearing stale keys matches the isolated-session docs but current tests intentionally preserve creator-session fallback for delivery.
