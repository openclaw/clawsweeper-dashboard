## Summary

Keep open: current main still has source-level paths where an isolated cron job can preserve a stored channel-shaped `sessionKey` and feed it into delivery or failure-notification routing, while the OpenAI/Codex timeout portion remains a separate live beta reliability investigation.

## What This Changes

The issue reports beta cron reliability failures: isolated cron jobs retaining Discord DM-shaped session keys, Discord DM responsiveness blocking, and OpenAI/Codex cron turns timing out after acceptance.

## Best Possible Solution

Split the work into a narrow cron isolation repair for legacy channel-shaped `sessionKey` values plus a doctor/startup warning or fix, while keeping the OpenAI/Codex timeout and fallback-trace investigation tied to the related beta timeout thread.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full Discord DM blockage on a live beta host; source inspection does show that isolated cron jobs can preserve and pass a stored `sessionKey` into delivery and failure-notification routing on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: clearing or warning on stale channel-shaped keys is likely right, but the final fix should preserve deliberate `session:<id>` delivery semantics and avoid conflating the separate OpenAI/Codex timeout investigation.
