## Summary

Keep open. The proposed polling/backoff fix is directionally valid, but the branch is stale against current main: it edits the old core GitHub Copilot auth file while the active implementation now lives in the bundled GitHub Copilot plugin, and the advertised `--wait` flag is not wired through the current provider-auth context.

## What This Changes

The PR adds a `--wait` option to the GitHub Copilot login command and changes GitHub device-flow polling/backoff timing in the old core GitHub Copilot auth provider path.

## Best Possible Solution

Port the behavior into the current plugin-owned GitHub Copilot auth implementation: delay the first token request until the device-code interval has elapsed, update subsequent polling after `slow_down` according to GitHub's returned/new interval, expose a typed direct-login wait option through provider auth, render copyable URL/code plus confirmation through the plugin prompter, and cover it with focused fake-fetch/fake-timer tests plus docs.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main posts the first access-token request before sleeping and still lacks `--wait`; a fake-fetch/fake-timer unit test against `extensions/github-copilot/login.ts` can reproduce the immediate poll and `slow_down` interval behavior without live GitHub credentials.

## Solution Assessment

No. The polling/backoff direction is correct, but this PR is not the best current implementation because it edits the old core provider path and does not plumb `--wait` into the active plugin-owned auth flow.
