## Summary

Keep open: the reported GitHub device-flow problem is valid, but this PR is stale against current main's plugin-owned GitHub Copilot auth path. Main still posts the first token request before sleeping and has no `--wait` option, while the PR edits a removed core provider path and passes a wait option that the current provider-auth surface does not carry.

## What This Changes

The PR adds a `--wait` option to the GitHub Copilot login shortcut and changes device-flow polling/backoff timing in the old core GitHub Copilot auth implementation.

## Best Possible Solution

Move the timing/backoff behavior into `extensions/github-copilot/login.ts`, expose a typed wait/confirmation option through the provider-auth command path, preserve the #73290 GUI/RPC prompter flow, and add focused tests, docs, and changelog coverage without reintroducing core-owned GitHub Copilot auth.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main posts the access-token request before sleeping, handles `slow_down` with a fixed non-persistent delay, and does not register `--wait`; a fake-fetch/fake-timer test can reproduce this without live GitHub credentials.

## Solution Assessment

No. The polling/backoff direction is correct, but the proposed patch is not the best current implementation because it targets a removed core path and does not add typed option plumbing into the active plugin-owned auth flow.
