## Summary

Keep open: the PR targets a real Telegram menu-limit bug, but the patch moves alias suppression into the shared native-command registry, which would change non-Telegram/native command behavior and needs rework plus real Telegram proof before merge.

## What This Changes

The PR collapses native alias specs into a new `aliases` field and adds separate Telegram alias handlers so aliases remain callable while omitted from the Bot API menu payload.

## Best Possible Solution

Keep the shared registry behavior intact and make the Telegram `setMyCommands` payload canonical-first or alias-filtered while still registering alias handlers from the full native command catalog.

## Reproduction Assessment

Yes for source-level reproduction: current main expands `nativeAliases` into separate native specs, and Telegram caps that expanded list before `setMyCommands`. I did not run a live Telegram `getMyCommands` probe in this read-only review.

## Solution Assessment

No: the PR fixes the Telegram symptom by changing shared registry semantics that other providers rely on. The safer solution is a Telegram-specific menu payload change with alias dispatch preserved and regression tests for Telegram plus unaffected non-Telegram aliases.
