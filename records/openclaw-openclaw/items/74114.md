## Summary

Keep open. The PR targets a real linked bug in #72021, but the current patch conflicts with existing memory-core tests and docs that deliberately allow daily and grounded evidence to satisfy deep-promotion thresholds.

## What This Changes

The PR changes memory-core short-term promotion so `minRecallCount` is checked against `recallCount` alone in ranking and apply filtering while aggregate `signalCount` remains used for scoring.

## Best Possible Solution

Define the promotion contract explicitly: separate organic user recalls from automated daily/session evidence where #72021 needs protection, while preserving or deliberately retiring daily and grounded deep-promotion paths with matching tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with this diff applied, the current tests expecting `recallCount: 0, dailyCount: 3` and `groundedCount: 3` candidates to pass default gates would fail the new recall-only threshold.

## Solution Assessment

No, not as-is. A recall-only gate is a plausible direction for #72021, but the narrow patch silently contradicts supported daily and grounded promotion semantics instead of reconciling or intentionally retiring them.
