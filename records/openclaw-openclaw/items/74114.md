## Summary

Keep open. The PR addresses a real source-level ambiguity, but it is not mergeable as-is because a strict recall-only gate would disable daily-ingestion and grounded-backfill promotion paths that current tests and docs still treat as valid, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes memory-core short-term promotion gates so `minRecallCount` checks `recallCount` instead of aggregate `signalCount`.

## Best Possible Solution

Define one memory-core promotion signal contract for organic recalls, daily/session evidence, and grounded evidence, then update implementation, regression tests, docs, and changelog together.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main aggregates recall, daily, and grounded counts for the gate, and current fixtures show daily-only and grounded-only candidates clearing default promotion paths; I did not run tests in this read-only review.

## Solution Assessment

No, not as-is. A recall-only gate addresses the reported false eligibility, but it silently disables current daily and grounded promotion behavior unless maintainers intentionally retire or replace those contracts.
