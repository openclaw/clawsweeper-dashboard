## Label Justifications

- `P1`: The linked issue is a user-facing isolated cron completion delivery failure that can lose final controller messages.
- `merge-risk: 🚨 message-delivery`: The diff changes final completion routing for cron/background announcements and can still drop explicit-channel messages if merged as-is.
- `merge-risk: 🚨 session-state`: The PR adds process-local route state keyed by session identity, and the current key/lifetime does not match requester run state.

## Summary

Keep the PR open, but it is not merge-ready: the proposed lease is registered under a key that the completion callers do not use for default cron runs, and it is retired before delayed background completions are guaranteed to have delivered.

## What This Changes

The PR adds an in-memory delivery route lease store for isolated cron runs, wires announce-origin lookup through that store, registers/retires leases from cron execution, and adds unit tests for the store.

## Best Possible Solution

Register the route lease under the actual requester run key and retire it from the shared background-task completion/delivery lifecycle only after all supported completions settle, with delayed-completion regression coverage and live route proof.

## Reproduction Assessment

Source-reproducible: current main uses a distinct runSessionKey for default cron execution, while the PR registers leases under agentSessionKey and later lookup uses requesterSessionKey.

## Solution Assessment

No. The lease-store direction matches the maintainer-requested shape, but this implementation is not the best fix because its keying and retirement lifecycle do not match the completion callers it is meant to support.
