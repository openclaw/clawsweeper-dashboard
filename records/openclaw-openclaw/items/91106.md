## Summary

Keep open: current main and the latest release still have the provider-layer OpenAI Completions compat detector set `requiresReasoningContentOnAssistantMessages` only for DeepSeek, while the agent transport compat path already treats Xiaomi as requiring the same replay field. The issue is a narrow provider-compatibility bug with a source-level reproduction and official Xiaomi docs confirming the 400 failure mode for missing `reasoning_content`.

## What This Changes

The issue asks OpenClaw to make provider-layer OpenAI Completions compatibility detection set `requiresReasoningContentOnAssistantMessages` for Xiaomi-native endpoints, matching the agent transport path.

## Best Possible Solution

Centralize or mirror the Xiaomi-native OpenAI Completions compat default so both direct provider and agent transport paths agree, with regression coverage for Xiaomi direct-provider replay.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction: direct provider `detectCompat` leaves the Xiaomi flag false, and `convertMessages` only emits `reasoning_content` when that flag is true. I did not run a live Xiaomi-key request.

## Solution Assessment

Yes, a small compat-default fix is the best direction; prefer reusing or aligning with `src/agents/openai-completions-compat.ts` rather than asking Xiaomi users to add per-model config that bundled Xiaomi docs intentionally omit.
