## Summary

Keep open: current main still treats startup sidecars as part of Gateway readiness and connection gating, while the requested core-ready milestone would change the Gateway/TUI startup contract rather than close a stale or already-implemented item.

## What This Changes

The issue requests an explicit Gateway core-ready boundary so TUI/core access comes up before channel/plugin sidecars and bundled-plugin dependency checks finish.

## Best Possible Solution

Define the desired core-ready semantics first, then implement a narrow Gateway/TUI boundary that exposes core control surfaces while representing channels, plugin services, and dependency diagnostics as degraded sidecar state.

## Reproduction Assessment

No, not for the reported downstream wedge: I did not run a macOS/Node 24 recovery scenario. Source and tests do provide a clear current-main path showing startup sidecars gate readiness, WebSocket attach, and several TUI-needed methods.

## Solution Assessment

Unclear: the requested direction is plausible, but it changes an existing documented/tested readiness contract. The best next step is maintainer agreement on which surfaces are core-ready-safe before changing Gateway/TUI behavior.
