## Label Justifications

- `P1`: The issue covers Gateway/TUI startup wedges that can block real users from attaching to the core runtime during restart recovery.
- `impact:crash-loop`: The reported failure mode is process-level startup unavailability or apparent hangs while channel/plugin sidecars run.

## Summary

Keep open: current main and the latest release still gate normal Gateway readiness on startup sidecars, while the deferred path exists only for containment flags. The related implementation PR remains open and conflicting, so there is no merged or safe canonical fix to close against.

## What This Changes

Request that Gateway/TUI expose a core-ready state before optional channel/plugin sidecars and bundled-plugin dependency work can delay startup.

## Best Possible Solution

Define and document a core-ready versus sidecars-ready Gateway contract, expose core/TUI state before optional sidecars finish, and report sidecar failures as degraded status with retryable method-level gating.

## Reproduction Assessment

Yes, at source level: current main only defers startup sidecars for containment flags, and normal readiness remains false until sidecars finish. I did not run a live restart reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable direction is an explicit core-ready/sidecars-ready split, not another emergency skip flag or a narrow timing-only workaround.
