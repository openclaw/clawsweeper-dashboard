## Summary

Keep this open: current main and v2026.6.6 still tie normal gateway readiness to startup sidecars, while the related implementation PR remains open and conflicting rather than a landed canonical fix.

## What This Changes

The issue requests an explicit gateway core-ready milestone so TUI/control surfaces can attach before channel and bundled-plugin sidecars finish or fail.

## Best Possible Solution

Define and land a maintainer-approved core-ready versus sidecar-ready contract in the gateway, with TUI/status surfaces showing degraded sidecars asynchronously and regression coverage for normal startup without containment flags.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: start the gateway without containment flags and the default post-attach runtime awaits startup sidecars before readiness completes.

## Solution Assessment

Unclear until maintainers choose the readiness contract. The best direction is not another emergency skip flag; it is an explicit core-ready/sidecars-ready split with narrow method gating and operator-visible degraded sidecar status.
