## Summary

Keep open: current main still treats channel/plugin sidecars as part of Gateway readiness and WebSocket/TUI connection gating, so the requested core-ready boundary is not already implemented and needs maintainer startup-contract review.

## What This Changes

Request an explicit Gateway/TUI core-ready milestone that comes up before channel startup, plugin services, bundled-plugin dependency diagnostics, or other sidecars can block attachment.

## Best Possible Solution

Define the core-ready contract first, then implement an additive Gateway/TUI readiness model that exposes core control surfaces while representing channels, plugin services, and dependency diagnostics as sidecar degraded or pending state.

## Reproduction Assessment

No for the reported downstream wedge: I did not run a macOS/Node 24 recovery scenario. Yes for the current-main source shape: readiness, WebSocket connect, and selected Gateway methods are still gated on startup sidecars.

## Solution Assessment

Unclear: the requested direction is plausible, but it changes documented and tested readiness behavior. The maintainable path is to decide which Gateway/TUI surfaces are core-ready-safe before changing startup gating.
