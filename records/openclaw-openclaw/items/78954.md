## Summary

Keep open: current main still gates Gateway readiness, WebSocket attachment, and several TUI-needed methods on startup sidecars. The related ACPX readiness work is partly fixed elsewhere, but the broader core-ready/degraded-sidecar contract requested here is not implemented and needs maintainer product/API agreement.

## What This Changes

The issue requests an explicit Gateway/TUI core-ready milestone before channel/plugin sidecars and bundled-plugin dependency checks can block startup.

## Best Possible Solution

Define the core-ready protocol/readiness semantics first, then implement a narrow Gateway/TUI boundary that exposes core control surfaces while reporting channels, plugin services, and dependency diagnostics as asynchronous degraded sidecar state.

## Reproduction Assessment

No high-confidence live reproduction of the reported downstream macOS wedge was established. Source inspection does show a current-main path where startup sidecars gate `/readyz`, WebSocket attach, and selected Gateway methods.

## Solution Assessment

Unclear until maintainers decide the startup contract. The requested direction is plausible, but it changes current docs/tests that intentionally treat startup sidecars as readiness and connection gates.
