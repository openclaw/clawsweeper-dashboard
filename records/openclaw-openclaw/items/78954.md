## Label Justifications

- `P1`: The issue concerns Gateway/TUI startup becoming unavailable or wedged for real users during normal channel/plugin sidecar startup.
- `impact:crash-loop`: The reported impact is process-level availability: startup hangs or delayed sidecars can prevent Gateway/TUI use after restart.

## Summary

Keep open: current main has partial containment behavior, but normal Gateway startup still treats sidecar completion as the point where startup-gated RPC/connect surfaces become available, so the requested default core-ready boundary is not implemented. The related open PR and lazy-channel issue cover narrower pieces, not the whole startup contract.

## What This Changes

Request an explicit Gateway/TUI core-ready startup milestone before channel startup, bundled-plugin dependency work, plugin services, and other sidecars can delay or block user attachment.

## Best Possible Solution

Define a maintainer-approved Gateway startup contract where core HTTP/WebSocket/RPC attachment becomes available before optional channel/plugin sidecars, while full sidecar readiness remains visible as degraded or pending state through diagnostics and `/readyz` semantics.

## Reproduction Assessment

Yes at source level, but not from a live restart in this read-only review. Current main shows normal startup keeps Gateway startup pending until sidecars finish, and tests explicitly expect readiness/connect gating while `startup-sidecars` is pending.

## Solution Assessment

Unclear until maintainers choose the startup contract. The direction is aligned with startup reliability, but the safest fix likely needs coordinated Gateway readiness, WebSocket/RPC gating, TUI attach behavior, and sidecar diagnostics rather than a narrow one-file patch.
