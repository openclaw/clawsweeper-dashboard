## Label Justifications

- `P1`: The issue covers gateway/TUI startup wedges that can block real users from attaching to the core runtime during restart recovery.
- `impact:crash-loop`: The reported failure mode is process-level startup unavailability or apparent hangs while channel/plugin sidecars run.

## Summary

Current main still does not provide the requested normal-mode core-ready boundary: the gateway has a deferred-sidecar path, but the CLI only enables it for containment flags and the default server path still waits for sidecars before reporting ready. The issue should stay open for maintainer product review because an open related PR exists but is conflicting and not a safe canonical fix yet.

## What This Changes

Gateway/TUI startup should expose core readiness before channel and bundled-plugin sidecars can block attachment, with sidecar failures surfaced as degraded status instead of startup wedges.

## Best Possible Solution

Define the core-ready versus sidecars-ready contract in gateway runtime and docs, make normal startup expose core/TUI state before optional sidecars finish, and keep sidecar failures visible through degraded status and retryable method-level gating.

## Reproduction Assessment

Yes, at source level: current main only defers startup sidecars when skip-channel/provider containment flags are set, and the default post-attach runtime awaits sidecars before ready. I did not run a live restart reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. The best direction is not another emergency skip flag, but an explicit core-ready/sidecars-ready split with narrow method gating and operator-visible degraded sidecar status.
