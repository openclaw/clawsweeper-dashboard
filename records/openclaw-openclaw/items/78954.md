## Summary

Keep this open: current main and the latest release still make normal gateway readiness wait on startup sidecars, and the related implementation PR is still open and conflicting rather than a landed fix.

## What This Changes

The issue asks for an explicit gateway core-ready startup boundary before channel, plugin, bundled-plugin, dependency-check, and TUI-blocking sidecar work can delay attachment.

## Best Possible Solution

Define and land a maintainer-approved core-ready versus sidecar-ready gateway contract, with TUI/status surfaces showing degraded sidecars asynchronously and regression coverage for normal startup without containment flags.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the current behavior: start the gateway without containment flags and the default post-attach runtime awaits startup sidecars before readiness completes; I did not run a live restart reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the readiness contract. The best direction is not another emergency skip flag, but an explicit core-ready/sidecars-ready split with narrow method gating and operator-visible degraded sidecar status.
