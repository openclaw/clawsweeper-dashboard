## Summary

Keep this open: current main and the latest release still make normal gateway readiness wait on startup sidecars, while the related implementation PR remains open and conflicting rather than a landed fix.

## What This Changes

The issue requests an explicit gateway core-ready milestone so TUI and gateway access are available before channel and plugin sidecars can delay or wedge startup.

## Best Possible Solution

Define and land a maintainer-approved core-ready versus sidecar-ready gateway contract, with TUI/status surfaces showing degraded sidecars asynchronously and regression coverage for normal startup without containment flags.

## Reproduction Assessment

Yes, source-level: start the gateway normally without OPENCLAW_SKIP_CHANNELS or OPENCLAW_SKIP_PROVIDERS and a slow startChannels/startPluginServices path keeps readiness on the sidecar wait. I did not run a live packaged restart in this read-only review.

## Solution Assessment

Unclear until maintainers choose the readiness contract. The best direction is an explicit core-ready/sidecars-ready split with narrow method gating and degraded sidecar status, not another emergency skip flag.
