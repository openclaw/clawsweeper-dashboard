## Summary

Keep open: current main still treats startup sidecars as part of Gateway readiness and WebSocket/method availability, while the requested `core-ready` milestone would change a documented and tested Gateway/TUI startup contract.

## What This Changes

Request an explicit Gateway/TUI `core-ready` milestone so channel, plugin, bundled-plugin dependency, and other sidecar startup work cannot block core gateway/TUI attachment.

## Best Possible Solution

Define the desired core-ready semantics and safe core surfaces first, then split Gateway/TUI readiness from sidecar status with degraded async reporting and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction for the downstream macOS/Node 24 wedge was established. Source inspection does show a current-main path where sidecars keep `/readyz`, WebSocket connect, and startup-gated methods unavailable.

## Solution Assessment

Unclear: the direction is plausible, but it changes current documented/tested readiness semantics. Maintainers should first decide which Gateway/TUI surfaces are core-ready-safe before implementation.
