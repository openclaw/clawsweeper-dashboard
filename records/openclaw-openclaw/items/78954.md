## Summary

Keep open: current main still makes startup sidecars part of Gateway readiness and WebSocket attachment, so the requested core-ready-before-sidecars contract is not implemented and is not a stale cleanup close.

## What This Changes

The issue requests an explicit Gateway/TUI core-ready milestone before channel activation, bundled-plugin dependency diagnostics, or plugin sidecars can block startup or TUI attachment.

## Best Possible Solution

Define the core-ready semantics and client-safe method set first, then expose core control surfaces while representing channels, bundled plugins, dependency diagnostics, and plugin services as degraded sidecar state.

## Reproduction Assessment

No, not for the exact downstream wedge: I did not reproduce the macOS/Node 24 recovery scenario. Source and tests do show the current-main path where startup sidecars gate readiness, WebSocket attach, and several TUI-needed methods.

## Solution Assessment

Unclear: the requested direction is plausible, but it changes the current documented and tested readiness contract. Maintainers should decide which Gateway/TUI operations are safe at core-ready before implementation.
