## Summary

Keep open: current main still makes startup sidecars part of Gateway readiness and WebSocket/RPC attachment, while the requested core-ready milestone would change the Gateway/TUI startup contract rather than close a stale or already-implemented report.

## What This Changes

Gateway/TUI startup should expose a core-ready boundary before channel startup, plugin sidecars, bundled-plugin dependency work, or sidecar failures can block core attachment.

## Best Possible Solution

Define the desired core-ready semantics first, then split Gateway/TUI core surfaces from channel/plugin sidecar readiness with degraded sidecar status, updated protocol/docs, and focused tests for HTTP readiness, WebSocket attach, startup-gated RPCs, and TUI history/model loading.

## Reproduction Assessment

No, not for the reported downstream wedge: I did not run the macOS/Node 24 recovery scenario. Source and tests do provide a clear current-main path showing sidecars gate readiness, WebSocket connect, and startup-gated Gateway methods.

## Solution Assessment

Unclear: the requested direction is plausible, but it changes an existing documented and tested readiness contract. The narrowest safe next step is maintainer agreement on which surfaces are core-ready-safe before implementation.
