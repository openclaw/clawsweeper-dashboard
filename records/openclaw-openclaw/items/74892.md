## Summary

Keep this PR open. Current main still has the response-flag bug this branch targets: native subagent model resolution always falls back to a model, and the accepted response reports `modelApplied: true` whenever that resolved model is persisted. The code direction is narrow, but the PR still needs the required changelog entry and focused no-override regression coverage before merge.

## What This Changes

The PR changes native subagent planning and direct spawn response handling so `modelApplied` is derived from a non-empty `sessions_spawn.model` override rather than any resolved model.

## Best Possible Solution

Land a small update that preserves the already-fixed runtime model persistence path, makes the `modelApplied` response contract explicit with no-override regression coverage, and records the user-visible response change in `CHANGELOG.md`. After that merges, the linked runtime report can close through the PR.

## Reproduction Assessment

Yes. Source tracing on current main gives a high-confidence unit reproduction: call native `sessions_spawn` or `resolveSubagentModelAndThinkingPlan` without a `model` override and it still reports `modelApplied` true because model resolution falls back to a default.

## Solution Assessment

Yes, if `modelApplied` means the caller supplied a non-empty `sessions_spawn.model`; checking `modelOverride?.trim()` is the narrowest maintainable fix. If maintainers want the field to mean any resolved child model, the safer path is to document or rename the field instead of landing this semantic change.
