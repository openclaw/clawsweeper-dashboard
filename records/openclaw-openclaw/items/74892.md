## Summary

Keep open. Current main has the runtime model-persistence path and focused coverage for explicit subagent model selection, but it still reports `modelApplied` from any resolved/default model, so the PR's caller-override-only response semantics are not already implemented on main. The diff is narrow and appears functionally sound for that contract, but it still needs the required user-facing changelog entry before landing.

## What This Changes

The PR changes native sub-agent model planning and direct spawn response handling so `modelApplied` is true only when the caller passes a non-empty `model` override.

## Best Possible Solution

Land a small update that keeps current-main runtime model persistence intact, makes the public `modelApplied` semantics explicit through focused regression coverage, and records the user-visible response change in the active changelog.

## Reproduction Assessment

Yes for the PR-level flag behavior: current main's resolver always returns a model and direct spawn sets `modelApplied` true for any resolved model, so a native spawn without `model` still reports true. No live provider reproduction was run for the linked runtime report, but current main has unit-level persistence coverage for the runtime path.

## Solution Assessment

Yes, if the intended contract is that `modelApplied` means the caller supplied a non-empty `sessions_spawn.model`; deriving it from `modelOverride?.trim()` is the narrowest code change. If maintainers want the field to mean any selected child model, this PR should not land without renaming or documenting the response semantics.
