## Summary

Keep this PR open: it uses the right shared bounded-reader primitive for successful Voyage batch status/error bodies, but the same untrusted response paths still call the existing non-OK handler first, which buffers provider error bodies with `Response.text()` before the new cap can run.

## What This Changes

The PR routes Voyage embedding batch status JSON and batch error-file content through bounded response readers and adds tests for overflow, under-cap parsing, and empty error-file behavior.

## Best Possible Solution

Complete Voyage batch hardening by bounding both success bodies and non-OK diagnostic bodies with existing response-limit helpers, preserve the error-file fail-soft behavior, and add focused overflow proof for both paths.

## Reproduction Assessment

Yes. Source inspection shows a non-OK Voyage batch status or error-file response still reaches `assertVoyageResponseOk`, which buffers the whole body with `await res.text()` before the PR's bounded success reader runs.

## Solution Assessment

No. The shared bounded reader is the right primitive, but the best fix also bounds the non-OK diagnostic bodies on the same untrusted provider response surface.
