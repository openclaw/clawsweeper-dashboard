## Summary

Keep this PR open, but it is not merge-ready: the root public-asset fallback is a useful candidate for the linked basePath manifest problem, yet the implementation bypasses the existing non-read method fallthrough boundary for configured-basePath deployments.

## What This Changes

The PR changes the Gateway Control UI HTTP handler to serve known root-mounted public assets outside a configured basePath and adds GET regression coverage for those assets.

## Best Possible Solution

Land one canonical fix that prevents the browser asset failures while preserving the gateway read-method boundary for any fallback static-asset route.

## Reproduction Assessment

Yes. Source inspection shows current main does not handle root public assets outside a configured basePath, and the PR-introduced regression is visible by tracing the non-read method classifier into the new carveout.

## Solution Assessment

No. The fallback is plausible, but as written it is not the best fix because it skips the existing read-method boundary; the safer version gates the carveout to read methods, while maintainers may still prefer an HTML/client URL rewrite candidate.
