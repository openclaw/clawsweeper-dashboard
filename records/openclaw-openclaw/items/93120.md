## Summary

Keep open: current main and v2026.6.9 still route the reported Gemini `current quota` plus `/rate-limits` 429 away from same-model retry, and the two related implementation PRs were closed unmerged with maintainer guidance that the issue remains valid.

## What This Changes

Gemini 429 messages containing `current quota` and the Gemini `rate-limits` URL should be treated as short-window rate limits instead of immediately falling back.

## Best Possible Solution

Keep this issue open for a provider-scoped classifier fix backed by real Gemini short- and long-window payload evidence, and defer the retry-count config knob to the broader retry-policy decision.

## Reproduction Assessment

Yes, at source level: current `main` treats `current quota` as long-window unless explicit per-minute/RPM/TPM text or a short Retry-After is present, and the reported Gemini URL is not a short-window token. I did not run a live Gemini quota-exhaustion scenario.

## Solution Assessment

No for the original config-only request: increasing the retry count does not fix the classifier path, and a generic `rate-limits` URL regex was closed unmerged as too broad. The safer solution is a provider-scoped classifier contract with real Gemini payload evidence, plus a separate config-policy decision if needed.
