## Summary

Keep open: current main and latest release still hard-code the same-model rate-limit retry cap at 3, and no existing open item cleanly owns this exact retry-count configurability request. The request is concrete, but it adds a new provider-routing config surface and overlaps broader provider retry/cooldown design work, so it needs maintainer product review before implementation.

## What This Changes

The issue requests making the hardcoded same-model rate-limit retry cap configurable, for example via `auth.cooldowns.maxSameModelRateLimitRetries`.

## Best Possible Solution

Define one explicit provider rate-limit retry policy that covers retry count, delay, Retry-After handling, and interaction with profile rotation/fallback, rather than exposing only this constant as an isolated override.

## Reproduction Assessment

Yes at source level: the retry limit remains a hardcoded constant and the cited runtime path still consumes it. No live Gemini reproduction was run in this read-only review.

## Solution Assessment

No for the exact proposed knob as an automatic fix: it would add a new config surface adjacent to existing rate-limit profile-rotation controls and broader provider retry policy work. A unified provider-rate-limit policy is the safer direction.
