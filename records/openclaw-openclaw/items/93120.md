## Summary

Keep open: current main and the latest release still misclassify the reported Gemini 429 text as long-window quota, and a maintainer closed the linked config PR as the wrong fix while explicitly leaving this issue open for a narrow classifier regression.

## What This Changes

The issue asks OpenClaw to treat Gemini TPM/RPM 429 rate-limit responses as retryable same-model short-window throttles instead of falling through to profile rotation or model fallback.

## Best Possible Solution

Land a focused classifier regression fix for the exact Gemini `/rate-limits` 429 text, with tests that keep hard quota and long-window failures on the existing escalation path; leave the retry-count config knob out unless reopened as a separate design.

## Reproduction Assessment

Yes at source level: current main treats `current quota` as long-window unless a short-window hint is present, and the reported `/rate-limits` URL is not recognized as that hint. I did not run a live Gemini quota exhaustion test.

## Solution Assessment

The original config knob is not the best fix; the linked PR was closed for that reason. The best path is the narrower classifier fix with regression coverage for the exact Gemini message and existing hard-quota cases.
