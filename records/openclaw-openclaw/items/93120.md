## Summary

Keep open: current main and v2026.6.9 still route the reported Gemini `current quota` plus `/rate-limits` 429 away from same-model retry, and the related implementation attempts were closed unmerged with maintainer guidance that this issue remains valid.

## What This Changes

The issue asks OpenClaw to treat Gemini TPM/RPM quota messages as short-window rate limits and possibly make the same-model retry budget configurable.

## Best Possible Solution

Keep this as the canonical provider-rate-limit classifier bug, fix the Gemini short-window path only with a reliable structured/provider-scoped signal, and leave the retry-count config knob to maintainer product review or a separate policy tracker.

## Reproduction Assessment

Yes, source-reproducible: the reported Gemini message contains `current quota` and `/rate-limits`, while current main only treats explicit per-minute/RPM/TPM text or short Retry-After as short-window signals.

## Solution Assessment

No closeable implementation exists yet. The classifier layer is the right place for the narrow bug fix, but the prior URL-token PR was closed because the shared Google `rate-limits` URL also covers daily limits, and the retry-count knob remains a separate product/config decision.
