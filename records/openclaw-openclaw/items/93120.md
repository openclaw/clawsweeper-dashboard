## Summary

Keep open: current main and the latest release still classify the reported Gemini `current quota` plus `/rate-limits` 429 as a long-window quota path, and both attempted linked fixes were closed unmerged rather than adopted.

## What This Changes

The issue asks OpenClaw to retry same-model Gemini TPM/RPM 429s that are currently misclassified as long-window quota failures, with an earlier broader request for a configurable retry budget.

## Best Possible Solution

Keep this issue open for a provider-scoped Gemini classifier fix that distinguishes minute-scale RPM/TPM evidence from explicit daily/RPD quota evidence, with focused regression tests and no new retry config unless maintainers separately approve that policy.

## Reproduction Assessment

Yes at source level: current main still classifies `current quota` as long-window unless a separate short-window token or short Retry-After is present, and `/rate-limits` is not such a token. I did not run a live Gemini quota exhaustion path in this read-only review.

## Solution Assessment

No, the original config-only proposal and the later generic URL regex are not the best fixes. The maintainable path is a narrow provider-aware classifier with positive and negative Gemini quota coverage, while the retry-count config remains a separate product decision.
