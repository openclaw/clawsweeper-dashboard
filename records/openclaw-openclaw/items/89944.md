## Summary

Keep open: the Teams table feature has convincing screenshot proof, but the branch still exposes `adaptive` as a shared markdown table mode and sends plain IR text around table cards, which can lose links/formatting and create cross-channel table-delivery risk.

## What This Changes

The branch adds an opt-in Microsoft Teams adaptive markdown table mode that emits Adaptive Card Table attachments, plus shared markdown config/types, docs, and tests.

## Best Possible Solution

Land a Teams-owned adaptive table renderer that preserves existing markdown/link rendering for non-table slices, prevents shared `adaptive` mode from breaking sibling renderers, and keeps the Teams client compatibility boundary explicit.

## Reproduction Assessment

Yes, from source inspection: an adaptive-mode Teams reply with a markdown link or emphasis around a table will split and send plain `ir.text`, losing the URL or formatting. A non-Teams channel configured with `markdown.tables: adaptive` can also route that value into IR table parsing without a rendering branch.

## Solution Assessment

No, not yet. The Teams-owned renderer is a reasonable location, but the implementation should render IR slices back to channel-safe markdown and make the shared `adaptive` config/API behavior safe or explicitly Teams-scoped before merge.
