## Summary

This PR is still useful and should stay open, but the latest repair only proves the CLI upgrade path; the shared `memory_search` path can still pause on legacy indexes because the new FTS-format mismatch is not a full-reindex trigger for `reason: "search"`.

## What This Changes

The PR prefixes memory file paths into memory-core FTS rows, strips that prefix from keyword snippets/scoring, versions the FTS text format in index metadata, and adds regression tests for filename-token search and CLI-triggered legacy reindex.

## Best Possible Solution

Land after FTS text-format mismatches trigger a one-time rebuild for both CLI and shared `memory_search` paths while preserving provider/model mismatch pause behavior.

## Reproduction Assessment

Yes, source-reproducible. The linked issue gives concrete CLI output, and current source shows filename/path tokens are not indexed into FTS text; the new blocker is also traceable from source through the default `memory_search` path.

## Solution Assessment

No. Prefixing path text and versioning the FTS payload are the right local mechanisms, but the implementation still needs the default `memory_search` upgrade path to rebuild legacy rows, not only the CLI path.
