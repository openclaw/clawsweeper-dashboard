- `impact:security`: The affected behavior crosses a documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway auth secret reuse and password-mode shared-secret authentication.

## Summary

Keep open: current main and v2026.5.18 still allow the source-level auth-boundary path, and the open closing PR at https://github.com/openclaw/openclaw/pull/84338 should be reviewed and landed before this issue closes.

## What This Changes

The issue reports that a hook ingress bearer can also satisfy password-mode Gateway HTTP auth when `hooks.token` equals `gateway.auth.password`.

## Best Possible Solution

Land the linked PR or an equivalent focused fix that rejects hook-token reuse against both Gateway token and password shared secrets at startup/reload and reports both modes in security audit coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `hooks.enabled` with `hooks.token` equal to `gateway.auth.password`, then reuse the hook bearer against a password-mode Gateway HTTP surface.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from Gateway shared-secret auth in startup/reload validation and security audit, while leaving the documented trusted-operator semantics of `/tools/invoke` intact.
