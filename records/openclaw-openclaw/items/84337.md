## Label Justifications

- `P0`: The report describes a shared-secret auth boundary bypass from hook ingress credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses the documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway auth secret reuse and password-mode shared-secret authentication.

## Summary

Keep open: current main and the latest release still allow the reported hook-token/password-mode Gateway auth reuse path, and an open closing PR should be reviewed and landed before this issue closes.

## What This Changes

The issue reports that reusing `hooks.token` as `gateway.auth.password` lets a hook ingress bearer authenticate to password-mode Gateway HTTP operator surfaces.

## Best Possible Solution

Review and land the focused closing PR with startup/reload validation, security-audit coverage for token and password modes, regression tests, and operator-facing docs or upgrade guidance; then close this issue as fixed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure `hooks.enabled` with `hooks.token` equal to `gateway.auth.password`, then reuse the hook bearer against a password-mode Gateway HTTP surface.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from Gateway shared-secret auth in startup/reload validation and security audit, while preserving the documented full-operator semantics of `/tools/invoke`.
