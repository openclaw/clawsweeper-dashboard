- `P0`: The report describes a concrete auth boundary bypass where a hook ingress secret can become a full Gateway operator credential.
- `impact:security`: The affected behavior crosses a documented security boundary involving credentials and operator authorization.
- `impact:auth-provider`: The issue is specifically about Gateway shared-secret auth mode and credential routing between hook and operator surfaces.

## Summary

Keep open: current main still has the reported hook-token/password reuse path, and the linked fix PR at https://github.com/openclaw/openclaw/pull/84338 is open with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that reusing `hooks.token` as `gateway.auth.password` lets hook callers authenticate to password-mode Gateway HTTP endpoints such as `/tools/invoke`.

## Best Possible Solution

Land a narrow fix that treats token and password shared secrets as the Gateway auth credential set for hook separation, updates the security audit and regression tests, and then close this issue from the merged PR.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: configure hooks enabled with password-mode Gateway auth using the same secret, then reuse the hook bearer on a shared-secret HTTP surface; I did not run a live Gateway because this review is read-only.

## Solution Assessment

Yes. The maintainable fix is to reject hook-token reuse against both token and password Gateway shared secrets and mirror that in the security audit/tests, which is the direction proposed by https://github.com/openclaw/openclaw/pull/84338.
