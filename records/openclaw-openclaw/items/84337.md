## Label Justifications

- `P0`: The report describes a shared-secret auth boundary bypass from hook ingress credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses a documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway auth secret reuse and password-mode shared-secret authentication.

## Summary

Keep open: current main and the latest release still allow the reported hook-token/password-mode Gateway auth boundary crossing, and the issue is paired with an open closing PR that should be reviewed before closure.

## What This Changes

The issue reports that a hook ingress bearer can also authenticate to password-mode Gateway HTTP surfaces when `hooks.token` matches `gateway.auth.password`.

## Best Possible Solution

Land the linked PR or an equivalent focused fix that rejects hook-token reuse with both Gateway token and password shared secrets at startup/reload and in security audit, with regression tests and operator-facing guidance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable hooks, set `hooks.token` equal to `gateway.auth.password`, then reuse the hook bearer against a password-mode Gateway HTTP surface; I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from Gateway shared-secret auth in startup/reload validation and security audit, rather than changing the documented full-operator semantics of `/tools/invoke`.
