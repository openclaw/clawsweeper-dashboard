## Summary

Keep open: current main and the latest release still allow hook-token reuse to satisfy password-mode Gateway auth, and the linked closing PR is open rather than merged.

## What This Changes

The issue reports that reusing the same secret for `hooks.token` and `gateway.auth.password` lets a hook-only bearer authenticate to full Gateway HTTP operator surfaces.

## Best Possible Solution

Land a focused fix that rejects or flags hook-token reuse against both Gateway token and password shared-secret auth, with regression coverage for startup validation and security audit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `hooks.enabled` with `hooks.token` equal to `gateway.auth.password`, then reuse the hook bearer against a password-mode Gateway HTTP surface.

## Solution Assessment

Yes. The narrow maintainable solution is to keep hook ingress credentials distinct from Gateway shared-secret auth in startup/reload validation and security audit, rather than changing `/tools/invoke` trusted-operator semantics.
