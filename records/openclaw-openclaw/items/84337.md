## Label Justifications

- `P0`: The issue describes a shared-secret auth boundary bypass from hook ingress credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses the documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The report is specifically about Gateway shared-secret auth handling and password-mode credential reuse.

## Summary

Keep open: current main still lets a hook bearer be presented as password-mode Gateway auth when the same secret is reused, and the linked closing PR is still open and needs maintainer security/compatibility review before this issue can close.

## What This Changes

The issue asks OpenClaw to reject or report hook-token reuse with password-mode Gateway auth so hook ingress credentials cannot become full Gateway operator credentials.

## Best Possible Solution

Land or replace the linked fix with a narrow startup/reload guard and security-audit coverage for both Gateway token and password secret reuse, plus clear docs or upgrade guidance for operators who reused secrets.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure hooks.enabled with hooks.token equal to gateway.auth.password, then reuse the hook bearer on a password-mode Gateway HTTP surface; I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The maintainable direction is to keep hook ingress secrets distinct from Gateway shared-secret operator auth in startup/reload validation and security audit, while reviewing the linked PR's fail-closed compatibility impact.
