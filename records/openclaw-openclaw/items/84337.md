## Summary

Keep open: current main and the latest release still have the reported auth-boundary gap, and the linked closing PR is open rather than merged.

## What This Changes

The issue reports that reusing the same secret for `hooks.token` and password-mode Gateway auth lets a hook bearer authenticate to full Gateway HTTP operator surfaces.

## Best Possible Solution

Land a focused fix that rejects hook-token reuse with both Gateway token and password shared secrets at startup/reload, reports it in security audit, and includes regression coverage plus upgrade guidance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure hooks enabled with `hooks.token` equal to `gateway.auth.password`, then reuse the hook bearer against a password-mode Gateway HTTP endpoint.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from Gateway shared-secret operator credentials and cover both startup validation and security audit.
