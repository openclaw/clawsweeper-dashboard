## Label Justifications

- `P0`: The report describes a shared-secret auth boundary bypass from hook ingress credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses the documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway auth secret reuse and password-mode shared-secret authentication.

## Summary

Keep open: current main and the latest shipped tag still allow the reported source-level auth-boundary path, and the open closing PR should be reviewed and landed before this issue closes.

## What This Changes

The issue reports that reusing one secret for `hooks.token` and `gateway.auth.password` lets a hook bearer authenticate to password-mode Gateway HTTP surfaces as a trusted operator.

## Best Possible Solution

Land a focused security fix that treats hook-token reuse with either Gateway token or password auth as invalid at startup/reload and critical in security audit, with regression tests and operator-facing upgrade guidance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure `hooks.enabled` with `hooks.token` equal to `gateway.auth.password`, then reuse the hook bearer against a password-mode Gateway HTTP surface; I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from Gateway shared-secret auth in startup/reload validation and security audit, while preserving the documented trusted-operator semantics of `/tools/invoke`.
