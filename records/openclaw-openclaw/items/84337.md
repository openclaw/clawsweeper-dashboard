## Label Justifications

- `P0`: The report describes a security bypass where possession of a hook ingress secret can become full Gateway operator access.
- `impact:security`: The affected behavior crosses the documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway shared-secret auth resolution and password-mode credential reuse.

## Summary

Keep open: current main and v2026.5.20 still have the password-mode hook-token reuse gap, and https://github.com/openclaw/openclaw/pull/84338 is an open unmerged closing PR, so this should close only after that fix or an equivalent lands.

## What This Changes

The issue asks OpenClaw to prevent a hook ingress bearer from authenticating to password-mode Gateway operator HTTP surfaces when `hooks.token` equals `gateway.auth.password`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84338 or an equivalent focused fix that rejects hook-token reuse with both Gateway token and password auth and adds matching audit coverage, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: configure hooks with `hooks.token` equal to `gateway.auth.password`, then reuse that bearer against a password-mode Gateway HTTP surface; I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable solution is to keep hook ingress credentials distinct from Gateway shared-secret operator credentials in startup/reload validation and security audit, and the linked open PR is the current implementation candidate.
