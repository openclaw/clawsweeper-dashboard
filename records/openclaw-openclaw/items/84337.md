## Label Justifications

- `P0`: The issue describes a shared-secret auth boundary bypass from hook ingress credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses a documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway shared-secret auth reuse and password-mode credential handling.

## Summary

Keep open: current main and v2026.5.18 still allow a hook ingress bearer to match password-mode Gateway auth, and the linked closing PR is open and needs maintainer/security review before this issue can close.

## What This Changes

Reject hook-token reuse with password-mode Gateway auth so a hook ingress bearer cannot become a full Gateway operator credential.

## Best Possible Solution

Land a focused fix that treats hook-token reuse with either Gateway token or password shared-secret auth as invalid at startup/reload and critical in security audit, with regression coverage and upgrade guidance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure hooks.enabled with hooks.token equal to gateway.auth.password, then reuse that bearer against a password-mode Gateway HTTP surface.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from Gateway shared-secret auth in startup/reload validation and security audit, rather than changing the documented full-operator semantics of /tools/invoke.
