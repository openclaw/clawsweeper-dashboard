## Label Justifications

- `P0`: The report describes a shared-secret auth boundary bypass from hook ingress credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses a documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway auth secret reuse and password-mode shared-secret authentication.

## Summary

Keep open: current main and the latest release still allow the reported hook-token/password-mode Gateway auth path, and the linked closing PR at https://github.com/openclaw/openclaw/pull/84338 is still open and unmerged.

## What This Changes

The issue reports that reusing the same secret for `hooks.token` and `gateway.auth.password` lets a hook bearer authenticate to password-mode Gateway HTTP operator surfaces.

## Best Possible Solution

Keep this issue open until a focused fix lands that rejects hook-token reuse with both Gateway token and password shared secrets at startup/reload and reports the same condition in security audit coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure hooks with a token equal to `gateway.auth.password`, then reuse that bearer against a password-mode Gateway HTTP surface; I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The maintainable direction is to keep hook ingress credentials distinct from Gateway shared-secret operator credentials in startup/reload validation and audit reporting, while leaving the documented `/tools/invoke` operator semantics intact.
