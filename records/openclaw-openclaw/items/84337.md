## Label Justifications

- `P0`: The issue describes a hook ingress credential becoming full Gateway operator authentication, including owner-style HTTP surfaces.
- `impact:security`: The affected behavior crosses the documented boundary between hook ingress credentials and trusted Gateway operator auth.
- `impact:auth-provider`: The issue is specifically about Gateway shared-secret auth resolution and password-mode credential reuse.

## Summary

Keep open: current main and the latest release still show the reported auth-boundary path, and https://github.com/openclaw/openclaw/pull/84338 is an open closing PR that should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that a hook ingress bearer can unlock password-mode Gateway operator auth when `hooks.token` equals `gateway.auth.password`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84338 or an equivalent focused fix that rejects hook-token reuse with any Gateway shared-secret auth and adds matching audit, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure hooks with `hooks.token` equal to `gateway.auth.password`, then reuse that bearer on a password-mode Gateway HTTP surface where the bearer is also passed as `connectAuth.password`.

## Solution Assessment

Yes. The narrow maintainable solution is to keep hook ingress credentials distinct from Gateway token/password shared-secret auth at startup/reload and in the security audit, without changing `/tools/invoke` operator semantics.
