## Label Justifications

- `P0`: The issue describes a shared-secret auth boundary bypass from hook ingress credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses a documented Gateway authentication and trusted-operator security boundary.
- `impact:auth-provider`: The issue is specifically about Gateway auth secret reuse and password-mode shared-secret authentication.

## Summary

Keep open: current main and v2026.5.19 still have the password-mode hook-token reuse gap, and the issue is paired with an open closing PR that should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that reusing the same secret for `hooks.token` and password-mode Gateway auth lets a hook bearer authenticate to full Gateway HTTP surfaces.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/84338` or an equivalent focused fix that rejects hook-token reuse with Gateway token or password shared secrets at startup/reload and reports the same reuse in security audit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure hooks with a token equal to `gateway.auth.password`, then reuse that bearer against a password-mode Gateway HTTP endpoint; I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from Gateway shared-secret auth in startup/reload validation and security audit rather than changing `/tools/invoke` operator semantics.
