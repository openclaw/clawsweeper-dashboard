## Summary

Keep open. The feature is useful and current main does not implement Copilot model discovery, but the latest PR head still has a security blocker in the authenticated discovery fetch and leaves duplicate active changelog sections.

## What This Changes

The PR adds GitHub Copilot `/models` discovery, merges discovered non-built-in models into the provider catalog with Copilot IDE/configured headers, updates implicit header merging, and adds focused tests plus changelog entries.

## Best Possible Solution

Keep the provider-local additive discovery design, but make hostname scoping preserve private-address checks unless `request.allowPrivateNetwork` is true, then merge the changelog bullets into the existing Unreleased sections and rerun the focused Copilot discovery checks.

## Reproduction Assessment

Yes. On the PR head, configure `models.providers.github-copilot.baseUrl` to a loopback or LAN URL without `request.allowPrivateNetwork`; the discovery fetch policy includes `allowedHostnames`, which current main treats as skipping private-network checks while sending the Copilot bearer token.

## Solution Assessment

No. Auto-discovery is a good provider-owned direction, and the Gemini filter is a useful response to #74159, but this patch is not the best mergeable form until it preserves the provider private-network opt-in contract and fixes the changelog structure.

## Review Findings

Full review comments:

- **[P1] Require opt-in before sending Copilot tokens to private hosts:** `extensions/github-copilot/discovery.ts:98-100`
  - body: This policy still includes `allowedHostnames` for the configured `baseUrl`. Current SSRF code treats an allowed hostname as a reason to skip private-network checks, so a configured loopback/LAN/split-DNS endpoint can receive the Copilot bearer token even when `request.allowPrivateNetwork` is not enabled. Use hostname scoping that still checks resolved addresses, or only allow private targets when the provider request opt-in is true.
