## Summary

Keep open. Current main still lacks Copilot `/models` discovery, and the PR is useful but not merge-ready because the latest head still has an authenticated discovery SSRF-policy blocker, still auto-advertises Copilot Gemini models onto the known-broken Responses transport path tracked in #74159, and adds a stale raw-fetch guard exception.

## What This Changes

The PR adds GitHub Copilot `/models` catalog discovery, merges discovered model definitions and IDE headers into implicit provider config, updates provider discovery test hooks, and adds discovery/unit tests plus a raw-fetch guard allowlist entry.

## Best Possible Solution

Keep the provider-owned additive discovery design, but make discovery use the same private-network request policy contract as normal provider calls, do not advertise Copilot Gemini models through the unsupported Responses path until #74159 is fixed or filter them out, remove the stale raw-fetch allowlist entry, and add focused regression tests for configured private base URLs and Gemini discovery routing.

## Reproduction Assessment

Yes. On PR head, configure `models.providers.github-copilot.baseUrl` to a loopback/private/link-local endpoint without `request.allowPrivateNetwork`; catalog discovery passes that URL to `discoverCopilotModels`, which sends the Copilot bearer token under an `allowedHostnames` policy. For the transport bug, mock `/models` returning `gemini-3-flash-preview` and observe the discovered definition uses the non-Claude `openai-responses` route.

## Solution Assessment

No. Auto-discovery is a reasonable provider-local direction, but this implementation is not the narrowest maintainable solution until it preserves the private-network opt-in contract and avoids auto-registering known unsupported Gemini transport entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: `baseUrl` can come from `models.providers.github-copilot.baseUrl`, but this policy builds `{ allowedHostnames: [host] }`, which current SSRF handling treats as enough to skip private-network checks. A configured loopback, private, or link-local discovery target can therefore receive the Copilot bearer token without `request.allowPrivateNetwork`; thread the provider request opt-in into discovery instead of using this bypassing helper alone.
  - confidence: 0.91
- **[P2] Do not discover Gemini models onto Responses transport:** `extensions/github-copilot/discovery.ts:64`
  - body: Discovered models call `resolveCopilotTransportApi`, which maps every non-Claude ID to `openai-responses`. Related #74159 shows Copilot Gemini IDs returned by `/models` fail on `/responses` and require `/chat/completions`, so this PR would auto-advertise Gemini models that fail at runtime unless the Copilot completions transport/header path is fixed or those models are filtered.
  - confidence: 0.86
- **[P3] Remove the stale raw-fetch exception:** `scripts/check-no-raw-channel-fetch.mjs:30`
  - body: The discovery helper now uses `fetchWithSsrFGuard`, so the new raw-fetch allowlist entry no longer corresponds to a real raw `fetch` call. Keeping an unused exception weakens the guard for this provider file and can hide a future raw fetch if it lands on that line.
  - confidence: 0.78
