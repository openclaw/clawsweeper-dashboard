## Summary

Keep open. Current main still lacks Copilot `/models` discovery, but the PR head still has blocking request-policy and transport issues, plus a stale guard exception and missing changelog entry.

## What This Changes

This PR adds GitHub Copilot `/models` discovery, merges discovered models into the provider catalog, preserves Copilot IDE headers, and adds focused discovery/merge tests.

## Best Possible Solution

Keep the provider-owned additive discovery design, but patch this PR to honor the provider private-network opt-in, avoid unsupported Gemini discovery until #74159 is fixed, remove the stale raw-fetch exception, add tests, and add the changelog entry.

## Reproduction Assessment

Yes. Configure `models.providers.github-copilot.baseUrl` to a loopback/private URL without `request.allowPrivateNetwork` and discovery will pass that host through an allowed-hostname policy while sending the Copilot bearer token; separately, mock `/models` with a Gemini ID and it is classified as `openai-responses`.

## Solution Assessment

No. Auto-discovery is the right provider-local direction, but this implementation should not merge until it preserves the request-policy contract, avoids advertising unsupported Gemini runtime entries, removes the stale guard exception, and records the user-facing feature in the changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:88`
  - body: `baseUrl` can come from `models.providers.github-copilot.baseUrl`, but this policy turns that configured host into `allowedHostnames`; current SSRF policy skips private-network checks for allowed hostnames. A loopback, LAN, link-local, or private configured discovery target can therefore receive the Copilot bearer token without `request.allowPrivateNetwork`, so thread the provider request opt-in into the policy instead.
  - confidence: 0.92
- **[P2] Do not discover Gemini models onto Responses transport:** `extensions/github-copilot/discovery.ts:61`
  - body: Discovered models delegate to `resolveCopilotTransportApi`, which maps every non-Claude ID to `openai-responses`. Related #74159 reports Copilot Gemini IDs returned by `/models` fail on `/responses`, so this would auto-advertise Gemini entries that fail at runtime unless the completions transport/header path is fixed or those IDs are filtered.
  - confidence: 0.86
- **[P3] Remove the stale raw-fetch exception:** `scripts/check-no-raw-channel-fetch.mjs:30`
  - body: The new discovery helper uses `fetchWithSsrFGuard`, so the added raw-fetch allowlist entry no longer corresponds to a real raw `fetch` call. Keeping it weakens the guard for this provider file and can hide a future raw fetch on that line.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `extensions/github-copilot/index.ts:414-421`
  - body: This is a user-facing `feat` that makes new Copilot models appear automatically, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing feature changes before merge.
  - confidence: 0.9
