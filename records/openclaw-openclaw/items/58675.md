## Summary

Keep open. Current main still lacks Copilot `/models` auto-discovery, and the latest PR branch remains blocked by the authenticated discovery SSRF policy, the open Gemini transport issue, a stale raw-fetch guard exception, and a missing changelog entry.

## What This Changes

The PR adds GitHub Copilot `/models` discovery, returns discovered model definitions and IDE headers from the provider catalog, deep-merges Copilot headers during implicit provider config, and adds focused contract/unit coverage.

## Best Possible Solution

Keep the provider-owned additive discovery design, but require the configured provider request private-network opt-in for private discovery URLs, defer or filter Gemini discovery until #74159 is fixed, remove the stale guard exception, add focused regression tests, and add the required changelog entry.

## Reproduction Assessment

Yes. On the PR head, configure `models.providers.github-copilot.baseUrl` to a loopback or LAN URL without `request.allowPrivateNetwork`; the discovery path sends the Copilot bearer token using an `allowedHostnames` policy that current SSRF code treats as skipping private-network checks. A mocked `/models` response containing a Gemini preview ID also reproduces the unsupported `openai-responses` registration path.

## Solution Assessment

No. Auto-discovery is a reasonable provider-local solution, but this implementation is not the best mergeable form until it preserves the provider request-policy contract, avoids advertising currently unsupported Gemini entries, removes the stale script exception, and includes the user-facing changelog entry.

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: `baseUrl` can come from `models.providers.github-copilot.baseUrl`, but the discovery fetch builds an `allowedHostnames` policy from that same value. Current SSRF code skips private-network checks for allowed hostnames, so a loopback, LAN, link-local, or private configured discovery target can receive the Copilot bearer token without `request.allowPrivateNetwork`; thread the provider request opt-in into this policy instead.
  - confidence: 0.92
- **[P2] Avoid auto-registering unsupported Gemini discoveries:** `extensions/github-copilot/discovery.ts:64`

Related #74159 shows Copilot Gemini IDs returned by `/models` fail on `/responses`, so discovery can advertise Gemini entries that fail at runtime unless the Gemini transport/header path is fixed or those IDs are filtered.
  - confidence: 0.86
