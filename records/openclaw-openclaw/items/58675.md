## Summary

Keep open. Current main still lacks GitHub Copilot chat model discovery, so the PR is not obsolete, but the latest branch has blocking issues: the authenticated discovery request can bypass the provider private-network opt-in when `baseUrl` is configured, and discovered Gemini models inherit the existing `openai-responses` routing despite related issue #74159 showing Copilot rejects those Gemini IDs on that endpoint.

## What This Changes

The PR adds GitHub Copilot `/models` discovery during provider catalog resolution, returns discovered model definitions and IDE headers, deep-merges Copilot headers with explicit provider config, and adds related provider discovery tests plus a raw-fetch guard allowlist entry.

## Best Possible Solution

Keep the provider-owned additive discovery design, but make discovery use the same private-network policy contract as normal provider requests, reconcile Copilot Gemini transport selection with #74159 before advertising those discovered models, remove the stale raw-fetch allowlist entry, and add focused regression tests for private configured base URLs and Gemini discovery routing.

## Reproduction Assessment

Yes. On the PR head, configure `models.providers.github-copilot.baseUrl` to a private or link-local endpoint without `request.allowPrivateNetwork`; catalog discovery passes that URL into `discoverCopilotModels` with an `allowedHostnames` policy. For the transport bug, mock `/models` returning `gemini-3-flash-preview` and observe the discovered definition uses `openai-responses`.

## Solution Assessment

No. Auto-discovery is a reasonable product direction, but the current implementation is not the narrowest maintainable solution until it preserves the provider private-network opt-in and avoids auto-advertising Copilot Gemini models on the unsupported Responses transport.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: `baseUrl` can come from `models.providers.github-copilot.baseUrl`, but this policy uses `allowedHostnames`, which current SSRF handling treats as enough to skip private-network checks. A private, loopback, or link-local configured discovery target can therefore receive the Copilot bearer token without `request.allowPrivateNetwork`; thread the provider request opt-in into the discovery policy instead of using this bypassing helper alone.
  - confidence: 0.91
- **[P2] Do not discover Gemini models onto the Responses transport:** `extensions/github-copilot/discovery.ts:64`
  - body: Discovered models call `resolveCopilotTransportApi`, which maps every non-Claude ID to `openai-responses`. Related #74159 shows Copilot Gemini IDs returned by `/models` fail on `/responses` and require `/chat/completions`, so this PR would auto-advertise Gemini models that fail at runtime unless the Copilot completions transport/header path is fixed or those models are filtered.
  - confidence: 0.86
- **[P3] Remove the stale raw-fetch allowlist entry:** `scripts/check-no-raw-channel-fetch.mjs:30`
  - body: The latest discovery helper uses `fetchWithSsrFGuard`, so this new raw-fetch allowlist entry no longer corresponds to a real raw `fetch` call. Keeping an unused exception weakens the guard for a security-sensitive provider file and can hide a future raw fetch if it lands on that line.
  - confidence: 0.78
