## Summary

Keep open. The feature direction is useful, but the latest PR head still introduces an authenticated Copilot discovery fetch whose configured base URL can bypass the existing private-network opt-in contract through the allowed-hostname SSRF policy.

## What This Changes

The PR adds GitHub Copilot chat model discovery via authenticated `/models` requests, merges discovered models and IDE headers into provider catalog resolution, deep-merges Copilot headers with explicit provider config, and adds discovery/test-helper coverage.

## Best Possible Solution

Keep the provider-owned additive discovery design, but build discovery SSRF policy from a hostname allowlist that still performs private-network checks, merge in the configured provider request private-network opt-in only when enabled, and add a focused regression test for private/link-local configured discovery base URLs.

## Reproduction Assessment

Yes for the review finding. On the PR head, configure `models.providers.github-copilot.baseUrl` to a private, link-local, or private-DNS HTTP(S) endpoint without `models.providers.github-copilot.request.allowPrivateNetwork`; the catalog path passes that URL into `discoverCopilotModels`, which sends the bearer token under an `allowedHostnames` policy that skips private-network checks.

## Solution Assessment

No, not as currently proposed. The discovery feature is the right product shape, but the fetch policy must preserve the existing provider private-network contract before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
