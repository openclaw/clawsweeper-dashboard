## Summary

Keep open. The provider-local discovery direction is useful, but the latest PR head still has an authenticated discovery security blocker, auto-registers known-problem Copilot Gemini models onto the Responses transport, leaves a stale raw-fetch guard exception, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds GitHub Copilot `/models` catalog discovery, merges discovered model definitions and IDE headers into implicit provider config, and adds discovery/unit contract coverage.

## Best Possible Solution

Keep the provider-owned additive discovery design, but thread `models.providers.github-copilot.request.allowPrivateNetwork` into the discovery SSRF policy, filter or defer Gemini discovery until #74159 is fixed, remove the stale raw-fetch exception, add focused regression tests, and add the changelog entry.

## Reproduction Assessment

Yes. On PR head, configure `models.providers.github-copilot.baseUrl` to a loopback/private URL without `request.allowPrivateNetwork`; discovery passes that URL to `discoverCopilotModels`, which sends the Copilot bearer token under an allowed-hostname policy that skips private-network checks. Separately, mock `/models` with `gemini-3-flash-preview` and the discovered definition gets the current non-Claude `openai-responses` route.

## Solution Assessment

No. Auto-discovery is a reasonable provider-local solution, but this implementation should not merge until it preserves the provider request-policy contract, avoids advertising unsupported Gemini transport entries, removes the unused guard exception, and includes the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: `baseUrl` can come from `models.providers.github-copilot.baseUrl`, but this policy converts that configured host into `allowedHostnames`; the current SSRF guard skips private-network checks for allowed hostnames. That means a loopback, LAN, link-local, or private configured discovery target can receive the Copilot bearer token without `request.allowPrivateNetwork`; thread the provider request opt-in into the discovery policy instead.
  - confidence: 0.92
- **[P2] Do not discover Gemini models onto Responses transport:** `extensions/github-copilot/discovery.ts:64`
  - body: Discovered models delegate to `resolveCopilotTransportApi`, which maps every non-Claude ID to `openai-responses`. Related #74159 shows Copilot Gemini IDs returned by `/models` fail on `/responses`, so this would auto-advertise Gemini entries that fail at runtime unless the completions transport/header path is fixed or those IDs are filtered.
  - confidence: 0.86
- **[P3] Remove the stale raw-fetch exception:** `scripts/check-no-raw-channel-fetch.mjs:30`
  - body: The discovery helper uses `fetchWithSsrFGuard`, so the new raw-fetch allowlist entry no longer corresponds to a real raw `fetch` call. Keeping the exception weakens the guard for this provider file and can hide a future raw fetch on that line.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `extensions/github-copilot/index.ts:420`
  - body: This adds user-visible Copilot model auto-discovery, but the PR diff does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.9
