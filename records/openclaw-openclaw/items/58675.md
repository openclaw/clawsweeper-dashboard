## Summary

Keep open. The latest head still has blocking authenticated discovery issues: a configured Copilot `baseUrl` can receive a bearer-token `/models` request under a hostname-only SSRF policy that skips private-network checks, and discovery can still advertise Gemini models affected by open #74159.

## What This Changes

The PR adds GitHub Copilot `/models` discovery during provider catalog resolution, preserves Copilot IDE headers with configured headers, and adds focused discovery/implicit-provider test coverage.

## Best Possible Solution

Keep the provider-owned additive discovery design, but merge the configured provider request private-network policy into discovery fetches, defer or filter Gemini discoveries until #74159 is fixed, remove the stale raw-fetch exception, and add the required changelog entry.

## Reproduction Assessment

Yes. On the PR head, configure `models.providers.github-copilot.baseUrl` to loopback or LAN without `request.allowPrivateNetwork` and the discovery fetch still sends the bearer token under an allowed-hostname policy; a mocked `/models` payload with a new Gemini ID reproduces registration through the unsupported `openai-responses` path.

## Solution Assessment

No. Auto-discovery is a reasonable provider-local approach, but this implementation is not the narrowest safe fix until it preserves the provider request-policy contract and avoids advertising currently unsupported Gemini entries.

  - body: `baseUrl` can come from `models.providers.github-copilot.baseUrl`, but this fetch sends the Copilot bearer token with only an `allowedHostnames` policy. Current SSRF handling treats that as skipping private-network checks, so loopback, LAN, link-local, or split-DNS targets can receive the token without `request.allowPrivateNetwork`; merge the provider request opt-in into this policy.
