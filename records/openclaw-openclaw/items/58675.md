## Summary

Keep open. The latest head still has merge-blocking issues: authenticated discovery can send the Copilot bearer token to a configured private base URL without the provider request opt-in, and discovery can advertise Gemini models that current Copilot transport code cannot run.

## What This Changes

The PR adds GitHub Copilot `/models` discovery, merges discovered models into the provider catalog, preserves Copilot IDE headers with configured headers, and adds focused discovery and contract coverage.

## Best Possible Solution

Keep the provider-owned additive discovery design, but require the configured provider request private-network opt-in for private discovery URLs, defer or filter Gemini discovery until #74159 is fixed, remove the stale guard exception, add focused regression tests, and add the required changelog entry.

## Reproduction Assessment

Yes. On the PR head, configure `models.providers.github-copilot.baseUrl` to loopback or LAN without `request.allowPrivateNetwork`; the discovery path sends the bearer token with an allowed-hostname policy that current SSRF code treats as skipping private-network checks. A mocked `/models` response containing a new Gemini ID reproduces the unsupported `openai-responses` registration path described by #74159.

## Solution Assessment

No. Auto-discovery is a reasonable provider-local solution, but this implementation is not mergeable until it preserves the provider request-policy contract and avoids advertising currently unsupported Gemini entries.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: `baseUrl` can come from `models.providers.github-copilot.baseUrl`, but this discovery fetch sends the Copilot bearer token using an `allowedHostnames` policy. Current SSRF code treats allowed hostnames as skipping private-network checks, so loopback, LAN, link-local, or split-DNS endpoints can receive the token without `request.allowPrivateNetwork`; thread the provider request opt-in into this policy instead.
  - confidence: 0.92
- **[P2] Avoid auto-registering unsupported Gemini discoveries:** `extensions/github-copilot/discovery.ts:64`
  - body: Discovered models delegate to `resolveCopilotTransportApi`, which maps every non-Claude ID to `openai-responses`. Related #74159 shows Copilot Gemini IDs returned by `/models` fail on `/responses`, so discovery can advertise Gemini entries that fail at runtime unless the Gemini transport/header path is fixed or those IDs are filtered.
  - confidence: 0.86

  - body: The discovery helper now uses `fetchWithSsrFGuard`, so this new raw-fetch allowlist entry no longer corresponds to a real raw `fetch` call. Keeping it weakens the guard for this provider file and can hide a future raw fetch on that line.
