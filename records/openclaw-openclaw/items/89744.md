## Summary

Keep open: the PR fixes the right Discord startup surface, but the new default-account promotion can still prioritize an account the gateway will not start, which can delay the actual runtime-enabled account in duplicate-token configurations.

## What This Changes

The branch reorders Discord startup delay calculation so `channels.discord.defaultAccount` takes the first startup slot and adds a regression test for secondary stagger delays.

## Best Possible Solution

Make Discord startup priority operate on the same accounts the gateway will actually start, then preserve the configured default-first behavior and secondary stagger with focused regression coverage.

## Reproduction Assessment

Yes, source inspection shows current main sorts Discord account IDs before applying the startup stagger, so a configured default account that sorts later receives a delayed slot. The PR proof also records the reported pre-patch sorted slot and the after-fix default startup timing.

## Solution Assessment

No, not yet: the fix is in the right Discord plugin layer, but it should build the promoted startup list from runtime-enabled accounts rather than enabled/configured accounts only. That keeps the default-first behavior without delaying the duplicate-token owner that gateway startup will actually run.
