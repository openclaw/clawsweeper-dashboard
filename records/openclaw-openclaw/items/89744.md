## Summary

Keep open: the PR fixes the ordinary default-account-first case, but the startup slot helper still counts Discord accounts that gateway startup will skip as runtime-disabled, so a duplicate-token default alias can keep delaying the real bot.

## What This Changes

The PR changes Discord startup delay ordering to promote the configured default account before secondary accounts and adds a focused regression test.

## Best Possible Solution

Use one Discord startup candidate list that filters to runtime-enabled accounts, promotes the resolved default only when that account will actually start, and preserves the existing stagger for secondary accounts.

## Reproduction Assessment

Yes. Current source and the PR helper make the remaining blocker source-reproducible: gateway startup skips duplicate-token aliases, while the PR's delay helper still counts enabled/configured aliases as startup slots.

## Solution Assessment

No as submitted. Promoting the configured default account is the right layer, but the helper needs to share Discord's runtime-enabled account predicate before this is the best fix.
