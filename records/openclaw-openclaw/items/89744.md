## Summary

Keep open. The PR is aimed at the right Discord startup bug and includes useful live proof, but the patch still lets Discord accounts that gateway startup will skip consume delay slots, so the previous blocker remains actionable.

## What This Changes

The PR promotes the configured Discord default account to the first startup-delay slot and adds a focused Discord channel test for default-first staggering.

## Best Possible Solution

Build Discord startup-delay ordering from the same runtime-enabled account set gateway startup uses, then promote the configured default only if that account is actually startable.

## Reproduction Assessment

Yes, by source inspection. Configure a duplicate-token Discord default/env fallback plus a config-token owner account: gateway skips the duplicate through config.isEnabled, while the PR delay helper can still let the skipped account consume the zero-delay slot.

## Solution Assessment

No. The PR is a plausible default-order fix, but the best fix should reuse Discord's runtime-enabled account filter so the delay slots match the accounts gateway will actually start.
