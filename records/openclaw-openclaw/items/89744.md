## Summary

Keep open: the PR fixes the available-default Discord startup path and includes strong live proof, but the new promotion still counts configured-but-unavailable SecretRefs as startup candidates, so an unavailable default can delay available secondary bots.

## What This Changes

The PR reorders Discord multi-account startup delays so the configured default account is placed before secondary accounts, with a focused channel test.

## Best Possible Solution

Keep the default-first behavior, but make Discord startup slot ordering consider only runtime-startable accounts so configured-unavailable defaults do not delay available secondaries.

## Reproduction Assessment

Yes for the underlying bug from source: current main sorts Discord account ids and computes startup delay from that list, while docs define defaultAccount as the configured default route. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not quite: the fix location is appropriate, but the promotion predicate needs to skip configured_unavailable default accounts and cover that state with a regression test.
