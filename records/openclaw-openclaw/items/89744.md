## Summary

Keep open: current main and the latest release still compute Discord startup delay from account list order, while this PR is the canonical proof-backed fix after the attempted replacement was closed as superseded.

## What This Changes

The PR changes Discord startup slot ordering so an eligible configured `defaultAccount` starts first, while preserving runtime-enabled/configured filtering and adding regression tests.

## Best Possible Solution

Land this Discord-local fix after maintainer acceptance that an eligible `channels.discord.defaultAccount` should own the zero-delay startup slot while secondary accounts keep the existing stagger.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.6` assign Discord startup delay from account-list order, so a configured default account that sorts later can receive a nonzero delay.

## Solution Assessment

Yes. This is the narrowest maintainable fix I found: it stays in the Discord plugin, reuses the runtime-enabled and configured-token predicates, and avoids hard-coded account aliases or a new config knob.
