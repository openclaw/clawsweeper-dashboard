## Summary

Keep open: this PR has useful Discord default-account startup proof, but the branch is not merge-ready because its startup slot list can still let a duplicate-token default account take the zero-delay slot while the gateway treats that account as disabled. The open replacement at https://github.com/openclaw/openclaw/pull/90660 is useful context, but it is not a safe supersession target yet because it still lacks real behavior proof and has its own predicate finding.

## What This Changes

This PR refactors Discord startup delay calculation to promote `channels.discord.defaultAccount` to the zero-delay slot and adds a regression test for multi-account startup ordering.

## Best Possible Solution

Land a review-clean Discord plugin patch that builds startup slots from accounts that are both configured and runtime-enabled, promotes an eligible `defaultAccount` first, and keeps focused regression tests plus real startup proof.

## Reproduction Assessment

Yes. Source inspection shows current main indexes Discord startup slots by account list order while the PR proof shows a configured default account that would otherwise be delayed to index 3; the duplicate-token gap is also source-reproducible from the runtime-enabled predicate mismatch.

## Solution Assessment

No, not as submitted. Promoting the configured default account in the Discord plugin is the right direction, but the slot list must also match Discord's runtime-enabled and configured-account eligibility before merge.
