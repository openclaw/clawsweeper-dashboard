## Summary

Keep open: the PR fixes the configured-default-first path for fully configured Discord accounts, but it still uses a delay-candidate predicate that diverges from gateway runtime startup and can delay runnable accounts in duplicate-token setups.

## What This Changes

The PR changes Discord multi-account startup delay ordering so the configured default account is promoted ahead of secondary accounts and adds a focused channel test.

## Best Possible Solution

Reuse Discord's runtime-enabled account predicate for startup delay candidates, then promote the configured default only among accounts the gateway will actually start.

## Reproduction Assessment

Yes from source: gateway startup skips accounts that plugin.config.isEnabled marks disabled, and Discord marks duplicate-token accounts disabled there, but the PR's delay candidate list does not use that predicate. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: promoting the configured default account is the right fix direction, but the best fix must share the gateway runtime-enabled account predicate so skipped accounts cannot consume delay slots.
