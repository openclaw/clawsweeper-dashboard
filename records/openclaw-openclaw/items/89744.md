## Summary

Keep open; the default-account startup fix is useful and the live proof is strong, but the merge should wait for the startup ordering helper to use Discord's existing runtime-enabled account predicate so skipped duplicate-token aliases cannot consume stagger slots.

## What This Changes

The PR promotes the configured Discord default account to the first startup-delay slot and adds a channel test for the reordered stagger timing.

## Best Possible Solution

Keep the default-account-first behavior, but compute stagger slots from the same runtime-enabled, configured Discord accounts the gateway can actually start, with duplicate-token regression coverage before merge.

## Reproduction Assessment

Yes. Source inspection and the PR's live proof show the current sorted account list can delay a configured default account; the remaining blocker is a source-reproducible duplicate-token slot case that I did not execute locally in this read-only review.

## Solution Assessment

No, not yet. Promoting `defaultAccount` is the right shape, but the helper should reuse Discord's runtime-enabled predicate so accounts the gateway will skip cannot change delay slots.
