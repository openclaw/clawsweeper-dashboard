## Summary

Keep this PR open: the default-account startup priority fix is useful, but the current patch can still assign the zero-delay slot to a Discord account that the gateway will skip for duplicate-token handling, delaying the real runtime-enabled account.

## What This Changes

The PR reorders Discord multi-account startup slots so the configured `channels.discord.defaultAccount` starts before secondary accounts and adds a regression test for that ordering.

## Best Possible Solution

Merge after the startup slot builder reuses Discord's runtime-enabled account filtering and the PR includes a duplicate-token default-account regression test.

## Reproduction Assessment

Yes. The central issue and the PR-introduced duplicate-token edge case are source-reproducible from the Discord startup delay helper, gateway `isEnabled` gate, and Discord duplicate-token runtime predicate.

## Solution Assessment

No. Promoting `defaultAccount` is the right owner-boundary fix, but the implementation must build slots from the same runtime-enabled account set that the gateway actually starts.
