## Summary

Keep this PR open, but it still needs a focused repair before merge: the happy-path default-account promotion is useful and backed by real Discord startup logs, yet the new ordering can reserve the first stagger slot for a configured but non-startable default account and delay available secondary bots.

## What This Changes

The PR reorders Discord multi-account startup delay slots so the configured default account starts before secondary accounts and adds a focused channel test.

## Best Possible Solution

Use a single Discord startup-slot ordering path that promotes the configured default only when that account is actually runtime-startable, while configured-unavailable defaults still report their resolver error without consuming a stagger slot.

## Reproduction Assessment

Yes, source inspection and the linked report establish the current-main behavior: Discord delay slots follow account-list order instead of the configured default account. The remaining PR regression is also source-reproducible through configured_unavailable credentials counting as configured while startAccount throws before provider startup.

## Solution Assessment

No, not quite yet. Promoting the configured default is the right layer, but the helper must match the runtime-startable account set so unavailable SecretRefs or skipped duplicate-token accounts do not consume stagger slots.
