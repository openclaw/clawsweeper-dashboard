- `P2`: This is a normal-priority Discord multi-account startup bug with limited blast radius but real restart ergonomics impact.
- `merge-risk: 🚨 compatibility`: The PR changes behavior for an existing Discord config key and can alter startup timing for existing multi-account configs.
- `merge-risk: 🚨 availability`: A skipped duplicate-token default account can still consume the first startup slot and stall the runnable Discord account during restart.

## Summary

Keep open: the fix is in the right Discord plugin boundary and has live proof for the normal default-account path, but the new startup order still uses configured accounts instead of Discord's runtime-enabled account predicate, so a duplicate-token default account that the gateway skips can still take the first slot and delay the runnable account.

## What This Changes

The PR reorders Discord gateway startup delay calculation so `channels.discord.defaultAccount` starts before secondary accounts and adds a focused regression test.

## Best Possible Solution

Keep the plugin-local ordering fix, but build the startup delay list from Discord runtime-enabled accounts before promoting the default and add a duplicate-token default-account regression.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the remaining PR defect: configure accounts ordered `["work", "zdefault"]` with the same token and `channels.discord.defaultAccount: "zdefault"`; the gateway skips `zdefault`, but the PR helper still promotes it before calculating `work`'s delay.

## Solution Assessment

No, not yet. The fix belongs in the Discord plugin startup-delay helper, but the best implementation must reuse the runtime-enabled account predicate before default promotion instead of maintaining a looser duplicate candidate filter.
