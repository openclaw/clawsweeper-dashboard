## Summary

## What This Changes

The PR changes Discord startup-delay calculation so an eligible configured `channels.discord.defaultAccount` gets the zero-delay startup slot, with regressions for duplicate-token and unconfigured-account cases.

## Best Possible Solution

Land the Discord-local fix after maintainer acceptance that an eligible configured default account should own the zero-delay startup slot while secondary accounts keep the existing stagger.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` still compute Discord startup delay from the sorted filtered account list, so a configured default account that sorts later can receive a nonzero delay.

## Solution Assessment

Yes as an implementation shape, with maintainer acceptance required for upgrade behavior. The PR stays inside the Discord plugin startup path and reuses existing default-account, runtime-enabled, and configured-token predicates instead of hard-coding account names or adding config.
