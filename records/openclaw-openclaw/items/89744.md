## Summary

Keep this PR open: current main still has the Discord sorted-account startup-delay behavior, and the PR fixes the right Discord-owned runtime surface. The patch should first avoid promoting a configured-unavailable default account into the first startup slot, because that can delay available secondary Discord bots behind an account that will throw before provider startup.

## What This Changes

The PR reorders Discord startup-delay slot calculation so the configured default Discord account starts before secondary accounts, with a focused regression test.

## Best Possible Solution

Keep the Discord-local startup ordering fix, but make the promoted default predicate match accounts that can actually start and cover the unavailable-SecretRef case.

## Reproduction Assessment

Yes. Source inspection shows current main sorts account ids before computing Discord stagger slots, and the related issue plus PR proof describe a default account that sorts after secondary accounts and is delayed.

## Solution Assessment

Mostly yes. The fix belongs in the Discord startup-delay path rather than the shared account-list helper, but it should require the promoted default account to be actually startable so unavailable SecretRefs do not delay healthy secondary accounts.
