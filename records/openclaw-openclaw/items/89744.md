## Summary

Keep open: the PR fixes the main Discord default-account startup ordering case, but the new startup list can still include a duplicate-token account that gateway startup will disable, leaving the actual runtime-enabled account in a delayed slot.

## What This Changes

The PR reorders Discord multi-account startup delay calculation to put the resolved default account first and adds focused channel test coverage for that ordering.

## Best Possible Solution

Merge after the startup delay order is derived from the same runtime-enabled Discord account set that gateway startup actually starts, with duplicate-token defaultAccount regression coverage.

## Reproduction Assessment

Yes. Current main source computes delay slots from sorted Discord account ids, and the linked issue plus PR proof show a configured default account can be delayed behind secondary accounts; I did not run a live Discord restart in this read-only review.

## Solution Assessment

No. Promoting the configured default is the right direction, but the helper should first use the same runtime-enabled account set as gateway startup so duplicate-token disabled accounts cannot consume slot 0.
