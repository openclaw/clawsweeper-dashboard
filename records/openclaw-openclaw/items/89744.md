## Summary

Keep open: the PR is a useful fix direction with strong live proof for the distinct-token default-account path, but the startup slot helper still uses a different predicate than gateway startup, so duplicate-token Discord configs can keep delaying the account that actually starts.

## What This Changes

The PR adds Discord startup ordering that promotes the resolved default account to the first stagger slot and adds a regression test for distinct-token multi-account startup order.

## Best Possible Solution

Keep the default-account-first ordering, but compute delay slots from the same configured and runtime-enabled account set the gateway will actually start, with duplicate-token owner coverage.

## Reproduction Assessment

Yes by source inspection: configure a duplicate-token Discord setup where an env-backed default account is disabled and a named config-token account owns the token; gateway startup skips the disabled account, but the PR helper still counts it in the delay slots. I did not run tests because this review is read-only.

## Solution Assessment

No: the direction is right, but the best fix must reuse Discord's runtime-enabled account predicate for slot calculation so gateway-skipped duplicate-token accounts cannot consume delay slots.
