## Summary

Keep open: the PR targets a real Discord multi-account startup bug, but the current patch can still delay the only runtime-startable account when the configured default account is skipped as a duplicate-token account.

## What This Changes

This PR reorders Discord gateway startup delay calculation so the configured Discord default account gets the first startup slot and adds a focused channel test.

## Best Possible Solution

Filter Discord startup ordering through the same runtime-enabled account predicate used by gateway startup, then promote the configured default only if it survives that filter.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.1 can assign a nonzero startup delay to a configured Discord default account that sorts after secondary accounts; the PR body also includes live fixed-path logs.

## Solution Assessment

No, not as-is. The helper is the right layer, but it should build the startup order from runtime-enabled Discord accounts before default promotion so skipped duplicate-token accounts cannot consume slots.
