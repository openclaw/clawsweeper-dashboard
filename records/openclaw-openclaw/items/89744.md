## Summary

Keep open: the PR targets a real Discord multi-account startup bug and has strong runtime proof, but the submitted helper can still assign the zero-delay slot to a duplicate-token default account that gateway startup will skip, delaying the actual enabled Discord bot.

## What This Changes

The PR promotes the configured Discord default account to the first startup slot before staggering secondary Discord accounts and adds focused channel test coverage.

## Best Possible Solution

Merge this behavior only after Discord startup slots are ordered from configured, runtime-enabled accounts, with the configured default first only when that same account would actually start.

## Reproduction Assessment

Yes. Source inspection shows current main derives Discord startup delay from sorted account ids, and the PR finding is source-reproducible from the gateway `isEnabled` skip path plus Discord duplicate-token tests.

## Solution Assessment

No, not as submitted. Promoting the configured default account inside the Discord plugin is the right layer, but the helper must reuse the existing runtime-enabled account contract before assigning delay slots.
