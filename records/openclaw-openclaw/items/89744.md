## Summary

Keep open: the PR addresses the right Discord multi-account startup bug and includes strong live proof, but the delay-slot helper still counts configured/token-present accounts that the gateway can skip as runtime-disabled duplicates, leaving an availability blocker before merge.

## What This Changes

The branch promotes the configured Discord default account to the first startup delay slot and adds a focused Discord channel regression test for default-before-secondary startup ordering.

## Best Possible Solution

Build Discord startup delay slots from the same runtime-enabled account set the gateway starts, then promote the configured default account and preserve the existing stagger for the remaining accounts.

## Reproduction Assessment

Yes. Current main computes Discord startup delay order from configured account IDs while the gateway starts only accounts that pass plugin.config.isEnabled; the linked issue and PR proof show the defaultAccount delay in multi-account Discord configs.

## Solution Assessment

No, not yet. Promoting defaultAccount is the right direction, but the PR should build its slot list from the same runtime-enabled predicate the gateway uses so skipped duplicate-token accounts cannot affect delays.
