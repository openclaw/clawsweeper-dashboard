## Summary

Keep this PR open: the default-account startup priority fix is directionally right and has strong live proof for the normal multi-account path, but the branch still assigns startup-delay slots to duplicate-token aliases that gateway startup will skip, leaving a known Discord startup-delay case unresolved.

## What This Changes

The PR changes Discord gateway startup delay ordering so the configured Discord default account is promoted to the first startup slot before secondary accounts are staggered.

## Best Possible Solution

Compute Discord startup slots from the same runtime-enabled and configured account set that gateway startup can actually start, then promote the configured default only if it survives that filter.

## Reproduction Assessment

Yes from source: gateway startup skips duplicate-token aliases via `plugin.config.isEnabled`, while the PR's startup-slot helper still includes accounts that are enabled/configured but not runtime-enabled.

## Solution Assessment

No as submitted; promoting `defaultAccount` is the right fix direction, but the maintainable fix must share the gateway's startable-account filter before assigning stagger slots.
