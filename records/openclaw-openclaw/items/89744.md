## Summary

Keep open: the default-account-first fix is the right direction and the supplied live Discord proof covers the main path, but the patch still builds delay slots from token-present accounts instead of the same runtime-enabled account set the gateway actually starts.

## What This Changes

The PR reorders Discord multi-account startup delays so the configured `channels.discord.defaultAccount` gets the first startup slot, with a focused regression test.

## Best Possible Solution

Keep the default-account-first behavior, but derive startup delay slots from the same runtime-enabled account set the gateway starts, including duplicate-token filtering.

## Reproduction Assessment

Yes. Source inspection shows gateway startup skips accounts via `plugin.config.isEnabled`, while the PR's delay helper still includes duplicate-token aliases; a config with `alpha` and `zzz` sharing a token and `defaultAccount: "zzz"` would make `alpha` wait one stagger.

## Solution Assessment

No as submitted. Promoting the configured default is the right owner-boundary fix, but the helper should first mirror runtime-enabled filtering so skipped aliases cannot influence provider startup slots.
