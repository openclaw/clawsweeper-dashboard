## Summary

Keep open: this PR targets a real Discord multi-account startup bug, but the current diff is not merge-ready because a duplicate-token configured default account can still consume the first startup slot even though the gateway skips that account.

## What This Changes

The PR changes Discord multi-account startup delay ordering so `channels.discord.defaultAccount` receives the first startup slot before secondary accounts.

## Best Possible Solution

Keep the plugin-local default-account-first fix, but derive the delay list from Discord runtime-enabled accounts before default promotion and cover duplicate-token defaults.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure accounts ordered `work`, `zdefault` with the same token and `channels.discord.defaultAccount: "zdefault"`; the gateway skips `zdefault`, but the PR helper still promotes it before calculating `work`'s delay.

## Solution Assessment

No, not yet. The fix belongs in the Discord plugin startup-delay helper, but the best implementation must reuse the runtime-enabled account predicate before default promotion instead of maintaining a looser candidate filter.
