## Summary

Keep open: the PR fixes the main default-account ordering path, but the startup candidate list still diverges from the gateway runtime-enabled account contract, so disabled duplicate-token Discord accounts can consume stagger slots before or among accounts that actually start.

## What This Changes

The PR promotes the configured Discord default account to the first startup-delay slot and adds a channel test for reordered stagger timing.

## Best Possible Solution

Keep the default-account-first behavior, but compute stagger slots from configured Discord accounts that pass the same runtime-enabled gate used by gateway startup, with duplicate-token regression coverage.

## Reproduction Assessment

Yes. Source inspection shows PR head's startup helper can include duplicate-token accounts that gateway startup marks disabled, so a DISCORD_BOT_TOKEN plus same-token configured account can create a phantom stagger slot.

## Solution Assessment

No. Promoting the configured default account is the right direction, but the best fix is to derive delay slots from the same configured and runtime-enabled account set that gateway startup uses.
