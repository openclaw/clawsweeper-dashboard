## Summary

Keep open: the PR targets a real Discord multi-account startup bug, but it is not merge-ready because the new startup-slot helper can still count duplicate-token aliases that gateway startup skips, leaving the real default bot delayed.

## What This Changes

The PR changes Discord startup delay ordering to promote the configured default account before secondary accounts and adds a focused regression test.

## Best Possible Solution

Use one Discord startup candidate list that filters to runtime-enabled accounts, promotes the resolved default only when that account will actually start, and preserves existing stagger behavior for secondary accounts.

## Reproduction Assessment

Yes, from source inspection. Gateway startup skips duplicate-token aliases through `plugin.config.isEnabled`, while the PR helper still counts enabled/configured aliases as startup slots.

## Solution Assessment

No as submitted. Promoting the configured default account is the right layer, but the helper needs to share Discord's runtime-enabled account predicate before this is the best fix.
