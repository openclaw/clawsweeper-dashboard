## Summary

Keep this PR open for maintainer review: it fixes the central Discord multi-account startup ordering bug with a narrow owner-local change and strong live proof, but I found one low-priority edge where the new ordering can count a runtime-disabled duplicate-token default account as the first startup slot.

## What This Changes

The branch promotes the resolved Discord default account to the front of Discord startup delay ordering and adds a focused regression test for secondary account staggering.

## Best Possible Solution

Keep the owner-local Discord startup-priority fix, but align the startup slot list with Discord's runtime-enabled account predicate before merge so duplicate-token disabled accounts cannot reserve a delay slot.

## Reproduction Assessment

Yes. Source inspection shows current main assigns delay slots from sorted/listed Discord account ids, and the linked issue gives a concrete multi-account config where `channels.discord.defaultAccount` starts later than secondary accounts; I did not run live Discord in this read-only review.

## Solution Assessment

Mostly yes. The PR uses the existing Discord default-account resolver in the Discord plugin boundary, which is the right layer for the central bug, but the finished version should use the same runtime-enabled filtering that gateway startup uses for duplicate-token accounts.
