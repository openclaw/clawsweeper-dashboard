## Summary

Keep open: the PR fixes the normal configured-default ordering case and has good live proof, but it still builds startup-delay slots from accounts that Discord gateway startup can skip, leaving a supported duplicate-token alias setup with the same delayed-default failure.

## What This Changes

The PR promotes the resolved Discord default account ahead of secondary Discord accounts when calculating multi-account startup stagger delays and adds a focused channel test for that order.

## Best Possible Solution

Make Discord startup-delay ordering use the same runtime-enabled account set as gateway startup before applying default-account promotion, then cover the duplicate-token alias case in tests.

## Reproduction Assessment

Yes. The remaining blocker is source-reproducible: gateway startup filters through plugin.config.isEnabled before startAccount, while the PR's delay-slot helper still counts accounts using only enabled/configured token checks.

## Solution Assessment

No. Default-account promotion is the right fix shape, but it needs to operate on Discord's runtime-enabled account list so the delay calculation matches the gateway startup boundary.
