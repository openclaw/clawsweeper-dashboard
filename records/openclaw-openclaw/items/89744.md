## Summary

Keep open. ## What This Changes

The PR promotes the configured Discord default account into the first startup slot before staggering secondary Discord accounts, with a focused regression test for the ordering.

## Best Possible Solution

Land a repaired version of this branch, or a proof-positive replacement, that promotes the configured default account only after filtering startup slots through Discord runtime eligibility.

## Reproduction Assessment

Yes. Source inspection shows current main sorts Discord account ids independently of `defaultAccount`, and the PR body supplies live Discord gateway logs where a configured default account that would have been index 3 starts immediately after the patch; the remaining duplicate-token defect is also source-reproducible from the mismatch between gateway `isEnabled` and the PR's manual delay-list filter.

## Solution Assessment

No, not as submitted. Promoting `defaultAccount` is the right owner-boundary fix, but the startup ordering must reuse `isDiscordAccountEnabledForRuntime` so the delay list and gateway startup agree on which accounts can actually run.
