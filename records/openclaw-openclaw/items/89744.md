## Summary

Keep open: the PR fixes the basic configured-default Discord startup case and has strong live log proof, but the helper still uses a broader configured-account filter than gateway startup, so duplicate-token aliases can reserve the zero-delay slot while the real runtime-started account remains delayed.

## What This Changes

The PR changes Discord startup delay ordering to promote the configured default account before secondary accounts and adds a focused regression test for default-first staggering.

## Best Possible Solution

Use one Discord startup candidate list that filters to runtime-enabled accounts, promotes the resolved default only when that account will actually start, and preserves the existing stagger for secondary accounts.

## Reproduction Assessment

Yes. Current source and latest release compute Discord startup delay from listed account order, and the remaining PR blocker is source-reproducible because gateway startup skips duplicate-token aliases while the PR helper still counts them.

## Solution Assessment

No as submitted. Promoting the configured default account is the right layer, but the helper needs to share Discord's runtime-enabled account predicate before this is the best fix.
