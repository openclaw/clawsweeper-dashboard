## Summary

Keep open; the PR fixes the central Discord default-account priority path with strong live proof, but it still calculates startup slots from accounts the gateway may skip as runtime-disabled duplicate-token aliases, so an existing default alias setup can delay the actual bot account during restart.

## What This Changes

The PR promotes `channels.discord.defaultAccount` to the first Discord multi-account startup delay slot and adds a focused channel test for that ordering.

## Best Possible Solution

Land the default-account priority fix after the startup-slot helper orders the same runtime-enabled Discord accounts that gateway startup uses, with regression coverage for duplicate-token default aliases and normal secondary staggering.

## Reproduction Assessment

Yes: current main's delay helper indexes Discord accounts in `listDiscordAccountIds` order, and the supplied live logs show a configured default account could otherwise land in a later slot. The remaining duplicate-token alias gap is source-reproducible from the mismatch between gateway `isEnabled` filtering and the PR helper's local predicate.

## Solution Assessment

No: promoting the configured default account in the Discord plugin is the right boundary, but the helper should order the same runtime-enabled accounts that gateway startup actually starts. The current PR is a good partial fix but not the best mergeable shape yet.

- **[P1] Filter startup slots through runtime-enabled accounts:** `extensions/discord/src/channel.ts:223-230`
