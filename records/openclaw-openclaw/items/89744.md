## Summary

Keep this PR open: the central Discord default-account startup fix is useful and has strong live proof, but the current helper can still let a runtime-disabled duplicate-token default account consume the first startup slot and delay the real enabled account.

## What This Changes

The PR promotes the configured Discord default account to the first startup slot and adds a regression test for secondary account staggering.

## Best Possible Solution

Build Discord startup slots from the same configured and runtime-enabled account set that the gateway actually starts, then promote the default account only if it survived that filtering.

## Reproduction Assessment

Yes. The linked bug is source-reproducible from current main because Discord startup delay follows sorted `listDiscordAccountIds`, and the PR body supplies live logs showing the configured default account would otherwise have been delayed to index 3.

## Solution Assessment

No, not yet. Promoting the configured default is the right direction, but the startup list should first match the gateway's runtime-enabled account filtering so skipped duplicate-token accounts cannot consume delay slots.

- **[P2] Filter skipped Discord accounts before promoting the default:** `extensions/discord/src/channel.ts:232-238`
  - body: This promotes `defaultAccount` after filtering only for `candidate.enabled` and configured token state, but gateway startup skips accounts through `plugin.config.isEnabled`, and Discord uses that hook to disable duplicate-token accounts.
