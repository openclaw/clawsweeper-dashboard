## Summary

Keep open: the default-account startup bug is real, but this PR is not the best fix yet because its new startup slot list does not match Discord gateway runtime eligibility, and the linked replacement PR is still open without positive real behavior proof.

## What This Changes

The PR promotes `channels.discord.defaultAccount` to the first Discord multi-account startup slot and adds a regression test for secondary account staggering.

## Best Possible Solution

Build Discord startup slots from the same runtime-enabled predicate used by gateway startup, promote `defaultAccount` only inside that eligible set, and keep focused coverage for normal default ordering plus duplicate-token defaults.

## Reproduction Assessment

Yes from source, though I did not run the test suite in this read-only review. Current main sorts Discord account ids for startup delay, so a configured default account that sorts later can be delayed behind secondary accounts.

## Solution Assessment

No. Promoting the configured default account is the right direction, but this PR duplicates a narrower eligibility check instead of reusing Discord's runtime-enabled predicate, which leaves a duplicate-token startup slot bug.
