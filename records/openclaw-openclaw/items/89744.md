## Summary

Keep open: the PR fixes the ordinary configured-default Discord account case, but it still computes startup-delay slots from configured accounts rather than the runtime-enabled accounts the gateway can actually start, leaving the duplicate-token alias case unresolved.

## What This Changes

The PR promotes Discord's resolved default account to the first startup-delay slot and adds a focused regression test for secondary-account staggering.

## Best Possible Solution

Compute Discord startup slots from accounts that are enabled, configured, and not disabled by duplicate-token ownership, then promote the configured default only if it survives that startable-account filter.

## Reproduction Assessment

Yes. Source inspection shows gateway startup skips duplicate-token aliases through `plugin.config.isEnabled`, while the PR's new startup-slot helper still includes enabled/configured aliases before assigning stagger delays.

## Solution Assessment

No as submitted. Promoting `defaultAccount` is the right direction, but the maintainable fix must build the delay list from the same startable account set that gateway startup uses.
