## Summary

Keep open: the PR fixes the normal default-account ordering case, but the merge result can still leave the runtime-started Discord account delayed when `defaultAccount` points at a duplicate-token alias that gateway startup skips.

## What This Changes

The PR promotes the resolved Discord `defaultAccount` to the first startup slot and adds a regression test for default-before-secondary staggering.

## Best Possible Solution

Use one Discord startup candidate list that filters to runtime-enabled accounts, promotes the resolved default only when that account will actually start, and preserves the existing stagger for enabled secondary accounts.

## Reproduction Assessment

Yes. Source inspection shows gateway startup skips duplicate-token aliases through `plugin.config.isEnabled`, while the PR helper still counts enabled/configured aliases as startup slots.

## Solution Assessment

No as submitted. Promoting the configured default account is the right layer, but the helper needs to share Discord's runtime-enabled account predicate before this is the best fix.
