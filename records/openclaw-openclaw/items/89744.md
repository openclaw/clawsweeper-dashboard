## Summary

Keep this PR open, but it still needs changes before merge: the default-account promotion is the right Discord owner-boundary fix, yet the new startup slot list can still count duplicate-token aliases that gateway startup will skip.

## What This Changes

This PR promotes the resolved Discord `channels.discord.defaultAccount` to the first startup slot and adds a focused channel test for default-first staggering.

## Best Possible Solution

Use one Discord startup ordering path that filters to configured runtime-enabled accounts, promotes the resolved default account only when that account will actually start, and preserves the existing stagger for secondary accounts.

## Reproduction Assessment

Yes. Current source shows the original sorted-account delay path, and the remaining blocker is source-reproducible because gateway startup skips duplicate-token aliases via `isDiscordAccountEnabledForRuntime` while the PR helper does not.

## Solution Assessment

No as submitted. Promoting `defaultAccount` is the right layer, but the helper needs to use the same runtime-enabled account predicate as gateway startup before it is the best fix.
