## Summary

Keep open: current main still has the Discord default-account startup bug, and the open replacement PR is not yet a proof-positive superseding target. This branch has strong real Discord proof, but the patch needs one narrow repair before merge because its startup slot list can still promote a duplicate-token disabled default account.

## What This Changes

The PR promotes the configured Discord default account into the first startup slot before staggering secondary accounts and adds a focused startup-order regression test.

## Best Possible Solution

Land a review-clean, proof-positive patch that builds Discord startup slots from the runtime-enabled account predicate and promotes the configured default account only when that account is actually eligible to start.

## Reproduction Assessment

Yes from source inspection and PR proof: current main indexes Discord startup slots by configured account order, while the PR's live logs show a configured default account that would otherwise be delayed to index 3. I did not run a live Discord gateway locally in this read-only review because no Discord secrets are available.

## Solution Assessment

No, not as submitted: default-account promotion is the right behavior, but the slot list should first use isDiscordAccountEnabledForRuntime so duplicate-token or otherwise skipped accounts cannot take the zero-delay slot.
