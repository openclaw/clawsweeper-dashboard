## Summary

Keep open for maintainer review: the PR looks like a narrow Discord-local fix with sufficient real runtime proof and focused tests, but it intentionally changes startup ordering for existing multi-account Discord configs and needs maintainer acceptance before merge.

## What This Changes

The PR changes Discord startup delay calculation to give an eligible configured `defaultAccount` the zero-delay startup slot, with regressions for default-first, duplicate-token, and unconfigured-account cases.

## Best Possible Solution

Land the Discord-local fix after maintainers accept that an eligible configured `channels.discord.defaultAccount` should own the zero-delay startup slot while secondary accounts keep the existing stagger.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.6` shows Discord startup delay still follows the filtered account-list index, so a configured default account that sorts later can be delayed.

## Solution Assessment

Yes, with maintainer acceptance of the behavior change. The PR stays in the Discord plugin, reuses the existing runtime-enabled and configured predicates, and adds focused regressions for the prior predicate gaps.
