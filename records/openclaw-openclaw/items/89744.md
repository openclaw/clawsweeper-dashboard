## Summary

Keep open: this is the canonical proof-backed candidate fix for the linked Discord default-account startup issue. I found no concrete patch defect, but the startup-order compatibility change still needs maintainer acceptance before merge.

## What This Changes

The PR promotes a runtime-eligible configured Discord default account to the first startup slot and adds regressions for default-account, duplicate-token, and unconfigured-account startup cases.

## Best Possible Solution

Land the Discord-local fix after maintainer acceptance that an eligible configured default account should own the zero-delay startup slot while secondary accounts keep the existing stagger.

## Reproduction Assessment

Yes. Current main and v2026.6.9 compute Discord startup delay from the sorted filtered account list, so a configured default account that sorts later can receive a nonzero startup delay.

## Solution Assessment

Yes as an implementation shape, with maintainer acceptance required for upgrade behavior. The PR stays inside the Discord plugin startup path and reuses existing default-account, runtime-enabled, and configured-token predicates instead of hard-coding account names or adding config.
