## Summary

Keep open: the default-account startup fix is useful and has real Discord proof, but the patch still lets a runtime-disabled duplicate-token default account occupy the first stagger slot and delay the account the gateway will actually start.

## What This Changes

The PR changes Discord startup-delay ordering so the configured defaultAccount is moved to the first startup slot before secondary accounts, with a focused channel test.

## Best Possible Solution

Keep the defaultAccount-first behavior, but compute stagger slots from the same runtime-enabled Discord account set that gateway startup uses, with duplicate-token regression coverage.

## Reproduction Assessment

Yes from source inspection: the delay list uses listDiscordAccountIds plus enabled/configured checks while gateway startup skips accounts through plugin.config.isEnabled. A duplicate-token defaultAccount can therefore occupy the first slot even though it will not start.

## Solution Assessment

No as-is. The owner boundary and defaultAccount-first idea are right, but the best fix must reuse Discord's runtime-enabled account contract so skipped duplicate-token accounts cannot consume stagger slots.
