## Summary

Keep open: current main and the latest release still need default-account-first Discord startup, but this PR needs a narrow fix because the merge result still builds delay slots from enabled/configured accounts rather than runtime-enabled accounts, so duplicate-token aliases can reserve slots.

## What This Changes

The PR promotes the resolved Discord default account to the first startup delay slot and adds a regression test for default-before-secondary ordering.

## Best Possible Solution

Use one Discord startup candidate list that filters to runtime-enabled accounts, promotes the resolved default only when that account will actually start, and preserves the existing stagger for enabled secondary accounts.

## Reproduction Assessment

Yes, from source inspection. Gateway startup skips duplicate-token aliases through plugin.config.isEnabled, while the PR helper still counts enabled/configured aliases as startup slots.

## Solution Assessment

No as submitted. Promoting the configured default account is the right layer, but the helper needs to share Discord's runtime-enabled account predicate before this is the best fix.
