## Label Justifications

- `P2`: This is a bounded Discord multi-account startup bug with real user impact but limited blast radius.
- `merge-risk: 🚨 availability`: The new default promotion can let a duplicate-token account that gateway startup skips consume the immediate startup slot, delaying the real Discord monitor.

## Summary

Keep open: the central fix is useful and has live proof for the normal multi-account default case, but the patch still diverges from Discord's runtime-enabled account filtering and can assign a stagger slot to a duplicate-token account the gateway will skip.

## What This Changes

The PR adds a Discord startup ordering helper that moves the configured default account to the first stagger slot and adds a regression test for that ordering.

## Best Possible Solution

Keep the default-account-first behavior, but make the startup delay list use the same duplicate-token-aware runtime-enabled account predicate as gateway startup before assigning stagger slots.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: configure two enabled Discord accounts with the same token, set `defaultAccount` to the later duplicate that loses token ownership, and the PR helper can still promote that skipped account into slot zero while the gateway starts only the owner.

## Solution Assessment

No. The PR is close, but the delay helper uses a configured-token predicate instead of the runtime-enabled account set used by gateway startup, so the narrow maintainable fix is to reuse that predicate and cover the duplicate-token default case.
