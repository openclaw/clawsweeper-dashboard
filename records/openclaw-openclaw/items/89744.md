## What This Changes

This PR promotes `channels.discord.defaultAccount` to Discord's zero-delay startup slot before applying the existing stagger, and adds focused ordering coverage.

## Best Possible Solution

Use Discord's shared runtime-enabled predicate for the startup slot list, keep the configured default first only when it is runtime-enabled, and land either this repaired branch or a proof-positive merged replacement.

## Reproduction Assessment

Yes by source inspection, without executing a local repro in this read-only review. The gateway skips accounts through `plugin.config.isEnabled`, while the PR's delay list still counts enabled/configured duplicate-token accounts, so a duplicate-token default can consume the first slot and delay the actual runtime-enabled bot.

## Solution Assessment

No. The PR is a good partial fix, but the best fix is to use the existing `isDiscordAccountEnabledForRuntime` predicate so the delay order and gateway startup eligibility share one source of truth.
