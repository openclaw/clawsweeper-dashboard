## Summary

Keep open: the normal default-account startup fix is useful and the real Discord proof is strong, but the PR still promotes the configured default before excluding Discord accounts that current main will skip at runtime, so a duplicate-token default can still delay the real provider account.

## What This Changes

This PR changes Discord gateway startup delay ordering so the configured Discord default account is placed before secondary accounts, and adds a focused channel test.

## Best Possible Solution

Keep the default-account-first behavior, but compute Discord startup delay slots from accounts the gateway will actually start, then add the focused duplicate-token default regression before merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Discord config with duplicate-token accounts where `channels.discord.defaultAccount` names the skipped duplicate; the gateway skips that account through `isDiscordAccountEnabledForRuntime`, while the PR's delay helper can still promote it into the first slot.

## Solution Assessment

No. Promoting the configured default is the right direction, but the maintainable fix must first align the delay candidate list with the same runtime-enabled predicate the gateway uses before calling `startAccount`.
