## Summary

Keep open: the PR is pointed at the right Discord startup bug and includes strong live proof, but the patch still assigns startup slots before applying Discord's runtime-enabled duplicate-token filter, so a skipped default alias can consume the zero-delay slot and leave the real account delayed.

## What This Changes

This PR promotes the resolved Discord default account to the first startup slot before applying secondary account stagger delays, with a focused regression test.

## Best Possible Solution

Keep the default-account promotion, but derive startup slots from Discord accounts that gateway would actually start, then prove duplicate-token default aliases do not consume delay slots.

## Reproduction Assessment

Yes. Current main computes Discord startup delay from sorted configured account ids, so a configured default account that sorts later than secondary enabled accounts is source-reproducible; I did not run a live Discord restart in this read-only review.

## Solution Assessment

No. Promoting defaultAccount is the right direction, but the best fix must use the same runtime-enabled account predicate as gateway startup so skipped duplicate-token accounts cannot affect delay slots.
