## Summary

Keep this PR open: it addresses the reported configured-default Discord startup ordering and the live proof is strong, but the new startup-slot helper still counts Discord accounts that gateway startup skips through the runtime `isEnabled` path, leaving duplicate-token default fallback setups able to delay the first actually-started account.

## What This Changes

The PR reorders Discord multi-account startup delay slots so the configured `defaultAccount` starts before secondary accounts and adds a focused channel test.

## Best Possible Solution

Build the Discord delay-slot list from the same runtime-enabled account set used by gateway startup, then keep the configured default account first among only those startable accounts.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure an implicit default Discord token and a named account with the same token, then the gateway skips the default account via `isEnabled` while the PR helper still counts it in the delay slot list.

## Solution Assessment

No, not yet: promoting `defaultAccount` is the right fix direction, but the implementation should reuse Discord's canonical runtime-enabled predicate instead of a partial enabled/configured check.
