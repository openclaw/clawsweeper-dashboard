- `merge-risk: 🚨 compatibility`: The diff intentionally changes startup behavior for existing Discord multi-account configs that set `channels.discord.defaultAccount`.

## Summary

Keep open: current main and the latest release still do not promote the configured Discord default account into the first startup slot, while this PR is a narrow proof-backed fix. The remaining decision is maintainer acceptance of the intentional startup-order compatibility and availability risk.

## What This Changes

This PR changes Discord startup delay calculation to place an eligible configured `defaultAccount` in the zero-delay startup slot and adds focused regression coverage for default-first, duplicate-token, and unconfigured-account cases.

## Best Possible Solution

Land the Discord-local fix after maintainers accept that an eligible configured `channels.discord.defaultAccount` should own the zero-delay startup slot while secondary accounts keep the existing stagger.

## Reproduction Assessment

Yes. Source inspection on current main shows Discord startup delay still uses sorted/list account order without promoting the resolved default account, so a configured default account such as `main` can be delayed behind accounts like `billy` or `farber`.

## Solution Assessment

Yes, as an implementation shape. The PR keeps the fix inside the Discord plugin, reuses the existing runtime-enabled and configured/token predicates, and avoids adding a new config surface; maintainers still need to accept the upgrade behavior.
