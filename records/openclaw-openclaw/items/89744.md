## Summary

Keep open: the PR fixes the normal configured-default Discord startup ordering path, but the patch still builds the delay list without the Discord runtime-enabled duplicate-token predicate, so a skipped duplicate account can consume slot 0 and delay the only account that actually starts.

## What This Changes

The PR promotes the configured Discord default account to the first startup slot and adds a regression test for secondary account staggering.

## Best Possible Solution

Use one Discord startup account list that applies the same runtime-enabled and configured-account filtering as gateway startup, then promote the configured default only if it remains in that list and cover the duplicate-token case in tests.

## Reproduction Assessment

Yes. By source inspection, current main delays by listDiscordAccountIds order, and the PR branch still includes duplicate-token skipped accounts in the delay list even though gateway startup will not call startAccount for those skipped accounts.

## Solution Assessment

No. The direction is right, but the best fix is to build the delay order from the same runtime-enabled account set that gateway startup uses and promote the default only within that filtered set.
