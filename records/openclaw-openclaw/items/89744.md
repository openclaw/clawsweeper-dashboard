## Summary

Keep open: the PR addresses the reported default-account ordering path and has convincing live proof, but the helper still builds delay slots from basic enabled/token state instead of the Discord runtime-enabled account contract, so runtime-disabled duplicate-token defaults can still consume the first slot.

## What This Changes

The PR promotes Discord `channels.discord.defaultAccount` to the first gateway startup stagger slot and adds a focused regression test for default-account-first ordering.

## Best Possible Solution

Keep the default-account promotion, but compute Discord startup slots from the same runtime-enabled account set the gateway uses so disabled and duplicate-token accounts cannot consume stagger positions.

## Reproduction Assessment

Yes. Current source shows the original default-account startup ordering issue, the PR body includes after-fix live Discord logs, and the remaining PR defect is source-reproducible through the existing duplicate-token runtime-enabled predicate.

## Solution Assessment

No. The direction is right, but the best fix is to share the existing Discord runtime-enabled account contract before assigning stagger slots instead of using only `candidate.enabled` and token presence.
