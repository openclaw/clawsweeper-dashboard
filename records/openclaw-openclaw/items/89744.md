## Summary

Keep open: this PR addresses a real Discord default-account startup-order bug with strong live proof, but it is not merge-ready because the new slot-order helper still counts accounts that the gateway later disables for duplicate tokens.

## What This Changes

The PR changes Discord multi-account startup delay ordering so the configured Discord default account is promoted to the first startup slot before secondary accounts.

## Best Possible Solution

Merge this behavior only after Discord startup slots are ordered from configured, runtime-enabled accounts, with the configured default first only when it is actually startable and a duplicate-token regression covers the skipped-default case.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main derives Discord startup delay from configured account order, while the PR proof shows a configured default account would otherwise be delayed. The new review finding is also source-reproducible from the gateway `isEnabled` skip path plus Discord duplicate-token tests.

## Solution Assessment

No, not as submitted. Promoting the configured default account inside the Discord plugin is the right layer, but the helper must reuse the existing runtime-enabled account contract before assigning delay slots.
