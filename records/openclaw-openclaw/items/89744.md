## Summary

Keep open: the PR fixes the right Discord multi-account startup bug and has strong live log proof, but the current patch still computes startup slots from enabled/configured token state instead of the gateway runtime-enabled account contract, so duplicate-token default accounts can still delay the actual bot.

## What This Changes

This PR promotes the configured Discord default account to the first gateway startup stagger slot and adds a regression test for default-account-first ordering.

## Best Possible Solution

Keep the default-account promotion, but compute Discord startup slots from the same runtime-enabled account set the gateway uses so disabled and duplicate-token accounts cannot consume stagger positions.

## Reproduction Assessment

Yes. Source inspection shows the remaining PR defect: the new startup slot helper filters on enabled/configured token state while gateway startup skips accounts through Discord's runtime-enabled predicate; I did not run a live Discord restart in this read-only review.

## Solution Assessment

No. The direction is right, but the best fix is to share the existing Discord runtime-enabled account contract before assigning stagger slots instead of using only candidate.enabled and token presence.
