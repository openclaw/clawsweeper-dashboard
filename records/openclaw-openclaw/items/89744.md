## Summary

Keep this PR open: it fixes the right Discord multi-account startup problem and has solid live proof, but the proposed helper still assigns delay slots to Discord accounts the gateway will not actually start, so a duplicate-token default account can delay the real runtime-enabled bot.

## What This Changes

This PR changes Discord gateway startup delay ordering so `channels.discord.defaultAccount` is moved to the first stagger slot and adds a regression test for that ordering.

## Best Possible Solution

Keep the default-account promotion, but compute Discord startup slots from the same runtime-enabled account set the gateway uses so disabled and duplicate-token accounts cannot consume stagger positions.

## Reproduction Assessment

Yes. Current source shows main computes delay from account-list order, and the linked issue plus PR logs show a configured default account can otherwise start late; the remaining PR bug is also source-reproducible through the existing duplicate-token runtime-enabled predicate.

## Solution Assessment

No. The direction is right, but the best fix is to share the existing Discord runtime-enabled account contract before assigning stagger slots, instead of using only `candidate.enabled` and token presence.
