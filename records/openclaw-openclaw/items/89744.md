## Summary

Keep this PR open with changes required: the happy-path default-first fix is useful, but the merged code can still assign the immediate startup slot to a Discord account alias that current gateway startup will skip, leaving the actual runtime-started bot delayed.

## What This Changes

The PR promotes the configured Discord default account to the first multi-account startup slot and adds a focused Discord channel test for default-first staggering.

## Best Possible Solution

Use one startup account ordering path that first filters to configured runtime-enabled Discord accounts, then promotes the resolved default account only when that same account will actually be started.

## Reproduction Assessment

Yes from source inspection: current main derives Discord startup delay from account-list order, so a configured default account sorted after other enabled/configured accounts receives a delayed slot.

## Solution Assessment

No as submitted: promoting the configured default account is the right owner-boundary fix, but the helper must use the same runtime-enabled account filter as gateway startup before it is the best solution.
