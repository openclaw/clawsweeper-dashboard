## Summary

Keep open: the PR fixes the main Discord default-account startup ordering path and includes strong live proof, but the submitted helper still uses an enabled/configured account list instead of the same runtime-enabled predicate gateway startup uses, so a skipped duplicate-token alias can still consume the zero-delay slot.

## What This Changes

The PR reorders Discord multi-account startup delays so the resolved `channels.discord.defaultAccount` gets the zero-delay slot and adds a channel test for default-first staggering.

## Best Possible Solution

Use one Discord startup ordering path that filters to configured runtime-enabled accounts, promotes the resolved default account only when that account will actually start, and preserves the existing stagger for secondary accounts.

## Reproduction Assessment

Yes. Current main source-reproduces the original default-account delay through sorted account ids, and the remaining PR blocker is source-reproducible because gateway skips duplicate-token aliases via `isDiscordAccountEnabledForRuntime` while the new slot helper does not.

## Solution Assessment

No as submitted. Promoting `defaultAccount` is the right owner-boundary fix, but the helper needs to use the same runtime-enabled account predicate as gateway startup before it is the best solution.
