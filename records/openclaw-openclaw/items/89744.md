## Summary

Keep open: this is the right Discord startup surface and it fixes the normal configured-default ordering case, but the branch still promotes a default account that the gateway would skip as duplicate-token disabled, which can leave the real token owner behind a startup delay.

## What This Changes

The PR changes Discord gateway startup delay ordering to move the configured default account ahead of secondary accounts and adds a focused channel test for that ordering.

## Best Possible Solution

Filter Discord startup ordering with the same runtime-enabled predicate used by gateway startup before promoting the default account, then cover the duplicate-token default/owner case.

## Reproduction Assessment

Yes. Current main and v2026.6.1 compute Discord startup delay from sorted/listed account ids, so a configured default account that sorts after secondary accounts is delayed; the PR body also supplies live logs for the fixed distinct-account path.

## Solution Assessment

No, not as-is. The Discord plugin startup helper is the right layer, but the implementation must reuse the runtime-enabled account set before default promotion so skipped duplicate-token accounts cannot consume startup slots.
