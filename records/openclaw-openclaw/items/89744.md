## What This Changes

The PR changes Discord multi-account startup ordering so the configured default account receives the first startup slot and adds a regression test for secondary account staggering.

## Best Possible Solution

Land a review-clean, proof-positive patch that builds Discord startup slots from the runtime-enabled account predicate and promotes the configured default account only when it is actually eligible to start.

## Reproduction Assessment

Yes from source inspection and PR proof: current main indexes Discord startup slots by configured account order, while the PR's live logs show a configured default account that would otherwise be delayed to index 3. ## Solution Assessment

No, not as submitted: default-account promotion is the right behavior, but the slot list should first use isDiscordAccountEnabledForRuntime so duplicate-token or otherwise skipped accounts cannot take the zero-delay slot.
