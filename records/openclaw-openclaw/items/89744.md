## Summary

Keep open: the PR is a useful fix with strong live Discord proof, but the new startup-order helper still counts accounts that gateway startup skips as runtime-disabled, so duplicate-token configs can assign extra stagger delay to a real secondary account.

## What This Changes

The PR promotes the configured Discord default account to the first startup slot and adds a regression test for secondary account staggering.

## Best Possible Solution

Promote the configured default account after filtering to the same runtime-enabled and configured account set that gateway startup uses, with a regression test for duplicate-token skipped accounts.

## Reproduction Assessment

Yes, from source inspection. Current main sorts Discord account ids before assigning startup delay slots, and the linked issue provides a concrete multi-account config where channels.discord.defaultAccount starts after secondary accounts; I did not run a live Discord restart in this read-only review.

## Solution Assessment

No, not quite. Promoting the configured default account is the right layer, but the startup-order helper should derive slots from the same runtime-enabled predicate used by gateway startup so skipped duplicate-token accounts do not affect delays.
