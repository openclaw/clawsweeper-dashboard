## Summary

Keep open: the PR addresses a real Discord multi-account startup bug and includes convincing live proof, but the new startup-slot helper still disagrees with Discord's existing runtime-enabled account predicate, so a disabled duplicate-token default account can consume the zero-delay slot and delay the real bot.

## What This Changes

The PR promotes the configured Discord default account to the first startup slot before staggering secondary Discord accounts and adds focused channel test coverage.

## Best Possible Solution

Merge the default-account startup priority only after the startup-slot list is built from configured, runtime-enabled Discord accounts, with regression coverage for a duplicate-token configured default account.

## Reproduction Assessment

Yes. Current main derives Discord startup delay from sorted account ids, the linked issue and PR proof show the configured default account can be delayed, and the remaining blocker is source-reproducible from the gateway isEnabled skip path plus duplicate-token tests.

## Solution Assessment

No, not as submitted. Promoting the configured default account inside the Discord plugin is the right layer, but the helper must reuse the existing runtime-enabled account contract before assigning delay slots.
