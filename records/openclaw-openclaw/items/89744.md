## Summary

Keep open: the PR addresses a real Discord multi-account startup bug and the supplied live log proof is strong, but the implementation still builds the stagger list from configured accounts rather than Discord runtime-enabled accounts, so duplicate-token skipped accounts can consume early startup slots and leave the availability fix incomplete.

## What This Changes

The PR promotes the configured Discord default account to the first startup slot and adds a regression test for secondary stagger delays.

## Best Possible Solution

Keep the default-account promotion, but derive startup slots from Discord runtime-enabled and configured accounts, then add a duplicate-token regression and rerun the focused Discord validation.

## Reproduction Assessment

Yes. Current main computes Discord startup delay from sorted configured account ids, so a configured default account that sorts later than secondary enabled accounts is source-reproducible without adding a new product mode.

## Solution Assessment

No. Promoting defaultAccount is the right direction, but the best fix must use the same runtime-enabled account predicate as gateway startup so skipped duplicate-token accounts cannot affect delay slots.
