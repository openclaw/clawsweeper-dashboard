## Summary

Keep this PR open, but it still needs changes before merge: the default-account promotion is the right direction, while the new startup ordering list still counts Discord accounts that the gateway disables before runtime startup, leaving duplicate-token configurations with the same availability delay class.

## What This Changes

This PR adds a Discord startup ordering helper that moves the resolved default account to the first stagger slot and adds a regression test for multi-account startup delay ordering.

## Best Possible Solution

Build Discord startup ordering from the same runtime-enabled account set used by the gateway/status surfaces, then promote the configured default account only within that filtered set and cover the duplicate-token case with a focused regression.

## Reproduction Assessment

Yes. Source inspection shows current main sorts Discord account IDs alphabetically for startup delay, and the PR proof demonstrates the distinct-token default-account case; the remaining duplicate-token gap is also source-reproducible from the gateway `isEnabled` filter and Discord account predicate.

## Solution Assessment

No. Promoting the configured default account is the right fix shape, but the PR should build the ordering from runtime-enabled Discord accounts so disabled duplicate-token accounts cannot reserve delay slots.
