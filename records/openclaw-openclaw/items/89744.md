## Summary

Keep open: current main and v2026.6.10 still compute Discord startup delay from the sorted eligible account list without promoting the configured default account, while this open PR is the proof-backed candidate for the canonical linked bug. The patch looks correct, but maintainers still need to accept the compatibility/availability impact of changing startup order for existing multi-account Discord installs.

## What This Changes

The PR changes Discord startup-delay ordering so an eligible configured `channels.discord.defaultAccount` receives the zero-delay startup slot before secondary Discord accounts.

## Best Possible Solution

Land the Discord-local fix after maintainer acceptance that an eligible configured default account should own the zero-delay startup slot while secondary accounts keep the existing stagger.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 compute Discord startup delay from the sorted eligible account list, so a configured default account that sorts later can receive a nonzero delay.

## Solution Assessment

Yes as an implementation shape, with maintainer acceptance required for upgrade behavior. The PR stays inside the Discord plugin startup path and reuses existing default-account, runtime-enabled, and configured-token predicates instead of hard-coding account names or adding config.
