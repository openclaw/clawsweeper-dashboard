## Label Justifications

- `P2`: This is a normal Discord multi-account startup bugfix with limited blast radius, not an emergency core-runtime failure.
- `merge-risk: 🚨 availability`: The diff changes provider startup ordering, and the current helper can delay the real runtime-enabled Discord bot when a duplicate-token disabled account consumes the zero-delay slot.

## Summary

Keep open: the PR fixes the straightforward distinct-token default-account ordering path and has live proof, but the new startup ordering helper still uses configured/enabled checks instead of Discord's existing runtime-enabled contract, so duplicate-token configs can put a non-started account in the zero-delay slot.

## What This Changes

The branch promotes the resolved Discord defaultAccount to the first gateway startup-delay slot for enabled/configured accounts and adds a focused unit test for that ordering.

## Best Possible Solution

Build Discord startup delay order from the same runtime-enabled account set that gateway startup uses, then promote the configured default only if it is in that set and cover duplicate-token ownership with a regression test.

## Reproduction Assessment

Yes, from source inspection rather than an executed local repro. Current main computes Discord startup delay from account-list order, and the PR head still includes duplicate-token disabled accounts in the new priority list while the linked issue and PR proof show the distinct-token default-account delay path.

## Solution Assessment

No. Promoting defaultAccount is the right direction, but the best fix is to share Discord's runtime-enabled account contract so disabled duplicate-token accounts cannot consume startup slots.
