- `merge-risk: 🚨 compatibility`: The PR changes restart/startup ordering semantics for existing Discord multi-account configurations.
- `merge-risk: 🚨 availability`: A runtime-disabled default alias can delay the real enabled Discord account during gateway startup.

## Summary

Keep open: the normal-token fix is in the right Discord plugin boundary and has strong live log proof, but the PR still uses a looser enabled/configured filter than gateway startup, leaving duplicate-token default aliases able to delay the real enabled account.

## What This Changes

The PR changes the Discord plugin startup delay helper and tests so `channels.discord.defaultAccount` is promoted ahead of secondary Discord accounts during staggered gateway startup.

## Best Possible Solution

Keep the Discord-plugin fix, but build startup order from runtime-enabled, configured accounts before promoting `defaultAccount` and cover duplicate-token aliases in channel tests.

## Reproduction Assessment

Yes, source-reproducible: current main computes Discord startup delay from sorted `listDiscordAccountIds` rather than `channels.discord.defaultAccount`, and the PR proof logs show the reported normal-token case on PR head. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the owner boundary is correct, but the helper should reuse `isDiscordAccountEnabledForRuntime` or equivalent gateway-enabled semantics before promotion so non-startable aliases cannot affect delay slots.
