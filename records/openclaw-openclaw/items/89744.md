## Summary

Keep this PR open, but it still needs one correctness repair before merge: the new Discord startup ordering list does not reuse the runtime-enabled account predicate that gateway startup already uses, so duplicate-token default-account configs can still leave the real runtime account delayed.

## What This Changes

The PR changes Discord multi-account startup delay ordering so the configured `channels.discord.defaultAccount` is promoted to the first startup slot, with a regression test for distinct-token secondary accounts.

## Best Possible Solution

Compute Discord startup order from the same runtime-enabled account set used by gateway startup, then promote the configured default only when that account is actually started, with a duplicate-token regression test.

## Reproduction Assessment

Yes, from source inspection: gateway startup filters Discord accounts through `plugin.config.isEnabled`, while the PR's new delay list does not, so a duplicate-token default account can remain in the delay calculation after the gateway skips it. I did not run a live current-main repro in this read-only review.

## Solution Assessment

No, the proposed shape is close but not yet the best fix because it introduces a second account-startability predicate. The maintainable fix is to reuse Discord's runtime-enabled account set for delay ordering and keep the new default-account promotion on top of that set.
