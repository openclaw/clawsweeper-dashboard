## Summary

Keep open: current main still computes Discord startup delay from account-list order, while this proof-backed PR makes an eligible configured default account start first and superseded PR https://github.com/openclaw/openclaw/pull/90660 is closed in favor of this branch. No blocking patch defect surfaced; maintainer review should accept the intentional startup-order compatibility risk before landing.

## What This Changes

This PR changes Discord startup delay calculation to put an eligible configured `channels.discord.defaultAccount` in the zero-delay slot while preserving staggered secondary accounts and adds Discord regression tests for default, duplicate-token, and unconfigured-account cases.

## Best Possible Solution

Land the Discord-local fix after maintainers accept that an eligible configured `channels.discord.defaultAccount` should own the zero-delay startup slot for existing multi-account deployments.

## Reproduction Assessment

Yes. Source inspection of current main shows Discord startup delay follows account-list order, so a configured default account that sorts later is delayed despite the documented `defaultAccount` preference.

## Solution Assessment

Yes. The PR keeps the fix inside the Discord plugin, composes the existing configured/token check with the runtime-enabled predicate, and rejects the closed replacement path that dropped part of that guard.
