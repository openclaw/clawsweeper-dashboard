## Summary

Keep open: the PR fixes the common configured-default Discord startup order and includes strong live proof, but the delay-slot list still diverges from the gateway's runtime-enabled account set, so duplicate-token setups can leave a skipped default account consuming the zero-delay slot.

## What This Changes

The branch reorders Discord multi-account startup delays so `channels.discord.defaultAccount` is first and adds a focused channel test for the default account ahead of secondary accounts.

## Best Possible Solution

Promote the configured Discord default only after building the same runtime-enabled account set the gateway will start, then preserve the existing stagger for the remaining enabled accounts.

## Reproduction Assessment

Yes, source-reproducible. Current main builds Discord startup delay slots from sorted configured accounts, while the gateway starts only accounts that pass `plugin.config.isEnabled`, so the default-account delay bug and the duplicate-token edge are both traceable without adding a new product mode.

## Solution Assessment

No, not yet. Promoting `defaultAccount` is the right direction, but the implementation should build its slot list from the same runtime-enabled accounts the gateway starts so skipped duplicate-token accounts cannot affect delays.
