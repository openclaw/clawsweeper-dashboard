## Summary

Needs changes before merge: the default-account promotion is the right direction, but the slot list still diverges from the gateway runtime-enabled account set, leaving a duplicate-token availability edge that can delay the account that actually starts.

## What This Changes

The PR promotes the configured Discord default account to the first startup slot and adds a regression test for default-before-secondary startup ordering.

## Best Possible Solution

Build Discord startup delay slots from the same runtime-enabled and configured account set the gateway starts, then promote the configured default account within that set and preserve the existing stagger for the rest.

## Reproduction Assessment

Yes. Source inspection shows current main derives Discord startup delay from account-list order while the gateway starts only plugin.config.isEnabled accounts, and the linked issue plus PR proof show the defaultAccount delay in multi-account Discord configs.

## Solution Assessment

No, not yet. Promoting the configured default account is the right fix shape, but the slot list must first match Discord's runtime-enabled predicate so skipped duplicate-token accounts cannot affect delays.
