## Summary

Keep open: current main still delays a configured Discord default account when it sorts after secondary accounts, and this PR is now the canonical proof-backed fix. I found no blocking patch defect, but merge still needs maintainer acceptance of the startup-order compatibility and availability risk.

## What This Changes

The PR changes Discord startup delay calculation to promote an eligible configured `defaultAccount` ahead of secondary accounts and adds regressions for default-first, duplicate-token, and unconfigured-account cases.

## Best Possible Solution

Land the Discord-local fix after maintainer acceptance that an eligible configured `channels.discord.defaultAccount` should own the zero-delay startup slot while secondary accounts keep the existing stagger.

## Reproduction Assessment

Yes. Source inspection of current main shows `resolveDiscordStartupDelayMs` uses sorted/list account order, so a configured default account such as `main` can be delayed behind accounts like `billy` or `farber`.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found: it stays in the Discord plugin, reuses the existing runtime-enabled and configured predicates, and adds regressions for the prior predicate gaps.
