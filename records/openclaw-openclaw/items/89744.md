## Summary

Keep open: the PR fixes the main Discord default-account startup ordering case, but it is not merge-ready because the new startup slot list can still include Discord accounts that gateway startup skips through the runtime-enabled duplicate-token contract.

## What This Changes

The branch promotes `channels.discord.defaultAccount` to the first Discord startup-delay slot and adds a focused channel test for enabled secondary accounts.

## Best Possible Solution

Keep the defaultAccount-first behavior, but compute stagger slots from the same runtime-enabled Discord account set that gateway startup uses, with duplicate-token regression coverage.

## Reproduction Assessment

Yes from source inspection: the PR helper builds delay slots without the Discord runtime-enabled duplicate-token filter, while gateway startup skips accounts through `plugin.config.isEnabled`. A duplicate-token `defaultAccount` can therefore take slot 0 even though it will not start.

## Solution Assessment

No as-is. The owner boundary and defaultAccount-first direction are right, but the best fix must reuse Discord's runtime-enabled account contract so skipped duplicate-token accounts cannot consume stagger slots.
