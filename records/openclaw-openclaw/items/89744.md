## Summary

Keep open: this is a useful Discord startup-order fix with strong live proof for the normal enabled-default path, but it is not merge-ready because the new delay ordering still diverges from the gateway runtime-enabled account predicate. Duplicate-token configs can still make the real token owner wait behind an account that gateway startup skips.

## What This Changes

The PR changes Discord gateway startup delay ordering so the configured Discord default account is promoted ahead of secondary accounts, with a focused channel test.

## Best Possible Solution

Use one startup-candidate predicate aligned with Discord gateway enablement, then promote the configured default only if that account will actually start, with regression coverage for duplicate-token upgrade configs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: gateway startup skips duplicate-token accounts through plugin.config.isEnabled, but the PR's delay helper still includes them, so the real token owner can receive a nonzero stagger index; I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is a plausible happy-path fix, but the best fix is to reuse Discord's runtime-enabled account predicate so delay ordering and gateway startup cannot drift.
