## Summary

Keep open: the PR addresses the Discord default-account startup bug, but the proposed helper can still assign the immediate startup slot to a duplicate-token account that gateway startup will skip, delaying the actual Discord monitor.

## What This Changes

The PR promotes the configured Discord default account to the first startup delay slot and adds a regression test for default-account-first staggering.

## Best Possible Solution

Keep the default-account-first behavior, but build the stagger order from the same runtime-enabled Discord account set that gateway startup uses before assigning delay slots.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: configure two enabled Discord accounts with the same token, set `channels.discord.defaultAccount` to the later duplicate-token account, and the PR helper can promote that skipped account into slot zero while gateway startup skips it.

## Solution Assessment

No. The PR is close, but the best fix is to reuse Discord's runtime-enabled account predicate before default promotion so startup delay order matches the accounts that the gateway will actually start.
