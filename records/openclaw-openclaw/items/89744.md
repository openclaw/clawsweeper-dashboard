## Summary

Keep open: the PR fixes the normal default-account ordering case, but it is not merge-ready because the new delay ordering still counts Discord accounts that gateway startup will skip for duplicate-token suppression, so the actual runtime-enabled bot can remain delayed during restart windows.

## What This Changes

The branch promotes the configured Discord default account to the first multi-account startup slot and adds a focused channel test for default-first staggering.

## Best Possible Solution

Build the Discord startup-delay account list from the same runtime-enabled account set used by gateway startup, then promote the resolved default within that set and cover the duplicate-token defaultAccount case.

## Reproduction Assessment

Yes from source inspection, though I did not execute the test suite in this read-only review. Gateway startup skips accounts through plugin.config.isEnabled, while the PR's delay list still includes token-configured accounts that Discord marks disabled due duplicate-token ownership.

## Solution Assessment

No. The default-first direction is the right fix shape, but the helper should align its slot list with the runtime-enabled accounts that can actually reach startAccount.
