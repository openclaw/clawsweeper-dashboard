## Summary

Keep this PR open, but it still needs a blocking repair: the normal default-account path is well covered, while the new delay ordering can still let a Discord account that gateway startup skips consume the first startup slot.

## What This Changes

The PR promotes the configured Discord default account to the first provider-startup slot and adds a channel test for secondary account staggering.

## Best Possible Solution

Use one Discord startup account list that applies the runtime-enabled duplicate-token/disabled/configured filtering first, promotes the configured default only if it remains in that list, and covers both normal-default and skipped-duplicate cases in focused tests.

## Reproduction Assessment

Yes, by source inspection: current main sorts Discord startup delay by listDiscordAccountIds, so a configured default account later in that list gets a delayed slot. I did not execute a local reproduction in this read-only review, but the PR body supplies live after-fix proof for the normal path.

## Solution Assessment

No. The direction is right, but the patch should compute the delay order from the same runtime-enabled account set that gateway startup uses, then promote the default account only inside that filtered set.
