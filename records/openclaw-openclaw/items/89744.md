## Summary

Keep this PR open. It fixes the main configured-default Discord startup case and has strong live log proof, but it is not merge-ready because the new startup ordering helper still counts duplicate-token accounts that Discord gateway startup disables, so documented env-fallback/config-token setups can still get a phantom stagger delay.

## What This Changes

The branch updates Discord gateway startup delay ordering to promote the configured default account ahead of secondary accounts and adds a focused channel test.

## Best Possible Solution

Keep the configured-default promotion, but compute stagger slots from the configured, runtime-enabled Discord accounts that will actually start, with duplicate-token regression coverage.

## Reproduction Assessment

Yes, source-reproducible. With DISCORD_BOT_TOKEN plus a configured Discord account sharing the same token, current docs and account tests mark the env fallback account disabled, but the PR helper still includes it when computing delay slots.

## Solution Assessment

No. Promoting the configured default account is the right direction, but the best fix is to derive delay slots from the same runtime-enabled account set as gateway startup before reordering the default account.
