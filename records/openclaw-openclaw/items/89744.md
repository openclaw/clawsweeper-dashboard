## What This Changes

The PR reorders Discord startup-delay candidates so the configured default account starts before staggered secondary accounts, with a regression test for the multi-account case.

## Best Possible Solution

Land this plugin-local startup-order fix after normal maintainer and CI gates, then let it close the linked Discord startup bug at https://github.com/openclaw/openclaw/issues/77429.

## Reproduction Assessment

Yes. Current main sorts Discord account IDs before computing startup delay, so a configured default account such as `main` can receive a nonzero delay when enabled accounts like `billy` or `farber` sort earlier.

## Solution Assessment

Yes. Reordering only Discord startup-delay candidates with the existing default-account resolver is the narrowest maintainable fix; changing shared account-list ordering would affect unrelated callers.
