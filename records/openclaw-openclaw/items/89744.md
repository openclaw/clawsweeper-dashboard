## Summary

Keep open. The common-path fix is useful and the supplied live Discord startup proof is strong, but the new startup-slot helper still diverges from the gateway's runtime-enabled account predicate, so duplicate-token default-account configurations can leave the real running account delayed.

## What This Changes

The PR reorders Discord multi-account startup delay slots so the configured `defaultAccount` starts before secondary accounts and adds a focused channel test.

## Best Possible Solution

Derive Discord startup slots from the same runtime-enabled account set used by gateway startup, preserving the 10s stagger only for accounts that will actually start.

## Reproduction Assessment

Yes. Current main computes Discord startup delay from configured account order, and the linked issue plus source path show how a configured default account can be assigned a delayed slot; this read-only review did not run the test suite.

## Solution Assessment

No. Promoting `defaultAccount` is the right direction, but the best fix should reuse the Discord runtime-enabled account predicate so skipped duplicate-token accounts cannot consume startup slots.
