## Summary

Keep open: the report is recent, detailed, and not fully covered by the closed related issue, which fixed section-scoped configure flows rather than the full `openclaw configure` path. Current main still wraps non-skipped CLI commands in a respawn layer before command dispatch, and `configure` is not classified as an interactive TTY command there, which is a plausible source for the reported stuck parent/child process pair.

## What This Changes

The issue reports that full `openclaw configure` hangs after the initial Gateway mode selection on Debian with v2026.5.12 and leaves high-CPU OpenClaw child processes behind.

## Best Possible Solution

Treat full `openclaw configure` as an interactive command for startup respawn purposes, or otherwise prevent the warning-suppression respawn wrapper from owning its PTY, then prove SIGINT/timeout cleanup with focused tests and a Linux PTY smoke.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection gives a clear current-main path for `configure` to be wrapped in a respawn child, matching the reported process tree, but I did not run a live Debian/PTTY reproduction of the full high-CPU hang.

## Solution Assessment

Is this the best way to solve the issue? Unclear until live PTY proof confirms the root cause; the narrowest likely fix is to mark `configure` as interactive in the respawn policy while preserving the existing Gateway prompt and section-scoped configure fixes.
