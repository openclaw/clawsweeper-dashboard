## Summary

Keep open. The approval-id dedupe direction is useful, but the PR can return an in-flight `status: "accepted"` cache entry to duplicate followup callers; `expectFinal` clients ignore accepted frames, so a duplicate async exec delivery can hang until timeout and then take the direct fallback path, reintroducing a duplicate Telegram message. The PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds approval-id-based gateway dedupe keys for exec approval followup agent requests and a regression test for two nonce idempotency keys sharing one approval id.

## Best Possible Solution

Keep the approval-id dedupe idea, but make duplicate followup deliveries resolve without spawning another run and without triggering the direct fallback, then prove the behavior in a redacted real Telegram or equivalent gateway-runtime setup.

## Reproduction Assessment

Yes, at source level for the PR regression: the new stable key can return cached `status: "accepted"`, while GatewayClient `expectFinal` ignores accepted frames and exec followups currently call with `expectFinal: true`. I did not run the live Telegram reproduction.

## Solution Assessment

No. Approval-id dedupe at the gateway boundary is the right direction, but returning cached accepted/in-flight payloads to duplicate followup calls is incomplete; it needs a terminal-sharing, non-accepted in-flight, or equivalent response strategy that does not cause fallback delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid hanging duplicate followup callers on accepted cache entries:** `src/gateway/server-methods/agent.ts:699-702`
  - body: The new approval-id key can hit the first request's in-flight `status: "accepted"` entry. Exec approval followups call Gateway `agent` with `{ expectFinal: true }`, and the TS client deliberately ignores accepted frames while waiting for a final response, so a duplicate delivery will wait until its 60s timeout and then take the direct fallback path, causing another Telegram message instead of only suppressing the extra continuation. The dedupe path needs to return a non-accepted in-flight result, share a final waiter, or otherwise avoid an accepted-only cached response for duplicate followup deliveries.
  - confidence: 0.9
