## Summary

Keep open: the patch targets a valid P1 duplicate-followup bug, but the alias cache can return an in-flight `accepted` response to `expectFinal` callers and leave them pending; the PR also lacks real behavior proof.

## What This Changes

This PR adds approval-id-based gateway dedupe for exec approval follow-up agent requests while preserving nonce handoff keys, plus a regression test for duplicate elevated followups.

## Best Possible Solution

Collapse duplicate exec followup requests by approval id without leaving `expectFinal` callers pending, then prove the Telegram DM async exec flow emits one completion reply.

## Reproduction Assessment

No high-confidence live reproduction from this review; the linked report gives detailed Telegram steps and source inspection shows the nonce followup path, but I did not run the Telegram DM flow.

## Solution Assessment

No. Approval-id dedupe is the right boundary, but cached in-flight aliases must either wait for and return the original terminal result or return a non-accepted deduped status so `expectFinal` callers do not hang.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Return a final response for cached followups:** `src/gateway/server-methods/agent.ts:699-702`
  - body: When the second nonce followup hits the new approval-id alias while the first run is still in flight, `readGatewayDedupeEntry` returns the first run's cached `{status: "accepted"}` and the handler returns immediately. Backend `sendExecApprovalFollowup` calls gateway `agent` with `expectFinal: true`, and the TS gateway client ignores `accepted` frames until a final frame arrives; this duplicate request never gets one, so it can wait until the 60s timeout and then take the direct fallback path instead of cleanly deduping. Please make cached in-flight followups resolve with the original terminal result or a non-accepted deduped response.
  - confidence: 0.88
