## What This Changes

The PR adds reply-phase and context-assembly telemetry to diagnostics-otel, extends the QA OTel smoke report with latency buckets, and updates related docs and Plugin SDK baseline metadata.

## Best Possible Solution

Fix the context duration boundary so the metric measures the assembly work it names, then merge the additive reply-phase telemetry only after maintainer acceptance of the public diagnostic SDK contract.

## Reproduction Assessment

Yes, source inspection reproduces the missing behavior: current main exports no reply phase OTel signal and context.assembled has no duration field. I did not run the real OTel smoke in this read-only checkout.

## Solution Assessment

No, not yet: the reply-phase implementation is a good fit, but the context assembly duration currently starts too late to include context-engine assembly. The safer fix is to move the timing boundary to cover the actual assembly work or rename the metric/docs to match what it measures.
