## Summary

Keep open. Current main still sends the raw `memory_search` query to QMD `lex`, `vec`, and `hyde` sub-searches, and any QMD search exception flips the manager to the builtin fallback; the latest tagged QMD release still has the hyphen validator that rejects `vec`/`hyde` queries, so this is actionable rather than cleanup-closeable.

## What This Changes

Prevent hyphenated `memory_search` queries from causing QMD query validation failure to drop the whole QMD backend to the builtin index.

## Best Possible Solution

Keep lex queries raw, prevent known QMD semantic-query validation failures from disabling the whole QMD backend, and add a focused regression test for hyphenated `qmd.query` searches.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: QMD v2.1.0 rejects `/-\w/` in `vec`/`hyde`, OpenClaw current main passes the raw hyphenated query to those sub-searches, and the fallback wrapper treats that QMD error as a whole-backend failure.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable path is in `qmd-manager`: preserve raw lex behavior while either normalizing or retrying only semantic subqueries for known QMD validation failures, with tests around the mcporter `qmd.query` payload and fallback behavior.
