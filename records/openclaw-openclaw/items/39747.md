## Summary

Keep this PR open, but do not merge it as-is. The branch targets a real heartbeat/plugin-channel load-order gap that still exists on current main, but the proposed fix can still validate against a stale pinned channel registry and leave plugin-only heartbeat targets unresolved.

## What This Changes

The PR changes outbound channel normalization to optionally bootstrap configured plugin channels and passes config when resolving explicit heartbeat targets.

## Best Possible Solution

Fix the outbound bootstrap seam so explicit heartbeat targets can resolve configured plugin-only channel ids before deliverable-channel recognition, including pinned-registry cases, then cover that behavior with focused channel-resolution and heartbeat-target tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure heartbeat.target to a plugin-only channel that is not already in the registered deliverable-channel set; current main normalizes it to undefined before any cfg-backed bootstrap can run.

## Solution Assessment

No, the PR is not the best final fix as written. Bootstrapping before deliverable recognition is the right direction, but the implementation should validate against the refreshed or returned plugin registry surface instead of re-reading a potentially stale pinned channel registry.
