## Summary

Close as implemented on current main: the observable heartbeat plugin-target problem is now handled by startup-time plugin loading and channel-registry pinning before heartbeat services start, so this branch’s per-normalization bootstrap approach is obsolete. The branch diff is still not the best patch if reviewed alone because it can validate through a stale channel registry after bootstrap.

## What This Changes

The branch changes outbound channel normalization so explicit heartbeat targets can pass config into plugin-channel bootstrap before falling back to no target.

## Best Possible Solution

Keep the current main startup-load and pinned-channel-registry implementation, and close this stale branch rather than repairing the older lazy-bootstrap approach.

## Reproduction Assessment

No. On current main, the Gateway loads and pins plugin channel registry state before activating the heartbeat runner, so I do not have a current-main failing path for `heartbeat.target: "telegram"`.

## Solution Assessment

Yes for current main; no for this branch. The startup registry solution is narrower and more maintainable than per-call lazy bootstrapping, while the PR diff can still revalidate through a stale channel surface.
