## Summary

Current main already handles the heartbeat plugin-target problem through startup-time plugin loading and pinned channel-registry state before heartbeat services activate, so this stale PR is obsolete.

## What This Changes

The branch passes config into outbound channel normalization so heartbeat target resolution can try bootstrapping plugin channel names before falling back to no external delivery.

## Best Possible Solution

Keep the current startup plugin loading and pinned channel-registry implementation, and close this stale lazy-bootstrap PR rather than repairing it.

## Reproduction Assessment

No. I do not have a current-main failing path because source inspection shows plugin channel registries are loaded and pinned before the heartbeat runner can resolve `heartbeat.target: "telegram"`.

## Solution Assessment

Yes for current main. Startup-level registry pinning is the more maintainable fix because it makes plugin channel ids available to all channel-resolution consumers instead of adding heartbeat-only lazy bootstrap behavior.
