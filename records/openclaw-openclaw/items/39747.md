## Summary

Keep this PR open. Current main still has the heartbeat/plugin-channel load-order gap, and the PR is aimed at the right area, but the proposed normalization retry is not robust against the pinned channel-registry path used by current gateway startup.

## What This Changes

The PR adds cfg-aware plugin bootstrap to outbound channel normalization and passes config when resolving explicit heartbeat targets.

## Best Possible Solution

Fix the load-order gap in the outbound bootstrap seam so explicit heartbeat targets can resolve configured plugin-only channel ids before deliverable-channel recognition, including pinned-registry cases, then add focused tests for the empty-registry and pinned-registry paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with a plugin-only heartbeat target that is not already in the registered deliverable-channel set, current main leaves the heartbeat delivery target as none before any cfg-backed bootstrap can run.

## Solution Assessment

No, not as written. Bootstrapping before the deliverable check is the right direction, but the PR re-checks deliverability through a registry helper that can still read a pinned registry and miss the newly loaded plugin channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Validate against the refreshed channel registry:** `src/infra/outbound/channel-resolution.ts:32-33`
  - body: After bootstrapping, this re-check still calls `isDeliverableMessageChannel(normalized)`, which reads the pinned channel registry when one exists. If gateway startup has a non-empty pinned registry that does not include the newly configured plugin channel, the bootstrap can load the plugin into the active registry while this check still returns false, leaving `heartbeat.target` as `none`. Have the bootstrap path refresh or return the channel surface, or validate the loaded registry directly before returning the target.
  - confidence: 0.78
