## What This Changes

The PR changes shared identity-name mention regex derivation so Han names use explicit lookaround boundaries, and adds regression tests for single- and two-character CJK names.

## Best Possible Solution

Land after normal checks if maintainers accept shared identity-derived Han names as group mention triggers; otherwise request live transport proof or a narrower boundary policy before merge.

## Reproduction Assessment

## Solution Assessment

Yes. Fixing `deriveMentionPatterns` is the best owner boundary because `buildMentionRegexes` is shared through core/channel callers and the plugin SDK; a DingTalk-only parser fix would leave sibling consumers with the same boundary bug.
