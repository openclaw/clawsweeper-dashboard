## Summary

Keep open: this PR is the current candidate fix for https://github.com/openclaw/openclaw/issues/87303, and current main plus the latest release still use ASCII word boundaries for identity-derived mention names. The patch looks correct, but it intentionally broadens shared mention routing, so maintainer acceptance remains the right next gate.

## What This Changes

This PR changes the shared auto-reply mention helper so Han-containing identity names use Unicode-aware lookaround boundaries, with regression tests for single- and two-character Chinese names.

## Best Possible Solution

Land this PR after normal checks if maintainers accept shared Han identity names as fallback group mentions; otherwise request live transport proof or a narrower boundary policy before merge.

## Reproduction Assessment

Yes. Current main derives `\b@?包\b` for a one-character Han identity name, and a focused Node regex check shows that regex does not match standalone `@包 你好`; the PR body also supplies terminal output from production helpers after the patch.

## Solution Assessment

Yes. Fixing deriveMentionPatterns is the best owner boundary because buildMentionRegexes is shared through the runtime and plugin SDK; a channel-specific parser fix would leave sibling shared-helper consumers with the same boundary bug.
