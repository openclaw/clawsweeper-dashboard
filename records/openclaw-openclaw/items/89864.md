## Summary

Keep open: this PR is still the focused candidate fix for the open single-character Han mention bug, and current main plus v2026.6.10 still derive identity-name mention regexes with ASCII word boundaries that miss `@包`. The patch has no blocking correctness finding, but it intentionally changes shared mention routing, so maintainers should accept or waive the cross-channel delivery risk before merge.

## What This Changes

The PR changes shared identity-name mention regex derivation to use Han-aware lookaround boundaries for CJK-containing names and adds regression tests for single- and two-character Han names.

## Best Possible Solution

Land the shared-helper fix after normal checks if maintainers accept standalone Han identity names as fallback group mentions; otherwise request live transport proof or a narrower mention policy before merge.

## Reproduction Assessment

Yes. A focused current-main regex check shows `\b@?包\b` does not match `@包 你好` or `包 你好`, while the PR-style boundary matches both and rejects `面包好吃`.

## Solution Assessment

Yes. Fixing `deriveMentionPatterns` is the best owner boundary because `buildMentionRegexes` is exported through the channel SDK and consumed by multiple channel paths; a DingTalk-only parser fix would leave sibling consumers with the same boundary bug.
