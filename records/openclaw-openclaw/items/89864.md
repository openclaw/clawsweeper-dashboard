## Summary

The PR is still necessary and remains a focused candidate for maintainer review: current main still derives identity mention patterns with JavaScript word boundaries, which do not match the reported single-character Chinese name case, while the branch fixes the shared helper used by multiple channel plugins.

## What This Changes

This PR changes the shared derived mention regex builder to use Unicode-aware boundaries for CJK identity names and adds regression tests for single- and two-character Chinese names.

## Best Possible Solution

Land the shared helper fix after maintainer and CI validation, keeping identity-derived CJK mention matching in core with focused regression coverage rather than duplicating transport-specific parsers.

## Reproduction Assessment

Yes, at source level: current main derives `\b@?包\b`, and a read-only Node probe confirms that pattern fails for `@包 你好` while the PR-style boundary matches and rejects longer CJK words. I did not reproduce a live DingTalk group-chat flow.

## Solution Assessment

Yes: the shared `deriveMentionPatterns` path is the narrow maintainable fix because multiple channel plugins consume `buildMentionRegexes` and `matchesMentionPatterns`. A DingTalk-only parser would leave the same identity-derived CJK text-pattern bug in other shared mention paths.
