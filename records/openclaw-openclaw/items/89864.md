## Summary

Keep open: current main and v2026.6.1 still derive identity mention names with ASCII word boundaries, while this PR is the active narrow fix with sufficient helper-level proof and no blocking review findings.

## What This Changes

The PR changes shared mention pattern derivation to use Unicode-aware boundaries for CJK identity names and adds regression tests for single- and two-character CJK mentions.

## Best Possible Solution

Land the shared helper fix after maintainer and CI validation, keeping CJK mention derivation in core instead of duplicating channel-specific parsers.

## Reproduction Assessment

Yes, at source level: current main derives `\b@?包\b`, and a read-only Node probe shows that pattern fails for `@包 你好` while the PR-style boundary matches and rejects longer CJK words. I did not reproduce the full live DingTalk channel flow.

## Solution Assessment

Yes: the shared `deriveMentionPatterns` owner boundary is the narrow maintainable fix because multiple channel plugins consume `buildMentionRegexes` and `matchesMentionPatterns`. A channel-specific DingTalk parser would leave the same CJK identity bug in other shared mention paths.
