## Summary

Keep open: the fix direction is useful for the linked CJK mention bug and the updated proof now runs the production helper, but the current patch double-escapes the word-character boundary and would create false-positive group mentions for ASCII-adjacent CJK names.

## What This Changes

The PR changes shared auto-reply mention pattern derivation to use Unicode-aware boundaries for CJK identity names and adds CJK mention regression tests.

## Best Possible Solution

Land a corrected shared mention-boundary helper that treats both ASCII word characters and CJK characters as non-boundary neighbors, with focused tests for CJK positives and ASCII/CJK substring false positives.

## Reproduction Assessment

Yes. Current main derives `\b@?包\b`, and a direct Node check shows that pattern does not match `@包 你好` or `包 你好`; the patch defect is also source-reproducible from the proposed raw regex text.

## Solution Assessment

No. The shared helper is the right owner boundary, but this implementation escapes `\w` incorrectly in the CJK lookaround; fixing that and adding ASCII adjacency tests is the narrow maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep ASCII word boundaries around CJK mentions:** `src/auto-reply/reply/mentions.ts:29-30`
  - body: The new CJK branch builds the lookaround with `String.raw` and `[\\w...]`, which RegExp treats as a literal backslash/`w` class instead of the `\w` word-character class. That lets text like `a包`, `1包`, and `_包` pass as mentions, so group messages containing the CJK name inside an ASCII identifier can now wake the agent. Use a single raw backslash or a normal string that produces `\w`, and add adjacency tests for ASCII letters, digits, and underscores.
  - confidence: 0.94
