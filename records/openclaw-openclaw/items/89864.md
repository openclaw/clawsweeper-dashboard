## Summary

## What This Changes

The PR wraps identity-derived Chinese/Han mention regex names with explicit Unicode-aware boundaries and adds regression tests for single- and two-character Chinese agent names.

## Best Possible Solution

Land the shared mention-helper fix after CI and maintainer acceptance of helper-level proof; request live DingTalk or comparable group-chat proof only if transport-level confidence is required.

## Reproduction Assessment

Yes, at source level: current main derives `\b@?包\b`, and a read-only Node probe shows that regex fails on `@包 你好` while the PR-style boundary matches and rejects `面包好吃`. I did not reproduce a live DingTalk group-chat flow.

## Solution Assessment

Yes: the shared `deriveMentionPatterns` path is the narrow maintainable fix because many channel plugins consume `buildMentionRegexes` and `matchesMentionPatterns`. A DingTalk-only parser change would leave the same identity-derived text-pattern bug in other shared mention paths.
