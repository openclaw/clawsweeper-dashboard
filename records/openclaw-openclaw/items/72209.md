## Summary

Keep this PR open. Current main does not already implement passive recovery checkpoints, and the PR still has blocking design/correctness issues: it records recovery state but does not wire saved bundles into recreated-session prompt construction, and it adds broad prose redaction plus synchronous checkpoint persistence on tool paths.

## What This Changes

The PR adds agent session recovery prompt helpers, local Task Ledger/Recovery Bundle persistence, and checkpoint writes from `update_plan`, `sessions_yield`, and `sessions_spawn`, with focused tests.

## Best Possible Solution

Complete the passive recovery path end to end: define how a recreated session finds the prior task bundle, load that bundle through the actual embedded/CLI prompt builders as informational-only context, preserve the no-approval-inheritance rule, make persistence non-blocking or queued, tighten redaction to high-confidence secret shapes, and cover the live prompt-composition plus tool checkpoint paths together.

## Reproduction Assessment

Yes. Static source inspection is enough for the review findings: the PR writes bundles, but live prompt callers do not load/pass them, and the redaction and synchronous I/O behavior are directly visible in the added state module.

## Solution Assessment

No. The direction is plausible, but the current patch records recovery data without delivering it to automatically recreated sessions and puts best-effort persistence on hot tool paths; the safer solution is to wire the real recovery prompt path first and queue or async-write checkpoint state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire recovered bundles into live prompts:** `src/agents/system-prompt.ts:502`
  - body: This adds `sessionRecovery` only to the leaf prompt builder, but the embedded/CLI prompt wrappers are not changed and no runtime path loads a saved Recovery Bundle for a recreated session. As written, checkpoints are persisted but never appear in live prompts, so the advertised recovery context is unavailable.
  - confidence: 0.93
- **[P2] Avoid redacting normal recovery prose:** `src/agents/session-recovery-state.ts:115`
  - body: The bare `token|secret|authorization|password` pattern treats ordinary summaries like `authorization: waiting-for-review` as secret material and stores them redacted. That silently damages the recovered context this feature is meant to preserve; keep high-confidence secret patterns or require stronger value shapes.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: Every enabled `update_plan`, `sessions_yield`, and `sessions_spawn` checkpoint calls synchronous filesystem writes before returning the tool result. Since checkpoints are best-effort, slow local or network-backed state directories can block unrelated agent progress; queue the write or use async persistence.
  - confidence: 0.88
