## Summary

Keep open. The PR is useful directionally, but it is not merge-ready because it records recovery checkpoints without wiring recovered bundles into live recreated-session prompts, and it adds broad prose redaction plus synchronous checkpoint I/O on tool execution paths.

## What This Changes

The PR adds session recovery prompt guidance, local Task Ledger and Recovery Bundle persistence, and checkpoint writes from `update_plan`, `sessions_yield`, and `sessions_spawn` with focused tests.

## Best Possible Solution

Complete the passive recovery path end to end: load scoped recovery bundles for recreated sessions, inject them as informational-only prompt context, keep approvals expired, make checkpoint persistence non-blocking, tighten redaction, add tests, and document the user-facing behavior in the changelog.

## Reproduction Assessment

Yes. Static inspection is a high-confidence reproduction path: the PR persists checkpoint state, but the live embedded/CLI prompt builders have no recovered-bundle load or pass-through path, and the redaction and synchronous I/O behavior are directly visible in the added module.

## Solution Assessment

No. The direction is plausible, but this branch records recovery data before it can reach real recreated-session prompts and puts best-effort persistence on synchronous tool paths; the safer solution is to wire live recovery first with non-blocking storage and narrower redaction.
