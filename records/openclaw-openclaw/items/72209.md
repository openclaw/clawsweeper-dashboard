## Summary

Keep open: the PR is a plausible session-continuity feature, but the branch still lacks end-to-end recovery-bundle loading, real behavior proof, and resolution of concrete security and availability risks.

## What This Changes

The PR adds passive session recovery prompt guidance, local task-ledger/recovery-bundle persistence, and checkpoint writes from update_plan, sessions_yield, and sessions_spawn.

## Best Possible Solution

Define the passive recovery contract end to end: load scoped bundles into recreated prompt builders, reuse shared secret redaction before persistence, make checkpoint writes asynchronous best-effort, align with https://github.com/openclaw/openclaw/issues/17211, and require real behavior proof before merge.

## Reproduction Assessment

Yes for the PR review findings by source inspection: PR head writes recovery bundles, but CLI and embedded prompt builders do not load or pass sessionRecovery, and the added state module shows local redaction plus synchronous writes. No end-to-end recreated-session behavior proof is present.

## Solution Assessment

No. The maintainable solution needs complete recovery loading, shared redaction, non-blocking persistence, maintainer agreement on session-state semantics, and real behavior proof before merge.

- **[P1] Wire recovery bundles into prompt callers:** `src/agents/system-prompt.ts:502`
  - body: The PR writes recovery bundles and only renders them when buildAgentSystemPrompt receives sessionRecovery, but the CLI and embedded prompt builders never load a bundle or pass that value. Recreated sessions therefore cannot consume the persisted recovery context this PR creates.

- **[P2] Use shared redaction before persisting recovery text:** `src/agents/session-recovery-state.ts:115`
  - body: Recovery summaries and bundles are persisted through a local regex list that can miss structured secret forms already covered by the shared redaction helpers. Persisted recovery text should reuse the shared redaction path or equivalent key-aware coverage.
  - confidence: 0.86

  - body: appendTaskLedgerEvent performs mkdirSync and appendFileSync before update_plan, sessions_yield, or sessions_spawn returns. Since these checkpoints are best-effort, slow state storage should not block normal tool execution.
  - confidence: 0.88
