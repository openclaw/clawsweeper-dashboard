## Summary

Keep open. The PR is useful source work, but the latest head still records recovery data without wiring recovered bundles into the production recreated-session prompt path, and it keeps two concrete persistence-quality issues in the new state module.

## What This Changes

The PR adds session recovery prompt discipline, a local Task Ledger and Recovery Bundle store, and checkpoint writes from `update_plan`, `sessions_yield`, and accepted `sessions_spawn` calls with focused agent tests.

## Best Possible Solution

Complete the recovery path end to end: write bounded checkpoints off the hot path, load the matching Recovery Bundle only for the recreated/interrupted session context, pass it through embedded and CLI prompt builders as informational-only data, and keep approvals/auth/permissions explicitly expired rather than inherited.

## Reproduction Assessment

Yes. Static inspection of the PR head is enough: the checkpoint writer and prompt section are added, but no production recreated-session caller loads `loadRecoveryBundle(...)` or passes `sessionRecovery` through the embedded prompt wrapper.

## Solution Assessment

No. The overall direction is plausible, but the current patch is incomplete because persisted checkpoints are not delivered to live recreated-session prompts and the best-effort checkpoint writes remain synchronous on tool execution paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire recovered bundles into real prompt creation:** `src/agents/system-prompt.ts:502`
  - body: This adds `sessionRecovery` to the leaf prompt builder, but no production path loads a saved Recovery Bundle or passes it through the embedded prompt wrapper when a session is recreated. As written, checkpoints are persisted but never appear in live prompts, so the advertised recovery context is unavailable.
  - confidence: 0.93
- **[P2] Avoid redacting normal recovery prose:** `src/agents/session-recovery-state.ts:115`
  - body: The bare `token|secret|authorization|password` pattern treats ordinary summaries like `authorization: waiting for review` as secret material and stores them as redacted text. That silently damages the recovered context this feature is meant to preserve; keep high-confidence secret patterns or require stronger value shapes.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: Every enabled `update_plan`, `sessions_yield`, and accepted `sessions_spawn` checkpoint performs synchronous filesystem writes before returning the tool result. Since checkpoints are best-effort, slow local or network-backed state directories can block unrelated agent progress; queue the write or use async persistence.
  - confidence: 0.88
