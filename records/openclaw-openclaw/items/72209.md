## Label Justifications

- `P2`: This is a meaningful session-continuity feature PR, but it is not an urgent shipped regression and needs proof/design work before merge.
- `merge-risk: 🚨 session-state`: The PR persists and reintroduces agent context, so wrong bundle selection/loading or stale semantics could mis-associate recovered session state.
- `merge-risk: 🚨 security-boundary`: The PR persists free-text recovery data using bespoke redaction instead of the shared secret-redaction contract.
- `merge-risk: 🚨 availability`: The PR adds synchronous filesystem writes to update_plan, sessions_yield, and sessions_spawn hot paths.

## Summary

Keep open: this is a useful session-continuity direction, but the PR is not merge-ready because it persists recovery bundles without wiring them into real prompt construction, lacks real behavior proof, carries secret-redaction and hot-path I/O concerns, and is currently conflicting against main.

## What This Changes

This PR adds a session recovery prompt section, local task ledger/recovery-bundle persistence, and passive checkpoint writes from update_plan, sessions_yield, and sessions_spawn.

## Best Possible Solution

Define the passive recovery contract end to end: load scoped bundles into recreated prompt builders, reuse shared secret redaction before persistence, make checkpoint writes asynchronous best-effort, align with https://github.com/openclaw/openclaw/issues/17211, and require real behavior proof before merge.

## Reproduction Assessment

Yes for the PR review findings. PR-head source shows recovery bundles can be written but are not loaded into real CLI or embedded prompt callers, and the bespoke redaction plus synchronous I/O are visible in the added recovery-state module; I did not run tests because this was a read-only review.

## Solution Assessment

No. The patch is a plausible direction, but the maintainable solution needs complete recovery loading, shared redaction, non-blocking persistence, maintainer agreement on the session-state contract, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire recovery bundles into prompt callers:** `src/agents/system-prompt.ts:502`
  - body: The PR writes recovery bundles and only renders them when buildAgentSystemPrompt receives sessionRecovery, but the CLI and embedded prompt builders never load a bundle or pass that value. Recreated sessions therefore cannot consume the persisted recovery context this PR creates.
  - confidence: 0.93
- **[P2] Use shared redaction before persisting recovery text:** `src/agents/session-recovery-state.ts:115`
  - body: Recovery summaries and bundles are persisted through a local regex list that can miss structured secret forms already covered by the shared redaction helpers. Persisted recovery text should reuse the shared redaction path or equivalent key-aware coverage.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: appendTaskLedgerEvent performs mkdirSync and appendFileSync before update_plan, sessions_yield, or sessions_spawn returns. Since these checkpoints are best-effort, slow state storage should not block normal tool execution.
  - confidence: 0.88
