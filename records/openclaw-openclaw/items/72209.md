## Summary

Keep this PR open. The current branch is not merge-ready because the persisted recovery bundle is not wired into the real recreated-session prompt path, and the new checkpoint persistence has concrete data-integrity and hot-path I/O issues.

## What This Changes

The PR adds session recovery prompt discipline, local task-ledger and recovery-bundle storage, and checkpoint writes from `update_plan`, `sessions_yield`, and `sessions_spawn`, with focused agent tests.

## Best Possible Solution

Complete the passive recovery path end to end: load the correct Recovery Bundle when a session is recreated, pass it through the embedded and CLI prompt builders as informational-only context, preserve the no-approval-inheritance invariant, tighten redaction to high-confidence secret shapes, and move best-effort checkpoint persistence off the synchronous tool path.

## Reproduction Assessment

Yes. Static inspection is enough for the review findings: the PR persists checkpoint state, but no live recreated-session caller loads a bundle into prompt creation, and the redaction and synchronous I/O behavior are directly visible in the added state module.

## Solution Assessment

No. The direction is plausible, but this branch records recovery data without delivering it to automatically recreated sessions and puts synchronous best-effort writes on tool execution paths; the safer implementation should wire live prompt recovery first and make persistence non-blocking.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire recovered bundles into real prompt creation:** `src/agents/system-prompt.ts:502`
  - body: This adds `sessionRecovery` to the leaf prompt builder, but no production path loads a saved Recovery Bundle or passes it through the embedded/CLI prompt wrappers when a session is recreated. As written, checkpoints are persisted but never appear in live prompts, so the advertised recovery context is unavailable.
  - confidence: 0.93
- **[P2] Avoid redacting normal recovery prose:** `src/agents/session-recovery-state.ts:115`
  - body: The bare `token|secret|authorization|password` pattern treats ordinary summaries like `authorization: waiting for review` as secret material and stores them as redacted text. That silently damages the recovered context this feature is meant to preserve; keep high-confidence secret patterns or require stronger value shapes.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: Every enabled `update_plan`, `sessions_yield`, and accepted `sessions_spawn` checkpoint calls synchronous filesystem writes before returning the tool result. Since checkpoints are best-effort, slow local or network-backed state directories can block unrelated agent progress; queue the write or use async persistence.
  - confidence: 0.88
