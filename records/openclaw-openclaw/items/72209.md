## Summary

Keep this PR open. Current main does not already provide the requested passive recovery checkpoint surface, and the PR branch is useful directionally but still has blocking review issues: persisted bundles are not wired into real recreated-session prompt construction, free-text redaction can corrupt normal recovery prose, and best-effort checkpoint writes run synchronously on tool paths.

## What This Changes

The PR adds session recovery prompt helpers, local JSONL/JSON checkpoint storage, and checkpoint writes from `update_plan`, `sessions_yield`, and `sessions_spawn` with focused tests.

## Best Possible Solution

Complete the passive recovery path end to end: load the appropriate Recovery Bundle when a session is recreated, pass it through the actual embedded/CLI prompt builders as informational-only context, preserve the no-approval-inheritance invariant, queue or async-write checkpoint persistence, tighten redaction to high-confidence secret shapes, and cover the live prompt-composition plus update_plan/yield/spawn paths together.

## Reproduction Assessment

Yes. Static source inspection is enough for the review findings: the PR writes checkpoint state, but no live recreated-session caller loads a bundle and passes `sessionRecovery` into the production prompt wrappers, and the redaction and synchronous I/O behavior are directly visible in `session-recovery-state.ts`.

## Solution Assessment

No. The product direction is plausible, but the current patch records recovery data without delivering it to automatically recreated sessions and puts synchronous best-effort writes on tool execution paths; a safer solution should first wire the live prompt path and make persistence non-blocking.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire recovered bundles into real prompt creation:** `src/agents/system-prompt.ts:502`
  - body: This adds `sessionRecovery` only to the leaf prompt builder, but the actual embedded/CLI prompt callers are not changed and no runtime path loads a saved Recovery Bundle for a recreated session. As written, checkpoints are persisted but never appear in live prompts, so the advertised recovery context is unavailable.
  - confidence: 0.93
- **[P2] Avoid redacting normal recovery prose:** `src/agents/session-recovery-state.ts:115`
  - body: The bare `token|secret|authorization|password` pattern treats ordinary summaries like `authorization: waiting for review` as secret material and stores them as redacted text. That silently damages the recovered context this feature is meant to preserve; keep high-confidence secret patterns or require stronger value shapes.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: Every enabled `update_plan`, `sessions_yield`, and `sessions_spawn` checkpoint calls synchronous filesystem writes before returning the tool result. Since checkpoints are best-effort, slow local or network-backed state directories can block unrelated agent progress; queue the write or use async persistence.
  - confidence: 0.88
