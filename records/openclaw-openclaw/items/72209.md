## Summary

Keep open. Current main still lacks passive session recovery checkpoints, and this PR has blocking gaps: recovered bundles are not wired into real prompt creation, recovery persistence needs safer redaction, checkpoint writes block tool execution, and the contributor has only provided test/CI proof rather than real behavior proof.

## What This Changes

Adds session recovery prompt discipline, local ledger/bundle persistence, and checkpoint recording for `update_plan`, `sessions_yield`, and `sessions_spawn`.

## Best Possible Solution

Define an end-to-end passive recovery contract, then wire scoped bundle loading into real recreated-session prompts with shared redaction, non-blocking persistence, focused tests, and explicit user confirmation semantics.

## Reproduction Assessment

Yes. Static source inspection is a high-confidence reproduction path: the PR writes checkpoint state but does not load or pass recovered bundles into embedded/CLI prompt builders, and the redaction plus synchronous I/O behaviors are directly visible in the added module.

## Solution Assessment

No. The direction is useful, but the current branch records recovery data before it can reach real prompts and uses bespoke blocking persistence; the safer solution is a maintainer-approved recovery contract with shared redaction and async writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire recovered bundles into live prompt creation:** `src/agents/system-prompt.ts:502`
  - body: This only adds `sessionRecovery` to the leaf prompt builder. The embedded and CLI prompt builders do not accept or load a recovery bundle, so persisted checkpoints cannot reach recreated-session prompts and the advertised recovery context is unavailable.
  - confidence: 0.93
- **[P2] Use shared redaction for persisted recovery text:** `src/agents/session-recovery-state.ts:115`
  - body: The local `SECRET_PATTERNS` branch both redacts ordinary recovery text like `authorization: waiting for review` and misses common secret forms such as provider env assignments. Since these summaries are persisted, reuse the repo's shared redaction or tighten this to high-confidence secret shapes.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: Every enabled plan update, yield, and accepted spawn can reach `appendTaskLedgerEvent`, which performs `mkdirSync` and `appendFileSync` before the tool returns. Best-effort checkpoints should be queued or written asynchronously so slow state storage does not block agent progress.
  - confidence: 0.88
