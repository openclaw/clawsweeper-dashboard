## Summary

Keep open: the branch writes recovery state, but recreated sessions still do not load that state into real prompt builders, and the persistence path needs security and hot-path fixes before merge.

## What This Changes

The PR adds agent session recovery prompt guidance, a task-ledger/recovery-bundle store, and checkpoint writes for `update_plan`, `sessions_yield`, and `sessions_spawn`.

## Best Possible Solution

Define the passive recovery contract end to end: load scoped bundles into recreated prompt builders, use shared redaction before persistence, make checkpoint writes asynchronous best-effort, add focused tests, and require real behavior proof.

## Reproduction Assessment

Yes for the PR review findings from source inspection. PR-head code persists recovery bundles, but the real CLI and embedded prompt builders do not load or pass them, and the redaction plus synchronous I/O issues are directly visible in the added recovery-state module.

## Solution Assessment

No. The current patch is a useful direction, but the maintainable solution needs a complete bundle-loading contract, shared redaction, asynchronous best-effort persistence, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Wire recovery bundles into real prompt builders:** `src/agents/system-prompt.ts:502`
  - body: The PR writes bundles and only renders recovery context when `buildAgentSystemPrompt` receives `sessionRecovery`, but the CLI and embedded prompt builders never load a bundle or pass that value. Recreated sessions therefore cannot consume the persisted recovery context this PR creates.
  - confidence: 0.93
- **[P2] Use shared redaction before persisting recovery text:** `src/agents/session-recovery-state.ts:115`
  - body: Recovery summaries and bundles are persisted through a local regex list that can miss structured secret forms already covered by the shared redaction helpers. Persisted recovery text should reuse the shared redaction path or equivalent key-aware coverage.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: `appendTaskLedgerEvent` performs `mkdirSync` and `appendFileSync` before `update_plan`, `sessions_yield`, or `sessions_spawn` returns. Since these checkpoints are best-effort, slow state storage should not block normal tool execution.
  - confidence: 0.88
