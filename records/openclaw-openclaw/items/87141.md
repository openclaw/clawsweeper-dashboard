## Summary

Keep open: this is useful maintainer-directed hardening, but the patch still lets unreadable schema object fields escape the new shared JSON-compatibility walker, which can break the intended quarantine/diagnostic path for plugin tools and SDK/config schema inputs.

## What This Changes

The PR hardens Codex dynamic-tool registration and related plugin, MCP, provider, config-schema, Responses-payload, and trajectory paths against unsupported or non-JSON-compatible schemas.

## Best Possible Solution

Fix unreadable-field handling in the shared walker, add focused coverage through an uncaught plugin/runtime or SDK caller, then let maintainers decide whether the stricter schema rejection policy is acceptable for plugin compatibility.

## Reproduction Assessment

Yes from source inspection: a schema object with an enumerable getter that throws will escape `describeNonJsonCompatibleValue` at `Object.entries(value)` instead of producing a controlled issue. That path is reachable through plugin tool validation and SDK/config schema callers added by this PR.

## Solution Assessment

No, not yet; the quarantine architecture is the right direction, but the shared walker must handle unreadable fields before the patch fully solves malformed-schema availability hazards. The safer repair is narrow: catch enumeration/property-read failures and report an unreadable path.
