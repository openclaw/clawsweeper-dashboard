- `merge-risk: 🚨 compatibility`: The patch changes how existing Node runtime flags and entry arguments are carried into a respawned gateway process.

## Summary

Keep this PR open: the linked gateway update/respawn bug is real and the PR is a focused repair attempt, but the patch rewrites the first combined spawn argument instead of the actual script entry. That misses non-empty `process.execArgv` launches and still needs real update/respawn proof before merge.

## What This Changes

The PR adds pnpm-versioned entry-path rewriting in `src/infra/process-respawn.ts` and tests the update respawn path in `src/infra/process-respawn.test.ts`.

## Best Possible Solution

The end state should be one respawn-argument resolver that preserves Node runtime flags, rewrites only the OpenClaw entry path across supported separators, and is backed by focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes at source level: current main reaches `respawnGatewayProcessForUpdate()` for `update.run`, and the current helper builds child args directly from `process.execArgv` plus `process.argv.slice(1)`. I did not run a live package self-update in this read-only review.

## Solution Assessment

No as proposed: the implementation rewrites `args[0]` after prepending `process.execArgv`, so the entry path is only corrected when there are no Node runtime flags. The safer fix is to rewrite the `process.argv.slice(1)` entry element before concatenating it with `process.execArgv`, then add non-empty `execArgv` and Windows-path regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rewrite the script argument instead of args[0]:** `src/infra/process-respawn.ts:88`
  - body: Here `args` starts with `process.execArgv`, so with any Node runtime flag (`--enable-source-maps`, `--import`, etc.) `args[0]` is not the entry script. The pnpm versioned entry remains later in the array, leaving the reported self-update failure unfixed, and a versioned loader/import flag could be replaced as if it were the script. Build the script argv slice, rewrite its first element, then concatenate `process.execArgv`.
  - confidence: 0.95
- **[P2] Normalize Windows pnpm entry paths:** `src/infra/process-respawn.ts:44-45`
  - body: The marker only matches `node_modules/.pnpm/`, so Windows pnpm realpaths with backslashes pass through unchanged. `respawnGatewayProcessForUpdate` already supports unmanaged Windows update respawn, so this leaves a supported package-manager path on the stale entrypoint. Normalize separators or parse path parts and add a Windows-style regression test.
  - confidence: 0.86
