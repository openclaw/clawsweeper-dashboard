Full review comments:

- **[P1] Rewrite the actual entrypoint argument:** `src/infra/process-respawn.ts:92`
  - body: `args` starts with `process.execArgv`, so when the gateway was launched with any Node flag the real entrypoint is at `process.execArgv.length`, not `0`. In that case this line rewrites the flag and leaves the pnpm versioned `process.argv[1]` unchanged, so the update child can still spawn the deleted entrypoint.
  - confidence: 0.92
- **[P1] Normalize separators before matching pnpm paths:** `src/infra/process-respawn.ts:44-55`
  - body: The resolver only searches for `node_modules/.pnpm/` and then applies a forward-slash-only regex. `respawnGatewayProcessForUpdate()` explicitly supports unmanaged Windows update respawns, so native backslash pnpm realpaths are returned unchanged and can still point at the removed versioned entrypoint.
  - confidence: 0.89
- **[P2] Remove the unused `_rest` binding:** `src/infra/process-respawn.ts:63`
  - body: The new helper assigns `match[3]` to `_rest` but never reads it. Current CI confirms this fails `tsgo:prod` with TS6133 and `check-lint` with `no-underscore-dangle`, so the branch cannot land until the binding is removed or used.
  - confidence: 0.98
