## Summary

Keep open: the resolver hardening addresses a real current Control UI canvas behavior, but the branch is not safe to merge because its refresh path closes the websocket during active sessions, targets deprecated `canvasHostUrl`, and lacks real after-fix behavior proof.

## What This Changes

The PR hardens Control UI canvas/a2ui iframe URL resolution and adds a timer-driven Gateway websocket reconnect to refresh scoped canvas URLs.

## Best Possible Solution

Keep the fail-closed protected iframe behavior, add positive scoped rewrite coverage for canvas and a2ui, and refresh `pluginSurfaceUrls.canvas` through a non-teardown or explicitly idle-safe maintainer-reviewed path.

## Reproduction Assessment

Yes, by source inspection rather than a live browser run: current main can return raw protected canvas/a2ui paths without a scoped surface URL, and the PR regression follows from its timer close plus current close/reconnect state handling.

## Solution Assessment

No. The resolver hardening is a good direction, but closing the full Control UI websocket and keying refresh off deprecated `hello.canvasHostUrl` is not the narrowest maintainable solution on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid refreshing canvas capability by closing the socket:** `ui/src/ui/gateway.ts:406`
  - body: The new timer closes every scoped Control UI websocket after about nine minutes. Current close handling rejects pending browser requests, and the reconnect `onHello` path clears active chat run/stream state, so long-running chats or RPCs crossing the refresh boundary can be dropped even when no canvas is active. Refresh the plugin surface without transport teardown, or gate it to an idle-safe path.
  - confidence: 0.92

This refresh scheduling path still keys off `hello.canvasHostUrl`, so after rebase it will either fail type checking or miss the canvas surface entirely.
  - confidence: 0.87
