## Summary

Keep this PR open for rework. Current main still has the raw protected canvas URL fallback the PR targets, and the fail-closed resolver direction is useful, but the PR's timer closes the Control UI websocket to refresh capability state; current connection handling rejects pending RPCs and reconnect handling clears active chat run state, so the patch can interrupt long-running UI work.

## What This Changes

The PR makes protected Control UI canvas/a2ui iframe URLs fail closed unless a scoped capability host is present, adds a timer-driven browser websocket reconnect before canvas capability expiry, and updates related UI/gateway tests.

## Best Possible Solution

Keep the fail-closed protected iframe URL behavior, add positive coverage for scoped canvas and a2ui rewrites, and replace the scheduled websocket teardown with an idle-safe or non-teardown Control UI capability refresh path that preserves pending RPCs and active chat run state.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original bug by calling `resolveCanvasIframeUrl('/__openclaw__/canvas/...')` without a scoped host, and the PR regression is traceable from the added timer close through current close/hello handlers that reject pending calls and clear active run state.

## Solution Assessment

No. The resolver hardening is a good part of the fix, but closing the whole Control UI websocket every refresh interval is not the narrowest maintainable solution because capability renewal should not reset unrelated browser session state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid refreshing canvas capability by closing the socket:** `ui/src/ui/gateway.ts:404`

Refresh the canvas capability without tearing down the websocket, or gate refresh to an idle-safe path.
