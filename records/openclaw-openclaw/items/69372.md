## Summary

Keep open: the underlying Control UI canvas capability problem is still real on current main, but this PR should not merge as written because its timed refresh path closes the whole browser websocket and can drop pending RPCs or active chat run state.

## What This Changes

The PR branch makes protected Control UI canvas/a2ui iframe URLs fail closed without a scoped capability host, schedules a browser websocket reconnect before canvas capability expiry, and updates related UI and gateway tests.

## Best Possible Solution

Keep the fail-closed protected iframe URL behavior, add positive coverage for both canvas and a2ui scoped rewrites, and refresh Control UI canvas capabilities through an operator-safe non-teardown path or a clearly idle-gated flow that preserves pending RPCs and active chat run state.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original bug by calling `resolveCanvasIframeUrl('/__openclaw__/canvas/...')` without a scoped host, and the PR regression is traceable from the added timer close through current close and hello handlers that reject pending calls and clear active run state.

## Solution Assessment

No. The resolver hardening is a good part of the fix, but closing the entire Control UI websocket on a timer is not the narrowest maintainable solution for refreshing an iframe capability token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid refreshing canvas capability by closing the socket:** `ui/src/ui/gateway.ts:404`
  - body: The new timer closes every scoped Control UI websocket after about 9 minutes. Current close handling rejects pending browser requests, and the next hello clears active chat run and stream state, so long-running chats or RPCs crossing the refresh boundary can be dropped. Refresh the canvas capability without tearing down the websocket, or gate refresh to an idle-safe path.
  - confidence: 0.92
