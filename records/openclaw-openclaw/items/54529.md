## Label Justifications

- `P2`: The report describes a real Telegram reliability bug on a supported but specialized local Bot API configuration, with limited blast radius and no confirmed current-main live repro.
- `impact:message-loss`: Inbound Telegram updates can be received by polling but skipped by the persisted offset path before reaching agent sessions.
- `impact:crash-loop`: The reported `deleteWebhook` `400: Logged out` path remains able to abort the Telegram polling startup cycle.

## Summary

Keep open: current main and the latest release still store Telegram polling watermarks by account/token rather than by Bot API endpoint, while startup still treats non-network `deleteWebhook` failures as fatal. That leaves the reported local `apiRoot` message-loss and restart-loop path unresolved pending maintainer policy and live local Bot API proof.

## What This Changes

The issue asks OpenClaw to dispatch Telegram updates through a configured local Bot API server, reset or scope update offsets when `apiRoot` changes, and handle `deleteWebhook` `400: Logged out` during migration without a crash loop.

## Best Possible Solution

Keep this open until maintainers choose endpoint-scoped offset migration semantics and `400: Logged out` startup behavior, then fix the Telegram plugin with live local Bot API proof.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection gives a clear source-reproducible path for the offset mismatch: offsets are keyed by account/token while `apiRoot` is applied separately to polling.

## Solution Assessment

Unclear pending maintainer choice. Endpoint-scoped offsets plus a doctor/migration path look like the maintainable fix boundary, but `400: Logged out` handling needs explicit startup compatibility semantics.
