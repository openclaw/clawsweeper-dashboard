## Summary

Keep open. Current main and `v2026.6.10` still keep Telegram update offsets scoped to account/token rather than API root, pass that watermark into the configured local `getUpdates` endpoint, and only continue past `deleteWebhook` failures when they look like recoverable network errors; the remaining fix needs maintainer policy and live local Bot API proof.

## What This Changes

The issue reports local Telegram Bot API `apiRoot` polling updates being received but not dispatched, offset reuse across endpoints, and fatal `deleteWebhook` behavior during migration.

## Best Possible Solution

Keep this as the canonical local Bot API migration tracker; decide endpoint-scoped offset/reset policy and narrow `Logged out` cleanup tolerance, then prove the fix with a live local Bot API migration run.
