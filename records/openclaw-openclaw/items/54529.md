## Summary

Keep open. Current main has the basic Telegram `apiRoot` plumbing and some related startup/network fixes, but the persisted update offset is still keyed by account/token rather than API endpoint, and `400: Logged out` is still outside the recoverable `deleteWebhook` path before polling starts.

## What This Changes

The issue reports that Telegram polling through a self-hosted Bot API `apiRoot` receives raw updates but does not dispatch them, reuses incompatible update offsets, and crash-loops on `deleteWebhook` during migration.

## Best Possible Solution

Keep this issue as the canonical tracker for endpoint-switch polling state: decide whether `apiRoot` changes should reset/quarantine offsets or be repaired by doctor, define the narrow `Logged out` startup policy, then prove it with a real local Bot API migration run.

No. Source inspection gives a plausible current-main reproduction path for stale offsets when the same token moves between API roots, but this read-only pass did not run a local Bot API server or reproduce the full raw-update drop.

## Solution Assessment

Unclear. The fix likely belongs in Telegram plugin polling state and startup error handling, but endpoint-scoped offsets versus doctor/reset behavior and `400: Logged out` tolerance need maintainer direction first.
