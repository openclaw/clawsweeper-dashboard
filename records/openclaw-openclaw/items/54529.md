## Summary

Keep open. Current main and the latest release still persist Telegram polling offsets by account/token rather than API endpoint, and `deleteWebhook` startup only treats network-shaped failures as nonfatal, so the reported local Bot API endpoint-switch message-loss and restart-loop path is not fully resolved.

## What This Changes

The issue asks OpenClaw to dispatch Telegram updates from a self-hosted Bot API server, reset or scope polling offsets when `apiRoot` changes, and avoid a `deleteWebhook` crash loop during Telegram `logOut` cooldown.

## Best Possible Solution

Keep this as the canonical Telegram local Bot API migration bug and resolve it in the Telegram plugin with endpoint-aware offset identity or migration plus an explicit, tested `deleteWebhook` cooldown policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live yet: current source shows the stale-offset and non-network `deleteWebhook` failure paths, but a full self-hosted Telegram Bot API server repro is still needed for the raw-update/no-dispatch behavior on current main.

## Solution Assessment

Is this the best way to solve the issue? The requested direction is broadly right, but the best fix should avoid adding another config knob and instead make offset identity/migration plus `Logged out` cooldown handling explicit in the Telegram polling owner code.
