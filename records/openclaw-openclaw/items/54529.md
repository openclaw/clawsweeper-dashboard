## Label Justifications

- `P2`: This is a bounded Telegram channel reliability bug affecting local Bot API users, with real inbound delivery and startup impact but no evidence of a core-wide outage.
- `impact:message-loss`: The reported failure silently receives or consumes Telegram updates without dispatching them to agent sessions.
- `impact:crash-loop`: The report includes repeated Telegram startup failure when `deleteWebhook` returns `400: Logged out` during local Bot API migration.

## Summary

Keep open: current main and the latest release still expose Telegram `apiRoot`, but update offsets are scoped by account/token identity rather than Bot API endpoint, and the reported `deleteWebhook` `400: Logged out` startup behavior still needs a maintainer decision plus live local Bot API proof. Related reports cover adjacent generic polling or session-key failures, not this endpoint-switch offset/cooldown case.

## What This Changes

The issue reports that switching Telegram to a self-hosted `channels.telegram.apiRoot` can receive or consume updates without agent dispatch, reuse an incompatible update offset, and crash-loop on `deleteWebhook` during local Bot API migration.

## Best Possible Solution

Repair the bundled Telegram plugin so endpoint changes reset or scope polling offsets, define tested local Bot API migration behavior for `400: Logged out`, and prove inbound dispatch plus reply delivery against a real local Bot API server.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for the endpoint-offset failure because current main and `v2026.6.8` key offsets by account/token but not `apiRoot`. A full live reproduction is still missing for the raw-update dispatch drop and `deleteWebhook` cooldown path.

## Solution Assessment

No final patch is proposed here; closing as implemented would be premature because the endpoint-aware offset and `400: Logged out` startup semantics are not present. The best fix likely belongs in the Telegram plugin's offset state/migration and polling startup policy, followed by live local Bot API proof.
