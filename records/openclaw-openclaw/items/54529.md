## Summary

Keep open: current main still preserves Telegram polling offsets by account/token without including the configured Bot API endpoint, so switching the same bot token from the official API to a local `apiRoot` can keep polling with an incompatible watermark. The configured `apiRoot` is now wired into polling, so the remaining work is endpoint-offset migration/scoping plus live confirmation of the local Bot API dispatch and `deleteWebhook` cooldown behavior.

## What This Changes

The issue reports that Telegram local Bot API `apiRoot` polling receives raw updates but fails to dispatch them, with endpoint-switch offset and `deleteWebhook` cooldown failure modes.

## Best Possible Solution

Keep this open and repair the bundled Telegram plugin so polling offsets reset or scope by Bot API endpoint changes, startup webhook cleanup handles the migration cooldown deliberately, and a live local Bot API smoke proves inbound dispatch plus reply delivery after switching roots.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for the endpoint-offset failure because current main keys offsets by account/token but not `apiRoot`. A full live reproduction is still missing for the local Bot API dispatch drop and `logOut` cooldown behavior.

## Solution Assessment

No final patch is proposed here; the best path is a bounded Telegram-plugin repair plus live local Bot API validation. Closing as implemented would be premature because current main and v2026.6.6 still show the endpoint-offset gap.
