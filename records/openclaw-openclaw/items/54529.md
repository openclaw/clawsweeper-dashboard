## Label Justifications

- `P2`: The bug affects a documented Telegram self-hosted Bot API deployment path with limited blast radius to custom apiRoot users.
- `impact:message-loss`: The stale-offset path can silently skip inbound Telegram updates before they reach agent session dispatch.
- `impact:crash-loop`: The reported deleteWebhook Logged out response can keep the Telegram provider in startup retry or exit behavior during local Bot API migration.

## Summary

Keep open. Current main and v2026.5.20 still scope Telegram polling offsets to account/token rather than apiRoot, and startup webhook cleanup still treats a non-network deleteWebhook Logged out response as fatal during local Bot API migration.

## What This Changes

The issue reports that Telegram custom apiRoot/local Bot API polling receives updates but fails to dispatch them after endpoint migration because persisted offsets and startup webhook cleanup are not endpoint-aware.

## Best Possible Solution

Make Telegram offset state endpoint-aware so apiRoot changes invalidate or separate same-token watermarks, and narrowly treat local Bot API migration Logged out deleteWebhook responses as recoverable without masking real auth failures.

## Reproduction Assessment

Yes, at source level: persist a same-token offset from one Bot API endpoint, switch only apiRoot to a local endpoint with lower update ids, and current main seeds local getUpdates from the stale higher watermark. I did not run the live Docker Bot API migration in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Telegram plugin offset store, monitor threading, isolated polling startup, and cleanup-error classification; core plugin APIs and new user config are not required.
