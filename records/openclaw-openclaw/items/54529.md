## Label Justifications

- `P2`: The report affects a supported Telegram configuration with inbound drops and startup retries, but it is scoped to local Bot API endpoint migration rather than the whole runtime.
- `impact:message-loss`: The reported failure drops or skips inbound Telegram updates before they reach agent sessions.
- `impact:crash-loop`: The report includes repeated Telegram provider startup retries when deleteWebhook fails during local Bot API migration.

## Summary

Keep open: `apiRoot` support and token-rotation offset handling are present on current main, but the remaining local Bot API endpoint-switch problem is not solved because the persisted Telegram update offset is still scoped by account/token rather than Bot API endpoint, and `deleteWebhook` logged-out startup handling still needs maintainer policy plus live local Bot API proof.

## What This Changes

The issue asks OpenClaw to dispatch inbound Telegram updates through a configured local Bot API `apiRoot`, reset or scope stale update offsets across endpoint switches, and avoid a `deleteWebhook` crash loop during Telegram `logOut` migration.

## Best Possible Solution

Define Telegram Bot API endpoint identity in the offset lifecycle, add a tested reset or migration path for endpoint switches, and decide the explicit `deleteWebhook` logged-out startup policy with live local Bot API validation.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The source path is reproducible for the offset mismatch: current polling reuses an account/token-scoped `lastUpdateId` when `apiRoot` changes, then applies it to the local endpoint.

## Solution Assessment

Unclear. The endpoint-offset repair belongs in the Telegram plugin, but the exact reset/migration semantics and whether `400: Logged out` should fail open need maintainer direction before an automated fix is safe.
