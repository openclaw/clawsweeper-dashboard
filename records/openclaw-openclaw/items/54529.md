## Label Justifications

- `P2`: The bug affects a real Telegram local Bot API migration path with silent inbound message loss and startup retry loops, but the blast radius is limited to configured custom `apiRoot` deployments.
- `impact:message-loss`: The reported and source-reproducible failure drops inbound Telegram updates before agent session dispatch.
- `impact:crash-loop`: The same report includes a startup retry loop when `deleteWebhook` hits the migration-time logged-out response.

## Summary

Keep open. Current main still scopes Telegram polling offsets to account and token identity, but not the configured Bot API endpoint, so a same-token switch from the official API to a local `apiRoot` can keep polling from a stale higher watermark; the `deleteWebhook` startup path also still lacks a narrow recoverable path for the migration-time logged-out rejection.

## What This Changes

The issue asks OpenClaw to make Telegram local Bot API `apiRoot` migrations dispatch inbound updates, reset stale offsets on endpoint changes, and avoid fatal `deleteWebhook` cooldown loops.

## Best Possible Solution

Fix Telegram offset persistence to include a normalized Bot API endpoint identity and handle the narrow local Bot API migration `deleteWebhook` logged-out rejection without weakening ordinary token/auth failures.

## Reproduction Assessment

Yes, at source level: write a same-token v3 offset from the official Bot API, switch only `apiRoot` to a local Bot API server whose update IDs are lower, and current main seeds polling from the stale higher offset. I did not run the live Docker Bot API path in this read-only review.

## Solution Assessment

Yes. The maintainable repair belongs in Telegram plugin offset state, monitor persistence threading, isolated/non-isolated polling startup, and specific startup-cleanup error classification; core plugin APIs and user config do not need to change.
