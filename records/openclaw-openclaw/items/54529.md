## Summary

Keep open. Current main implements the basic Telegram `apiRoot` plumbing, but source inspection still shows the persisted polling offset is keyed by account/token rather than API endpoint, and startup only treats network-shaped `deleteWebhook` failures as nonfatal, so the reported endpoint-switch message loss/crash-loop scenario is not fully resolved.

## What This Changes

The issue requests reliable Telegram inbound dispatch when `channels.telegram.apiRoot` points to a self-hosted Bot API server, including endpoint-aware offset handling and nonfatal startup behavior during Bot API migration.

## Best Possible Solution

Keep this as the canonical Telegram local Bot API migration bug and resolve it with endpoint-aware offset handling plus an explicit, tested policy for the `logOut`/`deleteWebhook` cooldown path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live yet: current source shows the stale-offset and non-network `deleteWebhook` failure paths, but a full self-hosted Telegram Bot API repro is still needed for the raw-update/no-dispatch behavior on current main.

## Solution Assessment

Is this the best way to solve the issue? The requested direction is broadly right, but the best fix should avoid a new config knob and instead make offset identity/migration and the `Logged out` cooldown policy explicit in the Telegram polling owner code.
