## Label Justifications

- `P2`: The issue describes a bounded but real Telegram local Bot API delivery/startup failure, not a project-wide outage.
- `impact:message-loss`: Inbound Telegram updates can be polled but then fail to reach agent sessions when endpoint migration state is stale.
- `impact:crash-loop`: The report includes repeated Telegram channel restarts when `deleteWebhook` fails during the local Bot API migration window.

## Summary

Keep open. Current main and the latest release still keep Telegram polling offsets scoped to account/token rather than API root, and startup still treats non-network `deleteWebhook` rejections as fatal before polling can dispatch local Bot API updates.

## What This Changes

The issue asks OpenClaw to make Telegram `apiRoot` local Bot API migrations preserve inbound dispatch by handling endpoint-specific offsets and `deleteWebhook` startup failures.

## Best Possible Solution

Keep this issue as the canonical tracker for local Bot API endpoint migration, decide the endpoint-scoped offset and `Logged out` startup policy, then prove the fix with a real local Bot API migration run.

## Reproduction Assessment

No. The report is detailed and source inspection shows a plausible current-main path for stale offsets and fatal webhook cleanup, but this review did not run a self-hosted Bot API server or reproduce the full raw-update drop.

## Solution Assessment

Unclear. The fix likely belongs in Telegram polling state and startup error handling, but endpoint-scoped offsets versus doctor/reset behavior and `400: Logged out` tolerance need maintainer direction first.
