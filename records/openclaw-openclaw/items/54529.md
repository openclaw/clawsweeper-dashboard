## Label Justifications

- `P2`: This is a normal-priority Telegram channel reliability bug with bounded blast radius but real inbound delivery and startup impact for local Bot API users.
- `impact:message-loss`: The reported behavior silently drops Telegram inbound updates after receipt instead of dispatching them to agent sessions.
- `impact:crash-loop`: The report includes repeated Telegram startup failures during local Bot API migration when `deleteWebhook` sees `400: Logged out`.

## Summary

Keep open: current main and the latest release still scope Telegram polling offsets by account/token, not by the configured Bot API endpoint, so switching the same bot token from the official API to a local `apiRoot` can reuse an incompatible watermark. The `deleteWebhook` `400: Logged out` startup behavior also remains a product/compatibility decision needing live local Bot API proof.

## What This Changes

The issue asks OpenClaw to make Telegram local Bot API `apiRoot` polling dispatch inbound messages reliably, reset or scope update offsets on endpoint changes, and avoid a startup crash loop during Bot API migration cooldown.

## Best Possible Solution

Keep this open and repair the bundled Telegram plugin so endpoint changes reset or scope polling offsets, startup webhook cleanup handles migration cooldown deliberately, and a live local Bot API smoke proves inbound dispatch plus reply delivery.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for the endpoint-offset failure because current main keys offsets by account/token but not `apiRoot`. A full live reproduction is still missing for the local Bot API dispatch drop and `logOut` cooldown behavior.

## Solution Assessment

No final patch is proposed here; the best path is a bounded Telegram-plugin repair plus live local Bot API validation. Closing as implemented would be premature because current main and `v2026.6.8` still show the endpoint-offset gap.
