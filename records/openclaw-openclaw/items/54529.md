## Summary

Keep open. Current main still has the Telegram apiRoot plumbing, but the reported local Bot API polling migration gap remains: update offsets are keyed by account and bot token rather than endpoint, and the polling cleanup path still only downgrades recoverable network failures, not the reported Telegram 400 logged-out migration response.

## What This Changes

The issue requests Telegram local Bot API polling migration support so inbound updates dispatch after switching apiRoot, persisted offsets do not suppress local-server updates, and deleteWebhook logged-out responses do not crash-loop startup.

## Best Possible Solution

Fix this in the Telegram plugin by persisting a normalized Bot API endpoint identity with polling offset state, treating legacy or mismatched endpoint state as stale while preserving bot-token safety, and allowing only the expected local Bot API migration cleanup response to proceed without weakening normal auth/error handling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist a higher update offset for the same account/token, switch apiRoot to a local Bot API server whose update ids are lower, and observe current main's tracker skip those updates before handler dispatch.

## Solution Assessment

Yes. A narrow Telegram plugin fix is the best path; this should not require core/plugin SDK changes, and the apiRoot client passthrough already exists.

- Keep deleteWebhook using the configured apiRoot through the existing bot client; add only a narrow 400 logged-out migration allowance if safe and well-tested.
- Do not swallow 401 bad-token failures, malformed apiRoot 404s, or unrelated Telegram 4xx errors.
