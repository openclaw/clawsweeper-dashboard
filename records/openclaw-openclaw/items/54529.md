## Summary

Keep open. Current main still has the Telegram apiRoot plumbing, but the local Bot API migration failure remains source-reproducible because polling offsets are not scoped to the Bot API endpoint and deleteWebhook only downgrades recoverable network failures.

## What This Changes

Requests that Telegram apiRoot polling dispatch inbound local Bot API updates, reset or scope offsets when the endpoint changes, and survive expected deleteWebhook logged-out migration responses.

## Best Possible Solution

Fix this in the Telegram plugin by storing a normalized endpoint identity with offset state and handling the specific local Bot API migration deleteWebhook response without weakening normal auth or arbitrary 4xx failures.

## Reproduction Assessment

Yes, for a source-level reproduction: persist a higher lastUpdateId for the same Telegram account/token, switch apiRoot to a local Bot API server with lower update_id values, and current main will skip those updates before handler dispatch. I did not run a live Bot API container in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Telegram plugin polling state and startup cleanup path; the core/plugin SDK and public config surface do not need to change.
