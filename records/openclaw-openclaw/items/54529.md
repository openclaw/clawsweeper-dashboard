- `P2`: The issue affects a real Telegram configuration with message delivery and startup reliability impact, but it is limited to local Bot API migration rather than the default Telegram path.
- `impact:message-loss`: The reported and source-supported failure can leave Telegram updates received by polling but not dispatched to agent sessions.

## Summary

Keep this open: current main has basic Telegram `apiRoot` passthrough, but the reported local Bot API migration failure is still plausible because update offsets remain endpoint-insensitive and `deleteWebhook` logged-out responses are not handled as a documented nonfatal migration state.

## What This Changes

The issue asks OpenClaw to make Telegram `channels.telegram.apiRoot` work reliably with a self-hosted Bot API server for inbound dispatch, offset migration, and startup webhook cleanup.

## Best Possible Solution

Keep the issue open for maintainer-reviewed Telegram local Bot API migration work: make offset handling endpoint-aware, narrowly handle logged-out startup during migration, document the recovery path, and prove it against a real local Bot API server.

## Reproduction Assessment

No high-confidence live repro was performed, but current source confirms two central failure paths: endpoint-insensitive persisted offsets and fatal handling for non-network `deleteWebhook` setup errors.

## Solution Assessment

Unclear until maintainers choose the migration contract; the best technical direction is endpoint-aware offset reset/scoping plus narrow logged-out startup handling with live local Bot API proof.
