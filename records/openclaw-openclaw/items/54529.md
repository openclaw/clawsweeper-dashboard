## Label Justifications

- `P2`: The bug affects a documented Telegram apiRoot deployment path with limited blast radius to custom or local Bot API users.
- `impact:message-loss`: The source-reproducible stale-offset path can silently skip inbound Telegram updates before agent session dispatch.
- `impact:crash-loop`: The report and current startup cleanup path can repeatedly restart when deleteWebhook returns the expected migration-time logged-out rejection.

## Summary

Keep open. Current main and the latest release still document Telegram `apiRoot`, but polling offsets are persisted without endpoint identity and `deleteWebhook` startup cleanup still only treats network-style failures as recoverable, so the local Bot API migration path can still lose inbound updates or restart-loop.

## What This Changes

The issue asks for Telegram `apiRoot` local Bot API polling to dispatch inbound updates, reset or partition offsets on endpoint changes, and avoid fatal `deleteWebhook` failures during migration.

## Best Possible Solution

Endpoint-scoped Telegram offset state should make same-token apiRoot switches start without stale watermarks, while startup cleanup should treat the local Bot API migration logged-out response as a recoverable transition.

## Reproduction Assessment

Yes, at source level: persist a same-token offset from the official Bot API, switch only `apiRoot` to a local Bot API server with lower update IDs, and current main seeds local `getUpdates` from the stale higher offset. I did not run the live Docker Bot API migration in this read-only review.

## Solution Assessment

Yes. The maintainable fix belongs inside the Telegram plugin offset state, monitor persistence threading, polling startup, and narrow startup-cleanup error classification; core plugin APIs and user config do not need to change.
