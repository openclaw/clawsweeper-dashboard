## Summary

Keep open: current main and v2026.6.6 still persist Telegram polling offsets by account/token only while polling uses apiRoot separately, so switching the same bot token between the official and a local Bot API root can reuse an incompatible offset; webhook cleanup also still treats non-network 400 errors such as Logged out as fatal setup failures.

## What This Changes

The issue reports that Telegram local Bot API apiRoot migrations can drop inbound updates because stale official-api offsets are reused, and can crash-loop during deleteWebhook after logOut.

## Best Possible Solution

Fix this inside the Telegram plugin by making update-offset persistence endpoint-aware for apiRoot switches and by handling the expected local-server migration deleteWebhook cooldown without a fatal restart loop.

## Reproduction Assessment

No live local Bot API repro was run, but the source path is clear: the stored update offset is not keyed by apiRoot, so switching endpoints with the same token can seed getUpdates with the wrong update sequence.

## Solution Assessment

Yes, the best fix is a narrow Telegram plugin repair to carry normalized apiRoot into offset identity or rotation decisions and to add focused cleanup handling for the Logged out migration window.
