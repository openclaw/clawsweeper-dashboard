## Summary

Keep open. Current main still has the Telegram apiRoot plumbing, but the reported local Bot API migration failure remains source-reproducible because persisted polling offsets are not scoped to the configured Bot API endpoint and the deleteWebhook startup path only downgrades recoverable network failures.

## What This Changes

The issue requests that Telegram local Bot API apiRoot migrations dispatch inbound updates normally, reset or scope incompatible offsets, and avoid a fatal deleteWebhook loop during logOut cooldown.

## Best Possible Solution

Fix this inside the Telegram plugin by recording a normalized endpoint identity with offset state and handling the specific local Bot API migration deleteWebhook Logged out case without weakening normal 401 auth or arbitrary 4xx failures.

## Reproduction Assessment

Yes, at source level: persist a higher lastUpdateId for the same Telegram account/token, switch apiRoot to a local Bot API server that emits lower update_id values, and current main will reject those updates before handler dispatch. I did not run the live Bot API container path in this read-only review.

## Solution Assessment

Yes. The narrow maintainable repair belongs in Telegram polling offset state and startup cleanup handling; the core plugin SDK and public config surface do not need to change.
