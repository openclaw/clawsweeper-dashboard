## Summary

Keep open: current main and the latest release support Telegram `apiRoot` generally, but endpoint switches still preserve the update watermark by account/token rather than Bot API endpoint, and the `deleteWebhook` `400: Logged out` behavior still needs a maintainer policy decision plus live local Bot API proof.

## What This Changes

The issue asks OpenClaw to make Telegram local Bot API `apiRoot` endpoint switches dispatch inbound updates, reset incompatible polling offsets, and avoid fatal `deleteWebhook` cooldown loops.

## Best Possible Solution

Define Telegram local Bot API endpoint identity in the offset lifecycle, add a tested reset/migration or doctor path for endpoint switches, and decide the explicit startup policy for `deleteWebhook` logged-out cooldowns with live local Bot API validation.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The source path is reproducible for the offset mismatch: current polling reuses an account/token-scoped `lastUpdateId` when `apiRoot` changes, then applies it to the local endpoint.

## Solution Assessment

Unclear. The endpoint-offset repair belongs in the Telegram plugin, but the exact migration/reset semantics and `400: Logged out` fail-open behavior need maintainer direction before an automated fix is safe.
