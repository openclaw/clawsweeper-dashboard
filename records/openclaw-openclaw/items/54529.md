## Label Justifications

- `P2`: This is a normal-priority Telegram channel reliability bug with bounded blast radius but real inbound-delivery and startup impact.
- `impact:message-loss`: The reported behavior silently drops Telegram inbound updates after receipt instead of dispatching them to agent sessions.
- `impact:crash-loop`: The report includes repeated Telegram startup failures during local Bot API migration when `deleteWebhook` sees `400: Logged out`.

## Summary

Keep open: current main and v2026.6.6 still seed local Bot API polling from an account/token-scoped Telegram update offset that does not include the Bot API endpoint, and startup webhook cleanup still treats non-network Bot API errors as fatal setup errors. The configured apiRoot is now passed into polling and Bot construction, so the remaining issue is migration-state handling plus live confirmation of the local Bot API dispatch path.

## What This Changes

The issue asks for Telegram `apiRoot` local Bot API polling to dispatch inbound updates after endpoint migration, reset or scope incompatible offsets, and avoid `deleteWebhook` crash loops during `logOut` cooldown.

## Best Possible Solution

Keep the issue open and repair the bundled Telegram plugin so update offsets are reset or scoped across Bot API endpoint changes, migration-time webhook cleanup errors are handled deliberately, and a local Bot API smoke proves inbound dispatch and outbound reply after switching roots.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for the endpoint-offset failure because current main keys offsets by account/token but not `apiRoot`. A full live reproduction is still missing for the local Bot API dispatch drop and `logOut` cooldown behavior.

## Solution Assessment

No final patch is proposed here; the best path is a bounded Telegram-plugin repair plus live local Bot API validation. Closing as implemented would be premature because current main and v2026.6.6 still show the endpoint-offset gap.
