## Summary

Keep open. Current main and the latest release still preserve Telegram update offsets by account/token rather than API root, pass that watermark into the configured local `getUpdates` endpoint, and only continue past `deleteWebhook` failures for recoverable network errors; the remaining fix needs maintainer policy and live local Bot API migration proof.

## What This Changes

This issue reports that Telegram `channels.telegram.apiRoot` can receive local Bot API updates without dispatching them to sessions, with stale update offsets and `deleteWebhook` startup failures during migration.

## Best Possible Solution

Keep this as the canonical local Bot API migration tracker; define endpoint-switch offset/reset policy and a narrow `400: Logged out` cleanup rule, then prove inbound dispatch and outbound reply delivery against a live local Bot API server.

## Reproduction Assessment

No high-confidence live reproduction was established. The report has detailed logs and current source still shows the stale-offset and non-network `deleteWebhook` paths, but a live self-hosted Bot API server run is needed to prove the full raw-update-to-dispatch failure.

## Solution Assessment

Unclear. The fix likely belongs in Telegram polling state and startup cleanup, but maintainers need to choose endpoint-scoped offsets versus reset/doctor behavior and the exact safe scope for tolerating `400: Logged out`.
