## Label Justifications

- `P2`: The bug affects configured Telegram local Bot API deployments with a bounded Telegram plugin repair surface.
- `impact:message-loss`: The stale endpoint-agnostic offset can make inbound Telegram updates from the local server never reach agent dispatch.
- `impact:crash-loop`: The same report and source path show startup retry/exit behavior when deleteWebhook rejects during Bot API migration.

## Summary

Keep open. Current main still preserves Telegram polling offsets by account and bot token, but not by the configured Bot API endpoint, so switching the same token from the official API to a local apiRoot can keep polling above the local server's update_id range; the deleteWebhook startup path also still only treats network-style failures as recoverable.

## What This Changes

The issue asks OpenClaw to dispatch Telegram inbound updates correctly after switching a bot token from the official Bot API to a local apiRoot, including endpoint-aware offset handling and nonfatal deleteWebhook behavior during migration.

## Best Possible Solution

Store Telegram update offsets with a normalized Bot API endpoint identity and handle the migration-time logged-out deleteWebhook response narrowly, while preserving existing apiRoot routing and ordinary auth failure behavior.

## Reproduction Assessment

Yes, at source level: write a same-token v3 offset from the official Bot API, switch only channels.telegram.apiRoot to a local server whose update IDs are lower, and current main seeds getUpdates from the stale higher offset. I did not run the live Docker Bot API path.

## Solution Assessment

Yes. The narrow maintainable fix belongs in Telegram plugin offset persistence, monitor threading, polling startup, and specific startup-cleanup error classification; core plugin APIs and user config do not need to change.
