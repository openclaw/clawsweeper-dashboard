## Label Justifications

- `P2`: The bug affects a documented Telegram self-hosted Bot API deployment path with limited blast radius to custom `apiRoot` users.
- `impact:message-loss`: The stale-offset path can silently skip inbound Telegram updates before they reach agent session dispatch.
- `impact:crash-loop`: The reported `deleteWebhook` logged-out response can keep the Telegram provider in startup retry/exit behavior during local Bot API migration.

## Summary

Keep open. Current main and the latest release still expose Telegram `apiRoot`, but the persisted polling offset is scoped to account/token identity rather than endpoint identity, and startup cleanup still treats a Telegram `400: Logged out` `deleteWebhook` rejection as fatal rather than a recoverable local Bot API migration state.

## What This Changes

The issue asks OpenClaw to make Telegram local Bot API `apiRoot` migrations dispatch inbound updates reliably, avoid stale update offsets, and handle migration-time `deleteWebhook` logged-out failures without restart loops.

## Best Possible Solution

Endpoint-aware Telegram offset state should invalidate or separate same-token watermarks when `apiRoot` changes, while webhook startup cleanup should narrowly treat local Bot API migration `Logged out` responses as recoverable without weakening real auth failures.

## Reproduction Assessment

Yes, at source level: persist a same-token offset from one Bot API endpoint, switch only `apiRoot` to a local endpoint with lower update ids, and current main seeds local `getUpdates` from the stale higher watermark. I did not run the live Docker Bot API migration in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Telegram plugin offset store, monitor threading, isolated polling startup, and focused cleanup-error classification; core plugin APIs and new user config are not required.
