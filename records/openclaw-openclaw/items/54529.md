- `impact:message-loss`: The reported and source-reproducible stale-offset path can silently skip inbound Telegram updates before agent session dispatch.

## Summary

Keep open. Current main and the latest release still persist Telegram polling offsets by account and bot-token identity, but not by the configured Bot API endpoint, so switching the same token from the official API to a local apiRoot can keep polling from a stale higher watermark; the deleteWebhook startup path also still only treats network-style failures as recoverable.

## What This Changes

The issue reports that Telegram local Bot API apiRoot polling can receive updates without dispatching them after endpoint migration, and asks for endpoint-aware offset handling plus safer deleteWebhook behavior during migration.

## Best Possible Solution

Fix Telegram offset persistence to include a normalized Bot API endpoint identity, thread that identity through monitor read/write and isolated/non-isolated polling startup, and add a narrow recoverable classification for the local Bot API migration logged-out deleteWebhook response without weakening ordinary auth failures.

## Reproduction Assessment

Yes, at source level: write a same-token offset from the official Bot API, switch only apiRoot to a local Bot API server with lower update IDs, and current main seeds local getUpdates from the stale higher offset. I did not run the live Docker Bot API path in this read-only review.

## Solution Assessment

Yes. The maintainable fix belongs inside the Telegram plugin offset state, monitor persistence threading, polling startup, and specific startup-cleanup error classification; core plugin APIs and user config do not need to change.
