## Label Justifications

- `P2`: The issue reports a bounded Telegram local Bot API migration failure with real inbound delivery and startup impact, but no evidence of a core-wide outage.
- `impact:message-loss`: The reported failure receives or consumes Telegram updates without dispatching them to agent sessions.
- `impact:crash-loop`: The report includes repeated Telegram startup failure when `deleteWebhook` returns `400: Logged out` during local Bot API migration.

## Summary

Keep open: current main and the latest release still support Telegram `apiRoot`, but the remaining reported endpoint-switch behavior is not implemented because offset state is keyed by account/token identity rather than the Bot API root, and `400: Logged out` startup handling still needs a maintainer policy decision plus live local Bot API proof.

## What This Changes

The issue requests reliable Telegram local Bot API `apiRoot` migration behavior so inbound updates dispatch to agent sessions, endpoint changes do not reuse incompatible update offsets, and `deleteWebhook` logOut cooldown failures do not crash-loop the channel.

## Best Possible Solution

Repair the bundled Telegram plugin so apiRoot endpoint changes reset or scope polling offsets, define tested startup behavior for `400: Logged out` during local Bot API migration, and prove inbound dispatch plus reply delivery against a real local Bot API server.

## Reproduction Assessment

Partly. Source inspection gives a concrete path for the endpoint-offset failure because current main and `v2026.6.8` key offsets by account/token but not apiRoot; a full live reproduction is still missing for the raw-update dispatch drop and `deleteWebhook` cooldown path.

## Solution Assessment

No final patch is proposed here, and closing as implemented would be premature. The best fix likely belongs in the Telegram plugin’s offset state/migration and polling startup policy, but maintainers need to choose the apiRoot-reset semantics and prove them live.
