## Summary

Keep open. Current main supports Telegram `apiRoot` generally, but the reported local Bot API polling migration gap still appears valid: persisted polling offsets are keyed only by account and bot token, so switching the same bot token from the official API to a local Bot API server can reuse an incompatible update watermark and skip handler dispatch. The startup `deleteWebhook` path also still only downgrades recoverable network failures, not the reported Telegram 400 `Logged out` migration response.

## What This Changes

Make Telegram local Bot API polling migration dispatch inbound updates by endpoint-scoping or resetting persisted offsets and handling the expected `deleteWebhook` cooldown response without masking real auth failures.

## Best Possible Solution

Fix the Telegram plugin so polling offset state includes a normalized Bot API endpoint identity, legacy or mismatched endpoint state is treated as stale while preserving bot-token mismatch safety, and the local-server migration cleanup path can pass through the expected cooldown response without weakening normal Telegram auth/error handling.

## Repair Work Prompt

Fix openclaw/openclaw#54529 in the Telegram plugin only. The observable bug is that switching `channels.telegram.apiRoot` from the official Telegram API to a self-hosted/local Bot API server can reuse the same account/bot-token update offset file, causing lower local-server update ids to be skipped after raw-update logging and never dispatched to sessions. Also handle the local Bot API migration `deleteWebhook` 400 `Logged out` cooldown response narrowly so polling can continue when appropriate.

Expected implementation boundary:
- Add a normalized endpoint identity/fingerprint to Telegram polling offset state and APIs, based on the effective normalized Bot API root.
- Treat missing or mismatched endpoint identity as stale at startup, while preserving existing bot-token mismatch safety and invalid-offset guards.
- Persist endpoint identity with future offset writes.
- Ensure switching from official API to local `apiRoot` starts without the incompatible old watermark and dispatches valid inbound updates.
- Keep `deleteWebhook` using the configured `apiRoot` through the existing bot client; add only a narrow `400: Logged out` migration allowance if it is safe and well-tested. Do not swallow 401 bad-token failures, malformed `apiRoot` 404s, or unrelated Telegram 4xx errors.
- Do not change core/plugin SDK contracts unless a generic seam is clearly needed, do not delete all Telegram offsets blindly, and do not add dependencies.
- Add a single-line user-facing changelog fix entry under the active version with `Thanks @marieldejesus12`.
