## Summary

Keep open. Current main improves Telegram polling and legacy token-offset rotation, but same-token apiRoot endpoint switches still reuse the persisted polling watermark and the migration-time deleteWebhook `400: Logged out` path remains fatal, so the local Bot API migration bug is still source-reproducible.

## What This Changes

The issue asks OpenClaw to dispatch Telegram updates after switching `channels.telegram.apiRoot` to a self-hosted Bot API server, reset incompatible polling offsets, and tolerate migration-time deleteWebhook `Logged out` responses.

## Best Possible Solution

Fix the Telegram plugin by persisting a normalized Bot API endpoint identity with update-offset state and by handling the specific local Bot API migration `deleteWebhook` Logged out case without weakening normal token/auth failures.

## Reproduction Assessment

Yes, at source level: write a same-token offset from the official Bot API, switch the same account to a local apiRoot whose update IDs are lower, and current main seeds polling from the stale higher offset. I did not run the live Docker Bot API server path in this read-only review.

## Solution Assessment

Yes. The narrow maintainable repair belongs in Telegram offset state, monitor/isolated-ingress persistence, and startup cleanup classification; core plugin APIs and user config do not need to change.
