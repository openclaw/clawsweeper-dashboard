- `impact:crash-loop`: The report is about an agent turn looping on repeated identical failed edits, which can hang the run and waste tokens until manually interrupted.

## Summary

Keep open: current main has related loop-detection infrastructure, but it is opt-in for ordinary rolling tool-call loops and still does not provide the requested default three-strike guard for repeated identical failed edit calls.

## What This Changes

The issue requests runtime protection that stops a turn after repeated identical failed tool calls and proposes a configurable `agents.defaults.maxIdenticalToolFailures` defaulting to 3.

## Best Possible Solution

Extend the existing `tools.loopDetection` runtime path with a maintainer-approved default failed-outcome guard and clear user-facing abort event, instead of adding a parallel config surface unless maintainers explicitly want that API.

## Reproduction Assessment

No live high-confidence replay was established. Source inspection does confirm that current main’s default path does not block repeated identical failed calls unless rolling loop detection is explicitly enabled or the run is in the post-compaction guard window.

## Solution Assessment

Unclear as proposed. The desired behavior fits existing loop-detection ownership, but maintainers should decide the default threshold, event semantics, and config path before implementation.
