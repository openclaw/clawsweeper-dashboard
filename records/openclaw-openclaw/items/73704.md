## Summary

Keep open. Current main still ignores the configured compaction model in the safeguard auto-compaction runtime, and this PR is a relevant implementation path, but the current patch has a blocking gap for configured inline provider models and the PR head is dirty with failing checks.

## What This Changes

This PR threads `modelRegistry` into safeguard extension factory registration, resolves a compaction override before setting runtime state, flips the safeguard hook to prefer the runtime model over the context model, and adds tests plus a changelog entry.

## Best Possible Solution

Land this PR or a focused successor after updating the safeguard runtime target resolver to honor the same supported model-selection inputs as normal OpenClaw model resolution, including inline `models.providers.*.models`, while preserving no-override fallback, provider-switch auth behavior, the Greptile-requested warning, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure safeguard compaction with `agents.defaults.compaction.model` different from the active session model; current main stores the session model in runtime and the hook prefers `ctx.model`. For the PR finding, configure the override to a custom inline `models.providers.*.models` entry; the new resolver only calls `modelRegistry.find`, while normal model resolution handles inline entries first.

## Solution Assessment

Not yet. The proposed direction is the right narrow fix, but the patch should not resolve changed overrides with a registry-only lookup because that leaves supported configured provider models falling back to the session model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

- **[P2] Resolve inline compaction model targets:** `src/agents/pi-embedded-runner/extensions.ts:163-166`
  - body: The new resolver only calls `modelRegistry.find` for overrides that differ from the session model. That skips OpenClaw's supported inline `models.providers.*.models` path, which normal `resolveModel` handles before registry lookup, so a custom configured compaction model still falls back to the session model instead of honoring `agents.defaults.compaction.model`. Please resolve the safeguard target through the same model-resolution contract or add equivalent inline-model handling and coverage.
