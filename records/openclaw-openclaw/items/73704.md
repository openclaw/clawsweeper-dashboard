## Summary

Keep open: the PR targets a real safeguard compaction bug and the main valid-override path looks directionally right, but it still needs observed real behavior proof and the unresolved-override path has an inconsistent fallback/cancel behavior that should be fixed before merge.

## What This Changes

The PR resolves configured safeguard compaction provider/model targets before embedded runtime registration, gives the runtime model precedence in the safeguard hook, threads registry/workspace context through callers, and adds tests plus a changelog entry.

## Best Possible Solution

Fix the unresolved-override path so it behaves consistently in both initialized runs and the direct `compact.ts` workflow, keep the focused valid-override behavior, and require redacted real safeguard auto-compaction proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main registers the session model in safeguard runtime and then prefers `ctx.model`, so a configured different compaction model cannot win in initialized safeguard sessions. I did not run a live provider-backed compaction.

## Solution Assessment

No, not yet. Reusing the existing compaction target resolver and preferring the resolved runtime model is the right narrow direction, but the unresolved-override branch currently claims fallback while direct `compact.ts` can cancel, and the PR still lacks real behavior proof.
