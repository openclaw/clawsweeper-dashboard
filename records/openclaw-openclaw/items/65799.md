- `P2`: This is a focused Feishu inbound reliability and proxy-policy bug with an open implementation candidate, but it is limited to proxy-configured deployments.
- `impact:message-loss`: The reported failure mode stops real Feishu inbound messages from arriving while the account can still appear healthy.
- `impact:security`: The issue concerns whether a channel WebSocket should inherit operator proxy environment settings and route egress through an ambient proxy by default.

## Summary

Keep open. Current main and latest release still route Feishu WebSocket creation through ambient proxy-agent resolution, and this issue has an open closing PR that should be reviewed, rebased, or replaced before the issue closes.

## What This Changes

The issue requests that Feishu WebSocket stop inheriting ambient `HTTP_PROXY`, `HTTPS_PROXY`, or `ALL_PROXY` by default and use proxying only through an explicit opt-in.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/65802 with a Feishu-owned default-direct WebSocket proxy policy, an explicit opt-in path, docs if maintainers keep the new surface, and regression coverage for both default and opt-in behavior.

## Reproduction Assessment

Yes, at source level. Setting an ambient HTTP(S) proxy before Feishu WebSocket creation still flows through `getWsProxyAgent()` into the Lark `WSClient` agent option on current main; I did not run a live Feishu gateway in this read-only review.

## Solution Assessment

Unclear as a final merge path. A Feishu-local default-direct gate is the narrow code boundary, but maintainers still need to settle env-only versus config-backed opt-in behavior and document how this interacts with intentional proxy routing.
