## Label Justifications

- `P1`: The current default can break the documented default Feishu WebSocket inbound path for real users with ambient proxy env set.
- `impact:message-loss`: The report is about Feishu inbound messages stopping when the WebSocket cannot connect.
- `impact:security`: The issue concerns proxy egress policy for Feishu WebSocket traffic and needs an explicit routing boundary.

## Summary

Current main still resolves and passes an ambient proxy agent into Feishu WebSocket creation, so the reported breakage remains valid; this should stay open while the linked fix PR is reviewed, rebased, or replaced.

## What This Changes

The issue requests Feishu WebSocket to stop using ambient proxy environment variables by default and instead use an explicit opt-in for proxy routing.

## Best Possible Solution

Land a Feishu-owned default-direct WebSocket proxy fix with an explicit documented opt-in and regression tests, using the linked open PR or a maintainer replacement if that branch remains unmergeable.

## Reproduction Assessment

Yes, at source level. Setting proxy env before `createFeishuWSClient()` follows `getWsProxyAgent()` into `resolveAmbientNodeProxyAgent()` and passes the resulting agent into the Lark WS client; I did not run a live Feishu gateway in this read-only review.

## Solution Assessment

Unclear as a final product path. A Feishu-local default-direct gate is the narrow code fix, but maintainers still need to approve the opt-in surface and document how intentional proxy routing should work.
