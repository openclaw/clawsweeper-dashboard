## Label Justifications

- `P1`: The report describes an official Feishu channel workflow where inbound messages stop arriving when ambient proxy env misroutes the WebSocket connection.
- `impact:message-loss`: The affected Feishu WebSocket path is the default inbound transport, and failure stops real Feishu messages from reaching OpenClaw.
- `impact:security`: The final fix determines whether Feishu/Lark WebSocket egress honors or bypasses operator proxy boundaries.

## Summary

Keep open. Current main and the latest release still route Feishu WebSocket creation through ambient proxy resolution, and an open closing PR remains the direct implementation candidate while broader Feishu proxy policy is still under maintainer discussion.

## What This Changes

The issue requests making Feishu WebSocket direct-by-default instead of inheriting ambient HTTP(S)/ALL proxy environment variables, with explicit opt-in for proxy use.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/65802 with a Feishu-owned default-direct WebSocket proxy policy, documented explicit opt-in, regression coverage, and redacted real Feishu proof.

## Reproduction Assessment

Yes, at source level. Setting HTTP(S)/ALL proxy env before Feishu WebSocket creation follows getWsProxyAgent() to resolveAmbientNodeProxyAgent() and passes an agent into Lark WSClient; I did not run a live Feishu gateway.

## Solution Assessment

Unclear as a final merge path. A Feishu-local direct-by-default gate is the narrow code path, but maintainers still need to settle the explicit opt-in surface and document how it interacts with managed proxy routing.
