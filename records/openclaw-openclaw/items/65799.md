## Label Justifications

- `P1`: The report describes a real broken Feishu channel workflow where inbound messages stop arriving in proxy-enabled environments.
- `impact:message-loss`: A failed Feishu WebSocket long connection prevents real inbound messages from reaching OpenClaw.
- `impact:security`: The remaining fix controls whether Feishu WebSocket egress honors or bypasses operator proxy routing expectations.

## Summary

Keep open. Current main and the latest v2026.5.18 release still pass an ambient proxy-derived agent into Feishu WebSocket creation, and the open closing PR at https://github.com/openclaw/openclaw/pull/65802 should be reviewed, rebased, or replaced before this issue is closed.

## What This Changes

The issue asks Feishu WebSocket connections to stop inheriting ambient proxy environment variables by default and to use explicit opt-in behavior for proxy routing.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/65802 with a Feishu-owned default-direct WebSocket proxy fix, documented opt-in behavior, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. On current main, setting HTTP(S) proxy env before createFeishuWSClient() follows getWsProxyAgent() to resolveAmbientNodeProxyAgent() and passes the resulting agent into the Lark WSClient; I did not run a live Feishu gateway.

## Solution Assessment

No, current main is not the solution. A Feishu-local default-direct gate is plausible, but the final path still needs maintainer agreement on env-only versus config-backed proxy opt-in and managed-proxy documentation.
