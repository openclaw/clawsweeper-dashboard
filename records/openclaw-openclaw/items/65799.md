## Label Justifications

- `P1`: The reported behavior can break a real Feishu inbound channel workflow for users whose gateway inherits incompatible proxy variables.
- `impact:security`: Ambient proxy inheritance can unintentionally route Feishu WebSocket traffic through an operator-uncontrolled proxy path.
- `impact:message-loss`: When the inherited proxy breaks the WebSocket connection, Feishu inbound messages stop arriving.

## Summary

Keep open: current main and the latest release still pass an ambient proxy agent into Feishu WebSocket, while https://github.com/openclaw/openclaw/pull/90324 is the open closing PR that owns the proposed fix path.

## What This Changes

Request to stop Feishu WebSocket from using ambient HTTP(S)/ALL_PROXY by default and add an explicit proxy opt-in for deployments that need it.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90324 or a maintainer-approved replacement, then close this issue once the Feishu WS proxy default, opt-in contract, tests, docs, and proof are complete.

## Reproduction Assessment

Yes at source level: current main unconditionally resolves an ambient proxy agent and passes it to the Feishu Lark WSClient, whose pinned SDK uses that agent for the WebSocket connection. I did not run a live Feishu/proxy gateway scenario in this read-only review.

## Solution Assessment

Unclear until maintainer review resolves the compatibility tradeoff: direct-by-default with an explicit opt-in is a narrow technical fix, but it changes existing proxy-dependent deployments. The best process is to finish the linked PR rather than queue a competing fix or close this issue early.
