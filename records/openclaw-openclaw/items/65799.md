## Label Justifications

- `P1`: The current default can break the normal Feishu WebSocket inbound path for real users with ambient proxy env set.
- `impact:security`: The issue concerns whether Feishu WebSocket egress follows ambient process proxy routing by default.
- `impact:message-loss`: The reported failure stops Feishu inbound messages when the WebSocket cannot connect.

## Summary

Keep open: current main still passes an ambient proxy agent into Feishu WebSocket creation, and an open closing PR already targets this issue but is conflicting and needs maintainer proxy-policy review.

## What This Changes

The issue requests changing Feishu WebSocket proxy behavior so ambient proxy environment variables are not inherited by default, with an explicit opt-in for proxy use.

## Best Possible Solution

Land or replace the open Feishu-owned default-direct WebSocket proxy fix with an explicit opt-in, regression tests, user-facing docs, and coordination with the broader Feishu mixed-proxy policy work.

## Reproduction Assessment

Yes, at source level. Setting proxy env before `createFeishuWSClient()` follows `getWsProxyAgent()` into `resolveAmbientNodeProxyAgent()` and passes the resulting agent into the Lark WS client; I did not run a live Feishu gateway in this read-only review.

## Solution Assessment

Unclear as a final product path. A Feishu-local default-direct gate is the narrow code surface, but maintainers still need to approve the opt-in surface and document how intentional proxy routing should work.
