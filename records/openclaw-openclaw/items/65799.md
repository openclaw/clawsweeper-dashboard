## Label Justifications

- `P1`: The reported behavior can break a real Feishu inbound channel workflow while credentials and status probes may still appear healthy.
- `impact:security`: Ambient proxy inheritance can unintentionally route Feishu WebSocket traffic through an operator-uncontrolled proxy path.
- `impact:message-loss`: When the inherited proxy breaks the WebSocket connection, Feishu inbound messages stop arriving.

## Summary

Keep open: current main and latest release still create a Feishu WebSocket proxy agent from ambient proxy configuration, while https://github.com/openclaw/openclaw/pull/90324 is the open implementation candidate and has not merged.

## What This Changes

The issue requests Feishu WebSocket connections to avoid ambient proxy environment variables by default, with an explicit opt-in for deployments that need proxy routing.

## Best Possible Solution

Review and land a narrow Feishu-owned fix via https://github.com/openclaw/openclaw/pull/90324, including opt-in documentation, upgrade safety, tests for default and opt-in behavior, and current-head real transport proof before closing this issue.

## Reproduction Assessment

Yes at source level: current main unconditionally resolves an ambient proxy agent for Feishu WebSocket and passes it into the Lark SDK WSClient, whose dependency source uses that agent for the WebSocket connection. I did not run a live Feishu gateway scenario in this read-only review.

## Solution Assessment

The Feishu-owner direct-by-default fix looks like the narrowest technical shape, but the explicit env opt-in is a compatibility and security product decision. The best path is to finish and review the linked PR rather than close this issue early or queue a competing fix.
