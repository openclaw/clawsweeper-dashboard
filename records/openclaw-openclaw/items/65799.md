## Label Justifications

- `P1`: The reported behavior can break a real Feishu inbound channel workflow while credentials or status checks appear healthy.
- `impact:security`: The issue concerns unintended Feishu WebSocket routing through ambient proxy settings that may be outside the operator's intended Feishu trust boundary.
- `impact:message-loss`: When the WebSocket connection fails through the inherited proxy, Feishu inbound messages stop arriving.

## Summary

Keep open: current main and v2026.6.6 still unconditionally pass an ambient proxy agent into Feishu WebSocket creation, while the linked implementation PR is open and not yet merged.

## What This Changes

The issue requests that Feishu WebSocket stop inheriting ambient proxy environment variables by default and only use them through an explicit opt-in.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90324 or an equivalent Feishu-owned fix that defaults WebSocket transport to direct, preserves an explicit proxy opt-in, documents the upgrade path, and includes proof before closing this issue.

## Reproduction Assessment

Yes, at source level: current main unconditionally resolves an ambient proxy agent and passes it to Feishu WSClient, and current tests assert proxy-env inheritance. I did not run a live Feishu gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainer review completes: a Feishu-owned direct-by-default fix with an explicit opt-in is the narrowest apparent path, but the new env contract and proxy-required upgrade behavior need product/security approval.
