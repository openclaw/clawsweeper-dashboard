## Summary

Keep open. Current main still resolves an ambient proxy agent for Feishu WebSocket setup and passes it into the Lark WSClient when HTTP(S) proxy env is configured, while the open implementation PR #65802 explicitly closes this issue and is not merged yet.

## What This Changes

The issue requests changing Feishu WebSocket setup so it bypasses ambient HTTP(S)/SOCKS proxy environment variables by default and only uses them behind explicit opt-in.

## Best Possible Solution

Review and land PR #65802, or an equivalent Feishu-plugin-local change, so Feishu WebSocket does not inherit ambient proxy env by default. If proxy support is retained for special deployments, keep it explicit, tested, and documented under a stable opt-in environment variable.

## Reproduction Assessment

Yes. With HTTP_PROXY or HTTPS_PROXY set, current source creates a ProxyAgent and passes it to Feishu WSClient; with a broken ALL_PROXY also present, proxy-agent's wss routing can select the broken SOCKS proxy. ALL_PROXY-only is not a complete current-main reproduction because the OpenClaw gate ignores ALL_PROXY unless HTTP(S) proxy env also triggers agent creation.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Feishu plugin's WebSocket client construction, gating getWsProxyAgent behind explicit opt-in while leaving the shared ambient proxy helper available to transports that intentionally inherit process proxy settings.
