## Summary

Keep open: current main still lacks user-role memory-prefix stripping in Gateway and Control UI display paths, and this PR targets the right surfaces, but the branch still has a P2 indentation-loss bug in the new helper and lacks after-fix real behavior proof.

## What This Changes

The PR adds a shared sanitizer for memory-plugin `<relevant-memories>` prefixes, wires it into Gateway chat history and Control UI user-message rendering/normalization, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a prefix-specific shared sanitizer that removes only the memory plugin's injected block and generated separator, preserves user-authored tags and prompt indentation, and covers Gateway history plus Control UI rendering/normalization with regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the memory plugin prepends the exact block while Gateway and Control UI user display paths do not remove it; I did not run the live WebUI in this read-only review.

## Solution Assessment

No, not yet. The PR is scoped to the right code paths, but the helper should strip only the injected block plus generated separator instead of consuming all following whitespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve prompt indentation after memory prefixes:** `src/shared/text/assistant-visible-text.ts:560-561`
  - body: `stripInjectedRelevantMemoriesPrefix` skips every whitespace character after the closing memory tag. When the runtime stores the injected block, its separator, and then an indented user prompt, this removes user-authored indentation from the visible transcript. Strip only the generated separator and add a regression for an injected prefix followed by indented user text.
