## Summary

Keep open. Current main still leaks memory-plugin injected `<relevant-memories>` prefixes through user-role Gateway and Control UI display paths, and the PR addresses the right surfaces, but the latest branch still has a blocking whitespace-preservation bug in the new shared helper.

## What This Changes

The PR adds a shared injected memory-prefix sanitizer, wires it into Gateway and Control UI user-message display paths, adds regression tests, and updates the changelog.

## Best Possible Solution

Land this PR after narrowing `stripInjectedRelevantMemoriesPrefix` so it removes only the memory plugin's injected leading block plus the generated separator, preserves user-authored tags and prompt indentation, and keeps coverage across Gateway history, Control UI extraction, normalized grouped rendering, and the shared helper.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `memory-lancedb` returns the leading memory block as `prependContext`, while current main's user-message display paths strip only internal runtime context, inbound metadata, envelopes, and message-id hints.

## Solution Assessment

No, not yet. The PR is the right maintainable shape, but the helper's post-prefix whitespace loop is too broad and should preserve indentation belonging to the user's actual prompt.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve prompt indentation after the injected memory prefix:** `src/shared/text/assistant-visible-text.ts:560-562`
  - body: `stripInjectedRelevantMemoriesPrefix` skips every whitespace character after the closing memory tag. The runtime stores this as injected block plus a generated separator before the user prompt, so an indented prompt such as pasted code loses its leading spaces after the prefix is removed. Strip only the generated separator and add a regression for an injected prefix followed by indented user text.
  - confidence: 0.87
