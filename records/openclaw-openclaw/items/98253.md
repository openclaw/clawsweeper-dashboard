## Summary

Keep open: the PR is not merge-ready because the added stream controller is not actually used to terminate after `response.completed`, so the reported 120s post-completion stall can still happen. The PR also lacks real after-fix behavior proof in the body.

## What This Changes

This PR changes `src/agents/openai-transport-stream.ts` to add internal termination controls to the OpenAI Responses first-event timeout wrapper and to keep consuming the stream after `response.completed`.

## Best Possible Solution

Implement cleanup-aware terminal-event handling for both Responses stream processors, add stalled-after-`response.completed` regression coverage, and include redacted live or loopback after-fix proof.

## Reproduction Assessment

Yes, at source level: a Responses async iterable that yields `response.completed` and then never resolves would still leave the current and proposed loops awaiting the next event. I did not run a credentialed live OpenAI request in this read-only review.

## Solution Assessment

No: this patch adds termination controls but never invokes them, and it leaves the sibling shared Responses processor untouched. The best fix is a focused terminal-event cleanup path with regression coverage and proof.
