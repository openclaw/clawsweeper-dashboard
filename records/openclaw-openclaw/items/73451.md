## Summary

Keep open for maintainer handling. The reported optimizer-failure bug is valid, but this PR is not merge-ready: returning original bytes from `optimizeImageToJpeg` can mislabel non-JPEG inputs as JPEG and can bypass current image input guard failures. The safer consolidation path is already tracked in open PR #74243, which explicitly preserves MIME, keeps guard failures fail-closed, and supersedes this PR, but it has not merged yet and the paired issue #73424 remains open.

## What This Changes

The PR changes `optimizeImageToJpeg` to return the original source buffer with `resizeSide` and `quality` set to `0` when all JPEG resize/quality attempts fail instead of throwing `Failed to optimize image`.

## Best Possible Solution

Resolve the optimizer-failure family with a loader-bound fallback: return original bytes only when the original image is already under the channel cap, preserve the original content type, keep HEIC conversion, pixel-limit, unknown-dimension, and over-cap paths fail-closed, retain optimizer diagnostics, add focused regression tests and a changelog entry, then close this PR as superseded once the canonical path lands.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction exists: make `resizeToJpeg` throw for a non-JPEG or byte-small oversized-pixel image, and this PR's terminal fallback returns the original buffer; current `optimizeImageWithFallback` and `optimizeAndClampImage` then treat that result as JPEG and byte-cap-only.

## Solution Assessment

No. Returning original bytes from `optimizeImageToJpeg` is too low-level for this policy because that helper no longer knows the original MIME boundary or which optimizer errors are safety guards; the safer approach is the `loadWebMedia`/`optimizeAndClampImage` boundary used by #74243.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve image input guard failures:** `src/media/web-media.ts:657-662`
  - body: The fallback returns `source` after every `resizeToJpeg` attempt throws. `resizeToJpeg` is where the pixel-limit and unknown-dimension guards run, so a byte-small oversized PNG would now be accepted if it is under `maxBytes`. Re-throw those guard errors before falling back.
  - confidence: 0.9
- **[P2] Preserve original MIME on fallback:** `src/media/web-media.ts:657-662`
  - body: For non-HEIC PNG/WebP inputs, `source` is still the original bytes, but callers wrap any `optimizeImageToJpeg` result as JPEG and emit `image/jpeg`. Move fallback to the loader boundary or carry the original content type so callers do not receive PNG/WebP bytes labeled as JPEG.
  - confidence: 0.92
