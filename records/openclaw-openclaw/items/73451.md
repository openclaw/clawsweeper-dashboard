## Summary

Close this PR as superseded: the useful runtime fallback is now implemented more safely at the media-loader boundary, while the remaining root `sharp` package gap is tracked by a canonical open issue/PR. This branch still returns original bytes from the wrong helper and would weaken image guard and MIME contracts.

## What This Changes

The PR changes `optimizeImageToJpeg` to return the original source buffer with zero resize/quality metadata instead of throwing after all JPEG optimization attempts fail.

## Best Possible Solution

Keep the shipped loader-level fallback, finish the root `sharp` dependency work in the canonical package issue/PR, and do not merge the direct `optimizeImageToJpeg` passthrough.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: force `resizeToJpeg` to throw a guard or non-JPEG optimizer error, and this PR returns original bytes before the caller applies JPEG format/content-type mapping.

## Solution Assessment

No. The safer solution is the current loader-level fallback plus the canonical root dependency fix, because `optimizeImageToJpeg` lacks MIME, filename, and policy context for safe passthrough.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve image input guard failures:** `src/media/web-media.ts:657-662`
  - body: The fallback returns `source` after every `resizeToJpeg` attempt throws. `resizeToJpeg` is where pixel-limit and unknown-dimension guard errors are enforced, so this can accept byte-small unsafe images instead of failing closed.
  - confidence: 0.91
- **[P2] Preserve original MIME on fallback:** `src/media/web-media.ts:657-662`
  - body: For non-HEIC PNG/WebP inputs, `source` is still the original bytes, but callers wrap any `optimizeImageToJpeg` result as JPEG and emit `image/jpeg`. That can send PNG/WebP bytes with the wrong content type.
  - confidence: 0.92
- **[P3] Keep optimizer diagnostics on failed fallbacks:** `src/media/web-media.ts:657-662`
  - body: When the original is still over cap and all resize attempts fail, this return path turns the detailed optimizer failure into a generic cap error from the caller. Preserve the underlying optimizer cause for unsendable originals.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch has no supply-chain changes, but it weakens an existing image resource-safety boundary by treating guard failures as successful original-buffer fallbacks.

Concerns:

- **[medium] Fallback bypasses image guard errors:** `src/media/web-media.ts:657`
  - body: Returning the original buffer from `optimizeImageToJpeg` after every resize failure catches the same path that enforces pixel-limit and unknown-dimension checks, allowing under-cap unsafe images to continue downstream.
  - confidence: 0.9
