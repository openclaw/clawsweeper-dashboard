## Summary

Keep open. The PR targets a real media-preprocessing failure, but moving fallback into `optimizeImageToJpeg` weakens the image safety boundary, can mislabel original bytes as JPEG, and loses optimizer diagnostics; current `main` still lacks the safe loader-level fallback, while the paired #73424 issue remains open.

## What This Changes

The PR changes `optimizeImageToJpeg` so all failed resize attempts return the original source buffer with zero resize/quality metadata instead of throwing.

## Best Possible Solution

Implement fallback at the `loadWebMedia`/`optimizeAndClampImage` boundary: pass through only under-cap non-HEIC originals, preserve MIME and filename, rethrow image guard failures, keep optimizer cause diagnostics, add regression tests, and add the changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction exists by making `resizeToJpeg` throw for an under-cap image: current `main` propagates `Failed to optimize image`, while this PR returns the original buffer; the guard and MIME regressions follow from the current call path and tests.

## Solution Assessment

No. `optimizeImageToJpeg` lacks original content-type and policy context, so the maintainable fix belongs at the loader/clamp boundary rather than returning arbitrary original bytes from a JPEG optimizer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve image input guard failures:** `src/media/web-media.ts:657-662`
  - body: The fallback returns `source` after every `resizeToJpeg` attempt throws. `resizeToJpeg` is where pixel-limit and unknown-dimension guards run, so byte-small but unsafe images can be accepted if they are under `maxBytes`; rethrow those guard errors before falling back.
  - confidence: 0.91
- **[P2] Preserve original MIME on fallback:** `src/media/web-media.ts:657-662`
  - body: For non-HEIC PNG/WebP inputs, `source` is still the original bytes, but callers wrap every `optimizeImageToJpeg` result as JPEG and emit `image/jpeg`. Move fallback to the loader boundary or carry the original content type so consumers do not receive PNG/WebP bytes labeled as JPEG.
  - confidence: 0.92
- **[P3] Keep optimizer diagnostics on failed fallbacks:** `src/media/web-media.ts:657-662`
  - body: When the original is over cap and all resize attempts fail, this return path turns the detailed optimizer failure into a generic cap error from the caller. Preserve the underlying optimizer cause for unsendable originals so broken image backends stay diagnosable.
  - confidence: 0.82
- **[P3] Add the media fallback changelog entry:** `src/media/web-media.ts:657-662`
  - body: This is a user-facing `fix(media)` behavior change, but the PR only changes `src/media/web-media.ts`. Add the required single-line `CHANGELOG.md` entry so the release notes capture the image fallback behavior.
  - confidence: 0.95
