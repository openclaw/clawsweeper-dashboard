## Summary

Keep open. The PR is not safe to merge as written: it moves fallback policy into the low-level JPEG optimizer, where it cannot preserve original MIME type and can turn image input guard failures into accepted original bytes. Current main has not implemented the requested fallback, and open PR #74243 is the better tracked replacement path, but it has not landed yet and the paired source issue is still open.

## What This Changes

The PR changes `optimizeImageToJpeg` to return the original source buffer with `resizeSide` and `quality` set to `0` when all JPEG optimization attempts fail instead of throwing.

## Best Possible Solution

Resolve this optimizer-failure family at the loader or clamp boundary: return original bytes only when the original image is already under the media cap, preserve the original content type, keep HEIC conversion plus pixel-limit and unknown-dimension guard errors fail-closed, retain optimizer diagnostics, add focused regression tests and a changelog entry, then close this PR after the canonical replacement lands or maintainers explicitly choose a different path.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction exists: make `resizeToJpeg` throw for a non-JPEG input or a byte-small oversized-pixel image, and this PR returns the original buffer; the current caller then treats that result as JPEG and only applies the byte cap.

## Solution Assessment

No. Returning original bytes from `optimizeImageToJpeg` is not the best fix because that helper no longer has enough context to preserve the original MIME type or distinguish optimizer backend failures from safety guard failures; the safer approach is the loader-bound policy described by PR #74243.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve image input guard failures:** `src/media/web-media.ts:657-662`
  - body: The fallback returns `source` after every `resizeToJpeg` attempt throws. `resizeToJpeg` is where the pixel-limit and unknown-dimension guards run, so a byte-small oversized or invalid image can be accepted if it is under `maxBytes`. Re-throw those guard errors before falling back.
  - confidence: 0.9
- **[P2] Preserve original MIME on fallback:** `src/media/web-media.ts:657-662`
  - body: For non-HEIC PNG/WebP inputs, `source` is still the original bytes, but callers wrap any `optimizeImageToJpeg` result as JPEG and emit `image/jpeg`. Move fallback to the loader boundary or carry the original content type so consumers do not receive PNG/WebP bytes labeled as JPEG.
  - confidence: 0.92
