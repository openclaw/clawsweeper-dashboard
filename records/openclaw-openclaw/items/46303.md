## Summary

Keep open: current main still lacks the restart-time debounce/followup drain behavior, and the related persistence PR is still open rather than a merged canonical replacement. This branch is useful but not merge-ready because the new followup-drain waiter can report drained while overflow summary work remains pending, and the broad SDK/channel restart path still needs compatibility-aware review and stronger real restart proof.

## What This Changes

Adds a global inbound debouncer flush registry, followup and channel run-queue drain waiters, gateway-drain internal context, SIGUSR1 restart integration, and channel/test updates to preserve buffered inbound work before reload.

## Best Possible Solution

Land a narrow restart-drain fix only after the followup waiter mirrors the queue drain predicate, plugin SDK compatibility is explicit, and redacted real gateway/channel SIGUSR1 proof shows the buffered message survives reload.

## Reproduction Assessment

Yes from source inspection, though I did not run a live channel repro. Current main keeps inbound debounce buffers in process-local closures and the SIGUSR1 path drains active tasks only, so a process reload can drop timeout-buffered messages before their callbacks fire.

## Solution Assessment

No, not yet. The branch is a plausible owner-boundary fix, but it is not the best merge-ready shape until the followup waiter counts the same pending states as the drain loop and the SDK/channel compatibility plus live restart proof gaps are closed.
