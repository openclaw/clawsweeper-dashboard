Full review comments:

- **[P1] Surface incomplete debounce flushes before restart continues:** `src/auto-reply/inbound-debounce.ts:89`
  - body: `handle.flushAll()` and the timeout race can return `{ drained: false }`, but this function throws that state away and returns only `result.flushedCount`. If the deadline is hit with buffers still present, the restart path can proceed toward `server.close()` with no signal that inbound work remains, recreating the message-loss failure this PR is trying to prevent.
  - confidence: 0.9
- **[P2] Count overflow summaries as pending followup work:** `src/auto-reply/reply/queue/drain-all.ts:21`
  - body: The production drain loop keeps running while `queue.droppedCount > 0`, because summarize-on-overflow still has user-visible work to emit. This waiter ignores `droppedCount`, so a queue with no `items` but pending overflow summary state can report `drained: true` and let restart continue too early.
  - confidence: 0.88
- **[P2] Preserve the plugin-facing debouncer contract:** `src/auto-reply/inbound-debounce.ts:478`
  - body: `createInboundDebouncer` is re-exported through the plugin SDK, and returning required `flushKeyWithCount`, `flushAll`, and `unregister` methods changes the inferred public return shape. The PR already had to update bundled typed mocks, so third-party runtime stubs can see the same break unless this is made compatible or explicitly documented and tested as an SDK change.
  - confidence: 0.74
