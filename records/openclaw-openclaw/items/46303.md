## Summary

Keep this PR open, but do not merge it yet: the latest head still has a correctness bug in the new followup drain waiter, and the supplied proof does not show the production config.patch/SIGUSR1 channel path.

## What This Changes

The PR adds restart-time flushing/draining for inbound debouncers, channel run queues, and followup queues, plus related tests and a small E2E assertion cleanup.

## Best Possible Solution

Land a focused restart-drain fix that preserves the early gateway-draining gate, flushes accepted debounced work under an internal drain context, waits with the same pending-work predicate as the real drain loops, and includes redacted production-path proof.

## Reproduction Assessment

Yes for the PR-introduced issue by source inspection: a followup queue with only droppedCount pending will be reported as drained by the new waiter. The original message-loss report is credible from source, but I did not establish a live Signal/config.patch/SIGUSR1 reproduction in this read-only review.

## Solution Assessment

No, not yet. The implementation direction is maintainable, but waitForFollowupQueueDrain needs to include droppedCount and the PR needs production-path proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

Full review comments:

- **[P2] Count dropped summaries while waiting for queue drain:** `src/auto-reply/reply/queue/drain-all.ts:21`
  - body: `waitForFollowupQueueDrain` can return `drained: true` when a queue has no items and is not currently draining, even though `droppedCount` still represents pending overflow-summary work.
