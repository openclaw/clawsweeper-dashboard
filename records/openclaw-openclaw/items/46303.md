## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Count overflow summaries as pending follow-up work:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `waitForFollowupQueueDrain()` returns drained when every queue has `items.length === 0` and is not actively draining, but the real drain loop continues while `queue.droppedCount > 0`. A restart can therefore close while summarized overflow follow-up work is still pending; include `droppedCount` or reuse the drain loop's pending predicate.
  - confidence: 0.91
- **[P2] Contain flush work after the timeout wins:** `src/auto-reply/inbound-debounce.ts:66-75`
  - body: The timeout branch in this `Promise.race` only returns `{drained:false}` to the caller; it does not cancel or quarantine the still-running `handle.flushAll(...)`. If that flush later reaches `onFlush`, it can enqueue follow-up or channel work after the restart path has already waited on queues, recreating the shutdown message-loss race.
  - confidence: 0.82
- **[P1] Keep the drain bypass out of the public SDK contract:** `src/plugin-sdk/channel-lifecycle.core.ts:33-36`
  - body: `ChannelRunQueue` is exported through public plugin SDK subpaths, so adding `enqueueInternal` exposes an internal gateway-drain bypass to plugin callers without docs, API baseline discussion, or compatibility tests. Keep this bypass in a core-owned internal path, or make the new plugin contract explicit and owner-approved before merge.
  - confidence: 0.78
- **[P2] Update the mock queue shape:** `src/auto-reply/reply/queue/drain-all.test.ts:7-20`
  - body: This helper returns a `FollowupQueueState`, but the object is missing the current required `summaryElisions` and `evictedSummaryCount` fields from `state.ts`. The added test file will fail type checking until the mock state matches the real queue shape.
  - confidence: 0.88
