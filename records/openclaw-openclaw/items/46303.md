## Summary

Keep this PR open: current main still has a source-reproducible restart-time message-loss bug, and the contributor has now provided terminal proof for the helper behavior, but the patch remains not merge-ready because it leaves restart ingress open, misses channel-local run queues, and changes a public SDK return contract. https://github.com/openclaw/openclaw/pull/46303

## What This Changes

The branch adds global inbound debouncer flushing, followup-queue drain waiting, SIGUSR1 restart integration, and bundled channel teardown/test updates for registered debouncers.

## Best Possible Solution

Revise the restart lifecycle so external ingress is gated immediately, trusted internally flushed work can still drain through a compatible seam, channel-local run queues are covered, and SDK-facing debounce APIs remain backward-compatible.

## Reproduction Assessment

Yes. Current main keeps inbound debounce state in closure-local timers and the restart path closes the server without a global debounce flush or all-followup drain, so the reported loss path is source-reproducible.

## Solution Assessment

No. The PR targets the right failure mode, but the current patch is not the best solution because it delays external ingress gating, waits only on followup queues, and changes a public SDK method contract instead of adding an internal drain/count seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate external ingress before restart drain awaits:** `src/cli/gateway-cli/run-loop.ts:490`
  - body: `markGatewayDraining()` still runs only after debounce flushing, active-work waiting, and followup draining. Until then, remote channel traffic can enqueue new command work during restart, prolonging the drain and escaping the active-task snapshot; gate external ingress at restart start and provide an explicit internal path for trusted flushed/followup work.
  - confidence: 0.87
- **[P2] Drain channel run queues before closing the gateway:** `src/cli/gateway-cli/run-loop.ts:477`
  - body: The restart path waits for `FOLLOWUP_QUEUES`, but Discord debounce flushing enqueues into `messageRunQueue`, a `createChannelRunQueue()` wrapper that this path never waits on. Those flushed jobs can still be queued or starting when channel teardown deactivates the run queue, so debounced Discord messages can still be skipped.
  - confidence: 0.82
- **[P2] Preserve the public flushKey return contract:** `src/auto-reply/inbound-debounce.ts:222-227`
  - body: `createInboundDebouncer` is exported through public plugin SDK subpaths, but this patch changes `flushKey` from the existing `Promise<void>` behavior to `Promise<boolean>`. Keep the public method compatible and expose delivery/count state through an additive internal helper or a new method.
  - confidence: 0.78
