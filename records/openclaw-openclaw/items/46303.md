Full review comments:

- **[P1] Do not proceed after incomplete debounce flushes:** `src/auto-reply/inbound-debounce.ts:64-74`
  - body: `flushAllInboundDebouncers()` races each `handle.flushAll()` against a timeout and then returns only `flushedCount`, discarding whether that handle actually drained. If the timeout wins, the original flush can still enqueue work later, or `flushAllInternal` can return `drained:false`; either way `run-loop.ts` may run the channel/followup waits while no work is visible yet and then shut down with messages still buffered.
  - confidence: 0.89
- **[P1] Count overflow summaries while waiting for followup drain:** `src/auto-reply/reply/queue/drain-all.ts:19-23`
  - body: The production drain loop keeps running while `queue.droppedCount > 0`, but this waiter only counts queued items plus an active-drain sentinel. A queue with only overflow-summary work can be reported as drained, so restart can tear down before the summary followup is emitted.
  - confidence: 0.92
