## Summary

Keep open: the reported loss mode is real on current main, but latest PR head still has restart-drain correctness and availability blockers, plus no after-fix real behavior proof from a live setup.

## What This Changes

The branch adds restart-time inbound debouncer flushing, followup-queue draining, and bundled channel debounce teardown/test updates around SIGUSR1 reload.

## Best Possible Solution

Revise the restart lifecycle so external ingress is gated immediately, trusted internally flushed work can still drain through an explicit compatible path, channel-local queues are covered, and SDK-facing debounce APIs stay backward-compatible.

## Reproduction Assessment

Yes, by source inspection. Current main keeps debounce buffers in closure-local timer state and the SIGUSR1 restart path closes the server without a global debounce flush or all-followup drain.

## Solution Assessment

No. The PR targets the right failure mode, but the current design still accepts external work during awaited restart drains, misses at least one channel-local queue, and changes a public SDK return type instead of adding a compatible drain seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate external ingress before restart drain awaits:** `src/cli/gateway-cli/run-loop.ts:490`
  - body: At PR head, `markGatewayDraining()` still runs only after debounce flushing, active-work draining, and followup draining. Since `enqueueCommandInLane()` accepts work until that flag is set, remote channel traffic can create new agent work during restart and prolong or recreate the loss window; gate external ingress immediately and use a trusted internal path for flushed/followup work.
  - confidence: 0.87
- **[P2] Drain channel run queues before closing the gateway:** `src/cli/gateway-cli/run-loop.ts:477`
  - body: The restart path waits for followup queues, but Discord debounce flushing only enqueues into `messageRunQueue`, which is a `createChannelRunQueue()` wrapper with no drain awaited here. Those flushed jobs can still be queued when channel teardown deactivates the run queue, so messages can be skipped despite the debounce flush completing.
  - confidence: 0.82
- **[P2] Preserve the public flushKey return contract:** `src/auto-reply/inbound-debounce.ts:222-227`
  - body: `createInboundDebouncer` is re-exported through the public `openclaw/plugin-sdk/channel-inbound` subpath, but this patch changes `flushKey` from the prior `Promise<void>` behavior to `Promise<boolean>`. Keep the existing return contract for plugin consumers and expose delivery/count state through an additive internal or new method.
  - confidence: 0.78
