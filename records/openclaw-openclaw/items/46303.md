## Best Possible Solution

Land a bounded restart-drain fix only after the drain predicates match production queues, incomplete flushes are surfaced, SDK compatibility is explicit and tested, and current-head live delivery proof is attached; keep durable replay tracked in https://github.com/openclaw/openclaw/issues/51620.

## Reproduction Assessment

Yes, at source level: current main stores debounce buffers in a closure-local Map and followup queues in an in-memory global Map, while the SIGUSR1 shutdown path does not flush either before process replacement. I did not run a live Signal or Telegram restart reproduction in this read-only review.

## Solution Assessment

No. The branch is a plausible partial mitigation, but it needs drain-completeness fixes, a droppedCount-aligned followup waiter, and an explicit SDK compatibility path before it is the narrowest maintainable solution.

Full review comments:

- **[P1] Propagate incomplete debounce flushes before restarting:** `src/auto-reply/inbound-debounce.ts:64-74`
  - body: The timeout race can resolve `{ flushedCount: 0, drained: false }`, but this function discards `drained` and returns only a count while the underlying flush promise may still be running or buffers may remain. That lets the restart path continue after a slow or hung `onFlush`, recreating the message-loss case this drain is meant to prevent.
  - confidence: 0.86
- **[P1] Preserve the public debouncer contract:** `src/auto-reply/inbound-debounce.ts:478`
  - body: `createInboundDebouncer` is re-exported from the plugin SDK, so adding new required return methods changes `ReturnType` and plugin runtime mock contracts for external plugins. Keep the public return shape compatible, or add an explicit compat/deprecation path with SDK contract coverage before making `flushAll`/`unregister` required.
  - confidence: 0.78
- **[P2] Count dropped summaries while waiting for followup drain:** `src/auto-reply/reply/queue/drain-all.ts:21`
  - body: `scheduleFollowupDrain` keeps draining while either `items.length` or `droppedCount` is nonzero, but this waiter ignores `droppedCount`. A restart can report the followup queues drained while an overflow summary is still pending, then close the gateway before that queued message summary is delivered.
  - confidence: 0.9
