## Summary

Keep open. The PR addresses a real message-loss gap that is still present on current main, but the latest branch is not merge-ready because the restart drain design still delays the external enqueue gate, undercounts summary-only followup work, can evict live debouncers during restart, and changes a public SDK return contract.

## What This Changes

The PR adds a process-global inbound debouncer registry and flush path, a followup queue drain waiter, SIGUSR1 restart-loop integration, and bundled channel teardown/test updates to reduce message loss during gateway reloads.

## Best Possible Solution

Revise the branch around a backward-compatible restart-drain design: stop or gate external ingress at restart start while allowing trusted internally flushed work, mirror the existing followup drain predicate including dropped summaries, keep live debouncers discoverable until their transports are closed, and expose new drain metadata through additive SDK APIs instead of changing flushKey's public contract.

## Reproduction Assessment

Yes. The original user problem is reproducible by source inspection on current main: debounced messages live in closure-local timer buffers, while SIGUSR1 restart only drains active command work before server close. The PR-specific blockers are also high-confidence static reproductions from the latest head and current queue semantics.

## Solution Assessment

No. The PR is aimed at the right failure mode, but the current implementation is not the best fix because it leaves an external enqueue window during restart, does not count summary-only followup work, can unregister live idle debouncers during the restart window, and changes a public SDK return type instead of adding a compatible drain API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count summary-only followup work:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps a queue alive while `queue.items.length > 0 || queue.droppedCount > 0`, but this waiter only counts queued items plus the active drain sentinel. A summarize-overflow queue with only `droppedCount` pending can report drained and let restart close before the summary followup is sent.
  - confidence: 0.91
- **[P2] Gate external command enqueues at restart start:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits through debounce flushing, active-task drain, and followup drain before setting `gatewayDraining`. During that async window external chat traffic can start new command work, and `waitForActiveTasks()` only waits for tasks active at its snapshot, so restarts can be prolonged and late work can still be lost.
  - confidence: 0.84
- **[P2] Keep live debouncers registered until ingress is closed:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This stale-entry branch unregisters debouncers during restart flush based only on age, while channel ingress can still deliver events. A live but quiet channel can then buffer a late message on a handle removed from the global registry, so no later global sweep can flush it before teardown.
  - confidence: 0.86
- **[P2] Preserve the public flushKey return contract:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through public plugin SDK subpaths, but this changes `flushKey` from `Promise<void>` to `Promise<boolean>`. Existing plugins or helpers typed to the old contract can fail to type-check; keep `flushKey` compatible and expose delivery/count state through additive APIs.
  - confidence: 0.78
