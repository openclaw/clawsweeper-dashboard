## Summary

Keep open: current main does not include this restart-buffer drain work, but the PR still has a blocking correctness gap because `waitForFollowupQueueDrain()` ignores overflow-summary work in `droppedCount`; the supplied proof also does not exercise the real gateway SIGUSR1/config.patch channel path.

## What This Changes

The branch adds restart-time flushing for inbound debouncers, channel run queues, and followup queues with gateway-drain bypass plumbing and tests, plus incidental formatting/script churn.

## Best Possible Solution

Keep the restart-drain design focused, make the waiter mirror the production pending predicate including `droppedCount`, add a summary-only overflow regression test, trim unrelated churn where practical, and require live gateway restart proof before merge.

## Reproduction Assessment

Yes, by source inspection. A queue with `droppedCount > 0`, no `items`, and not currently `draining` is still pending in `scheduleFollowupDrain()`, while the PR waiter returns `drained: true`.

## Solution Assessment

No. The current approach is close, but `waitForFollowupQueueDrain()` must use the same pending-work predicate as the real drain loop before it can safely gate restart teardown.
