## Summary

Keep open: the live PR head still has a correctness bug in the new followup drain waiter because overflow summary work in droppedCount can be reported as fully drained.

## What This Changes

The PR adds restart-time inbound debouncer flushing, channel/followup queue drain waiters, gateway-drain bypass plumbing, and related tests across auto-reply, gateway, Discord, plugin SDK, and command queue code.

## Best Possible Solution

Land a focused restart-drain fix that marks draining early, drains accepted debounced and queued work with the same pending-work predicate as the real drain loop, and includes redacted production-path proof.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: a followup queue with droppedCount > 0, no items, and no draining sentinel is reported drained by the new waiter. I did not run a live Signal/config.patch/SIGUSR1 reproduction in this read-only review.

## Solution Assessment

No, not yet. The implementation direction is maintainable and the earlier external-enqueue restart-window concern is mitigated, but waitForFollowupQueueDrain must include droppedCount and the PR still needs production-path proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Count dropped summaries while waiting for drain:** `src/auto-reply/reply/queue/drain-all.ts:21`
