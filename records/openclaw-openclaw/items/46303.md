Yes, source-level: current main keeps debounce buffers and followup queues in process memory and the restart path has no flush/drain waiters for them. I did not run a live config.patch/SIGUSR1 transport reproduction in this read-only review.

No, not as currently written: the in-memory drain approach is a useful partial repair, but it must not proceed after incomplete debouncer flushes or treat droppedCount summaries as drained. The broader durable replay design remains separately tracked.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not proceed after incomplete debounce flushes:** `src/auto-reply/inbound-debounce.ts:64-74`
  - body: This races `handle.flushAll()` against a timeout, but the caller only reduces `flushedCount` and ignores whether the handle reported `drained: false`. When the timeout wins, the underlying flush can still be running or buffers can remain, yet `run-loop.ts` proceeds to queue waits and server close, preserving the restart-window message-loss race this PR is trying to fix. Return an aggregate drained state to the restart path and fail/abort/retry instead of treating this as a completed flush.
  - confidence: 0.89
- **[P1] Count overflow summaries while waiting for followup drain:** `src/auto-reply/reply/queue/drain-all.ts:19-22`
  - body: `waitForFollowupQueueDrain()` only counts `items.length` and `draining`, but the production drain loop continues while `queue.droppedCount > 0` because those dropped messages are emitted as an overflow summary. A queue with only summary work left can be reported as drained, allowing restart shutdown before the user-visible summary is delivered. Include `droppedCount` in the pending count and cover that case in the drain-all test.
  - confidence: 0.86
