## Summary

Keep open. Current main still does not implement the requested debounce/followup drain on SIGUSR1, and the PR remains useful, but the latest head is not merge-ready because it introduces restart-lifecycle and public SDK contract blockers plus an availability-sensitive enqueue window.

## What This Changes

The PR adds a process-global inbound debouncer registry and flush path, a followup queue drain waiter, SIGUSR1 restart-loop integration, and bundled channel teardown/test updates to reduce message loss during gateway reloads.

## Best Possible Solution

Revise the branch around a backward-compatible restart-drain design: stop or gate external ingress at restart start while allowing trusted internally flushed work, mirror the existing followup drain predicate including dropped summaries, keep live debouncers discoverable until transports are closed, and expose any new drain metadata through additive SDK APIs.

## Reproduction Assessment

Yes. The original user problem is reproducible by source inspection on current main: debounced messages live in closure-local timer buffers, while SIGUSR1 restart does not flush them or wait for followup queues. The PR-specific blockers are also high-confidence static reproductions from the latest PR head and current queue/command-lane semantics.

## Solution Assessment

No. The PR targets the right failure mode, but this is not the best current solution because it leaves an external enqueue window during restart, undercounts summary-only followup work, can unregister live idle debouncers during restart, and changes a public SDK return type instead of adding compatible drain APIs.

- **[P2] Count dropped-summary followup work:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps draining while `queue.items.length > 0 || queue.droppedCount > 0`, but this waiter only counts queued items plus the active-drain sentinel. A summarize-overflow queue with only `droppedCount` pending can report drained and let restart close before the summary followup is sent.
  - confidence: 0.91
- **[P2] Gate external command enqueues when restart begins:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits through debounce flushing, active-task drain, and followup drain before setting `gatewayDraining`. During that async window, external chat traffic can start new command work, and the active-task wait only covers work visible at its snapshot, so restarts can be prolonged and late work can still be lost.
  - confidence: 0.84
- **[P2] Keep stale live debouncers registered until teardown:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This stale-entry branch unregisters debouncers during restart flush based only on age while channel ingress can still deliver events. A live but quiet channel can then buffer a late message on a handle removed from the global registry, leaving no later global sweep to flush it before teardown.
  - confidence: 0.86
- **[P2] Preserve the public flushKey return contract:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through public plugin SDK subpaths, but this changes `flushKey` from `Promise<void>` to `Promise<boolean>`. Existing plugins or helpers typed to the old contract can fail to type-check; keep `flushKey` compatible and expose delivery/count state through additive APIs.
  - confidence: 0.78
