## Best Possible Solution

Keep the in-process drain idea only after the restart path treats timed-out debounce flushes as not drained, mirrors the production followup pending predicate, and is backed by current-head live transport proof; leave durable replay and provider ACK policy to the canonical restart-message-loss tracker.

## Solution Assessment

No: the direction is appropriate, but current head needs non-detached flush timeout handling, followup pending-count parity with the real drain loop, and current-head live transport proof before this is the best mergeable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not detach timed-out debounce flushes:** `src/auto-reply/inbound-debounce.ts:64-74`
  - body: `Promise.race` lets `flushAllInboundDebouncers` return after the timeout while the underlying `handle.flushAll()` continues running. The restart path can then wait on empty channel/followup queues and proceed to server close before that late flush enqueues work, reintroducing the message-loss race this PR is meant to fix. Surface `drained:false` to the caller or keep the restart drain waiting/cancelling deterministically instead of allowing detached flush work.
  - confidence: 0.91
- **[P1] Count overflow summaries as pending followup work:** `src/auto-reply/reply/queue/drain-all.ts:21`
  - body: `waitForFollowupQueueDrain` only counts queued items and an active drain sentinel, but the production drain loop keeps running while `droppedCount > 0` to emit overflow summaries. A queue with summary-only work can be reported as drained here, letting restart teardown proceed before that followup is delivered. Include `queue.droppedCount` in the pending count and cover the summary-only case.
  - confidence: 0.9
