## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Count dropped summaries before declaring drain complete:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain()` keeps draining while either `queue.items.length > 0` or `queue.droppedCount > 0`, because dropped overflow entries still need a summary followup. This helper only counts `items` plus the active-drain sentinel, so a queue with `droppedCount > 0` and no active drain is reported as fully drained and the restart can continue before those summarized messages are processed.
  - confidence: 0.92
- **[P2] Budget the new restart waits in the watchdog:** `src/cli/gateway-cli/run-loop.ts:550-554`
  - body: The force-exit timer is still armed as `restartDrainTimeoutMs + SHUTDOWN_TIMEOUT_MS` before the newly added debounce flush and channel/followup drain waits run. Those added waits can consume up to their timeout budget before `waitForActiveTasks(restartDrainTimeoutMs)` starts, so a capped restart can hard-exit while otherwise drainable active work is still waiting.
  - confidence: 0.87
- **[P2] Initialize full followup queue state in the new test helper:** `src/auto-reply/reply/queue/drain-all.test.ts:7-20`
  - body: `FollowupQueueState` now requires `summaryElisions` and `evictedSummaryCount`, but this new test helper returns an object without those fields. That leaves the added drain-all suite out of sync with the real queue state shape and can fail strict test type checks before the assertions run.
  - confidence: 0.84
