## Summary

The PR changes SIGUSR1 gateway restart handling to flush inbound debouncers, channel run queues, and followup queues before shutdown so buffered inbound messages are less likely to be lost.

## Best Possible Solution

Keep the restart-drain approach only if the new drain waiters mirror the real queue invariants and maintainers explicitly accept the plugin-facing queue contract changes.

## Reproduction Assessment

Yes for the source-level remaining bug: a queue with droppedCount > 0 and no items is pending according to the real drain loop but drained according to the new waiter. No live channel restart reproduction was run in this read-only review.

## Solution Assessment

No, not yet: the restart-drain direction is plausible, but the new waiter must match the existing drain loop's pending-work predicate and the plugin SDK contract change needs maintainer signoff.
