## Summary

Keep open: current main still has the restart-time message-loss bug, but this PR is not merge-ready because the latest branch still leaves external ingress open during awaited restart drains, undercounts summary-only followup work, and changes a plugin-facing debounce contract.

## What This Changes

The branch adds a process-wide inbound debouncer flush registry, an all-followup queue drain waiter, SIGUSR1 restart integration, and bundled channel teardown/test updates.

## Best Possible Solution

Revise the branch around a restart-drain design that gates external ingress immediately, allows trusted internally flushed work to complete, mirrors followup queue pending semantics, and preserves plugin SDK compatibility with additive lifecycle APIs.

## Reproduction Assessment

Yes by source inspection, though not by executing a live Signal/config reload scenario. Current main keeps debounced items in closure-local timer buffers and the SIGUSR1 path does not flush those buffers or wait for all followup queue work before teardown.

## Solution Assessment

No. The PR targets the right failure mode, but the current patch delays the external enqueue gate, misses `droppedCount` followup work, and changes a public debounce return contract instead of using a compatible lifecycle seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate external command enqueues before restart drains:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits for debounce flushing, active/runtime-deps drains, and followup drains before `markGatewayDraining()` runs. During those awaits, remote channel traffic can still enqueue new LLM/tool work through `enqueueCommandInLane`, prolonging restart and recreating a late-work loss window; gate external ingress immediately or add an internal-only bypass for trusted flushed work.
  - confidence: 0.86
- **[P2] Count dropped summaries as pending followup work:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps draining while `queue.items.length > 0 || queue.droppedCount > 0`, but this waiter only counts queued items plus the active-drain sentinel. A summary-only overflow queue can therefore report drained before the summary followup is sent.
  - confidence: 0.91
- **[P2] Preserve the public flushKey contract:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through the plugin SDK, but this patch changes `flushKey` from the existing `Promise<void>` behavior to a boolean result. That can break third-party plugin typings; keep `flushKey` compatible and expose delivery/count state through an additive API.
  - confidence: 0.8
- **[P2] Keep live debouncers registered during restart flush:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: The stale-entry branch can unregister a live but quiet debouncer while the server is still accepting ingress. A message arriving after that eviction can buffer on a handle no later global sweep can see before teardown, so defer stale cleanup until ingress is closed or prove the handle is no longer live.
  - confidence: 0.84
