Full review comments:

- **[P1] Count overflow summaries as pending follow-up work:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `waitForFollowupQueueDrain()` reports a queue drained when `items.length === 0` and it is not actively draining, but the real drain loop continues while `droppedCount > 0` to deliver overflow summaries. A restart can therefore proceed before summarized follow-up work is processed; include the same `droppedCount`/summary predicate used by the drain loop and add a regression case.
  - confidence: 0.88
- **[P2] Contain timed-out debounce flushes before proceeding:** `src/auto-reply/inbound-debounce.ts:65-75`
  - body: The timeout branch races `handle.flushAll()` and then returns while the original flush promise can still be running. If that late flush enqueues follow-up or channel work after the restart drain waits have completed, the PR can still drop the message it is trying to preserve; the restart path must either await/contain the in-flight flush or avoid starting work it will not wait for.
  - confidence: 0.82
- **[P2] Make global debouncer registration lifecycle-owned:** `src/auto-reply/inbound-debounce.ts:478`
  - body: Registering every debouncer globally makes teardown coverage mandatory, but existing call sites such as Telegram, Signal, and iMessage still create inbound debouncers without an inspected unregister path. That can retain stale closures across in-process restarts/reconnects and make restart sweeps operate on old handlers; either make registration opt-in/lifecycle-owned or update all long-lived debouncer owners with tested unregister coverage.
  - confidence: 0.78
