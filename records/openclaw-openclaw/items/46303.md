## Summary

Keep open: current main still lacks the central restart-time debounce/followup drain behavior, but this PR has a concrete correctness gap in the new followup-drain waiter and the supplied proof does not yet show the production config.patch/SIGUSR1 path.

## What This Changes

The PR adds restart-time inbound debouncer flushing, followup and channel queue drain waits, and internal gateway-drain enqueue bypasses across the command, followup, Discord, and SDK channel queues.

## Best Possible Solution

Land a focused restart-drain fix that preserves the early gateway-draining gate, flushes debounced work under an internal drain context, waits channel and followup queues using the same pending-work predicate as the real drain loops, and includes redacted production-path proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection shows the current main debounce buffers are closure-local timers and the SIGUSR1 restart path has no global debounce or followup-queue flush before server close.

## Solution Assessment

No, not yet. The implementation direction is maintainable, but waitForFollowupQueueDrain should mirror the production drain loop's droppedCount-aware predicate and the PR still needs production-path proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Count dropped summaries while waiting for queue drain:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: waitForFollowupQueueDrain can report drained when a queue has no items and is not currently draining, even though droppedCount still represents overflow-summary work. The production drain loop continues while queue.items.length > 0 || queue.droppedCount > 0, so the restart path can proceed to server close before that summary followup is processed. Include queue.droppedCount in the pending count and cover the summary-only case.
  - confidence: 0.84
