## Summary

Keep this PR open: it targets a real source-reproducible restart message-loss gap that current main does not fully cover, but the current patch is not merge-ready because it still has restart-drain correctness issues and exposes an internal drain bypass through a public SDK surface.

## What This Changes

This PR adds global inbound debouncer flushing, follow-up/channel run queue drain waiters, and gateway-drain bypass plumbing so SIGUSR1 reloads can drain pending inbound and follow-up work before restart.

## Best Possible Solution

Land a narrowed restart-drain fix that keeps drain bypasses internal, waits for all follow-up queue state including overflow summaries after active turns, keeps debouncer handles registered until safe teardown, and refreshes proof against the rebased head.

## Reproduction Assessment

Yes. Current main still has closure-local inbound debounce buffers and no global debounce/follow-up drain path before restart close, so the reported SIGUSR1 loss path is source-reproducible even though I did not run a live Signal roundtrip.

## Solution Assessment

No. The direction is useful, but the current patch still misses summary-only follow-up work, can unregister live or undrained debouncers, and exposes an internal drain bypass through the public SDK.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count summary-only followups before reporting drained:** `src/auto-reply/reply/queue/drain-all.ts:21`
  - body: `scheduleFollowupDrain` keeps running while `queue.droppedCount > 0`, because overflow summaries are still pending work. This waiter ignores `droppedCount`, so a restart can see `{drained: true}` when a queue has no `items` but still has a summary follow-up to run, then close the server and lose that summary. Mirror the drain loop predicate by including `queue.droppedCount` in the pending count.
  - confidence: 0.92
- **[P2] Keep live debouncers registered during restart drain:** `src/auto-reply/inbound-debounce.ts:99-100`
  - body: This age-based unregister runs before `server.close()` and does not check `result.drained`. A live but quiet channel, or a debouncer whose flush timed out with buffered work remaining, can be removed from the global registry and any later buffered message will no longer be discoverable by the restart sweep. Defer cleanup to channel teardown or only evict handles that are proven inactive and fully drained.
  - confidence: 0.88
- **[P2] Keep the drain bypass out of the public SDK:** `src/plugin-sdk/channel-lifecycle.core.ts:22-26`
  - body: `channel-lifecycle` is exported as a public plugin SDK subpath, so adding `enqueueInternal` exposes the gateway-drain bypass to plugin callers. That lets a plugin opt into `runWithGatewayDrainInternalContext` and enqueue work after the restart gate is meant to reject external ingress. Keep this bypass private to core-owned restart flushing or use an unexported capability.
  - confidence: 0.86
