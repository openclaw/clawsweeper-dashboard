## Review Findings

Full review comments:

- **[P1] Count dropped summaries before reporting queues drained:** `src/auto-reply/reply/queue/drain-all.ts:21`
  - body: `waitForFollowupQueueDrain()` only counts `items.length` and `draining`, but the real drain loop keeps running while `droppedCount > 0`. A queue with only overflow-summary work can return `drained: true` during restart, so server close may cut off the summary/followup. Mirror the production pending predicate or include `droppedCount`, and add a regression test for a droppedCount-only queue.
  - confidence: 0.92
- **[P2] Keep live idle debouncers registered until ingress closes:** `src/auto-reply/inbound-debounce.ts:99-100`
  - body: During restart flush, the server is still accepting channel ingress, but debouncers idle for five minutes are unregistered here. A normal live-but-idle channel can receive a message after this point and buffer it on a handle no longer visible to the global flush, recreating the message-loss race this PR is trying to fix. Remove this eviction from restart flush or move stale cleanup to a point where the owning ingress is already closed.
  - confidence: 0.86
