## Summary

Keep open. The PR still has a concrete message-loss bug in the restart drain waiter, stale real-behavior proof for the current head, a conflicting/dirty merge state with failing checks, and compatibility-sensitive changes to restart admission and plugin-facing queue contracts.

## What This Changes

The PR adds SIGUSR1 restart-time flushing for registered inbound debouncers plus channel-run and follow-up queue drain waits, with gateway-drain bypass plumbing across core and bundled channel paths.

## Best Possible Solution

Land only a rebased version whose restart waiter mirrors the real follow-up drain predicate, whose plugin-facing bypass semantics are maintainer-approved, and whose current head has redacted live or terminal proof for the full restart-drain path.

## Reproduction Assessment

Yes, by source. Current main keeps debounce and follow-up queue state in memory, and the PR-head bug is clear because the new waiter ignores droppedCount while the real drain loop treats droppedCount as pending work.

## Solution Assessment

No, not as currently proposed. The in-memory drain direction is a plausible mitigation, but the best fix must mirror the actual queue drain predicate, preserve plugin compatibility, and prove current-head restart behavior before merge.
