## Summary

Keep open. The branch addresses a real source-proven restart message-loss path and the author has followed up on several earlier review issues, but latest head still has blocking restart-lifecycle, followup-drain, registry, and SDK-compatibility problems.

## What This Changes

The PR adds a global inbound debouncer registry/flush API, a followup-queue drain waiter, SIGUSR1 run-loop integration, and bundled channel teardown/test updates.

## Best Possible Solution

Revise the branch around an explicit restart-drain contract: gate external ingress early, preserve internal followup execution, mirror followup queue pending semantics, and keep plugin SDK debounce APIs backward-compatible.

## Reproduction Assessment

Yes. Source inspection on current main shows debounced inbound messages can live only in closure-local timer buffers, and the SIGUSR1 path does not flush those buffers or all followup queues before server teardown.

## Solution Assessment

No. The PR targets the right loss mode, but latest head is not the best mergeable solution because it accepts external work during restart draining, undercounts summary-only followup work, can evict live debouncers, and changes an SDK-exposed return contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate external enqueues before restart drain awaits:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The restart path now waits for debounce flushing, active work, and followup draining before `markGatewayDraining()` runs. During that window, remote channel traffic can still enqueue new LLM/tool work, prolonging restart and creating more late work that may be lost; add an external-ingress gate or internal-only drain bypass so trusted flushed work can run without accepting new external commands.
  - confidence: 0.84
- **[P2] Include dropped summaries in followup drain waiting:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps draining while `queue.items.length > 0 || queue.droppedCount > 0`, but this waiter only counts queued items plus an active-drain sentinel. A summarize-on-overflow queue with only `droppedCount` pending can report drained before its summary followup is sent.
  - confidence: 0.91
- **[P2] Keep live idle debouncers registered during restart flush:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: The stale-entry branch can unregister a live but quiet debouncer during the restart flush while channel ingress is still accepting events. A late message for that channel can then buffer on a handle no later global sweep can see before teardown.
  - confidence: 0.86
- **[P2] Preserve the public flushKey return contract:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through plugin SDK subpaths, but the PR changes `flushKey` from the old void-like promise contract to `Promise<boolean>`. Keep existing plugin-facing calls compatible and expose delivery/count state through an additive API.
  - confidence: 0.78
