## Summary

Keep this PR open. The restart-drain direction is useful and current main does not have the proposed debounce/followup drain helpers, but the latest head still has blocking source-inspection issues in the new drain logic and its real behavior proof is stale relative to the current PR head.

## What This Changes

The PR adds restart-time inbound debouncer flushing, followup and channel queue drain waiters, gateway-drain internal enqueue bypasses, and tests across gateway, auto-reply, plugin SDK, Discord, and Feishu surfaces.

## Best Possible Solution

Land the restart-drain approach only after the wait helper mirrors the real followup drain predicate, active debouncers remain globally trackable until ingress is actually closed, and the contributor refreshes latest-head real restart proof.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: droppedCount summary work is ignored by the new waiter, and idle live debouncers can be unregistered before ingress is closed. The original live Signal/config.patch restart loss was not reproduced in this read-only sweep.

## Solution Assessment

No, not yet. The overall implementation direction is maintainable, but the drain predicates and debouncer registry lifetime need to match the existing runtime behavior before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count dropped summaries while waiting for drain:** `src/auto-reply/reply/queue/drain-all.ts:21`
  - body: `waitForFollowupQueueDrain` can return `drained: true` when a queue has no items and is not currently draining even though `droppedCount` still represents summary work. The real drain loop continues while `queue.items.length > 0 || queue.droppedCount > 0`, so include `droppedCount` in the pending count and add a summary-only case.
  - confidence: 0.9
- **[P2] Keep live debouncers registered until ingress closes:** `src/auto-reply/inbound-debounce.ts:99-100`
  - body: The stale-entry branch unregisters debouncers during restart flush even though the server may still accept inbound traffic. A quiet live channel can receive a message after this auto-eviction and buffer it on a handle that the global restart sweep can no longer see, recreating the restart-time message-loss race this PR is trying to fix.
  - confidence: 0.82
