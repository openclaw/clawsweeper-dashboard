## Summary

Keep this PR open: it targets a valid current-main message-loss bug, but the proposed restart drain design still has correctness, SDK-compatibility, and availability blockers that should be resolved before merge.

## What This Changes

The branch adds a global inbound debouncer flush registry, an all-followup queue drain waiter, SIGUSR1 restart integration, and bundled channel teardown/test updates.

## Best Possible Solution

Revise the restart-drain design so external ingress is gated immediately, trusted internally flushed work can still drain, followup pending semantics match the real queue loop, and SDK debounce contracts remain backward-compatible.

## Reproduction Assessment

Yes. Source inspection on current main shows debounce buffers are closure-local and the SIGUSR1 path does not flush them or wait for all followup queues before closing.

## Solution Assessment

No. The PR addresses the right failure mode, but the current implementation delays the external enqueue gate, misses dropped-summary followups, and changes a public plugin SDK return contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Gate external ingress before restart drain awaits:** `src/cli/gateway-cli/run-loop.ts:340-342`
  - body: The branch awaits debounce flushing, active-work drain, and followup drain before `markGatewayDraining()` runs at the end of the block. During those awaits, remote channel traffic can still enqueue new LLM/tool work and create late buffers after the global sweep; gate external ingress immediately or add an internal-only bypass for trusted flushed work.
  - confidence: 0.86
- **[P2] Count dropped summaries as pending followup work:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps draining while `droppedCount > 0`, but this waiter only counts queued items plus the active-drain sentinel. A summary-only overflow queue can therefore report drained before its summary followup is sent.
  - confidence: 0.91
- **[P2] Keep live idle debouncers registered during restart flush:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This stale-entry branch unregisters an idle debouncer while the server can still accept ingress. A late message for that live channel can then buffer on a handle that no later global sweep can see before teardown.
  - confidence: 0.85
- **[P2] Preserve the public flushKey return contract:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is exported through plugin SDK subpaths, but this changes `flushKey` from the previous `Promise<void>` behavior to `Promise<boolean>`. Keep existing plugin-facing calls compatible and expose delivery counts through an additive API.
  - confidence: 0.78
- **[P2] Do not let timed-out flushes continue behind restart:** `src/auto-reply/inbound-debounce.ts:60-62`
  - body: `Promise.race` returns on timeout, but it does not cancel `handle.flushAll()`. That background flush can still enqueue followup work after the restart path has already moved past its flush/followup drain decisions, recreating a late-work loss window.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces an availability-sensitive restart window where external inbound traffic can enqueue new command work before the gateway is marked draining.

Concerns:

- **[medium] External work accepted during restart drain:** `src/cli/gateway-cli/run-loop.ts:340`
  - body: Because `markGatewayDraining()` is delayed until after debounce flushing, active-work waiting, and followup draining, remote chat traffic can create additional command work during the restart drain window, increasing cost/DoS exposure and undermining restart guarantees.
