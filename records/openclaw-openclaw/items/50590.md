## Summary

Keep open. Current `main` still has the sandbox/skills mismatch: non-readwrite sandbox setup syncs skills into the sandbox workspace, but embedded and cron-style runs can keep using a prebuilt skills snapshot whose prompt takes precedence over sandbox-local entries.

## What This Changes

The issue asks OpenClaw to make managed skill `SKILL.md` paths usable from non-main sandboxed runs instead of advertising host-global `~/.openclaw/skills` paths that the sandbox cannot read.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable skill copies, or remap snapshot prompt locations after sandbox context resolution, while keeping host canonical metadata for config, env, and session state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to build a skills snapshot from the agent workspace, enter a non-readwrite sandbox run, and observe that the snapshot prompt is preferred over entries loaded from the sandbox workspace.

## Solution Assessment

No for a host allowlist or prompt-only workaround as the primary fix, because that either weakens the sandbox boundary or leaves inaccessible prompt paths possible. The maintainable fix is a runtime remap or post-sandbox prompt rebuild that keeps file reads sandbox-local.
