## Summary

Keep open. Current main still has the sandbox/skills mismatch: non-rw sandbox setup mirrors skills into the sandbox workspace, but embedded and cron-style runs can reuse a host-built skills snapshot whose prompt is returned before sandbox-loaded entries.

## What This Changes

The issue reports non-main sandbox sessions advertising `~/.openclaw/skills/<name>/SKILL.md` instead of the sandbox-local skill copy, causing sandboxed reads to fail.

## Best Possible Solution

Sandboxed non-rw runs should rebuild or remap `<available_skills>` after sandbox context resolution so prompt-facing `<location>` values point to sandbox-readable skill copies while preserving canonical host metadata for runtime state and env injection.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to build a skills snapshot from the host agent workspace, enter a non-rw sandbox run, and observe that the snapshot prompt is preferred over entries loaded from the sandbox workspace.

## Solution Assessment

No for a host allowlist or prompt-only workaround as the primary fix, because that sidesteps the sandbox contract. The narrow maintainable fix is to rebuild or remap the runtime skills prompt after sandbox workspace resolution.
