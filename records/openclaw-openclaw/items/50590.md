## Summary

Keep open. Current main still has the sandbox/skills mismatch: sandbox setup copies eligible skills into the sandbox workspace, but embedded and cron runs can continue using a host-built skills snapshot whose prompt wins over sandbox-loaded entries.

## What This Changes

The issue reports that sandboxed non-main runs advertise or resolve global `~/.openclaw/skills/<name>/SKILL.md` paths instead of sandbox-readable skill copies.

## Best Possible Solution

Sandboxed non-rw runs should render `<available_skills>` from sandbox-readable skill copies, or remap snapshot locations after sandbox context resolution, while preserving host-resolved skill metadata for env and session state.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: build a skills snapshot from the host agent workspace, enter a non-rw sandboxed embedded or cron run, and observe that the snapshot prompt is preferred over entries from the sandbox workspace.

## Solution Assessment

No for a host allowlist or #74161-style prompt-only workaround as the primary fix. The narrow maintainable solution is to rebuild or remap the skills prompt after sandbox context resolution so `<location>` values are sandbox-readable.
