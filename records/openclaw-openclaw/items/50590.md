## Summary

Keep open. Current `main` still has the sandbox/skills mismatch: non-rw sandbox setup syncs eligible skills into the sandbox workspace, but embedded and cron-style runs can still reuse a host-built skills snapshot whose prompt takes precedence over sandbox-loaded entries.

## What This Changes

The issue requests that non-main sandboxed agents resolve skill `SKILL.md` paths to sandbox-readable copies instead of global `~/.openclaw/skills` paths.

## Best Possible Solution

Sandboxed non-rw runs should rebuild or remap `<available_skills>` after sandbox context resolution so prompt-facing `<location>` values point to sandbox-readable skill copies while host metadata remains available for env and runtime state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to build a skills snapshot from the host agent workspace, enter a non-rw sandbox run, and observe that the snapshot prompt is preferred over entries from the sandbox workspace.

## Solution Assessment

No for a host allowlist or prompt-only workaround as the primary fix, because that weakens or sidesteps the sandbox contract. The narrow maintainable fix is a runtime prompt rebuild or path remap after sandbox context resolution.
