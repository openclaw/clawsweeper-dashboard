## Summary

Keep open: this is a useful Control UI bug fix with real proof, but the submitted patch is stale against current main because it changes the old inline picker path while current main renders Chat session controls through the shared `session-controls.ts` module. The underlying selected-option issue still appears present in current main, so the best next step is a narrow repair or replacement rather than closing or merging this branch as-is.

## What This Changes

The branch updates Control UI Chat session picker rendering, mobile styling, and tests to force the active session option selected after rerender and distinguish desktop/mobile selector targets.

## Best Possible Solution

Port the live selected-option binding into the shared `ui/src/ui/chat/session-controls.ts` renderer and add focused fallback-rerender coverage while preserving the current shared desktop/mobile control contract.

## Reproduction Assessment

Yes, at source level. Current main's shared picker binds `<select>.value` while options still use `?selected`, so a focused Lit rerender test can drift the live selected option, switch state to a fallback session, and assert both `.value` and `selectedOptions[0].value`.

## Solution Assessment

No, not as currently proposed. The live selected-option binding is the right narrow fix, but it needs to move from the old inline helper renderer to the current shared `session-controls.ts` path used by both desktop and mobile controls.
