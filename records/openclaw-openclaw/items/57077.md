## Summary

Keep open. The PR addresses a real Control UI session-picker desync with sufficient screenshot/manual proof, but current main moved the active desktop and mobile picker through the shared `ui/src/ui/chat/session-controls.ts` renderer, while this branch still applies the live selected-option binding in the obsolete helper implementation.

## What This Changes

The PR updates Control UI Chat session picker rendering and tests to force the active session option selected after rerender and distinguish desktop/mobile selector targets.

## Best Possible Solution

Port the live DOM-state binding for Chat session selection into the shared `ui/src/ui/chat/session-controls.ts` renderer with focused fallback-rerender coverage, rather than merging the stale helper-targeted branch as-is.

## Reproduction Assessment

Yes, at source level. Current main still renders session options with `?selected` in the shared picker, so a rerender where the live select state has drifted and the active session is injected as a fallback can be covered by a focused Lit render test that asserts both `value` and `selectedOptions[0].value`.

## Solution Assessment

No, not as currently proposed. The live selected-option binding is the right narrow fix, but it needs to be moved from the old inline helper renderer to the current shared `session-controls.ts` path used by both desktop and mobile controls.
