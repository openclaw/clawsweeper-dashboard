## Summary

Keep open. The underlying inbound debounce behavior already exists through `messages.inbound` config, so this PR is a chat-command convenience layer, but latest head is not mergeable because origin-channel `configWrites` authorization can still be skipped for external channels with no `channelId`.

## What This Changes

The PR adds `/messaging_window` and `/messaging-window` parsing, handling, command metadata, docs, tests, and a changelog entry for viewing or updating `messages.inbound` debounce windows from chat.

## Best Possible Solution

Keep the command as a convenience layer over existing `messages.inbound`, but normalize the origin channel with the same safe external fallback used for targets and add a denied-origin regression test before merge.

## Reproduction Assessment

Yes. Use a safe external channel such as `custom-sms` with no `command.channelId` and `channels.custom-sms.configWrites=false`, then run a global `/messaging_window 3s`; source review shows the handler passes a null origin into authorization.

## Solution Assessment

No. The direction is maintainable, but latest head is not the best merge state until origin-channel authorization is normalized and covered by tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Normalize origin channel before authorization:** `src/auto-reply/reply/commands-messaging-window.ts:178`
  - body: For non-bundled channel commands where `command.channelId` is unset, this still passes `params.command.channelId ?? normalizeChannelId(params.command.channel)` into config-write authorization. That can be null even though this handler accepts the same raw channel through the safe fallback for targets, so a global `/messaging_window 3s` skips `channels.<origin>.configWrites=false`. Reuse the same origin fallback before calling `resolveConfigWriteDeniedText`.
  - confidence: 0.93
