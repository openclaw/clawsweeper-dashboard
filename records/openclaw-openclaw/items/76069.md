## Summary

Keep open. Current main already supports the underlying inbound debounce behavior through `messages.inbound` config, but this PR adds a chat/native command convenience layer and latest head still leaves a security-sensitive origin-channel `configWrites` bypass for some external channel contexts.

## What This Changes

The branch adds `/messaging_window` and `/messaging-window` command support, parser/handler tests, Telegram quick-pick metadata, docs, and a changelog entry for viewing and updating `messages.inbound` batching windows.

## Best Possible Solution

If the command UX is desired, derive the origin channel with the same safe external-channel fallback used for targets, add a denied-origin regression test, and keep the command as a convenience layer over `messages.inbound`.

## Reproduction Assessment

Yes. Source review gives a high-confidence path: use a safe external channel such as `custom-sms` with no `command.channelId` and `channels.custom-sms.configWrites=false`, then a global `/messaging_window 3s` write passes null origin into authorization.

## Solution Assessment

No. The direction is a reasonable shortcut over existing config, but latest head is not the best merge state until origin-channel authorization is normalized and covered by tests.
