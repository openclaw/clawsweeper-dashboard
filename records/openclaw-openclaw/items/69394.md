## Summary

Keep open. Current main still has the reported final-payload leak, but this PR's length-based trailing-token check would also suppress legitimate non-trailing `NO_REPLY` mentions that merely have trailing whitespace, and the broader `NO_REPLY` contract still overlaps with competing open fixes.

## What This Changes

The PR changes `normalizeReplyPayload` to suppress a payload when trailing `NO_REPLY` stripping shortens the text, and updates normalize tests to expect full suppression for trailing-token emoji, substantive-text, and reasoning-preamble cases.

## Best Possible Solution

Define one explicit silent-token predicate for final payload normalization and related delivery paths, avoid trim-length inference, preserve intentional non-trailing mention behavior, add focused regression coverage for both leak and false-positive cases, and include the required changelog entry.

## Reproduction Assessment

Yes. The current main source and tests give a high-confidence reproduction for the reported leak via `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })`, and the PR regression can be reasoned from `stripSilentToken()` trimming plus `stripped.length < text.length`.

## Solution Assessment

No. Suppressing the reported trailing-token leak is the right direction, but this implementation is not the best fix because it infers token removal from raw string length instead of an explicit trailing-token match or the existing compare-against-trimmed approach.
