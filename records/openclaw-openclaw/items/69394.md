## Summary

Keep open: current main still has the reported silent-reply leak, but this PR has a blocking false-positive regression and only test-based proof. Related open PRs also show that maintainers still need to choose the intended suppression boundary.

## What This Changes

The PR changes `normalizeReplyPayload` and tests so any payload shortened by trailing `NO_REPLY` stripping is treated as silent instead of delivering the remaining text.

## Best Possible Solution

Land one canonical silent-reply classifier that suppresses reasoning-prefaced silent outputs without dropping legitimate non-trailing `NO_REPLY` mentions, with focused regression tests and redacted dispatcher or channel proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-empty preamble before `NO_REPLY` normalizes to deliverable text, and current tests still assert delivery for text before a trailing silent token.

## Solution Assessment

No, not as written. Suppressing reasoning-prefaced silent replies is valid, but this patch infers a trailing token from raw length after a trimming helper; use an explicit trailing-token predicate or compare against the pre-strip trimmed text after maintainers choose the policy.

## Review Findings

Overall correctness: patch is incorrect
