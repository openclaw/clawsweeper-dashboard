## Summary

Keep open. Current main still has the reported final-payload leak, but this PR introduces a blocking false-positive by inferring trailing-token removal from raw length after a helper that always trims; it also needs maintainer agreement on the intended NO_REPLY semantics across several overlapping open alternatives before merge or replacement.

## What This Changes

The PR changes `normalizeReplyPayload` to suppress payloads when `stripSilentToken` shortens text, flips two mixed `NO_REPLY` tests to expect full suppression, and adds a reasoning-preamble regression test.

## Best Possible Solution

Define one explicit silent-token predicate for final payload delivery, use it consistently in the normalizer and adjacent delivery gates, preserve non-trailing mention behavior with regression tests, cover the false-positive cases already represented in cron tests, and add the required changelog entry once maintainers choose the intended mixed-token semantics.

## Reproduction Assessment

Yes. Current main can be reproduced from source and tests with `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })`, which is still expected to return non-null. The proposed patch regression is also reproducible by reasoning from `stripSilentToken()` trimming plus `stripped.length < text.length` on non-trailing `NO_REPLY` text with trailing whitespace.

## Solution Assessment

No. Full suppression for the reported leak may be the right product direction, but this implementation is not the best fix because it uses trim-length inference instead of an explicit trailing-token check or comparison against the pre-strip trimmed value.

- **[P2] Detect trailing NO_REPLY without trim-length inference:** `src/auto-reply/reply/normalize-reply.ts:73`
  - body: `stripSilentToken()` trims the result even when it removes no token. Use an explicit trailing-token predicate or compare against the pre-strip trimmed value before suppressing the whole payload.
