## Summary

Keep this PR open: the reported leak still exists on current main, but this patch is not merge-ready because it infers trailing-token removal from trim-related length changes and can suppress legitimate non-trailing `NO_REPLY` mentions. The broader `NO_REPLY` semantics also overlap several open alternatives, so maintainer choice is still needed before landing one direction.

## What This Changes

The PR changes `normalizeReplyPayload` to suppress the whole payload when trailing `NO_REPLY` stripping shortens text, and updates normalize tests to expect full suppression for trailing-token emoji, substantive-text, and reasoning-preamble cases.

## Best Possible Solution

Introduce an explicit final-payload silent-token predicate for the non-streaming normalize path, cover leak and false-positive cases, preserve intentional non-trailing mentions, and add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` strips the token and keeps the non-empty preamble, and current tests assert that behavior.

## Solution Assessment

No. Suppressing reasoning-prefaced trailing silent replies is the right direction, but this implementation should use an explicit trailing-token match or compare against the pre-strip trimmed value instead of raw length inference.
