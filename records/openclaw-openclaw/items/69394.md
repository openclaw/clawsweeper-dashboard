## Summary

Keep open for maintainer review. Current main still has the reported non-streaming auto-reply behavior, but the proposed patch should not merge as-is because it infers a trailing token from string length after trimming and can suppress legitimate messages that only mention `NO_REPLY` with trailing whitespace; several open alternatives also make different product choices about the existing mixed-content contract.

## What This Changes

The PR changes `normalizeReplyPayload` to suppress a payload whenever `stripSilentToken` shortens the text, and updates tests to expect full suppression for mixed or trailing `NO_REPLY` content plus a reasoning-preamble case.

## Best Possible Solution

Adopt one canonical silent-token predicate across the final payload, directive, cron, and streaming paths; make the #30916/#19537 compatibility decision explicit; and cover both reasoning-preamble suppression and non-trailing `NO_REPLY` mention false positives with focused regression tests.

## Reproduction Assessment

Yes. The current source path and tests give a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is expected on main to return a non-null payload with `NO_REPLY` stripped, which matches the reported leak shape for preamble text.

## Solution Assessment

No. The proposed direction addresses the target leak, but the trim-length heuristic is not the narrowest maintainable fix; an explicit trailing-token or own-line predicate, or the cron helper's compare-to-trimmed approach, would avoid suppressing legitimate non-trailing mentions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Detect trailing NO_REPLY without trim-length inference:** `src/auto-reply/reply/normalize-reply.ts:73`
  - body: `stripSilentToken()` trims the string even when it removes no token. With this change, inputs like `NO_REPLY -- explanation\n` or any non-trailing `NO_REPLY` mention with trailing whitespace satisfy `stripped.length < text.length`, causing `normalizeReplyPayload` to return null and drop legitimate content. Compare against the pre-strip trimmed value or use an explicit trailing-token match before suppressing the whole payload.
  - confidence: 0.91
