## Summary

Keep open: the reported leak is source-reproducible on current main, but this PR still needs real behavior proof and its implementation can suppress legitimate non-trailing `NO_REPLY` mentions because it infers a trailing token from post-trim length.

## What This Changes

The PR changes `normalizeReplyPayload` and tests so payloads shortened by trailing `NO_REPLY` stripping are treated as silent instead of returning the remaining text.

## Best Possible Solution

Land one canonical shared silent-reply classifier that suppresses reasoning-prefaced silent outputs without dropping legitimate non-trailing mentions, with focused regression tests and real dispatcher or channel proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-empty preamble before `NO_REPLY` normalizes to deliverable text, and current tests still assert delivery for trailing-token content; I did not run tests because this review is read-only.

## Solution Assessment

No. Suppressing reasoning-prefaced silent replies is valid, but this patch uses whitespace-sensitive length inference and broad trailing-token suppression; a safer fix should use an explicit trailing-token predicate or trimmed comparator after maintainers choose the policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use explicit trailing NO_REPLY detection:** `src/auto-reply/reply/normalize-reply.ts:73-74`
  - body: `stripSilentToken()` trims even when it removes no trailing token. With this change, text like `NO_REPLY -- explanation\n` or any non-trailing mention with trailing whitespace makes `stripped.length < text.length` true and gets incorrectly suppressed as silent; compare against the pre-strip trimmed value or use an explicit trailing-token predicate.
  - confidence: 0.91
