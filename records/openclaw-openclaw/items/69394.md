## Summary

Keep this PR open: it targets a real current-main final-payload leak, but the patch has a concrete false-positive regression and still needs changelog coverage before it is mergeable.

## What This Changes

The PR changes `normalizeReplyPayload` to suppress the whole payload when trailing `NO_REPLY` stripping shortens text, and flips normalize tests to expect suppression for emoji, substantive-text, and reasoning-preamble trailing-token cases.

## Best Possible Solution

Define one explicit final-payload silent-token predicate, use a real trailing-token match or trimmed-before comparison, preserve non-trailing mentions, cover leak and false-positive cases, and add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection of current main shows `normalizeReplyPayload({ text: "...\n\nNO_REPLY" })` strips the token and then returns the remaining text when `hasContent` succeeds; existing tests assert that old mixed-content behavior.

## Solution Assessment

No. Suppressing the leak is the right direction, but this patch infers token removal from raw length and broadens legacy mixed-content semantics without an explicit maintainable predicate.
