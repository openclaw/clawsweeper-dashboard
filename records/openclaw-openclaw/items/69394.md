## Summary

Keep open: the reported leak is still source-reproducible on current main, but this PR has a blocking false-positive in its trailing-token detection, lacks contributor real behavior proof, and overlaps other open implementation candidates that need maintainer policy selection.

## What This Changes

The PR changes reply normalization to suppress the whole payload when trailing `NO_REPLY` stripping shortens the text, and flips related normalizer tests to expect full suppression.

## Best Possible Solution

Land one canonical shared silent-reply classifier that suppresses reasoning-only final `NO_REPLY` outputs without dropping substantive non-trailing mentions, with focused tests and real dispatcher or channel proof.

## Reproduction Assessment

Yes, source-reproducible. Current main strips a trailing silent token and returns non-empty preamble text, and current tests still assert strip-and-deliver behavior for trailing-token content.

## Solution Assessment

No. Suppressing reasoning-prefaced silent replies is a valid direction, but this patch should use an explicit trailing-token predicate or compare with pre-strip trimmed text, and the broader any-trailing-token policy needs maintainer selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Detect trailing NO_REPLY without trim-length inference:** `src/auto-reply/reply/normalize-reply.ts:73-74`
  - body: `stripSilentToken()` trims even when it removes no trailing token. With this change, text like `NO_REPLY -- explanation\n` or any non-trailing mention with trailing whitespace can satisfy `stripped.length < text.length` and be dropped as silent; compare against `text.trim()` or use an explicit trailing-token predicate before suppressing the whole payload.
