## Summary

This PR should stay open: it targets a real main-session `sessions_yield` lifecycle gap, but the current branch is not merge-ready because it adds a new persisted `SessionEntry` key without reserving it for plugin slot protection, leaves Control UI consumers unable to model `paused`, and still lacks after-fix real behavior proof.

## What This Changes

The PR persists top-level `sessions_yield` lifecycle endings as `status: "paused"` with `pauseReason: "sessions_yield"`, propagates that metadata to session rows and broadcasts, skips restart recovery for paused-yield entries, and adds focused tests plus changelog text.

## Best Possible Solution

Land the lifecycle fix only after reserving the new core session key, wiring `paused` through UI/shared session consumers as a non-terminal state, and adding redacted real runtime proof for main-session yield plus restart recovery.

## Reproduction Assessment

Yes, at source level: current main emits `yielded: true` from the runner lifecycle path while gateway session persistence ignores that flag and maps a clean terminal lifecycle event to `done`. I did not run a live smoke in this read-only review.

## Solution Assessment

No, not yet. The gateway lifecycle seam is the right place to classify yielded main-session runs, but this branch must reserve the new session key and update Control UI/shared consumers before the fix is complete.
