## Summary

Keep open: the gateway-side pause mapping is a useful direction and the latest PR head fixes the earlier error/yield leak, but the patch is not merge-ready because `paused` is exposed to consumers that still treat non-running statuses as inactive/interrupted, one exported terminal-status guard is now unsound, and the supplied proof is still diff-only rather than a patched runtime smoke.

## What This Changes

The PR marks clean main-session `sessions_yield` lifecycle ends as `paused` with `pauseReason: sessions_yield`, propagates that through session rows/list tooling, skips restart recovery for paused entries, and adds focused tests.

## Best Possible Solution

Land a version that treats `paused` as a first-class nonterminal state across gateway broadcasts, sessions list consumers, Control UI run lifecycle/status badges, CLI/session helpers, and config type guards, with redacted runtime proof of main-session yield plus restart recovery.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a credible path: current main ignores lifecycle `yielded` in `deriveGatewaySessionLifecycleSnapshot`, but I did not run a live main-session yield/restart smoke.

## Solution Assessment

Is this the best way to solve the issue? Not as-is: gateway lifecycle persistence is the right core layer, but the PR must update every consumer of the new status contract and prove the runtime path before this is the best fix.
