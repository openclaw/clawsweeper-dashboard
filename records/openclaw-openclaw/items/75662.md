## Label Justifications

- `P2`: This is a normal-priority session-state/message-delivery bug fix with meaningful but bounded agent/gateway/UI blast radius.
- `merge-risk: 🚨 session-state`: The PR changes persisted and broadcast session status semantics, and current UI consumers still treat the new paused state as terminal.
- `merge-risk: 🚨 message-delivery`: The affected workflow is a yielded main-session continuation where misclassified state can still race follow-up turns or make the session look finished.

## Summary

Keep open: the backend pause-state direction is plausible, but the PR adds a new `paused` session status without updating Control UI/session consumers, so yielded sessions can still be treated as terminal or unknown. The supplied proof is also not enough real behavior proof for an external bug-fix PR.

## What This Changes

The PR persists top-level `sessions_yield` lifecycle ends as paused session entries, propagates `pauseReason`, skips restart recovery for those entries, and adds focused gateway/agent tests plus a changelog entry.

## Best Possible Solution

Land a complete paused-session contract that treats `status: "paused"` and `pauseReason: "sessions_yield"` as an awaiting-continuation state across gateway, `sessions_list`, and Control UI, with targeted tests and live yield/restart proof.

## Reproduction Assessment

Yes, at source level: current main emits `yielded: true`/`livenessState: "paused"` from the runner, while gateway lifecycle persistence ignores `yielded` and falls through to `done`. I did not establish a live current-main repro in this read-only review.

## Solution Assessment

No, not yet. The backend persistence change is the right direction, but the PR needs paused-state consumer updates and live proof before it is the safest complete fix.
