## Summary

Keep this PR open: the gateway-side fix direction is sound, but the branch is not merge-ready because the new `paused` session state is not taught to Control UI consumers and the supplied proof shows the diff rather than after-fix runtime behavior.

## What This Changes

The PR maps yielded main-session lifecycle ends to a persisted `paused`/`sessions_yield` state, propagates that state through gateway session surfaces, skips restart recovery for paused entries, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a complete pause-state contract where gateway, sessions_list, Control UI, and chat-run reconciliation all treat `paused`/`sessions_yield` as a non-terminal yielded state, with focused UI/backend tests and redacted runtime proof for a top-level yield plus restart recovery.

## Reproduction Assessment

Yes, at source level: current main already carries `yielded` lifecycle metadata, while `deriveGatewaySessionLifecycleSnapshot` ignores it and maps the terminal event to `done`. I did not run a live smoke in this read-only review.

## Solution Assessment

No, not yet. The gateway persistence approach is the right shape, but the solution is incomplete until Control UI and other session consumers treat `paused` as non-terminal and real behavior proof demonstrates the flow.

Full review comments:

- **[P1] Teach UI consumers that paused is non-terminal:** `src/gateway/session-utils.types.ts:26`
  - body: Adding `paused` as a Gateway session status is not enough: the Control UI still declares `SessionRunStatus` without `paused`, and `reconcileChatRunFromCurrentSessionRow` treats every non-`running` status as terminal/interrupted.
