## Summary

Keep open. Current main still does not expose an exec-tool `description` or `intent` parameter: the schema, docs, and runtime arg shape list only command/execution controls, while `before_tool_call` hooks only receive whatever tool params the schema/provider path allows. The request is valid but security-sensitive because a model-supplied intent string must remain informational rather than becoming an approval authority.

## What This Changes

Request to add an optional human-readable exec tool description or intent field for hooks, approval prompts, audit logs, and safety guardrails.

## Best Possible Solution

Add a backward-compatible optional string field to the exec tool contract, pass it through to hook payloads and display/audit surfaces as informational metadata, and document that approval enforcement remains based on the actual command and existing binding policy.

## Reproduction Assessment

Yes. A read-only current-main check reproduces the feature gap by inspecting the exec schema, runtime arg shape, hook forwarding path, and docs; the official exec contract has no description/intent parameter today.

## Solution Assessment

Unclear until maintainers choose the trust model. The narrow implementation is straightforward, but the best version should explicitly keep the field informational and avoid changing allowlist or approval-bypass behavior based on natural-language text.

## Risks / Open Questions

- The field name and trust semantics are not settled: `description` is intuitive for UI/logs, while `intent` may better signal model-supplied intent.

- A complete implementation likely needs schema, runtime typing, docs, and focused hook/approval tests so provider tool-schema conversion does not silently drop the new field.
