## Summary

Keep open. Current main still lacks the requested exec `description`/`intent` field in the exec tool schema and runtime approval/hook path, and the only related duplicate is closed in favor of this canonical request, so there is remaining product/security design work rather than a cleanup close.

## What This Changes

The issue requests an optional natural-language `description` or `intent` field on exec tool calls so hooks and approval flows can reason about command purpose.

## Best Possible Solution

Add a backward-compatible optional exec intent metadata field through the exec schema/runtime, hook-visible params, approval payloads, docs, and tests, while documenting that enforcement must remain command-bound.

## Reproduction Assessment

Yes. Source inspection on current main shows the exec contract accepts no `description` or `intent` field, and the hook path receives only the params admitted by that contract.

## Solution Assessment

Unclear until maintainers settle the field name and trust model. The requested direction is maintainable only if the new field is informational and cannot authorize execution by itself.
