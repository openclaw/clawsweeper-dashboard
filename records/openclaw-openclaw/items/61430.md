## Summary

Keep open. Current main and the latest release still lack the requested exec `description`/`intent` field; https://github.com/openclaw/openclaw/issues/61562 was closed as the duplicate, and adjacent guardrail/provenance issues do not replace this narrow exec parameter contract. The remaining work needs product/security judgment on naming and trust semantics before implementation.

## What This Changes

Request adding optional `description` or `intent` metadata to exec tool calls so hooks, approvals, and audits can see model-supplied command intent.

## Best Possible Solution

Add a backward-compatible optional exec intent string through schema/runtime/docs and hook-visible params, while documenting and testing that approval enforcement remains bound to the actual command and existing approval policy.

## Reproduction Assessment

Yes. Source and docs inspection on current main reproduces the feature gap: exec has no `description` or `intent` field, and hooks only receive params exposed by the tool contract.

## Solution Assessment

Unclear until maintainers choose the field name and trust model. The requested direction is maintainable only if the field is informational and cannot bypass command-bound approval checks.
