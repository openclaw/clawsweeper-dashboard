## Summary

Keep open. Current main still does not expose an exec-tool `description` or `intent` parameter, while the hook surface can only inspect parameters that reach the tool-call payload; #61562 was closed as the duplicate, so this remains the canonical tracker and needs maintainer/security judgment before implementation.

## What This Changes

Request to add an optional human-readable exec `description` or `intent` field for hooks, approval prompts, audit logs, and safety guardrails.

## Best Possible Solution

Expose a backward-compatible optional string field in exec’s schema/runtime and hook-visible params, while documenting that approval enforcement remains bound to the actual command and existing approval policy.

## Reproduction Assessment

Yes. A read-only current-main check reproduces the feature gap by inspecting the exec schema, runtime arg shape, hook forwarding path, and user docs; the official exec contract has no `description` or `intent` parameter today.

## Solution Assessment

Unclear until maintainers choose the trust model and field name. The narrow code change is feasible, but the maintainable version must keep the field informational and avoid approval bypass behavior based on natural-language text.

## Risks / Open Questions

- The field name and trust semantics are not settled: `description` is clear for UI/logs, while `intent` better signals model-supplied intent.
- Using model-supplied intent as an allow/block authority would be unsafe; command text, cwd, env, canonical binding, allowlists, and explicit approvals still need to govern execution.
- A complete implementation needs schema, runtime typing, docs, and focused hook/approval tests so provider schema conversion does not silently drop the new field.
