## Summary

Keep open. Current main still lacks optional model-supplied exec `description` or `intent` metadata, and adding it crosses the exec approval and hook trust boundary, so this needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests an optional exec `description` or `intent` field so hooks and approval flows can reason about command purpose without parsing only the command string.

## Best Possible Solution

Add an additive optional exec intent metadata field only as advisory data, carry it through exec schema/runtime, hook-visible params, approval payloads, docs, and focused tests, and document that enforcement remains command-bound.

## Reproduction Assessment

Yes. Source inspection on current main shows the exec schema, runtime args, hook-visible params, approval protocol, and public exec docs still do not include a per-call `description` or `intent` field.

## Solution Assessment

Unclear until maintainers settle the field name and trust boundary. The requested direction is maintainable if the field is explicitly advisory and never authorizes execution by itself.
