## Summary

Keep this PR open. Current main does not implement the requested workspace file-management API, but the latest PR head still introduces a security-sensitive workspace read/write surface with raw recursive delete and rename operations after prechecks, plus API-default and documentation/changelog gaps that should be addressed before merge.

## What This Changes

The PR adds `agents.workspace.*` gateway schemas, scopes, handlers, tests, generated Swift protocol models, and a Control UI Workspace tab for browsing, editing, uploading, downloading, deleting, creating, moving, and stat-ing agent workspace files.

## Best Possible Solution

Keep the feature direction, but repair the filesystem boundary model before merge: destructive recursive operations should use root-pinned helpers or a narrower supported contract, and the gateway schema, UI behavior, protocol docs, changelog, generated clients, and focused tests should agree on defaults and limits.

## Reproduction Assessment

Yes. The blocking findings have a high-confidence static reproduction path on the latest PR head: the code visibly calls raw `fs.rm` and `fs.rename` after boundary prechecks, and omitted API defaults are visibly coerced to `false`.

## Solution Assessment

No. The feature matches the linked workspace-management request, but the current patch is not the best maintainable solution until destructive filesystem operations are root-pinned and the public API contract is aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pin recursive deletes to the workspace root:** `src/gateway/server-methods/agents-workspace.ts:234`
  - body: `workspaceDelete` verifies `realPath`, then calls `fs.rm(fullPath, { recursive: true })` on the original path string. A workspace-writable process can swap an already checked ancestor before `fs.rm` resolves it again, so this admin RPC can delete outside the workspace; use a root-pinned recursive delete helper or narrow the contract until one exists.
  - confidence: 0.84
- **[P1] Pin moves instead of renaming checked paths:** `src/gateway/server-methods/agents-workspace.ts:343`
  - body: `workspaceMove` validates the source and destination ancestors, but then performs `fs.rename(fromFullPath, toFullPath)` on mutable path strings. A symlink-ancestor swap between validation and rename can redirect the operation outside the workspace, so this needs a root-pinned move implementation or a narrower move contract.
  - confidence: 0.86
- **[P2] Restore default parent creation for writes:** `src/gateway/server-methods/agents-workspace.ts:527`
  - body: Omitted `createDirs` is converted to `false`, but the linked workspace API contract says `agents.workspace.set` creates parent directories by default. API clients sending a nested new path without the optional flag will fail despite that contract; use `params.createDirs !== false` or change the contract and tests together.
  - confidence: 0.88
- **[P2] Restore recursive mkdir as the default:** `src/gateway/server-methods/agents-workspace.ts:617`
  - body: Omitted `parents` is converted to `false`, while the linked workspace API contract describes `agents.workspace.mkdir` as creating parents by default. Use `params.parents !== false`, or update the public contract, UI assumptions, and tests to make non-recursive the default.
  - confidence: 0.88
- **[P3] Document the workspace RPC contract:** `src/gateway/protocol/schema/agents-workspace.ts:24`
  - body: This adds public `agents.workspace.*` protocol methods, but the user-facing protocol docs still only describe existing operator helpers and `agents.files.*` behavior. Add the method summary, scopes, encoding, path, and size-limit behavior so clients have a stable contract to follow.
  - confidence: 0.86
