## Summary

Keep open. Current main does not implement the requested workspace file-management RPC/UI, and this PR is the active implementation for the linked feature request, but the PR head still has blocking security and correctness findings in the new workspace move path, refresh behavior, and tests.

## What This Changes

The PR adds `agents.workspace.*` gateway methods, protocol/schema/client model updates, scope registrations, server handlers/tests, and a Control UI Workspace tab for browsing, uploading, downloading, editing, deleting, creating, and moving agent workspace files.

## Best Possible Solution

Keep the PR open and require fixes before merge: the server move path should use a root-pinned/safe move primitive or propagate outside-root ancestor errors, the Workspace refresh flow should reload the tree without discarding dirty editor state, and the new tests should be aligned with the final lstat/schema behavior before targeted and changed-lane validation.

## Reproduction Assessment

Yes for the blocking review findings. The move escape is reproducible by reasoning from the PR head code with a workspace symlink such as `out -> /tmp` and destination `out/new/file.txt`; the refresh data-loss path is visible from the Refresh button calling `onNavigate`, which clears editor state.

## Solution Assessment

No. The feature direction is coherent and not implemented on current main, but this PR is not the best mergeable solution until the new write-capable RPC preserves workspace boundaries, the UI preserves unsaved edits on refresh, and the tests accurately cover the final implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Propagate outside-root destination errors:** `src/gateway/server-methods/agents-workspace.ts:307-313`
  - body: The destination ancestor check throws when `realDir` resolves outside the workspace, but the broad `catch` immediately swallows that throw and walks to the parent. With a workspace symlink like `out -> /tmp` and `to: "out/new/file.txt"`, the later `fs.mkdir`/`fs.rename` can write outside the workspace boundary.
  - confidence: 0.93
- **[P2] Preserve edits when refreshing the workspace tree:** `ui/src/ui/views/agents-panels-workspace.ts:81`
  - body: The Refresh button calls `onNavigate(currentPath)`, and that handler clears `workspaceSelectedFile`, `workspaceFileContent`, and `workspaceEditedContent` before reloading. A user with dirty editor content loses unsaved changes by pressing Refresh.
  - confidence: 0.9
- **[P2] Mock lstat in delete tests:** `src/gateway/server-methods/agents-workspace.test.ts:545-549`
  - body: `workspaceDelete` now starts with `fs.lstat`, but these tests only set up `mocks.fsStat`, so `lstat` returns `undefined` and `lstat.isDirectory()` throws before the delete behavior is exercised. Update the tests to mock the call the implementation actually uses.
  - confidence: 0.82
- **[P2] Align empty get-path coverage with the schema:** `src/gateway/server-methods/agents-workspace.test.ts:992-997`
  - body: The test expects `agents.workspace.get` with `path: ""` to succeed, but the new params schema uses `NonEmptyString` for get paths, so validation rejects the request before the handler reaches the read helper. Either make the schema allow the root path for get, or change this to assert the invalid-params response.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a write-capable workspace RPC surface, and the final move implementation still has a concrete symlink destination escape.

Concerns:

- **[high] Move destination can escape workspace root:** `src/gateway/server-methods/agents-workspace.ts:307`
  - body: `workspaceMove` detects an outside-root resolved ancestor, but throws inside a broad `try` whose `catch` treats every error as a missing ancestor. A symlinked destination ancestor can therefore be ignored and the subsequent raw `fs.mkdir`/`fs.rename` can write outside the agent workspace.
  - confidence: 0.93
