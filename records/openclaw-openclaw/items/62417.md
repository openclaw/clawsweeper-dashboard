## Summary

Keep open. Current main still lacks the workspace-management RPC/UI capability, so the PR is not obsolete, but the latest head is not merge-ready because the new security-sensitive workspace filesystem surface still performs raw path-string filesystem operations after boundary prechecks and has protocol/defaults/docs/changelog gaps.

## What This Changes

The PR adds `agents.workspace.*` gateway schemas, scopes, handlers, tests, generated Swift protocol models, a Control UI Workspace tab for browsing/editing/uploading/downloading agent workspace files, and a changelog entry.

## Best Possible Solution

Keep the PR open and repair the server-side workspace filesystem model around root-pinned helpers or an explicit narrower contract, then align defaults, protocol docs, generated clients, tests, and changelog credit before maintainer/security review considers merge.

## Reproduction Assessment

Yes. The blockers have high-confidence source reproductions on the PR head: raw `fs.readdir`, `fs.rm`, `fs.mkdir`, and `fs.rename` are still called after mutable-path prechecks, and omitted `createDirs`/`parents` are visibly coerced to `false`.

## Solution Assessment

No. The feature direction fits the linked workspace-management request, but this implementation is not the safest maintainable solution until filesystem boundary semantics and the public API contract are aligned.

- **[P1] Pin directory listings to the workspace root:** `src/gateway/server-methods/agents-workspace.ts:114`
  - body: `workspaceList` verifies `realTarget`, then calls `fs.readdir(targetPath)` on the original mutable path. A workspace-writable process can swap a checked ancestor before `readdir` resolves it again, exposing directory names outside the workspace; use a root-pinned directory listing helper or reject this operation until one exists.
  - confidence: 0.82

  - body: The recursive delete path checks `realPath`, then calls `fs.rm(fullPath, { recursive: true })` on the unchecked path string. A symlink-ancestor swap between those steps can redirect deletion outside the agent workspace, so destructive deletes need a root-pinned helper or a narrower supported contract.
  - confidence: 0.86
- **[P1] Pin non-recursive mkdir to the workspace root:** `src/gateway/server-methods/agents-workspace.ts:274`
  - body: When `parents` is false, the handler validates the parent with `realpath` and then calls raw `fs.mkdir(fullPath)`. Since the parent path can be swapped after validation, this admin RPC can create directories outside the workspace; use a pinned mkdir helper or keep only the pinned recursive path.
  - confidence: 0.81

  - confidence: 0.88
