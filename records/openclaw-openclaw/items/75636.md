## Summary

Keep this open. The report is concrete and current main still routes bundled plugin runtime-deps repair through the same generated manifest plus package-manager install path, with no inspected handling for npm's `Invalid Version:` Arborist dedupe failure. The earlier #74949 fix is present, but it only ensures the generated manifest has a dependency map; this issue describes a later failure after npm starts placing transitive packages.

## What This Changes

The issue reports that `openclaw@2026.4.29` fails to install bundled plugin runtime dependencies on Node v22/npm v10 with npm Arborist `Invalid Version:` during dedupe, leaving runtime-deps plugins unloaded.

## Best Possible Solution

Harden the bundled runtime-deps installer for npm v10/Node v22 while preserving the manifest-first package-level plan. The desired end state is that a fresh packaged Gateway can converge the complete bundled plugin runtime-deps set or retry through a narrowly validated npm fallback without leaving acpx, bonjour, browser, file-transfer, and memory-core unavailable.

## Reproduction Assessment

Yes for the shipped-release bug: the report includes exact install and Gateway commands, Node/npm/macOS versions, npm log stack, and degraded plugin-load symptoms. Reproduction against current main is not yet proven because this read-only review did not run the package install path and no existing test simulates the npm `Invalid Version:` failure.

## Solution Assessment

Unclear. The #74949 manifest fix is correct for the earlier empty-manifest failure, but this report shows populated manifests are not enough for npm's later dedupe path; a narrow installer fallback or install-plan adjustment is safer than closing this as already implemented.
