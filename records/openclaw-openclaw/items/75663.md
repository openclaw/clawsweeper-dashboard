## Summary

Keep open. Current main still has the reported active-requester completion path: when steering fails and the active-session direct fallback cannot send, delivery returns false before the existing gateway `agent` call can run. Existing coverage proves the successful fallback case, but not the fallback-miss path described here.

## What This Changes

The issue requests that subagent completion announcements fall through to direct gateway delivery when Telegram cannot be woken through the parent session's active embedded run path.

## Best Possible Solution

Update completion announce delivery so an active requester that cannot be steered only stops on a confirmed successful direct fallback; fallback miss or recoverable fallback failure should continue into the existing gateway `agent` delivery path, with regression coverage for Telegram-style active-not-streaming completion delivery.

## Reproduction Assessment

Yes. A focused unit reproduction can set `expectsCompletionMessage: true`, `requesterActivity.sessionId`, `requesterActivity.isActive: true`, `queueEmbeddedPiMessage` returning false, and an empty or unroutable fallback, then assert current main returns false before `callGateway` is invoked.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix if implemented with a regression test: preserve successful direct fallback, but let the existing gateway delivery path handle fallback miss/failure instead of ending the direct phase early.
