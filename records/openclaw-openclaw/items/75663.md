## Summary

Keep open. Current main still has the reported active-requester path: if steering fails and the active-session fallback cannot send, delivery returns false before the existing gateway `agent` delivery path can run; an open linked PR already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Telegram subagent completion announcements are queued but not pushed when the parent session is active yet not consuming messages between turns.

## Best Possible Solution

Review and land #75669 or an equivalent narrow fix that preserves successful direct fallback, but lets fallback miss or recoverable fallback failure continue into the existing gateway `agent` delivery path with regression tests for active Telegram-style completion delivery; close this issue only after that fix merges.

## Reproduction Assessment

Yes. A focused unit reproduction can set `expectsCompletionMessage: true`, an active requester session id, `queueEmbeddedPiMessage` returning false, and an empty or failing completion fallback, then assert current main returns false before `callGateway` is invoked.

## Solution Assessment

Yes. The narrowest maintainable solution is to keep the successful fallback early return but allow fallback miss/failure to fall through to the already-wired gateway delivery path, covered by a regression test; the open linked PR appears aimed at that path.
