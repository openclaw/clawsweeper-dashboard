## Summary

Keep open: current main still has the active-requester fallback dead end, and #75669 is the open, unmerged implementation candidate with closing syntax.

## What This Changes

The issue asks for subagent completion announcements to fall through to existing gateway delivery when Telegram steering and direct completion fallback cannot wake an active requester.

## Best Possible Solution

Land #75669 or an equivalent narrow patch with regression coverage that lets fallback miss or recoverable fallback failure continue into the existing gateway delivery path while preserving successful steering and direct fallback behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction can set `expectsCompletionMessage` true, an active requester session id, `queueEmbeddedPiMessage` returning false, and missing or throwing fallback delivery; current main returns undelivered before `callGateway` is invoked.

## Solution Assessment

Yes. Letting the failure path fall through to the existing gateway `agent` delivery call is the narrowest maintainable fix because the delivery mechanism already exists and no new channel, config, or product behavior is needed.
