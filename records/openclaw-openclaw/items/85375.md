## Summary

Keep open: the redesign heads in the right direction, but the branch still exposes dynamic hook fields incompletely and its static append cacheability contract does not match marker-bearing prompts. Current-head real behavior proof is also insufficient after the redesign.

## What This Changes

The PR adds prompt-cache boundary helpers, new dynamic before_prompt_build system-context fields, runtime/override routing below the cache marker, SDK API baseline updates, JSDoc updates, and regression tests.

## Best Possible Solution

Finish one coherent hook contract: static fields either truly stay in the cacheable prefix for marker-bearing prompts or the docs say otherwise, dynamic fields are applied across all prompt-build consumers, and current-head live cache proof covers the redesigned path.

## Reproduction Assessment

Yes. Source inspection of current main shows volatile runtime context flows through appendSystemContext and flat-joins with marker-bearing system prompts, while the linked issue provides live provider cache-miss telemetry for the same class.

## Solution Assessment

No. Separating volatile dynamic context is the right direction, but the current PR is not the best final shape until the new fields are wired through every prompt-build runtime and the static append cache contract is made true or corrected.
