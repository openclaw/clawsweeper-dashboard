## Summary

This PR should stay open, but it is not merge-ready: current main still lacks the shared `_shared` memory store, while the PR head has unresolved correctness/security defects, is conflicting with the current memory-core layout, and lacks inspectable real behavior proof.

## What This Changes

The PR adds `memorySearch.sharedPaths`, a reserved `_shared` memory namespace, shared-memory CLI flags, shared-result merging, local-index exclusion for shared paths, and config/memory tests.

## Best Possible Solution

Rework the feature against the current memory-core plugin and memory-host SDK contracts, with explicit shared-memory config/docs, strict path validation, regression tests, security review, and redacted live proof.

## Reproduction Assessment

Yes for the PR review defects: PR-head source inspection shows blank `sharedPaths` resolution, default `extraPaths` behavior changes, eager directory creation, first-match weighting, and hard-coded shared status paths. This is not a current-main bug reproduction because current main has no shared `_shared` memory store.

## Solution Assessment

No. A shared memory store may be useful, but this branch is not the best merge path until it is rebased to the current memory-core and memory-host SDK layout, preserves or explicitly redesigns `extraPaths` semantics, fixes path-scope bugs, and supplies real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject blank sharedPaths before resolving:** `src/agents/memory-search.ts:228-231`
  - body: Blank string or object paths are mapped into `settings.sharedPaths`; the later shared read check can treat the empty path as a broad prefix on POSIX systems, widening markdown read access beyond explicitly configured shared directories.
  - confidence: 0.86
- **[P1] Keep default extraPaths agent-scoped:** `src/agents/memory-search.ts:234-263`
  - body: This migrates `agents.defaults.memorySearch.extraPaths` into the shared store and removes those defaults from local `extraPaths`, changing the documented/current behavior where defaults are merged into each agent's own index.
  - confidence: 0.84
- **[P1] Resolve extraPaths relative to each workspace:** `src/agents/memory-search.ts:262-263`
  - body: The new `resolveUserPath()` call resolves relative agent `extraPaths` against the process working directory, while current indexing resolves them relative to the agent workspace; configs like `../team-docs` can silently point at the wrong location.
  - confidence: 0.82
- **[P2] Avoid mkdir on shared file paths:** `src/memory/manager.ts:250-251`
  - body: Shared paths may be individual markdown files, but the constructor calls `ensureDir(sp.path)` for every configured shared path; a configured `policy.md` path can be created as a directory and block the intended file from being indexed.
  - confidence: 0.8
- **[P2] Apply shared path overrides last:** `src/memory/manager.ts:444-447`
  - body: The weighting loop uses the first matching shared path, so when defaults and an agent override contain the same path with different weights, the default entry wins and the agent-specific override is ignored.
  - confidence: 0.78
- **[P2] Use the actual shared store path for status:** `src/agents/memory-search.ts:265`
  - body: `sharedStorePath` is hard-coded from the `_shared.sqlite` helper, but the `_shared` manager can use a custom `memorySearch.store.path` template, so status can report counts from the wrong database.
  - confidence: 0.76
