## Summary

This PR addresses a real current-main gap, but it should not merge as-is: current main still lacks broad `web_fetch` private-network config, the branch uses a top-level API shape that conflicts with the newer nested `ssrfPolicy` direction, it is conflicting/stale, and it still needs maintainer security approval plus real behavior proof.

## Best Possible Solution

Rework around one maintainer-approved nested `web_fetch` SSRF policy contract with metadata protections, cache isolation for every policy flag, redirect coverage, generated config metadata, public docs, changelog, targeted tests, and redacted real private-target proof.

## Solution Assessment

No, not as this branch stands. The capability is useful, but the public API shape, metadata-denylist policy, stale branch state, missing docs/proof, and security approval gap make a maintainer-approved rework safer.
