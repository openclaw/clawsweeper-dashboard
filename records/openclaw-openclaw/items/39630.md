## Summary

Keep open: current main still lacks this broad web_fetch private-network opt-in, but this PR is not merge-ready because it conflicts with the newer `ssrfPolicy` shape and leaves a security-sensitive metadata-endpoint policy unresolved; #74790 is a newer open alternative for the same capability.

## What This Changes

The branch adds a top-level `tools.web.fetch.allowPrivateNetwork` boolean, threads it through `web_fetch` and SSRF resolution/cache keys, hard-blocks selected metadata endpoints, and updates config metadata, tests, and changelog.

## Best Possible Solution

Implement the feature under the current `tools.web.fetch.ssrfPolicy` namespace with generated config metadata, docs, cache partitioning, targeted SSRF tests, and an explicit maintained metadata-endpoint policy.

## Reproduction Assessment

Yes. Source inspection shows current main rejects the requested top-level key and only exposes narrow `ssrfPolicy` flags; the PR defect is source-reproducible from the branch diff because broad private mode skips link-local blocking while only hard-blocking selected metadata endpoints.

## Solution Assessment

No. The PR is useful source work, but the best current solution should use the existing `ssrfPolicy` namespace and resolve the metadata-endpoint policy before adding a broad private-network opt-in.
