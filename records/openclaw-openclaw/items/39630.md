## Summary

Keep open. Current main still does not expose the requested broad private-network opt-in for `web_fetch`, and the paired feature issue #39604 is still open. This PR is also not merge-ready as-is: it predates the current `fetchWithWebToolsNetworkGuard`/`tools.web.fetch.ssrfPolicy` shape and still carries security/product-policy risk around metadata endpoints, while newer PR #74790 is tracking a current-main-shaped implementation.

## What This Changes

The PR adds a top-level `tools.web.fetch.allowPrivateNetwork` config option, threads it through the older `web_fetch` SSRF resolver/cache/tests, adds hard-coded IMDS denylist handling, and updates config metadata plus changelog text.

## Best Possible Solution

Keep the feature tracked, but implement it against the current `tools.web.fetch.ssrfPolicy` surface rather than a new top-level key, with generated config metadata, docs, cache partitioning, targeted SSRF tests, and an explicit maintainer decision on whether broad private-network mode should also allow cloud metadata endpoints or use a documented hard/soft denylist.

## Reproduction Assessment

Yes. For current main, the missing feature is reproducible by schema/type/runtime inspection: the strict `web_fetch` config surface rejects the requested key and the runtime only passes the two narrow SSRF policy flags. For the PR finding, the PR head's broad allow path skips the generic link-local block while only hard-blocking `169.254.169.254`, leaving `169.254.170.2` reachable.

## Solution Assessment

No. This PR is useful source work, but it is not the best current solution because it uses an older top-level config shape and old fetch path; the maintainable path is to align with current `ssrfPolicy` naming and guarded-fetch plumbing, then resolve the metadata-endpoint policy explicitly.
