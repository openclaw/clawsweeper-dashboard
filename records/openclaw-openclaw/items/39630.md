## Summary

Keep open. The capability is still relevant and current main does not expose a broad private-network opt-in for arbitrary `web_fetch` URLs, but this branch is not merge-ready because it uses the disfavored top-level config shape, expands the SSRF boundary, omits documented container credential metadata endpoints, and still lacks contributor real behavior proof.

## What This Changes

The PR adds a default-off `tools.web.fetch.allowPrivateNetwork` setting, threads it through `web_fetch` SSRF/cache/config surfaces, adds SSRF tests, and updates the changelog.

## Best Possible Solution

Rework or replace this with one maintainer-approved nested `web_fetch` SSRF policy that preserves existing fake-IP flags/cache isolation, blocks approved metadata endpoints, updates docs/generated config/changelog, and includes real behavior proof.

## Reproduction Assessment

Yes, as source proof: current main's strict `web_fetch` schema lacks the requested broad private-network key and the runtime forwards only the two narrow fake-IP SSRF flags. I did not run a live private-network fetch in this read-only review.

## Solution Assessment

No, not as this branch stands. The capability is valid, but the top-level API shape and metadata-denylist policy are not the safest maintainable solution without maintainer security/API approval.
