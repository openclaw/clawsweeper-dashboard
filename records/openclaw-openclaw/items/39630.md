## Summary

Keep open: current main still lacks a broad private-network opt-in for arbitrary `web_fetch` URLs, but this stale/conflicting branch should not land as-is because the API shape, metadata endpoint policy, and real behavior proof are still unresolved.

## What This Changes

This PR adds a default-off top-level `tools.web.fetch.allowPrivateNetwork` config, threads it through `web_fetch` SSRF/cache handling, adds IMDS-focused tests, and updates config metadata, types, schema, and changelog.

## Best Possible Solution

Land or replace this with one maintainer-approved, default-off nested `web_fetch` SSRF policy contract that preserves fake-IP flags, blocks credential metadata endpoints, partitions cache keys by every policy bit, covers redirects, and ships with docs, tests, changelog, and redacted real private-target proof.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection on current main shows the strict `web_fetch` schema accepts only two narrow `ssrfPolicy` flags and runtime forwards only those flags.

## Solution Assessment

No, not as this branch stands. The safer path is a maintainer-approved nested SSRF policy shape with complete metadata endpoint policy, cache partitioning, docs, tests, changelog, and real behavior proof.
