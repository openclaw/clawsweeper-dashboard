## Summary

Keep open: current main and v2026.6.10 still resolve a single Google LLM API key before SSE setup, and https://github.com/openclaw/openclaw/pull/97328 is an open candidate fix rather than a merged implementation.

## What This Changes

The issue asks for Google Generative AI LLM streaming to rotate configured Gemini API keys on rate-limit setup failures instead of using only the first resolved key.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/97328 or an equivalent provider-owned pre-SSE rotation fix with credential-boundary tests, then close this issue after the fix merges.

## Reproduction Assessment

Yes at source level: current main resolves one Google LLM API key and opens SSE without the rotation helper, while adjacent provider paths already rotate on 429. I did not exercise live Gemini rate limiting with real keys.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing API-key rotation helper at the provider-owned pre-SSE request boundary, with tests for Vertex/OAuth/custom-endpoint exclusions.
