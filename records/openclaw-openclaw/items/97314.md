- `impact:auth-provider`: The issue concerns provider credential selection and API-key rotation for Google/Gemini model requests.

## Summary

Keep open: current main and the latest release still resolve a single Google/Gemini LLM API key before opening the SSE request, while https://github.com/openclaw/openclaw/pull/97328 is an open candidate fix rather than merged behavior.

## What This Changes

Issue requests that Google/Gemini LLM streaming requests rotate across configured Gemini API keys on rate-limit failures like the adjacent embedding path already does.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/97328; the desired implementation is provider-owned pre-SSE key rotation for official HTTPS Gemini LLM requests without adding new config surface.

## Reproduction Assessment

Yes, source-reproducible: current main resolves one Google LLM API key and never collects the configured rotation key set before opening the SSE request. I did not run a live Gemini 429 against real credentials in this read-only review.

## Solution Assessment

Yes for the issue boundary: the maintainable fix is to reuse the existing rotation helper at the pre-SSE request setup boundary while preserving Vertex, OAuth JSON, custom/proxy, and non-TLS exclusions. PR-specific acceptance should happen on the open candidate PR.
