## Summary

Keep open: current main and latest release still build Google/Gemini LLM stream requests with a single resolved API key, while the existing key-rotation helper is wired into adjacent Google embedding behavior and documented as the gateway contract.

## What This Changes

The issue reports that Google/Gemini LLM streaming uses only the first configured API key instead of rotating across configured GEMINI keys on rate-limit failures.

## Best Possible Solution

Wire the provider-owned Google LLM pre-stream request through the existing API-key rotation helper for rate-limit failures, with focused transport tests and no new config surface.

## Reproduction Assessment

Yes, at source level: current main resolves one Google LLM API key and never collects the configured rotation key set for that request path. I did not run a live 429 against real Gemini credentials.

## Solution Assessment

Yes, the best fix is to reuse the existing generic rotation helper at the pre-SSE request boundary. A custom rotator or new config knob would duplicate existing provider-auth behavior.

## Repair Work Prompt

Fix the Google direct LLM transport so rate-limit failures rotate across configured Google/Gemini API keys. In `extensions/google/transport-stream.ts`, reuse `collectProviderApiKeysForExecution` and `executeWithApiKeyRotation` from `openclaw/plugin-sdk/provider-auth-runtime` at the pre-SSE request boundary, rebuilding headers per candidate key and rotating only before any SSE chunk is emitted. Preserve Google Vertex ADC/OAuth JSON auth, custom headers, guarded fetch behavior, and Gemini first-response retry behavior. Do not add new config, new env vars, bucket cooldown, changelog entries, or a custom rotator. Add focused regression tests in `extensions/google/transport-stream.test.ts` for first-key 429 then second-key success and all-keys-fail behavior, and keep the existing helper/embedding tests passing.
