## Summary

Keep open: current main and the latest release still resolve one Google/Gemini LLM API key before opening the SSE stream, while an open proof-positive PR is now the active candidate fix rather than a reason to close the issue early.

## What This Changes

The issue asks for Google/Gemini LLM streaming requests to rotate configured API keys on 429/rate-limit responses like the adjacent embedding path already does.

## Best Possible Solution

Land the provider-owned pre-SSE rotation fix in https://github.com/openclaw/openclaw/pull/97328 with credential-boundary tests, then close this issue as implemented after the fix reaches main.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a single Google LLM API key and throws setup-time SSE HTTP errors without collecting or rotating the configured Google key set. I did not run a live Gemini 429 with real credentials in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable solution is to reuse the existing provider-auth rotation helper at the Google plugin's pre-SSE request boundary, while excluding Vertex/OAuth/custom endpoint cases that should not receive global Gemini env keys.
