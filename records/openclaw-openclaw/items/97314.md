## Summary

Keep open: current main and the latest release still send Google/Gemini LLM requests with one resolved API key, while the documented rotation contract and adjacent Google embedding path already use the shared rotation helper.

## What This Changes

The issue reports that Google/Gemini LLM requests use only the first configured API key instead of rotating across configured Gemini keys on rate-limit failures.

## Best Possible Solution

Wire the provider-owned Google LLM pre-stream request through the existing API-key rotation helper for rate-limit failures, with focused tests and no new config surface.

## Reproduction Assessment

Yes, source-reproducible: current main resolves one Google LLM API key and never collects the configured rotation key set for that request path. I did not run a live 429 against real Gemini credentials.

## Solution Assessment

Yes, the best fix is to reuse the existing generic rotation helper at the pre-SSE request boundary. A custom rotator or new config knob would duplicate existing provider-auth behavior.
