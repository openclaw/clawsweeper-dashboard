## Summary

Keep open. Current main still lacks the requested prompt/heartbeat race UX: the Control UI queues only when it owns a local chat run, other-run heartbeat deltas can still be ignored during a prompt-owned run, and backend follow-up queueing preserves execution order without a visible pending prompt/cancel-inject surface.

## What This Changes

Request visible handling for prompt/heartbeat races so heartbeat progress remains visible and the user prompt is shown as queued, cancellable, and injectable after the heartbeat reaches a safe boundary.

## Best Possible Solution

Implement an additive session-scoped active-run and pending-follow-up surface shared by Gateway chat/session events and Control UI. Heartbeat progress should keep rendering, the submitted prompt should remain visible as queued or pending with existing-style remove/steer controls, and backend follow-up draining should update that state when the heartbeat finishes or reaches the next safe tool boundary.

## Reproduction Assessment

Yes, at source level. With a heartbeat active but not represented as host.chatRunId, handleSendChat sends the prompt immediately; once chatRunId points at that prompt-owned run, heartbeat deltas with a different runId are ignored by handleChatEvent.

## Solution Assessment

Yes, the best direction is a generic session-scoped active-run/pending-follow-up contract rather than a heartbeat-specific special case or only changing isChatBusy. That keeps the fix aligned with existing queue/steer semantics and avoids plugin/provider-specific policy.
