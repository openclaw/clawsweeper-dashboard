## Summary

Keep open. Current main still has backend follow-up queueing, but the Control UI only treats its own chat run as busy, so a prompt submitted while a heartbeat is active can become a separate prompt-owned run and other-run heartbeat deltas remain visually hidden.

## What This Changes

The issue requests immediate, visible handling for prompt/heartbeat races so heartbeat progress is shown and the prompt remains queued or cancellable instead of appearing lost.

## Best Possible Solution

Add a generic session-scoped active-run and pending-follow-up UI contract so heartbeat progress can keep rendering while the submitted prompt remains visible with existing-style remove/steer controls until the backend drains it.

## Reproduction Assessment

Yes, at source level. With a heartbeat active only in session/backend state, isChatBusy remains false, handleSendChat starts a prompt-owned run, and handleChatEvent ignores non-final heartbeat deltas whose runId differs from that prompt run.

## Solution Assessment

Yes. The maintainable direction is a generic session-scoped active-run and pending-follow-up surface shared by Gateway events and Control UI, not a heartbeat-only special case or a config knob.
