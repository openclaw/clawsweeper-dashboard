## Summary

Keep open: the PR addresses a real media-diagnostics gap, but the current branch still duplicates all-media-dropped warnings and is missing required release-note hygiene for a user-visible fix.

## What This Changes

The PR adds a `DroppedMediaItem` pipeline that records failed `MEDIA:` normalization, preserves those entries through reply/outbound delivery, renders sanitized user-visible notices, mirrors them into transcripts, and adds focused tests.

## Best Possible Solution

Land a refined version that removes the legacy generic warning once structured dropped-media notices exist, keeps the sanitized single-notice behavior across delivery paths, updates focused tests, and adds the required changelog entry.

## Reproduction Assessment

Yes, at source level: current main and the PR branch expose the media-drop path clearly in `reply-media-paths.ts`, and the PR branch's all-dropped path carries both legacy warning text and structured dropped-media. I did not run a live channel reproduction because this review was read-only.

## Solution Assessment

No: the overall direction is maintainable, but the branch should emit one sanitized attachment-failure notice per dropped set rather than layering the old generic warning with the new detailed warning. A single changelog entry should accompany the user-visible change.
