## Summary

Keep open: current main still has a source-visible partial-media silent-drop path, but this PR should not merge yet because the patch can duplicate user warnings, bypasses `message_sending` policy hooks for the new notice text, and lacks real behavior proof.

## What This Changes

The PR adds `droppedMedia` metadata, reason-code/display-name helpers, outbound delivery notice rendering, routing/payload preservation, and regression tests for failed `MEDIA:` normalization.

## Best Possible Solution

Keep the focused dropped-media pipeline, but render exactly one sanitized warning per failed attachment and make that final warning text participate in the existing outbound hook policy before merge.

## Reproduction Assessment

Yes, by source inspection: current main catches failed media normalization and continues, and mixed success/failure media lists do not get a user-visible failure notice. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The direction is maintainable, but the current PR is not the best fix until it removes the legacy duplicate warning path and routes the final notice text through `message_sending` hooks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid duplicating media failure notices:** `src/auto-reply/reply/reply-media-paths.ts:240-246`
  - body: When every media item fails, this branch still appends the existing `⚠️ Media failed.` text and now also returns `droppedMedia`. Delivery later renders `droppedMedia` as an `Attachment not sent` notice, so users get two warnings for one failure. Drop the legacy text when `droppedMedia` will produce the notice, or render one unified notice.
  - confidence: 0.91
- **[P2] Run dropped-media notices through sending hooks:** `src/infra/outbound/deliver.ts:1487-1497`
  - body: `message_sending` hooks run before `droppedMediaNotice` is composed, then this branch appends or sends the warning afterward. Hooks documented as able to rewrite or cancel outbound content cannot inspect or block the new user-visible warning. Compose the notice before the hook or invoke the hook for the notice send.
  - confidence: 0.86
