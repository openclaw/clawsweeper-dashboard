## Summary

Keep this PR open. The dropped-media pipeline is a good direction for a real user-facing media failure, but the current patch can emit both the old generic warning and the new detailed warning for the same all-failed MEDIA payload, and the external contributor proof gate is still missing.

## What This Changes

The PR adds DroppedMedia tracking so failed MEDIA attachments are carried through reply normalization, routing, outbound delivery, JSON projection, logs, and transcript mirroring as user-visible attachment warnings.

## Best Possible Solution

Keep the dropped-media pipeline, remove the legacy generic warning when detailed droppedMedia notices will be emitted, then require a redacted real setup proof showing a failed MEDIA attachment produces the intended user-visible warning.

## Reproduction Assessment

Yes at source level: current main strips media on normalization failure without a per-file user notice, and the PR head has a deterministic all-failed MEDIA path that would emit duplicate warning text. I did not run a live channel proof in this read-only review.

## Solution Assessment

No as-is. The central DroppedMedia approach is maintainable, but it should not preserve the legacy generic warning alongside the new detailed notice, and the PR still needs real behavior proof before merge.
