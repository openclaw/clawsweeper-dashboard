## Summary

Keep open: the PR targets a real source-visible media-drop bug, but the latest diff is not merge-ready because it introduces unused imports that should fail the strict core typecheck and still lacks after-fix real behavior proof.

## What This Changes

Adds a dropped-media payload field and warning pipeline through reply normalization, outbound delivery, JSON projection, transcript mirroring, and tests so failed MEDIA attachments produce user-visible notices.

## Best Possible Solution

Keep the dropped-media pipeline direction, remove the compile blockers and duplicate all-failed warning behavior, then require redacted live channel proof showing the attachment-failure notice reaches users.

## Reproduction Assessment

Yes, from source inspection: a mixed MEDIA payload with one invalid and one valid media item hits the per-item catch, logs verbose, and returns only the surviving media without a per-item user notice. I did not execute a live repro in this read-only review.

## Solution Assessment

Mostly yes: carrying droppedMedia as structured payload metadata is a maintainable fix boundary for per-item failures across delivery and JSON paths. It is not merge-ready until the unused imports, duplicate all-failed notice, and real behavior proof gap are addressed.
