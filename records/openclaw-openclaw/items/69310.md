## Summary

Keep open: the PR addresses a valid source-reproducible dropped-media bug, but the current patch would double-report all-failed MEDIA payloads and still lacks required real behavior proof.

## What This Changes

The PR adds dropped-media tracking and user-visible attachment failure notices across reply media normalization, routing, outbound delivery, JSON projection, tests, and transcript mirroring.

## Best Possible Solution

Keep the dropped-media pipeline approach, remove the legacy generic all-failed warning once structured dropped-media notices are emitted, and add real setup proof showing the user-visible warning in a channel path such as Telegram.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current main still silently loses partial MEDIA failures and catastrophic normalization failures, while all-failed paths only show a generic warning. The affected code paths are in `reply-media-paths.ts` and `agent-runner-payloads.ts`.

## Solution Assessment

Mostly yes, but not as currently patched: carrying sanitized dropped-media metadata through delivery is the right maintainable direction, while retaining the older generic `Media failed` text creates duplicate user warnings. The safer fix is to let the new dropped-media notice be the single user-facing warning for those failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid double-warning all-failed media payloads:** `src/auto-reply/reply/reply-media-paths.ts:240-243`
  - body: When every media item fails, this branch still appends the legacy `Media failed` text while also returning `droppedMedia`. Delivery later formats `droppedMedia` into the new `Attachment not sent` notice, so users get two failure warnings for the same attachment. Drop the legacy text once `droppedMedia` is populated, or suppress the later notice for this branch.
