## Summary

Keep open: the silent dropped-media problem is source-reproducible on current main and this PR is directionally useful, but it still has a duplicate-warning regression and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `droppedMedia` pipeline through reply media normalization, outbound planning and delivery, routing, JSON projection, transcript mirroring, warning logs, and regression tests.

## Best Possible Solution

Keep the dropped-media pipeline, remove the legacy generic warning when structured drop details are present, and add redacted real output proof from an all-failed and ideally mixed-success media run.

## Reproduction Assessment

Yes by source inspection: a payload with one invalid `MEDIA:` item and one valid media item takes the catch-and-continue path on current main, then returns surviving media without notifying the user about the dropped item. I did not run a live channel repro in this read-only review.

## Solution Assessment

No, not yet: carrying dropped-media details through delivery is the right shape, but the PR should avoid combining the legacy `⚠️ Media failed.` text with the new detailed notice and should include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drop the legacy all-media warning when tracking drops:** `src/auto-reply/reply/reply-media-paths.ts:240-246`
  - body: When every media source fails, this branch still appends the legacy `⚠️ Media failed.` text while also preserving `droppedMedia`. `deliver.ts` later formats and appends the detailed dropped-media notice for no-media payloads, so users get two warnings for the same failed attachment. Return only the tracked drop details, or suppress the later notice, so all-failed `MEDIA:` replies show one actionable warning.
