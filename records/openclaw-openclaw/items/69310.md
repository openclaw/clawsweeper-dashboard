## Summary

Keep open: the PR is a useful bug-fix direction, but the current head does not typecheck, regresses the existing silent-reply contract, and still lacks after-fix real behavior proof from the external contributor.

## What This Changes

The PR threads dropped-media diagnostics through reply and outbound payload handling, logs sanitized media normalization failures, emits user-visible attachment-failure notices, mirrors them to transcripts, and adds focused tests.

## Best Possible Solution

Keep the dropped-media diagnostic pipeline, but fix the wrapper access, preserve silent-turn suppression, keep typecheck/tests green, and add redacted live proof that users see the warning without leaking paths or tokens.

## Reproduction Assessment

Yes for the PR blockers: CI reproduces both the `deliver.ts` type error and the failing silent-media regression test on the PR merge commit. I did not run local tests because this review is read-only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Read dropped media from the payload wrapper:** `src/infra/outbound/deliver.ts:1908`
  - body: `normalizedPayloads` entries are shaped as `{ index, payload }`, so reading `p.droppedMedia` here does not typecheck and the PR currently fails CI. Use the nested payload (`p.payload.droppedMedia`) or reshape the loop before collecting mirror notices.
  - confidence: 0.96
- **[P2] Keep silent media failures silent:** `src/auto-reply/reply/reply-payloads-base.ts:85`
  - body: Making every `droppedMedia` payload renderable lets `NO_REPLY` media failures survive after `agent-runner-payloads` strips the silent text, which breaks the existing test that suppresses warning text for silent media-normalization failures. Gate or clear `droppedMedia` for parsed silent replies that no longer have sendable media.
  - confidence: 0.9
