## Summary

Keep open: current main only has a coarse all-media-failed warning, while this PR adds the fuller dropped-media pipeline; however the PR still duplicates warnings when every media item fails and is missing the required changelog entry.

## What This Changes

The PR threads a new `droppedMedia` payload field through reply media normalization, routing, outbound delivery, JSON projection, and tests so failed `MEDIA:` attachments produce sanitized user-visible notices.

## Best Possible Solution

Keep the dropped-media pipeline, but make one path responsible for user text so all-media-failed replies show only the detailed attachment notice, then add focused coverage and a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes for the PR defect: on the PR branch, a payload whose every `MEDIA:` item fails normalization keeps the legacy generic warning and also carries `droppedMedia`, so delivery inlines a second detailed notice. The original user bug is also source-backed by current main’s lack of mixed/per-item dropped-media propagation.

## Solution Assessment

No, not as currently written: the overall pipeline is the right maintainable direction, but it should remove or suppress the legacy generic warning when `droppedMedia` will render the detailed notice. A changelog entry is also required before merge.
