## Summary

Keep open. The PR addresses a real remaining gap because current main only warns when all reply media is dropped, but this branch should not merge yet: it can double-report all-dropped media and it lacks required after-fix real behavior proof.

## What This Changes

The PR adds a DroppedMedia pipeline that carries sanitized attachment failure details through reply normalization, routing, outbound delivery, transcript mirroring, and JSON projections.

## Best Possible Solution

Land a single-warning dropped-media implementation that reports every failed attachment, including partial drops, while preserving path/privacy redaction and current channel safety boundaries.

## Reproduction Assessment

Yes, from source inspection: current main catches failed media items and only emits `⚠️ Media failed.` when all media are dropped, so a mixed valid-plus-invalid MEDIA reply still hides the dropped item. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The richer DroppedMedia pipeline is a reasonable direction, but this branch should first replace the existing generic all-dropped warning instead of appending a second warning, then show real after-fix behavior proof.
