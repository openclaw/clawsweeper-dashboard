## Summary

Keep this PR open. Current main still has media-drop visibility gaps, and the branch is directionally useful, but the latest head still misses one actual delivery path: payloads that contain only `droppedMedia` can be filtered out before the notice is formatted. I also found a duplicate-warning case and a missing required changelog entry.

## What This Changes

The PR adds a `DroppedMediaItem` pipeline for MEDIA normalization failures, formats user/transcript notices, carries dropped-media metadata through reply/outbound planning, redacts warning logs, and adds focused tests.

## Best Possible Solution

Keep the contributor PR active, update the final delivery normalization so `droppedMedia` is treated as renderable content all the way to notice formatting, remove the legacy duplicate generic warning for all-failed media, add the required changelog entry, and then validate the focused reply/outbound tests plus the changed gate.

## Reproduction Assessment

Yes. A media-only payload whose `normalizeMediaPaths` throws can be traced on the PR head into a `{ droppedMedia }`-only payload, then through delivery’s empty-payload guard where it is filtered before `formatDroppedMediaNotice` can run.

## Solution Assessment

No. The overall design is the right narrow direction, but the current patch is incomplete until the final delivery path preserves dropped-media-only payloads and covers that case with an integration-style delivery test.
