## Summary

Keep open: the underlying media-drop bug is real on current main, and this PR is aimed at the right surface, but the latest diff has blocking typecheck/behavior defects and lacks after-fix real behavior proof.

## What This Changes

The PR threads dropped-media metadata through reply normalization, routing, outbound delivery, JSON projection, transcript mirroring, and tests so failed MEDIA attachments become visible warnings.

## Best Possible Solution

Keep the dropped-media pipeline direction, but fix the typecheck blockers, collect mirror drops from `entry.payload.droppedMedia`, remove the legacy duplicate warning, and require a redacted real-run proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main drops per-item media failures with only verbose logging when another media item survives, and the catastrophic normalizer catch clears media after only verbose logging.

## Solution Assessment

No, not in the latest PR shape. The approach is maintainable, but this diff must fix the type errors, mirror collection bug, duplicate warning, and real-proof gap before it is the best solution.
