## Summary

## What This Changes

The PR adds a dropped-media tracking pipeline so failed `MEDIA:` attachments are logged, preserved through reply/outbound payloads, rendered as user-visible notices, mirrored to transcripts, and covered by new tests.

## Best Possible Solution

Keep the dropped-media pipeline direction, remove the legacy duplicate warning and unused imports, add the required changelog entry, and require redacted real behavior proof from a real media-drop send before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main catches failed media normalization and either continues without an item-level user notice or strips media in the catastrophic fallback. I did not run a live channel send in this read-only review.

## Solution Assessment

No, the PR is the right maintainable direction but the current patch is not the best final form because it can render duplicate warnings and leaves unused imports that should fail lint; the safer fix is to make `droppedMedia` the single warning path and keep the delivery imports minimal.
