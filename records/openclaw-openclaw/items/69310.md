## Summary

Keep this PR open: the direction is useful and overlaps a real media-drop bug, but the current head has blocking type/test failures and lacks required after-fix real behavior proof.

## What This Changes

This PR adds a dropped-media pipeline that records failed MEDIA normalization, carries droppedMedia through reply/outbound delivery, formats user-visible attachment warnings, mirrors them to transcripts, and adds regression tests.

## Best Possible Solution

Keep the richer dropped-media design only after it compiles, preserves NO_REPLY/silent-turn behavior, and includes redacted real OpenClaw proof that a failed MEDIA path is surfaced to the user.

## Reproduction Assessment

Yes for the PR defects: CI reproduces the TS2339 type error and the silent-media regression on the current PR head. The original dropped-media problem is also source-reproducible from current main's partial-drop and normalization-catch paths, though I did not run a live channel scenario in this read-only review.

## Solution Assessment

No, not in its current form. The overall direction is maintainable, but the implementation must fix the normalized payload access, preserve the existing NO_REPLY contract, and provide real behavior proof before it is the best solution.

## Review Findings

Overall correctness: patch is incorrect

  - confidence: 0.92
