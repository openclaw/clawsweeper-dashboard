## Summary

Keep open: this PR addresses a real current-main `MEDIA:` failure visibility gap, but the head still duplicates warnings for all-media-failed payloads and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `DroppedMediaItem` pipeline that classifies failed `MEDIA:` normalization, preserves dropped entries through reply/outbound payloads, and emits sanitized attachment-failure notices in delivery and transcript mirror paths.

## Best Possible Solution

Land a version that carries only the detailed dropped-media notice, preserves sanitized diagnostics across direct/routed/follow-up delivery, and includes focused regression tests plus real channel or CLI proof.

## Reproduction Assessment

Yes, source inspection provides a high-confidence path: current main documents `MEDIA:` delivery, while failed normalization paths either log verbosely and continue or clear media without a detailed user-visible notice. I did not run a live channel repro in this read-only review.

## Solution Assessment

No, not as-is. Tracking `droppedMedia` is the right narrow direction, but the branch should remove the legacy generic warning on dropped-media payloads and add real behavior proof before merge.
