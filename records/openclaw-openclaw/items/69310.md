## Summary

Keep this PR open: the dropped-media pipeline is a plausible fix for a real source-reproducible bug, but the current head still produces duplicate user warnings when every MEDIA item fails and it still lacks required after-fix real behavior proof.

## What This Changes

The PR adds a DroppedMedia pipeline across reply media normalization, outbound payload planning/delivery, JSON projection, transcript mirroring, and tests so failed MEDIA attachments produce sanitized warnings.

## Best Possible Solution

Use droppedMedia as the single source for attachment-failure notices, remove the legacy generic warning on the all-dropped path, add a focused regression for that case, and attach redacted real send proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main catches and continues past failed MEDIA normalization, and mixed valid/invalid media has no per-item user notice. I did not run a live channel reproduction.

## Solution Assessment

No. The droppedMedia pipeline is the right direction, but this implementation should not also append the legacy generic media-failed warning when droppedMedia will produce the detailed notice.
