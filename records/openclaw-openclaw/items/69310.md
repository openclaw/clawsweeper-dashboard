## Summary

Keep this PR open: the dropped-media direction addresses a real media-loss gap, but the current head is not mergeable because it fails type/lint checks, regresses silent reply behavior, can emit duplicate warnings, and has no after-fix real behavior proof.

## What This Changes

The PR adds a dropped-media payload pipeline that sanitizes failed MEDIA sources, carries reason codes through auto-reply and outbound delivery, emits user-visible notices, mirrors them to transcripts, and adds regression tests.

## Best Possible Solution

Keep the dropped-media pipeline direction, but make it type-clean, preserve silent-reply policy, remove duplicate legacy warning text, and require redacted real behavior proof showing the final warning output without path or token leakage.

## Reproduction Assessment

Yes at source level: current main still logs and continues on per-item media normalization failures, so a mixed MEDIA reply can lose an attachment without a per-file notice. I did not run a live channel path in this read-only review, but CI provides high-confidence reproduction for the PR regressions.

## Solution Assessment

No for this branch as-is: the droppedMedia pipeline is a maintainable direction, but it must typecheck, use the normalized payload entry correctly, respect silent reply tests, and avoid duplicate warning text before it is the best fix.
