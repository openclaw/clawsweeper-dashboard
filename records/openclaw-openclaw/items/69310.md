## Summary

Keep open: this PR targets a real source-visible MEDIA drop path, but the current head still double-renders the all-media-failed warning and the external PR lacks required real behavior proof.

## What This Changes

The PR adds a DroppedMediaItem pipeline from reply media normalization through outbound delivery, JSON projection, routing, transcript mirroring, warning logs, and focused tests.

## Best Possible Solution

Keep the dropped-media pipeline, remove the legacy generic warning when the detailed droppedMedia notice will render, and add redacted real runtime proof for an affected CLI or channel path.

## Reproduction Assessment

Yes, from source inspection: current main catches failed MEDIA normalization and continues, so a mixed valid-plus-invalid MEDIA reply can lose the failed attachment without user-visible feedback. I did not run a live channel scenario in this read-only review.

## Solution Assessment

No, not quite yet: the proposed DroppedMedia pipeline is the right narrow direction, but it should not render both the old generic warning and the new detailed notice for the same all-failed payload.
