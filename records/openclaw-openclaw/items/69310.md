## Summary

Keep this PR open: the core direction is useful, but the current branch still has a user-visible duplicate-warning bug, lacks required real behavior proof, and needs a changelog entry before merge.

## What This Changes

The PR adds DroppedMedia tracking for reply media normalization failures, carries it through reply/outbound routing, formats user-visible attachment-not-sent notices, updates JSON/mirror paths, and adds dropped-media tests.

## Best Possible Solution

Keep the droppedMedia pipeline, but remove the legacy coarse warning when a structured droppedMedia notice will be delivered, add the required changelog entry, and have the contributor attach after-fix real behavior proof.

## Reproduction Assessment

Yes, source-reproducible by inspection: a mixed valid-plus-failing MEDIA list on current main exercises the catch-and-continue path without notifying the user about the failed item. I did not live-run the channel workflow in this read-only review.

## Solution Assessment

No, not in the current PR form. The droppedMedia pipeline is the right narrow direction, but it should be the sole user-facing notice for all-failed media payloads instead of stacking on the legacy `Media failed` text.
