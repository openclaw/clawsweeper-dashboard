## Summary

Keep open. Current main still does not fully solve dropped-media visibility, and the latest PR head mostly implements the requested pipeline, but it has a concrete duplicate-warning bug in the all-media-dropped path and is missing the required user-facing changelog entry. The earlier review issue around the stale `reason` field appears fixed in the latest tests.

## What This Changes

The PR adds a `DroppedMediaItem` payload contract and delivery pipeline that records media normalization failures, formats user/transcript notices, preserves dropped-media-only payloads through routing/outbound projections, upgrades warning logs, and adds focused tests.

## Best Possible Solution

Land the dropped-media pipeline after it emits exactly one user-facing notice for each dropped-media event, keeps sanitized display names and transcript/JSON preservation, adds regression coverage for the all-failed case, and includes the required Unreleased changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: on the PR head, a payload whose only `MEDIA:` item fails normalization returns both legacy warning text and `droppedMedia`, then delivery inlines the formatted dropped-media notice for the no-media payload.

## Solution Assessment

No. The proposed direction is the right narrow fix for the silent-drop bug, but the latest implementation should remove the legacy generic warning from the `droppedMedia` path and add the missing changelog before merge.
