## What This Changes

The PR adds dropped-media tracking, sanitized warning text, routing/payload preservation, transcript mirror handling, and regression tests for failed MEDIA attachment normalization.

## Best Possible Solution

Land a version that emits exactly one sanitized dropped-media notice for each failed attachment, includes the required changelog entry, and is backed by real runtime proof showing the delivered warning.

## Reproduction Assessment

Yes, by source inspection: current main still drops per-item media failures after a verbose log and clears all media in the catastrophic fallback without a durable user-visible drop record. I did not run a live channel scenario in this read-only review.

## Solution Assessment

No, not yet: the droppedMedia pipeline is the right direction, but the all-media-failed path still carries the legacy generic warning and will get the new detailed warning appended later. The narrow fix is to remove the legacy `formatBlockedReplyMediaWarning()` text when droppedMedia is present and update final-delivery coverage.
