## Summary

Keep open: the PR addresses a real media-drop visibility bug that current main still only partially handles, but the proposed patch has blocking delivery semantics issues around outbound hooks and transcript mirroring before it should merge.

## What This Changes

The PR threads dropped-media metadata through reply normalization and outbound delivery, formats user-visible attachment-failure notices, and adds tests for sanitization and preservation.

## Best Possible Solution

Land a revised PR that emits one sanitized failed-attachment notice, keeps outbound hook and transcript semantics accurate, and documents the user-facing fix in the changelog.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main catches a failed media item, logs and continues, so mixed valid/invalid media sends can still hide the failed attachment from users. I did not execute a live reproduction because this review is read-only.

## Solution Assessment

No: the droppedMedia pipeline is the right direction, but the current patch appends notices after outbound hooks, mirrors notices that may not have been delivered, and keeps the legacy generic warning in all-failed cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Run drop notices through outbound hooks:** `src/infra/outbound/deliver.ts:1146-1164`
  - body: The dropped-media notice is built after `message_sending` has already run, then appended to text or sent with `sendTextChunks`. Hook plugins documented to rewrite or cancel outbound content never see the final user-visible warning, so compliance/redaction hooks can be bypassed by filename diagnostics. Compose the notice before the hook or run the hook for the generated notice message too.
  - confidence: 0.86
- **[P2] Keep transcript mirrors aligned with delivered notices:** `src/infra/outbound/deliver.ts:1412-1414`
  - body: This collects `droppedMedia` from every normalized payload and writes a mirror even when `results` is empty. If a hook cancels delivery or the payload fails before any notice is sent, the session transcript records an attachment warning that the user never saw. Track notice delivery or delivered payload state instead of mirroring all drops unconditionally.
  - confidence: 0.8
- **[P3] Remove the legacy media-failed text when carrying drops:** `src/auto-reply/reply/reply-media-paths.ts:240-243`
  - body: When all media fail, this branch still injects `⚠️ Media failed.` while also preserving `droppedMedia`, so delivery later adds the formatted `Attachment not sent` notice and users get two warnings for one failure. Drop the legacy text once `droppedMedia` is used for the visible notice.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/infra/outbound/deliver.ts:85-96`
  - body: This changes user-facing failed-media output, but the PR does not update `CHANGELOG.md`. Add a single-line Unreleased/Fixes entry for the new attachment failure notice behavior.
  - confidence: 0.95
