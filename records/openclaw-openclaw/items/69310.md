## Summary

Keep open: the PR is a useful focused bug fix, but the current head fails prod typecheck, leaks silent NO_REPLY media failures into visible replies, can emit duplicate all-dropped media warnings, and still lacks required real behavior proof for the visible chat change.

## What This Changes

The PR adds a DroppedMediaItem pipeline for failed MEDIA normalization, formats user/transcript notices, preserves dropped-media-only payloads through routing/outbound projections, upgrades warning logs, and adds tests.

## Best Possible Solution

Keep the dropped-media direction, but fix the typecheck blockers, preserve silent NO_REPLY semantics, let one layer own the user-facing notice text, and add redacted real Telegram or CLI proof before maintainer merge.

## Reproduction Assessment

Yes for the review blockers: CI and source inspection show the TypeScript failure, silent NO_REPLY regression, and duplicate-notice path on the current PR head. The original silent-drop issue is partially addressed on current main by a generic warning, but this branch still needs correctness work for the richer notice pipeline.

## Solution Assessment

No; the overall direction is maintainable, but the current implementation is not the best mergeable form because it breaks prod typecheck, preserves droppedMedia across silent replies, and layers the new notice over the legacy warning. The safer fix is to route droppedMedia through the existing payload wrapper correctly and make delivery own exactly one visible notice.
