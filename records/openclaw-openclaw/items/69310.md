## What This Changes

The PR adds a dropped-media payload pipeline that records failed reply media normalization, formats user-visible attachment warnings, preserves those payloads through routing/outbound projections, and adds tests.

## Best Possible Solution

Keep the broader dropped-media pipeline under review, fix the type and silent-reply blockers, resolve the hook-policy question, then require real after-fix runtime proof before merge.

## Reproduction Assessment

Yes for the PR blockers: CI reproduces both the typecheck failure and the silent-reply regression on the current PR head. The original user-facing media-drop behavior is also source-reproducible from current main and partially covered by the existing all-dropped warning path.

## Solution Assessment

No, not as currently implemented: the direction is plausible, but the patch must read `droppedMedia` from normalized payload entries correctly and preserve or intentionally redesign silent-reply policy before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Read dropped media from payload entries:** `src/infra/outbound/deliver.ts:1908`
  - body: `normalizedPayloads` contains `{ index, payload }` entries, so reading `p.droppedMedia` fails the test typecheck lane with TS2339. Use `p.payload.droppedMedia` here before this can build.
  - confidence: 0.97
- **[P2] Preserve silent replies when media normalization fails:** `src/auto-reply/reply/agent-runner-payloads.ts:58-60`
  - body: The catch path now leaves `droppedMedia` on a parsed `NO_REPLY` payload after the text is cleared, and `isRenderablePayload` keeps that warning payload alive. CI's existing silent-media test expects zero payloads, so this changes silent reply behavior without an explicit policy update.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff avoids obvious supply-chain changes, but the new generated warning messages can bypass `message_sending` rewrite/cancel hooks and need maintainer policy review.

Concerns:

- **[medium] Dropped-media notices bypass send policy hooks:** `src/infra/outbound/deliver.ts:1676`
  - body: The primary payload runs through `applyMessageSendingHook`, but the later `sendTextChunks(droppedMediaNotice, ...)` calls send a new user-visible message without passing that notice through the same hook. Even with sanitized filenames, this can bypass compliance, moderation, or cancellation hooks that expect to govern outbound text.
