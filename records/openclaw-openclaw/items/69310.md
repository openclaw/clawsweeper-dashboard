## Summary

Keep this PR open: the bug is real and the DroppedMedia direction is useful, but the latest head is still red in CI, still emits dropped-media payloads during silent replies, and still lacks required real behavior proof from an external contributor.

## What This Changes

The PR adds dropped-media tracking through reply normalization, routing, outbound delivery, JSON projection, warning logs, and transcript mirroring so failed `MEDIA:` attachments can produce user-visible warnings.

## Best Possible Solution

Keep the DroppedMedia pipeline, fix the build/type and silent-reply regressions, avoid duplicate legacy plus detailed warnings, then require redacted real setup proof showing a failed MEDIA attachment produces exactly the intended user-visible warning.

## Reproduction Assessment

Yes for the patch defects: latest CI reproduces the TypeScript error and silent-turn test failure, and source inspection shows current main still drops or strips failed media without detailed per-file user feedback. I did not run a live channel proof in this read-only review.

## Solution Assessment

No as-is. The DroppedMedia approach is a maintainable fix direction, but the current branch is not mergeable until CI is green, silent `NO_REPLY` media failures remain silent, duplicate warnings are removed, and real behavior proof is added.
