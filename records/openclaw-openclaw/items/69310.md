## Summary

Keep open. The PR targets a real dropped-media visibility bug, but it is not merge-ready because source inspection shows an all-media-failed reply would send both the legacy generic warning and the new detailed notice, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a `droppedMedia` payload pipeline, sanitized display names, user-facing attachment-failure notices, transcript mirroring, and focused tests for failed `MEDIA:` normalization.

## Best Possible Solution

Use one canonical dropped-media notice generated from `droppedMedia`, remove the legacy all-failed text injection, and validate the visible warning in a real channel run.

## Reproduction Assessment

Yes, by source inspection: a payload whose media all fail normalization leaves `⚠️ Media failed.` in `text` and also carries `droppedMedia`, which delivery appends as the detailed attachment warning.

## Solution Assessment

No. The new `droppedMedia` pipeline is the right boundary, but it should replace the old generic warning for all-failed media instead of sending both messages.
