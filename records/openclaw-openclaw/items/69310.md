## Summary

Keep this PR open: it targets a real source-backed dropped-media visibility gap, but the patch currently duplicates user-facing warnings on the all-media-failed path and still needs external real behavior proof before merge.

## What This Changes

The PR threads dropped-media metadata through reply media normalization, outbound payload planning, routing, delivery, JSON projection, transcript mirroring, and tests so failed `MEDIA:` attachments can be shown to users.

## Best Possible Solution

Land a version that makes the new sanitized dropped-media notice the single user-visible diagnostic, preserves the pipeline coverage, adds the required changelog entry, and includes real OpenClaw proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a payload with one blocked `MEDIA:` source and one surviving media URL still hides the failed item on current main, while the all-failed path only emits a generic warning. I did not run a live channel scenario in this read-only review.

## Solution Assessment

No, not as currently patched. The direction is maintainable, but the branch should remove the legacy `⚠️ Media failed.` text when `droppedMedia` will render the detailed notice, then add changelog and real behavior proof.
