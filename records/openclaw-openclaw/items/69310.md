## Summary

Keep open. The PR addresses a real dropped-media visibility bug, but the current head still has a blocking transcript-mirror mistake, a duplicated all-media-failed warning path, and no after-fix real behavior proof for the visible Telegram/media behavior.

## What This Changes

The branch adds a DroppedMedia pipeline for failed MEDIA normalization, formats user/transcript notices, preserves dropped-media-only payloads through routing/outbound projections, upgrades warning logs, and adds focused tests.

## Best Possible Solution

Keep the DroppedMedia direction, fix the wrapper access and duplicate warning behavior, then require live redacted proof that a failed MEDIA directive produces exactly one visible warning in a real channel.

## Reproduction Assessment

Yes for the review findings: source inspection shows the wrapper shape mismatch in deliver.ts and the duplicate all-failed warning path without needing a live run. The linked user-visible media-drop behavior is also source-reproducible, but the PR still needs real after-fix channel proof.

## Solution Assessment

No, not yet. The overall pipeline is a maintainable direction, but this head should read dropped media from entry.payload, avoid emitting the legacy generic warning alongside the new notice, and add live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Read droppedMedia from the normalized payload:** `src/infra/outbound/deliver.ts:1913`
  - body: `normalizedPayloads` entries are `{ index, payload }`, so this reads a non-existent `droppedMedia` property on the wrapper. That should fail type checking, and if it were bypassed the transcript mirror would never record the dropped-media notice; use `p.payload.droppedMedia` here.
  - confidence: 0.93
- **[P2] Drop the legacy warning when recording dropped media:** `src/auto-reply/reply/reply-media-paths.ts:240-246`
  - body: When every media item fails, this branch still appends the legacy `Media failed` text and also sets `droppedMedia`. The delivery layer then inlines the new per-file notice for no-media payloads, so users see two warnings for one failure; let the dropped-media notice be the single rendered warning.
  - confidence: 0.84
