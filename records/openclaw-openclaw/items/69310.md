## Summary

Keep open: this targets a real source-proven media-drop path, but the current PR still has blocking delivery-hook/security concerns, duplicate warning behavior, stale imports, no changelog entry, and missing external real behavior proof.

## What This Changes

This PR threads `DroppedMediaItem` through reply media normalization, outbound payload projections, delivery, transcript mirroring, and tests so failed `MEDIA:` attachments produce user-visible warnings.

## Best Possible Solution

Keep the structured dropped-media pipeline, but compose warning text through the normal outbound hook/delivery path, remove duplicate legacy warning text and stale imports, add the required changelog entry, and attach redacted real channel proof.

## Reproduction Assessment

Yes from source inspection, not live execution: current main has clear catch paths that continue after failed media normalization or strip media entirely in fallback. I did not establish an end-to-end failing channel run in this read-only review.

## Solution Assessment

No; the proposed data model direction is plausible, but sending generated notices after `message_sending`, retaining the legacy generic warning, and leaving stale imports is not the narrowest maintainable merge shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Run drop notices through message_sending hooks:** `src/infra/outbound/deliver.ts:1555`
  - body: The new dropped-media notice is user-visible outbound text, but this path sends it directly with `sendTextChunks` after `applyMessageSendingHook` has already run. That bypasses the documented hook that can rewrite or cancel outbound content, so moderation/compliance hooks cannot see or stop the final warning text; compose the notice before the hook or route the notice through the same hook pipeline as normal text.
  - confidence: 0.87

  - body: When all media fail, the normalizer still injects the old `⚠️ Media failed.` text while also setting `droppedMedia`. Delivery later inlines `formatDroppedMediaNotice` for a dropped-media-only payload, so users can receive both the generic warning and the new detailed warning in the same reply, contrary to the PR’s documented output.
  - confidence: 0.84

  - body: The final diff imports `sendMediaWithLeadingCaption`, `chunkByParagraph`, and `chunkMarkdownTextWithMode`, but none of them are referenced in `deliver.ts`. The core lint lane uses oxlint, so these stale imports should be removed before merge.
  - confidence: 0.8
