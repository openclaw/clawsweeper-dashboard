## Summary

Keep open. The user problem is valid, but this head still has blocking issues: silent `NO_REPLY` media failures now survive as visible dropped-media payloads, all-failed media can show both the old generic warning and the new detailed warning, and there is no after-fix real behavior proof.

## What This Changes

The PR adds dropped-media tracking and user-visible attachment-failure notices across reply media normalization, outbound delivery, transcript mirroring, JSON projection, routing, and tests.

## Best Possible Solution

Land a corrected version that carries sanitized dropped-media details through reply/outbound paths, honors silent responses, emits one clear warning per dropped-media event, and includes real Telegram-visible proof before merge.

## Reproduction Assessment

Yes. Source inspection plus the live check annotation gives a high-confidence path: `NO_REPLY\nMEDIA: ./missing.png` with normalization failure now leaves a dropped-media-only payload instead of being suppressed.

## Solution Assessment

No. The dropped-media pipeline is the right direction, but this branch must preserve silent/no-reply semantics, remove the old generic all-failed warning when the detailed notice is used, and clean up the unused import before merge.
