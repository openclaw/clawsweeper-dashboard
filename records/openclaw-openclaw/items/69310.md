## Summary

Keep this PR open: it is a plausible bug-fix branch with meaningful unique structured dropped-media work, but it is not mergeable because the current patch fails typecheck, can duplicate user warnings on the all-media-dropped path, and lacks required real behavior proof.

## What This Changes

The PR adds a `DroppedMediaItem` pipeline from reply media normalization through routing, outbound delivery, JSON projection, logs, transcript mirroring, and tests so dropped `MEDIA:` attachments become visible warnings.

## Best Possible Solution

Keep the PR open only if maintainers want the richer structured dropped-media contract; before merge, make it compile, replace rather than duplicate the existing generic warning, and add real channel proof.

## Reproduction Assessment

Yes for the patch defects: source inspection and the latest `check-prod-types` log show a deterministic compile failure on the PR branch. The original silent-drop symptom is partly superseded by current-main source and tests that now emit a generic media-failure warning.

## Solution Assessment

No. The structured dropped-media design is plausible, but this patch is not yet the best implementation because it fails typecheck, can double-notify users, and has not been proven in a real channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Read dropped media from the wrapped payload:** `src/infra/outbound/deliver.ts:1921`
  - body: `normalizedPayloads` stores entries as `{ index, payload }`, so `p.droppedMedia` does not exist and the branch fails `check-prod-types` with TS2339. Read `p.payload.droppedMedia` here so transcript mirroring compiles and actually sees the normalized payload's dropped-media entries.
  - confidence: 0.98
- **[P1] Remove the unused imports that break typecheck:** `src/infra/outbound/deliver.ts:1-4`
  - body: The PR adds imports that are never used (`sendMediaWithLeadingCaption`, `chunkByParagraph`, `chunkMarkdownTextWithMode`), and CI reports TS6133 for them. Drop these imports or use the existing helpers so the production type check can pass.
  - confidence: 0.96
- **[P1] Remove the unused config-dir import:** `src/auto-reply/reply/reply-media-paths.ts:18`
  - body: `resolveConfigDir` is imported but never read, which is another TS6133 failure in the latest `check-prod-types` job. Removing it is required before the branch can merge.
  - confidence: 0.95
- **[P2] Avoid emitting both media-failure warnings:** `src/auto-reply/reply/reply-media-paths.ts:240-246`
  - body: When all media fail, this branch appends the existing `⚠️ Media failed.` text and also carries `droppedMedia`; outbound delivery then formats `droppedMedia` into the new detailed notice. That produces two warnings for one failed attachment, so this path should either set `droppedMedia` without adding the legacy text or suppress the later detailed notice for already-rendered warnings.
  - confidence: 0.88
