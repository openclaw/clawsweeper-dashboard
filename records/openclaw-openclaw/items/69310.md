## Summary

Keep open: the bug-fix direction is useful, but the current PR head is not merge-ready because it has deterministic typecheck blockers, a broken transcript-mirror collection path, duplicate all-failed media warnings, and no real behavior proof.

## What This Changes

The PR adds a DroppedMediaItem pipeline for failed MEDIA normalization, emits user and transcript notices, preserves dropped-media payloads through reply/outbound routing and JSON projections, upgrades warning logs, and adds tests.

## Best Possible Solution

Keep the structured droppedMedia pipeline direction, fix the typecheck and mirror/double-warning blockers, then require redacted live channel proof before merge.

## Reproduction Assessment

Yes. Current main source shows a mixed MEDIA payload can hit the per-item catch, log verbosely, and continue with only the surviving media, while the catastrophic catch strips all media without a structured user-visible failure marker.

## Solution Assessment

Mostly yes. Carrying droppedMedia as structured payload metadata is a maintainable fix boundary, but this branch is not merge-ready until the typecheck blockers, mirror collection bug, duplicate legacy warning, and real behavior proof gap are addressed.
