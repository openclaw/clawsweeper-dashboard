## Summary

Keep this PR open: the overall dropped-media pipeline addresses a real source-reproducible bug, but the current head can still show duplicate warnings on the all-media-dropped path and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds dropped-media tracking through reply normalization, routing, outbound delivery, JSON projection, logs, transcript mirroring, and tests so failed `MEDIA:` attachments produce user-visible warnings.

## Best Possible Solution

Keep the dropped-media pipeline direction, remove the legacy generic all-dropped warning so `droppedMedia` is the single user-facing notice source, add a regression for that case, and require redacted real send proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main catches and continues past failed per-item `MEDIA:` normalization, and the catastrophic fallback strips media without user-visible dropped-media details. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The PR's architecture is the right fix direction, but the current implementation should not also append the legacy generic `Media failed` warning when the new detailed dropped-media notice will be rendered.
