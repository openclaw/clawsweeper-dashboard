## Summary

Keep open: the PR is still useful in direction, but the current head has blocking correctness issues. The transcript mirror still reads `droppedMedia` from the wrapper entry instead of `entry.payload`, and the dropped-media renderability path conflicts with the existing silent-turn behavior; the external PR also still lacks after-fix real behavior proof.

## What This Changes

Adds a `DroppedMediaItem` pipeline from reply media normalization through outbound delivery, JSON projection, transcript mirroring, and tests so failed `MEDIA:` attachments produce sanitized user-visible warnings.

## Best Possible Solution

Land a narrow version that compiles, reads dropped media from the normalized payload entry correctly, preserves silent reply semantics, and includes real OpenClaw or Telegram after-fix proof for the visible warning behavior.

## Reproduction Assessment

Yes for the PR blockers: the current PR head source still contains a type-invalid `normalizedPayloads.flatMap((p) => p.droppedMedia ?? [])`, and current main has a focused silent-media normalization test that the dropped-media renderability path violates. I did not run tests because this review is read-only.

## Solution Assessment

No: the product direction is reasonable, but this patch head is not mergeable until it fixes the wrapper access, preserves silent-turn suppression, and provides real after-fix behavior proof. The safer path is to keep dropped-media visibility in the normal delivery path while explicitly excluding silent `NO_REPLY` failures.

  - confidence: 0.9
