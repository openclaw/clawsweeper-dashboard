## Summary

Keep this PR open: the dropped-media direction is useful, but the current head fails CI type checks and an existing silent-reply regression test, and it still lacks real behavior proof.

## What This Changes

The PR adds a sanitized dropped-media pipeline for failed `MEDIA:` normalization, carries it through reply/outbound routing, emits user/transcript notices, and adds focused tests.

## Best Possible Solution

Land a cleaned-up version that keeps silent-reply suppression intact, fixes the payload-wrapper access and unused imports, and then proves the warning in a real channel with private details redacted.

## Reproduction Assessment

Yes for the PR blockers: GitHub CI reproduces the typecheck failures and the silent-media test regression on the PR merge commit. For the original media-drop behavior, source inspection shows current main still lacks the structured dropped-media pipeline this PR proposes.

## Solution Assessment

Mostly yes, but not in this head. Carrying sanitized dropped-media items through reply and outbound delivery is maintainable, but the implementation must respect `NO_REPLY`, use the normalized payload wrapper correctly, and pass core checks before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Read dropped media from the payload wrapper:** `src/infra/outbound/deliver.ts:1921`
  - body: `normalizedPayloads` entries are shaped as `{ index, payload }`, so `p.droppedMedia` does not exist and core typecheck fails. If this were ignored at runtime, the transcript mirror would never append the dropped-media notices because it is reading the wrong object.
  - confidence: 0.97
- **[P2] Keep dropped media from reviving silent replies:** `src/auto-reply/reply/reply-payloads-base.ts:85`
  - body: Treating any `droppedMedia` as renderable lets `NO_REPLY\nMEDIA: ./missing.png` survive after normalization with `text` cleared, so a silent reply can still send an attachment-failure warning. The existing `agent-runner-payloads` test reproduces this as one returned payload where zero are expected.
  - confidence: 0.93
- **[P2] Remove the unused delivery imports:** `src/infra/outbound/deliver.ts:1-4`
  - body: These new imports are not referenced anywhere in the PR head, and `tsconfig.core.json` enables `noUnusedLocals`. Core typecheck is failing on them, so the branch cannot merge until they are removed or actually used.
  - confidence: 0.94
- **[P2] Remove the unused config-dir import:** `src/auto-reply/reply/reply-media-paths.ts:18`
  - body: `resolveConfigDir` is added but never used in `reply-media-paths.ts`. Because core typecheck rejects unused locals, this is another current CI blocker.
  - confidence: 0.93
