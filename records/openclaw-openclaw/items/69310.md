## Summary

Keep open: the PR addresses a real source-reproducible media-drop bug and has handled substantial review feedback, but the current diff still duplicates the legacy media-failure warning with the new detailed notice and lacks after-fix real behavior proof.

## What This Changes

The PR adds a DroppedMediaItem pipeline through reply media normalization, outbound payload planning, delivery, transcript mirroring, and tests so failed MEDIA attachments produce sanitized user-visible notices.

## Best Possible Solution

Use droppedMedia as the single source for user-visible attachment-failure notices, remove the legacy generic warning from the dropped-media path, add a focused regression for the all-dropped case, then verify a real redacted send.

## Reproduction Assessment

Yes, by source inspection: on current main, a mixed payload with one invalid and one valid MEDIA source catches and continues past the invalid item, and the existing warning only appears when all media fail. I did not run a live channel reproduction.

## Solution Assessment

No. The droppedMedia pipeline is the right direction, but this implementation should not also append the legacy `⚠️ Media failed.` text because users should receive one sanitized notice.
