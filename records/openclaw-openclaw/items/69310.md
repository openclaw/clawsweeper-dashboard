## Summary

Keep open. The PR targets a real dropped-media visibility problem, but the current head is not merge-ready because CI reproduces a TypeScript error, an existing silent-reply regression, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR threads dropped-media metadata through reply media normalization, outbound payload planning, delivery, JSON projection, transcript mirroring, and tests so failed MEDIA directives produce user-visible notices.

## Best Possible Solution

Land a narrow implementation that extends current main’s media-failure warning without breaking silent replies, compiles cleanly, and includes a redacted live channel proof for the visible notice.

## Reproduction Assessment

Yes for the PR blockers: CI at head 4bd2c95a87454be3d2699b7aa09dcd59949bc5d6 reproduces the TypeScript error and the silent-media regression. The original all-dropped media symptom is already partially covered on current main by generic warning tests.

## Solution Assessment

No; the direction is useful, but this diff must first read dropped media from the normalized payload entry, preserve the silent-reply contract, and provide real behavior proof before it is a mergeable fix.
