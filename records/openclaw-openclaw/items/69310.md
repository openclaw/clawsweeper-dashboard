## Summary

Keep this PR open, but it is not merge-ready: current main still has the dropped-media visibility bug, while the PR head introduces a silent-reply regression and lacks required real behavior proof.

## What This Changes

The PR threads dropped-media records through reply media normalization, outbound delivery, routing, JSON projection, transcript mirroring, and tests so failed MEDIA attachments can be shown as user-visible warnings.

## Best Possible Solution

Land a version that preserves NO_REPLY semantics, respects or explicitly documents the outbound hook policy, and demonstrates the warning in a real runtime flow before merge.

## Reproduction Assessment

Yes. Source inspection shows current main still drops failed media without per-item user feedback, and PR CI reproduces a new NO_REPLY plus missing MEDIA regression in `src/auto-reply/reply/agent-runner-payloads.test.ts:596`; no live after-fix proof is present yet.

## Solution Assessment

No. The direction is useful, but this implementation leaves droppedMedia renderable after an explicit silent reply and composes generated notice text after the outbound hook gate; the safer fix should honor silence first and settle hook handling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Honor silent replies after media normalization fails:** `src/auto-reply/reply/agent-runner-payloads.ts:58-61`
  - body: This catch now returns a droppedMedia-only payload when media normalization throws. For a reply like `NO_REPLY\nMEDIA: ./missing.png`, the existing silent-directive branch clears `text` after this catch, but `droppedMedia` remains and `isRenderablePayload` treats it as sendable, so OpenClaw emits an attachment warning despite the explicit no-reply directive. The current PR CI reproduces this in `agent-runner-payloads.test.ts`; clear the dropped entries or drop the payload when the parsed reply is silent and no media remains.
  - confidence: 0.95
