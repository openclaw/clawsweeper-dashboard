The PR adds a DroppedMediaItem pipeline that records failed MEDIA normalization, carries those entries through reply routing and outbound payloads, and sends user-visible attachment-failure notices with tests.

## Best Possible Solution

Land a corrected dropped-media pipeline that preserves silent replies, mirrors notices from the payload objects, removes unused imports, and demonstrates the notice in a real channel or terminal run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Read dropped media from each payload:** `src/infra/outbound/deliver.ts:1921`
  - body: `normalizedPayloads` entries are `{ index, payload }`, so `p.droppedMedia` is not a real property. This both fails TypeScript with TS2339 and prevents the transcript mirror from collecting the dropped-media notices; read from `p.payload.droppedMedia` instead.
  - confidence: 0.97
- **[P2] Keep dropped-media notices out of silent replies:** `src/auto-reply/reply/agent-runner-payloads.ts:58-61`
  - body: The catch path now attaches `droppedMedia` even for a parsed silent `NO_REPLY` payload. After the silent branch clears `text`, `isRenderablePayload` treats the dropped-media entry as visible content, so the existing silent-media test returns one warning payload instead of zero.
  - confidence: 0.92
- **[P3] Remove unused deliver imports:** `src/infra/outbound/deliver.ts:1-4`
  - body: These new imports are never read, and `check-prod-types` reports TS6133 for them. Removing `sendMediaWithLeadingCaption`, `chunkByParagraph`, and `chunkMarkdownTextWithMode` is needed before the production type lane can pass.
  - confidence: 0.99
- **[P3] Remove the unused config-dir import:** `src/auto-reply/reply/reply-media-paths.ts:18`
  - body: `resolveConfigDir` is imported in this patch but not used, and `check-prod-types` reports TS6133 on this line. Drop the import so the production type lane can pass.
  - confidence: 0.99
