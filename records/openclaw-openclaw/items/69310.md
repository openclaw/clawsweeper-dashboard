## Summary

Keep open: the PR targets a real dropped-media visibility problem, but the current head regresses NO_REPLY/silent media semantics and still lacks required after-fix real behavior proof.

## What This Changes

The PR adds DroppedMedia types, normalization propagation, delivery notices, transcript mirroring, and regression tests so failed MEDIA attachments become visible warnings.

## Best Possible Solution

Keep the dropped-media notice pipeline, but preserve NO_REPLY/silent payload suppression when failed media leaves no surviving attachment, then add real after-fix proof from a redacted live setup.

## Reproduction Assessment

Yes for the PR blocker: the PR head CI reproduces the silent-media regression against an existing test. The original current-main media-drop bug is source-reproducible from the normalization catch paths, but I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not yet. The pipeline direction is maintainable, but the current implementation makes a failed-media NO_REPLY payload renderable and needs both a narrow silent-behavior fix and real behavior proof before merge.

- **[P2] Respect NO_REPLY silence on failed media:** `src/auto-reply/reply/agent-runner-payloads.ts:58-61`
  - body: This catch now attaches `droppedMedia` when media normalization throws. For a payload parsed from `NO_REPLY\nMEDIA: ...`, the existing silent guard only clears `text`; because `isRenderablePayload` now treats `droppedMedia` as content, the existing `suppresses warning text when silent media payloads fail normalization` test fails and users can receive an attachment warning for a silent reply. Clear or suppress `droppedMedia` for parsed silent payloads with no surviving media.
