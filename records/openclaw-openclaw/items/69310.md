## Summary

Keep open: current main still lacks the PR’s full dropped-media visibility path, and this PR is useful, but it is not merge-ready because all-media-failed payloads will show both the legacy generic warning and the new detailed dropped-media notice.

## What This Changes

The PR adds a `DroppedMediaItem` pipeline for failed `MEDIA:` normalization, threads it through reply/outbound delivery, formats user-visible attachment warnings, and adds regression coverage.

## Best Possible Solution

Keep the dropped-media metadata pipeline, but make it the single source of user-visible attachment failure text by removing the legacy generic warning from media normalization and covering all-media-failed output with a focused regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a reply whose `MEDIA:` entries all fail normalization gets legacy `⚠️ Media failed.` text in `reply-media-paths.ts`, then the PR’s delivery code appends the new `Attachment not sent` notice from `droppedMedia`. I did not run a live channel send in this read-only review.

## Solution Assessment

No, the PR direction is right but the current implementation should not render both warning systems. The maintainable fix is to preserve `droppedMedia` during normalization and let outbound delivery render exactly one user-facing notice.

## Review Findings
