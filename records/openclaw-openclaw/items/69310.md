## Summary

Keep open: current main still has source-visible media-drop paths that can hide attachment failures, and this PR is pointed at the right problem, but the patch still introduces an observer/audit regression for separately sent warning messages.

## What This Changes

The PR adds a DroppedMediaItem pipeline that records failed MEDIA normalization, surfaces sanitized attachment-failure notices through delivery/transcript/JSON paths, and adds regression tests.

## Best Possible Solution

Keep the dropped-media pipeline, but make every user-visible warning delivery observable through the existing sent-message/audit hooks, avoid duplicate legacy and detailed warnings, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows MEDIA failures are caught and stripped in the reply-media normalizer and agent-runner fallback, while the docs define MEDIA as attachment delivery metadata.

## Solution Assessment

No. The proposed shape is maintainable, but the current patch sends separate warning messages outside the sent-message observer path and can double-report all-failed media, so it needs a narrow repair before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Emit hooks for the separate warning send:** `src/infra/outbound/deliver.ts:1235`
  - body: When a payload has surviving media plus dropped media, this branch sends the dropped-media warning as a second text message after the primary `message_sent` event has already fired, then truncates the warning result out of `results`. That means plugin `message_sent`, internal `message:sent`, and after-delivery observers never see a real outbound warning message, even though the text-only fallback path reports the notice in hook content. Emit an observer event for the warning or move the notice into the delivery accounting before firing hooks.
  - confidence: 0.86
- **[P3] Remove the duplicate all-failed media warning:** `src/auto-reply/reply/reply-media-paths.ts:243`
  - body: When every media item fails, the normalizer still adds the legacy `Media failed` text while also returning `droppedMedia`. The delivery path then appends the new detailed dropped-media notice, so users receive two warnings for the same failed attachment. Once `droppedMedia` is present, skip the legacy generic warning and let the detailed notice be the only user-facing diagnostic.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/infra/outbound/deliver.ts:800`
  - body: This is a user-facing fix that changes what users see when attachments are dropped, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix changes to add an active-version changelog entry; add one without forcing bot or maintainer attribution.
  - confidence: 0.78
