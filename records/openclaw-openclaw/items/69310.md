## Summary

Keep open: the PR addresses a real dropped-media visibility gap and the discussion shows several privacy/routing review comments were fixed, but the current head still sends a duplicate generic warning on all-media-failed replies and is missing the required changelog entry for a user-facing fix.

## What This Changes

The PR adds a DroppedMediaItem pipeline that records media-normalization failures, preserves dropped-media-only payloads through routing/outbound projections, and emits sanitized user-visible and transcript notices.

## Best Possible Solution

Keep the DroppedMedia pipeline, remove the legacy generic normalizer warning so deliver.ts is the single notice renderer, add regression coverage for the all-media-failed path, and add the required changelog entry.

## Reproduction Assessment

Yes. Current main source shows MEDIA normalization errors can be logged and stripped without per-attachment user feedback, and the PR regression is source-reproducible by following an all-media-failed payload from reply-media-paths.ts into deliver.ts notice composition.

## Solution Assessment

No. The DroppedMedia design is the right narrow direction, but the current PR should stop injecting the old generic warning before the new dropped-media renderer and should add the required changelog entry.
