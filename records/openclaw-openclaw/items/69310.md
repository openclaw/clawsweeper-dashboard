## Summary

Keep open: current main still has source-visible dropped-media gaps, and this PR is useful but not merge-ready because the latest diff can emit duplicate warnings for all-failed media payloads and lacks after-fix real behavior proof.

## What This Changes

This PR adds DroppedMedia tracking for failed MEDIA normalization, emits sanitized attachment-failure notices in outbound delivery, transcript, and JSON paths, and adds regression tests.

## Best Possible Solution

Revise the PR so all-failed media payloads produce one sanitized dropped-media notice, add the required changelog entry, and include redacted real runtime proof before merge.

## Reproduction Assessment

Yes, source-level reproduction is clear: current main drops individual failed media in the catch-and-continue path when other media survives, and the catastrophic fallback clears media without user-facing dropped-media metadata. I did not run a live channel repro in this read-only review.

## Solution Assessment

No, not yet. The dropped-media pipeline is the right direction, but the implementation should remove the older generic all-media-failed warning so users do not see duplicate failure text, then provide real behavior proof and changelog coverage.
