## Summary

Keep this PR open. Current main still has source-visible dropped-media gaps, and the branch addresses much of the right surface, but the patch still sends generated warning text outside the documented outbound hook policy and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a dropped-media payload pipeline for failed MEDIA normalization, surfaces sanitized attachment warnings through outbound delivery/transcript/JSON paths, and adds focused tests.

## Best Possible Solution

Keep the dropped-media tracking direction, but route the final warning text through the normal outbound hook/sanitization/lifecycle path and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows mixed media failures can be logged and continued without user-visible per-file feedback, and the catastrophic normalizeMediaPaths catch clears media without carrying drop details; I did not run a live channel repro in this read-only review.

## Solution Assessment

No. The PR is directionally useful, but the notice must be composed before hook inspection or re-enter the message_sending/message_sent lifecycle so policy hooks can rewrite or cancel the actual outbound text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Run dropped-media notices through outbound hooks:** `src/infra/outbound/deliver.ts:1676`
  - body: This sends a new user-visible warning after the one `message_sending` hook run. For structured and media payloads the warning is delivered directly with `sendTextChunks`, so plugins that rewrite or cancel outbound content cannot inspect or block it despite the documented terminal cancel contract. Compose the notice before the hook sees the payload, or run the notice through the same hook/lifecycle path before sending.
  - confidence: 0.88
