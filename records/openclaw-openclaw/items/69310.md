## Summary

Keep open: the bug-fix direction is valid and current main only partially covers all-dropped media with a generic warning, but this PR is not merge-ready because latest CI reproduces a type-threading failure, a silent-reply regression, and missing external real behavior proof.

## What This Changes

The PR adds a DroppedMediaItem pipeline through reply media normalization, outbound projection, delivery, routing, JSON output, transcript mirroring, and tests so failed MEDIA attachments can produce user-visible notices.

## Best Possible Solution

Land a corrected narrow dropped-media pipeline that preserves silent NO_REPLY semantics, fixes the normalized-payload threading, covers partial-drop notices, and includes real chat or CLI proof plus focused regression tests.

## Reproduction Assessment

Yes for the PR blockers: GitHub CI reproduces both the type failure and silent-reply regression on the PR merge SHA. For the original media-drop behavior, current-main source inspection shows per-item failures are still only logged when other media survive; I did not run a live channel repro in this read-only review.

## Solution Assessment

No. The approach is reasonable, but the current patch is not the best merge shape until it reads droppedMedia from the normalized payload entry, keeps silent replies silent, and adds required real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Read dropped media from the payload entry:** `src/infra/outbound/deliver.ts:1921`
  - body: `normalizedPayloads` stores entries as `{ index, payload }`, so this reads a property that does not exist on the wrapper and CI fails with TS2339. It also means mirror notices would be collected from the wrong place; use `p.payload.droppedMedia` or carry the dropped list explicitly.
  - confidence: 0.96
- **[P2] Keep silent failed-media replies suppressed:** `src/auto-reply/reply/agent-runner-payloads.ts:58-61`
  - body: When `NO_REPLY` media normalization throws, this catch leaves a dropped-media-only payload behind; the later silent-token path clears text, but the new renderability rules still deliver the warning. That breaks the existing silent-turn contract and the PR CI test that expects no payload for silent failed media.
  - confidence: 0.9
