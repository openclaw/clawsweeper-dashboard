## Summary

Keep this PR open, but current head is not merge-ready: the useful structured dropped-media idea still has compile blockers, a silent-turn regression, duplicate-warning risk against current main, and no contributor real-behavior proof. Current main already covers the baseline user problem with a generic media-failure warning, so the remaining value is the richer dropped-media pipeline and needs a clean, proven implementation.

## What This Changes

The PR adds a `DroppedMediaItem` pipeline through reply normalization, routing, outbound delivery, JSON projection, logs, transcript mirroring, and tests so failed `MEDIA:` attachments produce user-visible warning text.

## Best Possible Solution

Either keep the shipped generic warning or narrow this PR to a clean structured-diagnostics delta over current main, preserving silent-turn semantics and proving the visible warning in a real channel setup.

## Reproduction Assessment

Yes for the PR blockers: CI and source inspection show a type error at `deliver.ts:1921` and a failing silent-turn test where a `NO_REPLY` media failure emits a payload. The original all-dropped media visibility bug is already covered on current main by source and tests.

## Solution Assessment

No, not at this head. The structured pipeline may be a reasonable direction, but it must compile, preserve silent replies, avoid double warnings over the existing main behavior, and include real behavior proof before it is the best fix.
