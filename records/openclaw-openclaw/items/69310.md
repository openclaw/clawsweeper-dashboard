## Summary

Keep open: the direction is valid and current main still has a source-reproducible mixed-media drop path, but this external PR is not merge-ready because after-fix real behavior proof is missing, one path can render duplicate media-failure warnings, and the user-facing fix lacks the required changelog entry.

## What This Changes

The branch adds a DroppedMediaItem pipeline through reply media normalization, outbound payload planning, delivery notices, transcript mirroring, routing guards, JSON projections, and targeted tests.

## Best Possible Solution

Keep the dropped-media pipeline, but make the renderer emit exactly one sanitized user notice per failed attachment, add the required changelog entry, and require real CLI or channel proof before merge.

## Reproduction Assessment

Yes from source inspection: current main's per-item normalizer catches and continues on failed media while preserving surviving media, so mixed MEDIA payloads can still lose a failed attachment without detailed user feedback. I did not run a live channel reproduction because this review is read-only.

## Solution Assessment

No. The proposed pipeline is the right shape, but the current PR head composes the legacy generic all-dropped warning with the new detailed notice; the safer fix is to remove that generic text when `droppedMedia` will render the user-facing notice.
