## Summary

Keep open. The PR targets a real dropped-media visibility gap, but the latest head is not merge-ready because all-failed media payloads would render both the legacy generic warning and the new detailed notice, and the outbound delivery file has unused imports/tracking that should fail lint.

## What This Changes

The PR adds dropped-media tracking through reply normalization, routing, outbound delivery, JSON projection, transcript mirroring, warning logs, and tests so failed attachments are surfaced to users.

## Best Possible Solution

Keep the droppedMedia pipeline, make the detailed sanitized notice the only user-facing fallback for dropped attachments, remove dead code/imports, and lock the all-media-dropped case with targeted tests.

## Reproduction Assessment

Yes. Current main can be checked by sending or normalizing an invalid MEDIA source, which reaches the catch path in reply-media-paths; the PR regression is visible statically because the all-failed branch appends the legacy warning before deliver.ts appends the detailed dropped-media notice.

## Solution Assessment

No, not yet. The droppedMedia pipeline is the right maintainable direction, but this PR should first remove the stale generic warning path and lint-dead outbound delivery additions.
