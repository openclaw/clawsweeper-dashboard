## Summary

Keep open: current main still hides the requested device purpose, client/platform metadata, activity, and editable identity from `openclaw devices list`, while the related naming issue covers only part of the broader UX.

## What This Changes

Request to make `openclaw devices list` identify paired devices by client purpose, platform or OS, activity, and optional user notes so default CLI/webchat entries are understandable.

## Best Possible Solution

Adopt one coordinated device-identification UX: show sanitized client, platform, stable ID, and activity metadata in the CLI, preserve `--json`, and route editable labels or notes through one shared Gateway/CLI/UI contract coordinated with https://github.com/openclaw/openclaw/issues/13870.

## Reproduction Assessment

Yes, at source level: current main deterministically renders paired devices without client purpose, platform/device family, activity, or notes while storage already carries some of that metadata. I did not run the CLI because this review is read-only.

## Solution Assessment

No single implementation path is confirmed as best yet. Displaying existing sanitized metadata is a bounded first step, but editable names or notes need a shared persisted Gateway contract rather than a CLI-only field.
