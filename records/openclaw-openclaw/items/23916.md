## Summary

Keep open. Current main still hides client purpose, platform/device family, activity, and editable identity from the paired-device table; the related naming issue and timestamp PR each cover only part of this broader request, and the earlier combined implementation PR closed unmerged.

## What This Changes

Show clearer paired-device identity and purpose in `openclaw devices list`, including client/platform/activity details and potentially user-editable names or notes.

## Best Possible Solution

Adopt one coordinated device-identification design: display sanitized client purpose, platform/device family, stable device ID, and agreed activity metadata in the CLI while defining editable names or notes through a single persisted Gateway contract shared with CLI and UI.

## Reproduction Assessment

Yes, at source level. Current main deterministically renders the paired-device table without client purpose, platform/device family, activity, or notes, although no live CLI run was performed during this read-only review.

## Solution Assessment

No single proposed path is yet the complete best fix. Surfacing existing sanitized metadata is a bounded first step, but editable names or notes should use one persisted Gateway contract shared with the web UI rather than becoming a CLI-only field.
