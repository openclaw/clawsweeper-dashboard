## Summary

Keep open: current main and v2026.5.12 still render the compact devices list table, while the requested client purpose, OS/device-family, activity/detail mode, and editable note/name UX remain unimplemented. The storage and Gateway layers already carry much of the raw metadata, but the user-facing CLI/docs work is still pending and related display PRs remain open/conflicting.

## What This Changes

The issue requests richer `openclaw devices list` output so users can identify paired devices by purpose, client/platform details, activity, and possibly editable notes or a full-detail view.

## Best Possible Solution

Land or replace the open display PRs with sanitized client/platform/activity columns and docs, preserve `--json`, then coordinate durable editable names or notes through https://github.com/openclaw/openclaw/issues/13870.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, release, and related-PR inspection give high confidence that the requested metadata, detail mode, and editable note/name surfaces are still absent.

## Solution Assessment

Partly yes: surfacing already-collected pairing metadata in the CLI is the narrow first slice. Editable notes/names and any `--full` detail mode still need an additive API/UX decision coordinated with the durable naming workflow.
