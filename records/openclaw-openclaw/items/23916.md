## Summary

Keep open: current main and the latest release still render the compact paired-device table without client purpose, platform/device family, activity, notes, or a detail mode; the open naming issue covers only part of this broader device-identification UX.

## What This Changes

The issue requests richer `openclaw devices list` output so users can identify paired devices by purpose, client/platform metadata, activity, and possibly editable notes or names.

## Best Possible Solution

Adopt one coordinated device-identification UX: show sanitized client, platform, stable ID, and activity metadata in the CLI, preserve `--json`, and route editable labels or notes through one shared Gateway/CLI/UI contract coordinated with https://github.com/openclaw/openclaw/issues/13870.

## Reproduction Assessment

Yes, at source level: current main deterministically renders paired devices without client purpose, platform/device family, activity, or notes while storage already carries some of that metadata. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear for the full request. Displaying existing sanitized metadata is a bounded first step, but editable names or notes need a shared persisted Gateway contract rather than a CLI-only field.
