## Summary

Keep open: current main and v2026.6.10 still render the compact paired-device list without client purpose, platform/device family, last activity, notes, or a detail mode. The open label PR only covers the human-friendly naming slice, so the broader device-list metadata request still needs maintainer scope decisions.

## What This Changes

Request clearer paired-device identity/details in `openclaw devices list`, including client purpose, host/OS-style metadata, default-device explanation, and editable notes or names.

## Best Possible Solution

Adopt one coordinated device-identification UX that shows sanitized client, platform, stable ID, and activity metadata in the CLI, preserves `--json`, and coordinates editable labels or notes with https://github.com/openclaw/openclaw/issues/13870 and https://github.com/openclaw/openclaw/pull/94517.

## Reproduction Assessment

Yes, at source level: current main deterministically renders paired devices without client purpose, platform/device family, activity, notes, or detail output while storage carries some of that metadata. This is a missing feature rather than a failing runtime path needing live reproduction.

## Solution Assessment

Unclear for the full request. Displaying existing sanitized metadata is bounded, but editable labels/notes and any `--full` mode need a shared Gateway/CLI/UI contract rather than a CLI-only field.
