## Summary

Keep open. Current main still has the CLI identification gap: the paired-device table is compact, docs do not describe richer output or notes, and related implementation work is still open/unmerged rather than shipped.

## What This Changes

The issue requests richer `openclaw devices list` output so users can identify the default CLI/service device and distinguish paired clients by purpose, platform, activity, host/user-agent-like details, and optional notes.

## Best Possible Solution

Land a focused device-list UX update that surfaces existing non-secret metadata in the CLI, preserves `--json`, sanitizes every device-controlled string, updates `src/cli/devices-cli.test.ts` and `docs/cli/devices.md`, and coordinates display/name/timestamp scope with #57449, #55973, and #13870.

## Reproduction Assessment

Yes for the observable CLI gap: source and tests show the current paired-device table only renders Device/Roles/Scopes/Tokens/IP while storage carries richer metadata. Editable notes are a feature-design request rather than a runtime bug reproduction.

## Solution Assessment

Unclear until an implementation PR lands. #57449 looks like a plausible base for display columns, but the complete solution should reconcile timestamp semantics with #55973 and keep editable names/notes coordinated with #13870.
