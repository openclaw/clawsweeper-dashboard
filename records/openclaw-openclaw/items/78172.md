Keep open: the PR has solid real-provider proof and the shared speech-core boundary is the right place for the behavior, but the latest head still removes emoji/symbol runs with an empty string, which can merge adjacent words in enabled TTS output.

The PR adds a default-off `messages.tts.skipEmojiSymbols` config option, applies TTS text preprocessing across file, stream, and telephony synthesis paths, updates config metadata/docs, and adds focused TTS tests.

## Best Possible Solution

Keep the shared speech-core preprocessing approach, but replace removed emoji/symbol sequences with a word boundary and collapse whitespace, then add a no-space separator regression test.
