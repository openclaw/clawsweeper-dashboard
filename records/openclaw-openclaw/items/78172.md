## Summary

Keep this PR open: it is a plausible opt-in TTS feature, but it is not merge-ready because the filter still misses direct synthesis/Talk, the emoji matcher is incomplete for common modern emoji, and the supplied proof is unit-test-only.

## What This Changes

Adds an opt-in `messages.tts.skipEmojiSymbols` config/schema field, filters several TTS wrapper paths before provider synthesis, updates generated config hash metadata, and adds mocked TTS tool tests.

## Best Possible Solution

Put one shared TTS text-preparation step after effective config resolution and before every provider request, document the option and scope, and require redacted real provider/audio proof before merge.

## Reproduction Assessment

Yes. Source inspection shows the PR applies filtering in wrapper functions while current `talk.speak` uses direct `synthesizeSpeech`, and the added tests mock the wrapper instead of proving provider input.

## Solution Assessment

No. The maintainable fix is a shared runtime text-preparation step with Unicode-aware sequence handling and docs, not separate wrapper-level filtering.
