## Summary

Keep this PR open: the shared speech-core filtering approach is reasonable and backed by real-provider proof, but the branch still leaves a scoped config contract gap where Feishu channel/account TTS overrides reject the new key.

## What This Changes

The PR adds an opt-in `messages.tts.skipEmojiSymbols` TTS preprocessing option, applies it before file/stream/telephony synthesis, updates config/docs metadata, and adds focused TTS tests.

## Best Possible Solution

Land the shared speech-core filtering after Feishu top-level/account TTS override schemas, parse coverage, and generated metadata accept the same opt-in key, or explicitly document the key as global-only.

## Reproduction Assessment

Yes. Source inspection at PR head shows the core TTS schema accepts `skipEmojiSymbols`, `resolveEffectiveTtsConfig` merges channel/account TTS overrides, and Feishu's strict scoped override schema omits the same key.

## Solution Assessment

No, not yet. Centralizing filtering in speech-core is the right boundary, but the config contract should accept the key everywhere `resolveEffectiveTtsConfig` can merge TTS overrides or explicitly document a global-only scope.
