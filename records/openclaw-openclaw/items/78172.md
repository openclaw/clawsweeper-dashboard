## Summary

Keep open: the PR is a plausible additive TTS option, but source review found that the telephony path still sends the original text to synthesis and the new filter can create empty provider requests; the supplied proof is mocked Vitest output rather than real TTS behavior.

## What This Changes

The PR adds `messages.tts.skipEmojiSymbols` across config/schema/docs/tests and filters emoji or decorative symbols before TTS synthesis when the option is enabled.

## Best Possible Solution

Apply the filter at a single synthesis boundary that feeds file, stream, and telephony provider requests, guard empty text after stripping, keep the default disabled, and require redacted real TTS proof before landing.

## Reproduction Assessment

Yes for the patch defects by source inspection: the telephony path computes filtered text but still prepares synthesis from `params.text`, and wrapper-level filtering can produce an empty request after earlier guards. No high-confidence real provider reproduction is supplied for the user-audible behavior.

## Solution Assessment

No: the direction is reasonable, but the current implementation should move filtered text through every provider request and handle empty-after-strip before synthesis. The safer path is a shared helper or boundary used by file, stream, and telephony TTS plus live proof.
