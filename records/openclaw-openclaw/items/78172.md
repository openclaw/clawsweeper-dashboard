## Summary

Keep this PR open: the shared TTS preprocessing approach and real-provider proof are strong, but the patch is not merge-ready because the new public TTS config field is not accepted by Feishu's strict channel/account TTS override schema even though the docs say those overrides use the same shape as `messages.tts`.

## What This Changes

Adds an opt-in `messages.tts.skipEmojiSymbols` setting, strips emoji/symbol text in shared speech-core file/stream/telephony synthesis, guards empty post-strip input, updates TTS docs/config metadata, and adds focused tests.

## Best Possible Solution

Land the shared speech-core preprocessing after every TTS config and override schema plus generated metadata accepts the field, with focused config regression coverage and maintainer acceptance of the public option name and stripping policy.

## Reproduction Assessment

Yes, at source level: current `main` has no `skipEmojiSymbols` option and passes original text through TTS synthesis, while this PR demonstrates the new behavior with real Microsoft speech output. The remaining PR blocker is also source-reproducible because Feishu's strict TTS override schema lacks the new field.

## Solution Assessment

Not yet. Centralizing preprocessing in speech-core is the right implementation shape, but the public config surface needs all strict override schemas and generated metadata kept in sync before merge.
