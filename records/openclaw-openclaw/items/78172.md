## Summary

This PR should stay open but needs one compatibility fix before merge: the shared TTS preprocessing and proof are strong, but Feishu's strict scoped TTS override schema still rejects the new key for top-level and account overrides.

## What This Changes

The PR adds an opt-in `messages.tts.skipEmojiSymbols` config, applies shared emoji/symbol stripping before file, stream, and telephony TTS synthesis, updates docs/config metadata, and adds focused TTS/tool tests.

## Best Possible Solution

Land the shared speech-core filtering after Feishu top-level/account TTS override schemas, parse coverage, and generated metadata accept the same opt-in key, or explicitly document the key as global-only.

## Reproduction Assessment

Yes. Source inspection at PR head shows the core TTS schema accepts `skipEmojiSymbols`, while Feishu's strict top-level/account TTS override schema reuses `TtsOverrideSchema` without that key.

## Solution Assessment

No, not yet. Centralizing filtering in speech-core is the right boundary, but the config contract should either accept the key everywhere `resolveEffectiveTtsConfig` can merge TTS overrides or explicitly document global-only scope.
