## Summary

Keep this PR open: the real-provider proof is now adequate, but the patch still applies `skipEmojiSymbols` only in wrapper entry points, leaving direct speech-core synthesis and `talk.speak` able to send the original emoji/symbol text to providers.

## What This Changes

The PR adds an opt-in `messages.tts.skipEmojiSymbols` config/schema/docs surface and filters emoji/symbols before selected TTS synthesis paths, with focused tests and generated config baseline updates.

## Best Possible Solution

Put emoji/symbol preprocessing at the shared speech synthesis boundary so every configured file, stream, Talk/direct, and telephony caller honors the same effective config while the default remains disabled.

## Reproduction Assessment

No for a current-main bug because this PR adds a new opt-in config feature. The PR defect is source-reproducible: `talk.speak` and direct `synthesizeSpeech` bypass the wrapper-only filter while using the same TTS config.

## Solution Assessment

No; wrapper-level filtering is not the best shared implementation because public direct synthesis paths bypass it. The safer fix is to prepare filtered text once at the shared synthesis boundary and test the bypass callers.
