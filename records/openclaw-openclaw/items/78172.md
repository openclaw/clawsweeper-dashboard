## Summary

Keep open: the TTS option is useful and the real-provider proof is strong, but the new strip path deletes emoji/symbol sequences without preserving a word boundary, so enabled synthesis can merge adjacent words.

## What This Changes

Adds a default-off `messages.tts.skipEmojiSymbols` config option, shared speech-core filtering, docs/config metadata, and focused TTS tests.

## Best Possible Solution

Keep the shared speech-core preprocessing approach, but preserve word boundaries when removing emoji/symbol sequences and leave path/system-message elision tracked separately on https://github.com/openclaw/openclaw/issues/77969.

## Reproduction Assessment

Yes for the PR finding from source inspection: the new stripping helper replaces matched emoji/symbol sequences with `""`, which merges adjacent words when an emoji or symbol is used without spaces. I did not run tests because this review is read-only.

## Solution Assessment

No as written; centralizing filtering in speech-core is the right boundary, but the implementation should replace removed sequences with a boundary and cover no-space separator cases before merge.
