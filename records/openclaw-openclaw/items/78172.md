## Summary

Keep this PR open as a useful implementation candidate for the linked TTS emoji/symbol request, but the current head is not merge-ready because telephony still synthesizes the original text, the new speech-core test has source-visible helper/result-field mistakes, and the supplied proof is mocked unit-test output rather than real TTS behavior.

## What This Changes

The branch adds an opt-in `messages.tts.skipEmojiSymbols` config/schema/docs field and applies emoji/symbol filtering in TTS synthesis paths with agent-tool and speech-core tests.

## Best Possible Solution

Keep the opt-in config shape only if maintainers accept the TTS preprocessing scope, then route one filtered text value through file, stream, and telephony synthesis, repair the tests, rebase, and add redacted live provider proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of the current head shows the telephony synthesis path still using `params.text`, plus the undefined helper and wrong result field in the new test. A current-main reproduction of the feature itself is not applicable because the option is absent on `main`.

## Solution Assessment

No: the shared-path direction is reasonable, but this patch does not consistently use the filtered text and its new test is broken. The safer solution is one shared filtered-text handoff with coverage for regular, stream, and telephony TTS plus live enabled/disabled proof.
