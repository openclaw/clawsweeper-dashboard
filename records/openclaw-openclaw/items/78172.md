The PR adds an opt-in `messages.tts.skipEmojiSymbols` setting, applies emoji/symbol stripping across TTS synthesis paths, updates config/docs metadata, and adds focused speech-core and TTS tool tests.

## Best Possible Solution

Land the shared speech-core filtering after Feishu top-level/account TTS override schemas, parse tests, and generated config metadata accept the same opt-in key, or explicitly document the key as global-only.

## Reproduction Assessment

Yes. Source inspection is enough: parsing `channels.feishu.tts.skipEmojiSymbols` or an account-level equivalent reaches Feishu's strict `TtsOverrideSchema`, which lacks the key while docs promise scoped overrides use the `messages.tts` shape.

## Solution Assessment

No, not yet. Centralizing filtering in speech-core is the right boundary, but the config contract needs Feishu schema/test/generated parity or an explicit global-only documentation choice.
