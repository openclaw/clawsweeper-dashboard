## Summary

## What This Changes

The PR adds an opt-in `messages.tts.skipEmojiSymbols` config field, filters text before `textToSpeech` synthesis, updates TTS tool tests, and refreshes config baseline hashes.

## Best Possible Solution

Apply an opt-in Unicode-aware text cleanup step after effective TTS config resolution and before provider preparation for all public TTS synthesis modes, document the key, and keep long path replacement as separate scoped work.

## Reproduction Assessment

Yes for the PR review findings by source inspection: the PR filters only `textToSpeech`, while current main exposes direct, streaming, and telephony synthesis paths that pass the original text to providers. I did not run tests because this review is read-only.

## Solution Assessment

No. The maintainable fix is to apply cleanup once in the shared synthesis preparation path after effective config resolution, use Unicode-aware sequence handling, and document the new public config key.
