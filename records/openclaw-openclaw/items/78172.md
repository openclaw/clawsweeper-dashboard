## Summary

Keep open: the PR is focused and now has real provider proof, but it is not merge-ready because the new TTS filter is applied only by wrapper APIs while the public Talk synthesis path still calls the lower-level synthesis API directly and would ignore the new setting.

## What This Changes

Adds an opt-in `messages.tts.skipEmojiSymbols` config option with schema/docs/help updates and speech-core preprocessing tests for file, stream, and telephony TTS wrappers.

## Best Possible Solution

Move the preprocessing and empty-after-strip guard into the shared synthesis setup used by direct synthesis, stream synthesis, file output, and telephony, keep the default disabled, and add coverage for the Talk/direct-synthesis bypass.

## Reproduction Assessment

Not applicable as a strict current-main bug repro because the PR adds a new config option. The blocking PR issue is source-reproducible: `talk.speak` uses `synthesizeSpeech` directly while the PR filters only wrapper calls.

## Solution Assessment

No, not yet: the option belongs in the shared synthesis preparation layer rather than only in `textToSpeech` wrappers. That preserves one source of truth for file, stream, Talk, and telephony provider requests.
