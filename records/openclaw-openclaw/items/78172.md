## Summary

This PR should stay open because it targets a useful opt-in TTS config feature, but the branch still violates the existing TTS effective-config contract and the supplied proof is mocked test output rather than a real post-fix TTS run.

## What This Changes

The PR adds `messages.tts.skipEmojiSymbols`, schema/type wiring, emoji/symbol stripping before TTS synthesis, and focused agent-tool tests.

## Best Possible Solution

Implement this as one shared TTS text-preparation step that resolves effective config with agent, channel, and account context, then document the new setting after maintainers confirm the filtering scope.

## Reproduction Assessment

No high-confidence current-main bug reproduction is applicable because this PR adds a new opt-in config option. The blocking PR defects are source-reproducible from the diff against current-main TTS config precedence and call-site context.

## Solution Assessment

No. The feature belongs in shared TTS preparation, but this branch resolves the option without full effective-config context and duplicates filtering in the agent tool before channel/account overrides can apply.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve the filter with channel context:** `extensions/speech-core/src/tts.ts:1104-1107`
  - body: The shared filter calls `resolveEffectiveTtsConfig` with agent and account IDs but not the channel. Current TTS config resolution applies `channels.<id>.tts` and account overrides through channel context, so automatic TTS and slash-command paths can ignore channel policy for this new option.
  - confidence: 0.95
- **[P2] Remove the tool-local emoji prefilter:** `src/agents/tools/tts-tool.ts:94-99`
  - body: The agent tool strips text before calling the shared TTS path using only the agent ID. Once the text is stripped, a channel or account override that disables `skipEmojiSymbols` cannot restore it, so the tool can violate the same effective-config precedence used by the rest of TTS.
  - confidence: 0.93
- **[P3] Document the new TTS option:** `src/config/types.tts.ts:81-82`
  - body: This adds a user-facing `messages.tts.skipEmojiSymbols` setting, but the TTS docs do not list the option. Users would not be able to discover the field or understand how it participates in the existing global, agent, channel, and account precedence.
  - confidence: 0.84
