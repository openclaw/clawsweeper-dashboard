## Summary

Keep open: this is a useful, well-proven TTS improvement, but the new public config field is not wired into Feishu's strict TTS override schema, so one existing channel/account override surface would reject the option before speech-core can apply it.

## What This Changes

Adds `messages.tts.skipEmojiSymbols`, strips emoji/symbols in shared TTS synthesis for file, stream, and telephony paths, documents the option, updates generated config metadata, and adds focused tests.

## Best Possible Solution

Keep the shared speech-core preprocessing approach, but update every TTS config and override schema plus generated metadata so the new option is accepted consistently before merge.

## Reproduction Assessment

Yes, at source level: current main has no `skipEmojiSymbols` field and passes original text into provider preparation. This is a new opt-in capability, not a broken existing documented behavior.

## Solution Assessment

Not yet. Centralizing the preprocessing in speech-core is the right implementation shape, but the PR needs to keep strict TTS override schemas aligned with the new public config field.
