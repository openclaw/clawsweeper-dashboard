## Summary

Keep open: this is a useful opt-in TTS improvement, but the current patch is not merge-ready because filtering is wrapper-local, common emoji sequences are not handled, the public config key is undocumented, and external proof is only mocked tests.

## What This Changes

The PR adds `messages.tts.skipEmojiSymbols`, registers it in TTS config schema/types, applies regex filtering before `textToSpeech` synthesis, updates mocked TTS tool tests, and refreshes config baseline hashes.

## Best Possible Solution

Implement an opt-in, Unicode-aware TTS input preparation step after effective config resolution and before provider preparation for all synthesis modes, document the config key, and keep path handling separately tracked unless it is also solved.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR filters only the `textToSpeech` wrapper, while current runtime also has direct, streaming, and telephony synthesis paths. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but the narrow maintainable solution is a shared Unicode-aware preparation step plus docs and real proof, not a wrapper-local regex filter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Filter every TTS synthesis path:** `extensions/speech-core/src/tts.ts:1133`
  - body: This only rewrites text before `textToSpeech` calls `synthesizeSpeech`; direct Talk calls `synthesizeSpeech`, Discord voice can use `textToSpeechStream`, and telephony uses `textToSpeechTelephony`, all of which still pass the original text into provider preparation. With the option enabled, those paths can still speak emoji, so move the filtering into the common request/preparation path used by every synthesis mode.
  - confidence: 0.88
- **[P2] Use complete emoji and symbol matching:** `extensions/speech-core/src/tts.ts:1107-1110`
  - body: The hand-maintained range misses common emoji sequences such as regional-indicator flags and newer U+1FAxx emoji, and it can leave variation selectors, ZWJ, or keycap remnants behind. Use Unicode-aware sequence handling or a tested emoji/grapheme matcher and cover those cases so the option works for ordinary emoji-heavy input.
  - confidence: 0.86
- **[P3] Document the public TTS config key:** `src/config/zod-schema.core.ts:592`
  - body: The schema adds `messages.tts.skipEmojiSymbols`, but the TTS field reference does not describe the key, its default, or which synthesis paths it affects. Add it to the user-facing TTS docs alongside the baseline update so operators can discover and configure it correctly.
  - confidence: 0.82
