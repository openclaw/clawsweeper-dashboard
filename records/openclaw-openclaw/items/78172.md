## Summary

Keep this PR open with changes requested: the shared TTS preprocessing design and real-provider proof are good, but the current helper still deletes emoji/symbol separators without preserving word boundaries, which can change the spoken text for enabled users.

## What This Changes

Adds a default-off `messages.tts.skipEmojiSymbols` config option, shared TTS preprocessing for file/stream/telephony synthesis, empty-after-strip guards, metadata/docs updates, and focused TTS tests.

## Best Possible Solution

Keep the shared speech-core preprocessing design, but replace stripped emoji/symbol runs with a whitespace boundary, normalize whitespace, add a no-space separator regression test, and leave long path/file elision to the linked broader issue.

## Reproduction Assessment

Yes for the PR finding: source inspection and an exact regex probe show `stripEmojiAndSymbols` turns no-space separators like `hello✅world` into `helloworld`. I did not run the repository test suite because this review is read-only.

## Solution Assessment

No as written; centralizing filtering in speech-core is maintainable, but the helper should preserve word boundaries before this opt-in config is merged.
