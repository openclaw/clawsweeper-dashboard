## Summary

Keep open: the shared speech-core approach and real-provider proof are strong, but the current helper deletes emoji/symbol runs without inserting a boundary, so enabled TTS can merge words such as `hello✅world` into `helloworld`.

## What This Changes

Adds a default-off `messages.tts.skipEmojiSymbols` config option, shared TTS preprocessing and empty-text guards for file, stream, and telephony synthesis, plus docs, metadata, and focused tests.

## Best Possible Solution

Keep the shared speech-core preprocessing design, but replace stripped emoji/symbol runs with a whitespace boundary, normalize whitespace, add a no-space separator regression test, and leave long path/file elision to the linked broader issue.

## Reproduction Assessment

Yes for the PR finding from source inspection: `stripEmojiAndSymbols` replaces matched sequences with `""`, so `hello✅world` would become `helloworld`. I did not execute tests because this review is read-only.

## Solution Assessment

No as written; centralizing filtering in speech-core is maintainable, but the helper should preserve word boundaries before this opt-in config is merged.
