## Summary

Keep open: the PR is a useful response to #77969, but the current patch checks a non-existent root `cfg.tts`, omits the strict config schema, and filters only the agent `tts` tool while shared TTS surfaces still send unfiltered text. The supplied terminal proof shows gateway status and branch commits, not an actual after-fix TTS conversion.

## What This Changes

The PR adds a `skipEmojiSymbols` TTS config type field, strips selected emoji/symbol characters in the agent `tts` tool, and adds tool-level tests.

## Best Possible Solution

Implement TTS text preparation in the shared speech pipeline using the effective `messages.tts` policy, with schema, docs, changelog, and regression coverage for the chosen emoji/symbol/path behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new configurable TTS filtering feature. The patch defects are source-reproducible from current main's config resolution, strict schema, and shared TTS call paths.

## Solution Assessment

No. Filtering selected code points inside `createTtsTool` is too narrow; the maintainable fix belongs in shared TTS preparation under `resolveEffectiveTtsConfig` after maintainers decide the exact filtering semantics.
