## Summary

Keep open: the shared TTS filtering approach and supplied real-provider proof are strong, but the new public TTS config key is not accepted by Feishu's strict channel/account override schema even though the docs say those overrides use the same shape as `messages.tts`.

## What This Changes

The PR adds an opt-in `messages.tts.skipEmojiSymbols` setting, shared speech-core filtering and empty-text guards for file, stream, and telephony synthesis, config/docs metadata, and focused TTS tests.

## Best Possible Solution

Land after Feishu top-level and account-level TTS overrides accept the opt-in key with focused schema coverage and regenerated config artifacts; keep the broader long path/file elision request in the linked issue separate.

## Reproduction Assessment

Yes, source-level for the remaining blocker: on the PR head, `skipEmojiSymbols` is in `TtsConfigSchema` but absent from Feishu's strict `TtsOverrideSchema`, so the documented Feishu channel/account override paths would reject it. I kept the checkout read-only and did not run a generated parse test.

## Solution Assessment

No, not yet: centralizing filtering in speech-core is the right runtime shape, but the public config contract needs Feishu override parity or an explicit root-only scope before merge.
