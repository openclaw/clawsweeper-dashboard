## Summary

Keep open: the fix is aimed at a valid memory-core bug and now has terminal proof, but the patch still misses marker-only placeholder shapes from the linked report and adds an ASCII-only relocation gate that can regress non-English snippets.

## What This Changes

The PR adds shared short-term promotion placeholder filtering, applies it across record/rank/apply paths, adds a relocation substring guard, and adds focused memory-core regression tests.

## Best Possible Solution

Broaden the shared unpromotable-snippet predicate to require Unicode letter/number content after stripping markdown structure, keep ordinary content bullets promotable, and make relocation substring matching Unicode-aware with regression tests for the reported marker-only cases.

## Reproduction Assessment

Yes. Current main’s source path shows non-empty placeholders pass the existing gates, and the PR body now includes before/after terminal output from a real temporary workspace exercising the affected promotion functions.

## Solution Assessment

No. Centralizing the filter is the right direction, but this implementation is still too narrow for marker-only snippets and should not add an ASCII-only relocation constraint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject marker-only placeholders:** `extensions/memory-core/src/short-term-promotion.ts:330`
  - body: The linked report at https://github.com/openclaw/openclaw/issues/80582 calls out malformed marker-only snippets such as `- -`, `>`, and fence-only fragments. This predicate only rejects empty, single bare list markers, and heading-only lines, so those snippets still flow through the new shared `isUnpromotableShortTermSnippet` gates and can be recorded, ranked, or applied; strip markdown structure and require real letter/number content before treating a snippet as promotable.
  - confidence: 0.9
- **[P2] Handle Unicode relocation text:** `extensions/memory-core/src/short-term-promotion.ts:342-345`
  - body: The new relocation guard tokenizes only ASCII words with `/[^A-Za-z0-9_]+/`, so meaningful CJK or other non-ASCII memory text produces no tokens and loses substring relocation even when the target and window contain each other. Use Unicode letter/number checks or a Unicode-aware tokenizer so non-English snippets still relocate while marker-only fragments stay blocked.
  - confidence: 0.82
