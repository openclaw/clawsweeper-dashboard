## Summary

Keep open: the PR addresses the main bare-bullet path, but the new predicate still admits marker-only placeholder snippets called out in the linked discussion, and the external PR proof is test-only.

## What This Changes

The PR adds memory-core short-term promotion placeholder filtering and relocation substring guards, plus regression tests for bare bullets and heading-only skeleton ranges.

## Best Possible Solution

Broaden the shared unpromotable-snippet predicate to reject marker-only markdown after stripping structure, preserve Unicode letter/number content as meaningful, apply it consistently across promotion entry points, and add focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, source-level: current main only rejects empty or contaminated snippets in the record/rank/apply path, so a non-empty placeholder can pass. I did not run filesystem tests because this review was read-only.

## Solution Assessment

No. Centralizing placeholder filtering is the right direction, but this implementation is too narrow for the reported marker-only cases and needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject marker-only placeholder snippets:** `extensions/memory-core/src/short-term-promotion.ts:330`
  - body: The linked discussion reports malformed promoted placeholders beyond a single bare bullet, including `- -`, `>`, fence-only, and other marker-only fragments. This predicate only returns true for empty, bare `-/*/+`, or heading-only lines, so those fragments still flow through every new `isUnpromotableShortTermSnippet` callsite and can be appended to `MEMORY.md`; strip markdown markers and require real letters/numbers before treating a snippet as meaningful.
  - confidence: 0.89
- **[P2] Handle Unicode text in relocation matching:** `extensions/memory-core/src/short-term-promotion.ts:342-345`
  - body: The new meaningful-substring check tokenizes only `[A-Za-z0-9_]`, so meaningful CJK or other non-ASCII snippets produce zero tokens and lose substring relocation even when the target and window contain each other. Use Unicode letter/number checks or an existing tokenizer so non-English memory still relocates while marker-only fragments stay blocked.
  - confidence: 0.82
