## Summary

Keep open: the PR addresses a real current-main bug and the supplied terminal proof is enough for the main Linux/macOS path, but the new Claude command detector misses generated wrapper commands launched by `node.exe` on Windows.

## What This Changes

The branch adds a `claude-agent-acp` command detector, suppresses `timeout`/`timeout_seconds` config forwarding for Codex or Claude ACP commands, and adds ACPX runtime tests.

## Best Possible Solution

Keep the ACPX timeout suppression, but make the backend detector match generated wrappers on all supported Node launch names and cover that with focused regression tests.

## Reproduction Assessment

Yes. Current main emits `timeout` from `timeoutSeconds` and forwards it for non-Codex ACP commands, while upstream `claude-agent-acp@0.33.1` rejects unknown config options; the PR body also includes terminal output showing the same rejection and the patched wrapper suppression.

## Solution Assessment

Not yet. Suppressing the unsupported wire option in ACPX is the narrow maintainable fix because OpenClaw still enforces the timeout locally, but the detector needs to cover `node.exe` generated wrapper commands before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Accept node.exe for generated Claude wrappers:** `extensions/acpx/src/runtime.ts:412`
  - body: The generated ACP wrapper command is built from `process.execPath`, which is `node.exe` on Windows, but this detector only checks the wrapper script after an exact `node` launcher. Bundled Claude ACP on Windows will miss the new suppression and still forward `timeout`, so `sessions_spawn` with `timeoutSeconds` can keep failing there; accept `node.exe` and add a regression test.
  - confidence: 0.86
