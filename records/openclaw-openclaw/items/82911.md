## Summary

Current main still threads stale ambient `webchat` channel context through the message tool, and the outbound runner can convert that exact context into an internal UI source-reply sink before normal channel resolution. The related reports remain useful context, but this issue describes a distinct, narrow routing defense that is not implemented yet.

## What This Changes

The issue requests that `message` tool delivery prefer the non-webchat channel and target encoded by the agent session key when ambient current-channel context has drifted to webchat.

## Best Possible Solution

Implement a narrow effective-current-channel resolver in `createMessageTool`, backed by a regression test that stale webchat ambient context cannot override a routable non-webchat session key.

## Reproduction Assessment

Yes, at source level: current main still passes stale `currentChannelProvider: webchat` into `toolContext`, and the outbound runner maps that to `sourceReplySink: internal-ui` before channel resolution. I did not perform a live Signal run.

## Solution Assessment

Yes, the proposed guarded session-key preference is the narrowest maintainable direction for this issue. It should be implemented in source with tests, not as a generated `dist` patch, while leaving the broader upstream contamination issue separate.

In `src/agents/tools/message-tool.ts`, compute effective current channel/current target values in `createMessageTool` before schema discovery, description discovery, secret-scope fallback, and runtime `toolContext`; prefer values derived from `options.agentSessionKey` only when the ambient provider normalizes to `webchat`, the session key identifies a non-webchat channel, and the inferred target is routable. Preserve normal webchat behavior and explicit `channel`/`target` calls. Add focused coverage in `src/agents/tools/message-tool.test.ts` proving stale webchat ambient context plus a Signal-style session key does not pass webchat into the runner.
