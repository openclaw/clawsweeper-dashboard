## Summary

Keep open: current main still builds the message tool schema, description, secret scope, and runtime `toolContext` from stale ambient `currentChannelProvider/currentChannelId`, and the outbound runner still short-circuits `webchat` message-tool-only sends to the internal UI sink before normal channel resolution.

## What This Changes

The issue requests a guarded session-key-derived delivery context in `createMessageTool` so stale ambient `webchat` state cannot override a routable non-webchat session when the model sends an untargeted reply.

## Best Possible Solution

Implement a small effective-current-channel resolver in `createMessageTool`, use it consistently for schema, description, secret scoping, and runtime `toolContext`, and keep the broader upstream session contamination work in https://github.com/openclaw/openclaw/issues/36614 separate.

## Reproduction Assessment

Yes, at source level: current main forwards stale `currentChannelProvider: webchat` into `toolContext`, and `runMessageAction` maps that exact context to `sourceReplySink: internal-ui` before normal channel resolution. I did not perform a live Signal run.

## Solution Assessment

Yes, the proposed guarded session-key preference is the narrowest maintainable direction for this issue. It should be implemented in source with regression coverage, not as a generated `dist` patch, while leaving the upstream contamination issue separate.
