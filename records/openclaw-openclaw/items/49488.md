## Summary

Keep open. Current main still requires Android API 31, so the requested older-device support is not implemented there; the PR is useful and has real device/API 26 context in the discussion, but it is still dirty against base and leaves API 26-28 compatibility gaps around assistant-role APIs and motion command advertising.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds API-level fallbacks or guards for dynamic colors, foreground service startup, DNS discovery, location, notification access, thermal status, and motion UI/permission reporting.

## Best Possible Solution

Decide the Android support floor explicitly. If API 26 support is accepted, rebase this PR and land a comprehensive compatibility pass that guards every API 29+ runtime/UI surface, keeps advertised node capabilities aligned with invocable commands, updates Android docs to the new floor, and includes focused API 26, 29, 30, and current-device validation.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to apply the PR support floor, then open Settings on API 26-28 for the unguarded RoleManager path, or run an API 26-28 node with motion sensors and observe motion commands advertised while invocations fail the API-29 ACTIVITY_RECOGNITION permission check.

## Solution Assessment

No. The direction may be valid and the PR addresses earlier foreground-service and DNS blockers, but this branch is not the best mergeable solution until it is rebased and the remaining API-level gates and runtime capability alignment are fixed.
