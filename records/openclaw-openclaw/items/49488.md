## Summary

Keep open: the API 26 support floor is not on current main, a human review has requested changes on the support-floor direction, and the latest branch still needs broader API 26-30 compatibility work plus stronger real-device or emulator proof before it is mergeable.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds API guards or fallbacks for theme colors, foreground service startup, DNS discovery, location, notifications, thermal state, and motion UI paths.

## Best Possible Solution

Decide the official Android support floor; if API 26 is accepted, rebase and land a complete API 26-30 compatibility pass with guards for every reachable newer-API surface and device or emulator proof for the affected flows.

## Reproduction Assessment

Yes for the review blockers by source inspection: lowering minSdk to 26 makes API 29+ assistant-role and activity-recognition motion paths reachable or advertised on API 26-28 unless they are fully guarded. I did not run an emulator in this read-only review.

## Solution Assessment

No. Lowering minSdk may be worthwhile, but this PR is not the best mergeable solution until maintainers accept the support floor, the branch is rebased, all API 26-30 gaps are audited, and runtime proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard assistant-role APIs before lowering minSdk:** `apps/android/app/build.gradle.kts:66`
  - body: Lowering `minSdk` to 26 means this branch must merge with current main's SettingsSheet assistant-role path, where `RoleManager` and `ROLE_ASSISTANT` are API 29 surfaces. Without an SDK guard or exclusion, API 26-28 devices can hit that path as soon as Settings computes assistant role state, so the minSdk change should stay blocked until this current-main path is made safe.
  - confidence: 0.88
- **[P2] Apply the motion SDK gate to runtime registration:** `apps/android/app/src/main/java/ai/openclaw/app/tools/Features.kt:9`
  - body: `Features.Motion` hides some UI and permission payload state below Android Q, but NodeRuntime still advertises motion availability from sensor presence and MotionHandler still requires the API-29 `ACTIVITY_RECOGNITION` permission. On API 26-28 devices with sensors, `motion.*` commands can still be registered and then fail with `MOTION_PERMISSION_REQUIRED`; the SDK gate needs to apply to command/capability registration and handler checks too.
  - confidence: 0.87
