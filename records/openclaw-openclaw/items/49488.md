## Summary

Keep open: current main still requires API 31, and this PR still has merge-blocking API 26-28 compatibility gaps around assistant-role settings and advertised motion commands. The discussion also includes a maintainer concern that lowering the Android floor is a broader support decision than this branch currently proves.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds compatibility fallbacks or guards for dynamic colors, foreground service startup, DNS discovery, location, notification access, thermal status, and motion UI/permission reporting.

## Best Possible Solution

Decide the Android support floor explicitly; if API 26 is accepted, rebase this PR and land a comprehensive compatibility pass that guards every API 29+ surface, keeps advertised node capabilities aligned with invocable commands, updates user-facing docs/changelog, and includes API 26, 29, 30, and current-device validation.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to apply the PR support floor, then open Settings on API 26-28 for the unguarded RoleManager path or connect a sensor-equipped API 26-28 node and observe motion commands advertised while invocations fail the API-29 ACTIVITY_RECOGNITION permission check.

## Solution Assessment

No. The direction may be valid and the PR fixes several earlier blockers, but it is not the best mergeable solution until the remaining API-level guards, runtime capability alignment, rebase, and support-floor decision are resolved.
