Keep open: current main has not lowered the Android support floor, and the PR still needs maintainer product approval plus source-backed compatibility fixes before merge.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds pre-API compatibility guards or fallbacks for theme colors, foreground service startup, DNS, location, notifications, thermal status, motion, and permission UI paths.

## Best Possible Solution

Decide the Android support floor explicitly; if API 26 is accepted, rebase this work and land a comprehensive API 26-30 compatibility pass with guarded API 29/30/31+ surfaces and device/emulator proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the PR findings: apply the minSdk 26 floor, then open Settings on API 26-28 for the unguarded RoleManager path or advertise/invoke motion on a pre-Q sensor-equipped device. I did not run an emulator in this read-only review.

## Solution Assessment

No. The PR fixes several compatibility paths, but the current branch is not the best mergeable solution until maintainers accept the support floor and the remaining assistant-role and runtime motion gates are fixed and validated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard assistant-role APIs below Android Q:** `apps/android/app/build.gradle.kts:66`
  - body: Lowering minSdk to 26 makes API 26-28 devices supported, but Settings still initializes RoleManager-backed state and launches ROLE_ASSISTANT without an SDK guard. RoleManager is an API-29 class, so opening Settings on pre-Q devices can crash; hide or guard the assistant role block and helpers below Q.
  - confidence: 0.86
- **[P2] Gate runtime motion availability below Android Q:** `apps/android/app/src/main/java/ai/openclaw/app/tools/Features.kt:9`
  - body: Features.Motion gates parts of the UI, but NodeRuntime still advertises motion commands from sensor availability while MotionHandler still requires ACTIVITY_RECOGNITION. On API 26-28 that permission is not in the platform API, so exposed motion commands can be advertised but cannot succeed; include the SDK gate in runtime availability or handler behavior.
  - confidence: 0.88
