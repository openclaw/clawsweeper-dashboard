## Summary

Keep open: older-Android support is not implemented on main and this PR contains useful work, but it is conflicting/stale and needs a support-floor decision, a complete API 26-30 audit, and stronger real-device proof before it can merge.

## What This Changes

The branch lowers the Android app minSdk from 31 to 26 and adds API-level fallbacks or guards for theme colors, foreground service startup, DNS discovery, location, notification access, thermal state, and motion UI/settings paths.

## Best Possible Solution

Treat API 26 support as a dedicated Android compatibility project: rebase, audit every current-main Android API use against SDK contracts, align advertised capabilities with runtime behavior, update docs/tests, and land only with API 26-30 real-device or emulator proof; otherwise pause this branch and keep minSdk 31.

## Reproduction Assessment

Yes for the review blockers by source inspection: current main calls API 29/30 Android APIs that become reachable if minSdk drops to 26, and the PR's motion gate does not align command advertisement with handler permission behavior. I did not run an API 26 emulator in this read-only review.

## Solution Assessment

No, not as currently proposed for merge. The direction is useful, but the safest solution is either an explicit maintainer decision to keep minSdk 31 or a rebased API 26 compatibility project with guards, tests, docs, and real-device/emulator proof for the supported flows.
