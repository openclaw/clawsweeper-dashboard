## Summary

Keep open. The PR contains useful Android compatibility work, but current main still sets and documents `minSdk=31`, live PR metadata is changes-requested, and lowering the support floor still needs a maintainer product decision plus a broader API 26-30 compatibility pass and stronger proof.

## What This Changes

The PR lowers the Android app `minSdk` to 26 and adds compatibility guards or fallbacks for theme colors, foreground service startup, DNS discovery, location, notification access, thermal status, and motion UI/device permission reporting.

## Best Possible Solution

Decide the Android support floor explicitly; if API 26 is accepted, rebase this work and land a comprehensive API 26-30 compatibility pass with guarded API 29/30/31+ surfaces and real device or emulator proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path once the PR lowers `minSdk` to 26: current-main Settings uses API-29 `RoleManager`, and PR-head runtime motion availability can still be advertised from sensors while invocation requires the API-29 `ACTIVITY_RECOGNITION` permission.

## Solution Assessment

No. The PR contains useful compatibility guards, but it is not the best mergeable solution until the support-floor decision is made, the remaining API 26-30 gates are fixed, and the affected flows are validated on real or emulated devices.
