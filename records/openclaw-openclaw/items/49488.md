## Summary

Keep open: current main still requires Android API 31, the PR is merge-conflicting, and the requested API 26 support needs a broader compatibility audit plus stronger device proof before maintainers can take the support-floor change.

## What This Changes

The PR lowers the Android app minimum SDK to 26 and adds older-API fallbacks for theme colors, foreground service startup, DNS discovery, permissions, motion UI, notification access, location, and thermal status.

## Best Possible Solution

Treat API 26 support as a deliberate Android support-floor project: rebase, audit every reachable API 29/30/31+ path, share capability gates from UI through runtime command handling, and require API 26-28 proof before lowering minSdk.

## Reproduction Assessment

Yes for the review blocker by source inspection: Android SDK metadata marks ACTIVITY_RECOGNITION as API 29+, while the PR's SDK gate is not propagated to runtime motion command advertisement or invocation. I did not run an API 26 emulator in this read-only review.

## Solution Assessment

No as-is. Lowering the support floor may be the right direction, but the maintainable path needs a current-main rebase, a complete API 26-30 audit, runtime capability gating, and real older-device proof before merge.
