## Summary

Keep open: the older-Android support direction is useful, but the PR is still dirty/conflicting against current main and has a source-reproducible runtime capability mismatch for motion on API 26-28.

## What This Changes

The PR lowers the Android app minSdk to 26 and adds older-API fallbacks for theme colors, foreground service startup, DNS discovery, permissions, notification access, location, thermal status, and motion UI state.

## Best Possible Solution

Rebase onto current Android main, make an explicit maintainer decision that API 26 is supported, guard every reachable newer-API runtime path including command advertisement and invocation, and prove API 26-28 plus current API 31+ behavior before merge.

## Reproduction Assessment

Yes. The review blocker is reproducible by source inspection: Android API metadata puts ACTIVITY_RECOGNITION at API 29, while the PR gates UI/permissions below API 29 but leaves runtime motion command advertisement and invocation driven by sensor availability and permission checks.

## Solution Assessment

No. The support-floor change should be a full Android API 26-30 audit and rebase over current main, not a partial UI/permission gate that leaves runtime command behavior inconsistent.
