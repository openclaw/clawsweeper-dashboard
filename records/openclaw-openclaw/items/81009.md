## Summary

Keep open. The feature direction is plausible, but current head still exposes `/progress` as a shared command while only Telegram honors the session override, can report the wrong status when config disables tool progress, and lacks inspectable real Telegram behavior proof.

## What This Changes

Adds a shared `/progress` command/directive, a `SessionEntry.previewToolProgress` override, Telegram dispatch consumption, and docs/tests for session-level tool-progress preview toggling.

## Best Possible Solution

Make the command scope, runtime consumers, status text, docs, and regression coverage agree, then attach redacted live Telegram proof showing `/progress off` and `/progress on` changing a real chat stream.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows the shared command is broader than the only runtime consumer, and the status path hard-codes an unset override as `on (config)`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The intended UX is reasonable, but this is not the best mergeable shape until shared-versus-Telegram scope and effective status reporting are fixed and real Telegram proof is attached.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Make `/progress` affect every exposed surface:** `src/auto-reply/commands-registry.shared.ts:831-834`
  - body: Registering `/progress` in the shared built-in command catalog exposes it outside Telegram, and the shared directive handler can acknowledge and persist the override for those surfaces. Only Telegram dispatch reads the session override; Discord, Slack, Matrix, Teams, and Mattermost still use config-only progress checks, so `/progress off` can report success while progress lines continue. Either scope this to Telegram or thread the override through every surface that exposes it.
  - confidence: 0.91
- **[P2] Report the configured progress state:** `src/auto-reply/reply/directive-handling.impl.ts:234-236`
  - body: When no session override exists, this status path hard-codes the state as `on (config)`. A user with `streaming.preview.toolProgress: false` will be told progress is on even though the resolver suppresses it, so `/progress` needs to compute the same effective session/config value that dispatch uses.
  - confidence: 0.86
