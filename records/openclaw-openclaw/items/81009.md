## Summary

Keep open. The PR is a plausible implementation of the linked feature request, but the new `/progress` directive is not wired through the full apply path, so progress-only commands return before persistence or acknowledgement, and the external PR still lacks artifact-backed real Telegram proof.

## What This Changes

Adds a session-level `/progress` command/directive, `SessionEntry.previewToolProgress`, resolver override support, and Telegram dispatch plumbing/tests for tool-progress preview visibility.

## Best Possible Solution

Wire progress into the full directive application path, compute status from the effective session/config resolver, add coverage through the real resolve/apply path, and attach redacted Telegram proof before maintainer acceptance.

## Reproduction Assessment

Yes for the PR-introduced bug: source inspection shows parsed progress directives are omitted from `applyInlineDirectiveOverrides`' directive gate, so progress-only commands return before persistence. I did not run tests because this is a read-only review.

## Solution Assessment

No. The requested direction is reasonable, but this patch is not the best mergeable solution until the directive is routed through every apply gate, status reflects effective config, and real Telegram behavior is proven.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route progress directives through the apply gate:** `src/auto-reply/reply/get-reply-directives.ts:309`
  - body: Adding `hasProgressDirective` here makes the parser notice `/progress`, but `applyInlineDirectiveOverrides` still computes `hasAnyDirective` without that flag. A message containing only `/progress off`, `/progress on`, or `/progress default` will be treated as having no directive and return before the new persistence/ack code runs, so the advertised command does not toggle anything unless another directive happens to be present.
  - confidence: 0.93
- **[P2] Report the effective configured progress state:** `src/auto-reply/reply/directive-handling.impl.ts:236`
  - body: The status path reports an unset session override as `on (config)` without reading the channel streaming config. When `streaming.preview.toolProgress` is configured `false`, `/progress` would tell the user progress is on even though the resolver suppresses it, so the status command needs to use the effective config/session value.
  - confidence: 0.85
