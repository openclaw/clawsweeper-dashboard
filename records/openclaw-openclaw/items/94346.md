Keep open: the linked cron bug is valid and the patch moves in the right direction, but the new deferral state is still opt-in at call sites, leaves sibling maintenance recomputes and cleanup uncovered, and lacks after-fix real behavior proof from this branch.

## What This Changes

The PR adds `deferredCatchupJobIds` to cron service state, assigns it after startup catch-up, and passes it as `skipFutureRepairJobIds` from read and timer maintenance recomputes.

## Best Possible Solution

Make pending startup deferrals a canonical cron maintenance invariant, clear each id when its deferred slot is consumed, and prove the read/status plus non-read recompute sibling paths on the patched branch.

## Reproduction Assessment

Yes. The linked issue gives a high-confidence source reproduction: startup defers an overflow cron job, then a maintenance recompute from read/status advances that deferred slot to the natural schedule before it fires.

## Solution Assessment

No. Persisting the ids in state is the right direction, but the patch does not make the exemption canonical for all maintenance recompute callers and does not clear pending ids after consumption.
