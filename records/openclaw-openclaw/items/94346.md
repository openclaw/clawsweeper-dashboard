## Summary

Keep open: the PR points at a valid cron deferral-loss bug, but this branch leaves the new deferred-job skip set alive for the rest of the service lifetime and lacks branch-specific after-fix real behavior proof.

## What This Changes

The PR stores startup deferred catch-up job IDs in `CronServiceState` and threads that set into read and timer maintenance recomputes.

## Best Possible Solution

Use one canonical pending catch-up deferral state that is populated when deferrals are assigned, consulted by every maintenance recompute caller, and cleared when each deferred slot is reached or the job disappears.

## Reproduction Assessment

Yes. The linked issue and current source show a clear source-level path: startup produces a deferred catch-up skip set, while read and timer maintenance recomputes on current main do not receive it.

## Solution Assessment

No. Moving the exemption into service state is the right direction, but the set must represent pending deferrals only and this branch never clears IDs after the deferred slot is reached.

- **[P2] Clear deferral IDs after the catch-up slot fires:** `src/cron/service/ops.ts:265`
  - body: This assignment makes every later read/timer recompute pass the same deferred ID set, but nothing removes an ID once its staggered catch-up run has fired.
