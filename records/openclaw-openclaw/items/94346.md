## Summary

Keep open: current main still needs a fix for the linked cron deferral bug, but this branch is not merge-ready because it leaves some maintenance recomputes unguarded and the new pending-deferral Set has no expiry path.

## What This Changes

The PR stores startup overflow cron catch-up deferral job IDs in cron service state and passes that Set to selected read and timer maintenance recomputes.

## Best Possible Solution

Make pending startup deferrals a single cron maintenance invariant, clear IDs when their deferred slot is consumed, cover read/status plus non-read recompute siblings with regression tests, and add real branch proof.

## Reproduction Assessment

Yes. The linked issue and current source show startup can defer an overflow cron job, then read/status or timer maintenance can call future-slot repair without that deferral skip before the staggered tick; I did not execute tests because this review is read-only.

## Solution Assessment

No. Persisting pending IDs in state is the right direction, but this PR leaves non-read ops.ts recomputes unguarded and does not expire the pending IDs after the deferred run.

Full review comments:

- **[P1] Make pending deferrals central and expiring:** `src/cron/service/state.ts:217`
  - body: Adding `deferredCatchupJobIds` to state and manually passing it through selected call sites leaves other maintenance recomputes, such as manual-run preflight/finalization in `ops.ts`, calling `recomputeNextRunsForMaintenance(state)` with future repair enabled. Those paths can still advance a pending overflow deferral to its natural schedule before the staggered catch-up fires, and the Set is never cleared after the deferred slot is consumed, suppressing later future-slot repair for those job IDs. Put the pending-deferral check in one canonical maintenance path and clear IDs when their catch-up slot is reached or executed.
