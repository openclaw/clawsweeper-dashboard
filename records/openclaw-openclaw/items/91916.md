## Label Justifications

- `P1`: Independent users on shipped releases report a broken Feishu workflow that can leave only the last character or an error instead of the correct reply.
- `impact:message-loss`: Correct assistant content is visibly truncated or overwritten in the delivered Feishu card.

## Summary

This issue is still necessary: current main deliberately replaces accumulated Feishu stream text with each later final payload, while independent users still reproduce severe card truncation on shipped v2026.6.5 after the cumulative-update fix. The truncation portion overlaps https://github.com/openclaw/openclaw/issues/88867, but the valid-reply-then-tool-error overwrite sequence is distinct and not covered by the current tests or a proven canonical PR.

## What This Changes

Fix Feishu streaming cards so long replies remain complete and later tool failures do not replace an already-correct assistant reply.

## Best Possible Solution

Define one canonical assistant-visible content state across partial, block, final, idle, and failed-tool events; preserve valid reply text when later diagnostics arrive; then prove long replies and failed-tool sequences in a real Feishu desktop/mobile card before choosing any Card Kit layout or append-mode fallback.

## Reproduction Assessment

Yes at source level for the later-final replacement mechanism, with multiple shipped-release reports supporting truncation; a live current-main Feishu trace is still required to isolate Card Kit payload loss from client-side clipping.

## Solution Assessment

No, none of the current one-sided proposals is yet proven best; the maintainable fix must first define multiple-final/error ownership in the dispatcher and then validate the real Card Kit rendering boundary.
