## Label Justifications

- `P1`: The shipped default Feishu streaming workflow can reduce a complete response to one character or replace useful output, breaking real channel use.
- `impact:message-loss`: Correct agent reply content is visibly truncated or overwritten in the Feishu card.

## Summary

This remains necessary: v2026.6.5 includes the earlier cumulative CardKit update fix, yet multiple users still reproduce severe truncation, and current main still assigns final text directly over accumulated `streamText`; the additional tool-error overwrite symptom is not fully owned by the narrower open truncation report.

## What This Changes

Fix Feishu streaming cards so long accumulated replies remain complete after finalization and subsequent tool errors do not replace an already-correct reply.

## Best Possible Solution

Define one monotonic visible-reply lifecycle in the Feishu dispatcher: merge final text with accumulated streaming content when appropriate, preserve a committed valid answer from later tool-error events, close the card with the canonical full text, and cover both behaviors with focused tests plus live Feishu proof.

## Reproduction Assessment

Yes at source and shipped-report confidence: multiple users reproduce on v2026.6.5 with a controlled configuration matrix, and current source contains a final assignment that can replace accumulated streaming text. A live current-main Feishu run is still required for `reproduced` status.

## Solution Assessment

No, the earlier streaming-card-only fix is not the complete solution. The narrow maintainable repair is to fix final/error precedence in the owning Feishu dispatcher while retaining cumulative CardKit updates.
