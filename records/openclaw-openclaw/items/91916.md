## Summary

This issue remains necessary because current main still replaces accumulated Feishu streaming text with any distinct final payload, allowing a late tool error to overwrite the correct answer; the truncation symptom overlaps an existing report but does not make this distinct defect redundant.

## What This Changes

Report Feishu streaming cards truncating long replies and replacing already-correct assistant text with later tool-error finals.

## Best Possible Solution

Keep accumulated assistant output authoritative through closeout, present later tool failures non-destructively, and use tested chunked or static delivery when complete final content cannot safely fit the streaming-card contract.

## Reproduction Assessment

Yes for the overwrite defect: current main deterministically replaces accumulated `streamText` with a distinct final payload before card close. The precise long-card failure mechanism still requires a live Feishu trace.

## Solution Assessment

No, closing this as wholly duplicated would lose the distinct overwrite defect; the narrow maintainable repair is in the Feishu dispatcher/session boundary, with maintainer confirmation of how late tool errors should be displayed.
