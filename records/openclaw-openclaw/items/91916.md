## Label Justifications

- `P1`: Feishu users can lose substantial or complete intended reply content during ordinary streamed agent and tool workflows.
- `impact:message-loss`: The issue concerns silently truncated replies and replacement of already visible answer content.

## Summary

Current main still has both reported failure classes: long final text is diverted away from the active streaming card without first preserving its preview, while a later distinct final payload can become replacement content for that same card. The truncation symptom overlaps https://github.com/openclaw/openclaw/issues/91159, but the overwrite behavior is a separate, source-backed Feishu dispatcher defect, so this issue remains necessary.

## What This Changes

Prevent Feishu streaming cards from truncating long replies or replacing a completed answer with a later tool-error message.

## Best Possible Solution

Keep one canonical cumulative visible-answer state through Feishu streaming finalization, route oversized or Card Kit-incompatible output to a complete fallback without leaving a truncated preview, and append or separately deliver actionable tool errors instead of replacing the completed answer.

## Reproduction Assessment

Yes—current main provides a high-confidence source reproduction: oversized finals leave the streaming-card path, and non-extending final content is explicitly replaced during card close. A live Feishu account would still be needed to verify the exact Card Kit limit and end-user rendering.

## Solution Assessment

Yes, a bounded repair in the existing Feishu dispatcher and streaming-card lifecycle is the best path; new configuration or a parallel channel delivery mechanism would add unnecessary sources of truth.
