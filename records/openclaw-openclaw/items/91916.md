- `impact:message-loss`: The reported behavior drops accumulated reply text or replaces a valid visible reply with later error content.

## Summary

This remains a valid Feishu reliability issue: current main still replaces the accumulated streaming buffer with each `final` payload, and the report adds a distinct late tool-error overwrite symptom that is not clearly covered by the existing related work. The issue should stay open while maintainers decide whether to extend the active incremental-final repair or track the error-arbitration path separately.

## What This Changes

Fix Feishu streaming cards so long replies retain their complete accumulated content and later tool errors do not replace an already-valid assistant reply.

## Best Possible Solution

Keep one canonical Feishu reply state per lifecycle, merge only proven cumulative or incremental assistant-final payloads, and prevent later tool-error output from replacing a completed visible reply; add focused dispatcher tests for long multi-chunk output, trailing final deltas, complete rerenders, unrelated finals, and reply-then-tool-error ordering.

## Reproduction Assessment

Yes at source level, with medium confidence. The direct `streamText = text` final-payload path reproduces accumulated-text loss, but the exact elevated-tool failure sequence has not been established in a live Feishu tenant or a focused current-main test.

## Solution Assessment

Unclear as a single existing patch. https://github.com/openclaw/openclaw/pull/91698 plausibly repairs incremental finals, but this issue's tool-error overwrite behavior also needs explicit reply-lifecycle arbitration and regression coverage before the issue is resolved.
