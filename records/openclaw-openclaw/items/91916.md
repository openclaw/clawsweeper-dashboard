## Summary

This issue remains necessary: the shipped cumulative-update fix did not resolve Feishu truncation, and current main still permits a later non-prefix final payload to replace already-visible card content. The related open issue covers the last-character symptom, but this report adds a distinct failed-tool overwrite sequence that should remain in the canonical repair scope.

## What This Changes

Fix Feishu streaming cards that truncate long replies and allow later tool errors to overwrite an already-correct response.

## Best Possible Solution

Maintain one explicit Feishu visible-reply state that distinguishes transient progress, accepted assistant output, and tool errors; close the card from that state without letting later errors erase accepted text, and validate both long-response completion and failed-tool sequencing in a real Feishu chat.

## Reproduction Assessment

Yes, at high confidence from source and shipped-user evidence: current finalization can replace accumulated card content, and reporters provide repeatable streaming/non-streaming comparisons and a five-case configuration matrix. A real Feishu run against current main remains the final transport-level proof gap.

## Solution Assessment

No, treating every non-prefix final payload as authoritative replacement is not the best solution. The maintainable fix is to model transient status, accepted assistant text, and later tool errors separately, then close from the canonical visible state.
