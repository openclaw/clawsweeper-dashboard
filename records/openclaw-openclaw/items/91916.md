## Summary

The long-response truncation overlaps the still-open canonical report at https://github.com/openclaw/openclaw/issues/88867, but this issue also describes a distinct later tool-error overwrite sequence. Current main contains newer final-card replacement and duplicate-final logic, yet there is no real Feishu proof that it fixes either the shipped 2026.6.5 truncation or the valid-answer overwrite, so closing now would risk losing actionable behavior.

## What This Changes

Fix Feishu streaming cards so long responses remain complete and later tool failures do not replace an already-correct assistant reply.

## Best Possible Solution

Consolidate the last-character truncation evidence into https://github.com/openclaw/openclaw/issues/88867, then reproduce the valid-answer-followed-by-tool-error sequence on current main and fix the Feishu dispatcher so only the authoritative assistant final can replace accumulated card content.

## Reproduction Assessment

No, not yet for current main. The shipped 2026.6.5 truncation has strong repeated real-Feishu reproduction, and current source exposes the relevant final-replacement path, but the later-error overwrite still needs an isolated current-main transcript and event sequence.

## Solution Assessment

Unclear until live verification. The existing Feishu dispatcher and streaming-session boundary is the right place to solve this, but a repair must distinguish authoritative final answers from later tool errors rather than broadly suppressing final updates.
