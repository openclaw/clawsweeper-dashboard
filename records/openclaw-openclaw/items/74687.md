## Summary

Keep open. The PR addresses a real current-main recovery gap, but its new channel-backed predicate is too broad and would also stop resumable internal WebChat/local main sessions from using the existing restart-recovery path; it also still needs the required user-facing changelog entry.

## What This Changes

The PR marks channel-backed restart-aborted main sessions failed for manual recovery before synthetic replay and adds a focused regression test for that path.

## Best Possible Solution

Keep the guard at the restart-recovery decision point, but base it on routable external delivery metadata rather than any origin/channel metadata, preserve automatic recovery for internal WebChat/local main sessions, add regression coverage for both paths, and add the required single-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a running `abortedLastRun` main session with a resumable tail: current main reaches `callGateway`, while the PR's predicate would mark the same session failed when it has persisted internal `webchat` origin/route metadata.

## Solution Assessment

No, not as written. The decision point is appropriate, but the predicate should require an external deliverable route and exclude internal WebChat/control-UI metadata instead of treating `origin.provider`, `origin.surface`, or `lastChannel` alone as channel-backed.
