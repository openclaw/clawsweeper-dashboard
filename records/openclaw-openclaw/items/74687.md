## Summary

Keep open: the bug-fix direction is plausible, but the current patch is overbroad and would treat ordinary WebChat/local session metadata as external channel backing; it also lacks real channel restart-recovery proof.

## What This Changes

The PR marks channel-backed restart-aborted main sessions failed for manual recovery instead of replaying a synthetic recovery turn, and adds a Discord-shaped regression test.

## Best Possible Solution

Keep the restart-recovery guard at this decision point, but base it on a normalized external deliverable route with a target and add WebChat/local plus external-channel regression coverage before merge.

## Reproduction Assessment

Yes for the review finding at source level: current main can store WebChat/local origin or last-route metadata and still resume a resumable aborted session, while the PR predicate would mark it failed. The original live Discord restart symptom is not yet proven with real after-fix evidence.

## Solution Assessment

No. The guard belongs in restart recovery, but the implementation should require a routable external delivery channel plus target and explicitly exclude internal WebChat/control-UI metadata.
