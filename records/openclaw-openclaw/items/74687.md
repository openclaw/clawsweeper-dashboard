## Summary

Keep open: the recovery safety concern is plausible, but the proposed predicate is overbroad and would regress internal WebChat/local restart recovery. The PR also needs real behavior proof beyond mocked/unit validation before merge.

## What This Changes

The PR marks channel-backed restart-aborted main sessions failed for manual recovery instead of replaying a synthetic recovery turn, with a Discord-shaped regression test.

## Best Possible Solution

Keep the restart-recovery guard, but classify only normalized external deliverable routes with a target as manual-recovery and preserve WebChat/local auto-recovery with regression coverage for both cases.

## Reproduction Assessment

Yes for the review finding at source level: a WebChat/local abortedLastRun session with origin or last-route metadata and a resumable tail currently resumes on main, while the PR predicate would mark it failed. I did not run a live restart E2E in this read-only review.

## Solution Assessment

No. The guard belongs at restart recovery, but the predicate should require a routable external delivery channel plus target and explicitly exclude internal WebChat/control-UI metadata.
