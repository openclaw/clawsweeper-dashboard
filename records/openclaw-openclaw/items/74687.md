## Summary

Keep open: the branch targets a real restart-recovery hazard, but the current predicate overclassifies ordinary stored WebChat/control UI route metadata as channel-backed and the PR still lacks real channel restart-recovery proof.

## What This Changes

The PR marks channel-backed restart-aborted main sessions failed for manual recovery instead of replaying a synthetic recovery turn, and adds a Discord-shaped regression test.

## Best Possible Solution

Keep the guard at the restart-recovery decision point, but base it on canonical routable external delivery context with a target, explicitly exclude WebChat/internal metadata, and add both internal-route and external-channel regression coverage plus redacted real channel proof.

## Reproduction Assessment

Yes at source level: current main can replay a marked channel-scoped restart-aborted main session through the hidden `deliver: false` recovery path. The PR-introduced false positive is also source-reproducible because WebChat/control UI metadata can populate the same fields that the new predicate treats as channel-backed.

## Solution Assessment

No. The decision point is plausible, but the implementation should require a normalized external deliverable route with a target and exclude internal WebChat/control UI metadata before marking a session failed.
