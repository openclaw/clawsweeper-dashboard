## Summary

Keep open. The PR targets a real restart-recovery risk, but the proposed predicate is too broad: current main shows internal WebChat/local main sessions can persist the same origin and route fields the PR treats as channel-backed, so the patch would disable automatic recovery outside its stated scope. It also lacks the required changelog entry for a user-visible fix.

## What This Changes

The PR marks channel/delivery/origin-tagged restart-aborted main sessions as failed for manual recovery and adds a focused regression test for a Discord-backed session.

## Best Possible Solution

Keep the guard at the restart-recovery decision point, but classify only real external deliverable routes as manual-recovery candidates, preserve automatic restart recovery for internal WebChat/local main sessions, add regression coverage for both paths, and add the required single-line changelog entry.

## Reproduction Assessment

Yes. Seed a running `abortedLastRun` main session with a resumable tail plus persisted internal WebChat origin or route metadata; current main reaches `callGateway`, while the PR predicate would mark that same session failed before replay.

## Solution Assessment

No. The decision point is appropriate, but the predicate should require external deliverable route evidence and exclude internal WebChat/control-UI metadata instead of treating origin or last-route fields alone as channel-backed.
