## Summary

Keep this open. The report matches a current source-level policy interaction: root or agent `tools.alsoAllow` can allow `message` at the profile layer, but sandboxed runs are filtered again by the sandbox tool policy, whose default allowlist does not include `message`.

## What This Changes

A sandboxed `openclaw-weixin` agent configured with `tools.alsoAllow: ["message"]` no longer receives the `message` tool at runtime.

## Best Possible Solution

Maintainers should choose one clear contract for sandboxed channel agents, then add regression coverage: either `message` is allowed by sandbox defaults for outbound channel delivery, or users get an explicit doctor/docs path to add `tools.sandbox.tools.alsoAllow: ["message"]`.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: root `tools.alsoAllow` can admit `message` at the profile layer, but sandbox default tool policy excludes it and is applied later. I did not perform a live weixin/provider run.

## Solution Assessment

Unclear. Passing root `alsoAllow` through sandbox policy would restore the reported workflow but may widen sandboxed side effects, so the best fix needs maintainer policy judgment before implementation.
