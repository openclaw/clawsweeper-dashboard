## Summary

Keep open: current main still has a separate sandbox tool-policy pass that can remove `message` after top-level `tools.alsoAllow` grants it, and the right fix requires a maintainer decision about the sandbox policy contract.

## What This Changes

The issue requests restoring `message` tool availability for sandboxed `openclaw-weixin` agents when top-level `tools.alsoAllow` includes `message`.

## Best Possible Solution

Decide the sandbox/tool-policy contract, then either restore compatibility for explicit top-level `message` grants in sandboxed channel agents or document and diagnose the required `tools.sandbox.tools.alsoAllow` grant.

## Reproduction Assessment

Yes, from source inspection: a sandboxed agent with ordinary top-level `tools.alsoAllow: ["message"]` but no sandbox-specific `message` grant can have `message` removed by the sandbox policy pass. I did not run a live Weixin session in this read-only review.

## Solution Assessment

Unclear: the narrow code path is identifiable, but the best fix depends on whether maintainers want top-level grants to widen sandbox policy or want strict sandbox-specific grants plus clearer diagnostics/docs.
