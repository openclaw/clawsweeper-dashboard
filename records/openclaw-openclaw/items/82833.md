## Summary

Keep open: the report is consistent with current source for sandboxed agents, where top-level `tools.alsoAllow` can expose `message` at the profile layer but the active sandbox tool policy can still remove it from the final tool list.

## What This Changes

The issue asks OpenClaw to keep the `message` tool available to a sandboxed Weixin agent when `tools.alsoAllow` explicitly includes `message`.

## Best Possible Solution

Decide the sandbox/tool-policy contract first: either restore the beta.2 compatibility path for explicit top-level `message` grants in sandboxed channel agents, or keep sandbox enforcement strict and update doctor/docs to point users to `tools.sandbox.tools.alsoAllow` for `message` and related channel tools.

## Reproduction Assessment

Yes, from source inspection: a sandboxed agent with ordinary `tools.alsoAllow: ["message"]` but no sandbox-specific message grant can have `message` removed by the sandbox policy pass. I did not run a live Weixin session in this read-only review.

## Solution Assessment

Unclear: restoring the old behavior may be the right regression fix, but because sandbox tool policy is a security boundary, maintainers should decide whether top-level grants should widen sandbox policy or whether diagnostics/docs should require `tools.sandbox.tools.alsoAllow`.
