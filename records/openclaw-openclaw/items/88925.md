## Summary

This PR still contains a necessary fix: current main can preserve recently progressing embedded work at the diagnostic coordinator, but automatic recovery through the session-id fallback still becomes a user abort. The proposed system-abort plumbing follows the existing reply-operation owner boundary and the focused tests cover the central regression; the PR should remain open for a clean rebase and real runtime proof.

## What This Changes

The PR propagates a `stuck_recovery` cancellation reason through reply and embedded-run abort paths and prevents recovery from aborting active runs with recent tracked progress.

## Best Possible Solution

Rebase onto current main, retain the truthful `stuck_recovery` terminal-state propagation and only the liveness checks not already present, then prove that a progressing long run completes while genuinely stale work is still recovered.

## Reproduction Assessment

Yes. The linked report at https://github.com/openclaw/openclaw/issues/88870 contains a captured automatic-recovery stack through `abortReplyRunBySessionId`, and current main still maps that path to `abortByUser()`; this read-only review did not run the live Feishu or CLI scenario.

## Solution Assessment

Yes in direction, but not yet in merge-ready form. System-originated cancellation belongs in the reply-operation and backend cancellation contracts, while the progress guard must be rebased and de-duplicated against the protection already on current main.
