## Summary

Keep open: current main and latest release still omit per-conversation identity from provider `createStreamFn` context, and the open implementation candidate has not merged yet.

## What This Changes

The issue requests adding per-conversation `sessionId` and `sessionKey` to provider plugin `createStreamFn` context so stream providers can isolate conversation-scoped state.

## Best Possible Solution

Keep this issue open as the canonical API/product decision, then review and either land or deliberately decline https://github.com/openclaw/openclaw/pull/93025.

## Reproduction Assessment

Yes at source level: current main lacks `sessionId` and `sessionKey` on `ProviderCreateStreamFnContext`, and the central stream-registration call does not pass them. I did not run the external GlueClaw two-DM repro in this read-only review.

## Solution Assessment

Unclear pending maintainer API/privacy direction, but optional fields on the existing provider stream hook are the narrowest plausible core seam. Plugin-side hashing, prompt parsing, or filesystem watching would be weaker than passing identity through the existing runtime boundary.
