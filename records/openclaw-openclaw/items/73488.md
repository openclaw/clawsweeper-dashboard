## Summary

Keep open. Current `main` and latest release still omit per-conversation identity from provider `createStreamFn` context, the paired OpenClaw PR was closed unmerged, and the merged Glueclaw-side fix still needs OpenClaw to expose these fields before it can take effect.

## What This Changes

The issue requests optional per-conversation `sessionId` and `sessionKey` fields in provider plugin `createStreamFn` context.

## Best Possible Solution

Define a documented provider stream identity contract that exposes `sessionKey` and `sessionId` to `createStreamFn` only on session-bearing paths, leaves one-off completions identity-less, and adds focused passthrough coverage.

## Reproduction Assessment

Yes. Source inspection shows `sessionId` and `sessionKey` exist in session-bearing runner paths but are not included in `ProviderCreateStreamFnContext` or provider stream registration, and the linked Glueclaw issue gives a concrete concurrent-conversation transcript-bleed path.

## Solution Assessment

Yes directionally, but it needs maintainer API/privacy scope. Optional fields on `ProviderCreateStreamFnContext` are the narrow seam for custom provider transports; the adjacent `prepareExtraParams` PR at https://github.com/openclaw/openclaw/pull/54725 does not replace this hook.
