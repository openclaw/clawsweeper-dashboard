- `impact:session-state`: The reported failure mode is transcript/session bleed between concurrent conversations when a provider plugin cannot key state by conversation.

## Summary

Keep open: current main still does not expose per-conversation identity to provider `createStreamFn` hooks, the direct implementation PR was closed unmerged, and the remaining open provider-identity PR targets a different cached hook rather than this custom transport seam.

## What This Changes

The issue requests adding optional `sessionId` and `sessionKey` to provider plugin `createStreamFn` context so provider plugins can scope state per conversation.

## Best Possible Solution

Add optional request-scoped identity to `ProviderCreateStreamFnContext` and thread it through the embedded attempt, compaction, and BTW provider stream registration paths, with focused passthrough tests and no cached extra-param behavior change.

## Reproduction Assessment

Yes. Source inspection shows current main has `sessionId` and `sessionKey` in the relevant runner flows, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` do not expose them to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing `createStreamFn` context are the narrow maintainable seam for custom provider transports; the adjacent `prepareExtraParams` proposal is a different cached hook and is not a safe substitute for this request.
