Keep open: current main and the latest release still omit `sessionId`/`sessionKey` from the provider `createStreamFn` context, and the live implementation candidate remains open rather than merged.

## What This Changes

The issue requests adding per-conversation `sessionId` and `sessionKey` fields to the provider plugin `createStreamFn` context.

## Best Possible Solution

Keep this issue as the canonical API decision and review, land, or deliberately decline https://github.com/openclaw/openclaw/pull/93025 with focused tests and docs.

## Reproduction Assessment

Yes at source level: current main lacks `sessionId` and `sessionKey` on `ProviderCreateStreamFnContext`, and the stream-registration call sites do not pass them. I did not run the external GlueClaw two-DM repro in this read-only review.

## Solution Assessment

Unclear pending maintainer API/privacy direction, but optional fields on the existing provider stream hook are the narrowest plausible core seam. Plugin-side hashing, prompt parsing, or filesystem watching would be weaker than passing identity through the existing runtime boundary.
