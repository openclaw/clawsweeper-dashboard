## Summary

Keep open. Current main still does not expose `sessionId` or `sessionKey` through the provider plugin `createStreamFn` context, and open PR #73492 is already linked with closing syntax as the implementation candidate. The issue also includes useful downstream Glueclaw context showing why per-agent fallback state can cause cross-conversation transcript bleed.

## What This Changes

Expose per-conversation identity fields, `sessionId` and `sessionKey`, to provider plugin `createStreamFn` context so custom provider transports can scope state per conversation.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core patch, that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them only from session-bearing paths, leaves one-off completion paths unset, updates provider SDK documentation if needed, and adds focused contract coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity available in the runner and related paths, but the provider stream context type and registration helper cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can scope state by `sessionKey` or `sessionId`.
