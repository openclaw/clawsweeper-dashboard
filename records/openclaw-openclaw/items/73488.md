## Summary

Keep open. Current main still does not expose per-conversation `sessionId` or `sessionKey` to provider plugin `createStreamFn`, and open PR #73492 uses closing syntax for this issue, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

Request to add optional `sessionId` and `sessionKey` to provider plugin `createStreamFn` context so provider plugins can scope per-conversation state.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional session identity to provider stream context and forwards it only from session-bearing paths.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity in runner, compaction, and `/btw` flows, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can scope state by `sessionKey` or `sessionId`.
