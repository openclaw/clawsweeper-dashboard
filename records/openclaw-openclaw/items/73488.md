## Summary

Keep open. Current main and release v2026.5.7 still omit per-conversation identity from provider `createStreamFn` context, the paired OpenClaw PR is closed unmerged, and the merged Glueclaw-side fix still cannot activate until OpenClaw exposes the fields.

## What This Changes

Expose optional per-conversation `sessionId` and `sessionKey` in provider plugin `createStreamFn` context and pass them from session-bearing agent paths.

## Best Possible Solution

Define and document an additive provider-stream identity contract that passes `sessionKey`/`sessionId` only from session-bearing paths, leaves one-off completions unset, and adds focused passthrough coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows run parameters carry `sessionId`/`sessionKey` but provider `createStreamFn` context and registration omit them, and the linked Glueclaw issue/PR gives a concrete concurrent-conversation transcript-bleed scenario.

## Solution Assessment

Yes directionally, but it needs maintainer-owned API/privacy scope. Optional fields on the existing provider `createStreamFn` context are the narrow OpenClaw seam for custom provider transports, provided the semantics and session-bearing call paths are documented and tested.
