## Summary

Keep open. Current main and v2026.5.7 still omit per-conversation identity from provider `createStreamFn` context, the paired OpenClaw PR is closed unmerged, and the merged Glueclaw fix remains a no-op until OpenClaw exposes the fields.

## What This Changes

Expose per-conversation session identity to provider plugin `createStreamFn` contexts so custom provider transports can scope state by conversation.

## Best Possible Solution

Define and document an optional provider stream identity contract, plumb `sessionKey`/`sessionId` through session-bearing paths, add passthrough coverage, and leave one-off completions identity-less.

## Reproduction Assessment

Yes, at source level. Current main exposes session identity in runner flows but not in provider `createStreamFn` context, and the linked Glueclaw issue describes a concrete two-conversation transcript-bleed reproduction.

## Solution Assessment

Yes directionally, but it needs maintainer-owned API/privacy scope. Optional fields on `ProviderCreateStreamFnContext` are the narrow OpenClaw seam for custom provider transports because this identity is not available through the current provider stream context.
