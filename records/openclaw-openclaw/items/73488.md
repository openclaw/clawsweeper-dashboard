## Label Justifications

- `P2`: This is a normal-priority provider plugin API/session-state issue with a concrete external failure mode and bounded core surface.
- `impact:session-state`: The reported failure mode is transcript/session bleed between concurrent conversations of the same agent when provider plugins can only key state by agent-level data.

## Summary

Keep open: current main and latest release still do not expose `sessionId` or `sessionKey` through `ProviderCreateStreamFnContext`, and the open implementation candidate has not merged yet.

## What This Changes

The issue requests adding per-conversation `sessionId` and `sessionKey` to provider plugin `createStreamFn` context so custom provider transports can isolate state per conversation.

## Best Possible Solution

Review and either land or deliberately decline the open provider API PR; if accepted, keep the fields optional, document their local per-conversation use, and leave one-off completion paths identity-free.

## Reproduction Assessment

Yes, from source inspection: the provider `createStreamFn` context and registration path currently omit session identity even though agent turn and compaction paths already carry it. I did not run the external GlueClaw concurrent-DM repro.

## Solution Assessment

Yes, pending maintainer API sign-off: optional fields on the existing provider `createStreamFn` context are the narrowest core seam for custom provider transports. Plugin-side hashing, prompt parsing, or filesystem watching would be weaker and less reliable.
