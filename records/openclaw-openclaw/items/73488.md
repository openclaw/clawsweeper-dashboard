## Summary

Keep open. Current main still does not expose `sessionId` or `sessionKey` to provider plugin `createStreamFn` contexts, and open PR #73492 references this issue with closing syntax as the implementation candidate, so the issue should not be closed before that PR or an equivalent fix lands.

## What This Changes

Add optional per-conversation `sessionId` and `sessionKey` fields to `ProviderCreateStreamFnContext` and plumb them from conversation-bearing provider stream call sites.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional provider stream context identity fields, forwards them only from session-bearing paths, leaves one-off completion paths unset, and adds focused contract coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity available in runner-side contexts, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional `sessionId` and `sessionKey` fields on the existing provider stream context are the narrowest backward-compatible API fix; provider plugins can prefer `sessionKey` when present without breaking plugins that ignore the new fields.
