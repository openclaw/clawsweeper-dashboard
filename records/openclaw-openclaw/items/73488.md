## Summary

Keep open: current main still lacks per-conversation identity in provider `createStreamFn` context, and #73492 is an open implementation PR with closing syntax, so this issue should track until that fix or an equivalent one merges.

## What This Changes

The issue asks core to expose optional per-conversation `sessionId` and `sessionKey` to provider plugins through `ProviderCreateStreamFnContext`.

## Best Possible Solution

Expose optional `sessionKey` and `sessionId` on the provider stream context, forward them only from session-bearing paths, and add focused contract coverage for the new plugin-facing fields.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity in the runner, compaction, and BTW flows, but the provider context type and registration helper cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can key state by `sessionKey` or `sessionId`.
