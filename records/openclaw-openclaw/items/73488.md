## Summary

Keep open. Current main still does not expose per-conversation identity to provider plugin `createStreamFn` contexts, and open PR #73492 is already the closing implementation candidate, so this issue should remain open until that PR or an equivalent fix merges.

## What This Changes

The issue requests optional `sessionId` and `sessionKey` fields on `ProviderCreateStreamFnContext` and plumbing from session-bearing provider stream registration paths.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them only from session-bearing paths, leaves one-off completion paths unset, and adds focused contract coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity in runner-side flows, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible API fix; provider plugins can prefer `sessionKey` when present without breaking plugins that ignore the new fields.
