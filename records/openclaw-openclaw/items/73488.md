## Summary

Keep open. Current main still does not expose `sessionId` or `sessionKey` through the provider plugin `createStreamFn` context, and the open linked PR #73492 uses closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to add optional per-conversation `sessionId` and `sessionKey` fields to provider plugin `createStreamFn` context and forward them from session-bearing runtime paths.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional `sessionId` and `sessionKey` to provider stream context, forwards them only from session-bearing paths, leaves one-off completion paths unset, and adds focused contract coverage plus plugin SDK docs if needed.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity in runner, compaction, and `/btw` paths, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can scope state by `sessionKey` or `sessionId`.
