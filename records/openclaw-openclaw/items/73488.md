## Summary

Keep open. Current main still does not expose per-conversation `sessionId` or `sessionKey` to provider plugin `createStreamFn`, and open PR #73492 uses closing syntax for this issue; the linked Glueclaw reports provide a concrete downstream transcript-bleed failure mode.

## What This Changes

Expose optional `sessionId` and `sessionKey` in `ProviderCreateStreamFnContext` and forward them from session-bearing provider stream call sites.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional session identity to provider stream context and forwards it only from session-bearing paths.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity in runner, compaction, and `/btw` flows, but the provider `createStreamFn` contract and registration helper cannot deliver it to plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can scope state by `sessionKey` or `sessionId`.
