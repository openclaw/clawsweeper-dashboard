## Summary

Keep open. Current main still does not expose `sessionId` or `sessionKey` through provider plugin `createStreamFn` context, and the linked open PR #73492 uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Provider plugins should receive optional per-conversation `sessionId` and `sessionKey` values in `createStreamFn` context so custom stream transports can isolate per-conversation state.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them from session-bearing paths, leaves one-off completion paths unset, and adds focused contract coverage plus SDK docs if needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has session identity in runner, compaction, and `/btw` paths, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can scope state by `sessionKey` or `sessionId`.
