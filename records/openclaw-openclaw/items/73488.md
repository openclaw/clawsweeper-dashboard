## Summary

Keep open. Current main still does not expose `sessionId` or `sessionKey` through the provider plugin `createStreamFn` context, and the linked open PR #73492 is the implementation candidate with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Expose optional per-conversation `sessionId` and `sessionKey` fields to provider plugin `createStreamFn` contexts and plumb them from session-bearing runner paths.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them only from session-bearing paths, leaves one-off completion paths unset, updates SDK docs if needed, and adds focused contract coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity available in runner-side code, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can scope state by `sessionKey` or `sessionId`.

## Security Review

Status: needs_attention

Summary: The issue describes a concrete privacy/isolation risk for provider plugins that need to persist upstream sessions per conversation.
