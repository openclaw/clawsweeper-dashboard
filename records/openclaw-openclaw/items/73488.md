## Summary

Keep open: current main and the latest release still lack the requested `sessionId`/`sessionKey` fields on `ProviderCreateStreamFnContext`, and the open implementation candidate has not merged yet.

## What This Changes

The issue requests exposing per-conversation `sessionId` and `sessionKey` on provider plugin `createStreamFn` context so stream providers can isolate per-conversation state.

## Best Possible Solution

Keep this issue as the canonical tracker until https://github.com/openclaw/openclaw/pull/93025 lands or maintainers choose a different provider API shape for request-scoped conversation identity.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessionId` and `sessionKey` are present in agent run inputs, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` do not expose them to provider plugins. I did not run the downstream GlueClaw two-DM repro.

## Solution Assessment

Yes, pending maintainer API sign-off. Optional fields on the existing provider `createStreamFn` context are the narrowest maintainable seam; plugin-side hashing, prompt parsing, or lockfile watching would be brittle workarounds.
