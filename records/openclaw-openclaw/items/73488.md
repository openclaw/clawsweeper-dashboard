## Summary

Keep open. Current main and latest release still omit `sessionId`/`sessionKey` from the provider `createStreamFn` context, while the session-bearing runner, compaction, and `/btw` paths already have that identity; open PR #73492 is the active closing implementation candidate.

## What This Changes

Expose per-conversation session identity to provider plugin `createStreamFn` contexts so provider-owned transports can scope upstream state by conversation.

## Best Possible Solution

Land a reviewed version of PR #73492, or an equivalent narrow core change, that adds optional `sessionId`/`sessionKey` to provider stream context and forwards them only from session-bearing paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: session identity exists on run and `/btw` inputs, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible API extension because existing plugins can ignore them while conversation-aware providers can key state by `sessionKey` or `sessionId`.
