## Summary

Keep open. Current main still does not expose per-conversation `sessionId` or `sessionKey` to provider plugin `createStreamFn`, and open PR #73492 is already the linked implementation candidate for this issue.

## What This Changes

Expose optional per-conversation `sessionId` and `sessionKey` values to provider plugin `createStreamFn` context so providers can isolate state by conversation.

## Best Possible Solution

Land a reviewed version of PR #73492, or an equivalent narrow core change, that adds optional session identity to provider stream context and forwards it only from session-bearing paths.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has session identity in runner, compaction, and `/btw` flows, but the provider context type and registration helper cannot deliver it to provider plugins.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix because existing plugins can ignore them while conversation-aware providers can scope state by `sessionKey` or `sessionId`.
