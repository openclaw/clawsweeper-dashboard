## Summary

Keep open. Current main and the latest release still omit per-conversation identity from provider `createStreamFn` context, the paired OpenClaw implementation PR is closed unmerged, and the merged Glueclaw-side fix still needs OpenClaw to expose the identity fields before it can take effect.

## What This Changes

Add optional `sessionId` and `sessionKey` to provider `createStreamFn` context and forward them from session-bearing provider stream registration paths.

## Best Possible Solution

Define a documented provider stream identity contract that exposes `sessionKey`/`sessionId` only on session-bearing paths, adds focused passthrough coverage, and leaves one-off completions identity-less.

## Reproduction Assessment

Yes. Source inspection against current main shows session identity exists in runner paths but is not available to provider `createStreamFn`, and the linked Glueclaw issue/PR gives a concrete concurrent-conversation transcript-bleed scenario.

## Solution Assessment

Yes directionally, but it needs maintainer-owned API/privacy scope. Optional fields on `ProviderCreateStreamFnContext` are the narrow OpenClaw seam for custom provider transports because `sessionKey` is not a pi-ai concept, while the implementation should also consider consistent `sessionId` forwarding to provider-owned stream options.
