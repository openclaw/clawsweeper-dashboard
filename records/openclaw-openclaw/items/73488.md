## Label Justifications

## Summary

Keep open: current main and v2026.6.10 still omit per-conversation identity from the provider `createStreamFn` context, while the live implementation candidate remains open and needs maintainer API/privacy direction before this issue can close.

## What This Changes

Expose per-conversation `sessionId` and `sessionKey` to provider plugin `createStreamFn` context so plugins can scope stream state by conversation.

## Best Possible Solution

Keep this issue as the canonical API decision and review, land, or deliberately decline https://github.com/openclaw/openclaw/pull/93025 with focused tests and docs.

## Reproduction Assessment

Yes at source level: current main lacks `sessionId` and `sessionKey` on `ProviderCreateStreamFnContext`, and the stream-registration call sites do not pass them even when session identity is nearby. I did not run the external GlueClaw two-DM repro in this read-only review.

## Solution Assessment

Unclear pending maintainer API/privacy direction, but optional fields on the existing provider stream hook are the narrowest plausible core seam if the direction is accepted. Plugin-side hashing, prompt parsing, or filesystem watching would be weaker than passing identity through the existing runtime boundary.
