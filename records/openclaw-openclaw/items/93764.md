## Summary

Keep open: current main has partial docs and resolver work around Codex context caps, but it still has code paths and tests that make configured OpenAI context overrides authoritative for native Codex runs while Codex itself clamps the runtime window. The closed follow-up PR was a candidate fix, not merged, so the central mismatch remains owned by this issue.

## What This Changes

The issue asks OpenClaw to stop implying that a configured 1M/950k OpenAI/Codex context budget is active when native Codex gpt-5.5 turns are capped around 258k effective tokens.

## Best Possible Solution

Define one runtime-specific effective context contract for native Codex OAuth: report and cap around the Codex catalog limit, and separately document any supported API/OpenClaw runtime path for larger contexts.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: OpenClaw can resolve configured OpenAI context metadata for native Codex runs, while upstream Codex clamps and emits the smaller effective runtime window.

## Solution Assessment

No current solution is sufficient yet; the unmerged PR addressed one resolver path, but maintainers still need to choose the canonical status/runtime/config contract for native Codex versus larger API/OpenClaw runtime paths.
