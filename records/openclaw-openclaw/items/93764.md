- `impact:auth-provider`: The issue centers on OpenAI/Codex OAuth routing, model choice, and the difference between provider metadata and native Codex runtime limits.

## Summary

Keep open: current main and v2026.6.8 still allow OpenAI model config to imply a larger Codex OAuth context than the native Codex runtime can use, and the only linked fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to make Codex OAuth gpt-5.5 context reporting/routing match the native runtime cap or provide a supported OAuth-preserving 1M context path.

## Best Possible Solution

Keep provider model metadata and native Codex runtime metadata as separate facts: when the Codex harness will run, status/guards should show Codex's effective cap, while docs/config should direct users to another runtime only when that path actually preserves the desired auth and context.

## Reproduction Assessment

Yes, at source/dependency level: current main resolves the native Codex harness path from OpenAI configured metadata while sibling Codex source reports and enforces 272000 * 95% = 258400. I did not run a live Codex/OAuth turn in this read-only review.

## Solution Assessment

Unclear. Capping the value everywhere is only one plausible fix; the more maintainable direction is for maintainers to choose a runtime-specific status/config contract that does not conflate OpenAI provider metadata with Codex native limits.
