## Summary

Keep open: current main still allows configured OpenAI model metadata to advertise a larger context budget while the native Codex app-server contract clamps turns to the Codex catalog maximum, and the only direct fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to stop implying a 1M/950k budget for `openai/gpt-5.5` native Codex OAuth turns unless that runtime can actually use it, or to document/support a runtime switch that preserves OAuth.

## Best Possible Solution

Maintainers should define the supported contract: report the native Codex cap where that runtime is selected, document a safe OpenClaw-runtime/API path for 1M use, or wait for upstream Codex support before claiming 1M on native Codex OAuth.

## Reproduction Assessment

Yes at source/dependency-contract level: configure a larger `models.providers.openai.models[]` context for `gpt-5.5`; OpenClaw can resolve/display that value while Codex clamps native turns to the model `max_context_window`. I did not run a live OAuth turn because credentials/live proof were not available in this read-only review.

## Solution Assessment

No current main solution is sufficient. The closed cap-only PR was a partial mitigation and could be the wrong layer unless maintainers choose the runtime-specific status, runtime-switching, or upstream-Codex contract.
