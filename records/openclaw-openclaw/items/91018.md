## Summary

Keep open: current main does not prove this DeepSeek cache regression is fixed, and source inspection shows DeepSeek is not currently opted into the cache-key path that OpenClaw documents for OpenAI-compatible Completions providers. The full production symptom still needs DeepSeek-live proof because DeepSeek documents automatic prefix caching and `prompt_cache_hit_tokens`, not an OpenAI `prompt_cache_key` requirement.

## What This Changes

The issue reports that upgrading to v2026.6.1 caused DeepSeek prompt caching to stop working for an OpenAI-compatible completions configuration, with high token spend and a suspected missing upgrade/default cache-key setting.

## Best Possible Solution

Keep this open for a provider-owner investigation that runs DeepSeek-live repeated-prefix proof, then updates DeepSeek model metadata/provider policy or cache shaping only if the upstream behavior proves that is the right contract for fresh and upgraded configs.

## Reproduction Assessment

Yes for the missing request-key path from source: current main omits DeepSeek `prompt_cache_key` unless model config explicitly opts in. No for the full production symptom: this review did not run a DeepSeek-live repeated-prefix cache test.

## Solution Assessment

Unclear: adding `supportsPromptCacheKey` may address part of the source-visible OpenClaw gate, but DeepSeek's documented cache behavior is prefix-based and the report says manual opt-in still did not restore hits. The best fix needs live provider proof plus a maintainer decision at the DeepSeek provider boundary.
