## Summary

Keep open. Current main and v2026.6.6 still truncate older tool results before newer ones, which rewrites the reusable prompt prefix as sessions grow and matches the newer source-backed regression explanation. The identical same-author report at https://github.com/openclaw/openclaw/issues/91016 is also open, so maintainers should consolidate the pair before promoting one shared-runtime fix; the original `supportsPromptCacheKey` proposal is not the best primary repair for DeepSeek’s automatic prefix cache.

Likely related people: Jianpan is a medium-confidence recent investigator of the truncation mechanism; Vincent Koc is a low-confidence release-context contact because the available grafted history does not expose the implementation’s original author reliably.

## What This Changes

Restore cache-friendly prompt-prefix stability for DeepSeek/OpenAI-compatible sessions after tool-result truncation began rewriting older prompt content first.

## Best Possible Solution

Designate https://github.com/openclaw/openclaw/issues/91016 or this issue as canonical, update the shared truncation policy to preserve the longest stable prompt prefix without discarding needed recent tool context, add deterministic multi-turn coverage, and verify redacted DeepSeek cache-hit and billed-token output on both an upgraded configuration and a fresh configuration.

## Reproduction Assessment

Yes at source level: current main deterministically rewrites older tool-result content first when enforcing the aggregate budget, which destabilizes repeated prompt prefixes. A live DeepSeek run is still needed to quantify the provider-side cache-hit and cost regression.

## Solution Assessment

No, a DeepSeek-only `supportsPromptCacheKey` default or new cache-mode option is not the best primary fix. The narrower maintainable direction is to restore the shared stable-prefix invariant, then add provider metadata only if live upstream evidence proves it is independently required.
