## Summary

Keep open: current main still invokes the dashboard helper through `docker compose run --rm openclaw-cli` without `--no-deps`, and Docker Compose dry-run shows that command starts the `openclaw-gateway` dependency. The reported `GATEWAY_IMAGE` variable is not part of the current base compose contract, but the helper can still touch the running gateway while only fetching a URL.

## What This Changes

Running `clawdock-dashboard --no-open` after a manually started Docker Compose gateway can cause Compose to replace the running gateway container image while only trying to fetch the Control UI URL.

## Best Possible Solution

Change the ClawDock dashboard URL lookup so it cannot start or recreate `openclaw-gateway`, add a regression test around the helper command, and keep Docker docs aligned with the supported `OPENCLAW_IMAGE` variable.

## Reproduction Assessment

No, not as a high-confidence live reproduction in this read-only review. Source inspection and Docker Compose dry-run show the current helper has a plausible dependency-start path, but the exact reported `GATEWAY_IMAGE` setup needs a real Docker environment or local compose override to confirm end to end.

## Solution Assessment

Yes, the narrowest maintainable fix appears to be making `clawdock-dashboard` run the post-start CLI with `--no-deps` so fetching the URL cannot alter the gateway service. A broader image-variable migration would be riskier and is not needed for the observed helper behavior.
