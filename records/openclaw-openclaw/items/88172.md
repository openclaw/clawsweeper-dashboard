## Summary

Keep open: the teardown half is plausible, but the reconnect path can still treat normal MCP server errors as dead transports and replay a tool call, and the added proof shows before-fix orphan accumulation rather than after-fix cleanup or reconnect success.

## What This Changes

The PR retires bundled MCP runtimes during CLI and isolated cron teardown, then adds one-shot reconnect-and-retry behavior when bundled MCP tool calls hit selected dead-transport error messages.

## Best Possible Solution

Land a coordinated MCP lifecycle fix that retires run-owned runtimes on teardown, and merge reconnect only after it distinguishes transport death from normal MCP server errors and includes after-fix process/reconnect proof.

## Reproduction Assessment

Yes, source-reproducible: current main only closes the MCP loopback server on CLI cleanup, isolated cron cleanup only retires browser sessions, and bundled MCP tool calls directly call session.client.callTool without reconnect. I did not run a live cron/MCP process reproduction in this read-only review.

## Solution Assessment

No, not yet: the teardown direction is narrow, but the reconnect implementation should classify true transport failures instead of message-matching all MCP callTool rejections. The safer path is to preserve normal MCP server errors and prove the patched runtime behavior after the fix.
