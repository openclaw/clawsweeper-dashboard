## Summary

Keep open. The MCP teardown half is plausible, but the reconnect half has a concrete correctness bug: retry-call failures are treated as reconnect failures and can leave a disposed session cached; the external PR also lacks real behavior proof for the process cleanup and reconnect paths.

## What This Changes

The PR adds MCP runtime retirement on CLI and isolated cron teardown, plus one-shot reconnect/retry handling when bundle MCP tool calls hit dead transport errors.

## Best Possible Solution

Land a bounded MCP lifecycle fix that disposes run-owned runtimes on teardown and reconnects only confirmed dead transports while preserving retry error/state semantics and proving the real cron/stdout process path.

## Reproduction Assessment

Source-reproducible, not live-reproduced: current main only closes the MCP loopback server for CLI cleanup and directly calls session.client.callTool without reconnect. I did not run a live cron/MCP process reproduction in this read-only review.

## Solution Assessment

No: the teardown direction is plausible, but the reconnect implementation should separate reconnect failures from retried tool-call failures, keep the sessions map consistent, and include focused tests plus real process proof.
