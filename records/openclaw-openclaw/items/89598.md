## Summary

Keep open: this maintainer-owned draft addresses a real MCP loopback robustness bug, but the current patch still partially merges a union variant after an unreadable `required` getter, which can misstate required tool arguments.

## What This Changes

The PR adds guarded MCP loopback union-schema reads and a regression for an unreadable `properties` getter while preserving usable sibling variants.

## Best Possible Solution

Make union-variant flattening transactional: read properties, property entries, and required before mutating `mergedProps`, skip unreadable variants with a warning, and cover both throwing `properties` and throwing `required`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a plugin-provided union variant with a throwing schema getter reaches `buildMcpToolSchema` through the MCP loopback cache. I did not run tests in this read-only review.

## Solution Assessment

No. The PR is close, but the best fix is to skip a variant when any field needed for its merge is unreadable rather than merging its properties with an empty required set.
