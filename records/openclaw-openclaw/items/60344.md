## Summary

Keep open. Current main and v2026.5.7 still use the exact reported literal as model-visible image-prune replay text while WebChat passes that literal as ordinary input, so neither current nor shipped behavior resolves the marker-collision bug.

## What This Changes

The issue asks OpenClaw to stop recursion caused by the `[image data removed - already processed by model]` image-pruning marker when that text is sent normally to Kimi Coding.

## Best Possible Solution

Replace or encode the internal pruned-image replay marker so it cannot collide with user-authored text, while preserving stale media reference scrubbing and current-turn image hydration.

## Reproduction Assessment

No, not end-to-end. Source inspection on current main and v2026.5.7 shows the literal marker can still be accepted as WebChat text and emitted into model replay, but I did not run a live kimi-coding/k2p5 session to observe the recursive model output.

## Solution Assessment

Yes. The narrowest maintainable fix is at the core replay marker representation or sanitization boundary; a Kimi-only output filter would leave the shared model-visible marker ambiguous.
