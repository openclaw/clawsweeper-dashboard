## Summary

Keep open. Current main still exposes the exact reported pruned-image marker as model-visible replay text, and WebChat still accepts that same literal as normal user input, so the marker-recursion bug has not been resolved by the adjacent stale-media fixes.

## What This Changes

The issue asks OpenClaw to prevent `[image data removed - already processed by model]` from being confused with internal image-pruning state when sent as user text or replayed to Kimi Coding.

## Best Possible Solution

Replace or encode the internal pruned-image replay marker so it cannot collide with user-authored text, while preserving stale-media reference scrubbing and current-turn image hydration.

## Reproduction Assessment

No, not end-to-end. Static inspection gives a high-confidence path showing the literal marker is accepted by WebChat and still emitted into provider replay, but the Kimi loop needs a live Kimi run or a focused mocked regression.

## Solution Assessment

Yes. The narrowest maintainable solution is in the core replay marker representation or sanitization boundary; a Kimi-only output filter would leave the ambiguous model-visible marker in shared replay context.
