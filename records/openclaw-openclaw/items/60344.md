## Summary

Keep open. Current main still uses the exact reported pruned-image marker as model-visible replay text, WebChat accepts that literal as ordinary user text, and the adjacent stale-media fix for #71868 does not cover this marker-recursion case.

## What This Changes

Prevent the internal pruned-image marker from being exposed to or echoed recursively by `kimi-coding/k2p5` when a WebChat user sends the marker text literally.

## Best Possible Solution

Change the core replay marker boundary so internally pruned images cannot be confused with user-authored text, while preserving stale-media pruning and current-turn image hydration.

## Reproduction Assessment

No, not end-to-end. Static inspection gives a high-confidence path showing the literal marker is accepted by WebChat and still injected into provider replay, but the Kimi recursive output itself still needs a live Kimi run or focused mocked regression.

## Solution Assessment

Yes, the narrowest maintainable fix belongs in core replay marker representation or sanitization. A provider-specific Kimi output filter would be weaker because the ambiguous marker is produced before provider dispatch.
