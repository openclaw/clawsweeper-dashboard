Keep open. Current main still emits the exact reported pruned-image marker as provider-visible replay text, WebChat still accepts the same literal as ordinary user input, and there is no merged closing PR or canonical duplicate that resolves this narrower marker-collision bug.

The issue asks OpenClaw to treat the literal `[image data removed - already processed by model]` as normal user text and prevent the internal image-prune marker from causing recursive model output with Kimi.

## Best Possible Solution

Replace or encode the internal pruned-image replay marker so it cannot collide with user-authored text, while preserving stale-media reference scrubbing and current-turn image hydration.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the core replay marker representation or sanitization boundary; a Kimi-only output filter would leave the ambiguous shared replay text in other provider contexts.
