Keep open: current main still has a provider-visible history-image pruning marker, and the related stale-image replay work is active, but the exact Kimi literal-marker recursion is not proven enough to close or hand to automation.

## What This Changes

The issue asks OpenClaw to stop kimi-coding/k2p5 from recursively echoing the `[image data removed - already processed by model]` pruning marker when a user sends that literal text.

## Best Possible Solution

Keep this open until a Kimi/WebChat reproduction proves or disproves the loop; coordinate any provider-visible marker change with https://github.com/openclaw/openclaw/issues/66702 and https://github.com/openclaw/openclaw/pull/76732.

## Reproduction Assessment

No. The reporter gives plausible steps, but there are no logs or live Kimi proof; source inspection shows the literal marker is not parsed as an image ref, while the marker can still be sent as old-history replay text.

## Solution Assessment

Unclear. The narrow maintainable path is to first reproduce Kimi's behavior, then decide whether to reword/remove the provider-visible marker or fold it into the stale-image pruning work instead of adding a blind output filter.
