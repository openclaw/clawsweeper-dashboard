## Summary

Keep open. Current main still emits the exact reported pruned-image marker into provider replay and WebChat still accepts the same literal as normal printable input, while the related merged work only changed the pruning window rather than removing the marker collision.

## What This Changes

The issue asks OpenClaw to treat `[image data removed - already processed by model]` as ordinary user text and stop that internal replay marker from triggering recursive Kimi Coding output.

## Best Possible Solution

Disambiguate or encode the internal pruned-image replay marker so it cannot collide with user-authored text, while preserving stale-media cleanup and current-turn image hydration.

## Reproduction Assessment

No, not end-to-end with live Kimi in this review. Source inspection on current main shows the exact literal is accepted by `chat.send` and still emitted into provider replay, but the provider recursion needs a live Kimi run or focused mocked regression to prove the loop itself.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared replay marker representation or sanitization boundary; a Kimi-only output filter would leave the ambiguous marker in shared model-visible context.
