Keep open: current main still uses the exact reported pruned-image marker as model-visible replay text and still accepts the same literal from WebChat without escaping; the Kimi model-id normalization and stale-media fix do not cover this marker-recursion case.

## What This Changes

The issue asks OpenClaw to prevent a literal user message containing the internal pruned-image marker from causing Kimi Coding to recursively output that marker.

## Best Possible Solution

Move the pruned-image marker out of ambiguous model-visible plain text or escape the user/replay boundary, with regression coverage for literal user marker text and old image replay pruning.

## Reproduction Assessment

No, not end-to-end. Source inspection shows WebChat passes the literal marker unchanged and old image replay still injects the same marker, but a live Kimi run or focused mocked regression is still missing.

## Solution Assessment

Yes, a narrow core marker-boundary fix is the best direction. Provider-specific Kimi output filtering would leave the ambiguous replay text intact before provider dispatch.
