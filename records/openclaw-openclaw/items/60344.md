## Summary

Keep open: current main still has a model-visible pruned-image marker that matches the reported text, but source review does not prove the Kimi recursive-output symptom and the closest related vision-history work is still open rather than shipped.

## What This Changes

The issue reports that sending the literal text `[image data removed - already processed by model]` to Kimi `kimi-coding/k2p5` in WebChat causes repeated output of that marker instead of normal text handling.

## Best Possible Solution

Define a replay-safe representation for pruned images that does not look like ordinary user prompt content, or add a narrowly scoped Kimi/provider guard, with focused regression coverage and live WebChat or Kimi proof; coordinate only the stale-image overlap with https://github.com/openclaw/openclaw/issues/66702 and https://github.com/openclaw/openclaw/pull/76732.

## Reproduction Assessment

No. Source review confirms where the exact marker enters model-visible replay and that the literal marker is not parsed as an image reference, but a live Kimi/WebChat run is needed to reproduce the reported recursive output.

## Solution Assessment

Unclear. Filtering the literal user input would be risky; a replay-local marker design or provider-specific handling with live proof looks safer than closing or applying a broad text rewrite.
