## Summary

Keep open. Current main still uses the exact reported pruned-image marker as replay text, and WebChat still accepts the same literal as ordinary input, so the marker-collision bug remains unresolved.

## What This Changes

The issue asks OpenClaw to prevent the literal `[image data removed - already processed by model]` from triggering recursive Kimi output when sent as normal WebChat text.

## Best Possible Solution

Replace or encode the internal pruned-image replay marker so it cannot collide with user-authored text, while preserving stale-media reference scrubbing and current-turn image hydration.

## Reproduction Assessment

No, not end-to-end. Source inspection on current main shows the exact literal is accepted by `chat.send` and still emitted into provider replay, but the Kimi recursion itself needs a live Kimi run or a focused mocked regression.

## Solution Assessment

Yes. The narrowest maintainable fix is in the core replay marker representation or sanitization boundary; a Kimi-only output filter would leave the ambiguous marker in shared replay context.
