## Summary

Keep open. Current main has provider-owned `reasoning_content` replay/backfill support for adjacent DeepSeek-style paths, but the Qianfan plugin still only registers auth/catalog metadata and does not preserve or backfill `reasoning_content` for Qianfan tool-call replay. The reported Coding Plan route also is not in the current bundled Qianfan catalog, so closing would conflate a related DeepSeek fix with an unresolved Qianfan/custom-provider behavior.

## What This Changes

The issue requests that OpenClaw stop Qianfan thinking-mode tool-call replays from failing when assistant tool-call messages are missing `reasoning_content`.

## Best Possible Solution

Keep the issue open and add provider-owned replay support for Qianfan-like OpenAI-compatible thinking models, either by wiring a Qianfan stream/replay hook that preserves or backfills `reasoning_content` when thinking is enabled, or by adding a documented custom-provider compatibility option for this contract. The Qianfan docs should also state whether the old Coding Plan route is retired, renamed, or supported through custom provider configuration.

## Reproduction Assessment

Unclear for live reproduction: the issue gives a concrete model/tool-call path and error, but this checkout has no Qianfan key in env/profile for a current-main live run. The source-level reproduction path is high confidence because Qianfan lacks the provider hook that adjacent DeepSeek coverage uses for this exact replay shape.

## Solution Assessment

Unclear. The DeepSeek-style provider-owned replay wrapper is the narrowest proven pattern in current main, but applying it to this issue needs a maintainer choice because the reported provider/model route is not bundled under the current Qianfan catalog.
