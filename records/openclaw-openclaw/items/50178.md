## Summary

Keep open. Current main still has Qianfan registered as a plain OpenAI-compatible provider with auth and catalog metadata only, while adjacent DeepSeek/OpenRouter paths use provider-owned replay/stream wrappers to preserve or backfill `reasoning_content`; the reported Coding Plan route is also absent from the bundled Qianfan catalog.

## What This Changes

The issue asks OpenClaw to handle or disable Qianfan `reasoning_content` during thinking-enabled tool-call replay so tool loops do not emit 400 errors.

## Best Possible Solution

Add a provider-owned Qianfan compatibility path only after deciding the supported route boundary, then document whether Coding Plan models are bundled, custom-provider only, or retired.

## Reproduction Assessment

Yes for source-level reproduction, but not for a live API run: Qianfan currently lacks the replay/stream hooks that adjacent providers use to preserve or backfill `reasoning_content`. The exact reported model route is not in the bundled catalog, so live current-main reproduction remains unproven.

## Solution Assessment

Unclear. The DeepSeek-style provider-owned replay wrapper is the narrowest proven pattern, but applying it here needs a maintainer decision about Qianfan versus custom Coding Plan route support.
