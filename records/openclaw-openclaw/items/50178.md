Keep open. Current main still has no Qianfan-owned `reasoning_content` replay path, and the reported Coding Plan route is not in the bundled Qianfan catalog, so this needs a maintainer support-boundary decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent Baidu Qianfan `qianfan-code-latest` tool-call turns from failing with missing `reasoning_content` when thinking is enabled.

## Best Possible Solution

If maintainers support the Coding Plan/Qianfan route, add a Qianfan-owned OpenAI-compatible replay/stream compatibility path that preserves or backfills `reasoning_content` for thinking tool turns; otherwise document the unsupported custom-route boundary.

## Reproduction Assessment

No, not with a high-confidence live current-main run. Source inspection shows Qianfan lacks the replay/stream hooks used by DeepSeek for this error class, but the exact reported Coding Plan route is absent from the bundled catalog.

## Solution Assessment

Unclear. A provider-owned wrapper like DeepSeek is the narrowest technical shape for supported Qianfan models, but a broad generic replay backfill could affect custom providers and the reported route needs a support decision first.
