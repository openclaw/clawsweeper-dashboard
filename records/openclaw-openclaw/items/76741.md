## Summary

Keep open: current main still lacks Kimi cache-marker stripping, and the PR uses the right provider-local boundary, but the updated recursive strip deletes arbitrary `cache_control` keys from tool schemas and tool-use inputs, so it needs a narrow repair before merge.

## What This Changes

The branch adds Kimi-local recursive removal of `cache_control` fields from request payloads, regression coverage for stripped payload blocks, and a changelog fix entry for #76612.

## Best Possible Solution

Keep the fix local to the Kimi provider, but strip only Anthropic prompt-cache marker metadata while preserving tool schemas and historical tool inputs named `cache_control`.

## Reproduction Assessment

Yes for a source-level reproduction: Kimi uses `anthropic-messages`, `buildAnthropicParams` can add `cache_control` from `cacheRetention`, and current main's Kimi wrapper does not remove those fields. I did not perform a live Kimi request.

## Solution Assessment

No for the current diff. A Kimi-local strip is the narrow maintainable direction, but deleting every `cache_control` key in the full payload is broader than necessary and can corrupt legitimate tool data.
