## Summary

Keep open: current main and the latest release still retain the reported verifier path where a hook can request Telegram announce delivery without a concrete `to`, while source-delivery verification refuses to mark the successful message-tool send verified without that planned target.

## What This Changes

The issue asks hook delivery verification to accept a successful Telegram `message` tool send using `target`, or report the exact expected and observed mismatch instead of the generic cron target error.

## Best Possible Solution

Define the hook delivery verification policy, then add focused regression coverage so successful explicit Telegram sends either satisfy the hook run under a safe match rule or fail with clear expected and observed target details.

## Reproduction Assessment

Yes from source inspection: hook announce delivery can be built without `to`, and the verifier requires both `delivery.to` and message-tool target evidence. I did not run a live Gmail/Telegram hook.

## Solution Assessment

Unclear until maintainers choose policy: verifying targetless hook sends is plausible but delivery-sensitive. The narrow maintainable fix belongs in hook/cron/source-delivery verification with explicit diagnostics and regression tests.
