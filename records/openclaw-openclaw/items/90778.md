## Label Justifications

- `P2`: The report describes a real hook delivery-status bug with limited blast radius, noisy failures, and duplicate-send risk rather than a core outage.
- `impact:message-loss`: The issue concerns channel delivery bookkeeping after a Telegram send and can lead operators to retry an already-delivered message.

## Summary

Keep open: current main and the latest release still have the reported mismatch path. Hook dispatch can request announce delivery without a concrete `to`, the message tool does capture `target`, but source-delivery verification still requires the planned delivery target to include `to` before it marks the message-tool send verified.

## What This Changes

The issue asks OpenClaw to stop marking Gmail hook runs failed after a successful explicit Telegram `message` tool send, or to report the exact configured/observed target mismatch.

## Best Possible Solution

Define the hook delivery policy, then add focused regression coverage so successful explicit Telegram sends either satisfy an intentionally targetless hook path or fail with clear expected and observed target details.

## Reproduction Assessment

Yes. Source inspection shows hook dispatch can build an announce delivery with no `to`, while message-tool evidence is only verified when the planned source delivery target also has `to`; I did not run a live Gmail-to-Telegram hook.

## Solution Assessment

Unclear until maintainers choose policy. Verifying targetless hook sends is plausible but delivery-sensitive, so the safest implementation is a focused hook/cron/source-delivery change with explicit diagnostics and regression tests.
