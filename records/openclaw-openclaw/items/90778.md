## Summary

Keep open: current main still has the reported mismatch path because hook dispatch can create announce delivery without a concrete `to`, while source-delivery verification refuses to mark a message-tool send verified unless the planned delivery target also has `to`.

## What This Changes

The issue asks OpenClaw to treat a successful Telegram `message(action="send", target=...)` from a Gmail hook as verified delivery, or to report the exact target mismatch.

## Best Possible Solution

Define the hook delivery policy, then add focused regression coverage so successful explicit Telegram sends either satisfy an intentionally targetless hook path or fail with clear expected and observed target details.

## Reproduction Assessment

Yes from source inspection: hook announce delivery can be built without `to`, and the verifier requires both `delivery.to` and message-tool target evidence. I did not run a live Gmail/Telegram hook.

## Solution Assessment

Unclear until maintainers choose policy: verifying targetless hook sends is plausible but delivery-sensitive. The safest implementation is a focused hook/cron/source-delivery change with explicit diagnostics and regression tests.
