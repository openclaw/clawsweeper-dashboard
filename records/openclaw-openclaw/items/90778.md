## Summary

Keep open: current main and latest v2026.6.8 still require a resolved delivery target before a message-tool target can verify source delivery, while hook dispatch can create announce runs with an optional target. The merged related cron PR fixed the empty-target-evidence false-success case, but it does not close this explicit-target false-error path.

## What This Changes

The issue asks for Gmail hook delivery verification to accept a successful Telegram `message` tool send with an explicit `target`, or to report the exact expected/observed mismatch.

## Best Possible Solution

Define the targetless hook delivery contract, then add focused hook/cron/source-delivery coverage so successful explicit Telegram sends either verify under that rule or fail with expected and observed target details.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: hook dispatch can create announce delivery without `to`, and the verifier requires `delivery.to` before explicit message-tool target evidence can match.

## Solution Assessment

Unclear until maintainers choose policy. The safest fix is a focused hook/cron/source-delivery change that either verifies successful explicit sends under a defined targetless-hook rule or emits clearer expected/observed mismatch diagnostics.
