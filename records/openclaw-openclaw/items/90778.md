## Label Justifications

- `P2`: The issue describes a real hook delivery-status bug with duplicate-send risk, but the blast radius is limited to hook/cron delivery accounting.
- `impact:message-loss`: The affected behavior can make operators retry an already-delivered Telegram message because the hook run is recorded as failed.

## Summary

Keep open: current main still has the reported targetless hook verification shape. Explicit message-tool target evidence is captured, but source delivery verification still refuses to mark it verified when the hook/cron delivery plan has no concrete `to`, and current tests preserve that unresolved-target failure path.

## What This Changes

The issue asks OpenClaw hook delivery verification to accept or clearly explain successful Telegram `message` tool sends that use the canonical `target` argument.

## Best Possible Solution

Define the hook delivery policy, then add focused regression coverage so successful explicit Telegram sends either satisfy an intentionally targetless hook path or fail with expected and observed target details.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a hook announce delivery can be created without `to`, and the verifier requires `delivery.to` before explicit message-tool target evidence can match. I did not run a live Gmail-to-Telegram hook.

## Solution Assessment

Unclear until maintainers choose policy. The safest fix is a focused hook/cron/source-delivery change that either treats targetless hook sends as explicitly verified under a defined rule or emits clearer expected/observed target diagnostics.
