## Label Justifications

- `P2`: This is a real hook/channel delivery-status bug with limited blast radius, noisy failures, and duplicate-send risk rather than a core outage.
- `impact:message-loss`: The report is about channel delivery bookkeeping after a Telegram send and can lead operators to retry an already-delivered message.

## Summary

Keep open: current main still appears vulnerable to this hook path because hook dispatch can create announce delivery with only a channel while source-delivery verification requires a concrete delivery target before a successful message-tool send can satisfy the wrapper.

## What This Changes

The issue asks OpenClaw to treat a successful Telegram `message(action="send", target=...)` hook send as verified, or to report the exact expected-vs-observed target mismatch.

## Best Possible Solution

Define the hook delivery verification policy, then add regression coverage so successful explicit Telegram sends either satisfy an intentionally targetless hook path or fail with clear expected and observed target details.

## Reproduction Assessment

Yes from source inspection, though I did not run a live hook: a hook announce delivery can have `channel` without `to`, while the verifier refuses to match a successful message-tool target unless the delivery plan also has `to`. The reporter's logs line up with that path.

## Solution Assessment

Unclear until maintainers choose policy: marking targetless hook sends verified is plausible but security/delivery-sensitive. The narrow maintainable path is to align hook dispatch, source-delivery verification, and diagnostics in one focused change with regression tests.
