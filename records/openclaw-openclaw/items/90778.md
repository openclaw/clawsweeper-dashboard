## Summary

Current main still has the hook delivery-status gap: hook dispatch can create an announce delivery with a channel but no concrete target, while source delivery verification refuses to treat a message-tool send as verified unless the resolved delivery target has both channel and recipient.

## What This Changes

The issue asks OpenClaw to treat successful Telegram `message` tool sends from Gmail hook runs as verified delivery, or to report an exact expected/observed target mismatch instead of a generic wrapper failure.

## Best Possible Solution

Define the hook delivery policy, then make source-delivery verification and diagnostics agree: either verified explicit hook sends satisfy the run, or targetless hook delivery fails with expected and observed target details.

## Reproduction Assessment

Yes. Source inspection on current main shows a hook announce path can lack `to`, and the verifier cannot mark a message-tool send verified without a resolved delivery recipient.

## Solution Assessment

Unclear. The narrow fix location is source-delivery verification or hook delivery construction, but maintainers need to decide whether targetless hook sends may satisfy delivery or should only produce clearer diagnostics.
