## Summary

Keep open: current main still captures explicit Telegram `target` evidence, but hook-created announce delivery can have no planned `to`, and the verifier still refuses to mark that send as matched without a resolved delivery target.

## What This Changes

The issue reports a Gmail hook run that successfully sent a Telegram message via `message(action="send", channel="telegram", target=...)` but was still recorded as a hook delivery error.

## Best Possible Solution

Define the targetless hook delivery contract, then add focused hook/cron/source-delivery coverage so successful explicit Telegram sends either verify under that rule or fail with expected and observed target details.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: hook dispatch can create announce delivery with no `to`, while `sourceDeliveryTargetsMatch()` rejects observed message-tool target evidence unless the planned delivery has `delivery.to`.

## Solution Assessment

Unclear until maintainers choose policy. The narrowest maintainable fix should stay in the hook/cron/source-delivery boundary, but the safe rule for targetless hook sends needs explicit delivery policy direction.
