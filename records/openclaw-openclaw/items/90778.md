## Summary

Keep open: current main still has the reported mismatch path. Hook dispatch can create announce delivery with a channel but no concrete `to`, while source-delivery verification refuses to mark a message-tool send verified unless the planned delivery target also has `to`.

## What This Changes

The issue reports hook delivery verification marking a Gmail-triggered Telegram message-tool send as failed even after the tool successfully sends to the explicit Telegram target.

## Best Possible Solution

Define the hook delivery policy, then add focused regression coverage so successful explicit Telegram sends either satisfy an intentionally targetless hook path or fail with clear expected and observed target details.

## Reproduction Assessment

Yes from source inspection: hook announce delivery can be built without `to`, and source-delivery verification requires `delivery.to` before it will verify the message-tool target. I did not run a live Gmail-to-Telegram hook.

## Solution Assessment

Unclear until maintainers choose policy: verifying targetless hook sends is plausible but delivery-sensitive. The safest implementation is a focused hook/cron/source-delivery change with explicit diagnostics and regression tests.
