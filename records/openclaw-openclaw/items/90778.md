## Label Justifications

- `P2`: The issue reports a real hook delivery-accounting bug with duplicate-send risk, but the blast radius is limited to hook and cron delivery verification.
- `impact:message-loss`: A Telegram message can be sent successfully while the hook run is recorded as failed, making operator retries or duplicate sends more likely.

## Summary

Keep open: current main and v2026.6.9 still have the source-level false-error path for targetless hook announce delivery. The related merged cron PR fixed empty target-evidence false success, but this issue still needs a maintainer policy choice for successful explicit message-tool sends when the hook wrapper has no configured expected target.

## What This Changes

The issue requests that Gmail hook runs which successfully send via `message(action="send", channel="telegram", target=...)` verify delivery correctly or report the exact target mismatch instead of recording the hook as failed.

## Best Possible Solution

Define the targetless hook delivery rule, then update hook, cron, and source-delivery verification so successful explicit Telegram sends either verify under that rule or fail with expected and observed target details.

## Reproduction Assessment

Yes, source-level. Current main can build a hook announce delivery with `channel` but no `to`, captures the message-tool `target` as a sent target, and still requires `delivery.to` before verification can pass.

## Solution Assessment

Unclear until maintainers choose policy. The narrow maintainable fix is to define whether targetless hook announce runs may be satisfied by explicit message-tool sends, then cover that rule with hook/cron/source-delivery tests.
