## Summary

Keep open. Current main still has the same source-level lane collision described in the report: deferred turn maintenance waits for the resolved session lane to drain, while normal inference is enqueued onto that same resolved session lane, and existing tests currently assert that foreground turns can overtake deferred maintenance work.

## What This Changes

The issue requests a fix for background context-engine turn maintenance that can starve behind steady same-session chat traffic and expose stale transcript state to later turns.

## Best Possible Solution

Define the session-lane ordering contract so transcript-correctness maintenance reaches a safe checkpoint before later same-session inference reads history, while leaving true indexing/summarization bookkeeping deferred.

## Reproduction Assessment

Yes, source-reproducible but not locally executed: configure a context engine with `turnMaintenanceMode: "background"`, keep one session lane continuously active or queued, and the current worker will keep polling that same lane before running maintenance. The source and existing tests are enough to show the starvation path under steady same-session traffic.

## Solution Assessment

Unclear. The issue's Option B looks like the smallest plausible mitigation, but current tests assert the opposite priority, so the best fix should be chosen by maintainers before a repair PR changes queue ordering.
