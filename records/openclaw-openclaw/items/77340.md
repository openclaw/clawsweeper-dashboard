## Summary

Keep open: current main still waits for the same resolved session lane used by inference, and existing tests explicitly allow foreground turns and deferred rewrite work to run ahead of maintenance, so there is no implemented or duplicate close path.

## What This Changes

The issue asks OpenClaw to prevent background context-engine turn maintenance from livelocking behind continuous same-session chat traffic and accumulating trailing assistant turns.

## Best Possible Solution

Ensure transcript-correctness maintenance for a session reaches a safe committed state before the next same-session inference reads history, with regression coverage for a saturated session lane and for deferred rewrites that currently race behind foreground turns.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main waits for the same resolved session lane used by inference, and tests show foreground turns run before deferred maintenance. I did not execute a live Telegram timing repro, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear. Option B from the report looks like the smallest plausible repair, but current tests encode the opposite priority, so maintainers should choose whether to gate inference on prior maintenance or move correctness writes inline.
