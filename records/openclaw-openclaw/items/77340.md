## Summary

Keep open. Current main still has the same source-level lane collision, and the existing tests intentionally let foreground turns pass deferred maintenance, but changing that behavior would revise the background-maintenance scheduling contract introduced for the lossless-claw no-stall work.

## What This Changes

The issue reports a deferred turn-maintenance livelock where maintenance waits on the same session lane used by user-turn inference, allowing trailing assistant messages to accumulate under steady chat traffic.

## Best Possible Solution

Define the transcript-correctness invariant for background context-engine maintenance, then update the worker and tests so a same-session user turn cannot read pre-maintenance transcript state while heavy bookkeeping can still stay deferred.

## Reproduction Assessment

Yes, at source level with high confidence. Current main resolves deferred maintenance and user-turn inference through the same `sessionKey` lane, and existing tests prove foreground turns are allowed to pass maintenance while it waits.

## Solution Assessment

Unclear. The report’s Option B is a plausible narrow mitigation, but current tests and the original merged PR intentionally prefer foreground turns, so maintainers should choose the invariant before code changes.
