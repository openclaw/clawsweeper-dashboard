## Summary

Keep open: current main still has the lane-drain wait and session-lane rewrite handoff described, and existing tests encode the contested ordering, so this is not cleanup-closeable.

## What This Changes

The issue reports that background context-engine turn maintenance waits for the same session lane used by user-turn inference, allowing steady same-session traffic to defer transcript rewrites indefinitely and accumulate trailing assistant turns.

## Best Possible Solution

Pick a transcript-correctness invariant for context-engine turn maintenance, then update the session-lane scheduling and rewrite handoff so prior-turn maintenance reaches a safe checkpoint before the next inference reads that session transcript.

## Reproduction Assessment

Yes, source-reproducible: enqueue repeated work on one `resolveSessionLane(sessionKey)` while a background turn-maintenance task is waiting, and the current wait loop can remain unsatisfied. I did not run the live Telegram/lossless-claw signature in this read-only review.

## Solution Assessment

Unclear pending maintainer choice: the issue's Option B is the smallest plausible mitigation, but changing the existing foreground-before-maintenance contract should be decided deliberately and covered by regression tests.
