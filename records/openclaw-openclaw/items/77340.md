## Summary

Keep open: current main still waits for the same session lane used by user turns to drain before background turn maintenance runs, and existing tests assert that foreground turns and deferred rewrites can overtake that maintenance. This should not be closed as stale or implemented; the fix needs maintainer scheduling-contract judgment because it changes the foreground-responsiveness guarantee shipped in #65233.

## What This Changes

The issue asks OpenClaw to prevent deferred context-engine turn maintenance from livelocking behind rapid same-session traffic and accumulating uncommitted trailing assistant turns.

## Best Possible Solution

Split the post-turn lifecycle so transcript-correctness state is committed before the next same-session inference while heavier context-engine bookkeeping remains coalesced background work, with a regression test for rapid same-session turns.

## Reproduction Assessment

Yes, from source inspection: user turns and deferred maintenance use the same resolved session lane, and the worker waits for that lane to be empty before committing maintenance. I did not run a live Telegram reproduction in this read-only environment.

## Solution Assessment

Unclear as an automation target: Options A or B are directionally aligned with transcript correctness, but they intentionally change the foreground-turn priority contract added by #65233. Maintainers should choose the exact split between correctness work and deferrable bookkeeping before implementation.
