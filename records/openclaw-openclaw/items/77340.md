## Summary

Keep open: current main still contains the source-backed same-session lane wait described in the report, and existing tests intentionally let foreground turns and deferred rewrites outrun background maintenance. A fix likely needs maintainer direction because it changes the context-engine scheduling contract rather than being a trivial cleanup.

## What This Changes

The issue reports that deferred context-engine turn maintenance can starve behind steady same-session inbound traffic, leaving prior-turn transcript maintenance uncommitted and allowing stale trailing assistant messages to accumulate.

## Best Possible Solution

Define a non-starving transcript-correctness checkpoint for background context-engine maintenance, likely by gating the next same-session inference on the prior turn’s safe maintenance point or moving correctness rewrites inline while leaving heavier indexing/bookkeeping deferred.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not end-to-end in this review; source inspection gives a clear synthetic repro where a same-session lane kept non-empty leaves the worker in its wait loop, but no live Telegram or plugin harness was run.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the report’s Option B is a plausible short-term direction, but current tests intentionally encode the opposite priority, so maintainers should choose the scheduler contract before an automated patch changes it.
