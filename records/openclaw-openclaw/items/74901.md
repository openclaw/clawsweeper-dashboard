## Summary

This PR should stay open because it targets a real TTS voice-delivery gap, but the proposed patch is too broad for the local-media trust boundary. Current main still drops the local TTS media when the exact raw `tts` name is absent from `builtinToolNames`, while the PR would let any result-controlled `trustedLocalMedia` flag preserve local filesystem paths.

## What This Changes

The PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` preserves local media paths even when `builtinToolNames` does not include the emitting tool name.

## Best Possible Solution

Revise this into a narrow TTS-owned fix: honor trusted local media only for the exact raw built-in `tts` result with no external provenance, keep the existing registered-name gate for every other tool, and add positive plus negative regression tests.

## Reproduction Assessment

Yes. Current main can be checked at the code level by calling `filterToolResultMediaUrls("tts", ["/tmp/reply.opus"], ttsResult, new Set(["web_search"]))`, which filters the local path because `tts` is absent; the PR regression is reproduced with an unregistered or MCP-provenance result that sets `trustedLocalMedia: true`.

## Solution Assessment

No. Respecting TTS-owned trusted media is the right direction, but this implementation is too broad because the result field alone is not proof that the local path came from the built-in TTS tool.
