## Summary

Keep open. The PR targets a real TTS voice-delivery gap, but the patch broadens a local-media trust boundary by letting a result-controlled `trustedLocalMedia` flag bypass the existing provenance and exact registered-name checks.

## What This Changes

The PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` preserves local media paths even when `builtinToolNames` does not include the emitting tool name.

## Best Possible Solution

Revise this PR or replace it with a narrow fix that honors `trustedLocalMedia` only for an exact raw built-in `tts` result with no external provenance, while preserving the existing gate for every other tool and adding positive plus negative regression tests.

## Reproduction Assessment

Yes. Current main can be checked at the code level with a TTS local OGG result and `builtinToolNames` defined without `tts`, which filters the media to an empty list; the PR regression is an unregistered or MCP-provenance result that sets `trustedLocalMedia: true` and would preserve a local path.

## Solution Assessment

No. Respecting TTS-owned trusted media is the right direction, but this implementation is too broad because the result field alone is not proof that the local path came from the built-in TTS tool.
