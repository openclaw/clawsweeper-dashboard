## Summary

Keep open. The PR targets a real TTS/Telegram delivery gap from #74752, but the patch is not merge-ready because it lets a result-controlled `details.media.trustedLocalMedia` flag bypass the existing local-file media provenance and exact registered-name gates.

## What This Changes

The PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` preserves local media paths even when `builtinToolNames` does not include the emitting tool name.

## Best Possible Solution

Revise this change so trusted local TTS media is honored only for an exact raw built-in `tts` result with no external provenance, while preserving the existing registered-name gate for every other tool and adding focused positive and negative regression tests.

## Reproduction Assessment

Yes. At the code level, current main filters a local TTS OGG path to `[]` when `builtinToolNames` is defined without `tts`; the proposed patch can be checked by an unregistered or MCP-provenance result that sets `trustedLocalMedia: true` and would preserve a local path.

## Solution Assessment

No. Respecting TTS-owned trusted media is the right direction, but this PR treats the result field itself as authorization; the safer fix is to bind the bypass to exact built-in TTS provenance and keep the generic local-media gate intact.
