## Summary

Keep open. The linked TTS voice-delivery bug is plausible, but this PR is not merge-ready because it lets a result-controlled `details.media.trustedLocalMedia` flag bypass the existing exact registered-tool-name gate for local media paths.

## What This Changes

The PR changes `filterToolResultMediaUrls` so local media paths are preserved when `details.media.trustedLocalMedia === true`, even if `builtinToolNames` does not include the emitting tool name.

## Best Possible Solution

Revise or replace this PR with a narrow fix that preserves local media only for a proven built-in TTS result with exact raw `tts` provenance and no external MCP provenance, or fixes the construction of `builtinToolNames` so the actual built-in TTS execution is represented. Add positive and negative regression tests plus the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is available by passing a TTS result with `/tmp/reply.opus`, `details.media.trustedLocalMedia: true`, and a `builtinToolNames` set that omits `tts`; current main filters the local path out, while the PR would keep it. The security regression is also reproducible by giving an unregistered or external result the same flag and a local path.

## Solution Assessment

No. Respecting TTS-owned structured media is the right direction, but this implementation is not the best fix because the result flag alone is not proof that the local path came from the built-in TTS tool.
