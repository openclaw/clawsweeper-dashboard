## Summary

Keep open. The PR targets a real remaining TTS delivery bug, but the proposed bypass is too broad because it lets a result-controlled `details.media.trustedLocalMedia` flag preserve local filesystem paths without proving the result came from the built-in TTS tool.

## What This Changes

The PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` preserves local media paths even when `builtinToolNames` does not include the emitting tool name.

## Best Possible Solution

Revise this PR or replace it with a narrow fix that lets trusted structured media bypass the registration gate only for an exact raw built-in `tts` result with no external provenance, while keeping the existing local-media restrictions for every other tool and adding positive plus negative regression tests.

## Reproduction Assessment

Yes. Current main can be reproduced at the source level with a `tts` tool result containing `/tmp/reply.opus`, `details.media.trustedLocalMedia: true`, and `builtinToolNames` set without `tts`, which filters the media to an empty list before queueing.

## Solution Assessment

No. Respecting trusted TTS-owned media is the right direction, but this implementation is not the best fix because the result flag alone is not proof that the local path came from the built-in TTS tool.
