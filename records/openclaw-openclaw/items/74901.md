## Summary

Keep open. The PR targets a real TTS voice-delivery failure, but the current diff broadens a local-file media trust boundary by letting a result-controlled `trustedLocalMedia` flag bypass external-provenance and exact registered-name checks.

## What This Changes

The branch changes `filterToolResultMediaUrls` so local media URLs are preserved whenever `details.media.trustedLocalMedia === true`, bypassing the registered built-in tool-name gate.

## Best Possible Solution

Keep the TTS delivery fix narrow: allow `trustedLocalMedia` to preserve local paths only for an exact, non-external built-in `tts` result, keep the raw registered-name gate for all other tools, add positive and negative media-filter tests, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes. At the code level, current main filters a local TTS OGG path when `builtinToolNames` is defined but omits `tts`; the PR regression can be reproduced by giving an external or unregistered tool result `details.media.trustedLocalMedia: true` and observing that the diff would preserve the local path.

## Solution Assessment

No. Respecting TTS-owned trusted media is the right direction, but this implementation is too broad because it treats the result field alone as authorization instead of tying it to exact built-in TTS provenance.
