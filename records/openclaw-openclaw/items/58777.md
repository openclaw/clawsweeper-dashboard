## Summary

Keep open. Current main has internal fallback/model trace data and partial hook context improvements, but it still does not expose the requested closed-loop effective provider/model/thinking metadata on the sub-agent run, task, lifecycle, or wait surfaces that a parent run would use to audit pin fidelity. The related router fallback contract in #63139 is still open, so this issue is not superseded.

## What This Changes

The issue requests telemetry and lifecycle/session reconciliation so sub-agent runs report the effective provider, model, and thinking level after fallback or router selection differs from an explicit model pin.

## Best Possible Solution

Define an additive effective-run metadata contract, populate it from the final runtime provider/model/thinking after fallback and hook selection, and reconcile it across sub-agent registry records, background tasks or wait results, lifecycle events, and docs with regression coverage for explicit pins, fallback to another model, and unsupported-thinking re-clamping.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: spawn a sub-agent with an explicit model and a fallback or router rewrite, let `runWithModelFallback()` select a different candidate, then inspect `agent.wait`, sub-agent registry/task records, and lifecycle events and observe that no effective provider/model/thinking tuple is emitted.

## Solution Assessment

Unclear. The requested direction is valid, but the narrowest maintainable fix needs maintainer agreement on which public surfaces own the effective tuple and how that interacts with the still-open router-hook fallback contract.
