## Summary

Keep open, but not merge-ready: the default-account priority direction is useful, yet the new startup-slot predicate can count enabled but unconfigured Discord accounts and delay the first runnable bot; the PR also has only mock/test validation, not real behavior proof.

## What This Changes

The PR changes Discord startup delay calculation to promote the resolved default account first and filter candidates through `isDiscordAccountEnabledForRuntime`, with regression tests for default ordering and duplicate-token defaults.

## Best Possible Solution

Keep the default-account promotion, but build startup slots from accounts that are both configured and runtime-enabled, then prove the fixed ordering in a real Discord gateway startup run before merge.

## Reproduction Assessment

Yes. Source inspection shows an enabled Discord account with no token is skipped by the gateway as unconfigured, but the PR's startup-slot list would still count it and delay configured accounts sorted after it.

## Solution Assessment

No. Promoting the configured default account is the right direction, but the new predicate is too broad; the fix should preserve the configured/token check alongside duplicate-token runtime filtering.
