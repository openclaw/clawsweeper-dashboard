## Summary

Keep open. Current main narrows the blast radius by treating WhatsApp as an on-demand external plugin, but the workspace still contains the Baileys-to-libsignal GitHub tarball resolution and two open closing PRs are already attached for the dependency-policy repair.

## What This Changes

The issue asks OpenClaw to stop Baileys' transitive libsignal dependency from failing pnpm installs with blockExoticSubdeps and breaking inbound replies when WhatsApp is disabled.

## Best Possible Solution

The end state should keep WhatsApp from blocking unrelated channels and make the WhatsApp package dependency graph install under pnpm's exotic-subdependency policy without broadening supply-chain trust globally.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the dependency-resolution failure: current main still resolves Baileys' `libsignal` through a GitHub tarball, but I did not establish a full live gateway reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the linked PRs are reviewed; replacing the transitive resolution with a registry package directly addresses `blockExoticSubdeps`, while a build-script allowlist alone appears to target a different pnpm policy axis.
