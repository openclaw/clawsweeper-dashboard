## Summary

Keep open: current main still sends Xiaomi TTS requests with the `api-key` header, and the newer live Token Plan comment shows the user problem remains unresolved. The linked closing PR is open but has no file changes, so this issue should stay open until a real Xiaomi TTS endpoint/auth fix lands.

## What This Changes

Allow Xiaomi MiMo TTS Token Plan users to use their Token Plan credentials and endpoint successfully from OpenClaw.

## Best Possible Solution

Fix Xiaomi Token Plan TTS inside the bundled Xiaomi plugin by correcting endpoint resolution for Gateway TTS, deciding the auth header from verified provider behavior, and updating focused tests plus Token Plan docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection plus the May 10 live report: current main still defaults Xiaomi TTS to the pay-as-you-go base URL path and sends `api-key`, while a real Token Plan Gateway run failed until the Token Plan base URL was supplied through `XIAOMI_BASE_URL`. I did not rerun the live endpoint locally because no Token Plan key was available.

## Solution Assessment

Is this the best way to solve the issue? Unclear as originally framed: switching only to Bearer may be incomplete because live AMS data says `api-key` also worked on the Token Plan endpoint. The safer fix is to make Xiaomi TTS reliably use the Token Plan endpoint/config first, then change or preserve auth based on verified compatibility.
