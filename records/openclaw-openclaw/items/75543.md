## Summary

Keep this PR open. Current main still lacks a lower-level `writeConfigFile` no-op guard, so the branch addresses a real writer-side churn path, but the proposed implementation is not merge-ready because its early returns violate the writer return contract and its semantic comparison is still order-sensitive.

## What This Changes

The PR adds byte-identical and meta-stripped no-op short-circuits to `createConfigIO().writeConfigFile`, plus a regression test asserting a repeated write does not change mtime, log an overwrite, or rotate backups.

## Best Possible Solution

Land a narrow writer-level no-op guard that preserves the full `writeConfigFile` return shape, compares a stable canonical representation after removing only volatile metadata, leaves real config changes on the existing audit/backup/suspicious-write path, and documents the user-facing fix in the changelog.

## Solution Assessment

No. The direction is appropriate for the lower-level writer, but this implementation should return `{ persistedHash, persistedConfig }`, use a key-order-insensitive stable comparison for semantic no-ops, and add the required changelog entry before merge.
