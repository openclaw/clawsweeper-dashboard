## Summary

This PR targets a valid config-writer bug, but the proposed implementation still has merge-blocking correctness issues: the new no-op branches do not satisfy the `writeConfigFile` return contract and the semantic comparison is not aligned with the repository's JSON5/canonical config handling.

## What This Changes

The PR adds early returns in `src/config/io.ts` to skip config file writes when serialized config content is byte-identical or differs only by volatile metadata, plus a regression test in `src/config/io.write-config.test.ts`.

## Best Possible Solution

Land a narrow writer-boundary guard that returns the complete persisted result, uses the repository's canonical JSON5-aware stable config representation after removing only volatile metadata, and preserves audit, backup, and suspicious-write behavior for real writes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createConfigIO().writeConfigFile` twice with unchanged stable config and current main continues past hash calculation into write side effects instead of returning early.

## Solution Assessment

No. The writer-boundary fix is the right direction, but this PR is not the best mergeable solution until it returns `persistedConfig` and compares canonical JSON5-aware stable content instead of raw JSON insertion order.
