## Summary

Keep open. The underlying writer no-op bug is still valid on current main, but this PR is not mergeable as written because the new early returns violate the writer return contract and the semantic comparison can miss valid JSON5 or key-order-only no-ops.

## What This Changes

The branch adds early returns in `src/config/io.ts` to skip byte-identical or volatile-meta-only config writes, plus a regression test in `src/config/io.write-config.test.ts`.

## Best Possible Solution

Land a narrow writer-boundary guard that returns the complete persisted result, compares JSON5-parsed stable content after removing only volatile metadata, and preserves audit, backup, and suspicious-write behavior for real changes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createConfigIO().writeConfigFile` twice with equivalent persisted output and current main still proceeds from hash calculation into write, backup, and audit side effects instead of returning.

## Solution Assessment

No. The writer-boundary direction is right, but this patch should not merge until the no-op branches return `persistedConfig` and the semantic comparison uses the repository's JSON5-aware, stable config representation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Return persistedConfig from no-op writes:** `src/config/io.ts:1603-1607`
  - body: `writeConfigFile` is declared as returning both `persistedHash` and `persistedConfig`, but both new no-op branches return only `{ persistedHash }`. That breaks the inner writer contract and direct callers/tests that rely on the canonical persisted config result.
  - confidence: 0.96

  - confidence: 0.88
