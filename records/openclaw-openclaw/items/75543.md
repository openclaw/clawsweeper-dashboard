## Summary

Keep this PR open. The lower-level writer optimization is still useful beyond the current gateway-level `config.patch` no-op, but this patch is not merge-ready because the new early returns violate the existing return contract and the semantic comparison still misses key-order-only no-ops described by the linked report.

## What This Changes

The PR adds an early no-op path in `src/config/io.ts` that skips writing `openclaw.json` when the next config matches the current file by byte hash or by a meta-stripped hash, plus a regression test for repeated `writeConfigFile` calls.

## Best Possible Solution

Land a narrow writer-level no-op guard that preserves the existing return contract, uses a stable canonical comparison after removing only volatile metadata, keeps the existing gateway `config.patch` no-op behavior intact, and covers repeated direct writer calls with focused regression tests.

## Reproduction Assessment

Yes. The writer path on current main stamps `meta.lastTouchedAt` on every write and then proceeds to backup and rename even when only that metadata changed, while the PR's compile-time return-contract issue is reproducible by applying the diff or inspecting the declared return type.

## Solution Assessment

No. The direction is appropriate for the lower-level writer, but this implementation should return the full `{ persistedHash, persistedConfig }` shape and use a key-order-insensitive stable comparison instead of hashing insertion-order JSON strings.
