## Summary

Keep this PR open, but do not merge it as written. The reported config-writer thrash is a real current-main gap and the PR targets the right boundary, but the patch breaks the writer return contract, only partially handles semantic no-ops, and is missing the required changelog entry.

## What This Changes

The PR adds early returns to `createConfigIO().writeConfigFile` for byte-identical or volatile-meta-only config writes and adds a regression test for avoiding disk writes, overwrite logs, and backup rotation.

## Best Possible Solution

Land a narrow writer-boundary no-op guard that preserves the full return shape, compares canonical JSON5-aware stable config content after removing only volatile metadata, leaves real-write audit/backup/suspicious-write behavior intact, and documents the fix.

## Reproduction Assessment

Yes. A focused reproduction can call `createConfigIO().writeConfigFile` twice with the same stable config on current main and observe the second call continue into writer side effects; the PR-specific return-contract break is visible from the declared return type and diff.

## Solution Assessment

No. The writer-boundary direction is appropriate, but this implementation is not the best mergeable fix until it returns `persistedConfig`, uses canonical config comparison, and adds the changelog entry.
