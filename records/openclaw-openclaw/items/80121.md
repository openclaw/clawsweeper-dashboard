## Summary

Keep open. The PR targets a valid Feishu bug, but the current guard is too broad because it drops every tool-kind error payload while shared dispatch deliberately preserves error tool payloads; it also still lacks required real Feishu behavior proof.

## What This Changes

The PR adds a Feishu reply-dispatcher guard for `isError` tool-kind payloads and unit tests covering suppressed tool warnings plus preserved final errors.

## Best Possible Solution

Keep the fix Feishu-owned, but narrow it so stale recovered tool warnings are suppressed without hiding genuine tool failures, then land after real Feishu proof and focused regression coverage.

## Reproduction Assessment

Yes for a source-level path: current main can deliver `isError` tool payloads through Feishu, and the linked Feishu report describes the exact warning in channel messages. I did not see after-fix proof from a live Feishu setup in the PR.

## Solution Assessment

No, not as written. A Feishu dispatcher fix is the right owner boundary, but this implementation should distinguish stale recovered warnings from genuine tool errors instead of dropping every tool-kind error payload.
