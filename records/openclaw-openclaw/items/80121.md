Keep open: the PR still filters only tool-kind Feishu deliveries, while the linked warning is source-proven to be built as an `isError` reply and delivered through the final-reply path; the supplied proof is also unit-test-only.

## What This Changes

The PR adds a Feishu dispatcher guard that drops `payload.isError` tool-kind deliveries and adds dispatcher tests for suppressing tool errors while preserving final errors.

## Best Possible Solution

Fix the stale Feishu warning at the shared embedded-run warning policy or the Feishu final-delivery boundary that sees recovered mutating-tool warnings, with coverage for the actual final-dispatch path and genuine final failures preserved.

## Reproduction Assessment

Yes, source-level: current main builds the mutating-tool warning with `isError: true`, returns it as a reply item, then sends it through `sendFinalPayload`/`sendFinalReply`, so Feishu receives `kind: "final"`. I did not live-test a Feishu workspace.

## Solution Assessment

No. The PR targets `kind: "tool"` dispatcher deliveries, but the reported warning is source-proven as a final reply; the narrower maintainable fix should suppress that recovered warning before or at the final-delivery boundary.
