## Summary

Keep open: the patch targets `kind: "tool"` deliveries, but the linked Feishu warning is source-proven to be built as an `isError` final reply, and the PR still only supplies unit-test proof rather than real Feishu behavior proof.

## What This Changes

The PR adds a Feishu reply-dispatcher guard that drops `isError` tool-kind payloads and tests that tool errors are suppressed while final errors still deliver.

## Best Possible Solution

Suppress recovered mutating-tool warnings at the shared warning policy or the Feishu final-delivery boundary that can identify those warnings, while preserving genuine final failures and adding coverage for the actual final-dispatch path.

## Reproduction Assessment

Yes, source-level: current main creates a mutating-tool warning as an `isError` reply item, then the final reply loop sends it through `sendFinalReply`, so Feishu receives it as `kind: "final"`. I did not run a live Feishu session.

## Solution Assessment

No. This PR filters only `kind: "tool"`, while the reported warning is delivered as a final reply, so the maintainable fix should target the final-warning path or an earlier warning policy.
