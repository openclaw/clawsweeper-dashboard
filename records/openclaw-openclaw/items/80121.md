## Summary

Keep open: this is a plausible focused Feishu bug fix, but the current patch does not cover the linked warning path because that warning is built as an error reply and delivered as a final payload, while the PR only suppresses tool-kind deliveries. The proof is also unit-test-only, so it does not satisfy the external PR real-behavior proof gate.

## What This Changes

The PR adds an early Feishu reply-dispatcher return for `payload.isError` tool deliveries and adds dispatcher tests for tool-error suppression and final-error delivery.

## Best Possible Solution

Fix the stale Feishu warning at the embedded-run warning policy or Feishu reply-options boundary, with coverage that drives the actual final-dispatch warning path and still preserves genuine final failure replies.

## Reproduction Assessment

Yes, source-level: current main builds the mutating-tool warning as an `isError` reply payload and dispatches reply results through `sendFinalPayload`, which reaches Feishu as `kind: "final"`. I did not establish a live Feishu workspace reproduction in this read-only review.

## Solution Assessment

No. The current PR guard is not the best fix because it only suppresses `kind: "tool"` deliveries, while the linked warning path reaches Feishu as a final error payload; the safer fix should target that actual path.
