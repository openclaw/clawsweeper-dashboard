## Summary

Keep open: the branch fixes the reported exec warning direction in the shared presentation layer, but source inspection confirms it still mishandles documented raw tool-progress metadata and the PR has no real after-fix cron/chat/CLI proof.

## What This Changes

The PR changes embedded-runner tool-failure warning formatting so exec-like failures show the tool label, command subject, and concise exit code as separate pieces.

## Best Possible Solution

Keep the shared failure-label approach, extract raw-mode trailing inline command metadata before wrapping the subject, add a regression for that shape, and require redacted real user-path proof before merge.

## Reproduction Assessment

Yes, at source level: current main passes raw tool progress detail into exec metadata, stores that value on `lastToolError`, and the warning payload path consumes the metadata as user-visible failure text. I did not run a live cron/chat reproduction in this read-only review.

## Solution Assessment

No, not yet: moving the failure label outside the command subject is the right layer, but the current patch misses the documented raw-detail metadata shape. A narrow extraction of trailing raw inline command metadata plus a regression is the safer fix.
