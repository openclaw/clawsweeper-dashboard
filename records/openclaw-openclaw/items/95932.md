## Label Justifications

- `merge-risk: 🚨 compatibility`: Existing contributors with relative or same-file global hooksPath values can see their local commit flow break after merge.
- `merge-risk: 🚨 availability`: The same-file case can recurse repeatedly and hang or stall local pre-commit execution.

## Summary

Keep this PR open, but it is not merge-ready: the global-hook chaining goal is reasonable, yet the patch can recurse into the repo hook when the global hooksPath resolves to the same file.

## What This Changes

The PR changes `git-hooks/pre-commit` to run an executable global `pre-commit` hook before the existing local staged-file formatting flow.

## Best Possible Solution

Keep the global pre-commit chaining behavior, but resolve and compare the candidate global hook with the current repo hook and skip chaining when they are the same file.

## Reproduction Assessment

Yes. Source plus Git docs show that with global `core.hooksPath=git-hooks` and the repo-local hook path also set to `git-hooks`, the added block calls `git-hooks/pre-commit` from inside itself.

## Solution Assessment

No. The hook is the right layer for the hardening, but the implementation needs a same-file recursion guard before it is the maintainable fix.

- **[P2] Guard against chaining the hook into itself:** `git-hooks/pre-commit:10-12`
  - body: Git accepts relative `core.hooksPath` values, and hooks run from the worktree root, so a global value like `git-hooks` resolves to this repository's hook.
