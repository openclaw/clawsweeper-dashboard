## Summary

Keep open. The PR targets a real source-visible timeout propagation gap, but it is not merge-ready because the branch is conflicting, lacks after-fix real behavior proof, and introduces blocking timeout-contract regressions.

## What This Changes

The branch forwards `sessions_send` and `runAgentStep` timeouts into nested agent calls, passes run timeout into session write-lock acquisition, and adds focused tests.

## Best Possible Solution

Rework the branch on current main to use numeric target timeouts, preserve fire-and-forget's existing run budget, and align lock acquisition, run execution, and waits under one maintainer-approved deadline contract with real A2A/gateway proof.

## Reproduction Assessment

Yes, at source level: current main accepts `sessions_send.timeoutSeconds` and waits with that budget, but the nested target `agent` call does not receive a corresponding run timeout. I did not run a live contended-session scenario in this read-only review.

## Solution Assessment

No. The direction is plausible, but this patch is not the best implementation because it sends string timeouts, changes fire-and-forget semantics, and keeps lock, run, and wait deadlines inconsistent.

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass numeric agent timeouts:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: `AgentParamsSchema` requires `timeout` to be an integer and the handler validates before dispatch. Passing `String(targetTimeoutSeconds)` here means real `sessions_send` gateway calls can fail with `invalid agent params` even though the mocked tests pass.
  - confidence: 0.97
- **[P1] Preserve fire-and-forget run budgets:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For `timeoutSeconds: 0`, current behavior is enqueue-and-return while the child run keeps the normal agent budget. Forcing a 30-second target timeout can kill legitimate background work early.
  - confidence: 0.91
- **[P2] Count lock waiting against one deadline:** `src/agents/pi-embedded-runner/run/attempt.ts:1705`
  - body: Passing the full `params.timeoutMs` into lock acquisition lets a run spend the whole caller budget waiting for the session lock before execution starts, so total runtime can still exceed what the sender waited for.
  - confidence: 0.86
- **[P2] Match helper run budget to helper wait budget:** `src/agents/tools/agent-step.ts:52`
  - body: `runAgentStep` derives the target run timeout from the full `params.timeoutMs`, but the helper still waits with a 60-second cap. For larger values, the target run can continue after the helper returns `undefined`.
  - confidence: 0.88
