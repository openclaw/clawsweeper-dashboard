## Summary

Keep open: the PR targets a real source-level timeout propagation gap, but the branch still introduces gateway validation and timeout-budget regressions, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR propagates `sessions_send` and `runAgentStep` timeouts into nested `agent` calls, passes run timeout into session write-lock acquisition, and adds timeout propagation tests.

## Best Possible Solution

Rework the branch to use numeric target timeouts only for waited paths, preserve fire-and-forget run budgets, align lock/wait/run deadlines against current main, and add focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main accepts `sessions_send.timeoutSeconds` and waits with that budget, but the target `agent` RPC receives no corresponding run timeout; I did not run a live contended-session repro.

## Solution Assessment

No. The direction is reasonable, but this implementation should not merge until it preserves gateway types and fire-and-forget semantics and makes the effective lock, wait, and run budgets line up on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass agent timeouts as numbers:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: The `agent` RPC schema requires `timeout` to be an integer and validates before dispatch. Passing `String(targetTimeoutSeconds)` here, and the matching string in `runAgentStep`, makes real gateway calls fail with `invalid agent params` even though the mocked tests pass.
  - confidence: 0.97
- **[P2] Preserve fire-and-forget run budgets:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For `timeoutSeconds: 0`, current behavior is enqueue-and-return while the target run keeps the normal configured agent budget. Forcing `timeout: 30` would terminate legitimate background child work after 30 seconds.
  - confidence: 0.9
- **[P2] Count lock waiting against the run deadline:** `src/agents/pi-embedded-runner/run/attempt.ts:1705`
  - body: Passing the full `params.timeoutMs` into lock acquisition lets a target spend the caller's whole budget waiting for the session lock before the embedded run abort timer starts later, so the run can still exceed what the sender waited for.
  - confidence: 0.86
- **[P2] Align nested helper run and wait budgets:** `src/agents/tools/agent-step.ts:42-52`
  - body: `runAgentStep` derives the target run timeout from the full `params.timeoutMs`, but the helper still waits with `Math.min(params.timeoutMs, 60000)`. Values above 60 seconds leave the target run alive after the helper returns `undefined`.
  - confidence: 0.88
