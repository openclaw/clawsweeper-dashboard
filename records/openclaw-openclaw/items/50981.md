## Summary

Keep open: the timeout plumbing problem is source-reproducible, but this PR is not mergeable as-is because it violates the gateway timeout type contract, regresses fire-and-forget sends, leaves deadlines misaligned, and still lacks real behavior proof.

## What This Changes

The PR forwards `sessions_send` and `runAgentStep` timeout budgets into child `agent` calls, passes run timeout to session write-lock acquisition, and adds focused tests.

## Best Possible Solution

Rework the branch to send numeric target timeouts only for waited paths, preserve fire-and-forget’s normal run budget, and align lock acquisition, run execution, and caller wait against one coherent deadline.

## Reproduction Assessment

Yes, at source level. Current main accepts `sessions_send.timeoutSeconds` and waits with that budget, but the nested target `agent` call does not receive a corresponding run timeout; I did not run a live contended-session scenario.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best implementation because it sends string timeouts, changes fire-and-forget semantics, and keeps lock/wait/run deadlines inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass numeric agent timeouts:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: `agent` RPC params require `timeout` to be an integer, and the handler only treats numeric values as overrides. Passing `String(targetTimeoutSeconds)` here, and in `runAgentStep`, makes real gateway calls reject or ignore the timeout while the mocks still pass.
  - confidence: 0.97
- **[P2] Keep fire-and-forget runs on the normal budget:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For `timeoutSeconds: 0`, current behavior is enqueue-and-return while the child run inherits the normal agent timeout. Forcing a 30-second target timeout kills legitimate background work early.
  - confidence: 0.91
- **[P2] Count lock waiting against the same deadline:** `src/agents/pi-embedded-runner/run/attempt.ts:1705`
  - body: Passing the full `params.timeoutMs` into lock acquisition lets a run spend the whole caller budget waiting for the session lock before the embedded abort timer starts later, so total runtime can still exceed what the sender waited for.
  - confidence: 0.86
- **[P2] Match helper run budget to helper wait budget:** `src/agents/tools/agent-step.ts:42-52`
  - body: `runAgentStep` derives the target run timeout from the full `params.timeoutMs`, but the helper still waits with a 60-second cap. For larger values the target run can continue after the helper returns `undefined`.
  - confidence: 0.88
