## Summary

Keep open: current main still shows a source-level timeout propagation gap, but this PR is not merge-ready because it sends invalid string timeout params, changes fire-and-forget run budgets, leaves lock/run/wait deadlines inconsistent, lacks real behavior proof, and is currently conflicting.

## What This Changes

The branch forwards sessions_send and runAgentStep timeouts into nested agent calls, passes run timeout into session lock acquisition, and adds focused tests.

## Best Possible Solution

Rework the branch to send numeric target timeouts only for waited paths, preserve fire-and-forget's normal run budget, and align lock acquisition, run execution, and caller wait around one coherent deadline with real A2A/gateway proof.

## Reproduction Assessment

Source-reproducible, but not live-reproduced: current main accepts `sessions_send.timeoutSeconds` and waits with that budget while the nested target `agent` call does not receive a corresponding run timeout. I did not run a live contended-session scenario, and current main has changed lock defaults since the PR was opened.

## Solution Assessment

No. The direction is plausible, but this patch is not the best implementation because it sends string timeouts, changes fire-and-forget semantics, and keeps lock, run, and wait deadlines inconsistent.

  - confidence: 0.91

  - confidence: 0.86

  - body: `runAgentStep` derives the target run timeout from the full `params.timeoutMs`, but the helper still waits with a 60-second cap. For larger values, the target run can continue after the helper returns `undefined`.
