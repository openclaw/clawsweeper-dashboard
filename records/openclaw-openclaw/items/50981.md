## Summary

Keep this PR open. Current main still has the timeout propagation gaps, but the proposed patch is not merge-ready because it forwards string timeout values across a numeric gateway contract, changes fire-and-forget behavior, and leaves target-run, wait, and lock budgets inconsistent.

## What This Changes

This PR forwards sessions_send and A2A timeout values into nested agent runs, passes embedded-runner timeoutMs into session write-lock acquisition, and adds tests for those propagation paths.

## Best Possible Solution

Revise this PR or replace it with a narrower patch that forwards numeric agent.timeout only when appropriate, preserves fire-and-forget target-run budget unless maintainers intentionally change it, accounts for session-lock wait inside the effective run or caller deadline, aligns A2A target and wait budgets, and adds regression coverage for protocol validation, fire-and-forget sends, contended locks, and A2A helper waits.

## Reproduction Assessment

Yes. The PR's string timeout failure is deterministic by following the nested agent calls through validateAgentParams, and the current-main 10-second lock fallback is visible from runEmbeddedAttempt omitting timeoutMs plus acquireSessionWriteLock defaulting to 10_000.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it violates the gateway type contract and changes timeout semantics; the safer solution should use numeric protocol values and a shared deadline or remaining-budget model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward numeric sessions_send timeouts:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: The agent RPC schema declares timeout as an integer and agent requests are rejected before dispatch when validation fails. Sending String(targetTimeoutSeconds) makes real sessions_send calls fail with invalid agent params instead of starting the target run.
  - confidence: 0.94
- **[P1] Forward numeric A2A helper timeouts:** `src/agents/tools/agent-step.ts:52`
  - body: This helper also sends timeout as a string through the same validated agent RPC. Real A2A helper runs that include this field will be rejected by gateway validation before the nested run starts.
  - confidence: 0.93
- **[P1] Preserve fire-and-forget target budgets:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For timeoutSeconds: 0, the caller should return immediately while the target run keeps the normal agent budget. Forcing 30 seconds here cuts legitimate background work short and changes fire-and-forget behavior.
  - confidence: 0.88
- **[P2] Keep lock waits inside the run budget:** `src/agents/pi-embedded-runner/run/attempt.ts:1706`
  - body: The embedded abort timer is started later, so using the full params.timeoutMs for lock acquisition lets a contended run spend the whole caller budget waiting for the lock and then receive another full model runtime budget. Use a shared deadline or remaining budget instead.
  - confidence: 0.82
- **[P2] Align helper target and wait budgets:** `src/agents/tools/agent-step.ts:42`
  - body: For timeoutMs values above 60 seconds, the target run gets the full timeout while the helper still waits at most 60 seconds. That leaves the child run continuing after runAgentStep has already returned undefined.
  - confidence: 0.84
