## Summary

Keep open, but the current PR should not merge as-is: it targets a plausible timeout plumbing bug, yet the branch is stale against current main and introduces gateway type, fire-and-forget, lock-deadline, and helper wait-budget regressions.

## What This Changes

The PR forwards `sessions_send` and nested `runAgentStep` timeouts into child `agent` runs, passes embedded run timeout into session lock acquisition, and adds timeout propagation tests.

## Best Possible Solution

Rework the branch to send numeric target timeouts only for waited paths, preserve fire-and-forget's normal run budget, and align lock, wait, and run deadlines against current main semantics.

## Reproduction Assessment

Yes, at source level. Current main accepts `sessions_send.timeoutSeconds` and waits with that budget, but the nested target `agent` call does not receive a corresponding run timeout; I did not run a live contended-session scenario.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best implementation because it violates the gateway type contract, shortens fire-and-forget runs, and leaves the lock/wait/run budgets misaligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass numeric agent timeouts:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: The `agent` RPC schema requires `timeout` to be an integer and the handler only treats numeric values as overrides. Passing `String(targetTimeoutSeconds)` here, and in `runAgentStep`, makes real gateway calls reject or ignore the timeout even though the mocks pass.
  - confidence: 0.97
- **[P2] Keep fire-and-forget runs on the normal budget:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For `timeoutSeconds: 0`, current behavior is enqueue-and-return while the target run keeps the configured agent budget. Forcing `timeout: 30` makes legitimate background child work terminate after 30 seconds.
  - confidence: 0.91
- **[P2] Count lock waiting against the same deadline:** `src/agents/pi-embedded-runner/run/attempt.ts:1705`
  - body: Passing the full `params.timeoutMs` into lock acquisition lets a run spend the caller's whole budget waiting for the session lock before the embedded run abort timer starts later, so the total runtime can still exceed what the sender waited for.
  - confidence: 0.86
- **[P2] Match helper run budget to helper wait budget:** `src/agents/tools/agent-step.ts:42-52`
  - body: `runAgentStep` derives the target run timeout from the full `params.timeoutMs`, but the helper still waits with a 60-second cap. For larger values the target run can continue after the helper returns `undefined`.
  - confidence: 0.88
