## Summary

Keep open: the timeout propagation problem is still real on current main, but this PR is not correct enough to merge because it forwards string `agent.timeout` values rejected by the gateway contract and changes fire-and-forget/deadline semantics.

## What This Changes

The PR forwards `sessions_send` and A2A timeouts into nested `agent` run params, passes embedded-runner `timeoutMs` into session write-lock acquisition, and adds focused tests.

## Best Possible Solution

Revise or replace the PR with a deadline-aware timeout fix: forward numeric target-run timeouts only for waited sends, preserve fire-and-forget target budgets, align A2A wait and target budgets, and keep lock wait inside the same effective run deadline.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main omits target-run timeout propagation and does not derive lock wait from the caller's run deadline, while the PR's new string timeout values would be rejected by `validateAgentParams`.

## Solution Assessment

No. The direction is useful, but this implementation is not the best fix because it violates the numeric gateway contract and changes established fire-and-forget and deadline behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Send numeric sessions_send timeouts:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: The `agent` RPC schema declares `timeout` as an integer and the gateway rejects invalid params before dispatch. Passing `String(targetTimeoutSeconds)` makes real `sessions_send` calls fail with invalid agent params instead of starting the target run.
  - confidence: 0.94
- **[P1] Send numeric A2A helper timeouts:** `src/agents/tools/agent-step.ts:52`
  - body: This helper sends the new `timeout` field through the same validated `agent` RPC. Passing a string here means A2A helper runs that include the field are rejected before the nested run starts.
  - confidence: 0.93
- **[P1] Preserve fire-and-forget target budgets:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For `timeoutSeconds: 0`, the caller should return immediately while the target run keeps the normal agent budget. Forcing a 30-second target timeout cuts legitimate background work short and changes fire-and-forget behavior.
  - confidence: 0.89
- **[P2] Keep lock waits inside the run budget:** `src/agents/pi-embedded-runner/run/attempt.ts:1706`
  - body: Using the full `params.timeoutMs` for lock acquisition lets a contended run spend the whole caller budget waiting for the lock and then receive another full model runtime budget. Use a shared deadline or remaining budget instead.
  - confidence: 0.83
- **[P2] Align helper target and wait budgets:** `src/agents/tools/agent-step.ts:45`
  - body: For `timeoutMs` values above 60 seconds, the target run gets the full timeout while the helper still waits at most 60 seconds, leaving the child run active after `runAgentStep` has returned `undefined`.
  - confidence: 0.84
