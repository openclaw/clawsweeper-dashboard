## Summary

Keep open. The underlying timeout-propagation bug is plausible on current main, but this PR is not mergeable as-is because it sends an integer-validated gateway timeout as a string, shortens fire-and-forget child runs to 30 seconds, and still leaves run/wait/lock budgets misaligned.

## What This Changes

The PR forwards `sessions_send` and `runAgentStep` timeouts into nested `agent` RPCs, passes run timeout into session lock acquisition, and adds timeout propagation tests.

## Best Possible Solution

Rework the branch to propagate numeric target timeouts for waited `sessions_send` and helper paths, preserve normal fire-and-forget run budgets, and align lock acquisition with the effective run deadline on current main with focused regression tests and real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main accepts `sessions_send.timeoutSeconds` and waits with that budget, but the target `agent` RPC receives no corresponding run timeout; I did not execute a live contended-session repro in this read-only review.

## Solution Assessment

No. The direction is reasonable, but this implementation should not merge until it keeps the gateway timeout numeric, preserves fire-and-forget semantics, and makes lock/wait/run budgets line up on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass agent timeouts as numbers:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: The `agent` RPC schema requires `timeout` to be an integer and the gateway validates params before dispatch. Sending `String(targetTimeoutSeconds)` makes real `sessions_send` calls fail with `invalid agent params`; the same problem exists in `runAgentStep`.
  - confidence: 0.97
- **[P2] Preserve fire-and-forget run budgets:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For `timeoutSeconds: 0`, the existing behavior is to enqueue and return immediately while the target uses the normal agent run budget. Forcing `timeout: 30` kills legitimate background child work after 30 seconds.
  - confidence: 0.9
- **[P2] Count lock waiting against the run deadline:** `src/agents/pi-embedded-runner/run/attempt.ts:1705`
  - body: Passing the full `params.timeoutMs` into lock acquisition lets a target spend the caller's whole budget waiting for the session lock before the embedded run abort timer starts later, so the run can still exceed what the sender waited for.
  - confidence: 0.86
- **[P2] Align nested helper run and wait budgets:** `src/agents/tools/agent-step.ts:42`
  - body: `runAgentStep` derives the target run timeout from the full `params.timeoutMs`, but the helper still waits with `Math.min(params.timeoutMs, 60000)`. Values above 60 seconds leave the target run alive after the helper has returned `undefined`.
  - confidence: 0.88
