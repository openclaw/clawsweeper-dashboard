## Summary

Keep open: the underlying timeout propagation problem is plausible, but this PR branch introduces blocking runtime regressions and still needs contributor real-behavior proof before merge.

## What This Changes

The PR forwards sessions_send/runAgentStep timeouts into nested agent RPCs, passes embedded run timeout into session-lock acquisition, and adds focused timeout propagation tests.

## Best Possible Solution

Thread a nonzero caller wait budget into target agent runs as a numeric timeout, preserve fire-and-forget target run defaults, and align session-lock waiting with a documented end-to-end deadline or the existing write-lock config.

## Reproduction Assessment

Yes for the PR review blockers: source inspection shows the PR sends string timeouts where the gateway requires integers, and it changes timeoutSeconds: 0 into a 30-second target run budget. I did not execute a live contended-session run in this read-only review.

## Solution Assessment

No. The requested direction is plausible, but this implementation is not mergeable because it violates the gateway timeout contract and regresses fire-and-forget/background run semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Send a numeric sessions_send timeout:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: `AgentParamsSchema` requires `timeout` to be an integer and the agent handler only honors numeric values. Forwarding `String(targetTimeoutSeconds)` means real gateway dispatch rejects or ignores the override even though the mocked tests pass; keep this value numeric.
  - confidence: 0.96
- **[P1] Send a numeric agent-step timeout:** `src/agents/tools/agent-step.ts:52`
  - body: This helper goes through the same `agent` RPC contract, where `timeout` is an integer. Passing a string here breaks real A2A helper calls at gateway validation; keep the forwarded timeout numeric and update the tests accordingly.
  - confidence: 0.95
- **[P1] Preserve fire-and-forget run budgets:** `src/agents/tools/sessions-send-tool.ts:220`
  - body: For `timeoutSeconds: 0`, current behavior returns immediately while leaving the target run on the normal agent timeout/default. This change forces the target run to 30 seconds, so legitimate background work can be killed early; leave the target timeout unset for fire-and-forget sends.
  - confidence: 0.92
- **[P2] Match the target budget to the wait cap:** `src/agents/tools/agent-step.ts:45`
  - body: For `timeoutMs > 60_000`, `targetTimeoutSeconds` grants the child run the full timeout while `runAgentStep` still waits at most 60 seconds. The caller can return `undefined` while the child keeps running; either cap the target timeout to the wait budget or wait for the same budget.
  - confidence: 0.88
- **[P2] Count lock waits against the caller budget:** `src/agents/pi-embedded-runner/run/attempt.ts:1705`
  - body: The embedded run abort timer is scheduled after session lock acquisition, so using the full `params.timeoutMs` as the lock wait can spend the whole caller budget on the lock and then start another full model run. Use an end-to-end deadline or the existing lock acquire policy instead of doubling the effective timeout.
  - confidence: 0.82
