## Summary

Keep open: current main still forwards direct chat route fields into mediated run-mode subagent starts, while this PR narrowly gates those fields and includes focused regression assertions plus sufficient live proof. I found no blocking correctness or security finding in the diff.

## What This Changes

The PR changes run-mode subagent startup so child agent calls omit direct chat route fields unless direct initial child delivery is explicitly enabled, with regression assertions for mediated starts.

## Best Possible Solution

Land this narrow route-gating fix after rebase and current check rerun, while coordinating with the related announce-side delivery PRs.

## Reproduction Assessment

No independent live reproduction was run in this read-only review. The issue is source-reproducible on current main because run-mode child starts set `deliver:false` while still forwarding route fields, and the gateway consumes those fields as delivery-context hints; the contributor also supplied before/after live Telegram output.

## Solution Assessment

Yes. Gating route fields on `deliverInitialChildRunDirectly` is the narrowest maintainable fix because explicit direct thread-bound session starts keep their route context while ordinary run-mode completions stay mediated through the existing announce path.
