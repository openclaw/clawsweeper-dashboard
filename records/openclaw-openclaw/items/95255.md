## Summary

Keep this PR open for now: the dispatch-side normalization is plausible, but it is too narrow for the linked Slack thread-drop bug and the PR has no after-fix real Slack proof. Current `main` still has a sibling `sendMessageSlack` path that records participation and receipts from the requested `opts.threadTs`, while https://github.com/openclaw/openclaw/pull/95250 appears to cover that broader path with tests and live proof.

## What This Changes

This PR changes Slack monitor dispatch to record thread participation under the inbound root `thread_ts` for thread replies instead of the delivered reply/status thread fallback.

## Best Possible Solution

Normalize Slack thread participation and receipts around the canonical delivered root thread id for every Slack send/dispatch path, keep this dispatch normalization only if it still covers a separate path, then add a focused regression and redacted live Slack proof.

## Reproduction Assessment

Yes from source inspection and the linked report: the prepare path looks up participation by root `message.thread_ts`, while current send/dispatch recording can still persist a requested or delivered child timestamp. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

No. The PR is a plausible dispatch-side mitigation, but it is not the best complete fix because the sibling Slack send/receipt path still owns a participation write and needs canonical delivered-thread handling plus regression and live proof.
