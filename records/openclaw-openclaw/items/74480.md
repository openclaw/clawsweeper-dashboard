## Summary

Keep open. Current main still has the raw embedded recall timeout handoff, and this PR is a narrow, correct fix for that boundary, but the user-visible timeout behavior change still needs a required changelog entry before merge.

## What This Changes

The PR centralizes Active Memory's effective embedded recall timeout and applies `timeoutMs + setupGraceTimeoutMs` to the embedded runner, watchdog, hook registration, and unit assertions.

## Best Possible Solution

Land the timeout-budget alignment with the strengthened unit guard and a single-line changelog entry, while keeping broader stalled/aborted Active Memory recall behavior in the narrower follow-up threads.

## Reproduction Assessment

Yes. The Active Memory unit harness can reproduce the boundary by setting a tiny `timeoutMs`, enabling setup grace, delaying the mocked embedded runner past the raw timeout, and asserting the runner receives the extended budget.

## Solution Assessment

Yes, with the changelog fix. A shared effective-timeout helper is the narrowest maintainable solution because it keeps the embedded runner, watchdog, and hook timeout on the same budget without changing search, routing, or circuit-breaker behavior.
