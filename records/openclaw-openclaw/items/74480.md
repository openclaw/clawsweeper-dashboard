## Summary

Keep open. The PR targets a real Active Memory timeout boundary, but current code shows the runner consumes the passed timeout as the embedded attempt/model budget, so adding setup grace there may broaden model runtime rather than only preserving setup grace; the user-facing fix also still needs a changelog entry.

## What This Changes

The PR adds an Active Memory effective-timeout helper that applies `timeoutMs + setupGraceTimeoutMs` to the embedded runner handoff, watchdog, hook registration, and related unit assertions.

## Best Possible Solution

Keep setup grace explicit at the Active Memory wrapper boundary, preserve a separately defined embedded model/idle budget, and add a changelog entry once the intended timeout semantics are settled.

## Reproduction Assessment

Unclear for the real failure. The unit harness can assert the proposed timeout argument, but code inspection shows current main already gives the wrapper/hook a setup-grace budget while the embedded runner uses the passed timeout as its own model/idle deadline.

## Solution Assessment

No, not as written. Centralizing the arithmetic is tidy, but passing the grace-extended value into `runEmbeddedPiAgent` changes the runner's internal deadline; a safer fix would keep setup and model budgets separate or add a dedicated runner setup-budget seam.
