## Summary

Keep this PR open. Current main still has the Active Memory timeout-budget mismatch at the embedded-runner handoff, and the proposed source/test change is the narrow fix for that mismatch; the remaining actionable blocker is the missing changelog entry required for a user-facing fix.

## What This Changes

The PR centralizes Active Memory's embedded recall timeout budget and updates the embedded runner handoff, outer watchdog, hook timeout, and regression test to use `timeoutMs + setupGraceTimeoutMs`.

## Best Possible Solution

Land the narrow timeout-budget alignment with the strengthened unit guard, add the required changelog entry, then re-evaluate whether #73306 is fully resolved or whether the broader Active Memory timeout reports need separate follow-up.

Yes. A high-confidence reproduction path exists through the Active Memory unit harness: configure a tiny `timeoutMs`, set setup grace above the simulated setup delay, and assert that `runEmbeddedPiAgent` receives the extended budget rather than the raw timeout.

## Solution Assessment

Yes, with the changelog fix. A shared helper for the effective embedded recall timeout is the narrowest maintainable code path because it keeps the embedded runner, watchdog, and hook registration on the same budget without changing memory search, routing, or circuit-breaker behavior.
