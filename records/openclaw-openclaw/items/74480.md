## Summary

Keep open. Current main still passes the raw Active Memory `timeoutMs` into the embedded recall runner while the outer watchdog and hook budget already include setup grace, so the PR addresses a real remaining inconsistency. The patch looks functionally narrow and security-safe, but it is missing the required user-facing changelog entry before merge.

## What This Changes

The PR centralizes Active Memory's effective embedded recall timeout as `timeoutMs + setupGraceTimeoutMs`, applies it to the embedded runner, watchdog, and hook timeout, and updates the focused unit coverage.

## Best Possible Solution

Land the narrow timeout-budget alignment with a single-line changelog entry and targeted Active Memory tests, then keep separate broader Active Memory embedded-run timeout and abort reports open unless those symptoms are independently verified fixed.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists in the Active Memory unit harness: configure a small `timeoutMs`, simulate setup work longer than that raw timeout but within setup grace, and assert the embedded runner receives the extended budget.

## Solution Assessment

Yes, with the changelog fix. Reusing one helper for the effective embedded recall timeout is the narrowest maintainable solution because it keeps the embedded runner, watchdog, and hook timeout on the same budget without changing search behavior, provider routing, or circuit-breaker semantics.
