## Summary

Keep this PR open. Current main still passes the raw Active Memory timeout into the embedded runner while the surrounding watchdog and hook already include setup grace, and the PR is a narrow, well-scoped correction; the remaining blocker is the required user-facing changelog entry.

## What This Changes

The PR centralizes Active Memory's effective embedded recall timeout and applies `timeoutMs + setupGraceTimeoutMs` to the embedded runner handoff, watchdog, hook registration, and regression assertions.

## Best Possible Solution

Land the timeout-budget alignment with the strengthened unit guard and a single-line changelog entry, then handle broader embedded Active Memory timeout reports as separate follow-up if they still reproduce.

## Reproduction Assessment

Yes. The Active Memory unit harness can reproduce the boundary: configure a tiny `timeoutMs`, add setup grace, simulate setup taking longer than the raw timeout, and assert the embedded runner receives the extended budget.

## Solution Assessment

Yes, with the changelog fix. A shared effective-timeout helper is the narrowest maintainable solution because it keeps the embedded runner, watchdog, and hook timeout on the same budget without changing memory search, routing, or circuit-breaker semantics.
