## Summary

Keep open: current main and the latest shipped tag still lack a supported way to keep memory-core dreaming machine artifacts enabled while disabling Dream Diary prose. The existing discussion correctly treats this as distinct from adjacent Dream Diary language and reliability issues, and the remaining work is a new public config/product decision rather than conservative cleanup.

## What This Changes

Request a memory-core dreaming config option that skips Dream Diary narrative generation and `DREAMS.md` writes while preserving managed dreaming phases, reports, events, and promotion artifacts.

## Best Possible Solution

Add a memory-core-owned diary/output toggle with schema, docs, UI/status handling, and focused tests that skips narrative subagents and `DREAMS.md` writes while leaving phase reports, events, promotion, and current defaults intact.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request for a missing public config boundary. Source inspection gives high confidence that current main has no supported way to keep dreaming enabled while disabling Dream Diary prose.

## Solution Assessment

Unclear until maintainers choose the public contract. The memory-core plugin config boundary is the right place, but the exact naming and Dreams UI/doctor semantics need product judgment before implementation.
