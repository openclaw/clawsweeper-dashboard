## Summary

Keep open: current main still has no supported Dream Diary or human-readable prose opt-out, and adding one is a new public memory-core config/product decision rather than conservative cleanup. The related open prompt-customization work covers language/style, not machine-only dreaming.

## What This Changes

Request a memory-core dreaming option that keeps machine-readable phase artifacts and promotion outputs enabled while disabling Dream Diary narrative generation and `DREAMS.md` writes.

## Best Possible Solution

Add a memory-core-owned diary/narrative output toggle, coordinated with the Dream Diary prompt-config discussion, that skips narrative subagents and `DREAMS.md` writes while preserving phase reports, events, promotion, defaults, schema, docs, and tests.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken existing contract. Source inspection gives high confidence that current main has no supported way to keep dreaming enabled while disabling Dream Diary prose.

## Solution Assessment

Yes on the owner boundary: memory-core plugin config is the right place if maintainers accept the feature. The exact naming and Dreams UI/doctor semantics need product judgment before an implementation lane is safe.
