## Summary

Keep open: current main still lacks a supported Dream Diary or human-readable output opt-out, and adding one is a new public memory-core config/product decision rather than conservative cleanup. The related Dream Diary language and reliability issues are adjacent but do not supersede the requested machine-only dreaming mode.

## What This Changes

Add a memory-core dreaming option that skips Dream Diary narrative generation and DREAMS.md writes while preserving machine phase reports, events, and promotion artifacts.

## Best Possible Solution

Add a memory-core-owned diary or narrative output toggle with schema, docs, status/UI semantics, and focused tests that skips the narrative subagent and DREAMS.md writes while leaving phase reports, events, promotion, and current defaults intact.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection confirms current main has no supported way to keep dreaming enabled while disabling Dream Diary prose.

## Solution Assessment

Yes for the owner boundary: memory-core plugin config is the right place if maintainers accept the feature. The exact option name and Dreams UI semantics need maintainer product judgment before implementation.
