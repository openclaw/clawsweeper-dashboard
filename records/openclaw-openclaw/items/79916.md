## Summary

Keep open: current main still has no supported Dream Diary or human-readable-output opt-out, and the request adds a new public memory-core configuration boundary rather than fixing a broken existing contract. Adjacent Dream Diary prompt work would customize the prose, but it does not provide the requested machine-only dreaming mode.

## What This Changes

Add a memory-core dreaming option that keeps phase/report/promotion artifacts enabled while skipping Dream Diary narrative generation and `DREAMS.md` writes.

## Best Possible Solution

Add a memory-core-owned diary/output toggle with schema, docs, UI/status handling, and focused tests that skips narrative subagents and `DREAMS.md` writes while preserving phase reports, events, promotion, and current defaults.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection confirms current main has no supported way to keep dreaming enabled while disabling Dream Diary prose.

## Solution Assessment

Yes on the owner boundary: memory-core plugin config is the maintainable place if maintainers accept the feature. The exact naming, default, and Dreams UI semantics need maintainer product judgment before implementation.
