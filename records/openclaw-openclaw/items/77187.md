## Summary

Keep open: current main still lacks the proposed per-call resolver cache beyond the existing thinking-level row-context cache, so the sessions.list performance work is not obsolete, but this PR is not merge-ready because the added perf test is source-failing and one cost-resolution path still bypasses the new cache.

## What This Changes

The PR adds a per-call resolver cache for `sessions.list` row building and a synthetic perf regression test, while the branch also carries unrelated release and workflow changes from an older base.

## Best Possible Solution

Land a narrow gateway-only replacement or rebased PR that memoizes per-list resolver work, adds deterministic coverage with an explicit row limit, and excludes stale release/workflow churn.

## Reproduction Assessment

Yes. Source inspection shows the new perf test would fail because `listSessionsFromStoreAsync` applies the default 100-row limit when `opts.limit` is omitted, while the test expects 1000 rows.

## Solution Assessment

No as currently proposed. The cache direction is maintainable, but the PR should be narrowed onto current main, fix the test limit, pass the cache through the remaining cost probe, and drop unrelated release/CI changes.
