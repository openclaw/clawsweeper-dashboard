## Summary

Keep open. Current main still has the reported cache shape: a fixed-size plugin registry LRU, cache keys that include `onlyPluginIds`, exact active-registry compatibility checks, and synchronous cold-load fallback on a cache miss. The related open/closed reports are adjacent latency paths, not a merged fix for cron-driven scoped-key churn evicting the warm workspace registry.

## What This Changes

The issue requests a plugin-loader cache policy change to prevent cron or auxiliary scoped `onlyPluginIds` registry loads from evicting warm full-workspace plugin registries and causing synchronous event-loop stalls on cold reload.

## Best Possible Solution

Keep this open for a targeted plugin-loader performance fix that prevents scoped snapshot/provider/tool loads from evicting or forcing cold reloads of warm full workspace registries, while preserving scoped startup registry safeguards. The fix should add small-cache regression coverage for full versus scoped keys, multiple workspaces, `onlyPluginIds` variants, and `ensureRuntimePluginsLoaded` after cache churn.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by lowering the plugin registry cache cap, warming a full workspace key, inserting more than the cap of distinct `onlyPluginIds` scoped keys, then asserting the next full `ensureRuntimePluginsLoaded` path misses and reloads; this review did not execute it because the task was read-only.

## Solution Assessment

Unclear. The proposed options are plausible, but prewarming configured agents, off-thread loading, full-to-scoped reuse, and cache pinning have different startup, compatibility, and plugin visibility tradeoffs; a maintainer should choose the cache policy before implementation.
