## Summary

Keep open. Current main still has the cache shape and exact scoped-cache compatibility behavior described by the report, and the related open issues cover adjacent latency paths rather than this cron-driven LRU eviction pattern.

## What This Changes

Avoid synchronous mid-session plugin registry cold reloads caused by scoped plugin cache-key churn evicting warmed workspace registries during cron bursts.

## Best Possible Solution

Keep this open for a targeted plugin-loader performance fix that prevents scoped snapshot/provider/tool loads from evicting or forcing cold reloads of warm full workspace registries, while preserving the existing scoped startup registry safeguards. The fix should include regression coverage with a small cache cap, multiple workspace keys, scoped onlyPluginIds variants, and cron/embedded ensureRuntimePluginsLoaded behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction can be built by setting a small plugin registry cache cap, warming a full workspace key, creating many distinct onlyPluginIds keys, switching the active registry to another workspace, then asserting the next ensureRuntimePluginsLoaded path misses and reloads; I did not run it because this review is read-only.

## Solution Assessment

Unclear. The report’s proposed fixes are plausible, but prewarming configured agents changes startup cost, off-thread loading is larger architecture work, and full-to-scoped reuse needs care after the recent scoped registry compatibility fixes.
