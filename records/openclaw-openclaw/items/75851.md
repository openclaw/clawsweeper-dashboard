## Summary

Keep this issue open. Current main still has a single 128-entry plugin registry LRU whose key includes `onlyPluginIds`, provider/runtime paths still create cached scoped registry entries, and a cache miss falls back to synchronous plugin loading before the embedded runner records the `runtime-plugins` stage.

## What This Changes

The issue requests preventing cron-driven scoped plugin registry cache churn from evicting warm per-workspace runtime registries and causing long synchronous cold plugin reloads.

## Best Possible Solution

Add a generic plugin-loader cache policy that prevents scoped/snapshot/provider registry variants from evicting warm full-scope per-workspace runtime registries, with focused regression coverage around low-cap LRU churn and embedded runtime plugin loading.

## Reproduction Assessment

Yes at source/unit level: lower the plugin registry cache cap via the existing test hook, load a full-scope workspace registry, fill the cache with distinct scoped `onlyPluginIds` or workspace keys, then verify a later `ensureRuntimePluginsLoaded` for the original workspace does not re-register/cold-load. I did not run this because the review was required to be read-only.

## Solution Assessment

Unclear for the issue’s proposed Option A: prewarming configured agent workspaces is useful but does not protect entries from later eviction. A narrower maintainable fix is likely cache partitioning/pinning or a compatible full-scope active/superset registry fallback that preserves scoped-load correctness.
