## Summary

Keep open. The PR addresses real current-main helper dispatch gaps, but it should not merge as-is because the new cron CLI alias branch is not policy-equivalent to embedded cron execution and can broaden tool/message access.

## What This Changes

The PR routes queued followups, model probes, slug generation, and canonical-provider cron runs through CLI-backed execution, adding session/routing metadata and MCP message-send tracking tests.

## Best Possible Solution

Preserve CLI dispatch while making cron CLI alias runs enforce the same toolsAllow and message-tool policy as embedded runs, or keep those alias runs on embedded execution until CLI/MCP loopback supports that contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a cron agent turn whose canonical provider resolves to a CLI runtime and set a restrictive toolsAllow such as read, or a delivery mode that disables the message tool; the PR routes that run through runCliAgent without the embedded policy flags.

## Solution Assessment

No. Routing helper paths through runCliAgent is directionally right, but the current PR is not the best solution until cron CLI alias runs are policy-equivalent to embedded cron runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve cron tool policy on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:170-205`
  - body: This CLI branch only forwards the owner-only cron grant, while the embedded branch still receives toolsAllow plus the message-tool policy fields. A cron job configured with a restrictive tools list or with message delivery disabled can now enter the CLI alias path with a broader bundled MCP/message tool surface than configured. Add equivalent CLI-side enforcement or keep these cron runs on the embedded path until policy is preserved.
  - confidence: 0.9
