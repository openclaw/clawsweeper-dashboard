## Summary

Keep open. The PR targets real helper-path gaps still present on current main, but the latest head should not merge as-is because the new cron CLI alias branch is not policy-equivalent to the embedded cron path and can expose a broader MCP/message tool surface than the cron job configured.

## What This Changes

The PR routes queued followups, model status probes, LLM slug generation, and canonical-provider cron runs through CLI runtime dispatch with session binding, MCP loopback routing, and regression tests.

## Best Possible Solution

Add policy-equivalent CLI enforcement for cron runs, including toolsAllow and message-tool require/disable/force behavior, or keep those alias cron runs on the embedded path until runCliAgent and MCP loopback can preserve the same contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure an isolated cron job whose canonical provider resolves to a CLI runtime, then set toolsAllow to a restrictive list such as read or disable the message tool; the PR routes that run through runCliAgent without the embedded policy fields.

## Solution Assessment

No. Routing CLI-backed helper paths through runCliAgent is the right direction, but this PR is not the best implementation until cron CLI execution preserves the same per-run tool policy as the embedded runner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve cron tool policy on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:170-205`
  - body: This CLI branch only forwards the owner-only cron grant, while the embedded branch still receives toolsAllow plus the message-tool policy fields. A cron job configured with a restrictive tools list or with message delivery disabled can now enter the CLI alias path with a broader bundled MCP/message tool surface than configured. Add equivalent CLI-side enforcement or keep these cron runs on the embedded path until policy is preserved.
  - confidence: 0.9
