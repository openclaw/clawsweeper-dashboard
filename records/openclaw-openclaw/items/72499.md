## Summary

Keep open. The PR still targets real current-main helper-path gaps, but the latest head has a blocking cron CLI alias regression because it does not preserve the embedded cron tool-policy contract.

## What This Changes

The PR routes queued followups, model auth probes, LLM slug generation, and canonical-provider cron runs through CLI-backed execution with session/routing metadata, MCP message-send tracking, and regression tests.

## Best Possible Solution

Update the PR so CLI-backed helper paths preserve auth, session, routing, cancellation, and cron tool-policy contracts; for cron, add CLI/MCP loopback enforcement for toolsAllow and message-tool policy or keep those alias runs on the embedded path until enforcement exists.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure an isolated cron job whose canonical provider resolves to a CLI runtime and set a restrictive toolsAllow such as read or a delivery mode that disables the message tool; the PR routes that run through runCliAgent without the embedded policy flags.

## Solution Assessment

No. Routing these helper paths through runCliAgent is directionally correct, but the current PR is not the best way to solve it until the cron CLI path is policy-equivalent to the embedded path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve cron tool policy on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:170-205`
  - body: This CLI branch only forwards the owner-only cron grant, while the embedded branch still receives toolsAllow plus the message-tool policy fields. A cron job configured with a restrictive --tools list or with message delivery disabled can now enter the CLI alias path with a broader bundled MCP/message tool surface than configured. Add equivalent CLI-side enforcement or keep these cron runs on the embedded path until policy is preserved.
  - confidence: 0.9
