## What This Changes

The PR routes queued followups, model status probes, LLM slug generation, and canonical-provider cron helper runs through CLI-backed execution paths, with added session/MCP routing and message-send tracking.

## Best Possible Solution

Land a rebased or split change that routes queued followups, model probes, and slug generation through the same CLI dispatch pattern while preserving or fail-closing cron tool and message-policy restrictions.

## Reproduction Assessment

Yes, at source level. Configure a cron agent whose canonical provider resolves to a CLI runtime and set restrictive toolsAllow or message-tool policy; PR head reaches runCliAgent without the policy fields that the embedded cron branch still receives.

## Solution Assessment

No, not as-is. Routing helper paths through runCliAgent is the right direction, but this branch should not merge until it is rebased across current-main cron work, preserves or fail-closes tool policy, and includes real behavior proof.
