## Summary

Keep open. The PR still targets a real helper-dispatch bug that current main has for followups, model probes, and slug generation, but the branch still needs rebase and policy work before merge, and it has no after-fix real behavior proof.

## What This Changes

The PR routes queued followups, model status probes, LLM slug generation, and canonical-provider cron helper runs through CLI-backed execution, with added session/MCP routing and message-send tracking tests.

## Best Possible Solution

Rebase the useful followup, probe, and slug dispatch fixes onto current main, split or update the overlapping cron changes, and make any CLI cron path preserve or fail closed on tool and message-policy restrictions.

## Reproduction Assessment

Yes, at source level. Configure a cron agent whose canonical provider resolves to a CLI runtime and set restrictive toolsAllow or message-tool policy; PR head enters runCliAgent without the policy fields that the embedded cron path still receives.

## Solution Assessment

No. Routing helper paths through runCliAgent is the right direction, but this branch is not the best merge shape until it is rebased across current-main cron changes, preserves cron tool/message policy, and includes real behavior proof.
