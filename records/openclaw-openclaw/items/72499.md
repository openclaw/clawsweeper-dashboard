## Summary

Keep open. Current main still has the helper-dispatch gaps this branch targets, so the PR is not obsolete, but the latest head has a blocking security-sensitive cron policy regression: canonical-provider cron runs that resolve to CLI-backed runtimes do not receive the same tool restrictions as the embedded cron path.

## What This Changes

The branch routes queued followups, model status probes, slug generation, and canonical-provider cron runs through CLI-backed execution, adding CLI session/routing/MCP send-tracking plumbing and regression tests.

## Best Possible Solution

Update the PR so CLI-backed helper paths preserve the same auth, session, routing, cancellation, and cron tool-policy contracts as the embedded paths. For cron specifically, canonical-provider CLI alias runs should either enforce toolsAllow and message-tool policy in the CLI/MCP loopback path or stay on the embedded path until that enforcement exists.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure an isolated cron job whose canonical provider resolves to a CLI runtime and set a restrictive toolsAllow such as read; the PR routes it through runCliAgent while the CLI params omit the allowlist and message-tool policy that the embedded path receives.

## Solution Assessment

No. Routing the helper paths through runCliAgent is the right direction, but this PR is not yet the best fix because the new cron CLI alias path is not policy-equivalent to the existing embedded path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve cron tool policy on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:203-205`
  - body: This CLI branch only forwards the owner-only grant, while the embedded branch still receives `toolsAllow` plus the message-tool policy fields. A cron job configured with `--tools read` or with the message tool disabled can now enter the CLI alias path and get the default bundled MCP/message tool surface. Add equivalent CLI-side enforcement or keep these cron runs on the embedded path until the policy is preserved.
  - confidence: 0.86
