## Summary

Keep open. Current main still has the helper-path CLI dispatch gaps, so this PR is not obsolete, but PR head still leaves the new cron CLI alias path less restrictive than the existing embedded path for cron tool and message-tool policy.

## What This Changes

The PR routes queued followups, model probes, slug generation, and canonical-provider cron runs through CLI-backed execution with session, auth, routing, MCP-send tracking, and regression-test updates.

## Best Possible Solution

Make the CLI-backed helper paths policy-equivalent to the embedded paths; for cron, either enforce runtime `toolsAllow` and message-tool flags through the CLI/MCP loopback or leave those alias runs on the embedded path until that enforcement exists.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an isolated cron job whose canonical provider resolves to a CLI backend while `toolsAllow` excludes `message` or delivery mode disables the message tool; PR head enters the CLI branch without carrying those policy fields.

## Solution Assessment

No. Routing helper paths through `runCliAgent` is the right direction, but this patch is not the best solution until cron CLI execution preserves the same tool-policy contract as embedded cron execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve cron tool policy on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:170-205`
  - body: This new CLI branch only forwards the cron owner-only grant, while the embedded branch still applies `toolsAllow` plus `requireExplicitMessageTarget`, `disableMessageTool`, and `forceMessageTool`. An isolated cron job with a restrictive allowlist or webhook delivery can therefore reach a CLI alias run with broader bundled MCP/message tools than configured. Add equivalent CLI-side enforcement or keep these runs on the embedded path until policy is preserved.
  - confidence: 0.9

## Security Review

Status: needs_attention
