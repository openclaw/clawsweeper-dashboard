## Summary

Keep open. Current main still has source-level gaps for queued followups, model probes, and slug generation, but this branch is not merge-ready because its cron CLI path can bypass configured tool/message policy and the external PR still lacks real behavior proof.

## What This Changes

The branch routes queued followups, model status probes, LLM slug generation, and canonical-provider cron helper runs through `runCliAgent()`, adding CLI session/MCP routing context and tests.

## Best Possible Solution

Rebase or split the branch so queued followup, probe, and slug dispatch fixes land against current main, while cron CLI handling preserves or fails closed on runtime tool/message policy and includes real behavior proof.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider with a CLI runtime and trigger queued followup execution, `openclaw models status --probe`, or LLM slug generation; current main still reaches `runEmbeddedPiAgent` directly in those helper paths, and PR head's cron policy gap is visible from the new `runCliAgent` branch.

## Solution Assessment

No. Routing helper paths through `runCliAgent()` is the right direction, but this branch is not the best merge shape until it is rebased across current-main cron work, preserves tool/message policy parity, and includes real behavior proof.
