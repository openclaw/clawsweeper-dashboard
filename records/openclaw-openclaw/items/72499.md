## Summary

Keep open. The PR is a useful bug-fix direction for CLI-backed helper dispatch, but it is not merge-ready because the PR cron path can bypass existing tool/message policy restrictions, the branch overlaps newer current-main cron routing work, and the external PR still lacks real after-fix CLI/MCP behavior proof.

## What This Changes

The PR routes queued followups, model status probes, LLM slug generation, and canonical-provider cron helper runs through CLI-backed execution with added session/MCP routing and message-send tracking.

## Best Possible Solution

Land a rebased or split change that routes queued followups, model probes, and slug generation through the same CLI dispatch pattern while preserving or fail-closing cron tool and message-policy restrictions.

## Reproduction Assessment

Yes, at source level. Configure a cron agent whose canonical provider resolves to a CLI runtime and set restrictive toolsAllow or message-tool policy; PR head enters runCliAgent without the policy fields that the embedded branch still receives.

## Solution Assessment

No. Routing helper paths through runCliAgent is the right direction, but this branch is not the best merge shape until it is rebased across current-main cron work, preserves cron tool/message policy, and includes real behavior proof.
