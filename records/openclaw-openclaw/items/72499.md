## Summary

Keep open. The PR still addresses real CLI-backed helper gaps on current main, but it is not merge-ready because its cron CLI path is not policy-equivalent to the embedded cron path and it overlaps newer cron runtime work that already landed.

## What This Changes

The PR routes queued followups, model status probes, LLM slug generation, and canonical-provider cron runs through CLI-backed execution, with CLI session/MCP routing and message-send tracking coverage.

## Best Possible Solution

Rebase onto current main, retain the queued followup/probe/slug CLI dispatch fixes, and either drop superseded cron routing or make the CLI cron path preserve the same tool and message policy as the embedded path.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure a cron agent turn whose canonical provider resolves to a CLI runtime and set toolsAllow restrictively or disable message delivery; the PR head enters runCliAgent without the embedded path's policy fields.

## Solution Assessment

No. Routing CLI-backed helper paths through runCliAgent is the right direction, but this PR is not the best implementation until it is rebased over current main and the cron CLI path is policy-equivalent or removed as superseded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve cron tool policy on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:170-205`
  - body: This CLI branch forwards only the owner-only cron grant, while the embedded branch still receives toolsAllow plus requireExplicitMessageTarget, disableMessageTool, and forceMessageTool. A cron job configured with a restrictive toolsAllow list or disabled message delivery can enter the CLI alias path with a broader bundled MCP/message tool surface than configured. Add equivalent CLI-side enforcement or keep these cron runs on the embedded path until policy is preserved.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR can weaken per-job cron tool restrictions for canonical-provider cron runs that resolve to CLI-backed runtimes.

Concerns:

- **[medium] Cron CLI alias path can bypass tool restrictions:** `src/cron/isolated-agent/run-executor.ts:170`
  - body: The new CLI cron branch does not carry the runtime toolsAllow allowlist or message-tool disable/require flags that the embedded branch enforces. Because bundled MCP tools are exposed through the CLI loopback, restrictive cron jobs can receive a broader tool surface than configured.
  - confidence: 0.88
