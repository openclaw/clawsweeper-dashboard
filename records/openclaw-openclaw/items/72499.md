## Summary

Keep open. The PR still carries useful work for queued followups, model probes, and slug generation, but it is not mergeable as-is because the cron CLI branch can bypass tool/message policy and the external contributor has not provided real behavior proof.

## What This Changes

The branch routes queued followups, model status probes, LLM slug generation, and canonical-provider cron helper runs through CLI-backed execution with added session, MCP routing, and message-send tracking.

## Best Possible Solution

Land a rebased or split change that routes followups, probes, and slug generation through the main CLI dispatch pattern while preserving cron tool/message policy or leaving restricted cron runs on the embedded path.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider with a CLI runtime and restrictive cron tool/message policy; the PR head reaches runCliAgent without policy fields that the embedded branch still receives, and current main still has direct embedded helper calls for followups, probes, and slug generation.

## Solution Assessment

No, not as-is. Routing helper paths through runCliAgent is the right direction, but the branch needs rebase/split handling, policy parity for cron CLI runs, and contributor real behavior proof before merge.
