## Summary

Keep open. Current main still has the helper-path gaps this PR targets, but the latest PR head is not merge-ready because its new canonical-provider cron CLI path is not policy-equivalent to the embedded path: it forwards only the owner-only cron grant and omits the runtime `toolsAllow` and message-tool policy contract.

## What This Changes

The PR routes queued followups, model auth probes, LLM slug generation, and canonical-provider cron runs through CLI-backed execution, adding CLI session/routing metadata, MCP message-send tracking, and regression tests.

## Best Possible Solution

Update the PR so CLI-backed helper paths preserve the same auth, session, routing, cancellation, and cron tool-policy contracts as the embedded paths. For cron, either add CLI/MCP loopback enforcement for `toolsAllow` and message-tool policy or leave canonical-provider cron alias runs on the embedded path until that enforcement exists.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure an isolated cron job whose canonical provider resolves to a CLI runtime and set a restrictive `--tools` policy such as `read`; the PR routes that run through `runCliAgent` while omitting the allowlist and message-tool restrictions that the embedded branch receives.

## Solution Assessment

No. Routing these helper paths through `runCliAgent` is directionally correct, but the current PR is not the best solution until the new cron CLI path is policy-equivalent to the existing embedded path.
