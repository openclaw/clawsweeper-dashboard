## What This Changes

The PR adds a helper and cleanup branch in `src/agents/embedded-agent-runner/run/attempt.ts` to suppress `EmbeddedAttemptSessionTakeoverError` when the attempt abort signal is aborted and timeout flags are not set.

## Best Possible Solution

Thread explicit user-stop provenance into the embedded attempt and suppress only that case, while keeping timeout, restart, RPC/session abort, post-compaction, and other internal abort cleanup takeovers fatal with focused regression coverage.

## Reproduction Assessment

Partially: source inspection shows current main rejects cleanup takeover errors into the attempt error path, but I did not reproduce the Stop-button UI flow live. The blocking review finding is source-reproducible from the abort reason plumbing.

## Solution Assessment

No: suppressing at cleanup is a plausible layer, but the proposed `abortSignal.aborted && !timeout` gate is too broad for OpenClaw's current abort contract. The safer fix is explicit user-stop provenance plus tests for the sibling abort causes.

Full review comments:

- **[P1] Gate suppression on explicit user abort provenance:** `src/agents/embedded-agent-runner/run/attempt.ts:5097-5101`
  - body: This treats every aborted non-timeout attempt as user initiated, but current abort routing does not preserve that distinction: `chat.abort`/`sessions.abort` use `rpc`, restart uses `restart`, and `abortChatRunById` only turns `timeout` into an AbortSignal reason. Those paths reach `runEmbeddedAttempt` as `abortSignal.aborted === true` with no timeout flag, so a cleanup `EmbeddedAttemptSessionTakeoverError` during restart/internal/RPC abort would be logged and suppressed instead of remaining fatal. Please thread an explicit user-stop reason/provenance into this predicate and keep the other abort causes on the existing rejection path.
