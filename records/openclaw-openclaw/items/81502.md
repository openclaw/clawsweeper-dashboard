## Summary

Keep open: current main still has the reported `readlinkSync`-before-type-check path, and this PR is a plausible narrow fix, but it is draft and lacks after-fix real behavior proof from an affected setup.

## What This Changes

The branch guards plugin skill publication with `lstatSync`, skips regular directories, only calls `readlinkSync` for symlinks, and adds a regression test for a regular-directory collision.

## Best Possible Solution

Land the narrow guarded-readlink fix with regression coverage once the draft state and real behavior proof gate are resolved, while preserving generated symlink and junction replacement semantics.

## Reproduction Assessment

Yes, at source level: create a regular directory in the managed plugin skills directory with the same basename as a publishable plugin skill, then refresh or list skills; current main reaches `fs.readlinkSync(linkPath)` before checking the entry type. I did not run it because this review was read-only.

## Solution Assessment

Yes, the proposed `lstatSync` guard is the narrowest source-level fix: it still replaces existing symlinks and avoids treating regular directories as generated symlinks. The merge path still needs real behavior proof from an affected setup.
