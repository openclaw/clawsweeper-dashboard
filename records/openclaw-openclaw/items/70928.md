## Summary

Keep open: current main still routes WebChat and Gateway-backed TUI replies through `chat.send` paths where final delivery and Gateway-injected transcript writes bypass the existing outbound/write hook seams.

## What This Changes

Ensure WebChat and Gateway-backed openclaw-tui outbound replies invoke existing message lifecycle and transcript write hooks.

## Best Possible Solution

Add a focused Gateway chat lifecycle bridge so WebChat and Gateway-backed TUI final delivery and Gateway-owned assistant transcript appends invoke the existing hooks exactly once with session/run context, while leaving the broader cross-channel observer redesign to the umbrella issue.

Yes, at source level. WebChat docs and the TUI client both enter `chat.send`, and current main still shows direct final broadcast and injected transcript append paths that do not call the relevant hook runners.
