## Summary

Keep open. Current main and the latest release still route WebChat and Gateway-backed TUI through `chat.send`, where final chat delivery and Gateway-owned assistant transcript appends bypass the existing outbound and write hook seams.

## What This Changes

The issue asks for `message_sent`, `message_sending`, `agent_end`, and `before_message_write` hooks to fire for WebChat and Gateway-backed TUI assistant outbound paths.

## Best Possible Solution

Add a focused Gateway chat lifecycle seam so WebChat and Gateway-backed TUI final delivery and Gateway-owned assistant transcript appends invoke the existing hooks exactly once with session/run context, while leaving broad outbound observer design to https://github.com/openclaw/openclaw/issues/50126.

## Reproduction Assessment

Yes, at source level. WebChat docs and `GatewayChatClient.sendChat()` both enter `chat.send`, and current main still shows that path bypassing hook-aware pre-delivery dispatch, sent-hook emission, and `before_message_write` for Gateway-owned assistant transcript appends; no live run was performed in this read-only review.

## Solution Assessment

Yes. A focused Gateway chat fix is the narrowest maintainable solution for this issue; a larger exactly-once outbound observer should stay with https://github.com/openclaw/openclaw/issues/50126 unless maintainers choose that architecture first.
