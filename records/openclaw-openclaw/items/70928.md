## Summary

Keep open. Current main still routes WebChat and Gateway-backed TUI replies through `chat.send`, and that path bypasses the existing hook-aware reply wrapper, final sent-hook emission, and `before_message_write` for Gateway-owned transcript appends.

## What This Changes

The issue reports that webchat and openclaw-tui outbound assistant replies do not fire documented message lifecycle hooks such as `message_sent`, `message_sending`, `agent_end`, and `before_message_write`.

## Best Possible Solution

Add a narrow Gateway chat lifecycle hook seam for `chat.send` final delivery and Gateway-owned transcript appends, while leaving the broader cross-channel observer design to https://github.com/openclaw/openclaw/issues/50126.

## Reproduction Assessment

Yes, as a source-level reproduction. WebChat docs and the TUI client both enter `chat.send`, and current main still shows that path bypassing the hook-aware dispatch wrapper, sent-hook emission, and `before_message_write` before Gateway assistant transcript appends; no live run was performed in this read-only review.

## Solution Assessment

Yes. A focused Gateway chat fix is the narrowest maintainable solution for this issue, using existing hook runner and reply-dispatch seams without taking on the full observer architecture tracked by https://github.com/openclaw/openclaw/issues/50126.
