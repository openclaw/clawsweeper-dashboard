## Summary

Keep open. Current main still routes WebChat and openclaw-tui through `chat.send`, where final chat delivery and Gateway-injected transcript appends bypass the existing outbound lifecycle and write-hook seams; the older hook PR was closed unmerged with the remaining WebChat/TUI gap intentionally left to narrower issues.

## What This Changes

The issue requests that webchat and openclaw-tui outbound assistant replies fire the existing message lifecycle hooks, including `message_sending`, `message_sent`, `agent_end`, and `before_message_write`.

## Best Possible Solution

Add a focused Gateway chat lifecycle seam so WebChat and Gateway-backed TUI final delivery plus Gateway-owned assistant transcript appends invoke the existing hooks exactly once with session/run context, leaving the broader cross-channel observer design to https://github.com/openclaw/openclaw/issues/50126.

## Reproduction Assessment

Yes, at source level. WebChat docs and the TUI client both enter `chat.send`, and current main still shows that path building, broadcasting, and appending final assistant output without the relevant hook calls; no live runtime run was performed in this read-only review.

## Solution Assessment

Yes. A focused Gateway chat fix is the narrowest maintainable path for this issue; a larger exactly-once outbound observer belongs to https://github.com/openclaw/openclaw/issues/50126 if maintainers choose that architecture.
