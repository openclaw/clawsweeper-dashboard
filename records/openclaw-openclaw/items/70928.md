## Summary

Keep open: current main and the latest release still show WebChat and Gateway-backed TUI reply paths that bypass the documented outbound and transcript-write hook seams, while the related lifecycle PR was closed unmerged and the umbrella issue does not replace this focused WebChat/TUI report.

## What This Changes

The issue asks for documented outbound lifecycle and transcript-write hooks to fire for assistant replies delivered through WebChat and openclaw-tui.

## Best Possible Solution

Add a focused Gateway chat lifecycle bridge so WebChat and Gateway-backed TUI final delivery plus Gateway-owned assistant transcript appends invoke the existing hooks exactly once with session/run context, while leaving the broader outbound observer design to https://github.com/openclaw/openclaw/issues/50126.

## Reproduction Assessment

Yes, at source level. WebChat and Gateway-backed TUI enter `chat.send`, and current main still reaches direct final broadcasts and Gateway-injected transcript appends without the relevant hook runners.

## Solution Assessment

Unclear until maintainers decide exact streaming and exactly-once semantics. The narrowest maintainable direction is a Gateway chat hook bridge, not a broad cross-channel rewrite inside this issue.
