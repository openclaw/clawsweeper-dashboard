## Summary

Keep open: this PR targets a real P1 fsGroup chmod regression, but the reviewed head still has a permission-boundary bug for private 0600 files and the posted proof does not cover that path.

## What This Changes

Adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state permission setup through it for selected chmod failures.

## Best Possible Solution

Keep the shared helper, but make its fallback target-aware: tolerate fsGroup directory read/execute bits like `2775`, reject unexpected `other` bits for private `0600` files, and refresh tests plus real K8s proof after the fix.

## Reproduction Assessment

Yes, at source level: current main still calls `chmodSync` on the approvals directory and rethrows non-Windows `EPERM`, and the linked issue plus PR proof give a concrete Kubernetes fsGroup `2775` path. I did not run a live Kubernetes cluster in this read-only review.

## Solution Assessment

No for the current head: the shared helper is the right direction, but the fallback is too broad for private `0600` files. The safer solution is directory-specific fsGroup tolerance plus stricter private-file checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep private file fallbacks from accepting other-read bits:** `src/infra/chmod.ts:29-31`
  - body: After a chmod failure, this helper accepts any accessible path unless the `other-write` bit is set. The new call sites also pass `0o600` for SQLite sidecar/state files, so a root-owned file that remains `0644` or `0664` will be accepted even though the existing permission contract expects owner-only files; make the fallback mode-aware so directories can tolerate fsGroup `r-x` while private files reject extra `other` bits.
  - confidence: 0.92
