## Summary

Keep open: the fsGroup compatibility direction addresses a valid regression, and the latest Docker proof demonstrates the chmod EPERM condition, but the current helper still weakens private state permission hardening because `accessSync` does not verify the actual mode/ownership boundary before swallowing chmod failures.

## What This Changes

The PR adds a `tryChmodSync` helper and switches exec approvals, task/flow registry, and plugin-state permission hardening to tolerate expected chmod failures after access checks.

## Best Possible Solution

Land the fsGroup tolerance with a stat-based post-failure policy that permits the intended fsGroup-compatible owner/group access while rejecting unsafe world-accessible private state paths.

## Reproduction Assessment

Yes. The linked issue and PR proof show the fsGroup chmod EPERM shape, and current main rethrows the same non-Windows chmod failure in exec approvals while related registries use the same strict pattern.

## Solution Assessment

No for the current PR head. Tolerating chmod failures on fsGroup-mounted state is the right direction, but `accessSync` alone is not the safest maintainable replacement for private permission hardening.
