## Summary

Keep open: this targets a real P1 chmod regression, but the current PR head still uses an access-only fallback for private state paths and the latest proof does not exercise the changed OpenClaw exec or registry initialization paths.

## What This Changes

The PR adds `tryChmodSync` and switches exec approvals, task/task-flow SQLite registries, and plugin-state permission hardening from strict chmod calls to that tolerant helper with mocked tests.

## Best Possible Solution

Keep the fsGroup compatibility direction, but require stat-based owner/group/mode validation, focused regression tests for acceptable and unsafe modes, and redacted OpenClaw runtime proof before merge.

## Reproduction Assessment

Yes. The linked issue logs and current main source path show a non-Windows `chmodSync` rethrow before exec approval persistence can continue; I did not run a live Kubernetes pod, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No for the current PR head. Tolerating managed-volume chmod failures is the right direction, but `accessSync` alone is not a maintainable replacement for private permission hardening.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate permissions before tolerating chmod failures:** `src/infra/chmod.ts:18`
