## Summary

Keep open: the PR targets a real Kubernetes fsGroup regression, but the live head still contains the access-only chmod fallback despite the author’s follow-up comment, so it needs permission-policy validation and real deployment proof before merge.

## What This Changes

This PR adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state SQLite permission hardening through it to tolerate chmod `EPERM`/`EACCES`/`EROFS` on accessible paths.

## Best Possible Solution

Keep the fsGroup compatibility fix, but only tolerate chmod failure after a stat-based permission policy approves the path for these state surfaces, then land with redacted Kubernetes/PVC or equivalent proof.

## Reproduction Assessment

Yes. Current main still rethrows non-Windows `chmodSync` errors in exec approvals and strict SQLite state permission paths, and the linked issue provides concrete Kubernetes fsGroup logs; I did not run a live PVC reproduction in this read-only review.

## Solution Assessment

No. Tolerating managed-volume chmod failure is the right direction, but `accessSync` alone is not a sufficient replacement for permission hardening on private state paths.

## Review Findings

Full review comments:

- **[P1] Validate permissions before swallowing chmod failures:** `src/infra/chmod.ts:18`
