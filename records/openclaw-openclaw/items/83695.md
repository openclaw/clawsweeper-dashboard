## Summary

Keep open: this is a useful regression fix direction, but the current patch relaxes permission hardening with only `accessSync` verification and still lacks after-fix proof from a Kubernetes fsGroup PVC or equivalent non-owner filesystem setup.

## What This Changes

The PR adds `tryChmodSync` and applies it to exec approvals plus task, flow, and plugin-state SQLite permission hardening paths so selected chmod failures are tolerated when access checks pass.

## Best Possible Solution

Land a scoped chmod tolerance that re-stats tolerated paths and rejects unsafe owner/group/mode states while preserving exec approval hardening, then attach redacted fsGroup/PVC proof.

## Reproduction Assessment

Yes, source inspection plus the linked report give a high-confidence path: current main and `v2026.5.18` rethrow non-Windows chmod EPERM in exec approvals while the repo Kubernetes manifest uses the reported fsGroup/non-root PVC shape. I did not run a live Kubernetes pod in this read-only review.

## Solution Assessment

No, not yet. Centralizing chmod tolerance is reasonable, but access-only verification is too broad for private state and exec approval hardening without stat-based scope or explicit maintainer security acceptance.
