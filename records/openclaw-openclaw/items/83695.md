## Summary

Keep this PR open: the latest head fixes the earlier access-only security gap, but it still appears not to handle the linked Kubernetes fsGroup regression because the reported chmod target is the state directory with mode `2775`, which the new `other`-bit check rejects.

## What This Changes

The PR adds `tryChmodSync` and routes exec approvals, task/task-flow registry, and plugin-state permission hardening through it so expected chmod failures on fsGroup PVCs can be tolerated after access and mode checks.

## Best Possible Solution

Update the helper, tests, and proof around the exact fsGroup state-root directory case, with maintainer-approved policy for non-writable `other` bits before claiming the linked regression is fixed.

## Reproduction Assessment

Yes, at source level. Current main rethrows the non-Windows chmod failure in `ensureDir`, and the linked issue gives the concrete fsGroup ownership/mode that reaches that path; I did not run a live Kubernetes deployment in this read-only review.

## Solution Assessment

No for the current PR head. The shared helper is the right shape, but the mode check must be aligned with the actual `2775` directory case or the PR remains a partial fix with a security-policy gap.
