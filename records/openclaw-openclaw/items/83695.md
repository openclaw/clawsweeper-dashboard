## Label Justifications

- `P1`: The PR targets a user-facing regression that can break every exec-dependent agent workflow on Kubernetes fsGroup PVC deployments.
- `merge-risk: 🚨 compatibility`: Some fsGroup-mounted existing state files may remain too broadly readable for the new private-file guard and still fail closed after the fix.
- `merge-risk: 🚨 security-boundary`: The patch deliberately softens chmod hardening for state directories and files when access and mode checks indicate the path is usable.

## Summary

This PR is a focused fix for a real K8s fsGroup chmod regression and the current patch has no blocking code-review finding, but it should stay open for maintainer acceptance of the permission-boundary tradeoff before merge.

## What This Changes

The PR adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state chmod hardening through access-verified EPERM/EACCES/EROFS tolerance.

## Best Possible Solution

Merge the helper only after maintainers accept the fsGroup permission boundary, with the strict private-file guard preserved and operator guidance for pre-existing files that remain too broadly readable.

## Reproduction Assessment

Yes. Source inspection shows current main rethrows `chmodSync(dir, 0o700)` failures on non-Windows, and the linked issue plus PR terminal proof give a concrete K8s/Docker fsGroup path that triggers EPERM.

## Solution Assessment

Yes, with maintainer acceptance of the boundary. A shared access-verified chmod helper is narrower and safer than making each call best-effort, and the current patch preserves fail-closed checks for inaccessible paths, world-writable directories, and private files with excessive other bits.
