## Summary

Keep open for maintainer review: this is a focused P1 regression fix with sufficient terminal proof and no blocking code findings, but it intentionally relaxes chmod hardening for fsGroup-owned state paths and needs explicit acceptance of that security boundary before merge.

## What This Changes

The PR adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state SQLite permission hardening through it with focused chmod fallback tests.

## Best Possible Solution

Land the shared helper once maintainers accept the fsGroup permission boundary, preserving fail-closed checks for inaccessible paths and unsafe private-file modes and adding operator guidance only if maintainers want cleanup steps documented.

## Reproduction Assessment

Yes. Current main source rethrows non-Windows chmod failures in exec approvals and uses raw chmod in the store paths, while the PR discussion gives concrete Kubernetes-style and Docker fsGroup terminal proof for the EPERM behavior.

## Solution Assessment

Yes, pending maintainer acceptance of the boundary. A shared access-verified helper is narrower than scattered catches, and the tests preserve fail-closed behavior for inaccessible paths, other-writable directories, and private files with unsafe other bits.
