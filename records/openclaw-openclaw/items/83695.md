- `P1`: The linked regression blocks exec and state initialization workflows on real Kubernetes fsGroup deployments.

## Summary

Keep this PR open for maintainer/security review: the patch is focused and has strong fsGroup proof, but it intentionally changes sensitive state paths from unconditional chmod failure to access- and mode-verified continuation.

## What This Changes

The PR adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state SQLite permission hardening through it, with focused tests.

## Best Possible Solution

Land the shared helper after security/compatibility approval and exact-head validation, preserving rejection for inaccessible paths, other-write directories, and private files with unsafe other bits.

## Reproduction Assessment

Yes. Current main has raw non-Windows `chmodSync` calls on the affected state paths, and the PR body/comments include Kubernetes and Docker fsGroup terminal output showing EPERM before the patch and success after it; I did not rerun the live scenario in this read-only review.

## Solution Assessment

Mostly yes. A shared helper is the maintainable shape for repeated chmod hardening, and the tests cover the key permission boundary; the remaining question is maintainer/security approval of the exact compatibility policy.
