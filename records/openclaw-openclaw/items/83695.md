## Label Justifications

- `P1`: The linked regression blocks exec and state initialization workflows for Kubernetes fsGroup-mounted PVC deployments.
- `merge-risk: 🚨 security-boundary`: The PR changes how private exec approval and state-store permission hardening behaves when chmod fails on non-owned managed volumes.

## Summary

Keep open for maintainer review: the latest PR head addresses the prior access-only security finding by rejecting world-accessible paths and includes Docker terminal proof of the chmod EPERM kernel behavior, but it still intentionally changes private state hardening from strict owner-only chmod to fsGroup-compatible group access when chmod cannot be applied.

## What This Changes

The PR adds a shared `tryChmodSync` helper, applies it to exec approvals, task registry, task-flow registry, and plugin-state permission hardening, and adds focused chmod fallback tests.

## Best Possible Solution

Land a security-reviewed fsGroup-compatible chmod helper that preserves symlink and world-access protections while allowing expected managed-volume owner/group access on affected state paths.

## Reproduction Assessment

Yes, source inspection plus the linked reporter logs provide a high-confidence path: current main rethrows non-Windows chmod EPERM in exec approvals and related registries use the same strict pattern. I did not run a live Kubernetes PVC reproduction in this read-only review.

## Solution Assessment

Yes, with maintainer policy acceptance: a shared helper at the state permission boundary is a maintainable fix, and the latest head adds the missing world-bit rejection. The remaining question is whether OpenClaw should accept fsGroup-compatible group access for these private state paths.
