## Summary

This PR targets a real Kubernetes fsGroup chmod regression, but the current head still needs security review because the shared fallback can accept chmod failures for private `0600` file targets that remain other-readable.

## What This Changes

The branch adds a shared `tryChmodSync` helper, tests EPERM/EACCES/EROFS fallback handling, and routes exec approvals plus task/plugin SQLite permission setup through it.

## Best Possible Solution

Keep the shared fallback only with a target-aware boundary: allow fsGroup directory shapes like `2775`, but preserve the `0600` private-file contract unless maintainers explicitly approve and document a looser Kubernetes policy.

## Reproduction Assessment

Yes at source level: current main rethrows non-Windows chmod failures in the implicated exec approvals path, and the PR body gives a concrete Kubernetes fsGroup `2775` terminal reproduction. I did not run a live Kubernetes cluster in this read-only review.

## Solution Assessment

No, not yet. The shared helper is a reasonable direction for fsGroup directories, but the current implementation should not silently weaken `0600` file hardening without a maintainer security decision and proof.
