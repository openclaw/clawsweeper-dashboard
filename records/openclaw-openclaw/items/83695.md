## Summary

Keep open: this is a plausible fix for a real shipped regression, but the current patch still lacks real Kubernetes/PVC proof and has a blocking security-boundary issue in its access-only chmod fallback.

## What This Changes

The PR adds a shared `tryChmodSync` helper and applies it to exec approvals plus task, flow, and plugin-state SQLite permission hardening paths.

## Best Possible Solution

Land a scoped chmod tolerance that re-stats and rejects unsafe modes or owners except an accepted fsGroup state-path policy, preserves exec approval hardening, and includes redacted real Kubernetes/PVC proof.

## Reproduction Assessment

Yes, from source inspection and the linked issue logs: current main rethrows non-Windows chmod `EPERM` in `ensureDir`, and the repo Kubernetes manifest uses the fsGroup/non-root PVC pattern. I did not run a live PVC reproduction in this read-only review.

## Solution Assessment

No, not yet. Centralizing chmod tolerance is directionally reasonable, but access-only verification is too broad for private state and exec approval hardening without stat-based scope or an explicit maintainer security decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate permissions before swallowing chmod failures:** `src/infra/chmod.ts:17-18`
  - body: `accessSync` only proves OpenClaw can use the path; it also passes for root-owned `0777` directories or `0666` files. If chmod fails there, exec approvals and state registries now continue with private state still globally or unexpectedly group accessible, so re-stat and reject modes outside the accepted fsGroup policy before swallowing the chmod error.
  - confidence: 0.84
