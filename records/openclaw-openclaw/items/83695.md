## Summary

Keep this PR open: it targets a valid P1 Kubernetes fsGroup regression, but the current patch is not merge-ready because it lacks after-fix real deployment proof and relaxes permission hardening using access checks alone.

## What This Changes

The PR adds a `tryChmodSync` helper and applies it to exec approvals plus task, flow, and plugin-state SQLite permission hardening so chmod EPERM/EACCES/EROFS can be tolerated on accessible fsGroup-mounted state paths.

## Best Possible Solution

Keep the fsGroup compatibility fix, but make the fallback verify an acceptable owner/group/mode policy before tolerating chmod failure and require redacted real-environment proof for the affected deployment path.

## Reproduction Assessment

Yes from source inspection and the linked issue logs: current main rethrows non-Windows chmod EPERM in `ensureDir`, and the same strict chmod pattern exists in the state registries. I did not run a live PVC reproduction in this read-only review.

## Solution Assessment

No for the current patch shape: tolerating fsGroup chmod failure is the right direction, but `accessSync` alone is not a sufficient replacement for permission hardening on private state paths.
