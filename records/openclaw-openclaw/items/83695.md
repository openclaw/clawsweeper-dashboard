## Summary

Keep this PR open: the shared helper is the right direction, but the reviewed head still rejects the reported Kubernetes `2775` fsGroup directory mode, so it does not yet fix the linked exec/state startup regression.

## What This Changes

The PR adds a shared `tryChmodSync` helper and routes exec approvals, task/task-flow registries, and plugin-state permission setup through it, with focused chmod fallback tests.

## Best Possible Solution

Land the shared helper only after it accepts accessible fsGroup directories like `2775`, rejects unexpected `other` write such as `2777`, and keeps focused regression coverage for the affected exec and state-store paths.

## Reproduction Assessment

Yes, at source level: current main rethrows non-Windows chmod failures in the affected setup paths, and the linked Kubernetes report/proof gives a concrete `0:1000` `2775` fsGroup state root where chmod fails while access succeeds. I did not run live Kubernetes in this read-only review.

## Solution Assessment

No for the current PR head: the shared helper is the maintainable shape, but its `actualOther` check rejects the central `2775` case. The safer fix is to reject only unexpected `other` write while preserving access-verified fsGroup read/execute bits.
