## Summary

Keep open for maintainer review: the patch appears focused and well-supported, but it intentionally changes the state-directory chmod failure boundary for Kubernetes fsGroup volumes, so this is not a conservative cleanup close.

## What This Changes

The PR adds a shared access-verified `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state SQLite permission hardening through it.

## Best Possible Solution

Land the centralized helper after explicit maintainer approval of the fsGroup directory/private-file permission policy, then let it close https://github.com/openclaw/openclaw/issues/83619.

## Reproduction Assessment

Yes. Source inspection shows current main can rethrow raw chmod failures in the implicated paths, and the PR discussion provides Docker/K8s-style fsGroup output demonstrating EPERM before the patch and success with the helper afterward.

## Solution Assessment

Mostly yes. A single shared helper is the narrow maintainable fix for duplicated chmod hardening, but the exact directory/file permission boundary needs maintainer approval because it changes a security-sensitive state-volume policy.
