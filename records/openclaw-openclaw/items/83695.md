## Label Justifications

- `P1`: The underlying regression can block exec and state-store initialization for Kubernetes users on fsGroup-mounted state volumes.
- `merge-risk: 🚨 compatibility`: Merging changes upgrade behavior from unconditional chmod failure to access-verified tolerance, while still failing some existing file modes that operators may need to repair.
- `merge-risk: 🚨 security-boundary`: The helper deliberately changes private state permission hardening by accepting chmod-failing fsGroup directories with group/other read-execute bits.
- `merge-risk: 🚨 availability`: The touched chmod paths determine whether exec approvals and task/plugin state stores can initialize on affected deployments.

## Summary

Keep this PR open for maintainer review: the chmod EPERM regression is real on current main and the patch has strong real-environment proof, but merge still needs an explicit security/availability decision around tolerating access-verified fsGroup directory modes and failing closed on private files with extra `other` bits.

## What This Changes

Adds a shared `tryChmodSync` helper, routes exec approvals, task registry, task-flow registry, and plugin-state chmod calls through it, and adds focused tests for chmod failure handling.

## Best Possible Solution

Land the shared helper after maintainers explicitly accept the fsGroup permission boundary, or split directory tolerance from SQLite sidecar policy if private state file behavior needs a separate decision.

## Reproduction Assessment

Yes at source/proof level: current main rethrows non-Windows chmod `EPERM` in the exec-approvals path, and the PR includes Docker/K8s-style live output showing current-main failure and patched success across the affected stores. I did not run local tests because this review is read-only.

## Solution Assessment

Mostly yes: centralizing the chmod fallback is a maintainable fix for the repeated chmod pattern, and the tests cover the intended boundaries. The remaining question is not mechanical code shape but whether maintainers accept the directory and private-file permission policy.
