## Label Justifications

- `P1`: The PR targets a regression that can break exec and state initialization for Kubernetes/fsgroup deployments used by real operators.
- `merge-risk: 🚨 compatibility`: Merging changes the upgrade behavior for root-owned fsGroup state volumes and may still fail on sidecars with broader existing file modes.
- `merge-risk: 🚨 security-boundary`: The patch deliberately tolerates private state directories that remain group/other-readable and traversable when chmod cannot tighten them.
- `merge-risk: 🚨 availability`: The affected code runs in exec approvals and SQLite state setup paths, so an incorrect permission policy can keep core agent/store workflows unavailable.

## Summary

Keep this PR open: the chmod EPERM regression is well-supported by current source and the contributor’s Docker/K8s-style proof, but the patch still needs maintainer approval for the fsGroup permission boundary before merge.

## What This Changes

Adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state chmod calls through an access-verified fallback for fsGroup-style permission failures.

## Best Possible Solution

Land the shared helper after maintainers explicitly accept the fsGroup permission boundary, or split directory tolerance from SQLite sidecar policy if private state file behavior needs a separate decision.

## Reproduction Assessment

Yes at source/proof level: current main rethrows non-Windows chmod `EPERM` in exec approvals and raw chmod calls remain in the task/plugin-state stores. The PR discussion includes Docker/K8s-style output demonstrating current-main failure and patched success across the affected stores.

## Solution Assessment

Mostly yes: centralizing the chmod fallback is the maintainable shape for the repeated chmod pattern, and the tests cover the intended boundaries. The remaining question is maintainer acceptance of the security and upgrade policy, not a discrete code defect in the patch.
