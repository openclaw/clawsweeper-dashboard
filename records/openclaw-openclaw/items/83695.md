## Summary

Keep this PR open, but not merge-ready: the focused regression fix is useful, yet the current live head still rejects the reported Kubernetes fsGroup directory modes after access succeeds, so it likely leaves the linked startup/exec failure unresolved.

## What This Changes

The branch adds a shared `tryChmodSync` helper and routes exec approvals, task/task-flow SQLite stores, and plugin-state SQLite permission setup through it.

## Best Possible Solution

Align the helper, tests, and proof around the exact fsGroup state-root modes OpenClaw intends to support, while still rejecting writable public access to private state.

## Reproduction Assessment

Yes, at source level: current main rethrows non-Windows `chmodSync(dir, 0o700)` failures, and the linked issue plus PR proof describe accessible fsGroup-owned directories where chmod fails. I did not run live K8s proof during this read-only review.

## Solution Assessment

No, not yet: the shared helper is the right shape, but the current guard rejects the exact fsGroup directory modes in the report and proof. The narrow maintainable fix is to encode the maintainer-approved mode boundary and test the `2775` case.
