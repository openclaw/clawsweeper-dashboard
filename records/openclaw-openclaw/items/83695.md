## Label Justifications

- `P1`: The PR addresses an urgent regression that blocks exec and state-store workflows on Kubernetes fsGroup-mounted state volumes.
- `merge-risk: 🚨 compatibility`: The fix changes the runtime response to chmod failures on existing root-owned volume layouts and may still fail closed for sidecar modes that operators must repair.
- `merge-risk: 🚨 security-boundary`: The PR intentionally tolerates chmod failure for accessible private state directories, leaving fsGroup directory modes such as 2775 in place after verification.

## Summary

Keep open: this is a focused P1 regression fix with source-backed reproduction and strong real-environment proof, but merging it changes the private-state chmod fail-closed boundary on fsGroup-style volumes, so maintainer security and upgrade-policy approval is still needed.

## What This Changes

Adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state chmod hardening through access-verified EPERM/EACCES/EROFS tolerance with regression tests.

## Best Possible Solution

Land the shared helper once maintainers approve the fsGroup directory and private-file boundary, with operator guidance or a split follow-up if that boundary is not accepted.

## Reproduction Assessment

Yes. Current main has raw non-Windows `chmodSync` rethrows in the affected setup paths, and the PR discussion provides Docker/K8s-style output showing EPERM on root:fsGroup state paths before the patch and success with the helper afterward.

## Solution Assessment

Mostly yes. Centralizing the access-verified chmod fallback is the maintainable shape for the repeated chmod pattern; the remaining question is maintainer acceptance of the security and upgrade policy, not a discrete code defect in the patch.
