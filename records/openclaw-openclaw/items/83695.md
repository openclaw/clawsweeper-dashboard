## Summary

Keep open: the PR targets a real P1 regression, but the live head still uses an access-only chmod fallback for private state paths and has no after-fix Kubernetes/PVC proof. The latest author comment says stat validation was added, but `gh pr diff` for the current head still shows only `accessSync`, so the branch needs another update before merge.

## What This Changes

The PR adds a `tryChmodSync` helper and applies it to exec approvals, task registry, task-flow registry, and plugin-state SQLite permission hardening so selected chmod failures can be tolerated on fsGroup-style mounts.

## Best Possible Solution

Keep the fsGroup compatibility direction, but land a helper that stat-validates acceptable owner/group/mode before tolerating chmod failure, with focused regression tests and redacted real Kubernetes/PVC proof.

## Reproduction Assessment

Yes, source inspection plus the linked deployment logs give a high-confidence reproduction path: current main rethrows non-Windows `chmodSync` EPERM in `ensureDir` before exec approval persistence can continue. I did not run a live Kubernetes PVC reproduction in this read-only review.

## Solution Assessment

No for the current PR head: tolerating fsGroup chmod failures is the right direction, but an access-only fallback is not the narrowest safe replacement for private state permission hardening. The safer path is stat-based owner/group/mode validation before swallowing the chmod failure.
