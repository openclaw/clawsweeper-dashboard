## Summary

Keep open for normal maintainer review: current main still has the raw chmod crash path, and this PR is a focused, tested fix with no blocking correctness findings. The remaining question is maintainer acceptance of the chmod tolerance and private-file security boundary before merge.

## What This Changes

The branch adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state SQLite permission setup through it.

## Best Possible Solution

Land the shared helper once maintainers accept the fsGroup permission boundary, preserving fail-closed checks for unsafe private files and adding operator guidance only if maintainers want cleanup steps called out.

Yes. Current main rethrows non-Windows chmod failures in exec approvals and uses raw chmod in the store permission paths, while the PR discussion provides concrete Kubernetes-style and Docker fsGroup terminal proof for the EPERM behavior.

## Solution Assessment

Yes, pending maintainer acceptance of the boundary. A shared access-verified helper is narrower than scattered best-effort catches, and the tests preserve fail-closed behavior for inaccessible paths, other-writable directories, and private files with unsafe other bits.
