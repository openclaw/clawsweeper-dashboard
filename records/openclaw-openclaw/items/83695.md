## Summary

Keep open: this PR targets a real Kubernetes fsGroup regression, but the live branch still swallows chmod failures after only `accessSync`, which is too weak for exec approvals and persistent state permission hardening.

## What This Changes

The PR adds a `tryChmodSync` helper and replaces strict chmod calls in exec approvals, task registry, task-flow registry, and plugin-state SQLite permission setup.

## Best Possible Solution

Keep the fsGroup compatibility fix, but make chmod fallback validate an acceptable owner/group/mode policy before tolerating failure and require redacted real-environment proof for the affected deployment path.

## Reproduction Assessment

Yes from source inspection and the linked issue logs: current main still rethrows non-Windows chmod EPERM in exec approvals, and the same strict chmod pattern exists in the state registries. I did not run a live Kubernetes PVC reproduction in this read-only review.

## Solution Assessment

No for the current patch shape: tolerating fsGroup chmod failure is the right direction, but `accessSync` alone is not a sufficient replacement for permission hardening on private state paths.
