## Summary

Keep open: this PR targets a real P1 chmod regression, but the current head still relies on an access-only fallback for private state paths and the submitted proof is mocked unit-test output rather than real fsGroup/PVC behavior.

## What This Changes

The PR adds `tryChmodSync`, uses it for exec approvals plus task, flow, and plugin-state SQLite permission hardening, and adds mocked helper tests.

## Best Possible Solution

Keep the fsGroup compatibility direction, but land a stat-based owner/group/mode fallback for private state paths and require redacted real-environment proof before merge.

## Reproduction Assessment

Yes. The linked issue logs show EPERM on the same chmod path, and current main still rethrows non-Windows chmod errors in exec approvals while related registries use the same strict pattern.

## Solution Assessment

No for the current PR head. Tolerating fsGroup chmod failure is the right direction, but `accessSync` alone is not a maintainable replacement for private permission hardening.
