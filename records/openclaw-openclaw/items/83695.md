## Label Justifications

- `P1`: The linked reports describe a regression that blocks exec and state initialization on Kubernetes fsGroup deployments.
- `merge-risk: 🚨 compatibility`: Merging changes upgrade behavior for deployments whose state directories or SQLite sidecars cannot be chmodded by the OpenClaw process.
- `merge-risk: 🚨 security-boundary`: Merging relaxes fail-closed chmod hardening on private state paths when access and mode checks pass.

## Summary

Keep this PR open for maintainer/security review: the bug is real and the patch shape is plausible, but it intentionally changes private state chmod from fail-closed to access-verified continuation on fsGroup-owned paths.

## What This Changes

The PR adds a shared chmod helper and routes exec approvals, task registry, task-flow registry, and plugin-state SQLite permission hardening through it for Kubernetes fsGroup-owned state paths.

## Best Possible Solution

Land the shared helper after explicit security/compatibility approval and exact-head fsGroup validation, keeping fail-closed behavior for inaccessible paths, other-write directories, and private files with unsafe other bits.

## Reproduction Assessment

Yes. Current main still has the raw chmod/rethrow paths, and the PR body/comments include Docker and Kubernetes-style fsGroup output showing EPERM before the patch and successful access-verified continuation after it.

## Solution Assessment

Mostly yes. A shared helper is the maintainable shape for the repeated chmod hardening paths, but the exact directory and file permission tolerance is a security/compatibility boundary that maintainers should explicitly approve.
