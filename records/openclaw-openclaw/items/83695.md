## Label Justifications

- `P1`: The PR targets a regression that blocks exec and startup state paths for Kubernetes fsGroup deployments.
- `merge-risk: 🚨 compatibility`: The correct fallback policy can change whether existing fsGroup-mounted state files continue to work or fail closed on upgrade.
- `merge-risk: 🚨 security-boundary`: The patch changes permission hardening for private SQLite state files and can leave `0o600` targets other-readable.
- `merge-risk: 🚨 availability`: The affected chmod path can crash exec approvals and state initialization, and the proof does not yet cover full startup after the permission-boundary choice.

## Summary

Keep open: the Kubernetes chmod regression is real, but the current head still weakens the existing private state-file permission boundary by tolerating other-readable files when chmod fails.

## What This Changes

The branch adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state permission setup through it for Kubernetes fsGroup chmod failures.

## Best Possible Solution

Keep fsGroup tolerance for state directories, but make the fallback target- and mode-aware so private files requested as `0o600` do not silently keep broader permissions unless maintainers explicitly approve and document that policy.

## Reproduction Assessment

Yes at source level: current main rethrows non-Windows chmod failures in `src/infra/exec-approvals.ts`, and Kubernetes fsGroup source/docs support the root-owned group-access volume mode described by the reporter. I did not run a live cluster in this read-only review.

## Solution Assessment

No: a shared helper is a good direction, but the current implementation applies a directory-style other-read tolerance to `0o600` file targets. The safer solution is a mode-aware boundary or an explicit maintainer-approved exception for fsGroup file modes.
