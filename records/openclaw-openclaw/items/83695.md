## Summary

Keep open: this is a focused fix for a real Kubernetes fsGroup chmod failure, and I did not find a discrete code defect in the current patch. Merge still needs maintainer acceptance of the permission-boundary and upgrade behavior because the helper intentionally relaxes directory chmod hardening when access is otherwise verified.

## What This Changes

Adds a shared `tryChmodSync` helper and uses it for exec approvals, task registry, task-flow registry, and plugin-state chmod hardening on fsGroup-style filesystems.

## Best Possible Solution

Land the shared helper after maintainers explicitly approve the fsGroup directory and private-file permission boundary, or split any unsettled sidecar policy into a narrower follow-up before merge.

## Reproduction Assessment

Yes. Current main has raw non-Windows `chmodSync` rethrows in the affected setup paths, and the PR discussion includes Docker/K8s-style proof showing EPERM on root:fsGroup state paths before the patch and success with the helper afterward.

## Solution Assessment

Mostly yes. Centralizing the access-verified chmod fallback is the maintainable shape for the repeated chmod pattern; the remaining issue is maintainer acceptance of the security and upgrade policy, not a concrete code defect I can point to in the patch.
