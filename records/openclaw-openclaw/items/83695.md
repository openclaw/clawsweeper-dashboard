## Summary

Keep open: this is a focused regression fix with sufficient live-output proof and no discrete blocking code finding, but it changes sensitive state-permission behavior and needs explicit maintainer/security approval before merge.

## What This Changes

The PR adds a shared `tryChmodSync` helper with tests and routes exec approvals, task registry, task-flow registry, and plugin-state chmod hardening through it.

## Best Possible Solution

Land the shared helper after explicit security/compatibility approval and exact-head fsGroup validation, preserving fail-closed behavior for inaccessible paths, other-write directories, and private files with unsafe other bits.

## Reproduction Assessment

Yes. Current main has raw chmod/rethrow paths, and the PR body/comments provide Docker and Kubernetes-style fsGroup output showing EPERM before the patch and access-verified success after it.

## Solution Assessment

Mostly yes. A shared helper is the maintainable shape for the repeated chmod hardening paths, but the exact directory and file permission tolerance is a security/compatibility boundary that maintainers should explicitly approve.
