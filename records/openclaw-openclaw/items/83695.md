## Summary

Keep this PR open for maintainer review. The patch is focused and the real fsGroup proof is sufficient, but it intentionally changes a security-sensitive filesystem permission boundary for OpenClaw state paths.

## What This Changes

Adds a shared `tryChmodSync` helper for access-verified chmod fallback on fsGroup-owned state paths and routes exec approvals, task registry, task-flow registry, and plugin-state permission setup through it.

## Best Possible Solution

Land the centralized helper after explicit maintainer approval of the fsGroup directory/private-file permission policy, then use it as the canonical fix for the linked Kubernetes chmod reports.

## Reproduction Assessment

Yes. Source inspection shows current main rethrows raw chmod failures in the affected paths, and the PR discussion provides Docker/K8s-style terminal output demonstrating EPERM before the patch and success with the helper afterward.

## Solution Assessment

Mostly yes. A single shared helper is the narrow maintainable fix for duplicated chmod hardening, but the exact directory/file permission boundary needs maintainer approval because it changes security-sensitive state-volume behavior.
