## Label Justifications

- `P1`: The linked reports show a regression that can make exec-dependent agent workflows unusable on Kubernetes fsGroup deployments.
- `merge-risk: 🚨 compatibility`: The PR changes how existing chmod failures on persisted state paths are handled during upgrades and runtime initialization.
- `merge-risk: 🚨 security-boundary`: The PR deliberately relaxes directory chmod enforcement when access and mode checks pass, which is a private-state security boundary.
- `merge-risk: 🚨 availability`: The change is meant to prevent Kubernetes exec and state initialization crashes, but the wrong boundary could still leave affected deployments broken.

## Summary

Keep open: current main still has the hard chmod paths, and this PR is the active fix for a real Kubernetes fsGroup regression; the remaining blocker is maintainer approval of the filesystem permission boundary.

## What This Changes

The PR adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state permission hardening through it so selected chmod failures can be tolerated after access and mode checks.

## Best Possible Solution

Land the shared helper after maintainers explicitly accept the fsGroup permission boundary, or split directory tolerance from private-file sidecar policy if the state-store upgrade behavior needs separate review.

## Reproduction Assessment

Yes at source/proof level: current main rethrows non-Windows chmod `EPERM` in the exec-approvals path, and the PR discussion includes Docker/K8s-style terminal proof showing the fsGroup failure and patched success. I did not run a live cluster or tests during this read-only review.

## Solution Assessment

Mostly yes: centralizing the chmod fallback is maintainable and directly targets the implicated paths, but the exact directory and private-file permission boundary is a maintainer security policy decision before merge.
