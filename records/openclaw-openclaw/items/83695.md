## Summary

Keep open: this targets a real Kubernetes fsGroup chmod regression, but the live PR head still only rejects `other-write`, so private `0600` SQLite sidecars can remain world-readable when chmod fails. A later comment says the helper is mode-aware, but the current diff and commit list do not show that implementation.

## What This Changes

The branch adds a shared `tryChmodSync` helper, tests it, and routes exec approvals, task registry, task-flow registry, and plugin-state SQLite permission setup through that helper.

## Best Possible Solution

Keep the shared helper, but make the fallback target-aware: tolerate fsGroup directory read/execute bits like `2775`, reject unexpected `other` bits on `0600` file targets, add regression tests, and refresh real K8s or equivalent startup proof.

## Reproduction Assessment

Yes, at source level: current main still rethrows the exec approvals chmod failure, and the linked issue plus PR terminal output give a concrete Kubernetes fsGroup `2775` path. I did not run a live Kubernetes cluster in this read-only review.

## Solution Assessment

No for the current head: the shared helper is the right direction, but its fallback boundary is too broad for `0600` files. The safer solution is directory-specific fsGroup tolerance plus stricter private-file checks.
