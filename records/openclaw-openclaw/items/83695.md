## Summary

Keep open: the patch is a focused fix for a real Kubernetes fsGroup chmod regression and now has sufficient copied live output, but merge still needs maintainer acceptance of the private state-directory permission boundary and upgrade behavior for strict sidecar file modes.

## What This Changes

Adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state chmod calls through an access-verified fallback for fsGroup-style permission failures.

## Best Possible Solution

Land the shared helper after maintainers explicitly accept the fsGroup permission boundary, or split directory tolerance from SQLite sidecar policy if private state file behavior needs a separate decision.

## Reproduction Assessment

Yes at source/proof level: current main rethrows non-Windows chmod `EPERM` in exec approvals and raw chmod calls remain in the task/plugin-state stores. The PR discussion includes Docker/K8s-style output demonstrating current-main failure and patched success across the affected stores.

## Solution Assessment

Mostly yes: centralizing the chmod fallback is the maintainable shape for the repeated chmod pattern, and the tests cover the intended boundaries. The remaining question is a maintainer security/upgrade policy decision, not a discrete code defect I can point to in the patch.
