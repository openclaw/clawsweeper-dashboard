## Summary

Keep open: the PR addresses a real Kubernetes fsGroup regression that is still present on current main, but merge should wait for maintainer/security approval of the chmod tolerance boundary and exact-head validation.

## What This Changes

The PR adds a shared chmod fallback helper and routes exec approvals, task registry, task-flow registry, and plugin-state permission hardening through it for fsGroup-mounted state paths.

## Best Possible Solution

Land the shared helper after explicit security/compatibility approval and exact-head fsGroup validation, preserving fail-closed behavior for inaccessible paths, other-write directories, and private files with unsafe other bits.

## Reproduction Assessment

Yes, with high confidence from current-main source and submitted real-environment output. I did not run a local Kubernetes repro in this read-only pass, but current main still has the raw chmod paths and the PR discussion shows fsGroup EPERM before the patch plus successful access/stat-verified continuation after it.

## Solution Assessment

Mostly yes. A shared helper is the maintainable shape for the repeated chmod hardening paths, but the exact directory/file permission tolerance is a security and compatibility policy that maintainers should approve before merge.
