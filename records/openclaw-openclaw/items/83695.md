## Summary

Keep open: the PR targets a real P1 fsGroup chmod regression, but the live diff still only uses `accessSync` after chmod failure and no after-fix Kubernetes or equivalent filesystem proof is attached.

## What This Changes

Adds a `tryChmodSync` helper and replaces strict `chmodSync` calls in exec approvals plus SQLite task, flow, and plugin-state permission hardening with that tolerant helper and unit tests.

## Best Possible Solution

Keep the fsGroup compatibility fix, but require stat-based acceptable owner/group/mode validation before swallowing chmod failures and add redacted real-environment proof for the affected deployment path.

## Reproduction Assessment

Yes, from source inspection and the linked issue logs: current main rethrows non-Windows chmod EPERM in `ensureDir`, and the same strict chmod pattern exists in the state registries. I did not run a live Kubernetes PVC reproduction in this read-only review.

## Solution Assessment

No for the live PR shape: tolerating managed-volume chmod failure is the right direction, but `accessSync` alone is not a sufficient replacement for permission hardening on private state paths.
