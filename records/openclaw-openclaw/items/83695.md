## Label Justifications

- `P1`: The PR targets a regression that can block exec and startup/state initialization on Kubernetes fsGroup deployments.
- `merge-risk: 🚨 compatibility`: The helper changes which existing mounted state paths are accepted after chmod failures during upgrade/runtime startup.
- `merge-risk: 🚨 security-boundary`: The diff changes private state permission hardening and currently tolerates world-readable `0600` file targets when chmod fails.
- `merge-risk: 🚨 availability`: An incorrect permission boundary can either preserve the reported startup/exec failure or allow unsafe state paths to proceed.

## Summary

Keep this PR open: it addresses a real P1 chmod regression, but the current helper weakens the private-file permission boundary for SQLite state sidecars by tolerating world-readable files when chmod fails.

## What This Changes

Adds a shared `tryChmodSync` helper and routes exec approvals, task registry, task-flow registry, and plugin-state permission setup through it for selected chmod failures.

## Best Possible Solution

Keep the shared helper, but make the fallback boundary target-aware: tolerate fsGroup directory read/execute bits like `2775`, reject unexpected `other` bits on `0600` files, and refresh tests plus real K8s proof after that change.

## Reproduction Assessment

Yes, at source level: current main still rethrows the exec approvals chmod failure, and the linked issue plus PR proof give a concrete Kubernetes fsGroup `2775` path. I did not run a live Kubernetes cluster in this read-only review.

## Solution Assessment

No for the current head: the shared helper is the right direction, but the security check is too broad for `0600` files. The safer solution is a directory-specific fsGroup exception plus stricter private-file checks.
