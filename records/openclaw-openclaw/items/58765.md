## Summary

Keep open: current main and the latest v2026.5.18 release still accept and forward `memorySearch.outputDimensionality`, but the built-in local GGUF provider ignores it and returns full normalized vectors. The closed OpenAI-compatible dimensions issue does not cover this local provider path.

## What This Changes

Request support for truncating local GGUF memory embedding vectors via the existing `memorySearch.outputDimensionality` setting, with reindex/cache identity handling when the dimension changes.

## Best Possible Solution

Implement local GGUF truncation through `memorySearch.outputDimensionality`, include the dimension in local provider cache/reindex identity, add focused tests/docs, and leave query-instruction support for a separate product/config discussion.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: create the local provider with `outputDimensionality` and a mocked `node-llama-cpp` vector, then both query and batch paths return the full normalized vector because the option is never read. I did not run the repro because this review was read-only.

## Solution Assessment

Yes for the main requested fix: reusing the existing `outputDimensionality` contract is narrower than adding a duplicate local-only config key. The bonus query-instruction idea is separate and would need its own config/product decision.
