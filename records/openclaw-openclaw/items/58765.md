## Summary

Keep open: current main and v2026.5.22 still carry `memorySearch.outputDimensionality` through config resolution, but the built-in local GGUF provider never reads it and still returns full normalized vectors. The closed related dimension issue solved the OpenAI-compatible path, not this local provider path.

## What This Changes

The issue requests applying `memorySearch.outputDimensionality` to local GGUF embedding vectors and reindexing when that dimension changes, with query-instruction support as a separate bonus idea.

## Best Possible Solution

Implement local GGUF vector truncation through the existing `memorySearch.outputDimensionality` field, include that dimension in local provider cache/reindex identity, and track query-instruction support separately.

## Reproduction Assessment

Yes, source-level. Configure local memory search with `outputDimensionality` and a mocked `node-llama-cpp` vector longer than the requested size; current query and batch paths return the full normalized vector because the local provider never reads the option.

## Solution Assessment

Yes for the main request. Reusing the existing `outputDimensionality` field is the narrowest maintainable path; the query-instruction bonus should remain separate because it adds new behavior beyond dimensionality truncation.
