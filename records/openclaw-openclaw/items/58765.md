## Summary

Keep open: current `main` and the latest release still preserve `memorySearch.outputDimensionality` through generic memory-search config, but the built-in local GGUF provider never reads it and still returns full normalized vectors. The related closed dimension work covers OpenAI-compatible and generic provider plumbing, not this local provider truncation path.

## What This Changes

Support truncating local GGUF memory embeddings through the existing `memorySearch.outputDimensionality` setting, with matching reindex/cache identity.

## Best Possible Solution

Implement local GGUF truncation by slicing raw vectors before normalization, include the selected dimension in local provider cache/reindex identity, document local support, and leave query-instruction support to a separate item.

## Reproduction Assessment

Yes, at source level. Configure local memory search with `outputDimensionality`, mock `node-llama-cpp` to return a longer vector, and both query and batch paths still return the full normalized vector because the local provider never reads the option.

## Solution Assessment

Yes for the main requested fix. Reusing `outputDimensionality` is the narrowest maintainable path because schema, config resolution, and provider request plumbing already carry it; `queryInstruction` should stay separate.
