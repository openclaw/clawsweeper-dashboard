## Summary

Keep open. Current main still accepts and resolves `memorySearch.outputDimensionality`, but the built-in local GGUF provider never reads it and returns full normalized vectors; the local provider runtime key also omits dimensionality, so a dimension change would not reliably trigger the existing full-reindex path. The open OpenAI-compatible dimensions PR is adjacent, not a replacement for this local GGUF request.

## What This Changes

The issue requests making `memorySearch.outputDimensionality` truncate local GGUF embedding vectors and reindex when the configured dimension changes.

## Best Possible Solution

Implement local GGUF truncation by slicing the raw vector to the configured positive dimension before normalization, include that dimension in the local adapter runtime/cache key so provider-key reindexing fires, add focused query and batch tests, and document local support in the memory config reference. Keep query instructions as a separate product/API follow-up.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to use the existing mocked `createLocalEmbeddingProvider` test harness with a known vector and `outputDimensionality: 2`; current code never reads that option, so `embedQuery` and `embedBatch` return normalized full-length vectors.

## Solution Assessment

Yes for the main requested fix. The narrowest maintainable solution is local-provider truncation plus provider-key/reindex identity and docs/tests; adding a new `truncateDimensions` field or query-instruction API is unnecessary for this bug.
