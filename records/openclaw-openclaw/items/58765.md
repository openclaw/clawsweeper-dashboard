## Summary

Keep open: current `main` and latest release still preserve `memorySearch.outputDimensionality` through generic memory-search config, but the built-in local GGUF provider never reads it and still returns full normalized vectors. The related closed dimension issues cover OpenAI-compatible and generic dimension plumbing, not this local provider path.

## What This Changes

Request to make `memorySearch.outputDimensionality` truncate vectors from local GGUF embedding providers and reindex when the configured dimension changes, with a separate bonus query-instruction idea.

## Best Possible Solution

Implement local GGUF truncation through the existing `memorySearch.outputDimensionality` field by slicing raw vectors before normalization, include the dimension in local provider cache/reindex identity, add focused tests/docs, and leave query-instruction support to a separate item or PR.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure local memory search with `outputDimensionality`, mock `node-llama-cpp` to return a longer vector, and both query and batch paths return the full normalized vector because the local provider never reads the option.

## Solution Assessment

Yes for the main requested fix. Reusing `outputDimensionality` is the narrowest maintainable path because config resolution and provider requests already carry it; the bonus query-instruction behavior should stay separate.
