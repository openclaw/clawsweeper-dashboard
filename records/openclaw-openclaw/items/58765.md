## Summary

Keep open: current main still resolves `memorySearch.outputDimensionality`, but the built-in local GGUF provider ignores it and the local provider cache identity still omits the configured dimension. The related OpenAI-compatible dimensions work is shipped, but it does not cover this local GGUF path.

## What This Changes

The issue requests that local GGUF memory embedding providers honor the existing `memorySearch.outputDimensionality` setting by truncating vectors and reindexing when the dimension changes.

## Best Possible Solution

Implement local GGUF truncation through the existing `outputDimensionality` contract, add the dimension to local runtime/cache identity, update focused tests and docs, and leave configurable query instructions as separate follow-up work.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: instantiate `createLocalEmbeddingProvider` with a mocked `node-llama-cpp` vector and `outputDimensionality: 2`; current code never reads that option and returns the full normalized vector.

## Solution Assessment

Yes for the main requested fix: the narrow maintainable path is to truncate the local raw vector before normalization, include the dimension in local provider identity, and add tests/docs. Adding `queryInstruction` would broaden the public config surface and should be a separate decision.
