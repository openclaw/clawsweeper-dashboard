## Label Justifications

- `P1`: The PR targets an urgent warm-turn performance regression, but the current patch can break existing-session agent runs before they start.
- `merge-risk: 🚨 availability`: Frozen loaded entries can cause a runtime `TypeError` in the embedded-runner preparation path for existing session files.
- `merge-risk: 🚨 session-state`: The cache changes transcript object ownership and mutability for loaded session state across all `SessionManager.open` callers.

## Summary

Keep open: the performance target and supplied proof are useful, but the patch currently freezes entries that `SessionManager` stores, while the embedded runner still mutates the loaded session header during existing-session preparation; that can turn warm agent runs into a runtime failure.

## What This Changes

The PR adds a bounded snapshot-keyed cache around `loadEntriesFromFile` and focused session-manager tests for warm transcript reuse, append updates, migration bypass, and cache invalidation.

## Best Possible Solution

Keep the transcript parse cache private and immutable, but return mutable manager-owned entries or move header normalization behind a `SessionManager` API that also updates the cache.

## Reproduction Assessment

Yes for the review finding from source inspection: PR head returns frozen current-version entries, and current embedded-runner preparation mutates the loaded header for existing session files. I did not run a write-producing reproduction command in this read-only review.

## Solution Assessment

No: caching the parse result is a plausible performance fix, but returning frozen objects into the manager conflicts with current session preparation. The safer fix is to keep only the private cached snapshot frozen while returning mutable manager-owned entries, or to move the mutation behind a cache-aware manager API.
