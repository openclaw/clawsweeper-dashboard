## Summary

Keep open. Current main does not contain the central follow-up SDK surfaces, the PR is still a draft stacked on open #73384 with a dirty merge state, and the PR head has a concrete P2 stale-registry side-effect bug that should block merge.

## What This Changes

The PR adds additive plugin SDK and Gateway seams for host runtime reads, derived tool paths, attachment hints, input-suppression descriptors, tagged scheduled-turn cleanup, session-entry slot projection, public channel helper exports, QA boundary cleanup, generated clients, and contract tests.

## Best Possible Solution

Keep the draft available, add the same live-plugin side-effect guard to tagged unscheduling that the sibling workflow APIs use, preserve the additive SDK boundary, then rebase after #73384 and rerun focused SDK/protocol/generated-client validation before marking it ready.

## Reproduction Assessment

Yes for the review finding: retain a plugin API object from a retired registry, load a replacement registry with tagged scheduled turns for the same plugin/session, then call unscheduleSessionTurnsByTag on the stale API object. The PR head reaches cron.list and cron.remove without the live-plugin guard.

## Solution Assessment

No, the PR is not mergeable as-is. The additive SDK direction fits the plugin boundary, but tagged unscheduling needs the existing shouldCommitWorkflowSideEffect guard before it can safely mutate cron state.
