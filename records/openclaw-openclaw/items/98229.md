## Summary

Keep open: current main and the latest release still render reseed history user turns with the hardcoded `User:` prefix, and no duplicate or merged fix appears to own the remaining sender-attribution problem.

## What This Changes

The issue requests that CLI history rendering preserve group-chat sender attribution instead of labeling every prior user turn as `User:`.

## Best Possible Solution

Update the source renderer and focused session-history tests so user turns with trusted sender metadata render with a sanitized sender label while preserving the `User` fallback for unattributed direct or legacy turns.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call `buildCliSessionHistoryPrompt` with two user history messages that carry different sender metadata and current main will still render both with `User:`. I did not run a live group-chat repro in this read-only review.

## Solution Assessment
