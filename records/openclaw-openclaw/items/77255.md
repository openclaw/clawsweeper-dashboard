- `impact:other`: The issue affects user-visible message ordering and failure clarity, which is maintainer-visible but does not cleanly fit the owned message-loss/session/security labels.

## Summary

Keep open: current main and the latest release still assemble eligible tool-error warning payloads after assistant answer payloads, and current tests assert that ordering for several unacknowledged failed-tool cases. The exact blanket move suggested in the issue is not obviously safe because current tests and adjacent PR review discussion distinguish terminal warnings, direct progress leaks, and non-terminal middleware warnings.

## What This Changes

The issue requests that tool error warnings be shown before or inline with the assistant response instead of appended after it.

## Best Possible Solution

Define the intended ordering for true in-run tool failures versus post-output or middleware warnings, then update the payload builder and focused tests so the warning is shown before/inline only where that policy is correct.

## Reproduction Assessment

Yes from source inspection: current main and latest release still append eligible `lastToolError` warnings after assistant answer payloads, and current tests assert that order for `Write`, `Exec`, and `Edit` warnings. I did not run a live channel flow because this read-only review had enough source and test evidence to verify the ordering claim.

## Solution Assessment

No for the reporter's exact suggested move as a blanket change. A better fix should distinguish true in-run tool failures from post-output or middleware warnings and should account for adjacent direct-chat progress suppression work before changing the ordering contract.
