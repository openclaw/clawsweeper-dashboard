## Label Justifications

- `P2`: This is a normal-priority user-visible reply-shaping issue with bounded agent payload code but real confusion when a failed tool warning follows a successful-looking answer.
- `impact:other`: The issue affects user-visible message ordering and failure clarity, which is maintainer-visible but does not cleanly fit the owned message-loss, session, auth, or security labels.

## Summary

Keep open: current main and the latest release still append eligible tool-error warnings after assistant answer payloads, and adjacent open work only covers specific direct-chat or UI leak surfaces rather than this broader ordering contract.

## What This Changes

Request that tool-error warning payloads appear before or inline with assistant response text instead of after the final answer.

## Best Possible Solution

Define the product contract for terminal versus non-terminal tool errors, then update `buildEmbeddedRunPayloads` and focused tests only for warnings that should be pre-answer or inline while preserving guaranteed fallback behavior.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.6` append eligible `lastToolError` warnings after assistant answer payloads, and current tests assert that order for `Write`, `Exec`, and `Edit` warning cases.

## Solution Assessment

No for the exact blanket move suggested in the issue. The safer solution is a maintainer-approved warning-order policy that distinguishes true in-run failures from terminal fallback, direct progress leaks, and middleware/non-terminal warnings.
