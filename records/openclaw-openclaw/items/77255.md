## Summary

Keep open: current main still appends eligible tool-error warnings after assistant answer payloads, and the latest release plus focused tests still pin that ordering. The issue is valid at source level, but the safest fix needs a maintainer decision on warning order versus recovery-aware warning wording before changing behavior.

## What This Changes

The issue asks OpenClaw to show tool-error warnings before or inline with assistant responses instead of appending them after the final answer.

## Best Possible Solution

Define a recovery-aware warning-order contract, then update buildEmbeddedRunPayloads and focused payload tests so warnings are shown before/inline only when that does not conflict with acknowledgement and recovery semantics.

## Reproduction Assessment

Yes, source-level with high confidence: current main pushes assistant answer payloads before appending eligible lastToolError warnings, and focused tests assert that order. I did not run tests because this review is read-only.

## Solution Assessment

No for the suggested one-line blanket move; it is plausible but not safely proven because the current code scans assistant text before deciding whether a mutating failure warning is needed. The better fix is a maintainer-approved warning-order and recovery policy with updated tests.
