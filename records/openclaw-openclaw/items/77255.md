- `impact:other`: The issue affects how users interpret channel-visible tool failure warnings, but it is not data loss, message loss, session state, auth, crash, or security by itself.

## Summary

Keep open: current main and the latest release still append visible tool-error warnings after assistant answer payloads, and adjacent tests assert that order for edit/exec failures after assistant text. The report is source-reproducible, but the fix should preserve the existing warning policy instead of only moving the block named in the issue body.

## What This Changes

Request tool-error warning payloads be shown before or inline with the assistant response instead of being appended after it.

## Best Possible Solution

Refactor payload assembly so it computes assistant payloads and warning policy first, then emits real user-visible tool-error warnings before or inline with the final answer while preserving suppression, acknowledgement, duplicate, redaction/detail, and non-terminal metadata rules.

## Reproduction Assessment

Yes, by source inspection. Current `buildEmbeddedRunPayloads` pushes assistant answer payloads before the `lastToolError` warning, and tests assert assistant text at index 0 with the warning at index 1 for representative failures.

## Solution Assessment

No, not exactly as suggested. Moving the `lastToolError` block before the answer loop would lose the existing answer-aware policy inputs unless the code first parses answer payloads, computes acknowledgement/suppression state, and then emits in the chosen order.
