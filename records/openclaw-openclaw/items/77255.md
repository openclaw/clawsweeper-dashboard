## Summary

Keep open: current main and v2026.6.10 still append eligible tool-error warnings after assistant answer payloads, and focused tests intentionally pin that order. Related open issues overlap on failure recovery and warning content, but none supersedes this specific warning-order contract.

## What This Changes

The issue requests that embedded-runner tool failure warnings appear before or inline with assistant answer text instead of after the answer.

## Best Possible Solution

Define a shared warning-order and recovery-aware policy for terminal versus recovered/non-terminal tool failures, then update the payload builder and focused tests to match that maintainer-approved contract.

## Reproduction Assessment

Yes, source-level with high confidence. Current main pushes answer payloads before appending eligible `lastToolError` warnings, and focused tests assert the assistant answer at index 0 with the warning at index 1.

## Solution Assessment

No for the suggested blanket one-line move. The safer solution is a maintainer-approved warning-order policy because current code uses assistant text to decide whether a mutating failure was already acknowledged, and related issues may require recovery-aware wording or suppression instead of simple reordering.
