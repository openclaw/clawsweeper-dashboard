## Summary

Keep open: current main and v2026.6.8 still append eligible `lastToolError` warnings after assistant answer payloads, while related open and merged work only handles narrower direct-chat, Discord, Telegram, UI, or config-policy surfaces.

## What This Changes

The issue asks OpenClaw to place tool-error warning payloads before or inline with assistant text instead of appending them after a successful-looking assistant response.

## Best Possible Solution

Define and test a shared warning-order policy for terminal versus non-terminal tool errors, then update payload assembly and channel suppression consistently.

## Reproduction Assessment

Yes. Source inspection on current main shows the answer loop pushes payloads before the `lastToolError` warning block, and tests assert assistant text at index 0 with the warning at index 1 for write, exec, and edit cases.

## Solution Assessment

No for the blanket move suggested in the issue. The safer fix is a maintainer-approved policy that distinguishes terminal failure warnings, non-terminal warnings, and channel progress leaks before changing ordering.
