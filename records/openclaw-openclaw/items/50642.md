## Summary

Keep open. Current main still enables macOS direct `wss://` TOFU for a fresh gateway, accepts and stores the presented certificate before requiring trust success, and the linked closing PR remains open rather than merged.

## What This Changes

The issue requests hardening macOS node direct `wss://` gateway connections so first-use TLS cannot silently pin and trust a rogue certificate.

## Best Possible Solution

Harden macOS direct `wss://` node mode so first connection fails closed unless normal trust succeeds or an explicit configured or confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into that session, and add focused Swift coverage for first-use trust and pin behavior.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure macOS direct remote mode to a fresh `wss://` host with no stored Keychain fingerprint, and current main sets `allowTOFU: true` then accepts and stores the presented certificate before requiring `systemTrustOk`.

## Solution Assessment

Yes for the requested hardening direction, but current main is not the solution. The maintainable fix should reject silent untrusted first-use TOFU and carry the existing documented `gateway.remote.tlsFingerprint` contract into macOS direct node mode.
