## Summary

Keep open. Current main still has the macOS direct `wss://` first-use TOFU path, still does not carry `gateway.remote.tlsFingerprint` into the macOS node TLS params, and the report is paired with open unmerged PR #50643 rather than a shipped fix.

## What This Changes

The report asks OpenClaw to stop macOS node direct `wss://` from auto-trusting a first-seen gateway certificate and to require normal TLS trust or an explicit configured/confirmed fingerprint before node credentials and commands are exchanged.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal trust evaluation succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into the macOS direct remote session, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure macOS remote direct mode to a fresh `wss://` host with no stored keychain fingerprint, and current main sets `allowTOFU: true` then accepts and stores the presented certificate before `SecTrustEvaluateWithError`.

## Solution Assessment

Unclear until maintainers choose the supported private-TLS enrollment policy. The narrow maintainable direction is a macOS TLS enrollment change plus config pin plumbing, with security review of PR #50643 or a replacement that covers both gaps.
