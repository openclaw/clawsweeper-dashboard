## Summary

Keep open. Current main still has a source-reproducible macOS direct `wss://` first-use trust path: it computes system trust but the `allowTOFU` branch still saves and accepts the presented certificate without enforcing that result, the advertised `gateway.remote.tlsFingerprint` does not reach the macOS session builder, and the linked fixing PR is still open and unmerged at https://github.com/openclaw/openclaw/pull/50643.

## What This Changes

Harden macOS node direct `wss://` first-use TLS so rogue certificates are not pinned or accepted and configured remote TLS fingerprints can be honored.

## Best Possible Solution

Harden the macOS direct `wss://` node path so first use fails closed unless system trust succeeds or an explicit configured/confirmed fingerprint matches, plumb `gateway.remote.tlsFingerprint` into the macOS session path, and add focused Swift/config coverage for both behaviors.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: a fresh `wss://` host has no stored pin, so macOS builds `allowTOFU: true`, and the delegate saves and accepts the presented certificate without enforcing `systemTrustOk`.

## Solution Assessment

Yes, the issue’s fail-closed or explicit-fingerprint remediation is the right security boundary. A delegate-only patch is useful but incomplete unless the macOS direct config path also honors `gateway.remote.tlsFingerprint` and maintainers settle the first-use policy for private/self-signed gateways.
