## Summary

Keep open. Current main still accepts a fresh macOS direct `wss://` certificate in the TOFU branch without enforcing `systemTrustOk`, still enables that branch for a new `host:port`, and the issue has an open unmerged closing PR that needs maintainer/security review.

## What This Changes

The issue requests hardening macOS node direct `wss://` connections so first-use gateway certificates are not auto-trusted before system trust or an explicit fingerprint is enforced.

## Best Possible Solution

Harden macOS direct `wss://` node mode so first connection fails closed unless normal trust succeeds or an explicit configured/confirmed fingerprint matches, carry `gateway.remote.tlsFingerprint` into that session, and add focused Swift coverage for first-use trust and pin behavior.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure macOS direct remote mode to a fresh `wss://` host with no stored keychain fingerprint, and current main sets `allowTOFU: true` then accepts/stores the presented certificate before checking `systemTrustOk`.

## Solution Assessment

Yes for the requested hardening direction, but current main is not the solution. The narrow maintainable fix is to enforce system trust or an explicit configured/confirmed pin on first use and wire the documented pin through the macOS direct path.
