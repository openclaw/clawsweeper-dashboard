## Summary

Keep open. Current main still accepts and stores the first presented certificate for a fresh macOS direct `wss://` gateway because the TOFU branch does not require `systemTrustOk`, and the macOS direct remote path still does not carry the documented `gateway.remote.tlsFingerprint` into `GatewayTLSParams`. This is security-sensitive and already paired with open closing PR #50643, so cleanup closure would be premature.

## What This Changes

The issue asks OpenClaw to harden the macOS node direct `wss://` TLS path so first-use connections cannot pin an arbitrary untrusted certificate and configured remote TLS fingerprints are honored.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal trust evaluation succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into the macOS direct session, and add focused Swift coverage for first-use trust, pin mismatch, and configured fingerprint behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure macOS direct remote mode to a fresh `wss://` host with no stored keychain fingerprint and a self-signed or otherwise untrusted cert; current main sets `allowTOFU: true` and accepts/stores that fingerprint without requiring `systemTrustOk`.

## Solution Assessment

Unclear for the open PR and no for current main. The narrowest maintainable fix should cover both the trust gate and the documented configured-pin path; PR #50643 handles the delegate gate but does not appear to address the macOS config plumbing called out in the report.
