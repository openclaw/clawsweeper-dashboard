## Summary

Keep open. Current main still has the macOS direct `wss://` first-use trust path that accepts and stores a fresh certificate before enforcing system trust, and the macOS direct remote config still does not carry the documented TLS fingerprint into node-mode TLS params. The issue is also paired with open closing PR #50643, so it should remain open for maintainer/security review rather than cleanup closure.

## What This Changes

The issue requests hardening the macOS node direct `wss://` gateway connection so first-use TLS does not auto-trust arbitrary certificates and can use the configured remote TLS fingerprint.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal trust succeeds or an explicit configured/confirmed fingerprint matches, carry `gateway.remote.tlsFingerprint` into the macOS direct session, and add focused Swift coverage for first-use trust, pin mismatch, and configured fingerprint behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure macOS direct remote mode to a fresh `wss://` host with no stored keychain fingerprint and an untrusted certificate; current main sets `allowTOFU: true` and accepts/stores the presented fingerprint before enforcing `systemTrustOk`.

## Solution Assessment

Yes for the requested hardening direction, but current main is not the solution. The narrow maintainable fix is to gate first-use acceptance on system trust or an explicit configured/confirmed pin and to wire the documented pin into the macOS direct node path.
