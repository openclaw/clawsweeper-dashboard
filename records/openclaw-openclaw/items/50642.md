## Summary

Keep open. Current main and the latest release still contain the macOS direct `wss://` first-use TOFU path, the macOS config path still does not carry `gateway.remote.tlsFingerprint`, and the linked closing PR remains open rather than merged.

## What This Changes

Harden macOS node direct `wss://` first-use TLS handling so a fresh gateway certificate is not silently pinned and trusted, and configured remote TLS pins are honored.

## Best Possible Solution

Review, land, or replace the linked fix with a complete macOS direct `wss://` hardening change that rejects silent untrusted first-use TOFU, honors `gateway.remote.tlsFingerprint`, and adds focused Swift regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure macOS direct remote mode to a fresh `wss://` host with no stored Keychain fingerprint, and current main sets `allowTOFU: true` then accepts and stores the presented certificate before using `systemTrustOk` to reject.

## Solution Assessment

Yes for the requested hardening direction, but current main is not the solution. The maintainable fix should fail closed for silent untrusted first use, carry the existing `gateway.remote.tlsFingerprint` contract into macOS direct node mode, and preserve deliberate enrollment flows by an explicit maintainer-selected policy.
