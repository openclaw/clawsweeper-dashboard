## Summary

Keep open. Current main still accepts and stores the first presented certificate for a fresh macOS direct `wss://` gateway when no pin exists, even when system trust evaluation fails, and the macOS direct remote path still does not carry the documented `gateway.remote.tlsFingerprint` into node TLS params. The issue is also paired with open unmerged PR #50643, so it should stay open for maintainer/security review rather than cleanup closure.

## What This Changes

The issue requests hardening the macOS app node's direct `wss://` gateway connection so first-use TLS cannot trust a rogue certificate and configured remote TLS fingerprints are honored.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection never accepts an untrusted certificate merely because no pin is stored, plumb `gateway.remote.tlsFingerprint` into the macOS direct session, and add focused Swift coverage for first-use trust, pin mismatch, and configured fingerprint behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure macOS direct remote mode to a fresh `wss://` host with no stored keychain fingerprint and a self-signed or otherwise untrusted cert; current main sets `allowTOFU: true` and accepts/stores that fingerprint without requiring `systemTrustOk`.

## Solution Assessment

No for current main. Computing `SecTrustEvaluateWithError` is not sufficient because the TOFU branch ignores the result; the safer fix is to gate first-use acceptance on normal trust or an explicit configured/confirmed fingerprint and to wire the documented pin into the macOS direct path.
