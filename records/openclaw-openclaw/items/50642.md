## Summary

Keep open. Current main still allows macOS direct wss first-use TOFU before enforcing system trust, still does not plumb the documented remote TLS fingerprint into the macOS direct path, and the open closing PR is not merged.

## What This Changes

The issue reports that macOS node-mode direct wss connections accept and pin the first presented TLS certificate without system trust validation or configured fingerprint enforcement.

## Best Possible Solution

Harden the macOS node direct wss path so first connection fails closed unless normal trust evaluation succeeds or an explicit configured/confirmed fingerprint matches, wire gateway.remote.tlsFingerprint into that session, and cover first-use, pin mismatch, and configured-pin behavior with focused Swift tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: use macOS direct remote mode with a fresh wss host and no stored keychain pin; current main sets allowTOFU: true and accepts/stores the presented fingerprint before requiring systemTrustOk.

## Solution Assessment

Yes for the requested hardening direction, but current main is not the solution. The narrow maintainable fix is to gate first-use acceptance on system trust or an explicit configured/confirmed pin, not to auto-trust any first leaf certificate.
