## Label Justifications

## Summary

Close as implemented on main: current `main` and the shipped `v2026.5.22` release resolve npm updates through the package root's owning npm/prefix when that npm exists, and regression coverage exercises the mismatched PATH/global-root case that caused `/usr/local` staging.

## What This Changes

The issue reports that `openclaw update` / `gateway update.run` stages npm updates under `/usr/local` instead of the nvm-owned npm prefix, causing EACCES on nvm-only macOS installs.

## Best Possible Solution

Keep the shipped owning-prefix updater behavior; any future setup where `<prefix>/bin/npm` is absent should come back as a separate fresh repro with that exact install layout.

## Reproduction Assessment

No current-main failure path remains from source inspection: the old EACCES report is credible, but current main and `v2026.5.22` select the package root's owning npm/prefix when `<prefix>/bin/npm` exists. I did not run a live macOS nvm update replay.

## Solution Assessment

Yes. Resolving the npm command and staging root from the installed package root is the narrow maintainable fix for this nvm-prefix staging bug, while the no-owning-npm-binary edge remains a separate design question.
