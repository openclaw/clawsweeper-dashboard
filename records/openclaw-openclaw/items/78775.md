## Summary

Keep open: current main and the latest release appear to cover the reported nvm-owned npm-prefix staging path, but an open PR still references this issue with closing syntax and needs maintainer resolution before the issue is cleaned up.

## What This Changes

The issue reports that nvm-owned npm installs can have `openclaw update` or `gateway update.run` stage under `/usr/local` and fail with EACCES instead of using the owning nvm prefix.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/78778 by landing it if it still has unique value or closing it as superseded, then close this issue with current-main or release proof if no nvm repro remains.

## Reproduction Assessment

No, not as a current-main failure path from source inspection: the issue log reproduces the old EACCES behavior, but current main and `v2026.5.12` route npm resolution through the package-root prefix when `<prefix>/bin/npm` exists. I did not run a live macOS nvm update.

## Solution Assessment

Yes for the code direction: using the npm binary that owns the resolved package root is the narrow maintainable fix for this nvm-prefix staging bug. The remaining best process step is resolving the open closing PR rather than opening another repair lane.
