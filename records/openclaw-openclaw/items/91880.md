## Summary

The updated branch fixes the earlier core/memory-host resolver divergence, but it still changes a persisted workspace path that shipped in v2026.6.5 without migrating existing nested workspaces. That can make an upgraded non-default agent appear to lose its memory, skills, and workspace files, so the PR needs a maintainer-approved canonical layout and upgrade path before merge.

## What This Changes

The PR changes configured non-default agent workspace fallbacks from nested directories to hyphen-suffixed sibling directories in core and memory-host SDK, with regression tests and a reproduction script.

## Best Possible Solution

Define one canonical configured-workspace layout and either retain the shipped nested path or add a doctor-owned migration that detects, moves, and verifies existing nested workspaces before both runtime resolvers adopt the hyphenated path.

## Reproduction Assessment

Yes. Current main deterministically resolves a configured non-default workspace as `<workspace>/<agentId>`, while this PR deterministically changes both runtime copies to `<workspace>-<agentId>`.

## Solution Assessment

No, not yet. Keeping both resolvers synchronized is correct, but directly changing a shipped persistent path is not the best complete fix without either preserving that contract or supplying a doctor-owned migration and upgrade proof.
