## Summary

Keep open: current main still does not provide a gateway-backed WebChat per-message Read aloud caller, and the STT half remains a product/UX decision rather than cleanup. The current discussion and source point toward splitting the work: TTS through existing `tts.convert`, and STT through an approved Talk/transcription or `tools.media.audio` design instead of a new parallel `messages.stt` surface.

## What This Changes

Request gateway-backed WebChat TTS and STT so configured local or self-hosted voice providers are used instead of browser speech APIs.

## Best Possible Solution

Land or replace the TTS-only `tts.convert` WebChat read-aloud path, then settle STT separately around Talk transcription or a dedicated `tools.media.audio` draft-dictation UX without adding `messages.stt` unless maintainers approve that config contract.

## Reproduction Assessment

Not applicable as a strict runtime bug reproduction because this is a feature request. Source inspection is high-confidence: current main lacks a WebChat `tts.convert` read-aloud caller and a current advertised `chat.transcribeAudio`-style draft STT method.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The maintainable direction is split: reuse `tts.convert` for read-aloud, and make a separate STT recorder/transport decision that preserves expected VAD, interim transcript, and provider-routing behavior.
