## Summary

Keep open. Current main removed the stale browser speech controls but still lacks a gateway-backed per-message Read aloud control and has no WebChat draft transcription RPC/button; the active TTS work is an open draft PR and the STT half still needs a separate `tools.media.audio`-backed design.

## What This Changes

The issue requests gateway-backed WebChat TTS and STT so configured self-hosted providers are used instead of browser Web Speech APIs.

## Best Possible Solution

Land or replace the focused TTS read-aloud implementation using `tts.convert` and assistant-media, then handle STT as a separate authenticated WebChat transcription path backed by `tools.media.audio`, avoiding a new `messages.stt` surface unless maintainers explicitly choose it.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main has Talk/audio attachment pieces and `tts.convert`, but no gateway-backed assistant Read aloud control and no WebChat `chat.transcribeAudio`/server dictation method.

## Solution Assessment

Partly yes. The TTS half is best solved by the open `tts.convert` plus assistant-media approach; the STT half should reuse `tools.media.audio` behind a dedicated gateway/UI path rather than add a parallel `messages.stt` config block.
