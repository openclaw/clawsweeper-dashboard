## Label Justifications

## Summary

Keep open: current main and release v2026.6.5 still do not provide configured-provider WebChat Read aloud or a recorder-to-agent STT control; the TTS half has an open conflicting draft, while the STT half remains a distinct product/API decision.

## What This Changes

Add provider-aware WebChat voice output and input by routing Read aloud through Gateway TTS and microphone recordings through the configured transcription pipeline.

## Best Possible Solution

Ship two coherent WebChat capabilities: refresh the existing `tts.convert`-backed Read aloud work, then add an explicit recorder/upload flow that reuses `tools.media.audio` and submits the transcript through normal agent chat without creating `messages.stt`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature gap, and current-main plus v2026.6.5 source clearly show the requested controls are absent.

## Solution Assessment

Partly. Reusing `tts.convert` is the narrowest TTS solution, but the proposed `messages.stt` block is not the best STT design because `tools.media.audio` already owns transcription-provider configuration.
