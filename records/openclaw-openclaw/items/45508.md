## Label Justifications

- `P2`: Configured-provider WebChat voice is a meaningful privacy and usability improvement, but text chat, file upload transcription, and realtime Talk provide partial alternatives.
- `impact:auth-provider`: The missing WebChat controls bypass configured speech providers and normal agent-model routing for self-hosted STT/TTS setups.

## Summary

Keep open: current main removes the stale browser speech controls but still does not provide a configured-provider WebChat Read aloud action or a one-click recorder-to-agent STT flow. The TTS half has an open draft PR, while the STT half still needs maintainer direction around recorder UX and the gateway/API shape.

## What This Changes

The issue asks WebChat to use configured gateway TTS/STT providers instead of browser speech APIs for assistant Read aloud and voice input.

## Best Possible Solution

Land or replace the `tts.convert`-backed Read aloud work, then design STT as a push-to-talk/record/upload flow that reuses `tools.media.audio` instead of adding `messages.stt`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature gap. Source inspection gives high confidence that current main lacks a configured-provider Read aloud action and a batch recorder-to-agent STT flow.

## Solution Assessment

Partly. Reusing `tts.convert` is the right TTS shape, but the proposed `messages.stt` block is not the best STT path because `tools.media.audio` already owns batch transcription configuration.
