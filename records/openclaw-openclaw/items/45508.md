## Summary

Keep open: current main has removed the old browser Web Speech implementation and added adjacent Talk/inline-audio support, but it still does not provide the requested gateway-backed per-message Read aloud flow or one-shot WebChat STT path backed by `tools.media.audio`. The STT half is already tracked more narrowly by #47311, but this issue still needs maintainer scoping rather than an automatic close.

## What This Changes

Request WebChat voice controls that use OpenClaw gateway TTS/STT configuration instead of browser `speechSynthesis` and Web Speech recognition.

## Best Possible Solution

Split the outcome explicitly: keep TTS provider configuration under `messages.tts`, keep STT under `tools.media.audio`, decide whether WebChat should regain a per-message gateway Read aloud button, and route any one-shot server-side dictation work through the narrower #47311 path instead of adding `messages.stt`.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: current main has no production browser Web Speech helper and no gateway `stt.transcribe`-style method, while WebChat exposes Talk mode and audio attachment playback instead of the requested one-shot dictation/Read aloud behavior.

## Solution Assessment

No. The proposed `messages.stt` block is not the best direction because current docs and the issue discussion point to `tools.media.audio`; the maintainable path is a gateway transcription method backed by existing media-understanding plus a deliberate UI decision for TTS Read aloud.
