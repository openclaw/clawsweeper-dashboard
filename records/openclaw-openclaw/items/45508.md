## Label Justifications

- `impact:auth-provider`: The issue is about WebChat not consistently using configured local/provider TTS and STT routing, including self-hosted OpenAI-compatible endpoints.

## Summary

Keep open: current main still lacks gateway-backed WebChat Read aloud, and the prior server-side dictation path was merged then removed. Existing Talk relay/transcription support is related, but it is not a replacement for draft mic dictation or the requested configured-provider Read aloud path.

## What This Changes

Request WebChat voice controls to use configured self-hosted TTS/STT providers through Gateway-backed paths instead of browser speech APIs.

## Best Possible Solution

Land or replace the TTS-only Read aloud path through existing `tts.convert`, then settle WebChat dictation as a gateway-mediated draft STT flow or an explicit Talk transcription UX without adding a parallel `messages.stt` config surface.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection is high confidence: current main has `tts.convert` and Talk relay/transcription machinery, but WebChat lacks gateway-backed Read aloud and draft STT controls/RPC.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The safer solution is split: reuse `messages.tts` plus `tts.convert` for Read aloud, and decide STT around existing `tools.media.audio` or Talk transcription contracts.
