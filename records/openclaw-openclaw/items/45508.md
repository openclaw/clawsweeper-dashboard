## Summary

Keep this open. Current main removed the old browser Web Speech API path and added Browser Talk surfaces, but the one-shot WebChat Read aloud path is still only in an open draft PR, and the STT side still needs maintainer direction on the gateway/config contract.

## What This Changes

The issue requests gateway-backed WebChat read-aloud and voice input so configured self-hosted TTS/STT providers are used instead of browser Web Speech APIs.

## Best Possible Solution

Review and land or replace the TTS-only gateway Read aloud PR, then handle WebChat STT as a separate Talk/transcription design using an existing or explicitly approved config surface rather than adding `messages.stt` by default.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks the requested one-shot gateway Read aloud path and does not expose the proposed self-hosted STT/dictation route through `messages.stt`.

## Solution Assessment

Partially. Reusing `tts.convert` and assistant-media is the best narrow TTS solution, but the proposed `messages.stt` config shape is not clearly the best STT solution because current Talk/transcription docs point to a separate pending config/API decision.
