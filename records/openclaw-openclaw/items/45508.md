## Label Justifications

- `P2`: Configured-provider WebChat voice is a meaningful privacy and usability improvement, but text chat, file upload transcription, and realtime Talk provide partial alternatives.
- `impact:auth-provider`: The missing WebChat controls bypass configured speech providers and normal agent-model routing for self-hosted STT/TTS setups.

## Summary

Keep open: current main and v2026.6.6 still do not provide a configured-provider WebChat Read aloud action or a recorder-to-agent STT control. The TTS half has an open protected draft, while the STT half still needs a product/API decision around recorder UX and the existing `tools.media.audio` pipeline.

## What This Changes

Request WebChat voice output and input to use configured gateway TTS/STT providers, including self-hosted OpenAI-compatible audio providers, instead of browser speech APIs.

## Best Possible Solution

Ship WebChat TTS as a refreshed `tts.convert`-backed Read aloud action, then design STT as a recorder/upload/agent-send flow that reuses `tools.media.audio` instead of adding `messages.stt`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature gap. Source and release inspection show the requested Read aloud and batch recorder-to-agent controls are still absent.

## Solution Assessment

Partly. Reusing `tts.convert` is the best TTS path, but the proposed `messages.stt` config is not the best STT shape because `tools.media.audio` already owns transcription-provider configuration.
