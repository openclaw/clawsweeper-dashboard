## Summary

Keep open: current main still lacks a gateway-backed WebChat Read aloud path, and the STT side remains a recorder/config product decision rather than a drop-in browser Speech API replacement.

## What This Changes

Request to replace WebChat browser speech APIs with gateway-backed TTS playback and STT transcription for self-hosted audio providers.

## Best Possible Solution

Land or replace the TTS-only WebChat Read aloud path using existing `tts.convert` and assistant-media, then scope STT separately around `tools.media.audio` or Talk transcription with an explicit recorder UX decision.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks a WebChat `tts.convert` caller and does not expose a settled server-side STT recorder path for this request.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The maintainable direction is split: reuse existing `messages.tts` plus `tts.convert` for TTS, and decide STT around `tools.media.audio` or Talk transcription without adding a parallel config surface prematurely.
