## Summary

Keep open: current main and v2026.6.6 still do not provide the requested WebChat provider-backed voice flow. The Gateway has TTS and media-audio primitives, but WebChat still lacks a completed gateway Read aloud path and a one-click mic flow that records audio, transcribes through `tools.media.audio`, and sends the transcript to the agent.

## What This Changes

The issue requests routing WebChat Read aloud and microphone input through configured Gateway/self-hosted TTS and STT providers instead of browser speech APIs or provider-bypassing voice paths.

## Best Possible Solution

Keep this as the canonical WebChat provider-backed voice issue: land a safe Gateway `tts.convert` Read aloud path, then design the mic flow around `tools.media.audio` or an approved Talk transport without adding parallel STT config.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 have Gateway TTS/media-audio primitives but still assert no Read aloud action and no WebChat voice-input control that routes recorded audio through `tools.media.audio`.

## Solution Assessment

No for the originally proposed `messages.stt` shape. The maintainable path is to reuse existing `messages.tts`, `tts.convert`, and `tools.media.audio`, with an explicit WebChat recorder/Talk UX decision.
