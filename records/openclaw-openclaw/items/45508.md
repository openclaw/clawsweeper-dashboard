## Label Justifications

- `P2`: Configured-provider WebChat voice is a meaningful privacy and usability improvement, but text chat, file upload transcription, and realtime Talk provide partial alternatives.
- `impact:auth-provider`: The missing WebChat controls bypass configured speech providers and normal agent-model routing for self-hosted STT/TTS setups.

## Summary

Keep open: current main and latest release still do not provide a configured-provider WebChat Read aloud action or a one-click recorder-to-agent STT flow. The TTS half has an open draft implementation, and the STT half still needs maintainer product/API direction around recorder UX and reuse of the existing audio transcription pipeline.

## What This Changes

The issue requests routing WebChat read-aloud and microphone voice input through configured gateway-backed TTS/STT providers instead of browser speech APIs.

## Best Possible Solution

Ship WebChat TTS as a refreshed `tts.convert`-backed Read aloud action, then design STT as a recorder/upload/agent-send flow that reuses `tools.media.audio` instead of adding `messages.stt`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature gap. Source inspection gives high confidence that current WebChat lacks a configured-provider Read aloud action and a batch recorder-to-agent STT flow.

## Solution Assessment

Partly. Reusing `tts.convert` is the best TTS path, but the proposed `messages.stt` config is not the best STT shape because `tools.media.audio` already owns batch transcription provider configuration.
