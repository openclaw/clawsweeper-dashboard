## Summary

Keep open: current main removed the stale browser speech controls and has Talk/audio attachment support, but it still lacks the requested gateway-backed per-message Read aloud action and does not expose a current `chat.transcribeAudio` WebChat dictation RPC/UI; the open TTS PR is still draft.

## What This Changes

Request to route WebChat read-aloud and voice input through Gateway-backed TTS/STT providers instead of browser speech APIs.

## Best Possible Solution

Ship the TTS half through existing `tts.convert`, and settle the STT half on the existing `tools.media.audio`/Talk contracts rather than adding a parallel `messages.stt` config path.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main has Talk/audio playback pieces but lacks both Gateway Read aloud wiring and a current WebChat draft-transcription RPC.

## Solution Assessment

No. The request is valid, but the proposed new `messages.stt` block is not the best shape; TTS should use `tts.convert`, and STT should reuse `tools.media.audio` or the existing Talk/session contracts.
