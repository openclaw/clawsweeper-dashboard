## Summary

Keep open: current main removed the stale browser speech buttons but still does not provide gateway-backed WebChat Read aloud, and WebChat dictation remains split between an absent draft-transcription path and broader Talk transcription UX choices.

## What This Changes

The issue requests routing WebChat speech output and voice input through configured gateway-backed TTS/STT providers instead of browser Speech APIs.

## Best Possible Solution

Land or replace the TTS-only WebChat Read aloud path using existing `tts.convert`, then settle WebChat dictation around existing Talk transcription or `tools.media.audio` contracts without adding a parallel `messages.stt` surface.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current `main` still lacks WebChat Read aloud wiring to `tts.convert` and has no current composer dictation path through `chat.transcribeAudio`.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The safer solution is split: reuse `messages.tts` plus `tts.convert` for TTS, and decide STT around existing `tools.media.audio` or Talk transcription rather than adding a parallel config surface prematurely.
