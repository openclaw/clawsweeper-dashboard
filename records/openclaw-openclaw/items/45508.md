## Summary

Keep open: current main and v2026.6.6 still do not provide the full requested WebChat provider-backed voice flow. The gateway has TTS and media-audio plumbing, and WebChat now has Talk controls plus inline audio rendering, but there is no completed provider-backed Read aloud integration and no one-click mic path that captures audio and sends it through `tools.media.audio` to the agent.

## What This Changes

The issue requests WebChat voice controls that use configured gateway TTS/STT providers instead of browser speech APIs, especially self-hosted OpenAI-compatible audio services.

## Best Possible Solution

Keep this as the canonical WebChat voice-provider issue: land or replace the gateway Read aloud path for TTS, then design a mic flow around `tools.media.audio` or a supported Talk transport without adding parallel STT config.

## Reproduction Assessment

Yes, at source level. Current WebChat exposes Start Talk and lower-level gateway voice plumbing, but the requested provider-backed Read aloud and one-click mic-to-`tools.media.audio` agent flow are not complete.

## Solution Assessment

No for the originally proposed `messages.stt` block. The better path is to reuse `tools.media.audio` for batch STT and make an explicit WebChat recorder/Talk UX decision.
