## Summary

Keep open: current main and latest release still do not provide a completed WebChat provider-backed voice flow. The old browser read-aloud/dictation affordances are absent, Gateway has TTS and transcription primitives, and related PRs cover only partial or conflicting slices of the requested TTS/STT behavior.

## What This Changes

Requests WebChat voice controls that use configured Gateway/self-hosted TTS and STT providers instead of browser speech or realtime-model-only paths.

## Best Possible Solution

Keep this as the canonical WebChat provider-backed voice issue: land a safe `tts.convert` Read aloud path, then design the mic flow around `tools.media.audio` or an approved Talk transcription mode without adding parallel STT config.

## Reproduction Assessment

Yes, at source level. Current main has no Read aloud action, no WebChat `Voice input` or `chat.transcribeAudio` path, and latest release `v2026.6.8` shows the same absent-control assertions.

## Solution Assessment

No for the originally proposed `messages.stt` shape. The maintainable path is to reuse existing `messages.tts`, `tts.convert`, and `tools.media.audio`, with an explicit WebChat recorder/Talk UX decision.
