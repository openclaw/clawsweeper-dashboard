## Summary

Keep open: current main and v2026.6.6 still do not provide the requested WebChat provider-backed Read aloud action or a one-click mic-to-agent STT flow. The lower-level TTS, media-audio, and Talk RPC plumbing exists, but the UI path is still incomplete and the linked TTS PR remains open, draft, and conflicting.

## What This Changes

Route WebChat Read aloud and microphone voice input through configured gateway TTS/STT providers instead of browser or provider-owned frontend speech paths.

## Best Possible Solution

Keep this as the canonical WebChat voice-provider issue: rework or land the gateway Read aloud path for TTS, then design the mic path around `tools.media.audio` or a managed-room Talk UI without adding parallel STT config.

## Reproduction Assessment

Yes, at source level. Current WebChat exposes Start Talk and Talk settings, `rg` finds no old browser speech helpers or Read aloud action, and the only server-side STT/TTS Talk transport is rejected by the UI as unavailable.

## Solution Assessment

No, the originally proposed `messages.stt` block is not the best shape because `tools.media.audio` already owns batch STT config. The maintainable path is to wire TTS through `tts.convert` and choose a WebChat recorder/Talk UX that reuses the existing transcription pipeline.
