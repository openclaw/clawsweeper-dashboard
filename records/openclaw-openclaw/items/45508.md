## Summary

Keep open: current main still lacks gateway-backed WebChat Read aloud, the related TTS implementation remains an open draft PR, and the WebChat STT/dictation path needs maintainer reconciliation before this feature can be considered done.

## What This Changes

Route WebChat read-aloud and voice input through configured Gateway TTS/STT providers instead of browser Speech APIs.

## Best Possible Solution

Land or replace the TTS-only WebChat Read aloud path using existing `tts.convert`, then handle WebChat STT through the existing `tools.media.audio` or Talk transcription contracts after the recorder UX and current-main provenance are settled.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks WebChat `tts.convert` Read aloud wiring and a current `chat.transcribeAudio` server-dictation path.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The safer solution is split: reuse `messages.tts` plus `tts.convert` for TTS, and decide STT around existing `tools.media.audio` or Talk transcription rather than adding a parallel config surface prematurely.
