## Summary

Keep open. Current main has removed the old browser Web Speech buttons, but it still does not ship the requested gateway-backed per-message Read aloud flow; the active TTS path is the open draft PR #76027, and the broader WebChat voice UX still needs maintainer scoping.

## What This Changes

Request gateway-backed WebChat voice support: use configured `messages.tts` for Read aloud and server-side transcription for microphone input instead of browser speech APIs.

## Best Possible Solution

Review or land #76027 for gateway Read aloud if that UX is still desired, keep STT on `tools.media.audio`, and document the boundary between Talk mode, one-shot dictation, and TTS playback.

## Reproduction Assessment

Yes. Source inspection shows current main has no per-message Read aloud action and no `Voice input` dictation button, while `tts.convert` and attachment audio playback exist as separate pieces.

## Solution Assessment

No. The proposed `messages.stt` config block is not the best direction because existing docs and implementation route transcription through `tools.media.audio`; the maintainable path is to finish the active gateway Read aloud work and keep STT on the shared audio pipeline.
