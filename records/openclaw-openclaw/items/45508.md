Keep open: current main still lacks a WebChat read-aloud caller for the existing Gateway TTS RPC, and the remaining STT shape is a feature/product decision rather than cleanup closure.

## What This Changes

Request WebChat voice controls to use configured Gateway-backed self-hosted TTS/STT providers instead of browser SpeechSynthesis and Web Speech APIs.

## Best Possible Solution

Land or replace the TTS-only WebChat read-aloud path using existing `tts.convert`/assistant-media, and settle WebChat STT on existing `tools.media.audio` or Talk transcription contracts rather than adding a parallel `messages.stt` surface without maintainer approval.

## Solution Assessment
