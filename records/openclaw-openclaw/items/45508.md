## Summary

Keep open: current main removed the old browser Speech API buttons and has Talk plus inline audio attachment playback, but it still lacks the requested per-message gateway Read aloud action, and the server-side WebChat dictation path was merged and then removed from current main.

## What This Changes

The issue requests routing WebChat read-aloud and voice input through configured gateway TTS/STT providers instead of browser speech APIs, including self-hosted OpenAI-compatible audio backends.

## Best Possible Solution

Keep the browser speech removal, land or replace the gateway Read aloud work using existing `tts.convert`, and decide whether WebChat dictation should be restored through `tools.media.audio` or folded into Talk rather than adding `messages.stt`.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main lacks both the requested Read aloud action and `chat.transcribeAudio`, while the old browser speech controls are gone.

## Solution Assessment

No, not exactly as proposed. TTS should reuse `tts.convert` and STT should reuse `tools.media.audio` or Talk contracts instead of introducing a parallel `messages.stt` config block.
