## Summary

Keep open: current main and v2026.6.6 still lack a configured-provider WebChat Read aloud action and a one-click mic-to-agent STT flow, even though the old browser Speech API buttons are gone. The TTS half has an open draft PR, and the STT half still needs maintainer direction after the prior server-side dictation PR is no longer present on current main.

## What This Changes

Request WebChat voice input/output support that routes Read aloud and microphone capture through gateway-backed configured TTS/STT providers instead of browser or realtime-provider defaults.

## Best Possible Solution

Review or replace the TTS draft, and design the STT path around existing `tools.media.audio` or transcription-only Talk primitives without adding a parallel config surface unless maintainers approve it.

## Reproduction Assessment

Yes for source reproducibility: current main has no browser Speech API path, no `chat.transcribeAudio`/MediaRecorder dictation path, and tests assert the old Voice input and Read aloud buttons are absent. I did not run a live browser repro because this review is read-only.

## Solution Assessment

Unclear as a single implementation: the TTS half should reuse `tts.convert`, while the STT half should avoid a parallel `messages.stt` config and needs a maintainer decision on recorder UX and whether to restore or replace the prior dictation PR.
