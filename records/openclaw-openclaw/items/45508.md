## Summary

Keep open: current main has removed the stale browser speech buttons and has Talk/transcription plumbing, but the requested gateway-backed WebChat Read aloud path is still only in an open draft PR and the STT recorder/config shape remains a product/UX decision.

## What This Changes

Request to make WebChat voice input and Read aloud use configured gateway TTS/STT providers instead of browser Web Speech APIs, including a proposed STT config surface.

## Best Possible Solution

Land or replace the TTS-only WebChat Read aloud path using existing `tts.convert` and assistant-media, then handle STT separately through Talk transcription or `tools.media.audio` contracts after the recorder UX is decided.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks a WebChat `tts.convert` Read aloud caller and that the proposed STT path is not a drop-in replacement for browser streaming recognition.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The safer path is split: reuse existing `messages.tts` plus `tts.convert` for TTS, and decide STT around Talk transcription or `tools.media.audio` without adding a parallel config surface prematurely.
