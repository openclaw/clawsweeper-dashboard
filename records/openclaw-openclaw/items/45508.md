## Summary

Keep open: current main no longer has the old browser Speech API controls, and it has Talk plus audio attachment playback, but the requested gateway-backed per-message Read aloud action is still only in an open draft PR and current main does not expose a WebChat `chat.transcribeAudio` dictation RPC/UI.

## What This Changes

Route WebChat read-aloud and voice input through gateway-backed TTS/STT providers instead of browser speech APIs.

## Best Possible Solution

Land the TTS half through the existing `tts.convert`/assistant-media path, then settle WebChat dictation on the existing `tools.media.audio` or Talk contracts rather than adding `messages.stt`.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection of current main gives high confidence that browser speech controls are gone, but the requested gateway Read aloud action and standalone WebChat dictation RPC are not both present.

## Solution Assessment

No, not exactly as proposed. The TTS direction should use existing `tts.convert`, while STT should reuse `tools.media.audio` or current Talk/session contracts instead of introducing a parallel `messages.stt` config block.
