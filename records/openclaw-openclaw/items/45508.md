## Summary

Keep open. Current main still lacks the requested gateway-backed per-message Read aloud action, and the WebChat server-dictation path is inconsistent after it was merged and then removed while docs still describe a microphone dictation button.

## What This Changes

Route WebChat Read aloud and microphone input through configured gateway TTS/STT providers instead of browser Web Speech APIs.

## Best Possible Solution

Restore or explicitly de-scope WebChat server dictation through `tools.media.audio`, and review the gateway Read aloud PR that reuses `tts.convert`; avoid adding a parallel `messages.stt` config path.

## Reproduction Assessment

Yes. Source inspection on current main shows no per-message Read aloud action and no current `chat.transcribeAudio`/dictation UI, while docs still describe microphone dictation and the TTS RPC exists separately.

## Solution Assessment

No. The proposed `messages.stt` block is not the best direction because OpenClaw already has `tools.media.audio`; the maintainable path is to restore or clarify server dictation and finish the separate `tts.convert` Read aloud work.
