## Summary

Keep open: current main still lacks a completed provider-backed WebChat voice flow. Gateway TTS, Talk transcription, and media-audio primitives exist, but WebChat has no Read aloud action and no one-click provider-backed mic capture path; the related PRs cover partial or conflicting slices rather than a landed solution.

## What This Changes

The issue requests WebChat voice input/output that uses configured Gateway-backed STT/TTS providers instead of browser speech APIs or unrelated realtime Talk behavior.

## Best Possible Solution

Keep this as the canonical WebChat provider-backed voice thread; land or replace the TTS slice using existing `messages.tts`/`tts.convert` plus scoped media tickets, and settle a Gateway-backed STT UX that reuses `tools.media.audio` or Talk transcription rather than adding a parallel `messages.stt` surface by default.

## Reproduction Assessment

Yes, at source level: current main renders Talk and file-attachment controls, asserts no `Voice input` dictation button, and asserts no `Read aloud` action. This is not a strict bug repro because the requested provider-backed WebChat voice flow is a feature/product gap.

## Solution Assessment

No final implementation is present. The issue should not be solved by a new `messages.stt` config block by default because existing `tools.media.audio`, Talk transcription, and `messages.tts` surfaces already define the likely provider boundaries.
