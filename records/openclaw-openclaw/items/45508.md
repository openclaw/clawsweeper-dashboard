## Summary

Keep open: current main still lacks a gateway-backed per-message WebChat Read aloud action, and the open TTS implementation PR is still draft. The narrower server-side dictation work was handled through https://github.com/openclaw/openclaw/pull/76021 and https://github.com/openclaw/openclaw/issues/47311, but that does not fully satisfy the broader self-hosted WebChat STT/TTS request.

## What This Changes

Request gateway-backed WebChat voice support so Read aloud uses configured TTS and microphone input uses configured server-side STT instead of browser Speech APIs.

## Best Possible Solution

Review and land or replace the TTS PR at https://github.com/openclaw/openclaw/pull/76027, reconcile the WebChat dictation state from https://github.com/openclaw/openclaw/pull/76021, and keep STT configuration on the existing `tools.media.audio`/Talk transcription contracts rather than adding `messages.stt`.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks the requested gateway-backed per-message Read aloud path, while the browser Speech API controls are no longer the active UI surface.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The maintainable solution is split: use `tts.convert` for Read aloud and use the existing `tools.media.audio`/Talk transcription architecture for STT.
