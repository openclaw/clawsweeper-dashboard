## Summary

Keep open. Current main no longer has the old browser Web Speech buttons, and the STT half appears partly handled by the #47311 direction, but the requested gateway-backed per-message Read aloud flow is still not shipped; #76027 is an active draft rather than current main behavior.

## What This Changes

The issue requests gateway-backed WebChat TTS/STT so configured self-hosted voice providers are used instead of browser speech APIs.

## Best Possible Solution

Review or land #76027 for gateway-backed Read aloud if that UX is desired, keep STT on `tools.media.audio`, and document the boundary between Talk mode, one-shot dictation, and TTS playback.

## Reproduction Assessment

Partly. Source inspection confirms the requested per-message gateway Read aloud surface is still absent, while the original browser Web Speech buttons no longer exist on current main.

## Solution Assessment

No. The proposed `messages.stt` config block is not the best direction because current docs and implementation route transcription through `tools.media.audio`; the maintainable path is to finish the active gateway Read aloud work and keep STT on the shared audio pipeline.
