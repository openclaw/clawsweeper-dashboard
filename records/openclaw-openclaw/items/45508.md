## Summary

Keep open: current main removed the stale browser speech controls, but WebChat still lacks a gateway-backed Read aloud action and the STT/dictation half still needs maintainer sequencing around recorder UX and existing transcription contracts.

## What This Changes

The issue requests routing WebChat TTS/STT through configured gateway-backed voice providers instead of browser Web Speech APIs.

## Best Possible Solution

Land or replace the TTS-only WebChat Read aloud path using existing `tts.convert`, then settle WebChat dictation around existing Talk transcription or `tools.media.audio` contracts without adding a parallel `messages.stt` surface.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main still lacks WebChat Read aloud wiring to `tts.convert` and no current draft-dictation RPC is exposed.

## Solution Assessment

No for the exact proposed `messages.stt` shape. The safer solution is split: reuse `messages.tts` plus `tts.convert` for TTS, and decide STT around existing `tools.media.audio` or Talk transcription rather than adding a parallel config surface prematurely.
