## Summary

Keep open. Current main has removed the old browser Speech API controls, but it still lacks the requested gateway-backed WebChat Read aloud flow and a current server-side dictation path; the TTS half is covered only by an open draft PR, while the prior STT implementation was merged and then removed from main.

## What This Changes

Request gateway-backed WebChat voice controls that use configured TTS for Read aloud and self-hosted STT for mic input instead of browser speech APIs.

## Best Possible Solution

Split the work: land or replace the TTS Read aloud PR using existing `tts.convert`, and restore a maintained WebChat dictation path that reuses `tools.media.audio` or transcription-only Talk without adding `messages.stt`.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks both the gateway-backed Read aloud UI and the server-side WebChat dictation RPC/UI path.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow maintainable direction is the split proposed in the discussion: reuse `tts.convert` for TTS and existing `tools.media.audio` or transcription-only Talk for STT, rather than adding a parallel `messages.stt` config surface.
