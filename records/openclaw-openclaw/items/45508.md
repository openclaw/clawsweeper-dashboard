## Summary

Keep open: current main and latest release still lack the completed provider-backed WebChat voice flow. Gateway TTS and media-audio primitives exist, but WebChat has no Read aloud action and no one-click provider-backed mic capture path; the open related PRs cover partial or conflicting slices rather than a landed solution.

## What This Changes

The issue requests routing WebChat read-aloud and microphone input through OpenClaw's configured TTS/STT providers instead of browser-only speech APIs.

## Best Possible Solution

Keep this issue as the canonical WebChat provider-backed voice thread; land or replace the TTS PR using existing `messages.tts`/`tts.convert` plus scoped media tickets, and settle a Gateway-backed STT UX that reuses `tools.media.audio` or Talk transcription instead of adding a parallel `messages.stt` surface by default.

## Reproduction Assessment

Not applicable as a strict bug reproduction, but yes at source level for the reported gap: current main and v2026.6.10 assert no `Voice input` dictation button and no `Read aloud` action while exposing Talk and file upload controls.

## Solution Assessment

No final implementation is present. The issue should not be solved by a new `messages.stt` config block by default because existing `tools.media.audio`, Talk, and `messages.tts` surfaces already define the likely provider boundaries.
