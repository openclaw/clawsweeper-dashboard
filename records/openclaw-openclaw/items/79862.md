## Summary

Keep open for maintainer/security review. Current main and docs already support the reporter's Docker path with `request.allowPrivateNetwork: true`, while this PR broadens `hostnameAllowlist` into private-network trust, which is an SSRF policy contract change rather than a narrow bug fix.

## What This Changes

The PR changes SSRF hostname policy handling so `hostnameAllowlist` can bypass reserved hostname suffix blocks and private DNS-answer checks, with focused tests.

## Best Possible Solution

Keep private DNS and private-IP access gated by explicit `request.allowPrivateNetwork`, and only broaden `hostnameAllowlist` semantics if security owners approve and the contract is documented and tested.

## Reproduction Assessment

Yes for the no-private-opt-in path by source inspection: current main blocks `.internal` hostnames and still validates private DNS answers unless private-network policy is explicitly enabled. The original Docker use case with `request.allowPrivateNetwork: true` is no longer a high-confidence failing path because the reporter confirmed it works.

## Solution Assessment

No. The patch fixes the demonstrated no-opt-in flow by broadening `hostnameAllowlist` into private-network trust, but the narrower maintainable direction is to keep private-network access explicit unless maintainers approve a security contract change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep hostname allowlists from bypassing private DNS checks:** `src/infra/net/ssrf.ts:159-161`
  - body: Adding `hostnameAllowlist` here makes any matching provider host skip the resolved-address check in `resolvePinnedHostnameWithPolicy`. Provider fetches auto-generate that allowlist from `baseUrl` without `allowPrivateNetwork`, so this would let an allowlisted hostname DNS-rebind to RFC1918, link-local, or metadata addresses; keep this bypass gated by explicit private-network policy or an approved contract change.
  - confidence: 0.9
