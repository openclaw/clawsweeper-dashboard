## Summary

Keep open for human review. The PR addresses a real Docker/local-provider pain point, but it broadens `hostnameAllowlist` into private-network trust despite the current provider contract keeping fake-IP host scoping separate from explicit private-network opt-in.

## What This Changes

The PR changes SSRF policy handling so `hostnameAllowlist` entries bypass reserved hostname checks and resolved private-network checks, with new hostname-classifier tests.

## Best Possible Solution

Preserve explicit private-network opt-in as the trust boundary, then add focused `host.docker.internal` coverage for the documented provider config or make a maintainer-approved contract change with DNS-rebinding regressions.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR makes a matching `hostnameAllowlist` skip resolved private/internal IP checks. The original Docker failure is unclear without the effective config including `request.allowPrivateNetwork`.

## Solution Assessment

No. The patch solves the demonstrated Docker flow by changing a security-policy boundary; the narrower maintainable path is to keep private-network access tied to explicit opt-in unless maintainers approve new allowlist semantics.
