## Summary

Keep open as not merge-ready. The reporter later confirmed the documented `request.allowPrivateNetwork: true` configuration works, while this PR would broaden `hostnameAllowlist` into private-network trust and weaken the SSRF DNS-answer boundary without an approved contract change.

## What This Changes

The PR changes SSRF hostname policy handling so `hostnameAllowlist` can bypass blocked local suffixes and private DNS-address checks, with focused SSRF tests.

## Best Possible Solution

Preserve explicit private-network opt-in for private DNS and private IP access; if maintainers want `hostnameAllowlist` to imply that trust, document the new security contract and add regression tests for metadata and DNS-rebinding boundaries.

## Reproduction Assessment

Yes for the PR-introduced security regression by source inspection: matching `hostnameAllowlist` would skip resolved private/internal IP checks. The original Docker failure is no longer high-confidence because the reporter confirmed the documented `request.allowPrivateNetwork: true` config works.

## Solution Assessment

No. The patch fixes the no-opt-in Docker path by expanding `hostnameAllowlist` into private-network trust, but current main already supports the documented opt-in and the safer direction is to keep that boundary explicit unless maintainers approve a security contract change.
