## Summary

Keep open: the Docker provider failure is worth fixing, and the PR has real behavior proof, but this patch broadens `hostnameAllowlist` into a private-network DNS bypass and needs maintainer security-policy review before merge.

## What This Changes

The branch changes SSRF hostname checks and tests so `hostnameAllowlist` can bypass reserved `.internal`/`.localhost`/`.local` names and resolved private-network checks.

## Best Possible Solution

Preserve private-network DNS-answer bypass behind explicit `allowPrivateNetwork`, then add focused `host.docker.internal` coverage for the documented opt-in path or get maintainer approval for a broader allowlist contract change.

## Reproduction Assessment

Yes for the no-private-opt-in path by source inspection: current main auto-adds `hostnameAllowlist` from `baseUrl` but still blocks `.internal` before allowing that hostname. It is unclear for the linked issue's `allowPrivateNetwork` claim because the shown config omits the documented nested option and current main appears to thread that option through.

## Solution Assessment

No. The patch fixes the demonstrated Docker flow by treating `hostnameAllowlist` as private-network trust, but the narrower maintainable solution should keep resolved private-IP skips gated by explicit private-network policy or a maintainer-approved contract change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep hostname allowlists from bypassing private DNS checks:** `src/infra/net/ssrf.ts:159-161`
  - body: `hostnameAllowlist` is used as a target-host/fake-IP scope, and provider fetches auto-create it from any configured `baseUrl`.
