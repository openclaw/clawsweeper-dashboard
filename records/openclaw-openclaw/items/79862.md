## Summary

Keep this PR open for changes: the reported Docker provider flow is plausible and the contributor supplied real-run proof, but the patch weakens SSRF protection by making `hostnameAllowlist` behave like private-network trust instead of a hostname restriction.

## What This Changes

The PR changes SSRF hostname and private-network checks so policy hostname allowlists can permit `.internal`, `.localhost`, and `.local` hostnames, with regression tests for those hostname cases.

## Best Possible Solution

Preserve private-network access as an explicit trust boundary, then add focused `host.docker.internal` coverage for the documented `request.allowPrivateNetwork: true` path or get maintainer approval before changing the `hostnameAllowlist` contract.

## Reproduction Assessment

Yes for the no-private-opt-in path by source inspection: current main blocks `.internal` hostnames when only the auto-generated `hostnameAllowlist` is present and checks private DNS answers. It is unclear for the linked issue's `allowPrivateNetwork` claim because the shown config omits `request.allowPrivateNetwork`, and current main appears to pass that option through when present.

## Solution Assessment

No. The patch fixes the demonstrated Docker flow by broadening hostname allowlists into private-network bypasses, but the narrower maintainable fix should keep DNS private-IP skips gated by explicit private-network trust or a maintainer-approved contract change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep hostname allowlists from bypassing private DNS checks:** `src/infra/net/ssrf.ts:159-161`
  - body: `hostnameAllowlist` is used as a target-host restriction for public and fake-IP requests, not as private-network trust. Returning true here skips the resolved-address check in `resolvePinnedHostnameWithPolicy`, so an allowlisted provider host can DNS-rebind to RFC1918, link-local, or metadata addresses without `allowPrivateNetwork`; keep that skip gated by explicit private-network policy and add a regression for an allowlisted hostname resolving to a blocked IP.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete SSRF hardening regression by treating hostname allowlists as resolved private-IP bypasses.

Concerns:

- **[high] Hostname allowlist bypasses private DNS protection:** `src/infra/net/ssrf.ts:159`

That weakens the guard against DNS rebinding and metadata/private-network SSRF unless an explicit private-network trust flag is present.
