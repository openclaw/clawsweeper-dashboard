## Summary

Keep open for maintainer/security review because the linked Docker problem works with the documented private-network opt-in, while this branch broadens the SSRF trust boundary beyond the current docs and tests.

## What This Changes

The PR changes `src/infra/net/ssrf.ts` so `hostnameAllowlist` can bypass reserved hostname suffix blocks and resolved private-network checks, with unit tests for `.internal`, `.localhost`, and `.local` hostname allowlisting.

## Best Possible Solution

Keep private DNS and private-IP access behind explicit `request.allowPrivateNetwork` opt-in, and document/test any broader `hostnameAllowlist` semantics only after security-owner approval.

## Reproduction Assessment

Yes for the branch’s no-private-opt-in path by source inspection: current main blocks `.internal` hostnames and rechecks private DNS answers unless private-network policy is enabled. No for the original Docker failure with `request.allowPrivateNetwork: true`, because the current source and the reporter’s follow-up both indicate that configuration works.

## Solution Assessment

No. The patch solves the unconfigured Docker path by broadening `hostnameAllowlist` into private-network trust, but the narrower maintainable direction is to keep private-network access explicit unless maintainers approve a security policy change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep hostname allowlists from bypassing private DNS checks:** `src/infra/net/ssrf.ts:159-161`
  - body: Adding `hostnameAllowlist` here makes any matching provider host skip the resolved-address check in `resolvePinnedHostnameWithPolicy`. Provider fetches auto-generate that allowlist from `baseUrl` without `allowPrivateNetwork`, so this would let an allowlisted hostname DNS-rebind to RFC1918, link-local, or metadata addresses; keep this bypass gated by explicit private-network policy or an approved contract change.
  - confidence: 0.89
