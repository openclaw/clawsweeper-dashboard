## Summary

Keep open: current main and v2026.6.10 still route an isolated cron setup timeout into an immediate Gateway restart and scheduler restart-recovery pause, while the open closing PR has not merged and still needs maintainer policy review.

## What This Changes

The issue asks OpenClaw to treat isolated cron setup timeouts as cron-job failures with backoff instead of forcing a full Gateway restart that can kill in-flight work under event-loop saturation.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/96396 and either land the no-restart/backoff behavior with maintainer-approved escalation semantics, or keep this issue canonical until a chosen recovery policy lands.

## Reproduction Assessment

Yes for the source-level failure path: current code, release source, and tests show a 60-second isolated setup watchdog can trigger the Gateway restart hook and `restartRecoveryPending`. I did not run the full live systemd saturation scenario in this read-only review.

## Solution Assessment

Mostly yes: moving the default response to cron-job error/backoff is the narrowest fix for the destructive restart path. The remaining maintainer decision is whether to add a stronger corruption-class escalation predicate before merge.
