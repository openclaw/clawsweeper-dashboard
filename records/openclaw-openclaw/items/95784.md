## Summary

Keep open: current main and v2026.6.10 still route isolated cron setup timeouts into the Gateway restart/restart-recovery path, and the open closing PR is the right candidate to review rather than a reason to close early.

## What This Changes

The issue reports that isolated cron setup timeouts under event-loop saturation trigger Gateway restarts that can kill in-flight embedded work instead of failing and backing off the cron tick.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/96396 or replace it with an equivalent maintainer-approved cron-local backoff plus explicit restart-escalation policy, then close this issue after the fix merges.

## Reproduction Assessment

Yes for the source-level path: current main and v2026.6.10 still call requestSafeGatewayRestart and set restartRecoveryPending when isolated setup timeout is notified. I did not run the full live systemd saturation incident.

## Solution Assessment

Mostly yes: the open no-restart/backoff PR is the narrowest visible fix for the destructive restart trigger, but maintainers still need to confirm the policy for corruption-class setup failures and recovery escalation.
