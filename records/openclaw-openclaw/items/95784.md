## Summary

Keep open: current main and v2026.6.9 still route an isolated cron setup timeout into a Gateway restart and scheduler restart-recovery pause, while the no-restart/backoff PR that referenced this issue was closed unmerged.

## What This Changes

The issue requests changing isolated cron setup timeout handling from a full Gateway restart into tick-level failure/backoff, with saturation-aware escalation and clearer restart semantics.

## Best Possible Solution

Maintainers should replace the merged setup-timeout restart policy with a default tick-level error/backoff path, and only allow Gateway restart behind a stronger corruption signal with a nonzero drain/recovery budget.

## Reproduction Assessment

Yes for the source-level path: current main and v2026.6.9 show setup timeout invoking `requestSafeGatewayRestart` and setting `restartRecoveryPending`. I did not run the full live systemd saturation incident, so the operational reproduction remains source-backed rather than live-reproduced here.

## Solution Assessment

Unclear until maintainer policy is confirmed: the no-restart/backoff direction matches the reported saturation failure better than a process restart, but it intentionally supersedes the recently merged automatic restart policy and needs owner review.
