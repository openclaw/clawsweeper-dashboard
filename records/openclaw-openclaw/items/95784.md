## Label Justifications

- `P1`: The report describes a shipped Gateway restart loop that can kill active embedded runs and exhaust a supervised service in real operation.
- `impact:crash-loop`: The central symptom is repeated Gateway restart under systemd after cron setup timeouts during event-loop saturation.
- `impact:message-loss`: Forced restart can interrupt active embedded/channel work before replies settle.
- `impact:session-state`: The restart path marks and recovers in-flight main-session state, and the report ties failures to restart-recovery state drift.

## Summary

Keep open: current main and v2026.6.9 still route an isolated cron setup timeout into Gateway SIGUSR1 restart scheduling, and the force-after-deferral path can skip active-work drain, so the central reported behavior is not fixed or superseded.

## What This Changes

The issue asks OpenClaw to stop treating a single isolated-cron setup timeout under event-loop saturation as a process-level Gateway restart trigger that can kill in-flight work.

## Best Possible Solution

Maintainers should choose a revised availability policy for setup timeouts: cancel/back off the cron tick under saturation, and reserve Gateway restart for stronger corruption-class evidence with non-destructive drain or recovery proof.

## Reproduction Assessment

Partly: source and tests show the single setup-timeout to restart-scheduling path on current main and v2026.6.9. I did not live-reproduce sustained event-loop saturation plus systemd restart cycling in this read-only pass.

## Solution Assessment

Unclear: cancelling/backing off a saturated cron tick is plausible and probably safer for in-flight work, but reversing the recently merged automatic restart policy needs maintainer agreement on the corruption signal and recovery boundary.
