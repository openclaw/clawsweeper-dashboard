## Summary

Keep open: current main and v2026.6.10 still route isolated cron setup timeouts into a Gateway restart and scheduler restart-recovery pause. The open closing pull request https://github.com/openclaw/openclaw/pull/96396 is the right candidate path, so this issue should stay open until that PR lands or maintainers choose a different recovery policy.

## What This Changes

The issue asks OpenClaw to treat isolated cron setup timeouts as job-local failures under event-loop saturation instead of immediately restarting the Gateway and disrupting unrelated in-flight work.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/96396 with maintainer-approved escalation semantics, or close that PR and choose a different cron setup-timeout recovery policy before closing this issue.

## Reproduction Assessment

Yes for the source-level path: current main and v2026.6.10 still call requestSafeGatewayRestart and set restartRecoveryPending when an isolated setup timeout is notified. I did not run the full live systemd saturation incident in this read-only review.

## Solution Assessment

Mostly yes: the linked PR's job-local backoff direction matches the reported saturation failure better than a process restart. The remaining question is the maintainer-owned policy for when setup timeouts should escalate to Gateway restart at all.
