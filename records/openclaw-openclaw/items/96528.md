## Summary

Keep open: the patch is not merge-ready because it only lowers the lock payload limit while the contended `hold-exceeded` path still refuses removal, and it also silently caps explicit `maxHoldMs` tuning that current docs expose as an independent setting.

## What This Changes

The branch caps `acquireSessionWriteLock`'s effective `maxHoldMs` by the acquire timeout and adds tests for default, explicit-low, and explicit-high hold limits.

## Best Possible Solution

Use one coherent session-lock policy: either coordinate with the `hold-exceeded` reclaim path and prove real stale-holder recovery, or preserve explicit `maxHoldMs` tuning while changing only the default fallback with docs and upgrade proof.

## Reproduction Assessment

Yes, source-level: current main has a 60s acquire timeout, 300s default max-hold, and report-only `hold-exceeded` handling. I did not run a live gateway contention reproduction in this read-only review.

## Solution Assessment

No. The acquire layer is a plausible place to address the mismatch, but this patch only changes lock metadata and explicit tuning semantics; the safer solution must also address contender reclaim or coordinate with the existing reclaim PR.
