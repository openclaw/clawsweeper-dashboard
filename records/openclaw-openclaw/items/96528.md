## Summary

Keep open for human review: current main still has a real session-lock timing problem, but this PR is not merge-ready because it only lowers the recorded max-hold value while current reclaim policy still treats `hold-exceeded` as report-only, and it silently caps explicit high `maxHoldMs` settings.

## What This Changes

The branch caps `acquireSessionWriteLock`'s effective `maxHoldMs` by the acquire timeout and adds payload-value tests for default, explicit-low, and explicit-high values.

## Best Possible Solution

Use one coherent session-lock policy: either coordinate with the `hold-exceeded` reclaim path and prove real stale-holder recovery, or preserve explicit `maxHoldMs` tuning while changing only the default fallback with docs and upgrade proof.

## Reproduction Assessment

Yes, source-level: current main resolves default acquire timeout to 60000ms and default `maxHoldMs` to 300000ms, while `hold-exceeded` remains report-only. I did not run a live gateway contention reproduction in this read-only review.

## Solution Assessment

No. The acquire layer is a plausible place to address the mismatch, but this patch only changes lock metadata and explicit tuning semantics; the safer solution must also address contender reclaim or coordinate with the existing reclaim PR.
