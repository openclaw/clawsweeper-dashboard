## Summary

Keep open: current main and latest release still reject explicit `transport: "stdio"` in MCP config validation, while the runtime resolves command-bearing MCP servers as stdio before transport filtering. This is a narrow, source-reproducible validation/runtime consistency bug with no merged or open canonical fix found.

## What This Changes

The issue requests aligning MCP config validation with runtime behavior for command-bearing stdio MCP servers that include an explicit `transport: "stdio"`.

## Best Possible Solution

Accept explicit `stdio` on the existing MCP `transport` field for command-bearing server definitions, align the config type/docs as needed, and add focused validation plus resolver regression coverage.

## Reproduction Assessment

Yes, at source level: `openclaw config validate` reaches `OpenClawSchema.safeParse`, whose MCP server schema rejects `transport: "stdio"`, while the runtime resolves a command-bearing server as stdio before transport filtering. I did not run the CLI because this review is read-only.

## Solution Assessment

Yes. Adding `stdio` to the existing MCP transport validation/type surface, with regression coverage, is the narrowest maintainable way to make validation match the runtime’s existing command-bearing stdio behavior.
