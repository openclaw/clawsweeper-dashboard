## Summary

Keep open: current main and v2026.6.8 still reject explicit `transport: "stdio"` in MCP config validation while runtime resolution accepts command-bearing MCP servers as stdio. Open candidate PRs reference this issue, but none has merged yet, so this issue is still necessary.

## What This Changes

Align MCP config validation with runtime behavior for command-bearing MCP servers that explicitly set `transport: "stdio"`.

## Best Possible Solution

Review and land one narrow candidate PR, or an equivalent fix, that aligns the MCP schema, exported config type, and validation coverage; then close this issue after merge.

## Reproduction Assessment

Yes, at source level: `openclaw config validate` reaches `OpenClawSchema.safeParse`, whose MCP server schema rejects `transport: "stdio"`, while the runtime resolver accepts command-bearing servers as stdio. I did not run the CLI because this review is read-only.

## Solution Assessment

Yes: aligning the existing MCP config schema and exported type with runtime stdio precedence is the narrow maintainable fix. A targeted validator hint is only preferable if maintainers intentionally want `transport` to stay remote-only.
