Keep open: current main still strips hidden tagged TTS text from visible block delivery and only sends the accumulated final TTS payload when synthesis produces `mediaUrl`, so the reported synth-miss silent drop remains possible. The open related PR may repair this surface, but it is still unmerged and broader than this narrow WhatsApp fallback bug.

## What This Changes

The issue reports that hidden `[[tts:text]]...[[/tts:text]]` auto-TTS replies in final mode can be stripped from visible output and then silently dropped when synthesis returns no media URL.

## Best Possible Solution

Ensure final-mode hidden-tag TTS always produces a visible outcome: synthesized media when available, otherwise cleaned text fallback, either through the related TTS PR if maintainers accept it or a narrow non-ACP dispatcher repair if they do not.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main strips hidden `[[tts:text]]` content from visible block delivery and only sends the accumulated final TTS payload when `mediaUrl` exists. I did not run the live WhatsApp/OpenAI SecretRef scenario.

## Solution Assessment
