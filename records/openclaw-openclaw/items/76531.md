## Summary

Keep open. Current main still has the reported silent-drop path: hidden tagged block text is removed before visible delivery, then the accumulated final TTS branch only queues a reply when synthesis returns `mediaUrl`.

## What This Changes

Prevent hidden `[[tts:text]]...[[/tts:text]]` block-stream replies from disappearing when final TTS synthesis returns no media.

## Best Possible Solution

Add a narrow accumulated-block finalization fallback that sends the stripped hidden TTS body as text only when final synthesis produces no media and no visible/final reply was delivered, with regression coverage for the generic dispatcher and matching ACP behavior if affected.

## Reproduction Assessment

Yes, source-reproducible: a final-mode streamed block containing only `[[tts:text]]hello[[/tts:text]]`, with `maybeApplyTtsToPayload` returning no `mediaUrl`, reaches the block-delivery empty-content skip and then no final fallback. I did not run a live channel/provider test in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is a fallback in the accumulated-block synthetic TTS branch rather than changing provider failure semantics. It should preserve the audio-only payload when synthesis succeeds and expose stripped text only when the alternative is a silent drop.
