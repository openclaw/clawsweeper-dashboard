## Summary

Keep open. Current `main` still has the accumulated hidden-tag TTS miss path described here: the streamed hidden text is stripped from visible block delivery, synthesis failure returns a payload with no `mediaUrl`, and the dispatcher only queues the synthetic final when media exists.

## What This Changes

Prevent hidden `[[tts:text]]...[[/tts:text]]` accumulated block replies from disappearing when final-mode TTS synthesis returns no media URL.

## Best Possible Solution

Add a narrow dispatcher fallback that routes stripped accumulated TTS text when final-mode synthetic TTS returns no media, preserving media-only delivery on synth success and covering the generic and parallel ACP accumulated-block paths where applicable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: stream a hidden-only `[[tts:text]]hello[[/tts:text]]` block with final-mode tagged TTS and make `maybeApplyTtsToPayload` return without `mediaUrl`; current main strips the visible block and queues no fallback final reply.

## Solution Assessment

Yes. The narrow maintainable fix is in the accumulated-block dispatch fallback, not provider config or channel delivery: successful synthesis should remain audio-only, and synth misses should route the cleaned hidden TTS body as text.
