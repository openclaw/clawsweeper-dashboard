## Summary

Keep open. Current main has the #73758 short hidden-tag synthesis fix, but the accumulated-block final TTS path still only delivers when synthesis returns a `mediaUrl`; if synthesis misses, hidden streamed text has already been stripped and no fallback is queued.

## What This Changes

The issue requests that hidden `[[tts:text]]...[[/tts:text]]` auto-TTS replies fall back to stripped text delivery when final-mode accumulated-block synthesis returns no media URL.

## Best Possible Solution

Add a narrow accumulated-block finalization fallback that sends stripped hidden TTS body as text only when final synthetic TTS produces no media and no visible/final reply was delivered, with regression coverage for the generic dispatcher and any matching ACP path.

## Reproduction Assessment

Yes, source-reproducible: a final-mode streamed block containing only `[[tts:text]]hello[[/tts:text]]`, with `maybeApplyTtsToPayload` returning no `mediaUrl`, reaches an early block-delivery skip and then no final fallback. I did not run a live WhatsApp/provider test in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is a fallback in the accumulated-block synthetic TTS branch rather than changing provider failure semantics. It should preserve the audio-only payload when synthesis succeeds and only expose stripped text when the alternative is a silent drop.
