## Summary

Keep open: current main still strips hidden tagged TTS block text from visible delivery and only sends the accumulated final TTS payload when synthesis returns `mediaUrl`, so the reported synth-miss silent drop remains possible. The related TTS PR is a plausible candidate repair but is still open and broader than this narrow fallback bug.

## What This Changes

Bug report that hidden tagged final-mode TTS block replies should fall back to cleaned text instead of disappearing when synthesis produces no media URL.

## Best Possible Solution

Ship a focused dispatcher fix, or land the existing TTS PR after maintainers accept its broader caption capability, so hidden tagged TTS always delivers synthesized audio or cleaned fallback text.

## Reproduction Assessment

Yes for source-level reproduction: current main strips hidden `[[tts:text]]` content from visible block delivery and only sends the accumulated final TTS payload when `mediaUrl` exists. I did not run the live WhatsApp/OpenAI SecretRef scenario, so this is not marked as fully reproduced.

## Solution Assessment

No, current main is not the solved state because the no-media branch still has no fallback. The narrow maintainable fix is to route cleaned fallback text from the accumulated hidden-tag body when final synthesis returns no media, unless maintainers choose to land the broader related TTS PR.
