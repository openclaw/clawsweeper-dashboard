## Summary

Keep open. Current main still strips hidden streamed TTS text before block delivery, then only sends the accumulated final TTS payload when synthesis returns a `mediaUrl`, so a no-media synth result can still silently drop the reply.

## What This Changes

Hidden tagged TTS block replies should fall back to stripped text instead of disappearing when final TTS synthesis returns no media.

## Best Possible Solution

Add a narrow accumulated-block finalization fallback that sends stripped hidden TTS body as text only when final synthetic TTS produces no media and no visible or final reply was delivered.

## Reproduction Assessment

Yes, source-reproducible. A final-mode streamed block containing only hidden `[[tts:text]]hello[[/tts:text]]` can be stripped before block delivery, then the final synthetic TTS branch exits when no `mediaUrl` is returned.

## Solution Assessment

Yes. The narrow maintainable fix is a fallback in the accumulated-block synthetic TTS branch, not changing provider failure semantics or making hidden tags globally visible.

## Repair Work Prompt

Fix the source-reproducible hidden tagged TTS synth-miss drop. In final-mode streamed block delivery with `messages.tts.auto = tagged`, a hidden-only `[[tts:text]]...[[/tts:text]]` block is stripped from visible block delivery, accumulated, and passed to final TTS synthesis; when `maybeApplyTtsToReplyPayload` returns without `mediaUrl`, current `dispatch-from-config` sends nothing. Add a narrow fallback in the accumulated-block finalization path that derives stripped fallback text from the accumulated hidden TTS body and routes or queues it only when no final reply and no visible delivery already covered the user-visible response. Preserve the media-only payload when synthesis succeeds, avoid duplicate text for mixed visible-plus-hidden streamed blocks, and check the sibling ACP accumulated-block final TTS path for the same media-only gap. Add regression tests for hidden-only synth miss and mixed visible-plus-hidden behavior. Add a changelog entry because this is user-visible reply delivery behavior.
