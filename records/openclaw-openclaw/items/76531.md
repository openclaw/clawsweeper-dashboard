## Summary

Keep open. Current main still has the reported silent-drop path: hidden streamed TTS text is stripped before block delivery, and the accumulated final TTS branch only sends anything when synthesis returns a `mediaUrl`.

## What This Changes

Prevent hidden `[[tts:text]]...[[/tts:text]]` accumulated final-mode TTS replies from disappearing when synthesis returns no media URL.

## Best Possible Solution

Add a narrow accumulated-block finalization fallback that sends stripped hidden TTS body as text only when no media, visible block, or final reply was delivered, with generic dispatcher coverage and ACP parity if applicable.

## Reproduction Assessment

Yes, source-reproducible: a final-mode streamed block containing only `[[tts:text]]hello[[/tts:text]]`, with `maybeApplyTtsToPayload` returning no `mediaUrl`, is stripped before block delivery and then skipped by the final synthetic TTS branch. I did not run live WhatsApp or provider synthesis in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is in the accumulated-block dispatcher finalization path, not in provider failure semantics. It should preserve audio-only delivery when synthesis succeeds and only send stripped text when the current behavior would silently drop the reply.
