## Summary

Keep open: current main and the latest release still have a source-visible hidden tagged TTS drop when final-mode synthesis returns no media, and the overlapping open fix PR has not merged yet.

## What This Changes

The issue reports that hidden `[[tts:text]]...[[/tts:text]]` final-mode TTS block replies can disappear instead of falling back to text when synthesis misses.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83988 if maintainers accept the broader TTS delivery change, or land a narrower equivalent that strips hidden TTS wrappers and sends text fallback whenever the accumulated final-mode TTS media is not delivered.

## Reproduction Assessment

Yes. Source inspection on current main shows a hidden-only tagged block is stripped from visible delivery, then the accumulated final-mode TTS branch only sends media and has no fallback when synthesis returns no `mediaUrl`.

## Solution Assessment

No, current main is not the best end state because it still drops the fallback text. The best fix is to keep the existing successful voice path and add a stripped text fallback when synthesized media is missing or fails to deliver.
