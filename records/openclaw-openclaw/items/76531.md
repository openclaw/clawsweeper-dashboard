## Summary

Keep open. Current main still strips hidden streamed TTS text before block delivery and the accumulated final TTS branch only sends when synthesis returns a media URL, so a synth miss can still silently drop the reply.

## What This Changes

The issue requests a fallback for hidden `[[tts:text]]...[[/tts:text]]` block replies when accumulated final TTS synthesis returns no media.

## Best Possible Solution

Add a narrow accumulated-block finalization fallback that sends stripped hidden TTS body as text only when final synthetic TTS produces no media and no visible/final reply was delivered.

## Reproduction Assessment

Yes, source-reproducible: a final-mode streamed block containing only hidden `[[tts:text]]hello[[/tts:text]]` can be stripped before block delivery, then the final synthetic TTS branch exits when no `mediaUrl` is returned. I did not run a live WhatsApp/provider test in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is a fallback in the accumulated-block synthetic TTS branch rather than changing provider failure semantics or making hidden tags globally visible.
