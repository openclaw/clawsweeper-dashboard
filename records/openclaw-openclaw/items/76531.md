## Summary

Keep this open. Current main and the latest release still have the reported accumulated hidden-tag TTS branch: hidden text is stripped from block delivery, then the synthetic final TTS path only routes a reply when synthesis returns `mediaUrl`, leaving no text fallback on a synth miss. The related closed issue https://github.com/openclaw/openclaw/issues/73758 fixed the short-tag synthesis guard, and open PR https://github.com/openclaw/openclaw/pull/83988 contains a broader fallback direction, but that PR is not merged.

## What This Changes

The issue asks OpenClaw to deliver stripped fallback text for hidden `[[tts:text]]` accumulated block replies when final-mode TTS synthesis returns no media.

## Best Possible Solution

Add a focused dispatcher fallback that delivers stripped accumulated text whenever hidden tagged final-mode TTS produces no media, with regression tests for both synthesized media success and synth-miss fallback.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: emit a hidden-only `[[tts:text]]...[[/tts:text]]` streamed block, have final-mode synthesis return no `mediaUrl`, and the current branch has no text fallback to route. I did not run a live WhatsApp/Gateway repro in this read-only review.

## Solution Assessment

Yes, the narrow fallback proposed by the issue is the best repair shape for this specific bug. The broader open PR may solve it too, but this issue should remain open until that fallback lands or maintainers intentionally choose the broader captioned-final path.
