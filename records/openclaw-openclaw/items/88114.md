## Label Justifications

- `P2`: This is a bounded Windows package-update reliability bug with limited platform blast radius but real availability impact.
- `merge-risk: 🚨 compatibility`: The patch changes existing Windows `gateway stop` semantics by persistently disabling Scheduled Task autostart.
- `merge-risk: 🚨 availability`: A persistently disabled Scheduled Task can leave the gateway unavailable after logon or reboot if restart or recovery does not run.

## Summary

Keep open: this addresses a valid Windows update race, but the submitted patch changes generic Windows gateway stop semantics and still lacks after-fix real behavior proof. The narrower companion implementation is promising but remains draft/conflicting, so this PR is not safely superseded yet.

## What This Changes

Adds Windows Scheduled Task disable/enable calls around the existing stop and restart task lifecycle functions.

## Best Possible Solution

Land an update-scoped suspend/resume guard around mutable package replacement, restore only tasks suspended by that update invocation, and keep ordinary Windows gateway stop semantics intact.

## Reproduction Assessment

Yes, source-backed. Current main stops the managed Windows Scheduled Task before package replacement using `/End` only, and the linked issue describes legacy PT3M tasks that can fire during that window; I did not run a native Windows update in this read-only review.

## Solution Assessment

No. Disabling inside generic `stopScheduledTask` is broader than the bug requires; the narrower maintainable fix is update-scoped suspension and restoration around package replacement.

- **[P1] Keep persistent disable out of generic stop:** `src/daemon/schtasks.ts:1324`
  - body: This call runs for every Windows gateway stop, not only the update flow. Move this guard into the package-update window and restore only the task that update suspended.
