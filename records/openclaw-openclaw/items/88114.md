## Label Justifications

- `P2`: This is a bounded Windows package-update reliability bug with limited platform blast radius but real availability impact.
- `merge-risk: 🚨 compatibility`: The patch changes existing Windows `gateway stop` behavior by persistently disabling Scheduled Task autostart.

## Summary

Keep open: the Windows package-update race is still real, but this branch is not merge-ready because it fixes the update window by changing generic Windows stop/restart semantics and it still lacks real Windows after-fix proof. The safer update-scoped companion at https://github.com/openclaw/openclaw/pull/88292 is relevant, but it is draft/conflicting, so it is not a safe superseding close target yet.

## What This Changes

Adds `/Change /DISABLE` after Windows Scheduled Task stop and `/Change /ENABLE` before Scheduled Task restart in `src/daemon/schtasks.ts`.

## Best Possible Solution

Land an update-scoped suspend/resume guard around mutable package replacement, restore only tasks suspended by that update invocation, and preserve ordinary Windows gateway stop semantics.

## Reproduction Assessment

Yes, source-backed. Current main stops the managed Windows Scheduled Task before package replacement using `/End` only, and the canonical issue describes legacy PT3M tasks that can fire during that window; I did not run a native Windows update in this read-only review.

## Solution Assessment

No. Disabling inside generic `stopScheduledTask` is broader than the bug requires; the narrower maintainable fix is update-scoped suspension and restoration around package replacement.
