## Summary

Keep open: this PR targets a real Windows update race, but the branch is not merge-ready because it changes generic Scheduled Task stop/restart semantics and still lacks contributor real-behavior proof. The companion update-scoped PR is relevant, but it is draft/dirty, so it is not a safe superseding close target yet.

## What This Changes

The branch adds `/Change /DISABLE` after `stopScheduledTask()` ends a Windows Scheduled Task and `/Change /ENABLE` before `restartScheduledTask()` runs it again.

## Best Possible Solution

Land an update-scoped suspend/resume guard around mutable package replacement, restoring only tasks suspended by that update invocation while preserving ordinary Windows gateway stop semantics.

## Reproduction Assessment

Yes, source-reproducible. Current main stops the managed Windows service before package replacement and `stopScheduledTask()` uses `schtasks /End` without disabling already-registered repeated triggers; I did not run a native Windows update in this read-only review.

## Solution Assessment

No. Disabling inside generic `stopScheduledTask()` is broader than the bug requires; the narrower maintainable fix is update-scoped suspension and restoration around package replacement.
