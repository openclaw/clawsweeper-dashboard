## Label Justifications

- `merge-risk: 🚨 compatibility`: The patch changes the semantics of existing Windows `gateway stop` by persistently disabling task autostart.

## Summary

Keep open: the PR addresses a real Windows update race, but the patch changes generic Windows `gateway stop` semantics instead of guarding only the package-update window, and the narrower related PR is still draft/conflicting rather than a clean supersession target.

## What This Changes

The PR adds `/DISABLE` after Windows Scheduled Task stop and `/ENABLE` before scheduled-task restart in `src/daemon/schtasks.ts`.

## Best Possible Solution

Keep generic Windows stop semantics intact and implement a package-update-scoped suspend/resume guard that restores only the task disabled by that update invocation.

## Reproduction Assessment

Yes, source-reproducible: current main stops the Windows Scheduled Task before package replacement with `/End` only, and the linked issue documents legacy PT3M tasks that can fire during that window. I did not run a native Windows package update in this read-only PR pass.

## Solution Assessment

No. Disabling in generic `stopScheduledTask` is broader than the bug requires; the maintainable fix is update-scoped suspension and restoration around package replacement.

- **[P1] Keep persistent disable out of generic stop:** `src/daemon/schtasks.ts:1324`
  - body: This line runs for every Windows `openclaw gateway stop`, not only the update flow.
