## Label Justifications

- `P2`: This is a bounded Windows package-update reliability fix with limited platform blast radius but real availability impact.
- `merge-risk: 🚨 compatibility`: The patch changes existing Windows `gateway stop` semantics by persistently disabling Scheduled Task autostart.
- `merge-risk: 🚨 availability`: A persistently disabled Scheduled Task can leave the gateway unavailable after logon or reboot if restart or recovery does not run.

## Summary

Keep open: this PR targets a real Windows package-update race, but the patch disables the Scheduled Task in the generic stop path, which changes ordinary Windows gateway stop semantics and can leave logon autostart disabled until a later restart succeeds.

## What This Changes

Adds Windows Scheduled Task `/DISABLE` after generic stop and `/ENABLE` before generic restart in `src/daemon/schtasks.ts`.

## Best Possible Solution

Land an update-scoped suspend/resume guard around mutable package replacement, restore only tasks suspended by that update invocation, and keep generic Windows gateway stop semantics intact.

## Reproduction Assessment

Yes, source-backed. Current main calls the managed service stop before package replacement, and the Windows stop implementation ends the running Scheduled Task without disabling an already-registered repeated trigger.

## Solution Assessment

No. Disabling inside generic `stopScheduledTask` is broader than the bug requires; the maintainable fix is update-scoped suspension and restoration around package replacement.
