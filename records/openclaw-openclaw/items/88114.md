## Label Justifications

- `P2`: This is a bounded Windows package-update reliability fix with limited platform blast radius but real availability impact.
- `merge-risk: 🚨 compatibility`: The patch changes existing Windows `gateway stop` semantics by persistently disabling task autostart.
- `merge-risk: 🚨 availability`: A disabled Scheduled Task can leave the gateway unavailable after logon or reboot if restart/recovery does not run.

## Summary

Keep open: current main and v2026.6.6 still need the Windows update race fixed, but this PR is not merge-ready because it makes generic Windows Scheduled Task stop persistently disable autostart and has no after-fix native Windows proof. The narrower implementation path in https://github.com/openclaw/openclaw/pull/88292 is promising but still draft/conflicting, so it is not a safe supersession close target yet.

## What This Changes

The PR adds `/DISABLE` after Windows Scheduled Task stop and `/ENABLE` before scheduled-task restart in `src/daemon/schtasks.ts`.

## Best Possible Solution

Keep generic Windows stop semantics intact and land an update-scoped suspend/resume guard that restores only the task disabled by the same update invocation.

## Reproduction Assessment

Yes. Current main stops the managed Windows Scheduled Task before package replacement using `/End` only, and the linked report describes legacy PT3M tasks that can fire during that window; I did not run a native Windows update in this read-only review.

## Solution Assessment

No. Disabling inside generic `stopScheduledTask` is broader than the bug requires; the narrower maintainable fix is update-scoped suspension and restoration around package replacement.

- **[P1] Keep persistent disable out of generic stop:** `src/daemon/schtasks.ts:1324`
  - body: This added call runs for every Windows `openclaw gateway stop`, not only the update flow.
