## Label Justifications

- `P2`: This is a bounded Windows package-update reliability bug with limited platform blast radius but real availability impact.
- `merge-risk: 🚨 compatibility`: The patch changes existing Windows gateway stop semantics by persistently disabling Scheduled Task autostart.
- `merge-risk: 🚨 availability`: A persistently disabled Scheduled Task can leave the gateway unavailable after logon or reboot if restart or recovery does not run.

## Summary

Keep open, but this branch is not merge-ready: it addresses a real Windows update race by changing generic Scheduled Task stop/restart semantics, which can persistently disable normal Windows gateway autostart outside the update window.

## What This Changes

The PR adds `/Change /DISABLE` after Windows Scheduled Task stop and `/Change /ENABLE` before scheduled-task restart in `src/daemon/schtasks.ts`.

## Best Possible Solution

Land an update-scoped suspend/resume guard around package replacement, restore only tasks suspended by that update invocation, and preserve normal Windows gateway stop semantics.

## Reproduction Assessment

Yes, source-backed. Current main stops the managed Windows Scheduled Task before package replacement using `/End` only, and the linked report describes legacy repeated tasks that can fire during that window.

## Solution Assessment

No. Disabling inside generic `stopScheduledTask` is broader than the bug requires; the narrower maintainable fix is update-scoped suspension and restoration around package replacement.

- **[P1] Keep persistent disable out of generic stop:** `src/daemon/schtasks.ts:1324`
  - body: This call runs for every Windows gateway stop, not only the update flow. Move this guard into the package-update window and restore only the task that update suspended.
