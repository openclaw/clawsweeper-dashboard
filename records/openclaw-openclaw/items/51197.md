## Summary

Keep open. Current main still has the gateway-lock post-open write failure leak, while the plugin SDK half is already fixed and the docs-i18n hunk is stale plus functionally incomplete, so this PR is useful source work but not mergeable as-is.

## What This Changes

This PR adds cleanup around lock-file writes in the gateway and plugin SDK file-lock helpers and adds partial pipe cleanup to the old docs-i18n Pi RPC client startup path.

## Best Possible Solution

Port only the still-missing gateway-lock cleanup onto current main, add a focused gateway regression test for fs.open success followed by handle.writeFile failure, keep the shipped plugin SDK implementation, and avoid reviving the removed docs-i18n RPC client.

## Reproduction Assessment

Yes. Mock fs.open in acquireGatewayLock to return a FileHandle-like object whose writeFile rejects; current main will throw GatewayLockError without calling close or removing the partial lock because cleanup exists only in the returned release closure.

## Solution Assessment

No. The cleanup pattern is the right shape for the remaining gateway bug, but this PR duplicates already-shipped plugin SDK work and carries an obsolete Go script hunk with a real stdout pipe cleanup defect.
