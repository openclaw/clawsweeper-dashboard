## Summary

Keep open: current main still has the gateway-lock write-failure leak, but this PR is not merge-ready because the plugin SDK part is now handled by the fs-safe path, the docs-i18n Pi RPC client hunk is obsolete and contains a real stdout pipe leak, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR wraps gateway/plugin lock owner-payload writes with cleanup on failure and adds pipe cleanup to the old docs-i18n Pi RPC client.

## Best Possible Solution

Port the gateway-lock write-failure cleanup onto current main with focused regression coverage, keep the fs-safe-backed plugin SDK implementation, and drop the obsolete docs-i18n Pi client hunk.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mock fs.open in acquireGatewayLock to return a FileHandle-like object whose writeFile rejects, and current main throws before any close or partial-lock cleanup path is reachable.

## Solution Assessment

No. The cleanup pattern is the right fix for the remaining gateway bug, but this PR is not the best merge unit because one hunk is obsolete on main and another obsolete hunk introduces a new pipe cleanup leak.
