## Summary

Keep open. Current main still has the gateway lock post-open write failure path, but the plugin SDK file-lock half is already fixed on main and in v2026.4.27; the PR also carries a stale docs-i18n RPC client change with an unresolved stdout pipe cleanup bug from review context.

## What This Changes

The PR wraps post-open lock payload writes with cleanup in the gateway and plugin SDK file-lock helpers, and adds partial pipe cleanup to the old docs-i18n RPC client startup path.

## Best Possible Solution

Port only the still-missing gateway-lock cleanup onto current main, add a focused gateway regression test for fs.open success followed by handle.writeFile failure, keep the shipped plugin SDK file-lock implementation, and drop the obsolete docs-i18n RPC client change.

## Reproduction Assessment

Yes. The bug is reproducible by mocking fs.open to return a FileHandle-like object and then making handle.writeFile reject in acquireGatewayLock; current main has no inner catch on that path, unlike the already-covered plugin SDK file-lock helper.

## Solution Assessment

No, not as the PR currently stands. The cleanup pattern is the right shape for the remaining gateway bug, but the branch is stale because one TS hunk is already shipped and the old Go script hunk has a real pipe cleanup defect.
