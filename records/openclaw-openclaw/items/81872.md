## What This Changes

The PR adds bundled-plugin middleware contracts and PI runner/tool-path wiring for streaming LLM calls and tool-call execution, plus SDK/manifest registration support and focused tests.

## Best Possible Solution

Approve this seam only after middleware lookup is active/request-scoped, effective tool params are tracked consistently, SDK docs and API baseline are aligned, and a real bundled-plugin PI run demonstrates the behavior.

## Reproduction Assessment

No live reproduction was run. Source inspection is high-confidence for the review findings because the branch helpers read global registry state while current main can preserve a different global registry, and the tool-call wrapper permits nextParams while later tracking outcome.params.

## Solution Assessment

No. The generic plugin seam fits the owner boundary, but this implementation should use active/request-scoped registry state, preserve effective execution facts, and complete SDK contract follow-through before merge.

  - body: Both new middleware helpers enumerate registrations from `getGlobalPluginRegistry()`, but current main can preserve the gateway global registry while installing a different active registry. That makes audit and loop detection record different params from the actual tool invocation when middleware rewrites arguments.
  - confidence: 0.78

  - confidence: 0.88
