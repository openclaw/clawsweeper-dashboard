## Summary

Keep open: this draft feature PR has blocking registry-scope and SDK baseline issues, and the supplied proof is test-only for a new LLM/tool execution interception surface.

## What This Changes

Adds PI agent streaming LLM and tool-call middleware contracts for bundled plugins, wires them into PI runtime paths, exposes SDK types/API methods, and adds focused middleware registration tests.

## Best Possible Solution

Keep the seam draft until runtime lookup uses the active or explicitly passed registry, SDK baseline/docs are aligned, and maintainers approve the bundled-only LLM/tool interception contract with live redacted proof.

## Reproduction Assessment

No high-confidence runtime reproduction was run. Source inspection gives a clear path: after a gateway-bindable load is preserved and a default registry is activated, these helpers still read the preserved global registry while the existing middleware precedent reads the active registry.

## Solution Assessment

No. The generic seam direction fits the plugin boundary, but the implementation should use active/request-scoped registry state or an explicit registry handle before exposing tool-call and LLM-stream interception.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Use the active registry for middleware lookup:** `src/plugins/agent-tool-call-middleware.ts:61-62`
  - body: These helpers build the runtime middleware list from `getGlobalPluginRegistry()`, but current loader behavior can preserve a gateway registry while swapping in a different active registry.
