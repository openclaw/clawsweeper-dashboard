## Summary

Keep this draft PR open: the middleware seam is plausible, but the current diff has a registry-scope bug that can make middleware run from the preserved gateway registry instead of the active runtime registry, and the external PR proof is only focused test output rather than a real agent/plugin run.

## What This Changes

The PR adds bundled-plugin PI middleware contracts for streaming LLM calls and tool calls, wires them into the PI runner/tool path, exposes SDK types/API methods, and adds focused tests.

## Best Possible Solution

Keep the PR as a draft until middleware lookup follows the active/request-scoped runtime registry, the SDK/docs/API contract surface is deliberately approved, and a live agent run with a real bundled plugin middleware is supplied.

## Reproduction Assessment

No high-confidence runtime reproduction was run in this read-only review. Source inspection shows the likely failure path: loader can preserve a global gateway registry while setting a different active registry, and the new middleware helpers read the global registry.

## Solution Assessment

No. The seam direction is reasonable, but the implementation should use the active/request-scoped plugin runtime registry, or explicitly pass the intended registry, before exposing tool-call and LLM-stream interception.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Use the active registry for middleware lookup:** `src/plugins/agent-tool-call-middleware.ts:61-62`
  - body: These wrappers read middleware from `getGlobalPluginRegistry()`, but the loader can preserve the gateway hook runner while replacing the active plugin registry. In that state, middleware registered in the current active registry will be skipped, and stale gateway middleware can still wrap PI LLM/tool calls. Follow the existing tool-result middleware pattern by using the active runtime registry or by passing the intended registry explicitly.
  - confidence: 0.82
