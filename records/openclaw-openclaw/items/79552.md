## Label Justifications

- `P1`: Android notification forwarding can permanently drop automation-triggering events during reconnect for real users.
- `impact:message-loss`: The reported failure drops notifications.changed node events before they reach the gateway and agent automation path.

## Summary

Keep open: current main and v2026.6.10 still expose the Android node connection before the gateway connect RPC completes, and notification events still go through a no-retry send path. The linked fix candidate at https://github.com/openclaw/openclaw/pull/92602 is open but currently conflicting, so closing this issue now would hide a current message-loss bug.

## What This Changes

The issue requests fixing Android notification forwarding so node.event frames wait for gateway connect and reconnect-window notifications are queued or replayed instead of being lost.

## Best Possible Solution

Review, repair, or replace the linked Android PR, then close this issue only after a fix reaches main with regression proof for the handshake and reconnect notification paths.

## Reproduction Assessment

Yes, at source level: current main publishes currentConnection before the connect RPC finishes, Android notification forwarding sends through that readiness gate without retry, and the gateway rejects non-connect first requests. I did not run live Android or Robolectric proof in this read-only review.

## Solution Assessment

Yes, the best fix direction is to make currentConnection represent only a ready gateway socket, then add bounded notification retry and reconnect reconciliation. The current issue should stay open until the linked PR lands or maintainers choose a replacement.
