## Summary

Keep open: current main still exposes the Android node connection before the connect RPC completes and notification forwarding still drops failed node.event sends. The two linked fix PRs are closed and unmerged, so this issue remains the canonical work item.

## What This Changes

The issue reports Android notification node.event frames being sent during a reconnect before the gateway WebSocket connect handshake completes, causing notification wake events to be lost.

## Best Possible Solution

Implement an Android-side fix that only sends node.event after authenticated connect is complete and preserves notifications that arrive during reconnect with bounded replay or reconciliation, without changing the gateway protocol.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: currentConnection is visible before conn.connect() finishes, and notification forwarding can call sendNodeEvent during that pre-handshake window. I did not run a live Pixel/Gateway reconnect scenario in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is Android-side readiness plus bounded notification retry/reconciliation; loosening the gateway first-frame connect contract would be the wrong layer.

## Repair Work Prompt

Fix the Android reconnect notification delivery bug reported in https://github.com/openclaw/openclaw/issues/79552. On current main, GatewaySession.connectOnce publishes currentConnection before conn.connect() completes, while NodeRuntime forwards DeviceNotificationListenerService notifications directly through sendNodeEvent and ignores false/failed sends. Keep the gateway WebSocket contract unchanged: first frame must remain connect. Implement an Android-side readiness guard or connection publication ordering so node.event cannot be sent before authenticated connect completes, and add bounded retry/queueing or active-notification reconciliation so notifications that arrive during reconnect are delivered after reconnect instead of being dropped. Use the closed PRs as reference only: https://github.com/openclaw/openclaw/pull/79565 had the fuller queueing/replay direction; https://github.com/openclaw/openclaw/pull/79927 only guarded sends and left notification loss unresolved.
