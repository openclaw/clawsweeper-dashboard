## Summary

Keep open: current main still exposes a connecting Android GatewaySession before the connect RPC completes, while the Gateway contract requires the first WebSocket request to be connect; notification forwarding can therefore still lose node.event frames during reconnect.

## What This Changes

Ensure Android notification node.event frames wait for authenticated WebSocket connect completion and survive reconnect via queue, retry, or reconciliation.

## Best Possible Solution

Fix Android-side connection readiness and bounded notification replay so node.event frames are sent only after authenticated connect completes, while keeping the Gateway protocol unchanged.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: connectOnce stores currentConnection before conn.connect() completes, and the notification sink can call sendNodeEvent in that window. I did not run a live Pixel reconnect scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is Android-side: guard or delay node.event until connect completion and add bounded notification retry/reconciliation on reconnect. A Gateway protocol change does not look necessary.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79552 on current main. The observable bug is that Android notification forwarding can call node.event during the WebSocket connect handshake because GatewaySession.currentConnection is set before conn.connect() finishes; the Gateway then rejects node.event as the first frame and the notification is lost. Keep the fix Android-owned and do not change the Gateway protocol or add config. Ensure node.event cannot be sent before authenticated connect completes, add a bounded retry/queue/reconciliation path for notifications.changed events that arrive during reconnect, preserve existing notification filters/rate limits/sessionKey handling, and avoid duplicate heartbeat wakes for the same notification key. Add focused Android unit/Robolectric coverage for pre-connect node.event suppression/flush and notification reconnect replay; update CHANGELOG.md under the active Fixes section because this is user-facing. Stop if the implementation would require a new protocol method, new config option, or product-policy decision.
