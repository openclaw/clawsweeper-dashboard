## Summary

Keep open: current `main` still has the Android reconnect race and the related fix PRs were closed without merging. The reporter’s local patch comment is useful corroboration, but the source tree still lacks both a handshake-ready guard and bounded notification replay/reconciliation.

## What This Changes

Fix Android reconnect handling so `node.event` is not sent before the WebSocket `connect` handshake completes and notification events are queued or replayed after reconnect.

## Best Possible Solution

Fix this on the Android side by marking/publishing the connection only after authenticated connect readiness and adding bounded notification retry or active-notification reconciliation after reconnect, without changing the Gateway protocol.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `connectOnce` exposes `currentConnection` before `conn.connect()` completes, and notification forwarding can call `sendNodeEvent` in that window while the gateway requires the first frame to be `connect`.

## Solution Assessment

Yes. The narrow maintainable fix is Android-side readiness plus bounded retry/reconciliation for notification events; a gateway protocol change is not needed.
