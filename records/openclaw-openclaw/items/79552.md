## Summary

Keep open: current main still exposes a connecting Android GatewaySession before the connect RPC completes, and notification forwarding still sends directly through that session with no visible retry/replay path for events that fail during the pre-handshake window.

## What This Changes

The issue requests Android notification node events to wait for the websocket connect handshake, retry or queue during reconnect, and reconcile missed posted notifications after reconnect.

## Best Possible Solution

Fix this on the Android side by making GatewaySession readiness reflect completed connect, then add bounded notification retry/reconciliation that preserves existing notification policy and gateway protocol behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: connectOnce stores currentConnection before conn.connect() completes, and the notification sink can call sendNodeEvent in that window. I did not run a live Pixel reconnect scenario in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is Android-side readiness plus bounded notification replay; the reporter's local patch direction matches that shape and does not require a Gateway protocol change.

## Repair Work Prompt

Fix the Android notification forwarding reconnect race reported at https://github.com/openclaw/openclaw/issues/79552. In GatewaySession, do not let outbound RPC/node.event callers use a connection until the connect handshake has completed. In NodeRuntime/DeviceNotificationListenerService, add bounded notification event queue/retry for notifications.changed sends that fail during reconnect, flush queued events after node reconnect, and reconcile active posted notifications without bypassing existing forwarding filters, quiet hours, rate limits, or package policy. Dedupe replayed notifications so reconnect reconciliation does not double-wake the same notification. Keep the Gateway protocol/server behavior unchanged and avoid adding a config option. Add or update Android unit/Robolectric coverage for pre-handshake send gating, queued notification flush, and active notification reconciliation. Add a single-line CHANGELOG.md fix entry crediting @hectorrp13 if the change is user-facing. Stop and ask for maintainer input if the implementation would require changing the websocket protocol, notification policy semantics, or Android permission model.
