## Summary

Keep open: current main and v2026.5.12 still expose the Android reconnect race, and both linked fix PRs are closed unmerged. The reporter's local patch comment is useful confirmation of the right direction: Android should wait for a completed gateway connect handshake and preserve notification events that arrive during reconnect.

## What This Changes

The issue reports that Android notification node.event frames can be sent before the gateway WebSocket connect handshake completes during reconnect, causing the gateway to reject and drop notification events.

## Best Possible Solution

Fix Android-side readiness and bounded notification replay/reconciliation while keeping the Gateway protocol contract unchanged.

## Reproduction Assessment

Yes, source-reproducible with high confidence: connectOnce publishes currentConnection before conn.connect() completes, and the notification sink can call sendNodeEvent in that window while the gateway rejects non-connect first frames. I did not run a live Android/Gateway reconnect scenario in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is Android-side: do not expose or use a node.event-capable connection until connect succeeds, then preserve notification events through a bounded queue or active-notification reconciliation after reconnect.
