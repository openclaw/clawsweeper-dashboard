## Summary

Keep open: current main and the latest release still expose the Android reconnect race, and both linked fix PRs are closed unmerged. The remaining work is a narrow Android-side fix: make node.event send paths wait for a completed Gateway connect handshake and preserve notification events that arrive during reconnect.

## What This Changes

The issue asks Android nodes to avoid sending node.event before the Gateway connect handshake completes and to preserve notification events that arrive during reconnect.

## Best Possible Solution

Fix Android-side readiness and bounded notification replay/reconciliation while keeping the Gateway protocol contract unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: connectOnce publishes currentConnection before conn.connect() completes, and the notification sink can call sendNodeEvent in that window while the Gateway rejects non-connect first frames.

## Solution Assessment

Yes. The narrow maintainable solution is Android-side: do not expose or use a node.event-capable connection until connect succeeds, and preserve notification events through a bounded queue or active-notification reconciliation after reconnect.
