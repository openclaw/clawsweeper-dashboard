## Repair Work Prompt

Fix the Android notification reconnect race described in https://github.com/openclaw/openclaw/issues/79552. Keep the Gateway protocol unchanged: the first WebSocket request must remain connect. In Android, do not let sendNodeEvent/sendNodeEventDetailed use a GatewaySession connection before the connect RPC has completed successfully; either publish currentConnection only after connect succeeds or add an explicit ready/handshake-complete guard that is set before onConnected callbacks can flush work. Preserve notification events that arrive during reconnect with a bounded queue and/or active-notification reconciliation after reconnect, respecting existing notification forwarding filters and avoiding duplicate wakes. Use the closed PRs https://github.com/openclaw/openclaw/pull/79565 and https://github.com/openclaw/openclaw/pull/79927 as prior art, but do not copy their incomplete behavior blindly. Add focused Android tests that prove node.event is not sent before the connect ACK and that queued/reconciled notification events flush after the ready connection is available. Do not change Gateway connect-first semantics or add new config options.
