## Summary

Keep open: current main still exposes a connecting Android GatewaySession before the connect RPC completes, notification forwarding still drops failed sends, and the gateway still rejects a non-connect first frame. The two linked fix PRs were closed unmerged, so the canonical issue remains actionable.

## What This Changes

Ensure Android notification node.event frames are delayed, retried, or replayed until after the gateway WebSocket connect handshake completes.

## Best Possible Solution

Fix Android-side connection readiness and bounded notification replay/reconciliation, with regression tests and redacted real reconnect proof, while leaving the gateway protocol unchanged.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: connectOnce publishes currentConnection before conn.connect(), NodeRuntime can send notification node.event in that window, and the server rejects non-connect first frames. I did not perform a live Pixel reconnect run.

## Solution Assessment

Yes, the maintainable direction is Android-side: only expose or use a ready connection after connect succeeds, then queue/dedupe/reconcile notification events across reconnect. A gateway protocol or config change is not needed.
