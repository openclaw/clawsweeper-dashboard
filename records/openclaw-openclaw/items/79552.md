## Summary

Keep open: current main still exposes a pre-handshake Android node connection and the notification forwarding path still drops failed `sendNodeEvent` calls, so the reported reconnect-time notification loss remains valid.

## What This Changes

The issue requests that Android notification `node.event` frames wait for the Gateway connect handshake and that reconnect-window notifications be queued, retried, or reconciled after reconnect.

## Best Possible Solution

Make the Android node treat the Gateway connection as send-ready only after the connect ACK, then preserve reconnect-window notification events with bounded retry or active-notification reconciliation while leaving the Gateway protocol unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `connectOnce` exposes `currentConnection` before `conn.connect()` completes, and the notification sink can call `sendNodeEvent` during that window; I did not run a live Android reconnect scenario in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is Android-side connection readiness plus bounded notification replay/reconciliation, not a Gateway protocol change.
