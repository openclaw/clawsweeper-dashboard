## Label Justifications

- `P1`: Android notification forwarding can permanently lose automation-triggering events during reconnect for real users.
- `impact:message-loss`: The reported failure drops `notifications.changed` node events before they reach the gateway and agent automation path.

## Summary

Keep open: current main and the latest release still expose the Android node connection before the connect RPC finishes, notification forwarding still sends directly without retry/replay, and an open PR now explicitly owns the fix path.

## What This Changes

Android notification forwarding can lose `notifications.changed` events when reconnect sends `node.event` before the gateway connect handshake completes.

## Best Possible Solution

Review and land a narrow Android fix that separates connecting from ready gateway sockets, queues failed notification events, flushes after reconnect, and reconciles active notifications without duplicate replay.

## Reproduction Assessment

Yes, at source level: `connectOnce` publishes `currentConnection` before the connect RPC completes, and notification forwarding sends directly through that readiness gate. I did not run Android/Robolectric or live device proof in this read-only review.

## Solution Assessment

Yes, the best fix direction is the linked PR path: make `currentConnection` represent a ready connection only, then add bounded notification retry and reconnect reconciliation. The issue should remain open until that PR lands or a replacement is chosen.
