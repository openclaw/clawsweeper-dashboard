## Summary

Keep open: current main still exposes the Android reconnect race, and the two linked fix PRs are closed unmerged rather than landed. The reporter's local patch/testing is useful supporting context, but the repository still needs a mainline fix.

## What This Changes

Ensure Android notification node events are not sent or lost during reconnect before the Gateway WebSocket connect handshake completes.

## Best Possible Solution

Land an Android-side fix that gates node.event on completed connect and replays only bounded, policy-filtered notification events after reconnect without changing the Gateway protocol.

## Reproduction Assessment

Yes. Source inspection on current main shows connectOnce publishes currentConnection before conn.connect() finishes, and the notification sink can call sendNodeEvent in that window; I did not run a live Pixel reconnect scenario.

## Solution Assessment

Yes. The narrow maintainable fix is Android-side readiness gating plus bounded notification retry/reconciliation after reconnect, while keeping the Gateway connect-first handshake unchanged.
