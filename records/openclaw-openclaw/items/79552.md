## Label Justifications

- `P1`: Android notification events can be permanently lost during reconnect, preventing notification-triggered automations from waking agents.
- `impact:message-loss`: The issue is specifically about `notifications.changed` node events being rejected and not retried, which silently drops a user-visible automation trigger.

## Summary

Keep open: current main and the latest release still expose the Android gateway connection before the connect RPC finishes, and notification events are still sent directly without a retry/replay path. The related fix PR was closed unmerged, so no merged implementation currently owns the remaining work.

## What This Changes

Fix Android notification forwarding so `node.event` frames are not sent before the websocket connect handshake, and reconnect-window notifications are queued, retried, or reconciled.

## Best Possible Solution

Land a focused Android fix that exposes a node connection only after the connect RPC succeeds, then uses a bounded notification-event queue with reconnect flush and active-notification reconciliation without duplicate delivery.

## Reproduction Assessment

Yes at source level: trigger a notification after `connectOnce` assigns `currentConnection` but before `Connection.connect()` completes, and `sendNodeEvent` can send `node.event` as the first websocket request. No live Android/gateway run was performed in this read-only review.

## Solution Assessment

Yes, the requested direction is the right repair shape, but it needs the callback ordering fixed too: publish the connection only after connect succeeds, then flush queued notification events after that ready connection is visible.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79552 in the Android app. Current main sets `GatewaySession.currentConnection` before the websocket `connect` RPC completes, while `NodeRuntime` forwards `DeviceNotificationListenerService` notification events directly through `nodeSession.sendNodeEvent`; the gateway rejects a pre-connect `node.event` as `invalid handshake: first request must be connect`, and Android drops the event. Implement a focused fix in the Android runtime: make a connection visible for normal RPC/event sends only after the connect handshake succeeds; preserve `onConnected` behavior by returning/holding the ready hello metadata and invoking callbacks after the ready connection is visible; add a bounded retry queue for forwarded notification events that failed or arrived while reconnecting; flush it after reconnect; reconcile active posted notifications after reconnect without duplicating already delivered keys. Add regression tests for the pre-connect send race and notification queue/flush behavior. Do not change the gateway protocol, add config options, or edit CHANGELOG.md.
