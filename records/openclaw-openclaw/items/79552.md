## Summary

Keep open: current main and the latest release still expose the Android reconnect race, and the two related fix PRs were closed unmerged. The remaining work is a narrow Android-side fix to make node.event sends wait for a completed gateway connect handshake and preserve notification events that arrive during reconnect.

## What This Changes

The issue requests that Android wait for the gateway connect handshake before sending node.event frames and preserve notification events during reconnect.

## Best Possible Solution

Fix Android-side connection readiness and bounded notification replay/reconciliation while keeping the gateway connect-first protocol unchanged.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: connectOnce publishes currentConnection before conn.connect() completes, and the notification sink can call sendNodeEvent in that window while the gateway rejects non-connect first frames. I did not live-run a Pixel/Gateway reconnect scenario, so this is source-reproducible rather than live reproduced.

## Solution Assessment

Yes, the narrow maintainable solution is Android-side: do not expose or use a node.event-capable connection until connect succeeds, and preserve notification events through a bounded queue or active-notification reconciliation after reconnect. Changing the gateway handshake contract would conflict with the documented protocol.
