## Summary

Keep open: current main still exposes an Android node connection before the Gateway connect RPC completes, and notification forwarding still drops the failed send result instead of retrying or replaying it.

## What This Changes

Prevent Android node.event frames from being sent before the WebSocket connect handshake and preserve notification events that arrive during reconnect.

## Best Possible Solution

Fix the Android runtime path so node events are sent only after authenticated connect completion, then add bounded notification retry or active-notification reconciliation after reconnect without changing the Gateway protocol.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delay the connect ACK, call sendNodeEvent while currentConnection is already assigned, and the Gateway rejects node.event as the first request.

## Solution Assessment

Yes. The narrow maintainable solution is Android-side readiness gating plus bounded notification retry/reconciliation; a Gateway protocol change is not needed.

## Repair Work Prompt

Fix the Android notification reconnect delivery bug reported in https://github.com/openclaw/openclaw/issues/79552. Current main assigns GatewaySession.currentConnection before conn.connect() finishes, so nodeSession.sendNodeEvent can send node.event before the Gateway connect handshake; NodeRuntime also ignores failed notification sends. Implement an Android-side fix only: make sendNodeEvent/sendNodeEventDetailed refuse pre-handshake connections or publish currentConnection only after connect is ready, then preserve notification events that arrive during reconnect with a bounded queue and/or active-notification reconciliation after reconnect. Do not change the Gateway protocol, do not bypass notification forwarding policy, quiet-hours, package filtering, or burst limits, and avoid unbounded memory growth. Use the useful direction from the closed attempts at https://github.com/openclaw/openclaw/pull/79565 and https://github.com/openclaw/openclaw/pull/79927, but ensure the final fix actually delivers or reconciles notification events after reconnect. Add focused Android tests covering no node.event before connect ACK and queued/replayed notification delivery after reconnect. Add a single changelog entry because this is user-visible.
