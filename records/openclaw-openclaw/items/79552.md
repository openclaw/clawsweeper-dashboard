Keep open: current main still exposes the Android node reconnect race and the notification path can still drop events; the two related fix PRs are closed and unmerged, so there is no implemented or superseding fix to close against.

## What This Changes

The issue asks Android to avoid sending `node.event` before the WebSocket `connect` handshake completes and to preserve notification events that arrive during reconnect.

## Best Possible Solution

Keep the issue open and land a narrow Android fix that gates node events on a completed connect handshake and preserves notification events through reconnect without changing the Gateway protocol.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `connectOnce` stores `currentConnection` before `conn.connect()` completes, and the notification sink can call `sendNodeEvent` in that window; I did not run a live Pixel reconnect scenario.

## Solution Assessment

Yes. The best fix is Android-side readiness plus bounded notification retry/reconciliation after reconnect; a Gateway protocol change does not appear necessary.

## Repair Work Prompt

Fix the Android reconnect notification delivery bug reported in https://github.com/openclaw/openclaw/issues/79552. On current main, `GatewaySession.connectOnce` assigns `currentConnection` before `conn.connect()` completes, while `sendNodeEvent` sends through any non-null current connection; `NodeRuntime` forwards notification events without checking the Boolean result, and `DeviceNotificationListenerService` emits posted notifications once. Keep the Gateway protocol unchanged. Implement an Android-side readiness guard or publish `currentConnection` only after the connect ACK, and add bounded retry/queue/reconciliation so `notifications.changed` events arriving during reconnect are delivered after reconnect without unbounded memory growth or duplicate wakes. Preserve existing notification forwarding filters and sessionKey behavior. Add focused Android regression coverage for pre-handshake node.event behavior and notification queue/replay behavior. Do not change unrelated Gateway protocol, provider, or channel surfaces. Add a user-facing changelog entry if the project policy requires it for the fix branch.
