## Summary

Keep open: current main still exposes the Android reconnect race, and the two linked fix PRs are closed unmerged, so notification events can still be lost before the Gateway connect handshake completes.

## What This Changes

The issue asks Android to wait for a completed Gateway WebSocket connect handshake before sending node.event frames and to preserve notification events that arrive during reconnect.

## Best Possible Solution

Fix Android-side readiness and bounded notification replay/reconciliation while keeping the Gateway connect-first protocol unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows currentConnection is published before conn.connect completes, and the notification sink can call sendNodeEvent during that window while the Gateway rejects non-connect first frames.

## Solution Assessment

Yes. The narrow maintainable direction is Android-side: do not expose or use a node.event-capable connection until connect succeeds, and preserve notification events through a bounded queue or active-notification reconciliation after reconnect.

## Repair Work Prompt

Fix the Android reconnect notification-loss bug from https://github.com/openclaw/openclaw/issues/79552. On current main, GatewaySession.connectOnce publishes currentConnection before conn.connect completes, sendNodeEvent can send node.event before the Gateway connect ACK, and NodeRuntime drops notification sends that return false. Implement the fix on the Android side: make node.event sends wait for a ready/handshake-complete Gateway connection, preserve notification events that arrive during reconnect with a bounded queue and/or active-notification reconciliation, flush only after a send-ready node session is visible, and avoid duplicate wakeups. Preserve existing notification forwarding policy, rate limiting, quiet-hours filtering, and the Gateway connect-first protocol; do not add a config option or change core Gateway protocol behavior. Use the closed PRs https://github.com/openclaw/openclaw/pull/79565 and https://github.com/openclaw/openclaw/pull/79927 as prior approaches but produce a clean current-main implementation with focused Android tests. Add a changelog entry if the repair is user-visible.
