## What This Changes

The PR threads WhatsApp authDir into createWebSendApi, adds a forward lid-mapping resolver for phone-number outbound sends, routes message, poll, and composing paths through it, and adds helper/send API tests.

## Best Possible Solution

Land this focused WhatsApp plugin fix after adding the required changelog entry, keeping PN-to-LID resolution at the active Web send boundary and tracking any re-link reverse-mapping persistence issue separately.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction: even with lid-mapping-{phoneDigits}.json available, createWebSendApi has no authDir input and still sends +1555-style targets as 1555@s.whatsapp.net.

## Solution Assessment

Yes, with one process fix. Resolving phone-number recipients against the listener account authDir inside createWebSendApi is the narrowest maintainable fix for message, poll, and composing delivery, and the PR should add the missing changelog line before merge.
