## Summary

Keep open: current main implements fixed same-sender inbound debouncing for rapid text-message batching, but this issue still asks for semantic long-message splitting and adaptive aggregation behavior that are not in the current config, schema, docs, or debouncer contract. The closed coalescing RFC at https://github.com/openclaw/openclaw/issues/51361 covers the implemented rapid-batching subset, while the adaptive timing subset remains separately open at https://github.com/openclaw/openclaw/issues/57123.

## What This Changes

Request a semantic message preprocessing layer that can merge rapid short-message bursts, split or structure long multi-topic messages, and adapt aggregation timing to user behavior.

## Best Possible Solution

Use this issue as the broader product/API design thread for semantic splitting and adaptive aggregation, while treating `messages.inbound` and https://github.com/openclaw/openclaw/issues/57123 as the current rapid-batching baseline and narrower adaptive-timing follow-up.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior. Source and docs inspection show a fixed text-batching capability, not an existing semantic-splitting contract to reproduce.

## Solution Assessment

No for closure or a single automated bugfix. The maintainable path needs an explicit product/API decision for semantic splitting and adaptive timing, preserving current fixed-window behavior unless maintainers choose a migration.
