## Summary

Keep open. Current main covers timing-based rapid-message batching, but semantic long-message topic detection/splitting and adaptive aggregation remain absent and need maintainer product/API design.

## What This Changes

The issue requests semantic preprocessing that merges rapid short inbound messages, splits or structures multi-topic long messages, and optionally adapts the aggregation window.

## Best Possible Solution

Keep the issue open and narrow the remaining scope into a typed inbound preprocessing design, either gateway-owned or exposed as an additive plugin SDK seam with explicit transcript, command, media, routing, and adaptive-timing semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence on the current gap: `messages.inbound` batches same-sender text by timing, but no source/config/docs path implements semantic topic splitting or adaptive windows.

## Solution Assessment

Not applicable as a proposed patch. A one-shot automatic fix is not the best route yet because the remaining behavior changes core conversation semantics and probably needs a versioned plugin/core contract.
