## Summary

Keep open: current main covers the rapid-message batching subset with fixed inbound debouncing, but the issue still asks for semantic long-message splitting and adaptive aggregation behavior that are not in the current config, schema, docs, or debouncer contract. The closed coalescing RFC at https://github.com/openclaw/openclaw/issues/51361 covers the implemented rapid-batching baseline, while https://github.com/openclaw/openclaw/issues/57123 separately tracks the adaptive-timing subset.

## What This Changes

The issue requests semantic preprocessing for inbound chat messages, including rapid short-message aggregation, long-message multi-topic handling, and adaptive aggregation windows.

## Best Possible Solution

Use this issue as the broader product/API design thread for semantic splitting and cross-channel aggregation semantics, while treating `messages.inbound` and https://github.com/openclaw/openclaw/issues/57123 as the current rapid-batching baseline and narrower adaptive-timing follow-up.
