## Summary

Keep open. Current main covers the rapid-message batching part with timing-based inbound debounce, but the issue also asks for semantic long-message topic detection/splitting and adaptive aggregation windows, which are still absent from config, source behavior, docs, and plugin hook contracts.

## What This Changes

Request a semantic inbound preprocessing layer that merges rapid short message bursts, detects or splits multi-topic long messages, and optionally adapts aggregation timing to user cadence.

## Best Possible Solution

Keep the issue open and narrow the remaining scope around semantic preprocessing beyond shipped `messages.inbound` debounce. The maintainable end state is either a gateway-owned, typed preprocessing stage or an additive plugin SDK contract that can rewrite or split inbound content with explicit rules for transcripts, commands, media, threading, and adaptive timing.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. There is a high-confidence source-inspection path for the gap: current main can batch rapid same-sender text via `messages.inbound`, but no config/source/docs path implements long-message semantic topic detection/splitting or adaptive windows.

## Solution Assessment

Not applicable as a proposed patch. A narrow automatic fix is not the best route yet; the remaining request needs maintainer product/API judgment before code because the obvious implementation choices affect core conversation semantics and plugin contracts.
