## Summary

Keep open: current main and v2026.6.10 already provide fixed same-sender inbound text batching, but semantic long-message topic splitting and adaptive timing remain unimplemented and need maintainer product/API direction.

## What This Changes

The issue requests semantic inbound preprocessing that merges rapid short-message bursts, optionally splits long multi-topic messages, and adapts aggregation timing to user behavior.

## Best Possible Solution

Keep the fixed inbound debounce baseline, let the open adaptive and Telegram metadata trackers own their narrower pieces, and require a separate maintainer-approved design for long-message topic splitting before implementation.

## Reproduction Assessment

Not applicable: this is a feature request rather than a report of broken documented behavior. Source and docs show the current supported behavior is fixed same-sender text debounce, not semantic topic splitting or adaptive timing.

## Solution Assessment

Unclear: the all-in-one semantic preprocessing layer is broader than the current fixed debounce model. A safer path is to separate fixed/adaptive timing, bundle metadata, and topic splitting into explicit maintainer-approved API/product decisions.
