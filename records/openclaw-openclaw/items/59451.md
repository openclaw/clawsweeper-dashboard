## Summary

Keep open: current main and the latest release still return an unavailable memory_search payload when the SQLite-backed memory manager cannot initialize, and the open related PR only addresses the narrower error-message UX rather than preserving file-based recall.

## What This Changes

Request to add a file-based keyword fallback over canonical memory files when node:sqlite is unavailable.

## Best Possible Solution

Add a bounded memory-core fallback that detects missing node:sqlite, scans only allowed agent memory files, ranks snippets deterministically, and clearly labels results as fallback keyword search.

## Reproduction Assessment

Yes, at source level: make require("node:sqlite") fail, then MemoryIndexManager.get cannot initialize, getMemoryManagerContext returns an error, and memory_search returns the unavailable payload before any file scan can run. I did not run a live affected Node build because the requested review was read-only and current source is sufficient to prove the path.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if it is limited to missing-node:sqlite initialization and reuses existing memory file constraints. The separately linked messaging PR should not be treated as resolving this fallback capability.
