## Summary

Keep open: current main still has no file-based fallback when SQLite-backed memory manager initialization fails, and the active related PR only addresses the narrower unavailable-message UX.

## What This Changes

The issue asks for `memory_search` to fall back to deterministic keyword scanning of `MEMORY.md` and `memory/*.md` when `node:sqlite` is unavailable.

## Best Possible Solution

Add a bounded memory-core fallback that detects missing-`node:sqlite` initialization, scans only canonical workspace memory files, ranks snippets deterministically, and clearly labels results as fallback keyword search.

## Reproduction Assessment

Yes, source-level: make `require("node:sqlite")` throw, create an agent workspace with `MEMORY.md` or `memory/*.md`, and call `memory_search`; current main reaches the unavailable-result path before any file scan can run.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it is limited to the missing-SQLite runtime case and reuses existing memory file/path constraints rather than changing broader provider fallback policy.
