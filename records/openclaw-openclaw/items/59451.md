## Summary

Keep open: current main and v2026.5.18 still turn missing node:sqlite into an unavailable memory_search response before any MEMORY.md file scan can run, while the linked PR only targets error wording.

## What This Changes

The issue requests a file-based keyword fallback for memory_search when node:sqlite is unavailable.

## Best Possible Solution

Add a bounded memory-core fallback that detects missing node:sqlite, scans only allowed agent memory files, ranks snippets deterministically, and clearly labels results as fallback keyword search.

## Reproduction Assessment

Yes, at source level: make require("node:sqlite") fail, then MemoryIndexManager construction fails, getMemoryManagerContext returns an error, and memory_search returns the unavailable payload before any file scan can run. I did not run a live affected Node build in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if it is limited to missing-node:sqlite initialization and reuses existing memory file constraints. The linked messaging PR should not be treated as resolving this fallback capability.
