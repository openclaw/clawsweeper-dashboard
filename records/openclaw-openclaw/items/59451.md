## Summary

Keep open. Current main still has no file-based keyword fallback when `node:sqlite` cannot initialize; memory-only `memory_search` returns the unavailable payload before any file scan can run, and the related open PR only handles diagnostic copy for the narrower error-message issue.

## What This Changes

The issue requests a file-based keyword fallback for `memory_search` over `MEMORY.md` and `memory/*.md` when `node:sqlite` is unavailable.

## Best Possible Solution

Add a bounded memory-core fallback that detects the missing-`node:sqlite` manager-init failure, scans only allowed `MEMORY.md` and `memory/*.md` files, ranks snippets deterministically, and labels the response as fallback keyword search.

## Reproduction Assessment

Yes. A source-level reproduction is to make `require("node:sqlite")` throw, create a workspace with `MEMORY.md` or `memory/*.md`, and call `memory_search`; current code returns the unavailable payload before any file scan can run.

## Solution Assessment

Yes. The requested fallback is the narrowest maintainable direction if it is limited to the SQLite-runtime-unavailable case and reuses existing memory read/path constraints instead of replacing the SQLite backend or changing provider fallback policy.
