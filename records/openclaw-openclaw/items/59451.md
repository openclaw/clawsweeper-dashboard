## Summary

Keep open: current main and v2026.5.18 still turn missing `node:sqlite` manager initialization into an unavailable `memory_search` response with empty results. The related messaging PR only targets clearer error copy for https://github.com/openclaw/openclaw/issues/59457, while this issue's file keyword fallback remains unimplemented.

## What This Changes

This issue asks for `memory_search` to fall back to deterministic file keyword search over `MEMORY.md` and `memory/*.md` when `node:sqlite` cannot be loaded.

## Best Possible Solution

Add a bounded memory-core fallback that detects only missing `node:sqlite`, scans allowed agent memory files, ranks snippets deterministically, and labels the result as fallback keyword search.

## Reproduction Assessment

Yes, at source level: when `require("node:sqlite")` fails, `MemoryIndexManager.get` cannot initialize, `getMemoryManagerContext` returns an error, and `memory_search` returns the unavailable payload before any file scan can run. I did not run an affected Node build in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if it is limited to missing-`node:sqlite` initialization and reuses existing memory file constraints. It should not replace provider quota, embedding failure, QMD transient, or in-database FTS fallback handling.
