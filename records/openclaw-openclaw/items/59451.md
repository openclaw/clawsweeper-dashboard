## Summary

Keep open. Current main still routes `memory_search` through the active memory manager and the SQLite-backed builtin manager; when the builtin manager cannot initialize because `node:sqlite` is missing, the tool returns an unavailable payload rather than scanning `MEMORY.md` and `memory/*.md`. The linked #69199/#59457 work is explicitly the narrower error-message path and does not implement this fallback search capability.

## What This Changes

Add a file-based keyword fallback for `memory_search` over `MEMORY.md` and `memory/*.md` when `node:sqlite` is unavailable.

## Best Possible Solution

Implement a bounded, clearly marked fallback inside the memory-core search tool or adjacent runtime helper: when builtin memory manager initialization fails specifically because `node:sqlite` is unavailable, scan the configured agent workspace memory files with existing memory path rules, return ranked keyword snippets with `fallback`/`mode` metadata, and leave the separate #69199 error-message improvement focused on unavailable cases where fallback cannot run.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to simulate a Node runtime where `require("node:sqlite")` throws, create an agent workspace with `MEMORY.md` or `memory/*.md`, and call `memory_search`; current main reaches the manager-unavailable path instead of returning file keyword hits.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix for this capability gap. The fallback should stay in the memory-core owner boundary and reuse existing memory file/path helpers rather than adding startup migrations, core-wide policy, or a new backend selection mechanism first.
