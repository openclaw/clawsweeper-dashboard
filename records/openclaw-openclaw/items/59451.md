## Summary

Keep open: current main and v2026.5.18 still turn a missing node:sqlite initialization into an unavailable memory_search response with empty results, while the linked open PR only covers the narrower error-message UX and the nearby FTS fallback PR targets a different SQLite capability failure.

## What This Changes

Add a file-based keyword-search fallback over allowed memory files when node:sqlite is unavailable so memory_search preserves basic recall and labels the degraded mode clearly.

## Best Possible Solution

Add a bounded memory-core fallback that detects missing node:sqlite, scans only allowed agent memory Markdown files, ranks snippets deterministically, and marks the response as fallback keyword search.

## Reproduction Assessment

Yes, at source level: make require("node:sqlite") fail, then MemoryIndexManager.get cannot initialize, getMemoryManagerContext returns an error, and memory_search returns the unavailable payload before any file scan can run.

## Solution Assessment

Yes, the requested fallback is the narrow maintainable direction if it is limited to missing node:sqlite initialization and reuses existing memory file constraints; the linked messaging PR should not be treated as resolving this capability.
