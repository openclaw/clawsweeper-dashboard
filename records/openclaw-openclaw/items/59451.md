## Summary

Keep open: current main still turns missing `node:sqlite` during memory manager initialization into an unavailable `memory_search` response with empty results. Existing lexical fallbacks only work after a SQLite-backed index opens, and the related messaging PR targets clearer error text rather than the file-based fallback requested here.

## What This Changes

Add a file-based keyword fallback over canonical memory files when `node:sqlite` is unavailable, and label those results as fallback search.

## Best Possible Solution

Add a bounded memory-core fallback for missing `node:sqlite` only, scanning canonical memory files through existing host path constraints, ranking snippets deterministically, and labeling results as fallback keyword search.

## Reproduction Assessment

Yes, at source level: when `require("node:sqlite")` fails, `MemoryIndexManager.get` cannot initialize, `getMemoryManagerContext` returns an error, and `memory_search` returns the unavailable payload before any file scan can run. I did not run a live affected Node build in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if it is limited to missing-`node:sqlite` initialization and reuses existing memory file constraints. It should not replace provider quota, embedding failure, QMD transient, or SQLite-backed FTS fallback handling.
