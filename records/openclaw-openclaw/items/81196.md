## Summary

Keep open: the report is source-corroborated against current main, and the active fix is still an open PR rather than merged behavior. The current code still imports a private Codex SDK subpath from the packaged Codex runtime while non-QA root-alias/export behavior does not expose that path for npm-installed `@openclaw/codex`.

## What This Changes

The issue reports that beta 2026.5.12-beta.2 still fails to run `openai/gpt-*` through the Codex runtime because `openclaw/plugin-sdk/codex-native-task-runtime` resolves under `root-alias.cjs` as if it were a directory.

## Best Possible Solution

Review and land the focused Codex npm-plugin runtime resolution fix, then prove `openai/gpt-*` runs through the packaged Codex runtime without fallback; split the metadata/updater observations only if they remain afterward.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: update/install the beta Codex plugin and run a direct `openclaw agent --model openai/gpt-*` smoke. I did not execute the packaged beta smoke locally, but the report, a second Linux confirmation, and current source all line up on the same missing private subpath.

## Solution Assessment

Yes, the narrow maintainable path is to repair the Codex plugin runtime/package-loader boundary for npm-installed `@openclaw/codex` while keeping the helper Codex-scoped. The active open PR is the right current vehicle to review rather than starting a separate repair branch.
