- `impact:crash-loop`: The issue is about process-level availability after gateway crashes or exits under the Windows Scheduled Task path.

## Summary

Keep open. Current main still maps native Windows gateway lifecycle to a Scheduled Task that starts at logon, while macOS and Linux have explicit crash-restart policies; the newer Windows 11 field report and the closed Windows Service discussion at https://github.com/openclaw/openclaw/issues/83085 add useful design evidence rather than making this obsolete.

## What This Changes

Request Windows gateway crash-recovery parity by adding a watchdog or service-backed supervisor instead of relying only on a logon Scheduled Task.

## Best Possible Solution

Define a first-class Windows gateway supervisor contract, then implement it consistently across install, update, stop, restart, uninstall, status, doctor/docs, and real Windows upgrade proof.

## Reproduction Assessment

Yes, for the source-level gap: current main creates and updates only the Windows logon Scheduled Task path, with no watchdog or Windows Service supervisor. I did not run a live Windows Task Scheduler crash-restart reproduction in this read-only review.

## Solution Assessment

Unclear. The dedicated watchdog design is plausible, and the Windows Service report is also plausible, but maintainers should choose the Windows supervisor contract before implementation.
