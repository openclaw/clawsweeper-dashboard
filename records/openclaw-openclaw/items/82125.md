## Label Justifications

- `P1`: A native Windows gateway crash can leave agents and channels down until manual intervention, and field comments report this affecting real Windows deployments.
- `impact:crash-loop`: The issue is about process-level gateway availability and missing auto-recovery after a Windows gateway crash.

## Summary

Keep open. Current main still gives native Windows only Scheduled Task or Startup-folder login startup, while Linux and macOS have explicit restart semantics; the related field reports make this a real Windows gateway lifecycle gap, but the watchdog-vs-service design needs maintainer judgment before implementation.

## What This Changes

Add Windows gateway crash-recovery behavior so native Windows installs can recover after mid-session gateway crashes like the current systemd and launchd paths do.

## Best Possible Solution

Define a first-class Windows gateway supervisor contract, then implement the chosen watchdog or Windows Service path across install, update, stop, uninstall, status, doctor, docs, and real Windows proof.

## Reproduction Assessment

Yes for the current source behavior: current main maps Windows to Scheduled Tasks and creates or updates only the logon task path, while the field comment reports killing the gateway PID leaves it dead on Windows 11. I did not run live Windows proof in this read-only review.

## Solution Assessment

Unclear. The dedicated watchdog design is plausible, and the Windows Service field workaround is also credible, but maintainers should choose the lifecycle contract before automation implements one path.
