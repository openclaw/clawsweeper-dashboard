## Summary

Keep open. Current main and the latest shipped release still route native Windows gateway lifecycle through a Scheduled Task logon launcher with no watchdog or Windows Service adapter, and the newer field report strengthens the same crash-recovery gap rather than resolving it.

## What This Changes

Request Windows gateway crash-recovery parity by adding a Scheduled Task watchdog or Windows Service adapter instead of relying only on logon startup.

## Best Possible Solution

Define one first-class Windows gateway supervisor contract, then implement install/update/status/stop/uninstall and doctor repair with focused tests and real Windows proof.

## Reproduction Assessment

Yes for the source-level behavior: current main and `v2026.5.12` create/update only the Scheduled Task logon path, with no watchdog or Windows Service adapter. I did not run a live Windows crash-restart repro in this read-only review.

## Solution Assessment

Unclear. A dedicated watchdog and a Windows Service adapter are both plausible, but maintainers should choose the Windows lifecycle contract before implementation to avoid split supervisors.
