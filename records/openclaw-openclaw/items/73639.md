## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though interactive onboarding, runtime validation, and docs support that auth mode; the same-author draft PR #49107 remains the active implementation path and needs maintainer/security review.

## What This Changes

The issue requests non-interactive `openclaw onboard` flags to select and fully configure gateway `trusted-proxy` auth for reverse-proxy deployments.

## Best Possible Solution

Review and land #49107 or an equivalent narrow onboarding patch with typed trusted-proxy flags, fail-closed validation, tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk --skip-health`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before trusted-proxy fields can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; manual config edits or a plugin would bypass the typed setup contract for a security-sensitive path.
