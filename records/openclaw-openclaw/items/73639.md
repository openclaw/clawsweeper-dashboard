## Summary

Keep open. Current main and v2026.5.2 still reject `trusted-proxy` in non-interactive onboarding, while interactive onboarding, runtime auth, schema, and docs already support trusted-proxy; the same-author draft PR #49107 is the active implementation path and should be reviewed rather than bypassed.

## What This Changes

Request non-interactive `openclaw onboard` support for selecting and configuring trusted-proxy gateway auth.

## Best Possible Solution

Land a reviewed core onboarding change that adds typed trusted-proxy non-interactive flags with docs, tests, and changelog coverage while preserving existing gateway auth guardrails.

## Reproduction Assessment

Yes, source-level. `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk` reaches `applyNonInteractiveGatewayConfig`, where `trusted-proxy` is rejected before config can be written.

## Solution Assessment

Yes. Core onboarding parity is the maintainable direction because trusted-proxy is already a core gateway auth mode; manual config edits or a plugin would bypass the typed onboarding contract.
