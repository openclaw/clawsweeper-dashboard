## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime, interactive gateway configurator, and docs support trusted-proxy auth; the same-author draft PR #49107 remains the active implementation path and needs maintainer/security review rather than cleanup closure.

## What This Changes

The issue asks for non-interactive `openclaw onboard` support to select and configure trusted-proxy gateway authentication for scripted reverse-proxy deployments.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch that adds typed trusted-proxy flags, tests, docs, and fail-closed validation parity with the existing interactive wizard.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before trusted-proxy fields can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; a plugin or manual config edit would bypass the typed onboarding contract for a security-sensitive setup.
