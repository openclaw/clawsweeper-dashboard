## Summary

Keep open. Current main still rejects `trusted-proxy` for non-interactive onboarding while the runtime, interactive gateway configurator, and docs already support trusted-proxy auth; same-author draft PR #49107 remains the active implementation path and needs maintainer/security review.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for configuring trusted-proxy gateway authentication in scripted reverse-proxy deployments.

## Best Possible Solution

Core onboarding should expose typed trusted-proxy non-interactive flags with validation, docs, tests, and a changelog entry while preserving the existing gateway auth trust model.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before trusted-proxy fields can be written.

## Solution Assessment

Yes. Core onboarding parity is the maintainable direction because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; manual config edits or a plugin would bypass the typed onboarding contract.
