## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though runtime, docs, and the interactive gateway configurator support trusted-proxy auth; draft PR #49107 explicitly addresses this issue but still needs maintainer/security review rather than cleanup closure.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring trusted-proxy gateway auth behind reverse proxies.

## Best Possible Solution

Review and land #49107 or a replacement patch that mirrors the interactive trusted-proxy contract, clears incompatible credentials, adds focused tests/docs, and keeps loopback trust explicit.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before trusted-proxy fields can be written.

## Solution Assessment

Yes for the requested layer: core non-interactive onboarding is the maintainable place because trusted-proxy is already a core gateway auth mode. The existing draft PR is not ready as submitted and needs the security-sensitive fixes called out in its review before it becomes the best implementation.
