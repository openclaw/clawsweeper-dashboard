## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though trusted-proxy is supported by runtime config, interactive gateway setup, tests, and docs; draft PR #49107 remains the active same-author implementation path and needs maintainer/security review.

## What This Changes

The issue asks for non-interactive `openclaw onboard` support to select and configure trusted-proxy gateway authentication for scripted reverse-proxy deployments.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch that adds typed trusted-proxy flags, fail-closed validation parity with the interactive wizard, focused tests, docs, and changelog coverage if user-facing.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before trusted-proxy fields can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and interactive onboarding already configures it; manual config edits or plugin-side setup would bypass the typed onboarding contract for a security-sensitive setup.
