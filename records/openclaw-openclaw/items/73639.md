## Summary

Keep open. Current main still has the reported gap: non-interactive onboarding rejects `trusted-proxy`, while the gateway runtime, config schema, docs, and interactive gateway setup already support that auth mode. The same-author draft PR #49107 is the active implementation candidate, so this should remain open rather than being closed or converted into a competing repair job.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring `trusted-proxy` gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Add a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, exposes typed non-interactive flags for trusted proxy IPs, user header, required headers, allowUsers, allowLoopback, and Control UI allowed origins, writes the existing gateway config schema, updates docs/tests, and keeps runtime validation fail-closed.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the narrowest maintainable solution because `trusted-proxy` is already a core gateway auth mode and the interactive wizard already configures it; manual config editing or a plugin would bypass the typed onboarding/security contract.
