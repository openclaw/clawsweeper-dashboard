## Summary

Keep open. Current main and the latest stable release still only allow token/password in non-interactive gateway onboarding, while trusted-proxy already exists in runtime and interactive gateway configuration; the open same-author draft PR is the active implementation path.

## What This Changes

Request non-interactive `openclaw onboard` flags and config handling for trusted-proxy gateway auth in scripted deployments.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/49107 with a reviewed core onboarding change that writes the same trusted-proxy config shape as interactive onboarding, with focused tests and public docs for the new non-interactive flags.

## Reproduction Assessment

Yes. A high-confidence source-level path is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`, which reaches `applyNonInteractiveGatewayConfig` and is rejected by the token/password-only validation.

## Solution Assessment

Yes on the requested direction, but not by closing this issue now. Core onboarding parity is the maintainable fix because trusted-proxy already exists as runtime and interactive gateway behavior, while the open draft PR still needs maintainer/security review and proof.
