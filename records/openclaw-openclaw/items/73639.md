## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime, config schema, interactive gateway configurator, and docs all support trusted-proxy gateway auth. The same-author draft PR #49107 remains the active implementation candidate, so this should not be closed or replaced by a competing repair lane yet.

## What This Changes

Support configuring trusted-proxy gateway auth and related trusted proxy/Control UI fields through `openclaw onboard --non-interactive`.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, adds typed flags for the existing trusted-proxy fields plus Control UI origins, updates docs/tests, and keeps the runtime's fail-closed trusted-proxy validation intact.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the maintainable fix because trusted-proxy is already a core gateway auth mode with interactive setup, runtime validation, config schema, and public docs; a plugin or manual config edit would bypass that typed security-sensitive setup path.
