## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while the runtime, config contract, docs, and interactive gateway setup already support that auth mode; the same-author draft PR #49107 and duplicate-looking issue #73638 should be reviewed or consolidated before cleanup closure.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring trusted-proxy gateway authentication behind an identity-aware reverse proxy.

## Best Possible Solution

Add a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, exposes typed flags for trusted proxy IPs, identity headers, required headers, allowUsers, loopback opt-in, and Control UI origins where needed, writes the existing gateway config schema, updates focused tests/docs, and preserves fail-closed runtime validation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right layer because trusted-proxy is already a core gateway auth mode and interactive setup supports it; raw config editing or a plugin would bypass typed onboarding and security guidance.
