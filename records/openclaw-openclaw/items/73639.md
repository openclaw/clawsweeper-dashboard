## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime, interactive gateway setup, schema, and docs already support that auth mode. The issue is also paired with same-author draft PR #49107 and duplicate-looking issue #73638, so cleanup should wait for maintainer review, merge, replacement, or explicit consolidation.

## What This Changes

Add non-interactive onboarding support for configuring `trusted-proxy` gateway auth behind an identity-aware reverse proxy.

## Best Possible Solution

Land a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, adds typed non-interactive flags for trusted proxy IPs, identity headers, allowUsers, loopback opt-in, and Control UI origins where needed, writes the existing gateway config schema, updates docs/changelog, and keeps runtime validation fail-closed.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes. The best fix belongs in core onboarding because `trusted-proxy` is already a core gateway auth mode and interactive setup already configures it; raw config edits or a plugin would bypass the typed onboarding and security guidance surface.
