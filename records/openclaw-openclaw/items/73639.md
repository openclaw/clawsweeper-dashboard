## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while the runtime, interactive gateway configurator, and docs support that auth mode; the same-author draft PR #49107 remains the active implementation path and needs maintainer/security review.

## What This Changes

Add non-interactive onboarding support for configuring `trusted-proxy` gateway auth for automated reverse-proxy deployments.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch that adds typed trusted-proxy flags, tests, docs, and fail-closed validation parity with the interactive wizard.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk --skip-health`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and interactive setup configures it; a plugin or manual config edit would bypass the typed onboarding contract for a security-sensitive setup.
