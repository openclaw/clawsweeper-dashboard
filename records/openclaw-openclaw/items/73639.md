## Summary

Keep open. Current main still has the reported gap: runtime, docs, and interactive setup support `trusted-proxy`, but `onboard --non-interactive` rejects it and the same-author draft PR #49107 is already the active implementation candidate.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring `gateway.auth.mode: "trusted-proxy"` and its reverse-proxy identity settings.

## Best Possible Solution

Add a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, exposes typed non-interactive flags for trusted proxy IPs, user header, required headers, allowUsers, allowLoopback, and Control UI allowed origins, writes the existing gateway config schema, updates docs/tests, and keeps runtime validation fail-closed.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the narrowest maintainable fix because `trusted-proxy` is already a core gateway auth mode and the interactive wizard already configures it; manual config editing or a plugin would bypass the typed onboarding contract and security prompts.
