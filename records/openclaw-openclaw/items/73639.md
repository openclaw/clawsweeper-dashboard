## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while the runtime, interactive gateway configurator, and docs all support trusted-proxy auth; open draft PR #49107 remains the active implementation path and needs maintainer/security review.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for configuring the core `trusted-proxy` gateway auth mode.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch with typed flags, fail-closed validation, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before trusted-proxy fields can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; a plugin or manual config edit would bypass the typed onboarding contract for a security-sensitive setup.
