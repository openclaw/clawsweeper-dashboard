## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while runtime, interactive configuration, and docs support it, and same-author draft PR #49107 is already the active implementation path for https://github.com/openclaw/openclaw/issues/73639.

## What This Changes

Add non-interactive onboarding support for gateway `trusted-proxy` auth so scripted reverse-proxy deployments can avoid interactive setup or manual config edits.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch that adds typed trusted-proxy flags, tests, docs/changelog, and validation parity with the interactive gateway wizard.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits before any trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the maintainable direction because `trusted-proxy` is already a core gateway auth mode and the interactive wizard configures it; a plugin or manual config edit would bypass the typed setup contract for a security-sensitive mode.
