## Summary

Keep open. Current main still has the reported onboarding gap: trusted-proxy is a supported gateway auth mode at runtime and in interactive gateway setup, but the non-interactive onboard path and CLI option type still only accept token/password. The issue is also paired with same-author open draft PR #49107, so cleanup should wait for maintainer/security review of that implementation or a maintainer-directed replacement.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring `gateway.auth.mode="trusted-proxy"` and the related trusted proxy identity settings for scripted deployments.

## Best Possible Solution

Add a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, exposes typed non-interactive flags for trusted proxy IPs, identity header, required headers, allowUsers, allowLoopback, and Control UI allowed origins, writes the existing gateway config schema, updates user-facing docs, and preserves current fail-closed runtime validation.

## Reproduction Assessment

Yes. The source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits before config write because only `token` and `password` are accepted.

## Solution Assessment

Yes. The requested direction belongs in core onboarding because trusted-proxy is already a core gateway auth mode and interactive setup already configures it; raw config editing or plugin-side setup would bypass the typed onboarding/security contract.
