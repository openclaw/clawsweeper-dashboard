## Summary

Keep open. Current main still has the reported gap: trusted-proxy is supported by the gateway runtime and interactive configurator, but non-interactive onboarding still only accepts token or password auth; the same-author draft PR #49107 remains the active implementation candidate.

## What This Changes

The issue requests non-interactive `openclaw onboard` flags and config writing for `gateway.auth.mode: "trusted-proxy"` deployments behind an identity-aware reverse proxy.

## Best Possible Solution

Add a narrow core onboarding change, likely by reviewing and landing #49107 or an equivalent patch, that accepts `--gateway-auth trusted-proxy`, exposes typed flags for the existing trusted-proxy config fields and Control UI origins, updates docs/tests, and preserves the runtime's fail-closed validation.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; a plugin or manual config edit would bypass the typed onboarding contract for a security-sensitive setup.
