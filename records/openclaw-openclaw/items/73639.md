## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime, interactive gateway configurator, and docs support trusted-proxy auth; same-author draft PR #49107 and duplicate report #73638 should be handled together rather than closing this issue in isolation.

## What This Changes

Allow `openclaw onboard --non-interactive` to configure `gateway.auth.mode: "trusted-proxy"` without manual config edits.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch with typed flags for trusted proxy IPs, user header, required headers, allowed users, loopback/origin policy, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy fields can be written.

## Solution Assessment

Yes. Core non-interactive onboarding is the right place because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; manual config edits or a plugin would bypass the typed setup contract for a security-sensitive mode.
