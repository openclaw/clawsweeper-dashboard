## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while the runtime, interactive gateway wizard, and docs already support trusted-proxy auth; the open same-author draft PR #49107 is the active implementation path for https://github.com/openclaw/openclaw/issues/73639.

## What This Changes

The issue requests non-interactive onboarding support for configuring trusted-proxy gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch that adds typed trusted-proxy flags, tests, docs, and validation parity with the existing interactive wizard.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits on the token/password-only gate before writing trusted-proxy config.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; a plugin or manual config edit would bypass the typed onboarding contract for a security-sensitive setup.
