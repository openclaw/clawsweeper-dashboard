## Summary

Keep open. Current main still leaves the reported onboarding gap: `trusted-proxy` is a supported runtime and interactive gateway auth mode, but `onboard --non-interactive` still restricts `--gateway-auth` to `token` or `password`. This issue is also paired with same-author open draft PR #49107, so cleanup should wait for that implementation to be reviewed, replaced, or closed.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring trusted-proxy gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Add a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, exposes typed flags for trusted proxy sources and identity constraints, writes the existing gateway config schema, updates user-facing docs, and preserves current fail-closed runtime validation.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode because only `token` and `password` are accepted.

## Solution Assessment

Yes. The maintainable fix belongs in core onboarding because `trusted-proxy` is already a core gateway auth mode and interactive setup already configures it; raw config editing or a plugin would bypass the typed onboarding contract and security prompts.
