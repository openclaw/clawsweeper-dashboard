## Summary

Keep open. Current main still blocks `trusted-proxy` in non-interactive onboarding, while runtime auth, interactive gateway setup, and docs already support trusted-proxy; the same-author draft PR #49107 is the active implementation path and needs maintainer/security review.

## What This Changes

The issue requests non-interactive `openclaw onboard` flags for configuring trusted-proxy gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch that adds typed trusted-proxy flags, tests, docs, and validation parity with the interactive wizard.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits before trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and the interactive wizard already configures it; a plugin or manual config edit would bypass the typed onboarding contract for a security-sensitive setup.
