## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while interactive onboarding, runtime gateway auth, and public docs already support trusted-proxy mode; draft PR #49107 remains the active implementation path and needs maintainer/security review.

## What This Changes

Request non-interactive `openclaw onboard` support for `trusted-proxy` gateway auth so scripted reverse-proxy deployments do not require interactive setup or manual config edits.

## Best Possible Solution

Land #49107 or an equivalent narrow core onboarding patch with typed trusted-proxy flags, fail-closed validation, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before trusted-proxy fields can be written.

## Solution Assessment

Yes in direction. Core onboarding parity is the maintainable solution because trusted-proxy is already a core gateway auth mode; the exact PR still needs security-conscious review before landing.
