## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding, while interactive onboarding, runtime validation, tests, and docs all treat trusted-proxy as a core gateway auth mode; open draft PR #49107 remains the active implementation path.

## What This Changes

The issue requests non-interactive onboarding support for configuring trusted-proxy gateway authentication.

## Best Possible Solution

Review and land #49107 or an equivalent narrow core onboarding patch with typed trusted-proxy flags, fail-closed validation, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the mode before any trusted-proxy config can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode; a plugin or manual config edit would bypass the typed onboarding contract for security-sensitive setup.
