## Summary

Keep open. Current main and the latest stable release still reject `trusted-proxy` in non-interactive onboarding, while runtime config, interactive gateway setup, validation, and docs already support trusted-proxy; the same-author draft PR at https://github.com/openclaw/openclaw/pull/49107 remains the active implementation path but is not merged.

## What This Changes

Add non-interactive `onboard` CLI/config support for selecting and configuring trusted-proxy gateway authentication.

## Best Possible Solution

Review, fix, and land or replace https://github.com/openclaw/openclaw/pull/49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding and updates focused tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level path is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`, which reaches `applyNonInteractiveGatewayConfig` and exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes on the requested direction, but not by closing this issue now. Non-interactive onboarding is the right core surface because trusted-proxy is already a core runtime and interactive onboarding mode; the safe implementation still needs the paired PR’s fixes, tests, proof, and security review.
