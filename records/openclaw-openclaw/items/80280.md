## Summary

Keep open: the change is useful and has supplied real behavior proof, but the latest head still has a blocking timeout-budget mismatch where Chrome MCP status can spend up to 10s on route probes while public status and non-deep doctor clients still abort at 7.5s.

## What This Changes

The PR adds `BrowserStatus.pageReady` for Chrome MCP status using an ephemeral page-reachability probe, threads the ephemeral option through browser availability, and updates existing-session status tests.

## Best Possible Solution

Keep the `pageReady` contract and ephemeral page probe, but make the route and public client budgets share a bounded timeout contract that reliably returns a status response.

## Reproduction Assessment

Yes, at source level: current main reports Chrome MCP status from the transport probe only, and the existing test encodes transport-up/page-unreachable as running without a page-readiness field. I did not run a live macOS Chrome reproduction in this read-only review.

## Solution Assessment

No, not yet. Adding `pageReady` with an ephemeral page probe is the right narrow direction, but the PR still needs to align route and client timeout budgets so callers see `pageReady: false` instead of a client timeout.
