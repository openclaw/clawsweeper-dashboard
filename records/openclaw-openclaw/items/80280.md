## Summary

Keep this PR open: the `pageReady` status contract is useful and the PR includes real CLI proof, but the current patch makes a passive browser status request run the normal cached Chrome MCP page-tool path, which can seed a persistent MCP session instead of staying status-safe.

## What This Changes

The PR adds `BrowserStatus.pageReady`, probes Chrome MCP page reachability with a 5s timeout, forwards that timeout into `listChromeMcpTabs`, and adds focused browser status tests.

## Best Possible Solution

Expose `pageReady` from a bounded status-safe page probe that does not seed or replace the shared Chrome MCP attach session, while keeping the transport-up/page-down tests.

## Reproduction Assessment

Yes for the reported gap at source level: current main only checks Chrome MCP transport for status, and the existing route test keeps `cdpReady: true` when page reachability is false. The PR body also supplies live macOS CLI output showing the intended new `pageReady: false` signal after the patch.

## Solution Assessment

No, not as written. Adding `pageReady` is the right narrow contract, but the status route should use an ephemeral/status-safe page probe instead of the normal cached page-tool path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep the page readiness probe status-safe:** `extensions/browser/src/browser/routes/basic.ts:88`
  - body: This status path now calls `profileCtx.isReachable()`, whose Chrome MCP implementation goes through `listChromeMcpTabs()` without `ephemeral: true`. When the probe succeeds, `leaseSession()` takes the normal cached-session branch, so a passive `browser status` call can create and retain a persistent Chrome MCP session and change `pid`/subsequent behavior as if the user had started page tooling. Use an ephemeral/status-safe page probe for `pageReady` so status remains observational.
  - confidence: 0.87
