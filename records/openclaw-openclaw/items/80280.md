## Summary

Keep open: the PR addresses a real browser status gap and has sufficient live proof, but the current branch still makes passive status call the normal cached Chrome MCP page-tool path. That can create or retain a shared attach session during `browser status`, so the branch needs a narrow status-safe repair before merge.

## What This Changes

The PR adds `BrowserStatus.pageReady`, probes Chrome MCP page reachability for status/doctor output, forwards a timeout into `listChromeMcpTabs`, and updates browser docs, diagnostics, and tests.

## Best Possible Solution

Expose `pageReady` from a bounded Chrome MCP page probe that is explicitly ephemeral/status-safe, preserves the shared attach-session cache, and keeps the new doctor/CLI/docs/tests aligned.

## Reproduction Assessment

Yes for the reported status-contract gap at source level: current main reports Chrome MCP transport readiness only, while the linked issue and PR body show the transport-up/page-tools-down state. I did not run a live macOS Chrome session in this read-only review.

## Solution Assessment

No, not as written. Adding `pageReady` is the right narrow contract, but the page-readiness check must avoid the normal cached Chrome MCP session path used by real page tools.
