## Summary

Keep open: the PR is aimed at the right status-contract gap, but the proposed status probe uses the non-ephemeral Chrome MCP page path and would make a read-only status call seed a persistent MCP session. It also still needs inspectable after-fix real behavior proof before merge.

## What This Changes

The PR adds `pageReady` to `BrowserStatus`, derives it from a bounded Chrome MCP `list_pages` reachability probe, threads the timeout into `listChromeMcpTabs`, and updates existing-session status tests.

## Best Possible Solution

Keep the additive `pageReady` contract, but compute it with a bounded non-invasive Chrome MCP page probe that does not seed shared sessions, and require redacted real status output before merge.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main lacks `pageReady`, maps Chrome MCP status to transport readiness only, and already has a separate page-level `list_pages` reachability path. I did not run the macOS Chrome live setup.

## Solution Assessment

No, not as written. The additive field is the right direction, but the status path should use an ephemeral page probe and the PR needs inspectable real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the status page probe ephemeral:** `extensions/browser/src/browser/routes/basic.ts:86`
  - body: This call reaches `listChromeMcpTabs` through `isReachable`, but that path does not pass `ephemeral`, so `leaseSession` uses `getSession` and stores a shared Chrome MCP session.
