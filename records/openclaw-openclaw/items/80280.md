## Summary

Keep open: this PR adds a useful `pageReady` status signal with real behavior proof, but the latest head still has a blocking timeout-budget bug where the new sequential Chrome MCP probes can exceed the public status/non-deep doctor client timeout.

## What This Changes

This PR adds `BrowserStatus.pageReady`, computes it from an ephemeral Chrome MCP page-tool probe for existing-session profiles, threads the option through browser availability actions, and updates status tests.

## Best Possible Solution

Keep `pageReady` and the ephemeral probe, but align the status/doctor client timeout budget with the route probe budget and add focused route/client regression coverage.

## Reproduction Assessment

Yes at source level, not live-reproduced in this read-only pass. Current main lacks a page-readiness field and the remaining PR bug follows from the 5s plus 5s route probe budget versus the 7.5s public client timeout.

## Solution Assessment

No, not yet. Adding `pageReady` with an ephemeral Chrome MCP page probe is the right narrow direction, but the route and client timeout contract must be repaired so status reliably returns the new signal.
