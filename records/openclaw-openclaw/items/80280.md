## Summary

Keep open: the direction and supplied live proof are good, but the current PR head still lets the new sequential Chrome MCP page-readiness probe exceed the public status/doctor client timeout before it can return `pageReady: false`.

## What This Changes

The branch adds `BrowserStatus.pageReady`, computes it with an ephemeral Chrome MCP page probe in browser status, forwards the ephemeral option through availability actions, and updates existing-session status tests.

## Best Possible Solution

Keep `pageReady` and the status-safe ephemeral page probe, but make the route and public client budgets share a contract that reliably returns `pageReady: false` instead of timing out.

## Reproduction Assessment

Yes at source level: current main only reports Chrome MCP status from the transport probe, and the existing test keeps `running: true`/`cdpReady: true` even when page reachability is false. The PR body also includes copied live output showing the intended after-fix `pageReady: false` field.

## Solution Assessment

No, not yet. Adding `pageReady` with an ephemeral probe is the right narrow direction, but the patch needs to align the new sequential probe with the status and doctor timeout contract.
