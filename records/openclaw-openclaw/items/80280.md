## Summary

Keep open: the PR fixes a real Chrome MCP status gap and has sufficient live-output proof, but the latest head still lets status spend up to 5s on transport plus 5s on page readiness while the public status/doctor client budget remains 7.5s.

## What This Changes

The PR adds `BrowserStatus.pageReady`, computes it via an ephemeral Chrome MCP page probe for existing-session profiles, threads the option through availability actions, and updates browser status tests.

## Best Possible Solution

Keep `pageReady` and the ephemeral probe, but make the route and public status/doctor timeout budgets share a tested contract that reliably returns a status response.

Yes at source level. Current main lacks a page-readiness field and its existing-session status test preserves the misleading `cdpReady: true` case when page reachability is false; the remaining PR bug follows from the 10s route budget versus 7.5s client timeout.

## Solution Assessment

No, not yet. Adding `pageReady` with an ephemeral page probe is the right narrow direction, but the latest patch still needs the timeout-budget repair and focused client coverage.
