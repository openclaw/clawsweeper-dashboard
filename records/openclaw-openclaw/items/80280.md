## Summary

Keep open: the direction and supplied live output are credible, but the PR can spend up to 10s in status while public status/doctor clients still abort at 7.5s, so callers may get a timeout instead of the intended pageReady:false signal.

## What This Changes

Adds BrowserStatus.pageReady plus a Chrome MCP existing-session status probe that calls isReachable with ephemeral listChromeMcpTabs, updating availability types and route tests.

## Best Possible Solution

Keep pageReady and the status-safe ephemeral probe, but make the status/doctor route fit one public timeout budget or update the public budgets with focused coverage.

## Reproduction Assessment

Yes at source level: current main lacks a page-readiness status field, and the PR head has a deterministic slow-path where 5s transport plus 5s page probing can exceed the 7.5s public client timeout. I did not run a live Chrome session in this read-only review.

## Solution Assessment

No, not yet. Adding pageReady with an ephemeral page probe is the right narrow direction, but the implementation needs to preserve the status/doctor timeout contract so callers receive pageReady:false instead of a client timeout.
