## Summary

Keep open: the core `pageReady` direction and proof are useful, but the latest head still lets status spend up to 5s on Chrome MCP transport plus 5s on the page probe while the public status and non-deep doctor clients abort after 7.5s, so callers can get a timeout instead of the new `pageReady: false` signal.

## What This Changes

Adds `BrowserStatus.pageReady` for Chrome MCP existing-session profiles, probes page reachability in ephemeral mode, threads the `isReachable` option through browser profile actions, and updates status route tests.

## Best Possible Solution

Keep `pageReady` and the ephemeral page probe, but align the route and public client timeout budgets and cover that contract in focused route/client regression tests.

## Reproduction Assessment

Yes at source level. Current main lacks a page-readiness field, and the remaining PR defect follows from the latest diff's sequential 5s transport plus 5s page probe versus the unchanged 7.5s public client timeout.

## Solution Assessment

No, not yet. The `pageReady` field with an ephemeral Chrome MCP page probe is the right narrow fix direction, but the timeout contract must be repaired so status reliably returns the new signal.
