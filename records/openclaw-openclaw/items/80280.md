## Summary

Keep open: the PR fixes the earlier cache-safety issue by forwarding `ephemeral: true`, but the new status path can still overrun the public status/doctor client timeout because it runs a 5s transport probe and then a separate 5s page probe while those clients still default to 7.5s.

## What This Changes

Adds `BrowserStatus.pageReady` and a status-time Chrome MCP page-readiness probe, with availability types/tests updated to pass an ephemeral probe option.

## Best Possible Solution

Keep `pageReady` with the status-safe ephemeral page probe, but align the route and public client budgets so a slow attach plus page-tool timeout returns `pageReady: false` instead of a client-side timeout.

## Reproduction Assessment

Yes at source level: current main only reports Chrome MCP status from the transport probe, and the existing test keeps `running: true`/`cdpReady: true` even when page reachability is false. The PR body also includes copied live output showing the intended after-fix `pageReady: false` status field.

## Solution Assessment

No, not yet. Adding `pageReady` and using an ephemeral probe is the right narrow direction, but the patch needs to make the new sequential probe fit the public status/doctor timeout contract.
