## Summary

Keep open: the `pageReady` direction is useful and well-scoped, but the current PR can spend up to 10s in the status route while the public `browserStatus()` and non-deep `browserDoctor()` clients still abort after 7.5s, so callers can receive a timeout instead of the new `pageReady: false` signal.

## What This Changes

The PR adds `BrowserStatus.pageReady`, computes it for Chrome MCP existing-session profiles with an ephemeral page-level reachability probe, forwards the optional `ephemeral` flag through availability actions, and updates status tests.

## Best Possible Solution

Keep the additive `pageReady` contract and status-safe ephemeral probe, but make the route and client timeout budgets share a tested contract so slow page probes return status JSON instead of timing out.

## Reproduction Assessment

Yes at source level: current main reports Chrome MCP status from transport readiness only, and the existing test covers transport-up/page-unreachable without a page-readiness field. I did not run a live macOS Chrome MCP session in this read-only review.

## Solution Assessment

No, not yet. Adding `pageReady` with an ephemeral page probe is the right narrow fix, but the patch still needs to align the new sequential probe with the public status and doctor timeout budget.
