## Summary

Keep open: the patch is a useful, well-scoped direction with sufficient real behavior proof, but the current head can still spend up to 10 seconds in sequential Chrome MCP status probes while public status/doctor clients default to 7.5 seconds, so callers can see a client timeout instead of the new `pageReady: false` signal.

## What This Changes

The PR adds `BrowserStatus.pageReady`, computes it from an ephemeral Chrome MCP page-level reachability probe for existing-session profiles, forwards the ephemeral option through browser profile actions, and updates status-route tests.

## Best Possible Solution

Keep the additive `pageReady` contract and ephemeral probe, but make the route, public client, and doctor timeout budgets share a tested contract that returns `pageReady: false` instead of timing out.

## Reproduction Assessment

Yes, at source level: current main only reports Chrome MCP status from transport readiness and the existing-session route tests encode transport-up/page-reachability-down as running without any page-readiness field. I did not run a live macOS Chrome session in this read-only review.

## Solution Assessment

No, not yet. Adding `pageReady` with a status-safe ephemeral probe is the right narrow direction, but the current patch still needs the status and doctor timeout budget aligned with the new sequential probe.
