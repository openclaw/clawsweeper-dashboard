Keep open: current main still has the reported memory-core close/sync ordering hazard, and the related FD-leak PRs are open adjacent work rather than merged fixes for this specific race.

## What This Changes

The issue asks for `MemoryIndexManager.close()` to wait for or isolate in-flight sync/provider-initialization work before closing provider and SQLite resources.

## Best Possible Solution

Land a focused memory-core lifecycle fix that prevents provider/DB teardown until all already-started sync/provider-initialization work is settled or using stable resource snapshots.

## Reproduction Assessment

No deterministic runtime reproduction is available. The source reproduction is high-confidence: an already-started `sync()` can be awaiting provider initialization while `close()` snapshots `this.syncing` as null and later closes resources before that sync settles.

## Solution Assessment

Yes, the best fix is a narrow lifecycle repair in memory-core: either re-check and await newly attached sync work after awaited initialization steps, or make sync hold stable provider/DB state across awaits. The fix should add tests for both interleavings before landing.
