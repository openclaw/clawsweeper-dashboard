## Summary

Keep open: the PR addresses a valid Docker onboarding bug, but the patch currently treats PATH-only brew detection as Homebrew absence and can hide brew-only skills even when Linuxbrew is present in a standard resolver path.

## What This Changes

The PR filters brew-only skill dependency installs during Linux-container onboarding, adds a container-specific note, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Keep the container-specific onboarding mitigation, but make brew availability use the same resolver contract as installSkill before landing; leave broader install/fallback policy to #42303, #55940, or the canonical bug thread.

## Reproduction Assessment

Yes by source inspection. Current main offers brew-backed missing skills during onboarding, while the PR's changed path can be evaluated against the existing installer and brew resolver contracts without needing a live Docker run.

## Solution Assessment

No as currently written. Hiding brew-only installs in Linux containers is the right narrow UX direction, but the availability check should match installSkill's resolveBrewExecutable fallback instead of only checking PATH.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the installer brew resolver before hiding options:** `src/commands/onboard-skills.ts:83`
  - body: `detectBinary("brew")` only checks PATH, but the installer currently falls back to `resolveBrewExecutable()` and can run brew from standard Linuxbrew locations like `/home/linuxbrew/.linuxbrew/bin/brew`. In a Linux container with that supported layout but no PATH entry, this branch hides every brew-only skill even though the installer can still use Homebrew. Please base `brewAvailable` on the same resolver contract used by `installSkill`.
  - confidence: 0.86
