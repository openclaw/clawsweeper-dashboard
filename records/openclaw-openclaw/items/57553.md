## Summary

Keep this PR open. The reported Docker onboarding problem is real on current main, but this patch is not ready because it adds a weaker duplicate container detector and the branch is dirty against current main while related Linuxbrew alternatives remain open.

## What This Changes

The branch hides brew-only missing skill installers during Linux-container onboarding when Homebrew is absent, shows a container-specific note, and adds onboarding/container-context tests.

## Best Possible Solution

Keep the PR open, rebase or recreate it from current main, reuse the centralized container detector, add focused coverage for the supported container signals, add the required changelog entry, and coordinate the final Docker skill-install policy with #42303 and #55940.

## Reproduction Assessment

Yes. The code path is high-confidence: current main offers brew-only missing skills during onboarding and shows the Homebrew prompt after selection when `brew` is absent, while Docker setup runs that onboarding path inside the container.

## Solution Assessment

No, not as-is. Hiding unavailable brew-only installs is a plausible narrow UX fix, but the implementation should use the existing cgroup-aware detector instead of adding a partial duplicate, and maintainers should decide how it coexists with Linuxbrew support.
