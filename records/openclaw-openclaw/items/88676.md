## Label Justifications

- `P1`: The PR is for a broken agent/channel workflow where memory-flush failure can wedge user replies, and the proposed fix still risks session-state breakage.
- `merge-risk: 🚨 session-state`: The diff writes session-store memory-flush markers after failed flushes, which can stale or suppress future session memory recovery behavior.

## Summary

Keep open: the PR targets a real P1 memory-flush retry bug, but the current patch reuses the successful-flush dedupe marker as failure state, so one transient non-abort provider failure can permanently suppress future memory flushes for that compaction cycle.

## What This Changes

The PR persists `memoryFlushAt` and `memoryFlushCompactionCount` on non-abort memory-flush failures and adds mocked regression tests for marker persistence and retry suppression.

## Best Possible Solution

Use separate failure metadata with maintainer-approved retry/escalation semantics, and only mark `memoryFlushCompactionCount` when a flush actually ran for the successful dedupe contract.

## Reproduction Assessment

Yes, at source level: current main retries after non-abort flush failure because no marker is persisted, while this PR makes the next call skip by writing the current compaction count into the dedupe field. I did not run a live provider-crash reproduction in this read-only review.

## Solution Assessment

No: the patch breaks the retry loop by marking a failed attempt as if the flush already ran, which can permanently suppress memory flush after a transient 5xx or timeout. A separate failure tracker with bounded retry or escalation is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not mark failed flushes as completed:** `src/auto-reply/reply/agent-runner-memory.ts:1362`
  - body: This writes the current `compactionCount` into the same field that `shouldRunMemoryFlush()` uses to decide a flush already ran. Because a failed flush does not increment `compactionCount`, one transient non-abort failure such as a 5xx or timeout will make future messages skip memory flush for this bloated session even after the provider recovers. Track failure attempts separately or bound the suppression instead of reusing the successful-flush marker.
