## Summary

Keep this PR open, but do not merge it as-is: it targets a real memory-flush retry loop, but it marks failed flushes as completed and can suppress all later memory-flush attempts for the same compaction cycle. The supplied proof is Docker/Vitest output against mocked provider failures, not real or closer integration recovery proof.

## What This Changes

The PR writes `memoryFlushAt` and `memoryFlushCompactionCount` on non-abort memory-flush failures and adds mocked regression tests for marker persistence and retry suppression.

## Best Possible Solution

Preserve `memoryFlushCompactionCount` for successful flushes only, and add a separate maintainer-approved failure/retry/escalation state with proof that the session either retries safely or degrades through an explicit recovery path.

## Reproduction Assessment

Yes for the source-level loop: current main catches a non-abort flush failure without updating `memoryFlushCompactionCount`, and the next over-threshold message can re-enter the same flush path. A real provider-crash reproduction is still not supplied.

## Solution Assessment

No. The proposed branch is a narrow loop breaker, but marking a failed flush as completed is not the safest durable fix; a separate failure tracker with retry or escalation policy is the maintainable direction.
