## Label Justifications

- `P1`: The issue reports real-world loss of irreplaceable session transcripts tied to current archive naming and retention semantics.
- `impact:data-loss`: The issue is about session transcript artifacts being treated as disposable and physically removed.
- `impact:session-state`: The issue concerns session transcript lifecycle, reset/delete archival, and retained conversation state.

## Summary

Keep open: current `main` and the latest release still use `.jsonl.deleted.*` / `.jsonl.reset.*` archive names and still physically remove archive artifacts through retention and disk-budget paths. Related open work improves archive visibility/search, but it does not decide the core naming and “never delete session transcripts” product contract.

## What This Changes

The issue asks OpenClaw to replace destructive-looking session archive names with archival semantics and prevent session transcript files from being physically deleted by OpenClaw code paths.

## Best Possible Solution

Maintainers should decide the durable archive contract first, then implement a compatibility-aware migration that keeps old `.deleted.*` / `.reset.*` files readable while moving new archives to clearer permanent semantics and updating retention/docs/tests together.

## Reproduction Assessment

Yes at source level: current main still produces `.jsonl.deleted.*` and `.jsonl.reset.*` names and still has retention cleanup that physically removes archive artifacts. I did not run a live `/new` or `/reset` flow because this review is read-only.

## Solution Assessment

No for a blind rename-only fix: current main has filename parsers, docs, tests, cleanup, cron, memory/search, and UI-adjacent paths depending on the shipped suffixes. The safer solution is a maintainer-approved retention and migration design that preserves old archives while changing new durable-archive semantics.
