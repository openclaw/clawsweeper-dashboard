## Label Justifications

- `P1`: The issue reports real-world irreversible transcript loss tied to current archive naming and cleanup semantics that still exist on main and in the latest release.
- `impact:data-loss`: The report concerns retained conversation transcripts being treated as disposable and later physically lost.
- `impact:session-state`: The affected surface is persisted session transcript lifecycle state across reset, delete, maintenance, archive cleanup, and indexing paths.

## Summary

Keep open: current main and the latest release still create `.jsonl.deleted.*` and `.jsonl.reset.*` retained transcripts, and current retention/disk-budget paths can still physically remove those artifacts. The issue is valid, but the fix needs a maintainer decision on durable archive semantics rather than stale cleanup.

## What This Changes

Request to rename retained session archive suffixes to non-destructive archived naming or an archive directory, and to stop physically deleting session transcript files.

## Best Possible Solution

Maintainers should define a durable session-archive contract that separates retained-history naming from physical deletion policy, preserves existing suffixes through migration/doctor guidance, and updates runtime, docs, and tests together.

## Reproduction Assessment

Yes, at source level: current main still produces `.jsonl.deleted.*` and `.jsonl.reset.*` archive names and still has retention/disk-budget paths that physically remove archive artifacts. I did not run a live `/new` or `/reset` flow because this review is read-only.

## Solution Assessment

No for a rename-only fix: current runtime parsers, docs, tests, cleanup, disk-budget behavior, and related memory/session paths depend on the shipped suffixes. The best path is a maintainer-approved retention and migration design before implementation.
