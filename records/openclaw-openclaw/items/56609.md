## Label Justifications

- `P1`: The issue reports real-world irreversible session transcript loss tied to current archive naming and cleanup semantics that still exist on main and in the latest release.
- `impact:data-loss`: The report is about retained transcript artifacts being treated as disposable and later lost or physically removed.
- `impact:session-state`: The affected data is persisted session transcript lifecycle state across reset, delete, maintenance, and archive cleanup paths.

## Summary

Keep open: current main and v2026.6.8 still use `.jsonl.deleted.*` and `.jsonl.reset.*` for retained transcripts, and cleanup/disk-budget paths can still physically remove those archive artifacts. The related retention-semantics issue overlaps, but it does not supersede this issue's data-loss and never-delete product decision.

## What This Changes

The issue requests replacing destructive-looking session archive names like `.deleted.*` and `.reset.*` with durable archive semantics, preferably without any OpenClaw code path physically deleting session transcripts.

## Best Possible Solution

Define a maintainer-approved durable archive contract, keep existing `.deleted.*` and `.reset.*` files readable or migratable, then update naming, cleanup defaults, disk-budget behavior, docs, and tests together.

## Reproduction Assessment

Yes, at source level: current main still produces `.jsonl.deleted.*` and `.jsonl.reset.*` archive names and has cleanup paths that physically remove archive artifacts. I did not run a live `/new` or `/reset` flow because this review is read-only.

## Solution Assessment

No for a rename-only fix: current parsers, docs, tests, cleanup, disk-budget behavior, and related memory/session paths depend on the shipped suffixes. The best fix is a maintainer-approved retention and migration design that changes new archive semantics without losing old archives.
