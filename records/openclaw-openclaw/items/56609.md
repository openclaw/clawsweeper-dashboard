- `impact:data-loss`: The report is about retained transcript artifacts being treated as disposable and physically removed.
- `impact:session-state`: The affected artifacts are session transcripts and lifecycle state created by reset/delete/session maintenance paths.

## Summary

Keep open: current `main` and latest `v2026.6.6` still use `.jsonl.deleted.*` / `.jsonl.reset.*` archive names and still have retention and disk-budget paths that physically remove archive artifacts. The related open retention-semantics issue is useful context, but this data-loss-oriented product contract is not implemented or safely superseded.

## What This Changes

The issue requests renaming retained session transcript artifacts from `.deleted.*`/`.reset.*` to archival terminology or moving them into a durable archive directory, and avoiding physical session transcript deletion paths.

## Best Possible Solution

Maintainers should decide the durable session-archive contract first, then implement a compatibility-aware migration that keeps old `.deleted.*` / `.reset.*` files readable while giving new archives clear permanent-retention semantics and updating docs/tests together.

## Reproduction Assessment

Yes at source level: current main still produces `.jsonl.deleted.*` and `.jsonl.reset.*` names and still has retention cleanup that physically removes archive artifacts. I did not run a live `/new` or `/reset` flow because this review is read-only.

## Solution Assessment

No for a blind rename-only fix: current main has filename parsers, docs, tests, cleanup, disk-budget, memory/search, and history paths depending on the shipped suffixes. The safer solution is a maintainer-approved retention and migration design that preserves old archives while changing new durable-archive semantics.
