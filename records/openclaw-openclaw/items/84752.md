## Summary

Keep this PR open: the OAuth and Telegram recovery goals are plausible, but the diagnostics change introduces a blocking serialization risk by resetting an active command lane from a normal idle transition, and the real-behavior-proof check is currently failing because the proof is not in the required parsed field format.

## What This Changes

The PR changes diagnostics recovery, legacy OAuth sidecar loading for auth-profile store helpers, and Telegram isolated polling restart behavior for bot reinitialization failures.

## Best Possible Solution

Keep the OAuth and Telegram recovery fixes if they survive review, but replace the immediate idle-time lane reset with a guarded stale-recovery or pump path that preserves command-lane serialization and add focused regression coverage.

## Reproduction Assessment

Yes for the blocking review finding at source level: resetCommandLane clears activeTaskIds and drains queued work, while embedded-run idle transitions can occur before the command task has settled. I did not run tests because this review is read-only.

## Solution Assessment

No. The OAuth and Telegram recovery directions look maintainable, but the diagnostics fix should not clear the command lane from an ordinary idle diagnostic transition without proving the active lane task is stale.
