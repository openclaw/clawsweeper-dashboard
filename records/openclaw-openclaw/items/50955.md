## Summary

Keep this PR open. The underlying ACP thread-bound session problem is still real on current main, but the proposed patch is not mergeable as submitted because it drops Feishu advertised action handlers, opens a current-binding ownership bypass, and does not fully update bootstrap delivery routing for the new current-placement path.

## What This Changes

The PR adds shared ACP spawn-origin, placement, conversation-canonicalization, and bound-delivery helpers, updates Discord/Telegram/Feishu channel adapters plus session announce/send routing, and regenerates config/plugin SDK baselines.

## Best Possible Solution

Revise and rebase the PR before merge: keep one shared origin-aware placement resolver for native and slash ACP spawning, preserve Feishu action handlers, enforce the same boundBy ownership gate for current native ACP bindings, route bootstrap/completion through the binding-aware channel seam, preserve store-backed internal requester classification except for ACP's explicit external-delivery case, and update generated baseline artifacts to current main's sha256 layout with focused regression tests.

## Reproduction Assessment

Yes. The PR body gives concrete Telegram/Discord repro flows, and current main has static test evidence for the native Discord child-placement behavior plus source evidence for the slash-path ownership gate that native binding lacks.

## Solution Assessment

No. The direction is maintainable, but this patch is not the best mergeable solution until it preserves existing Feishu behavior, carries the ownership check into native current bindings, and updates bootstrap routing and generated baselines against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reinstate Feishu advertised action handlers:** `extensions/feishu/src/channel.ts:585`
  - body: The diff removes the Feishu handlers for pins, channel/member lookup, channel-list, and reactions after the edit branch, but the plugin still advertises those actions. Users can discover and invoke them, then every call falls through to unsupported-action behavior.
  - confidence: 0.93
- **[P1] Preserve bootstrap routing for current ACP bindings:** `src/agents/acp-spawn.ts:589`
  - body: This change allows placement: "current", but the bootstrap delivery plan still reconstructs a target from the bound conversation id through the generic delivery helper. For current Discord/Telegram surfaces that can prefer malformed targets like channel:user:<id> or channel:<chatId> over the requester origin, so the initial ACP task can be misrouted even after binding succeeds.
  - confidence: 0.86
- **[P1] Enforce ownership before current ACP rebinds:** `src/agents/acp-spawn.ts:589`
  - body: When placement resolves to current, this path binds immediately and records boundBy: "system"; unlike /acp spawn, it does not check an existing conversation binding's boundBy sender first. A different participant can therefore rebind an already-bound ACP DM/thread and clear the previous ownership metadata.
  - confidence: 0.88
- **[P2] Keep store-backed internal announce detection:** `src/agents/subagent-announce.ts:93-101`
  - body: Replacing getSubagentDepthFromSessionStore with a key-prefix check loses the stored spawnDepth/spawnedBy fallback used for legacy or alias requester sessions. Those internal sessions can be treated as external, causing orchestration completion messages to be delivered visibly to user channels.
  - confidence: 0.72
