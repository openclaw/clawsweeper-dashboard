## Summary

Keep open. Current main has not absorbed the PR's shared ACP origin/placement and bound-delivery seams, and the proposed patch is not mergeable as-is because it removes existing Feishu action handlers and opens a current-binding rebind path that lacks the sender ownership guard already present in `/acp spawn`.

## What This Changes

The PR adds shared ACP conversation canonicalization, origin-aware placement, channel-aware bound delivery, threaded announce routing, and related Discord/Telegram/Feishu tests and generated baselines.

## Best Possible Solution

Keep the PR open for contributor or maintainer follow-up. The best end state is a rebased, narrower patch that shares origin-aware placement between `/acp spawn` and native `sessions_spawn`, preserves existing channel action handlers, enforces the same `boundBy` ownership gate for current bindings, and routes bootstrap/completion through a binding-aware channel seam with focused Discord, Telegram, Feishu, and A2A regression coverage.

## Reproduction Assessment

Yes. The PR body provides concrete live repro scenarios, and current main has static/test proof: native ACP spawn still expects child placement for an existing Discord thread, while `/acp spawn` has a current-context resolver and owner gate that the native path lacks.

## Solution Assessment

No. The direction is maintainable, but this branch is not currently the best mergeable solution because it regresses Feishu advertised actions, does not preserve the current-binding ownership check, and needs binding-aware bootstrap/completion delivery before landing.

Full review comments:

- **[P1] Reinstate Feishu advertised action handlers:** `extensions/feishu/src/channel.ts:585`
  - body: The diff removes the Feishu handlers for pins, channel/member lookup, channel-list, and reactions after the `edit` branch, but the plugin still advertises those actions. Users can discover and invoke them, then every call falls through to unsupported-action behavior.
  - confidence: 0.93
- **[P1] Preserve bootstrap routing for current ACP bindings:** `src/agents/acp-spawn.ts:589`
  - body: This change allows `placement: "current"`, but bootstrap delivery still reconstructs the target from the bound conversation id through the generic delivery helper. For current Discord/Telegram surfaces that can prefer malformed targets over the requester origin, so the initial ACP task can be misrouted even after binding succeeds.
  - confidence: 0.86
- **[P1] Enforce ownership before current ACP rebinds:** `src/agents/acp-spawn.ts:589`
  - body: When placement resolves to `current`, this path binds immediately and records `boundBy: "system"`; unlike `/acp spawn`, it does not check an existing conversation binding's `boundBy` sender first. A different participant can therefore rebind an already-bound ACP DM/thread and clear the previous ownership metadata.
  - confidence: 0.88
- **[P2] Keep spawn-depth fallback for internal announce sessions:** `src/agents/subagent-announce.ts:93`
  - body: The diff classifies internal requester sessions only by key shape and cron state, dropping the stored `spawnDepth`/`spawnedBy` fallback that current main uses. Flat or legacy subagent sessions with stored depth can then be treated as external and post internal orchestration completions to user channels.
  - confidence: 0.78
