## Summary

Keep open: current main still has the TUI aborted-event path rendering only `run aborted`, while the validation failure is preserved upstream as an error tool result and can be hidden from the terminal surface.

## What This Changes

Surface tool-call JSON Schema validation failures in the TUI instead of showing only `run aborted` after repeated failed tool calls.

## Best Possible Solution

Propagate a sanitized last tool validation error or repeated-tool-error summary into the TUI abort surface, with focused coverage for aborted runs and no new config option.

## Reproduction Assessment

No live high-confidence reproduction was run, but the source path is reproducible: validation errors become error tool results while the TUI aborted branch renders only `run aborted`. A focused test can simulate an aborted run after a failed tool result.

## Solution Assessment

Yes, the best fix appears narrow: reuse existing tool-error summary or lifecycle metadata and render a sanitized one-line abort reason in the TUI. A broader provider/model retry-policy change would not address the terminal visibility gap directly.
