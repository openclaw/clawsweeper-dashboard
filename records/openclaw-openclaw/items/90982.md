## Summary

Keep open: current main and the latest release still render aborted TUI chat events as bare `run aborted`, and the open closing PR remains the active fix path rather than a reason to close the issue early.

## What This Changes

The issue requests surfacing a sanitized tool-call validation error in the TUI when a run aborts after repeated validation failures.

## Best Possible Solution

A single sanitized last-tool-error summary should reach the TUI abort line across the supported local and Gateway-backed paths without dumping received tool arguments.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live, but yes at source level: validation errors become stored tool-error state while current TUI abort rendering still emits only `run aborted`.

## Solution Assessment

Is this the best way to solve the issue? Yes, the maintainable direction is narrow error-summary propagation into the abort surface; retry policy, parser changes, or a new config option would be broader than the reported visibility bug.
