## Summary

Keep this PR open. It targets a real Telegram group-chat context gap, but the latest branch still does not solve the linked sibling-bot mention case and is stale against current main’s group prompt baseline.

## What This Changes

The PR adds optional bot-targeting metadata to auto-reply history/template context, renders it in inbound prompt metadata, and updates Telegram parsing/session plumbing plus tests to populate mention/reply bot fields.

## Best Possible Solution

Keep the additive metadata contract, but require Telegram to use a trustworthy sibling-bot source or a safer explicit unknown-other-bot state before claiming the linked repro is fixed. The final branch should rebase onto current main, update the group prompt byte-stability assertion, add the required changelog entry, and prove current-bot, sibling-bot, reply-to-bot, and ambient history behavior with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main stores mention-skipped Telegram group history without target metadata, and the PR still calls the new detector with only the current bot username for skipped history, so the @bot-b history case for bot-a remains unclassified.

## Solution Assessment

No. The core field shape is reasonable, but the proposed Telegram wiring is not yet the best complete fix because it cannot identify sibling-bot mentions in the main repro and the branch needs a rebase/test update for current main.
