## Summary

Keep open: current main still lacks the requested Telegram group-history targeting metadata, and the latest PR branch does not fully fix the linked sibling-bot repro because skipped-history detection only knows the current bot. The branch also needs a stale test expectation updated and a required changelog entry before merge.

## What This Changes

The PR adds optional auto-reply bot-targeting metadata and Telegram parsing/session plumbing to render mentioned/replied bot signals in group history prompts.

## Best Possible Solution

Keep the additive metadata contract, but finish Telegram with an approved sibling-bot username source, pass that source through skipped-history and session context, update docs/changelog as needed, and land focused tests for current-bot mentions, sibling-bot mentions, reply-to-bot context, and ambient history.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main stores mention-skipped Telegram group history without target metadata, and the PR still cannot classify a skipped @bot-b message for bot-a without a sibling-bot source.

## Solution Assessment

No. The core field shape is reasonable, but the proposed Telegram wiring is incomplete for the linked repro until it can identify sibling-bot mentions or represent that state explicitly; the stale test and changelog issues are smaller merge blockers.
