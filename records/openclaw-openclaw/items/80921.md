## What This Changes

The PR adds a 3-second timeout to the gateway `canBindToHost()` probe so a non-settling IPv6 loopback bind can fall back to IPv4-only startup.

## Best Possible Solution

Land a rebased narrow timeout or abort around the existing bind probe, with regression coverage for a never-settling probe and redacted real Big Sur gateway startup proof.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection shows the gateway can await `canBindToHost("::1")` indefinitely if neither socket event fires, but the Big Sur event behavior was not reproduced here.

## Solution Assessment

Yes, with proof. Bounding the existing bind probe is the narrowest maintainable fix and preserves non-Windows dual-loopback behavior when the probe succeeds; it should be paired with a regression test and real setup evidence.
