## What This Changes

The issue reports that `openclaw commitments` table columns overflow when truncated ID or scope values append a three-character ellipsis after reserving one character.

## Best Possible Solution

Land one narrow linked PR that uses a width-safe truncation suffix, preferably the existing single-character ellipsis convention plus a long ID/scope alignment regression test, then let this issue close through the merged fix.

## Reproduction Assessment

Yes from source inspection: the current helper returns 18 characters for a 16-character truncated ID, and that value is passed to fixed-width `padEnd()` columns. I did not run the full CLI because this was a read-only review, but the root path is deterministic.

## Solution Assessment

Yes, the best fix shape is narrow: keep the existing table and change truncation so returned cells never exceed `maxChars`, with a regression test for long ID and scope rows. The issue itself should remain open until one linked candidate PR merges.
