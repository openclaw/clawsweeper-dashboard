## Summary

Keep this PR open, but do not merge the current patch: the bug report is valid and the index/id accumulation direction is right, yet the patch can still lose later argument deltas because its maps keep pointing at tool-call objects after those objects are replaced in `output.content`.

## What This Changes

The PR changes `src/agents/openai-transport-stream.ts` to accumulate OpenAI-compatible streamed tool calls in maps keyed by `tool_call.index` and id, with per-block argument byte tracking.

## Best Possible Solution

Keep the index/id accumulation design, but either finalize tool-call blocks in place or refresh both maps whenever a block object is replaced, then add a focused regression for split parallel tool-call indices.

## Reproduction Assessment

Yes, by source inspection: a mocked OpenAI completions stream with two tool-call indices introduced in one chunk and later argument deltas for the first index exercises both current-main failure and the PR’s stale-map bug. I did not run tests because this review is read-only.

## Solution Assessment

No. Mapping streamed tool calls by index/id is the right solution direction, but this patch replaces block objects while maps still reference the old objects; the safer fix is in-place finalization or map refresh plus regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep tool-call maps pointed at live blocks:** `src/agents/openai-transport-stream.ts:1798`
  - body: When a second parallel tool-call slot is created, `finishCurrentBlock()` can replace the first block in `output.content`, but both maps still point at the old object. Later deltas for that index/id then mutate a detached block, `output.content.indexOf(block)` can become `-1`, and split arguments can still be lost. Finalize in place or refresh both maps whenever replacing a block.
