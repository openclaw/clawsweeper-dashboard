## Summary

Keep open. This is an active, focused bug-fix PR rather than a cleanup-close candidate: current main still has the single-block OpenAI-completions tool-call parser, while the PR matches the upstream index contract and existing pi-ai parser pattern with no blocking code or security finding found in this read-only review.

## What This Changes

The PR changes `src/agents/openai-transport-stream.ts` to accumulate OpenAI-completions streaming tool-call deltas by index/id with per-block byte accounting and adds a regression test for parallel split tool-call arguments.

## Best Possible Solution

Land the indexed OpenAI-completions tool-call accumulation on top of current main while preserving recent stream-yield/abort handling, then keep the separate Kimi anthropic-messages loop issue on its own evidence path.

## Reproduction Assessment

Yes for source-level reproduction: current main can switch tool-call blocks without considering `toolCall.index`, and the PR adds a mocked two-index stream that exercises split arguments. I did not run a live provider or local tests in this read-only review.

## Solution Assessment

Yes. Keying streamed tool-call deltas by the required numeric `toolCall.index` with an id fallback and per-block byte counts is the narrow maintainable fix, and it matches the existing pi-ai parser strategy.
