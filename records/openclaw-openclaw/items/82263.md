## What This Changes

The PR changes `src/agents/openai-transport-stream.ts` to accumulate streamed tool-call deltas by index/id with per-block byte accounting and adds a regression test for split parallel tool-call arguments.

## Best Possible Solution

Land the indexed OpenAI-completions tool-call accumulation with the focused regression test once the relevant CI/proof gates are reconciled; keep the linked Kimi anthropic-messages loop issue separate unless this parser fix is proven to cover it.

## Reproduction Assessment

Yes for source-level reproduction: current main can switch tool-call blocks without considering `toolCall.index`, and the added mocked stream exercises two split indices across chunks. I did not run a live provider or local tests in this read-only review.

## Solution Assessment

Yes. Keying streamed tool-call deltas by numeric `toolCall.index` with an id fallback is the narrow maintainable fix and matches both the OpenAI SDK stream type contract and the bundled pi-ai parser pattern.
