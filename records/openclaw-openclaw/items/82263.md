## Summary

Keep open: the reported parser bug is valid and the index/id accumulation direction is useful, but this PR still has a blocking stale-reference bug when a mapped tool-call block is replaced while the maps keep pointing at the old object.

## What This Changes

The PR changes the OpenAI completions streaming parser to accumulate streamed tool-call deltas by tool_call index/id instead of a single active block.

## Best Possible Solution

Keep the index/id accumulation design, but finalize tool-call blocks in place or refresh both maps whenever replacing a block, then add a focused regression for split parallel tool-call indices.

## Reproduction Assessment

Yes. By source inspection, a mocked OpenAI completions stream with two tool_call indices in one chunk and later argument deltas for the first index exercises both current-main misrouting and this PR’s stale-map bug.

## Solution Assessment

No. Mapping streamed tool calls by index/id is the right direction, but this implementation replaces block objects while the maps can still reference the old objects; in-place finalization or map refresh is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep tool-call maps pointed at live blocks:** `src/agents/openai-transport-stream.ts:1798`
  - body: When a second parallel tool-call slot is created, finishCurrentBlock() can replace the first block in output.content, but both maps still point at the old object. Later deltas for that index/id then mutate a detached block, output.content.indexOf(block) can become -1, and split arguments can still be lost. Finalize in place or refresh both maps whenever replacing a block.
