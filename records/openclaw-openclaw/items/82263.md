## Summary

Keep open. This PR is a focused bug fix for a real OpenAI-completions stream parser gap: current main still tracks one tool-call block, while the PR head keys streamed tool-call deltas by index/id and adds targeted regression coverage. I found no blocking diff-level correctness or security findings; merge readiness still depends on normal CI and maintainer review.

## What This Changes

The PR updates `src/agents/openai-transport-stream.ts` to accumulate streamed OpenAI-completions tool-call deltas by index/id with per-block byte accounting, and adds a regression test for split parallel tool-call arguments.

## Best Possible Solution

Land the indexed OpenAI-completions tool-call accumulation with the regression test once relevant CI/proof gates are reconciled; keep https://github.com/openclaw/openclaw/issues/71273 separate unless this parser fix is proven to cover that anthropic-messages Kimi loop.

## Reproduction Assessment

Yes for source-level reproduction: current main can switch tool-call blocks without considering `toolCall.index`, and the added mocked stream exercises two split indices across chunks. I did not rerun a live provider in this read-only review.

## Solution Assessment

Yes: keying streamed tool-call deltas by `toolCall.index` with an id fallback is the narrow maintainable fix, and it aligns with both openai-node's delta contract and the bundled pi-ai parser pattern.
