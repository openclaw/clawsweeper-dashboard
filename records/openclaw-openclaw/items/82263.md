## Summary

Keep open: this is a focused bug fix for a current-main parser path that still ignores streamed tool-call indices, and the latest head appears to address the earlier ClawSweeper blockers.

## What This Changes

The PR changes `src/agents/openai-transport-stream.ts` to accumulate OpenAI-completions streamed tool-call deltas by index/id and adds a regression test for split parallel tool-call arguments.

## Best Possible Solution

Land the indexed block accumulation with focused parser validation, keeping the fix in the OpenAI-completions transport rather than provider-specific Kimi code.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: a mocked `ChatCompletionChunk` stream with two `delta.tool_calls` indices in one chunk and split later argument deltas exercises the current-main single-`currentBlock` path without needing live provider access.

## Solution Assessment

Is this the best way to solve the issue? Yes: keying streamed tool-call accumulation by OpenAI's `toolCall.index` with id fallback matches the upstream type contract and the existing pi-ai parser pattern while preserving the current transport boundary.
