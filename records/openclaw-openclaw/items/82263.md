## Summary

This PR should stay open because the reported OpenAI-compatible completions stream bug is real by source inspection, but the current patch still has a correctness blocker: map entries can retain references to pre-finalized tool-call objects that are no longer in `output.content`, so later indexed deltas can still be dropped.

## What This Changes

The PR changes `src/agents/openai-transport-stream.ts` so streamed OpenAI-compatible tool calls are accumulated in maps keyed by `tool_call.index` and id instead of a single active tool-call block.

## Best Possible Solution

Keep the index/id accumulation design, but finalize tool-call blocks in place or refresh both maps whenever a block object is replaced, then add a focused stream fixture for interleaved parallel tool-call indices.

## Reproduction Assessment

Yes by source inspection: a mocked completions stream with two `tool_calls` indices introduced in one chunk and later argument deltas for the first index exercises the current parser failure. I did not run tests because this review is read-only.

## Solution Assessment

No. Mapping streamed tool calls by index/id is the right direction, but the current patch replaces block objects while maps still reference the old objects; a safer fix keeps mapped block objects stable or updates all map entries after finalization.
