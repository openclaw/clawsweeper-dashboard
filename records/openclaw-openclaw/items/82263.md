## Summary

Keep open. The PR addresses a real parser bug and includes sufficient before/after live-output proof, but the branch still replaces tool-call block objects while the new index/id maps keep references to the old objects, so later deltas can mutate detached blocks and lose arguments.

## What This Changes

The PR changes `src/agents/openai-transport-stream.ts` to accumulate streamed OpenAI completions tool-call blocks in maps keyed by `tool_call.index` and id, with per-block argument byte tracking.

## Best Possible Solution

Keep the index/id accumulation design, but finalize tool-call blocks in place or refresh both maps whenever a block object is replaced, then add a regression for split parallel tool-call indices.

## Reproduction Assessment

Yes. By source inspection, a mocked OpenAI completions stream with two tool-call indices introduced in one chunk and later argument deltas for the first index exercises both current-main failure and the PR branch’s stale-map bug; no tests were run because this review is read-only.

## Solution Assessment

No. Mapping streamed tool calls by index/id is the right direction, but this implementation should keep mapped blocks live through in-place finalization or map refresh, plus focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep tool-call maps pointed at live blocks:** `src/agents/openai-transport-stream.ts:1802`
  - body: When a second parallel tool-call slot is created, `finishCurrentBlock()` can replace the first block in `output.content`, but both maps still point at the old object. Later deltas for that index/id then mutate a detached block, `output.content.indexOf(block)` becomes `-1`, and split arguments can still be lost. Finalize in place or refresh both maps whenever replacing a block.
