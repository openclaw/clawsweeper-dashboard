## Summary

This PR targets a real OpenAI-completions streaming parser bug, and the updated branch fixes the prior stale-map problem, but it now leaves an internal `partialArgsBytes` counter on emitted tool-call blocks. That changes the output shape for ordinary tool calls and conflicts with existing parser expectations, so it should not merge as-is.

## What This Changes

The PR updates the OpenAI completions stream parser to accumulate streamed tool-call deltas by tool-call index/id and adds a regression test for split parallel tool-call arguments.

## Best Possible Solution

Keep the index/id accumulation design, but store per-block argument byte counts outside emitted tool-call blocks and add an exact regression that proves split parallel calls parse without leaking parser scratch fields.

## Reproduction Assessment

Yes. A mocked OpenAI-completions stream with two tool-call indices introduced together and later argument deltas for each index exercises the current-main parser bug; I did not run it because this review is read-only.

## Solution Assessment

No. Mapping tool calls by index/id is the right fix direction, but this implementation should keep byte-count scratch state in a side map or other non-emitted structure so the public tool-call block shape stays unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep byte counters out of emitted tool-call blocks:** `src/agents/openai-transport-stream.ts:1990`

Existing parser tests assert the exact emitted block shape without this field, and real transcripts/events would expose the counter, so keep the byte count in a side map or delete it before any emitted/final output can observe it.
