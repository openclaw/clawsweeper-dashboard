## Summary

Keep open. The bug report is plausible and the map-by-index direction is right, but this patch is not mergeable because the proposed finalization replaces output blocks while the maps still point at the old objects, so later deltas can still be dropped; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/agents/openai-transport-stream.ts` to accumulate streamed OpenAI-compatible tool calls in maps keyed by `tool_call.index` and id instead of a single current block.

## Best Possible Solution

Keep the index/id accumulation approach, but finalize tool-call blocks in place or refresh the maps whenever an output block is replaced, then add a focused stream regression and redacted real-run proof.

## Reproduction Assessment

Yes, by source inspection: a mocked completions stream with two tool-call indices introduced in one chunk and later argument deltas for the first index would exercise the reported parser failure. I did not execute it because this review is read-only, and the PR still needs real Kimi/openai-completions proof before merge.

## Solution Assessment

No. The intended map-by-index repair is the right shape, but replacing block objects while retaining map references is not a safe implementation; mirroring pi-ai's stable in-place block finalization is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep tool-call maps pointed at live blocks:** `src/agents/openai-transport-stream.ts:1798`
  - body: When a new parallel tool-call slot is first seen, `finishCurrentBlock()` replaces the previous block in `output.content`, but `toolCallBlocksByIndex` and `toolCallBlocksById` still point at the old object. Later deltas for that slot mutate an object no longer in the output and emit `contentIndex: -1`, so split parallel tool calls can still lose arguments. Finalize in place or update both maps when replacing the block.
  - confidence: 0.91
