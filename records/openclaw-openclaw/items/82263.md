## Summary

Keep open. The reported parser bug is plausible and the PR is aimed at the right code path, but the branch can still lose split arguments because it replaces finalized tool-call blocks while the index/id maps keep references to the old objects.

## What This Changes

The PR updates `src/agents/openai-transport-stream.ts` to accumulate OpenAI completions streaming tool-call chunks by `tool_call.index` and id with per-block argument byte accounting.

## Best Possible Solution

Keep the index/id accumulation design, but finalize tool-call blocks in place or refresh both maps whenever a block object is replaced, then add a regression for split parallel tool-call continuations.

## Reproduction Assessment

Yes. A mocked OpenAI completions stream with two tool-call indices introduced in one chunk and a later continuation for the first index follows both current main's single-block failure and the PR branch's stale-map failure by source inspection.

## Solution Assessment

No. Mapping streamed tool calls by index and id is the right direction, but this implementation should keep mapped blocks live through in-place finalization or map refresh and add focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep tool-call maps pointed at live blocks:** `src/agents/openai-transport-stream.ts:1802`
  - body: When a later tool-call slot is created, `finishCurrentBlock()` replaces the first block in `output.content`, but the index/id maps still point at the old object. A later delta for that index mutates the detached object and emits `contentIndex: -1`, so split arguments can still be lost. Finalize in place or refresh both maps when replacing a block.
