## Summary

Keep open: the xAI device-code direction is useful, but this branch is not merge-ready because it removes shipped xAI device-code entry points, retries rotating refresh grants on ambiguous responses, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes the bundled xAI provider so the `oauth` auth method runs device-code login, removes the separate xAI device-code method/choice, updates docs/tests, and adds refresh endpoint rediscovery/retry handling.

## Best Possible Solution

Make `oauth` use device-code only with a tested compatibility path for the shipped device-code entry points, and keep refresh-token retries limited to cases proven not to consume the token.

## Reproduction Assessment

Yes. Source inspection shows current CLI maps `--device-code` to method `device-code`, while this PR removes xAI's `device-code` auth method and `xai-device-code` manifest choice, so the shipped command path becomes unresolvable.

## Solution Assessment

No. Routing xAI OAuth through device-code is a plausible owner-boundary fix, but the safer merge shape preserves shipped entry points and narrows refresh retry behavior.
