## Summary

## Best Possible Solution

Keep the single visible xAI OAuth direction if maintainers want it, but preserve the shipped `--device-code` and `xai-device-code` aliases through a tested deprecation path and keep refresh-token retries limited to demonstrably non-origin challenge responses.

Full review comments:

- **[P1] Preserve xAI device-code entry points:** `extensions/xai/index.ts:177`
  - body: This leaves xAI with only the `oauth` auth method, but the shipped CLI still maps `openclaw models auth login --provider xai --device-code` to method `device-code`, and the current docs also publish `--auth-choice xai-device-code`. After this change those existing commands no longer resolve. Please keep the `device-code` method/manifest choice as a deprecated alias to the new implementation, or add an explicit tested migration path before removing it from docs.
  - confidence: 0.94
- **[P2] Do not retry ambiguous refresh-token responses:** `extensions/xai/xai-oauth.ts:311-312`
  - body: `refresh_token` grants now get three attempts, so 5xx/429 token responses resend the same refresh token. The PR notes xAI rotates refresh tokens, and OAuth allows the old token to be revoked when a replacement is issued; a server error after rotation can make the retry burn the credential and force re-login. Restrict retry to Cloudflare challenge responses that clearly did not reach the token service, or keep refresh grants fail-fast.
  - confidence: 0.8
