## Summary

Current main has the lifecycle-generation guard and regression coverage that prevent completed current-generation sessions from being reopened by restart recovery; the fix came from merged pull request https://github.com/openclaw/openclaw/pull/95472 and is main-only because v2026.6.10 still has the old predicate.

## What This Changes

The issue reports that a completed Telegram main session can be silently reset by restart recovery when current-generation lifecycle metadata is mistaken for stale restart evidence.

## Best Possible Solution

Keep the current main lifecycle-generation guard, close this issue as fixed on main, and handle the still-open overlapping pull request separately as superseded or duplicate cleanup.

## Reproduction Assessment

Yes at source level: the old predicate could match a completed session when a same-generation active run had observedAt greater than updatedAt, and current main now guards that path. I did not run a live Telegram overnight replay in this read-only review.

## Solution Assessment

Yes for the reported lifecycleGeneration false positive: the current main fix changes the implicated predicate without removing stale-generation recovery or explicit running-session recovery. Any broader desire to disable documented daily resets would be a separate product/config request.
