## Summary

Keep open: current main and v2026.6.9 still have the restart-recovery timing predicate that can mark a completed session from a same-generation candidate, and open candidate PRs now own the proposed fix. The issue should remain the canonical user report until one candidate lands or maintainers choose a different session-policy path.

## What This Changes

Prevent normally completed Telegram main sessions from being silently reopened/reset by current-generation restart-recovery metadata, preserving conversation context before the configured idle timeout.

## Best Possible Solution

Land exactly one narrow lifecycle-generation guard for restart-recovery timing, keep documented reset semantics separate, and close this issue through the merged fix.

## Reproduction Assessment

Yes at source level: current main can match a completed session when a same-generation restart-recovery candidate has observedAt greater than updatedAt. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

Yes for the central false-positive: lifecycleGeneration is the existing discriminator between current maintenance candidates and stale pre-restart candidates. The best current path is maintainer review and landing of one open candidate PR, not a new duplicate repair job.
