## Summary

Keep open: current main and v2026.6.8 still have the daily-plus-idle OR reset behavior and the implicit rollover path can preserve old restart-recovery metadata, so this is not safe to close as implemented, duplicate, or not-actionable.

## What This Changes

The issue reports that a long-lived Telegram session was silently reset before the configured idle window and then appeared to carry restart-recovery lifecycle metadata, causing lost conversation context.

## Best Possible Solution

Keep this as a focused Telegram/session-state tracker until maintainers decide the daily-plus-idle UX and either clear stale restart-recovery metadata on implicit rollover or route the behavior to an accepted adaptive/reset-policy change.

## Reproduction Assessment

Yes at source level, but not as a live Telegram replay: current main can rotate a session when daily-plus-idle freshness is stale, and the rotated patch does not clear restart-recovery metadata. The exact reported local timeline still needs live or log-level confirmation.

## Solution Assessment

Unclear as an automatic fix: clearing stale restart-recovery fields on rollover is narrow, but changing whether a session resets before the idle window expires conflicts with current documented daily-plus-idle semantics and needs maintainer direction.
