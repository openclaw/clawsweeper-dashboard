## Summary

Current main now prevents same-lifecycle restart-recovery candidates from reopening completed main sessions, and the regression is covered by a focused test; the fix also appears in the v2026.6.11-beta.1 prerelease while v2026.6.10 still had the old unguarded predicate.

## What This Changes

The issue reports Telegram main-session context loss when restart recovery silently reopens a normally completed session before the configured idle timeout.

## Best Possible Solution

Keep the lifecycle-generation guard and regression coverage now on main, and treat the still-open overlapping PR as separate duplicate cleanup rather than leaving this fixed issue open.

## Reproduction Assessment

Yes. The old v2026.6.10 source can match a completed session when `updatedAt < observedAt`, and current main now has a focused regression test showing the completed current-generation session is not reopened.

## Solution Assessment

Yes. The implemented fix is the narrow owner-boundary change: use the existing lifecycle generation to distinguish stale pre-restart candidates from current maintenance-expired candidates while preserving running-session recovery.
