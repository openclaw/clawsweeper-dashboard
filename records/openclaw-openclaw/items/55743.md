## Summary

Keep this open. Current main still renders assistant shell snippets as sanitized markdown with Copy-only code blocks, while the existing Control UI approval flow only resolves approval requests that were already created by exec/plugin runtime paths; adding Approve and Run is a new security-sensitive product/runtime feature.

## What This Changes

Request to render assistant-suggested shell commands in the Control UI as structured command cards with Copy and human-approved execution actions.

## Best Possible Solution

Add a structured command-card path only when explicit command metadata is present, bind it to the existing exec approval/systemRunPlan machinery, stream results back into chat, and document the security model.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is source-verifiable: assistant shell snippets render as Copy-only markdown, while approvals only resolve existing runtime-created approval records.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The safe direction is not to execute arbitrary markdown code blocks, but to introduce structured command metadata that reuses existing approval and node systemRunPlan binding.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the requested Control UI action would let browser-rendered assistant output initiate local or node-host command execution.

Concerns:

- **[high] Bind command cards to structured approvals:** `ui/src/ui/markdown.ts:435`
  - body: Any implementation should bind exact command text, argv, cwd, host/node, session, and approval identity before execution; approving arbitrary markdown code blocks would bypass the current queued approval and systemRunPlan safeguards.
  - confidence: 0.9
