## Summary

Current main has clipboard support for assistant code blocks and a separate exec-approval path for real execution requests, but it does not yet implement the requested Control UI bridge from assistant-suggested shell commands to an approved run action. The item should stay open for maintainer product and security design because executing commands derived from chat text is a security boundary, not a narrow bug fix.

## What This Changes

Request a Control UI command-card experience for assistant-suggested shell commands with Copy, Approve and Run, visible command metadata, and risk-aware approval safeguards.

## Best Possible Solution

Design this as a narrow extension of the existing exec approval and Codex command-execution contracts: command cards should run only exact, structured command proposals with visible cwd/host/risk metadata and approval records that preserve current systemRunPlan binding.

## Reproduction Assessment

Yes, from source inspection: assistant code blocks currently expose copy behavior, while executable approvals are driven by explicit approval events. I did not find a current-main path that turns an ordinary assistant-suggested shell block into an approved run action.

## Solution Assessment

Unclear until maintainers choose the product/security contract. The maintainable direction is to reuse the existing exec approval and Codex approval-backed execution path, not infer executable commands directly from rendered chat text.
