## Summary

Keep open. Current main supports copying assistant code blocks and separately resolving explicit exec approval events, but it still does not turn assistant-suggested shell command text into a structured Control UI approve-and-run action. The request crosses a command-execution security boundary, so the remaining work needs maintainer product/security design rather than an automatic fix PR.

## What This Changes

Request a Control UI command card for assistant-suggested shell commands with copy, approve-and-run, cancel, visible command metadata, and risk classification.

## Best Possible Solution

Design a structured command-proposal path that reuses the existing exec approval/systemRunPlan contract, shows exact command/cwd/host/risk metadata, and never treats arbitrary markdown code blocks as executable without explicit structured intent.

## Reproduction Assessment

Yes for source behavior: current main shows assistant code blocks only support copy, while actual command execution approvals are driven by explicit `exec.approval.requested` events. There is no current-main bridge from suggested shell text to an approved run action.

## Solution Assessment

Unclear until maintainers choose the product/security boundary. The maintainable direction is a structured command proposal that feeds the existing approval contract, not markdown scraping or a direct run button on arbitrary code blocks.
