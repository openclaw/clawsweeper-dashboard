## Summary

Keep this open. Current main still renders assistant shell snippets as copy-only markdown, while the existing approval UI only resolves already-created exec/plugin approval records; adding browser-initiated command execution needs product and security design.

## What This Changes

Request a Control UI command card for assistant-suggested shell commands with Copy, Approve and Run, Cancel, target metadata, risk labels, and approval safeguards.

## Best Possible Solution

Add a structured command-proposal path backed by existing exec approval and systemRunPlan binding, with conservative metadata, output streaming, tests, and security docs.

## Reproduction Assessment

Yes. Source inspection shows assistant code fences render as copy-only markdown and the current approval modal only resolves pending gateway/plugin approval records.

## Solution Assessment

Unclear until maintainers define the security contract. The safer direction is structured command metadata that reuses exec approvals and systemRunPlan, not executing arbitrary markdown code blocks.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would let the browser Control UI initiate local or node-host shell execution from assistant output.

Concerns:

- **[high] Bind approved runs to structured metadata:**
  - body: An implementation should not execute arbitrary markdown code blocks directly; command text, cwd, host/node, agent, session, and approval id need to be bound before execution.
  - confidence: 0.9
- **[medium] Keep risk labels conservative:**
  - body: Read-only, destructive, network, and config labels can mislead if inferred only from shell text, so uncertain commands should take the strongest approval path.
  - confidence: 0.78
