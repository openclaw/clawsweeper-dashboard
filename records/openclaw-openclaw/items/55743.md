## Summary

Keep this issue open. Current main still provides Copy-only markdown code blocks in chat, while the existing exec approval UI only resolves pending approval events created by gateway/node exec paths; adding Approve and Run requires security-sensitive product design.

## What This Changes

Request Control UI command cards for assistant-suggested shell commands with Copy and Approve and Run actions, target metadata, and risk labels.

## Best Possible Solution

Add a structured command-card path only for explicit command metadata, reuse exec approval and node systemRunPlan binding, stream results back into chat, and document the security model.

## Reproduction Assessment

Yes. A source-level check shows assistant shell snippets render through sanitized markdown into Copy-only code blocks, and the chat click handler only copies the command text.

## Solution Assessment

Unclear until maintainers settle the product/security contract. The maintainable direction is to reuse the existing exec approval and systemRunPlan machinery rather than executing arbitrary markdown code blocks directly.

## Security Review

Status: needs_attention

Summary: This request is security-sensitive because it would let the browser Control UI initiate local or node-host shell execution from assistant output.

Concerns:

- **[high] Bind approved runs to structured execution metadata:**
  - body: An implementation should not execute arbitrary markdown code blocks directly; the exact command, cwd, host/node, session, and approval record need to be bound before execution.
  - confidence: 0.88
- **[medium] Avoid optimistic risk labels from shell text alone:**
  - body: Read-only, destructive, network, and config labels can mislead if inferred only from command text, so uncertain commands should take the strongest approval path.
  - confidence: 0.78
