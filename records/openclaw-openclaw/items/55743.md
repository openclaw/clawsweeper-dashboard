## Summary

Keep this open. Current main still renders assistant shell snippets as ordinary sanitized markdown code blocks with Copy-only behavior, while the existing exec approval UI only resolves already-pending approval events and does not create or launch structured assistant-suggested command cards.

## What This Changes

Request to add Control UI command cards for assistant-suggested shell commands with Copy, Approve and Run, Cancel, target metadata, and risk labels.

## Best Possible Solution

Design a narrow structured command-card path for assistant output that carries exact command text, cwd, host/node, session, and execution target metadata; reuse the existing exec approval and `systemRunPlan` machinery for execution, keep Copy as a non-executing fallback, stream results back into chat, and document the security model.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an assistant message with a shell fenced code block is rendered through `toSanitizedMarkdownHtml`, produces only `.code-block-copy`, and the click handler only writes to the clipboard.

## Solution Assessment

Unclear until maintainers settle the security and product contract. The requested UX belongs in Control UI, but the maintainable implementation should use structured metadata and existing approval plumbing rather than executing arbitrary markdown code blocks.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would let the browser Control UI initiate host or node shell execution from assistant-proposed content.

Concerns:

- **[high] Bind approved runs to structured execution metadata:** `docs/gateway/protocol.md:531`
  - body: An implementation should not execute arbitrary markdown code blocks directly. It needs exact command, cwd, host/node, session, and approval record binding so the executed payload cannot drift from what the user approved.
  - confidence: 0.9
- **[medium] Classify command risk conservatively:**
  - body: Read-only, destructive, network, and config labels could be unsafe if inferred only from shell text. Unknown or ambiguous commands should route through the strongest approval path.
  - confidence: 0.78
