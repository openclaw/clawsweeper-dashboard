## Summary

Keep open because current main implements configurable text-only inbound debouncing, but media-aware delayed aggregation for image/attachment plus text remains explicitly excluded or lossy in shared policy and several channel handlers.

## What This Changes

The issue requests configurable delayed batching so multi-part user inputs, especially image or attachment followed by text, reach the agent as one conversation turn.

## Best Possible Solution

Extend the existing `messages.inbound` contract into a documented media-aware batching policy that preserves ordered parts, attachments, sender/session keys, dedupe IDs, and latest-message reply identity across representative channels.

## Reproduction Assessment

Yes, source inspection gives a high-confidence way to observe the missing behavior: configure non-zero `messages.inbound`, then send an attachment or image followed quickly by text on Discord, Slack, or Microsoft Teams. Current main bypasses debounce for media or clears attachments in synthetic combined turns.

## Solution Assessment

No for the literal new `messageAggregationDelayMs` key; current main already has `messages.inbound.debounceMs` and `messages.inbound.byChannel`. The maintainable path is to extend that contract for media-aware batching rather than add a parallel channel option.
