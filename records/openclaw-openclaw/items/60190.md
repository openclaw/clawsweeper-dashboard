## Summary

Keep open. Current main has a shared inbound debounce config and a few channel-specific coalescing paths, but the common policy and several mainstream channel handlers still treat media-bearing messages as immediate single turns, so image-plus-text aggregation remains only partially implemented.

## What This Changes

Add configurable per-channel delayed aggregation so related multi-part inbound user messages, especially media followed by text, can be processed as one conversation turn.

## Best Possible Solution

Extend the existing `messages.inbound` debouncer into a documented media-aware contract, batching by channel, conversation, and sender while preserving text plus media metadata and using the latest message for reply identity. The change should include focused coverage for QQ Bot or another media-heavy channel plus a shared contract test for media-plus-text batching.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is to send a media-bearing message followed quickly by text on Discord, Slack, Signal, legacy iMessage, or QQ Bot DM; current code either bypasses debounce because `hasMedia` is true or drains the QQ queue immediately.

## Solution Assessment

Unclear. The requested per-channel delay is directionally right, but the maintainable solution should probably extend the existing `messages.inbound` contract rather than introduce a separate QQ-style `messageAggregationDelayMs` surface.
