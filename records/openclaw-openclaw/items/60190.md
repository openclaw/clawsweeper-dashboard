## Summary

Keep open. Current main has `messages.inbound` debounce config and a few channel-specific coalescers, but the shared policy and mainstream channel handlers still exclude or lose media-bearing messages, so image-plus-text aggregation remains only partially implemented.

## What This Changes

Request configurable per-channel delayed aggregation so rapid multi-part user messages, especially media followed by text, reach the agent as one turn.

## Best Possible Solution

Extend the existing `messages.inbound` debouncer into a documented media-aware batching contract that preserves text, media metadata, dedupe IDs, and latest-message reply identity, then ship focused coverage for at least one media-heavy channel and shared contract behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a media-bearing message followed quickly by text on Discord, Slack, Signal, or QQ Bot: current code either bypasses debounce through `hasMedia` or drains the queue immediately.

## Solution Assessment

Unclear. The requested per-channel delay is directionally right, but current main already has `messages.inbound.byChannel`; the maintainable fix should extend that contract instead of adding a separate `messageAggregationDelayMs` surface.
