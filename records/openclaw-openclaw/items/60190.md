## Summary

Close as duplicate/superseded because the remaining media-aware delayed batching problem is already tracked by the open canonical media buffering issue, while current main only covers text-only inbound debounce and still excludes media from the shared debounce path.

## What This Changes

The issue requests configurable per-channel delayed aggregation so rapid multi-part user messages, especially image plus text, reach the agent as one turn.

## Best Possible Solution

Keep the canonical media batching work in https://github.com/openclaw/openclaw/issues/39343 and coordinate narrower channel-specific follow-ups there rather than maintaining this overlapping tracker.

## Reproduction Assessment

Yes, source-level. Current main excludes media-bearing messages from shared inbound debounce, so sending media followed quickly by text on a channel using that policy still exercises the missing aggregation behavior.

## Solution Assessment

Yes for cleanup closure. The best solution is not a separate `messageAggregationDelayMs` tracker, but extending the existing `messages.inbound` and channel media batching contracts under the canonical media buffering issue.
