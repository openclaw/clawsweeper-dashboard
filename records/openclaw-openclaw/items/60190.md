## Summary

Keep open. Current main has configurable text-only inbound debouncing, but the shared policy, tests, and docs still exclude media, and several channel handlers either bypass debounce for attachments or drop attachments when synthesizing a combined turn.

## What This Changes

Request media-aware delayed batching of rapid same-sender multi-part channel messages, ideally configurable per channel.

## Best Possible Solution

Extend the existing `messages.inbound` debounce contract into documented media-aware batching that preserves attachments, sender/session keys, dedupe IDs, and latest-message reply identity, then cover shared policy plus at least one media-heavy channel.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `messages.inbound`, then send an image or attachment followed quickly by text on Discord, Slack, Signal, or Microsoft Teams; current main bypasses debounce for media or strips attachments in the synthetic combined turn.

## Solution Assessment

Unclear. The requested behavior is directionally useful, but adding a new `messageAggregationDelayMs` surface is not the narrowest path because current main already has `messages.inbound.byChannel`; the maintainable fix should extend that contract for media.
