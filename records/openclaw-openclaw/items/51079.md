## Summary

Keep this PR open. Current main and the latest release still reuse the persistent session id as the node `agent.request` run id, while the PR is a narrow runtime fix that matches the existing voice-ingress pattern; the remaining actionable gap is the required changelog entry for a user-facing gateway fix.

## What This Changes

The PR changes gateway node `agent.request` handling to pass a fresh UUID run id into `agentCommandFromIngress` and updates the focused gateway test to assert the run id differs from the session id.

## Best Possible Solution

Add the missing changelog entry, then land this focused gateway fix so each node-originated agent request gets a per-run UUID while preserving the existing session id, session key, delivery route, and attachment handling semantics.

## Reproduction Assessment

Yes. A code-level reproduction exists on current main: invoke `handleNodeEvent` with an `agent.request` payload for a stored session and the mocked `agentCommandFromIngress` receives `opts.runId === opts.sessionId`; the current test asserts that old behavior.

## Solution Assessment

Yes, with one process fix. Generating `randomUUID()` at the node ingress boundary is the narrowest maintainable runtime change and matches the established `voice.transcript` path; the PR should also add the required changelog entry before merge.
