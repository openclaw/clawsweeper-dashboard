## Summary

Keep open: current main still has the run-id reuse that this PR targets, while the proposed diff is narrow, matches the existing voice ingress pattern, and does not introduce a concrete functional or security blocker.

## What This Changes

The PR changes node `agent.request` handling to pass a fresh `randomUUID()` as `runId` to `agentCommandFromIngress` and updates the gateway test to assert the run id differs from the persistent session id.

## Best Possible Solution

Land this focused gateway fix, or an equivalent patch, so each node `agent.request` gets a per-run UUID while preserving the existing session id, session key, delivery route, and attachment handling semantics.

## Reproduction Assessment

Yes. A code-level reproduction is available on current main: call `handleNodeEvent` with an `agent.request` payload for a stored session and the mocked `agentCommandFromIngress` receives `opts.runId === opts.sessionId`; the current test also asserts that old behavior.

## Solution Assessment

Yes. Generating `randomUUID()` at the node ingress boundary is the narrowest maintainable fix and matches the established `voice.transcript` path; no broader gateway API change is needed.
