## Summary

Keep open: current main and the latest release still reuse the session id as the node `agent.request` run id, while this PR’s runtime/test change is the right narrow fix; the only concrete blocker is the missing required changelog entry.

## What This Changes

The PR changes gateway node `agent.request` handling to pass a fresh UUID run id into `agentCommandFromIngress` and updates the focused gateway test to assert the run id differs from the session id.

## Best Possible Solution

Add the missing changelog entry, then land this focused gateway fix so each node-originated agent request gets a per-run UUID while preserving session, delivery-route, and attachment semantics.

## Reproduction Assessment

Yes, at source level. On current main, invoking `handleNodeEvent` with an `agent.request` payload for an existing session reaches `agentCommandFromIngress` with `opts.runId === opts.sessionId`, and the current focused test asserts that old behavior.

## Solution Assessment

Yes, with one process fix. Generating `randomUUID()` at the node ingress boundary is the narrowest maintainable runtime change and matches the established `voice.transcript` path; the PR still needs the required changelog entry.
