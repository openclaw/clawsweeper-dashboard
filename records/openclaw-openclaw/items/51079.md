## Summary

Keep open. Current main still reuses the session id as the node `agent.request` run id, so the bug remains; this PR is the right narrow runtime/test fix but still needs the required changelog entry before merge.

## What This Changes

The PR changes node `agent.request` handling to pass a fresh UUID run id to `agentCommandFromIngress` and updates the focused gateway test to assert the run id differs from the session id.

## Best Possible Solution

Add the missing `CHANGELOG.md` entry, then land this focused gateway fix so each node-originated agent request gets a per-run UUID while preserving session, delivery-route, and attachment semantics.

## Reproduction Assessment

Yes, at source level. On current main, an `agent.request` payload for an existing session reaches `agentCommandFromIngress` with `opts.runId === opts.sessionId`, and the focused test asserts that old behavior.

## Solution Assessment

Yes, with one process fix. Generating `randomUUID()` at the node ingress boundary is the narrowest maintainable runtime change and matches the established `voice.transcript` path; the PR still needs the required changelog entry.
