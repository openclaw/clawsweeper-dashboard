## Summary

Keep open: current main and the latest release still reuse the session id as the run id for node `agent.request`, while this PR's runtime and focused test change are the right narrow fix. It still needs the required changelog entry before merge.

## What This Changes

The PR changes node-originated gateway `agent.request` handling to pass a fresh UUID run id and updates the focused gateway test to assert it differs from the session id.

## Best Possible Solution

Add the missing `CHANGELOG.md` entry, then land the focused gateway fix so each node-originated agent request gets a distinct run id while preserving session and delivery-route semantics.

## Reproduction Assessment

Yes, at source level. On current main, `handleNodeEvent` for an `agent.request` payload reaches `agentCommandFromIngress` with `opts.runId === opts.sessionId`, and the focused test currently asserts that equality.

## Solution Assessment

Yes, with one process fix. Generating `randomUUID()` at the node ingress boundary is the narrowest maintainable runtime change and matches the established `voice.transcript` path; the PR still needs the required changelog entry.
