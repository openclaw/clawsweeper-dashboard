## Summary

Keep open: current main still reuses the session id as the run id for `agent.request`, and the PR’s runtime/test change is the right narrow fix, but it still needs the required changelog entry before merge.

## What This Changes

The PR changes gateway node `agent.request` dispatch to pass a fresh UUID run id to `agentCommandFromIngress` and updates the focused test to assert the run id differs from the session id.

## Best Possible Solution

Add the missing `CHANGELOG.md` entry, then land this focused gateway fix so each node-originated agent request gets a per-run UUID while preserving session and delivery semantics.

## Reproduction Assessment

Yes, at source level. On current main, an `agent.request` payload for an existing session reaches `agentCommandFromIngress` with `opts.runId === opts.sessionId`, and the focused test asserts that old behavior.

## Solution Assessment

Yes, with one process fix. Generating `randomUUID()` at the node ingress boundary is the narrowest maintainable runtime change and matches the established `voice.transcript` path; the PR still needs the required changelog entry.
