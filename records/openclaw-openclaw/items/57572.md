## Summary

Keep open: current main does not implement this capability, and this draft external PR still has blocking correctness issues plus missing real behavior proof for the claimed stale-socket/prompt-cache improvement.

## What This Changes

The PR adds global and per-channel channel-health restart-mode config and threads it into gateway startup/reload so stale-channel recovery can use a graceful stop/start path.

## Best Possible Solution

If maintainers accept the capability, implement a schema-backed and documented restart-mode contract that actually preserves the intended runtime state, add focused coverage, and require redacted live stale-socket proof before merge.

## Reproduction Assessment

Partially yes. Source inspection of current main shows the health monitor uses a stop/reset/start path for unhealthy channels, but there is no high-confidence live reproduction of the Anthropic prompt-cache cost spike or after-fix improvement.

## Solution Assessment

No. The proposed branch is not mergeable as-is because the new config keys are not accepted by strict schemas, the graceful path still goes through a start path that clears restart attempts, and typed ChannelManager doubles are not updated.
