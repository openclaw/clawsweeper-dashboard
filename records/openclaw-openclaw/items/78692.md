## Summary

Keep open, but do not merge as-is: the patch removes a documented and client-planned trusted-proxy password fallback without aligning the client credential path, public docs, or maintainer security policy, and the PR only provides test/typecheck proof rather than real behavior proof.

## What This Changes

The PR removes the trusted-proxy local-direct password fallback, changes gateway auth tests to expect rejection, and adds a changelog entry.

## Best Possible Solution

Have secops/maintainers decide the trusted-proxy policy, then align server auth, local client credential resolution, docs, changelog, security audit messaging, and migration guidance around either preserving the compatibility path or removing it deliberately.

## Reproduction Assessment

Yes for source-level reproduction: current main has the fallback branch in `src/gateway/auth.ts`, existing tests assert password success, and docs describe the same path. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. Removing only the server branch is not the best fix because current docs and client credential resolution still rely on the password path; the maintainable solution needs an explicit security policy decision and a coordinated update across all affected surfaces.
