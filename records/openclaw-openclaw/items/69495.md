## Summary

Keep open: current main still has the heartbeat single-model limitation, and this PR is the canonical implementation thread, but the branch is not merge-ready because it still needs real behavior proof and has two fallback-semantics defects.

## What This Changes

The PR widens `agents.*.heartbeat.model` from a string to `{ primary, fallbacks }`, threads heartbeat fallback overrides through reply execution, updates pricing metadata, docs/help/generated schema, tests, and changelog.

## Best Possible Solution

Land a rebased implementation that widens the heartbeat model contract, preserves explicit `[]` versus inherited fallbacks, covers direct heartbeat calls, regenerates docs/schema outputs, and includes redacted live or terminal proof of fallback behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live provider run: current main rejects object-form heartbeat models and only forwards a single heartbeat model override, while the PR defects are visible against the existing `undefined` versus `[]` fallback contract.

## Solution Assessment

No, not as currently written. The requested direction matches existing model-list config, but this patch should preserve explicit empty fallback overrides, forward fallback lists for direct heartbeat calls, rebase onto current main, and add real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve explicit empty heartbeat fallbacks:** `src/infra/heartbeat-runner.ts:1035`
  - body: The new option is emitted only when the sanitized fallback list is non-empty. Downstream fallback resolution treats `undefined` as inherit and `[]` as an explicit no-fallback override, so `{ primary, fallbacks: [] }` still inherits agent-level fallbacks despite the documented disable semantics.
  - confidence: 0.95
- **[P2] Forward fallback lists for direct heartbeat calls:** `src/auto-reply/reply/get-reply.ts:203`
  - body: This fallback path resolves the object-form heartbeat primary when `heartbeatModelOverride` is absent, but it never derives `heartbeatModelFallbacks`. Direct `getReplyFromConfig(..., { isHeartbeat: true })` callers that rely on `heartbeat.model` fallbacks will run only the primary and lose failover.
  - confidence: 0.86
