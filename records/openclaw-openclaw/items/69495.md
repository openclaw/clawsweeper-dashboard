## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve empty heartbeat fallback overrides:** `src/infra/heartbeat-runner.ts:1038`
  - body: The PR body/docs say `fallbacks: []` disables the inherited chain, but this only forwards `heartbeatModelFallbacks` when the list is non-empty. An explicit empty array therefore reaches fallback resolution as `undefined`, so heartbeats can still use agent/default fallback providers the user tried to disable. Forward the override whenever the `fallbacks` property is present, and add a regression test for the empty array case.
  - confidence: 0.95
- **[P1] Update model-ref scanners for heartbeat object config:** `src/config/zod-schema.agent-runtime.ts:30`
  - body: Changing `HeartbeatSchema` to accept `AgentModelSchema` makes object primary/fallback refs valid config, but shared model-ref collection still treats `heartbeat.model` as a string-only slot and config write-prep only normalizes it as a string. Provider plugin auto-enable/startup validation/doctor/auth-profile paths that depend on those scanners can miss fallback-only providers. Treat `heartbeat.model` like the other model-list configs and cover it in the shared tests.
  - confidence: 0.88
- **[P2] Carry config fallback values through the compat path:** `src/auto-reply/reply/get-reply.ts:206`
  - body: This fallback path now resolves the primary from object-form `agentCfg?.heartbeat?.model`, but it does not also pass that object's fallback list into the run. Any heartbeat caller that relies on the documented config fallback instead of `heartbeatModelFallbacks` gets the primary without the configured fallbacks, creating a second source of behavior for the same config shape.
  - confidence: 0.78
- **[P3] Remove the normal-PR changelog edit:** `CHANGELOG.md:8`
  - body: `CHANGELOG.md` is release-owned in this repo. Keep the release-note context in the PR body or eventual squash/commit message, and leave changelog generation to the release flow.
  - confidence: 0.92
