## Summary

Keep open: current main still lacks heartbeat object-form fallback support, but this branch needs correctness fixes before it can merge.

## What This Changes

The PR widens `agents.*.heartbeat.model` to accept `{ primary, fallbacks }`, threads heartbeat fallback lists into reply/model fallback execution, and updates pricing, schema/help, docs, tests, and changelog.

## Best Possible Solution

Land a rebased implementation that treats heartbeat fallback lists as an explicit per-run override, keeps string compatibility, updates generated schema/docs/pricing/tests, and proves a redacted real heartbeat fallback path.

## Reproduction Assessment

Yes, at source level: current main's heartbeat schema/type are string-only and `runHeartbeatOnce` forwards only `heartbeatModelOverride` with no heartbeat-specific fallback list. I did not reproduce a live provider-429 failover path.

## Solution Assessment

No, not as currently written. The object-form direction matches the existing model config contract, but the patch needs empty-array preservation, direct-call fallback forwarding, a rebase onto current fallback plumbing, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve explicit empty heartbeat fallbacks:** `src/infra/heartbeat-runner.ts:1038`
  - body: The PR documents `fallbacks: []` as disabling the fallback chain, and the downstream fallback contract treats `[]` differently from `undefined`. Because this line only passes `heartbeatModelFallbacks` when the list is non-empty, an explicit empty array is dropped and the run can still inherit configured fallbacks.
  - confidence: 0.95
- **[P2] Forward heartbeat fallback lists for direct calls:** `src/auto-reply/reply/get-reply.ts:205-207`
  - body: This fallback path now resolves the object-form heartbeat primary from defaults when no runner override is supplied, but it never derives the matching fallback list. Direct heartbeat callers using object-form defaults would run only the primary and lose the configured failover chain.
  - confidence: 0.86
