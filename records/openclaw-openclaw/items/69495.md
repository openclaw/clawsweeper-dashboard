## Summary

Keep open: current main still lacks heartbeat object-form fallback support, but this PR is not merge-ready because it drops explicit empty fallback overrides, misses fallback forwarding on a direct heartbeat path, and has no after-fix real behavior proof.

## What This Changes

The PR widens `agents.*.heartbeat.model` to accept `{ primary, fallbacks }`, threads heartbeat fallback lists through reply execution, and updates pricing, schema/help, docs, tests, and changelog.

## Best Possible Solution

Land a rebased implementation that widens heartbeat model config, preserves explicit `[]` fallback semantics, forwards fallback overrides through every heartbeat reply path, updates current schema/docs/pricing/tests, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main has a string-only heartbeat schema/type and forwards only one heartbeat model override; I did not establish a live provider-429 failover path.

## Solution Assessment

No, not as currently written. The direction matches the existing model-list contract, but the patch needs empty-array preservation, direct-call fallback forwarding, a rebase to current config/schema surfaces, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve explicit empty heartbeat fallbacks:** `src/infra/heartbeat-runner.ts:1038`
  - body: The runner only sets `heartbeatModelFallbacks` when the sanitized list is non-empty. Downstream fallback resolution treats `undefined` as inherit and `[]` as an explicit override, so `{ primary, fallbacks: [] }` can still inherit configured fallbacks despite the documented disable semantics.
  - confidence: 0.95
- **[P2] Forward heartbeat fallback lists for direct calls:** `src/auto-reply/reply/get-reply.ts:203`
  - body: This fallback path resolves the object-form heartbeat primary when `heartbeatModelOverride` is absent, but it never derives `heartbeatModelFallbacks`. Direct `getReplyFromConfig(..., { isHeartbeat: true })` callers using object-form config will run only the primary and lose failover.
  - confidence: 0.86
