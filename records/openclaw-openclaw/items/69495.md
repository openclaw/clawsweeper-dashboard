## Summary

Keep this PR open. Current main still does not implement heartbeat `{ primary, fallbacks }` support, and the PR is the active implementation lane, but the proposed patch has merge-blocking behavior gaps around explicit empty fallback arrays and direct heartbeat reply calls, plus a changelog policy miss.

## What This Changes

The PR branch changes heartbeat model config to accept `{ primary, fallbacks }`, threads heartbeat fallback overrides through reply execution, updates pricing-cache/config-schema/docs/help/changelog surfaces, and adds heartbeat model override tests.

## Best Possible Solution

Land a rebased version that reuses the shared model schema/type for heartbeat config, preserves `undefined` versus explicit empty fallback arrays end to end, derives fallbacks for both runner and direct heartbeat reply paths, updates pricing/docs/generated schema/help/tests, and keeps the shipped docs aligned with actual retry semantics.

## Reproduction Assessment

Yes. Current main can be verified from source by checking that the schema/type/docs are string-only and that `runHeartbeatOnce` only forwards `heartbeatModelOverride`; the PR defects are reproducible by comparing `fallbacks: []` with downstream `fallbacksOverride !== undefined` behavior and by following the direct WhatsApp heartbeat caller.

## Solution Assessment

No. The PR chooses the right product direction, but the current patch is not yet the best implementation because it drops explicit empty fallback overrides, misses direct heartbeat callers, and needs current-main fallback-helper adaptation plus changelog attribution.
