Current main still has string-only heartbeat model config, and this PR is the canonical implementation thread for https://github.com/openclaw/openclaw/issues/69434, but the branch is not merge-ready because it drops explicit empty fallback overrides, omits direct heartbeat fallback plumbing, lacks real behavior proof, and is currently conflicting with main.

The PR extends `agents.*.heartbeat.model` to accept `{ primary, fallbacks }` and threads heartbeat fallback models through schema, runtime reply/run plumbing, pricing metadata, docs, tests, and changelog.

## Best Possible Solution

Rebase onto current main, implement the object form through the current schema/runtime fallback seams, preserve `undefined` versus explicit `[]`, cover direct heartbeat calls, regenerate docs/schema outputs, and attach redacted real runtime proof.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Current main still rejects object-form heartbeat models and forwards only one heartbeat model string, while the PR defects are visible against the existing fallback contract for `undefined` versus `[]`.

## Solution Assessment

No. The product direction is the narrow maintainable fix, but this patch is not the best mergeable version until it preserves empty fallback overrides, forwards direct heartbeat fallback lists, and rebases onto current main’s runtime/schema layout.
