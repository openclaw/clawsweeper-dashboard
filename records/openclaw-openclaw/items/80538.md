## Summary

Keep open: current main has the intended WhatsApp allowlist access checks, but the active web listener still passes a startup-captured config into DM access control while `channels.whatsapp` config changes are marked no-op for channel reload, matching the reported hot-reload bypass. The issue discussion also already identified this same focused failure mode and fix direction.

## What This Changes

Honor hot-reloaded WhatsApp `dmPolicy: "allowlist"` and `allowFrom` changes so non-allowlisted inbound DMs stop reaching the AI without requiring a gateway restart.

## Best Possible Solution

Make the active WhatsApp listener consult the current runtime config, or restart the WhatsApp channel when DM access-control config changes, and add regression coverage for hot-reloading from permissive to restrictive allowlist.

## Reproduction Assessment

Yes, we have a high-confidence source reproduction path: start the WhatsApp listener with permissive config, hot-reload `channels.whatsapp.dmPolicy="allowlist"` plus a restrictive `allowFrom`, then an existing listener continues passing its startup `cfg` into `checkInboundAccessControl`. I did not execute a live WhatsApp run in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to refresh the config used by the active listener's DM access-control path or make WhatsApp config changes restart the channel. Changing allowlist semantics or adding new policy knobs would be broader than the reported bug.
