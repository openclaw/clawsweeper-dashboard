## Summary

Keep open. Current main still has a source-level path that matches the report: WhatsApp config changes are classified as a reload no-op while inbound DM access control evaluates a config snapshot captured when the listener started.

## What This Changes

The issue requests that WhatsApp `dmPolicy: "allowlist"` block inbound DMs from numbers outside `allowFrom`, including after config reload.

## Best Possible Solution

Keep this open and fix WhatsApp hot reload so DM access-control policy changes apply to the active listener before the next inbound message, with regression coverage for changing `dmPolicy` and `allowFrom` after listener startup.

## Reproduction Assessment

Yes, by source inspection: start a WhatsApp listener, tighten `channels.whatsapp.dmPolicy` or `allowFrom` through config reload, then send a DM from a now-disallowed number; current main has no channel restart and the access-control path still uses the listener's startup config snapshot. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Yes, the requested behavior is the right contract. The narrow maintainable fix is to either restart the WhatsApp channel on relevant `channels.whatsapp` changes or make inbound access control consume the current runtime policy per message, then cover the hot-reload path in tests.
