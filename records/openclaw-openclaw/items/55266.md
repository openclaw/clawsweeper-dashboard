## Summary

Keep open. The reported bug still exists on current main, and this PR is useful source work, but it targets the older `shouldForceResponsesStore` helper path instead of the current centralized Responses payload policy that now controls embedded OpenAI Responses payload mutation.

## What This Changes

The PR exports and updates the embedded OpenAI stream-wrapper store helper to force `store: true` for proxied OpenAI Responses models with `compat.supportsStore: true`, decouples default server compaction from that proxy opt-in, and adds helper-level unit tests.

## Best Possible Solution

Implement the explicit proxy `compat.supportsStore: true` opt-in in `resolveOpenAIResponsesPayloadPolicy` and the matching provider capability surface, preserve `supportsStore: false` stripping and provider allowlist behavior, keep default server compaction limited to native direct OpenAI unless explicitly configured, and add focused tests plus docs/changelog coverage for the user-facing config.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: a model with `provider: "openai"`, `api: "openai-responses"`, proxy `baseUrl`, and `compat.supportsStore: true` still fails the native-endpoint gate, so the embedded payload wrapper does not force `store: true`.

## Solution Assessment

No. The requested behavior is the right narrow fix, but this PR is not the best current implementation because it edits an obsolete helper instead of the centralized policy/capability path now used by embedded and transport code.
