## Summary

Keep open: the reported bug is still present on current main, but this PR is stale because the active embedded Responses path now uses `resolveOpenAIResponsesPayloadPolicy`; the old helper targeted by the branch is no longer the right implementation point.

## What This Changes

The PR exports and tests `shouldForceResponsesStore`, forces `store: true` for proxied OpenAI Responses models with `compat.supportsStore: true`, and keeps default server compaction limited to direct OpenAI URLs.

## Best Possible Solution

Implement the explicit proxy `compat.supportsStore: true` opt-in in the central Responses payload policy and provider capability surfaces, keep server compaction opt-in separate, and update focused tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection is high confidence: with `provider: "openai"`, `api: "openai-responses"`, a proxy `baseUrl`, and `compat.supportsStore: true`, the native-endpoint gate keeps `explicitStore` undefined, so embedded payload mutation does not force `store: true`.

## Solution Assessment

No. The requested behavior is the right narrow fix, but this PR is not the best current implementation because it targets an obsolete helper instead of the central policy and capability path used on current main.
