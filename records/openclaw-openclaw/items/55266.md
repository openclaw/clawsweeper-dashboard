## Summary

Keep open. The bug is valid, but this PR is stale against current main: Responses store handling now lives in the central payload policy, which still blocks proxy `compat.supportsStore: true` from forcing `store: true`.

## What This Changes

The PR exports/tests the old embedded runner store helper, makes `compat.supportsStore: true` force `store: true` for proxied OpenAI Responses routes, and limits default server compaction to direct OpenAI URLs.

## Best Possible Solution

Implement explicit proxy `compat.supportsStore: true` support in `resolveOpenAIResponsesPayloadPolicy` and `resolveProviderRequestCapabilities`, keep compaction separately opt-in for proxies, then update focused tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection is high confidence: with `provider: "openai"`, `api: "openai-responses"`, a proxy `baseUrl`, and `compat.supportsStore: true`, the current central `allowsResponsesStore` gate remains false, so provider-policy mode does not force `store: true`.

## Solution Assessment

No. The requested behavior is the right narrow fix, but this PR changes an obsolete helper instead of the central payload policy and capability path used on current main.
