## Summary

Keep open. The PR is a useful repair for a real proxy Responses API gap, but current main has moved the active behavior into the centralized OpenAI Responses payload policy, while the PR still targets the older embedded-wrapper helper path; the same opt-in should be rebased or replaced in the current policy/capability layer before merge.

## What This Changes

The PR exports and updates the embedded OpenAI stream-wrapper store helper to force `store: true` for proxied OpenAI Responses models with `compat.supportsStore: true`, decouples default server compaction from that proxy opt-in, and adds helper-level unit tests.

## Best Possible Solution

Implement the proxy `compat.supportsStore: true` opt-in in the centralized OpenAI Responses payload policy and matching provider capability surface, keep the provider allowlist and `supportsStore: false` stripping semantics, leave default server compaction limited to direct OpenAI unless explicitly configured, and update focused tests plus docs/changelog for the user-facing config behavior.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: a model with `provider: "openai"`, `api: "openai-responses"`, a proxy `baseUrl`, and `compat.supportsStore: true` still fails the `usesKnownNativeOpenAIEndpoint` gate, so `explicitStore` remains undefined and the embedded wrapper does not force `store: true`.

## Solution Assessment

No. The PR’s desired behavior is correct, but the implementation is no longer the best path because it edits an obsolete helper instead of the current centralized payload policy and provider capability layer.
