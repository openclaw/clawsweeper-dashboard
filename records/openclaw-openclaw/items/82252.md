## Summary

Keep open. The report is fresh, concrete, and current main still routes Discord read/search through the gateway path while the Discord read handler assumes the returned value is an array before calling `.map()`. I did not find a merged follow-up for this newer crash after the prior read/search hang fix.

## What This Changes

The issue reports a Discord `message read` and likely `message search` gateway crash in OpenClaw 2026.5.12 with `TypeError: (intermediate value).map is not a function`.

## Best Possible Solution

Add a narrow regression around Discord read/search through the gateway message.action path, then fix the runtime or gateway serialization boundary so read/search return structured JSON or a clear structured error instead of throwing `.map is not a function`.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path from the report: run Discord `message read` against a real configured channel through the CLI or agent message tool. I did not execute that live Discord setup here, so the current review is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear whether a guard only at the `messages.map` call is the best fix, because the reporter says instrumenting that compiled site did not change the crash. The safer solution is to reproduce the gateway-owned Discord read/search path in a focused test and fix the boundary that actually returns the non-array value.
