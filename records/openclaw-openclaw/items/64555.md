## Summary

Keep open. Current main and the latest release still resolve WhatsApp default/account auth directories through ambient OAuth state, while the target-repo fix PR was closed unmerged and the remaining fork PR has not landed in openclaw/openclaw.

## What This Changes

The issue reports that WhatsApp Baileys credentials can be read or written under the default state directory instead of the active `--profile` state directory.

## Best Possible Solution

Land a focused WhatsApp plugin fix that lazily resolves default, account, and legacy Baileys auth directories from the active profile state, with regression coverage for `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes. The issue provides concrete profile-gateway commands and logs, and current main remains source-reproducible because WhatsApp default/account auth paths still resolve through ambient OAuth state instead of explicit active profile state.

## Solution Assessment

Yes. A WhatsApp plugin-owned path-resolution fix with regression tests is the narrowest maintainable solution; documenting the `OPENCLAW_OAUTH_DIR` workaround alone would leave the isolation bug in place.
