## Summary

Keep open. Current `main` and the latest release still contain the WhatsApp auth-dir resolution patterns implicated by the report, while the target-repo fix PR is closed unmerged and the external fork PR has not landed in `openclaw/openclaw`.

## What This Changes

WhatsApp Baileys credentials should stay inside the active `--profile` state directory instead of resolving to the default `~/.openclaw` credentials directory.

## Best Possible Solution

Land a focused WhatsApp plugin fix that binds default, account, and legacy Baileys auth dirs to the active profile state and adds regression coverage for `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes. The issue provides concrete profile-gateway commands and logs, and current `main` is source-reproducible because the default/account auth paths still resolve through ambient `resolveOAuthDir()` instead of an explicit active profile state dir.

## Solution Assessment

Yes. A WhatsApp plugin-owned fix that lazily resolves the default auth path and threads explicit env/state through account auth resolution is the narrowest maintainable solution; documenting the workaround alone would leave the isolation bug in place.
