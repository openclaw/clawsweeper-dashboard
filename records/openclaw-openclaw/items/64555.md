## Summary

Keep open. Current main and the latest release still eagerly resolve the default WhatsApp web auth directory and still route WhatsApp account auth discovery through ambient OAuth resolution, while the target-repo fix PR is closed unmerged and the remaining fork candidate has not landed in openclaw/openclaw.

## What This Changes

The issue reports WhatsApp Baileys credentials resolving to the default credentials root instead of the active named profile's state directory.

## Best Possible Solution

Land or recreate a focused WhatsApp fix that removes eager default auth-dir resolution, binds account/default/legacy auth-dir helpers to the active profile env or state dir, and adds a regression test proving default Baileys auth stays under `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes. The issue provides concrete profile-gateway steps and logs, and current source inspection still shows the default and account auth paths resolving from ambient OAuth state rather than an explicit active profile state dir.

## Solution Assessment

Yes. A plugin-owned path-resolution fix with a narrow regression test is the smallest maintainable correction; docs or the `OPENCLAW_OAUTH_DIR` workaround alone would leave the credential isolation bug in place.
