## Summary

Keep open. Current main still has the WhatsApp default auth dir and account/legacy auth-dir helpers resolving through ambient `resolveOAuthDir()`, while the CLI/docs promise per-profile state isolation and the target-repo fix PR is closed unmerged.

## What This Changes

Ensure WhatsApp Baileys credentials respect `--profile` / `OPENCLAW_STATE_DIR` isolation instead of falling back to the default state directory.

## Best Possible Solution

Land a focused WhatsApp plugin fix that lazily resolves the default auth path, threads active profile env/state through account and legacy auth-dir resolution, and adds regression coverage for `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes. The issue provides concrete profile-gateway commands and logs, and current main is source-reproducible because the default/account auth paths still resolve through ambient OAuth-dir state instead of an explicit active profile boundary.

## Solution Assessment

Yes. The narrowest maintainable solution is a WhatsApp plugin-owned path-resolution fix with regression tests; documenting `OPENCLAW_OAUTH_DIR` alone would leave the credential isolation bug in place.
