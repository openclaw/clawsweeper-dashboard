## Summary

Keep open. Current main still resolves the WhatsApp default and account auth directories through ambient OAuth state, and the open fixing PR at https://github.com/openclaw/openclaw/pull/82492 explicitly closes this issue but has not landed.

## What This Changes

The issue reports WhatsApp Baileys credentials resolving to the default state directory instead of the active `--profile` state directory.

## Best Possible Solution

Review and land a focused WhatsApp auth-dir fix through https://github.com/openclaw/openclaw/pull/82492 or an equivalent patch that resolves Baileys auth paths from the active profile state with regression coverage.

## Reproduction Assessment

Yes, at source level. The issue provides concrete profile-gateway steps and logs, and current main still has the default and account WhatsApp auth-dir paths resolving from ambient OAuth state instead of explicit active profile context.

## Solution Assessment

Yes. A WhatsApp-owned auth-dir resolution fix with regression coverage is the narrow maintainable solution; documenting `OPENCLAW_OAUTH_DIR` alone would leave the credential-isolation bug in place.
