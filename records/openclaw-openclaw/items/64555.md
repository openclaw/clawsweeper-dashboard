## Label Justifications

- `P1`: The report describes cross-profile WhatsApp credential and session leakage affecting multi-profile gateway deployments.
- `impact:security`: The issue is about credentials being read or written outside the active profile boundary.
- `impact:session-state`: The failure can make one profile reuse another profile's WhatsApp session state and fight over Baileys state files.

## Summary

Keep open. Current main and the latest shipped release still contain the WhatsApp auth-directory paths implicated by the report, and the open fixing PR that closes this issue has not merged.

## What This Changes

Make WhatsApp Baileys auth storage respect `--profile` and `OPENCLAW_STATE_DIR` so credentials stay inside the active profile.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82492 or an equivalent focused WhatsApp auth-dir fix that resolves Baileys credential paths from the active profile state with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. The issue provides concrete profile-gateway steps and logs, and current main still has the default and account WhatsApp auth-dir paths resolving from ambient OAuth state instead of explicit active profile context.

## Solution Assessment

Yes. A WhatsApp-owned auth-dir resolution fix with lazy/default-path regression coverage is the narrow maintainable solution; documenting `OPENCLAW_OAUTH_DIR` alone would leave the isolation bug in place.
