## Summary

Keep open. Current main still resolves WhatsApp default/account auth paths from ambient OAuth state, the target-repo fix PR is closed unmerged, and the reported profile-isolation boundary remains a real credential/session isolation bug.

## What This Changes

Ensure WhatsApp Baileys credentials for `--profile` gateways resolve under the active profile state directory instead of the default `~/.openclaw` credentials root.

## Best Possible Solution

Land a focused WhatsApp plugin fix that lazy-resolves the default web auth dir, binds default/account/legacy auth-dir helpers to the active profile env or state dir, and adds regression coverage for `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes. The issue provides concrete profile-gateway steps and logs, and current source still shows the default and account auth paths resolving from ambient OAuth state rather than an explicit active profile state dir.

## Solution Assessment

Yes. A plugin-owned path-resolution fix with focused regression coverage is the narrowest maintainable solution; documenting the `OPENCLAW_OAUTH_DIR` workaround alone would leave the isolation bug in place.
