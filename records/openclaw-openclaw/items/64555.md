## Summary

Keep open. Current main still resolves WhatsApp default and account auth directories through ambient `resolveOAuthDir()` and the target-repo fix was closed unmerged while the remaining fork PR has not landed in `openclaw/openclaw`.

## What This Changes

The issue asks OpenClaw to keep WhatsApp Baileys credentials inside the active `--profile` / `OPENCLAW_STATE_DIR` boundary instead of the main credentials directory.

## Best Possible Solution

Land a focused WhatsApp plugin fix that lazily resolves default, account, and legacy Baileys auth directories from the active profile state, with regression coverage for `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes. The issue provides concrete profile-gateway commands and logs, and current main remains source-reproducible because WhatsApp default/account auth paths still resolve through ambient OAuth state instead of explicit active profile state.

## Solution Assessment

Yes. A WhatsApp plugin-owned path-resolution fix with regression tests is the narrowest maintainable solution; documenting the workaround alone would leave the isolation bug in place.
