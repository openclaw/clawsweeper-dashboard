## Summary

Keep open. Current main and the latest release still contain the WhatsApp credential path patterns described in the report, and the target-repo fix candidate was closed unmerged while the remaining fork PR has not landed in openclaw/openclaw.

## What This Changes

WhatsApp Baileys credentials should stay under the active `--profile` state directory instead of resolving to the default `~/.openclaw` credentials root.

## Best Possible Solution

Land a focused WhatsApp plugin fix that resolves default, account, and legacy Baileys auth dirs from the active profile state and adds regression coverage for `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes. The issue provides concrete profile-gateway commands and logs, and current main remains source-reproducible because the default/account auth paths still resolve through ambient `resolveOAuthDir()` instead of an explicit active profile state dir.

## Solution Assessment

Yes. A WhatsApp plugin-owned fix that lazily resolves the default auth path and threads explicit env/state through account auth resolution is the narrowest maintainable solution; documenting the workaround alone would leave the isolation bug in place.
