## Label Justifications

- `P1`: The report describes cross-profile WhatsApp credential and session leakage affecting multi-profile gateway deployments.
- `impact:security`: The issue is about WhatsApp credentials being read or written outside the active profile boundary.

## Summary

Keep open. Current main and the latest release still have the WhatsApp default/account auth-dir paths resolving through ambient OAuth state, and the open fix PR at https://github.com/openclaw/openclaw/pull/82492 explicitly closes this issue but has not landed.

## What This Changes

The issue reports that WhatsApp Baileys credentials can resolve under the default state directory instead of the active `--profile` state directory.

## Best Possible Solution

Review and land a focused WhatsApp plugin fix, likely through https://github.com/openclaw/openclaw/pull/82492 after its blockers are addressed, that resolves default/account/legacy Baileys auth dirs from the active profile state and includes regression plus redacted real profile-gateway proof.

## Reproduction Assessment

Yes, at source level. The issue gives concrete profile-gateway steps and logs, and current main still eagerly exports the default WhatsApp auth dir while related account helpers resolve OAuth state without explicit active profile context.

## Solution Assessment

Yes. A WhatsApp-owned auth-dir resolution fix with regression coverage is the narrow maintainable solution; documenting `OPENCLAW_OAUTH_DIR` alone would leave the credential-isolation bug in place.
