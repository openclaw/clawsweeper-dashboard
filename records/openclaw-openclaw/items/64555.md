## Label Justifications

- `P1`: The report describes cross-profile WhatsApp credential/session leakage affecting real multi-profile gateway deployments.
- `impact:security`: Credentials can be read or written outside the active profile boundary.
- `impact:session-state`: The failure can make one profile reuse another profile's WhatsApp session state and fight over Baileys files.
- `impact:auth-provider`: The affected surface is credential and auth-directory resolution for the WhatsApp channel provider.

## Summary

Keep open. Current main still resolves WhatsApp default and account auth directories through ambient OAuth state, and the open fixing PR has a closing reference to this issue but has not landed.

## What This Changes

The issue reports that WhatsApp Baileys credentials ignore `--profile` isolation and are read/written under the default state directory instead of the active profile directory.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82492 or an equivalent WhatsApp-owned fix that resolves Baileys auth paths from the active profile state with regression coverage.

## Reproduction Assessment

Yes, at source level. The reported profile-gateway steps and logs match current main: the default WhatsApp auth export is eager, and account discovery resolves OAuth directories from ambient process env instead of explicit active profile state.

## Solution Assessment

Yes. A focused WhatsApp auth-dir resolution fix with regression coverage is the narrow maintainable solution; documenting `OPENCLAW_OAUTH_DIR` alone would leave the isolation bug in place.
