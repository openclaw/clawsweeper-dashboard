## Summary

Keep open: this is now a focused draft replacement for https://github.com/openclaw/openclaw/pull/78621 with supplied real behavior proof, but the latest diff still has a blocking security-diagnostic regression and a test environment isolation bug.

## What This Changes

The PR adds `OPENCLAW_IGNORE_BIND_WARNING` and `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` doctor env suppressors for managed Docker-out-of-Docker setups, plus focused doctor tests.

## Best Possible Solution

Suppress only non-critical bind diagnostics after trusted-proxy validity is established, preserve caller env state in tests, and document the operator-owned env knobs if maintainers accept them.

## Reproduction Assessment

Yes, source-level: current main unconditionally emits the relevant exposed-gateway bind and split-state warnings when those conditions are present, and the PR changes those branches. I did not run doctor or tests because this review must keep the checkout read-only.

## Solution Assessment

No, not yet. The suppressors may be acceptable for managed Docker deployments, but the bind warning path should only suppress after trusted-proxy `userHeader` and `trustedProxies` validation, and the state-integrity test must preserve caller env state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep invalid trusted-proxy configs visible:** `src/commands/doctor-security.ts:241`
  - body: With `OPENCLAW_IGNORE_BIND_WARNING=true`, this branch treats raw `gateway.auth.mode: "trusted-proxy"` as enough to skip the critical exposed-without-auth diagnostic, then suppresses the only remaining bind warning. Startup and the gateway audit still require `gateway.auth.trustedProxy.userHeader` and `gateway.trustedProxies`, so doctor should keep those critical findings visible or suppress only after the same validation passes.
  - confidence: 0.9
- **[P3] Restore the split-state env override after the test:** `src/commands/doctor-state-integrity.test.ts:828-833`
  - body: The new test sets `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` outside the suite env snapshot and always deletes it in `finally`. If a developer or CI lane already has that variable set, the default-warning assertion can run under suppression and the test will erase the caller's value afterward.
  - confidence: 0.78
