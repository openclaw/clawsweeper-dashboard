## Summary

Keep open: the branch still has a blocking security-diagnostic issue because the bind suppressor treats raw trusted-proxy mode as authenticated even when startup and audit contracts would reject or flag the configuration as critical. The PR is also draft and needs maintainer policy approval for persistent env-based warning suppression.

## What This Changes

The PR adds `OPENCLAW_IGNORE_BIND_WARNING` and `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` handling plus focused doctor tests for managed Docker-out-of-Docker deployments.

## Best Possible Solution

If maintainers accept these env knobs, suppress only non-critical diagnostics after validating trusted-proxy requirements and keep critical startup/audit findings visible.

## Reproduction Assessment

Yes, source-level: current main unconditionally emits the relevant exposed-gateway bind warning and split-state warning, and the PR diff clearly changes those branches. I did not run doctor or tests because this review is required to keep the checkout read-only.

## Solution Assessment

No, not yet. The suppression approach may be acceptable for managed Docker deployments, but the bind path should only suppress after trusted-proxy config is valid or should surface the same critical trusted-proxy diagnostics as startup and audit.
