## Summary

Keep open. The narrow PR is useful and has supplied after-fix terminal proof, but the latest head still lets `OPENCLAW_IGNORE_BIND_WARNING` hide invalid trusted-proxy exposure diagnostics and has one test env-isolation bug; the warning-suppression policy also needs maintainer approval before merge.

## What This Changes

The PR adds `OPENCLAW_IGNORE_BIND_WARNING` and `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` to suppress doctor gateway-bind and split-state warnings, plus focused tests.

## Best Possible Solution

Validate trusted-proxy auth with the same startup/audit contract before suppressing non-critical bind warnings, preserve test env state, and document the operator-owned env knobs if maintainers approve them.

## Reproduction Assessment

Yes, source-level. Current main unconditionally emits the relevant exposed-gateway and multiple-state-dir doctor warnings when those conditions are present; I did not run doctor commands because this review is read-only.

## Solution Assessment

No, not yet. The suppressor should only silence non-critical bind diagnostics after trusted-proxy validity is established, and the new test needs to restore the caller's env value.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep invalid trusted-proxy configs visible:** `src/commands/doctor-security.ts:241`
  - body: With `OPENCLAW_IGNORE_BIND_WARNING=true`, this branch treats raw `gateway.auth.mode: "trusted-proxy"` as enough to skip the critical exposed-without-auth diagnostic, then suppresses the only remaining bind warning. Startup and the gateway audit still require `gateway.auth.trustedProxy.userHeader` and `gateway.trustedProxies`, so doctor should keep those critical findings visible or suppress only after the same validation passes.
  - confidence: 0.88

  - confidence: 0.76
