## Summary

Keep this PR open but not merge-ready: the focused doctor warning suppressors are plausible, but the current diff can hide invalid trusted-proxy exposure diagnostics and one new test does not preserve caller env state.

## What This Changes

The PR adds `OPENCLAW_IGNORE_BIND_WARNING` and `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` to suppress Docker bind and split-state doctor warnings with focused tests.

## Best Possible Solution

Suppress only non-critical bind diagnostics after trusted-proxy validity is established, restore the split-state test env snapshot, and document the operator-owned env knobs if maintainers accept this policy.

## Reproduction Assessment

Yes, source-level: current main unconditionally emits the exposed-gateway bind and split-state warnings when those conditions are present, and the PR changes those branches. I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

No, not yet. The suppressors may be acceptable for managed Docker deployments, but the bind warning path should suppress only after trusted-proxy `userHeader` and `trustedProxies` validation, and the test must preserve caller env state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep invalid trusted-proxy configs visible:** `src/commands/doctor-security.ts:241`
  - body: With `OPENCLAW_IGNORE_BIND_WARNING=true`, this branch treats `gateway.auth.mode: "trusted-proxy"` as enough to skip the critical exposed-without-auth diagnostic, then suppresses the only remaining bind warning. Startup and the gateway audit still require `gateway.auth.trustedProxy.userHeader` and `gateway.trustedProxies`, so doctor should keep those critical findings visible or suppress only after the same validation passes.
  - confidence: 0.9

If a developer or CI lane already has that variable set, the default-warning assertion can run under suppression and the test will erase the caller's value afterward.
