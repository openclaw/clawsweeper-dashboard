## Summary

Keep open. This PR is a focused replacement for the earlier broad attempt and has sufficient terminal proof, but the current head still lets raw trusted-proxy mode hide bind diagnostics before the trusted-proxy config is validated, and the split-state test still leaks the new env override across caller state.

## What This Changes

Adds `OPENCLAW_IGNORE_BIND_WARNING` and `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` handling to Doctor security/state-integrity checks with focused tests.

## Best Possible Solution

If maintainers accept these env knobs, suppress only non-critical bind diagnostics after trusted-proxy validity is established, preserve test env state, document the operator-owned switches, and keep the broader sibling-home scan issue separate.

## Reproduction Assessment

Yes, source-level. Current main unconditionally emits the relevant exposed-gateway bind and split-state warnings when those conditions are present, and this PR changes those branches; I did not run Doctor because this review is read-only.

## Solution Assessment

No, not yet. The suppressors may be acceptable for managed Docker deployments, but the bind warning path should only suppress after trusted-proxy `userHeader` and `trustedProxies` validation, and the test must preserve caller env state.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Keep invalid trusted-proxy configs visible:** `src/commands/doctor-security.ts:241`
  - body: With `OPENCLAW_IGNORE_BIND_WARNING=true`, this condition treats `gateway.auth.mode: "trusted-proxy"` as authenticated even when `gateway.auth.trustedProxy.userHeader` or `gateway.trustedProxies` are missing. Startup and security audit fail those configs closed, so Doctor should keep a critical diagnostic visible until the same trusted-proxy validation passes.
  - confidence: 0.88

  - confidence: 0.76
