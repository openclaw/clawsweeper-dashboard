## Summary

Keep open: this is an active draft feature PR with supplied terminal proof, but the latest diff still has a blocking correctness/security issue in trusted-proxy suppression plus a test env cleanup issue.

## What This Changes

The PR adds `OPENCLAW_IGNORE_BIND_WARNING` and `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` support to suppress doctor gateway-bind and split-state warnings, with focused tests.

## Best Possible Solution

If maintainers accept these knobs, implement them as documented operator suppressors that only silence non-critical diagnostics after trusted-proxy config validity is established.

## Reproduction Assessment

Yes, source-level: current main emits the exposed-gateway and split-state doctor warnings on the affected branches, and the PR changes those paths. I did not run doctor or tests because this review must keep the checkout read-only.

## Solution Assessment

No, not yet. The suppressors may be acceptable for managed Docker deployments, but the bind path should validate trusted-proxy requirements before suppression and the new state test should restore the env snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep invalid trusted-proxy configs visible:** `src/commands/doctor-security.ts:241`
  - body: With `OPENCLAW_IGNORE_BIND_WARNING=true`, an exposed config using `gateway.auth.mode: "trusted-proxy"` skips the critical no-auth branch even when required trusted-proxy pieces such as `gateway.auth.trustedProxy.userHeader` or `gateway.trustedProxies` are missing. Startup and security audit treat those states as invalid or critical, so doctor should keep a critical diagnostic or suppress only after the same validation passes.
  - confidence: 0.88
- **[P3] Restore the split-state env override after the test:** `src/commands/doctor-state-integrity.test.ts:832-840`
  - body: The new split-state test sets and then deletes `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` without adding it to the suite env snapshot. If a developer or CI lane already has that variable set, the default-warning assertion can run under suppression and the test also erases the caller's env value afterward.
  - confidence: 0.77
