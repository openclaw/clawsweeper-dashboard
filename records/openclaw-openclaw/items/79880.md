## Summary

Keep open: this focused draft PR has sufficient real behavior proof, but it still has a security-sensitive correctness blocker because `OPENCLAW_IGNORE_BIND_WARNING` can suppress exposed-gateway diagnostics for raw `trusted-proxy` mode before the trusted-proxy configuration is validated.

## What This Changes

The PR adds env-variable suppressors for doctor bind and split-state warnings, plus focused doctor tests for managed Docker-out-of-Docker deployments.

## Best Possible Solution

If maintainers accept these env knobs, suppress only non-critical diagnostics after trusted-proxy config validity is established, and document the operator-owned suppressors as explicit managed-deployment overrides.

## Reproduction Assessment

Yes, source-level: current main unconditionally emits the relevant exposed-gateway bind warning and split-state warning, and the PR diff changes those branches. I did not run doctor or tests because this review must keep the checkout read-only.

## Solution Assessment

No, not yet. The suppressors may be acceptable for managed Docker deployments, but the bind warning path should only suppress after trusted-proxy `userHeader` and `trustedProxies` validation, or should surface the same critical findings as startup and audit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep invalid trusted-proxy configs visible:** `src/commands/doctor-security.ts:241`
  - body: With `OPENCLAW_IGNORE_BIND_WARNING=true`, an exposed config using `gateway.auth.mode: "trusted-proxy"` skips the critical no-auth branch even when required trusted-proxy pieces such as `gateway.auth.trustedProxy.userHeader` or `gateway.trustedProxies` are missing. Startup and security audit treat those states as invalid or critical, so doctor should keep a critical diagnostic or suppress only after the same validation passes.
  - confidence: 0.87
- **[P3] Restore the split-state env override after the test:** `src/commands/doctor-state-integrity.test.ts:828-833`
  - body: The new split-state test sets and deletes `OPENCLAW_IGNORE_SPLIT_STATE_WARNING` without adding it to the suite env snapshot. If a developer or CI lane already has that variable set, the default-warning assertion can run under suppression and the test will also erase the caller's env value afterward.
  - confidence: 0.75
