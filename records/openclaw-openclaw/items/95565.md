## Label Justifications

- `P2`: This is a normal-priority WebChat/session-history improvement with meaningful long-session UX impact but no unusable runtime or data deletion evidence.
- `impact:session-state`: The request concerns session transcript continuity and the distinction between archived history and the model's active context.
- `impact:message-loss`: Older transcript messages become inaccessible from the WebChat display after compaction, making retained messages appear lost to the user.

## Summary

Keep open: current main already shows a compaction divider and checkpoint link, but WebChat still fetches and replaces only a bounded recent active-tail history with no cursor/offset path or inline read-only older scrollback. Related open work covers reset/family archives and model-facing history pagination, but not this exact WebChat pre-compaction scrollback UX.

## What This Changes

The issue requests paginated WebChat scrollback for older pre-compaction transcript messages, with a visible active-context boundary and archival styling.

## Best Possible Solution

Design one bounded, additive Gateway/WebChat history model that can page older active-session/pre-compaction transcript records, reuses existing chat projection, and aligns with the includeFamily and sessions_history pagination decisions instead of adding a competing history path.

## Reproduction Assessment

Yes, at source level. Current main's chat.history path reads a bounded recent tail and the UI replaces the visible history from that response; I did not run a live WebChat compaction scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. The requested UX is useful, but the safest implementation should share one bounded history API with the existing checkpoint, includeFamily, and sessions_history pagination work rather than inventing a separate scrollback source.
