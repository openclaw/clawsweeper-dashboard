## Label Justifications

- `P2`: This is a normal-priority WebChat history and UX improvement for long sessions, with no evidence that transcript data is actually deleted.
- `impact:session-state`: The issue concerns durable session transcript entries becoming inaccessible from the active WebChat view after compaction.

## Summary

Keep open: current main and v2026.6.9 still load active WebChat through one bounded `chat.history` page with no older-page cursor, while a separate REST history endpoint and an open maintainer PR show the likely implementation direction but not a landed fix.

## What This Changes

Request WebChat scrollback for older pre-compaction transcript history, with a clear active-context boundary indicator and archival styling.

## Best Possible Solution

Approve one bounded active WebChat history paging contract, then land a repaired version of https://github.com/openclaw/openclaw/pull/87111 or an equivalent replacement that preserves payload caps, active-branch semantics, and the compaction boundary affordance.

## Reproduction Assessment

Yes, at source level: active WebChat loads one bounded `chat.history` page and top-of-thread scrolling only expands already-loaded messages, while the active RPC schema has no cursor or offset. I did not run a live WebChat compaction flow in this read-only review.

## Solution Assessment

Unclear. Cursor pagination plus a WebChat load-older control is likely better than raising limits, but maintainers still need to accept the public `chat.history`/session-state semantics and choose whether the open PR or a replacement is the right landing path.
