## Summary

Keep open: current main and v2026.6.9 already have a compaction divider and a direct REST cursor-history surface, but active WebChat still fetches one bounded `chat.history` page and has no scrollback path for older pre-compaction transcript entries. This is a real WebChat/Gateway API and UX decision, not an implemented or stale cleanup item.

## What This Changes

Request WebChat support for paginated, scrollable transcript history before the active model context window, with a clear compaction boundary indicator.

## Best Possible Solution

Define one bounded Gateway contract for active WebChat history paging, then implement scrollback that preserves UI performance and clearly distinguishes model-visible context from archival transcript entries.

## Reproduction Assessment

Yes, at source level: WebChat loads one fixed `chat.history` page and top-of-thread scrolling only expands the already-loaded render window, while the active chat RPC schema has no cursor or offset. I did not run a live browser compaction flow in this read-only review.

## Solution Assessment

Unclear. The existing HTTP cursor history endpoint and current `chat.history` projection are plausible building blocks, but maintainers should choose the durable WebChat/Gateway API and UX contract before implementation.
