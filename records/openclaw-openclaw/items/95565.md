## Summary

Keep open: current main and v2026.6.9 already have a compaction divider and a direct REST cursor-history endpoint, but active WebChat still loads one bounded chat history page and has no older-page scrollback path.

## What This Changes

The issue requests WebChat scrollback for older pre-compaction transcript messages, plus UI affordances that distinguish archival history from the active model context.

## Best Possible Solution

Define one bounded Gateway contract for active WebChat history paging, then implement scrollback that preserves UI performance and clearly distinguishes model-visible context from archival transcript entries.

## Reproduction Assessment

Yes, at source level: current WebChat requests one bounded chat.history/chat.startup page and top-scroll only expands the already-loaded render window. I did not run a live browser compaction flow in this read-only review.

## Solution Assessment

Unclear; the existing REST cursor-history endpoint and current chat.history projection are plausible building blocks, but maintainers still need to choose the durable Gateway/WebChat paging contract and UX semantics.
