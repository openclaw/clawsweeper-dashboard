## Summary

Keep open. Current main still reads the bare Chrome MCP temp screenshot path even though Chrome DevTools MCP saves the file with the requested format extension, and the related open PR #74685 is the active implementation candidate rather than a reason to close early. citeturn2view0turn3view0

## What This Changes

Fix Chrome DevTools MCP user-profile screenshots so the browser tool returns image data instead of ENOENT when the MCP server saves the screenshot with a format extension.

## Best Possible Solution

Review and land #74685, or an equivalent browser plugin fix that reads the MCP-created extension-suffixed screenshot file and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level: a fake Chrome MCP session that writes only `<filePath>.<format>` after a successful `take_screenshot` call would make current main read the unwritten bare path and hit ENOENT. I did not run a live browser session here.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/browser/src/browser/chrome-mcp.ts`, reading the extension-suffixed screenshot output or the MCP-reported file path rather than changing temp-directory policy.
