## Summary

Keep open. Current main still reads Chrome MCP screenshot output from the bare temporary path even though upstream Chrome DevTools MCP writes the extension-suffixed file; open PR #74685 already targets the same bug and should be reviewed or landed instead of starting another cleanup path.

## What This Changes

The issue reports that `browser` screenshots with Chrome MCP `profile="user"` fail with ENOENT because OpenClaw reads a temp screenshot path that Chrome MCP did not write.

## Best Possible Solution

Review and land #74685, or an equivalent browser plugin fix that reads the MCP-created extension-suffixed screenshot file and keeps focused regression coverage.

## Reproduction Assessment

Yes, at source level. A fake Chrome MCP session that writes only `<filePath>.<format>` after a successful `take_screenshot` call would make current main read the unwritten bare path and hit ENOENT.

## Solution Assessment

Yes. The narrow maintainable fix is in `extensions/browser/src/browser/chrome-mcp.ts`, matching PR #74685's approach of reading the extension-suffixed file and adding a regression test rather than changing core temp-directory policy.
