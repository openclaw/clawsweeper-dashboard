## Summary

Keep open. Current main still has the Chrome MCP screenshot filename mismatch, and the related open PR #74685 targets the exact bug but is not merged yet.

## What This Changes

The issue reports that browser screenshots for Chrome DevTools MCP user profiles fail with ENOENT because OpenClaw reads a temporary screenshot path that Chrome MCP does not write.

## Best Possible Solution

Review and land #74685, or an equivalent browser plugin fix that reads the Chrome MCP extension-suffixed screenshot file and keeps focused regression coverage.

## Reproduction Assessment

Yes, at source level: a fake Chrome MCP session that writes only `<filePath>.<format>` after `take_screenshot` would make current main read the unwritten bare `<filePath>` and hit ENOENT. I did not run a live Chrome MCP session in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in `extensions/browser/src/browser/chrome-mcp.ts`: read the actual extension-suffixed screenshot file and add a regression test, without changing core temp-dir policy or adding configuration.
