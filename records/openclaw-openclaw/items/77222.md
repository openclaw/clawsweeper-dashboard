## Summary

Keep open. Current main still appears to read Chrome MCP screenshots from the bare temporary file path while Chrome DevTools MCP writes the requested screenshot with a format extension, and the related open PR #74685 is the active implementation candidate rather than a reason to close this issue early.

## What This Changes

The issue reports that browser screenshots through the Chrome DevTools MCP user profile fail with ENOENT because OpenClaw reads a temporary screenshot path that Chrome MCP did not create.

## Best Possible Solution

Review and land #74685, or an equivalent browser plugin fix that reads the MCP-created extension-suffixed screenshot file and keeps focused regression coverage.

## Reproduction Assessment

Yes, at source level: a fake Chrome MCP session that writes only `<filePath>.<format>` after a successful `take_screenshot` call would make current main read the unwritten bare path and hit ENOENT. I did not perform a live Chrome MCP run in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable solution is in `extensions/browser/src/browser/chrome-mcp.ts`, matching #74685's approach of reading the extension-suffixed file and adding a regression test rather than changing core temp-directory policy or adding configuration.
