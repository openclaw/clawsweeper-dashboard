## Summary

Keep open. Current main still reads Chrome MCP screenshots from the bare temporary file path, while the upstream Chrome MCP screenshot implementation saves the file with the requested format extension; the related open PR #74685 targets that exact mismatch and should be reviewed or landed instead of closing this issue early.

## What This Changes

The issue reports that browser screenshots through the Chrome DevTools MCP `user` profile fail with ENOENT because OpenClaw reads a temporary screenshot path that Chrome MCP does not write.

## Best Possible Solution

Review and land #74685, or an equivalent browser plugin fix that reads the MCP-created extension-suffixed screenshot file and keeps focused regression coverage.

## Reproduction Assessment

Yes, at source level: a fake Chrome MCP session that writes only `<filePath>.<format>` after a successful `take_screenshot` call would make current main read the unwritten bare path and hit ENOENT. I did not perform a live Chrome MCP run here.

## Solution Assessment

Yes. The narrowest maintainable solution is in `extensions/browser/src/browser/chrome-mcp.ts`, matching #74685's approach of reading the extension-suffixed file and adding a regression test rather than changing core temp-dir policy or adding configuration.
