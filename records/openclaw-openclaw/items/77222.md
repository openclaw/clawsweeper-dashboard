## Summary

Keep open. Current main still routes existing-session/user screenshots through a Chrome MCP temp file path and then reads that file; the docs promise this screenshot surface works, and I found no current fix or regression coverage for the reported missing-file ENOENT.

## What This Changes

The issue asks for Chrome DevTools MCP user-profile browser screenshots to return image data instead of failing with ENOENT on the temporary screenshot path.

## Best Possible Solution

Fix the browser plugin’s Chrome MCP screenshot helper so it robustly returns bytes when the MCP server does not create the requested file, with focused regression coverage and no new user-facing config.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection does show a direct current-main path where a successful Chrome MCP `take_screenshot` call followed by a missing output file yields the reported ENOENT.

## Solution Assessment

Yes, the best fix belongs in `extensions/browser` around Chrome MCP screenshot file handling. A safer repair should preserve the existing docs contract and add fallback or validation around the MCP output instead of adding a new config knob or changing core temp-dir policy.
