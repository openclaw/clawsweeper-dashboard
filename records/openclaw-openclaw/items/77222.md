## Summary

Keep open. Current main still reads the bare Chrome MCP screenshot temp path, while upstream Chrome DevTools MCP writes the extension-suffixed file; open PR #74685 is the right implementation candidate to review or land.

## What This Changes

Fix Chrome MCP user-profile browser screenshots so they read the screenshot file actually written by Chrome DevTools MCP instead of failing with ENOENT.

## Best Possible Solution

Review and land #74685, or an equivalent browser plugin fix that reads the extension-suffixed Chrome MCP screenshot file and keeps focused regression coverage.

## Reproduction Assessment

Yes, at source level: a fake Chrome MCP session that writes only `<filePath>.<format>` after a successful `take_screenshot` call makes current main read the unwritten bare path. I did not perform a live Chrome MCP run.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/browser/src/browser/chrome-mcp.ts`, matching #74685's approach of reading the extension-suffixed file and adding regression coverage rather than changing core temp-dir policy.
