## Summary

Keep open. Current main still reads a bare Chrome MCP screenshot temp path, the documented existing-session screenshot surface remains supported, and the related fix PR is open but not merged.

## What This Changes

The issue reports that Chrome MCP `profile="user"` browser screenshots return ENOENT from a temp-file read instead of captured image data.

## Best Possible Solution

Land PR #74685 or an equivalent browser-plugin fix that reads the actual Chrome MCP screenshot output path, with regression coverage and no new config knob.

## Reproduction Assessment

No. I did not establish a live Chrome MCP reproduction, but current-main source inspection shows a direct path from successful `take_screenshot` to `fs.readFile(filePath)`, which matches the reported ENOENT when the file is absent or written under a different name.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/browser/src/browser/chrome-mcp.ts`, and the already-open PR #74685 is aligned with that boundary, so reviewing or landing it is preferable to opening a duplicate repair lane.
