## Summary

Keep open. Current main still has the Chrome MCP screenshot filename mismatch, and the related open PR #74685 is the right implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue requests that `browser` screenshots for the Chrome MCP-backed `user` profile return image data instead of failing with ENOENT on the temporary screenshot path.

## Best Possible Solution

Land #74685, or an equivalent browser plugin fix, so OpenClaw reads the Chrome MCP-created extension-suffixed screenshot file and keeps focused regression coverage.

## Reproduction Assessment

Yes, at source level. A Chrome MCP session that writes only `<filePath>.<format>` after `take_screenshot` will make current main read the unwritten bare `filePath` and throw ENOENT.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/browser/src/browser/chrome-mcp.ts`: read the same format-suffixed file that Chrome MCP writes, with a unit regression test.
