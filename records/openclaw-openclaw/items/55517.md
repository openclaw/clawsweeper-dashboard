## Summary

Keep open: current main and v2026.6.10 still register plugin-runtime subagent runs with completion delivery disabled, so the reported `subagent_delivery_target` and final external delivery path remains unresolved; the related merged PR fixed only lifecycle tracking for `subagent_ended`.

## What This Changes

The issue requests a supported way for plugin-runtime `api.runtime.subagent.run()` calls made from `before_dispatch` to participate in completion delivery and reach the originating external channel.

## Best Possible Solution

Add a maintainer-approved explicit plugin-runtime completion-delivery contract, likely an opt-in `subagent.run` mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests while preserving background-run defaults unless maintainers decide otherwise.

## Reproduction Assessment

Yes, at source level: current main marks plugin-runtime runs as tracked subagents but registers them with `expectsCompletionMessage: false`, and that skips the announce path that would run `subagent_delivery_target`. I did not execute a live iMessage/plugin harness in this read-only review.

## Solution Assessment

Unclear pending maintainer API/default direction. Flipping the default is risky; an explicit opt-in completion-delivery mode with docs and regression coverage is the safer likely solution.
