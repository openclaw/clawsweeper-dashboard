## Label Justifications

- `P2`: The issue describes a real plugin SDK delivery gap with limited blast radius that needs normal maintainer follow-up rather than emergency handling.
- `impact:message-loss`: The user-visible failure is that the final child result is not delivered back to the external requester channel.
- `impact:session-state`: Plugin-created runs are not registered as subagent lifecycle records with the metadata used by completion handling.

## Summary

Keep open. Current main and v2026.5.19 still route plugin-runtime subagent runs through the generic Gateway agent path without the registry and announce metadata that drives `subagent_delivery_target` and final external completion delivery, so this remains a real SDK contract question rather than cleanup.

## What This Changes

The issue asks for plugin `api.runtime.subagent.run()` calls from `before_dispatch` to participate in subagent completion target resolution and final external delivery, or for that unsupported behavior to be documented clearly.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add an opt-in bridge into the existing registry and announce machinery or document plugin-created runs as manual polling and delivery only.

## Reproduction Assessment

Yes, at source level. A plugin call to `api.runtime.subagent.run()` reaches Gateway `agent`, while only registry-tracked subagent runs enter the completion announce path that invokes `subagent_delivery_target`; I did not execute a live iMessage plugin run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the SDK contract. Reusing the existing registry and announce machinery is likely the right implementation substrate, but automatic default delivery versus explicit opt-in versus documented manual delivery is a product/API decision.
