## Summary

Keep open. Current main still routes plugin `api.runtime.subagent.run()` through the generic Gateway `agent` RPC without creating the subagent registry record that drives completion announce delivery, so the reported `subagent_delivery_target` and final external-delivery gap remains valid.

## What This Changes

The issue asks for plugin runtime subagent runs started from `before_dispatch` to have a supported deferred completion delivery path back to the originating external channel, or for the manual-delivery-only behavior to be documented.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add a deliberate registration path that reuses the existing subagent registry and announce machinery, or document plugin-created runs as manual polling/delivery only. The chosen path should update SDK docs and add regression coverage for a `before_dispatch` hook that returns immediately while the child completes later.

## Reproduction Assessment

Yes. The issue gives concrete reproduction steps and logs, and current-main source inspection gives a high-confidence static reproduction: plugin `runtime.subagent.run()` creates a Gateway `agent` run but never creates the registry record consumed by completion announce delivery.

## Solution Assessment

Unclear until maintainer API direction is chosen. Reusing the existing registry/announce flow is likely the right implementation substrate, but the maintainable decision is whether deferred delivery is default, explicit opt-in, or documented as unsupported for plugin-created runs.
