## Summary

Keep open. Current main and the latest release still keep plugin runtime `subagent.run()` runs on the non-completion background path, so the requested `subagent_delivery_target` hook and final external delivery path remains unresolved even though the related `subagent_ended` lifecycle gap was fixed.

## What This Changes

The issue asks for plugin-started subagent runs from `before_dispatch` to participate in completion target resolution and final external delivery back to the originating external channel.

## Best Possible Solution

Add an explicit plugin runtime completion-delivery contract, likely an opt-in `subagent.run` option or documented binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: plugin runtime `subagent.run()` marks the Gateway request as `plugin_subagent`, the Gateway registers it with `expectsCompletionMessage: false`, and lifecycle cleanup skips the announce path that would consult `subagent_delivery_target`.

## Solution Assessment

No, flipping the existing default for all plugin subagent runs is not clearly the best fix because current tests encode them as background tasks with completion delivery not required. The safer solution is an explicit opt-in completion-delivery API or a maintainer decision to document plugin runtime runs as manually delivered.
