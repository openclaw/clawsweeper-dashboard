## Summary

Keep open. Current main still sends plugin `api.runtime.subagent.run()` through the Gateway `agent` method without registering the child run in the subagent registry, so the completion announce path that invokes `subagent_delivery_target` is still skipped. The existing discussion correctly frames this as a valid plugin-runtime subagent delivery contract question rather than a simple `before_dispatch` hook failure.

## What This Changes

The issue asks for plugin runtime subagent runs started from `before_dispatch` to have a supported deferred completion delivery path back to the originating external channel, or for the manual-delivery-only behavior to be documented.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add a deliberate registration path that reuses the existing subagent registry and announce machinery, or document plugin-created runs as manual polling/delivery only. The chosen path should update SDK docs and add a regression test for a `before_dispatch` hook returning immediately while the child completes later.

## Reproduction Assessment

Yes. The report gives concrete reproduction steps, and the current source path gives a high-confidence static reproduction: `runtime.subagent.run()` creates a Gateway `agent` run but never creates the registry record that completion announce delivery consumes.

## Solution Assessment

Unclear until maintainer API direction is chosen. Reusing the existing registry/announce flow is likely the right implementation substrate, but the safe product decision is whether that behavior is default or explicit opt-in for plugin-created runs.
