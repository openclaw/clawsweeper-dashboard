## Summary

Keep open. Current main and v2026.6.8 still register plugin-runtime `subagent.run()` as tracked background subagent runs with completion delivery disabled, so the requested `subagent_delivery_target` and final external-channel delivery path remains unresolved; the merged related fix only covered lifecycle tracking for `subagent_ended`.

## What This Changes

The issue requests completion-target hook execution and final external delivery for plugin runtime subagent runs started from `before_dispatch`.

## Best Possible Solution

Add a maintainer-approved explicit plugin-runtime completion-delivery contract, likely an opt-in `subagent.run` option or binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes. Source inspection shows plugin runtime `subagent.run()` marks Gateway requests as `plugin_subagent`, the Gateway registers those runs with `expectsCompletionMessage: false`, and lifecycle cleanup skips the announce path that would consult `subagent_delivery_target`.

## Solution Assessment

Unclear pending maintainer product/API direction. Flipping the default for all plugin subagent runs is not the narrowest maintainable fix; an explicit opt-in completion-delivery contract or clear documentation of manual delivery is safer.
