## Summary

Keep open. Current main and the latest release still mark plugin SDK subagent runs as completion-delivery-not-required, so the related `subagent_ended` fix does not solve the requested `subagent_delivery_target` hook invocation or final external delivery path.

## What This Changes

The issue requests supported final external delivery for plugin runtime subagent runs started inside `before_dispatch`, including running `subagent_delivery_target` when the child completes.

## Best Possible Solution

Add an explicit, documented plugin runtime completion-delivery contract, likely an opt-in `subagent.run` option or binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: plugin runtime `subagent.run()` marks the Gateway request as `plugin_subagent`, the Gateway registers it with `expectsCompletionMessage: false`, and lifecycle cleanup skips the announce path that would consult `subagent_delivery_target`.

## Solution Assessment

No, closing is not the best outcome because the central requested behavior remains missing. The safer solution is an explicit opt-in completion-delivery API or a maintainer decision to document plugin runtime runs as manually delivered.
