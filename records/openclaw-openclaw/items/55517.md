## Summary

Keep open. Current `main` still routes plugin `api.runtime.subagent.run()` through the generic Gateway `agent` path without the registry and announce metadata that drives `subagent_delivery_target` and final external delivery; the narrower lifecycle issue at https://github.com/openclaw/openclaw/issues/59164 and the broader delivery-owner design issue at https://github.com/openclaw/openclaw/issues/80450 do not supersede this SDK contract question.

## What This Changes

The issue requests that plugin-created subagent runs started from `before_dispatch` participate in `subagent_delivery_target` and deferred final delivery back to the external requester.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add an opt-in bridge into the existing registry and announce machinery or document plugin-created runs as manual polling and delivery only.

## Reproduction Assessment

Yes, at source level. A plugin call to `api.runtime.subagent.run()` reaches Gateway `agent`, while only registry-tracked subagent runs enter the completion announce path that invokes `subagent_delivery_target`; I did not execute a live iMessage plugin run.

## Solution Assessment

Unclear until maintainers choose the SDK contract. Reusing the existing registry and announce machinery is likely the right implementation substrate, but automatic default delivery versus explicit opt-in versus documented manual delivery is a product/API decision.
