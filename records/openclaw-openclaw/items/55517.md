## Summary

Keep open. Current main and the latest release still track plugin-runtime `subagent.run()` calls as background subagent runs with completion delivery disabled, so the requested `subagent_delivery_target` and final external-channel delivery path remains unresolved; the merged related fix at https://github.com/openclaw/openclaw/pull/87698 only covered lifecycle tracking for `subagent_ended`.

## What This Changes

Issue requests that plugin-runtime subagent runs started from a `before_dispatch` hook can later resolve `subagent_delivery_target` and deliver final completion output back to the originating external channel.

## Best Possible Solution

Add an explicit plugin-runtime completion-delivery contract, likely an opt-in `subagent.run` option or binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes. Source inspection shows plugin runtime `subagent.run()` marks Gateway requests as `plugin_subagent`, the Gateway registers those runs with `expectsCompletionMessage: false`, and lifecycle cleanup skips the announce path that would consult `subagent_delivery_target`.

## Solution Assessment

Unclear pending maintainer product/API direction. Flipping the default for all plugin subagent runs is not the narrowest maintainable fix; an explicit opt-in completion-delivery contract or clear documentation of manual delivery is safer.
