## Summary

Keep open. Current main and v2026.6.8 still register plugin-runtime `api.runtime.subagent.run()` as a tracked background subagent run with completion delivery disabled, so the requested `subagent_delivery_target` and final external delivery behavior remains unresolved; the merged related PR only fixed `subagent_ended` lifecycle tracking.

## What This Changes

The issue requests a supported plugin-runtime path for `api.runtime.subagent.run()` launched from `before_dispatch` to participate in subagent completion delivery and `subagent_delivery_target` so final results can reach the originating external channel.

## Best Possible Solution

Add a maintainer-approved explicit plugin-runtime completion-delivery contract, likely an opt-in `subagent.run` option or binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests while keeping default background runs non-delivering.

## Reproduction Assessment

Yes, at source level. A plugin call to `api.runtime.subagent.run()` is marked as `plugin_subagent`, registered with `expectsCompletionMessage: false`, and then lifecycle cleanup skips the announce path that would consult `subagent_delivery_target`.

## Solution Assessment

Unclear pending maintainer product/API direction. Flipping the default for all plugin subagent runs is not the narrowest maintainable fix; an explicit opt-in completion-delivery contract or clear documentation of manual delivery is safer.
