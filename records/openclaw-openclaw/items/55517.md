## Summary

Keep open. Current main and the latest release still register plugin-runtime `subagent.run()` runs with completion delivery disabled, so the reported `subagent_delivery_target` and final external-channel delivery path remains unresolved; the related merged work only fixed lifecycle tracking for `subagent_ended`.

## What This Changes

The issue asks for plugin-runtime subagent runs started from `before_dispatch` to participate in completion target resolution and deliver the final child result back to the originating external channel.

## Best Possible Solution

Add a maintainer-approved explicit plugin-runtime completion-delivery contract, likely an opt-in `subagent.run` option or binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests while preserving current background-run defaults unless maintainers decide otherwise.

## Reproduction Assessment

Yes, at source level. Plugin runtime `subagent.run()` marks Gateway requests as `plugin_subagent`, current registration sets `expectsCompletionMessage: false`, and lifecycle cleanup skips the announce path that would call `subagent_delivery_target`.

## Solution Assessment

Unclear until maintainers choose the public Plugin SDK contract. Flipping the default is risky; an explicit opt-in completion-delivery mode or clear documentation of manual delivery is the safer direction.
