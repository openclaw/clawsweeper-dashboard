## Summary

Keep open. Current main still treats plugin runtime `subagent.run()` runs as tracked background subagents with completion delivery disabled, so `subagent_delivery_target` and final external delivery remain unresolved and need a maintainer API/default decision.

## What This Changes

The issue requests a supported core path for plugin `before_dispatch` hooks that start `api.runtime.subagent.run()` child runs to deliver the later child result back to the original external channel.

## Best Possible Solution

Add an explicit plugin runtime completion-delivery contract, likely an opt-in `subagent.run` option or documented binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes. Source inspection shows plugin runtime `subagent.run()` marks the Gateway request as `plugin_subagent`, the Gateway registers it with `expectsCompletionMessage: false`, and lifecycle cleanup skips the announce path that would consult `subagent_delivery_target`.

## Solution Assessment

No, flipping the existing default for all plugin subagent runs is not clearly the best fix because current tests encode them as background tasks. The safer path is an explicit opt-in completion-delivery API or a maintainer decision to document plugin runtime runs as manually delivered.
