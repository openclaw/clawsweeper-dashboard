## Summary

Keep open. Current `main` still sends plugin `api.runtime.subagent.run()` through the generic Gateway `agent` path rather than the registered subagent completion lifecycle, and the remaining question is an SDK/API delivery contract decision rather than a stale cleanup candidate.

## What This Changes

The issue asks for plugin runtime subagent runs launched from `before_dispatch` to participate in `subagent_delivery_target` and final external delivery after the hook returns an immediate progress reply.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either bridge opted-in plugin-created runs into the existing subagent registry and announce machinery or document them as manual polling/delivery only, with SDK docs and regression coverage updated accordingly.

## Reproduction Assessment

Yes. The issue provides concrete steps and logs, and current-main source inspection confirms that plugin `runtime.subagent.run()` creates a generic Gateway `agent` run without the registry record consumed by completion announce delivery.

## Solution Assessment

Unclear until maintainers choose the API contract. Reusing the existing registry/announce flow is likely the right implementation substrate, but the decision is whether deferred delivery is default, explicit opt-in, or documented as unsupported for plugin-created runs.
