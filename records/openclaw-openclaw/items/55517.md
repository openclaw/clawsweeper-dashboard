## Summary

Keep open. Current main and the latest release still track plugin runtime subagent runs as non-completion background runs, so the requested final external delivery and `subagent_delivery_target` path remains unresolved.

## What This Changes

The issue requests that plugin runtime `api.runtime.subagent.run()` calls started from `before_dispatch` have a supported way to later deliver child completion back to the original external requester.

## Best Possible Solution

Add an explicit plugin runtime completion-delivery contract, likely an opt-in `subagent.run` option or documented binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction: plugin runtime `subagent.run()` marks the Gateway request as `plugin_subagent`, the Gateway registers it with `expectsCompletionMessage: false`, and lifecycle cleanup skips the announce path that would consult `subagent_delivery_target`. I did not rerun the original external iMessage scenario.

## Solution Assessment

No, flipping the existing default for all plugin subagent runs is not clearly the best fix because current tests encode them as background tasks with completion delivery not required. The safer solution is an explicit opt-in completion-delivery API or a maintainer decision to document plugin runtime runs as manually delivered.
