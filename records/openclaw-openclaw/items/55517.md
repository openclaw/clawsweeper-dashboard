## Label Justifications

- `P2`: The report describes a real plugin SDK delivery gap with limited blast radius that needs normal maintainer follow-up rather than emergency handling.
- `impact:message-loss`: The user-visible failure is that the final child result is not delivered back to the external requester channel.
- `impact:session-state`: The source gap is that plugin-created runs are not registered as subagent lifecycle records with the metadata used by completion handling.

## Summary

Keep open. Current main and the latest stable tag still send plugin-runtime subagent runs through the generic Gateway agent path, while `subagent_delivery_target` and final external completion delivery are driven by registry-tracked subagent announce flows; this needs an explicit plugin SDK delivery contract rather than cleanup.

## What This Changes

The issue requests that plugin-created subagent runs started from `before_dispatch` participate in `subagent_delivery_target` and final external reply delivery.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add an opt-in bridge into the existing registry and announce machinery or document plugin-created runs as manual polling and delivery only.

## Reproduction Assessment

Yes. The reported flow is source-reproducible on current main: plugin `api.runtime.subagent.run()` reaches Gateway `agent`, while only registry-tracked subagent runs enter the completion announce path that invokes `subagent_delivery_target`.

## Solution Assessment

Unclear until maintainers choose the SDK contract. Reusing the existing registry and announce machinery is likely the right implementation substrate, but automatic default delivery versus explicit opt-in versus documented manual delivery is a product/API decision.
