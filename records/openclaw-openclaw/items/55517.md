## Label Justifications

- `P2`: This is a real plugin SDK/runtime integration gap with source-level proof and limited blast radius, but it is not a core-runtime outage.
- `impact:message-loss`: The reported external-channel final result is not delivered after the plugin-created child run completes.
- `impact:session-state`: Plugin-created child runs lack the registry state that completion hooks and announce delivery depend on.

## Summary

Keep open. Current main and the latest tagged release still route plugin `api.runtime.subagent.run()` through the generic Gateway `agent` path without the registry metadata that drives `subagent_delivery_target`, `subagent_ended`, and final completion announce delivery; the related open issues cover narrower lifecycle hooks or broader delivery ownership, but neither supersedes this plugin-runtime delivery contract question.

## What This Changes

The issue asks for plugin `api.runtime.subagent.run()` calls made from `before_dispatch` to participate in subagent completion target resolution and final external-channel delivery, or for that unsupported behavior to be documented.

## Best Possible Solution

Define the plugin-runtime completion contract explicitly, then bridge plugin-created subagent runs into the existing registry/announce machinery with compatibility-safe defaults and focused docs/tests for lifecycle hooks and final delivery.

## Reproduction Assessment

Yes, at source level. A plugin `api.runtime.subagent.run()` call reaches Gateway `agent`, while `subagent_delivery_target` and final completion announce flow require a registered `SubagentRunRecord`; I did not run a live external-channel harness in this read-only review.

## Solution Assessment

Unclear until maintainers decide the contract. Reusing the existing registry and announce machinery is the likely maintainable path, but automatic final external delivery should be explicit or compatibility-safe rather than silently changing all plugin-created background runs.
