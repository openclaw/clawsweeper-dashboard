## Summary

Keep open. Current main and the latest release still register plugin runtime subagent runs with completion delivery not required and cleanup skips the announce path, so the final external delivery and `subagent_delivery_target` request remains unresolved; the related merged work only fixed registry tracking for `subagent_ended`.

## What This Changes

Request plugin runtime `api.runtime.subagent.run()` calls started from `before_dispatch` to participate in supported completion delivery back to the original external requester channel.

## Best Possible Solution

Add an explicit plugin runtime completion-delivery contract, likely an opt-in `subagent.run` option or documented binding mode, then wire it through registry registration, announce delivery, `subagent_delivery_target`, docs, and focused upgrade-safe tests.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: plugin runtime `subagent.run()` marks the Gateway request as `plugin_subagent`, the Gateway registers it with `expectsCompletionMessage: false`, and lifecycle cleanup skips `runSubagentAnnounceFlow`. I did not rerun the original external iMessage setup.

## Solution Assessment

No, flipping the existing default for all plugin subagent runs is not clearly the best fix because current tests treat them as background tasks with delivery not required. The safer path is an explicit opt-in completion-delivery API or a maintainer decision to document that plugin runtime runs do not own final external delivery.
