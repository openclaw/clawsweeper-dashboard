## Summary

Keep open. Current main still routes plugin `api.runtime.subagent.run()` through the generic Gateway `agent` task path, while `subagent_delivery_target` and final completion delivery are driven by registry-tracked subagent runs; the remaining question is an explicit plugin SDK/API delivery contract, not stale cleanup.

## What This Changes

Request deferred completion delivery for plugin-created subagent runs launched from `before_dispatch`, including `subagent_delivery_target` resolution back to the external requester.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add an opt-in bridge into the existing registry/announce machinery or document plugin-created runs as manual polling/delivery only.

## Reproduction Assessment

Yes. The issue provides concrete steps/logs, and current-main source inspection confirms `api.runtime.subagent.run()` reaches Gateway `agent` without the registry metadata that drives `subagent_delivery_target` and completion delivery.

## Solution Assessment

Unclear until maintainers choose the API contract. Reusing the registry and announce machinery is likely the right implementation substrate, but default delivery versus explicit opt-in versus documented manual delivery is a product/SDK decision.
