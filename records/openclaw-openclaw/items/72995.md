## Summary

Keep open. Current main still has the credible failure shape: managed `web_search` drops the tool AbortSignal, the provider contract/runtime execute providers with args only, Gemini search uses only timeout-derived guarded fetch, and `AbortError` bypasses normal tool failure logging.

## What This Changes

The issue reports intermittent instant Gemini-backed `web_search` aborts that ignore timeout configuration and do not appear in `gateway.err.log`.

## Best Possible Solution

Add a backward-compatible web-search execution context carrying the caller AbortSignal through the managed tool, runtime, SDK/provider types, and Gemini provider, then classify/log aborts so intentional cancellations stay quiet while unexpected provider aborts are visible.

## Reproduction Assessment

Unclear for the live intermittent failure because this read-only review had no Gemini/Google key to run the live test. A source-level reproduction path is clear: a provider can throw `AbortError`, and current main rethrows it before normal `[tools]` failure logging while no signal context reaches the provider.

## Solution Assessment

Not applicable as a proposed patch because this is an issue. The narrow maintainable direction is an additive execution context plus abort classification, not a Gemini-only catch, because the runtime/provider contract currently lacks cancellation-source context.
