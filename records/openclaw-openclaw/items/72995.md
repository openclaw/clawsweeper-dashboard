## Summary

Keep open. Current main still matches the reported failure class: the managed `web_search` path and plugin-facing web-search provider contract do not pass the agent tool `AbortSignal`, the Gemini provider only receives a timeout-derived guarded-fetch signal, and `AbortError` still bypasses the normal tool failure logging path.

## What This Changes

The issue asks OpenClaw to stop intermittently returning an instant silent `This operation was aborted` from Gemini-backed `web_search`, and to make the failure observable in normal gateway logs.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries `AbortSignal` and abort diagnostics through `AnyAgentTool`, `runWebSearch`, provider SDK/runtime types, and the Gemini provider; then log and classify unexpected provider aborts while preserving true user/session cancellation as cancellation.

## Reproduction Assessment

Unclear for the intermittent live Gemini failure because this read-only review did not run live Gemini credentials or tests. The source-level reproduction path is clear: a selected web-search provider can throw an `AbortError`, and the current adapter rethrows it before normal tool failure logging while the managed web-search path lacks caller signal context.

## Solution Assessment

Not applicable as a proposed patch; there is no PR to evaluate. The current implementation is not the best end state because a timeout-only Gemini path cannot distinguish true cancellation from provider/internal aborts, so the safer direction is a generic web-search execution-context seam plus logging tests rather than a Gemini-only timeout tweak.
