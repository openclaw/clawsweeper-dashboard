## Summary

Keep open. Current main still matches the reported source-level failure shape: managed `web_search` does not propagate the tool execution `AbortSignal` into web-search providers, Gemini search only receives a timeout-derived fetch guard, and provider-shaped `AbortError`s still bypass normal `[tools]` failure logging.

## What This Changes

The issue reports intermittent near-instant `web_search` aborts with the Gemini provider and no corresponding `gateway.err.log` entry.

## Best Possible Solution

Add a backward-compatible web-search execution context that carries caller `AbortSignal` and cancellation diagnostics through the managed tool, runtime, SDK/provider types, and Gemini fetch path, then log unexpected provider aborts while preserving intentional run cancellation behavior.

## Reproduction Assessment

Unclear for the live intermittent Gemini failure because this read-only review had no Gemini/Google API key and did not run tests. A source-level reproduction path is clear: a selected provider can throw `AbortError`, and current main rethrows it before normal `[tools]` failure logging while lacking caller-signal context.

## Solution Assessment

Not applicable as a proposed patch because this is an issue. The narrow maintainable direction is an additive execution context plus abort classification rather than a Gemini-only catch, because the runtime/provider contract currently lacks cancellation-source context.
