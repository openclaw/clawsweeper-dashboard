## Summary

Keep open. Current main still has a credible source-level match for the report: the managed `web_search` tool has an agent abort signal available at the tool boundary, but the web-search provider path still executes providers with args only, Gemini still uses timeout-only guarded fetch, and provider-shaped `AbortError`s still bypass normal tool failure logging.

## What This Changes

The issue reports intermittent near-instant Gemini `web_search` aborts that return `This operation was aborted` without a corresponding `gateway.err.log` entry.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries caller `AbortSignal` and abort diagnostics through the managed tool, `runWebSearch`, provider SDK/runtime types, and Gemini provider, then classify and log unexpected provider aborts while preserving true user/session cancellation behavior.

## Reproduction Assessment

Unclear. The live intermittent failure needs a Gemini key and timing-sensitive provider run, but the source-level path is clear: a selected provider can throw an `AbortError`, current main rethrows it before normal tool failure logging, and the managed web-search provider path lacks caller signal context.

## Solution Assessment

Not applicable as a proposed patch because this is an issue, not a PR. The current implementation is not the best end state because timeout-only provider execution cannot reliably distinguish true cancellation from internal or provider aborts.
