## Summary

Keep open. Current main still has a credible source-level match for the report: the managed `web_search` tool receives an agent tool signal but does not forward it through the web-search provider path, Gemini search uses only timeout-derived guarded fetch, and `AbortError` still bypasses normal tool failure logging.

## What This Changes

The issue reports intermittent, near-instant Gemini `web_search` aborts that return `This operation was aborted` without a corresponding `gateway.err.log` entry.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries caller `AbortSignal` and abort diagnostics through the managed tool, `runWebSearch`, provider SDK/runtime types, and Gemini provider, then classify and log unexpected provider aborts while preserving true user/session cancellation behavior.

## Reproduction Assessment

Unclear for the live intermittent failure because no Gemini key was available in the environment or `~/.profile`, and this read-only review did not run tests. The source-level path is clear: a selected provider can throw an `AbortError`, and current main rethrows it before normal tool failure logging while the managed web-search path lacks caller signal context.

## Solution Assessment

Not applicable as a proposed patch because this is an issue, not a PR. The current implementation is not the best end state because timeout-only provider execution cannot reliably distinguish true cancellation from internal/provider aborts.
