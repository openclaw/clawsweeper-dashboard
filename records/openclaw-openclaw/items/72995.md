## Summary

Keep open. Current main still has the same source-level failure shape: managed `web_search` does not forward the tool `AbortSignal` into provider execution, Gemini search uses only timeout-derived guarded fetch, and `AbortError` is rethrown before normal tool failure logging.

## What This Changes

The issue reports intermittent sub-second `web_search` aborts when using the Gemini Google Search grounding provider, with no corresponding gateway error log entry.

## Best Possible Solution

Add a backward-compatible web-search execution context carrying caller `AbortSignal` and cancellation diagnostics through the managed tool, runtime, SDK/provider types, and Gemini provider, then log unexpected provider aborts without treating intentional cancellation as a tool failure.

## Reproduction Assessment

Unclear for the live intermittent Gemini failure because no Gemini/Google API key was available and this read-only review did not run tests. A source-level reproduction path is clear: a selected provider can throw `AbortError`, and current main rethrows it before normal `[tools]` failure logging while lacking caller signal context.

## Solution Assessment

Not applicable as a proposed patch because this is an issue. The narrow maintainable direction is an additive execution context plus abort classification, not a Gemini-only catch, because the runtime/provider contract currently lacks cancellation-source context.
