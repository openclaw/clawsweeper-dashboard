## Summary

Keep open. Current main still has a credible source-level path matching the report: managed `web_search` drops the run/tool `AbortSignal` before provider execution, the Gemini provider only receives timeout-derived cancellation, and provider-raised `AbortError`s still bypass the normal tool failure logging branch.

## What This Changes

The issue reports intermittent near-instant Gemini-backed `web_search` aborts that return “This operation was aborted” without a matching `gateway.err.log` entry.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries `AbortSignal` and abort diagnostics through `AnyAgentTool`, `createWebSearchTool`, `runWebSearch`, the provider SDK/runtime types, and the Gemini provider; log and classify unexpected provider aborts while preserving true user/session cancellation as cancellation.

## Reproduction Assessment

Unclear for the intermittent live Gemini failure because this read-only review did not run live credentials. The source-level reproduction path is clear: a selected web-search provider can throw an `AbortError`, and the current adapter rethrows it before normal tool failure logging while the managed web-search path lacks caller signal context.

## Solution Assessment

Not applicable as a proposed patch; there is no PR to evaluate. The current implementation is not the best end state because timeout-only provider execution cannot distinguish true cancellation from provider/internal aborts, so the safer direction is a generic web-search execution-context seam plus logging tests.
