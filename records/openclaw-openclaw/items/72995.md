## Summary

Keep open. Current main still matches the report at source level: managed `web_search` receives a tool `AbortSignal` but does not forward it, the provider-facing web-search contract executes with args only, Gemini runs through a timeout-only guarded fetch path, and `AbortError` still bypasses normal tool failure logging.

## What This Changes

Investigate and fix intermittent Gemini `web_search` aborts that return `This operation was aborted` without corresponding Gateway error-log diagnostics.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries caller `AbortSignal` and abort diagnostics through `createWebSearchTool`, `runWebSearch`, provider SDK/runtime types, and the Gemini provider, then log unexpected provider aborts while preserving true user/session cancellation behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the live intermittent Gemini failure: the reporter provided concrete steps and current source shows the abort/logging path, but this review had no Gemini credential and did not run live search.

## Solution Assessment

Is this the best way to solve the issue? Not applicable as a proposed patch because this is an issue. The best repair is not a Gemini-only catch block; it should preserve public provider compatibility while adding cancellation context and diagnostics through the shared web-search path.
