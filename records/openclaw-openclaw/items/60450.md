## Summary

Keep open. Current main has finite fallback candidate chains, auth-profile cooldowns, and shorter `auth_permanent` disables, but it still does not implement the central billing-safety request: a spend/rate guard or user-visible lockout that stops autonomous cross-provider model calls after repeated auth failures. The broader #13615 rate/budget issue is related and still open, but this report is a concrete recent auth-failover incident with enough detail to track separately until maintainer/product follow-up decides the lockout and alert policy.

## What This Changes

Prevent OpenClaw from autonomously cycling through configured LLM providers after repeated auth failures by adding hard failover/spend guards, stronger billing lockouts, and user-visible alerts.

## Best Possible Solution

Treat this as an active billing-safety regression and add a tested auth-failover guard that counts repeated auth/billing failures across the configured fallback chain, pauses further provider calls within a session or time window, surfaces a clear user-visible lockout/alert, and documents how that targeted protection relates to the broader #13615 budget-control work.

## Reproduction Assessment

Unclear: the report gives enough concrete log patterns and configuration context to build a high-confidence simulated repro, but a safe live reproduction would require paid provider accounts and intentionally failing auth across providers. A focused unit/integration test can force repeated 401 `FailoverError` results across configured fallback providers.

## Solution Assessment

No: current main's cooldowns, finite candidate list, and shorter `auth_permanent` backoff are useful partial safeguards, but they are not the best complete solution to this billing-safety failure. The safer path is a targeted auth-failover lockout/alert coordinated with, but not hidden behind, the broader #13615 rate/budget-control design.
