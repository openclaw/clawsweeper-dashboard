## Summary

Keep open: current main has finite fallback chains and auth/billing cooldowns, but it still lacks the central billing-safety behavior requested here: a spend/rate guard or user-visible lockout after repeated auth failures across fallback providers.

## What This Changes

The issue requests hard failover limits, user-visible alerts, and safer billing defaults for autonomous cross-provider model failover after repeated auth failures.

## Best Possible Solution

Add a targeted auth-failover safety guard that counts repeated auth/billing failures across configured fallback providers, pauses further model calls for a defined window, surfaces a clear user-visible alert, and documents how it complements #13615.

## Reproduction Assessment

Unclear: the issue has concrete version, environment, provider, log-event, and workaround details, so a simulated repro can be built by forcing repeated 401 FailoverError results across fallback providers. I would not use a live paid-provider repro for routine triage.

## Solution Assessment

No: current main's finite candidate loop and cooldowns are useful partial protections, but they are not the best complete solution because they still allow cross-provider fallback without a spend/rate guard or user-visible lockout.
