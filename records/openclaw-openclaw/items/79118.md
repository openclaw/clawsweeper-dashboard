## Summary

Keep open: current main still records provider-webhook speech without entering the auto-response loop, and the linked open fix PR should be reviewed or landed before this issue closes.

## What This Changes

The issue requests that Telnyx and other provider webhook `call.speech` transcripts trigger the same voice-call auto-response path used by streaming transcripts.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79336, or an equivalent waiter-aware webhook-path fix, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows Telnyx `call.transcription` becomes `call.speech`, current main only forwards that event to the manager, and the manager records it without triggering `handleInboundResponse`.

## Solution Assessment

Yes. A provider-agnostic webhook-server handoff after successful final transcript persistence is the narrowest maintainable fix, as long as it preserves transcript-waiter and replay/dedupe behavior.
