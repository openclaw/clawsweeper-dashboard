## Summary

Keep open: current main still lacks the provider-webhook speech-to-auto-response handoff, and an open same-reporter fixing PR with closing syntax should be reviewed or landed before this issue closes.

## What This Changes

The issue reports that Telnyx webhook-delivered final speech is recorded after the inbound greeting but never handed to the voice-call auto-response loop.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79336 or an equivalent voice-call patch that routes final generic webhook `call.speech` events into auto-response while preserving transcript waiter dedupe.

## Reproduction Assessment

Yes. Source inspection shows Telnyx `call.transcription` becomes `call.speech`, then current main records the transcript and transitions to `listening` without invoking the response loop; I did not run a live carrier call in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is the linked PR's provider-agnostic handoff from the webhook event path into `handleInboundResponse`, gated so transcript waiters and duplicate/replay cases do not double-reply.
