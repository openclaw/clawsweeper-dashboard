## Summary

Keep open: current main still has a provider-webhook transcript path that records Telnyx `call.transcription` as `call.speech` but never enters the classic auto-response handoff. The related Telnyx answering issue is fixed, so this is a distinct follow-up after the greeting/listening step.

## What This Changes

The issue asks the voice-call plugin to trigger agent auto-responses after Telnyx webhook `call.transcription` events, not only after Twilio media-stream transcripts.

## Best Possible Solution

Keep this issue open and land a voice-call plugin fix that routes final non-streaming provider webhook transcripts into the same auto-response handoff used by media-stream transcripts.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a Telnyx `call.transcription` webhook becomes `call.speech`, then current main only records the transcript and transitions to `listening`. I did not run a live Telnyx call in this read-only review.

## Solution Assessment

Yes, the maintainable fix is a provider-agnostic final-speech auto-response hook owned by the voice-call runtime, gated so active transcript waiters and mismatched turn tokens do not produce duplicate replies.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79118 in the `voice-call` plugin. Current main answers Telnyx inbound calls and plays the greeting, but final Telnyx `call.transcription` webhooks normalize to `call.speech` and only run `manager.processEvent`; unlike the Twilio media-stream transcript path, they never trigger `handleInboundResponse`, so the call remains in `listening`. Add a provider-agnostic auto-response handoff for final webhook speech events for inbound calls and outbound `conversation` calls. Keep the manager/provider boundary clean: prefer a typed runtime hook or webhook-runtime orchestration rather than importing core agent runtime into manager helper modules. Preserve `continueCall` transcript waiter and turn-token semantics so resolved waiter events do not double-trigger an agent response. Add regression coverage for the Telnyx/non-streaming webhook path and a no-double-response waiter case, and add a single-line user-facing `CHANGELOG.md` fix entry. Do not change Telnyx answer-call behavior from the prior https://github.com/openclaw/openclaw/issues/58231 fix unless needed for the new tests.
