## Summary

Keep open. Current main still has the hot-mode skip path, so the PR is not obsolete, but it is not mergeable as-is because it changes the documented hot-mode restart contract by default, lacks after-fix real behavior proof, and needs current-main config/docs alignment.

## What This Changes

The PR adds `gateway.reload.fallbackToRestart`, defaults it to true, routes restart-required hot-mode reload plans through restart when enabled, and adds focused reload tests plus zod/type coverage.

## Best Possible Solution

Land a rebased version only after maintainers decide the hot-mode default policy, with the setting implemented in the current settings module and the types, zod schema, help/labels, Gateway docs, and focused tests kept in sync.

## Reproduction Assessment

Yes, at source level. On current main, a restart-required config change while `gateway.reload.mode` is `hot` reaches the `plan.restartGateway` branch, logs the hot-mode warning, and returns before scheduling a restart.

## Solution Assessment

No as written. The fallback mechanism is narrow, but defaulting it on reverses the current hot-mode no-auto-restart contract and the branch must be adapted to the current reload-settings split and public config surfaces.
