## Summary

Keep this PR open. Current main still has the legacy hot-mode behavior, so the proposed fallback is not obsolete, but the branch is not merge-ready because the new public config option is not carried through the generated schema/help/docs surfaces and the default restart behavior needs maintainer judgment.

## What This Changes

The PR adds `gateway.reload.fallbackToRestart`, defaults it to `true`, makes hot-mode restart-required reloads queue a restart unless opted out, and extends gateway reload schema/types/tests.

## Best Possible Solution

Keep this PR as an implementation candidate, rebase it onto current main, decide whether hot-mode restart fallback should default on or require opt-in, then update the complete public config contract: types, zod schema, field help/labels, generated base schema, gateway docs, and focused reload tests.

## Reproduction Assessment

Yes. A focused reproduction is to start from `gateway.reload.mode: "hot"`, change a restart-required path such as `gateway.port` or `plugins`, and observe that current main logs the hot-mode warning and does not call the restart path; the PR’s added harness tests model that path.

## Solution Assessment

Unclear as currently proposed. The fallback mechanism is narrow and plausible, but the default behavior and public config contract need maintainer review and generated/docs alignment before this is the best mergeable solution.
