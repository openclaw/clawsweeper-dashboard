## Summary

Keep open: current main has adjacent startup retry/connect fixes but does not implement this server-side hold, and this active PR still proposes a meaningful Gateway behavior change. It needs conflict resolution, current-head real behavior proof, and maintainer acceptance of the 20s held-request semantics before merge.

## What This Changes

The PR adds a Gateway startup barrier that waits startup-gated RPCs until post-attach sidecars finish, then falls back to the existing retryable startup `UNAVAILABLE` response on timeout.

## Best Possible Solution

Rebase onto current main and either merge a Gateway-owned bounded barrier with fresh restart proof, or explicitly close it as superseded if maintainers choose the fail-fast retryable startup path as the canonical design.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run in this read-only review. Source inspection shows the method-level gate still returns immediate retryable `UNAVAILABLE`, but current main also rejects startup-window WebSocket connects with retryable startup details, so the exact cached-browser symptom needs fresh proof.

## Solution Assessment

Unclear: a bounded server-side barrier is a plausible durable fix for old or cached clients, but it now competes with current main's canonical startup retry strategy and must be rebased and proven before maintainers can judge it best.
