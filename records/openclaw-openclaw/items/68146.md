## Summary

Keep open: current main and the latest release still fail fast for startup-gated RPCs and do not include the server-side barrier, but the PR is conflicting and still needs exact-head real restart proof plus maintainer acceptance of the delayed-request tradeoff.

## What This Changes

The branch adds a StartupGateBarrier and gateway request-context wiring so startup-gated RPCs wait server-side until post-attach clears the startup set, with tests for wait, timeout, bypass, and legacy no-barrier behavior.

## Best Possible Solution

Refresh the branch, preserve the descriptor-derived startup method set, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Solution Assessment

Conditionally yes. A gateway-owned barrier is the right layer for already-open cached UI bundles that cannot receive new client retry code, but the branch still needs conflict refresh, exact-head proof, and maintainer acceptance of the delayed-failure tradeoff.
