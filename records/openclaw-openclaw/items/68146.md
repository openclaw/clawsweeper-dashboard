## Label Justifications

- `P2`: This is a normal-priority Gateway startup race fix with limited blast radius and no emergency data-loss or security signal.
- `merge-risk: 🚨 compatibility`: Existing clients that expect immediate retryable startup errors for gated RPCs would instead wait for the server-side barrier or timeout.
- `merge-risk: 🚨 availability`: The PR intentionally holds startup-gated requests on the server for up to 20 seconds, increasing pending work if sidecars stall.

## Summary

Keep open: current main still fail-fasts startup-gated RPCs, so the server-side hold is not implemented on main, but this PR is blocked by merge conflicts, insufficient exact-head proof, and compatibility/availability tradeoffs.

## What This Changes

The branch adds a Gateway startup barrier that holds startup-gated requests until post-attach sidecars finish, then falls back to the existing retryable UNAVAILABLE response on timeout.

## Best Possible Solution

Refresh the branch, keep the descriptor-derived startup method set as the single source of truth, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Reproduction Assessment

Yes. Source inspection shows current main returns UNAVAILABLE immediately while startup-gated methods remain in the set, and the linked issue logs show a 0 ms `chat.history` failure during the post-ready sidecar window; I did not run a live restart in this read-only review.

## Solution Assessment

Conditionally yes. A server-side barrier is the right layer for already-open cached UI bundles because client-only retry cannot update an already-loaded old bundle, but the branch still needs refresh, exact-head proof, and maintainer acceptance of the delayed-failure tradeoff.
