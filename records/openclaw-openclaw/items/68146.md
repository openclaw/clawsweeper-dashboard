## Summary

Keep this PR open, but it is not merge-ready: the server-side startup barrier is a plausible Gateway fix, yet the branch is stale against current startup retry contracts and now conflicts with main.

## What This Changes

The PR threads a Gateway startup barrier through request context so startup-gated methods wait for post-attach sidecars, with timeout fallback and focused Gateway tests.

## Best Possible Solution

If maintainers want server-side recovery for cached or older clients, rebase the barrier over the current core method descriptors and startup retry-detail contract; otherwise close it in favor of the shipped handshake/client retry behavior.

## Reproduction Assessment

No high-confidence current-main end-to-end reproduction was established for the cached Control UI restart case. Source inspection still shows the method-level gate fail-fasts when reached, but current main also rejects startup-pending WebSocket handshakes before post-connect RPCs can run.

## Solution Assessment

No, not as currently written. A server-side barrier may be the right maintainable fix, but this branch must preserve current startup retry details, derive the current startup-gated method set, and reconcile with the newer WebSocket startup policy before merge.
