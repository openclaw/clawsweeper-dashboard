## Summary

Keep this PR open: current main and the latest release still fail fast for startup-gated Gateway RPCs, while this branch is the open server-side mitigation for cached or older clients; it is not merge-ready because the branch conflicts, exact-head real gateway proof is still insufficient, and the delayed-failure tradeoff needs maintainer acceptance.

## What This Changes

The PR adds a Gateway startup barrier so startup-gated RPCs wait server-side for post-attach sidecar registration before falling back to retryable startup UNAVAILABLE.

## Best Possible Solution

Refresh the branch onto current main, keep the descriptor-derived startup method set as the source of truth, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

Conditionally yes: a server-side barrier is the right layer for already-open cached UI bundles because client-only retry cannot update an already-loaded old bundle. The branch still needs refresh, exact-head proof, and maintainer acceptance of the delayed-failure tradeoff.
