## Summary

Keep open: current main still lacks the server-side hold this PR is aiming for, so the central change is not obsolete. The branch is not mergeable as-is because it is stale against newer Gateway startup retry contracts and needs Gateway policy review for bounded request holding during startup.

## What This Changes

The PR adds a Gateway startup barrier that waits up to 20 seconds for post-attach sidecars before dispatching startup-gated requests, threads it through request context and startup, and adds focused Gateway tests.

## Best Possible Solution

Finish a Gateway-owned bounded barrier or equivalent queue on top of the descriptor-derived startup method set, preserving the shared startup-sidecars retry details on timeout and validating restart behavior with real Gateway logs.

## Reproduction Assessment

No high-confidence current-main end-to-end reproduction was run. Source inspection shows the fail-fast startup-gated RPC path still exists, but newer websocket startup-pending handshake handling changes how often a connected client can reach that path.

## Solution Assessment

No, not as currently written. A server-side startup barrier may be the right durable fix for cached or older clients, but this branch must preserve current retry details and the descriptor-derived startup-gated method set before it can be mergeable.
