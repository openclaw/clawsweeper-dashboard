## Summary

Keep open: current main still fail-fasts startup-gated RPCs during the post-attach window, and this PR is a plausible targeted fix. It is not merge-ready yet because the latest head still relies on historical gateway logs plus a direct barrier exercise rather than current-head real restart proof, and the server-side hold semantics need maintainer acceptance.

## What This Changes

The branch adds a `StartupGateBarrier`, threads it through Gateway request and post-attach context, waits up to 20 seconds before falling back to retryable `UNAVAILABLE`, and adds focused unit coverage.

## Best Possible Solution

Merge a Gateway-owned bounded server-side wait or queue only after current-head restart proof shows held startup-gated requests release after sidecars and maintainers accept the semantics for all startup-gated methods.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection and the linked issue logs show current main still returns immediate retryable `UNAVAILABLE` between Gateway listen and sidecar readiness.

## Solution Assessment

Unclear as a merge-ready solution. A bounded barrier is plausible for cached clients, but applying it to every startup-gated RPC changes request lifetimes and needs current-head live proof plus maintainer approval.
