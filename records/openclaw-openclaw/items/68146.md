## Summary

This PR should stay open because current main still lacks the server-side startup barrier and the branch contains plausible Gateway bug-fix work, but it is not mergeable as written after newer startup retry changes on main.

## What This Changes

The PR adds a Gateway startup barrier that waits up to 20 seconds for post-attach sidecars before dispatching startup-gated requests, with tests for wait, timeout, bypass, and post-attach opening.

## Best Possible Solution

If maintainers want server-side recovery for cached or older clients, finish the barrier on top of current descriptor-derived startup methods and startup-sidecars retry details, then validate bounded request holding during restart.

## Reproduction Assessment

No high-confidence current-main end-to-end reproduction was run in this read-only review. Source inspection confirms current main still has an immediate startup-gated UNAVAILABLE path, while newer websocket reconnect handling changes how often post-connect RPCs can reach that path.

## Solution Assessment

No, not as currently written. A Gateway-side barrier may be the right durable fix for cached clients, but this branch must preserve current retry details and the descriptor-derived startup-gated method set before it can be mergeable.
