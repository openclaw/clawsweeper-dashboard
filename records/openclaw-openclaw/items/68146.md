## Label Justifications

- `P2`: This is a normal-priority Gateway startup race fix with limited blast radius and no emergency data-loss or security signal.
- `merge-risk: 🚨 compatibility`: Existing clients that expect immediate retryable startup errors for gated RPCs would instead wait for the server-side barrier or timeout.
- `merge-risk: 🚨 availability`: The PR intentionally holds startup-gated requests on the server for up to 20 seconds, increasing pending work if sidecars stall.

## Summary

Keep this PR open: current main still fail-fasts startup-gated Gateway RPCs, and this branch contains a meaningful server-side hold that is not implemented by the shipped client retry mitigations. It is not merge-ready because the branch conflicts with main, exact-head real restart proof is still missing, and maintainers need to accept the compatibility and availability tradeoff of holding requests server-side.

## What This Changes

The PR adds a Gateway startup barrier that waits up to 20s for post-attach sidecars before dispatching startup-gated RPCs or returning the existing retryable UNAVAILABLE fallback, plus request-context wiring and unit tests.

## Best Possible Solution

Refresh the branch onto current main, keep the descriptor-derived startup method set as the source of truth, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Reproduction Assessment

Yes. Source inspection shows current main returns UNAVAILABLE immediately while startup-gated methods remain in the set, and the linked issue logs show a 0ms chat.history failure during the post-ready sidecar window; I did not run a live restart in this read-only review.

## Solution Assessment

Conditionally yes. A server-side barrier is the right layer for already-open cached UI bundles because client-only retry cannot update an already-loaded old bundle, but the branch still needs refresh, exact-head proof, and maintainer acceptance of the delayed-failure tradeoff.
