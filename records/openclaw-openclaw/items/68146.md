## Summary

Keep open for maintainer review. The PR is a plausible server-side bug fix with real after-fix logs, but current main has since changed the gateway startup contract and this branch would regress the current startup retry metadata and gated-method coverage if rebased as-is.

## What This Changes

The PR adds a Gateway startup barrier so startup-gated requests wait for post-attach sidecars before dispatch, with a timeout fallback and focused gateway tests.

## Best Possible Solution

Land a rebased gateway-owned barrier only if maintainers want server-side request holding; otherwise keep the shipped startup-pending handshake/client retry policy and close this PR explicitly.

## Reproduction Assessment

No high-confidence current-main end-to-end cached-Control-UI reproduction was established. Source inspection does show the method-level gate still fail-fasts if reached, while current main also rejects startup-pending WebSocket handshakes before post-connect RPCs can run.

## Solution Assessment

No, not as currently written. The server-side barrier may be a maintainable fix, but this branch must be rebased to preserve the current startup-sidecars detail contract and expanded gated method set before it is a safe merge candidate.
