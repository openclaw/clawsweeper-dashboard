## Label Justifications

- `P2`: This is a normal-priority Gateway startup race fix with limited blast radius and no emergency data-loss or security signal.
- `merge-risk: 🚨 compatibility`: Existing clients that expect immediate retryable startup errors for gated RPCs would instead wait for the server-side barrier or timeout.
- `merge-risk: 🚨 availability`: The PR intentionally holds startup-gated requests on the server for up to 20 seconds, increasing pending work if sidecars stall.

## Summary

Keep open: current main still fail-fasts startup-gated Gateway RPCs, and the PR’s server-side hold is not already implemented by the shipped UI/TUI retry work. The branch is not merge-ready because it conflicts with main, exact-head real behavior proof is still insufficient, and maintainers need to accept the changed startup failure timing.

## What This Changes

The PR adds a Gateway `StartupGateBarrier` and threads it through request handling so startup-gated RPCs wait for post-attach completion before falling back to retryable `UNAVAILABLE`.

## Best Possible Solution

Refresh the branch, keep the descriptor-derived startup method set as the single source of truth, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Reproduction Assessment

Yes, source-level. Current main returns retryable `UNAVAILABLE` immediately while startup-gated methods remain in the set, and the linked issue logs show a 0 ms `chat.history` failure during the post-ready sidecar window; I did not run a live restart in this read-only review.

## Solution Assessment

Conditionally yes. A server-side barrier is the right layer for already-open cached UI bundles because client-only retry cannot update an already-loaded old bundle, but the branch still needs conflict resolution, exact-head proof, and maintainer acceptance of the delayed-failure tradeoff.
