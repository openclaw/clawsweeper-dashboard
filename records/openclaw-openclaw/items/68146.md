## Summary

## What This Changes

The PR adds a Gateway `StartupGateBarrier` that lets startup-gated RPCs wait up to 20 seconds for post-attach before falling back to retryable `UNAVAILABLE`, threads it through request context/startup wiring, and adds focused unit tests.

## Best Possible Solution

Refresh the branch against current main, preserve the descriptor-derived method set, and land the bounded server-side wait only after exact-head restart proof and maintainer acceptance of the startup timing tradeoff.

## Reproduction Assessment

Yes. Source inspection shows current main returns immediate retryable `UNAVAILABLE` while startup-gated methods remain unavailable, and the linked logs show the 0 ms `chat.history` failure during the post-ready sidecar window.

## Solution Assessment

Conditionally yes. A gateway-owned barrier is the right layer for already-open cached UI bundles that cannot receive new retry code, but this branch still needs conflict refresh, exact-head proof, and maintainer acceptance of delayed startup failures.
