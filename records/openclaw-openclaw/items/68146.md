## Summary

Current main already solves the reported restart race with a shipped WebSocket startup-connect retry path, so the PR's request-level barrier is now redundant and would add request-lifetime semantics without a confirmed remaining gap.

## What This Changes

The PR adds a Gateway `StartupGateBarrier` that waits startup-gated RPCs until post-attach sidecars open, with request-context wiring and focused unit tests.

## Best Possible Solution

Keep the shipped connect-level startup retry behavior and open a new focused issue only if a live client can still send startup-gated RPCs after handshake during sidecar startup.

## Reproduction Assessment

No. Current main no longer exposes the normal repro path because WebSocket clients receive a retryable startup connect response before a successful hello while sidecars are pending.

## Solution Assessment

No as a merge path. A bounded request barrier was plausible earlier, but the narrower shipped connect-level gate avoids accepting clients during the sidecar window and makes the PR's extra request hold unnecessary.
