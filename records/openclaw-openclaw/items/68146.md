## Summary

Keep this PR open: current main still returns immediate retryable `UNAVAILABLE` for startup-gated RPCs, and this branch contains a distinct server-side mitigation for already-open cached clients. It is not merge-ready because the branch is conflicting, the after-fix gateway journal is not from this exact re-architected head, and maintainers still need to accept the startup timing tradeoff.

## What This Changes

The PR adds a Gateway `StartupGateBarrier` so startup-gated requests wait until post-attach sidecars open the barrier, with a 20-second fallback to the existing retryable startup `UNAVAILABLE` response, plus focused dispatch tests.

## Best Possible Solution

Refresh the branch, keep the descriptor-derived startup method set as the source of truth, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Reproduction Assessment

## Solution Assessment

Conditionally yes. A server-side barrier is the right layer for already-open cached UI bundles because client-only retry cannot update an already-loaded old bundle, but the branch still needs conflict refresh, exact-head proof, and maintainer acceptance of delayed-failure tradeoffs.
