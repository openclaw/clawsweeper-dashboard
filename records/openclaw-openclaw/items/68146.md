## Summary

Keep this PR open: it is a plausible gateway bug fix with real after-fix logs, but the branch is conflicting and stale against current main’s startup retry contract, so it is not safe to merge as written.

## What This Changes

The PR adds a Gateway startup barrier so startup-gated RPCs wait for post-attach sidecars before dispatching, with a timeout fallback and focused gateway tests.

## Best Possible Solution

Land a rebased gateway-owned barrier or equivalent queue only if maintainers want older cached clients to recover without relying on client retries; the finished version should preserve the shared startup-sidecars retry contract and the full current gated method set.

## Reproduction Assessment

No high-confidence current-main end-to-end cached-Control-UI reproduction was established. Source inspection does show startup-gated RPCs still fail fast if they reach handleGatewayRequest, while current main also rejects startup-pending WebSocket handshakes before post-connect RPCs can run.

## Solution Assessment

No, not as currently written. The server-side barrier may be a maintainable fix, but this branch must be rebased to preserve the current startup-sidecars detail contract and expanded gated method set before it is safe to merge.
