## Summary

Keep this PR open. Current main still fail-fasts startup-gated `chat.history` and `models.list` requests, so the server-side barrier proposed here is not implemented or superseded; the patch is plausibly targeted, but it is missing the required changelog entry and still needs maintainer review against current main.

## What This Changes

This PR adds a Gateway startup barrier so startup-gated `chat.history` and `models.list` requests wait for post-attach sidecars before dispatching, with a retryable `UNAVAILABLE` timeout fallback and focused gateway tests.

## Best Possible Solution

Land a gateway-owned server-side wait or equivalent bounded queue for startup-gated methods if maintainers want cached or older clients to recover without relying on client retries. The final version should keep the retryable timeout fallback and cover dispatch-after-sidecars, timeout fallback, non-gated bypass, and legacy no-barrier contexts.

## Reproduction Assessment

Yes. On current main, a high-confidence static reproduction is to call `handleGatewayRequest` for `models.list` or `chat.history` while `context.unavailableGatewayMethods` contains the method; it immediately returns retryable `UNAVAILABLE` without dispatching.

## Solution Assessment

Yes, with maintainer validation. A Gateway-owned barrier is the narrowest maintainable fix for cached clients that cannot retry, while preserving the existing retryable fallback if sidecars do not become ready.
