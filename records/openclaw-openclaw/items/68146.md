## Summary

Keep open: current main still fail-fasts startup-gated requests, and this PR is a plausible targeted server-side fix with real after-fix log proof. It is not mergeable as-is because the branch conflicts with main and the timeout/no-barrier fallback drops the current startup retry metadata.

## What This Changes

The PR adds a Gateway startup barrier so startup-gated requests wait for post-attach sidecars before dispatching, with timeout fallback handling and focused Gateway tests.

## Best Possible Solution

Land a rebased Gateway-owned barrier or equivalent server-side queue only if maintainers accept the request-lifetime behavior, while preserving current startup retry metadata, the expanded gated-method list, current handshake semantics, and focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main immediately returns startup `UNAVAILABLE` from `handleGatewayRequest` while a method is in `context.unavailableGatewayMethods`, and startup keeps that set populated until post-attach sidecars clear it. I did not run a live restart flow in this read-only review.

## Solution Assessment

No, not as-is. The server-side barrier is a reasonable narrow fix direction, but the branch must be rebased and the fallback must preserve the current `startup-sidecars` retry details before it is mergeable.
