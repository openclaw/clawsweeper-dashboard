## Summary

Keep open: current main still returns immediate startup `UNAVAILABLE` for gated `chat.history` and `models.list` requests, while this PR's server-side barrier remains a plausible hardening patch beyond the shipped client/TUI retry work. The only discrete review finding is the missing required changelog entry.

## What This Changes

This PR adds a Gateway startup barrier that waits up to 20 seconds for post-attach sidecars before dispatching startup-gated `chat.history` and `models.list` requests, with request-context wiring and focused Gateway tests.

## Best Possible Solution

Land a rebased Gateway-owned barrier or equivalent queue that opens when post-attach clears startup-gated methods, preserves the retryable timeout fallback, keeps focused coverage, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. On current main, calling `handleGatewayRequest` for `models.list` or `chat.history` with that method in `context.unavailableGatewayMethods` returns retryable `UNAVAILABLE` without dispatching, and the startup path creates that gated set before post-attach sidecars clear it.

## Solution Assessment

Yes, with maintainer validation. A server-owned barrier is the narrowest maintainable fix for cached clients that cannot retry, while the existing client/TUI retries remain complementary fallbacks.
