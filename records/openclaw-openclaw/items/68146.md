## Label Justifications

- `P2`: This is a normal-priority Gateway startup race fix with limited blast radius and no emergency data-loss or security signal.
- `merge-risk: 🚨 compatibility`: Existing clients that expect immediate retryable startup errors for gated RPCs would instead wait for the server-side barrier or timeout.
- `merge-risk: 🚨 availability`: The PR intentionally holds startup-gated requests on the server for up to 20 seconds, increasing pending work if sidecars stall.

## Summary

Keep open: current main still returns startup-gated Gateway requests with immediate retryable `UNAVAILABLE`, and this PR remains the only open server-side candidate for the cached-Control-UI restart edge; it is not merge-ready because the branch conflicts with main, exact-head real gateway proof is insufficient, and maintainers need to accept the delayed-failure tradeoff.

## What This Changes

The PR adds a Gateway startup barrier so startup-gated RPCs wait server-side until post-attach sidecar registration opens the gate, with timeout fallback to the existing retryable startup `UNAVAILABLE` response.

## Best Possible Solution

Refresh the branch onto current main, keep the descriptor-derived startup method set as the source of truth, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Reproduction Assessment

Yes from source for the startup window: current main immediately returns retryable UNAVAILABLE while methods remain startup-gated, and the linked logs show a 0 ms chat.history failure during that gap. I did not run a live gateway restart in this read-only review.

## Solution Assessment

Conditionally yes: a server-side barrier is the right layer for already-open cached UI bundles because client-only retry cannot update an already-loaded old bundle. The branch still needs refresh, exact-head proof, and maintainer acceptance of the delayed-failure tradeoff.
