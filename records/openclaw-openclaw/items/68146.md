## Summary

Keep this PR open: it is a recent, plausible Gateway bug-fix PR with real after-fix logs, but it is stale/conflicting against current Gateway startup behavior and still has correctness blockers before merge.

## What This Changes

Adds a Gateway startup barrier that waits startup-gated `chat.history` and `models.list` requests until post-attach sidecars finish, with timeout fallback and focused Gateway tests.

## Best Possible Solution

If maintainers want server-side recovery for cached or older clients, rebase the barrier onto current Gateway method specs and startup retry details, then validate it against current reconnect and restart-burst behavior.

## Reproduction Assessment

No high-confidence end-to-end current-main reproduction was established for the cached Control UI restart case. Source inspection does show the method-level startup gate still fail-fasts when reached, while current main also rejects startup-pending WebSocket handshakes before post-connect RPCs can run.

## Solution Assessment

No, not as currently written. A server-side barrier may be the right fix, but this branch must preserve current retry metadata, use the current startup-gated method set, and be revalidated against the newer startup handshake policy.
