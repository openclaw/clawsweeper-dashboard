## Summary

Keep this PR open: current main does not implement the server-side startup barrier, and the PR has useful real-behavior evidence, but the branch is conflicting and would need to preserve newer startup retry contracts before merge.

## What This Changes

This PR threads a `StartupGateBarrier` through gateway request handling so startup-gated requests wait for post-attach sidecars before dispatching, with timeout fallback tests.

## Best Possible Solution

Rebase the gateway-owned barrier over current main only if maintainers want server-side request holding; the finished version should preserve `startup-sidecars` retry details and the expanded gated method set.

## Reproduction Assessment

No high-confidence current-main end-to-end reproduction was established for the cached Control UI restart case. Source inspection does show the method-level gate still fail-fasts when reached, while current main also rejects startup-pending websocket handshakes before post-connect RPCs can run.

## Solution Assessment

No, not as currently written. A server-side barrier may be a maintainable fix, but this branch must be rebased to preserve current startup retry details and the expanded gated method set before it is safe to merge.
