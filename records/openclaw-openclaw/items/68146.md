## What This Changes

The PR adds a Gateway startup barrier that waits up to 20 seconds for startup-gated requests to dispatch after post-attach sidecars, with request-context wiring and focused Gateway tests.

## Best Possible Solution

Finish a Gateway-owned bounded barrier or equivalent queue on current main, using the descriptor-derived startup method set and canonical `startup-sidecars` retry details, then validate the restart-while-tab-open flow with real Gateway logs and focused tests.

## Reproduction Assessment

No high-confidence end-to-end current-main reproduction was run in this read-only review. Source inspection shows current main still fail-fasts startup-gated RPC methods, while newer websocket startup-pending handling and client retries affect how often a connected user reaches that exact path.

## Solution Assessment

No, not as currently written. A server-side startup barrier may be the durable fix for cached or older clients, but this branch must preserve current retry details, carry forward the descriptor-derived gated method set, and be rebased before it is mergeable.
