## Summary

Keep this PR open: current main still fail-fasts startup-gated Gateway RPCs, so the server-side hold is not already implemented, but the branch is dirty and still needs exact-head real behavior proof plus maintainer acceptance of delayed startup failures.

## What This Changes

The PR adds a Gateway startup barrier so startup-gated requests wait for post-attach sidecars, then fall back to the existing retryable startup UNAVAILABLE response on timeout.

## Best Possible Solution

Refresh the branch, keep the descriptor-derived startup method set as the source of truth, and land the bounded server wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Reproduction Assessment

Yes. Source inspection shows current main returns retryable UNAVAILABLE immediately while startup-gated methods remain in the set, and the linked issue logs show a 0 ms `chat.history` failure during the post-ready sidecar window; I did not run a live restart in this read-only review.

## Solution Assessment

Conditionally yes. A server-side barrier is the right layer for already-open cached UI bundles because client-only retry cannot update an already-loaded old bundle, but the branch still needs conflict refresh, exact-head proof, and maintainer acceptance of delayed-failure tradeoffs.
