## Label Justifications

- `P2`: This is a normal-priority gateway startup race fix with limited blast radius and no emergency data-loss or security signal.
- `merge-risk: 🚨 compatibility`: Existing clients that expect an immediate retryable startup error for gated RPCs would instead wait for the server-side barrier or timeout.
- `merge-risk: 🚨 availability`: The PR intentionally holds startup-gated requests on the server for up to 20 seconds, increasing pending work if sidecars stall.

## Summary

Keep open: current main and the latest release still fail fast for startup-gated gateway RPCs instead of holding them server-side, so this PR is not obsolete; it is not merge-ready because the branch is conflicting and the exact-head restart proof/compatibility decision are still missing.

## What This Changes

Adds a StartupGateBarrier to hold startup-gated gateway requests until post-attach sidecars are ready or a 20 second timeout fires, and adds focused gateway dispatch tests.

## Best Possible Solution

Refresh the branch onto current main, keep the descriptor-derived startup method set as the source of truth, and land the bounded server-side wait only after maintainers accept the delayed-failure tradeoff and exact-head restart proof shows held requests releasing after post-attach.

## Reproduction Assessment

Yes, by source inspection and reporter logs, though I did not run a live gateway restart repro in this read-only review. Current main still returns retryable UNAVAILABLE immediately when a startup-gated method is present in unavailableGatewayMethods, while shipped UI/TUI retries mitigate the normal client path.

## Solution Assessment

Conditionally yes: a server-side barrier is the right layer for cached pre-retry browser bundles because client-only retry cannot change an already-loaded old UI bundle. The broader timeout and pending-request behavior is a maintainer compatibility decision, and the branch still needs exact-head real behavior proof.
