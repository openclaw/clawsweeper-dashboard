## Summary

Keep open. Current main and the latest release still use the immediate retryable startup `UNAVAILABLE` path, so the server-side hold remains unique, but the PR is currently conflicting and still needs exact-head real gateway proof plus maintainer acceptance of the delayed-failure tradeoff.

## What This Changes

The PR adds a Gateway startup barrier that waits up to 20 seconds for post-attach sidecar completion before dispatching startup-gated RPCs, threads it through server request context, and adds gateway dispatch tests.

## Best Possible Solution

Refresh the branch onto current main, keep the descriptor-derived startup method set as the source of truth, and land the bounded wait only after exact-head restart proof and maintainer acceptance of delayed startup failures.

## Reproduction Assessment

Yes, by source inspection. Current main returns retryable `UNAVAILABLE` immediately while a method remains in `unavailableGatewayMethods`, and the linked logs show `chat.history` failing at 0 ms during the post-ready sidecar gap; I did not run a live restart in this read-only review.

## Solution Assessment

Conditionally yes. A server-side barrier is the right layer for cached pre-retry browser bundles because client-only retry cannot change an already-loaded old UI bundle, but the branch needs refresh, exact-head proof, and maintainer acceptance of the delayed-failure tradeoff.
