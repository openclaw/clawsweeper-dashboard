## Summary

Keep open: this is a plausible Gateway bug fix, but the current PR needs changes before merge because its fallback startup error shape drops retry metadata that current main now requires, and the branch is conflicting with main.

## What This Changes

The PR adds a Gateway startup barrier that holds startup-gated requests until post-attach sidecars finish, then dispatches them or returns the existing retryable startup fallback on timeout.

## Best Possible Solution

Land a rebased Gateway-owned barrier that reuses the current startup-unavailable protocol helpers, opens when post-attach clears the gate, preserves timeout and legacy fallback semantics, and keeps focused coverage for the barrier and existing client retries.

## Reproduction Assessment

Yes, source-level: current main returns retryable startup `UNAVAILABLE` before dispatch when a method is in `context.unavailableGatewayMethods`, and startup creates that gated set before post-attach sidecars clear it. I did not run a live restart flow in this read-only review.

## Solution Assessment

No, not as-is. The server-side barrier is a reasonable narrow fix direction, but the PR must preserve current `startup-sidecars` retry metadata and be rebased before it is mergeable.
