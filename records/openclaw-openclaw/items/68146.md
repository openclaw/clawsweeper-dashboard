## Summary

Keep this PR open for maintainer review: the server-side barrier is not implemented on current main and may still be a valid hardening direction, but the branch is stale against the current startup retry contract and method registry, so it is not safe to merge as written.

## What This Changes

The PR adds a Gateway startup barrier so startup-gated RPCs wait for post-attach sidecars before dispatching, with timeout fallback and focused Gateway tests.

## Best Possible Solution

If maintainers want server-side recovery for cached or older clients, rebase a Gateway-owned barrier over current core method specs and `startup-sidecars` retry details; otherwise close it in favor of the shipped retry and reconnect behavior.

## Reproduction Assessment

No high-confidence current-main end-to-end reproduction was established for the cached Control UI restart case. Source inspection does show the method-level gate still fail-fasts when reached, while current main also rejects startup-pending websocket handshakes before post-connect RPCs can run.

## Solution Assessment

No, not as currently written. A server-side barrier may be the right maintainable fix, but this branch must preserve current startup retry details and the expanded startup-gated method set before it is safe to merge.
