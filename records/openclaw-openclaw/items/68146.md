## Summary

Keep open for maintainer review: the server-side barrier targets a plausible Gateway startup race and includes copied after-fix logs, but the branch is conflicting and stale against current startup-sidecars retry metadata and method coverage, so it is not merge-ready.

## What This Changes

This PR adds a Gateway startup barrier that waits startup-gated requests until post-attach sidecars are ready, wires it through request context, and adds focused Gateway tests.

## Best Possible Solution

If maintainers want older or cached clients to recover without relying on client retry logic, rebase the server-side barrier onto the current startup-sidecars contract, preserve retry metadata, cover every startup-gated RPC, and validate restart/concurrency behavior before merge.

## Reproduction Assessment

Unclear for a full current-main user repro. Source inspection shows startup-gated RPCs still fail fast if reached, and the PR body includes before/after live logs, but I did not live-reproduce the cached pre-fix Control UI path against current main.

## Solution Assessment

Unclear as-is. A Gateway-owned barrier is a plausible fix for cached clients, but this branch must be rebased against the current startup-sidecars protocol shape and expanded method set before maintainers can judge it safely.
