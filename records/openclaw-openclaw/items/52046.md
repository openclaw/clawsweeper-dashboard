## Summary

Keep open: current main and v2026.6.8 still run heartbeat work through the normal heartbeat runner and retry/backpressure path, while the related open retry PR only addresses aggressive retry cadence rather than true non-blocking heartbeat isolation.

## What This Changes

The issue requests non-blocking heartbeat execution so heartbeat checks do not delay user message processing.

## Best Possible Solution

Keep this as the tracker for a maintainer-approved heartbeat isolation model, while handling the narrower retry regression through https://github.com/openclaw/openclaw/issues/40611 and https://github.com/openclaw/openclaw/pull/89989.

## Reproduction Assessment

Yes at source level: current main can be traced to serialized heartbeat admission, shared busy-skip retry, and an awaited getReplyFromConfig heartbeat turn. I did not establish a high-confidence live transport reproduction of the reported stall duration.

## Solution Assessment

No implementation shape is ready yet: a background worker thread is plausible, but the safer maintainer decision is the heartbeat isolation contract across session state, delivery ordering, and tool side effects. The open retry PR is only a partial mitigation for a narrower related bug.
