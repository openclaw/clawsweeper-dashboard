## Summary

Keep open: current main and v2026.6.8 still run heartbeats through the existing heartbeat runner/session path rather than a non-blocking worker or isolated execution lane. The narrower aggressive retry regression is already tracked by https://github.com/openclaw/openclaw/issues/40611 and https://github.com/openclaw/openclaw/pull/89989, but that does not cover the broader product request here.

## What This Changes

The issue requests non-blocking heartbeat execution, likely via a background worker or separate execution lane, so heartbeat checks do not delay user message processing.

## Best Possible Solution

Define a maintainer-approved heartbeat execution isolation model that preserves session state, delivery ordering, tool authority, and busy-lane semantics; keep the retry regression on https://github.com/openclaw/openclaw/issues/40611 separate.

## Reproduction Assessment

Yes at source level for the missing capability: current main and v2026.6.8 still run heartbeats through the normal heartbeat runner/session path. No live Telegram, Feishu, WebChat, or equivalent reproduction was run for the claimed 5-10 second user-visible delay.

## Solution Assessment

Unclear as an implementation: a background worker thread is plausible, but the best solution needs a maintainer-chosen heartbeat isolation and session/delivery contract rather than a timer-only or worker-only patch.
