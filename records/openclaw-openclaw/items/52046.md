## Summary

Keep open: current main and v2026.5.20 still run heartbeat as serialized, awaited main-session/session-lane work with busy-skip retry, while related open issues cover narrower retry spacing and idle-timeout behavior rather than this broader non-blocking heartbeat architecture request.

## What This Changes

The issue requests heartbeat checks that do not block normal message processing, likely through a background worker or isolated heartbeat execution lane.

## Best Possible Solution

Design a heartbeat-owned execution isolation contract, likely a dedicated lane or worker model, while preserving busy-skip safety, user/manual queue priority, session ordering, delivery guarantees, cancellation, and narrower retry/backoff fixes tracked separately.

## Reproduction Assessment

No high-confidence live reproduction of the exact 5-10 second delay was run. Source inspection does confirm the relevant current behavior: heartbeat is awaited in the existing reply/session execution path and guarded by busy-lane backpressure rather than a detached worker.

## Solution Assessment

Unclear as proposed. A background worker may be viable, but the maintainable solution needs an explicit heartbeat isolation contract so session state, delivery ordering, wake retries, cancellation, and tool side effects remain well-defined.
