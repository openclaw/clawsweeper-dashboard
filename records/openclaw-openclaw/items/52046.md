## Summary

Keep open: current main and the latest release still serialize/backpressure heartbeat turns rather than running heartbeat work on a non-blocking execution boundary, and related open work only covers narrower retry-delay or idle-gating mitigations.

## What This Changes

Request non-blocking heartbeat execution so heartbeat work does not delay normal message handling.

## Best Possible Solution

Keep this open as a product/runtime design item and define an isolated heartbeat execution boundary that preserves delivery ordering, session state, tool side effects, and existing retry/idle mitigations.

## Reproduction Assessment

Yes at source level for the missing capability: current main still defers on busy lanes and awaits heartbeat reply generation through the embedded runner. No live channel stall reproduction was run in this read-only review.

## Solution Assessment

Unclear: a background worker is a plausible direction, but not proven as the best fix until maintainers define the session/tool/delivery isolation contract. The safer end state is a designed heartbeat execution boundary rather than a timer-only change.
