## Summary

Keep open. Current main has added useful mitigations such as per-agent heartbeat fan-out, active-reply deferral, and queue priority, but heartbeat still runs as an awaited session/global agent turn rather than a detached worker or separate isolation lane, so the requested architecture remains unimplemented.

## What This Changes

The issue requests non-blocking heartbeat execution so heartbeat checks do not delay user message processing.

## Best Possible Solution

Design a heartbeat-owned execution isolation contract, likely a dedicated lane or worker model, while preserving current busy-skip safety, queue priority, session ordering, delivery guarantees, and narrower retry/backoff fixes tracked separately.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the relevant current behavior: heartbeat is awaited in the existing reply/session execution path and guarded by busy-lane backpressure rather than a detached worker.

## Solution Assessment

Unclear as proposed. A background worker may be viable, but the maintainable solution needs an explicit heartbeat isolation contract so session state, delivery ordering, wake retries, cancellation, and tool side effects remain well-defined.
