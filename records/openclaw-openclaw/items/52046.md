## Summary

Keep open: current main and the latest release still do not provide non-blocking heartbeat execution. Existing mitigations reduce heartbeat cost or retry timing, and related open work covers narrower retry/activity problems, but the requested separate heartbeat execution model remains a core product/architecture decision.

## What This Changes

Request non-blocking heartbeat execution so scheduled heartbeat work does not delay user message processing.

## Best Possible Solution

Define a maintainer-approved heartbeat isolation model that can run heartbeat turns without blocking user-message handling while preserving session ordering, delivery policy, and tool side-effect safety; keep the narrower retry-delay and idle-timeout work tracked separately.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source and release inspection do confirm that heartbeat work is still serialized/backpressured around the main lane and existing options are mitigations, not non-blocking execution.

## Solution Assessment

Unclear: a background worker is a plausible direction, but not yet proven as the best implementation boundary. The safer next step is maintainer design for an isolated heartbeat turn or lane that preserves session state, ordering, and delivery semantics.
