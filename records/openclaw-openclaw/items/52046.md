## Summary

Keep open. Current main still runs heartbeat as an awaited main/session turn with backpressure and retry semantics, while the requested non-blocking worker or separate execution-lane model remains unimplemented architecture work.

## What This Changes

Request non-blocking heartbeat execution, likely through a background worker or isolated lane, so heartbeat checks do not delay user message processing.

## Best Possible Solution

Design a dedicated heartbeat execution lane or worker isolation contract that preserves session consistency, delivery ordering, wake retries, and tool side-effect boundaries, while reviewing the narrower heartbeat race fixes separately.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show current main awaits heartbeat reply work after main/session busy checks, so a concrete reproduction would run a long heartbeat turn while inbound user traffic arrives.

## Solution Assessment

Unclear as stated. A background worker may be viable, but the maintainable solution needs a heartbeat isolation and lane contract before moving full heartbeat turns out of the serialized execution model.
