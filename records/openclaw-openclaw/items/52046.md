## Summary

Keep open: current main and the latest release still execute heartbeat as a serialized, awaited agent/session turn with busy-lane deferral and wake retries, not as a detached worker or isolated execution lane. Related open issues cover narrower retry, idle-timeout, and pending-delivery bugs, but none replaces the broader non-blocking heartbeat architecture request.

## What This Changes

Request a non-blocking heartbeat execution model so heartbeat checks do not delay user message processing.

## Best Possible Solution

Design a heartbeat-owned execution isolation contract, likely a dedicated lane or worker model, while preserving busy-skip safety, user/manual queue priority, session ordering, delivery guarantees, and narrower retry/backoff fixes tracked separately.

## Reproduction Assessment

No high-confidence live reproduction of the exact 5-10 second delay was run. Source inspection does confirm the relevant current behavior: heartbeat is awaited in the existing reply/session execution path and guarded by busy-lane backpressure rather than a detached worker.

## Solution Assessment

Unclear as proposed. A background worker may be viable, but the maintainable solution needs an explicit heartbeat isolation contract so session state, delivery ordering, wake retries, cancellation, and tool side effects remain well-defined.
