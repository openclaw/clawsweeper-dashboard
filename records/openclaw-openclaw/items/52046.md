## Summary

Keep open. Current main still runs heartbeat as an awaited full agent turn coordinated through shared main and session lane backpressure; recent work adds safer skips and retries, but it does not implement a non-blocking heartbeat worker or independent execution lane.

## What This Changes

The issue requests non-blocking heartbeat execution so scheduled heartbeat checks do not delay normal user message processing.

## Best Possible Solution

Design a dedicated heartbeat execution lane or worker-style isolation contract that preserves session consistency, delivery ordering, wake retries, and tool side effects, while reviewing narrower busy-skip fixes separately.

## Reproduction Assessment

No high-confidence live timing reproduction was established. Source inspection does show current main still awaits heartbeat reply work after shared main/session lane checks, so a concrete repro would need a long heartbeat run concurrent with an inbound user message.

## Solution Assessment

Unclear as stated. A background worker may be viable, but the safer maintainable path is to define the heartbeat isolation and lane contract before moving full heartbeat turns out of the current serialized execution model.
