## Summary

Keep open: current main and v2026.5.22 still implement heartbeat as serialized, awaited heartbeat/session-lane work with busy-skip retry, while the related open items cover narrower retry-spacing or idle-timeout behavior rather than this broader non-blocking heartbeat architecture request.

## What This Changes

The issue requests non-blocking heartbeat execution so heartbeat checks do not delay user message processing.

## Best Possible Solution

Design a heartbeat-owned execution isolation contract, likely a dedicated lane or worker model, while preserving busy-skip safety, user/manual queue priority, session ordering, delivery guarantees, cancellation, and separately tracked retry/backoff fixes.

## Reproduction Assessment

No high-confidence live reproduction of the exact 5-10 second user-visible delay was run. Source inspection does confirm the relevant current behavior: heartbeat remains awaited serialized work with busy-lane backpressure rather than a detached worker.

## Solution Assessment

Unclear as proposed. A background worker may be viable, but the maintainable solution needs an explicit heartbeat isolation contract so session state, delivery ordering, wake retries, cancellation, and tool side effects remain well-defined.
