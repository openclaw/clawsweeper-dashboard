## Summary

Keep open: current main and v2026.6.6 still run heartbeat work through the awaited reply path with main/session-lane deferral and wake-layer retry, while the open retry-delay PR only mitigates a narrower timing symptom.

## What This Changes

The issue requests a non-blocking heartbeat execution model so heartbeat checks do not delay user message processing.

## Best Possible Solution

Define a heartbeat-owned isolated execution contract, likely a dedicated lane or worker model, that prioritizes user turns while preserving session ordering, delivery, cancellation, wake retry behavior, and tool side effects.

## Reproduction Assessment

Partly: source inspection shows current main heartbeats are awaited reply turns and are backpressured by main/session lanes. The exact 5-10 second live channel delay was not reproduced in this read-only review.

## Solution Assessment

Unclear as proposed. A background worker may be viable, but the best solution needs an explicit heartbeat isolation contract rather than a timer-only tweak or the narrower retry-delay PR.
