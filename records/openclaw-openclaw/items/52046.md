## Summary

Keep open. Current main still runs heartbeat as a scheduled agent turn that is backpressured by the main/global command lane and the resolved session lane; it has useful mitigations for busy lanes, light context, timeouts, and isolated sessions, but it does not implement a separate heartbeat worker or non-blocking execution lane. The issue discussion remains accurate: GMTekAI's comment frames this as an architecture request rather than a timer bug, and Ryce's KeepMyClaw snapshot note is an interim guardrail rather than a core fix.

## What This Changes

Request non-blocking heartbeat execution so scheduled heartbeat checks do not delay normal inbound message processing, likely through a worker or isolated execution lane.

## Best Possible Solution

Keep this open as an architecture/product feature request. The desired end state is a dedicated heartbeat execution lane or worker-style isolated heartbeat turn model that can run heartbeat checks without delaying inbound user messages while preserving session consistency, tool side effects, delivery ordering, wake retries, and existing cost/context mitigations such as isolatedSession, lightContext, timeoutSeconds, and skipWhenBusy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction is included, but the current-main code path gives high-confidence static evidence: heartbeat is skipped around busy lanes and, when it runs, awaits the normal reply pipeline on shared session/global lanes. A timing repro would still need a config and a long heartbeat turn concurrent with an inbound message.

## Solution Assessment

Is this the best way to solve the issue? Unclear as stated: a background worker may be viable, but the maintainable fix should first define the heartbeat isolation/lane contract rather than only moving work into worker_threads. The current implementation is a deliberate serialized/backpressured design, not the requested non-blocking solution.
