## Summary

Keep open: current main and the latest release still defer and retry heartbeat work on shared lanes and then await the normal reply path, while the open related retry PR only addresses a narrower cadence problem rather than independent heartbeat execution.

## What This Changes

The issue asks for heartbeat checks to run in a non-blocking execution path so incoming messages are not delayed while heartbeat work is running.

## Best Possible Solution

Keep this as the canonical tracker for a maintainer-approved heartbeat isolation model, while handling the narrower retry cadence work through https://github.com/openclaw/openclaw/issues/40611 and https://github.com/openclaw/openclaw/pull/89989.

## Reproduction Assessment

Yes at source level: current main can be traced to busy-skip deferral and an awaited getReplyFromConfig heartbeat turn. I did not establish a live transport reproduction of the exact 5-10 second user-visible delay.

## Solution Assessment

Unclear: a background worker or separate lane is plausible, but the best fix needs a maintainer-defined isolation contract across session state, delivery ordering, and tool side effects.
