## Summary

Keep open. Current main still treats heartbeat as a scheduled full agent turn coordinated through shared main/session reply lanes, and the discussion correctly frames this as architecture/product work rather than an already-fixed timer bug; Ryce's KeepMyClaw snapshot note is useful interim context, not a core implementation.

## What This Changes

Request heartbeat checks run on a separate non-blocking execution path so inbound user messages are not delayed during long heartbeat turns.

## Best Possible Solution

Keep this as an architecture feature request and design a dedicated heartbeat execution lane or isolated heartbeat turn contract that preserves session consistency, delivery ordering, wake retries, and tool side effects before implementation.

## Reproduction Assessment

No high-confidence live timing reproduction was established. Source inspection does show the current-main path still runs heartbeats as full awaited agent turns through shared lanes, so a timing repro would need a long heartbeat run concurrent with an inbound user message.

## Solution Assessment

Unclear as stated. A background worker may be viable, but the safer maintainable solution is to first define the heartbeat isolation and lane contract rather than only moving existing work into worker_threads.
