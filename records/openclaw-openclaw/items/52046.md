## Summary

Keep open. Current main still treats heartbeat as a serialized, awaited agent turn with main/session lane backpressure, and the requested non-blocking worker/lane model remains an architecture feature rather than an already-implemented cleanup close.

## What This Changes

Request non-blocking heartbeat execution so periodic heartbeat checks do not delay normal message processing.

## Best Possible Solution

Design a dedicated heartbeat execution lane or worker-style isolation contract that preserves session consistency, delivery ordering, wake retries, and tool side effects; review the narrower active-reply guard separately.

## Reproduction Assessment

No high-confidence live timing reproduction was established. Source inspection does confirm current main awaits heartbeat reply work after shared main/session lane checks, so a concrete repro would need a long heartbeat run concurrent with inbound user traffic.

## Solution Assessment

Unclear as stated. A background worker may be viable, but the maintainable fix needs a heartbeat isolation and lane contract before moving full heartbeat turns out of the serialized execution model.
