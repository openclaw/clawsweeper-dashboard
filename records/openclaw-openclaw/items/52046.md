## Summary

Keep open. Current main still runs heartbeat as a serialized, awaited agent turn coordinated with main/session lane backpressure; the related active-reply guard PR is useful but does not implement the requested non-blocking heartbeat execution model.

## What This Changes

The issue requests non-blocking heartbeat execution so scheduled heartbeat checks do not delay user message processing.

## Best Possible Solution

Design a dedicated heartbeat execution lane or worker-style isolation contract that preserves session consistency, delivery ordering, wake retries, and tool side effects, while reviewing the active-reply guard separately.

## Reproduction Assessment

No high-confidence live timing reproduction was established. Source inspection does confirm current main still awaits heartbeat reply work after shared main/session lane checks, so a concrete repro would need a long heartbeat run concurrent with inbound user traffic.

## Solution Assessment

Unclear as stated. A background worker may be viable, but the maintainable fix needs a heartbeat isolation and lane contract before moving full heartbeat turns out of the serialized execution model.
