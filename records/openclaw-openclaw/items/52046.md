## Summary

Keep open. Current main and the latest release still implement heartbeat as a serialized, awaited session turn with busy-skip backpressure and wake-layer retries, so the requested non-blocking worker or separate isolation lane remains unimplemented architecture work.

## What This Changes

The issue requests running heartbeat checks without blocking normal message processing, likely by moving heartbeat work to a background worker or separate execution lane.

## Best Possible Solution

Define a heartbeat-owned execution isolation contract, likely a dedicated lane or worker model, while preserving busy-skip safety, session ordering, delivery guarantees, cancellation, and narrower retry/backoff fixes tracked separately.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the relevant current behavior: heartbeat is awaited in the existing reply/session path and guarded by busy-lane backpressure rather than a detached worker.

## Solution Assessment

Unclear as proposed. A background worker may be viable, but the maintainable solution needs an explicit heartbeat isolation contract so session state, delivery ordering, wake retries, cancellation, and tool side effects remain well-defined.
