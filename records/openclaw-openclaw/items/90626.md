## Summary

Keep open: the PR targets a real cron delivery regression and now has copied live-output proof, but the patch still changes compatibility-sensitive fallbacks in the wrong places and would silently suppress explicit mode-less delivery targets while leaving Gateway/API defaults unchanged.

## What This Changes

The branch changes the cron CLI fallback and delivery-plan fallback from `announce` to `none` for isolated cron delivery paths.

## Best Possible Solution

Make one maintainer-approved cron delivery policy change at the canonical create/normalization/service boundaries, preserve explicit target semantics, coordinate with https://github.com/openclaw/openclaw/pull/91399 for no-channel run status, and update focused tests/docs.

The contributor's copied logs also show `delivery.mode="none"` producing `not-requested` delivery instead of the missing-channel error.

## Solution Assessment

No, not as submitted. The patch is a plausible mitigation for CLI-created channel-less jobs, but the best fix needs the default decided and implemented at the canonical create/default boundary while preserving explicit mode-less delivery targets.
