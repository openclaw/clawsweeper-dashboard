Full review comments:

- **[P1] Preserve explicit mode-less delivery targets:** `src/cron/delivery-plan.ts:85`
  - body: Changing this fallback to `none` makes an existing delivery object such as `{ channel: "telegram", to: "123" }` resolve as no delivery. Current coverage expects mode-less explicit targets to remain announce delivery, so this would silently drop cron output for stored or legacy jobs with a target but no mode.
  - confidence: 0.92
- **[P1] Keep explicit CLI chat targets from becoming no-delivery:** `src/cli/cron-cli/register.cron-add.ts:338`
  - body: With this default, `cron add --channel ... --to ...` and related target flags still build a delivery object, but the mode becomes `none` unless `--announce` is also passed. That suppresses a user-supplied target instead of treating it as an explicit announce request or rejecting the ambiguous command.
  - confidence: 0.88
- **[P2] Move the default policy into canonical create defaults:** `src/cli/cron-cli/register.cron-add.ts:338`
  - body: The CLI wrapper now sends `delivery.mode="none"`, but agent tool and Gateway RPC callers still pass through `normalizeCronJobCreate` and `resolveInitialCronDelivery`, which default detached agentTurn/command jobs to announce. Non-CLI cron.add paths can therefore keep hitting the same channel-required behavior.
  - confidence: 0.86
