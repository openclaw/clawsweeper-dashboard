## Summary

Keep open. The reported cron delivery problem is real, but this branch is not merge-ready because it changes local fallbacks while leaving canonical create defaults and legacy explicit-target semantics inconsistent, and it still lacks usable real behavior proof.

## What This Changes

The branch changes two cron delivery fallbacks from `announce` to `none` for CLI-created detached jobs and mode-less delivery plans.

## Best Possible Solution

Make a maintainer-approved cron delivery policy change in the canonical create/default layer, preserve explicit legacy delivery targets, update docs/tests, and require redacted real cron add/run proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.6` default isolated cron delivery to `announce`, and unresolved non-best-effort delivery can still fail a run; I did not execute a live cron flow in this read-only review.

## Solution Assessment

No. This is a plausible mitigation, but not the best fix because it changes two fallbacks rather than applying a maintainer-approved default through the canonical CLI, Gateway, service, docs, and test boundary while preserving explicit targets.
