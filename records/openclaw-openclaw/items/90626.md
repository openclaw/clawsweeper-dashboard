## Summary

Keep open. The cron failure is real and still present on current main/latest release, but this PR is not merge-ready because it changes a compatibility-sensitive delivery default, suppresses explicit mode-less delivery targets, and only updates the CLI path while shared service/API creation still defaults to announce.

## What This Changes

The branch changes two cron delivery fallback expressions so CLI-created isolated output jobs and mode-less delivery objects resolve to `delivery.mode: "none"` instead of `"announce"`.

## Best Possible Solution

Confirm the cron silent-default policy first, then implement it once in the shared create/default path while preserving explicit legacy delivery targets and tracking SQLite migration repair separately under the canonical issue.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.8` still default omitted isolated cron delivery to `announce`, mode-less delivery objects to `announce`, and unresolved requested delivery can still fail the run; the issue and contributor live output show the channel-required symptom.

## Solution Assessment

No. The PR is a plausible mitigation, but not the best fix as submitted because it suppresses explicit mode-less delivery targets and changes only the CLI fallback while shared service/API creation still defaults omitted detached delivery to `announce`.
