## Summary

Keep open. The PR targets a real cron delivery regression, but the patch is still not merge-ready: it changes two fallbacks while current main and v2026.6.6 still define `announce` as the canonical default, leaves Gateway/tool create paths inconsistent, can suppress explicit mode-less delivery targets, and still lacks branch-tied real behavior proof.

## What This Changes

The branch changes two cron delivery fallbacks so omitted or unresolved delivery resolves to `none` instead of `announce` in CLI add and delivery-plan resolution.

## Best Possible Solution

Make one maintainer-approved cron delivery policy change at the canonical create, normalization, delivery-plan, docs, and test boundaries; preserve explicit stored targets and leave the SQLite migration/data-loss work tracked by the linked issues.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 still default isolated cron delivery to `announce`, and required delivery can still fail the run when the target is unresolved; I did not run a live cron upgrade flow in this read-only review.

## Solution Assessment

No. This is a plausible mitigation, but not the best fix because it changes local fallbacks while leaving canonical create defaults, explicit legacy delivery targets, docs, and the linked migration/error-precedence work unresolved.
