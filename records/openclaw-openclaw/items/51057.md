## Summary

Keep open. Current main and the latest release still intentionally skip auth-profile cooldown marking for pure timeout/server-error failures, while tests explicitly preserve timeout rotation without cooldown; changing that needs maintainer failover-policy judgment rather than cleanup closure.

## What This Changes

The issue requests faster model fallback when repeated provider/auth-profile timeouts occur, instead of cycling timed-out auth profiles until a broad retry or turn-timeout budget is exhausted.

## Best Possible Solution

Define a bounded timeout-failure policy for the embedded runner and model fallback boundary, with threshold/cooldown semantics that distinguish pure provider timeouts from run-budget aborts and classified rate limits, then update docs and regression tests.

## Reproduction Assessment

Yes, at source level. Current main still skips cooldown marking for timeout failures and has E2E coverage asserting timeout rotation without cooldown; I did not replay a live OpenAI Codex outage.

## Solution Assessment

No, not as an automatic guard flip. Marking every timeout as an auth-profile failure is too broad; the maintainable solution needs a thresholded policy that avoids poisoning shared auth state and does not conflict with run-timeout fallback work.
