## Summary

Keep open. Current main has useful mitigations, but the timeout path still intentionally avoids auth-profile cooldown persistence and has no timeout-specific same-provider rotation cap comparable to overloaded/rate-limit failover, so the faster fallback policy requested here remains unresolved.

## What This Changes

The issue requests faster timeout-driven model fallback by treating repeated auth-profile timeouts as failed profiles or by adding a bounded timeout retry/failover budget.

## Best Possible Solution

Add a non-persistent timeout-specific same-provider rotation budget or circuit breaker that reaches configured model fallback after bounded LLM timeouts while preserving external abort, compaction, and tool-execution timeout exceptions.

## Reproduction Assessment

Yes for source-level reproduction: current main explicitly drops timeout reasons before auth-profile cooldown, tests assert timeout no-cooldown behavior, and no timeout-specific rotation cap exists. I did not run a live provider-outage replay in this read-only review.

## Solution Assessment

No, the current behavior is not the complete solution. The narrow maintainable fix is not to persist every timeout as auth failure, but to add an in-run timeout failover budget that reaches the fallback chain quickly after repeated same-provider stalls.
