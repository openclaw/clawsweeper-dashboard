## Summary

Keep open. Current main still has the requested failover-policy gap: timeout-shaped failures can rotate through the in-memory auth-profile candidate list and eventually fall back, but pure transport/model timeouts are still intentionally excluded from auth-profile cooldown persistence and there is no timeout-specific profile-rotation cap comparable to the overloaded/rate-limit caps.

## What This Changes

The issue requests that provider/model timeouts count toward auth-profile health or a bounded failover budget so OpenClaw reaches fallback models faster instead of spending long periods cycling primary-provider profiles.

## Best Possible Solution

Add a timeout-specific failover policy, preferably a model-scoped short circuit breaker or configurable same-provider timeout rotation budget, so repeated profile timeouts reach the fallback chain quickly while isolated transport stalls can still recover without permanent auth-profile damage. Cover multi-profile timeout fallback, single-timeout recovery, external aborts, and compaction/cron timeout boundaries.

## Reproduction Assessment

Yes for the code-level behavior: the current helper test proves timeout failover reasons resolve to null, and the runner guard returns before markAuthProfileFailure for timeout reasons. A live provider-outage replay was not run in this read-only review.

## Solution Assessment

No, current main is not the complete solution. The existing in-run profile rotation and idle watchdog are useful mitigations, but the narrow maintainable fix is a timeout-specific rotation budget or circuit breaker that reaches fallback without making every timeout a persisted auth failure.
