## Summary

Keep open. Current main still deliberately skips auth-profile cooldown marking for timeout failures, and the auth-profile rotation tests explicitly assert timeout rotation without cooldown, so this needs maintainer failover-policy judgment rather than cleanup closure.

## What This Changes

Request that repeated model timeouts cause auth profiles or the primary route to cool down sooner so model fallback starts faster.

## Best Possible Solution

Define a bounded timeout-failure policy in the embedded runner/model fallback path, with explicit threshold and cooldown semantics, updated docs, and tests that distinguish provider timeouts from run-budget timeouts.

## Reproduction Assessment

Yes, from source and tests rather than a live outage. Current main still skips timeout failure marking and has E2E coverage asserting timed-out profiles rotate without cooldown.

## Solution Assessment

No as an automatic fix. Marking every timeout as an auth-profile failure is too broad; maintainers should choose a thresholded transient-failure policy that does not poison shared auth state or conflict with run-timeout fallback work.
