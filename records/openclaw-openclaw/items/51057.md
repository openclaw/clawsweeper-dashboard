## Summary

Current main and the shipped v2026.5.18 release now solve the reported slow timeout-to-fallback symptom by escalating timeout failures to configured model fallbacks after profile rotation is exhausted, while intentionally keeping pure timeouts out of shared auth-profile cooldown state.

## What This Changes

The issue asks for timeout failures to count toward auth-profile failure or otherwise bound primary-provider retry time so fallback models are reached faster.

## Best Possible Solution

Keep the shipped v2026.5.18 timeout fallback behavior, and leave broader timeout/failure-class policy refinements to the narrower open policy issues.

## Reproduction Assessment

No current failing reproduction path remains from source inspection. Current code advances profile candidates one-way and falls back after timeout rotation is exhausted, with E2E coverage for timeout-to-model-fallback behavior.

## Solution Assessment

Yes. The shipped solution solves the observable slow fallback problem without treating every timeout as shared auth-profile health, which is safer for ambiguous transport and run-budget timeouts.
