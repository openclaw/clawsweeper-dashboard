## Summary

Keep open. Current main still deliberately keeps generic timeouts out of shared auth-profile cooldowns; the remaining ask is a timeout-specific failover policy decision, not a conservative cleanup close.

## What This Changes

Issue requests that provider/model timeouts count toward auth-profile failure/cooldown, or a bounded failover budget, so configured fallback models are reached faster.

## Best Possible Solution

Design a timeout-specific failover guard that reaches configured model fallbacks quickly without recording generic transport timeouts as shared auth health, then document and test that contract.

## Reproduction Assessment

Yes, at source level for the narrow behavior: current main still excludes `timeout` from auth-profile cooldown and has a regression test asserting no cooldown. No live provider-timeout replay was run for the full historical outage timing.

## Solution Assessment

No. Marking generic timeouts as shared auth-profile failures is not clearly the best fix because current policy intentionally avoids poisoning shared auth health; a scoped timeout failover budget or consecutive-timeout rule would be safer.
