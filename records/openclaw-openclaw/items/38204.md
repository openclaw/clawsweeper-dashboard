## Summary

Keep this open. Current main still deliberately excludes Grok 4.20 multi-agent model IDs from the bundled xAI provider path, while the issue discussion has shifted from an informational workaround to a request for native multi-agent endpoint support.

## What This Changes

The issue requests native support for Grok 4.2 multi-agent models in OpenClaw’s xAI provider instead of using chat-completions-compatible Grok variants.

## Best Possible Solution

If maintainers want bundled support, implement it inside `extensions/xai` as an explicit provider-owned multi-agent path with model aliases, payload semantics, unsupported-parameter handling, docs, focused regression tests, and live xAI proof when credentials are available. If maintainers decide this beta surface should live outside the bundled provider, document the plugin/ClawHub direction and then close with that product decision.

## Reproduction Assessment

Yes. Against current main, selecting a Grok 4.20 multi-agent ID follows source paths that explicitly reject `multi-agent` IDs, and existing tests assert the unsupported resolution result without needing a live xAI call.

## Solution Assessment

Not applicable because there is no proposed patch. A catalog-only model ID addition is not the best fix unless the upstream multi-agent API matches the existing OpenClaw xAI transport, which current docs and tests say it does not.
