## Summary

Keep this PR open, but it should not merge as-is: it replaces an opaque cast with a named assertion while the Discord setup resolver still returns `guildId`/`channelId` rows that the config setter does not read.

## What This Changes

The PR adds a local Discord guild/channel allowlist entry type in `extensions/discord/src/setup-core.ts` and casts setup allowlist results to that type before calling the Discord allowlist setter.

## Best Possible Solution

Keep the typing cleanup only after the Discord setup apply boundary explicitly converts resolver rows into the config writer's `{ guildKey, channelKey? }` shape and adds a focused regression test.

## Reproduction Assessment

Yes. A focused unit test can pass a resolved Discord channel row with `guildId` and `channelId` through the setup apply path and assert the selected guild/channel is written; current main and this PR feed that row to a setter that reads `guildKey`/`channelKey`.

## Solution Assessment

No. The PR is not the best fix because it replaces `never` with a more specific but still false assertion; the maintainable fix is to type the real resolver output and map it before writing config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Map resolver output before asserting setter entries:** `extensions/discord/src/setup-core.ts:291-295`
  - body: `resolveDiscordGroupAllowlist` passes through `resolveDiscordChannelAllowlist`, whose rows use `guildId` and `channelId`, but this assertion tells the setter they are `{ guildKey, channelKey? }` entries. Because `setDiscordGuildChannelAllowlist` reads `entry.guildKey`, resolved setup entries still fall through to `*` instead of the selected guild/channel. Map the resolver rows to the setter shape before applying them.
  - confidence: 0.86
