## Summary

Keep this PR open. The cleanup target is real, but the patch is not safe to merge as-is because it replaces an opaque cast with a named assertion that still does not match the Discord setup resolver’s actual `guildId`/`channelId` output shape.

## What This Changes

The PR extracts a local Discord guild/channel allowlist entry type in `extensions/discord/src/setup-core.ts` and replaces the setup apply path’s `resolved as never` cast with a cast to that explicit entry array type.

## Best Possible Solution

Type the Discord setup boundary around the real resolver output, then convert resolved `guildId`/`channelId` entries and unresolved fallback inputs into the config writer’s `{ guildKey, channelKey? }` shape before calling the setter.

## Reproduction Assessment

Yes. A focused unit test can have the Discord setup resolver return a resolved guild/channel entry with `guildId` and `channelId`; the current apply path and this PR’s asserted type then feed those keys to a setter that reads `guildKey` and `channelKey`.

## Solution Assessment

No. The PR replaces an opaque cast with a more specific but still false assertion; the safer fix is to map or type the resolver result explicitly before applying it to config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Map resolver output before asserting setter entries:** `extensions/discord/src/setup-core.ts:291-295`
  - body: `resolveDiscordGroupAllowlist` passes through `resolveDiscordChannelAllowlist`, whose results use `guildId` and `channelId`, but this assertion tells the apply path they are `{ guildKey, channelKey? }` entries. `setDiscordGuildChannelAllowlist` reads `entry.guildKey`, so resolved setup entries still fall through to the `*` guild instead of the selected guild/channel. Map the resolver output to the setter shape before applying it.
