## Summary

Keep this PR open but do not merge it as written. The underlying Telegram `message.send` buffer bug is valid on current main, but this branch modifies legacy files that have been removed from the active implementation and still decodes inline base64 before an encoded-size guard.

## What This Changes

The PR adds inline `buffer`/`filename`/`contentType` support for Telegram send actions by threading those fields through legacy action plumbing and constructing Telegram media from base64 data.

## Best Possible Solution

Rebuild the fix on current main by hydrating and forwarding inline send buffers through `src/infra/outbound` and `extensions/telegram/src`, with an encoded-size guard before base64 decoding in the Telegram transport.

## Reproduction Assessment

Yes. Source inspection is enough: current `message.send` advertises inline buffer fields, but current hydration, runner, Telegram runtime, and transport paths only handle URL/path media for `send`.

## Solution Assessment

No. The requested behavior is the right bug fix, but this PR targets removed legacy files and needs a pre-decode base64 size guard before it would be safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Rebase onto the active Telegram plugin files:** `src/telegram/send.ts:47-49`
  - body: This patch adds the feature under legacy `src/telegram/*`, `src/agents/tools/telegram-actions.ts`, and `src/channels/plugins/actions/telegram.ts`, but current main uses `extensions/telegram/src/*` plus `src/infra/outbound/*`. As written, it will not apply cleanly or fix the shipped send path.
  - confidence: 0.95
- **[P2] Reject oversized base64 before decoding:** `src/telegram/send.ts:763-764`
  - body: The inline path calls `Buffer.from(bufferBase64, "base64")` before enforcing `mediaMaxBytes`, so an oversized tool payload can allocate memory and spend CPU before rejection. Add an encoded-length guard before decoding, then keep the decoded-size check as a backstop.
  - confidence: 0.91

## Security Review
