## Summary

Keep open: this PR targets a real current-main Telegram `send` bug, but the branch is stale against the extension refactor and still needs fixes for pre-decode buffer sizing and over-broad send hydration before it is mergeable.

## What This Changes

The PR threads inline buffer, filename, and content type fields through Telegram message send handling, normalizes send-action attachment params, and adds unit coverage for inline media sending.

## Best Possible Solution

Port the fix to current `extensions/telegram` and outbound helper paths, hydrate `send` only when attachment hints exist, and enforce inline media limits before and after base64 decode with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main shows `send` advertises inline attachment fields, but send-action hydration skips `send` and the Telegram runtime/transport only forwards URL media, so buffers are dropped before delivery.

## Solution Assessment

No. The direction is right, but this stale PR is not the best merge shape until it is ported to the current plugin paths and adds an encoded-length guard plus attachment-hint-gated send hydration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject oversized inline buffers before decoding:** `src/telegram/send.ts:763-764`
  - body: The inline path decodes the caller-supplied base64 string with `Buffer.from(...)` before any encoded-length guard. Even though the PR now checks `decoded.byteLength` afterward, an over-limit payload can still allocate and consume CPU before `mediaMaxBytes` is enforced, so add a pre-decode base64 length cap and keep the decoded-byte check.
  - confidence: 0.9
- **[P3] Gate send hydration on attachment hints:** `src/infra/outbound/message-action-params.ts:358-361`
  - body: Adding `send` to this action gate sends every text-only send through `hydrateAttachmentPayload`. That helper can synthesize `filename: "attachment"` even without media, changing plugin-visible params for ordinary sends; only hydrate `send` when media/path/file/buffer hints are present.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR adds a caller-controlled inline base64 decode path without bounding encoded input before allocation.

Concerns:

- **[low] Inline base64 can allocate before the media limit:** `src/telegram/send.ts:763`
  - body: `Buffer.from(bufferBase64, "base64")` runs before the PR checks `mediaMaxBytes`, so an oversized string can consume memory and CPU even when it would later be rejected.
  - confidence: 0.9
