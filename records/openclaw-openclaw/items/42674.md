## Summary

Keep open. The reported Telegram inline-buffer send bug is still present on current main, but this PR is not mergeable as-is: it edits pre-pluginized Telegram files that no longer exist on main, and its inline base64 path still decodes before a pre-allocation size guard. The useful path is a narrow replacement or rebase onto the active Telegram plugin and shared outbound runner, coordinated with the related shared hydration PR.

## What This Changes

The PR threads inline `buffer`/`filename`/`contentType` support through the older Telegram send action path and adds tests for base64-backed Telegram media sends without a URL.

## Best Possible Solution

Preserve the requested behavior, but implement it against current main: hydrate inline `buffer`/`filename`/`contentType` for `send` through the shared outbound path, pass those fields through `extensions/telegram/src/channel-actions.ts` and `extensions/telegram/src/action-runtime.ts`, add inline-buffer support to `extensions/telegram/src/send.ts`, and enforce encoded-length limits before decoding base64.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current `message.send` advertises `buffer` and filename fields, but the active shared runner and Telegram plugin only route URL/path media into `sendMessageTelegram`, so a Telegram `send` with `buffer` is either rejected for missing text or sent as text while dropping the attachment.

## Solution Assessment

No, this PR is not the best way as-is. The product direction is right, but the patch targets obsolete files and needs a pre-decode size guard; a narrow replacement or substantial rebase onto the active pluginized paths is the maintainable route.

## Security Review

Status: needs_attention

Summary: The PR introduces inline Telegram base64 media decoding without a pre-decode size guard, which is a concrete low-severity resource exhaustion concern.

Concerns:

- **[low] Inline base64 decodes before enforcing size limits:** `src/telegram/send.ts:763`
  - body: `Buffer.from(bufferBase64, "base64")` allocates before the proposed `mediaMaxBytes` check. A large tool payload can consume memory/CPU even when the configured media limit should reject it; add an encoded-length guard before decoding.
