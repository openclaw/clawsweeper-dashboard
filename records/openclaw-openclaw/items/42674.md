## Summary

Keep open: current main still has the Telegram `send` buffer mismatch, and this PR is useful source work, but it needs to be ported to the current pluginized Telegram paths and tightened before merge.

## What This Changes

The branch threads inline buffer, filename, and content type params through Telegram `send` handling, hydrates data-URL buffers for send actions, and adds regression tests for buffer forwarding and media sending.

## Best Possible Solution

Port the fix to the current Telegram plugin and outbound-helper files, hydrate `send` only when attachment hints are present, forward inline media metadata through the Telegram action/runtime boundary, and enforce media size before and after base64 decode with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `send` advertises `buffer`/`filename`/MIME fields, but current main skips `send` hydration and the Telegram action/runtime path only forwards `mediaUrl`, so the reported fields are dropped before delivery.

## Solution Assessment

No. The intended direction is right, but this PR is stale against current main and its inline buffer size check occurs after allocation; the maintainable fix is a current-path port with attachment-hint-gated hydration and a pre-decode size guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject oversized inline buffers before decoding:** `src/telegram/send.ts:763-764`
  - body: The inline path decodes the full base64 string with `Buffer.from(...)` before enforcing `mediaMaxBytes`. A large caller-controlled payload can allocate far beyond the configured limit before this check runs, so add a base64-length guard before decoding and keep the decoded-byte check afterward.
  - confidence: 0.9
- **[P3] Hydrate send only when attachment hints exist:** `src/infra/outbound/message-action-params.ts:358-361`
  - body: Adding `send` to this gate makes every text-only send run through attachment hydration. With no media or buffer, `hydrateAttachmentPayload` can still synthesize `filename: "attachment"`, changing the plugin-visible params for ordinary sends; gate the `send` hydration path on actual attachment fields.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR adds inline base64 decoding on a tool/action path without bounding encoded input before allocation.

Concerns:

- **[low] Inline base64 can allocate before the media limit:** `src/telegram/send.ts:763`
  - body: The proposed Telegram inline-buffer path calls `Buffer.from(bufferBase64, "base64")` before checking `mediaMaxBytes`, so an oversized string can consume memory and CPU even when it would later be rejected.
  - confidence: 0.9
