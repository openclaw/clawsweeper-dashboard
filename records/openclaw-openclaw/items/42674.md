## Summary

Keep open: the underlying Telegram buffer-send bug is still valid on current main, but this PR is not mergeable as-is because it targets stale pre-extension Telegram paths, still decodes caller-controlled base64 before an encoded-size guard, hydrates all send actions unconditionally, and has no inspectable live Telegram proof.

## What This Changes

The PR threads inline buffer, filename, and content-type fields through Telegram send action handling so agents can send memory-backed attachments without a URL.

## Best Possible Solution

Port a consolidated fix to current extensions/telegram and shared outbound paths, use gated send-media hydration, enforce encoded and decoded size limits, coordinate with https://github.com/openclaw/openclaw/pull/63578, and add focused tests plus live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main advertises buffer/filename/contentType on message.send, but send hydration skips action "send" and Telegram only receives URL/path media.

## Solution Assessment

No. The direction is valid, but this branch needs to be ported to the current Telegram plugin runtime, bounded before base64 decode, and reconciled with the generic hydration PR before it is the maintainable fix.
