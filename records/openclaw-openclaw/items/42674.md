## Summary

Keep open. Current main still advertises inline attachment fields on shared `message.send`, but the shared hydration, runner, active Telegram action runtime, and Telegram transport path still route only URL/path media for `send`, so this PR is not obsolete. The branch is stale against current pluginized Telegram files and still needs a pre-decode base64 size guard before it can be merged or replaced.

## What This Changes

The PR threads inline `buffer`/`filename`/`contentType` support through the older Telegram message action path and adds tests for sending base64-backed media without a URL.

## Best Possible Solution

Preserve the intent, but refresh it against current main: add inline attachment support through the shared send hydration/runner and active Telegram plugin files, coordinate the shared hydration piece with #63578, and enforce base64 size limits before allocation.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a `message.send` call for Telegram with `buffer`, `filename`, and `contentType`/`mimeType`: current main advertises those fields but the code path drops them before Telegram transport.

## Solution Assessment

No, not as-is. The product direction is right, but this branch targets obsolete files and needs a pre-decode base64 guard; a narrow replacement or substantial rebase onto `extensions/telegram/src/*` is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject oversized base64 before decoding:** `src/telegram/send.ts:763-764`
  - body: The inline path decodes `bufferBase64` with `Buffer.from(..., "base64")` before checking `mediaMaxBytes`, so a caller can force allocation of a payload much larger than the configured media limit. Estimate or cap the base64 payload before decoding, then keep the decoded-size check as a backstop.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The PR introduces inline base64 Telegram media decoding without a pre-decode size guard, leaving a concrete low-severity resource exhaustion concern.

Concerns:

- **[low] Inline base64 decodes before enforcing size limits:** `src/telegram/send.ts:763`
  - body: `Buffer.from(bufferBase64, "base64")` allocates before `mediaMaxBytes` is checked. A large tool payload can consume memory/CPU even when the configured media limit should reject it; use the existing base64 decoded-size estimator before decoding.
  - confidence: 0.9
