## Summary

Keep open: the reported Telegram buffer-send bug is valid, but this PR is not mergeable as-is because it targets removed legacy paths and still needs a pre-decode size guard for inline base64 media.

## What This Changes

The PR threads inline `buffer`/`filename`/`contentType` support for Telegram send actions through legacy action and send files, with tests.

## Best Possible Solution

Implement the fix against current main by hydrating and forwarding inline send buffers through `src/infra/outbound` and `extensions/telegram/src`, with a pre-decode encoded-size guard in the Telegram transport.

## Reproduction Assessment

Yes. Source inspection is enough: current `message.send` advertises inline buffer fields, but `send` hydration skips them and the active Telegram runtime/transport only route URL-style media.

## Solution Assessment

No. The requested behavior is the right fix, but this PR targets removed legacy files and needs a safer inline base64 limit check before decoding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Rebase onto the active Telegram plugin files:** `src/telegram/send.ts:47-49`
  - body: This patch adds the feature under legacy `src/telegram/*`, `src/agents/tools/telegram-actions.ts`, and `src/channels/plugins/actions/telegram.ts`, but current main uses `extensions/telegram/src/*` plus `src/infra/outbound/*`.   - body: The inline path still calls `Buffer.from(bufferBase64, "base64")` before enforcing `mediaMaxBytes`, so an oversized tool payload can allocate and burn CPU before it is rejected. Add an encoded-length guard before decoding, then keep the decoded-size check as a backstop.
  - confidence: 0.91
