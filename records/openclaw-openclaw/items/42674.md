## Summary

Keep this PR open, but it is not merge-ready. Current main still has the Telegram `message.send` buffer/filename gap, while this branch targets stale pre-extension paths and still has a pre-decode base64 allocation issue plus an overly broad `send` hydration gate; the related open work at https://github.com/openclaw/openclaw/pull/63578 should be consolidated rather than merged independently.

## What This Changes

The PR threads inline `buffer`, `filename`, and `contentType` support through Telegram send-action hydration and transport, with unit coverage for inline media sending.

## Best Possible Solution

Port the Telegram inline-buffer path to current `extensions/telegram` code, gate send hydration only when attachment hints or an approved opt-in are present, enforce encoded and decoded media limits, reconcile the shared hydration work with https://github.com/openclaw/openclaw/pull/63578, and add focused tests plus live Telegram proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with `buffer`, `filename`, and `mimeType`/`contentType`; current main accepts those fields in the schema, skips `send` attachment hydration, and reaches Telegram with URL-only media transport.

## Solution Assessment

No. The direction is right, but this patch is not the best merge shape until it is ported to current extension paths, bounded before base64 decode, and coordinated with the open generic hydration PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject oversized inline buffers before decoding:** `src/telegram/send.ts:763-764`
  - body: The inline path decodes the caller-supplied base64 string with `Buffer.from(...)` before any encoded-length guard. Even though the PR checks `decoded.byteLength` afterward, an over-limit payload can still allocate and consume CPU before `mediaMaxBytes` is enforced; add a pre-decode base64 length cap and keep the decoded-byte check.
  - confidence: 0.9
- **[P3] Gate send hydration on attachment hints:** `src/infra/outbound/message-action-params.ts:358-361`
  - body: Adding `send` to this action gate sends every text-only send through `hydrateAttachmentPayload`. That helper can synthesize `filename: "attachment"` even without media, changing plugin-visible params for ordinary sends; hydrate `send` only when media/path/file/buffer hints are present or behind the approved plugin opt-in.
  - confidence: 0.78
