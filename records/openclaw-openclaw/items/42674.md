## Summary

Keep open: current main still has the Telegram `message.send` inline-attachment mismatch, and this PR is a plausible fix direction, but the branch is not merge-ready because it targets stale paths, overlaps the generic hydration work, lacks real Telegram proof, and still has concrete review findings.

## What This Changes

The PR threads inline `buffer`, `filename`, and `contentType`/`mimeType` through Telegram send-action handling so agents can send memory-backed attachments without a URL.

## Best Possible Solution

Port the fix to current `extensions/telegram` and shared outbound paths, gate `send` hydration only on attachment hints or an approved opt-in, enforce encoded and decoded size limits, reconcile with https://github.com/openclaw/openclaw/pull/63578, and add focused tests plus live Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `message.send` advertises inline attachment fields, while send hydration returns before `action: "send"` and Telegram runtime/transport only forward URL/path media.

## Solution Assessment

No. The direction is valid, but this patch is not the best merge shape until it is ported to current extension files, coordinated with the generic hydration work, and bounded before base64 decode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject oversized inline buffers before decoding:** `src/telegram/send.ts:763-764`
  - body: The inline path decodes caller-controlled base64 with `Buffer.from(...)` before any encoded-length guard. The decoded-byte check happens after allocation, so an over-limit payload can still consume memory and CPU before `mediaMaxBytes` is enforced; add a pre-decode base64 length cap and keep the decoded-byte check.
  - confidence: 0.9
- **[P3] Gate send hydration on attachment hints:** `src/infra/outbound/message-action-params.ts:358-361`
  - body: Adding `send` unconditionally sends text-only sends through `hydrateAttachmentPayload`, which can synthesize `filename: "attachment"` even without media. Hydrate `send` only when media/path/file/buffer hints are present or through an approved plugin opt-in.
  - confidence: 0.78
