## Summary

Keep open: current main and v2026.6.9 still have the permissive doctor migration path that can write `groupAllowFrom` into a schemaless external email channel account. The right cleanup path is to review and land https://github.com/openclaw/openclaw/pull/95540 or an equivalent narrow fix, not to close this issue before the fixing PR merges.

## What This Changes

The issue reports that the 2026.6.8 to 2026.6.9 upgrade writes an invalid `groupAllowFrom` field into an external email channel account, causing Gateway config validation to fail.

## Best Possible Solution

Land the schema-verified doctor guard in https://github.com/openclaw/openclaw/pull/95540, or an equivalent narrow fix, so core never writes plugin-owned `groupAllowFrom` fields unless metadata proves the target channel schema accepts them.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.6.9 permit missing-schema writes and then assign `groupAllowFrom` onto channel account records when `allowFrom` exists. I did not run the full macOS ClawMail upgrade path in this read-only review.

## Solution Assessment

Yes, the best fix is at the doctor migration write guard: core should fail closed when it cannot verify an external channel schema accepts `groupAllowFrom`. If maintainers want external plugins to receive this migration, the safer alternative is an explicit plugin schema or doctor opt-in contract.
