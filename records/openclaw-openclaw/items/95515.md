## Label Justifications

## Summary

Keep open: current main still permits the doctor migration to write `groupAllowFrom` when bundled channel schema metadata is missing, and the open linked pull request is the active fix candidate rather than a merged fix.

## What This Changes

The issue reports that upgrade-time `doctor --fix` injects `groupAllowFrom` into `channels.email.accounts.default`, corrupting the external email channel config and blocking gateway startup.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95540, or an equivalent doctor migration patch that only writes `groupAllowFrom` when core can prove the target schema accepts it.

## Reproduction Assessment

Yes from source inspection: current main still allows `groupAllowFrom` writes when generated bundled channel schema metadata is absent, and then writes the field onto account records. I did not run the macOS ClawMail upgrade path because this review is read-only, but the reporter supplied live v2026.6.10 evidence.

## Solution Assessment

Yes, the narrow schema-guard fix is the best owner-boundary solution: core doctor should not mutate external plugin account config unless bundled metadata proves the path is valid. A broader post-write validation check would be useful defense in depth, but it does not replace fixing the bad write decision.
