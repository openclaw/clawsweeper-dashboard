## Summary

Keep open: current `main` and v2026.6.9 still contain a source-reproducible path where the doctor fallback migration can write `groupAllowFrom` to an external `email` channel account with no bundled schema metadata. ## What This Changes

The issue reports that upgrading from 2026.6.8 to 2026.6.9 injects a Feishu/group-only `groupAllowFrom` field into `channels.email.accounts.default`, causing email plugin validation and Gateway startup to fail.

## Best Possible Solution

Land a narrow doctor migration fix that does not write `groupAllowFrom` unless channel schema or plugin metadata explicitly proves that field is valid, with regression coverage for an email-like external channel.

## Reproduction Assessment

Yes from source: current main and v2026.6.9 can treat an unknown external `email` channel schema as writable and copy `allowFrom` into account `groupAllowFrom`. I did not run the live upgrade flow because this review is read-only.

## Solution Assessment

Yes: the best fix is a narrow doctor/config migration guard, not broadening the email schema or adding a runtime compatibility shim for invalid config.

## Repair Work Prompt

Fix the doctor/upgrade path that copies `allowFrom` into `groupAllowFrom` for external or unknown channel schemas. The reported failure is that upgrading from 2026.6.8 to 2026.6.9 writes `channels.email.accounts.default.groupAllowFrom`, which the email plugin schema rejects and can prevent Gateway startup. Keep the fix in the doctor/config migration boundary; do not add a runtime compatibility shim and do not broaden the email schema. Use https://github.com/openclaw/openclaw/pull/95518 as context only because it is closed unmerged. Likely make `maybeRepairGroupAllowFromFallback` require explicit schema/capability proof before writing root or account `groupAllowFrom`, then add regression coverage for an email-like external channel with `accounts.default.allowFrom` that must remain unchanged. Preserve existing supported migrations for bundled channels whose schemas accept `groupAllowFrom`. Do not edit `CHANGELOG.md`.
