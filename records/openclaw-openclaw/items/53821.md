## Summary

Keep open. Current main still lacks the per-agent runtime sandbox-clamp resolution this PR targets, and the broader open #59149/#59170 work does not fully supersede this narrower sandbox-only slice. The latest PR head appears to address the prior override/guard wiring blockers, but it is still missing required user-facing changelog and docs updates before merge.

## What This Changes

The PR threads `requesterAgentIdOverride` through session list/history/send/status tooling and session-visibility helpers so `agents.list[].sandbox.sessionToolsVisibility` can override the default sandbox session-tools clamp, with focused tests and a plugin SDK API baseline hash update.

## Best Possible Solution

Land the narrow per-agent sandbox clamp after adding a changelog entry and documenting the new `agents.list[].sandbox.sessionToolsVisibility` precedence, while keeping the broader per-agent `tools.sessions` and `agentToAgent` policy design in #59149/#59170 separate.

## Reproduction Assessment

Yes. On current main, a sandboxed agent with `agents.list[].sandbox.sessionToolsVisibility: "all"` still resolves through the global/default-only sandbox clamp because the runtime resolver has no requester agent id; PR head statically changes that path. The remaining review findings are reproducible by inspecting PR head: no `CHANGELOG.md` entry and docs still describe only the global default clamp.

## Solution Assessment

Mostly yes. The runtime direction is the narrow maintainable fix for this sandbox-clamp field and preserves compatibility with optional SDK parameters, but the PR is not merge-complete until the user-facing docs and changelog are added.
