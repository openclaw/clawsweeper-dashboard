## Summary

Keep open. Current main still only reads the global sandbox session-tools clamp, and this PR is a useful narrow implementation slice, but the current head has merge-blocking requester-context and public SDK compatibility bugs in session access control.

## What This Changes

This PR threads requesterAgentIdOverride through session list/history/send/status tooling and session-visibility helpers so agents.list[].sandbox.sessionToolsVisibility can override the default sandbox session-tools clamp, with focused tests.

## Best Possible Solution

Land the narrow per-agent sandbox clamp only after every session tool consumes one override-aware requester context for visibility, spawned filtering, A2A checks, and store/session-id fallback, while preserving the existing public SDK call shape and documenting/changelogging the new behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is sandboxed true with agentSessionKey "global" and requesterAgentIdOverride "tony": sessions_list still uses the reconstructed key for spawnedBy, and session_status treats the bare override as main while sessionId fallback remains forced to spawned-only.

## Solution Assessment

No. The implementation direction is reasonable, but the current patch is not the best merge candidate because requester identity is still split across several derivations and the SDK signature change breaks existing callers; a single compatible context resolver is the safer fix.

Full review comments:

- **[P2] Use the override-scoped requester key for list filters:** `src/agents/tools/sessions-list-tool.ts:85`
  - body: resolveSandboxedSessionToolContext now returns an override-rebased effectiveRequesterKey, but sessions_list discards it and rebuilds the key from requesterInternalKey ?? alias. In sandboxed override runs that still use the spawned clamp, the gateway query uses global/main for spawnedBy instead of agent:<override>:<mainKey>, so same-agent spawned sessions are hidden.
  - confidence: 0.93
- **[P2] Normalize the status requester override before A2A checks:** `src/agents/tools/session-status-tool.ts:276-278`
  - body: requesterAgentIdOverride is a bare agent id, not a session key, so resolveAgentIdFromSessionKey("tony") resolves to the default main agent. That makes session_status treat agent:tony:* as cross-agent and reject same-agent status when A2A is disabled; normalize the override as an agent id before falling back to session-key parsing.
  - confidence: 0.91
- **[P2] Reuse the sandbox context for status session-id lookups:** `src/agents/tools/session-status-tool.ts:269-273`
  - body: The status tool computes an override-aware sandbox context, but later sessionId/current fallback paths still use opts?.sandboxed === true for restrictToSpawned. An override agent configured with sessionToolsVisibility: "all" can still be forced through spawned-only lookup and incorrectly fail; carry restrictToSpawned from the resolver into both fallback calls.
  - confidence: 0.89
- **[P2] Keep the SDK guard signature backwards-compatible:** `src/plugin-sdk/session-visibility.ts:279-284`
  - body: openclaw/plugin-sdk/session-visibility is an exported SDK subpath, and this change makes mainKey required for createSessionVisibilityGuard. Existing callers, including the bundled memory-core plugin, still call it without mainKey, so TypeScript plugin checks will break unless mainKey is optional/defaulted or a compatible overload is kept.
  - confidence: 0.9
