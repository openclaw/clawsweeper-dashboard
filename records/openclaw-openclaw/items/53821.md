## Summary

Keep open. Current main has the config schema for per-agent sandbox session visibility, but the runtime still uses the global default on the affected session-tool paths, so this PR is not obsolete. The branch is not merge-ready because it leaves two override paths incorrectly clamped and changes a public SDK function signature in a breaking way.

## What This Changes

The PR threads `requesterAgentIdOverride` through session list/history/send/status tooling and session-visibility helpers so `agents.list[].sandbox.sessionToolsVisibility` can override the default sandbox session-tools clamp, with tests for fallback, guard identity, and custom main keys.

## Best Possible Solution

Keep the narrow per-agent sandbox visibility direction, but update the branch so all session tools consume one override-aware context (`effectiveRequesterKey`, `restrictToSpawned`, and visibility) consistently, preserve the existing public SDK call shape or provide a compatible default, and then add focused docs/changelog plus changed-lane validation.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `sandboxed: true`, `agentSessionKey: "global"`, `requesterAgentIdOverride: "tony"`: `sessions_list` still sends `spawnedBy: "global"` under the spawned clamp, and `session_status` still uses `opts?.sandboxed === true` for sessionId fallback even when the override resolves to `all`.

## Solution Assessment

No. The implementation direction is reasonable, but the current patch is not the best merge candidate because it only partially applies the override-derived context and introduces a public SDK compatibility break; using the resolver's returned context consistently is safer and narrower.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the override-scoped requester key for spawned list filters:** `src/agents/tools/sessions-list-tool.ts:85`
  - body: `resolveSandboxedSessionToolContext` now returns an `effectiveRequesterKey` rebased to `agent:<override>:<mainKey>`, but `sessions_list` discards it and rebuilds the key from `requesterInternalKey ?? alias`. In sandboxed override runs that still use the `spawned` clamp, the gateway query is made with `spawnedBy: "global"` or the main alias instead of the override agent, so same-agent spawned sessions are hidden. Destructure and use the helper's `effectiveRequesterKey` for both `spawnedBy` and guard setup.
  - confidence: 0.93
- **[P2] Reuse the sandbox context for status session-id lookups:** `src/agents/tools/session-status-tool.ts:269-273`
  - body: The status tool asks the resolver for an override-aware sandbox context, but later sessionId/current fallback resolution still passes `restrictToSpawned: opts?.sandboxed === true`. With an override agent configured for `sessionToolsVisibility: "all"`, direct keys can be allowed while equivalent sessionId lookups remain spawned-only and incorrectly fail. Carry `restrictToSpawned` out of `resolveSandboxedSessionToolContext` and use it in both fallback calls.
  - confidence: 0.88
- **[P2] Preserve the public SDK guard signature:** `src/plugin-sdk/session-visibility.ts:279-284`
  - body: `openclaw/plugin-sdk/session-visibility` is an exported SDK subpath, so making `mainKey` a required property on `createSessionVisibilityGuard` forces existing TypeScript plugin callers to update for a patch-level feature. The runtime can default missing values to the current `main` behavior, so make `mainKey` optional or add a backward-compatible overload before merging.
  - confidence: 0.86
