## Summary

Keep open. Current main still resolves the sandbox session-tools clamp only from the default sandbox config, and this PR implements most of the per-agent override path, but the latest head still leaves a `session_status` derived-requester path clamped to defaults.

## What This Changes

The PR adds per-agent `agents.list[].sandbox.sessionToolsVisibility` precedence, wires requester identity through session tools, and documents/tests the sandbox clamp behavior.

## Best Possible Solution

Land the narrow sandbox-clamp override after `session_status` uses one resolved requester identity for helper context, effective visibility, guard checks, and session-id fallback; leave broader per-agent sessions/A2A policy to #59149/#59170.

## Reproduction Assessment

Yes. A static reproduction is clear: configure sandboxed `session_status` for `agent:tony:main` with `tools.sessions.visibility: "all"`, default sandbox visibility `"spawned"`, and `agents.list[{id:"tony"}].sandbox.sessionToolsVisibility: "all"`; without an explicit override option, PR head still computes effective visibility from the default clamp.

## Solution Assessment

No. The PR's narrow sandbox-clamp direction is maintainable, but the status path should pass the computed requester id into visibility resolution and base fallback restriction on the effective clamp instead of on whether an explicit override option was present.
