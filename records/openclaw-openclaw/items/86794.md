## Summary

Keep this PR open, but it is not merge-ready: the hot-path problem is source-plausible and the branch targets the right gateway surface, but the new cache can survive same-object session-store mutations and the external proof is still tests-only.

## What This Changes

The PR adds a process-local cache for single-row gateway session child-session candidate indexes and focused Vitest coverage for repeated row loads.

## Best Possible Solution

Keep the hot-path optimization direction, but tie cache validity to the session-store write lifecycle or a versioned snapshot and require redacted real gateway workload/profiler proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear current-main path: single-row loads call buildGatewaySessionRow without a row context, which falls through to a full store child-session index build; I did not run a live profiler workload.

## Solution Assessment

Is this the best way to solve the issue? Not yet: caching child candidates is a plausible narrow optimization, but the current object-identity cache needs write invalidation or versioning before it is safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Invalidate the single-row candidate cache after store writes:** `src/gateway/session-utils.ts:495-496`
  - body: The cache is reused whenever `loadSessionStore(..., { clone: false })` returns the same store object. Current store writes can take that mutable cached object, let callers mutate it, and save it back, so a child added or reparented after an earlier row read can keep using the old candidate map until a different object is cached. That can omit or mis-associate `childSessions`; invalidate/version this cache on writes and add a write-after-read regression.
  - confidence: 0.89
