## Summary

Keep open: the added Crabbox-style perf output satisfies the real-proof gap, but the PR still has a blocking session-state cache lifecycle issue because the new candidate cache is keyed only by store object identity while current writer paths can mutate and re-cache that same object.

## What This Changes

The PR adds a process-local single-row gateway session child-candidate cache and focused regression coverage for repeated row loads.

## Best Possible Solution

Tie the single-row cache to the session-store write lifecycle, or invalidate/version it on every writer-owned store save, and keep the focused perf proof plus a regression test for post-warmup parent/child mutations.

## Reproduction Assessment

Yes at source/proof level: current main loads a single row without list row context, and resolveChildSessionKeys rebuilds the store child index for that row. I did not run a local profiler in this read-only review.

## Solution Assessment

No as written: caching child candidates is a reasonable hot-path fix, but object-identity freshness is not enough for the current mutable session-store writer contract. The safer solution is write-lifecycle invalidation or versioning with a mutation regression test.

- **[P1] Invalidate the candidate cache after store writes:** `src/gateway/session-utils.ts:496-498`
  - body: `getSingleRowChildSessionCandidates` reuses the index whenever the cached `store` object is identical. Current session-store writers can take that same cached object, mutate it, and re-cache it with ownership, so a warmed single-row cache can survive `parentSessionKey` or `spawnedBy` changes. After that write, old parents can keep showing moved/deleted children or new parents can miss children until eviction; please invalidate/version this cache on store writes and add a regression test that mutates the store after the cache is populated.
