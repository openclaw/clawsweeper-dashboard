## Summary

Keep open: the performance fix is useful, but the final head is not merge-ready because it still adds an identity-keyed process-local child-session candidate cache without invalidation for session-store writer mutations, and the external real behavior proof remains focused Vitest/mock output rather than redacted gateway workload or profiler proof.

## What This Changes

The PR changes gateway single-session row loading to borrow the cached session store, cache store child-session candidate indexes by store path/object identity, and add focused Vitest coverage for repeated single-row loads.

## Best Possible Solution

Keep the optimization, but tie the candidate cache to the session-store cache/write lifecycle or a content/version token, add write-after-cache coverage, and include redacted after-fix gateway/profiler proof before merge.

## Reproduction Assessment

Yes from source for the patch defect: the PR caches candidates by store object identity, while current writer paths can mutate and re-cache the same borrowed object. I did not run a live gateway/profiler replay in this read-only review.

## Solution Assessment

No: the performance direction is reasonable, but the cache needs a lifecycle-owned invalidation/version boundary and a write-after-cache test before this is the narrow maintainable fix.

- **[P1] Invalidate the single-row child cache after store writes:** `src/gateway/session-utils.ts:496-497`
  - body: The cache is reused whenever the store path and store object identity match, but `loadGatewaySessionRow` now borrows the session-store object with `clone:false`. Invalidate/version this cache on session-store writes and cover a write-after-cache case.
