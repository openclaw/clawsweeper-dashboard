## Summary

Keep open. This is a focused gateway performance fix candidate, but it should not merge as-is because the new cache can reuse a rowContext whose subagent-run index is stale within the one-second cache bucket, and the external-PR proof is only focused tests/autoreview output.

## What This Changes

The PR adds a process-local single-row gateway session row context cache, extends loadSessionEntry with a cloneStore option, and adds a focused Vitest regression test.

## Best Possible Solution

Preserve the performance fix, but key or invalidate the cache on both store and subagent registry freshness, or separate store-derived child-index reuse from live subagent state, then add real after-fix gateway/profiler proof.

## Reproduction Assessment

Yes, from source inspection rather than a local run: current main's single-row loader builds rows without a shared rowContext, so child-session resolution can rebuild the store child index per call. The PR body also cites redacted profiler evidence for the hot path.

## Solution Assessment

No, not as written. The narrow performance direction is right, but the cache needs a freshness boundary for live subagent registry state before it is the safest maintainable fix.
