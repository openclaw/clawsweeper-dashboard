## Summary

Keep open: the latest head still has the explicit-250 ms Node runtime policy bug from the prior review, the contributor’s real-network claim has no inspectable after-fix artifact, and current main has not made this PR’s process-wide Node timeout change.

## What This Changes

The PR adds a guarded process-wide `net.setDefaultAutoSelectFamilyAttemptTimeout(300)` call when Node reports the default 250 ms family autoselection timeout, plus unit coverage for default and non-default values.

## Best Possible Solution

Preserve explicit Node CLI and `NODE_OPTIONS` timeout choices while solving any remaining broken-IPv6 fallback gap in the shared network policy or in affected plugin-owned transport paths; keep the Telegram-specific regression on its focused PR.

## Reproduction Assessment

Yes for the PR-introduced policy regression: Node reports `250` for both the default and an explicit `--network-family-autoselection-attempt-timeout=250`, matching the PR’s value-only guard. No live broken-IPv6 provider outage was reproduced during this read-only review.

## Solution Assessment

No. The direction addresses a plausible fallback gap, but this patch is not the narrowest maintainable fix because it silently overwrites an explicit operator timeout policy that Node exposes as a runtime option.
