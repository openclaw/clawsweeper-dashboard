## Summary

Keep open, but do not merge as-is: the latest branch still overwrites an explicit operator timeout value when it matches Node's 250 ms default, and the PR body has no inspectable after-fix real behavior proof. Current main does not fully make this PR obsolete, so this is a review-and-revise case rather than a cleanup close.

## What This Changes

The branch adds a guarded process-wide bump of Node's default autoSelectFamily attempt timeout from 250 ms to OpenClaw's 300 ms policy during `ensureGlobalUndiciStreamTimeouts()`, plus unit coverage for default and non-default timeout values.

## Best Possible Solution

Implement any needed fallback in current main's shared network policy while preserving explicit Node CLI and `NODE_OPTIONS` timeout choices; keep the Telegram-specific regression tracked separately at https://github.com/openclaw/openclaw/issues/76852.

## Reproduction Assessment

Yes for the PR-introduced policy regression: Node returns `250` for both the default and explicit CLI or `NODE_OPTIONS` timeout value, matching the PR's value-only guard. No live broken-IPv6 provider outage was reproduced in this read-only review.

## Solution Assessment

No. The direction addresses a plausible fallback gap, but the current patch is not the narrowest maintainable fix because it can overwrite explicit operator timeout policy; a safer version should detect explicit runtime options or make the process-wide mutation opt-in.
