## Summary

Keep open. The PR addresses a real networking pain point, but the latest patch still has a concrete blocker: it treats every observed 250 ms auto-family timeout as Node's untouched default and can overwrite an explicit operator override; it also lacks the required user-facing changelog entry.

## What This Changes

The PR adds a process-wide Node `net` auto-family attempt-timeout bootstrap in the Undici global dispatcher path and extends the dispatcher tests for default and explicit timeout behavior.

## Best Possible Solution

Revise the PR so process-wide timeout mutation preserves explicit Node runtime policy, add focused regression tests for explicit CLI and `NODE_OPTIONS` overrides, and add the required changelog entry while keeping the networking change scoped to the dispatcher bootstrap.

## Reproduction Assessment

Yes for the PR-introduced regression: run Node with `--network-family-autoselection-attempt-timeout=250` or matching `NODE_OPTIONS` and the getter returns the same 250 value the new guard treats as default. The original IPv6-broken network failure was not live-reproduced in this read-only pass.

## Solution Assessment

No. The proposed direction is close, but value-only detection is not the narrowest maintainable fix because it can clobber an explicit operator timeout; explicit flag/`NODE_OPTIONS` detection or an opt-in process-wide default is safer.
