## Summary

Keep open: the network fix is plausible, but the latest PR still overwrites an explicit 250 ms Node family-selection timeout and lacks inspectable real behavior proof. Current main has adjacent guarded-fetch IPv4/IPv6 fallback work, but it does not implement this PR's process-wide timeout mutation, so this is not obsolete.

## What This Changes

The PR adds a process-wide Node `net` autoSelectFamily attempt-timeout bump from 250 ms to 300 ms before applying Undici global dispatcher timeouts, plus dispatcher tests for that behavior.

## Best Possible Solution

Port the fallback onto current main's guarded-fetch and lazy-Undici structure while preserving explicit Node CLI and `NODE_OPTIONS` timeout policy; keep Telegram-specific getMe fallback work in its separate Telegram issue/PR path.

## Reproduction Assessment

Yes for the PR-introduced regression: Node exposes an explicit 250 ms runtime override through the same getter value that the PR treats as default. The original broken-IPv6 provider outage was not live-reproduced here.

## Solution Assessment

No. The proposed direction is close, but value-only detection is not the narrowest maintainable fix because it can clobber an explicit operator timeout; detect explicit runtime flags or make the process-wide mutation opt-in.
