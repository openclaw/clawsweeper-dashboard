## Summary

Keep open but do not merge as-is: the PR targets a plausible IPv6-broken network bug, but the current patch still clobbers explicit 250 ms Node timeout policy and lacks inspectable real-behavior proof.

## What This Changes

The PR adds a process-wide Node `net` autoSelectFamily attempt-timeout adjustment in `undici-global-dispatcher.ts` plus tests that expect the default 250 ms value to be raised to OpenClaw's 300 ms dispatcher policy.

## Best Possible Solution

Port any needed fallback onto current main's guarded-fetch and lazy-Undici structure while preserving explicit Node CLI and `NODE_OPTIONS` family-timeout policy; keep Telegram-specific fallback work in its own Telegram tracker.

## Reproduction Assessment

Yes for the PR-introduced policy regression: Node v24.14.1 reports 250 ms for both the default and explicit CLI/`NODE_OPTIONS` `--network-family-autoselection-attempt-timeout=250`, matching the PR's value-only guard. No live broken-IPv6 provider outage was reproduced.

## Solution Assessment

No. The proposed direction is close, but the current patch is not the narrowest maintainable fix because it can overwrite explicit operator timeout policy and is stale against current main's network transport structure.
