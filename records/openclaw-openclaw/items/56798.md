## Summary

Keep open. The latest PR branch still has a concrete correctness blocker: it treats any observed 250 ms Node family-selection attempt timeout as an untouched default and mutates it to 300 ms, even when an operator explicitly selected 250 ms. The related Telegram IPv4 fallback issue is adjacent context, not a superseding fix for this generic process-wide networking change.

## What This Changes

The PR adds a guarded net-level autoSelectFamily attempt-timeout bootstrap in `src/infra/net/undici-global-dispatcher.ts` and tests that a 250 ms default is raised to 300 ms while other timeout values are preserved.

## Best Possible Solution

Revise the patch so process-wide timeout mutation preserves explicit Node runtime policy, including the 250 ms case, or make that process-wide change opt-in. Keep the existing OpenClaw-owned dispatcher behavior, add focused regression tests for CLI and `NODE_OPTIONS` overrides, and add the required changelog entry before merge.

## Reproduction Assessment

Yes for the PR-introduced regression: run Node with `--network-family-autoselection-attempt-timeout=250` or matching `NODE_OPTIONS` and the getter returns the same 250 value the new guard treats as a default. The original IPv6-broken network outage was not live-reproduced in this read-only review.

## Solution Assessment

No. The patch is close, but value-only detection is not the narrowest maintainable solution because it overrides an explicit operator timeout; explicit flag/`NODE_OPTIONS` detection or an opt-in process-wide default is safer.
