## Summary

Keep this PR open, but it is not merge-ready: the patch conflicts with the plugin metadata freshness boundary and lacks real gateway proof for the performance claim. The linked latency issue remains valid-looking, and related open PRs are exploring safer snapshot-owned caching approaches.

## What This Changes

The PR adds 30-second in-process caches and environment controls around plugin discovery and manifest registry loading, with tests adjusted for the new cache behavior.

## Best Possible Solution

Preserve fresh default discovery and registry calls, and optimize the gateway/status hot path through a caller-owned PluginMetadataSnapshot, PluginLookUpTable, manifest registry, or narrowly fingerprinted snapshot memo.

## Reproduction Assessment

No high-confidence live reproduction of the gateway latency was run in this review. The stale-cache regression introduced by this PR is source-reproducible because the diff caches discovery and manifest registry results for 30 seconds while current main has policy and tests requiring next-call freshness.

## Solution Assessment

No. Process-wide TTL caches are not the best way to solve this because plugin metadata freshness is an explicit boundary; the safer alternative is snapshot-owned or flow-owned reuse in the gateway/status path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop the process-wide discovery cache:** `src/plugins/discovery.ts:78-82`
  - body: Caching discoverOpenClawPlugins for 30 seconds hides plugin add/remove events and path-safety changes until the TTL expires. Current main requires discovery metadata to stay fresh unless the current flow explicitly owns a snapshot or lookup table, so the gateway optimization should reuse a flow-owned metadata object instead of changing the default discovery contract.
  - confidence: 0.9
- **[P2] Keep manifest registry loads fresh by default:** `src/plugins/manifest-registry.ts:183-187`
  - body: A 30-second cache around loadPluginManifestRegistry returns stale manifest/package metadata after openclaw.plugin.json or package changes and delays registry diagnostics. Current tests expect the next load to reflect manifest edits, so this should be solved by passing a registry or snapshot through the gateway path rather than adding a module-level TTL.
  - confidence: 0.88
