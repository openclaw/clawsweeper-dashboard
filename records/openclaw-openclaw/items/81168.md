## Summary

Keep open for review, but the current patch should not merge as-is because it reintroduces persistent plugin metadata caches that conflict with the current plugin boundary and can leave discovery/manifest safety checks stale.

## What This Changes

The PR adds 30-second in-process cache maps and environment controls around plugin discovery and manifest registry loading, plus tests for cache hits across short gateway poll intervals.

## Best Possible Solution

Fix the gateway polling cost by carrying a gateway-owned PluginMetadataSnapshot, PluginLookUpTable, or explicit manifest registry through auth/provider resolution while preserving fresh default discovery and registry calls.

## Reproduction Assessment

No high-confidence live reproduction of the linked gateway latency on current main was established. The introduced stale-cache behavior is source-reproducible from the diff because discovery and manifest registry results are cached for 30 seconds while current tests require next-call freshness.

## Solution Assessment

No. A module-level 30-second metadata cache is not the best fix because current plugin policy requires fresh metadata unless a caller owns an explicit snapshot or registry, and the gateway already has snapshot handoff points for flow-owned reuse.
