## Summary

Keep open, but this draft PR is not mergeable as submitted: the TTL caches conflict with the plugin metadata freshness boundary, and the external PR still lacks after-fix real gateway proof while the linked reporter issue remains open.

## What This Changes

The PR adds 30-second in-process caches and environment controls for plugin discovery and manifest registry loading, plus tests expecting cache hits across short poll intervals.

## Best Possible Solution

Carry the existing compatible PluginMetadataSnapshot, PluginLookUpTable, or manifest registry through gateway/auth/provider hot paths while keeping discovery and manifest registry calls fresh by default.

## Reproduction Assessment

No high-confidence live after-fix reproduction is present for this PR. The PR regression is source-reproducible: removing or editing plugin files within the 30-second TTL would return stale discovery or registry data where current main tests require next-call freshness.

## Solution Assessment

No. A module-level TTL cache is not the best fix because plugin metadata freshness is an explicit boundary; the safer approach is gateway-owned snapshot or lookup-table reuse with compatibility checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop the persistent discovery cache:** `src/plugins/discovery.ts:78-82`
  - body: This caches discovery results for 30 seconds, so plugin add/remove events and path-safety changes can be hidden until the TTL expires. Current plugin guidance requires fresh discovery metadata unless a caller owns an explicit snapshot or lookup table for the current flow.
  - confidence: 0.9

  - confidence: 0.88
