## Summary

Keep open, but the current PR is not mergeable as written: it extends a persistent plugin metadata cache that current main removed, conflicts with the scoped plugin cache boundary, and still lacks real gateway proof for the linked performance problem.

## What This Changes

The branch raises old plugin discovery and manifest-registry cache defaults from 1 second to 30 seconds and adds tests expecting removed plugin metadata to remain cached after 1.5 seconds.

## Best Possible Solution

Fix the linked gateway polling cost on current main by carrying an explicit flow-owned `PluginMetadataSnapshot`, `PluginLookUpTable`, or manifest registry through auth/provider checks, or by narrowing auth/status no-op coalescing, without reintroducing persistent discovery or manifest-registry TTLs.

## Reproduction Assessment

No high-confidence live current-main gateway reproduction was run. Source and PR diff inspection do reproduce the review problem: this branch edits cache APIs no longer present on current main and adds tests for stale metadata retention.

## Solution Assessment

No. Extending removed persistent metadata TTLs is not the best fix because current main intentionally moved this surface toward fresh metadata plus caller-owned snapshots or lookup tables.
