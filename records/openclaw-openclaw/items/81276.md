## Summary

Keep open: the code change is focused, but it creates a new user-facing `HEARTBEAT.md` pause contract that overlaps with the documented `0m` and CLI disable controls, and a maintainer has already asked for that product boundary to be resolved before merge. The updated proof is still mocked Vitest output rather than a real scheduler/provider run showing skipped dispatch after the fix.

## What This Changes

The PR adds a case-insensitive exact-content `HEARTBEAT.md` `PAUSED` sentinel that skips heartbeat preflight with reason `heartbeat-paused`, plus regression coverage for paused, actionable, and empty heartbeat files.

## Best Possible Solution

Choose one supported heartbeat pause contract, then land the code with matching docs, diagnostics, regression coverage, and real runtime proof that dispatch is suppressed for that contract.

## Reproduction Assessment

Yes for the observable current behavior: source inspection shows a literal `PAUSED` file is non-empty content and falls through to the normal interval heartbeat path. No live scheduler/provider credit-burn reproduction was established, and the expected `PAUSED` contract is not documented today.

## Solution Assessment

Unclear as-is: the preflight location is plausible, but recognizing `PAUSED` adds a new user-facing pause contract that overlaps with the existing `0m` and CLI disable controls. The safer path is maintainer agreement on the contract plus docs and real runtime proof before merge.
