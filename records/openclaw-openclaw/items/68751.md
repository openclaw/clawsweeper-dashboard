## Summary

Keep open. Current main still writes prior-session transcript-shaped content directly into durable session-memory files on both `/new` and `/reset`; startup-context wording mitigates only one injection path and does not fix the raw persisted artifact or exact memory reads.

## What This Changes

The issue asks for bundled `session-memory` to stop replaying prior-session turns as actionable current context after `/reset`, using durable untrusted framing, neutral summaries, or reset-specific non-persistence.

## Best Possible Solution

Define one end-to-end trust boundary for bundled session memory: `/reset` should not carry prior-session instructions forward by default, and any `/new` continuity should be persisted and retrieved only as untrusted historical reference or neutral summaries with regression coverage for write, startup, and memory-tool paths.

## Reproduction Assessment

Yes, at source level. A session JSONL with ordinary user/assistant messages followed by a `command:reset` event still reaches code and tests that write raw `user:` and `assistant:` transcript lines into `memory/YYYY-MM-DD-*.md`; live autonomous re-execution depends on the deployed model and tool environment.

## Solution Assessment

Unclear for the final policy. Wrapping historical content is a narrow mitigation, but the durable fix should also decide `/reset` semantics and cover both startup injection and exact memory-tool retrieval so the same transcript-shaped data cannot be treated as current instructions through another path.
