## Label Justifications

- `P1`: The report includes production Discord and Telegram cases where reset-era memory replay repeated write-capable external actions.
- `impact:security`: The issue is about historical instructions crossing a trust boundary into future tool-capable agent context.
- `impact:session-state`: The affected surface is persisted session memory and startup/reset context state drifting across sessions.

## Summary

Keep open. Current main and the latest release still preserve ordinary prior user/assistant turns as transcript-shaped memory on `/reset`, while the existing untrusted startup prelude and related sanitizer PR only partially mitigate separate read-path or malformed-token cases.

## What This Changes

The issue asks OpenClaw to prevent session-memory from replaying raw prior-session turns as actionable context after `/reset`.

## Best Possible Solution

Define the session-memory trust boundary end to end: `/reset` should break continuity by default, and any `/new` continuity should be stored and injected only as untrusted historical reference or a neutral summary with regression coverage.

## Reproduction Assessment

Yes, at source level. A session JSONL with ordinary user and assistant messages is formatted as role-prefixed text and written under `## Conversation Summary` when the hook handles `action: "reset"`; live autonomous replay depends on the model/tool environment and was not executed in this read-only review.

## Solution Assessment

Unclear for the final policy. Wrapping or summarizing historical content is directionally right, but the open sanitizer PR is not sufficient by itself because it does not address ordinary prior commands or `/reset` continuity semantics.
