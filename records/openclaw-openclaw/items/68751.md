## Summary

Keep open. Current main still persists transcript-shaped prior user/assistant turns into bundled session-memory files on both `/new` and `/reset`; startup-context framing helps one first-turn path, but the durable Markdown and memory-tool retrieval surfaces remain unresolved.

## What This Changes

The issue asks for bundled `session-memory` to prevent prior-session raw turns from replaying as fresh instructions, likely by changing `/reset` persistence and adding untrusted historical framing or neutral summaries.

## Best Possible Solution

Define the bundled session-memory trust boundary end to end: `/reset` should break continuity by default, and any `/new` continuity should be stored and retrieved only as clearly untrusted historical reference or as a neutral summary, with tests covering the persisted file, startup prelude, and memory-tool retrieval paths.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: create a session JSONL with ordinary user/assistant messages, trigger the hook with `action: "reset"`, and current code/tests show raw transcript-shaped lines written into `memory/YYYY-MM-DD-*.md`. Live autonomous re-execution depends on model and tool environment, but the unsafe persistence and recall surface are directly reproducible.

## Solution Assessment

Yes, the requested direction is the right fix boundary, but it should combine reset semantics with persisted-content framing. Token-only or metadata-only sanitizers are useful subcase fixes, but they do not neutralize ordinary prior user commands saved as historical `user:` turns.
