## Label Justifications

- `P1`: The issue reports production agents repeating stopped external-API work, and the source path that can persist prior turns on `/reset` is still present.
- `impact:security`: The issue is about a prompt-context trust boundary where historical instructions can be persisted and later treated as actionable context.
- `impact:session-state`: The reported failure carries prior-session transcript state into a reset/new session and can make the agent resume old work.

## Summary

Keep open: current main and v2026.5.18 still persist role-prefixed prior-session transcript content on `/reset`; the startup untrusted wrapper is only a read-path mitigation, and https://github.com/openclaw/openclaw/pull/69961 targets malformed token poisoning rather than ordinary prior-command replay.

## What This Changes

The issue requests preventing `session-memory` from carrying prior-session transcript-shaped turns forward as executable context on `/reset`, preferably through untrusted framing, neutral summarization, or skipping reset persistence.

## Best Possible Solution

Define the session-memory trust boundary end to end: `/reset` should break continuity by default, and any `/new` continuity should be persisted and later injected only as untrusted historical reference or a neutral summary with regression coverage for write and startup/read paths.

## Reproduction Assessment

Yes, at source level. A JSONL session with ordinary user and assistant messages is formatted as role-prefixed text and written under `## Conversation Summary` when the hook handles `action: "reset"`; I did not run a live model/tool reproduction in this read-only pass.

## Solution Assessment

Unclear for the final policy. Wrapping or neutral summarization is directionally right, but maintainers still need to decide the `/reset` continuity semantics and how existing memory artifacts should be handled.
