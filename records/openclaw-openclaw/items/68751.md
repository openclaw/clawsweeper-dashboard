## Label Justifications

- `P1`: The report describes a real high-impact agent/session workflow where reset can carry prior instructions back into a write-capable agent context.
- `impact:session-state`: The affected surface is persisted session memory and reset/new startup context, which can make session state drift across supposedly fresh sessions.
- `impact:security`: Raw prior user commands from write-capable sessions can be preserved as prompt material without a durable trust boundary.

## Summary

Keep open: current main and the latest release still persist transcript-shaped prior user/assistant turns from `session-memory` on `/reset`; the newer startup-context wrapper is a partial read-path mitigation, not an end-to-end write-path or reset-semantics fix.

## What This Changes

Prevent bundled `session-memory` from saving raw prior-session turns that can be reintroduced after `/reset` as executable-looking context.

## Best Possible Solution

Make `session-memory` preserve continuity only through explicitly historical, untrusted-framed or neutral summarized content, and either stop capture on `/reset` or document and test why reset should still carry memory forward.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection reproduces the raw `/reset` persistence path on current main, but I did not live-reproduce autonomous re-execution after the newer startup-context wrapper.

## Solution Assessment

Is this the best way to solve the issue? No final fix is present yet; the current read-path wrapper is a useful mitigation, but the best fix needs a maintainer decision on `/reset` continuity plus write-path trust framing or neutral summarization.
