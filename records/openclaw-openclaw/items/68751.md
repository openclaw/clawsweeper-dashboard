## Summary

Keep open. Current main and the latest release still write prior-session transcript-shaped `user:`/`assistant:` content directly into session-memory Markdown on both `/new` and `/reset`; the newer startup-context wrapper only protects one later load path, and the related sanitizer PR at https://github.com/openclaw/openclaw/pull/69961 covers malformed-token poisoning rather than the broader raw prior-command replay policy.

## What This Changes

The issue asks OpenClaw to prevent session-memory from replaying prior-session transcript turns as executable current context after `/reset`, by adding durable untrusted framing, neutral summaries, or changed reset persistence semantics.

## Best Possible Solution

Define the bundled session-memory trust boundary end to end: `/reset` should break continuity by default, and `/new` continuity should be stored and retrieved only as clearly untrusted historical reference or a neutral summary, with regression coverage for persistence, startup context, and memory-tool retrieval.

## Reproduction Assessment

Yes, at source level. A session JSONL with ordinary user and assistant messages will be formatted as role-prefixed text and written raw under `## Conversation Summary` when the hook handles `action: "reset"`; live autonomous re-execution still depends on model and tool environment.

## Solution Assessment

Unclear for the exact final policy. Wrapping or summarizing historical content is the right direction, but the open sanitizer PR at https://github.com/openclaw/openclaw/pull/69961 is not sufficient by itself because it does not address ordinary prior commands or `/reset` continuity semantics.
