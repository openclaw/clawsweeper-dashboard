## Label Justifications

- `P1`: The issue reports production agents repeating stopped external-API work, and the source path that persists prior turns on `/reset` is still present in current main and the latest release.
- `impact:security`: The issue concerns a prompt-context trust boundary where historical instructions can be persisted and later treated as actionable context.
- `impact:session-state`: The reported failure carries prior-session transcript state into a reset/new session and can make the agent resume old work.

## Summary

Keep open: current main and v2026.5.18 still persist transcript-shaped prior user/assistant turns on `/reset`, and the existing startup untrusted wrapper is only a partial read-path mitigation, not a durable write-path trust boundary.

## What This Changes

The issue asks to stop `session-memory` from persisting raw prior-session transcript turns on `/reset` so they cannot later re-enter context as actionable instructions.

## Best Possible Solution

Define the session-memory trust boundary end to end: `/reset` should break continuity by default, while any `/new` continuity should be persisted and later injected only as untrusted historical reference or a neutral summary, with regression coverage for write and startup/read paths.

## Reproduction Assessment

Yes, at source level. A JSONL session with ordinary user and assistant messages is formatted as role-prefixed text and written under `## Conversation Summary` when the hook handles `action: "reset"`; I did not run a live model/tool reproduction in this read-only pass.

## Solution Assessment

Unclear for the final policy. Wrapping or neutral summarization is directionally right, but maintainers still need to decide `/reset` continuity semantics and what to do with existing memory artifacts.
