## Label Justifications

- `P1`: The issue describes a broken agent/session-memory workflow that can progressively poison future runs and affect real users of `/reset` or `/new`.
- `impact:session-state`: The affected artifact is persisted session memory that can corrupt future session context and agent behavior.
- `impact:security`: The reported failure mode is prompt-injection-like replay of prior raw turns and tool-call scaffolding as current input.

## Summary

Keep open. Current main mitigates one startup-injection path by framing daily/session memory as untrusted quoted notes, but the reported reset-memory poisoning is not fully fixed: the bundled session-memory hook still writes raw transcript-shaped content on `/new` and `/reset`, and memory tools can still return exact raw memory excerpts outside the startup prelude. Related sanitizer work did not merge, so a product/security decision remains on whether persistence itself must sanitize, elide, or otherwise neutralize raw prior-session turns.

## What This Changes

The issue asks OpenClaw to prevent session-memory files saved during `/reset` or `/new` from replaying prior raw `user:`/`assistant:` transcript turns as active input.

## Best Possible Solution

Keep the issue open until maintainers choose and implement a canonical memory safety boundary for saved session summaries, with tests covering both startup injection and memory-tool recall.

## Reproduction Assessment

Yes, source inspection provides a high-confidence path for the remaining risk: `/new` and `/reset` still persist role-prefixed session content, and memory tools can still retrieve exact raw excerpts, although I did not run a live poisoned-memory scenario in this read-only review.

## Solution Assessment

No, current main is only a partial solution: startup prelude framing is the right mitigation for one injection path, but it is not the complete fix for raw persisted session summaries being recalled outside that path.
