## Summary

Keep open: current main still keeps outer model fallback failures only in `agentMeta.fallbackAttempts` while `executionTrace` is built from embedded-runner-local attempts, and the reported OpenAI/Codex timeout still needs live auth-profile/app-server investigation. The merged loader fix for the earlier beta `MODULE_NOT_FOUND` path is useful context, but it does not address this beta.3 timeout or trace mismatch.

## What This Changes

The issue reports that beta.3 now loads the Codex runtime, but `openai/gpt-5.5` times out, Anthropic/Pi wins via fallback, and result metadata disagrees about whether fallback was used.

## Best Possible Solution

Investigate the beta OpenAI/Codex auth-profile timeout on a real installed macOS/global-npm setup, and add focused trace assembly coverage so outer fallback attempts appear in `executionTrace.attempts` with `fallbackUsed: true`.

## Reproduction Assessment

Partially: there is a high-confidence source path for the telemetry mismatch because outer fallback attempts are copied only to `agentMeta`, while `executionTrace` is assembled from a separate local list. There is not yet a high-confidence current-main live reproduction of the OpenAI/Codex timeout itself.

## Solution Assessment

Unclear for the timeout root cause; it needs live Codex auth-profile/app-server diagnosis before choosing the transport fix. The telemetry part should be solved narrowly in agent-runtime trace assembly rather than by changing fallback selection or making the private Codex subpath public.
