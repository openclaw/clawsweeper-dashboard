## Summary

Keep open: current main still chooses an adjacent owning npm from any inferred npm prefix with no guard for Homebrew Cellar or other per-Node version-manager prefixes, and the linked closing PR is still open with unresolved review feedback.

## What This Changes

The issue reports that `openclaw update` can preserve a per-version Node npm install location, leaving the `openclaw` command unreachable after a version-manager upgrade deletes that location.

## Best Possible Solution

Land or replace the linked PR with a narrow resolver change that separates npm ownership detection from install-command selection, rejects known ephemeral per-Node install prefixes for update argv, and adds regression coverage for the full self-update path.

## Reproduction Assessment

Yes. The source path is high-confidence reproducible: current main derives an owning npm from the package root and would use it for global update, matching the reporter's dangling Homebrew Cellar symlink evidence.

## Solution Assessment

Yes, with a caveat. Rejecting per-version npm command selection is the right narrow fix, but the linked PR currently needs adjustment so trapped npm installs are still detected as npm and then updated through a stable command path.
