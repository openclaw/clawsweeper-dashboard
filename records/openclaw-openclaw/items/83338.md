## Summary

Keep open: current main and v2026.6.8 still keep `sessions_history` on the active `chat.history` path with no archive, family, or compaction-checkpoint selector. The reset-archive fallback only helps when the active transcript is missing, while this report has a small active successor transcript after compaction rotation.

## What This Changes

Request to let `sessions_history` recover transcript content preserved before `truncateAfterCompaction` rotates the active session, likely through an explicit archived or checkpoint history mode.

## Best Possible Solution

Adopt one maintainer-approved, bounded, permission-conscious compaction history recovery contract, then wire it through `chat.history`/`sessions_history` or keep recovery in operator checkpoint branch/restore flows with focused tests and docs.

## Reproduction Assessment

Yes, at source level. A compacted session with a small active successor transcript flows through `sessions_history` to `chat.history`, and the lower reader returns that active transcript before considering any archive fallback.

## Solution Assessment

Unclear. Explicit opt-in is safer than implicit fallback, but maintainers still need to choose whether this belongs in the model-facing tool, an operator-controlled RPC/UI recovery view, or existing compaction checkpoint branch/restore flows.
