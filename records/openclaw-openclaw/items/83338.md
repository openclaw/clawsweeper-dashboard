## Summary

Keep open: current main and v2026.6.8 still expose only the active `chat.history` tail to `sessions_history`; the reset-archive fallback is active-missing only, and the open family-history PR is still unmerged and needs maintainer/security review.

## What This Changes

The issue asks for `sessions_history`/agent history access to recover pre-compaction rotated transcript content when `truncateAfterCompaction` leaves a small active successor transcript.

## Best Possible Solution

Land or narrow a maintainer-approved explicit family/archived-history contract with bounded reads, visibility checks, tests, and docs; keep this issue open until that contract ships or is intentionally rejected.

## Reproduction Assessment

Yes, at source level. A compacted session with a small active successor transcript flows through `sessions_history` to `chat.history`, and the lower reader returns that active transcript before considering reset archives.

## Solution Assessment

Unclear. Explicit opt-in family/history lookup is safer than implicit fallback, but maintainers still need to choose whether this belongs in the model-facing tool, an operator-only recovery view, or checkpoint restore flows.
