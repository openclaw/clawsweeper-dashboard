## Summary

Keep open: current main and the latest release still expose only the active successor transcript through `sessions_history`, while compaction checkpoints are available only through separate operator/session RPCs. The remaining work is a product/API decision about how model-facing history recall should expose archived pre-compaction content.

## What This Changes

The issue requests that `sessions_history` expose or opt into archived pre-compaction transcripts when `truncateAfterCompaction: true` rotates the active JSONL to a small successor file.

## Best Possible Solution

Add an explicit, bounded, redacted checkpoint-history recall path that keeps active-branch history as the default, reuses existing session visibility guards, returns clear segment/checkpoint metadata, and documents the tool/protocol contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a session with an active successor transcript plus a pre-compaction checkpoint, call `sessions_history`, and it will request only `chat.history` for the active transcript.

## Solution Assessment

Unclear. The explicit opt-in direction is safer than silent fallback, but maintainers should decide whether the product surface is `sessions_history`, a separate checkpoint recall tool, or operator-only checkpoint controls.
