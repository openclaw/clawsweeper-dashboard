## Summary

Keep open: current main still keeps `sessions_history` on the bounded active-transcript path, while pre-compaction checkpoint transcripts are exposed only through separate operator/UI recovery controls. The remaining work is an API and visibility-policy decision, not a cleanup-close candidate.

## What This Changes

The issue requests a way for `sessions_history` to recover or opt into pre-compaction archived transcripts after `truncateAfterCompaction` rotates the active JSONL.

## Best Possible Solution

Add an explicit, redacted checkpoint/archive recall mode or separate recovery tool that reuses existing session visibility guards and checkpoint metadata while preserving active-history defaults.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a session with an active successor transcript plus a pre-compaction checkpoint, then call `sessions_history` and it will read only the active transcript through `chat.history`.

## Solution Assessment

Unclear. The explicit opt-in direction is likely safer than implicit fallback, but maintainers should decide whether this belongs in `sessions_history`, a separate checkpoint recall tool, or operator-only checkpoint controls.
