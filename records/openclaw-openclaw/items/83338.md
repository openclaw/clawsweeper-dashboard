## Summary

Keep open: current main still keeps `sessions_history` on the active `chat.history` path, while checkpoint/archive recovery remains a separate operator surface and related archive-history PRs are open but not merged.

## What This Changes

Request model-facing session history recovery for transcript content preserved before compaction rotation instead of returning only the active successor transcript.

## Best Possible Solution

Define one bounded, permission-conscious archived/checkpoint history contract, then wire it through `chat.history`/`sessions_history` or a separate checkpoint recall tool with focused gateway and tool tests.

## Reproduction Assessment

Yes, at source level. A session with a small active successor transcript and preserved pre-compaction checkpoint will flow through `sessions_history` to `chat.history`, which returns the active transcript path before considering any archive/checkpoint content.

## Solution Assessment

Unclear. Explicit opt-in is safer than implicit fallback, but maintainers still need to choose whether this belongs in the model-facing tool, a user-controlled RPC/UI recovery view, or a separate checkpoint recall surface.
