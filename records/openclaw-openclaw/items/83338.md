## Summary

Keep open: current main still gives `sessions_history` only the active transcript tail after compaction rotation, while the pre-rotation JSONL is preserved as a predecessor file that the agent tool cannot request. Related reset-archive work is useful context, but it targets `.reset.*` archives rather than the `parentSession`/successor transcript chain described here.

## What This Changes

The issue asks for `sessions_history` to expose archived pre-rotation transcript content when `truncateAfterCompaction: true` leaves the active successor transcript nearly empty.

## Best Possible Solution

Add a bounded, explicit archived-history mode that can follow compaction successor metadata (`parentSession` or stored checkpoint references), preserves existing redaction and byte caps, documents the contract, and coordinates with the reset-archive API work.

## Reproduction Assessment

Yes, at source level. Current main rotates compaction into a successor transcript with `parentSession`, but `sessions_history` forwards only to active-session `chat.history`, which reads one active transcript path and never follows that predecessor chain.

## Solution Assessment

Partly: an explicit opt-in archive mode is the safest direction, but it should use compaction metadata and bounded reads rather than mtime-only discovery or an implicit fallback that silently changes what `sessions_history` returns.
