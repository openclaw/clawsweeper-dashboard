## Summary

Keep open: current main and the latest release still require the cron minute field to be exactly `0`, so Croner-valid step/list/range minute fields that include minute zero still miss the default stagger. An open cross-repo closing PR exists, but it is unmerged and needs maintainer adoption, replacement, or closure before this issue can be resolved.

## What This Changes

The issue asks OpenClaw to apply the default top-of-hour cron stagger to Croner-valid minute fields that include minute zero, such as step, list, and range expressions, instead of only literal `0`.

## Best Possible Solution

Land one narrow target-repo fix that parses Croner-compatible minute fields for minute-zero inclusion, preserves explicit `staggerMs: 0`, and adds focused coverage for helper, runtime default, normalization, and doctor migration paths.

## Reproduction Assessment

Yes. At source level, `isRecurringTopOfHourCronExpr("*/15 * * * *")` cannot return true because current main requires the minute field to equal `"0"`; I did not run tests because this review is read-only.

## Solution Assessment

Yes, with a tightened implementation. The narrow maintainable fix is to parse Croner-compatible minute fields for minute-zero inclusion in `src/cron/stagger.ts` and cover the runtime/persisted default paths, rather than broadening unrelated scheduler semantics.
