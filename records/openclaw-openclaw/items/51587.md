## Summary

Keep open: current main and v2026.5.22 still require a literal minute field of `"0"`, so Croner-valid step, list, and range minute fields that fire at minute zero still miss the default stagger.

## What This Changes

The issue requests that top-of-hour cron stagger detection recognize Croner minute fields that include minute zero, not only the literal `0` field.

## Best Possible Solution

Land one narrow target-repo fix that parses Croner-compatible minute fields for minute zero and adds regression coverage across helper, create/update, normalization, and doctor migration paths.

## Reproduction Assessment

Yes. Source inspection on current main shows `isRecurringTopOfHourCronExpr` requires a literal minute field of `"0"`, so the reported step/list/range expressions cannot receive the default stagger; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Keeping the issue open is the right handling until a target-repo fix lands; the implementation should be a narrow Croner-compatible minute-zero parser with regression coverage for persisted scheduling paths.
