## Summary

Keep this open: current main and the latest release still use literal minute-field equality, and the live linked PR https://github.com/openclaw/openclaw/pull/51594 still uses closing syntax for this issue. The report is a valid runtime correctness bug with production-impact context, but it should close only after that PR or a narrow replacement lands.

## What This Changes

The issue asks OpenClaw to treat Croner-valid minute fields that include minute zero, such as step/list/range expressions, as eligible for the default top-of-hour cron stagger.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/51594 or a narrow replacement with Croner-compatible minute-zero parsing and persisted-path regression coverage, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes, from source inspection. Current main still rejects `*/15 * * * *`, `0,30 * * * *`, and range/list variants because `isRecurringTopOfHourCronExpr` only accepts a literal minute field of `0`.

## Solution Assessment

Yes, keeping this issue open is the best handling because an open PR already references it with closing syntax. The implementation should stay narrow and match Croner minute-field semantics across helper, service, normalization, and doctor migration paths.
