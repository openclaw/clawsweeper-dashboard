## Summary

Keep open: current main and the latest release still require a literal minute field of `0`, so step/list/range minute-zero cron schedules remain unstaggered; the linked open fork PR needs maintainer adoption, replacement, or closure before this can resolve.

## What This Changes

This issue reports that top-of-hour cron stagger detection misses valid cron minute fields such as `*/15`, `0,30`, and `0-30` that include minute zero.

## Best Possible Solution

Review, land, or replace the linked fork PR with one narrow core fix that parses Croner-compatible minute-zero fields and covers helper, runtime defaulting, normalization, and doctor migration paths.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `isRecurringTopOfHourCronExpr("*/15 * * * *")` cannot return true because the minute field must equal `"0"`; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/cron/stagger.ts`, with regression coverage on the helper and persisted/default-stagger callers rather than broad scheduler or product-policy changes.
