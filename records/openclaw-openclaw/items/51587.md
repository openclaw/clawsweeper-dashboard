## Summary

Keep open. Current main still only recognizes a literal minute field of `0` for default top-of-hour staggering, the create and normalization paths still depend on that helper, and open PR #51594 is already the active implementation candidate with closing syntax.

## What This Changes

The issue requests that top-of-hour cron stagger detection recognize minute fields that include zero through step, list, and range syntax instead of only the literal `0`.

## Best Possible Solution

Land PR #51594 or an equivalent narrow fix that defines Croner-aligned minute-zero semantics, updates `src/cron/stagger.ts` for both 5-field and 6-field expressions, and adds focused regression coverage for the helper plus the create/normalization paths before closing this issue.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: `isRecurringTopOfHourCronExpr("*/15 * * * *")` reaches the `minuteField === "0"` check and returns false, and the default-stagger call sites depend on that result.

## Solution Assessment

Yes for the triage outcome: keeping the issue open while reviewing the linked PR is the narrowest maintainable path. The code fix should stay localized to the cron stagger helper and tests unless Croner-aligned parsing requires a shared helper.
