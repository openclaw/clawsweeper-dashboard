## Summary

Keep open: current main and the latest release still require the cron minute field to be exactly "0", so Croner-valid step/list/range minute fields that fire at minute zero still miss the default stagger. The open cross-repo closing PR is not a merged target-repo fix and reuses the same stale head as an older closed attempt, so the issue still needs maintainer-owned resolution.

## What This Changes

Detect Croner-valid minute fields that include minute zero so recurring top-of-hour cron schedules receive the default stagger window.

## Best Possible Solution

Land or adopt one narrow target-repo fix that parses minute fields according to Croner 10.0.1, preserves explicit `staggerMs: 0`, and covers runtime, normalization, and doctor migration paths before closing this issue.

## Reproduction Assessment

Yes, at source level: `isRecurringTopOfHourCronExpr("*/15 * * * *")` cannot return true because current main requires the minute field to equal `"0"`. I did not run tests because this review is read-only, but the code path is direct and high-confidence.

## Solution Assessment

Yes, with a tightened implementation: the best fix is a narrow Croner-compatible minute-field helper plus focused regression coverage in the existing cron stagger, service, normalization, and doctor paths. Stale duplicate branches should not be treated as the final fix without rebase and review.
