## Summary

Keep open: current main and the latest release still require the cron minute field to be exactly `0`, so the reported step/list/range minute-zero schedules still miss the default stagger. A linked open fork PR appears to be the active fix candidate, so maintainers should adopt, replace, or close that path before another repair lane queues.

## What This Changes

The issue reports that top-of-hour cron stagger detection only recognizes literal minute `0` and misses cron minute syntax that also includes minute zero.

## Best Possible Solution

Land one narrow core fix, either by adopting the linked fork PR or replacing it, that parses Croner-compatible minute fields for minute-zero inclusion and adds focused regression coverage while preserving explicit `staggerMs: 0`.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `isRecurringTopOfHourCronExpr("*/15 * * * *")` cannot return true because the minute field must equal `"0"`; tests were not run because this review must keep the checkout read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/cron/stagger.ts`, with regression coverage on the helper and persisted/default-stagger callers rather than broad scheduler or product-policy changes.
