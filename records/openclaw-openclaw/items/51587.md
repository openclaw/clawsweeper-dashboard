## Summary

Keep open: current main and v2026.5.22 still only apply the default cron stagger when the minute field is the literal `0`, so the reported step/list/range schedules remain unstaggered. The target-repo fix attempts are closed unmerged, and the remaining open implementation path is a fork PR that maintainers still need to adopt, replace, or close.

## What This Changes

The issue asks OpenClaw to detect cron minute fields that include minute zero, such as `*/15`, `0,30`, and `0-30`, when deciding whether to apply the default top-of-hour stagger.

## Best Possible Solution

Keep this issue open until maintainers adopt or replace the linked fork PR with one Croner-compatible fix that adds regression coverage for step, list, range, and 6-field cron expressions while preserving explicit `staggerMs` and `--exact` behavior.

## Reproduction Assessment

Yes, from source inspection: on current main, `isRecurringTopOfHourCronExpr("*/15 * * * *")` reaches the 5-field path and compares `minuteField` to the literal string `"0"`, so it returns false. I did not run tests because this review is read-only and test commands may write artifacts.

## Solution Assessment

Yes for the central direction: the narrow maintainable fix is to parse Croner-valid minute fields enough to detect minute zero, then add focused regression coverage. The open fork PR should still be reviewed or tightened before merge because previous target-repo reviews identified composite-list and wildcard/every-minute semantics to settle.
