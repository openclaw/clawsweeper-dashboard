## Summary

Keep open: current main and latest release still only detect literal minute `0`, while related discussion shows real production impact and the linked implementation path is unresolved.

## What This Changes

The issue requests that default top-of-hour cron staggering recognize step, list, and range minute fields that include minute zero.

## Best Possible Solution

Land or adopt a narrow Croner-compatible fix that detects minute zero in valid step, list, and range fields, with regression coverage across helper, create/normalize, and doctor repair paths.

## Reproduction Assessment

Yes from source inspection: `isRecurringTopOfHourCronExpr("*/15 * * * *")` reaches the 5-field branch and compares the minute field to the literal string `"0"`, so it returns false on current main. I did not run tests because this review is read-only and test commands may write artifacts.

## Solution Assessment

Yes: the maintainable fix is to parse the Croner-compatible minute field enough to know whether minute zero is included, then add focused regression tests without changing explicit `staggerMs` or `--exact` behavior.
