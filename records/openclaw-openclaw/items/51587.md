## Summary

Keep this open: current main and the latest release still require the cron minute field to be the literal string `"0"`, so the reported step/list/range minute-field gap is not fixed. Target-repo fix attempts are closed unmerged, while the only open closing-reference PR is fork-local, so maintainer routing is still needed.

## What This Changes

The issue requests that recurring top-of-hour cron stagger detection recognize minute fields such as step, list, and range expressions that include minute zero.

## Best Possible Solution

Bring a narrow patch into `openclaw/openclaw` that parses cron minute fields for minute zero, preserves explicit `staggerMs: 0` exact scheduling, and adds focused regression coverage for helper, normalization, and job-creation behavior.

## Reproduction Assessment

Yes, at source level: current main requires `minuteField === "0"`, so the reported step/list/range minute fields do not receive the default stagger. I did not run tests because this review is read-only and must not create artifacts.

## Solution Assessment

Yes in direction: the best fix is a small parser/helper in `src/cron/stagger.ts` plus regression tests, not a broad scheduler rewrite. The implementation should explicitly settle wildcard and step semantics such as `*`, `*/1`, and `*/7` in tests.
