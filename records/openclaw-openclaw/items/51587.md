## Summary

Keep open: current main and v2026.6.10 still require the cron minute field to be exactly `0`, so the reported step/list/range minute-zero schedules still miss the default stagger; an open linked fork PR exists but is unmerged and needs maintainer adoption, replacement, or closure.

## What This Changes

The issue asks for top-of-hour cron stagger detection to recognize Croner-valid step, list, and range minute fields that include minute zero.

## Best Possible Solution

Land one narrow core fix that parses Croner-compatible minute fields for minute-zero inclusion, preserves explicit `staggerMs: 0`, and adds focused coverage for helper, runtime default, normalization, and doctor migration paths.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `isRecurringTopOfHourCronExpr("*/15 * * * *")` cannot return true because the minute field must equal `"0"`; I did not run tests because this review is read-only.

Yes. The narrow maintainable fix belongs in `src/cron/stagger.ts`, with regression coverage on the helper and persisted/default-stagger callers rather than broad scheduler or product-policy changes.
