## Summary

Keep open: current main and the latest release still use literal minute-zero checks, while an open PR with closing syntax remains the active implementation path for this cron stagger bug.

## What This Changes

The issue asks OpenClaw to apply the default cron stagger when Croner-valid minute fields such as `*/15`, `0,30`, or `0-30` include minute zero instead of only matching literal `0`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/51594 or an equivalent narrow fix that parses minute fields consistently with Croner and adds regression coverage across helper, normalization, service default-stagger, and doctor migration paths.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `*/15 * * * *` reaches the literal `minuteField === "0"` check and returns false, and creation, patch, normalization, and doctor migration paths depend on that helper.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable solution is the linked PR or an equivalent replacement that follows Croner minute-field semantics, then this issue can close after merge.
