## Summary

Keep open: current main and the latest release still require the cron minute field to be literal `0`, so the reported step/list/range minute-zero schedules still miss the default stagger; an open linked fork PR exists but is unmerged and not maintainer-editable, so this needs maintainer adoption, replacement, or closure rather than cleanup close.

## What This Changes

The issue reports that cron top-of-hour stagger detection misses step/list/range minute fields, such as `*/15`, `0,30`, and `0-30`, that also fire at minute zero.

## Best Possible Solution

Land one narrow core fix that parses minute fields for minute-zero inclusion, preserves explicit `staggerMs: 0`, and adds focused coverage for helper, runtime default, normalization, and doctor migration paths.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `isRecurringTopOfHourCronExpr("*/15 * * * *")` cannot return true because the minute field must equal `"0"`; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/cron/stagger.ts`, with regression coverage on the helper and persisted/default-stagger callers rather than broad scheduler or product-policy changes.
