## Summary

Keep this open: current main and the latest release still only apply the default cron stagger when the minute field is literally `0`, and the linked open fix PR already uses closing syntax for this issue.

## What This Changes

The issue asks for top-of-hour cron stagger detection to parse cron minute fields that include minute zero instead of accepting only a literal `0`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/51594 or a narrow replacement with Croner-compatible minute-zero parsing and regression coverage across the helper, service, normalization, and doctor paths, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection on current main shows `isRecurringTopOfHourCronExpr` checks literal `minuteField === "0"`, so step/list/range expressions like `*/15 * * * *` and `0,30 * * * *` do not receive the default stagger.

## Solution Assessment

Yes. Keeping this issue open is the right handling because an open PR already references it with closing syntax; the code change should stay narrow and align with Croner minute-field parsing.
