## Summary

Current main already routes Feishu async completions through the stored requester session key; channel/account/user metadata is carried as delivery context, so the reported split between the main session and a Feishu direct session is no longer present in the implementation.

## What This Changes

The issue asks for Feishu user DMs and async completions for the same agent/user to reuse one session key instead of splitting between `agent:<agentId>:main` and a Feishu direct key.

## Best Possible Solution

Keep the current requester-session completion routing and ship it in the next release; no new config or product behavior is needed.

## Reproduction Assessment

Yes for the originally reported release: the supplied Feishu DM/session-key evidence plus current source make the old split-session path clear. I did not run a live Feishu websocket repro, and current main's source path now keeps completions on the requester session key.

## Solution Assessment

Yes. Routing async completions through the stored requester session key while keeping Feishu channel/account/to as delivery metadata is the narrow maintainable fix for this state desynchronization.
