## Summary

Keep open: the schtasks-probe direction is plausible, but this branch probes only the default Scheduled Task while current restart handoff supports profile-derived and custom task names, and the PR still lacks real Windows PID/uptime proof.

## What This Changes

The PR adds a synchronous Windows `schtasks /Query /TN OpenClaw Gateway` fallback in supervisor detection when service environment markers are absent.

## Best Possible Solution

Resolve and probe the same Windows task name the restart handoff will run, cover default/profile/custom-name cases, and require real Windows PID or uptime proof before merge.

## Reproduction Assessment

Yes, source-level: current main returns disabled for win32 without Scheduled Task markers, and the run loop then uses the in-process restart fallback rather than replacing the Node process. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No: probing `schtasks` is a plausible safer direction than detached respawn, but this implementation probes only the default task while the handoff can target profile or custom task names.
