## Summary

Keep open: the PR is a plausible safer direction than detached Windows respawn, but it still probes only the default Scheduled Task while the restart handoff can run profile-specific or override task names, and the external real-behavior proof gate is still missing.

## What This Changes

The PR adds a synchronous Windows `schtasks /Query` fallback to `detectRespawnSupervisor()` plus mocked coverage for success, failure, exception, and skip paths.

## Best Possible Solution

Resolve and probe the same Windows task name the restart handoff will run, cover default/profile/custom cases, and require redacted Windows proof before merge.

## Reproduction Assessment

Yes, at source level: current main returns disabled for env-less win32 supervisor detection, and the linked issue describes a manually started gateway with an installed Scheduled Task but missing service env. I did not run live Windows proof in this read-only review.

## Solution Assessment

No/unclear: probing `schtasks` is a plausible safer direction than detached respawn, but this implementation probes only the default task while the restart path can use profile-specific names. The safer solution is to query the exact task name the handoff will run and prove it on Windows.
