## Summary

Keep open: this PR is a useful candidate for the Windows restart issue, but it is not merge-ready because it probes only the default Scheduled Task while the handoff can run profile-specific or explicit task names, and the external PR still has only mock/test evidence rather than real Windows restart proof.

## What This Changes

The PR adds a synchronous Windows `schtasks.exe /Query` fallback to `detectRespawnSupervisor()` plus mocked tests for success, failure, exception, and skip paths.

## Best Possible Solution

Resolve and probe the same Windows task name the restart handoff will run, cover default/profile/custom cases, and require redacted Windows proof before merge.

## Reproduction Assessment

Yes, source-level: current main returns disabled for env-less win32 supervisor detection, and the linked issue describes a manually started gateway with an installed Scheduled Task but missing service env. I did not run live Windows proof in this read-only review.

## Solution Assessment

No/unclear: probing `schtasks` is a plausible safer direction than detached respawn, but this implementation probes only the default task while the restart path can use profile-specific or explicit names. The safer fix is to query the exact task name the handoff will run and prove it on Windows.
