## Label Justifications

## Summary

Keep open: the schtasks probing direction is useful, but this branch can still miss or mis-target profiled Windows Scheduled Tasks and it lacks real Windows after-fix proof.

## What This Changes

This PR adds a synchronous Windows `schtasks /Query` fallback to `detectRespawnSupervisor()` when supervisor env markers are absent, with mocked coverage for success, failure, and skip paths.

## Best Possible Solution

Resolve and probe the same Windows task name the restart handoff will run, cover default/profile/custom cases, and require redacted Windows proof before merge.

## Reproduction Assessment

Yes, at source level: current main returns null for env-less win32 supervisor detection, and the linked issue describes a manually started gateway with an installed Scheduled Task but missing service env. I did not run live Windows proof in this read-only review.

## Solution Assessment

No/unclear: probing `schtasks` is a plausible safer direction than detached respawn, but this implementation probes only the default task while the restart path can use profile-specific names. The safer solution is to query the exact task name the handoff will run and prove it on Windows.
