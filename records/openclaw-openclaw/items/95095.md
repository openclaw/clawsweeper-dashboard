## Summary

Keep open: this is the active candidate for the linked Windows restart issue, but it is not merge-ready because the fallback probes the fixed default Scheduled Task instead of the task the handoff will actually run, and the PR still lacks real Windows behavior proof.

## What This Changes

The PR adds a Windows `schtasks /Query` fallback to supervisor detection when service markers are missing, expands supervisor-marker tests, and adds an unrelated bounded-response test.

## Best Possible Solution

Resolve and probe the same Windows task name the restart handoff will run, cover default/profile/custom cases, split unrelated test churn, and require redacted Windows `/restart` plus `/status` proof before merge.

## Reproduction Assessment

Yes, at source level: current main returns no `schtasks` supervisor for env-less Windows and then keeps ordinary restart on the in-process fallback. I did not run live Windows `/restart` proof in this read-only review.

## Solution Assessment

No/unclear: probing `schtasks` is a safer direction than detached respawn, but this implementation probes only the default task while the restart handoff can use profile-specific or custom task names.
