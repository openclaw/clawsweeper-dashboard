## Summary

Keep open: this is a plausible candidate fix for the linked Windows restart issue, but it is not merge-ready because the probe uses the default task while current restart code supports profile/custom task names, and the PR still lacks real Windows proof.

## What This Changes

The PR adds a Windows `schtasks /Query` fallback in supervisor detection, plus mocked unit coverage, so env-less gateway processes can be treated as Scheduled Task supervised.

## Best Possible Solution

Resolve and probe the same profile/custom task name that the restart handoff will run, preserve the safe fallback when that exact task is absent, add focused tests, and require real Windows proof before merge.

## Reproduction Assessment

Yes, source-level: current main returns null for env-less win32 supervisor detection, and the linked issue identifies a manually started gateway with an installed Scheduled Task but missing service env. I did not run live Windows proof in this read-only review.

## Solution Assessment

No/unclear: probing schtasks is a plausible safer direction, but this implementation probes only the default task and has no after-fix Windows proof. The safer solution is a profile-aware exact-task probe plus regression tests and live proof.
