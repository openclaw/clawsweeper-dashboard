## Summary

Keep open. The missing provider-usage TTL cache is still real on current main, and this PR is useful source work, but the patch is not mergeable as-is because it is based on an older loader contract and its cache key does not cover current runtime/plugin/auth context boundaries.

## What This Changes

This PR adds a configurable `usage.cacheTtlMs` option and an in-memory TTL cache around `loadProviderUsageSummary`, with config schema/help/label/type updates and cache-focused tests.

## Best Possible Solution

Rework the cache on top of current main’s loader, preserve every existing loader option and plugin/auth boundary, key or bypass caching across all inputs that can change the summary, avoid caching transient failures, document the new config, and add a changelog entry plus focused regression tests.

## Reproduction Assessment

Yes for the missing-cache behavior: current main has direct status and gateway calls into `loadProviderUsageSummary`, and the Anthropic fetcher performs a live usage endpoint request with no loader-level TTL cache. The provider-side 429 rate-limit symptom was not live-tested.

## Solution Assessment

No, this PR is not the best mergeable solution as-is. A loader-level TTL cache is the right narrow direction, but the patch must be rebased onto the current loader contract and keyed or bypassed across current runtime/plugin/auth context boundaries.
