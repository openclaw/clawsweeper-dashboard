## Summary

Keep open. Current main still lacks provider usage TTL caching, and this PR is a useful direction, but the branch is stale against the current loader contract and would cache across runtime/plugin contexts unless repaired before merge.

## What This Changes

This PR adds a configurable in-memory TTL cache around provider usage summaries, updates config schema/help/labels/types, and adds cache-focused provider usage tests.

## Best Possible Solution

Rebase the loader-level TTL cache onto current main, preserve every existing loader option and plugin/auth boundary, key or bypass caching across all inputs that can change the summary, avoid caching transient failures, and add focused tests plus the required changelog entry.

## Reproduction Assessment

Yes for the missing-cache behavior: current main’s status paths call `loadProviderUsageSummary`, and the loader reaches the Anthropic usage fetcher without any TTL cache. I did not live-test Anthropic 429s, so the rate-limit symptom is source-backed rather than live reproduced.

## Solution Assessment

No, this PR is not the best mergeable solution as-is. A loader-level TTL cache is the right narrow direction, but this patch needs to be rebased onto the current loader contract and keyed or bypassed across config/env/workspace/plugin/auth boundaries.
