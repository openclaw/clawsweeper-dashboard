## Summary

Keep open: the linked bug is still source-reproducible on current main, and this PR is the active implementation candidate, but the patch is not mergeable as-is because it predates the current provider-usage plugin/context boundary and lacks real OAuth proof.

## What This Changes

The PR adds a configurable in-memory TTL cache for provider usage summaries to reduce repeated Anthropic OAuth usage endpoint calls.

## Best Possible Solution

Implement a loader-level TTL cache that preserves current `opts.config`, `env`, `workspaceDir`, plugin usage hooks, and credential-source policy, isolates cache entries by the relevant runtime boundaries, and includes focused regression tests plus real OAuth status-call proof.

## Reproduction Assessment

Yes, at source level: current status paths call `loadProviderUsageSummary`, and the Anthropic path performs live usage endpoint I/O without TTL caching. I did not live-trigger provider 429s, so the rate-limit symptom remains source-backed rather than live reproduced.

## Solution Assessment

No, not as currently proposed. A loader-level TTL cache is the right narrow direction, but this PR must preserve current runtime context and cache isolation before it is maintainable.
