## Summary

Keep open: current main still lacks a provider usage TTL cache, and this PR is the canonical implementation candidate, but the head is conflicting/outdated and still has cache-boundary correctness issues plus missing real OAuth proof.

## What This Changes

Adds a configurable `usage.cacheTtlMs` in-memory TTL cache around `loadProviderUsageSummary`, with config schema/help/labels/types updates and cache-focused provider usage tests.

## Best Possible Solution

Land a rebased, context-isolated provider usage cache that avoids caching transient failures and includes redacted real Anthropic OAuth proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level for duplicate upstream calls: current main status and gateway usage paths still enter the uncached loader, and the Anthropic fetcher performs a live usage request per snapshot; I did not reproduce a live 429.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed; a loader-level cache is plausible, but it must be rebased to preserve current context inputs and key or bypass cached results across every runtime boundary that affects provider usage output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve usage loader context before caching:** `src/infra/provider-usage.load.ts:73-80`
  - body: Current main callers can pass `config`, `env`, `workspaceDir`, and credential-source policy into provider usage resolution. This patch reads ambient config and caches around an options type that lacks those fields, so a rebased version can resolve auth and fetch plugin snapshots under the wrong runtime context.
  - confidence: 0.9
- **[P2] Key cached usage by runtime context:** `src/infra/provider-usage.load.ts:30-33`
  - body: The cache key only includes agent directory and provider ids, but provider usage output can vary by config, env, workspace plugin resolution, credential-source policy, and injected provider hooks. Within the TTL, one runtime context can receive another context's cached usage or plan data.
  - confidence: 0.87
