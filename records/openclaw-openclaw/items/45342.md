## Summary

Keep open: current main still lacks provider usage TTL caching, but this branch is not mergeable as-is because it drops current runtime context/plugin usage hooks, uses too-narrow cache boundaries, and lacks real OAuth proof.

## What This Changes

The PR adds a `usage.cacheTtlMs` config setting and an in-memory cache around `loadProviderUsageSummary`, with tests for cache hits and explicit-auth bypass.

## Best Possible Solution

Replace or rebase this as a loader-level TTL cache that preserves `opts.config`, `env`, `workspaceDir`, plugin usage hooks, credential-source policy, and cache isolation boundaries, with focused regression tests and real OAuth proof.

## Reproduction Assessment

Yes, at source level: current status paths call `loadProviderUsageSummary`, and the Anthropic path performs live usage endpoint I/O without TTL caching. I did not live-trigger provider 429s, so the rate-limit symptom remains source-backed rather than live reproduced.

## Solution Assessment

No, not as currently proposed. A loader-level TTL cache is the right narrow direction, but this PR must preserve current runtime context and cache isolation before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the current loader context before caching:** `src/infra/provider-usage.load.ts:76-80`
  - body: Current main callers can pass `config`, `env`, `workspaceDir`, and plugin-auth policy into the usage loader, and the loader forwards that context into provider usage hooks. This patch reads ambient config and keeps the older direct fetch path, so a rebase would resolve credentials and plugin snapshots from the wrong runtime context.
  - confidence: 0.9

On current main, the same provider summary can differ by config, env, workspace plugin resolution, credential-source policy, and provider usage hooks, so one status call can receive another context's cached usage or plan data until the TTL expires.
  - confidence: 0.86
