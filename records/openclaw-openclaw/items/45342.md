## Summary

Keep open: the PR targets a real current-main gap, and the author addressed earlier clock/key/transient-error review comments, but the reviewed head predates current provider-usage hook context, the cache boundary is too narrow, and required real Anthropic OAuth proof is still missing.

## What This Changes

The PR adds a configurable `usage.cacheTtlMs` in-memory cache around `loadProviderUsageSummary`, updates config schema/help/labels/types, and adds cache-focused provider usage tests.

## Best Possible Solution

Rebase onto current main and implement a context-preserving provider usage cache that isolates every usage-output boundary, avoids caching transient failures, keeps focused regression coverage, and includes redacted real OAuth proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level for duplicate upstream calls: status, session_status, and gateway usage paths still enter the uncached loader, and the Anthropic fetcher performs a live usage request per snapshot; I did not reproduce an actual 429 with a real OAuth account.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed; a loader-level cache is plausible, but it must be rebased to preserve current context inputs and key cached results by every runtime boundary that affects provider usage output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve usage loader context before caching:** `src/infra/provider-usage.load.ts:76-80`
  - body: Current main callers and hooks can pass `config`, `env`, `workspaceDir`, and credential-source policy into provider usage resolution. This patch reads ambient config and then resolves auth/fetches snapshots with only providers/auth/agentDir, so a rebased cache can fetch summaries under the wrong runtime context and skip provider plugin usage hooks.
  - confidence: 0.9
- **[P2] Key cached usage by runtime context:** `src/infra/provider-usage.load.ts:30-33`
  - body: The cache key only includes agent directory and provider ids, but provider usage output can vary by config, env, workspace plugin resolution, credential-source policy, and injected provider hooks. Within the TTL, one runtime context can receive another context's cached usage or plan data.
  - confidence: 0.86
