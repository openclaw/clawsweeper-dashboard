## Summary

Keep open: the linked bug is still source-reproducible on current main, but this PR is not merge-ready because the branch is stale/conflicting, lacks real Anthropic OAuth proof, and the proposed cache does not preserve current runtime context boundaries.

## What This Changes

The PR adds a configurable 5-minute in-memory cache for provider usage summaries, updates `usage.cacheTtlMs` config schema/help/labels/types, and adds cache behavior tests.

## Best Possible Solution

Rebase and implement a context-preserving provider usage cache that isolates every usage-output boundary, avoids caching transient failures, keeps focused regression coverage, and includes redacted real OAuth proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main status paths call `loadProviderUsageSummary`, the loader has no cache, and the Anthropic fetcher performs a live upstream usage request per snapshot; I did not reproduce a live 429 in a real OAuth account.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed; a loader-level cache is plausible, but it must be rebased to preserve current context inputs and key cached results by every runtime boundary that affects provider usage output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve usage loader context before caching:** `src/infra/provider-usage.load.ts:76-95`
  - body: Current main callers can pass `config`, `env`, `workspaceDir`, and credential-source policy into `loadProviderUsageSummary`, and the loader forwards that context to auth and provider usage hooks. This patch reads ambient config and calls auth resolution with only providers/auth/agentDir, so a rebased cache can fetch summaries under the wrong runtime context.
  - confidence: 0.9
