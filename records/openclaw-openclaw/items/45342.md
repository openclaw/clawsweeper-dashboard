## Summary

Keep open: current main still lacks a provider-usage TTL cache, so the underlying bug is valid, but this PR is not mergeable as-is because it is stale against the current loader contract and still needs cache-boundary and changelog repairs.

## What This Changes

The PR adds an in-memory TTL cache for `loadProviderUsageSummary`, exposes `usage.cacheTtlMs` in config schema/help/labels/types, and adds provider usage cache tests.

## Best Possible Solution

Rebase the loader-level TTL cache onto current main, preserve every existing loader option and plugin/auth boundary, avoid caching transient failures, key or bypass all inputs that can change the summary, and add focused tests plus the required changelog entry.

## Reproduction Assessment

Yes, source-reproducible: current main status and usage paths call `loadProviderUsageSummary`, and the Anthropic path performs a live `api.anthropic.com/api/oauth/usage` fetch with no TTL cache. I did not live-trigger provider 429s in this read-only review.

## Solution Assessment

No, this PR is not the best mergeable solution as-is. A loader-level TTL cache is the right narrow direction, but the patch must be rebased onto the current loader contract and isolated across config, env, workspace, provider, plugin, and credential-policy boundaries.
