## Summary

Keep open: current main still has the uncached provider-usage path, but this PR is not merge-ready because the cache drops current runtime context boundaries and lacks real Anthropic OAuth proof.

## What This Changes

This PR adds a configurable in-memory TTL cache for provider usage summaries, wires `usage.cacheTtlMs` into config schema/help/labels/types, and adds provider usage cache tests.

## Best Possible Solution

Rebase onto current main and implement a context-preserving provider usage cache that isolates every usage-output boundary, avoids caching transient failures, keeps focused regression coverage, and includes redacted real OAuth proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level for duplicate upstream calls: current main status/session_status and gateway usage paths still call the uncached loader, and the Anthropic fetcher performs a live usage request per snapshot; I did not run a real OAuth account to reproduce an actual 429.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed; a loader-level cache is plausible, but it must be rebased to preserve current context inputs and key cached results by every runtime boundary that affects provider usage output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve usage loader context before caching:** `src/infra/provider-usage.load.ts:76-80`
  - body: Current main callers and hooks can pass `config`, `env`, `workspaceDir`, and credential-source policy into provider usage resolution. This patch reads ambient config and then resolves auth with only providers/auth/agentDir, so a rebased cache can fetch summaries under the wrong runtime context.
  - confidence: 0.9

  - body: The cache key only includes agent directory and provider ids, but provider usage output can vary by config, env, workspace plugin resolution, credential-source policy, and injected provider hooks. Within the TTL, one runtime context can receive another context's cached usage or plan data.
  - confidence: 0.86
