## Summary

Keep open: current main and the latest release still lack a provider-usage TTL cache, and the linked bug was closed as superseded by this PR. The PR is not merge-ready because it lacks real Anthropic OAuth proof, is conflicting against current main, and its cache design does not preserve the current context-sensitive usage boundary.

## What This Changes

The PR adds a configurable in-memory TTL cache for provider usage summaries, config schema/help/labels/types, and focused provider-usage cache tests.

## Best Possible Solution

Implement a current-main loader-level cache that preserves injected runtime context, isolates entries by every usage-output boundary, skips transient failures, adds regression coverage, and includes real OAuth status-call proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main status/gateway paths call `loadProviderUsageSummary`, and Anthropic usage fetching performs a live OAuth usage request for each snapshot; I did not reproduce a live 429 in a real OAuth account.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed; a loader-level TTL cache is plausible, but this branch must be rebased around the current context-aware provider usage contract and cache isolation rules.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve usage loader context before caching:** `src/infra/provider-usage.load.ts:70-80`
  - body: Current main callers can pass `config`, `env`, `workspaceDir`, and credential-source policy into `loadProviderUsageSummary`, and the loader forwards that context into auth and provider usage hooks. This patch reads ambient config and is based on the older options shape, so a rebased cache can fetch or return summaries under the wrong runtime context.
  - confidence: 0.9
- **[P2] Key cached usage by runtime context:** `src/infra/provider-usage.load.ts:30-33`
  - body: The cache key only includes `agentDir` and provider ids. Provider usage output can vary by config, env, workspace plugin resolution, credential-source policy, and provider hooks, so one status call can receive another context's cached usage or plan data until the TTL expires.
  - confidence: 0.86
