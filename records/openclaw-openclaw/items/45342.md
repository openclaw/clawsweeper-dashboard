## Summary

Keep open: current main still lacks a provider-usage TTL cache and the underlying bug remains source-provable, but this PR cannot merge as-is because its cache is stale against the current loader/plugin contract and lacks real OAuth behavior proof.

## What This Changes

The PR adds `usage.cacheTtlMs` config/schema/help text plus an in-memory `loadProviderUsageSummary` cache with tests for cache hits and explicit-auth bypass.

## Best Possible Solution

Rebase or replace the patch on current main with a loader-level TTL cache that preserves `opts.config`, `env`, `workspaceDir`, plugin usage hooks, credential-source policy, and all cache boundaries, with focused regression tests and real OAuth proof.

## Reproduction Assessment

Yes for the missing-cache behavior: current main status paths call `loadProviderUsageSummary`, and the Anthropic provider path performs a live usage endpoint request without TTL caching. I did not live-trigger provider 429s, so the rate-limit symptom remains source-backed rather than live reproduced.

## Solution Assessment

No, not as currently proposed. A loader-level TTL cache is the right narrow direction, but this PR uses an older loader shape and must preserve current runtime context and cache isolation boundaries before it is mergeable.
