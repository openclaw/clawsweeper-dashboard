## Summary

Keep open: current main still has the uncached provider-usage path, so the PR is not obsolete, but the branch should not merge as-is because it is stale against the current config/env/workspace/plugin-aware loader contract, has too-narrow cache isolation, and lacks real Anthropic OAuth proof.

## What This Changes

The PR adds a configurable in-memory TTL cache for provider usage summaries, updates usage config schema/help/labels/types, and adds focused provider-usage cache tests.

## Best Possible Solution

Implement a current-main loader-level cache that preserves injected runtime context, isolates entries by every usage-output boundary, skips transient failures, adds regression coverage, and includes real OAuth status-call proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main status/gateway paths call loadProviderUsageSummary and Anthropic usage fetching performs a live OAuth usage request; I did not reproduce a live 429 in a real OAuth setup.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed; a loader-level TTL cache is the right direction, but this branch must be rebased around the current context-aware provider usage contract and cache isolation rules.
