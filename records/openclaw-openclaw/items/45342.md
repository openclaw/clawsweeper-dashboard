## Summary

Keep this PR open. Current main still lacks the requested provider-usage TTL cache, so the PR is not obsolete, but the branch is stale against the current provider-usage loader contract and should not merge as-is without preserving the newer config/env/workspace/plugin-auth boundaries.

## What This Changes

This PR adds a configurable `usage.cacheTtlMs` option and an in-memory TTL cache around `loadProviderUsageSummary`, with tests for a cache hit and explicit-auth cache bypass.

## Best Possible Solution

Rework the cache in or near `src/infra/provider-usage.load.ts` on top of current main, keep the existing provider/auth/plugin options intact, key or bypass the cache across all context boundaries that can change a usage summary, avoid caching transient failures, and add focused regression tests for cache hit, expiry, disabled TTL, explicit-auth bypass, provider-list boundaries, plugin/config/env/workspace boundaries, and transient-error retry behavior.

## Reproduction Assessment

Yes for the missing-cache behavior: current main has direct status and gateway calls into `loadProviderUsageSummary`, and the Anthropic helper performs a live usage endpoint fetch with no provider-summary TTL layer. The live provider-side 429 symptom was not reproduced in this read-only pass.

## Solution Assessment

No, this PR is not the best mergeable solution as-is. The loader-level cache is the right narrow direction, but it needs to be rebased onto current main and preserve the current runtime/plugin/auth context before merge.
