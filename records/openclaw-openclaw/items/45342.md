## Summary

Keep open: current main still has no provider usage cache and this PR remains the canonical open fix path, but the branch is conflicting and the proposed cache is not safe to merge as-is.

## What This Changes

The PR adds a configurable in-memory TTL cache for provider usage summaries, plus config schema/help/labels/types updates and provider-usage cache tests.

## Best Possible Solution

Rebase and implement a context-preserving provider usage cache that isolates every usage-output boundary, avoids caching transient failures, has focused regression coverage, and includes redacted real OAuth proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main status/gateway paths call `loadProviderUsageSummary`, and the Anthropic fetcher performs a live upstream usage request for each snapshot; I did not reproduce a live 429 in a real OAuth account.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed; a loader-level cache is plausible, but this branch must preserve current-main context inputs and key cached results by all runtime boundaries that affect provider usage output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve usage loader context before caching:** `src/infra/provider-usage.load.ts:76-80`
  - body: Current main callers can pass `config`, `env`, `workspaceDir`, and credential-source policy into `loadProviderUsageSummary`, and the loader forwards that context to auth and provider usage hooks. This patch reads ambient config and is based on the older options shape, so a rebased cache can fetch or return summaries under the wrong runtime context.
  - confidence: 0.9
- **[P2] Key cached usage by runtime context:** `src/infra/provider-usage.load.ts:30-33`
  - body: The cache key only includes `agentDir` and provider ids. Provider usage output can vary by config, env, workspace plugin resolution, credential-source policy, and provider hooks, so one status call can receive another context's cached usage or plan data until the TTL expires.
  - confidence: 0.86
