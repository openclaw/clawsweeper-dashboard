## What This Changes

The PR adds a configurable in-memory TTL cache for provider usage summaries plus config schema/help/labels/types entries and provider usage cache tests.

## Best Possible Solution

Land a fresh or rebased provider-usage cache only after it is scoped or bypassed for every auth/snapshot runtime input, avoids caching transient failures, documents the public config behavior, and includes real Anthropic OAuth proof.

## Reproduction Assessment

Yes at source level: current main status/auth-status paths call loadProviderUsageSummary, and the Anthropic fetcher performs a live usage request per snapshot. I did not reproduce a live 429 in a real OAuth setup.

## Solution Assessment

No, not as currently proposed. A loader-level cache is plausible, but it must be rebased and scoped to all runtime inputs that affect provider usage output before it is the maintainable fix.
