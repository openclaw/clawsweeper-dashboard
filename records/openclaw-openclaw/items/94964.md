## Summary

## What This Changes

The PR adds a pending channel-reload abort flag called from SIGUSR1/in-process restart paths, updates deferred channel reload checks, and also changes shared reply dispatch for Telegram reasoning payloads.

## Best Possible Solution

Land one focused gateway lifecycle fix that aborts and drains active config hot reload before restart/session/channel shutdown can release stale channel stop/start work, and keep Telegram durable reasoning in its own reviewed PR.

## Reproduction Assessment

Yes, at source level: current main lets deferred channel reloads resume into stop/start while shutdown does not abort or drain the active config reloader before channel teardown. I did not run a live Telegram webhook EADDRINUSE reproduction in this read-only review.

## Solution Assessment

No. The better fix is lifecycle-owned cancellation and drain in the config reloader plus close ordering; this PR’s global flag is too narrow, and the branch should not carry the unrelated Telegram reasoning change.
