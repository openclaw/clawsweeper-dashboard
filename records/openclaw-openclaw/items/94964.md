## Summary

Keep open: this PR targets a real gateway reload/restart race, but the current abort branch is indistinguishable from a successful wait, so stale reload callers can still continue into channel stop/start, and the reset hook fires after the old shutdown lifecycle is already underway.

## What This Changes

The PR adds a process-wide deferred channel reload abort flag and triggers it from the in-process restart reset path via dynamic import.

## Best Possible Solution

Use lifecycle-owned reload cancellation and drain from restart acceptance through shutdown, return a caller-visible cancelled result, recheck before channel start, and add real gateway/Telegram proof.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main waits for active work and then calls channel stop/start without a restart cancellation result, while shutdown does not abort or drain active config reload work.

## Solution Assessment

No. The PR targets the right race, but a process-wide flag fired from next-start reset is too late and does not actually stop callers from continuing after the wait helper returns.
