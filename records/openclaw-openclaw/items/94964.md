## Summary

Keep open: the latest revision fixes the earlier void-return problem, but it can still miss a restart abort during the poll sleep and it cancels reloads before SIGUSR1 is authorized or accepted. The contributor proof is still static/module checks plus tests, not a real gateway/Telegram/SIGUSR1 after-fix run.

## What This Changes

The PR adds a module-level channel-reload abort flag, wires SIGUSR1 and restart reset paths to set it, and skips channel stop/start work when the deferred reload wait reports cancellation.

## Best Possible Solution

Tie cancellation to an accepted restart lifecycle, check it after every wait and immediately before channel start, and prove the fixed path with redacted real gateway/Telegram SIGUSR1 output.

## Reproduction Assessment

Yes at source level: the linked issue gives a concrete deferred reload plus SIGUSR1 EADDRINUSE timeline, and current main still has reload wait and channel restart paths with no restart cancellation result. I did not run a live Telegram webhook reproduction in this read-only review.

## Solution Assessment

No. The PR targets the right race, but the best fix needs lifecycle-owned cancellation that is only armed for accepted restarts and is rechecked after waits and before channel starts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Check cancellation after the poll sleep:** `src/gateway/server-reload-handlers.ts:318-326`
  - body: `pendingChannelReloadAborted` is only checked before the 500 ms sleep. If SIGUSR1 arrives during that sleep and active work drains before the timer fires, this code returns `false` at the zero-active branch and proceeds to restart channels, which is the reported race window.
  - confidence: 0.93
- **[P2] Abort only after SIGUSR1 is accepted:** `src/cli/gateway-cli/run-loop.ts:724-733`
  - body: This abort runs before the handler reads restart intent or authorization. An unauthorized SIGUSR1 that current code would ignore can now cancel a pending channel reload without any restart to apply the new channel config.
  - confidence: 0.9
- **[P2] Recheck cancellation before starting the channel:** `src/gateway/server-reload-handlers.ts:529-533`
  - body: Once the wait has returned, a restart accepted while `stopChannel()` is in flight is never observed before `startChannel()` runs. Recheck the durable cancellation signal after stop and before start so the old reload cannot race the restarted gateway.
  - confidence: 0.84
