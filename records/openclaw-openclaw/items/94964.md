## Summary

Keep open: this is a focused candidate for the deferred channel reload plus in-process restart race, but the current head still accepts a file-intent SIGUSR1 restart without cancelling the pending deferred reload when SIGUSR1 authorization is false, so the original duplicate channel start race can survive on that supported restart path.

## What This Changes

The PR adds a generation-scoped deferred channel reload abort hook, wires it into SIGUSR1/restart reset paths through the gateway lifecycle runtime, and adds reload-handler regression tests.

## Best Possible Solution

Land one canonical gateway-owned cancellation/drain fix that aborts stale deferred reload work for every accepted restart path, then supersede the overlapping candidate PRs.

## Reproduction Assessment

Yes, at source and log level: the canonical issue gives a concrete Telegram EADDRINUSE timeline, current main resumes deferred channel reloads into stop/start without cancellation, and the PR gap is traceable through the file-intent SIGUSR1 branch. I did not run a live Telegram webhook reproduction in this read-only review.

## Solution Assessment

No. The generation-based abort is a plausible mitigation, but this branch still misses an accepted restart path; the narrow repair is to abort once restartIntent is present before request("restart", ...), or for maintainers to choose the broader lifecycle-owned cancellation candidate.
