## Summary

Keep this PR open, but it is not merge-ready: the latest head removed the unrelated memory change and now targets the right gateway race, yet the process-wide abort flag can be cleared by the next gateway lifecycle while an old active hot reload is still running, and real gateway/Telegram/SIGUSR1 proof is still missing.

## What This Changes

The PR adds a deferred channel reload abort flag, triggers it from SIGUSR1 and in-process restart reset paths, and skips channel stop/start when the reload wait reports cancellation.

## Best Possible Solution

Use lifecycle-owned cancellation and drain for active config reloads during restart shutdown, preserve the cancellation state until old reload callbacks finish, recheck before channel start, and require real gateway/Telegram proof before merge.

## Reproduction Assessment

Yes, at source level: current main can leave active hot reload callbacks undrained during restart, and the linked issue provides a concrete Telegram EADDRINUSE timeline. I did not run a live Telegram webhook reproduction.

## Solution Assessment

No. Returning a cancellation result fixed one previous bug, but a process-global flag that resets at new handler creation is not the best lifecycle boundary while old reload callbacks can still be active.

- **[P1] Keep cancellation alive until old reloads drain:** `src/gateway/server-reload-handlers.ts:222`
