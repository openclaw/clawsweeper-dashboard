## Summary

Needs changes before merge: the latest head fixed the channel-only abort path, but the plugin-reload abort is still checked only before `reloadPlugins()` starts, while the flag can only be set from `beforeReplace` during that call. The callee then continues into plugin metadata/runtime replacement, so a superseded hot reload can still mutate plugin state during an in-process restart.

## What This Changes

The PR adds generation-scoped cancellation for deferred channel reloads during SIGUSR1/in-process restarts, wires the abort into gateway restart paths, and adds gateway reload/run-loop regression coverage.

## Best Possible Solution

Land one canonical gateway lifecycle cancellation fix that propagates abort through plugin reload before any plugin/runtime or channel side effects, then close or supersede overlapping candidates.

## Reproduction Assessment

Yes. The canonical issue gives a concrete Telegram EADDRINUSE timeline, and source inspection shows the current PR still does not abort the plugin reload callee after `beforeReplace` observes restart cancellation.

## Solution Assessment

No. The generation-scoped abort is a plausible channel-only fix, but this PR is not the best fix until plugin reload cancellation is propagated through the callee before plugin runtime replacement.
