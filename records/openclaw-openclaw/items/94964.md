## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep restart cancellation scoped to the old reload lifecycle:** `src/gateway/server-reload-handlers.ts:222`
  - body: `pendingChannelReloadAborted` is module-global, and this reset runs whenever the restarted gateway creates new reload handlers. Current `configReloader.stop()` does not await an already-running hot reload, so an old reload can still be waiting in `stopChannel()` when the new gateway clears the flag; its later pre-start check then sees `false` and can still start the stale channel. Use a lifecycle-owned abort signal/generation or drain the active config reload instead of resetting shared cancellation state from the next lifecycle.
  - confidence: 0.88
- **[P2] Split out the Telegram reasoning delivery change:** `src/auto-reply/reply/dispatch-from-config.ts:3177`
  - body: This reload PR also lets Telegram reasoning payloads through the generic auto-reply path, and the same change appears again in final delivery below. That is a separate visible Telegram message-delivery behavior with no matching proof in this PR, so it should be removed from this branch or moved to a focused Telegram PR with proof and tests.
  - confidence: 0.83
