## Summary

Keep open. Current main still lacks package.json-only library-extension coverage in the bundled runtime-deps scan, so this PR is not obsolete, but the current head is not landable because it deliberately bypasses plugin disable/deny gates and still lacks real startup/bootstrap proof after maintainer review called those blockers out.

## What This Changes

The PR adds package.json `openclaw.bundle.stageRuntimeDependencies` detection for manifest-less bundled library extensions, documents a library-extension gate exception, and adds scan/doctor regression tests.

## Best Possible Solution

Land this PR or an equivalent narrow change after maintainers choose the library-extension contract: either keep existing disable/deny gates and scope the fix to doctor/unscoped repair, or document an explicit infrastructure exception and add bootstrap coverage for any claimed startup prestage behavior while preserving the existing on-demand public-artifact runtime preparation.

## Reproduction Assessment

Yes. Current main can be inspected with high confidence: the packaged scan and doctor path use the manifest/policy-based shared package plan, while `media-understanding-core` has no manifest and only a package-level `stageRuntimeDependencies` flag.

## Solution Assessment

No, not as-is. The package-level discovery belongs near the shared runtime-deps selector, but the current PR’s gate bypass and startup-facing documentation need maintainer-approved policy and coverage before merge.
