## Summary

Keep open. Current main still has the manifest-only packaged runtime-deps selection gap this PR is trying to close, but the latest patch is not landable because it moves restrictive `plugins.allow` filtering ahead of existing explicit channel-enable handling and also lacks the required changelog entry for a user-visible repair fix.

## What This Changes

The PR adds package.json `openclaw.bundle.stageRuntimeDependencies` discovery for manifest-less bundled library extensions in packaged runtime-deps scan and doctor repair, with regression tests.

## Best Possible Solution

Keep the PR focused on packaged scan and doctor repair: add the manifest-less package.json fallback without changing the existing channel-enable, deny, global-disable, entry-disable, and allowlist semantics, cover those cases in tests, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. The regression path is a packaged runtime-deps plan with `plugins.allow: ["browser"]` and `channels.telegram.enabled: true`; current tests expect Telegram deps to be included, but the PR's new early allowlist return rejects Telegram before channel inspection.

## Solution Assessment

No. The package.json fallback is the right narrow direction, but the current placement changes unrelated allowlist behavior; the safer implementation preserves the existing selector order and gates the new library fallback only where it cannot suppress explicit channel activation.
