## Summary

The frozen `ingress-spool-*` directory is explained by the shipped v2026.6.1 migration of channel ingress queues to SQLite, not by itself a dormant writer. The issue should remain open narrowly because the reported drain error may still be real and the plugin’s need for full raw updates or probeable queue health may expose a supported-API gap requiring product judgment.

## What This Changes

Investigate Telegram ingress after the filesystem-to-SQLite queue migration and provide a supported way for plugins to consume full raw updates or detect queue dormancy if current hooks and status are insufficient.

## Best Possible Solution

Keep SQLite as the only ingress queue, obtain the complete drain error and `openclaw channels status --probe` evidence, then extend the documented Telegram/plugin ingress or status contract only if existing hooks cannot deliver the required raw update and health information.

## Reproduction Assessment

No. Current source and v2026.6.1 history show that filesystem directory writes intentionally stopped when ingress queues moved to SQLite, while the report lacks the complete drain exception and status data needed to reproduce a separate current failure.

## Solution Assessment

No. Restoring or duplicating filesystem writes is not the best solution; the maintainable path is the canonical SQLite queue plus a supported raw-update or health API if current plugin hooks and channel status do not meet the use case.
