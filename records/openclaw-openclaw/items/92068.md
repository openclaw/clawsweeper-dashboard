## Summary

The reported filesystem symptom is explained by the shipped migration from per-update spool files to the shared SQLite ingress queue, so current main is not expected to update that directory. The item should remain open because the same author’s linked Telegram media-hook report may account for the actual lost-media behavior, and deciding whether plugins need a supported queue-health surface requires explicit product/API judgment.

## What This Changes

Clarify the post-upgrade Telegram ingress storage behavior and request simultaneous hook/spool ingestion plus a probeable writer-health signal for plugin consumers.

## Best Possible Solution

Keep one canonical isolated polling consumer, resolve the supported `message:received` media gap in the linked report, and separately design a narrow typed status/plugin API for ingress queue progress only if existing account health cannot meet watchdog needs.

## Reproduction Assessment

No. Current main intentionally stores ingress records in SQLite, so a frozen legacy spool directory is expected; the missing full error and queue database evidence prevent a high-confidence reproduction of actual writer dormancy.

## Solution Assessment

No. Simultaneous polling and restoration of the retired file spool would create competing consumers and duplicate state; the maintainable path is supported-hook correctness plus an explicitly designed health contract if needed.
