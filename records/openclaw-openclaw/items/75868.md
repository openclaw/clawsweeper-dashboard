## Summary

Keep open. Current main still has a real Telegram plugin bug: `sendMessageTelegram` defaults to markdown mode, but the default text and media follow-up branches bypass the existing chunk planner and can send one oversized Bot API message. The latest shipped release contains the same gate, and the related merged work only covered HTML-mode chunking.

## What This Changes

Fix Telegram outbound delivery so default markdown-mode text over Telegram's message limit is split into safe chunks instead of being sent as one rejected `sendMessage` call.

## Best Possible Solution

Fix the Telegram plugin helper so every `sendMessageTelegram` text path, including default markdown sends and media caption follow-up text, plans safe chunks before Bot API delivery while preserving markdown rendering, plain-text fallback, HTML parse fallback, thread/reply parameters, silent/link-preview options, and reply markup placement on the final chunk.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `sendMessageTelegram` with default options and a >4096-character text using a mocked `api.sendMessage`; current main follows the markdown branch and constructs one send payload instead of multiple <=4000-character sends.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/telegram/src/send.ts`, because direct helper callers can bypass higher-level pre-chunking. The fix should reuse the existing planner rather than moving Telegram-specific chunking into core.
