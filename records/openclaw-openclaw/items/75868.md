## Summary

Keep open. Current `main` still has the reported Telegram helper bug: `sendMessageTelegram` defaults to markdown mode, but only the `textMode === "html"` branch invokes the chunk planner; the default branch sends a single rendered HTML message. The same branch is present in the latest release tag, and the documented `textChunkLimit` expectation is not consistently enforced for payload paths that call `sendMessageTelegram` without `textMode: "html"`.

## What This Changes

The issue requests that Telegram outbound markdown/default-mode text longer than Telegram's 4096-character limit be split into safe chunks instead of being sent as one rejected Bot API message.

## Best Possible Solution

Fix the Telegram plugin boundary so every `sendMessageTelegram` text path, including default markdown sends and media caption follow-up text, plans safe chunks before Bot API delivery while preserving markdown rendering, plain-text fallback, reply markup placement, thread/reply parameters, and existing HTML-mode behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `sendMessageTelegram` with default options and a >4096-character text plus a mocked `api.sendMessage`; current `main` follows the markdown branch and constructs one `sendMessage` payload instead of multiple chunked sends.

## Solution Assessment

Yes, fixing `extensions/telegram/src/send.ts` is the right boundary because direct helper callers and payload delivery can bypass core pre-chunking. The safest implementation is to route markdown/default text through the existing planner using rendered HTML plus the original/plain fallback text, rather than moving Telegram-specific behavior into core.
