## Summary

Keep open for maintainer reconciliation: current main still has loop detection disabled by default and still lacks read-offset recovery, but this PR is stale/mixed-scope and has correctness issues before it could merge.

## What This Changes

The PR flips tool-loop detection on by default, updates related tests/docs/schema metadata, and wraps read offset-beyond-EOF failures with a diagnostic result.

## Best Possible Solution

Split the work: land a canonical minimal loop-default change like #77491, then handle read offset-beyond-EOF in a separate adaptive-pager fix that preserves partial output.

## Reproduction Assessment

Yes, from source inspection. Current main still resolves omitted loop-detection config to enabled=false, and the read wrapper still delegates offset failures through adaptive paging without a read-aware recovery path.

## Solution Assessment

No. The goals are valid, but this PR is not the best merge shape because it mixes unrelated runtime changes, misses current config/doc/checksum/changelog surfaces, and handles read EOF recovery outside the adaptive pager where partial output can be lost.
