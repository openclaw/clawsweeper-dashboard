## Summary

Keep open: the goals are valid, but this PR is not merge-ready because it mixes the loop-detection default change with a separate read-offset change, misses current config/doc/changelog surfaces, and has a read recovery path that can discard already-read output.

## What This Changes

The PR enables tool-loop detection by default, updates related tests/docs/schema help, and adds read-tool handling for offset-beyond-EOF failures.

## Best Possible Solution

Land a focused loop-detection default change like #77491 or an equivalent rebased patch, and handle read offset-beyond-EOF separately inside adaptive paging with partial-output regression coverage if more recovery is still needed.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted loop-detection config still resolves to disabled, and the read wrapper still has no read-specific offset-beyond-EOF recovery path inside adaptive paging.

## Solution Assessment

No. The requested outcomes are reasonable, but this PR is not the best merge shape because it bundles unrelated runtime changes and does not align with the current run-scoped loop history, generated-doc checksum, changelog, and read-pager semantics.
