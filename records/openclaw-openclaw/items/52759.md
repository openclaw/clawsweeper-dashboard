## Summary

Keep open. Current main still ships loop detection disabled by default and still lacks read-aware out-of-range offset recovery, so the PR is not obsolete; however the branch is not merge-ready because it mixes two behavior changes, overlaps newer loop-default PRs, and still has correctness/docs/generated-artifact blockers.

## What This Changes

The PR flips `DEFAULT_LOOP_DETECTION_CONFIG.enabled` to true, updates related loop-detection tests/docs/schema metadata, commits generated config baseline artifacts, and wraps read-tool offset-beyond-EOF errors into a diagnostic result.

## Best Possible Solution

Use two focused, rebased changes if maintainers want this direction: one canonical loop-detection default change that preserves run-id scoping and aligns tests, exported type comments, schema help, generated SHA baselines, and current docs; and one separate read-offset recovery change inside adaptive paging with partial-output regression coverage.

## Reproduction Assessment

Yes. The loop-default behavior is reproducible by recording repeated read calls and calling `detectToolCallLoop` without config; current main still returns the disabled default. The read-offset path is reproducible by invoking `tools.read` with an offset past EOF; current main has no read-specific catch or diagnostic path.

## Solution Assessment

No. The safety goal may be valid, but this branch is not the best merge shape because it combines unrelated runtime changes, commits local-only generated artifacts, misses current config/doc surfaces, and handles EOF recovery outside the adaptive pager where partial output can be lost.
