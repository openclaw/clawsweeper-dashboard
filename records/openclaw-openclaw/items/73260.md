## Summary

Keep open: latest PR head still has a source-reproducible size-only oversize hash cache bypass, the required changelog entry is absent, and the external-PR real behavior proof gate is unmet.

## What This Changes

The PR replaces mtime-based models-config ready-cache inputs with content hashes, adds a models.json drift hash, hardens file handling, updates cache state/tests, and regenerates plugin SDK API baseline hashes.

## Best Possible Solution

Keep the content-hash direction, but make over-cap files force a miss or use a bounded content digest with size/truncation metadata, add same-size over-cap coverage, add the changelog entry, and require real behavior proof before merge.

## Reproduction Assessment

Yes, from source inspection of PR head: create an over-cap `models.json` or `auth-profiles.json`, warm the ready cache, then rewrite different content with the same byte length. `safeHashRegularFile` can return the same `oversize:<size>` value and preserve the cache hit; I did not run it because this review was read-only.

## Solution Assessment

No. The content-hash approach is the right direction, but a size-only oversize sentinel is not a maintainable drift-detection implementation; fail closed or hash bounded content plus size/truncation metadata instead.
