## Summary

Keep this PR open: the content-hash direction is useful and current `main` has not implemented it, but the PR still has a concrete over-cap cache-integrity bug and is missing the required changelog entry.

## What This Changes

The PR replaces mtime-based `ensureOpenClawModelsJson` cache inputs with stable auth-profile content hashes, adds a `modelsJsonHash` drift check, hardens file handling, and adds fingerprint-cache tests.

## Best Possible Solution

Keep the content-hash cache design, but make over-cap file handling content-sensitive or miss-forcing, add same-size over-cap regression coverage, and add a single-line changelog entry for the user-facing perf change.

## Reproduction Assessment

Yes. On PR head, warm the ready cache with an over-cap `models.json` or `auth-profiles.json`, then replace it with different content of the same byte length; the size-only sentinel can remain equal and preserve the cache hit.

## Solution Assessment

No. The proposed design is the right direction, but the deterministic `oversize:<size>` sentinel is not a maintainable integrity check for drift detection; failing closed on oversize is the narrowest safer alternative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Force a miss for oversized file hashes:** `src/agents/models-config.ts:115`
  - body: When a file is over the cap, `safeHashRegularFile()` returns `oversize:${lst.size}` without hashing any bytes. Since the cache hit path compares this value directly, a same-size rewrite of an oversized `models.json` or `auth-profiles.json` can keep the ready cache hot and skip re-planning. Return `null`, or include bounded content plus size/truncation metadata in the digest.
  - confidence: 0.9
- **[P3] Keep the no-follow fallback from chmoding symlink targets:** `src/agents/models-config.ts:334-337`
  - body: When `O_NOFOLLOW` is unavailable, this falls back to plain `O_RDONLY` and then `fh.chmod(0o600)` on whatever target was opened. Because this helper no longer performs an `lstat()` symlink check first, fallback platforms can still chmod a symlink target; either no-op without no-follow support or add a best-effort symlink/non-regular check before the fallback open.
  - confidence: 0.77
- **[P3] Add the required changelog entry:** `src/agents/models-config.ts:267`
  - body: This is a user-facing `perf` change: OAuth token refreshes and `models.json` writes should no longer repeatedly rerun provider discovery. Repo policy requires a `CHANGELOG.md` entry for user-facing perf/fix/feat changes, but this diff only updates source and tests.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR improves file handling but leaves concrete cache-integrity and fallback symlink-hardening gaps.

Concerns:

- **[medium] Size-only oversize hash can bypass drift detection:** `src/agents/models-config.ts:115`
  - body: For over-cap `models.json` or `auth-profiles.json`, the helper returns a deterministic `oversize:<size>` value without content. An external writer that changes an oversized file while preserving byte length can keep the cache hash unchanged and bypass the new re-plan trigger.
  - confidence: 0.9
- **[low] Fallback chmod can still follow symlink targets:** `src/agents/models-config.ts:334`
  - body: On platforms where `O_NOFOLLOW` is unavailable, the chmod helper opens with plain `O_RDONLY` and fchmods the opened target without a preceding `lstat()` rejection, leaving the fallback path vulnerable to the symlink-following behavior this PR is meant to harden.
  - confidence: 0.77
