## Summary

Keep open: the latest follow-up fixed the changelog gap, but the oversize handling still permits stale cache hits because oversized reads collapse to the same `null` value that the cache treats as stable. The PR also still needs after-fix real behavior proof from a real OpenClaw setup before merge.

## What This Changes

The PR replaces mtime-based models-config cache inputs with auth-profile content hashing, adds a stored `models.json` drift hash, hardens file reads/chmod, adds fingerprint-cache tests, updates the plugin SDK API hash baseline, and adds a changelog entry.

## Best Possible Solution

Keep the content-hash direction, but distinguish oversized/unhashable files from stable absence so over-cap files always miss or use bounded content metadata, add already-oversize regression coverage, then require real OpenClaw proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: warm the PR cache while `auth-profiles.json` or `models.json` is already over cap, then change its content while it remains over cap; both reads return `null`, so the cache can still hit.

## Solution Assessment

No. Returning `null` is not sufficient because the PR also treats stable `null` values as cache hits; the safer fix is an explicit always-miss oversize state or a bounded digest with size/truncation metadata.
