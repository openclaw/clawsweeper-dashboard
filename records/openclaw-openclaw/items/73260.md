## Summary

Keep open. Current main has not implemented the PR’s content-hash/drift-detection behavior, but the PR head still has blocking review issues: oversized files can preserve cache hits through a size-only hash sentinel, and the branch is stale against current main’s `providerDiscoveryEntriesOnly` startup path. Head checks also include failing `checks-node-core` and `checks-node-core-support-boundary`, so this needs revision before merge.

## What This Changes

This PR replaces mtime-based models-config ready-cache inputs with content hashes, stores a `modelsJsonHash` beside ready-cache entries, hardens `models.json` hashing/chmod paths, and adds fingerprint-cache tests.

## Best Possible Solution

Keep the content-hash direction, but revise the PR so oversized files cannot produce stable cache hits from length alone, preserve current main’s `providerDiscoveryEntriesOnly` option through the fingerprint and planning calls, add regression coverage for those paths, and get the failing checks green on a rebased head.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the main finding: create an over-cap `models.json` or `auth-profiles.json`, warm the ready cache, then rewrite different content with the same byte length; the PR head can return the same `oversize:<size>` value and keep the cache hot.

## Solution Assessment

No. The content-hash approach is the right maintainable direction, but the current patch is not the best implementation because over-cap files use a deterministic non-content hash and the branch no longer matches current main’s metadata-only provider-discovery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Force a miss for oversized file hashes:** `src/agents/models-config.ts:115`
  - body: When a file is already over the cap, `safeHashRegularFile()` returns `oversize:${lst.size}` without reading any content. Since cache hits compare this value to the captured `modelsJsonHash`/fingerprint input, a same-size edit to an oversized `models.json` or `auth-profiles.json` can keep the ready cache hot and skip re-planning. Return `null` for oversize or include bounded content in the digest so content drift cannot collide by size alone.
  - confidence: 0.9
- **[P2] Preserve providerDiscoveryEntriesOnly:** `src/agents/models-config.ts:411-416`
  - body: Current main threads `providerDiscoveryEntriesOnly` through `ensureOpenClawModelsJson` so startup prewarm can stay metadata-only. This branch’s replacement options/fingerprint/plan path omits that option, so rebasing or merging as-is would either fail current call sites that pass it or regress startup provider discovery back to runtime-loading behavior. Carry the option through the new content-hash path as current main does.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR improves several file-handling paths, but the size-only oversize hash is a concrete cache-integrity weakness in the proposed drift-detection mechanism.

Concerns:

- **[medium] Size-only oversize hash can bypass drift detection:** `src/agents/models-config.ts:115`
  - body: For over-cap `models.json` or `auth-profiles.json`, the helper returns a deterministic `oversize:<size>` value without content. An attacker or sibling process that can rewrite an oversized file to different content with the same length can preserve the cache hash and bypass the new re-plan trigger.
  - confidence: 0.88
