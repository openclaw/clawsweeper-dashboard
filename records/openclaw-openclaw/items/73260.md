## Summary

Keep open: current main does not already contain this content-hash cache behavior, and the PR still has a P2 cache-integrity bug plus missing real behavior proof before merge.

## What This Changes

The PR replaces models-config mtime cache inputs with auth-profile content hashing, adds a stored models.json drift hash, hardens models-config file handling, adds fingerprint-cache tests, and refreshes plugin SDK API baseline hashes.

## Best Possible Solution

Keep the content-hash direction, but make over-cap files force a miss or hash bounded content with size/truncation metadata, add regression coverage and the changelog entry, then require real behavior proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: warm the cache with an over-cap `models.json` or `auth-profiles.json`, rewrite different same-length content, and the PR can keep the same `oversize:<size>` hash. I did not execute it because this review was read-only.

## Solution Assessment

No. The content-hash approach is maintainable, but the size-only oversize sentinel undermines the promised drift detection; fail closed or use a bounded content digest instead.
