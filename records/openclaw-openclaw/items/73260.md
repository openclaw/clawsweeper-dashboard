## Summary

Keep open. Current main still uses the mtime-based cache fingerprint this PR is meant to replace, but the latest PR head still has a concrete cache-integrity problem: oversized files return a size-only sentinel, so same-size content drift can bypass the new hash comparison.

## What This Changes

The PR replaces mtime-based auth-profile and models.json cache fingerprinting with stable content hashes, stores a models.json hash in readyCache, hardens models.json file handling, and adds fingerprint-cache tests.

## Best Possible Solution

Keep the content-hash direction, but make over-cap files unable to produce stable cache hits from byte length alone, then add regression coverage for same-size over-cap edits to both models.json and auth-profiles.json.

## Reproduction Assessment

Yes. On PR head, create an over-cap models.json or auth-profiles.json, warm the ready cache, then rewrite different content with the same byte length; safeHashRegularFile can return the same oversize:<size> value and preserve the cache hit.

## Solution Assessment

No. The proposed content-hash design is the right direction, but a deterministic size-only sentinel is not the narrowest maintainable implementation for a drift-detection path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Force a miss for oversized file hashes:** `src/agents/models-config.ts:115`
  - body: When a file is over the cap, safeHashRegularFile() returns `oversize:${lst.size}` without hashing any bytes. Since the cache hit path compares this value directly, a same-size rewrite of an oversized models.json or auth-profiles.json can keep the ready cache hot and skip re-planning. Return null for oversize, or include bounded content plus size/truncation metadata in the digest.
  - confidence: 0.88
