## Summary

Keep open: this PR is still a meaningful candidate for the open mixed hook/tool plugin closure-state bug, but it is not merge-ready because the branch conflicts with current main, proof is only tests/checks, and the guard misses current live hook-registry surfaces.

## What This Changes

The PR skips cached plugin tool descriptors when an active or channel registry already contains the same plugin's hooks and tools, and adds a regression test for preserving hook/tool closure state.

## Best Possible Solution

Rebase or replace this PR with a resolver fix that uses the same hook-owner registry source set as current hook dispatch for mixed hook/tool plugins, then add focused regression coverage and redacted real behavior proof before landing.

Yes at source level: current main resolves cached plugin tools before the runtime path, and mixed hook/tool plugins are documented as supported. I did not run a live mem0-style gateway reproduction in this read-only review.
