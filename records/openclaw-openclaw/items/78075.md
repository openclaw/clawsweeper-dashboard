## Summary

Keep this PR open because it is still the active candidate for the linked plugin hook/tool closure-state bug, but it is not merge-ready: current main added a request-local `runtimeRegistry` path that this branch's bypass does not cover, and the PR still lacks real behavior proof.

## What This Changes

The PR changes plugin tool resolution to skip cached descriptors when a live active/channel registry already owns a plugin's hooks and tools, and adds regression coverage for that active-registry path.

## Best Possible Solution

Refresh the branch, make the cached-descriptor bypass honor every registry source used by `resolvePluginTools`, add real hook-and-tool plugin proof, and get maintainer confirmation of the intended plugin state contract.

## Reproduction Assessment

Yes at source level: current main resolves cached descriptor-backed plugin tools before it considers live runtime registries, which can split tool execution from hook registry state. I did not run a live mem0 or equivalent gateway reproduction in this read-only review.

## Solution Assessment

No as written: the PR is a plausible narrow mitigation, but it should include the request-local `runtimeRegistry` path that current `resolvePluginTools` supports. Maintainers still need to confirm whether same-register closure sharing is the intended plugin contract versus an explicit shared-state API.
