## Summary

Keep this PR open: the linked plugin closure-state bug is still present on current main, but the branch is not merge-ready because it misses the request-local registry path and still lacks after-fix real behavior proof.

## What This Changes

The PR skips cached plugin tool descriptors when global active/channel registries already own both hooks and tools for a plugin, and adds a regression test for that active-registry path.

## Best Possible Solution

Refresh the branch, make the cached-descriptor bypass honor every registry source resolvePluginTools can use, add redacted live proof from a real hook-and-tool plugin setup, and land only after maintainers confirm the intended plugin state contract.

## Reproduction Assessment

Yes, at source level: current main can return cached descriptor-backed tools before the live registry path while hooks remain on registry entries. I did not run a live mem0 or equivalent gateway reproduction in this read-only review.

## Solution Assessment

No as written: the PR is a plausible narrow mitigation, but it should include the request-local runtimeRegistry path that resolvePluginTools already supports. Maintainers still need to confirm whether closure sharing is the intended plugin contract versus an explicit shared-state API.
