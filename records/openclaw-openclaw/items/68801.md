## Summary

Keep open: the orphan-cleanup problem is plausible, but this patch is not safe because it prunes `agentRunStarts` by the 10-minute cache TTL even though active runs can run much longer and terminal lifecycle events may rely on that map for `startedAt`. The external PR also still lacks inspectable after-fix real behavior proof.

## What This Changes

The PR adds a TTL sweep for `agentRunStarts` inside `pruneAgentRunCache()` so orphaned run-start timestamps are deleted after the existing agent-run cache TTL.

## Best Possible Solution

Use lifecycle-aware cleanup that drains true orphan starts without deleting timestamps for active runs, with focused fake-timer coverage for both orphan cleanup and long active runs crossing the cache TTL.

## Reproduction Assessment

Yes. By source inspection against the PR diff, a lifecycle `start` with `startedAt`, a prune after the 10-minute TTL, and a later terminal `end` without `startedAt` loses the original timestamp.

## Solution Assessment

No. Reusing the completed snapshot cache TTL for active-run starts is not the narrowest maintainable fix because active runs can last up to 48 hours and terminal events often rely on the fallback map.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve start timestamps for active runs:** `src/gateway/server-methods/agent-job.ts:39-42`
  - body: This loop deletes `agentRunStarts` entries based only on age, but that map is also the fallback `startedAt` source for terminal lifecycle events that omit it. Active runs can exceed the 10-minute cache TTL, so a later `end` or `error` for a still-active run can make `agent.wait` return a snapshot without its original `startedAt`.
  - confidence: 0.94
