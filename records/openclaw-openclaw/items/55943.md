## Label Justifications

- `impact:session-state`: The requested behavior is about missing session-memory artifacts for completed subagent sessions.

## Summary

Keep open. Current main and v2026.6.6 still keep bundled session-memory on `/new` and `/reset`, and current subagent tests intentionally avoid `subagent_ended` for persistent session-mode completion, so this remains a real lifecycle/product-contract gap rather than cleanup-safe stale work.

## What This Changes

The issue requests memory capture when a subagent session completes, not only when a main session is reset or replaced.

## Best Possible Solution

Maintainers should choose the canonical lifecycle contract for completed persistent subagent sessions, then wire session-memory through that contract with focused coverage.

## Reproduction Assessment

Yes from source, but not as a live runtime repro in this read-only review: current session-memory only handles `/new` and `/reset`, and current tests explicitly suppress `subagent_ended` for persistent session-mode completion.

## Solution Assessment

Unclear. The requested behavior is useful, but the best fix requires deciding whether persistent subagent completion should fire a generic lifecycle event, a subagent event, or remain separate until explicit reset/delete.
