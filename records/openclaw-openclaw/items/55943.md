## Summary

Keep open. Current main and v2026.6.10 still keep bundled session-memory scoped to `command:new` and `command:reset`, while completed subagents emit the separate `subagent_ended` plugin lifecycle hook; maintainers still need to choose the lifecycle contract before wiring memory persistence.

## What This Changes

The issue asks for completed subagent sessions to trigger session-memory output under `memory/` the same way reset/new session boundaries do.

## Best Possible Solution

Define one canonical completed-child-session lifecycle contract, then wire session-memory through that path with focused tests for transcript lookup, workspace resolution, and duplicate-write prevention.

## Reproduction Assessment

Yes, at source level: the bundled session-memory handler accepts only `command:new` and `command:reset`, while completed subagents emit a separate `subagent_ended` plugin lifecycle event. I did not run a live filesystem reproduction in this read-only review.

## Solution Assessment

Unclear. Directly special-casing `subagent_ended` in session-memory is plausible, but the maintainable fix should settle the session/subagent lifecycle contract first instead of adding a parallel one-off completion path.
