## Label Justifications

- `P2`: This is a normal-priority lifecycle/memory persistence gap with clear source evidence and limited blast radius.
- `impact:session-state`: The issue concerns missing transcript-derived memory artifacts for completed child agent sessions.

## Summary

Keep open. Current main and v2026.5.20 still keep bundled session-memory scoped to `/new` and `/reset`, while subagent completion emits the typed `subagent_ended` plugin hook; the related open PR does not update bundled session-memory itself, so this specific memory persistence gap is not implemented yet.

## What This Changes

Request that completed subagent sessions trigger session-memory persistence so child session memory files are written under `memory/`.

## Best Possible Solution

Choose one supported lifecycle contract, then persist the child session transcript to the correct agent workspace exactly once while preserving the existing `/new` and `/reset` behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable bundled session-memory, complete a `sessions_spawn` child, and current main emits `subagent_ended` while session-memory accepts only `command:new` and `command:reset`; I did not run a live subagent session in this read-only review.

## Solution Assessment

Unclear. Directly teaching session-memory about `subagent_ended` may work, but maintainers should first choose whether the durable contract is internal `session:end`, typed `session_end`, or `subagent_ended`.
