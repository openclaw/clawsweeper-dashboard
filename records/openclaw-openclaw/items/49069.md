## Best Possible Solution

Land a revised implementation after maintainers choose the accepted config, CLI, slash-command, or settings surface; carry timestamp data through pending, live, streaming, final, and history replay paths with focused tests and real TUI proof.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict config schema has no `tui.showTimestamps`, current `theme` has no `muted`, and assistant update/final paths replace prefixed text. The requested timestamp display itself is a feature request, not a broken existing behavior on current main.

## Solution Assessment

No. Prefixing rendered strings from `Date.now()` while reading an undeclared config key is not the best shape; the maintainable path is an accepted settings/toggle contract plus timestamp data carried through live, history, pending, streaming, and final render paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Define the timestamp config contract:** `src/tui/tui.ts:544-547`
  - body: The PR reads `config.tui.showTimestamps`, but `OpenClawSchema` is strict and has no top-level `tui` block. A user who follows the PR body's config example would get an invalid config error instead of enabling timestamps, so this needs an accepted schema/types/help/docs surface or a different supported toggle before merge.
  - confidence: 0.97
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:35`
  - body: `theme.muted` is not exported by `src/tui/theme/theme.ts`; the current formatter is `theme.dim`. As soon as timestamp rendering is reachable, the system/user timestamp paths can throw or fail typechecking instead of rendering the TUI.
  - confidence: 0.98
- **[P2] Preserve assistant timestamps through updates:** `src/tui/components/chat-log.ts:100-102`
  - body: Only the initial `startAssistant` text is prefixed, while `updateAssistant` and `finalizeAssistant` still replace the component with raw text. Streaming responses will lose the timestamp after the first update, and fallback finalization never gets a prefix.
  - confidence: 0.95
- **[P2] Render stored message times instead of reload time:** `src/tui/components/chat-log.ts:9-13`
  - body: `formatTimestamp()` always uses the local clock, but history replay currently passes only message text into ChatLog. Reopening a session would label old messages with the time the TUI loaded, which is misleading for the main use case this feature targets.
  - confidence: 0.9
