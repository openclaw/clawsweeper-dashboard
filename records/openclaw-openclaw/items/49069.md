## Summary

Keep this PR open. Current main still does not provide configurable user-facing timestamps in the terminal TUI, and the linked feature request remains open, but the submitted patch is not merge-ready because the proposed config key is not schema-backed, it calls a formatter that current `theme` does not export, and it loses or misstates timestamps in assistant streaming and history replay paths.

## What This Changes

The PR adds a Date-based timestamp prefix helper and `showTimestamps` switch to `ChatLog`, then reads `tui.showTimestamps` from config during TUI startup.

## Best Possible Solution

Revise this PR or replace it with a narrow implementation that adds a typed and documented TUI timestamp setting or toggle, renders timestamp prefixes through `ChatLog` with existing theme formatters, preserves prefixes during assistant streaming/final updates, uses persisted transcript timestamps during history replay, and covers the behavior with focused TUI tests.

## Reproduction Assessment

Yes. The failure path is source-reproducible: following the PR's documented config shape hits strict config validation before TUI startup, and forcing the flag on would exercise `theme.muted` plus raw assistant `setText` updates that drop prefixes.

## Solution Assessment

No. The user-facing direction is reasonable, but this patch is not the best maintainable fix because it bypasses the config contract and treats timestamps as render-time decoration instead of data that must survive streaming and replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add showTimestamps to the config contract:** `src/tui/tui.ts:541-545`
  - body: The new startup code reads `config.tui.showTimestamps`, but the root config schema is strict and has no `tui` key. Users who follow the PR's usage example will get `INVALID_CONFIG` before the TUI can start, so the option needs schema/types/help/docs support or a different existing settings surface.
  - confidence: 0.94
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:33`
  - body: With timestamps enabled this path calls `theme.muted(...)`, but current `theme` exports `dim` and not `muted`. That makes the patch fail type checking and can throw as soon as a timestamped message renders.
  - confidence: 0.98
- **[P2] Keep assistant timestamps through stream updates:** `src/tui/components/chat-log.ts:87-90`
  - body: The patch only prefixes the assistant text when the component is created. `updateAssistant` and `finalizeAssistant` later replace the component text with the raw response body, so streaming responses lose the timestamp after the first update.
  - confidence: 0.92
- **[P2] Use stored timestamps when replaying history:** `src/tui/components/chat-log.ts:7-14`
  - body: `formatTimestamp()` always uses the current local clock, while replayed transcript messages already carry timestamps. Reopening a session would relabel old messages with the time history was loaded unless `ChatLog` accepts and formats the message timestamp from history.
  - confidence: 0.86
