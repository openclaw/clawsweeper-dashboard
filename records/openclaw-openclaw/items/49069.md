## Summary

Keep open: current main still lacks user-facing TUI chat timestamps, and this external PR has blocking correctness issues plus missing real behavior proof before it can merge.

## What This Changes

The branch adds a `tui.showTimestamps` config read and prefixes system, user, and initial assistant ChatLog messages with `[HH:MM]` timestamps.

## Best Possible Solution

Land a rebased implementation that defines the config or toggle contract, carries stored timestamps through live, pending, streaming, final, and replay paths, and adds focused TUI/config coverage.

## Reproduction Assessment

Yes for the PR blockers via source inspection: the strict config schema lacks `tui`, `theme.muted` is not exported, and streaming/final assistant updates replace text with raw content. For the requested feature, current main source also confirms there is no TUI timestamp rendering yet.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best maintainable solution until it extends the accepted settings contract and preserves timestamp data separately from transient rendered text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add the timestamp setting to the config contract:** `src/tui/tui.ts:544-547`
  - body: The patch reads `config.tui.showTimestamps`, but the current config type has no top-level `tui` object and the root Zod schema is strict. Users following the PR usage example would fail config validation before the TUI starts, so this needs schema/types/help/docs support or a different accepted surface.
  - confidence: 0.95
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:34`
  - body: When timestamps are enabled this calls `theme.muted(...)`, but current `theme` exports `dim` and no `muted`. That is a type error and would throw as soon as a timestamped system or user message renders.
  - confidence: 0.98
- **[P2] Preserve assistant prefixes during streaming updates:** `src/tui/components/chat-log.ts:98-100`
  - body: Only `startAssistant` adds the prefix. `updateAssistant` and `finalizeAssistant` still replace the component with raw text, so streamed replies lose the timestamp after the first update and fallback finals never get one.
  - confidence: 0.94
- **[P2] Use persisted message times for history replay:** `src/tui/components/chat-log.ts:9-14`
  - body: `formatTimestamp()` always reads the local clock, while history messages already carry timestamps before `loadHistory` reduces them to text. Reopening a session would relabel old messages with the load time instead of the original message time.
  - confidence: 0.88
