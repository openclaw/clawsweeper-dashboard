## Summary

Keep open: this is a plausible user-facing TUI feature, but the current PR still has blocking correctness issues, lacks contributor-owned real behavior proof, and needs maintainer agreement on the new timestamp config/UX contract before merge.

## What This Changes

The branch modifies ChatLog and runTui to read `tui.showTimestamps` from config and prepend `[HH:MM]` to system, user, and initial assistant messages.

## Best Possible Solution

Land a rebased implementation that defines the user-facing config or settings contract, formats stored message timestamps through live, pending, streaming, final, and replay paths, and adds focused TUI/config coverage.

## Reproduction Assessment

Yes for the PR blockers via source inspection: the diff reads an undefined strict-config key, calls missing `theme.muted`, and prefixes only the initial assistant component before raw streaming/final updates replace it. I did not execute the TUI in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best maintainable solution until it extends the config contract and carries real stored timestamps through streaming and history replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add the timestamp setting to the config contract:** `src/tui/tui.ts:544-547`
  - body: The patch reads `config.tui.showTimestamps`, but the current config type has no top-level `tui` object and the root schema is strict. Users following the PR usage example would fail config validation before the TUI starts, so this needs schema/types/help/docs support or a different existing surface.
  - confidence: 0.95
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:38`
  - body: When timestamps are enabled, this path calls `theme.muted(...)`, but current `theme` exports `dim` and no `muted`. That is a type error and would throw as soon as a timestamped system or user message renders.
  - confidence: 0.98
- **[P2] Preserve assistant prefixes during streaming updates:** `src/tui/components/chat-log.ts:99-101`
  - body: Only `startAssistant` adds the prefix. `updateAssistant` and `finalizeAssistant` still replace the component with raw text, so streamed replies lose the timestamp after the first update and fallback finals never get one.
  - confidence: 0.94
- **[P2] Use persisted message times for history replay:** `src/tui/components/chat-log.ts:9-14`
  - body: `formatTimestamp()` always reads the local clock, while history messages already carry timestamps before `loadHistory` reduces them to text. Reopening a session would relabel old messages with the load time instead of the original message time.
  - confidence: 0.88
