## Summary

Keep this PR open for human follow-up, but it is not merge-ready: current main still lacks TUI message timestamps, while this branch adds a new undeclared config surface and has source-visible rendering defects plus no real behavior proof.

## What This Changes

The PR adds a `showTimestamps` flag to `ChatLog` and reads `config.tui.showTimestamps` in `runTui` to prefix TUI chat messages with local `[HH:MM]` timestamps.

## Best Possible Solution

Land a revised implementation only after maintainers choose the accepted config, CLI, slash-command, or settings surface, then carry message timestamps as data through live, pending, streaming, final, and history replay paths with focused coverage and real TUI proof.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict schema lacks `tui.showTimestamps`, the theme object lacks `muted`, and assistant updates still overwrite prefixed text. The underlying timestamp display request is a feature request rather than a broken existing behavior.

## Solution Assessment

No. Prefixing rendered strings from `Date.now()` while reading an undeclared config key is not the best shape; the maintainable path is an accepted settings/toggle contract plus timestamp data carried through live, history, pending, streaming, and final render paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Define the timestamp config contract:** `src/tui/tui.ts:542-547`
  - body: The PR reads `config.tui.showTimestamps`, but `OpenClawSchema` is strict and has no top-level `tui` block. A user who follows the PR body's config example would get an invalid config error instead of enabling timestamps, so this needs an accepted schema/types/help/docs surface or a different supported toggle before merge.
  - confidence: 0.97
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:34`
  - body: `theme.muted` is not exported by `src/tui/theme/theme.ts`; the current formatter is `theme.dim`. As soon as timestamp rendering is reachable, the system/user timestamp paths can throw or fail typechecking instead of rendering the TUI.
  - confidence: 0.98
- **[P2] Preserve assistant timestamps through updates:** `src/tui/components/chat-log.ts:98-100`
  - body: Only the initial `startAssistant` text is prefixed, while `updateAssistant` and `finalizeAssistant` still replace the component with raw text. Streaming responses will lose the timestamp after the first update, and fallback finalization never gets a prefix.
  - confidence: 0.95
- **[P2] Render stored message times instead of reload time:** `src/tui/components/chat-log.ts:9-13`
  - body: `formatTimestamp()` always uses the local clock, but history replay already loads persisted messages and currently passes only text into ChatLog. Reopening a session would label old messages with the time the TUI loaded, which is misleading for the main use case this feature targets.
  - confidence: 0.9
