## Summary

Keep open: current main still lacks user-facing TUI chat timestamps, but this PR is not mergeable as written because the config key is not part of the strict config contract, the patch calls a missing theme formatter, assistant/history paths lose or misstate timestamps, and no real TUI behavior proof is present.

## What This Changes

The PR adds an opt-in `tui.showTimestamps` config read and timestamp prefixes for TUI system, user, and initial assistant chat messages.

## Best Possible Solution

Land a rebased TUI implementation that defines the accepted config or toggle contract, preserves stored timestamps through live, pending, streaming, final, and history replay paths, and adds focused config plus ChatLog coverage.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict config contract rejects the proposed key, `theme.muted` is absent, and assistant update/final paths replace prefixed content with raw text. For the requested feature itself, reproduction is not applicable because current main has no TUI display timestamp capability.

## Solution Assessment

No. Optional timestamps are a reasonable feature direction, but this patch is not the best maintainable solution until it extends an accepted settings/toggle contract and stores timestamp data separately from transient rendered text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add the timestamp setting to the config contract:** `src/tui/tui.ts:544-547`
  - body: The patch reads `config.tui.showTimestamps`, but current `OpenClawConfig` has no top-level `tui` object and the root schema is strict. Users following the PR's usage example would fail config validation before the TUI starts, so this needs schema/types/help/docs support or a different accepted setting surface.
  - confidence: 0.95
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:34`
  - body: When timestamps are enabled this calls `theme.muted(...)`, but current `theme` exports `dim` and no `muted`. That is a type error and would throw as soon as a timestamped system or user message renders.
  - confidence: 0.98
- **[P2] Preserve assistant prefixes during streaming updates:** `src/tui/components/chat-log.ts:98-100`
  - body: Only `startAssistant` adds the prefix. `updateAssistant` and `finalizeAssistant` still replace the component with raw text, so streamed replies lose the timestamp after the first update and fallback finals never get one.
  - confidence: 0.94
- **[P2] Use persisted message times for history replay:** `src/tui/components/chat-log.ts:9-14`
  - body: `formatTimestamp()` always reads the local clock, while history messages already carry timestamps before `loadHistory` reduces them to text. Reopening a session would relabel old messages with the load time instead of the original message time.
  - confidence: 0.88
