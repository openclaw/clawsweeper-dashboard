## Summary

Keep open: current main still lacks TUI message timestamping, and this PR remains an open implementation tied to https://github.com/openclaw/openclaw/issues/48708, but the branch still has blocking config/rendering defects and missing real behavior proof.

## What This Changes

The PR adds an opt-in `tui.showTimestamps` config read and prefixes TUI system, user, and initial assistant messages with `[HH:MM]`.

## Best Possible Solution

Land a revised timestamp implementation that defines the accepted setting or toggle contract, renders timestamp chrome without corrupting message content, preserves real message times through live, streaming, pending, final, and history paths, and adds focused TUI/config coverage.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict config contract has no `tui` key, current `theme` has no `muted`, and assistant update/final paths replace prefixed content with raw text. The requested timestamp display itself is a feature request, not a broken existing behavior on current main.

## Solution Assessment

No. Optional timestamps are a reasonable direction, but this patch reads an unaccepted config key and attaches timestamps to transient rendered text instead of carrying message timestamp data through renderer paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add the timestamp setting to the config contract:** `src/tui/tui.ts:544-547`
  - body: The patch reads `config.tui.showTimestamps`, but current `OpenClawSchema`/`OpenClawConfig` do not define a top-level `tui` object and config validation is strict. Users following the PR usage example would fail config validation before the TUI starts, so this needs schema/types/help/docs support or a different accepted setting surface.
  - confidence: 0.95
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:35`
  - body: When timestamps are enabled this calls `theme.muted(...)`, but current `theme` exports `dim` and no `muted`. That is a type error and would throw as soon as a timestamped system or user message renders.
  - confidence: 0.98
- **[P2] Preserve assistant timestamps during streaming updates:** `src/tui/components/chat-log.ts:98-100`
  - body: Only `startAssistant` adds the prefix. `updateAssistant` and `finalizeAssistant` still replace the component with raw text, so streamed replies lose the timestamp after the first update and fallback finals never get one.
  - confidence: 0.94
- **[P2] Use persisted message times for history replay:** `src/tui/components/chat-log.ts:10-13`
  - body: `formatTimestamp()` always reads the local clock, while history messages already carry timestamps before `loadHistory` reduces them to text. Reopening a session would relabel old messages with the load time instead of the original message time.
  - confidence: 0.88
- **[P3] Keep timestamp styling outside message theme wrappers:** `src/tui/components/chat-log.ts:78`
  - body: `addSystem` wraps a prefixed string in `theme.system(...)`, so once the undefined formatter is fixed, the outer system color can still override timestamp styling. Render the timestamp prefix consistently outside the body theme.
  - confidence: 0.78
