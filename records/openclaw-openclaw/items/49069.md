## Summary

Keep open: current main still has no TUI message timestamp option, and this PR’s implementation has blocking config and rendering defects before it can merge.

## What This Changes

This PR adds an opt-in `tui.showTimestamps` setting that prepends `[HH:MM]` timestamps to TUI chat messages.

## Best Possible Solution

Land a narrow rebased implementation that defines the TUI timestamp setting in the config contract/help/docs, preserves real message timestamps through live, pending, streaming, final, and replay paths, and adds focused TUI/config coverage.

## Reproduction Assessment

Yes for the PR blockers. Source inspection shows the PR reads an unknown strict-config key, calls a non-existent theme formatter, and prefixes assistant text only before later raw `setText` updates replace it.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best maintainable solution because timestamp display needs a typed config contract and timestamp data that survives streaming and history replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add showTimestamps to the config contract:** `src/tui/tui.ts:541-545`
  - body: The new startup code reads `config.tui.showTimestamps`, but the root config schema is strict and has no `tui` key. Users following the documented example will hit invalid-config handling before the TUI can start, so the setting needs schema/types/help/docs support or a different existing config surface.
  - confidence: 0.94
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:31-34`
  - body: With timestamps enabled this path calls `theme.muted(...)`, but current `theme` exports `dim` and not `muted`. That makes the patch fail type checking and can throw as soon as a timestamped message renders.
  - confidence: 0.98
- **[P2] Keep assistant timestamps through stream updates:** `src/tui/components/chat-log.ts:87-90`
  - body: The patch only prefixes the assistant text when the component is created. `updateAssistant` and `finalizeAssistant` later replace the component text with the raw response body, so streaming responses lose the timestamp after the first update.
  - confidence: 0.93
- **[P2] Use stored timestamps when replaying history:** `src/tui/components/chat-log.ts:7-14`
  - body: `formatTimestamp()` always uses the current local clock, while replayed transcript messages already carry timestamps. Reopening a session would relabel old messages with the time history was loaded unless the TUI passes and formats the message timestamp from history.
  - confidence: 0.87
- **[P2] Add the required changelog entry:** `src/tui/tui.ts:541-545`
  - body: This is a user-facing `feat`, but the PR only changes TUI code. Repo policy requires a single-line `CHANGELOG.md` entry before merge so the new timestamp option is visible in release notes.
  - confidence: 0.9
