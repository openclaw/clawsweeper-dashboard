## Summary

Keep open: current main does not implement user-facing TUI message timestamps, the linked canonical feature issue remains open, and this PR still has blocking source-level defects plus missing real behavior proof.

## What This Changes

The PR adds an opt-in `tui.showTimestamps` config read and prefixes TUI chat messages with `[HH:MM]` when enabled.

## Best Possible Solution

Land a revised timestamp implementation that defines the accepted setting or toggle contract, preserves real message times through live, pending, streaming, final, and history replay paths, and adds focused TUI/config coverage.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict config contract has no accepted `tui.showTimestamps` key, current `theme` has no `muted`, and assistant update/final paths replace prefixed content with raw text. The requested timestamp display itself is a feature request, not a broken existing behavior on current main.

## Solution Assessment

No. Optional TUI timestamps are a reasonable direction, but this patch reads an unaccepted config key and attaches timestamps to transient rendered text instead of carrying timestamp data through the renderer and history paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Define the timestamp config contract:** `src/tui/tui.ts:544-547`
  - body: The patch reads `config.tui.showTimestamps`, but current config validation is strict and there is no top-level `tui` object in `OpenClawConfig` or `OpenClawSchema`. Users following the PR body example would fail config validation before the TUI starts, so this needs schema/types/help/docs support or a different accepted setting surface.
  - confidence: 0.95
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:35`
  - body: When timestamps are enabled this calls `theme.muted(...)`, but current `theme` exports `dim` and no `muted`. That is a type error and would throw as soon as a timestamped system or user message renders.
  - confidence: 0.98
- **[P2] Preserve assistant timestamps during streaming updates:** `src/tui/components/chat-log.ts:98-100`
  - body: Only `startAssistant` adds the prefix. `updateAssistant` and `finalizeAssistant` still replace the component with raw text, so streamed replies lose the timestamp after the first update and fallback finals never get one.
  - confidence: 0.94
- **[P2] Use persisted message times for history replay:** `src/tui/components/chat-log.ts:9-14`
  - body: `formatTimestamp()` always reads the local clock, while history messages already carry timestamps before `loadHistory` reduces them to text. Reopening a session would relabel old messages with the load time instead of the original message time.
  - confidence: 0.88
- **[P3] Keep timestamp styling outside message theme wrappers:** `src/tui/components/chat-log.ts:78`
  - body: `addSystem` wraps a prefixed string in `theme.system(...)`, so after the undefined formatter is fixed the outer system color can still override timestamp styling. Render the timestamp prefix consistently outside the body theme.
  - confidence: 0.78
