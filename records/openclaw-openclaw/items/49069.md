## Summary

Keep open: current main still does not implement TUI chat timestamps, but this PR is not merge-ready because it adds an unsupported config surface, calls a nonexistent theme formatter, loses timestamps across assistant updates/history paths, is conflicting, and lacks real behavior proof.

## What This Changes

The PR adds an opt-in TUI timestamp toggle that prefixes chat messages with `[HH:MM]` through `ChatLog` and reads `tui.showTimestamps` during TUI startup.

## Best Possible Solution

Define the accepted TUI timestamp UX/config contract, carry real message timestamps through live, pending, streaming, final, and history replay rendering, add focused schema/TUI coverage, and attach redacted real TUI proof before merge.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the branch reads an unsupported strict config key, calls `theme.muted`, and prefixes only the initial assistant text while current update/final/history paths use raw text. The underlying timestamp display request is a feature request, not a broken existing behavior reproduction.

## Solution Assessment

No. Optional TUI timestamps are reasonable, but this patch is not the best implementation because it uses transient string prefixes and local clock time instead of a defined contract and persisted message timestamps across render paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Define the timestamp config contract before reading it:** `src/tui/tui.ts:542-545`
  - body: The PR body tells users to set `tui.showTimestamps`, but current `OpenClawSchema` is root-strict and has no `tui` key. Following the documented usage can fail config validation before the TUI starts, so this needs an accepted schema/types/help/docs update or a different toggle surface.
  - confidence: 0.96
- **[P1] Use an existing theme formatter for timestamps:** `src/tui/components/chat-log.ts:33`
  - body: `theme.muted(...)` is not part of the current theme object; the first timestamped system or user render can fail typechecking or throw at runtime. Use a defined formatter such as `theme.dim` and keep system/user styling order explicit.
  - confidence: 0.97
- **[P2] Preserve assistant timestamps after updates:** `src/tui/components/chat-log.ts:98-100`
  - body: The branch prefixes only the initial `AssistantMessageComponent`, but `updateAssistant` and `finalizeAssistant` replace the component text with raw text, and the fallback final path creates an unprefixed component. Streaming assistant messages will lose or skip timestamps unless the timestamp is stored and reapplied on every update path.
  - confidence: 0.94
- **[P2] Use stored message times for history replay:** `src/tui/components/chat-log.ts:9-13`
  - body: `formatTimestamp()` always uses the current local clock, while `chat.history` rows already carry message timestamps. Replaying history through this path would relabel old messages with the time the TUI loaded instead of when the messages were sent.
  - confidence: 0.91
