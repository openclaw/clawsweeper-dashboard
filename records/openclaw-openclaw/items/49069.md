## Summary

Keep open: this PR is a plausible implementation candidate for the open TUI timestamp feature, but it is not merge-ready because the proposed config key is not part of the strict config contract, the rendering code calls an undefined theme formatter, assistant timestamps are lost on update/final paths, and contributor real behavior proof is still missing.

## What This Changes

The PR adds an opt-in `tui.showTimestamps` read and prefixes TUI system, user, and initial assistant chat messages with `[HH:MM]`.

## Best Possible Solution

Land a revised implementation that chooses an accepted config, CLI, slash-command, or settings surface, carries message timestamps as data through live, pending, streaming, final, and history replay paths, and includes focused TUI/config coverage plus real TUI proof.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict config schema has no `tui.showTimestamps`, current `theme` has no `muted` formatter, and ChatLog update/final paths replace prefixed assistant text with raw text. The requested timestamp display itself is a feature request, not a broken existing behavior on current main.

## Solution Assessment

No. Prefixing rendered strings from `Date.now()` and reading an undeclared config key is not the best shape; the maintainable path is an accepted settings/toggle contract plus timestamp data carried through live, history, pending, streaming, and final render paths.

- **[P1] Define the timestamp config contract:** `src/tui/tui.ts:542-547`
  - body: The PR reads `config.tui.showTimestamps`, but `OpenClawSchema` is strict and has no top-level `tui` block.   - body: `theme.muted` is not exported by `src/tui/theme/theme.ts`; the current formatter is `theme.dim`. As soon as timestamp rendering is reachable, the system/user timestamp paths can throw or fail typechecking instead of rendering the TUI.
  - confidence: 0.98
- **[P2] Preserve assistant timestamps through updates:** `src/tui/components/chat-log.ts:98-100`
  - body: Only the initial `startAssistant` text is prefixed, while `updateAssistant` and `finalizeAssistant` still replace the component with raw text. Streaming responses will lose the timestamp after the first update, and fallback finalization never gets a prefix.
  - confidence: 0.95
- **[P2] Render stored message times instead of reload time:** `src/tui/components/chat-log.ts:9-13`
  - body: `formatTimestamp()` always uses the local clock, but history replay currently passes only message text into `ChatLog`. Reopening a session would label old user/system/assistant messages with the time the TUI loaded, which is misleading for the main use case this feature targets.
  - confidence: 0.9
