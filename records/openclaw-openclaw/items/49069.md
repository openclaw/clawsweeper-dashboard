## Summary

Keep open: current main still does not implement user-facing TUI chat timestamps, but this PR is not mergeable as written because the proposed config key is outside the accepted config contract, the timestamp rendering calls a missing theme formatter, assistant/history paths lose or misstate timestamps, and contributor-owned real behavior proof is missing.

## What This Changes

Adds an opt-in `tui.showTimestamps` config read and prefixes TUI system, user, and assistant messages with `[HH:MM]` timestamps.

## Best Possible Solution

Land a revised TUI timestamp implementation that defines the accepted setting or toggle contract, carries stored timestamps through live, pending, streaming, final, and history replay paths, and adds focused TUI/config coverage.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the proposed config key is outside the strict schema, `theme.muted` is absent, and assistant/history paths replace or invent timestamp text. The requested timestamp display itself is a feature request, not a broken existing behavior reproduction.

## Solution Assessment

No. Optional TUI timestamps are a reasonable direction, but this patch is not the narrowest maintainable solution until the setting/toggle contract is accepted and timestamps are carried as message data instead of transient rendered text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add the timestamp setting to the config contract:** `src/tui/tui.ts:544-547`
  - body: The patch reads `config.tui.showTimestamps`, but current config validation has no top-level `tui` object and the root schema is strict. Users following the PR usage example would fail config validation before the TUI starts, so this needs schema/types/help/docs support or a different accepted setting surface.
  - confidence: 0.95
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:34`
  - body: When timestamps are enabled this calls `theme.muted(...)`, but current `theme` exports `dim` and no `muted`. That is a type error and would throw as soon as a timestamped system or user message renders.
  - confidence: 0.98
- **[P2] Preserve assistant prefixes across stream updates:** `src/tui/components/chat-log.ts:98-100`
  - body: Only `startAssistant` adds the prefix. `updateAssistant` and `finalizeAssistant` still replace the component with raw text, so streamed replies lose the timestamp after the first update and fallback finals never get one.
  - confidence: 0.94
- **[P2] Use persisted message times for history replay:** `src/tui/components/chat-log.ts:10-14`
  - body: `formatTimestamp()` always reads the local clock, while history messages already carry timestamps before `loadHistory` reduces them to text. Reopening a session would relabel old messages with the load time instead of the original message time.
  - confidence: 0.88
