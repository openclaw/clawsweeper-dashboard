## Summary

Keep open; this is a valid feature PR for open #48708, but it cannot merge because the proposed config is rejected by the strict schema, enabled timestamps call a non-existent theme formatter, and assistant/history paths would render disappearing or misleading timestamps.

## What This Changes

The PR adds an opt-in `tui.showTimestamps` setting read in the TUI and prefixes ChatLog user/system/assistant message text with `[HH:MM]`.

## Best Possible Solution

Repair this PR against current main with typed config/schema/help/docs/changelog coverage plus ChatLog APIs that render stable per-message timestamps from live event and history metadata.

## Reproduction Assessment

Yes, from source inspection. Enabling the documented config reaches `theme.muted`, which does not exist, and validation rejects the strict-schema `tui` key; assistant timestamp loss follows from `updateAssistant` and `finalizeAssistant` replacing raw text.

## Solution Assessment

No. The patch decorates strings at the edges and reads an unschematized config key; the maintainable fix is a typed config surface plus timestamp-aware ChatLog rendering that survives stream updates and history replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use an existing theme formatter for timestamps:** `src/tui/components/chat-log.ts:35`
  - body: With timestamps enabled this calls `theme.muted(...)`, but the TUI theme exports `dim`, `system`, and related formatters, not `muted`. The same undefined call is used for user messages, so the advertised option can fail typecheck and throw before the chat log renders.
  - confidence: 0.98
- **[P1] Add the timestamp option to the config schema:** `src/tui/tui.ts:544-547`
  - body: This reads `config.tui.showTimestamps`, but the root OpenClaw config schema is strict and has no `tui` key. A user following the PR's documented config will get an invalid-config failure before the TUI can start, so the feature needs the normal schema/types/help/generated-doc coverage or a supported setting surface.
  - confidence: 0.97
- **[P2] Preserve assistant prefixes through stream updates:** `src/tui/components/chat-log.ts:100-102`
  - body: The timestamp is only baked into the initial assistant component. Later `updateAssistant` and `finalizeAssistant` calls replace the component text with raw `text`, and the no-existing-run final path appends raw text, so streamed assistant messages lose or skip the timestamp.
  - confidence: 0.96
- **[P2] Use transcript times when replaying history:** `src/tui/components/chat-log.ts:9-13`
  - body: `formatTimestamp()` always formats the current client clock, but history replay passes old messages through the same `addUser`/`addSystem`/`finalizeAssistant` methods. Reopening a session would label prior messages with the load minute instead of the message timestamp already available in history.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/tui/tui.ts:544-547`
  - body: This is a user-facing TUI feature and new config surface, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.86
