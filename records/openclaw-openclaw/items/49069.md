## Summary

Keep open. Current main still lacks optional user-facing TUI message timestamps and linked #48708 remains open, but this PR is not merge-ready because the proposed config key is not schema-backed, the timestamp rendering calls a non-existent theme formatter, and assistant/history paths can lose or misstate timestamps.

## What This Changes

The PR adds a Date-based `[HH:MM]` prefix option to `ChatLog` and reads `tui.showTimestamps` during TUI startup.

## Best Possible Solution

Revise this PR or replace it with a narrow implementation that adds a typed and documented TUI timestamp setting or toggle, renders prefixes through existing theme formatters, preserves timestamps during streaming/final updates, uses persisted transcript timestamps during history replay, and covers user, assistant, pending-user, and replay behavior with focused TUI tests.

## Reproduction Assessment

Yes. The failure path is source-reproducible: the PR's documented config shape is rejected by the strict config contract, and the timestamp-enabled render path would call `theme.muted` plus raw assistant `setText` updates that remove prefixes.

## Solution Assessment

No. The requested feature direction is reasonable, but this patch is not the best maintainable implementation because it bypasses the public config contract and treats timestamps as render-time decoration instead of data that must survive streaming and replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add showTimestamps to the config contract:** `src/tui/tui.ts:541-545`
  - body: The new startup code reads `config.tui.showTimestamps`, but the root config schema is strict and has no `tui` key. Users who follow the PR's usage example will get `INVALID_CONFIG` before the TUI can start, so the option needs schema/types/help/docs support or a different existing settings surface.
  - confidence: 0.94
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:33`
  - body: With timestamps enabled this path calls `theme.muted(...)`, but current `theme` exports `dim` and not `muted`. That makes the patch fail type checking and can throw as soon as a timestamped message renders.
  - confidence: 0.98
- **[P2] Keep assistant timestamps through stream updates:** `src/tui/components/chat-log.ts:87-90`
  - body: The patch only prefixes the assistant text when the component is created. `updateAssistant` and `finalizeAssistant` later replace the component text with the raw response body, so streaming responses lose the timestamp after the first update.
  - confidence: 0.92
- **[P2] Use stored timestamps when replaying history:** `src/tui/components/chat-log.ts:7-14`
  - body: `formatTimestamp()` always uses the current local clock, while replayed transcript messages already carry timestamps. Reopening a session would relabel old messages with the time history was loaded unless `ChatLog` accepts and formats the message timestamp from history.
  - confidence: 0.86
- **[P2] Add the required changelog entry:** `src/tui/tui.ts:541-545`
  - body: This is a user-facing `feat`, but the PR only changes TUI code. Repo policy requires a single-line `CHANGELOG.md` entry with contributor credit before merge so the new option is visible in release notes.
  - confidence: 0.9
