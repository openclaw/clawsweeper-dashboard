## Summary

Keep open. Current main still has no TUI message timestamp option, and this PR is a relevant implementation candidate for the linked user-facing feature request, but the proposed patch is not mergeable as-is because it bypasses the strict config contract and has concrete TUI rendering bugs that would make the option fail or behave inconsistently.

## What This Changes

This PR adds a `ChatLog.showTimestamps` flag that prepends `[HH:MM]` to TUI chat messages and reads `tui.showTimestamps` from config during TUI startup.

## Best Possible Solution

Implement the timestamp option through the public config or settings contract, render prefixes with existing theme formatters, preserve the same timestamp through pending, streaming, final, and replayed messages, and document/test the behavior before merging.

## Reproduction Assessment

Yes. The failure path is source-reproducible: the proposed config shape is not in the strict config contract, `theme.muted` is absent from the current theme, and the current streaming/replay paths replace message text without carrying a timestamp prefix.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best maintainable solution because it treats timestamps as ad hoc render-time string decoration and reads an untyped config key instead of extending the supported config/settings surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add showTimestamps to the config contract:** `src/tui/tui.ts:541-545`
  - body: The new startup code reads `config.tui.showTimestamps`, but the root config schema is strict and current config types do not define a `tui` object. Users who follow the PR's usage example can fail config validation before the TUI starts, so the setting needs schema/types/help/docs support or an existing supported settings surface.
  - confidence: 0.94
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:33`
  - body: With timestamps enabled this path calls `theme.muted(...)`, but the current theme exports `dim` and not `muted`. That makes the patch fail type checking and can throw as soon as a timestamped message renders.
  - confidence: 0.98
- **[P2] Keep assistant timestamps through stream updates:** `src/tui/components/chat-log.ts:87-90`
  - body: The patch prefixes assistant text only when the component is created. `updateAssistant` and `finalizeAssistant` later replace the component text with the raw response body, so streaming responses lose the timestamp after the first update and fallback final messages can skip it entirely.
  - confidence: 0.92
- **[P2] Use stored timestamps when replaying history:** `src/tui/components/chat-log.ts:7-13`
  - body: `formatTimestamp()` always uses the current local clock, while replayed transcript messages already have timestamps. Reopening a session would relabel old messages with the time history was loaded unless the history path passes the message timestamp into the chat log renderer.
  - confidence: 0.86
- **[P2] Add the required changelog entry:** `src/tui/tui.ts:541-545`
  - body: This is a user-facing `feat`, but the PR only changes TUI code. Repo policy requires a single-line `CHANGELOG.md` entry with contributor credit before merge so the new option appears in release notes.
  - confidence: 0.9
