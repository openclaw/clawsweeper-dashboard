## Summary

Keep open: current main still does not implement TUI chat timestamps, and this PR is a plausible implementation path but has source-confirmed blockers in config validation, theme usage, streaming updates, and history replay.

## What This Changes

Adds an opt-in `tui.showTimestamps` setting that prepends `[HH:MM]` timestamps to TUI chat messages.

## Best Possible Solution

Land a rebased or replacement implementation that defines the timestamp setting in the config/help/docs contract, formats stored message timestamps across live, pending, streaming, final, and replay paths, and adds focused TUI/config coverage.

## Reproduction Assessment

Yes for the PR blockers via source inspection: the diff reads an undefined strict-config key, calls missing `theme.muted`, and only prefixes assistant text before later raw `setText` updates replace it. I did not execute the TUI in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best maintainable solution until it extends the config contract and carries real stored timestamps through streaming and history replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Add the timestamp setting to the config contract:** `src/tui/tui.ts:541-545`
  - body: The diff reads `config.tui.showTimestamps`, but current config has no `tui` object and the root schema is strict. Users following the PR usage example will fail config validation before the TUI starts, so this needs schema/types/help/docs support or a different existing surface.
  - confidence: 0.95
- **[P1] Use an exported theme formatter:** `src/tui/components/chat-log.ts:31-34`
  - body: When timestamps are enabled, the new prefix code calls `theme.muted(...)`, but current `theme` exports `dim` and no `muted`. That is a type error and would throw as soon as a timestamped system or user message renders.
  - confidence: 0.98
- **[P2] Preserve assistant prefixes during streaming updates:** `src/tui/components/chat-log.ts:97-100`
  - body: Only `startAssistant` adds the prefix. `updateAssistant` and `finalizeAssistant` still replace the component with raw text, so streamed replies lose the timestamp after the first update and fallback finals never get one.
  - confidence: 0.94
- **[P2] Use persisted message times for history replay:** `src/tui/components/chat-log.ts:8-14`
  - body: `formatTimestamp()` always reads the local clock. Replayed transcript messages already carry timestamps, but `loadHistory` passes only text into ChatLog, so reopening a session would relabel old messages with the load time.
  - confidence: 0.88
