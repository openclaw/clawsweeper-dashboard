## Summary

Close: current main now addresses the Telegram-specific event-loop starvation path by moving Bot API polling into an isolated worker with durable spooling, and the multi-account startup probe burst is separately bounded. Broader single-threaded gateway architecture concerns remain better tracked by the related open architecture reports.

## What This Changes

The issue reports that many Telegram bot accounts can starve the Gateway event loop and asks for polling isolation, throttling, backoff, and startup protection.

## Best Possible Solution

Keep the current worker-based Telegram ingress implementation for this issue, and leave broader gateway architecture work to https://github.com/openclaw/openclaw/issues/78808 and https://github.com/openclaw/openclaw/issues/78861.

## Reproduction Assessment

Yes for the central root cause: the merged fix for https://github.com/openclaw/openclaw/issues/81132 reproduced the main-event-loop-blocked Telegram polling failure with a fake Bot API server and deliberate event-loop block. I did not establish a fresh live reproduction with the reporter's exact 16-account Windows proxy setup.

## Solution Assessment

Yes. Moving getUpdates ingress into an isolated worker with durable spooling is a narrower and more robust fix for main event-loop starvation than tuning same-loop timers, and the separate startup probe limiter covers the reported multi-account getMe fanout.
