## Summary

Keep open: current main still starts each Telegram account as an independent long-lived poller and only prevents duplicate pollers for the same bot token. The reporter’s logs are not locally reproduced, but source, docs, and tests still show no cross-account `getUpdates` budget, startup stagger, or Telegram-owned event-loop degradation policy; related web-fetch and watchdog fixes do not close this multi-account starvation report.

## What This Changes

The issue reports severe Gateway event-loop starvation when 16 Telegram bot accounts long-poll concurrently through a local proxy.

## Best Possible Solution

Add a Telegram-owned multi-account polling scheduler that staggers account startup, bounds concurrent active `getUpdates` requests, and documents any operator-facing cap, with synthetic multi-account/proxy stress coverage plus targeted polling tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The report includes detailed logs and current source still allows many distinct-token Telegram pollers concurrently, but I did not reproduce the full 16-account Windows/proxy failure on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The requested stagger, cap, and backoff direction matches the source gap, but exact defaults, config shape, and event-loop degradation behavior need maintainer design before an automated fix is safe.
