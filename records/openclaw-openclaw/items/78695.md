## Summary

Keep open: current source and docs still show Telegram long polling has per-account watchdogs but no cross-account getUpdates budget or event-loop degradation policy. The linked web-fetch fix addresses a related timeout cleanup path, not Telegram multi-account polling pressure.

## What This Changes

The issue requests Telegram multi-account polling controls to prevent event-loop starvation when many bot accounts long-poll through a constrained proxy.

## Best Possible Solution

Add a Telegram-owned multi-account polling scheduler that staggers startup and bounds concurrent getUpdates activity, then document any operator-facing cap and prove it with a synthetic multi-account/proxy stress test plus targeted Telegram polling tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The report includes detailed logs and the source path shows missing cross-account polling coordination, but I did not reproduce the full 16-account/proxy/Windows failure on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The requested stagger/cap/backoff direction matches the source gaps, but the exact defaults, config surface, and ELU circuit-breaker behavior need maintainer design before a narrow implementation.
