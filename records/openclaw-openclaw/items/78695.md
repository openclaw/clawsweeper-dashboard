## Summary

Keep open: current main still has credible Telegram multi-account starvation gaps. The source shows per-account Telegram startup probes and independent polling sessions with per-session watchdog/backoff, but no cross-account getUpdates budget, startup stagger, or event-loop load-shedding policy; related open reports add narrower and newer evidence rather than superseding this canonical issue.

## What This Changes

The issue reports severe Gateway event-loop starvation when 16 Telegram bot accounts long-poll concurrently through a local proxy and asks for staggered polling, concurrency limits, backoff, and overload protection.

## Best Possible Solution

Define a Telegram-owned multi-account startup and polling backpressure policy, likely splitting bounded startup probes from steady-state getUpdates scheduling, then prove it with targeted tests plus live multi-account Telegram evidence.

## Reproduction Assessment

No, not for the full reported failure: I did not run the Windows 16-account proxy setup. Source inspection does show a plausible current-main path with per-account startup probes, independent polling sessions, and no cross-account polling budget.

## Solution Assessment

Unclear: staggered startup, bounded getUpdates concurrency, and backoff are plausible, but the exact defaults, config surface, and load-shedding behavior need maintainer design before implementation.
