## Summary

Keep open: the PR targets a real current-main CLI option mismatch and includes copied terminal proof, but the patch is not merge-ready because `gateway health --port` is converted into an explicit `--url` path that changes local auth behavior for secured gateways.

## What This Changes

This PR adds `--port` to gateway RPC subcommands and `gateway probe`, resolves it to a loopback WebSocket URL when `--url` is absent, and adds a changelog entry for the linked gateway CLI request.

## Best Possible Solution

If maintainers accept the local `--port` alias, implement it as a local port override that preserves local auth/TLS semantics, keeps `--url` precedence, and updates tests plus gateway CLI docs before landing.

## Reproduction Assessment

Yes, source-reproducible: current main declares `--port` for gateway run but not for the `gateway health` or `gateway probe` child commands, and the linked issue reports the same failure on v2026.5.7. I did not run commands locally because this was a read-only review.

## Solution Assessment

No: adding a local alias is plausible, but this implementation reuses explicit `--url` semantics for `gateway health` and therefore breaks configured local auth on secured gateways. A safer implementation should carry the port as a trusted loopback/local target fact instead of pretending the user supplied an arbitrary URL.
