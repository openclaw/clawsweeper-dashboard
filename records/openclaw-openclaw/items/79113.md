## Summary

Keep this PR open: it addresses a real gateway CLI consistency gap, but the current patch still drops existing config on the health path and forces plaintext probe URLs for TLS-enabled local gateways.

## What This Changes

The PR adds `--port` support to gateway RPC commands and `gateway probe`, plus a changelog entry for non-default local gateway ports.

## Best Possible Solution

Implement `--port` as a local port override that preserves the loaded gateway config, `--url` precedence, configured auth, TLS scheme, and local probe semantics, then document and test that contract.

## Reproduction Assessment

Yes, source-reproducible: current main declares `--port` for `gateway run` but not for `gateway health` or `gateway probe`, so Commander rejects the child flag. The linked issue also includes released-version output showing the same unknown-option failure.

## Solution Assessment

No: the intended local alias is plausible, but this implementation replaces the loaded config on the health path and hard-codes `ws://` for probe `--port`. The safer fix is to carry a typed local port override through existing config-backed target resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve loaded config when applying --port:** `src/cli/gateway-cli/call.ts:48`
  - body: When `--port` is supplied, this passes a synthetic config into `callGateway`. Normal CLI opts do not include a loaded config, and `callGateway` treats any provided config as complete, so configured auth, TLS, mode, and remote settings are dropped; secured local gateways can still fail or use the wrong scheme. Load and merge the real config, or pass a typed local port override without replacing config.
  - confidence: 0.93
- **[P2] Use configured TLS for probe --port:** `src/cli/gateway-cli/register.ts:619`
  - body: `gateway probe --port` is converted to `ws://127.0.0.1:<port>` before the status command resolves targets. Existing local probe resolution uses `wss://` when `gateway.tls.enabled` is true, so TLS-enabled local gateways on a non-default port will be probed over plaintext and fail. Route the port through the same local target/config path and add a TLS regression test.
  - confidence: 0.88
