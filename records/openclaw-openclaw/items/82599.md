## Summary

Keep open. Current main still has the reported `accepted`/`pending` behavior, but the existing docs, tests, gateway ACK, and SDK status type make this a protocol/API design request rather than a narrowly defined bug fix.

## What This Changes

Requests a `sessions_send` receipt ACK or clearer queued semantics before target processing completes.

## Best Possible Solution

Define an additive receipt contract for `sessions_send` if maintainers want this guarantee, then update the tool/gateway/SDK/docs/tests together without silently changing the existing public `accepted` status.

## Reproduction Assessment

Yes for the current behavior: source and tests show `sessions_send` returns `accepted`/pending for fire-and-forget and active soft-timeout paths, not a distinct `received` ACK. I did not run a live multi-agent flow because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the protocol contract. A receipt ACK may be useful, but renaming `accepted` to `queued` is not the safest path because current tests and SDK types treat `accepted` as public API; an additive receipt field/state would be more maintainable.
