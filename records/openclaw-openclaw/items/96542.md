## Summary

Keep open: current main and the latest release still use the colliding cwd-to-session-dir encoding and still choose the newest valid session file without verifying that its stored cwd matches the requested project.

## What This Changes

The issue requests preventing distinct project cwd paths that collapse to the same dashed session directory from sharing and resuming each other's JSONL sessions.

## Best Possible Solution

Make default project session lookup injective or cwd-aware so resume/list/fork never adopt a transcript whose header cwd differs from the requested cwd, while preserving or migrating existing shipped session directories.

## Reproduction Assessment

Yes at source level: current main still maps the two example cwds to the same directory and `continueRecent` opens the newest valid file without a header-cwd match. I did not run an artifact-writing repro because this was a read-only review.

## Solution Assessment

Partly: fixing the shared session directory/selection path is the right owner boundary, but an injective encoder must include migration or bounded cwd-aware legacy handling so existing shipped sessions are not orphaned.
