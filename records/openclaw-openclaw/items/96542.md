## Summary

Keep open: current main and v2026.6.10 still have the lossy cwd-to-session-dir encoder and cwd-blind recent-session resume path, while the linked fixing PRs are still open rather than merged.

## What This Changes

The issue reports that distinct project cwds differing only by path separators can share one session directory, causing `continueRecent` to resume another project's transcript.

## Best Possible Solution

Land one focused session-manager fix that makes the cwd-derived directory name injective and refuses to resume a transcript whose stored cwd does not match the requested project, with upgrade-safe handling for shipped session directories.

## Reproduction Assessment

Yes, at source level: current main still maps the reported cwd pair to one encoded directory and resumes the newest valid JSONL file from that directory without checking `header.cwd`. I did not run artifact-writing tests because this review is read-only.
