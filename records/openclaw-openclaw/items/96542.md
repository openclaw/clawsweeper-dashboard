## Label Justifications

- `P2`: This is a deterministic session-state correctness bug with limited blast radius to colliding project paths.
- `impact:session-state`: The failure can silently associate one project's transcript/history with another project's cwd during resume.

## Summary

Keep open: current main and v2026.6.10 still use a non-injective cwd-to-session-dir encoder and still resume the newest valid JSONL file without checking that the stored header cwd matches the requested project.

## What This Changes

The issue requests separate session storage and resume behavior for distinct project cwds whose paths differ only by path separators versus literal hyphens.

## Best Possible Solution

Make default project session lookup injective or cwd-aware so resume/list/fork never adopt a transcript whose header cwd differs from the requested cwd, while preserving or migrating shipped session directories.

## Reproduction Assessment

Yes, at source level: current main still maps the two example cwds to the same directory and continueRecent opens the newest valid file without a header-cwd match. I did not run an artifact-writing repro because this was a read-only review.

## Solution Assessment

Yes: the right boundary is the shared default session directory and recent-session selection path. The fix should be injective or cwd-aware and include upgrade-safe handling for existing shipped directories.
