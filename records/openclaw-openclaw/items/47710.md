## Summary

Keep open: current main still treats WhatsApp QR-login 408 as a terminal failure, but this PR is not mergeable because the retry is in the wrong layer, loses existing socket state, and can recreate sockets without a bound.

## What This Changes

The PR adds WhatsApp Web QR-login handling that recreates the socket after a 408 timeout in `waitForWebLogin` and logs timed-out session disconnects.

## Best Possible Solution

Move bounded `DisconnectReason.timedOut` retry handling into the shared WhatsApp login waiter so both initial QR start and scan wait preserve auth directory, legacy flag, socket timing, QR callbacks, and socket replacement state, with regression coverage for success and exhaustion.

## Reproduction Assessment

Yes, source-level. Mocking `waitForWaConnection` to reject with status 408 during QR login follows current main into the generic failed result, which `startWebLoginWithQr` or `waitForWebLogin` reports as a terminal login failure.

## Solution Assessment

No. The PR targets the right bug, but the retry belongs in `waitForWhatsAppLoginResult` with a cap and preserved socket state rather than as an ad hoc `waitForWebLogin` branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Retry 408 before the QR-start failure path:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: The retry only runs after `waitForWebLogin` sees `login.error`. If the 408 happens before the first QR wins `waitForQrOrRecoveredLogin`, `startWebLoginWithQr` returns a terminal failure and resets the active login before this branch can run, which is the linked failure mode. Put the retry in the shared login waiter so both start and wait paths use it.
  - confidence: 0.9
- **[P1] Preserve QR callbacks on 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it drops the `onQr` callback and socket timing options captured by the current login flow. A fresh QR after timeout will not update the active login state, and slow-network timing config is lost.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch after every replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry ceiling in the shared waiter before recreating sockets.
  - confidence: 0.88
- **[P3] Match timedOut exactly:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so the string `includes('408')` check is imprecise and can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.82
- **[P3] Keep the timeout log comment accurate:** `extensions/whatsapp/src/session.ts:149-151`
  - body: This session handler only logs the timed-out disconnect; it does not trigger reconnect from this branch. The comment should not imply recovery happens in the socket event handler.
  - confidence: 0.78
