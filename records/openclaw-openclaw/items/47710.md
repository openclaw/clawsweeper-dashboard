## Summary

Keep open: current main still treats a non-401 QR-login waiter error, including a 408, as terminal, so the PR is aimed at a real remaining gap. The proposed patch should not merge as-is because its retry path drops the existing QR/timing socket contract and can repeatedly churn sockets on persistent 408s.

## What This Changes

The PR adds a WhatsApp Web QR-login 408 timeout branch that closes and recreates the socket in `waitForWebLogin`, plus a timed-out disconnect log branch in WhatsApp session socket handling.

## Best Possible Solution

Move the useful 408 handling into `waitForWhatsAppLoginResult` as a bounded `DisconnectReason.timedOut` restart path that preserves `authDir`, `isLegacyAuthDir`, `socketTiming`, `onQr`, and `onSocketReplaced`, with regression tests for retry success, retry exhaustion, QR refresh preservation, and unchanged 401/515 behavior.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `waitForWaConnection` to reject with status 408: current main records a terminal QR-login failure, and the PR diff can be inspected or tested to show callback loss and unbounded retry behavior.

## Solution Assessment

No. The PR targets the right symptom, but the maintainable fix belongs in the shared WhatsApp login waiter where socket replacement already has the callback-preserving restart contract and can enforce an explicit retry bound.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QR callbacks and timing on retry:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The 408 retry creates the replacement socket with only `authDir`. That drops `onQr`, socket timing options, and the existing `onSocketReplaced` state update, so a fresh QR after the timeout may never refresh the active login state and configured timing knobs are lost.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch on each failed replacement until the outer wait deadline, repeatedly closing and opening sockets. Add a bounded retry policy, preferably in the shared login waiter, before merging this recovery path.
  - confidence: 0.87
- **[P3] Match timedOut exactly:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `String(login.errorStatus).includes('408')` is imprecise for a numeric status and can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.8
- **[P3] Fix the timeout reconnect comment:** `extensions/whatsapp/src/session.ts:150`
  - body: The session handler branch only logs the timeout; it does not trigger reconnect from that handler. The comment should not imply recovery happens there.
  - confidence: 0.78
