## Summary

Keep open. The underlying WhatsApp QR-login 408 timeout bug is still source-reproducible on current main, but this PR is not mergeable as written because the retry runs too late, drops the shared socket-replacement callback contract, lacks a retry bound, has no real behavior proof, and is currently conflicting.

## What This Changes

The PR adds WhatsApp Web QR-login handling that recreates the socket after a 408 timeout in `waitForWebLogin`, plus a timeout log branch in the WhatsApp session socket handler.

## Best Possible Solution

A corrected branch should implement bounded `DisconnectReason.timedOut` restart in the shared WhatsApp login waiter, preserve auth directory, legacy-auth state, socket timing, QR callbacks, and socket replacement state, then add focused regression tests plus redacted live QR-login evidence.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` to reject with a Boom-style 408 during QR login. Current main records the error and returns a terminal `WhatsApp login failed` path instead of retrying; no live WhatsApp pairing run was performed.

## Solution Assessment

No. The branch targets the right bug, but retrying in `waitForWebLogin` is too late and bypasses the shared socket-replacement contract; the safer fix belongs in `waitForWhatsAppLoginResult`.
