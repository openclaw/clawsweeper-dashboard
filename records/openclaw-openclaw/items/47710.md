## Summary

Keep open: the WhatsApp 408 QR-login problem is still valid, but this PR is not merge-ready because the retry is late in the flow, drops the existing socket replacement contract, and lacks real WhatsApp proof.

## What This Changes

The PR adds a WhatsApp Web QR-login 408 retry inside `waitForWebLogin` and a 408 timeout log branch in the WhatsApp session socket handler.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` restart in `waitForWhatsAppLoginResult`, preserve socket timing and QR replacement callbacks, add focused retry/exhaustion/QR-refresh regression coverage, and require redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` to reject with a Boom-style 408 during QR login. Current main records the error status and then returns a terminal `WhatsApp login failed` path instead of retrying; no live WhatsApp pairing run was performed.

## Solution Assessment

No. The PR targets the right bug class, but the retry belongs in the shared login waiter with bounded attempts and preserved callbacks; the current late retry is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This retry only runs after `waitForWebLogin` observes an already-active login error. If the 408 happens before the first QR is returned, the start path still fails before this code can retry, which matches the linked QR-not-showing symptom. Handle timed-out status in `waitForWhatsAppLoginResult` so start and wait paths share one restart contract.
  - confidence: 0.9
- **[P1] Preserve QR callbacks on 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it drops the `onQr` handler and socket timing captured by the active login. A fresh QR emitted after timeout may never update active login state, leaving users with an expired QR.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry cap or reuse a bounded shared restart path.
  - confidence: 0.88
