## Summary

Keep open: current main still treats WhatsApp QR-login 408 timeouts as terminal, but this PR is not mergeable as written because the retry runs in the wrong layer, drops QR/timing state, can recreate sockets repeatedly, and lacks required real behavior proof.

## What This Changes

The PR adds a WhatsApp Web QR-login 408 timeout retry branch in `waitForWebLogin` and a timed-out disconnect log branch in WhatsApp socket setup.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` retry in `waitForWhatsAppLoginResult`, preserving auth directory, legacy flag, socket timing, QR callbacks, and socket replacement state, with regression coverage and redacted live WhatsApp QR-login proof.

## Reproduction Assessment

Yes, source-level: mocking `waitForWaConnection` to reject with status 408 during QR login follows current main into the shared waiter's generic failed result and then into a terminal `WhatsApp login failed` result. I did not run a live WhatsApp device pairing flow in this read-only review.

## Solution Assessment

No. The PR targets a real bug, but the retry should be a bounded shared-waiter restart path; this branch misses pre-QR 408s, loses QR/timing state, and can churn sockets on persistent failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This branch only runs after `waitForWebLogin` observes `login.error`. If 408 happens while `startWebLoginWithQr` is waiting for the first QR, `waitForQrOrRecoveredLogin` returns failed and resets the active login before this retry can run, so the reported pre-QR timeout remains terminal.
  - confidence: 0.9
- **[P1] Preserve QR callbacks and timing on retry sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, dropping the `onQr` callback and configured socket timing captured by the active login flow. A fresh QR after timeout may not update active state, and slow-network timing config is lost.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this block after each replacement waiter fails until the outer deadline, repeatedly closing and opening sockets. Track a bounded retry count or handle 408 in the shared waiter with an explicit maximum.
  - confidence: 0.88
- **[P3] Use an exact timeout status check:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so `String(login.errorStatus).includes('408')` can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.82
