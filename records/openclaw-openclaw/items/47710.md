## Summary

Keep this PR open for changes: current main still treats WhatsApp QR-login 408 timeouts as terminal, but this patch retries in the wrong layer, drops replacement-socket state, lacks a retry bound, and has no required real behavior proof.

## What This Changes

The PR adds WhatsApp Web QR-login 408 timeout retry logic in `waitForWebLogin` and a timed-out disconnect log branch in WhatsApp socket setup.

## Best Possible Solution

Move bounded `DisconnectReason.timedOut` retry handling into the shared WhatsApp login waiter so initial QR start and scan wait both preserve auth directory, legacy flag, socket timing, QR callbacks, and socket replacement state, with regression coverage for retry success and exhaustion.

## Reproduction Assessment

Yes, source-level. Mocking `waitForWaConnection` to reject with status 408 during QR login follows current main into the generic failed result, which `startWebLoginWithQr` or `waitForWebLogin` reports as terminal.

## Solution Assessment

No. The PR targets a real bug, but the retry belongs in `waitForWhatsAppLoginResult` with a cap and preserved socket state rather than as an ad hoc `waitForWebLogin` branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This branch only runs after `waitForWebLogin` observes `login.error`. If 408 happens before the first QR wins `waitForQrOrRecoveredLogin`, `startWebLoginWithQr` returns a failure and clears the active login before this retry can run.
  - confidence: 0.9
- **[P1] Preserve QR and timing state on retry sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it drops the `onQr` callback and socket timing options captured by the active login flow. A fresh QR after timeout may not update active state, and slow-network timing config is lost.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch after each replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry ceiling in the shared waiter before recreating sockets.
  - confidence: 0.88
- **[P3] Use an exact timeout status check:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so `String(login.errorStatus).includes('408')` is imprecise and can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.82
