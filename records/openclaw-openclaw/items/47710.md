## Summary

Keep open: the WhatsApp QR-login 408 problem is still a real source-level gap on current main, but this PR is not merge-ready because the retry is in the wrong layer, loses socket callbacks, is unbounded, and lacks real WhatsApp proof.

## What This Changes

The PR adds a 408 timeout retry branch in `waitForWebLogin` and a WhatsApp session log branch for `DisconnectReason.timedOut`.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` handling inside `waitForWhatsAppLoginResult`, preserving socket timing, QR callbacks, socket replacement state, and focused 408 success/exhaustion coverage.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` to reject with a Boom-style 408 during QR login. Current main records that status as failed and reports it instead of retrying; no live WhatsApp device run was performed.

## Solution Assessment

No. The PR retries too late in `waitForWebLogin` and recreates the socket without the shared waiter callback/timing contract; the maintainable fix belongs in `waitForWhatsAppLoginResult`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: The retry only runs after `waitForWebLogin` is polling an active login. If the 408 happens before a QR is returned, `startWebLoginWithQr` sees the failed waiter result and returns `WhatsApp login failed`, so the reported no-QR path remains terminal. Put the retry in `waitForWhatsAppLoginResult` where start and wait flows share it.
  - confidence: 0.92
- **[P1] Preserve callbacks on replacement sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`. `createWaSocket` captures `onQr` and socket timing at construction time, and `attachLoginWaiter` relies on `onSocketReplaced`, so a fresh QR emitted after timeout will not update active login state and can leave users looking at an expired code.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Add an explicit retry limit in the shared restart path so network faults do not churn sockets for the full wait window.
  - confidence: 0.87
- **[P3] Correct the reconnect comment:** `extensions/whatsapp/src/session.ts:149-152`
  - body: This session handler only logs a 408 timeout; it does not trigger reconnect by itself. The comment should not claim a reconnect happens here because that misstates the socket ownership boundary.
  - confidence: 0.83
