## Summary

Keep open: the underlying 408 QR-login problem is still source-visible on current main, but this PR’s implementation is not mergeable as-is because it recreates sockets outside the shared waiter contract and can churn sockets on persistent timeout failures.

## What This Changes

The PR adds WhatsApp QR-login handling that closes and recreates the socket after a 408 timeout in `waitForWebLogin`, plus a 408 timeout log branch in the session socket handler.

## Best Possible Solution

Move bounded 408/timedOut handling into the shared WhatsApp login waiter, preserving QR/timing/replacement callbacks and keeping existing 401 logout plus 515 restart behavior covered by regression tests.

## Reproduction Assessment

Yes for a source-level reproduction: when `waitForWhatsAppLoginResult` records a 408 failure, current `waitForWebLogin` resets the active login through the generic non-401 error path. I did not run a live WhatsApp pairing attempt in this read-only review.

## Solution Assessment

No. The retry idea targets the right failure mode, but this PR implements it in the wrong layer without preserving callback/timing state and without a retry bound; the safer fix is a shared waiter retry path with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks on 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` when the socket is constructed, so a fresh QR emitted by the replacement socket is not written back to the active login state; users can keep seeing the expired QR after the 408. Reuse the shared restart path or pass the same QR/timing callbacks when recreating the socket.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Track a bounded retry count, or handle 408 in the shared waiter with an explicit maximum, so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.86
