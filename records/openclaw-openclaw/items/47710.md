## Summary

Keep open, but do not merge this patch as-is. Current main still lacks a bounded WhatsApp QR-login 408 retry, while this branch is dirty against main, missing real QR-login proof, and introduces retry-placement and socket-lifecycle defects.

## What This Changes

The PR adds 408 timeout socket retry handling during WhatsApp Web QR login and logs 408 disconnects in the WhatsApp session socket handler.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` restart in `waitForWhatsAppLoginResult`, preserve `authDir`, legacy auth state, socket timing, `onQr`, and `onSocketReplaced`, add retry/exhaustion/QR-refresh regression coverage, and require redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` or the shared login waiter to reject with a Boom-style 408 during QR login. Current main records that status as a failed login instead of retrying; no live WhatsApp pairing run was performed.

## Solution Assessment

No. The PR places retry logic too late and outside the shared waiter, so it misses pre-QR failures and drops the callback/timing contract; the maintainable fix belongs in `waitForWhatsAppLoginResult`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: The retry runs only from `waitForWebLogin` after `login.waitPromise` has already recorded failure. If the 408 arrives before a QR is returned, `startWebLoginWithQr` still wins the race with `WhatsApp login failed`, which is the linked user flow. Put the retry in `waitForWhatsAppLoginResult` so both start and wait paths share it.
  - confidence: 0.92
- **[P1] Preserve QR callbacks on 408 replacement sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`. `createWaSocket` captures `onQr` and socket timing at construction time, so a fresh QR emitted after the timeout will not update active login state and can leave users looking at the expired code.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Add an explicit retry limit in the shared restart path so network faults do not churn sockets for the full wait window.
  - confidence: 0.87
- **[P3] Correct the reconnect comment:** `extensions/whatsapp/src/session.ts:149-152`
  - body: This session handler only logs a 408 timeout; it does not trigger reconnect by itself. The comment should not claim a reconnect happens here because that misstates the socket ownership boundary.
  - confidence: 0.83
