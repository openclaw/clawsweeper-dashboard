## Summary

Keep open: current main still treats QR-login 408 as a terminal failure, while this PR's retry path bypasses the shared socket replacement contract and can churn sockets without a cap.

## What This Changes

The PR adds a WhatsApp QR-login 408 timeout socket retry in `waitForWebLogin` and a 408 timeout log branch in session socket handling.

## Best Possible Solution

Move the retry into `waitForWhatsAppLoginResult` as a bounded `DisconnectReason.timedOut` restart path that reuses the existing socket replacement contract and adds focused QR-login regression tests.

## Reproduction Assessment

Yes, source-level. Mock `waitForWaConnection` to reject with status 408 during an active QR login; current main records `login.errorStatus = 408`, and `waitForWebLogin` returns a terminal failed login result.

## Solution Assessment

No. The PR targets the right symptom, but retrying inside `waitForWebLogin` is not the narrowest maintainable fix because the shared waiter already owns socket replacement, callback preservation, and 515 restart behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QR callbacks when retrying 408:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it drops the `onQr` callback and socket timing options captured by `createWaSocket`. If WhatsApp emits a fresh QR after the timeout, the active login state will not update and users can keep seeing an expired code.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch after every failed replacement until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry ceiling in the shared waiter so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.88
- **[P3] Match timedOut exactly:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `String(login.errorStatus).includes('408')` is imprecise for a numeric status and can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.81
- **[P3] Fix the timeout reconnect comment:** `extensions/whatsapp/src/session.ts:150`
  - body: This session handler branch only logs the timeout; it does not trigger reconnect from that handler. The comment should not imply recovery happens there.
  - confidence: 0.78
