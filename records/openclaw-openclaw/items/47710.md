## Summary

Keep open: current main still lacks a bounded WhatsApp QR-login 408 retry and this PR remains the canonical follow-up for the closed QR-timeout report, but the branch is not mergeable as written because the retry is late, drops socket callbacks/timing, can loop repeatedly, and has no real QR-login proof.

## What This Changes

The PR adds a WhatsApp Web QR-login retry after a 408 timeout in `waitForWebLogin` and adds a timeout log branch in the WhatsApp session socket handler.

## Best Possible Solution

Port the idea into `waitForWhatsAppLoginResult` as a bounded `DisconnectReason.timedOut` socket-restart path that preserves auth directory, legacy auth state, socket timing, QR callbacks, socket replacement state, and existing 401/515 behavior, with focused regression tests and redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` to reject with a Boom-style 408 before or during QR login. Current main routes that through the shared waiter as a failed result, and `waitForWebLogin` resets the active login instead of retrying.

## Solution Assessment

No. The PR targets a valid bug, but the retry belongs in the shared WhatsApp login waiter with preserved callbacks/timing and an explicit retry cap rather than in a late `waitForWebLogin` branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This retry only runs after `waitForWebLogin` sees `login.error`; if the 408 happens before the first QR is returned, `startWebLoginWithQr` can return failure before this branch can help. Handle timed-out status in `waitForWhatsAppLoginResult` so start and wait paths share one bounded restart contract.
  - confidence: 0.9
- **[P1] Preserve callbacks and timing on retry sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it drops the socket timing and `onQr` handler captured by `createWaSocket`. A fresh QR emitted after the timeout may never update active login state, leaving users with an expired QR.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry cap or reuse a bounded shared restart path.
  - confidence: 0.88
- **[P3] Use an exact timeout status check:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so converting it to a string and searching for `408` can match unrelated values such as 1408 or 4080. Compare exactly against `DisconnectReason.timedOut`.
  - confidence: 0.82
