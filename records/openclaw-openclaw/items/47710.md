## Summary

Keep this PR open as useful source work, but the patch is not merge-ready: current main still treats QR-login 408 as terminal, while this PR recreates sockets without preserving the existing QR/timing callback contract and without a retry bound.

## What This Changes

The PR branch adds a 408 timeout retry in `waitForWebLogin` that recreates the WhatsApp socket, plus a timed-out disconnect log branch in `session.ts`.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` handling in `waitForWhatsAppLoginResult`, preserving `authDir`, `isLegacyAuthDir`, `socketTiming`, `onQr`, and `onSocketReplaced`, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a focused path: mock `waitForWaConnection` to reject with status 408 during an active QR login, and current main will return a terminal failed login through `waitForWebLogin`.

## Solution Assessment

No. The PR targets the right symptom, but placing ad hoc socket recreation in `waitForWebLogin` bypasses the shared waiter that already owns socket replacement and state preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QR callbacks when retrying 408:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it loses the QR callback and socket timing options captured by `createWaSocket`. A fresh QR emitted after the timeout will not update active login state, leaving callers with the expired code.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch after each replacement waiter fails until the outer deadline, repeatedly closing and opening sockets. Add an explicit retry ceiling in the shared login waiter so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.88
- **[P3] Match timedOut exactly:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `String(login.errorStatus).includes('408')` is imprecise for a numeric status and can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.81
- **[P3] Fix the timeout reconnect comment:** `extensions/whatsapp/src/session.ts:150`
  - body: The session handler branch only logs the timeout; it does not trigger reconnect from that handler. The comment should not imply recovery happens there.
  - confidence: 0.78
