## Summary

Keep open: current main still treats WhatsApp QR-login 408 failures as terminal, so the fix idea is useful, but this branch is conflicting, lacks after-fix real WhatsApp proof, and the implementation drops shared waiter state while allowing repeated socket churn.

## What This Changes

The PR adds a 408 timeout retry path in WhatsApp Web QR-login polling and logs 408 disconnects in the WhatsApp session socket handler.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` handling in `waitForWhatsAppLoginResult`, preserving `authDir`, `isLegacyAuthDir`, `socketTiming`, `onQr`, and `onSocketReplaced`, with focused retry success, retry exhaustion, QR refresh, 401, and 515 coverage plus redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` to reject with a Boom-style 408 during QR login. Current main records that status as failed and reports it instead of retrying; no live WhatsApp device run was performed.

## Solution Assessment

No. The PR retries too late in `waitForWebLogin` and recreates the socket without the shared waiter callback/timing contract; the maintainable fix belongs in `waitForWhatsAppLoginResult`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: The retry runs from `waitForWebLogin` after `login.waitPromise` has already recorded failure. If the 408 happens before a QR is returned, the start path can still fail with `WhatsApp login failed`, so the linked QR-not-showing flow is not reliably fixed. Put the retry in `waitForWhatsAppLoginResult` so start and wait paths share it.
  - confidence: 0.92
- **[P1] Preserve QR callbacks on 408 replacement sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`. `createWaSocket` captures `onQr` and socket timing at construction time, so a fresh QR emitted after the timeout will not update active login state and can leave users looking at the expired code.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Add an explicit retry limit in the shared restart path so network faults do not churn sockets for the full wait window.
  - confidence: 0.87
- **[P3] Correct the reconnect comment:** `extensions/whatsapp/src/session.ts:149-152`
  - body: This session handler only logs a 408 timeout; it does not trigger reconnect by itself. The comment should not claim a reconnect happens here because that misstates the socket ownership boundary.
  - confidence: 0.83
