## Summary

Keep open: the PR targets a real WhatsApp QR-login 408 timeout bug, but current main still treats that status as a terminal login failure and this branch is not safe to merge because it misses part of the flow, drops socket callback/timing state, can retry unboundedly, and lacks required real behavior proof.

## What This Changes

The PR adds a WhatsApp Web QR-login 408 timeout retry in `waitForWebLogin` and a 408 timeout log branch in WhatsApp socket setup.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` retry in the shared WhatsApp login waiter while preserving auth directory, legacy state, socket timing, QR callbacks, and socket replacement state, with focused regression tests and redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mocking `waitForWaConnection` to reject with a Boom-style 408 during QR login follows current main into the shared waiter's generic failed result and then into a terminal `WhatsApp login failed` path. I did not run a live WhatsApp pairing flow.

## Solution Assessment

No. The PR targets the right bug, but the retry should be a bounded shared-waiter restart path; this branch misses pre-QR failures, drops QR/timing state, and can churn sockets on persistent 408s.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This branch only runs after `waitForWebLogin` observes `login.error`. If a 408 happens while `startWebLoginWithQr` is still waiting for the first QR, the start path returns failed and resets the active login before this retry can run, so the original pre-QR timeout remains terminal.
  - confidence: 0.9
- **[P1] Preserve QR callbacks on retry sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, dropping the `onQr` callback, socket timing, and socket replacement state used by the active login flow. A fresh QR after timeout may not update the active QR image, and slow-network timing config is lost.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this block each time the new waiter fails until the outer deadline, repeatedly closing and opening sockets. Track a bounded retry count or handle 408 in the shared waiter with an explicit maximum.
  - confidence: 0.88
- **[P3] Use an exact timeout status check:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so `String(login.errorStatus).includes('408')` can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.82
