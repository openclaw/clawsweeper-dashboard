## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: The retry runs only after `waitForWebLogin` sees `login.error`. If the 408 happens before the first QR is returned, `startWebLoginWithQr` can race the failed waiter and return `WhatsApp login failed` before this retry block ever runs, which misses the reported no-QR path. Put the bounded timeout restart in `waitForWhatsAppLoginResult` so start and wait flows share it.
  - confidence: 0.92
- **[P1] Preserve QR callbacks and socket timing on retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, dropping the `onQr` callback and configured socket timing captured by `createWaSocket`. A fresh QR emitted after the timeout will not update active login state, and slow-link timing settings are lost for the retry.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Add an explicit retry limit in the shared restart path so network faults do not churn sockets for the full wait window.
  - confidence: 0.87
- **[P3] Correct the reconnect comment:** `extensions/whatsapp/src/session.ts:149-152`
  - body: This session handler only logs a 408 timeout; it does not trigger reconnect by itself. The comment should not claim reconnect happens here because that misstates the socket ownership boundary.
  - confidence: 0.83
