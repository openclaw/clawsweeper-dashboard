## Summary

Keep open: current main still lacks bounded QR-login 408 recovery, but this branch is not mergeable as-is because it retries at the wrong layer, drops QR/timing callbacks, has no retry cap, and lacks real behavior proof.

## What This Changes

The PR adds WhatsApp Web QR-login 408 retry logic in `waitForWebLogin` and a timed-out log branch in the WhatsApp session handler.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` retry in the shared WhatsApp login waiter, preserving auth directory, legacy auth state, socket timing, QR callbacks, and socket replacement state, with focused regression tests and redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mocking `waitForWaConnection` to reject with a Boom-style 408 during QR login follows current main through the shared waiter into a generic failed result and then a terminal `WhatsApp login failed` path. I did not run a live WhatsApp pairing flow in this read-only review.

## Solution Assessment

No. The branch targets the right bug, but the retry belongs in `waitForWhatsAppLoginResult` as a bounded shared restart path that preserves QR callbacks, socket timing, and replacement state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This block only runs after `waitForWebLogin` observes `login.error`; if the 408 happens before the first QR is returned, `startWebLoginWithQr` resets the active login and returns failed before this retry can run. Handle timed-out status in the shared waiter so both start and wait paths share the same bounded restart contract.
  - confidence: 0.9
- **[P1] Preserve QR callbacks on retry sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, dropping `onQr`, socket timing, and replacement-state callbacks. A fresh QR emitted after timeout may not update the active QR image, leaving users with the expired code.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add a retry cap or reuse a bounded shared waiter restart path.
  - confidence: 0.88
- **[P3] Use an exact timeout status check:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so `String(login.errorStatus).includes('408')` can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.82
