## Summary

Keep open: current main still treats QR-login 408 as a terminal login failure and adjacent merged WhatsApp reliability fixes explicitly did not replace this work, but this PR cannot merge as-is because the retry is late, unbounded, drops retry socket callbacks/timing, conflicts with main, and lacks real behavior proof.

## What This Changes

The PR adds a WhatsApp QR-login 408 retry in `waitForWebLogin` and a timed-out socket log branch in `createWaSocket`.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` restart inside `waitForWhatsAppLoginResult`, preserving auth directory, legacy auth state, socket timing, QR callbacks, and socket replacement state, with focused regression tests and redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` to reject with a Boom-style status 408 during QR login. Current main routes that through `waitForWhatsAppLoginResult` as `failed`, then `waitForWebLogin` resets the active login with `WhatsApp login failed`; no live WhatsApp pairing flow was run.

## Solution Assessment

No. The PR targets the right bug, but placing retry logic in `waitForWebLogin` is too late and creates a second restart path; the maintainable fix belongs in the shared WhatsApp login waiter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This block only runs after `waitForWebLogin` sees `login.error`; if the 408 ends the waiter before the first QR/start response, the start path can return failure before this retry helps. Handle `DisconnectReason.timedOut` inside `waitForWhatsAppLoginResult` so start and wait flows share one bounded restart contract.
  - confidence: 0.9
- **[P1] Preserve QR and timing callbacks on retry sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it drops `socketTiming` and the `onQr` callback captured by `createWaSocket`. A fresh QR emitted after timeout may never update active login state, leaving users with an expired QR.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry cap or reuse a bounded shared restart path.
  - confidence: 0.88
- **[P3] Use an exact timeout status check:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so converting it to a string and searching for `408` can match unrelated values such as 1408 or 4080. Compare exactly against `DisconnectReason.timedOut`.
  - confidence: 0.82
- **[P3] Correct the timeout reconnect comment:** `extensions/whatsapp/src/session.ts:149-151`
  - body: This handler only logs a timed-out close; it does not trigger reconnect itself. The comment should not claim reconnect ownership in `session.ts` when retry belongs in the login/controller path.
  - confidence: 0.78
