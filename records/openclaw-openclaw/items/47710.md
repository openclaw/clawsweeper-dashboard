## Summary

Keep open: current main still treats WhatsApp QR-login 408 errors as terminal, but this PR should not merge as-is because its stale retry block drops the existing socket-replacement contract and can churn sockets under persistent timeouts.

## What This Changes

The PR adds a WhatsApp QR-login 408 timeout branch that closes and recreates the socket, plus a session handler log branch for timed-out disconnects.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` handling in `waitForWhatsAppLoginResult`, preserving `authDir`, `isLegacyAuthDir`, `socketTiming`, `onQr`, and `onSocketReplaced`, with QR-login regression tests and a changelog entry.

## Reproduction Assessment

Yes. A focused source-level reproduction can mock `waitForWaConnection` to reject with status 408 during an active QR login; current main will flow through the generic failed result and `waitForWebLogin` will return a terminal login failure.

## Solution Assessment

No. The PR targets the right symptom, but adding ad hoc retry logic in `waitForWebLogin` bypasses the shared waiter that already owns socket replacement and state preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QR callbacks when retrying 408:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it loses the QR callback and socket timing options captured by `createWaSocket`. A fresh QR emitted after the timeout will not update the active login state, leaving callers with the expired code.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch after each replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry ceiling in the shared login waiter so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.88
- **[P3] Match timedOut exactly:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `String(login.errorStatus).includes('408')` is imprecise for a numeric status and can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.81
- **[P3] Fix the timeout reconnect comment:** `extensions/whatsapp/src/session.ts:150`
  - body: The session handler branch only logs the timeout; it does not trigger reconnect from that handler. The comment should not imply recovery happens there.
  - confidence: 0.78
