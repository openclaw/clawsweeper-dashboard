## Summary

Keep open. Current main still treats WhatsApp QR-login 408 timeouts as terminal, and this PR targets a real bug, but the branch is not mergeable as-is because the retry is too late, drops required QR/timing callbacks, is unbounded, and lacks real behavior proof.

## What This Changes

The PR adds a WhatsApp Web QR-login 408 timeout retry in `waitForWebLogin` and logs timed-out socket closes in `createWaSocket`.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` restart inside the shared WhatsApp login waiter, preserving auth directory, legacy auth state, socket timing, QR callbacks, and socket replacement state, with focused regression tests and redacted live QR-login proof.

## Reproduction Assessment

Yes, source-level: mock `waitForWaConnection` to reject with a Boom-style 408 during QR login. Current main routes that through the shared waiter into a failed result and then `waitForWebLogin` resets the login with `WhatsApp login failed`; no live WhatsApp pairing flow was run.

## Solution Assessment

No. The branch targets the right bug, but placing the retry in `waitForWebLogin` is too late and bypasses the shared restart contract; the safer fix belongs in `waitForWhatsAppLoginResult`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move 408 retry into the shared login waiter:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: This retry only runs after `waitForWebLogin` observes `login.error`; if the 408 happens before the first QR is returned, the start path can reset the active login and return failure before this block can help. Handle timed-out status in `waitForWhatsAppLoginResult` so start and wait paths share one bounded restart contract.
  - confidence: 0.9
- **[P1] Preserve QR and timing callbacks on retry sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The replacement socket is created with only `authDir`, so it drops the socket timing and `onQr` handler captured by `createWaSocket`. A fresh QR emitted after timeout may never update active login state, leaving users with an expired QR.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry cap or reuse a bounded shared restart path.
  - confidence: 0.88
- **[P3] Use an exact timeout status check:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `login.errorStatus` is numeric, so converting it to a string and searching for `408` can match unrelated status values such as 1408 or 4080. Compare exactly against `DisconnectReason.timedOut`.
  - confidence: 0.82
- **[P3] Correct the reconnect comment in the session handler:** `extensions/whatsapp/src/session.ts:149-151`
  - body: This handler only logs a timed-out close; it does not trigger reconnect itself. The comment should not claim reconnect ownership in `session.ts` when retry belongs in the login/controller path.
  - confidence: 0.78
