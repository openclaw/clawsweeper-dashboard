## Summary

Keep open: current main still treats a WhatsApp QR-login 408 as terminal, but this PR is not mergeable as-is because the retry path drops the existing socket replacement contract and has no retry bound.

## What This Changes

The PR adds a 408 timeout branch to WhatsApp QR login that closes and recreates the socket, plus a timed-out disconnect log branch in session socket handling.

## Best Possible Solution

Implement bounded `DisconnectReason.timedOut` handling in `waitForWhatsAppLoginResult`, preserving the existing socket replacement contract and adding focused QR-login regression tests plus a changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `waitForWaConnection` to reject with status 408 during an active QR login; current main records a terminal login failure, and the PR diff shows the replacement socket loses callbacks and retry bounds.

## Solution Assessment

No. The PR targets the right symptom, but the maintainable fix belongs in the shared WhatsApp login waiter where socket replacement already preserves QR/timing state and can enforce an explicit retry cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QR callbacks and timing on retry:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The 408 branch recreates the socket with only `authDir`, dropping `onQr`, `socketTiming`, and the `onSocketReplaced` state update. A fresh QR emitted after the timeout can fail to update active login state, leaving users with the expired QR.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 can re-enter this branch on each failed replacement until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry ceiling, preferably in the shared login waiter.
  - confidence: 0.87
- **[P3] Match timedOut exactly:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `String(login.errorStatus).includes('408')` is imprecise for a numeric status and can match unrelated values such as 1408 or 4080. Use an exact `DisconnectReason.timedOut` comparison.
  - confidence: 0.8
- **[P3] Fix the timeout reconnect comment:** `extensions/whatsapp/src/session.ts:150`
  - body: The session handler branch only logs the timeout; it does not trigger reconnect from that handler. The comment should not imply recovery happens there.
  - confidence: 0.78
