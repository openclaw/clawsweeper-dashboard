## Summary

Keep open: the PR targets a real Telegram diagnostic bug, but the patch suppresses the warning whenever `defaultAccount` is merely present, including invalid values that still fall back to another account. The added regression test also does not fail on current `main` because `charles` is not the first sorted account id in that fixture.

## What This Changes

The PR changes Telegram default account selection to suppress the missing-`accounts.default` warning when `channels.telegram.defaultAccount` is set, and adds a multi-account regression test.

## Best Possible Solution

Keep the fix, but suppress the warning only when the normalized `defaultAccount` is valid for the configured account ids, and add a before-failing regression fixture where the explicit default is the first sorted fallback account.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: on current `main`, a valid `defaultAccount` that is also the first sorted account id still satisfies `shouldWarnMissingDefault`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written: the intended suppression is right for valid explicit defaults, but presence-only gating also suppresses invalid-default fallback diagnostics. The safer solution is to check that the configured default is the resolved/listed account before suppressing.
