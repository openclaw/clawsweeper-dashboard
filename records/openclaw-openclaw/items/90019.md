## Summary

Keep open: the PR targets a real stale builtin memory-search index path that current main and the latest release still do not solve, but the patch should not merge as-is because the new interval callback marks memory dirty even for managers that do not index memory files; the default-on 30-minute fallback also remains a maintainer upgrade decision.

## What This Changes

The PR changes builtin memory search to default `sync.intervalMinutes` to 30 minutes, forces dirty state before interval sync, documents the default and `0` opt-out, and adds focused regression tests.

## Best Possible Solution

Scope the interval dirty mark to managers that actually include the memory source, then land only if maintainers explicitly accept the default-on periodic fallback; otherwise keep the default at `0` and document `30` as the workaround.

## Reproduction Assessment

Yes at source level: current main defaults interval sync to `0`, and the current interval timer does not force dirty state before sync. I did not reproduce a real multi-hour chokidar outage.

## Solution Assessment

No as submitted: the default resolver plus interval dirty mark is the right narrow code path if maintainers accept the upgrade behavior, but the dirty assignment must be scoped so non-memory-source managers do not retain stale dirty state.
