Full review comments:

- **[P2] Use policy matching for web_search allowlists:** `src/cron/isolated-agent/run.ts:356-360`
  - body: This checks only for an exact/group-expanded `web_search` entry. Runtime tool policy uses glob matching, so an explicit allowlist like `web_*` permits `web_search` but would skip this diagnostic; use the shared matcher/helper so the preflight matches the actual tool policy.
  - confidence: 0.9
- **[P2] Skip default-derived toolsAllow caps:** `src/cron/isolated-agent/run.ts:1494-1501`
  - body: Agent-created cron jobs can store the creator tool surface with `toolsAllowIsDefault: true`. This call passes only `toolsAllow`, so a default cap containing `web_search` is treated as an explicit operator request and can persist a misleading no-provider warning.
  - confidence: 0.88
- **[P2] Resolve model API before native-search checks:** `src/cron/isolated-agent/run.ts:383-388`
  - body: The native-search check falls back only to configured provider model rows, but cron model API is often supplied by the runtime catalog. Without passing the catalog API, eligible OpenAI/Codex Responses models can look `managed_only` and still get a false no-provider warning.
  - confidence: 0.84
- **[P2] Point users to provider selection:** `src/cron/isolated-agent/run.ts:139-140`
  - body: The warning tells operators to enable DuckDuckGo, but key-free DuckDuckGo is not auto-selected merely because it is enabled. The diagnostic should direct users to `openclaw configure --section web` or `tools.web.search.provider` so following it actually clears the warning.
  - confidence: 0.9
