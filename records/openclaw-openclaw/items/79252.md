## Label Justifications

- `P2`: This is a normal-priority loop-safety bug that can burn tool/model calls and require manual stop, but it does not make the core runtime unusable by default.
- `impact:crash-loop`: The issue is about runaway cross-tool agent loops continuing because the global breaker resets by tool signature.

## Summary

Keep open: current main and the latest release still compute the global breaker from the current tool/args no-progress streak, so cross-tool no-progress loops can still evade it. The open closing PR is the current implementation candidate, but it has not merged.

## What This Changes

The issue asks for `globalCircuitBreakerThreshold` to count no-progress outcomes across the run/session instead of per tool/args signature.

## Best Possible Solution

Review and land the focused session-global detector fix in https://github.com/openclaw/openclaw/pull/97577, or replace it with an equivalent narrow PR that includes detector and before-tool-call regression coverage.

## Reproduction Assessment

Yes at source level: current main computes the global breaker count from `getNoProgressStreak(history, toolName, currentHash)`, so switching tool signatures can reset the count. I did not run a live Feishu/Gateway replay in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow detector-level aggregate across scoped session history because the wrapper already supplies the run scope and docs already define the setting as global. The adjacent exec-hash and consecutive-error-cascade issues should remain separate.
