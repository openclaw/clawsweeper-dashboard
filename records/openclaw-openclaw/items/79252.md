## Summary

Keep open: current main and v2026.6.10 still compare the global breaker against the current tool/args no-progress streak, so cross-tool no-progress loops can still evade the documented global breaker. Earlier same-problem PRs are closed unmerged, and the currently open related work tracks adjacent loop-detection problems rather than this exact session-global counter bug.

## What This Changes

The issue asks for `globalCircuitBreakerThreshold` to count stable no-progress outcomes across all tools in a session/run instead of per tool signature.

## Best Possible Solution

Land a focused detector-level fix that aggregates stable no-progress outcomes across the run/session for the global breaker, with pure detector and before-tool-call wrapper regression coverage.

## Reproduction Assessment

Yes, source-level: current main computes the global breaker count with `getNoProgressStreak(history, toolName, currentHash)`, so switching tool signatures can reset the count. I did not run a live Feishu/Gateway replay in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow detector-boundary aggregate count because the wrapper already supplies run-scoped session history and the docs already define global behavior. The adjacent consecutive-error-cascade work should stay separate from this stable no-progress counter bug.
