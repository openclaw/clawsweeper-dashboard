## Summary

Keep open: current `main` and latest release still compare the global breaker against the current tool/args no-progress streak, so switching tools can evade the documented session-global breaker. The earlier focused fix was closed unmerged, and nearby open loop-detection PRs cover different problems rather than this exact stable no-progress aggregate counter.

## What This Changes

The issue requests that `globalCircuitBreakerThreshold` count no-progress tool outcomes across a session rather than per tool signature so tool switching cannot evade the loop breaker.

## Best Possible Solution

Implement a focused detector-level aggregate no-progress streak and add pure detector plus wrapped before-tool-call regression coverage for blocking a different next tool once the threshold is reached.

## Reproduction Assessment

Yes at source level: the detector compares the global threshold against a per-signature no-progress streak, and the wrapper calls that detector with session history before every tool call. I did not run tests because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is in the detector, because the caller already provides session history and the docs already define a global no-progress breaker. A separate new config or caller-specific block list is unnecessary for this report.
