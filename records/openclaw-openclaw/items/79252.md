## Summary

Keep open: current main and v2026.6.8 still compare the global breaker against the current tool/args no-progress streak, so cross-tool no-progress loops can still evade the documented global breaker. The open linked candidate fix at https://github.com/openclaw/openclaw/pull/94606 uses closing syntax but is not merged, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The report asks OpenClaw to make `globalCircuitBreakerThreshold` count stable no-progress tool outcomes across the session rather than per tool/args signature.

## Best Possible Solution

Land a focused detector-level aggregate no-progress fix with pure detector and before-tool-call wrapper coverage, then close the issue with the merged commit and release provenance.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.8 compute the global breaker count with `getNoProgressStreak(history, toolName, currentHash)`, while docs define a global no-progress breaker. I did not run live Feishu/Gateway reproduction in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable solution is a detector-boundary aggregate no-progress count because the caller already supplies session history and the docs already define global behavior. The issue should stay open until the focused linked PR lands or is replaced.
