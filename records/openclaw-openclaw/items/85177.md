## Summary

Keep open: current main and the latest release still preserve the no-raw auth-boundary behavior, while the remaining user problem is a real claude-cli continuity loss when cosmetic auth-profile or auth-epoch churn drops an otherwise resumable session. The active replacement path is the open auth-epoch PR, but it still needs maintainer/security agreement on Claude CLI identity semantics before this issue can close.

## What This Changes

The issue asks OpenClaw to preserve claude-cli conversation continuity after auth-profile or auth-epoch invalidation instead of starting a fresh session with no usable prior context.

## Best Possible Solution

Land a narrowed auth-identity fix that preserves claude-cli reuse when the host CLI identity is unchanged, keeps raw transcript replay blocked when identity changed or is unproven, and covers auth-profile/auth-epoch paths with focused tests and real proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection and related PR proof: current main can return auth-profile/auth-epoch invalidation and then produce no raw reseed across that boundary, and the open replacement PR includes production reset logs. I did not run a live WSL2 gateway scenario in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No for the broad raw-reseed suggestion: it conflicts with current docs/tests and the accepted auth-boundary policy. The best path is the narrower identity/trigger fix now under review, after maintainers settle the Claude CLI auth identity contract.
