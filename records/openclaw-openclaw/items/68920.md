## Summary

Keep open. Current main still lacks an HTTP light-context or voice-mode policy for `/v1/chat/completions`, and the related stable-session work in #69060 is useful but only covers one subproblem.

## What This Changes

Request adding low-latency light-context behavior and stable session handling to the OpenAI-compatible chat completions endpoint for voice/realtime callers.

## Best Possible Solution

Define a documented low-latency HTTP prompt policy for trusted callers, wire it through existing `promptMode` and `bootstrapContextMode` seams, and keep the stable-session work scoped separately.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no HTTP `lightContext` config, and no `x-openclaw-session` stable hint, while regular HTTP sessions resolve to full prompt mode. The reported latency magnitude still needs the reporter's live setup to reproduce numerically.

## Solution Assessment

Unclear. The direction is valid, but a bare header is not enough; maintainers should first define safe HTTP prompt-policy semantics for a full operator-access endpoint and then implement the narrow code path.
