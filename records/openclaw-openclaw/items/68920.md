## Summary

Keep open: current main and v2026.6.10 still route `/v1/chat/completions` through the normal agent ingress path with random-session fallback when no supported stable key is provided, and there is still no HTTP light-context or voice-mode contract. The related session-key issues and PRs cover narrower session-continuity subproblems, not the main full-context TTFB request.

## What This Changes

The issue requests a low-latency HTTP `/v1/chat/completions` mode that avoids full agent context assembly and also asks for stable HTTP session reuse.

## Best Possible Solution

Define a maintainer-approved low-latency Gateway HTTP contract that explicitly scopes prompt, tools, memory, bootstrap, and session behavior, then cover it with docs, tests, upgrade semantics, and live latency proof.

## Reproduction Assessment

Yes, at source level: current `/v1/chat/completions` reaches the random-session fallback and full `agentCommandFromIngress` path without a light-context HTTP branch. I did not run a live TTFB benchmark in this read-only review.

## Solution Assessment

Unclear as proposed: a light-context header and per-agent prompt override are plausible, but the best fix needs maintainer-owned API semantics for prompt, tool, memory, bootstrap, and session behavior.
