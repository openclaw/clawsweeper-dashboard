## Summary

Keep open: current main and v2026.5.18 still run OpenAI-compatible Chat Completions through the normal full-agent path, and there is no public `x-openclaw-light-context`, `http.systemPrompt`, or stable `x-openclaw-session` contract yet; the related stable-session PR covers only the QMD/session-warm portion.

## What This Changes

Request a low-latency HTTP Chat Completions mode that avoids full agent context assembly and supports stable sessions or HTTP-specific slim prompts for voice/realtime clients.

## Best Possible Solution

Define and document a trusted low-context Chat Completions policy, wire it through existing prompt/bootstrap/tool/memory seams with focused Gateway tests, and land stable-session reuse through https://github.com/openclaw/openclaw/pull/69060 or a replacement.

## Reproduction Assessment

Yes for the source-level gap: current main has no low-context Chat Completions contract, no HTTP-specific prompt override, no stable `x-openclaw-session` on main, and still forwards Chat Completions through normal agent ingress. I did not rerun the reporter's live latency benchmark.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix requires a maintainer-owned public contract for low-context HTTP behavior; stable-session reuse should land through https://github.com/openclaw/openclaw/pull/69060 or a replacement while voice-mode policy remains separate.
