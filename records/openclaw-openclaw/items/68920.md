## Summary

Keep open. Current main still routes `/v1/chat/completions` through the normal agent ingress path without an HTTP light-context or voice-mode prompt policy, and the related stable-session PR is explicitly only a partial warm-cache fix.

## What This Changes

The issue asks for a low-latency `/v1/chat/completions` mode that reduces full agent context assembly, supports stable HTTP sessions, and optionally lets agents override the HTTP system prompt for realtime voice callers.

## Best Possible Solution

Define an explicit low-latency HTTP prompt/context policy, document its security boundary for the full operator-access endpoint, then wire it through existing `promptMode` and `bootstrapContextMode` seams with focused endpoint tests and docs; keep stable session reuse tracked separately through #69060 or an equivalent narrow change.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no HTTP `lightContext` config, and no `x-openclaw-session` parsing, while regular HTTP requests resolve to full prompt mode. The exact TTFB values still need the reporter's live workspace/model setup to reproduce numerically.

## Solution Assessment

Unclear. A bare header is not enough by itself; the maintainable fix should first define safe HTTP prompt-policy semantics for this full operator-access endpoint, then wire the existing runtime prompt controls and document the tradeoffs.
