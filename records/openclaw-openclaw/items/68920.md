## Summary

Keep open. Current main and the latest release still route `/v1/chat/completions` through the normal ingress agent pipeline with full context assembly, and the related stable-session work is either closed unmerged or only covers a narrower Open-WebUI session-correlation problem.

## What This Changes

The issue requests a low-latency HTTP chat-completions mode that avoids full agent context assembly, plus stable HTTP session behavior for voice and OpenAI-compatible clients.

## Best Possible Solution

Maintainers should either split the stable-session work into the narrower Open-WebUI/session-correlation tracker or sponsor a Gateway HTTP light-context design that preserves the current agent-first default while making the low-context mode explicit, documented, and tested.

## Reproduction Assessment

Yes at source level: the current Chat Completions handler still calls the normal ingress agent pipeline, and the resolver falls back to a random session when no `user` or explicit session key is supplied. I did not run a live latency reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API contract. An HTTP light-context mode is plausible, but headers, defaults, context pruning, tool exposure, and session reuse are public Gateway behavior and should not be added as a narrow autonomous bug fix.
