## Summary

Keep open. Current main and the latest release still route `/v1/chat/completions` through the normal full agent ingress path; the stable-session work is isolated in open PRs and does not solve the requested low-context/voice-mode prompt policy.

## What This Changes

The issue requests a low-latency `/v1/chat/completions` mode that avoids full agent context assembly and supports stable HTTP sessions or HTTP-specific system prompts for voice-style callers.

## Best Possible Solution

Define a documented trusted HTTP low-latency prompt policy, wire it through existing prompt/bootstrap context seams with Gateway tests and docs, and coordinate stable session reuse as a compatible separate contract.

## Reproduction Assessment

Yes for the source-level gap: current main has no HTTP light-context, voice-mode, or HTTP-specific prompt policy, and ordinary Chat Completions requests still use full prompt mode. The exact latency numbers still need the reporter's live environment to reproduce numerically.

## Solution Assessment

Unclear. The requested direction is valid, but an ad hoc header alone is not the best fix; the maintainable solution needs explicit prompt, tool, memory, session, and security semantics for the full operator-access HTTP surface.
