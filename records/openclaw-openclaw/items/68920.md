## Summary

Keep open. Current main still runs `/v1/chat/completions` through the normal agent ingress path without an HTTP light-context, voice-mode, or HTTP-specific prompt policy; the related stable-session PR only addresses the QMD warm-session subproblem.

## What This Changes

The issue requests a low-latency OpenAI-compatible Chat Completions mode with lighter prompt assembly, stable HTTP sessions, and optional HTTP-specific system prompts for voice and realtime callers.

## Best Possible Solution

Define a documented trusted HTTP low-latency prompt policy, wire it through existing prompt and bootstrap context seams with Gateway tests/docs, and close this only after the stable-session and light-context pieces are both resolved or split into separate canonical items.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no HTTP prompt policy, and no stable `x-openclaw-session` hint, while ordinary HTTP sessions use full prompt mode. I did not reproduce the reporter's live 10-15s timing numbers.

## Solution Assessment

Unclear. The requested direction is valid, but the maintainable solution needs explicit prompt, tool, memory, session, and security semantics for this full operator-access HTTP surface rather than only an ad hoc header.
