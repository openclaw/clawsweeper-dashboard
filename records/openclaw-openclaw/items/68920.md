## Summary

Keep open. Current main still routes OpenAI-compatible chat completions through the normal agent ingress path, has no HTTP light-context or voice-mode prompt policy, and still lacks the agent-scoped `x-openclaw-session` hint on main; the related stable-session PR is useful but only covers one subproblem.

## What This Changes

The issue requests a low-latency `/v1/chat/completions` mode with lighter prompt context and reusable HTTP sessions for voice and realtime callers.

## Best Possible Solution

Define a documented trusted HTTP low-latency prompt policy, then wire it through the existing `promptMode` and `bootstrapContextMode` seams with tests and docs while keeping stable-session reuse as a separate compatible API change.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no HTTP `lightContext` prompt policy, and no `x-openclaw-session` stable hint, while normal HTTP sessions resolve to full prompt mode. The exact latency numbers still need the reporter's live environment to reproduce numerically.

## Solution Assessment

Unclear. The direction is valid, but an ad hoc header is not enough; the maintainable fix should first define safe prompt, tool, memory, session, and security semantics for this full operator-access HTTP surface.
