## Summary

Keep open: current main and v2026.6.8 still execute `/v1/chat/completions` as a normal agent run, and the requested low-context HTTP mode, `x-openclaw-session` stable hint, and per-agent HTTP prompt override are not present. The related stable-session PR was closed unmerged and only addressed one session-warm subproblem, not the larger prompt-assembly latency request.

## What This Changes

The issue requests a low-context or voice-oriented mode for HTTP Chat Completions, plus stable HTTP session hints and optional per-agent HTTP system prompt overrides.

## Best Possible Solution

Keep this as the canonical product issue and design an explicit HTTP low-context or voice mode that reuses existing prompt/bootstrap controls while preserving default compatibility, session semantics, security boundaries, docs, and focused tests.

## Reproduction Assessment

Yes at source level: current `/v1/chat/completions` builds a normal agent command and enters `agentCommandFromIngress`, while the session resolver only supports `x-openclaw-session-key`, `user`, or random UUID sessions. I did not run a live latency benchmark in this read-only review.

## Solution Assessment

Unclear: the requested direction is plausible, but the best permanent fix requires maintainer choice among a header, config surface, prompt-mode wiring, or a dedicated voice/API path, plus explicit safety and session semantics.
