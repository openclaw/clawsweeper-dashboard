Keep open because current main still lacks a low-context or voice-mode contract for OpenAI-compatible Chat Completions; the only related implementation path is an open partial stable-session PR, and the discussion explicitly keeps the larger prompt/context decision open.

## What This Changes

The issue requests a low-latency HTTP Chat Completions mode that avoids full agent context assembly, plus stable session reuse and an HTTP-specific prompt override for voice/realtime callers.

## Best Possible Solution

Define and document a trusted low-context Chat Completions policy, wire it through the existing prompt/bootstrap/tool/memory seams with focused Gateway tests, and handle the stable-session header through the narrower open PR or a replacement.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no `http.systemPrompt`, no stable `x-openclaw-session`, and Chat Completions still runs through normal agent ingress with full default prompt mode. I did not reproduce the live latency timing numbers.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix requires a maintainer-owned public contract for low-context HTTP behavior; the stable-session part should land through https://github.com/openclaw/openclaw/pull/69060 or a replacement while the voice-mode policy remains separate.
