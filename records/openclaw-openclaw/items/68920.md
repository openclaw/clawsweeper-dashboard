## Summary

Keep open. Current main still runs `/v1/chat/completions` through the normal agent ingress path with no HTTP light-context, voice-mode, or HTTP-specific prompt policy; the linked stable-session PR only addresses the warm-session/QMD subproblem.

## What This Changes

The issue requests a low-latency HTTP Chat Completions mode that avoids full agent context assembly, plus stable-session and HTTP-specific prompt controls.

## Best Possible Solution

Define a documented trusted HTTP low-latency prompt/session policy, then wire it through existing `promptMode` and `bootstrapContextMode` seams with Gateway tests and docs while coordinating stable-session reuse separately.

## Reproduction Assessment

Yes for the source-level gap: current main has no HTTP light-context prompt policy, and Chat Completions still uses the normal full agent ingress path. The exact latency figures still require the reporter's live environment.

## Solution Assessment

Unclear. A low-context HTTP mode is plausible, but an ad hoc header alone is not the best complete solution; maintainers need explicit prompt, tool, memory, session, and security semantics for this full operator-access surface.
