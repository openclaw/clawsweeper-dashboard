## Summary

Keep open. Current main still routes `/v1/chat/completions` through the normal agent ingress path without an HTTP light-context, voice-mode, or HTTP-specific prompt policy; the open stable-session work only covers the QMD warm-session subproblem.

## What This Changes

The issue requests a low-latency OpenAI-compatible HTTP mode that avoids full agent prompt/context assembly and supports stable session reuse for voice-style callers.

## Best Possible Solution

Define a documented trusted HTTP low-latency prompt policy, then wire it through existing `promptMode` and `bootstrapContextMode` seams with Gateway tests and docs while handling stable session reuse as a compatible, separate API decision.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no HTTP low-context prompt policy, and no `x-openclaw-session` stable hint, while ordinary HTTP sessions use full prompt mode. The exact latency figures still require the reporter's live environment to reproduce numerically.

## Solution Assessment

Unclear. The requested direction is valid, but an ad hoc header is not enough; the maintainable solution needs explicit prompt, tool, memory, session, and security semantics for this full operator-access HTTP surface.
