## Summary

Keep open: current main and v2026.6.8 still run HTTP Chat Completions through normal agent ingress with no low-context, voice-mode, stable `x-openclaw-session`, or per-agent HTTP prompt override. The closed related PR only attempted the stable-session subproblem and did not land or address full prompt assembly latency.

## What This Changes

Request a low-latency `/v1/chat/completions` mode for voice/realtime callers by reducing full agent context assembly and improving HTTP session reuse or prompt override controls.

## Best Possible Solution

Keep this as the canonical product issue and design an explicit HTTP low-context or voice mode that reuses existing prompt/bootstrap/tool-policy primitives while preserving default compatibility, security boundaries, session semantics, docs, and focused tests.

## Reproduction Assessment

Yes at source level: current main shows `/v1/chat/completions` building a normal agent command and entering `agentCommandFromIngress`, while the session resolver only supports `x-openclaw-session-key`, `user`, or random UUID sessions. I did not run a live TTFB benchmark.

## Solution Assessment

Unclear: low-context or voice mode is plausible, but the best permanent fix needs maintainer choice over the header/config shape, prompt/tool/memory safety, and session precedence.
