## Summary

Keep open. Current main still runs `/v1/chat/completions` through the normal agent ingress path without an HTTP light-context, voice-mode, or HTTP-specific prompt policy; the open stable-session work at https://github.com/openclaw/openclaw/pull/69060 only addresses one QMD/session-warm subproblem.

## What This Changes

Request a low-latency HTTP Chat Completions mode that avoids full agent context assembly and supports stable sessions for voice/realtime callers.

## Best Possible Solution

Define a documented trusted HTTP low-latency prompt policy, wire it through the existing prompt/bootstrap/tool seams with Gateway tests and docs, and keep the stable-session work split from the broader light-context decision until both are resolved.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no HTTP prompt policy, and no stable `x-openclaw-session` hint, while ordinary HTTP sessions still use the full prompt path. I did not reproduce the reporter's live 10-15s timing numbers.

## Solution Assessment

Unclear. The requested direction is valid, but the maintainable fix needs explicit prompt, tool, memory, bootstrap, session, and security semantics for the trusted HTTP surface rather than only an ad hoc header.
