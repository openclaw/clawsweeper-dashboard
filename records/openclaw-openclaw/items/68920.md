## Summary

Keep open. Current main still sends OpenAI-compatible Chat Completions through the normal agent ingress path with full prompt assembly and no HTTP light-context or voice-mode policy; the open stable-session PR only addresses the QMD warm-session subproblem.

## What This Changes

Request a low-latency `/v1/chat/completions` mode that avoids full tool, skill, memory, and bootstrap context assembly and supports stable HTTP sessions for realtime voice clients.

## Best Possible Solution

Define a documented, trusted low-latency HTTP prompt/session policy, then wire it through existing `promptMode` and `bootstrapContextMode` seams with Gateway tests and docs while keeping stable session reuse as a compatible, separately coordinated API decision.

## Reproduction Assessment

Yes for the source-level behavior: current main has no `x-openclaw-light-context`, no HTTP low-context prompt policy, and no `x-openclaw-session` stable hint while ordinary HTTP sessions use full prompt mode. The exact latency figures still require a live reproduction in the reporter's environment.

## Solution Assessment

Unclear. The requested direction is valid, but an ad hoc header alone is not the best fix; maintainers should define explicit prompt, tool, memory, session, persistence, and security semantics for this HTTP surface.
