## Summary

Keep open: current main and v2026.6.10 still route `/v1/chat/completions` through full agent ingress with no HTTP light-context path; the linked session-key work only covers a narrower cold-session subproblem.

## What This Changes

The issue requests a low-latency HTTP Chat Completions mode with lighter context and stable session handling for realtime voice and chatbot clients.

## Best Possible Solution

Define a maintainer-approved low-latency Gateway HTTP contract that scopes prompt, tools, memory, bootstrap, and session behavior explicitly, then cover it with docs, tests, upgrade semantics, and live latency proof.

## Reproduction Assessment

Yes, at source level: `/v1/chat/completions` reaches the random session fallback and full `agentCommandFromIngress` path without a light-context HTTP branch. I did not run a live TTFB benchmark in this read-only review.

## Solution Assessment

Unclear as proposed: the header and per-agent prompt ideas are plausible, but the best fix needs a maintainer-owned API contract for prompt, tool, memory, bootstrap, and session semantics before implementation.
