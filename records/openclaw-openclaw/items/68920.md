## Summary

Keep open. Current main still routes `/v1/chat/completions` through the normal agent ingress path with full prompt defaults, no HTTP light-context or voice-mode policy, and no `x-openclaw-session` stable hint; the related #69060 PR remains only a partial session-warm fix.

## What This Changes

Request low-latency OpenAI-compatible HTTP chat completions via light-context/voice-mode prompt policy, stable session reuse, and an optional HTTP-specific system prompt override.

## Best Possible Solution

Define a documented HTTP low-latency prompt policy for trusted callers, then wire it through existing `promptMode` and `bootstrapContextMode` seams with tests, docs, and a separate stable-session resolution change.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no HTTP `lightContext` config, and no `x-openclaw-session` stable hint, while regular HTTP sessions resolve to full prompt mode. The precise latency numbers still need the reporter's live environment to reproduce numerically.

## Solution Assessment

Unclear. The request is directionally valid, but a bare header is not enough; the maintainable solution should first define safe HTTP prompt-policy semantics for this full operator-access endpoint and keep #69060 or an equivalent PR scoped to stable session reuse.
