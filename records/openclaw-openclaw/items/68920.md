## Summary

Keep open. Current main still sends `/v1/chat/completions` through the normal agent ingress path without an HTTP light-context or voice-mode prompt policy, and the related stable-session work in #69060 is only a partial session-warm fix per the provided discussion.

## What This Changes

The issue requests a low-latency `/v1/chat/completions` mode that can avoid full agent context assembly and add stable session or HTTP-specific prompt controls for realtime voice callers.

## Best Possible Solution

Define an explicit HTTP prompt/context policy for low-latency callers, document its security boundary, then wire it through existing `promptMode` and `bootstrapContextMode` seams with tests and docs while keeping #69060 or an equivalent change scoped to stable session reuse.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, HTTP `lightContext`, or `x-openclaw-session` path, and normal HTTP requests resolve to full prompt mode. The precise latency figures still need the reporter's live workspace/model setup to reproduce numerically.

## Solution Assessment

Unclear. A bare header alone is not enough; the maintainable fix should first define safe HTTP prompt-policy semantics for this full operator-access endpoint, then wire the existing runtime prompt controls and document the tradeoffs.
