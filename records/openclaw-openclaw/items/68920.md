## Summary

Keep open: current main and the latest release still lack a low-context or voice-mode contract for OpenAI-compatible Chat Completions, and the linked stable-session work only covers one smaller part of the reported latency path.

## What This Changes

Request a low-context or voice-mode path for `/v1/chat/completions`, plus stable HTTP session reuse, to reduce first-token latency for realtime voice and chatbot callers.

## Best Possible Solution

Define and document a trusted low-context Chat Completions policy, wire it through the existing prompt/bootstrap/tool/memory seams with focused Gateway tests, and handle stable-session reuse through https://github.com/openclaw/openclaw/pull/69060 or a replacement.

## Reproduction Assessment

Yes for the source-level gap: current main has no `x-openclaw-light-context`, no `http.systemPrompt`, no stable `x-openclaw-session`, and Chat Completions still runs through normal agent ingress with full default prompt mode. I did not reproduce the live latency timing numbers.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix requires a maintainer-owned public contract for low-context HTTP behavior; the stable-session part should land through https://github.com/openclaw/openclaw/pull/69060 or a replacement while the voice-mode policy remains separate.
