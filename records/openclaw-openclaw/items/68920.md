## Summary

Keep open: current main and v2026.5.18 still route OpenAI-compatible Chat Completions through normal full-agent ingress and lack a low-context or HTTP prompt-override contract; the linked stable-session work only covers one smaller latency component.

## What This Changes

The issue requests a low-context/voice-mode OpenAI-compatible Chat Completions path, plus stable HTTP session reuse, to reduce high TTFB for realtime voice callers.

## Best Possible Solution

Define and document a trusted low-context Chat Completions policy, wire it through existing prompt/bootstrap/tool/memory seams with focused Gateway tests, and handle stable-session reuse through https://github.com/openclaw/openclaw/pull/69060 or a replacement.

## Reproduction Assessment

Yes for the source-level gap: current main has no low-context Chat Completions contract, no `http.systemPrompt`, no stable `x-openclaw-session`, and still forwards Chat Completions through normal agent ingress. I did not rerun the reporter's live latency benchmark.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix requires a maintainer-owned public contract for low-context HTTP behavior; the stable-session part should land through https://github.com/openclaw/openclaw/pull/69060 or a replacement while voice-mode policy remains separate.
