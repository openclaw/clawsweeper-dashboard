## Summary

Keep this PR open for maintainer review rather than closing it as stale or implemented. Current main still has no HarmonyOS app or platform surface, and the PR is a large new first-party client with gateway auth, device credential storage, build tooling, dependencies, and docs implications; the diff also has concrete setup-code/auth and supply-chain issues that should block merge until addressed.

## What This Changes

This PR adds a new HarmonyOS ArkTS app under `apps/HarmonyOS` with secure preferences, device auth, gateway connect/chat/settings UI, DevEco/Hvigor project files, screenshots/docs, and an `openclaw-harmonyos` gateway client ID validation case.

## Best Possible Solution

Maintainers should first decide whether HarmonyOS is accepted as a first-party OpenClaw companion app. If yes, harden or split the work so setup-code bootstrap auth, mobile-node/platform policy, deterministic DevEco/Hvigor dependencies, secure storage, docs/labeler coverage, and real-device validation land together; if not, incubate the HarmonyOS app outside core until the platform contract is agreed.

## Reproduction Assessment

Yes for the main PR defects: current main emits setup-code payloads with `bootstrapToken`, while the PR parser checks host fallback first and serializes only `token`/`password`, which statically reproduces the normal setup-code auth failure path. Full app runtime reproduction is unclear without DevEco Studio and a HarmonyOS device.

## Solution Assessment

No. The current PR is useful source work, but merging it as one XL first-party app addition is not the narrowest maintainable solution because it leaves setup-code auth, platform policy, deterministic build inputs, and security review unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Decode setup codes before host fallback:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:829-832`
  - body: `openclaw qr --setup-code-only` emits base64url JSON, which is usually a bare alphanumeric string. Because this parser accepts bare alphanumeric strings as hostnames before trying to decode the payload, normal setup codes will be treated as `<setup-code>:18789` instead of extracting the gateway URL and credential.
  - confidence: 0.92
- **[P1] Send setup bootstrap tokens as bootstrapToken:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:848-850`
  - body: Current setup-code payloads carry `bootstrapToken`, and the gateway reads it from `auth.bootstrapToken`; this PR only reads `token`/`gatewayToken` and the session auth object only serializes `token` or `password`. Normal QR/setup-code pairing will therefore connect without the bootstrap credential and be rejected on bootstrap-token gateways.
  - confidence: 0.94
- **[P1] Initialize secure prefs before reading saved state:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:139-140`
  - body: `mainViewModel` starts `initialize()` from a module-level constructor, but `securePrefs.init(this.context)` is only kicked off later from `EntryAbility.onCreate`. Since `SecurePrefs` returns defaults when the preference handles are still null, startup can lose the saved host, token/password, and onboarding state for this session.
  - confidence: 0.86
- **[P2] Gate requests until gateway auth completes:** `apps/HarmonyOS/entry/src/main/ets/api/GatewaySession.ets:464-468`
  - body: `connected` is set to true immediately after the WebSocket opens, before `performAuthentication()` waits for the challenge and sends `connect`. Public callers of `isConnected()`, `request()`, or `sendNodeEvent()` can therefore enqueue pre-auth frames during the handshake window.
  - confidence: 0.84
- **[P2] Wire HarmonyOS into mobile platform policy:** `src/gateway/protocol/client-info.ts:13`
  - body: The PR adds the `openclaw-harmonyos` client id and reports platform `harmonyos`, but the gateway mobile-node predicate and node command policy on main only recognize iOS/iPadOS/Android as mobile defaults. A HarmonyOS-only mobile node would not satisfy WebChat `talk.mode`, and node command defaults would fall back to the unknown-platform policy.
  - confidence: 0.78
- **[P2] Run hvigor without a shell command string:** `apps/HarmonyOS/hvigorw.js:28-32`
  - body: The wrapper joins forwarded argv into a single command string and passes it to `execSync`, which invokes a shell and can interpret metacharacters in arguments; it also runs `hvigor` from PATH after checking for a local install. Use `spawnSync`/`execFileSync` with an args array and the local tool entrypoint.
  - confidence: 0.91
- **[P2] Commit deterministic HarmonyOS dependency inputs:** `apps/HarmonyOS/.gitignore:26-27`
  - body: The app adds a ranged third-party dependency while `.gitignore` excludes Harmony lockfiles, so different developers or CI runs can resolve different package contents for the same source commit. First-party app builds should commit the appropriate lockfile or otherwise pin/verify the DevEco dependency graph.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces concrete build and supply-chain risks in the HarmonyOS wrapper/dependency setup and should not merge without security review.

Concerns:

- **[medium] Shell execution of forwarded wrapper arguments:** `apps/HarmonyOS/hvigorw.js:29`
  - body: `apps/HarmonyOS/hvigorw.js` builds `hvigor ${args}` from `process.argv.slice(2).join(' ')` and runs it through `execSync`, allowing shell metacharacters in wrapper arguments to be interpreted during local builds.
  - confidence: 0.91
- **[medium] Automatic install plus ignored lockfiles:** `apps/HarmonyOS/.gitignore:26`
  - body: The wrapper auto-runs `npm install` when a local Hvigor package is absent, while the app adds a ranged Harmony dependency and ignores lockfiles. That makes first-party app builds non-deterministic and broadens the supply-chain surface.
  - confidence: 0.84
- **[low] New client credential storage needs secops review:** `apps/HarmonyOS/entry/src/main/ets/common/SecurePrefs.ets:367`
  - body: The PR adds client-side storage and use of gateway tokens/passwords and device key material. The approach may be reasonable, but it is a new auth/secret-handling surface and overlaps CODEOWNERS secops-protected gateway auth paths.
  - confidence: 0.72
