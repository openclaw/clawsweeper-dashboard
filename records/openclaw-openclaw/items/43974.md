## Summary

Keep open: current main still lacks HarmonyOS support, and this PR still has concrete source-level auth, startup, build, and supply-chain defects plus insufficient real-device proof. First-party HarmonyOS app ownership needs maintainer judgment before repair automation is useful.

## What This Changes

Adds a new HarmonyOS ArkTS app under apps/HarmonyOS, registers a HarmonyOS gateway client ID with validation coverage, and adds DevEco/hvigor project docs and assets.

## Best Possible Solution

Decide whether OpenClaw will own HarmonyOS as a first-party app; if yes, land a hardened integration aligned with setup-code/bootstrap auth, deterministic build inputs, public docs/labeler/changelog coverage, and real-device validation.

## Reproduction Assessment

Yes for the main source-level PR defects: current setup-code payloads use bootstrapToken, current gateway auth reads bootstrapToken, and the PR parser/serializer/startup paths conflict with those contracts. Full runtime reproduction on a HarmonyOS device was not established.

## Solution Assessment

No. A complete XL first-party app is not the narrowest maintainable path while ownership, auth compatibility, deterministic build inputs, and real-device proof remain unresolved.
