## Summary

Keep this PR open. Current main still has no HarmonyOS app, so the contribution is not obsolete, but the branch is not mergeable as-is: the setup-code auth path is incompatible with current bootstrap-token pairing, saved secure preferences can be read before initialization, HarmonyOS is not wired into the current mobile-node policy, and the new build wrapper/dependency setup needs security and ownership review.

## What This Changes

The PR adds a new first-party HarmonyOS ArkTS app under `apps/HarmonyOS`, DevEco/Hvigor project files and docs/screenshots, plus a gateway client-id addition and validation test for `openclaw-harmonyos`.

## Best Possible Solution

Accept HarmonyOS only after maintainers decide it belongs as a first-party companion app; then land a hardened, smaller integration that matches current setup-code/bootstrap auth, mobile-node policy, deterministic DevEco/Hvigor dependencies, docs/labeler/changelog coverage, and real-device validation. If that product decision is not made, keep the implementation outside core or incubate it until the platform contract is settled.

## Reproduction Assessment

Yes for the main correctness defects: current main emits setup-code payloads with `bootstrapToken`, while the PR parses host fallback first and serializes only `token`/`password`, so the normal setup-code auth failure is statically reproducible from source. Full HarmonyOS runtime reproduction is unclear without DevEco Studio and a real or representative HarmonyOS device.

## Solution Assessment

No. Adding a complete XL first-party app in one PR is not the narrowest maintainable path while auth compatibility, mobile platform policy, deterministic dependency inputs, and security review remain unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Decode setup codes before host fallback:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:829-832`
  - body: `openclaw qr --setup-code-only` emits a base64url JSON payload that can look like a bare alphanumeric host. Because this parser accepts host fallback before decoding, normal setup codes are treated as `<setup-code>:18789` instead of extracting the gateway URL and credential.
  - confidence: 0.92
- **[P1] Send bootstrap setup credentials as bootstrapToken:** `apps/HarmonyOS/entry/src/main/ets/api/GatewaySession.ets:263-267`
  - body: Current setup-code payloads carry `bootstrapToken`, and gateway bootstrap auth reads `auth.bootstrapToken`. This serializer only sends `token` or `password`, so setup-code pairing connects without the bootstrap credential and is rejected on bootstrap-token gateways.
  - confidence: 0.94
- **[P1] Initialize secure prefs before reading saved state:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:139-140`
  - body: `mainViewModel` starts `initialize()` from the constructor, but `securePrefs.init(this.context)` is kicked off later by `EntryAbility`. Reads before initialization return defaults, so saved host, credentials, and onboarding state can be lost for the session.
  - confidence: 0.86
- **[P2] Wire HarmonyOS into mobile platform policy:** `src/gateway/protocol/client-info.ts:13`
  - body: The PR adds `openclaw-harmonyos` as a client ID and the app reports platform `harmonyos`, but current mobile-node policy only recognizes iOS, iPadOS, and Android. A HarmonyOS-only mobile node will not satisfy WebChat `talk.mode` availability.
  - confidence: 0.78
- **[P2] Run hvigor without a shell command string:** `apps/HarmonyOS/hvigorw.js:29`
  - body: The wrapper joins forwarded argv into `hvigor ${args}` and passes it to `execSync`, so shell metacharacters in build arguments are interpreted and PATH resolution decides which hvigor runs. Use a non-shell invocation with an args array and a deterministic local tool entrypoint.
  - confidence: 0.91
- **[P2] Commit deterministic HarmonyOS dependency inputs:** `apps/HarmonyOS/.gitignore:26-27`
  - body: The app adds a ranged third-party dependency while `.gitignore` excludes Harmony lockfiles. First-party app builds should commit the appropriate lockfile or otherwise pin and verify the DevEco dependency graph so developers and CI resolve the same package contents.
  - confidence: 0.82
