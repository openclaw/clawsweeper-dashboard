## Summary

Keep this PR open for maintainer review, but it is not merge-ready: current main does not have a HarmonyOS app, while the branch still has source-proven setup-code/auth/startup defects, supply-chain/build-wrapper concerns, and missing real-device behavior proof.

## What This Changes

The PR adds a HarmonyOS ArkTS client app under `apps/HarmonyOS`, registers an `openclaw-harmonyos` gateway client ID, and adds DevEco/hvigor project files, assets, docs, and validation coverage.

## Best Possible Solution

Decide whether OpenClaw will own HarmonyOS as a first-party app; if yes, land a hardened app aligned with setup-code/bootstrap auth, deterministic build inputs, labeler/docs/changelog coverage, and real-device validation.

## Reproduction Assessment

Yes for the source-level PR defects: current setup codes carry `bootstrapToken`, gateway auth reads `auth.bootstrapToken`, and the PR parser/serializer/startup paths conflict with those contracts. I did not establish a full runtime reproduction on a physical HarmonyOS device.

## Solution Assessment

No. A large new first-party app is not the narrowest maintainable merge path while ownership, setup-code/auth compatibility, deterministic build inputs, and real-device proof remain unresolved.
