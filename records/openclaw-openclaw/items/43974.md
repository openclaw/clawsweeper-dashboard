## Summary

Keep open. Current main still lacks the HarmonyOS app and `openclaw-harmonyos` gateway client ID, while this PR adds a large new first-party app surface with blocking auth/setup-code defects, supply-chain concerns, and insufficient real-device proof.

## What This Changes

Adds a HarmonyOS ArkTS app under `apps/HarmonyOS`, registers an `openclaw-harmonyos` gateway client ID, and adds DevEco/hvigor project files, docs, assets, and validation coverage.

## Best Possible Solution

Decide whether OpenClaw will own HarmonyOS as a first-party app; if yes, land a hardened app aligned with current setup-code/bootstrap auth, deterministic build inputs, labeler/docs/changelog coverage, and real-device validation.

## Reproduction Assessment

Yes for the source-level PR defects: current setup codes carry `bootstrapToken`, gateway auth reads `auth.bootstrapToken`, and the PR parser/serializer/startup paths conflict with those contracts. I did not establish a full runtime reproduction on a physical HarmonyOS device.

## Solution Assessment

No. A large new first-party app is not the narrowest maintainable merge path while ownership, setup-code/auth compatibility, deterministic build inputs, and real-device proof remain unresolved.
