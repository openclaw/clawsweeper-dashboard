## Summary

Keep open: current main still has no HarmonyOS app, and this PR still has blocking setup-code/bootstrap auth bugs, persisted-state loss, security/supply-chain concerns, and insufficient real-device proof. This needs first-party app ownership review, not conservative cleanup closure.

## What This Changes

The PR adds a new HarmonyOS ArkTS client app under apps/HarmonyOS, registers openclaw-harmonyos as a gateway client ID, and adds DevEco/hvigor project files, assets, docs, and validation coverage.

## Best Possible Solution

Decide whether OpenClaw will own HarmonyOS as a first-party app; if yes, land a hardened app aligned with setup-code/bootstrap auth, deterministic build inputs, labeler/docs/changelog coverage, and real-device validation.

## Reproduction Assessment

Yes for the source-level PR defects: current setup codes encode base64url JSON with bootstrapToken, and gateway auth reads auth.bootstrapToken, while the PR parses host fallback before decoding and serializes only token/password. I did not establish a full runtime reproduction on a physical HarmonyOS device.

## Solution Assessment

No. A large new first-party app is not ready as-is while setup-code auth, persisted state, deterministic build inputs, and real behavior proof remain unresolved; the app should match existing mobile pairing contracts before merge.
