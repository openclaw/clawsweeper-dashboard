## Summary

Keep this PR open: current main does not implement HarmonyOS support, and the proposed first-party app still has concrete auth, startup, mobile-policy, build, and supply-chain blockers plus a maintainer product/security ownership decision.

## What This Changes

This PR adds a HarmonyOS ArkTS client app under `apps/HarmonyOS`, DevEco project/docs/screenshots, and a gateway client-id/test update for `openclaw-harmonyos`.

## Best Possible Solution

Accept HarmonyOS only after maintainers decide to own it as a first-party companion app; then land a hardened integration that matches current mobile setup-code/bootstrap auth, mobile-node policy, deterministic DevEco/Hvigor inputs, docs/labeler/changelog coverage, and real-device validation.

## Reproduction Assessment

Yes for the main correctness defects: current main emits setup-code payloads with `bootstrapToken`, gateway auth reads `auth.bootstrapToken`, and mobile policy currently excludes `harmonyos`. Full HarmonyOS runtime reproduction remains unclear without DevEco Studio and a real or representative device.

## Solution Assessment

No. Adding a complete XL first-party app in one PR is not the narrowest maintainable solution while auth compatibility, mobile platform policy, deterministic build inputs, and first-party ownership remain unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Decode setup codes before host fallback:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:829-832`
  - body: `openclaw qr --setup-code-only` emits a base64url JSON payload that can look like a bare alphanumeric host. Because this parser accepts host fallback before decoding, normal setup codes are treated as `<setup-code>:18789` instead of extracting the gateway URL and credential.
  - confidence: 0.92
- **[P1] Send bootstrap credentials as bootstrapToken:** `apps/HarmonyOS/entry/src/main/ets/api/GatewaySession.ets:263-267`
  - body: Current setup-code payloads carry `bootstrapToken`, and gateway bootstrap auth reads `auth.bootstrapToken`. This serializer only sends shared `token` or `password`, so setup-code pairing connects without the bootstrap credential and is rejected on bootstrap-token gateways.
  - confidence: 0.94
- **[P1] Initialize secure prefs before reading saved state:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:139-140`
  - body: `mainViewModel` starts initialization from the constructor, but `securePrefs.init(this.context)` is started later by `EntryAbility`. Reads before secure prefs initialization return defaults, so saved host, credentials, and onboarding state can be lost for the session.
  - confidence: 0.86
- **[P2] Wire HarmonyOS into mobile platform policy:** `src/gateway/protocol/client-info.ts:13`
  - body: The PR adds `openclaw-harmonyos` and the app reports platform `harmonyos`, but current mobile-node policy only recognizes iOS, iPadOS, and Android. A HarmonyOS-only mobile node will not satisfy WebChat `talk.mode` availability.
  - confidence: 0.78
- **[P2] Run hvigor without a shell command string:** `apps/HarmonyOS/hvigorw.js:29`
  - body: The wrapper joins forwarded argv into `hvigor ${args}` and passes it to `execSync`, so shell metacharacters in build arguments are interpreted and PATH decides which `hvigor` runs. Use a non-shell invocation with an args array and a deterministic local tool entrypoint.
  - confidence: 0.91
- **[P2] Commit deterministic HarmonyOS dependency inputs:** `apps/HarmonyOS/.gitignore:26-27`
  - body: The app declares a ranged third-party dependency while `.gitignore` excludes Harmony lockfiles. First-party app builds should commit the appropriate lockfile or otherwise pin and verify the DevEco dependency graph so developers and CI resolve the same package contents.
  - confidence: 0.82
