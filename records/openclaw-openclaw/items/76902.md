## Summary

The PR is not ready to merge: it fixes the CLI cron path to be non-owner and adds the changelog entry, but the new scoped MCP bearer grant only applies a generic owner-only allowlist after tool construction, so the allowlisted `cron` tool is still unrestricted instead of current-job self-removal only.

## What This Changes

The branch changes CLI-backed isolated cron runs to use non-owner loopback authorization, adds a scoped MCP bearer-token allowlist for cron self-removal, extends CLI/gateway tests and types, and adds a changelog entry.

## Best Possible Solution

Keep the non-owner CLI authorization change, but make the scoped cron grant carry the active job id through the loopback path and construct the same current-job-removal-only cron tool used by the embedded isolated path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run an isolated cron job through a compatible CLI runtime with `toolsAllow: ["cron"]`; on the PR branch the scoped token only carries an owner-only allowlist, while loopback tool construction has no job id/self-remove scope.

## Solution Assessment

No. Switching CLI-backed isolated cron runs to non-owner auth is the right direction, but the scoped grant must be job-bound and action-limited before this is the narrow maintainable fix.
