## Summary

Keep this PR open: the latest branch fixes the original owner-token path and the changelog gap, but the scoped loopback grant still exposes the full cron owner-only tool instead of the intended self-removal-only cron grant.

## What This Changes

This PR changes CLI-backed isolated cron runs to use non-owner loopback authorization, adds scoped MCP bearer-token handling for owner-only tool grants, updates cron/gateway/CLI tests, and adds a changelog entry.

## Best Possible Solution

Keep CLI-backed isolated cron runs non-owner, but thread a job-scoped cron self-removal grant through the CLI/MCP loopback so `toolsAllow: ["cron"]` can only remove the active cron job.

## Reproduction Assessment

Yes. Source inspection shows the PR's scoped token can make `cron` visible to non-owner loopback callers, while the loopback tool factory lacks the active job id needed to apply the existing self-removal-only cron wrapper.

## Solution Assessment

No. Switching CLI-backed isolated cron runs to non-owner is the right direction, but a raw owner-only tool allowlist is broader than the embedded path's job-scoped self-removal grant.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep the scoped cron grant self-removal-only:** `src/gateway/mcp-http.runtime.ts:56`
  - body: The new scoped bearer path re-adds `cron` with only `ownerOnlyToolAllowlist`, after loopback tools were created without the active cron `jobId` or `cronSelfRemoveOnlyJobId`. That means a CLI-backed isolated cron run with `toolsAllow: ["cron"]` can see the normal cron tool and call list/add/update/run/remove for arbitrary jobs instead of only removing its current job. Please carry a job-scoped grant through loopback and cover denied non-remove and non-current-job calls.
