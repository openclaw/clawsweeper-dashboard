## Summary

Keep open: the least-privilege direction is correct, but the patch drops the embedded path's scoped cron self-removal grant for CLI-backed isolated cron runs and lacks the required changelog entry for a runtime auth fix.

## What This Changes

The branch changes CLI-backed isolated cron execution to pass `senderIsOwner: false` to `runCliAgent` and adds a regression test for that CLI runtime dispatch path.

## Best Possible Solution

Keep CLI-backed isolated cron runs non-owner while preserving the same narrowly scoped cron self-removal grant that embedded isolated cron already has, then document the runtime auth fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an isolated cron job through a compatible CLI runtime and current main passes `senderIsOwner: true`; the proposed change flips that to non-owner but does not carry the existing cron self-removal grant through the CLI loopback path.

## Solution Assessment

No. Setting CLI-backed isolated cron runs to non-owner is the right core fix, but it should also thread an equivalent self-removal-only cron grant through the CLI/MCP loopback path and add the required changelog entry.
