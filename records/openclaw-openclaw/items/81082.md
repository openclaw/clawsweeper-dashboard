## Summary

Current main fixes this false-positive path: `channels.status` uses the central gateway event-loop health monitor, and that monitor now requires delay co-evidence before ELU/CPU can mark the gateway degraded. The fix landed through the merged gateway sampler PR and shipped in `v2026.5.9-beta.1`; the stable `v2026.5.7` release predates it.

## What This Changes

The issue requests that idle `openclaw channels status` stop reporting `Gateway event loop degraded` when `eventLoopDelayMaxMs=0` but utilization and CPU ratios are saturated artifacts.

## Best Possible Solution

Keep the merged central gateway sampler fix and direct affected 5.6/5.7 users to a release containing PR 77028; open a new issue only if current main or a post-fix release still shows zero-delay `channels status` degradation.

## Reproduction Assessment

No for current main: the source and regression tests now prove that the reported zero-delay ELU/CPU sample does not mark the gateway degraded. The issue body remains strong historical reproduction evidence for 2026.5.6.

## Solution Assessment

Yes. Fixing the central gateway event-loop health classifier is the narrow maintainable solution because it keeps the existing payload schema and corrects health, readiness, and `channels status` consistently, instead of adding a formatter-only filter.
