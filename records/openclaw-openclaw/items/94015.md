## Summary

Keep open. The PR is a useful stale-payload mitigation, but it does not fix the linked first-audio latency bug because callback payloads are only read after the awaited embedded run returns.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and an `onBlockReply` collector to voice-call response generation, then prefers collected callback payloads over completed-run payloads when extracting spoken TTS text.

## Best Possible Solution

Rework voice-call delivery so a valid block reply can start TTS before the embedded run waits on compaction, or explicitly narrow this PR to a stale-payload mitigation while leaving the latency issue open.

## Reproduction Assessment

Yes at source level: the linked issue gives live logs, and current code shows voice-call speaks only after `generateVoiceResponse()` resolves while the embedded runner can flush block replies before compaction wait. I did not run a live Twilio reproduction in this read-only review.

## Solution Assessment
