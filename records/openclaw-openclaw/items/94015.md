## Summary

Keep open. The PR is a useful stale-payload mitigation, but it does not yet fix the linked voice-call latency bug because TTS still receives the collected callback text only after the embedded run returns from compaction wait.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and `onBlockReply` collection to voice-call response generation, plus a unit test that prefers callback payloads over completed payloads.

## Best Possible Solution

Rework the voice-call path so the first valid block reply can start TTS before the embedded run waits on compaction, while runner cleanup continues and focused tests plus redacted real voice-call proof cover the timing.

## Reproduction Assessment

Yes, source-level. Current main and PR head both have the voice-call webhook speak only after `generateVoiceResponse()` returns, while the embedded runner can flush block replies and then continue waiting on compaction.

## Solution Assessment

No. The patch is a plausible stale-payload mitigation, but it is not the best fix for the reported latency because it only reads callback payloads after the awaited embedded run settles.
