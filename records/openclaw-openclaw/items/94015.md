## Summary

Keep open. The PR uses the right embedded-agent seam, but it is not merge-ready because it marks early TTS as spoken before `manager.speak()` has actually succeeded, so a failed early playback attempt can suppress the existing final speech fallback and drop the audible response.

## What This Changes

The PR adds an early voice-call TTS callback from `generateVoiceResponse`, wires the webhook to speak the first block reply before compaction finishes, and adds response-generator regression tests.

## Best Possible Solution

Keep the `onBlockReply` early handoff direction, but make final-speak suppression depend on confirmed early playback success and prove both early delivery and fallback failure behavior with focused tests plus redacted live voice-call evidence.

## Reproduction Assessment

Yes at source level: current main waits for `runEmbeddedAgent()` before speaking, and the PR-head code can skip the final speak after an early callback even when `manager.speak()` returns failure. I did not run a live Twilio call in this read-only review.

## Solution Assessment

No as currently implemented: `onBlockReply` is the right narrow seam, but the PR must tie fallback suppression to successful early playback rather than callback invocation.
