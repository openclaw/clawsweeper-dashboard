## Summary

Keep open; the PR is not merge-ready because it collects early block replies but still waits for the embedded run to settle before voice-call starts speaking, so the reported first-audio latency remains.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and an `onBlockReply` collector to voice-call response generation, then prefers collected callback payloads over final embedded-run payloads.

## Best Possible Solution

The voice-call path should start or queue TTS from a valid block-reply callback before the embedded run settles, while preserving final-result fallback and error handling for runs that never emit usable spoken text.

## Reproduction Assessment

Yes for source reproduction: the linked issue has live Twilio evidence, and current source shows the webhook speaks only after `generateVoiceResponse()` returns while the embedded result still waits through compaction.

## Solution Assessment

No; collecting `onBlockReply` payloads is useful, but reading them only after the awaited embedded run returns is not the best fix for first-audio latency.
