## Summary

Keep open. The PR is a focused attempt at the linked voice-call compaction latency bug, but it still waits for the embedded run to finish before TTS can speak the collected block reply, so it does not yet fix the reported early-audio behavior.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and an `onBlockReply` payload collector to voice-call response generation, plus a unit test preferring callback payloads over stale final payloads.

## Best Possible Solution

Deliver the first valid voice-call block reply to the TTS/playback path before the embedded run can wait on compaction, with regression coverage for delayed completion and redacted real voice-call proof.

## Reproduction Assessment

Yes, at source level: current main and the PR both show the webhook speaks only after `generateVoiceResponse()` resolves, while the embedded runner can flush block replies before a later compaction wait. I did not replay a live Twilio call in this read-only review.

## Solution Assessment

No. The PR is a plausible stale-payload mitigation, but it is not the best fix for the reported latency because it only reads callback payloads after the awaited embedded run settles.
