## Summary

Keep open: current main and the latest release still have the voice-call compaction wait, and this PR is useful context, but the branch is not merge-ready because it only reads callback payloads after the embedded run finishes and lacks real voice-call proof.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and an `onBlockReply` collector to voice-call response generation, plus unit coverage preferring callback payloads over final payloads.

## Best Possible Solution

Hand the first valid block reply to the voice-call TTS path before the embedded run waits on compaction, then prove that path with focused regression coverage and redacted live voice-call evidence.

## Reproduction Assessment

Yes, at source level: current main and the PR both show the webhook speaks only after `generateVoiceResponse()` resolves, while the embedded runner can wait on compaction after block replies flush. I did not replay a live Twilio call in this read-only review.

## Solution Assessment

No. The patch is a plausible stale-payload mitigation, but it is not the best fix for the reported latency because it only consumes callback payloads after the awaited embedded run settles.
