## Summary

Keep open: this is the right bug area, but the patch is not merge-ready because it only stores `onBlockReply` payloads and still waits for the embedded run to return before voice-call TTS can speak.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and an `onBlockReply` payload collector to voice-call response generation, preferring callback payloads over final embedded-run payloads.

## Best Possible Solution

Rework the voice-call path so the first valid block reply can be handed to TTS before the embedded run waits on compaction, then prove that with a delayed-completion regression test and redacted real voice-call evidence.

## Reproduction Assessment

Yes, at source level: current main and the PR both show the webhook speaks only after `generateVoiceResponse()` resolves, while the embedded runner can still wait on compaction after block replies flush. I did not replay a live Twilio call in this read-only review.

## Solution Assessment

No. The patch is a plausible stale-payload mitigation, but it is not the best fix for the reported latency because callback payloads are only used after the awaited embedded run settles.
