## Summary

Keep open. The branch is a focused attempt at the linked voice-call bug, but it only records early block replies and still waits for the embedded run to finish before `manager.speak()` is called, so the reported compaction latency remains unresolved.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and an `onBlockReply` collector to voice-call response generation, then prefers collected callback payloads over final `result.payloads` in tests and production code.

## Best Possible Solution

Rework the voice-call path so the first valid block reply is handed to TTS before the embedded run waits on compaction, or explicitly choose a voice-lane compaction bypass/decoupling strategy, then prove it with focused tests and redacted live voice-call evidence.

## Reproduction Assessment

Yes, source-level: current main and the PR both show the webhook only speaks after `generateVoiceResponse()` resolves, while the embedded attempt can still wait on compaction after flushing block replies. I did not replay a live Twilio call in this read-only review.

## Solution Assessment

No. This is a plausible stale-payload mitigation, but it is not the best fix for the reported latency because the callback payload is only consumed after the awaited embedded run settles.
