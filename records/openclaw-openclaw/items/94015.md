## Summary

Keep open: the branch is a focused attempt at the linked voice-call bug, but it still consumes block-reply payloads only after the awaited embedded run settles, so it does not prove or implement first-audio delivery before compaction wait. Real behavior proof is also still test/build/type output only, with no live or redacted voice-call log proof of improved first-audio timing.

## What This Changes

The PR adds `blockReplyBreak: "text_end"` and an `onBlockReply` collector to voice-call response generation, preferring collected block payloads over final `result.payloads` and adding unit coverage for that preference.

## Best Possible Solution

Rework voice-call so the first valid block reply is handed to TTS before the embedded run waits on compaction, or explicitly choose a voice-lane compaction bypass/decoupling strategy, then prove it with focused tests and redacted real voice-call evidence.

## Reproduction Assessment

Yes, at source level: current main and the PR both show the webhook speaks only after `generateVoiceResponse()` resolves, while the embedded runner can still wait on compaction after block replies flush. I did not replay a live Twilio call in this read-only review.

## Solution Assessment

No. The patch is a plausible stale-payload mitigation, but it is not the best fix for the reported latency because it only consumes callback payloads after the awaited embedded run settles.
