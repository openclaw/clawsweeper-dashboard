## Summary

Keep open. The PR targets a valid linked sandbox diagnostic bug, but the current diff can still emit the wrong container path for supported custom workdir configurations and it lacks after-fix real behavior proof from a real sandboxed run.

## What This Changes

This PR extends sandbox path-escape errors in the core path guard, sandbox fs mount resolver, OpenShell fs bridge, and changelog to mention the in-container workspace path.

## Best Possible Solution

Keep the diagnostic-only direction, but build the escape hint from the configured container root at the actual throwing guard and add focused coverage for custom workdir sandboxes plus real sandbox proof.

## Reproduction Assessment

Yes, source-reproducible: current main rejects an outside path through `resolveSandboxFsPathWithMounts` without using the available container mapping. I did not run a live Docker/OpenShell session in this read-only review.

## Solution Assessment

No. The requested diagnostic-only fix is the right shape, but this implementation should use the configured container workdir before the escape throw and prove the after-fix behavior in a real sandboxed run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use configured workdir before the escape throw:** `src/agents/sandbox/fs-paths.ts:160`
  - body: For escaping inputs, the preceding `resolveSandboxPath(...)` call throws before this new `params.defaultContainerRoot` hint can run. In sandboxes configured with `sandbox.docker.workdir: "/sandbox-root"`, the earlier generic throw now still tells the agent to use `/workspace/`, so the PR returns the wrong recovery path for a supported config. Build this error from `defaultContainerRoot` at the throwing guard and add custom-workdir coverage.
  - confidence: 0.88
