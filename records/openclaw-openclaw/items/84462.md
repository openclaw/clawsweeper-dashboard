## Summary

Keep open. Source inspection shows current main can still drop a generic `/command@TargetBot` group message behind `requireMention` because Telegram mention detection only accepts standalone `@username` text or `mention` entities, while the existing slash-command bypass only covers known authorized OpenClaw control commands.

## What This Changes

The issue asks OpenClaw to treat Telegram `/command@TargetBot` messages in groups as an explicit mention for the targeted bot when `requireMention` is enabled.

## Best Possible Solution

Add a narrow Telegram-owned parser path that treats a `bot_command` addressed to the current bot username as `wasMentioned`, while preserving other-bot ignoring, reply-to-bot implicit mentions, and command authorization semantics.

## Reproduction Assessment

Yes, source-level. A group message like `/ping@TargetBot` with a Telegram `bot_command` entity is not accepted by the current mention helper, so it reaches the `no-mention` skip path when `requireMention` is true unless it is also a known authorized OpenClaw control command.

## Solution Assessment

Yes, the best fix is narrow. Recognize addressed Telegram `bot_command` text for the current bot as an explicit mention, and add regression coverage that also proves commands addressed to another bot still stay quiet.
