## Summary

Keep open. Current main does not already implement the proposed scoped memory RPC surface, and this branch is not merge-ready because it removes existing Gateway/protocol contracts and retains unbounded in-process RPC state.

## What This Changes

The PR adds scoped `memory.*` Gateway RPC handlers, memory protocol schemas, generated Swift models, and related memory-core scoping/provenance/indexing changes.

## Best Possible Solution

Land the memory management RPCs as an additive, current-main-based Gateway/protocol change after the prerequisite memory stack is resolved; preserve existing plugin/session contracts, add bounded retention for jobs/source refs, document the new RPCs, regenerate TS/Swift artifacts from current schemas, and validate the focused Gateway/protocol/memory lanes.

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature PR, but the blocking findings have a high-confidence static reproduction path by comparing PR head `bb0481c0ba56c8b0af18e305b905052a5af0b9fb` with current main `442f59508e4cad2448af364370f184fabf0b5d2a`.

Is this the best way to solve the issue? No. The scoped memory RPC direction is plausible, but the current branch should not merge until it is rebased and made strictly additive with existing Gateway/protocol contracts preserved and retention bounds added.
