## Summary

Keep open. Current main still lacks the proposed scoped Memory Gateway RPCs, so this PR is not obsolete, but the patch is not mergeable as-is because it removes existing Gateway/protocol contracts and retains unbounded in-process memory state in new long-lived RPC paths.

## What This Changes

The PR adds scoped Memory Gateway `memory.*` RPC handlers, protocol schemas, generated Swift models, and memory-core scoping/indexing support for status, source listing, debug search, index jobs, and server-issued source opening.

## Best Possible Solution

Rebase and narrow the branch after the prerequisite memory PRs are resolved, preserve all existing Gateway/protocol/generated-client contracts, add bounded retention for new memory job/source-ref state, document the additive RPCs, regenerate TS/Swift protocol artifacts from current schemas, and validate with focused gateway/protocol/memory tests plus the changed gate.

## Reproduction Assessment

Not applicable as a feature PR, but the review findings have a high-confidence static reproduction path: compare the PR diff against current main and verify that existing handlers/schemas/generated fields are removed while the main handlers still consume them.

## Solution Assessment

No. The scoped memory RPC direction is plausible, but this is not the best merge shape until it is additive against current main, bounded for long-lived Gateway state, and sequenced with the open prerequisite stack.

  - body: The generated Swift models delete existing protocol types and fields such as node presence/event payloads, `ModelsListParams.view`, and tool `risk`/`tags` while current schemas and handlers still define or consume them. The memory RPC addition needs regenerated clients from current main without unrelated contract removals.
  - confidence: 0.89
- **[P2] Bound retained memory RPC job state:** `src/gateway/server-methods/memory.ts:53-55`
  - body: Each `memory.index.run` inserts a job into the module-level `indexJobs` map, but neither the run nor jobs endpoint prunes or caps completed/failed entries. Repeated indexing in a long-lived Gateway will retain job records until process restart.
  - confidence: 0.91
- **[P2] Prune source refs when issuing search results:** `src/gateway/server-methods/memory.ts:188-196`
  - body: `memory.search.debug` can mint up to 50 `sourceRef` entries per call, but stale refs are pruned only inside `memory.source.open`. Search-only/debug-only workflows will accumulate expired refs until a later open call happens.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff does not add dependency, workflow, secret, or supply-chain changes, but the new Gateway RPC state retention has a concrete availability risk before merge.

Concerns:

- **[medium] Bound memory RPC retained state:** `src/gateway/server-methods/memory.ts:53`
  - body: `memory.index.run` retains jobs without a cap or TTL, and `memory.search.debug` can create source refs that are not pruned unless a later open call occurs. An authorized client repeatedly calling these RPCs can grow Gateway process memory in a long-lived server.
