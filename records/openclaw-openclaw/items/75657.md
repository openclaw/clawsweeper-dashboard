## Summary

Keep open. Current main still initializes the explicit local GGUF embedding provider inside the Gateway Node.js process, so startup deferral and Promise.race timeouts do not isolate native node-llama-cpp model/context work from the event loop.

## What This Changes

Prevent explicit local GGUF memory embedding initialization from blocking Gateway startup and first-use memory paths on slow ARM64/Pi hosts.

## Best Possible Solution

Keep the local embedding provider contract, but move GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary with regression coverage for startup, deep status, boot sync, and first semantic search.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, QMD boot sync with startup enabled, memory index/sync, or first `memory_search`; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy startup, deferral, and Promise.race timers cannot interrupt native work that blocks the main Node.js event loop.
