## Summary

Keep open. Current main still creates the explicit local GGUF embedding provider inside the Gateway Node.js process; recent startup/status mitigations avoid some eager work, but they do not isolate node-llama-cpp model/context creation from the event loop when local embeddings are actually initialized.

## What This Changes

The issue asks for local GGUF memory embedding initialization to stop blocking the Gateway event loop on slow ARM64/Pi startup and first-use paths.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while keeping the memory embedding provider contract stable and adding regression coverage for startup, deep status, sync, and first search paths.

## Reproduction Assessment

Yes. This is source-reproducible rather than live-reproduced: configure explicit local memory embeddings and trigger deep memory status, memory sync/indexing, QMD boot sync with startup enabled, or first memory_search, and current main still reaches in-process node-llama-cpp initialization.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix; lazy initialization or Promise.race timeouts can reduce eager work but cannot keep the Gateway event loop responsive while native initialization blocks it.
