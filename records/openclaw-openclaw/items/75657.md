## Summary

Keep open. Current main still leaves explicit local GGUF embedding initialization and embedding calls in the Gateway Node.js process; recent startup/status mitigations reduce some eager probes, but they do not isolate node-llama-cpp model/context work from the event loop.

## What This Changes

The issue asks OpenClaw to stop local GGUF memory embedding warmup from blocking Gateway startup on slow ARM64/Pi hosts.

## Best Possible Solution

Keep the local embedding provider contract, but move GGUF model resolution, model/context creation, and embedding execution behind a worker-thread or child-process boundary with regression coverage for startup, first-use, status, and indexing paths.

## Reproduction Assessment

Yes, from source inspection rather than live ARM64 proof. Configure explicit local memory embeddings and trigger QMD boot sync, `openclaw memory status --deep`, `openclaw memory index --force`, or a first `memory_search`; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process runtime boundary is the narrowest maintainable fix because startup laziness and Promise.race timeouts cannot preempt native work that blocks the main Node.js event loop.

Keep the existing local provider/user config contract, but move node-llama-cpp model resolution, model/context creation, and embedQuery/embedBatch execution out of the Gateway main Node.js event loop, preferably through a worker-thread or child-process runtime owned by the memory host SDK.
