## Summary

Keep open. Current main still initializes and uses the explicit local GGUF embedding provider inside the Gateway Node.js process; recent startup/status mitigations reduce eager probing, but they do not isolate node-llama-cpp model/context creation from the event loop.

## What This Changes

The issue asks OpenClaw to keep Gateway startup and first-use memory paths responsive when `memorySearch.provider` is `local` and GGUF model initialization is slow on ARM64/Pi.

## Best Possible Solution

Keep the local embedding provider contract, but move GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime with regression coverage for startup, first-use search, indexing, and deep status paths.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, memory index/sync, or the first `memory_search`; current main still reaches in-process node-llama-cpp provider initialization.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because laziness and Promise timeouts cannot preempt native work once it blocks the main Node.js event loop.
