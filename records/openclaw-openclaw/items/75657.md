## Summary

Keep open. Current main still creates and uses the explicit local GGUF embedding provider inside the Gateway Node.js process; startup deferral and JS-level timeouts do not isolate native model/context creation from the event loop when local embeddings are first used.

## What This Changes

The issue requests preventing local GGUF memory embedding warmup from blocking Gateway startup and responsiveness on ARM64/Pi-class hardware.

## Best Possible Solution

Preserve the local embedding provider contract, but run GGUF model resolution, model/context creation, and embedding calls behind an internal worker-thread or child-process boundary with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger deep memory status, boot/index sync, or first memory_search; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. Runtime isolation is the narrowest maintainable fix because lazy startup, deferral, and Promise.race timeouts cannot keep the main event loop responsive during native GGUF initialization.
