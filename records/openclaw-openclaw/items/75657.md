## Summary

Keep open. Current main still runs local GGUF embedding provider setup and first model/context creation in the Gateway Node process; the recent QMD startup laziness narrows one startup trigger but does not remove the event-loop blocking path for local embeddings.

## What This Changes

Make local GGUF memory embedding initialization stop blocking the Gateway event loop on slow ARM64/Pi hosts.

## Best Possible Solution

Put local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider contract and remote-provider behavior.

## Reproduction Assessment

Yes. Configure `memorySearch.provider: "local"` with the reported GGUF model, then trigger `openclaw memory status --deep`, a first vector memory search, or an opted-in QMD startup refresh; current main still reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable fix because timeouts and lazy scheduling cannot keep the Gateway event loop responsive if native GGUF initialization monopolizes the main thread.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75657. Local GGUF memory embeddings (`memorySearch.provider: "local"`) initialize `node-llama-cpp` in the Gateway Node process, so slow ARM64/Pi model load can block the event loop during startup/deep probes/first memory search. Implement a worker-thread or child-process backed local embedding runtime so `resolveModelFile`, `getLlama`, `loadModel`, `createEmbeddingContext`, and embedding calls run off the main event loop while preserving the existing memory embedding provider API, default model, config schema, optional dependency error messages, cache-key behavior, and remote providers. Collapse concurrent initialization, handle worker startup/load/embed failures cleanly, close worker resources when providers/managers close where applicable, and do not change provider selection or swap the default GGUF model. Add focused tests around local-provider initialization delegation, concurrent first embed, startup/deep-probe nonblocking behavior, and error propagation. Add a `CHANGELOG.md` fix entry for the user-facing startup outage.
