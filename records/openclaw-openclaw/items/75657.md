## Summary

Keep open. Current main has improved startup defaults around QMD refresh and primary model prewarm, but the local GGUF embedding provider still resolves and loads node-llama-cpp models in the Gateway process, so the event-loop blocking risk remains whenever startup/diagnostic/first-use memory work enters that provider path.

## What This Changes

Fix local GGUF memory embedding initialization so slow node-llama-cpp model loading does not block Gateway startup or the main Node.js event loop on ARM64/Pi-class hardware.

## Best Possible Solution

Move local GGUF model resolution, model loading, embedding context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider API and remote provider behavior.

## Reproduction Assessment

Yes. Configure local memory embeddings with the reported GGUF model on ARM64/Pi-class hardware, then trigger an eager startup memory sync, deep memory probe, or first local embedding request; current main still reaches in-process node-llama-cpp loadModel/createEmbeddingContext.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable fix; promise timeouts, setImmediate scheduling, or default-off startup refresh only move when the stall happens and cannot keep the Gateway event loop responsive during native model load.
