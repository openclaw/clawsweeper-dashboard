## Summary

Keep open. Current main still creates the explicit local GGUF embedding provider inside the Gateway process; startup mitigations defer some paths, but first use or opt-in QMD startup can still reach in-process node-llama-cpp model/context creation and block the event loop.

## What This Changes

The issue requests moving or deferring local GGUF embedding initialization so Gateway startup remains reachable on slow ARM64/Pi hosts using local memory embeddings.

## Best Possible Solution

Keep the local embedding provider contract, but move GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime with regression coverage for startup and first-use paths.

## Reproduction Assessment

Yes, source-reproducible. Configure explicit local memory embeddings and trigger deep memory status, indexing, or the first memory_search; current main still awaits node-llama-cpp model/context creation in the Gateway process.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrow maintainable fix because JS Promise.race timeouts and startup deferral cannot preempt native work that blocks the main event loop.
