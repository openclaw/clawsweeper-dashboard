## Summary

Keep open. Current main still loads the local GGUF embedding model and creates the embedding context inside the Gateway Node process path, so the reported ARM64/Pi startup stall remains a valid runtime bug rather than stale cleanup.

## What This Changes

The issue requests moving or deferring local GGUF embedding initialization so Gateway startup stays responsive on slow ARM64 hosts.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or process-backed runtime while preserving the existing memory embedding provider API, cache-key behavior, and remote providers.

## Reproduction Assessment

Yes. Configure memory search with `provider: "local"` and the reported GGUF model, then trigger QMD boot sync, a deep embedding probe, or first vector search; current main reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. An off-main-thread worker or process boundary is the narrow maintainable fix; lazy init alone may reduce startup work but would still let the first embedding request block the Gateway event loop.
