## Summary

Keep open. Current main reduced the original default/startup exposure, but explicit local GGUF embeddings still initialize `node-llama-cpp` inside the Gateway process on first embedding use or opt-in memory startup work, with no worker or child-process boundary to protect the event loop.

## What This Changes

The issue requests that local GGUF memory embedding model initialization stop blocking Gateway startup and responsiveness on slow ARM64/Pi hardware.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider contract and remote-provider behavior.

## Reproduction Assessment

Yes, source-reproducible. Configure `agents.defaults.memorySearch.provider` to `local` and trigger `openclaw memory status --deep`, a first `memory_search`, or memory sync; current main reaches in-process `node-llama-cpp` model resolution/loading and embedding context creation.

## Solution Assessment

Yes, the reported worker/child-process direction is the narrowest maintainable fix. Laziness and longer timeouts reduce when the stall happens, but they cannot preempt native work that blocks the Gateway event loop once local GGUF initialization starts.
