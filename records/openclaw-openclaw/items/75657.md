## Summary

Keep open. Current main still performs local `node-llama-cpp` model resolution, model loading, context creation, and embedding calls in the Gateway process for explicit local memory embeddings, so the event-loop blocking risk remains despite newer startup mitigations.

## What This Changes

The issue asks OpenClaw to keep Gateway startup and first-use memory search responsive by moving or deferring local GGUF embedding initialization off the main Node.js event loop.

## Best Possible Solution

Introduce a worker-thread or process-backed local embedding runtime for GGUF model resolution, model/context creation, query embeddings, and batch embeddings while preserving the existing memory provider API, cache identity, remote providers, and current startup mitigations.

## Reproduction Assessment

Yes. Configure `agents.defaults.memorySearch.provider` or an agent override to `"local"` with the reported GGUF model, then trigger a first vector search, `openclaw memory status --deep`, or an opt-in QMD startup refresh; current main still reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. The suggested worker/process boundary is the narrow maintainable fix; lazy init or timeout changes alone can move when the stall occurs, but cannot keep the Gateway event loop responsive if native model loading monopolizes the main thread.
