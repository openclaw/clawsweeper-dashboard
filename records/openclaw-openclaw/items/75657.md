## Summary

Keep open. Current main still runs explicit local GGUF embedding initialization inside the Gateway Node.js process; startup deferral and long Promise.race timeouts do not isolate node-llama-cpp native work from the event loop once local embeddings are initialized or probed.

## What This Changes

Report requests that local GGUF memory embedding warmup stop blocking Gateway startup and reachability on slow ARM64/Pi hardware.

## Best Possible Solution

Keep the local embedding provider contract, but run GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary with regression coverage for startup, deep status, sync, and first search.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, QMD boot sync with startup enabled, memory index/sync, or the first `memory_search`; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because startup deferral, prewarm reshuffling, and Promise.race timeouts cannot interrupt native work that blocks the main Node.js event loop.
