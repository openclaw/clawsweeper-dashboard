## Summary

Keep open. Current main still constructs the explicit local GGUF embedding provider inside the Gateway process; recent startup laziness reduces some cold-start paths, but it does not isolate node-llama-cpp model/context creation from the main event loop.

## What This Changes

Prevent explicit local GGUF memory embeddings from blocking Gateway startup or runtime responsiveness on ARM64/Pi-class hosts.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the current memory embedding provider contract.

## Reproduction Assessment

Yes, source-reproducible. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, memory index/sync, or the first vector `memory_search`; current main still reaches in-process node-llama-cpp loadModel/createEmbeddingContext.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because startup laziness and promise timeouts cannot preempt native main-thread blocking during local GGUF initialization.
