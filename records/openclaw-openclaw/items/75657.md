## Summary

Keep open. Current main avoids builtin/local startup work by default, but the local GGUF provider still resolves and loads `node-llama-cpp` in the Gateway process whenever local embeddings are initialized, so the core event-loop blocking risk remains. The related open PRs cover prewarm or diagnostic-budget changes, not the needed worker or child-process isolation.

## What This Changes

The issue asks OpenClaw to stop local GGUF memory embedding initialization from blocking Gateway startup and the Node.js event loop on slow ARM64/Pi systems.

## Best Possible Solution

Run GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary while preserving the existing local provider contract.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced: configure explicit local memory embeddings and trigger deep status, memory index/sync, or the first `memory_search`; current main still enters the in-process `node-llama-cpp` provider path. I did not live-measure the ARM64/Pi startup timing.

## Solution Assessment

Yes, worker-thread or child-process isolation is the best direction. Startup deferral, prewarm reshuffling, or outer `Promise.race` timeouts do not keep native model loading off the Gateway event loop.
