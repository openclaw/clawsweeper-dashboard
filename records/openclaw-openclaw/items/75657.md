## Summary

Keep open. Current main has improved QMD startup defaults, but the explicit local GGUF embedding provider still imports and initializes node-llama-cpp in the Gateway process, so a slow native model load can still monopolize the event loop when that path is reached.

## What This Changes

Prevent local GGUF memory embedding initialization from blocking the Gateway Node.js event loop on slow ARM64/Pi startup or first-use paths.

## Best Possible Solution

Move local node-llama-cpp model resolution, model loading, embedding context creation, and embedding calls behind a worker-thread or process-backed runtime while preserving the existing memory embedding provider contract and remote provider behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.memorySearch.provider` as `local` with the reported GGUF model and trigger startup memory sync, `openclaw memory status --deep`, `openclaw memory index --force`, or a first memory search; current main still reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable fix; deferring startup work or adding Promise.race timeouts can reduce when the stall happens, but cannot keep the Gateway event loop responsive during native model initialization.
