## Summary

Keep open. Current main still runs the built-in local GGUF embedding provider in the Gateway process, so first-use model resolution, loading, context creation, and embedding can still monopolize the Node.js event loop despite startup promise timeouts.

## What This Changes

Keep Gateway startup responsive when local GGUF memory embeddings initialize on ARM64/Pi or other slow hosts.

## Best Possible Solution

Preserve the memory embedding provider API while isolating the built-in local GGUF runtime behind a worker or child process, leaving remote providers and provider selection semantics unchanged.

## Reproduction Assessment

Yes. Configure memory search to use the built-in local provider with the reported GGUF model, then start a Gateway path that performs boot sync or run a deep memory probe; current main still reaches in-process loadModel/createEmbeddingContext before local embeddings are available.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable fix; a timeout or scheduling-only change cannot keep the Gateway event loop responsive if native GGUF initialization monopolizes the main thread.
