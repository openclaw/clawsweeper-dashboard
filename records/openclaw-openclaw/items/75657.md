## Summary

Keep open. Current main still runs the explicit local GGUF embedding provider in the Gateway Node.js process; startup deferral and timeouts reduce eager work, but they do not isolate node-llama-cpp model/context creation from the event loop when local embeddings are used.

## What This Changes

The issue requests that local GGUF memory embeddings stop blocking Gateway startup and event-loop availability on ARM64/Pi when `memorySearch.provider` is `local`.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process provider boundary while preserving the existing `memorySearch.provider: "local"` contract and fallback semantics.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. On current main, explicit local memory embeddings plus `openclaw memory status --deep`, memory index/sync, QMD boot sync with startup enabled, or first semantic `memory_search` reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy deferral and Promise.race timeouts cannot protect Gateway responsiveness once native embedding work runs on the main Node.js event loop.

Fix issue https://github.com/openclaw/openclaw/issues/75657. Current main still creates and uses the explicit local GGUF memory embedding provider in the Gateway Node.js process, so node-llama-cpp model resolution/loading/context creation and embedding calls can block the event loop on slow ARM64 hosts. Implement a focused internal isolation boundary, preferably a worker thread or child process owned by the local embedding provider, while preserving existing `memorySearch.provider: "local"`, local model path/cache/contextSize behavior, fallback behavior, and public config/docs shape. Add regression coverage showing provider creation/first query does not synchronously perform the native initialization on the caller path, and cover deep status, sync/index, and first search paths where practical.
