## Summary

Keep open. Current main still routes explicit local GGUF memory embeddings through node-llama-cpp inside the Gateway Node.js process; recent startup/status deferrals reduce eager probing but do not isolate model resolution, model load, context creation, or embedding calls from the event loop.

## What This Changes

The issue asks OpenClaw to prevent explicit local GGUF memory embedding initialization from blocking Gateway startup and control-plane responsiveness on slow ARM64 hosts.

## Best Possible Solution

Keep the local embedding provider contract, but move node-llama model resolution, model loading, embedding-context creation, and embedding calls behind a worker-thread or child-process runtime with regression coverage for startup, status, sync, and first-use search paths.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger QMD boot sync, `openclaw memory status --deep`, memory index/sync, or the first `memory_search`; current main still reaches in-process node-llama model/context creation.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because startup deferral and Promise.race timeouts cannot preempt native work that blocks the main Node.js event loop.
