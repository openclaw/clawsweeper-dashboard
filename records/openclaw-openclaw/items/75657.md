## Summary

Keep open. Current main still runs the built-in local GGUF embedding provider in the Gateway process, and first local embedding use resolves the model, calls node-llama-cpp `loadModel`, and creates the embedding context without a Worker or process boundary. The report has detailed ARM64/Pi logs, and the current source does not contain a fix for the event-loop blocking class of failure.

## What This Changes

The issue requests making local GGUF memory embedding warmup non-blocking so Gateway startup remains reachable on slow ARM64/Pi hosts.

## Best Possible Solution

Move local GGUF model resolution, model loading, context creation, and embedding calls off the main Gateway event loop behind a worker-thread or process-backed local embedding runtime, preserving the existing memory embedding provider API and remote provider behavior.

## Reproduction Assessment

Yes. The reporter provided detailed v2026.4.29 ARM64/Pi logs, and current main still has a source-level path where explicit local memory embeddings load the GGUF model in-process; a full timing repro should run on similar ARM64 hardware with `memorySearch.provider: "local"` and the same startup/probe path enabled.

## Solution Assessment

Yes. A worker or process boundary for the local GGUF provider is the narrow maintainable fix; merely racing a timeout cannot keep the Gateway responsive if native model loading monopolizes the Node event loop.
