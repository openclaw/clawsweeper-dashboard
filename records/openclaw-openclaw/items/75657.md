## Summary

Keep open. Current main has partial startup mitigations, but explicit local GGUF embeddings still initialize `node-llama-cpp` in the Gateway process, so deep probes, first semantic search, sync, or opt-in memory startup can still block the main event loop.

## What This Changes

Request to keep the Gateway responsive by moving local GGUF embedding initialization and warmup off the main Node.js event loop.

## Best Possible Solution

Keep the local embedding provider contract, but isolate GGUF resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary with cleanup, timeout/error propagation, and regression coverage.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit `memorySearch.provider: "local"` and trigger `openclaw memory status --deep`, first `memory_search`, memory sync, or an opted-in memory startup path; current main still reaches in-process `node-llama-cpp` model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy scheduling and `Promise.race` timeouts cannot preempt native work that blocks the main Node.js event loop.
