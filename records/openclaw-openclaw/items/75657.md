## Summary

Keep open. Current main has reduced some eager startup work, but explicit local memory embeddings still call node-llama-cpp model/context creation inside the Gateway process, so the reported event-loop starvation remains source-reproducible.

## What This Changes

Ensure local GGUF memory embeddings do not block the Gateway event loop during startup or first use on slow ARM64/Pi systems.

## Best Possible Solution

Keep the local embedding provider contract, but run GGUF model resolution, model/context creation, and embedding calls behind an internal worker-thread or child-process boundary with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger deep memory status, memory index/sync, opt-in QMD startup refresh, or first memory_search; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy startup, scheduling, and Promise.race timeouts cannot preempt native work once it blocks the main Node.js event loop.

## Repair Work Prompt

Fix the local GGUF embedding event-loop starvation reported in this issue. Current main still creates and first-uses the local node-llama-cpp embedding provider inside the Gateway process; move importNodeLlamaCpp, resolveModelFile, getLlama/loadModel/createEmbeddingContext, and embedQuery/embedBatch execution behind a worker-thread or child-process boundary while preserving the existing memory embedding provider API, provider selection defaults, local config shape, cache key behavior, fallback behavior, and timeout semantics. Keep ownership in the memory-host-sdk/memory-core boundary; do not add a user-facing config flag and do not broaden QMD or active-memory product behavior. Add focused regression coverage for provider creation/first embedding and the Gateway startup/deep-status/sync paths so the main event loop is not tied to native model initialization. Add a single CHANGELOG.md fix entry because this is user-facing reliability work. Stop and return for maintainer review if packaging the worker requires dependency, release, or public SDK contract changes beyond additive internal files.
