## Summary

Keep open. Current main still creates the explicit local GGUF embedding provider inside the Gateway Node.js process; recent lazy/status mitigations avoid some eager probes, but they do not isolate node-llama-cpp model resolution, model loading, context creation, or embedding calls from the event loop.

## What This Changes

The issue requests that local GGUF memory embedding warmup stop blocking Gateway startup on slow ARM64/Pi hosts.

## Best Possible Solution

Keep the local embedding provider contract, but run GGUF model resolution, load/context creation, and embedding calls behind a worker-thread or child-process boundary while preserving lazy initialization and remote-provider behavior.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger deep memory status, memory index/sync, QMD boot sync with startup enabled, or first memory_search; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because laziness and Promise.race timeouts do not prevent native initialization from blocking the main Node.js event loop.
