## Summary

Keep open. Current main still routes explicit local memory embeddings through in-process `node-llama-cpp` model/context creation; recent startup deferral helps readiness paths but does not isolate native GGUF work from the Gateway event loop when local embeddings are used.

## What This Changes

The issue asks OpenClaw to prevent local GGUF memory embedding initialization from blocking Gateway startup and runtime responsiveness on ARM64/Pi-class hosts.

## Best Possible Solution

Keep the local embedding provider contract but run GGUF model resolution, model/context creation, and embedding calls behind an internal worker-thread or child-process boundary with regression coverage for startup, deep status, sync, and first search.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit `memorySearch.provider: "local"` and trigger QMD boot sync, `openclaw memory status --deep`, memory index/sync, or first semantic search; current main reaches in-process `node-llama-cpp` initialization.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy startup and `Promise.race` timeouts cannot interrupt native work that blocks the main Node.js event loop.

Fix issue https://github.com/openclaw/openclaw/issues/75657. Current main still initializes explicit local GGUF memory embeddings through in-process `node-llama-cpp` calls, so slow native model/context creation can block the Gateway event loop. Keep the existing `memorySearch.provider: "local"` contract and do not add a new config option. Move local provider model resolution, `loadModel`, `createEmbeddingContext`, and embedding calls behind a worker-thread or child-process boundary, preserving provider output normalization, cache key behavior, local model/cache/contextSize options, fallback behavior, and existing remote providers. Add focused regression coverage showing startup/deep status/sync/first-search paths do not perform blocking local initialization on the main Gateway path; use mocks rather than downloading a real GGUF. Include a user-facing changelog entry because this fixes Gateway responsiveness for local embeddings. Stop and report if `node-llama-cpp` cannot safely run in the chosen isolation boundary without a broader product decision.
