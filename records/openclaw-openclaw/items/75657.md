## Summary

Keep open. Current main has reduced the default cold-start exposure by keeping QMD startup opt-in and primary-model warmup metadata-only, but the explicit local GGUF embedding provider still initializes node-llama-cpp in the Gateway process on first use without a worker or child-process boundary.

## What This Changes

The issue asks OpenClaw to keep Gateway responsiveness when explicit local GGUF memory embeddings initialize on slow ARM64/Pi systems.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider contract and remote-provider behavior.

## Reproduction Assessment

Yes, source-reproducible. Configure explicit local memory embeddings and trigger a deep memory status probe, first vector memory search, or memory sync; current main still reaches in-process node-llama-cpp loadModel and createEmbeddingContext.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because startup laziness reduces cold-boot exposure but cannot preempt native main-thread blocking during the first local embedding call.
