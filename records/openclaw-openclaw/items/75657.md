## Summary

Keep open. Current main still runs the explicit local GGUF embedding provider inside the Gateway Node.js process; recent startup and status-path mitigations reduce eager probing, but they do not isolate node-llama-cpp model/context creation from the event loop.

## What This Changes

Prevent explicit local GGUF memory embeddings from blocking Gateway startup or first memory use on slow ARM64/local hosts.

## Best Possible Solution

Keep the local embedding provider contract, but move GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime with regression coverage for first-use and startup paths.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, memory index/sync, QMD boot sync with startup enabled, or the first `memory_search`; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because startup laziness, status deferral, and Promise.race timeouts cannot preempt native work that blocks the main Node.js event loop.
