## Summary

Keep open. Current main still runs explicit local GGUF memory embedding provider setup inside the Gateway process; startup paths are lazier now, but first live local embedding, deep memory status, or memory sync still reaches in-process `node-llama-cpp` model/context creation without a worker or child-process boundary.

## What This Changes

The issue requests moving local GGUF memory embedding initialization off the Gateway event loop so ARM64/Pi startup and first-use memory paths stay responsive.

## Best Possible Solution

Move local GGUF model resolution, `loadModel`, `createEmbeddingContext`, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider contract and avoiding new user-facing config.

## Reproduction Assessment

Yes, source-reproducible. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, memory sync/indexing, or the first `memory_search`; current main reaches in-process `node-llama-cpp` model/context creation from the Gateway process.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because startup laziness and Promise timeout wrappers cannot preempt a native main-event-loop stall.
