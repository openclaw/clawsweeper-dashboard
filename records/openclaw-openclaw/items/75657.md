## Summary

Keep open. Current main has narrowed some default startup triggers, but the built-in local GGUF embedding provider still loads `node-llama-cpp` model/context work inside the Gateway Node process, and explicit QMD startup, deep probes, or first-use memory sync/search can still reach that path without a worker or process boundary.

## What This Changes

Make local GGUF memory embedding initialization non-blocking so Gateway startup stays responsive on ARM64/Pi and other slow hosts.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or process-backed runtime while keeping the existing memory embedding provider API and remote providers unchanged.

## Reproduction Assessment

Yes. Configure `agents.defaults.memorySearch.provider` to `"local"` with a GGUF model, then trigger explicit QMD startup boot sync, `memory status --deep`, or first vector sync/search; current main still reaches the in-process `loadModel` and `createEmbeddingContext` path. I did not run a live Pi timing test in this read-only review.

## Solution Assessment

Yes. A worker or process boundary around the local GGUF runtime is the narrow maintainable fix; timeout or default-startup changes only defer the same main-thread native load.
