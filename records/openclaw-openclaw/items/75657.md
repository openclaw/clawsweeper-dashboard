## Summary

Keep open. Current main still creates and initializes the local node-llama-cpp embedding provider in the Gateway process, and both startup memory sync and deep probe paths can reach that first-use GGUF load despite the existing promise timeout.

## What This Changes

Make local GGUF memory embedding initialization non-blocking so Gateway startup remains reachable on ARM64/Pi.

## Best Possible Solution

Move node-llama-cpp model resolution, GGUF loading, embedding context creation, and embedding calls behind a worker-thread or process-backed local runtime while preserving the existing memory embedding provider API.

## Reproduction Assessment

Yes. Configure `agents.defaults.memorySearch.provider` as `local` with the reported GGUF model, then start the Gateway with QMD boot sync or run a deep memory probe; current main still reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable fix because Promise.race timeouts and scheduling tweaks cannot keep the Gateway event loop responsive when native model initialization monopolizes the main thread.
