## Summary

Keep open. Current main reduces the startup-specific exposure by keeping QMD startup opt-in and primary-model prewarm metadata-only, but the explicit local GGUF memory embedding provider still initializes node-llama-cpp in the Gateway process on first embedding use, so a slow native load can still block the main event loop.

## What This Changes

Prevent explicit local GGUF memory embeddings from blocking Gateway startup or runtime responsiveness on ARM64/Pi-class hosts.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider contract.

## Reproduction Assessment

Yes, source-reproducible. Configure explicit local memory embeddings and trigger memory status --deep, memory index/sync, or a first vector memory_search; current main still reaches in-process node-llama-cpp loadModel and createEmbeddingContext.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because startup laziness and promise timeouts cannot preempt native main-thread blocking during local GGUF initialization.
