## Summary

Keep open. Current main moved some memory and model startup work off the readiness path, but explicit local GGUF memory embeddings still initialize `node-llama-cpp` and load/create the embedding context inside the Gateway Node.js process on sync, deep status, or first semantic search.

## What This Changes

The issue requests that local GGUF memory embedding warmup stop blocking the Gateway event loop during startup or first use on slow ARM64 hosts.

## Best Possible Solution

Run local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary while preserving the local provider contract and adding startup and first-use regression coverage.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, memory sync/index, QMD boot sync with startup enabled, or first `memory_search`; current main still reaches in-process `node-llama-cpp` model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy scheduling and JS-level `Promise.race` timeouts cannot interrupt native work that blocks the main Node.js event loop.

Add regression coverage showing startup/deep status/sync/first search do not synchronously execute the heavy local provider path in the Gateway process, and preserve plugin-local dependency ownership for `node-llama-cpp`.
