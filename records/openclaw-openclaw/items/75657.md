## Summary

Keep open. Current main still routes explicit local memory embeddings through in-process `node-llama-cpp`; startup deferral and JS-level timeouts do not isolate native model/context creation from the Gateway event loop once local embeddings are used.

## What This Changes

Request to keep the Gateway responsive when explicit local GGUF memory embeddings initialize on slow ARM64 hosts.

## Best Possible Solution

Keep the local embedding provider contract, but move GGUF model resolution, model load, context creation, and embedding calls behind a worker-thread or child-process boundary with first-use queueing and focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: configure explicit local memory embeddings and trigger `openclaw memory status --deep`, memory index/sync, QMD boot sync, or first `memory_search`; those paths still reach in-process `node-llama-cpp` model/context creation. I did not live-run on ARM64/Pi in this read-only review.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy startup and `Promise.race` timeouts cannot preempt native work that blocks the main Node.js event loop.

## Repair Work Prompt

Fix the local GGUF memory embedding startup/event-loop blocking reported in https://github.com/openclaw/openclaw/issues/75657. Keep `memorySearch.provider: "local"` and offline local embeddings supported, but move node-llama-cpp model resolution, model load, embedding context creation, and embedding calls out of the Gateway main Node.js event loop using a worker-thread or child-process boundary. Preserve the existing memory embedding provider SDK contract and provider selection behavior; do not add a new config knob, do not switch the default provider semantics, and do not remove the local provider. Add regression coverage that proves local provider initialization is isolated/lazy enough for Gateway startup, and that deep status, memory index/sync/QMD boot sync, and first `memory_search` still work through the isolated provider. Add a single-line user-facing changelog fix entry because this changes Gateway reliability for local embeddings.
