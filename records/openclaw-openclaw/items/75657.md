## Summary

Keep open. Current main still initializes the local GGUF embedding provider in the Gateway process, and the startup timeout/scheduling code does not isolate node-llama-cpp model loading from the Node.js event loop. The reporter supplied concrete ARM64/Pi logs that match the current source-level behavior.

## What This Changes

The issue requests that local GGUF memory embedding initialization stop blocking the Gateway event loop during startup on low-powered ARM64 hosts.

## Best Possible Solution

Move local GGUF model resolution, model loading, embedding context creation, and embedding calls behind a worker-thread or process-backed runtime for the local memory embedding provider while preserving the existing provider API and remote provider behavior.

## Reproduction Assessment

Yes. Configure memory search with `provider: "local"` and the reported GGUF model on ARM64/Pi, then trigger Gateway startup memory sync or a deep memory probe; current main still reaches in-process `loadModel` and `createEmbeddingContext` on first local embedding use.

## Solution Assessment

Yes. A worker or process boundary for the local GGUF provider is the narrow maintainable fix; a promise timeout or `setImmediate` scheduling alone cannot keep the Gateway responsive if native model loading monopolizes the main event loop.
