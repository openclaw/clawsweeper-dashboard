## Summary

Keep open. Current main has reduced some default cold-start exposure, but explicit local GGUF memory embeddings still initialize node-llama-cpp inside the Gateway process on first deep probe, sync, or embedding request, so the reported main-thread block remains source-reproducible.

## What This Changes

The issue requests keeping the Gateway responsive when explicit local GGUF memory embeddings initialize node-llama-cpp on slow ARM64 hosts.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider contract and remote-provider behavior.

## Reproduction Assessment

Yes, source-reproducible. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, a memory sync, or the first vector memory search; current main still reaches in-process node-llama-cpp initialization.

## Solution Assessment

Yes. A worker or child-process boundary is the narrowest maintainable fix because laziness only moves the native main-thread block to first use and cannot make the Gateway event loop responsive during that load.
