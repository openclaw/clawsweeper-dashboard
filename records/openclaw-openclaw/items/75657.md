## Summary

Keep open. Current main still initializes the local `node-llama-cpp` GGUF embedding provider inside the Gateway Node process, and both QMD boot sync and deep embedding probes can still reach first-use model loading without a worker or process boundary.

## What This Changes

The issue reports that local GGUF memory embedding warmup can block the Gateway event loop for minutes on ARM64/Pi startup.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or process-backed runtime while preserving the existing memory provider API and remote provider behavior.

## Reproduction Assessment

Yes. Configure memory search to use the local GGUF provider with the reported model, then trigger QMD boot sync, a deep embedding probe, or the first vector search; current main still reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable solution; timeout or scheduling changes alone cannot keep the Gateway event loop responsive if native model loading monopolizes the main thread.
