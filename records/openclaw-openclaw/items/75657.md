## Summary

Keep open. Current main has reduced some eager startup work, but explicit local GGUF embeddings still initialize node-llama-cpp model/context state inside the Gateway Node.js process on first embedding use, so the reported event-loop stall remains a valid source-level bug.

## What This Changes

The issue reports that explicit local GGUF memory embeddings can block Gateway readiness and WebSocket access for minutes on slow ARM64/Pi hardware.

## Best Possible Solution

Keep the local embedding provider contract, but isolate GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary with regression coverage for startup refresh, deep status/index, and first semantic search.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit `memorySearch.provider: "local"` and trigger `openclaw memory status --deep`, `openclaw memory index --force`, opt-in QMD startup sync, or first `memory_search`; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy startup scheduling and Promise.race timeouts cannot keep the Gateway event loop responsive while native model initialization blocks it.
