## Summary

Keep open. Current main still initializes the local GGUF embedding provider in the Gateway process and still reaches `node-llama-cpp` model resolution, `loadModel`, and `createEmbeddingContext` without a worker-thread or child-process boundary; the prior ClawSweeper keep-open comment still matches the current code.

## What This Changes

The issue asks OpenClaw to keep the Gateway responsive by moving local GGUF embedding model warmup and first-use initialization off the main Node.js event loop on slow ARM64/Pi systems.

## Best Possible Solution

Put local GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider API and remote-provider behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.memorySearch.provider` to `"local"` with the reported GGUF model, then trigger QMD boot sync, `openclaw memory status --deep`, or the first vector memory search; current main reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. A worker or child-process boundary is the narrow maintainable fix because startup timeouts and lazy scheduling do not keep the Gateway event loop responsive if native GGUF initialization monopolizes the main thread.
