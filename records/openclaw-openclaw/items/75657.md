## Summary

Keep open. Current main still routes the built-in local GGUF memory embedding provider through node-llama-cpp inside the Gateway process, and startup memory sync can enter that path without a worker or process boundary. The existing timeout only races the awaited warmup path; it cannot prevent native model loading from monopolizing the Node event loop once that work starts.

## What This Changes

The issue requests that local GGUF memory embedding initialization stop blocking Gateway startup on ARM64/Pi, likely by moving model load and embedding work to a worker/process boundary or an equivalent non-blocking lazy queue.

## Best Possible Solution

Move node-llama-cpp model resolution, model loading, embedding context creation, and embedding calls for the built-in local memory embedding provider behind a worker-thread or child-process runtime while keeping the existing memory embedding provider contract and remote provider behavior unchanged.

## Reproduction Assessment

Yes. The issue gives detailed v2026.4.29 ARM64/Pi startup logs with provider "local", and current main still has a source-level path where startup memory sync can initialize the local GGUF provider in the Gateway process.

## Solution Assessment

Yes. A worker or process boundary for the local GGUF provider is the narrow maintainable fix; a timeout or plain lazy init alone cannot keep the Gateway responsive if the native model load continues on the main event loop.
