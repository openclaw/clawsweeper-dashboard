## Summary

Keep open. Current main still creates the explicit local GGUF embedding provider in the Gateway Node.js process; startup deferral and JS-level timeouts reduce some eager work, but they do not isolate node-llama-cpp model/context creation from the event loop once local embeddings are first used.

## What This Changes

The issue asks for local GGUF embedding initialization to stop blocking Gateway startup and liveness on slow ARM64/Pi systems.

## Best Possible Solution

Keep the local embedding provider contract, but run GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary with regression coverage for startup, deep status, sync, and first search.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, memory index/sync, QMD boot sync with startup enabled, or first `memory_search`; current main still reaches in-process node-llama-cpp model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because lazy startup, deferral, and Promise.race timeouts cannot interrupt native work that blocks the main Node.js event loop.
