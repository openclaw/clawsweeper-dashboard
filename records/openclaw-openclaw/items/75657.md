## Summary

Keep open. Current main still runs the built-in local GGUF embedding runtime inside the Gateway process, and both QMD startup sync and deep memory probes can reach first-use `node-llama-cpp` model loading despite the existing promise timeout.

## What This Changes

The issue reports that explicit local GGUF memory embeddings can block Gateway startup for minutes on ARM64/Pi because native model initialization runs on the main Node.js event loop.

## Best Possible Solution

Isolate local GGUF model resolution, model loading, embedding context creation, and embedding calls behind a worker-thread or child-process runtime while preserving the existing memory embedding provider API and remote-provider behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.memorySearch.provider` as `local` with the reported GGUF model on ARM64/Pi, then start the Gateway with QMD boot sync enabled or run a deep memory probe; current main still reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable fix because promise timeouts and lazy scheduling cannot keep the Gateway event loop responsive if native model initialization monopolizes the main thread.
