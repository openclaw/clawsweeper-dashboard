## Summary

Keep open. Current main reduces some eager startup work, but explicit local memory embeddings still create the node-llama-cpp model and embedding context inside the Gateway Node.js process, so first use or opt-in startup sync can still starve the event loop.

## What This Changes

The issue requests making local GGUF memory embedding warmup non-blocking so Gateway startup and control paths stay reachable on slow ARM64/Pi hardware.

## Best Possible Solution

Keep the local embedding provider contract, but run GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary with regression coverage for startup scheduling, deep status, sync, and first search.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure memorySearch.provider as "local" with a GGUF model and trigger openclaw memory status --deep, memory index/sync, opt-in QMD boot sync, or first memory_search; current main reaches in-process node-llama-cpp initialization.

## Solution Assessment

Yes. Runtime isolation is the narrowest maintainable fix because startup deferral and Promise.race timeouts cannot prevent synchronous or native model work from starving the Node event loop.
