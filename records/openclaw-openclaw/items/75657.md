## Summary

Keep open. Current main still initializes the in-process `node-llama-cpp` local embedding provider inside the Gateway process; the startup deferral that exists today reduces some eager paths, but it does not isolate GGUF model resolution, model load, context creation, or embedding calls from the Node.js event loop. The related prewarm PR at https://github.com/openclaw/openclaw/pull/70596 overlaps the startup files but moves local probing earlier rather than adding the needed worker or child-process boundary.

## What This Changes

The issue requests preventing local GGUF memory embedding warmup/model initialization from blocking Gateway startup and the Node.js event loop on slow ARM64 hosts.

## Best Possible Solution

Keep the local embedding provider contract, but run GGUF model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary with regression coverage for startup, deep status, sync, and first search.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, QMD boot sync with startup enabled, memory index/sync, or the first `memory_search`; current main still reaches in-process `node-llama-cpp` model/context creation.

## Solution Assessment

Yes. A worker-thread or child-process boundary is the narrowest maintainable fix because startup deferral, prewarm reshuffling, and `Promise.race` timeouts cannot interrupt native work that blocks the main Node.js event loop.
