## Summary

Keep open. Current main has reduced default cold-start exposure by making QMD startup refresh opt-in and metadata-only, but an explicit local GGUF memory embedding provider still initializes node-llama-cpp in the Gateway process on first embedding use with no Worker or child-process boundary.

## What This Changes

Fix local GGUF memory embedding initialization so slow node-llama-cpp model warmup cannot block the Gateway Node.js event loop for minutes on ARM64/Pi startup or first use.

## Best Possible Solution

Move local GGUF model resolution, model/context creation, and embedding calls behind a Worker-thread or child-process runtime while preserving the existing memory embedding provider contract, cache keys, timeout behavior, and remote-provider behavior.

## Reproduction Assessment

Yes, source-reproducible. Configure explicit local memory embeddings and trigger `openclaw memory status --deep`, first vector memory search, or memory sync; current main still reaches in-process node-llama-cpp loadModel/createEmbeddingContext on the Gateway thread.

## Solution Assessment

Yes. A Worker-thread or child-process boundary is the narrowest maintainable fix because lazy startup and promise timeouts reduce when the cost is paid but cannot preempt native main-thread blocking during local GGUF initialization.
