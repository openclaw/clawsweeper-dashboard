## Summary

Keep open. Current main still creates the local node-llama-cpp embedding provider in the Gateway process, and startup memory sync/probe paths can still reach in-process GGUF model loading despite the existing promise timeout.

## What This Changes

The issue asks for local GGUF memory embedding warmup to stop blocking Gateway startup and the Node.js event loop on ARM64/Pi systems.

## Best Possible Solution

Move node-llama-cpp model resolution, GGUF model loading, embedding context creation, and embedding calls behind a worker-thread or process-backed local runtime while preserving the existing provider API and remote provider behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.memorySearch.provider` as `local` with the reported GGUF model on ARM64/Pi, then start the Gateway with QMD boot sync or run a deep memory probe; current main still reaches in-process `loadModel` and `createEmbeddingContext`.

## Solution Assessment

Yes. A worker or process boundary is the narrow maintainable fix; a timeout or scheduling change alone cannot keep the Gateway event loop responsive if native model loading monopolizes the main thread.
