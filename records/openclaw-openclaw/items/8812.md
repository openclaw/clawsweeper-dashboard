## Summary

Keep open: current main still renders URL-bearing tool output as escaped/plain text in Control UI tool-card paths and plain SwiftUI text in native WebChat, while the related implementation PR was closed unmerged.

## What This Changes

The issue requests clickable HTTP(S) URLs inside Control UI and native WebChat tool output cards.

## Best Possible Solution

Implement safe URL-aware rendering for all tool-output card surfaces, preserving JSON/plain-text formatting and adding equivalent native URL rendering when WebChat remains in scope.

## Reproduction Assessment

Yes by source inspection: Control UI binds tool-output strings as text in expanded, preview, and inline paths, and native WebChat renders formatter output with SwiftUI Text. I did not run a live browser or native app because this review was constrained to read-only inspection.

## Solution Assessment

Yes, keeping this open for a safe renderer is the right direction; the maintainable fix should escape untrusted output, linkify only safe URL schemes, and cover every tool-card path rather than applying a narrow regex replacement in one view.

In Control UI, add a safe helper that escapes arbitrary text first, linkifies only safe http/https URLs, handles trailing punctuation/whitespace, and preserves JSON/plain-text layout. Apply it across expanded output, raw details, collapsed previews, short inline cards, grouped tool messages, and side-panel/raw views as appropriate. Add focused tests for escaping, safe scheme filtering, URL edge cases, and each visible tool-card path. If native WebChat remains in scope, add equivalent SwiftUI URL rendering using AttributedString/Link-style handling without losing preview truncation. Add a user-facing CHANGELOG.md entry. Do not introduce core plugin IDs, broad markdown renderer changes, or unsafeHTML on unescaped tool output.
