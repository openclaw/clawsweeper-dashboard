## Summary

Current main still renders URL-bearing tool results as plain text in both Control UI and native WebChat, and both linked implementation attempts closed unmerged, so the feature remains necessary; the main unresolved decision is whether one change must cover both clients.

## What This Changes

Render safe clickable HTTP(S) links inside tool output while preserving plain-text/JSON formatting and copy behavior.

## Best Possible Solution

Keep this as the canonical request, choose the intended client scope, and add a shared escape-preserving URL-aware text renderer in each selected UI stack with coverage for every selected tool-result presentation.

## Reproduction Assessment

Yes at source level. Current main sends tool results through direct text renderers in Control UI and native WebChat, while automatic linkification is confined to the separate markdown path.

## Solution Assessment

The shared safe-renderer direction is the best maintainable approach; the proposed one-off regex replacement is not, because it risks unsafe HTML, punctuation/whitespace errors, and incomplete coverage across sibling render paths.
