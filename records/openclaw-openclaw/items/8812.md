## Label Justifications

- `P3`: This is a useful but non-urgent ergonomics improvement with limited blast radius.
- `impact:other`: Users must copy and paste URLs from tool results, creating concrete UI friction outside the specific impact categories.

## Summary

Current main still renders tool-result URLs as plain text in Control UI and native WebChat, and both linked implementation PRs closed unmerged. The feature remains necessary, but maintainers need to decide the cross-client scope and approve a canonical safe-rendering contract.

## What This Changes

Add clickable HTTP(S) links to URL-bearing tool output in Control UI and, if retained in scope, native WebChat.

## Best Possible Solution

Define one safe tool-output link presentation contract that preserves whitespace and copyable text, recognizes only allowed HTTP(S) URLs, handles punctuation correctly, and apply it consistently across all approved Control UI and native surfaces with focused behavior tests.

## Reproduction Assessment

Yes at source level: current-main Control UI and native tool-result paths demonstrably render URL-bearing strings as text, although this read-only review did not launch either client.

## Solution Assessment

No, the proposed generic regex-to-HTML helper is not the best final design; a reusable escape-safe, scheme-restricted presentation primitive covering all selected render paths is narrower and more maintainable.
