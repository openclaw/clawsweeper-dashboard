## Summary

Keep open: current main and v2026.6.5 support Slack plugin-owned modal submission/close routing, but still do not expose the requested first-class Slack modal open/update/close/validation workflow surface. The related full-state callback issue is fixed by https://github.com/openclaw/openclaw/pull/82133, while the closest lifecycle-action PR, https://github.com/openclaw/openclaw/pull/22084, closed unmerged and does not supersede this request.

## What This Changes

The issue requests first-class Slack modal lifecycle support for workflows/plugins, including opening, updating, closing, validation ACKs, dynamic options, and routed modal submissions.

## Best Possible Solution

Define a Slack plugin-owned modal lifecycle contract that reuses the existing plugin interactive routing, supports open/update/close and validation responses, handles dynamic modal options deliberately, and documents the ownership and data boundaries with focused tests.

## Reproduction Assessment

Not applicable: this is a missing feature/API surface request, not a broken existing behavior report. Source and docs inspection show receive-side modal callbacks exist, while modal open/update/close workflow actions do not.

## Solution Assessment

Unclear until maintainers choose the contract. The likely maintainable direction is a Slack plugin-owned or narrow Plugin SDK surface that builds on existing modal routing, not a raw ad hoc Slack API escape hatch.
