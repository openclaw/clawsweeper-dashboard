## Summary

Keep open: current main and the latest release support Slack modal submission and close callbacks for OpenClaw-routed/plugin-owned modals, but they still do not provide the requested first-class workflow surface for opening, updating, closing, and validating Slack modals.

## What This Changes

Add first-class Slack modal support so workflows can open Block Kit modals, update or close them, receive submissions, expose values, and return validation errors.

## Best Possible Solution

Define a Slack plugin-owned modal contract that can open, update, and close views, route submissions through the existing interactive handler/system-event path, handle validation responses, and document the workflow-facing API.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection shows the current Slack plugin handles modal submissions and closes but does not expose a modal open/update/close workflow action.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is likely a Slack plugin or Plugin SDK surface that reuses the existing modal lifecycle handling, not an ad hoc raw Slack API call path.
