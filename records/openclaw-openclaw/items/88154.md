## Summary

Keep open: current main and v2026.6.6 support plugin-owned Slack modal submission/close routing, but they still do not expose a documented workflow/plugin surface for opening, updating, closing, or validation-ACKing Slack modals.

## What This Changes

Request first-class Slack modal lifecycle support for workflows/plugins, including open/update/close, submission routing, validation errors, dynamic options, and documented ownership.

## Best Possible Solution

Define a Slack plugin-owned modal lifecycle contract that reuses current plugin modal submission routing, supports open/update/close plus validation ACK responses, documents ownership and callback routing, and coordinates with shortcut trigger work where needed.

## Reproduction Assessment

Not applicable: this is a missing workflow/plugin API surface, not a broken existing behavior report. Source inspection verifies the absence of modal lifecycle actions, but there is no failing current behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the API boundary. A Slack plugin-owned contract built on current callback routing is likely the maintainable direction, while a raw Slack API escape hatch would be a weaker long-term shape.
