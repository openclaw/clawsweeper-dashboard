Keep open for maintainer review: the patch addresses the linked P1 install-hook bug with sufficient live CLI proof, but it intentionally changes CLI plugin install startup and failure behavior for existing configured-plugin setups.

## What This Changes

The PR makes `openclaw plugins install` load the plugin registry during CLI startup, threads a recovery-specific plugin-load failure override, and extends CLI/plugin-registry tests.

## Best Possible Solution

Land the narrow CLI bootstrap fix with the recovery-path tests preserved once maintainers accept the intentional startup compatibility tradeoff.

## Reproduction Assessment

Yes, source-level: current main resolves `plugins install` to no plugin loading, bootstrap returns before registry activation, and `before_install` sees no process-local hook runner. I did not run a live failing install in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance: adding a CLI startup policy override plus a narrow recovery load-error escape is the right owner boundary for the documented hook contract. Gateway RPC delegation would be broader and is not needed for this bug.
