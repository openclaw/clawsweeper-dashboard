## Summary

Keep open: current main and the latest shipped tag still run the optional Tailscale hostname probe immediately after enabling Serve without retry/backoff, and the linked closing PR is open/draft rather than merged.

## What This Changes

The issue reports noisy transient Tailscale `status --json` failures during gateway Serve startup and requests retry/backoff or a narrow equivalent fix.

## Best Possible Solution

Land the linked PR or an equivalent narrow fix that retries the optional post-Serve status probe with final-attempt-only failure logging while leaving real Serve/Funnel setup failures visible.

## Reproduction Assessment

Yes, at source level: current main enables Serve and immediately queries `tailscale status --json`, and `runExec` emits verbose failure logs before the caller's catch can swallow the optional hostname failure. I did not run a live macOS Tailscale restart.

## Solution Assessment

Yes, the narrow retry/final-attempt logging direction is the maintainable fix for this bug. Adding `gateway.tailscale.binaryPath` would be broader config surface and is not needed for the startup-noise race.
