The issue requests retry/backoff for transient `tailscale status --json` failures during Gateway Tailscale Serve startup hostname discovery.

## Best Possible Solution

Resolve this through https://github.com/openclaw/openclaw/pull/91553 or a narrow replacement that retries the status JSON read, avoids new config surface, preserves fast fallback for missing binaries, and suppresses non-final retry noise.

## Reproduction Assessment

Yes at source level, not live: current main enables Serve and then immediately calls the single-read `getTailnetHostname` path, which can surface a verbose exec failure before being swallowed. A live macOS/Homebrew Tailscale restart under load was not reproduced in this read-only review.

## Solution Assessment

Yes for direction: retrying the existing status JSON read is the narrowest maintainable fix. Adding `gateway.tailscale.binaryPath` would expand config surface and does not directly solve the transient post-Serve status race.
