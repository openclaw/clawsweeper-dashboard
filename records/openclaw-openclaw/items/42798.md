## Summary

Current main still has the single-shot Tailscale status lookup after Serve/Funnel startup, so the reported transient verbose `Command failed: tailscale status --json` path remains plausible and narrow enough to fix.

## What This Changes

The issue requests eliminating a Tailscale gateway startup race where `tailscale status --json` can fail transiently after enabling `tailscale serve`.

## Best Possible Solution

Add a narrow retry/backoff around the best-effort hostname lookup after Serve/Funnel setup, without adding a new public Tailscale binary-path setting.

## Reproduction Assessment

Yes, at source level. Mocking a successful Serve setup followed by one failing `tailscale status --json` exercises the current single-attempt path and the verbose `runExec` failure log.

## Solution Assessment

Yes. A small retry/backoff for the best-effort status lookup is the narrow maintainable fix; adding `gateway.tailscale.binaryPath` would be a broader config change that does not directly solve the startup race.

## Repair Work Prompt

Fix the Tailscale gateway startup race reported in https://github.com/openclaw/openclaw/issues/42798. Keep the boundary narrow: add a small retry/backoff for the best-effort `tailscale status --json` hostname lookup used after `enableTailscaleServe`/`enableTailscaleFunnel`, preferably in `src/infra/tailscale.ts` or via a small gateway helper. Do not add `gateway.tailscale.binaryPath` or otherwise expand the public config contract. Add focused Vitest coverage showing a transient first `status --json` failure is retried and a later success returns the hostname, while persistent failures still degrade to the existing generic `serve enabled`/`funnel enabled` behavior. Add a `CHANGELOG.md` fix entry because this is user-facing logging/startup behavior.
