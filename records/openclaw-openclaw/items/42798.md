## Label Justifications

- `P2`: The bug creates confusing startup error logs for optional Tailscale exposure but the report says the gateway continues and Serve usually configures correctly.

## Summary

Review evidence still supports keeping this open: current main and the latest shipped tag still query `tailscale status --json` immediately after enabling Serve without retry/backoff, and the linked closing PR remains open/draft rather than merged.

## What This Changes

The issue asks for retry/backoff around the post-Serve `tailscale status --json` hostname probe to avoid transient startup error logs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/82727 or an equivalent narrow fix that retries the optional post-Serve status probe with final-attempt-only failure logging while leaving real Serve/Funnel setup failures visible.

## Reproduction Assessment

Yes, at source level: current main enables Serve and immediately queries `tailscale status --json`, and `runExec` emits verbose failure logs before the caller's catch can swallow the optional hostname failure. I did not run a live macOS Tailscale restart.

## Solution Assessment

Yes, the narrow retry/final-attempt logging direction is the maintainable fix for this bug. Adding `gateway.tailscale.binaryPath` would be broader config surface and is not needed for the startup-noise race.
