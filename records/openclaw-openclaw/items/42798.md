## Summary

Keep open: current main and the latest shipped tag still query `tailscale status --json` immediately after enabling Serve without retry/backoff, and the linked closing PR is still open and draft rather than merged.

## What This Changes

The issue reports confusing transient `tailscale status --json` startup errors when Tailscale Serve mode queries status immediately after enabling Serve.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/82727 or an equivalent narrow fix that retries the optional post-Serve status probe while keeping real Serve/Funnel setup failures visible.

## Reproduction Assessment

Yes, at source level: current main enables Serve and immediately queries `tailscale status --json`, and `runExec` emits verbose failure logs before the caller's catch can swallow the optional hostname failure. I did not run a live macOS Tailscale restart.

## Solution Assessment

Yes. Retrying the optional status probe is the narrow maintainable fix; adding `gateway.tailscale.binaryPath` would be broader config surface and does not directly address post-Serve daemon settling.
