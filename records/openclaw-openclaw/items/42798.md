## What This Changes

The issue asks OpenClaw to avoid transient verbose/debug `tailscale status --json` failure logs immediately after enabling Tailscale Serve during gateway startup.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/82727 with a narrow Tailscale-local retry or quiet best-effort hostname probe that preserves real Serve/Funnel setup failures and avoids adding a public binary-path config knob.

## Reproduction Assessment

Yes, at source level: current main immediately probes `tailscale status --json` after Serve setup, and `runExec` logs failures before the optional hostname catch can hide them. I did not run the live macOS launchd/Tailscale restart path.

## Solution Assessment

Yes for the narrow retry/backoff or quiet best-effort status probe; the alternative `gateway.tailscale.binaryPath` config option is broader and does not directly address the post-Serve status race.
