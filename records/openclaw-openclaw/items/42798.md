## Summary

Keep this issue open: current main and v2026.6.6 still perform a one-shot `tailscale status --json` hostname read immediately after Serve setup, and https://github.com/openclaw/openclaw/pull/91553 is an open closing PR for the narrow retry fix.

## What This Changes

The issue asks OpenClaw to avoid transient `tailscale status --json` startup errors in Tailscale Serve mode, preferably through retry/backoff rather than a new binary-path config option.

## Best Possible Solution

Land the narrow retry in https://github.com/openclaw/openclaw/pull/91553 after acceptable live or maintainer-approved proof, or close that PR and reopen the fix path if the mitigation is rejected.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection confirms the current Serve startup path immediately performs an unretried `tailscale status --json` lookup, matching the reported timing window.

## Solution Assessment

Yes, the narrow retry/backoff direction is the best current fix shape. Adding `gateway.tailscale.binaryPath` would expand config surface and would not directly address the post-Serve status timing race.
