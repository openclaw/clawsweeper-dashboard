## Summary

Keep this open: current `main` still immediately runs the best-effort Tailnet hostname lookup after `tailscale serve`, and a transient `tailscale status --json` failure can still be logged by the shared exec helper before the caller swallows the lookup failure.

## What This Changes

The issue reports noisy transient `tailscale status --json` failures during gateway startup in Tailscale Serve mode.

## Best Possible Solution

Make only the post-Serve/Funnel hostname status probe truly best-effort, by retrying or suppressing the shared verbose failure log for this lookup while preserving normal warnings for failed Serve/Funnel setup.

## Reproduction Assessment

Yes, at source level: with verbose/debug logging enabled, any non-zero `tailscale status --json` from `getTailnetHostname` causes `runExec` to log the command failure before `startGatewayTailscaleExposure` catches the best-effort lookup. I did not run the reporter's live macOS launchd restart path.

## Solution Assessment

Yes for a narrow retry or quiet best-effort status lookup. Adding `gateway.tailscale.binaryPath` would widen the public config contract and addresses a different discovery problem.
