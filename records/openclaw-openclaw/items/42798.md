## Summary

This should stay open: current main still has the reported Serve startup shape, and the shared exec helper still logs the exact verbose `Command failed: tailscale status --json` line before the caller suppresses the best-effort hostname failure.

## What This Changes

The issue reports transient startup failures from `tailscale status --json` immediately after `tailscale serve --bg --yes` when gateway Tailscale mode is `serve`.

## Best Possible Solution

Add a small bounded retry/backoff for the post-Serve Tailscale status lookup, keep the hostname lookup best-effort, and cover the first-failure-then-success path in focused gateway or infra Tailscale tests.

## Reproduction Assessment

Yes from source inspection for the observable log path: when `tailscale status --json` rejects under verbose logging, `runExec` emits the exact `Command failed` line before `startGatewayTailscaleExposure` suppresses the hostname lookup failure. I did not establish a live macOS/Tailscale timing reproduction in this read-only pass.

## Solution Assessment

Yes. A bounded retry/backoff around `getTailnetHostname` or the underlying status read is narrower and more maintainable than adding a new `gateway.tailscale.binaryPath` config option for this startup race.
