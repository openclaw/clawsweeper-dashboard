## Summary

Keep open: current main still has the source-level noisy Tailscale startup path, and the linked closing PR is open, so this issue should remain open until that PR lands or is rejected.

## What This Changes

The issue requests suppressing transient `tailscale status --json` failure noise during gateway startup in Serve mode, preferably by retrying or backing off the optional hostname probe.

## Best Possible Solution

Resolve the linked PR or an equivalent narrow fix so optional post-Serve hostname discovery retries or stays quiet while real Serve/Funnel setup failures remain visible.

## Reproduction Assessment

Yes, at source level: Serve startup immediately calls `getTailnetHostname`, and a failing `tailscale status --json` reaches `runExec`'s verbose failure log before the caller catch swallows the optional hostname failure. I did not run a live macOS Tailscale restart.

## Solution Assessment

Yes for the narrow bug shape: a quiet or delayed retry around the optional status probe is the maintainable fix. The suggested `gateway.tailscale.binaryPath` option would be broader config surface and is not needed to address this startup-noise race.
