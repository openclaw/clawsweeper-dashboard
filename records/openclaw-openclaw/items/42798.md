## Summary

Keep open: current main still has the source-level path where a transient `tailscale status --json` failure is logged by the shared exec helper before the best-effort hostname lookup swallows it, so the reported noisy startup race remains plausible and narrowly fixable.

## What This Changes

The issue asks OpenClaw to avoid noisy transient `tailscale status --json` failures immediately after enabling Tailscale Serve during gateway startup.

## Best Possible Solution

Make the post-Serve hostname probe truly best-effort by retrying/backing off or quieting only this status lookup, while preserving visible warnings for failed Serve/Funnel setup and cleanup.

## Reproduction Assessment

Yes, at source level: with verbose/debug logging enabled, any non-zero `tailscale status --json` from `getTailnetHostname` causes `runExec` to log before the caller catch treats hostname discovery as best-effort. I did not run the live macOS launchd/Tailscale restart path.

## Solution Assessment

Yes for a narrow retry/backoff or quiet best-effort status probe. Adding `gateway.tailscale.binaryPath` would be a broader public config change and does not directly fix the noisy post-Serve status race.
