## Summary

Keep open: current main still has the source-level path reported by the issue, where a best-effort post-Serve hostname lookup can log `Command failed: tailscale status --json` before the caller swallows the failure. The prior ClawSweeper comment matches the current code, and I did not find a merged or canonical item that resolves this specific startup-noise race.

## What This Changes

The issue asks for Tailscale Serve startup to avoid transient verbose `tailscale status --json` failure logs after `tailscale serve --bg --yes` succeeds.

## Best Possible Solution

Keep Serve/Funnel setup and cleanup failures visible, but make only the optional hostname probe after exposure setup retry briefly or run quietly when it is being used as best-effort URL decoration.

## Reproduction Assessment

Yes, at source level: with verbose/debug logging enabled, any non-zero `tailscale status --json` from `getTailnetHostname` reaches `runExec` and logs before the caller catch treats hostname discovery as optional. I did not run the live macOS launchd/Tailscale restart path.

## Solution Assessment

Yes for a narrow retry/backoff or quiet best-effort status probe. The suggested `gateway.tailscale.binaryPath` setting would be a broader public config change and would not directly fix the immediate post-Serve status race.

Fix the Tailscale Serve startup race reported in https://github.com/openclaw/openclaw/issues/42798.
