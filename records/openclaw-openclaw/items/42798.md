## Summary

Keep open. Current main still matches the reported Gateway Tailscale startup race: Serve/Funnel setup is followed by a single hostname/status lookup, the shared Tailscale status helpers do not retry transient `tailscale status --json` failures, and verbose exec logging can still emit the confusing failure before the startup path swallows it. I also found no direct open PR that references this issue as a fix candidate.

## What This Changes

Request retry/backoff or equivalent quieter transient handling for Gateway Tailscale startup after enabling Serve/Funnel, with `gateway.tailscale.binaryPath` suggested as a separate operator escape hatch.

## Best Possible Solution

Add bounded retry/backoff for the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status/hostname helper so all callers get the same transient handling. Intermediate failed attempts should not emit verbose `Command failed: tailscale status --json` noise until retries are exhausted; the binary-path config idea should stay separate unless maintainers explicitly want that schema/docs surface.

## Reproduction Assessment

Yes. A deterministic regression test can mock the post-Serve `tailscale status --json` call to fail once and then succeed; current main performs no retry and `runExec` logs the first failure before the startup catch can hide it. A live daemon restart was not run.

## Solution Assessment

Yes, the narrow maintainable fix is a small bounded retry/backoff with quiet intermediate logging around Tailscale status/hostname reads. Adding `gateway.tailscale.binaryPath` would be a broader operator-control feature and should not be bundled into the race fix unless maintainers request it.
