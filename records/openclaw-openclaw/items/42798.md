## Summary

Keep open: current main still has the source-level noisy Tailscale startup path, and the linked closing PR is open with changes requested, so this issue should remain open until that PR lands or is rejected.

## What This Changes

The issue reports a transient noisy `tailscale status --json` failure when Serve startup queries hostname immediately after enabling Tailscale Serve.

## Best Possible Solution

Resolve the linked PR or an equivalent narrow fix so optional post-Serve hostname discovery retries quietly while final status failures and real Serve/Funnel setup failures remain visible.

## Reproduction Assessment

Yes, at source level: Serve startup immediately calls `getTailnetHostname` after `enableTailscaleServe`, and a failing `tailscale status --json` reaches `runExec`'s verbose failure log before the caller catch swallows the optional hostname failure. I did not run a live macOS Tailscale restart.

## Solution Assessment

Yes for the narrow bug shape: a quiet or final-attempt-only retry around the optional status probe is the maintainable solution. Adding `gateway.tailscale.binaryPath` would be broader config surface and is not needed for this startup-noise race.
