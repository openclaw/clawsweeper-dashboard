## Summary

Current main still has the reported single-shot Tailscale hostname lookup after Serve/Funnel startup, so the prior ClawSweeper keep-open review remains accurate and this should be fixed rather than closed.

## What This Changes

The issue requests retry/backoff for transient `tailscale status --json` failures after Tailscale Serve startup.

## Best Possible Solution

Add a small bounded retry/backoff around the best-effort Tailscale hostname/status lookup after Serve/Funnel setup, with regression coverage and no new public binary-path setting.

## Reproduction Assessment

## Solution Assessment

Yes. A narrow retry/backoff around the status lookup is the maintainable fix; adding `gateway.tailscale.binaryPath` would be broader config surface and does not directly address the startup race.

## Repair Work Prompt

Fix the Tailscale startup race from https://github.com/openclaw/openclaw/issues/42798. Keep the change narrow: after `enableTailscaleServe`/`enableTailscaleFunnel`, retry the best-effort hostname lookup or `tailscale status --json` read once or twice with a short backoff so a transient first CLI failure does not emit misleading verbose startup noise. Add regression coverage for a first rejected status call followed by success. Do not add a new `gateway.tailscale.binaryPath` config option or otherwise expand the public config contract. Include a user-facing changelog entry if the startup/logging behavior changes. Validate with the targeted Tailscale helper/startup tests and then the changed gate in Testbox.
