## Summary

Keep open: current main still runs the post-Serve Tailscale hostname probe without a stabilization delay or retry, and a failed `tailscale status --json` still reaches the shared verbose/debug command-failure logger before the caller catches it.

## What This Changes

The issue reports transient `tailscale status --json` failures during `gateway.tailscale.mode: "serve"` startup and asks for retry/backoff or a binary-path configuration option.

## Best Possible Solution

Add a bounded stabilization delay or quiet retry around the post-Serve/Funnel hostname status probe, keeping fatal Serve/Funnel enable failures visible and leaving public config unchanged.

## Reproduction Assessment

No live high-confidence reproduction was established in this review. Source inspection gives a clear reproduction path if `tailscale status --json` exits non-zero immediately after Serve: `runExec()` logs the command failure under verbose/debug before `startGatewayTailscaleExposure()` catches the hostname lookup failure.

## Solution Assessment

Yes, the retry/backoff direction is the narrow maintainable fix. A new `gateway.tailscale.binaryPath` option is broader than the reported startup timing/logging bug and should not be required unless maintainers separately want that config surface.

## Repair Work Prompt

Fix the gateway Tailscale startup race reported in https://github.com/openclaw/openclaw/issues/42798. Current main enables Serve/Funnel and immediately probes `tailscale status --json`; when that transiently fails, `runExec()` logs `Command failed: tailscale status --json` under verbose/debug before the caller catches the hostname lookup failure. Add a bounded stabilization delay and/or quiet retry for the post-Serve/Funnel hostname probe so transient status failures during startup do not surface as one-off command-failed logs. Keep the scope to gateway Tailscale startup; do not add a public `gateway.tailscale.binaryPath` config option unless a maintainer explicitly asks. Preserve existing behavior for real Serve/Funnel enable failures, and if hostname resolution still fails after bounded attempts, keep startup non-fatal with the generic `<mode> enabled` log. Add focused Vitest coverage for transient status failure after Serve and update CHANGELOG.md with a user-facing fix entry. Do not change unrelated Tailscale auth, Funnel policy, or config schema behavior.
