## Summary

Keep open: current main and v2026.5.7 still perform a best-effort Tailscale hostname lookup with no bounded retry after Serve/Funnel setup, so the transient verbose error path remains plausible and narrow to fix.

## What This Changes

The issue asks for a bounded retry/backoff around `tailscale status --json` after Tailscale Serve/Funnel startup to avoid confusing transient startup log noise.

## Best Possible Solution

Add a small bounded retry/backoff around the best-effort Tailscale hostname/status lookup after Serve/Funnel setup, with regression coverage and no new public binary-path config option.

## Reproduction Assessment

Yes, at source level: mock a successful Serve setup followed by one rejected `tailscale status --json` lookup and the current path drops the hostname without retrying while verbose `runExec` logs the command failure. A live macOS launchd/Tailscale restart was not run in this read-only review.

## Solution Assessment

Yes. A bounded retry/backoff around the status lookup is the narrow maintainable fix; adding `gateway.tailscale.binaryPath` would expand public config and does not directly solve the startup race.
