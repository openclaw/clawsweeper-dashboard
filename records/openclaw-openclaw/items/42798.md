## Summary

Keep open: current main and the latest release still call `tailscale status --json` without a same-binary retry after Serve setup, and the linked closing PR is open rather than merged.

## What This Changes

The issue requests a narrow retry/backoff for transient `tailscale status --json` failures during Tailscale Serve gateway startup.

## Best Possible Solution

Review, refresh, and land https://github.com/openclaw/openclaw/pull/91553 or an equivalent narrow retry/backoff in `getTailnetHostname`, keeping the config surface unchanged.

## Reproduction Assessment

Yes at source level: current main runs Serve setup and immediately asks for `tailscale status --json` without a same-binary retry. I did not run a live macOS Homebrew Tailscale daemon restart under load in this read-only review, so this is not a live reproduction.

## Solution Assessment

Yes: a narrow retry/backoff around the status JSON read is the best current fix shape. The alternative binary-path config option would add config surface and does not directly address the transient status race.
