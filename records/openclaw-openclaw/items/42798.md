- `P2`: The bug creates confusing startup error logs for optional Tailscale exposure, but the report says the gateway continues and Serve usually configures correctly.

## Summary

Keep open: current main and the latest shipped release still query `tailscale status --json` immediately after enabling Serve without retry/backoff, and the linked closing fix at https://github.com/openclaw/openclaw/pull/82727 is still open and draft.

## What This Changes

The issue requests retry/backoff around the Tailscale status probe after Serve startup so transient daemon settling failures do not emit confusing verbose error logs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82727, or an equivalent narrow retry that quiets intermediate status failures while preserving final failure visibility, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: current main enables Serve and immediately queries `tailscale status --json`, and `runExec` emits verbose failure logs before the caller catch can swallow the optional hostname failure. I did not run a live macOS Tailscale restart.

## Solution Assessment

Yes. Retrying the optional post-Serve status probe is the narrow maintainable fix; adding `gateway.tailscale.binaryPath` would be broader config surface and would not directly address daemon settling after Serve setup.
