## Summary

Keep this open: current main still has a source-level path where a transient `tailscale status --json` failure is logged by the shared exec helper before the best-effort hostname lookup can swallow it, so the reported noisy startup failure is still plausible and narrowly fixable.

## What This Changes

The issue requests that Tailscale Serve startup tolerate transient `tailscale status --json` failures after `tailscale serve --bg --yes` without showing a confusing command-failed log.

## Best Possible Solution

Make the post-Serve hostname lookup truly best-effort by retrying or quieting only this status probe while preserving normal warnings for failed Serve/Funnel setup.

## Reproduction Assessment

Yes, at source level: with verbose/debug logging enabled, any non-zero `tailscale status --json` from `getTailnetHostname` causes `runExec` to log the command failure before `startGatewayTailscaleExposure` catches the hostname lookup. I did not run the reporter's live macOS launchd/Tailscale restart path.

## Solution Assessment

Yes for the retry or quiet best-effort status lookup; it is narrower than adding a new `gateway.tailscale.binaryPath` config option, which would solve a different discovery concern and widen the public config contract.
