## Summary

Keep this open: current main still has a source-level path where a transient `tailscale status --json` failure is logged by the shared exec helper before the best-effort hostname lookup can swallow it, so the reported noisy startup failure remains plausible and narrowly fixable.

## What This Changes

The issue asks OpenClaw to avoid transient verbose/debug `tailscale status --json` failure logs during `gateway.tailscale.mode: "serve"` startup, preferably via retry/backoff or a quiet best-effort status probe.

## Best Possible Solution

Make the post-Serve hostname lookup truly best-effort by retrying or quieting only this status probe while preserving normal visible warnings for failed Serve/Funnel setup and cleanup.

## Reproduction Assessment

Yes, at source level: with verbose/debug logging enabled, any non-zero `tailscale status --json` from `getTailnetHostname` causes `runExec` to log the command failure before the startup catch can treat hostname discovery as best-effort. I did not run the reporter's live macOS launchd/Tailscale restart path.

## Solution Assessment

Yes for a narrow retry or quiet best-effort status lookup; adding `gateway.tailscale.binaryPath` would be a broader public config change and does not directly address the noisy post-Serve status race.
