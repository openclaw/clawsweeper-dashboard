Keep open: the PR targets a real current-main allowlist regression, but the patch as written lets generic ZIP sniffs bypass the fallback MIME guard and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds gzip, tar, 7z, and zip MIME types to the host-read document allowlist and updates the unsupported-media error text.

## Best Possible Solution

Restore archive host-read sends with scoped MIME handling, regression tests for both valid archive sends and generic-ZIP masquerade cases, and contributor-provided live output showing the attachment send succeeds.

## Reproduction Assessment

Yes from source inspection: current main reads local media, sniffs application/zip, then rejects it because the host-read document allowlist lacks that MIME. I did not run a live Discord or message-tool send in this read-only review.

## Solution Assessment

No, not as written. Allowing archives is the right direction, but application/zip needs to be scoped so generic ZIP sniffs do not bypass fallback MIME rejection, and the PR needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep generic ZIP from bypassing fallback MIME checks:** `src/media/web-media.ts:112`
  - body: Adding `application/zip` to `HOST_READ_ALLOWED_DOCUMENT_MIMES` makes the sniffed-MIME branch return before the fallback branch that rejects extension-derived Office/PDF document types. Since `detectMime` preserves generic ZIP buffers as `.xlsx`/`.docx` based on the file extension, a plain ZIP named `report.xlsx` would now pass host-read as an Office document instead of being rejected or normalized to an archive; scope the ZIP allowance to real archive MIME/filenames and cover that case.
  - confidence: 0.84

## Security Review

Status: needs_attention
