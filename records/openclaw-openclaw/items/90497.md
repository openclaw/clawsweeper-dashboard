## Summary

Keep open: the PR targets a real asymmetric timeout gap, but the implementation appears too broad because it exposes every pending timeout before the 15s retry grace, including soft aborted/retry snapshots that current code intentionally delays.

## What This Changes

The PR changes the gateway `agent.wait` lifecycle fallback to return pending sub-agent timeout snapshots before retry-grace cleanup and adds a focused Vitest regression.

## Best Possible Solution

Surface only hard sub-agent run-timeout snapshots early, keep retry-grace protection for soft aborted timeouts, and add real parent-session timeout proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: emit a pending timeout lifecycle snapshot, let the caller timeout fire before the 15s grace window, and observe the fallback result. I did not run tests because this review is read-only.

## Solution Assessment

No. The fallback is the right layer for the reported hard-timeout gap, but the current PR should distinguish hard run timeouts from soft aborted retry-grace snapshots before returning terminal fields early.
