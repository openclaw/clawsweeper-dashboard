## Summary

Keep open: the linked Telegram duplicate-approval bug is still source-reproducible, but this PR is not merge-ready because the shared helper change can suppress Google Chat fallback delivery when its native handler cannot handle a missing turn source, and the PR still lacks real Telegram proof.

## What This Changes

The PR changes the shared plugin approval fallback suppression helper so a null `turnSourceChannel` no longer blocks native suppression, and adds a regression test for that helper-level case.

## Best Possible Solution

Limit missing-channel suppression to channels whose native handler can actually handle the request, preserve fallback delivery for Google Chat and other non-handleable cases, then land with visible Telegram proof of one approval prompt.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main exits suppression on null `turnSourceChannel` before checking the resolved Telegram account, while Telegram native matching can still accept the request by account/session binding; I did not run a live Telegram session.

## Solution Assessment

Is this the best way to solve the issue? No as submitted: the Telegram alignment is a good target, but changing the shared helper without request-specific native eligibility breaks the Google Chat sibling path.
