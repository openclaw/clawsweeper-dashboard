## Summary

Keep open: this is a narrow, likely correct fix for the linked Telegram duplicate approval bug, but the external PR still lacks real Telegram behavior proof. Current main and v2026.6.6 still have the old null-channel mismatch behavior, so the work is still necessary.

## What This Changes

The PR changes the shared plugin SDK approval fallback suppression helper so a missing `turnSourceChannel` no longer blocks native-delivery suppression, and adds a regression test for that null-channel case.

## Best Possible Solution

Land the helper fix after visible Telegram proof confirms one native approval prompt with no forwarding duplicate, while preserving explicit foreign-channel rejection and disabled-native fallback delivery.

## Reproduction Assessment

Yes at source level: current main exits fallback suppression on null `turnSourceChannel` before checking Telegram native account eligibility, while Telegram native matching can still accept by account/session. The patch's Google Chat regression is also source-reproducible from the shared helper plus Google Chat's turn-source-only native handler gate.

## Solution Assessment

No for the submitted patch: the helper-level direction is plausible for Telegram, but this exact shared condition is too broad for Google Chat. The safer fix is to align suppression with each channel's native `shouldHandle` or target resolver before suppressing fallback.
