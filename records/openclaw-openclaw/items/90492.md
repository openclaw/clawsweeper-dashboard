## Summary

Keep open: this PR targets the right Agent Teams lifecycle bug, but the patch currently treats any displayable child history string as success, which can turn a stale run with only tool-call activity into an `ok` completion. The external proof is also mock-only and does not show a real `sessions_spawn` / `sessions_yield` run after the change.

## What This Changes

The PR adds a lost-context outcome resolver, wires the subagent sweeper to complete stale runs as `ok` when child history has output, and adds focused regression tests.

## Best Possible Solution

Keep the narrow sweeper reconciliation, but gate `ok` on classified visible child result text, handle output-read failures as lost-context, and require real Agent Teams proof before merge.

## Reproduction Assessment

Yes, source-level: current main has a stale-run lost-context fallback, and the linked issue shows that the announce path can still carry child output beside that failure status. I did not run a live gateway reproduction in this read-only pass.

## Solution Assessment

No, not yet: the fix location is plausible, but using `readSubagentOutput` as a success predicate is too broad and needs a classified result plus a read-failure fallback.

- **[P1] Require real child output before marking lost context ok:** `src/agents/subagent-lost-context-completion.ts:16-18`

  - confidence: 0.88

  - body: `readSubagentOutput` reaches `chat.history` through `callGateway` and can reject. Because this await happens before `completeSubagentRunWithRecovery`, a gateway/history failure now aborts the sweep instead of completing the stale run with the existing lost-context error; catch lookup failures and return the error outcome.
  - confidence: 0.82
