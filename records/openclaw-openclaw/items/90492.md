## Summary

Keep open: current main and v2026.6.6 still have the reported lost-context fallback, and this PR contains useful work, but the patch is not merge-ready because it treats announce-display output as terminal success and can abort the fallback if output lookup fails. The later replacement attempts are closed unmerged, so they are not a safe superseding target.

## What This Changes

The PR adds a resolver that turns stale lost-context subagent runs into `ok` when `readSubagentOutput` returns text, wires it into the registry sweeper, and adds focused Vitest coverage.

## Best Possible Solution

Keep the narrow sweeper reconciliation idea, but gate recovery on a classified genuine terminal child result, catch output-read failures as lost-context, and require real Agent Teams proof before merge.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.6 still have a lost-context sweeper fallback, and the linked issue includes a transcript where the parent receives that failure status alongside child output. I did not run a live Agent Teams gateway repro in this read-only review.

## Solution Assessment

No, not yet: the fix location is plausible, but using `readSubagentOutput` as the success predicate is too broad and the new await needs a lost-context fallback on lookup failure. A stricter terminal-result reader is the safer solution shape.

- **[P1] Require a terminal child result before success:** `src/agents/subagent-lost-context-completion.ts:16-18`
  - body: `readSubagentOutput` is an announce-display helper, not a terminal-success predicate: it can return synthetic tool-call summaries or visible pre-tool text that does not prove the child finished.   - confidence: 0.88
- **[P2] Preserve the lost-context fallback on lookup errors:** `src/agents/subagent-lost-context-completion.ts:16`
  - body: This await can reject through `chat.history` before `completeSubagentRunWithRecovery` runs, which means the sweeper may fail to settle the stale run at all. Catch output lookup failures and return the existing lost-context error outcome instead.
  - confidence: 0.82
