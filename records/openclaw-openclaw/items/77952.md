## Summary

Keep open: the Discord cache fix is in the right owner boundary and has after-fix runtime proof, but the new expiry test can fail nondeterministically because it relies on a 1 ms wall-clock TTL before asserting the pre-sweep size.

## What This Changes

The PR bounds the Discord REST entity cache by adding write-time expiry sweeps, a 5,000-entry FIFO cap, a size getter, regression tests, and a changelog entry.

## Best Possible Solution

Keep the plugin-local cache cap/sweep approach, but make the new expiry regression test deterministic before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows unique cache keys are never swept or capped, while the PR body provides after-fix runtime output for the bounded-cache behavior.

## Solution Assessment

Mostly yes. The owner-local cap and write-time sweep are a narrow maintainable fix, but the timer-based test should use fake timers or another deterministic clock path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use deterministic time in the expiry sweep test:** `extensions/discord/src/internal/entity-cache.test.ts:33`
  - body: This test inserts `u1`, then awaits another async fetch with a 1 ms TTL and `sweepIntervalMs: 0`, so if the second insert lands a millisecond later it can sweep `u1` before the `expect(cache.size).toBe(2)` precondition. Use fake timers or an injected clock for the expiry path instead of wall-clock timing.
  - confidence: 0.87
