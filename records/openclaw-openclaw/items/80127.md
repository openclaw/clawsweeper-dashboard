## Summary

Keep open: the report is concrete and recent, and current docs promise one poller per Telegram bot token, but current source already has duplicate-start guards, so the exact bypass needs log-backed investigation rather than cleanup closure.

## What This Changes

The issue reports that gateway startup initializes each configured Telegram account twice, leaving two `getUpdates` pollers per bot token and causing persistent Telegram 409 conflicts.

## Best Possible Solution

Identify the startup or restart path that bypasses the existing account and token guards, then add a focused regression test and ensure a second `getUpdates` poller cannot start for the same Telegram bot token.

## Reproduction Assessment

No, we do not have a high-confidence current-main reproduction path. The issue gives a clear live reproduction, but source inspection shows existing duplicate-start and duplicate-token guards, so the failing path still needs logs or a live multi-account setup to isolate.

## Solution Assessment

No, adding another generic guard is not yet proven to be the best fix. The maintainable path is to determine why the existing `server-channels` and Telegram polling lease guards did not prevent the reported duplicate starts, then patch that specific lifecycle path.
