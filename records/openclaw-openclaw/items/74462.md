## Summary

Keep open. Current main still rejects both requested `contextSafety` config paths, and the runtime surface named by the reporter is not present in current source, so this needs maintainer product/architecture handling rather than cleanup closure or a schema-only autofix.

## What This Changes

The issue requests a durable `contextSafety` configuration contract because the reported runtime behavior reads `session.contextSafety` or top-level `contextSafety` while strict config validation rejects both keys.

## Best Possible Solution

Maintainers should consolidate this with #75699 and decide the intended contract. If automatic context rollover is intended, implement it end to end across runtime behavior, strict schema, exported config types, schema/help/label metadata, docs, and focused validation/runtime tests; if it is not intended, document that `agents.defaults.compaction` and existing `/context`/`/compact` commands are the supported surfaces and close the legacy `contextSafety` path explicitly.

## Reproduction Assessment

Yes for the schema rejection: current main's strict `session` and root schemas reject unknown `contextSafety` keys before config can be persisted. The runtime-support half is unclear because the reported `contextSafety` runtime file and `/context_status`/handoff/rollover slash commands are absent from current source.

## Solution Assessment

Unclear. The suggested schema-only fix is not the best path until maintainers confirm whether `contextSafety` is a supported contract; the safer alternatives are either an end-to-end rollover implementation or explicit documentation that the supported controls live under `agents.defaults.compaction`.
