## Summary

Keep open. Current main still rejects both requested `contextSafety` config paths, and the exact runtime surface the reporter cites is not present in current source, so this is a real config-contract/product question rather than a safe cleanup close or schema-only autofix.

## What This Changes

Expose a durable validated `contextSafety` configuration path, preferably under `session.contextSafety`, because the reporter observed config validation rejecting explicit context-safety settings.

## Best Possible Solution

Maintainers should decide the intended product contract first. If `contextSafety` is intended, implement it end to end across runtime behavior, strict schema, exported config types, schema/help/label metadata, docs, and focused validation/runtime tests; if it is not intended, document the supported `agents.defaults.compaction` controls and ask for a fresh installed-package reproduction of the divergent dist artifact.

## Reproduction Assessment

Yes for the schema rejection: the current strict schemas make both `session.contextSafety` and top-level `contextSafety` unknown keys. The runtime-support half is unclear because the reported runtime file and slash commands are absent from current main source.

## Solution Assessment

Unclear. A schema-only patch is not the best solution until maintainers confirm whether `contextSafety` is a supported contract; the safer path is either an end-to-end implementation or explicit documentation that `agents.defaults.compaction` is the supported configuration surface.
