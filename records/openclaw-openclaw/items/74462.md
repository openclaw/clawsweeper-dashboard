## Summary

Keep open. Current main still rejects both requested `contextSafety` config paths, and current source does not contain the runtime symbol or slash commands cited from the installed artifact, so this remains a config-contract/product decision rather than a safe cleanup close.

## What This Changes

The issue asks for a durable supported config schema, type, metadata, and docs path for `contextSafety` because installed runtime artifacts reportedly read it while `openclaw.json` validation rejects it.

## Best Possible Solution

Maintainers should choose the intended durable contract: implement `contextSafety` end to end if it is still a supported feature, or document `agents.defaults.compaction` as the supported surface and ask for fresh installed-package provenance for the divergent artifact.

## Reproduction Assessment

Yes for the schema rejection: the current strict schemas make both `session.contextSafety` and top-level `contextSafety` unknown keys. The runtime-support half is unclear because the cited `contextSafety` symbol and slash commands are absent from current source.

## Solution Assessment

Unclear. A schema-only patch is not the best fix until maintainers confirm whether `contextSafety` is intended public config; the safer solution is either an end-to-end implementation or a docs/support correction pointing users to the existing compaction config surface.
