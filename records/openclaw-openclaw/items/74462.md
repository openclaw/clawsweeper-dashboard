## Summary

Keep open. Current main still rejects both requested `contextSafety` config paths, but the runtime and slash-command surface described in the report is not present in current source, so this is a real config-contract question that needs maintainer product/architecture confirmation before a safe fix.

## What This Changes

The issue requests durable support for context-safety configuration, preferably under `session.contextSafety`, with schema, exported types, config metadata, docs, and runtime behavior aligned.

## Best Possible Solution

Decide the product contract first. If `contextSafety` is intended, implement it end to end across runtime, strict schema, exported config types, generated schema/help/labels, docs, and focused validation/runtime tests; if it is not intended, document the supported `agents.defaults.compaction` controls and ask for a fresh reproduction of the divergent runtime artifact.

## Reproduction Assessment

Yes for the schema rejection: current source shows both `session.contextSafety` and top-level `contextSafety` are rejected by strict schemas. The runtime-support half is unclear because the reported runtime file and commands are absent from current main.

## Solution Assessment

Unclear. A schema-only change is not the best fix unless maintainers confirm the intended runtime contract; the safer path is an end-to-end config/runtime/docs decision or explicit documentation that the supported path is `agents.defaults.compaction`.
