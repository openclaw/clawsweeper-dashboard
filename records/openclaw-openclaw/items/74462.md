## Summary

Keep open. Current main still rejects both requested `contextSafety` config paths, and the current source does not contain the runtime or slash-command surface named in the report, so this is a real config-contract question that needs maintainer direction before implementation.

## What This Changes

The issue requests a durable `contextSafety` config contract, preferably under `session.contextSafety`, because the reporter saw runtime code reading that key while config validation rejects it.

## Best Possible Solution

Decide the product contract first. If `contextSafety` is intended, implement it end to end across runtime behavior, schema, exported config types, generated schema/help metadata, docs, and focused validation/runtime tests; if it is not intended, document the supported `agents.defaults.compaction` path and investigate the divergent dist artifact the reporter saw.

Yes for the validation failure: current source shows both `session.contextSafety` and top-level `contextSafety` are rejected by strict schemas. The runtime-support half is unclear because current main has no matching `contextSafety` symbols or the named context-safety slash commands.

## Solution Assessment

Unclear. A schema-only addition is not the best fix unless maintainers confirm the runtime contract; the safer path is either an end-to-end `contextSafety` feature or explicit docs steering users to the existing `agents.defaults.compaction` controls.
