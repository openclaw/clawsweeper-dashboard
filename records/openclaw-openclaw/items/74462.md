## Summary

Keep open. Current main still rejects both requested `contextSafety` config paths, but the reported runtime reader and slash commands are absent from current source, so this is a real config-contract/product decision rather than a safe cleanup close or schema-only repair.

## What This Changes

The issue asks OpenClaw to provide a durable config schema/type/docs surface for context safety settings that the reporter says installed runtime artifacts already read.

## Best Possible Solution

Maintainers should decide the intended contract first. If `contextSafety` is intended, implement it end to end across runtime behavior, strict schema, exported config types, generated schema/help/label metadata, docs, and focused tests; if not, document `agents.defaults.compaction` as the supported control surface and ask for a fresh installed-package reproduction of the divergent dist artifact.

## Reproduction Assessment

Yes for the schema rejection: source inspection shows both `session.contextSafety` and top-level `contextSafety` are unknown under strict schemas. The runtime-support half is unclear because the reported runtime reader and slash commands are absent from current main source.

## Solution Assessment

Unclear. A schema-only patch is not the best way to solve this until maintainers confirm whether `contextSafety` is a supported contract; the safer alternatives are an end-to-end implementation or explicit documentation that existing `agents.defaults.compaction` controls are the supported path.
