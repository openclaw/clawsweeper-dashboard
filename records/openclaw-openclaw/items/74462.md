## Summary

Keep open. Current main still rejects both requested durable config paths, and the current source tree has no `contextSafety` runtime reader or reported slash-command surface, so this is a real config-contract/product question rather than an obsolete or implemented issue.

## What This Changes

The issue requests a durable config schema, TypeScript type, metadata, and docs path for context safety settings that the reporter says runtime code already consumes.

## Best Possible Solution

Decide the intended product contract first. If context safety is intended, implement it end to end with schema, exported types, generated schema/help metadata, docs, validation tests, and runtime tests proving the durable key is consumed; otherwise document the supported `agents.defaults.compaction` path and avoid accepting dead `contextSafety` aliases.

## Reproduction Assessment

Yes for the schema rejection: current source shows both `session.contextSafety` and top-level `contextSafety` are rejected by strict schemas used in raw config validation. The runtime-support half is unclear because the reported runtime reader and slash commands are absent from current main.

## Solution Assessment

Unclear. A schema-only addition is not the best fix unless maintainers confirm the runtime contract; the safer path is an end-to-end config/runtime/docs decision or a clear redirect to existing compaction settings.
