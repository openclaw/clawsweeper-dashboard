## Summary

This PR should stay open: it targets a real current-main TUI watchdog notice bug, but the current patch has a small render-path defect and the PR is still draft/protected with only test-based proof.

## What This Changes

The PR adds run-scoped pending system notices to the TUI chat log, uses them for the streaming watchdog warning, dismisses the warning on later same-run chat events, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the run-scoped pending notice approach, but make dismissal trigger a render even when the next delta has no displayable text, then verify the slow-run and stuck-run TUI behavior in a real session.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: let the current-main watchdog append its notice, then deliver a later same-run delta or final; current main has no dismissal path for the earlier system message.

## Solution Assessment

No, not as written. The run-scoped pending notice is the right narrow direction, but the patch must also render after dismissal on no-display deltas so the visible notice actually disappears when the run first resumes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Render after dismissing stale watchdog notices:** `src/tui/tui-event-handlers.ts:399`
  - body: `dismissPendingSystem` mutates the chat log before delta processing, but a later delta can still return at the no-display path without calling `tui.requestRender()`. That means a commentary-only or tool-call delta can remove the notice from state while leaving it visible until some later render, which misses the PR's intended "moment the response continues" behavior.
  - confidence: 0.86
