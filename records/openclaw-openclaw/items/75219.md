## Summary

Keep open: the PR targets a real provider retry gap, but the current patch still changes timeout abort semantics in a way that can break existing timeout handling, still misses TimeoutError DOMExceptions with numeric codes, and the external real behavior proof remains mock-only.

## What This Changes

The PR adds an opt-in transient retry policy for provider execution, exposes provider HTTP retry helpers, wires read/poll/download retries into media, embedding, and video providers, and adds tests plus a changelog entry.

## Best Possible Solution

Land the stage-aware retry seam after preserving timeout abort compatibility, classifying timeout-shaped errors before numeric status/code handling, and adding redacted real provider retry proof; keep https://github.com/openclaw/openclaw/issues/60422 open until a fix merges.

## Reproduction Assessment

Yes. Current main source shows executeWithApiKeyRotation only rotates on caller policy or 429-like rate limits, so a mocked single-key 500/timeout path can reproduce the missing retry with high confidence.

## Solution Assessment

No, not yet. The central opt-in retry seam is a maintainable direction, but the patch needs the timeout compatibility and classifier fixes plus real provider proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve abort-compatible timeout handling:** `src/utils/fetch-timeout.ts:90-92`
  - body: `fetchWithTimeout` now aborts with an `Error` named `TimeoutError`. Existing callers such as media attachment caching only convert `isAbortError(err)` into a controlled timeout skip, and `isAbortError` does not recognize this new shape, so a fetch timeout can now escape as a hard media-understanding failure instead of being skipped.
  - confidence: 0.92
- **[P2] Check timeout-shaped errors before numeric codes:** `src/provider-runtime/operation-retry.ts:145-149`
  - body: `readErrorStatus` treats any numeric `code` as a status and returns before `hasTimeoutSignal` runs. Node timeout DOMExceptions use `name === "TimeoutError"` with `code === 23`, so SDK/provider timeout errors with that shape still skip the new transient retry path.
  - confidence: 0.93
