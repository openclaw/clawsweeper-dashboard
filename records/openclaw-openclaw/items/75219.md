## Summary

Keep open: the retry fix is still relevant to the open provider reliability bug, but the latest PR head has blocking timeout-handling defects and still lacks after-fix real provider proof.

## What This Changes

The branch adds an internal transient provider retry helper, wires opt-in same-key retries into API-key rotation, media audio/video, and Google embeddings, tags fetch timeouts, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the opt-in same-key retry approach, but classify timeout-named errors before numeric status checks, preserve existing timeout/AbortError handling, rebase onto current main, and require redacted real provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main only rotates on 429-style errors and the PR finding is backed by a Node 22 probe showing real TimeoutError values carry numeric code 23.

## Solution Assessment

No, not yet. The opt-in retry boundary is maintainable, but the current PR head skips real DOMException timeouts and changes timeout abort compatibility without updating existing timeout handlers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve AbortError timeout handling:** `src/utils/fetch-timeout.ts:75`
  - body: Changing fetch timeout aborts to a `TimeoutError` means existing callers that only treat `isAbortError(err)` as a controlled timeout no longer match. For example, media attachment fetching maps only `isAbortError(err)` to `MediaUnderstandingSkipError("timeout")`, so a remote media fetch timeout can escape as a hard failure instead of the existing skip path.
  - confidence: 0.88
- **[P2] Classify TimeoutError before numeric status codes:** `src/agents/provider-operation-retry.ts:129-131`
  - body: `AbortSignal.timeout()` produces a `TimeoutError` DOMException with numeric `code === 23`. Because `readErrorStatus` treats any integer `code` as a status and returns before `hasTimeoutSignal(...)`, these real provider/SDK timeout errors are classified non-transient and skip the opt-in retry path.
  - confidence: 0.94
