## Summary

Close as ClawHub-scoped: current OpenClaw no longer ships or supports the BlueBubbles channel, the supported iMessage path is the bundled `imessage` plugin via `imsg`, and any BlueBubbles-specific Tahoe GUID handling now belongs in an externally maintained channel plugin rather than core.

## What This Changes

The issue asks OpenClaw to stop BlueBubbles catchup/history polling from assuming `iMessage;-;` direct-message chat GUIDs on macOS Tahoe.

## Best Possible Solution

Keep BlueBubbles out of OpenClaw core, guide users to the supported `imsg` iMessage path, and publish any maintained BlueBubbles-specific channel behavior through ClawHub using the channel plugin API.

## Reproduction Assessment

No for current main: the report is plausible for OpenClaw 2026.4.24 with the retired BlueBubbles channel, but current main no longer contains a BlueBubbles runtime or `channels.bluebubbles` support path to reproduce against.

## Solution Assessment

No for core: querying BlueBubbles for the actual chat GUID may be the right repair for an external BlueBubbles plugin, but the maintainable OpenClaw-core solution is the already-documented migration to `channels.imessage` via `imsg`.
