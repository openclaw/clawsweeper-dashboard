## Summary

Keep open: current main still has the reported disconnect path, but an open linked fix PR already references this issue with closing syntax, and the disconnect-cancels-run behavior needs maintainer review against existing reconnect/queued-abort semantics.

## What This Changes

The issue requests immediate cancellation of in-flight chat/agent runs whose `ownerConnId` matches a WebSocket connection when that connection closes.

## Best Possible Solution

Make an explicit maintainer decision on whether WS close means cancel or transient reconnect, then either land the linked PR with regression coverage or close that PR with the intended reconnect-preservation policy documented in tests.

## Reproduction Assessment

Yes for the current behavior: source inspection shows the close handler leaves `chatAbortControllers` untouched while chat and agent runs record `ownerConnId`, and the linked PR provides a concrete wire-level probe. I did not rerun the live probe in this read-only review.

## Solution Assessment

Unclear until maintainers settle the disconnect policy. Technically, using `abortChatRunById` for matching `ownerConnId` entries is narrow, but it may conflict with existing reconnect and queued-abort behavior.
