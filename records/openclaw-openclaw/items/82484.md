## Summary

Keep open: current main still lacks disconnect-driven cleanup for owned chat abort controllers, and the open linked PR is the focused fix path that should be reviewed and landed before this issue closes.

## What This Changes

The issue reports that gateway WebSocket disconnect cleanup does not abort in-flight chat or agent runs whose `chatAbortControllers` entry belongs to the closing connection.

## Best Possible Solution

Review and land the linked PR or an equivalent small patch that performs owner-connection cleanup in `src/gateway/server/ws-connection.ts` using the existing abort semantics.

## Reproduction Assessment

Yes. Current main shows the close handler does not inspect `chatAbortControllers` while chat and agent handlers register entries with `ownerConnId`, and the reporter supplied a concrete wire-level reproduction with five failing trials.

## Solution Assessment

Yes. The narrow maintainable fix is to abort only `chatAbortControllers` entries whose `ownerConnId` matches the closing WebSocket by calling the existing `abortChatRunById` helper.
