## Summary

Keep open. Current main still accepts arbitrary `tools.web.search.provider` strings during config validation and only warns/falls back at runtime, while plugin manifests already expose provider IDs that could support validation.

## What This Changes

Request to validate `tools.web.search.provider` against built-in and plugin-registered web search provider IDs instead of accepting any string.

## Best Possible Solution

Add a manifest-backed validation step in the plugin-aware config validation path that derives allowed web-search provider IDs from bundled and installed plugin manifests, reports clear unknown-provider diagnostics with allowed values for likely typos, and preserves an intentional warning path for stale plugin-owned config when the owning plugin is missing.

## Repair Work Prompt

Fix the config-validation gap from https://github.com/openclaw/openclaw/issues/53092. Current main accepts arbitrary `tools.web.search.provider` strings because the base schema uses `z.string().optional()` and plugin-aware validation never checks that selector against manifest web-search provider IDs. Add a control-plane validation step in `src/config/validation.ts` that derives valid provider IDs from bundled/installed plugin manifest `contracts.webSearchProviders` plus any built-in/provider compatibility IDs already represented in manifests. Unknown typos should produce a clear config issue on `tools.web.search.provider` with allowed values where practical. Preserve startup resilience for stale plugin-owned config: if the provider appears to belong to a missing/removed plugin or there is explicit stale plugin evidence, prefer the existing warning/doctor path rather than a hard startup break, per the issue proposal. Do not import runtime provider implementations during config validation, do not add bundled provider IDs directly to core when manifest lookup can provide them, and do not add startup/load-time migrations. Update focused tests in `src/config/config.web-search-provider.test.ts` and adjust runtime tests only if the validation boundary changes expectations. Update generated config docs/schema baselines if the diagnostics or allowed values are surfaced there. Add a user-facing changelog fix entry only if behavior changes are visible, crediting `Thanks @TinyTb` and not crediting bots.
