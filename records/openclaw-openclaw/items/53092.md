## Summary

Keep open. Current main still leaves `tools.web.search.provider` as an unconstrained string in the base config schema and the plugin-aware validator still does not check that field against manifest `webSearchProviders`; runtime instead warns and falls back for an unknown configured provider, so the reported validation gap remains real.

## What This Changes

The issue requests plugin-aware validation for `tools.web.search.provider` so built-in and manifest-registered web search provider ids are accepted while typos or unknown ids are reported.

## Best Possible Solution

Add a manifest-backed validation step in the plugin-aware config validator that derives known web search provider ids from bundled and installed plugin metadata, reports a clear issue with allowed values for likely typos, and preserves a documented warning path only for intentionally resilient stale-plugin cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: pass a config with `tools.web.search.provider: "missing-id"` through the current config validator and it is accepted by the base `z.string()` path, while runtime tests show the typo later falls back to DuckDuckGo.

## Solution Assessment

Yes. The issue’s proposed manifest-backed validation is the narrowest maintainable direction because provider ownership already lives in `contracts.webSearchProviders`, so core does not need hardcoded provider ids or runtime plugin imports.
