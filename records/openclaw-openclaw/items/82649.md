## Summary

Keep open: source and dependency checks support the reported failure path, but this branch broadens the default secret-file permission contract and still may not fix the root-owned `/etc` setup. The prior ClawSweeper review already flagged the same proof and security gaps, and no later real-environment proof is present.

## What This Changes

The PR changes the file secret provider to pass `allowReadableByOthers` to `readSecureFile`, includes the underlying SecretRef resolution cause in gateway auth errors, and adds a POSIX regression test plus changelog entry.

## Best Possible Solution

Define a narrow shared-system-secret permission policy that preserves strict defaults, then add regression coverage and redacted real Ubuntu CLI/TUI proof for the accepted `/etc/openclaw/secrets.json` setup.

## Reproduction Assessment

Yes, by source inspection: current main rejects group/world-readable file providers, and the CLI/TUI auth paths surface SecretRef resolution failures before connecting. I did not run a live cross-uid `/etc` setup in this read-only review.

## Solution Assessment

No. The unconditional `allowReadableByOthers` change is broader than the reported group-readable system-secret case and does not address the upstream not-owned check, so the safer path is an explicit, security-reviewed permission contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep secret files from becoming world-readable:** `src/secrets/resolve.ts:299`
  - body: `allowReadableByOthers` permits both group- and world-readable files in `@openclaw/fs-safe`, while `readSecureFile()` still rejects files owned by another POSIX uid. Applying it to every file secret provider means a 0644 secret can resolve without `allowInsecurePath`, but a root-owned `/etc` secret can still fail as not-owned, weakening the default secret-file contract without reliably fixing the reported deployment.
  - confidence: 0.89
