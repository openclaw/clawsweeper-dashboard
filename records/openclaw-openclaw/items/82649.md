## Summary

Keep this PR open: it targets a real, recent gateway SecretRef bug, but the current diff broadens the file-secret permission contract in a security-sensitive way and still lacks after-fix real CLI/TUI proof.

## What This Changes

The PR passes `allowReadableByOthers` into file-provider `readSecureFile`, appends provider failure causes to unresolved gateway SecretRef messages, adds POSIX regression coverage, and updates the changelog.

## Best Possible Solution

Define a narrow shared-system-secret permission policy that preserves strict defaults, then add regression coverage and redacted real Ubuntu CLI/TUI proof for the accepted `/etc/openclaw/secrets.json` setup.

## Reproduction Assessment

Yes, source-reproducible: current `main` rejects group/world-readable file-provider payloads, and gateway CLI/TUI auth paths must resolve `gateway.auth.token` before connecting. I did not establish a live cross-uid `/etc` reproduction in this read-only review.

## Solution Assessment

No. The unconditional `allowReadableByOthers` change is broader than the reported group-readable system-secret case and does not address the not-owned check, so the safer path is an explicit security-reviewed permission contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep file SecretRefs from accepting world-readable secrets:** `src/secrets/resolve.ts:299`
  - body: `allowReadableByOthers` in `@openclaw/fs-safe` permits both group- and world-readable files, and the new test uses `0o644`, so this makes 0644 secret payloads resolve without `allowInsecurePath`. It also still cannot read a root-owned `/etc` file because the helper keeps the current-uid ownership check, so this weakens the default secret-file contract without reliably fixing the reported deployment.
  - confidence: 0.89
