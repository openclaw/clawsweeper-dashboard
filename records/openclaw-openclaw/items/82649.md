## Summary

Keep this PR open, but the current patch should not merge as-is: it broadens the file-secret permission contract and still may not solve the reported cross-uid `/etc/openclaw/secrets.json` setup.

## What This Changes

The PR adds `allowReadableByOthers` to file-provider secure reads, includes SecretRef failure causes in gateway auth errors, and adds a POSIX regression test plus changelog entry.

## Best Possible Solution

Define a narrow, security-reviewed system-secret file contract that preserves strict defaults and then prove CLI/TUI resolution against the real `/etc/openclaw/secrets.json` deployment shape.

## Reproduction Assessment

Yes from source inspection: current main rejects group/world-readable file-provider payloads, and CLI/TUI gateway auth resolution reports the unresolved SecretRef path before connecting. I did not establish a live cross-uid `/etc` reproduction in this read-only review.

## Solution Assessment

No. The proposed unconditional `allowReadableByOthers` change is broader than the reported group-readable system-secret case and does not address the remaining not-owned check in `readSecureFile`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep secret files from becoming world-readable:** `src/secrets/resolve.ts:299`
  - body: `allowReadableByOthers` permits both group- and world-readable POSIX files in the secure-file helper, while the helper still rejects files owned by another uid. Applying it to every file secret provider lets 0644 secret payloads resolve without `allowInsecurePath`, but a root-owned `/etc` secret can still fail as `not-owned`, weakening the default secret-file contract without reliably fixing the reported deployment.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: Needs attention: the patch can accept world-readable secret payload files without fixing cross-uid owner rejection.

Concerns:

  - body: The added `allowReadableByOthers: true` maps to both group- and world-readable POSIX files in `@openclaw/fs-safe`. That distinction is acceptable for executable command paths, but applying it to secret payload files can silently accept secrets readable by all local users.
