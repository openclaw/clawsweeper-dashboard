## Summary

This PR should stay open because it addresses a real linked CLI/TUI SecretRef failure, but the proposed permission change broadens the secret-file default to accept world-readable payloads and the PR still lacks after-fix real environment proof.

## What This Changes

The PR passes `allowReadableByOthers: true` into file SecretRef payload reads, adds provider error detail to unresolved gateway SecretRef messages, adds POSIX regression coverage, and updates the changelog.

## Best Possible Solution

Define a narrow, security-reviewed shared-system-secret permission contract that fixes the `/etc/openclaw/secrets.json` CLI/TUI case without making ordinary world-readable secret payloads valid by default.

## Reproduction Assessment

Yes, source-reproducible: current main rejects readable-by-others file-provider payloads, and the linked issue shows CLI/TUI gateway auth paths failing when the token comes from `/etc/openclaw/secrets.json`. I did not establish a live cross-uid Ubuntu reproduction in this read-only review.

## Solution Assessment

No. The current PR is broader than the reported group-readable system-secret case because `allowReadableByOthers` also allows world-readable files and does not solve ownership mismatch by itself.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep file SecretRefs from accepting world-readable secrets:** `src/secrets/resolve.ts:294-300`
  - body: Passing `allowReadableByOthers: true` for every file-provider payload makes 0644 secret files resolve without `allowInsecurePath`, and the added regression test exercises exactly that mode. The underlying secure-file helper treats this option as bypassing both group- and world-readable rejections, while still enforcing ownership, so the change weakens the default secret-file contract without reliably covering root-owned `/etc` deployments.
  - confidence: 0.89
