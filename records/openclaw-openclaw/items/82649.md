## Summary

Keep open: this is a plausible fix for the linked CLI/TUI SecretRef bug, but it should not merge as-is because it globally relaxes secret-file readability and still does not prove the reported cross-uid `/etc/openclaw/secrets.json` setup is fixed.

## What This Changes

The PR passes `allowReadableByOthers` to file secret-provider reads, appends underlying SecretRef resolution causes to gateway auth errors, and adds a POSIX regression test plus changelog entry.

## Best Possible Solution

Define a narrow shared-system-secret permission contract that keeps default secret files strict, then prove CLI/TUI resolution against a real `/etc/openclaw/secrets.json` style setup.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main rejects readable-by-others file providers, while `readSecureFile()` still rejects different POSIX owners even with `allowReadableByOthers`. I did not run a live `/etc` cross-uid repro in this read-only review.

## Solution Assessment

No. The unconditional `allowReadableByOthers` change is broader than the reported group-readable system-secret case and does not address the upstream not-owned check, so a narrower security-reviewed permission model is needed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep secret files from becoming world-readable:** `src/secrets/resolve.ts:299`
  - body: `allowReadableByOthers` permits both group- and world-readable files in `@openclaw/fs-safe`, while `readSecureFile()` still rejects files owned by another POSIX uid. Applying it to every file secret provider means a 0644 user secret starts resolving without `allowInsecurePath`, but a root-owned `/etc` secret can still fail as `not-owned`, so this weakens the default secret-file contract without fixing the reported deployment.
  - confidence: 0.89
