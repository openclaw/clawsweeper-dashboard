## Summary

Keep open: the linked auth failure is source-reproducible, but this PR broadens the default secret-file permission contract in a security-sensitive way and still lacks real CLI/TUI proof for the reported `/etc` setup.

## What This Changes

The PR adds `allowReadableByOthers` to file SecretRef payload reads, surfaces resolver failure causes in gateway SecretRef errors, adds a POSIX regression test, and updates the changelog.

## Best Possible Solution

Define an explicit shared-system-secret permission policy that preserves strict defaults, then update resolver coverage, docs, and redacted CLI/TUI proof for the accepted `/etc` setup.

## Reproduction Assessment

Yes, source-reproducible: current main rejects group/world-readable file-provider payloads, and gateway CLI/TUI auth paths must resolve `gateway.auth.token` before connecting. I did not establish a live cross-uid `/etc` reproduction in this read-only review.

## Solution Assessment

No. Passing `allowReadableByOthers` unconditionally is broader than the reported shared `/etc` case because it accepts world-readable secret payloads and still does not solve files owned by another uid.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep file SecretRefs from accepting world-readable secrets:** `src/secrets/resolve.ts:300`
