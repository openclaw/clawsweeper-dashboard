## Summary

Keep this PR open for human follow-up: the linked bug is plausible from source, but this patch broadens the default secret-file permission contract and still lacks after-fix proof from a real `/etc` CLI/TUI setup.

## What This Changes

The PR passes `allowReadableByOthers` to file-backed SecretRef reads, appends provider failure causes to unresolved gateway SecretRef messages, adds a POSIX resolver test, and updates the changelog.

## Best Possible Solution

Define a narrow shared-system-secret permission policy that preserves strict defaults, then add focused regression coverage and redacted Ubuntu package proof for the accepted `/etc/openclaw/secrets.json` flow.

## Reproduction Assessment

Yes, source-reproducible: current main fails closed for group/world-readable file-provider payloads, and gateway CLI/TUI auth paths must resolve `gateway.auth.token` before connecting. I did not establish a live cross-uid `/etc` reproduction in this read-only review.

## Solution Assessment

No. The unconditional `allowReadableByOthers` change is broader than the reported group-readable system-secret case and still leaves the current-uid ownership check in place; the safer path is an explicit security-reviewed permission contract.
