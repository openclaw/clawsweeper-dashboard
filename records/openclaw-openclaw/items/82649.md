## Summary

Keep open: the reported failure path is plausible on current main, but this PR should not merge as-is because it weakens the default secret-file permission contract, may still miss the root-owned `/etc` case, and lacks real CLI/TUI proof.

## What This Changes

The PR passes `allowReadableByOthers: true` to file-backed SecretRef reads, appends underlying resolution errors to gateway auth messages, adds a POSIX regression test, and updates the changelog.

## Best Possible Solution

Define a narrow, security-reviewed shared-system-secret permission contract that keeps ordinary secret files strict, then prove CLI/TUI resolution against a real `/etc/openclaw/secrets.json` style setup.

## Reproduction Assessment

Yes from source for the core failure mode: current main rejects group/world-readable file providers before CLI/TUI gateway auth can resolve. The exact reporter `/etc` cross-uid setup was not live reproduced, and the PR test does not cover a different owner uid.

## Solution Assessment

No. Unconditionally allowing readable-by-others files is broader than the reported group-readable case and does not address the upstream `not-owned` check; the safer path is a narrower permission model plus real CLI/TUI proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep secret files from becoming world-readable:** `src/secrets/resolve.ts:299`
  - body: `allowReadableByOthers` allows both group- and world-readable POSIX files in `@openclaw/fs-safe`. Applying it to every file-backed secret means a 0644 secret payload can resolve without `allowInsecurePath`, while a root-owned `/etc` secret can still fail as `not-owned`, so the patch weakens the default secret contract without proving the reported deployment works.
  - confidence: 0.89
