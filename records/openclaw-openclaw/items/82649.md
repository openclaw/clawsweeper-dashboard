## Summary

Keep this PR open, but it should not merge as-is: the reported failure is source-reproducible on current main, while the proposed fix broadens default secret-file permissions to include world-readable payloads and still does not address the current-uid ownership check for root-owned `/etc` files.

## What This Changes

This PR passes `allowReadableByOthers` into file-backed SecretRef reads, appends provider failure causes to gateway unresolved SecretRef messages, adds a POSIX regression test, and updates the changelog.

## Best Possible Solution

Keep strict secret-file defaults unless maintainers approve a narrow shared-system-secret contract, then add focused regression coverage plus redacted real Ubuntu CLI/TUI proof.

## Reproduction Assessment

Yes, from source: current main calls `readSecureFile` for file providers without `allowReadableByOthers`, and the fs-safe contract rejects group/world-readable files unless opted in. I did not run the exact `/etc` cross-uid CLI/TUI setup in this read-only review.

## Solution Assessment

No. Unconditionally passing `allowReadableByOthers` is broader than the reported group-readable system-secret case, accepts world-readable payloads, and does not fix the not-owned `/etc` case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict readable-by-others secret files:** `src/secrets/resolve.ts:299`
  - body: `allowReadableByOthers` in `@openclaw/fs-safe` covers both group- and world-readable files, so this default makes `0644` secret payloads resolve without `allowInsecurePath`.
