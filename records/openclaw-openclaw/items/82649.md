## Summary

Keep this PR open, but do not merge it as-is: the linked bug is concrete and recent, yet the patch broadens the default secret-file permission contract to accept world-readable payloads and may still miss the reported root-owned `/etc/openclaw/secrets.json` case.

## What This Changes

The branch passes `allowReadableByOthers` into file secret-provider reads, appends caught resolver error messages to gateway unresolved SecretRef text, adds a POSIX resolver regression test, and adds a changelog entry.

## Best Possible Solution

Define a narrow, security-reviewed shared-system-secret permission contract that does not accept world-readable secrets by default, then prove CLI/TUI resolution against a real `/etc/openclaw/secrets.json` setup.

## Reproduction Assessment

Yes, from source: current main reads file providers through `readSecureFile()` without `allowReadableByOthers`, and that helper rejects group/world readable files. I did not establish a live cross-uid `/etc` reproduction in this read-only review.

## Solution Assessment

No. The unconditional `allowReadableByOthers` change is broader than the reported group-readable system-secret case and does not handle the dependency's `not-owned` rejection, so the safer fix needs a narrower permission policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep secret files from becoming world-readable:** `src/secrets/resolve.ts:299`
  - body: `allowReadableByOthers` permits both group- and world-readable files in `@openclaw/fs-safe`, while `readSecureFile()` still rejects files owned by another POSIX uid. Applying it to every file secret provider means a 0644 user secret starts resolving without `allowInsecurePath`, but a root-owned `/etc` secret can still fail as `not-owned`.
  - confidence: 0.89
