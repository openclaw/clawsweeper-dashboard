## Summary

Keep open. Current main still has adjacent memory-search timeout/cache/retry behavior, but the requested schema-backed `agents.defaults.memorySearch.query.timeoutMs`, `cacheTtlMs`, and `retry` controls are still absent from the config type, strict schema, resolved runtime config, generated config help, docs, and builtin memory-search query path.

## What This Changes

Add durable, schema-backed per-agent memory-search query reliability controls for timeout, query-result cache TTL, and retry policy.

## Best Possible Solution

Implement the controls at the memory-search config/runtime boundary: add typed and zod-validated `memorySearch.query.timeoutMs`, `cacheTtlMs`, and `retry`; resolve them per agent; apply them to builtin memory search with graceful fallback; update generated schema/help, docs, focused tests, and a user-facing changelog entry.

## Reproduction Assessment

Not applicable as a runtime bug report. The high-confidence verification path is static source inspection: the proposed keys are absent from the strict schema, resolved config, generated help, docs, and memory-search execution path on current main.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution. QMD-only limits and Active Memory plugin-local cache/timeout controls cover adjacent surfaces and should not be treated as substitutes for schema-backed builtin memory-search query controls.
