## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Export the channel broker runtime setter:** `extensions/channel-broker/index.ts:17-18`
  - body: `index.ts` tells the bundled loader to import `setChannelBrokerRuntime` from `./runtime-api.js`, but `runtime-api.ts` never exports that symbol. When the host tries to wire runtime helpers for this bundled channel, the declared export is missing, so the channel-broker runtime injection path cannot load correctly. Export the setter from `runtime-api.ts` and keep the guardrail test aligned.
  - confidence: 0.93
- **[P2] Infer broker Discord DMs as direct targets:** `extensions/channel-broker/src/channel.ts:58-65`
  - body: For `broker:discord:user:...`, `parseBrokerConversationTarget` leaves `conversationId` as `discord:user:...`, so this inference helper sees `discord` as the first segment and falls through to `channel`. Other routing/account-binding callers use `inferTargetChatType`, so they can persist a DM as non-direct even though the send and session-route paths now treat it as direct.
  - confidence: 0.88
- **[P2] Match capability config to the SDK shape:** `extensions/channel-broker/src/config-schema.ts:5-22`
  - body: The strict config schema accepts a flat capability object (`inbound`, `outbound`, `draftPreview`, etc.), while `ChannelBrokerProviderConfig` and the exported SDK type use `BrokerPlatformCapabilities` with nested `delivery`, `live`, `receive`, and `native` fields. A provider following the public SDK shape will be rejected by config validation, and the accepted flat fields will not match the typed capability checks.
  - confidence: 0.9
