## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Export the runtime setter from runtime-api:** `extensions/channel-broker/index.ts:30`
  - body: `defineBundledChannelEntry` will load `setChannelBrokerRuntime` from `./runtime-api.js` during registration, but that file does not export the setter. As soon as the bundled channel is registered with runtime wiring, `loadBundledEntryExportSync` will throw `missing export`, so the new channel cannot be safely enabled until `runtime-api.ts` re-exports the setter and the guard test expects it.
  - confidence: 0.97
- **[P2] Accept the SDK capability shape:** `extensions/channel-broker/src/config-schema.ts:39`
  - body: The public broker type exposes capability groups as `delivery`, `live`, `receive`, and `native`, but the strict config schema only accepts flat keys like `inbound`, `outbound`, and `draftPreview`. Users following the SDK/docs shape would have their channel config rejected before `normalizeCapabilities` can use it, so the schema, docs, and tests need one canonical shape.
  - confidence: 0.93
- **[P2] Verify broker webhooks before post-auth parsing:** `extensions/channel-broker/src/http-routes.ts:80-84`
  - body: The handler reads a post-auth-sized body and parses JSON to discover `event.providerId` before checking the HMAC signature. That lets unauthenticated requests take the larger post-auth parse path; put the provider/account identity in an authenticated route/header lookup, use pre-auth limits, and verify the signature over the raw body before parsing the event.
  - confidence: 0.9
