## Label Justifications

- `P2`: This is a normal-priority Telegram picker bug with a bounded channel/UI blast radius and a clear source-level cause.
- `impact:message-loss`: The Telegram channel surface suppresses model-selection buttons even though the message header says those models are available.

## Summary

Keep open. Current main and the latest release still implement the reported silent drop path: overlong compact model callbacks return `null`, the Telegram keyboard skips those models, and the visible header still counts the full provider model set.

## What This Changes

The issue reports that Telegram's `/model` inline picker hides long model IDs when both full and compact callback data exceed Telegram's 64-byte callback limit.

## Best Possible Solution

Implement an index-based or otherwise opaque Telegram model selection callback that stays under 64 bytes while keeping every listed configured model selectable, with a clear drift/error path if the model list changes before click handling.

## Reproduction Assessment

Yes, source-level. A model whose compact `mdl_sel/{model}` payload is 65 bytes or more makes `buildModelSelectionCallbackData` return `null`, and `buildModelsKeyboard` then skips that row while the caller reports the full model count.

## Solution Assessment

Yes, the issue's index/short-id direction is the best fix. A footer-only note would explain the mismatch but would still leave valid configured models unselectable from the Telegram picker.
