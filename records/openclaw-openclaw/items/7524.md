## Summary

Keep this issue open: current main and the latest release still have only DM-specific session scoping for this path, while the linked implementation PR is open and explicitly closes this issue. The request is valid but remains a feature/config/security-boundary decision, not a stale cleanup close.

## What This Changes

The issue requests a `groupScope` option so selected group or channel conversations can share an agent's main session instead of always using isolated group session keys.

## Best Possible Solution

Land a backwards-compatible, opt-in group/session routing contract with docs, security-audit posture, inbound/outbound parity, and regression coverage, or explicitly make a broader shared-session issue canonical before closing this focused tracker.

## Reproduction Assessment

Yes, at source level. Current main deterministically derives isolated keys for non-direct group/channel peers, and the strict config schemas cannot express a group-to-main option.

## Solution Assessment

Unclear as a final product path. A `groupScope` option is plausible and has an open implementation PR, but maintainers still need to settle the security posture and exact scope before this issue can close.
