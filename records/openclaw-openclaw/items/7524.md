## Summary

Keep open. Current main still has only DM-specific session scoping, while group/channel peers continue to resolve to isolated session keys and the strict config/types/docs do not expose a group-to-main option. The discussion and related items support this as the focused tracker, with open PR #47234 as an implementation candidate rather than a reason to close before merge.

## What This Changes

The issue requests an opt-in groupScope-style setting so selected group chats can consolidate into the agent main session while isolated group sessions remain the default.

## Best Possible Solution

Keep this issue as the focused tracker and resolve it by reviewing/landing #47234 or an equivalent additive design coordinated with #19929, covering routing, strict config schemas/types/generated metadata, public docs, channel group configs, queue/session serialization, reply delivery, and trust-boundary behavior.

## Reproduction Assessment

Yes. A high-confidence current-main check is to inspect buildAgentSessionKey/buildAgentPeerSessionKey or run the focused routing tests: non-direct peers still produce agent:<agentId>:<channel>:<peerKind>:<peerId>, and the existing continuity test asserts channel sessions stay distinct from main DM sessions.

## Solution Assessment

No current-main solution exists yet. The best direction is a narrow, backward-compatible opt-in session routing contract for selected group/channel peers, with #47234 treated as a review candidate and #19929 as broader design context.
