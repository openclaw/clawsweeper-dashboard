## What This Changes

Adds a per-instance bounded LRU query-embedding cache to memory-lancedb, wires it through both embedding implementations, exposes optional `query.embeddingCache` config, and adds unit/integration coverage.

## Best Possible Solution

Keep the cache at the memory-lancedb embeddings boundary, document the new opt-out/tuning settings publicly, and then let maintainers accept the default-on upgrade behavior.

## Reproduction Assessment

Yes for the performance behavior, from source inspection: current main calls the embedding provider on each identical `embed()` call, while the PR body includes real Ollama output showing duplicate calls collapse to one. I did not run a live reproduction in this read-only review.

## Solution Assessment

Yes, this is the right layer: caching inside the per-instance memory-lancedb embeddings owner avoids call-site duplication and keeps memory-core’s persistent chunk cache separate. The remaining improvement is docs/config-contract completion, not a different runtime design.
