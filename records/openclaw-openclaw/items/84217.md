## Summary

Keep open: current main still has the heartbeat fallback gap, and the only linked fix PR was closed unmerged after route-scoping and proof concerns. Adjacent message-tool dedupe work does not cover this distinct heartbeat narration leak.

## What This Changes

The issue asks OpenClaw to stop delivering heartbeat free-text fallback when the same heartbeat turn already completed a generic message-tool send to the heartbeat target.

## Best Possible Solution

Carry route-scoped successful generic message-tool delivery evidence into the heartbeat reply path and suppress fallback text only when that send already served the configured heartbeat target, with focused regressions and live or equivalent proof.

## Reproduction Assessment

Yes, at source level: current main still enables heartbeat message-tool-only mode, exposes both generic message and heartbeat_respond, then only treats heartbeat_respond metadata as owning delivery before sending fallback text. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Partly: the requested suppression direction is right, but the maintainable fix should key off successful route-matched message-tool delivery evidence rather than any generic message tool call in the transcript.
