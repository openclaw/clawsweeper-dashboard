## What This Changes

The issue asks OpenClaw to suppress fallback heartbeat free text when a heartbeat run already delivered visible output through the generic message tool.

## Best Possible Solution

Carry successful, route-matched generic message-tool delivery evidence from the reply runner into heartbeat handling, suppress fallback only for the same heartbeat target, and cover it with focused regression plus live or equivalent Telegram proof.

## Reproduction Assessment

Yes, at source level: current main enables heartbeat message-tool-only mode, exposes both `message` and `heartbeat_respond`, and still sends fallback free text when no `heartbeat_respond` metadata is present. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, with one constraint: metadata-based suppression is the maintainable direction, but it should be based on successful route-matched message-tool delivery rather than any generic message-tool call in the turn.
