## What This Changes

The issue reports that heartbeat runs can send free-form assistant narration to Telegram after a structured generic `message` tool call already delivered the intended heartbeat message.

## Best Possible Solution

Carry verified generic message-tool delivery evidence from the agent runner to heartbeat dispatch, match it against the heartbeat delivery route before suppressing fallback text, and cover both message-tool-only and automatic heartbeat modes with focused regressions.

## Reproduction Assessment

Yes, at source level: current main enables heartbeat message-tool-only mode, exposes both delivery tools, recognizes only `heartbeat_respond` metadata, and then sends remaining free text through the heartbeat fallback path. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, with one constraint: the best fix is the metadata-based suppression shape from the closed PR, but route-scoped to verified provider/target/account/thread delivery rather than blindly suppressing after any generic `message` tool send.
