## Summary

Keep open: current main still has the heartbeat dispatch gap, and the only directly linked fix PR was closed unmerged after review found route-scoping/proof gaps.

## What This Changes

The issue reports that heartbeat runs can deliver stray assistant narration alongside a generic `message` tool send when chatty non-Codex providers emit both in one turn.

## Best Possible Solution

Carry successful generic message-tool delivery evidence into the heartbeat result, route-match it against the heartbeat target before suppressing fallback text, and add focused regression coverage for both matched and mismatched routes.

## Reproduction Assessment

Yes, at source level: current main allows heartbeat message-tool-only runs to expose both generic `message` and `heartbeat_respond`, then only recognizes `heartbeat_respond` metadata before falling back to text delivery. I did not run a live Telegram tick in this read-only review.

## Solution Assessment

Mostly yes: metadata-based suppression is the right shape, but it should key off successful, route-matched message-tool delivery rather than any generic message tool call in the turn.
