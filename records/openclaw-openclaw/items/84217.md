## What This Changes

The issue asks OpenClaw to suppress heartbeat fallback text when a heartbeat turn has already delivered the intended visible response through the message tool.

## Best Possible Solution

Land a route-aware delivery-evidence fix: after a successful generic message tool send to the heartbeat target, mark the reply as already visibly delivered and have runHeartbeatOnce skip only the duplicate fallback text, with focused regression tests and redacted Telegram or Mantis proof.

## Reproduction Assessment

Yes, at source level. Current main still exposes both message and heartbeat_respond tools, only treats heartbeat_respond metadata as owning heartbeat delivery, and sends ordinary fallback text when that metadata is absent.

## Solution Assessment

Yes, with one important constraint: the fix should use successful, route-matched message-tool delivery evidence rather than suppressing fallback text whenever a generic message tool call appears anywhere in the turn.
