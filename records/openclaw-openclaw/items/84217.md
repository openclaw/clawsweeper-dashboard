## Label Justifications

- `P1`: The issue affects the current release and can duplicate Telegram heartbeat messages from real provider runs with noisy or private narration.
- `impact:message-loss`: The report is about duplicated/unwanted channel messages from one heartbeat turn, which fits the message delivery impact class.

## Summary

Keep open: the report matches a current-main heartbeat dispatch gap where message-tool-only heartbeat runs can still send fallback assistant text after a generic `message` tool send, and the related closed issues do not cover this structured-tool-call shape.

## What This Changes

The issue asks OpenClaw to suppress free-text heartbeat fallback delivery when the same heartbeat turn already sent a visible response through the generic `message` tool.

## Best Possible Solution

Teach the heartbeat runner to honor successful message-tool delivery evidence in message-tool-only heartbeat mode, suppressing fallback text only after a real visible tool send and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main enables message-tool-only heartbeat mode, exposes both `message` and `heartbeat_respond`, then only treats `heartbeat_respond` metadata as owning delivery before sending fallback text. A focused unit test can reproduce this without a live Telegram run.

## Solution Assessment

The suggested direction is mostly right, but the safer implementation should key off successful message-tool delivery evidence and target matching rather than blindly suppressing whenever a transcript contains a generic `message` tool call.
