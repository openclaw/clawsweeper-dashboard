## Summary

Keep open: current main still has the source-level heartbeat dispatch gap, and https://github.com/openclaw/openclaw/pull/84273 is an open closing PR for this issue rather than merged implementation. The issue should close only after that PR or an equivalent route-scoped fix lands.

## What This Changes

Heartbeat runs can leak free-text assistant narration to Telegram alongside a generic message-tool send, so the request is to suppress fallback text once structured delivery already owns the response.

## Best Possible Solution

Land a route-scoped implementation that marks/suppresses heartbeat fallback only after a successful message-tool send to the heartbeat target, then close the issue through the linked PR.

## Reproduction Assessment

Yes, at source level. Current main can expose both `message` and `heartbeat_respond`, then only recognizes `heartbeat_respond` metadata before delivering remaining free-text fallback through the heartbeat send path.

## Solution Assessment

Yes, with route scoping. The maintainable fix is to carry successful generic message-tool delivery evidence into the heartbeat runner and suppress fallback text only when that delivery matches the heartbeat route.
