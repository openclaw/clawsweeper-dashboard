## Summary

Keep open: current main and the latest shipped tag still have the heartbeat fallback path that can deliver free text after a generic message-tool send, and the linked fix PR was closed unmerged after proof and route-scoping concerns.

## What This Changes

The issue asks OpenClaw to suppress heartbeat fallback text when a structured message-tool send has already handled the visible heartbeat reply.

## Best Possible Solution

Carry successful generic message-tool delivery evidence into the heartbeat runner, suppress fallback only for the same provider/target/account/thread route, and keep `heartbeat_respond` precedence unchanged.

## Reproduction Assessment

Yes, at source level: current main still allows a generic message-tool send plus free text to leave `heartbeatToolResponse` undefined and then deliver the free-text fallback through `sendDurableMessageBatch`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, with a route-scoping caveat: the maintainable fix is internal delivery evidence from `runReplyAgent` to `runHeartbeatOnce`, but suppression must use the existing provider/target/account/thread matching boundary so unrelated message-tool sends do not drop heartbeat fallback text.
