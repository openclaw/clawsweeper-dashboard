## Summary

Keep this PR open, but it still needs changes before merge. Current main does not implement the loose `media` alias, so the contribution targets a real SDK reply-payload gap; the PR head still mishandles blank `mediaUrl` fallback and is stale against current main.

## What This Changes

The PR updates plugin SDK reply-payload normalization and media helper functions to accept a loose `media` alias and adds regression tests for normalization, delivery wrapping, media resolution, and sendable reply parts.

## Best Possible Solution

Refresh the change onto current main, keep alias support narrowly inside `src/plugin-sdk/reply-payload.ts`, treat blank `mediaUrl` values as missing before alias backfill, preserve current-main reply metadata behavior, and add regression coverage for string and array aliases, blank alias entries, empty `mediaUrl` fallback, normalized delivery, and sendable-part extraction.

## Reproduction Assessment

Yes. Static reproduction is high confidence: on the PR head, `normalizeOutboundReplyPayload({ mediaUrl: "", media: ["https://example.com/a.png"] })` keeps `mediaUrl` as an empty string because `readStringValue` returns empty strings and the `??` fallback does not run.

## Solution Assessment

No. The direction is narrow and appropriate, but the current implementation is not the best fix until it normalizes `mediaUrl` as a non-empty string before alias fallback and is refreshed without dropping newer reply-payload behavior.
