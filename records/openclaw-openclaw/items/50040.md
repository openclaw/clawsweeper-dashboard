## Summary

Keep this open. Current main has partial Telegram draft-stream and safe-send retry improvements, but the durable runtime recovery gap remains for direct interactive final replies that fail during Telegram polling/network recovery windows.

## What This Changes

The issue asks OpenClaw to preserve and recover Telegram outbound replies when polling stalls or transient send failures occur after an assistant response has already been computed.

## Best Possible Solution

Add a maintainer-approved Telegram reply-delivery recovery model that records already-computed final/block reply chunks, retries only failures classified as safe, preserves or surfaces ambiguous failures, and drains eligible work after polling recovery and startup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make a Telegram final reply sendMessage fail inside deliverReplies, and current main rethrows through the direct sendPayload path without creating durable pending/retryable/ambiguous recovery state.

## Solution Assessment

Unclear. The requested direction is right, but the best implementation needs maintainer judgment around duplicate-safe classification and whether recovery belongs in shared outbound delivery or a Telegram-specific reply outbox.
