## Summary

Keep this issue open. Current main has Telegram polling recovery, strict safe-send retry for wrapped pre-connect failures, and a shared outbound queue, but interactive Telegram final replies still go through direct `deliverReplies` delivery and are not persisted as pending/retryable/ambiguous work that can drain after polling recovery.

## What This Changes

The issue asks for runtime Telegram outbound delivery recovery so polling stalls or restarts do not silently lose already-computed final reply chunks.

## Best Possible Solution

Add a maintainer-approved Telegram reply-delivery recovery design that records already-computed final/block reply chunks, retries only failures classified as safe, preserves or surfaces ambiguous failures, and hooks eligible drain work into polling restart as well as startup where the shared queue contract fits.

## Reproduction Assessment

Yes. The issue includes production timelines, and current-main code gives a static reproduction path: make a Telegram final reply `sendMessage` fail in `deliverReplies`, and the direct path rethrows without durable pending/retryable/ambiguous state.

## Solution Assessment

Unclear. The requested direction is correct, but the exact fix needs maintainer design judgment because non-idempotent Telegram sends cannot be blindly retried without duplicate-message risk.
