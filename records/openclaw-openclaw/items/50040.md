## Summary

Keep open: current main has important partial fixes for normal final replies and update acknowledgment, but Telegram polling recovery still does not drain queued outbound delivery, and streamed overflow follow-up chunks still use a direct send path that can fail without durable recovery.

## What This Changes

Ensure Telegram outbound replies are not silently lost during polling stalls by retrying safe failures and preserving ambiguous failures without requiring a gateway restart.

## Best Possible Solution

Keep the partial durable-final and delayed-ack work, then add a maintainer-approved Telegram recovery design that covers queued polling-restart drains and streamed follow-up chunks while preserving ambiguous failures instead of blindly replaying them.

## Reproduction Assessment

Yes, at source level. Make a Telegram streamed final follow-up chunk or queued final send fail during polling recovery: current main either sends the follow-up directly without durable state or leaves queued work without a Telegram polling-restart drain.

## Solution Assessment

Unclear. The current durable queue direction is the right foundation, but the remaining fix needs maintainer judgment for duplicate-safe classification, polling-restart drain timing, and streamed overflow chunk semantics.
