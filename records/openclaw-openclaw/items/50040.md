## Summary

Keep open. Current main has Telegram polling recovery, bounded pre-connect send retry, and the shared outbound write-ahead queue, but interactive Telegram final replies still call `deliverReplies` directly and rethrow failed sends without creating durable pending/retryable/ambiguous work that can drain after polling recovery.

## What This Changes

The issue requests durable runtime recovery for Telegram outbound `sendMessage` failures during polling stall or restart windows, including safe retry classification and non-lossy handling of ambiguous deliveries.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and triggers eligible recovery during polling restart as well as startup where the shared queue contract fits.

## Reproduction Assessment

Yes. The issue includes production timelines, and current-main code gives a static reproduction path: make a Telegram final reply `sendMessage` fail in `deliverReplies`, and the direct path rethrows without durable pending/retryable/ambiguous state.

## Solution Assessment

Unclear. The requested direction is right, but the exact implementation needs maintainer design judgment because Telegram sends are non-idempotent and the wrong recovery model can trade message loss for duplicate visible messages.

## Risks / Open Questions

- A naive runtime retry can duplicate Telegram messages when the request reached Telegram but the client saw a post-send network error.
- Maintainers need to choose whether this belongs in a Telegram-owned reply outbox or in the shared outbound delivery queue with explicit non-idempotent send states.
- The current generic queue has retry/backoff primitives, but it does not yet model Telegram-specific safe, ambiguous, and permanent final-reply outcomes.
