## Summary

Keep open: current main still protects only narrow safe-send and queued outbound paths, while interactive Telegram final replies can still fail through the direct deliverReplies path without durable pending, retryable, or ambiguous recovery.

## What This Changes

The issue asks for runtime recovery of Telegram outbound reply delivery during polling stalls and transient sendMessage failures.

## Best Possible Solution

Add a maintainer-approved Telegram reply-delivery recovery design that records already-computed final reply chunks, retries only classified safe failures, and preserves or surfaces ambiguous failures without blind duplication.

## Reproduction Assessment

Yes. The issue has production timelines, and current-main source gives a static path: make a Telegram final reply sendMessage fail in deliverReplies and the direct path rethrows without durable pending, retryable, or ambiguous state.

## Solution Assessment

Unclear. The durable recovery direction is likely right, but the exact state model needs maintainer judgment because non-idempotent Telegram sends cannot be retried blindly.
