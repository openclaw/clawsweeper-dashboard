## Summary

Keep this issue open: current main has partial Telegram retry and stream-preview safeguards, but direct interactive final replies still are not recorded as pending/retryable/ambiguous work that can recover after polling stalls.

## What This Changes

Request robust Telegram outbound delivery recovery so polling stalls or restarts do not silently lose already-computed replies.

## Best Possible Solution

Add a maintainer-approved Telegram reply-delivery recovery design that records already-computed final/block reply chunks, retries only classified safe failures, preserves or surfaces ambiguous failures, and hooks eligible drain work into polling recovery and startup where the shared queue contract fits.

## Reproduction Assessment

Yes. The issue has concrete production timelines, and current-main source gives a static reproduction path: make a Telegram final reply sendMessage fail in deliverReplies and the direct path rethrows without durable pending, retryable, or ambiguous state.

## Solution Assessment

Unclear. The durable recovery direction is likely right, but the exact implementation needs maintainer design judgment because non-idempotent Telegram sends cannot be retried blindly without duplicate-message risk.
