## Summary

The one-line change correctly closes the current-main producer/consumer gap: startup catch-up creates a running cron task without marking its job active, although authoritative reconciliation uses that marker to decide whether the task is still live. No patch defect was found, but the external-PR merge gate remains unmet because the supplied evidence is test-harness output rather than after-fix behavior from a real Gateway restart.

## What This Changes

The PR marks a missed cron job active when its startup catch-up execution begins.

## Best Possible Solution

Keep this producer-side marker placement, add a focused startup catch-up test that blocks execution and verifies the marker is true in flight and false afterward, then provide redacted real Gateway terminal or log proof covering reconciliation beyond the grace period.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: startup catch-up omits the active marker consumed by authoritative task reconciliation, and the linked issue supplies an exact blocked-run test design that fails before the change.

## Solution Assessment

Yes for the implementation direction: marking the producer path is narrower and safer than weakening reconciliation. A direct startup catch-up regression test should accompany it, and real runtime proof is still required before merge.
