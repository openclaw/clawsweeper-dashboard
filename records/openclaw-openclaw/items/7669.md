## Summary

Current main has clarified `dev` as a Git checkout of moving `main`, not an npm dist-tag, and the live registry still has no `openclaw@dev`. The npm snapshot request therefore remains a valid but optional release feature that needs explicit maintainer product and supply-chain approval rather than conservative cleanup.

## What This Changes

Add a supported npm `dev` distribution channel for moving main builds, including publish automation, registry verification, and aligned documentation.

## Best Possible Solution

Keep the source-based `dev` channel as the current contract unless maintainers approve a complete snapshot-release design that publishes uniquely versioned, provenance-backed main builds under `dev` and verifies the registry result.

## Reproduction Assessment

Yes. Current main has no npm `dev` publish option, and `npm view openclaw@dev version --json` returns E404 on June 12, 2026; the existing source-based `openclaw update --channel dev` path is separately documented and available.

## Solution Assessment

No to merely adding a `dev` choice or dist-tag mutation. The narrowest maintainable npm solution must define snapshot versioning, trusted publish triggers, provenance, verification, updater behavior, and lifecycle policy together; otherwise the current Git-based channel is safer.
