## Summary

Keep open. Current main and the latest shipped tag still lack a first-class npm `dev` dist-tag publish path, the live npm registry has no `dev` tag, and the remaining decision is release/security policy rather than conservative cleanup.

## What This Changes

Request to re-enable an npm `dev` dist-tag so moving-main OpenClaw builds can be installed as `openclaw@dev`.

## Best Possible Solution

Maintainers should either design and implement a first-class npm dev snapshot channel with explicit versioning, trigger policy, credential/provenance boundaries, tests, and registry verification, or intentionally retire npm dev snapshots and document `dev` as git/source-only.

## Reproduction Assessment

Yes for the source and registry gap: current main's workflows, release-plan types, publish helper, and live npm dist-tags do not provide `openclaw@dev`. This is a feature/release-policy request rather than a broken existing bug contract.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired end state, but maintainers need to approve snapshot semantics and credential/provenance boundaries before implementation; the safer alternative is to document `dev` as git/source-only if npm snapshots are not intended.
