## Summary

Keep open: current main still documents an npm `dev` channel but the source release workflow, publish helper, release-plan contract, tests, and live npm registry do not provide a first-class `dev` dist-tag publishing path. This remains a maintainer release-channel and supply-chain policy decision, not conservative cleanup that can close.

## What This Changes

The issue requests re-enabling the npm `dev` dist-tag for moving-main OpenClaw releases and adding verification that the documented development channel is published correctly.

## Best Possible Solution

Maintain one coherent preview-channel contract: either retire npm `dev` in docs and keep git dev plus npm alpha, or implement `dev` as an approved package channel with versioning, permissions, provenance, verification, updater behavior, and docs aligned.

## Reproduction Assessment

Yes for the release-surface mismatch: current source lacks a `dev` npm publish path, and the live npm registry has no `dev` dist-tag. I did not run a mutating publish test, but the workflow/helper/source and registry checks reproduce the missing channel with high confidence.

## Solution Assessment

Unclear. Directly re-enabling npm `dev` is not proven to be the best fix until maintainers decide whether current npm `alpha` plus git dev supersede, alias, or intentionally differ from the requested moving-main npm dev channel.
