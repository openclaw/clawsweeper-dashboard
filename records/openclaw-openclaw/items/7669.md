## Summary

Keep open. Current main still has no first-class npm `dev` dist-tag publish path, and the remaining decision is release/security policy rather than conservative cleanup.

## What This Changes

Request to re-enable an npm `dev` dist-tag for moving `main` builds and verify that the documented development channel is publishable.

## Best Possible Solution

Maintainers should either design and implement a first-class npm dev snapshot channel with explicit versioning, trigger policy, credential/provenance boundaries, tests, and registry verification, or intentionally retire npm dev snapshots and document `dev` as git-only/source-only.

## Reproduction Assessment

Yes for the source and registry gap: current main lacks a `dev` publish path in workflow choices, release-plan types, publish helpers, and tests, and the live npm registry has no `openclaw@dev` dist-tag. This is a feature/release-policy request rather than a broken existing bug contract.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired end state, but the current release architecture is tag-based and OIDC-focused with token dist-tag mutation kept private, so maintainers should approve snapshot semantics and credential/provenance boundaries first.
