## Summary

Keep open. Current main, the latest shipped release tag, and the live npm registry still do not provide a first-class npm `dev` dist-tag publish path, and enabling one needs release/security product judgment rather than conservative cleanup.

## What This Changes

The issue requests restoring an npm `dev` dist-tag channel for moving main builds and verifying that the documented development channel is published correctly.

## Best Possible Solution

Maintainers should either design a first-class npm dev snapshot channel with explicit versioning, trigger policy, credential/provenance boundaries, tests, and registry verification, or intentionally document `dev` as git/source-only.

## Reproduction Assessment

Yes. Source inspection plus live registry checks reproduce the gap: current workflows, release-plan types, publish helper, tests, latest release tag, and npm registry do not provide `openclaw@dev`.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired end state, but maintainers need to approve snapshot semantics and credential/provenance boundaries first; documenting `dev` as git/source-only is the safer alternative if npm snapshots are not intended.
