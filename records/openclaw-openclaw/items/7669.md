## Summary

Keep open: current main and the latest release still lack an npm `dev` publish path while user-facing docs still mention an optional `dev` dist-tag. This remains a release-channel and supply-chain policy decision, not conservative cleanup that can close.

## What This Changes

Issue requests re-enabling an npm `dev` dist-tag for moving main-branch OpenClaw builds and adding verification/docs alignment for that channel.

## Best Possible Solution

Maintain one coherent preview-channel contract: either retire npm `dev` in public docs and keep git dev plus npm alpha, or implement `dev` as an approved package channel with versioning, permissions, provenance, verification, updater behavior, and docs aligned.

## Reproduction Assessment

Yes for the release-surface mismatch: current source lacks a `dev` npm publish path, latest release source lacks it too, and the live npm registry has no `dev` dist-tag. I did not run a mutating publish test, but workflow/helper/source and registry checks reproduce the missing channel with high confidence.

## Solution Assessment

Unclear. Directly re-enabling npm `dev` is not proven to be the best fix until maintainers decide whether current npm `alpha` plus git-backed dev supersede, alias, or intentionally differ from the requested moving-main npm dev channel.
