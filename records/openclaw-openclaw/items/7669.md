## Summary

Current main and the live npm registry still have no moving-main `dev` publication path; tagged `alpha` prereleases were added instead, while public docs continue to mention an optional npm `dev` tag, so the issue remains necessary pending an explicit release and security policy decision.

## What This Changes

Requests restoring an npm `dev` dist-tag that tracks moving `main`, with publishing verification and accurate channel documentation.

## Best Possible Solution

Adopt one coherent channel contract: either implement securely triggered, uniquely versioned npm `dev` snapshots with provenance and post-publish verification, or remove the npm `dev` promise everywhere and document `dev` as git-only while npm prereleases use `alpha` and `beta`.

## Reproduction Assessment

Yes. Current main lacks a `dev` option in the npm workflow and publish contract, and the live registry exposes only `alpha`, `beta`, and `latest`.

## Solution Assessment

No, re-enabling a dist-tag alone is not the best solution; maintainers first need a complete snapshot versioning, trigger, provenance, verification, retention, and channel-semantics design.
