## Summary

Keep open. Current main, the latest shipped release tag, and the live npm registry still lack a first-class npm `dev` dist-tag publish path, and the remaining work needs release/security product judgment rather than conservative cleanup.

## What This Changes

Requests re-enabling an npm `dev` dist-tag for moving main branch builds, plus CI/CD verification and docs alignment.

## Best Possible Solution

Maintainers should either design a first-class npm dev snapshot channel with explicit versioning, trigger policy, credential/provenance boundaries, tests, and registry verification, or intentionally document `dev` as git/source-only.

## Reproduction Assessment

Yes. Source inspection plus a live registry fetch reproduce the gap: current workflows, release-plan types, publish helper, and registry dist-tags do not provide `openclaw@dev`.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired end state, but maintainers need to approve snapshot semantics and credential/provenance boundaries first; the safer alternative is to document `dev` as git/source-only if npm snapshots are not intended.
