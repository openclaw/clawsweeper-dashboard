Keep open. Current main, the latest shipped tag, and the live npm registry still do not provide a first-class npm `dev` dist-tag path, and the remaining work is release/security policy rather than conservative cleanup.

## What This Changes

Request to re-enable npm `dev` dist-tag publishing for moving-main OpenClaw builds and verify CI/CD, registry state, and docs.

## Best Possible Solution

Maintainers should either design a first-class npm dev snapshot channel with explicit versioning, trigger policy, credential/provenance boundaries, tests, and registry verification, or intentionally document `dev` as git/source-only.

## Reproduction Assessment

Yes for the source and registry gap: current main's workflows, release-plan types, publish helper, and live npm dist-tags do not provide `openclaw@dev`. This is a feature/release-policy request rather than a broken existing bug contract.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired end state, but maintainers need to approve snapshot semantics and credential/provenance boundaries before implementation; the safer alternative is to document `dev` as git/source-only if npm snapshots are not intended.
