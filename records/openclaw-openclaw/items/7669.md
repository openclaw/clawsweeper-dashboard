## Summary

Keep open. Current main still has no supported npm `dev` dist-tag publish path: the public release workflows, publish helper, release-plan contract, and release tests are stable/alpha/beta/latest-only, while the live npm registry still exposes only `latest` and `beta`.

## What This Changes

The issue requests re-enabling npm `dev` dist-tag publishing for moving `main` builds and adding registry verification plus matching documentation.

## Best Possible Solution

Maintainers should either design and implement a first-class npm dev snapshot channel with explicit versioning, trusted-publish or token boundaries, tests, and registry verification, or intentionally retire npm dev snapshots and document `dev` as source/git-only.

## Reproduction Assessment

Yes for the source and registry gap: current main lacks a `dev` publish path in workflow choices, release-plan types, publish helpers, and tests, and the live npm registry has no `openclaw@dev` dist-tag. This is a feature/release-policy gap rather than a broken existing bug contract.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired result, but current release architecture intentionally keeps real publishes tag-based and OIDC-focused, so maintainers should first approve snapshot semantics and the credential/provenance boundary.
