## Summary

Keep open. Current main and the latest shipped release still lack a supported npm `dev` dist-tag publish path; the docs only describe that tag as optional, `openclaw update --channel dev` uses a git checkout, and moving-main npm publishing needs release/security owner approval before implementation.

## What This Changes

Request to re-enable npm `dev` dist-tag publishing for moving `main` builds and verify the registry, CI/CD path, and documentation.

## Best Possible Solution

Maintainers should either design and implement a first-class npm dev snapshot channel with explicit versioning, trigger policy, credential/provenance boundaries, tests, and registry verification, or intentionally retire npm dev snapshots and document `dev` as git-only/source-only.

## Reproduction Assessment

Yes for the source and registry gap: current main lacks a `dev` publish path in workflow choices, release-plan types, publish helpers, and tests, and the live npm registry has no `openclaw@dev` dist-tag. This is a feature/release-policy request rather than a broken existing bug contract.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired result, but current release architecture is tag-based and OIDC-focused with token dist-tag mutation kept private, so maintainers should first approve snapshot semantics and the credential/provenance boundary.
