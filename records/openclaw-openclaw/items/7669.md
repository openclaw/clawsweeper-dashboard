## Summary

Keep open. Current main still has no supported npm `dev` dist-tag publish path: docs still mention it as optional, release workflows and helpers only model alpha/beta/latest release-tag publishing, `dev` updates are git-based, and the live npm registry has no `dev` tag.

## What This Changes

The issue requests re-enabling an npm `dev` dist-tag for moving `main` builds, plus CI/CD verification and matching documentation.

## Best Possible Solution

Maintainers should either design and implement a first-class npm dev snapshot channel with explicit versioning, credential/provenance boundaries, tests, and registry verification, or intentionally retire npm dev snapshots and document `dev` as source/git-only.

## Reproduction Assessment

Yes for the source and registry gap: current main lacks a `dev` publish path in workflow choices, release-plan types, publish helpers, and tests, and the live npm registry has no `openclaw@dev` dist-tag. This is a feature/release-policy gap rather than a broken existing bug contract.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the desired outcome, but the current release architecture deliberately keeps real publishes tag-based and OIDC-focused, so maintainers should first approve snapshot semantics and the credential/provenance boundary.
