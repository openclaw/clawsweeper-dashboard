## Label Justifications

## Summary

Current `main` confirms the central request remains unresolved: `dev` is still documented as an optional npm dist-tag, but the live registry and public release workflow expose `alpha`, `beta`, and `latest` only. The newer `alpha` lane is a versioned prerelease channel, not an established moving-head replacement for `dev`, so this needs an explicit maintainer product and security decision rather than cleanup closure.

## What This Changes

Restore an npm `dev` dist-tag that publishes moving `main` builds and add release verification and accurate channel documentation.

## Best Possible Solution

Choose one explicit contract: either implement a securely versioned moving-main npm snapshot lane and align workflow, release-plan helper, tests, registry verification, installer/updater behavior, and docs, or retire npm `dev` and document `dev` as git-only while explaining how `alpha` differs.

## Reproduction Assessment

Yes. On current `main`, the release workflow and helper have no `dev` path, and the live npm registry has no `dev` dist-tag despite the public docs still naming it.

## Solution Assessment

Unclear pending maintainer policy. Adding `dev` mechanically is not sufficient; the best solution must first define whether `alpha` supersedes it and how moving-main versions and secure publication should work.
