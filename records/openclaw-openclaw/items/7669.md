## Summary

Keep open. Current main still does not have a first-class npm `dev` dist-tag publish path: the public npm release workflow only offers `beta` and `latest`, real publishes still require release tags, the publish helper and release-plan types only model stable/beta, tests cover stable/beta only, and the live npm registry currently exposes only `latest` and `beta` dist-tags. The existing maintainer/Codex discussion reached the same conclusion, and the later release-infra changes did not add `dev`.

## What This Changes

Re-enable an npm `dev` dist-tag for moving `main` builds, verify `openclaw@dev` publication, and align the release-channel documentation.

## Best Possible Solution

Keep the issue open until maintainers either implement a full npm `dev` snapshot channel for moving `main` builds, with workflow support, versioning rules, release-plan and publish-helper support, tests, registry verification, and docs; or explicitly retire npm `dev` snapshots and make the docs/update behavior consistently git-only for `dev`.

## Reproduction Assessment

Yes. The current-main source inspection and live registry JSON provide a high-confidence reproduction path: the release workflow cannot select `dev`, the publish planner cannot emit `dev`, and the registry has no `dev` dist-tag.

## Solution Assessment

Unclear, because there is no proposed patch and the requested direction needs maintainer release-policy design. A safe end state is either a fully designed `dev` snapshot publishing lane or a deliberate docs/runtime cleanup that stops advertising npm `dev`.
