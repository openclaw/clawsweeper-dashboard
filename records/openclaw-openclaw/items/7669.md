## Summary

Keep this open. Current main still lacks a first-class npm `dev` dist-tag publish path: the public release workflows accept `alpha`, `beta`, and `latest` only, the real publish path is release-tag based, the publish helper/release-plan contract cannot emit `dev`, docs still describe optional npm `dev` snapshots, and the live npm registry has no `dev` dist-tag.

## What This Changes

The issue requests re-enabling npm `dev` dist-tag publishing for moving main builds and adding matching verification and documentation.

## Best Possible Solution

Maintainership should either design and implement a first-class npm `dev` snapshot channel with explicit versioning, triggers, provenance, verification, and docs, or retire npm `dev` snapshots and document `dev` as git/source-only.

## Reproduction Assessment

Yes for the source-level gap: current main exposes only `alpha`, `beta`, and `latest` in release workflows/helpers, and the live npm registry has no `dev` dist-tag. This is a feature/release-policy gap rather than a broken existing command contract.

## Solution Assessment

Unclear. Re-enabling npm `dev` may be the right product direction, but the maintainable solution needs an owner decision on whether moving-main npm snapshots should exist at all and how to publish them safely.
