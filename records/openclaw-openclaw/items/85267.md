## Summary

Keep open: current main still has a QQBot outbound response watchdog capped at 300s, and that channel-level cap can undercut the reporter's configured 1800s Ollama/provider timeout before a long local-model reply is delivered.

## What This Changes

The issue reports that long Ollama qwen3.5:27b replies time out through QQBot on OpenClaw 5.19/5.20 despite an 1800s provider timeout, while older builds and direct Ollama usage complete.

## Best Possible Solution

Make QQBot's outbound response watchdog derive from the active turn timeout budget or move that wait policy into the shared channel-turn runtime, then cover the slow-first-response path with a focused dispatcher regression test.

## Reproduction Assessment

Yes, at source level: a QQBot inbound turn with no block response before 300,000ms hits the hardcoded response timeout even when the configured provider timeout is longer. I did not run a live Ollama/Qwen/QQBot reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable direction is to align QQBot's response wait with the existing active turn/provider timeout contract, not to add a separate QQBot timeout setting or change Ollama transport behavior.
