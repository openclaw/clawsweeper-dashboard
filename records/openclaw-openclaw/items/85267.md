## Summary

Keep open: current main still has QQBot’s outbound response watchdog fixed at 300,000ms, while the configured provider timeout path already supports longer slow-local-model waits. The linked fix PR https://github.com/openclaw/openclaw/pull/85271 is open and references this issue with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to stop timing out QQBot replies for long Ollama/qwen local model turns when provider and agent `timeoutSeconds` are configured longer.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/85271 or an equivalent QQBot watchdog fix that derives the wait budget from existing timeout config while preserving the default guard, then close this issue after release or main-branch proof.

## Reproduction Assessment

Yes, at source level: a QQBot inbound turn with no block response before 300,000ms can hit the hardcoded response timeout even when the configured provider timeout is 1,800s. I did not run a live QQBot/Ollama/qwen reproduction in this read-only review.

## Solution Assessment

Yes: aligning QQBot’s outbound watchdog with existing agent/provider timeout settings is the narrow maintainable fix and avoids adding a competing QQBot-only timeout knob.
