## Summary

Keep open: current main still has a QQBot outbound response watchdog hardcoded to 300s, while the Ollama/provider path already honors longer provider timeouts, so this channel route can still undercut the reporter's 1800s model timeout.

## What This Changes

The issue reports slow QQBot replies through local Ollama/Qwen timing out after upgrade even with `models.providers.ollama.timeoutSeconds` set to 1800.

## Best Possible Solution

Make QQBot's outbound response watchdog derive from the existing agent/run timeout ceiling, and avoid letting a plugin-local 300s cap undercut longer configured local-model timeouts.

## Reproduction Assessment

Yes, at source level: a QQBot inbound turn with no block response before 300,000ms hits the hardcoded response timeout even when the configured model/provider timeout is longer. I did not run a live Ollama/Qwen/QQBot reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to align QQBot's wait budget with existing agent/provider timeout settings instead of adding a competing timeout policy or changing Ollama transport behavior.
