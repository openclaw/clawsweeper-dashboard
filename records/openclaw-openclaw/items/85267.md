## Summary

Keep open: current main and the latest release still have QQBot’s outbound response watchdog hardcoded to 300s, while the linked fix PR is open and not merged. The source-level failure still matches the reporter’s OpenClaw -> QQBot -> Ollama route, so the issue should stay open until the fix PR lands or is replaced.

## What This Changes

The issue reports a regression where long Ollama local-model replies through the QQBot channel time out around five minutes despite `models.providers.ollama.timeoutSeconds` being set much higher.

## Best Possible Solution

Keep this issue open and land or replace https://github.com/openclaw/openclaw/pull/85271 so QQBot’s outbound watchdog follows the existing agent/provider timeout ceiling instead of a separate 300s cap.

## Reproduction Assessment

Yes, at source level: a QQBot turn with no delivered response before 300,000ms races against the hardcoded watchdog even when `models.providers.ollama.timeoutSeconds` is 1,800s. I did not run a live QQBot/Ollama reproduction in this read-only review.

## Solution Assessment

Yes: deriving the QQBot watchdog from existing timeout configuration is the narrow maintainable fix and avoids adding a competing QQBot-only timeout knob. The current open PR follows that direction, but the issue should remain open until it merges.
