## Summary

Keep open: current main still has a QQBot 300s outbound response watchdog that can undercut the reporter's 1800s Ollama provider timeout, and the linked unmerged fix PR is the right path to review before closing this issue.

## What This Changes

The issue reports a regression where long Ollama Qwen responses time out through the QQBot route despite longer configured provider timeouts and working direct Ollama calls.

## Best Possible Solution

Review and land, or replace, the linked QQBot fix so the outbound response watchdog derives from existing agent/provider timeout ceilings without adding a separate QQBot-only timeout setting.

## Reproduction Assessment

Yes, at source level: a QQBot turn that takes longer than 300,000ms to produce its first response can hit the channel watchdog even when models.providers.<id>.timeoutSeconds is longer. I did not run a live Ollama/Qwen/QQBot reproduction in this read-only review.

## Solution Assessment

Yes: aligning QQBot's response wait budget with existing configured run/provider timeouts is the narrow maintainable fix. The linked open PR is the current implementation candidate, so this issue should close only after that fix lands or another equivalent fix replaces it.
