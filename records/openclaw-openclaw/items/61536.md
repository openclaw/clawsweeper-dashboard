## Summary

Keep open. Current main still matches the reported native Discord voice output-contract gap: voice STT can reach the agent as normal Discord ingress, the `tts` tool is only hidden for provider `voice`, and Discord VC playback still depends on a non-empty returned text payload. The maintainer analysis and second-user confirmation in the issue discussion remain consistent with the inspected source.

## What This Changes

The issue requests that native Discord `/vc join` voice turns produce a speakable reply for playback instead of allowing a `tts` tool call plus `NO_REPLY` to leave the voice channel silent.

## Best Possible Solution

Add an explicit native Discord VC spoken-output contract that preserves Discord session/routing metadata but runs voice-channel transcript turns under a voice-safe output/tool policy, so `tts` is hidden and the agent must return speakable text for Discord VC synthesis and playback.

## Reproduction Assessment

Yes. The issue gives concrete `/vc join` setup steps, and the current source provides a high-confidence contract-level reproduction path: successful STT enters the agent as `discord`, can expose `tts`, and playback is skipped when no text payload returns.

## Solution Assessment

Not applicable for a proposed patch, but the best fix direction is clear. A voice-specific policy and prompt contract is safer than special-casing final `NO_REPLY`, because it prevents the tool-only `tts` turn shape before Discord VC reaches the empty-payload playback branch.
