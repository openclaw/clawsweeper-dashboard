## Summary

Keep open. Current main still has the Discord voice output-contract gap described in the report: native Discord VC sends the transcribed turn as Discord ingress, exposes the normal Discord tool policy, and only synthesizes/playbacks audio after a non-empty text payload comes back. The maintainer deep-review comment and another user confirmation remain consistent with the current source.

## What This Changes

Native Discord `/vc join` should produce an audible voice-channel reply after successful STT instead of allowing a `tts` tool call plus `NO_REPLY` or empty final payload to result in no playback.

## Best Possible Solution

Add an explicit native Discord VC spoken-output contract: preserve Discord session/routing metadata, but run voice-channel transcript turns under a voice-safe output/tool policy and prompt contract so `tts` is hidden and the agent must return speakable text for `processDiscordVoiceSegment` to synthesize and play.

## Reproduction Assessment

Yes. The issue gives concrete `/vc join` setup steps, and the current source provides a high-confidence contract-level reproduction path: successful STT can enter the agent as `discord`, expose `tts`, and then skip playback when no text payload is returned.

## Solution Assessment

Not applicable for a proposed patch, but the best fix direction is clear. A voice-specific policy/contract is safer than special-casing final `NO_REPLY`, because it prevents the tool-only `tts` turn shape before Discord VC reaches the empty-payload playback branch.
