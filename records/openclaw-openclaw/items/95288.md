## Label Justifications

- `P1`: The report describes supervised gateways entering restart loops with channel delivery failures and session recovery churn on real deployments.
- `impact:crash-loop`: The central symptom is a gateway supervisor restart loop that can exhaust systemd start limits.
- `impact:message-loss`: The affected Telegram/WhatsApp channel turns can fail or be replayed during recovery instead of delivering normally.
- `impact:session-state`: The restart-recovery loop is driven by persisted session fields such as `abortedLastRun` and `restartRecoveryRuns`.

## Summary

Keep open. Current main has useful partial safeguards for Telegram startup, channel health-monitor restart caps, and restart-recovery lifecycle cleanup, but `src/agents/main-session-restart-recovery.ts` still auto-resumes every `running` + `abortedLastRun` main session without a persistent cross-boot retry/quarantine gate; the narrower restart-recovery work is now represented by https://github.com/openclaw/openclaw/issues/95750 and open candidate PRs https://github.com/openclaw/openclaw/pull/96307 and https://github.com/openclaw/openclaw/pull/96083.

## What This Changes

The issue asks OpenClaw to prevent degraded Telegram/WhatsApp channel recovery and restart-recovery paths from starving the gateway event loop and putting supervised gateways into repeated restart loops.

## Best Possible Solution

Land one maintainer-approved restart-recovery budget/quarantine fix, then keep the channel no-settlement or channel-health follow-up tracked separately if gateways can still wedge under degraded Telegram/WhatsApp load.

## Reproduction Assessment

Unclear for the full WSL2 Telegram+WhatsApp event-loop starvation loop; I did not run a live multi-channel repro. The restart-recovery subproblem is source-reproducible because current main still resumes `running` + `abortedLastRun` entries without a persisted cross-boot retry budget.

## Solution Assessment

Unclear for the full issue, but closing as implemented would not be the best solution. The best narrow path is to review and land one persistent restart-recovery budget/quarantine fix, while treating the channel-turn wall-clock deadline in https://github.com/openclaw/openclaw/issues/95630 as a separate product decision.
