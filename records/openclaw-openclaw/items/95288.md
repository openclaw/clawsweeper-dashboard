## Summary

Keep open. Current main has partial Telegram, WhatsApp, channel-health, and restart-recovery safeguards, and https://github.com/openclaw/openclaw/issues/95750 now tracks the focused main-session retry-budget slice, but no single current-main or shipped fix fully resolves this broader Telegram/WhatsApp gateway starvation and supervisor restart-loop report.

## What This Changes

The issue reports that degraded Telegram and WhatsApp recovery on WSL2 can saturate the gateway event loop, trigger supervisor restarts, and leave the gateway in a restart loop.

## Best Possible Solution

Keep this as the broad production incident until maintainers either close it after the focused recovery and trigger-layer issues land, or split the remaining channel-specific work into narrower canonical items.

## Reproduction Assessment

Yes for part of the issue: current source shows main-session restart recovery can reselect running aborted rows across fresh processes without a persisted retry ceiling. No full live WSL2 Telegram/WhatsApp repro was run in this read-only review.

## Solution Assessment

Unclear for the whole issue. A persisted restart-recovery budget is a clear survivability fix, but channel-turn deadlines, channel restart policy, and cron restart behavior need maintainer product judgment to avoid config/default churn or one-sided fixes.
