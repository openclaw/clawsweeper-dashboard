## Summary

Keep open. The report is fresh, credible, and not solved by current main or the latest release: the gateway still restarts unhealthy channel accounts on the health-monitor cadence, WhatsApp still has bounded but repeated setup/auth retry loops, and the related event-loop restart-cascade fix was closed unmerged after maintainer feedback that the design was too broad.

## What This Changes

The issue reports that degraded Telegram and WhatsApp channel recovery on WSL2 can saturate the gateway event loop, lose channel delivery, and trigger supervisor restart loops.

## Best Possible Solution

Keep this open and use it with https://github.com/openclaw/openclaw/issues/89785 as live evidence for a bounded gateway channel restart/startup-progress design that is proven on the actual Telegram/WhatsApp supervisor path.

## Reproduction Assessment

No high-confidence live current-main reproduction is available. Source inspection shows credible current paths for the reported health-monitor restarts, Telegram fallback retries, WhatsApp setup/auth retries, and stop-timeout recovery state, but the full WSL2 supervisor loop was not reproduced here.

## Solution Assessment

Unclear. Disabling both channels is only a workaround; the best fix likely belongs at the gateway restart scheduling/startup-progress boundary with targeted Telegram or WhatsApp owner fixes only after live proof identifies the pressure source.
