## Summary

Keep open: current main and the latest release still disable only Gemini async batch mode, while inline memory indexing groups chunks and calls Gemini's synchronous batch endpoint. The report is source-reproducible, but the fix needs maintainer direction because it likely changes public memory/provider config or retry policy.

## What This Changes

The issue asks for a low-throughput Gemini memory embedding mode so disabled remote batch plus low non-batch concurrency can avoid low-quota 429 failures.

## Best Possible Solution

Design one canonical low-throughput inline embedding policy for remote memory providers, with Gemini regression coverage and docs, while preserving existing fast defaults unless maintainers intentionally change them.

## Reproduction Assessment

Yes at source level: current main reaches Gemini batchEmbedContents even when remote.batch.enabled=false. No live low-quota Gemini 429 reproduction was run, so the quota failure itself still needs live confirmation.

## Solution Assessment

Unclear: single embedContent mode, configurable inline batch size/delay, and adaptive 429 splitting are all plausible. The best fix needs a maintainer-chosen provider/config policy rather than a narrow automatic patch.
