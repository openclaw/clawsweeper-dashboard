## Summary

Keep open: current main and the latest checked release still bypass the async remote batch hook when `remote.batch.enabled=false`, but Gemini inline indexing still groups chunks and calls the synchronous `batchEmbedContents` endpoint. The request is valid, but the fix needs maintainer product/config direction rather than conservative cleanup closure.

## What This Changes

Requests an officially supported low-throughput Gemini memory embedding mode for low-quota projects when async remote batch is disabled.

## Best Possible Solution

Design one canonical low-throughput inline embedding policy for remote memory providers, then document and test Gemini behavior for disabled async batch, inline batch sizing, 429 backoff, and any opt-in delay mode.

## Reproduction Assessment

No, not for the live quota failure because it requires a constrained Gemini project. Source inspection does confirm the central request-shape concern: disabled async batch still routes inline indexing through `provider.embedBatch` and Gemini `batchEmbedContents`.

## Solution Assessment

Unclear: the request lists several plausible fixes, and choosing between adaptive 429 splitting, single-request mode, or new batch/delay/backoff config is a product/API decision. The narrowest maintainable path is likely provider-generic inline throttling with Gemini-specific regression coverage.
