## Summary

Keep open: current main and v2026.6.9 still route non-async-batch memory indexing through grouped `embedBatch` calls that Gemini maps to `batchEmbedContents`, and the 429/quota path still needs a maintainer-chosen low-throughput or quota-safety policy.

## What This Changes

The issue requests a safer Gemini memory embedding mode so `remote.batch.enabled=false` can avoid aggressive inline grouped requests and reduce 429 quota/cost failures.

## Best Possible Solution

Define one memory-owned quota-safety policy for Gemini and remote embeddings, then implement it with schema/docs/tests while preserving existing fast defaults unless maintainers explicitly choose a breaking behavior change.

## Reproduction Assessment

Yes at source level: current main still reaches Gemini `batchEmbedContents` through grouped `provider.embedBatch` calls when async batch is disabled. The exact low-quota 429/cost spike still needs live Gemini quota proof.

## Solution Assessment

Unclear: forcing single `embedContent`, adding inline batch size/delay knobs, and adding a provider-level quota circuit breaker are all plausible. The best fix needs a maintainer-selected memory throughput and quota-safety contract.
