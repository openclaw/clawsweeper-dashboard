## Summary

Keep open: current main and the latest release still route non-async Gemini memory indexing through grouped inline batches and Gemini's synchronous batch endpoint, while 429 quota-exhaustion remains treated as a short retryable embedding error. The new 2026.6.9 incident adds concrete production cost-safety evidence, but the right fix needs maintainer policy on low-throughput mode, circuit breaking, and public memorySearch config.

## What This Changes

The issue requests a low-throughput Gemini memory embedding path and safer quota-exhaustion handling when async remote batch is disabled.

## Best Possible Solution

Design one maintainer-approved Gemini and remote memory embedding quota-safety policy that preserves current fast defaults unless intentionally changed, adds focused schema/docs/tests for any public knobs, and stops background retry loops from creating cost runaway after provider-level quota exhaustion.

## Reproduction Assessment

Yes at source level: current main routes disabled async batch mode into grouped inline `provider.embedBatch` calls, and Gemini maps those calls to `batchEmbedContents`. A live low-quota Gemini repro was not run in this read-only review.

## Solution Assessment

Unclear: true single-request `embedContent`, configurable inline batch size/delay, and provider-level circuit breaking are all plausible. The best solution needs maintainer policy on remote embedding throughput and quota exhaustion before a narrow implementation can be selected.
