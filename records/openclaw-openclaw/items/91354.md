## Summary

Keep open: current main and the latest release still route disabled remote async batch mode through inline embedding batches, and Gemini's inline batch path still posts to batchEmbedContents. The remaining work is valid but needs maintainer direction because it likely adds or changes public memorySearch throttling, batch-size, delay, or adaptive 429 behavior.

## What This Changes

Request a supported low-throughput Gemini memory indexing mode when remote async batch is disabled, such as single embedContent calls, smaller inline batches, request delays, configurable backoff, or adaptive 429 splitting.

## Best Possible Solution

Design one provider-generic inline throttling and backoff policy for remote memory embeddings, with Gemini-specific regression coverage proving disabled async batch can run at low quota without changing faster defaults unexpectedly.

## Reproduction Assessment

Yes for the source path, but not for the live quota failure. Current main with remote async batch disabled still reaches inline batching and Gemini batchEmbedContents; a constrained Google/Gemini project is still needed to reproduce the exact 429.

## Solution Assessment

Unclear: the issue lists several plausible fixes, but choosing between adaptive 429 splitting, a single-request mode, and new batch/delay/backoff config is a maintainer product/API decision. The narrowest maintainable direction is likely provider-generic inline throttling with Gemini-specific tests.
