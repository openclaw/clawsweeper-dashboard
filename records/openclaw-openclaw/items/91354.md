## Summary

Keep open: current main and latest release still route disabled remote async batch mode through inline memory embedding batches, and Gemini maps those inline batches to `batchEmbedContents`. The report is valid, but the remedy needs maintainer direction because it likely changes public config or provider retry policy.

## What This Changes

The issue requests a low-throughput Gemini memory embedding path so disabling remote async batch can avoid inline batch requests that hit low quota limits.

## Best Possible Solution

Design one canonical low-throughput inline embedding policy for remote memory providers, then document and test Gemini behavior for disabled async batch, inline batch sizing, 429 backoff, and any opt-in delay mode.

## Reproduction Assessment

We have a high-confidence source reproduction of the central config path, but not a live low-quota Gemini 429 reproduction. Current main bypasses `asyncBatchEmbedContent` when `remote.batch.enabled=false`, yet still calls `provider.embedBatch`, which Gemini implements with `batchEmbedContents`.

## Solution Assessment

Unclear: the request lists several plausible fixes, and choosing between adaptive 429 splitting, a single-request mode, or new batch/delay/backoff config is a product/API decision. The narrowest maintainable path is likely provider-generic inline throttling with Gemini-specific regression coverage.
