## Summary

Keep open: current main and the latest release still disable only Gemini async batch mode, while non-batch memory indexing groups chunks and sends them through Gemini's synchronous batch endpoint. The remaining work is useful but needs maintainer product direction because the likely fixes add or reinterpret public memorySearch/provider throughput policy.

## What This Changes

The issue requests a low-throughput Gemini memory indexing mode so `remote.batch.enabled=false` can avoid grouped inline batch requests that trigger 429s on low-quota Gemini projects.

## Best Possible Solution

Design one canonical low-throughput inline embedding policy for Gemini and remote memory indexing, with schema/docs/tests and existing fast defaults preserved unless maintainers intentionally change them.

## Reproduction Assessment

Yes at source level: current main reaches Gemini `batchEmbedContents` through `provider.embedBatch` even when async `remote.batch.enabled` is false. The actual low-quota 429 needs live Gemini quota proof.

## Solution Assessment

Unclear: single `embedContent`, configurable inline batch size/delay, and adaptive 429 splitting are all plausible. The best fix needs a maintainer-chosen provider/config policy rather than a narrow automatic patch.
