## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and delegates them to the underlying edit tool, while the default loop detector remains opt-in and the related open loop-detection PR only covers an adjacent cross-tool error-cascade case.

## What This Changes

Request to add no-op edit pre-validation, clearer non-retryable tool errors, and a circuit breaker for repeated tool failures.

## Best Possible Solution

Add canonical edit validation that rejects identical `oldText`/`newText` replacements before the wrapped edit tool runs, and leave broader run-level circuit-breaker behavior to the existing loop-detection cluster.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow repair: rejecting no-op replacements in OpenClaw's edit parameter validation is the smallest maintainable fix. The broader circuit-breaker/error-classification design is better handled separately through the loop-detection work already referenced by #75923/#75924.
