## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and delegates them to the wrapped edit tool, while the existing loop-detection circuit breaker remains opt-in and the related open PR only covers an adjacent cross-tool error-cascade detector.

## What This Changes

Request to add tool safety guardrails for no-op `edit` replacements and repeated tool-error loops.

## Best Possible Solution

Add a canonical edit validator that rejects any `edits[]` entry where `oldText` and `newText` are identical before the upstream edit tool runs, with focused tests and no default loop-detection policy change.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow pre-validation fix: rejecting no-op edit replacements in OpenClaw's canonical edit parameter validation is the smallest maintainable solution. The broader circuit-breaker behavior should remain with the related loop-detection review rather than be bundled into this repair.
