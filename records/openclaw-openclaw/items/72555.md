## Summary

Keep open. Current main still accepts identical edit replacements and delegates them to the wrapped edit tool, while the existing loop detector is opt-in and the related open PR only covers adjacent cross-tool error cascades.

## What This Changes

Request edit-tool pre-validation for identical `oldText`/`newText` replacements plus broader tool-error retry and circuit-breaker guardrails.

## Best Possible Solution

Add canonical edit validation that rejects any replacement whose `oldText` and `newText` are identical before the upstream edit tool runs, while leaving broader loop-detection policy to the existing circuit-breaker cluster.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op replacements in OpenClaw's edit parameter validation is the smallest maintainable fix; cross-tool retry/circuit-breaker behavior should stay separate from this edit-specific guard.
