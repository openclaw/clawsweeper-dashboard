## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and delegates them to the wrapped edit tool, while the generic tool-loop breaker is opt-in and the related open PR only covers an adjacent cross-tool error-cascade detector.

## What This Changes

Add edit no-op pre-validation plus clearer tool error circuit-breaker/error-classification guardrails to prevent repeated failing tool calls from wasting tokens.

## Best Possible Solution

Add canonical edit validation that rejects any replacement where `oldText` and `newText` are identical before the upstream edit tool runs, with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow repair: rejecting no-op replacements in OpenClaw's edit parameter validation is the smallest maintainable fix. Broader retry-budget or cross-tool circuit-breaker behavior should remain in the loop-detection cluster.
