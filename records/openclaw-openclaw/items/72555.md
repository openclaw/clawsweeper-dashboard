## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and delegates them to the wrapped edit tool, while the generic tool-loop circuit breaker remains an opt-in policy rather than a default guard.

## What This Changes

The issue requests an edit-tool guard against identical `oldText`/`newText` replacements plus broader repeated tool-error circuit-breaker behavior.

## Best Possible Solution

Add a canonical edit validation guard that rejects any `edits[]` entry where `oldText === newText` with a clear non-retryable parameter error before the upstream edit tool runs; leave default loop-detection policy unchanged unless maintainers revisit it separately.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts that payload and delegates to the base tool.

## Solution Assessment

Yes for the narrow pre-validation portion: rejecting no-op edit replacements in OpenClaw's canonical edit validator is the smallest maintainable fix. The broader default circuit-breaker change is not the best same-step repair because it changes runtime policy beyond this parameter-validation bug.
