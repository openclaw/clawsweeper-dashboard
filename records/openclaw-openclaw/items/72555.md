## Summary

Keep open. Current main still lacks OpenClaw-owned pre-validation for identical edit replacements, and the related loop-detection PR only addresses an adjacent cross-tool error-cascade guard rather than this edit no-op failure mode.

## What This Changes

Request to add edit-tool no-op pre-validation plus broader circuit-breaker and retry/error-classification guardrails for repeated tool failures.

## Best Possible Solution

Add central edit replacement validation that rejects identical `oldText`/`newText` before the upstream edit tool runs, while leaving broader loop-detection policy to the existing related cluster.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts that payload and delegates to the base tool.

## Solution Assessment

Yes for the narrow repair. The validator is the smallest maintainable fix point because it covers host and sandbox edit tools without changing global loop-detection thresholds or retry policy.
