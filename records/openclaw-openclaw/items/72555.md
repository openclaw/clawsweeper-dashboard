## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and delegates them to the underlying edit tool, while the available loop-detection guard remains opt-in and the open consecutive-error work does not cover this narrow edit pre-validation gap.

## What This Changes

The issue requests edit-tool pre-validation for identical replacements plus broader repeated tool-error circuit breaking and non-retryable error classification.

## Best Possible Solution

Add a focused edit validation path that rejects identical replacements with a clear non-retryable message, while leaving generic pre-dispatch validation to https://github.com/openclaw/openclaw/issues/56771 and broader run-scoped error cascade handling to https://github.com/openclaw/openclaw/issues/75923 and https://github.com/openclaw/openclaw/pull/77318.

## Reproduction Assessment

Yes by source inspection. A dummy edit tool wrapped with `REQUIRED_PARAM_GROUPS.edit` and called with `edits: [{ oldText: "same", newText: "same" }]` will pass current validation and reach the wrapped tool.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op edit replacements in the existing edit parameter validation path is the smallest maintainable fix; the broader circuit-breaker and retry-classification policy should stay separate.
