## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements, delegates them unchanged to the upstream edit tool, and the separately tracked loop-detection work does not cover this narrow edit pre-validation gap.

## What This Changes

The issue requests edit-tool pre-validation for identical replacements plus broader tool error circuit-breaker and retry-classification safeguards.

## Best Possible Solution

Add focused edit validation that rejects identical `oldText`/`newText` replacements with a clear non-retryable parameter error, while leaving broader run-scoped circuit-breaker policy to the separate loop-detection thread.

## Reproduction Assessment

Yes by source inspection. A wrapped edit tool called with `edits: [{ oldText: "same", newText: "same" }]` passes current OpenClaw validation and reaches the pinned upstream implementation, which throws the reported no-change error.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op replacements in the existing edit parameter validation path is the smallest maintainable fix; the broader circuit-breaker and retry-classification policy should remain separate.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72555 narrowly. Current main lets edit replacements with identical `oldText` and `newText` pass `REQUIRED_PARAM_GROUPS.edit` validation, then delegates them to the pinned upstream edit implementation, which throws a misleading no-change error that models can retry. Add pre-dispatch validation in the central edit parameter path, likely `src/agents/pi-tools.params.ts`, so any `edits[]` entry where `oldText === newText` is rejected with a clear non-retryable parameter validation message before file I/O. Add focused regression coverage for direct param validation and the created OpenClaw edit tool path. Do not implement generic circuit breakers, new config thresholds, retry budgets, fuzzy matching, apply_patch fallback, or already-at-target/concurrent noop semantics; those are tracked separately by https://github.com/openclaw/openclaw/issues/75923, https://github.com/openclaw/openclaw/pull/77318, https://github.com/openclaw/openclaw/issues/60816, and https://github.com/openclaw/openclaw/pull/77638. Add a changelog fix entry if this user-visible behavior change needs one under repo policy.
