## Summary

Keep open. Current main still validates edit replacements without rejecting `oldText === newText`, delegates those params to the wrapped edit tool, and the existing loop breaker is opt-in while the related open PR covers an adjacent cross-tool error-cascade guard rather than edit no-op pre-validation.

## What This Changes

The issue requests edit no-op pre-validation plus tool error circuit-breaker/error-classification guidance to prevent repeated failed tool calls.

## Best Possible Solution

Add canonical edit validation that rejects identical `oldText`/`newText` replacements before the upstream edit tool runs, and leave broader loop-detection policy to the existing cross-tool error-cascade cluster.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow repair: rejecting no-op replacements in OpenClaw's edit parameter validation is the smallest maintainable fix. The broader circuit-breaker request should stay separate from edit validation because #75923/#75924 already track adjacent loop-detection work.
