## Summary

Keep open. Current main has shipped generic loop safeguards and a dependency-level no-change edit error, but OpenClaw still accepts identical `oldText`/`newText` through its edit parameter validation and delegates the call to the upstream edit tool. The narrow edit pre-validation gap remains actionable, while changing default circuit-breaker policy is a broader maintainer decision.

## What This Changes

The issue requests edit-tool pre-validation for identical replacements, a repeated tool-failure circuit breaker, non-retryable error classification, and agent guidance to prevent failed tool-call retry loops.

## Best Possible Solution

Add an edit-specific preflight guard that rejects identical replacement text with a clear no-op/non-retryable error before the upstream edit tool runs, with focused host and sandbox tests plus a changelog entry. Keep global loop-detection defaults and circuit-breaker policy as separate maintainer-owned work.

## Reproduction Assessment

Yes. A focused unit path can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` or call the host/sandbox edit factory with `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes OpenClaw validation and reaches the base edit tool. I did not execute a mutating edit during this read-only review.

## Solution Assessment

Yes for the narrow edit pre-validation portion, but no for changing global circuit-breaker defaults in the same automated fix. The maintainable repair is an edit-specific no-op guard and tests; broader retry classification and turn termination should be handled separately.
