## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and forwards them to the wrapped edit tool; recent loop-detection work only improves enabled rolling-history guards and does not provide the requested edit pre-validation on the default path.

## What This Changes

The issue requests edit-tool no-op pre-validation plus broader circuit-breaker and error-classification behavior for repeated tool failures.

## Best Possible Solution

Add a focused edit validation path that rejects identical replacements with a clear non-retryable message, while leaving generic pre-dispatch validation and broader loop policy to https://github.com/openclaw/openclaw/issues/56771 and https://github.com/openclaw/openclaw/issues/75923.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op edit replacements in the existing OpenClaw edit parameter validation is the smallest maintainable fix; broader circuit-breaker behavior should stay separate.
