## Summary

Keep open. Current main still lacks OpenClaw-owned pre-validation for identical `oldText`/`newText` edit replacements, and the broader tool-loop guard remains opt-in/generic while the related consecutive-error work is still open and separate.

## What This Changes

Request to add edit no-op pre-validation plus tool-error circuit breaker/error-classification guidance to prevent repeated failed tool loops.

## Best Possible Solution

Add canonical edit validation that rejects identical replacements before the upstream edit tool runs, and leave broader cross-tool retry policy to the open loop-detection cluster.

## Reproduction Assessment

Yes by source inspection. Wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op edit replacements in OpenClaw's edit parameter validation is the smallest maintainable fix, while broader consecutive-error policy should stay with #75923/#75924.

## Repair Work Prompt

Implement the narrow edit safety part of #72555 on current main. In `src/agents/pi-tools.params.ts`, reject edit replacements whose `oldText` and `newText` are identical before the upstream edit tool executes, with a clear parameter-validation error that tells the agent no change will be made and to supply different replacement text. Preserve deletion edits with empty `newText`, multi-edit support, existing path/content validation, and host/sandbox edit recovery behavior. Add focused regression coverage in the existing agent tool tests, update `CHANGELOG.md` because this is user-visible behavior, and do not change loop-detection defaults or implement cross-tool consecutive-error policy; leave that to #75923/#75924.
