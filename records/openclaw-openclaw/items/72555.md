## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and delegates them to the edit tool; the broader loop detector remains opt-in and the related open loop-detection PR only covers adjacent cross-tool error cascades.

## What This Changes

Request a guard that rejects identical edit replacements plus circuit-breaker/error-classification behavior for repeated tool failures.

## Best Possible Solution

Add canonical no-op edit rejection in the edit parameter validator with regression coverage, while leaving broader repeated-error policy to #75923/#75924 or a separate maintainer-approved loop-detection change.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the wrapped edit tool.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op edit replacements before the upstream edit tool runs is the smallest maintainable fix; the broader circuit-breaker behavior should remain in the related loop-detection cluster.
