## Summary

Keep open. Current main still only partially addresses the report: configurable loop detection exists, but it is disabled by default and the OpenClaw edit-tool validation path still accepts identical `oldText`/`newText` replacements before delegating to the base edit tool.

## What This Changes

The issue requests an edit-tool pre-validation guard for identical replacement text plus broader repeated-tool-error circuit breaker behavior.

## Best Possible Solution

Add an OpenClaw-owned edit validation guard that rejects identical replacement text with a clear non-retryable no-op parameter error before the base edit tool runs, covering host and sandbox edit creation paths. Keep global loop-detection default policy and the separate concurrent-edit noop semantics as independent maintainer decisions.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call the current edit validation path with `edits: [{ oldText: "same", newText: "same" }]`; current main accepts it and reaches the base edit tool. I did not run a mutating edit because this review is read-only.

## Solution Assessment

No, current main is not a complete solution because it relies on downstream edit behavior and optional loop detection rather than rejecting the non-changing parameters up front. The narrowest maintainable fix is edit-specific pre-validation; changing global circuit-breaker defaults is a separate policy choice.
