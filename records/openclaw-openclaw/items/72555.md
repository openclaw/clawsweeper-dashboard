## Summary

Keep open. Current main still has the narrow edit no-op validation gap: identical `oldText` and `newText` values pass OpenClaw parameter validation and reach the upstream edit tool, while the broader loop detector remains opt-in for registered tool loops.

## What This Changes

Request to reject no-op edit payloads where `oldText` equals `newText` and add broader safeguards against repeated tool-error loops.

## Best Possible Solution

Add an OpenClaw-owned edit validation guard that rejects identical replacement text with a clear non-retryable no-op error before the base edit tool runs, with focused coverage for shared validation plus host and sandbox edit construction. Leave global loop-detection defaults and turn-abort policy to the existing broader loop-safety discussions.

## Reproduction Assessment

Yes. A focused unit path can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` or use the host/sandbox edit factory and pass `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes OpenClaw validation and reaches the base edit tool.

## Solution Assessment

Yes for the narrow pre-validation repair, and no for bundling broad circuit-breaker policy changes into the same fix. The maintainable path is an edit-specific guard and tests, with broader retry/error-classification behavior handled separately by maintainers.
