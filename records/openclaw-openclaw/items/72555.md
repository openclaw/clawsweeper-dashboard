## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements, passes them through to the upstream edit implementation, and only has opt-in rolling loop detection; the duplicate reports and open related work do not yet cover this narrow pre-validation gap.

## What This Changes

Request a tool-safety guard that rejects no-op edit replacements and broader repeated tool-error circuit-breaker behavior.

## Best Possible Solution

Add focused edit validation that rejects identical replacements with a clear non-retryable message, while leaving generic pre-dispatch validation and broader run-scoped error-cascade policy to their separate issues.

## Reproduction Assessment

Yes by source inspection. A wrapped edit tool called with `edits: [{ oldText: "same", newText: "same" }]` passes current OpenClaw validation and reaches the pinned upstream edit implementation, which throws the reported no-change error.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op replacements in the existing edit parameter validation path is the smallest maintainable step; the broader circuit-breaker and retry-classification policy should remain separate.
