## Summary

Keep open. Current main has opt-in loop detection and shipped generic loop safeguards, but the edit validation path still accepts identical `oldText`/`newText` and delegates to the upstream edit tool, so the narrow no-op edit loop remains actionable.

## What This Changes

Request to prevent repeated edit-tool failures by rejecting identical `oldText` and `newText` before execution and by adding broader tool-loop circuit-breaker/error-classification guidance.

## Best Possible Solution

Add a shared edit validation guard that rejects identical replacement text with a clear no-op/non-retryable error before the base edit tool runs, with focused host and sandbox coverage plus a changelog entry; leave global loop-detection default policy unchanged unless maintainers separately choose to revisit it.

## Reproduction Assessment

Yes. A focused unit path can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` or use the host/sandbox edit factory and pass `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes OpenClaw validation and reaches the base edit tool. I did not execute a mutating edit during this read-only review.

## Solution Assessment

Yes for the narrow pre-validation portion, and no for changing global circuit-breaker defaults as part of the same repair. The maintainable fix is an edit-specific guard and tests; broader retry/error-classification behavior should remain a separate maintainer product decision.
