## Summary

Keep open. Current main has opt-in tool-loop detection and an upstream post-application no-change error, but OpenClaw still accepts identical edit replacements at its validation layer and delegates them to the edit tool instead of rejecting them as a clear non-retryable no-op.

## What This Changes

The issue requests edit-tool pre-validation for identical `oldText`/`newText`, repeated tool-failure circuit breaking, error classification, and agent guidance to prevent retry loops.

## Best Possible Solution

Add a shared edit validation guard that rejects any canonical `edits[]` entry with identical `oldText` and `newText` before the upstream edit tool runs, with a clear non-retryable error, focused host/sandbox tests, and a changelog entry; leave global loop-detection defaults and broader circuit-breaker policy unchanged unless maintainers choose to revisit them separately.

## Reproduction Assessment

Yes. A focused unit path can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` or use the host/sandbox edit factory and pass `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes OpenClaw validation and reaches the base edit path. I did not execute a mutating edit because this was a read-only review.

## Solution Assessment

Yes for the narrow pre-validation portion: an edit-specific validator is the smallest maintainable fix. No for bundling a default global circuit-breaker change into the same repair, because that changes broader agent runtime policy and false-positive tradeoffs.
