## Summary

Keep open. Current main has generic tool-loop safeguards, but the edit-specific no-op path remains actionable: identical `oldText`/`newText` entries still pass OpenClaw validation and reach the base edit tool.

## What This Changes

The issue requests edit-tool pre-validation for identical replacement text plus broader repeated tool-failure circuit-breaker behavior.

## Best Possible Solution

Add a focused edit pre-validation guard that rejects exact identical `oldText`/`newText` entries with a clear non-retryable no-op error before the base edit tool runs, with host/sandbox coverage and a changelog entry; leave broader circuit-breaker defaults to separate maintainer product judgment.

## Reproduction Assessment

Yes. I did not run a mutating edit during this read-only review.

## Solution Assessment

Yes for the narrow pre-validation fix, and no for bundling global circuit-breaker default changes into the same repair. The maintainable fix is edit-specific validation plus tests; broader retry/error-classification behavior should remain a separate product decision.
