## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements at OpenClaw's validation layer, and the broader repeated-tool circuit breaker remains opt-in/default-off, so the central reported token-waste path is not fully addressed.

## What This Changes

The issue requests edit-tool pre-validation for no-op replacements plus repeated tool-failure circuit breaking, error classification, and agent retry guidance.

## Best Possible Solution

Add an OpenClaw-owned canonical edit validation guard that rejects each `edits[]` entry where `oldText` and `newText` are identical, with a clear non-retryable parameter error before the upstream edit tool runs; keep default loop-detection policy unchanged unless maintainers explicitly revisit it.

## Reproduction Assessment

Yes. A focused unit can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes validation and reaches the wrapped tool.

## Solution Assessment

Yes for the narrow pre-validation portion: an edit-specific validator is the smallest maintainable fix. No for bundling default global circuit-breaker changes into the same repair, because that changes broader runtime policy.
