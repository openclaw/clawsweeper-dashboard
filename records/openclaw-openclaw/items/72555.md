## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements before delegating to the wrapped edit tool, while the broader loop-detection behavior remains opt-in and is already being tracked separately by #75923/#75924.

## What This Changes

Add pre-validation for identical edit replacements plus circuit-breaker/error-classification guardrails for repeated tool failures.

## Best Possible Solution

Add a canonical edit validator that rejects `edits[]` entries where `oldText` and `newText` are identical, while leaving broader loop-policy changes to the existing loop-detection track.

## Reproduction Assessment

Yes. A source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow pre-validation fix: rejecting no-op replacements in OpenClaw's canonical edit parameter validation is the smallest maintainable change. The broader circuit-breaker behavior should be reviewed through #75923/#75924 instead of being mixed into this repair.
