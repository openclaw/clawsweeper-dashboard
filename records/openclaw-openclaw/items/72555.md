## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements and delegates them to the wrapped edit tool, while the existing loop detector is opt-in and the related open loop-detection PR only covers an adjacent cross-tool error-cascade guard.

## What This Changes

Requests rejecting no-op edit replacements and adding clearer tool-error guardrails so repeated identical tool failures do not burn turns or tokens.

## Best Possible Solution

Add canonical edit validation that rejects identical replacements before the upstream edit tool runs, and leave broader cross-tool retry policy to the existing loop-detection cluster.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts the payload and reaches the base tool.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op edit replacements in OpenClaw's edit parameter validation is the smallest maintainable fix; broader circuit-breaker behavior should stay with #75923/#75924.
