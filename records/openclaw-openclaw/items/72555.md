## Summary

Keep open. Current main still lets identical `oldText`/`newText` edit replacements pass OpenClaw's edit validation and reach the wrapped edit tool, while the broader tool-loop circuit breaker remains configurable and disabled by default.

## What This Changes

The issue requests edit pre-validation for no-op replacements plus repeated tool-failure circuit breakers, retry classification, and agent guidance to prevent runaway retry loops.

## Best Possible Solution

Add a canonical edit validation guard that rejects any `edits[]` entry where `oldText` and `newText` are identical with a clear non-retryable parameter error before the upstream edit tool runs; leave default loop-detection policy unchanged unless maintainers decide to revisit it separately.

## Reproduction Assessment

Yes. A focused source-level reproduction is to wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts that payload and delegates to the base tool.

## Solution Assessment

Yes for the narrow pre-validation portion: rejecting no-op edit replacements at OpenClaw's canonical edit validator is the smallest maintainable fix. The broader default circuit-breaker change is not the best same-step solution because it changes runtime policy beyond the reported edit parameter bug.
