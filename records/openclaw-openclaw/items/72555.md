## Summary

Keep open. Current main and the latest shipped tag still allow identical `oldText`/`newText` edit replacements through OpenClaw validation, and the related loop-detection work does not replace this narrow edit pre-validation gap.

## What This Changes

The issue requests rejecting no-op edit replacements and adding repeated tool-error guardrails so agents stop retrying identical edit failures.

## Best Possible Solution

Add focused edit pre-validation in the existing edit parameter wrapper, and leave run-scoped/cross-tool circuit-breaker policy to https://github.com/openclaw/openclaw/issues/75923 and https://github.com/openclaw/openclaw/pull/77318.

## Reproduction Assessment

Yes, by source inspection. A wrapped edit call with `edits: [{ oldText: "same", newText: "same" }]` passes current OpenClaw validation and reaches the pinned edit implementation, which throws the reported no-change error.

## Solution Assessment

Yes for the narrow edit repair. Rejecting identical replacement pairs in the existing edit parameter validation path is the smallest maintainable fix; broad circuit-breaker and retry taxonomy changes should stay separate.

## Repair Work Prompt

Add focused validation in the existing edit parameter validation path so any identical replacement pair is rejected before file I/O with a clear non-retryable parameter message. Preserve valid deletion edits (`newText: ""`), existing missing-param behavior, sandbox and host edit surfaces, and host edit recovery semantics. Do not implement generic retry budgets, per-session circuit breakers, or new loop-detection config in this PR; those belong to https://github.com/openclaw/openclaw/issues/75923 and https://github.com/openclaw/openclaw/pull/77318. Add regression coverage and a user-facing changelog entry.
