## Summary

Keep open. Current main still accepts identical `oldText`/`newText` edit replacements, passes them to the pinned upstream edit implementation, and the related loop-detection work covers broader repeated-error policy rather than this narrow edit pre-validation gap.

## What This Changes

The issue requests edit-tool no-op pre-validation plus broader tool error circuit-breaker behavior to prevent repeated failed tool calls.

## Best Possible Solution

Add focused edit validation that rejects identical replacement pairs with a clear non-retryable parameter error, while leaving broader loop/circuit-breaker policy to the separate loop-detection thread.

## Solution Assessment

Yes for the narrow repair. Rejecting no-op replacements in the existing edit parameter validation path is the smallest maintainable fix; broader circuit-breaker and retry-classification policy should remain separate.
