## Summary

Keep open. Re-review against current main shows no relevant change since the prior ClawSweeper comment: OpenClaw still accepts canonical edit entries where `oldText` and `newText` are identical, and the generic circuit-breaker remains opt-in/default-off rather than a default fix for this edit retry loop.

## What This Changes

The issue requests edit-tool pre-validation for identical `oldText`/`newText`, repeated tool-failure circuit breaking, error classification, and agent guidance to prevent retry loops.

## Best Possible Solution

Add a shared edit validation guard that rejects every canonical `edits[]` entry whose `oldText` and `newText` are identical before upstream edit execution, with a clear non-retryable/no-op message, focused host/sandbox coverage, and a changelog entry; leave global loop-detection defaults and broader retry policy to separate maintainer decisions.

## Reproduction Assessment

Yes. A focused unit path can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` or use the host/sandbox edit factories and pass `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes validation and reaches the base edit path. I did not execute a mutating edit because this review is read-only.

## Solution Assessment

Yes for the narrow edit pre-validation portion: an edit-specific validator is the smallest maintainable fix. No for bundling a default global circuit-breaker or broad retry-classification change into the same repair, because that changes agent runtime policy and false-positive tradeoffs.
