## Summary

Keep open. Current main still has no APNG-to-PNG normalization in the shared MIME sniffing or gateway attachment parsing path, so a sniffed `image/apng` can still flow into Anthropic-style image payloads unchanged. Open PR #51964 is an unmerged implementation candidate with closing syntax, so the issue should remain open until that PR or a replacement lands.

## What This Changes

The issue requests normalizing PNG/APNG attachment MIME handling so Claude-style providers receive `image/png` instead of unsupported `image/apng`.

## Best Possible Solution

Keep this issue open and review, rebase, or replace #51964 with a narrow patch that normalizes `image/apng` to `image/png` before provider-facing image MIME values are produced. The most maintainable place appears to be the shared MIME sniff helper, with regression coverage for the APNG sniff result and the gateway attachment path.

## Reproduction Assessment

Yes. A high-confidence unit repro can mock `file-type` to return `image/apng` or use a PNG/APNG buffer with an `acTL` chunk before `IDAT`; current `parseMessageWithAttachments` will choose that sniffed MIME and the Anthropic transport will forward it as `media_type`.

## Solution Assessment

Yes, normalizing APNG in the shared MIME detection path is the narrowest maintainable fix because gateway attachments already consume `sniffMimeFromBase64` through `detectMime`. The open PR points in that direction, but it needs review/rebase or replacement before this issue can close.
