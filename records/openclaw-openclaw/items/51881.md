## Summary

Keep open. Current `main` still lacks APNG-to-PNG normalization in the shared MIME sniffing and gateway attachment path, and open PR #51964 is an unmerged implementation candidate for this issue.

## What This Changes

The issue reports that PNG/APNG MIME sniffing can produce `image/apng`, which is then forwarded to Claude-style providers that only accept `image/png` and related supported image MIME values.

## Best Possible Solution

Review, rebase, or replace #51964 with a current-main patch that normalizes `image/apng` to `image/png` in the shared media MIME detector and keeps focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro path: `file-type` can return `image/apng`, `detectMime` returns it unchanged, `parseMessageWithAttachments` prefers it as `finalMime`, and the Anthropic transport forwards it as `media_type`.

## Solution Assessment

Yes. Shared `detectMime` normalization is the narrowest maintainable fix because gateway attachments and plugin/runtime media helpers already consume that detector; the open PR points in that direction but must be adapted to current `main`.
