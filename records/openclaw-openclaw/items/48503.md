## Summary

Keep open. Current main still exposes `before_tool_call` without action classification or input provenance, and related capability/provenance/security threads remain complementary design work rather than merged superseding implementations.

## What This Changes

Request to add optional action classification and input-provenance fields to `PluginHookBeforeToolCallEvent` for security policy plugins.

## Best Possible Solution

Define an additive, versioned Plugin SDK contract for host-owned action/capability metadata and redacted active input provenance, then thread it through trusted and ordinary `before_tool_call` hooks with docs and contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature request. The gap is source-verifiable: the current hook type, runtime dispatch, and public docs omit action classification and input provenance while input provenance exists elsewhere in the runner/session pipeline.

## Solution Assessment

Unclear until maintainers choose the public security/API contract. The safer path is to coordinate this with tool capability descriptors and provenance trust-model work instead of adding a one-off enum directly to the event.
