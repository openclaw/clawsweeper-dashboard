## What This Changes

Request to enrich `before_tool_call` hook events with optional action classification and input provenance for security policy plugins.

## Best Possible Solution

Define a stable optional policy-context contract for `before_tool_call`, coordinate it with tool capability descriptors and external-content provenance work, then land the SDK types, runtime propagation, docs, and focused hook tests together after maintainer security review.

## Reproduction Assessment

Not a runtime bug, but yes at the source level: the current hook type, runtime event builder, and docs lack the requested `actionCategory` and `inputProvenance` fields.

## Solution Assessment

Unclear until maintainers choose the public shape. The narrowest maintainable path is likely a coordinated plugin API addition that reuses existing provenance helpers and aligns with the separate tool-capability descriptor work.
