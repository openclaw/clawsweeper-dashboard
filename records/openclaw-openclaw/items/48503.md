## Summary

Keep open. Current main still exposes `before_tool_call` without action classification or input provenance, and the related host-hook/tool-metadata work explicitly leaves this capability/provenance request unresolved; the HELM OSS comment is useful prior art for the remaining API and security design.

## What This Changes

Requests optional action classification and input provenance fields on `PluginHookBeforeToolCallEvent` so security policy plugins can make stable policy decisions without hardcoded tool-name maps.

## Best Possible Solution

Define an additive, versioned Plugin SDK contract for host-owned action/capability metadata and redacted active input provenance, then thread it through trusted policies and ordinary `before_tool_call` hooks with docs and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The gap is source-verifiable because current hook types, dispatch, docs, and tests omit action classification and `inputProvenance` while input provenance exists elsewhere.

## Solution Assessment

Unclear until maintainers choose the public contract. The safer direction is to coordinate this with capability descriptors, tool security classification, and provenance/trust-model work before exposing a redacted stable view.
