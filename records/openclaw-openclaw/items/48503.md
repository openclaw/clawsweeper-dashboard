## Summary

Keep open. Current main and the latest shipped tag still expose `before_tool_call` without action classification, capability flags, or `inputProvenance`; the request fits the security/plugin-API direction, but the public taxonomy and redaction model need maintainer decisions first.

## What This Changes

Request additive `before_tool_call` hook fields for host-owned action classification and redacted input provenance so security policy plugins can avoid brittle tool-name mappings.

## Best Possible Solution

Define an additive, versioned Plugin SDK contract for host-owned action/capability metadata and redacted active input provenance, then thread it through trusted policies and ordinary `before_tool_call` hooks with docs, API baselines, and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The gap is source-verifiable because current hook types, runtime dispatch, docs, and `v2026.5.22` omit action classification and `inputProvenance` while input provenance exists elsewhere.

## Solution Assessment

Unclear until maintainers choose the public contract. The safer path is to coordinate action categories, capability flags, provenance redaction, and trusted-policy ordering before exposing a stable SDK field.
