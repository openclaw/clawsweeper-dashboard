## Summary

Keep open. Current main still exposes only toolName, params, runId, and toolCallId on before_tool_call events, while provenance and action/capability classification remain adjacent SDK/runtime concepts rather than a stable public hook contract.

## What This Changes

Add optional action classification and input provenance fields to before_tool_call hook events so security plugins can make policy decisions without brittle tool-name mapping.

## Best Possible Solution

Define a stable additive SDK contract for tool action metadata and redacted active input provenance, thread it through before_tool_call, document the trust/privacy model, and cover the runtime contract with focused tests.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-verifiable on main because the hook type, runtime construction, and docs omit the requested fields while input provenance exists elsewhere.

## Solution Assessment

Unclear until maintainers choose the public shape. A safer direction is to standardize capability/action metadata and a redacted provenance view first, then expose that through before_tool_call instead of adding an ad hoc enum only in the event.
