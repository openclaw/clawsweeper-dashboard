## Summary

Keep open. Current main has adjacent hook enrichments such as `derivedPaths`, trusted tool policies, and run/tool/session context, but it still lacks the requested action classification or input provenance on `before_tool_call`; the remaining shape is a public SDK/security design decision.

## What This Changes

Request to add optional action classification and input provenance fields to `PluginHookBeforeToolCallEvent` for security policy plugins.

## Best Possible Solution

Define an additive, versioned Plugin SDK contract for tool action/capability metadata and redacted active input provenance, thread it through `before_tool_call` and trusted policies, then document and test the trust model.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-verifiable: current main’s hook type, runtime dispatch, and docs omit action classification and `inputProvenance` while input provenance exists elsewhere in the runner/session pipeline.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is to align this with tool capability descriptors and durable provenance work, then expose a redacted, stable view rather than adding a one-off enum only to the event.
