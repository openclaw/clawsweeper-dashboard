## Summary

Keep open: current main and v2026.6.10 still lack the requested stable `before_tool_call` action classification and hook-level input provenance fields, while the open related PR covers only a narrower external-content signal. This remains a public Plugin SDK and security-policy design item, not a stale cleanup close.

## What This Changes

The issue requests optional `actionCategory` and `inputProvenance` fields on `before_tool_call` events so plugins can make security, audit, and compliance decisions without brittle tool-name maps.

## Best Possible Solution

Define one additive, maintainer-approved `before_tool_call` policy-context contract, coordinate it with tool capability descriptors and external-content provenance work, then update SDK types, runtime event construction, docs, and focused hook tests.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 can be inspected and show the hook event and runtime builder omit `actionCategory` and hook-level `inputProvenance` while internal provenance helpers exist elsewhere.

## Solution Assessment

Unclear until maintainers choose the public shape. The narrowest maintainable path is a coordinated plugin API addition that reuses existing provenance helpers and aligns with the separate tool-capability descriptor work.
