## Summary

Keep open. Current main still exposes only toolName, params, runId, and toolCallId on PluginHookBeforeToolCallEvent; session identity is in the hook context, and existing input-provenance plus trusted-policy/tool-metadata plumbing remains adjacent rather than surfaced as the requested action classification or provenance fields.

## What This Changes

Request to enrich before_tool_call with optional action classification or capability context plus input provenance so policy plugins can enforce by stable metadata instead of tool-name matching.

## Best Possible Solution

Define a stable additive SDK contract that combines tool action metadata or capability descriptors with normalized active input provenance, thread it through the before_tool_call runtime path, document the trust/privacy model, and add contract/runtime tests while aligning with #48510, #48519, #58079, #73384, and #74483.

## Reproduction Assessment

Not applicable as a feature request, but the current gap is directly verifiable on main: the hook event type, runtime event construction, and docs all omit action classification and inputProvenance while input provenance exists elsewhere in the runner/session pipeline.

## Solution Assessment

Unclear until maintainers choose the public shape. The narrowest maintainable direction is likely to define stable tool capability/action metadata first, then expose that metadata plus a redacted provenance view through before_tool_call instead of adding an ad hoc enum only in the event.
