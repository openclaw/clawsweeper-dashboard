## Summary

Keep open: current main and v2026.6.6 still do not expose action classification or input provenance on `before_tool_call`, and the related open capability/provenance items are adjacent design work rather than a merged or clearly superseding fix.

## What This Changes

The issue requests optional action-category and input-provenance fields on the `before_tool_call` plugin hook event so security policy plugins can avoid brittle tool-name classification.

## Best Possible Solution

Define one additive, maintainer-approved Plugin SDK hook context contract coordinated with https://github.com/openclaw/openclaw/issues/48510 and related security-classification work, then thread it through trusted policies, normal hooks, docs, API baselines, and focused contract tests.

## Reproduction Assessment

Yes at source level: the current hook type, hook dispatch, public docs, and latest release can be inspected and do not include `actionCategory` or `inputProvenance`. A runtime failing repro is not applicable because this is a feature/API gap rather than broken existing behavior.

## Solution Assessment

Unclear pending maintainer design review. Enriching `before_tool_call` is plausible, but the best fix needs a stable taxonomy and provenance disclosure model coordinated with existing trusted-policy, derived-path, and tool-metadata surfaces.
