## Summary

Keep open: current main still has the source-visible 60-second Telegram health-probe path that calls `getMe` through a cached fetch closure, and the linked closing PR is open but not merged.

## What This Changes

The issue reports a regression where Telegram `getMe` health checks log repeated 10s fetch timeouts every 60 seconds on IPv4-only networks after upgrading to OpenClaw 2026.5.2.

## Best Possible Solution

Review and land PR #76856, or an equivalent Telegram-owned patch, so the health `getMe` probe retains transport fallback state and can promote sticky IPv4 fallback with regression coverage.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance runs a probed health refresh every 60 seconds, health snapshots call the Telegram account probe, and that probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable solution is to fix the Telegram probe transport boundary so health checks reuse or promote the same sticky IPv4 fallback behavior as the main Telegram transport. The linked PR targets that boundary but is not merged yet.
