## Summary

Keep open: current main still contains the 60-second Telegram health-probe path that calls `getMe` through a cached fetch closure, and the closing PR that targets this regression remains open rather than merged.

## What This Changes

The issue reports a Telegram `getMe` health-check timeout storm on IPv4-only networks with broken IPv6 egress, regressed in OpenClaw 2026.5.2 versus 2026.4.26.

## Best Possible Solution

Land #76856 or an equivalent focused Telegram-owned patch that lets health `getMe` probes reuse or promote sticky IPv4 fallback, with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance runs a probed health refresh every 60 seconds, health snapshots call the Telegram account probe, and that probe calls `getMe` via `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable direction is to fix the Telegram probe transport path so the health `getMe` call can promote to the same sticky IPv4 fallback used elsewhere. The linked PR targets that boundary but is not merged into current main.
