## Summary

Keep open: current main and v2026.5.7 still route Telegram getMe health probes through a cached fetch closure without probe-level transport fallback, and the linked closing PR remains open.

## What This Changes

The issue reports a Telegram getMe health-check timeout storm on IPv4-only networks where DNS still returns IPv6 records.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/76856, or an equivalent Telegram-owned patch that lets getMe probes reuse or promote sticky IPv4 fallback, then close after merge and release verification.

## Reproduction Assessment

Yes. The reporter provides concrete broken-IPv6 logs, and current main still routes the 60-second health refresh through probeAccount into probeTelegram getMe using a cached fetch closure without probe-level transport.forceFallback.

## Solution Assessment

Yes. The narrow maintainable fix is to keep the fallback behavior in TelegramTransport and let the getMe probe cache/promote that transport; the linked open PR targets that boundary, so this issue should remain open until that work lands or is superseded.
