## Summary

Keep open: current main and the latest release still route the Telegram getMe health probe through a cached fetch closure without a probe-level transport fallback, and the focused closing implementation remains open rather than merged.

## What This Changes

The issue reports a regression where periodic Telegram getMe health checks time out every 60 seconds on IPv4-only networks when DNS still returns AAAA records.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/76856, or an equivalent Telegram-owned patch that lets getMe health probes reuse or promote sticky IPv4 fallback, then close this issue after merge and release verification.

## Reproduction Assessment

Yes. The reporter provides concrete broken-IPv6 logs, and current main still runs the 60-second health refresh through probeAccount into probeTelegram getMe using a cached fetch closure without probe-level transport.forceFallback; I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the Telegram probe transport path so getMe retries can promote the same sticky IPv4 fallback; the linked open PR targets that boundary, so this issue should stay open until that work lands or is superseded.
