## Summary

Keep open: current main still contains the periodic Telegram health probe path that reaches `getMe` through a cached fetch closure, and the focused closing PR is still open rather than merged.

## What This Changes

The issue reports a regression where periodic Telegram health `getMe` probes timeout every about 60 seconds on hosts whose DNS returns IPv6 records but whose network lacks working IPv6 egress.

## Best Possible Solution

Land #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse or promote sticky IPv4 fallback, with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance refreshes probed health every 60 seconds, health collection invokes Telegram `probeAccount`, and the probe calls `getMe` via `fetchWithTimeout`. I did not live-test on a host with broken IPv6 egress.

## Solution Assessment

Yes, the narrow maintainable fix is in the Telegram probe transport path so health `getMe` can promote to the same sticky IPv4 fallback used by the full transport. The linked PR targets that boundary but is not merged into current main.
