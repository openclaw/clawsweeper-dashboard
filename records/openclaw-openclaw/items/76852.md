## Summary

Keep open: current main still matches the reported Telegram health-probe timeout path, and #76856 is an open unmerged closing PR for the focused fix.

## What This Changes

Fix the Telegram `getMe` health probe so IPv4-only networks with broken IPv6 egress do not emit recurring 10s fetch-timeout logs every health refresh.

## Best Possible Solution

Land #76856 or an equivalent Telegram-owned patch that keeps the health `getMe` probe on the full transport and promotes sticky IPv4 fallback after timeout, with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: the gateway refreshes probed health every 60 seconds, health calls the Telegram `probeAccount`, and that probe issues `/getMe` through `fetchWithTimeout` using a cached fetch-only transport. I did not run a live Debian host with broken IPv6 egress.

## Solution Assessment

Yes, the narrow maintainable fix is the linked PR's direction: make the Telegram health probe keep access to the full transport and call or reuse sticky IPv4 fallback on probe timeout. A broader network-policy or config-knob change is not needed for this issue.
