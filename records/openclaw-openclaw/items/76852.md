## Summary

Keep open: current main still routes the periodic Telegram health probe through a cached fetch closure without promoting the transport fallback, and the focused closing PR remains open and unmerged.

## What This Changes

The issue asks OpenClaw to stop recurring Telegram getMe fetch-timeout storms on IPv4-working hosts with broken IPv6 egress by reusing or promoting Telegram's sticky IPv4 fallback for health probes.

## Best Possible Solution

Land #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse and promote the sticky IPv4 fallback, with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: current main schedules a 60-second probed health refresh, the health snapshot calls Telegram `probeAccount`, and the probe calls `/getMe` through `fetchWithTimeout` without a transport fallback promotion path. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Telegram probe transport path, not generic gateway health scheduling. The linked PR targets that boundary by caching the full transport and calling `forceFallback`, but it is not merged.
