## Summary

Keep open: current main still routes periodic Telegram health `getMe` probes through a cached fetch closure without probe-owned `forceFallback`, and the closing PR that targets this regression is still open rather than merged.

## What This Changes

Fix the regression where periodic Telegram `getMe` health probes repeatedly time out on IPv4-working hosts with unreachable IPv6 egress instead of promoting to sticky IPv4 fallback.

## Best Possible Solution

Review and land #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse/promote the sticky IPv4 fallback, with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance runs a probed health refresh every 60 seconds, health snapshots call Telegram `probeAccount`, and that probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable fix is to change the Telegram probe transport path so the health `getMe` retry can promote to the same sticky IPv4 fallback used by startup control requests. The open linked PR targets that boundary and should be reviewed or superseded, not bypassed with a competing issue-close.
