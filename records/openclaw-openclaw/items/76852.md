## Summary

Keep open: current main still shows the periodic Telegram health-probe path calling `getMe` through a cached fetch closure, while the focused closing PR remains open rather than merged.

## What This Changes

The issue reports a regression where periodic Telegram `getMe` health probes time out every roughly 60 seconds on IPv4-only networks with broken IPv6 egress.

## Best Possible Solution

Land #76856 or an equivalent focused Telegram-owned patch that lets health `getMe` probes reuse and promote sticky IPv4 fallback, with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance runs a probed health refresh every 60 seconds, health snapshots call the Telegram account probe, and that probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Telegram probe transport path so health `getMe` can use or promote the same sticky IPv4 fallback as other Telegram control calls. The linked PR targets that boundary but is not merged into current main.
