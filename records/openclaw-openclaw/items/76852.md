## Summary

Keep open: current main still routes the 60-second gateway health probe into Telegram `getMe` through `fetchWithTimeout` and a cached `resolveTelegramFetch` closure, while the focused closing PR is open and unmerged.

## What This Changes

The issue asks OpenClaw to stop recurring Telegram `getMe` fetch-timeout storms on IPv4-only hosts with broken IPv6 egress after the 2026.5.2 release.

## Best Possible Solution

Land PR #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse or promote the sticky IPv4 fallback, with regression coverage for the timeout path.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance refreshes health with `probe: true` every 60 seconds, health snapshots call Telegram `probeAccount`, and that probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes for the repair direction: the narrow maintainable fix is to cover the Telegram probe transport with the existing sticky IPv4 fallback seam. The linked PR targets that boundary, but the issue should remain open until it merges or is replaced.
