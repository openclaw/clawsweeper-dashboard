## Summary

Keep open: current main still has the source-visible periodic Telegram health `getMe` path that can repeat `fetchWithTimeout` without promoting the probe transport to the sticky IPv4 fallback, and the linked closing PR #76856 is open rather than merged.

## What This Changes

The issue reports a 2026.5.2 Telegram regression where periodic health `getMe` probes time out every ~60 seconds on networks with working IPv4 but no usable IPv6 egress.

## Best Possible Solution

Review and land #76856 or an equivalent Telegram-owned fix that lets the health `getMe` probe reuse or promote sticky IPv4 fallback, with regression coverage, before closing this issue.

## Reproduction Assessment

Yes, source-reproducible: current main runs a probed gateway health refresh every 60 seconds, routes it into the Telegram account probe, and calls `getMe` via `fetchWithTimeout` through a cached fetch closure. I did not live-test on a host with broken IPv6 egress.

## Solution Assessment

Yes, the best fix direction is narrow: keep this in the Telegram probe/transport boundary so health probes can reuse or force the existing sticky IPv4 fallback. The linked PR targets that boundary but still needs normal review and merge.
