## Summary

Keep open: current main still has the 60-second probed Telegram health refresh path, and the Telegram probe still caches only a fetch closure from `resolveTelegramFetch`, so the reported `getMe` timeout path cannot explicitly promote the transport to sticky IPv4 fallback. The linked closing PR #76856 remains the right implementation candidate, but it is not merged in the provided context.

## What This Changes

Request to stop recurring Telegram health `getMe` fetch timeouts on IPv4-working, IPv6-broken networks by reusing the Telegram IPv4 fallback path for periodic probes.

## Best Possible Solution

Land #76856 or an equivalent focused Telegram-owned patch that lets health `getMe` probes reuse or promote sticky IPv4 fallback, with regression coverage, then close this after merge.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance refreshes probed health every 60 seconds, health snapshots call Telegram `probeAccount`, and the probe calls `getMe` through `fetchWithTimeout`. I did not establish a live failing run on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable fix is in the Telegram probe transport path, not core gateway scheduling: retain the full transport and force/promote fallback after a probe timeout. The linked PR targets that boundary but is not merged into current main.
