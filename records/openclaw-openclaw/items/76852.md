## Summary

Keep open: current main still routes periodic Telegram health `getMe` probes through a cached fetch closure that cannot explicitly promote the full Telegram transport on timeout, and #76856 remains the open closing PR for this regression.

## What This Changes

The issue reports a 2026.5.2 regression where Telegram health `getMe` probes repeatedly hit 10s fetch timeouts on hosts with IPv4 egress but broken IPv6 egress.

## Best Possible Solution

Land #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse and promote the sticky IPv4 fallback transport, with regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main refreshes probed gateway health every 60 seconds, that health path calls Telegram `probeAccount`, and the probe still sends `getMe` through `fetchWithTimeout` using a cached fetch closure. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable fix is to change the Telegram probe transport boundary so health probes can force or reuse the same sticky IPv4 fallback path as the full transport. The linked PR targets that boundary but is not merged.
