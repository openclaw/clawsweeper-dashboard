## Summary

Keep open: current main still has the 60-second Telegram health-probe path that calls `getMe` through `fetchWithTimeout`, and the linked closing PR is open/unmerged in the provided context.

## What This Changes

The issue reports a regression where Telegram health `getMe` probes repeatedly time out every 60 seconds on hosts with IPv4 egress but broken IPv6 egress.

## Best Possible Solution

Land PR #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse or promote sticky IPv4 fallback state, with regression coverage for the retry path.

## Reproduction Assessment

Yes, at source level: current main refreshes health every 60 seconds, probes configured Telegram accounts, and calls `getMe` through `fetchWithTimeout`. I did not live-test a broken-IPv6 host, but the reporter's network probes and logs match that path.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the Telegram probe transport path so the health check uses the same sticky IPv4 fallback behavior as the rest of the Telegram transport. PR #76856 appears aligned with that direction but is not merged yet.
