## Summary

Keep open: current main still has the source-visible Telegram health path that probes `getMe` every 60 seconds through `fetchWithTimeout`, and the focused closing PR is still open rather than merged.

## What This Changes

The issue requests fixing a Telegram regression where periodic health-check `getMe` probes repeatedly time out on IPv4-only networks with broken IPv6 egress.

## Best Possible Solution

Land PR #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse or promote sticky IPv4 fallback and adds regression coverage.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance runs probed health refresh every 60 seconds, health snapshots call the Telegram account probe, and that probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a host with broken IPv6 egress.

## Solution Assessment

Yes: the narrow maintainable fix is to cover the Telegram probe transport with the same sticky IPv4 fallback behavior already present in the Telegram transport layer. The linked PR targets that boundary but has not merged yet.
