## Summary

Keep open: current main and the latest release still route the periodic Telegram getMe probe through a cached fetch closure that cannot force the Telegram transport IPv4 fallback, and the focused closing implementation remains open at https://github.com/openclaw/openclaw/pull/76856.

## What This Changes

The issue reports a regression where Telegram getMe health checks time out every 60 seconds on hosts with broken IPv6 egress.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/76856, or an equivalent Telegram-owned patch that lets getMe health probes reuse and promote the sticky IPv4 fallback, then close this issue after merge and release verification.

## Reproduction Assessment

Yes, source-reproducible with high confidence: the reporter provides concrete broken-IPv6 logs, and current main still sends the 60-second health refresh through Telegram probeTelegram getMe using a cached fetch closure without probe-level forceFallback. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the Telegram probe transport path so getMe retries can promote and reuse the same sticky IPv4 fallback; the linked open PR targets that boundary, so this issue should stay open until that work lands or is superseded.
