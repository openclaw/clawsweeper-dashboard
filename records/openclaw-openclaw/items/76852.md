## Summary

Keep open: current main still contains the recurring Telegram health-probe path that can call `getMe` every 60 seconds through a cached fetch closure, and the closing PR that targets this regression remains open and unmerged.

## What This Changes

Report a v2026.5.2 regression where periodic Telegram health probes repeatedly time out on hosts with IPv4 egress but broken IPv6 egress.

## Best Possible Solution

Review and land #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse and promote the full transport fallback, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: the gateway refreshes probed health every 60 seconds, health calls the Telegram account probe, and that probe calls `/getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable fix is in the Telegram probe transport boundary: reuse the full transport and force/promote fallback after probe timeouts. The linked PR targets that boundary but is not merged into current main.
