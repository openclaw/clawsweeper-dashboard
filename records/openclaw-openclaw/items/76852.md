## Summary

Keep open: current main still contains the source-visible 60-second Telegram health probe path that reaches `getMe` through a cached fetch closure, and the closing PR that targets this regression remains open rather than merged.

## What This Changes

The issue reports a regression where periodic Telegram health `getMe` probes in 2026.5.2 repeatedly time out on IPv4-only networks with broken IPv6 egress.

## Best Possible Solution

Land #76856 or an equivalent focused Telegram-owned patch that lets health `getMe` probes reuse or promote sticky IPv4 fallback, with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance refreshes health with probing every 60 seconds, health collection invokes the Telegram account probe, and the probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a host with broken IPv6 egress.

## Solution Assessment

Yes, the narrow maintainable fix is to change the Telegram probe transport path so health `getMe` can reuse or promote the same sticky IPv4 fallback used by the Telegram transport. The linked PR targets that boundary but is not merged into current main.
