## Summary

Keep open: current main still routes the periodic Telegram health getMe probe through a cached fetch closure, so it cannot promote the cached transport to the sticky IPv4 fallback after a fetch-timeout. The open closing PR #76856 already targets this exact regression and should be reviewed or landed before this issue is closed.

## What This Changes

The issue reports recurring Telegram getMe health-check fetch timeouts on IPv4-only egress networks after upgrading to 2026.5.2.

## Best Possible Solution

Make Telegram health probes reuse the full Telegram transport and promote sticky IPv4 fallback after getMe timeout failures, with regression coverage, then close this issue after #76856 or an equivalent patch merges.

## Reproduction Assessment

Yes. The current source path shows a 60-second gateway health refresh invoking Telegram probeAccount, which calls getMe through fetchWithTimeout using a cached fetch closure that cannot force sticky IPv4 fallback after timeout.

## Solution Assessment

Yes. The narrow maintainable fix is in the Telegram probe transport boundary, not a broad core networking change; #76856 appears to target that boundary but is not merged into current main.
