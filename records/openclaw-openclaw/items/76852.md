## Summary

Keep open: current main still has the source-visible 60-second Telegram `getMe` health-probe path, and the focused closing PR for this regression remains open rather than merged.

## What This Changes

The issue requests that Telegram health checks on IPv4-only hosts avoid recurring `getMe` fetch-timeout storms by reusing the IPv4 fallback behavior observed in 2026.4.26.

## Best Possible Solution

Land #76856 or an equivalent Telegram-owned patch that lets health `getMe` probes reuse and promote the sticky IPv4 fallback transport, with focused regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: current main schedules a 60-second probed health refresh, the health path calls the Telegram account probe, and the probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes, the narrow maintainable solution is to fix the Telegram probe transport path so `getMe` health checks can reuse or force the same sticky IPv4 fallback already present in the Telegram transport. The linked PR targets that boundary but is not merged.
