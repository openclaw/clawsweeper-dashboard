## Summary

Keep open: current main still has a source-visible Telegram `getMe` timeout path in the 60s gateway health refresh, and probe request timeouts are not wired to the sticky IPv4 fallback used by Telegram bot control requests.

## What This Changes

Fix the Telegram health/startup probe so `getMe` does not repeatedly time out every 60 seconds on IPv4-only or broken-IPv6 networks.

## Best Possible Solution

Make the Telegram probe transport reuse or promote sticky IPv4 fallback when `getMe` times out, with focused regression tests and no new user config knob.

## Reproduction Assessment

Yes, source-reproducible: the gateway refreshes health every 60 seconds, enabled Telegram accounts are live-probed, and the probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host, but the source path matches the reported cadence and log text.

## Solution Assessment

Unclear for a specific patch because none is proposed here. The narrow maintainable fix is to make probe-time request timeouts promote or reuse Telegram sticky IPv4 fallback, not to disable probes or add another config option.
