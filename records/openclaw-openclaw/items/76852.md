## Summary

Keep open: current main still routes periodic Telegram health checks through the getMe probe path that can repeat the reported fetch timeout, and the focused closing PR #76856 is still open rather than merged.

## What This Changes

The issue requests restoring Telegram health checks so getMe probes fall back to IPv4 instead of producing recurring fetch-timeout logs on broken-IPv6 networks.

## Best Possible Solution

Land #76856 or an equivalent focused Telegram-owned patch that lets health getMe probes reuse and promote sticky IPv4 fallback, with regression coverage.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance refreshes health every 60 seconds, health snapshots invoke channel probes, and the Telegram probe calls getMe through `fetchWithTimeout`. I did not perform live validation on a broken-IPv6 host.

## Solution Assessment

Yes: the narrow maintainable fix is to adjust the Telegram probe transport path so it can reuse or promote the existing sticky IPv4 fallback, rather than changing gateway health cadence or adding a new user option.
