## Summary

Keep open: current main still has the source-visible 60-second Telegram health-probe path that calls `getMe` through `fetchWithTimeout`, and the linked closing PR is open but not merged.

## What This Changes

The issue asks OpenClaw to stop the Telegram `getMe` fetch-timeout storm on IPv4-only or broken-IPv6 networks, reported as a regression in 2026.5.2 versus 2026.4.26.

## Best Possible Solution

Land a focused Telegram-owned fix, likely PR #76856 or an equivalent patch, that lets health `getMe` probes reuse or promote sticky IPv4 fallback and adds regression coverage before closing this issue.

## Reproduction Assessment

Yes, source-reproducible: gateway maintenance runs a probed health refresh every 60 seconds, health snapshots call the Telegram account probe, and that probe calls `getMe` via `fetchWithTimeout`. I did not live-test on a broken-IPv6 host.

## Solution Assessment

Yes for the implementation direction: the narrow maintainable fix is to cover the Telegram probe transport with the same sticky IPv4 fallback behavior already present in the transport layer. The linked PR targets that boundary but is not merged yet.
