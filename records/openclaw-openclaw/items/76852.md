## Summary

Keep open: current main still has a source-visible 60s health-probe path that calls Telegram `getMe` through `fetchWithTimeout`, while the Telegram bot-client timeout fallback does not cover that probe path.

## What This Changes

The issue asks OpenClaw to stop recurring Telegram `getMe` timeout logs and latency on hosts with broken IPv6 egress by falling back cleanly to IPv4 during health/startup probes.

## Best Possible Solution

Make Telegram probe transport reuse or arm sticky IPv4 fallback on request timeout so periodic health snapshots remain quiet and useful on IPv4-only networks.

## Reproduction Assessment

Yes, source-reproducible: an enabled Telegram account is live-probed by the gateway health refresh every 60 seconds, and that probe calls `getMe` through `fetchWithTimeout`. I did not live-test on a broken-IPv6 host, but the current-main call chain matches the reported cadence and log subsystem.

## Solution Assessment

Unclear for a specific patch because none is proposed here. The narrow maintainable fix is to cover the Telegram health/startup probe path with the same sticky IPv4 timeout fallback behavior already used by the bot client, rather than adding a new config knob.

## Repair Work Prompt

Fix the Telegram `getMe` fetch-timeout storm reported in https://github.com/openclaw/openclaw/issues/76852. Current main refreshes gateway health with `probe: true` every 60 seconds; `src/commands/health.ts` calls Telegram `probeAccount`, which calls `probeTelegram`, which uses `fetchWithTimeout` for `getMe`. On networks where IPv4 works but IPv6 egress times out, this can log `[fetch-timeout]` every health refresh. Keep the fix narrow: make the Telegram probe/startup getMe path reuse or force the existing Telegram sticky IPv4 fallback on request timeout, or otherwise avoid repeated timeout logging while preserving real token/reachability failures. Do not add a new config option and do not change unrelated global networking behavior; treat open PR #56798 as related background for built-in fetch, not as the Telegram probe fix. Add regression coverage for the periodic health-probe path and the Telegram probe timeout/fallback behavior, and add a user-facing changelog entry because this is a release-visible channel reliability fix.
