## Summary

Keep open; current main still does not use `probe.npmResolution.version` when formatting npm plugin dry-run update targets, and the two linked fix PRs are open rather than merged.

## What This Changes

The issue asks for `openclaw plugins update --all --dry-run` to show the resolved npm target version instead of `unknown` when npm metadata resolution already found a version.

## Best Possible Solution

Land one canonical fix that preserves the exact-spec fallback, falls back to `probe.npmResolution?.version`, adds focused regression coverage, and then close this issue after the PR merges.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: mock a successful npm dry-run probe with `npmResolution.version` and no top-level `version` for a bare or tagged npm spec, and current main can still build an `unknown` target label. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is in `src/plugins/update.ts`: derive the dry-run target from `probe.version ?? probe.npmResolution?.version ?? exactSpecFallback` and use that same resolved value for the message, outcome field, and unchanged check.
