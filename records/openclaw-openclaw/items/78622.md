## Summary

Keep open; current main still has the reported dry-run path where npm update probes can include `npmResolution.version` without a top-level `version`, while the formatter only reads `probe.version`.

## What This Changes

Dry-run plugin updates should show the resolved npm target version instead of `unknown` when npm metadata includes the version.

## Best Possible Solution

Use `probe.version ?? probe.npmResolution?.version` consistently for dry-run display, outcome `nextVersion`, and unchanged comparison where applicable, then add focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, from source: mock an npm dry-run install result with `ok: true`, `npmResolution.version`, and no top-level `version`; current main formats only `probe.version`, so the outcome is `unknown`. I did not run a live command because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/plugins/update.ts`; changing only the CLI logger would be too late because it prints the already-built outcome message.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78622. In `src/plugins/update.ts`, make npm plugin update dry-run outcomes report the resolved target version from `probe.version ?? probe.npmResolution?.version` instead of falling back to `unknown` when the installer returned npm resolution metadata only. Use the same resolved value for `nextVersion` in the outcome object and for unchanged/update comparison where appropriate. Add a focused regression test in `src/plugins/update.test.ts` that mocks `installPluginFromNpmSpec({ dryRun: true })` returning `ok: true`, `npmResolution.version`, and no top-level `version`, then asserts the dry-run message and outcome use that version. Add a single-line user-facing changelog fix entry. Do not change npm resolution semantics, install records, plugin index migration, or hook-pack update behavior.
