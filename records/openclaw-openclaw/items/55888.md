## Summary

Keep this issue open. Current main supports the reported `agents.defaults.compaction` knobs, but it still defaults to a 20,000-token Pi reserve floor plus a 4,000-token memory-flush soft threshold, so a 32,768-token local Ollama model still reaches the reported preflight threshold around 8.7k unless users opt in. The remaining gap is a documented high-memory local-model profile or similarly explicit opt-in guidance, not an implemented fix or exact duplicate.

## What This Changes

The issue requests high-memory local Ollama compaction tuning guidance or an opt-in profile so 32k-context models on 32GB/64GB machines do not compact around 8k tokens by default.

## Best Possible Solution

Keep the conservative runtime defaults, and add a documented high-memory local Ollama compaction profile that explains `reserveTokensFloor`, `keepRecentTokens`, the 4,000-token memory-flush buffer, expected threshold math, and low-RAM/OOM cautions. If maintainers want more than documentation, add an explicit opt-in preset with focused threshold/config precedence tests rather than silently changing defaults.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main computes the relevant threshold as `32768 - 20000 - 4000 = 8768`, and the test suite explicitly keeps the 20k floor for a 32,768-token context window; the live M4 Pro latency claim was not reproduced in this read-only review.

## Solution Assessment

Yes for an opt-in docs/profile fix, no for a blanket default change. The narrow maintainable path is to document the supported high-memory configuration and warnings first; hardware-aware automation would need product approval and live validation.
