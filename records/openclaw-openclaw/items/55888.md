## Summary

Keep open. Current main and the latest release still expose the existing compaction knobs and keep the conservative default threshold behavior, while the docs still lack a high-memory local Ollama tuning profile for this reported use case.

## What This Changes

Request to document or tune compaction thresholds for high-memory local Ollama models so a 32,768-token model can avoid early pre-compaction around 8k tokens.

## Best Possible Solution

Keep the conservative runtime defaults and add a documented high-memory local Ollama profile with threshold math, RAM cautions, and supported agents.defaults.compaction keys.

Yes for the source-level threshold behavior: current main computes 32,768 - 20,000 - 4,000 = 8,768 by default. The hardware-specific M4 Pro latency claim was not reproduced in this read-only review.

## Solution Assessment

Yes for a docs/profile fix, no for a blanket default change. The narrow maintainable solution is explicit high-memory guidance using existing config knobs; runtime auto-tuning would need separate validation.

Document the threshold math using existing behavior: effective preflight threshold is contextWindowTokens - reserveTokensFloor - memoryFlush.softThresholdTokens, so a 32,768-token Ollama model with the default 20,000 reserve and 4,000 soft threshold triggers around 8,768 tokens; lowering reserveTokensFloor for high-memory local setups can raise usable context. Include supported keys under agents.defaults.compaction: mode, reserveTokensFloor, keepRecentTokens, and optionally memoryFlush.softThresholdTokens.
