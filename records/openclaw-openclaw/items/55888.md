## Summary

Keep open. Current main still supports the tuning knobs but keeps the conservative 20,000-token reserve plus 4,000-token memory-flush buffer, and the docs still lack the requested high-memory local Ollama tuning profile.

## What This Changes

The issue requests high-memory local Ollama compaction tuning guidance or an opt-in profile so 32k-context models on 32GB/64GB machines avoid premature compaction around 8k tokens.

## Best Possible Solution

Keep the conservative defaults and add a documented high-memory local Ollama tuning profile with threshold math, RAM cautions, and supported config keys.

## Reproduction Assessment

Yes for the source-level threshold behavior: current main computes 32,768 - 20,000 - 4,000 = 8,768 by default. The live M4 Pro latency claim was not reproduced in this read-only review.

## Solution Assessment

Yes for a docs/profile fix, no for a blanket default change. The narrow maintainable path is explicit high-memory guidance using existing config knobs; runtime auto-tuning would need separate product validation.
