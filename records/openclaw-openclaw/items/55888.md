## Summary

Keep open. Current main still has the conservative 32k-window threshold behavior, the config knobs already exist, and the public docs still lack the high-memory local Ollama profile, threshold math, and RAM caution that would make this report actionable for users.

## What This Changes

Request to document or support high-memory local Ollama compaction tuning so 32k-context models do not hit the default 8k-ish pre-compaction threshold on capable machines.

## Best Possible Solution

Keep conservative runtime defaults and add a focused docs profile for high-memory local Ollama users with the threshold formula, supported `agents.defaults.compaction` keys, and low-RAM/OOM cautions.

## Reproduction Assessment

Yes for the source-level threshold behavior: current main computes `32768 - 20000 - 4000 = 8768` with defaults. The hardware-specific M4 Pro latency claim was not reproduced in this read-only review.

## Solution Assessment

Yes for a docs/profile fix, no for a blanket default change. The narrow maintainable path is to document the existing high-memory tuning knobs rather than auto-tune defaults for all hosts.
