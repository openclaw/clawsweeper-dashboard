## Summary

Keep open: current main still has a single run deadline for chat/agent turns, while this asks for new liveness timeout semantics, new config, and channel-visible progress UX. A related open PR addresses part of the Telegram visibility problem, but it does not implement the core per-step timeout/reset contract.

## What This Changes

Request liveness-based per-step timeout resets, a hard max-turn ceiling, graceful timeout reporting, and progress notifications for long agent turns.

## Best Possible Solution

Define a backward-compatible core timeout/liveness contract with an explicit absolute guardrail, channel-aware progress delivery, docs, and focused regression coverage; review the open Telegram PR as a possible visibility slice rather than the whole fix.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction of the current behavior: chat.send and agent RPC register one `expiresAtMs`, and the embedded runner schedules one `params.timeoutMs` abort timer; no live Telegram end-to-end run was performed.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact timeout semantics, defaults, migration behavior, reset events, and notification policy need maintainer product/architecture decisions before a narrow fix is safe.

## Risks / Open Questions

- Changing `timeoutSeconds` from a total-turn budget to a liveness budget could let active turns run much longer than operators expect unless an absolute ceiling and migration story are defined.
- Progress notices may duplicate existing tool summaries, typing indicators, progress drafts, and Telegram-specific acknowledgements without channel-aware throttling.
- The reset boundary must align with model idle timeouts, tool/subprocess execution, compaction grace, diagnostics recovery, and failover so real hangs are not hidden by unrelated progress.
