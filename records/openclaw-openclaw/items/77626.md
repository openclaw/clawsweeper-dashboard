## Summary

Keep open: current main still treats `timeoutSeconds` as an overall embedded agent run deadline, while this request changes timeout semantics and adds new config/notification UX that needs maintainer product design.

## What This Changes

Request liveness-based per-step agent turn timeouts, periodic progress notifications, and clearer timeout reporting for long tool-heavy turns.

## Best Possible Solution

Design a core agent-run liveness contract with an explicit absolute guardrail, well-defined idle-reset events, channel-aware progress delivery, docs, and regression coverage.

## Reproduction Assessment

No high-confidence end-to-end Telegram reproduction was run. Source inspection does confirm current main resolves one run timeout and schedules a single embedded attempt abort timer rather than a liveness-resetting per-step window.

## Solution Assessment

Unclear. The direction is plausible, but changing `timeoutSeconds` from total-turn to per-step semantics and adding progress/max-turn config needs a maintainer decision on migration, defaults, and per-channel notification behavior.
