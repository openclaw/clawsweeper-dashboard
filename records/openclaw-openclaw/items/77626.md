## Summary

Keep open: current main has related timeout and progress pieces, but it does not implement the requested liveness-resetting turn timeout semantics, `maxTurnTimeSeconds`/`progressNotifyAfterSeconds` config, or detailed timeout reporting. This is a feature and compatibility design request rather than a narrow bug fix.

## What This Changes

The issue requests liveness-based per-step timeout semantics, progress notifications during long turns, and clearer timeout termination messages for long agent turns.

## Best Possible Solution

Settle an additive timeout contract, then wire liveness events and timeout details through the agent runner, Gateway, cron, and channel delivery paths with schema, docs, and regression coverage.

## Reproduction Assessment

No bug-style live reproduction was run in this read-only review, but source inspection gives a high-confidence current-main path: the run gets one fixed timeout/expiry and the embedded runner schedules one abort timer. The requested behavior is therefore source-reproducible as missing rather than a reproduced regression.

## Solution Assessment

Unclear: the proposed direction is plausible, but reinterpreting `timeoutSeconds` is a compatibility/product decision. A safer final design may keep the existing key as a total-turn guard and add explicit liveness/idle keys or an opt-in migration.
