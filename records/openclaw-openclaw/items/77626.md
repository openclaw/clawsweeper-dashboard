## Summary

Keep open: current main still uses a single overall agent-turn timeout for embedded reply runs, while this request changes public timeout semantics and adds new config plus notification UX that needs maintainer product and architecture review.

## What This Changes

Request liveness-resetting per-step turn timeouts, an absolute max-turn guardrail, progress notifications, and clearer timeout reporting for long tool-heavy agent turns.

## Best Possible Solution

Design and document a core liveness contract with explicit idle reset events, a separate absolute guardrail, channel-aware progress delivery, clear timeout errors, and regression coverage across embedded runs and at least one chat channel.

## Reproduction Assessment

No high-confidence end-to-end Telegram reproduction was run. Source inspection does confirm the current embedded reply path uses one overall abort timer rather than a per-step liveness-resetting timeout window.

## Solution Assessment

Unclear. The direction is plausible, but reinterpreting timeoutSeconds is not obviously the safest migration path; maintainers should decide whether to use additive config, opt-in semantics, or a staged compatibility path.
