## Summary

Keep open: current main still implements a single overall agent-run timeout, while this request asks for new liveness semantics, new config keys, and channel notification policy that need maintainer product and architecture review.

## What This Changes

Request liveness-based per-step turn timeouts, a hard turn ceiling, clearer timeout reporting, and progress notifications for long agent turns.

## Best Possible Solution

Design a core agent-run liveness contract with explicit reset events, an absolute timeout guardrail, channel-aware progress delivery, docs, and focused regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because current docs and code define `timeoutSeconds` as an overall runtime timeout. Source inspection does confirm the requested liveness-reset behavior is not present on current main.

## Solution Assessment

Unclear. The proposed direction is plausible, but the exact config shape, default behavior, migration from existing `timeoutSeconds`, and notification policy need maintainer design before implementation.
