## Summary

Keep open: current main still uses a single run deadline, while this request asks for new timeout semantics, config, and channel notification UX that need maintainer design before implementation.

## What This Changes

Request liveness-based per-step agent turn timeouts, progress notifications for long steps, and clearer timeout reporting.

## Best Possible Solution

Design a backward-compatible core liveness contract with an explicit absolute guardrail, well-defined reset events, channel-aware progress delivery, docs, and focused regression coverage across chat.send, agent RPC, and the embedded runner.

## Risks / Open Questions

- Changing `timeoutSeconds` semantics could let active turns run far longer than operators expect unless an absolute ceiling and migration story are designed first.
- Periodic progress notices could duplicate existing tool summaries, typing indicators, and Telegram progress drafts without channel-aware throttling.
- The liveness boundary must align with model idle timeouts, subprocess/tool execution, diagnostics, and compaction grace so real hangs are not hidden by unrelated progress events.
