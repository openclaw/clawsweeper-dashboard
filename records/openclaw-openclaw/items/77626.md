## Summary

Keep open: current main still enforces a single configured run deadline, while this request asks for new liveness semantics, config keys, progress UX, and graceful timeout reporting that need maintainer design.

## What This Changes

The issue requests per-step liveness timeouts, explicit hard turn ceilings, progress notifications, and graceful timeout messages for long agent turns.

## Best Possible Solution

Design a backward-compatible core liveness contract with explicit reset events, an absolute guardrail, channel-aware progress delivery, docs, and focused regression coverage across chat.send, agent RPC, diagnostics, and the embedded runner.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence way to verify the current behavior: chat.send and agent RPC register one wall-clock expiry, and the embedded runner schedules one total-run abort timer; no live Telegram end-to-end run was performed.

## Solution Assessment

Unclear. The direction is plausible, but the exact reset events, defaults, migration behavior, diagnostics interaction, and channel notification policy need maintainer product and architecture decisions before a narrow implementation is safe.

## Risks / Open Questions

- Changing timeoutSeconds semantics could let active turns run much longer than operators expect unless an absolute ceiling and migration story are designed first.
- Periodic progress notices could duplicate existing tool summaries, typing indicators, streaming drafts, and channel rate-limit behavior without channel-aware throttling.
- The liveness boundary needs to align with model idle timeouts, subprocess/tool execution, diagnostics stuck-session recovery, and compaction grace so real hangs are not masked by unrelated progress.
