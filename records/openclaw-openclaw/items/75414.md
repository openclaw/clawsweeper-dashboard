## Summary

Keep open. Current main has partial mitigations for transcript size and async trajectory appends, but the reported Gateway event-loop starvation class is still present: compaction checkpoint capture still performs synchronous filesystem work on session transcripts, successor transcript rotation still serializes the full retained transcript in one event-loop turn, and QQBot heartbeats plus cron scheduling still share the Gateway event loop. The related open PR #75595 is the right implementation candidate to review, not a reason to close before merge.

## What This Changes

The issue reports long Gateway event-loop stalls during large session compaction/trajectory handling that cause QQBot WebSocket heartbeat timeouts and cron failures, and asks for nonblocking transcript handling plus guardrails.

## Best Possible Solution

Land PR #75595 or an equivalent focused runtime fix that removes synchronous transcript checkpoint/copy work from Gateway compaction hot paths, bounds or chunks successor transcript serialization, and adds a synthetic large-transcript liveness regression plus QQBot heartbeat and cron validation notes.

## Reproduction Assessment

Yes. The report gives a concrete field signature, and current code shows a 20.5 MiB transcript can enter the synchronous checkpoint-copy path while QQBot heartbeats and cron share the Gateway event loop; the focused repro is to trigger compaction with a large active transcript and observe `eventLoopDelayMaxMs` plus QQBot 4009/1006 behavior.

## Solution Assessment

No. Current main's size-triggered preflight compaction and async trajectory append path are useful mitigations, but the narrow maintainable fix is to make the compaction transcript I/O and serialization path nonblocking enough that Gateway timers are not starved.
