## Summary

Keep open. Current main has useful partial mitigations for oversized active transcripts and trajectory recording, but it still does not prove the reported Gateway event-loop starvation class is fixed because QQBot heartbeats share the Gateway event loop and successor transcript rotation still performs full in-process JSON serialization before writing.

## What This Changes

The issue requests a liveness fix so large session or trajectory compaction does not block the Gateway event loop, drop QQBot WebSocket heartbeats, or disrupt cron jobs.

## Best Possible Solution

Keep this open and land a focused liveness fix that makes large-session compaction and transcript rotation nonblocking for the Gateway event loop, with a synthetic large-transcript/event-loop-delay regression and QQBot/cron validation notes. Reconnect handling should remain a fallback, not the primary fix.

## Reproduction Assessment

Yes. The issue gives concrete steps, file sizes, close codes, and a liveness warning signature; current code inspection confirms the QQBot heartbeat timer shares the Gateway event loop and successor transcript rotation can still do full synchronous serialization.

## Solution Assessment

No. Current main's byte threshold, async queued trajectory appends, mid-turn precheck, and reconnect handling are useful mitigations, but they do not prove the Gateway event loop is protected from large compaction/session persistence stalls.
