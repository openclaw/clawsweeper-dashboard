## Summary

Keep open. Current main has partial mitigations for oversized active transcripts and queued trajectory writes, but the reported Gateway event-loop starvation class is still plausible: QQBot heartbeats and cron both live in the Gateway process, and compaction successor rotation still performs full in-memory JSON serialization before the async file write. The earlier ClawSweeper comment remains consistent with current main because the central files have not changed since that review.

## What This Changes

The issue asks OpenClaw to prevent large session compaction or trajectory persistence from blocking the Gateway event loop and causing QQBot WebSocket disconnects plus missed cron work.

## Best Possible Solution

Land a focused liveness fix that bounds or isolates large-session compaction and successor transcript persistence so Gateway timers remain responsive, then prove it with a large-transcript event-loop-delay regression and QQBot/cron validation notes. Reconnect handling should remain a fallback, not the primary fix.

## Reproduction Assessment

Yes. The issue gives concrete field steps and a precise liveness signature, and current code inspection shows a high-confidence path: large-session compaction can synchronously serialize successor transcript content while QQBot heartbeat timers and cron scheduling share the Gateway event loop.

## Solution Assessment

No. Current main's byte threshold, queued trajectory appends, and reconnect handling are useful mitigations, but they are not the best complete solution because they do not prove compaction/session serialization is bounded or off the Gateway event loop.
