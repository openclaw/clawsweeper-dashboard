## Summary

Keep open. Current main still materializes plugin tools by calling registered factories during per-turn tool resolution, and open PR #75974 explicitly targets this issue, so this should stay open until that PR is reviewed and merged or replaced.

## What This Changes

The issue asks to reduce native-Windows Gateway per-turn latency by avoiding uncached plugin tool factory calls in the `core-plugin-tools` stage.

## Best Possible Solution

Ship a context-safe plugin tool materialization strategy, with focused regression coverage and a changelog entry, then close this issue through #75974 or a canonical replacement PR.

## Reproduction Assessment

Yes. The issue includes concrete native-Windows steps and timing traces, and current main still has the same per-resolution factory call path; live Windows verification was not performed here.

## Solution Assessment

Unclear. Avoiding repeated factory work is the right target, but a cache keyed only by plugin id or tool names is not clearly safe because factories receive per-run context.
