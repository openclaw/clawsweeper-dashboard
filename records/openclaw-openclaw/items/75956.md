## Summary

Keep open. Current main still invokes plugin tool factories during each `resolvePluginTools` pass, the current docs only add timing diagnostics, and open PR #75974 explicitly targets this issue but has not merged.

## What This Changes

Report requests reducing Windows Gateway per-turn latency by avoiding uncached per-request plugin tool factory calls in `resolvePluginTools`.

## Best Possible Solution

Keep this open until #75974 is reviewed and landed or replaced by a context-safe plugin tool preparation change with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The issue provides concrete native-Windows timing traces and current main still executes the factory call on each plugin tool resolution; this review verified the source path but did not run Windows live testing.

## Solution Assessment

Unclear. The requested latency target is right, but the pluginId-only cache proposed in the issue and PR needs design review because current factories receive per-run context.
