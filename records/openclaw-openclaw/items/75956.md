## Summary

Keep open. Current `main` still resolves plugin tools by calling each registered factory during `resolvePluginTools`, so the reported per-turn hot path is not implemented away; the existing tracing helps diagnosis but does not cache or defer factory materialization.

## What This Changes

The issue reports severe native-Windows gateway latency caused by uncached plugin tool factory calls during per-message tool resolution.

## Best Possible Solution

Add a context-safe plugin tool preparation strategy in the plugin runtime, likely by separating stable tool definitions from per-run bindings or adding an explicit cacheability contract, then cover the hot path with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The issue provides a concrete native-Windows trace showing `core-plugin-tools` at 78.58s, and current source confirms the same stage still calls plugin factories on each resolution; live Windows verification was not performed in this read-only sweep.

## Solution Assessment

Unclear. The proposed cache keyed only by plugin id and tool names is not safely sufficient because factories receive per-run context; a narrower cacheability contract or lazy materialization split would be safer.
