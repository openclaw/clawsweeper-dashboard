## Summary

Keep open. Current main narrows plugin-tool discovery and adds timing diagnostics, but `resolvePluginTools` still materializes scoped plugin tools by calling each factory during every resolution, so the remaining uncached hot path has not been designed away.

## What This Changes

Request to cache or defer plugin tool factory materialization so native Windows Gateway turns stop spending tens of seconds in `core-plugin-tools`.

## Best Possible Solution

Add a context-safe plugin tool preparation strategy, likely by separating stable tool definitions from per-run bindings or adding an explicit cacheability contract, then cover the hot path with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The issue provides a concrete native-Windows trace, and current source gives a high-confidence source-level reproduction: call `resolvePluginTools` twice with a counted factory and the factory is invoked on each resolution.

## Solution Assessment

No for the naive cache sketched in the issue. A cache keyed only by plugin id and tool names is not the safest fix because factories receive dynamic per-run context; a lazy-definition or explicit cacheability contract is more maintainable.
