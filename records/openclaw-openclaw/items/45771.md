## Summary

Keep open because current main has provider usage-window visibility and static subagent fan-out guards, but it still lacks pace-aware autonomous-agent control: no pace tiers, projected end-of-window usage, Anthropic response-header capture, persisted per-profile pace state, prompt-context policy, notifications, or throttling. Related open rate-limit, usage-aware routing, and failover trackers overlap but do not supersede this narrower 5h/7d autonomous-agent pace design.

## What This Changes

The request is to capture Anthropic 5h/7d quota utilization, project consumption pace, expose it in status/session tooling, and use it to guide or throttle autonomous agent work.

## Best Possible Solution

Keep this as a maintainer design tracker; a good first slice would add provider-owned structured usage freshness and pace projection for status/session visibility, then separately design prompt guidance and enforcement around persistence and cost calibration.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source/docs inspection on current main, which shows usage-window reporting and static concurrency limits but not the requested pace policy surfaces.

## Solution Assessment

Unclear for the full proposal. Usage/header capture plus status/session visibility is a maintainable first slice, while automatic throttling should wait for provider-aware cost calibration and resume-safe agent UX.
