## Summary

Keep open. Current main has provider usage snapshots and status visibility, but it does not implement the requested pace projection, Anthropic response-header capture, persistent pace state, agent prompt injection, or autonomous/subagent throttling; the related open rate-limit, failover, and routing issues are adjacent rather than exact superseders.

## What This Changes

Requested built-in pace-aware Anthropic usage tracking, status exposure, agent context, and autonomous throttling based on 5-hour and 7-day utilization windows.

## Best Possible Solution

Define a maintainer-scoped core pace policy that reuses provider usage snapshots, adds an explicit freshness/state contract, exposes status/session_status data, and decides opt-in warning/throttle behavior for autonomous work before implementation.

## Reproduction Assessment

Not applicable; this is a feature proposal, not a regression. Source inspection gives high confidence that current main lacks the requested pace config, header capture, projection, and autonomous throttling while only partially covering usage visibility.

## Solution Assessment

Unclear as a single patch. The direction likely belongs in core agent/provider policy rather than only in the existing skill, but maintainers should first settle the config, freshness, and throttle semantics.
