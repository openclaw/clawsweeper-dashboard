## Summary

Keep open. Current main has provider usage-window visibility and static subagent fan-out guards, but it does not implement pace-aware autonomous-agent policy, projected 5h/7d tiers, persisted per-profile pace state, prompt/session structured pace context, notifications, or throttling based on consumption pace. Related rate-limit and failover trackers overlap but do not fully supersede this narrower pace-control design.

## What This Changes

Add built-in pace-aware rate-limit tracking for autonomous agents by capturing Anthropic utilization data, projecting 5h/7d consumption, surfacing pace in status/session tools, and optionally throttling subagents or non-interactive work.

## Best Possible Solution

Keep this as a maintainer design tracker for a provider-aware pace model. A maintainable first slice would add structured usage source/freshness and status/session visibility, keep Anthropic-specific capture inside the Anthropic provider path, and defer enforcement until per-profile persistence, request-cost calibration, and subagent UX are settled.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows usage-window reporting and static subagent limits exist but the requested pace policy surfaces do not.

## Solution Assessment

Unclear for the full proposal. Header or usage snapshot capture plus status/session visibility is a reasonable first slice, but automatic throttling should be designed around provider windows, observed request cost, and autonomous-agent UX before it becomes enforcement policy.
