## Summary

Keep open. Current main has provider usage snapshots, status usage display, and reactive cooldown/fallback handling, but it still lacks the requested pace projection, normal Anthropic model-response header capture, persistent pace state, prompt/session_status pace guidance, and autonomous/subagent throttling policy.

## What This Changes

The issue requests built-in pace-aware rate-limit tracking and autonomous-agent throttling based on Anthropic 5-hour and 7-day utilization windows.

## Best Possible Solution

Define a provider-aware pace contract that extends usage snapshots with freshness and projection data, expose it in status/session_status and prompt context, then add narrowly scoped throttle behavior only after the policy is accepted.

Not applicable; this is a feature proposal rather than broken existing behavior. Source inspection gives high confidence that current main lacks the requested pace tiers, response-header capture, persisted pace state, prompt/session_status guidance, and throttle policy.

## Solution Assessment
