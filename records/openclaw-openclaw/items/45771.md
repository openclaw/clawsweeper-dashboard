## Summary

Keep open. Current main and v2026.6.8 have provider usage readouts plus reactive rate-limit cooldown/fallback handling, but not built-in pace tiers, pace context in `session_status`, or configurable autonomous/subagent throttling; the open related trackers overlap but do not fully replace this narrower Anthropic/autonomous-agent design.

## What This Changes

Request built-in pace-aware rate limiting for autonomous agents using Anthropic usage/reset signals, status visibility, session tool context, and optional throttling controls.

## Best Possible Solution

Keep this open for a maintainer-scoped design that builds on existing provider usage visibility, then adds an opt-in generic pace policy for status/session context and autonomous throttling only after provider and session-state semantics are settled.

## Reproduction Assessment

Yes, at source level: current main has usage readouts and reactive rate-limit handling, but lacks the requested pace tier, session_status context, and autonomous throttling surfaces. This is a feature gap, not a failing existing-behavior bug reproduction.

## Solution Assessment

Unclear. The direction is plausible, but the proposed automatic throttling/config shape is not yet the best confirmed implementation; maintainers should first choose a generic provider-usage pace contract and opt-in failure modes.
