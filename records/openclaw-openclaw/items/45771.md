## Summary

Keep open: current main and v2026.6.6 already expose provider usage windows in status/session surfaces, but they do not implement projected pace tiers, persisted pace state, prompt/context injection, or autonomous/subagent throttling; the discussion and labels point to a product/API decision rather than conservative cleanup.

## What This Changes

Request to add provider-usage pace tracking that projects Anthropic quota windows and feeds status, session_status, prompts, and autonomous/subagent throttling.

## Best Possible Solution

Design a maintainer-approved provider-agnostic pace signal on top of the existing usage hooks, then add opt-in prompt/status exposure and upgrade-safe config only if core owns enforcement.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current behavior with an expected contract. Source inspection verifies that current main lacks the requested pace policy/config/throttling, but there is no failing current-main repro to run.

## Solution Assessment

Unclear: the visibility portion is already implemented, but converting provider utilization into autonomous throttling and prompt context needs maintainer product/API decisions. A safer direction is provider-agnostic pace/status exposure first, with explicit opt-in enforcement after the policy is settled.
