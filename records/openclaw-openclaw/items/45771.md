## Summary

Keep open. Current main has normalized provider usage snapshots and reactive cooldown/fallback behavior, but it still does not implement pace projection, response-header capture from normal Anthropic model calls, persistent pace state, prompt/session_status pace guidance, or autonomous/subagent throttling policy.

## What This Changes

Request built-in pace-aware rate-limit tracking, status/session_status exposure, optional agent pace config, and autonomous throttling for long-running agents.

## Best Possible Solution

Define an opt-in provider/agent pace contract that extends usage snapshots with freshness and projection data, exposes it in status/session_status and prompt context, and only then adds carefully scoped throttle behavior.

## Reproduction Assessment

Not applicable; this is a feature proposal rather than broken existing behavior. Source inspection gives high confidence that current main lacks pace tiers, response-header capture, persistent pace state, prompt exposure, and throttle policy.

## Solution Assessment

Unclear as a single patch. Usage visibility is already partly present, but the remaining pace and throttling design needs maintainer decisions around provider semantics, prompt exposure, persistence, cost modeling, and autonomous interruption policy.
