## Summary

Keep open: the PR targets a real WebChat transcript persistence bug, but the new duplicate guard can append a second assistant turn for long Pi-written JSONL records, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds an embedded-runner assistant transcript repair helper, exports it from the attempt runtime barrel, wires it from `agent-command.ts`, and adds a changelog entry for #77518.

## Best Possible Solution

Use a transcript-aware embedded fallback that checks complete active transcript records under the session write lock, appends only a missing current assistant turn, preserves metadata, and covers both missing-turn repair and long-message no-duplicate cases.

## Reproduction Assessment

Yes for the PR regression by source inspection: a normal embedded run whose Pi-written final assistant JSONL record exceeds the 4096-byte tail window will make the new guard parse a partial line and append a duplicate. The original WebChat loss remains source- and incident-supported, but I did not live-run WebChat in this read-only pass.

## Solution Assessment

No. The embedded fallback direction is reasonable, but a fixed-size tail parse is not a reliable duplicate check; the safer solution is to inspect complete transcript records or use an active-branch transcript helper before appending.

## Review Findings
