## Summary

Keep open: the performance direction is useful, but the patch is not merge-ready because the new pending-block latch can be cleared while an idle wait is still in flight, and the external PR has no required real behavior proof.

## What This Changes

The branch changes auto-reply and gateway send paths to stop awaiting direct block dispatcher idle and source transcript mirroring in the hot path, with regression tests for delayed block delivery and async source mirroring.

## Best Possible Solution

Keep the non-blocking reply hot path, but use a shared pending-idle promise or latch that every concurrent tool/progress path awaits until the prior block delivery drains, and prove the async transcript mirror behavior in a real run.

Yes for source-level behavior: current main visibly awaits block dispatcher idle and source transcript mirroring, and the PR diff moves those waits off the hot path. I did not run the patched tests because this review is read-only.

Full review comments:

- **[P2] Keep the pending block wait latched while idle is in flight:** `src/auto-reply/reply/dispatch-from-config.ts:1889-1890`
  - body: The helper clears `hasPendingDirectBlockReplyDelivery` before `waitForReplyDispatcherIdle` finishes.   - confidence: 0.88
- **[P2] Apply the same pending wait latch to ACP tool delivery:** `src/auto-reply/reply/dispatch-acp-delivery.ts:242-243`
  - body: The ACP coordinator has the same pre-cleared pending flag. Concurrent `deliver("tool")` calls after a direct block can let a later tool delivery resolve without waiting for the prior block drain, weakening the block-before-tool ordering this PR says it preserves.
  - confidence: 0.8
