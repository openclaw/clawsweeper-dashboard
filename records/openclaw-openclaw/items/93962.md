## What This Changes

## Best Possible Solution

Preserve the registration-time warning, but recursively validate availability expressions or make the evaluator defensive before descriptor capture evaluates plugin-authored data.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass a plugin tool with availability like { allOf: [{ anyOf: "not-array" }] } through capturePluginToolDescriptor and the PR reaches evaluateToolAvailability, where current evaluator code still calls .map on the nested non-array value.

## Solution Assessment

No. Descriptor capture is a reasonable boundary for the warning, but the implementation is not the best fix until nested plugin-authored availability shapes are validated or the evaluator is defensive.
