## Summary

Keep open: current main and v2026.5.22 still route opt-in LLM slug generation through the configured default agent model and embedded runner lane. The default blast radius is lower now because LLM slugs are opt-in and session-memory capture is backgrounded, but the requested lightweight slug-routing policy remains unresolved and the related implementation PR is still open.

## What This Changes

The issue requests routing LLM-generated session-memory slugs through a lightweight model or fallback path instead of the agent primary model to avoid lane congestion.

## Best Possible Solution

Land or replace the focused slug-generation policy in https://github.com/openclaw/openclaw/pull/65134 so optional descriptive slugs avoid embedded-runner/default-primary routing while preserving timestamp defaults, background hook behavior, timeout/fallback behavior, and auth/provider contracts.

## Reproduction Assessment

Yes, at source level: enable `hooks.internal.entries.session-memory.llmSlug`, and session memory reaches `generateSlugViaLLM`, which resolves the default model and runs `runEmbeddedPiAgent` on `temp:slug-generator`. I did not live-reproduce the reported multi-agent queue buildup.

## Solution Assessment

Unclear. A dedicated slug model config addresses the reported deployment need, but maintainers still need to choose whether the durable policy is a new config key, heartbeat-model reuse, direct simple completion, a shorter heuristic fallback, or a combination.
