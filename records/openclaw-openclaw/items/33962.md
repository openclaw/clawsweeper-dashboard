## Summary

Keep this issue open. Current main still routes session-memory LLM slug generation through the default/effective agent model and the embedded runner session lane, so the reported cost and congestion problem remains; the linked #65134 PR is an open implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request to decouple session-memory slug generation from the agent primary model by using a lightweight slug-specific model, faster fallback, or debounce so resets do not congest the embedded runner lane.

## Best Possible Solution

Move slug generation off the full embedded-agent run path and its temp session lane, use a simple completion or approved lightweight model-selection contract, preserve `llmSlug: false` plus timestamp fallback behavior, and add regression coverage proving slug generation no longer queues on `session:temp:slug-generator`.

## Reproduction Assessment

Yes for the reported root cause: with session-memory enabled and `llmSlug` not set to false, current main awaits `generateSlugViaLLM`, which uses the effective agent model and `runEmbeddedPiAgent` with `temp:slug-generator`. The precise multi-agent queue duration is not live-reproduced here, but the lane behavior is directly visible in the queue implementation and reporter logs.

## Solution Assessment

Unclear whether the proposed dedicated `slugModel` config is the best final API, but increasing only the timeout is not enough. The narrower maintainable direction is to avoid the full embedded runner for this helper and select a cheap/simple model path with focused tests.
