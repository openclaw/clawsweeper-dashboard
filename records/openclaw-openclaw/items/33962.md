## Summary

Keep open: current main and v2026.6.10 still route opt-in LLM slug generation through the default agent model and embedded-agent lanes, while the supported lightweight model or fallback contract still needs maintainer product review.

## What This Changes

The issue requests a lightweight or dedicated model path, faster fallback, or debounce for LLM session-memory slug generation instead of always using the agent primary model.

## Best Possible Solution

Adopt one maintainer-approved slug model-selection and fallback contract, preferably reusing an existing model-selection surface if sufficient, then cover current-main and upgrade behavior with focused tests and docs.

## Reproduction Assessment

Yes for the source-level routing path: enabling `session-memory` with `llmSlug: true` still reaches `generateSlugViaLLM()`, which resolves the default agent model. No for the full live congestion claim: this review did not run a multi-agent reset scenario with a slow primary model.

## Solution Assessment

Unclear: the problem is real enough to keep open, but the requested dedicated config key is not yet proven to be the best permanent surface. Maintainers should choose between hook-level model selection, a generic lightweight-task model policy, direct completion, and fallback/debounce semantics before code lands.
