## Summary

Keep open: current main mitigates the default reset hot path by making LLM slugs opt-in and backgrounded, but opt-in descriptive slug generation still uses the configured/default agent model through the embedded runner instead of a lightweight slug-specific path.

## What This Changes

The issue requests moving session-memory LLM slug generation off the primary agent model, preferably to a lightweight dedicated model or fast fallback, to avoid slug-generator lane congestion during resets.

## Best Possible Solution

Land or replace the focused direct-completion/model-routing work so opt-in descriptive slugging avoids the embedded agent lane, preserves fast fallback semantics, and documents the chosen lightweight model policy.

## Reproduction Assessment

Yes for the source-level behavior: when llmSlug is enabled, current main still resolves the configured/default agent model and runs the embedded slug session. I did not establish a live multi-agent queue reproduction in this read-only pass.

## Solution Assessment

No. Current main's opt-in/background mitigation is useful, but it does not implement the requested lightweight slug model or direct-completion routing; the open related PR is the concrete path to review or replace.
