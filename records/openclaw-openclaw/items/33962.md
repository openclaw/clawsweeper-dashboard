- `impact:message-loss`: The report and follow-up production comment tie slug-generator lane congestion to delayed or dropped user-facing messages during resets.

## Summary

Keep open: current main and v2026.5.20 still route opt-in LLM slug generation through the configured default agent model and embedded runner. The default path is safer now because LLM slugs are opt-in and backgrounded, but the lightweight slug-routing policy remains unresolved and https://github.com/openclaw/openclaw/pull/65134 still needs maintainer review, compatibility work, and proof.

## What This Changes

The issue requests routing session-memory LLM slug generation to a lightweight model or fallback path instead of the agent primary model to avoid lane congestion.

## Best Possible Solution

Settle and land one focused slug-generation policy that avoids embedded-runner/default-primary routing for optional descriptive slugs while preserving timestamp defaults, background hook behavior, timeouts, fallback behavior, and auth/provider runtime contracts.

## Reproduction Assessment

Yes, at source level: enable `hooks.internal.entries.session-memory.llmSlug`, and session memory reaches `generateSlugViaLLM`, which resolves the default model and runs `runEmbeddedPiAgent` on `temp:slug-generator`. I did not live-reproduce the reported multi-agent lane buildup.

## Solution Assessment

Unclear. Direct simple completion and heartbeat-model reuse are plausible, but maintainers still need to choose whether the right policy is a dedicated slug model key, an existing lightweight model knob, a shorter fallback, or a combination.
