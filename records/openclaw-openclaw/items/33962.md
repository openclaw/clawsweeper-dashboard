## Summary

Keep open: current main and v2026.5.12 mitigate the default blocking path by making LLM slugs opt-in and backgrounded, but the enabled slug path still resolves the configured default agent model and runs through `runEmbeddedPiAgent`; the related open PR is not merged and the lightweight routing policy still needs maintainer choice.

## What This Changes

Request lightweight or dedicated model routing for session-memory LLM slug generation so trivial filename generation does not compete with primary user-facing model traffic.

## Best Possible Solution

Settle and land one focused slug-generation policy that avoids embedded-runner/default-primary routing for optional descriptive slugs while preserving timestamp defaults, background hook behavior, timeout fallback, auth/provider runtime contracts, and focused hook coverage.

## Reproduction Assessment

Yes at source level: enable `hooks.internal.entries.session-memory.llmSlug`, and session memory reaches `generateSlugViaLLM`, which resolves the default model and runs `runEmbeddedPiAgent` on `temp:slug-generator`. I did not live-reproduce the reported multi-agent lane buildup.

## Solution Assessment

Unclear. Direct simple completion and heartbeat-model reuse are plausible, but maintainers still need to choose whether the right policy is a dedicated slug model key, an existing lightweight model knob, a shorter fallback, or some combination.
