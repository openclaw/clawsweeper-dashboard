## Summary

Keep open: current main and v2026.6.6 still route opt-in LLM slugs through the default agent model and embedded-agent runner, while the remaining fix requires a maintainer decision on the lightweight slug-model/config contract.

## What This Changes

Requests a lightweight or dedicated model/fallback policy for LLM session-memory slug generation so opt-in slugging does not queue on the primary agent model.

## Best Possible Solution

Adopt a maintainer-approved lightweight slug model path using canonical alias-aware model resolution, document and doctor-check the config contract, preserve timestamp fallback, and avoid a slug-only config if a generic lightweight-task model policy can cover it.

## Reproduction Assessment

Yes for the source-level routing path: current main resolves slug provider/model through resolveDefaultModelForAgent and runs temp:slug-generator through runEmbeddedAgent. No live current-main multi-agent congestion repro was established in this review.

## Solution Assessment

No: a dedicated slugModel key is not clearly the best permanent solution. Maintainers should first choose whether the supported contract is hook-level model selection, a generic lightweight-task model policy, direct completion, or some combination.
