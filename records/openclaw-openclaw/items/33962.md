## Summary

Keep open: current main and the latest release still route opt-in LLM slug generation through the default agent model and embedded session lane. Timestamp slugs and configurable timeouts reduce the default pain, but the requested lightweight model/routing policy remains unresolved and the related open PR is not merged.

## What This Changes

Request to route session-memory LLM slug generation to a lightweight or dedicated model, or a fast fallback, instead of the agent primary embedded run.

## Best Possible Solution

Land a focused slug-generation policy that avoids embedded-runner/default-primary routing for opt-in descriptive slugs, preserves timestamp defaults and timeout/fallback behavior, and updates focused tests plus user-facing hook docs.

## Reproduction Assessment

Yes, at source level: enable `hooks.internal.entries.session-memory.llmSlug` and current main calls `generateSlugViaLLM`, resolves the default agent model, and runs the embedded agent on `temp:slug-generator`. I did not live-reproduce the multi-agent queue buildup.

## Solution Assessment

Unclear. Direct simple completion and heartbeat-model reuse are plausible, but maintainers still need to choose the model-routing policy before deciding whether to add dedicated slug config or reuse an existing lightweight-model knob.
