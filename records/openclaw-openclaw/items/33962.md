## Summary

Keep this issue open. Current main still routes session-memory LLM slug generation through the effective/default agent model and the full embedded runner on `session:temp:slug-generator`; the related #65134 PR is still an open implementation candidate, not a reason to close the issue before merge.

## What This Changes

Request to move session-memory slug generation away from the agent primary model/full embedded runner toward a lightweight model or faster non-congesting fallback.

## Best Possible Solution

Review and land or replace the #65134-style fix that moves slug generation off the full embedded-agent session lane, selects an approved cheap/simple model path, preserves `llmSlug: false` plus timestamp fallback behavior, and adds regression coverage for the new path.

## Reproduction Assessment

Yes for the source-level root cause: current main awaits `generateSlugViaLLM`, which uses the effective/default agent model and `runEmbeddedPiAgent` on `temp:slug-generator`. I did not live-run a seven-agent restart or reproduce the exact queue duration.

## Solution Assessment

Unclear whether the requested dedicated `slugModel` config is the best final API. The narrower maintainable direction is to avoid the full embedded runner for this helper and use a simple completion or approved lightweight model-selection contract instead of only extending timeouts.
