## Summary

Keep open: current main and v2026.6.8 still route opt-in LLM slug generation through the default agent model and embedded-agent queue path, while the requested lightweight slug model behavior still needs a maintainer-owned config/product decision.

## What This Changes

Request to keep opt-in session-memory LLM slug generation off the agent primary model, preferably via a lightweight slug model config plus faster fallback or debounce behavior.

## Best Possible Solution

Adopt one maintainer-approved slug model-selection contract with alias-aware resolution, focused tests, user-facing docs, timestamp fallback, and no extra config if an existing hook-level or generic lightweight-task policy is enough.

## Reproduction Assessment

Yes for the source-level routing path, but not for the full live congestion claim. Current main still resolves slug generation through resolveDefaultModelForAgent and queueing source shows temp:slug-generator is serialized through session/global lanes, but no live multi-agent reset proof was run.

## Solution Assessment

Unclear: a dedicated slugModel key is plausible, but not proven to be the best permanent surface. Maintainers should choose between hook-level model selection, a generic lightweight-task model policy, direct completion, and fallback/debounce semantics before code lands.
