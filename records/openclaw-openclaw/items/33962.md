## Summary

Keep open: current main and the latest tagged release still route opt-in session-memory LLM slug generation through the resolved default agent model and `runEmbeddedPiAgent`, so the lightweight model-routing request is not implemented. The linked open PR targets the right surface, but it is not merged and the remaining choice between a dedicated slug model, existing lightweight model reuse, direct simple completion, and fallback behavior still needs maintainer policy review.

## What This Changes

The issue requests routing session-memory LLM slug generation away from the agent primary model, preferably to a lightweight dedicated model or faster fallback, to avoid lane congestion.

## Best Possible Solution

Land a focused slug-generation policy that avoids embedded-runner/default-primary routing for opt-in descriptive slugs while preserving timestamp defaults, timeout/fallback behavior, runtime auth contracts, and focused hook tests/docs.

## Reproduction Assessment

Yes, at source level: enable `hooks.internal.entries.session-memory.llmSlug`, and session memory calls `generateSlugViaLLM`, which resolves the default model and runs `runEmbeddedPiAgent` on `temp:slug-generator`. I did not live-reproduce the multi-agent queue buildup.

## Solution Assessment

Unclear. Direct simple completion and lightweight-model reuse are plausible, but maintainers still need to choose the slug model-routing policy before adding a dedicated config key or reusing an existing model knob.
