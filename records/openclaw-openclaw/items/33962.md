## Summary

Keep open: current main and v2026.6.10 still route opt-in LLM slug generation through the default agent model and embedded-agent lanes. The narrower hook-model issue only partially overlaps, and its linked candidate PR is closed unmerged, so the lightweight slug-model and fallback policy still needs maintainer direction.

## What This Changes

The issue requests a lightweight or dedicated model path, fast fallback, or debounce behavior for session-memory LLM slug generation so trivial filename work does not congest the agent primary model lane.

## Best Possible Solution

Choose and document one supported slug model-selection and fallback contract, preferably reusing an existing hook or lightweight-task model surface if that solves the congestion without adding another config key.

## Reproduction Assessment

Yes for the source-level model routing and lane shape: current main still resolves slug generation through the agent default model and runs it through `temp:slug-generator`. No live multi-agent reset proof was run, so the full congestion or dropped-message outcome remains unverified.

## Solution Assessment

Unclear: the request identifies a real pressure point, but adding a dedicated `slugModel` key is not proven to be the best permanent surface under OpenClaw's high bar for new config and fallback behavior. A hook-level model override or generic lightweight-task model policy may be cleaner if maintainers accept that contract.
