## Summary

Keep this issue open. Current main still routes session-memory slug generation through the effective/default agent model and `runEmbeddedPiAgent` on the `session:temp:slug-generator` lane, so the reported lightweight-model/lane-congestion problem remains live; the open #65134 PR is a focused implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue requests moving session-memory slug generation away from the agent primary model toward a lightweight slug-specific model or fast fallback so concurrent resets do not congest agent lanes.

## Best Possible Solution

Make slug generation a cheap, non-blocking helper path: avoid `runEmbeddedPiAgent` and `session:temp:slug-generator`, use `completeSimple` or an approved lightweight slug-model selection contract, preserve `llmSlug: false` and timestamp fallback behavior, and add regression coverage proving concurrent slug generation no longer queues on the embedded-runner temp session lane. If maintainers add a new `slugModel` config surface, align schema, help text, docs, and tests in the same change.
