## Summary

Current main and the latest release still route opt-in LLM slug generation through the default agent model and full embedded-agent runner, so the requested lightweight/model-isolated path remains necessary. Related open PRs are partial or not merge-ready, so this issue should stay open for maintainer product review.

## What This Changes

The issue requests moving session-memory LLM slug generation off the agent primary model to a lightweight or otherwise isolated model path to reduce latency and lane contention.

## Best Possible Solution

Land a focused, maintainer-approved slug-generation change that preserves timestamp fallback, avoids a one-off config knob unless necessary, resolves model overrides through the existing config-aware model-selection path, and includes live slow-provider proof.

## Reproduction Assessment

Partly yes: source inspection gives a high-confidence path for the model-selection part because opt-in LLM slugging still uses `resolveDefaultModelForAgent` and `runEmbeddedAgent`. No: the reported multi-agent lane congestion and message drops still need live current-main proof with a slow local provider.

## Solution Assessment

No, the issue should not close yet and the proposed dedicated `slugModel` key should not be accepted blindly. The best path is a maintainer-approved model-routing rule, likely using existing hook/model-selection surfaces or direct simple completion, with docs and live proof.
