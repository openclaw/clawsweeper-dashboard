## Label Justifications

- `P2`: The opt-in helper can cause substantial latency in multi-agent and cold local-model setups, but timestamp slugs remain the default and reset acknowledgements are backgrounded.
- `impact:message-loss`: The report and confirming user account describe user-facing messages delayed for minutes or effectively suppressed during slug-lane congestion.
- `impact:auth-provider`: Slug generation consumes the default agent provider/model and participates in its timeout and failover behavior.

## Summary

Current main and v2026.6.6 still send opt-in LLM slug generation through the default agent model and full embedded-agent runner. Existing changes address timeout configuration and profile-local failure handling, but neither the cost/congestion problem nor the lightweight model policy is resolved; the closest linked PR remains open and conflicted.

## What This Changes

The issue requests isolating LLM session-slug generation from the agent’s primary model to reduce cost, latency, and lane congestion.

## Best Possible Solution

Replace the full embedded-agent slug run with a bounded direct completion using a maintainer-approved generic lightweight-task model policy, preserve timestamp fallback, and avoid a slug-only config key unless existing model-selection seams cannot express the requirement.

## Reproduction Assessment

Partly yes: current source proves the primary-model, shared-key, full embedded-run path, and two users supplied consistent operational logs. This review did not establish a failing current-main multi-agent reproduction of the claimed service-wide blockage.

## Solution Assessment

No, the proposed dedicated config key is not yet the narrowest maintainable solution. A direct completion path backed by a generic lightweight-task model policy would reduce overhead without creating another task-specific configuration source of truth.
