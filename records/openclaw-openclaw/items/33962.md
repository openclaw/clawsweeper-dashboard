## Summary

Keep open: current main and v2026.5.12 still route opt-in LLM slug generation through the default agent model and embedded session lane. The shipped timeout fix reduced one failure mode, and an open PR targets direct simple completion, but the lightweight model/routing policy remains unresolved.

## What This Changes

Request decoupling LLM-backed session-memory slug generation from the default agent primary model/lane by using a lightweight slug model, faster fallback, or debouncing.

## Best Possible Solution

Land a focused slug-generation policy that avoids embedded-runner/default-primary routing for opt-in descriptive slugs, preserves timestamp defaults and timeout/fallback behavior, and updates docs/tests.

## Reproduction Assessment

Yes, at source level: enable hooks.internal.entries.session-memory.llmSlug and current main calls generateSlugViaLLM, resolves the default agent model, and runs the embedded agent on temp:slug-generator. I did not live-reproduce the multi-agent queue buildup.

## Solution Assessment

Unclear. Direct simple completion is a plausible direction, but the final model policy needs maintainer resolution before deciding between a dedicated slug model config, heartbeat-model reuse, direct simple-completion runtime, or heuristic fallback.
