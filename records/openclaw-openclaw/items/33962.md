## Summary

Current main still sends optional LLM slug generation through a full embedded-agent run using the default agent model, so the requested optimization remains necessary. The related pull request is not yet a safe resolution because it is conflicting, includes unrelated failover-observation changes, and lacks real behavior proof.

## What This Changes

Move opt-in LLM session-memory slug generation to a lightweight model and completion path so filename work does not consume the primary agent execution path.

## Best Possible Solution

Land a focused slug-only change that uses direct lightweight completion, preserves deterministic timestamp fallback, and follows a maintainer-approved existing model-selection policy without adding a one-off slug configuration by default.

## Reproduction Assessment

Partly yes: enabling llmSlug reliably reaches a temporary full embedded-agent run using the default model. A high-confidence live reproduction of current-main message blockage was not established.

## Solution Assessment

No to a dedicated slugModel setting as the default solution; direct completion with an approved existing lightweight-task model policy is narrower and avoids expanding configuration.
