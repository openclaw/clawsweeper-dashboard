## Summary

Keep open: current main and the latest release still route opt-in LLM slug generation through the default agent model and embedded `session:temp:slug-generator` lane. The related open PR at https://github.com/openclaw/openclaw/pull/65134 targets the right surface but has not landed and previously needed proof plus patch repairs, so the lightweight model/routing policy remains unresolved.

## What This Changes

Request to route opt-in session-memory LLM slug generation away from the primary agent model to a lightweight or fallback path so resets do not congest model lanes.

## Best Possible Solution

Land a focused slug-generation policy that avoids embedded-runner/default-primary routing for opt-in descriptive slugs, preserves timestamp defaults and fallback behavior, and updates docs/tests.

## Reproduction Assessment

Yes, at source level: enable `hooks.internal.entries.session-memory.llmSlug` and current main calls `generateSlugViaLLM`, resolves the default agent model, and runs the embedded agent on `temp:slug-generator`. I did not live-reproduce the multi-agent queue buildup.

## Solution Assessment

Unclear. Direct simple completion and heartbeat-model reuse are plausible, but maintainers still need to choose the model-routing policy before deciding whether to add dedicated slug config, reuse existing heartbeat config, or prefer a heuristic fallback.
