## What This Changes

The PR derives `memory_search`'s outer deadline from `memory.qmd.limits.timeoutMs` for QMD searches and adds fake-timer regression tests for slow and stalled QMD searches.

## Best Possible Solution

Use the resolved active QMD backend config for the wrapper deadline, preserve the 15-second guard for non-QMD and wiki-only paths, keep bounded overhead, and add live QMD terminal/log proof before merge.

## Reproduction Assessment

Yes at source level. Current `main` wraps `memory_search` in a fixed 15-second deadline while the QMD manager uses `memory.qmd.limits.timeoutMs`; the linked report supplies a 46.7-second QMD rerank trace, though I did not run a live QMD binary.

## Solution Assessment

No, not yet. The fix direction is right, but the PR should derive the deadline from canonical resolved QMD configuration instead of duplicating raw timeout normalization, and live QMD proof is still missing.
