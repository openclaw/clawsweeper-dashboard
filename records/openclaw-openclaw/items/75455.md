## Summary

Keep open. Current main still has the BlueBubbles DM chat-identifier routing gap, but this PR should not merge as-is because it treats every DM `chatIdentifier` as a `chat_identifier` target, which bypasses the existing service-aware handle ranking for bare phone/email identifiers and can regress iMessage-vs-SMS routing; it also still needs the required changelog entry.

## What This Changes

The PR changes BlueBubbles inbound direct-message chat resolution to prefer an inbound `chatIdentifier` before falling back to `senderId`, and updates the focused resolver tests.

## Best Possible Solution

Update the BlueBubbles plugin so direct-message inbounds use the precise inbound chat identifier when it is a full chat GUID or stable non-handle chat identifier, but route bare phone/email identifiers through the existing handle target so iMessage-vs-SMS ranking remains intact; keep group behavior unchanged, add regression coverage, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main has a focused unit-level path showing DMs with `chatIdentifier` still resolve to a sender handle, and the PR regression can be reasoned from `resolveChatGuidForTarget`: bare `chatIdentifier` targets return on the first identifier match while handle targets preserve iMessage/SMS ranking.

## Solution Assessment

No, not as currently proposed. Preferring the inbound identifier is the right boundary, but the implementation is too broad for bare handle-like identifiers; a safer fix should classify identifier shape and keep those bare values on the service-aware handle path.
