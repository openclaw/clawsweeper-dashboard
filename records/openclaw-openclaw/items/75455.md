## Summary

Keep open. The PR addresses a real BlueBubbles DM fallback gap on current main, but as written it can regress iMessage-vs-SMS routing for bare phone/email chat identifiers and still lacks the required changelog entry.

## What This Changes

The PR changes BlueBubbles inbound direct-message chat-guid fallback to prefer an inbound `chatIdentifier` over `senderId` and updates the focused resolver tests.

## Best Possible Solution

Keep the PR open and update it so BlueBubbles DM fallback uses the inbound identifier without bypassing the existing handle service-ranking contract; then add a user-facing changelog entry and focused tests for both full-GUID and bare handle-like identifiers.

## Reproduction Assessment

Yes. Current main reproduces the original gap through the focused resolver helper/test, and the PR regression can be reproduced with a unit case where `chat_identifier` is a bare phone/email value while SMS appears before iMessage in the chat query results.

## Solution Assessment

No, not as currently proposed. Preferring the inbound identifier is the right direction, but handle-like identifiers should remain on the handle resolver path or otherwise preserve service ranking; blindly returning `chat_identifier` is not the narrowest safe fix.
