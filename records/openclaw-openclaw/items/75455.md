## Summary

Keep open. Current main still has the BlueBubbles DM fallback behavior the PR targets, but the proposed unconditional DM `chatIdentifier` preference can regress the existing iMessage-over-SMS routing for handle-like identifiers; the PR also still needs the required changelog entry.

## What This Changes

The PR changes BlueBubbles inbound direct-message chat resolution to prefer a provided chat identifier before falling back to the sender handle, with focused resolver tests.

## Best Possible Solution

Keep the fix in the BlueBubbles plugin, but make the DM fallback distinguish precise chat identifiers from handle-like identifiers so existing service-aware handle routing is preserved; then add focused regression coverage and the required `CHANGELOG.md` entry before merge.

## Reproduction Assessment

Yes. The original issue is reproducible from current main with the focused helper test: a DM inbound with a provided chat identifier still resolves to a sender-handle target. The PR regression is also reproducible by adding a targeted resolver case where a bare phone/email `chatIdentifier` encounters an SMS chat before an iMessage chat.

## Solution Assessment

No, not as proposed. Preferring `chatIdentifier` for DMs is the right direction for full GUID or opaque identifiers, but applying it to every nonblank identifier bypasses the existing handle path that intentionally preserves iMessage-over-SMS ranking.
