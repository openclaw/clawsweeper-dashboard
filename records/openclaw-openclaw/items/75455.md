## Summary

Keep open. The PR addresses a real current-main BlueBubbles DM routing gap, but the patch is not merge-ready because unconditional DM `chat_identifier` routing can bypass the existing handle-path iMessage-over-SMS ranking for bare handle-like identifiers; it also still needs the required changelog entry.

## What This Changes

The PR changes BlueBubbles inbound direct-message chat resolution to prefer a provided `chatIdentifier` before falling back to `senderId`, and updates the focused resolver tests.

## Best Possible Solution

Update the PR so BlueBubbles DMs prefer `chatIdentifier` only when it is a full service-qualified/opaque or sender-distinct identifier, keep bare sender-equivalent identifiers on the existing handle resolver path, preserve the group fail-closed behavior, add focused regression coverage, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction for the old DM fallback behavior, and the PR regression can be reproduced with a bare handle-like `chatIdentifier`: the new helper returns `chat_identifier`, whose resolver path can return an SMS GUID before the existing handle path's iMessage preference runs.

## Solution Assessment

No as written. The intended routing direction is sound, but unconditional `chatIdentifier` preference is broader than the safe fix; the maintainable solution should distinguish full/opaque or sender-distinct identifiers from bare handles and retain the established handle resolver for the latter.
