## Summary

Keep this PR open, but do not merge it as-is. The intended BlueBubbles DM fix is valid on current main, yet the branch routes every non-empty DM `chatIdentifier` through the `chat_identifier` resolver, which can bypass the existing iMessage-over-SMS handle ranking for bare phone/email identifiers; it also lacks the required user-facing changelog entry.

## What This Changes

The PR changes BlueBubbles inbound DM chat resolution and focused tests so direct-message webhooks prefer an inbound `chatIdentifier` before falling back to `senderId`.

## Best Possible Solution

Update the branch so DM fallback prefers precise inbound chat identity without bypassing handle service ranking: full GUIDs or clearly opaque chat identifiers can use the identifier path, while bare phone/email-like values should remain on the handle resolver path or receive equivalent service-aware ranking. Keep the existing group fallback protections unchanged and cover both full-GUID and bare-identifier cases in tests plus a changelog entry.

## Reproduction Assessment

Yes. Current main reproduces the original gap through the helper path that ignores DM `chatIdentifier`, and the PR regression is reproducible by a unit case where a bare `chatIdentifier` matches an SMS chat before an iMessage chat while the handle path would prefer iMessage.

## Solution Assessment

No, not as currently proposed. Preferring inbound chat identity is the right direction, but blindly returning `chat_identifier` for every non-empty DM identifier is not the narrowest maintainable fix because it drops the shipped service-ranking contract for handle-like identifiers.
