## Summary

Keep open: the feature direction is plausible, but the latest patch still uses ad hoc file-existence checks for plugin entry health, so it disagrees with current installed package entry resolution and boundary rules.

## What This Changes

The PR adds `openclaw doctor --post-upgrade [--json]` with plugin entry and manifest-drift probes, CLI/docs/changelog wiring, and focused tests.

## Best Possible Solution

Keep the new mode scoped, but make the post-upgrade probe reuse or exactly mirror the existing boundary-safe installed package entry resolver contract before merge.

## Reproduction Assessment

Yes. The blockers are source-reproducible: a package with `openclaw.extensions: ["./src/index.ts"]`, source present, and no built `dist` peer is rejected by current install validation but produces no PR finding; an escaping existing path is also treated as healthy by the PR check.

## Solution Assessment

No. The requested CLI direction is reasonable, but the implementation should use or share the existing package-entry resolver instead of re-encoding a weaker `fs.access` contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Mirror installed entry resolution before reporting health:** `src/commands/doctor-post-upgrade.ts:85-86`
  - body: When no `runtimeExtensions` entry is present, current install/loading logic checks inferred built runtime candidates and treats an installed TypeScript source entry without compiled JS as unhealthy. This branch only checks whether the source path exists, so `doctor --post-upgrade` can miss the TS-without-dist failure it is meant to catch and can still flag source-missing/dist-present packages that the loader can resolve.
  - confidence: 0.95
- **[P2] Reject entries that escape the plugin root:** `src/commands/doctor-post-upgrade.ts:68-69`
  - body: The new probe resolves `runtimeExtensions` with `path.resolve(record.rootDir, runtimeEntry)` and accepts any existing file, even an absolute or `../` path outside the plugin package. Current package validation rejects those paths, so this doctor mode can report an invalid plugin as healthy; reuse the boundary-safe package entry validation/resolution path for both runtime and source entries.
  - confidence: 0.9
