## Summary

The PR should stay open because the latest head still duplicates plugin entry validation with weaker direct filesystem checks, so it can miss invalid installed plugins and can follow entries outside the plugin package.

## What This Changes

The PR adds `openclaw doctor --post-upgrade [--json]` with plugin entry and manifest-drift probes, CLI/docs wiring, tests, and a changelog entry.

## Best Possible Solution

Keep the scoped post-upgrade mode, but implement `plugin.entry_unresolved` through the same boundary-safe package-entry resolver contract or a shared probe adapter before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an installed package with `openclaw.extensions: ["./src/index.ts"]`, source present, and no explicit or inferred compiled runtime output is rejected by current resolver code but produces no PR finding; escaping entries are likewise accepted when the target exists.

## Solution Assessment

No. The CLI direction is reasonable, but duplicating entry checks with direct `fs.access` is weaker than the current package-entry resolver; the narrow maintainable fix is to share or exactly mirror that resolver logic.

## Review Findings

- **[P2] Report source-only TypeScript package entries:** `src/commands/doctor-post-upgrade.ts:85-86`
  - body: This branch treats a package as healthy whenever the declared source entry exists on disk. Current install/discovery rejects installed package entries like `./src/index.ts` when no inferred or explicit compiled runtime exists, so the post-upgrade check can miss the plugin-load regression it is meant to report. Reuse the package-entry resolver or check built runtime candidates before accepting source entries.
  - confidence: 0.95

Absolute or `../` entries that point to an existing file outside the package will be reported healthy, while current validation rejects escaping entries; use the root-boundary helper or shared resolver before `fs.access`.
  - confidence: 0.92
