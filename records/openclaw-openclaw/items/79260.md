## Summary

Keep open: this is a useful CLI feature with supplied real behavior proof, but the patch currently reads the wrong plugin index for custom state directories and can false-flag valid packages that use `openclaw.runtimeExtensions`.

## What This Changes

The PR adds `openclaw doctor --post-upgrade` with JSON and human output for plugin entry and manifest-hash probes, plus tests and a changelog entry.

## Best Possible Solution

Keep the feature scoped, but route the probe through the existing plugin-index path and package-entrypoint contracts, document the new CLI flags, and rerun focused doctor/CLI validation before merge.

## Reproduction Assessment

Yes for the review blockers: current source and docs show the active state-dir and `runtimeExtensions` contracts, while the PR hardcodes `~/.openclaw` and checks only `openclaw.extensions`. I did not run tests because this review is read-only.

## Solution Assessment

No; the current approach duplicates plugin-index and package-entrypoint resolution instead of reusing the existing helpers. The safer solution is to call the established state-dir/index resolver and honor `runtimeExtensions` or built-runtime inference before emitting findings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Resolve the plugin index through the state-dir helper:** `src/commands/doctor.ts:9`
  - body: `doctor --post-upgrade` is supposed to inspect the active install, but this hardcodes `~/.openclaw/plugins/installs.json`. OpenClaw supports `OPENCLAW_STATE_DIR` and legacy/profiled state dirs through the installed-plugin-index resolver, so server, Nix, test, and profiled installs will read the wrong index or fail even though their real plugin index exists.
  - confidence: 0.93
- **[P2] Honor runtimeExtensions before flagging missing entries:** `src/commands/doctor-post-upgrade.ts:60-63`
  - body: This checks only `openclaw.extensions` on disk. Valid packages may declare source entries such as `./src/index.ts` and runtime entries such as `./dist/index.js`; current install/discovery accepts that and does not require the source file to be present when the runtime entry exists. Those packages would now get a false `plugin.entry_unresolved` error after upgrade.
  - confidence: 0.91
- **[P3] Document the new doctor mode:** `src/cli/program/register.maintenance.ts:28-33`
  - body: The PR adds public `doctor` flags but only updates the changelog. The CLI doctor docs list supported options, so users will not see `--post-upgrade` or the `--json` restriction unless this page is updated with the new mode and output/exit-code contract.
  - confidence: 0.78
