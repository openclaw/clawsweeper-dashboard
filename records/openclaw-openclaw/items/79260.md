## Summary

Keep open: the feature is plausible and real terminal proof was supplied, but the latest diff still does not mirror the installed plugin entry-resolution contract, so the new probe can miss broken installed TypeScript plugins and still flag valid inferred built peers incorrectly.

## What This Changes

The PR adds `openclaw doctor --post-upgrade [--json]`, plugin entry and manifest-drift probes, CLI flags, doctor docs, tests, and a changelog entry.

## Best Possible Solution

Keep the feature scoped, but make the post-upgrade entry probe reuse or exactly mirror existing package entry validation/resolution semantics, with regression fixtures for installed TypeScript source without dist, inferred dist peer, explicit `runtimeExtensions`, and manifest drift.

## Reproduction Assessment

Yes for the PR blocker by source inspection. A package with `openclaw.extensions: ["./src/index.ts"]`, source present, and no built `dist` peer is rejected by current install validation but would produce no finding here; the inverse source-missing plus inferred dist case can be flagged incorrectly.

## Solution Assessment

No. The direction is reasonable, but the probe should use the existing entry resolver or a shared pure-filesystem contract helper instead of an ad hoc source-path `fs.access` check.
