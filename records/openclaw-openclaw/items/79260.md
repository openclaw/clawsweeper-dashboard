## Summary

Keep open. The feature direction is useful and the PR has real terminal proof, but the current head still duplicates only part of the existing plugin package entry resolver, leaving P2 correctness and low-severity filesystem-boundary security issues before merge.

## What This Changes

The PR adds `openclaw doctor --post-upgrade [--json]` with structured plugin entry-unresolved and manifest-drift findings, CLI/docs/changelog wiring, and focused tests.

## Best Possible Solution

Keep the scoped doctor mode, but make post-upgrade entry probing reuse or share the existing package-entry validation contract so doctor reports the same package health as install and discovery.

## Reproduction Assessment

Yes. Source inspection gives concrete current-main cases for escaping entries, inferred built `dist` output, source-only TypeScript entries, and malformed `runtimeExtensions`, while the PR probe only does direct positional file-existence checks.

## Solution Assessment

No. The CLI mode is a reasonable direction, but duplicating partial `fs.access` checks is not the best fix; the probe should reuse or exactly mirror the current package-entry resolver contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Mirror package runtime entry resolution:** `src/commands/doctor-post-upgrade.ts:84-86`
  - body: When no explicit `runtimeExtensions` entry is present, installed packages can be valid with only inferred built `dist/*.js` output, while source-only TypeScript packages are unhealthy. This direct source-entry existence check will both flag valid packages and miss packages the loader rejects.
  - confidence: 0.95
- **[P2] Keep entry checks inside plugin roots:** `src/commands/doctor-post-upgrade.ts:68-69`
  - body: Resolving plugin-authored runtime entries with `path.resolve(record.rootDir, runtimeEntry)` and `fs.access` accepts absolute paths, parent-directory escapes, and symlink escapes when the target exists. Current install/discovery validation rejects those entries, so doctor can mark an invalid plugin healthy.
  - confidence: 0.92
- **[P2] Validate runtimeExtensions before indexing:** `src/commands/doctor-post-upgrade.ts:62-67`
  - body: The current resolver rejects blank `openclaw.runtimeExtensions` values and length mismatches before resolving entries. This code treats the field as an unchecked positional list, so malformed packages can fall through to source-entry checks and avoid the same compatibility error that install/discovery would report.
  - confidence: 0.89
