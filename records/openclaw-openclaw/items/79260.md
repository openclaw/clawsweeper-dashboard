## Summary

Keep open: the PR adds a useful doctor mode, but the live head still checks plugin entries with direct filesystem probes instead of the existing package-entry resolver, so it can disagree with loader/install health for valid built-runtime packages and invalid escaping entries.

## What This Changes

Adds `openclaw doctor --post-upgrade [--json]` with plugin entry and manifest-drift probes, CLI/docs/changelog wiring, and focused tests.

## Best Possible Solution

Keep the probe-only doctor mode, but drive `plugin.entry_unresolved` through the existing package-entry resolution and boundary contract, with regression fixtures for inferred built peers, source-only TypeScript, escaping paths, and malformed runtimeExtensions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a package with `openclaw.extensions: ["./src/index.ts"]` and a built `dist/index.js` peer is accepted by current resolver rules but would be reported as unresolved by the PR; escaping entries can also be reported healthy when the target exists.

## Solution Assessment

No. The CLI direction is reasonable, but the implementation should reuse or exactly mirror the existing package-entry resolver instead of maintaining weaker direct `path.resolve` and `fs.access` checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Mirror the package entry resolver:** `src/commands/doctor-post-upgrade.ts:84-86`
  - body: The probe falls back to checking only the declared source path when no explicit runtime entry exists, so it reports valid installed packages with an inferred `dist/*.js` peer as broken and misses source-only TypeScript packages that the loader rejects. Reuse or mirror the package-entry resolver so this doctor mode reports the same health as install/discovery.
  - confidence: 0.95
- **[P2] Enforce plugin-root boundaries:** `src/commands/doctor-post-upgrade.ts:68-69`
  - body: A `runtimeExtensions` value such as an absolute path or `../outside.js` is accepted whenever that file exists, and an escaping source entry is skipped once a runtime entry exists. Current plugin validation rejects entries that leave the package root, so this mode can report an invalid plugin as healthy.
  - confidence: 0.9
