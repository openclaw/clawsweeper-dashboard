## Summary

This PR should stay open: the doctor mode is useful and has real terminal proof, but the entry probe still diverges from OpenClaw's existing plugin package-entry contract in ways that can produce false positives and false negatives.

## What This Changes

The PR adds `openclaw doctor --post-upgrade [--json]`, plugin entry and manifest-drift probes, CLI/docs/changelog wiring, and focused tests.

## Best Possible Solution

Keep the scoped doctor mode, but route entry probing through the existing package-entry resolver or a shared helper so doctor reports the same package health as install and discovery.

## Reproduction Assessment

Yes. Source inspection gives concrete current-main fixtures for escaping entries, inferred built `dist` output, source-only TypeScript entries, and malformed `runtimeExtensions`, and the PR probe does not apply those same checks.

## Solution Assessment

No. The CLI direction is useful, but duplicating partial `fs.access` checks is not the narrowest maintainable solution; the probe should reuse or exactly mirror the current resolver contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Mirror inferred runtime entry resolution:** `src/commands/doctor-post-upgrade.ts:84-86`
  - body: When no explicit `runtimeExtensions` entry is present, current OpenClaw checks inferred built outputs and rejects installed TypeScript source-only packages without compiled JavaScript. This branch only checks whether the source entry exists, so it can flag valid packages that only ship `dist` output and miss source-only TypeScript packages that the loader rejects.
  - confidence: 0.95
- **[P2] Keep entry checks inside plugin roots:** `src/commands/doctor-post-upgrade.ts:68-69`
  - body: Resolving plugin-authored entries with `path.resolve(record.rootDir, entry)` plus `fs.access` accepts absolute paths or `..` escapes when the target exists. Current install/discovery validation rejects entries that leave the plugin package directory, so the post-upgrade probe can mark an invalid plugin healthy.
  - confidence: 0.92
- **[P2] Validate runtimeExtensions before indexing:** `src/commands/doctor-post-upgrade.ts:62-67`
  - body: Current resolver rejects blank `openclaw.runtimeExtensions` values and length mismatches before resolving entries. This code treats the field as an unchecked positional list, so malformed packages can fall through to source-entry checks and avoid the same compatibility error that install/discovery would report.
  - confidence: 0.89
