## Summary

Keep open: the new post-upgrade doctor mode is useful and has terminal proof, but the latest head still duplicates weaker entry-path checks instead of using the existing package-entry resolver, so it can miss unhealthy installed plugins and accept path-escaping entries.

## What This Changes

The branch adds `openclaw doctor --post-upgrade [--json]`, plugin entry and manifest-drift probes, CLI/docs/changelog wiring, and focused tests.

## Best Possible Solution

Keep the scoped post-upgrade mode, but route `plugin.entry_unresolved` through the existing boundary-safe package-entry validation contract and add fixtures for the missed installed-package cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an installed package with `openclaw.extensions: ["./src/index.ts"]`, source present, and no runtime entry or built `dist` peer is rejected by current install/discovery code but produces no PR finding; absolute or `../` entries also pass if the target exists.

## Solution Assessment

No. The CLI direction is reasonable, but the implementation should share or exactly mirror the current package-entry resolver instead of maintaining weaker direct filesystem checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Report source-only TypeScript entries:** `src/commands/doctor-post-upgrade.ts:84-86`
  - body: This treats an installed package as healthy whenever `openclaw.extensions[i]` exists on disk. Current install/discovery rejects `./src/index.ts` without a compiled `dist/*.js` or explicit runtime entry, so a plugin that the loader skips can produce no `plugin.entry_unresolved` finding. Reuse the package entry resolver or add the built-peer and TypeScript-source checks here.
  - confidence: 0.95
- **[P2] Keep entry probes inside plugin roots:** `src/commands/doctor-post-upgrade.ts:68-69`
  - body: Resolving a runtime entry with `path.resolve(record.rootDir, runtimeEntry)` and then `fs.access` accepts absolute or `../` paths outside the plugin package when that file exists. Current plugin validation rejects escaping entries, so this mode can report an invalid installed plugin as healthy and probe arbitrary local paths.
  - confidence: 0.91
