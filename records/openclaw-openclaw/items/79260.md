## Summary

Keep open: this is a useful focused CLI feature with sufficient terminal proof, but the new probe still diverges from current plugin package-entry validation and can both miss broken installed plugins and report invalid entries as healthy.

## What This Changes

The PR adds `openclaw doctor --post-upgrade [--json]`, plugin entry and manifest-drift findings, CLI/docs/changelog wiring, and focused tests.

## Best Possible Solution

Keep the scoped post-upgrade mode, but implement `plugin.entry_unresolved` by reusing or exactly mirroring the existing package-entry resolver and adding fixtures for the resolver contracts it currently misses.

## Reproduction Assessment

Yes. Source inspection gives high-confidence repro fixtures: TypeScript entries with inferred `dist` output, source-only TypeScript entries without compiled output, escaping paths, and malformed `runtimeExtensions` arrays all differ from current main's package-entry validation.

## Solution Assessment

No. The CLI direction is reasonable, but direct `path.resolve` plus `fs.access` is not the narrow maintainable fix; the probe should share or faithfully mirror the existing boundary-safe package-entry validation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Mirror TypeScript entry resolution:** `src/commands/doctor-post-upgrade.ts:84-86`
  - body: The fallback checks only whether the source entry exists. Current package validation also accepts inferred built outputs for `./src/*.ts` and rejects source-only TypeScript entries without compiled runtime output, so this can both flag valid packaged plugins as broken and miss the source-only case the mode is meant to catch.
  - confidence: 0.95
- **[P2] Keep entry checks inside plugin roots:** `src/commands/doctor-post-upgrade.ts:68-69`
  - body: Resolving plugin-declared entries with `path.resolve(record.rootDir, runtimeEntry)` and `fs.access` accepts absolute or `../` paths outside the installed package when the target exists. Current validation rejects escaping entries, so doctor can report an invalid plugin as healthy and inspect arbitrary local paths.
  - confidence: 0.91
- **[P2] Validate the runtimeExtensions array:** `src/commands/doctor-post-upgrade.ts:63-67`
  - body: Treating `runtimeExtensions[index]` as optional lets malformed arrays pass: extra entries are ignored, length mismatches fall back to source checks, and blank entries are treated as absent. Current package validation rejects those cases, so this probe can miss invalid installed metadata.
  - confidence: 0.9
