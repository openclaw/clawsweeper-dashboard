## Summary

Keep open. Current main still has the silent-success fallback for send results with no delivery result, so the PR is addressing a valid bug, but the proposed helper would incorrectly turn valid gateway-executed plugin sends with a top-level messageId payload into tool errors.

## What This Changes

The PR adds a missing-delivery guard to shared message-tool send handling and regression tests for no-result, dry-run, and concrete nested messageId send paths.

## Best Possible Solution

Keep the missing-delivery behavior, but scope it so valid plugin/gateway send successes are preserved, either by limiting the guard to core direct sends or by accepting the top-level payload.messageId shape used by gateway-executed plugin sends. Add a regression for that shape, keep the dry-run and nested result coverage, and include the required changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. The original bug can be reproduced with a mocked core send MessageActionRunResult whose payload and sendResult lack result.messageId; the PR regression can be reproduced from the current gateway plugin send shape that returns a top-level payload.messageId.

## Solution Assessment

No, not as written. Returning status:error for a true missing core delivery result is the right direction, but the proposed helper is too broad and would misclassify valid gateway/plugin sends as failures.
