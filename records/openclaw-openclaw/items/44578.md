## Summary

Keep this PR open: current main still has the success-shaped fallback for a core send with no concrete delivery receipt, but the branch still classifies some valid successful send shapes as failures and lacks after-fix real behavior proof.

## What This Changes

The PR changes `message-tool` to return an error for non-dry-run sends without a nested delivery `messageId` and adds regression tests for missing, dry-run, and nested-success cases.

## Best Possible Solution

Constrain the missing-delivery error to true core non-dry-run sends that lack any accepted receipt, preserve gateway/plugin and internal-source success shapes, and add focused regressions plus redacted runtime proof.

## Reproduction Assessment

Yes. A focused source-level reproduction is to stub `runMessageAction` to return a non-dry-run core send with no `toolResult` and no `sendResult.result.messageId`; current main then falls through to `jsonResult(result.payload)`.

## Solution Assessment

No. The direction is right, but the current PR guard is too broad because it rejects existing successful gateway/plugin and internal-source send receipts.
