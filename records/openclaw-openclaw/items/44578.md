## Summary

Keep open: the bug is source-reproducible on current main and the PR is pointed at the right failure class, but the patch would mark valid gateway/plugin sends with top-level `payload.messageId` as delivery failures and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch adds a message-tool guard that returns `status: "error"` for non-dry-run send results without a nested delivery `messageId` and adds regression tests for missing, dry-run, and nested-success paths.

## Best Possible Solution

Land a clean branch that reports true non-dry-run sends without a delivery id as tool errors while preserving both nested and top-level `messageId` success shapes, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection can reproduce the current-main bug by mocking a non-dry-run `kind: "send"` result whose payload and sendResult lack a delivery `messageId`; current main falls through to `jsonResult(result.payload)`. The PR regression is also source-reproducible from the existing gateway plugin send shape with top-level `payload.messageId`.

## Solution Assessment

No, not as written. Returning `status: "error"` for a true missing delivery result is the right direction, but the guard must accept valid top-level `payload.messageId` successes or be scoped more narrowly before merge.
