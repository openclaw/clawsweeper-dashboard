## Summary

Keep this PR open because the underlying silent-success message-tool bug is still present on current main, but the branch is not merge-ready: it lacks real after-fix behavior proof and its guard can misclassify existing successful send paths as errors.

## What This Changes

The branch adds a message-tool guard that returns an error for non-dry-run send results without a nested delivery `result.messageId`, plus focused regression tests for missing-result, dry-run, and nested-messageId cases.

## Best Possible Solution

Narrow the guard so only true core sends without an accepted delivery receipt become errors, preserve gateway top-level `messageId` and internal-source successes, and require redacted after-fix proof from a real send path before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a non-dry-run send result with no toolResult and no nested delivery `result.messageId` reaches the current `jsonResult(result.payload)` fallback on main. I did not run tests because this review is read-only.

## Solution Assessment

No, the current PR is not yet the best solution because its receipt check is too narrow and breaks current accepted send-success shapes. The better fix is an explicit delivery-receipt predicate over `handledBy`, gateway payloads, internal-source sinks, and core `MessageSendResult`.
