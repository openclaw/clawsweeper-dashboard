## Summary

Keep open: the PR targets a real current-main silent-success path, but the latest branch still misclassifies valid gateway plugin sends with top-level delivery messageIds and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a message-tool guard that returns an error for non-dry-run sends without a delivery messageId and adds unit coverage for missing, dry-run, and nested-success cases.

## Best Possible Solution

Keep the missing-delivery guard, but recognize top-level payload.messageId or scope the guard to core sends, add gateway regression coverage, then provide redacted after-fix delivery proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: inject a non-dry-run send result with no toolResult and no delivery result, and createMessageTool falls through to jsonResult(result.payload); I did not run tests in this read-only review.

## Solution Assessment

No. The direction is narrow and maintainable, but the current PR must recognize the existing top-level gateway payload.messageId receipt shape or avoid applying the guard to gateway plugin sends before it is safe.
