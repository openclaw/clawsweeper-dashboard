## Summary

Keep open: the reported silent-success shape is still plausible on current main, but this PR is not mergeable as written because its broad missing-delivery guard would turn existing successful gateway/plugin and internal WebChat/TUI sends into errors, and the external after-fix real behavior proof is still missing.

## What This Changes

The PR changes `message-tool` to return an error for non-dry-run sends without a nested delivery `result.messageId` and adds unit tests for missing, dry-run, and nested-success cases.

## Best Possible Solution

Constrain the new error path to true core delivery attempts that lack a delivery receipt, preserve gateway/plugin top-level receipts and internal UI source replies, then add focused regressions plus redacted real runtime proof.

## Reproduction Assessment

Yes. A focused source-level reproduction is to stub `runMessageAction` to return a non-dry-run core send with no `toolResult`, no `sendResult.result.messageId`, and a metadata payload; current main then falls through to `jsonResult(result.payload)`.

## Solution Assessment

No. The direction is maintainable, but this branch should only surface missing delivery for core sends that truly lack a receipt and must explicitly accept existing successful gateway and internal-source result shapes.
