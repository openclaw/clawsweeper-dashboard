## Summary

## What This Changes

The branch adds message-tool logic and tests to return a status error when a non-dry-run send has no nested delivery messageId.

## Best Possible Solution

Land a refreshed fix that errors only true core non-dry-run sends without an accepted delivery receipt, preserves gateway-plugin top-level receipts and internal-source success shapes, adds focused regressions, and includes redacted real send proof.

## Reproduction Assessment

Yes, by source inspection. Stub runMessageActionForTool to return a non-dry-run core send with no toolResult and no sendResult.result.messageId; current main then falls through to jsonResult(result.payload).

## Solution Assessment

No. The direction is right, but the proposed guard is too broad because it only recognizes nested messageId receipts and would turn existing successful gateway-plugin and internal-source sends into tool errors.
