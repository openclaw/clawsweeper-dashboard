## Summary

Keep open: current main still has the silent success-shaped fallback, and the PR targets a real message-tool bug, but the current patch would also turn existing successful non-core send receipts into errors and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `message-tool` to return an error when a non-dry-run send result lacks a nested delivery `messageId` and adds focused regression tests for missing, dry-run, and nested-success send results.

## Best Possible Solution

Update the PR so missing delivery is an error only for core sends that truly lack a delivery receipt, preserve gateway and internal-source success contracts, add focused regressions, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source-level reproduction is clear: stub `runMessageAction` to return a non-dry-run core send with no `toolResult`, no `sendResult.result.messageId`, and a metadata payload; current main falls through to `jsonResult(result.payload)`. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is narrow and maintainable, but the current patch is too broad because it treats successful gateway plugin and internal-source sends as missing-delivery errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve successful non-core send receipts:** `src/agents/tools/message-tool.ts:693`
  - body: The guard runs for every non-dry-run send without a nested `result.messageId`. Current main also has successful gateway plugin sends that return `payload.messageId` at the top level and internal WebChat/TUI source replies that return `status/sourceReplySink` without any messageId, so this PR would convert those successful sends into errors. Limit the error path to core sends that truly lack a delivery receipt, or explicitly accept those existing success shapes.
  - confidence: 0.92
