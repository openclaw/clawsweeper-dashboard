## Summary

Keep open: current main still has the silent-success fallthrough for send payloads with no delivery result, so the PR is aimed at a real bug, but the patch still misclassifies valid gateway/plugin sends with top-level messageId payloads and the external PR has only test proof, not after-fix real behavior proof.

## What This Changes

The PR adds a missing-delivery guard to message-tool send results and adds regression tests for missing, dry-run, and nested messageId payloads.

## Best Possible Solution

Land a clean branch that treats true non-dry-run sends without a delivery id as tool errors while preserving valid nested and top-level messageId successes, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection can reproduce the current-main bug by mocking a non-dry-run kind: send result whose payload and sendResult lack a delivery messageId; current main falls through to jsonResult(result.payload). The PR regression is also source-reproducible from the existing gateway plugin send shape with a top-level payload.messageId.

## Solution Assessment

No, not as written. Returning status:error for a true missing delivery result is the right direction, but the guard must accept valid top-level gateway/plugin payload.messageId successes or be scoped to core direct sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92
