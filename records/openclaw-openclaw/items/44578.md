## Summary

Keep this PR open: it targets a real remaining message-tool failure mode on current main, but the proposed guard incorrectly converts successful gateway-plugin sends into errors and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes message-tool send handling so non-dry-run sends without a concrete delivery messageId return an explicit error payload, with regression coverage for missing, dry-run, and successful core delivery-result paths.

## Best Possible Solution

Revise the guard so it only flags sends that truly lack a confirmed delivery identity, preserves gateway/plugin send successes, and is backed by focused regression coverage plus redacted real delivery proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the PR regression: a gateway-executed Telegram send returns top-level messageId, while the new guard only reads nested result.messageId. I did not run tests because this review is read-only.

## Solution Assessment

No. Surfacing real missing core delivery results is the right direction, but this implementation should either gate the new error path to core/sendResult results or accept top-level plugin payload messageId before treating delivery as missing.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve gateway plugin send successes:** `src/agents/tools/message-tool.ts:682`
  - body: Telegram runs message actions through the gateway, so `runGatewayPluginMessageActionOrNull()` returns a `kind: "send"` payload with no `sendResult` or `toolResult`. Telegram send success is shaped as top-level `{ ok: true, messageId, chatId }`, but this helper only accepts `payload.result.messageId`, so a successful Telegram send would be converted into `status: "error"`. Gate this missing-delivery check to core sends or accept top-level `payload.messageId` before returning the error.
  - confidence: 0.92
