## Summary

Keep open: the current-main bug is still present, but this PR is not mergeable as-is because the new guard would classify successful gateway-plugin sends, including Telegram, as missing delivery results. External after-fix real behavior proof is also still absent.

## What This Changes

The PR changes message-tool send handling to turn a non-dry-run core send without a concrete delivery messageId into an explicit tool error and adds regression coverage.

## Best Possible Solution

Refresh the patch so missing-delivery detection either applies only to core direct sends or recognizes existing payload shapes such as top-level messageId, then prove the Feishu or equivalent runtime path with redacted real output.

## Reproduction Assessment

Yes, by source inspection. Current main can return a send payload without result.messageId as a normal jsonResult, and the PR regression is also source-reproducible through Telegram gateway sends that return a top-level messageId.

## Solution Assessment

No, not as-is. The direction is narrow and appropriate, but the implementation must preserve gateway-plugin send successes before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve gateway-plugin send successes:** `src/agents/tools/message-tool.ts:682`
