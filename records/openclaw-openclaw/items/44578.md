## Summary

Keep open: current main still has the silent-success path, and this PR is a useful fix direction, but the patch would misclassify successful gateway plugin sends that return a top-level messageId and it also lacks the required changelog entry.

## What This Changes

The PR adds a message-tool guard that returns status:error for non-dry-run send results without a nested delivery messageId, plus regression tests for missing, dry-run, and nested-success send outputs.

## Best Possible Solution

Keep the missing-delivery error behavior, but recognize valid gateway/plugin successes by accepting top-level payload.messageId or narrowing the guard to true core/direct missing results, then add gateway regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mock a non-dry-run core send result whose payload and sendResult lack result.messageId, and current main falls through to jsonResult(result.payload).

## Solution Assessment

No, not as written. Returning status:error for a true missing delivery result is the right direction, but the helper is too broad for gateway/plugin sends that already expose a top-level messageId.
