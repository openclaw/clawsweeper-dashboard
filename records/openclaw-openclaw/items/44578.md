## Summary

Keep open: current main still has the silent-success fallback for send results without a delivery receipt, but this external PR is not merge-ready because the proposed guard would misclassify existing successful send shapes and lacks redacted real behavior proof.

## What This Changes

The PR adds message-tool logic and tests that turn non-dry-run send results without a nested delivery messageId into a status error.

## Best Possible Solution

Refresh the branch onto main with a narrower check that errors only true core non-dry-run sends without an accepted receipt, preserves top-level gateway receipts and internal-source success, adds regressions, and includes redacted real send proof.

## Reproduction Assessment

Yes, by source inspection: stub runMessageActionForTool to return a non-dry-run core send with no toolResult and no sendResult.result.messageId, and current main reaches jsonResult(result.payload). I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is right, but the implementation should distinguish true core missing receipts from existing successful gateway-plugin and internal-source send shapes instead of treating every non-dry-run send without nested messageId as a failure.
