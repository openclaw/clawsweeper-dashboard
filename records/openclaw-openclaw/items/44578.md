## Summary

Keep open: current main still has the success-shaped fallback for send results without a delivery receipt, but this PR still misclassifies existing gateway plugin sends with top-level `payload.messageId` and lacks after-fix real behavior proof.

## What This Changes

The PR adds a message-tool guard that turns non-dry-run send results without a delivery `messageId` into an error and adds regression tests for missing, dry-run, and nested-success cases.

## Best Possible Solution

Keep the missing-delivery guard, but recognize all valid receipt shapes, including top-level gateway `payload.messageId`, then validate with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: inject a non-dry-run send result with no `toolResult` and no delivery result, and current main falls through to `jsonResult(result.payload)`; the PR also has a separate source-reproducible regression for top-level gateway `payload.messageId`.

## Solution Assessment

No. The direction is narrow and maintainable, but the current PR must accept existing top-level gateway receipt payloads or scope the guard to core sends before it is safe to merge.
