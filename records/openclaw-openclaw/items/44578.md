## Summary

Keep open. Current main still has the silent-success fallback this PR is trying to fix, but the latest PR head still misclassifies valid gateway/plugin sends that return a top-level messageId payload, and the branch is currently dirty against main.

## What This Changes

The PR adds a missing-delivery guard to the shared message tool, returning status:error for non-dry-run send results without a nested delivery messageId, and adds regression coverage for missing, dry-run, and successful nested-messageId send results.

## Best Possible Solution

Keep the missing-delivery behavior, but revise the helper so valid gateway/plugin send successes are preserved, either by scoping the guard to core send results or by accepting the top-level payload.messageId shape used by gateway-executed plugin sends. Then rebase or cherry-pick the focused change onto current main and validate both message-tool and plugin-dispatch paths.

## Reproduction Assessment

Yes. The original bug can be reproduced with a mocked core send MessageActionRunResult whose payload and sendResult lack result.messageId; the PR regression can be reproduced with the existing gateway plugin send shape that returns a top-level messageId payload.

## Solution Assessment

No, not as written. Returning status:error for true missing delivery is the right direction, but the current helper is too broad because it treats valid plugin/gateway send payloads as missing delivery.
