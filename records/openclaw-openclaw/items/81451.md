## Summary

Keep open: the proposed optimization has blocking correctness issues around watcher reconfiguration and cache lifetime, and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds a fast return for existing skill watchers and introduces a module-level `resolvedSkills` cache in `ensureSkillSnapshot` with targeted tests.

## Best Possible Solution

Keep the performance direction, but only skip watcher setup after confirming the same watch signature, bound or clear the `resolvedSkills` cache on invalidation, and add inspectable real-run proof for the latency claim.

## Reproduction Assessment

Yes for the review findings by source inspection: current main compares watcher paths/debounce before returning, while the PR returns before that comparison and adds a version-keyed cache with no production eviction. I did not run a live watcher benchmark in this read-only pass.

## Solution Assessment

No. The warm-cache goal is reasonable, but this implementation removes existing config-change watcher behavior and adds an unbounded cache; a safer solution should preserve the watcher signature check and define cache invalidation bounds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve watcher reconfiguration on config changes:** `src/agents/skills/refresh.ts:126-128`
  - body: Returning for any existing watcher prevents the `pathsKey`/`debounceMs` comparison below from running. If `skills.load.extraDirs`, plugin skill dirs, or `watchDebounceMs` change while the process is alive, the old watcher stays attached and changes in newly configured roots will not bump snapshot versions. Keep the fast path only after confirming the watcher signature is unchanged.
  - confidence: 0.92
- **[P2] Bound the resolvedSkills cache:** `src/auto-reply/reply/session-updates.ts:36`
  - body: This cache stores full `resolvedSkills` arrays under a key that includes `snapshotVersion`, but old versions are never deleted outside the test reset helper. A long-lived gateway that sees repeated skill edits, config invalidations, remote-node changes, workspaces, or filters will retain historical `SKILL.md` bodies indefinitely, so this needs eviction or invalidation when versions advance.
  - confidence: 0.86
