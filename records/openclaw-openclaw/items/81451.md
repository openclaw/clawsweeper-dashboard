## Summary

Keep open: the latest head addressed the earlier unbounded-cache and unconditional-log blockers, but the remaining cache key can hydrate a session with parsed skills from an incompatible agent/config context. The PR also still needs real behavior proof that directly exercises the new module-level cache path, not only the warm dispatch path served by session-local state.

## What This Changes

The branch caches resolvedSkills in a bounded module-level Map, skips watcher target scans when debounce/config watch roots are unchanged, and adds focused skills/session tests.

## Best Possible Solution

Keep the optimization, but make resolvedSkills cache reuse prove compatibility with the full effective skill context before serving cached parsed skills.

## Reproduction Assessment

Yes for the review finding: source inspection shows a concrete path where one stripped snapshot can hydrate from another session’s cached resolvedSkills when workspace/version/raw skillFilter match but effective agent/config/eligibility differ. A focused unit test can reproduce this by mocking different effective skill outputs under the same cache key.

## Solution Assessment

No: caching this hot path is reasonable, but keying only on workspace/version/raw skillFilter is not the safest maintainable solution. The cache must include the effective skill context or avoid cross-session reuse when compatibility cannot be proven.
