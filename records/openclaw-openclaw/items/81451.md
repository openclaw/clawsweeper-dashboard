## Summary

Keep open: the latest head partially addresses the earlier config-omission review, but the new cache key still uses raw secret-bearing skills config and does not cover every config path that can affect skill eligibility. The PR also still needs contributor-supplied real gateway proof for the new module-level cache path before merge.

## What This Changes

The PR adds a bounded module-level `resolvedSkills` cache in `ensureSkillSnapshot`, skips redundant watcher target resolution when basic watch config is unchanged, and adds focused tests.

## Best Possible Solution

Keep the watcher reuse direction only with an explicit compatibility boundary, and replace raw config serialization with a redacted deterministic eligibility fingerprint or rebuild when compatibility cannot be proven.

## Reproduction Assessment

Yes for the review finding by source inspection. A focused regression can seed the module cache, then change an eligibility-affecting config path outside `skills.*` or verify that secret-bearing `skills.entries.*` values are never serialized into a cache key.

## Solution Assessment

No: caching this hot path is reasonable, but the current `cfg.skills` key is not the safest maintainable boundary. A redacted effective eligibility fingerprint, or no cross-session module reuse when compatibility is uncertain, is safer.
