## Summary

Keep open: the optimization is useful and has sufficient real behavior proof, but the current PR still introduces an unbounded `resolvedSkills` cache and unconditional hot-path perf logs that should block merge.

## What This Changes

The PR caches skill snapshot `resolvedSkills` across turns and skips repeated skills watcher target resolution when watcher config is unchanged.

## Best Possible Solution

Keep the warm-path optimization, but bound or prune the process cache and remove or gate timing logs behind the existing debug timing mechanism.

## Reproduction Assessment

Yes for the review findings: PR head source shows an unbounded module-level Map and unconditional `console.log` calls in `ensureSkillSnapshot`. The contributor supplied real live gateway timing logs for the performance improvement itself.

## Solution Assessment

No: the PR targets the right hot path and the watcher reconfiguration fix is present, but the cache must be bounded and the perf logs must be debug-gated or removed before this is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the resolvedSkills cache:** `src/auto-reply/reply/session-updates.ts:36`
  - body: This module-level Map retains every full `resolvedSkills` array by workspace/version/filter and is only cleared in tests. Because `resolvedSkills` includes parsed SKILL.md bodies and `snapshotVersion` changes on invalidation, a long-lived gateway can retain stale snapshots indefinitely; keep only current entries or add a small TTL/LRU bound.
  - confidence: 0.87
- **[P2] Gate the perf logging:** `src/auto-reply/reply/session-updates.ts:188-190`
  - body: `ensureSkillSnapshot` runs on normal reply dispatches, but this prints `[perf]` timing lines unconditionally. That will spam gateway stdout and add work on the hot path for every user, so remove these logs or guard them behind the existing debug timing mechanism.
  - confidence: 0.9
