## Summary

Keep open: the watcher reuse optimization looks useful, but the latest head still has a blocking cache-boundary bug because `resolvedSkills` reuse is keyed only by `cfg.skills` while current skill eligibility reads other config paths. The external real-behavior proof also shows warm-session timing logs, but it does not directly prove the new module-level cache path after `resolvedSkills` has been stripped from persisted session state.

## What This Changes

The PR reuses existing skill watchers and adds a bounded module-level `resolvedSkills` cache in `ensureSkillSnapshot`, with focused tests for watcher reuse, reconfiguration, and cache invalidation.

## Best Possible Solution

Keep the watcher reuse optimization, but either key module-level `resolvedSkills` reuse by all effective skill eligibility inputs with redacted fingerprints, or remove that cache until the compatibility boundary can be proven with regression coverage and real runtime proof.

## Reproduction Assessment

Yes for the review finding by source inspection. Seed the module cache while a config-gated skill such as Discord is eligible, then remove or change `channels.discord.token`; the PR cache key does not change even though current-main eligibility should.

## Solution Assessment

No. Caching this hot path is reasonable, but the current implementation is not the safest maintainable boundary because `buildWorkspaceSkillSnapshot` evaluates full config when deciding skill inclusion.
