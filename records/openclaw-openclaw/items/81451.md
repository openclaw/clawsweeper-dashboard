## Summary

Keep open: the latest head partially addresses the earlier cache-key review, but the module cache still omits config inputs that determine skill eligibility, and the supplied proof does not directly exercise the new module-level cache path.

## What This Changes

The PR adds a bounded in-memory `resolvedSkills` cache in `ensureSkillSnapshot`, skips redundant skills watcher target scans when watch config is unchanged, and adds focused tests.

## Best Possible Solution

Keep the watcher fast path, but only reuse cached `resolvedSkills` when the full effective skill context is proven compatible, then add cfg-sensitive regression coverage and real module-cache proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows a same workspace/version/filter/agent/remote cache key can cross different cfg-derived skill eligibility. A focused unit test can mock two configs where `skills.entries`, `allowBundled`, or a required config path changes under the same cache key.

## Solution Assessment

No: caching this hot path is reasonable, but the current cache key is still not the safest boundary because `cfg` changes the resolved skill set. The safer fix is a deterministic cfg-derived compatibility key, or no cross-session module reuse when compatibility cannot be proven.
