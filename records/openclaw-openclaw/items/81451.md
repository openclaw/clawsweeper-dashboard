## Summary

Keep open: the performance direction is useful and the latest commits address part of the earlier cache-key feedback, but the cache key still omits config sections that affect resolved skills, and the supplied proof does not directly exercise the new module-level cache path.

## What This Changes

This PR adds a bounded module-level `resolvedSkills` cache in `ensureSkillSnapshot`, skips redundant skills watcher scans when watcher config is unchanged, and adds focused tests.

## Best Possible Solution

Keep the watcher fast path, but reuse cached `resolvedSkills` only behind a deterministic compatibility key for the full effective skill context, then add cfg-sensitive regression coverage and direct real gateway proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows the cache key omits `cfg.agents` and `cfg.plugins` while current-main skill snapshot building depends on both. A focused unit test can change an agent skill filter or plugin activation under the same workspace/version/`cfg.skills` key and assert a rebuild occurs.

## Solution Assessment

No: caching this hot path is reasonable, but the current key is still not the safest boundary. The safer solution is a deterministic effective skill-context key, or rebuilding when compatibility cannot be proven.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Include agent and plugin skill context in the cache key:** `src/auto-reply/reply/session-updates.ts:200-207`
