## Summary

Keep open: the PR is a focused performance cache, but the current cache key misses runtime eligibility inputs and the supplied proof still does not show real warm-gateway behavior after the patch.

## What This Changes

The branch adds a bounded in-memory cache for hydrated `resolvedSkills` in `ensureSkillSnapshot`, keyed by workspace/version/filter/agent/remote/config fingerprint, with focused tests and a changelog entry.

## Best Possible Solution

Keep the cache only after cache reuse follows all skill eligibility inputs, or explicitly avoids caching eligibility-sensitive cases, and after redacted live warm-turn gateway/runtime proof is added.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a performance cleanup PR. Source inspection gives a high-confidence review path for the cache boundary and the missing runtime proof gate.

## Solution Assessment

No: the scoped cache direction is reasonable, but the proposed key is not complete because `shouldIncludeSkill` also depends on env and local binary eligibility. The safer fix is to key or invalidate on those runtime inputs, or not cache entries whose eligibility depends on them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Key the cache by runtime eligibility inputs:** `src/auto-reply/reply/session-updates.ts:277-285`
  - body: `shouldIncludeSkill` also depends on `process.env` and local binary/PATH state through `hasEnv` and `hasBinary`. With this key, a warm gateway can reuse cached `resolvedSkills` after an API env var is added/removed or PATH changes, whereas current main rebuilds and re-evaluates that eligibility on hydration. Include a redacted env/presence and local binary/PATH eligibility boundary, or avoid caching when those requirements are involved.
