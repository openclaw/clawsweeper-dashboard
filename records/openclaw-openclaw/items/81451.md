## Summary

Keep open. The optimization is plausible, but current PR head still keys the new module-level `resolvedSkills` cache without non-skills config paths that affect skill eligibility, so it can reuse stale capability-visible skills after channel/plugin config changes; the supplied proof also does not directly exercise the new module-cache path.

## What This Changes

The PR reuses existing skill watchers and adds a bounded module-level `resolvedSkills` cache with focused watcher and cache tests.

## Best Possible Solution

Keep the watcher reuse optimization, but either key module-level `resolvedSkills` reuse by all effective eligibility inputs using redacted fingerprints, or remove that cache until correctness and runtime proof are established.

## Reproduction Assessment

Yes for the review finding by source inspection. Seed the module cache while a config-gated skill such as Discord is eligible, then remove or change `channels.discord.token`; the PR cache key remains unchanged while current-main eligibility should change.

## Solution Assessment

No. Caching this hot path is reasonable, but the current PR is not the safest maintainable boundary because `buildWorkspaceSkillSnapshot` evaluates full config when deciding skill inclusion.
