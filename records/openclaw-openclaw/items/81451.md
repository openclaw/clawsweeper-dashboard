## Summary

Keep this PR open: the performance direction is useful, but the latest cache key still crosses skill eligibility boundaries because it only fingerprints `cfg.skills` while skill inclusion can depend on arbitrary config paths, and the supplied live proof still does not directly exercise the new module-level cache path.

## What This Changes

The PR adds watcher reuse and a bounded module-level `resolvedSkills` cache in `ensureSkillSnapshot`, with focused tests for watcher reuse and cache invalidation.

## Best Possible Solution

Keep the watcher reuse optimization only with a cache boundary that covers all effective skill eligibility inputs, or bypass module-level reuse when that compatibility cannot be proven.

## Reproduction Assessment

Yes for the review finding by source inspection. A focused regression can seed the module cache with a config-gated skill, then change a non-skills config path such as `channels.discord.token` and verify the cache key does not change while eligibility should.

## Solution Assessment

No. Caching this hot path is a reasonable approach, but the current key is not the narrowest safe compatibility boundary because current skill filtering evaluates the full config.
