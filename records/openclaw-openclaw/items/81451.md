## Summary

Keep open: the optimization targets a real skills hot path and the PR body supplies live gateway timing proof, but the current PR head still has blocking review findings around cache correctness, cache growth, and unconditional hot-path logging.

## What This Changes

The PR adds watcher reuse checks and a module-level `resolvedSkills` cache for `ensureSkillSnapshot`, plus tests for watcher reuse, watcher reconfiguration, and cache hits.

## Best Possible Solution

Keep the watcher reuse optimization, but make resolvedSkills reuse bounded and keyed by every snapshot-affecting input, or keep it session/current-version scoped, and remove or debug-gate the timing logs.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows the incomplete cache key, unbounded Map, and unconditional `console.log` calls. I did not execute heap-growth, log-volume, or live gateway proof in this read-only pass.

## Solution Assessment

No: the warm-path direction is useful, but the current implementation needs a complete cache boundary, bounded eviction/pruning, and gated logging before it is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Key the cache by all snapshot-affecting inputs:** `src/auto-reply/reply/session-updates.ts:207`
  - body: The cache key only includes `workspaceDir`, `snapshotVersion`, and the explicit `skillFilter`, but `buildWorkspaceSkillSnapshot` also filters by config, resolved agent, and remote exec eligibility. A stripped snapshot can therefore hydrate `resolvedSkills` computed for a different config/eligibility context, so the embedded runner may see skills that the current session did not advertise or may miss skills it should have.
  - confidence: 0.86
- **[P2] Bound the resolvedSkills cache:** `src/auto-reply/reply/session-updates.ts:36`
  - body: This module-level Map retains every full `resolvedSkills` array, including parsed SKILL.md bodies, and production never evicts stale workspace/version/filter entries. Long-lived gateways can accumulate old skill snapshots as versions or filters churn, so keep only current entries or add a small TTL/LRU bound.
  - confidence: 0.9
- **[P2] Gate the perf logging:** `src/auto-reply/reply/session-updates.ts:188-190`
  - body: `ensureSkillSnapshot` runs on normal reply dispatches, but the PR prints `[perf]` timing lines unconditionally on the hot path. That will spam gateway stdout and add work for every user, so remove these logs or guard them behind the existing debug timing mechanism.
  - confidence: 0.91
