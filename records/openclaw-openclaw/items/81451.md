## Summary

Keep open: the performance direction and live gateway timing proof are useful, and the watcher reconfiguration concern has been addressed, but the new module-level `resolvedSkills` cache is still unbounded and can retain stale full skill snapshots in a long-lived process.

## What This Changes

The PR caches `ensureSkillSnapshot` resolved skills across warm turns, skips redundant skills watcher filesystem scans when config is unchanged, and adds focused tests for cache hits and watcher reuse/reconfiguration.

## Best Possible Solution

Keep the warm-path optimization, but prune or bound stale `resolvedSkills` cache entries without changing the session persistence format or public skills contract.

## Reproduction Assessment

Yes for the review finding: source inspection of PR head shows every cache miss stores a full `resolvedSkills` array in a module-level `Map`, and there is no production eviction path. I did not run a heap-growth benchmark in this read-only review.

## Solution Assessment

No: the watcher short-circuit and warm-cache direction are reasonable, but the cache needs a small bounded invalidation strategy before this is the safest maintainable implementation.
