## Summary

Keep open: current main and v2026.6.6 still only support single shared token/password plus trusted-proxy identity, and the related open issues cover complementary slices rather than replacing this broader team-collaboration/auth-management request. Pinchy is useful external proof that a front-layer can solve much of the team workflow today, but it does not remove the remaining Gateway principal/scoped-token product decision.

## What This Changes

The issue requests multi-token Gateway authentication with per-user or per-agent permissions, caller identity/audit support, and optional API-key management for team collaboration.

## Best Possible Solution

Consolidate the Gateway-principal design across https://github.com/openclaw/openclaw/issues/43903 and https://github.com/openclaw/openclaw/issues/92367, then decide whether API-key UI and auditing belong in OpenClaw core or in an external team layer such as Pinchy.

## Reproduction Assessment

## Solution Assessment

Unclear as proposed: the umbrella direction is valid, but the sketched JSON/header/UI design is not yet proven as the best solution. The safer path is one maintainer-owned Gateway principal/scoped-token design that composes with trusted-proxy, device tokens, operator scopes, sessions, docs, and UI.
