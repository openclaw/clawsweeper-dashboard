## Summary

Keep open: this PR is not ready as written because the catch-all timeout handling would hide existing Feishu diagnostics, the shared timeout increase worsens startup/status waits already tracked in #67466, and the claimed TDZ import cycle is not source-backed yet.

## What This Changes

The branch catches rejected Feishu async races as timeout results, retries the lazy `monitor.account` import after a failure, and raises the default Feishu probe request timeout from 10s to 30s.

## Best Possible Solution

Revise this into a narrow Feishu timeout-noise fix that preserves non-timeout diagnostics, keeps startup/status probe waits bounded, and changes monitor import lifecycle only after a focused TDZ reproduction identifies the actual cycle.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows rejected probe/comment requests would now return `timeout` before existing diagnostic catches run. No high-confidence current-main reproduction was established for the claimed TDZ startup crash.

## Solution Assessment

No. The safer path is to classify timeout-shaped failures in probe/startup-specific code while preserving real request errors, then handle any TDZ crash only after a failing reproduction proves the import cycle.
