## Summary

Keep this PR open, but the current patch should not merge because it hides Feishu request diagnostics and lengthens shared status probes while the claimed TDZ startup cycle is not established from current main.

## What This Changes

The PR catches rejected Feishu async races as timeout results, retries the monitor account dynamic import after rejection, and raises the shared Feishu probe timeout default from 10s to 30s.

## Best Possible Solution

Revise this into a narrow Feishu timeout-noise fix that preserves non-timeout diagnostics, keeps status/startup probe waits bounded, and changes monitor import lifecycle only after a focused TDZ reproduction identifies the actual cycle.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows rejected probe/comment requests would now return `timeout` before existing diagnostic catches run. No high-confidence current-main reproduction was established for the claimed TDZ startup crash.

## Solution Assessment

No. The safer path is to classify timeout-shaped failures in probe/startup-specific code while preserving real request errors, then handle any TDZ crash only after a failing reproduction proves the import cycle.
