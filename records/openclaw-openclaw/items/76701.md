## Summary

Keep open: the PR targets plausible Feishu startup/probe problems, but the submitted diff is not safe to merge because it broadens timeout handling into all rejected Feishu requests and loses existing error semantics.

## What This Changes

The PR changes Feishu monitor runtime loading, converts shared timeout-helper rejections into timeout results, and raises the default Feishu probe request timeout from 10s to 30s.

## Best Possible Solution

Keep the lazy monitor runtime boundary, handle only true timeout-shaped probe failures without swallowing other rejections, and add focused regression coverage plus a Feishu changelog entry.

## Reproduction Assessment

Unclear for the original startup ReferenceError because the PR body reports it but current source inspection did not prove the TDZ path. Yes for the blocking PR regression: existing `probe.test.ts` expects rejected request errors to remain visible, while the PR catch would return `timeout` for that path.

## Solution Assessment

No. Catching all rejections inside the shared timeout helper is broader than the stated Axios timeout-noise fix; the safer path is to classify only timeout-shaped failures at the probe boundary and preserve non-timeout error details.
