## Summary

This PR should stay open and should not merge as-is: the shared timeout helper change hides real Feishu request failures, the import retry does not have a proven current-main TDZ path, and the generic timeout increase conflicts with existing startup/status-latency concerns.

## What This Changes

The PR changes Feishu’s shared async race helper to treat promise rejections as timeouts, adds retry logic around the monitor account dynamic import, and increases the default Feishu probe timeout from 10 seconds to 30 seconds.

## Best Possible Solution

Handle Feishu probe timeout noise in probe/startup-specific code by classifying timeout-shaped errors, preserve non-timeout diagnostics, and either prove the TDZ import path with focused coverage or remove that retry change.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the catch-all helper change would convert rejected Feishu requests into timeout results and bypass existing probe/comment diagnostic paths. No high-confidence current-main reproduction was established for the claimed TDZ startup crash.

## Solution Assessment

No. The narrow maintainable fix is to classify timeout-shaped probe errors at the Feishu probe/startup caller, keep real request errors visible, and require a focused TDZ reproduction before changing monitor import lifecycle.
