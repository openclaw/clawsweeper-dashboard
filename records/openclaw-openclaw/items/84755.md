## Summary

Keep open. This reports a current 2026.5.19 failure, and current main still has a plausible gap where diagnostic force-clear recovery can clear active work without proving the persisted session row is terminalized; an adjacent open PR may cover the same invariant but is not yet proven for this Telegram/Codex path.

## What This Changes

Ensure Codex-harness Telegram sessions that stall after `embedded_run:started` either make progress or recover with persisted session status consistent with active-run state.

## Best Possible Solution

Make diagnostic stuck-session recovery and lifecycle persistence share one invariant: when active work is force-cleared or released, `sessions.list` must not keep a stale `running` row with `hasActiveRun:false`; fold the Telegram/Codex case into https://github.com/openclaw/openclaw/pull/84352 if that PR owns the invariant.

## Reproduction Assessment

No high-confidence live reproduction is available because the reporter could not provide a deterministic trigger. The log shape is source-reproducible enough to keep open: current recovery can force-clear active work while persisted session lifecycle state is updated only by lifecycle events.

## Solution Assessment

Unclear whether the adjacent WebChat PR is sufficient on its own. The maintainable fix is to enforce the persisted-session/active-run invariant at the shared lifecycle or diagnostic recovery boundary, with regression coverage for the force-cleared Codex embedded-run path.
