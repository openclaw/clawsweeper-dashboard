## Summary

Keep open. The report is current against v2026.5.19, current main still separates diagnostic recovery from persisted session lifecycle state, and the adjacent open WebChat PR is not yet proven to cover the Telegram/Codex force-clear path.

## What This Changes

The issue reports Telegram/Codex harness turns stalling after `embedded_run:started` and leaving persisted sessions as `running` while `hasActiveRun` is false.

## Best Possible Solution

Enforce one shared invariant: any diagnostic recovery that force-clears or releases active work must also terminalize or reconcile the persisted session row so `sessions.list` cannot return `running` with `hasActiveRun:false`, then prove it on the Telegram/Codex path or fold it into the existing gateway lifecycle PR if that PR owns the invariant.

## Reproduction Assessment

No high-confidence live reproduction is available because the reporter could not provide a deterministic trigger. The reported log and state shape is source-reproducible enough to keep open: current recovery can force-clear active work while persisted session lifecycle state is updated through a separate lifecycle-event path.

## Solution Assessment

Unclear whether the adjacent WebChat PR is sufficient by itself. The narrow maintainable fix is to enforce the persisted-session and active-run invariant at the shared lifecycle or diagnostic recovery boundary, with regression coverage for `abort_embedded_run` plus `drained:false`/`forceCleared:true`.
