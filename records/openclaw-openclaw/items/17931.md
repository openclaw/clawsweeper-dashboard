## Summary

Keep open: current main still exposes sandbox skill material through writable copies or writable workspace paths, and the remaining fix is security-sensitive cross-backend sandbox design rather than cleanup-closeable work.

## What This Changes

Request to mount eligible skill directories into sandbox containers as read-only sources instead of exposing writable skill copies or writable workspace skill directories.

## Best Possible Solution

Design this as sandbox/security work: derive eligible skill sources from the skills loader, add read-only generated mounts after broader workspace mounts, avoid writable duplicate copies when source mounts are available, include generated mounts in config hashing and FS bridges, define non-Docker backend behavior, and update docs/tests for all `workspaceAccess` modes.

## Reproduction Assessment

Yes, from source inspection: `workspaceAccess: "ro"` and `"none"` still call `syncSkillsToWorkspace`, which copies eligible skills into the sandbox workspace, while `"rw"` uses the agent workspace as `/workspace`. I did not run a live container write attempt in this read-only review.

## Solution Assessment

Unclear as stated. Read-only bind mounts are the right Docker primitive, but the maintainable solution needs secops-owned design for loader-derived mounts, config hashing, FS bridge behavior, duplicate copies, docs, tests, and non-Docker backends.
