## Summary

Keep open: current main still lacks generated read-only per-skill sandbox mounts. The code still copies skills for non-rw sandboxes and exposes workspace skills through the writable workspace mount in rw mode, while the prior custom-bind precedence fix does not create per-skill overlays.

## What This Changes

Request kernel-enforced read-only per-skill mounts in sandbox containers instead of writable skill copies or writable workspace skill paths.

## Best Possible Solution

Implement a secops-reviewed sandbox contract that derives eligible skill source directories from the skills loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, updates FS bridge and config-hash behavior, defines non-Docker behavior, and documents/tests all workspace access modes.

## Reproduction Assessment

Yes. Source inspection shows `workspaceAccess: "rw"` uses the agent workspace as `/workspace`, while non-rw modes call `syncSkillsToWorkspace` and copy skills into the sandbox workspace; I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as filed. Read-only Docker overlays are a plausible enforcement direction, but the maintainable solution needs secops-owned design across loader-derived mounts, duplicate staging, config hashing, FS bridge writability, and non-Docker backends.
