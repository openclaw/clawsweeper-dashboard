## Summary

Keep open: current main still copies eligible skills into writable sandbox workspaces for non-rw modes and still exposes workspace skills through the writable `/workspace` mount in rw mode. Prior steipete and ClawSweeper comments already identified this as unresolved security-sensitive sandbox design work, and current source at b79de62 confirms that no read-only per-skill mount implementation has landed.

## What This Changes

Request read-only bind mounts for skill directories in sandbox containers instead of writable workspace copies or writable workspace skill paths.

## Best Possible Solution

Treat this as sandbox/security design work: derive eligible skill source directories from the skills loader, mount them read-only after broader workspace mounts, avoid writable duplicate copies, update hash and FS bridge contracts, define non-Docker behavior, and document/test all `workspaceAccess` modes.

## Reproduction Assessment

Yes, from source inspection: `workspaceAccess: "ro"` and `"none"` still call `syncSkillsToWorkspace`, which copies skills into the sandbox workspace, while `"rw"` uses the agent workspace as `/workspace`. I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as filed. Read-only Docker bind mounts are the right primitive, but the maintainable fix needs secops-owned design across loader-derived mounts, duplicate copies, config hashing, FS bridge writability, and non-Docker backends.
