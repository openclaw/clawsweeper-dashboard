## Summary

Keep open: current main still does not generate read-only per-skill sandbox mounts, and the recent discussion already kept this as sandbox/secops follow-up rather than cleanup closure.

## What This Changes

Request to mount eligible skill directories into sandbox containers as read-only per-skill bind mounts instead of exposing writable workspace paths or writable copied skill trees.

## Best Possible Solution

Implement a secops-reviewed sandbox contract that derives eligible skill source directories from the skills loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, updates FS bridge/config-hash behavior, defines non-Docker semantics, and documents/tests all workspaceAccess modes.

## Reproduction Assessment

Yes. Source inspection on current main shows rw mode uses the agent workspace as /workspace while non-rw modes call syncSkillsToWorkspace and copy skills into the writable sandbox workspace; I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as filed. Read-only Docker overlays are a plausible enforcement direction, but the narrow maintainable solution needs loader-derived mounts plus config-hash, FS-bridge, SSH/OpenShell, docs, and test follow-through.
