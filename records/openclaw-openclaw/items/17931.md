## Summary

Keep open. Current main still uses writable skill copies for non-rw sandbox workspaces and writable workspace skill directories for rw mode, and the live discussion already marks this as a security-sensitive sandbox design item rather than stale cleanup.

## What This Changes

The issue requests kernel-enforced read-only sandbox mounts for skill directories across all workspaceAccess modes.

## Best Possible Solution

Design a sandbox skill-mount contract that derives eligible skill source directories from the skills loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies when source mounts are available, and updates FS bridges, config hashing, docs, and tests for each backend.

## Reproduction Assessment

Yes, at source level. Current main still routes non-rw sandbox sessions through copy-based skill staging and rw sessions through the writable workspace mount, with no generated per-skill read-only overlay path.

## Solution Assessment

Yes on the direction, but not as a narrow autonomous patch. Kernel-enforced read-only mounts derived from the skills loader are the maintainable shape, while always-on behavior and non-Docker backend semantics need maintainer/security judgment.
