## Summary

Keep open: current main still stages sandbox skills as writable copies in non-rw modes and still exposes workspace skills through the writable `/workspace` mount in rw mode. Prior review comments already identified this as unresolved sandbox/security design work, and current source at `c90fd7e` shows no per-skill read-only mount implementation has landed.

## What This Changes

The issue asks for sandboxed skill directories to be exposed as read-only mounts instead of writable workspace copies or writable workspace paths.

## Best Possible Solution

Define and implement a secops-approved sandbox contract that derives eligible skill sources from the loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, updates hash and FS bridge behavior, documents backend differences, and tests all `workspaceAccess` modes.

## Reproduction Assessment

Yes, from source inspection: `workspaceAccess: "ro"` and `"none"` still call the copy-based `syncSkillsToWorkspace` path, while `"rw"` uses the agent workspace as `/workspace` without per-skill overlays. I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as filed. Read-only Docker bind mounts are the right primitive for the Docker backend, but the maintainable fix needs maintainer/secops decisions across loader-derived mounts, duplicate copies, config hashing, FS bridge writability, and non-Docker backends.
