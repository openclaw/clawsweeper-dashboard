## Summary

Keep open: current main still stages skills as writable sandbox workspace copies for non-rw modes and still exposes workspace skills through the writable `/workspace` mount in rw mode. Prior maintainer/ClawSweeper comments identified this as unresolved sandbox security design work, and source inspection at `c8f7cea` confirms no automatic read-only per-skill mount implementation has landed.

## What This Changes

The issue requests mounting skill directories into sandbox containers as read-only bind mounts instead of exposing writable skill copies or writable workspace skill directories.

## Best Possible Solution

Define and implement a sandbox/skills contract that derives eligible skill source directories, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, and keeps Docker, FS bridge, config hashing, docs, and non-Docker backend behavior aligned.

## Reproduction Assessment

Yes, from source inspection: `workspaceAccess: "ro"` and `"none"` still call `syncSkillsToWorkspace`, which copies skills into the sandbox workspace, while `"rw"` uses the agent workspace as `/workspace`. I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as filed. Read-only Docker bind mounts are the right primitive, but the maintainable fix needs secops-owned design across loader-derived mounts, duplicate copies, config hashing, FS bridge writability, and non-Docker backends.
