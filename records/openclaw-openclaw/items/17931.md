## Summary

Keep this issue open. Current main still lacks automatic read-only skill directory mounts: non-rw sandbox modes copy skills into a writable sandbox workspace, rw mode uses the agent workspace directly, and the existing read-only bind support only applies to configured binds rather than generated per-skill overlays.

## What This Changes

The issue requests kernel-enforced read-only skill directory mounts for sandboxed agents across all workspace access modes.

## Best Possible Solution

Design and implement a sandbox skill-mount contract that derives eligible skill source directories from the skill loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies when source mounts are available, updates FS bridge and config-hash modeling, defines non-Docker backend semantics, and documents/tests all workspaceAccess modes.

## Reproduction Assessment

Yes, at source level. Current main still routes non-rw sandbox sessions through copy-based skill staging and rw sessions through the agent workspace mount, with no generated per-skill read-only overlay path.

## Solution Assessment

Yes on the direction, but not as a narrow automated patch. Kernel-enforced read-only mounts derived from the existing skill loader are the maintainable shape; the exact always-on/configured behavior and non-Docker backend semantics need maintainer/security design.
