## Summary

Keep this open. Current main still copies eligible skills into writable sandbox workspaces for non-rw modes, exposes workspace skills through the writable workspace mount for rw mode, and has no generated read-only per-skill overlay mounts or matching fs-bridge/hash model.

## What This Changes

The issue requests read-only sandbox mounts for skill directories instead of writable workspace copies or writable workspace-mounted skills.

## Best Possible Solution

Keep this open for sandbox/secops design that derives eligible skill source directories, mounts them read-only across sandbox modes where supported, updates fs-bridge/hash/docs/tests, and explicitly defines non-Docker backend behavior.

## Reproduction Assessment

Yes, at source level: workspaceAccess ro/none still call syncSkillsToWorkspace into the sandbox workspace, and workspaceAccess rw still exposes workspace skills through the writable /workspace mount. I did not run a live Docker container, so this is source_reproducible rather than reproduced.

## Solution Assessment

Yes on direction: kernel-enforced read-only bind overlays are the right boundary for Docker and stronger than tool-layer guards. A complete solution still needs maintainer design for generated mount metadata, fs-bridge/config hashing, and SSH/OpenShell behavior.
