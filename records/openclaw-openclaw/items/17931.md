## Summary

Keep open: current main still has no generated read-only per-skill mount contract. In `rw` mode workspace skills remain under the writable `/workspace` mount, and non-`rw` modes still stage skills by copying into the sandbox workspace rather than deriving immutable source mounts.

## What This Changes

Request read-only bind mounts for skill directories in sandbox containers across all workspace access modes so agents cannot modify skill instruction material.

## Best Possible Solution

Implement a secops-reviewed sandbox contract that derives eligible skill source directories from the skills loader, mounts them read-only after broader workspace mounts, avoids writable duplicate skill copies, updates FS bridge and config-hash behavior, defines non-Docker behavior, and documents/tests all workspace access modes.

## Reproduction Assessment

Yes, from source inspection: `rw` still exposes workspace skills through the writable `/workspace` mount, and non-`rw` modes still copy skills into the sandbox workspace rather than generating source read-only mounts. I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as filed. Read-only Docker bind mounts are the right enforcement primitive, but the maintainable solution needs secops-owned design across loader-derived mounts, duplicate staging, config hashing, FS bridge writability, and non-Docker backends.
