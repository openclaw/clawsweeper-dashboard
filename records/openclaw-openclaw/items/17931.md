## Summary

Keep open: current main still has the writable-copy/writable-workspace behavior, and the closed bind-precedence fix in https://github.com/openclaw/openclaw/issues/22669 does not generate read-only skill mounts.

## What This Changes

The issue requests mounting skill directories into sandbox containers as read-only bind mounts instead of exposing them through writable workspace paths or writable copied sandbox workspaces.

## Best Possible Solution

Design the fix as sandbox/security work: derive eligible skill source directories from the skills loader, add read-only generated mounts after broader workspace mounts, avoid writable duplicate copies when mounts are available, include generated mounts in config hashing and FS bridges, define non-Docker backend behavior, and update docs/tests across all `workspaceAccess` modes.

## Reproduction Assessment

Yes, from source inspection: `workspaceAccess: "ro"`/`"none"` still calls `syncSkillsToWorkspace`, which copies eligible skill directories into the sandbox workspace, while `"rw"` uses the agent workspace as `/workspace`. I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as stated. Read-only bind mounts are the right Docker primitive, but the best maintainable solution needs a secops-owned design covering loader-derived mounts, config hash/FS bridge behavior, duplicate copies, docs, tests, and non-Docker backends.
