- `impact:security`: The reported behavior concerns sandbox boundaries and mutable instruction files inside sandboxed agent environments.

## Summary

Keep open. Current main still lacks generated read-only skill-directory mounts: non-`rw` sandboxes still copy eligible skills into the sandbox workspace, `rw` mode still exposes workspace skills through the writable workspace mount, and the issue remains a security-sensitive sandbox design item rather than stale cleanup.

## What This Changes

Request kernel-enforced read-only sandbox mounts for eligible skill directories instead of writable workspace exposure or copy-based skill staging.

## Best Possible Solution

Design and land an owner-reviewed sandbox skill-mount contract that derives eligible skill sources from the loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, and updates Docker hashing, fs-path authority, backend semantics, docs, and tests for every `workspaceAccess` mode.

## Reproduction Assessment

Yes, at source level. Current main still has no generated per-skill read-only overlay path, `rw` workspace skills remain under the writable workspace mount, and non-`rw` skill access is still copy-based; no live container run was performed.

## Solution Assessment

Yes on direction, but not as a narrow autonomous patch. Kernel-enforced read-only mounts derived from the skills loader look maintainable, while rollout and non-Docker backend semantics need maintainer/security design.
