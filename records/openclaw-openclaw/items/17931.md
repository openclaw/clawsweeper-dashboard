## Summary

Keep open. Current main still has the reported sandbox skill immutability gap: non-rw sandbox modes copy eligible skills into the sandbox workspace, rw mode exposes workspace skills through the writable workspace mount, and Docker still lacks generated per-skill read-only overlays.

## What This Changes

Request read-only bind mounts for each loaded skill directory in sandbox containers so agents cannot mutate skill instructions regardless of workspaceAccess mode.

## Best Possible Solution

Implement a maintainer-approved sandbox skill mount model that derives eligible skill source directories from the skills loader, appends read-only overlays after broader workspace mounts, avoids writable duplicate skill copies when source mounts are available, and aligns Docker, FS bridge, config hashing, SSH/OpenShell behavior, docs, and tests across all workspaceAccess modes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: in ro/none modes current main calls syncSkillsToWorkspace into the writable sandbox workspace, while in rw mode workspace skills remain under the writable /workspace mount; current tests also assert the ro sync path.

## Solution Assessment

Unclear as a final patch direction. Read-only bind overlays are the right Docker-level primitive, but the maintainable solution needs a secops-approved cross-backend design rather than a Docker-only mount append.
