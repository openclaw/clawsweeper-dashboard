## Summary

Keep open. Current main still uses writable skill copies for non-rw sandbox workspaces and writable workspace skill directories for rw mode, while the live discussion already marks this as a security-sensitive sandbox design item.

## What This Changes

Request automatic read-only skill directory mounts in sandbox containers so agents cannot mutate skill instruction files across workspaceAccess modes.

## Best Possible Solution

Design a sandbox skill-mount contract that derives eligible skill sources from the loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, and updates backend authority modeling, docs, and tests for every workspaceAccess mode.

## Reproduction Assessment

Yes, at source level. Current main still routes non-rw sandbox sessions through copy-based skill staging and rw sessions through the writable workspace mount, with no generated per-skill read-only overlay path.

## Solution Assessment

Yes on the direction, but not as a narrow autonomous patch. Kernel-enforced read-only mounts derived from the skills loader are the likely maintainable shape, while rollout and non-Docker backend semantics need maintainer/security design.
