## Summary

Keep open: current main still exposes skill material through writable copies or the writable workspace path, and the related bind-precedence work only made custom binds override workspace mounts; it did not add generated read-only skill directory overlays.

## What This Changes

Request kernel-enforced read-only exposure for skill directories inside sandbox containers across all `workspaceAccess` modes.

## Best Possible Solution

Design this as sandbox/security work: derive eligible skill source directories from the skills loader, append Docker read-only overlays after broader workspace mounts, avoid writable duplicate copies when source mounts are available, include generated mounts in config hashing and FS bridges, define non-Docker backend behavior, and update docs/tests across all `workspaceAccess` modes.

## Reproduction Assessment

Yes, from source inspection. `workspaceAccess: "ro"` and `"none"` still call `syncSkillsToWorkspace` into a writable sandbox workspace, while `"rw"` exposes workspace skills through the writable `/workspace` mount.

## Solution Assessment

Unclear as stated. Read-only bind overlays are the right Docker primitive, but the maintainable fix needs secops-owned design for loader-derived mounts, config hash and FS bridge behavior, duplicate copies, docs, tests, and non-Docker backends.
