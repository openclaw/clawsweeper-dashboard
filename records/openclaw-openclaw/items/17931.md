## Summary

Keep open. Current main still does not generate read-only skill directory mounts: non-rw sandbox modes still stage skills by copying them into the sandbox workspace, rw mode still uses the agent workspace directly, and the closed bind-precedence work at https://github.com/openclaw/openclaw/issues/22669 only made configured binds override workspace mounts rather than deriving per-skill overlays.

## What This Changes

Request to make sandboxed skill directories kernel-enforced read-only by generating per-skill bind mounts instead of exposing writable copies or workspace paths.

## Best Possible Solution

Design and implement a sandbox skill-mount contract that derives eligible skill source directories from the skill loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies when source mounts are available, updates FS bridge and config-hash modeling, defines non-Docker backend semantics, and documents/tests every workspaceAccess mode.

## Reproduction Assessment

Yes, at source level. Current main still routes non-rw sandbox sessions through copy-based skill staging and rw sessions through the agent workspace mount, with no generated per-skill read-only overlay path.

## Solution Assessment

Yes on the direction, but not as a narrow automated patch. Kernel-enforced read-only mounts derived from the existing skill loader are the maintainable shape, while always-on versus configurable behavior and non-Docker backend semantics need maintainer/security design.
