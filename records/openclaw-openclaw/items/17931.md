## Summary

Keep open. Current main and the latest release still lack automatic read-only skill directory mounts, and the live discussion already treats this as a security-sensitive sandbox design item rather than cleanup closure.

## What This Changes

Request read-only bind-mounted skill directories in sandbox containers across all workspaceAccess modes instead of writable workspace mounts or copied skill trees.

## Best Possible Solution

Design a sandbox skill-mount contract that derives eligible skill sources from the loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, and updates Docker/container hashing, fs-path authority, docs, and tests for every workspaceAccess mode.

## Reproduction Assessment

Yes, at source level. Current main still copies eligible skills into writable non-rw sandbox workspaces and exposes workspace skills through the writable rw workspace mount, with no generated per-skill read-only overlay path.

## Solution Assessment

Yes on the direction, but not as a narrow autonomous patch. Kernel-enforced read-only mounts derived from the skills loader look like the maintainable shape, while rollout and non-Docker backend semantics need maintainer/security design.
