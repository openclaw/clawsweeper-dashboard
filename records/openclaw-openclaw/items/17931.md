## Summary

Keep open: current main still has the writable skill-copy and writable workspace-skill behavior, and the related closed bind-precedence fix only makes explicit custom binds override broader workspace mounts rather than generating read-only skill mounts.

## What This Changes

The issue requests sandbox behavior that exposes eligible skill directories through read-only mounts instead of writable workspace copies or writable workspace paths.

## Best Possible Solution

Design this as sandbox/security work: derive eligible skill source directories from the skills loader, mount them read-only after broader workspace mounts, avoid writable duplicate copies when mounts exist, update hash and FS bridge contracts, define non-Docker behavior, and document/test all `workspaceAccess` modes.

## Reproduction Assessment

Yes, from source inspection: `workspaceAccess: "ro"` and `"none"` still call `syncSkillsToWorkspace`, which copies eligible skill directories into the sandbox workspace, while `"rw"` uses the agent workspace as `/workspace`. I did not run a live Docker write attempt in this read-only review.

## Solution Assessment

Unclear as stated. Read-only bind mounts are the right Docker primitive, but the maintainable fix needs secops-owned design across loader-derived mounts, duplicate copies, config hashing, FS bridge writability, and non-Docker backends.
