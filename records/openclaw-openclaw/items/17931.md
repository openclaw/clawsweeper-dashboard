## Summary

Keep open. Current main still stages skills as writable copies for non-rw sandbox modes and exposes workspace skills through the writable workspace mount for rw mode; the closed bind-precedence work at https://github.com/openclaw/openclaw/issues/22669 only makes configured binds override workspace mounts, not generated per-skill read-only overlays.

## What This Changes

Request to replace writable sandbox skill copies/directories with read-only skill directory mounts across sandbox workspace access modes.

## Best Possible Solution

Design a sandbox skill-mount contract that derives eligible skill source directories from the skill loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies when source mounts exist, updates FS bridge/config-hash modeling, defines non-Docker backend behavior, and documents/tests every workspaceAccess mode.

## Reproduction Assessment

Yes, at source level. Current main still routes non-rw sandbox sessions through copy-based skill staging and rw sessions through the agent workspace mount, with no generated per-skill read-only overlay path.

## Solution Assessment

Yes on the direction, but not as a narrow automated patch. Kernel-enforced read-only mounts derived from the existing skill loader are the maintainable shape, while always-on versus configurable behavior and non-Docker backend semantics need maintainer/security design.
