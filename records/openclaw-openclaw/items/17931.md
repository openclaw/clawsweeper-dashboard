## Summary

Keep this issue open. Current main still does not generate read-only skill directory mounts: non-rw sandbox modes still copy eligible skills into the sandbox workspace, rw mode still exposes workspace skills through the writable workspace mount, and the Docker custom-bind precedence work only helps manually configured binds rather than deriving per-skill read-only overlays.

## What This Changes

Request read-only bind mounts for each loaded skill directory in sandbox containers so agents cannot mutate skill instructions regardless of workspaceAccess mode.

## Best Possible Solution

Implement a maintainer-approved sandbox skill mount model that derives eligible skill source directories from the skills loader, appends Docker read-only overlays after broader workspace mounts, avoids writable duplicate skill copies when source mounts are available, and aligns FS bridge, config hashing, backend docs, and tests across all workspaceAccess modes.
