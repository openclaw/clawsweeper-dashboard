## Summary

Keep this open. Current main still has copy-based skill staging for non-rw sandbox workspaces, writable workspace skill paths in rw mode, and no generated read-only per-skill mount contract across Docker, fs bridges, config hashing, docs, or non-Docker backends.

## What This Changes

The issue requests kernel-enforced read-only mounting of skill directories inside sandbox containers across all workspaceAccess modes.

## Best Possible Solution

Define a secops-reviewed sandbox contract that derives eligible skill source directories from the skills loader, mounts or otherwise exposes them read-only where supported, avoids writable duplicates, and updates fs-bridge/hash/docs/tests plus non-Docker backend behavior.

## Reproduction Assessment

Yes, at source level: current main still copies eligible skills into non-rw sandbox workspaces and exposes workspace skills through the writable rw workspace mount. I did not run a live Docker container, so this is source_reproducible rather than reproduced.

## Solution Assessment

Yes on direction: kernel-enforced read-only bind overlays are the right Docker boundary and stronger than tool-layer write guards. A complete fix still needs maintainer design for generated mount metadata, fs-bridge/config hashing, and SSH/OpenShell behavior.
