## Summary

Keep open. Current main still copies eligible skills into writable sandbox workspaces for non-rw modes and exposes workspace skills through the rw workspace mount, while the prior live reviews also kept this open as a security-sensitive sandbox design item.

## What This Changes

Request read-only sandbox mounts for skill directories instead of writable skill copies or writable workspace skill paths across workspaceAccess modes.

## Best Possible Solution

Design a sandbox skill-mount contract that derives eligible skill source directories from the skill loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies when source mounts are available, and updates FS bridge, config hashing, docs, and tests for every backend and workspaceAccess mode.

## Reproduction Assessment

Yes, at source level. Current main still routes non-rw sandbox sessions through copy-based skill staging and rw sessions through the writable agent workspace mount, with no generated per-skill read-only overlay path.

## Solution Assessment

Yes on the direction, but not as a narrow autonomous patch. Kernel-enforced read-only mounts derived from the skill loader are the maintainable shape, while always-on versus configurable behavior and non-Docker backend semantics need maintainer/security judgment.
