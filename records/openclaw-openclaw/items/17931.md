## Summary

Keep open. Current main still leaves sandbox-visible skill directories writable in the reported paths, and prior maintainer review already routed this to sandbox/security design rather than cleanup closure.

## What This Changes

Request to make sandbox-visible skill directories kernel-enforced read-only by bind-mounting eligible skill sources instead of copying them or exposing them through writable workspace mounts.

## Best Possible Solution

Define an approved sandbox skill-source mount model that derives eligible skill directories from the loader, exposes them read-only without writable duplicates, and updates Docker, fs bridges, config hashing, backend docs, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ro/none modes call syncSkillsToWorkspace into the sandbox workspace, while rw mode exposes workspace skills through the writable /workspace mount.

## Solution Assessment

No current implementation solves it. Read-only Docker overlays are likely the right primitive for Docker, but the best fix needs a cross-backend sandbox contract rather than a Docker-only patch.
