## Summary

Keep open. Current main still has no generated read-only skill-directory mount contract: `rw` mode exposes workspace skills through the writable workspace mount, non-`rw` modes still use copy-based skill staging, and the current release docs still describe mirrored/workspace-mounted skill access. Prior discussion correctly treats this as security-sensitive sandbox design rather than stale cleanup.

## What This Changes

Request to make sandbox-provided skill directories immutable by bind-mounting eligible skill sources read-only across all sandbox `workspaceAccess` modes.

## Best Possible Solution

Design and land an owner-reviewed sandbox skill-mount contract that derives eligible skill sources from the loader, mounts them read-only after broader workspace mounts, avoids writable duplicate copies, and updates Docker hashing, fs-path authority, backend semantics, docs, and tests for every `workspaceAccess` mode.

## Reproduction Assessment

Yes, at source level. Current main still has no generated per-skill read-only overlay path, `rw` workspace skills remain under the writable workspace mount, and non-`rw` skill access is still copy-based; no live container run was performed.

## Solution Assessment

Yes on the direction, but not as a narrow autonomous patch. Kernel-enforced read-only mounts derived from the skills loader look like the maintainable shape, while rollout and non-Docker backend semantics need maintainer/security design.
