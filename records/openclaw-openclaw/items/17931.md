## Summary

Keep open. Current main still exposes sandbox skills as writable copies in non-rw modes or through the writable workspace in rw mode, and existing maintainer discussion already routes this to secops design rather than cleanup closure.

## What This Changes

Request read-only sandbox mounting for eligible skill directories instead of writable workspace copies or writable workspace exposure.

## Best Possible Solution

Implement an approved sandbox skill mount model that derives eligible skill sources from the skills loader, installs read-only overlays after broader mounts, avoids writable duplicates, and updates Docker, remote backends, fs bridges, hashing, docs, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ro/none modes call syncSkillsToWorkspace into a writable sandbox workspace, while rw mode still exposes workspace skills through the writable /workspace mount.

## Solution Assessment

Unclear as a final patch. Read-only bind overlays are the right Docker primitive, but the maintainable solution needs secops-approved cross-backend semantics rather than a Docker-only change.
