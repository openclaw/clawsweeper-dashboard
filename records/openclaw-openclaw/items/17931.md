## Label Justifications

- `P2`: The issue describes a concrete sandbox hardening gap for sandbox-enabled users, but it is not a confirmed active exploit, crash loop, or outage.
- `impact:security`: The requested behavior concerns sandbox boundaries and mutable skill instruction files inside sandboxed agent environments.

## Summary

Keep open. Current main and the latest release still do not generate read-only skill directory mounts; non-rw sandbox modes still stage writable skill copies, rw mode still exposes workspace skills through the writable workspace mount, and the fs bridge/config-hash path has no generated skill-mount authority model.

## What This Changes

The issue asks OpenClaw to mount eligible sandbox skill directories read-only instead of copying them into writable sandbox workspaces or exposing them through writable workspace mounts.

## Best Possible Solution

Design and land an owner-reviewed sandbox skill-mount contract that derives eligible skill source directories from the skills loader, mounts them read-only after broader workspace mounts, avoids writable duplicates, and updates hashing, fs-path authority, backend docs, and tests across every `workspaceAccess` mode.

## Reproduction Assessment

Yes, at source level. Current main still has no generated per-skill read-only overlay path, rw workspace skills remain under the writable workspace mount, and non-rw skill access is still copy-based; no live container run was performed in this read-only review.

## Solution Assessment

Yes on direction, but not as a narrow autonomous patch. Kernel-enforced read-only mounts derived from the skills loader look like the right end state, while rollout, config hashing, fs bridge behavior, and non-Docker backend semantics need maintainer/security design.
