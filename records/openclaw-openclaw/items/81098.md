## What This Changes

The PR catches ENOTSUP from session usage-cost lock hard-link creation and falls back to exclusive-create lock-file creation with `writeFile(..., { flag: "wx" })`.

## Best Possible Solution

Keep the existing lock protocol, add the narrow ENOTSUP exclusive-create fallback while preserving EEXIST and stale-lock behavior, and pair it with real behavior proof plus focused regression coverage if practical.

## Reproduction Assessment

Yes, the linked issue provides a concrete reproduction class and ENOTSUP log, and current main shows an unhandled `fs.promises.link` path. I did not run a live SMB/NFS/virtiofs mount in this read-only review, so the reproduction is source-backed rather than locally executed.

## Solution Assessment

Yes, trying the existing hard-link path first and falling back to `wx` exclusive create only on ENOTSUP is a narrow maintainable fix for the reported failure. A focused regression test would make the branch safer, but I do not see a blocking correctness issue in the patch itself.
