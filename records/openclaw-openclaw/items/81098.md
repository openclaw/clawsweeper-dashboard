## Summary

Keep open: this is a plausible narrow fix for the source-proven ENOTSUP crash path, but this external contributor PR still needs after-fix real behavior proof before merge.

## What This Changes

The PR catches ENOTSUP from the usage-cost cache hard-link lock publish step and falls back to writing the lock file with flag "wx".

## Best Possible Solution

Land a tested unsupported-hard-link fallback for the usage-cost refresh lock, ideally with a mocked ENOTSUP regression plus real setup proof showing the cache refresh no longer crashes.

## Reproduction Assessment

Yes at source level: current main calls fs.promises.link for the lock and rethrows non-EEXIST errors, so an ENOTSUP from that call will escape. I did not run a live unsupported-filesystem reproduction in this read-only review.

## Solution Assessment

Yes, the proposed fallback is the narrowest code path for the reported crash and preserves the existing stale-lock and EEXIST handling. It should still be backed by regression coverage and real unsupported-filesystem proof before merge.
