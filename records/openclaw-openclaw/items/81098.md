## Summary

Keep open: the patch is a narrow, plausible fix for the source-proven ENOTSUP crash path, but this external PR still lacks the required after-fix real behavior proof before merge.

## What This Changes

Adds an ENOTSUP catch in `writeUsageCostCacheLockAtomically` that falls back from hard-link lock creation to `writeFile(..., { flag: "wx" })`.

## Best Possible Solution

Keep the existing lock protocol, add the narrow unsupported-hard-link fallback, and merge only after contributor-supplied real behavior proof shows the usage-cost refresh path succeeding on an ENOTSUP filesystem.

## Reproduction Assessment

Yes, source-backed but not live-executed: the linked issue provides an ENOTSUP lock-creation log, and current main rethrows non-EEXIST errors from `fs.promises.link`. I did not mount SMB/NFS/virtiofs in this read-only review.

## Solution Assessment

Yes, the PR's hard-link-first and ENOTSUP-only `wx` fallback is a narrow maintainable fix for the reported crash path. A focused regression test would reduce risk, but I do not see a blocking correctness bug in the diff.
