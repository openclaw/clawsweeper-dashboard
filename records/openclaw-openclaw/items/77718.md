## Summary

Keep open: current main already handles a missing agent workspace by falling back through the agent workspace resolver, but `workspace: null` still fails schema validation and the doctor repair path only attempts prefixed-config or last-known-good recovery, not a targeted agent workspace repair. The related restart-throttle PR is a separate mitigation for restart cadence, not the malformed config root cause.

## What This Changes

Request that null or missing `agents.list[].workspace` entries no longer trap the gateway in an invalid-config restart loop and that doctor repair can recover the config.

## Best Possible Solution

Add a targeted `openclaw doctor --fix` repair that treats `agents.list[].workspace: null` as absent or restores the effective default workspace, while keeping gateway startup validation fail-closed for untrusted config edits.

## Reproduction Assessment

Yes from source inspection: current main rejects `agents.list[].workspace: null` at schema validation, then gateway startup throws on the invalid snapshot. I did not run the external service restart loop, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes for a doctor repair, not for a startup guard. The maintainable direction is to repair this known malformed optional field in `openclaw doctor --fix` and leave runtime startup validation strict.
