## Summary

Keep open: current main handles an omitted agent workspace via fallback resolution, but a literal `workspace: null` still fails schema validation before gateway startup and doctor recovery still has no targeted repair. The related restart-throttle PR only mitigates restart cadence, not the malformed config root cause.

## What This Changes

The issue asks OpenClaw to recover from `agents.list[].workspace` being null or missing so gateway startup does not remain in a crash loop after doctor repair.

## Best Possible Solution

Add a targeted `openclaw doctor --fix` repair for `agents.list[].workspace: null` while preserving strict gateway startup validation for invalid direct config edits.

## Reproduction Assessment

Yes, from source inspection: set `agents.list[].workspace` to null and current schema validation rejects it before gateway startup proceeds. An omitted workspace alone is not a repro because the resolver already falls back to default or state-dir workspaces.

## Solution Assessment

Yes for a doctor repair, not for startup schema relaxation. The narrow maintainable fix is to repair this known malformed optional field in `openclaw doctor --fix` and keep runtime startup fail-closed for invalid config.
