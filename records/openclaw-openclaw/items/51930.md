## Summary

Keep open. Current main still emits the OpenClawBrain manifest/package mismatch path, and the PR is a relevant narrow fix, but it is dirty against current main and needs maintainer judgment on whether to accept a core hard-coded plugin alias or move the equivalence into one shared compatibility helper.

## What This Changes

The PR maps `@openclawbrain/openclaw` to the canonical `openclawbrain` plugin id during discovery, suppresses the install-time mismatch info log for that known pair, and adds targeted discovery/install tests.

## Best Possible Solution

Keep manifest ids as the canonical install/config key. If OpenClawBrain compatibility is accepted, implement the package equivalence through one shared helper or manifest/registry-backed table used by both install warning suppression and package-derived discovery, with tests that exercise the actual alias branch.

## Reproduction Assessment

Yes. On current main, a native plugin with package name `@openclawbrain/openclaw` and manifest id `openclawbrain` follows the install path into `packageNameMatchesId(...) === false`, producing the info log; the discovery alias path needs a manifestless package-derived fixture because manifest ids win first.

## Solution Assessment

Unclear. The PR addresses the user-visible install chatter narrowly, but the duplicated alias tables and core plugin-specific exception are not obviously the best long-term contract; a shared equivalence helper or metadata-backed compatibility contract would be safer if maintainers accept this behavior.
