## Summary

Keep this PR open. The install failure is valid on current main, but the proposed override fixes it by switching Baileys from the WhiskeySockets libsignal fork to the older npm `libsignal` package, which is a WhatsApp signal-crypto runtime and supply-chain change not covered by the new string-only guardrail test.

## What This Changes

The PR adds a scoped pnpm override and lockfile refresh so Baileys resolves `libsignal` from npm, plus a changelog entry and lockfile guardrail test.

## Best Possible Solution

Preserve Baileys' expected WhiskeySockets libsignal implementation while making installs registry-safe, with crypto/runtime regression coverage and the normal changed gate before merge.

## Reproduction Assessment

Yes, from source and reporter evidence. Current main still locks Baileys' `libsignal` to a GitHub codeload tarball, and the linked report gives the pnpm `ERR_PNPM_EXOTIC_SUBDEP` path; I did not run an install because this review is read-only.

## Solution Assessment

No. The PR removes the exotic subdependency, but it does so by substituting a different libsignal package/fork in the WhatsApp crypto path; the safer solution needs to keep the WhiskeySockets fork semantics or prove the replacement is runtime-equivalent.
