## Summary

This should stay open: current main and the latest shipped tag still intentionally force Discord room events into message-tool-only delivery, while an open related PR is exploring a guarded fallback and the remaining decision is product policy, not cleanup-closeable implementation drift.

## What This Changes

The issue asks for Discord room/thread events to avoid transcript-only final assistant replies when configured visible-reply policy or response-owed activity implies a visible Discord response.

## Best Possible Solution

Maintain the issue as the canonical tracking item while maintainers decide the room-event contract, then either land a narrow tested guard/fallback or reaffirm the strict ambient-room behavior in docs and diagnostics.

## Reproduction Assessment

Yes for source-level reproduction: current source and tests show a Discord room_event with groupChat.visibleReplies set to automatic still resolves to message_tool_only and suppresses normal final delivery. I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No for the broad hotfix described in the issue body: removing every room-event hard force conflicts with the current ambient-room contract. The safer path is a maintainer-approved, narrow fallback or diagnostic change with Discord regression coverage.
