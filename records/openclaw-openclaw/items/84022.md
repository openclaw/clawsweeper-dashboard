## Summary

Keep open: current main and the latest release still enforce documented strict `message_tool_only` delivery for `room_event`, even when group visible replies are configured as automatic. The related open PR at https://github.com/openclaw/openclaw/pull/84485 is still under review and intentionally excludes `room_event`, so this remains a maintainer delivery-policy decision rather than cleanup-closeable.

## What This Changes

The issue requests that Discord room/thread events configured for automatic visible replies stop forcing final assistant text into transcript-private `message_tool_only` delivery.

## Best Possible Solution

Maintainers should decide the `room_event` contract: either reaffirm the documented quiet ambient-room behavior, or approve a narrow fallback with Discord regression coverage, privacy/message-delivery tests, docs, and live proof.

## Reproduction Assessment

Yes. Source and tests show that a Discord `room_event` with `messages.groupChat.visibleReplies: "automatic"` still resolves to `message_tool_only`; I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No. Removing every room-event hard-force is not clearly the best fix because current docs and tests intentionally keep ambient room events quiet; a safer path needs maintainer policy input and then a narrow fallback or documentation clarification.
