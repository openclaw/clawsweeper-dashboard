## Summary

Keep this open: current main still has the reported room-event suppression path, but source, docs, and tests also show it is an intentional ambient-room policy that needs maintainer judgment rather than cleanup closure or a safe automatic fix.

## What This Changes

The issue asks OpenClaw to prevent Discord room/thread events from leaving a final assistant answer transcript-only when room events force message-tool-only delivery.

## Best Possible Solution

Maintainers should decide the room_event contract, then either reaffirm the current quiet ambient-room behavior in docs or land a narrow Discord guard that prevents accidental transcript-only finals without making ambient rooms chatty.

## Reproduction Assessment

Yes. Current source and tests show that a Discord room_event with messages.groupChat.visibleReplies set to "automatic" still resolves to message_tool_only and suppresses normal final delivery unless an explicit bypass payload is present; I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No for the issue’s hotfix shape. Removing every room_event hard-force conflicts with current docs/tests for quiet ambient rooms; the safer path is a maintainer-approved narrow fallback or the linked Discord guard approach with live proof and regression coverage.
