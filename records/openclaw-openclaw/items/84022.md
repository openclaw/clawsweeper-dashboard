## Summary

Keep open: current main deliberately keeps `room_event` turns message-tool-only even when group visible replies are automatic, so this is a real, source-reproducible Discord delivery concern but not a safe cleanup close. Changing it needs maintainer policy judgment because a broad hotfix would undo the ambient-room quiet semantics that were recently merged and documented.

## What This Changes

The issue asks for Discord room/thread events to deliver final assistant text when visible replies are configured as automatic, or to provide a visible fallback/failure when message-tool delivery is missed.

## Best Possible Solution

Preserve intentional ambient-room silence by default, then either fix any Discord misclassification that turns normal user requests into room events or add an explicit maintainer-approved automatic/fallback policy for room events with tests and docs.

## Reproduction Assessment

Yes, source-reproducible for the suppression path: current main forces `room_event` turns to `message_tool_only`, Discord passes the inbound event kind into that resolver, and tests assert automatic group config is still suppressed for room events. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No, the proposed broad dist hotfix is not the best safe fix because it conflicts with documented ambient-room behavior. The maintainable path is to first decide whether this was misclassification or an intentional policy change, then update the narrow classification or room-event fallback contract with focused coverage.
