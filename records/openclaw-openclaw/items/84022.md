## Summary

Keep open: current main and the latest release still make Discord room events strict message-tool-only, while the remaining fix path is an explicit maintainer policy/guard decision now represented by an open related PR.

## What This Changes

The issue requests that Discord room/thread events not silently suppress final assistant text when visible replies are configured for automatic delivery, or that they emit a visible fallback/failure.

## Best Possible Solution

Review the policy and the open guard proposal at https://github.com/openclaw/openclaw/pull/84485, then either land a tested fallback that preserves quiet ambient rooms or reaffirm the strict room_event contract in docs and diagnostics.

## Reproduction Assessment

Yes, at source level: current main resolves a Discord room_event with groupChat.visibleReplies set to "automatic" to message_tool_only, and the current test suite encodes that expectation. I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No, the local hotfix of removing every room_event hard-force is not clearly the best solution because it conflicts with the documented ambient-room contract. The safer path is a maintainer-approved narrow guard or an explicit decision to keep strict room-event behavior.
