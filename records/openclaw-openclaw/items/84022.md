## Summary

Keep open: current main and v2026.6.8 still force external room events into message-tool-only delivery, and delegated ACP parent-review completions still queue through the requester session instead of visibly posting to the bound Discord thread. The open linked PR https://github.com/openclaw/openclaw/pull/89279 owns the narrow ACP bound-thread path, while the broader room-event visibility/fallback policy still needs maintainer judgment.

## What This Changes

The issue requests visible Discord final replies or an explicit visible fallback when room/thread events leave final assistant text only in transcript/session state.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/89279 for the bound Discord thread ACP path, then keep the broader room_event visible-reply policy on an explicit maintainer decision track.

## Reproduction Assessment

Yes, at source level. Current resolver, dispatch, docs, and tests still force external room_event turns to message_tool_only, and current task-registry tests still queue delegated ACP completions through the requester session instead of direct Discord thread delivery.

## Solution Assessment

Unclear. Removing every room_event hard-force is not the best proven fix because current docs intentionally keep ambient rooms strict; the maintainable path is a narrow ACP delivery fix plus an explicit policy decision for any broader fallback.
