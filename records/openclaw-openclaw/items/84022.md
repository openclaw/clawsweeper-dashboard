## Summary

Keep this open: current main and the latest shipped tag still make room events strict `message_tool_only`, and the linked fix PR is open/conflicting with unresolved delivery-policy risk rather than merged implementation evidence.

## What This Changes

The issue asks OpenClaw to prevent Discord room/thread events from ending with final assistant text only in the transcript when visible replies are configured automatic.

## Best Possible Solution

Maintainers should settle the room-event delivery contract, then either keep the documented strict behavior with clearer diagnostics or land a narrow fallback that preserves quiet ambient rooms while preventing unintended transcript-only conversational replies.

## Reproduction Assessment

Yes. Source and tests show that a Discord `room_event` with `messages.groupChat.visibleReplies: "automatic"` still resolves to `message_tool_only`, so normal final delivery is suppressed; I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No. The reported dist hotfix removes the room-event hard-force broadly, but current docs and tests intentionally keep ambient room events quiet; a safer solution needs maintainer policy approval plus targeted Discord regression coverage.
