## Summary

Keep open: current main and v2026.6.6 still keep external room events strict message-tool-only, and current main still queues successful ACP parent-review completions through the requester session. ## What This Changes

The issue asks OpenClaw to prevent Discord room/thread events from leaving completed assistant or ACP replies visible only in the transcript when visible replies are configured or expected.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/89279 for the bound Discord thread ACP path, then make a separate maintainer decision on whether ambient room_event policy should stay strict or gain a narrow visible fallback.

## Reproduction Assessment

Yes. Current source and tests show room_event plus automatic visible replies still resolves to message_tool_only, and current task-registry code queues successful ACP parent-review completions through the requester session instead of directly delivering to the bound Discord thread; I did not run a live Discord gateway repro in this read-only review.

## Solution Assessment

Unclear. Removing every room_event hard-force is not the best proven fix because docs/tests intentionally keep ambient rooms quiet; the maintainable path is a narrow ACP delivery fix plus an explicit maintainer policy decision for any broader fallback.
