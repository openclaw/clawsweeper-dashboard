## Summary

Keep open: current main and v2026.6.9 still leave memory-search config resolution on the legacy memory-provider lookup while the official llama.cpp `local` provider is registered through the generic embedding-provider seam; https://github.com/openclaw/openclaw/pull/94355 is the open candidate fix and has not merged.

## What This Changes

The issue reports that in-session `memory_search` cannot resolve `memorySearch.provider: "local"` from an installed llama.cpp provider even though CLI/plugin status shows local embeddings as ready.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94355, or an equivalent resolver fix, with focused regression coverage and redacted installed-plugin or in-session `memory_search` proof.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 can miss a `local` provider registered only through `registerEmbeddingProvider` because config resolution checks only the legacy memory registry before alias fallback. I did not run a live installed-plugin Gateway reproduction in this read-only review.

## Solution Assessment

Yes, pending live proof: making memory-search config resolution consult the same generic embedding-provider registry used by runtime creation is the narrowest maintainable fix and avoids pushing llama.cpp back onto the deprecated memory-specific seam.
