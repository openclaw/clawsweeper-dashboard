## Summary

Keep open: the older `local` provider CLI failure is already closed, but this report describes the opposite runtime split on 2026.6.8, where CLI/status can see the official llama.cpp provider while the live agent tool still throws the unknown-provider error. Current main has the intended startup and adapter paths, but I found no merged fix or canonical open issue that proves this specific in-session failure is resolved.

## What This Changes

The issue reports that `memory_search()` fails with `Unknown memory embedding provider: local` even though `openclaw memory status --deep` and `plugins list` show the llama.cpp local embedding provider as ready.

## Best Possible Solution

Make the live Gateway and CLI use the same provider activation/registration path for the official llama.cpp plugin, then add a regression test that exercises `memory_search` with `memorySearch.provider: "local"` after a cold Gateway startup.

## Reproduction Assessment

Yes, at source level with medium confidence: the issue provides concrete config and commands, and current code has the exact throw path when `local` is absent from runtime provider registries. I did not live-run a packaged llama.cpp Gateway setup in this read-only review.

## Solution Assessment

Unclear: startup/registration parity is the likely repair boundary, but current source already intends to load the llama.cpp owner for configured local memory embeddings. A live reproducer should confirm which runtime inventory path is stale before changing code.
