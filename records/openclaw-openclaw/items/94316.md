## Summary

## What This Changes

The issue reports in-session `memory_search` failing to resolve the official llama.cpp local embedding provider even though CLI/plugin status reports it ready.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94355, or an equivalent config-resolver fallback, with focused regression coverage and redacted installed-plugin or in-session `memory_search` proof.

## Reproduction Assessment

Yes at source level with medium confidence: current main still shows the registry split between memory-search config resolution and runtime provider creation, and the reporter supplied a concrete 2026.6.8 installed-plugin setup. I did not live-run an installed llama.cpp Gateway or in-session `memory_search` path in this read-only review.

## Solution Assessment

Yes, pending proof: making memory-search config resolution consult the same generic embedding-provider registry used by runtime creation is the narrowest maintainable fix. The safer path is to land the focused fallback after installed-plugin proof rather than making llama.cpp dual-register on the deprecated memory-specific seam.
