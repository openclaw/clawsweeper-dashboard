## Summary

Keep open: current main still has the memory-search config resolver gap reported here, and the linked fix is an open PR rather than a merged implementation. Source and docs show `local` is now provided through the generic embedding-provider registry, while `src/agents/memory-search.ts` still derives provider metadata from the legacy memory registry only.

## What This Changes

The issue reports that in-session `memory_search` cannot resolve the `local` llama.cpp embedding provider even though CLI memory status reports the provider as ready.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/94355 or an equivalent narrow fix lands, with regression coverage and preferably live installed-plugin proof for `memorySearch.provider: "local"`.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a medium-confidence reproduction path: a generic-only `local` provider can be registered by llama.cpp while current memory-search config resolution still reads only the legacy memory registry.

## Solution Assessment

Yes in direction: the best fix is to align memory-search config lookup with the generic embedding-provider registry already used by the memory-core runtime bridge. The open linked PR targets that boundary, but should finish normal PR review and proof before the issue closes.
