## Summary

Keep open: the docs and plugin manifest say `memorySearch.provider: "local"` is supported through the official llama.cpp provider, but the current in-session memory-core path can still throw the reported missing-provider error when the generic `local` provider is not visible to the tool runtime. No merged current-main or release-backed fix was found for this specific CLI-ready but in-session-failing case.

## What This Changes

The issue asks for in-session `memory_search()` to resolve the installed `local` llama.cpp embedding provider when CLI memory status reports embeddings ready.

## Best Possible Solution

Make the in-session memory-core tool consume the same loaded generic `local` embedding provider that CLI status proves ready, with a focused regression test for an installed/enabled generic provider in the agent tool path.

## Reproduction Assessment

Unclear for a live high-confidence reproduction because I did not execute the external llama.cpp model path. Source inspection does identify a concrete reproduction boundary: CLI can show the generic `local` provider loaded while in-session `memory_search()` still reaches the memory-core missing-provider branch if that generic provider is not visible to the tool runtime.

## Solution Assessment

Yes, the best fix is a narrow runtime/provider lookup repair that keeps ownership in the generic embedding provider/plugin contracts. A core special case for llama.cpp or a new config option would be the wrong layer.
