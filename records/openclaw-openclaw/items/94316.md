## Summary

Keep open: current main and the latest release still have the memory-search config resolver using the legacy memory embedding registry while the official llama.cpp provider registers `local` through the generic embedding-provider registry; the focused candidate fix is open but not merged.

## What This Changes

The issue reports that in-session `memory_search()` cannot resolve the local llama.cpp embedding provider even though CLI/plugin status shows the provider as loaded and ready.

## Best Possible Solution

Review and land the focused registry-fallback fix in https://github.com/openclaw/openclaw/pull/94355, or an equivalent fix, with regression coverage and redacted installed-plugin or in-session `memory_search` proof.

## Solution Assessment

Yes, pending proof: making memory-search config resolution consult the same generic embedding-provider registry used by runtime creation is the narrowest maintainable fix, and it avoids making llama.cpp dual-register on the deprecated memory-specific seam.
