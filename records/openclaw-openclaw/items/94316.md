## Summary

Keep open: current main still leaves memory-search config resolution on the legacy memory-provider lookup, while the official llama.cpp `local` provider registers through the generic embedding-provider seam; https://github.com/openclaw/openclaw/pull/94355 remains the open candidate fix.

## What This Changes

The issue reports that in-session `memory_search` cannot resolve `memorySearch.provider: "local"` from the installed llama.cpp embedding plugin even though CLI status sees embeddings as ready.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94355, or an equivalent resolver fix, with focused regression coverage and redacted installed-plugin or in-session `memory_search` proof.

## Reproduction Assessment

Yes, at source level: current main can miss a `local` provider registered only through `registerEmbeddingProvider` because config resolution checks the legacy memory registry and owner aliases before model and transport defaults. I did not establish a live installed-plugin Gateway reproduction in this read-only review.

## Solution Assessment

Yes, pending live proof: making memory-search config resolution consult the same generic embedding-provider registry used by runtime creation is the narrowest maintainable fix and avoids pushing llama.cpp back onto the deprecated memory-specific seam.
