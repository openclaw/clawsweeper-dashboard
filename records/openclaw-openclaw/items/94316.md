## Summary

Keep open: current main and the latest stable release still resolve memory-search provider metadata through the legacy memory registry path, while the official llama.cpp plugin registers `local` through the generic embedding-provider registry. The linked fix PR is still open and not on main, so this issue should stay paired with that review path rather than be closed.

## What This Changes

The issue requests that in-session `memory_search` resolve the configured local llama.cpp embedding provider when CLI status and plugin listing show it as ready.

## Best Possible Solution

Land a narrow fix that makes memory-search config resolution consult the same generic provider registry path used by runtime provider creation, then verify it with a focused regression test and redacted installed-plugin or in-session `memory_search` proof.
