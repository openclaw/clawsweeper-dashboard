## Summary

Keep open. Current main partially supersedes the old per-chat queue shape by routing Feishu through `createFeishuMessageReceiveHandler` and giving `/stop` and `/btw` out-of-band sequential keys, but it does not implement this PR's broader `/new` or `/status` command bypass. The PR is also dirty against current main and still has blocking queue-ordering and authorization-adjacent defects, so it is not conservative cleanup material.

## What This Changes

The PR branch changes Feishu inbound handling to detect text control commands before debounce, dispatch them directly outside the per-chat queue, add a shared inbound debounce `dropKey` API, and update Feishu test mocks.

## Best Possible Solution

Rebase or replace the work on the current `createFeishuMessageReceiveHandler` plus sequential-key design: explicitly classify only commands that should be out-of-band, keep destructive cleanup authorization-aware, preserve normal FIFO/debounce behavior for ordinary commands, and add focused tests for pending debounce buffers and already queued same-chat work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: with the PR branch, queue normal Feishu work behind an active run, then send `/stop`, `/new`, or `/reset`; older same-chat work already in the queue can still run afterward, and an unauthorized destructive command can drop buffered text before normal authorization rejects it.

## Solution Assessment

No. The direct-dispatch plus pre-authorization debounce-drop approach is not the safest maintainable fix; extending the current sequential-key policy with authorization-aware cleanup and explicit command classification is the better path.

## Review Findings

  - body: This direct path only drops the debounce buffer before running `/stop`, `/new`, or `/reset`. Messages that already flushed into the per-chat queue still run after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
  - confidence: 0.86
