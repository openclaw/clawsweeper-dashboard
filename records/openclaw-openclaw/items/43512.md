## Summary

Close as superseded. Current main moved Feishu ingress to the sequential-key design from merged #64324, which shipped the safe `/stop` and `/btw` out-of-band lanes, while the broader `/new`/`/status` queue-fast-path request remains tracked by open #42803 and related Feishu queue issues; this stale branch still has blocking queue and authorization regressions.

## What This Changes

The PR branch changes Feishu inbound handling to detect text control commands before debounce, dispatch them directly outside the per-chat queue, add a shared inbound debounce `dropKey` API, and update Feishu test mocks.

## Best Possible Solution

Keep the shipped Feishu abort/BTW sequential-key implementation, close this branch, and use the canonical Feishu queue issues for any broader current-main policy change with explicit command classification, authorization-aware cleanup, and focused queue/debounce tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists for the PR blockers: queue same-chat Feishu work behind an active run, then send `/stop`, `/new`, or `/reset`; older queued work can still run after the direct command, and an unauthorized destructive command can drop buffered text before normal authorization rejects it.

## Solution Assessment

No. The PR’s direct-dispatch plus pre-authorization debounce-drop approach is not the best fix; the safer maintainable direction is the current sequential-key seam with explicit out-of-band classifications and authorization-aware behavior for any future expansion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear queued chat work before destructive commands:** `extensions/feishu/src/monitor.account.ts:441`
  - body: This direct path only drops the debounce buffer before running `/stop`, `/new`, or `/reset`. Messages that already flushed into the per-chat queue still run after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
  - confidence: 0.86
- **[P2] Gate destructive drops on command authorization:** `extensions/feishu/src/monitor.account.ts:438`
  - body: `dropKey` runs from text classification before `handleFeishuMessage` computes command authorization. In chats where a sender can speak but cannot run destructive slash commands, an unauthorized `/new`, `/reset`, or `/stop` can still discard pending debounced text.
  - confidence: 0.82
