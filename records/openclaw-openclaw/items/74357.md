## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools endpoint used by bundled web-search helpers and plugin-facing SDK exports, and the paired implementation PR #74360 remains open rather than merged.

## What This Changes

The issue requests splitting web-tools SSRF policy so public web-search providers use strict defaults while self-hosted providers such as SearXNG explicitly opt into private-network access.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted private-network opt-in, additive SDK exports, and tests covering public-provider private redirects plus SearXNG private-host access.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always attaches a private-network-allowing policy, and the shared web-search wrappers used by bundled providers call that endpoint.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. A strict default plus explicit self-hosted opt-in is the narrowest maintainable approach because it removes broad private-network access while preserving SearXNG's intended private-host behavior.
