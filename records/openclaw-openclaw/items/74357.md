## Summary

Keep open. Current main still routes shared trusted web-tools and web-search calls through a private-network-allowing SSRF policy, and the provided GitHub context shows the closing PR is still open rather than merged. Because this is SSRF-sensitive and touches plugin-facing SDK helpers, it should stay open for maintainer/security review of the paired fix path.

## What This Changes

The issue asks to make web-tools SSRF policy strict by default and allow private-network access only through explicit self-hosted provider opt-in, such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: withTrustedWebToolsEndpoint always attaches a private-network-allowing policy, and the shared web-search wrappers used by bundled providers call that endpoint.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it removes broad private-network access while preserving SearXNG's intended private-host behavior.
