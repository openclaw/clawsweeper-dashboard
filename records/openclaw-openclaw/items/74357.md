## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools endpoint used by bundled web-search helpers and plugin-facing SDK exports, and the paired closing PR #74360 remains open rather than merged.

## What This Changes

Issue requests splitting the shared trusted web-tools SSRF policy so private-network access is only available to explicit self-hosted web search providers such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always applies the private-network-allowing policy, shared web-search helpers call it, and redirect handling reuses the same policy for redirected targets.

## Solution Assessment

Yes for the requested direction, with implementation review still required. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
