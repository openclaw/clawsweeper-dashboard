## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools endpoint used by bundled web-search providers and plugin SDK exports, and the paired closing PR #74360 remains open rather than merged. Because the report is SSRF-sensitive and touches public plugin-facing helpers, it should stay with maintainer/security review.

## What This Changes

The issue asks to make web-tools SSRF handling strict by default and reserve private-network access for explicitly self-hosted providers such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main routes bundled provider calls through `withTrustedWebSearchEndpoint` or `postTrustedWebToolsJson`, which load `withTrustedWebToolsEndpoint` with `dangerouslyAllowPrivateNetwork: true`.

## Solution Assessment

Yes for the requested direction, with implementation review still required. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
