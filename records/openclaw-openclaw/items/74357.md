## Summary

Keep open. Current main still routes shared web-tools and web-search calls through a private-network-allowing SSRF policy, the behavior is covered by an existing test expectation, and the paired closing PR #74360 remains open rather than merged. Because this is SSRF-sensitive and touches plugin-facing SDK helpers, it needs maintainer/security review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make web-tools SSRF policy strict by default and reserve private-network access for self-hosted providers such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, explicit self-hosted private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers reject private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always passes a private-network-allowing policy, shared web-search helpers call it, and current tests assert that broad policy.

## Solution Assessment

Yes for the requested direction, with implementation review still required. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
