## Summary

Keep open. Current main still routes bundled web-search and plugin-facing helpers through a shared trusted web-tools SSRF policy that permits private-network targets, and the paired closing PR #74360 is still open rather than merged.

## What This Changes

The issue requests splitting the shared web-tools SSRF policy so public web-search providers stay strict while SearXNG-style self-hosted providers explicitly opt into private-network access.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search path, an explicit self-hosted private-network opt-in, additive SDK exports, docs/changelog updates, and regression tests for public-provider private redirects plus SearXNG private-host access.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: withTrustedWebToolsEndpoint always attaches a private-network-allowing policy, and bundled web-search helpers route public provider requests through it.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. A strict default plus explicit self-hosted opt-in is the narrowest maintainable approach because it removes broad private-network access while preserving SearXNG's intended private-host behavior.
