## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools endpoint used by bundled web-search providers and SDK exports, and the linked closing PR #74360 is still the active implementation path rather than a merged fix.

## What This Changes

The issue asks to split trusted web-tools SSRF policy so public web-search providers stay strict and only self-hosted providers such as SearXNG can explicitly opt into private-network access.

## Best Possible Solution

Ship a strict-by-default trusted web-tools/web-search path, add an explicit self-hosted private-network opt-in for SearXNG-style providers, and lock both behaviors with provider and SDK tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: shared web-search helpers route through withTrustedWebToolsEndpoint, which always attaches a policy with dangerouslyAllowPrivateNetwork enabled on current main.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. A strict default plus explicit self-hosted opt-in is the narrow maintainable solution because it removes broad private-network access while preserving SearXNG's intended private-host behavior.
