## Summary

Keep open. Current main still applies the private-network-allowing SSRF policy through the shared trusted web-tools/web-search helper used by bundled public providers and plugin-facing SDK exports, and the supplied GitHub context shows the closing PR is still open rather than merged.

## What This Changes

The issue requests splitting the shared web-tools SSRF policy so only SearXNG/self-hosted web-search providers can opt into private-network access while public API providers and SDK helpers use strict defaults.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports, and tests for public-provider private redirects plus SearXNG private-host behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always attaches a private-network-allowing policy, and shared web-search wrappers used by bundled providers call that endpoint.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
