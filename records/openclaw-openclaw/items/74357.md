## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools endpoint used by bundled web-search providers and plugin-facing helpers, and the provided context shows PR #74360 is still open rather than merged. Because this is SSRF-sensitive and touches public SDK helpers, it needs maintainer/security review of that paired fix path before closure.

## What This Changes

Request to split the shared web-tools SSRF policy so only self-hosted providers such as SearXNG can opt into private-network access while public web-search providers use strict defaults.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always attaches a policy with `dangerouslyAllowPrivateNetwork`, and the shared web-search wrappers used by bundled providers call that endpoint.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. A strict default plus explicit self-hosted opt-in is the narrowest maintainable solution because it removes broad private-network access while preserving SearXNG's intended private-host behavior.
