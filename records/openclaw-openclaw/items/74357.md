## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools/web-search helper used by bundled providers and plugin-facing SDK exports, and the paired closing PR #74360 remains open rather than merged.

## What This Changes

The issue asks to split web-tools SSRF policies so private-network access is an explicit self-hosted provider opt-in instead of applying to all bundled web-search providers and SDK helpers.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always passes a private-network-allowing policy, and the shared web-search helpers used by bundled providers call that endpoint.

## Solution Assessment

Yes for the requested direction, with implementation review still required. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
