## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools endpoint used by bundled web-search helpers and SDK exports, and the linked implementation path, PR #74360, remains open rather than merged. Because the remaining work changes SSRF behavior and plugin-facing helpers, it needs maintainer/security review before cleanup.

## What This Changes

The issue requests splitting the trusted web-tools SSRF policy so private-network access is explicit for self-hosted providers instead of inherited by every bundled web-search provider.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always passes a private-network-allowing policy, the shared web-search helpers call it, and the bundled provider search shows public providers using those helpers.

## Solution Assessment

Yes for the requested direction, but the exact patch still needs maintainer/security review. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it removes broad private-network access while preserving SearXNG’s documented private-network use case.
