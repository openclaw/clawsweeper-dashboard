## Summary

Keep open. Current main still applies a private-network-allowing SSRF policy through the shared trusted web-tools/web-search helper used by bundled public providers and plugin-facing SDK exports, and PR #74360 remains the open paired implementation path rather than a merged fix.

## What This Changes

The issue asks OpenClaw to make web-tools SSRF policy strict by default and reserve private-network access for explicit self-hosted providers such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always attaches a private-network-allowing policy, shared web-search wrappers call it, and bundled providers route through those wrappers.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
