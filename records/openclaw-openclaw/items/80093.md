## Summary

Kept open: this PR is not mergeable as-is. It reintroduces Telegram `sendMessageDraft` despite a merged removal and a stronger closed related PR, has blocking integration issues in config and lane finalization, carries unrelated Fal provider changes, and lacks real behavior proof.

## What This Changes

The PR adds a Telegram `sendMessageDraft` preview transport and tests while also carrying unrelated Fal image-provider changes.

## Best Possible Solution

Keep the linked feature request as the product discussion, and only reintroduce native drafts through an owner-approved `nativeTransport` design with strict schema/docs, DM-only fallback, finalization coverage, and live Telegram proof.

## Reproduction Assessment

Yes for the blocking review findings via source inspection: the advertised config key is absent from the strict schema, and draft previews do not satisfy the current lane-delivery message-id contract. The intended Telegram UX itself still lacks live proof.

## Solution Assessment

No. This patch bypasses the established streaming config seam, does not adapt the no-message-id draft lifecycle to current finalization logic, lacks DM/group gating, and conflicts with the prior removal decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep draft previews out of normal send fallback:** `extensions/telegram/src/draft-stream.ts:154-171`
  - body: `sendMessageDraft` returns `true`, not a message id, and this branch never gives the stream a finalizable id or draft-specific delivery result. Current lane delivery treats `messageId() === undefined` as "not streamed" and then falls through to normal `sendPayload`, so non-final preview updates can become visible Telegram messages instead of only native drafts.
  - confidence: 0.93
- **[P2] Expose the transport through the validated config schema:** `extensions/telegram/src/bot-message-dispatch.ts:431-432`
  - body: This reads `channels.telegram.streaming.transport`, but the current streaming schema is strict and does not allow `transport`, so the advertised opt-in config is rejected by the canonical config path. Use the existing `nativeTransport` seam or update the schema, types, docs, UI hints, generated config baseline, and tests together.
  - confidence: 0.91
- **[P2] Gate draft transport before group and non-partial lanes:** `extensions/telegram/src/bot-message-dispatch.ts:431-442`
  - body: The selected transport is passed to every Telegram draft lane once configured, but the feature request and prior implementation both require falling back outside private partial-mode chats. Without dispatcher gating or draft-error fallback, an opted-in group/topic route can hit Telegram's DM-only API limitation instead of using the existing edit-message transport.
  - confidence: 0.86
- **[P2] Split the Fal provider changes out of this PR:** `extensions/fal/image-generation-provider.ts:99-107`
  - body: This Telegram PR also changes Fal model routing, image edit limits, aspect-ratio behavior, and payload shape for a separate issue. Those provider changes need their own review and proof, otherwise merging this PR would silently ship unrelated Fal behavior with the Telegram feature.
  - confidence: 0.88
