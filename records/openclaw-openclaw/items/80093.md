## Summary

Keep open: this PR is active and focused, but the current patch does not persist final partial-mode draft replies even though Telegram documents `sendMessageDraft` previews as ephemeral and requiring a final `sendMessage`. The supplied proof shows direct Telegram API calls, not the OpenClaw final-delivery path.

## What This Changes

The PR adds an opt-in Telegram `sendMessageDraft` preview transport via `channels.telegram.streaming.nativeTransport` plus schema/type support and unit coverage.

## Best Possible Solution

Keep native drafts as a preview-only transport, then finalize through the existing durable Telegram send path so private-chat draft animation ends with one persisted message and normal fallback behavior still works.

## Reproduction Assessment

Yes for the blocking review findings via source and dependency inspection: final partial-mode draft delivery can be marked retained without a durable Telegram message. I did not run a live OpenClaw Telegram gateway repro.

## Solution Assessment

No. The narrow maintainable fix is to make draft transport participate in final delivery explicitly, likely by sending the complete message through the existing `sendPayload`/finalization seam instead of treating an ephemeral draft as the final message.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Persist draft previews on final delivery:** `extensions/telegram/src/draft-stream.ts:203-210`
  - body: `sendMessageDraft` only creates an ephemeral preview; Telegram's Bot API says the bot must call `sendMessage` with the complete output when generation is finalized. This path returns success without creating a message id, and current lane delivery then treats `sendMayHaveLanded()` as `preview-retained`, so partial-mode native drafts can finish without a permanent Telegram reply.
  - confidence: 0.94
- **[P2] Clear landed state when draft sends fail:** `extensions/telegram/src/draft-stream.ts:202-203`
  - body: This sets `messageSendAttempted` before awaiting `sendMessageDraft`, but the draft path has no catch that clears it on Telegram rejections or safe retry errors. On a final update, a failed draft call can still leave `sendMayHaveLanded()` true and make lane delivery mark the reply delivered instead of falling back to the normal send path.
  - confidence: 0.88
