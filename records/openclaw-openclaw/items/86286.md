## Label Justifications

- `P1`: The PR attempts to fix a user-visible reasoning leak affecting active Ollama Kimi Cloud chat flows, but the current patch leaves the live streaming path exposed.
- `merge-risk: 🚨 message-delivery`: The new delimiter handling can silently strip legitimate assistant text when U+FE0F appears as part of an emoji in Kimi Cloud output.
- `merge-risk: 🚨 security-boundary`: The change is meant to prevent internal reasoning from crossing into user-visible chat, but streamed deltas can still publish that reasoning before the final sanitizer runs.

## Summary

Keep this PR open: the fix targets a real Ollama Kimi Cloud reasoning leak, but the patch only sanitizes the final assistant message and leaves live streamed text deltas able to expose the same prefix; it also uses a bare variation selector delimiter that can match normal emoji in Kimi output.

## What This Changes

The PR adds Kimi Cloud-specific inline reasoning stripping to the Ollama assistant-message builder and adds regression tests for Kimi and non-Kimi final-message behavior.

## Best Possible Solution

Sanitize at the streaming boundary as well as final message construction, use a delimiter matcher that cannot catch normal emoji variation selectors, and require after-fix Kimi Cloud proof or a clearly documented proof gap before merge.

## Reproduction Assessment

Yes from source inspection for the patch defect: `createOllamaStreamFn` publishes raw `text_delta` events before the new final-message sanitizer runs. Live Kimi Cloud proof is still missing because the contributor's attempts returned subscription 403s.

## Solution Assessment

No: sanitizing only `buildAssistantMessage(...)` is not the best fix for a streaming chat leak. The safer solution is a stream-aware sanitizer plus final-message normalization and delimiter tests that cover emoji false positives.
