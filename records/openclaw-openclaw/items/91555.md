## Label Justifications

- `P1`: A documented recovery command can become unusable from a live Feishu chat when long model work occupies the same queue, leaving real users unable to recover the session from their primary channel.
- `impact:session-state`: The affected command is session compaction/recovery, and starvation leaves the large Feishu session in the unhealthy state it was meant to repair.
- `impact:message-loss`: The queued `/compact` message can be evicted by the per-chat cap before command handling, effectively dropping the recovery request.

## Summary

Keep open: current main and v2026.6.1 still route Feishu `/compact` through the per-chat sequential queue before core command handling, so the reported starvation and 300s eviction path remains real and distinct from the related reply-drop and text-command reports.

## What This Changes

The issue reports that Feishu `/compact` can queue behind long same-chat model work and expire under the 300s per-chat cap instead of acting as a recovery command.

## Best Possible Solution

Choose and implement a command-aware recovery path that lets authorized Feishu maintenance commands such as `/compact` reach core command handling without waiting behind normal same-chat model work, with regression coverage for queue ordering and authorization.

## Reproduction Assessment

Yes from source and logs, not from a live run: Feishu text control commands skip debounce but still enter `dispatchFeishuMessage`, await the same per-chat queue, and only then reach core `/compact` handling.

## Solution Assessment

No current implementation is not the desired end state; the 300s cap addresses indefinite normal-message starvation but is the wrong behavior for a command intended to unblock the active session.
