## Summary

Keep open: current main still builds and delivers the subagent announce without a runtime-side parent-assistant transcript fingerprint check. The related open PRs only cover adjacent SILENT-token prompt parity and same-idempotency-key dispatch dedupe, not whether the parent already delivered the child result.

## What This Changes

The issue requests runtime-side subagent announce duplicate suppression by comparing child findings with the requester parent session's latest assistant message before sending the completion announce.

## Best Possible Solution

Add a conservative, fail-open parent latest-assistant transcript check before announce delivery, with regression coverage for duplicate, unrelated, stale, and transcript-read-failure cases.

## Reproduction Assessment

Yes, source-level. Configure runSubagentAnnounceFlow with child result X and a requester session whose latest assistant reply after the child ended already contains X; current main still builds and delivers the announce without checking parent history.

## Solution Assessment

Yes. A pre-delivery, fail-open parent transcript fingerprint check is the narrow structural fix; prompt guards and same-key dispatch dedupe reduce adjacent failure modes but do not observe whether the parent already delivered the result.

Read the requester parent session's latest assistant message from chat.history or an equivalent helper, compare only when the assistant message timestamp is at or after the child endedAt when timestamp data is available, and fail open if history/timing cannot be read safely.
