## Summary

Keep open: current main and the latest shipped release still build and deliver subagent completion announces without checking the requester/parent transcript, and the related open PRs cover adjacent prompt-level or same-idempotency delivery dedupe rather than this parent-assistant fingerprint short-circuit.

## What This Changes

Request a runtime-side duplicate short-circuit that matches subagent findings against the parent/requester's latest assistant message and skips already-delivered announces.

## Best Possible Solution

Add a conservative, fail-open latest parent-assistant transcript check before announce delivery, with regression coverage for duplicate, unrelated, stale, and transcript-read-failure cases.

## Reproduction Assessment

Yes, source-level: runSubagentAnnounceFlow with child result X and a requester session whose latest assistant reply after endedAt contains X still proceeds to build and deliver the announce because no parent history check exists. I did not run a live channel scenario.

## Solution Assessment

Yes. A pre-delivery, fail-open parent transcript fingerprint check is the narrow structural fix; prompt guards and same-key delivery dedupe reduce adjacent failure modes but do not observe whether the parent already delivered the result.
