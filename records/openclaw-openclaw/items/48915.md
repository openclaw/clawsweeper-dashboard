## Summary

Keep open. Current main and the latest release still recognize only camelCase tool-call block types in the session-history repair path, and an open PR already references this issue as its fix candidate, so this should stay open until that PR or a replacement lands.

## What This Changes

The issue requests that session-history repair recognize snake_case tool-call block types such as `tool_use`, `tool_call`, and `function_call` so malformed Anthropic history blocks are sanitized before replay.

## Best Possible Solution

Land a focused fix that makes the shared tool-call recognition paths treat `tool_use`, `tool_call`, and `function_call` as tool-call blocks, with regression coverage for malformed snake_case blocks missing `input` or `arguments`; then close this issue with the fixing SHA and release provenance.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an assistant message containing `{ type: "tool_use", id: "call_1", name: "read" }` without `input` or `arguments`; current main's repair predicate does not recognize it, so the malformed block survives `sanitizeToolCallInputs`.

## Solution Assessment

Yes. Recognizing the snake_case variants in the shared tool-call recognition path is the narrowest maintainable fix; the safer version updates the parallel replay/tool-call recognizers together so repair and extraction do not drift.
