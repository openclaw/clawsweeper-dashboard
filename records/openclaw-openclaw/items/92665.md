## Summary

Keep the PR open: it is the right candidate for the open LiteLLM cache-retention bug, but the current patch still has a blocking compat-merging defect, stale provider docs, and no real LiteLLM-to-Anthropic proof.

## What This Changes

The PR adds LiteLLM Claude detection for Anthropic-family cache-retention resolution, enables Anthropic `cache_control` payload markers in OpenAI-compatible completions when `cacheRetention` is explicit, and adds source/payload regression tests.

## Best Possible Solution

Carry the explicit-cache flag through `getCompat`, add the compat-override regression, update the LiteLLM/proxy docs, and require redacted live LiteLLM-to-Anthropic proof before merge.

Yes at source level: current main has no LiteLLM OpenAI-completions cache-control route, and the issue path is visible through the cache-retention resolver and transport wrapper, though no live proxy reproduction was run in this review.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve the explicit cache gate through compat merging:** `src/llm/providers/openai-completions.ts:1351`
  - body: `detectCompat()` marks LiteLLM cache-control support as requiring explicit cache config, but `getCompat()` drops `requiresExplicitCacheConfig` when a model has any `compat` override. That means a LiteLLM Claude model with an unrelated compat tweak and no `cacheRetention` falls back to the default `short` retention path and emits `cache_control` anyway, which violates the explicit-only safety gate. Carry this flag through the merge and add a regression for the override case.
  - confidence: 0.9
- **[P3] Update the proxy cache documentation:** `src/llm/providers/openai-completions.ts:1320`
  - body: This change makes explicit LiteLLM Claude `cacheRetention` send Anthropic prompt-cache hints, but the public LiteLLM and OpenAI-compatible proxy docs still say those routes get no prompt-cache hints. Update `docs/providers/litellm.md` and `docs/gateway/local-models.md` so users do not treat the new config as unsupported.
  - confidence: 0.82
