## Summary

Keep this PR open, but do not merge it yet: it targets the right LiteLLM cache-retention bug, but the new OpenAI-completions compatibility gate would inject Anthropic cache_control for LiteLLM Claude aliases even when no explicit cacheRetention was configured, and the supplied proof is still source-level terminal output rather than a live LiteLLM proxy request.

## What This Changes

Adds LiteLLM Claude model detection to Anthropic cache-retention family resolution, enables Anthropic cache_control compatibility in openai-completions for LiteLLM models, and adds unit coverage for the new helper.

## Best Possible Solution

Make the LiteLLM cache_control path explicit-cacheRetention-only, add serialized payload coverage for explicit/no-config/non-Claude cases, update LiteLLM docs, and require redacted live LiteLLM-to-Anthropic proof before merge.

## Reproduction Assessment

Yes at source level: current main keeps LiteLLM on openai-completions while both retention-family detection and cache_control payload compatibility omit the default route. I did not run a live LiteLLM proxy reproduction in this read-only review.

## Solution Assessment

No: the two-gate fix shape is right, but the current detectCompat change is too broad because absent cacheRetention defaults to short. The safer solution is explicit-only LiteLLM payload behavior with docs and live proxy proof.
