## What This Changes

The PR removes the Codex bridge-tool surface and adds native Codex app-server plugin migration, mention-based docs, config updates, protocol normalization, and generic wildcard tool-contract coverage.

## Best Possible Solution

Land the native Codex path after fixing migration activation to target the same resolved Codex home future sessions will use, supporting pathless remote curated marketplaces, and getting maintainer signoff on bridge removal.

I did not execute the PR branch in this read-only review.

## Solution Assessment

No: the native app-server direction is plausible, but this PR should first keep apply-time activation aligned with the future runtime home and handle remote-only curated marketplaces. Maintainers also need to approve the intentional bridge-removal compatibility break.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use one resolved Codex home for activation and runtime:** `extensions/codex/src/migration/apply.ts:108`
  - body: `--from` is a migration source override, but this exports it as `CODEX_HOME` for apply-time `plugin/install`. Normal Codex app-server launches later use the agent-owned home unless an explicit override is present, so migrating from a non-default or `~` source can report success while installing the plugin in a home the next Gateway session will not use. Resolve the source if it is used, and activate against or persist the same home future sessions will run with.
  - confidence: 0.9
- **[P2] Pass remote marketplace names to plugin/install:** `extensions/codex/src/migration/apply.ts:215`
  - body: For an installed-but-disabled `openai-curated` plugin from a remote-only marketplace, `marketplace.path` is null by protocol, so this branch errors before trying to enable it. `PluginInstallParams` accepts `remoteMarketplaceName`, so pathless curated marketplaces should use that instead of failing migration.
  - confidence: 0.84
