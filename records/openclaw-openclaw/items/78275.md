## Summary

Keep open: this feature PR has sufficient real behavior proof, but latest head still has source-level blockers in migration activation, optional tool discovery, and Codex app-server protocol handling; the protected `maintainer` label also means final handling needs explicit maintainer approval.

## What This Changes

The PR adds an opt-in Codex plugin bridge that migrates selected source-installed `openai-curated` Codex plugins into `codexPlugins` config and exposes generated `codex_plugin_*` OpenClaw tools through Codex app-server activation and invocation.

## Best Possible Solution

Repair the current-head blockers, keep the bridge opt-in and fail-closed, then land only after maintainer approval of the new app-backed Codex tool surface.

## Reproduction Assessment

Yes for the PR blockers: source inspection of latest head `c15e3f6a913087ecdb4029a7f4221f2d188e4500`, current main optional-tool behavior, and generated app-server protocol types gives high-confidence reproduction paths. I did not run the branch because this review is read-only.

## Solution Assessment

No: bridging through Codex app-server is plausible, but this implementation is not yet the best mergeable form because migration does not reliably activate/discover the generated tools and the runtime bridge still diverges from app-server pagination, model, elicitation, and long-turn contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Enable the Codex plugin during plugin migration:** `extensions/codex/src/migration/plan.ts:201-205`
  - body: `openclaw migrate apply codex --plugin ...` writes only the nested `codexPlugins` config. The bundled `codex` plugin is disabled unless explicitly enabled or activated by a runtime hint, so the documented migrate-and-restart path still will not load the plugin or register these tools for users who had not already enabled Codex.
  - confidence: 0.9
- **[P1] Make migrated Codex plugin tools discoverable:** `extensions/codex/index.ts:41-44`
  - body: The generated plugin tools are registered as `optional`, but migration does not add the generated tool names, `codex`, or a wildcard to the tool allowlist. Current resolver behavior skips optional tools without an explicit allowlist, so the migrated `codex_plugin_*` tools can remain absent from the catalog after restart.
  - confidence: 0.9
- **[P2] Handle auth-required plugin installs during migration:** `extensions/codex/src/migration/apply.ts:151-156`
  - body: `plugin/install` can return `appsNeedingAuth`, but this path discards the response and marks the item migrated. OAuth-required plugins then get enabled in OpenClaw config even though the resulting tool immediately fails until the user fixes auth out of band.
  - confidence: 0.9
- **[P2] Page through Codex app inventory before enabling apps:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:120-122`
  - body: This reads only the first `app/list` page before deciding which Codex apps to enable. If the selected plugin's app is behind `nextCursor`, `appIds` stays empty and the bridge starts the plugin turn without enabling the required app.
  - confidence: 0.9
- **[P2] Accept nullable plugin elicitation turn IDs:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:367`
  - body: Codex app-server allows `mcpServer/elicitation/request` to carry `turnId: null`, and the existing elicitation bridge accepts that shape when the thread matches. This matcher rejects nullable turn IDs, so valid plugin approval prompts can be ignored and the bridged tool can hang or fail.
  - confidence: 0.92
- **[P2] Refresh the plugin idle timer on progress:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:221-225`
  - body: The plugin turn starts a fixed 30s idle timer after `turn/start` and never refreshes it when deltas, completed items, elicitations, or other progress arrives. Active plugin turns that take longer than 30 seconds before `turn/completed` can reject despite the surrounding 5-minute timeout.
  - confidence: 0.86
- **[P2] Use a valid Codex model provider for plugin threads:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:76-78`
  - body: This sends only the stripped model id from the OpenClaw default model. In mixed-provider configs such as `anthropic/claude-sonnet-4.6`, app-server receives `claude-sonnet-4.6` without a `modelProvider`, so `thread/start` can fail before the plugin runs.
  - confidence: 0.88
- **[P2] Preserve cached plugin review items after discovery:** `extensions/codex/src/migration/source.ts:310`
  - body: When app-server plugin discovery succeeds, this sets cached plugin bundles to `[]`, so non-curated or local cached bundles disappear from the migration plan instead of being reported as manual-review items as the docs promise.
  - confidence: 0.87
- **[P3] Credit the external contributor in the changelog:** `CHANGELOG.md:12`
  - body: This user-facing feature entry is for an external human contributor, but the changelog line does not include the required `Thanks @kevinslin` attribution.
  - confidence: 0.82
