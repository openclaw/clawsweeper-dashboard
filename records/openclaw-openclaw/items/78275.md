## Summary

Keep open: latest head still has concrete migration/runtime blockers, and the protected `maintainer` label means the new app-backed Codex tool surface needs explicit maintainer handling.

## What This Changes

The PR adds an opt-in `codexPlugins` bridge that migrates selected `openai-curated` Codex plugins, registers generated `codex_plugin_*` tools, updates docs/config metadata, and adds targeted Codex/plugin-tool tests.

## Best Possible Solution

Fix the migration/app-inventory blockers, keep the bridge opt-in and fail-closed, add regression coverage for `--from` and paginated/auth-required app lists, then land only after maintainer approval of the new Codex plugin tool surface.

## Reproduction Assessment

Yes for the PR blockers: source inspection of latest head shows apply uses app-server start options without the `ctx.source` `CODEX_HOME` override, and inventory reads only the first app-list page despite a generated `nextCursor` contract. I did not execute the branch because this was a read-only review.

## Solution Assessment

No: bridging through Codex app-server is plausible, but this implementation is not yet the best mergeable form because migration can target the wrong Codex home and still misses auth/pagination edge cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor --from when activating plugins:** `extensions/codex/src/migration/apply.ts:110`
  - body: `migrate apply codex --from <path> --plugin ...` plans against `ctx.source`, but activation starts app-server with `runtimeOptions.start` unchanged. That can run `plugin/list`/`plugin/install` against the default Codex home and then write OpenClaw config for the alternate source, so pass the source home through `CODEX_HOME` as discovery does.
  - confidence: 0.91
- **[P2] Verify auth for already-enabled plugins:** `extensions/codex/src/migration/apply.ts:186-191`
  - body: When app-server reports a plugin as installed and enabled, this marks the migration item `migrated` without checking the related app's auth/access state. A plugin with revoked OAuth can therefore enable the OpenClaw bridge config and fail immediately at runtime; check app accessibility or run the install/auth path before declaring success.
  - confidence: 0.86
- **[P2] Page app inventory before auth checks:** `extensions/codex/src/app-server/plugin-inventory.ts:233-234`
  - body: `readCodexPluginInventory` derives `authRequired` from `app/list`, but `readAppsBestEffort` consumes only the first page and ignores `nextCursor`. If the selected plugin's inaccessible app is on a later page, the bridge can treat it as ready and start the plugin turn without required auth.
  - confidence: 0.88
