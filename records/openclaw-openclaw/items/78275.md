## Summary

Keep open: this is a useful Codex plugin-bridge feature, but current-head source inspection still finds migration/discovery blockers and Codex app-server contract mismatches that should block merge; the protected `maintainer` label also means final approval needs maintainer handling.

## What This Changes

The PR adds opt-in Codex plugin bridge config/tool registrations, migration CLI support for `--plugin`, Codex app-server inventory/activation/invocation helpers, docs, generated config baselines, targeted tests, and a Codex plugin `typebox` dependency.

## Best Possible Solution

Repair the current-head blockers, keep the bridge opt-in and fail-closed, then land only after maintainer approval of the new app-backed Codex tool surface.

## Reproduction Assessment

Yes for the PR blockers: source inspection against the latest head, current main plugin resolver behavior, and generated Codex protocol types gives a high-confidence reproduction path. I did not run the PR branch locally because this review is read-only.

## Solution Assessment

No: bridging through Codex app-server is a plausible direction, but this implementation is not yet the best mergeable form because the documented migration path can leave tools undiscoverable and several invoker paths diverge from existing app-server contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Make migrated Codex plugin tools discoverable:** `extensions/codex/index.ts:43`
  - body: Registering the generated `codex_plugin_*` tools as optional means the normal migrated-and-restart path still drops them unless the user has also allowlisted each generated tool or plugin group. Either have migration write the required allowlist entries or make these explicit migration-created tools visible without the optional-tool gate.
  - confidence: 0.94
- **[P1] Enable the Codex plugin during migration:** `extensions/codex/src/migration/plan.ts:203-204`
  - body: The migration only writes `plugins.entries.codex.config.codexPlugins`, but the bundled Codex plugin is disabled by default. After `openclaw migrate apply codex --plugin ...`, the gateway will not load the Codex plugin or register these tools unless `plugins.entries.codex.enabled` is also written.
  - confidence: 0.96
- **[P2] Accept nullable plugin elicitation turn IDs:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:367`
  - body: Codex app-server allows `mcpServer/elicitation/request` to carry `turnId: null`, and the existing elicitation bridge accepts that shape when the thread matches. This matcher rejects nullable turn IDs, so valid plugin approval prompts can be ignored and the bridged tool can hang or fail.
  - confidence: 0.92
- **[P2] Refresh the plugin idle timer on progress:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:221-225`
  - body: The plugin turn starts a fixed 30s idle timer after `turn/start` and never refreshes it when deltas, completed items, elicitations, or other progress arrives. Active plugin turns that take longer than 30 seconds before `turn/completed` can reject despite the surrounding 5-minute tool timeout.
  - confidence: 0.88
- **[P2] Page through app inventory before enabling apps:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:120-122`
  - body: This reads only the first `app/list` page before deciding which Codex apps to enable. If the selected plugin's app is behind `nextCursor`, `appIds` stays empty and the bridge starts the plugin turn without enabling the required app.
  - confidence: 0.9
- **[P2] Surface plugin auth failures during migration:** `extensions/codex/src/migration/apply.ts:151-156`
  - body: `plugin/install` can return `appsNeedingAuth`, but this path marks the item migrated and then enables the OpenClaw tool in config. For plugins needing OAuth, migration reports success while the resulting tool immediately fails until the user fixes auth out of band.
  - confidence: 0.87
- **[P2] Pass a valid Codex model/provider pair:** `extensions/codex/src/app-server/plugin-tool-invoker.ts:370-374`
  - body: This strips any configured provider from the OpenClaw default model and sends only the model id to app-server. In common multi-provider configs such as `anthropic/claude-sonnet-4.6`, the plugin thread starts with a model Codex cannot resolve as a native Codex model; use a Codex fallback or pass `modelProvider` like the existing Codex thread lifecycle does.
  - confidence: 0.89
- **[P3] Credit the external contributor in the changelog:** `CHANGELOG.md:11`
  - body: This user-facing feature entry is for an external human contributor, but the changelog line does not include the required `Thanks @kevinslin` attribution.
  - confidence: 0.82
