## Summary

Keep open. The patch is a focused, likely-correct fix for the cron announce control-token leak, but it is still waiting on real cron-to-channel behavior proof before merge.

## What This Changes

The PR replaces cron delivery's `NO_REPLY`-only normalization with the shared suppressed-control-reply predicate and expands cron delivery regression coverage for token-only `NO_REPLY`, `ANNOUNCE_SKIP`, and `REPLY_SKIP`.

## Best Possible Solution

Land the narrow exact-token suppression fix after real cron-to-channel proof shows control-token replies stay silent and a normal cron reply still posts; track markdown-wrapped variants separately if desired.

## Reproduction Assessment

Yes. Source inspection on current main shows cron delivery only normalizes `NO_REPLY`, while the shared predicate already suppresses `ANNOUNCE_SKIP` and `REPLY_SKIP`; the linked issue also gives repeated channel observations.

## Solution Assessment

Yes for the exact-token leak. Reusing the shared suppression predicate in cron delivery is the narrow maintainable fix; broader markdown-wrapped suppression should only be added with explicit tests and maintainer agreement.
