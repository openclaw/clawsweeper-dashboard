## Summary

This PR is still useful and should stay open, but latest head still has a blocking standalone-signin example bug: it queues verification commands immediately after an interactive `op signin` in tmux.

## What This Changes

The PR changes the bundled 1Password skill, its get-started reference, and the changelog so agents choose direct `op` execution or tmux based on 1Password authentication mode.

## Best Possible Solution

Land the auth-mode-aware skill update after rewriting the standalone tmux signin example into an explicit attach-and-wait/manual verification flow.

## Reproduction Assessment

Yes. The underlying issue is reproducible from current main's tmux-only skill text plus the linked direct-vs-tmux report, and the remaining PR defect is reproducible by inspecting latest head at `skills/1password/SKILL.md:82-85`.

## Solution Assessment

No, not as written. The PR direction is the narrow maintainable fix, but the standalone signin example must not queue verification commands before the interactive signin has completed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for signin before queuing verification:** `skills/1password/SKILL.md:82`
  - body: The standalone flow sends `eval "$(op signin ...)"` to a detached tmux pane and immediately queues `op whoami`, `op vault list`, and `capture-pane`. Since manual signin blocks for user input before it emits the `OP_SESSION_*` export, those follow-up commands can be consumed by the prompt or run before the session exists; make this an attach/manual-wait flow before verification.
  - confidence: 0.9
