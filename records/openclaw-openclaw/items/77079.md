## Summary

Keep open: the report describes a current WebChat regression, and current main still has history-refresh paths that can replace previously visible live assistant text with the persisted `chat.history` snapshot while preserving tool cards.

## What This Changes

The issue requests that completed WebChat assistant replies remain visible after later user messages, especially for tool-heavy turns.

## Best Possible Solution

Add a focused regression test at the WebChat history-refresh boundary, then fix the narrow layer that drops delivered assistant text without duplicating normal Pi-persisted turns.

## Reproduction Assessment

Yes, at source level: a history refresh where `chat.history` includes tool results and a later user message but omits the previously displayed assistant answer will replace the UI state and drop that answer. I did not live-replay the reporter's exact macOS/provider session because the raw transcript was not included.

## Solution Assessment

Unclear until the failing transcript shape is pinned down; the best fix is a targeted regression test plus either projection/reconciliation repair or a SessionManager-backed persistence fallback, not raw JSONL appends.
