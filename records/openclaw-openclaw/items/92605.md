The PR changes `docs/platforms/windows.md` to pin Windows Hub companion downloads to v2026.6.5 and adds a note pointing users to GitHub releases if links fail.

## Best Possible Solution

Use this as a short-term docs mitigation after tightening the fallback wording, then land the release-process fix so Windows Hub assets are present on the current stable release and the docs can point at that exact release.

## Reproduction Assessment

Yes. Current main links to `releases/latest/download`, and live HEAD checks for x64, ARM64, and checksum downloads resolve to v2026.6.6 and return 404 while the pinned v2026.6.5 assets return 200.

## Solution Assessment

Partly. The docs-only pin is an acceptable immediate mitigation, but the durable fix is to promote or gate Windows Hub assets for the current stable release and avoid fallback wording that points users back to a release known to lack those assets.
