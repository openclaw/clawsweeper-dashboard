## Summary

Keep open. Current main still has the override-first resolver and raw configured/persisted override paths that can over-report a context window larger than a known selected-model limit, and the same-author fix PR #39859 remains open and unmerged.

## What This Changes

The issue asks for effective context-token resolution to cap configured or persisted overrides by the selected model's known smaller context window.

## Best Possible Solution

Keep this issue open until #39859 or an equivalent narrow fix lands. The desired end state is that positive configured or persisted contextTokens act as ceilings whenever the active model window is known, while preserving existing override behavior for unknown model windows and covering the resolver, reply setup, session persistence, and /status display paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure agents.defaults.contextTokens above a configured or discovered model window, then follow the resolver/session/status paths shown in current main; the code returns or stores the override before applying the smaller known model limit.

## Solution Assessment

Yes. The maintainable fix belongs in the shared model-aware context resolution and the raw persistence/reporting callers; a docs-only or status-only change would leave downstream stored metadata and reply setup behavior incorrect.
