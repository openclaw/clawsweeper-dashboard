## Summary

Keep open: current main and the latest v2026.6.6 release still have override-first context-token paths, and the issue is already paired with open closing-reference repair PRs. The immediate maintainer action is to review and land or reject those linked repairs, not close this issue before a merge.

## What This Changes

The issue asks OpenClaw to cap resolved session and status context windows to the selected model's known context limit when both an override and a model limit are known.

## Best Possible Solution

Land a focused shared-resolver/session-store repair that returns min(override, knownModelWindow) when both values are known, preserves unknown-window override behavior, and keeps the broader maxTokens/config-precedence question in its own canonical discussion.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the bug: pass a positive override with a provider/model whose known window is smaller, and current main returns or persists the override before applying the model limit.

## Solution Assessment

Yes. The narrow best fix is to cap in the shared resolver and route session persistence through that capped result; the linked repair PRs should be reviewed rather than opening another fix lane.
