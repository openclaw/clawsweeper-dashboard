- `impact:session-state`: The incorrect context limit can be persisted in session metadata and later consumed by status and follow-up paths.

## Summary

Current main and the latest release still allow a positive context-token override to bypass the selected model’s known window, while an open replacement PR already owns the focused repair; keep the issue open until that PR is refreshed, reviewed, and merged or replaced.

## What This Changes

Cap an explicit session context-token override by the selected model’s known context window before persisting or displaying the effective value.

## Best Possible Solution

Refresh the linked replacement PR against current main, keep unknown-model behavior unchanged, compute `min(override, knownModelWindow)` in the shared resolver, and make session persistence consume that canonical result before landing focused resolver, persistence, and status coverage.

## Reproduction Assessment

Yes from source inspection: a positive override returns before model-window lookup and can be persisted directly on current main. This read-only review did not execute a live agent turn, so the status is source-reproducible rather than reproduced.

## Solution Assessment

Yes in principle: applying the conservative minimum in the shared resolver is the narrowest maintainable fix, provided unknown-window override behavior remains unchanged and persistence stops bypassing that resolver.
