## Summary

Keep open: current main still has source-level override-first context-token paths that can over-report or persist a larger configured cap before checking the selected model's known window, and an open merge-conflicting PR already references this issue as the fix path.

## What This Changes

The issue asks OpenClaw to cap effective status/session `contextTokens` to `min(config override, known model window)` when both values are known.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/39859 with a current-main fix that caps overrides consistently in the shared resolver, session persistence, and status summary paths while preserving override behavior when the model window is unknown.

## Reproduction Assessment

Yes, source-level. Configure `agents.defaults.contextTokens` above a selected model's known smaller window; current main still has shared resolver, session-store, and status-summary paths that choose the override before the smaller model window.

## Solution Assessment

Yes. The requested `min(override, knownWindow)` behavior is the narrow maintainable fix, but it needs to cover all resolver, persistence, and status-summary bypasses rather than only display wording.
