- `impact:session-state`: The reported behavior affects persisted and displayed effective session context limits used by downstream session accounting.

## Summary

Keep open: current main and the latest release still have the override-first shared resolver/session-store path, and https://github.com/openclaw/openclaw/pull/90889 remains the open closing-reference repair for the central bug. The newer https://github.com/openclaw/openclaw/pull/92709 context adds a complementary slash-containing proxy-model lookup issue, so this issue should not be closed until maintainers land or deliberately split the linked work.

## What This Changes

The issue reports that an explicit `agents.defaults.contextTokens` override can be persisted or displayed as the effective session window even when the selected model has a smaller known context limit.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90889 or an equivalent shared resolver/session-store cap, then intentionally handle the complementary slash-containing model-ID lookup from https://github.com/openclaw/openclaw/pull/92709 if maintainers want both under this issue.

## Reproduction Assessment

Yes, at source level. On current main, the shared resolver returns a positive override before resolving the selected model window, and the session-store path can persist the same value before calling that resolver.

## Solution Assessment

Yes for the central problem: the linked repair in https://github.com/openclaw/openclaw/pull/90889 moves the override into shared resolver capping and removes the session-store bypass. The slash-ID lookup reported in https://github.com/openclaw/openclaw/pull/92709 is complementary and should be reviewed separately or folded deliberately.
