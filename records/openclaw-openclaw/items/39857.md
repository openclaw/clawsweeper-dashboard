## Summary

Keep open: current main still has source-level override-first context-token paths that can over-report a smaller known model window, and the same-author fixing PR with closing syntax is still open.

## What This Changes

The issue asks OpenClaw to cap effective status/session `contextTokens` overrides to the selected model's known context window before reporting or persisting them.

## Best Possible Solution

Rebase or replace https://github.com/openclaw/openclaw/pull/39859 with a current-main fix that applies `min(override, knownWindow)` consistently in the shared resolver, session persistence, and status reporting while preserving override behavior when the model window is unknown.

## Reproduction Assessment

Yes, source-level. Configure an agent/default `contextTokens` above a known selected-model window; current main still has resolver, session-store, and status-summary paths that choose the larger override before the known window.

## Solution Assessment

Yes, the requested cap semantics are the right narrow fix. The implementation should be current-main-aware and cover the shared resolver plus session/status bypasses, not only display wording.
