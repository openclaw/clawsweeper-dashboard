## Summary

Keep open: current main and the latest release still let an oversized positive context-token override win before the smaller known model window in the shared resolver and session-store persistence path. The immediate repair is already owned by the open closing-reference PR at https://github.com/openclaw/openclaw/pull/90889, so this issue should wait for that PR to land or be closed/replaced.

## What This Changes

The issue asks OpenClaw to cap effective session context-token overrides by the selected model's known context window before reporting or persisting session context metadata.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90889 or an equivalent shared resolver/session-store fix, then close this issue from merged current-main evidence and keep slash-model CLI/status follow-up work on its narrower PR or issue.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveContextTokensForModel` with a known provider/model window smaller than a positive override and current main returns the override first; the session-store path can then persist that value.

## Solution Assessment

Yes. The linked repair direction is the narrowest maintainable solution because it makes the shared resolver apply the cap and routes session persistence through that same capped result.
