## Summary

Keep open: current main and the latest release still have shared resolver/session-store paths that can prefer a positive context-token override over a known smaller model limit, and the open linked repair PR owns the focused fix before this issue can close.

## What This Changes

The issue asks OpenClaw to cap effective session context tokens at the selected model's known context window when an explicit override is larger.

## Best Possible Solution

Refresh and land the focused linked PR, or an equivalent replacement, that makes the shared resolver return `min(override, knownModelWindow)`, routes persistence through that result, and preserves unknown-model override behavior.

## Reproduction Assessment

Yes from source inspection: current main still returns a positive override before model-window lookup and can persist that value through the session-store helper. I did not run a live agent turn in this read-only review, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes in principle: applying the conservative minimum in the shared resolver is the narrowest maintainable fix, as long as unknown-window override behavior remains unchanged and persistence stops bypassing the resolver.
