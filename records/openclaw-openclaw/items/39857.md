## Summary

Keep open: current main still lets override-first values bypass known model windows in the shared resolver, session-store persistence, and CLI status summary; the linked closing PR is open and merge-conflicting, so this issue should remain paired with that fix path.

## What This Changes

The issue requests that effective session context-token reporting and persistence clamp a larger configured override to the selected model's known smaller context window.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/39859 with a current-main fix that clamps known-window overrides in the shared resolver, session persistence, and CLI status summary while leaving broader context/max-token precedence policy to https://github.com/openclaw/openclaw/issues/71136.

## Reproduction Assessment

Yes. Source inspection shows that a larger agents.defaults.contextTokens or persisted entry.contextTokens can still flow through override-first resolver, session-store, and CLI status-summary paths instead of being clamped to a known smaller model window.

## Solution Assessment

Yes for the narrow bug: when both an override and model window are known, resolving min(override, modelWindow) in the shared resolver and downstream persistence/status paths is the maintainable fix. The best path is the existing linked PR rebased or replaced, not a new parallel repair lane.
