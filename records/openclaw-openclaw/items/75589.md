## Summary

Keep open: current main still has the implicit noninteractive Gateway service install/rewrite paths, and this open maintainer-labeled automerge PR is the active fix. The code direction is narrow and security-positive, but the behavior change still needs a small doctor CLI docs update before automerge should proceed.

## What This Changes

The PR makes Gateway doctor service install/rewrite repairs require interactive confirmation outside update-mode repair, updates focused Gateway doctor tests, and adds a changelog entry.

## Best Possible Solution

Update the doctor CLI docs to state that Gateway service install/rewrite repairs now require an interactive confirmation or an explicit `openclaw gateway install` command, then let automerge rebase and gate the exact PR head.

## Reproduction Assessment

Yes. Current main can be reproduced statically: repair mode auto-approves runtime repairs unless `requiresInteractiveConfirmation` is set, both targeted Gateway service paths lack that guard, and the existing service-config test expects noninteractive install.

## Solution Assessment

Mostly yes. Reusing `requiresInteractiveConfirmation` is the narrowest maintainable fix for preventing noninteractive service mutation while preserving update-mode staging; the safer completion is a docs update for the changed operator contract.
