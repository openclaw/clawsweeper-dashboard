## Summary

Keep this draft, maintainer-labeled PR open for maintainer handling. The code change is narrow and uses the existing doctor prompter seam to keep service install/rewrite actions explicit, but the CLI docs should be checked because the PR changes what `doctor --repair` can do without prompting.

## What This Changes

The PR changes Gateway doctor service repair so noninteractive repair does not install or rewrite launchd/systemd service definitions, preserves update-mode staging, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land the service-repair guard after maintainer review and a small docs clarification, keeping service installs and service definition rewrites behind interactive confirmation or explicit `openclaw gateway install` commands while preserving update-mode staging.

## Reproduction Assessment

Yes. Current main can be traced statically: `--repair` sets `shouldRepair`, the current service install/config paths use auto-fix confirmations, and the PR's focused tests exercise the missing-service and entrypoint-drift paths.

## Solution Assessment

Mostly yes. Reusing `confirmRuntimeRepair` with `requiresInteractiveConfirmation` is the narrowest maintainable fix; the safer completion is to update the doctor CLI docs so operators understand the new service-repair exception.
