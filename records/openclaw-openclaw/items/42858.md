## Summary

Keep open. Current main still lacks a `bailian` provider/auth alias and `BAILIAN_API_KEY` candidate in the provider auth path, so the resolver can still fall through to unresolved auth for the reported provider id. The report is clear, has reporter verification of the mapping workaround, and has prior/open implementation PR context, so it should not be closed as stale or implemented.

## What This Changes

The issue requests Bailian provider auth resolution so `bailian` configs can use `BAILIAN_API_KEY` environment credentials instead of failing with HTTP 401.

## Best Possible Solution

If Bailian naming is accepted, add it through the Qwen/plugin-owned auth surface: include `bailian` in the Qwen provider/alias metadata as needed, add `BAILIAN_API_KEY` to the Qwen-compatible auth env candidates, and cover resolver/model-auth behavior with focused regression tests. If the alias is intentionally unsupported, update the Qwen docs to make the supported provider ids and env names explicit and close with that product decision.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `resolveEnvApiKey("bailian")` normalizes to `bailian`, finds no current candidate/evidence entry, and can return null; the reporter also confirmed that adding a Bailian mapping made their `bailian/qwen3.5-plus` path work.

## Solution Assessment

Unclear. The one-line core env-map addition fixes the immediate lookup, but the maintainable solution should live with the Qwen provider metadata/aliases so bundled provider ownership, docs, and tests stay aligned.
