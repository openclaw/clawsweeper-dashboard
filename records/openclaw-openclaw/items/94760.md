## Summary

Keep open: the direct-send target split addresses the linked Feishu p2p DM regression, but the PR still routes the streaming Card Kit start through a prefixed `user:ou_*` value and lacks real proof for that changed streaming path.

## What This Changes

The PR adds a Feishu reply-dispatcher `sendTarget`, passes `feishuTo` from both bot dispatch paths, and retargets visible Feishu replies for non-thread p2p DMs.

## Best Possible Solution

Keep the delivery-target split, normalize the streaming-card receive id before `FeishuStreamingSession.start`, add focused p2p direct and streaming coverage, then require redacted Feishu proof or maintainer override before merge.

## Reproduction Assessment

Yes. Current main computes `user:<open_id>` for Feishu DMs but still passes raw `ctx.chatId` to the reply dispatcher while reply metadata is suppressed, making the p2p `oc_*` delivery target source-reproducible.

## Solution Assessment

No, not yet. Splitting delivery target from inbound chat metadata is the right layer, but the streaming-card start path must receive a raw normalized id while preserving the receive-id type decision.
