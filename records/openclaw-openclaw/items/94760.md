## Summary

Keep open: the PR is the right general target-split shape for the Feishu p2p delivery bug, but the branch still routes the prefixed delivery target into the streaming-card create path, which can break streaming-card delivery.

## What This Changes

The branch adds a Feishu reply-dispatcher `sendTarget` parameter, passes `feishuTo` from both bot dispatch paths, and uses it for visible direct sends plus streaming-card start.

## Best Possible Solution

Keep raw `chatId` for lifecycle state, use a separate delivery target for visible sends, normalize the streaming-card receive id before Feishu API calls, and add focused p2p direct plus streaming coverage before merge.

## Reproduction Assessment

Yes. Current main computes `user:<open_id>` for Feishu DMs but passes raw p2p `ctx.chatId` to the reply dispatcher while reply metadata is suppressed; the linked issue and PR body also include Feishu p2p logs.

## Solution Assessment

No, not yet. Splitting delivery target from raw chat metadata is the right layer, but the streaming-card start path must normalize or strip prefixes before Feishu API create.
