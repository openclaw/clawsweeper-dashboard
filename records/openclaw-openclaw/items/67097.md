## Summary

Keep open. Current main still generates `models.json` through implicit provider discovery before applying `models.mode`, and the Codex and Arcee catalog paths named in the report are still present. Related #66957 covers the narrower `models.mode: "replace"` ordering/delay bug, but this issue also asks for a product/API decision about whether generated default/merge `models.json` should include bundled provider-owned catalogs without explicit provider configuration.

## What This Changes

The issue asks for generated `models.json` to reflect only providers explicitly introduced by onboarding or `openclaw.json`, instead of including bundled Codex and Arcee catalog rows from plugin-owned hooks.

## Best Possible Solution

Define the generated `models.json` contract explicitly. If merge/default generated manifests are meant to include bundled provider-owned catalogs, document that and keep only #66957 for replace-mode behavior; otherwise gate generated catalog rows on explicit activation/auth/config intent while preserving read-only discovery views such as `models list --all`.

## Reproduction Assessment

Yes. The source path is high-confidence: generated `models.json` planning calls implicit provider discovery before mode handling, and the named Codex and Arcee provider catalog hooks are still reachable. I did not run a live generation because this review is read-only.

## Solution Assessment

Unclear. A narrow `replace` short-circuit belongs with #66957, but this issue's broader requested behavior is not clearly the best fix until maintainers choose whether default/merge `models.json` should be a configured-only manifest or a generated provider catalog.
