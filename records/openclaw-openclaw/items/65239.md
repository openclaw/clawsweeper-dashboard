## Summary

Keep open. Current main has substantial SecretRef support plus one-way scrubbing, but it still lacks built-in encrypted-at-rest `.env` values, `ENC:` transparent decryption, master-key/keyring setup, and rotation; this remains security-sensitive product/API design rather than conservative cleanup.

## What This Changes

Add built-in encrypted-at-rest handling for sensitive `.env` configuration values, including master-key setup, `ENC:` storage/decryption, and key rotation.

## Best Possible Solution

Design a first-party encrypted/keyring-backed SecretRef provider or local vault that fits the existing activation snapshot, audit/configure/apply flow, onboarding UX, docs, fail-fast semantics, migration from plaintext `.env` and auth-profile values, and explicit rotation/recovery policy.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. The current-main negative check is high confidence: supported secret sources are env/file/exec and the `.env` path scrubs plaintext instead of encrypting it.

## Solution Assessment

No current implementation or proposed patch is ready to approve. A quick `ENC:` dotenv rewrite would be less maintainable than a reviewed key-management design that defines runtime plaintext boundaries, key storage, migration, and rotation.
