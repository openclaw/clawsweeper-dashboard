## Summary

Keep open. Current main and the latest release still clear the Gateway Token for non-identical WebSocket URL edits, while this PR is a narrow implementation candidate that repairs the linked source PR and follows the existing scoped-token storage contract. It needs maintainer review, mergeability/rebase handling, and exact-head validation rather than cleanup closure.

## What This Changes

The PR adds a Control UI storage helper, wires Overview Gateway Access URL edits through it, extends UI regression tests, and adds a changelog fix entry for preserving or reloading gateway-scoped tokens during WebSocket URL edits.

## Best Possible Solution

Land this PR or an equivalent scoped Control UI fix after maintainer review, preserving the normalized-scope behavior: keep the token for same-endpoint edits, reload an existing scoped token for another endpoint, clear when no scoped token exists, and close #41545 through the merged fix.

Yes. On current main, enter a Gateway Token in Overview > Gateway Access, edit the WebSocket URL to a same-endpoint but non-identical value such as adding a trailing slash, and the handler at `ui/src/ui/views/overview.ts:271` clears the token immediately.

## Solution Assessment

Yes. The PR's normalized-scope helper is the narrow maintainable fix because it avoids losing tokens for harmless edits while still preventing a gateway-scoped credential from being carried silently to a different endpoint.
