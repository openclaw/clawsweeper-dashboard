## Summary

Keep open: current main still deletes `agents.defaults.llm` without preserving or echoing `idleTimeoutSeconds`, the runtime/startup guard still gives no signal for valid configs carrying that legacy key, and open PR #74940 is already the paired implementation candidate.

## What This Changes

The issue asks for `doctor --fix` to preserve the legacy LLM idle-timeout value as a breadcrumb or targeted migration and for runtime/startup to warn when the retired key remains configured.

## Best Possible Solution

Keep `agents.defaults.llm` retired, but make doctor preserve the removed numeric intent in a clear breadcrumb or migrate only when the provider target is unambiguous, and add a one-shot runtime/startup warning through PR #74940 or an equivalent narrow patch.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main's migration test expects deletion-only behavior, and the runtime idle-timeout resolver cannot consume `agents.defaults.llm.idleTimeoutSeconds`, so a config relying only on that legacy key falls back to the supported provider/run/default timeout paths.

## Solution Assessment

Yes. The narrow maintainable direction is to improve migration diagnostics and pre-doctor visibility while keeping the retired key out of runtime semantics; restoring the legacy knob would extend a deprecated config surface.
