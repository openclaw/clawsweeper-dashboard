## Summary

Keep open. Current main still deletes `agents.defaults.llm` without preserving or echoing `idleTimeoutSeconds`, the checked-in test still expects deletion-only behavior, and the runtime/config guard still has no warning path for otherwise-valid configs carrying the legacy key. The issue is paired with open PR #74940 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Request that `openclaw doctor --fix` preserve or echo `agents.defaults.llm.idleTimeoutSeconds` when retiring the legacy block and that runtime startup warn when the legacy key remains.

## Best Possible Solution

Keep `agents.defaults.llm` retired, but make doctor repair preserve user intent by echoing or safely applying the numeric timeout to the provider timeout path, add a one-shot warning when valid runtime config still contains the legacy key, and leave the adjacent model-primary shadowing concern to #75720.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the migration test passes `agents.defaults.llm.idleTimeoutSeconds` and expects deletion-only output, while the idle-timeout resolver has no legacy-key input and falls back through provider/run/default timeout values.

## Solution Assessment

Yes. The narrow fix direction in open PR #74940 or an equivalent patch is the maintainable path; reintroducing `agents.defaults.llm` as a supported runtime knob would conflict with the documented provider-timeout replacement.
