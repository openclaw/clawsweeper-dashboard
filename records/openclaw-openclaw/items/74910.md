## Summary

Keep open: current main still drops the legacy timeout number during `doctor --fix`, even though current config validation now gives a stronger signal for legacy keys; open PR #74940 is already the linked closing implementation candidate.

## What This Changes

The issue asks for `doctor --fix` to preserve or echo legacy `idleTimeoutSeconds` values and for startup/runtime diagnostics before users fall back to the default 120s model idle timeout.

## Best Possible Solution

Keep `agents.defaults.llm` retired, but land or rework PR #74940 or an equivalent narrow patch that records the removed numeric value and keeps the startup diagnostic path covered.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the migration code and focused test delete the legacy block without retaining the number, while the timeout resolver cannot read that legacy path.

## Solution Assessment

Yes. The narrow maintainable direction is better migration diagnostics and pre-runtime visibility; reviving the retired key as a runtime input would extend a deprecated config surface.
