## Summary

Keep open: current main still drops the legacy timeout value during doctor migration, still has no valid-config startup warning for that legacy key, and open PR #74940 is already the paired implementation candidate.

## What This Changes

The issue asks doctor/runtime to preserve or surface `agents.defaults.llm.idleTimeoutSeconds` intent when migrating users to `models.providers.<id>.timeoutSeconds`.

## Best Possible Solution

Keep the retired key out of runtime semantics, but land or decline PR #74940 or an equivalent narrow patch that echoes the removed numeric value and adds a one-shot runtime/config warning.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current migration code and its test delete the legacy block without retaining the number, while the runtime timeout resolver cannot consume that legacy path.

## Solution Assessment

Yes. The maintainable fix is better migration diagnostics and pre-doctor visibility; reviving `agents.defaults.llm` as a runtime input would extend a deprecated config surface.
