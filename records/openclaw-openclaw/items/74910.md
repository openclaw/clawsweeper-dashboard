## Summary

Keep open. Current main still has the reported doctor/runtime gap: the legacy migration deletes `agents.defaults.llm` with a generic message, runtime timeout resolution still only honors provider/run/default timeout inputs, and config startup does not surface otherwise-valid legacy configs. The newer Linux/v2026.4.29 confirmation strengthens the report, while open PR #74940 remains the paired implementation candidate and should be reviewed or repaired before this issue closes.

## What This Changes

The issue asks for `openclaw doctor --fix` to preserve or echo the legacy `agents.defaults.llm.idleTimeoutSeconds` value and for runtime startup/config loading to warn when the retired key remains.

## Best Possible Solution

Keep `agents.defaults.llm` retired, improve doctor repair so the legacy numeric timeout is preserved or clearly echoed, and add a one-shot runtime/config-load warning that points users to `models.providers.<id>.timeoutSeconds`, with focused migration/config warning tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the checked-in migration test passes `agents.defaults.llm.idleTimeoutSeconds` and asserts deletion-only output, while the runtime resolver and config-load paths still omit a legacy-key warning path.

## Solution Assessment

Yes. The narrow maintainable direction is to improve diagnostics and migration intent while keeping the retired runtime key unsupported; restoring `agents.defaults.llm` as an active runtime contract would be broader and less consistent with the documented replacement path.
