## Summary

Keep open. Current main still has the reported deletion-only doctor migration and no runtime/startup warning for otherwise valid configs containing the retired timeout key; the issue is also paired with open PR #74940 via closing syntax, so it should wait for that PR or an equivalent fix to land.

## What This Changes

The issue asks doctor repair to preserve or echo `agents.defaults.llm.idleTimeoutSeconds` when removing the legacy block and to warn users at runtime when that legacy key is still present.

## Best Possible Solution

Keep `agents.defaults.llm` retired, but update the doctor repair path so the removed numeric timeout is visible or carried to the correct provider timeout target, add a one-shot warning before users run doctor, and resolve the issue through review of PR #74940 or an equivalent narrow patch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the checked-in migration test expects deletion-only output, and the runtime resolver has no input path for the legacy key, so configs relying on it fall back to the provider/run/default timeout behavior.

## Solution Assessment

Yes. The narrow maintainable fix is to improve migration diagnostics and pre-doctor warning behavior, not to reintroduce `agents.defaults.llm` as a supported runtime timeout knob.
