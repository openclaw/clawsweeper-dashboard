## Summary

Keep open: current main still deletes the legacy timeout block without preserving or echoing the numeric value, the runtime/startup path still gives no warning for valid configs carrying that legacy key, and open PR #74940 is already linked as the implementation candidate.

## What This Changes

The issue asks OpenClaw to stop silently discarding `agents.defaults.llm.idleTimeoutSeconds` during doctor repair and to warn users before they run doctor.

## Best Possible Solution

Keep `agents.defaults.llm` retired, but make doctor preserve the removed numeric intent in a clear breadcrumb or only migrate it when the provider target is unambiguous, and add a one-shot runtime warning through PR #74940 or an equivalent narrow patch.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main's migration test expects deletion-only behavior, and the idle-timeout resolver cannot consume `agents.defaults.llm.idleTimeoutSeconds`, so a config relying only on that legacy key falls back to the supported provider/run/default timeout paths.

## Solution Assessment

Yes. The narrow maintainable direction is to improve migration diagnostics and pre-doctor visibility while keeping the retired key out of runtime semantics; restoring the legacy knob would extend a deprecated config surface.
