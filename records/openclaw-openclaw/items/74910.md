## Summary

Keep open: current main still deletes `agents.defaults.llm` with a generic doctor message, has no runtime path from `agents.defaults.llm.idleTimeoutSeconds` into the LLM idle timeout resolver, and does not surface legacy-key guidance for otherwise valid config before returning from the config guard. Open PR #74940 references this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks `doctor --fix` to preserve or echo the legacy `agents.defaults.llm.idleTimeoutSeconds` value and to warn at runtime when the legacy key is still present.

## Best Possible Solution

Keep the retired key out of runtime semantics, but land or decline PR #74940 or an equivalent narrow patch that echoes the removed numeric value and adds pre-doctor visibility for the legacy key.

## Reproduction Assessment

Yes. Source inspection is sufficient: `doctor --fix` deletes the legacy block without retaining the number, the focused test asserts that behavior, and the resolver has no path from `agents.defaults.llm.idleTimeoutSeconds`.

## Solution Assessment

Yes. The narrow maintainable direction is better migration diagnostics and a one-shot warning, not reviving deprecated runtime config semantics; #74940 is the active implementation candidate for that path.
