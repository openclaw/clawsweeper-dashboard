## Summary

Keep open: current main still drops the legacy timeout value during doctor migration, still gives no valid-config runtime warning before doctor runs, and open PR #74940 is already the paired implementation candidate.

## What This Changes

The issue requests safer handling for legacy `agents.defaults.llm.idleTimeoutSeconds`: preserve or echo the value during `doctor --fix` and warn users before doctor runs.

## Best Possible Solution

Keep `agents.defaults.llm` retired, but land or decline PR #74940 or an equivalent narrow patch that echoes the removed numeric value and adds a one-shot legacy-key warning.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current migration code and its test delete the legacy block without retaining the number, while the runtime timeout resolver cannot consume that legacy path.

## Solution Assessment

Yes. The narrow maintainable fix is better migration diagnostics and pre-doctor visibility; reviving the retired key as a runtime input would extend deprecated config semantics.
