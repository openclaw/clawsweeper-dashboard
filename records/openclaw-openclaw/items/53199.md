## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested Slack `channel-list` and `search` actions, while the PR head is stale against the current Slack runtime boundary and still has concrete search/discovery/truncation blockers; the Slack search portion also overlaps the active #65691 implementation candidate.

## What This Changes

The PR adds Slack `channel-list` and `search` message actions with Slack API helpers, runtime/dispatch/action discovery wiring, Slack action config schema metadata, generated config baselines, and a channel action contract update.

## Best Possible Solution

Land one coherent Slack plugin implementation: keep `channel-list`/Slack search aligned with the current resolved-config runtime boundary, return search data through the shared `results.messages` contract, surface channel-list pagination or truncation metadata, add focused Slack tests/docs/generated config metadata, and coordinate the search portion with #65691 while preserving #11492’s channel-resolution intent.

## Reproduction Assessment

Yes for the PR blockers: source inspection against the PR head and current main reproduces them without a Slack token. The branch calls `loadConfig()` from an action helper, returns search hits outside the current formatter envelope, advertises search independently of the invokable account context, and truncates channel lists without metadata.

## Solution Assessment

No: the feature direction is useful, but this branch is not the best mergeable solution today. A rebased or coordinated implementation should follow current Slack runtime config/account patterns and avoid duplicating the active #65691 search work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass resolved Slack config into search helpers:** `extensions/slack/src/actions.ts:333-347`
  - body: `resolveUserToken()` calls ambient `loadConfig()` from the Slack action helper when no token override is passed. Current main requires action helpers to receive already-resolved config/account data from the runtime boundary, so search should use the account/config already resolved in `handleSlackAction()`.
  - confidence: 0.94
- **[P2] Return Slack search hits in the results envelope:** `extensions/slack/src/action-runtime.ts:494`
  - body: `handleSlackAction()` returns search data as top-level `messages`/`total`, but the shared CLI search formatter reads `payload.results.messages`. Successful Slack searches will skip the normal search-results table unless this returns the shared `results` envelope.
  - confidence: 0.9
- **[P2] Scope search discovery to the invokable account:** `extensions/slack/src/message-actions.ts:55-63`
  - body: Discovery adds `search` when any enabled Slack account has a user token and search enabled, but invocation without `accountId` resolves the default account. A non-default search-capable account can therefore expose an action the default invocation cannot run.
  - confidence: 0.84
- **[P2] Report channel-list truncation to callers:** `extensions/slack/src/action-runtime.ts:479`
  - body: `channel-list` stops at the default 500-channel cap but returns only `channels`. Larger workspaces are silently truncated, leaving callers unable to tell whether discovery is complete or needs pagination.
  - confidence: 0.82
