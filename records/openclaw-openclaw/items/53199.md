## Summary

Keep open. The requested Slack actions are still absent on current main, but this PR is not merge-ready because the branch is stale against the current Slack runtime boundary, has source-reproducible response/discovery bugs, and overlaps the separate open Slack search PR #65691.

## What This Changes

The PR adds Slack `channel-list` and `search` message actions, `channelList` config/schema metadata, generated config baselines, and action dispatch/discovery wiring.

## Best Possible Solution

Land one coherent Slack plugin implementation that preserves the current resolved-config and account-scoped discovery boundaries, coordinates search with #65691, returns search data through the shared `results.messages` contract, and exposes channel-list pagination or truncation metadata with focused Slack tests and docs.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head and current main reproduces them without a Slack token. The branch calls `loadConfig()` from an action helper, returns search hits outside the current formatter envelope, advertises search outside the selected account context, and truncates channel lists without metadata.

## Solution Assessment

No. The feature direction is useful, but this branch is not the narrowest maintainable solution today because it predates current Slack runtime boundaries and duplicates active search work in #65691.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass resolved Slack config into search helpers:** `extensions/slack/src/actions.ts:333-347`
  - body: `resolveUserToken()` calls ambient `loadConfig()` from the Slack action helper when no token override is passed. Current main requires Slack action helpers to receive already-resolved config/account data from the runtime boundary, so search should use the account/config already resolved in `handleSlackAction()` instead of re-reading raw config.
  - confidence: 0.94
- **[P2] Return Slack search hits in the results envelope:** `extensions/slack/src/action-runtime.ts:494`
  - body: `handleSlackAction()` returns search data as top-level `messages`/`total`, but the shared CLI search formatter reads `payload.results.messages`. Successful Slack searches will skip the normal search-results table unless this returns the shared `results` envelope.
  - confidence: 0.9
- **[P2] Scope search discovery to the selected account:** `extensions/slack/src/message-actions.ts:55-63`
  - body: Discovery adds `search` when any enabled Slack account has a user token and search enabled, but current main supports account-scoped discovery and invocation without `accountId` resolves the selected/default account. A non-default search-capable account can therefore expose an action the selected account cannot run.
  - confidence: 0.86
- **[P2] Expose channel-list truncation metadata:** `extensions/slack/src/action-runtime.ts:479`
  - body: `channel-list` stops at the default 500-channel cap but returns only `channels`. Larger workspaces are silently truncated, leaving callers unable to tell whether discovery is complete or whether they need another page/cursor.
  - confidence: 0.82
