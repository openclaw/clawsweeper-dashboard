## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested Slack `channel-list` and `search` actions, while the PR branch has concrete blockers around runtime config boundaries, shared search output shape, account-scoped action discovery, and silent `channel-list` truncation; it also overlaps active Slack search work in #65691.

## What This Changes

The PR adds Slack `channel-list` and `search` message actions, Slack API helpers, runtime and dispatch wiring, action discovery, Slack action config/schema metadata, generated config baselines, and a channel action contract update.

## Best Possible Solution

Land one coherent Slack plugin implementation: add `channel-list` with explicit pagination or truncation metadata, add Slack `search` using already-resolved account/config data and the shared `results.messages` output contract, update focused Slack tests/docs/generated config metadata, and coordinate the search portion with #65691. Because this PR closes #11492, maintainers should also decide whether `channel-info` belongs in this PR or remains a separate canonical follow-up.

## Reproduction Assessment

Yes for the PR blockers: the source-level checks reproduce them without a live Slack token. The PR head calls `loadConfig()` from an action helper, returns search hits outside the current formatter envelope, and advertises search independently of the invokable account context.

## Solution Assessment

No: the feature direction is useful, but this branch is not the best mergeable solution today. A rebased Slack-plugin implementation should use the current runtime config/account boundary, shared search result envelope, account-scoped discovery, and non-silent channel-list pagination semantics.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass resolved Slack config into search helpers:** `extensions/slack/src/actions.ts:333-347`
  - body: `resolveUserToken()` calls ambient `loadConfig()` from a Slack action helper when no token override is passed. Current main's runtime-action guard rejects `loadConfig()` in channel action helpers, so this path should use the already-resolved config/account data passed through `handleSlackAction()`.
  - confidence: 0.94
- **[P2] Return search hits under the results envelope:** `extensions/slack/src/action-runtime.ts:494`
  - body: `handleSlackAction()` returns Slack search data as top-level `messages`/`total`, but the shared CLI formatter renders search rows from `payload.results.messages`. Successful Slack searches will skip the normal search-results table unless this returns the shared `results` envelope.
  - confidence: 0.9
- **[P2] Advertise search only for the invokable account:** `extensions/slack/src/message-actions.ts:55-63`
  - body: Unscoped discovery advertises `search` if any Slack account has a user token and search enabled, but invocation without `accountId` resolves the default account. A non-default search-capable account can therefore expose an action the default invocation cannot run.
  - confidence: 0.84
- **[P2] Report channel-list truncation to callers:** `extensions/slack/src/actions.ts:302`
  - body: `channel-list` stops at the default 500-channel cap but returns only `channels`. Larger workspaces will be silently truncated, leaving callers unable to tell whether discovery is complete or needs another page/cursor.
  - confidence: 0.82
