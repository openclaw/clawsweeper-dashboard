## Summary

Keep open. Current main still lacks the proposed `agents.defaults.enforceFinalTag` config surface, and this PR is a focused opt-in mitigation for the open messaging-leak tracker with useful local-backport context, but it needs a current-main rebase, fresh checks, and real behavior proof before merge.

## What This Changes

The PR adds an `agents.defaults.enforceFinalTag` boolean to config/docs and propagates it through interactive and cron embedded agent runs so existing `<final>` filtering can be enabled for any provider.

## Best Possible Solution

Rebase this PR onto current main, preserve the existing final-tag runtime contracts, add or refresh focused current-base coverage if needed, and land it as an opt-in mitigation while keeping the canonical leak issue open for default-policy decisions.

## Reproduction Assessment

Yes. Source inspection shows current main already has strict final-tag filtering when `enforceFinalTag` is true, but the interactive path only enables it for reasoning-tag providers and the config schema/docs do not expose the proposed override.

## Solution Assessment

Yes for the PR's narrow opt-in scope: it reuses the existing `<final>` filter instead of adding another delivery layer. It should not be treated as the complete solution for https://github.com/openclaw/openclaw/issues/25592 because default direct/source delivery policy remains separate.
