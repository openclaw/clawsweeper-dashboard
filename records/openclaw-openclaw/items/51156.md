## Summary

Keep open. The opt-in final-tag config is still absent from current main and targets a real, still-open channel-leak problem, but this PR needs changes before merge because one added unit test is source-failing and external real behavior proof is not yet inspectable.

## What This Changes

Adds `agents.defaults.enforceFinalTag` with schema/docs/changelog coverage and threads it into reply and cron agent runs so final-tag filtering can be enabled for any provider.

## Best Possible Solution

Rebase the PR onto current main, preserve the existing provider-context checks while adding the opt-in config flag, port cron propagation to the current executor path, fix the unit test mock, and keep the broader default-delivery policy on the canonical issue.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main only enables final-tag enforcement for reasoning-tag providers, while direct/source delivery can still flush unphased assistant text before tool execution on automatic delivery paths.

## Solution Assessment

Mostly yes as a narrow opt-in escape hatch, but not as a complete solution for the broader leak class. The current PR needs a rebase, a test fix, and inspectable real behavior proof before it is the best mergeable version.
