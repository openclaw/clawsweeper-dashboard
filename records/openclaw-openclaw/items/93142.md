## Summary

Keep open: current `main` and latest `v2026.6.6` still do not provide a Docker/Compose env input for `gateway.controlUi.allowedOrigins`, and the request is valid but needs maintainer approval because it adds a security-sensitive env/config surface.

## What This Changes

The issue requests a Docker/Compose-friendly way to configure `gateway.controlUi.allowedOrigins` for remote Control UI deployments such as Portainer-managed stacks.

## Best Possible Solution

Adopt an explicit Docker/setup-owned strict-JSON input only if maintainers approve the new env surface, persist it into canonical `gateway.controlUi.allowedOrigins` before gateway startup, and document precedence with focused setup and runtime coverage.

## Reproduction Assessment

Not a bug reproduction, but the request is source-reproducible: current compose/setup/docs have no env input for remote allowed origins, and runtime only accepts persisted origins or the dangerous fallback. I did not run Docker because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the config contract; the requested direction is valid, but Docker setup persistence looks safer than a steady-state runtime env fallback because runtime config is supposed to read canonical config only.
