## Summary

Keep open: current main and v2026.6.9 still lack the requested Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and adding one controls a security-sensitive browser-origin allowlist that needs maintainer product/security approval.

## What This Changes

Request support for setting `gateway.controlUi.allowedOrigins` from Docker/Compose environment for Portainer-style deployments without a manual `config set` command.

## Best Possible Solution

Approve a strict-JSON Docker/Compose setup-time input that validates origins fail-closed, preserves existing config, and documents precedence and security caveats.

## Reproduction Assessment

Yes, source-level: current Compose and Docker setup paths do not expose a non-localhost allowed-origins env input, while gateway startup still enforces explicit origins or deliberate fallback for non-loopback Control UI. I did not run Docker because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict JSON persistence path looks safer than a runtime-only env shim because runtime reads canonical config and this setting gates browser-origin authorization.
