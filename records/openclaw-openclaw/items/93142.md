## Summary

Keep open: current main and v2026.6.9 still lack a Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and adding one would create a security-sensitive config surface that needs maintainer product approval.

## What This Changes

The issue requests a compose-friendly environment/config input for `gateway.controlUi.allowedOrigins` so Portainer-style Docker deployments can configure remote Control UI browser origins without an extra CLI shell step.

## Best Possible Solution

Add a maintainer-approved strict-JSON Docker/Compose setup path that validates fail-closed, preserves existing origins, documents precedence and security caveats, and avoids a silent runtime-only shadow of canonical config.

## Reproduction Assessment

Yes, source-level: current Compose and Docker setup paths do not expose a non-localhost allowed-origins env input, while gateway startup still enforces explicit origins or deliberate fallback for non-loopback Control UI. I did not run Docker because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict JSON persistence path looks safer than a runtime-only env shim because runtime reads canonical config and this setting gates browser-origin authorization.
