## Summary

## What This Changes

The issue asks for a Docker/Compose-friendly environment input to configure `gateway.controlUi.allowedOrigins` for Portainer-style remote Control UI deployments without an extra shell config command.

## Best Possible Solution

Add a maintainer-approved Docker/Compose path, preferably strict-JSON setup-time persistence, that validates fail-closed, preserves existing explicit origins, documents precedence/security caveats, and covers Docker setup plus gateway config tests.

## Reproduction Assessment

Yes, source-level: current Compose and Docker setup paths do not expose a non-localhost allowed-origins env input, while runtime still requires configured origins or an explicit dangerous fallback for non-loopback Control UI.

## Solution Assessment

Unclear until maintainers choose the contract; setup-time strict JSON persistence looks safer than a runtime-only env shim because runtime reads canonical config and this setting gates browser-origin authorization.
