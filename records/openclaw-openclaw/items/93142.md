## Summary

Keep open: current main and v2026.6.8 still lack a Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and the request remains a security-sensitive setup/product decision rather than conservative cleanup.

## What This Changes

The issue requests Docker/Compose environment-variable support for configuring `gateway.controlUi.allowedOrigins` in Portainer-style deployments.

## Best Possible Solution

Add a maintainer-approved Docker/Compose path that accepts explicit remote Control UI origins with fail-closed parsing, clear precedence, preservation of existing origins, focused tests, and public docs that keep the current security boundary explicit.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.8 shows no compose-friendly allowed-origins env input, Docker setup only seeds loopback origins, and non-loopback runtime still requires configured origins or the explicit dangerous fallback.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict-JSON persistence path looks safer than a runtime-only env shim because runtime code is expected to consume canonical config.
