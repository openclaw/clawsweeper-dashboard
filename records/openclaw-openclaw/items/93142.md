## Summary

Keep open: current main and latest release still do not provide a Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and the requested surface is security-sensitive enough to need maintainer product/security review before implementation.

## What This Changes

Request support for configuring `gateway.controlUi.allowedOrigins` from Docker/Compose environment values for Portainer-style deployments without a separate config command.

## Best Possible Solution

Add a maintainer-approved Docker/Compose path, preferably strict-JSON setup-time persistence, that validates fail-closed, preserves existing explicit origins, documents precedence and security caveats, and includes Docker setup plus gateway config coverage.

## Reproduction Assessment

Yes, source-level. Current Compose and Docker setup paths do not expose a non-localhost allowed-origins env input, while gateway startup and tests still require configured origins or deliberate dangerous fallback for non-loopback Control UI.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict JSON persistence path looks safer than a runtime-only env shim because runtime reads canonical config and this setting gates browser-origin authorization.
