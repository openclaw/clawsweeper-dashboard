## Summary

Keep open: current main and `v2026.6.9` still lack a Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and adding one changes a security-sensitive origin allowlist that needs maintainer contract approval.

## What This Changes

The issue requests a Docker/Compose-friendly environment input for `gateway.controlUi.allowedOrigins` so remote Control UI origins can be configured in Portainer-style deployments without running a separate config command.

## Best Possible Solution

Add a maintainer-approved strict-JSON Docker/Compose setup-time path that validates fail-closed, preserves existing explicit origins, documents precedence and security caveats, and avoids a runtime-only config shadow.

## Reproduction Assessment

Yes, source-level: current Compose and Docker setup paths do not expose a non-localhost allowed-origins env input, while gateway startup still enforces explicit origins or deliberate fallback for non-loopback Control UI. I did not run Docker because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict JSON persistence path looks safer than a runtime-only env shim because runtime reads canonical config and this setting gates browser-origin authorization.
