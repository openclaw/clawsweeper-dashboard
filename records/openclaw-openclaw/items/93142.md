## Summary

Keep open: current main and v2026.6.8 still do not provide a Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and the request remains a useful but security-sensitive setup/config decision rather than cleanup closure.

## What This Changes

The issue requests a Docker/Compose-friendly way to configure `gateway.controlUi.allowedOrigins` for remote Control UI deployments without running an extra config command.

## Best Possible Solution

Add a maintainer-approved Docker/Compose path that accepts a strict, validated origin list, preserves existing explicit origins unless deliberately replaced, documents precedence, and fails closed for malformed or unsafe input.

## Reproduction Assessment

Yes, source-level: current main has no Docker/Compose env input for non-localhost `gateway.controlUi.allowedOrigins`, and the startup helper only seeds loopback/custom defaults without persistence. I did not run Docker because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict-JSON persistence path looks safer than a runtime-only env override, but either shape touches a security-sensitive config surface.
