## Summary

Keep open: current main and the latest release still do not expose a Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and adding one is a security-sensitive config contract that needs maintainer approval.

## What This Changes

The issue requests a Docker/Compose-friendly environment input for `gateway.controlUi.allowedOrigins` so managed stacks can configure remote Control UI origins without a post-start CLI command.

## Best Possible Solution

Add a maintainer-approved strict-JSON Docker/Compose setup path that persists validated explicit origins without overwriting existing config, documents precedence and security caveats, and coordinates with the broader Docker-manager first-run work.

## Reproduction Assessment

Yes, source-level: current Compose and Docker setup paths do not expose a non-localhost allowed-origins env input, while gateway startup still enforces explicit origins or deliberate fallback for non-loopback Control UI. I did not run Docker because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict JSON persistence path looks safer than a runtime-only env shim because runtime reads canonical config and this setting gates browser-origin authorization.
