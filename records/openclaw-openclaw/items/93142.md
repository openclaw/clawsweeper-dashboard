## Summary

Keep open: current main still lacks a first-class Docker/Compose environment input for non-localhost `gateway.controlUi.allowedOrigins`, and adding one is a security-sensitive config-surface decision that maintainers should choose deliberately.

## What This Changes

The issue requests a Docker/Compose-friendly environment configuration path for `gateway.controlUi.allowedOrigins`, mainly for Portainer-style remote Control UI deployments.

## Best Possible Solution

Maintainers should choose a strict JSON Docker/Compose contract, preferably setup-time persistence that validates fail-closed, preserves existing origins, documents precedence, and fits the broader Docker-manager bootstrap direction.

## Reproduction Assessment

Yes, source-level. Current main has no Compose/env input for remote `gateway.controlUi.allowedOrigins`, Docker setup only seeds loopback origins, and non-loopback startup still requires configured origins or the dangerous fallback.

## Solution Assessment

Unclear until maintainers choose the contract. A setup-time strict-JSON persistence path looks safer than a runtime-only env shim, but security and precedence policy are the key decisions.
