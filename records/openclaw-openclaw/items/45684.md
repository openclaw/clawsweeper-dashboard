## Summary

Keep this PR open. Current main still does not implement the Slack DM message-tool threading fix, and the PR remains relevant, but the branch is not ready to land because it is stale against the current `extensions/slack` layout and mixes the Slack fix with unrelated dependency/test changes.

## What This Changes

The PR attempts to keep Slack DM message-tool attachments in the active thread by adding a DM user-address threading field, changing Slack auto-thread matching, adding regression tests, and also adding unrelated `@pierre/diffs` and Windows test changes.

## Best Possible Solution

Land a focused Slack plugin fix under `extensions/slack`: preserve `currentChannelId` as the native `D...` id for inferred Slack actions, carry the current DM `user:<id>` recipient through a generic or Slack-owned threading context, update `resolveSlackAutoThreadId` to match DM sends, and add targeted regression tests without unrelated dependency or platform-test changes.

## Reproduction Assessment

Yes. A high-confidence static/unit reproduction exists on current main: build a DM tool context with `To = user:<id>` and `NativeChannelId = D...`, then call `resolveSlackAutoThreadId` with `to: user:<id>`; current tests and code show it returns `undefined` instead of the active thread timestamp.

## Solution Assessment

No, not as currently packaged. The author's two-field direction is likely the right behavioral shape, but the branch needs a focused rebase onto current Slack plugin paths and should split the unrelated dependency/CI edits before maintainers evaluate the API contract change.
