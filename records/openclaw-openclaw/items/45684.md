## Summary

Keep this PR open, but it is not mergeable as-is: the Slack DM threading bug still appears valid on current main, while the branch is stale against the plugin-owned Slack paths, lacks real Slack behavior proof, and mixes in unrelated dependency/Windows test changes.

## What This Changes

The PR tries to keep Slack DM message-tool media sends in the active thread by carrying a DM user target in threading context and matching it during Slack auto-thread resolution, while also adding unrelated dependency and Windows test changes.

## Best Possible Solution

Land a clean Slack-plugin fix in `extensions/slack/src/threading-tool-context.ts` and `extensions/slack/src/action-threading.ts`, with regression coverage through the current message-action normalization path and without unrelated CI or dependency changes.

## Reproduction Assessment

Yes for a source-level reproduction path: current main stores native DM channel ids in Slack tool context, then the Slack auto-thread resolver rejects user targets and only compares channel ids. I did not run a live Slack workspace reproduction.

## Solution Assessment

No. The two-field direction is reasonable, but this branch must be updated for the current plugin-owned Slack files and current bare-user-id normalization, and the unrelated dependency/test changes should be split out.
