## Summary

Keep open: current main still has the source-reproducible Slack DM message-tool threading bug, and this PR has the right two-field direction but is stale against the active Slack plugin paths and mixed with unrelated package/test changes.

## What This Changes

The PR adds Slack DM user-target threading context and resolver coverage so message-tool media/file sends can inherit the current Slack DM thread, while also changing package resolution and unrelated Windows test guards.

## Best Possible Solution

Land a narrow Slack plugin fix in extensions/slack that preserves native D-channel IDs for action inference, adds a separate DM user target for auto-thread matching, includes regression tests and a changelog entry, and omits dependency or unrelated CI-test changes.

## Reproduction Assessment

Yes, from source inspection on current main. A Slack DM tool context stores only the native D... channel id, while the message-tool auto-thread resolver rejects user:<id> targets before it can return the active thread timestamp.

## Solution Assessment

No, not as currently packaged. The author's later two-field design is the maintainable direction, but it needs to be applied to the active extensions/slack files and split from unrelated dependency and platform-test changes.
