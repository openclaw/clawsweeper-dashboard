## Summary

Keep open. Current main still has the macOS Voice Wake path forwarding with default main/WebChat options, while the selected session state used elsewhere in the macOS app is not consulted by Voice Wake. The gateway trigger-based voicewake.routing layer exists, but it does not satisfy the issue’s requested binding to the macOS session picker.

## What This Changes

Voice Wake in the macOS app should send captured voice input to the selected session target, such as a Telegram group, instead of routing to the main WebChat session.

## Best Possible Solution

Wire macOS Voice Wake forwarding to resolve the currently selected session target at send time, including delivery metadata for channel sessions, and add a regression test covering a selected non-WebChat session target. If maintainers prefer gateway voicewake.routing as the source of truth, the macOS picker should explicitly update that config and the docs should say so.

## Reproduction Assessment

Yes. The issue’s manual path is clear, and current source provides a high-confidence reproduction path: both Voice Wake send routes pass only voiceWakeTrigger into ForwardOptions, which defaults to main/WebChat.

## Solution Assessment

No current solution is in place on main. The narrow maintainable fix is to reuse the selected-session/session-row data already exposed by the macOS app and gateway, then pass the resolved sessionKey and delivery target into VoiceWakeForwarder instead of relying on its defaults.
