## Summary

Keep open. The PR targets a real current-main gap in WebSocket reasoning delivery, but the proposed patch removes the reasoning-visibility gate too broadly and can expose thinking events in sessions where normal assistant output is suppressed.

## What This Changes

The PR changes agent subscription handling so `stream: "thinking"` WebSocket events can be emitted without requiring a channel-specific `onReasoningStream` callback, and broadens the reasoning stream call sites in message update/end handlers.

## Best Possible Solution

Land a narrow agent subscription fix that decouples WebSocket thinking broadcasts from the channel callback only for explicit reasoning-stream visibility, preserves silent/background and deterministic-approval suppression, adds regression coverage for callback-less gateway sessions, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Static branch tracing gives a high-confidence reproduction: on current main, a `reasoningMode: "stream"` run without `onReasoningStream` cannot reach `emitAgentEvent({ stream: "thinking" })`; with this PR, a non-silent default/off or deterministic-approval-suppressed run with `<think>` content can reach that event path.

## Solution Assessment

No. The maintainable fix is to decouple `streamReasoning` from `onReasoningStream` while keeping it tied to explicit reasoning visibility, then make the callback optional and carry existing suppression rules into thinking-event emission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep thinking events behind reasoning visibility:** `src/agents/pi-embedded-subscribe.ts:636`
  - body: Removing the `state.streamReasoning` guard from `emitReasoningStream` lets the broadened callers publish `stream: "thinking"` events for ordinary non-silent runs where reasoning visibility is still off. That also bypasses deterministic-approval suppression, so hidden reasoning can appear in WebSocket clients even though normal assistant output is suppressed. Decouple the callback requirement without dropping the reasoning-mode and suppression gates.
  - confidence: 0.9
- **[P2] Avoid scanning every normal text delta:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:307-309`
  - body: `ctx.state.streamReasoning || !ctx.params.silentExpected` is true for nearly every visible non-reasoning run, so this now runs a full-buffer tagged-thinking extraction on each text delta. Since `deltaBuffer` grows across the turn, long normal replies take repeated scans on a hot streaming path. Keep the scan behind an explicit reasoning/WS-thinking visibility condition or add a cheap incremental/tag-presence guard.
  - confidence: 0.84
- **[P2] Handle final-only thinking for callback-less streams:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:640`
  - body: This new unconditional `rawThinking` call still depends on `rawThinking`, but `rawThinking` is only extracted when `includeReasoning || streamReasoning` is true. On current main, `streamReasoning` remains false without `onReasoningStream`, so gateway clients still miss providers that only surface thinking at `message_end`. The fix should update the stream-mode state or extraction path rather than only changing this final guard.
  - confidence: 0.78
