## Summary

Keep open: this looks like a valid fresh regression, not a duplicate close of the earlier subagent fix. Current main has a CLI compaction path that resolves the default `legacy` context engine without first initializing built-in engines, while the repaired `sessions_spawn` path explicitly initializes before resolving.

## What This Changes

The issue reports that CLI transcript persistence on `2026.5.3-1` logs `Context engine "legacy" is not registered` for claude-cli turns and leaves `sessions_history` backed by incomplete transcript data.

## Best Possible Solution

Mirror the subagent fix in the CLI compaction lifecycle by initializing built-in context engines before resolving the context engine, then add a regression test that fails if the default `legacy` engine is resolved before initialization.

## Reproduction Assessment

Yes, at source level: a CLI turn with `sessionFile` and `contextTokens` enters `runCliTurnCompactionLifecycle`, then calls `resolveContextEngine` without the initializer that registers `legacy`. I did not perform the live claude-cli gateway run, so this is not marked as fully reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is to call the existing context-engine initialization guard in the CLI compaction path and cover that ordering in `cli-compaction.test.ts`. A doctor check may be useful later, but it is not needed to fix the runtime regression.
