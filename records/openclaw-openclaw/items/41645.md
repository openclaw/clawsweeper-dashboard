## Summary

Keep open: current main and v2026.6.6 still have session/model/provider/agent/channel/date usage reporting, but no canonical user or accounting-subject dimension. The request remains valid product work, but it needs maintainer product and security direction before implementation.

## What This Changes

Request per-user token usage accounting for input, output, and total tokens with day/week/month views through `/usage` or the Dashboard.

## Best Possible Solution

Define a canonical accounting subject, retention/data-minimization policy, and viewer authorization model before extending the existing session usage schema, Gateway aggregation, `/usage`, and Dashboard filters with per-subject daily aggregates.

## Reproduction Assessment

## Solution Assessment

No: the raw `user_id` log-field proposal is not the best standalone shape. The safer solution is a canonical accounting subject plus access-control and retention semantics, then a schema/Gateway/UI extension.
