## Summary

Keep open: current main still lacks token usage accounting grouped or filtered by user, and this needs maintainer product decisions about identity, privacy, and reporting scope before implementation.

## What This Changes

The issue requests per-user token usage accounting for input, output, and total tokens with day/week/month reporting through `/usage` or the Dashboard.

## Best Possible Solution

Define the canonical accounting identity and privacy model, then add by-user aggregates and filters through the session usage API, `/usage`, Dashboard, tests, and docs.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing behavior report. Source inspection gives a high-confidence absence check for by-user usage aggregation on current main.

## Solution Assessment

Unclear as proposed; adding `user_id` fields to logs after model calls would not define cross-channel identity, access controls, or UI/API contracts. The maintainable path is an additive user-accounting contract wired through the existing usage aggregation surfaces.
