## Summary

Keep open: current main has usage tracking, `/usage cost`, and the Control UI usage dashboard, but it still does not provide token accounting grouped by user. The request is a real feature that needs maintainer product/privacy decisions before implementation.

## What This Changes

The issue requests per-user input/output/total token usage reporting with day/week/month views through `/usage` or the Dashboard.

## Best Possible Solution

Define a canonical usage-accounting user identity, then add by-user aggregates and filters through the session usage API, `/usage`, Control UI, and docs with explicit privacy semantics.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug. Source and docs inspection show existing usage surfaces, but no current-main path provides user-dimensional token aggregation.

## Solution Assessment

Unclear as proposed; adding log fields after each model call would not define cross-channel user identity or access controls. The maintainable solution is an additive user-identity/accounting contract wired through existing usage aggregation and UI surfaces.
