## Summary

Keep open: current main still lacks token usage accounting grouped by user, and this is a real feature request that needs maintainer decisions on identity, privacy, and reporting scope before implementation.

## What This Changes

Add per-user token usage accounting for input, output, and total tokens, with day/week/month reporting through `/usage` or the Dashboard.

## Best Possible Solution

Define a canonical usage-accounting identity and privacy model, then add user aggregates and filters through the existing sessions usage API, `/usage`, Dashboard, tests, and docs.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug. Source inspection gives a high-confidence absence check for by-user usage aggregation on current main.

## Solution Assessment

No, not as proposed; adding `user_id` fields to model-call logs would not define cross-channel identity, authorization, or UI/API contracts. The maintainable path is an additive user-accounting contract wired through the existing usage aggregation surfaces.
