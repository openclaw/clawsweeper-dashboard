## Summary

Keep open. Current main has session/provider/model/agent/channel usage accounting and Dashboard usage views, but it still lacks per-user token rollups, user filtering, weekly/monthly user reporting, and quota/billing semantics.

## What This Changes

Request per-user token usage accounting with input/output/total token totals, daily/weekly/monthly reporting, and `/usage` or Dashboard views for billing and quota scenarios.

## Best Possible Solution

Define a typed normalized user/principal contract for usage accounting, expose per-user period rollups through `sessions.usage`, then add Dashboard and `/usage` views with documented privacy and quota boundaries.

## Reproduction Assessment

Not applicable as a feature request; source inspection confirms the missing capability. Current usage types, protocol params, gateway aggregation, Dashboard query helpers, and token-use docs expose session/model/provider/agent/channel/daily metrics but no user aggregate or filter.

## Solution Assessment

Unclear as proposed: adding raw `user_id` fields to logs is not enough for OpenClaw because it needs a normalized cross-channel principal and access-controlled reporting. The maintainable path is to model that principal first, then expose typed aggregates and UI/docs around it.
