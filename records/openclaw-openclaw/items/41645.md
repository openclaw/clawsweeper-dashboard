## Summary

Keep open. Current main has substantial session, provider, agent, channel, and daily usage accounting, but it still lacks the requested per-user input/output/total token aggregation, user filter/reporting surface, weekly/monthly user rollups, and quota/billing semantics. The prior ClawSweeper comment also kept this open, and the adjacent usage issues/PRs I found cover log format or agent/channel aggregation rather than this per-user accounting request.

## What This Changes

Request per-user token usage accounting with input/output/total token totals, daily/weekly/monthly rollups, and `/usage` or Dashboard reporting for multi-user billing and quota management.

## Best Possible Solution

Define a core usage-accounting contract around a normalized principal key, extend the typed `sessions.usage` protocol/result with per-user and period rollups, add Dashboard and `/usage` reporting/filtering, and document identity, privacy, retention, and any quota-enforcement boundaries.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature request. The missing capability is high-confidence from source inspection: current `sessions.usage`, protocol params, Dashboard query helpers, overview UI, and docs lack a user aggregate or user filter.

## Solution Assessment

Unclear as proposed: adding raw `user_id` fields to logs would not be enough because OpenClaw needs a normalized cross-channel principal and access-controlled reporting. The maintainable path is to model that principal first, then expose typed aggregates and UI/docs around it.
