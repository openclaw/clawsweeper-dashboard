## Summary

Keep open: current `main` has session/global token usage tracking, `/usage cost`, exports, and a Control UI usage dashboard, but it still does not provide token accounting grouped by user. This is a real feature request that needs product and privacy decisions before implementation.

## What This Changes

The issue requests per-user input/output/total token usage accounting with daily, weekly, and monthly views through `/usage` or the dashboard.

## Best Possible Solution

Define a canonical usage-accounting user identity, then add by-user aggregates and filters through the session usage API, `/usage`, Control UI, exports, and docs with explicit privacy semantics.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug. Source and docs inspection on current `main` shows existing usage surfaces, but no path provides token aggregation by user.

## Solution Assessment

No as proposed; adding log fields after each model call would not define cross-channel user identity, visibility rules, or access controls. The maintainable solution is an additive accounting identity contract wired through existing usage aggregation and UI surfaces.
