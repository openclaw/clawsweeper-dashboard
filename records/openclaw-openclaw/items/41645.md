## Summary

Keep open: current main still lacks token usage accounting grouped by user. The existing usage path aggregates by session, model, provider, agent, channel, and day, and the requested user dimension needs maintainer product decisions about canonical identity, privacy, and reporting scope.

## What This Changes

Request by-user input/output/total token usage accounting with daily, weekly, and monthly reporting in `/usage` or the Dashboard.

## Best Possible Solution

Define a canonical usage-accounting user identity and privacy model, then add by-user aggregates and filters through the session usage API, `/usage`, Dashboard, tests, and docs.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug. Source inspection gives a high-confidence absence check for by-user usage aggregation on current main.

## Solution Assessment

Unclear as proposed; adding `user_id` log fields after model calls would not define cross-channel identity, access controls, or UI/API contracts. The maintainable path is an additive user-accounting contract wired through the existing usage aggregation surfaces.
