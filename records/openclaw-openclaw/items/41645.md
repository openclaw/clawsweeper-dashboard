## Summary

Keep this open: current main has substantial session/provider/agent/channel/date usage accounting, but it does not implement the requested per-user token usage dimension, and choosing that identity boundary is a product/API decision rather than conservative cleanup.

## What This Changes

Request per-user input/output/total token usage statistics with day/week/month views available through `/usage` or the dashboard.

## Best Possible Solution

Define the canonical user identity and access policy first, then extend the session usage cache, `sessions.usage` schema/result, dashboard filters/exports, and docs to aggregate input/output/total tokens by user and time bucket.

## Reproduction Assessment

Not applicable; this is an enhancement request, not a report of broken existing behavior. Source inspection shows current usage surfaces lack the requested user-level grouping.

## Solution Assessment

Unclear until maintainers choose the user identity boundary. A narrow implementation should build on existing session-cost aggregates rather than only adding ad hoc log fields.
