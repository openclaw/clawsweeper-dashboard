## Summary

Keep open: current main and v2026.6.8 still expose token usage by session, date, model, provider, agent, and channel, but not by user/accounting subject. The request is valid product work, but it needs product and security direction before implementation because identity-linked usage and spend data need a canonical subject, retention policy, and viewer authorization model.

## What This Changes

The issue requests per-user token accounting with input/output/total token totals by day/week/month, visible through /usage or the dashboard.

## Best Possible Solution

Define a canonical accounting subject plus authorization and retention rules, then extend the session usage schema, gateway aggregation, /usage output, and dashboard filters around that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source and release inspection show the requested user/accounting-subject dimension is absent from current usage surfaces.

## Solution Assessment

No: the suggested raw user_id log field is not the best standalone solution. A safer implementation needs a canonical accounting subject, viewer authorization, retention policy, and then schema/Gateway/UI changes.
