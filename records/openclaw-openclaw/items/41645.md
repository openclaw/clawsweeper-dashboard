## Summary

Keep open: current main still reports usage by session, model, provider, agent, channel, and date, but it does not define per-user token accounting. The request is useful, but it needs a maintainer product decision around canonical identity, privacy, authorization, and reporting scope before implementation.

## What This Changes

Request per-user token usage accounting with input/output/total tokens over day/week/month and `/usage` or Dashboard reporting.

## Best Possible Solution

Define a canonical, privacy-aware usage identity and authorization model, then add by-user aggregates and filters through `sessions.usage`, `/usage`, Dashboard, tests, and docs.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug. Source inspection gives a high-confidence absence check because the schema, result type, gateway aggregation, and UI filters all lack a user dimension.

## Solution Assessment

No, not as proposed; adding `user_id` fields to model-call logs would not define cross-channel identity, authorization, or UI/API contracts. The maintainable path is an additive user-accounting contract wired through the existing usage aggregation surfaces after maintainer product review.
