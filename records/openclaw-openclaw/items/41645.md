## Summary

Keep open: current main and the latest tagged release still expose session/model/provider/agent/channel/date usage reporting, but not a canonical user or accounting-subject dimension; the request remains valid product work with security implications.

## What This Changes

Request adds per-user token usage accounting with input/output/total tokens, day/week/month grouping, and `/usage` or Dashboard visibility.

## Best Possible Solution

Define a canonical accounting subject plus retention and viewer-authorization rules, then extend the session usage schema, Gateway aggregation, `/usage`, Dashboard filters, and docs with per-subject daily aggregates.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source and release inspection give a high-confidence feature-gap check because current usage reporting lacks a user/accounting-subject dimension.

## Solution Assessment

No: the raw `user_id` log-field proposal is not the best standalone solution. The safer shape is a canonical accounting subject with access-control and retention semantics before adding schema, Gateway, and UI reporting.
