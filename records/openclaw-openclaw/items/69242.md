## Summary

Keep open. Current main still captures exec termination cause internally, but foreground exec and `process.poll` structured details drop fields that would distinguish SIGKILL, timeout, and no-output-timeout cases; the merged Linux OOM victim-selection work is related but explicitly leaves SIGKILL observability as separate work.

## What This Changes

The issue asks OpenClaw to stop surfacing broad Linux exec discovery-command SIGKILLs as abrupt, low-context failures and instead complete or report a clear termination cause.

## Best Possible Solution

Add backward-compatible termination metadata to foreground exec and `process.poll` details using the existing supervisor/runtime fields, with focused regression coverage for signal, timeout, and no-output-timeout paths.

## Reproduction Assessment

Yes for the remaining diagnostics gap at source level: current main records exitSignal, failureKind, reason, timedOut, and noOutputTimedOut internally, then omits key fields from foreground exec and `process.poll` details. The intermittent broad-search SIGKILL trigger itself was not live-reproduced.

## Solution Assessment

Yes for the narrow first repair. Additive metadata propagation is the maintainable fix here; OOM victim selection, SIGTERM escalation, OS-cause telemetry, and broad recursive-scan guardrails should stay separate unless a maintainer explicitly scopes them in.
