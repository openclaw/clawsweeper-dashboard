## Summary

Keep open. Current main still records termination reason and signal inside the supervisor/runtime, but the foreground exec result and process poll details do not consistently expose that structured cause to callers, so the reported diagnostic gap remains actionable.

## What This Changes

The issue requests clearer diagnostics for intermittent Linux broad `find`/`grep` exec commands that are killed with `SIGKILL` without host OOM evidence.

## Best Possible Solution

Add an additive, generic exec/process termination diagnostics path that surfaces the existing supervisor/runtime metadata in foreground exec results and background process list, poll, and log details; keep Linux OOM victim selection and recursive-scan guardrails as separate product/resource-policy work.

## Reproduction Assessment

Partially yes: the exact intermittent Linux broad-search SIGKILL is not deterministic from the report, but the remaining diagnostics gap is source-verifiable by following a forced signal or timeout through RunExit, ExecProcessOutcome, ExecToolDetails, and process.poll details.

## Solution Assessment

Yes: the narrowest maintainable solution is to expose existing termination metadata and cover it with focused signal/timeout tests. Changing Linux resource policy or adding broad `find`/`grep` guardrails should not be folded into this repair.
