## Summary

Keep open. The Linux broad-search SIGKILL trigger is still intermittent, but current main still source-reproduces the actionable diagnostics gap: supervisor/runtime code records signal and failure metadata, while foreground exec and parts of process polling do not expose that metadata in structured result details.

## What This Changes

The issue asks for broad Linux `exec` discovery commands killed with SIGKILL to either complete or fail with clear timeout/resource/termination diagnostics.

## Best Possible Solution

Expose existing supervisor/runtime termination metadata through backward-compatible foreground exec and process poll/log details, then add focused regression tests for signal, timeout, and no-output-timeout paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining diagnostics gap: source inspection shows `RunExit` and `ExecProcessOutcome` carry the signal/failure data, while foreground exec and process poll details drop it. The original broad Linux search SIGKILL itself remains intermittent and was not deterministically reproduced here.

## Solution Assessment

Is this the best way to solve the issue? Yes: additive metadata propagation is the narrowest maintainable first fix. Changing Linux OOM policy, signal escalation, or adding broad-scan guardrails should remain separate focused items.
