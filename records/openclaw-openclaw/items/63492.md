## Summary

Keep open. Current main and v2026.6.10 still expose path-based `session.store`, local `sessions.json` metadata, and JSONL transcript files; the active SQLite/accessor work is prerequisite overlap, not the requested external multi-instance session backend.

## What This Changes

The issue requests a pluggable external session storage backend so multiple OpenClaw instances can share session state without sticky routing.

## Best Possible Solution

Keep this as the canonical external shared session-storage design item, coordinate it with the SQLite/accessor migration and broader database-abstraction work, and require a maintainer-approved storage/API boundary before implementation.

## Reproduction Assessment

Yes for the architecture gap: source and docs show current main resolves `session.store` to a local `sessions.json` path, reads JSON from disk, writes JSON back to disk, and uses JSONL transcript files. I did not run a live multi-instance deployment because the missing backend is clear from current source and release inspection.

## Solution Assessment

No for the literal Redis/Mongo key-value factory as written; it is a useful direction but too narrow for OpenClaw's session/transcript lifecycle. The maintainable path is a maintainer-approved storage boundary covering session metadata, transcripts, transaction semantics, pruning, migrations, credentials, and distributed consistency.
