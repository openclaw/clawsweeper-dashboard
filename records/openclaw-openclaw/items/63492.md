## Summary

Keep open. Current main still uses path-based `session.store`, local `sessions.json` metadata, per-session JSONL transcripts, and in-process writer queues; the related closed reports now point to this issue, while broader horizontal-scaling and session-identity items do not implement the requested external backend contract.

## What This Changes

Request adds a pluggable external session storage abstraction, such as Redis or MongoDB, so multiple OpenClaw instances can share stateless session state.

## Best Possible Solution

Define a versioned gateway-owned session persistence contract covering metadata and transcripts, keep filesystem storage as the compatibility default, prove one multi-instance-safe backend, and update config/schema/docs, migration/doctor, and SDK/plugin boundaries together.

## Reproduction Assessment

Not applicable: this is a feature and architecture request, not a reproducible bug. Source and docs inspection gives high confidence that current main still lacks an external session-storage backend selector or adapter API.

## Solution Assessment

Unclear for the exact proposed `get`/`set`/`delete` interface. The external-backend direction is reasonable, but the maintainable solution must include metadata, transcript ordering, distributed consistency, pruning, migration, credentials, and SDK/plugin ownership rather than only a simple storage wrapper.
