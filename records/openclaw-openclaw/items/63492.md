## Summary

Keep open. Current main still lacks an external session-storage backend selector or adapter API: session config is path-only, runtime metadata is read/written through local `sessions.json`, transcripts remain local JSONL files, and the closed adapter PR never landed.

## What This Changes

Request pluggable external session storage backends such as Redis or MongoDB so multiple OpenClaw instances can share session state without sticky routing.

## Best Possible Solution

Define a versioned gateway-owned session persistence contract that covers metadata and transcripts, keep filesystem storage as the compatibility default, prove one multi-instance-safe backend, and update config/schema/docs, migration/doctor, and plugin SDK boundaries together.

## Reproduction Assessment

Not applicable: this is a feature and architecture request, not a reproducible bug. Source and docs inspection gives high confidence that current main still lacks an external session-storage backend selector or adapter API.

## Solution Assessment

Unclear for the exact proposed `get`/`set`/`delete` interface. The external-backend direction is reasonable, but the maintainable solution must include metadata, transcript ordering, distributed consistency, pruning, migration, credentials, and SDK/plugin ownership rather than only a simple storage wrapper.
