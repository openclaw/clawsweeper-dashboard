## Summary

Keep open. Current main and v2026.6.8 still use path-based local session metadata plus JSONL transcript files, and the active SQLite/accessor migration is a prerequisite/partial overlap rather than the requested external multi-instance session backend.

## What This Changes

The issue requests a pluggable external session storage abstraction, such as Redis or MongoDB, so multiple OpenClaw instances can share session state without sticky routing.

## Best Possible Solution

Keep this open as the external shared session-storage design item, coordinate it with https://github.com/openclaw/openclaw/issues/88838, and have maintainers decide whether the backend belongs in core, a plugin/API seam, or deployment guidance.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source and release inspection show current main and v2026.6.8 still use local path-based session metadata and JSONL transcript files, without an external backend selector.

## Solution Assessment

No for the literal Redis/Mongo key-value storage factory as written. The maintainable path is a maintainer-approved backend boundary that covers session metadata, transcripts, ordering, pruning, migrations, credentials, plugin/API ownership, and multi-instance consistency.
