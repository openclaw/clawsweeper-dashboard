## Summary

Keep open. Current main and v2026.6.8 still expose only path-based `session.store`, local `sessions.json` metadata, and JSONL transcript files; the active SQLite/accessor work is prerequisite overlap, not the requested external multi-instance session backend.

## What This Changes

The issue requests a pluggable external session-storage backend for stateless, multi-instance OpenClaw deployments.

## Best Possible Solution

Keep this as the canonical external shared session-storage design item, coordinate it with the SQLite/accessor migration, and require a maintainer-approved storage/API boundary before implementation.

## Reproduction Assessment

No as a strict bug reproduction; this is a feature request. Source and release inspection show current main and v2026.6.8 still use local path-based session metadata and JSONL transcript files without an external backend selector.

## Solution Assessment

No for the literal Redis/Mongo key-value storage factory as written. The maintainable path is a maintainer-approved storage boundary that covers session metadata, transcripts, transaction semantics, pruning, migrations, credentials, and multi-instance consistency.
