- `impact:session-state`: The request is about sharing session state and conversation context across multiple OpenClaw instances.
- `impact:data-loss`: The issue describes sessions becoming unavailable or lost when traffic reaches another instance or an ephemeral instance crashes.

## Summary

Keep open. Current main and the latest release still use a path-only filesystem session store with local JSON metadata and JSONL transcripts, while related issues for PostgreSQL and generic pluggable stores were closed in favor of this issue as the canonical tracker; the later horizontal-scaling issue is broader architecture context, not a replacement for this session-storage request.

## What This Changes

Request pluggable external session storage, such as Redis or MongoDB, so multiple OpenClaw instances can share session state without sticky routing.

## Best Possible Solution

Define a versioned gateway-owned session persistence contract that covers metadata and transcripts, keep filesystem storage as the compatibility default, prove one multi-instance-safe backend, and update config/schema/docs, migration or doctor behavior, and plugin SDK boundaries together.

## Reproduction Assessment

Not applicable: this is a feature and architecture request, not a reproducible bug. Source and release inspection gives high confidence that current main and the latest release still lack an external session-storage backend selector or adapter API.

## Solution Assessment

Unclear for the exact proposed `get`/`set`/`delete` storage wrapper. The external-backend direction is reasonable, but the maintainable solution must include metadata, transcript ordering, distributed consistency, pruning, migration, credentials, and SDK or plugin ownership.
