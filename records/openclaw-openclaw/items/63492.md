## Label Justifications

- `P2`: External shared session storage is a significant production scalability improvement, but it is a product/architecture feature rather than an urgent confirmed regression.
- `impact:session-state`: The request is about preserving session metadata and conversation continuity across multiple OpenClaw instances.
- `impact:data-loss`: The reported failure mode is losing local session context when requests hit another instance or an instance-local store disappears.

## Summary

Keep open: current main and v2026.6.10 still lack an external shared session backend; the active SQLite/accessor and broader database-abstraction work are important prerequisites and partial overlaps, not replacements for this session-specific request.

## What This Changes

The issue requests a pluggable external session storage backend so multiple OpenClaw instances can share session state without load-balancer stickiness.

## Best Possible Solution

Keep this open as the session-specific external storage design tracker and coordinate it with the SQLite/accessor migration plus the broader database abstraction discussion before implementing any new config surface.

## Reproduction Assessment

Yes, from source inspection rather than live multi-instance deployment. Current main still resolves a string `session.store` to local `sessions.json` and reads transcript JSONL files, so there is no external shared backend path to exercise.

## Solution Assessment

No, not as the literal Redis/Mongo key-value factory described in the issue. The best fix is a maintainer-approved storage boundary that covers session metadata, transcripts, migrations, pruning, credentials, and consistency semantics, likely after the SQLite/accessor seam lands.
