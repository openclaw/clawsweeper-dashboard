## Label Justifications

- `P2`: The request targets meaningful production scalability and reliability impact, but it is a broad feature/design item rather than an urgent confirmed regression.
- `impact:session-state`: The reported failure mode is losing or mis-associating session context when requests land on different OpenClaw instances.
- `impact:data-loss`: The issue describes local session state disappearing from the user's perspective when routing reaches an instance without the original local store or when an instance-local store is lost.

## Summary

Keep open: current main and v2026.6.10 still expose session storage as a local path-backed store, while related SQLite/accessor and database-abstraction work is prerequisite or broader architecture work rather than a shared external session backend.

## What This Changes

The issue requests a configurable external session storage backend, such as Redis or MongoDB, so multiple OpenClaw instances can share session state without sticky routing.

## Best Possible Solution

Keep this issue open as the canonical session-specific external storage tracker, then design the backend boundary after the session accessor/SQLite migration and broader database-abstraction direction settle.

## Reproduction Assessment

Yes, source confirmation is high-confidence: current main only accepts a string session.store and resolves it to local sessions.json plus local transcript files. This is not a bug reproduction, but the missing external shared backend is directly visible in code and docs.

## Solution Assessment

No, not as the literal Redis/Mongo key-value factory described. The maintainable path needs a maintainer-approved storage boundary covering session metadata, transcripts, migrations, pruning, credentials, consistency, and upgrade behavior.
