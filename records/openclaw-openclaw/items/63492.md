## Summary

Keep open. Current main and v2026.6.6 still use path-only local session storage and local JSONL transcripts, while the active SQLite/accessor migration is a related prerequisite path rather than a full replacement for this issue's external multi-instance storage request.

## What This Changes

Request external/pluggable session storage so multiple OpenClaw instances can share session state without local filesystem affinity.

## Best Possible Solution

Keep this open as the canonical external session-storage product/design item, and coordinate it with the active SQLite/accessor migration before choosing whether external backends belong in core, a plugin API, or documented deployment guidance.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature and architecture request. Source and release inspection gives high confidence that current main and v2026.6.6 still lack an external session-storage backend selector or adapter API.

## Solution Assessment

Unclear for the exact proposed `get`/`set`/`delete` wrapper. The external-backend direction is valid input, but the maintainable solution must cover metadata, transcripts, ordering, pruning, migrations, credentials, plugin/API ownership, and multi-instance consistency.
