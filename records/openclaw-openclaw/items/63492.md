## Summary

Keep open: current main and v2026.6.10 still lack a configurable external shared session backend, and the open SQLite/accessor work is prerequisite storage migration work rather than a Redis/Mongo-style stateless multi-instance solution.

## What This Changes

Request a pluggable external session storage backend so multiple OpenClaw instances can share session metadata and transcripts without sticky routing.

## Best Possible Solution

Keep this as the canonical session-specific external storage design tracker, then decide the external backend boundary after the session accessor/SQLite migration and broader storage-backend direction settle.

## Reproduction Assessment

Yes, source-reproducible: current main only accepts a string session.store and resolves it to local sessions.json plus local transcript files, so there is no external shared backend path to configure or exercise.

## Solution Assessment

No, not as the literal Redis/Mongo key-value factory described. The maintainable path needs a maintainer-approved storage boundary covering session metadata, transcripts, migrations, pruning, credentials, and consistency semantics.
