## Summary

Keep open. Current main still has the same broad Onboard persistence conflict: provider auth writes/update config, but generated `models.json` merge mode intentionally preserves existing non-empty provider `apiKey` and same-surface `baseUrl`, so a same-provider Onboard edit can still be masked by older generated state.

## What This Changes

The issue asks for the Onboard UI to persist same-provider credential and endpoint edits instead of silently leaving older generated provider values in place.

## Best Possible Solution

Add an explicit user-initiated provider update path that updates source config and generated per-agent state consistently while preserving normal merge-mode and SecretRef invariants.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: same provider, merge mode, and existing generated provider state with a non-empty `apiKey` or same-surface `baseUrl`; the planner reaches the preservation helper and current tests assert the old values survive.

## Solution Assessment

Unclear for a concrete patch because none is proposed here. The maintainable direction is an explicit deliberate-update path for Onboard/configure, not a blanket reversal of documented merge-mode precedence.
