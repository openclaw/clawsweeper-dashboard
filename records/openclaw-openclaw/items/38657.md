## Summary

Keep open. Current main and v2026.5.22 still route Onboard provider saves into merge-mode generated provider state where existing baseUrl and non-SecretRef apiKey values can win, while the related landed work only fixed a narrower explicit-config baseUrl precedence path.

## What This Changes

Onboard provider edits should either persist updates to existing provider configuration across canonical/generated auth and model state, or clearly tell users how to update an existing provider.

## Best Possible Solution

Define explicit Onboard update semantics that persist deliberate provider edits to canonical config and targeted generated provider state while preserving normal merge-mode regeneration and SecretRef-safe marker handling.

## Reproduction Assessment

Yes, source-reproducible. Onboard provider saves still flow through merge-mode generated-state reconciliation, and current source/tests show existing generated provider values can preserve or override incoming config values for the fields this issue reports.

## Solution Assessment

Unclear because no patch is proposed. The narrowest maintainable fix is an explicit user-initiated Onboard update path or intent marker, not a blanket reversal of merge-mode precedence.
