## Summary

Keep open. Current main still preserves existing generated provider credentials/endpoints in merge mode, and the shared Onboard path does not carry explicit same-provider overwrite intent, so the broader persistence bug remains unresolved.

## What This Changes

Onboard should let users update an existing provider API key or base URL and persist the change across source config, generated models state, and auth profiles.

## Best Possible Solution

Add an explicit deliberate provider-update path from Onboard/configure that updates canonical config and generated per-agent state consistently while preserving merge-mode and SecretRef invariants.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: same-provider Onboard save with merge mode and existing non-empty generated `apiKey` or same-surface `baseUrl` reaches `mergeWithExistingProviderSecrets`, and current tests assert those existing values survive.

## Solution Assessment

Unclear because no patch is proposed here. The maintainable direction is an explicit user-initiated update path, not a blanket reversal of documented merge-mode precedence.
