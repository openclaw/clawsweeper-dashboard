## Summary

Keep open. Current main still preserves existing generated provider `apiKey` and `baseUrl` in merge mode, and docs/tests now make that precedence intentional, so the broader Onboard edit/update contract remains unresolved rather than implemented.

## What This Changes

Onboard should update or clearly reject edits to an existing model provider API key or base URL instead of showing success while generated config keeps old values.

## Best Possible Solution

Define an explicit user-initiated provider update path for Onboard/configure that updates source config and generated per-agent state consistently while preserving documented merge-mode and SecretRef invariants.

## Reproduction Assessment

Yes, by source inspection: start with merge mode, a same-provider config entry, and an existing generated provider entry with non-empty `apiKey`/`baseUrl`; current planning reaches the preservation helper and tests assert the existing values survive. I did not run the live UI in this read-only review.

## Solution Assessment

Unclear for a concrete patch because none is proposed here. The maintainable direction is an explicit deliberate-update path for Onboard/configure, not a global reversal of merge-mode precedence.
