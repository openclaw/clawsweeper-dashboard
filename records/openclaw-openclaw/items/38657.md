## Summary

Keep open. Current main still encodes merge-mode preservation of existing generated provider credentials/endpoints, while the shared provider onboarding path has no explicit signal for a deliberate same-provider edit, so the broader Onboard persistence behavior remains unresolved.

## What This Changes

Allow the Onboard UI/config flow to update an existing provider API key or base URL, or clearly report that the provider already exists and needs an update path.

## Best Possible Solution

Define an explicit user-initiated provider update path that updates source config and generated per-agent state consistently while preserving merge-mode and SecretRef invariants.

## Reproduction Assessment

Yes, by source inspection: start with merge mode, a same-provider config entry, and an existing generated provider entry with non-empty `apiKey`/`baseUrl`; current planning reaches the preservation helper and tests assert the existing values survive.

## Solution Assessment

Unclear for a concrete patch because none is proposed here. The maintainable direction is an explicit deliberate-update path for Onboard/configure, not a blanket reversal of documented merge-mode precedence.
