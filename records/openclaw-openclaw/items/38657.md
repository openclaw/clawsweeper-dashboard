## Summary

Keep open. Current main still preserves existing generated provider credentials/endpoints in merge mode, and the shared Onboard provider path still has no explicit same-provider overwrite intent; the narrower stale baseUrl reports and PRs do not resolve this broader Onboard persistence issue.

## What This Changes

The issue asks Onboard provider saves to overwrite or clearly reject updates to existing provider apiKey/baseUrl configuration across canonical config and generated state.

## Best Possible Solution

Add an explicit deliberate provider-update path from Onboard/configure that updates canonical config and generated per-agent state consistently while preserving merge-mode and SecretRef invariants.

## Reproduction Assessment

Yes, from source inspection rather than a live UI run. A same-provider Onboard save with merge mode and existing non-empty generated apiKey or same-surface baseUrl reaches `mergeWithExistingProviderSecrets`, and current tests assert those existing values survive.

## Solution Assessment

Unclear because no patch is proposed here. The maintainable direction is explicit user-initiated overwrite intent, not a blanket reversal of the documented merge-mode precedence.
