## Summary

Keep open. Current main still routes Onboard/provider saves through merge-mode generated state, and matching providers can still preserve existing generated `baseUrl` and non-SecretRef `apiKey` values over newly supplied config values; the broader overwrite/persistence path remains a maintainer-scoped behavior decision.

## What This Changes

The issue asks for Onboard edits to existing provider `apiKey`/`baseUrl` settings to persist across config and generated provider state instead of silently retaining stale values.

## Best Possible Solution

Define explicit user-initiated Onboard update semantics that update canonical config and targeted generated provider state while preserving normal merge-mode regeneration and SecretRef marker safety.

## Reproduction Assessment

Yes. This is source-reproducible: Onboard writes through the provider config helpers into default merge mode, and current merge tests assert that existing generated `apiKey`/`baseUrl` values can be preserved over incoming config.

## Solution Assessment

Unclear because no patch is proposed. The maintainable direction is an explicit Onboard update path or intent marker, not a global reversal of merge-mode precedence.
