## Summary

Keep this open. Current main still preserves existing generated provider credentials/endpoints in merge mode, and the Onboard/provider setup helper still has no separate signal for a deliberate same-provider update, so the broader persistence contract described in the discussion remains unresolved.

## What This Changes

Onboard/configure should persist deliberate edits to an existing provider configuration, including API key and base URL changes, across source config, generated per-agent models state, and auth-profile state.

## Best Possible Solution

Define an explicit provider-update contract for user-initiated Onboard/configure edits. It should update the source config and generated state consistently, distinguish deliberate edits from ordinary merge-mode regeneration, preserve SecretRef markers, and cover Moonshot `.ai` to `.cn` base URL updates plus plaintext and SecretRef key replacement cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start with merge mode, a same-provider config entry, and an existing generated `models.json` entry with non-empty `apiKey`/`baseUrl`; current planning reaches `mergeWithExistingProviderSecrets`, and tests assert those existing values are preserved. I did not run the live UI because this review was read-only.

## Solution Assessment

Unclear for a concrete patch because none is proposed here. The narrow maintainable direction is not to flip merge precedence globally, but to add an explicit deliberate-update path for Onboard/configure while keeping documented merge-mode preservation intact.
