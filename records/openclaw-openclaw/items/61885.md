## Summary

Keep open. Current main matches the report because the Control UI intentionally keeps the gateway token in sessionStorage, not localStorage, and changing that default is a security/product decision rather than conservative cleanup.

## What This Changes

The issue requests persisting the Control UI gateway token in localStorage so reopening the browser does not require re-entering it.

## Best Possible Solution

Keep the current session-scoped default unless maintainers approve a documented, opt-in persistent-token mode with explicit logout/clear-token behavior and security guidance.

## Reproduction Assessment

Yes. Source and tests establish that the token is written to sessionStorage and omitted from localStorage, so browser-session teardown loses it by design.

## Solution Assessment

No. The suggested helper swap to localStorage is not the best automatic fix because it changes the admin-surface security posture; a safer path is maintainer-approved opt-in persistence or leaving sessionStorage as-is.
