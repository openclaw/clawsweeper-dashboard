## Summary

Keep open. The reported browser-close behavior is real on current main, but current code, tests, and docs define gateway-token storage as session-scoped; changing it to localStorage is a security/product feature request rather than conservative bug cleanup.

## What This Changes

Request that the Control UI persist the gateway token in localStorage so it survives closing and reopening the browser.

## Best Possible Solution

Keep sessionStorage as the default; if maintainers want longer retention, design an explicit remember-token mode with clear logout/clear-token UX and docs.

## Reproduction Assessment

Yes. Source and tests on current main show the token is written through sessionStorage and deliberately omitted from localStorage, so browser-session teardown loses it by design.

## Solution Assessment

No. The suggested direct helper swap to localStorage is not the best conservative fix because Control UI is an admin surface; persistent retention needs an opt-in product/security design.
