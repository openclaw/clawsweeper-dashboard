## Summary

Keep this issue open. The reported browser-close token loss matches current main, but the current code, tests, docs, and recent history intentionally keep the Control UI gateway token out of localStorage, so the requested persistence change needs maintainer security/product judgment rather than conservative cleanup.

## What This Changes

The issue requests persisting the Control UI gateway token in localStorage so reopening the browser does not require re-entering the token.

## Best Possible Solution

Keep the current session-scoped behavior unless maintainers explicitly approve a documented, opt-in persistent-token mode with clear logout/clear-token handling and matching security guidance.

## Reproduction Assessment

Yes. Current main writes the gateway token to sessionStorage and keeps localStorage token-free, so browser-session teardown clears the token by design; this is covered by storage.node.test.ts.

## Solution Assessment

No. Swapping the helper to localStorage is not the best automatic fix because it conflicts with the documented admin-surface security posture; a safer path would be a maintainer-approved opt-in persistence setting or leaving sessionStorage as-is.
