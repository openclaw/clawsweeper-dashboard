## Summary

Keep open: this is a focused QMD memory fix with sufficient real terminal proof and no line-level correctness finding, but it changes upgrade behavior for existing configs that set a large `memory.qmd.limits.timeoutMs`; maintainers should decide whether `qmd status` probes should share the search timeout or keep a separately bounded health-check timeout.

## What This Changes

The PR changes `QmdMemoryManager.probeVectorAvailability()` to pass `memory.qmd.limits.timeoutMs` directly to `qmd status` and adds focused regression tests for that timeout plus checkpoint-named session transcript export.

## Best Possible Solution

Land the focused fix after maintainers confirm that the QMD search timeout is also the intended status-probe budget, or add a dedicated bounded status-probe timeout while preserving the default fast health-check behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path even though I did not run it locally: current main caps `qmd status` at 5000 ms, `memory status --deep` calls that probe, and the PR supplies terminal proof with a delayed real QMD status command.

## Solution Assessment

Unclear as a product choice: removing the hidden cap fixes false vector-health failures, but maintainers still need to decide whether the search timeout should also be the status-probe timeout or whether a separately bounded status timeout is safer.
