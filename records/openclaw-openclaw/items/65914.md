## Summary

Keep open. The PR targets a real, source-reproducible QMD timeout bug and the production change is narrow, but the external PR still has only test/check validation rather than after-fix real behavior proof, and the timeout regression still uses a value below the old 5000 ms cap.

## What This Changes

The PR changes memory-core's QMD status probe to pass `memory.qmd.limits.timeoutMs`, adds QMD timeout/session transcript regression coverage, and adds a changelog entry.

## Best Possible Solution

Land the focused QMD timeout fix after contributor real behavior proof and strengthen the regression to use a timeout above 5000 ms; keep the checkpoint status-count mismatch tracked in https://github.com/openclaw/openclaw/issues/69759.

## Reproduction Assessment

Yes. Source inspection shows current main caps `qmd status` at 5000 ms even though docs allow increasing `memory.qmd.limits.timeoutMs`; a hanging status probe with a configured timeout above 5000 ms would still stop at the cap.

## Solution Assessment

Yes for the production code. Passing the configured timeout directly is the narrow maintainable fix, but merge readiness still needs real behavior proof and a regression that fails under the old cap.
