## Summary

Keep open: the fix is narrow and has useful proof, but merging still needs a maintainer decision because it changes an existing QMD search-timeout setting into the vector status-probe timeout too.

## What This Changes

The branch removes the hard 5s cap from QMD vector status probes so they use `memory.qmd.limits.timeoutMs`, and adds QMD manager regression tests for status timeouts and checkpoint-like session transcript export.

## Best Possible Solution

Settle the QMD status-probe timeout contract before merge: either explicitly accept and document that the search timeout also bounds status probes, or keep health checks separately bounded with a dedicated status-probe timeout path and tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main clamps `qmd status` at 5s while QMD config can resolve a timeout above 5s, and the PR adds a focused fake-timer regression for that path. I did not run the command locally because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout contract. Reusing the existing setting fixes the false timeout report, but the safer maintainable solution is to make the status-probe semantics explicit in docs/help or introduce a separate status timeout instead of silently expanding the search timeout.
