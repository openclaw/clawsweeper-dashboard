## Summary

Keep open: current main still contains the timeout cap that contradicts documented QMD timeout tuning, and the PR's production change is narrow. The latest branch also keeps checkpoint filtering centralized, but one regression test still uses a timeout below the old cap, so it does not prove the timeout bug fix.

## What This Changes

The PR removes the 5000 ms cap from QMD vector status probes, adds QMD manager regression coverage for configured timeouts and checkpoint-named transcript exports, and adds a changelog entry.

## Best Possible Solution

Land the focused QMD status-timeout fix after strengthening the timeout regression test to prove behavior above the old 5000 ms cap, while keeping checkpoint snapshot filtering in the shared session artifact helpers and leaving the separate memory status file-count inconsistency tracked by #69759.

## Reproduction Assessment

Yes. On current main, configure QMD vector search with `memory.qmd.limits.timeoutMs` above 5000 ms and make `qmd status` hang; the status probe still times out at the hard cap, and the same path can be reproduced with the existing mocked child-process test harness.

## Solution Assessment

Yes, with a test adjustment. Passing the configured timeout directly is the narrowest production fix, and centralized session artifact filtering is the right boundary; the timeout regression test should assert a value above 5000 ms without waiting in real time.
