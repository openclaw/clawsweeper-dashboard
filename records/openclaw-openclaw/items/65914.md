## Summary

Keep open: the production fix is narrow and addresses a source-reproducible QMD timeout bug, but this external PR still lacks after-fix real behavior proof and its timeout regression should distinguish the old 5000 ms cap.

## What This Changes

This PR changes memory-core's QMD status probe to use `memory.qmd.limits.timeoutMs`, adds QMD timeout/session transcript regression tests, and adds a changelog entry.

## Best Possible Solution

Land the focused QMD timeout fix after contributor real behavior proof and a stronger regression that uses a timeout above 5000 ms; keep the checkpoint status-count mismatch tracked in https://github.com/openclaw/openclaw/issues/69759.

## Reproduction Assessment

Yes. Source inspection shows current main caps `qmd status` at 5000 ms even though docs allow increasing `memory.qmd.limits.timeoutMs`; a hanging status probe with a configured timeout above 5000 ms would still stop at the cap.

## Solution Assessment

Yes for the production code. Passing the configured timeout directly is the narrow maintainable fix, but the regression should use a value above the old cap and the external PR still needs real behavior proof before merge.
