## Summary

Keep open. Current main still caps QMD vector status probes at 5000 ms, and the PR’s production fix is narrow, but this external PR still has only mock/CI validation and its timeout regression uses a value below the old cap.

## What This Changes

The PR changes memory-core QMD vector status probes to pass `memory.qmd.limits.timeoutMs` directly, adds QMD timeout/checkpoint transcript regression tests, and updates the changelog.

## Best Possible Solution

Land the focused QMD timeout fix after strengthening the regression to use a configured timeout above 5000 ms with mocked time, keep checkpoint filtering in the shared session artifact helpers, and leave the CLI status-count mismatch to https://github.com/openclaw/openclaw/issues/69759.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable QMD vector mode, set `memory.qmd.limits.timeoutMs` above 5000 ms, and make `qmd status` hang; current main still passes the 5000 ms cap to the subprocess.

## Solution Assessment

Yes for the production code. Passing the configured timeout directly is the narrow maintainable fix, but the test should assert a value above the old cap and the external PR still needs real behavior proof before merge.
