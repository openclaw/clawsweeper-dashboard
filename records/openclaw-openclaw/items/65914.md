## What This Changes

The branch changes memory-core's QMD vector status probe to use the configured timeout, adds focused QMD timeout/session transcript tests, and adds a changelog entry.

## Best Possible Solution

Land the focused QMD timeout fix after contributor real behavior proof and a stronger regression that uses a timeout above 5000 ms; keep the checkpoint status-count mismatch with https://github.com/openclaw/openclaw/issues/69759.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable QMD vector mode, set `memory.qmd.limits.timeoutMs` above 5000 ms, and make `qmd status` hang; current main still caps that subprocess at 5000 ms.

## Solution Assessment

Yes for the production code. Passing the configured timeout directly is the narrow maintainable fix, but the regression should use a value above the old cap and the external PR still needs real behavior proof before merge.
