## Summary

Keep open: current main still caps the QMD vector status probe at 5000 ms, and this PR contains a focused fix with regression coverage and real terminal proof. The latest maintainer comment raises the remaining timeout-policy question, and the branch is currently conflicting, so this should stay in maintainer review rather than be cleaned up.

## What This Changes

The branch removes the hard 5 second cap from the QMD vector status probe, adds memory-core regression tests for configured timeout and checkpoint-like transcript IDs, and updates CHANGELOG.md.

## Best Possible Solution

Keep the focused timeout fix if maintainers accept status probes waiting for the configured QMD timeout; otherwise rework this to a documented status-specific cap while preserving the checkpoint-helper regression coverage.

## Reproduction Assessment

Yes by source inspection and contributor proof. Current main caps `qmd status` at 5000 ms even when `memory.qmd.limits.timeoutMs` is configured higher, and the PR body shows a delayed real QMD status probe completing under an 8000 ms configured timeout after the patch.

## Solution Assessment

Unclear until the maintainer timeout-policy question is answered. Passing the configured timeout directly is the narrow fix if the existing config is meant to govern status probes, while a status-specific cap would be safer if deep status must stay more responsive than other QMD operations.
