## Summary

Keep open. Current main still caps the QMD vector status probe at 5000 ms, and this PR’s production change is the right narrow fix, but the external PR still lacks after-fix real behavior proof and its timeout regression does not fail against the old cap.

## What This Changes

The PR changes memory-core’s QMD vector status probe to pass `memory.qmd.limits.timeoutMs`, adds focused QMD timeout/session transcript coverage, and updates `CHANGELOG.md`.

## Best Possible Solution

Land the focused QMD timeout fix after contributor real behavior proof, strengthen the regression to use a timeout above 5000 ms, and keep the checkpoint status-count mismatch tracked in https://github.com/openclaw/openclaw/issues/69759.

## Reproduction Assessment

Yes. Source inspection shows current main caps `qmd status` at 5000 ms even though the documented `memory.qmd.limits.timeoutMs` setting allows larger values; a hanging status probe with a configured timeout above 5000 ms would still stop at the cap.

## Solution Assessment

Yes for the production code. Passing the configured timeout directly is the narrow maintainable fix, but the regression should use a value above 5000 ms so it fails against current main.
