## Summary

Keep open: current main still has the QMD status-probe timeout cap, and the PR’s production fix is narrow, but the timeout regression still does not prove behavior above the old 5000 ms cap.

## What This Changes

The PR removes the 5000 ms cap from QMD vector status probes, adds QMD timeout/checkpoint transcript regression coverage, and adds a changelog entry.

## Best Possible Solution

Land the focused QMD timeout fix after strengthening the timeout regression to assert a configured value above 5000 ms, while keeping checkpoint filtering in the shared session artifact helpers and leaving the separate status-count issue to #69759.

## Reproduction Assessment

Yes. On current main, set QMD vector mode with `memory.qmd.limits.timeoutMs` above 5000 ms and make `qmd status` hang; the code path still times out at the hard cap, and the existing mocked child-process harness can reproduce it with fake timers.

## Solution Assessment

Yes, with a test repair. Passing the configured timeout directly is the narrowest production fix, and the checkpoint export boundary is now correct; the regression test should use a value above 5000 ms so it fails against the old implementation.
