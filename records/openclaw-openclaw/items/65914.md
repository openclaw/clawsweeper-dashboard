## Summary

Current main still caps the QMD vector status probe at 5 seconds, while this PR makes that path honor the existing QMD timeout setting and adds focused regression coverage. The code change looks correct, but a maintainer has explicitly asked whether the hidden 5-second health-check bound was intentional, so this should stay open for that timeout-contract decision before merge.

## What This Changes

This PR changes QMD vector status probes to use the configured `memory.qmd.limits.timeoutMs` and adds focused regression tests for the timeout path and checkpoint-like session IDs.

## Best Possible Solution

Land this after maintainers confirm the existing QMD timeout should govern vector status probes, or add an explicit status-probe timeout/default that preserves fast health checks while allowing opt-in longer waits.

## Reproduction Assessment

Yes. Current main visibly caps `qmd status` at `Math.min(timeoutMs, 5_000)`, and the PR adds a focused fake-timer regression plus real terminal proof for the delayed-status path.

## Solution Assessment

Unclear pending maintainer policy. Reusing the existing timeout is the smallest code fix if status probes should honor the same QMD timeout, but a separate explicit status-probe timeout is safer if the old fast health-check cap was intentional.
