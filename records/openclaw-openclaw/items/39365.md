## Summary

Keep open: the docs gap is real and current main still lacks the recipes page/nav entry, but this PR is not merge-ready because the webhook recipe and anti-pattern table disagree with current trust-boundary, agent selector, and channel dispatch behavior.

## What This Changes

This PR adds a `docs/automation/system-events.md` recipes page, navigation entry, and CLI system-doc links for programmatic agent wake via `openclaw system event`.

## Best Possible Solution

Revise the page to keep trusted local CLI wake recipes, route external webhooks to authenticated `/hooks/wake` or explicit validation guidance, and align the anti-pattern table with current agent selector and channel dispatch behavior.

## Reproduction Assessment

Yes. The missing docs surface and the review findings are source-reproducible by comparing the PR-added docs lines with current `system event`, `/hooks/wake`, `agent`, Slack, and Discord code paths.

## Solution Assessment

No, not as-is. A dedicated recipes page is the right direction, but the narrow maintainable solution must document trusted CLI wakes separately from authenticated or validated external webhook input.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route webhook payloads through untrusted hooks:** `docs/automation/system-events.md:70-75`
  - body: The webhook recipe forwards external arguments through `openclaw system event`, but that CLI path currently enqueues trusted system events. Copy-pasted webhook payloads should use authenticated `POST /hooks/wake` or add explicit validation and trust-boundary guidance.
  - confidence: 0.88
- **[P2] Describe `openclaw agent` failure accurately:** `docs/automation/system-events.md:122`
  - body: This row says `openclaw agent --message` hangs in scripts, but current docs require a selector such as `--to`, `--session-id`, or `--agent`. The docs should describe the selector requirement so automation users debug the real failure mode.
  - confidence: 0.9
- **[P2] Avoid saying channel posts bypass the Gateway:** `docs/automation/system-events.md:124`
  - body: Inbound Slack and Discord messages dispatch through Gateway-integrated handlers, so channel posts do not bypass Gateway. This row should instead explain sender filtering, bot self-message suppression, routing, or channel policy limits.
  - confidence: 0.82
