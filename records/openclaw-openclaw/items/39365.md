## Summary

Keep this PR open: the docs gap is real, but the current patch still has source-backed blockers around webhook trust boundaries and two misleading anti-pattern descriptions.

## What This Changes

This PR adds a new `docs/automation/system-events.md` recipes page, links it from `docs/cli/system.md`, and adds it to the Automation docs navigation.

## Best Possible Solution

Revise the page to separate trusted local CLI wakes from external webhook ingress, use `/hooks/wake` or explicit validation for webhook recipes, and correct the agent/channel anti-pattern rows.

## Reproduction Assessment

Yes. The docs defects are source-reproducible by comparing the PR-added lines with current `system event`, `/hooks/wake`, `agent`, Slack, and Discord implementations on main.

## Solution Assessment

No, not as-is. A recipes page is the right direction, but the maintainable version should document trusted CLI wakeups separately from authenticated or validated external webhook input.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Route webhook input through untrusted hooks:** `docs/automation/system-events.md:73-75`
  - body: The webhook recipe forwards external arguments through `openclaw system event`, but that CLI path enqueues trusted system events. Copy-pasted webhook payloads should use authenticated `/hooks/wake` or clearly validate and constrain input before using the trusted local CLI path.
  - confidence: 0.88
- **[P2] Describe agent CLI targeting failures accurately:** `docs/automation/system-events.md:122`
  - body: This row says `openclaw agent --message` hangs in scripts, but current main fails fast unless a selector such as `--to`, `--session-id`, or `--agent` is provided, and valid runs are bounded by the timeout path. The docs should point users at the real failure mode.
  - confidence: 0.9
- **[P2] Correct channel-trigger guidance:** `docs/automation/system-events.md:124`
  - body: Inbound Slack and Discord messages dispatch through Gateway-integrated handlers, so direct channel posts do not fail because they bypass the Gateway. This row should instead explain routing, sender filtering, bot self-message suppression, or channel policy limits.
  - confidence: 0.82
