## Summary

Keep open: the documentation gap is still real, but the PR should not merge as-is because the added recipes misstate current behavior and include an unsafe webhook trust-boundary pattern.

## What This Changes

The PR adds a new `docs/automation/system-events.md` recipes page, links it from `docs/cli/system.md`, and adds it to docs navigation.

## Best Possible Solution

Revise the docs-only page so trusted local CLI wake recipes are separated from external webhook guidance, webhooks use authenticated `/hooks/wake` or explicit validation boundaries, and the anti-pattern table matches current agent/channel behavior.

## Reproduction Assessment

Yes. The blockers are source-reproducible by comparing the PR text with current `system event`, `/hooks/wake`, `agent`, Slack, and Discord code paths.

## Solution Assessment

No, not as-is. A dedicated recipes page is a reasonable docs direction, but the safe version needs to route external webhook input through the untrusted hook path and correct the agent/channel explanations.
