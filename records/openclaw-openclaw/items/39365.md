Keep open. The linked discussion confirms a real documentation need, but this PR still has source-backed P2 docs/security issues, missing real behavior proof, and a failing docs check; current main has lower-level wake docs but not this dedicated recipes page.

## What This Changes

The PR adds a `docs/automation/system-events.md` recipe page, links it from `docs/cli/system.md`, and adds it to the Automation navigation group.

## Best Possible Solution

Revise the docs page so trusted local CLI wake recipes are separated from external webhook guidance, webhooks use authenticated `/hooks/wake` or an explicit validation boundary, and the anti-pattern table matches current agent and channel behavior.

## Reproduction Assessment

Yes. The blockers are source-reproducible by comparing the PR text with current `system event`, `/hooks/wake`, `agent`, Slack, and Discord code paths.

## Solution Assessment

No, not as-is. A dedicated recipes page is a reasonable docs direction, but the safe version needs to route external webhook input through the untrusted hook path and correct the agent/channel explanations.
