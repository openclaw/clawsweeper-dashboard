## Summary

Keep open. Current main still lacks the dedicated system-events recipes page, so this PR is not obsolete, but the proposed docs are not merge-ready because the new recipes misstate current `openclaw agent` and channel behavior and blur the trusted CLI wake path with untrusted webhook input.

## What This Changes

This PR adds `docs/automation/system-events.md`, adds it to the Automation navigation, and cross-links it from `docs/cli/system.md`.

## Best Possible Solution

Keep the PR open and revise the docs so the page documents the supported CLI wake path, authenticated `POST /hooks/wake`, and `openclaw agent` separately, with explicit validation and prompt-trust guidance for external payloads and corrected anti-pattern wording.

## Reproduction Assessment

Yes. The missing docs surface is reproducible from `pnpm docs:list`, `find`, and current nav/CLI docs, and the review findings are reproducible by comparing the PR file lines with current `agent`, hooks, system-event, Slack, and Discord source paths.

## Solution Assessment

No, not as-is. The page and cross-links are the right direction, but the narrow maintainable solution is to refresh the examples against current wake, webhook, agent, and channel semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route webhook payloads through untrusted hooks:** `docs/automation/system-events.md:73-75`
  - body: The webhook recipe forwards external arguments through `openclaw system event`, but the CLI wake path enqueues a trusted system event while built-in `/hooks/wake` marks webhook text untrusted. Copy-pasted webhook payloads would enter heartbeat prompt context as trusted `System:` content, so this recipe should use the authenticated hook endpoint or add clear validation/trust-boundary guidance.
  - confidence: 0.86
- **[P2] Describe `openclaw agent` failure accurately:** `docs/automation/system-events.md:122`
  - body: This row says `openclaw agent --message` hangs in scripts, but current main fails fast without `--to`, `--session-id`, or `--agent`. The docs should describe that selector error so automation users debug the real failure mode.
  - confidence: 0.9
- **[P2] Avoid saying channel posts bypass the Gateway:** `docs/automation/system-events.md:124`
  - body: Inbound Slack and Discord messages dispatch through the Gateway reply pipeline, so channel-posted events are not invalid because they bypass Gateway. This row should instead explain sender filtering, bot self-message suppression, or channel policy limits.
  - confidence: 0.82
