## Summary

Current main and the shipped v2026.5.2 line already address the reported failure: explicit `agents.defaults.imageGenerationModel` now makes `image_generate` register without depending on runtime provider listing, and spawned subagent runs use the same tool factory path before applying the requested allowlist.

## What This Changes

The issue requests that spawned subagent sessions expose `image_generate` when image generation is configured and allowed, matching the main agent and CLI behavior.

## Best Possible Solution

Keep the shipped registration behavior: explicit media-generation model config should make the tool visible without loading runtime provider registries, while provider resolution remains in execution and list actions.

## Reproduction Assessment

No for current main: the reported old failure depended on registration falling through to runtime provider listing, but current source short-circuits explicit image-generation config and has regression coverage for that path. The issue’s reproduction is still credible for the older 2026.4.29 build.

## Solution Assessment

Yes: the shipped fix is the narrow maintainable solution because it preserves provider runtime resolution for execution/listing while making prompt-time tool visibility depend on explicit config and manifest/auth availability.
