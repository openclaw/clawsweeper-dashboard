## Summary

Current main already removes the reported registration failure: explicit image-generation model config makes `image_generate` eligible without eager provider enumeration, and the subagent tool-construction path carries the same config/auth context into the core tool factory.

## What This Changes

The issue requests that `image_generate` be registered for spawned subagent sessions when `agents.defaults.imageGenerationModel` and provider auth are configured.

## Best Possible Solution

Keep the current main implementation, which registers the tool from explicit config and uses manifest/provider availability only when inference is needed.

## Reproduction Assessment

Yes. Source inspection shows the old failure mode would be provider-list enumeration suppressing registration, while current main registers from explicit `agents.defaults.imageGenerationModel` before provider enumeration is needed.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it preserves provider-based inference while avoiding eager provider resolution for explicit image-generation config.
