## Summary

Current main and the shipped v2026.5.2 release already fix the reported registration path: explicit `agents.defaults.imageGenerationModel` now keeps `image_generate` registered without asking the runtime provider registry during tool construction, which is the failure mode described from 2026.4.29.

## What This Changes

The issue asks for spawned subagent sessions to receive `image_generate` when image generation is configured, matching main-agent and CLI behavior.

## Best Possible Solution

Keep the shipped explicit-config and manifest/auth availability path; users on 2026.4.29 should upgrade to v2026.5.2 or newer.

## Reproduction Assessment

No current-main failing reproduction remains. Source inspection shows the reported explicit `agents.defaults.imageGenerationModel` path registers `image_generate` without runtime provider listing, and the shipped regression test covers that behavior.

## Solution Assessment

Yes. The implemented fix is the narrow maintainable solution because registration now uses explicit config and control-plane availability, while runtime provider resolution remains deferred to execution/list actions.
