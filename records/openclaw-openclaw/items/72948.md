## Summary

Keep open. Current main still rejects a single-token `openclaw-gateway` argv before the `allowGatewayBinary` branch can match, so `openclaw gateway stop` can still miss a foreground Gateway and fall through to the misleading service-disabled path; #72224 is the open implementation candidate.

## What This Changes

Request that `openclaw gateway stop` stop foreground or unmanaged Gateway processes instead of reporting service-disabled while leaving them running.

## Best Possible Solution

Land #72224 after rebase and maintainer review, or land a narrower equivalent that keeps verified process signaling while recognizing a bare `openclaw-gateway` listener and adds stop-path regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. The reporter provided a concrete foreground-Gateway repro and retest, and current main still has the same source-level path where a bare `openclaw-gateway` argv yields no verified PID and falls through to `Gateway service disabled.`

## Solution Assessment

Yes. A narrow verifier fix is the best direction because it preserves the safety check before SIGTERM; a message-only change would leave container and sandbox users manually killing a valid Gateway process.
