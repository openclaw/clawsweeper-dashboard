## Summary

Keep open. Current main still has the reported foreground-gateway stop failure path: `gateway stop` has an unmanaged-process fallback, but it only signals PIDs that pass listener discovery and `isGatewayArgv`; a bare `openclaw-gateway` argv still fails the verifier before the opt-in gateway-binary branch can match. The related #72224 PR is still the right implementation candidate rather than a reason to close this issue before merge.

## What This Changes

`openclaw gateway stop` should stop a directly foreground-launched Gateway in no-service-manager environments instead of reporting only that the service registration is disabled.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that recognizes a bare `openclaw-gateway` listener under the existing opt-in verifier, adds Linux listener discovery that works without `lsof`, and covers foreground `gateway stop` with regression tests.

## Reproduction Assessment

Yes. The reporter supplied a concrete foreground-run reproduction, and current source shows the same path can still fall through: bare `openclaw-gateway` argv is rejected, no verified PID is returned, and Linux prints `Gateway service disabled.` when the service is not loaded.

## Solution Assessment

Yes. A narrow verified unmanaged-process fallback is the best direction because it keeps the safety check before signaling while making `gateway stop` work for container and sandbox foreground Gateway runs; a message-only change would leave the operational bug in place.
