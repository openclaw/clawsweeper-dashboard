- `impact:other`: The issue affects operator-visible gateway lifecycle control in containers and sandboxes outside the more specific owned impact labels.

## Summary

Keep open: current main and v2026.6.8 still miss the reported foreground `openclaw-gateway` case when no service manager is loaded, because Unix listener discovery can return no PIDs when `lsof` is unavailable and the argv verifier still rejects a bare `openclaw-gateway` argv. The related implementation PR appears to cover the central bug, but it is still open and conflicting, so this issue remains useful until that PR or a narrow replacement lands.

## What This Changes

The issue requests that `openclaw gateway stop` stop a foreground-launched gateway in service-manager-free environments instead of returning `Gateway service disabled` while the process keeps running.

## Best Possible Solution

Land a current-main-compatible gateway lifecycle/process-discovery fix that accepts bare `openclaw-gateway` argv, does not depend solely on `lsof` in Linux containers, and proves foreground `openclaw gateway stop` releases the listener.

## Reproduction Assessment

Yes. Source inspection plus the reporter's logs show a high-confidence path: without a service manager, missing `lsof` can yield no listener PIDs, and current `isGatewayArgv` rejects the reported bare `openclaw-gateway` argv shape.

## Solution Assessment

Yes. The best fix belongs in gateway process verification and listener discovery because `gateway stop` already has an unmanaged listener fallback; service-manager messaging alone would not stop the foreground process.
