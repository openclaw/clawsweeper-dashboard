## Summary

Keep open: current main and the latest release still reject the reported single-token `openclaw-gateway` argv shape before the unmanaged stop fallback can signal it. The linked fix at https://github.com/openclaw/openclaw/pull/72224 appears to address the central bug, but it is still open and currently conflicting, so this issue should remain open until that fix or a narrow replacement lands.

## What This Changes

The issue reports that `openclaw gateway stop` disables the service-manager registration but leaves a directly launched foreground gateway process running.

## Best Possible Solution

Land a current-main-compatible fix in the existing gateway lifecycle and process-discovery boundary, preferably by refreshing and proving https://github.com/openclaw/openclaw/pull/72224 or extracting the bare argv and no-lsof fallback into a narrower PR.

## Reproduction Assessment

Yes by source inspection and reporter logs, without a fresh live run in this read-only review. Current main still rejects the reported `["openclaw-gateway"]` argv shape, which makes the stop fallback miss the live process and fall through to the not-loaded service message.

## Solution Assessment

Yes, fixing gateway process verification and listener discovery is the narrow maintainable solution because `gateway stop` already has an unmanaged listener fallback. A pid-file-only or docs-only workaround would leave the existing verified-signal path broken.
