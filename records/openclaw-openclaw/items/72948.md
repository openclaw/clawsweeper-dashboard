## Summary

Keep open. Current main still contains the reported foreground-stop failure path: unmanaged stop support exists, but listener discovery can return no candidates when `lsof` is unavailable and the argv verifier still rejects a single-token `openclaw-gateway` process before the opt-in gateway-binary check can match. The related #72224 PR is the active fix candidate, but it is open and not merged.

## What This Changes

The issue asks for `openclaw gateway stop` to stop a foreground-launched gateway process in non-service-manager environments instead of only reporting that the gateway service is disabled.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing opt-in gate, adds Linux listener discovery that works without `lsof`, and covers foreground `gateway stop` with regression tests.

## Reproduction Assessment

Yes. The reporter supplied concrete foreground-gateway steps, and current source inspection shows the same failure path: no verified listener PID causes the service-disabled fallback instead of sending SIGTERM.

## Solution Assessment

Yes. A narrow verified-process fallback is the best maintainable fix because it preserves gateway-only process checks before signaling; a message-only change would leave foreground gateway users manually killing a valid OpenClaw process.
