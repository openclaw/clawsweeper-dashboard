## Summary

Keep open. Current `main` still contains the reported foreground-stop failure path: unmanaged stop exists, but it depends on verified listener PID discovery, which still returns no Unix PIDs when `lsof` cannot spawn and still rejects a bare single-token `openclaw-gateway` argv before the opt-in binary check can match. The related #72224 PR is the right fix direction but is open, so this issue should remain open until a merged fix proves the behavior.

## What This Changes

`openclaw gateway stop` should stop a foreground-launched gateway process in non-service-manager environments instead of only reporting that the Gateway service is disabled.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that accepts verified bare `openclaw-gateway` listener processes under the existing opt-in gate, adds Linux listener discovery that works without `lsof`, and covers foreground `gateway stop` with regression tests.

## Reproduction Assessment

Yes. Start a foreground Gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop` from another shell, and current source shows a bare `openclaw-gateway` listener or missing `lsof` can produce no verified PID, leaving the process alive and falling through to the service-disabled path.

## Solution Assessment

Yes. A narrow verified-process fallback is the best fix because it preserves gateway-only verification before sending SIGTERM; a message-only change would still leave foreground container users manually killing a valid OpenClaw gateway process.
