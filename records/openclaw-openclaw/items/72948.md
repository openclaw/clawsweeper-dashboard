## What This Changes

This issue asks `openclaw gateway stop` to actually stop a foreground/no-service gateway instead of reporting service disablement while leaving the listener running.

## Best Possible Solution

Land a refreshed gateway lifecycle/process-discovery fix that accepts bare `openclaw-gateway` argv, can discover verified Linux listener PIDs without relying only on `lsof`, and proves foreground `openclaw gateway stop` releases the listener.

## Reproduction Assessment

Yes, at source level: without a loaded service, `gateway stop` only signals verified listener PIDs, and current discovery rejects the reported bare `openclaw-gateway` argv shape while the Unix path returns no PIDs when `lsof` is unavailable. I did not run a live no-systemd/no-lsof container flow in this read-only review.

## Solution Assessment

Yes, the best fix belongs in shared gateway process verification and Linux listener discovery because the stop command already has an unmanaged listener fallback. A message-only change would leave the foreground process alive.
