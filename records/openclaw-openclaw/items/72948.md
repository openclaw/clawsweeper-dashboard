## Summary

Keep open. Current main still rejects the reported bare `openclaw-gateway` argv shape before the opt-in gateway-binary check and still depends on `lsof` for Unix listener discovery, so `gateway stop` can fall through to the reported service-disabled path; #72224 is the open implementation candidate.

## What This Changes

Request that `openclaw gateway stop` stop foreground/unmanaged Gateway processes instead of reporting the service as disabled while leaving the listener alive.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that recognizes a bare `openclaw-gateway` listener, avoids a hard `lsof` dependency in Linux containers, and keeps strict PID verification before SIGTERM.

## Reproduction Assessment

Yes. Start a foreground Gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop` from another shell, and current source shows a bare `openclaw-gateway` argv or missing `lsof` can leave no verified PID and fall through to the service-disabled message.

## Solution Assessment

Yes. A verified unmanaged-process fallback is the narrow maintainable fix because it preserves safety before signaling; a message-only change would still leave container and sandbox users killing the valid Gateway process manually.
