## Summary

Keep open. Current main still rejects the bare `openclaw-gateway` argv shape before the unmanaged stop fallback can verify the listener, and linked PR #72224 is still open/dirty rather than merged.

## What This Changes

Request that `openclaw gateway stop` terminate a foreground-launched unmanaged Gateway process instead of reporting service-disabled success while leaving the listener alive.

## Best Possible Solution

Rebase and land #72224, or a narrow equivalent, preserving verified-PID safeguards while accepting bare `openclaw-gateway`, adding Linux listener discovery without a hard `lsof` dependency, and tying the PR to this issue.

## Reproduction Assessment

Yes. The issue and retest comments give a concrete foreground `openclaw gateway run --dev --port 38789` then `openclaw gateway stop` path, and current source explains the failure for `/proc/<pid>/cmdline` equal to `openclaw-gateway`.

## Solution Assessment

Yes. A verified unmanaged-process fallback is the right direction because it avoids killing arbitrary port owners while fixing container/no-systemd foreground Gateway cleanup; a message-only change would leave the operational bug in place.
