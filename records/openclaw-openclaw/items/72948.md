## Summary

Keep open. Current main still has the unmanaged stop fallback, but the verifier can reject a foreground child whose argv is only `openclaw-gateway`, so `gateway stop` can still fall through to the misleading service-disabled path; #72224 is the open implementation candidate but has not landed.

## What This Changes

The issue reports that `openclaw gateway stop` does not terminate a foreground-launched Gateway process when no service manager is registered.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that keeps strict process verification while recognizing a bare `openclaw-gateway` listener, avoids a hard dependency on `lsof` in Linux containers, and adds regression coverage for foreground stop.

## Reproduction Assessment

Yes. Start a foreground Gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop` from another shell, and current source shows a bare `openclaw-gateway` argv or missing `lsof` can leave no verified PID and fall through to `Gateway service disabled.`

## Solution Assessment

Yes. A narrow verified unmanaged-process fallback is the maintainable direction because it preserves the safety check before sending SIGTERM; a message-only change would leave container and sandbox users manually killing a valid OpenClaw Gateway process.
