## Summary

Keep open. Current main still contains the reported foreground-stop failure path: `gateway stop` only falls back to unmanaged SIGTERM when it finds verified listener PIDs, Unix listener discovery still returns no PIDs when `lsof` is unavailable, and `isGatewayArgv()` still rejects a bare single-token `openclaw-gateway` argv before the opt-in binary check can match. The related #72224 PR appears to carry the right fix direction, but it is still open, so this issue should remain open until that or a narrower equivalent lands.

## What This Changes

The issue requests that `openclaw gateway stop` stop foreground-launched Gateway processes in no-service-manager environments instead of reporting `Gateway service disabled` while leaving the listener alive.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that recognizes verified bare `openclaw-gateway` listener processes, discovers Linux listeners without requiring `lsof`, preserves strict process verification before signaling, and adds regression tests for foreground `gateway stop`.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to start a foreground Gateway whose listener argv is a single `openclaw-gateway` token in an environment without `lsof`, then run `openclaw gateway stop`; current source shows verified PID discovery returns empty and the stop command falls through to the service-disabled path.

## Solution Assessment

Yes. A verified-process fallback is the right maintainable fix because it keeps SIGTERM restricted to OpenClaw Gateway processes while making foreground container/sandbox runs manageable; a message-only change would still leave users killing the process manually.
