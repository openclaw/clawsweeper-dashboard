## Summary

Keep open. Current main still contains the unmanaged stop fallback, but the process verifier still rejects a foreground child whose argv is only `openclaw-gateway`, and the Linux listener lookup still returns no PIDs when `lsof` is unavailable; #72224 remains the open implementation candidate for this path.

## What This Changes

The issue asks for `openclaw gateway stop` to stop a directly launched foreground Gateway instead of only reporting service-manager disablement.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that keeps strict process verification while recognizing a bare `openclaw-gateway` listener, avoids depending only on `lsof` in Linux containers, and adds regression coverage for foreground stop.

## Reproduction Assessment

Yes. The provided sandbox reproduction is backed by current source: a bare `openclaw-gateway` argv is rejected by `isGatewayArgv`, and missing `lsof` returns no verified listener PIDs before `gateway stop` falls back to the service-disabled message.

## Solution Assessment

Yes. A narrow verified unmanaged-process fallback is the maintainable direction because it preserves the safety check before sending SIGTERM; a message-only change would leave container and sandbox users manually killing a valid OpenClaw Gateway process.
