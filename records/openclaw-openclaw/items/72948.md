## Summary

Keep open. Current main still routes `gateway stop` through the unmanaged fallback, but the verifier rejects a bare `openclaw-gateway` argv before the opt-in gateway-binary branch can match, so the reported no-op stop path remains possible until #72224 or an equivalent fix lands.

## What This Changes

The issue requests that `openclaw gateway stop` terminate a directly foreground-launched Gateway process instead of only reporting service-manager disabled state.

## Best Possible Solution

Land #72224 after rebase/review, or land a narrower equivalent that recognizes a bare `openclaw-gateway` listener only under the existing verified-process gate, with stop regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The reporter provided a foreground Gateway repro, and current main source shows the single-token `openclaw-gateway` argv shape can still yield no verified PID and fall through to the service-disabled message.

## Solution Assessment

Yes. A narrow verifier fix is the best path because it preserves the safety check before SIGTERM; a message-only change would leave container and sandbox users manually killing a valid OpenClaw Gateway process.
