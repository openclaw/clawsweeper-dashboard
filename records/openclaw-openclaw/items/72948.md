## Summary

Keep open. Current main still has the unmanaged stop fallback, but it rejects the reported single-token `openclaw-gateway` argv shape and still returns no Unix listener PIDs when `lsof` is unavailable, so `gateway stop` can fall through to the Linux service-disabled message without signaling the foreground child. #72224 is an open dirty PR containing the relevant fix direction, so this should stay open until that PR or a narrower equivalent lands.

## What This Changes

The issue requests that `openclaw gateway stop` terminate a directly launched foreground gateway in service-manager-free Linux/container environments, or at least report accurately when only service registration was disabled.

## Best Possible Solution

Rebase, review, and land #72224 or a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing opt-in gate, adds Linux listener discovery that works without `lsof`, covers foreground stop with regression tests, and includes the required changelog entry.

## Reproduction Assessment

Yes. The reporter supplied a concrete foreground `openclaw gateway run --dev --port 38789` reproduction, and current source shows why it still fails when the listener cmdline is `["openclaw-gateway"]` or when `lsof` is unavailable.

## Solution Assessment

Yes for the proposed direction. A verified unmanaged SIGTERM fallback plus robust listener discovery is the narrow maintainable fix; changing only the message would leave foreground container users manually killing a valid OpenClaw gateway process.
