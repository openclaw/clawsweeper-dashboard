## Summary

Keep open. Current `main` still contains the reported foreground-gateway stop failure path: unmanaged stop exists, but verified PID discovery can return no PID when `lsof` is unavailable, and the argv verifier still rejects a single-token `openclaw-gateway` process before the opt-in gateway-binary check can match. The related #72224 PR appears to carry the right fix direction, but it is still open and not merged.

## What This Changes

The issue asks for `openclaw gateway stop` to stop a foreground-launched gateway instead of only reporting that the service-manager registration is disabled.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing opt-in gate, adds Linux listener discovery that works without `lsof`, and covers foreground `gateway stop` with regression tests.

## Reproduction Assessment

Yes. Start a foreground gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop` from another shell, and current source shows both reported failure gates: single-token `openclaw-gateway` does not verify, and missing `lsof` returns no listener PID.

## Solution Assessment

Yes. A narrow verified-process fallback is the best solution because it preserves gateway-only verification before sending SIGTERM; a message-only change would leave container and sandbox users manually killing a valid OpenClaw gateway process.
