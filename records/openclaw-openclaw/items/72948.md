## Summary

Keep open. Current main still requires a standalone `gateway` argv token before accepting the `openclaw-gateway` binary, so the unmanaged stop fallback can still miss the foreground listener and fall through to the misleading service-disabled path; #72224 is the existing implementation candidate.

## What This Changes

Make `openclaw gateway stop` terminate a directly launched Gateway process in service-manager-less environments instead of reporting service disabled while it remains alive.

## Best Possible Solution

Land #72224 after rebase and maintainer review, or land a narrower equivalent that recognizes a bare `openclaw-gateway` listener while preserving verified-process signaling, with stop-path regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The reporter provided concrete foreground-Gateway steps and a retest, and current main still has the same source-level path where a bare `openclaw-gateway` argv yields no verified PID and falls through to `Gateway service disabled.`

## Solution Assessment

Yes. A narrow verifier fix is the best direction because it keeps the safety check before SIGTERM; a message-only change would leave container and sandbox users manually killing a valid Gateway process.
