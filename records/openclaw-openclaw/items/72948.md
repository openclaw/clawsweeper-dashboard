## Summary

Keep open: current main still has a process-verification/listener-discovery path that can miss a foreground `openclaw-gateway` child and fall through to the misleading disabled-service message; #72224 is open and appears to cover the same gap.

## What This Changes

The issue requests that `openclaw gateway stop` stop a directly started Gateway process in service-manager-free/container environments instead of returning the service-disabled no-op path.

## Best Possible Solution

Land #72224 after rebase and maintainer review, or land a narrower equivalent that keeps verified process signaling while recognizing bare `openclaw-gateway`, adds Linux listener discovery without relying only on `lsof`, and covers the stop path with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The reporter provided concrete foreground-Gateway repro and retest details, and current main still has the source-level path where PID discovery/verifier failure returns null and prints `Gateway service disabled.`.

## Solution Assessment

Yes. A narrow verifier/listener-discovery fix is the maintainable direction because it preserves the safety check before SIGTERM; a message-only change would leave container users manually killing a valid Gateway process.
