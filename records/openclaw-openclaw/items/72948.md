## Summary

Keep open. Current main still has an unmanaged stop fallback, but it depends on verified listener discovery that can return no PID when `lsof` is missing and still rejects a single-token `openclaw-gateway` argv before the opt-in gateway-binary branch can match; #72224 is the open implementation candidate, but it has not landed.

## What This Changes

Request that `openclaw gateway stop` terminate a directly launched foreground Gateway in no-service-manager/container environments instead of reporting service-disabled while leaving the process and port alive.

## Best Possible Solution

Land #72224 after rebase/review, or land a narrower equivalent that preserves strict process verification while accepting bare `openclaw-gateway`, adds non-`lsof` Linux listener discovery, and covers foreground stop in tests.

## Reproduction Assessment

Yes. Start a foreground Gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop` from another shell, and current source shows the bare `openclaw-gateway` argv or missing `lsof` paths can leave no verified PID and fall through to the service-disabled message.

## Solution Assessment

Yes. A narrow verified unmanaged-process fallback is the maintainable direction because it keeps the SIGTERM safety gate; a message-only change would still leave container and sandbox users manually killing a valid OpenClaw Gateway process.
