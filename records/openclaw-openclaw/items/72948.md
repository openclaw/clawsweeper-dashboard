## Summary

Keep open. Current main still has the unmanaged stop fallback, but the verifier rejects a bare `openclaw-gateway` argv before the `allowGatewayBinary` branch can match, so `gateway stop` can still fall through to the misleading service-disabled path; #72224 remains the open implementation candidate.

## What This Changes

Make `openclaw gateway stop` terminate a directly launched foreground Gateway instead of only reporting service-manager state.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that keeps strict process verification while recognizing a bare `openclaw-gateway` listener, avoids a hard `lsof` dependency in Linux containers, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Start a foreground Gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop`, and current source shows a bare `openclaw-gateway` argv can leave no verified PID and fall through to `Gateway service disabled.`

## Solution Assessment

Yes. A narrow verified unmanaged-process fallback is the maintainable direction because it preserves the safety check before sending SIGTERM; a message-only change would leave container and sandbox users manually killing a valid OpenClaw Gateway process.
