## Summary

Keep open. Current main still has the same verifier ordering gap: a single-token `openclaw-gateway` argv is rejected before the opt-in gateway-binary check can run, so `openclaw gateway stop` can still fall through to the service-disabled message instead of signaling the foreground Gateway.

## What This Changes

The issue requests that `openclaw gateway stop` stop foreground or unmanaged Gateway processes instead of only reporting that the service is disabled.

## Best Possible Solution

Land #72224 after rebase and maintainer review, or land a narrower equivalent that recognizes a bare `openclaw-gateway` listener while preserving verified-process signaling and adds stop-path regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. The reporter provided concrete foreground-Gateway steps and a retest, and current main still has a source-level path where `['openclaw-gateway']` yields no verified PID and falls through to the service-disabled result.

## Solution Assessment

Yes. A narrow argv-verifier fix is the best direction because it keeps the safety gate before SIGTERM; a message-only change would leave container and sandbox users manually killing a valid Gateway process.
