## Summary

Keep open: current `main` and v2026.6.8 still miss the reported foreground `openclaw-gateway` process when no service manager is loaded, because listener discovery can return no PIDs when `lsof` is unavailable and the argv verifier still rejects a bare `openclaw-gateway` argv. The open related PR appears to address the central bug, but it is still open and conflicting, so this issue remains useful until that fix or a narrow replacement lands.

## What This Changes

The issue asks for `openclaw gateway stop` to stop a foreground-launched gateway process instead of only reporting that the service-manager registration is disabled.

## Best Possible Solution

Land a current-main-compatible gateway lifecycle/process-discovery fix that accepts bare `openclaw-gateway` argv, does not depend solely on `lsof` in Linux containers, and proves foreground `openclaw gateway stop` releases the listener.

## Reproduction Assessment

Yes by source inspection and reporter logs, without a fresh live run in this read-only review. Current `main` still rejects the reported `["openclaw-gateway"]` argv shape and can return no listener PIDs when `lsof` is missing, which makes `gateway stop` fall through to the not-loaded service message.

## Solution Assessment

Yes, the best fix belongs in gateway process verification and listener discovery because `gateway stop` already has an unmanaged listener fallback. A docs-only, message-only, or pid-file-only workaround would leave the existing verified-signal path broken.
