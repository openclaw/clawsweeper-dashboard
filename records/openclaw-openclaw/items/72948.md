## Summary

Keep open. Current main still has the reported failure path: `gateway stop` has an unmanaged-process fallback, but it only runs after verified listener PID discovery, and the current argv verifier still rejects a bare `openclaw-gateway` process before the opt-in gateway-binary check can match. The related #72224 PR is the right fix direction, but it is still open rather than merged.

## What This Changes

The issue asks for `openclaw gateway stop` to stop or clearly report a foreground-launched Gateway process when no launchd/systemd/schtasks service is managing it.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that keeps strict gateway-process verification, recognizes a bare `openclaw-gateway` listener under the existing opt-in gate, adds Linux listener discovery that works without `lsof`, and covers foreground `gateway stop` with regression tests.

## Reproduction Assessment

Yes. Start a foreground Gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop` from another shell, and current source shows that a missing `lsof` or bare `openclaw-gateway` argv can produce no verified PID and fall through to `Gateway service disabled.` while the process remains alive.

## Solution Assessment

Yes. A narrow verified unmanaged-process fallback is the best direction because it preserves the safety check before sending SIGTERM; a message-only fix would still leave container and sandbox users manually killing a valid OpenClaw Gateway process.
