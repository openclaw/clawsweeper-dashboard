## Summary

Keep open. Current main still contains the unmanaged stop fallback, but the verified PID path can reject a foreground child whose argv is only `openclaw-gateway`, and the Unix listener discovery path still falls back to an empty PID list when `lsof` is unavailable, so the reported service-disabled no-op remains plausible and source-backed.

## What This Changes

The issue asks for `openclaw gateway stop` to stop a directly launched foreground Gateway instead of reporting service-disabled while leaving the process and port alive.

## Best Possible Solution

Land #72224 after rebase, changelog, and an explicit #72948 closing reference, or land a narrower equivalent that keeps strict process verification while accepting bare `openclaw-gateway` listeners and avoiding a hard `lsof` dependency.

## Reproduction Assessment

Yes. The reporter provided a live foreground `openclaw gateway run --dev --port 38789` reproduction, and current source still shows the matching verifier miss plus the service-disabled fallback path.

## Solution Assessment

Yes, the linked technical direction is the narrow maintainable fix: keep SIGTERM behind verified Gateway process detection, add Linux listener discovery that works without `lsof`, and cover the bare `openclaw-gateway` argv shape with tests.
