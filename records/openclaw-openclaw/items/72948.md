## Summary

Keep open. Current main has an unmanaged stop fallback, but it can still miss the reported foreground/container shape because bare `openclaw-gateway` argv is rejected before the opt-in binary check and Linux PID discovery still returns no candidates when `lsof` is unavailable; #72224 is the open implementation candidate.

## What This Changes

The issue asks for `openclaw gateway stop` to terminate or clearly report foreground-launched gateway processes when no service manager is available.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that keeps strict process verification while recognizing bare `openclaw-gateway` listeners, works without `lsof` in Linux containers, and adds foreground-stop regression coverage.

## Reproduction Assessment

Yes. The issue has concrete shell steps and a 2026.4.26 retest, and current source still shows the two failure conditions: bare `openclaw-gateway` argv is rejected and missing `lsof` yields no verified PID.

## Solution Assessment

Yes. A narrow verified unmanaged-process fallback is the maintainable fix because it preserves safety before sending SIGTERM; a message-only change would leave container and sandbox users manually killing a valid Gateway process.
