## Summary

Keep open. Current `main` still contains the reported failure path: unmanaged stop exists, but the verified listener path can return no PID when `lsof` is unavailable, and the argv verifier still rejects a single-token `openclaw-gateway` process before its opt-in gateway-binary allowance can match. The related #72224 PR appears to contain the right fix direction, but it is still open and not merged.

## What This Changes

The issue requests that `openclaw gateway stop` terminate a foreground-launched gateway process instead of only reporting that the service-manager registration is disabled.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing opt-in gate, adds Linux listener discovery that works without `lsof`, and covers foreground `gateway stop` with regression tests.

## Reproduction Assessment

Yes. Start a foreground gateway such as `openclaw gateway run --dev --port 38789`, run `openclaw gateway stop` from another shell, and current source shows a single-token `openclaw-gateway` listener will not verify, causing the command to fall back to the service-disabled path instead of SIGTERM.

## Solution Assessment

Yes. A narrow verified-process fallback is the best fix because it preserves gateway-only verification before signaling; a message-only change would leave foreground container users manually killing a valid OpenClaw gateway process.
