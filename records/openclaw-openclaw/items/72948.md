## Summary

Keep open. Current main still has the reported foreground-stop failure path: unmanaged stop only runs after finding a verified listener PID, but Linux discovery can return no PID when `lsof` is unavailable, and the argv verifier still rejects a bare `openclaw-gateway` process before its opt-in gateway-binary allowance can match. The open #72224 PR contains the matching fix direction, so this should wait for that PR or a narrow equivalent.

## What This Changes

Request that `openclaw gateway stop` terminate verified foreground-launched or otherwise unmanaged gateway processes instead of reporting the service as disabled while leaving the listener running.

## Best Possible Solution

Land #72224 after rebase and maintainer review, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing opt-in gate, adds Linux listener discovery that works without `lsof`, and covers foreground `gateway stop` with regression tests.

## Reproduction Assessment

Yes. The reporter provided concrete container commands, and current source shows the same path: a foreground listener with bare `openclaw-gateway` argv or no `lsof` yields no verified PID, so stop falls through to the service-disabled result instead of SIGTERM.

## Solution Assessment

Yes. A narrow verified-process fallback that recognizes opt-in bare `openclaw-gateway` argv and uses non-`lsof` Linux listener discovery is the maintainable fix; a message-only change would leave foreground container users manually killing a valid gateway.
