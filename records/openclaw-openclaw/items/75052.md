## Summary

Keep open: current main still has the source-reproducible state-only uninstall data-loss path, and the linked closing PR #75061 is open but unmerged.

## What This Changes

The issue reports that selecting State + config without Workspace during uninstall deletes workspaces located under `~/.openclaw`.

## Best Possible Solution

Land #75061 or an equivalent narrow fix that preserves configured nested workspaces during state cleanup unless the workspace scope is explicitly selected.

## Reproduction Assessment

Yes. Source inspection on current main shows the default workspace is inside the default state directory, and state cleanup recursively removes that parent before the workspace scope branch is considered.

## Solution Assessment

Yes. The narrow maintainable solution is to add a preservation guard for configured nested workspaces during state cleanup while keeping explicit `--workspace` and `--all` deletion behavior destructive.
