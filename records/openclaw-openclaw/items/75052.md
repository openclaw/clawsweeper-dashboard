## Summary

Keep open. Current `origin/main` still removes the whole state directory recursively before the workspace scope is considered, while the default workspace path is nested under `.openclaw`; open PR #75061 is the canonical active fix candidate, so this issue should stay open until that or an equivalent fix lands.

## What This Changes

The issue reports that uninstalling with state/config deletion selected can delete workspaces nested under `.openclaw` even when workspace deletion is not selected.

## Best Possible Solution

Land one canonical uninstall fix that keeps state/config and workspace deletion scopes independent: when workspace deletion is not selected, preserve configured/default workspaces nested under the state directory or stop with an explicit warning, then add focused regression coverage and update uninstall docs.

## Reproduction Assessment

Yes. A destructive live uninstall was not run during this read-only review, but the source path is high-confidence: default workspace resolves inside `.openclaw`, and state-only uninstall recursively removes that parent directory before checking the workspace scope.

## Solution Assessment

Yes. A narrow guard in the uninstall cleanup path, plus regression tests and docs, is the maintainable fix; the best next step is to review or consolidate the already-open PR #75061 rather than queue another repair branch.
