## Summary

Keep open. Current `main` at `757af70bf7cadbd4b9e9303dd065ad48eda09c4d` still removes the state directory recursively during state-only uninstall before workspace scope is considered, and the default workspace remains nested under `.openclaw`; open PR #75061 is the active closing fix candidate and is still unmerged.

## What This Changes

The issue requests that uninstall preserve workspace directories nested under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Land one canonical uninstall fix, preferably by reviewing and landing #75061 or an equivalent patch, that keeps state/config and workspace deletion scopes independent; it should include focused regression coverage, docs for the nested-default-workspace case, and a user-facing changelog entry when merged.

## Reproduction Assessment

Yes. A destructive live uninstall was not run during this read-only review, but the source path is high-confidence: default workspace resolution nests under `.openclaw`, and state-only uninstall recursively removes that parent directory before any workspace-scope decision can preserve it.

## Solution Assessment

Yes. A narrow guard in the uninstall cleanup path, with regression tests and docs, is the maintainable fix; because #75061 already targets the same bug with closing syntax, the best next step is to review or consolidate that PR rather than queue another repair branch.
