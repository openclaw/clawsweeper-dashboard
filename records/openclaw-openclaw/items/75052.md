## Summary

Current main still has the reported data-loss path, so this issue should stay open. The linked open PR #75061 uses closing syntax and remains the canonical repair path rather than a reason to close the issue early.

## What This Changes

The issue reports that selecting state/config deletion during uninstall can remove a workspace stored under `.openclaw` even when workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that preserves configured workspaces nested inside the state directory during state-only uninstall, with regression coverage, docs, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the default workspace is `$HOME/.openclaw/workspace`, and state-only uninstall recursively removes `$HOME/.openclaw` before the separate workspace scope can preserve it.

## Solution Assessment

Yes. The best path is the already-linked narrow preservation fix in the uninstall cleanup path, reviewed through PR #75061 rather than starting a duplicate repair.
