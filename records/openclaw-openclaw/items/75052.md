## Summary

Current main still has the reported data-loss path: state/config uninstall removes the state directory recursively, while the default workspace remains nested under that directory and is not protected when workspace deletion is unchecked. This should stay open while the active closing PR #75061 is reviewed or replaced by an equivalent fix.

## What This Changes

Preserve workspace directories under `~/.openclaw` when uninstalling state/config without selecting workspace deletion.

## Best Possible Solution

Land one canonical uninstall fix, preferably #75061 or an equivalent patch, that keeps state/config deletion and workspace deletion independent for nested workspace paths and adds focused regression coverage, docs guidance, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with the default `~/.openclaw/workspace`, selecting state/config deletion without workspace deletion still calls recursive state-dir removal before any workspace-specific branch can preserve that nested path.

## Solution Assessment

Yes. A narrow preservation guard in the uninstall cleanup path is the maintainable fix, and #75061 already targets that path with tests and docs, so the best next step is review or consolidation of that PR rather than another repair branch.
