## Summary

Keep open: current main still has the state-only uninstall data-loss path, and the linked closing PR #75061 is open but unmerged.

## What This Changes

The issue asks uninstall to preserve workspace directories under `~/.openclaw` when state/config deletion is selected but workspace deletion is not.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that skips or preserves configured workspace directories nested under the state directory unless `--workspace` or `--all` is selected.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: default state is `~/.openclaw`, default workspace is `~/.openclaw/workspace`, state cleanup recursively removes the state directory, and workspace cleanup is skipped when that scope is unchecked.

## Solution Assessment

Yes. The narrow maintainable fix is to add a preservation guard for configured nested workspaces during state cleanup while keeping explicit workspace/all deletion destructive.
