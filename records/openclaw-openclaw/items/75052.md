## Summary

Keep open. Current main still has the data-loss path: state cleanup recursively removes the state directory before the separate workspace scope is considered, and the default workspace is nested under that same `.openclaw` directory. The open closing PR #75061 is the active fix candidate, so this issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue asks for uninstall state/config deletion to preserve workspace files when workspace deletion is not selected, including workspaces nested under `.openclaw`.

## Best Possible Solution

Land one canonical fix, preferably by reviewing and adjusting PR #75061, that keeps state/config deletion independent from workspace deletion, preserves or explicitly protects nested workspaces when workspace deletion is unchecked, and adds focused regression coverage plus uninstall docs/changelog updates.

## Reproduction Assessment

Yes. The source path is high-confidence: default workspaces resolve under `.openclaw`, and state-only uninstall currently removes the parent state directory recursively before workspace-scope handling runs.

## Solution Assessment

Yes, the narrowest maintainable solution is in the uninstall cleanup path: pass the known workspace directories into state cleanup or otherwise guard parent deletion, then prove the behavior with unit coverage and docs. A second automated repair branch is not the best next step while PR #75061 is open.
