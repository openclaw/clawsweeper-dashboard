## Summary

Keep open. Current main still has the reported data-loss path, and open PR #75061 uses closing syntax as the canonical active repair, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that uninstalling with state/config deletion selected can remove workspaces nested under `.openclaw` even when workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that preserves configured workspaces nested inside the state directory during state-only uninstall, while keeping explicit workspace/all deletion destructive and updating regression tests, docs, and changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: default state cleanup recursively removes the state directory, while the default workspace is inside that directory and workspace deletion is a separate later scope.

## Solution Assessment
