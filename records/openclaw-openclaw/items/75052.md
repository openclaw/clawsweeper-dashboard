## Summary

Current main still has the reported state-only uninstall data-loss path, and the open closing PR #75061 remains the canonical fix candidate rather than a reason to close this issue early.

## What This Changes

The issue reports that selecting state/config deletion without workspace deletion can remove workspaces nested under the OpenClaw state directory.

## Best Possible Solution

State cleanup should preserve configured workspace directories when workspace deletion is not selected, with focused regression tests and updated uninstall docs warning about manual parent-directory removal.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with the default `$HOME/.openclaw/workspace`, `openclaw uninstall --state --yes --non-interactive` reaches recursive state-dir removal without any nested workspace preservation step.

## Solution Assessment

Yes. The narrow maintainable fix is to add a preservation guard or equivalent split-delete behavior in the uninstall cleanup path, and #75061 already targets that path with tests and docs.
