## Summary

Keep open. Current main still has the reported data-loss path: state/config cleanup recursively removes the whole state directory, while the default workspace is nested under that directory, and the workspace scope is checked only afterward. Open PR #75061 is an active closing fix candidate, so this issue should stay open until that or an equivalent fix lands.

## What This Changes

The issue reports that uninstalling with state/config deletion selected but workspace deletion unchecked can delete workspaces stored inside `.openclaw`.

## Best Possible Solution

Land one canonical uninstall fix, likely PR #75061 or an equivalent updated patch, that keeps state/config and workspace deletion scopes independent, preserves configured/default workspaces nested under the state directory when workspace deletion is not selected, and adds focused regression coverage plus uninstall documentation.

## Reproduction Assessment

Yes. A destructive live uninstall was not run during this read-only review, but the source path is high confidence: default workspaces are under `.openclaw`, and state-only uninstall recursively removes that parent directory before considering the workspace scope.

## Solution Assessment

Yes. A narrow guard in the uninstall/shared cleanup path with regression tests and docs is the maintainable solution; because PR #75061 already implements that direction, the best next step is to review, rebase, and land or replace that PR rather than opening another fix lane.
