## Summary

Current `main` still has the reported data-loss path, and open PR #75061 is the active linked repair with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

State/config-only uninstall should preserve configured workspace files nested under `.openclaw` when workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow guard that removes state contents while preserving configured nested workspaces unless workspace deletion is selected, with regression tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the default workspace is under `.openclaw`, and `openclaw uninstall --state` recursively removes that parent state directory before any workspace-specific deletion choice runs.

## Solution Assessment

Yes. The narrowest maintainable solution is to pass the configured workspace dirs into the state cleanup path as preservation targets when workspace deletion is not selected; PR #75061 is already the canonical repair path.
