## Summary

Current main still has the reported data-loss path: state-only uninstall removes the whole state directory before any workspace-specific decision can preserve a nested workspace, and open PR #75061 remains the canonical linked repair path.

## What This Changes

Uninstall should preserve workspace directories when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that skips or safely partitions state-directory removal when configured workspaces are nested inside it, with regression tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the default workspace is `~/.openclaw/workspace`, and state-only uninstall recursively removes `~/.openclaw` before any workspace deletion choice can preserve that nested path.

## Solution Assessment

Yes. The best path is the already-linked narrow preservation fix in the uninstall cleanup path, reviewed through PR #75061 rather than starting a duplicate repair.
