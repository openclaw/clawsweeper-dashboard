## Summary

Current main still has the reported data-loss path: state/config uninstall removes the state directory recursively, while the default workspace is nested under that same directory. This issue should stay open while the linked unmerged fix PR #75061 is reviewed or replaced by an equivalent patch.

## What This Changes

The issue requests that uninstall preserve workspace files when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent canonical fix that keeps state/config deletion independent from workspace deletion for nested workspace paths, with regression tests, docs, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the default workspace resolves to `~/.openclaw/workspace`, and state-only uninstall recursively removes `~/.openclaw` before any workspace-specific deletion decision can preserve that nested path.

## Solution Assessment

Yes. A narrow preservation guard in the uninstall cleanup path is the maintainable fix, and the already-linked PR #75061 is the right canonical path to review rather than starting another repair branch.
