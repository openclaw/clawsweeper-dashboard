## Summary

Keep open because current main still has the reported data-loss path: state/config cleanup recursively removes the state directory, while the default workspace is nested under that same `.openclaw` directory and workspace deletion is only checked afterward. The linked open PR #75061 is the active closing fix candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that uninstall state/config deletion preserve workspace directories when workspace deletion is not selected, including workspaces nested under `.openclaw`.

## Best Possible Solution

Land one canonical uninstall fix, preferably by reviewing and landing PR #75061 after its merge hygiene is complete, that keeps state/config deletion and workspace deletion independent for nested workspace paths while adding focused regression coverage, docs guidance, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists with the default `~/.openclaw/workspace`: selecting state/config deletion without workspace deletion still recursively removes the parent state directory before workspace-scope cleanup can preserve the nested workspace.

## Solution Assessment

Yes. Preserving configured workspace paths during state-only cleanup is the narrow maintainable fix; because PR #75061 already targets that path, the best next step is review or consolidation of that PR rather than another repair branch.
