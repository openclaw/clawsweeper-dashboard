## Summary

Keep open: current main still has the reported state-only uninstall data-loss path, and the linked repair PR #75061 is open but unmerged.

## What This Changes

The issue asks for state/config uninstall to preserve workspace directories nested under the OpenClaw state directory when workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that preserves configured nested workspace directories during state cleanup unless the workspace scope is explicitly selected.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the default workspace is inside `~/.openclaw`, state cleanup recursively removes `~/.openclaw`, and workspace cleanup is skipped when the workspace scope is unchecked.

## Solution Assessment

Yes. The narrow maintainable solution is to add a preservation guard for configured nested workspaces during state cleanup while keeping explicit `--workspace` and `--all` deletion behavior destructive.
