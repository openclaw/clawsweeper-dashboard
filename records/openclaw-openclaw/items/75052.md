## Summary

Current main still has the reported data-loss path: state/config uninstall recursively removes the state directory, while the default workspace is nested inside that directory and workspace removal is gated by a separate scope. This should stay open because #75061 is the active open closing PR and has not merged yet.

## What This Changes

The issue requests that uninstall preserve workspace directories under `.openclaw` when configuration/state deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Land one canonical uninstall fix, preferably #75061 or an equivalent narrow patch, that keeps state/config deletion and workspace deletion independent for nested workspace paths, adds focused regression coverage, updates uninstall docs, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with the default workspace under `$HOME/.openclaw/workspace`, selecting state/config deletion without workspace deletion still recursively removes the parent state directory.

## Solution Assessment

Yes. The maintainable fix is a narrow preservation guard in the uninstall cleanup path for configured workspace directories when workspace deletion is not selected; the active linked PR is the right place to review or consolidate that change.
