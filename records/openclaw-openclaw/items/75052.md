## Summary

Keep open. Current main still has the reported data-loss path: state/config uninstall recursively removes the `.openclaw` state directory before workspace scope is applied, while the default workspace remains nested inside that state directory. The issue is also paired with open, unmerged PR #75061 using closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Uninstall should preserve workspace directories under `.openclaw` when state/config deletion is selected but workspace deletion is not.

## Best Possible Solution

Land one canonical uninstall fix, preferably by reviewing and landing #75061 or an equivalent narrow patch, that keeps state/config and workspace deletion scopes independent for nested workspaces and adds regression coverage plus user-facing docs/changelog updates.

## Reproduction Assessment

Yes. A destructive live uninstall was not run during this read-only review, but the source path is high-confidence: the default workspace is under `.openclaw`, and state-only uninstall recursively removes that parent before workspace deletion scope is considered.

## Solution Assessment

Yes. The best fix is a narrow guard in the uninstall cleanup path that preserves configured workspace directories when workspace deletion is not selected; because #75061 already targets that path with closing syntax, maintainers should review or consolidate that PR rather than queue another repair branch.
