## Summary

Keep open: current main still has the state-only uninstall path that recursively removes the state directory before workspace cleanup, and the linked repair PR #75061 is open but unmerged.

## What This Changes

The issue reports that uninstalling state/config without workspace cleanup can delete a workspace nested under `~/.openclaw`.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that preserves configured workspaces nested under the state directory unless `--workspace` or `--all` is selected, with regression coverage and docs.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: default workspace under `~/.openclaw`, state cleanup recursively removes `~/.openclaw`, and workspace cleanup is skipped when that scope is unchecked.

## Solution Assessment

Yes. The best path is a narrow preservation guard for configured nested workspace directories during state cleanup while preserving explicit destructive behavior for `--workspace` and `--all`; the linked open PR is the implementation path to review.
