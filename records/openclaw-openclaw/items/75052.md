## Summary

Keep open: current main still has the state-only uninstall data-loss path, and PR #75061 is the open unmerged closing-syntax fix for the same issue.

## What This Changes

The issue asks for uninstall state/config cleanup to preserve workspaces under `.openclaw` when workspace deletion is not selected.

## Best Possible Solution

Review and land PR #75061 or an equivalent narrow fix that preserves configured nested workspaces during state cleanup unless workspace deletion is explicitly selected.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main removes the state directory recursively before considering workspace cleanup, and the default workspace lives under that state directory.

## Solution Assessment

Yes. The narrow maintainable fix is to add a preservation guard for configured workspaces nested under the state directory while keeping explicit `--workspace` and `--all` deletion behavior destructive.
