## Summary

Current main still has the reported state-only uninstall data-loss path: the state directory is removed recursively before the workspace scope is considered, and the default workspace lives under that state directory. The issue should stay open while PR #75061 is reviewed or replaced by an equivalent fix.

## What This Changes

The issue asks the uninstaller to preserve workspace directories under the state directory when state/config deletion is selected without workspace deletion.

## Best Possible Solution

Land PR #75061 or an equivalent focused fix that keeps state/config deletion independent from workspace deletion for nested workspace paths, with regression tests, docs, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the default workspace resolves to `~/.openclaw/workspace`, and state-only uninstall recursively removes `~/.openclaw` before any workspace-specific deletion decision can preserve that nested path.

## Solution Assessment

Yes. A narrow preservation guard in the uninstall cleanup path is the maintainable fix, and the already-linked PR #75061 is the right canonical path to review rather than starting another repair branch.
