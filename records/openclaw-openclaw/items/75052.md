## Summary

Current main still has the reported state-only uninstall data-loss path, and the issue should stay open while linked PR #75061 is reviewed or replaced by an equivalent fix.

## What This Changes

State/config-only uninstall can delete a workspace nested under `~/.openclaw` even when workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that keeps state/config deletion independent from workspace deletion for nested configured/default workspaces, with regression coverage, docs, and changelog treatment as needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: default workspace resolves to `~/.openclaw/workspace`, and state-only uninstall recursively removes `~/.openclaw` before any workspace-specific deletion decision can preserve that nested path.

## Solution Assessment

Yes. Preserving configured/default workspace directories during state-only cleanup is the narrow maintainable direction; because PR #75061 is already open and linked, maintainers should review or replace that PR rather than start a separate automated repair.
