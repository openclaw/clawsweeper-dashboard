## Summary

Keep open: current main still has the state-only uninstall data-loss path, and open PR #75061 is the unmerged closing-syntax fix for the same behavior.

## What This Changes

The issue requests that uninstall preserve workspace files when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Review and land PR #75061, or an equivalent narrow fix that preserves configured nested workspaces during state-only cleanup while keeping explicit workspace deletion destructive.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: default workspace is inside `~/.openclaw`, state-only uninstall recursively removes `~/.openclaw`, and the workspace branch is skipped when that scope is unchecked.

## Solution Assessment

Yes. The active PR direction is the narrow maintainable fix: add a preservation guard for configured nested workspaces during state cleanup without changing explicit `--workspace` or `--all` deletion semantics.
