## Summary

Keep open: current main still has the reported data-loss path, and PR #75061 is the open linked repair but has not merged.

## What This Changes

The issue asks uninstall to preserve workspace directories under the OpenClaw state directory when only state/config deletion is selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that preserves configured nested workspace directories when workspace deletion is not selected, with regression coverage and docs/changelog alignment.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the default workspace is under ~/.openclaw/workspace, state-only uninstall recursively removes ~/.openclaw, and workspace cleanup is skipped when the workspace scope is unchecked.

## Solution Assessment

Yes. The narrow maintainable solution is to preserve configured workspace directories during state cleanup when workspace deletion is not selected, as PR #75061 attempts, rather than opening a duplicate repair.
