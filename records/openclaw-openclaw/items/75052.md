## Summary

Current main still has the data-loss path described in the report, and an open closing PR already targets the exact fix, so this issue should remain open for maintainer review rather than be closed or queued again.

## What This Changes

The issue reports that selecting state/config deletion during uninstall can delete a workspace nested under `.openclaw` even when workspace deletion is not selected.

## Best Possible Solution

Review and land #75061, or an equivalent narrow patch, so state/config deletion preserves configured nested workspaces unless the workspace scope is explicitly selected, with regression tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the default workspace resolves to `$HOME/.openclaw/workspace`, while state cleanup removes `$HOME/.openclaw` recursively before any unchecked workspace scope can preserve that nested path.

## Solution Assessment

Yes. A preservation guard in the uninstall cleanup path is the narrow maintainable solution, and the already-open PR #75061 is the canonical implementation candidate to review or adjust.
