## Summary

Keep open. Current main still has the reported data-loss path: state/config uninstall recursively removes the state directory, while default and configured workspaces can live under that same `.openclaw` tree. Two open, unmerged closing PRs (#75061 and #75062) already target this bug, so the issue should remain open until maintainers choose or consolidate one fix.

## What This Changes

The issue asks `openclaw uninstall` to preserve workspaces stored under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion scopes independently. When a workspace to preserve is inside the state directory, either selectively remove non-workspace state paths or stop with a clear warning, then add focused regression coverage and update uninstall docs before landing one canonical PR.

## Reproduction Assessment

Yes. A destructive live uninstall was not run, but source inspection gives a high-confidence reproduction path: a workspace under `.openclaw` is nested inside the state directory, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. A narrow uninstall cleanup guard with regression tests is the best maintainable direction; the immediate path is to review or consolidate the already-open fix PRs rather than create another repair branch.
