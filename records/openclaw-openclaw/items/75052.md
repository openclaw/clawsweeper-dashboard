## Summary

Keep open because current main still has the reported data-loss path: state-only uninstall recursively removes the state directory, while the default workspace remains nested under that directory. The linked open PR #75061 is the active closing fix candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

State/config uninstall should preserve workspaces stored under `.openclaw` when the workspace deletion option is not selected.

## Best Possible Solution

Land one canonical uninstall fix, preferably by reviewing and landing #75061 or an equivalent patch, that keeps state/config deletion and workspace deletion independent for nested workspace paths while adding focused regression coverage, docs guidance for manual removal, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with the default `~/.openclaw/workspace`, selecting state/config deletion without workspace deletion still calls recursive state-dir removal before any workspace-scope cleanup decision can preserve that nested workspace.

## Solution Assessment

Yes. A narrow guard in the uninstall cleanup path that preserves configured workspace directories during state-only removal is the maintainable fix; because #75061 already targets that path, the best next step is review or consolidation of that PR rather than another repair branch.
