## Summary

Current main still has the reported data-loss path: state/config uninstall removes the state directory recursively, while the default workspace lives under that directory. The linked closing PR #75061 remains open and unmerged, so this issue should stay open for review or an equivalent fix.

## What This Changes

Preserve workspace directories under the OpenClaw state directory when uninstalling state/config without selecting workspace deletion.

## Best Possible Solution

Add a narrow uninstall cleanup guard that keeps configured workspace directories independent from state/config deletion, with regression tests, docs, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the default workspace resolves to `~/.openclaw/workspace`, and state-only uninstall recursively removes `~/.openclaw` before any workspace-specific deletion decision can preserve that nested path.

## Solution Assessment

Yes. The narrow maintainable fix is to make state cleanup aware of workspace paths that must be preserved when the workspace scope is not selected; PR #75061 is the current canonical implementation path to review.
