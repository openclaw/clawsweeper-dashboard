## Summary

Keep open: current main still has the reported data-loss path, and PR #75061 is the open linked repair but has not merged.

## What This Changes

The issue asks OpenClaw uninstall to preserve workspace directories under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make state cleanup preserve configured/default workspace directories when the workspace scope is not selected, with regression coverage and aligned uninstall docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the default workspace is under `~/.openclaw/workspace`, state cleanup recursively removes `~/.openclaw`, and the workspace cleanup branch is skipped when workspace deletion is unchecked.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing uninstall cleanup path workspace-aware when the workspace scope is not selected, rather than changing the uninstall product model or opening a duplicate repair.
