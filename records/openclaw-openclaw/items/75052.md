## Summary

Keep open: current main still has the reported data-loss path, and the linked repair PR is open but not merged.

## What This Changes

The issue reports that uninstalling with state/config deletion selected but workspace deletion unchecked can remove workspaces nested under the OpenClaw state directory.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that preserves configured nested workspace directories when workspace deletion is not selected, with regression coverage and uninstall docs aligned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the default workspace is under ~/.openclaw/workspace, state cleanup recursively removes ~/.openclaw, and workspace cleanup is skipped when the workspace scope is unchecked.

## Solution Assessment

Yes. The best path is the open narrow repair PR or an equivalent owner-reviewed fix that adds a preservation guard to state cleanup rather than a duplicate ClawSweeper PR.
