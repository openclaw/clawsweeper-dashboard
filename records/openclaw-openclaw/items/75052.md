## Summary

Keep open: current main still has the state-only uninstall data-loss path, and PR #75061 is the open closing-syntax fix for the same bug.

## What This Changes

The request is to preserve workspaces nested under `.openclaw` when uninstall removes state/config but workspace deletion is not selected.

## Best Possible Solution

Land PR #75061 or an equivalent narrow guard that preserves configured nested workspaces during state cleanup unless the workspace scope is explicitly selected.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the default workspace is inside `~/.openclaw`, state cleanup recursively removes `~/.openclaw`, and workspace cleanup is skipped when the workspace scope is unchecked.

## Solution Assessment

Yes. The narrow maintainable solution is to guard state-directory removal when a configured workspace must be preserved, while keeping explicit `--workspace` and `--all` deletion behavior destructive.
