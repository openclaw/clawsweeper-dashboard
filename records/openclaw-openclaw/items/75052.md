## Summary

Keep open: current main still has the state-only uninstall data-loss path, and PR #75061 is an open, unmerged closing-syntax repair for the same bug.

## What This Changes

The issue reports that uninstalling with state/config cleanup selected but workspace cleanup unchecked can delete workspaces nested under `~/.openclaw`.

## Best Possible Solution

Land PR #75061 or an equivalent narrow fix that preserves configured nested workspace directories during state cleanup unless workspace deletion is explicitly selected.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the default workspace is inside `~/.openclaw`, state cleanup recursively removes `~/.openclaw`, and workspace cleanup is skipped when the workspace scope is unchecked.

## Solution Assessment

Yes. The narrow maintainable fix is to add a preservation guard for configured nested workspaces during state cleanup while keeping explicit `--workspace` and `--all` deletion behavior destructive.
