## Summary

Keep open. Current main still has the reported data-loss path: state cleanup recursively removes the state directory even though the default workspace can live inside it, and the issue is already paired with open PR #75061 using closing syntax.

## What This Changes

Uninstall should preserve workspace files nested under the OpenClaw state directory when workspace deletion is not selected.

## Best Possible Solution

Land one canonical uninstall fix that keeps state/config deletion independent from workspace deletion, preserves configured and default workspaces nested under the state directory when workspace deletion is not selected, and adds focused regression tests plus uninstall docs.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but the source path is high-confidence: the default workspace is inside `.openclaw`, and state-only uninstall recursively removes that parent directory before any workspace-scope removal branch runs.

## Solution Assessment

Yes. A narrow guard in the uninstall cleanup path, with regression tests and docs, is the maintainable fix; the best next step is to review or land the already-open PR #75061 rather than queue another repair branch.
