## Summary

Keep open. Current main does not implement `/trust` or `/untrust`, and the proposed patch is security-sensitive and still has blocking semantic issues around time-window enforcement and status visibility.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` chat commands, in-memory session-scoped trust-window state, a reply exec-security override, docs/changelog entries, and unit tests.

## Best Possible Solution

Land one consolidated trust-window design that keeps time-bounded authority explicit through execution, reports active trust in user-facing status/help surfaces, and aligns with the broader exec-approval design tracked by the related open work.

## Reproduction Assessment

Yes. Current-main absence is reproducible by source/docs search, and the queued-run issue is reproducible from the PR diff plus current main's `followupRun.run.execOverrides` storage and replay path.

## Solution Assessment

No. The feature direction is plausible, but resolving trust into parse-time `execOverrides.security` is not the safest maintainable solution for a time-bounded security grant; execution-time revalidation or an explicit expiring authority is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Revalidate trust before queued runs execute:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: The PR resolves active trust into `execOverrides.security` while parsing the message. Queued and follow-up runs later replay the stored `run.execOverrides`, so a run accepted near expiry can execute with `security=full` after the window elapsed. Represent trust as an expiring authority or recheck it at execution time.
  - confidence: 0.88
- **[P2] Disclose active trust in exec status:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: During an active `/trust` window, effective exec security can become `full`, but `/exec` status still reports config/session defaults from `resolveExecDefaults`. Operators can be told `security=deny` or `allowlist` while trust is active, so status should include the override and remaining time.
  - confidence: 0.82

  - confidence: 0.9

## Security Review

Concerns:

- **[high] Queued runs can outlive the trust window:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: Trust is resolved into a stored `execOverrides.security` value at message parse time, while queued and follow-up runs replay that stored value later. This can extend `security=full` beyond the promised trust window for delayed execution.
