## Summary

Keep this PR open, but do not merge it as-is: current main still lacks `/trust` and `/untrust`, while this patch introduces a security-sensitive privilege-elevation path with blocking time-window and status-disclosure problems plus overlapping open trust-window designs.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` slash commands, in-memory session trust-window state, exec-security override resolution, docs, changelog text, and unit tests.

## Best Possible Solution

Land one consolidated trust-window design that validates time-bounded authority at execution time, reports active trust and remaining duration in user-visible status/help surfaces, and keeps docs/changelog scoped to that design.

## Reproduction Assessment

Yes. Source inspection shows the PR resolves trust into `execOverrides` during directive parsing, while current main stores and replays `run.execOverrides` for queued/follow-up execution; exact search also confirms current main has no `/trust` or `/untrust` implementation.

## Solution Assessment

No. The feature direction is plausible, but a parse-time `security=full` override is not the narrowest safe model for time-bounded privilege; execution-time revalidation or an explicit expiring authority is safer, with status disclosure included.
