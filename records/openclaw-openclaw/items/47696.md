## Summary

Keep open. The PR is a plausible feature direction, but it still needs real behavior proof and has source-reproducible security/UX issues around queued runs and effective exec-status reporting before maintainer review can consider it mergeable.

## What This Changes

The branch adds `/trust [minutes]` and `/untrust` slash commands, in-memory session-scoped trust windows, exec-security override resolution, tests, docs, and changelog text.

## Best Possible Solution

Define the canonical timed exec approval/trust model, then land a rebased implementation that keeps expiry authoritative at execution time, exposes active trust in status/help surfaces, and includes real proof from the contributor’s setup.

## Reproduction Assessment

Yes for source review. The queue and status issues are reproducible by tracing the PR's trust override into current main's queued-run storage/replay and `/exec` status code; no live runtime was run in this read-only review.

## Solution Assessment

No as-is. The slash-command UX is useful, but the implementation should recheck or carry trust expiry through execution and status reporting, and it should align with the broader timed-approval design before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Recheck trust when queued runs execute:** `src/auto-reply/reply/get-reply-directives.ts:74`
  - body: The trust window is resolved into `execOverrides.security = "full"` before the run is queued. Current queued/follow-up runs replay `run.execOverrides` later, so a message accepted near expiry can still execute with full security after `/trust` has expired or been revoked. Store an expiring trust marker or resolve trust at execution time instead of persisting a bare `full` override.
  - confidence: 0.86
- **[P2] Show active trust in exec status:** `src/auto-reply/reply/get-reply-directives.ts:74`
  - body: The new override changes effective exec security, but `/exec` status still reports defaults from session/config state only. During an active trust window, users can see `security=deny` or `allowlist` while commands actually run as `full`; include the active trust window or otherwise disclose the effective override.
  - confidence: 0.82
