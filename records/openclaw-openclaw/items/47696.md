## Summary

Keep open. Current main does not implement `/trust`, and this PR is meaningful but not merge-ready: source review shows active trust is converted into a static exec override that can outlive expiry in queued runs, `/exec` status would report stale security, real behavior proof is missing, and the broader timed-approval design remains open.

## What This Changes

This PR adds core slash-command docs/registry entries and a `commands-trust` handler that stores session-scoped in-memory trust windows and resolves them as `exec.security=full` during reply directive handling.

## Best Possible Solution

Keep timed exec approval work in a canonical design path that revalidates expiry at execution time, reports active trust consistently in status surfaces, rebases onto the current command registry, and includes redacted real runtime proof before merge.

## Reproduction Assessment

Yes, for the review findings: source inspection shows the PR snapshots trust into `execOverrides`, while current queue execution reuses stored overrides and `/exec` status reads only `sessionEntry` defaults. I did not run the PR branch because this is a read-only review and the source path is sufficient.

## Solution Assessment

No. The proposed narrow slash-command wrapper is not the best mergeable solution until it handles queued-run expiry, status visibility, current registry placement, and the broader timed-approval product decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Re-check trust before queued runs execute:** `src/auto-reply/reply/get-reply-directives.ts:77`
  - body: The added resolver turns an active trust window into a plain `execOverrides.security = "full"` while parsing the message. Queued and follow-up runs persist that object and later pass it unchanged to the agent runner, so a message queued before expiry can execute after expiry with full exec security. Carry the trust expiry through and revalidate when the run starts, or avoid snapshotting transient trust into stored run overrides.
  - confidence: 0.9
- **[P2] Reflect active trust in `/exec` status:** `src/auto-reply/reply/get-reply-directives.ts:77`
  - body: During an active `/trust` window, effective exec security becomes `full`, but `/exec` status still reports `resolveExecDefaults` from the session entry and can show `deny` or `allowlist`. That gives users an inaccurate security state; the status reply should disclose the active trust override and remaining time or use the same effective resolver.
  - confidence: 0.88
