## Summary

Keep open: this is a new exec-privilege feature that is not implemented on current main, and the proposed patch still has security-relevant correctness gaps plus missing real behavior proof.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` slash commands with in-memory session trust state, exec override resolution, tests, docs, and a changelog entry.

## Best Possible Solution

Settle the timed exec-approval product shape, then implement trust at the execution/approval boundary with live expiry checks, visible status/audit, focused tests, and cross-surface proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: the PR resolves trust during message parsing, while current queue code stores and later reuses `run.execOverrides`; no live test was run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. A temporary exec trust window should be rechecked where execution/approval is resolved and disclosed in `/exec` status, rather than being flattened into a parse-time session override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Re-check trust at execution time:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: The new call resolves an active trust window into `execOverrides.security` when directives are parsed. Queued/follow-up runs store that object and later pass `run.execOverrides` to the agent, so a message queued before expiry can still execute with `security=full` after the window expires. Keep the trust state or expiry live until the run starts, or re-resolve it in the execution path.
  - confidence: 0.88
- **[P2] Show active trust in `/exec` status:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: While trust is active, effective security can be `full`, but the existing `/exec` status path reports `resolveExecDefaults(...)` from the session/config and has no trust state. Users can see `security=deny` or `allowlist` while the next turn will run full, so the status output needs to disclose the active trust window.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch introduces a temporary exec escalation path but snapshots and hides trust state in ways that can exceed or obscure the intended security window.

Concerns:

- **[high] Queued runs can outlive trust expiry:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: Trust is converted into a plain `execOverrides.security=full` during directive parsing, and queued runs later execute with the stored override even if the trust timer has expired.
  - confidence: 0.88
- **[medium] Effective full trust is not reported by `/exec`:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: The status command still reports persisted exec defaults, so users may believe approvals are enabled while the trust override makes execution full-trust.
  - confidence: 0.84
