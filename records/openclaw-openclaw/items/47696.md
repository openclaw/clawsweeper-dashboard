## Summary

Keep open. Current main still does not implement `/trust` or `/untrust`, but this PR should not merge as-is because it adds a security-sensitive temporary exec elevation path that snapshots trust into queued run state and does not disclose active trust in `/exec` status; it also overlaps broader open trust-window proposals that need maintainer design selection.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` chat commands, in-memory session-scoped trust-window state, an exec-security override during reply directive resolution, command registry/docs/changelog updates, and unit tests.

## Best Possible Solution

Land one consolidated trust-window design that keeps the time-bound authority explicit until execution, handles queued and delayed runs intentionally, reports active trust in status/help surfaces, preserves current authorization boundaries, and includes scoped docs, changelog, and tests for the chosen user surfaces.

## Reproduction Assessment

Yes. Current-main absence is reproducible with targeted source/docs search, and the queued-run issue is reproducible from the PR diff plus current main's `followupRun.run.execOverrides` storage and replay path.

## Solution Assessment

No. The feature direction is plausible, but resolving trust into parse-time `execOverrides.security` is not the safest maintainable implementation for a time-bounded security grant; execution-time revalidation or an explicit expiring authority is safer.
