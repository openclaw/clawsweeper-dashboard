## Summary

Keep open. Current main still lacks the requested trust commands, but this PR is not merge-ready because the proposed static exec-security override has expiry/status security gaps, lacks real behavior proof, and overlaps an unsettled timed-approval design thread.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` slash commands, in-memory session trust state, command registry/docs/changelog entries, unit tests, and a trust-derived exec security override.

## Best Possible Solution

Define one canonical timed exec approval or trust model that rechecks expiry at execution time, discloses active trust in status, preserves hard security ceilings, covers audit/authority semantics, and includes redacted real-runtime proof.

## Reproduction Assessment

Yes for source-level PR review. The queue-expiry and `/exec` status issues are visible by tracing the PR’s trust override into current main’s static queued-run `execOverrides` storage/replay and status formatting; no live runtime was run in this read-only review.

## Solution Assessment

No as-is. The slash-command UX may be useful, but resolving trust to a static `security=full` override is not an authoritative expiry design, and the status path would hide the active escalation.
