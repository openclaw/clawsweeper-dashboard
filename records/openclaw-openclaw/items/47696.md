## Summary

Keep open: current main does not implement the requested trust commands, and this PR should not merge as-is because the temporary trust decision is snapshotted before queued execution and the exec status path can report safer defaults while trust is active.

## What This Changes

The PR adds session-scoped `/trust [minutes]` and `/untrust` slash commands, docs/changelog entries, and tests that temporarily resolve exec security to `full`.

## Best Possible Solution

Adopt one canonical trust-window design, then update this PR or a replacement against current main so trust is checked at execution time and active trust is visible in exec status before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows trust is resolved once into stored `execOverrides`, while queued/follow-up runs later replay that stored object. This is a PR-diff reproduction path, not a current-main product bug.

## Solution Assessment

No. The narrow slash-command approach is plausible, but the current patch needs execution-time expiry enforcement, active-trust status reporting, and a rebase onto the current command/directive structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Re-check trust when queued runs execute:** `src/auto-reply/reply/get-reply-directives.ts:77`
  - body: The trust window is resolved here into a static `execOverrides.security = "full"` value, but queued and follow-up runs replay `run.execOverrides` later. A message queued near expiry can therefore execute after the trust window has elapsed while still carrying `security=full`, which breaks the time-bound security promise. Store enough trust state to re-evaluate at run start, or move the trust check to the execution path.
  - confidence: 0.93
- **[P2] Show active trust in `/exec` status:** `src/auto-reply/reply/get-reply-directives.ts:77`
  - body: This override changes effective exec security without updating the no-args `/exec` status path, which still reports session/config defaults from `resolveExecDefaults`. During an active trust window, users can be told `security=deny` or `allowlist` even though execution is effectively `full`; disclose the active trust window or include it in the status calculation.
  - confidence: 0.89
