## What This Changes

The PR adds a catch block in `downloadFile()` that removes the destination archive with `rmSync(dest, { force: true })` before rethrowing download failures.

## Best Possible Solution

Land the centralized cleanup after redacted real behavior proof is posted, ideally with a focused pipeline-failure regression test and a refresh that preserves current response-body cancellation.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection shows a clear path where `pipeline()` can reject after `createWriteStream(dest)` creates a partial archive and before `downloadTool()` reaches its later cleanup.

## Solution Assessment

Yes, this is the right narrow boundary because `downloadFile()` owns the temporary archive write and future callers should not duplicate cleanup. A temp-file-then-rename design would be stronger, but it is a broader rewrite than this bug needs.
