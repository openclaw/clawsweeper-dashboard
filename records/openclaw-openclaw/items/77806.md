## Summary

Keep open: current main has partial Feishu/Lark doctor repair coverage, but the missing-plugin repair path can still install a catalog candidate when the same plugin id is already discoverable locally but lacks a managed install record.

## What This Changes

The issue asks `openclaw doctor --fix` to avoid installing Feishu/Lark plugin packages that duplicate an already-installed local plugin id or channel owner.

## Best Possible Solution

Doctor’s missing-plugin repair should treat an already discovered local/config/global plugin id as present, and use channel ownership metadata to avoid installing a second Feishu/Lark owner unless the user explicitly forces a reinstall through the plugin installer.

## Reproduction Assessment

No live high-confidence run was established; `pnpm` is unavailable and this review is read-only. The source-level reproduction is clear enough to keep open: discovered local plugin ids without managed records can still be installed from catalog candidates, and tests already assert the explicit Feishu/openclaw-lark conflict-adjacent path.

## Solution Assessment

Yes, a narrow source fix is available: make doctor de-duplicate missing-plugin install candidates against discovered manifest/plugin ownership before package install, with regression coverage around local `openclaw-lark` and Feishu channel ownership. Avoid a Feishu-only alias table unless maintainers decide that alias migration is part of the product contract.
