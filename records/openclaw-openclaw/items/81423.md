## Summary

Keep open: the PR is a useful bug-fix direction, but the latest head still leaves a QMD-rejected dashed semantic query case, corrupts unrelated changelog UTF-8 content, and has only mock/runtime-boundary proof rather than real QMD behavior proof.

## What This Changes

The PR changes memory-core QMD v2 payloads so lexical search keeps the raw query while vec/hyde use a hyphen-normalized query, adds QMD manager tests, and adds a changelog entry.

## Best Possible Solution

Land a cleaned patch that preserves raw QMD lexical search, normalizes every word-internal hyphen only for semantic searches, restores changelog encoding, and includes live QMD proof or an explicit maintainer waiver.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main sends raw hyphenated queries to QMD vec/hyde, QMD v2.0.1 rejects those semantic queries, and OpenClaw falls back to builtin on the primary QMD error.

## Solution Assessment

No for this exact patch. Semantic-only normalization is the right narrow direction, but the regex must replace every internal hyphen without consuming neighboring characters, and the changelog/proof blockers need resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Normalize every word-internal hyphen:** `extensions/memory-core/src/memory/qmd-manager.ts:3155`
  - body: `normalizeQmdSemanticQuery()` consumes both neighboring word characters during each replacement, so adjacent segments are skipped: `x-y-z` becomes `x y-z`. QMD still rejects that semantic query via `/-\w/`, leaving short dashed identifiers able to drop the whole QMD query path; replace the hyphen itself and add coverage for this case.
  - confidence: 0.94
- **[P2] Restore CHANGELOG.md UTF-8 content:** `CHANGELOG.md:1`
  - body: The diff adds a BOM at the start of `CHANGELOG.md` and rewrites unrelated existing non-ASCII release-note text as mojibake, including arrows and emoji. That would publish corrupted release notes, so restore the original UTF-8 content and keep only the intended Memory/QMD entry.
  - confidence: 0.98
