## Summary

Keep open: the memory-code direction is right, but the current PR is not merge-ready because the semantic hyphen normalizer still misses some word-internal hyphens, CHANGELOG.md is re-encoded, and the contributor proof is mocked/source-level rather than live QMD behavior.

## What This Changes

The PR changes QMD v2 search payload construction so lex uses the raw query, vec/hyde use a hyphen-normalized query, adds QMD manager tests, and edits CHANGELOG.md.

## Best Possible Solution

Land a cleaned fix that preserves raw QMD lexical search, normalizes every word-internal hyphen only for semantic searches, restores changelog encoding, and includes live QMD proof or an explicit waiver.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main sends raw hyphenated queries to QMD vec/hyde and falls back to builtin when QMD rejects them. The PR's new regex gap is also source-reproducible because x-y-z normalizes to x y-z, which still matches QMD's semantic-query rejection regex.

## Solution Assessment

Yes for the intended direction, but no for this exact patch. Semantic-only normalization is the narrow maintainable fix, but the regex needs to replace the hyphen itself rather than consume neighboring word characters, and the changelog/proof blockers must be resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Normalize every word-internal hyphen:** `extensions/memory-core/src/memory/qmd-manager.ts:3155`
  - body: `normalizeQmdSemanticQuery()` consumes the characters on both sides of each replacement, so adjacent single-character segments are skipped: `x-y-z` becomes `x y-z`. QMD v2.0.1 still rejects that semantic query via `/-\w/`, so dashed identifiers with short components can still drop the whole QMD query path; replace the hyphen itself, for example with a lookaround/equivalent, and add coverage for this case.
  - confidence: 0.93

  - confidence: 0.98
