## Summary

Keep open for maintainer review. Current main has absorbed or reworked some of the intended Telegram fixes, but it has not fully implemented the batch, and the proposed branch is stale against the plugin-owned Telegram layout with a concrete approval-authorization regression in the diff.

## What This Changes

The PR batches Telegram delivery, media handling, command-menu caching, network/proxy, draft-preview, metadata-stripping, retry, ACP thread-binding, and approval-flow fixes across shared code plus the older `src/telegram` layout.

## Best Possible Solution

Split the useful source work into focused current-main PRs under `extensions/telegram` and shared approval/ACP seams only where needed. Keep the current implementations for already-solved retry, ACP placement, reply-parameter, proxy, and streaming behavior, and give the approval authorization change a separate security-focused review.

## Reproduction Assessment

Yes for source-level reproduction of the remaining gaps: current main still hashes only the command list, returns `null` on exhausted `getFile`, rethrows non-recoverable media-group failures, and lacks an explicit-network guard around sticky IPv4 fallback. There is no single runtime reproduction because this PR batches many independent Telegram bugs.

## Solution Assessment

No. A stale all-in-one branch against the old Telegram layout is not the best way to solve this; the maintainable path is focused plugin-owned changes with targeted tests and separate review for the approval authorization behavior.
