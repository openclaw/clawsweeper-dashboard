This PR should remain open: it still contains useful Telegram fixes, but it is broad, old/conflicting, lacks contributor real Telegram proof, and the approval authorization refactor introduces a security-sensitive behavior regression where non-allowlisted Telegram users can get explicit /approve responses.

## What This Changes

The PR batches Telegram fixes across draft streaming cleanup, reply fallback parameters, reasoning-tag stripping, media download/error handling, media-group recovery, mention-gating logs, ACP thread placement, IPv4/proxy handling, command-menu hash invalidation, metadata stripping, and exec approval handling.

## Best Possible Solution

Rebase or split the still-useful Telegram fixes into the current plugin layout, preserve allowFrom silence for non-approvers, and attach redacted real Telegram proof for the visible flows before merge.

## Reproduction Assessment

Yes. The PR diff is enough to reproduce the approval regression by source: a Telegram sender outside allowFrom can reach explicit /approve replies instead of the prior silent authorization drop, though no live Telegram run was performed in this review.

## Solution Assessment

No. A broad conflicting batch without real Telegram proof and with an approval authorization regression is not the safest merge path; focused rebased fixes with proof are the maintainable route.
