## Summary

Keep this PR open; the failed-count accounting and provider guard changes look useful, but the current head still retries wrapped EPIPE/ECONNRESET/ETIMEDOUT/ECONNABORTED `chat.postMessage` failures that can duplicate visible Slack replies.

## What This Changes

The PR adds Slack `chat.postMessage` retry handling and failed-count-aware delivery accounting, plus Slack tests, a changelog entry, SSRF-guarded Chutes/Hugging Face model discovery, and a plugin dependency contract adjustment.

## Best Possible Solution

Land the failed-count accounting and SSRF guard improvements after limiting Slack send retries to non-duplicating failure modes or adding idempotency/dedupe for replayed posts.

## Reproduction Assessment

Yes. The source-level path is clear: PR head tests and code retry a wrapped `EPIPE` from `chat.postMessage`, while current main only retries DNS request failures and disables generic Slack write-client retries.

## Solution Assessment

No. The delivery accounting change is narrow, but replaying ambiguous `chat.postMessage` transport failures is not the safest fix without a proof of non-delivery or a concrete dedupe/idempotency mechanism.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
