## Best Possible Solution

Rebase onto current main, keep the readiness-only retry if desired, preserve existing Slack payload options, and rely on exactly one normalized dispatch-count contract with mixed success/failure regression coverage.

No. The retry boundary is maintainable, but the accounting correction should live in one layer only and the retry wrapper must be applied on top of current main’s Slack payload builder.
