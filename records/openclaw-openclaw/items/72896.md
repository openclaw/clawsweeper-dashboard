## Summary

Keep this PR open. The Slack retry wrapper and wrapped-error unwrapping are well-scoped, but the latest PR head still drops successful `tool` reply counts from the visible-delivery check and still lacks the required user-facing changelog entry.

## What This Changes

The PR adds bounded Slack `chat.postMessage` transient retries, unwraps Slack Web API request errors, adjusts failed-delivery accounting for Slack inbound replies, and adds focused Slack send/dispatch tests.

## Best Possible Solution

Land a Slack plugin fix that keeps the bounded retry wrapper and wrapped-error unwrapping, computes delivered counts for every `ReplyDispatchKind`, covers both failed-final and tool-only successful delivery cases, and records the user-facing Slack fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. A source-level reproduction is a Slack turn with `counts.tool > 0`, no block/final counts, no queued final, and no fallback signal; current main treats it as visible, while this PR's corrected snapshot passes only block/final counts and marks it undelivered.

## Solution Assessment

No, not as-is. The retry approach is narrow and supported by the Slack dependency source, but the accounting fix should preserve the full shared dispatch-count contract instead of adapting only block/final counts.
