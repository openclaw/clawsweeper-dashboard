## Summary

Keep open: the failed-count accounting and tests are useful, but the current PR still replays ambiguous Slack `chat.postMessage` request errors without idempotency, conflicting with main's intentional DNS-only write retry policy and risking duplicate visible replies.

## What This Changes

The PR adds Slack outbound send retries, adjusts Slack reply visibility accounting with dispatcher failed counts, adds focused Slack tests and a changelog entry, and includes SSRF-guarded Chutes/Hugging Face model discovery plus a plugin dependency contract tweak.

## Best Possible Solution

Land the failed-count accounting after limiting Slack send retries to non-duplicating failure modes or adding a concrete idempotency/dedupe mechanism for retried posts.

## Reproduction Assessment

Yes. The source-level path is clear: PR head replays mocked wrapped `EPIPE`/transport failures from `chat.postMessage`, while current main only retries DNS request failures and keeps write-client retries disabled.

## Solution Assessment

No. The accounting fix is narrow, but the retry classifier should not replay ambiguous `chat.postMessage` transport errors unless the implementation can prove non-delivery or dedupe the visible post.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid retrying ambiguous postMessage writes:** `extensions/slack/src/send.ts:260-263`
  - body: This classifier retries wrapped `EPIPE`, `ECONNRESET`, `ETIMEDOUT`, and `ECONNABORTED` from `chat.postMessage`. Those failures can happen after Slack accepts the request but before the client receives the response, so replaying the call can duplicate a visible reply. Current main deliberately keeps Slack write clients at `retries: 0` and only retries DNS-resolution failures; please remove these ambiguous code-based retries or add a dedupe/idempotency mechanism before retrying them.
  - confidence: 0.86
