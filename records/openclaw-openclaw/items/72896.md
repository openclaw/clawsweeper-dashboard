## Summary

Keep open: the retry scope is now narrow, but the latest head double-subtracts failed reply counts after shared dispatch already normalizes them, and the external PR still lacks after-fix real Slack behavior proof.

## What This Changes

The PR adds Slack client-readiness retry handling around `chat.postMessage`, adjusts Slack visible-delivery accounting, adds Slack regression tests, updates the changelog, and tweaks the plugin runtime dependency contract test.

## Best Possible Solution

Update Slack accounting to trust the normalized dispatch result, keep readiness-only retries, add mixed success/failure regression coverage, and attach after-fix Slack runtime proof before merge.

## Reproduction Assessment

Yes, source-level. The PR head shows shared dispatch already subtracts failed/cancelled deliveries before Slack receives `result.counts`, while Slack subtracts `dispatcher.getFailedCounts()` again; a mixed success/failure turn would be undercounted without needing live Slack to see the code path.

## Solution Assessment

No. The readiness retry boundary is the right shape, but the accounting fix belongs in one layer only, and this head applies the failure correction twice.

## Review Findings
