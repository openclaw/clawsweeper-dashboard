## Summary

Keep open. The Slack reliability problem is real and the PR has useful tests, but the current PR head is dirty against main and its retry helper would regress current Slack send behavior while also retrying ambiguous non-idempotent `chat.postMessage` failures.

## What This Changes

The PR adds bounded Slack `chat.postMessage` transient retries, subtracts failed dispatcher counts from Slack delivery visibility accounting, adds focused Slack tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased Slack-local fix that composes current DNS request retry with safe socket readiness handling and failed-count delivery accounting, with the changelog moved into the active release section.

## Reproduction Assessment

Yes. A source-level reproduction is a mocked Slack send that first throws a wrapped socket/readiness error plus a Slack turn where the dispatcher reports a failed final delivery after a queued final count; current main lacks the new socket retry and failed-count correction.

## Solution Assessment

No, not as the current PR head stands. The direction is narrow, but the retry policy should preserve current-main DNS handling and avoid replaying ambiguous non-idempotent Slack posts without a real dedupe mechanism.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve existing Slack DNS retry handling:** `extensions/slack/src/send.ts:130-139`
  - body: Current main already retries Slack Web API request errors for transient DNS failures such as `EAI_AGAIN` and `ENOTFOUND` by walking `.original`/`.cause` and wrapping `chat.postMessage`. This PR's replacement retry path only classifies socket/write codes and readiness messages, so rebasing it as-is would drop that shipped retry coverage for message sends. Please layer the new socket/readiness handling into the existing DNS retry path or share one classifier.
  - confidence: 0.9
- **[P2] Avoid ambiguous duplicate Slack sends:** `extensions/slack/src/send.ts:132-136`
  - body: `chat.postMessage` is user-visible and the pinned `@slack/web-api` type exposes no idempotency key for this call, but the new classifier retries `ECONNRESET`, `ETIMEDOUT`, and `ECONNABORTED`. Those failures can happen after Slack received the POST but before the client saw the response, so the retry can post a duplicate reply. Please narrow retries to errors that prove no message was sent, or add a real dedupe mechanism before retrying ambiguous request failures.
  - confidence: 0.78
