## Summary

Keep this PR open, but do not merge it as-is. The retry and Slack SDK wrapped-error handling are useful, but the latest head still drops successful `tool` dispatch counts before visible-delivery inference, and the user-facing Slack fix still lacks the required changelog entry.

## What This Changes

The PR adds bounded Slack `chat.postMessage` transient retries, adjusts failed Slack delivery accounting, and adds focused Slack send and preview-fallback tests.

## Best Possible Solution

Keep the bounded Slack retry wrapper and wrapped-error unwrapping, but make failed-count correction generic over `ReplyDispatchKind`, subtract failures for `tool`, `block`, and `final`, pass the full corrected count record to the shared helper, add a tool-only Slack regression test, and add the required Unreleased changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a Slack turn with `counts.tool > 0`, no block/final counts, no queued final, and no fallback signal; current main treats that as visible, while the PR's corrected snapshot passes only block/final counts and marks it undelivered.

## Solution Assessment

No, not as-is. The retry path is narrow and dependency-backed, but the accounting fix should preserve the full shared dispatch-count contract instead of adapting only block/final counts.
