## Summary

Keep open. The failed-delivery accounting is useful, but the latest head still retries ambiguous Slack Web API write/request failures that can duplicate visible `chat.postMessage` replies; current main deliberately limits postMessage retries to DNS request failures and disables Slack SDK write retries.

## What This Changes

The branch adds bounded Slack `chat.postMessage` transient retry handling, failed-count-aware Slack dispatch delivery accounting, focused tests, a changelog entry, and a plugin dependency contract tweak.

## Best Possible Solution

Land the failed-count delivery accounting and any proven non-duplicating socket-readiness retry, but remove code-based ambiguous Web API write retries unless a real Slack post idempotency guard is added.

## Reproduction Assessment

Yes, source-level. The latest PR diff retries wrapped/raw `EPIPE`, `ECONNRESET`, `ETIMEDOUT`, and `ECONNABORTED` from `chat.postMessage`, while current main only retries DNS request failures and sets Slack write-client SDK retries to 0; tests were not run because this pass is read-only.

## Solution Assessment

No. Failed-count accounting is a narrow fix, but replaying ambiguous `chat.postMessage` request failures is not the safest solution without non-delivery proof or an idempotency/dedupe mechanism.
