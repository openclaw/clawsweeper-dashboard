## Summary

Keep open. Current main fixes the exec-approvals atomic-write concern, but the transcript-read concern remains valid because several current gateway/session helpers still read whole transcript files before scanning or aggregating.

## What This Changes

The issue requests atomic exec-approvals writes and bounded or streaming transcript reads to avoid approval-file corruption and large-transcript memory pressure.

## Best Possible Solution

Keep the atomic exec-approvals implementation, then land a focused core transcript change that adds shared streaming or bounded JSONL scan/tail helpers and applies them to the remaining idempotency, latest-assistant, equivalent-message, all-message, and usage paths.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: create a sufficiently large transcript and call the listed helpers, which still read the whole file into memory before scanning; the exec-approvals corruption path no longer reproduces from source because writes now use temp-file replacement.

## Solution Assessment

Yes. A shared streaming or bounded transcript scanner is the narrow maintainable fix; a size cap alone would not cover callers that need forward scans, reverse scans, branch fallback, or usage aggregation semantics.
