## Summary

Keep open. Current main has addressed the exec-approvals atomic-write half and recently bounded some transcript history/usage hot paths, but several current gateway/session helpers still materialize complete transcript files before idempotency, latest-assistant, equivalent-message, and all-message scans, so the large-transcript memory-pressure concern remains valid.

## What This Changes

The issue requests atomic writes for the exec-approvals file and streaming or bounded transcript reads to avoid corrupted approval state and large-transcript memory pressure.

## Best Possible Solution

Keep the shipped exec-approvals atomic replacement, then land a focused core transcript fix that provides shared streaming or bounded JSONL scan/tail helpers and applies them consistently to the remaining idempotency, latest-assistant, equivalent-message, history, sequence, and usage paths.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the remaining resource pattern: with a sufficiently large transcript, the cited helpers still call whole-file read and split before scanning; the exec-approvals corruption path is no longer reproducible from source because writes now go through temp-file replacement.

## Solution Assessment

Yes for the remaining work: streaming or bounded transcript scans are the narrow maintainable fix while preserving the existing atomic exec-approvals implementation. A size cap alone would be less complete because several callers need tail, forward-scan, or branch-aware semantics.
