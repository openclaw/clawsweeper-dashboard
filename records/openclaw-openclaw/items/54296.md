## Summary

Keep open. Current main fixes the exec-approvals atomic-write half, and it has bounded several session-history hot paths, but the remaining transcript-read concern is still present in current source through idempotency, latest-assistant, equivalent-message, full-history, and usage helpers that still read complete transcript files.

## What This Changes

The issue asks to make exec-approvals persistence crash-safe and replace unbounded whole-file transcript reads with bounded or streaming scans.

## Best Possible Solution

Keep the current atomic exec-approvals implementation, then land a focused core transcript change that provides shared streaming or bounded JSONL scan/tail helpers and applies them consistently to idempotency, latest-assistant, equivalent-message, full-history fallback, sequence, recovery, attachment, and usage paths while preserving Pi transcript branch behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a sufficiently large transcript still reaches whole-file readFileSync/readFile plus split in the cited helpers, while the exec-approvals corruption path is no longer present in the write implementation.

## Solution Assessment

Yes. The requested streaming or bounded scan direction is the narrow maintainable solution for the remaining issue; a size cap alone would be less complete because callers need forward scans, reverse scans, tail windows, and branch-aware full-history behavior.
