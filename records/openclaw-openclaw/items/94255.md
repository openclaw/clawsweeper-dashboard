## Summary

Keep open: current main still compacts padded Media* holes before positional lookup, and the latest release has the same reader behavior. The open focused PR is the right fix candidate, so this issue should close only after that fix merges or maintainers choose a different path.

## What This Changes

The issue reports that the persisted user-turn media reader compacts padded MediaUrls/MediaTypes holes, shifting attachment URL and content-type metadata between attachments.

## Best Possible Solution

Keep this issue as the canonical tracker, review or land https://github.com/openclaw/openclaw/pull/94257, and close this issue only after the fix merges or maintainers reject that path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `MediaPaths: ["/media/a.bin", "/media/b.png"]` with `MediaTypes: ["", "image/png"]` is compacted to one type before positional lookup, so the first attachment can receive the later type.

## Solution Assessment

Yes. Preserving array cardinality in the reader is the narrow owner-boundary fix because the writer and sibling media normalizer already treat Media* arrays as positional, while single-field fallbacks can remain unchanged.
