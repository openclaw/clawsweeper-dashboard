## Summary

Keep open: current main still compacts padded Media* holes before positional lookup, and the open linked PR is the right fix candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that padded holes in parallel inbound Media* arrays are compacted by the transcript reader, shifting URLs or content types onto the wrong persisted attachment.

## Best Possible Solution

Keep this issue as the canonical tracker and review or land https://github.com/openclaw/openclaw/pull/94257, closing the issue only after the fix merges or maintainers reject that PR path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `MediaPaths: ["/media/a.bin", "/media/b.png"]` with `MediaTypes: ["", "image/png"]` is compacted to one type before positional lookup, so the first attachment can receive the later type.

## Solution Assessment

Yes. Preserving array cardinality in the reader is the narrow owner-boundary fix because the writer and sibling media code already treat the Media* arrays as positional, while the single-field fallbacks can remain unchanged.
