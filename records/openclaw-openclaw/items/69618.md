The PR branch adds `messages.responseFooter`, enriches `/usage full`, introduces `agents.defaults.responseUsage`, updates docs/schema/help/tests, and also changes gateway session lookup, memory-lancedb runtime fallback, gitignore, and package resolution metadata.

## Best Possible Solution

Land a narrower implementation that keeps the footer/default/usage-context contract, generated schema/docs/help surfaces, inheritance semantics, and block-streaming footer delivery, while fixing the review findings and splitting package policy, memory runtime, and unrelated session-store refactors into separate PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Return the target store data from loadSessionEntry:** `src/gateway/session-utils.ts:461-463`
  - body: `resolveGatewaySessionStoreTarget()` can choose a different store path and canonical key for deleted-main or legacy sessions, but this code still searches and returns the pre-target store/canonical key. Callers can then read or patch the wrong session even though the target helper found the intended backing store.
  - confidence: 0.9
- **[P2] Preserve the unsupported-platform fast fail:** `extensions/memory-lancedb/lancedb-runtime.ts:292-299`
  - body: The new `explicitPlatformOverride` condition disables the existing fast-fail for normal unsupported native platforms such as `darwin-x64`. Those installs now fall through to runtime install attempts before failing, so keep the unsupported-platform guard active unless the platform is actually supported.
  - confidence: 0.88
- **[P2] Compose caller response-template callbacks:** `src/plugin-sdk/inbound-reply-dispatch.ts:143-146`
  - body: `params.replyOptions` may already include `onResponseTemplateContextResolved`, but this assignment overwrites it after spreading caller options. SDK callers that supply the callback will silently lose their hook, so compose both callbacks or omit this field from accepted options.
  - confidence: 0.86
- **[P2] Regenerate the base config schema:** `src/config/zod-schema.session.ts:162`
  - body: The PR adds accepted config keys, but the generated base schema still lacks `messages.responseFooter` and `agents.defaults.responseUsage`. Schema-driven clients and config help cannot discover the new settings until the generated artifact is refreshed.
  - confidence: 0.84
- **[P2] Gate late-template work on configured consumers:** `src/auto-reply/reply/agent-runner.ts:1475-1478`
  - body: Gateway and inbound dispatch now pass `onResponseTemplateContextResolved`, so this condition becomes true for default replies even when `/usage` is off and no footer or late-bound prefix is configured. That forces fresh session-store reads and late usage assembly on the hot reply path for no visible output.
  - confidence: 0.82
- **[P2] Add an inherit mode for channel usage commands:** `src/auto-reply/reply/commands-session.ts:355-357`
  - body: Channel text commands now always persist an explicit `responseUsage` value, but the parser still accepts only concrete modes plus `cost`. After a chat user changes `/usage`, that session cannot return to inheriting `agents.defaults.responseUsage`.
  - confidence: 0.82
