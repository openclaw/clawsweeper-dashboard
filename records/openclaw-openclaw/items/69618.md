Full review comments:

- **[P1] Define loader deps before using sidecar fallback:** `extensions/memory-lancedb/lancedb-runtime.ts:66-74`
  - body: The new fallback calls `deps.resolveStateDir`, `deps.env`, `deps.resolveRuntimeEntry`, `deps.runtimeManifest`, `deps.importResolved`, and `os.homedir()`, but the loader dependency type still only declares `platform`, `arch`, and `importBundled`, and no `os` import is added in the diff. This will fail TypeScript compilation before the runtime fallback can be exercised.
  - confidence: 0.95
- **[P1] Return the target session store data:** `src/gateway/session-utils.ts:570-593`
  - body: `resolveGatewaySessionStoreTarget()` can choose a different store path and canonical key for deleted-main or legacy sessions, but this code still searches and returns the pre-target `storePath`, `store`, and `canonicalKey`. Callers can then read or patch the wrong session even though the target helper found the intended backing store.
  - confidence: 0.9
- **[P2] Update the lockfile for the new dev dependency:** `package.json:1657-1658`
  - body: `package.json` adds `@types/qrcode-terminal`, but the shown `pnpm-lock.yaml` importer patch only changes the `@types/node` specifier. Frozen installs will treat the lockfile as stale until the new dev dependency is represented in the lockfile or removed from this PR.
  - confidence: 0.89
- **[P2] Compose caller response-template callbacks:** `src/plugin-sdk/inbound-reply-dispatch.ts:181-182`
  - body: `params.replyOptions` may already include `onResponseTemplateContextResolved`, but this assignment is written after the spread and overwrites the caller's callback. SDK callers that supply the hook will silently lose it; compose both callbacks or omit this field from accepted options.
  - confidence: 0.86
- **[P2] Regenerate the base config schema:** `src/config/zod-schema.session.ts:147-150`
  - body: The PR adds accepted config keys, but the generated base schema consumed by config-schema clients is not updated for `messages.responseFooter` or `agents.defaults.responseUsage`. Schema-driven UI/help surfaces will not discover the new settings until the generated artifact is refreshed.
  - confidence: 0.84
- **[P2] Gate late-template work on configured consumers:** `src/auto-reply/reply/agent-runner.ts:1558-1564`
  - body: Gateway and inbound dispatch now provide `onResponseTemplateContextResolved`, so this condition becomes true even when `/usage` is off and no footer or late-bound prefix is configured. That forces fresh session-store reads and late usage assembly on the default hot reply path for no visible output.
  - confidence: 0.82
- **[P2] Support restoring inherited usage mode in chat commands:** `src/auto-reply/reply/commands-session.ts:347-356`
  - body: The channel `/usage` command now persists an explicit `responseUsage` value, but this parser still accepts only concrete modes plus `cost`. After a chat user changes `/usage`, that session cannot return to inheriting `agents.defaults.responseUsage`, unlike the TUI/gateway `null` path.
  - confidence: 0.82
- **[P2] Add the required changelog entry:** `docs/response-footer.md:1`
  - body: This PR adds user-facing config and reply behavior, but the diff does not include `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` and `fix` changes to land with a changelog entry so release notes cover the new footer/default behavior.
  - confidence: 0.86
