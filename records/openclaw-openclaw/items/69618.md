## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Define sidecar loader dependencies before use:** `extensions/memory-lancedb/lancedb-runtime.ts:64-77`
  - body: The new fallback calls `resolveRuntimeDir`, `os.homedir()`, and several `deps.*` fields that are neither imported nor declared on `LanceDbRuntimeLoaderDeps`. This will fail TypeScript compilation before the fallback can run.
  - confidence: 0.96
- **[P1] Return the target session store data:** `src/gateway/session-utils.ts:570-592`
  - body: `resolveGatewaySessionStoreTarget()` can choose a different store path and canonical key for deleted-main or legacy sessions, but this function still returns the pre-target `storePath`, `store`, and `canonicalKey`. Callers can then read or patch the wrong session.
  - confidence: 0.91
- **[P2] Update the lockfile for the new dev dependency:** `package.json:1661`
  - body: `package.json` adds `@types/qrcode-terminal`, but the lockfile does not contain the new dev dependency in the root importer. Frozen installs will fail until the lockfile is regenerated or the dependency is removed.
  - confidence: 0.9
- **[P2] Compose caller response-template callbacks:** `src/plugin-sdk/inbound-reply-dispatch.ts:181-184`
  - body: `params.replyOptions` may already include `onResponseTemplateContextResolved`, but this assignment is written after the spread and overwrites the caller's callback. SDK callers that provide the hook will silently lose it.
  - confidence: 0.87
- **[P2] Regenerate the base config schema:** `src/config/zod-schema.session.ts:150`
  - body: The PR adds accepted config keys, but the generated base schema used by config-schema clients is not updated for `messages.responseFooter` or `agents.defaults.responseUsage`. Schema-driven clients will not discover the new settings.
  - confidence: 0.85
- **[P2] Gate late-template work on configured consumers:** `src/auto-reply/reply/agent-runner.ts:1564-1567`
  - body: Gateway and inbound dispatch now provide `onResponseTemplateContextResolved`, so this condition becomes true even when `/usage` is off and no footer or late-bound prefix is configured. That forces fresh session-store reads and late usage assembly on the default reply path for no visible output.
  - confidence: 0.84
- **[P2] Support restoring inherited usage mode in chat commands:** `src/auto-reply/reply/commands-session.ts:342-356`
  - body: The channel `/usage` command now always persists an explicit value, but this parser still accepts only concrete modes plus `cost`. After a chat user changes `/usage`, that session cannot return to inheriting `agents.defaults.responseUsage`.
  - confidence: 0.83
- **[P2] Add the required changelog entry:** `docs/response-footer.md:1`
  - body: This PR adds user-facing config and reply behavior, but the diff does not include `CHANGELOG.md`. OpenClaw policy requires user-facing feature and fix changes to land with a changelog entry.
  - confidence: 0.86
