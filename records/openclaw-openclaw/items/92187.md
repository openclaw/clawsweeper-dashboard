## Label Justifications

- `P1`: The released failure repeatedly disabled a core memory-search workflow after gateway restarts.
- `impact:crash-loop`: The report describes an OOM-killed indexing process followed by repeated rebuild attempts.
- `impact:session-state`: The failure left the persisted memory index empty or without identity metadata, making agent memory unavailable.

## Summary

Current main fixes the destructive v2026.6.5 index-swap behavior behind the repeated empty-index cycle: rebuilds publish over the existing POSIX database without first removing it, and post-swap reopening no longer creates an empty database when the target is absent. An OOM may still abort a rebuild, but it should leave the prior usable index intact rather than repeatedly breaking memory search.

## What This Changes

Prevent gateway restart rebuilds and OOM interruption from leaving the memory index empty and memory search unavailable.

## Best Possible Solution

Keep the current atomic publication and no-create reopen fix; open a narrower resource-usage report only if a current-main rebuild still OOMs without corrupting or replacing the prior index.

## Reproduction Assessment

Yes, by source inspection against v2026.6.5: the old swap protocol temporarily removed the live database and could allow an empty no-metadata database to be opened. Current main has focused regression coverage for the corrected publication and reopen behavior.

## Solution Assessment

Yes for the central user-visible failure. Preserving the valid SQLite database until a completed temporary build can be atomically published is narrower and safer than backup sidecars, checkpoint state, runtime fallback readers, or a new memory-limit option.
