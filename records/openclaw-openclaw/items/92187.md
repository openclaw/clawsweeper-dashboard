## Label Justifications

- `P1`: The report describes a recurring restart-triggered process kill that disables an established memory-search workflow and requires manual recovery.
- `impact:crash-loop`: The embedding rebuild is reportedly killed by memory pressure and retried following gateway restarts.
- `impact:session-state`: Interrupted rebuilding leaves the persistent memory index empty or without usable identity metadata, making stored memory unavailable to searches.

## Summary

Current main can self-heal missing index identity and preserves the prior database during an atomic rebuild, but the reported Ollama rebuild OOM remains unresolved. The linked open PR adds WAL durability and stale-file cleanup rather than bounding or resuming embedding work, so this issue is still necessary as the tracker for the resource-exhaustion failure.

## What This Changes

Prevent gateway restarts or interrupted memory-index rebuilds from repeatedly leaving memory search unavailable after an OOM.

## Best Possible Solution

Keep the last known-good index searchable while rebuilding, bound rebuild memory with streaming or small committed work units, persist enough progress to resume safely after termination, and validate the result with a constrained-memory Ollama rebuild plus forced-kill/restart proof.

## Reproduction Assessment

Partly yes. Source and tests establish the full-reindex, atomic-temp-database, and missing-identity recovery sequence, but this review did not reproduce the reported SIGKILL with 415 files, Ollama bge-m3, and an 8 GB host.

## Solution Assessment

No. WAL checkpointing and stale-file cleanup are useful hardening, but the best fix must also prevent or resume the memory-intensive rebuild that triggers the interruption.
