## Label Justifications

- `P1`: The report describes a restart-triggered failure that repeatedly disables the established memory-search workflow for a real user.
- `impact:crash-loop`: The recovery indexing process is reportedly terminated by the OOM killer and retriggered on later gateway restarts.
- `impact:session-state`: Interrupted index replacement leaves persistent memory identity and search state unavailable.

## Summary

The report remains necessary: current main can self-heal missing index identity and already embeds each file in token-bounded batches, but there is no current-main proof that the reported constrained-memory rebuild completes without OOM. The open linked PR addresses WAL durability and stale swap artifacts rather than directly bounding the full rebuild’s resident memory, so this issue should remain open while that PR and the distinct OOM behavior are evaluated.

## What This Changes

Prevent gateway-restart memory-index rebuilds from exhausting memory and leaving memory search repeatedly unavailable after an interrupted atomic reindex.

## Best Possible Solution

Preserve the existing atomic replacement and missing-identity self-heal design, measure the actual memory owner during a 415-file Ollama rebuild, then bound that owner with streaming or smaller-lifetime batches and prove that both successful rebuild and kill-and-restart recovery retain a usable prior index.

## Reproduction Assessment

Partially yes: current source and focused tests establish the missing-identity and interrupted atomic-reindex paths, but no reviewed evidence reproduces the reported 415-file OOM against current main on an 8 GB macOS/Ollama setup.

## Solution Assessment

No complete solution is established yet. Current self-healing and the linked durability PR address index integrity and cleanup, while the narrowest fix for this issue must also identify and bound the rebuild's actual peak-memory owner without adding an unnecessary memory-limit setting.
