## Summary

Keep open. Current main does not yet have the proposed workflow SDK/Gateway surface, but the current PR head still has blocking correctness issues in attachment delivery context propagation and default Gateway scope routing for dynamic session actions.

## What This Changes

The PR adds Plugin SDK workflow seams for session actions, host-mediated session attachments, scheduled session turns, finalize retry support, Gateway/Swift protocol updates, workflow fixtures, and QA hardening.

## Best Possible Solution

Keep the generic workflow-seam direction, but update the PR head so attachment delivery uses the registry-scoped config for both route lookup and send execution, and dynamic session actions get a nonempty default operator scope path for CLI/default callers while action-specific requiredScopes remain enforced in the handler.

## Reproduction Assessment

Yes. The attachment bug is reproducible by using a non-default session store/config where lookup succeeds through params.config but sendMessage falls back to global config; the scope bug is reproducible by invoking plugins.sessionAction through callGatewayCli without explicit scopes.

## Solution Assessment

No. The proposed additive SDK/Gateway design is plausible, but the current PR head is not the best mergeable solution until the scoped outbound config propagation and dynamic method default-scope routing are fixed with focused regression coverage.
