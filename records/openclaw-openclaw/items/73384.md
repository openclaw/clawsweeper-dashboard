## Summary

Keep open. Current main does not contain the requested workflow SDK seams, and this PR remains the active implementation surface, but the current head still has a blocking attachment-delivery bug and is reported as dirty/unmergeable against main.

## What This Changes

The PR adds generic Plugin SDK workflow seams for session actions, host-mediated attachments, scheduled session turns, finalize retry, protocol/Swift model updates, executable workflow fixtures, and QA hardening.

## Best Possible Solution

Land the generic workflow seams only after the branch is rebased, attachment delivery carries the registry-scoped config through both route lookup and send execution, finalize-event normalization is made consistent, and focused contract tests plus the appropriate changed gate are green.

## Reproduction Assessment

Yes. The attachment bug has a high-confidence static reproduction: call sendSessionAttachment from a registry with a non-default config/session store, observe route lookup use params.config, and assert the mocked sendMessage call lacks cfg. The finalize event issue is reproducible by invoking runAgentHarnessBeforeAgentFinalizeHook with ctx.runId and an event without runId, then checking the hook receives undefined.

## Solution Assessment

No. The additive SDK/Gateway direction is plausible, but the current PR head is not the best mergeable solution until scoped config is passed into sendMessage, finalize event normalization is made consistent, and the dirty branch is updated against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass the scoped config into attachment sends:** `src/plugins/host-hook-workflow.ts:168-177`
  - body: sendPluginSessionAttachment resolves the delivery route with params.config, but the actual sendMessage call omits cfg. In a custom session store or multi-workspace runtime, lookup and send execution can use different configs, causing failed delivery or delivery through the wrong channel/account. Pass cfg: params.config into sendMessage and extend the workflow test to assert it.
  - confidence: 0.91
- **[P3] Pass the normalized finalize event to hooks:** `src/agents/harness/lifecycle-hook-helpers.ts:111-113`
  - body: eventForNormalization fills in runId from ctx.runId, but the hook runner still receives params.event. A plugin can observe a missing event.runId while the retry budget is keyed with the normalized runId, so hook behavior and host budgeting disagree. Pass eventForNormalization to runBeforeAgentFinalize.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces host-mediated attachment delivery, and the current head can send with global runtime config after a registry-scoped route lookup.

Concerns:

- **[medium] Attachment delivery may use global config:** `src/plugins/host-hook-workflow.ts:168`
  - body: sendPluginSessionAttachment uses params.config to find the session delivery route but calls sendMessage without cfg, so outbound channel/account resolution can fall back to global runtime config. In isolated or multi-workspace contexts this can fail delivery or send sensitive artifacts to the wrong configured target.
