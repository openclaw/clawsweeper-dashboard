## Summary

Keep open. Current main and the latest stable tag still reject unsigned Feishu/Lark URL-verification requests before challenge handling, while the focused fix remains open and conflicting.

## What This Changes

Feishu webhook mode should complete encrypted URL-verification challenges that arrive without Lark signature headers when `encryptKey` is configured.

## Best Possible Solution

Land or replace the focused Feishu plugin fix so verified URL-verification challenges are handled before normal event dispatch while preserving strict signatures for non-challenge events.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST an unsigned `url_verification` body; current main returns `401` before challenge handling.

## Solution Assessment

Yes. The maintainable direction is a phase-aware challenge path authorized by signature or verification token, not a broad unsigned webhook bypass.
