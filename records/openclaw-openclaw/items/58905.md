## Summary

Keep this issue open: current main and the latest release still reject unsigned Feishu/Lark `url_verification` requests before challenge handling, while several related fix PRs were closed unmerged. The report is source-reproducible and needs a narrow auth-boundary fix rather than stale cleanup.

## What This Changes

The issue reports that Feishu/Lark webhook URL verification returns `401 Invalid signature` when `encryptKey` is configured because the initial challenge request lacks `X-Lark-Signature`.

## Best Possible Solution

Implement a narrow phase-aware Feishu webhook fix in the plugin: detect and verify URL verification challenges before normal event dispatch, add unsigned encrypted/plaintext challenge regressions, and keep unsigned non-challenge events rejected.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: webhook mode requires `encryptKey`, unsigned requests fail signature validation before `generateChallenge`, and the e2e test currently locks in 401 for missing-signature `url_verification`.

## Solution Assessment

Yes for the direction, but not for current code. The best fix is a narrow challenge-first authentication path that verifies challenge payloads and preserves strict signature validation for normal webhook events.
