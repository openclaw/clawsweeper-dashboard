## What This Changes

The issue reports that Feishu/Lark webhook URL verification fails because encrypted setup-time challenge requests without `X-Lark-Signature` are rejected with `401 Invalid signature` before challenge handling.

## Best Possible Solution

Keep this issue as the canonical user report until a focused Feishu plugin patch lands that handles setup-time `url_verification` before normal event dispatch while preserving strict auth for non-challenge traffic.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: webhook mode requires `encryptKey`, unsigned `url_verification` requests fail signature validation before challenge handling, and current e2e coverage asserts that `401` behavior.

## Solution Assessment

Yes, the phase-aware challenge direction is the best fix shape. The current open PR is a useful candidate, but it still needs conflict refresh, security-boundary review, and real setup proof before this issue can close.
