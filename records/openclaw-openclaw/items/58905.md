## Summary

Keep open. Current main still validates Feishu/Lark webhook signatures before parsing the body or calling the SDK challenge helper, so unsigned setup-time `url_verification` requests are rejected with `401 Invalid signature`; the related open fix candidate is https://github.com/openclaw/openclaw/pull/60485.

## What This Changes

Allow Feishu/Lark webhook URL-verification challenge requests without Lark signature headers to complete when `encryptKey` is configured.

## Best Possible Solution

Land a current-main Feishu plugin fix that accepts only valid `url_verification` setup challenges via valid signature or timing-safe verification-token/decryption validation while preserving current pre-auth guards and unsigned non-challenge rejection.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure Feishu webhook mode with `encryptKey`, then POST a plaintext or encrypted `url_verification` body without Lark signature headers; current main checks `isFeishuWebhookSignatureValid()` before `Lark.generateChallenge()` and the e2e suite currently expects `401`.

## Solution Assessment

Yes. The best direction is a phase-aware challenge path, not a broad unsigned webhook bypass: detect/decrypt `url_verification`, validate the configured token or normal signature, and keep regular event delivery fail-closed.
