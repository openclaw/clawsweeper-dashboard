## Label Justifications

- `P1`: Feishu/Lark webhook setup is blocked for affected users, preventing that channel mode from receiving events.
- `impact:security`: The remaining fix must change the Feishu webhook signature boundary around setup challenges, verification tokens, and encrypted payloads.
- `impact:message-loss`: The failed URL-verification handshake prevents Feishu webhook delivery from being configured, so messages/events cannot reach OpenClaw through that mode.

## Summary

Keep open. Current main and v2026.5.19 still reject unsigned Feishu/Lark URL-verification requests at the signature gate before challenge handling, and https://github.com/openclaw/openclaw/pull/60485 remains the focused open fix path.

## What This Changes

Feishu/Lark webhook mode should complete URL verification when the platform sends an encrypted setup challenge without `X-Lark-Signature` instead of returning `401 Invalid signature`.

## Best Possible Solution

Implement a narrow Feishu webhook setup path that authenticates `url_verification` challenges by signature or verification token/encryptKey, keeps normal event delivery signature-gated, and adds regression coverage for unsigned valid, wrong-token, and malformed encrypted challenges.

## Reproduction Assessment

Yes. Source inspection shows an unsigned Feishu `url_verification` request with `encryptKey` configured returns `401` at the signature gate before challenge parsing, and current e2e coverage explicitly expects that failure.

## Solution Assessment

Yes. The maintainable direction is a phase-aware challenge path authorized by signature or verification token/encryptKey, not a broad unsigned webhook bypass.
