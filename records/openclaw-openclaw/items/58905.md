## Summary

Keep open. Current main and the latest release still reject unsigned Feishu/Lark setup-time `url_verification` requests before challenge handling, and the remaining work is already represented by the focused open fix path at https://github.com/openclaw/openclaw/pull/60485.

## What This Changes

The issue reports that Feishu/Lark webhook URL verification fails because encrypted setup challenge requests are rejected with `401 Invalid signature` when `encryptKey` is configured.

## Best Possible Solution

Land or replace the focused Feishu/Lark fix so token-verified `url_verification` challenges can complete setup while non-challenge webhook events remain signature-gated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an unsigned Feishu/Lark `url_verification` request reaches the signature gate, fails for missing Lark signature headers, and returns `401` before challenge parsing/decryption.

## Solution Assessment

Yes. The narrow maintainable solution is a phase-aware challenge exception that verifies the configured token for `url_verification` while leaving ordinary events on the strict signature path.
