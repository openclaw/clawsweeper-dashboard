## Summary

Keep this issue open: current main and the shipped v2026.6.10 release still reject missing-signature Feishu/Lark webhook verification requests before challenge handling, and the direct fix PR attempts are closed unmerged with no active replacement PR found.

## What This Changes

The issue reports that Feishu/Lark webhook URL verification cannot complete because encrypted setup challenges arrive without signature headers while OpenClaw requires signatures before challenge handling.

## Best Possible Solution

A narrow Feishu plugin fix should detect encrypted setup-time url_verification challenges before normal signature rejection, validate the decrypted verification token, and continue rejecting unsigned normal event deliveries.

## Reproduction Assessment

Yes. Source inspection shows webhook mode requires encryptKey, missing signatures return 401 before challenge handling, and the current e2e test asserts that missing-signature url_verification requests fail.

## Solution Assessment

Yes for the direction. The maintainable fix is phase-aware challenge handling with token validation, not a broad unsigned-webhook bypass.
