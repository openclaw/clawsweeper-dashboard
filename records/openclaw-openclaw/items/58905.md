## Summary

Keep open. Current main still rejects unsigned Feishu/Lark `url_verification` requests before challenge handling, latest release `v2026.6.6` has the same ordering, and the focused repair at https://github.com/openclaw/openclaw/pull/60485 is still open/conflicting rather than a merged fix.

## What This Changes

Feishu/Lark webhook mode should complete the initial encrypted `url_verification` handshake without weakening signature checks for normal webhook events.

## Best Possible Solution

Land or replace a focused Feishu plugin patch that handles setup-time `url_verification` before normal event dispatch only when the challenge is authenticated by the documented token/decryption path, while preserving strict signature validation for non-challenge traffic.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: webhook mode requires `encryptKey`, unsigned `url_verification` requests fail signature validation before challenge handling, and current e2e source asserts the `401` response.

## Solution Assessment

Yes. A phase-aware Feishu plugin challenge path is the right fix shape; a broad unsigned bypass would be unsafe, and the current open PR still needs refresh plus security/proof clearance.
