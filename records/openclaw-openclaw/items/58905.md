## What This Changes

The issue reports that Feishu/Lark webhook URL verification returns 401 when `encryptKey` is configured because the initial challenge request lacks signature headers.

## Best Possible Solution

Land or replace a focused Feishu plugin patch that handles setup-time `url_verification` before normal event dispatch only when the challenge is authenticated by the documented token/decryption path, while preserving strict signature validation for non-challenge traffic.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: webhook mode requires `encryptKey`, unsigned `url_verification` requests fail signature validation before challenge handling, and current e2e source asserts the `401` response.

## Solution Assessment

Yes, a phase-aware Feishu plugin challenge path is the right fix shape. The current open PR is a plausible candidate, but it must be refreshed and security/proof-cleared before it can resolve this issue.
