## Summary

Keep open. Current main and the latest release still reject unsigned Feishu/Lark URL verification requests before challenge handling, and the related fix PR is open but conflicting and not security/proof-cleared.

## What This Changes

The issue reports that Feishu/Lark webhook URL verification is rejected with 401 when `encryptKey` is configured because the setup challenge arrives without `X-Lark-Signature`.

## Best Possible Solution

Land or replace the focused Feishu plugin fix so URL verification challenges are accepted through a narrow phase-aware path while normal event deliveries still require signature validation.

## Reproduction Assessment

Yes. Current main validates signatures before payload parsing or `generateChallenge`, and the adjacent e2e test expects unsigned `url_verification` requests to return 401.

## Solution Assessment

Yes, the best direction is a narrow phase-aware Feishu plugin fix that authenticates setup-time `url_verification` challenges without relaxing normal webhook event signature validation.
