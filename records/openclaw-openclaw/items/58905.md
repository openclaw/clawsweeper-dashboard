## Summary

Keep this issue open: current main and v2026.6.10 still reject Feishu/Lark webhook requests with missing signature headers before challenge handling, and the direct fix PRs are closed unmerged with no viable open replacement.

## What This Changes

The issue requests Feishu/Lark webhook mode to accept encrypted URL-verification challenges without `X-Lark-Signature` when `encryptKey` is configured, while preserving normal event authentication.

## Best Possible Solution

Add a narrow Feishu plugin fix that decrypts or parses setup-time encrypted `url_verification` challenges before normal signature rejection, validates the decrypted verification token, and keeps unsigned non-challenge events on the signature-required path.

## Reproduction Assessment

Yes. Source inspection shows webhook mode requires `encryptKey`, missing signatures return `401` before `Lark.generateChallenge`, and the current e2e test asserts that missing-signature `url_verification` requests fail.

## Solution Assessment

Yes for the requested direction, but only with a narrow security shape. The best fix is phase-aware challenge handling with decrypted token validation, not a general unsigned-webhook bypass.
