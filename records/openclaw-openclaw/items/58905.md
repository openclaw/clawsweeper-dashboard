## Summary

Keep open. Current main and the latest release still reject unsigned Feishu/Lark `url_verification` requests before challenge handling, and the focused open fix path is https://github.com/openclaw/openclaw/pull/60485 rather than closing this issue now.

## What This Changes

The issue asks OpenClaw to let Feishu/Lark webhook URL-verification challenges complete when `encryptKey` is set but the setup request lacks Lark signature headers.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/60485 with a current-main Feishu plugin fix that accepts only token/encryptKey-verified setup challenges, keeps normal events signature-gated, and includes regression coverage plus redacted live Lark proof.

## Reproduction Assessment

Yes. Sending an unsigned JSON `url_verification` POST with `verificationToken` and `encryptKey` configured follows the current source path to `401 Invalid signature` before challenge parsing, and current e2e coverage explicitly expects that failure.

## Solution Assessment

Yes for the issue direction. The maintainable fix is a narrow phase-aware challenge path authorized by signature or verification token/encryptKey, not a broad unsigned webhook bypass.
