## Summary

Keep open. Current main still rejects unsigned Feishu/Lark `url_verification` requests before challenge handling, and the focused open PR has not landed.

## What This Changes

The issue asks Feishu webhook mode to accept initial unsigned URL-verification challenges when `encryptKey` is configured without relaxing normal event signature validation.

## Best Possible Solution

Land or replace the focused Feishu plugin fix so verified URL-verification challenges are handled before normal event dispatch while preserving pre-auth guards and strict signatures for non-challenge events.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST an unsigned plaintext or encrypted `url_verification` body; current main returns `401` before `Lark.generateChallenge()`.

## Solution Assessment

Yes. The narrow maintainable direction is phase-aware challenge handling authorized by signature or verification token, not a broad unsigned webhook bypass.
