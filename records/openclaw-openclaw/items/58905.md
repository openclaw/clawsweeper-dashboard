## Summary

Keep open. Current main and v2026.6.5 still reject unsigned Feishu/Lark URL verification requests before challenge handling, and the focused remaining implementation path is the open fix PR at https://github.com/openclaw/openclaw/pull/60485.

## What This Changes

Feishu/Lark webhook mode should complete encrypted URL verification requests that lack `X-Lark-Signature` without weakening normal event signature validation.

## Best Possible Solution

Land or replace the focused Feishu webhook fix so only verified `url_verification` challenges can bypass the normal signature gate, while ordinary event deliveries still require valid Feishu signatures.

## Reproduction Assessment

Yes, source-reproducible without a live Feishu run: current main requires signature headers before parsing or `generateChallenge`, so an unsigned encrypted challenge body reaches `401 Invalid signature` first.

## Solution Assessment

Yes, a phase-aware challenge path is the best direction. The open fix PR at https://github.com/openclaw/openclaw/pull/60485 is the canonical implementation candidate, but it still needs security-boundary review and proof before merge.
