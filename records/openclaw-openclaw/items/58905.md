## Summary

Keep open. Current main still rejects unsigned Feishu/Lark `url_verification` requests before parsing or challenge handling, and the discussion plus related open PR show this is a real handshake-phase/auth-boundary bug rather than stale support noise.

## What This Changes

Feishu/Lark webhook mode should complete setup-time URL verification when a valid plaintext or encrypted challenge arrives without normal Lark signature headers.

## Best Possible Solution

Land or replace the current Feishu plugin fix so only verified `url_verification` challenges are handled before normal event dispatch, while preserving current pre-auth guards and strict non-challenge signature validation.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST a plaintext or encrypted `url_verification` body without Lark signature headers; current main reaches the `401 Invalid signature` branch before `Lark.generateChallenge()`.

## Solution Assessment

Yes. The narrow maintainable direction is phase-aware challenge handling with verification-token or signature validation, not a broad unsigned webhook bypass.
