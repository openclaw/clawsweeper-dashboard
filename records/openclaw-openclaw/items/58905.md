## Summary

Keep open. Current main and v2026.6.6 still reject unsigned Feishu/Lark `url_verification` requests before challenge handling, and the focused open repair PR is still conflicting and security/proof blocked rather than a merged fix.

## What This Changes

The issue reports Feishu/Lark webhook setup failing URL verification because unsigned encrypted challenge requests are rejected with `401 Invalid signature` when `encryptKey` is configured.

## Best Possible Solution

Land a focused Feishu plugin patch that handles setup-time `url_verification` before normal event dispatch only when the challenge is authenticated by the documented token/decryption path, while preserving strict signature validation for non-challenge traffic.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: webhook mode requires `encryptKey`, unsigned `url_verification` requests fail signature validation before challenge handling, and current e2e coverage asserts the `401` response.

## Solution Assessment

Yes, the phase-aware Feishu plugin challenge path is the best fix shape. The existing open PR is a plausible candidate, but it must be refreshed and security/proof-cleared before it can resolve this issue.
