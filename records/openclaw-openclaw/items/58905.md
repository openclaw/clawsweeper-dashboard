## Summary

Keep open. Current main still performs Feishu webhook signature validation before parsing the body or calling `Lark.generateChallenge`, so an unsigned `url_verification` request is rejected with `401 Invalid signature`; the current e2e test suite even asserts that behavior for missing signature headers. Related PR #60485 is an open fix candidate, while earlier PRs #58938 and #59990 were closed unmerged.

## What This Changes

The issue requests fixing Feishu/Lark webhook URL verification so unsigned encrypted `url_verification` challenge requests can be decrypted and answered when webhook mode has `encryptKey` set.

## Best Possible Solution

Land a narrow Feishu plugin fix that recognizes `url_verification` challenge payloads before the normal event-delivery signature gate, validates the setup-time challenge with the configured token/decryption semantics, and adds regression coverage for unsigned plaintext and encrypted challenges plus unsigned non-challenge rejection.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: configure Feishu webhook mode with `encryptKey`, send a JSON `url_verification` request without Lark signature headers, and current main returns `401` before `Lark.generateChallenge`; the existing e2e test asserts the same missing-signature outcome.

## Solution Assessment

Yes, the phase-aware direction described in the issue comments and open PR #60485 is the right shape. A broad unsigned webhook bypass would be unsafe; the maintainable fix is limited to verified `url_verification` setup requests while keeping normal event signature validation strict.
