## Summary

Keep this issue open: current main and the latest release still reject missing-signature Feishu/Lark webhook challenge requests before challenge handling, and the open linked PR is the active fix candidate rather than a reason to close the report early.

## What This Changes

The issue reports that Feishu/Lark webhook URL verification fails because unsigned encrypted challenge requests are rejected with `401 Invalid signature` when `encryptKey` is configured.

## Best Possible Solution

A narrow Feishu plugin fix should recognize setup-time `url_verification` challenges before normal signature rejection, validate the verification token after decrypting the challenge payload, and keep unsigned normal event delivery rejected.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: webhook mode requires `encryptKey`, current main rejects missing signature headers before `generateChallenge`, and the current e2e test asserts 401 for a missing-signature `url_verification` request.

## Solution Assessment

Yes for the direction. The best fix is a phase-aware challenge path for verified URL verification requests, not a broad unsigned webhook bypass; the linked PR should be reviewed or repaired on that boundary.
