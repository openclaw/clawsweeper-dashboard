## Best Possible Solution

Re-port only identity-safe canonicalization onto the current resolver, preserving current auth/session semantics and remapping no-identity defaults only when the alias is unique or otherwise provably equivalent.

## Solution Assessment

No, not as currently proposed. The canonicalization direction is useful, but it should be re-ported onto current main and tightened so no-identity defaults do not remap to arbitrary OAuth aliases.
