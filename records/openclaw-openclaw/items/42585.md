## Summary

Keep open. The reported auth-profile drift is still a real gap on current main, but the PR head is stale against later auth-routing work and would regress explicit user credential selection and provider alias handling if merged as-is.

## What This Changes

The PR adds resolver-side canonicalization of automatic legacy OAuth `provider:default` session auth-profile overrides to non-default OAuth aliases, plus regression tests for identity matching, non-OAuth defaults, and rotation progress.

## Best Possible Solution

Port the canonicalization into current `src/agents/auth-profiles/session-override.ts` on top of main, reusing the existing OAuth identity helpers and preserving the no-source fast path, provider auth alias resolution, explicit user override behavior, cooldown handling, and rotation progress. Add the PR's negative identity/non-OAuth/rotation tests in the current test harness and include the required changelog entry.

## Reproduction Assessment

Yes. Create an auth store with `openai-codex:default` OAuth plus a matching non-default OAuth alias, set an automatic session `authProfileOverride` to `openai-codex:default`, and call `resolveSessionAuthProfileOverride`; current main leaves the legacy default because the resolver never canonicalizes it.

## Solution Assessment

No, not as currently branched. Resolver-side canonicalization is the right narrow approach, but this head must be rebased or ported so it preserves current-main auth alias and explicit user override semantics.
