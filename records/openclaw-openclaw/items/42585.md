## Summary

Keep open. Current main still does not implement legacy OAuth `provider:default` session-override canonicalization, but this PR branch is stale relative to current main and would regress existing auth-profile safeguards around explicit user overrides and provider aliases if merged as-is.

## What This Changes

The PR adds resolver-side canonicalization of automatic legacy OAuth `provider:default` auth-profile session overrides to non-default OAuth aliases, with regression tests for identity mismatch, non-OAuth defaults, and rotation progress.

## Best Possible Solution

Port the canonicalization into current `src/agents/auth-profiles/session-override.ts` instead of merging the stale branch directly: reuse or align with the existing OAuth identity helpers, preserve the no-source fast path, provider auth alias resolution, explicit user override behavior, cooldown handling, and rotation progress, then carry forward the PR’s negative identity and non-OAuth regression tests.

## Reproduction Assessment

Yes. The reported legacy-default behavior is reproducible by the PR’s fixture: an automatic `openai-codex:default` override remains unchanged on current main even when a matching non-default OAuth alias is present, because the resolver never canonicalizes it.

## Solution Assessment

No, not as currently branched. Resolver-side canonicalization is the right narrow place to solve the requested bug, but this PR must be rebased or ported onto current main so it preserves existing user override and provider alias behavior.
