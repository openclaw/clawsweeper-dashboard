## Summary

Keep open. Current main has adjacent diagnostics and safer reinstall hardening, but the update path still discovers and refreshes the managed service from the updater environment, while forced reinstall still drops prior service control roots. The reported cross-HOME package update can still abandon a dedicated-user config/state root instead of preserving, migrating, warning, or blocking.

## What This Changes

The issue requests that Linux package updates preserve, migrate, or warn/block when a managed gateway service refresh changes HOME and would abandon the existing config/state root.

## Best Possible Solution

Add a Linux package-update/service-refresh guard that discovers the previously installed managed gateway root before rewriting it, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH against the updater environment, and either preserves the old root when safe or emits a clear blocking migration warning.

## Reproduction Assessment

Yes. The report gives concrete update steps, and current source inspection provides a high-confidence static reproduction path: service discovery is HOME-keyed and starts from process.env, so a root-invoked refresh can miss a dedicated-user unit/config root.

## Solution Assessment

Not applicable as there is no proposed patch on this issue. The requested preserve/migrate/warn direction is right, but the exact policy needs maintainer/security review to avoid undoing stale-env hardening.
