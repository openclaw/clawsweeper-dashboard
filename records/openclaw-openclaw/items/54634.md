## Summary

Keep open: current main and the latest release still start managed-service discovery from the updater process environment, while systemd user-unit lookup is HOME-keyed and reinstall intentionally drops prior HOME/OPENCLAW_* roots, so the reported cross-HOME config abandonment is not proven fixed.

## What This Changes

The issue reports that updating from 2026.3.23-2 to 2026.3.24 changed the Linux service HOME and abandoned the existing OpenClaw config/state root.

## Best Possible Solution

Add a Linux update/service-refresh guard that discovers the previously installed gateway service root before rewriting, compares HOME/OPENCLAW_HOME/OPENCLAW_STATE_DIR/OPENCLAW_CONFIG_PATH, and either preserves the prior root safely or blocks with doctor-backed migration guidance.

## Reproduction Assessment

Yes, source-level: current main still reads service state from process.env before the previous service env is known, and systemd user-unit lookup follows HOME. I did not run a live cross-user systemd upgrade in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility and privacy policy. The narrow durable fix should handle cross-HOME service roots before update or reinstall writes new config/service state, without reintroducing unsafe stale service-env reuse.
