## Label Justifications

- `P1`: A package update can break an existing managed gateway and make the prior config/state appear lost when service HOME changes.
- `impact:data-loss`: The report centers on abandoning a complete existing config/state directory and creating a sparse replacement under another HOME.
- `impact:crash-loop`: The reported update left the gateway inaccessible with stale competing processes and missing effective gateway configuration.

## Summary

Keep open: current main has adjacent update-mode systemd hardening, but it still does not handle the reported cross-HOME package-update path before service discovery, doctor, or restart logic run from the updater process environment.

## What This Changes

The issue reports that upgrading from 2026.3.23-2 to 2026.3.24 changed the Linux gateway service HOME from /home/openclaw to /root, abandoning the existing config/state and breaking the gateway.

## Best Possible Solution

Add a Linux package-update guard that identifies the previously installed managed gateway root before doctor/restart/refresh, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH, then either safely preserves the prior root or blocks with explicit migration guidance.

## Reproduction Assessment

Yes, at source level: install a managed Linux systemd gateway under HOME=/home/openclaw, then invoke package update with HOME=/root. Current code still discovers the service from process.env while the systemd unit path is HOME-keyed; I did not run that destructive cross-user flow.

## Solution Assessment

Unclear because there is no patch on this issue. The requested preserve, migrate, warn, or block direction is correct, but maintainers need to choose the service-root and credential-root policy before implementation.
