## Summary

Keep open: current main still leaves the reported cross-HOME update path unresolved, and the open related PR only addresses adjacent `moltbot.json` recovery rather than service HOME/root migration.

## What This Changes

The issue reports that upgrading to 2026.3.24 changed the service HOME from a dedicated OpenClaw user to root, abandoning the existing config/state and starting the gateway from a skeletal root config.

## Best Possible Solution

Add a Linux update/service-refresh guard that identifies the previously installed gateway service root before rewriting, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH against the updater environment, and either preserves the prior root when safe or emits a clear migration/blocking warning.

## Reproduction Assessment

Yes at source level: the report gives concrete upgrade steps, and current code still has HOME/env-keyed service discovery plus forced reinstall behavior that drops the prior service roots. I did not run a live cross-user systemd upgrade in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility and privacy policy. The durable fix should prevent cross-HOME config abandonment without broadly reusing stale service env or leaking sibling-user state.
