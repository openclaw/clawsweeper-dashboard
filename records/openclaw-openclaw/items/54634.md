## Summary

Keep open: current main has useful last-known-good config recovery and service-env hardening, but it still does not prove that a Linux update run under a different HOME can find, preserve, migrate, or block before abandoning the prior service config root.

## What This Changes

The issue reports that upgrading from 2026.3.23-2 to 2026.3.24 changed the systemd HOME from /home/openclaw to /root, abandoned the existing config/state, and broke the gateway.

## Best Possible Solution

Add a Linux update/service-refresh guard that discovers the previously installed gateway service root before rewriting, compares HOME/OPENCLAW_HOME/OPENCLAW_STATE_DIR/OPENCLAW_CONFIG_PATH, and either preserves the prior root safely or blocks with doctor-backed migration guidance.

## Reproduction Assessment

Yes, source-level: systemd user-unit lookup derives from current HOME, update/refresh reads service state from process.env, and reinstall drops prior HOME/OPENCLAW_* roots. I did not run a live cross-user systemd upgrade in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility and privacy policy. The narrow durable fix should handle cross-HOME service roots before update or reinstall writes new config/service state, without reintroducing unsafe stale service-env reuse.
