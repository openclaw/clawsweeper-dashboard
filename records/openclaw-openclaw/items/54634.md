## Label Justifications

- `P1`: A version update can break an existing managed gateway and make the user's prior config/state appear lost after the service HOME changes.
- `impact:data-loss`: The report centers on OpenClaw abandoning an existing config/state directory and creating a sparse replacement under another HOME.
- `impact:crash-loop`: The reported update left the gateway inaccessible with competing/stale service processes and a missing effective gateway config.

## Summary

Keep open. Current main and v2026.5.18 still start managed-service discovery from the updater process environment, so a root-run update can miss the prior user-HOME systemd unit before any preserve/migrate/warn safeguard applies; newer related issues cover pieces of the problem, but no merged fix owns this cross-HOME update path end to end.

## What This Changes

The issue reports that upgrading from 2026.3.23-2 to 2026.3.24 changed the managed gateway HOME from /home/openclaw to /root and abandoned the prior config/state without migration or warning.

## Best Possible Solution

Add a Linux package-update guard that identifies the previously installed managed gateway root before rewriting or restarting it, compares HOME and OPENCLAW_* roots, and either preserves the prior root when safe or blocks with explicit migration guidance.

## Reproduction Assessment

Yes, at source level: install the managed systemd gateway under HOME=/home/openclaw, then invoke update with HOME=/root and the current code will derive the user-unit path and config/state roots from the updater environment. I did not run the destructive cross-user systemd flow in this read-only review.

## Solution Assessment

Unclear because there is no patch on this issue. The preserve/migrate/warn direction is correct, but maintainers need to choose the service-root and credential-root policy before implementation.
