## Label Justifications

- `P1`: A package update can break an existing managed gateway and make the previous config/state appear lost when service HOME changes.
- `impact:data-loss`: The issue centers on abandoning a complete config/state directory and creating or using a sparse replacement under another HOME.
- `impact:crash-loop`: The reported update left the gateway inaccessible with competing processes, missing effective gateway config, and token mismatch symptoms.

## Summary

Keep open: current main has partial service-env preservation, but the cross-HOME package-update path still starts discovery from the updater process environment and can miss the old user-systemd unit before it can preserve, migrate, or warn.

## What This Changes

The issue asks OpenClaw update to preserve, migrate, warn, or block when a package update changes the managed gateway service HOME and abandons the previous config/state root.

## Best Possible Solution

Add a Linux package-update guard that identifies the previously installed managed gateway root before service refresh, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH, then preserves the prior root when safe or blocks with explicit migration guidance and live upgrade proof.

## Reproduction Assessment

Yes, at source level: install a managed Linux systemd gateway under HOME=/home/openclaw, then invoke package update with HOME=/root. I did not run that destructive cross-user flow, but current code still shows the HOME-keyed lookup and process.env starting point.

## Solution Assessment

Unclear because there is no patch on this issue. The requested preserve, migrate, warn, or block direction is correct, but maintainers need to choose the service-root and credential-root policy before implementation.
