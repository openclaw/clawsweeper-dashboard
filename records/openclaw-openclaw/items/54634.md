## Summary

Keep open. Current main has adjacent diagnostics and service-env hardening, but it still lacks the reported cross-HOME update safeguard: update/service refresh starts from the updater process environment, Linux user-unit lookup remains HOME-keyed, and forced reinstall drops prior HOME and OPENCLAW_* control roots rather than preserving, migrating, warning, or blocking when a dedicated-user install is refreshed from root.

## What This Changes

The issue asks OpenClaw update/service refresh to avoid abandoning a dedicated-user config and state root when the updater runs under a different HOME.

## Best Possible Solution

Add a Linux update/service-refresh guard that identifies the previously installed managed gateway service root before rewriting it, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH against the updater environment, and either preserves the prior root when safe or stops with a clear migration warning. Cover the dedicated-user service refreshed from root case across update, gateway install, systemd, and doctor/status diagnostics; treat GHCR version-tag rollback as separate work.
