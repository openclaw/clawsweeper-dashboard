## Summary

Keep open: current main and v2026.6.10 still leave the reported cross-HOME update path unresolved, and the open related PR only addresses the adjacent sibling `moltbot.json` migration case, not service HOME/root migration.

## What This Changes

The issue reports that an update changed the managed Linux systemd service HOME from a dedicated OpenClaw user to root, abandoning the existing config/state and starting from a sparse replacement config.

## Best Possible Solution

Add a Linux update/service-refresh guard that discovers the previously installed gateway service root before rewriting, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH, and either preserves the prior root safely or blocks with doctor-backed migration guidance.

## Reproduction Assessment

Yes at source level: current main still starts service discovery from the updater process environment, systemd user-unit lookup follows HOME, and reinstall intentionally drops prior HOME/OPENCLAW_* roots. I did not run a live cross-user systemd upgrade in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility and privacy policy. The durable fix should handle cross-HOME service roots before update or reinstall writes new config/service state, without reintroducing unsafe stale service-env reuse.
