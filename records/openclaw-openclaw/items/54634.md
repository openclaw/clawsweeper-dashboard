## Label Justifications

- `P1`: A released update path can break an existing managed gateway and make the user's prior config/state appear lost after the service HOME changes.
- `impact:data-loss`: The report centers on OpenClaw abandoning an existing config/state directory and creating a sparse replacement under another HOME.
- `impact:crash-loop`: The reported update left the gateway inaccessible with stale/competing service processes and a missing effective gateway config.

## Summary

Keep open. Current main and v2026.5.18 still derive managed-service discovery from the updater process environment, so a root-run update can miss a prior dedicated-user systemd unit before any preserve, migrate, warn, or block policy is applied.

## What This Changes

The issue reports that updating from 2026.3.23-2 to 2026.3.24 changed the managed gateway service HOME and caused existing config/state under a dedicated user to be abandoned.

## Best Possible Solution

Add a Linux package-update guard that identifies the previously installed managed gateway root before rewriting or restarting it, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH, and then preserves, migrates, or blocks with explicit guidance according to maintainer policy.

## Reproduction Assessment

Yes, at source level: install the managed systemd gateway under HOME=/home/openclaw, then invoke update with HOME=/root and current code derives the service unit and config/state roots from process.env. I did not run the destructive live cross-user systemd flow.

## Solution Assessment

Unclear because there is no patch on this issue. The requested preserve, migrate, or warn direction is correct, but maintainers need to choose the service-root and credential-root policy before implementation.
