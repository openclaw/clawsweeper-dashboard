## Label Justifications

- `P1`: A package update can break an existing managed gateway and make the prior config/state appear lost when the service HOME changes.
- `impact:data-loss`: The report centers on abandoning an existing complete config/state directory and creating a sparse replacement under another HOME.
- `impact:crash-loop`: The reported update left the gateway inaccessible with stale competing processes and missing effective gateway configuration.

## Summary

Keep open: current main and v2026.5.18 still discover and refresh the managed gateway from the updater process environment, so a root-run update can miss a prior dedicated-user systemd unit before any cross-HOME preserve, migrate, warn, or block safeguard applies.

## What This Changes

The issue asks OpenClaw update to preserve, migrate, or warn/block when a package update changes the managed gateway HOME/config/state root.

## Best Possible Solution

Add a Linux package-update guard that identifies the previously installed managed gateway root before rewriting or restarting it, compares HOME/OPENCLAW_HOME/OPENCLAW_STATE_DIR/OPENCLAW_CONFIG_PATH, and either safely preserves the prior root or blocks with explicit migration guidance.

## Reproduction Assessment

Yes, at source level: install the managed Linux systemd gateway under HOME=/home/openclaw, then invoke package update with HOME=/root and the current updater derives discovery and config roots from process.env. I did not run that destructive cross-user systemd flow in this read-only review.

## Solution Assessment

Unclear because there is no patch on this issue. The requested preserve, migrate, or warn/block direction is correct, but maintainers need to choose the service-root and credential-root policy before implementation.
