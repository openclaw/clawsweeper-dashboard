## Summary

Keep open. Current main has adjacent diagnostics and config recovery, but it still lacks a cross-HOME package-update safeguard that discovers or preserves the previously installed managed gateway service root before a root-invoked refresh rewrites service environment and abandons a dedicated-user config/state directory.

## What This Changes

The issue requests an update-time safeguard for Linux systemd installs where upgrading from a dedicated user to HOME=/root silently drops the existing OpenClaw config and state.

## Best Possible Solution

Add a Linux package-update/service-refresh guard that discovers the previously installed managed gateway service root before rewriting it, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH against the updater environment, and either safely preserves the old root or emits a clear blocking migration warning. Cover the dedicated-user service refreshed from HOME=/root case across update, gateway install, systemd, and doctor/status regression tests.

## Reproduction Assessment

Yes. The report gives concrete steps, and current source inspection provides a high-confidence static reproduction path: user-unit discovery is HOME-keyed and both pre/post package-update service reads start from process.env, so HOME=/root will miss a service/config rooted under /home/openclaw.

## Solution Assessment

Not applicable as there is no proposed patch on this issue. The requested preserve/migrate/warn direction is the right end state, but the exact policy should be chosen with maintainer/security review because it touches service environment and config/secrets-adjacent roots.
