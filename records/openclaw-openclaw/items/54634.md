## Summary

Keep open: current main has adjacent package-root and service-env hardening, but it still does not prove a cross-HOME Linux systemd update can find, preserve, migrate, or block before abandoning the prior service config root.

## What This Changes

The issue reports that updating from 2026.3.23-2 to 2026.3.24 changed the managed gateway HOME from /home/openclaw to /root, leaving the old config/state behind and starting from a sparse new config.

## Best Possible Solution

Add a Linux update/service-refresh guard that discovers the previously installed gateway service root before rewriting it, compares HOME/OPENCLAW_HOME/OPENCLAW_STATE_DIR/OPENCLAW_CONFIG_PATH, and either safely preserves the prior root or blocks with clear doctor-backed migration guidance.

## Reproduction Assessment

Yes, source-level: a user-scope systemd unit is resolved from the caller HOME, while update root detection and service-env capture also start from process.env. I did not run a live cross-user systemd scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. The current partial safeguards are useful but the best fix should explicitly handle cross-HOME service roots before update/reinstall writes new config or service state.
