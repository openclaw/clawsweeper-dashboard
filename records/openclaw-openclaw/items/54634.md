## Label Justifications

- `P1`: A package update can break an existing managed gateway and make the prior config/state appear lost when the service HOME changes.
- `impact:data-loss`: The report centers on abandoning an existing config/state directory and creating a sparse replacement under another HOME.
- `impact:crash-loop`: The reported update left the gateway inaccessible with stale competing processes and a missing effective gateway config.

## Summary

Keep open: current main and v2026.5.18 still discover the managed gateway service from the updater process environment, so a root-run package update can miss a prior dedicated-user systemd unit before any preserve, migrate, warn, or block safeguard applies.

## What This Changes

The issue requests an update safeguard so package updates do not abandon a dedicated-user gateway config/state directory when the updater runs under a different HOME.

## Best Possible Solution

Add a Linux package-update guard that identifies the previously installed managed gateway root before rewriting or restarting it, compares HOME and OPENCLAW_* roots, and preserves the prior root when safe or blocks with explicit migration guidance.

## Reproduction Assessment

Yes, at source level: install the managed systemd gateway under HOME=/home/openclaw, then invoke package update with HOME=/root and current code derives service discovery and config/state roots from the updater environment. I did not run the destructive cross-user systemd flow in this read-only review.

## Solution Assessment

Unclear because there is no patch on this issue. The requested preserve, migrate, or warn direction is correct, but maintainers need to choose the service-root and credential-root policy before implementation.
