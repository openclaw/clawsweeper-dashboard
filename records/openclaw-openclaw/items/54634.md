## Label Justifications

- `P1`: A package update can break an existing managed gateway and make the prior config/state appear lost when service HOME changes.
- `impact:data-loss`: The report centers on abandoning a complete existing config/state directory and creating a sparse replacement under another HOME.
- `impact:crash-loop`: The reported update left the gateway inaccessible with stale competing processes and missing effective gateway configuration.

## Summary

Keep open: current main has a merged adjacent fix for preserving already-captured package service env, but the reported cross-HOME Linux update path can still miss the old user systemd unit before any preserve/migrate/warn policy runs.

## What This Changes

The issue asks OpenClaw updates to preserve, migrate, or clearly block/warn when a Linux systemd gateway update changes HOME and would abandon an existing config/state directory.

## Best Possible Solution

Add a Linux package-update guard that identifies the previously installed managed gateway root before service refresh, compares HOME and OpenClaw root/config/state env, then preserves the old root when safe or blocks with explicit migration guidance.

## Reproduction Assessment

Yes, at source level: install a Linux user systemd gateway under HOME=/home/openclaw, then run a package update with HOME=/root. Current main still begins service discovery from process.env and the systemd unit path is HOME-keyed, so the old service env may never be captured.

## Solution Assessment

Unclear because there is no patch for the remaining cross-HOME path. The preserve, migrate, warn, or fail-closed direction is correct, but maintainers need to choose the root-preservation and credential-boundary policy before implementation.
