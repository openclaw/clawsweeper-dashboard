## Summary

Keep open: current main and v2026.6.10 still lack a `memory.dream.started`/`interrupted` lifecycle contract, and the related open work only covers overlapping pieces such as completed-event outcome normalization and stale child-session cleanup.

## What This Changes

The issue requests dreaming-phase crash recovery that records phase starts, detects orphaned runs on startup, emits interrupted outcomes, and marks partial inline memory writes.

## Best Possible Solution

Define an additive memory-core dreaming run lifecycle contract, keep existing completed-event readers compatible, and coordinate startup orphan handling with the existing cron and stale-session cleanup work.

## Reproduction Assessment

Yes, at source level: current main has only a completed dream event shape and no `memory.dream.started` or interrupted recovery path. I did not run a live process-kill, OOM, or timeout reproduction.

## Solution Assessment

Unclear as a final design: the proposed start/orphan/interrupted event shape is plausible, but the public event taxonomy and partial-write recovery policy need maintainer approval before a narrow implementation PR.
