## Summary

Keep open. Current main still does not archive ACP-created Discord threads on `/acp close`, and this PR remains useful but needs rework for the current Discord plugin architecture before merge.

## What This Changes

The branch adds Discord ACP-thread ownership metadata, `/acp close` archive-after-reply signaling, Discord thread archive delivery, and related command/outbound/thread-binding tests.

## Best Possible Solution

Add an additive channel conversation-binding close hook or typed reply-payload contract, preserve ACP-spawned child-thread ownership in generic binding metadata, and implement archive-after-reply plus partial-failure messaging inside the Discord plugin with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence path: `/acp close` closes and unbinds only, while the active Discord outbound payload path has no archive-after-reply handling, so an ACP-created Discord thread remains unarchived after the command.

## Solution Assessment

No. The requested behavior is useful, but this PR is not the best current implementation because it targets retired `src/discord`/legacy outbound paths and embeds Discord policy in ACP core; the safer fix is a generic close hook with Discord-owned archive delivery.
