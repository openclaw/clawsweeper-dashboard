## Summary

Keep open. Current main still closes and unbinds ACP sessions without archiving ACP-created Discord threads, but this branch is not merge-ready because it targets retired Discord paths and puts Discord-specific close behavior in ACP core instead of the Discord plugin boundary.

## What This Changes

The branch records ACP-created Discord thread ownership metadata and attempts to archive those threads after `/acp close` replies, with command, outbound, thread-binding, and test changes.

## Best Possible Solution

Add an additive channel/conversation-binding close hook or typed reply-payload contract, keep ACP core extension-agnostic, preserve ACP-owned child-thread metadata through the generic binding metadata path, and implement archive-after-reply plus partial-failure messaging inside the Discord plugin with docs, changelog, and focused tests.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: `/acp close` only closes and unbinds, while the active Discord plugin payload path has no archive-after-reply handling.

## Solution Assessment

No. The requested behavior is reasonable, but this PR is not the best current implementation because it targets retired paths and embeds Discord policy in ACP core; the safer fix is a generic seam with Discord-owned archive delivery.
