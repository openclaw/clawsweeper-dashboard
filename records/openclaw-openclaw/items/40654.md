## Summary

Keep open. The requested Discord cleanup behavior is useful and still missing on current main, but this PR is not merge-ready because it patches retired Discord paths and puts Discord-specific close/archive policy in ACP core instead of the active Discord plugin boundary.

## What This Changes

The PR records ACP-created Discord thread ownership metadata and attempts to archive those threads after `/acp close` closes and unbinds the ACP session.

## Best Possible Solution

Add an additive channel/conversation-binding close hook or typed reply-payload contract, preserve ACP-owned child-thread metadata through the existing binding metadata path, and implement archive-after-reply inside `extensions/discord` with unchanged non-Discord and manually attached thread behavior.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence path: `/acp close` closes and unbinds only, while the active Discord outbound/plugin code has no archive-after-reply handler.

## Solution Assessment

No. The requested outcome is reasonable, but this PR is not the best implementation because it targets retired Discord paths and embeds Discord-specific behavior in ACP core; a plugin-owned implementation behind a generic seam is safer.
