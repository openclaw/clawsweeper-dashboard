## Summary

Keep open. Current main still has the iMessage RPC stdin write without local write-error or stream-error handling, and the issue is now paired with open PR #75473 using closing syntax, so cleanup should wait for that PR to merge or be closed. The global uncaught-EPIPE backstop reduces the crash blast radius, but it does not make the iMessage request path reject promptly and cleanly.

## What This Changes

The issue asks OpenClaw to handle broken-pipe and asynchronous stdin write failures in the iMessage RPC child-process client so pending requests fail cleanly instead of escaping through process-level exception handling.

## Best Possible Solution

Review and land #75473 or an equivalent narrow patch that adds local `child.stdin` error handling in `extensions/imessage/src/client.ts`, rejects and clears pending requests on write callback errors, adds focused iMessage regression tests, and keeps the global uncaught-EPIPE handler as a last-resort backstop.

## Reproduction Assessment

Yes. Current main shows the missing local handling in the iMessage client, and an inline Node Writable probe reproduced that a write callback error can still emit an uncaught stream error unless a stream `error` listener is attached; a live `imsg rpc` run was not needed to confirm the code-level gap.

## Solution Assessment

Yes in direction. Local stdin failure handling in the iMessage RPC client is the narrow maintainable fix; relying on the global uncaught-EPIPE suppression alone is only a process-level fallback and leaves request-level behavior degraded.
