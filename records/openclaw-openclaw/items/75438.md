## Summary

Keep open. Current main still has the iMessage RPC request-level gap: `request()` writes to child stdin without a callback, and `start()` does not attach a stdin `error` listener. The global benign-EPIPE handler reduces the original gateway-exit severity, but it does not make pending iMessage RPC requests reject cleanly, and the linked closing PR is still open rather than merged.

## What This Changes

The issue asks for iMessage RPC child-stdin write failures to reject pending requests instead of surfacing as uncaught EPIPE exceptions that can crash or destabilize the gateway.

## Best Possible Solution

Review and land #75473 or an equivalent narrow fix that keeps this behavior owned by the iMessage plugin: add a stdin `error` listener that fails pending requests and closes the client state, make write callback errors reject the affected pending request immediately, and keep focused regression coverage plus the changelog entry.

## Reproduction Assessment

Yes. Current-main source shows the unguarded iMessage stdin write and missing stdin error listener, and the adjacent Codex app-server test demonstrates the same EPIPE stream-failure class; a live long-running gateway incident was not repeated in this read-only pass.

## Solution Assessment

Yes. Local iMessage RPC stdin handling is the narrowest maintainable solution; the global uncaught-EPIPE handler is only a last-resort crash backstop and should not be the primary request failure path.
