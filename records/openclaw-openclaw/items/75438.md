## Summary

Keep open. Current main has a broad global backstop that treats uncaught `EPIPE`/`EIO` as non-fatal, but the iMessage RPC client still writes to the child stdin without a write callback or stdin `error` listener, so the request-level failure path remains unhandled and can leave callers waiting for timeout instead of rejecting cleanly.

## What This Changes

Harden the iMessage `imsg rpc` JSON-RPC client so broken stdin writes reject pending requests instead of surfacing as uncaught stream errors or delayed timeouts.

## Best Possible Solution

Add local stdin failure handling in `extensions/imessage/src/client.ts`: attach a `child.stdin` `error` listener during `start()`, route it through `failAll()` and closed-client state, and make `request()` clear/reject its pending entry on write callback errors. Cover the EPIPE path with a focused iMessage client unit test and keep the global uncaught-EPIPE handler as a last-resort backstop.

## Reproduction Assessment

Yes for the underlying stream failure. Current source shows the unguarded iMessage stdin write, and a focused Node Writable probe reproduced that write errors emit an uncaught `EPIPE` unless the stream has an `error` listener; the exact process-exit part is now partly mitigated by current main's global handler.

## Solution Assessment

No, the current global uncaught-EPIPE suppression is not the best complete fix. The narrow maintainable solution is local iMessage RPC stdin handling that rejects pending requests immediately and prevents the stream error from reaching process-level exception handling.
