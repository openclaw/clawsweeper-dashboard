## Summary

Keep open. Current main still has the unguarded iMessage RPC stdin write, and the existing global uncaught-EPIPE backstop only reduces crash severity instead of rejecting the affected request cleanly; an open unmerged fix PR already targets this issue, so the issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue asks for `IMessageRpcClient.request()` to handle child stdin write and stream errors so `EPIPE` rejects pending RPC requests instead of surfacing as an uncaught exception that can crash the gateway.

## Best Possible Solution

Review and land #75473, or an equivalent narrow iMessage-client patch that attaches a `child.stdin` error listener, rejects/clears the pending request on write-callback errors, and covers both paths with focused regression tests. After that merge, this issue can close as implemented.

## Reproduction Assessment

Yes. Current source shows the missing iMessage stdin callback/listener, and a focused Node Writable probe reproduces callback plus uncaught `EPIPE` behavior when no stream `error` listener is attached; the exact gateway-exit symptom is now partly masked by the global backstop.

## Solution Assessment

Yes for the intended fix direction. Local stdin error handling in `IMessageRpcClient` is the narrow maintainable fix; the global uncaught-EPIPE handler should remain only a last-resort backstop, not the primary request failure path.
