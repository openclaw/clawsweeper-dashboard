## Label Justifications

- `P1`: The proposed startup behavior can affect Gateway availability for existing explicit-local memory users and already has an open owner-level blocker discussion.
- `merge-risk: 🚨 compatibility`: Existing explicit-local memory configs would start doing model availability work during Gateway startup instead of only on first memory use.
- `merge-risk: 🚨 availability`: The startup probe can reach in-process GGUF model loading, which linked live reports show can leave the Gateway unreachable for minutes.

## Summary

Keep open: the PR has useful retry and coverage work, but it still moves builtin/local GGUF initialization into Gateway startup before local embedding isolation is solved. The linked open runtime issue shows this same in-process path can block Gateway reachability for minutes, so this needs an owner-level startup-boundary decision despite green CI and sufficient routing proof.

## What This Changes

The branch adds Gateway startup prewarm for explicit builtin/local memory search, makes failed provider initialization retryable, adds gateway/memory tests and a changelog entry, and relaxes a Codex auth-hint test helper.

## Best Possible Solution

Keep builtin/local prewarm disabled until GGUF model resolution, model/context creation, and the ping embedding run outside the Gateway event loop, then add startup prewarm with upgrade-safe tests.

## Reproduction Assessment

Yes, source-reproducible. Current main skips builtin/local startup, while the PR schedules it and calls probeEmbeddingAvailability(), which enters the in-process local provider path documented by the linked ARM64/Pi availability report.

## Solution Assessment

No. Prewarming the cached manager and clearing failed provider-init promises are useful, but enabling local GGUF probes from Gateway startup before provider isolation is not the safest solution.
