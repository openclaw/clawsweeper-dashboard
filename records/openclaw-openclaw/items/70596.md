## Summary

Keep open: the latest PR fixes the prior retry/changelog concerns, but the new builtin-local prewarm is still not reached by real gateway startup because post-attach scheduling remains QMD-only, and external real behavior proof is still missing.

## What This Changes

The PR prewarms explicit builtin local memory embeddings during gateway startup, makes provider initialization retry after transient failure, updates memory/gateway tests, relaxes a Codex/OpenAI auth-hint test helper, and adds a changelog entry.

## Best Possible Solution

Wire explicit builtin-local memory into the post-attach startup policy, keep existing QMD semantics and retry-safe provider initialization, add scheduler-level coverage, then require real gateway proof before merge.

## Reproduction Assessment

Yes from source inspection. At PR head, the builtin-local branch exists inside `startGatewayMemoryBackend`, but post-attach `resolveGatewayMemoryStartupPolicy` still returns `off` for non-QMD backends, so a real builtin gateway never schedules that branch.

## Solution Assessment

No. The retry-safe manager fix is good, but the prewarm must also be scheduled from `server-startup-post-attach.ts` and covered by a post-attach scheduler test before this solves the reported gateway behavior.
