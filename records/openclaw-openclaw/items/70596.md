## Summary

Keep open. The PR targets a real remaining builtin-local memory cold-start gap, but the patch is not safe to merge because its startup probe can poison the cached default builtin manager after a transient provider-init failure, and the branch needs to be refreshed over current QMD startup behavior.

## What This Changes

The PR adds gateway startup logic and tests to prewarm builtin `memorySearch.provider = "local"` embeddings, plus a shared OpenAI/Codex missing-auth test-helper relaxation.

## Best Possible Solution

Land a rebased, retry-safe explicit-local builtin prewarm that preserves current QMD startup semantics, avoids `auto` and remote providers, lets real searches retry after transient provider-init failure, and records the user-facing performance fix in the changelog.

## Reproduction Assessment

Yes. Configure builtin memory with `memorySearch.provider = "local"`, make provider initialization fail during startup prewarm, then call `memory_search` later in the same process after the provider would be available; current code shows the cached default manager retains the rejected initialization promise.

## Solution Assessment

No. Prewarming explicit local embeddings is the right target, but doing it through the cached default manager without retry-safe initialization is not the narrowest maintainable fix; clearing failed init state or otherwise making the prewarm/default manager retry-safe is safer.
