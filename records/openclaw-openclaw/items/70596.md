## Summary

Keep open. The PR’s retry fix is useful, but the central startup prewarm still calls the builtin local embedding probe from Gateway startup, and that path can resolve/download/load a GGUF model and run embeddings in the Gateway process before the local provider is isolated.

## What This Changes

The PR schedules builtin explicit-local memory startup prewarm, calls `probeEmbeddingAvailability()` on the cached builtin manager, retries failed provider initialization, and adds gateway/memory tests plus a changelog entry.

## Best Possible Solution

Coordinate this with https://github.com/openclaw/openclaw/issues/75657: keep the provider-init retry behavior if useful, but do not schedule builtin local GGUF prewarm at Gateway startup until model resolution, model/context creation, and embedding calls are isolated from the Gateway event loop.

## Reproduction Assessment

Yes. Source inspection shows PR head schedules builtin/local startup and calls `probeEmbeddingAvailability()`, which reaches in-process `node-llama-cpp` GGUF load and ping embedding; the related ARM64/Pi report supplies real logs for the blocking path.

## Solution Assessment

No. Retrying failed provider initialization is a good narrow fix, but moving local GGUF probing into Gateway startup is not the safest solution until the local provider has a worker or child-process boundary.

Full review comments:

- **[P1] Gate local prewarm until GGUF loads are isolated:** `src/gateway/server-startup-memory.ts:125`
  - body: This startup probe reaches the builtin local provider, which still resolves and loads the GGUF model and runs the ping embedding inside the Gateway process. Gate this behind worker/child-process isolation or defer the prewarm until that issue is fixed.
