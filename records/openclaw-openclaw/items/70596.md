## Summary

This PR should stay open with a blocking maintainer-review finding: the useful retry/cache fix is narrow, but the startup prewarm change moves builtin local GGUF embedding initialization into Gateway startup before that provider is isolated from the main event loop. The linked ARM64/Pi report shows this local warmup path can block Gateway reachability for minutes, so the remaining decision is an owner-level startup policy and isolation tradeoff rather than stale cleanup.

## What This Changes

The PR schedules builtin explicit-local memory search during post-attach Gateway startup, prewarms the cached builtin manager with `probeEmbeddingAvailability()`, retries failed provider initialization, and adds gateway/memory regression tests plus a changelog entry.

## Best Possible Solution

Keep the provider-init retry fix, but defer or gate builtin local startup prewarm until GGUF model resolution, model/context creation, and embedding calls run behind a worker-thread or child-process boundary, or until an owner explicitly accepts the startup tradeoff.

## Reproduction Assessment

Yes. Current main is source-reproducible as QMD-only startup, while PR head schedules builtin explicit-local startup and calls the local embedding probe path tied to the linked ARM64/Pi event-loop blocking report.

## Solution Assessment

No. Retrying failed provider initialization is a good narrow fix, but moving in-process GGUF probing into Gateway startup is not the safest solution until local embedding isolation or an explicit owner-approved startup policy is in place.
