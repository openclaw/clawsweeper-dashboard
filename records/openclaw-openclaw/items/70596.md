## Summary

Keep open. The PR contains useful cold-start and retry work, but it should not merge as-is because it moves explicit builtin-local embedding probing into Gateway startup while the local GGUF provider still resolves, loads, and pings the model inside the Gateway process; that overlaps the open ARM64/Pi event-loop outage report.

## What This Changes

The PR schedules builtin memory startup for explicit `memorySearch.provider = "local"`, prewarms the cached manager with `probeEmbeddingAvailability()`, retries failed provider initialization, and adds gateway/memory tests plus a changelog entry.

## Best Possible Solution

Coordinate this with the local embedding isolation work: keep startup prewarm disabled or narrowly gated until GGUF resolution, model/context creation, and the ping embedding run outside the Gateway event loop.

## Reproduction Assessment

Yes, source-reproducible. Current main's memory startup is QMD-only, while PR head schedules builtin/local and the probe reaches in-process `node-llama-cpp` model/context creation; the related ARM64/Pi issue supplies real logs for the event-loop block.

## Solution Assessment

No. Prewarming the cached default manager is the right cache target and the retry fix is useful, but scheduling the local GGUF probe from Gateway startup before provider isolation is not the safest solution for explicit-local configs.

- **[P1] Avoid startup prewarm for in-process GGUF loads:** `src/gateway/server-startup-memory.ts:125`
  - body: This startup probe reaches the builtin local provider, which still resolves/loads the GGUF model and runs the ping embedding inside the Gateway process. Gate prewarm behind worker/child-process isolation or defer it until that issue is fixed.
