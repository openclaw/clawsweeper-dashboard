## Summary

Keep open: current main still has a plausible source-reproducible macOS Gateway restart false-failure path, and an open closing PR should be reviewed instead of closing this issue before merge.

## What This Changes

The issue reports that macOS `openclaw gateway restart` can report a launchctl bootstrap failure during concurrent LaunchAgent bootstrap even though launchd has loaded and spawned the Gateway.

## Best Possible Solution

Land a narrow launchd restart fix that treats only the concurrent-bootstrap already-loaded race as success after `launchctl print` confirms the LaunchAgent is loaded or running, while genuine bootstrap failures remain fatal.

## Reproduction Assessment

Yes at source level. Current main still has a restart path that can throw from `launchctl bootstrap` without checking whether another bootstrap already loaded the LaunchAgent, and the issue provides a concrete macOS unified-log scenario.

## Solution Assessment

Yes in direction. The maintainable fix belongs in `src/daemon/launchd.ts` and should narrowly probe `launchctl print` after the specific bootstrap race before reporting success.
