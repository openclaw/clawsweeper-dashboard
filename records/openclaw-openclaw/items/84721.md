## What This Changes

The issue reports that macOS Gateway restart can falsely fail when a concurrent launchd bootstrap has already loaded the Gateway LaunchAgent.

## Best Possible Solution

Land a narrow launchd restart fix in `src/daemon/launchd.ts` with regression coverage that only treats the recoverable concurrent-bootstrap race as success after `launchctl print` confirms the Gateway LaunchAgent is loaded or running.

## Reproduction Assessment

Yes for source-level reproduction: current main throws from the daemon bootstrap helper on the reported nonzero launchctl result, and the issue gives a concrete concurrent-bootstrap macOS log scenario. I did not establish a live failing macOS race in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to probe launchd state after the specific recoverable bootstrap race and return success only when the intended LaunchAgent is loaded or running. The existing linked PR should be reviewed or corrected rather than queueing a second independent fix lane.
