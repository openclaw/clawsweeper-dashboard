## Summary

Keep open: the PR targets a valid source-reproducible bug in an existing silent-reply config contract, but the patch still misses the dispatcher’s explicit conversation-type context, its new tests use a non-generic error path that returns before the policy branch, and contributor real behavior proof is absent.

## What This Changes

The branch threads OpenClaw config into agent-runner failure reply handling, calls `resolveSilentReplyPolicy` before returning `NO_REPLY`, passes config from `runReplyAgent`, and adds silent-reply policy tests.

## Best Possible Solution

Carry the dispatcher-style explicit conversation type into failure-fallback policy resolution, cover the generic runner failure branch with focused tests, and require redacted real group/channel proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main returns `NO_REPLY` for generic group/channel runner failures before consulting the existing `silentReply` config, while docs and config code define group policy overrides. I did not run a live Feishu or Telegram timeout flow in this read-only review.

## Solution Assessment

No, not yet: using `resolveSilentReplyPolicy` is the right direction, but this PR omits the known conversation type and its tests exercise a rate-limit branch rather than the generic fallback branch. The narrower maintainable fix is to mirror the dispatcher’s `ChatType` to `conversationType` derivation and test the intended branch.
