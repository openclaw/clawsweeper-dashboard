## Summary

Keep open: the reported bug is source-reproducible on current main, but this PR is not merge-ready because the policy resolver call can still miss group/channel policy for opaque session keys, the added tests do not exercise the branch they assert, and the contributor has not provided real after-fix behavior proof.

## What This Changes

The branch threads config into the agent-runner failure reply path, calls `resolveSilentReplyPolicy` before returning `NO_REPLY`, and adds focused silent-reply policy tests.

## Best Possible Solution

Mirror the dispatcher’s silent-reply context contract in the failure-fallback path, cover the generic group/channel failure branch with focused tests, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main returns `NO_REPLY` for generic non-direct runner failures without reading the existing `silentReply` config, while the docs and config types define policy overrides. I did not run a live Feishu or Telegram flow in this read-only review.

## Solution Assessment

No, not yet: using `resolveSilentReplyPolicy` is the right direction, but this implementation omits the known conversation type and the new tests do not reliably exercise the intended generic-fallback behavior. The safer fix is to reuse the dispatcher-style `conversationType` derivation from `ChatType`.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Pass the chat type into silent policy resolution:** `src/auto-reply/reply/agent-runner-execution.ts:490-496`
  - body: This failure path already knows it is handling a group/channel from `ChatType`, but the new resolver call only passes `sessionKey` and `surface`. `classifySilentReplyConversationType` falls back to `internal` when the session key does not contain `:group:` or `:channel:`, so `silentReply.group: "disallow"` can still be ignored for opaque session keys.   - confidence: 0.93

As written, the default group expectations at lines 46 and 98 expect `NO_REPLY` even though the implementation returns rate-limit copy, and the direct allow case also returns before policy. These tests need to exercise the generic fallback path or the implementation needs to intentionally broaden policy handling.
