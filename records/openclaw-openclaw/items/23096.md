## Summary

Keep this PR open as a focused Bitwarden/Vaultwarden secrets candidate, but it is not mergeable as-is: current main still has only the generic env/file/exec SecretRef transports, the PR adds a security-sensitive wrapper that resolves `bw` through PATH, its docs point to an unpackaged bundled script, and maintainer discussion still needs to choose the plugin-seam route versus a hardened core exec-wrapper route.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script with type declarations, and tests for batch field resolution.

## Best Possible Solution

Maintainers should choose one path: either port this into a Bitwarden secret-provider plugin once the seam exists, or land a hardened and packaged exec-wrapper/docs integration coordinated with #29110, with command-path hardening, packaging, tests, and changelog covered.

## Solution Assessment

## Security Review

Status: needs_attention

Summary: Needs attention because the wrapper bypasses OpenClaw exec command hardening by resolving `bw` from PATH while receiving Bitwarden session environment.

Concerns:

- **[medium] PATH-resolved Bitwarden CLI bypasses exec hardening:** `scripts/bw-exec-resolver.mjs:27`
  - body: OpenClaw validates the configured exec provider command path, but this wrapper starts a second executable with `execFile("bw", ...)`; combined with docs passing `PATH`, the actual Bitwarden CLI binary is selected outside OpenClaw's trusted-dir checks and receives secret-bearing env such as `BW_SESSION`.
  - confidence: 0.87
