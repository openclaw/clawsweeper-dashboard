## Summary

Keep open: current main still has only the built-in env/file/exec SecretRef transports and no Bitwarden/Vaultwarden provider, but this PR needs maintainer route choice and command-execution hardening before merge.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script with declarations, and tests for parsing, field extraction, batching, and child-process behavior.

## Best Possible Solution

Choose the route deliberately: port this as a `secrets-bitwarden` plugin on the #72548 seam, or land a hardened exec wrapper that pins the Bitwarden CLI path, updates docs/changelog, and passes focused secrets tests.

## Reproduction Assessment

Not applicable as a feature PR rather than a broken-current-behavior bug. Source inspection confirms current main lacks a Bitwarden/Vaultwarden SecretRef provider or docs, so the remaining work is design and patch review rather than reproducing a failure.

## Solution Assessment

No, not as currently proposed. The exec-wrapper approach could be viable, but the current patch resolves the nested `bw` binary through PATH and maintainers still need to choose between that route and the plugin-provider seam tracked by #72548.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Pin the Bitwarden CLI instead of resolving through PATH:** `scripts/bw-exec-resolver.mjs:28-30`
  - body: The wrapper is documented as the multi-secret resolver while the docs pass `PATH` into its environment, but the wrapper then calls `execFile("bw", ...)`. Current SecretRef exec hardening validates only the configured wrapper command path, so the actual Bitwarden binary can be any earlier `bw` on PATH and will receive the passed Bitwarden session/secret requests. Require an absolute Bitwarden CLI path, validate it in the wrapper, and update the docs/tests so the nested command is not PATH-selected.
  - confidence: 0.86
