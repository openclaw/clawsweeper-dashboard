## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script, type declarations, and resolver tests for batched secret resolution.

## Best Possible Solution

Keep this focused PR open until maintainers choose the exec-wrapper versus plugin-provider route, then land only after hardening command execution, redacting errors, refreshing proof, and coordinating overlapping Bitwarden work.

## Solution Assessment

No. The requested integration is useful, but this patch needs command-execution hardening, redacted error handling, live proof, and a maintainer decision on whether it should remain an exec wrapper or move to the plugin-provider seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require a trusted Bitwarden binary path:** `scripts/bw-exec-resolver.mjs:29`
  - body: The wrapper is launched through OpenClaw's hardened exec provider, but it then spawns `bw` by searching PATH. The documented multi-secret config passes PATH, so the actual secret-manager binary bypasses the absolute-command and trusted-dir checks that protect exec SecretRefs; require an absolute `BW_BIN` or equivalent validated path before spawning the nested CLI.
  - confidence: 0.89
- **[P2] Redact CLI stderr before returning resolver errors:** `scripts/bw-exec-resolver.mjs:38`
  - body: `runBw` rejects with raw stderr, and `resolveSecrets` serializes that message into SecretRef protocol errors. Core can surface per-id exec errors during SecretRef resolution, so arbitrary Bitwarden CLI stderr can leak account, vault, or secret-adjacent data into diagnostics; map expected failures to fixed redacted messages instead.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds a security-sensitive secret resolver with concrete command-path and error-redaction issues.

Concerns:

- **[medium] Nested bw lookup bypasses exec hardening:** `scripts/bw-exec-resolver.mjs:29`
  - body: The resolver calls `execFile("bw", ...)`, so the Bitwarden CLI is selected from PATH inside the wrapper instead of being validated by OpenClaw's absolute command and trusted-directory checks.
  - confidence: 0.89
- **[medium] Raw CLI stderr reaches SecretRef errors:** `scripts/bw-exec-resolver.mjs:38`
  - body: Nonzero `bw` exits are converted from raw stderr into protocol errors, which can be surfaced by OpenClaw during SecretRef resolution and may expose account, vault, or secret-adjacent data.
  - confidence: 0.84
