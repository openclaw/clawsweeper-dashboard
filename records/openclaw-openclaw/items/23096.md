## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Ship the resolver where the docs point:** `docs/gateway/secrets.md:339-347`
  - body: The docs tell users to configure the bundled `scripts/bw-exec-resolver.mjs`, but the package `files` allowlist only includes a small set of scripts and does not include this new resolver. Packaged installs would publish the docs without the referenced executable, so users following this example would configure a missing command. Either include the resolver in the package at a stable installed path or change the docs to a path that actually ships.
  - confidence: 0.88
- **[P2] Pin the Bitwarden CLI path before passing secrets:** `scripts/bw-exec-resolver.mjs:29-30`
  - body: The core exec provider validates only the wrapper command; inside the wrapper, `execFile("bw", ...)` searches the inherited PATH while the docs pass `BW_SESSION` and `PATH`. A PATH-precedent `bw` binary would receive the vault session and item lookups, bypassing the absolute/trusted command checks that SecretRef exec providers otherwise enforce. Require an absolute configured Bitwarden binary path and validate it before spawning.
  - confidence: 0.9
- **[P2] Redact CLI errors before returning them:** `scripts/bw-exec-resolver.mjs:126-129`
  - body: The wrapper copies raw Bitwarden stderr or error text into the protocol `errors` object. Core includes those per-id messages in SecretRef resolution failures, so external CLI diagnostics from a secret-bearing path can surface in logs or UI. Map expected Bitwarden failures to bounded messages and avoid forwarding raw stderr.
  - confidence: 0.79
