## Summary

Keep open: current main still has no Bitwarden/Vaultwarden SecretRef support, and this PR remains the focused candidate, but the branch has a security-sensitive command-execution finding and an unresolved routing choice between an exec wrapper and the plugin-provider seam.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef docs, a protocol-v1 Bitwarden CLI exec resolver script with declarations, and tests for parsing, field extraction, grouping, and child-process resolution.

## Best Possible Solution

Land Bitwarden/Vaultwarden support only after maintainers choose the route: port it to a secrets-bitwarden plugin once #72548 lands, or harden the exec-wrapper/docs path with absolute CLI binary validation and redaction review.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature PR. For the blocking finding, source inspection gives a high-confidence path: configure the documented wrapper, put another bw earlier in PATH, and the wrapper executes that binary with the credential env.

## Solution Assessment

No, not as currently proposed. The exec-wrapper route can work only after binary selection is hardened, and maintainers still need to decide whether the plugin seam from #72548 is the long-term home.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve the Bitwarden CLI from a validated absolute path:** `scripts/bw-exec-resolver.mjs:29`
  - body: The configured exec provider validates only this wrapper script, but the wrapper then calls `execFile("bw", ...)` while the docs pass `PATH` along with Bitwarden credentials. That lets the actual secrets-reading binary be selected outside OpenClaw's command path validation and `trustedDirs` checks. Require an absolute validated Bitwarden binary path, such as `BW_BIN`, before spawning it.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds a command-execution path that can run a PATH-selected Bitwarden CLI with credential environment outside OpenClaw's exec-provider hardening.

Concerns:

- **[medium] PATH-selected bw bypasses exec hardening:** `scripts/bw-exec-resolver.mjs:29`
  - body: The trusted configured command is the wrapper, but the wrapper invokes `bw` by name and the docs pass `PATH` plus Bitwarden credential variables into it. A different earlier PATH entry can receive those credentials without passing OpenClaw's absolute-path, symlink, or trusted-dir checks.
  - confidence: 0.87
