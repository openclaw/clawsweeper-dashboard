## Summary

Keep open: current main still lacks Bitwarden/Vaultwarden support, and maintainer discussion has kept this as the focused candidate. The branch is not merge-ready because the integration route still needs human choice, real Bitwarden/Vaultwarden proof is missing, and the nested Bitwarden CLI invocation bypasses the core exec-provider command-path checks.

## What This Changes

Adds Bitwarden/Vaultwarden SecretRef docs, a `bw` protocol-v1 exec resolver script, TypeScript declarations, and tests for batched item/field secret resolution.

## Best Possible Solution

Pick the route explicitly, then either land a hardened exec wrapper that resolves `bw` from a trusted absolute path with real-vault proof, or port the work to a Bitwarden secret-provider plugin after the plugin seam lands.

## Reproduction Assessment

Not applicable as a current-main bug reproduction; this is feature work. The PR blocker is source-reproducible because `scripts/bw-exec-resolver.mjs` launches `bw` by PATH while the documented provider passes secret-bearing environment into the wrapper.

## Solution Assessment

No for merge as-is. The direction is useful, but maintainers still need to choose the integration route and the current wrapper must preserve the exec-provider command trust boundary before it can safely land.

## Review Findings

Full review comments:

- **[P2] Resolve the Bitwarden CLI from a trusted path:** `scripts/bw-exec-resolver.mjs:29-30`
