## Summary

Keep open. The Bitwarden/Vaultwarden gap is still present on current main, the branch is the focused candidate for that work, and recent maintainer-facing review asked to keep it open while deciding between a plugin-port path and a hardened exec-wrapper path. The current patch is not merge-ready because the wrapper invokes `bw` via `PATH` while passing vault credentials, and the PR still lacks real Bitwarden/Vaultwarden behavior proof.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef docs, a protocol-v1 `bw` exec resolver script, TypeScript declarations, and tests for batch secret resolution.

## Best Possible Solution

Keep a focused Bitwarden/Vaultwarden path open, either as a plugin after the secret-provider seam lands or as a hardened exec wrapper with explicit binary validation and live proof.

## Reproduction Assessment

Not applicable for a feature PR. Source inspection confirms current main lacks Bitwarden/Vaultwarden integration, but there is no live after-fix proof that the PR behavior works against a real vault.

## Solution Assessment

No, not as-is. The feature direction is useful, but the branch needs a maintainer route decision plus trusted nested CLI resolution before it is the safest implementation path.
