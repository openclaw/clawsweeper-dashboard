## Summary

Close as ClawHub scope. The proposal is a useful payment capability, but current main explicitly routes optional commercial integrations and new plugin-like capabilities out of core unless a missing generic plugin API is shown; the current SDK already exposes tools, CLI commands, HTTP routes, tool-call parameter rewriting with plugin approvals, and transcript-write hooks that cover the requested architecture without bundling a Stripe-specific payment plugin.

## What This Changes

Request to add a bundled payment plugin that supports Stripe Link virtual cards for browser checkout sentinel substitution and HTTP 402 machine payments.

## Best Possible Solution

Keep OpenClaw core focused on generic plugin and safety seams, and ship this as an external ClawHub code plugin. If implementation work exposes a missing generic API, the narrow follow-up should be an SDK/plugin-seam issue or PR rather than a bundled Stripe-specific plugin request.

## Reproduction Assessment

Not applicable. This is an RFC/feature request, and the relevant verification path is current-main scope and SDK inspection rather than reproducing a bug.

## Solution Assessment

Yes, closing to the ClawHub/plugin path is the narrowest maintainable outcome. The requested tool, CLI, HTTP, approval, parameter rewrite, and transcript guard surfaces already exist without adding a new bundled commercial integration to core.
