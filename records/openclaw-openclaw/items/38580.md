## Summary

Current main still does not implement Telegram pinned-message reads. The shared `list-pins` action and `openclaw message pins` command exist, but the bundled Telegram plugin does not advertise, route, gate, normalize, or execute that action; its current `getChat` helper only extracts `result.id`. The linked contributor PRs are useful prior art, but both were closed unmerged, so the feature request remains valid and narrow enough for a focused repair PR.

## What This Changes

Request Telegram support for fetching the current pinned chat message on demand through the existing bot/API message-action surface.

## Best Possible Solution

Add bundled Telegram plugin support for the existing `list-pins` action by calling `getChat`, normalizing `pinned_message` into the shared `pins` result shape, documenting Telegram's zero-or-one pinned-message behavior, and keeping automatic startup ingestion out of the initial change.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: dispatching shared `list-pins` to Telegram reaches `telegramMessageActions.handleAction`, where the action is not mapped and throws `Unsupported Telegram action`; the lower runtime also has no matching alias or handler.

## Solution Assessment

Yes. The narrow maintainable path is to implement this in the Telegram plugin through the existing shared message-action surface, with a Telegram `actions.pins` gate and tests, rather than adding a separate `/getpinned` command or core-specific Telegram logic.
