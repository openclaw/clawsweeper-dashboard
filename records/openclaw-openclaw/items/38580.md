## Summary

Keep open: current main and v2026.6.5 still do not expose a Telegram pinned-message read action, and the two linked implementation attempts are closed unmerged. The related open pin/unpin request is a distinct write capability, so it does not supersede this read-only session-context feature.

## What This Changes

Request to let Telegram agents fetch the current pinned chat message on demand, and possibly at session start, using Telegram Bot API support.

## Best Possible Solution

Add a plugin-owned, read-only, on-demand Telegram list-pins action that returns zero or one current chat pinned message without automatic session injection, storage, or a new core action surface.

## Reproduction Assessment

Yes, at source level. Current main exposes the generic list-pins action name but Telegram discovery and runtime omit any pinned-message read mapping, so the requested Telegram behavior cannot be invoked.

## Solution Assessment

Yes, if kept narrow. The best implementation is an on-demand Telegram plugin action using getChat().pinned_message; automatic session-start prompt injection would add avoidable state and policy surface.

Add an on-demand read-only `list-pins` message action for Telegram using the existing generic `CHANNEL_MESSAGE_ACTION_NAMES` entry and the Telegram Bot API `getChat` contract. Keep ownership in `extensions/telegram`: update discovery/action mapping, runtime dispatch, and a small helper that resolves the target chat using existing target normalization/getChat patterns, then returns a `pins` array with zero or one current pinned message from `pinned_message`. Do not add automatic session-start injection, persistence/cache, topic pin history, a new core action name, or unrelated pin/unpin write support.
