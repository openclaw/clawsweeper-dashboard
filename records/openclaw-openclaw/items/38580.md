## Summary

Keep open: current main still lacks a Telegram pinned-message read path, and the related implementation PRs were closed unmerged. This is a narrow Telegram plugin feature rather than a core API gap because the shared message-action contract already includes `list-pins`.

## What This Changes

Request Telegram support for reading a chat's current pinned message through the assistant's bot/API tool surface.

## Best Possible Solution

Add a Telegram plugin-owned on-demand `list-pins` message action backed by Bot API `getChat.pinned_message`, with a Telegram `actions.pins` gate and docs, while leaving automatic session-start context injection for a later product decision.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection gives high confidence that current main has no Telegram `list-pins`, `getpinned`, or documented pinned-message read path.

## Solution Assessment

Yes: the narrow maintainable solution is an on-demand Telegram `list-pins` action inside the existing Telegram plugin message-action adapter. That avoids core changes and avoids bundling the larger automatic startup-context behavior into the first implementation.

## Repair Work Prompt

Implement a narrow Telegram plugin feature for https://github.com/openclaw/openclaw/issues/38580. Add an on-demand Telegram message action for `list-pins` / `listPins` backed by Telegram Bot API `getChat`, returning `pins: []` when there is no current pinned message and one normalized pin when `pinned_message` is present. Keep the work in the Telegram plugin and existing message-action surfaces; do not add a new core action name, do not add automatic session-start pinned-context injection, and do not add a broad slash-command-only implementation. Add a `channels.telegram.actions.pins` type/schema/docs/config-hint gate consistent with Slack/Discord `actions.pins`, default it like other read actions unless existing gate helpers require an explicit default, and cover account-scoped discovery/gating. Add or update regression tests for action discovery, runtime success, no-pin, disabled gate, missing token/error handling, and target/account handling. Update `docs/channels/telegram.md` and `CHANGELOG.md` because this is user-facing. Preserve plugin boundary rules: Telegram prod code should use plugin SDK/public Telegram helpers, not core internals.
