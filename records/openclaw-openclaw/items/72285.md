## Summary

Keep open: current main still lacks checkpoint-backed post-compaction status output, and this PR is a useful implementation path, but the branch can reuse an older checkpoint's `tokensAfter` when the newest checkpoint lacks token data and it still lacks real behavior proof.

## What This Changes

The PR adds a status formatter fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test for that status output.

## Best Possible Solution

Land a corrected shared status formatter fix that preserves fresh totals, uses only the actual latest checkpoint's finite non-negative `tokensAfter` when totals are stale or unavailable, and adds regression coverage for the no-older-reuse case.

## Reproduction Assessment

Yes, source-level. On current main, calling `buildStatusMessage()` with `totalTokensFresh: false`, stale `totalTokens`, and a compaction checkpoint containing `tokensAfter` leaves the status context unknown instead of checkpoint-backed post-compaction usage.

## Solution Assessment

No. The PR direction is narrow and appropriate, but the helper should select the newest checkpoint before validating `tokensAfter`, with a regression that proves a newer checkpoint without token data does not reuse an older value.
