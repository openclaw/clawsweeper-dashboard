## Summary

Keep open. Current `main` still lacks this `/status` fallback, and the PR targets a valid stale-context bug, but the diff can report an older checkpoint when the newest checkpoint lacks `tokensAfter`; it also still needs the small cleanup and changelog follow-through already noted in review discussion.

## What This Changes

The PR adds a `/status` fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test for stale totals after compaction.

## Best Possible Solution

Land a focused status formatter fix that selects the newest checkpoint by `createdAt` first, uses that checkpoint's finite non-negative `tokensAfter` only when totals are stale, preserves unknown context when absent, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence unit path is `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and compaction checkpoints; the PR-specific regression is a newer checkpoint without `tokensAfter` plus an older checkpoint that has it.

## Solution Assessment

No. The fallback source is appropriate, but this implementation filters checkpoints before choosing the newest one; the safer solution is to choose the newest checkpoint first and then validate that checkpoint's `tokensAfter`.
