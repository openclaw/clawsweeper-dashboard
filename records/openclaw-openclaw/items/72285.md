## Summary

Keep open. The PR targets a real `/status` stale-context bug, but the current patch is not mergeable because it can reuse an older compaction checkpoint when the newest checkpoint lacks `tokensAfter`, and it is missing required cleanup and changelog work.

## What This Changes

The PR adds a `/status` context fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test.

## Best Possible Solution

Land a focused status formatter fix that selects the newest compaction checkpoint by `createdAt` first, uses that checkpoint's finite non-negative `tokensAfter` only when totals are stale, preserves unknown context otherwise, covers the no-older-reuse case, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The reported path is reproducible by calling `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a checkpoint with `tokensAfter`; the PR-specific regression is a newer checkpoint without `tokensAfter` plus an older checkpoint that has it.

## Solution Assessment

No. The requested direction is right, but this implementation filters checkpoints before selecting the newest one; the safer solution is to select the newest checkpoint first and then validate that checkpoint's `tokensAfter`.
