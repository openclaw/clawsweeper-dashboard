## Summary

Keep open. The PR targets a real current-main status bug, but the current diff is not merge-ready because it can reuse an older checkpoint's `tokensAfter` after a newer compaction checkpoint exists without token data; it also leaves an unreachable fallback branch and misses the required changelog entry for a user-facing fix.

## What This Changes

The PR adds a status formatter fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test for the stale-total post-compaction display.

## Best Possible Solution

Land a focused status formatter fix that first selects the newest compaction checkpoint by timestamp, uses that checkpoint's `tokensAfter` only when it is a finite non-negative number, leaves the Context total unknown otherwise, removes the unreachable fallback branch, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. The intended bug is reproducible by calling `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a compaction checkpoint with `tokensAfter`; the PR-specific bug is reproducible with a newer checkpoint that lacks `tokensAfter` plus an older checkpoint that has it.

## Solution Assessment

No. The requested direction is right, but this implementation filters checkpoints before deciding which one is latest; the safer solution is to choose the newest checkpoint first and then validate that checkpoint's post-compaction token count.
