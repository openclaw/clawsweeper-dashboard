## Summary

Keep open. Current main still lacks the checkpoint-backed status fallback, and this PR is useful, but the branch can display an older checkpoint as current context and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR updates shared status message formatting to use compaction checkpoint `tokensAfter` when stored totals are stale and adds a regression test for post-compaction context output.

## Best Possible Solution

Land a corrected shared status formatter fix that selects the newest checkpoint first, uses that checkpoint's finite non-negative `tokensAfter` only when totals are stale or unavailable, and covers the no-older-reuse regression.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` on current main with `totalTokensFresh: false`, a stale `totalTokens`, and compaction checkpoint metadata; current main renders unknown context instead of the checkpoint post-compaction size.

## Solution Assessment

No. The PR direction is right, but filtering checkpoints before selecting the newest one can display stale older checkpoint tokens, so the safer fix is latest-checkpoint-first and then `tokensAfter` validation.
