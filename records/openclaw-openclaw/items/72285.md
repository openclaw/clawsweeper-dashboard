## Summary

Keep open. Current main still lacks the requested post-compaction checkpoint fallback, and the PR addresses a valid user-visible `/status` bug, but the current patch can report an older checkpoint when the newest checkpoint has no `tokensAfter`, so it should not merge as-is.

## What This Changes

The PR adds a `/status` formatter fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test for stale totals after compaction.

## Best Possible Solution

Land a focused status formatter fix that chooses the newest compaction checkpoint by `createdAt` first, uses that checkpoint's finite non-negative `tokensAfter` only when session totals are stale, leaves context usage unknown otherwise, covers the no-older-reuse case, removes dead fallback code, and adds the required user-facing changelog entry.

## Reproduction Assessment

Yes. The issue path is reproducible by calling `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a compaction checkpoint with `tokensAfter`; the PR-specific regression is a newer checkpoint without `tokensAfter` plus an older checkpoint that has it.

## Solution Assessment

No. The requested direction is right, but this implementation filters checkpoints before selecting the newest one; the safer solution is to select the newest checkpoint first and then validate that checkpoint's `tokensAfter`.
