## Summary

Keep open: this PR addresses a real `/status` accuracy bug, but as written it can reuse an older compaction checkpoint when the newest checkpoint lacks `tokensAfter`, and the external PR still has only test proof.

## What This Changes

The PR adds status-message logic to use compaction checkpoint `tokensAfter` when session totals are stale, plus a regression test.

## Best Possible Solution

Land or replace this with a corrected shared status formatter fix that selects the newest checkpoint first, uses its finite non-negative `tokensAfter` only when appropriate, and covers the no-reuse case for older checkpoints.

## Reproduction Assessment

Yes, source-level. Calling `buildStatusMessage()` on current main with `totalTokensFresh: false`, a stale `totalTokens`, and a latest checkpoint with `tokensAfter` cannot render the checkpoint-backed post-compaction context size.

## Solution Assessment

No. The direction is right, but the PR filters checkpoints before selecting the newest one; the safer implementation chooses the latest checkpoint first and only then validates optional `tokensAfter`.
