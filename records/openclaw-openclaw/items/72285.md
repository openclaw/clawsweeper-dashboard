## Summary

Keep open. The PR targets a real shared `/status` bug, but the current patch can pick an older checkpoint's `tokensAfter` when the newest checkpoint lacks a usable token count, and the external contributor still has not provided after-fix real behavior proof.

## What This Changes

The PR adds a shared status formatter fallback from stale session totals to compaction checkpoint `tokensAfter` plus a regression test for post-compaction context output.

## Best Possible Solution

Land a corrected shared formatter fix that selects the latest checkpoint first, uses its finite non-negative `tokensAfter` only when totals are stale or unavailable, and covers the older-checkpoint no-reuse case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` on current main with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context instead of the checkpoint post-compaction size.

## Solution Assessment

No. The PR direction is right, but filtering checkpoints before selecting the newest one can display stale older checkpoint tokens; the safer fix is latest-checkpoint-first and then `tokensAfter` validation.
