## Summary

Keep open: this PR targets a real shared status bug that current main still does not fully solve, but the branch needs a correctness fix before merge and lacks required real behavior proof.

## What This Changes

The PR adds a `/status` formatter fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test.

## Best Possible Solution

Land a corrected shared status formatter fix that selects the newest checkpoint first, uses that checkpoint's finite non-negative `tokensAfter` only when totals are stale or unavailable, preserves unknown context otherwise, and covers the no-older-reuse regression.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and compaction checkpoint metadata; the PR-specific failing case is a newer checkpoint without `tokensAfter` plus an older one with it.

## Solution Assessment

No. The PR direction is right, but the helper filters checkpoints before selecting the newest one, so the safer implementation is to select the newest checkpoint first and then validate its `tokensAfter`.
