## Summary

Keep open. The PR targets a real shared status bug that is still present on current main, but the current diff can report an older checkpoint's token count when the newest checkpoint lacks `tokensAfter`, and the external PR still lacks after-fix real behavior proof.

## What This Changes

This PR adds a shared status formatter fallback from stale session totals to compaction checkpoint `tokensAfter` and a regression test for post-compaction context output.

## Best Possible Solution

Land a corrected shared formatter fix that selects the latest checkpoint first, uses its finite non-negative `tokensAfter` only when totals are stale or unavailable, and covers the older-checkpoint no-reuse regression.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` on current main with `totalTokensFresh: false`, a stale `totalTokens`, and compaction checkpoint metadata; current main resolves the context numerator as unknown instead of the checkpoint post-compaction size.

## Solution Assessment

No. The PR direction is right, but filtering checkpoints before selecting the newest one can display stale older checkpoint tokens; the safer fix is latest-checkpoint-first and then `tokensAfter` validation.
