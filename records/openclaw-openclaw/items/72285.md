## Summary

Keep open: the PR targets a real shared status bug that remains on main, but the current branch should not merge because it can reuse an older checkpoint count when the newest checkpoint lacks `tokensAfter`, and the external contributor has not posted real after-fix behavior proof.

## What This Changes

The PR adds a shared status formatter fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test.

## Best Possible Solution

Land a corrected shared formatter fix that selects the newest checkpoint first, uses its finite non-negative `tokensAfter` only when totals are stale or unavailable, preserves fresh totals, and covers the no-older-reuse regression.

## Reproduction Assessment

Yes. A high-confidence source-level path is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and compaction checkpoint metadata; the PR-specific failing case is a newer checkpoint without `tokensAfter` plus an older one with it.

## Solution Assessment

No. The PR direction is right, but filtering checkpoints before choosing the newest one can present an older post-compaction count as current context; the safer fix is to select the latest checkpoint first and then validate its `tokensAfter`.
