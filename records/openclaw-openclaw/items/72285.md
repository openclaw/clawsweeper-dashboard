## Summary

Keep open. Current main still does not show checkpoint-backed post-compaction context when totals are stale, and this PR is a plausible implementation path, but the patch has a blocking ordering bug and lacks real behavior proof.

## What This Changes

The PR changes shared status context rendering to use compaction checkpoint `tokensAfter` when stored session totals are stale and adds a regression test.

## Best Possible Solution

Land a corrected shared formatter fix that selects the newest checkpoint first, uses its finite non-negative `tokensAfter` only when totals are stale or unavailable, preserves fresh totals, and covers the no-older-reuse regression.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and compaction checkpoint metadata; the PR-specific failing case is a newer checkpoint without `tokensAfter` plus an older one with it.

## Solution Assessment

No. The PR direction is right, but filtering checkpoints before choosing the newest one can show an older post-compaction count as current context; the safer fix is to select latest first and then validate its `tokensAfter`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Select the newest checkpoint before reading tokensAfter:** `src/status/status-message.ts:262-274`
  - body: `tokensAfter` is optional, but this helper filters out checkpoints without it before choosing by `createdAt`. If a later compaction checkpoint has no token count, `/status` will reuse an older checkpoint's `tokensAfter` as current context; pick the latest checkpoint first, then use its `tokensAfter` only when valid.
