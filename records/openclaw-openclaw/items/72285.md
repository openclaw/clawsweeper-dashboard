## Summary

Keep this PR open: current main still has the post-compaction status gap, but the branch can reuse an older checkpoint token count and it still lacks required real behavior proof.

## What This Changes

The PR adds a status formatter fallback from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test for post-compaction context output.

## Best Possible Solution

Land a corrected shared status formatter fix that selects the newest checkpoint first, uses its finite non-negative `tokensAfter` only when appropriate, and covers the no-reuse case for older checkpoints.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` on current main with `totalTokensFresh: false`, stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context instead of the checkpoint post-compaction size.

## Solution Assessment

No. The PR direction is right, but filtering checkpoints before selecting the newest one can display stale older checkpoint tokens; the safer fix is latest-checkpoint-first and then `tokensAfter` validation.
