## Summary

Keep open. Current main and the latest release still have a source-reproducible all-error-payload path that can persist a recovered cron run as failed, and the linked focused fix PR is still open rather than merged.

## What This Changes

Issue reports cron isolated agent-turn runs selecting an intermediate tool error over the final assistant recovery text when summarizing and persisting the run outcome.

## Best Possible Solution

Review, refresh if needed, and land https://github.com/openclaw/openclaw/pull/84156 or an equivalent resolver-level fix that marks recovered tool-error payloads nonfatal while keeping run-level errors and typed fatal signals fatal.

## Reproduction Assessment

Yes, at source level. Current main has a focused helper test that passes all-error payloads plus terminal assistant-visible text and expects the last error payload to win; I did not execute tests because this review is read-only.

## Solution Assessment

Yes, the best fix belongs in `resolveCronPayloadOutcome` because both interim retry handling and final cron persistence consume that classifier. A summary-only change would still leave `hasFatalErrorPayload` and persisted `status` wrong.
