## Summary

Keep this PR open: it addresses a real local gateway false-negative cluster and extends surfaces that current main does not fully cover, but the latest head still skips the fallback for normal token/password loopback setups over unpinned `ws://127.0.0.1`, which is part of the linked reproduction shape.

## What This Changes

The PR adds a shared local loopback `status` RPC fallback for CLI status/probe paths, a guarded loopback-only `callGateway` override, regression tests, and a changelog entry.

## Best Possible Solution

Keep the bounded local `status` RPC fallback, but distinguish trusted config/default local loopback credentials from arbitrary URL overrides so token-auth loopback gateways can recover without expanding remote or untrusted credential reuse.

## Reproduction Assessment

Yes. A local `ws://127.0.0.1` gateway with `gateway.auth.mode=token`, an initial timeout or unknown-capability probe, and a successful direct `status` RPC follows #48360/#73535; the PR helper exits before `callStatus` because `hasSharedCredentials` is true and no TLS fingerprint is present.

## Solution Assessment

No. The shared fallback is the right shape, but the current credential gate is too broad; a safer fix is to allow config-derived active local loopback credentials while still rejecting implicit credentials for arbitrary URL overrides and remote targets.
