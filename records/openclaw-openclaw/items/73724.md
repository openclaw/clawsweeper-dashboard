## Summary

Keep open: the PR is pointed at a real gateway/status false-negative path, but the current head blocks the fallback for normal local token/password loopback setups, including the linked token-auth reproductions.

## What This Changes

The PR adds a shared local loopback `status` RPC fallback for CLI status/probe paths, tests credential/pinning behavior, opens a narrowly scoped `callGateway` loopback override, and adds a changelog entry.

## Best Possible Solution

Repair the fallback gate so config-derived local token/password loopback targets can use the bounded status RPC without allowing implicit credentials on arbitrary URL overrides, then rerun changed-gate validation.

## Reproduction Assessment

Yes. A local `ws://127.0.0.1` gateway with `gateway.auth.mode=token`, an initial probe timeout, and a successful direct `status` RPC follows the linked #48360/#73535 shape; the PR helper exits before `callStatus` because `hasSharedCredentials` is true.

## Solution Assessment

No. The bounded local status-RPC fallback is the right direction, but the current shared-credential guard is too broad; the safer fix is to distinguish config-derived loopback credentials from untrusted URL overrides.
