## Summary

Keep this PR open: it is a plausible bug/privacy fix, but current main now routes OpenAI-compatible audio uploads through the shared SDK-facing form-data helper, while the PR still adds sanitization to the older private wrapper path and lacks contributor-provided real behavior proof.

## What This Changes

The PR adds a private OpenAI-compatible audio upload filename resolver and unit tests for path-shaped and missing runtime filenames.

## Best Possible Solution

Move POSIX-and-Windows basename stripping into resolveAudioTranscriptionUploadFileName, preserve the existing audio fallback and AAC/M4A normalization, and cover the shared helper path with focused tests.

## Reproduction Assessment

Yes, at source level. Current main’s shared resolver calls path.basename on the trimmed filename, so Windows-style backslash paths remain intact on POSIX while missing filenames already fall back to audio.

## Solution Assessment

No, not in the PR’s current shape. The basename algorithm is reasonable, but the maintainable fix belongs in the shared SDK-facing resolver used by buildAudioTranscriptionFormData, not in a private wrapper path that current main no longer uses for form construction.
