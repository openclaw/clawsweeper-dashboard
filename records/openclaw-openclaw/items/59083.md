## Summary

Keep open. The Windows-style host-path filename leak is still source-reproducible on current main, but this PR patches a private OpenAI-compatible path that current main no longer uses for multipart construction and it lacks contributor-owned real behavior proof.

## What This Changes

The PR adds a private OpenAI-compatible audio upload filename resolver and unit tests for path-shaped and missing runtime filenames.

## Best Possible Solution

Move POSIX and Windows basename stripping into the shared SDK-facing resolver, preserve the existing audio fallback and AAC/M4A normalization, and cover shared-helper callers with focused tests.

## Reproduction Assessment

Yes, at source level. Current main routes multipart audio through buildAudioTranscriptionFormData, whose shared resolver uses platform path.basename, so a Windows-style filename such as C:\Users\alice\note.wav remains intact on POSIX while missing filenames already fall back to audio.

## Solution Assessment

No, not in this PR's current shape. The basename algorithm is reasonable, but the maintainable fix belongs in resolveAudioTranscriptionUploadFileName so OpenAI-compatible and SDK provider callers get the same behavior.

## Review Findings

Overall correctness: patch is incorrect
