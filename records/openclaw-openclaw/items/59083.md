## Summary

Keep open. The PR addresses a real privacy/crash bug, but current main has since moved multipart filename assembly into `src/media-understanding/shared.ts`; main already fixes the missing filename crash and POSIX basename case, while Windows backslash paths can still leak on POSIX hosts. The PR head is also dirty against main, so the useful fix needs to be adapted into the shared helper rather than merged as-is.

## What This Changes

The PR adds an OpenAI-compatible audio upload filename resolver that strips POSIX/Windows path segments and falls back to `audio`, plus unit coverage for a POSIX host path and an undefined runtime filename.

## Best Possible Solution

Update `resolveAudioTranscriptionUploadFileName` in `src/media-understanding/shared.ts` so every OpenAI-compatible audio provider strips both POSIX and Windows path segments, keeps the existing missing-name fallback and AAC/M4A handling, and has focused regression tests for those cases.

## Reproduction Assessment

Yes. Current main can be reproduced by exercising the shared filename resolver shape with a Windows backslash path on a POSIX host; the basename stays as the full path, while missing and blank filenames already fall back safely to `audio`.

## Solution Assessment

No, not as currently placed. The PR's basename algorithm is the right direction, but the maintainable current-main fix belongs in the shared upload filename resolver so it covers all callers and preserves later filename-extension behavior.
