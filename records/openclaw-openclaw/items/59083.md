## Summary

Keep this PR open, but it is not mergeable as written: current main still has the Windows-path filename leak in the active shared resolver, while this branch patches a stale private OpenAI-compatible wrapper path.

## What This Changes

The PR adds a private OpenAI-compatible audio upload filename resolver and unit tests for path-shaped and missing filenames.

## Best Possible Solution

Put POSIX-and-Windows basename stripping in `resolveAudioTranscriptionUploadFileName`, preserve the existing `audio` fallback and AAC/M4A normalization, and cover the shared helper path with focused tests.

## Reproduction Assessment

Yes, at source level: current main’s `resolveAudioTranscriptionUploadFileName` uses platform `path.basename`, so Windows-style backslash paths remain intact on POSIX while missing filenames already fall back to `audio`. A focused helper test or Node probe against the resolver shows the leak without needing a live provider call.

## Solution Assessment

No, not in this PR’s current shape. The basename algorithm is reasonable, but the fix belongs in the shared SDK-facing resolver used by `buildAudioTranscriptionFormData`, not in a stale private OpenAI wrapper path.
