## Summary

Keep open: the filename privacy bug is valid, but this PR is not merge-ready against current main because the active upload path now goes through the shared SDK-facing form-data helper, while the PR patches the older local OpenAI-compatible helper path. The external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds an OpenAI-compatible audio upload filename resolver and unit tests so path-like or missing filenames become a basename or `audio`.

## Best Possible Solution

Move POSIX-and-Windows basename stripping into `resolveAudioTranscriptionUploadFileName`, keep the existing fallback and AAC/M4A behavior, and cover the shared helper with focused tests.

## Reproduction Assessment

Yes at source level. Current main’s shared resolver leaves a Windows-style `C:\Users\...` upload filename unchanged, while the missing-filename fallback already returns `audio`.

## Solution Assessment

No, not in the PR’s current shape. The basename algorithm is reasonable, but the maintainable fix belongs in the shared SDK-facing resolver instead of a stale local wrapper path.

## Review Findings
