## Summary

Keep open: this is a plausible privacy/bug fix, but the branch is stale against current main and sanitizes the old local upload path instead of the shared helper that now builds audio transcription multipart forms. The external PR also still needs after-fix real behavior proof before merge.

## What This Changes

The PR adds a local upload filename resolver and unit tests to strip path-shaped audio filenames and fall back to `audio` for missing filenames.

## Best Possible Solution

Move POSIX-and-Windows basename stripping into `resolveAudioTranscriptionUploadFileName`, preserve the existing `audio` fallback and AAC/M4A normalization, and cover the shared helper with focused tests.

## Reproduction Assessment

Yes at source level: current main’s shared resolver uses platform `path.basename`, and a POSIX Node probe leaves a Windows-style path unchanged. The missing-filename crash part no longer reproduces on current main because the shared resolver falls back to `audio`.

## Solution Assessment

No, not as currently placed. The basename algorithm is right, but the maintainable fix belongs in the shared upload filename resolver used by current main and SDK-facing provider helpers.
