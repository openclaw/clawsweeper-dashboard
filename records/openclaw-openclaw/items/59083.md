## Summary

Current main still leaves the active shared audio upload filename resolver vulnerable to Windows-style path leakage, so this PR is not obsolete. The branch is stale against main’s current shared helper shape and lacks after-fix real behavior proof, so it should stay open for contributor follow-up rather than be closed or auto-repaired.

## What This Changes

The PR adds a private OpenAI-compatible audio upload filename resolver and tests for path-shaped and missing filenames.

## Best Possible Solution

Move POSIX-and-Windows basename stripping into resolveAudioTranscriptionUploadFileName, preserve the existing audio fallback and AAC/M4A normalization, and cover the shared helper path with focused tests.

## Reproduction Assessment

Yes, at source level: current main’s shared resolver uses platform path.basename, so a Windows-style filename like C:\Users\alice\private\note.wav remains unchanged on POSIX while undefined still falls back to audio. The focused Node probe and source lines show the privacy leak without needing a live provider call.

## Solution Assessment

No, not in this PR’s current shape. The basename strategy is directionally right, but it must live in the exported shared resolver used by buildAudioTranscriptionFormData, not in a private OpenAI wrapper path that current main no longer uses for filename normalization.
