## Summary

Keep open: the bug/security-hardening goal is valid, but the branch now patches a stale local upload path while current main routes OpenAI-compatible audio uploads through the shared form-data resolver exported to the provider HTTP SDK; the PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds a local OpenAI-compatible audio upload filename sanitizer plus unit tests for host-path filenames and missing runtime filenames.

## Best Possible Solution

Move POSIX-and-Windows basename stripping into resolveAudioTranscriptionUploadFileName, preserve the existing audio fallback and AAC/M4A normalization, and cover the shared helper with focused tests.

## Reproduction Assessment

Yes at source level: current main uses platform path.basename in resolveAudioTranscriptionUploadFileName, and a POSIX Node probe leaves a Windows-style C:\\Users\\... path unchanged. The missing-filename crash no longer reproduces in the active shared resolver because it falls back to audio.

## Solution Assessment

No, not as currently placed. The basename algorithm is right, but the maintainable fix belongs in the shared upload filename resolver used by current main and SDK-facing callers.
