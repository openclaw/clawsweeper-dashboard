## Summary

Keep open: the underlying filename metadata bug still exists on current main for Windows-style paths, but this PR patches the stale local OpenAI-compatible helper instead of the current shared resolver exported through the provider HTTP SDK. The PR also lacks required after-fix real behavior proof.

## What This Changes

The PR adds a local OpenAI-compatible audio upload filename resolver and tests to strip host paths and fall back to `audio`.

## Best Possible Solution

Move POSIX-and-Windows basename stripping into `resolveAudioTranscriptionUploadFileName`, preserve the existing `audio` fallback and AAC/M4A normalization, and cover the shared helper with focused tests.

## Reproduction Assessment

Yes at source level: a probe matching current main's resolver leaves a Windows-style `C:\\Users\\...` filename unchanged while the shared path is the active upload path. The missing-filename crash no longer reproduces on current main because the shared resolver falls back to `audio`.

## Solution Assessment

No, not as currently placed. The basename algorithm is reasonable, but the maintainable fix belongs in the shared upload filename resolver used by current main and SDK-facing provider callers.
