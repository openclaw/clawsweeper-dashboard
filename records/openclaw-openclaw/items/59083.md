## Summary

Keep open: current main still has the Windows-style audio upload filename leak in the shared provider HTTP resolver, but this PR needs to move the fix from the stale private wrapper path into the shared SDK-facing helper and add contributor real behavior proof before merge.

## What This Changes

The PR adds a private OpenAI-compatible audio upload filename resolver and unit tests for path-shaped and missing filenames.

## Best Possible Solution

Move POSIX-and-Windows basename stripping into resolveAudioTranscriptionUploadFileName, preserve the existing audio fallback and AAC/M4A normalization, and cover the shared helper path with focused tests.

## Reproduction Assessment

Yes. Source inspection plus a Node probe against the current resolver algorithm shows Windows-style backslash paths remain intact on POSIX, while missing filenames already fall back to audio.

## Solution Assessment

No, not in the PR's current shape. The basename algorithm is reasonable, but the maintainable fix belongs in the shared SDK-facing resolver used by buildAudioTranscriptionFormData, not in a private stale wrapper path.
