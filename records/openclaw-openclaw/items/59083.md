## What This Changes

The PR adds a local OpenAI-compatible audio upload filename resolver and unit tests for host-path filenames and missing runtime filenames.

## Best Possible Solution

Move the POSIX-and-Windows basename sanitizer into `resolveAudioTranscriptionUploadFileName` while preserving the existing `audio` fallback and AAC/M4A normalization.

## Reproduction Assessment

Yes at source level for the remaining path leak: current main uses platform `path.basename`, and a Node probe shows Windows-style paths remain unchanged on this POSIX host. The missing-filename crash no longer reproduces in the active shared resolver because it falls back to `audio`.

## Solution Assessment

No, not as currently placed. The basename algorithm is right, but the maintainable fix belongs in the shared upload filename resolver used by current main and SDK-facing callers.
