## Summary

Keep open: the reported audio filename privacy bug is still valid on current main, but this PR is now conflicting and patches the old private multipart construction path instead of the active shared helper used by OpenAI-compatible and SDK provider callers.

## What This Changes

The PR adds a local OpenAI-compatible audio upload filename resolver and unit tests for path-shaped and missing filenames.

## Best Possible Solution

Move POSIX and Windows basename stripping into resolveAudioTranscriptionUploadFileName, preserve the existing audio fallback and AAC/M4A normalization, and cover shared-helper callers with focused tests.

## Reproduction Assessment

Yes, at source level. Current main builds multipart audio bodies through buildAudioTranscriptionFormData, and the shared resolver uses platform path.basename so a Windows-style filename remains unsanitized on POSIX.

## Solution Assessment

No, not in this PR's current shape. The basename algorithm is reasonable, but the maintainable fix belongs in the shared SDK-facing resolver rather than a private helper in the stale OpenAI-compatible implementation.
