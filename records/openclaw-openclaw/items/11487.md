## Summary

Current main and the latest release still lack a supported human-readable metadata field for strict channel group entries, so the request remains useful; the related WhatsApp issue covers functional name-based matching and does not supersede this metadata-only contract change.

## What This Changes

Requests one optional non-functional string field on channel group configuration entries so numeric or opaque IDs can carry human-readable names.

## Best Possible Solution

Choose one canonical optional string field, add it only to equivalent strict group/room entry types and schemas, preserve it through config read/write and generated metadata, ignore it for routing, document its purpose, and add acceptance plus round-trip tests.

## Reproduction Assessment

Yes at the source-contract level: current main's strict Telegram and WhatsApp group schemas omit the proposed field, so the reported validation limitation follows directly from the parser definitions.

## Solution Assessment

Yes in principle, but the best implementation needs one maintainer-approved field name and a carefully defined cross-channel scope; the separate WhatsApp name-key matching proposal should remain distinct.
