## Summary

Keep open. Current main still lacks a cross-channel optional human-readable group metadata field in the strict config schemas, TypeScript types, generated metadata, and shared docs; QQBot has a plugin-local `name` precedent, but the broader public config contract remains unresolved.

## What This Changes

Add an optional human-readable `name`, `label`, or `description` string to group/channel config entries so numeric group IDs are easier for people and agents to maintain.

## Best Possible Solution

Choose one documented optional string metadata key and add it consistently to relevant group/channel entry types, strict Zod schemas, generated metadata, docs, and validation coverage while preserving and ignoring it at runtime.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config-contract request. Source inspection gives high confidence that current main still omits and rejects the proposed field for the central Telegram/WhatsApp/Discord/Slack-style group entries.

## Solution Assessment

Yes directionally, but not as an automated cleanup yet. The narrow maintainable path is a deliberate public config-contract change after maintainers choose the field name and exact cross-channel scope.
