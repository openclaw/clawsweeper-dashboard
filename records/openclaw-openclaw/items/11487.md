## Summary

Keep open. Current main still rejects optional human-readable metadata on representative strict group/channel config entries, so this remains a public config-contract enhancement rather than cleanup that can be closed.

## What This Changes

Request adds an optional string metadata field such as `name`, `label`, or `description` to group/channel config entries for readability while leaving runtime behavior unchanged.

## Best Possible Solution

End state should be one documented optional string metadata key added consistently to group/room/channel entry contracts, generated metadata, docs, and validation coverage without changing runtime routing behavior.

## Reproduction Assessment

Yes, as a source-level reproduction. Adding `name` under `channels.telegram.groups.<id>` still hits `TelegramGroupSchema.strict()` and generated Telegram metadata with `additionalProperties: false`, so the field is not accepted on current main.

## Solution Assessment

Yes directionally, but not as an automatic fix until maintainers choose the canonical public key and exact cross-channel scope. The maintainable path is a config-contract/schema/docs/test update that leaves runtime behavior unchanged.
