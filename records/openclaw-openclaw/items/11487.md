## Summary

Keep this open. Current main and the latest release still reject the requested inert group-entry metadata in representative strict channel schemas, and the open WhatsApp name-matching work is a separate functional/security-sensitive path rather than an implementation of this broader config metadata request.

## What This Changes

The issue requests an optional human-readable metadata field, such as `name`, `label`, or `description`, on channel group config entries.

## Best Possible Solution

Keep this issue as the canonical metadata request, then add one maintainer-approved optional string field consistently across group-entry config schemas, TypeScript types, generated metadata, docs, and validation tests while keeping runtime behavior unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows the relevant group-entry schemas are strict and representative channel group config types do not accept a metadata key.

## Solution Assessment

Yes, with a product decision first. The maintainable solution is a single inert config-contract addition across schemas/types/metadata/docs, not a runtime group-name matching feature or a channel-local workaround.
