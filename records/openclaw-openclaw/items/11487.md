## Summary

Keep open. Current main and the latest release still do not accept a consistent ignored `name`/`label`/`description` metadata field on representative channel group config entries, and the related WhatsApp work is a distinct runtime name-matching feature rather than this config readability contract.

## What This Changes

Request to add an optional ignored human-readable string metadata field, such as `name`, `label`, or `description`, to channel group config entries.

## Best Possible Solution

Approve one canonical descriptive string field and cross-channel scope, then update the relevant types, strict schemas, generated config metadata, docs, and validation coverage together.

## Reproduction Assessment

Yes, at source level. Current strict group-entry schemas and representative TypeScript types omit the requested descriptive field, so an unknown `name`, `label`, or `description` key remains invalid.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable shape is one canonical metadata field across schemas, types, metadata, docs, and tests rather than ad hoc per-channel labels or multiple aliases.
