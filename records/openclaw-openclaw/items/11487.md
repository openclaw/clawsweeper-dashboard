## Summary

Current main still rejects descriptive metadata on representative group entries, and no merged canonical work implements the request. The related WhatsApp issue is a distinct runtime name-matching feature, so this issue remains useful but needs a maintainer product decision before implementation.

## What This Changes

Add one optional human-readable metadata field to strict channel group configuration entries without changing their runtime behavior or stable identifier keys.

## Best Possible Solution

Choose one canonical non-functional display field, define the coherent group/channel-entry levels that support it, preserve stable IDs as keys, and update the corresponding types, strict schemas, generated metadata, round-trip validation tests, and public configuration docs together.

## Reproduction Assessment

Yes, at source level. Representative current-main strict schemas deterministically reject an unknown descriptive field on group entries, and their corresponding TypeScript types omit it.

## Solution Assessment

Yes in broad shape, provided maintainers first choose one canonical field and a precise cross-channel boundary. Runtime matching by mutable group names is not the same solution and should remain separately tracked.
