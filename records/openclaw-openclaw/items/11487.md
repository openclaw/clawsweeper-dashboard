## Summary

Keep this issue open. Current main still lacks the requested optional human-readable metadata field on the central group/room/channel config entry surfaces, the generated config metadata still omits it for strict entries, and the docs still describe only functional per-group fields; this is a real config-contract enhancement rather than something already implemented or superseded.

## What This Changes

Request to add an optional human-readable string metadata field to group, room, or channel config entries so numeric IDs can be identified by people and agents.

## Best Possible Solution

Add one optional string metadata field to the relevant group, room, guild-channel, and channel-entry config contracts, preserve it through config load/write, ignore it at runtime, regenerate bundled config metadata, update public docs, and add validation coverage showing the field is accepted without changing group behavior.

## Reproduction Assessment

Yes. A static reproduction is adding name to a strict entry such as channels.telegram.groups.<id>; TelegramGroupSchema.strict() and generated metadata additionalProperties: false establish that current validation/tooling rejects the extra field, without needing to run a write-producing test command.

## Solution Assessment

Yes, with one constraint: the maintainable fix is a single canonical optional string field, not multiple aliases. The narrowest safe implementation is a config-contract/schema/docs/test update that leaves runtime routing and group behavior unchanged.
