## Summary

Keep open. Current main and the latest release still do not accept a consistent ignored `name`/`label`/`description` metadata field on representative channel group config entries, and the related WhatsApp work is a distinct runtime name-matching feature rather than this metadata-only config readability contract.

## What This Changes

Add an optional ignored string metadata field to channel group config entries so numeric group IDs can be annotated for humans and agents.

## Best Possible Solution

Approve one canonical descriptive string field and apply it consistently across the relevant group-entry types, strict schemas, generated metadata, docs, and validation coverage while keeping runtime behavior unchanged.

## Reproduction Assessment

Yes, at source level. Current strict representative group-entry schemas and config types omit the requested descriptive field, so unrecognized metadata remains invalid.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable shape is one canonical metadata field across schemas, types, generated metadata, docs, and tests rather than per-channel labels or multiple aliases.
