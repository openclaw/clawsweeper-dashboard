## Summary

Keep this issue open. Current main still lacks an optional human-readable metadata field on strict group/channel entry schemas and generated config metadata, so the request remains a real config-contract enhancement rather than implemented, duplicated, or out of scope.

## What This Changes

Request an optional string metadata field such as `name` or `label` on per-group/channel config entries so numeric room IDs can be identified by humans and agents.

## Best Possible Solution

Choose one canonical optional string metadata field, add it consistently to group/room/channel entry config contracts, preserve it through config load/write, ignore it at runtime, regenerate metadata, update docs, and add validation coverage.

## Reproduction Assessment

Yes. A static reproduction is adding `name` to a strict entry such as `channels.telegram.groups.<id>`; the current Zod schemas and generated metadata show that field is rejected as an unknown property.

## Solution Assessment

Yes, directionally, but the best implementation needs one canonical field name rather than accepting several aliases. The narrow maintainable fix is a config-contract/schema/docs/test update that leaves runtime routing behavior unchanged.
