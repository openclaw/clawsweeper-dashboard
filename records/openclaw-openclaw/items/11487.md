## Summary

Keep open. Current main still treats group/channel entries as strict behavior-only config, so the requested human-readable metadata field is not implemented and needs a public config-contract decision.

## What This Changes

Add an optional string metadata field such as `name`, `label`, or `description` to group/channel config entries so numeric room IDs are easier for people and agents to identify.

## Best Possible Solution

Choose one documented optional string metadata key and add it consistently to the relevant group/channel entry contracts, generated metadata, docs, and validation coverage while leaving runtime routing behavior unchanged.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, as a source-level validation path: `channels.telegram.groups.<id>.name` is absent from the strict Zod schema, TypeScript config type, and generated metadata on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but not as an automatic cleanup until maintainers choose the canonical public key and exact cross-channel scope.
