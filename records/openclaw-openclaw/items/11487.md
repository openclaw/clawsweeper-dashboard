## Summary

Keep open. Current main still treats group/channel entry objects as strict behavior config and does not accept the requested optional human-readable metadata field; prior review discussion and live issue state also show this is awaiting a config-contract product decision, not cleanup.

## What This Changes

The issue requests an optional ignored string metadata key such as `name`, `label`, or `description` on channel group config entries across supported channels.

## Best Possible Solution

Choose one documented optional string metadata key and add it consistently to the relevant group/channel entry types, strict schemas, generated metadata, docs, changelog, and validation tests while preserving and ignoring it at runtime.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config-contract request. Source inspection on current main gives high-confidence proof that strict schemas, TypeScript types, generated metadata, and docs still omit the requested metadata key.

## Solution Assessment

Unclear until maintainers choose the public key name and exact cross-channel scope. The maintainable path is a deliberate config-contract addition across schemas, types, generated metadata, docs, and tests rather than an ad hoc single-channel patch.
