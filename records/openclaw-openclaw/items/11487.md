## Summary

Keep open. Current main and v2026.6.8 still lack an ignored human-readable metadata field on channel group entries, and the related WhatsApp work is functional exact-name matching rather than this broader config-readability contract.

## What This Changes

The issue requests an optional descriptive string field such as `name`, `label`, or `description` on channel group config entries so numeric group IDs are readable to humans and agents.

## Best Possible Solution

Approve one canonical descriptive string field and apply it consistently across relevant group-entry types, strict schemas, generated metadata, docs, and validation coverage while preserving it through config load/write and ignoring it at runtime.

## Reproduction Assessment

Yes, at source level. Current strict representative group-entry schemas and config types omit the requested descriptive field, so unrecognized metadata remains invalid without needing a live channel reproduction.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable shape is one canonical metadata field across schemas, types, generated metadata, docs, and tests rather than per-channel labels or multiple aliases.
