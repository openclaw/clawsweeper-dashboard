## Summary

Keep this issue open. Current main still lacks an optional human-readable metadata field on representative strict group/channel entry schemas and generated config metadata, so this remains a config-contract enhancement that needs maintainer API naming/scope judgment.

## What This Changes

Request adds an optional string metadata field such as `name`, `label`, or `description` to group/channel config entries so numeric room IDs are readable for humans and agents.

## Best Possible Solution

Choose one canonical metadata key, add it consistently to group/room/channel entry contracts, preserve it through config load/write while ignoring it at runtime, regenerate metadata, update docs, and add validation coverage.

## Reproduction Assessment

Yes. A source-level reproduction is adding `name` under a strict entry such as `channels.telegram.groups.<id>`; current `TelegramGroupSchema.strict()` and generated metadata with `additionalProperties: false` show that field is not accepted.

## Solution Assessment

Yes directionally, but not as an automatic fix until maintainers choose the canonical public key and scope. The narrow maintainable path is a config-contract/schema/docs/test update that leaves runtime routing behavior unchanged.
