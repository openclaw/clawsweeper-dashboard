## Summary

Keep open. Current main and the latest release still do not provide a consistent ignored human-readable metadata field for group/channel config entries, and the linked WhatsApp work is a distinct runtime name-matching feature rather than this config-readability contract.

## What This Changes

Request to add one optional string metadata field such as `name`, `label`, or `description` to channel group config entries for human and agent readability.

## Best Possible Solution

Approve one canonical descriptive string field and apply it consistently across relevant group-entry types, strict schemas, generated metadata, docs, and validation coverage while keeping runtime behavior unchanged.

## Reproduction Assessment

Yes, at source level. Current strict representative group-entry schemas and config types omit the requested descriptive field, so unrecognized metadata remains invalid.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable shape is one canonical metadata field across schemas, types, generated metadata, docs, and tests rather than per-channel labels or multiple aliases.
