## What This Changes

The PR changes webchat chat-event run adoption and sessions.changed reconciliation so terminal events do not resurrect chatRunId and terminal session rows can clear stale chatRunId state.

## Best Possible Solution

Keep the terminal-event no-resurrection behavior, but make runless sessions.changed cleanup narrowly prove it belongs to the current/stale run and add real webchat proof before merge.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: a runless terminal sessions.changed event for the current session now passes the removed guard and clears whatever chatRunId is active. I did not run a browser repro for the original intermittent report.

## Solution Assessment

No. The chat.ts terminal no-adopt direction is plausible, but the sessions.ts change is too broad because it removes run ownership for no-runId terminal patches instead of proving the row belongs to the stale run.
