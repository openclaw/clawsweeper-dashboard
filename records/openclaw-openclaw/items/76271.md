This should stay open: current main exposes Discord channel/category management through plugin action discovery and runtime handlers, but the public `openclaw message channel` CLI and CLI docs still expose only `info` and `list`. The earlier ClawSweeper comment’s keep-open rationale remains valid against `15bbf4f2f304b2b116e588643fc8d99daa9f245d`.

## What This Changes

Request that Discord multi-account readiness/capability reporting and executable admin surfaces agree for channel creation and management.

## Best Possible Solution

Expose the existing Discord channel/category management actions in the message CLI and docs behind the current action gates, or explicitly document them as agent-tool-only if maintainers reject CLI exposure.

Yes. Source inspection on current main reproduces the mismatch: discovery/runtime include Discord channel-management actions, while the registered CLI and CLI docs only include `channel info` and `channel list`.

## Solution Assessment

Yes. CLI/docs parity over the existing gated runtime handlers is the narrowest maintainable fix; changing readiness or action discovery would hide a capability that already exists in the plugin path.
