## Summary

Keep open: current main and the latest release still use a non-greedy regex to skip Markdown code regions, so a literal triple-backtick sequence inside a fenced block can expose later code text to Discord mention rewriting despite the documented no-rewrite-in-code behavior.

## What This Changes

The issue reports that Discord mention alias rewriting can convert an `@handle` inside a fenced Markdown code block when the fence body contains a triple-backtick literal.

## Best Possible Solution

Replace the Discord mention rewriter's regex-based code-region skip with a Markdown-aware scanner that preserves fenced and inline code regions, then add focused regression coverage for literal backtick runs inside fences and multi-backtick inline spans without changing the `mentionAliases` config contract.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main uses a non-greedy fenced-code regex and rewrites the text after the prematurely matched literal backtick run. I could not execute the supplied repro in this checkout because `tsx` is missing, but the code path and reporter output line up.

## Solution Assessment

Yes, the best fix is a narrow Markdown code-region scanner in the Discord mention rewriter with regression tests. A config option or broad mention-policy change would be the wrong layer for this documented invariant.
