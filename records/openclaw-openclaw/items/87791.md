## Label Justifications

- `P1`: This security-sensitive CI gate can incorrectly authorize or re-block dependency graph changes in real PR workflows.
- `merge-risk: 🚨 automation`: Merging the PR changes dependency awareness from advisory automation into a hard-failing check whose approval state can become unstable.

## Summary

Keep this PR open: the move from an advisory dependency label/comment workflow to a trusted-base hard gate is useful, and the prior shell-quoting problem is fixed, but the approval state is still not safe enough for a security-sensitive required check.

## What This Changes

This PR replaces the inline dependency-awareness workflow with a checked-in Node script, adds dependency graph blocking with a `/allow-dependencies-change` override, extends CODEOWNERS, and adds workflow/script tests.

## Best Possible Solution

Keep the trusted-base script direction, but bind approvals to the exact blocked head SHA, preserve same-head approvals across reruns, add state-machine tests, and get secops approval before making the check required.

## Reproduction Assessment

Yes by source inspection: comment A after a blocked guard can be consumed for a later head B because the code stamps the current head SHA onto any newer security command. A same-head rerun after authorization also fails because the authorized guard update becomes the new freshness barrier.

## Solution Assessment

No, not yet. Moving the logic into a trusted checked-in script is a maintainable direction, but the override state machine needs SHA-bound semantics before this can safely become a required dependency gate.

- **[P1] Bind dependency approvals to the blocked head SHA:** `scripts/github/dependency-change-awareness.mjs:543-548`
  - body: The override lookup accepts any security-member command newer than the existing guard comment, then records `pullRequest.head?.sha` as approved.
