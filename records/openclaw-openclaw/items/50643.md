## Summary

Keep open. The security issue is still source-reproducible on current main, but this PR is incomplete because it hardens first-use TLS without carrying the documented `gateway.remote.tlsFingerprint` path into macOS direct mode, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds OSLog-backed `SecTrustEvaluateWithError` checking before macOS gateway TLS first-use certificate pinning.

## Best Possible Solution

Update this PR or replace it so macOS direct `wss://` first use fails closed unless system trust succeeds or an explicit configured/stored fingerprint matches, then add focused Swift coverage and redacted runtime proof for rogue rejection and explicit-pin acceptance.

## Reproduction Assessment

Yes, by source inspection. On current main, a fresh macOS direct `wss://` endpoint has no stored Keychain pin, `buildSessionBox` enables `allowTOFU`, and `GatewayTLSPinningSession` saves/accepts the presented certificate before enforcing trust success.

## Solution Assessment

No. The trust-evaluation guard is the right hardening direction, but this patch should also wire the existing documented `gateway.remote.tlsFingerprint` contract into macOS direct mode before failing closed for private TLS deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire the explicit private-TLS pin path:** `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayTLSPinning.swift:107-110`
  - body: Once this guard cancels a first-use cert that fails system trust, macOS direct mode still has no path to pass the documented `gateway.remote.tlsFingerprint` into `GatewayTLSParams`; `buildSessionBox` only loads the stored Keychain pin. Self-signed/private-CA gateways cannot use explicit pins after this change, so carry the existing config pin into the session before failing closed.
  - confidence: 0.88
