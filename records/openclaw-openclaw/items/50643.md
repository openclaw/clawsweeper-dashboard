## Summary

Keep this PR open: current main still accepts and stores a first-use macOS gateway TLS certificate in the TOFU path, but this patch is not safe to merge as-is because it rejects private-TLS first-use connections before the macOS direct node path can consume the existing explicit fingerprint config. The paired vulnerability report remains open, and the PR is currently unmergeable against main.

## What This Changes

The PR imports OSLog and adds a `SecTrustEvaluateWithError` guard and diagnostic log before `GatewayTLSPinningSession` stores and accepts a first-use gateway TLS fingerprint.

## Best Possible Solution

Replace or update this PR with a focused macOS TLS fix that rejects silent untrusted first-use TOFU, honors the existing `gateway.remote.tlsFingerprint` contract or another explicit confirmation path, and covers both rogue rejection and explicit-pin acceptance.

## Reproduction Assessment

Yes by source inspection. A fresh macOS direct `wss://` endpoint has no stored Keychain pin, `buildSessionBox` sets `allowTOFU`, and current main saves and accepts the presented certificate before any trust failure can block the connection.

## Solution Assessment

No, not as-is. The trust-evaluation gate is the right hardening direction, but the maintainable fix should preserve the documented explicit-pin path or an explicit user-confirmed enrollment path instead of making private TLS direct macOS setups fail closed with no app-side escape hatch.
