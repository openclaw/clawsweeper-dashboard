## Summary

Keep open. This PR moves in the right security direction, but it still blocks the documented explicit-pin path for private or self-signed macOS direct WSS gateways because the macOS config/session path does not carry `gateway.remote.tlsFingerprint` into `expectedFingerprint`; the paired vulnerability report at https://github.com/openclaw/openclaw/issues/50642 also remains open.

## What This Changes

This PR adds OSLog-backed trust evaluation before `GatewayTLSPinningSession` stores and accepts a first-use macOS gateway TLS certificate.

## Best Possible Solution

Update or replace this PR so macOS direct WSS first use fails closed unless system trust succeeds or the existing configured/confirmed fingerprint matches, with Swift regression coverage for rogue rejection and explicit-pin acceptance.

## Reproduction Assessment

Yes, by source inspection. A fresh macOS direct WSS endpoint has no keychain pin, `buildSessionBox` enables `allowTOFU`, and current main accepts/stores the presented certificate before enforcing system trust.

## Solution Assessment

No. The trust-evaluation guard is the right hardening direction, but this patch should also wire the documented `gateway.remote.tlsFingerprint` path into macOS direct mode before failing closed for private TLS deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire the explicit private-TLS pin path:** `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayTLSPinning.swift:107-110`
