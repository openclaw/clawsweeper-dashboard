## Summary

Keep open. The PR is aimed at a real macOS direct WSS TLS trust problem, but the proposed fail-closed TOFU guard still blocks private/self-signed first-use endpoints before the macOS app has any documented explicit fingerprint path wired in, and the security-sensitive behavior needs maintainer review before merge.

## What This Changes

The branch adds OSLog diagnostics and a SecTrustEvaluateWithError guard before GatewayTLSPinningSession saves and accepts a first-use gateway TLS fingerprint.

## Best Possible Solution

Rebase or replace the PR with a focused macOS TLS fix that rejects silent untrusted first-use TOFU, wires the existing gateway.remote.tlsFingerprint contract into direct macOS node mode or another explicit confirmation path, and covers both rogue first-use rejection and explicit-pin acceptance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a fresh macOS direct wss:// endpoint has no stored Keychain pin, buildSessionBox sets allowTOFU, and current main accepts and stores the presented fingerprint; the PR then rejects untrusted first-use certificates before any explicit macOS config pin can be supplied.

## Solution Assessment

No, not as-is. The trust-evaluation gate is the right hardening direction, but the maintainable fix should preserve a documented explicit-pin or user-confirmed enrollment path instead of making private TLS direct macOS setups fail closed with no app-side escape hatch.
