## Summary

Keep open. The security issue is still present on current main and the PR is directionally useful, but the branch needs current-main integration, an explicit pin enrollment path for macOS direct mode, and real behavior proof before it is mergeable.

## What This Changes

The PR adds a system-trust check and OSLog diagnostic before macOS Gateway TLS TOFU saves and accepts a first-use certificate.

## Best Possible Solution

Rebase the hardening onto current main by rejecting untrusted first-use TLS while preserving current TLS failure reporting and wiring the existing explicit fingerprint path, or an explicit confirmation flow, into macOS direct node setup.

## Reproduction Assessment

Yes, from source inspection: a fresh macOS direct wss endpoint with no keychain pin enters allowTOFU and accepts the certificate before a failed trust result can block it. I did not run a live rogue gateway.

## Solution Assessment

No. The trust-evaluation guard is the right hardening direction, but this PR only patches the old TOFU branch and leaves macOS without a documented explicit-pin enrollment path for private TLS endpoints.
