## Summary

Keep open. Current main still auto-pins and accepts a fresh macOS direct WSS certificate before enforcing `SecTrustEvaluateWithError`, while this PR is conflict-dirty against current main and its fail-closed approach still lacks a wired macOS explicit-pin path for documented `gateway.remote.tlsFingerprint`. The security direction is valid, but the branch needs maintainer security/product review and a rebase onto the current TLS failure plumbing before it can land.

## What This Changes

The PR adds `OSLog` logging and a `SecTrustEvaluateWithError` guard before `GatewayTLSPinningSession` stores and accepts a first-use TLS fingerprint.

## Best Possible Solution

Harden macOS direct WSS first use so a fresh endpoint is accepted only after normal system trust validation or an explicit out-of-band pin/user confirmation. The maintainable fix should rebase onto current `GatewayTLSValidationError` plumbing, wire or document `gateway.remote.tlsFingerprint` for the macOS direct node path, and add focused regression coverage for rogue first-use certificates plus the intended explicit-pin path.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure a fresh macOS direct `wss://` endpoint with no Keychain pin, observe `buildSessionBox` sets `allowTOFU: true`, and current `GatewayTLSPinningSession` saves and accepts the presented fingerprint before rejecting on `systemTrustOk`. I did not run a live rogue WebSocket server because this review was required to stay read-only.

## Solution Assessment

No, not as-is. The PR’s trust-evaluation gate is the right hardening direction, but the best fix should be rebased onto current main and should preserve a documented explicit-pin or user-confirmed enrollment path rather than leaving macOS direct private TLS with no app-side escape hatch.
