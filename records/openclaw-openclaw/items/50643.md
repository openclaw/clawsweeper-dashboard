## Summary

Keep open. The vulnerability is still present on current main, and this PR is useful but not merge-ready because it blocks untrusted first-use TLS before the macOS direct node path can consume the documented explicit fingerprint contract.

## What This Changes

The PR imports OSLog and adds a trust-evaluation guard with logging before macOS GatewayTLSPinningSession stores and accepts a first-use TOFU certificate.

## Best Possible Solution

Update this PR or replace it with a focused macOS TLS fix that rejects silent untrusted first-use TOFU, honors gateway.remote.tlsFingerprint for explicit pins, and adds regression coverage for rogue rejection and explicit-pin acceptance.

## Reproduction Assessment

Yes, by source inspection. A fresh macOS direct wss:// endpoint has no stored Keychain pin, buildSessionBox enables allowTOFU, and current main stores and accepts the presented certificate before system trust can reject it.

## Solution Assessment

No, not as-is. Requiring trust evaluation is the right hardening direction, but the patch should also wire the documented explicit fingerprint path into macOS direct node mode so private TLS deployments have a secure enrollment path.

## Review Findings

Overall correctness: patch is incorrect
