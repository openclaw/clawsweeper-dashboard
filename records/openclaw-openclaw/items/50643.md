## Summary

Keep open: current main still has the macOS first-use WSS TOFU path, and this PR is a plausible security hardening direction but is not merge-ready because it disables first-use enrollment for non-system-trusted gateways without wiring the documented explicit fingerprint path into the macOS direct node flow.

## What This Changes

The PR adds a `SecTrustEvaluateWithError` check and OSLog diagnostic before `GatewayTLSPinningSession` stores and accepts a first-use TLS certificate.

## Best Possible Solution

Adapt current main so macOS direct `wss://` first use succeeds only for a system-trusted certificate or an explicit configured/confirmed fingerprint, plumb `gateway.remote.tlsFingerprint` into `GatewayTLSParams`, and cover both paths with Swift tests plus real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main builds `allowTOFU: true` for a fresh macOS direct `wss://` host and the TLS delegate saves and accepts the first presented certificate before any trust-failure branch; I did not run a live macOS MITM setup.

## Solution Assessment

No. Adding a trust-evaluation gate is the right hardening primitive, but this PR is not the best complete fix until the macOS direct path supports the documented explicit fingerprint enrollment and is rebased onto the current TLS failure model.
