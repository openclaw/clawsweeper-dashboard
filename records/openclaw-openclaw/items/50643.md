## Summary

Keep open. Current main still accepts and stores the first presented macOS node TLS certificate in the `allowTOFU` path without enforcing the system trust result, so the reported security problem is not implemented on main. The PR is a focused security fix candidate, but it needs maintainer security/product review because the exact diff can break first-time private-TLS direct endpoints unless the macOS app gets an explicit pin or confirmation path.

## What This Changes

The PR adds `OSLog` logging and a `SecTrustEvaluateWithError` gate before `GatewayTLSPinningSession` saves and accepts a first-use TLS fingerprint.

## Best Possible Solution

Harden macOS node first-use TLS so a fresh direct WSS connection is accepted only after normal system trust validation or an explicit out-of-band pin/user confirmation. The maintainable end state should integrate with current `GatewayTLSValidationError` reporting, consume or document `gateway.remote.tlsFingerprint` for macOS direct mode, and include focused regression coverage for rogue first-use certificates and the intended private-TLS enrollment path.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure a fresh macOS direct `wss://` endpoint with no stored Keychain pin, observe `MacNodeModeCoordinator` sets `allowTOFU: true`, and `GatewayTLSPinningSession` saves and accepts the presented fingerprint before using `systemTrustOk` to reject it. I did not run a live rogue WebSocket server because this review was required to stay read-only.

## Solution Assessment

No, not as-is. The PR's security direction is right, but the best fix should be rebased onto current main's TLS failure plumbing and should preserve a documented explicit-pin or user-confirmed enrollment path before removing silent TOFU.
