## Summary

Keep open. The PR targets a real security bug, but the patch is incomplete because macOS direct mode still does not carry the existing `gateway.remote.tlsFingerprint` contract into `GatewayTLSParams`, and the contributor has not provided real after-fix macOS TLS proof.

## What This Changes

The PR adds OSLog-backed `SecTrustEvaluateWithError` checking before saving and accepting first-use macOS gateway TLS pins in `GatewayTLSPinningSession`.

## Best Possible Solution

Harden macOS direct `wss://` first use so it fails closed unless system trust succeeds or an explicit configured/confirmed fingerprint matches, then add focused Swift coverage and redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main sets `allowTOFU` for a fresh macOS direct `wss://` endpoint and the TLS delegate saves/accepts that first certificate before enforcing `systemTrustOk`.

## Solution Assessment

No. The trust-evaluation guard is the right direction, but this patch should also wire the existing configured or confirmed fingerprint path into macOS direct mode before failing closed.
