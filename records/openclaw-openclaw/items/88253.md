## Label Justifications

- `P1`: Sandboxed channel-bound agents can silently fail to deliver DM replies, breaking a real user-facing channel workflow.
- `impact:message-loss`: The reported symptom is a generated reply that never reaches the originating Telegram DM.
- `impact:security`: The likely fix changes the sandbox default channel-send capability and therefore the sandbox security boundary.
- `impact:session-state`: The failure depends on sandboxed session routing and fallback to an internal webchat lane instead of the inbound channel context.

## Summary

Keep open: current main and the latest release still omit `message` from the default sandbox allowlist, so sandboxed channel-bound agents can still lose the deliverable channel reply tool. The closed related PR attempted the narrow allowlist fix but was closed unmerged, and broader routing issues do not fully supersede this sandbox-default/security decision.

## What This Changes

Allow sandboxed channel-bound agents to reply to originating DMs by making a deliverable `message` path available under sandbox policy and fixing internal webchat/plain-text routing.

## Best Possible Solution

Have maintainers choose the sandbox-default policy, then land a focused fix that preserves explicit deny behavior, adds regression coverage, and includes redacted live Telegram or Discord proof; keep broader out-of-session binding work on https://github.com/openclaw/openclaw/issues/84989 and announce-routing work on https://github.com/openclaw/openclaw/issues/27445.

## Reproduction Assessment

Yes at source level: current main omits `message` from the nonempty sandbox default allowlist, applies that allowlist to runtime tools, and leaves `sessions_send` on internal non-delivery routing. I did not run a live Telegram/Docker reproduction in this read-only review.

## Solution Assessment

Unclear: adding `message` to the default sandbox allowlist is the narrowest fix for the tool-availability half, but it needs maintainer approval and live proof because it expands default sandbox channel egress. The plain-text webchat fallback likely needs separate routing verification.
