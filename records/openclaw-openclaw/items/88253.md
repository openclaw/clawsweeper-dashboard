## Summary

Keep open: current main and the latest release still omit `message` from the default sandbox allowlist, the linked fix PR closed unmerged, and the direct allowlist fix needs maintainer acceptance of the sandbox channel-egress change plus live transport proof.

## What This Changes

The issue reports that sandboxed channel-bound agents cannot deliver visible DM replies because `message` is not allowed by the default sandbox policy and fallback text output is not routed to the originating channel.

## Best Possible Solution

Keep this issue open for a maintainer decision on the sandbox reply contract: either accept and prove default `message` egress for sandboxed channel agents, or keep `message` opt-in and add a narrower no-delivery diagnostic or routing fix.

## Reproduction Assessment

Yes, by source inspection: current `DEFAULT_TOOL_ALLOW` still omits `message`, the sandbox policy uses that default, and `sessions_send` sends through an internal non-delivery lane. I did not run a live Telegram sandbox reproduction in this read-only review.

## Solution Assessment

Unclear. Adding `message` to the default allowlist is the direct fix for the message-tool half, but it expands sandbox channel egress and needs maintainer acceptance plus live channel proof; a narrower diagnostic or opt-in path may be preferable.
