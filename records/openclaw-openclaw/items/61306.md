## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Validate mission ids before resolving paths:** `src/claw/service.ts:1717-1722`
  - body: `missionId` comes from Gateway RPC params and is passed through `resolveMissionStatePath()` into `path.join(...)` without checking the generated id format or normalized containment. A crafted id containing `..` or separators can escape the missions root and target a different `mission-state.json` path, so validate mission ids before any filesystem resolution.
  - confidence: 0.91
- **[P1] Keep mission runtime state in SQLite:** `src/claw/service.ts:67-70`
  - body: This adds mission, control, and audit runtime state as JSON/JSONL files under the workspace. Current repository policy treats OpenClaw-owned runtime state as SQLite-backed unless it is a named artifact, so this would create durable sidecar state without an approved migration or upgrade model.
  - confidence: 0.87
- **[P1] Preserve HTTP/1-only pinned dispatchers:** `src/infra/net/ssrf.ts:430-447`
  - body: The PR constructs Undici dispatchers directly here instead of current main's `createHttp1*` helpers, which enforce `allowH2: false` and related proxy/timeout policy. That can re-enable HTTP/2 on SSRF-guarded pinned requests and weaken the network boundary those helpers protect.
  - confidence: 0.9
- **[P1] Revalidate mission status after agent calls:** `src/claw/service.ts:2954-2960`
  - body: After `runMissionAgent()` returns, this path mutates and persists the pre-call `state` without reloading the current mission. If an operator pauses or cancels the mission while the agent is in flight, the returned runner/helper/verifier branch can overwrite that control action and resume or complete the mission unexpectedly.
  - confidence: 0.86
