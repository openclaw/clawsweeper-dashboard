## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate mission IDs before resolving mission paths:** `src/claw/service.ts:1717-1718`
  - body: RPC handlers only require a non-empty `missionId`, but this resolver joins that value directly under the missions root. A value containing `..` or path separators can escape the mission directory for state, audit, and artifact reads/writes, so mission IDs need generated-format validation or a normalized containment check before any filesystem access.
  - confidence: 0.9
- **[P1] Preserve HTTP/1-only pinned dispatchers:** `src/infra/net/ssrf.ts:430-447`
  - body: Current main routes pinned SSRF dispatchers through the `createHttp1*` helpers, where the guarded networking path enforces the HTTP/1-only contract. Constructing Undici agents directly here can re-enable HTTP/2 on SSRF-guarded requests and weaken the pinned-lookup assumptions, so this path should keep the wrappers or preserve the same `allowH2: false` behavior explicitly.
  - confidence: 0.88
- **[P1] Move mission runtime state to approved storage:** `src/claw/service.ts:1774`
  - body: This writes mission runtime state to `mission-state.json`, with companion control/audit files under the workspace mission folder. OpenClaw-owned runtime state is supposed to live in an approved SQLite-backed store unless maintainers explicitly classify the files as product artifacts with migration/upgrade proof, so the durable mission model needs to move or be approved before merge.
  - confidence: 0.86
- **[P1] Revalidate mission state after agent runs:** `src/claw/service.ts:2922`
  - body: The runner/helper/verifier paths await long-running agent calls and then mutate the same in-memory mission record. If an operator pauses or cancels the mission while the agent call is in flight, these branches can persist over that control action; reload and validate the live mission status before applying any post-run transition.
  - confidence: 0.84
