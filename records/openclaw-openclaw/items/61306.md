## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate mission ids before resolving paths:** `src/claw/service.ts:1717-1722`
  - body: Gateway handlers pass caller-supplied `missionId` strings through after only trimming them, and this resolver joins that value directly under the missions root. A value containing path separators or `..` can make reads and later mutations target paths outside the mission directory, so mission ids need format validation or resolved-path containment before any file access.
  - confidence: 0.91
- **[P1] Keep mission runtime state in SQLite:** `src/claw/service.ts:67-70`
  - body: These constants establish Claw mission, control, and audit data as durable JSON/JSONL sidecars. Current runtime state policy and adjacent task-flow persistence are SQLite-backed, so landing a parallel file store here creates migration and upgrade debt for a brand-new stateful subsystem.
  - confidence: 0.88
- **[P1] Preserve HTTP/1-only pinned dispatchers:** `src/infra/net/ssrf.ts:430-447`
  - body: This path now constructs Undici dispatchers directly instead of the current `createHttp1*` helpers. That bypasses `allowH2: false` plus the managed proxy TLS, timeout, and IP-safe proxy-client handling used by guarded SSRF fetches, so pinned requests can lose the security assumptions those helpers enforce.
  - confidence: 0.93
- **[P1] Revalidate mission status after agent calls:** `src/claw/service.ts:3050-3079`
  - body: The verifier, runner, and helper paths await a long `runMissionAgent` call and then mutate the same in-memory mission record they loaded before the call. If an operator pauses or cancels the mission while the agent is running, these branches can persist `done`, `running`, or `blocked` over that newer control action; reload and validate the live mission state before applying outcomes.
  - confidence: 0.87
