## Summary

Keep open: current main has added useful Codex app-server timeout watches, client retirement, replay-safe retry, and diagnostic abort-drain recovery, but I could not prove the full reported inter-session stall is solved. The remaining `sessions_send` timeout behavior still intentionally preserves delayed target replies, so the cancel-versus-detach policy needs maintainer runtime judgment.

## What This Changes

The issue asks OpenClaw to recover Codex app-server `model_call` stalls after a `sessions_send` timeout, release the caller, safely cancel or detach the target, and prevent one stuck delegation from wedging unrelated sessions.

## Best Possible Solution

Keep the current timeout and recovery primitives, add a focused repro for post-tool Codex silence plus inter-session timeout, then make the maintainer-approved cancel/detach/recovery semantics explicit before closing this issue.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source and tests do show the relevant dynamic-tool timeout, wait-only `sessions_send` timeout, delayed A2A reply behavior, and partial recovery paths, so the failure family is source-reproducible but not fully proven end to end.

## Solution Assessment

Unclear: current main is a safer partial direction, but it is not enough to call the issue solved. The best fix needs maintainer-approved `sessions_send` cancel versus detach semantics plus proof that Codex app-server and diagnostic recovery isolate unrelated sessions.
