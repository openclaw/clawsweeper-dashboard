## Summary

Keep open: this is a focused fix for the open GBK read bug, but the current patch is not mergeable because the new parameter is not wired into execution and the advertised GBK path uses unsupported Buffer encodings; the supplied proof is mock-only.

## What This Changes

The PR adds an optional `encoding` parameter to the built-in read tool, a decoding helper, and regression tests for GBK and default UTF-8 reads.

## Best Possible Solution

Land one canonical fix for https://github.com/openclaw/openclaw/issues/92664 that decodes GBK through a supported decoder/codepage path, keeps the read schema/input/executor contract aligned, and includes real read-tool proof on a GBK file.

## Reproduction Assessment

Yes for the PR defects: CI and source inspection show the patched read path throws because `encoding` is undefined, and Node rejects the advertised Buffer encodings. The linked GBK bug is source-reproducible from current main's UTF-8-only decode path, but I did not run a live Windows repro.

## Solution Assessment

No. Explicit encoding is a plausible fix direction, but this patch is not the best current solution because it is not wired into execution and should use `TextDecoder` or the existing Windows codepage helper instead of casting arbitrary strings to `BufferEncoding`.

Full review comments:

- **[P1] Wire encoding through the read executor:** `src/agents/sessions/tools/read.ts:374`
  - body: The new call passes `encoding` into `decodeBuffer`, but `execute` still destructures only `{ path, offset, limit }`. Any text read now throws `ReferenceError: encoding is not defined`; CI shows the read-tool shard failing at this line. Add the parameter to the executor and input contract before using it.
  - confidence: 0.99
- **[P1] Use a decoder that supports the advertised encodings:** `src/agents/sessions/tools/read.ts:74`
  - body: The fallback calls `Buffer.toString(encoding as BufferEncoding)`, but Node's `Buffer` does not support `gbk`, `gb2312`, `shift_jis`, or `utf-16be`; `Buffer.from(...).toString('gbk')` throws `TypeError: Unknown encoding: gbk`. Use `TextDecoder` or the existing Windows codepage decoder instead of casting arbitrary strings to `BufferEncoding`.
  - confidence: 0.97
