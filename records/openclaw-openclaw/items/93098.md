## Summary

The current patch is useful but not merge-ready: it proves explicit GBK decoding, yet it also advertises `encoding: "auto"` as auto-detection even though the implementation only checks BOMs and otherwise decodes as UTF-8. Several sibling PRs are still open for the same bug, so this PR should stay open for maintainer/API direction rather than be closed as duplicate or implemented.

## What This Changes

The PR adds an optional `encoding` argument to the built-in session read tool, decodes text buffers with `TextDecoder`/fallback logic, updates `ReadToolInput`, and adds GBK plus default UTF-8 tests.

## Best Possible Solution

Choose one canonical fix for https://github.com/openclaw/openclaw/issues/92664: either make an explicit `read.encoding` API accurate and intentionally supported, or reuse the existing Windows codepage fallback without expanding the read-tool schema.

## Reproduction Assessment

Yes, at source level: current main decodes read-tool text buffers as UTF-8, which matches the linked GBK mojibake report. I did not run a live Chinese Windows gateway session in this read-only review.

## Solution Assessment

No as written: explicit `encoding` is a plausible mitigation, but the advertised `auto` behavior is inaccurate and maintainers still need to choose the new API versus the existing Windows codepage fallback.
