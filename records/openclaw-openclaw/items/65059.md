## Summary

Keep open. The PR targets a valid MiniMax global-endpoint usage bug, but the proposed host selection can route a bearer token to a host chosen from a different provider config than the credential source, so it should not merge as-is.

## What This Changes

The PR adds baseUrl-aware MiniMax usage host derivation and passes a provider-config baseUrl into MiniMax usage polling so `coding_plan/remains` can use a CN or global host.

## Best Possible Solution

Implement source-aligned MiniMax usage host resolution: carry or derive the usage host from the same credential source that supplied the token, preserve CN/global fallback behavior intentionally, add focused URL-selection tests, and reconcile the overlapping #63073 and #65054 work before landing.

## Reproduction Assessment

Yes. The current-main bug is reproducible from source by configuring MiniMax with a global `baseUrl` and observing that `fetchMinimaxUsage` still calls the CN host; the PR-specific regression is reproducible with both `minimax` and `minimax-portal` configured in different regions and no portal OAuth token.

## Solution Assessment

No. Reading the configured host is the right direction, but this patch is not the best fix because it selects host priority independently from credential priority; the safer solution binds token source and usage host together and covers that with tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Select the usage host from the credential source:** `extensions/minimax/provider-registration.ts:248-250`
  - body: The new host selection prefers `minimax-portal` `baseUrl`, but usage auth can fall back to an API key from `minimax` first when no portal OAuth token exists. With both provider configs present in different regions, a `minimax` key can be sent to the other region's usage host, causing the same 403 or wrong-quota behavior this patch is trying to fix.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain changes, but it changes MiniMax bearer-token routing and can send a token to a host selected from a different provider config than the credential source.

Concerns:

- **[medium] Bearer token can be routed to the wrong MiniMax host:** `extensions/minimax/provider-registration.ts:248`
  - body: Because the PR chooses the `minimax-portal` baseUrl before the `minimax` baseUrl while auth can choose a `minimax` API key first, dual-region configs can send a bearer token to the wrong regional endpoint. The patch should align token source and host before merge.
  - confidence: 0.86
