## Summary

Keep this PR open: current main still has the fixed-interval Discord heartbeat ACK race and the production change targets the right owner module, but the PR needs test cleanup, a changelog entry, and contributor-provided real behavior proof before merge.

## What This Changes

The PR changes Discord gateway heartbeat scheduling from a fixed `setInterval` ACK check to recursive `setTimeout` scheduling after each heartbeat and adds fake-timer coverage for the lifecycle helper.

## Best Possible Solution

Land a narrow Discord-owned fix that schedules ACK checks relative to sent heartbeats, preserves genuine timeout reconnect behavior, and keeps tests/changelog/proof aligned with repo policy.

Yes, source inspection gives a high-confidence reproduction path: set the first heartbeat random delay near the full interval, let the heartbeat set `lastHeartbeatAck` false, and the current fixed interval can check before an ACK round trip. I did not run a live Discord gateway reproduction.
