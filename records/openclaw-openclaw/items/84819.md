## Best Possible Solution

Rebase and narrow the PR to the Control UI locale-picker work, move Vietnamese copy into the generated locale map path, and prove both login and authenticated dashboard locale switching in a browser.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows viDashboardText reads i18n.getLocale() immediately while several module-scope constants call it during import. The browser behavior claimed by the PR is not fully reproduced because the provided proof covers only the login gate.

## Solution Assessment

No; the current implementation is not the best path because it adds a manual Vietnamese-only translation layer instead of extending the existing locale-map pipeline. The safer solution is a focused UI picker change plus generated vi locale entries and render-time label resolution.
