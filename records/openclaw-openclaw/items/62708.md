## Summary

Keep open. Current main still treats `claudeCliSessionId` as an active legacy compatibility field, including fallback reads, dual-writes, preservation/reset handling, and tests that intentionally protect old Claude CLI session resume behavior. The cleanup is concrete, but it needs a maintainer compatibility decision before removal.

## What This Changes

The issue requests removing the legacy `claudeCliSessionId` field from `SessionEntry` and deleting related fallback, dual-write, preservation, and test coverage paths.

## Best Possible Solution

Make an explicit compatibility decision for legacy Claude-only session rows. If OpenClaw is ready to retire that path, remove the field from the session type, CLI helper fallback and dual-write paths, preservation/reset patches, Claude history import fallback, and tests; otherwise keep the fallback and document the reason or deprecation plan.

## Reproduction Assessment

Not applicable as a runtime bug. The current state is reproducible with high confidence by static inspection: `rg claudeCliSessionId` and focused file reads show active type, read, write, preservation, and test coverage on current main.

## Solution Assessment

Unclear without maintainer compatibility judgment. The requested deletion is the narrow cleanup if old-session resume support can be dropped, but current tests show the safer existing behavior is intentionally preserving that compatibility.
