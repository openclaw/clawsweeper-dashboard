## Summary

Keep open. Current main still treats `claudeCliSessionId` as an active legacy compatibility field, including fallback reads, dual-writes, preservation/reset paths, and tests that protect legacy Claude CLI resume/import behavior.

## What This Changes

The issue requests removing the legacy `claudeCliSessionId` field from session state and deleting its read/write compatibility paths.

## Best Possible Solution

Make an explicit compatibility decision for legacy Claude-only session rows; if OpenClaw is ready to retire that path, remove the type field, helper fallback and dual-write paths, preservation/reset handling, Claude history fallback, and tests, otherwise keep the compatibility path and document the retention/deprecation plan.

## Reproduction Assessment

Not applicable as a runtime bug. The current state is reproducible with high confidence by static inspection: focused `rg` and file reads show active type, read, write, preservation, and test coverage on current main.

## Solution Assessment

Unclear without maintainer compatibility judgment. The requested deletion is the narrow cleanup only if old-session resume/import support can be dropped; current tests show the safer existing behavior intentionally preserves that compatibility.
