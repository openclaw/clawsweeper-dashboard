## Summary

Keep open: current main still has the alphanumeric-only Feishu bitable URL parser, but this PR is not merge-ready because the patch still truncates any future token character outside `-` and `_`, lacks real behavior proof, and conflicts with current main's existing bitable test file.

## What This Changes

The PR widens the Feishu bitable URL parser to accept hyphen/underscore tokens, exports the parser/meta helpers for direct tests, includes the invalid URL in `getBitableMeta` errors, and adds unit tests.

## Best Possible Solution

Refresh the branch, merge the new cases into the existing bitable test file, parse the complete `/base/` or `/wiki/` path segment without allow-list truncation, and add redacted real Feishu/Lark proof or a maintainer proof override before merge.

## Reproduction Assessment

Yes from source inspection: current main's `parseBitableUrl` captures only the alphanumeric prefix, so `/base/abc-123_xyz` would parse as `abc`. I did not run a live Lark tenant, so the behavior is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The direction is reasonable, but the best fix is to capture the whole path segment and let the Lark API validate it, not keep expanding a character allow-list that can still truncate future token shapes.
