## Summary

Keep open. The PR addresses a plausible provenance bug, but the current one-line patch is still not mergeable because it reads the raw requester origin instead of the normalized, possibly re-resolved announce origin, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes the subagent completion announce path in `src/agents/subagent-announce.ts` to pass `params.requesterOrigin?.channel` as `sourceChannel` instead of always using the internal webchat channel.

## Best Possible Solution

Derive announce provenance from the normalized resolved origin chain, with the internal channel only as the final fallback, then add focused regression coverage for `inputProvenance.sourceChannel` and redacted Telegram or Discord proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction for the PR defect: raw or absent `requesterOrigin` can differ from `targetRequesterOrigin`, `directOrigin`, or `completionDirectOrigin`, while delivery copies the caller-supplied `sourceChannel` into `inputProvenance`.

## Solution Assessment

No. The patch points at the right metadata field, but the maintainable fix should use the already-normalized and possibly re-resolved origin chain rather than raw `params.requesterOrigin`.
