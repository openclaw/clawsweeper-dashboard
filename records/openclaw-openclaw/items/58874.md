## Summary

Keep this PR open. It targets a real subagent provenance bug that still exists on current main, but the current one-line fix reads the raw requester origin instead of the normalized and possibly re-resolved requester origin that the announce flow actually delivers to, so important non-webchat fallback paths can still be tagged as webchat or with stale/raw channel text.

## What This Changes

The PR changes `src/agents/subagent-announce.ts` so completion announcements pass `params.requesterOrigin?.channel ?? INTERNAL_MESSAGE_CHANNEL` as `sourceChannel` instead of always passing `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Update the PR so completion announce provenance is derived from the same normalized and resolved delivery origin used for the requester handoff, add focused regression assertions for non-webchat sourceChannel and missing-requester fallback, and add the required changelog entry. Keep the timeout, fire-and-forget, and concurrency suggestions from the discussion as separate follow-up work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run `runSubagentAnnounceFlow` with a non-webchat requester or with a missing requester subagent that resolves to a WhatsApp/Telegram parent; current main hardcodes webchat, and this PR still misses the recovered/normalized origin in the fallback case.

## Solution Assessment

No. The PR points at the right field, but reading raw `params.requesterOrigin` is not the narrowest maintainable fix because the flow already normalizes and may replace the requester origin before delivery.
