## Summary

Keep open: the reported bug is still real, but this PR is not sufficient because it reads raw requester params instead of the normalized/re-resolved delivery origin used by the announce flow.

## What This Changes

The PR changes the completion announce `sourceChannel` argument in `src/agents/subagent-announce.ts` from the internal channel to `params.requesterOrigin?.channel` with the same fallback.

## Best Possible Solution

Derive completion provenance from the normalized resolved delivery origin, cover normal and reparented external-channel paths, and leave timeout/concurrency changes for separate work.

## Reproduction Assessment

Yes, at source level. In a missing requester-subagent fallback, `resolveRequesterForChildSession` can supply a WhatsApp or Discord parent that delivery uses, while this PR would still read absent or stale original params and can fall back to webchat.

## Solution Assessment

No. The PR targets the right bug, but the maintainable fix is to derive `sourceChannel` from `completionDirectOrigin`, `directOrigin`, or `targetRequesterOrigin` after normalization and fallback resolution.
