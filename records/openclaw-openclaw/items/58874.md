## Summary

Keep this PR open, but it still needs changes before merge. Current main still sends `webchat` as the completion announce provenance channel, while this branch reads raw `params.requesterOrigin`, which misses normalized and re-resolved requester origins; the later Discord canary comment supports that this area is still worth fixing.

## What This Changes

The PR changes `runSubagentAnnounceFlow` to pass `params.requesterOrigin?.channel` as the subagent completion announce `sourceChannel` instead of `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Use the normalized, resolved delivery origin chain for completion provenance, with the internal channel as the final fallback, and add regression assertions for non-webchat, normalized, and reparented requester channels.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a completion announce whose requester subagent session is gone and `resolveRequesterForChildSession` returns a WhatsApp or Discord parent; delivery uses the recovered origin while this PR would still read raw or missing `params.requesterOrigin`.

## Solution Assessment

No. The PR points at the right field, but raw `params.requesterOrigin` is not the best source because it bypasses normalization and the flow's resolved `targetRequesterOrigin`, `directOrigin`, and `completionDirectOrigin`.
