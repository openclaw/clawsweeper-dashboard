## Summary

Keep open: the PR targets a real provenance gap, but the one-line patch reads raw `params.requesterOrigin` after the flow has already normalized or re-resolved delivery origin, so it can still emit stale, missing, or unnormalized `sourceChannel` and it lacks after-fix real behavior proof.

## What This Changes

The branch changes completion announces in `src/agents/subagent-announce.ts` to set `sourceChannel` from `params.requesterOrigin?.channel` instead of `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Derive announce provenance from the normalized resolved origin chain, with the internal channel only as the final fallback, then add focused regression coverage and redacted Telegram or Discord proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction for the PR defect: raw or absent `requesterOrigin` can differ from `targetRequesterOrigin`, `directOrigin`, or `completionDirectOrigin`, while delivery copies the caller-supplied `sourceChannel` into `inputProvenance`.

## Solution Assessment

No. The patch points at the right metadata, but the maintainable fix should use the already-normalized and possibly re-resolved origin chain instead of raw `params.requesterOrigin`.
