## Summary

Keep open: the PR targets a real announce-provenance gap, but the current patch reads raw `params.requesterOrigin` instead of the normalized/resolved origin chain and the external PR still lacks after-fix real behavior proof.

## What This Changes

This PR changes the subagent completion announce path to pass the requester origin channel into `sourceChannel` instead of always using the internal webchat channel.

## Best Possible Solution

Derive completion provenance from the normalized resolved origin chain, with the internal channel only as the final fallback, and add regression assertions for non-webchat, normalized, and fallback requester-origin cases.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction: when the requester subagent session is gone and `resolveRequesterForChildSession` recovers a WhatsApp or Discord parent, delivery uses the recovered origin while this PR would still read raw or missing `params.requesterOrigin`. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, this is not the best fix as written. The patch points at the right field, but it should use the already normalized and resolved origin chain rather than raw PR input.
