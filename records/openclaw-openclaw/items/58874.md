## Summary

Keep open: the bug is real and current main still has it, but this PR is not merge-ready because the changed line uses raw requester origin rather than the normalized or recovered delivery origin, leaving fallback requester paths with stale or webchat provenance.

## What This Changes

The PR changes completion subagent announce provenance to pass the requester channel as `sourceChannel` instead of always using the internal webchat channel.

## Best Possible Solution

Use the resolved delivery origin for completion announce provenance and add focused regression coverage for non-webchat, normalized, and reparented requester channels; keep timeout and concurrency suggestions as separate work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a completion announce where the requester subagent session is missing and `resolveRequesterForChildSession` returns a WhatsApp or Discord parent; delivery uses that recovered origin while the PR would still read the original params and can fall back to webchat.

## Solution Assessment

No. The PR points at the right behavior, but raw `params.requesterOrigin` is not the narrowest maintainable source; `sourceChannel` should come from `completionDirectOrigin`, `directOrigin`, or `targetRequesterOrigin` with the internal fallback last.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the resolved requester origin for provenance:** `src/agents/subagent-announce.ts:559`
  - body: The announce flow can replace `targetRequesterOrigin` when a missing requester subagent is reparented, and the direct/completion origin helpers can also recover the actual channel. Reading `params.requesterOrigin` here leaves that path reporting `webchat` or an unnormalized raw channel even while delivery goes to the recovered external channel, so derive `sourceChannel` from the resolved origin used for delivery.
