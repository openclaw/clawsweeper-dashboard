## Summary

Keep open: the PR targets a real subagent announce provenance gap, but the one-line patch is not correct as-is because it reads the raw requester origin instead of the resolved and normalized origin chain, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes subagent completion announcements to set `sourceChannel` from `params.requesterOrigin?.channel` instead of always using `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Derive announce provenance from `completionDirectOrigin`, `directOrigin`, or `targetRequesterOrigin` after normalization, fall back to the internal channel last, and add focused regression coverage plus redacted Telegram or Discord proof.

## Reproduction Assessment

Yes for the PR defect: source inspection shows raw or absent `params.requesterOrigin` can differ from `targetRequesterOrigin`, `directOrigin`, or `completionDirectOrigin`, while delivery copies the caller-supplied `sourceChannel` into `inputProvenance`. I did not run a live channel reproduction.

## Solution Assessment

No. The patch points at the right provenance field, but the maintainable fix should use the already-normalized and possibly re-resolved origin chain rather than raw `params.requesterOrigin`.
