The PR changes the subagent completion announce call to derive `sourceChannel` from `params.requesterOrigin?.channel` instead of always using the internal webchat channel.

## Best Possible Solution

Derive announce provenance from the normalized resolved origin chain, fall back to the internal channel last, and add focused regression coverage plus redacted real-channel proof.

Yes for source-level review: current main still passes the internal channel while delivery copies the caller-supplied value into `inputProvenance`, and PR head reads a raw origin that can differ from normalized or re-resolved origins. I did not run a live Telegram or Discord reproduction.

## Solution Assessment

No. The one-line patch points at the right field family, but the maintainable fix should use `completionDirectOrigin`, `directOrigin`, or `targetRequesterOrigin` after normalization and fallback resolution.
