## Summary

Keep open: the PR targets a plausible announce-provenance bug that is still present on current main, but the patch is not mergeable because it reads raw requester input instead of the normalized/resolved delivery origin chain and has no after-fix real behavior proof.

## What This Changes

The PR changes `src/agents/subagent-announce.ts` so completion announces use `params.requesterOrigin?.channel` as `sourceChannel` instead of always using `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Derive completion announce provenance from the normalized resolved origin chain, with the internal channel only as the final fallback, and add regression coverage plus redacted Telegram or Discord real behavior proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a completion announce where `requesterOrigin` is raw or absent but `targetRequesterOrigin`, `directOrigin`, or `completionDirectOrigin` is normalized or recovered; this PR would still emit raw, missing, or `webchat` provenance while delivery uses the resolved external origin.

## Solution Assessment

No. The PR points at the right provenance field, but the maintainable fix should use the already-normalized and possibly re-resolved origin chain instead of raw `params.requesterOrigin`.
