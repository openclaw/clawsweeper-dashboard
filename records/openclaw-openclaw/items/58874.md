Keep open. The PR targets a real agents/channel provenance surface, but the current patch reads raw `params.requesterOrigin` instead of the normalized, possibly re-resolved delivery origin chain, and it still lacks after-fix real-channel proof.

## What This Changes

The PR changes the subagent completion announce call to set `sourceChannel` from `params.requesterOrigin?.channel` instead of always using the internal webchat channel.

## Best Possible Solution

Derive announce provenance from the normalized resolved origin chain, fall back to the internal channel last, add focused regression coverage, and require redacted real-channel proof before merge.

## Reproduction Assessment

Yes for source-level review: current main still passes the internal channel while delivery copies the caller-supplied value into `inputProvenance`, and the PR head reads a raw origin that can differ from normalized or re-resolved origins. I did not run a live Telegram or Discord reproduction in this read-only review.

## Solution Assessment

No. The one-line patch points at the right field family, but the maintainable fix should use `completionDirectOrigin`, `directOrigin`, or `targetRequesterOrigin` after normalization and fallback resolution.
