## Summary

Keep open; the memory-dreaming direction is plausible for the linked bug, but the TTS timestamp portion has blocking correctness issues and the external PR has no real behavior proof.

## What This Changes

The PR changes ElevenLabs TTS to return timestamp metadata, adds `wordTimestamps` to TTS result types, and adjusts memory dreaming chunk/dedup tokenization.

## Best Possible Solution

Revise the PR so the memory bug fix has regression tests and the TTS timestamp feature uses the documented ElevenLabs endpoint, updates provider/core contracts end-to-end, forwards metadata to callers, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, the blocking PR issues are source-reproducible: the helper return type changes while an existing telephony caller still treats it as a Buffer, and speech-core never forwards the new timestamp field. The endpoint mismatch is also verifiable against the official ElevenLabs API reference.

## Solution Assessment

No. The current approach is not the best solution because it changes a shared helper shape without updating all callers, uses the wrong timestamp endpoint, and only adds public result types without end-to-end propagation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Update every ElevenLabs TTS caller for the new return shape:** `extensions/elevenlabs/tts.ts:121-128`
  - body: Changing `elevenLabsTTS` from `Promise<Buffer>` to an object requires updating all callers. The telephony path still assigns `await elevenLabsTTS(...)` directly to `audioBuffer`, so ElevenLabs telephony would return `{ audioBuffer, wordTimestamps }` where the contract expects a `Buffer`.
  - confidence: 0.93
- **[P2] Call the documented timestamp endpoint:** `extensions/elevenlabs/tts.ts:136`
  - body: ElevenLabs exposes timing via `/v1/text-to-speech/:voice_id/with-timestamps`, returning JSON with `audio_base64` and `alignment`. Adding `with_timestamps=true` to the regular audio endpoint leaves successful responses as audio, so `wordTimestamps` will usually stay undefined. ([elevenlabs.io](https://elevenlabs.io/docs/api-reference/text-to-speech/convert-with-timestamps?utm_source=openai))
  - confidence: 0.9
- **[P2] Return word timestamps from speech-core:** `extensions/speech-core/src/tts.ts:128-132`
  - body: The public synthesis type now advertises `wordTimestamps`, but `synthesizeSpeech` constructs a fresh result object and does not copy `synthesis.wordTimestamps` from the provider. Callers of the new SDK field will never receive the metadata.
  - confidence: 0.89
