## Summary

Keep open: the patch is not ready because it breaks ElevenLabs telephony type checks, uses the wrong ElevenLabs timing API shape, adds timestamp result fields without propagating them through speech-core, and lacks after-fix real behavior proof. The memory CJK half is also already being handled more narrowly in https://github.com/openclaw/openclaw/pull/80620, while the managed-block leak still needs a targeted promotion-path fix.

## What This Changes

This PR changes memory-core dreaming dedupe/leak handling, adds TTS word timestamp fields to public result types, and changes ElevenLabs non-streaming synthesis to try to return alignment metadata.

## Best Possible Solution

Split the work: handle the memory bug through a narrow memory-core PR, and redesign TTS timestamps around the documented ElevenLabs timing endpoint with typed response validation, runtime propagation, tests, docs if public, and real behavior proof.

## Reproduction Assessment

Yes for the PR defects: CI logs show a concrete type failure, and source inspection shows the timestamp data would not be propagated through `synthesizeSpeech`. The contributor has not provided a real after-fix run for the changed behavior.

## Solution Assessment

No. The maintainable path is to split the memory bug from the TTS feature and implement the TTS API contract end to end instead of adding a query flag and type fields only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Update every TTS caller for the new return shape:** `extensions/elevenlabs/tts.ts:121-128`
  - body: Changing `elevenLabsTTS` from `Promise<Buffer>` to an object return requires updating all callers. The telephony path still assigns the whole object to `audioBuffer`, so `check-prod-types` fails and telephony synthesis would no longer return a Buffer.
  - confidence: 0.98
- **[P2] Call the ElevenLabs timestamp endpoint:** `extensions/elevenlabs/tts.ts:138-139`
  - body: Adding `with_timestamps=true` to the normal text-to-speech URL does not match the documented ElevenLabs timing API, which uses the `/with-timestamps` path and returns `audio_base64` plus `alignment`. As written, the new branch is unlikely to ever receive the JSON shape it parses.
  - confidence: 0.91
- **[P2] Propagate timestamps through speech-core:** `extensions/speech-core/src/tts.ts:125-128`
  - body: The public result types now advertise `wordTimestamps`, but `synthesizeSpeech` still returns only `outputFormat`, `voiceCompatible`, `fileExtension`, and `target` from provider synthesis. Even if ElevenLabs produced timestamps, callers of the TTS runtime would never see them.
  - confidence: 0.9
- **[P2] Fix the managed-block leak where promotion rehydrates snippets:** `extensions/memory-core/src/dreaming-phases.ts:259-263`
  - body: This extra `startMarker` check only fires on marker comment lines already covered by `trimmed.startsWith("<!--")`, so it does not address the reported leak path where promotion rehydrates source ranges containing managed dreaming block text. The memory leak half still needs a targeted promotion-path sanitizer and regression.
  - confidence: 0.82
