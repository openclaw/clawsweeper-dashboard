## What This Changes

The PR updates memory-core dreaming chunking/tokenization and adds word timestamp fields through ElevenLabs, speech-core, and the plugin SDK result types.

## Best Possible Solution

Split or revise the patch so the memory bug has focused regression coverage, and route TTS timestamps through a complete provider/core/SDK contract using the documented ElevenLabs timestamp endpoint with all callers updated.

Yes for the blocking PR defects: the diff is source-reproducible and CI-confirmed because the shared ElevenLabs helper return shape changes while telephony still expects Buffer, and speech-core reads a field that provider types do not expose. ## Solution Assessment

No. The current PR is not the best path because it mixes a memory regression fix with a new public TTS timestamp feature and does not update the provider contract, all callers, or verification end-to-end.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Update telephony for the new TTS return shape:** `extensions/elevenlabs/tts.ts:121-128`
  - body: Changing elevenLabsTTS from Promise<Buffer> to an object means every caller must unwrap audioBuffer. The telephony synthesis path still assigns await elevenLabsTTS(...) directly to audioBuffer, so it returns an object where SpeechTelephonySynthesisResult expects a Buffer.
  - confidence: 0.97
- **[P2] Call the ElevenLabs timing endpoint:** `extensions/elevenlabs/tts.ts:136`
  - body: The PR adds with_timestamps=true to the regular audio endpoint and then expects JSON alignment data. The vendor contract exposes timestamp synthesis at the with-timestamps endpoint, so this path can still receive raw audio bytes and never populate wordTimestamps.
  - confidence: 0.9
- **[P1] Add timestamps to the provider contract first:** `extensions/speech-core/src/tts.ts:1317`
  - body: speech-core reads synthesis.wordTimestamps, but the provider-facing SpeechSynthesisResult type in src/tts/provider-types.ts does not define that field. The SDK/runtime result type change alone does not make provider implementations expose the value safely.
  - confidence: 0.96
- **[P3] Replace the no-op proof test with a real regression:** `extensions/speech-core/src/tts-proof.test.ts:31`
  - body: The added test never registers the mock provider or calls synthesizeSpeech; it only asserts true, so timestamp propagation can remain broken while the test passes. Use the existing speech-core harness to assert the returned wordTimestamps value.
  - confidence: 0.9
