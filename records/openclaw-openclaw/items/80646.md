## Summary

Keep open: the PR is not mergeable because the TTS changes introduce blocking source-level regressions and the external contributor still has not supplied real after-fix behavior proof. The memory portion overlaps a narrower open fix path, while the TTS timestamp work needs a coherent provider/core/SDK contract and product review.

## What This Changes

The PR changes memory-core dreaming ingestion/deduplication and expands ElevenLabs, speech-core, and plugin SDK TTS result types to carry word timestamp metadata.

## Best Possible Solution

Split or rework the branch: handle the memory bug through the focused memory-core fix path, and implement TTS timestamps through the documented ElevenLabs timing endpoint with provider, SDK, core, tests, and UI/API expectations aligned before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head reproduces the blockers: malformed `textToSpeech`, invalid ElevenLabs timestamp parsing, telephony returning the new object shape as a Buffer, and `synthesizeSpeech` reading a field missing from the provider contract.

## Solution Assessment

No. The current patch mixes a focused memory bug fix with a broader TTS feature and changes shared return shapes without updating all callers/contracts; the safer path is a split, end-to-end contract change plus focused memory regression coverage.

  - body: The added early return exits before `textToSpeech` writes the `audioPath`, runs channel pre-transcode, or computes `audioAsVoice`, and it leaves the original block outside the function referencing `synthesis` out of scope.   - confidence: 0.96

  - confidence: 0.95

  - body: `synthesizeSpeech` reads `synthesis.wordTimestamps`, but `SpeechProviderPlugin.synthesize` still returns `SpeechSynthesisResult` from `src/tts/provider-types.ts`, which has no timestamp field. Add the provider-side contract and direct tests before reading the field in speech-core.
  - confidence: 0.93
