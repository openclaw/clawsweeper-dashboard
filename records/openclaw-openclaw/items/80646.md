## Summary

Keep open: the PR is a plausible attempt at the linked memory bug and a new TTS timestamp feature, but the current patch still breaks TTS contracts/callers, uses the wrong ElevenLabs timestamp API path, and lacks after-fix real behavior proof.

## What This Changes

This PR changes memory-core dreaming ingestion/deduplication and expands ElevenLabs/speech-core TTS result types to carry word timestamp metadata.

## Best Possible Solution

Revise the PR so the memory bug has focused regression coverage and the TTS timestamp feature uses the documented ElevenLabs timing endpoint with provider/core/SDK contracts updated end-to-end.

## Reproduction Assessment

Yes. The blocking PR issues are source-reproducible from the diff and current main: telephony still expects a Buffer, speech-core reads a field missing from the provider contract, and the ElevenLabs API path does not match the documented timing endpoint.

## Solution Assessment

No. The current approach changes a shared helper shape without updating all callers, exposes a public result field before the provider contract is aligned, and uses a query parameter where the vendor documents a separate timing endpoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Update telephony for the new TTS return shape:** `extensions/elevenlabs/tts.ts:121-128`
  - body: Changing elevenLabsTTS from Promise<Buffer> to an object requires every caller to unwrap audioBuffer. The telephony path still assigns await elevenLabsTTS(...) directly to audioBuffer, so it would return the object where SpeechTelephonySynthesisResult expects a Buffer.
  - confidence: 0.94
- **[P2] Call the ElevenLabs timing endpoint:** `extensions/elevenlabs/tts.ts:136`
  - body: ElevenLabs documents timing at /v1/text-to-speech/:voice_id/with-timestamps and returns JSON with audio_base64 plus alignment. Adding with_timestamps=true to the regular audio endpoint will normally keep returning audio bytes, so wordTimestamps stays absent. ([elevenlabs.io](https://elevenlabs.io/docs/api-reference/text-to-speech/convert-with-timestamps))
  - confidence: 0.91
- **[P1] Add timestamps to the provider contract first:** `extensions/speech-core/src/tts.ts:1317`
  - body: This reads synthesis.wordTimestamps, but SpeechProviderPlugin.synthesize still returns SpeechSynthesisResult from src/tts/provider-types.ts, which has no timestamp field. The SDK result type change alone leaves the provider-to-core field unsupported and should fail type checking.
  - confidence: 0.9
- **[P3] Replace the no-op proof test with a real regression:** `extensions/speech-core/src/tts-proof.test.ts:31`
  - body: The added test never installs the mock provider or calls synthesizeSpeech; it only asserts true, so timestamp propagation can stay broken while the test passes. Use the existing speech-core test harness to assert the returned wordTimestamps value.
  - confidence: 0.88
