## Summary

Current main still has only the generic Cron surface: it can filter by schedule kind and status, but it has no Reminders route, reminder-purpose metadata, safe reminder display fields, or reminder-specific filtering. The broader cron tags request remains open and overlaps the classification decision without replacing this issue’s dedicated reminder UX, so this issue is still necessary and needs maintainer product direction.

## What This Changes

Add a dedicated Mission Control Reminders tab that separates reminder jobs from ordinary cron jobs and presents safe human-readable reminder details and deletion controls.

## Best Possible Solution

Define a single additive, producer-owned cron purpose and user-facing display metadata contract, use typed service filters to separate reminders from other one-shot jobs, and build the dedicated Control UI view on that contract rather than inferring semantics from names or payloads.

## Reproduction Assessment

Yes, at source level. Current main directly shows no Reminders route and no reminder-purpose metadata or list filter, so the requested feature gap is reproducible without relying on the reporter’s old installation.

## Solution Assessment

No, a UI-only tab or payload-string heuristic would not be the best solution. The narrow maintainable path is a canonical typed metadata contract shared by reminder producers, storage, protocol, service filtering, and UI.
