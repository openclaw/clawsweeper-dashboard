## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though runtime config, gateway auth enforcement, interactive onboarding, and docs all treat it as a supported core gateway auth mode; draft PR #49107 is the active same-author implementation candidate.

## What This Changes

Add non-interactive onboarding support for configuring `gateway.auth.mode: "trusted-proxy"` without manual config edits.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding, with focused tests, docs, changelog, and explicit security review.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; the safe path is a typed CLI/config/docs/test change, not plugin-side setup or manual config editing guidance.
