## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while the runtime, interactive onboarding flow, and public docs treat trusted-proxy as a supported core gateway auth mode; draft PR #49107 is the active implementation candidate rather than a merged fix.

## What This Changes

Add non-interactive `openclaw onboard` support for selecting and configuring `gateway.auth.mode: "trusted-proxy"` for scripted reverse-proxy deployments.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding and updates focused tests, docs, and changelog as needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits in the token/password-only validation before any trusted-proxy config can be written.

## Solution Assessment

Yes for the requested direction. Core non-interactive onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; plugin-side setup or manual config editing would not address the CLI automation gap.
