## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime config, gateway auth implementation, interactive onboarding, and trusted-proxy docs all treat it as a core gateway auth mode; draft PR #49107 is the active same-author implementation candidate.

## What This Changes

Add non-interactive `openclaw onboard` support for selecting and configuring `gateway.auth.mode: "trusted-proxy"`.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding, preserves fail-closed runtime validation, and updates focused tests, docs, and changelog if user-facing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; the safe fix is a typed CLI/docs/test change with maintainer security review, not plugin-side setup or manual config editing guidance.
