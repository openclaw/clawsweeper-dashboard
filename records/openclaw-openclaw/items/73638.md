## Summary

Keep open: current main and the latest release still reject `trusted-proxy` in non-interactive onboarding while runtime config and interactive gateway configuration support it. The remaining work is real, but it adds a security-sensitive public CLI/config surface and needs maintainer product/security direction before implementation.

## What This Changes

The issue asks for non-interactive `openclaw onboard` support to select and configure `gateway.auth.mode: trusted-proxy` for scripted reverse-proxy deployments.

## Best Possible Solution

Add an additive, maintainer-approved non-interactive trusted-proxy onboarding path in core CLI setup, with explicit security flags, docs, focused regression tests, and fresh-install plus upgrade proof.

## Reproduction Assessment

Yes, from source. Current main and `v2026.6.10` reject non-interactive `--gateway-auth trusted-proxy`, while the runtime schema and interactive gateway setup support trusted-proxy configuration.

## Solution Assessment

Partly. Core onboarding is the right owner because trusted-proxy is core Gateway auth, but the exact CLI/config shape needs maintainer product and security approval before implementation.
