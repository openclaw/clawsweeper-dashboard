## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime config, gateway auth implementation, interactive gateway setup, and trusted-proxy docs all support it; draft PR #49107 is the active paired implementation candidate and needs maintainer/security review.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring `gateway.auth.mode: "trusted-proxy"` and its required proxy identity settings.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding, adds focused tests and docs, and preserves the existing gateway auth guardrails.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive setup mode; a plugin or manual config-edit workaround would not close the automation gap safely.
