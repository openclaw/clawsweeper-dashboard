## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime contract, gateway auth implementation, interactive onboarding, and gateway docs all support it; draft PR #49107 remains the active implementation candidate.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring `gateway.auth.mode: "trusted-proxy"` behind reverse proxies.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding and updates focused tests, docs, and changelog if user-facing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; plugin-side setup or manual config editing would bypass the existing config contract.
