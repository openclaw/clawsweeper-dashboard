## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though core runtime config, interactive onboarding, gateway validation, and public docs all support trusted-proxy auth; draft PR #49107 is the active implementation candidate for this same surface.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring `trusted-proxy` gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding and updates focused tests, docs, and changelog as needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before any trusted-proxy config can be written.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right place because trusted-proxy is already a core gateway runtime and interactive onboarding mode; the maintainable fix is a typed CLI/config/docs/test change, not plugin-side setup or manual config editing guidance.
