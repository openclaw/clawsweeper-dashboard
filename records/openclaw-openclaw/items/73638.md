## Summary

Keep open: current main and v2026.6.8 still only accept token/password for non-interactive onboarding, while trusted-proxy is a real runtime/schema/docs mode and interactive setup already supports it. The gap is valid, but the fix changes a security-sensitive public setup surface and needs maintainer product/security review rather than cleanup closure.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring trusted-proxy gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Add an additive, maintainer-approved non-interactive trusted-proxy onboarding path in core CLI setup with explicit security flags, focused docs, and regression tests that preserve token/password behavior.

## Reproduction Assessment

Yes. Source inspection shows current main accepts only token/password in non-interactive `--gateway-auth`, while runtime config, docs, and interactive setup support trusted-proxy.

## Solution Assessment

Partly. Core onboarding is the right owner because trusted-proxy is core Gateway auth, but the exact non-interactive CLI/config shape needs maintainer product and security approval before implementation.
