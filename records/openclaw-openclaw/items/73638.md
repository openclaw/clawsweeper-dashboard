## Summary

Keep open: current main and the latest release still reject `trusted-proxy` in non-interactive onboarding, while runtime config, docs, security audit, and interactive setup already support that auth mode. A previous contributor PR attempted this but closed unmerged with proof and correctness blockers, so the remaining work still needs maintainer approval for the public CLI shape and guardrails.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for configuring trusted-proxy gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Add maintainer-approved, additive non-interactive trusted-proxy onboarding flags that reuse the existing trusted-proxy config shape, drop mutually exclusive shared secrets, keep loopback trust explicit, and document the required security prerequisites.

## Reproduction Assessment

Yes. From source inspection, `--gateway-auth trusted-proxy` reaches `applyNonInteractiveGatewayConfig`, which rejects every value except `token` or `password`; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes on direction, but not as an autonomous fix yet. Trusted-proxy auth is already core runtime and interactive setup behavior, but the public non-interactive flags and security validation need maintainer approval before implementation.
