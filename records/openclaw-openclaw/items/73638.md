## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while runtime config, interactive onboarding, gateway validation, and docs all support trusted-proxy as a core gateway auth mode; draft PR #49107 is the active implementation candidate for the same surface.

## What This Changes

Add non-interactive onboarding support for selecting and configuring trusted-proxy gateway auth for scripted reverse-proxy deployments.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding, preserves fail-closed runtime validation, and updates focused tests, docs, and changelog if the CLI surface changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes for the requested direction, pending maintainer security review. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; the safe fix is a typed CLI/docs/test change, not plugin-side setup or manual config editing guidance.
