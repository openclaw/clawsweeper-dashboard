## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime config, gateway auth implementation, interactive setup, and docs already recognize trusted-proxy as a core gateway auth mode; draft PR #49107 is the active implementation candidate and should be reviewed rather than closing this issue in cleanup.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring trusted-proxy gateway auth behind reverse proxies.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that adds typed trusted-proxy non-interactive flags, fail-closed validation matching the existing gateway auth contract, focused unit coverage, user-facing docs, and a changelog entry if the change is user-facing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core gateway auth mode and interactive setup supports it; the maintainable fix is a narrow CLI/docs/test change with security review, not plugin-side setup or manual config editing guidance.
