## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though runtime config, gateway auth, docs, and interactive gateway setup already support that mode; the same author also has open draft PR #49107 targeting the exact surface, and related issue #73639 preserves duplicate/context evidence.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring `gateway.auth.mode="trusted-proxy"` and the required trusted proxy settings from CLI flags.

## Best Possible Solution

Keep this issue open until PR #49107 is reviewed, landed, or replaced with a narrow core onboarding change that adds typed trusted-proxy non-interactive flags, docs, and focused tests while preserving the existing fail-closed gateway auth defaults. After that merges, close the duplicate issue cluster against the landed implementation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `opts.gatewayAuth = "trusted-proxy"` through `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation, matching the non-interactive CLI path advertised by `register.onboard.ts`.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core gateway auth mode and interactive setup supports it; the safer path is a narrow, security-reviewed CLI/docs/test change rather than a plugin or manual config editing guidance.
