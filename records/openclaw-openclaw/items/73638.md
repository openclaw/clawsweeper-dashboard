## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though runtime config, gateway validation, interactive onboarding, and docs support trusted-proxy as a core gateway auth mode.

## What This Changes

Add non-interactive onboarding support for configuring `trusted-proxy` gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Expose a typed, documented non-interactive trusted-proxy onboarding path that writes the same config shape as interactive onboarding and preserves runtime fail-closed guardrails.

## Reproduction Assessment

Yes. A source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password validation before writing gateway config.

## Solution Assessment

Yes for the core CLI direction, pending maintainer security review. Trusted-proxy is already a core runtime and interactive onboarding mode, so the maintainable fix belongs in onboarding types, flags, config writing, tests, and docs rather than in a plugin or manual config guidance only.
