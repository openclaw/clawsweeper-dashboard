## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding, while runtime config, gateway validation, interactive onboarding, and trusted-proxy docs already treat it as a core gateway auth mode; the same-author draft PR at https://github.com/openclaw/openclaw/pull/49107 and duplicate tracker at https://github.com/openclaw/openclaw/issues/73639 remain open.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring trusted-proxy gateway auth in scripted deployments.

## Best Possible Solution

Review, refresh, and land or replace https://github.com/openclaw/openclaw/pull/49107 with a narrow core onboarding change that preserves trusted-proxy guardrails and updates focused tests and public docs.

## Reproduction Assessment

Yes. A high-confidence source-level path is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`, which reaches `applyNonInteractiveGatewayConfig` and exits at the token/password-only validation.

## Solution Assessment

Yes on the requested direction, but not by closing now. Core non-interactive onboarding is the right surface because trusted-proxy already exists at runtime and in interactive onboarding; the open implementation still needs security-conscious review and proof.
