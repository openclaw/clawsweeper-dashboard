## Summary

Keep open. Current main still limits non-interactive onboarding gateway auth to token/password even though runtime config, interactive onboarding, validation, and docs already support trusted-proxy; draft PR #49107 remains the active implementation candidate for this surface.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring `gateway.auth.mode: "trusted-proxy"` and its trusted-proxy fields.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes for the requested direction, pending maintainer security review. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; the narrow solution is typed CLI flags plus docs/tests, not plugin-side setup or manual config editing guidance.
