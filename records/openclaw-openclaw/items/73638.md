## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding, while the runtime config, gateway auth implementation, interactive setup, and docs already support that auth mode. The same author also has draft PR #49107 targeting this exact CLI/docs/test surface, so this should stay open for review rather than cleanup closure.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring `gateway.auth.mode="trusted-proxy"` for scripted reverse-proxy deployments.

## Best Possible Solution

Keep this issue open until PR #49107 is reviewed, landed, or replaced by a narrow core onboarding change that adds typed trusted-proxy non-interactive flags, focused tests, docs updates, and fail-closed validation matching the existing gateway auth contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core gateway auth mode and interactive setup supports it; the maintainable fix is a narrow CLI/docs/test change with security review, not plugin-side setup or manual config editing guidance.
