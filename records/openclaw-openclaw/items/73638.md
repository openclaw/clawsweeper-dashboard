## Summary

Keep open. Current main still blocks `trusted-proxy` in non-interactive onboarding even though runtime config, interactive onboarding, and docs support it as a core gateway auth mode; draft PR #49107 is already the active implementation candidate.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring `gateway.auth.mode: "trusted-proxy"` and its trusted proxy settings.

## Best Possible Solution

Review and land PR #49107, or an equivalent narrow core onboarding change, that writes the same trusted-proxy config shape as interactive onboarding and updates focused tests, docs, and changelog as needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

Yes. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; the safe solution is a typed CLI/docs/test change with explicit auth guardrails, not a plugin or manual config-edit workaround.
