## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though runtime config, interactive setup, and docs support trusted-proxy gateway auth; the same author also has open draft PR #49107 targeting this exact onboarding gap, so cleanup should not close the issue ahead of maintainer/security review.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring `gateway.auth.mode: "trusted-proxy"` and its required reverse-proxy fields.

## Best Possible Solution

Keep the issue open until #49107 is reviewed, landed, or replaced with a narrow core onboarding change that adds typed trusted-proxy non-interactive flags, focused tests, docs, and a user-facing changelog entry while preserving the existing fail-closed gateway auth defaults.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `applyNonInteractiveGatewayConfig` with `opts.gatewayAuth = "trusted-proxy"`; current main exits at the token/password-only validation used by the non-interactive CLI path.

## Solution Assessment

Yes for the requested direction. Core onboarding is the maintainable surface because trusted-proxy is already a core gateway auth mode and interactive setup supports it; the safer path is a narrow, security-reviewed CLI/docs/test change rather than plugin-side setup or manual config editing guidance.
