## Summary

Keep open: current main and the latest release still only allow token/password in non-interactive onboarding, while trusted-proxy is a real runtime and interactive setup mode. The request belongs in core onboarding, but the exact CLI/config contract is security-sensitive and needs maintainer approval before implementation.

## What This Changes

Add non-interactive `openclaw onboard` support for configuring `gateway.auth.mode="trusted-proxy"` in scripted trusted reverse-proxy deployments.

## Best Possible Solution

Add an additive, maintainer-approved non-interactive trusted-proxy onboarding path in core CLI setup with explicit security flags, focused docs, and tests that preserve token/password behavior.

Yes, source-level: current main types and runtime validation allow only token/password for non-interactive `--gateway-auth`, while interactive setup and runtime docs support trusted-proxy. I did not run the CLI because this review is read-only and the source path is direct.
