## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while runtime config, gateway auth, docs, and interactive setup already support that auth mode; the same author also has open draft PR #49107 targeting this exact surface, so this should remain with maintainer/security review rather than cleanup closure.

## What This Changes

Add non-interactive `openclaw onboard` flags to configure trusted-proxy gateway authentication and its required proxy identity settings.

## Best Possible Solution

Keep this issue open until #49107 is reviewed, landed, or replaced with a narrow core onboarding change that adds typed trusted-proxy non-interactive flags, updates docs and focused tests, and preserves fail-closed gateway auth defaults.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `opts.gatewayAuth = "trusted-proxy"` through `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation, matching the non-interactive CLI path.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core gateway auth mode and interactive setup supports it; the exact flag contract still needs security review.
