## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though trusted-proxy is supported by runtime config, runtime validation, docs, and interactive onboarding; draft PR #49107 is the active same-surface implementation candidate and this auth surface needs maintainer security review.

## What This Changes

Add non-interactive `onboard` CLI support for configuring `trusted-proxy` gateway auth in scripted deployments.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that writes the same trusted-proxy config shape as interactive onboarding, with focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig` and current main exits at the token/password-only validation before writing config.

## Solution Assessment

Yes for the requested direction, pending maintainer security review. Core onboarding is the right surface because trusted-proxy is already a core runtime and interactive onboarding mode; plugin-side setup or manual config editing would be a weaker fit.
