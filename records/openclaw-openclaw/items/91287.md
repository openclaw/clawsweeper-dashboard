## Summary

Keep this PR open: current main still only excludes exec-completion events from the generic `System:` render, while cron-tagged events are also consumed by the heartbeat cron prompt. The patch is narrow, matches the existing heartbeat-owned event pattern, and I did not find a blocking correctness or security issue.

## What This Changes

The PR changes generic system-event draining to leave `cron:`-tagged events queued for heartbeat-owned cron prompts and adds a regression test for tagged cron reminders.

## Best Possible Solution

Land the narrow event-owner fix after normal checks, keeping cron-tagged prompt ownership in heartbeat and generic rendering for non-cron or direct-delivery mirror events.

## Reproduction Assessment

Yes, source-level: current main enqueues main-session cron events with `contextKey: cron:<jobId>` and heartbeat also wraps `cron:` events, while the generic drain only excludes exec completions. I did not run a live gateway schedule in this read-only review.

## Solution Assessment

Yes; excluding heartbeat-owned `cron:` context-keyed entries from the generic render is the narrowest owner-boundary fix and mirrors the existing exec-completion pattern. The direct-delivery `cron-direct-delivery:` mirror remains outside this prefix and should still render raw.
