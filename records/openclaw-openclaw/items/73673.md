## Summary

Keep open. Current main still generates and audits the macOS LaunchAgent with unconditional KeepAlive and a one-second throttle, so the reported startup-failure restart loop remains plausible. The issue is also already paired with open PR #73849 using closing syntax, so cleanup should wait for that PR to be reviewed, replaced, or closed.

## What This Changes

The issue requests a macOS launchd Gateway policy that prevents rapid restart loops after startup failures so users can inspect and debug the failure.

## Best Possible Solution

The end state should distinguish intentional managed restarts from deterministic startup failures on macOS launchd, stop or slow the relaunch loop for debuggable configuration errors, and update plist generation, audit checks, tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: generate or install the macOS LaunchAgent, configure an invalid startup value such as an invalid port, and launchd supervises a process that exits with status 1 while the current plist has unconditional `KeepAlive=true` and `ThrottleInterval=1`.

## Solution Assessment

No. Current main has not solved the issue, and the linked PR direction needs careful review because restarting only on unsuccessful exits does not by itself stop loops for nonzero startup failures.
