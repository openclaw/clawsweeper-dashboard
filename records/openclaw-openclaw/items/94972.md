## Summary

Keep open: the PR fixes a real remote-skills area, but the patch makes `system.which`-only macOS nodes eligible even though the documented execution path still requires `system.run`, so it is not merge-ready.

## What This Changes

The PR widens remote macOS skill eligibility from `system.run`-only to `system.run || system.which` in three runtime gates and updates one remote runtime test.

## Best Possible Solution

Keep remote skill eligibility tied to executable macOS nodes with `system.run`, leave `system.which` as a bin-probe transport, and solve the linked user report at the CLI/Gateway status boundary if that remains active.

## Reproduction Assessment

Yes. From source inspection, a connected macOS node with only `system.which` and cached bins would become eligible under the PR, while the execution path still rejects nodes without `system.run`.

## Solution Assessment

No. The maintainable fix is to keep eligibility aligned with the execution path; `system.which` should prove bin presence, not replace the `system.run` capability required to use the skill.
