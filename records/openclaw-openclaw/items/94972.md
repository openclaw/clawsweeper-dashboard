- `merge-risk: 🚨 compatibility`: Existing `system.which`-only node setups would start advertising macOS-only skills even though the documented execution path still requires `system.run`.

## Summary

Keep open: the branch changes the remote eligibility gate in a way that conflicts with the existing execution contract, because a `system.which`-only node can locate binaries but cannot run the macOS-only skills that the eligibility result exposes.

## What This Changes

The PR broadens remote macOS skill eligibility, cache priming, and disconnect refresh gates to accept nodes with `system.which` even when they lack `system.run`, and updates a runtime test for that case.

## Best Possible Solution

Keep `system.run` as the remote skill eligibility requirement, use `system.which` only as the safer bin-probe method for executable nodes, and fix the linked CLI status mismatch through https://github.com/openclaw/openclaw/pull/94973 or an equivalent combined branch.

## Reproduction Assessment

Yes. Source inspection shows the PR would make `system.which`-only nodes eligible, while current `exec host=node` code rejects nodes without `system.run`; I did not run a live paired-node setup in this read-only review.

## Solution Assessment

No. The narrow maintainable fix is to preserve `system.run` as the eligibility contract and address the linked CLI status mismatch separately, rather than treating a bin-probe command as proof that a skill can execute remotely.
