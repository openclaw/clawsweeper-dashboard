## Summary

Keep open: this PR is not merge-ready because it broadens remote skill eligibility from executable macOS nodes to `system.which`-only probe nodes, while the documented and implemented run path still requires `system.run`. The linked user-facing regression remains better owned by the CLI status-path fix candidate rather than by changing the runtime execution-capability gate.

## What This Changes

The PR changes remote skill eligibility gates so macOS nodes with `system.which` but without `system.run` are treated as eligible, and updates a runtime test for that case.

## Best Possible Solution

Keep `system.run` as the remote skill eligibility requirement, use `system.which` only as the safer bin-probe method for executable nodes, and fix the linked CLI status mismatch through https://github.com/openclaw/openclaw/pull/94973 or an equivalent combined branch.

## Reproduction Assessment

Yes for the PR-introduced behavior: source inspection shows it would make `system.which`-only nodes eligible while the node execution path still requires `system.run`. I did not run a live paired macOS node setup in this read-only review.

## Solution Assessment

No. Treating `system.which` as sufficient eligibility is broader than the current execution contract; the narrower maintainable fix is to preserve `system.run` eligibility and address the linked CLI status mismatch separately.
