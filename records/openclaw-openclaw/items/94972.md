## Summary

Keep open: the patch changes remote skill eligibility in the wrong direction because `system.which` can prove a binary exists, but current OpenClaw still requires `system.run` before a remote macOS skill can actually run via `exec host=node`.

## What This Changes

The PR changes three remote skill runtime gates and one test so macOS nodes with `system.which` but no `system.run` count as remotely eligible.

## Best Possible Solution

Keep remote skill eligibility tied to executable macOS nodes with `system.run`, leave `system.which` as a bin-probe transport, and solve the linked issue at the CLI/Gateway status boundary if that remains the active bug.

## Reproduction Assessment

Yes. From source inspection, a connected macOS node with only `system.which` and a cached `memo` bin would become eligible under the PR, while the actual `exec host=node` runtime rejects nodes that lack `system.run`.

## Solution Assessment

No. The maintainable fix is to keep eligibility aligned with the execution path; `system.which` should prove bin presence, not replace the `system.run` capability required to use the skill.
