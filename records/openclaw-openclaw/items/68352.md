## Summary

Keep open: current main still returns before checking populated `plugins.allow` entries against discovered plugin IDs, and the focused fix is already tracked by the open linked PR with a closing reference.

## What This Changes

The request is to make plugin allowlist startup diagnostics accurate when users configure channel-style IDs instead of discovered plugin IDs.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/68389 or an equivalent focused diagnostic with regression coverage lands; treat channel aliasing and extension auto-load policy as separate follow-ups.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load a workspace/global plugin while `plugins.allow` contains only a non-plugin ID, and `warnWhenAllowlistIsOpen` returns before any mismatch comparison.

## Solution Assessment

Yes for the narrow fix. A warning-only diagnostic in the plugin-loader provenance helper is the maintainable solution; automatic channel aliases or extension auto-load policy changes should remain separate product work.
