## Summary

Keep open. Current main still lacks the requested channel-id-to-plugin-id allowlist diagnostic, and the linked PR that explicitly closes this issue is still open rather than merged.

## What This Changes

The issue requests clearer plugin allowlist warnings and discovery guidance when users configure channel-style IDs, such as `feishu`, instead of discovered plugin IDs.

## Best Possible Solution

Keep this issue open until PR #68389 or a maintainer-approved replacement lands a focused diagnostic that uses discovered manifest/channel ownership to tell users when an allowlist entry looks like a channel ID and which plugin ID should be allowed.

## Reproduction Assessment

Yes. The report has concrete config steps, and current source inspection confirms populated-but-mismatched `plugins.allow` entries still do not get a specific channel-ID mismatch diagnostic; I did not run the gateway because this review is read-only.

## Solution Assessment

Yes for the narrow path. A diagnostic-only fix is the most maintainable first step because automatic aliasing or extension auto-load policy changes would affect plugin trust semantics and need separate product/security review.
