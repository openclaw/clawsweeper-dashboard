## Summary

Keep open: current main and the latest release still skip populated `plugins.allow` lists before checking whether entries match discovered plugin IDs, and the linked fix PR is still open with an explicit closing reference.

## What This Changes

The issue asks for clearer startup diagnostics when `plugins.allow` contains channel-style or otherwise unmatched entries instead of discovered plugin IDs.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/68389 or an equivalent focused diagnostic lands with regression coverage for unmatched `plugins.allow` entries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a populated `plugins.allow` with only non-plugin IDs while a non-bundled plugin is discoverable, and the helper returns before comparing the allowlist to discovered plugin IDs.

## Solution Assessment

Yes for the narrow fix. A warning-only diagnostic is the maintainable first step; automatic channel aliases or extension auto-load policy changes should remain separate product work.
