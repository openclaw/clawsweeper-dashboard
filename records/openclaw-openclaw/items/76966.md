## Summary

Keep open. Current main still has a source-level path where all chat scroll events update `chatUserNearBottom`, and #76991 is an open closing PR for the same programmatic-scroll root cause, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Control UI/webchat should stay pinned to the newest streamed assistant output while the user is already near the bottom, without overriding intentional scroll-away.

## Best Possible Solution

Land or replace #76991 with a narrow Control UI scroll-state fix plus regression coverage, then close this issue through the merged PR.

## Reproduction Assessment

Yes at source level, but not via a live browser run in this read-only review. The current handler can be driven by a scroll event after programmatic `scrollTo` and content growth, making `distanceFromBottom >= 450` and flipping `chatUserNearBottom` false.

## Solution Assessment

Yes. Distinguishing programmatic scroll events from user scroll events in the Control UI scroll state, with focused regression tests, is the narrow maintainable fix and matches the open linked PR’s boundary.
