## Summary

Keep open. Current main still has the Control UI scroll state path that can lose bottom stickiness during streaming, and an open linked PR is already the right implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Control UI webchat should stay pinned to the newest streaming assistant output while the user is at or near the bottom, without overriding intentional manual scroll-away.

## Best Possible Solution

Land a focused UI regression fix, likely PR #76991 or equivalent, that distinguishes programmatic chat scroll events from user scrolls while preserving manual scroll-away behavior.

## Reproduction Assessment

Yes at source level: `scheduleChatScroll` writes the chat thread to bottom, the view routes scroll events into `handleChatScroll`, and that handler can flip `chatUserNearBottom` to false when content growth leaves `distanceFromBottom >= 450`. I did not run a live browser reproduction in this read-only pass.

## Solution Assessment

Likely yes, based on the linked PR context: a narrow programmatic-scroll guard targets the state corruption without changing thresholds, force semantics, or user scroll-away behavior. Final approval should happen on PR #76991 with its tests.
