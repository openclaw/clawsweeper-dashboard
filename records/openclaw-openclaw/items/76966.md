## Summary

Keep open. Current main still has the source-level Control UI scroll-state gap, and open PR #76991 is the appropriate closing implementation but remains unmerged with a review blocker, so the issue should track that PR until a corrected fix lands.

## What This Changes

The issue reports that Control UI webchat can lose bottom stickiness while assistant output streams, leaving the newest streamed content out of view.

## Best Possible Solution

Land a corrected version of #76991 or an equivalent focused fix that suppresses OpenClaw’s own auto-scroll events without masking genuine user scroll-away behavior.

## Reproduction Assessment

Yes, at source level: current main has no programmatic-scroll distinction, so `handleChatScroll` with `scrollHeight` 3000, `scrollTop` 500, and `clientHeight` 400 deterministically flips `chatUserNearBottom` false. I did not run a live browser streaming session in this read-only review.

## Solution Assessment

No for closing now. The linked PR is the right implementation path, but it needs the reported guard-window user-scroll issue fixed and merged before this issue is resolved.
