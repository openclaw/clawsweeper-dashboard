## Summary

Keep open. Current main still treats every chat scroll event as scroll-position state and has no programmatic-scroll guard, while open PR #76991 explicitly targets this bug with closing syntax; this should be resolved by reviewing, updating, landing, or replacing that PR rather than closing the issue now.

## What This Changes

The issue asks for Control UI/webchat to stay pinned to the newest streamed assistant output while the user is already near the bottom.

## Best Possible Solution

Review and land #76991 or an equivalent narrow replacement that guards app-initiated scroll events while preserving intentional user scroll-away behavior and adding regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was established in this sweep. The source-level path is clear: current main schedules scroll on `chatStream`, writes `scrollTo`/`scrollTop`, and then lets every scroll event recompute `chatUserNearBottom` without a programmatic-scroll guard.

## Solution Assessment

Yes for the likely direction: distinguishing programmatic scroll events from user scrolls is a narrow fix that preserves the existing near-bottom contract. The exact implementation should still be reviewed through #76991 or a replacement PR with regression tests.
