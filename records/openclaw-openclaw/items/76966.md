## Summary

Keep open. Current main still has the vulnerable Control UI scroll state path, and an open linked PR explicitly targets this issue, so the issue should wait for that PR to merge or be closed.

## What This Changes

The issue reports that Control UI webchat sometimes stops staying pinned to the newest streaming assistant output when the user is already near the bottom.

## Best Possible Solution

Review and land #76991, or an equivalent narrow guard that ignores programmatic scroll events while preserving intentional user scroll-away, then let the merge close this issue.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only pass. The source path is clear: streaming updates schedule scrolls, and current `handleChatScroll` can flip `chatUserNearBottom` false from any scroll event when the computed bottom distance grows past the threshold.

## Solution Assessment

Likely yes. A focused programmatic-scroll guard in the Control UI scroll scheduler/handler is the narrowest maintainable direction, and the open linked PR is already scoped to that behavior plus regression coverage.
