## Summary

Keep open. Current main still leaves the Media Understanding docs without explicit credential-source guidance, and the runtime/schema still do not support inline `apiKey` on `tools.media` model entries. The right narrow fix is already represented by open PR #75164, which uses closing syntax for this issue, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Document or otherwise clarify where Media Understanding provider credentials must be configured so `tools.media` audio/video/image setup does not fail with a missing `apiKey`.

## Best Possible Solution

Land PR #75164 or an equivalent docs-only patch that adds a Media Understanding provider-credentials section: `tools.media` model entries do not accept inline `apiKey`; credentials resolve through auth profiles, provider environment variables, or `models.providers.<providerId>.apiKey`; include a short working example and link to the public auth/config docs. Runtime/schema changes are unnecessary unless maintainers intentionally want a new credential surface.

## Reproduction Assessment

Yes. The gap is reproducible by reading current main: the docs omit the credential-source rule, while the runtime and strict config schema show inline `tools.media.*.models[].apiKey` is not part of the supported contract.

## Solution Assessment

Yes. A narrow docs clarification is the lowest-risk fix for the reported setup failure because it matches current runtime behavior; PR #75164 is the active implementation candidate and should be reviewed or landed before closing the issue.
