## Summary

Keep open: this is a useful Telegram message-preservation fix, but the current patch can force-send below-debounce drafts that Telegram never displayed, and the PR still needs acceptable real Telegram proof before merge.

## What This Changes

The PR changes Telegram draft dispatch so an in-flight streamed answer preamble rotates to a fresh message at a tool boundary, with regression tests and a new opt-in live proof harness under `proof/`.

## Best Possible Solution

Keep the dispatch-layer rotation, but gate it on an actual message id or attempted/in-flight send state, add regression coverage for the below-debounce no-send case, remove or relocate the root proof harness if maintainers do not want PR-specific proof scripts in tree, and attach redacted real Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main has finalized-lane and rendered-tool-progress rotation paths, but an unfinalized answer lane with suppressed tool progress can reach post-tool text without rotating. I did not run the live Telegram repro in this read-only review.

## Solution Assessment

No, not yet: the dispatch-layer rotation is the right owner boundary, but the current guard is too broad because `hasStreamedMessage` can mean pending unsent draft text. The safer fix is to rotate only when there is a real preview message or an attempted/in-flight send, and prove it live.
