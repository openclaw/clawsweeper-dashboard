## Summary

Keep open: the PR addresses a real Telegram message-loss path that current main does not cover, but the new rotation guard is too broad and can turn a below-debounce, never-sent draft into an extra persistent Telegram message at a tool boundary.

## What This Changes

The PR adds Telegram dispatch/test coverage and an opt-in live proof harness so an in-flight streamed answer preamble is preserved as its own message when a tool boundary occurs mid-stream.

## Best Possible Solution

Keep the dispatch-layer tool-boundary rotation, but gate it on an existing preview message id or attempted/in-flight send state and add a below-debounce regression before merge.

## Reproduction Assessment

Yes. The PR body provides live fast/delayed Bot API output, and source inspection shows current main has finalized-lane and rendered-tool-progress rotation paths but lacks this unfinalized suppressed-progress tool-boundary rotation.

## Solution Assessment

No, not yet. The dispatch layer is the right owner boundary, but the guard should distinguish actually sent or attempted Telegram previews from below-debounce pending text before it can be the best fix.
