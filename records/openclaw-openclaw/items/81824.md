## Summary

This PR should stay open because the proposed size gate only reaches picker/drop attachment flows, while current main's paste handler can still read an oversized clipboard image into the same JSON-RPC send path. The missing real behavior proof is also still a merge gate for this external PR.

## What This Changes

The PR adds a 4 MB maximum-size constant to the WebChat attachment-support helper and rejects files over that size in `isSupportedChatAttachmentFile`.

## Best Possible Solution

Apply one attachment-size policy before FileReader across picker, drop, and paste, align the limit and user feedback with the gateway attachment contract, add focused regression coverage, and provide real browser proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show the reported path: accepted files become data URLs, then base64 JSON-RPC payloads, then a full-frame `JSON.stringify` before WebSocket send.

## Solution Assessment

No. A client-side guard is a reasonable mitigation, but this patch is incomplete because pasted images bypass it and the 4 MB silent-skip policy is not reconciled with the existing 20 MB gateway attachment limit.
