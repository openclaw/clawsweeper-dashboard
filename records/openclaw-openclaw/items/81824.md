## Summary

Keep open: this is a plausible WebChat crash fix, but the patch is incomplete and still lacks inspectable after-fix real behavior proof. Pasted images bypass the new size predicate, and the hard silent 4 MB UI cap is not reconciled with the existing 20 MB gateway attachment contract.

## What This Changes

Adds a 4 MB max-size check to WebChat's shared attachment support predicate so oversized picker/drop files are rejected before FileReader processing.

## Best Possible Solution

Apply one attachment-size preflight before FileReader across picker, drop, and paste, align oversize behavior with the gateway size contract and user feedback, add focused coverage, and provide real browser proof.

## Reproduction Assessment

No, not as a live current-main reproduction in this review. Source inspection does show the path from FileReader data URLs to inline base64 `chat.send` attachments and full WebSocket frame serialization.

## Solution Assessment

No. A client-side preflight is reasonable, but this PR only covers picker/drop files and needs paste coverage plus an agreed limit/UX that fits the gateway attachment contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Route pasted files through the same predicate and cover that path.
