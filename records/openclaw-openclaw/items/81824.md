## Summary

Keep open: this is a plausible focused WebChat bug fix, but the current patch is incomplete because pasted image attachments still bypass the new size predicate, and the PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds a 4 MB WebChat attachment size limit to `isSupportedChatAttachmentFile()` so picker/drop uploads larger than that are rejected before base64 JSON-RPC serialization.

## Best Possible Solution

Apply one attachment-size policy before `FileReader` across picker, drop, and paste; align the limit and feedback with the gateway contract; add focused regression coverage; and require real browser proof before merge.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection does show the reported path: accepted files become data URLs, base64 JSON-RPC attachments, and a full `JSON.stringify` WebSocket frame.

## Solution Assessment

No. A client-side preflight is a reasonable part of the fix, but this patch misses pasted images and hard-codes a silent 4 MB UI cap without reconciling it with the gateway attachment contract.
