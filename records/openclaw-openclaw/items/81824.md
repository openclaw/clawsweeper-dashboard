## Summary

Keep this PR open as useful work for the linked WebChat attachment crash, but the current branch is not merge-ready: the 20 MB client gate no longer blocks the reported ~4 MB failure path, the branch carries an unrelated package resolution change, it is dirty against current main, and real behavior proof is still not inspectable.

## What This Changes

The PR adds a WebChat client-side attachment size predicate and oversized-file alerts for paste/select/drop flows, plus an unrelated hosted-git-info package override.

## Best Possible Solution

Refresh this as a narrow WebChat fix that blocks oversized attachments before base64 JSON-RPC serialization at a proven safe client cap, or moves uploads off the JSON frame, while covering select/drop/paste paths, removing the package change, and adding inspectable real-run proof.

## Reproduction Assessment

No live browser reproduction was run in this read-only review. Source inspection gives a clear current-main path: WebChat reads attachments as data URLs, sends base64 in chat.send, and the browser client serializes the whole JSON-RPC frame with JSON.stringify.

## Solution Assessment

No. A client-side pre-serialization guard is a plausible fix shape, but the current 20 MB value does not block the linked ~4 MB repro; the safer solution is a proven browser-safe cap or an upload path that avoids embedding large base64 in the JSON-RPC frame.
