## Summary

Keep open. Current main still has the reported Control UI mismatch: array rows are visibly one-based while config validation errors are built from raw zero-based path segments, so users can be pointed at the wrong row label.

## What This Changes

Align Control UI config validation error numbering with the row numbers shown for model/config array entries.

## Best Possible Solution

Keep canonical ConfigValidationIssue.path values raw and zero-based for API/tests, then add a Control UI-facing formatter for config save/apply errors that renders numeric array segments with the same one-based row labels users see in the form while preserving raw paths in structured details.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make an invalid entry in a Control UI config array such as models.providers.<provider>.models, then save: the row badge is one-based while the returned validation path is formatted from zero-based path segments.

## Solution Assessment

Yes. The narrowest maintainable fix is display-layer formatting for Control UI config validation errors, not changing Zod path mapping or the ConfigValidationIssue.path contract globally.
