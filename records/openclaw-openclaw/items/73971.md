## Summary

Keep open. Current main still has the reported Control UI mismatch: config array rows are labeled one-based in the form, while save/apply validation errors still expose raw zero-based config paths.

## What This Changes

The issue requests that Control UI config row labels and backend validation error numbering use consistent indexing for model/config array entries.

## Best Possible Solution

Keep canonical ConfigValidationIssue.path values zero-based for API, CLI, tests, and structured details, but add Control UI display formatting for config save/apply errors so array entries render with the same one-based labels users see in the form.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the fourth item in a Control UI config array invalid, such as a models.providers.<provider>.models entry, then save or apply: the row badge is #4 while the returned validation path uses raw index 3.

## Solution Assessment

Yes. The narrow maintainable fix is a Control UI display-layer formatter that consumes GatewayRequestError.details.issues when available, not a global change to Zod path mapping or ConfigValidationIssue.path semantics.
