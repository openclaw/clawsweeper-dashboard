## Summary

Keep open. Current main still has the reported Control UI/config validation mismatch: array rows are displayed with one-based labels, while validation summaries keep raw zero-based dotted path segments, so a visible fourth model row can still be reported as index 3.

## What This Changes

The issue requests consistent numbering between Control UI model row labels and backend validation error messages so users are not confused by one-based UI labels versus zero-based validation paths.

## Best Possible Solution

Human-facing config validation messages should render numeric array path segments with the same one-based labels shown in the Control UI form, while structured validation issue details continue to expose raw zero-based schema paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make the fourth item in a Control UI config array such as models.providers.<provider>.models invalid, then save or apply; the visible row is #4 while the validation path is built from index 3.

## Solution Assessment

Yes. The narrow maintainable fix is a display-layer formatter for human config save/apply errors, not changing Zod path mapping or the shared ConfigValidationIssue contract globally.
