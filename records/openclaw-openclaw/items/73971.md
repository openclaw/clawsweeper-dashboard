## Summary

Keep open. The current main still matches the reported Control UI/config validation mismatch: array rows are labeled one-based in the form, but the validation summary preserves raw zero-based schema path segments, so the fourth visible model row can still be reported as index 3.

## What This Changes

The issue requests aligning Control UI model-array row labels and backend validation error numbering so a row displayed as #4 is not reported as the third/zero-based item.

## Best Possible Solution

Human-facing config validation messages should display array positions consistently with Control UI row badges while structured RPC details keep zero-based schema paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make the fourth item in models.providers.<provider>.models invalid through the Control UI form and save/apply; the visible badge is #4 while the validation summary formats the raw path segment as 3.

## Solution Assessment

Yes. The narrow maintainable fix is a display-layer formatter for human-facing config save/apply errors, not changing Zod path mapping or the shared ConfigValidationIssue contract globally.
