## Summary

Keep open. Current main still preserves the reported mismatch: Control UI array rows are shown with one-based badges, while config validation errors are summarized from raw zero-based path segments, so the save/apply error can point users at the wrong visible row.

## What This Changes

The issue requests that Control UI model configuration row numbers and backend validation error numbering use the same user-visible convention.

## Best Possible Solution

Preserve canonical ConfigValidationIssue.path values as raw schema paths, then add a Control UI-facing formatter for config save/apply errors that renders numeric array segments with the same one-based row labels users see in the form while keeping raw issues available in structured details.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an invalid fourth entry in a Control UI config array such as models.providers.<provider>.models, then save or apply: the form row badge is #4 while the validation path is built from index 3.

## Solution Assessment

Yes. The narrowest maintainable solution is a display-layer formatter for Control UI config validation errors, not changing Zod path mapping or the shared ConfigValidationIssue.path contract globally.
