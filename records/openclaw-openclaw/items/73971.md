## Summary

Keep open. Current main still has the reported Control UI mismatch: array rows are displayed with one-based labels, while config validation errors are summarized from raw zero-based path segments, so the user can be pointed at the wrong visible row.

## What This Changes

The issue requests that frontend row labels and backend/config validation error indexes match for available-model configuration entries.

## Best Possible Solution

Keep canonical ConfigValidationIssue.path values raw and zero-based for API/tests, then add a Control UI-facing formatter for config save/apply errors that renders numeric array segments with the same one-based row labels users see in the form while preserving raw paths in structured details.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make an invalid fourth entry in a Control UI config array such as models.providers.<provider>.models and save/apply it: the row badge is #4 while the validation path is built from raw zero-based segments.

## Solution Assessment

Yes. The narrowest maintainable fix is display-layer formatting for Control UI config validation errors, not changing Zod path mapping or the ConfigValidationIssue.path contract globally.
