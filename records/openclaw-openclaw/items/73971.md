## Summary

Keep open. Current main still has the reported Control UI/config validation mismatch: array rows are shown as one-based badges, while config validation paths and save/apply error text keep raw zero-based schema indexes, so the fourth visible model row can still be reported as index 3.

## What This Changes

Requests that Control UI model-provider array rows and config validation/save error messages use consistent item numbering.

## Best Possible Solution

Human-facing config validation messages should render numeric array path segments with the same one-based labels shown in the Control UI form, while structured issue details continue to expose raw zero-based schema paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make the fourth item in a Control UI config array such as models.providers.<provider>.models invalid, then save/apply; the visible row is #4 while the validation path is built from index 3.

## Solution Assessment

Yes. The narrow maintainable fix is a display-layer formatter for human config save/apply errors, not changing Zod path mapping or the shared ConfigValidationIssue contract globally.
