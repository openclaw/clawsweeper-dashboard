## Summary

Keep open. Current main still appears to have the reported Control UI mismatch: array rows are displayed with one-based labels, while config validation error paths are summarized from raw zero-based numeric path segments and surfaced directly to the UI.

## What This Changes

The issue asks for Control UI row numbering and backend config validation error numbering to match so users can identify the correct model/config row.

## Best Possible Solution

Preserve canonical ConfigValidationIssue.path values as raw machine paths, and add a Control UI-facing formatter for config save/apply errors that renders numeric array segments with the same one-based row labels users see in the form while keeping raw paths available in structured details.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make an invalid entry in a Control UI config array such as models.providers.<provider>.models and save/apply it: the visible row label is one-based while the returned validation path is formatted from zero-based segments.

## Solution Assessment

Yes. The narrow maintainable solution is display-layer formatting for Control UI config validation errors, not changing Zod path mapping or the ConfigValidationIssue.path contract globally.
