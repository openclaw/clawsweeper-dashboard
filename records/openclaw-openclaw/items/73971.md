## Summary

Keep open. Current main still has the reported Control UI/config validation mismatch: array rows are labeled one-based in the form, while validation error summaries are built from raw zero-based schema paths, so a visible fourth model row can still be reported with index 3.

## What This Changes

The issue asks for model-array row numbers shown in the Control UI and the backend validation error text to use consistent numbering.

## Best Possible Solution

Add a human-facing config issue path formatter for save/apply error text that renders numeric array positions with the same one-based labels users see in the Control UI, while leaving structured validation issue paths raw for programmatic consumers.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make the fourth item in a Control UI config array such as models.providers.<provider>.models invalid, then save or apply; the visible row label is #4 while the validation path is built from zero-based index 3.

## Solution Assessment

Yes. The narrow maintainable fix is display-layer formatting for human config validation summaries, not changing Zod path mapping or the shared ConfigValidationIssue contract globally.
