## Summary

Keep open. Current main still has the reported Control UI mismatch: array rows are presented to users with one-based badges, but config save/apply validation errors are summarized from raw zero-based schema paths.

## What This Changes

The issue asks for Control UI and backend validation messages to use consistent numbering for array items so an invalid fourth model row is not reported as the third item.

## Best Possible Solution

Preserve canonical ConfigValidationIssue.path values as raw schema paths, then add a Control UI-facing formatter for config save/apply errors that renders numeric array segments with the same one-based row labels users see in the form while keeping raw issues available in structured details.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit an invalid fourth entry in a Control UI config array such as models.providers.<provider>.models: the form badge renders #4 while the validation path remains the raw index 3 and is used in the request error summary.

## Solution Assessment

Yes. The narrowest maintainable solution is a display-layer formatter for Control UI config validation errors, not changing Zod path mapping or the shared ConfigValidationIssue.path contract globally.
