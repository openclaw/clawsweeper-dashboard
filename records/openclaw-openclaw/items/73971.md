## Summary

Keep open. Current main still has the reported Control UI/config validation mismatch: array rows render with one-based labels while validation paths and save/apply error text preserve raw zero-based path segments, so the fourth visible model row can be reported as index 3.

## What This Changes

Request that Control UI and backend-facing config validation messages use matching row numbering for configurable model entries so users are not pointed at the wrong visible row.

## Best Possible Solution

Human-facing config validation messages should render numeric array path segments with the same one-based labels shown in the Control UI form, while structured issue details continue to expose raw zero-based schema paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make the fourth item in a Control UI config array such as models.providers.<provider>.models invalid, then save/apply; the visible row is #4 while the validation path is built from index 3.

## Solution Assessment

Yes. The narrow maintainable direction is a display-layer formatter for human config save/apply errors, not changing Zod path mapping or the shared ConfigValidationIssue contract globally.
