## Summary

Keep open. Current main still has the reported Control UI/config validation numbering mismatch: array rows are labeled one-based in the form, while human-facing save/apply validation summaries still format raw zero-based schema paths.

## What This Changes

The issue asks for Control UI model rows and validation errors to use consistent numbering for configured model arrays.

## Best Possible Solution

Human-facing config validation messages should render array positions consistently with Control UI labels, while structured `ConfigValidationIssue` details keep raw zero-based schema paths for machines.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make the fourth entry in `models.providers.<provider>.models` invalid through the Control UI form and save/apply; the visible row is `#4`, but the validation path remains `models.providers.<provider>.models.3...`.

## Solution Assessment

Yes. The narrow maintainable fix is a display-layer formatter for human-facing config save/apply errors, not changing Zod path mapping or the shared machine-readable issue contract.
