## Summary

Keep open. Current main still has the reported Control UI mismatch: array rows are displayed with one-based labels, while config validation errors are formatted from raw zero-based Zod path segments and surfaced directly to the UI.

## What This Changes

The issue asks for Control UI config row labels and backend validation error paths to use consistent numbering for array-backed model provider entries.

## Best Possible Solution

Keep canonical ConfigValidationIssue.path values raw and zero-based for machine/API use, then add a Control UI-facing formatter for config save/apply errors that renders numeric array segments with the same one-based row labels users see in the form while preserving raw paths in structured details.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make an invalid fourth entry under models.providers.<provider>.models in the Control UI form and save/apply it: the visible row badge is #4 while the gateway error path is formatted from raw index 3.

## Solution Assessment

Yes. The best fix is a narrow display-layer formatter for Control UI config validation errors, not changing Zod path mapping or the ConfigValidationIssue.path contract globally.
