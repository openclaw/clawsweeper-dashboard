## Summary

Keep open: this protected maintainer-labeled external PR needs maintainer handling, real behavior proof is missing, and the diff has blocking auth metadata and cleanup regressions.

## What This Changes

The PR branch makes `openclaw models auth login --provider openai` prefer ChatGPT/Codex OAuth, keeps `--method api-key` explicit, and updates auth listing, docs, tests, and changelog.

## Best Possible Solution

Revise the branch so runtime setup provider metadata, static manifest auth choices, docs/tests, and stale lockout cleanup all treat OpenAI ChatGPT login consistently, then add redacted real CLI proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows manifest auth choices are loaded from `openclaw.plugin.json` and compared to runtime wizard metadata, and the lockout cleanup runs by selected provider before the delegated Codex OAuth method returns `openai-codex` profiles. I did not run tests because this review was read-only.

## Solution Assessment

No; the requested default may be a reasonable direction, but this patch is not the best mergeable form until it keeps manifest/control-plane metadata and auth cleanup semantics aligned and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep OpenAI auth metadata in sync:** `extensions/openai/setup-api.ts:82-86`
  - body: This changes the runtime `openai:api-key` wizard group and adds `openai` OAuth/device methods, but the static manifest `providerAuthChoices` remains on the old shape. Cold onboarding/configure reads those choices from the manifest and `openclaw.plugin.test.ts` compares them with the runtime wizard metadata, so this will either fail that contract or leave the user-facing setup flow advertising the old OpenAI API-key/Codex split.
  - confidence: 0.88

  - body: The new `openai` OAuth method delegates to the `openai-codex` provider, but `modelsAuthLoginCommand` clears stale lockouts using the selected provider id before running the method. A re-login through the new documented `--provider openai` path will therefore skip existing `openai-codex:*` disabled/cooldown entries, and `upsertAuthProfile` does not reset usage stats.
  - confidence: 0.82
