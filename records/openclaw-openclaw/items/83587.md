## Summary

Keep open: the reported cron preflight auth problem is plausible and the shared resolver direction is good, but this branch is not merge-ready because it fails current type contracts and would treat synthetic local auth markers as real Authorization credentials.

## What This Changes

The PR modifies cron local provider preflight to resolve provider auth via `resolveApiKeyForProvider` and attach a Bearer Authorization header to the `/models` or `/api/tags` probe.

## Best Possible Solution

Keep the fix in cron preflight, but implement it through the existing model-auth contract, pass only supported auth context, skip non-secret markers, and cover config/env/profile plus authless local cases with focused tests.

## Reproduction Assessment

Yes at source level: current main sends the local provider probe with only `GET`, so a local OpenAI-compatible provider requiring auth will receive an unauthenticated `/models` probe. I did not run a live LiteLLM repro in this read-only review.

## Solution Assessment

No for this branch as written: using the shared resolver is the right direction, but the patch must compile and must filter non-secret synthetic markers before emitting Authorization.
