## Summary

Keep this PR open, but it is not merge-ready: it targets a real source-visible cron preflight auth gap, yet the branch still has build, credential-scope, marker-handling, and after-fix proof blockers.

## What This Changes

The PR changes cron model-provider preflight probing to resolve a provider API key and add an Authorization header to the local `/models` probe.

## Best Possible Solution

Pass the resolved cron `agentDir` into preflight, use `resolveApiKeyForProvider` in that scope, suppress non-secret marker values before adding Authorization, add focused coverage, and require redacted after-fix runtime proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Current main source shows the preflight GET is sent without headers, and the PR body includes before LiteLLM 401 logs, so the reported symptom is source-reproducible but not live-proven here.

## Solution Assessment

No. Reusing the normal resolver is the right direction, but this implementation needs the cron run's scoped `agentDir`, marker filtering, build cleanup, tests, and real after-fix proof before it is the maintainable fix.
