## Summary

Keep open. Current main has relevant mitigations for this startup-stall family, but it still does not prove this WSL2/npm-global restart report is fixed: startup still waits for sidecar bootstrap by default, channel account work can still run after the sidecar handoff, the normal startup trace still disables at `ready`, and there is no current-main packaged CoClaw/local-WebSocket proof showing the restart path stays under the handshake budget.

## What This Changes

The issue asks OpenClaw to stop blocking the Gateway event loop for about 75 seconds across `sidecars.channels` and post-ready work during restart, so local plugin WebSocket handshakes do not hit the Gateway timeout.

## Best Possible Solution

Capture a current-main packaged restart on the reported WSL2 shape with startup trace, diagnostics timeline event-loop samples, `/readyz` event-loop health, per-channel timings, and a CoClaw or equivalent local Gateway WebSocket client. If the freeze remains, move the measured synchronous work off the Gateway hot path or keep readiness/local client handshakes gated until the event loop is actually responsive; if it is fixed, close with the exact fixing commit and shipped release or main-only timestamp.

## Reproduction Assessment

Yes for the reported 2026.4.26 host: the issue includes concrete restart steps, startup timing, and repeated plugin-side lag observations. Current-main reproduction is unclear because the same WSL2/npm-global plus CoClaw plugin path has not been live-verified after the newer mitigations.

## Solution Assessment

Unclear. The current implementation has useful mitigations, especially background model prewarm, startup-pending responses, `/readyz` event-loop diagnostics, and client readiness helpers, but closing as implemented would be premature without measured current-main proof for both the `sidecars.channels` stall and the post-ready WebSocket starvation.
