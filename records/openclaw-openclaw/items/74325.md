## Summary

Keep open. Current main contains meaningful mitigations for the likely startup/runtime-deps contributors, but it does not prove this combined report is fixed: normal startup still waits for sidecars, the default-wait behavior is covered by tests, startup event-loop sampling still stops at `ready`, and the post-ready work the reporter called out remains outside the startup trace.

## What This Changes

The issue reports gateway restart event-loop starvation on v2026.4.26, with about 45s in `sidecars.channels` plus a second about 28s freeze immediately after `gateway ready`, causing local WebSocket handshakes from plugins to time out.

## Best Possible Solution

Keep this open until maintainers capture a current-main or next-release packaged restart trace with after-ready attribution on an affected host. Then either fix the measured synchronous channel/runtime-deps/post-ready blocker directly, or close with explicit proof that the current runtime-deps/startup changes restore the CoClaw local WebSocket recovery path.

## Reproduction Assessment

Yes for the reported v2026.4.26 host: the issue gives concrete restart steps, startup trace timings, and repeated event-loop-lag observations. Current-main reproduction remains unclear because this read-only review did not run the WSL2/npm-global restart scenario after later runtime-deps changes.

## Solution Assessment

Unclear. Current main has the right partial mitigations, especially detached model prewarm and runtime-deps staging, but the narrowest maintainable solution still needs measurement of the current remaining stall before choosing channel readiness changes, after-ready instrumentation, or a specific runtime-deps/plugin fix.
