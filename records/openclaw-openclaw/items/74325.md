## Summary

Keep open: the reported 2026.4.26 and 2026.4.29 packaged builds have strong two-platform evidence, and current main has relevant mitigations but no live packaged proof that the two-phase restart and post-ready event-loop starvation is gone.

## What This Changes

The report asks OpenClaw to stop gateway restarts from blocking the main thread around channel sidecar startup and immediately after readiness, which causes local WebSocket handshakes and channel/plugin recovery to time out.

## Best Possible Solution

Capture a current-main or v2026.5.2 packaged restart and first-turn trace on the affected platforms, then remove or offload the measured synchronous startup/agent-prep work while keeping Gateway readiness and client retry behavior aligned with actual event-loop responsiveness.

## Reproduction Assessment

Yes for the reported packaged 2026.4.26 and 2026.4.29 builds, based on detailed timing logs and event-loop symptoms. Current-main reproduction is unclear because the source has adjacent mitigations but no live packaged proof that the same two-phase stall still occurs or is gone.

## Solution Assessment

Unclear. The current mitigations are directionally right, but the best fix should come from fresh current-main attribution of the remaining startup or post-ready block rather than assuming one adjacent runtime-deps or readiness change fully resolves the user-visible outage.
