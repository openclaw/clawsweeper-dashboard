## Summary

Keep open. Current main has several adjacent mitigations for this startup-stall family, but it does not prove this WSL2 restart report is resolved: sidecar/channel startup is still awaited by default, the normal startup trace still stops at `ready`, and the client-side event-loop readiness helper does not prove third-party plugin loopback WebSocket recovery stays inside the handshake budget.

## What This Changes

Investigate a reported gateway restart event-loop stall where `sidecars.channels` takes about 45 seconds and a second post-ready freeze starves local WebSocket handshakes.

## Best Possible Solution

Keep this issue open until maintainers capture a current-main packaged restart on the reported shape with startup and after-ready event-loop attribution plus per-channel timings, then move the measured synchronous work off the gateway hot path or keep readiness and loopback clients gated until the event loop is actually responsive.

Yes for the reported 2026.4.26 host: the report includes concrete restart steps, phase timings, and repeatable plugin-side event-loop lag measurements. Current-main reproduction is unclear because the newer runtime-deps, readiness, and diagnostics changes have not been verified against the same WSL2/npm-global plus CoClaw plugin path.

## Solution Assessment

Unclear. The current mitigations are directionally useful, but closing as implemented would be premature until current main proves the channel-sidecar stall and post-ready WebSocket starvation are fixed or narrowed to a canonical remaining tracker.
