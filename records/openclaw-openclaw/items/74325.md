## Summary

Keep open. Current main has meaningful mitigations for this startup/starvation family, but it does not prove the reported WSL2 restart path is fixed: default startup still waits on sidecars, `sidecars.channels` still wraps channel startup, normal startup trace still disables at `ready`, and the related runtime-deps/client-readiness fixes do not establish that the CoClaw local WebSocket recovery path is back under the handshake budget.

## What This Changes

The issue asks OpenClaw to fix a reproducible gateway restart stall where channel sidecar startup and a second post-ready event-loop block starve local WebSocket handshakes on WSL2.

## Best Possible Solution

Capture a fresh current-main packaged WSL2 restart trace with startup and post-ready event-loop attribution, then remove the measured synchronous blocker from the channel/runtime-deps/provider path or keep readiness and WebSocket clients gated until the gateway event loop is genuinely responsive.
