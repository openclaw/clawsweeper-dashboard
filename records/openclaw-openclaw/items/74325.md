## Summary

Keep open. Current main has real mitigations for the startup-stall family, but it still does not prove this WSL2 restart report is resolved: default startup still awaits channel sidecars, the normal startup trace still ends at ready, and the newer runtime-deps/client-readiness changes do not prove the CoClaw local WebSocket recovery path is back under the handshake budget.

## What This Changes

The issue asks OpenClaw to eliminate or diagnose a repeatable gateway restart stall where `sidecars.channels` blocks for about 45 seconds and a second post-ready event-loop freeze causes local plugin WebSocket handshakes to time out.

## Best Possible Solution

Capture a current-main packaged restart on the reported WSL2 shape with startup and after-ready event-loop attribution plus per-channel timings, then move the measured synchronous work off the gateway hot path or keep readiness and local WS clients gated until the event loop is actually responsive.

## Reproduction Assessment

Yes for the reported 2026.4.26 host: the issue includes concrete restart steps, startup trace timings, and repeated event-loop lag observations. Current-main reproduction is unclear because the newer mitigations have not been verified against the same WSL2/npm-global plus CoClaw plugin path.

## Solution Assessment

Unclear for the current implementation. The existing mitigations are directionally useful, but closing as implemented would be premature until current main proves both the `sidecars.channels` stall and the post-ready WS handshake starvation are fixed or attributable to a narrower canonical tracker.
