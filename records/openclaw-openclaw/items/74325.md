## Summary

Keep open. Current main has meaningful mitigations for adjacent runtime-deps and primary-model prewarm costs, but it does not prove this combined restart stall is resolved: normal startup still waits for sidecars, the startup trace still stops event-loop sampling at `ready`, and the post-ready work path remains unattributed. The CoClaw reproduction data and the related #75069/#75183 work are useful evidence, but they are not enough to close this report.

## What This Changes

The issue asks OpenClaw to eliminate or attribute gateway main-thread stalls during restart, especially the long `sidecars.channels` phase and the post-ready freeze that causes local WebSocket handshakes to time out.

## Best Possible Solution

Keep the report open until maintainers capture a current-main packaged restart trace with event-loop attribution through the post-ready window. Then either fix the measured blocker in the owning startup/channel/runtime-deps path or close with explicit proof that current main restores the CoClaw local WebSocket recovery path.

## Reproduction Assessment

Yes for the reported 2026.4.26 host: the issue includes concrete restart steps, startup trace timings, and repeated plugin-side event-loop-lag observations. Current-main reproduction is unclear because this review did not run the WSL2/npm-global restart scenario after the newer runtime-deps and prewarm changes.

## Solution Assessment

Unclear. The current mitigations are directionally correct, but they are not yet a proven best solution for this combined report because normal readiness still waits on channel sidecars and post-ready work still lacks startup-trace attribution.
