## Summary

Keep open. Current main contains real mitigations for the reported startup family, including detached/scoped model prewarm, runtime-deps staging fixes, startup trace improvements, and event-loop readiness diagnostics, but it does not prove this combined WSL2 restart report is resolved: default startup still waits for channel sidecars, the normal startup trace still stops at `ready`, and no current-main packaged restart trace shows the CoClaw local WebSocket recovery path is back under the handshake budget.

## What This Changes

The issue reports deterministic gateway restart event-loop starvation on 2026.4.26, with a long `sidecars.channels` phase followed by an unattributed post-ready freeze that causes local plugin WebSocket handshakes to time out.

## Best Possible Solution

Capture a current-main packaged restart trace on the reported WSL2 shape with startup and after-ready event-loop attribution, then fix the measured synchronous blocker directly or keep readiness/WS clients gated until the gateway event loop is genuinely responsive.

## Reproduction Assessment

Yes for the reported 2026.4.26 host: the issue includes concrete restart steps, startup trace timings, and repeated plugin-side event-loop lag observations. Current-main reproduction is unclear because this review did not run the WSL2/npm-global restart scenario after the newer runtime-deps and event-loop mitigations.

## Solution Assessment

Unclear. The current mitigations are directionally useful, but they are not yet a proven best solution for this combined report because sidecar readiness still gates the default path and post-ready work still lacks the requested trace attribution.
