## Summary

Keep open. Current main has meaningful mitigations after the reported 2026.4.26 build, including detached model prewarm and several runtime-deps fixes that reference #75069/#75183, but the combined issue is not proven resolved: normal startup still waits for channel sidecars, startup tracing still disables event-loop sampling at `ready`, and post-ready work still runs without the attribution the reporter asked for.

## What This Changes

The issue reports deterministic gateway restart event-loop starvation around `sidecars.channels` plus a second after-ready freeze that causes local WebSocket handshakes from plugins to time out.

## Best Possible Solution

Keep this report open until maintainers capture a current-main packaged restart trace with after-ready attribution, then either fix the measured blocker directly or close it with explicit proof that the runtime-deps/channel fixes also restore the CoClaw local WebSocket recovery path.

## Reproduction Assessment

Yes for the reported 2026.4.26 host: the issue includes concrete restart steps, startup trace timings, and repeated plugin-side event-loop-lag observations. Current-main reproduction is unclear because this read-only review did not run the WSL2/npm-global restart scenario after the newer runtime-deps changes.

## Solution Assessment

Unclear. The current mitigations are directionally right, but they are not yet the best proven solution for this combined report because normal readiness still waits on channel sidecars and post-ready work is still not attributed by startup tracing.
