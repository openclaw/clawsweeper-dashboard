## Summary

Keep open. Current main contains useful mitigations, including detached primary-model prewarm, client-side event-loop readiness waiting, and runtime-mirror caching/hardlink improvements, but the combined restart starvation report is not proven fixed: channel startup still gates the normal ready path, the startup trace still disables event-loop sampling at `ready`, and the related runtime-deps/mirror work is only partially addressed in current main and still cross-referenced to open follow-up work in the provided context.

## What This Changes

The issue reports gateway restart event-loop starvation where `sidecars.channels` and immediate post-ready work block loopback WebSocket handshakes for roughly 75 seconds on OpenClaw 2026.4.26.

## Best Possible Solution

Keep this report open until maintainers capture a current-main restart trace with after-ready event-loop attribution, then either fix the measured channel/runtime/post-ready blocker directly or close it only after a merged canonical runtime-deps/startup PR proves the CoClaw WebSocket recovery case is no longer affected.

## Reproduction Assessment

Yes for the reported 2026.4.26 release and host: the issue includes concrete restart steps, startup-trace timings, and repeated plugin-side event-loop-lag observations. Current-main reproduction is unclear because this read-only review did not run the reporter's WSL2 npm-global restart scenario.

## Solution Assessment

Unclear. The current mitigations are directionally correct, but the narrow maintainable path is to add after-ready attribution and rerun the reporter's scenario before deciding whether the fix belongs in channel startup, runtime-deps mirroring, post-ready services, or a combination.
