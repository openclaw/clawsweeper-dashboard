## Summary

Keep open. Current main has several adjacent mitigations for gateway startup stalls, runtime-deps staging, primary-model prewarm, per-channel tracing, and client event-loop readiness, but it still does not prove this WSL2 restart report is fixed: default startup still waits for channel sidecars, the normal startup trace still disables at `ready`, and the SDK transport path used by third-party plugins still starts the Gateway WebSocket client directly.

## What This Changes

The issue asks OpenClaw to eliminate or diagnose a repeatable gateway restart stall where `sidecars.channels` blocks for about 45 seconds and a second post-ready event-loop freeze causes local plugin WebSocket handshakes to time out.

## Best Possible Solution

Capture a current-main packaged restart on the reported WSL2 shape with per-channel timings, startup timeline event-loop samples, after-ready lag attribution, and a CoClaw or equivalent loopback plugin WebSocket probe, then move the measured synchronous work off the hot path or make readiness and SDK connection semantics reflect actual event-loop availability.

## Reproduction Assessment

Yes for the reported 2026.4.26 host: the issue includes concrete restart steps, timing traces, and repeated event-loop lag observations. Current-main reproduction remains unclear because the same WSL2/npm-global plus CoClaw plugin path has not been remeasured after the newer mitigations.

## Solution Assessment

Unclear for current main. The existing mitigations are directionally useful, but the best fix depends on fresh attribution of the remaining stall rather than guessing between sidecar startup, runtime-deps, post-ready services, and plugin transport readiness.
