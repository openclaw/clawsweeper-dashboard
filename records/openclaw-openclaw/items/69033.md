## Summary

Keep open. Current main still has the false-overflow gap because the tool-result estimator adds hidden `details` to the preemptive context estimate, but the PR’s persistence half is stale against current main’s newer whole-object details byte cap and should be rebased rather than merged as-is.

## What This Changes

The PR removes `toolResult.details` from tool-result character estimation and adds field-based truncation for oversized persisted detail strings with regression tests.

## Best Possible Solution

Rebase the PR on current main and narrow the production change to excluding model-invisible `toolResult.details` from the tool-result estimator, while preserving the existing persisted-details byte cap and adding the aggregate false-overflow regression test.

## Reproduction Assessment

Yes. A current-main context with modest visible content and a large `toolResult.details` blob can be reasoned through the estimator path: `details` is added at `tool-result-char-estimator.ts:130`, then the aggregate guard checks that inflated estimate at `tool-result-context-guard.ts:421`.

## Solution Assessment

Partly. The estimator change is the narrowest maintainable fix for the false overflow, but the PR's persistence truncation approach is no longer the best solution because current main already has a broader byte-bounded sanitizer.
