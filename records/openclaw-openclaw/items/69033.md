## Summary

Keep open. The estimator part of the PR still fixes a real current-main false-overflow path, but the branch is not safe to merge as-is because its persisted-details whitelist is weaker than the whole-object byte cap already on main and it lacks the required changelog entry for a user-facing reliability fix.

## What This Changes

The PR removes `toolResult.details` from tool-result character estimation and adds field-based truncation for oversized persisted detail strings with regression tests.

## Best Possible Solution

Land a rebased, narrow fix that excludes model-invisible `toolResult.details` from the context estimator, preserves current main's 8 KiB whole-object persisted-details sanitizer, adds the false-overflow regression test, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A current-main context with modest visible user/tool-result text and a large hidden `toolResult.details` object follows a source-backed path: the estimator adds `details`, and the aggregate guard uses that inflated estimate even though the LLM boundary strips details before provider conversion.

## Solution Assessment

No, not as-is. Excluding details from the estimator is the narrow maintainable fix, but the persistence portion should preserve the current byte-bounded sanitizer rather than replacing it with a top-level field whitelist.
