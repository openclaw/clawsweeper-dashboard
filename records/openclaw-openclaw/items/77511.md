## Summary

Keep open: this protected maintainer-labeled updater architecture follow-up is still necessary because current main has staged swaps, post-core respawn, and managed-service handoff mitigations, but it does not implement a durable external updater that owns package replacement outside the `openclaw` install root.

## What This Changes

Request to move `openclaw update` package replacement into a small external updater process outside the package being replaced.

## Best Possible Solution

Define and implement a core external-updater handoff where `openclaw update` writes a durable update plan, exits before package replacement, and a tiny updater outside `node_modules/openclaw` performs staged install, verification, swap, rollback, post-update doctor, plugin convergence, and restart handoff with cross-platform tests and docs.

## Reproduction Assessment

Not applicable: this is an architectural follow-up request, not a bug report with a failing current-main reproduction path. Source inspection gives high confidence that the requested external-updater boundary is still absent.

## Solution Assessment

Unclear as an implementation: the requested direction is coherent and likely durable, but the exact plan format, helper packaging, rollback model, doctor handoff, and service ownership need maintainer design before code should land.
