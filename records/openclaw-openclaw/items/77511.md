## Summary

The item still looks necessary: current `main` has staged npm swaps and managed-service handoffs for Gateway/control-plane update paths, but the ordinary CLI package replacement logic still lives in the normal updater process rather than a tiny external updater with a durable plan and rollback boundary. The protected `maintainer` label plus update/security architecture scope means this should stay open for maintainer design, not cleanup closure.

## What This Changes

Request to move `openclaw update` package replacement into a small external updater that runs outside the installed package being replaced.

## Best Possible Solution

Keep the staged-swap and managed-service handoff mitigations, then design a small trusted external updater with a durable plan schema, strict temp-file permissions, package verification, service handoff, post-update doctor/plugin convergence, and rollback policy before replacing the current in-process package swap.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this issue is a protected architecture follow-up, not a single deterministic failure report. Source inspection does confirm the central remaining gap: the standalone CLI updater still owns package swap execution, while related bug reports provide concrete failure modes that motivate the design.

## Solution Assessment

Yes in direction, but not yet as an implementation plan: moving package replacement out of the package being replaced is the right durable boundary. The best fix needs maintainer decisions on plan format, trusted helper packaging, rollback, service coordination, and security policy before code work starts.
