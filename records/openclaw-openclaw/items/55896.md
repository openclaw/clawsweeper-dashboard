## Summary

Keep open. Current main still enqueues ACP parent relay `:progress`, `:stall`, and `:resumed` events for visible parent sessions, and the spawn path does not pass any Discord/thread-specific suppression flag. The PR is a narrow implementation for the linked Discord ACP thread bug, and I found no concrete blocking review finding in the provided diff, but it still needs maintainer review and likely rebase against current main.

## What This Changes

The PR adds a relay suppression flag for ACP `progress`, `stall`, and `resumed` system events, wires it for Discord thread-bound parent sessions, re-exports the thread-parent session-key helper, and adds relay/spawn regression tests.

## Best Possible Solution

Review and land this narrow suppression, or reimplement the same behavior on current main: Discord thread-bound ACP parent relays should suppress only intermediate `progress`, `stall`, and `resumed` visible system events while preserving relay logs, start/done/error/timeout lifecycle notices, and all non-Discord or non-thread behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by starting an ACP parent relay for a Discord thread-bound parent session, emitting assistant/no-output/resumed events, and observing that current main enqueues the `:progress`, `:stall`, and `:resumed` system events into the parent session.

## Solution Assessment

Yes, the PR direction appears to be the narrowest maintainable fix: add an explicit suppression seam in the ACP relay and only enable it for Discord thread-bound parents. Disabling `surfaceUpdates` entirely would be broader and would also hide lifecycle notices that the report says should remain visible.
