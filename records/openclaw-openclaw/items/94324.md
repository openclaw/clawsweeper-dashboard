## Summary

Keep this PR open. Current main and v2026.6.8 still accept only plaintext strings for Codex app-server `authToken` and `headers`, but this branch is not merge-ready because `appServer.headers.*` can mis-assign dotted header names during SecretRef resolution and the PR still lacks real SecretRef-backed app-server proof.

## What This Changes

The PR expands Codex app-server `authToken` and `headers.*` to accept SecretInput values, resolves plugin SecretRefs before runtime option construction, hashes header values in shared-client keys, and updates SecretRef docs/tests.

## Best Possible Solution

Land the manifest-owned SecretInput expansion only after header wildcard assignment preserves literal map keys, dotted-header regression coverage is added, and redacted real app-server proof shows a SecretRef-backed token or header connects.

## Reproduction Assessment

Yes for the review finding: source inspection shows `appServer.headers.*` wildcard matches return a dotted path string and the assignment writer later splits that path on dots. The feature itself still lacks real app-server runtime proof.

## Solution Assessment

No. The manifest-owned SecretInput expansion is the right layer, but this implementation still needs segment-safe wildcard assignment and real Codex app-server proof before it is the best fix.
