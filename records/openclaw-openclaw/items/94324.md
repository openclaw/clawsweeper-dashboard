Keep open: current main does not support Codex app-server SecretRefs yet, and the PR is useful but not merge-ready because wildcard header SecretRefs can lose dotted header keys and the PR only provides test commands, not real app-server proof.

## What This Changes

The PR expands Codex app-server `authToken` and `headers.*` to accept SecretInput values, resolves them through plugin secret contracts, hashes header values in shared-client keys, and updates docs/tests.

## Best Possible Solution

Land the SecretRef expansion after preserving concrete wildcard header keys during assignment and adding redacted real Codex app-server proof for a SecretRef-backed authToken or header connection.

## Reproduction Assessment

Yes for the review finding: source inspection shows wildcard header keys are serialized as dotted paths and assignment later splits on dots, so a header key containing `.` would not receive the resolved SecretRef value. No real app-server runtime proof was provided for the new feature itself.

## Solution Assessment

No. The manifest-owned SecretInput expansion is the right layer, but it needs segment-safe wildcard assignment and real Codex app-server proof before it is the best fix.
