## Summary

The PR reproduces and repairs the released custom-provider auth regression, but its explicit-marker fallback can still return the active snapshot credential to a different caller config that reuses the same provider ID. That credential-isolation defect should be fixed before merge.

## What This Changes

The PR makes managed SecretRef custom-provider API keys available from the runtime config snapshot during both the fast availability check and final credential resolution, with regression tests and a file-backed reproduction script.

## Best Possible Solution

Require the custom-provider snapshot lookup to pass the canonical source-snapshot applicability check, then add a regression test where a mismatched caller also uses the managed marker and must not receive the active credential.

## Reproduction Assessment

Yes. The file-backed repro follows the production SecretRef preparation path and exercises both the fast availability check and final credential resolution with the reported source/runtime config shapes.

## Solution Assessment

No, not yet. Sharing snapshot-backed lookup between availability and resolution is the correct direction, but the lookup must also enforce the existing source-snapshot applicability contract rather than treating the marker alone as authorization.
