## Summary

Keep this PR open: current `main` still lacks the requested LLM log source, but the submitted patch is not mergeable as-is because it exposes cache-trace content through the read-scoped logs RPC and is stale against current logs fallback/reconnect behavior.

## What This Changes

The PR adds `openclaw logs --source gateway|llm|all`, extends `logs.tail` with source metadata/hints, tails cache-trace JSONL as LLM logs, pretty-prints trace JSON, raises the default max-byte window, and adds CLI/gateway tests.

## Best Possible Solution

Define a safe cache-trace access or redaction contract, rebase while preserving current logs fallback/reconnect behavior, and update CLI docs plus changelog before merge.

## Reproduction Assessment

Yes. Current-main absence is source-reproducible from the CLI registration, protocol schema, and `logs.tail` handler; PR-head blockers are source-reproducible from the submitted diff.

## Solution Assessment

No. Reusing cache trace may be a reasonable direction, but this PR is not the best mergeable form until the sensitive-data access contract and current logs reliability regressions are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate or redact cache-trace log results:** `src/gateway/server-methods/logs.ts:208`
  - body: `logs.tail` is currently available to `operator.read`, but the new LLM source returns cache-trace records that can include prompts, system content, and message payloads by default. Add a stricter scope, explicit opt-in, or redaction policy before serving these trace lines through this RPC.
  - confidence: 0.92
- **[P1] Keep LLM logs usable with invalid configs:** `src/gateway/server-methods/logs.ts:58-59`
  - body: `resolveLlmLogFile()` and `isLlmSourceEnabled()` call `loadConfig()`, which throws on invalid configs. That makes `openclaw logs --source llm` and `--source all` fail during config breakage, even though logs are a troubleshooting path that should still work.
  - confidence: 0.88
- **[P2] Preserve local log fallback while rebasing:** `src/cli/logs-cli.ts:61-64`
  - body: Current main falls back to the configured file log when implicit local loopback RPC closes, times out, or asks for pairing. This PR's `fetchLogs()` calls `logs.tail` directly and drops that fallback, so the rebase needs to keep it for gateway logs.
  - confidence: 0.84
- **[P2] Honor plain mode for trace JSON:** `src/cli/logs-cli.ts:347`
  - body: When stdout is rich and the user passes `--plain`, `rich` can still be true and trace rendering passes it into `formatPrettyJson()`. That can emit ANSI escapes in a mode documented as no styling.
  - confidence: 0.86
- **[P2] Default missing sources to gateway:** `src/cli/logs-cli.ts:87-89`
  - body: `source` is optional for older gateway responses, but `toPublicSource(undefined)` currently returns `llm`. Older gateway log payloads would be mislabeled in output and notices, so the safe fallback should be `gateway`.
  - confidence: 0.92
- **[P3] Document and changelog the new log source:** `src/cli/logs-cli.ts:455`
  - body: The PR adds a user-facing `--source` option and changes the default log byte window, but the CLI docs and changelog are not updated. Add both so command reference and release policy stay aligned.
  - confidence: 0.82
