## Summary

Keep open: this is a useful diagnostics feature, but the current PR is not merge-ready because it bypasses log-tail redaction, can break logs under invalid config, is dirty against current main, and still lacks real behavior proof.

## What This Changes

The PR adds a `--source gateway|llm|all` option to `openclaw logs`, expands `logs.tail` source metadata, and renders cache-trace JSONL as pretty CLI output.

## Best Possible Solution

Rebase onto current main and extend shared log-tail helpers to support source-aware gateway/LLM tails while preserving redaction, invalid-config diagnostics, local fallback/follow behavior, docs, tests, and redacted CLI proof.

## Reproduction Assessment

Yes for the review defects by source inspection. The PR branch returns raw `readLogSlice` output, calls `loadConfig()` in the LLM path, and passes `rich` into trace JSON rendering even when `--plain` disables pretty output.

## Solution Assessment

No. Exposing existing cache-trace data through `openclaw logs` is a reasonable direction, but this patch duplicates log reading instead of extending the redacted, config-safe log-tail contract on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve log-tail redaction before returning lines:** `src/gateway/server-methods/logs.ts:208`
  - body: Current main routes log tails through `readConfiguredLogTail`, which applies `redactSensitiveLines` before returning RPC lines. This branch responds with raw `readLogSlice` results here, so `logs.tail` can expose tokens or credentials in gateway logs.
  - confidence: 0.94
- **[P2] Keep LLM log lookup out of invalid-config paths:** `src/gateway/server-methods/logs.ts:59`
  - body: Docs say `openclaw logs` remains available when config validation fails. The LLM path calls `loadConfig()` in `resolveLlmLogFile()` and `isLlmSourceEnabled()`, so malformed config can make `--source llm` and `--source all` fail while troubleshooting.
  - confidence: 0.86
- **[P2] Honor --plain when rendering trace JSON:** `src/cli/logs-cli.ts:347`
  - body: When stdout is a TTY and the user passes `--plain`, `pretty` becomes false but `rich` can remain true. This trace rendering path still passes `rich` into `formatPrettyJson`, so JSON blocks can contain ANSI styling despite the plain-output contract.
  - confidence: 0.9
- **[P2] Default missing source metadata to gateway:** `src/cli/logs-cli.ts:88`
  - body: `LogsTailPayload.source` is optional for older gateways, but `toPublicSource(undefined)` returns `llm`. Older gateway responses would have normal gateway log metadata and notices mislabeled as LLM output.
  - confidence: 0.88
