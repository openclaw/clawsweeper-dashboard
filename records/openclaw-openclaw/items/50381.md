## Summary

Keep open: current main still lacks this log-source capability, but this PR is not mergeable as-is because it would expose cache-trace prompt/message/system content through the existing read-scoped logs RPC and is stale against current logs fallback/reconnect behavior.

## What This Changes

The PR adds `openclaw logs --source gateway|llm|all`, extends `logs.tail` source metadata, reads `diagnostics.cacheTrace` JSONL as LLM logs, and adds pretty JSON trace output with CLI/gateway tests.

## Best Possible Solution

Define a safe LLM trace access or redaction contract, rebase while preserving current logs reliability behavior, then update CLI/protocol docs and changelog before merge.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the `--source` option and LLM `logs.tail` branch, and the PR-head blockers are visible in the submitted diff plus the current cache-trace and scope contracts.

## Solution Assessment

No. Reusing cache trace is plausible, but the PR should first add a stricter access/redaction contract and preserve current diagnostic fallback behavior instead of exposing raw trace lines through the existing read-scoped logs method.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate or redact cache-trace log results:** `src/gateway/server-methods/logs.ts:208`
  - body: `logs.tail` is currently reachable with `operator.read`, but cache trace can include prompt text, system prompt, and full message payloads by default. Add a stricter scope, explicit opt-in, or redaction policy before returning LLM trace lines through this RPC.
  - confidence: 0.92
- **[P1] Keep LLM logs usable with invalid configs:** `src/gateway/server-methods/logs.ts:58-59`
  - body: `resolveLlmLogFile()` and `isLlmSourceEnabled()` call `loadConfig()`, which throws on invalid configs. That makes `openclaw logs --source llm` and `--source all` fail during config breakage, even though logs are a diagnostic path users need when config is broken.
  - confidence: 0.88
- **[P2] Preserve local log fallback while rebasing:** `src/cli/logs-cli.ts:61-64`
  - body: Current main falls back to the configured file log when implicit local loopback RPC closes, times out, or asks for pairing. This PR's `fetchLogs()` calls `logs.tail` directly and drops that fallback, so the rebase needs to keep it for gateway logs.
  - confidence: 0.84
- **[P2] Honor plain mode for trace JSON:** `src/cli/logs-cli.ts:347`
  - body: When stdout is rich and the user passes `--plain`, trace rendering can still pass a rich flag into `formatPrettyJson()`. That can emit ANSI escapes in a mode documented as no styling.
  - confidence: 0.86
- **[P2] Default missing sources to gateway:** `src/cli/logs-cli.ts:87-89`
  - body: `source` is optional for older gateway responses, but `toPublicSource(undefined)` currently returns `llm`. Older gateway payloads would be mislabeled in output and notices, so the safe fallback should be `gateway`.
  - confidence: 0.92
- **[P3] Document and changelog the new log source:** `src/cli/logs-cli.ts:455`
  - body: The PR adds a user-facing `--source` option and changes the default log byte window, but the CLI docs and changelog are not updated. Add both so command reference and release policy stay aligned.
  - confidence: 0.82
