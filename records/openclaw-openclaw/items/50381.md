## Summary

Keep open: current main still does not expose `openclaw logs --source llm`, so the feature is not obsolete, but the PR is not merge-ready because it bypasses the current redacted log-tail path, is stale/conflicting against newer logs fallback behavior, and lacks real behavior proof.

## What This Changes

The PR adds `openclaw logs --source gateway|llm|all`, extends `logs.tail` source metadata and hints, and pretty-prints cache-trace JSONL in CLI output.

## Best Possible Solution

Expose cache-trace logs through the existing redacted, config-safe log-tail contract, preserve current fallback/reconnect behavior during rebase, and require redacted real CLI proof before merge.

## Reproduction Assessment

Yes for the review defects by source inspection: the PR branch returns raw `readLogSlice` output, calls `loadConfig()` in the LLM path, passes `opts.rich` into trace JSON rendering under `--plain`, and maps missing source metadata to `llm`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The user-facing feature is useful, but the implementation should extend the shared redacted, config-safe log-tail path instead of adding a parallel raw reader in the gateway handler.

  - body: When stdout is a TTY and the user passes `--plain`, `pretty` becomes false but `rich` can remain true. This trace rendering path still passes `opts.rich` into `formatPrettyJson`, so JSON blocks can contain ANSI styling despite the plain-output contract.
  - confidence: 0.9
- **[P2] Default missing source metadata to gateway:** `src/cli/logs-cli.ts:88`
  - body: `LogsTailPayload.source` is optional, but `toPublicSource(undefined)` returns `llm`. Older gateways that omit the new field would have normal gateway lines and notices mislabeled as LLM output.
  - confidence: 0.88
