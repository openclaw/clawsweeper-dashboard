## What This Changes

Adds `openclaw logs --source gateway|llm|all`, extends `logs.tail` with source/hint fields, reads cache-trace JSONL for LLM traces, pretty-prints trace JSON, and adds focused tests.

## Best Possible Solution

Define a safe LLM trace access/redaction contract, preserve existing logs resilience and redaction, update protocol/docs, and land with real CLI proof.

## Reproduction Assessment

Yes, source-level. Current main has no logs `--source` support or LLM `logs.tail` branch, while the PR head adds both and the blocking defects are visible in the inspected source.

## Solution Assessment

No. Reusing cache trace may be reasonable, but this implementation is not the best mergeable solution until access/redaction is approved and current log-tail redaction, fallback, follow-retry, and protocol/docs contracts are preserved.

Full review comments:

- **[P1] Gate or redact cache-trace log results:** `src/gateway/server-methods/logs.ts:208`
  - body: `logs.tail` is read-scoped, but this new LLM path returns cache-trace lines that can include prompt, system, and message payloads. Add a stricter permission, redaction mode, or explicit documented access contract before serving this source.
  - confidence: 0.93
- **[P1] Keep gateway log tails on the redacted path:** `src/gateway/server-methods/logs.ts:208`
  - body: The new shared `readLogSlice` path also serves `source: "gateway"`, bypassing current main's `readConfiguredLogTail` redaction before responding. Keep the existing helper or apply the same redaction before returning gateway log lines.
  - confidence: 0.9
- **[P1] Avoid full config reads in logs.tail:** `src/gateway/server-methods/logs.ts:58-76`
  - body: `resolveLlmLogFile()` and `isLlmSourceEnabled()` call `loadConfig()`. Invalid config can make `--source llm` fail, and `--source all` can reject the whole request before showing gateway logs, which regresses a troubleshooting path.
  - confidence: 0.88
- **[P2] Preserve logs fallback and follow reconnects:** `src/cli/logs-cli.ts:61-64`
  - body: This fetch path exits directly on any RPC error and does not preserve current main's implicit local file-log fallback or transient `--follow` reconnect loop. `openclaw logs --source gateway` should keep the current resilience behavior.
  - confidence: 0.88
- **[P2] Honor plain mode for trace JSON:** `src/cli/logs-cli.ts:347`
  - body: When stdout is a rich TTY and the user passes `--plain`, `pretty` is false but `rich` remains true, so `formatPrettyJson(trace.parsed, opts.rich)` can still emit ANSI escapes. Pass a non-rich flag for plain output.
  - confidence: 0.86
- **[P2] Default missing log sources to gateway:** `src/cli/logs-cli.ts:87-88`
  - body: `source` is optional for compatibility with older gateway responses, but `toPublicSource(undefined)` returns `llm`. Default anything except an explicit `llm` to `gateway` so older gateway log lines and notices are not mislabeled.
  - confidence: 0.92
- **[P2] Update the log-source protocol and CLI docs:** `src/gateway/protocol/schema/logs-chat.ts:9-23`
  - body: The schema adds `source` and `hint`, and the CLI adds `--source` plus a new default byte window, but the protocol/CLI docs and generated protocol check are not in sync. Update the public contract and docs with the new fields and behavior.
  - confidence: 0.8
