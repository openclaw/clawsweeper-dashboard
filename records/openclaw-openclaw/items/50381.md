## Summary

Keep this PR open: the feature is still absent from current main, but the proposed patch is not merge-ready because it exposes cache-trace content through the existing read-scoped logs RPC and is stale against current `openclaw logs` fallback behavior.

## What This Changes

The PR adds `openclaw logs --source gateway|llm|all`, extends `logs.tail` source metadata, reads cache-trace JSONL as LLM logs, pretty-prints trace JSON, and adds CLI/gateway tests.

## Best Possible Solution

Land the feature only after defining a safe cache-trace access or redaction contract, rebasing onto current logs behavior, and updating CLI docs, protocol docs, and the changelog.

## Reproduction Assessment

Yes. Current-main absence is source-reproducible by inspecting the CLI option registration, protocol schema, and `logs.tail` handler; the PR-head defects are source-reproducible from the submitted diff.

## Solution Assessment

No. Reusing cache trace is a plausible implementation direction, but this patch is not the best mergeable solution until it gates or redacts sensitive trace content and preserves current logs fallback behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate or redact cache-trace log results:** `src/gateway/server-methods/logs.ts:208`
  - body: `logs.tail` is currently reachable with `operator.read`, but cache trace can include prompts, system content, and message payloads by default. Add a stricter scope, explicit opt-in, or redaction policy before returning LLM trace lines through this RPC.
  - confidence: 0.92
- **[P1] Keep LLM logs usable with invalid configs:** `src/gateway/server-methods/logs.ts:59`
  - body: `resolveLlmLogFile()` and `isLlmSourceEnabled()` call `loadConfig()`, which throws on invalid configs. That makes `openclaw logs --source llm` and `--source all` fail during config breakage, even though logs are documented as a diagnostic command that should still work.
  - confidence: 0.88
- **[P2] Preserve local log fallback while rebasing:** `src/cli/logs-cli.ts:61-64`
  - body: Current main falls back to the configured file log when implicit local loopback RPC closes, times out, or asks for pairing. This PR's `fetchLogs()` calls `logs.tail` directly and drops that fallback, so the rebase needs to keep it for gateway logs.
  - confidence: 0.84
- **[P2] Honor plain mode for trace JSON:** `src/cli/logs-cli.ts:347`
  - body: When stdout is rich and the user passes `--plain`, `rich` can still be true and trace rendering passes it into `formatPrettyJson()`. That can emit ANSI escapes in a mode documented as no styling.
  - confidence: 0.86
- **[P2] Default missing sources to gateway:** `src/cli/logs-cli.ts:87-89`
  - body: `source` is optional for older gateway responses, but `toPublicSource(undefined)` currently returns `llm`. Older gateway log payloads would be mislabeled in output and notices, so the safe fallback should be `gateway`.
  - confidence: 0.92
- **[P3] Document and changelog the new log source:** `src/cli/logs-cli.ts:455`
  - body: The PR adds a user-facing `--source` option and changes the default log byte window, but the CLI docs and changelog are not updated. Add both so command reference and release policy stay aligned.
  - confidence: 0.82
