## Summary

Keep open: current main still lacks the requested LLM log source, but this PR is not merge-ready because it exposes cache-trace content through read-scoped `logs.tail` and needs a rebase that preserves newer logs resilience behavior.

## What This Changes

The PR adds `openclaw logs --source gateway|llm|all`, extends `logs.tail` metadata for source-aware results, reads cache-trace JSONL as LLM logs, pretty-prints trace JSON, and adds CLI/gateway tests.

## Best Possible Solution

Define a safe LLM trace access or redaction contract, rebase onto current logs CLI behavior, then update CLI/protocol docs and changelog before merge.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `--source` and an LLM `logs.tail` branch, and the PR-head blockers are visible in the submitted diff and raw branch files.

## Solution Assessment

No. Reusing cache trace may be the right backend, but the PR should first settle a stricter access/redaction contract and preserve current logs reliability behavior during rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate or redact cache-trace log results:** `src/gateway/server-methods/logs.ts:208`
  - body: `logs.tail` is currently reachable with `operator.read`, but cache trace can include prompt text, system prompt, and message payloads by default. Add a stricter scope, explicit opt-in, or redaction policy before returning LLM trace lines through this RPC.
  - confidence: 0.92
- **[P1] Avoid config loads in diagnostic log reads:** `src/gateway/server-methods/logs.ts:58-59`
  - body: `resolveLlmLogFile()` and `isLlmSourceEnabled()` call `loadConfig()`, which throws on invalid configs. That makes `openclaw logs --source llm` and `--source all` fail during config breakage, even though logs are a diagnostic path users need when config is broken.
  - confidence: 0.88
- **[P2] Preserve current logs resilience on rebase:** `src/cli/logs-cli.ts:61-64`
  - body: Current main falls back to configured file logs for implicit local RPC failures and retries transient `--follow` disconnects. This branch calls `logs.tail` directly and exits through the generic error path, so the rebase needs to keep the current gateway-log fallback and follow retry behavior.
  - confidence: 0.86
- **[P2] Honor --plain for trace JSON blocks:** `src/cli/logs-cli.ts:347`
  - body: When stdout is rich and the user passes `--plain`, trace rendering can still pass `opts.rich` into `formatPrettyJson()`. That can emit ANSI escapes in a mode documented as no styling.
  - confidence: 0.86
- **[P2] Default missing sources to gateway:** `src/cli/logs-cli.ts:87-89`
  - body: `source` is optional for older gateway responses, but `toPublicSource(undefined)` currently returns `llm`. Older gateway payloads would be mislabeled in output and notices, so the safe fallback should be `gateway`.
  - confidence: 0.92
- **[P3] Document and changelog the new log source:** `src/cli/logs-cli.ts:455`
  - body: The PR adds a user-facing `--source` option and changes the default log byte window, but the CLI docs and changelog are not updated. Add both so command reference and release policy stay aligned.
  - confidence: 0.82
