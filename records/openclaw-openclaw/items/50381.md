## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate or redact cache-trace log results:** `src/gateway/server-methods/logs.ts:208`
  - body: `logs.tail` is available through `operator.read`, but this path returns cache-trace lines that can include prompt text, system content, and message payloads. Add a stricter scope, redaction mode, or explicit access contract before serving these lines through the existing read RPC.
  - confidence: 0.92
- **[P1] Avoid full config loads while serving logs:** `src/gateway/server-methods/logs.ts:58-59`
  - body: `resolveLlmLogFile()` and `isLlmSourceEnabled()` call `loadConfig()`, so an invalid config makes `openclaw logs --source llm` and `--source all` fail through `UNAVAILABLE`. Logs are a diagnostic path during config breakage, so the LLM path should degrade without hiding gateway logs.
  - confidence: 0.88
- **[P2] Preserve current logs fallback and retry behavior:** `src/cli/logs-cli.ts:61-64`
  - body: This branch calls `logs.tail` directly and falls into the generic error path, while current main falls back to configured file logs for implicit local RPC failures and retries transient `--follow` disconnects. Preserve those behaviors when adding multiple sources.
  - confidence: 0.86
- **[P2] Honor --plain when rendering trace JSON:** `src/cli/logs-cli.ts:347`
  - body: `buildParsedOutputLine()` renders trace JSON with `opts.rich`, so a rich TTY plus `--plain` can still emit ANSI escapes even though plain mode promises no styling. Gate trace JSON color on the plain/pretty decision too.
  - confidence: 0.86
- **[P2] Default missing log sources to gateway:** `src/cli/logs-cli.ts:87-88`
  - body: `source` is optional for older gateway responses, but `toPublicSource(undefined)` currently returns `llm`. Older gateway payloads would be mislabeled in output and notices; the safe fallback should be `gateway`.
  - confidence: 0.92
