## Summary

Keep open: current main does not implement LLM log-source support, and the PR still has merge-blocking issues around cache-trace exposure, stale local-fallback behavior, invalid-config handling, and plain-mode output.

## What This Changes

The PR adds `--source gateway|llm|all` to `openclaw logs`, extends `logs.tail` with source metadata and hints, reads cache-trace JSONL for LLM logs, pretty-prints trace JSON, and adds CLI/gateway tests.

## Best Possible Solution

Keep the feature direction, but land it only after maintainers choose a safe cache-trace access/redaction contract, the branch is rebased to preserve current local fallback behavior, LLM log resolution remains useful when config is broken, plain output is truly unstyled, and docs/changelog/protocol references are updated.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the CLI option registration, protocol schema, handler, and docs; the PR-head defects are reproducible by reading the `loadConfig()` calls, source fallback removal, `rich` calculation, and `toPublicSource(undefined)` branch.

## Solution Assessment

No. Reusing cache trace is a reasonable implementation direction, but the current patch is not the narrowest safe solution until it gates or redacts sensitive trace content and preserves the troubleshooting behavior already on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate or redact cache-trace log results:** `src/gateway/server-methods/logs.ts:208`
  - body: `logs.tail` is currently authorized by `operator.read`, but cache trace can include prompts, system content, and message payloads by default. Returning `source: "llm"` through this same RPC broadens read-scoped clients to sensitive trace content, so add a stricter scope or redaction policy before returning these lines.
  - confidence: 0.92
- **[P1] Keep LLM logs usable with invalid configs:** `src/gateway/server-methods/logs.ts:59`
  - body: `resolveLlmLogFile()` and `isLlmSourceEnabled()` call `loadConfig()`. If the config file is invalid, `openclaw logs --source llm` and `--source all` fail before showing logs, which regresses a troubleshooting command exactly when users need it.
  - confidence: 0.88
- **[P2] Preserve local log fallback while rebasing:** `src/cli/logs-cli.ts:61-64`
  - body: Current main catches local loopback close/timeout/pairing failures and falls back to the configured file log. This PR's `fetchLogs()` calls `logs.tail` directly and drops that fallback, so `openclaw logs` would regress on local gateway RPC failures unless the rebase keeps the fallback for gateway logs.
  - confidence: 0.84
- **[P2] Honor plain mode for trace JSON:** `src/cli/logs-cli.ts:347`
  - body: When stdout is rich and the user passes `--plain`, `rich` can still be true, and trace rendering passes that into `formatPrettyJson()`. That lets ANSI coloring appear in a mode documented as no-styling output.
  - confidence: 0.86
- **[P2] Default missing sources to gateway:** `src/cli/logs-cli.ts:87-89`
  - body: `source` is optional for older gateway responses, but `toPublicSource(undefined)` currently returns `llm`. Older gateway log payloads would be mislabeled in output and notices, so the safe fallback should be `gateway`.
  - confidence: 0.92
- **[P3] Document and changelog the new log source:** `src/cli/logs-cli.ts:455`
  - body: The PR adds a user-facing `--source` option and changes the default log byte window, but it does not update the CLI docs or `CHANGELOG.md`. Add the docs and release note so command reference and release policy stay aligned.
  - confidence: 0.82
