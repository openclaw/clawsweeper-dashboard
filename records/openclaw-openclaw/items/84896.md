## Summary

Keep open; the PR addresses a real memory-wiki/LanceDB bridge gap, but it is not merge-ready because the loader change can retain discovery-only memory capability state outside full plugin activation.

## What This Changes

The PR exports shared memory workspace artifacts through the SDK, adds memory-lancedb wiki bridge artifacts with URI-backed dbPath skipping, preserves discovered memory capability state on activate:false loads, and changes native wiki links to omit .md targets.

## Best Possible Solution

Keep the artifact provider and URI skip, but preserve only publicArtifacts from discovery loads and derive LanceDB artifact ownership metadata from the current config instead of hardcoding main.

## Reproduction Assessment

Yes. Source inspection shows current main's bridge reads active memory public artifacts while memory-lancedb does not provide them, and the PR's loader regression follows from preserving the full discovered memory capability on activate:false loads.

## Solution Assessment

No, not yet. The artifact ownership move and URI-backed dbPath skip are the right direction, but the loader restore path must stay artifact-only and the LanceDB artifact metadata should not assume the main agent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve only artifact state from discovery loads:** `src/plugins/loader.ts:2433`
  - body: This keeps the full discovered memory capability when `activate:false` runs with no previous capability. Discovery mode still lets memory plugins register prompt builders, flush plans, and runtimes, and `memory-core` registers all of those, so a read-only snapshot load can silently alter agent prompt, compaction, and memory runtime state before full activation. Keep runtime fields restored and expose only the public artifact provider needed by the bridge.
  - confidence: 0.88
- **[P2] Derive LanceDB artifact agent ids from config:** `extensions/memory-lancedb/index.ts:584`
  - body: The new LanceDB bridge artifact always declares `agentIds: ["main"]`, but bridge import uses artifact agent ids for source titles and frontmatter. Non-main or multi-agent configs will get incorrect ownership/provenance for the exported LanceDB memory page, while workspace artifacts already preserve real ids.
  - confidence: 0.76
