## Summary

Keep open; the fix direction matches the related Telegram delivery tracker, but this patch still weakens full shutdown abort behavior and omits the required user-facing changelog entry.

## What This Changes

The PR adds a Telegram fetch-abort method allow-list, applies it to polling sessions for getUpdates-only restart aborts, and adds regression coverage for overlapping sendMessage preservation.

## Best Possible Solution

Keep getUpdates-only aborts for polling-restart recovery, preserve full abort behavior for actual polling-session shutdown, add the active-version changelog entry, and revalidate the Telegram changed lane.

## Reproduction Assessment

Yes for the PR regression: the PR attaches fetchAbortMethods: ["getupdates"] to the same controller that stopRunner and the outer abortSignal abort, so sendMessage would ignore a real shutdown abort and wait for its request timeout. The broader related user report still lacks deterministic live reproduction.

## Solution Assessment

No. The scoped-abort idea is the right narrow direction, but the implementation should separate restart-only getUpdates aborts from full session-shutdown aborts before merge.
