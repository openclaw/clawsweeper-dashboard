## Summary

Keep open. The PR targets a real Telegram polling-restart reply-loss path, but the patch should not merge as-is because it also scopes full polling-session shutdown aborts away from in-flight sendMessage calls and is missing the required changelog entry.

## What This Changes

The PR adds a Telegram bot fetch-abort method allow-list, applies it to polling sessions for getUpdates-only restart aborts, and adds regression coverage for overlapping sendMessage preservation.

## Best Possible Solution

Keep the getUpdates-only abort for polling-restart recovery, but preserve full abort behavior for actual polling-session shutdown, then add the active-version changelog entry and revalidate the Telegram changed lane.

## Reproduction Assessment

Yes for the PR regression: in the PR head, the polling session passes fetchAbortMethods: ["getupdates"] to the same controller that outer abortSignal and stopRunner abort, so an in-flight sendMessage no longer observes shutdown abort. The broader related user report still lacks a deterministic live reproduction, but the source-level abort path is clear.

## Solution Assessment

No. The scoped-abort idea is the right narrow direction, but this implementation needs separate restart-vs-shutdown abort semantics so restart recovery preserves replies without weakening full shutdown cleanup.
