## Summary

Keep this PR open, but do not merge it as-is: the scoped abort direction is useful for Telegram polling restarts, yet the patch scopes the same abort controller used for full shutdown cleanup and would weaken shutdown abort behavior for in-flight replies.

## What This Changes

The PR adds a Telegram Bot API method allow-list for `fetchAbortSignal`, applies getUpdates-only aborting to polling sessions, and extends fetch-abort coverage for overlapping `sendMessage` calls.

## Best Possible Solution

Separate restart-only getUpdates aborts from full-session shutdown aborts, preserve default bot abort behavior, cover both paths with regression tests, and add the active-version changelog entry.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main wires one polling-session abort signal to all Telegram API fetches, while the PR scopes that same signal to getUpdates. I did not run tests in this read-only review.

## Solution Assessment

No. The proposed allow-list is the right narrow mechanism for restart recovery, but it is attached at the wrong lifetime boundary because shutdown and restart currently share the same controller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Keep shutdown aborts unscoped:** `extensions/telegram/src/polling-session.ts:190`
  - body: This allow-list is attached to the polling session's only `fetchAbortSignal`, but that same controller is also aborted by the outer session abort listener and final `stopRunner()` cleanup. After this change, a gateway shutdown with an in-flight `sendMessage` would no longer be aborted by the shutdown path and can sit until the outbound Telegram timeout instead of ending immediately. Please split restart-only aborts from full-shutdown aborts, or add a separate unscoped shutdown signal.
