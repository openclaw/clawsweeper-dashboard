## Summary

This PR should stay open but needs changes before merge: current main still lacks restart-only abort scoping, while the proposed patch scopes the single polling-session abort controller used for both restart recovery and full shutdown. It also lacks the required changelog entry for a user-facing Telegram delivery fix.

## What This Changes

The PR adds a Telegram bot fetch-abort method allow-list, wires polling sessions to abort only getUpdates during recovery, and adds regression coverage for overlapping sendMessage preservation.

## Best Possible Solution

Keep getUpdates-only aborts for polling-restart recovery, preserve unscoped aborts for real session shutdown, add the active-version changelog entry, and revalidate the Telegram changed lane.

## Reproduction Assessment

Yes for the PR-level regression: the PR passes fetchAbortMethods: ["getupdates"] to the same controller that stopRunner and the outer abortSignal abort, so an in-flight sendMessage during shutdown would no longer observe that shutdown abort. The broader related live report in #72867 still lacks a deterministic end-to-end reproduction, but the source-level path is clear.

## Solution Assessment

No. The method allow-list is a reasonable narrow seam for restart recovery, but applying it to the only polling-session abort controller conflates restart and shutdown semantics; a safer fix separates restart-only getUpdates aborts from full shutdown aborts.
