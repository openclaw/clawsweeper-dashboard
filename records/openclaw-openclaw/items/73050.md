## Summary

Keep open. The PR is aimed at a real Telegram reply-loss path and its allow-list mechanism is useful, but the current diff scopes the polling session's shared abort signal for shutdown as well as restart and still lacks the required changelog entry.

## What This Changes

The PR adds a Telegram Bot API method allow-list for `fetchAbortSignal`, applies getUpdates-only aborting in polling sessions, and adds fetch-abort regression coverage for overlapping `sendMessage` calls.

## Best Possible Solution

Separate restart-only getUpdates aborts from full-session shutdown aborts, preserve default bot abort behavior, cover both lifetimes with regression tests, and add the active-version changelog entry.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main wires one polling-session abort signal to all Telegram API fetches, while the PR scopes that same signal to getUpdates. I did not run tests in this read-only review.

## Solution Assessment

No. The allow-list is a reasonable narrow mechanism for restart recovery, but it is attached to the shared polling-session abort controller, so shutdown aborts become getUpdates-only too.

## Review Findings
