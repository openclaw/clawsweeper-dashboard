## Summary

Keep open: current main still forwards the polling restart abort signal to every wrapped Telegram API fetch, so the PR addresses a real remaining problem, but the submitted branch is stale against the current Telegram fetch wrapper and is missing the required changelog entry.

## What This Changes

The PR adds a Telegram bot fetch-abort method allow-list, scopes polling-session restart aborts to getUpdates, and adds regression coverage for overlapping sendMessage calls during polling recovery.

## Best Possible Solution

Land a rebased Telegram-owned fix that keeps fetchAbortMethods internal to TelegramBotOptions, applies the allow-list inside the current runFetch abort-forwarding path without weakening request timeouts or fallback retry behavior, passes getupdates from polling sessions, adds focused regression coverage, and records the user-facing fix in CHANGELOG.md.

## Reproduction Assessment

Yes for the PR's narrow bug: create a wrapped Telegram client with fetchAbortSignal, start overlapping getUpdates and sendMessage fetches, then abort the polling controller; current main forwards that abort to both requests because there is no method filter. The broader #72867 production report still needs separate live/session evidence to prove this is the only root cause.

## Solution Assessment

Yes in design, but no as submitted. Scoping polling restart aborts to getUpdates is the narrow maintainable solution, but the patch must be rebased into the current runFetch wrapper and include the required changelog entry before merge.
