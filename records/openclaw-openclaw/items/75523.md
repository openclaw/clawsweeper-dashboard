## Summary

Keep open: the PR is useful SSRF hardening, and the later dispatcher-override commit addresses part of the prior review, but PR head still lets `.internal` hostnames bypass the runtime resolver path when private-network access is enabled.

## What This Changes

The PR adds explicit cloud metadata host/IP checks and regression tests across SSRF classification, pinned DNS resolution, dispatcher overrides, guarded fetches, and the changelog.

## Best Possible Solution

Make hostname preflight, pinned DNS resolution, and dispatcher override validation share one always-enforced blocked-destination helper for metadata and `.internal` policy, then add resolver and guarded-fetch regressions for `foo.internal` under private-network opt-in.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` still passes because only exact metadata destinations are checked before the private-network skip.

## Solution Assessment

No. The metadata/IP hardening is the right direction, but the current implementation is not the narrowest complete fix because it lets the classifier contract and runtime resolver drift for `.internal` names.
