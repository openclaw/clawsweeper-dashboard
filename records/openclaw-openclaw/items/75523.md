## Summary

Keep open: the PR targets a real SSRF hardening gap, but the patch does not enforce the new metadata-host block on the guarded fetch and pinned DNS path because private-network opt-in still skips the shared hostname/IP checks.

## What This Changes

The PR adds a policy-aware cloud metadata hostname block to the SSRF hostname classifier and direct unit tests for private-network policy handling.

## Best Possible Solution

Land a revised patch that enforces always-blocked metadata destinations before any private-network skip in the shared SSRF policy gate, then cover the guarded fetch or pinned DNS path so future classifier-only changes cannot drift from runtime behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction is a unit test that calls `resolvePinnedHostnameWithPolicy("metadata.google.internal", { lookupFn, policy: { allowPrivateNetwork: true } })` with a mocked metadata/link-local DNS answer; source inspection shows the current and proposed paths skip the blocking phases.

## Solution Assessment

No. The direct classifier change is too low in the stack; the safer fix is to enforce metadata blocking in `resolveHostnamePolicyChecks` or an equivalent shared gate before `skipPrivateNetworkChecks` can bypass it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Enforce metadata blocking before private-network skips:** `src/infra/net/ssrf.ts:245-247`
  - body: The new always-blocked metadata set is only consulted by direct classifier calls. `resolveHostnamePolicyChecks` still skips that classifier when `allowPrivateNetwork` or `dangerouslyAllowPrivateNetwork` is true, so `fetchWithSsrFGuard` can still resolve and fetch `metadata.google.internal` through the main pinned-DNS path. Move this check into the shared policy gate and add coverage through `resolvePinnedHostnameWithPolicy` or guarded fetch.
