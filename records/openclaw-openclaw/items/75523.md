## Summary

Keep open: the PR moves the SSRF guard in the right direction, but PR head still leaves a `.internal` hostname-policy bypass when private-network access is enabled, so it is not ready to merge without a security fix/review.

## What This Changes

The PR adds metadata-host/IP blocking before SSRF private-network skips, dispatcher override checks, regression tests, and a changelog entry.

## Best Possible Solution

Make hostname preflight, pinned DNS resolution, and dispatcher override validation share one always-enforced blocked-destination helper for metadata and `.internal`, with targeted resolver and guarded-fetch regressions.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: publicLookup, policy: { allowPrivateNetwork: true } })` follows the private-network skip path and does not run the full blocked-host classifier.

## Solution Assessment

No. The metadata/IP hardening is the right direction, but the current implementation is incomplete because `.internal` is asserted in classifier tests while the runtime resolver can still skip that classifier.
