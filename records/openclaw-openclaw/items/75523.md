## Summary

Keep open: the PR is aiming at a real SSRF hardening gap, but the current patch only hardens the direct classifier path and does not protect the guarded fetch/pinned-DNS path that actually skips checks under private-network opt-in.

## What This Changes

The branch adds policy-aware metadata and loopback hostname handling in the SSRF classifier and direct classifier regression tests in `src/infra/net/ssrf.test.ts`.

## Best Possible Solution

Revise the existing branch to enforce always-blocked metadata destinations before any private-network skip in `resolveHostnamePolicyChecks` or an equivalent shared gate, then add regression coverage through `resolvePinnedHostnameWithPolicy` or guarded fetch while preserving ordinary LAN/self-hosted opt-in behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction is a unit test that calls `resolvePinnedHostnameWithPolicy("metadata.google.internal", { lookupFn, policy: { allowPrivateNetwork: true } })` with a mocked metadata/link-local DNS answer; source inspection shows the current and proposed paths skip both blocking phases.

## Solution Assessment

No. The classifier-only change is not the best fix because the runtime fetch path bypasses that classifier under private-network opt-in; the safer solution is to enforce the metadata deny rule in the shared policy gate before the skip is applied.
