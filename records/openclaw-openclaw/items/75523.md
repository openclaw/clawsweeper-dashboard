## Summary

Keep open: the latest PR head fixes the previously reported dispatcher override path for canonical cloud metadata IPs, but the patch still has a concrete SSRF policy gap. It adds tests/comments saying `.internal` hostnames stay blocked under private-network opt-in, while the actual guarded hostname path still skips the full hostname classifier whenever private-network access is allowed.

## What This Changes

The PR updates SSRF hostname/IP classification, pinned DNS and pinned dispatcher validation, regression tests, and the changelog so cloud metadata destinations remain blocked under private-network opt-in.

## Best Possible Solution

Revise the branch so the intended always-block hostname rules run before every private-network skip in the guarded hostname path, pinned DNS path, and dispatcher override path, with regression coverage for both exact metadata destinations and any generic `.internal` policy the maintainers want to enforce.

## Reproduction Assessment

Yes. A focused reproduction can call `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: publicLookup, policy: { allowPrivateNetwork: true } })` or `fetchWithSsrFGuard` with the same hostname on the PR head; static inspection shows it bypasses the full hostname classifier and proceeds unless DNS returns an exact metadata IP.

## Solution Assessment

No. The direction is right and the latest force-push closes the canonical metadata dispatcher override hole, but the proposed fix is not complete while its `.internal` blocking rule is only enforced by the standalone classifier and not by the guarded fetch/pinning policy path.
