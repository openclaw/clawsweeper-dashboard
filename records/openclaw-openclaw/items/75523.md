## Summary

Keep open: the PR moves the SSRF guard in the right direction for cloud metadata destinations, but the latest head still leaves the runtime resolver narrower than the hostname policy it documents and tests. A `.internal` hostname can still pass under private-network opt-in when DNS returns a public address, so this should not merge as-is.

## What This Changes

The PR adds always-blocked cloud metadata hostname/IP checks to SSRF classification, pinned DNS resolution, dispatcher override validation, regression tests, and the changelog.

## Best Possible Solution

Keep the metadata hardening, but make hostname preflight, pinned DNS resolution, and dispatcher override validation share one always-enforced blocked-destination helper so `.internal`, exact metadata hostnames, and metadata IP forms cannot drift under private-network opt-in. Add resolver and guarded-fetch regression coverage for `foo.internal` with `allowPrivateNetwork: true` and public DNS.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` still passes because the full hostname classifier is skipped and only exact metadata destinations are checked before the skip.

## Solution Assessment

No. The metadata/IP hardening is a good direction, but this implementation is incomplete because the runtime guard is narrower than the new classifier policy and tests.
