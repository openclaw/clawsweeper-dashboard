## Summary

Keep open: the PR addresses exact cloud metadata host/IP cases, but it still leaves `.internal` hostnames bypassing the full SSRF classifier when private-network access is enabled.

## What This Changes

The PR adds always-blocked cloud metadata host/IP checks to SSRF resolver and dispatcher paths, regression tests, and a changelog entry.

## Best Possible Solution

Make hostname preflight, pinned DNS resolution, and dispatcher override validation share one always-enforced blocked-destination helper for metadata and `.internal`, with resolver and guarded-fetch regression coverage.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: publicLookup, policy: { allowPrivateNetwork: true } })` follows the private-network skip path and avoids the classifier that the PR says should still block `.internal`.

## Solution Assessment

No. The metadata-specific guard is the right direction, but the proposed fix is incomplete because it does not enforce the broader internal-hostname policy in the runtime resolver path.

## Review Findings

Overall correctness: patch is incorrect
