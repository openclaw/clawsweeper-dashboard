## Summary

Keep open: the PR is useful hardening, but the branch still has an internal-host policy mismatch in the SSRF resolver path and needs maintainer/security review before merge.

## What This Changes

The PR adds SSRF metadata hostname/IP prechecks, dispatcher override validation, regression tests, and a changelog entry for private-network SSRF guard behavior.

## Best Possible Solution

Land a consistent SSRF policy that always blocks cloud metadata host/IPs under private-network opt-in and aligns .internal handling across classifier, resolver, dispatcher override checks, tests, and docs.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: publicLookup, policy: { allowPrivateNetwork: true } }) skips the full hostname classifier and can return pinned public addresses.

## Solution Assessment

No. The exact metadata/IP hardening is the right direction, but the branch should either enforce the same .internal policy before private-network skips or narrow the new classifier/tests to the intended metadata-only policy and update docs.
