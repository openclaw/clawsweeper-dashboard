## Summary

Keep open: this is active security-sensitive SSRF hardening, and the latest PR head still leaves the `.internal` hostname policy unenforced on the guarded pinned-resolution path when private-network access is enabled. The branch improves exact cloud metadata host/IP blocking, but it should not merge until the runtime path matches the classifier intent and CI is green.

## What This Changes

The PR updates SSRF hostname/IP classification and guarded fetch/pinned dispatcher checks to keep cloud metadata hosts and IPs blocked under private-network opt-in, with regression tests and a changelog entry.

## Best Possible Solution

Unify the always-blocked hostname policy so pre-DNS validation, pinned DNS resolution, guarded fetch, and dispatcher override validation enforce the same `.internal` and metadata protections before private-network skips, while preserving the intended localhost, `.localhost`, `.local`, and LAN allowances.

## Reproduction Assessment

Yes. On the PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` follows the exact-metadata precheck, performs DNS, skips the broader classifier, and returns a pinned host.

## Solution Assessment

No. The exact metadata host/IP hardening is the right direction, but the current patch is incomplete because the runtime guarded path and the classifier do not enforce the same `.internal` policy under private-network opt-in.
