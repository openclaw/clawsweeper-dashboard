## Summary

Keep open: the PR improves metadata SSRF hardening, but the current patch still lets `.internal` hostnames bypass resolver and guarded-fetch validation when private-network access is enabled, contrary to the PR's own classifier contract.

## What This Changes

The PR adds explicit cloud metadata hostname/IP checks to SSRF preflight, pinned DNS resolution, dispatcher override validation, regression tests, and the changelog.

## Best Possible Solution

Centralize an always-enforced blocked-destination helper for metadata and `.internal` policy, use it in hostname preflight, DNS-answer checks, and dispatcher overrides, then add resolver and guarded-fetch regressions for `.internal` under private-network opt-in.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` follows the private-network skip path after the exact metadata precheck and should resolve instead of throwing.

## Solution Assessment

No. The metadata/IP hardening is the right direction, but this patch is not complete because the classifier contract and runtime resolver still diverge for `.internal` hostnames.

## Review Findings
