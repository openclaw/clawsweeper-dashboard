## Summary

Keep open: the latest PR head fixes the exact cloud metadata host/IP and dispatcher-override cases, but it still leaves the `.internal` hostname boundary inconsistent under `allowPrivateNetwork`. The PR's own classifier tests say `.internal` should remain blocked, while the resolver path still skips the full hostname classifier after only checking exact metadata destinations.

## What This Changes

The PR adds always-block checks for cloud metadata hostnames and metadata IPs to SSRF classification, pinned DNS resolution, dispatcher override validation, regression tests, and the changelog.

## Best Possible Solution

Keep the metadata hardening, but make hostname preflight, pinned DNS resolution, and dispatcher override validation share one always-enforced blocked-destination helper so `.internal`, exact metadata hostnames, and metadata IP forms cannot drift under private-network opt-in. Add resolver/fetch regression coverage for `foo.internal` with `allowPrivateNetwork: true` and public DNS.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` still passes the preflight because only exact metadata destinations are checked before the private-network skip.

## Solution Assessment

No. The metadata/IP hardening is the right direction, but the current patch is incomplete because it narrows the pre-skip guard instead of enforcing the same blocked-host policy that the PR's classifier tests document.
