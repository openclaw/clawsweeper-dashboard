## Summary

Keep open: the SSRF hardening targets a real bypass and now includes direct runtime proof, but the diff still breaks documented strict `allowedHostnames` exceptions for non-metadata hosts such as `localhost`.

## What This Changes

The PR changes SSRF hostname/IP classification and pinned dispatcher validation so metadata and `.internal` destinations remain blocked under private-network opt-in, adds regression tests, and records a changelog entry.

## Best Possible Solution

Land the hardening after preserving exact non-metadata `allowedHostnames` exceptions, keeping metadata/`.internal` protections, and aligning focused regression coverage with the documented contract.

## Reproduction Assessment

Yes. Source inspection on current main shows the blocked host/IP assertion is skipped when `allowPrivateNetwork` or an exact `allowedHostnames` entry sets `skipPrivateNetworkChecks`; the PR body also provides a before/after node runner for the metadata and `.internal` cases.

## Solution Assessment

No. The hardening direction is right, but the current diff should separate always-blocked metadata/`.internal` policy from documented exact non-metadata exceptions such as `localhost`.
