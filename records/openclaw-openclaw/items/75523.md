## Summary

Keep open: the latest patch is useful but still leaves a security-sensitive pinned dispatcher override path unchecked when private-network access is enabled, so the PR should not merge yet.

## What This Changes

This PR modifies SSRF host/IP classification, pinned DNS validation, regression tests, and the changelog so cloud metadata hostnames and metadata IP DNS answers stay blocked under private-network opt-in.

## Best Possible Solution

Revise the branch so the always-blocked cloud metadata rule runs before every private-network skip path, including pinned dispatcher override records, then add focused dispatcher and guarded-fetch regression coverage for that route.

## Reproduction Assessment

Yes. A focused reproduction can call `fetchWithSsrFGuard` for a public URL with a public mocked DNS result, `policy: { allowPrivateNetwork: true }`, and `dispatcherPolicy.pinnedHostname.addresses` set to `['169.254.169.254']`; static inspection shows the override records still skip validation.

## Solution Assessment

No. The direction is right, but the current patch is not the narrowest safe fix because it applies the metadata rule to hostnames and DNS answers while leaving dispatcher override addresses outside the always-blocked check.

## Review Findings
