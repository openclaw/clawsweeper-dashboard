## Summary

Keep open: the latest branch addresses the prior `.internal` skip-path concern, but it introduces a blocking regression for documented strict-mode `allowedHostnames` exceptions and still only includes mocked/test proof for an external security-sensitive PR.

## What This Changes

The PR moves SSRF metadata and hostname checks ahead of private-network skips, adds pinned-DNS, dispatcher, and guarded-fetch regressions, and adds a changelog entry.

## Best Possible Solution

Keep metadata and `.internal` hardening in the shared SSRF path, preserve documented exact exceptions for non-metadata local hosts, add a regression for `{ allowedHostnames: ["localhost"] }`, and attach real runtime proof with private details redacted.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction with `resolvePinnedHostnameWithPolicy("localhost", { policy: { allowedHostnames: ["localhost"] }, lookupFn })`: current main skips the blocked-host assertion, while PR head checks the blocked-host classifier first.

## Solution Assessment

No. The hardening direction is right, but the patch should separate always-blocked metadata/internal destinations from documented exact `allowedHostnames` exceptions instead of overriding all blocked-host exceptions before the skip path.

## Review Findings
