## Summary

Keep open: the security-hardening direction is valid, but this PR is incomplete. It adds metadata blocking to the direct hostname/IP classifier, while the main guarded fetch and pinned-DNS path still skips both pre-DNS and resolved-address checks whenever private-network access is allowed.

## What This Changes

The PR adds an always-blocked cloud metadata hostname set, makes hostname blocking policy-aware so loopback and `.local` names are allowed under private-network opt-in while `.internal` stays blocked, and adds direct `isBlockedHostnameOrIp` regression tests.

## Best Possible Solution

Keep the PR open and move the always-blocked metadata decision into the shared policy gate used by guarded fetch and pinned DNS, then add regression coverage through `resolvePinnedHostnameWithPolicy` or `fetchWithSsrFGuard` so metadata hosts remain blocked even when local/private network endpoints are allowed.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resolvePinnedHostnameWithPolicy("metadata.google.internal", { lookupFn, policy: { allowPrivateNetwork: true } })` with a mocked `169.254.169.254` DNS result; the current and proposed code paths still skip both blocker phases when private-network access is allowed.

## Solution Assessment

No. The PR tests the direct classifier, but the narrow maintainable fix needs to enforce the metadata exception before `skipPrivateNetworkChecks` bypasses guarded fetch and resolved-address validation.
