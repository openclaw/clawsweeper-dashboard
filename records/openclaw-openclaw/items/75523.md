## Summary

Keep open: the PR is addressing a real SSRF hardening gap, but latest head still has a blocking policy mismatch. The classifier tests now say `.internal` hosts remain blocked with private-network opt-in, while the guarded runtime resolver only prechecks exact cloud metadata hosts/IPs before skipping the full hostname classifier.

## What This Changes

The PR changes SSRF host/IP classification, pinned DNS resolution, guarded fetch and dispatcher regression tests, plus the changelog so metadata destinations remain blocked when private-network access is enabled.

## Best Possible Solution

Keep the metadata hardening, but make pre-DNS checks, pinned DNS resolution, guarded fetch, and dispatcher override validation share one always-blocked destination helper so `.internal`, exact metadata hostnames, and metadata IP forms cannot drift under private-network opt-in.

## Reproduction Assessment

Yes. On PR head, call `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })`; source inspection shows it skips the full hostname classifier and can return a pinned host.

## Solution Assessment

No. The exact metadata host/IP hardening is the right direction, but the patch is incomplete because it adds a narrower precheck while its own classifier tests require `.internal` to remain blocked.
