## Summary

Keep open: the metadata hardening is useful, but the latest head remains incomplete because the runtime resolver still skips the broader blocked-host policy for `.internal` names under `allowPrivateNetwork`; this is a security-sensitive SSRF path and should be corrected before merge.

## What This Changes

The PR modifies SSRF host/IP classification and pinned DNS/dispatcher checks to keep cloud metadata hosts and IPs blocked under private-network opt-in, adds targeted regression tests, and updates the changelog.

## Best Possible Solution

Keep the metadata hardening, but update the PR so hostname preflight, pinned DNS resolution, and dispatcher override validation share one always-enforced blocked-destination helper for metadata hosts, metadata IP forms, and `.internal` names; add resolver and fetch regression coverage for `.internal` under private-network opt-in.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` follows a deterministic path that skips the full blocked-host classifier and returns a pinned hostname.

## Solution Assessment

No. The direction is right, but the patch is not the narrow complete fix because it enforces only a metadata subset before the private-network skip instead of the blocked-host policy that the PR's classifier tests document.
