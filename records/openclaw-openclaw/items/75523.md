## Summary

Keep open: this PR fixes part of the SSRF gap for exact cloud metadata hostnames and metadata IPs, but the latest head still leaves the broader `.internal` hostname policy bypassed in the pinned/guarded fetch path when private-network access is enabled.

## What This Changes

The PR updates `src/infra/net/ssrf.ts` to keep explicit cloud metadata hostnames and metadata IPs blocked under private-network opt-in, adds pinned DNS and dispatcher regression tests, and adds a changelog entry.

## Best Possible Solution

Revise the branch so `resolveHostnamePolicyChecks`, pinned DNS resolution, and dispatcher override validation enforce the same always-blocked hostname policy before private-network skips, with focused regression coverage for `.internal` through the pinned/fetch paths while preserving legitimate localhost, `.local`, and LAN use cases.

## Reproduction Assessment

Yes. On the PR head, a focused reproduction is `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })`; static inspection shows it passes the exact-metadata precheck, performs DNS, skips the broader classifier, and returns a pinned host.

## Solution Assessment

No. The exact metadata hostname/IP hardening is the right direction, but the proposed patch is incomplete because the runtime guarded path and the classifier no longer enforce the same `.internal` policy under private-network opt-in.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Apply the .internal block before private-network skips:** `src/infra/net/ssrf.ts:344-348`
  - body: The patch makes `isBlockedHostnameOrIp("foo.internal", { allowPrivateNetwork: true })` return true and adds classifier tests for that policy, but `resolveHostnamePolicyChecks` only checks exact cloud metadata host/IP values before honoring `skipPrivateNetworkChecks`. A guarded fetch to `foo.internal` with a public DNS answer can still resolve and proceed, leaving the stated `.internal` block unenforced.
