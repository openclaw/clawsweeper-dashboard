## Summary

Keep open: the latest head hardens exact cloud metadata host/IP cases, but it still leaves the claimed `.internal` hostname protection unenforced in the pinned-resolution path under private-network opt-in. Because this is an SSRF policy surface and exact-head checks are not clean yet, it needs maintainer network/security review before merge.

## What This Changes

The PR changes SSRF hostname/IP classification plus guarded fetch and pinned dispatcher checks to keep cloud metadata hosts/IPs blocked under private-network opt-in, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Define one always-blocked SSRF destination helper and use it consistently in pre-DNS validation, pinned DNS resolution, guarded fetch, and dispatcher override validation. If the intended scope is exact cloud metadata hosts/IPs only, remove the broad `.internal` expectation; if `.internal` must stay blocked, enforce it before private-network skips and update docs for the narrowed opt-in behavior.

## Reproduction Assessment

Yes. On the PR head, a focused unit can call `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })`; the code path performs only the exact-metadata precheck, skips the broader classifier, and returns a pinned host.

## Solution Assessment

No. The exact metadata host/IP hardening is the right direction, but the current patch is incomplete because the classifier and runtime guarded path enforce different hostname policies under private-network opt-in.
