## Summary

Keep open: the latest PR head fixes the exact cloud metadata host/IP cases, but it still leaves the advertised `.internal` hostname block unenforced on guarded fetches when private-network opt-in skips the full classifier.

## What This Changes

This PR updates SSRF hostname/IP classification, pinned resolution, guarded fetch, pinned dispatcher tests, and the changelog so cloud metadata destinations stay blocked when private-network access is enabled.

## Best Possible Solution

Keep the PR open and make the guarded runtime path enforce the same metadata and `.internal` boundary that the classifier promises, ideally through one shared always-blocked destination helper used by pre-DNS checks, pinned DNS resolution, guarded fetch, and dispatcher override validation.

## Reproduction Assessment

Yes. On PR head, call `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })`; the code marks private-network checks as skipped, bypasses the full hostname classifier, and accepts the public DNS answer.

## Solution Assessment

No. The exact metadata host/IP hardening is the right direction, but the current implementation is incomplete because it adds a narrower precheck while leaving the broader `.internal` classifier outside the private-network skip path.
