## Summary

Keep open: the PR fixes the exact cloud metadata host/IP cases it names, but the current head still leaves a private-network bypass for `.internal` hostnames in the runtime resolver path. That is a concrete security-sensitive blocker before merge.

## What This Changes

The PR adds SSRF metadata host/IP blocking logic, dispatcher override validation, regression tests, and a changelog entry for private-network opt-in behavior.

## Best Possible Solution

Keep the metadata hardening, but make hostname preflight, DNS-pinned resolution, and dispatcher override validation share an always-enforced blocked-destination policy so `.internal`, exact metadata hostnames, and metadata IP forms cannot drift under private-network opt-in.

## Reproduction Assessment

Yes. On PR head, `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` follows the private-network skip and avoids the full hostname classifier even though the PR's classifier tests say `.internal` remains blocked.

## Solution Assessment

No. The exact metadata hostname/IP checks are useful, but the patch is incomplete because it narrows the pre-skip runtime guard instead of enforcing the same blocked-host contract that the classifier exposes.
