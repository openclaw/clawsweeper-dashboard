## Summary

Keep open: this PR is a useful SSRF hardening change, but the latest head still leaves `.internal` hostnames allowed in the guarded/pinned path when private-network access is enabled, even though the new classifier logic and tests say those names should stay blocked.

## What This Changes

The PR updates SSRF hostname/IP classification, pinned DNS resolution, dispatcher override validation, regression tests, and the changelog so cloud metadata destinations remain blocked when private-network access is enabled.

## Best Possible Solution

Revise the branch so `resolveHostnamePolicyChecks`, pinned DNS, and dispatcher policy paths enforce the same always-block hostname policy before private-network skips; add pinned/fetch regression coverage for `.internal` while preserving the intended LAN, localhost, `.local`, and explicit metadata-IP behavior.

## Reproduction Assessment

Yes. A focused reproduction can call `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` on the PR head; static inspection shows it reaches DNS and skips the broader classifier.

## Solution Assessment

No. The direction is right for exact metadata hostnames, metadata IPs, and dispatcher overrides, but the proposed fix is incomplete because the runtime guarded path does not enforce the new `.internal` block before the private-network skip.
