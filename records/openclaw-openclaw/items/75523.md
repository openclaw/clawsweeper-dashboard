## Summary

Keep open: this is a useful security hardening PR, and the latest head fixes exact metadata hostname/IP and dispatcher-override cases, but it still leaves a concrete SSRF policy gap. The patch adds and tests a rule that `.internal` hostnames remain blocked under private-network opt-in, while the guarded pinned-hostname path still bypasses that classifier whenever `allowPrivateNetwork` is true.

## What This Changes

The PR updates SSRF hostname/IP classification, pinned hostname resolution, pinned dispatcher override validation, regression tests, and the changelog so cloud metadata destinations remain blocked when private-network access is allowed.

## Best Possible Solution

Revise this PR so the actual guarded fetch and pinned DNS policy paths enforce the same always-block hostname rules that the classifier and tests claim, especially `.internal`, while preserving the intended LAN/localhost private-network opt-in and the new metadata-IP checks.

## Reproduction Assessment

Yes. A focused reproduction can call `resolvePinnedHostnameWithPolicy("foo.internal", { lookupFn: async () => [{ address: "93.184.216.34", family: 4 }], policy: { allowPrivateNetwork: true } })` on the PR head; static inspection shows it reaches DNS and succeeds instead of throwing.

## Solution Assessment

No. The direction is right for exact metadata hostname/IP and dispatcher override blocking, but the proposed fix is incomplete while the new `.internal` policy is enforced only by the standalone classifier and not by the guarded hostname path used by fetch/pinning.
