## Summary

Keep open: current main and v2026.6.10 still keep Slack assistant thread status at a generic `is typing...` value during a run, while tool-start events only update reactions and draft/progress preview surfaces. The request is valid, but it should stay in maintainer/product review because direct per-tool status text overlaps the open Slack-safe progress visibility and status-customization decisions.

## What This Changes

The issue requests Slack assistant thread status updates that reflect the currently running tool instead of a static typing message.

## Best Possible Solution

Keep this issue as the canonical tool-level Slack assistant-status request, but require maintainers to first choose the Slack visibility/status contract and then implement the Slack plugin wiring with safe defaults, docs, tests, and live Slack proof.

## Reproduction Assessment

Yes, at source level. Current Slack dispatch sets assistant thread status to `is typing...` on reply start and clears it at the end, while `onToolStart` only updates reactions and progress previews; no live Slack workspace reproduction was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. The Slack plugin is the right owner boundary, but the implementation should be coordinated with Slack-safe visibility and custom status/loading-message decisions instead of adding another one-off status path.
