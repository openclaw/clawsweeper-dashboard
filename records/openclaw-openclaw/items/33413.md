## Summary

Keep open: current main still has Slack native assistant-thread status wired only to generic reply start/clear, while tool-start events drive reactions and draft-preview progress rather than `assistant.threads.setStatus`. The prior implementation PR at https://github.com/openclaw/openclaw/pull/33420 was closed unmerged, and the open customization request at https://github.com/openclaw/openclaw/issues/67129 tracks a separate config/API decision.

## What This Changes

Request Slack to update `assistant.threads.setStatus` with human-readable tool-specific progress labels during agent tool execution.

## Best Possible Solution

Add Slack-local native thread status updates from `onToolStart` with concise mapped labels, dedupe repeated labels, preserve stop/error cleanup, and keep custom status/loading-message configuration tracked separately at https://github.com/openclaw/openclaw/issues/67129.

## Reproduction Assessment

Yes, for source-level verification. Current main emits tool-start payloads and exposes the Slack status wrapper, but Slack dispatch only sends generic start/clear native status while tool starts update reactions or draft previews instead.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable direction because core already emits tool-start events and Slack already owns the native assistant-thread status wrapper.
