## Summary

Keep open: current main and the latest release still use Slack's native assistant-thread status only for static reply start/clear, while tool-start events update reactions and draft-preview progress but do not update `assistant.threads.setStatus`. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 confirms a viable Slack-local wiring shape, and the adjacent open config request at https://github.com/openclaw/openclaw/issues/67129 does not supersede this tool-level status behavior.

## What This Changes

The issue requests Slack native assistant-thread status updates that show the currently running tool instead of a static typing status.

## Best Possible Solution

Implement Slack-local native thread status updates from the existing `onToolStart` callback, reusing the established tool-display/progress vocabulary where practical, deduping repeated labels, and preserving the existing start/clear lifecycle without adding a public config option for the basic behavior.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack status wrapper, but the Slack `onToolStart` handler never calls `ctx.setSlackThreadStatus` with a tool label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new core hook or custom status config is not needed for the basic requested behavior.
