## Summary

Keep open: current main still only starts and clears Slack native assistant-thread status with static text; tool-start events update reactions and draft-preview progress but do not update `assistant.threads.setStatus` with tool labels. The merged Slack assistant lifecycle work and the open custom-status config request are adjacent, but neither owns this exact native tool-status gap.

## What This Changes

Request to show Slack assistant-thread status text that reflects the currently running tool during agent execution.

## Best Possible Solution

Add Slack-local native thread status updates from the existing `onToolStart` callback using the shared tool-display/progress-label helpers where practical, with stale-status cleanup, repeated-label dedupe, focused tests, docs/changelog coverage, and no new public config unless maintainers deliberately choose that in https://github.com/openclaw/openclaw/issues/67129.

## Reproduction Assessment

Yes, at source level: current main emits tool-start payloads and has the Slack status wrapper, but Slack dispatch does not call that wrapper from `onToolStart`. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable solution because core already emits tool events and the Slack plugin owns `assistant.threads.setStatus`. The implementation should avoid expanding plugin hooks or adding config for this narrow status-label request.
