## Summary

Keep open: current main and the latest release still use a static Slack native assistant-thread status, while tool-start events only drive reactions and draft-preview progress. The discussion and closed unmerged prototype make this a narrow Slack plugin feature gap rather than a cleanup close.

## What This Changes

Request Slack to update `assistant.threads.setStatus` with human-readable tool labels during agent tool execution instead of leaving the thread status static.

## Best Possible Solution

Implement Slack-local native thread status updates from the existing `onToolStart` callback with compact labels, repeated-label dedupe, and safe cleanup; leave custom status/loading-message configuration to the separate Slack status config discussion.

## Reproduction Assessment

Yes, at source level: current main emits tool-start payloads and has the Slack status wrapper, but Slack dispatch does not call that wrapper from `onToolStart` with tool-level labels. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable direction because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new core hook or config option is not needed for the basic requested behavior.
