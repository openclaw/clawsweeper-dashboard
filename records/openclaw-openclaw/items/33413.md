## Summary

Keep open: current main still sets Slack native assistant-thread status only to the generic start/clear text, while tool-start events update reactions and draft-preview progress but do not call the native status wrapper with tool-level labels. The prior implementation PR was closed unmerged, and the related open issues cover adjacent customization and broader assistant lifecycle work rather than this narrow native-status gap.

## What This Changes

Request Slack assistant-thread status text to change dynamically during tool execution with human-readable tool progress labels.

## Best Possible Solution

Add Slack-local native thread status updates from the existing `onToolStart` callback with concise label mapping, repeated-label dedupe, safe cleanup, and focused tests; leave custom status configuration to https://github.com/openclaw/openclaw/issues/67129 and broader assistant lifecycle work to https://github.com/openclaw/openclaw/issues/80787.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current main emits tool-start payloads and has the Slack status wrapper, but Slack dispatch does not call that wrapper from `onToolStart`.

## Solution Assessment

Is this the best way to solve the issue? Yes: Slack-local callback wiring is the narrowest maintainable path because core already emits the tool events and Slack owns the native assistant-thread status call.
