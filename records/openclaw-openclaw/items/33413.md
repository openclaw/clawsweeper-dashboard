## Summary

Keep open. Current main still sets Slack native assistant thread status only to the static start/clear values, while tool-start events update status reactions and draft-preview progress rather than `assistant.threads.setStatus`. The direct implementation PR #33420 was closed unmerged, and the adjacent #67129 and #73054 items cover customization/preview behavior rather than this native per-tool status request.

## What This Changes

The issue requests Slack-native assistant thread status updates that show human-readable tool-level progress during agent execution.

## Best Possible Solution

Implement a Slack-owned native status bridge: map known tool names to short human labels, update `ctx.setSlackThreadStatus` on tool-start events only when a status thread target is active, dedupe repeated labels, keep the existing static start and clear behavior, and document/test the native assistant status sequence separately from the customization and draft-preview discussions.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by exercising Slack dispatch with a reply start and an `onToolStart({ name: "exec" })` event: current main records only the static start and clear `setStatus` calls, and the tool-start callback never calls `ctx.setSlackThreadStatus`.

## Solution Assessment

Yes. The narrow maintainable fix is Slack-local wiring because core already emits the tool event and Slack already owns the native status wrapper; a broader plugin hook or global core status API is not needed for this single-channel UX request.
