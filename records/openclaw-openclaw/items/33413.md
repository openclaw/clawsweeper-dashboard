## Summary

Keep open: current main still uses Slack native assistant thread status only for static reply start/clear, while tool-start events update reactions and draft-preview progress rather than `assistant.threads.setStatus`; the direct implementation PR was closed unmerged and related items do not supersede this native per-tool status request.

## What This Changes

Request Slack native assistant thread status to show per-tool progress labels during agent tool execution instead of a static typing message.

## Best Possible Solution

Implement a Slack-owned native status bridge that maps known tool names to short labels, updates `ctx.setSlackThreadStatus` on tool-start events when a status thread target exists, dedupes repeated labels, preserves existing start/clear behavior, and adds focused Slack tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: exercise Slack dispatch with a reply start and `onToolStart({ name: "exec" })`; current main records the static start/clear native status calls while the tool-start path only updates reactions and draft preview progress.

## Solution Assessment

Yes. The narrow maintainable fix is Slack-local wiring because core already emits the tool event and Slack already owns the native status wrapper; a broader plugin hook or global core status API is not needed for this request.
