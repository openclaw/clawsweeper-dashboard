## Summary

Keep open: current main still uses Slack native assistant thread status only for reply start/clear, while tool-start callbacks update reactions and draft-preview progress instead of `assistant.threads.setStatus`; the direct implementation PR was closed unmerged and the related preview PR does not supersede native per-tool status.

## What This Changes

Request Slack to update `assistant.threads.setStatus` with tool-specific progress labels during agent tool execution.

## Best Possible Solution

Add a Slack-owned native status bridge that maps known tool names to short labels, updates `ctx.setSlackThreadStatus` on tool-start events when a status thread target exists, dedupes repeated labels, preserves existing start/clear behavior, and covers it with focused Slack tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is clear: exercise Slack dispatch with a reply start and then `onToolStart({ name: "exec" })`; current main has the static start/clear status path but routes tool start only to reactions and draft preview progress.

## Solution Assessment

Yes. The narrow maintainable solution is Slack-local wiring because core already emits tool-start events and Slack already owns the native status wrapper; a broader plugin hook or global status API is not needed for this request.
