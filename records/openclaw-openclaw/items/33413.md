## Summary

Keep open: current main still starts Slack native assistant-thread status with the generic typing text and clears it at the end, while the existing tool-start handler updates status reactions and preview progress only. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 is useful implementation context but was not merged, and related Slack status/customization items do not replace this specific tool-level native status request.

## What This Changes

Add Slack native assistant-thread status updates that reflect the currently running tool during agent execution.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established channel progress/tool labels where practical, dedupe repeated labels, guard by the current typing lifecycle, and add focused Slack dispatch coverage.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call; a new public config option is not needed for the basic requested behavior.
