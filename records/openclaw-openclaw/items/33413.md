## Summary

Keep open: current main and the latest release still set Slack native assistant-thread status to the static `is typing...` text, while tool-start handling only updates reactions and draft progress. The earlier direct implementation at https://github.com/openclaw/openclaw/pull/33420 was closed unmerged, and the open native progress/loading-message PRs are adjacent but do not own this exact status-label gap.

## What This Changes

The issue requests Slack assistant-thread status updates that show the currently running tool instead of a static typing message.

## Best Possible Solution

Slack should update native assistant-thread status from existing tool-start events using shared tool display/progress labels, while preserving the current typing clear lifecycle and draft/native progress behavior.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable direction because the agent loop already emits tool events and Slack owns the native assistant-thread status API call.
