## Summary

The remaining work is still valid: current main and the latest release expose Slack `assistant.threads.setStatus`, but Slack only sets the native thread status to `is typing...` at reply start and clears it at the end; tool-start events update reactions and draft-preview progress without updating the native assistant-thread status. The related closed prototype confirms a narrow Slack-local implementation shape, while the open custom-status config request is adjacent rather than a superseding tracker.

## What This Changes

The issue requests Slack assistant-thread status updates that show the currently running tool instead of only a static typing status.

## Best Possible Solution

Implement Slack-local native thread status updates from the existing `onToolStart` callback, reusing established tool-display/progress labels where practical, deduping repeated labels, and preserving the current start/clear lifecycle without adding a public config option for the basic behavior.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but the Slack `onToolStart` handler never updates `assistant.threads.setStatus` with a tool label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new core hook or custom status config is not needed for the basic requested behavior.
