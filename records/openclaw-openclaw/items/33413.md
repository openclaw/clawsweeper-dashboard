## Summary

Keep open: current main and the latest release still set Slack native assistant-thread status to the generic typing text, while tool-start handling only updates reactions and preview progress. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 is useful implementation context but was not merged, and the other related Slack status items are adjacent rather than canonical replacements.

## What This Changes

The issue requests Slack `assistant.threads.setStatus` text to change per tool call instead of staying on a generic typing indicator.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established channel progress labels where practical, dedupe repeated labels, guard by the current typing lifecycle, and add focused Slack dispatch coverage.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call; a new public config option is not needed for the basic requested behavior.
