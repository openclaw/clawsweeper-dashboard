## Summary

Keep open: current main and the latest release still start Slack native assistant-thread status with the generic typing text, while the tool-start path updates reactions and draft-preview progress only. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 is useful context but was not merged, and active adjacent Slack status/progress PRs do not replace this specific native status-label request.

## What This Changes

Request Slack to update assistant-thread status with human-readable per-tool labels during agent runs instead of only setting a generic typing status.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established compact progress labels where practical, dedupe repeated labels, clear through the existing typing lifecycle, and add focused Slack dispatch coverage.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call; a new public config option is not required for the basic requested behavior.
