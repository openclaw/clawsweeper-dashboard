## Summary

Keep open: current main and the latest release still only set Slack native assistant-thread status to the static typing text at reply start and clear it at the end; tool-start handling updates reactions and draft progress, but not `assistant.threads.setStatus`.

## What This Changes

The issue requests Slack native assistant-thread status updates that show which tool is currently running.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established progress/tool display labels where practical, and guard the status lifecycle so suppressed-typing turns do not leave stale Slack status text.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new public config option or core hook is not needed for the basic requested behavior.
