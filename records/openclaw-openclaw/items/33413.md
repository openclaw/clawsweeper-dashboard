## Summary

Keep open: current `main` still has Slack native assistant-thread status lifecycle plumbing and tool-start callbacks, but the Slack tool-start path does not update `assistant.threads.setStatus` with tool-specific text. Related closed prototypes and progress-card work are useful context, but they do not implement this requested native status behavior on `main`.

## What This Changes

Request Slack native assistant thread status text to update per tool call instead of staying on the generic typing status during agent execution.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established progress/tool display labels where practical, dedupe repeated status writes, and preserve the current start/clear lifecycle.

## Reproduction Assessment

Yes, at source level: current `main` emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new public config option or core hook is not needed for the basic requested behavior.
