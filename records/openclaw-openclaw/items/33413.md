## Summary

Keep open: current main still has the Slack-native assistant thread status limited to generic reply start/clear, while tool-start events only drive status reactions and draft preview progress. The earlier implementation attempt at https://github.com/openclaw/openclaw/pull/33420 closed unmerged, so the requested native Slack status behavior remains actionable.

## What This Changes

Show Slack assistant-thread status text for the currently running tool instead of leaving the native status at the generic typing message.

## Best Possible Solution

Add Slack-local native thread status updates from `onToolStart` with concise mapped labels, dedupe repeated labels, preserve cleanup on stop/error, and cover the behavior with focused Slack tests plus docs/changelog updates.

## Reproduction Assessment

Yes, for source-level verification. Current main emits tool-start payloads and has a Slack native status wrapper, but Slack dispatch only sends the generic start/clear status while tool starts update reactions or draft previews instead.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable direction because core already emits tool-start events and Slack already owns the native assistant-thread status wrapper; keep the separate custom status configuration request at https://github.com/openclaw/openclaw/issues/67129 out of this fix.
