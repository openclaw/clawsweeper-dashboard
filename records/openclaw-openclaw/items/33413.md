## Summary

Keep open: current main still sends Slack native assistant-thread status only at reply start and clear. Tool-start events update status reactions and draft-preview progress, but they do not update `assistant.threads.setStatus`; the prior implementation PR was closed unmerged, while related open items cover customization and broader Slack assistant lifecycle work.

## What This Changes

Request tool-specific Slack assistant-thread status text during agent tool execution instead of only a generic typing status.

## Best Possible Solution

Add Slack-local native thread status updates from the existing `onToolStart` callback with concise mapped labels, repeated-label dedupe, focused tests, and no new custom status config because that is tracked separately.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, for source-level verification: current main emits tool-start payloads and has the Slack status wrapper, but Slack dispatch does not call that wrapper from `onToolStart`.

## Solution Assessment

Is this the best way to solve the issue? Yes: Slack-local callback wiring is the narrowest maintainable path because core already emits tool-start events and Slack already owns the native assistant-thread status wrapper.
