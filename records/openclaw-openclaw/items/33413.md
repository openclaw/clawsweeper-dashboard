## Summary

Keep open: current main and the latest release still set Slack native assistant-thread status only to the generic start/clear text, while tool-start progress updates reactions and draft/streaming progress instead of `assistant.threads.setStatus`. The closed prototype and newer Slack progress-card PRs are useful context, but none implements this requested default native status behavior on main.

## What This Changes

Request Slack to update native assistant-thread status text during tool execution so users can see which tool is currently running.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established progress/tool display labels where practical, dedupe repeated status writes, and preserve the current start/clear lifecycle so suppressed or failed turns cannot leave stale status text.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new public config option or core hook is not needed for the basic requested behavior.
