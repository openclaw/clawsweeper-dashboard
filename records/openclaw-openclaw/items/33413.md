## Summary

Keep open: current main still exposes the core tool-start callback and Slack native status wrapper, but Slack dispatch only writes native assistant-thread status for generic reply start/clear while tool starts update reactions or draft previews instead. The linked implementation PR was closed unmerged, and its review identified a stale-status edge case that a replacement fix should handle.

## What This Changes

Request Slack to update `assistant.threads.setStatus` with tool-specific progress labels during agent tool execution instead of showing only the generic typing status.

## Best Possible Solution

Add Slack-local native thread status updates from `onToolStart` with concise mapped labels, dedupe repeated labels, preserve cleanup even when typing start is suppressed, and cover it with focused Slack tests plus docs/changelog updates.

## Reproduction Assessment

Yes, for source-level verification. In a Slack reply thread, current main calls `opts.onToolStart` for tool events, but Slack's handler updates only reactions and draft-preview progress while native `setSlackThreadStatus` remains limited to generic start/clear calls.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable direction because core already emits tool-start events and Slack already owns the native status wrapper; keep the separate custom-status config request at https://github.com/openclaw/openclaw/issues/67129 out of this fix.
