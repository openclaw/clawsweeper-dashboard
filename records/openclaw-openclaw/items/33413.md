## Summary

Keep open: current main still emits tool-start events and updates Slack reactions/draft-preview progress, but native Slack `assistant.threads.setStatus` remains the static reply start/clear indicator.

## What This Changes

Request Slack assistant thread status text to change per tool execution with human-readable progress labels.

## Best Possible Solution

Add a Slack-owned native status bridge that maps known tool names to concise labels, calls `ctx.setSlackThreadStatus` on tool-start events when a status thread exists, preserves start/clear cleanup, and covers the behavior with focused Slack tests and docs/changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is clear: exercise Slack dispatch through reply start and `onToolStart({ name: "exec" })`; current main only calls native Slack status for static start/clear while tool starts go to reactions and draft preview progress.

## Solution Assessment

Yes. The narrowest maintainable solution is Slack-local wiring because core already exposes `onToolStart` and Slack already owns `setSlackThreadStatus`; a broader plugin hook or global status API is unnecessary for this request.
