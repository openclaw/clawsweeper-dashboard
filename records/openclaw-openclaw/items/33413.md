## Summary

Keep open: current main still uses Slack native assistant-thread status only for generic typing start/clear, while tool-start handling updates status reactions and preview progress without updating `assistant.threads.setStatus`. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 is useful context but was not merged, and related Slack loading-message/native-card work does not replace this exact status-text request.

## What This Changes

Request Slack assistant thread status text update per tool with human-readable labels during agent execution instead of the static typing text.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established tool display/progress labels where practical, dedupe repeated labels, preserve the typing clear lifecycle, and add focused Slack dispatch coverage.

## Reproduction Assessment

Yes, at source level: current main emits tool-start callbacks and has the Slack native status wrapper, but Slack dispatch never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call; a new public config option is not needed for the basic requested behavior.
