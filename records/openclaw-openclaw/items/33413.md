## Summary

Current main still exposes only a coarse Slack assistant status while richer tool progress travels through a separate pipeline. The closed implementation PR did not merge, so the requested capability remains useful and actionable.

## What This Changes

Add tool-specific progress text to Slack assistant thread status while an agent executes tools.

## Best Possible Solution

Implement a Slack-owned assistant-status controller that consumes existing normalized tool-progress events, deduplicates updates, preserves current typing and preview behavior, and clears status on every terminal path.

## Reproduction Assessment

Yes, from source inspection. Current main sets one static status at run start and clears it at stop even though the reply pipeline receives tool start and update events.

## Solution Assessment

The requested outcome is appropriate, but the original hard-coded tool map is not the best current implementation; Slack should reuse normalized progress presentation so its status and progress drafts cannot drift.
