## Summary

Keep open: current main still starts Slack native assistant-thread status with the static `is typing...` text and clears it at completion, while tool-start events only update Slack reactions and draft-preview progress. The closed unmerged implementation PR confirms the narrow wiring shape, but no merged code or canonical open item owns this exact native tool-status label gap.

## What This Changes

Request Slack native assistant-thread status labels that reflect the currently running tool during an agent turn.

## Best Possible Solution

Add Slack-local native thread status updates from the existing `onToolStart` callback with concise label mapping, repeated-label dedupe, safe cleanup, and focused Slack dispatch tests; keep custom status configuration separate from this issue.

## Reproduction Assessment

Yes, at source level: current main emits tool-start payloads and has the Slack status wrapper, but Slack dispatch does not call that wrapper from `onToolStart` with tool-level labels. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable direction because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new config option or core plugin hook is not needed for the basic requested behavior.
