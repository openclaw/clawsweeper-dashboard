## Summary

Keep this open. Current main still does not update Slack native assistant thread status with tool-level labels: the native status path remains static start/clear, while tool-start events feed reactions and draft-preview progress only. The earlier implementation PR was closed unmerged, and the open related Slack PRs cover configurable/persona status text or preview-stream behavior rather than this native per-tool status request.

## What This Changes

Show human-readable tool-level progress in Slack `assistant.threads.setStatus` during agent tool execution instead of only the static typing status.

## Best Possible Solution

Implement this as a narrow Slack-owned behavior: map tool names to human-readable labels in Slack dispatch, update native assistant thread status only while a thread status is active, preserve existing start/clear semantics, dedupe repeated labels, keep the active tool label stable through typing keepalive, and document the native status behavior separately from preview-stream and persona customization work.

## Risks / Open Questions

- A naive fix can spam Slack `assistant.threads.setStatus` because the agent emits both tool `start` and repeated `update` phases; unchanged labels should be deduped or phase-gated.
- Typing keepalive currently reuses the same start callback that sends `is typing...`, so a replacement fix must preserve the active tool label across keepalive ticks instead of briefly flashing it.
- This read-only review did not exercise a live Slack workspace; a fix should still include focused unit coverage for API call sequencing and may warrant a maintainer live smoke for final UX.
