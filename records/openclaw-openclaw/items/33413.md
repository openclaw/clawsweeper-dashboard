## Summary

Keep open: current main and v2026.5.12 still use Slack's native assistant-thread status only for a static start/clear typing indicator, while existing tool-start events feed reactions and draft-preview progress but not `assistant.threads.setStatus`. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 confirms the intended wiring shape, but it was not merged and had a cleanup edge case to avoid.

## What This Changes

Request Slack native assistant-thread status text to update with a human-readable label for the currently running tool instead of staying at `is typing...`.

## Best Possible Solution

Implement Slack-local native thread status updates from the existing `onToolStart` callback with compact labels, repeated-label dedupe, and reliable cleanup, while leaving custom status/loading-message configuration to the separate Slack config discussion.

## Reproduction Assessment

Yes, at source level: current main emits tool-start payloads and has the Slack status wrapper, but Slack dispatch does not call that wrapper from `onToolStart` with tool-level labels. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: Slack-local callback wiring is the narrowest maintainable direction because the agent loop already emits tool events and Slack owns the native assistant-thread status API call. A new core hook or custom config option is not needed for the basic requested behavior.
