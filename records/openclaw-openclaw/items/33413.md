## Summary

Keep open: current main still only writes Slack native assistant-thread status as the generic start/clear text, while tool-start events update reactions and draft-preview progress without updating `assistant.threads.setStatus` with tool labels. The closed prior implementation PR confirms the narrow wiring idea, but it was not merged and no canonical open item fully owns this exact native-status gap.

## What This Changes

Request dynamic Slack `assistant.threads.setStatus` text that reflects the currently running tool during an agent turn.

## Best Possible Solution

Add Slack-local native thread status updates from the existing `onToolStart` callback with concise label mapping, repeated-label dedupe, safe cleanup, and focused Slack dispatch tests; leave custom status configuration to https://github.com/openclaw/openclaw/issues/67129 and broader assistant lifecycle work to https://github.com/openclaw/openclaw/issues/80787.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current main emits tool-start payloads and has the Slack status wrapper, but Slack dispatch does not call that wrapper from `onToolStart`.

## Solution Assessment

Is this the best way to solve the issue? Yes: Slack-local callback wiring is the narrowest maintainable path because core already emits tool events and Slack owns the native assistant-thread status call.
