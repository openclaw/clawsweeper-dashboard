## Summary

Keep open: current `main` and latest release still set Slack native assistant-thread status to the generic `is typing...` text and clear it at the end, while tool-start handling only updates reactions and preview progress. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 is useful implementation context, but it was not merged; the active native progress-card work is adjacent and does not implement this requested `assistant.threads.setStatus` behavior on `main`.

## What This Changes

The issue requests Slack assistant-thread status text that reflects the currently running tool instead of a static typing message.

## Best Possible Solution

Implement Slack-local native assistant status updates from the existing `onToolStart` callback, reuse established tool display labels where practical, dedupe repeated status writes, and preserve the current start/clear lifecycle.

## Reproduction Assessment

Yes, at source level: current `main` emits tool-start callbacks and has the Slack native status wrapper, but the Slack tool-start handler never calls that wrapper with a tool-specific label. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes. Slack-local callback wiring is the narrowest maintainable path because the agent loop already emits tool events and Slack owns the native assistant-thread status API call; a new public config option or core hook is not needed for the basic requested behavior.
