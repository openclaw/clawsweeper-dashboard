## Summary

Keep open: the PR addresses a real blocked-plugin diagnostic gap, but the validation change currently downgrades existing fatal missing-plugin paths for `plugins.deny` and `plugins.slots.memory` to warnings before merge.

## What This Changes

The PR propagates blocked plugin IDs into discovery diagnostics, deduplicates suspicious-ownership diagnostics, changes config warning text for ownership-blocked plugins, adds discovery tests, and adds a changelog entry.

## Best Possible Solution

Keep the blocked-present diagnostic and dedupe approach, but preserve existing issue-versus-warning severity per config surface, add config validation coverage and public ownership guidance, and leave container ownership policy to a separate maintainer-reviewed change.

## Reproduction Assessment

Yes. A non-bundled plugin path owned by a uid that is neither the gateway process uid nor root is deterministically blocked in discovery, and current config validation then exercises the missing-plugin path for explicit `plugins.entries`, allow, deny, and slot references.

## Solution Assessment

No. The overall direction is narrow and maintainable, but the PR should preserve fatal validation severity for deny/slot references and tighten the broader issue-closing/docs scope before merge.
