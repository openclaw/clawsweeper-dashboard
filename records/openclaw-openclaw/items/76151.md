## Summary

Keep open: the PR fixes the right diagnostic gap, but it currently turns ownership-blocked deny and memory-slot references into warnings and would close the broader linked ownership issue before the docs/container-durability parts are resolved.

## What This Changes

The branch propagates plugin IDs into suspicious-ownership discovery diagnostics, deduplicates those diagnostics, rephrases missing-plugin config warnings for ownership-blocked plugins, adds discovery tests, and adds a changelog entry.

## Best Possible Solution

Keep the diagnostic propagation and dedupe approach, preserve the existing issue-versus-warning contract, add config validation coverage and public ownership guidance, and make the linked issue reference partial unless the container/docs remainder is handled or split.

## Reproduction Assessment

Yes. A non-bundled plugin path with an ownership UID that differs from the gateway UID and root deterministically enters the suspicious-ownership block, and config validation then exercises the missing-plugin paths for entries, allow, deny, and memory slots.

## Solution Assessment

No. The direction is maintainable, but the PR should preserve fatal validation severity for non-warn-only callers and avoid claiming all of #76144 is fixed before merge.
