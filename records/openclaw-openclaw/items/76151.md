## Summary

Keep open: the PR addresses the ownership-blocked plugin diagnostics reported in #76144, but its new diagnostic dedupe key is too broad and can hide distinct diagnostics that share message text but have different sources.

## What This Changes

The PR propagates plugin id hints into ownership-blocked discovery diagnostics, rewrites missing-plugin config messages for ownership-blocked plugins, deduplicates repeated discovery diagnostics, and adds discovery tests plus a changelog entry.

## Best Possible Solution

Land the ownership-diagnostics fix after making discovery diagnostic dedupe source-aware, while leaving #76144 open for the remaining docs and container ownership durability work.

## Reproduction Assessment

Yes. Current main gives a deterministic source-level reproduction path: a UID mismatch creates a suspicious-ownership diagnostic without `pluginId`, scoped/shared discovery can append duplicate diagnostics, and configured refs then flow into the stale-config warning path.

## Solution Assessment

No as submitted. The ownership diagnostic bridge is the narrow maintainable direction, but the dedupe should include `source` or be limited to ownership-blocked diagnostics so unrelated diagnostics are not hidden.
