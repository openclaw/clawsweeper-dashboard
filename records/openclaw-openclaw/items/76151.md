## Summary

Keep open: the PR addresses a real plugin diagnostics problem, but it currently weakens validation severity for ownership-blocked deny and explicit memory-slot references and overstates the linked issue as fully fixed.

## What This Changes

The PR propagates plugin IDs into suspicious-ownership diagnostics, deduplicates those diagnostics, changes missing-plugin warnings for ownership-blocked plugins, adds discovery tests, and adds a changelog entry.

## Best Possible Solution

Keep the diagnostic propagation and dedupe approach, preserve the existing issue-versus-warning contract, add config validation coverage and public ownership guidance, and narrow the linked issue claim unless the container/docs remainder is handled or split.

## Reproduction Assessment

Yes. A non-bundled plugin path with an ownership UID mismatch deterministically enters the suspicious-ownership block, and config validation can then exercise entries, allow, deny, and memory-slot refs for the blocked plugin ID.

## Solution Assessment

No. The direction is maintainable, but the implementation should preserve fatal validation severity for non-warn-only callers and avoid claiming the broader ownership issue is fully resolved.
