## Label Justifications

- `impact:crash-loop`: The reported failure mode is unbounded heap growth leading to repeated gateway process termination.

## Summary

Keep open; current main and the latest release still route Matrix E2EE through fake-indexeddb 6.2.5, and upstream fake-indexeddb source still retains finished transactions in the backing Database array. The report is recent, source-backed, and distinct from the broader non-Matrix heap issue.

## What This Changes

The issue asks OpenClaw to stop Matrix E2EE deployments from accumulating finished fake-indexeddb transactions until the gateway OOMs.

## Best Possible Solution

Fix the Matrix plugin's E2EE storage path so completed IndexedDB transactions are not retained in long-running gateway processes, with focused regression coverage and no unrelated gateway state-storage changes.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the issue includes an isolated transaction loop, and current main still pins and imports the same fake-indexeddb version for Matrix E2EE. I did not execute the long-running loop in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear which implementation shape is best, but fixing retained finished transactions is the right problem. A narrow dependency-backed repair or upstream-compatible shim is safer than broad Matrix storage rewrites unless maintainers choose a production IndexedDB replacement.
