## Summary

Keep open. Current main accepts and deduplicates Feishu card-action callbacks, including the adjacent Schema 2.0 parser path, but it still does not return an immediate WebSocket callback payload that disables or marks the clicked button as processing. The related open PR is parser-focused and does not implement the requested instant callback response.

## What This Changes

Request immediate Feishu card-action WebSocket callback responses that disable or mark clicked Schema 2.0 card buttons as processing while agent handling continues asynchronously.

## Best Possible Solution

Implement a Feishu-plugin-owned instant response path: claim and validate the callback token first, launch eligible quick/approval-confirm command dispatch in the background with existing completion/release semantics preserved, and return an SDK-compatible transitional card or toast. If a ds field is added to ocf1 values, keep it optional and backward-compatible with focused Feishu tests and docs/changelog updates.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered card.action.trigger handler with a valid structured quick-action event and observe that it resolves without returning any callback response while the current action path awaits synthetic command dispatch.

## Solution Assessment

Yes, the proposed handler-return direction is the right narrow boundary. The safer implementation is to keep this inside the Feishu plugin, preserve replay-token dedupe and authorization checks, and avoid any core plugin API change unless multiple channels need the same seam.
