- `impact:message-loss`: The report is about repeated Feishu card button callbacks that can duplicate user actions before the agent finishes processing.

## Summary

Keep open. Current `main` still routes Feishu card-action callbacks through the async command path without returning an immediate WebSocket callback response, and the related in-place card-update PRs remain open rather than providing a merged canonical disable/update-state contract.

## What This Changes

Request Feishu card-action WebSocket handlers return an instant transitional card update, ideally controlled by an `ocf1` disable-state field, so clicked buttons become disabled before agent processing completes.

## Best Possible Solution

Implement the instant response in the Feishu plugin as a typed card-action feedback path, coordinate it with the open card-update PRs, and add both source-level regression coverage and redacted live Feishu proof of a real button disabling immediately.

## Reproduction Assessment

Yes, at source level. A valid Feishu `card.action.trigger` payload reaches the current handler, but current `main` resolves through `Promise<void>` command dispatch with no returned WebSocket callback payload or disable-state contract.

## Solution Assessment

Yes on product direction, but the implementation path still needs maintainer coordination. The narrow maintainable solution is a Feishu-plugin-owned typed callback response that reuses the existing SDK return-value plumbing while defining one canonical `ocf1` disable/update-state contract.
