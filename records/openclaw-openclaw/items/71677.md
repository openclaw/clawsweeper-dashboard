## Summary

Keep open. Current main handles Feishu card button callbacks through the existing dispatch/deduplication path, but it still does not return an immediate WebSocket callback response for a toast or disabled transitional card; the related Schema 2.0 callback fix only covers malformed-callback routing.

## What This Changes

Request instant Feishu interactive-card feedback by returning a disabled transitional card/toast from `card.action.trigger` while command dispatch continues asynchronously.

## Best Possible Solution

Implement this in the Feishu plugin by claiming and validating the callback token first, returning a typed transitional toast/card response immediately, and moving eligible command dispatch to a background path that preserves existing completion/release behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without returning a callback response while the normal dispatch path runs.

## Solution Assessment

Yes. The handler-return path is the narrowest maintainable fix because the Feishu/Lark SDK supports callback return responses and the behavior is Feishu plugin-owned; the related Schema 2.0 parser fix does not address this UX gap.
