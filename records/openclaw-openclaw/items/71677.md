## Summary

Keep open. The Feishu plugin currently parses and dispatches card-action callbacks, but the registered callback still returns no instant card-update payload, and the related open parser PR is scoped to Schema 2.0 routing rather than this disabled/loading-card UX path.

## What This Changes

Request an instant Feishu card-action callback response that disables or transitions clicked buttons immediately while synthetic command processing continues asynchronously.

## Best Possible Solution

Implement the instant response inside the Feishu plugin: claim and validate the callback first, return an eligible disabled/loading card response immediately, run synthetic command dispatch in the background, and document/test any backward-compatible ocf1 disable-state extension without moving Feishu-specific behavior into core.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered card.action.trigger handler with a valid structured quick-action event and observe that it resolves without a callback response while current tests only assert routing/dedupe behavior.

## Solution Assessment

Yes, with guardrails. The handler-return path is the narrowest maintainable fix because the SDK already supports card handler return values, but the implementation should keep Feishu validation and replay-token semantics intact before launching background dispatch.
