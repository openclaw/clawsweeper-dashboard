## Summary

Keep open. Current main still handles Feishu card button callbacks through the normal dispatch/deduplication path and does not return an immediate WebSocket callback payload for a disabled or processing card state; the related #71787 work is an open parser/routing fix and does not cover this UX path.

## What This Changes

Request Feishu `card.action.trigger` handling to return a transitional disabled-card or toast callback response immediately while command dispatch continues asynchronously.

## Best Possible Solution

Implement this inside the Feishu plugin: claim and validate the callback token first, keep user/conversation authorization intact, launch eligible command dispatch in the background with existing completion/release behavior, and return a typed transitional card/toast response through the Lark SDK callback path. Any `ocf1` disable-state field should be additive, Feishu-specific, documented, and covered by focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without returning a callback response while normal message dispatch proceeds.

## Solution Assessment

Yes. The requested handler-return path is the narrowest maintainable solution because the SDK already carries handler return values, and the behavior belongs in the Feishu plugin rather than core; #71787 does not solve the instant-update UX.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/71677 with a narrow Feishu plugin change. The observable bug is that Feishu/Lark `card.action.trigger` button callbacks currently return no immediate callback response, so buttons stay interactive until the normal agent reply/edit path finishes. Implement immediate transitional feedback through the existing Lark SDK dispatcher return path: validate/claim the callback token first, preserve stale/wrong-user/wrong-conversation/malformed checks, start eligible quick/approval-confirm command dispatch asynchronously with existing completion/release semantics, and return a typed disabled/processing card and/or info toast response from the handler. Keep the change inside `extensions/feishu`; do not add bundled plugin ids or Feishu-specific behavior to core. If adding an `ocf1` disable-state field, make it additive and Feishu-owned. Add focused tests for the handler return value, duplicate click behavior while dispatch is in flight, invalid/stale/wrong-user cases, and any new card rendering helper. Add a user-facing `CHANGELOG.md` entry crediting the issue reporter if policy allows. Related context: #71787 is parser/routing work and should not be treated as solving this UX path.
