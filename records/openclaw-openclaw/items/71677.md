## Label Justifications

- `P2`: The request affects Feishu card-button UX and duplicate callback pressure, but it is a bounded channel improvement rather than an urgent runtime outage.

## Summary

Keep open. Current main and the latest release still parse and dedupe Feishu card actions but do not return an immediate WebSocket callback payload or define an `ocf1` disable/update-state field.

## What This Changes

Request instant Feishu card callback responses that disable or update card buttons immediately after a click.

## Best Possible Solution

Define one Feishu-plugin-owned typed callback response and `ocf1` disable/update-state contract, then prove immediate disabled-card feedback with focused tests and live Feishu proof.

## Reproduction Assessment

Yes, at source level. A valid Feishu `card.action.trigger` reaches the current handler, but current main resolves through `Promise<void>` command dispatch with no returned WebSocket callback payload or disable-state contract.

## Solution Assessment

Yes on direction, but not safe for autonomous implementation yet. The maintainable path is one typed Feishu callback-response contract coordinated with `ocf1` card metadata, not a one-off return payload.
