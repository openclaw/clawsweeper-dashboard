## Summary

Keep open. Current main still parses, deduplicates, and dispatches Feishu card-action callbacks, but it does not return an immediate WebSocket callback payload for a disabled or processing card state; the related #71787 work is parser/routing coverage and does not implement this UX path.

## What This Changes

Add Feishu card.action.trigger instant WebSocket callback responses so clicked card buttons can show an immediate disabled or processing state before the agent finishes.

## Best Possible Solution

Implement a narrow Feishu plugin change: validate and claim the callback token first, preserve wrong-user/wrong-conversation checks, start eligible command dispatch in the background with completion/release semantics intact, and immediately return a typed transitional card/toast response through the existing Lark SDK callback path. If ocf1 gains a disable-state field, keep it additive, documented, and covered by Feishu-specific tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered card.action.trigger handler with a valid structured quick-action event and observe that it resolves without returning a callback response while the command dispatch proceeds through the normal message path.

## Solution Assessment

Yes. The requested handler-return path matches the locked SDK plumbing, and keeping the change inside the Feishu plugin is the narrowest maintainable solution as long as replay protection and authorization checks are preserved.
