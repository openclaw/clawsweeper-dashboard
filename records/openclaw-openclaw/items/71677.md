## Summary

Keep open. Current main handles and deduplicates Feishu card actions, including the adjacent Schema 2.0 parsing path, but it still does not return a WebSocket callback payload that can immediately disable or mark the clicked card as processing.

## What This Changes

Request Feishu `card.action.trigger` WebSocket callback responses that instantly update clicked interactive cards, optionally via an `ocf1` disable-state field.

## Best Possible Solution

Keep this open and land a focused Feishu plugin fix that returns an immediate, typed transitional card or toast after callback validation and token claim, while preserving replay dedupe, authorization, wrong-user, and wrong-conversation protections.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves with no callback response while command dispatch happens through the normal message path.

## Solution Assessment

Yes. The handler-return path is the narrow maintainable direction because the SDK already propagates handler return values; the fix should stay Feishu-plugin-owned unless a reusable plugin SDK seam becomes necessary.
