## Summary

Keep open. Current main still routes Feishu card button callbacks through the normal dispatch and dedupe path, but it does not return an immediate WebSocket callback payload to update the card or show feedback; the related Schema 2.0 callback work only fixes callback parsing/routing.

## What This Changes

Add Feishu/Lark card-action WebSocket callback responses that immediately update or disable clicked cards while background command handling continues.

## Best Possible Solution

Implement this inside the Feishu plugin as a typed card-action callback response path with transitional card/button state, while keeping the Schema 2.0 parser fix separate.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action payload and observe that it resolves to `undefined` while the normal dispatch path runs.

## Solution Assessment

Yes. A Feishu plugin-owned handler return path is the narrowest maintainable solution; core SDK or generic channel changes are not needed unless the implementation reveals a missing plugin contract.
