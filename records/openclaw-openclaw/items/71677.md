## Summary

Keep open. Current main still parses, deduplicates, and dispatches Feishu card-action callbacks, but it does not return an immediate WebSocket callback payload to disable or mark the clicked button as processing; the related #71787 work is parser-only and does not implement this UX path.

## What This Changes

Request support for instant Feishu card updates from `card.action.trigger` WebSocket callback responses so clicked buttons can show a disabled or processing state before agent dispatch completes.

## Best Possible Solution

Implement a narrow Feishu plugin change: claim and validate the callback token first, start eligible command dispatch in the background with completion/release semantics preserved, and immediately return a typed transitional card/toast response through the existing Lark SDK callback path. If a new `ocf1` disable-state field is added, keep it additive, documented, and covered by Feishu unit and lifecycle tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without a callback response while current code dispatches the synthetic command through the normal message path.

## Solution Assessment

Yes. The issue’s proposed handler-return path matches the locked Lark SDK return plumbing, and keeping the change inside the Feishu plugin is the narrowest maintainable boundary as long as token replay dedupe and authorization checks are preserved.
