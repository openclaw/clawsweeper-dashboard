## Summary

Keep open. Current main still parses, deduplicates, and dispatches Feishu card-action callbacks, but it does not return an immediate WebSocket callback response that could disable the clicked card button or show processing feedback; the related parser work only covers Schema 2.0 callback shape acceptance.

## What This Changes

Support Feishu `card.action.trigger` WebSocket callback responses so interactive card buttons can show an immediate disabled or processing state instead of waiting for normal agent processing and message edits.

## Best Possible Solution

Implement the instant response inside the Feishu plugin: parse and validate the action, claim the callback token, preserve stale/wrong-user/wrong-conversation checks, launch eligible command dispatch in the background with existing completion/release semantics, and return a typed transitional card/toast response through the SDK callback path. Any `ocf1` disable-state extension should be additive, Feishu-owned, documented, and covered by card-action lifecycle tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without returning a callback response while dispatch proceeds through the normal message path.

## Solution Assessment

Yes. The requested handler-return path matches the pinned SDK plumbing, and keeping the change in the Feishu plugin is the narrowest maintainable fix as long as replay protection and authorization checks are preserved.
