## Summary

Keep open. Current main still cannot deliver the requested instant Feishu card update because the registered card callback returns no payload and the card-action path is still implemented as a void dispatch flow. The related Schema 2 parser/routing work only makes callbacks dispatch normally; it does not add a WebSocket callback response or disabled-button rendering.

## What This Changes

Request Feishu `card.action.trigger` handling to return an immediate transitional disabled-button card or toast response, optionally controlled by an additive `ocf1` disable-state field.

## Best Possible Solution

Implement this inside the Feishu plugin: validate and claim the callback token first, keep user and conversation authorization intact, return a typed transitional card/toast response through the SDK callback path, and continue eligible command dispatch in the background with existing completion and retry behavior. Any `ocf1` disable-state field should be additive, Feishu-specific, documented if exposed, and covered by focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid quick-action payload and observe that current main resolves without returning a callback response while normal message dispatch proceeds.

## Solution Assessment

Yes. The handler-return path is the narrowest maintainable solution because the SDK already carries handler return values and the behavior belongs in the Feishu plugin; the open Schema 2 parser PR does not solve the instant-update UX.
