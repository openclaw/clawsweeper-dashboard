## Summary

Keep open. Current main still does not return an immediate Feishu/Lark WebSocket card callback response for quick-action buttons, and the related open parser/routing PR does not cover the instant disabled or processing-card UX path.

## What This Changes

Request Feishu/Lark Schema 2.0 card button callbacks to return an instant disabled-button card or toast response while command dispatch continues asynchronously.

## Best Possible Solution

Implement this as a narrow Feishu plugin UX repair: validate and claim the callback token first, preserve current authorization and replay behavior, return a typed transitional card/toast through the existing Lark SDK callback path, and keep any new `ocf1` disable-state metadata additive and Feishu-specific.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without returning any callback response while normal dispatch/deduplication proceeds.

## Solution Assessment

Yes. The requested handler-return path is the narrowest maintainable fix because the Lark SDK already propagates handler return values, and the behavior belongs in the Feishu plugin rather than core.
