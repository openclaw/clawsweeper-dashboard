## Summary

Keep open. Current main accepts and deduplicates Feishu card-action callbacks, including the adjacent Schema 2.0 `open_chat_id` parsing path, but it still does not return an immediate WebSocket callback payload that disables or marks the clicked card button as processing.

## What This Changes

Request Feishu `card.action.trigger` handling to return an instant card-update callback response while continuing synthetic command processing asynchronously.

## Best Possible Solution

Implement this as a narrow Feishu plugin change: validate and claim the callback token first, launch eligible structured quick/approval-confirm dispatch in the background with token completion/release preserved, and immediately return a typed disabled/loading card or toast through the existing SDK callback path. If an `ocf1` disable-state field is added, keep it backward-compatible, document it, and cover it with Feishu unit/lifecycle tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without any callback response while the current action path performs synthetic command dispatch separately.

## Solution Assessment

Yes. The issue's proposed handler-return path matches the locked Lark SDK behavior; the maintainable fix is Feishu-plugin-owned and should preserve callback validation, replay-token dedupe, wrong-user/wrong-conversation checks, and command authorization.
