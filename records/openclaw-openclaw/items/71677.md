## Summary

Keep open. Current main still handles Feishu card actions without returning an immediate WebSocket callback payload or defining a disable-state/update-card contract; the merged Schema 2 parser work only fixed callback routing, and the open Feishu card-update PRs are overlapping but not a complete/canonical implementation of this request.

## What This Changes

Support Feishu `card.action.trigger` WebSocket callback responses that immediately return a disabled/transitional card while agent processing continues.

## Best Possible Solution

Implement this inside the Feishu plugin as the canonical card-action feedback path: return a typed instant callback response from the handler, define the `ocf1` disabled/update-state contract, preserve token lifecycle semantics, and coordinate or replace the overlapping open card-update PRs.

## Reproduction Assessment

Yes, at source level. A valid `card.action.trigger` payload reaches the registered handler, but current main resolves with `undefined` while the command path runs, and the existing tests only prove routing/dedupe behavior.

## Solution Assessment

Yes on direction, but not yet on implementation path. Returning a Feishu-plugin-owned handler response is the narrowest way to get millisecond feedback, while the open card-update PRs need maintainer coordination because they do not settle the WebSocket callback/disable-state contract.
