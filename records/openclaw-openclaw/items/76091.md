## Summary

Keep open. The source gap on current main is real and the PR’s single-owner typing direction is reasonable, but the patch starts accepted typing before queue execution while cleanup only runs inside processing, so skipped queued jobs can leave typing alive; it also lacks the required user-facing changelog entry.

## What This Changes

The branch adds a Discord reply typing feedback owner, carries it from accepted preflight through queued processing, reuses it in the reply pipeline, refreshes shared typing keepalive starts, and adds focused tests.

## Best Possible Solution

Keep the single-owner Discord typing design, but make accepted feedback cleanup part of the queue/deactivation lifecycle and add an active-version changelog entry before merge.

## Reproduction Assessment

Yes for the blocking PR regression: extend the existing skipped queued-run deactivation test with an injected replyTypingFeedback, enqueue a second same-key message, deactivate the handler, and assert the skipped feedback owner is cleaned while processDiscordMessage still runs only once.

## Solution Assessment

No, not yet. Reusing one typing owner is the narrow maintainable direction, but cleanup must cover accepted feedback that never reaches processDiscordMessage, and the user-facing fix needs a changelog entry.
