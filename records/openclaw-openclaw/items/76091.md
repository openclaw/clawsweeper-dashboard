## Summary

Keep open. The PR is aimed at a real Discord typing lifecycle gap, but the proposed patch starts accepted typing before enqueue and does not clean it up when a queued job is skipped, and it also misses the required changelog entry for a user-visible Discord fix.

## What This Changes

The PR adds a Discord reply typing feedback owner, carries it through queued inbound processing, reuses it in the reply dispatcher, refreshes shared typing callbacks, and adds focused tests.

## Best Possible Solution

Keep the single-owner Discord typing design, but make accepted feedback cleanup part of the queue/deactivation lifecycle and add an active Fixes changelog entry before merge.

## Reproduction Assessment

Yes. Extend the existing skipped queued-run deactivation test with an injected replyTypingFeedback, enqueue a second same-key message, deactivate the handler, and assert the skipped feedback owner is cleaned while processDiscordMessage still runs only once.

## Solution Assessment

No, not yet. Reusing one typing owner is the right narrow direction, but the cleanup path must cover jobs that never enter processDiscordMessage and the user-facing fix needs changelog coverage.
