## Summary

Keep open. This is a focused Discord typing-lifecycle fix for a real current-main gap, and I found no blocking correctness or security finding; the main remaining question is the compatibility tradeoff around one-shot typing for unconfigured `message_tool_only` replies and coordination with overlapping related PRs.

## What This Changes

The PR carries Discord reply typing feedback from accepted preflight through queued processing and dispatch, adds skipped-run cleanup, and adds a core one-shot typing keepalive option with Discord tests, docs, and a changelog entry.

## Best Possible Solution

Land a single Discord typing lifecycle direction that carries accepted preflight feedback through dispatch, cleans up skipped queued work, and deliberately chooses whether the `message_tool_only` edge belongs in this PR or in the narrower related PRs.

## Solution Assessment

Yes for the accepted Discord typing delay: carrying one feedback owner from preflight through dispatch is the right boundary. The `message_tool_only` keepalive subset is a maintainer tradeoff because it changes default refresh behavior and overlaps related open PRs.
