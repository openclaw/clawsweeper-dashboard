## Summary

Keep open: this PR targets a real Discord typing regression, but the current head still starts accepted feedback for every non-`never` typing mode and for default guild messages before the dispatcher lifecycle, which conflicts with the documented typing-mode contract.

## What This Changes

The PR carries a Discord reply typing feedback owner from accepted preflight through queued dispatch, adds skipped-run cleanup to the shared queue, refreshes repeated typing starts, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the single carried Discord typing owner, but only pre-start it for resolved instant-mode cases; for `message`, `thinking`, and default unmentioned group mode, carry the owner without starting until the dispatcher lifecycle reaches the configured signal.

## Reproduction Assessment

Yes. Source inspection of current main shows Discord accepted typing is narrow while dispatcher typing starts only after reply-plan resolution, and related issue #76396 provides concrete Discord-channel delay steps after v2026.4.29.

## Solution Assessment

No. Reusing one feedback owner is the right implementation shape, but this PR starts it too early for documented conservative modes; the safer repair is a narrow preflight-start gate plus regression coverage.

## Review Findings

- **[P2] Respect typingMode before starting accepted feedback:** `extensions/discord/src/monitor/message-handler.ts:77`
