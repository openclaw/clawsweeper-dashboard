## Summary

Keep open: the PR addresses a real Discord reply typing lifecycle gap, but the current head still starts accepted typing feedback during preflight for every non-`never` mode, which regresses documented `message`, `thinking`, and default unmentioned-group timing.

## What This Changes

The PR carries Discord accepted typing feedback from message preflight through queued processing, reuses it in the reply dispatcher, adds skipped-run cleanup to the shared queue, refreshes shared typing restarts, and updates tests/docs/changelog.

## Best Possible Solution

Keep the single carried Discord typing owner, but only pre-start it when the resolved typing semantics are instant; conservative modes should carry the owner and let dispatcher stream events start it later.

## Reproduction Assessment

Yes. Source inspection shows current main still has a Discord pre-dispatch typing gap, and the PR head's non-`never` gate provides a high-confidence path for the introduced conservative-mode regression.

## Solution Assessment

No. Reusing one feedback owner is the maintainable shape, but the current PR starts it before applying the same typing-mode semantics used by reply runs; the safer fix is to gate preflight start to resolved instant cases.
