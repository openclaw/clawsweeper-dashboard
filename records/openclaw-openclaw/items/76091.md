## Summary

Keep open. The PR targets a real Discord feedback split and carries useful focused tests, but the shared typing callback change can leave a tripped start guard unrecoverable on a later reply start, so the patch is not merge-ready; it also needs the required changelog entry.

## What This Changes

The PR adds a Discord reply typing feedback owner, carries accepted-message typing through the inbound queue into dispatch, changes shared typing callbacks to refresh active keepalives, and adds focused tests.

## Best Possible Solution

Keep the single Discord typing owner approach, but make the shared lifecycle refresh recover after a breaker trip and add the required active-release changelog entry before merge.

## Reproduction Assessment

Yes. A focused unit path can reproduce the blocker by using createTypingCallbacks with a failing start until maxConsecutiveFailures trips, then making start succeed and calling onReplyStart again; with the PR's started latch, the guard remains tripped and start is not called again.

## Solution Assessment

No. Reusing one Discord typing owner is the right direction, but the current patch should preserve recovery semantics in createTypingCallbacks instead of making repeated starts unable to reset a tripped guard.
