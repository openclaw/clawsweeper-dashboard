## Summary

Keep open: the PR addresses a real Discord typing lifecycle gap, but the current diff starts accepted typing for `message` and `thinking` modes, which conflicts with the documented typing-mode contract and should be fixed before merge.

## What This Changes

The PR carries Discord reply typing feedback from accepted preflight through queued processing and dispatch, adds skipped-run cleanup and keepalive refresh behavior, and updates tests, changelog, SDK docs, and API baseline hashes.

## Best Possible Solution

Land a Discord-owned single typing-feedback owner after it preserves configured typing-mode semantics and passes the targeted Discord and shared typing checks.

## Reproduction Assessment

Yes at source level: current main starts an accepted Discord typing cue before queueing and later creates a separate dispatcher typing owner. The related Discord reports and PR proof add live symptom context, but I did not run a live Discord bot in this read-only pass.

## Solution Assessment

No as written: carrying one DiscordReplyTypingFeedback object is the right narrow shape, but prestarting it for every non-`never` mode overrides the existing `message` and `thinking` timing contract. The safer fix is to prestart only when the effective mode is instant/default, while still reusing the carried owner once dispatch begins.
