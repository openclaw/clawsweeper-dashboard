## Best Possible Solution

Keep the scoped Discord lifecycle approach, but first repair the testing-hook property and pass the skipped-run cleanup through createChannelRunQueue's new onSkip option, then rerun the focused Discord and SDK queue checks.

Not yet. The one-feedback-owner design is the right narrow fix, but the implementation currently misses the existing testing hook name and leaves the new skipped-run cleanup helper unwired.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the existing testing hook property:** `extensions/discord/src/monitor/message-handler.ts:213`
  - body: `DiscordMessageHandlerParams` still exposes test hooks as `testing`, but this branch reads `params.__testing` when creating reply typing feedback. That property does not exist, so the current head fails type/test checks and the injected feedback factory used by the new tests is never reached; use `params.testing?.createReplyTypingFeedback` and update the tests to pass `testing`.
  - confidence: 0.97
- **[P2] Wire skipped-run cleanup into the queue:** `extensions/discord/src/monitor/message-run-queue.ts:76-89`
  - body: `cleanupSkippedDiscordQueuedMessage` is added but never passed to `createChannelRunQueue` as the new `onSkip` option. When a message is accepted, starts typing, and is later skipped after deactivation, the queued task does not call `replyTypingFeedback.onCleanup()` or release the replay guard through this helper, so the typing loop can keep running until TTL. Pass an `onSkip` callback from `enqueue(job)` that calls this helper.
  - confidence: 0.94
