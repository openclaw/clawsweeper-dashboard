## Summary

Keep this PR open: it targets a real Discord typing lifecycle regression, but the current head still starts accepted typing feedback for documented delayed modes and default unmentioned guild messages.

## What This Changes

The PR carries Discord reply typing feedback from preflight through queued processing, adds skipped-run cleanup hooks, refreshes shared typing starts, and updates tests/docs/changelog.

## Best Possible Solution

Keep the single carried Discord typing owner, but only pre-start it when the resolved typing semantics are instant; otherwise carry it dormant until the dispatcher text/reasoning/tool lifecycle starts it.

## Reproduction Assessment

Yes. Current main source shows the accepted preflight cue is separate from the later dispatcher typing callback, and related issue #76396 provides concrete Discord-channel delay observations.

## Solution Assessment

No. Reusing one feedback owner is the right shape, but this PR starts it too early for `message`, `thinking`, and default unmentioned guild semantics; the narrower fix is to gate preflight start through resolved instant-mode behavior.
