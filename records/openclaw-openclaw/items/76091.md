## Summary

Keep open: this PR targets a valid Discord typing lifecycle regression, but the latest diff still starts accepted feedback too broadly and would break documented typing-mode timing.

## What This Changes

The PR adds a carried Discord reply typing feedback owner, skipped queued-run cleanup, shared typing keepalive changes, and matching tests/docs/changelog.

## Best Possible Solution

Land a revised PR that carries one Discord typing owner across preflight, queue, and dispatch while preserving documented `typingMode` timing and skipped-run cleanup coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: accepted Discord work can wait until the later dispatcher typing lifecycle, and the related report provides concrete Discord-channel delay observations. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. Reusing one carried feedback owner is the right direction, but the PR starts it during preflight for every non-`never` mode; the safer fix is to pre-start only for resolved instant cases and otherwise let dispatcher lifecycle events start it.
