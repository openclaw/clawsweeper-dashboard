## Summary

Keep open. Current main still has the split Discord typing lifecycle this PR is trying to fix, but the patch is not merge-ready because early accepted typing can be left running when queued Discord work is skipped before processing, and the user-facing fix is missing a changelog entry.

## What This Changes

The PR adds a reusable Discord reply typing feedback owner, starts it after accepted preflight, carries it through queued inbound jobs, reuses it in processing, and refreshes shared typing keepalive starts.

## Best Possible Solution

Keep the single Discord typing owner design, but make accepted feedback cleanup part of the queue/deactivation lifecycle and add the required active-version changelog entry before merging.

## Reproduction Assessment

Yes for the blocking review finding: extend the existing deactivation test with an injected `replyTypingFeedback`, enqueue a second same-key message, call `handler.deactivate()`, and assert the skipped second owner is cleaned while `processDiscordMessage` is still called only once. For the original flicker, the source path gives a credible reproduction but I did not live-test Discord.

## Solution Assessment

No, not yet. Reusing one typing feedback owner is the right direction, but cleanup must cover accepted feedback that never reaches `processDiscordMessage`; the changelog requirement is also still unmet.
