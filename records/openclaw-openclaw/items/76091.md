## Summary

Keep open: current main still has the Discord reply-typing gap this PR targets, and the branch has the right single-owner shape, but the current head still starts accepted feedback for delayed typing modes that should wait for reply lifecycle signals.

## What This Changes

The PR carries a Discord reply typing feedback owner from accepted preflight through queued processing, adds skipped-run cleanup and shared typing keepalive refresh behavior, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the single carried Discord typing owner, but decide preflight start from the same resolved typing-mode semantics as the reply run so only instant cases type immediately and delayed modes merely carry the owner forward.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted Discord work only gets the early cue for DMs with unset/instant typing mode, while Discord channel replies rely on later dispatcher typing; #76396 provides concrete post-upgrade delay observations.

## Solution Assessment

No. Reusing one carried feedback owner is the right direction, but the PR starts it too early for `message`, `thinking`, and default unmentioned group semantics; the safer fix is to carry the owner and only pre-start it for resolved instant cases.
