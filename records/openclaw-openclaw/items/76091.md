## Summary

Keep open: the carried typing owner design is plausible and real behavior proof is supplied, but the current head still skips accepted-preflight typing for default no-mention Discord guild/channel tool-only turns, leaving one P2 blocker before merge.

## What This Changes

The PR carries Discord reply typing feedback from accepted preflight through queued processing and dispatch, adds skipped-run cleanup plus shared typing keepalive changes, and updates focused tests, docs, changelog, and the SDK API hash.

## Best Possible Solution

Keep the single carried feedback owner, but derive accepted-preflight typing from the same delivery and typing policy that dispatch uses so default message_tool_only guild/channel replies prestart while explicit typing modes still win.

## Reproduction Assessment

Yes, at source level: current main has a DM-only accepted cue before queueing and a separate dispatcher typing owner after Discord reply-plan resolution. The linked report also describes delayed Discord channel typing after upgrading to 2026.4.29.

## Solution Assessment

No, not yet: carrying one feedback owner is the narrow maintainable fix, but the accepted-preflight gate must align with the existing tool-only group typing policy before this broadly fixes delayed Discord channel typing.
