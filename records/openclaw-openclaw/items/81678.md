## Summary

Keep open: the PR addresses a real auth stale-lock wedge and now has usable real-behavior proof, but the current head still enables stale sidecar removal through a dependency primitive that rechecks identity and then deletes by pathname, leaving the credential lock mutual-exclusion race unresolved.

## What This Changes

The branch bumps `@openclaw/fs-safe` to 0.2.4, adds dead-owner stale-lock policy/helpers/tests, and enables plugin SDK file-lock recovery for stale auth/profile lock sidecars.

## Best Possible Solution

Keep stale auth sidecar recovery fail-closed until fs-safe can bind deletion to the inspected lock identity, or land an equivalent supervised recovery path with multi-process contention proof.

## Reproduction Assessment

Yes, from source inspection. Current main routes auth-store and OAuth refresh writes through file-lock sidecars, and a dead-owner stale sidecar reaches the plugin SDK/fs-safe stale-lock path; I did not run a live OAuth repro in this read-only review.

## Solution Assessment

No, not at the current head. Moving stale-removal mechanics into fs-safe is the right boundary, but the 0.2.4 primitive still rechecks and then deletes by pathname instead of making deletion identity-bound.
