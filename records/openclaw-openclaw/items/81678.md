## Summary

Keep open: this protected-label PR targets a real auth lock recovery problem, but the patch introduces a lock-safety race by deleting stale lock files by path and it lacks after-fix real behavior proof.

## What This Changes

The PR changes plugin SDK file locking to remove reported stale lock files whose recorded PID is no longer alive, adds regression tests, and adds a changelog entry for auth-profile lock recovery.

## Best Possible Solution

Keep fs-safe’s fail-closed behavior until stale auth locks can be reclaimed through an identity-bound lower-level primitive or a narrowly supervised cleanup path, then prove the real auth-store recovery path.

## Reproduction Assessment

Yes from source inspection, but not by executing in this read-only review. A stale `auth-profiles.json.lock` with an old timestamp and dead PID reaches fs-safe’s stale path, which currently fails closed with `file_lock_stale` instead of acquiring; the PR’s added tests encode that path.

## Solution Assessment

No. The wrapper-level read/compare/delete-by-path approach does not satisfy the fs-safe stale-lock contract because another process can acquire a fresh lock between the final comparison and `unlink`, so the safer fix belongs in an identity-bound lock primitive or a supervised cleanup path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve lock identity before unlinking stale locks:** `src/plugin-sdk/file-lock.ts:150`
  - body: The new recovery path verifies the sidecar contents and then deletes by path. If two reclaimers race, one can remove the stale file, another process can acquire a fresh lock, and this caller can then unlink that fresh lock before retrying. That breaks mutual exclusion for `auth-profiles.json` writers; fs-safe currently fails closed here because Node cannot bind the delete to the inspected file.
  - confidence: 0.91
