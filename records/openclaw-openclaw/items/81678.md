## Summary

Keep open: this maintainer-labeled PR still has a blocking lock-safety issue and lacks real behavior proof. The latest head moves cleanup into a helper, but it still performs read/compare/unlink by path, which conflicts with fs-safe's fail-closed stale-sidecar contract for auth-profile writes.

## What This Changes

The branch adds a stale-lock helper, changes plugin SDK file locking to remove dead-owner stale locks and retry acquisition, adds PID/zombie tests, and adds a changelog entry.

## Best Possible Solution

Keep stale sidecar recovery fail-closed until cleanup can be tied to lock identity in the lower-level primitive or a supervised auth recovery path, then prove the real auth-store recovery flow.

## Reproduction Assessment

Yes, from source inspection. A stale auth-profiles.json.lock can reach fs-safe's file_lock_stale path, and the PR then attempts wrapper-level recovery; I did not execute it in this read-only review.

## Solution Assessment

No. The proposed wrapper-level read/compare/delete-by-path approach is not the safest fix because deletion is not bound to the inspected lock identity; the narrower maintainable fix is an identity-bound primitive or supervised recovery path.
