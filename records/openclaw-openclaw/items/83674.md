## Label Justifications

- `P2`: The report describes a serious but plugin-scoped event-loop stall with limited evidence of a current OpenClaw core regression.
- `impact:crash-loop`: The reported 21s event-loop stall is a process availability problem even though it is attributed to an external plugin.
- `impact:message-loss`: The report says the stall prevents other channels and fallback paths from working correctly during startup.

## Summary

Close: the reported synchronous startup work is in Tencent's external Weixin channel plugin, not in OpenClaw core, and current docs/catalog make that ownership boundary explicit.

## What This Changes

The report asks for Weixin channel startup to stop blocking the Gateway event loop by replacing synchronous plugin filesystem calls or deferring plugin startup.

## Best Possible Solution

Keep OpenClaw core channel-agnostic; fix the blocking startup filesystem work in @tencent-weixin/openclaw-weixin and update or reinstall that external plugin once published.

## Reproduction Assessment

No high-confidence repo-local reproduction exists from this review: the liveness log is plausible, but the named startup files are not in the OpenClaw checkout and the external npm package was not executed.

## Solution Assessment

No, adding a service-specific core delay knob is not the best fit; the maintainable fix is for the external Weixin plugin to make its startup async or move blocking work off the Node event loop.
