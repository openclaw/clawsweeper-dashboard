## Label Justifications

- `P2`: The report describes a real but externally owned Gateway availability and delivery problem with limited OpenClaw-core actionability.
- `impact:message-loss`: The issue reports that event-loop blocking prevents other channels and model fallback paths from processing messages correctly.
- `impact:crash-loop`: The issue is about a 21s event-loop stall and process-level liveness degradation during Gateway startup.

## Summary

The reported event-loop block is plausible, but the named fs.*Sync startup files and Weixin runtime live in Tencent's external @tencent-weixin/openclaw-weixin package, while this repo only owns the generic channel contract and startup handoff.

## What This Changes

The issue asks for Weixin channel startup to stop blocking the Gateway event loop by making the external plugin startup async or delaying that channel startup.

## Best Possible Solution

Keep OpenClaw core channel-agnostic and fix the blocking startup work in @tencent-weixin/openclaw-weixin, then have users update or reinstall that external plugin.

## Reproduction Assessment

No high-confidence repo-local reproduction exists. The liveness log is plausible, but the named startup files are not in this checkout and I did not run the external @tencent-weixin/openclaw-weixin package.

## Solution Assessment

Yes, closing this as outside the OpenClaw source repository is the best repo-maintenance outcome. The maintainable fix is for the external Weixin plugin to make startup async or move blocking work off the Node event loop, not to add a Weixin-specific core delay knob without a proven generic API gap.
