## Summary

Keep open. Current main still hard-fails Telegram delete-message API errors through the shared logging/rethrow path, and the linked fix attempts are closed unmerged, so the reported operator-log noise is still actionable.

## What This Changes

The issue requests that `deleteMessageTelegram` treat benign Telegram delete no-op 400s as soft failures, like the existing reaction fail-soft path, and surface a warning through the Telegram action runtime.

## Best Possible Solution

Patch only the Telegram plugin: classify exact benign delete no-op errors, suppress their ERROR-level API log with the existing per-call `shouldLog` seam, return a structured warning from `deleteMessageTelegram`, surface `deleted: false` in the action runtime, and add focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The reported CLI action maps to `handleTelegramAction` calling `deleteMessageTelegram`; a high-confidence unit reproduction can inject an API whose `deleteMessage` rejects with `Bad Request: message can't be deleted` and observe that current main logs/rethrows instead of returning a warning.

## Solution Assessment

Yes. A narrow Telegram-plugin fail-soft path is the best maintainable direction; the safer implementation is exact delete-error matching with tests proving non-benign errors still throw.

## Repair Work Prompt

Fix openclaw/openclaw#73726. In the Telegram plugin only, make `deleteMessageTelegram` fail softly for exact benign delete no-op Telegram Bot API errors: `message to delete not found`, `message can't be deleted` / `message cant be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`. Use the existing `createTelegramRequestWithDiag` per-call `shouldLog` seam so those exact benign cases do not emit the shared ERROR-level `telegram deleteMessage failed` log, then catch them and return `{ ok: false, warning }`. Preserve hard throw/log behavior for auth failures, rate limits, network errors, and unrelated Telegram errors. Update `handleTelegramAction` for `deleteMessage` so a soft result returns `deleted: false` with the warning instead of always `deleted: true`. Add focused regression coverage for all benign strings and for a non-benign error still rethrowing, plus action-runtime coverage for the warning result. Add a single-line user-facing `CHANGELOG.md` fix entry credited with `Thanks @Avicennasis`. Do not move behavior into core and do not broaden all Telegram 400s.
