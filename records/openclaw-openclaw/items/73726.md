## Summary

Keep open. Current main still lets Telegram delete-message API failures flow through the logged/rethrow path, while the same plugin has local fail-soft precedents and the related fix PRs were closed unmerged.

## What This Changes

The issue asks for Telegram delete-message actions to treat benign Bot API 400 delete failures as non-fatal warnings instead of noisy hard errors.

## Best Possible Solution

Patch the Telegram plugin boundary only: add a small benign-delete-error classifier, pass `shouldLog` so those errors do not emit ERROR-level Telegram API logs, return a structured warning from `deleteMessageTelegram`, propagate `deleted: false` through the action runtime, add focused regression tests, and include a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence reproduction exists by mocking `api.deleteMessage` to reject with `400: Bad Request: message can't be deleted` or `message to delete not found` and observing the current `requestWithDiag` log/rethrow path; the reported CLI action reaches the same runtime branch.

## Solution Assessment

Yes. A narrow Telegram plugin fail-soft path is the best maintainable fix; changing shared retry, delivery recovery, or global API logging would be broader and could hide unrelated Telegram failures.

## Repair Work Prompt

Fix the Telegram benign delete-message error handling for https://github.com/openclaw/openclaw/issues/73726. In `extensions/telegram/src/send.ts`, keep the change inside the Telegram plugin: add a small classifier for benign delete failures such as `message to delete not found`, `message can't be deleted` / `message can’t be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`; pass the existing `shouldLog` option into `requestWithDiag` so those benign failures do not log through `withTelegramApiErrorLogging`; catch them in `deleteMessageTelegram` and return `{ ok: false, warning }` while preserving hard throws for auth, network, and other non-benign failures. In `extensions/telegram/src/action-runtime.ts`, surface the helper result so soft failures return a warning and `deleted: false` instead of unconditional `deleted: true`. Add focused regression coverage in the Telegram tests for benign delete strings, non-benign rethrow, and action-runtime warning propagation; keep proxy coverage working. Add a single-line user-facing `CHANGELOG.md` fix entry with appropriate non-bot credit. Do not change shared core retry, delivery recovery, or global Telegram API logging behavior beyond using the existing per-call suppression seam.
