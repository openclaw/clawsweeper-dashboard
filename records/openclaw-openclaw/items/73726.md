## Summary

Keep open. Current main still routes benign Telegram delete-message 400s through the logged/rethrow path, while the same plugin already has fail-soft and log-suppression precedents for analogous Telegram action errors; the related fix PRs are closed unmerged.

## What This Changes

The issue asks for `deleteMessageTelegram` to treat known benign Telegram Bot API delete failures as soft no-op warnings instead of logged hard failures.

## Best Possible Solution

Patch only the Telegram plugin delete helper and action runtime: suppress logging for known benign delete no-ops, return a structured warning with `deleted: false`, and keep unrelated failures hard.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inject an `api.deleteMessage` mock that rejects with `400: Bad Request: message can't be deleted` and observe the current `requestWithDiag` plus API logger path rethrowing and logging it; the documented CLI command reaches the same `deleteMessage` action branch.

## Solution Assessment

Yes. A narrow Telegram-plugin fail-soft path is the maintainable fix; changing shared retry or API logging semantics would risk hiding unrelated Telegram failures.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73726 with a narrow Telegram plugin patch. In `extensions/telegram/src/send.ts`, make `deleteMessageTelegram` fail soft for only known benign delete no-op errors such as `message to delete not found`, `message can't be deleted`/`message cant be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`; suppress the shared API ERROR log for those cases via the existing `shouldLog` seam, return a structured warning, and continue throwing all non-benign errors. In `extensions/telegram/src/action-runtime.ts`, propagate the soft result as `deleted: false` with the warning instead of always returning `deleted: true`. Add focused regression coverage in the Telegram tests for the benign strings, non-benign rethrow/log behavior, and action-runtime warning propagation. Add a user-facing `CHANGELOG.md` fix entry if policy requires it. Do not change shared retry/logging semantics, core channel behavior, or unrelated Telegram delete call sites.
