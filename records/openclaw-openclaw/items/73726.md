## Summary

Keep open. Current main still routes benign Telegram delete-message 400s through the logged rethrow path, while the Telegram plugin already has a local fail-soft reaction precedent and the related fix PRs in the issue timeline are closed unmerged.

## What This Changes

The issue asks for `deleteMessageTelegram` to treat benign Telegram delete failures as non-fatal warnings and propagate that result through the Telegram message action.

## Best Possible Solution

Patch only the Telegram plugin action path: classify benign delete no-op errors, suppress their ERROR-level log, return a structured warning with `deleted: false`, and keep non-benign failures hard.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by injecting an `api.deleteMessage` mock that rejects with `400: Bad Request: message can't be deleted` and observing the current logged rethrow path; the reported CLI command reaches the same action branch.

## Solution Assessment

Yes. A narrow Telegram plugin fail-soft path is the best maintainable fix; changing shared retry or API logging semantics would risk hiding unrelated Telegram failures.

## Repair Work Prompt

Fix the Telegram delete-message fail-soft bug tracked at https://github.com/openclaw/openclaw/issues/73726. In `extensions/telegram/src/send.ts`, make `deleteMessageTelegram` classify benign Telegram delete no-op errors (`message to delete not found`, `message can't/cant be deleted`, `MESSAGE_ID_INVALID`, `MESSAGE_DELETE_FORBIDDEN`) as non-fatal, suppress the shared ERROR-level API log for those exact cases, and return a structured warning while preserving hard failures for auth/network/other errors. In `extensions/telegram/src/action-runtime.ts`, propagate that result as `deleted: false` with the warning instead of always returning `deleted: true`. Add focused regression tests in the Telegram send/action test files, include a user-facing `CHANGELOG.md` fix entry, and do not change shared Telegram retry or API logging semantics globally. Treat #73734, #73735, and #73927 as useful closed source attempts, but implement against current main.
