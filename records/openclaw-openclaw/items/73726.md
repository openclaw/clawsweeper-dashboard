## Summary

Keep open. Current main still hard-fails Telegram `deleteMessage` API errors through the shared logging/rethrow path, while the action runtime still reports every completed helper call as `deleted: true`; the related implementation PRs in the provided timeline are closed and unmerged.

## What This Changes

Request fail-soft handling for benign Telegram `deleteMessage` 400 responses, mirroring the existing reaction warning behavior and surfacing a delete warning to callers.

## Best Possible Solution

Patch the Telegram plugin only: add an exact benign-delete classifier, use the existing `shouldLog` seam to avoid ERROR-level noise for those cases, return a structured warning from `deleteMessageTelegram`, surface `deleted: false` through the action runtime, and add focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The reported CLI path maps to the documented Telegram delete action, and the current behavior can be reproduced deterministically by injecting a Telegram API whose `deleteMessage` rejects with one of the reported benign 400 messages.

## Solution Assessment

Yes. A narrow plugin-local fail-soft path is the maintainable solution; the safer alternative to general 400 handling is exact delete-error classification with tests proving non-benign errors still rethrow.

## Repair Work Prompt

Fix this issue in the Telegram plugin only. Classify exact benign Telegram delete no-op responses from `api.deleteMessage` (`message to delete not found`, `message can't be deleted` / `message can’t be deleted`, `MESSAGE_ID_INVALID`, `MESSAGE_DELETE_FORBIDDEN`) as soft delete results. Use the existing `createTelegramRequestWithDiag` per-call `shouldLog` option so these benign cases do not emit ERROR-level `telegram deleteMessage failed` logs, but auth, permission, rate-limit, network, and unrelated API errors must still log/rethrow. Widen `deleteMessageTelegram` to return `{ ok: true } | { ok: false; warning: string }`, update `extensions/telegram/src/action-runtime.ts` so the `deleteMessage` action returns `deleted: false` plus the warning for soft results, and keep public/runtime exports stable. Add focused tests for all benign strings, non-benign rethrow/log behavior, and action-runtime warning propagation. Add a single-line user-facing `CHANGELOG.md` fix entry credited to `@Avicennasis`. Do not move this into core, do not add broad 400 handling, and do not touch unrelated Telegram delete paths such as native command cleanup unless a test proves they share this exact helper path.
