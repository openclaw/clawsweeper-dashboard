## Summary

Keep open. Current main still routes Telegram delete-message API failures through the shared ERROR logging and rethrow path, while all related fix PRs in the provided context are closed unmerged, so the reported operator-log noise remains actionable.

## What This Changes

The issue requests that `deleteMessageTelegram` fail softly for benign Telegram Bot API delete 400s and surface a warning instead of logging and throwing them as hard runtime errors.

## Best Possible Solution

Patch only the Telegram plugin: classify exact benign delete no-op errors, suppress their ERROR-level API log through the existing per-call `shouldLog` seam, return a structured warning from `deleteMessageTelegram`, surface `deleted: false` in the action runtime, and add focused tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can inject a Telegram API override whose `deleteMessage` rejects with `Bad Request: message can't be deleted` and observe that current main logs and rethrows instead of returning a warning.

## Solution Assessment

Yes. The narrow plugin-local fail-soft direction is the best fix, but the implementation should use the existing `shouldLog` seam in addition to a catch block so the shared API logger does not emit the benign error first.

## Repair Work Prompt

Fix the Telegram delete-message fail-soft bug tracked at https://github.com/openclaw/openclaw/issues/73726. Scope the change to the Telegram plugin and user-facing changelog. In `extensions/telegram/src/send.ts`, make `deleteMessageTelegram` treat only exact benign delete no-op errors (`message to delete not found`, `message can't be deleted` / `message cant be deleted`, `MESSAGE_ID_INVALID`, `MESSAGE_DELETE_FORBIDDEN`) as non-fatal. Use the existing `requestWithDiag` per-call `shouldLog` seam so these benign errors do not emit ERROR-level `telegram deleteMessage failed` logs before the catch; keep non-benign Telegram/API/network/auth errors throwing. Return a structured union such as `{ ok: true } | { ok: false; warning: string }`. In `extensions/telegram/src/action-runtime.ts`, surface soft deletes as `deleted: false` with the warning instead of always returning `deleted: true`. Add focused regression tests in the existing Telegram tests for all benign strings, for a non-benign error still throwing, and for action-runtime warning propagation. Add a single-line `CHANGELOG.md` fix entry with `Thanks @Avicennasis`. Do not change core delivery recovery, broad Telegram retry policy, plugin SDK contracts, or unrelated Telegram actions.
