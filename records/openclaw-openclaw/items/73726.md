## Summary

Keep open. Current main still routes Telegram delete-message failures through the shared logged/rethrow path, while nearby reaction and edit-markup helpers already have soft-fail or log-suppression precedents for benign Telegram errors; the related fix PRs in the provided context are closed unmerged, so the reported operator-log noise remains actionable.

## What This Changes

The issue asks for Telegram `deleteMessageTelegram` to treat benign “already gone/not deletable” Bot API 400s as soft no-op results and surface a warning instead of logging and throwing.

## Best Possible Solution

Patch only the Telegram plugin wrapper: classify exact benign delete no-op errors, suppress their shared API error log, return a structured warning from `deleteMessageTelegram`, propagate `deleted: false` through the Telegram action runtime, and pin the behavior with focused tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can inject an `api.deleteMessage` mock that rejects with `400: Bad Request: message can't be deleted` and observe that current main logs/rethrows instead of returning a warning; the reported CLI path maps to the same `handleTelegramAction` delete branch.

## Solution Assessment

Yes. A narrow Telegram-plugin fail-soft path is the best maintainable solution; broad retry or core delivery changes would not address this specific action-runtime/logging surface and could hide non-benign failures.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73726 with a narrow Telegram plugin change only. In `extensions/telegram/src/send.ts`, add an exact benign delete-error classifier for Telegram Bot API no-op delete responses such as `message to delete not found`, `message can't be deleted` / `message cannot be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`. Use the existing `requestWithDiag` `shouldLog` option so `withTelegramApiErrorLogging` does not emit ERROR-level logs for those benign delete failures, catch them, log only verbose context, and return `{ ok: false, warning }`; preserve throws for auth, token, network, permission, and unexpected errors. Update `extensions/telegram/src/action-runtime.ts` so the `deleteMessage` action reports `{ ok: false, deleted: false, warning }` or the locally preferred soft-result shape instead of unconditionally `{ ok: true, deleted: true }` when the helper returns a warning. Add focused tests in the Telegram plugin for all benign strings, one non-benign rethrow, and action-runtime warning propagation. Add a single-line user-facing `CHANGELOG.md` fix entry thanking the issue reporter, following current changelog policy. Do not move this into core, do not broadly classify all Telegram 400s as soft, and do not change unrelated preview cleanup behavior.
