## Summary

Keep open. Current main still lets benign Telegram delete-message 400s pass through the shared logged/rethrow path, while the same plugin already has fail-soft precedent for analogous Telegram action errors and the related fix PRs are closed unmerged.

## What This Changes

The issue requests that Telegram message deletion treat already-gone or non-deletable message 400s as soft no-op results instead of noisy hard failures.

## Best Possible Solution

Patch only the Telegram plugin action path: classify benign delete no-op errors, suppress their ERROR-level log, return a structured warning, propagate `deleted: false`, and keep non-benign failures hard.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by injecting an `api.deleteMessage` mock that rejects with `400: Bad Request: message can't be deleted` and observing the current `requestWithDiag` log/rethrow path; the reported CLI command reaches the same action branch.

## Solution Assessment

Yes. A narrow Telegram plugin fail-soft path is the best maintainable fix; changing shared retry or API logging semantics would risk hiding unrelated Telegram failures.

## Repair Work Prompt

Fix openclaw/openclaw#73726 in the Telegram plugin only. In `extensions/telegram/src/send.ts`, add a small benign-delete-error classifier for Telegram delete no-op failures such as `message to delete not found`, `message can't be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`; use the existing `requestWithDiag` `shouldLog` option so these benign failures do not emit ERROR-level Telegram API logs; return `{ ok: false, warning }` from `deleteMessageTelegram` while preserving hard failures for auth, network, and other non-benign errors. In `extensions/telegram/src/action-runtime.ts`, propagate the soft result as a JSON action result with `deleted: false` and the warning instead of always reporting `deleted: true`. Add focused coverage in the Telegram tests for successful delete, benign delete warnings, non-benign rethrow, and action-runtime warning propagation. Add a user-facing CHANGELOG.md fix entry. Do not change shared Telegram retry/logging semantics outside the narrow call-site seam, and do not touch unrelated channel behavior.
