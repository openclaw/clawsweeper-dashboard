## Summary

Keep open: the duplicate completion-delivery problem is still plausible and not fully implemented on current main, but this PR is not mergeable as-is because it has blocking functional defects, lacks real behavior proof, and overlaps the open completion-delivery work in https://github.com/openclaw/openclaw/pull/64416.

## What This Changes

The PR adds an env-flagged CompletionDeliveryGate, ACP child-session silent wake handling, requester-origin and Discord child-thread routing changes, and related tests.

## Best Possible Solution

Consolidate on a narrower completion-ownership design that claims only after real delivery outcomes, preserves cleanup and retry paths, keeps shadow mode observational, and reconciles this branch with current main and https://github.com/openclaw/openclaw/pull/64416.

## Reproduction Assessment

Yes, source-reproducible but not live-run here. Current main still lacks a shared completion gate and tests still cover visible ACP completion banners, while the PR defects follow directly from the announce lifecycle and the diff.

## Solution Assessment

No. The current PR is not the best fix as-is because it claims completion before delivery succeeds, can bypass cleanup, changes shadow-mode behavior, and bundles Discord/requester-origin routing into the gate rollout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Claim announce completion after delivery succeeds:** `src/agents/subagent-announce.ts:231-235`
  - body: The gate claim is taken before the announce flow has waited for output or delivered anything. This function can still return false later for deferred or failed announce paths, but the retained claim can make fallback paths treat the run as already delivered and leave the requester with no completion message.
  - confidence: 0.9
- **[P1] Keep gate short-circuit inside cleanup path:** `src/agents/subagent-announce.ts:236-238`
  - body: When another path already claimed the completion, this early return exits before the function enters its try/finally. That skips label patching and session deletion while callers interpret the result as a successful announce with no retry path.
  - confidence: 0.88
- **[P2] Keep shadow mode observational for farewells:** `src/agents/subagent-registry-completion.ts:82-83`
  - body: isCompletionClaimed can return true in shadow mode because claims are stored, so this branch flips sendFarewell to false even though shadow rollout is supposed to log would-be blocks without changing user-visible behavior.
  - confidence: 0.86
- **[P1] Preserve parent-channel lookup for child thread bindings:** `extensions/discord/src/monitor/thread-bindings.manager.ts:641`
  - body: For child placement, conversationId can still be a thread target when parentConversationId is absent. Assigning it directly to channelId makes the binder create a child thread under a thread channel, which Discord rejects; keep the thread-to-parent resolution fallback.
  - confidence: 0.84
