## Summary

Keep open: current main still lacks the requested completion-delivery gate and still has source/tests showing visible ACP completion banner delivery, so the PR is not obsolete. The branch should not merge as-is because the proposed gate claims completion ownership before successful announce delivery, can skip announce cleanup on dedupe short-circuit, changes shadow-mode user-visible behavior, and bundles a Discord thread-binding change with its own regression risk while overlapping open PR #64416.

## What This Changes

The PR adds an env-flagged CompletionDeliveryGate across task-registry, subagent announce, and farewell-hook completion paths, adds ACP silent-wake policy/tests, and changes requester-origin plus Discord child-thread binding behavior.

## Best Possible Solution

Revise or split the branch into a narrower completion-ownership design that claims after actual successful delivery, preserves cleanup and retry behavior, keeps shadow mode observational, and separately reviews the Discord/requester-origin routing changes while reconciling the overlap with #64416.

## Reproduction Assessment

Yes. Current-main source and tests show the pre-existing visible ACP banner path, and the PR regressions are directly traceable from the supplied diff: claim-before-delivery, early return before cleanup, and shadow-mode farewell suppression. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. A central gate is a plausible direction, but this implementation claims ownership too early and mixes completion coordination with separate Discord routing changes; the safer fix is after-delivery coordination with explicit shadow semantics and isolated channel-routing review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Claim announce completion after successful delivery:** `src/agents/subagent-announce.ts:231-235`
  - body: The gate claim is taken before `runSubagentAnnounceFlow` has waited for output or delivered anything. This function can still return `false` later for deferred or failed announce paths, but the retained claim lets fallback paths treat the run as already delivered, leaving the requester with no completion message.
  - confidence: 0.9
- **[P1] Keep dedupe returns inside announce cleanup:** `src/agents/subagent-announce.ts:236-238`
  - body: When another path already claimed the completion, this `return true` exits before the function enters its `try`/`finally`. That skips cleanup such as label patching and `sessions.delete`, while callers interpret the result as a successful announce and do not retry.
  - confidence: 0.88
- **[P2] Keep shadow mode observational for farewells:** `src/agents/subagent-registry-completion.ts:82-83`
  - body: `isCompletionClaimed` can read claims stored during `OPENCLAW_COMPLETION_GATE=shadow`, so this branch can flip `sendFarewell` to false even though shadow rollout is supposed to log would-be blocks without changing user-visible behavior.
  - confidence: 0.86
- **[P1] Preserve parent-channel lookup for child bindings:** `extensions/discord/src/monitor/thread-bindings.manager.ts:641`
  - body: For child placement, `conversationId` can still be the current thread when `parentConversationId` is absent. Assigning it directly to `channelId` makes `bindTarget` try to create a child thread under a thread, which Discord rejects; keep the thread-to-parent resolution fallback.
  - confidence: 0.84
