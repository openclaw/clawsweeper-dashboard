## Summary

Keep open: current main does not implement the requested completion-delivery gate or ACP silent-wake behavior, and the PR targets a real completion-duplication area, but the current patch has blocking delivery/cleanup regressions and an overlapping open PR #64416.

## What This Changes

The PR adds an env-flagged completion delivery gate and ACP silent-wake path, adjusts requester-origin and Discord child-thread binding behavior, and adds task, subagent, ACP spawn, and Discord tests.

## Best Possible Solution

Land a narrower completion-ownership design that claims only after real delivery outcomes, preserves cleanup and retry paths, keeps shadow mode observational, and reconciles Discord/requester-origin routing with current main and #64416.

## Reproduction Assessment

Yes. Current main source and tests still show ACP child-session terminal completion delivering a visible banner, and the PR regressions are directly traceable from the supplied diff and review comments; I did not run tests because this was a read-only review.

## Solution Assessment

No. The current PR is not the best fix as-is because it can mark completion delivered before delivery succeeds, bypass cleanup, change shadow-mode behavior, and bundle Discord/requester-origin routing changes into the gate rollout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Claim announce completion after delivery succeeds:** `src/agents/subagent-announce.ts:231-235`
  - body: The new gate claim is taken before `runSubagentAnnounceFlow` has waited for output or delivered anything. This function can still return `false` later for deferred or failed announce paths, but the retained claim can make fallback paths treat the run as already delivered and leave the requester with no completion message.
  - confidence: 0.9
- **[P1] Keep gate short-circuit inside cleanup path:** `src/agents/subagent-announce.ts:236-238`
  - body: When another path already claimed the completion, this early `return true` exits before the function enters its `try`/`finally`. That skips cleanup such as label patching and session deletion while callers interpret the result as a successful announce with no retry path.
  - confidence: 0.88
- **[P2] Keep shadow mode observational for farewells:** `src/agents/subagent-registry-completion.ts:82-83`
  - body: `isCompletionClaimed` can return true in shadow mode because claims are stored, so this branch flips `sendFarewell` to false even though shadow rollout is supposed to log would-be blocks without changing user-visible behavior.
  - confidence: 0.86
- **[P1] Preserve parent lookup for child thread bindings:** `extensions/discord/src/monitor/thread-bindings.manager.ts:641`
  - body: For child placement, `conversationId` can still be a thread target when `parentConversationId` is absent. Assigning it directly to `channelId` makes the binder create a child thread under a thread, which Discord rejects; keep the existing thread-to-parent resolution fallback.
  - confidence: 0.84
