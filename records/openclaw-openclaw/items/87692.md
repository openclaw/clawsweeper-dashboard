## Label Justifications

- `P1`: The report affects real isolated cron automation by hanging before useful output and then recording a healthy run despite a failed model trajectory.
- `impact:message-loss`: Cron jobs can deliver no useful agent result while downstream health checks see the run as successful.

## Summary

The issue should stay open: current main has the related Claude CLI watchdog-progress fix from #87546, but source inspection shows the reported GitHub Copilot `anthropic-messages` isolated-cron path can still miss an aborted empty model attempt when computing cron success, and there is no current-main or shipped-release evidence that the Copilot transport hang/status mismatch has been fixed.

## What This Changes

The issue asks OpenClaw to make GitHub Copilot Claude `anthropic-messages` isolated cron hangs fail visibly, trigger the intended provider fallback/error path, and avoid recording successful cron runs when the model trajectory ended in error.

## Best Possible Solution

Fix the Copilot `anthropic-messages` no-progress path and cron terminal-status propagation together, with focused tests plus live Copilot cron proof that an aborted empty model attempt records an error and follows the intended fallback behavior.

## Reproduction Assessment

No live current-main reproduction was run, but reporter artifacts plus current source give a medium-confidence source reproduction of the status/fallback gap for aborted no-output model attempts.

## Solution Assessment

Unclear whether the permanent fix should live primarily in the Copilot transport, embedded failover policy, cron status propagation, or a combination; the required outcome is that terminal model failures cannot be hidden by payload-only cron success logic.
