## Summary

Keep open for maintainer decision: current main and v2026.5.12 deterministically reject non-`openai-curated` `codexPlugins` entries, but the docs and tests describe that as the intentional V1 boundary rather than an already-defined contract violation. I did not find an implementation PR for this exact marketplace allowlist change.

## What This Changes

The issue requests accepting `openai-bundled` and `openai-primary-runtime` Codex marketplaces in `codexPlugins` config and downstream native plugin policy resolution.

## Best Possible Solution

Decide whether native Codex plugins should expand from curated-only V1 support to a named first-party marketplace allowlist; if yes, update the manifest schema, Zod/types, inventory/activation/session/migration paths, docs, and regression tests together.

## Reproduction Assessment

Yes from source inspection: a `codexPlugins` entry with `marketplaceName: "openai-bundled"` is rejected by the manifest schema, then filtered by the Zod parser and policy resolver on current main. I did not run a live gateway because this was a read-only checkout review, but the existing validation test asserts the same rejection path.

## Solution Assessment

Unclear as a direct fix: the allowlist proposal is plausible, but current docs explicitly define curated-only V1 support and preserve other marketplaces for manual review. A maintainer should first confirm the first-party marketplace policy and activation/install safety boundary.
