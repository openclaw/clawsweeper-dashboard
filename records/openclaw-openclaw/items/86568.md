## Summary

Keep open. The patch edits the right pre-run auth cooldown gate, but the provider-only CLI check does not cover the documented canonical Claude CLI runtime path, so the linked auth-provider outage can remain after merge.

## What This Changes

The PR adds an `isCliProvider` guard around the model-fallback auth cooldown gate and adds a regression test for a provider configured under `agents.defaults.cliBackends`.

## Best Possible Solution

Align the pre-run cooldown gate with the resolved runtime or transport owner, skip OpenClaw auth cooldown only for CLI/harness-owned transport, keep normal provider cooldown behavior intact, and cover the documented canonical Claude CLI config.

## Reproduction Assessment

Yes. Source inspection shows current main runs the auth cooldown gate before candidate execution, and the documented Claude CLI setup keeps `candidate.provider` as `anthropic` while selecting `agentRuntime.id: "claude-cli"`; I did not run a live claude-tmux scenario.

## Solution Assessment

No. The patch is in the right area, but checking only `candidate.provider` is not the narrow complete fix because runtime ownership is selected separately from the canonical provider id.
