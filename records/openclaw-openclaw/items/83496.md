## Summary

Keep open: current main and v2026.6.6 still provide explicit auth order, cooldown rotation, and round-robin/session stickiness, but not a selectable primary-first/last-good/load-balance policy. ## What This Changes

Request to add an explicit Codex/OpenAI auth-profile selection policy so stable-primary accounts can stay first while backup accounts remain failover-only.

## Best Possible Solution

Define the policy in core auth routing with compatibility-preserving defaults, fix inherited-profile cooldown persistence, then expose schema/help/docs/UI wording for primary-first semantics.

## Reproduction Assessment

Yes at source level, but not through a live run in this review. Current source and v2026.6.6 show explicit order, cooldowns, round-robin, and session rotation without the requested selectable policy; the new cooldown persistence concern is also traceable through the current source path.

## Solution Assessment

Unclear until maintainers choose the product semantics. A narrow policy enum may be viable, but the safer path is to preserve current defaults, fix inherited-profile cooldown persistence, and then document how primary-first interacts with session overrides and fallback recovery.
