## Summary

Keep open: current main and v2026.6.8 still expose explicit auth ordering, session rotation/stickiness, and cooldown behavior, but no selectable primary-first/last-good/load-balance policy for Codex/OpenAI auth profiles. The request remains a product and config-surface decision rather than an implemented-on-main cleanup or exact duplicate.

## What This Changes

Add an explicit primary-first Codex/OpenAI auth-profile selection mode for stable primary plus cold-backup account setups.

## Best Possible Solution

Define one canonical OpenAI/Codex auth-profile selection policy in the core auth-routing/session layer, keep current behavior as the compatibility default unless maintainers choose otherwise, and split or resolve the external Codex cooldown persistence prerequisite before exposing failover-only UX.

## Reproduction Assessment

Yes for source-level behavior: current main and v2026.6.8 have no auth-profile policy field, and documented behavior is explicit order plus implicit round-robin/session stickiness rather than selectable primary-first routing.

## Solution Assessment

Unclear until maintainers choose semantics; a policy enum may be appropriate, but it adds config/default surface and must account for explicit order, auto-pinned sessions, cooldown sorting, and external Codex profile state.
