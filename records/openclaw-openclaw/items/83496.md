## Summary

Keep open: current main and the latest release still have explicit auth order, round-robin/session rotation, and capped cooldown rotation, but no selectable primary-first/last-good/load-balance policy; the new comment also identifies a source-level cooldown-persistence prerequisite for inherited/external Codex profiles that should be resolved or split before final policy design.

## What This Changes

Request an explicit Codex/OpenAI auth-profile selection mode for stable-primary plus cold-backup account setups instead of relying on implicit order, last-used, session, and cooldown heuristics.

## Best Possible Solution

Define one canonical OpenAI/Codex auth-profile policy in the core auth-routing layer, keep current behavior as the compatibility default unless maintainers choose otherwise, and fix or split the external-profile cooldown persistence prerequisite before exposing primary-first/failover-only UX.

## Reproduction Assessment

Yes for source-level behavior: current main has no policy field and still uses explicit order or round-robin/session rotation. The latest comment's cooldown persistence prerequisite is also source-reproducible from the lock-update path, but I did not run a live multi-account Codex repro.

## Solution Assessment

Unclear until maintainers choose semantics. A narrow policy enum may be the right shape, but the external-profile cooldown persistence path should be repaired or split first so primary-first/failover-only behavior can rely on durable cooldown state.
