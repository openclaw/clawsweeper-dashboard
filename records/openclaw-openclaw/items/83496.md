## Summary

Keep open: current main and the latest release support explicit auth order, session pinning, cooldown rotation, and the linked display fix, but they still do not expose a selectable primary-first/last-good/load-balance auth-profile policy for stable-primary plus cold-backup OpenAI/Codex setups.

## What This Changes

Request an explicit Codex/OpenAI auth-profile selection policy mode so a stable primary account can be preferred over sticky backup or rotation state.

## Best Possible Solution

Define one canonical auth-profile selection policy in the core auth-routing layer, keep current behavior as the compatibility default unless maintainers choose otherwise, and document how primary-first interacts with cooldowns, session overrides, and fallback recovery.

## Reproduction Assessment

No high-confidence bug repro is required because this is a feature/config-policy request. Source inspection confirms current main has explicit order, round-robin/session rotation, and cooldown controls, but no primary-first policy mode.

## Solution Assessment

Unclear until maintainers choose the product semantics. A narrow policy enum may be the right shape, but it must preserve compatibility for existing rotation users and define how auto-selected sessions return to the primary.
