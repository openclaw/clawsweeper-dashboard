## Summary

Keep open: current main and the latest release still provide explicit auth order, cooldown rotation, and session stickiness, but no selectable primary-first/last-good/load-balance policy for Codex/OpenAI auth profiles. Related open items overlap with provider pooling, in-run rotation, and UI exposure, but none owns this specific stable-primary Codex operator policy.

## What This Changes

The issue requests explicit Codex/OpenAI auth-profile selection modes so users can keep a stable primary account ahead of backup accounts unless the primary actually fails.

## Best Possible Solution

Define one canonical OpenAI/Codex auth-profile policy in the core auth/session routing layer, preserve the current default unless maintainers choose a migration, and expose/document it through canonical openai auth profile ids.

## Reproduction Assessment

Yes for the source-level product gap: current main and v2026.6.8 have no auth-profile policy field, and documented behavior is explicit order plus implicit round-robin/session stickiness. This is not a failing strict bug reproduction; it is a reproducible missing feature/config mode.

## Solution Assessment

Unclear until maintainers choose semantics. A policy enum may be the right shape, but it must be designed around existing auth.order precedence, auto-pinned sessions, cooldown sorting, canonical openai migration, and the external Codex cooldown persistence prerequisite.
