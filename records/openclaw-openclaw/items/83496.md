## Summary

Keep open: current main and v2026.6.6 still expose explicit auth order, cooldown rotation, and round-robin/session rotation, but no selectable primary-first/last-good/load-balance policy. The latest issue comment also gives a credible source-level prerequisite around external Codex cooldown persistence, so this needs a maintainer policy decision rather than cleanup closure.

## What This Changes

Request a selectable OpenAI/Codex auth-profile policy for primary-first or failover-only multi-account setups instead of relying on implicit round-robin and persisted selection state.

## Best Possible Solution

Define one canonical OpenAI/Codex auth-profile selection policy in core auth routing, keep the compatibility default explicit, and fix or split the external-profile cooldown persistence prerequisite before exposing UI or config.

## Reproduction Assessment

Yes for source-level behavior: current main and v2026.6.6 have no selectable auth-profile policy and still use explicit order or round-robin/session rotation. I did not run a live multi-account Codex repro.

## Solution Assessment

Unclear until maintainers choose the policy contract. A narrow policy enum may be the right shape, but the external-profile cooldown persistence path should be repaired or split first so primary-first/failover-only behavior can rely on durable cooldown state.
