## Summary

Keep open: current main supports explicit `auth.order`, cooldown-aware rotation, and documented round-robin/session pinning, but it does not expose the requested primary-first/last-good/load-balance auth-profile policy. The related threads cover stale `lastGood`, provider-pool routing, rotation caps, and UI visibility, but none is a clean canonical replacement for this Codex cold-backup policy request.

## What This Changes

The issue requests a new Codex auth-profile selection mode so operators can prefer a stable primary Pro account and use backup accounts only after real failure.

## Best Possible Solution

Add a documented auth-profile selection policy in the auth-routing layer, preserving current behavior by default unless maintainers intentionally approve a migration path, with schema/help/docs and focused tests for primary-first fallback and recovery.

## Reproduction Assessment

No high-confidence bug repro is needed because this is a feature/config-policy request. Source inspection confirms current main has explicit order, cooldown rotation, and round-robin/session behavior, but no primary-first policy mode.

## Solution Assessment

Unclear until maintainers choose the product semantics. A narrow policy enum could be a maintainable fit, but it should preserve compatibility for existing round-robin users and define how auto-selected sessions return to the primary.
