## Summary

Keep open: current main has explicit auth ordering, cooldown sorting, round-robin default ordering, and session auto-rotation, but it does not expose the requested primary-first/last-good/load-balance/failover-only policy mode. The related open reports cover stale lastGood recovery, same-provider rotation caps, UI exposure, or a broader pool-routing design, but none is a precise replacement for this Codex cold-backup account policy.

## What This Changes

The issue requests an opt-in Codex/OpenAI auth-profile selection mode that prefers a stable primary account before using backup profiles.

## Best Possible Solution

Add a documented, compatibility-preserving auth-profile selection policy in the auth-routing layer, with schema/help/docs and focused tests for primary-first fallback and recovery behavior.

## Reproduction Assessment

No high-confidence bug reproduction is needed because this is a feature/config-policy request. Source inspection confirms current main lacks a primary-first policy surface and instead uses explicit order, round-robin ordering, cooldowns, and session auto-rotation.

## Solution Assessment

Unclear until maintainers choose the product semantics. A narrow opt-in policy enum looks maintainable, but it should preserve existing round-robin behavior unless maintainers intentionally choose a migration path.
