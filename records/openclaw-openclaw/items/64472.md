## Summary

Keep open: current main and v2026.6.10 still read fixed workspace bootstrap files through a workspace-root fs-safe boundary that rejects symlink hops resolving outside that root. A maintainer explicitly kept this issue open after closing the old local-resolver PR as obsolete, and the remaining work is a security/product policy decision rather than conservative cleanup.

## What This Changes

The issue requests loading fixed workspace bootstrap files that are symlinked from the workspace into sibling directories under the OpenClaw home instead of rendering them as missing.

## Best Possible Solution

Define a narrow external-bootstrap symlink policy in the active fs-safe/OpenClaw wrapper path, then land tests that allow intended shared bootstrap files while rejecting private OpenClaw state roots.

## Reproduction Assessment

Yes. Source inspection shows current main reads fixed bootstrap files through openRootFile rooted at the workspace directory, fs-safe rejects symlink targets outside that root, and the loader renders non-ok reads as missing.

## Solution Assessment

No. The broad proposed allowance for OpenClaw-home sibling targets is not the best fix by itself; maintainers need a deliberate allowed/denied target policy in the active fs-safe or wrapper path with regression coverage.
