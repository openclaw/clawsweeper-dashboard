## Summary

Keep open: current main still reads fixed workspace bootstrap files through a workspace-root `openRootFile` boundary and renders rejected reads as missing. The requested `.openclaw` sibling symlink exception remains a security/product policy decision and is not fully owned by the open broader symlink PR.

## What This Changes

Request to allow fixed workspace bootstrap files symlinked from the workspace to selected sibling `~/.openclaw` directories to load instead of being rendered as `[MISSING]`.

## Best Possible Solution

Land a maintainer-approved external-bootstrap symlink policy in the active fs-safe/OpenClaw reader path, with tests for allowed shared files and rejected private OpenClaw state roots.

## Reproduction Assessment

Yes, at source level. Current main reads fixed bootstrap files through `openRootFile` bounded to the workspace root, fs-safe rejects symlink hops outside that root, and non-ok reads are rendered as missing.

## Solution Assessment

No, the broad proposed allowance for any target inside `~/.openclaw` is not the best fix by itself. The safer path is a narrow maintainer-approved allow/deny policy in the active reader layer with regression coverage for private OpenClaw state roots.
