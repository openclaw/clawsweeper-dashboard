## Summary

Keep open: current main and the latest release still read fixed workspace bootstrap files through a workspace-root fs-safe boundary, so symlink targets that canonicalize outside the workspace are still rendered as missing. A maintainer explicitly kept this issue open after closing the old boundary-path PR as obsolete, and the remaining fix needs a product/security decision about which OpenClaw-home targets may become startup prompt context.

## What This Changes

The issue asks OpenClaw to allow workspace bootstrap-file symlinks that point to selected OpenClaw-home sibling directories instead of treating them as missing.

## Best Possible Solution

Define a narrow external-bootstrap symlink policy in the active fs-safe/OpenClaw wrapper path, preserve rejection of private OpenClaw state roots, add focused regression tests, and close this issue only after that fix merges.

## Reproduction Assessment

Yes. Source inspection shows current main reads fixed bootstrap files through openRootFile rooted at the workspace directory and renders non-ok guarded reads as missing; I did not run a live filesystem repro because this review is read-only.

## Solution Assessment

No, the proposed broad OpenClaw-home allowance is not the best fix. The maintainable solution needs a narrow symlink-hop policy with explicit private-state exclusions and tests in the active fs-safe/OpenClaw wrapper path.
