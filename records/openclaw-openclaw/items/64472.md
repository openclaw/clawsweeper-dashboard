## Summary

Current main still rejects bootstrap symlinks whose canonical targets leave the workspace, and a repository member explicitly kept this security-sensitive issue open for a narrower active-owner fix; the linked maintainer PR remains open but conflicted, so no close reason applies.

## What This Changes

Allow fixed workspace bootstrap files to follow selected symlinks outside the workspace while preserving general filesystem-boundary protections.

## Best Possible Solution

Refresh the scoped bootstrap-reader implementation against current main so only explicit fixed bootstrap entries may follow approved symlink hops, while direct arbitrary external paths, extra/glob bootstrap paths, hardlinks, dangling links, non-regular files, and oversized targets remain rejected.

## Reproduction Assessment

Yes at source level: fixed bootstrap entries are opened under the workspace root, and canonical targets outside that root become validation failures that the loader reports as missing. The issue supplies a deterministic symlink layout, although this read-only review did not run a live filesystem reproduction.

## Solution Assessment

No, the proposed blanket fallback for the entire OpenClaw state root is broader than the demonstrated need. A fixed-bootstrap-only symlink policy at the active fs-safe or workspace-wrapper boundary is the safer maintainable solution.
