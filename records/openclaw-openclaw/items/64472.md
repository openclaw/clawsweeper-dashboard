## Summary

Keep open: current main still reads fixed bootstrap files through a workspace-root `openRootFile` boundary and turns any rejected read into `missing: true`, while the fs-safe dependency rejects symlink hops that resolve outside that root. A maintainer explicitly kept this issue open after closing the old boundary-path PR as obsolete, and the open maintainer PR is only a partial/conflicting candidate rather than a merged fix.

## What This Changes

Allow workspace bootstrap symlinks under `~/.openclaw/workspace` to target selected sibling `~/.openclaw` subdirectories without being reported as missing.

## Best Possible Solution

Keep this issue open until a maintainer-approved policy is implemented against the active fs-safe/OpenClaw wrapper boundary, with tests proving allowed symlink-hop targets and rejection of direct arbitrary `~/.openclaw` paths.

## Reproduction Assessment

Yes, at source level: current main reads fixed bootstrap files with `openRootFile` bounded to the workspace root, and fs-safe rejects symlink hops outside that root. I did not run a live filesystem repro because this review is constrained to read-only inspection.

## Solution Assessment

No, the originally proposed broad `~/.openclaw` fallback is not the best fix because it can widen direct path access as well as symlink hops. The better solution is a narrow, maintainer-approved symlink-hop policy in the active fs-safe/OpenClaw wrapper path with security regression tests.
