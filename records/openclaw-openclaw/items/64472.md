## Summary

Keep open: current main and the latest release still route fixed workspace bootstrap reads through a workspace-root fs-safe boundary, and fs-safe rejects symlink hops whose canonical target leaves that root. A maintainer explicitly kept this issue open after closing the old local-boundary PR as obsolete, and the open replacement PR only partially overlaps this `.openclaw` trust-policy question.

## What This Changes

The issue asks OpenClaw to load workspace bootstrap files that are symlinks from a workspace into another subdirectory under `~/.openclaw` instead of treating them as missing.

## Best Possible Solution

Keep this issue open until maintainers settle a narrow external-bootstrap symlink policy and land tests that allow intentional shared files while rejecting private OpenClaw state roots.

## Reproduction Assessment

Yes, at source level: current main reads bootstrap files through `openRootFile` bounded to the workspace root, and fs-safe rejects symlink hops that resolve outside that root. I did not run a live filesystem repro because this review is constrained to read-only inspection.

## Solution Assessment

No, the originally suggested broad `~/.openclaw` allowance is not the best fix. A safer fix needs a maintainer-approved symlink-hop policy in the active fs-safe/OpenClaw wrapper path with tests for allowed shared files and rejected private state paths.
