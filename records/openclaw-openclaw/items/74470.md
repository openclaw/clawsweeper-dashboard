## Summary

Keep open. This PR is protected by the `maintainer` label and still has blocking review work: the fallback override is wired into timeout compaction but not the separate overflow compaction context, and the new fallback classifier can drop provider status metadata before deciding whether to retry.

## What This Changes

The branch threads model fallback overrides through agent, auto-reply, and embedded compaction paths, wraps implicit compaction in model fallback, updates compaction docs and changelog, and adds focused regression tests.

## Best Possible Solution

Update this branch so implicit compaction receives the same effective fallback override on both timeout and overflow recovery paths, preserve provider status/code when classifying compaction failures, and add an overflow-focused regression test while leaving broader circuit-breaker or force-truncate behavior to the existing narrower follow-up discussions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Thread fallback override into overflow compaction:** `src/agents/pi-embedded-runner/run.ts:1197`
  - body: This passes `params.modelFallbacksOverride` into timeout-recovery compaction, but the overflow-recovery `buildEmbeddedCompactionRuntimeContext` later in `runEmbeddedPiAgent` still omits it. Overflow recovery is the path described by #64960; when the active run has an effective override such as `[]` from a user model switch, compaction can fall back to config/session defaults and use models the run intentionally disabled. Pass the same override into the overflow context and cover it in the overflow compaction tests.
  - confidence: 0.86
- **[P2] Preserve status metadata when classifying compaction failures:** `src/agents/pi-embedded-runner/compact.ts:362`
  - body: `compactEmbeddedPiSessionDirectOnce` catches provider errors and returns only `formatErrorMessage(err)`, but this classifier rebuilds `new Error(reason)`, dropping `.status` and `.code`. A fallback-eligible 400 content-filter rejection whose message does not start with `400` will be treated as a successful `{ ok: false }` result by `runWithModelFallback` and skip the fallback chain. Preserve the status/code in the compaction result or classify the original error before serializing it.
  - confidence: 0.78
