## Summary

Keep open. Current main still performs one SearXNG request with the requested category and returns a successful empty result set; generic web-search fallback only advances on unavailable providers, thrown execution errors, or structured missing-key errors. A GitHub search found no paired open PR or canonical duplicate implementing this behavior.

## What This Changes

The issue requests retrying a category-specific web search as a general web search when the category request returns zero results.

## Best Possible Solution

Add a SearXNG-provider-local retry: after an empty successful response for a non-general categorized request, retry once with categories set to general, while preserving endpoint validation, timeout, cache TTL behavior, language, count limits, content wrapping, and existing error handling.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A deterministic mocked SearXNG response where categories is weather and the JSON body is {"results":[]} should show current main issuing one weather request and returning count 0 without a second general request.

## Solution Assessment

Is this the best way to solve the issue? Yes. A provider-local retry is the narrowest maintainable solution because generic runWebSearch should continue treating successful zero-result payloads as valid for other providers.
