## Summary

Keep open. Current main still sends one categorized SearXNG request and returns a successful empty payload without retrying `general`; the core web-search fallback only advances on provider unavailability, thrown failures, or structured missing-key errors, so the earlier ClawSweeper keep-open comment still matches the current code.

## What This Changes

Request fallback for category-specific web searches by retrying the same query with general search when the categorized search returns no results.

## Best Possible Solution

Add a SearXNG-provider-local retry: after an empty categorized response, retry once with `categories` set to `general` when the effective category set is non-empty and does not already include `general`, while preserving endpoint validation, timeout, cache TTL, language, count, wrapping, and existing error behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mocked `runSearxngSearch` call with `categories: "weather"` and a SearXNG JSON response of `{ "results": [] }`; current code builds one categorized request and returns `count: 0` without a second general request.

## Solution Assessment

Yes. The narrowest maintainable solution is provider-local because zero results can be a valid successful search result in the generic provider loop, while this report is specifically about SearXNG category-engine compatibility.
