## Summary

Keep open. Current `main` still performs a single SearXNG category request and returns a successful zero-result payload without retrying `general`; the existing core provider fallback only moves to another provider for unavailable providers, thrown errors, or structured missing-key availability errors.

## What This Changes

Add a SearXNG `web_search` fallback that retries the same query with the general category when a category-specific search returns no results.

## Best Possible Solution

Implement the retry in the SearXNG plugin client: after an empty categorized response, retry once with `categories` set to `general` when the effective category set is non-empty and does not already include `general`, while preserving timeout, endpoint validation, caching, count, language, wrapping, and existing error behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mocked `runSearxngSearch` call with `categories: "weather"` where the SearXNG JSON response is `{ "results": [] }`; current code builds one categorized URL and returns `count: 0` without a second general request.

## Solution Assessment

Yes. A provider-local fallback is the narrowest maintainable fix because zero results can be a valid successful result in the generic provider loop, while this report is specifically about SearXNG category-engine compatibility.
