## Summary

Keep open. Current main still treats an empty successful SearXNG category response as final, and the generic web-search fallback only advances on unavailable providers, thrown errors, or structured missing-key payloads. This is a narrow provider-local reliability fix with clear implementation and validation paths.

## What This Changes

The issue requests retrying category-specific SearXNG web searches with the general category when the categorized response has no results.

## Best Possible Solution

Add a SearXNG client retry that runs once with categories set to general after an empty categorized response when the effective category set is non-empty and does not already include general, preserving existing endpoint validation, timeout, cache, count, language, wrapping, and error behavior.

## Reproduction Assessment

Yes. A source-level reproduction is a mocked runSearxngSearch call with categories: "weather" where the SearXNG JSON response is {"results":[]}; current code builds one categorized URL and returns a zero-count payload without a general retry.

## Solution Assessment

Yes. A provider-local fallback is the narrowest maintainable solution because generic zero-result web-search responses can be legitimate and should not automatically trigger cross-provider fallback.

## Repair Work Prompt

Fix the SearXNG category fallback requested by https://github.com/openclaw/openclaw/issues/73552. Implement the retry inside the SearXNG plugin client, not in the generic web-search runtime: after an empty response for an effective category set that is non-empty and does not already include `general`, retry the same query once with `categories=general`. Preserve endpoint validation, timeout, count, language, cache TTL behavior, result wrapping, existing error behavior, and the current provider payload shape unless a small metadata field is clearly needed. Do not disable category support and do not make generic `runWebSearch` treat zero results as provider failure. Add focused tests for categorized empty response followed by general results, no retry when results are present, no retry when categories already include `general`, and no retry when no category is configured. Update SearXNG docs if the fallback behavior should be user-visible, and add a user-facing `CHANGELOG.md` Fixes entry crediting `Thanks @Loukky`.
