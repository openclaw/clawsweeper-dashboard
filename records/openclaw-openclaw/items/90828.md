## Summary

Keep open. The launchd stderr restoration is useful, but this PR is not merge-ready because `status --all` still skips Darwin stderr, public docs still describe macOS stderr as suppressed, and the external PR only provides unit-test proof for a macOS LaunchAgent behavior.

## What This Changes

The PR changes macOS launchd gateway stderr from `/dev/null` to the resolved `gateway.err.log` path and updates daemon status, runtime hints, diagnostics, and tests around that logging contract.

## Best Possible Solution

Align every macOS launchd diagnostic surface with the restored stderr contract, update the public docs, and require redacted real macOS LaunchAgent proof before deciding whether to merge ahead of log retention.

## Reproduction Assessment

Yes for the source-level bug: current main and `v2026.6.8` still pass `/dev/null` as `StandardErrorPath`, and the linked issue includes live macOS comments showing diagnostics lost to that path. I did not run a real macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

No, not yet as the best merge shape. The plist and `readLastGatewayErrorLine` direction is plausible, but the PR leaves `status --all` and public docs on the old stderr-suppressed contract and lacks real macOS upgrade proof.
