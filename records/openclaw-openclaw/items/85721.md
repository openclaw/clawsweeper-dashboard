## Label Justifications

- `P1`: The report describes browser automation process accumulation that can consume gigabytes of memory and destabilize a user host.
- `impact:crash-loop`: The affected failure mode is process-level availability pressure from unreaped Chrome MCP subprocesses.

## Summary

Keep open: current main only contains the telemetry-watchdog opt-out, while Chrome MCP sessions in the browser plugin still tear down through SDK `client.close()` without owned subprocess-tree cleanup. The linked open PR is the right fix path, so this issue should remain open until that PR or an equivalent cleanup merges.

## What This Changes

The issue requests browser automation cleanup so repeated Chrome DevTools MCP sessions do not leave `npx`, `chrome-devtools-mcp`, or watchdog subprocesses behind until host memory is exhausted.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85832, or an equivalent browser-plugin cleanup that owns the Chrome MCP subprocess tree, then close this issue from the merged fix.

## Reproduction Assessment

Yes at source level: current main still closes Chrome MCP sessions through `client.close()`, while the SDK transport only signals the direct spawned process. I did not run the reporter's multi-hour Linux process-count loop in this read-only review.

## Solution Assessment

Yes: keeping this issue open for the linked browser-plugin cleanup PR is the narrow maintainable path. A long-lived extension bridge like the FSB reference may be useful future architecture, but it is not required to fix this leak.
