- `P1`: The report describes real browser automation process accumulation that can exhaust memory and destabilize a user host.

## Summary

Keep open: current main still launches Chrome DevTools MCP through the browser plugin with `npx chrome-devtools-mcp@latest` and closes sessions through `client.close()`, while the dependency only terminates the direct stdio child and Chrome DevTools MCP can spawn a detached telemetry watchdog. Related sidecar work fixed a Codex app-server path, but this browser-plugin cleanup path still needs a focused repair.

## What This Changes

The issue reports accumulating `chrome-devtools-mcp` and watchdog process trees after repeated browser automation, causing multi-GB memory pressure.

## Best Possible Solution

Give the browser plugin explicit ownership of Chrome MCP subprocess cleanup, including the `npx` wrapper and watchdog descendants, with regression coverage and live process-count proof.

## Reproduction Assessment

Yes at source level: current main still closes Chrome MCP sessions through `client.close()`, while the MCP SDK only terminates its direct stdio child and Chrome DevTools MCP can spawn a detached watchdog. I did not run a live multi-hour process-count reproduction.

## Solution Assessment

No, the current cleanup path is not sufficient for the reported process tree. The narrow maintainable fix is browser-plugin-owned subprocess lifecycle cleanup rather than relying on a global install or the MCP protocol close alone.
