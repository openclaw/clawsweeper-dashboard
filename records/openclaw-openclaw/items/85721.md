## Summary

Keep open: current main has a partial telemetry-watchdog mitigation, but the browser plugin still closes Chrome MCP sessions through `client.close()` without owning the full subprocess tree. The remaining cleanup is already owned by the open linked PR at https://github.com/openclaw/openclaw/pull/85832, so this issue should stay open until that PR merges or is superseded.

## What This Changes

The issue reports repeated browser automation leaving `chrome-devtools-mcp` and watchdog subprocesses running after browser sessions close.

## Best Possible Solution

Review and land the linked browser-plugin cleanup PR, or replace it with an equivalent narrow fix that owns the `npx -> chrome-devtools-mcp -> watchdog/helper` tree and includes process-tree regression coverage plus live process-count proof.

## Reproduction Assessment

Yes, source-level: current main still launches Chrome MCP through `StdioClientTransport` and closes sessions with `client.close()`, while upstream stdio close only owns the direct child and Chrome DevTools MCP can spawn a detached watchdog. I did not rerun a live multi-hour Linux process-count reproduction in this read-only sweep.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: make the browser plugin own Chrome MCP subprocess cleanup instead of relying on protocol close or a global install. The issue should remain open until that fix or an equivalent one lands.
