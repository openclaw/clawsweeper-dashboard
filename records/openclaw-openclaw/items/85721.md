## Label Justifications

- `P1`: The report describes repeated browser automation leaving many subprocesses alive until host memory and swap are exhausted.
- `impact:crash-loop`: The affected failure mode is process-level availability pressure from unreaped Chrome MCP subprocesses.

## Summary

Keep open: current main still routes browser-plugin Chrome MCP sessions through `npx chrome-devtools-mcp@latest` and relies on MCP `client.close()` for teardown, while the pinned MCP SDK only owns the direct stdio child and Chrome DevTools MCP can create detached watchdog descendants. The related configured-MCP and Codex app-server cleanup work does not cover this browser-plugin session lifecycle path.

## What This Changes

The issue reports that repeated browser automation leaves `chrome-devtools-mcp`, `npx`, and watchdog subprocesses alive after Chrome MCP session close, eventually exhausting memory.

## Best Possible Solution

Give the browser plugin explicit ownership of Chrome MCP subprocess teardown, including the `npx` wrapper and detached watchdog descendants, while keeping the process helper inside a valid plugin/SDK boundary and proving bounded process counts after repeated real browser operations.

## Reproduction Assessment

Yes at source level: current main still launches Chrome MCP through `npx chrome-devtools-mcp@latest` and closes sessions through MCP `client.close()`, while the dependency and Chrome DevTools MCP sources leave descendant watchdog cleanup outside that contract. I did not run a live multi-hour process-count reproduction.

## Solution Assessment

No, the current cleanup path is not the best solution because it delegates full subprocess lifecycle to a protocol close that only owns the direct stdio child. The narrow maintainable fix is browser-plugin-owned process-tree cleanup, not a global install requirement or manual `pkill` workaround.
