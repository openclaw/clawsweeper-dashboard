## Summary

Keep open: current main has the merged telemetry-watchdog opt-out, but it still uses the MCP SDK stdio transport directly and closes Chrome MCP sessions through `client.close()` rather than owning the spawned subprocess tree. The linked open repair PR uses closing syntax for this issue and is the right path to review or land before closing.

## What This Changes

The issue reports that repeated browser automation leaves `chrome-devtools-mcp` and watchdog subprocesses running after session cleanup, causing host memory pressure.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85832, or replace it with an equivalent browser-plugin-owned process-tree cleanup that preserves the telemetry opt-out and includes process-count proof.

## Reproduction Assessment

Yes at source level: current main still tears down Chrome MCP sessions with `client.close()`, while the MCP SDK close contract only owns the direct stdio child. I did not run a fresh multi-hour process-count loop against current main.

## Solution Assessment

Yes: the best narrow fix is browser-plugin-owned cleanup for the Chrome MCP process tree plus the already-merged telemetry opt-out, not a global install requirement or broad bridge rewrite.
