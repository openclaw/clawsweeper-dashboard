## Summary

Keep open: current main has xAI plugin tools, but source inspection shows a credential-source mismatch that can make a valid xAI model/auth profile fail to expose `x_search` and `code_execution`, while `web_search` can fall through to another provider.

## What This Changes

The issue asks OpenClaw to expose xAI-backed `x_search`, `code_execution`, and Grok `web_search` when a valid Grok model and xAI credential are configured.

## Best Possible Solution

Align xAI tool availability and execution with the same credential sources used for xAI model auth, or explicitly document and enforce a separate tool credential if maintainers want that separation.

## Reproduction Assessment

Yes, source-level: with an xAI auth profile but no `XAI_API_KEY` or `plugins.entries.xai.config.webSearch.apiKey`, manifest availability can select the xAI plugin while the xAI tool factory returns `null`. I did not run a live WSL2/xAI repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to make xAI plugin tool registration and execution use the existing provider-auth/profile helpers so the factory gate and manifest auth signal agree, while preserving current env and plugin-config paths.
