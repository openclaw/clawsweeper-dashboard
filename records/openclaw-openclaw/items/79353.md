## Summary

Keep open: the current xAI plugin declares and registers `x_search` and `code_execution`, but source inspection shows a likely remaining auth-path gap when the user has valid xAI model auth without `XAI_API_KEY` or `plugins.entries.xai.config.webSearch.apiKey`.

## What This Changes

The issue asks for the xAI provider to make `x_search` and `code_execution` available when a valid Grok model and xAI API key are configured.

## Best Possible Solution

Thread a generic provider-auth resolver or active auth-profile credential into plugin tool construction, then update the xAI tool auth helper to reuse that provider auth while preserving existing `webSearch`, legacy Grok, and `XAI_API_KEY` fallbacks.

## Reproduction Assessment

No live high-confidence run was performed. Source inspection gives a credible reproduction path: configure xAI through provider auth/profile/model config so `xai/grok-4.3` works, leave `XAI_API_KEY` and `plugins.entries.xai.config.webSearch.apiKey` unset, and the xAI tool factories still cannot resolve a key.

## Solution Assessment

Yes, the narrow maintainable fix is to make plugin-owned xAI tools reuse the same provider auth that makes the xAI model usable. Changing the selected `web_search` provider alone would not address `x_search` or `code_execution` registration.
