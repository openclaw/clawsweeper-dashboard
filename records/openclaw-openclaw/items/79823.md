## Summary

Keep open: current main does not implement this core Tool Search feature, and the PR still has blocking source-level issues around catalog scoping, ambiguous name lookup, bridged error semantics, and client-only allowlist prechecks. The supplied terminal proof covers the happy-path gateway E2E, but not these edge cases.

## What This Changes

The PR adds core PI Tool Search with code and structured bridge tools, `tools.toolSearch` config/docs/tests, catalog compaction for OpenClaw/MCP/client tools, and a gateway E2E harness.

## Best Possible Solution

Revise this PR so Tool Search uses strictly scoped per-run catalogs, rejects ambiguous tool identities, preserves direct-tool adapter semantics, and accounts for compacted client tools before allowlist prechecks; then rerun focused tests plus the gateway E2E proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows the singleton catalog fallback, first-match name lookup, raw bridged execution path, and precheck-before-client-catalog ordering. The feature itself is new capability work, not a current-main bug reproduction.

## Solution Assessment

No. Core Tool Search may be a reasonable direction, but this implementation should not merge until the bridge fails closed, uses deterministic tool identity, and preserves current direct-tool execution semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Count client catalog entries before allowlist precheck:** `src/agents/pi-embedded-runner/run/attempt.ts:1251-1255`
  - body: The empty explicit allowlist precheck runs before client tools are added to the Tool Search catalog, and it strips auto-added Tool Search controls while counting only the initial non-client catalog size. With an explicit allowlist whose callable tools are client-only, this can report no callable tools and skip prompt submission even though the client tools would become callable later through `addClientToolsToToolSearchCatalog`.
  - confidence: 0.86
- **[P2] Fail closed without a scoped catalog:** `src/agents/tool-search.ts:653-657`
  - body: `resolveCatalog` still returns the only process-global catalog after session, session key, and agent lookup miss. A Tool Search control with missing or stale context can therefore search or call another run's catalog; require a scoped match instead of falling back globally.
  - confidence: 0.84
- **[P2] Reject ambiguous tool-name lookups:** `src/agents/tool-search.ts:720-722`
  - body: `findEntry` accepts a catalog id or raw tool name and returns the first match, but client and plugin tools can legitimately share names. If the model passes a name to `tool_call` or `openclaw.tools.call`, the bridge can invoke the wrong implementation; require ids or fail on duplicate names.
  - confidence: 0.85
- **[P2] Preserve direct-tool error results for bridged calls:** `src/agents/pi-embedded-runner/run/attempt.ts:2558-2565`
  - body: The catalog executor calls the raw `tool.execute` path, so failures that direct PI tools convert into structured `status: "error"` results can instead reject `tool_search_code` or `tool_call`. Route bridged calls through the same adapter semantics, or normalize non-abort errors before returning through the bridge.
  - confidence: 0.8
