## Best Possible Solution

Keep the compact catalog/search direction, but merge only after the code bridge has an enforceable sandbox or structured executor plus session-scoped catalog lookup, cancellable tool execution, direct-path error semantics, and per-call tool IDs.

## Solution Assessment

No; the compact bridge is a plausible direction, but this implementation still exposes child-process capabilities and bypasses important tool runtime semantics. A hardened executor or non-JS structured mode should preserve normal OpenClaw tool behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Do not expose host bridge functions to VM code:** `src/agents/tool-search-code-mode.ts:160-162`
  - body: The child-process rewrite still passes outer-realm bridge functions into the VM as openclaw.tools.*. Code running in the VM can use openclaw.tools.search.constructor("return process.version")() to reach the child Node process, so model-authored code still gets process capability despite the claimed isolated runtime.
  - confidence: 0.96
- **[P1] Cancel host tool calls on code-mode timeout:** `src/agents/tool-search-code-mode.ts:648-653`
  - body: The timeout kills only the child process. If model code is awaiting openclaw.tools.call(...), the parent-side entry.tool.execute(...) promise keeps running without an AbortSignal, so side-effecting tools can continue after tool_search_code has already failed with a timeout.
  - confidence: 0.93
- **[P2] Generate unique IDs for bridged tool calls:** `src/agents/tool-search-code-mode.ts:652`
  - body: Every bridged call to the same catalog entry uses tool_search_code:<entry.name>. Client tools are recorded by toolCallId in the embedded runner, so two calls to the same client tool in one code-mode run can collapse into one recorded completion and lose earlier invocations.
  - confidence: 0.93
- **[P2] Fail closed when the session catalog is missing:** `src/agents/tool-search-code-mode.ts:481-487`
  - body: When lookup by sessionId/sessionKey/agentId misses, resolveCatalog falls back to the only global catalog in memory. Because the tool context fields are optional, a call with missing context can bind to another active run's catalog instead of failing closed.
  - confidence: 0.88
- **[P2] Return structured tool errors from bridged calls:** `src/agents/tool-search-code-mode.ts:648-653`
  - body: ToolSearchRuntime.call lets catalog tool exceptions escape, so a validation or runtime error aborts tool_search_code/tool_call. The direct toToolDefinitions path converts non-abort failures into structured status:error results, which lets the model recover from a failed tool call.
  - confidence: 0.9

  - body: The VM receives host-created bridge functions; a constructor probe reaches process.version from inside the model-authored code context, so the child process is still exposed to untrusted code.

  - body: If a tool context lacks session identifiers, resolveCatalog can return the sole in-memory catalog from another run, exposing that run's tool catalog to the wrong context.
  - confidence: 0.88
