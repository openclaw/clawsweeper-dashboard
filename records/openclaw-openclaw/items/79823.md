## Summary

Keep open: this is a protected-label feature PR, and the proposed code-mode execution path has blocking security and timeout findings that need maintainer review before merge.

## What This Changes

The branch adds a bundled Tool Search Code Mode plugin that compacts large OpenClaw, MCP, and client tool catalogs behind `tool_search_code`, plus agent/gateway integration, docs, tests, QA mock support, and an E2E script.

## Best Possible Solution

Move `tool_search_code` execution behind a real OpenClaw sandbox or hardened worker boundary with enforceable cancellation, then keep the compact catalog/search API layered on that boundary.

## Reproduction Assessment

Not applicable for the PR itself because it adds a new feature rather than reporting a broken current-main behavior. The blocking findings are source-reproducible with focused Node probes against the added VM execution shape.

## Solution Assessment

No; the compact catalog bridge is a plausible direction, but `node:vm` with host functions is not the best way to solve an untrusted code-mode tool surface. A sandbox or worker boundary aligned with OpenClaw's existing tool policy is the safer implementation path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Do not run model code in node:vm:** `src/agents/tool-search-code-mode.ts:585-591`
  - body: The new bridge runs model-supplied JavaScript in `node:vm` while exposing host-realm async functions. A local probe using `openclaw.tools.search.constructor.constructor("return process")()` reaches the host `process`, so the advertised isolated runtime can read environment data or access host APIs. Please move this execution behind a real sandbox or hardened worker boundary before exposing it.
  - confidence: 0.97
- **[P2] Prevent async code from starving the timeout:** `src/agents/tool-search-code-mode.ts:590-592`
  - body: The timeout only wraps `runInContext()` plus a `Promise.race` timer. Code such as `while (true) await Promise.resolve()` keeps the microtask queue busy so the timer never fires; in a local probe the process had to be killed by the outer OS timeout. The code-mode runner needs enforceable cancellation for async model code, not just this in-process timer.
  - confidence: 0.93
