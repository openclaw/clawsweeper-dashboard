## Summary

Keep open. The PR is a protected maintainer-labeled core feature and the current branch still has blocking runtime/security issues around bridged tool-call lifecycle, session catalog isolation, and compaction-time tool exposure; the supplied Testbox proof is useful but does not cover these failure modes.

## What This Changes

The branch adds core Tool Search for PI runs, including compact search/describe/call bridge tools, `tools.toolSearch` config, catalog compaction for OpenClaw/MCP/client tools, docs, tests, and a gateway E2E harness.

## Best Possible Solution

Revise the PR so Tool Search fails closed on session catalog lookup, cancels or drains all bridged host calls with the run lifecycle, and either catalogs compaction tools or keeps Tool Search controls out of compaction, then rerun the targeted tests and gateway E2E proof.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head `fdb22b8610755b77606ec262b20eaec145384c7a` shows the missing settlement guard, missing abort propagation, catalog fallback, and compaction exposure paths. The requested feature itself is not a bug reproduction case.

## Solution Assessment

No. Tool Search is plausibly a core PI feature, but this implementation is not merge-ready until the bridge lifecycle and session isolation failures are fixed and covered by regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop bridge calls once code mode settles:** `src/agents/tool-search.ts:848`
  - body: The parent dispatches every `bridge` message even after the outer `tool_search_code` call has already resolved or rejected. Model code can start `openclaw.tools.call(...)` without awaiting it, return a result, and still have the host execute that target tool afterward, detaching writes/network calls from the reported tool result and run lifecycle.
  - confidence: 0.88
- **[P1] Abort host tool calls on code-mode timeout:** `src/agents/tool-search.ts:660`
  - body: `runtime.call` invokes the cataloged tool without an `AbortSignal`, so the timeout path only kills the child process. A long-running target tool can keep running in the gateway and complete side effects after `tool_search_code` has already failed with a timeout; the code-mode controller should abort in-flight bridged calls and pass that signal into `execute`.
  - confidence: 0.86
- **[P2] Require a matching session catalog:** `src/agents/tool-search.ts:489-495`
  - body: When no catalog matches the current `sessionId`/`sessionKey`/`agentId`, `resolveCatalog` falls back to the only catalog in memory. A control tool created without complete context can therefore bind to another run's active or leftover catalog instead of failing closed, which can expose and execute the wrong session's tools.
  - confidence: 0.9
- **[P2] Register compaction catalogs before exposing controls:** `src/agents/pi-embedded-runner/compact.ts:726`
  - body: The compaction path now asks `createOpenClawCodingTools` to include Tool Search controls, but this file never registers a Tool Search catalog before passing those tools to the model. If the compaction model calls `tool_search_code` or the structured controls, `resolveCatalog` reports that the catalog is unavailable and the compaction turn can fail.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR adds model-authored code execution and currently has concrete fail-closed and lifecycle gaps that can detach host tool side effects or cross session boundaries.

Concerns:

  - body: Bridge requests are still dispatched after result settlement and are not aborted on timeout, so model-authored code can trigger host-side tool side effects outside the reported `tool_search_code` lifecycle.
