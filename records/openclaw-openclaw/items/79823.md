## Summary

Keep open: this protected PR adds a new security-sensitive code-mode tool surface, and the latest head still has blocking execution-boundary and bridged-call correctness issues. Current main does not already contain this feature, so there is no cleanup close reason.

## What This Changes

The PR adds a bundled Tool Search Code Mode plugin, agent/gateway integration, docs, tests, QA mock support, and an E2E script for compact search/describe/call access to large tool catalogs.

## Best Possible Solution

Keep the compact catalog/search direction, but run model-authored code behind a real sandbox or structured interpreter with enforceable cancellation and generate unique per-invocation bridged tool call IDs before merging.

## Reproduction Assessment

Not applicable for the PR as a new feature rather than a current-main bug. The blocking review findings are source-reproducible on the PR head with a focused Node VM probe and source inspection of the tool-call recorder.

## Solution Assessment

No; the compact bridge is a plausible direction, but the current implementation still exposes outer-realm bridge functions to VM code and reuses bridged toolCallIds. A hardened sandbox or non-JS execution boundary plus per-call IDs is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Do not expose bridge functions to VM code:** `src/agents/tool-search-code-mode.ts:160-162`
  - body: The child-process rewrite still passes outer-realm bridge functions into the VM as `openclaw.tools.*`. Code running in the VM can use `openclaw.tools.search.constructor("return process.version")()` to reach the child Node process, and Node's `vm` plus permission model are not a malicious-code sandbox; model-authored code should not get this process capability through the claimed isolated runtime.
  - confidence: 0.95
- **[P2] Generate unique bridged tool call IDs:** `src/agents/tool-search-code-mode.ts:652`
  - body: Every bridged call to the same catalog entry uses `tool_search_code:${entry.name}`. When the catalog entry is a client tool, the embedded runner's recorder keys slots by `toolCallId`, so two calls to the same client tool in one code-mode run collapse into one recorded completion and lose earlier invocations.
  - confidence: 0.93
