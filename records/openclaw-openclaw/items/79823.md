## What This Changes

The PR adds core Tool Search Code Mode with config/schema/docs/tests, PI tool-catalog compaction, a child-process JavaScript bridge, structured fallback tools, QA mock support, and a gateway E2E script.

## Best Possible Solution

Keep the compact catalog direction, but merge only after hardening the bridge/session/runtime semantics and rerunning the targeted security tests plus the gateway E2E proof.

## Reproduction Assessment

Not applicable as a current-main bug because this PR adds a new optional core feature. The blocking concerns are source-reproducible from the PR head, with a focused Node permission/VM probe confirming the process exposure path.

## Solution Assessment

No. The compact tool-catalog idea is plausible, but this implementation is not ready until sandbox/process exposure, timeout cancellation, session scoping, tool-call IDs, and direct-tool result semantics match the existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Do not expose process through bridge function constructors:** `src/agents/tool-search-code-mode.ts:161-163`
  - body: The VM still receives host-created bridge functions as `openclaw.tools.*`, so model code can use `openclaw.tools.search.constructor("return process")()` to recover the child `process` object despite the bridge-only boundary. The new escape test covers direct `Function` access but misses this host-function path.
  - confidence: 0.96
- **[P1] Cancel parent-side bridged tool calls on timeout:** `src/agents/tool-search-code-mode.ts:656`
  - body: The timeout path kills only the child process. If the child is waiting on `openclaw.tools.call(...)`, the parent is still awaiting `entry.tool.execute(...)` with no AbortSignal, so side-effecting tools can continue after `tool_search_code` has already failed.
  - confidence: 0.94
- **[P2] Generate unique IDs for bridged tool calls:** `src/agents/tool-search-code-mode.ts:656`
  - body: Every call to the same catalog entry uses `tool_search_code:<entry.name>` as the toolCallId. Client-tool completion is keyed by toolCallId, so repeated calls to one client tool can overwrite or collapse earlier invocations.
  - confidence: 0.93
- **[P2] Fail closed when session catalog lookup misses:** `src/agents/tool-search-code-mode.ts:485-491`
  - body: If lookup by sessionId, sessionKey, or agentId misses, the bridge falls back to the only global catalog in memory. A context-less call should not bind to another active run's tools; require a matching session-scoped key instead.
  - confidence: 0.89
- **[P2] Preserve direct-tool error results through the bridge:** `src/agents/tool-search-code-mode.ts:656`
  - body: Direct PI tools are wrapped so non-abort failures become structured error results. Calling catalog tools directly lets target-tool exceptions reject the whole bridge/snippet, which changes recoverable validation/runtime failures into `tool_search_code` failures.
  - confidence: 0.86
