## Summary

Keep open: this protected-label core feature PR has sufficient real behavior proof, but latest-head source review still shows merge-blocking bridge isolation, session scoping, allowlist, and runtime-contract defects.

## What This Changes

The PR adds PI Tool Search with `tools.toolSearch`, `tool_search_code` plus structured search/describe/call tools, catalog compaction for OpenClaw/MCP/client tools, docs, tests, and a gateway E2E script.

## Best Possible Solution

Revise the branch so PI Tool Search is PI-scoped, fail-closed, network-contained, aborts or drains bridge work on settlement, supports restricted catalog allowlists, rejects ambiguous name calls, preserves normal tool error results, and then reruns the targeted tests plus gateway E2E proof.

## Reproduction Assessment

Yes for the review findings: latest-head source inspection and local Node permission probes show the bridge lifecycle, network permission gap, allowlist gating, catalog fallback, ambiguous name lookup, and raw execution path. The feature itself is not a bug reproduction case.

## Solution Assessment

No. The core PI compaction direction is plausible, but this implementation is not the best mergeable shape until its sandbox boundary and tool-runtime contracts match OpenClaw's existing policy and session behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Block network from the code-mode subprocess:** `src/agents/tool-search.ts:840`
  - body: `--permission` does not block outbound sockets/fetch in Node, so model-authored `tool_search_code` can still open network connections directly and exfiltrate tool outputs outside OpenClaw tool policy. Add a real network-deny boundary or remove code mode until that boundary exists.
  - confidence: 0.9
- **[P1] Abort bridge work once code mode settles:** `src/agents/tool-search.ts:991-995`
  - body: A thenable-started call such as `Promise.resolve(openclaw.tools.call(...))` can enqueue a bridge message before the model returns. The parent starts the host tool call, then normal result settlement only kills the child and suppresses responses; it does not abort already-started bridge work, leaving side effects after `tool_search_code` has returned.
  - confidence: 0.87
- **[P2] Create Tool Search controls for plugin-only allowlists:** `src/agents/pi-tools.ts:717-720`
  - body: The controls are gated on `includeOpenClawTools`, but `tools.allow: ["group:plugins"]` builds plugin tools with `includeOpenClawTools=false`. With Tool Search enabled, that common restricted catalog never gets `tool_search_code`, so `applyToolSearchCatalog` cannot compact the plugin/MCP catalog the feature is meant to handle.
  - confidence: 0.86
- **[P2] Preserve bridged tool error results:** `src/agents/pi-embedded-runner/run/attempt.ts:2498-2510`
  - body: The catalog executor invokes the raw tool implementation through `runToolLifecycle`, while the normal PI adapter converts non-abort throws into structured tool error results. A validation or runtime throw from a bridged tool will reject `tool_search_code`/`tool_call` instead of giving the model recoverable tool output.
  - confidence: 0.84
- **[P2] Reject ambiguous catalog name lookups:** `src/agents/tool-search.ts:635-639`
  - body: `findEntry` accepts either id or name and returns the first matching entry. Because client and plugin tools can legitimately share a function name, passing a name instead of an id can silently call whichever entry sorts first rather than failing or requiring the unambiguous id.
  - confidence: 0.83
- **[P2] Fail closed when no scoped catalog matches:** `src/agents/tool-search.ts:570-575`
  - body: If the control tool loses its session, key, or agent scope, `resolveCatalog` returns the only in-memory catalog instead of failing closed. That can search or execute another run's catalog whenever only one catalog happens to be registered.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces model-authored JavaScript execution and still has concrete network, post-settlement side-effect, and cross-session catalog risks.

Concerns:

- **[high] Code-mode child can use direct network:** `src/agents/tool-search.ts:840`
  - body: The child runs with `--permission` but no network-deny mechanism; local Node v24 permission-mode probing allowed `node:net` socket creation while denying filesystem reads, so model-authored code can bypass OpenClaw tool policy for outbound network access.
  - confidence: 0.9
- **[high] Bridge calls can outlive the code-tool result:** `src/agents/tool-search.ts:991`
  - body: Thenable-started bridge calls can be dispatched before the final result, and normal settlement does not abort already-started host execution, allowing tool side effects after the reported lifecycle has ended.
  - confidence: 0.87
- **[medium] Catalog fallback can cross run boundaries:** `src/agents/tool-search.ts:570`
  - body: Returning the only in-memory catalog when no scoped key matches can expose or execute tools from the wrong active or leftover run if a control tool is created with missing or stale scope.
  - confidence: 0.82
