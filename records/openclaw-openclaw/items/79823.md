## Summary

Keep open. This is a protected-label core feature PR, current main does not contain the new Tool Search implementation, and latest PR-head source still has blocking catalog scoping, ambiguous tool identity, and bridged error-semantics issues before merge.

## What This Changes

This PR adds core PI Tool Search with `tools.toolSearch` config, code and structured bridge tools, catalog compaction for OpenClaw/MCP/client tools, docs, tests, and a gateway E2E harness.

## Best Possible Solution

Keep the PR open and revise it so Tool Search enforces strict per-run catalog scoping, requires unambiguous catalog ids or rejects duplicate names, and routes bridged calls through the same PI tool adapter semantics as direct tools before rerunning targeted tests and gateway E2E proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows the singleton catalog fallback, first-match name lookup, and raw bridged execution path. The feature itself is new capability work, not a current-main bug reproduction.

## Solution Assessment

No. Core catalog compaction may be a reasonable direction, but this implementation is not the best merge shape until it preserves per-run isolation, deterministic tool identity, and existing direct-tool error semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fail closed when no scoped Tool Search catalog matches:** `src/agents/tool-search.ts:623-629`
  - body: `resolveCatalog` still returns the only process-global catalog when the current control tool has no matching session, session key, or agent scope. A stale or under-scoped Tool Search control can therefore search or call another run's catalog; remove the singleton fallback and require a scoped match.
  - confidence: 0.84
- **[P2] Reject ambiguous name-based catalog calls:** `src/agents/tool-search.ts:690-692`
  - body: `findEntry` accepts an id or name and returns the first match, while client and plugin tools can legitimately share names. If the model passes a name instead of the catalog id, Tool Search can silently call the wrong implementation; require ids or fail when a name maps to more than one entry.
  - confidence: 0.85
- **[P2] Preserve direct-tool error results for bridged calls:** `src/agents/pi-embedded-runner/run/attempt.ts:2558-2565`
  - body: The catalog executor calls the raw `tool.execute` path, so non-abort failures that direct PI tools convert into structured `status: "error"` results can instead reject `tool_search_code` or `tool_call`. Route bridged OpenClaw/plugin calls through the same adapter or normalize those errors here.
  - confidence: 0.8
