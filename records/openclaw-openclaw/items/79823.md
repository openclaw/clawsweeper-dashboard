## Summary

This PR should stay open: it is a protected, security-sensitive core feature PR and the latest head still has concrete bridge lifecycle, catalog isolation, call-id, and tool-error semantic issues that should block merge.

## What This Changes

The PR adds PI Tool Search with `tools.toolSearch` config, `tool_search_code` and structured control tools, per-session catalog compaction across OpenClaw/MCP/client tools, docs, tests, and a gateway E2E script.

## Best Possible Solution

Revise the branch so Tool Search is PI-scoped, fail-closed, drains or aborts in-flight bridge work, uses run-scoped unique bridge call IDs, preserves normal PI tool error results, and then reruns the targeted tests plus gateway E2E proof.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head 5e613f9578c921be773472aef6d99f1d7e72b618 shows the bridge lifecycle, call-id reset, catalog fallback, and raw execution paths. The feature itself is not a bug reproduction case.

## Solution Assessment

No. The core PI compaction direction is plausible, but this implementation is not the best mergeable shape until its bridge lifecycle, session isolation, call identity, and error-result semantics match the existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop bridge work before returning code-mode results:** `src/agents/tool-search.ts:263-264`
  - body: A model can start a bridge call with `Promise.resolve(openclaw.tools.call(...))` and then return. The child pumps pending bridge messages before sending the final result, so the parent can dispatch the host tool call and then only suppress the response after `tool_search_code` has already settled; this leaves post-completion side effects outside the reported tool result.
  - confidence: 0.87
- **[P2] Keep bridge tool-call IDs unique across code runs:** `src/agents/tool-search.ts:805`
  - body: `runCodeMode` creates a fresh runtime for every `tool_search_code` execution, so `callSequence` restarts at 1. In one embedded run, repeated code-tool invocations against the same client tool can reuse IDs such as `tool_search_code:<name>:1`, and the client-tool recorder dedupes by `toolCallId`, collapsing distinct calls.
  - confidence: 0.86
- **[P2] Fail closed on missing catalog scope:** `src/agents/tool-search.ts:570-575`
  - body: When no session, key, or agent lookup matches, `resolveCatalog` returns the only catalog in process memory. A control tool created with missing or stale scope can therefore search or execute another run's catalog instead of failing closed.
  - confidence: 0.82
- **[P2] Preserve direct-tool error semantics in bridged calls:** `src/agents/pi-embedded-runner/run/attempt.ts:2503-2510`
  - body: The bridged executor calls the raw cataloged tool through `runToolLifecycle`, while the normal PI adapter converts non-abort throws into structured tool error results. A validation/runtime throw from a cataloged tool will reject `tool_search_code` or `tool_call` instead of giving the model recoverable tool output.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces model-authored code execution and still has concrete lifecycle and session-isolation risks at latest head.

Concerns:

- **[high] Bridge calls can outlive the code-tool result:** `src/agents/tool-search.ts:263`
  - body: Thenable-started bridge work can be dispatched before the final result is sent, and already-started host calls are not aborted or drained when `tool_search_code` settles, allowing side effects outside the reported lifecycle.
  - confidence: 0.87
- **[medium] Catalog fallback can cross run boundaries:** `src/agents/tool-search.ts:570`
  - body: Returning the only in-memory catalog when no scoped key matches can expose or execute tools from the wrong active or leftover run if a control tool loses session scope.
  - confidence: 0.82
