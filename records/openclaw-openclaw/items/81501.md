## Summary

Keep open. Current main still blocks installed plugin dependency trees when transitive runtime files hit critical scanner heuristics, and the requested non-break-glass path is a security/product policy decision rather than cleanup that can be closed.

## What This Changes

The issue asks for a supported install path for third-party memory/vector plugins whose LanceDB/OpenAI runtime dependencies trigger the plugin install dangerous-code scanner.

## Best Possible Solution

Define a maintained trust/attestation path for audited native or ML dependency trees, with focused tests that keep dangerous plugin-authored code blocked and avoid making `--dangerously-force-unsafe-install` the normal install path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: install any packaged plugin whose dependency tree contains files matching the current critical `child_process` or dynamic-code heuristics, and the installed-tree scan blocks before success. I did not run the full packaged beta smoke because this review was read-only.

## Solution Assessment

Unclear: the issue lists plausible directions, but choosing between a versioned allowlist, manifest trust policy, granular prompt, or ClawHub attestation needs maintainer security/product judgment. The narrow implementation should preserve the fail-closed default and add explicit coverage for legitimate dependency false positives.
