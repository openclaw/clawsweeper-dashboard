## Summary

Keep open: the PR is useful and targets a real gap, but current head still has blocking correctness issues in detector ordering, blocked-veto history recording, and config validation.

## What This Changes

The PR adds a run-scoped consecutive tool-error loop detector with a new `tools.loopDetection.consecutiveErrorThreshold` config surface, schema/docs/changelog updates, and detector/hook tests.

## Best Possible Solution

Keep the runId-scoped detector design, but evaluate the critical consecutive-error check before warning-only returns, preserve non-loop veto history, validate against the effective history window, remove the duplicate changelog line, and then rerun focused agent/config/docs checks.

## Reproduction Assessment

Yes. Source inspection of PR head shows ten same-run alternating failing A/B calls can hit a warning-only `ping_pong` return before `consecutive_errors`, and a threshold-only config above the default history window is accepted while history is still trimmed to 30.

## Solution Assessment

No, not as currently ordered and validated. The proposed run-scoped guard is the right narrow direction, but the PR needs targeted fixes before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move consecutive errors before warning returns:** `src/agents/tool-loop-detection.ts:635-636`
  - body: The new critical detector runs only after the warning-only ping-pong return. With default thresholds, ten same-run alternating failed A/B calls can return `ping_pong` as a warning, and the hook only blocks critical results, so the consecutive-error guard does not stop the cascade at its threshold.
  - confidence: 0.92
- **[P2] Keep recording non-loop veto outcomes:** `src/agents/pi-tools.before-tool-call.ts:643`
  - body: The sticky-blocking change skips outcome recording for every veto. That also removes result hashes for repeated `plugin-before-tool-call` or approval blocks, so existing no-progress/global loop detection can no longer escalate a model that keeps retrying the same policy-blocked tool; skip only `tool-loop` vetoes or keep non-loop blocks recorded.
  - confidence: 0.84
- **[P2] Validate thresholds against the effective history window:** `src/config/zod-schema.agent-runtime.ts:541-543`
  - body: This check only runs when `historySize` is explicit. A config with `consecutiveErrorThreshold: 40` and no `historySize` is accepted, but the loop history is still trimmed to the default 30 records, so the detector can never reach 40; compare against the resolved history size or expand it deliberately.
  - confidence: 0.82
- **[P3] Remove the duplicate gcloud changelog entry:** `CHANGELOG.md:400`
  - body: This PR adds the gcloud `CLOUDSDK_PYTHON` fix bullet even though the same bullet remains later in the current changelog, so the active release notes would duplicate an unrelated fix after merge.
  - confidence: 0.9
