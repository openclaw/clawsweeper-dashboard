## Summary

Keep open: the PR targets a real gap on current main, but the new critical detector can be bypassed by an earlier warning-only ping-pong return, and the branch also duplicates an unrelated changelog entry.

## What This Changes

The PR adds a runId-scoped consecutive-errors tool-loop detector with config/schema/docs/changelog updates, diagnostic typing, and unit plus hook-path coverage.

## Best Possible Solution

Keep the runId-scoped detector design, make critical consecutive-error detection take precedence over warning-only loop classifications, remove the duplicate changelog line, then merge after exact-head validation passes.

## Reproduction Assessment

Yes. Current main lacks the detector, and the PR behavior can be reproduced by recording same-run failed tool outcomes before the next tool call; the introduced ordering bug is reproducible with alternating A/B failures at the default threshold.

## Solution Assessment

No, not as currently ordered. The proposed run-scoped detector is the right narrow mechanism, but it should run before warning-only returns so `consecutiveErrorThreshold` reliably blocks cascades.
