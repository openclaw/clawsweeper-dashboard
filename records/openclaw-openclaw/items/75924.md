## Summary

Keep open: current main does not contain the requested run-scoped consecutive-error detector, but the latest PR head still has concrete merge-blocking behavior issues in the detector ordering, veto outcome recording, and config validation.

## What This Changes

Adds run-scoped `consecutive_errors` loop detection plus config/schema/docs/tests/diagnostic typing and changelog coverage for blocking consecutive failing tool calls.

## Best Possible Solution

Land a narrow runId-scoped consecutive-error detector only after the critical check wins over warning-only classifications, non-loop vetoes keep their existing outcome history behavior, impossible threshold configurations are rejected, and the changelog is cleaned up.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence reproduction path for the review blockers: default-threshold alternating failing A/B calls can return `ping_pong` before `consecutive_errors`, and the config validation gap follows directly from the default history size.

## Solution Assessment

No. The runId-scoped detector is a reasonable narrow direction, but the current implementation needs ordering, veto-recording, and effective-history validation fixes before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move consecutive errors before warning returns:** `src/agents/tool-loop-detection.ts:635-636`
  - body: The new critical check still runs after the warning-only ping-pong return. At the default threshold, an alternating failing A/B error tail can return `ping_pong` as a warning before `consecutive_errors` is evaluated, and the hook path only blocks critical loop results, so this guard does not stop that cascade at its threshold.
  - confidence: 0.93
- **[P2] Keep recording non-loop veto outcomes:** `src/agents/pi-tools.before-tool-call.ts:643`
  - body: The sticky-blocking change now skips outcome recording for every veto. That fixes tool-loop vetoes clearing the error streak, but it also stops plugin/trusted-policy blocks from feeding existing no-progress history, so repeated policy-blocked calls lose the escalation behavior current main has. Skip recording only `deniedReason: "tool-loop"` vetoes or otherwise preserve non-loop veto recording.
  - confidence: 0.86
- **[P2] Validate against the effective history size:** `src/config/zod-schema.agent-runtime.ts:541-543`
  - body: The schema only rejects `historySize < consecutiveErrorThreshold` when `historySize` is explicitly set. A config with only `consecutiveErrorThreshold: 50` still uses the default 30-entry history window, so the detector can never reach the configured threshold. Compare against the resolved/default history size and add a regression for that case.
  - confidence: 0.88
- **[P3] Remove the duplicate gcloud changelog line:** `CHANGELOG.md:404`
  - body: This PR still adds the gcloud `CLOUDSDK_PYTHON` fix bullet while the same bullet remains later in the active changelog section, so the release notes would duplicate an unrelated fix after merge.
  - confidence: 0.9
