## Summary

Keep open: the PR is useful and scoped to the right loop-detection surface, but it is not merge-ready because the new critical detector still runs after warning-only loop returns, so default alternating failure cascades can avoid the intended block.

## What This Changes

The PR adds a run-scoped `consecutive_errors` tool-loop detector with config/schema/docs/test/diagnostic coverage and a changelog entry.

## Best Possible Solution

Keep the runId-scoped detector design, prioritize its critical check before warning-only loop classifications, add alternating-failure coverage, and merge after exact-head validation.

## Reproduction Assessment

Yes. A same-run alternating A/B sequence of failing tool outcomes at the default threshold reaches the ping-pong warning path before `consecutive_errors`, and the hook path only blocks critical results.

## Solution Assessment

No, not as currently ordered. The proposed detector is the narrow maintainable mechanism, but it must run before warning-only returns to reliably stop the failure cascade it claims to block.
