## Summary

Keep open: the PR addresses a real loop-detection gap, but the current head still has blocking correctness issues in detector ordering and veto outcome recording, plus an unrelated duplicate changelog line.

## What This Changes

Adds a run-scoped `consecutive_errors` loop detector with `tools.loopDetection.consecutiveErrorThreshold` config/schema/docs/tests and adjusts blocked tool-call outcome handling.

## Best Possible Solution

Keep the runId-scoped detector design, but evaluate the critical consecutive-error check before warning-only returns, preserve outcome recording for non-tool-loop vetoes, add focused regressions, and remove the duplicate changelog line.

## Reproduction Assessment

Yes. Source inspection of PR head shows ten same-run alternating failing A/B calls can return a `ping_pong` warning before `consecutive_errors`, and the hook path only blocks critical loop results.

## Solution Assessment

No, not as currently implemented. The proposed detector is the right narrow shape, but it must run before warning-only classifications and the sticky-blocking change should skip recording only tool-loop vetoes, not all hook vetoes.
