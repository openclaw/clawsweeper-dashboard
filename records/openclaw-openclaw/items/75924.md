## Summary

Keep open: current main still lacks the requested run-scoped consecutive-error detector, and this PR is the active implementation for #75923. The design is narrow, but PR head still has source-backed blocking issues in detector ordering and veto outcome recording, plus a duplicate unrelated changelog line.

## What This Changes

Adds a run-scoped `consecutive_errors` tool-loop detector with `tools.loopDetection.consecutiveErrorThreshold`, config/schema/docs/changelog updates, and detector/hook regression tests.

## Best Possible Solution

Keep the runId-scoped detector design, but evaluate `consecutive_errors` before warning-only detections, preserve outcome recording for non-tool-loop vetoes, remove the duplicate changelog line, and land with focused agent/config/docs coverage.

## Reproduction Assessment

Yes. Source inspection of PR head shows default-threshold alternating failing tool calls can return a warning-only `ping_pong` result before `consecutive_errors`, and the hook path only blocks critical loop results.

## Solution Assessment

No, not as currently implemented. The run-scoped detector is the right narrow shape, but the ordering and veto-recording changes need repair before this is the safest maintainable fix.
