## Summary

Keep this PR open: current main still lacks the requested run-scoped cross-tool error detector, but the PR head has merge-blocking source issues in detector priority and failure classification, plus a duplicate changelog entry.

## What This Changes

The branch adds a run-scoped `consecutive_errors` tool-loop detector with config, schema, docs, tests, diagnostics, and a changelog entry.

## Best Possible Solution

Keep the runId-scoped detector design, run the critical consecutive-error check before warning-only returns, classify structured failed tool results as failures, and merge after focused regression coverage and changed-gate proof.

## Reproduction Assessment

Yes. A same-run alternating A/B failure sequence at the default threshold reaches the `ping_pong` warning return before `consecutive_errors`, and structured `details.status: "failed"` tool results never receive the `error:` hash prefix the new streak requires.

## Solution Assessment

No, not as currently implemented. The proposed detector is the right narrow mechanism, but it needs earlier priority and broader failure classification to reliably block the cross-tool cascades it targets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Evaluate consecutive errors before warn-only loops:** `src/agents/tool-loop-detection.ts:635-636`
  - body: The new critical check runs after the ping-pong warning return. With default thresholds, ten alternating failing calls can return a warning before `consecutive_errors` is evaluated, and the hook only blocks critical results, so the cascade continues past the intended threshold.
  - confidence: 0.91

  - body: Many tools report failures as normal tool results with `details.status: "failed"`, but the new streak only counts hashes starting with `error:`, which are produced for thrown exceptions. Cross-tool cascades of structured failed results therefore never trip the new detector.
