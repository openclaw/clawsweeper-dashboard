The PR adds a runId-scoped `consecutive_errors` tool-loop detector with config/schema/docs, diagnostic typing, tests, and a changelog entry.

## Best Possible Solution

Keep the runId-scoped detector design, move the critical consecutive-error check before warning-only detector returns, add an alternating-failure regression test, remove the duplicate changelog line, then merge after exact-head validation.

## Reproduction Assessment

Yes. On the PR head, a same-run alternating A/B failure sequence at the default threshold reaches the ping-pong warning return before the new `consecutive_errors` critical check, and the hook path does not block warning results.

## Solution Assessment

No, not as currently ordered. The proposed run-scoped detector is the right narrow mechanism, but critical consecutive-error detection needs priority over warning-only classifications to reliably stop cascades.
