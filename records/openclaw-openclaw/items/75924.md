## Summary

Keep open: the PR targets a real guardrail gap, but the current head still lets alternating same-run failure cascades hit a warning-only ping-pong return before the new critical detector, and the sticky-blocking fix drops outcome recording for all hook vetoes. A duplicate changelog entry also remains.

## What This Changes

Adds a run-scoped consecutive tool-error loop detector with config/schema/help/docs/changelog coverage and tests for same-run blocking plus runId isolation.

## Best Possible Solution

Keep the runId-scoped detector design, but evaluate the critical consecutive-error check before warning-only returns, preserve outcome recording for non-tool-loop vetoes, add alternating-failure and plugin-veto regression coverage, remove the duplicate changelog line, then rebase and validate.

## Reproduction Assessment

Yes. Source inspection of PR head shows ten same-run alternating failing A/B calls can satisfy the ping-pong warning path before consecutive_errors is evaluated, and the hook path only blocks critical loop results.

## Solution Assessment

No, not as currently ordered. The proposed detector is narrow and aligned with existing runId semantics, but it must run before warning-only classifications and the sticky-blocking fix should skip recording only tool-loop vetoes, not all hook vetoes.
