## What This Changes

The PR restores non-terminal failed commitment extraction batches to the front of the in-memory queue, re-arms the existing drain debounce, and adds regression tests for retry, ordering, overflow, timer-fired retry, and terminal-error behavior.

## Best Possible Solution

Land the narrow restore-and-rearm fix after maintainer review and required checks, keeping terminal cooldown/drop behavior unchanged and leaving durable queue persistence as a separate product decision.

## Reproduction Assessment

Yes. Current main has a source-reproducible path where `drainCommitmentExtractionQueue` removes a batch before extractor execution and rethrows a non-terminal error before restoration or persistence.

## Solution Assessment

Yes. Restoring the original batch at the extraction failure boundary and reusing the existing single-slot debounce is the narrowest maintainable fix; durable retry storage would be a larger follow-up design.
