## Summary

Keep open. Current main and the latest release still reject `sessions_send` calls that contain both `sessionKey` and `label`, and the active in-repo implementation path is still open and conflicting rather than merged.

## What This Changes

The issue reports that `sessions_send` receives an injected `label` alongside an explicit `sessionKey`, causing the tool to return its mutual-exclusion error before message delivery.

## Best Possible Solution

Land one canonical fix that treats an explicit `sessionKey` or session-id input as authoritative for redundant label-style fields, preserves visibility/A2A/thread safeguards, avoids leaking resolved keys on denial, updates tool guidance, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with both non-empty `sessionKey` and `label`, and current main returns the reported mutual-exclusion error before delivery.

## Solution Assessment

Yes, but not by closing this issue yet. The narrow maintainable direction is executor-side normalization plus focused tests and updated tool guidance; the linked PR still needs repair or replacement before merge.
