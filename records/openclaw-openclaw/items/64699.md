## Summary

Keep open. Current `main` and the latest release still reject mixed `sessionKey`/`label` `sessions_send` calls before delivery, and the active target-repo fix remains open and conflicting rather than merged.

## What This Changes

The issue reports that `sessions_send` fails when caller paths send both an explicit `sessionKey` and an injected `label`.

## Best Possible Solution

Land one canonical fix that makes explicit `sessionKey` or session-id targeting authoritative for redundant label-style fields while preserving visibility, A2A, and thread-target safeguards, then close this issue after merge/release provenance is clear.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with non-empty `sessionKey` and `label`, and current `main` returns the reported mutual-exclusion error before gateway delivery.

## Solution Assessment

Yes. Executor-side normalization plus focused tests is the narrow maintainable fix because schema or docs guidance alone cannot protect client paths that already submit redundant selectors.
