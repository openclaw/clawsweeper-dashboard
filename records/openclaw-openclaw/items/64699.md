## Summary

Keep open. Current main and the latest release still reject the reported mixed `sessionKey` plus `label` payload before `sessions_send` can resolve or deliver, and the active implementation path at https://github.com/openclaw/openclaw/pull/74009 is still open with unresolved review/security concerns.

## What This Changes

A `sessions_send` call intended to route by `sessionKey` should not fail when a client or tool layer also supplies a redundant `label`.

## Best Possible Solution

Land one canonical `sessions_send` fix that treats explicit `sessionKey` or sessionId routing as authoritative for redundant label-style fields, preserves visibility/A2A checks, updates tool/schema guidance, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with both `sessionKey` and `label`, and current main returns the reported mutual-exclusion error before gateway resolution; I did not run tests because this review is read-only.

## Solution Assessment

Yes in direction, but it is not landed. Executor-side normalization is the narrow maintainable fix because schema/docs guidance alone cannot protect client paths that already submit a polluted payload.
