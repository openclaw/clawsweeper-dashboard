## Summary

Keep open. Current main and the latest release still return the reported mixed-selector error when `sessions_send` receives both `sessionKey` and `label`, and the open implementation path has not landed yet.

## What This Changes

The issue requests that `sessions_send` stop failing when a `sessionKey`-targeted call receives a redundant `label` from client argument assembly.

## Best Possible Solution

Land one canonical fix, likely through https://github.com/openclaw/openclaw/pull/74009 or an equivalent replacement, that treats explicit `sessionKey`/sessionId as authoritative while preserving visibility/A2A confidentiality, updates tool-facing guidance, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `sessions_send` with both `sessionKey` and `label`, and current main returns the reported mutual-exclusion error before gateway resolution.

## Solution Assessment

Yes. Executor-side normalization is the narrow maintainable direction because schema or docs guidance alone would not protect client paths that already emitted a redundant `label`; the open PR still needs its review blockers resolved before merge.
