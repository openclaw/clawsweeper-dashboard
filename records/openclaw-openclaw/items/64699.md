## Summary

Keep open. Current `main` and the latest release still reject a `sessions_send` call containing both `sessionKey` and `label`, and the active implementation path is an open PR that has not landed.

## What This Changes

The issue reports that `sessions_send` rejects calls when a client supplies both `sessionKey` and `label`, even when `sessionKey` is the intended target.

## Best Possible Solution

Land one canonical `sessions_send` fix that makes explicit `sessionKey` or sessionId routing robust to redundant label-style fields, preserves denied-access confidentiality, updates tool guidance, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with both `sessionKey` and `label`, and current `main` returns the reported mutual-exclusion error before resolution.

## Solution Assessment

Yes in direction, but not yet landed. Executor-side normalization is the narrow maintainable fix because schema guidance alone would not protect client paths that already submit a polluted payload.
