## Summary

Keep open. Current main still rejects `sessions_send` calls that contain both `sessionKey` and `label` before any target resolution, matching the reported failure, and the latest release source still has the same guard. The issue also has active open implementation paths, including PR #74009 and fork PR BingqingLyu/openclaw#2317, so it should remain open until one canonical fix merges.

## What This Changes

The issue reports that `sessions_send` calls intended to target by `sessionKey` can receive a redundant `label` from client/control layers and then fail the tool's mutual-exclusion guard.

## Best Possible Solution

Land one canonical fix, preferably through PR #74009 or an equivalent replacement, that treats explicit `sessionKey` or session-id-shaped input as authoritative over redundant label-style hints, preserves label-only resolution and A2A visibility checks, updates the tool-facing description, and adds focused regression tests.

## Solution Assessment

Yes in direction, but not yet in current main. The narrow maintainable fix is executor-side `sessionKey` precedence plus description and regression coverage, with related open PRs needing review before this issue can close.
