## Summary

Keep open. Current main and the latest release still reject `sessions_send` payloads that contain both `sessionKey` and `label`, matching the reported failure, and the open implementation paths have not landed yet.

## What This Changes

Allow `sessions_send` calls with an explicit `sessionKey` plus an injected or redundant `label` to route by `sessionKey` instead of failing mutual-exclusion validation.

## Best Possible Solution

Land one canonical fix in `src/agents/tools/sessions-send-tool.ts` that makes explicit `sessionKey` or `sessionId` authoritative over redundant label-style fields, preserves label-only resolution and A2A visibility checks, updates the tool-facing description, and adds focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: call `sessions_send` with both `sessionKey` and `label`, and the executor hits the guard at `src/agents/tools/sessions-send-tool.ts:109` before any gateway resolution.

## Solution Assessment

Yes, the requested executor-side robustness is the right direction because schema/docs guidance alone would not protect polluted client payloads. The current main implementation is not the solution yet.
