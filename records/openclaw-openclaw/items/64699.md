## Summary

Keep open. Current main still returns the reported `sessions_send` mixed-selector error before target resolution, the latest release has the same guard, and open implementation candidates have not landed.

## What This Changes

The issue asks for `sessions_send` to handle client payloads that contain an explicit `sessionKey` plus an injected `label` without failing the mutual-exclusion guard.

## Best Possible Solution

Land one canonical fix in `src/agents/tools/sessions-send-tool.ts` that treats explicit `sessionKey` or sessionId routing as authoritative when redundant label-style metadata is present, preserves denied-access confidentiality, updates tool guidance, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with both `sessionKey` and `label`, and current main returns the reported mutual-exclusion error before resolution or delivery.

## Solution Assessment

Yes in direction, but not yet landed. Executor-side normalization is the narrow maintainable fix because schema or docs guidance alone does not protect client paths that already submit a polluted payload.
