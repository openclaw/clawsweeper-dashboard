## Summary

Keep open. Current main and the latest release still hard-reject `sessions_send` calls that contain both `sessionKey` and `label`, and the active fixes are still open and unmerged.

## What This Changes

The issue reports that `sessions_send` receives both `sessionKey` and `label` from client paths and rejects an otherwise valid sessionKey-targeted send.

## Best Possible Solution

Land one canonical fix in `src/agents/tools/sessions-send-tool.ts` that treats explicit `sessionKey` or session-id input as authoritative for redundant label-style fields, preserves A2A visibility and thread safeguards, updates tool guidance, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with non-empty `sessionKey` and `label`, and current main returns the reported mutual-exclusion error before any gateway call.

## Solution Assessment

Yes. Executor-side normalization plus regression coverage is the narrow maintainable fix because schema or docs guidance alone cannot protect client paths that already submit a redundant label.
