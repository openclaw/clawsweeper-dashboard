## Summary

Keep this PR open: the context-budget direction is valid and the contributor supplied real setup proof, but the patch introduces blocking auth-profile preservation and strict cap enforcement regressions.

## What This Changes

The PR adds Codex app-server guards to rotate over-budget native thread bindings and truncate dynamic tool text results, with focused Codex app-server tests.

## Best Possible Solution

Preserve the original binding auth profile across oversized-thread rotation and enforce the configured dynamic tool cap for every positive limit, then rerun the focused Codex app-server checks.

## Reproduction Assessment

Yes for the review findings by source inspection: an over-budget binding with only a persisted `authProfileId` will be cleared before startup auth resolution, and a tiny positive tool-result cap returns the full notice over budget. The original slow WebChat symptom was not locally reproduced in this read-only pass, but the PR includes live terminal proof for that path.

## Solution Assessment

No, not yet; the intended approach is narrow, but the current implementation needs to preserve auth context and make truncation strictly bounded. Capturing the original binding auth before rotation and using bounded suffix logic would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve bound auth when rotating oversized bindings:** `extensions/codex/src/app-server/run-attempt.ts:752-757`
  - body: When the new guard clears an oversized binding, `startupBinding` becomes `undefined` before `startupAuthProfileCandidate` is built. Existing coverage requires startup to reuse a bound auth profile when params omit one, but this path would start the fresh Codex app-server client without that profile. Capture the original binding auth before rotation and feed it into startup auth resolution.
  - confidence: 0.91
- **[P2] Keep truncation output within the configured cap:** `extensions/codex/src/app-server/dynamic-tools.ts:272`
  - body: `readPositiveInteger` accepts any positive `toolResultMaxChars`, but when the configured limit is smaller than the truncation notice, `sliceLength` becomes 0 and the returned notice still exceeds the limit. That breaks the documented cap for low values; bound the suffix itself or use existing bounded truncation behavior.
  - confidence: 0.86
