## Summary

Keep open: the direction is useful, but the current patch has blocking correctness issues around the byte guard default and aggregate tool-result budgeting, and the live PR body currently does not show after-fix real behavior proof.

## What This Changes

The PR adds Codex app-server startup binding rotation based on native rollout token/byte budgets and applies a configurable character cap to dynamic tool text returned to the Codex native thread, with focused tests.

## Best Possible Solution

Keep the Codex app-server guard, but make the byte rotation conditional on an explicitly configured positive maxActiveTranscriptBytes value and enforce one aggregate dynamic tool-result budget before returning text to Codex.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main resumes Codex app-server bindings without the proposed budget rotation and returns dynamic tool text to the native thread without applying toolResultMaxChars; I did not run live validation in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the patch should preserve the documented disabled byte-guard behavior and apply the configured tool-result cap across the whole result, not per text block.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve disabled byte guard when no byte limit is configured:** `extensions/codex/src/app-server/run-attempt.ts:482-483`
  - body: `parseCodexAppServerByteLimit` turns an unset, invalid, or zero `maxActiveTranscriptBytes` into a 1MiB limit, and the startup rotation runs whenever `truncateAfterCompaction` is true. The docs say this byte guard is disabled when unset or `0`, so users who only enabled transcript rotation can unexpectedly lose their existing native Codex thread once a rollout file exceeds 1MiB. Return no byte limit for unset/non-positive values and skip the byte-size rotation unless the option is explicitly configured.
  - confidence: 0.9
- **[P2] Apply the cap across the whole tool result:** `extensions/codex/src/app-server/dynamic-tools.ts:162-164`
  - body: This maps each content block through `convertToolContent`, and `convertToolContent` truncates each text block independently. Since `toolResultMaxChars` is documented as the cap for a single live tool result and `AgentToolResult.content` can contain multiple text blocks, two 16k text blocks would still send about 32k chars into the native Codex thread. Budget across all text blocks in the result before flattening the response.
  - confidence: 0.86
