## Summary

Keep open: the latest head fixes the earlier auth-preservation and tiny-cap review findings, but the patch still has narrow Codex app-server budget bugs around byte-guard activation, aggregate tool-result caps, and oversized rollout reads.

## What This Changes

The PR adds Codex app-server native-thread rotation guards and dynamic tool-result truncation, with focused Codex app-server tests.

## Best Possible Solution

Keep the Codex budget-guard direction, but gate byte rotation on an explicitly configured positive maxActiveTranscriptBytes, avoid full reads of already oversized rollout files, and enforce toolResultMaxChars across the aggregate dynamic tool response.

## Reproduction Assessment

Yes for the review findings by source inspection: unset maxActiveTranscriptBytes flows to a 1 MiB byte limit, multiple text blocks can exceed the aggregate cap, and rollout JSONL files are read before the byte-size branch clears them. The original live WebChat slowdown was not locally reproduced, but the PR includes terminal proof for that path.

## Solution Assessment

No, not yet; the approach is narrow and maintainable, but the current implementation needs to preserve documented config semantics and enforce caps on the actual aggregate surfaces. A safer repair is to reuse the existing positive-byte-limit semantics and apply one bounded truncation budget per dynamic tool result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate byte rotation on the configured byte limit:** `extensions/codex/src/app-server/run-attempt.ts:639-640`
  - body: The PR says the byte path stays tied to `agents.defaults.compaction.maxActiveTranscriptBytes`, and the docs say unset or `0` disables that guard. Here an unset, zero, or invalid value falls back to 1 MiB, so any Codex session with only `truncateAfterCompaction: true` can clear its native binding once the rollout file passes 1 MiB. Skip the byte branch unless the configured value parses to a positive limit.
  - confidence: 0.92
- **[P2] Enforce the cap across the whole tool result:** `extensions/codex/src/app-server/dynamic-tools.ts:162-164`
  - body: `toolResultMaxChars` is a live tool-result cap, but this flatMap caps each text content block separately. A dynamic tool returning several text blocks can still send multiple times the configured cap into the native Codex thread. Budget across `result.content` as one result before converting the blocks.
  - confidence: 0.88
- **[P2] Avoid full reads before clearing oversized rollouts:** `extensions/codex/src/app-server/run-attempt.ts:646-648`
  - body: The guard already has file sizes, but it reads every matched rollout JSONL in full to find token usage before checking whether any file exceeds the byte limit. An already oversized native rollout can therefore be loaded into memory on the recovery path that should clear it. Check the byte branch first when configured, or scan token usage with a bounded/streamed reader.
  - confidence: 0.82
