## Summary

Keep open: the underlying prompt-context gap is source-reproducible, but this PR still needs the surfaced key to line up with the Telegram sticker send tool and it lacks real Telegram behavior proof.

## What This Changes

The PR adds `sticker_file_id` to Telegram per-turn inbound conversation metadata and tests that it is omitted from stable system metadata and webchat turns.

## Best Possible Solution

Keep sticker file IDs in untrusted per-turn context, but align the exposed key with the accepted sticker-send contract or add an advertised alias, then attach redacted real Telegram proof of the sticker-context or sticker-echo flow.

## Reproduction Assessment

Yes, at source level: current `main` can receive `TemplateContext.Sticker.fileId`, but `buildInboundUserContextPrefix()` does not project it into conversation info. I did not run tests because this review had to keep the checkout read-only.

## Solution Assessment

No, not yet. The per-turn context location is the right boundary, but the key should line up with `message.send` sticker arguments or be supported as an alias before this reliably fixes sticker replay.
