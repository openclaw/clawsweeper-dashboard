## Summary

Keep open: current main still has the Telegram sticker prompt-context gap, and this PR uses the right per-turn metadata boundary, but it should not merge as-is because the surfaced key does not match the Telegram send contract and the PR lacks real Telegram proof.

## What This Changes

The branch adds `sticker_file_id` to inbound per-turn conversation metadata and tests system-prompt exclusion, webchat omission, and Telegram inclusion.

## Best Possible Solution

Expose the replayable Telegram sticker id in untrusted per-turn context using an accepted `fileId`/`stickerId` contract or a documented send-path alias, keep stable system metadata unchanged, and verify the real Telegram sticker replay flow.

## Reproduction Assessment

Yes, source-reproducible: current main passes `Sticker.fileId` into `TemplateContext` but `buildInboundUserContextPrefix()` omits it from conversation info. The PR's key mismatch is also source-checkable because the send path and tool schema accept `fileId`/`stickerId`, not `sticker_file_id`.

## Solution Assessment

No, not as-is. Per-turn untrusted conversation metadata is the right boundary, but the exposed key must line up with the send contract or be supported as an alias before merge.
