## Summary

Keep open: this is a focused prompt-context improvement, but it is not merge-ready because the exposed `sticker_file_id` key does not line up with the Telegram sticker send arguments and the external contributor has only synthetic proof.

## What This Changes

The PR adds `sticker_file_id` to inbound per-turn conversation info for eligible Telegram contexts and adds regression tests keeping it out of stable system metadata and direct webchat.

## Best Possible Solution

Keep the sticker id in per-turn untrusted metadata, align it with the accepted `message`/Telegram sticker action arguments, and prove the after-fix Telegram sticker workflow before merge.

## Reproduction Assessment

Yes, source-reproducible: construct a Telegram direct-chat `TemplateContext` with `Sticker.fileId` and inspect `buildInboundUserContextPrefix()` on current main; the metadata path omits the sticker id. I did not execute tests because this review is read-only.

## Solution Assessment

No, not yet. Per-turn untrusted conversation metadata is the right boundary, but the surfaced key must align with accepted send aliases before it reliably solves sticker replay.
