## Summary

Keep open: current main still has the Telegram sticker prompt-context gap, but this PR is not merge-ready because the surfaced `sticker_file_id` key does not match the accepted sticker send arguments and the PR still lacks after-fix real Telegram proof.

## What This Changes

The PR adds `sticker_file_id` to inbound per-turn conversation metadata and adds focused tests for stable system metadata exclusion, webchat omission, and Telegram direct-chat inclusion.

## Best Possible Solution

Expose a replayable Telegram sticker id in untrusted per-turn context using a tool-compatible key or a documented supported alias, keep stable system metadata unchanged, and prove the real Telegram sticker replay flow.

## Reproduction Assessment

Yes, source-reproducible: current main carries Telegram `Sticker.fileId` through ingress into `TemplateContext`, but `buildInboundUserContextPrefix()` does not project it into conversation info. The send-path key mismatch is also visible from the message tool schema and Telegram action runtime.

## Solution Assessment

No, not as-is. Per-turn untrusted conversation metadata is the right boundary, but the exposed key should line up with the documented `fileId`/`stickerId` send contract or the send path should support the alias before merge.
