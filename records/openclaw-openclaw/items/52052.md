## Summary

Keep open: current main still has the Telegram sticker prompt-context gap, but this PR is not ready to merge because the surfaced `sticker_file_id` key does not match the accepted sticker send arguments and the proof is limited to synthetic/unit checks.

## What This Changes

This PR adds `sticker_file_id` to inbound per-turn conversation metadata and adds focused tests for stable-system-metadata exclusion, direct webchat omission, and Telegram direct-chat inclusion.

## Best Possible Solution

Keep the sticker id in per-turn untrusted conversation metadata, align the exposed key or send-path aliases with the documented `fileId`/`stickerId` contract, and prove the Telegram sticker replay flow in a real setup.

## Reproduction Assessment

Yes, source-reproducible: current main carries Telegram `Sticker.fileId` through ingress, but `buildInboundUserContextPrefix()` does not project it into conversation info. The send-path mismatch is also source-reproducible from the message tool schema and Telegram action runtime.

## Solution Assessment

No, not as-is. Per-turn untrusted metadata is the right boundary, but the PR should expose a tool-compatible key or add a documented alias before merge.
