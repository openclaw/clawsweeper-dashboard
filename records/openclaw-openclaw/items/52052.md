## Summary

Keep open: current main still omits the Telegram sticker file id from agent-visible per-turn context even though Telegram ingress carries `Sticker.fileId`, so the PR addresses a real gap. It is not merge-ready because the surfaced key does not match the sticker send/tool contract, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `sticker_file_id` to inbound per-turn conversation info and tests that it is included for eligible Telegram direct chats while staying out of stable system metadata and direct webchat sticker-only turns.

## Best Possible Solution

Keep the sticker id in per-turn untrusted conversation metadata, but align the exposed key with the accepted sticker send contract or add a documented alias, then prove the sticker reply path with redacted real Telegram or terminal/log output.

## Reproduction Assessment

Yes, source-reproducible: construct a Telegram direct-chat `TemplateContext` with `Sticker.fileId` and inspect `buildInboundUserContextPrefix()` on current main. The Telegram ingress path already passes `Sticker` metadata, but the current conversation-info payload does not include the sticker file id.

## Solution Assessment

No, not yet. Per-turn untrusted conversation metadata is the right boundary, but the proposed `sticker_file_id` key must line up with accepted sticker send arguments before the PR reliably enables sticker replay.
