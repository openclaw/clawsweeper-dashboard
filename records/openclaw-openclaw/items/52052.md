## Summary

Keep open: the PR addresses a real Telegram sticker-context gap, but it is not merge-ready because the new prompt key does not match the accepted sticker send arguments and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `sticker_file_id` to Telegram inbound per-turn conversation metadata and adds focused tests that keep it out of stable system metadata and direct webchat turns.

## Best Possible Solution

Keep the sticker id in per-turn untrusted conversation metadata, but either surface an accepted key or add and advertise a `sticker_file_id` alias so the received id can drive a Telegram sticker reply.

## Reproduction Assessment

Yes, source-reproducible: construct a Telegram direct-chat `TemplateContext` with `Sticker.fileId` and inspect current main’s `buildInboundUserContextPrefix()`; it has no sticker file-id projection while Telegram ingress already carries `Sticker.fileId`. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Per-turn untrusted conversation metadata is the right boundary, but the surfaced key must align with accepted `message`/Telegram sticker action arguments before the PR reliably solves sticker replay.
