## Summary

Current main still routes Telegram message-tool edits through text-message editing, so the reported media-caption and button-only edit failure remains valid and narrowly fixable in the Telegram plugin.

## What This Changes

The issue requests Telegram `edit` action support for media captions and inline-keyboard-only updates via `editMessageCaption` and `editMessageReplyMarkup`.

## Best Possible Solution

Implement the fix inside the Telegram plugin edit path: route presentation-only edits to reply-markup editing, use caption editing for Telegram media-message failures, preserve existing text-edit behavior, and add focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main requires text for Telegram `edit` and then calls `editMessageText`, while Telegram/grammY expose separate caption and reply-markup edit methods for the reported media-button flow. I did not run a live Telegram bot repro in this read-only pass.

## Solution Assessment

Yes, the best fix is a plugin-local repair that reuses Telegram's standard edit APIs without changing the generic message-tool surface. The closed stale PR at https://github.com/openclaw/openclaw/pull/21898 is useful prior art, but current paths have moved and the action parser also needs presentation-only handling.

## Repair Work Prompt

Fix the Telegram message-tool edit action for media messages and presentation-only edits. In `extensions/telegram/src/action-runtime.ts`, allow `edit`/`editMessage` requests with presentation/buttons but no text to proceed, preserving inline button scope checks. In `extensions/telegram/src/send.ts`, preserve the existing `editMessageText` behavior for normal text edits, route button-only edits through `editMessageReplyMarkupTelegram`/`api.editMessageReplyMarkup`, and add an `editMessageCaption` path for media messages, preferably by handling Telegram's "there is no text in the message to edit" error while preserving HTML parse fallback, `MESSAGE_NOT_MODIFIED` success handling, retry/logging behavior, link-preview settings for text edits, and existing reply markup semantics. Use https://github.com/openclaw/openclaw/pull/21898 as prior art but adapt to the current `extensions/telegram` layout. Add focused regression tests in `extensions/telegram/src/action-runtime.test.ts`, `extensions/telegram/src/send.test.ts`, and the Telegram send test harness. Add a concise `CHANGELOG.md` Fixes entry if this repair lane owns landing; do not add unrelated refactors or change generic message-tool APIs.
