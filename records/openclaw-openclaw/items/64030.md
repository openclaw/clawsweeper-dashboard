## Summary

Keep open. Current main still rejects slash-bearing cron `session:<id>` targets even though custom session targets are documented and cron job IDs are UUID-backed; the related fix at https://github.com/openclaw/openclaw/pull/64057 was closed unmerged.

## What This Changes

Allow cron `session:<id>` targets to use channel-native conversation IDs that contain path separators, such as DingTalk group IDs.

## Best Possible Solution

Accept nonempty, non-NUL cron session-target payloads as opaque runtime session keys, keep path checks at run-log filename boundaries, and add DingTalk-style regression coverage.

## Reproduction Assessment

Yes. Importing the current `resolveCronSessionTargetSessionKey` helper and passing `session:agent:main:dingtalk:group:cid3tmd4xb19xjfk/wogxwy2a==` reproduces the `invalid cron sessionTarget session id` error on current main.

## Solution Assessment

Yes, with updated framing. Since current main already uses UUID cron job IDs, the narrow fix is to stop requiring session-target IDs to be path-safe while preserving run-log job-ID validation.
