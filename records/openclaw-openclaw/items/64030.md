## Summary

Keep open. Current main still rejects slash- and backslash-bearing cron `session:<id>` targets before job creation or patching, while the linked implementation PR at https://github.com/openclaw/openclaw/pull/64057 is closed unmerged.

## What This Changes

The issue requests cron support for channel-native session keys containing path separators by decoupling runtime session keys from filesystem-safe cron job/run-log IDs.

## Best Possible Solution

Accept opaque nonempty, non-NUL `session:<id>` payloads and `current`-resolved session keys at the cron session boundary while keeping filesystem-safe validation on run-log job IDs.

## Reproduction Assessment

Yes, by source inspection. A cron add or patch payload with `sessionTarget: "session:agent:main:dingtalk:group:cid3tmd4xb19xjfk/wogxwy2a=="` reaches `assertSafeCronSessionTargetId`, which throws on `/`, and current tests still assert separator rejection.

## Solution Assessment

Yes in direction, with one current-main adjustment. Since cron job IDs are already UUID-backed, the narrowest maintainable fix is to decouple runtime session-key validation from filename safety and keep strict checks only where actual run-log filenames are built.
