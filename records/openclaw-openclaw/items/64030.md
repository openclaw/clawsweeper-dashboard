## Summary

Keep open. Current main still rejects slash- and backslash-bearing cron `session:<id>` targets even though cron job ids are UUID-backed and run-log path safety is enforced separately; the related fix PR was closed unmerged.

## What This Changes

The issue requests a cron-boundary change so DingTalk/channel-native session keys containing `/` can be used as `session:<id>` targets without weakening run-log filename safety.

## Best Possible Solution

Accept opaque runtime session keys at cron session-target boundaries while keeping path-safety validation on run-log job ids.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `normalizeCronJobCreate` or gateway `cron.add` with `sessionTarget: "session:agent:main:dingtalk:group:cid3tmd4xb19xjfk/wogxwy2a=="`, which reaches `assertSafeCronSessionTargetId` and throws because the id contains `/`.

## Solution Assessment

Yes, with updated framing. Because current main already uses UUID job ids, the narrow maintainable fix is to stop requiring cron session-target ids to be path-safe while preserving the run-log filename guard, rather than changing global session-key normalization.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/64030. Current main still rejects DingTalk-style slash-bearing channel-native session keys in cron `session:<id>` targets even though cron job ids are UUID-backed and run-log paths validate `jobId` separately. Implement the narrow cron-boundary fix: treat explicit `session:<id>` payloads and resolved `current` session keys as opaque runtime session keys, preserve trimming and empty/NUL rejection, and do not relax `assertSafeCronRunLogJobId` or any actual filesystem path guard. Prefer accepting raw session keys at the session-target boundary; do not percent-encode persisted `sessionTarget` unless every resolution/delivery path decodes it back to the original runtime session key. Update regression coverage for `resolveCronSessionTargetSessionKey`, create/patch normalization, service create/update validation, gateway add/update, `current` binding with a slash-bearing context key, persisted/manual-run validation as appropriate, and run-log path safety. Add a user-facing `CHANGELOG.md` fixes entry crediting @BrilliantWang if policy allows. Do not change global `buildAgentSessionKey` normalization or migrate existing session keys.
