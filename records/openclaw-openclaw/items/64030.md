## Label Justifications

- `P2`: This is a normal-priority cron/channel bug with a clear source path and limited blast radius to persistent cron session targeting for native ids containing separators.
- `impact:session-state`: The failure is caused by treating valid runtime session keys as path-safe ids, blocking persistent session targeting for affected channel conversations.

## Summary

Keep open. Current main still rejects slash- and backslash-bearing cron `session:<id>` targets before create/update/runtime use, while docs describe `session:<id>` as an explicit persistent session key and the only linked fix PR is closed unmerged: https://github.com/openclaw/openclaw/pull/64057.

## What This Changes

The request is to let cron target channel-native session keys containing path separators without weakening run-log filename safety.

## Best Possible Solution

Implement a focused cron-boundary fix that treats `session:<id>` and `current`-resolved values as opaque runtime session keys while keeping empty/NUL rejection and strict run-log `jobId` filename validation.

## Reproduction Assessment

Yes, by source inspection. A cron add or patch payload with `sessionTarget: "session:agent:main:dingtalk:group:cid3tmd4xb19xjfk/wogxwy2a=="` reaches `assertSafeCronSessionTargetId`, which throws on `/`, and current tests still assert separator rejection.

## Solution Assessment

Yes in direction, but the current-main fix should decouple session-key validation from filename safety rather than change global session-key routing. Since cron job ids are UUID-backed, the narrow maintainable fix is to accept opaque session keys and keep strict checks where run-log filenames are built.
