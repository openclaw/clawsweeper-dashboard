## Label Justifications

- `P1`: A launchd-managed Gateway can leave the bundled iMessage channel unavailable for affected macOS users on the latest release.
- `impact:security`: The reported path involves Messages database access, Full Disk Access, Automation consent, and the process identity used for TCC grants.
- `impact:message-loss`: The iMessage channel fails before receive/send readiness, so messages may not be processed through the configured channel.
- `impact:crash-loop`: Provider startup repeatedly times out and can leave a stuck `imsg rpc` child process alive after readiness failure.

## Summary

Keep open. Current main and the shipped v2026.5.12 release still use the reported launchd-to-Node-to-stdio `imsg rpc` readiness path, the discussion now includes another matching LaunchAgent repro plus a partial independent-daemon workaround, and the broader macOS TCC tracker does not fully cover this issue's iMessage-specific timeout and stuck-child evidence.

## What This Changes

The issue asks OpenClaw to stop iMessage `imsg rpc` readiness hangs when Gateway is launched by macOS launchd, either by making it work or by failing with an actionable error.

## Best Possible Solution

Make LaunchAgent iMessage startup either run under a supported macOS TCC/session identity or fail quickly with actionable launchd/TCC guidance while reliably reaping hung `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The issue includes detailed v2026.5.12 logs and a second matching report, and current main still exercises the same LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, but this review did not run a real macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but maintainers still need to choose whether OpenClaw supports bare launchd, an app/session-owned launcher, or an explicit remote RPC endpoint pattern for this channel.
