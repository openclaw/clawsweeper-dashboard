## Label Justifications

- `P1`: A launchd-managed Gateway can leave the bundled iMessage channel unavailable for affected macOS users on the latest release.
- `impact:message-loss`: The reported startup hang prevents the iMessage channel from reaching receive/send readiness, suppressing channel delivery.
- `impact:security`: The path involves Messages database access, Full Disk Access, Automation consent, and a child process holding `chat.db` files open.
- `impact:crash-loop`: The provider startup path times out and can leave a stuck `imsg rpc` child process alive after readiness failure.

## Summary

Keep open. Current main still uses the reported LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, the issue now has a second matching LaunchAgent report plus an independent-daemon workaround discussion, and the broader macOS TCC tracker does not cover this issue's iMessage-specific timeout and stuck-child evidence.

## What This Changes

The issue reports that the bundled iMessage channel's `imsg rpc` readiness check times out under the macOS LaunchAgent while the same setup works from an interactive session.

## Best Possible Solution

Make LaunchAgent iMessage startup either use a supported macOS TCC/session identity or fail quickly with actionable launchd/TCC guidance while reliably reaping hung `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction in this review. The issue includes detailed v2026.5.12 logs plus a second matching report, and current main still exercises the same LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but maintainers still need to choose whether OpenClaw supports bare launchd, an app/session-owned launcher, or an explicit remote RPC endpoint pattern for this channel.
