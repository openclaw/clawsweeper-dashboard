## Summary

Keep open. Current main and the public v2026.5.12 source still use the reported Gateway-spawned `imsg rpc` readiness path, and the new workaround comment adds useful evidence that a separately launched `imsg` service avoids the inherited launchd process context rather than proving the OpenClaw path is fixed.

## What This Changes

The issue reports that the bundled iMessage channel’s `imsg rpc` readiness check hangs when Gateway is started by the macOS LaunchAgent.

## Best Possible Solution

Define the supported macOS iMessage launch path: either make Gateway-spawned `imsg` under LaunchAgent fail fast with TCC/session-specific diagnostics and verified child cleanup, or intentionally add a documented service/endpoint model with config and tests.

## Reproduction Assessment

No high-confidence live reproduction was established. The issue body gives concrete v2026.5.12 LaunchAgent logs, and current source still starts readiness by spawning `imsg rpc` and calling `chats.list`, but this review did not run a real macOS LaunchAgent with imsg 0.8.2.

## Solution Assessment

Unclear. A narrow diagnostic and child-cleanup fix may be possible, but choosing bare launchd, signed app/session ownership, Terminal-backed startup, or a new RPC endpoint/service mode is a maintainer product and security decision.
