## Label Justifications

- `P1`: The report and follow-up comment describe a broken bundled iMessage channel startup path affecting launchd-managed macOS users on the latest release.
- `impact:crash-loop`: The affected startup path hangs or times out during channel readiness and can leave an `imsg rpc` child process alive.
- `impact:security`: The issue crosses macOS Full Disk Access, Messages database access, and Automation process-ownership boundaries.

## Summary

Keep open. Current main and v2026.5.12 still use the reported launchd/Node/Gateway to stdio `imsg rpc` readiness path with no first-class endpoint config, and the latest human comment adds another same-release repro. The broader macOS TCC tracker is related but does not supersede this issue's distinct iMessage RPC hang and stuck-child evidence.

## What This Changes

The issue asks OpenClaw to make the bundled iMessage `imsg rpc` readiness path work under the macOS LaunchAgent, or fail quickly with actionable launchd/TCC/session guidance.

## Best Possible Solution

Keep this open and define a supported macOS iMessage launch contract: either make launchd-started Gateway spawn `imsg` in a process identity that works, or fail quickly with clear TCC/session guidance while reliably cleaning up stuck `imsg` children.

## Reproduction Assessment

No live high-confidence macOS reproduction was run. Source inspection confirms current main and v2026.5.12 still exercise the same `imsg rpc` child-process readiness path that matches the reported `imsg rpc timeout (chats.list)` logs.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but supporting an external RPC endpoint or changing the Gateway's macOS process owner needs maintainer product and security judgment.
