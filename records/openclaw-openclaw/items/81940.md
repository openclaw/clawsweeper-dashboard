## Summary

Keep open. Current main still launches the iMessage provider by spawning `imsg rpc` over stdio from the Gateway process and still probes readiness via `chats.list`, so the reported LaunchAgent hang is not obsolete; the workaround comment points to a possible architecture direction but does not resolve the bundled path.

## What This Changes

The issue requests that the bundled iMessage provider either succeed or fail actionably when `imsg rpc` is started from the macOS LaunchAgent-managed Gateway.

## Best Possible Solution

Keep this open and make LaunchAgent iMessage startup either work under a supported macOS process identity or fail quickly with clear launchd/TCC/session guidance while reliably terminating stuck `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The report includes detailed v2026.5.12 logs and current-main source still exercises the same LaunchAgent-to-stdio `imsg rpc` readiness path, but this review did not run a real macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A narrow timeout cleanup and diagnostic improvement may be possible, but deciding whether OpenClaw should support bare launchd, an app/session-owned launcher, or a daemon/RPC endpoint alternative needs maintainer product and security judgment.
