## Summary

Keep open. Current main still uses the reported macOS LaunchAgent -> Node Gateway -> stdio `imsg rpc` readiness path, the latest release still has the same iMessage RPC startup behavior, and the separate-launchd `imsg-server` workaround comment points to an unresolved product/security decision rather than making the report obsolete.

## What This Changes

The issue reports that bundled iMessage startup hangs on `imsg rpc` readiness when Gateway is launched by the default macOS LaunchAgent.

## Best Possible Solution

Make LaunchAgent iMessage startup either work under a supported macOS process identity or fail quickly with clear launchd/TCC/session guidance, while reliably cleaning up stuck `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The issue includes detailed v2026.5.12 logs and current-main source still exercises the same LaunchAgent-to-stdio `imsg rpc` readiness path, but this review did not run a real macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but deciding whether OpenClaw should support bare launchd, an app/session-owned launcher, or an RPC endpoint alternative needs maintainer product and security judgment.
