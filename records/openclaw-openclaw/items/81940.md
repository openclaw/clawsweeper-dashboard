## Summary

Keep open. Current main still runs the iMessage channel through the reported LaunchAgent-to-`imsg rpc` stdio readiness path, the latest stable tag still has the same client/probe behavior, and the newer workaround comment adds useful architecture evidence rather than making the issue obsolete.

## What This Changes

The issue reports that the bundled iMessage provider times out waiting for the first `imsg rpc` readiness response when Gateway is started by the macOS LaunchAgent.

## Best Possible Solution

Keep this open and make LaunchAgent iMessage startup either work under a supported macOS process identity or fail fast with clear launchd/TCC/session guidance while reliably cleaning up stuck `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction in this review. The report provides detailed current-release logs and the current-main source path matches them, but a real macOS LaunchAgent with `imsg` 0.8.2 or newer was not run here.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but deciding whether OpenClaw should support bare launchd, an app/session-owned launcher, or an RPC endpoint alternative needs maintainer product and security judgment.
