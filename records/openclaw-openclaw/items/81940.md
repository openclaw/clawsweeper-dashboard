## Label Justifications

- `P1`: A launchd-managed Gateway can leave the bundled iMessage channel unavailable for affected macOS users on the latest release.
- `impact:message-loss`: The issue is about iMessage channel startup failing before receive/send readiness, which can suppress channel delivery.
- `impact:security`: The reported path involves Messages database access, Full Disk Access, Automation consent, and a child process holding chat.db files open.
- `impact:crash-loop`: The provider startup path hangs/times out and can leave a stuck `imsg rpc` child process alive after readiness failure.

## Summary

Keep open. Current main still uses the reported LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, the latest comments add another matching LaunchAgent repro plus a partial independent-daemon workaround, and the broader TCC tracker does not fully cover this issue’s iMessage RPC hang and stuck-child evidence.

## What This Changes

The issue reports that the bundled iMessage channel’s `imsg rpc` readiness probe hangs when Gateway is started by the macOS LaunchAgent.

## Best Possible Solution

Make LaunchAgent iMessage startup either run under a supported macOS TCC/session identity or fail quickly with actionable launchd/TCC guidance while reliably reaping hung `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The issue includes detailed v2026.5.12 logs plus a second matching report, and current main still exercises the same LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, but this review did not run a real macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but maintainers still need to choose whether OpenClaw supports bare launchd, an app/session-owned launcher, or an explicit remote RPC endpoint pattern for this channel.
