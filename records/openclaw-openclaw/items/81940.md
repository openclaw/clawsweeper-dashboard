## Summary

Keep open. Current main and v2026.5.12 still use the reported Gateway-child `imsg rpc` readiness path, the newer imsg-server workaround is not an OpenClaw config/runtime path, and the remaining fix needs maintainer direction on the supported macOS launchd/TCC/session model.

## What This Changes

The issue reports that the bundled iMessage channel's `imsg rpc` readiness check times out under the default macOS LaunchAgent and can leave a child process alive, while interactive starts and a separate imsg-server workaround succeed.

## Best Possible Solution

Keep this open and make LaunchAgent iMessage startup either work through a supported macOS process identity or fail fast with explicit launchd/TCC/session guidance while reliably cleaning up stuck imsg children.

## Reproduction Assessment

No high-confidence live reproduction was established. The report gives current-release logs and source inspection confirms the same readiness path on current main, but this review did not run a real macOS LaunchAgent with imsg 0.8.2.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix may be safe, but deciding whether OpenClaw should support bare launchd, a signed app/session wrapper, or a separate imsg-server/TCP endpoint needs maintainer product and security judgment.
