## Summary

Keep open. Current main still uses the reported LaunchAgent to direct `imsg rpc` child-process readiness path, and the new `imsg-server` workaround is useful evidence for a possible product direction rather than a reason to close this issue.

## What This Changes

The issue reports that bundled iMessage `imsg rpc` readiness hangs when Gateway is started by the macOS LaunchAgent, with a later workaround suggesting an external `imsg-server` endpoint.

## Best Possible Solution

Define the supported macOS iMessage launch model: make LaunchAgent-started `imsg rpc` work, add an explicit supported external RPC endpoint/broker, or fail fast with actionable TCC/session diagnostics and reliable child cleanup.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The reporter provides detailed v2026.5.12 logs and current source still follows the same LaunchAgent-to-`imsg rpc` readiness path, but I did not run a real macOS LaunchAgent with imsg 0.8.2 or newer.

## Solution Assessment

Unclear. The separate `imsg-server` workaround is credible evidence for one direction, but maintainers need to decide whether OpenClaw should support a TCP endpoint/broker, a signed app/session launch model, or a narrower diagnostic and cleanup fix.
