## Summary

Keep open. Current main and the latest stable release still use the reported macOS LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, and the separate `imsg-server` workaround in the comments confirms a real architecture question rather than making the report obsolete.

## What This Changes

The issue reports that the bundled iMessage channel times out during `imsg rpc` readiness when Gateway is launched by the macOS LaunchAgent, while the same setup works from an interactive session.

## Best Possible Solution

Define the supported macOS iMessage LaunchAgent path: either make the subprocess readiness path work under an approved TCC/session owner or fail fast with clear launchd guidance while reliably terminating stuck `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The report includes concrete release logs and current-main source still exercises the same LaunchAgent-to-stdio `imsg rpc` readiness path, but this review did not run a real macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but deciding whether OpenClaw should support bare launchd, an app/session-owned launcher, or an RPC endpoint alternative needs maintainer product and security judgment.
