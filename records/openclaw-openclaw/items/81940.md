## Summary

Keep open. Current main and the latest stable release still use the reported macOS LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, and the newer workaround comment points to a real architecture choice rather than making the report obsolete. The broader TCC tracker remains relevant, but this issue has distinct iMessage RPC hang and child-process cleanup evidence.

## What This Changes

The issue reports that the bundled iMessage provider hangs during `imsg rpc` readiness when Gateway is started by macOS launchd, and asks for either successful startup or an actionable failure.

## Best Possible Solution

Keep this open and make LaunchAgent iMessage startup either work under a supported macOS process identity or fail quickly with clear launchd/TCC/session guidance while reliably cleaning up stuck `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The issue includes detailed v2026.5.12 logs and current-main source still exercises the same LaunchAgent-to-stdio `imsg rpc` readiness path, but this review did not run a real macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but deciding whether OpenClaw should support bare launchd, an app/session-owned launcher, or an RPC endpoint alternative needs maintainer product and security judgment.
