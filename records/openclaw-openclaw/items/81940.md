## Summary

Keep open. Current main still uses the same Gateway-spawned `imsg rpc` readiness path, and the report includes fresh launchd-specific evidence that is close to, but not cleanly replaced by, the broader open macOS TCC/iMessage issues.

## What This Changes

The issue reports that the bundled iMessage channel’s `imsg rpc` readiness check hangs when Gateway is launched by the macOS LaunchAgent.

## Best Possible Solution

Define the supported macOS iMessage LaunchAgent path, then either make `imsg rpc` readiness work in that signed/user-session context or fail fast with actionable launchd/TCC guidance while reliably cleaning up the stuck child process.

## Reproduction Assessment

No live high-confidence reproduction was run in this Linux checkout. The source path is reproducible: launchd-started Gateway still spawns `imsg rpc`, readiness still calls `chats.list`, and the reporter provided current-release logs for the timeout and surviving child process.

## Solution Assessment

Unclear. The issue identifies the right failure boundary, but the best fix needs maintainer macOS/iMessage judgment and live launchd proof before choosing between runtime support, upstream `imsg` changes, or a fail-fast diagnostic path.
