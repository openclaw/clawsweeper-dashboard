## Summary

Keep open. Current main still uses the reported LaunchAgent-to-stdio `imsg rpc` readiness path, has no native `rpcEndpoint`/server configuration, and the newer independent `imsg-server` workaround comment is useful context rather than a fix or duplicate of the broader macOS TCC tracker at https://github.com/openclaw/openclaw/issues/78049.

## What This Changes

Make bundled iMessage startup work or fail actionably when Gateway is launched by macOS LaunchAgent and `imsg rpc` readiness hangs.

## Best Possible Solution

Keep this open and define a supported macOS iMessage runtime path: either make LaunchAgent startup reliable under a supported process identity, or fail fast with actionable TCC/session guidance and verified cleanup of stuck `imsg` children.

## Reproduction Assessment

No, not as a high-confidence live reproduction in this review. The report has detailed v2026.5.12 logs and source inspection confirms the current readiness path, but I did not run a real macOS LaunchAgent with imsg 0.8.2.

## Solution Assessment

Unclear. A narrower diagnostic/cleanup fix may be possible, but deciding between supported launchd behavior, a signed app/session wrapper, or an endpoint/server mode needs maintainer product and security judgment.
