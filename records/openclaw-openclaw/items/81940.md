## Summary

Keep open. Current main still follows the reported macOS LaunchAgent-to-`imsg rpc` stdio readiness path, the latest stable tag has the same spawn/probe behavior, and the newer workaround comment is useful evidence for a possible architecture direction rather than proof that OpenClaw already supports it.

## What This Changes

This issue asks the bundled iMessage channel to avoid hanging when Gateway runs under macOS LaunchAgent, either by succeeding or failing with actionable diagnostics and cleanup.

## Best Possible Solution

Keep this open and make LaunchAgent iMessage startup either work under a supported macOS process identity or fail fast with clear launchd/TCC/session guidance while reliably cleaning up stuck `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The report has detailed current-release logs and current-main source still follows the same `imsg rpc` stdio readiness path, but no real macOS LaunchAgent with `imsg` 0.8.2 or newer was run here.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but deciding whether OpenClaw should support bare launchd, an app/session-owned launcher, or an RPC endpoint alternative needs maintainer product and security judgment.
