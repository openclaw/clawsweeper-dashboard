## Label Justifications

- `P1`: A launchd-managed Gateway can leave the bundled iMessage channel unavailable for affected macOS users during startup.
- `impact:security`: The issue involves Messages database access, Full Disk Access, Automation consent, and TCC process identity.
- `impact:message-loss`: The iMessage channel fails before send/receive readiness, suppressing channel delivery for affected installs.
- `impact:crash-loop`: The reported startup path hangs until timeout and can leave a long-lived `imsg rpc` child process with database files open.

## Summary

Keep open. Current main still starts bundled iMessage by spawning `imsg rpc` from the Gateway process and polling `chats.list` during the 30s readiness loop, while the issue discussion now has a second matching LaunchAgent repro plus an independent-daemon workaround that suggests the process context is still the unresolved part.

## What This Changes

The issue requests that LaunchAgent-started Gateways either complete bundled iMessage `imsg rpc` readiness or fail with actionable launchd/TCC diagnostics instead of timing out and leaving a stuck child.

## Best Possible Solution

Define the supported macOS LaunchAgent iMessage runtime: either make readiness run under a TCC-approved session identity, or fail quickly with actionable launchd/TCC guidance while reliably reaping hung `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The issue has detailed v2026.5.12 logs plus a second matching LaunchAgent report, and current main still exercises the same LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, but this review did not run a real macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but maintainers still need to choose whether OpenClaw supports bare launchd, an app/session-owned launcher, or a first-class external RPC endpoint pattern for this channel.
