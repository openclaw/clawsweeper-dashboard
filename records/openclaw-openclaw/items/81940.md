## Summary

Keep open. Current main and v2026.5.12 still use the reported macOS LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, and the independent `imsg-server` workaround is useful evidence for a real architecture decision rather than proof that the issue is resolved.

## What This Changes

The issue reports that the bundled iMessage provider's `imsg rpc` readiness check hangs when Gateway is started by the macOS LaunchAgent.

## Best Possible Solution

Define the supported macOS iMessage LaunchAgent model: either make startup work under an approved process identity or fail quickly with TCC/session guidance and always clean up stuck `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence current-main reproduction. The reporter supplied detailed v2026.5.12 logs and source still follows the same LaunchAgent-to-Node-to-stdio `imsg rpc` readiness path, but this review did not run a macOS LaunchAgent with `imsg`.

## Solution Assessment

Unclear. A bounded diagnostics and child-cleanup fix is likely valuable, but choosing whether OpenClaw should support bare launchd, a signed app/session-owned launcher, or an explicit RPC endpoint needs maintainer product and security judgment.
