## Label Justifications

- `P1`: A launchd-managed Gateway can leave the bundled iMessage channel unavailable for affected macOS users on the latest release.
- `impact:message-loss`: The failure happens before iMessage send/receive readiness, so channel delivery can be suppressed.
- `impact:security`: The reported path involves Messages database access, Full Disk Access, Automation consent, and macOS TCC process identity.
- `impact:crash-loop`: The startup path hangs or times out and can leave a stuck `imsg rpc` child process alive after readiness failure.

## Summary

Keep open. Current main and v2026.5.12 still use the reported launchd-managed Gateway to stdio `imsg rpc` readiness path, the latest comments add a second matching LaunchAgent repro plus a partial independent-daemon workaround, and the broader macOS TCC tracker does not cover this issue's iMessage-specific timeout and stuck-child evidence.

## What This Changes

The issue reports that a launchd-started macOS Gateway times out while probing bundled iMessage `imsg rpc` readiness and can leave the child process alive.

## Best Possible Solution

Make launchd iMessage startup either run under a supported macOS TCC/session identity or fail quickly with actionable launchd/TCC guidance while reliably reaping hung `imsg` children.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The issue includes detailed v2026.5.12 logs and another matching LaunchAgent report, and current main still exercises the same launchd-to-Node-to-stdio `imsg rpc` readiness path, but this review did not run the real macOS LaunchAgent setup.

## Solution Assessment

Unclear. A narrow diagnostics and child-cleanup fix is plausible, but maintainers still need to choose whether OpenClaw supports bare launchd, an app/session-owned launcher, or an explicit remote RPC endpoint pattern for this channel.
