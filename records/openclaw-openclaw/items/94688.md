## Label Justifications

- `P1`: A real Codex-backed fixed-session workflow can turn an already-generated assistant reply into an unavailable turn before channel delivery.
- `impact:session-state`: The failure is tied to fixed-session transcript compaction and post-turn session lifecycle state.

## Summary

Keep open. Current main and the latest release still await post-turn CLI transcript compaction before final delivery, and terminal compaction failures still throw; the issue now has multiple open fixing PRs that should be reviewed and landed or closed before this issue is resolved.

## What This Changes

The issue asks OpenClaw to keep an already-generated Codex/OpenAI fixed-session reply deliverable when later post-turn CLI transcript compaction fails.

## Best Possible Solution

Land a focused fix such as https://github.com/openclaw/openclaw/pull/94854 or an equivalent maintainer-selected PR that logs/records post-success compaction failure and continues normal delivery, with regression coverage for persisted reply plus compaction throw.

## Reproduction Assessment

Yes, source-reproducible: mock a successful CLI transcript persist, make post-turn compaction throw or return a failureReason, and delivery is skipped because current main awaits compaction before pending final delivery and deliverAgentCommandResult. I did not execute a live provider failure in this read-only pass.

## Solution Assessment

No, current main is not the best behavior because post-success maintenance can still fail a reply that already exists. The narrow maintainable fix is to make only post-success compaction non-fatal while preserving fail-closed behavior before a reply exists.
