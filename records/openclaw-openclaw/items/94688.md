## Label Justifications

- `P1`: A real Codex-backed agent/channel workflow can hide an already-generated assistant reply as unavailable after post-turn compaction fails.
- `impact:message-loss`: The generated assistant reply can fail to reach final channel delivery because compaction is awaited before delivery.
- `impact:session-state`: The failure is tied to fixed-session transcript compaction and recovery state after a reply is already persisted.

## Summary

Keep open. Current main still awaits post-success CLI transcript compaction before pending final delivery and channel delivery, and an open linked fix PR uses closing syntax for this issue, so the issue should remain open until one canonical fix lands.

## What This Changes

The issue requests that an already-generated and persisted Codex/OpenAI fixed-session reply stay deliverable when later post-turn CLI compaction fails.

## Best Possible Solution

Land one narrow canonical fix that logs or records post-success compaction failure and continues delivery, then close duplicate fix attempts after the canonical PR merges.

## Reproduction Assessment

Yes, source-level: current main persists the CLI transcript, awaits post-turn compaction, and only then persists pending final delivery and calls delivery. A mocked compaction throw after transcript persistence reproduces the unavailable-turn path without needing live provider credentials.

## Solution Assessment

Yes for the requested direction: the maintainable fix is to downgrade only post-success compaction failure after reply persistence, while preserving fail-closed behavior before a reply exists. The linked PR appears to target that boundary, but this issue should stay open until a reviewed fix merges.
