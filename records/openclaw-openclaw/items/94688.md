## Summary

Keep open. Current main and latest release v2026.6.8 still await fatal CLI transcript compaction after transcript persistence and before pending final/channel delivery, so the reported post-success UNAVAILABLE masking remains source-reproducible.

## What This Changes

Issue reports that a Codex/OpenAI fixed-session reply can be generated successfully, then a later post-turn CLI compaction failure surfaces the whole turn as unavailable.

## Best Possible Solution

Preserve generated reply delivery after post-turn compaction fails, record or log the session recovery state, and keep pre-reply/no-reply compaction failures fail-closed.

## Reproduction Assessment

Yes, source-level: mock successful CLI transcript persistence with payloads, make `runCliTurnCompactionLifecycle` reject, and observe that pending final delivery plus `deliverAgentCommandResult` are not reached.

## Solution Assessment

No, current main is not the best behavior because it lets post-success compaction failure override a generated reply; the narrow fix should downgrade only post-success compaction failure to recovery/logging before the next turn.
