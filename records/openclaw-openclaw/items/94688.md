## Label Justifications

- `P1`: A real Codex-backed channel workflow can still hide an already-generated assistant response as unavailable after post-turn compaction fails.
- `impact:message-loss`: The generated assistant reply can fail to reach final channel delivery because compaction is awaited first.
- `impact:session-state`: The failure is tied to fixed-session transcript compaction and recovery state after a reply is already in the transcript.

## Summary

Keep open. Current main still awaits post-success CLI transcript compaction before pending final delivery and channel delivery, so a compaction failure can still turn an already-generated reply into an unavailable turn.

## What This Changes

The issue reports that a Codex/OpenAI fixed-session reply can be generated and persisted, then a later post-turn compaction failure makes the whole turn surface as unavailable.

## Best Possible Solution

Make post-success CLI transcript compaction non-fatal once a reply is generated and persisted, while logging or marking session recovery before the next turn and preserving fail-closed behavior before a reply exists.

## Reproduction Assessment

Yes, source-level: mock a successful CLI transcript persist, make post-turn compaction return a failure reason or reject, and observe that pending final delivery and `deliverAgentCommandResult` are skipped because the compaction await happens first.

## Solution Assessment

No, current main is not the best behavior because post-success compaction remains fatal to a reply that is already available. The narrow maintainable fix is to downgrade that post-success failure to recovery metadata/logging before the next turn, not to weaken pre-turn compaction failures.
