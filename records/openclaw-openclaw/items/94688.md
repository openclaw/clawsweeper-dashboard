## Label Justifications

- `impact:message-loss`: The generated assistant reply can fail to reach final channel delivery because compaction is awaited first.

## Summary

Keep open. Current main still awaits post-turn CLI compaction before persisting pending final delivery and calling the delivery runtime, so a compaction throw can still turn an already-generated CLI reply into a failed turn. Several open fix PRs now target the same bug, with https://github.com/openclaw/openclaw/pull/94928 looking like the strongest current candidate rather than a reason to close before merge.

## What This Changes

The issue reports that a Codex/OpenAI fixed-session reply can be generated and persisted, then hidden from delivery when post-turn CLI transcript compaction fails.

## Best Possible Solution

Review and land one focused fix, preferably at the agent-command delivery boundary, that treats post-turn compaction failure as non-fatal only when a deliverable reply already exists and keeps no-reply compaction failures fatal.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: persist a CLI-backed reply transcript, then make post-turn `runCliTurnCompactionLifecycle` throw before pending final delivery is persisted. I did not run a live Codex/network repro in this read-only review.

## Solution Assessment

Yes, the best solution is to handle the post-turn maintenance failure at the lifecycle owner that knows whether a deliverable reply exists. Making the compaction helper globally non-fatal would be broader and could hide no-reply or session-health failures.
