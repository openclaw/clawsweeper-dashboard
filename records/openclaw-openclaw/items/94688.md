## Label Justifications

- `P1`: The issue reports a real agent/channel workflow where a completed assistant response can be hidden as unavailable after post-turn compaction fails.
- `impact:message-loss`: A generated assistant reply can fail to reach the downstream channel because compaction fails before final delivery runs.
- `impact:session-state`: The failure is tied to fixed-session transcript compaction and session recovery state after a reply is already written.

## Summary

Keep open. Current main and v2026.6.8 still run CLI transcript compaction after a successful CLI transcript persist but before pending final delivery and channel delivery; a compaction failure is thrown back through that path, matching the reported generated-reply-then-UNAVAILABLE failure.

## What This Changes

Prevent post-turn Codex/OpenAI CLI transcript compaction failures from converting an already-generated fixed-session reply into an unavailable turn.

## Best Possible Solution

Fix the post-success agent command ordering so a generated/persisted CLI reply is made durable and delivered even if later transcript compaction fails, while logging or marking the session for recovery before the next turn.

## Reproduction Assessment

Yes, source-level. A focused current-main test can mock a successful agent result plus persisted CLI transcript, then make `runCliTurnCompactionLifecycle` reject and observe that final delivery is never reached.

## Solution Assessment

No, current main is not the best behavior because post-success compaction failure still aborts the whole turn. The safer fix is to make post-turn compaction non-fatal once the assistant reply is available, without weakening pre-turn compaction failures.
