## Summary

Keep open. Current main and the latest release still await post-success CLI transcript compaction before pending final delivery, and compaction failure still throws, while several open linked PRs are candidate fixes that should be reviewed or landed before closing this issue.

## What This Changes

The issue requests that a Codex/OpenAI fixed-session reply already generated and persisted remain deliverable when later post-turn CLI transcript compaction fails.

## Best Possible Solution

Land one narrow reviewed fix that downgrades post-success CLI compaction failure to warning or recovery metadata after the reply is persisted, while preserving pre-reply fail-closed compaction behavior.

## Reproduction Assessment

Yes, source-level: persist a successful CLI reply transcript, make runCliTurnCompactionLifecycle throw or return a failure reason, and current main reaches the post-turn catch before pending final delivery or deliverAgentCommandResult. I did not run a live forced network-failure reproduction in this read-only pass.

## Solution Assessment

Yes, the best fix is the narrow runtime change proposed by the linked candidates: make only post-success compaction failure non-fatal after a reply is generated and persisted. A broader fallback or config option would add unnecessary product surface for a delivery-ordering bug.
