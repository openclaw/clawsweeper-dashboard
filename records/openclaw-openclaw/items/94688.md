## Label Justifications

- `P1`: A real Codex/OpenAI fixed-session workflow can hide an already-generated assistant reply as unavailable after post-turn compaction fails.
- `impact:message-loss`: The generated assistant reply can fail to reach final channel delivery because compaction is awaited first.
- `impact:session-state`: The failure is tied to fixed-session transcript compaction and recovery state after a reply has already been written.

## Summary

Keep open. Current main still persists the generated CLI transcript, then awaits post-turn compaction before pending final delivery and `deliverAgentCommandResult`, and there are open closing PRs rather than a merged fix.

## What This Changes

The issue reports that Codex/OpenAI fixed sessions can generate and persist a reply, then fail during post-turn CLI compaction and surface the turn as unavailable instead of delivering the reply.

## Best Possible Solution

Land one narrow lifecycle-owner fix that preserves delivery when a generated reply exists, keeps no-reply compaction failures fatal, and then supersede the sibling PRs.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence reproduction path: persist a CLI transcript with reply payloads, force `runCliTurnCompactionLifecycle` to throw, and delivery is skipped because compaction runs before pending final delivery and `deliverAgentCommandResult`.

## Solution Assessment

Yes. The best fix shape is a reply-gated catch in `agentCommandInternal`, which owns post-turn ordering and delivery, rather than making the compaction helper unconditionally swallow terminal failures.
