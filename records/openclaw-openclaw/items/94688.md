## Label Justifications

- `P1`: A successful generated reply can be hidden behind UNAVAILABLE in a live agent/channel workflow, which is urgent for affected long-lived sessions.
- `impact:message-loss`: The user-visible problem is that an already-generated assistant reply can be delayed, skipped, or surfaced as unavailable instead of delivered.
- `impact:session-state`: The failure happens after transcript persistence and during compaction/session maintenance, leaving transcript state and turn status out of sync.

## Summary

Keep open: current main still lets post-turn CLI compaction reject after a successful reply has been persisted, before normal delivery, so the reported UNAVAILABLE-after-success failure remains actionable.

## What This Changes

The issue asks OpenClaw to deliver a successfully generated Codex/OpenAI fixed-session reply even when later post-turn CLI compaction fails.

## Best Possible Solution

Make successful reply delivery independent from post-turn CLI compaction failures, while preserving a clear warning or recovery marker for the session and keeping true pre-generation failures fatal.

## Reproduction Assessment

Yes at source level: current main persists the successful CLI transcript, awaits post-turn compaction before delivery, and throws when compaction returns a failureReason. I did not replay the live macOS fixed-session failure in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to decouple delivery of a completed reply from post-turn compaction failure while recording the compaction problem for recovery. Adding a new config knob or broad background-worker redesign is not needed for the reported bug.
