## Label Justifications

- `P1`: A real Codex-backed fixed-session workflow can hide an already-generated assistant reply as unavailable after post-turn compaction fails.
- `impact:message-loss`: The generated assistant reply can fail to reach final channel delivery because compaction is awaited first.
- `impact:session-state`: The failure is tied to fixed-session transcript compaction and recovery state after a reply has already been written.

## Summary

Keep open. Current main and v2026.6.9 still await CLI compaction before final delivery, so a post-turn compaction throw can still suppress an already-generated reply; open linked PRs, especially https://github.com/openclaw/openclaw/pull/94928, now own the repair path.

## What This Changes

Prevent post-turn CLI compaction failures from turning an already-generated Codex/OpenAI fixed-session reply into an unavailable turn.

## Best Possible Solution

Land one narrow agent-command lifecycle fix that treats post-success compaction as non-fatal only when deliverable assistant text exists, while keeping pre-reply and no-reply compaction failures fatal.

## Reproduction Assessment

Yes. Source-level reproduction is clear: make CLI transcript persistence succeed with reply payloads, then make runCliTurnCompactionLifecycle throw and observe that current main reaches the post-turn error catch before pending final delivery or deliverAgentCommandResult.

## Solution Assessment

Yes. The best current fix shape is the reply-gated lifecycle fix in the agent-command owner path, not unconditional swallowing inside the compaction helper, because no-reply compaction failures should remain fatal.
