- `impact:session-state`: The failure is tied to fixed-session transcript compaction and recovery state after a reply has already been written.

## Summary

Keep open. Current main and the latest release still let a post-turn CLI compaction throw happen before pending final delivery and `deliverAgentCommandResult`, while open candidate PRs now own the exact repair path.

## What This Changes

The issue reports that a Codex/OpenAI fixed session can generate and persist a reply, then have post-turn CLI compaction fail and surface the whole turn as unavailable before delivery.

## Best Possible Solution

Land one narrow agent-command lifecycle fix that treats post-success compaction as non-fatal only when deliverable assistant text exists, while keeping pre-reply and no-reply compaction failures fatal.

## Reproduction Assessment

Yes. Source-level reproduction is clear: make CLI transcript persistence succeed with reply payloads, then make `runCliTurnCompactionLifecycle` throw and observe that current main reaches the post-turn error catch before pending final delivery or `deliverAgentCommandResult`.

## Solution Assessment

Yes, the best current fix shape is the reply-gated lifecycle fix in the agent-command owner path, not unconditional swallowing inside the compaction helper. That preserves honest compaction failure semantics while allowing an already-generated reply to be delivered.
