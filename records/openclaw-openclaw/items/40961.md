## Summary

Keep open, but do not merge as-is: current main already moved the announce default to 120 seconds, while this PR removes only one 60-second wait cap without adding a bounded timeout policy and leaves other A2A waits capped at 60 seconds.

## What This Changes

The PR raises the subagent announce default from 60s to 180s, removes the initial A2A wait's 60s cap, and updates the matching timeout test expectation.

## Best Possible Solution

Define one bounded A2A timeout policy, enforce it consistently for the initial wait and nested reply/announce waits, keep or deliberately change the current 120s default, and cover above-60-second plus timeout/fallback paths with focused tests.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: the changed line would pass `params.announceTimeoutMs` directly to `waitForAgentRun`, while `timeoutSeconds` accepts any nonnegative finite number and the wait helper forwards that duration to `agent.wait`. I did not run tests because this review was read-only and `pnpm` was unavailable in the environment.

## Solution Assessment

No. The patch is directionally related to the reported timeout pain, but removing only the initial cap without a replacement maximum is not the narrowest maintainable fix, especially after current main already changed the default to 120 seconds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:51`
  - body: This change lets `sessions_send.timeoutSeconds` drive the initial `agent.wait` duration without any upper bound. The schema accepts any nonnegative finite number and `waitForAgentRun` forwards the duration to the gateway wait, so large model-supplied values can hold the A2A flow for an excessive period. Add a shared maximum or schema/helper clamp before honoring waits above 60 seconds.
