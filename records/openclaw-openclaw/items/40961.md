## Summary

Keep open: current main does not fully implement this PR's intended A2A wait-cap change, and the proposed cap removal needs a bounded timeout policy before merge because it would allow excessive waits from valid tool input.

## What This Changes

The PR raises the subagent announce timeout default to 180 seconds, removes the 60-second cap from the `sessions_send` A2A wait phase, and updates the matching timeout test expectation.

## Best Possible Solution

Define one bounded A2A timeout policy on current main, enforce it consistently for delayed waits and nested reply/announce waits, and cover above-60-second plus timeout/fallback paths with focused tests.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: a large finite `sessions_send.timeoutSeconds` is schema-valid and this patch would pass the derived delayed A2A wait duration into `agent.wait` without a cap. I did not execute tests because this review was read-only.

## Solution Assessment

No. The patch is directionally related, but current main already uses a 120-second announce default and removing only this cap would leave unbounded and inconsistent A2A waits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:56`
  - body: This change lets `sessions_send.timeoutSeconds` drive the delayed A2A `agent.wait` duration without an upper bound. The schema accepts any nonnegative finite number and `waitForAgentRun` forwards the duration to the gateway with `timeoutMs + 2000`, so a large model-supplied value can hold the background A2A flow excessively. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
  - confidence: 0.9
