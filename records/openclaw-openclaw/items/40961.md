## Summary

Keep open: the PR targets a real A2A timeout reliability problem, but the patch is not safe to merge as-is because it removes the delayed wait cap without adding any upper bound and diverges from current main's overlapping 120s/default and async-timeout work.

## What This Changes

The PR raises the subagent announce default to 180s, removes the 60s delayed A2A wait cap, and updates the matching timeout test expectation.

## Best Possible Solution

Define one bounded A2A timeout policy on current main that honors intentional waits above 60 seconds, preserves current async announce-on-soft-timeout behavior, and covers delayed wait, reply, and announce paths with focused tests.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: `timeoutSeconds` accepts any nonnegative finite number and the patch would pass the derived delayed A2A wait duration into `agent.wait` without a cap. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction is related to the reported false timeouts, but raw cap removal without a shared maximum is not the narrowest maintainable fix on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:56`
  - body: Removing the `Math.min(..., 60_000)` cap lets `sessions_send.timeoutSeconds` drive the delayed A2A `agent.wait` duration, but the schema only requires a nonnegative number and `waitForAgentRun` forwards that duration with a gateway timeout of `timeoutMs + 2000`. A large model-supplied value can hold the background A2A flow excessively; add a clear maximum or shared clamp before honoring waits above 60 seconds.
  - confidence: 0.91
