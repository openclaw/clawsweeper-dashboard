## Summary

Keep open: the PR targets a real A2A timeout problem, but current main only partially overlaps it and the proposed cap removal would make model-supplied `sessions_send.timeoutSeconds` drive an unbounded initial wait.

## What This Changes

The PR raises the subagent announce timeout default to 180 seconds, removes the 60-second cap from the initial A2A wait, and updates the matching timeout test expectation.

## Best Possible Solution

Define one bounded A2A timeout policy on current main, enforce it through a shared helper or schema maximum, apply it consistently to the initial wait and nested reply/announce waits, and cover the above-60-second and timeout/fallback paths with focused tests.

## Reproduction Assessment

Yes for the PR-introduced defect: by inspection, a large finite `sessions_send.timeoutSeconds` is schema-valid and this PR would pass the derived wait duration directly into `agent.wait`. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No. The patch is directionally related to the timeout problem, but current main already uses a 120-second subagent announce default and removing only the initial wait cap would leave inconsistent and potentially excessive A2A waits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:50`
  - body: This change lets `sessions_send.timeoutSeconds` drive the initial `agent.wait` duration without an upper bound. The schema accepts any nonnegative finite number and `waitForAgentRun` forwards the duration to the gateway wait, so large model-supplied values can hold the A2A flow for an excessive time. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
