## Summary

Keep open, but do not merge as-is. The PR targets a real A2A timeout problem, yet current main has already moved the subagent announce default to 120s while the proposed cap removal would let `sessions_send.timeoutSeconds` drive an unbounded gateway wait; the remaining work needs a current-main timeout-policy rework and consolidation with overlapping A2A timeout reports.

## What This Changes

The PR raises the subagent announce timeout default and test expectation to 180 seconds and removes the 60-second cap from the initial `sessions_send` A2A wait phase.

## Best Possible Solution

Define one bounded `sessions_send` A2A timeout policy on current main, enforce it through a shared helper or schema maximum, apply it consistently to the initial wait and nested reply/announce waits, and add focused coverage for waits above 60 seconds plus fire-and-forget timeout behavior.

## Reproduction Assessment

Yes for the PR-introduced defect: by inspection, a large `sessions_send.timeoutSeconds` value is schema-valid, becomes `announceTimeoutMs`, and this PR would pass it directly into the initial `agent.wait`. I did not run tests because the review was explicitly read-only.

## Solution Assessment

No. The patch is directionally related to the reported timeout problem, but current main already uses a 120s subagent announce default, and removing only the first wait cap without a replacement bound leaves inconsistent and potentially excessive A2A waits.
