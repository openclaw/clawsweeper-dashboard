## Summary

This PR should close because the useful default-timeout part is already superseded by current main's 120s configurable subagent announce path, while the remaining sessions_send timeout-cap work is broader than this stale patch and is tracked by the open canonical timeout-policy issue.

## What This Changes

The PR raises the subagent announce default to 180s, removes the delayed sessions_send A2A 60s wait cap, and updates the matching timeout test expectation.

## Best Possible Solution

Use the canonical issue to define one bounded sessions_send A2A timeout policy on current main, keeping the shipped 120s configurable subagent announce path and covering initial wait, fire-and-forget, reply, and announce steps consistently.

## Reproduction Assessment

Yes. Source inspection shows current main still caps the delayed sessions_send A2A wait at 60s, and the PR head removes that cap while letting an unbounded timeoutSeconds-derived value drive agent.wait.

## Solution Assessment

No. The patch is directionally related, but raw cap removal lacks an upper bound, the 180s default is superseded by current main's 120s configurable path, and the broader sessions_send timeout policy is now tracked in https://github.com/openclaw/openclaw/issues/59339.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the delayed A2A wait before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: This change lets sessions_send.timeoutSeconds drive the delayed A2A agent.wait duration without an upper bound. The tool schema accepts any nonnegative finite number and the wait call extends the gateway RPC timeout from that value, so a large model-supplied value can hold the flow excessively. Add a shared maximum or bounded resolver before honoring waits above 60 seconds.
  - confidence: 0.9
