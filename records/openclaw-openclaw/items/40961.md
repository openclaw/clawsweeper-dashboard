## Summary

Keep open: the PR is related to a real A2A timeout problem, but current main has newer partial fixes and this patch still removes the delayed wait cap without a replacement bound.

## What This Changes

The PR raises the subagent announce timeout default from 60s to 180s, removes the 60s cap from the delayed `sessions_send` A2A wait, and updates the matching timeout test expectation.

## Best Possible Solution

Define one bounded A2A timeout policy on current main, honor legitimate waits above 60 seconds where intended, enforce a clear maximum, and cover sync, async, reply, and announce paths with focused tests.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: `timeoutSeconds` accepts any nonnegative finite number and this patch would pass the derived delayed A2A wait duration into `agent.wait` without a cap. I did not execute tests because this was a read-only review.

## Solution Assessment

No. The patch is directionally related, but raw cap removal lacks an upper bound, conflicts with current main's 120-second default, and leaves nested reply/announce waits capped elsewhere.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:56`
  - body: This change lets `sessions_send.timeoutSeconds` drive the delayed A2A `agent.wait` duration without an upper bound. The schema accepts any nonnegative finite number and `waitForAgentRun` forwards the duration to the gateway with `timeoutMs + 2000`, so a large model-supplied value can hold the background A2A flow excessively. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
  - confidence: 0.9
