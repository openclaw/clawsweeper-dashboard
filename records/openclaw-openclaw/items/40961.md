## Summary

Keep open: the PR targets a real A2A timeout problem, but current main has diverged and the patch removes the delayed wait cap without defining a bounded timeout policy.

## What This Changes

The PR raises the subagent announce default to 180s, removes the delayed `sessions_send` A2A wait cap, and updates the matching timeout test expectation.

## Best Possible Solution

Define one bounded A2A timeout policy on current main, honor legitimate waits above 60 seconds where intended, and cover sync, async, reply, and announce paths with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the cap removal lets an unbounded `timeoutSeconds`-derived value drive `agent.wait`; I did not execute tests because this was a read-only review.

## Solution Assessment

No. The patch is directionally related, but raw cap removal lacks an upper bound, conflicts with current main's 120-second default, and leaves nested reply/announce waits capped elsewhere.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound A2A wait time before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: This change lets `sessions_send.timeoutSeconds` drive the delayed A2A `agent.wait` duration without an upper bound. The schema accepts any nonnegative finite number and this call also extends the gateway RPC timeout to `waitMs + 2000`, so a large model-supplied value can hold the flow excessively. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `CHANGELOG.md:1`
  - body: This is a user-facing A2A reliability fix, but the PR does not update `CHANGELOG.md`. Add a single-line entry under the active `### Fixes` section so release notes capture the timeout behavior change.
  - confidence: 0.82
