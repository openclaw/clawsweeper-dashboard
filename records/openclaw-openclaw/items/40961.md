## Summary

Keep open and do not merge as-is. The PR is directionally related to the A2A timeout reports, but current main already moved the subagent announce default to 120 seconds and this patch removes the initial wait cap without defining a bounded timeout policy or addressing the remaining A2A timeout paths.

## What This Changes

This PR raises the subagent announce default to 180 seconds, removes the 60-second cap from sessions_send's initial A2A wait, and updates the timeout regression expectation.

## Best Possible Solution

Define one bounded A2A timeout policy, apply it consistently to initial waits and nested announce/reply waits, and cover the above-60-second plus timeout/fallback paths with focused regression tests.

## Reproduction Assessment

Yes for the PR blocker: by source inspection, a large finite sessions_send.timeoutSeconds is accepted by the schema and this PR would pass the derived duration into agent.wait without an upper bound. I did not run tests because this review was read-only.

## Solution Assessment

No. The patch is related to the timeout bug, but removing the cap without a maximum is not the narrowest maintainable fix and current main already uses a 120-second announce default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:50`
  - body: This change lets sessions_send.timeoutSeconds drive the initial agent.wait duration without an upper bound. The schema accepts any nonnegative finite number and waitForAgentRun forwards that duration to the gateway wait, so large model-supplied values can hold the A2A flow for an excessive time. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
