## Summary

Keep open. Current main has the isolated Telegram ingress worker and backlog-health reporting, but the reported media path can still remain coupled to an awaited downstream Telegram handler, inbound debounce flush, and image-tool turn; the report is recent, concrete, and reproduced by the reporter on the latest shipped release.

## What This Changes

The issue reports that sending Telegram image or video input on Windows can stall image processing, stop Telegram polling, and wedge gateway health until session files are manually removed.

## Best Possible Solution

Make Telegram media ingress resilient when downstream image analysis stalls: polling should keep fetching and spooling, health should stay accurate, stalled sessions should recover without deleting sessions.json, and same-chat ordering should remain intact.

## Reproduction Assessment

No high-confidence live reproduction was run. The reporter's logs plus current source provide a plausible source-level path through Telegram media dispatch, serialized inbound debouncing, runInboundReplyTurn, and image-tool execution, but Windows/Telegram proof is still needed.

## Solution Assessment

Unclear. The current isolated polling spool is the right direction for getUpdates durability, but it is only a partial mitigation for image-tool stalls because downstream media turns can still block processing and gateway health.
