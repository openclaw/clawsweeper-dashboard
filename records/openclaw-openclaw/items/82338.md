## Summary

Keep open. The report describes a current Telegram/gateway responsiveness failure, and current main still has a plausible path where Telegram image handling awaits a full media/tool turn on the gateway side; the newer isolated polling spool is a partial mitigation, not proof that Windows image-tool stalls are fixed.

## What This Changes

The issue reports that Telegram image/video handling can starve the gateway event loop and leave polling plus session recovery wedged on Windows.

## Best Possible Solution

Keep Telegram polling and gateway health responsive while image/media tool turns run, preserve per-chat/per-topic ordering, and make stalled Telegram sessions recover without deleting sessions.json.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection and the reporter's logs provide a plausible current-main path through Telegram media dispatch, awaited inbound turns, and image-tool buffer/provider work, but Windows/Telegram proof is still needed.

## Solution Assessment

The current isolated polling spool is not a complete solution because it mainly keeps Bot API ingress observable and marks backlog stalls unhealthy. The best fix should decouple or fail-safe long image/media turns so the gateway and Telegram session recovery remain responsive.
