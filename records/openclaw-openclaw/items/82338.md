## Summary

Keep open. Current main has important Telegram polling isolation from the merged ingress-spool work, but source inspection still shows Telegram media handling can stay coupled to an awaited downstream agent/image-tool turn, and the report is recent with concrete Windows/event-loop starvation logs after the latest shipped release.

## What This Changes

The issue reports that Telegram image/video input can stall gateway health and Telegram message processing on Windows while the image tool is active.

## Best Possible Solution

Make Telegram media ingress resilient when downstream image analysis stalls: polling should keep fetching/spooling, gateway health should stay responsive, stalled sessions should recover without deleting sessions.json, and same-chat ordering should remain intact.

## Reproduction Assessment

No high-confidence live reproduction was run. The reporter's logs plus current source provide a plausible source-level path through Telegram media dispatch, serialized inbound debouncing, runInboundReplyTurn, and image-tool execution, but Windows/Telegram proof is still needed.

## Solution Assessment

Unclear. The current isolated polling spool is the right direction for getUpdates durability, but it is only a partial mitigation for image-tool stalls because downstream media turns can still block processing and gateway health.
