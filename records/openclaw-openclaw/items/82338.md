## Summary

Keep open. The report is a current availability bug with plausible current-main source paths: Telegram media handling can still await an inbound agent/tool turn, and the image tool runs media load/optimization/provider work inside that turn; existing isolated polling and backlog-health work are useful mitigations but do not prove Windows image-tool stalls or gateway health timeouts are fixed.

## What This Changes

The issue requests that Telegram image/video processing not block polling, gateway health, or session recovery after the image tool runs.

## Best Possible Solution

Make Telegram polling, gateway health, and session recovery remain responsive while image/media tool turns run, with focused regression tests and preferably Windows or real Telegram proof for the stalled-image path.

## Reproduction Assessment

No high-confidence live reproduction was run. The reporter's logs plus current source provide a plausible current-main path through Telegram media dispatch, serialized inbound debounce, runInboundReplyTurn, and image-tool work, but Windows/Telegram proof is still missing.

## Solution Assessment

Unclear. Existing isolated polling/backlog-health work is a partial mitigation, but the best fix should decouple or fail-safe long image/media turns without breaking Telegram ordering or recovery semantics.
