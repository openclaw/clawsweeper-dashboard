## Summary

Keep open. Current main has the isolated Telegram ingress/spool mitigation from the related polling-stall fix, but the Telegram media path can still await a downstream inbound agent/image turn, and this recent Windows report includes concrete event-loop starvation logs after the latest shipped release.

## What This Changes

Reports that sending Telegram image/video media on Windows can freeze gateway responsiveness and stop later Telegram polling/dispatch until stalled session state is manually removed.

## Best Possible Solution

Make Telegram media ingress resilient when image analysis stalls: polling should keep fetching/spooling, health should remain responsive, stuck media turns should recover or fail visibly, and users should not need to delete sessions.json.

## Reproduction Assessment

No high-confidence live reproduction was run. The reporter's logs plus current source provide a plausible source-level path through Telegram media dispatch, serialized inbound debouncing, runInboundReplyTurn, and image-tool execution.

## Solution Assessment

Unclear as a complete fix. The existing isolated ingress worker is the right partial mitigation for getUpdates durability, but the remaining best fix should add a deterministic stuck-media regression and recover the downstream turn without broad channel semantics churn.
