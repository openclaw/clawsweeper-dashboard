## Summary

Keep open. Current main contains the merged Telegram isolated-ingress mitigation from https://github.com/openclaw/openclaw/pull/81746, but source inspection still shows Telegram media handling can remain coupled to an awaited downstream image/agent turn, and the recent Windows report includes concrete event-loop starvation logs after the latest shipped release.

## What This Changes

The issue reports that Telegram image/video media on Windows can trigger image-tool processing that stalls polling, gateway health, and subsequent message delivery until session state is manually removed.

## Best Possible Solution

Make Telegram media ingress resilient when downstream image analysis stalls: polling and gateway health should remain responsive, same-chat ordering should be preserved, and stuck sessions should recover without deleting sessions.json.

## Reproduction Assessment

No high-confidence live reproduction was run. The reporter's logs plus current source provide a plausible source-level path through Telegram media dispatch, serialized inbound debouncing, awaited channel dispatch, and image-tool execution, but Windows/Telegram proof is still needed.

## Solution Assessment

Unclear. The current isolated polling spool is the right direction for getUpdates durability, but it is only a partial mitigation for image-tool stalls because downstream media turns can still block a lane and leave the gateway unhealthy.
