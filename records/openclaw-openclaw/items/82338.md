## Summary

Keep open. Current main has partial mitigations for Telegram polling stalls and session recovery, but the central reported Windows image-tool freeze is still credible: Telegram media dispatch can remain coupled to an awaited same-lane agent/image turn, and no merged PR or release evidence shows this issue is fully fixed.

## What This Changes

Telegram image/video inbound should not freeze polling, gateway health, or session recovery when the image tool stalls on Windows.

## Best Possible Solution

Make Telegram media ingress resilient to stalled image analysis: polling should continue, same-lane backlog should surface and recover predictably, gateway health should remain responsive, and stalled sessions should not require manual sessions.json deletion.

## Reproduction Assessment

No high-confidence live reproduction has been run. Source inspection gives a plausible current-main path: Telegram media dispatch awaits a serialized inbound turn, and later same-lane work can block behind that unresolved handler.

## Solution Assessment

No, closing this as already fixed would not be the best path. The safest solution is a focused regression plus a narrow Telegram/media recovery change, with live Telegram or Windows proof if practical.
