## Summary

Keep open: current main still explains and preserves the reported Feishu group `queuedFinal=false, replies=0` behavior through the shared group visible-reply policy, and the remaining fix requires maintainer policy judgment plus real Feishu proof rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to restore or clarify Feishu group replies after 2026.5.3, where the agent generates final text but no Feishu group reply is delivered unless `messages.groupChat.visibleReplies` is set to `automatic`.

## Best Possible Solution

Decide the shared group visible-reply policy first, land a normal-level diagnostic or fallback through the canonical cross-channel tracker, and only accept a Feishu-specific legacy-default exception if maintainers approve it with redacted real Feishu proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: an unconfigured Feishu group turn with normal final assistant text and no message-tool send resolves to `message_tool_only`, suppresses automatic delivery, and produces `queuedFinal=false, replies=0`.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the narrow Feishu PR would help affected Feishu users but changes the documented group default, while a shared diagnostic/fallback path needs maintainer UX and privacy judgment.
