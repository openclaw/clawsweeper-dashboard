## Summary

Close as configuration/support: current main treats group/channel normal final replies as private by default, so Feishu can process the group turn and still log `queuedFinal=false, replies=0` unless the model sends via the message tool or `messages.groupChat.visibleReplies` is set to `automatic`.

## What This Changes

The report asks for Feishu group chats on 2026.5.3 with `groupPolicy: open` and `requireMention: false` to automatically send generated assistant replies back to the group.

## Best Possible Solution

Keep the current tool-only group reply default; operators who want 2026.5.2-style visible Feishu final replies should set `messages.groupChat.visibleReplies: "automatic"`, or use a model/runtime that reliably calls `message(action=send)`.

## Reproduction Assessment

Yes for the reported symptom shape: source inspection shows group/channel turns default to message-tool-only delivery and suppress normal final replies, producing `queuedFinal=false`/`replies=0` when the model answers privately. I did not run a live Feishu WebSocket account test.

## Solution Assessment

Yes: this is expected configuration policy rather than a Feishu send-path failure. The narrow solution is to set `messages.groupChat.visibleReplies: "automatic"` for legacy behavior, not to change the Feishu channel code.
