## Summary

Keep open: current main still explains the reported Feishu `queuedFinal=false, replies=0` behavior through the shared group/channel visible-reply policy, and fresh comments confirm the workaround is still needed on v2026.5.7. This remains an upgrade compatibility and product-policy decision, with an open Feishu PR candidate that still needs maintainer judgment and real behavior proof.

## What This Changes

The issue asks OpenClaw to restore or clarify Feishu group reply delivery after upgrading from 2026.5.2 to 2026.5.3+ changed generated group replies into `replies=0` unless `messages.groupChat.visibleReplies` is set to `automatic` or the model uses the message tool.

## Best Possible Solution

Make group reply visibility intentional for upgraded Feishu users: either accept a Feishu compatibility path that restores automatic replies for affected installs, or keep the message-tool default while adding visible diagnostics and Feishu/group migration guidance that points operators to the documented config.

## Reproduction Assessment

Yes from source inspection: a Feishu group turn reaches the shared group/channel reply path, which defaults to `message_tool_only` and suppresses ordinary final assistant text, matching `queuedFinal=false, replies=0`. I did not live-run Feishu, but current source, tests, and user confirmations make the reproduction path high confidence.

## Solution Assessment

Unclear: a Feishu-specific automatic default like the open PR is a narrow technical fix for unconfigured installs, but it intentionally diverges from the documented group/channel default. Maintainers need to choose compatibility, diagnostics, or documentation/migration before this should close.
