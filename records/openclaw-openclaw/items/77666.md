## Summary

Keep open: the reported Feishu symptom matches the current group/channel visible-reply policy, where ordinary assistant final text is processed but not posted unless the model uses the message tool or the operator opts back into automatic group replies. That looks like a recent upgrade/product-behavior break for this user, not a safe cleanup close.

## What This Changes

The issue asks OpenClaw to restore Feishu group-chat visible replies after upgrading to 2026.5.3, where group turns now log `replies=0` despite the agent generating assistant text.

## Best Possible Solution

Maintainers should decide whether Feishu upgrades should preserve legacy automatic group replies, add Feishu-specific upgrade guidance, or keep the new message-tool policy and direct affected users to `messages.groupChat.visibleReplies: "automatic"`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: a Feishu group turn reaches shared group/channel reply dispatch, which defaults to message-tool-only delivery and suppresses ordinary final assistant text, matching `queuedFinal=false, replies=0`.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the current implementation is internally consistent and documented in the group page, but restoring 2026.5.2-visible behavior for Feishu upgrades is a maintainer product/compatibility decision.
