## Summary

Keep open: current main still has a source-reproducible credential mismatch where marker-backed provider catalog discovery can reach `/models` with no runnable discovery key, and the only linked fix PR is closed unmerged. The reporter later questioned the exact production root cause, so the fix should stay narrow and covered by marker-specific tests rather than closing this as resolved.

## What This Changes

The issue asks OpenClaw to stop provider catalog discovery from sending unauthenticated `/models` requests when the resolved API key is only a non-secret auth marker.

## Best Possible Solution

Add a discovery-key-aware provider catalog guard that skips only marker-only dynamic `/models` discovery, preserves env-var and SecretRef/OAuth/SDK marker-backed catalogs, keeps authless local discovery intact, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a non-secret marker can pass the truthy catalog guard, resolve to no discovery key, and cause the downstream `/models` fetch to omit Authorization; I did not run a live LiteLLM repro in this read-only review.

## Solution Assessment

Yes for the direction, but not the closed PR's broad guard shape. The maintainable solution is a narrower discovery-key-aware guard with tests for marker-only, env-var-backed, SecretRef/OAuth marker, and authless local discovery behavior.
