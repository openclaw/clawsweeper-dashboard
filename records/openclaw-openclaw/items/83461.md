## Summary

Current main still has a source-reproducible auth-provider path: a truthy non-secret marker can satisfy the catalog guard while the runnable discovery key is undefined, and the linked fix PR was closed unmerged after the reporter questioned its exact scope. This should stay open for a narrower repair rather than close from the reporter’s conflicting later comments.

## What This Changes

The issue reports unauthenticated provider catalog `/models` discovery when provider API-key resolution returns a non-secret marker such as `custom-local` instead of a runnable discovery credential.

## Best Possible Solution

Add a discovery-key-aware guard for dynamic self-hosted `/models` discovery that skips true marker-only auth before network discovery while preserving env-var, SecretRef/OAuth/SDK static catalogs and intentionally authless local discovery paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-secret marker can pass the truthy catalog guard, `toDiscoveryApiKey()` removes the runnable key, and the downstream `/models` fetch omits Authorization when the key is undefined.

## Solution Assessment

Yes for the narrowed direction, not for the closed PR’s broad guard. The safer fix is to gate only dynamic discovery with no runnable discovery key, while leaving marker-backed static catalogs and env-ref discovery semantics intact.
