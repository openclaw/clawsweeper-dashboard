## Summary

Keep this issue open because current main and the latest release still have the reported provider-catalog auth gap, and an open linked PR explicitly closes it. The issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to skip provider catalog discovery when the resolved provider API key is a non-secret marker so `/models` discovery is not sent without Authorization.

## Best Possible Solution

Review and land the linked provider-catalog fix, or an equivalent guarded discovery fix, with regression coverage for marker-backed skip behavior in the shared catalog helpers and self-hosted discovery path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a truthy non-secret marker passes the current catalog guard, `toDiscoveryApiKey()` converts it to undefined, and discovery then omits Authorization; the reporter also supplied LiteLLM proxy logs matching that flow.

## Solution Assessment

Yes for the direction, but the exact linked PR still needs maintainer review. The maintainable fix is to treat non-secret markers as non-runnable discovery credentials wherever the catalog helpers decide whether to call `/models`, with regression tests covering the marker case.
