## Summary

Keep open: current main and the latest release still send the cron local-provider `GET /models` preflight without an Authorization header, and the newer same-author PR that tracks the corrected cron root cause is open but not merged.

## What This Changes

Fix provider/cron discovery or preflight so auth-required local OpenAI-compatible providers do not receive unauthenticated `GET /models` probes when OpenClaw already has usable provider credentials.

## Best Possible Solution

Land a corrected cron preflight fix that uses the selected cron agent context, shared provider auth/request contracts, non-secret marker filtering, and focused regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes at source level: current main sends the local-provider preflight `GET /models` without headers, and the linked PR includes LiteLLM logs showing 401 before and 200 after. I did not run a live LiteLLM instance in this read-only review.

## Solution Assessment

The direction is right, but the current open implementation is not yet the best fix because it still needs scoped provider auth, marker filtering, and cleanup. The maintainable solution should reuse existing auth/request contracts instead of duplicating credential parsing.
