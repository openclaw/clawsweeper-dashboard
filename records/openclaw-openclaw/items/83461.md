## Summary

Keep open: current main and the latest release still send the cron local-provider preflight `GET /models` without auth, and the active implementation path is the paired open PR rather than an already-landed fix. The original provider-catalog theory was narrowed by later discussion, but the same observable LiteLLM 401 problem remains source-reproducible in cron preflight.

## What This Changes

The issue reports unauthenticated `GET /models` probes against self-hosted OpenAI-compatible providers when provider auth handling does not produce a runnable request credential.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/83648 or a replacement that resolves cron preflight auth through the same scoped provider-auth contract as inference, filters non-secret markers, preserves authless local providers, and adds focused regression coverage.

## Reproduction Assessment

Yes at source level. Current main sends the cron local-provider `/models` probe with no headers, and the issue discussion includes LiteLLM logs showing the unauthenticated probe before an authenticated inference call.

## Solution Assessment

No for the original provider-catalog-only fix. The better solution is the cron preflight auth path now tracked by the paired open PR, but it must preserve scoped provider auth and marker filtering before merge.
