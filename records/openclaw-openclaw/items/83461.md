## Summary

Keep this issue open because current main still has the reported catalog-auth gap and an open linked PR explicitly closes it; the fix path should be reviewed before the issue is closed.

## What This Changes

The issue asks OpenClaw to skip provider catalog discovery when the resolved provider API key is only a non-secret marker, avoiding unauthenticated `/models` probes.

## Best Possible Solution

Land a focused fix that treats non-secret markers as non-runnable for catalog discovery across the shared catalog helpers and self-hosted provider discovery, with regression tests for the marker-backed skip case.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: resolve a provider API key to a truthy non-secret marker whose discovery key is undefined, then the current catalog path can proceed into discovery without an Authorization header. I did not run a live LiteLLM/vLLM proxy in this read-only review.

## Solution Assessment

Unclear for the linked PR as described: skipping discovery when only a marker is available is the right narrow direction, but the implementation should cover both the shared provider-catalog helpers and the self-hosted discovery helper used by vLLM/SGLang.
