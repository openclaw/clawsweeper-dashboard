## Summary

Keep open: current main and v2026.5.12 still have a source-backed catalog-auth gap, and the linked open fix PR at https://github.com/openclaw/openclaw/pull/83462 explicitly closes this issue.

## What This Changes

The issue reports that provider catalog discovery can send unauthenticated `GET /models` requests when a resolved API key is a non-secret local auth marker.

## Best Possible Solution

Land a focused fix that treats non-secret markers as non-runnable for catalog discovery across the shared catalog helpers and self-hosted provider discovery, with regression tests for marker-backed skip behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve a provider API key to a truthy non-secret marker with `discoveryApiKey` undefined, then the current discovery path can send `GET /models` without `Authorization`.

## Solution Assessment

Unclear for the linked PR as described. Skipping discovery when only a marker is available is the right narrow direction, but review should cover both the shared provider-catalog helpers and the self-hosted discovery helper path.
