## Summary

Keep open: current main still searches external channel secret-contract sidecars only at the plugin root, while the Discord docs support the reported env-backed SecretRef token shape and an open linked PR targets this exact gap.

## What This Changes

The issue reports that npm-installed `@openclaw/discord@2026.5.3` skips startup when `channels.discord.token` is an env-backed SecretRef because the external channel contract loader does not search compiled `dist/` sidecars.

## Best Possible Solution

Land a narrow fix that discovers compiled `dist/` channel secret-contract sidecars while preserving the existing plugin-root boundary checks, plus a regression test for the published npm layout.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a plugin record whose contract file exists only at `rootDir/dist/secret-contract-api.js` makes the current resolver return `null`, so no SecretRef assignment is queued; I did not run tests because this sweep is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to teach `resolvePluginContractApiPath` to recognize compiled `dist/` sidecars while preserving flat-layout compatibility and existing boundary-file checks.
