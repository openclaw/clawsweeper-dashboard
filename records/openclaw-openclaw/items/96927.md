## Summary

Keep open: the PR targets a real unbounded Comfy JSON read and uses the right SDK helper, but the current catch block masks helper overflow as a malformed-JSON error and the proof does not exercise the changed Comfy wrapper path.

## What This Changes

The PR imports `readProviderJsonResponse` and uses it in Comfy's shared workflow JSON reader for upload, prompt, status, and history responses.

## Best Possible Solution

Let the shared provider helper own both bounded reads and malformed JSON wrapping in Comfy, then merge only after maintainers accept the 16 MiB cap and the contributor supplies Comfy-wrapper proof.

## Reproduction Assessment

Yes from source: current main reads Comfy workflow JSON through `response.json()` without a byte cap, and the PR path still wraps helper overflow in the malformed-JSON catch. I did not run a live Comfy endpoint repro.

## Solution Assessment

No: routing through `readProviderJsonResponse` is the right layer, but the surrounding catch must be removed or narrowed so bounded overflow errors are not converted to malformed JSON.
