## Summary

Keep open: this is a useful availability hardening for Comfy JSON responses, but the current diff is not merge-ready because the bounded-reader overflow error is still caught and rewritten as malformed JSON, and the proof exercises the shared helper rather than the changed Comfy wrapper path.

## What This Changes

The PR imports `readProviderJsonResponse` into the Comfy workflow runtime and uses it in the shared JSON response reader for upload, prompt, status, and history responses.

## Best Possible Solution

Let `readProviderJsonResponse` own both bounded reads and malformed JSON wrapping for Comfy JSON responses, then merge only after maintainers accept the 16 MiB fail-closed cap for Comfy success JSON.

## Reproduction Assessment

Yes from source: current main reads Comfy JSON responses through `response.json()` without a byte cap in the shared helper. The proposed changed path is also source-reproducible as still wrapping helper overflow in the malformed-JSON catch.

## Solution Assessment

No: routing through the shared helper is the right layer, but the existing catch block must be removed or narrowed so bounded overflow errors are not converted to malformed JSON.
