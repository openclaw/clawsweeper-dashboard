## Summary

Keep open: current main and v2026.5.7 still use a 20k default compaction reserve floor, below the requested 25k-30k headroom, while safeguard quality guard only partly addresses malformed summaries. The exact long-session summary degradation was not live-reproduced, so the threshold change still needs maintainer validation and policy choice.

## What This Changes

The issue requests raising default compaction reserve-token headroom toward 25k-30k and optionally warning when summaries look like transcript tails.

## Best Possible Solution

Validate summary quality on realistic 200k-token sessions, choose a new default or relative reserve policy, then update focused tests, docs, and changelog; treat transcript-tail detection as a follow-up observability guard if needed.

## Reproduction Assessment

No high-confidence live reproduction was established. Source and docs do confirm the current effective default floor remains 20k, so the threshold concern is source-checkable but the 182k-token summary failure still needs real-session validation.

## Solution Assessment

Unclear. Raising the floor to 25k-30k may be the narrow fix, but current main already has a 20k floor, small-context caps, and safeguard quality guard behavior, so maintainers should choose the threshold after validating quality, cost, and compaction frequency.
