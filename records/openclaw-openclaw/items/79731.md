## Summary

Keep open: current main and the latest release still carry the Minimax portal catalog path that detects OAuth profiles by returning a provider marker instead of using the auth resolver seam; the exact fix PRs are closed unmerged, and the newer open OAuth refresh PR does not close or address this token-resolution path.

## What This Changes

The issue reports that Minimax portal OAuth profiles are written successfully but model requests still fail because the runtime does not resolve the stored OAuth token for API calls.

## Best Possible Solution

Patch the Minimax portal catalog to use the existing provider auth resolver seam for OAuth-backed catalog auth, with focused regression coverage and no unrelated provider or OAuth-refresh changes.

## Reproduction Assessment

No live high-confidence reproduction was established because this review did not run a real MiniMax OAuth request. Source inspection gives a clear current-main path: Minimax portal catalog discovery still emits the OAuth marker from profile presence while the resolver seam that should handle OAuth profiles already exists.

## Solution Assessment

Yes, the narrow maintainable fix is to use ctx.resolveProviderAuth(PORTAL_PROVIDER_ID, { oauthMarker: MINIMAX_OAUTH_MARKER }) in the Minimax portal catalog and add regression coverage. The repair should avoid endpoint migration, refresh behavior, or unrelated provider changes.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79731 narrowly. In current main, extensions/minimax/provider-registration.ts resolvePortalCatalog manually checks ensureAuthProfileStore/listProfilesForProvider and emits MINIMAX_OAUTH_MARKER for minimax-portal OAuth profiles. Replace that catalog auth selection with the existing ctx.resolveProviderAuth(PORTAL_PROVIDER_ID, { oauthMarker: MINIMAX_OAUTH_MARKER }) pattern used by Chutes, preserving explicit provider apiKey/baseUrl behavior as intentionally as possible. Add focused regression coverage in extensions/minimax/index.test.ts proving the portal catalog calls resolveProviderAuth with the marker and returns a usable catalog for OAuth profiles. Do not change MiniMax OAuth endpoints, refreshOAuth behavior, Google/provider defaults, or unrelated auth policy. Be aware that https://github.com/openclaw/openclaw/pull/82056 is adjacent MiniMax OAuth refresh work but does not close this token-resolution issue.
