## Summary

Keep open: current main and the latest shipped release still use the Minimax portal catalog path that detects OAuth profiles by returning the MiniMax OAuth marker directly, while the closed related fix PRs were not merged into main. The issue has a clear source-level reproduction path, but this read-only review did not run a live MiniMax OAuth request.

## What This Changes

Minimax portal OAuth profile tokens should be resolved for model requests instead of producing the "No API key found" runtime error.

## Best Possible Solution

Patch the Minimax portal catalog to use the existing provider auth resolver seam for OAuth profiles, add focused regression coverage, and keep endpoint refresh work separate unless a maintainer chooses to fold both into one reviewed Minimax OAuth change.

## Reproduction Assessment

No live high-confidence reproduction was established because this review did not run a real MiniMax OAuth request. The current source and latest release still show the marker-based catalog path with concrete user steps, so the failure is source-reproducible with the reported OAuth-profile setup.

## Solution Assessment

Yes, the narrow maintainable fix is to use `ctx.resolveProviderAuth(PORTAL_PROVIDER_ID, { oauthMarker: MINIMAX_OAUTH_MARKER })` instead of manually probing the auth store in the catalog path. The repair should add a regression test and avoid unrelated provider or endpoint changes.
