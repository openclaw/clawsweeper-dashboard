## Summary

Keep open: the Minimax Portal OAuth failure is still present on current main and the latest release, and both related fix PRs were closed unmerged. Current code still bypasses the existing catalog auth resolver seam used by comparable OAuth providers.

## What This Changes

Make MiniMax Portal OAuth profiles resolve to a usable runtime bearer token for `minimax-portal` model requests instead of failing with `No API key found`.

## Best Possible Solution

Land a focused Minimax plugin fix that uses the existing provider catalog auth resolver seam for portal OAuth, adds regression coverage, avoids unrelated provider changes, and verifies a real redacted MiniMax Portal request when feasible.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live reproduction in this review; source inspection shows current main and `v2026.5.12` still use the catalog marker path, but I did not run a real Minimax OAuth request.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow direction is to use `ctx.resolveProviderAuth(..., { oauthMarker })` in the Minimax portal catalog, matching the existing Chutes pattern, with a focused regression test and no unrelated provider changes.
