## Summary

Keep open: current main still has the Minimax portal catalog detect OAuth profiles by directly reading the auth store and returning the provider-owned marker, while the existing provider-auth resolver seam already supports OAuth marker resolution. The two linked fix PRs were closed unmerged, and one prior ClawSweeper review found the fix direction plausible but blocked that branch on an unrelated Google change and missing live proof.

## What This Changes

The issue asks for Minimax portal OAuth profiles written by login to be resolved and used for runtime model requests instead of failing with missing provider auth.

## Best Possible Solution

Patch the Minimax portal catalog to use the provider-auth resolver with an OAuth marker, add focused regression coverage, and avoid unrelated provider changes.

## Reproduction Assessment

Yes for source-level reproduction: current main still exposes the manual Minimax portal OAuth marker path, and the issue provides concrete CLI steps. I did not run a live OAuth request because this review is read-only and needs real Minimax credentials.

## Solution Assessment

Yes, the narrowest maintainable direction is to use the existing `ctx.resolveProviderAuth(..., { oauthMarker })` seam, matching the Chutes provider pattern, with a Minimax-specific regression test and no unrelated Google/default-model hunks.
