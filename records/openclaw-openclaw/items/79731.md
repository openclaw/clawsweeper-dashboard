## Summary

Keep open: current main still has the Minimax portal catalog path that the report and two closed unmerged PRs identify, and I did not find a merged fix or open replacement PR for this exact token-resolution failure.

## What This Changes

The issue reports that Minimax portal OAuth login stores credentials, but model requests still fail with `No API key found for provider "minimax-portal"`.

## Best Possible Solution

Keep the fix in the Minimax plugin catalog hook, use the existing `resolveProviderAuth` OAuth-marker seam, add focused regression coverage, and leave refresh-token work to the separate refresh issue.

## Reproduction Assessment

Unclear for a live request, but the source path is reproducible: current main still uses manual profile-store detection and marker injection for `minimax-portal`, while the existing resolver seam and Chutes pattern show the intended OAuth catalog path.

## Solution Assessment

Yes, the narrow fix direction is to call `ctx.resolveProviderAuth(PORTAL_PROVIDER_ID, { oauthMarker: MINIMAX_OAUTH_MARKER })` from the Minimax portal catalog and preserve explicit config/base URL behavior; a broader core auth change would cross the plugin boundary unnecessarily.
