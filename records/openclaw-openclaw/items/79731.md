## Summary

Keep open: current main and the latest release still use the Minimax portal catalog marker path that can advertise OAuth availability without resolving a request-ready credential, and the focused fix remains in an open, reviewed PR rather than merged code.

## What This Changes

The issue reports that Minimax Portal OAuth login writes credentials, but subsequent model requests still fail with a missing API key error.

## Best Possible Solution

Land a corrected narrow Minimax provider catalog fix through https://github.com/openclaw/openclaw/pull/82587 or a replacement, and keep the separate expired-token refresh work tracked at https://github.com/openclaw/openclaw/issues/77467.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolvePortalCatalog` can return the `minimax-oauth` marker for stored OAuth profiles, while runtime auth does not treat that non-secret marker as a direct request credential.

## Solution Assessment

Yes, the best fix belongs in the provider-owned Minimax catalog/auth path so stored OAuth profiles resolve through the existing auth seam before request execution. The linked PR is the right narrow work area, but it still needs correction/proof before this issue can close.
