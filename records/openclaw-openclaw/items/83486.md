## Label Justifications

- `P0`: The issue reports a security-boundary bypass where authenticated admin RPC on one gateway can execute against another gateway's control-plane context.
- `impact:security`: The affected behavior crosses the documented separate-gateway trust boundary for administrative RPC methods.
- `impact:session-state`: The bug is a request-context misassociation between live gateway instances in the same process.

## Summary

Keep this issue open: current main still has the reported source path, and the same author has an open linked fix PR with closing syntax. The issue is source-reproducible as a gateway/plugin context isolation bug and should close only after the linked fix lands or is replaced.

## What This Changes

The issue reports that authenticated admin HTTP RPC requests can dispatch gateway methods against the wrong live gateway instance when multiple gateway servers run in one Node process.

## Best Possible Solution

Land or replace the linked fix so plugin HTTP gateway-method dispatch carries the server-local `GatewayRequestContext` instead of relying on the process-global fallback.

## Reproduction Assessment

Yes from source inspection: plugin HTTP routes create a runtime scope without a gateway context, while gateway method dispatch falls back to a process-global resolver installed by `startGatewayServer`. I did not run a live two-gateway PoC because this review is read-only.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable path: bind plugin HTTP runtime scopes to the server-local gateway context. The issue should stay open until that PR is merged, revised, or superseded by an equivalent fix.
