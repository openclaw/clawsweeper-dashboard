## Summary

Keep open: this targets the right compaction bug, but the implementation only keys on a top-level `senderId` that current production user-turn messages do not carry, so the real multi-user message drop would remain.

## What This Changes

The PR adds an optional `senderId` field to duplicate-user-message compaction signatures and adds focused same-sender and different-sender dedup tests.

## Best Possible Solution

Use a maintainer-approved durable sender discriminator in user-turn transcript messages and have both compaction dedup consumers key on it while preserving same-sender retry pruning; if identity is unavailable, avoid sender-blind cross-user dedup.

## Reproduction Assessment

Yes, source-level: current main's `duplicateSignature` keys only on normalized text, and both compaction consumers feed it persisted `AgentMessage` or transcript `entry.message` values. I did not run a live multi-user compaction E2E in this read-only review.

## Solution Assessment

No. A sender-aware dedup key is the right direction, but this PR reads a field that production messages do not currently carry; the maintainable fix is to land or consume an approved durable sender discriminator first.
