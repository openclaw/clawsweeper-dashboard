## What This Changes

The issue reports continuously repeated `SECRETS_RELOADER_DEGRADED` logs for a `QCLAW_LLM_API_KEY` SecretRef that the reporter says is set in their shell but missing during gateway secret reloads.

## Best Possible Solution

Confirm the active gateway process env and configured SecretRef target first, then fix the managed env propagation path if current main loses a resolvable secret or document the supported `OPENCLAW_*`/daemon-readable setup if this is an unsupported legacy state.

## Reproduction Assessment

No for the exact race claim: we do not have live proof that current main loses an env var visible to the gateway process. Partially yes at source level: a missing env-backed SecretRef will produce repeated degraded warnings across repeated reload attempts while preserving last-known-good runtime state.

## Solution Assessment

Unclear: the suggested mutex/caching fix overlaps current serialization and last-known-good behavior. The safer direction is to prove whether the gateway process lacks the env var, uses an unsupported old env name, or still has a real reload race before changing core reload policy.
