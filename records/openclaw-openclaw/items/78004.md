## Summary

Keep open: the optimization is useful, but the current PR can skip existing native PreToolUse safety work when no ordinary before_tool_call hook is registered, so it needs a narrow repair before merge.

## What This Changes

The PR adds no-op command generation for Codex native hook relay events without matching global hooks and regression tests for relay versus no-op command selection.

## Best Possible Solution

Keep the no-handler relay optimization, but make the no-op decision account for all pre-tool relay work, especially trusted policies and session loop detection, then land with regression coverage.

## Reproduction Assessment

Yes for the PR regression: source inspection of the PR head shows pre_tool_use no-op selection based only on hasGlobalHooks("before_tool_call"), while current main performs loop detection and trusted policy/approval work before the ordinary hook gate. I did not run a live low-core stress test in this read-only review.

## Solution Assessment

No. The direction is narrow and useful, but the predicate models only registered hooks; it should preserve trusted policy, approval, and loop-detection work before skipping relay subprocesses.
