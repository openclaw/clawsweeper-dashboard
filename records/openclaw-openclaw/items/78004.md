## Summary

Keep this PR open: it targets a credible native Codex hook-relay fanout problem, but the current diff can skip existing pre-tool safety work whenever no ordinary `before_tool_call` hook is registered.

## What This Changes

The PR adds a native hook relay no-op command path for Codex native events without matching local hook handlers and adds regression tests for no-handler and matching-handler command generation.

## Best Possible Solution

Keep the no-handler relay optimization, but make the no-op decision preserve every pre-tool obligation and add regression coverage for trusted policies, plugin approval, and session loop detection before merge.

## Reproduction Assessment

Yes for the PR regression by source inspection: the proposed `pre_tool_use` predicate can return a no-op while current main's relay path runs loop detection, trusted policy checks, and approval. The original CPU fanout also has PR-body live-output proof, but I did not rerun the low-core stress scenario here.

## Solution Assessment

No, not as written. The optimization is a good narrow direction, but the predicate must keep `pre_tool_use` on the relay path whenever any loop-detection, trusted-policy, approval, or ordinary hook work can run.
