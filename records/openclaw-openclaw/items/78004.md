## Summary

Keep open; the CPU-reduction direction is valid and the PR includes useful live-host context, but the current diff would bypass existing native pre-tool safety work when no ordinary `before_tool_call` handler is registered.

## What This Changes

The PR changes Codex native hook relay command generation to emit cheap no-op commands for native events without matching local hook handlers and adds regression tests for no-op versus relay commands.

## Best Possible Solution

Keep the optimization, but make the local-work predicate cover every `pre_tool_use` path that can affect behavior, including trusted tool policies and loop detection, before allowing no-op command generation.

## Reproduction Assessment

Yes, from source inspection: with a trusted tool policy registered and no normal `before_tool_call` hook, the PR guard would return a no-op `pre_tool_use` command before the existing trusted-policy block path can run.

## Solution Assessment

No; the best fix is to avoid spawning relay subprocesses only when the full current-main relay path truly has no work, not only when ordinary plugin hook handlers are absent.
