## Summary

Keep open. The performance optimization is focused and the PR body has usable live host proof, but the current patch can skip existing Codex native pre-tool safety work when no ordinary hook handler is registered.

## What This Changes

The PR changes Codex native hook relay command generation to return a cheap no-op command for native events without matching local hook handlers and adds regression tests for relay versus no-op commands.

## Best Possible Solution

Keep the optimization, but return a no-op only when the full relay path truly has no work; pre_tool_use must still relay when loop detection, trusted policies, approval, or ordinary hooks can affect behavior.

## Reproduction Assessment

Yes, from source inspection: with trustedToolPolicies or session loop tracking present and no ordinary before_tool_call hook, the PR guard would return a no-op pre_tool_use command before current-main policy code can run. I did not run the PR branch or a live low-core performance stress test in this read-only review.

## Solution Assessment

No. The direction is good, but the predicate should model all existing relay work rather than only global hook registrations, otherwise the CPU reduction trades away a pre-tool safety path.
