## Summary

Keep open because the optimization is useful but currently skips existing pre-tool safety work when no ordinary hook is registered. The PR needs a narrower predicate that preserves loop detection, trusted tool policy blocks, and plugin approval before it can safely avoid spawning the relay subprocess.

## What This Changes

The PR adds a no-op native hook relay command for Codex native events without matching local hook handlers, keeps permission requests on the relay path, and adds focused command-selection tests.

## Best Possible Solution

Keep the no-handler relay optimization, but make the no-op decision account for every pre-tool obligation that `runBeforeToolCallHook` would perform, then cover trusted-policy and loop-detection cases in regression tests.

## Reproduction Assessment

Yes for the PR regression by source inspection: the new `pre_tool_use` predicate can return a no-op based only on ordinary hooks while current main performs loop detection, trusted policies, and approvals in the relay path. The original high-CPU fanout is supported by the PR body's live host proof, but I did not rerun that stress scenario.

## Solution Assessment

No, not as written. The narrow optimization is the right direction, but the predicate must preserve the current pre-tool safety pipeline before skipping relay subprocesses.
