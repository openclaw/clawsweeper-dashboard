## Summary

Keep open. The PR addresses a real Codex native hook relay CPU amplifier, but the proposed no-op guard can skip existing native pre-tool policy work and should not merge as written.

## What This Changes

The PR changes native Codex hook command generation to return no-op commands for selected native events when matching global hook handlers are absent, and adds regression coverage for relay versus no-op commands.

## Best Possible Solution

Keep the subprocess reduction, but make the no-op decision reflect all existing relay work for each event, including trusted pre-tool policy and loop-detection paths, then land it with regression coverage for both no-work and policy-work cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure trusted tool policy or loop tracking without an ordinary `before_tool_call` hook, then a Codex `pre_tool_use` event would receive the PR's no-op command before `runBeforeToolCallHook` can enforce policy.

## Solution Assessment

No. The optimization direction is narrow and useful, but the current predicate is not the best implementation because it models only global hook handlers rather than all relay-side work.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Keep pre-tool policies in the relay path:** `src/agents/harness/native-hook-relay.ts:354-356`

A Codex native tool call with only those policies would now get the no-op command and skip the existing block or approval path.
