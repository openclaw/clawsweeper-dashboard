## Summary

Keep open: the no-handler relay optimization is useful and has real host proof, but the current PR can skip native pre-tool safety work when no ordinary before_tool_call hook is registered.

## What This Changes

The PR adds a native hook relay no-op command path for Codex native events without matching local hook handlers and adds regression coverage for relay versus no-op selection.

## Best Possible Solution

Keep the no-handler relay optimization, but make the no-op decision account for all pre-tool relay obligations before skipping the subprocess.

## Reproduction Assessment

Yes for the PR regression by source inspection: pre_tool_use can return a no-op based only on hasGlobalHooks("before_tool_call"), while current main performs loop detection, trusted policy checks, and approval work inside runBeforeToolCallHook. I did not run a live low-core stress reproduction for the original performance issue.

## Solution Assessment

No, not as written. The optimization is narrow and useful, but the predicate should preserve trusted policy, approval, and loop-detection work before skipping relay subprocesses.
