## Summary

Keep open: the PR is pointed at the right QQBot credential-isolation bug, but the patch reimplements state-root resolution and still misses documented OpenClaw path semantics such as OPENCLAW_HOME and tilde expansion under that home. It also only supplies test/build claims, not real two-gateway behavior proof for the external PR gate.

## What This Changes

The PR moves QQBot credential backup files from the home-global QQBot data directory to a root derived from OPENCLAW_STATE_DIR/CLAWDBOT_STATE_DIR and adds focused path tests.

## Best Possible Solution

Route QQBot credential backups through the public plugin SDK state resolver, keep restore scoped to that resolved root, add tests for OPENCLAW_STATE_DIR, OPENCLAW_HOME, default, and tilde cases, and explicitly decide the legacy home-global backup compatibility story.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the boundary problem: current main writes QQBot credential backups under the OS home-global QQBot data path, and the PR only covers explicit OPENCLAW_STATE_DIR/CLAWDBOT_STATE_DIR. I did not run a live two-gateway QQBot scenario because this review is read-only.

## Solution Assessment

No. The narrow fix should use the canonical plugin SDK state resolver instead of manually reconstructing a subset of OpenClaw path semantics.
