## Summary

Keep open: current main improves the generic newer-config warning wording, but the package-update path can still have the old parent process read config during restart verification after the updated process has stamped `meta.lastTouchedVersion`, so the reported false warning is still source-reproducible.

## What This Changes

Suppress or make update-aware the false newer-config warning emitted by `openclaw update` after a successful package upgrade and restart.

## Best Possible Solution

Make only the old updater's post-update restart/probe config read silent or explicitly update-aware, while preserving the destructive future-config guard and generic stale-wrapper diagnostics elsewhere.

## Reproduction Assessment

Yes at source level: a package update can run post-core config/plugin work in the updated install, then the old parent process calls restart health verification and reads config through `warnIfConfigFromFuture`. I did not run the mutating 2026.5.4 to 2026.5.6 upgrade locally.

## Solution Assessment

Yes, keeping this open is the best current outcome. The narrow maintainable fix is update-scoped warning suppression or update-aware restart-probe config reads, not weakening `resolveFutureConfigActionBlock` or broadly muting newer-config warnings.
