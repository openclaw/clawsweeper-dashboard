## Summary

Keep open: current `main` still has a source-reproducible path where the old updater parent can read config during restart verification after the updated child has stamped `meta.lastTouchedVersion`; the later stale-wrapper diagnostic work improves the warning text but does not suppress this update-specific false positive.

## What This Changes

The issue requests that `openclaw update` avoid emitting a newer-config warning from the pre-update updater process after a successful package upgrade.

## Best Possible Solution

Make only the update handoff/restart-probe config read silent or update-aware, while preserving the general newer-config warning and destructive-action guard for real split-brain or downgrade cases.

## Reproduction Assessment

Yes at source level: a package update can run post-core config/plugin work in the updated child, then the old parent reads config during restart health probing and triggers the future-version warning. I did not execute the mutating self-update locally because this was a read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is update-scoped warning suppression or an update-aware/silent config read in restart verification. The stale-wrapper diagnostic fix in https://github.com/openclaw/openclaw/pull/79191 is useful context but does not fully solve this update-handoff false positive.
