## Summary

Keep open: current main still has a source-reproducible path where the pre-update parent process reads config during restart verification after the updated process has stamped `meta.lastTouchedVersion`, so the newer-config warning can be false even when the service is healthy.

## What This Changes

The issue requests that successful `openclaw update` runs not emit a misleading newer-config warning from the old updater process after upgrading and restarting successfully.

## Best Possible Solution

Suppress or make update-aware only the old-process restart/probe config read warning, while preserving the future-config destructive-action guard and adding regression coverage for package-update restart verification.

## Reproduction Assessment

Yes at source level: a package update can respawn post-core config/plugin work into the new install, then the old parent reads config during restart health probing and triggers the future-version warning. I did not execute the mutating upgrade locally because this was a read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is update-scoped warning suppression or a silent/update-aware config read in restart verification; do not disable `resolveFutureConfigActionBlock` or the service mutation guard.
