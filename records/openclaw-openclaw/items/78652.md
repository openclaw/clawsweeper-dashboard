## Summary

Keep open: current main still has a source-reproducible path where the old package-update parent process reads config during post-refresh/restart health verification after the new install can stamp `meta.lastTouchedVersion`, so the warning can still be false even though recent work improved restart routing and warning text.

## What This Changes

The issue requests that a successful `openclaw update` stop ending with a misleading newer-config warning from the pre-update updater process.

## Best Possible Solution

Make only the package-update parent-process restart/probe config read silent or update-aware, or move that verification into the updated install, while preserving normal stale-wrapper diagnostics and destructive-action protection.

## Reproduction Assessment

Yes at source level: a package update can run post-core/plugin writes in the updated install, then the old parent calls `waitForGatewayHealthyRestart`, whose auth setup reads config and can emit the newer-config warning. I did not execute the mutating 2026.5.4-to-current upgrade locally.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow fix should target the update-owned health/probe read, not disable the future-version warning globally or treat the stale-wrapper diagnostic PR as a complete fix.
