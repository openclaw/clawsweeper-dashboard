## Summary

Keep open: current main improved the generic newer-config warning, but the update-specific false-warning path still exists because the old updater process can read a config stamped by the newly installed package during restart verification.

## What This Changes

The issue asks for `openclaw update` to avoid a misleading newer-config warning after a successful self-update.

## Best Possible Solution

Make only the update-owned restart/probe config read silent or update-aware, while preserving the future-config warning and destructive-action guard for normal older-binary use.

## Reproduction Assessment

Yes at source level: a fresh-process package update can stamp config with the new version, then the old parent process reads config during restart health probing and triggers the newer-version warning. I did not execute the mutating 2026.5.4 to 2026.5.6 upgrade flow in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is update-scoped suppression or update-aware warning text for restart verification's config read, not disabling the general future-version warning or service mutation guard.

Fix https://github.com/openclaw/openclaw/issues/78652. During package self-update, the old parent process can call restart health probing after the newly installed OpenClaw has stamped config `meta.lastTouchedVersion`, causing a confusing newer-config warning even when the update and gateway restart are healthy. Keep the fix limited to update-owned restart/probe config reads: suppress the warning for that read or make it explicitly update-aware. Do not weaken `resolveFutureConfigActionBlock`, service mutation guards, normal config warning behavior, or broad install policy. Add focused regression coverage around `waitForGatewayHealthyRestart` or the update restart path showing that update restart verification does not emit the generic future-config warning while normal config loads still do.
