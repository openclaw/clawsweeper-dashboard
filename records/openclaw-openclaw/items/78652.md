## Summary

Keep open: current main still has a source-reproducible update handoff path where the old updater process can read config stamped by the new install and emit the newer-config warning after a healthy update.

## What This Changes

Prevent `openclaw update` from ending with a false newer-config warning after a successful package upgrade and service restart.

## Best Possible Solution

Suppress or make update-aware only the old-process restart/probe config read warning while preserving the future-config mutation guard and adding regression coverage.

## Reproduction Assessment

Yes at source level: package update post-core work can run in the fresh install, stamp config with the new version, then the original updater process reads that config during restart health probing. I did not run the mutating self-update locally because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is an update-scoped warning suppression or silent/update-aware config read for restart verification, not disabling `resolveFutureConfigActionBlock` or normal future-config warnings.
