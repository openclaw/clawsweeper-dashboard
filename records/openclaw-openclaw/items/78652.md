## Summary

Keep open: current main still has the source-reproducible false-warning path described in the issue, and no merged follow-up or current-source change removes the old parent process config read during restart verification.

## What This Changes

Prevent `openclaw update` from printing a misleading newer-config warning when the old updater parent verifies a successful package upgrade after the updated process has stamped config metadata.

## Best Possible Solution

Add an update-scoped silent or update-aware config read for restart probe auth, preserving the future-config destructive-action guard and adding regression coverage for package-update restart verification.

## Reproduction Assessment

Yes at source level. The updated post-core process can write `meta.lastTouchedVersion=2026.5.6`, then the still-running old parent calls restart health probing and reads config through the warning path with `VERSION=2026.5.4`.

## Solution Assessment

Yes. The narrow maintainable fix is to suppress or reword only this update-owned read warning, not to disable `resolveFutureConfigActionBlock` or the older-binary service mutation protections.
