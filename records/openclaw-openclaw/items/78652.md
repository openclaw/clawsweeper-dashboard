## Summary

Keep open: current `main` still has a source-reproducible path where the old update parent reads config during restart-health probing after the updated process can stamp `meta.lastTouchedVersion`, so the newer-config warning can be false after a healthy package upgrade. The merged per-process dedupe in https://github.com/openclaw/openclaw/pull/75927 reduces repeated warnings, but it does not suppress this old-parent update handoff warning.

## What This Changes

A successful `openclaw update` from an older package to a newer one should not finish by warning that the config was written by a newer OpenClaw when the live install and gateway are already upgraded.

## Best Possible Solution

Suppress or make update-aware only the old-process restart/probe config-read warning, while preserving normal future-config warnings and the destructive-action guard for real downgrade or split-brain cases.

## Reproduction Assessment

Yes at source level: a package update can run post-core work in the new install, stamp `meta.lastTouchedVersion` with the newer version, and then the old parent can read config during restart-health auth probing. I did not execute the mutating self-update locally because this was a read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is update-scoped warning suppression or a silent/update-aware config read in restart verification. Do not disable `resolveFutureConfigActionBlock` or globally remove the warning.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78652. During package-manager `openclaw update`, post-core work may run from the newly installed package and stamp `meta.lastTouchedVersion` with the new `VERSION`; the original older updater then calls `waitForGatewayHealthyRestart`, whose probe-auth config read can emit the false `Config was last written by a newer OpenClaw` warning. Implement a narrow update-aware suppression or silent config read only for the old-process restart/probe verification path. Preserve normal future-config warnings and preserve `resolveFutureConfigActionBlock` blocking for destructive service mutations from truly older binaries. Add regression coverage for restart-health/update verification with a newer `lastTouchedVersion` config and for the destructive guard still blocking. Add a user-facing `CHANGELOG.md` fix entry. Do not add a config option, broad warning-copy overhaul, or unrelated version-skew UX changes.
