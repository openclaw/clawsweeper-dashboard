## Summary

Keep open: current main still has a source-reproducible path where the old updater process can read config during restart verification after the new install has stamped `meta.lastTouchedVersion`, producing the misleading newer-config warning even when the restarted Gateway is healthy.

## What This Changes

The issue asks `openclaw update` to avoid a false newer-config warning after a successful self-update from an older running updater process.

## Best Possible Solution

Make only the update restart/probe config read warning-aware or silent for the old updater parent, preserve the destructive future-config guard, and add focused regression coverage.

## Reproduction Assessment

Yes, at source level: a package update can run post-core work in the fresh install, stamp `meta.lastTouchedVersion`, then have the old parent read config during restart probing and emit the warning. I did not execute the mutating self-update locally because this review was read-only.

## Solution Assessment

Yes: the narrow maintainable fix is update-scoped warning suppression or an update-aware/silent config read in restart verification. Do not disable `resolveFutureConfigActionBlock` or globally remove the newer-config warning.
