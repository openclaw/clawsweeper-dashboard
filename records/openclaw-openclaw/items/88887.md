## Summary

Keep open. Current main and v2026.6.6 still omit `rerank:false` for mcporter-backed `search`/`vsearch` calls, while QMD 2.1 supports the flag and this PR is the narrow call-site fix; I found no actionable patch defects.

## What This Changes

The PR sends `rerank:false` on memory-core QMD mcporter unified `query` calls for `search` and `vsearch` modes, and adds a vsearch call-argument regression test.

## Best Possible Solution

Land the narrow mcporter argument fix after normal maintainer review and exact-head CI, keeping full query-mode no-rerank control on the existing `memory.qmd.rerank` config path.

## Reproduction Assessment

Yes, source-reproducible. On current main, mcporter-backed `search`/`vsearch` resolve to QMD's unified `query` tool, send only `searches` plus `limit`, and therefore inherit QMD's default `rerank:true`.

## Solution Assessment

Yes. Setting `rerank:false` at `runQmdSearchViaMcporter` for the two lexical/vector-only modes is the narrowest maintainable fix because direct CLI and v1 fallback paths already avoid this unified reranker path and full `query` mode remains unchanged.
