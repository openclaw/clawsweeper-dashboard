- `P2`: This fixes a real memory-core QMD mcporter bug with limited blast radius in optional QMD search modes.
- `merge-risk: 🚨 compatibility`: The new default `rerank:false` argument relies on QMD 2.1+ unified-query support, while QMD 2.0.1 exposed the unified query tool without that parameter.

## Summary

Keep this PR open: current main and the latest release still send QMD unified `query` MCP calls for `search`/`vsearch` without disabling reranking, while the PR is a focused fix with convincing live proof. The only remaining maintainer question is the QMD 2.1+ version boundary for `rerank:false` on unified-query servers.

## What This Changes

The PR adds `rerank:false` to memory-core's QMD mcporter unified `query` call for `search` and `vsearch` modes and adds focused vsearch argument coverage.

## Best Possible Solution

Keep this focused PR open for maintainer review and land it after the QMD 2.1+ no-rerank boundary for default `search`/`vsearch` mcporter calls is accepted or explicitly guarded/documented.

## Reproduction Assessment

Yes. Source inspection reproduces the current-main path: QMD mcporter `searchMode: "search"` or `"vsearch"` resolves to the unified `query` tool, sends typed `searches` without `rerank:false`, and QMD current docs/source default `rerank` to true.

## Solution Assessment

Yes, with a version-boundary caveat. Setting `rerank:false` at the unified QMD MCP call site is the narrowest fix for QMD 2.1+ because the direct CLI and v1 fallback paths already avoid this reranker path, but maintainers should explicitly accept or handle older unified-query QMD versions.
