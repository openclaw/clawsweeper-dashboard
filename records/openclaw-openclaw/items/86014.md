## Summary

Keep open: current main still has the same embedded-runner/session-write-lock cleanup surface, and no post-v2026.5.22 change clearly fixes the reported live-gateway lock that survives timeout cleanup.

## What This Changes

The issue reports that embedded agent run timeout/error paths can leave a gateway-owned session JSONL write lock stuck, causing later requests to time out and retain session state.

## Best Possible Solution

Land a narrow embedded-runner/session-lock fix that guarantees timeout, tool-error, and abort cleanup releases or safely abandons same-process session locks, with regression coverage proving the next request can acquire the same session lock.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The report has concrete Docker logs and source inspection confirms the current same-process lock path can produce the exact 60s SessionWriteLockTimeoutError when a held lock survives cleanup.

## Solution Assessment

Yes: the best fix direction is a narrow cleanup and lock-lifecycle repair in the embedded runner, not a new user-facing cleanup mode. The patch should prove timeout, tool-error, and abort paths release or safely abandon the session lock before later requests retry.
