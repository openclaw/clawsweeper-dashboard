## Summary

Keep open: this PR targets a real edit-warning bug, but the current patch still only broadens the recovery wrapper while the exposed host and sandbox edit tools validate canonical `path` before recovery runs. The protected `maintainer` label, unresolved alias contract, and insufficient production-path proof make this a maintainer follow-up, not a cleanup close.

## What This Changes

The PR changes edit recovery path parsing to accept `file_path` and related aliases, and adds a direct recovery-wrapper regression test for AGENTS.md post-write failures.

## Best Possible Solution

Fix the exposed edit-tool boundary by either canonicalizing/accepting aliases before validation or ensuring callers emit canonical `path`, then add `createOpenClawCodingTools` coverage and redacted Feishu or production-path proof.

## Reproduction Assessment

Yes for the PR blocker by source inspection: production edit tools validate `path` before calling recovery, so `edit({ file_path, edits })` cannot reach the new recovery branch. No live Feishu reproduction was established in this review.

## Solution Assessment

No; the current PR changes recovery-only alias parsing and misses the validated production edit-tool path. The safer solution is to normalize or allow aliases before validation, or fix the caller to emit canonical `path`, then prove that path.
