The PR changes edit recovery to read `file_path`/`filePath`/`filepath`/`file` aliases and adds a direct AGENTS.md post-write recovery test.

## Best Possible Solution

Fix the exposed production edit path by normalizing accepted aliases before validation or by making the caller emit canonical `path`, then cover that boundary with regression tests and redacted runtime proof.

## Reproduction Assessment

Yes for the PR blocker by source inspection: alias-only edit calls cannot reach recovery because production edit tools validate canonical `path` before invoking `wrapEditToolWithRecovery`. No live Feishu/macOS reproduction was established.

## Solution Assessment

No. Recovery-only alias parsing is not the best fix because it misses the validated production edit-tool boundary; normalizing before validation or repairing the caller to send canonical parameters is safer.
