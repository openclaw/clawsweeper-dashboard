## What This Changes

The PR broadens edit recovery path parsing to accept `file_path`/`filePath`/`filepath`/`file` aliases and adds a direct recovery-wrapper regression test for `AGENTS.md` post-write failures.

## Best Possible Solution

Fix the exposed edit-tool boundary by either canonicalizing/accepting aliases before validation or ensuring callers emit canonical `path`, then add `createOpenClawCodingTools` coverage and redacted Feishu or production-path proof for the warning.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the PR blocker: production edit tools validate `path` before recovery, so `edit({ file_path, edits })` cannot reach the new recovery branch; no live Feishu reproduction was established.

## Solution Assessment

Is this the best way to solve the issue? No; recovery-only alias parsing misses the exposed tool path, so the safer fix is to normalize or allow aliases before validation, or fix the caller to emit canonical `path`, then prove the production path.
