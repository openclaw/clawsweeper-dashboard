## Summary

Keep open: current main has partial Codex timeout/client-retirement and stale recovery cleanup fixes, but the shared embedded-run, active-request, recovery, and channel-visible retry gaps are still not fully implemented or safely superseded.

## What This Changes

Reported that Codex app-server and channel-triggered embedded runs can emit early progress, then stop without terminal delivery, leaving user messages queued until stuck-session recovery or manual restart.

## Best Possible Solution

Keep this as the umbrella maintainer item until a shared recovery contract covers Codex app-server timeouts, stale active-work markers, queued channel turns, and one visible retry/failure outcome for the originating channel.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection still gives a medium-confidence path for important sub-gaps: active app-server request counters can suppress Codex terminal/completion watchdogs, and embedded-run recovery eligibility still has a session-age gate.

## Solution Assessment

No, the original Codex-only turn watchdog is only a partial mitigation. The better fix is a shared embedded-run and channel recovery policy that distinguishes stale progress from legitimate long work and sends one visible retryable failure when recovery cannot resume safely.
