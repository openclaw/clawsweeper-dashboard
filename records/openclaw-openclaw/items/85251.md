## Summary

Keep open: current main still has the source-proven Codex `turn/started` silent-gap behavior, and the latest discussion adds a broader shared embedded-run/plugin-discovery liveness direction that is not merged or resolved.

## What This Changes

The issue reports Codex and shared embedded-run turns becoming silent after `turn/started` or `embedded_run:started`, leaving user messages unreplied until stuck-session recovery aborts the run.

## Best Possible Solution

Land the narrow Codex `turn/started` watchdog fix as a safety net, then keep the shared plugin-discovery/event-loop starvation work tracked separately through the open plugin and diagnostic follow-ups.

## Reproduction Assessment

Yes at source level: current code arms the Codex completion-idle guard at `turn:start`, then current-turn `turn/started` can still disarm it before meaningful progress or a terminal event. I did not run a live macOS Codex OAuth or Slack replay in this read-only review.

## Solution Assessment

Yes for the narrow safety-net fix: use the existing `appServer.turnCompletionIdleTimeoutMs` guard and stop treating `turn/started` alone as enough progress to disable it. The plugin-discovery event-loop starvation thread should be handled as a separate root-cause repair rather than folded into this small guard change.
