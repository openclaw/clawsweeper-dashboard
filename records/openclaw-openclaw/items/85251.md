## Label Justifications

- `P1`: A Codex-backed agent/channel turn can accept user work and then remain silent until recovery, affecting live replies.
- `impact:message-loss`: The report says the inbound request receives no user-visible reply and is not retried after recovery.

## Summary

Keep open: the report matches a current Codex app-server liveness gap where a current-turn `turn/started` notification is treated as enough progress to disarm the short completion-idle watchdog, so a silent backend can leave the embedded run waiting for the longer outer recovery path.

## What This Changes

The issue asks OpenClaw to fail fast or surface a retryable error when Codex app-server emits `turn/started` and then no deltas or terminal event.

## Best Possible Solution

Keep the existing `appServer.turnCompletionIdleTimeoutMs` contract and make a lone current-turn `turn/started` keep or re-arm the short completion-idle guard until meaningful progress or a terminal event arrives.

## Reproduction Assessment

Yes at source level: current `main` arms the short completion-idle guard at `turn:start`, then disarms it for a non-terminal current-turn `turn/started` notification. I did not run a live Codex app-server replay, but the code and adjacent tests make the silent-gap path clear.

## Solution Assessment

Yes, with one shape adjustment: the fix should use the existing Codex app-server idle-timeout setting rather than adding `channels.codex.turnProgressThresholdMs`. The narrow repair is to stop treating `turn/started` alone as enough progress to disable the short watchdog.
