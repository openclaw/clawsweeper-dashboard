## Best Possible Solution

Land the narrow Codex `turn/started` watchdog fix as a safety net, then keep shared plugin-discovery/event-loop starvation work tracked separately through the open plugin and diagnostic follow-ups.

## Reproduction Assessment

Yes, at source level: current main arms the Codex completion-idle guard at `turn:start`, then current-turn `turn/started` can still disarm it before meaningful progress or a terminal event. I did not run a live macOS Codex OAuth or Slack/Telegram replay in this read-only review.

## Solution Assessment

Yes for the narrow safety-net fix: use the existing `appServer.turnCompletionIdleTimeoutMs` guard and stop treating `turn/started` alone as enough progress to disable it. The plugin-discovery event-loop starvation thread should be handled as a separate root-cause repair rather than folded into this small guard change.
