- `P1`: The issue describes live agent/channel turns accepting user work and then remaining silent until stuck-session recovery, affecting real replies.

## Summary

Keep open: current main still contains the source-level Codex guard gap, and the same issue now has newer discussion pointing to a broader gateway/plugin-discovery event-loop stall. An open PR with closing syntax targets the Codex watchdog path, and another open PR explores the plugin realpath storm, so the issue should remain open until those are reviewed, landed, or superseded.

## What This Changes

The issue asks OpenClaw to recover Codex or embedded-run turns that go silent after turn start instead of waiting for stuck-session recovery.

## Best Possible Solution

Land or supersede the narrow Codex watchdog fix for immediate bounded recovery, then handle the gateway-wide plugin discovery hot path with a freshness-safe design that removes sync FS from per-turn execution.

## Reproduction Assessment

Yes at source level: current main arms the Codex completion-idle guard at `turn:start`, and a current-turn `turn/started` notification can still disarm that guard without terminal progress. The newer plugin-discovery hypothesis is also source-plausible from the current `loadInstalledPluginIndex` and fingerprint paths, though I did not run the live gateway profile.

## Solution Assessment

Yes for the immediate safety net: using the existing Codex app-server idle-timeout contract to keep the guard armed after a lone `turn/started` is the narrowest fix for the original symptom. The broader event-loop starvation path should be fixed separately through a freshness-safe plugin metadata/discovery design, not by closing this issue before the linked PRs land.
