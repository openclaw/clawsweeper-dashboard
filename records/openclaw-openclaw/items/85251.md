## Summary

Keep open: current main and the latest release still allow a current-turn Codex `turn/started` notification to clear the short completion-idle guard before meaningful progress or a terminal event, and the related plugin-discovery PR is open, partial, and not a sufficient resolution.

## What This Changes

The issue requests bounded handling for embedded Codex/app-server turns that emit progress such as `turn/started` and then stop without a terminal event, avoiding a long silent stuck-session recovery window.

## Best Possible Solution

Land the narrow Codex `turn/started` watchdog fix as a safety net, and keep the broader plugin-discovery/event-loop starvation work tracked separately through the open plugin and diagnostic follow-ups.

## Reproduction Assessment

Yes at source level: current code arms the Codex completion-idle guard at `turn:start`, then a current-turn `turn/started` can still disarm it before meaningful progress or a terminal event. I did not run a live macOS Codex OAuth, Slack, or Telegram replay in this read-only review.

## Solution Assessment

Yes for the narrow safety-net fix: use the existing `appServer.turnCompletionIdleTimeoutMs` guard and stop treating `turn/started` alone as enough progress to disable it. The plugin-discovery event-loop starvation thread should be repaired separately instead of folded into this focused guard change.
