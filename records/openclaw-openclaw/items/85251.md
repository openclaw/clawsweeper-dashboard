## Summary

Keep open: current main still has a source-proven Codex app-server liveness gap where a lone current-turn `turn/started` notification can disarm the short completion-idle watchdog, leaving the run to wait for the outer timeout or stuck-session recovery.

## What This Changes

The issue asks OpenClaw to fail fast when a Codex app-server turn emits `notification:turn/started` and then produces no deltas, terminal event, or error.

## Best Possible Solution

Keep the existing `appServer.turnCompletionIdleTimeoutMs` contract and make a lone current-turn `turn/started` keep or re-arm the short idle guard until meaningful progress or a terminal event arrives, with focused regression coverage.

## Reproduction Assessment

Yes at source level: current code arms the short completion-idle guard at `turn:start`, then disarms it for a non-terminal current-turn `turn/started` notification. I did not run a live Codex app-server replay in this read-only review.

## Solution Assessment

Yes, with one adjustment: the maintainable fix should reuse the existing Codex app-server idle-timeout setting rather than adding a new channel-level knob. The key is to stop treating `turn/started` alone as enough progress to disable the short watchdog.
