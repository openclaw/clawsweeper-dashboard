## Summary

Keep open: the report matches a current Codex app-server liveness gap where a lone current-turn `turn/started` notification is treated as progress and disarms the short completion-idle watchdog, leaving the run to wait for the outer agent timeout or stuck-session recovery.

## What This Changes

The issue asks for Codex app-server turns that emit `turn/started` and then go silent to fail quickly with a retryable error instead of wedging until stuck-session recovery.

## Best Possible Solution

Keep the existing `appServer.turnCompletionIdleTimeoutMs` contract and make a lone `turn/started` keep or re-arm the short idle guard until meaningful progress or a terminal event arrives, with focused regression coverage.

## Reproduction Assessment

Yes at source level: current code arms the short completion-idle guard at `turn:start`, then disarms it for a non-terminal current-turn `turn/started` notification. I did not run a live Codex app-server replay, but the existing tests and source make the reported silent-gap path clear enough to keep open.

## Solution Assessment

Yes, with one adjustment: the maintainable fix should use the existing Codex app-server idle-timeout setting rather than adding a new channel-level knob. The key is to stop treating `turn/started` alone as enough progress to disable the short watchdog.
