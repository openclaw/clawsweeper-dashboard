## Summary

Keep open: current main still has a source-proven Codex `turn/started` silent-gap path, the latest release appears to carry the same guard-disarm behavior, and the related plugin-discovery PR remains open and partial rather than a merged fix.

## What This Changes

The issue asks OpenClaw to stop Codex/app-server and shared embedded-run turns from going silent after an initial progress event until stuck-session recovery aborts them.

## Best Possible Solution

Land a narrow Codex app-server watchdog repair that keeps the completion-idle guard armed after current-turn `turn/started` until real progress or a terminal event, and track the broader plugin-discovery/event-loop starvation work separately.

## Reproduction Assessment

Yes, at source level: current main arms the Codex completion-idle guard at `turn:start`, then a current-turn `turn/started` notification can still disarm it before meaningful progress or a terminal event. I did not run a live macOS Codex OAuth or channel replay in this read-only review.

## Solution Assessment

Yes for the narrow safety-net fix: use the existing `appServer.turnCompletionIdleTimeoutMs` guard and stop treating `turn/started` alone as enough progress to disable it. The plugin-discovery event-loop starvation thread should remain a separate root-cause repair rather than being bundled into this focused guard change.
