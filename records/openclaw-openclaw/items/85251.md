## Summary

Keep open: current main implements part of the Codex app-server idle-timeout and client-retirement mitigation, but source and live discussion evidence still show unresolved shared embedded-run recovery, active request watchdog gating, plugin-discovery hot-path pressure, and channel-visible retry/message-loss behavior.

## What This Changes

Bound Codex and other channel-triggered embedded runs that go silent after startup so recovery does not wedge the session lane or silently lose the user message.

## Best Possible Solution

Keep the current Codex timeout/client-retirement fixes, then land a cross-runtime channel-session recovery contract that keys stale active embedded-run recovery on last progress, bounds hung app-server requests, clears or rebinds lane/reply state after abort, and emits one visible retryable failure when a turn cannot be resumed safely.

## Reproduction Assessment

No live current-main repro was run, but source inspection still shows high-signal sub-gaps: terminal/completion idle watches are gated by active app-server requests, embedded-run recovery still uses session age for eligibility, and plugin index inspection can still rebuild discovery on demand.

## Solution Assessment

No: current main's Codex-only watchdog and cleanup work are useful partial fixes, but the best fix is a shared embedded-run/channel recovery policy with bounded app-server request hangs and visible retry/failure semantics.
