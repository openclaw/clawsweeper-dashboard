## What This Changes

The issue reports Codex and shared embedded-run turns going silent after `turn/started` or `embedded_run:started`, leaving users without a reply until stuck-session recovery aborts the run.

## Best Possible Solution

Land a narrow Codex app-server guard repair that treats `turn/started` as acceptance, not meaningful progress, while tracking plugin-discovery event-loop starvation and diagnostic recovery separately.

## Reproduction Assessment

Yes, at source level: current main arms the Codex completion-idle guard at `turn:start`, then a lone current-turn `turn/started` can still disarm it before meaningful progress or a terminal event. I did not run a live macOS Codex OAuth or Slack replay in this read-only review.

## Solution Assessment

Yes for the narrow safety-net fix: keep the existing `appServer.turnCompletionIdleTimeoutMs` guard armed until real item/progress or terminal completion arrives. The plugin-discovery event-loop starvation thread should be repaired separately rather than folded into this small guard change.
