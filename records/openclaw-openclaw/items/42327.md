## Summary

Keep open: current main still exposes `cron runs` as flag-only, and this PR is the canonical implementation path for the linked cron-runs positional-id report. The branch is useful, but it remains blocked on external-contributor real behavior proof and small review cleanup before merge.

## What This Changes

The PR adds an optional positional job id to `openclaw cron runs`, preserves `--id`, rejects conflicting ids, and adds cron CLI parsing tests.

## Best Possible Solution

Land a CLI-only positional alias that preserves `--id`, keeps gateway/protocol behavior unchanged, documents the new form, and validates id presence/conflicts through one clean command-validation path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main registers `cron runs` with required `--id` and no positional argument, while the gateway already accepts an id selector. I did not run the CLI because this review was read-only.

## Solution Assessment

Is this the best way to solve the issue? Yes with small amendments: a CLI-only alias is the narrow maintainable fix, but the PR should keep Commander validation errors out of the gateway catch and update cron docs before merge.
