## Summary

Keep open: current main still only supports `openclaw cron runs --id`, while this PR is the active narrow implementation for the linked cron-runs positional-id request. The branch looks functionally reasonable, but it still needs contributor real behavior proof before merge plus small P3 validation/docs follow-ups.

## What This Changes

The PR changes `openclaw cron runs` to accept an optional positional job id alongside `--id`, reject conflicting ids, and add cron CLI parsing tests.

## Best Possible Solution

Land the CLI alias after contributor proof, keeping gateway/protocol behavior unchanged and adding docs plus clean local Commander validation.

## Reproduction Assessment

Yes from source. Current main registers `cron runs` with required `--id` and no positional argument, while the gateway accepts `id`/`jobId`; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes with small amendments. The PR's CLI-only alias is the narrow maintainable direction, but the id validation should stay out of the gateway catch path and the public cron docs should mention the new form.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Keep Commander validation outside the gateway catch:** `src/cli/cron-cli/register.cron-simple.ts:80-90`
  - body: `command.error()` exits or throws under `exitOverride`; because these new id validation failures run inside the action `try`, harnesses can catch them in `handleCronCliError` and print a second runtime-style error. Move the id presence/conflict checks before the gateway `try` or use one local validation path.
  - confidence: 0.8
- **[P3] Document the positional cron runs form:** `src/cli/cron-cli/register.cron-simple.ts:67-68`
  - body: This adds `openclaw cron runs <id>`, but the public cron docs still show only `openclaw cron runs --id <job-id>`. Update the cron docs with the new alias so the behavior is discoverable when this lands.
  - confidence: 0.76
