## Summary

Keep this PR open: current main still only supports `openclaw cron runs --id <id>`, and this branch remains the canonical implementation for the linked cron-runs positional-ID request. The patch is directionally correct, but it still needs contributor real behavior proof before merge and has two small P3 follow-ups around validation flow and docs.

## What This Changes

The branch changes `openclaw cron runs` to accept an optional positional job id, preserve `--id`, reject conflicting ids, and add cron CLI parsing tests.

## Best Possible Solution

Land a CLI-only positional alias that preserves `--id`, leaves gateway/protocol behavior unchanged, documents the new form, and keeps id validation on a clean Commander command-validation path.

## Reproduction Assessment

Yes from source: current main registers `cron runs` with required `--id` and no positional argument, while the gateway already accepts an `id`/`jobId` selector. I did not execute the CLI because this was a read-only review.

## Solution Assessment

Yes with small amendments: the PR's CLI-only alias is the narrow maintainable direction, but validation errors should stay out of the gateway catch path and the public cron docs should mention the new form.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Keep Commander validation out of the gateway catch:** `src/cli/cron-cli/register.cron-simple.ts:77-88`
  - body: `command.error()` throws when the root program uses `exitOverride`; because these new validation failures run inside the action `try`, they can be caught and re-logged through `handleCronCliError` instead of staying as plain Commander usage errors. Move the id presence/conflict checks before the gateway `try` or use one local validation path.
  - confidence: 0.8
- **[P3] Document the positional cron runs form:** `src/cli/cron-cli/register.cron-simple.ts:67-68`
  - body: This adds a user-facing CLI invocation, but the public cron docs still show only `openclaw cron runs --id <job-id>`. Update the cron docs so the alias is discoverable after merge.
  - confidence: 0.76
