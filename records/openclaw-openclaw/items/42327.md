## Summary

Keep open. Current main still requires `--id`, and this PR is a narrow fix for the linked cron CLI gap, but the external contributor proof is mock-only and two small P3 amendments remain before merge.

## What This Changes

The PR adds an optional positional job id to `openclaw cron runs`, preserves `--id`, rejects conflicting ids, and adds cron CLI parsing tests.

## Best Possible Solution

Land the CLI alias after contributor proof, with docs for the new form and local Commander validation kept separate from gateway error handling.

## Reproduction Assessment

Yes, from source. Current main registers `cron runs` with required `--id` and no positional argument, while the gateway/protocol already accepts a job id selector; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes with small amendments. A CLI-only alias is the narrow maintainable solution, but validation should avoid the gateway catch path and the public cron docs should mention the new form.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Keep Commander validation out of the gateway catch:** `src/cli/cron-cli/register.cron-simple.ts:80-90`
  - body: `command.error()` exits or throws under `exitOverride`; placing the new presence/conflict checks inside the gateway `try` lets adapters catch the Commander error in `handleCronCliError` and print a second runtime-style error. Move these validations before the gateway `try` or use one local validation path.
  - confidence: 0.8
