## Summary

Keep open. Current main still requires `--id` for `openclaw cron runs`, while this PR is a focused fix for the reported CLI inconsistency; it needs small review cleanup, release-note/docs coverage, and validation before merge.

## What This Changes

The PR changes `openclaw cron runs` to accept an optional positional job id alongside `--id`, reject conflicting ids, and adds cron CLI tests for positional, flag-only, and conflict cases.

## Best Possible Solution

Keep this PR as the focused fix candidate: implement the positional alias in CLI parsing only, preserve `--id`, emit clean single-error failures for missing/conflicting ids, update docs and changelog for the user-facing CLI behavior, and validate the cron CLI test plus the changed gate before merge.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: `cron runs` has no positional argument and still requires `--id`, while adjacent cron commands already use positional job ids.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix because the gateway/protocol already support job-scoped `cron.runs`; the remaining gap is CLI parsing, docs, release notes, and tests rather than backend behavior.
