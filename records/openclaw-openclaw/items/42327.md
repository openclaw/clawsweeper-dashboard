## Summary

Keep open: current main still exposes `openclaw cron runs` as flag-only, and this PR remains the canonical open implementation for the related cron-runs positional-id request, but it needs real CLI proof and a refresh against current cron-run options before merge.

## What This Changes

The PR changes `openclaw cron runs` to accept an optional positional job id alongside `--id`, reject conflicting ids, and add focused CLI parsing tests.

## Best Possible Solution

Land a refreshed version of the CLI alias that preserves current `--run-id` support, keeps id validation local, updates cron docs, and includes redacted real CLI proof.

## Reproduction Assessment

Yes. From source, current main registers `cron runs` with required `--id` and no positional argument, while the gateway/protocol already accept a job id selector; I did not run a live CLI repro in this read-only review.

## Solution Assessment

Yes with small amendments. A CLI-level positional alias is the narrow maintainable direction, but the branch should keep validation out of the gateway catch path, preserve current `--run-id` support, and document the new invocation form.

- **[P3] Keep id validation outside the gateway catch:** `src/cli/cron-cli/register.cron-simple.ts:77-90`
  - body: The new `command.error()` checks run inside the gateway `try`. Under `exitOverride`, Commander throws and the surrounding `catch` routes that CLI validation through `handleCronCliError`, which can reprint it as a runtime error; move id validation before the gateway `try` or use one local validation path.
  - confidence: 0.8

  - body: This adds `openclaw cron runs <id>`, but the cron docs still show only `openclaw cron runs --id <job-id>`. Update the public cron docs when the alias lands so users can discover both forms.
  - confidence: 0.76
