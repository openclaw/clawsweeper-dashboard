## Summary

Keep open. Current main still implements `openclaw cron runs` as a flag-only command requiring `--id`, while this PR is the focused fix candidate for the positional job-id UX reported in #42300. The patch is narrow, but it still needs maintainer/contributor follow-up for small review cleanup, no-id regression coverage, docs alignment, and normal validation before merge.

## What This Changes

The PR changes the cron CLI so `openclaw cron runs` accepts an optional positional job id, preserves `--id`, rejects conflicting positional and option ids, and adds tests for positional, flag-only, and conflict cases.

## Best Possible Solution

Keep this PR open as the focused fix candidate: align it with current main, preserve both `cron runs <job-id>` and `cron runs --id <job-id>`, ensure conflicting and missing ids fail with clean single-error output, update cron docs to show both forms, and run the targeted cron CLI test plus the appropriate changed gate before merge.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the reported mismatch: `cron runs` has no positional argument and still requires `--id`, while adjacent `cron show` and `cron run` use positional job ids.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable solution because the gateway/protocol already support job-scoped `cron.runs` by id; the fix belongs in CLI parsing, tests, and docs rather than in gateway behavior.
