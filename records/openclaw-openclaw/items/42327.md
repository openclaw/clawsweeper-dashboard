## Summary

Keep open. Current main still has the flag-only CLI behavior, the PR remains a focused fix for a real cron CLI inconsistency, and the remaining concerns are narrow patch cleanup plus release/docs coverage rather than a reason to close or supersede it.

## What This Changes

The PR changes `openclaw cron runs` to accept an optional positional job id alongside `--id`, reject conflicting ids, and add cron CLI tests for positional, flag-only, and conflict cases.

## Best Possible Solution

Land this as a narrow CLI parsing fix after moving the new validation onto a single clean error path, covering the missing no-id branch, and updating the changelog plus the user-facing cron docs that currently only show `--id`. The gateway/protocol behavior should stay unchanged.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction path: `cron runs` has no positional argument and requires `--id`, while adjacent `cron show <id>` and `cron run <id>` already use positional job ids.

## Solution Assessment

Yes, with small amendments. The PR’s CLI-only direction is the narrow maintainable fix because the gateway already accepts a job id for `cron.runs`; the remaining work is error handling, test coverage, docs, and release notes.
