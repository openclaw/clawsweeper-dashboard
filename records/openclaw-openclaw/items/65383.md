## Summary

Keep open. Current main still has the duplicate iMessage watcher path, but this PR should not merge as-is because it scopes watcher dedupe through global account-enabled semantics and also adds an unrelated outbound target short-circuit that can bypass normal plugin resolution.

## What This Changes

The branch adds iMessage watcher-key dedupe helper/tests and wires the helper into the iMessage plugin config, plus an outbound target early return for missing implicit target/channel config/plugin cases.

## Best Possible Solution

Land one iMessage-owned startup dedupe that suppresses only duplicate monitor startup for the same effective `cliPath`/`dbPath` backend among startup-eligible accounts, while preserving account enabled/status semantics and leaving shared outbound target resolution unchanged. Cover the duplicate default/named config, disabled accounts, unconfigured placeholders, and distinct backends, then use that merged fix to resolve #65141 and the competing open PR.

## Reproduction Assessment

Yes. The source-level path is high-confidence: #65141 provides the duplicate `imsg rpc`/double-reply config, current main lists both `default` and named iMessage accounts, and gateway startup starts each enabled/configured listed account independently.

## Solution Assessment

No. The PR addresses part of the iMessage problem, including the unconfigured-placeholder edge, but the dedupe should be startup-scoped rather than a global `isEnabled` override, and the unrelated outbound resolver change should be removed or handled separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep watcher dedupe out of global enabled checks:** `extensions/imessage/src/shared.ts:88`
  - body: Putting `shouldStartIMessageAccount` in `config.isEnabled` changes the account's enabled state for runtime snapshots, `channels.status`, and channel selection, not just watcher startup. When `default` is shadowed by a named watcher, those surfaces can report or skip the default as disabled even though the account config is still enabled. Apply the duplicate-backend gate only where gateway startup decides which monitors to boot.
  - confidence: 0.9
- **[P2] Preserve outbound plugin resolution before missing-target returns:** `src/infra/outbound/targets.ts:70-82`
  - body: This return runs before `resolveOutboundChannelPlugin`, so bundled or lazily bootstrapped channel plugins never see implicit deliveries with no channel config. That bypasses plugin `resolveDefaultTo`/`allowFrom` resolution and changes non-iMessage delivery behavior to `Missing target`; remove it from this iMessage fix or keep plugin resolution first.
  - confidence: 0.87
