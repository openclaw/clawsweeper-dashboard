## Summary

Keep open. Current main still has the duplicate iMessage watcher startup path, but this PR is not merge-ready because it changes global enabled semantics, adds an unrelated outbound-resolution shortcut, overlaps another open implementation PR, and lacks external-contributor real behavior proof.

## What This Changes

The PR adds an iMessage watcher-key helper, wires it into the iMessage plugin enabled predicate, extends iMessage account tests, and adds an outbound target early return when no target or channel config exists.

## Best Possible Solution

Land one startup-only iMessage dedupe that suppresses fallback default watchers for the same effective backend without changing global account enabled semantics or shared outbound resolution, then close https://github.com/openclaw/openclaw/issues/65141.

## Reproduction Assessment

Yes. The linked report gives concrete config, process-list evidence with two imsg rpc children, duplicate message history, and a working workaround; current main also traces that config into separate startup account tasks.

## Solution Assessment

No. The dedupe direction is right, but this branch applies it through global plugin enabled-state checks and adds an unrelated shared outbound-resolution shortcut; the narrower maintainable fix belongs on startup account selection or an equivalent startup-only gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope watcher dedupe to startup:** `extensions/imessage/src/shared.ts:91`
  - body: Wiring `shouldStartIMessageAccount` into `config.isEnabled` changes enabled-state semantics for every iMessage account consumer, not only watcher startup. When `default` is shadowed by a named backend, status, audits, and channel selection can now treat `default` as disabled even though only its duplicate watcher should be suppressed.
  - confidence: 0.9
- **[P2] Preserve outbound plugin target resolution:** `src/infra/outbound/targets.ts:75-80`
  - body: This early return runs before normal plugin target/default handling. Implicit deliveries with no channel config and no active registry entry can no longer resolve through the channel plugin, which is unrelated to the iMessage watcher bug and can turn resolvable implicit targets into missing-target failures.
  - confidence: 0.86
