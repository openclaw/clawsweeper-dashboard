## Summary

Keep open. Current main still lacks the iMessage watcher dedupe, but this PR is not safe to merge because it applies startup dedupe through shared enabled-state semantics and adds an unrelated outbound target shortcut.

## What This Changes

This PR adds an iMessage watcher-key dedupe helper wired into plugin `isEnabled`, expands iMessage account tests, and adds an outbound target early return when no target or channel config exists.

## Best Possible Solution

Land one iMessage-owned startup-only dedupe that suppresses fallback `default` watchers for the same effective backend without changing plugin-wide enabled semantics or shared outbound target resolution, then close https://github.com/openclaw/openclaw/issues/65141 after merge.

## Reproduction Assessment

Yes. The linked report includes concrete config, process-list evidence with two `imsg rpc` children, duplicate message history, and a working workaround; current source also traces that config into separate startup account tasks.

## Solution Assessment

No. The iMessage dedupe direction is right, but this PR applies it through shared enabled-state checks and adds unrelated outbound resolution behavior; the narrower fix belongs on the startup account-selection path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope watcher dedupe to startup:** `extensions/imessage/src/shared.ts:91`
  - body: Wiring `shouldStartIMessageAccount` into `config.isEnabled` changes enabled-state semantics for status, channel selection, audits, and other consumers. When `default` is shadowed by a named backend, those surfaces can now report the default account as disabled even though only its watcher should be suppressed.
  - confidence: 0.9
- **[P2] Preserve outbound plugin target resolution:** `src/infra/outbound/targets.ts:75-80`
  - body: This early return runs before normal plugin target/default handling, so implicit deliveries with no channel config and no active registry entry can no longer bootstrap or resolve through the channel plugin. That cross-channel behavior change is unrelated to the iMessage watcher bug and can turn resolvable implicit targets into missing-target failures.
  - confidence: 0.86
