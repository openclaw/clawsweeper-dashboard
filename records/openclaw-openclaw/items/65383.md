## Summary

Keep open. The underlying iMessage duplicate-watcher bug is still present on current main, and this PR is not merge-ready because it wires startup dedupe into the plugin-wide enabled predicate and adds an unrelated outbound target resolver shortcut. There is also a competing open implementation in #65194, so a maintainer should choose the canonical path rather than closing or replacing this automatically.

## What This Changes

This PR adds watcher-key comparison for iMessage accounts, tests it in `accounts.test.ts`, wires the comparison into the iMessage plugin enabled predicate, and adds an outbound target short-circuit for missing implicit targets.

## Best Possible Solution

Land one iMessage-owned startup-only dedupe that suppresses duplicate monitors for the same effective `cliPath`/`dbPath` backend without changing global enabled semantics or shared outbound target resolution. Cover the reported default-plus-named config, disabled named accounts, unconfigured placeholders, and distinct backend cases, then close #65141 after the selected PR merges.

## Reproduction Assessment

Yes. #65141 provides a concrete config, macOS/imsg versions, process-list evidence with two `imsg rpc` children, and duplicate message history; current main still lists and starts each enabled/configured iMessage account independently.

## Solution Assessment

No. The iMessage-owned direction is right, but this PR applies the dedupe through global `config.isEnabled` and includes an unrelated outbound resolver shortcut; a startup-specific gate or account-start list filter is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope watcher dedupe to startup:** `extensions/imessage/src/shared.ts:88`
  - body: Wiring duplicate-backend suppression into `config.isEnabled` changes every consumer of enabled state, including status snapshots, channel summaries, channel selection, audits, and runtime snapshots. When `default` is shadowed by a named account, those surfaces can now treat the default account as disabled even though only its watcher should be suppressed; apply this as a startup-only gate instead.
  - confidence: 0.9
- **[P2] Preserve plugin target resolution:** `src/infra/outbound/targets.ts:70-82`
  - body: This early return runs before normal outbound plugin resolution, so implicit deliveries with no `channels.<channel>` config and no active registry entry can no longer bootstrap a channel plugin or use plugin target/default handling. That can turn previously resolvable delivery targets into `Missing target` failures outside the iMessage watcher fix.
  - confidence: 0.86
