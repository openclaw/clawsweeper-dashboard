## Summary

Keep open: current main fixes the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still launches with an isolated CODEX_HOME that contains only generated config and no Codex auth bridge; open PR #73943 tracks the remaining auth fix.

## What This Changes

Managed Codex ACP should reuse or safely bridge Codex authentication for its isolated CODEX_HOME and avoid sending unsupported timeout config options.

## Best Possible Solution

Keep the timeout guard, then land or replace #73943 with a tightly scoped Codex auth bridge or documented isolated-home login flow plus regression coverage.

Yes. The source-level reproduction is high confidence: current main sets CODEX_HOME to a generated acpx codex-home, writes only config.toml there, and still has a test asserting source auth.json is not bridged.

## Solution Assessment

Unclear for the remaining auth half until #73943 is reviewed. The timeout guard is the narrow maintainable fix for unsupported config forwarding, but credential bridging should expose only the required Codex auth material.
