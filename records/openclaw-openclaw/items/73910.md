## Summary

Keep open. Current main addresses the unsupported Codex ACP timeout forwarding half, but the managed ACPX Codex wrapper still forces an isolated CODEX_HOME containing only generated config and no Codex auth bridge; the issue is also paired with open, unmerged PR #73943.

## What This Changes

OpenClaw-managed Codex ACP should safely reuse or document Codex authentication for its isolated CODEX_HOME and stop sending unsupported timeout config to Codex ACP.

## Best Possible Solution

Keep the current Codex timeout guard, then land or replace PR #73943 with a tightly scoped reviewed fix that exposes only the minimum Codex auth material needed by the Codex ACP adapter, preserves generated config isolation, adds regression coverage, and updates the ACP setup docs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main creates an isolated Codex ACP home and wrapper without auth.json, and an existing test explicitly asserts source Codex auth is not copied; the timeout half is separately covered by a guard test.

## Solution Assessment

Unclear until the auth bridge receives maintainer/security review. The timeout guard is the right narrow fix for unsupported config forwarding, but the remaining auth path should be solved by a tightly scoped auth.json bridge or supported isolated-home login flow rather than importing the user's full Codex home.
