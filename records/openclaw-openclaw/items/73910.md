## Summary

Keep open. Current main has the Codex ACP timeout guard, but the managed Codex ACP wrapper still forces an isolated CODEX_HOME that only contains generated config and no auth bridge; the remaining auth fix is already tracked by open PR #73943 and should land or be replaced before this issue closes.

## What This Changes

The issue asks OpenClaw-managed Codex ACP sessions to bridge or document authentication for the isolated CODEX_HOME and avoid forwarding unsupported timeout options to codex-acp.

## Best Possible Solution

Keep the issue open until #73943 or an equivalent maintainer-reviewed fix lands: bridge only the minimum Codex auth material, or provide a supported isolated-home login flow, while preserving generated config isolation and adding focused tests plus setup docs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main generates an isolated Codex ACP home and wrapper without auth.json, and the existing test explicitly asserts no source Codex auth is copied while the timeout guard is already covered.

## Solution Assessment

Unclear until the auth bridge receives maintainer security review. The timeout guard is the right narrow fix for unsupported timeout config; the remaining auth fix should be a tightly scoped auth.json bridge or supported login flow, not a full Codex home import.
