## What This Changes

The issue asks OpenClaw-managed Codex ACP to reuse or set up Codex authentication safely while avoiding unsupported ACP timeout config forwarding.

## Best Possible Solution

Keep the shipped timeout suppression and land a maintainer/security-reviewed ACPX Codex auth contract, likely an explicit Codex-home override or documented isolated-home login flow, without silently materializing credentials into OpenClaw state.

## Reproduction Assessment

Yes. Source and dependency contracts show current main and v2026.5.18 still launch Codex ACP with an isolated `CODEX_HOME` that lacks auth while Codex ACP checks auth from that configured home; the issue comments add matching Ubuntu and macOS runtime evidence.

## Solution Assessment

No complete solution is on main yet. The timeout guard is the right narrow fix for the rejected config option, but the auth side needs a reviewed contract such as an explicit home override or documented isolated-home setup rather than unreviewed credential copying.
