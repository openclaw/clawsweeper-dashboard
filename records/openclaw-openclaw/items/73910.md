## Summary

Keep open: current main and v2026.6.1 already suppress unsupported ACP timeout config forwarding, but the Codex ACP OAuth-auth half remains unresolved because the managed wrapper still forces an isolated CODEX_HOME and does not bridge the host Codex OAuth auth.json.

## What This Changes

The issue asks OpenClaw-managed Codex ACP sessions to stop using an unauthenticated isolated Codex home and to avoid sending unsupported generic timeout config to Codex ACP.

## Best Possible Solution

Keep the timeout suppression already on main, and have maintainers choose a supported Codex ACP OAuth boundary such as a reviewed auth bridge, explicit Codex-home override, or documented setup/doctor path.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.1 still force Codex ACP into an isolated CODEX_HOME and only materialize API-key auth from env, while Codex source stores file-mode auth in CODEX_HOME/auth.json. I did not run a live OAuth ACP session in this read-only review.

## Solution Assessment

No: closing now would be premature because current main only fixes the unsupported timeout forwarding half; the supported OAuth-auth boundary for Codex ACP still needs maintainer/security direction.
