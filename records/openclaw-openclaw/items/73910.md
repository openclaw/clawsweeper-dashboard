## Summary

Keep open. Current main has fixed the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still runs with an isolated generated CODEX_HOME that does not bridge Codex auth; the open closing PR #73943 is the right item to review before this issue closes.

## What This Changes

The report asks OpenClaw-managed Codex ACP to make Codex auth available to its isolated home and stop sending timeout config options that Codex ACP rejects.

## Best Possible Solution

Keep the shipped timeout guard, then land or replace #73943 with a tightly scoped Codex auth bridge or documented isolated-home login flow that preserves generated config isolation and adds regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main sets CODEX_HOME to the generated acpx codex-home, writes only config.toml, and has a test asserting auth.json is not bridged; the timeout half is separately covered by a guard test.

## Solution Assessment

Unclear until the auth bridge is reviewed. The timeout guard is the narrow right fix for unsupported config forwarding, but the auth path should expose only the required Codex auth material rather than importing the full Codex home.
