## Summary

Keep open: current main fixes the unsupported Codex ACP timeout forwarding, but the Codex ACP auth path remains intentionally unresolved. Source, tests, docs, and pinned dependency contracts still show an isolated Codex ACP CODEX_HOME without host auth reuse, while related auth-bridge PRs were closed because credential materialization needs a maintainer/security decision.

## What This Changes

The issue reports that OpenClaw-managed Codex ACP sessions fail because the ACPX Codex wrapper uses an isolated CODEX_HOME without Codex auth and forwards unsupported timeout config.

## Best Possible Solution

Keep the timeout guard, then choose one supported Codex ACP auth path: a reviewed explicit Codex-home override or documented isolated-home login flow, without copying OpenClaw OAuth refresh material into ACPX state by default.

## Reproduction Assessment

Yes. Source and dependency contracts show current main still forces an isolated Codex ACP CODEX_HOME, writes no auth.json there, and the pinned Codex ACP/Codex auth stack loads auth from that resolved home; reporter and commenter logs provide matching real-world symptoms.

## Solution Assessment

No current solution is complete. The timeout filtering is the right narrow fix, but auth should be solved through a maintainer/security-approved contract such as the explicit home override lane, not by reintroducing unreviewed credential materialization.
