## Summary

Keep open. Current main fixes the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still forces an isolated CODEX_HOME with generated config only, and open PR #73943 is the active closing implementation candidate for the remaining auth gap.

## What This Changes

Reports managed Codex ACP sessions failing because OpenClaw isolates CODEX_HOME without Codex auth and forwards a timeout option rejected by Codex ACP.

## Best Possible Solution

Land or replace #73943 with a reviewed minimal Codex auth bridge or documented isolated-home login flow, while preserving generated config isolation and the existing timeout guard.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still creates an isolated Codex ACP home without auth.json, and the test suite explicitly locks in that no-auth behavior while separately covering the timeout guard.

## Solution Assessment

Unclear for the remaining auth bridge until maintainer/security review completes. The timeout guard is the narrow right fix for unsupported config forwarding, but credential handling should be constrained to the minimum Codex auth material or a documented isolated-home setup flow.
