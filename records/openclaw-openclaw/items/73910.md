## Summary

Keep open. Current main fixes the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still forces an isolated CODEX_HOME with only generated config and no auth bridge; the remaining auth work is already tracked by open PR #73943.

## What This Changes

OpenClaw-managed Codex ACP should safely reuse or bridge existing Codex auth in its isolated CODEX_HOME and avoid sending unsupported timeout config options.

## Best Possible Solution

Keep the timeout guard, then land or replace #73943 with a reviewed Codex auth bridge that exposes only the required auth.json material, preserves generated config isolation, and updates regression coverage and ACP setup docs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main creates an isolated Codex ACP home and wrapper without auth.json, and an existing test explicitly asserts source Codex auth is not copied; the timeout half is separately covered by a passing guard test.

## Solution Assessment

Unclear until the auth bridge receives maintainer/security review. The timeout guard is the narrow right fix for unsupported config forwarding, but the auth path should be solved with a tightly scoped auth.json bridge or documented isolated-home login flow rather than importing the full Codex home.
