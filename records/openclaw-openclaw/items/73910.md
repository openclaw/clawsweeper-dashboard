## Summary

Keep open. Current main has fixed the unsupported Codex ACP timeout forwarding, but it still launches Codex ACP with an isolated CODEX_HOME that contains generated config only and no Codex auth bridge; the remaining work is already tracked by the open, unmerged closing PR #73943.

## What This Changes

The issue asks OpenClaw-managed Codex ACP sessions to reuse or document Codex authentication for the isolated CODEX_HOME and avoid sending unsupported timeout config to Codex ACP.

## Best Possible Solution

Keep the current Codex ACP timeout guard, then land or replace #73943 with a narrowly scoped auth fix that exposes only the minimum Codex auth material or provides a supported isolated-home login flow, preserves generated config isolation, updates docs, and adds regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main generates a Codex ACP wrapper with isolated CODEX_HOME and config.toml only, while the existing test explicitly asserts source Codex auth is not bridged; the issue discussion also has independent Linux and macOS confirmations.

## Solution Assessment

Unclear for the remaining auth half until the bridge receives maintainer/security review. The timeout guard is the right narrow fix for the unsupported config option; the auth side should be solved by a minimal auth.json bridge or supported isolated-home login flow, not by copying the full Codex home.
