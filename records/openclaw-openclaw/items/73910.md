## Summary

Keep open: current main fixes the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still uses an isolated CODEX_HOME without bridging Codex auth, and open closing PR #73943 tracks the remaining auth fix.

## What This Changes

Request that managed Codex ACP sessions safely reuse or bridge Codex auth for the isolated CODEX_HOME and stop sending unsupported timeout options.

## Best Possible Solution

Land or replace #73943 with a tightly scoped auth.json bridge or documented isolated-home login flow, while preserving generated config/hook isolation and the shipped timeout guard.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main writes only config.toml into the isolated codex-home and still has a regression test asserting source auth.json is absent; the issue discussion also has direct ACPX and macOS confirmations.

## Solution Assessment

Unclear for the remaining auth bridge until #73943 gets maintainer/security review. The timeout guard already on main is the narrow maintainable fix for unsupported config forwarding, but credential bridging should expose only the required Codex auth material.
