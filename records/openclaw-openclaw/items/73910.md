## Summary

Keep open: current main has fixed the unsupported Codex ACP timeout forwarding, but the Codex ACP auth half is still not proven fixed and source plus pinned dependency contracts still show a forced isolated `CODEX_HOME` without a credential bridge or explicit ACP auth handoff.

## What This Changes

The issue asks OpenClaw-managed Codex ACP to handle isolated-home authentication safely and stop sending unsupported timeout config options.

## Best Possible Solution

Define one security-reviewed Codex ACP auth contract that either reuses host Codex auth without broadening backup exposure, injects a scoped ACP/Codex auth flow, or documents an explicit isolated-home login path while keeping the timeout guard.

## Reproduction Assessment

Yes, from source and dependency contracts: current main still forces an isolated Codex ACP `CODEX_HOME` without `auth.json`, while the pinned adapter and Codex auth manager load managed auth from that resolved home. I did not run a live credential smoke, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No current solution is fully proven. The timeout guard is the right narrow fix, but the auth side needs maintainer/security direction because copying or symlinking token files into OpenClaw state has credential-boundary and backup implications.
