## Summary

Keep open: current main and v2026.5.18 still force Codex ACP through an isolated CODEX_HOME with generated config only, while the unsupported timeout half is already fixed. The remaining auth path is active, auth-sensitive, and paired with an open explicit-home override PR that still needs maintainer/security review.

## What This Changes

The issue asks OpenClaw-managed Codex ACP to use a supported Codex auth path and stop sending unsupported timeout config options.

## Best Possible Solution

Keep the timeout suppression and land a reviewed ACPX Codex auth contract, likely an explicit Codex-home override or documented isolated-home login flow, without default credential materialization into OpenClaw state.

## Reproduction Assessment

Yes. Source inspection shows current main still launches Codex ACP with an isolated CODEX_HOME that lacks auth bridging, and codex-acp v0.14.0 checks auth from that configured home before session startup.

## Solution Assessment

No complete solution is on main. The timeout fix is already the right narrow behavior, but the remaining auth behavior needs a maintainer-approved contract such as the explicit home override lane rather than ad hoc credential copying.
