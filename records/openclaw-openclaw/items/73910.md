## Summary

Keep open. Current main and v2026.4.29 already suppress the unsupported Codex ACP timeout config option, but the managed Codex ACP path still creates an isolated CODEX_HOME with only generated config and no auth bridge. An open closing PR, #73943, tracks the remaining auth fix, so this issue should stay open until that PR or an equivalent maintainer-reviewed fix lands.

## What This Changes

OpenClaw-managed Codex ACP sessions should safely reuse or bridge valid Codex auth and avoid sending unsupported timeout config to Codex ACP.

## Best Possible Solution

Keep the timeout guard, then land or replace #73943 with a narrow maintainer-reviewed auth fix: bridge only the minimum Codex auth material or document a supported isolated-home login flow, preserve generated config isolation, add regression coverage, and update ACP docs/changelog as needed.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still forces an isolated Codex ACP home with no auth.json bridge, and current tests explicitly assert that source Codex auth is not copied while the timeout guard is covered.

## Solution Assessment

Unclear until the auth bridge receives maintainer security review. The timeout guard is the right narrow fix for the unsupported config option; the remaining auth part should be solved by a tightly scoped auth.json bridge or supported isolated-home login flow, not by importing the user's full Codex config.
