## Summary

Keep open. Current main has fixed the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still forces an isolated generated CODEX_HOME without bridging Codex auth; the open closing PR #73943 is the right item to review before this issue closes.

## What This Changes

OpenClaw-managed Codex ACP should reuse or explicitly set up Codex auth for its isolated home and avoid sending unsupported timeout config to Codex ACP.

## Best Possible Solution

Keep the shipped timeout guard, then land or replace #73943 with a minimal Codex ACP auth reuse or documented isolated-home login path plus regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main sets CODEX_HOME to the generated acpx codex-home, writes only config.toml, and has a test asserting auth.json is not bridged; the timeout half is separately covered by a guard test.

## Solution Assessment

Unclear until the auth bridge is reviewed. The timeout guard is the narrow right fix for unsupported config forwarding, but the remaining auth path should expose only the required Codex auth material rather than importing broader Codex home state.
