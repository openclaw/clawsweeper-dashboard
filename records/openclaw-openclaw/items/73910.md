## Summary

Keep open. Current main fixes the unsupported Codex ACP timeout forwarding, but the managed ACPX Codex wrapper still forces an isolated CODEX_HOME that contains only generated config and no Codex auth bridge; the remaining auth work is already tracked by open PR #73943 and should not be closed before that or an equivalent reviewed fix lands.

## What This Changes

The issue asks OpenClaw-managed Codex ACP sessions to reuse or safely bridge existing Codex authentication for the isolated CODEX_HOME and stop sending unsupported timeout config options to Codex ACP.

## Best Possible Solution

Keep the current Codex timeout guard, then land or replace #73943 with a narrow maintainer-reviewed auth fix: bridge only the minimum Codex auth material or provide a supported isolated-home login flow, preserve generated config isolation, add regression coverage, and update the ACP setup docs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main creates an isolated Codex ACP home and wrapper without auth.json, and the existing test explicitly asserts source Codex auth is not copied while the timeout guard is separately covered.

## Solution Assessment

Unclear until the auth bridge receives maintainer/security review. The timeout guard is the right narrow fix for the unsupported config option; the remaining auth part should be solved by a tightly scoped auth.json bridge or supported isolated-home login flow, not by importing the user's full Codex home.
