## Summary

Keep open: current main and v2026.6.8 cover the unsupported timeout forwarding, but the central Codex ACP auth-home failure remains because the generated ACPX wrapper still forces an isolated CODEX_HOME and only materializes API-key auth there.

## What This Changes

The issue reports that OpenClaw-managed Codex ACP fails when the wrapper uses an isolated CODEX_HOME without host Codex OAuth auth, and originally also forwarded a Codex-unsupported timeout config option.

## Best Possible Solution

Keep this issue as the canonical ACPX/Codex auth-home bug and resolve it with a maintainer-approved, documented auth bridge or explicit home behavior that preserves OAuth refresh without importing unrelated host Codex hooks/config, plus live Codex ACP proof.

## Reproduction Assessment

Yes. Source inspection shows OpenClaw forces Codex ACP through an isolated CODEX_HOME that lacks host auth.json, and inspected Codex source reads auth.json from the effective codex_home resolved from CODEX_HOME.

## Solution Assessment

Unclear until maintainers choose the credential boundary. The timeout suppression already on main is the right narrow fix for that half, but the auth-home half needs a security/product decision between a narrow auth bridge, an explicit home override, or documented setup/doctor behavior.
