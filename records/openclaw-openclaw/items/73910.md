## Summary

Keep open: current main fixes the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still creates an isolated CODEX_HOME with only generated config and no auth bridge; open PR #73943 remains the closing candidate.

## What This Changes

The issue requests that OpenClaw-managed Codex ACP sessions reuse or bridge valid Codex auth safely and stop sending unsupported Codex timeout config controls.

## Best Possible Solution

Land or replace #73943 with a narrowly scoped Codex auth bridge or documented isolated-home login flow while keeping config isolation and the existing timeout guard.

## Reproduction Assessment

Yes. Source inspection shows the wrapper still forces CODEX_HOME to the generated codex-home, that home only gets config.toml, and the current test asserts source auth.json is not bridged.

## Solution Assessment

Unclear for the remaining auth half until #73943 is reviewed. The timeout guard is already the narrow maintainable fix for unsupported config forwarding; the auth bridge should expose only the minimum Codex auth.json material.
