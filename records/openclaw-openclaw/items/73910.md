## Summary

Keep open: current main fixes the unsupported Codex ACP timeout forwarding, but the managed ACPX Codex auth path is not proven fixed. Source and pinned dependency contracts still show an isolated `CODEX_HOME` with no `auth.json` bridge or explicit OpenClaw auth handoff for normal ChatGPT/Codex OAuth.

## What This Changes

OpenClaw-managed Codex ACP should run successfully by safely reusing or documenting setup for Codex authentication in the isolated ACPX home and by not forwarding unsupported timeout config to Codex ACP.

## Best Possible Solution

Define one security-reviewed Codex ACP auth contract: either let Codex ACP see its normal authenticated home, inject a scoped ACP auth flow, or document and implement an explicit isolated-home login path, while keeping config hook isolation and the existing timeout guard.

## Reproduction Assessment

Yes, from source and dependency contracts: current main still forces an isolated `CODEX_HOME` without `auth.json`, while the pinned Codex ACP and Codex auth manager load managed auth from that resolved home. A live credential smoke was not run, so this is source-reproducible rather than tool-reproduced.

## Solution Assessment

No current solution is fully proven. The timeout guard is the right narrow fix, but the auth side needs a maintainer/security decision because copying or symlinking token files into OpenClaw state has backup and credential-boundary implications.
