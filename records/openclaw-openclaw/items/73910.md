## Summary

Keep open: current main fixes the unsupported Codex ACP timeout forwarding half, but the managed ACPX Codex wrapper still uses an isolated `CODEX_HOME` without bridging Codex auth, and open closing PR #73943 tracks the remaining auth fix.

## What This Changes

The issue requests that OpenClaw-managed Codex ACP sessions reuse or document Codex auth for the isolated ACPX home and stop sending unsupported timeout config to Codex ACP.

## Best Possible Solution

Review and land or replace #73943 with a minimal Codex `auth.json` bridge or documented isolated-home login flow while preserving the current timeout guard and config-hook isolation.

## Reproduction Assessment

Yes. The source-level path is high confidence: current main sets Codex ACP `CODEX_HOME` to the generated `codex-home`, writes only `config.toml`, and still has a regression test asserting source `auth.json` is not bridged.

## Solution Assessment

Unclear for the remaining auth half until #73943 is reviewed. The timeout guard is the narrow maintainable fix for unsupported config forwarding; the auth fix should be limited to the Codex auth material the adapter already expects.
