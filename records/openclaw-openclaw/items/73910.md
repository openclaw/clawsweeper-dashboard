## Summary

Keep open. Current main fixes the unsupported Codex ACP timeout forwarding path, but the managed ACPX Codex wrapper still forces an isolated CODEX_HOME that contains only generated config and no Codex auth bridge. The issue is also paired with open, unmerged PR #73943, so it should remain open until that PR or an equivalent reviewed fix lands.

## What This Changes

The issue reports that OpenClaw-managed Codex ACP sessions fail because the ACPX wrapper isolates CODEX_HOME without Codex auth and then forwards a timeout config option that Codex ACP rejects.

## Best Possible Solution

Keep the current Codex timeout guard, then land or replace #73943 with a narrow reviewed auth fix that exposes only the minimum Codex auth material needed by the Codex ACP adapter, preserves generated config isolation, adds regression coverage, and updates the ACP setup docs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main creates an isolated Codex ACP home and wrapper without auth.json, and the existing test explicitly asserts source Codex auth is not copied; the timeout half is separately covered by a passing guard test.

## Solution Assessment

Unclear until the auth bridge receives maintainer/security review. The timeout guard is the right narrow fix for unsupported config forwarding, but the remaining auth path should be solved by a tightly scoped auth.json bridge or supported isolated-home login flow rather than importing the user's full Codex home.
