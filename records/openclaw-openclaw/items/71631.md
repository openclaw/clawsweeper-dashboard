## Summary

Keep open. Current main has substantial chat responsiveness work and some gateway/session hot-path fixes, but this issue still spans gateway configuration UX, slow-RPC isolation, and end-to-end latency attribution that are not fully implemented or reducible to one shipped fix.

## What This Changes

Requests broad responsiveness improvements across gateway configuration screens and Control UI/WebChat chat, including immediate UI feedback, streaming or running states, slow-RPC isolation, and end-to-end latency instrumentation.

## Best Possible Solution

Keep this as a maintainer-owned umbrella only if that is useful; otherwise split it into focused tasks for UI-to-gateway timing spans, per-control config pending states, cancellation/stale-response guards for slow panels, and continued protection of the existing optimistic chat streaming path.

No, not for the whole issue. Current source and tests verify major chat responsiveness pieces, while #64004 gives a narrower Control UI slowness scenario; the combined instrumentation/config/slow-RPC acceptance criteria still lack one high-confidence reproduction path.

## Solution Assessment

Unclear as a single implementation item. The current non-blocking chat path is the right foundation for the chat subset, but the remaining work is broad enough that separate UI, gateway, and observability PRs would be more maintainable than one sweeping patch.
