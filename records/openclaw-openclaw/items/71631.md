Keep open. Current main covers major Web UI chat responsiveness pieces, but this issue also asks for gateway configuration responsiveness, slow-RPC isolation, and end-to-end latency attribution that are not fully implemented or tracked by a single merged fix.

## What This Changes

The issue requests lower-latency gateway configuration interactions, immediate Web UI chat feedback, non-blocking slow RPC behavior, and UI-to-gateway/provider latency instrumentation.

## Best Possible Solution

Keep this as an umbrella only if maintainers want a high-level performance tracker; otherwise split it into focused tasks for UI-to-gateway timing spans, per-control config pending states, stale-response/cancellation guards for slow panels, and preservation of the existing optimistic streaming chat path.
