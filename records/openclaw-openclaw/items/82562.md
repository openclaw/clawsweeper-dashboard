## Summary

Keep open: the PR targets a real source-reproducible plugin tool lifecycle bug and supplies convincing after-fix runtime proof, but the proposed retention path can replace broader active runtime state with a scoped tool-discovery registry.

## What This Changes

The PR exports the standalone runtime registry installer, uses it to retain forced tool-discovery registries for cached plugin tool descriptors, adds a lifecycle regression test, and updates the changelog.

## Best Possible Solution

Retain or reuse the successful scoped recovery registry for cached descriptor execution without replacing broader active/channel/http registry state, and add a regression proving unrelated live registry entries survive.

## Reproduction Assessment

Yes, source-reproducible. The linked issue provides concrete current-main steps/logs, and current main still has the cached descriptor path that rebuilds an activate:false tool-discovery registry and can throw when the resolved runtime registry lacks the matching tool.

## Solution Assessment

No. Retaining the successful recovery registry is plausible, but installing a scoped recovery registry as global active state can narrow unrelated runtime state; the safer fix should keep the scoped runtime available without replacing broader active/channel/http registrations.
