## Summary

Keep open: the reported bug is credible and the PR includes useful after-fix proof, but the patch has a blocking registry-lifetime issue because recovery for one cached descriptor can replace broader live plugin runtime state.

## What This Changes

The PR retains successful forced plugin tool-discovery registries for cached descriptor execution, adds a regression test, exports the standalone registry installer, and updates the changelog.

## Best Possible Solution

Retain or reuse the successful scoped tool runtime for cached descriptor execution without replacing a broader active/channel/http registry, and add a regression that proves unrelated live registry entries survive the recovery path.

## Reproduction Assessment

Yes, source-reproducible. The linked issue gives concrete current-main steps/logs, and the cached descriptor path in `src/plugins/tools.ts` still rebuilds an activate:false tool-discovery load that can throw `plugin tool runtime missing` when the resolved registry lacks the matching tool.

## Solution Assessment

No. Retaining the successful recovery registry is plausible, but installing the scoped tool-discovery registry as the active registry can narrow global runtime state; the safer fix should preserve unrelated active/channel/http registrations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the existing active registry when retaining tools:** `src/plugins/tools.ts:795`
  - body: This retention path installs the cold-loaded tool-discovery registry into global active state. Because cached descriptor execution loads only the owning plugin id, `installStandaloneRuntimePluginRegistry` can replace a broader active registry and sync unpinned channel/http surfaces to that scoped registry, dropping unrelated live plugin registrations. Retain this scoped runtime without replacing broader active state, or only install when it cannot narrow the current registry.
  - confidence: 0.87
