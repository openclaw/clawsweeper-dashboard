The PR exports the standalone registry installer, installs scoped cold-loaded tool-discovery registries during cached plugin tool execution, adds a regression test, and updates the changelog.

## Best Possible Solution

Retain or merge the successful scoped recovery registry for cached descriptor execution without replacing unrelated active, channel, or HTTP runtime state, and cover both repeated cached calls and survivor state in tests.

Yes. The linked issue and current main show a source-reproducible cached descriptor path that can throw plugin tool runtime missing after registry replacement, and the PR body includes live RED/GREEN runtime harness output for the proposed scenario.

## Solution Assessment

No. Retaining the recovered registry is the right direction, but installing a scoped cold-load registry with setActivePluginRegistry is not the narrowest maintainable fix because it can replace unrelated runtime surfaces.
