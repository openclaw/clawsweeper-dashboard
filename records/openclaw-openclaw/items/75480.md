## Summary

Keep open. Current main still contains the forced-stop failure shape described by the report: a timed-out `stopChannel` leaves the old account task registered, then the health monitor's immediate `startChannel` call is suppressed, so a replacement Slack adapter may never be instantiated.

## What This Changes

Fix Slack Socket Mode restart recovery so a channel-stop timeout cannot leave the old adapter half-stopped while suppressing replacement startup.

## Best Possible Solution

Add a regression-backed recovery path for health-monitor restarts where a stop timeout leaves an aborted provider task stuck, while keeping Slack-specific SDK cleanup inside the Slack plugin and preserving manual-stop deduplication.

## Reproduction Assessment

Yes. The report includes timestamped live logs, and current main has a focused unit-test reproduction of the core state machine: stop times out, `startChannel` is called, and no second account task starts.

## Solution Assessment

Unclear for the exact implementation because there is no patch yet. The maintainable direction is narrow: distinguish restart recovery from manual stop and/or make Slack's abort path settle promptly, instead of weakening the generic duplicate-task guard for every caller.

## Repair Work Prompt

Fix the Slack Socket Mode forced-stop timeout recovery described in https://github.com/openclaw/openclaw/issues/75480. Current main leaves the account task registered when `stopChannel` times out after abort, so the health monitor's subsequent `startChannel` call returns early and no replacement adapter starts. Implement a narrow recovery path for health-monitor restart/forced-stop timeout without allowing duplicate provider tasks during normal starts or manual stops. Prefer a generic gateway lifecycle fix only where the state machine is generic; keep Slack SDK internals such as `shuttingDown`, retry timers, and `app.stop()` behavior inside the Slack plugin. Update or replace the existing `does not allow a second account task to start when stop times out` test so it proves the desired recovery behavior, and add focused health-monitor/Slack coverage if the fix touches those paths. Add a `CHANGELOG.md` fix entry for this user-facing availability bug with `Thanks @jnikolaidis`. Do not change dependency versions, broad channel health policy defaults, or unrelated Slack routing behavior.
