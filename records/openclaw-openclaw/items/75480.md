## Summary

Keep open. Current main still has the reported forced-stop failure shape: the health monitor stops and then starts the account, but a stop timeout leaves the old task registered, and the start path refuses to create a replacement while that task remains.

## What This Changes

The issue asks for Slack Socket Mode recovery when a channel-stop timeout leaves the old adapter half-stopped and prevents a replacement adapter from starting.

## Best Possible Solution

Add a regression-backed recovery path for health-monitor restarts where a stop timeout leaves an aborted provider task stuck, while keeping Slack-specific SDK cleanup inside the Slack plugin and preserving the generic duplicate-task guard for normal manual stops and overlapping starts.

## Reproduction Assessment

Yes. The issue has timestamped live logs, and current main has a focused source-level reproduction of the core failure shape: after a stop timeout, `startChannel` is called but no second account task is allowed to start.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable direction is to distinguish restart recovery from manual stop and/or make Slack's abort path settle promptly, not to broadly weaken duplicate-task protection for every start caller.

Fix https://github.com/openclaw/openclaw/issues/75480. Current main leaves a channel account task registered after `stopChannel` times out at 5000 ms; the health monitor then calls `startChannel`, but `startChannelInternal` returns because `store.tasks.has(id)`, so Slack Socket Mode can stay silently dead until a manual gateway restart. Implement a narrow recovery path for health-monitor restart/forced-stop timeout without breaking manual stop deduplication or ordinary duplicate-start serialization. Update or replace the current `server-channels.test.ts` regression that asserts no second task starts after timeout, and add Slack/provider coverage if the plugin cleanup changes.
