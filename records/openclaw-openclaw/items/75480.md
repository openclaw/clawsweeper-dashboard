## Summary

Keep open. Current main still has the forced-stop restart failure shape described in the report: a health-monitor restart awaits `stopChannel`, then calls `startChannel`, but the stop-timeout branch leaves the old account task registered and the start path returns without creating a replacement task. The earlier Slack clean-stop fix from #56646 is present, but it does not cover the gateway manager path that stays stuck after the abort-timeout branch.

## What This Changes

The issue requests reliable Slack channel recovery after a health-monitor stop times out, including retry-timer cleanup and guaranteed replacement adapter startup for the forced-shutdown branch.

## Best Possible Solution

Add regression-backed recovery for health-monitor restarts when a provider task fails to settle after abort. The durable end state should replace or quarantine the stuck task for restart intent while preserving manual-stop deduplication; Slack-specific SDK cleanup should stay in the Slack plugin unless a generic lifecycle seam is needed.

## Reproduction Assessment

Yes. The issue includes timestamped live Slack logs, and current main has a focused unit-test reproduction of the core manager failure: after the 5s stop timeout, a subsequent `startChannel` does not create a second account task.

## Solution Assessment

No for current main. It preserves the stuck task as running and blocks the restart the health monitor requested; the safest repair is a narrow restart-recovery path with a flipped regression test, not a broad removal of duplicate-start protection.
