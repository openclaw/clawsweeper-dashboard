## Summary

Keep open. Current main still has the reported forced-stop restart failure shape: the health monitor calls stop then start, but a stop timeout leaves the old account task registered, and the start path returns without instantiating a replacement. The related Slack clean-stop fix from #56646 is present, but it does not cover the abort-timeout branch described here.

## What This Changes

The issue requests a Slack Socket Mode recovery fix for the case where a health-monitor restart hits the channel-stop timeout and leaves the adapter silently offline until a manual gateway restart.

## Best Possible Solution

Add regression-backed recovery for health-monitor restarts when a provider task fails to settle after abort. The durable end state should let restart-intent replace or quarantine the stuck task while preserving manual-stop deduplication, and Slack-specific SDK cleanup should stay in the Slack plugin unless a generic channel lifecycle seam is needed.

## Reproduction Assessment

Yes. The issue includes timestamped live Slack logs, and current main has a focused unit-test reproduction of the core failure: after the 5s stop timeout, a subsequent `startChannel` does not create a second account task.

## Solution Assessment

No for current main; it preserves the stuck task as running and blocks the restart that the health monitor requested. The safest repair is a narrow restart-recovery path plus a flipped regression test, not a broad removal of the duplicate-task guard for all starts.
