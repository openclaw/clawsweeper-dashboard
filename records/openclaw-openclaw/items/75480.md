## Summary

Keep open. Current main still has a source-level reproduction for the reported forced-stop branch: a channel stop timeout leaves the account task registered, and the existing regression test asserts that a subsequent start does not create a replacement task. The related merged Slack clean-stop fix covers the `app.stop()` race from #56508, but the abort-timeout restart path described here remains unresolved.

## What This Changes

The issue asks for Slack Socket Mode to recover after a channel-stop abort timeout instead of leaving the gateway healthy while the Slack adapter stops dispatching inbound events.

## Best Possible Solution

Add a regression-backed recovery path for health-monitor restarts where a stop timeout leaves an aborted provider task stuck, while preserving manual-stop deduplication and keeping Slack-specific SDK cleanup inside the Slack plugin when possible.

## Reproduction Assessment

Yes. The issue has timestamped live logs, and current main already contains a focused unit-test reproduction of the core failure shape: after a stop timeout, `startChannel` is called but no second account task is allowed to start.

## Solution Assessment

Unclear for the exact implementation because there is no patch yet. The direction should be narrow: distinguish restart recovery from manual stop and/or make Slack’s abort path settle promptly, instead of weakening the generic duplicate-task guard for all stop callers.
