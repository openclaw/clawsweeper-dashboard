## Summary

Keep open: current `main` and the latest release still return waited `sessions_send` replies inline and then start the normal A2A announce path for non-parent peer sessions. The report is credible, but existing docs and tests intentionally preserve normal peer A2A follow-up delivery, so the exact waited-send contract needs maintainer judgment before changing behavior.

## What This Changes

The issue reports duplicate requester-visible output when waited `sessions_send` returns `toolResult.reply` and later reinjects an inter-session A2A announce for the same logical reply.

## Best Possible Solution

Define the waited peer-session `sessions_send` contract, then update the session-send/A2A code and focused tests so inline replies are not re-announced as duplicate requester-visible messages while timeout and fire-and-forget delivery still work.

## Reproduction Assessment

Yes, source-reproducible: current main returns a waited reply inline and still invokes the A2A announce path, with tests asserting the combined behavior. I did not run the full webchat/Docker scenario.

## Solution Assessment

Unclear until maintainers settle the contract: suppressing duplicate waited replies matches the report, but current docs/tests intentionally keep normal peer A2A follow-up delivery. The safer path is a narrow behavior decision followed by code, test, and docs alignment.
