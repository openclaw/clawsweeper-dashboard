## Summary

The PR still has a meaningful unique batch CLI liveness fix: current `main` has live-session diagnostic progress, but the batch `executePreparedCliRun` stdout callback still only records output/parsing state and does not refresh `run.progress`. It should stay open for maintainer review because the branch is currently conflicting and the main unresolved question is whether raw batch stdout should count as diagnostic liveness for availability recovery.

## What This Changes

The PR adds a throttled `run.progress` diagnostic event with reason `cli:stdout` from the batch CLI runner stdout callback and adds focused Vitest seam coverage for emission, throttling, and no-output behavior.

## Best Possible Solution

Refresh the branch, update the PR body to describe the batch-only `cli:stdout` change, and have diagnostics owners decide whether raw batch stdout is the intended liveness contract before merge.

## Reproduction Assessment

Yes at source level: current `main` has no diagnostic progress emission in the batch stdout callback, while diagnostic recovery depends on `lastProgressAgeMs`. I did not run a live six-minute CLI repro in this read-only review.

## Solution Assessment

Yes as a narrow batch-path fix, but it is not merge-ready while conflicting and while the raw-stdout liveness contract awaits maintainer acceptance. A semantic-progress alternative is safer for noisy-output hangs but may miss the batch output activity this PR is trying to preserve.
