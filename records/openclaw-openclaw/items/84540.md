## Summary

Keep open: this targets a real CLI liveness bug, but the live Claude path is still incomplete because the new progress event only fires after text deltas, and the PR does not include after-fix real behavior proof from the patched build.

## What This Changes

The PR adds a throttled helper in `src/agents/cli-runner/execute.ts` to emit `run.progress` diagnostic events from CLI stdout and Claude live assistant-delta callbacks.

## Best Possible Solution

Emit throttled diagnostic progress from the CLI output boundary that sees live and batch output before assistant-delta filtering, cover text and thinking/event-only streams with focused tests, and require redacted after-fix diagnostic proof from a patched long-running Claude CLI session.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main starts embedded-run activity as `embedded_run:started`, the CLI path has no diagnostic progress emission, and stale active embedded runs become abort-eligible after `stuckSessionAbortMs`. The proposed live fix is also source-reproducibly incomplete for thinking-only streams because `onAssistantDelta` is text-delta-only.

## Solution Assessment

No. Emitting progress from batch stdout is directionally right, but the live Claude path should update progress before filtering to assistant text deltas, and the PR needs focused regression coverage plus real after-fix runtime proof.
