## Summary

Keep open: the batch stdout progress signal is useful and the supplied logs show the new diagnostic reasons firing, but the live-session hook is still too late because it only runs after text-delta parsing. The paired open follow-up at https://github.com/openclaw/openclaw/pull/84550 is the right shape for the live reasoning path.

## What This Changes

The PR adds throttled CLI-runner diagnostic `run.progress` emission from batch stdout and Claude live assistant-delta callbacks, plus a focused stdout-throttle test.

## Best Possible Solution

Keep the batch `cli:stdout` progress signal, move Claude live progress to the raw stdout boundary with thinking-delta coverage, and then drop or consciously keep the redundant text-delta emitter.

## Reproduction Assessment

Yes from source inspection and supplied logs. Current main records embedded-run starts and stalls when `lastProgressAgeMs` grows, while the Claude live parser only calls `onAssistantDelta` for `text_delta`, so a thinking-only stream can still reproduce the live-session stall path.

## Solution Assessment

No for the live-session portion. The maintainable fix is to emit live progress at raw stdout or parser-progress scope, while keeping the batch stdout emitter for non-live CLI runs.
