## Summary

Keep open: current main and the latest release still do not carry `skipDeferral` into the SIGUSR1 restart intent or close-stage drain decision, and the linked fixing PR is open rather than merged.

## What This Changes

The issue reports that `openclaw gateway restart --safe --skip-deferral` can still wait on close-stage pending reply drain for replies created by the same restart-requesting turn.

## Best Possible Solution

Review, refresh, and land or replace the linked fixing PR with a narrow intent-threading fix that preserves ordinary restart reply draining while making explicit skip-deferral restarts bypass close-stage reply drain.

## Reproduction Assessment

Yes, at source level: current main forwards `skipDeferral` into restart scheduling but does not carry it into the restart intent consumed by close-stage shutdown, so normal pending-reply drain timing still applies.

## Solution Assessment

Yes. Carrying the explicit skip-deferral intent into the SIGUSR1 close-drain decision is the narrow maintainable fix; a docs-only clarification would leave the operator escape hatch behavior mismatched.
