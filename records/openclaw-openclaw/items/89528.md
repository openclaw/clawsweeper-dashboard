## Summary

Keep open: current main and the latest release still lack `skipDeferral` state in the restart intent consumed by close-stage shutdown, and the linked fixing PR is open, unmerged, and currently conflicting.

## What This Changes

The issue reports that `openclaw gateway restart --safe --skip-deferral` can still block in the shutdown pending-reply drain for replies emitted by the same restart-requesting turn.

## Best Possible Solution

Review, refresh, and land or replace the linked fixing PR with a narrow intent-threading fix that preserves ordinary safe-restart reply draining while making explicit skip-deferral restarts bypass the close-stage reply drain.

## Reproduction Assessment

Yes, at source level. Current main records `skipDeferral` for scheduling but does not carry it in `GatewayRestartIntent`, and close-stage reply draining still computes the normal restart drain timeout.

## Solution Assessment

Yes. Carrying the explicit skip-deferral intent into the SIGUSR1 close-drain decision is the narrow maintainable fix; a docs-only clarification would leave the operator escape hatch behavior broken.
