## Summary

Keep open: current main still matches the reported mismatch. The safe skip path carries `skipDeferral` only far enough to bypass the pre-SIGUSR1 deferral gate, while restart shutdown still calls the close-stage pending-reply drain against the global reply dispatcher count.

## What This Changes

The issue asks for `openclaw gateway restart --safe --skip-deferral` to avoid being held by the restart-requesting turn's own tool/final replies, or to make the remaining drain semantics explicit.

## Best Possible Solution

Carry the explicit skip-deferral intent through SIGUSR1 restart shutdown and make only that path skip, tightly bound, or self-reply-filter the close-stage reply drain, with regression coverage that ordinary safe restarts still drain replies.

## Reproduction Assessment

Yes, at source level: run a channel-originated turn that requests `openclaw gateway restart --safe --skip-deferral` while its tool/final replies are still pending, then observe restart shutdown calling the pending-reply drain. I did not live-run the managed macOS Gateway path.

## Solution Assessment

Yes, keeping this open for a code fix is the best path because the docs and CLI present skip-deferral as an operator escape hatch. A docs-only clarification would leave the reported operational wait intact.
