## Label Justifications

- `P2`: The issue affects visible Telegram group/topic delivery but is a bounded behavior change with a privacy policy decision rather than an emergency regression.
- `impact:message-loss`: The reported failure mode is a successful agent run whose final output updates session history but is not delivered visibly to the Telegram group/topic.

## Summary

Keep open: current main still requires explicit `deliver === true` in the gateway and CLI agent-send paths. The recent shipped source-reply fixes cover adjacent Telegram group auto-reply and message-tool diagnostics, but they do not implement this gateway session-key delivery default, and changing group-visible delivery intent needs maintainer policy approval.

## What This Changes

Request that Telegram group/topic-bound agent sessions infer visible delivery from the session key when `deliver` is omitted while preserving explicit `deliver: false`.

## Best Possible Solution

Maintainers should decide whether a group-bound session key is sufficient visible-delivery intent; if yes, land a shared gateway/CLI delivery helper with explicit `deliver:false` override, focused regression tests, docs, and live Telegram group/topic proof.

## Reproduction Assessment

Yes, at source level: current gateway and CLI delivery paths require explicit `deliver === true`, so omitted `deliver` remains session-only even for Telegram group/topic session keys. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear. The proposed helper is narrow, but inferred visible delivery in shared Telegram rooms is policy-sensitive; a config-gated mode or agent-visible suppressed-final diagnostic may be safer than changing the default unconditionally.
