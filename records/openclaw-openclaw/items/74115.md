## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when the effective config is missing `gateway.mode`, so the central no-config startup request is not implemented; the WebUI no-provider and Raw-mode claims remain plausible but under-specified for a narrow code fix.

## What This Changes

The issue asks OpenClaw to let the Gateway and WebUI start without requiring full config/model/provider setup, and adds that Raw mode is not working.

## Best Possible Solution

Define the intended startup contract explicitly: either add a reviewed, loopback-only no-config/no-provider bootstrap mode with Gateway, Control UI, docs, and regression tests, or keep the current fail-closed policy and improve setup/repair messaging. Any verified Raw-mode or no-provider WebUI crash should be split into focused repro-driven UI/Gateway RPC issues.

## Reproduction Assessment

Partial. The Gateway policy portion is reproducible from source and tests: normal startup blocks missing `gateway.mode`, while `--allow-unconfigured` covers a bootstrap path; the no-provider WebUI crash and Raw-mode failure do not yet have enough concrete data for high-confidence reproduction.

## Solution Assessment

Unclear. The requested broad relaxation is not obviously the best fix because it changes an intentional security-sensitive startup policy; a safer path is an explicit loopback bootstrap mode or clearer repair guidance, plus separate fixes for any reproduced UI crash.
