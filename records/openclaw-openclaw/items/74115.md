## Summary

Keep open. Current main still intentionally blocks normal Gateway startup unless `gateway.mode=local` is present or `--allow-unconfigured` is used, so the central startup-policy request is not implemented; the no-provider WebUI and Raw-mode complaints remain plausible but under-specified.

## What This Changes

The issue asks for Gateway and Control UI startup to work before all config/model/provider setup is complete, including no crash when no provider is configured and a working Raw config mode.

## Best Possible Solution

Define the intended bootstrap contract first: either add a safe loopback-only no-config/no-provider Gateway and Control UI mode with regression coverage, or keep the current fail-closed guard and improve setup/recovery messaging. Any verified Raw-mode or no-provider WebUI crash should be split into a focused UI/Gateway RPC bug with a minimal fixture.

## Reproduction Assessment

Partial/unclear. The Gateway policy portion is reproducible from source and tests: a valid observed config snapshot without `gateway.mode` makes normal `gateway run` exit before server startup; the WebUI no-provider and Raw-mode claims lack enough concrete data to reproduce independently.

## Solution Assessment

Unclear. A simple default-to-local change is not clearly the best fix because current behavior is a deliberate fail-closed guard; the safer path is an explicit bootstrap policy decision plus narrow UI bug reports for any independently reproducible crashes.
