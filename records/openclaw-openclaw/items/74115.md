## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing, so the central no-config startup request is not implemented; the no-provider WebUI crash and Raw-mode follow-up still need narrower reproduction evidence.

## What This Changes

The issue asks for Gateway and WebUI startup to remain usable without complete settings or provider/model configuration.

## Best Possible Solution

Define a reviewed startup contract: either add a loopback-only bootstrap state that keeps Control UI reachable without provider setup, or keep fail-closed startup and improve setup/repair messaging while fixing any separately reproduced UI crash.

## Reproduction Assessment

Partial. The Gateway startup-policy portion is reproducible from source and tests; the no-provider WebUI crash and Raw-mode failure are unclear because the report gives no exact UI path, config snapshot, console output, or logs.

## Solution Assessment

Unclear. The requested broad relaxation is not yet proven to be the narrowest maintainable fix because current code and docs encode `gateway.mode` as an intentional safety guard; a reviewed bootstrap mode plus targeted UI fixes would be safer.
