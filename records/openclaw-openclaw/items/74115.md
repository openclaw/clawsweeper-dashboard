## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing, so the central no-config startup request is not implemented; the no-provider WebUI and Raw-mode follow-up remain plausible but need focused reproduction data.

## What This Changes

The issue requests that Gateway and the WebUI start without requiring full configuration or model/provider setup, and not crash when no provider is configured.

## Best Possible Solution

Define the intended startup contract explicitly: either add a reviewed, loopback-only no-config/no-provider bootstrap mode with Gateway, Control UI, docs, and regression tests, or keep the current fail-closed policy and improve setup/repair messaging. Any verified Raw-mode or no-provider WebUI crash should be split into focused repro-driven UI/Gateway RPC issues.

## Reproduction Assessment

Unclear overall. The Gateway policy portion is reproducible from source and tests, but the no-provider WebUI crash and Raw-mode failure lack concrete data for a high-confidence reproduction.

## Solution Assessment

Unclear. The requested broad relaxation is not obviously the safest fix because it changes an intentional startup policy; a safer path is an explicit loopback bootstrap contract plus focused fixes for any reproduced UI crash.
