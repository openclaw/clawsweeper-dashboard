## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing, so the central no-config startup request is not implemented; the no-provider WebUI and Raw-mode claims need narrower reproduction before a safe fix can be scoped.

## What This Changes

The issue asks for Gateway and Control UI startup to work without fully configured settings or providers, and adds that Raw mode is broken.

## Best Possible Solution

Define a reviewed startup contract: either add a loopback-only bootstrap state that keeps Control UI reachable without providers, or keep fail-closed startup and improve setup/repair messaging plus targeted UI fixes for reproduced crashes.

## Reproduction Assessment

Partial. The Gateway policy portion is reproducible from source and tests: normal startup blocks missing `gateway.mode`, while `--allow-unconfigured` covers a bootstrap path; the no-provider WebUI crash and Raw-mode failure still lack concrete reproduction data.

## Solution Assessment

Unclear. The requested broad relaxation is not yet the narrowest maintainable solution because it changes an intentional Gateway safety guard; a safer path is a maintainer-approved bootstrap mode plus separate fixes for any reproduced UI crash.
