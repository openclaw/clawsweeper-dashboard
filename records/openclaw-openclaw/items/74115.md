## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when the effective config lacks `gateway.mode`, while the no-provider WebUI crash and Raw-mode complaint remain plausible but under-specified.

## What This Changes

The issue asks for Gateway and WebUI startup to work without complete config, provider, or model setup, and not crash when no provider is configured.

## Best Possible Solution

Define the startup contract explicitly: either keep the fail-closed default with clearer setup/repair UI, or add a reviewed loopback-only bootstrap path covering Gateway, Control UI, no-provider model picker behavior, docs, and regression tests.

## Reproduction Assessment

Partial. The Gateway policy portion is reproducible from source and tests: normal startup blocks missing `gateway.mode`, while `--allow-unconfigured` covers a bootstrap path; the no-provider WebUI crash and Raw-mode failure do not yet have enough concrete data for high-confidence reproduction.

## Solution Assessment

Unclear. Broadly removing startup/config guards is not obviously the safest fix; a narrower reviewed loopback bootstrap contract plus separate reproduced UI/Raw-mode fixes would be more maintainable.
