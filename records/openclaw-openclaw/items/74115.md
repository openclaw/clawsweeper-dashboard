## Summary

Keep open. Current main has not implemented the central startup-policy request: normal Gateway startup still fails closed when `gateway.mode` is missing, and docs/tests encode that as intentional behavior. The WebUI no-provider crash and Raw-mode complaint remain plausible but under-specified, so this needs maintainer product/security judgment rather than conservative cleanup.

## What This Changes

The issue requests that Gateway and WebUI startup tolerate incomplete configuration, including no configured model/provider, rather than crashing or blocking before setup is complete.

## Best Possible Solution

The durable end state should be an explicit maintainer-approved startup contract: either normal loopback Gateway startup safely supports a no-config/no-provider bootstrap Control UI, or the existing fail-closed guard remains and setup/recovery guidance is improved. Any verified WebUI no-provider crash or Raw-mode regression should then be split into focused fixes with reproduction steps and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear. The Gateway policy portion is source- and test-reproducible because normal startup blocks missing `gateway.mode` while `--allow-unconfigured` starts, but the WebUI no-provider crash and Raw-mode failure lack concrete steps, logs, config, or version data.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A broad automatic change to make normal Gateway startup ignore incomplete config is not clearly the safest fix because current docs/tests intentionally fail closed; a safer path is maintainer-approved policy first, then narrow UI/model/raw fixes for any verified regressions.
