## Summary

Keep open. Current main still deliberately blocks normal Gateway startup when `gateway.mode` is missing, with `--allow-unconfigured` only as a bootstrap escape hatch; the no-provider WebUI and Raw-mode complaints need narrower reproduction before a safe code fix can be scoped.

## What This Changes

The issue requests that Gateway and Control UI startup not require complete configuration or configured model providers, and adds that Raw mode is failing.

## Best Possible Solution

Define a reviewed startup contract: either add a loopback-only bootstrap state that keeps Control UI reachable without provider setup, or keep fail-closed startup and improve setup/repair messaging while fixing separately reproduced UI crashes.

## Reproduction Assessment

Partial. The Gateway policy portion is reproducible from source and tests: ordinary startup blocks missing `gateway.mode`, while `--allow-unconfigured` starts the bootstrap path; the no-provider WebUI crash and Raw-mode failure still lack concrete reproduction data.

## Solution Assessment

Unclear. The requested broad relaxation is not yet proven to be the narrowest maintainable fix because it changes an intentional startup guard; a safer path is a maintainer-approved bootstrap mode plus targeted UI fixes for reproduced crashes.
