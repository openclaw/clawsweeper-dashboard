## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing, so the central no-config startup request is not implemented; the no-provider Web UI and Raw-mode claims need narrower reproduction before they are actionable fixes.

## What This Changes

The issue requests that Gateway and Web UI startup avoid crashing or blocking when provider/model settings are incomplete.

## Best Possible Solution

Define a reviewed startup contract: either add a loopback-only bootstrap state that keeps Control UI reachable without providers, or keep fail-closed startup and improve setup/doctor messaging plus targeted UI fixes for reproduced crashes.

## Reproduction Assessment

Partial. The Gateway policy portion is reproducible from source and tests: normal `gateway run` blocks missing `gateway.mode`, while `--allow-unconfigured` covers bootstrap; the Web UI no-provider crash and Raw-mode failure lack enough concrete data for high-confidence reproduction.

## Solution Assessment

Unclear. The requested broad relaxation is not yet the narrowest maintainable fix because it changes an intentional Gateway safety guard; a safer path is maintainer-approved bootstrap behavior plus separate fixes for any reproduced UI crash.
