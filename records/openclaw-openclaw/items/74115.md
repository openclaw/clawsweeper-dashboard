## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is absent unless `--allow-unconfigured` is used, so the central startup-policy request is not implemented; the WebUI no-provider and Raw-mode complaints are plausible but still lack concrete reproduction details for a narrow repair.

## What This Changes

Request that the Gateway and browser Control UI start in a useful bootstrap state without requiring all settings or model providers to be configured, and that Raw mode continue working.

## Best Possible Solution

Define the intended startup contract explicitly: either support a safe loopback no-config/no-provider bootstrap mode in Gateway and Control UI with tests, or keep the current fail-closed guard and improve setup/recovery messaging. Any verified Raw-mode or no-provider WebUI crash should then be split into a focused UI/Gateway RPC bug with a minimal config fixture and regression test.

## Reproduction Assessment

Partial/unclear. The Gateway policy portion is reproducible from source and tests: normal startup blocks missing `gateway.mode` while `--allow-unconfigured` starts; the WebUI no-provider crash and Raw-mode failure do not have enough concrete data to reproduce independently.

## Solution Assessment

Unclear. A single narrow code change is not currently the best path because the central behavior is an intentional documented guard, while the remaining UI complaints need separate reproduction before choosing a fix.
