## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing unless `--allow-unconfigured` is used, so the central startup-policy request is not implemented; the WebUI no-provider and Raw-mode complaints remain plausible but under-specified for a narrow repair.

## What This Changes

The issue requests that Gateway and the WebUI start without requiring all configuration, provider, or model setup first.

## Best Possible Solution

Define the intended startup contract explicitly: either add a safe loopback-only no-config/no-provider bootstrap mode with Gateway, Control UI, tests, docs, and security review, or keep the current fail-closed guard and improve setup/recovery messaging. Any verified Raw-mode or no-provider WebUI crash should be split into a focused UI/Gateway RPC bug with a minimal config fixture and regression test.

## Reproduction Assessment

Partial. The Gateway policy portion is reproducible from source and tests: normal startup blocks missing `gateway.mode`, while `--allow-unconfigured` covers the bootstrap path; the WebUI no-provider crash and Raw-mode failure do not yet have enough concrete data to reproduce.

## Solution Assessment

Unclear. The requested direction is not currently the narrowest maintainable fix because it would change an intentional security-sensitive startup policy; a safer path is either an explicit loopback bootstrap mode or better setup/repair guidance, plus separate repro-driven fixes for any UI crash.
