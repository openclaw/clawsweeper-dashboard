## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is absent unless `--allow-unconfigured` is used, so the central startup-policy request is not implemented; the WebUI no-provider and Raw-mode claims remain plausible but under-specified for a narrow automated fix.

## What This Changes

The issue asks OpenClaw to let the Gateway and WebUI start with incomplete configuration, including no configured provider/model, and not crash when setup is incomplete.

## Best Possible Solution

Define the intended first-run startup contract explicitly: either add a safe loopback no-config/no-provider bootstrap mode with Gateway and Control UI regression coverage, or keep the current fail-closed guard and improve setup/recovery messaging. Verified Raw-mode or no-provider UI crashes should be split into focused bugs with minimal config fixtures and UI/Gateway tests.

## Reproduction Assessment

Partial/unclear. The Gateway policy portion is reproducible from source and tests: normal startup blocks missing `gateway.mode`, while `--allow-unconfigured` starts the bootstrap path; the WebUI no-provider crash and Raw-mode failure do not yet have enough concrete data for high-confidence reproduction.

## Solution Assessment

Unclear. A one-shot code change is not currently the best path because the main behavior is an intentional documented guard, while the UI complaints need separate reproduction before choosing a narrow fix.
