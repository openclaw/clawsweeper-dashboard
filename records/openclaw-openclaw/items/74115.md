## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when the config is missing or lacks `gateway.mode`, unless `--allow-unconfigured` is used, and docs/tests encode that as current policy. The WebUI no-provider and Raw-mode claims are plausible but still lack the concrete config, route, log, or version detail needed for a narrow cleanup close or repair PR.

## What This Changes

The issue requests that Gateway and WebUI startup tolerate incomplete configuration and no configured model/provider instead of crashing or blocking startup.

## Best Possible Solution

Define the intended startup contract explicitly: either support a safe loopback no-config/no-provider bootstrap mode in Gateway and Control UI with regression coverage, or keep the fail-closed guard and improve setup/recovery messaging. Any verified Raw-mode or no-provider WebUI crash should be split into a focused UI/Gateway RPC bug with a minimal config fixture.

## Reproduction Assessment

Partial/unclear. The Gateway policy portion is reproducible from source and tests: normal startup blocks missing `gateway.mode` while `--allow-unconfigured` bypasses the guard; the WebUI no-provider crash and Raw-mode failure do not have enough concrete data to reproduce independently.

## Solution Assessment

Unclear. A single narrow code change is not currently the best path because the central behavior is an intentional documented guard, while the remaining UI complaints need separate reproduction before choosing a fix.
