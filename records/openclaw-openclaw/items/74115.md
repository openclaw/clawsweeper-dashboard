## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing unless `--allow-unconfigured` is used, so the central startup-policy request is not implemented; the no-provider WebUI and Raw-mode parts also lack enough concrete reproduction detail to identify a narrow source fix.

## What This Changes

The issue requests that the Gateway and Control UI start without requiring complete config/model/provider setup, avoid crashing when no provider is configured, and keep Raw mode working.

## Best Possible Solution

The durable end state should be an explicit maintainer-approved startup contract: either normal loopback Gateway startup safely supports a no-config/no-provider bootstrap state, or the existing guard remains and setup/recovery paths are made clearer. Any verified WebUI no-provider crash or Raw-mode regression should then get its own focused reproduction, regression test, and fix in the Control UI/Gateway RPC surface.

## Reproduction Assessment

Unclear. The Gateway policy portion is source- and test-reproducible because normal startup blocks missing `gateway.mode` while `--allow-unconfigured` starts, but the WebUI no-provider crash and Raw-mode failure do not have concrete steps, logs, or config snapshots.

## Solution Assessment

Unclear. This is not ready for a single narrow automated fix because current main already has setup/bootstrap helpers and model-list fallbacks, while changing the normal startup guard is a product/security policy decision.
