## Summary

Keep open: current main and the latest release still only log or silently drop unauthorized Telegram inbound attempts and expose static configuration audit findings, while the requested runtime audit trail, owner notification, webhook/digest handling, retention, and recent-attempt CLI/status surface remain missing. The related PR #89569 is useful context but is open, conflicting, and narrower than this DM/group audit and alert request.

## What This Changes

Request a supported Telegram security-event/audit path for unauthorized DM, group, group-allowlist, and command attempts with configurable alert handling and safe metadata defaults.

## Best Possible Solution

Define a Telegram-owned unauthorized-inbound security event/audit surface with redacted bounded storage, CLI/status inspection, and explicit opt-in delivery modes while keeping blocked content out of model context and preserving fail-closed authorization defaults.

Unclear as proposed: the security visibility goal is sound, but the best fix needs maintainer decisions on event/API shape, retention, redaction, delivery modes, defaults, and rate limiting before implementation.
