## Summary

Keep open: current main blocks unauthorized Telegram DM/group/callback/command paths and exposes static security-audit checks, but it does not provide the requested structured runtime alert or recent blocked-attempt audit surface.

## What This Changes

Request adding Telegram runtime security events, owner/webhook alerts, and an audit/history surface for unauthorized inbound attempts blocked by access policies and allowlists.

## Best Possible Solution

Design a Telegram-owned runtime security-event path with redacted metadata, rate limiting or aggregation, optional owner/webhook delivery, and durable CLI/status inspection, adding generic channel seams only if other channels need the same contract.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection show current main enforces Telegram allowlists and exposes static security-audit checks, but no supported runtime blocked-attempt alert or recent-history surface.

## Solution Assessment

Unclear until maintainers choose the product/security shape. A Telegram-owned implementation with privacy-safe metadata is more maintainable than documenting log scraping or routing unauthorized content into agent turns.
