## Summary

Keep open: current main and v2026.6.10 still block unauthorized Telegram DM/group attempts, but only through drop/log paths and existing config audits, not through the requested retained security-event stream, owner notification, webhook/digest handling, or CLI/status audit surface. The related access-request PR is useful adjacent work, but it is open, conflicting, and DM-focused, so it does not replace this broader Telegram DM/group audit request.

## What This Changes

Request a first-class audit and alert path for Telegram inbound attempts blocked by `dmPolicy`, `groupPolicy`, `allowFrom`, `groupAllowFrom`, or `groups` allowlists.

## Best Possible Solution

Define a privacy-preserving denied-inbound audit contract first, then implement Telegram on that contract with safe metadata defaults, rate limiting, retained inspection, and no unauthorized user content entering model context by default.

## Reproduction Assessment

Yes, at source level: current Telegram DM and group denial paths block unauthorized inbound events, while searched schema/docs/runtime surfaces do not expose the requested alert, webhook/digest, or recent-attempt audit view.

## Solution Assessment

Unclear until maintainers choose the security-event contract; Telegram is a sensible first target, but the durable solution should settle shared event shape, retention, redaction, notification, and rate limiting before code is queued.
