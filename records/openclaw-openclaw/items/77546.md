## Summary

Keep open: current main and v2026.6.10 still enforce the Telegram DM/group access denials, but only through drop/log paths and existing configuration audits, not through the requested retained security-event stream, owner notification, webhook/digest handling, or CLI/status audit surface. The related access-request PR is useful adjacent work, but it is open, conflicting, DM-focused, and does not replace the broader Telegram DM/group audit requirement.

## What This Changes

Request a first-class Telegram security-event, audit, and notification surface for denied DM, group, and command attempts without changing authorization semantics.

## Best Possible Solution

Define a privacy-preserving denied-inbound audit contract first, then implement Telegram on that contract with safe metadata defaults, rate limiting, retained inspection, and no unauthorized user content entering model context by default.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current Telegram DM/group denial paths block or log unauthorized inbound events, and the searched schema/docs/runtime paths do not expose a supported alert, audit, webhook/digest, or recent-attempt inspection surface.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the security-event contract; Telegram is a sensible first target, but the durable solution should settle shared event shape, redaction, storage, notification, and rate-limit semantics before code is queued.
