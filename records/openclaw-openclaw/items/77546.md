## Summary

Keep open: current main enforces Telegram allowlist blocking and has static security-audit checks, but it does not provide the requested runtime alert/history surface for blocked inbound attempts.

## What This Changes

Request structured alerts and audit history for Telegram inbound attempts blocked by DM policy, group policy, sender allowlists, or group allowlists.

## Best Possible Solution

Design a Telegram-owned runtime security-event path with redacted metadata, rate limiting, durable inspection, and optional owner/webhook delivery, adding generic channel seams only if other channels need the same contract.

## Reproduction Assessment

Not applicable as a feature request. Source and docs verification show current main enforces Telegram allowlists and exposes static security-audit checks, but no runtime alert or recent blocked-attempt history surface.

## Solution Assessment

Unclear until maintainers choose the product/security shape. A Telegram-owned event/audit implementation with privacy-safe metadata is more maintainable than documenting log scraping or routing unauthorized content into agent turns.
