## Summary

Keep open: current main enforces Telegram DM/group allowlists and has static security-audit checks, but it still lacks the requested runtime alert, notification, or recent blocked-attempt history surface.

## What This Changes

Request adds structured security visibility and optional operator alerts for unauthorized Telegram DM/group attempts blocked by existing allowlist policies.

## Best Possible Solution

Design a Telegram-owned runtime security-event path with minimal redacted metadata, bounded retention, rate limiting, and optional owner/webhook delivery; add generic channel seams only if maintainers want the same contract across channels.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection show existing blocking behavior and static audit coverage, but no runtime blocked-attempt alert or history surface to reproduce as a bug.

## Solution Assessment

Unclear until maintainers choose the product/security shape. A Telegram-owned event/audit implementation is safer than documenting log scraping or letting unauthorized content enter normal agent turns.
