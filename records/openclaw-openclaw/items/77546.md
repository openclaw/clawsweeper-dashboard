## Summary

Keep open: current main still blocks unauthorized Telegram DM and group inbound paths with runtime log/audit configuration checks, but it does not provide the requested first-class security event stream, owner notification, webhook/digest handling, retained recent-attempt audit surface, or CLI/status inspection path. The open access-request pull request is useful adjacent work for blocked direct-message requests, but it is not merged and does not cover this broader Telegram DM/group audit requirement.

## What This Changes

The issue requests a supported Telegram security-audit and alert surface for unauthorized inbound attempts blocked by DM, group, sender, or group allowlist policies.

## Best Possible Solution

Define a maintainer-approved, privacy-preserving denied-inbound audit contract first, then implement Telegram on that contract without changing existing authorization semantics or sending unauthorized content into agent context by default.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: Telegram unauthorized DM and group paths are blocked by policy/allowlist checks, while no supported audit, notification, webhook/digest, or recent-attempt inspection surface exists on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the cross-channel security-event contract; Telegram is a sensible first target, but the storage, redaction, rate-limit, notification, and config defaults need an explicit product/security decision.
