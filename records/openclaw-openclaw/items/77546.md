## Summary

Keep open: current main still enforces Telegram allowlist blocking, but blocked inbound attempts are handled through log/drop or command rejection paths rather than the requested supported runtime alert, event history, or audit surface.

## What This Changes

Request a first-class Telegram security-event/audit and notification surface for blocked unauthorized DM, group, and command attempts with privacy-safe metadata and rate limiting.

## Best Possible Solution

Design a Telegram-owned runtime security-event pipeline with redacted metadata, rate limiting, durable inspection, and optional owner/webhook/digest delivery, adding a generic channel seam only after maintainers agree on scope.

## Reproduction Assessment

Not applicable as a feature request. Source and docs verification show current main enforces Telegram allowlists and exposes static security-audit checks, but no supported runtime alert or recent blocked-attempt history surface.

## Solution Assessment

Unclear until maintainers define the product/security shape. A Telegram-owned event/audit implementation with privacy-safe metadata is more maintainable than log scraping or routing unauthorized content into agent turns.
