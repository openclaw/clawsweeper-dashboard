## Summary

Keep open: current main and v2026.6.8 still only block/log Telegram unauthorized DM and group attempts plus expose static configuration audit findings; the requested runtime audit trail, owner notification, webhook/digest handling, retention, and recent-attempt CLI/status surface remain missing. The related open PR at https://github.com/openclaw/openclaw/pull/89569 is useful partial context for blocked DM access requests, but it is open, currently conflicting, and narrower than this Telegram DM/group audit request.

## What This Changes

The issue requests first-class Telegram security visibility for blocked unauthorized DM/group access attempts, including structured audit events, owner notifications, rate limiting, and a recent-attempt inspection surface.

## Best Possible Solution

Define a Telegram-owned unauthorized-inbound security event and audit surface with redacted bounded storage, CLI/status inspection, and explicit opt-in delivery modes while preserving existing fail-closed authorization defaults.

## Reproduction Assessment

Yes, for current behavior at source level: unauthorized Telegram DMs and group messages are denied before normal dispatch, and current audit surfaces are static configuration posture rather than runtime denied-attempt records. No live Telegram deployment was run in this read-only review.

## Solution Assessment

Unclear as proposed: the visibility goal fits OpenClaw security priorities, but maintainers need to decide event/API shape, delivery defaults, redaction, storage, retention, and rate limiting before implementation.
