## Summary

Keep open: current main still only logs or silently drops unauthorized Telegram inbound attempts and exposes static configuration audit findings, but it does not provide the requested supported runtime security-event trail, owner notification, webhook/digest handling, retention, or recent-audit CLI surface. The related open PR is useful context but is not a safe close path because it is open, currently conflicting, and focused on blocked-DM pre-auth access requests rather than this broader Telegram audit/alert request.

## What This Changes

Request a supported security event, alert, and recent-audit surface for Telegram inbound attempts blocked by DM/group policies and allowlists.

## Best Possible Solution

Define a Telegram-owned unauthorized-inbound security event/audit surface with redacted bounded storage, CLI/status inspection, and explicit opt-in delivery modes, while keeping blocked content out of model context and preserving current fail-closed authorization defaults.

Unclear as proposed: the security visibility goal is sound, but the best fix needs maintainer decisions on event/API shape, retention, redaction, delivery modes, defaults, and rate limiting before implementation.
