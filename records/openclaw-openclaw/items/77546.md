## Summary

Keep open: current main enforces Telegram allowlist blocking by logging and dropping unauthorized inbound activity, but it still lacks the requested supported runtime alert, notification, or recent blocked-attempt audit history. This is a security-observability feature, not a broken existing contract, and it needs maintainer product/security design before implementation.

## What This Changes

Add configurable Telegram security visibility for unauthorized DM, group, allowlist, and command attempts, including structured events, redacted metadata, rate limiting, notifications, and an audit/status surface.

## Best Possible Solution

Design a Telegram-owned runtime security-event path with privacy-safe metadata, rate limiting, durable inspection, and optional owner/webhook delivery, adding a generic channel event seam only if maintainers want this across channels.

## Reproduction Assessment

Not applicable as a feature request. Current-main source, docs, and tests show existing Telegram authorization enforcement and static config audit checks, but no supported runtime alert or recent blocked-attempt history surface to reproduce as a broken contract.

## Solution Assessment

Unclear until maintainers choose the product/security shape. A Telegram-owned runtime audit implementation with redacted metadata is safer and more maintainable than documenting log scraping or routing unauthorized content into agent turns.
