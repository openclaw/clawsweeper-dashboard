## Summary

Keep open: current main still blocks unauthorized Telegram inbound activity by dropping/logging it, but it does not provide the requested supported alert, event, retention, or recent-audit surface. This is a security-observability feature with config and privacy choices, not a broken existing contract.

## What This Changes

Request first-class Telegram security alerts and audit history for inbound attempts blocked by DM, group, sender, and group allowlists.

## Best Possible Solution

Design a Telegram-owned runtime security-event path with redacted metadata, rate limiting, inspectable recent history, and optional owner/webhook delivery, adding a generic channel event seam only if maintainers want this across channels.

## Reproduction Assessment

Not applicable as a feature request. Source, tests, and docs show current allowlist enforcement, but there is no existing contract that requires runtime blocked-attempt alerts or history.

## Solution Assessment

Unclear until maintainers choose the security/product shape. A Telegram-owned audit path with privacy-safe metadata is a better direction than documenting log scraping or exposing unauthorized content to agent turns.
