## Summary

Keep open: current main enforces Telegram DM/group allowlist blocking, but the blocked paths still log/drop rather than emit structured runtime security events, notifications, or recent audit history.

## What This Changes

Add a Telegram security-event/audit surface for blocked unauthorized DM, group, group-add, and command attempts with privacy-safe metadata and configurable handling.

## Best Possible Solution

Design a Telegram-owned runtime security-event path with redacted metadata, rate limiting, durable inspection, and optional owner/webhook delivery, adding generic channel seams only if other channels need the same contract.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection show current main blocks unauthorized Telegram inbound events, but only through log/drop paths and static config audits rather than a supported alert or history surface.

## Solution Assessment

Unclear until maintainers choose the product/security shape. A Telegram-owned event/audit implementation with privacy-safe metadata is more maintainable than documenting log scraping or routing unauthorized content into agent turns.
