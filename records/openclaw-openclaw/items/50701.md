## Summary

Keep open. Current main is already archive-aware for usage totals, so this is not raw data loss, but the requested durable logical-session lineage contract and WebUI current-instance vs historical-lineage presentation are still missing; related archive-recovery PRs do not fully cover this usage-rollup request.

## What This Changes

The issue requests transcript-backed usage history rollups that survive updates, restarts, `/new`, `/reset`, and sessionId rebinding for the same logical session surface.

## Best Possible Solution

Define a first-class usage lineage contract: stable family keys derived from session metadata/origin, transcript-backed aggregation across included session IDs, explicit instance-vs-lineage query modes, and a Control UI view that makes the included instances and date scope visible while coordinating with the reset-archive recovery work.

## Reproduction Assessment

Yes for the core product gap: current source shows `sessions.usage` and detail endpoints operate on single session rows/keys with no family selector, and the UI drills into one selected session row. The exact post-update visual reset is not live-reproduced here, but the missing lineage path is directly inspectable.

## Solution Assessment

Unclear as proposed because the requested direction is right but broad. The maintainable fix should start with additive Gateway protocol semantics and lineage-key rules, then add UI controls and any cached/indexed summary needed for performance.
