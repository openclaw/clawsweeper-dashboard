## Summary

Keep open. Current main has archive-aware transcript usage scanning, so the report is not raw data loss, but it still lacks the requested durable logical-session lineage contract and WebUI presentation. The `sessions.usage` protocol and handler operate on individual session IDs/rows, while the usage UI selects and drills into one returned session row rather than a family of historical incarnations.

## What This Changes

The issue requests transcript-backed usage lineage aggregation in the WebUI so usage history for a logical session remains continuous across updates, restarts, resets, and new session IDs.

## Best Possible Solution

Define a first-class logical-session lineage contract for usage: stable family keys derived from session metadata/origin, transcript-backed aggregation across included session IDs, explicit instance-vs-lineage query modes, and a Control UI toggle that makes current-instance and historical-lineage scope visible. Preserve the existing archive-aware global usage behavior and coordinate with the reset-archive recovery work rather than folding this into an implicit one-off UI heuristic.

## Reproduction Assessment

Yes for the core product gap: source inspection shows `sessions.usage` can be queried by current key or discovered session row, but not by historical family, and the UI drills into one selected row. No live `openclaw update` run was performed, so the update-specific rebinding trigger is inferred from the report and current session/reset design rather than directly exercised here.

## Solution Assessment

Unclear. Transcript-backed lineage rollups are the right direction, but the proposed shape is broad; the maintainable fix should start with an explicit Gateway usage contract and lineage-key semantics before adding WebUI toggles or cached indexing.
