## Summary

Keep open. Current main has turned the active `plugin-runtime-deps` growth path into legacy cleanup, but it still lacks the requested first-class cross-cache inventory/prune surface for browser, tools, memory, and orphan transcript/trajectory state. The maintainer discussion also points to keeping the broader local-state feature open while updating the wording around `plugin-runtime-deps`.

## What This Changes

Add an `openclaw caches` command, or equivalent doctor integration, to list cache-like `~/.openclaw` directories by size/class and safely prune stale or regenerable state.

## Best Possible Solution

Keep this open as a core local-state maintenance feature: define a cache inventory model with explicit cache-vs-data classes, size/mtime reporting, dry-run-first output, allowlisted prune targets, active version/lock/session protections, and updated docs that treat `plugin-runtime-deps` as legacy cleanup rather than an actively growing cache on current main.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence verification path: current main has no `openclaw caches` command while only partial doctor, sessions, browser, and memory maintenance surfaces exist.

## Solution Assessment

Unclear whether the best public shape is a top-level `openclaw caches` command, doctor integration, or both. The best direction is the requested class-based inventory plus conservative prune rules, with maintainer decisions before implementation because local-state deletion boundaries are product- and safety-sensitive.
