## Summary

Keep open. Current main turns the `plugin-runtime-deps` growth into legacy cleanup, but the broader requested `~/.openclaw` cache inventory/prune surface is still missing; existing code only covers doctor cleanup for legacy plugin dependency debris and session-specific maintenance.

## What This Changes

Request a first-class `openclaw caches` or doctor-integrated command to list cache-class directories under `~/.openclaw`, show sizes and owners, and safely prune stale/runtime-regenerable data.

## Best Possible Solution

Design a core-owned cache inventory/prune surface that classifies cache versus data, reports sizes and populators, defaults to dry-run or conservative stale-only pruning, treats `plugin-runtime-deps` as legacy cleanup on current main, and keeps workspace, credentials, active sessions, active runtime roots, locks, and fresh transcripts protected.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence verification path: target main lacks `openclaw caches`, while existing doctor and sessions paths cover only partial cleanup slices.

## Solution Assessment

Unclear whether the public entry point should be top-level `openclaw caches`, doctor integration, or both. The requested class-based inventory plus conservative prune defaults is still the right product direction; current legacy plugin cleanup alone is not sufficient.
