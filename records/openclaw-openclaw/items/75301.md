## Summary

Keep open. Current main has useful partial cleanup for bundled runtime deps and session maintenance, but it still lacks the requested cross-cache `~/.openclaw` inventory/prune command and the destructive deletion boundaries need maintainer design.

## What This Changes

Add an `openclaw caches` or `doctor --prune-caches` surface that lists cache-class directories under `~/.openclaw`, explains what populates them, and safely prunes stale runtime-deps roots, orphan transcripts, and optional regenerable browser/tools/memory caches.

## Best Possible Solution

Design a core-owned cache inventory with dry-run-first pruning, explicit cache-vs-data classes, allowlisted deletion targets, lock/current-version protection for runtime deps, fresh/live transcript safeguards, and opt-in pruning for expensive or user-adjacent caches such as memory, browser, and tools.

## Reproduction Assessment

Not applicable as a feature request. The absence of the requested command is verified from current command descriptors, docs, and source search; reproducing the reported disk growth would require a long-lived install with updates and cron sessions.

## Solution Assessment

Unclear whether the best public surface is a top-level command or doctor integration. The maintainable direction is the requested class-based inventory plus conservative dry-run pruning rules, with any automatic deletion left for later approval.
