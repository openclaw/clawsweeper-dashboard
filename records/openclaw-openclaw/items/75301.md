## Summary

Keep open. Current main still lacks the requested cross-cache inventory/prune command; the existing runtime-deps, sessions cleanup, and doctor paths cover only partial slices. The feature is valid core local-state maintenance, but the deletion boundaries and public command shape need maintainer design before a safe automated repair PR.

## What This Changes

Request an `openclaw caches` or doctor cache-prune surface that lists and safely prunes cache-class state directories under `~/.openclaw/`.

## Best Possible Solution

Design a core-owned cache inventory with dry-run-first pruning, explicit cache-vs-data classes, allowlisted deletion targets, active/current-version and lock protection, fresh/live transcript safeguards, and opt-in pruning for browser, tools, memory, and other regenerable but expensive caches.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high-confidence verification that current main lacks the requested cross-cache command while only partial cleanup surfaces exist.

## Solution Assessment

Unclear whether the best public entry point is top-level `openclaw caches`, doctor integration, or both. The safest direction is the requested class-based inventory plus conservative dry-run pruning defaults, leaving automatic deletion for later maintainer approval.
