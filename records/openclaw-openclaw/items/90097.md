## Summary

Keep this issue open: current main still routes already-managed non-image PDF offloads through `prestageMediaPathOffloads`, where sandbox size/incomplete-stage handling can fail the send before dispatch. The request is a bounded gateway bug with clear safety constraints and no implemented-on-main or duplicate evidence found in the provided context or local source sweep.

## What This Changes

The issue requests that `chat.send` pass already-managed inbound PDFs through as `MediaPath`/`MediaPaths` and fall back to those paths if optional sandbox staging fails.

## Best Possible Solution

Land a narrow gateway fix that keeps arbitrary local paths and untrusted `media://` input blocked while allowing only already-saved managed inbound `application/pdf` offloads to reach the agent as media paths when sandbox staging is unavailable or incomplete.

## Reproduction Assessment

Yes from source inspection: a non-image PDF offload is saved as inbound media, then the sandboxed `chat.send` path can reject or fail in `prestageMediaPathOffloads` before dispatch. I did not run a live repro because this review was read-only.

## Solution Assessment

Yes, the requested PDF-only managed-inbound pass-through/fallback is the narrow maintainable fix if it reuses the existing inbound media reference checks and does not broaden arbitrary path access.

In `chat.send`, preserve current non-image offload behavior, but when every selected offloaded media-path ref is `application/pdf` and its path resolves to the managed inbound media store, pass the managed path/type through as `MediaPath`/`MediaPaths` and `MediaType`/`MediaTypes` even if sandbox staging is unavailable or incomplete. Keep the exception PDF-only and managed-inbound-only; do not accept arbitrary local paths, do not trust caller-supplied `media://` refs from untrusted input, do not grant locked-down agents `read` or `exec`, and do not broaden generic `stageSandboxMedia` behavior for other file types.
