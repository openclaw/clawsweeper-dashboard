## Summary

Keep open. The docs clarification is on the right topic, but the proposed setup text overstates `/whoami@<bot_username>` as a first-time group-ID discovery path even though Telegram native command auth still checks group allowlists first, and the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR updates Telegram channel documentation to distinguish Telegram user IDs from group chat IDs and adds an owner-only Telegram group setup example.

## Best Possible Solution

Keep the docs clarification, but qualify `/whoami` as usable only after native commands are enabled and the group is already allowed, while keeping logs, forwarded-ID bots, and Bot API `getUpdates` as first-time discovery paths.

## Reproduction Assessment

Yes, by source inspection: the docs place `/whoami@<bot_username>` in initial group setup, while the Telegram native-command path checks group allowlists before the `/whoami` handler can return identity details. I did not run a live Telegram setup.

## Solution Assessment

No: the docs surface is correct, but the current wording should not imply `/whoami` works before the group ID is configured under `channels.telegram.groups`. A narrower wording fix would make the proposed change maintainable.
