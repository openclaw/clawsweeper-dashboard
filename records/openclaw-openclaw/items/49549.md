## Summary

Keep open. Current main still has the WhatsApp outbound/inbound allowlist coupling, but this PR is not mergeable as-is because the latest head has concrete correctness gaps around direct allowTo adapter inputs, directory-backed target resolution, generated config metadata, and it also overlaps with the narrower open #48387 approach.

## What This Changes

The PR adds optional WhatsApp allowTo config/types/schema/docs and threads it into outbound target validation, action authorization, and implicit cron delivery so outbound recipients can differ from inbound allowFrom.

## Best Possible Solution

Decide the intended WhatsApp authorization model, then land one rebased implementation that consistently covers resolver inputs, channel outbound adapters, actions, cron/implicit delivery, directory-backed name lookup, config schema/types/metadata, docs, and focused regression tests.

## Reproduction Assessment

Yes. Current main can be checked statically: configure allowFrom to only the owner and attempt a WhatsApp DM target outside that list; the resolver still gates direct-message sends on allowFrom only. The PR-specific failures are also reproducible from the latest review context, which shows the added allowTo resolveTarget cases failing on the PR head.

## Solution Assessment

No, not as-is. The allowTo direction may be maintainable if maintainers want a new outbound allowlist, but the current patch misses direct adapter inputs, directory-backed target lookup, and generated metadata, and it needs a rebase onto the current WhatsApp plugin layout.

  - body: The adapter type now allows callers/tests to pass allowTo directly, but this implementation derives allowTo only from cfg/account.   - confidence: 0.9
