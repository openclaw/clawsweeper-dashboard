## Summary

Keep open: the request is a real WhatsApp feature gap, but this PR is not mergeable as-is because it targets stale outbound surfaces, still misses allowTo-backed directory resolution, and lacks real WhatsApp behavior proof.

## What This Changes

The branch adds an optional WhatsApp `allowTo` config field, threads it through outbound/action/cron target validation, and updates tests and docs.

## Best Possible Solution

Implement `allowTo` in the current WhatsApp plugin config/schema/resolved-account and extension-local outbound, action, cron, and directory paths, with focused regression tests, docs, and redacted live WhatsApp proof.

## Reproduction Assessment

Yes, source-level: current main's WhatsApp resolver rejects direct non-group targets outside `allowFrom`, and the PR changes that by adding a separate outbound list. I did not run a live WhatsApp session in this read-only review.

## Solution Assessment

No, not as-is. The opt-in `allowTo` direction is plausible, but the patch must be rebased into the current extension-local outbound adapters, cover directory lookup, and settle empty-list semantics before merge.
