## Summary

Keep open: current main still gates direct WhatsApp outbound targets through `allowFrom`, so the `allowTo` feature remains useful, but this PR should not merge as-is because it is stale against the current plugin-local WhatsApp layout, misses allowTo-only recipients in directory/name resolution, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a WhatsApp `allowTo` config field and threads it through outbound target validation, action authorization, cron implicit target fallback, tests, and docs.

## Best Possible Solution

Implement the opt-in outbound allowlist in the current WhatsApp plugin/config surfaces, keep inbound `allowFrom`/`dmPolicy` separate, include directory/cron/action coverage, and require redacted live WhatsApp proof before merge.

## Reproduction Assessment

Yes, at source level: current main's WhatsApp resolver rejects a direct target outside `allowFrom`, and the PR adds a separate outbound list to change that behavior. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

No, not as-is. The `allowTo` direction is narrower than an explicit-mode bypass, but this branch needs to be rebased onto current plugin-local paths, include allowTo recipients in directory lookups, and add real behavior proof.
