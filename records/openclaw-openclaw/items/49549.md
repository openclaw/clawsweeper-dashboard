## Summary

Keep open: the outbound-only WhatsApp use case is real and still not implemented on current main, but this PR is not ready as-is because it needs a current-main rebase, generated config metadata, one directory-path gap fixed, and real WhatsApp behavior proof.

## What This Changes

Adds `channels.whatsapp.allowTo` and per-account `allowTo` so WhatsApp outbound targets can be gated separately from inbound `allowFrom`, with resolver, cron delivery, action auth, docs, and tests.

## Best Possible Solution

Rebase onto current main and implement `allowTo` across the plugin-owned WhatsApp resolver, account config, action auth, cron delivery, directory/config metadata, docs, and focused tests, then add redacted live WhatsApp proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main passes only `allowFrom` into the WhatsApp target resolver, which blocks a valid direct target outside that list. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

No, not as-is. The `allowTo` direction is the safer solution than an explicit-mode bypass, but the branch needs current-main adaptation, generated metadata, directory behavior, and real proof before merge.
