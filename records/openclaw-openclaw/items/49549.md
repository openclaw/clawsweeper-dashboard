Keep this PR open, but it is not merge-ready: the feature is still useful and not implemented on current main, while the branch is stale against the current WhatsApp outbound surfaces, misses directory lookup, and lacks real WhatsApp behavior proof.

## What This Changes

The branch adds an optional WhatsApp `allowTo` config field, threads it through outbound/action/cron target validation, and updates tests and docs.

## Best Possible Solution

Implement `allowTo` in the current WhatsApp plugin config/schema/resolved account, registered outbound adapters, actions, cron delivery, and directory resolution, with focused tests, docs, and redacted live WhatsApp proof.

## Reproduction Assessment

Yes, source-level: current main's WhatsApp resolver builds the direct-message outbound allowlist from `allowFrom` only, so a direct target outside that list is rejected. I did not run a live WhatsApp session in this read-only review.

## Solution Assessment

No, not as-is. The `allowTo` direction is plausible, but the patch must be rebased onto current WhatsApp outbound adapters, cover directory lookup, and settle authorization semantics before merge.
