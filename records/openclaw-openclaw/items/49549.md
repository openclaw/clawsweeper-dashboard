## Summary

Keep open. Current main still lacks outbound-only WhatsApp gating, but this PR is not mergeable as-is because it targets stale outbound-contract/layout pieces and misses consistent `allowTo` coverage for direct adapter inputs, directory lookup, generated config metadata, and changelog follow-through; the product direction also overlaps the narrower open #48387 approach.

## What This Changes

The branch adds optional WhatsApp `allowTo` config/types/schema/docs and threads it into outbound target validation, action authorization, and implicit cron delivery so outbound recipients can differ from inbound `allowFrom`.

## Best Possible Solution

Decide the intended WhatsApp outbound authorization model, then land one rebased implementation covering the active WhatsApp plugin adapters, config schema/types/metadata, docs/config reference/changelog, directory-backed target lookup, cron/action/send paths, and focused regression tests.

## Reproduction Assessment

Yes. Current main can be checked statically by configuring `allowFrom` to only the owner and attempting a WhatsApp DM target outside that list: the resolver still gates direct-message sends on `allowFrom` only. The PR-specific blockers are also reproducible from the diff by direct `resolveTarget` calls with `allowTo` but no `cfg`, and by directory lookup for an allowTo-only contact.

## Solution Assessment

No, not as-is. `allowTo` may be a maintainable direction if maintainers want a new outbound allowlist, but this patch needs a rebase and consistent coverage of the active outbound adapters, directory lookup, generated metadata, and public docs before it is the narrowest safe solution.
