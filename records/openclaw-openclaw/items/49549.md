## Summary

Keep open: current main does not implement `allowTo`, but this PR is not ready as-is because it needs maintainer agreement on the new WhatsApp outbound authorization model plus a rebase across the active outbound adapter, directory, generated config metadata, docs, and changelog surfaces.

## What This Changes

The PR adds a WhatsApp `allowTo` config field and threads it into outbound target validation, action authorization, cron delivery, tests, and docs so outbound recipients can differ from inbound `allowFrom`.

## Best Possible Solution

Decide the intended WhatsApp outbound authorization model, then land one rebased implementation covering the active WhatsApp adapters, config schema/types/metadata, directory-backed target lookup, cron/action/send paths, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the resolver and active adapters still use `allowFrom` only, and the PR-specific blockers are visible from the diff and review discussion without needing a live WhatsApp device.

## Solution Assessment

No, not as-is. `allowTo` may be a maintainable solution if maintainers want that model, but this patch must be rebased and made consistent across the active adapter contract, directory lookup, generated config metadata, docs, and changelog before merge.
