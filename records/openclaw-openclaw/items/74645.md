## Summary

Keep this PR open. Current main still falls non-local Talk providers such as OpenAI back to system voice, while the PR’s small macOS runtime change routes those providers through the existing Gateway `talk.speak` path and adds focused planning coverage.

## What This Changes

The PR modifies macOS Talk playback planning to route providers other than ElevenLabs, MLX, and system through Gateway `talk.speak`, carries remote provider voice IDs, and updates the Talk runtime speech-plan test for OpenAI/custom providers.

## Best Possible Solution

Land this PR or an equivalent narrow macOS runtime change that uses Gateway `talk.speak` as the primary path for remote/custom Talk providers, keeps explicit `system` and `mlx` local behavior unchanged, and tracks provider-selection UI separately if maintainers still want that scope.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: on current main, `TalkModeRuntime.playbackPlan(provider: "openai", apiKey: nil, voiceId: "onyx")` reaches the default branch and returns `.systemVoiceOnly`, so no `talk.speak` call is attempted.

## Solution Assessment

Yes for the routing bug. The PR reuses the existing macOS `playGatewayTalkSpeak` helper and Gateway RPC contract instead of adding a new API, while preserving ElevenLabs direct playback, MLX, and system-voice behavior.
