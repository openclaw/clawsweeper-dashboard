## What This Changes

The issue requests suppressing or demoting the Gateway `auth mode=none` startup log for loopback-only deployments.

## Best Possible Solution

Review and land one exposure-aware no-auth startup warning predicate, likely via https://github.com/openclaw/openclaw/pull/91346, while preserving visible warnings for exposed configurations and avoiding a new config knob.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a loopback config with `gateway.auth.mode="none"` reaches the unconditional warning branch before bind/exposure filtering. I did not run the Gateway because this review is read-only.

## Solution Assessment

Yes, the best current route is an exposure-aware predicate without adding `gateway.quietNoAuthOnLoopback`. Blanket loopback suppression would be too broad because some loopback deployments are intentionally exposed through proxies or HTTP surfaces.
