## Summary

Keep open: current main still emits the no-auth Gateway startup warning unconditionally for `auth.mode="none"`, and the latest release has the same behavior. The request is valid source-level UX/security-log cleanup, but blanket suppression needs maintainer judgment because current audit logic has mixed loopback semantics rather than treating every loopback no-auth deployment as harmless.

## What This Changes

Request to suppress or demote the Gateway `auth mode=none` startup log when the effective Gateway bind is loopback-only.

## Best Possible Solution

Make the runtime log exposure-aware without adding a new config knob: quiet strictly local no-auth loopback startup, but keep a visible warning when proxy, tailnet, non-loopback, or HTTP exposure makes unauthenticated access meaningful.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `runGatewayCommand` logs the warning for every resolved `auth.mode="none"` before checking the bind host. I did not run the Gateway because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the security-log policy. A no-new-config, exposure-aware demotion is safer than blanket loopback suppression or adding `gateway.quietNoAuthOnLoopback`.
