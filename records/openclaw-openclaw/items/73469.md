## Summary

Keep open. Current main still sends Feishu streaming-card replies as an initial card reference and applies final mention text later through Card Kit updates, so there is no implemented path that can reliably trigger Feishu push notifications for agent-authored `<at>` tags. The related open syntax-normalization issue #60103 is adjacent, but it does not cover the streaming Card Kit notification boundary.

## What This Changes

Feishu streaming-card replies should trigger push notifications when an agent-authored `<at user_id="...">...</at>` mention is included in the delivered response text.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves mention notification semantics. The end state should cover raw `<at user_id>` parsing, existing `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and documented fallback behavior when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return a `<at user_id="...">...</at>` tag, and compare the current code path: the first Feishu message is only a card reference while the mention text is applied later by Card Kit updates.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` does not look sufficient because `close()` only updates Card Kit content/settings; the maintainable fix likely needs a Feishu-owned strategy around when to stream, fall back, or send an additional notification-bearing message.
