## Summary

Keep open. Current main still sends Feishu streaming-card replies by creating a Card Kit card reference first and then applying the final mention-bearing text through Card Kit element updates; there is no notification-bearing final send path for agent-authored `<at user_id="...">...</at>` text. The related open #60103 covers syntax normalization, while newer open PRs #74808, #75100, and #71396 may affect the design but are not merged and do not explicitly close the notification gap.

## What This Changes

The issue requests Feishu streaming-card replies to trigger push notifications when agent-authored text contains `<at user_id="...">...</at>` tags.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose one tested outbound strategy that preserves explicit mention notification semantics. The end state should cover raw `<at user_id>` parsing, structured `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and documented fallback behavior when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return `<at user_id="...">...</at>`, and current main will send the initial message as a card reference while applying the mention text later through Card Kit updates; live Feishu verification is still needed for the push-notification side effect.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` alone does not look sufficient because the current close path only updates Card Kit content/settings; a safer fix likely needs a Feishu-owned decision among fallback, segment delivery, or an additional notification-bearing message after API behavior is verified.
