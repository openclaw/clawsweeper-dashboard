## Summary

Keep open. Current main still routes Feishu streaming-card replies through an initial Card Kit card-reference message followed by Card Kit content/settings updates, so agent-authored `<at user_id="...">...</at>` text is not delivered through a normal final message path that would clearly carry mention notification semantics. The related #60103 is relevant for mention syntax normalization, but this remains a separate streaming-card notification-boundary bug.

## What This Changes

Request Feishu streaming-card replies to trigger push notifications when agent-authored response text includes `<at user_id="...">...</at>` mentions.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose an API-verified outbound strategy for mention-bearing streaming replies. The desired end state is either a proven Feishu pattern that preserves streaming and sends mention notifications, or a documented Feishu fallback that disables or bypasses streaming for mention-bearing replies without creating duplicate cards or breaking topic replies.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `"auto"`, have the agent return raw `<at user_id="...">...</at>` text, and current main will send an initial card reference before applying the final mention text through Card Kit updates; live push-notification confirmation is still needed for acceptance.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` alone does not look sufficient because the current close path updates Card Kit content/settings rather than sending a fresh notification-bearing message; a safer fix needs a Feishu-verified notification pattern or an explicit fallback policy.
