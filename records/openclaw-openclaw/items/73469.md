## Summary

Current main still appears to have the reported Feishu Card Kit notification gap: streaming replies create an initial card-reference message and then put final mention text into Card Kit content updates, which matches the issue's first-notification-only boundary rather than a notification-bearing message path.

## What This Changes

The issue requests that Feishu streaming-card replies notify users when the agent includes raw `<at user_id="...">...</at>` mentions in the final text.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and implement an API-verified strategy for mention-bearing streaming replies, such as a documented fallback to non-streaming or a controlled final notification-bearing message when Card Kit cannot notify on updates.

## Reproduction Assessment

Yes. The source path is clear: enable Feishu streaming in `card` or markdown-triggered `auto`, have the agent emit raw `<at user_id="...">...</at>` text, and current main sends the final text through Card Kit updates after the notification-time card-reference message; live Feishu notification confirmation remains the acceptance check.

## Solution Assessment

Unclear. The requested `close()` mentions parameter does not appear to solve the current implementation by itself because `close()` updates Card Kit content/settings instead of sending a new message that Feishu can use for mention notifications.
