## Summary

Keep open. Current main still sends Feishu streaming-card replies as an initial Card Kit card-reference message, then applies the final text through Card Kit content/settings updates; agent-authored raw `<at user_id="...">...</at>` text is not routed through a notification-bearing final message path. The related #60103 covers mention syntax normalization, but this issue remains a distinct Card Kit notification-boundary problem.

## What This Changes

The issue requests that Feishu streaming-card replies notify users when agent-authored raw `<at user_id="...">...</at>` mention tags appear in the final text.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose an API-verified outbound strategy for mention-bearing streaming replies. The end state should preserve streaming where Feishu can notify correctly, and otherwise use a documented fallback such as disabling streaming for mention-bearing replies or sending a controlled final notification-bearing message without reopening duplicate cards.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return raw `<at user_id="...">...</at>` text, and the code path sends only a card reference at message creation while applying the mention text later via Card Kit updates; live push-notification confirmation is still needed for acceptance.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` alone does not look sufficient because the current close path updates Card Kit content/settings instead of sending a fresh notification-bearing message; a safer solution needs a Feishu-owned fallback or verified Card Kit notification pattern.
