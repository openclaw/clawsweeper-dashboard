## Summary

Keep open. Current main still sends Feishu streaming replies as an initial Card Kit card reference, then applies final mention-bearing text through Card Kit element/settings updates; the non-streaming paths are the ones that attach mention markup at message-send time. Related #60103 tracks syntax normalization across Feishu formats, but it does not cover this Card Kit notification boundary.

## What This Changes

Ensure Feishu streaming-card replies that include agent-authored `<at user_id="...">...</at>` tags send push notifications to mentioned users like non-streaming Feishu replies do.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a verified outbound strategy that preserves mention notifications without surprising duplicate replies. The end state should cover raw `<at user_id>` parsing, `mentionTargets`, card/text syntax conversion, topic replies, streaming final dedupe, and documented fallback behavior if Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return a `<at user_id="...">...</at>` tag, then compare the current code path: the first Feishu message contains only a card reference while the mention text is applied later through Card Kit updates.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` alone does not look sufficient because `close()` only updates Card Kit content/settings; the safer solution needs a Feishu-owned decision between a verified API pattern, a notification-bearing fallback send, or explicit documented limitations.
