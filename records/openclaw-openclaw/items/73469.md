## Summary

Keep open. Current main still has the Feishu streaming-card notification boundary described by the reporter: the notification-time message is a Card Kit card reference, while the final mention-bearing text is applied later through Card Kit updates. The related #60103 issue covers syntax normalization, but this issue preserves a distinct push-notification problem for streaming cards.

## What This Changes

The issue asks Feishu streaming-card replies to trigger push notifications for users mentioned by raw agent-written `<at user_id="...">...</at>` tags, matching non-streaming Feishu delivery.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose an API-verified outbound strategy for mention-bearing streaming replies. The end state should preserve normal streaming where possible, handle raw `<at user_id>` tags and existing structured `mentionTargets`, coordinate with #60103 for syntax conversion, and document any fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: enable Feishu streaming with `renderMode: "card"` or markdown-triggered `auto`, have the agent return raw `<at user_id="...">...</at>` text, and observe that the initial Feishu message contains only a card reference while final text is applied through Card Kit updates. I did not run a live Feishu group check in this read-only review.

## Solution Assessment

Unclear. Adding a mentions parameter to `FeishuStreamingSession.close()` alone is not the best fix because `close()` updates Card Kit content and settings rather than creating a notification-bearing message; the safer path is a Feishu-owned fallback, companion send, or supported Card Kit/message API pattern after verification.
