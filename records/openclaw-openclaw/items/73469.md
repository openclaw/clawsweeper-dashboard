## Summary

Keep open. Current main still sends Feishu streaming replies as a Card Kit card-reference message followed by card content/settings updates, so the reported raw `<at user_id="...">...</at>` path is not routed through a notification-bearing final message; the related #60103 does not resolve the notification timing question.

## What This Changes

The issue asks Feishu streaming card replies to trigger push notifications when the agent emits raw `<at user_id="...">...</at>` mention tags.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and implement an API-verified notification strategy for mention-bearing streaming replies, with a documented fallback if Card Kit updates cannot notify.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled in `card` mode or markdown-triggered `auto`, have the agent emit `<at user_id="OPEN_ID">name</at>`, and current main places the final content into Card Kit updates after the initial card-reference message; a live group notification check is the acceptance proof.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` alone is unlikely to solve the reported behavior because `close()` currently mutates an existing Card Kit card rather than sending a new notification-bearing message.
