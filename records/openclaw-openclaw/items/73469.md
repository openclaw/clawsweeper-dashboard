## Summary

Keep open. Current main still routes Feishu card-mode streaming replies through an initial Card Kit card-reference message and later Card Kit content/settings updates, while raw agent-authored `<at user_id="...">...</at>` text is never sent through a notification-bearing final message path. The related #60103 issue covers mention syntax normalization, but this report adds a distinct Card Kit notification-boundary problem.

## What This Changes

Feishu streaming-card replies that include agent-authored `<at user_id="...">` tags should trigger the same push notifications as non-streaming mention messages.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose an API-verified outbound strategy for mention-bearing streaming replies. The end state should handle raw agent `<at user_id>` tags, existing structured `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and documented fallback behavior when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return a raw `<at user_id="...">...</at>` tag, and compare the code path: the notification-time message contains only a card reference while the mention text is applied later through Card Kit updates.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` alone does not look sufficient because the current close path updates Card Kit content/settings rather than sending a notification-bearing message; a safer solution needs a Feishu-owned fallback, additional message, or verified API pattern.
