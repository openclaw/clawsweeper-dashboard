## Summary

Keep open. Current main still sends Feishu streaming replies as an initial Card Kit card reference, then applies the final mention-bearing text through Card Kit element updates; the code does not show a notification-bearing final send path for agent-authored `<at user_id="...">` text, and adjacent #60103 covers syntax normalization rather than the Card Kit notification boundary.

## What This Changes

Ensure Feishu streaming-card replies that include agent-authored `<at user_id="...">...</at>` tags trigger Feishu @mention push notifications instead of only rendering the mention in the card.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves mention notification semantics. The end state should cover raw `<at user_id>` parsing, existing `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and documented fallback behavior when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return a `<at user_id="...">...</at>` tag, and compare current code: the initial Feishu message is only a card reference while the mention text is applied later through Card Kit updates.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` alone does not look sufficient because the current close path only updates Card Kit content/settings; the safer fix likely needs a Feishu-owned decision about fallback, an additional message, or a verified Feishu API pattern.
