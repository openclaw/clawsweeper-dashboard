## Summary

Close as duplicate/superseded: current `main` still has synchronous auto-TTS behavior, but the non-blocking Telegram voice pipeline work is already tracked in https://github.com/openclaw/openclaw/issues/48179 and the text-first/separate voice delivery-mode work is already tracked in https://github.com/openclaw/openclaw/issues/42539.

## What This Changes

Request non-blocking or asynchronous TTS delivery so Telegram text replies and inbound message handling are not blocked by audio generation.

## Best Possible Solution

Keep the remaining design in the two canonical issues and coordinate one backward-compatible TTS delivery policy that covers async/background generation, ordering, failure fallback, and Telegram text-first voice-note delivery.
