## Summary

Keep open. Current main still awaits automatic TTS synthesis before final/tool/block reply delivery, and Telegram ordinary updates remain per-chat/per-topic serialized, so the reported inbound blocking behavior is still plausible and not superseded by the related open voice/TTS UX issues.

## What This Changes

Request non-blocking automatic TTS delivery so text replies and new inbound messages do not wait on synchronous speech synthesis, especially for Telegram with slow local TTS endpoints.

## Best Possible Solution

Add a shared, config-backed automatic TTS delivery policy that preserves synchronous delivery as the compatibility default while allowing a bounded text-first/background mode with channel-safe follow-up media delivery and docs/schema coverage. Coordinate the Telegram split text/voice UX with #42539 and the broader voice pipeline work in #48179.

## Reproduction Assessment

Yes. Enable `messages.tts.auto` with a slow local/OpenAI-compatible TTS provider on Telegram, trigger a multi-chunk or streamed-final reply, then send another ordinary same-topic message; current dispatch awaits TTS and the Telegram lane keeps the second message pending.

## Solution Assessment

Unclear. The non-blocking direction is valid, but the best maintainable fix should be a shared delivery policy and bounded background media path rather than unbounded parallel TTS calls or a Telegram-only workaround.
