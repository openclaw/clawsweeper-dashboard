## Summary

Keep open. Current main still awaits automatic TTS synthesis inline before final/tool/block reply delivery, the TTS config schema has no async/background delivery policy, and Telegram ordinary updates remain serialized by chat/topic, so the reported blocking behavior remains valid.

## What This Changes

Request non-blocking automatic TTS delivery so text replies and subsequent inbound messages are not held while reply audio is synthesized.

## Best Possible Solution

Add a shared, config-backed automatic TTS delivery policy that preserves synchronous delivery as the compatibility default while allowing a bounded text-first/background mode with channel-safe follow-up media delivery and docs/schema coverage.

## Reproduction Assessment

Yes. Enable messages.tts.auto with a slow local/OpenAI-compatible TTS provider on Telegram, trigger a multi-part or streamed-block reply, then send another ordinary same-topic message; the inspected code awaits TTS and Telegram keeps the second message on the same sequential lane.

## Solution Assessment

Unclear. The non-blocking direction is sound, but the maintainable fix should be a shared delivery policy and bounded background media path rather than unbounded parallel TTS calls or a Telegram-only workaround.
