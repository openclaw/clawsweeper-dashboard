## Summary

Keep open: the PR fixes the earlier non-404 fallback problem, but the current head still changes documented Mattermost channel-id routing so a missing explicit id can fall through to a name lookup and post to a different channel; it also still lacks redacted real Mattermost proof.

## What This Changes

The PR adds Mattermost channel-id validation before send, falls back to channel-name lookup on 404, changes non-404 opaque target handling, and adds mocked tests for the new branches.

## Best Possible Solution

Preserve deterministic `channel:<id>` and bare-id channel routing unless the target was explicitly parsed as a channel name, add tests for both explicit-id 404 and ID-shaped-name cases, and require redacted live Mattermost proof before merge.

## Reproduction Assessment

No live Mattermost reproduction was run in this read-only review. The PR defect is source-reproducible: docs define `channel:<id>` as deterministic, and the head diff falls back from any channel-id 404 to channel-name lookup.

## Solution Assessment

No. Validating ID-shaped targets is a plausible repair, but the current implementation is too broad because it cannot distinguish explicit channel ids from intended name targets.
