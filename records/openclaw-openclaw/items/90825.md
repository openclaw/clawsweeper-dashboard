## Label Justifications

- `P2`: The PR addresses real silent cost under-reporting but introduces a bounded compatibility problem for zero-priced and custom models.
- `merge-risk: 🚨 compatibility`: Merging changes observable behavior for existing plugin SDK and local/custom model callers whose all-zero pricing is intentional or expected.

## Summary

The user problem is valid, but this patch treats every all-zero rate table as unknown pricing; bundled local/custom models and public plugin SDK callers legitimately use that shape, so the new warning can misdiagnose normal usage after upgrade.

## What This Changes

Adds a once-per-provider/model warning when `calculateCost` processes nonzero token usage with an all-zero model cost table, plus focused unit coverage.

## Best Possible Solution

Represent pricing knowledge explicitly at the model/catalog boundary—distinguishing known-free, known-priced, and unavailable pricing—then warn only for the unavailable state while preserving current behavior for intentional zero-cost and plugin-defined models.

## Reproduction Assessment

Yes from source and supplied runtime proof: current main calculates an all-zero cost silently, while the patched helper logs for the same model and usage. The broader false-positive case is also directly established by supported model definitions that use all-zero rates.

## Solution Assessment

No. Warning at the shared calculation boundary is reasonable, but inferring “unknown” solely from four zero values is not the narrowest maintainable solution because the current model contract also uses zero for legitimate local, proxy, and fallback pricing states.
