## Summary

Keep open: current main and the latest release still only recognize Claude CLI Keychain or `.claude/.credentials.json` credentials for this pre-spawn auth path, while Claude Code documents `apiKeyHelper` as a credential script source. This is a concrete auth-provider bug with a narrow source-reproducible path, not stale cleanup.

## What This Changes

The issue reports that Claude CLI auth backed by `apiKeyHelper` in Claude settings is rejected by OpenClaw's pre-spawn auth gate before the CLI can run.

## Best Possible Solution

Keep this open and add helper-aware Claude CLI auth availability in the Claude CLI credential/auth seam, preferably by detecting configured `apiKeyHelper` as CLI-owned auth evidence without exposing or executing the helper output in direct Anthropic API paths.

## Reproduction Assessment

Yes, from source: on Linux with no `.claude/.credentials.json`, the noninteractive Claude CLI resolver returns null, synthetic auth returns undefined, and provider auth falls through to `missing-provider-auth`. I did not run a live current-main reproduction with real Claude CLI credentials.

## Solution Assessment

Yes, the narrow maintainable fix is in the Claude CLI credential/auth availability path, with regression tests for helper-backed settings and no broad bypass of provider auth. A safer implementation should let Claude CLI execute the helper during the actual run rather than making OpenClaw run arbitrary helper commands during the gate.
