## Summary

Keep open: current main and the latest release still only recognize Claude CLI Keychain or `.credentials.json` credentials for synthetic `claude-cli` auth, while Claude Code supports `apiKeyHelper` as a settings-based credential source. The reported pre-spawn `missing-provider-auth` path remains source-reproducible and belongs in the Anthropic Claude CLI auth seam.

## What This Changes

Reported bug: OpenClaw rejects Claude CLI sessions authenticated through Claude Code `apiKeyHelper` settings before spawning the CLI.

## Best Possible Solution

Implement a narrow Anthropic-owned Claude CLI auth fix that recognizes `apiKeyHelper` as CLI-managed auth, with regression coverage and live proof when feasible, without adding a new OpenClaw config option.

## Reproduction Assessment

Yes. Source inspection shows that on Linux with no `.claude/.credentials.json`, the current resolver returns null because it does not read Claude settings, then synthetic auth returns undefined and the provider auth resolver can throw `missing-provider-auth`; I did not run a live proxy-backed Claude CLI turn.

## Solution Assessment
