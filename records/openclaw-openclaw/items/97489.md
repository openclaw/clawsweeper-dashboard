## Label Justifications

- `P2`: This is a real provider-auth compatibility bug for a bounded Claude CLI auth mode, but it is not a crash loop or broad outage.
- `impact:auth-provider`: The failure is an auth-provider resolution gate rejecting a valid Claude CLI authentication source before session spawn.
- `maturity:stable`: The affected CLI auth/setup surface maps to `cli-install-update-onboard-doctor`, which the maturity helper reports as M4 Stable.

## Summary

Keep open: current main and v2026.6.10 still only recognize Claude CLI Keychain or `.credentials.json` credentials for synthetic `claude-cli` auth, while Claude Code documents `apiKeyHelper` as a settings-based credential helper. The reported `missing-provider-auth` path remains a concrete auth-provider bug rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to treat Claude Code `apiKeyHelper` in `~/.claude/settings.json` as valid `claude-cli` auth instead of requiring Keychain or `.credentials.json`.

## Best Possible Solution

Keep this issue open and implement a narrow Anthropic/Claude CLI auth fix that recognizes `apiKeyHelper` as CLI-managed auth, with regression coverage and live proof when feasible.

## Reproduction Assessment

Yes, source-reproducible: on Linux with no `.claude/.credentials.json`, the current resolver returns null because it only checks Keychain on Darwin and `.claude/.credentials.json`, after which synthetic auth returns undefined and the provider auth resolver can throw `missing-provider-auth`. I did not run a live proxy-backed Claude CLI turn.

## Solution Assessment

Yes, the best path is a narrow fix in the Anthropic-owned Claude CLI credential/synthetic-auth seam rather than a new OpenClaw config option or a generic provider fallback. The repair should treat `apiKeyHelper` as CLI-managed auth without executing it in the auth gate unless maintainers explicitly choose that behavior.
