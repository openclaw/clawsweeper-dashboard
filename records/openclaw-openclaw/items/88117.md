Full review comments:

- **[P1] Import LLM package contracts through declared exports:** `packages/llm-runtime/src/api-registry.ts:9`
  - body: `@openclaw/llm-runtime` declares dist exports and depends on `@openclaw/llm-core`, but this source import points at `../../llm-core/src/index.js`. Declaration emit preserves module specifiers, so built consumers of `/api-registry` and `/stream` can resolve to sibling source paths outside the package contract; import through `@openclaw/llm-core` exported subpaths instead.
  - confidence: 0.86
- **[P1] Keep the public SDK facade on packaged paths:** `src/plugin-sdk/llm.ts:47-51`
  - body: `openclaw/plugin-sdk/llm` is a root package export, so re-exporting from `../../packages/llm-core/src/...` leaks monorepo source paths into the installed SDK surface. External plugins importing the SDK from a packaged OpenClaw install can fail without `packages/llm-core/src`; route this through existing `src/llm/**` shims or declared package exports.
  - confidence: 0.88
- **[P1] Restore built-in provider registration before registry reads:** `src/llm/stream.ts:3`
  - body: Built-in providers now register only when `src/llm/stream.ts` is imported. Direct registry readers such as `prepareModelForSimpleCompletion` can call `getApiProvider()` first, making built-in APIs look custom/provider-owned and bypassing the intended built-in provider/auth routing; keep registration in the registry lifecycle or add import-order coverage for those consumers.
  - confidence: 0.78
