## Summary

Keep open: the reported session-tool alias problem is real, but this branch is not merge-ready because it is based on an older session-access shape and its alias set broadens the self/tree visibility boundary.

## What This Changes

The PR adds a current-session alias set to session visibility checks, wires it through `sessions_list`, `sessions_history`, and `sessions_send`, and adds regression tests.

## Best Possible Solution

Adapt the fix on current main in the shared session-visibility layer so only aliases that actually resolve to the requester/current client pass self/tree checks, with regression coverage for allowed current aliases and denied `main` access from a non-main requester.

## Reproduction Assessment

Yes at source level: current main resolves direct key actions through `createSessionVisibilityGuard`, which only recognizes the exact requester key or `current`, so alias targets like `main` remain outside self/tree for non-main requesters. I did not run tests because this review is read-only.

## Solution Assessment

No. The proposed direction is plausible, but the patch is too broad and stale; the maintainable fix should live in the current shared visibility layer and distinguish true current aliases from the agent main session.
