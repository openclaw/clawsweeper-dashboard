## Summary

Keep open: current main still exposes only active `chat.history` for the current session entry, and the requested logical history across reset/archive segments remains an additive Gateway protocol/product feature. Related open PRs cover partial or proposed archive access, but none is merged on main or terminal enough to close this issue.

## What This Changes

Add an optional logical-history mode to `chat.history` so a stable session key can return current and prior rolled transcript segments with lightweight segment metadata.

## Best Possible Solution

Review and land a coordinated additive archive-history design, likely via the existing `includeArchived`/logical-history PR path, with bounded reads, stable segment metadata, docs, tests, and explicit approval for reset-archive visibility semantics.

## Reproduction Assessment

Yes, at source level. Current main rejects a logical-history parameter in `ChatHistoryParamsSchema`, then `chat.history` loads the current session entry and reads only that active transcript while reset rollover archives the old transcript into a separate `.reset.<timestamp>` file.

## Solution Assessment

Unclear as specified. A logical or explicit archive-history mode is the right general surface, but the best fix must preserve bounded history reads, define segment metadata, and settle reset-archive visibility semantics before merge.
