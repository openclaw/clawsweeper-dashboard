## Label Justifications

- `P2`: The recurring gap fragments visible history for long-lived chats but does not immediately delete the underlying transcript data.
- `impact:session-state`: The issue concerns continuity and recovery across related rolled session transcripts.

## Summary

Current main and v2026.6.5 still expose only the active session segment through `chat.history`; detailed field evidence confirms the continuity gap, and the related implementation PR remains open and conflicted, so this issue is still necessary for maintainer protocol and product review.

## What This Changes

Add an optional Gateway history mode that returns rolled transcript segments for one logical chat with segment metadata while preserving current-segment behavior.

## Best Possible Solution

Define one additive, bounded Gateway contract over a canonical logical-session chain, return ordered segment metadata and explicit unavailable/pruned states, keep current-segment behavior as the default, and have Control UI and agent tools consume that contract rather than scanning transcript files.

## Reproduction Assessment

Yes at source level: current `chat.history` resolves and reads only the active session ID, while the issue contains detailed real-use evidence of a stable logical key spanning multiple retained session IDs after rollover. I did not execute a new live rollover reproduction during this read-only review.

## Solution Assessment

The proposed Gateway-first boundary is the best direction because both local and remote clients need a canonical history source, but the exact chain identity, bounds, retention semantics, and segment representation require maintainer agreement before implementation.

## Risks / Open Questions

- The permanent contract must distinguish reset archives belonging to one session ID from `usageFamilySessionIds` spanning several session IDs; treating either as the whole logical chain could omit valid segments or combine unrelated history.
- Logical aggregation needs explicit message, byte, segment, and retention bounds so long-lived chats do not create oversized Gateway responses or Control UI performance regressions.
- Reset archives are retention-limited, so the API must represent unavailable or pruned segments rather than promise complete history indefinitely.
