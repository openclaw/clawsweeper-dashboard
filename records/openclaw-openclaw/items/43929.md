## Summary

Keep open: current main and the latest release still expose `chat.history` as a bounded current-session transcript read, while logical/archive history is only covered by active related PRs and still needs a maintainer API and reset-archive visibility decision.

## What This Changes

Add a logical-history mode to `chat.history` so WebChat and Control UI can show rolled transcript segments for the same long-lived logical session key.

## Best Possible Solution

Choose and land a backward-compatible Gateway protocol/tool contract for logical history that preserves bounded reads, keeps current behavior as the default, returns ordered reset-segment metadata, and coordinates WebChat with `sessions_history`.

## Reproduction Assessment

Yes, at source level. Current main rejects logical/archive parameters in the `chat.history` schema and the handler reads only the active session entry through `readRecentSessionMessagesAsync`; I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the protocol shape. A Gateway-first additive option is plausible, but the open archive-history PRs should be reviewed or revised rather than starting a competing implementation.
