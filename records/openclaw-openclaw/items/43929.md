## Summary

Keep open: current main still exposes only the active `chat.history` transcript for the resolved `sessionId`, while reset code archives prior transcripts and creates a fresh session file. Related open PRs cover narrower archive fallback or an `includeArchived` API direction, but none is merged and the requested logical segment chain remains a Gateway/API product decision.

## What This Changes

Add an optional logical-history mode to `chat.history` so long-lived chats can return the current rolled segment plus prior segments and lightweight segment metadata.

## Best Possible Solution

Define one additive Gateway history contract for rolled-session continuity, then implement it in the transcript/session read layer and update WebChat/Control UI docs and tests without changing reset behavior.

## Reproduction Assessment

Yes, for the current behavior from source inspection: `chat.history` accepts no logical-history selector and reads only the current resolved transcript path. I did not run a live Gateway reproduction because this read-only review cannot install missing tooling, but the source and docs agree on the active-branch boundary.

## Solution Assessment

Unclear. A logical mode is a plausible solution, but the best maintainable API should be chosen alongside the open `includeArchived` and archive-fallback PRs so OpenClaw does not grow overlapping history contracts.
