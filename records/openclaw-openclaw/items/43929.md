## Summary

Keep open: current main still exposes `chat.history` as a current-session transcript reader, and the reporter's continuity gap remains. The related #76134 PR is open and explicitly narrower than chained logical history.

## What This Changes

Add an optional logical-history mode to `chat.history` that returns current and rolled transcript segments for the same stable session key with lightweight segment metadata.

## Best Possible Solution

Design a Gateway protocol addition that keeps current behavior as the default, adds an explicit logical/archived segment mode with bounded reads and segment metadata, and coordinates WebChat/Control UI plus `sessions_history` behavior across the archive-discovery cluster.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence the capability is absent because `chat.history` has no `scope` parameter and reads only the current `entry.sessionId` transcript.

## Solution Assessment

Unclear. The proposed `scope: "logical"` direction is plausible, but maintainers should first settle the protocol shape, archive-retention semantics, and relationship to `sessions_history` and #76134.
