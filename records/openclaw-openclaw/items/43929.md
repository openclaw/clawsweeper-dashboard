## Summary

Keep open. Current main still exposes `chat.history` as a current active-session read, and the recent archive-indexing work only makes reset archives searchable elsewhere rather than returning logical current-plus-rolled history with segment metadata.

## What This Changes

Add an optional logical-history mode to `chat.history` so WebChat and Control UI can show one long-lived conversation across daily, idle, or manual session rollovers.

## Best Possible Solution

Add an additive Gateway protocol extension for logical history, backed by durable reset-lineage metadata and bounded ordered segment reads, then update WebChat/Control UI and docs to render continuity without direct transcript-file scanning.

## Reproduction Assessment

Yes. Source reproduction is high confidence: after a stable session key is reset to a new `sessionId`, `chat.history` reads only the current `entry.sessionId` transcript through `readRecentSessionMessagesAsync`; I did not run a live mutation test because this review was read-only.

## Solution Assessment

Yes at the boundary level: Gateway is the right layer because remote WebChat and Control UI clients should not scan transcript files. The concrete API fields and lineage storage need maintainer product/API judgment before a safe implementation.
