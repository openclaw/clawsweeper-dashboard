## Summary

Keep open: current main and the latest release still expose `chat.history` as a bounded current-session transcript read, while the logical/archive-history work remains in open related PRs and needs a maintainer API and archive-visibility decision.

## What This Changes

Request to add an optional logical-history mode to `chat.history` that can return rolled reset segments and segment metadata for a stable session key.

## Best Possible Solution

Choose and land a backward-compatible Gateway protocol contract for logical history that preserves bounded reads, keeps current behavior as the default, returns ordered reset-segment metadata, and coordinates WebChat plus `sessions_history` behavior with the existing related PRs.

## Reproduction Assessment

Yes, at source level. Current main rejects logical/archive parameters in the `chat.history` schema and the handler reads only the active session entry's transcript through `readRecentSessionMessagesAsync`; I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the protocol shape. A Gateway-first additive option is plausible, but the open archive-history PRs should be reviewed or revised rather than starting a competing implementation.
