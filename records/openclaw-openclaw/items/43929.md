## Summary

Keep open: current main still exposes `chat.history` as a bounded current-session transcript read, and the requested logical/archive history mode remains an unsettled Gateway protocol/tool product decision with active related PRs but no merged implementation.

## What This Changes

Request an optional logical-history mode for `chat.history` that chains rolled transcript segments for the same logical session key and returns lightweight segment metadata.

## Best Possible Solution

Choose and land a backward-compatible Gateway contract for logical history that preserves bounded reads, keeps current behavior as the default, returns ordered reset-segment metadata, documents the protocol/tool behavior, and coordinates WebChat plus `sessions_history`.

## Reproduction Assessment

Yes, at source level. Current main rejects logical/archive history parameters and the handler reads only the active session transcript through `readRecentSessionMessagesAsync`; I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive option is plausible, but the open archive-history PRs should be reviewed or revised rather than starting a competing implementation.
