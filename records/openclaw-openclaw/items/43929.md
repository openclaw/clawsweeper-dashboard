## Summary

Close as duplicate/superseded: current main still lacks a logical `chat.history` archive mode, but the same remaining Gateway history contract is already tracked by https://github.com/openclaw/openclaw/issues/45003 and active implementation work in https://github.com/openclaw/openclaw/pull/73883; https://github.com/openclaw/openclaw/pull/76134 is only a narrower missing-active fallback and explicitly leaves chained logical history out of scope.

## What This Changes

Request an optional logical-history mode on `chat.history` that returns current and prior rolled transcript segments for the same logical session key with segment metadata.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/45003 as the canonical tracker and land a bounded, explicit archive/logical-history Gateway contract with segment metadata, docs, and approved archive-visibility semantics.

## Reproduction Assessment

Yes, at source level: current main rejects any logical/archive selector on `chat.history`, reads the current resolved transcript through `readRecentSessionMessagesAsync`, and still creates `.jsonl.reset.*` archives on rollover. I did not run a live reset because this was a read-only review.

## Solution Assessment

Yes for cleanup: consolidating this under the canonical archive-recovery thread and active PR path is the narrowest maintainer outcome. The implementation itself should be a bounded explicit Gateway/tool archive-history mode, not UI-side file scanning.
