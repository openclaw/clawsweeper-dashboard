## Summary

Keep open: current main still has no logical-history selector, and `chat.history` reads only the active transcript for the current session entry. Related open PRs cover partial archive fallback or an explicit archive opt-in, but none is merged and the final archive visibility semantics need maintainer product/security judgment.

## What This Changes

Request an optional logical-history mode for `chat.history` that returns current and previous rolled transcript segments for the same stable session key with lightweight segment metadata.

## Best Possible Solution

Land a coordinated additive archive/logical-history design that preserves bounded reads, defines stable segment metadata, updates protocol/docs/tests/generated clients, and explicitly approves reset-archive visibility semantics.

## Reproduction Assessment

Yes, at source level. Current main rejects logical-history parameters in `ChatHistoryParamsSchema`, then `chat.history` reads only `entry.sessionId` through the active transcript reader while reset stores the previous transcript as a separate `.reset.<timestamp>` archive.

## Solution Assessment

Unclear as specified. A logical or explicit archive-history mode is the right direction, but the maintainable solution must preserve bounded reads, define segment metadata, and settle reset-archive visibility before merge.
