- `impact:session-state`: The issue is about transcript/session continuity across rolled session ids and user-visible session state drift in WebChat/Control UI.

## Summary

Keep open: current main and v2026.6.5 still expose `chat.history` as an active-session reader, while the requested logical-history API remains unmerged. The open implementation candidate at https://github.com/openclaw/openclaw/pull/73883 is still conflicting, and the adjacent archive fallback at https://github.com/openclaw/openclaw/pull/76134 explicitly does not implement chained logical history.

## What This Changes

Add a logical-history mode to `chat.history` so a stable chat/session key can return current and previous rolled transcript segments with lightweight segment metadata.

## Best Possible Solution

Define the Gateway-owned logical-history contract, then refresh or replace the conflicting implementation PR so `chat.history` and `sessions_history` expose explicit rolled-segment history without changing default current-session behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: create or inspect a session row with prior `usageFamilySessionIds`, then `chat.history` still validates no logical-history parameter and reads only the active `entry.sessionId` transcript. I did not run a live Gateway scenario in this read-only review.

## Solution Assessment

Unclear as an exact API shape, but Gateway/protocol is the right layer; a UI-only scan would not work reliably for remote Gateway clients. Maintainers still need to choose the canonical chain semantics and segment metadata before landing the implementation.
