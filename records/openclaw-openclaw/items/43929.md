## Summary

Keep open. Current main still implements `chat.history` as a current active-session read, while the issue asks for an additive logical-history mode that aggregates rolled transcript segments and exposes segment metadata.

## What This Changes

Add a logical-history mode to `chat.history` so long-lived chats can show current and previously rolled transcript segments for the same stable session key.

## Best Possible Solution

Add an additive Gateway protocol extension for logical history, backed by durable reset-lineage metadata and bounded ordered segment reads, then update WebChat/Control UI and docs to render continuity without direct transcript-file scanning.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: after reset assigns a stable key to a new `sessionId`, `chat.history` reads only that current `entry.sessionId` through a single active transcript resolver.

## Solution Assessment

Yes at the boundary level: Gateway is the right layer because remote WebChat and Control UI clients should not scan transcript files. The exact API fields, lineage storage, and retention semantics need maintainer product/API judgment.
