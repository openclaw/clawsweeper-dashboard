## Summary

Keep open. Current main still exposes `chat.history` as a current-session transcript read, while the request is an additive Gateway protocol feature for logical history across rolled transcript segments; related open fallback PRs do not implement aggregation or segment metadata.

## What This Changes

Request an optional `chat.history` logical-history mode that returns the current segment plus previous rolled transcript segments and lightweight segment metadata for the same stable session key.

## Best Possible Solution

Add a backwards-compatible Gateway protocol extension for logical history, backed by durable reset-lineage metadata and bounded ordered segment reads, then update WebChat/Control UI and docs to render continuity through the Gateway.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after reset moves a stable key to a new `sessionId`, `chat.history` reads only that current physical session transcript through a single active-path resolver.

## Solution Assessment

Yes at the architectural boundary: Gateway is the right layer because remote WebChat and Control UI clients should not scan transcript files. The exact API shape, lineage storage, limits, and metadata semantics need maintainer product/API judgment.
