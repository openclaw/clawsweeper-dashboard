## Label Justifications

- `impact:session-state`: The issue is about transcript and session continuity across rolled `sessionId` segments for one logical chat.

## Summary

Current main still lacks the requested logical-history mode for `chat.history`: the protocol accepts only active-session parameters, the gateway handler reads one current session entry, and reset archives are still maintained separately on disk. Related PRs are open implementation candidates, but none has merged, so this issue should stay open for maintainer product/API review.

## What This Changes

Request an additive `chat.history` mode that can return the current transcript plus prior rolled transcript segments for the same logical session key, with lightweight segment metadata for UI continuity.

## Best Possible Solution

Adopt one backward-compatible Gateway contract that keeps current history as the default, adds an explicit logical/archive mode with bounded reads and ordered segment metadata, and updates WebChat, `sessions_history`, docs, generated clients, and tests together.

## Reproduction Assessment

Yes, at source level. Current main rejects any logical/archive selector, `chat.history` and `sessions_history` read only the active transcript, and reset archive files are still created on disk.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive mode is plausible, but it should be coordinated with the related archive-history PRs instead of handled as a competing standalone repair.
