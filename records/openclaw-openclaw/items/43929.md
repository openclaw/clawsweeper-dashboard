## Label Justifications

- `P2`: This is a normal-priority session continuity improvement with recurring user impact but no crash, security bypass, or unusable runtime.
- `impact:session-state`: The issue is about transcript and chat continuity across rolled `sessionId` segments for one logical session key.

## Summary

Current main and the latest release still expose only the active session segment through `chat.history`; the requested logical/archive history mode remains unimplemented. Related implementation PRs are open but unmerged, so this should stay open for maintainer protocol/product review.

## What This Changes

Request an optional logical chat-history mode that returns the current segment plus prior rolled segments for the same stable session key, with lightweight segment metadata.

## Best Possible Solution

Adopt one additive Gateway contract that preserves current history by default, adds bounded logical/archive reads with ordered segment metadata, and updates WebChat, `sessions_history`, protocol docs/generated clients, and tests together.

## Reproduction Assessment

Yes, at source level. Current main rejects any logical/archive selector and reads only the active transcript while reset archives are still created on disk.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive mode is plausible, but it should be coordinated with the related archive-history PRs instead of handled as a competing standalone repair.
