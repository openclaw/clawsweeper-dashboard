## Label Justifications

- `P2`: This is a normal-priority session continuity improvement with clear UX impact but no emergency runtime outage or confirmed data deletion.
- `impact:session-state`: The issue is about transcript and session continuity across rolled `sessionId` segments for one logical chat.

## Summary

Keep open: current main and the latest shipped release still expose active-session `chat.history` only, while reset archive files remain real on-disk state and related archive-history PRs are open rather than merged.

## What This Changes

The issue requests an optional logical-history mode for `chat.history` that can return the current segment plus prior rolled/reset transcript segments for the same stable session key.

## Best Possible Solution

Choose one backward-compatible Gateway contract that keeps current active history as the default, adds explicit logical/archive history with bounded reads and ordered segment metadata, and updates WebChat, `sessions_history`, docs, generated clients, and tests together.

## Reproduction Assessment

Yes, at source level. Current main rejects any logical/archive selector, `chat.history` and `sessions_history` read only the active transcript, and reset archives are still created on disk.

## Solution Assessment

Unclear until maintainers choose the public contract. A Gateway-first additive mode is plausible, but it should be coordinated with the related archive-history PRs rather than implemented as a competing standalone repair.
