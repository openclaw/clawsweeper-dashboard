## Summary

Keep open: current main still exposes `chat.history` and `sessions_history` as active-session reads, while logical history across rolled reset segments remains an unresolved Gateway/API product decision with active related PRs.

## What This Changes

The issue requests an optional logical-history mode for `chat.history` that returns current and previous rolled session segments with lightweight segment metadata for WebChat and Control UI continuity.

## Best Possible Solution

Choose and land a backward-compatible Gateway contract for logical history that keeps current behavior as the default, preserves bounded reads, returns ordered reset-segment metadata, documents the protocol/tool behavior, and coordinates WebChat plus `sessions_history`.

## Reproduction Assessment

Yes, at source level. Current main rejects logical/archive history parameters and reads only the active transcript behind the current `sessionId`, while reset archives are still created on disk.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive option is plausible, but it should be coordinated with https://github.com/openclaw/openclaw/pull/73883 and https://github.com/openclaw/openclaw/pull/76134 rather than started as a competing repair.
