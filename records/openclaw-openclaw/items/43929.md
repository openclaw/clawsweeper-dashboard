- `impact:session-state`: The issue is about transcript and session continuity across rolled `sessionId` segments for one logical chat.

## Summary

Keep open: current main still exposes only active-session history for `chat.history` and `sessions_history`, while reset archives are created on disk but not returned as one logical chat. The remaining work is an additive Gateway/API product decision with related open PRs, not a stale or already-implemented cleanup item.

## What This Changes

Add an optional logical-history mode to `chat.history` that returns the current segment plus previous reset/rolled segments with lightweight segment metadata.

## Best Possible Solution

Choose one backward-compatible Gateway contract that keeps current history as the default, adds an explicit logical/archive mode, preserves bounded reads, returns ordered segment metadata, and updates WebChat, `sessions_history`, docs, and tests together.

## Reproduction Assessment

Yes, at source level. Current main rejects logical/archive history parameters, `chat.history` and `sessions_history` read the active transcript only, and reset archives are still created on disk.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive option is plausible, but it should be coordinated with the related open archive-history PRs instead of started as a competing repair.
