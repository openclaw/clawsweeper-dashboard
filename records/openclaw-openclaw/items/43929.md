## Summary

Keep this issue open. Current main still exposes `chat.history` as a current-session transcript read, while reset flows create a new `sessionId` and archive the old transcript; related archive-recovery work is adjacent but does not implement logical current-plus-rolled history with segment metadata for WebChat or Control UI.

## What This Changes

The issue requests an optional logical `chat.history` mode that returns the current chat segment plus previous rolled segments for the same logical session key, including lightweight segment metadata.

## Best Possible Solution

Add an additive Gateway protocol extension, likely `chat.history(scope: "logical")` while keeping current behavior as default, backed by persisted reset lineage and bounded ordered segment reads that include `sessionId`, start/end time, and reset reason metadata; then update WebChat/Control UI and docs to render continuity without direct file scanning.

## Reproduction Assessment

Yes. A high-confidence source reproduction is: roll a stable session key through `sessions.reset` or daily reset, then call `chat.history` for that same key; current main will read only the new `entry.sessionId` transcript even though the old transcript was archived as `.reset.*`. I did not run a live mutation test because this review is read-only.

## Solution Assessment

Yes at the boundary level: the Gateway is the right layer because remote WebChat and Control UI clients should not scan transcript files. The exact API fields and archive-chain persistence need maintainer design, especially alongside #45003, #56131, and open PR #60409.
