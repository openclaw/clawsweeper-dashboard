## Summary

Keep open: current main still exposes `chat.history` and `sessions_history` as current-segment reads, while logical history across reset archives remains an unresolved Gateway protocol/tool product decision with active related PRs covering only partial slices.

## What This Changes

Request a Gateway logical-history mode for `chat.history` that returns the current rolled session segment plus earlier reset segments for the same logical session key with segment metadata.

## Best Possible Solution

Choose and land a backward-compatible Gateway contract for logical history that keeps current behavior as the default, preserves bounded reads, returns ordered reset-segment metadata, documents the protocol/tool behavior, and coordinates WebChat plus `sessions_history`.

## Reproduction Assessment

Yes, at source level. Current main rejects logical/archive history parameters and reads only the active transcript behind the current `sessionId`, while reset archives are still created on disk.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive option is plausible, but it should be coordinated with https://github.com/openclaw/openclaw/pull/73883 and the narrower fallback work in https://github.com/openclaw/openclaw/pull/76134 rather than started as a competing repair.
