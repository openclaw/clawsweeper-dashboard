## Summary

Keep open: current main still exposes `chat.history` as a current-segment read, while reset rollover archives the prior transcript. Related PRs are open and useful, but the logical-history API and segment metadata requested here are not merged on main and need maintainer API/UX direction.

## What This Changes

Add a logical-history mode to `chat.history` so a stable chat can include previous rolled transcript segments with lightweight segment metadata.

## Best Possible Solution

Add an explicit, backward-compatible Gateway history mode that preserves current behavior by default, returns ordered rolled segments plus segment metadata when requested, and coordinates with the existing archive-history PRs before UI adoption.

## Reproduction Assessment

Not a bug reproduction in the strict sense, but source inspection gives high-confidence current-main proof of the gap: the schema lacks a logical/archive mode and the handler reads only the current session entry's transcript.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested Gateway-first direction is plausible, but the safest solution should align or revise the open `includeArchived`/archive-fallback PRs rather than creating a competing implementation.
