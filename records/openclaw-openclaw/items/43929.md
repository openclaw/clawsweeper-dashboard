## Summary

Keep open: current main and the latest release still expose `chat.history` as a current-session transcript read, while the archive/logical-history implementation work is still in open PRs. This is a Gateway protocol/API feature that needs maintainer direction rather than a cleanup close.

## What This Changes

Add an optional logical-history mode to `chat.history` so a stable logical session key can return rolled transcript segments and lightweight segment metadata across resets.

## Best Possible Solution

Add a backward-compatible Gateway protocol option for logical history, keep current behavior as the default, return ordered rolled-segment metadata, and align WebChat plus `sessions_history` after maintainers choose the API contract.

## Reproduction Assessment

Yes at source level. Current main rejects logical/archive parameters in the `chat.history` schema and the handler reads only the active session entry's transcript through `readRecentSessionMessagesAsync`.

## Solution Assessment

Unclear until maintainers choose the protocol shape. A Gateway-first additive option is plausible, but the existing open archive-history PRs should be reviewed or revised rather than starting a competing implementation.
