## Summary

Keep open: current main still exposes `chat.history` as a current active-transcript read and rejects logical/archive parameters; related PRs are open but no merged implementation provides the requested logical history and segment metadata.

## What This Changes

Add optional logical history to `chat.history` so long-lived session keys can return current and rolled transcript segments with lightweight metadata.

## Best Possible Solution

Add a backward-compatible Gateway logical-history mode that keeps current behavior by default, returns ordered rolled segments plus segment metadata when requested, and aligns `sessions_history` and UI usage after maintainer review of the open archive-history PRs.

## Reproduction Assessment

Yes at source level. Current main rejects any logical/archive parameter in the `chat.history` schema and the handler reads only the current session entry's transcript through `readRecentSessionMessagesAsync`.

## Solution Assessment

Unclear until maintainers choose the protocol shape. The Gateway-first direction is plausible, but the safest path is to review or revise the open archive-history PRs rather than start a competing implementation.
