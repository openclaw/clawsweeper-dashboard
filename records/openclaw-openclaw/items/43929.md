## Label Justifications

- `P2`: This is a normal-priority session continuity improvement with recurring user impact but no crash, security bypass, or unusable runtime.
- `impact:session-state`: The issue is about transcript and chat continuity across rolled `sessionId` segments for one logical session key.

## Summary

Current main still does not expose a logical/archive history mode: the Gateway protocol, `chat.history` handler, bounded transcript reader, and `sessions_history` tool all stay scoped to the current active session segment. The issue should remain open for maintainer product/API review because related implementation PRs exist but are still open and not yet merged.

## What This Changes

Add an optional logical-history mode to `chat.history` so clients can read the current transcript segment plus previous rolled segments for the same logical session key.

## Best Possible Solution

Adopt one backward-compatible Gateway contract that keeps current history as the default, adds explicit bounded logical/archive reads with ordered segment metadata, and updates WebChat, `sessions_history`, protocol docs/generated clients, and tests together.

## Reproduction Assessment

Yes, at source level. Current main rejects any logical/archive selector and reads only the active transcript while reset archives are still created on disk.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive mode is plausible, but it should be coordinated with the related archive-history PRs instead of handled as a competing standalone repair.
