- `P2`: This is a normal-priority session continuity improvement with clear user impact but no emergency runtime outage.
- `impact:session-state`: The issue is about transcript and session continuity across rolled `sessionId` segments for one logical chat.

## Summary

Keep open: current main and the latest release still expose active-session history only, while logical history across reset archives remains an unresolved Gateway/API product decision with active related PRs.

## What This Changes

Add a logical-history mode to `chat.history` so one stable logical session key can return current and rolled transcript segments with segment metadata.

## Best Possible Solution

Choose and land one backward-compatible Gateway contract for logical history that keeps current behavior as the default, preserves bounded reads, returns ordered reset-segment metadata, documents the protocol/tool behavior, and coordinates WebChat plus `sessions_history`.

## Reproduction Assessment

Yes, at source level. Current main rejects logical/archive history parameters, `chat.history` and `sessions_history` read the active transcript only, and reset archives are still created on disk.

## Solution Assessment

Unclear until maintainers choose the protocol and visibility contract. A Gateway-first additive option is plausible, but it should be coordinated with https://github.com/openclaw/openclaw/pull/73883 and https://github.com/openclaw/openclaw/pull/76134 rather than started as a competing repair.
