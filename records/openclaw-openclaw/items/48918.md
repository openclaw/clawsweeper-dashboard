## Summary

Keep open. Current main and v2026.6.10 still define skills around `SKILL.md` plus structured per-skill runtime config, but there is no supported `preferences.md` or `conventions.md` sidecar loaded with a triggered skill; the remaining work is a maintainer product/API decision, not stale cleanup.

## What This Changes

The issue requests a supported per-skill user preferences or conventions file that loads only when that skill is triggered.

## Best Possible Solution

Define one maintainer-approved skill-preferences contract with filename, precedence, load timing, snapshot/version invalidation, security boundaries, and docs, then implement it across loader, prompt/command expansion, tests, and public docs.

## Reproduction Assessment

Yes. The source-level reproduction is to inspect current skill loading and prompt construction: both still use `SKILL.md` as the loaded skill unit, and targeted source/docs search finds no supported preference sidecar contract.

## Solution Assessment

Unclear. A markdown sidecar is plausible, but the best solution depends on maintainer choices for filename, precedence, refresh behavior, security boundaries, and whether this is a docs convention or a new runtime contract.
