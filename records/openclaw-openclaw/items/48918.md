## Summary

Keep open. Current main and v2026.6.10 still define skills around `SKILL.md`, documented source precedence, and structured per-skill config, but they do not support a `preferences.md` or `conventions.md` sidecar that is loaded only when a skill is triggered; the remaining work is a maintainer product/API decision.

## What This Changes

Request support for user-scoped per-skill preference or convention files that load only when the matching skill is triggered.

## Best Possible Solution

Define one maintainer-approved skill preference contract with filename, precedence, load timing, root containment, snapshot invalidation, docs, and tests before implementing the loader change.

## Reproduction Assessment

Yes, as a source-level feature gap. Current main still loads `SKILL.md` as the skill unit, replaces same-named skills by source precedence, and has no supported `preferences.md` or `conventions.md` sidecar contract.

## Solution Assessment

Unclear. A markdown sidecar is plausible, but maintainers still need to choose the canonical filename, precedence, refresh behavior, security boundary, and whether this belongs in loader behavior or a different documented skill/config contract.
