## Summary

Keep open. Current main and v2026.6.6 still load `SKILL.md` plus existing per-skill config/env overrides, but do not standardize or auto-load `preferences.md` or `conventions.md`; this remains a product/API decision, not cleanup-close material.

## What This Changes

The issue requests a standardized per-skill user preferences or conventions file that is loaded only when the matching skill triggers.

## Best Possible Solution

Define a maintainer-approved skill preferences contract with file name, precedence, load timing, snapshot/watcher invalidation, docs, and security boundaries, then implement it consistently across skill loading, prompt construction, runtime snapshots, config/schema if needed, and tests.

## Reproduction Assessment

Yes, as a source-level feature gap rather than a failing runtime reproduction. The current loader and prompt contract do not auto-load `preferences.md` or `conventions.md` as first-class skill context.

## Solution Assessment

Unclear until maintainers choose the contract. A markdown convention may fit the user need, but a loader-only patch would be too narrow without precedence, refresh, security, and docs semantics.
