## Summary

Keep open. The report is recent, specific, and still maps to current main: each embedded attempt still materializes tools, builds the system prompt, and rewires stream setup, while current caching mainly covers provider prompt-cache shaping and pieces of skill snapshot hydration rather than a reusable per-request startup artifact.

## What This Changes

The issue requests reducing repeated per-message latency in embedded-run setup by caching stable tool/system-prompt work, improving remote stream setup, and avoiding event-loop blocking.

## Best Possible Solution

Add measured, versioned caches for stable tool/schema and prompt-prefix artifacts with explicit invalidation, and separately evaluate provider transport reuse for OpenAI-compatible remote endpoints without weakening per-turn policy checks.

## Reproduction Assessment

Yes. The issue gives a concrete environment and trace labels, and current main still emits those labels from the same per-attempt code path; reproduce by enabling embedded-run tracing and sending consecutive messages in the same Feishu-backed session.

## Solution Assessment

Unclear. The requested caching and keep-alive direction is plausible, but the safest implementation needs profiling and a cache-key design that separates stable artifacts from per-turn policy, auth, channel, sandbox, and runtime state.
