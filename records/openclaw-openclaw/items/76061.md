## Summary

Keep open. The report is recent, specific, and maps directly to current embedded-runner trace stages; current main still builds tool definitions, assembles the system prompt, and reconfigures stream setup during each run, while existing caching only covers pieces such as skill snapshots and provider-side prompt cache behavior.

## What This Changes

Reduce per-message embedded-run startup latency by caching stable tool/system-prompt work, improving remote stream setup reuse, and avoiding event-loop blocking during request preparation.

## Best Possible Solution

Add measured, versioned caches for stable tool/schema and prompt-prefix artifacts with explicit invalidation, and separately evaluate provider transport reuse/keep-alive for OpenAI-compatible remote endpoints without weakening per-turn policy checks.

## Reproduction Assessment

Yes. The issue gives a concrete environment and trace labels; a maintainer can reproduce by enabling embedded-run tracing and sending consecutive Feishu-backed messages through the same session while comparing `core-plugin-tools`, `system-prompt`, and `stream-setup` timings.

## Solution Assessment

Unclear. The requested direction is plausible, but the right fix should be staged from profiling because the bottlenecks span local prompt/tool construction, policy-sensitive invalidation, and provider transport behavior.
