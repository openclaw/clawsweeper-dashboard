## Summary

Keep open. The trusted-proxy WebSocket docs gap is real on current main, but this PR is not merge-ready because it still says `x-openclaw-scopes` is HTTP-only for WebSocket and overgeneralizes device-less token/password Control UI behavior; the supplied proof is source-inspection only.

## What This Changes

The branch adds trusted-proxy and Gateway protocol documentation about WebSocket scope clearing, `x-openclaw-scopes`, missing-scope troubleshooting, and `dangerouslyDisableDeviceAuth`.

## Best Possible Solution

Revise the docs to distinguish device-less trusted-proxy scope clearing from paired trusted-proxy WebSocket scope bounding, document token/password missing-device rejection/preserve exceptions accurately, and keep `dangerouslyDisableDeviceAuth` as narrow break-glass guidance.

## Reproduction Assessment

Yes for the review findings: current source and tests show WebSocket upgrades read `x-openclaw-scopes`, paired trusted-proxy WebSocket scopes are bounded by that header, and device-less trusted-proxy scopes clear. This is not a runtime bug reproduction; it is a source-verifiable docs mismatch.

## Solution Assessment

No. A docs-only correction is the right surface, but this wording is not the best solution until it describes the current WebSocket scope-header and missing-device token/password boundaries accurately.
