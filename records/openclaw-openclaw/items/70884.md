## Summary

This PR should stay open because the docs-only direction is useful, but the current wording conflicts with current main's trusted-proxy Control UI WebSocket behavior and changes security-sensitive auth guidance in the wrong direction.

## What This Changes

The PR adds trusted-proxy and protocol documentation notes about missing-device WebSocket scope clearing, `x-openclaw-scopes` scope, troubleshooting, and `dangerouslyDisableDeviceAuth` guidance.

## Best Possible Solution

Keep this as a focused docs correction, but state that successful trusted-proxy Control UI WebSockets can be device-less and have scopes bounded by `x-openclaw-scopes`; reserve clearing and break-glass guidance for paths that actually clear scopes.

## Reproduction Assessment

Yes. Source inspection and current tests show the mismatch: the WebSocket handler reads `x-openclaw-scopes`, the control-ui auth suite asserts WebSocket scope bounding, and the clearing predicate excludes successful `trustedProxyAuthOk`.

## Solution Assessment

No. A docs-only fix is appropriate, but the current PR overcorrects; the safer solution is to distinguish successful trusted-proxy Control UI WebSocket preservation/bounding from the remaining unbound scope-clearing paths.
