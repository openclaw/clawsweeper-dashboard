## Summary

Keep this PR open. The latest force-push is back to a docs-only change and removes the earlier unrelated provider-request code, but the new docs still conflict with current trusted-proxy WebSocket behavior and the merged scope-preservation work in https://github.com/openclaw/openclaw/pull/79643.

## What This Changes

The PR adds trusted-proxy and Gateway protocol docs about WebSocket scope clearing, the `x-openclaw-scopes` header, and missing-scope troubleshooting.

## Best Possible Solution

Rework the docs to describe current post-https://github.com/openclaw/openclaw/pull/79643 behavior: trusted-proxy Control UI WebSockets can preserve and header-bound scopes, `device: null` rejection is distinct from unpaired signed device identity, and `dangerouslyDisableDeviceAuth` remains break-glass guidance only.

## Reproduction Assessment

Yes. The mismatch is source-reproducible on current main: the WebSocket handler reads and applies `x-openclaw-scopes`, and current tests cover both header narrowing and `device: null` rejection; I did not run tests because this review is read-only.

## Solution Assessment

No. A focused docs correction is the right shape, but this wording documents older or over-broad behavior instead of the current trusted-proxy Control UI scope-preservation and device-identity rules.
