## Summary

Keep open: this docs-only correction belongs in the right area, but the current branch still conflicts with main because trusted-proxy Control UI WebSocket upgrades do read `x-openclaw-scopes`, and the protocol note overgeneralizes missing-device token/password cases. The external proof is still source inspection only.

## What This Changes

The PR adds Gateway trusted-proxy and protocol docs about device-less WebSocket scope clearing, `x-openclaw-scopes`, missing-scope troubleshooting, and `dangerouslyDisableDeviceAuth`.

## Best Possible Solution

Revise the docs to distinguish paired trusted-proxy WebSocket scope bounding from device-less scope clearing, and keep `dangerouslyDisableDeviceAuth` framed as a narrow break-glass exception.

## Reproduction Assessment

Yes for the review findings: current-main source and tests show WebSocket upgrades read `x-openclaw-scopes`, paired trusted-proxy WebSocket scopes are bounded by that header, and device-less trusted-proxy scopes clear. This is a source-verifiable docs mismatch, not a runtime bug reproduction.

## Solution Assessment

No. A docs-only correction is the right surface, but this patch needs to describe the current WebSocket header and missing-device boundaries accurately before it is the maintainable fix.
