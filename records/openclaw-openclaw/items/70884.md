## Summary

Keep open: the docs-only PR is still a plausible fix path for the linked docs issue, but current main shows the patch overstates HTTP-only scope handling and broadens missing-device guidance in security-sensitive trusted-proxy docs.

## What This Changes

The PR adds Gateway trusted-proxy and protocol documentation for WebSocket scope clearing, missing-scope troubleshooting, `x-openclaw-scopes`, and `dangerouslyDisableDeviceAuth`.

## Best Possible Solution

Revise the docs to describe current main: paired trusted-proxy Control UI WebSockets can be bounded by `x-openclaw-scopes`, fresh unpaired device identities require pairing, device-less trusted-proxy sessions connect with empty scopes, and break-glass device-auth bypass guidance stays narrow.

## Reproduction Assessment

Yes for the review findings: current-main source and tests show the WebSocket header is read and paired sessions are bounded by it, while device-less trusted-proxy sessions clear scopes. This is a source-verifiable docs mismatch, not a runtime bug reproduction.

## Solution Assessment

No. A docs-only correction is the right surface, but this patch should align with the current WebSocket contract and narrower missing-device exceptions before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Describe WebSocket scope headers accurately:** `docs/gateway/trusted-proxy-auth.md:323-325`
  - body: The added note says `x-openclaw-scopes` applies to HTTP endpoints only, but current main reads that header from trusted-proxy Control UI WebSocket upgrades and tests assert paired sessions are bounded to the proxy-declared scope set. Please document paired WebSocket scope bounding separately from device-less scope clearing.
  - confidence: 0.95
- **[P2] Narrow missing-device scope guidance:** `docs/gateway/protocol.md:719-725`
  - body: This says Control UI token/password/trusted-proxy connections without device identity are allowed and then lose scopes. Current policy only allows the trusted-proxy device-less path there; token/password Control UI without device identity is generally rejected, while local `allowInsecureAuth` preserves scopes. Please narrow the allowed-empty-scope case and avoid steering unrelated paths to the break-glass flag.
  - confidence: 0.92
