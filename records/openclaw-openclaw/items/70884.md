## Summary

Keep open: the current branch is docs-only after the force-push, but current main contradicts the central wording because trusted-proxy Control UI WebSocket upgrades read `x-openclaw-scopes`, and successful trusted-proxy auth is excluded from missing-device scope clearing.

## What This Changes

The PR adds trusted-proxy and protocol documentation notes about missing-device WebSocket scope clearing, `x-openclaw-scopes` scope, troubleshooting, and `dangerouslyDisableDeviceAuth` guidance.

## Best Possible Solution

Keep this as a focused docs correction, but state that successful trusted-proxy Control UI WebSockets can be device-less and have scopes bounded by `x-openclaw-scopes`; reserve scope-clearing guidance for the paths that actually clear scopes.

## Reproduction Assessment

Yes. Source inspection and existing tests on current main show the mismatch: the WebSocket handler reads `x-openclaw-scopes`, the control-ui auth suite asserts it bounds trusted-proxy WebSocket scopes, and the clearing predicate excludes `trustedProxyAuthOk`.

## Solution Assessment

No. The docs-only direction is appropriate, but the current wording overcorrects; the safer fix is to narrow the docs to the missing-device cases that actually clear scopes and preserve the current trusted-proxy WebSocket header behavior.
