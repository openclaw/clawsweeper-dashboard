## Summary

Keep open: the docs correction is useful, but the current branch is no longer docs-only and contains two blocking source-level issues plus a security-sensitive provider-header feature that belongs in the separate feature review.

## What This Changes

The branch adds trusted-proxy WebSocket scope-clearing documentation and also adds per-agent/per-cron provider request override plumbing, config types/schema, and tests.

## Best Possible Solution

Split or rebase this into a focused docs correction that distinguishes missing-device WebSocket scope clearing from paired trusted-proxy WebSocket scope bounding, and handle provider request headers in https://github.com/openclaw/openclaw/pull/79990 with protocol, security, docs, and proof coverage.

## Reproduction Assessment

Yes. The blockers are source-reproducible: current main WebSocket code/tests show the header also affects paired trusted-proxy WebSocket upgrades, and the cron protocol schema rejects undeclared providers fields.

## Solution Assessment

No. A docs-only clarification is the right direction for the scope-clearing gap, but this branch overstates WebSocket header behavior and bundles an unrelated provider-header feature that should be reviewed through its canonical feature PR.

## Review Findings

  - body: The added note says `x-openclaw-scopes` applies to HTTP endpoints only, but the WebSocket handler reads that header for trusted-proxy Control UI upgrades and current tests assert it bounds paired WebSocket scopes.   - confidence: 0.94

  - confidence: 0.88
