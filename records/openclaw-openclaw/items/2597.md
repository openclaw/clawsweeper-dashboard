## Summary

Keep open: current main still lacks the requested passive `context=X%` field in the system-prompt Runtime line. Existing `/status`, `/context`, memory-flush, post-compaction recovery, context-engine plugins, and the linked `openclaw-workspace-sync` community plugin are useful mitigations, but they do not implement the per-turn agent-visible prompt primitive, and the latest maintainer-facing discussion also kept this issue open.

## What This Changes

Add a passive `context=X%` field to the system-prompt Runtime line so embedded agents can see approximate context-window usage each turn.

## Best Possible Solution

Implement a bounded embedded-runner Runtime context percentage derived from a reliable prompt-token snapshot and resolved context window, omit it when unavailable, cover it with focused prompt/runtime tests, and keep existing diagnostics and sync plugins as complementary recovery surfaces.

## Reproduction Assessment

Yes for the missing requested surface: current source inspection of `RuntimeInfoInput`, `buildRuntimeLine()`, and embedded attempt runtime assembly shows no `context=` path. The real-world compaction data loss was not live-replayed, but the absence of the proposed agent-visible gauge is directly reproducible from source.

## Solution Assessment

Yes at the direction level: a bounded, omitted-when-unavailable `context=X%` Runtime field is the narrowest maintainable way to satisfy this issue. The safer implementation detail is to derive it from fresh prompt/context-window session metadata rather than provider cumulative usage totals, and to keep CLI backends excluded unless they gain reliable session tracking.
