## Summary

Keep open: current main still lacks the requested passive `context=X%` field in the system-prompt Runtime line. The existing `/status`, `/context`, UI metadata, memory-flush, and context-engine/compaction mitigations help users inspect or recover context state, but they do not give the agent the per-turn Runtime-line signal requested here, and the latest maintainer-facing discussion also kept the issue open.

## What This Changes

The issue requests adding a bounded context-usage percentage to the agent Runtime line for embedded runs so agents can proactively checkpoint state before compaction or reset.

## Best Possible Solution

Implement a small embedded-runner-only runtime prompt field derived from the latest reliable prompt-token snapshot and resolved context window, omit it when unavailable, and document that the value is one turn stale. Existing explicit diagnostics and sync plugins should remain complementary recovery/inspection surfaces, not replacements for the requested prompt primitive.

## Reproduction Assessment

Yes for the missing requested surface: current source inspection of `RuntimeInfoInput`, `buildRuntimeLine()`, and embedded runtime assembly shows no `context=` path. The field-reported compaction data loss itself was not locally replayed, but the absence of the proposed agent-visible gauge is directly verifiable.

## Solution Assessment

Yes at the direction level: a bounded, omitted-when-unavailable `context=X%` Runtime field is the narrowest way to satisfy the specific request. The safer implementation detail is to derive it from a fresh prompt/context snapshot rather than broad provider usage totals, and to leave CLI backends excluded unless they gain reliable session context tracking.

## Repair Work Prompt

Implement the requested passive context-usage percentage for embedded Pi runs. Add an optional bounded context percentage carrier from the latest reliable previous-turn prompt/context snapshot into `RunEmbeddedPiAgentParams`, `EmbeddedRunAttemptParams`, `RuntimeInfoInput`, and `buildRuntimeLine()`, rendering `context=X%` only when both used tokens and context window are finite positive numbers. Clamp to a safe display range such as 0-999, omit on `NaN`/`Infinity`/missing/stale data, and do not derive the percentage from broad provider `usage.total` when that can include output/tool-loop/cached totals rather than live prompt context. Keep CLI backends excluded unless a reliable context snapshot already exists for them. Add focused unit tests for runtime-line rendering/omission/clamping and the embedded-runner or agent-runner plumbing path. Update relevant docs for Runtime/token-use behavior and add a user-facing changelog entry if the repo’s active changelog section expects one, crediting the reporter and relevant non-bot commenters; do not touch unrelated compaction, workspace-sync, provider, or plugin policy behavior.
