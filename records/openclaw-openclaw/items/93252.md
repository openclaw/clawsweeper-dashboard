## Summary

Keep open: the abusive attribution claims are not code evidence, but the core concern is source-backed because current OpenClaw intentionally routes unconfigured managed web_search to Parallel Search (Free); changing that shipped default needs maintainer product/security judgment rather than conservative auto-close.

## What This Changes

The issue asks OpenClaw to stop or revisit unconfigured web_search traffic routing to Parallel.ai's hosted Search MCP.

## Best Possible Solution

Maintainers should explicitly decide whether Parallel Search (Free) remains the keyless default; if not, change provider selection, setup/docs, and tests so hosted Parallel routing requires explicit provider choice or credentials.

## Reproduction Assessment

Yes, at source level: current main marks `parallel-free` credentialless, selects the first keyless fallback when no configured credentials exist, and documents that Parallel Search (Free) is the zero-config default. I did not run a live Windows reproduction for the reporter's exact installation.

## Solution Assessment

Unclear as an implementation path: the requested direction may be appropriate for privacy and consent, but current docs and merged PR history show the default was intentional. The narrowest maintainable solution starts with a maintainer decision on the default policy, then a focused provider-selection/docs/test change if the policy changes.
