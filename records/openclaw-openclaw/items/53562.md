## Summary

Keep open. The requested Discord voice companion text-channel routing is still absent from current main, and the implementation paths in the context are unmerged: the OpenClaw PR is closed while an external fork PR remains open. This needs maintainer follow-up rather than cleanup closure.

## What This Changes

Add an optional Discord voice `autoJoin[].sessionChannelId` so voice STT transcripts and AI replies can route to a companion text channel while voice playback continues in the joined voice channel.

## Best Possible Solution

Implement this in the Discord plugin/config surface with a typed and schema-validated `autoJoin[].sessionChannelId`, validate the target text channel against the guild and bot permissions, route paired voice sessions through that text-channel session, enable Discord delivery only for that paired route, preserve voice playback, update docs/config metadata, and cover the behavior with focused config and voice manager/segment tests while coordinating with #73699 and #47095.

## Reproduction Assessment

Not applicable as a feature request, but current-main source inspection gives a high-confidence verification path: the config/schema/docs lack `sessionChannelId`, auto-join cannot pass it, and voice segment delivery remains disabled.

## Solution Assessment

Unclear until maintainers settle the narrow companion-channel option versus #73699's broader voice-as-I/O design. If the narrow option is accepted, a Discord plugin-owned config/routing/delivery change with explicit validation is the maintainable path.
