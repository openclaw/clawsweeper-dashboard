## Summary

Keep this open: current main and v2026.6.6 still lack the requested per-auto-join text-channel route and visible transcript/reply delivery, while the existing `voice.agentSession` target only covers session routing. The same-repository implementation attempts were closed, and the remaining fork PR is conflicting, so this needs maintainer product direction rather than cleanup closure.

## What This Changes

The issue requests a Discord voice `autoJoin[].sessionChannelId` option that routes voice transcripts and replies to a paired text channel while continuing audio playback in the voice channel.

## Best Possible Solution

Choose one canonical Discord voice bridge design that reuses `voice.agentSession` where possible, explicitly defines transcript/reply delivery semantics, validates the text target, documents the config, and covers routing, delivery, recovery, and mention behavior with focused tests.

## Reproduction Assessment

Yes. Source inspection shows `autoJoin` accepts only `guildId` and `channelId`, the strict schema rejects `sessionChannelId`, and Discord voice agent turns still use `deliver: false`; no live Discord voice run was attempted in this read-only review.

## Solution Assessment

Unclear. The per-auto-join field is plausible, but current main already has global `voice.agentSession`, so the best fix should avoid a competing config source and needs maintainer agreement on delivery and safety semantics.
