## Summary

Keep open. Current main still lacks the requested Kubernetes ServiceAccount TokenReview trust path for autoscaled headless node/worker pods; the linked implementation PR is closed unmerged, and the broader distributed-runtime RFC does not replace this specific authentication and pairing-bypass requirement.

## What This Changes

Add an opt-in Kubernetes ServiceAccount TokenReview authentication path so autoscaled Kubernetes node/worker pods can connect without interactive pairing, plus deployment manifests and Helm support.

## Best Possible Solution

Keep the issue open until maintainers either design and implement a security-reviewed, opt-in Kubernetes ServiceAccount Trust path in the gateway/node protocol or fold this exact auth requirement into a narrower canonical distributed-runtime Kubernetes item.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence source-level verification path: current config, schema, protocol auth payloads, node CLI/runtime, and Kubernetes manifests do not carry or validate a ServiceAccount TokenReview credential.

## Solution Assessment

Unclear. Kubernetes SA TokenReview is a plausible solution for headless autoscaling, but the best maintainable path needs maintainer/security design because it changes gateway trust and node pairing-bypass semantics.
