## Summary

Current main still lacks Kubernetes workload-identity authentication: its Kubernetes support is a single-pod, shared-token baseline, while node bootstrap tokens remain approval-oriented and CIDR auto-approval trusts network origin rather than ServiceAccount identity. The prior broad PR was closed unmerged with an explicit invitation to return the SA-trust portion as focused work, and no viable replacement currently owns it; the issue should remain open for a maintainer security and distributed-runtime product decision.

## What This Changes

Requests an opt-in Kubernetes ServiceAccount TokenReview authentication path so dynamically scaled worker pods can connect to a gateway without interactive pairing.

## Best Possible Solution

Approve a focused generic machine-workload identity contract at the distributed-runtime registration boundary, then implement fail-closed Kubernetes TokenReview verification with strict audience and namespace/ServiceAccount allowlists while keeping Helm and unrelated deployment packaging separate.

## Reproduction Assessment

Not applicable: this requests a new zero-touch Kubernetes worker authentication mode; current manual pairing, bootstrap-token, CIDR auto-approval, and shared-token behavior operate as designed.

## Solution Assessment

No, not in the original broad PR shape. A focused, security-reviewed workload-identity contract coordinated with distributed-runtime protocol work is narrower and more maintainable than combining auth, Helm, deployment assets, and unrelated provider changes.
