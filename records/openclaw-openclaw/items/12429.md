## Summary

Keep open. Current main still lacks the requested Kubernetes ServiceAccount TokenReview trust path for autoscaled node or worker pods; existing Kubernetes support is a single-pod token-auth deployment, and the linked implementation PR is closed unmerged while the distributed-runtime RFC is broader architecture work.

## What This Changes

Request an opt-in Kubernetes ServiceAccount Trust authentication mode, plus manifests/Helm support, so headless autoscaled workers can connect without per-pod interactive pairing.

## Best Possible Solution

Keep this issue open until maintainers either design and implement a security-reviewed, opt-in Kubernetes ServiceAccount Trust path in the gateway/node protocol or fold this exact auth requirement into a narrower canonical distributed-runtime Kubernetes item.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence source-level verification path: current config, strict schema, protocol auth payloads, node CLI/runtime, and Kubernetes manifests do not carry or validate a ServiceAccount TokenReview credential.

## Solution Assessment

Unclear. Kubernetes SA TokenReview is a plausible approach for headless autoscaling, but because it changes gateway trust and pairing bypass semantics, the best maintainable solution needs maintainer/security design rather than an automated port of the closed PR.
