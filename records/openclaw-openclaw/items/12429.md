## Summary

This issue should stay open: current main and the latest release still have single-pod Kubernetes deployment support plus token/password/device/trusted-proxy auth, but not the requested headless Kubernetes ServiceAccount TokenReview trust path for autoscaled worker pairing. The linked implementation at https://github.com/openclaw/openclaw/pull/12434 is closed unmerged, and the open distributed-runtime RFC at https://github.com/openclaw/openclaw/issues/42026 does not yet define this workload-identity contract.

## What This Changes

Request an opt-in Kubernetes ServiceAccount Trust authentication flow so autoscaled worker nodes can authenticate to a gateway without interactive PIN/device pairing.

## Best Possible Solution

Define a generic workload-identity contract at the gateway/runtime boundary, then add Kubernetes TokenReview as an opt-in fail-closed backend with explicit allowlists, TLS/CA handling, pairing-bypass tests, and focused deployment docs.

## Reproduction Assessment

Not applicable: this issue requests a new Kubernetes workload-identity mode rather than reporting broken behavior promised by current docs or tests. The absence of the capability is source-verifiable from the current auth types, config schema, protocol schema, node-host client setup, and Kubernetes manifests.

## Solution Assessment

No, the earlier broad Kubernetes patch is not the best final shape. The maintainable path is to first establish the gateway/runtime identity boundary, then implement a narrowly scoped, fail-closed Kubernetes TokenReview backend.
