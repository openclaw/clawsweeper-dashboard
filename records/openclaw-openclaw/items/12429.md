## Summary

Keep open. Current main still has shared token/password/trusted-proxy plus device/bootstrap-token gateway auth and a single-pod Kubernetes baseline, but no Kubernetes ServiceAccount Trust or TokenReview path for autoscaled worker/node pods; the prior implementation PR was closed unmerged and the open distributed-runtime RFC does not decide this specific trust boundary.

## What This Changes

The issue requests an opt-in Kubernetes ServiceAccount Trust authentication path so autoscaled worker nodes can authenticate to the gateway without interactive device pairing.

## Best Possible Solution

Keep this open until maintainers approve a focused Kubernetes identity design or explicitly fold this requirement into a reviewed distributed-runtime authentication plan.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new Kubernetes identity/authentication feature. Source, docs, release, and related-PR inspection give high-confidence negative capability evidence that current main has no k8sToken, k8sTrust, k8s-sa-trust, or TokenReview branch.

## Solution Assessment

Unclear until maintainers make the product and security decision. The safer path is a focused opt-in auth change across config/schema, protocol payloads, node-host credential handling, fail-closed TokenReview validation, tests, docs, and deployment assets rather than reopening the old broad PR wholesale.
