## Summary

Keep open. Current main still has a minimal single-pod Kubernetes deployment with token auth, but no opt-in Kubernetes ServiceAccount Trust / TokenReview path for autoscaled worker or node pods; the linked implementation pull request is closed unmerged, and the open distributed-runtime RFC does not specifically replace this auth request.

## What This Changes

Request an opt-in Kubernetes ServiceAccount Trust auth path so headless autoscaled Kubernetes workers can authenticate to a gateway without interactive device pairing.

## Best Possible Solution

Keep the issue open until maintainers either approve an opt-in Kubernetes ServiceAccount Trust design or fold the auth requirement into a narrower distributed-runtime/Kubernetes identity plan.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request for a new Kubernetes identity path. Source and docs inspection give high-confidence negative capability evidence that current main has no k8sToken, k8sTrust, k8s-sa-trust, or TokenReview branch.

## Solution Assessment

Unclear until maintainers make the product and security decision. The previous broad PR is not the best path to merge wholesale; a focused design should cover config/schema, protocol payloads, gateway validation, node-host credential handling, tests, docs, and deployment assets.
