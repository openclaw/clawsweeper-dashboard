## Summary

Keep open. Current main and the latest shipped release still lack the requested opt-in Kubernetes ServiceAccount/TokenReview authentication path for autoscaled worker pods, the linked implementation PR at https://github.com/openclaw/openclaw/pull/12434 was closed unmerged, and the broader distributed-runtime RFC at https://github.com/openclaw/openclaw/issues/42026 does not replace this focused gateway/node auth request.

## What This Changes

Request an opt-in Kubernetes ServiceAccount/TokenReview authentication path so autoscaled Kubernetes worker pods can pair with a gateway without interactive device pairing.

## Best Possible Solution

Keep this as a focused Kubernetes worker-auth design item until maintainers approve an opt-in ServiceAccount Trust contract or explicitly fold it into a narrower distributed-runtime auth plan.

Not applicable as a bug reproduction; this is a new Kubernetes identity/authentication feature request. Source, docs, related-item, and release inspection give high-confidence negative capability evidence that current OpenClaw lacks this path.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. A focused opt-in design across config/schema, protocol payloads, node-host credential handling, fail-closed TokenReview validation, tests, docs, and real Kubernetes proof is safer than reopening the old broad PR wholesale.
