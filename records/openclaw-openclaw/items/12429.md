## Summary

This issue should stay open. Current main and the latest release still have only shared token/password/trusted-proxy/device-token gateway authentication and a single-pod Kubernetes starter path; there is no Kubernetes ServiceAccount Trust or TokenReview pairing path for headless autoscaled worker nodes.

## What This Changes

The issue requests an opt-in Kubernetes ServiceAccount Trust authentication and deployment path so autoscaled worker pods can pair with a gateway without interactive PIN/device approval.

## Best Possible Solution

Keep this issue open until maintainers choose a secure opt-in worker identity design and split any approved work into focused auth/protocol/config/docs/deployment changes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested config, protocol, and handshake surfaces for Kubernetes ServiceAccount TokenReview auth.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Kubernetes ServiceAccount Trust is plausible, but the best maintainable solution needs maintainer agreement on the runtime/auth boundary, allowlisting model, TokenReview validation rules, and deployment ownership.
