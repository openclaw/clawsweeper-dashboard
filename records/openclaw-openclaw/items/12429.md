## Summary

Keep open. Current main still provides only the simpler Kubernetes/Kustomize deployment plus existing token/password/trusted-proxy/device-token paths; it does not implement the requested Kubernetes ServiceAccount Trust or TokenReview pairing bypass, and the linked implementation PR was closed unmerged while the distributed runtime RFC remains a broader architecture item rather than a replacement for this auth request.

## What This Changes

The issue requests an opt-in Kubernetes ServiceAccount Trust authentication path so autoscaled worker pods can connect to a gateway without interactive device pairing.

## Best Possible Solution

Keep this open until maintainers either approve a focused, fail-closed Kubernetes ServiceAccount Trust design or explicitly fold that auth contract into a narrower distributed-runtime/Kubernetes architecture item.
