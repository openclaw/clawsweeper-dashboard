## Summary

The request remains unimplemented and distinct: current main supports a hardened single-pod Kubernetes deployment plus token/password/device enrollment, while the closed prior PR was explicitly split away from the simpler Kubernetes baseline and the active distributed-runtime RFC does not yet define this workload-identity contract.

## What This Changes

Requests Kubernetes workload-identity authentication so independently autoscaled worker or node pods can securely connect to the gateway without interactive pairing.

## Best Possible Solution

Define a generic workload-identity contract at the distributed-runtime and gateway boundary, then implement Kubernetes TokenReview as an opt-in fail-closed backend with explicit audience and identity allowlists, projected short-lived tokens, TLS and CA handling, pairing-bypass tests, and focused deployment documentation.
