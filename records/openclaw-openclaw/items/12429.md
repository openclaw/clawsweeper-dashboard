## Summary

Keep open. Current main and the latest shipped Kubernetes baseline still do not provide Kubernetes ServiceAccount Trust, TokenReview validation, or a pairing-skip path for autoscaled worker/node pods, and the only linked implementation was closed unmerged in favor of a simpler Kubernetes install/docs baseline.

## What This Changes

The issue requests an opt-in Kubernetes ServiceAccount TokenReview authentication path so autoscaled worker/node pods can connect to a gateway without interactive device pairing.

## Best Possible Solution

Keep this open until maintainers approve a focused, fail-closed Kubernetes ServiceAccount Trust design or explicitly fold that auth contract into a narrower distributed-runtime/Kubernetes architecture item.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection gives high-confidence negative capability evidence that current main lacks the requested ServiceAccount TokenReview auth and pairing-skip path.

## Solution Assessment

Unclear. Kubernetes TokenReview is a plausible mechanism, but the best OpenClaw solution needs maintainer product and security decisions around runtime identity, allowlisting, pairing bypass semantics, and whether Helm/distributed-worker deployment belongs in this issue or a broader runtime item.
