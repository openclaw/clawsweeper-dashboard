## Summary

Keep this issue open: current main adds a hardened single-pod Kubernetes deployment and bounded device-bootstrap enrollment, but it still lacks workload-identity authentication for independently autoscaled node pods; the closed implementation PR was not merged, and the distributed-runtime RFC does not yet own this specific authentication contract.

## What This Changes

Add secure headless authentication for dynamically autoscaled Kubernetes worker nodes, using ServiceAccount identity, with supporting manifests and Helm deployment assets.

## Best Possible Solution

First approve a generic machine-enrollment or workload-identity contract for distributed runtimes, then implement Kubernetes TokenReview as a fail-closed adapter and land deployment assets separately after real multi-pod upgrade and autoscaling proof.

## Reproduction Assessment

Not applicable. This requests a new orchestration and workload-authentication capability rather than reporting broken behavior under an existing documented contract.

## Solution Assessment

No, the original all-in-one Kubernetes-specific implementation is not the best maintainable shape. The generic distributed-runtime identity boundary should be decided first, with Kubernetes validation and deployment packaging following as focused changes.

## Security Review

Status: needs_attention

Summary: The proposed pairing bypass is security-sensitive and requires an explicitly approved, fail-closed workload-identity design before implementation.

Concerns:

- **[high] Bind TokenReview identity to narrow authorization:**
  - body: Successful token authentication alone must not grant an unrestricted node pairing bypass; the design needs explicit audience, namespace, ServiceAccount, role, scope, and device-binding policy with deny-by-default configuration.
  - confidence: 0.96
- **[medium] Avoid a Kubernetes-only parallel auth contract:**
  - body: Adding a Kubernetes token directly to the handshake without a generic workload-identity boundary could create competing credential semantics and make future distributed-runtime authorization harder to audit.
  - confidence: 0.86
