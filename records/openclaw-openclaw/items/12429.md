## Summary

Current main partially addresses headless node enrollment with CIDR-based auto-approval and now includes a single-pod Kubernetes deployment, but it still has no Kubernetes ServiceAccount TokenReview credential or workload-identity authorization path. The issue remains useful, although the previously proposed broad core-plus-Helm solution should not be adopted without a maintainer decision on the generic distributed-node identity boundary and security model.

## What This Changes

Request an opt-in Kubernetes ServiceAccount TokenReview authentication path so autoscaled worker pods can connect to the gateway without interactive pairing.

## Best Possible Solution

First decide whether documented `gateway.nodes.pairing.autoApproveCidrs` plus existing gateway/bootstrap credentials is the supported trusted-cluster model; if stronger workload identity is required, define a generic node-authenticator contract and implement Kubernetes TokenReview as a fail-closed backend with explicit audience, namespace, service-account, and CA allowlists.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken established behavior. Source inspection gives high confidence that current main has CIDR auto-approval but no ServiceAccount TokenReview path.

## Solution Assessment

No, the closed broad Kubernetes-specific implementation is not yet established as the best permanent solution. Maintainers should first choose between the existing trusted-CIDR model and a generic workload-identity authenticator that can safely own Kubernetes TokenReview.
