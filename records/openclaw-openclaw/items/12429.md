## Summary

Keep open. Current main still does not implement the requested Kubernetes ServiceAccount TokenReview trust path for autoscaled headless worker/node pods, and the linked implementation PR is closed unmerged while the distributed-runtime RFC does not replace this specific auth requirement.

## What This Changes

The issue requests opt-in Kubernetes ServiceAccount Trust authentication plus deployment assets so autoscaled Kubernetes worker pods can authenticate to the gateway without interactive pairing.

## Best Possible Solution

Keep the issue open until maintainers design and implement a security-reviewed, opt-in Kubernetes ServiceAccount Trust path or explicitly fold this exact authentication requirement into a narrower canonical distributed-runtime Kubernetes item.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence source-level verification path: current config, schema, protocol auth payloads, node-host runtime, and Kubernetes manifests do not carry or validate a ServiceAccount TokenReview credential.

## Solution Assessment

Unclear. Kubernetes TokenReview is a plausible direction for headless autoscaling, but the best maintainable solution needs maintainer/security design because it changes gateway trust, node admission, and pairing-bypass semantics.
