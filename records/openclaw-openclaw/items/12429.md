## Summary

Keep open. Current main still does not implement the requested Kubernetes ServiceAccount Trust / TokenReview authentication path; it only has shared token/password/device/bootstrap/trusted-proxy auth plus an opt-in trusted-CIDR node pairing shortcut, which is not the same K8s identity model.

## What This Changes

Request to add opt-in Kubernetes ServiceAccount-based authentication so autoscaled headless worker or node pods can connect without interactive pairing.

## Best Possible Solution

Keep this open until maintainers approve a concrete core design for opt-in Kubernetes worker/node identity, or explicitly fold the same SA Trust requirement into a narrower distributed-runtime follow-up.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence current main lacks the requested mode because config, protocol, gateway client, node-host, and Kubernetes manifests do not expose ServiceAccount TokenReview auth.

## Solution Assessment

Unclear pending maintainer security and product design. Kubernetes SA Trust is plausible, but the maintainable solution needs reviewed gateway auth, protocol, pairing-bypass policy, node-host credential handling, TokenReview validation, tests, docs, and deployment changes rather than a small automated patch.
