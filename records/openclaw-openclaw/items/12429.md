## Summary

Keep open. Current main still lacks the requested Kubernetes ServiceAccount Trust / TokenReview authentication path for headless autoscaled worker or node pods, and the linked implementation PR is closed unmerged while the broader distributed-runtime RFC does not specifically resolve this auth requirement.

## What This Changes

Add opt-in Kubernetes ServiceAccount Trust authentication so autoscaled Kubernetes worker/node pods can connect to a gateway without interactive PIN pairing, plus deployment manifests or Helm support.

## Best Possible Solution

Keep this open until maintainers approve a concrete core design for opt-in Kubernetes worker/runtime authentication, or explicitly fold the same SA Trust requirement into a narrower distributed-runtime follow-up.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested mode because config, protocol, gateway client, node-host, and Kubernetes manifests only expose existing token/password/device/trusted-proxy paths.

## Solution Assessment

Unclear pending maintainer design review. The SA Trust direction is plausible, but the best solution needs reviewed core auth/protocol changes with fail-closed TokenReview validation rather than a small automated patch.
