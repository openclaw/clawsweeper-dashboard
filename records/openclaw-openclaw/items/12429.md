## Summary

Keep this issue open. Current main still has token/password/device/bootstrap/trusted-proxy gateway auth and a minimal single-pod Kubernetes deployment, but it does not implement the requested Kubernetes ServiceAccount TokenReview trust path for autoscaled node or worker pods; the linked implementation PR is closed unmerged and the distributed runtime RFC is broader rather than a replacement.

## What This Changes

The issue requests an opt-in Kubernetes ServiceAccount Trust authentication path so autoscaled Kubernetes worker pods can authenticate to the gateway without interactive PIN/device pairing.

## Best Possible Solution

Keep the issue open until maintainers either design and implement a security-reviewed, opt-in Kubernetes ServiceAccount Trust path in the gateway/node protocol or fold this exact auth requirement into a narrower canonical distributed-runtime/Kubernetes deployment item.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence source-level verification path: current config, protocol, node-host, and Kubernetes deployment files do not expose or accept a ServiceAccount TokenReview credential path.

## Solution Assessment

Unclear. Kubernetes SA TokenReview is a plausible approach for headless autoscaling, but because it changes gateway trust and pairing semantics, the best maintainable solution needs maintainer/security design rather than an automated direct port of the closed PR.
