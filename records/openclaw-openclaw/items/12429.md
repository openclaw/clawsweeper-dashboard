## Summary

The requested autoscaled-worker authentication remains absent on current main. OpenClaw now includes a single-pod Kubernetes/Kustomize deployment and a bounded setup-code bootstrap flow, but neither provides reusable Kubernetes workload identity for headless worker replicas; the issue should remain open for an explicit security and distributed-runtime product decision.

## What This Changes

Add an opt-in Kubernetes ServiceAccount TokenReview authentication path so autoscaled headless worker nodes can connect without interactive pairing, with Kubernetes deployment manifests and Helm packaging.

## Best Possible Solution

Define a generic, security-reviewed workload enrollment and identity seam for distributed nodes, then implement Kubernetes ServiceAccount TokenReview as an opt-in adapter with explicit audience, namespace, service-account, TLS, and pairing-skip policy; handle Helm or deployment packaging as a separately owned deliverable.

## Reproduction Assessment

Not applicable: this is a request for a new headless workload-identity capability. Current-main source inspection does provide high-confidence evidence that the capability is missing.

## Solution Assessment

No, the bundled Kubernetes-specific implementation is not yet proven to be the best permanent shape. The narrower maintainable direction is a generic distributed-node identity contract with a Kubernetes TokenReview adapter and separately scoped deployment assets.
