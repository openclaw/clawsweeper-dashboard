## Summary

Current main still lacks Kubernetes ServiceAccount or TokenReview workload authentication. Trusted-CIDR node auto-approval can remove manual enrollment on explicitly trusted private networks, but it authenticates network origin rather than pod identity and does not clearly solve multi-tenant clusters or ephemeral device lifecycle; neither the closed implementation PR nor the broader distributed-runtime RFC makes this request obsolete.

## What This Changes

Add a secure Kubernetes-native workload identity path so dynamically autoscaled headless worker nodes can enroll without interactive pairing.

## Best Possible Solution

Define a generic, fail-closed workload-attestation and ephemeral node lifecycle contract within the canonical device pairing flow, then implement Kubernetes TokenReview as one adapter with explicit audience, namespace, ServiceAccount, TLS/CA, rotation, revocation, and cleanup policy; handle deployment packaging as a separate focused deliverable.

## Reproduction Assessment

Not applicable: this is a feature request, and current source provides a high-confidence way to verify that Kubernetes workload identity is absent while trusted-CIDR enrollment is only a partial alternative.

## Solution Assessment

No. The original all-in-one `k8s-sa-trust` plus Helm implementation is not the best permanent shape; the identity contract and ephemeral lifecycle should be generic and security-reviewed before adding the Kubernetes adapter and separate deployment assets.

## Risks / Open Questions

- Trusted-CIDR auto-approval can be broader than Kubernetes ServiceAccount identity in shared or multi-tenant cluster networks.
- Disposable replicas may create durable device identities and pairing records without a defined lease, rotation, revocation, or cleanup lifecycle.
- A standalone Kubernetes auth mode could become a parallel source of authentication policy unless it reuses the canonical device identity, scope, and pairing authorization model.
- Bundling workload authentication, distributed runtime behavior, raw manifests, and a Helm chart into one implementation would cross several ownership and security review boundaries.
