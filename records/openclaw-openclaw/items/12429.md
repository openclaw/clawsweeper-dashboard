## Summary

Keep open. Current main still lacks the requested Kubernetes ServiceAccount Trust / TokenReview authentication path for autoscaled headless worker or node pods, and the related implementation PR is closed unmerged while the broader distributed-runtime RFC does not define this specific auth requirement.

## What This Changes

Request an opt-in Kubernetes ServiceAccount Trust / TokenReview auth path and deployment assets so autoscaled Kubernetes worker or node pods can connect without interactive pairing.

## Best Possible Solution

Keep this open until maintainers approve a concrete core design for opt-in Kubernetes worker/runtime authentication, or explicitly fold the same SA Trust requirement into a narrower distributed-runtime follow-up.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested mode because config, protocol, gateway client, node-host, and Kubernetes manifests only expose token/password/device/trusted-proxy paths.

## Solution Assessment

Unclear pending maintainer security and product design. Kubernetes SA Trust is plausible, but the maintainable solution needs reviewed gateway auth, protocol, pairing, node-host credential, TokenReview validation, docs, and deployment changes rather than a small automated patch.
