## Summary

Keep open: the optimization is plausible, but the current patch leaves wildcard exclusion inconsistent on forced message and heartbeat tool builds, and the PR still needs inspectable real behavior proof before merge.

## What This Changes

The PR adds an early return in Codex app-server dynamic tool building for `codexDynamicToolsExclude: ["*"]` and a regression test asserting the tool factory is skipped.

## Best Possible Solution

Make `*` semantics explicit and central: either document and implement wildcard exclude-all in the shared dynamic-tool filter with narrow forced-tool exceptions, or keep excludes name-only and remove the wildcard optimization.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: combine `codexDynamicToolsExclude: ["*"]` with message-tool-only delivery or heartbeat registration, and the new skip is bypassed while the existing filter treats `*` as a literal name. I did not run tests because this review is read-only.

## Solution Assessment

No. The early skip only handles the non-forced path; the maintainable fix is to put wildcard semantics in the central filter or decide not to support wildcard excludes at all.
