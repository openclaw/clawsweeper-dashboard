## Summary

Keep open. Current main still has the same plausible SwiftUI layout path where iOS composer height/focus changes can participate in message-list layout, and the related closed PR did not land or directly prove this composer-height freeze fixed.

## What This Changes

The issue reports an iOS/iPad UI freeze when the chat text input expands vertically, likely from composer inset or padding changes causing expensive message-list relayout.

## Best Possible Solution

Stabilize the shared SwiftUI chat layout so composer height, focus, and keyboard changes do not reflow long message lists, then prove the result on iOS/iPad with before-and-after responsiveness or profiling evidence.

## Reproduction Assessment

No, not high-confidence. Source inspection shows current main and v2026.5.22 still have a variable-height iOS composer composed alongside a scrollPosition-driven LazyVStack, but no simulator/device recording or profiler trace confirms the freeze path.

## Solution Assessment

Unclear. The fix likely belongs in shared SwiftUI chat layout, but profiling should choose the narrowest maintainable approach rather than guessing between composer isolation, geometry stabilization, scroll deferral, or row invalidation reduction.
