## Summary

Keep open. Current `main` still matches the reported iOS SwiftUI layout risk: the scrollable message list and composer are vertically coupled, the iOS composer still uses a bounded variable-height `TextEditor`, and I did not find a current-main, changelog, or related-PR change that isolates composer-height changes from message-list layout work.

## What This Changes

The issue reports brief iOS/iPad freezes when the multiline chat composer changes vertical size, likely because composer inset or padding changes force expensive message-list relayout.

## Best Possible Solution

Stabilize the shared SwiftUI chat layout so composer height changes do not force expensive message-list layout work, then verify with a long transcript on iPhone/iPad and before/after profiling evidence.

## Reproduction Assessment

Unclear. The best reproduction path is to run the iOS app with a long transcript, type enough text for multiline composer growth, and profile main-thread stalls or SwiftUI layout invalidation during the height changes.

## Solution Assessment

Unclear. The fix belongs in the shared SwiftUI chat UI, but profiling should choose the narrowest maintainable strategy, such as isolating the composer, reserving stable bottom space, deferring scroll updates, or reducing row invalidation.
