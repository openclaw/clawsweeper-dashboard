## Summary

Keep open. Current main still has the plausible SwiftUI layout coupling described in the report: the message list and composer are vertical siblings, the list claims flexible height, and the iOS composer remains a bounded `TextEditor` whose height/focus state can change during typing. I did not find a current-main source, test, changelog, release, or related-PR signal that isolates composer-height changes from chat-list layout work.

## What This Changes

The issue reports brief iOS/iPad app freezes when the multiline text input bar grows, likely from composer bottom-space changes triggering expensive message-list relayout.

## Best Possible Solution

Stabilize the shared SwiftUI chat layout so composer-height changes do not invalidate long message-list layout work, then verify on iPhone/iPad with a long transcript and profiling or responsiveness evidence.

## Reproduction Assessment

Unclear. The report gives a plausible path: run the iOS/iPad app with a long transcript, focus the composer, type enough text for multiline growth, and observe/profile main-thread stalls, but this review did not have device or simulator proof.

## Solution Assessment

Unclear. The fix likely belongs in the shared SwiftUI chat UI, but profiling should choose the narrowest approach, such as isolating the composer, reserving/stabilizing bottom space, deferring scroll updates, or reducing row invalidation.
