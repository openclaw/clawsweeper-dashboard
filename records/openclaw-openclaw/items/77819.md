## Summary

Keep open: current main still lacks a complete path for reset archives to appear in Sessions/WebChat history, even though part of the image-rendering surface has existing fixes. The remaining archive behavior needs an additive session-history/API and UI decision rather than cleanup closure.

## What This Changes

The issue asks WebChat/Control UI to keep reset archived transcripts discoverable and preserve image attachment display or placeholders after reload/reset.

## Best Possible Solution

Add a clear archived-transcript browsing path with stable identifiers, timestamps, and read-only history loading, then ensure history projection keeps or placeholders attachment-bearing messages even when text is empty or media access is unavailable.

## Reproduction Assessment

Yes, from source inspection for the archive half: reset archives the old JSONL, sessions.list only enumerates live sessions.json rows, and the UI does not send an archive-aware parameter. I did not establish a live browser reproduction in this read-only pass.

## Solution Assessment

Unclear as an implementation path: the desired end state is valid, but archived reset transcripts need an additive protocol and UI design before a narrow fix is safe. The image-only history projection can be handled as a focused regression once the archive browsing boundary is chosen.
