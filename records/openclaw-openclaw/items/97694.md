## Summary

Keep this PR open, but it is not merge-ready: the streaming JSONL direction is the right layer, yet the patch still stores provider-supplied output IDs that were never requested, lacks live Gemini proof or a maintainer waiver, and introduces a fail-closed per-record cap that needs explicit maintainer acceptance.

## What This Changes

The PR changes Gemini embedding batch output downloads from whole-body `res.text()` parsing to streaming JSONL parsing with a 16 MiB per-line cap and loopback HTTP regression tests.

## Best Possible Solution

Merge only after the streamed parser bounds accepted records to the requested batch, live Gemini proof or an explicit maintainer waiver exists, and maintainers accept or adjust the 16 MiB output-record cap.

## Reproduction Assessment

## Solution Assessment

No, not as submitted. Streaming JSONL is the right layer, but the implementation should reject or ignore output records outside the current batch before storing them and still needs live proof or a maintainer waiver.
