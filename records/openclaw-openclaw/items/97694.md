## Summary

Keep this PR open, but it is not merge-ready: the reported OOM vector is real, while the submitted fix can reject valid large Gemini batch result files and still has only test-style proof.

## What This Changes

The PR changes Gemini embedding batch output-file downloads from `res.text()` to the shared capped text-response reader and adds regression tests for oversized download bodies.

## Best Possible Solution

Prevent OOM by parsing the Gemini output JSONL incrementally, or by using an explicitly justified output-file size strategy that preserves provider-valid large batches, then add real setup proof.

## Reproduction Assessment

Yes from source inspection: current main calls `res.text()` on the successful Gemini `:download` response and then parses the full JSONL string. I did not run a live Gemini reproduction because this review was read-only.

## Solution Assessment

No: the PR is a plausible mitigation, but it applies a generic 16 MiB metadata/text-response cap to the normal successful output artifact. The better fix is streaming JSONL parsing or an output-file-specific bound aligned with valid batch sizes.
