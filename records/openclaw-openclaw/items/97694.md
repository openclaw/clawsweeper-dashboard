## Summary

Keep open: the PR is a useful Gemini batch-output streaming fix, but it is not merge-ready because the new streamed path still stores every provider-supplied output key and can grow memory without a total record bound; live Gemini/OpenClaw proof or maintainer waiver for the external provider path is also still missing.

## What This Changes

The PR changes Gemini embedding batch output handling from whole-file text buffering to streamed JSONL parsing with a 16 MiB per-record guard and loopback regression tests.

## Best Possible Solution

Merge only after the streamed parser bounds accepted output records to the requested batch, keeps the per-record byte guard, and maintainers either get live Gemini proof or explicitly waive it while accepting the record-cap failure mode.

## Reproduction Assessment

Yes, source-reproducible: current main calls `res.text()` on the successful Gemini batch output download and parses the full JSONL string afterward. I did not run a live Gemini reproduction in this read-only review.

## Solution Assessment

No. Streaming JSONL is the right layer for a finite batch output file, but this implementation still stores provider output records without checking that they belong to the current batch; the safer fix is to bound accepted records by `remaining` and then prove the real provider path or get a maintainer waiver.
