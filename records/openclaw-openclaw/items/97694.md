## Summary

Keep open; the central OOM hardening is still necessary, but this PR is not merge-ready because provider-controlled unknown output IDs can still grow memory and the external Gemini path still lacks live proof or an explicit maintainer waiver.

## What This Changes

The PR changes Gemini embedding batch output parsing from whole-body text buffering to streamed JSONL parsing with a per-record byte cap and loopback HTTP regression tests.

## Best Possible Solution

Merge only after the streamed parser bounds accepted records to the requested batch, live Gemini proof or an explicit maintainer waiver exists, and maintainers accept or adjust the 16 MiB output-record cap.

## Reproduction Assessment

Yes. Current main has a clear source reproduction path: successful Gemini batch output is downloaded with `res.text()` and only parsed afterward; I did not run live Gemini in this read-only review.

## Solution Assessment

No, not as submitted. Streaming JSONL is the right layer, but the implementation should reject or ignore output records outside the current batch before storing them and still needs live proof or a maintainer waiver.
