## Summary

Keep this PR open but do not merge as-is: it fixes a real unbounded-read path, yet the chosen 16 MiB text-response cap would reject valid successful Gemini batch output files and CI also shows a diff-caused lint failure.

## What This Changes

The PR replaces the unbounded Gemini batch output-file `res.text()` read with the shared capped provider text reader and adds regression tests around over-cap downloads.

## Best Possible Solution

Prevent OOM on Gemini batch output by processing the JSONL download incrementally or by using an output-file-specific limit aligned with provider-valid batch sizes, then add real setup proof.

## Reproduction Assessment

Yes from source inspection: current main calls `res.text()` on the successful Gemini `:download` response, so an overlarge streamed body is buffered before parsing. I did not run a local repro because this review is read-only and CI/test commands can create artifacts.

## Solution Assessment

No: reusing the generic 16 MiB provider text-response cap is only a plausible mitigation because the capped body is the successful batch output data. A streaming JSONL parser or output-specific size strategy would prevent OOM without rejecting provider-valid large jobs at the metadata-response limit.

- **[P1] Avoid rejecting valid batch output files at 16 MiB:** `extensions/google/embedding-batch.ts:224`
  - body: This applies the shared provider text-response cap to the successful Gemini batch output file. Use block-bodied executors that call `resolve` or `reject` without returning the server method result.
