## Summary

Keep open: current main and v2026.5.22 still reject Feishu/Lark plain group-chat `thread=true` subagent binding, while static group bindings, DM binding, and topic binding are adjacent but do not restore the requested plain-group behavior. The related Feishu group-subagents and plugin-threading-policy issues overlap but do not cleanly supersede this narrower binding-policy decision.

## What This Changes

The issue requests restoring Feishu/Lark plain group-chat `sessions_spawn({ thread: true, mode: "session" })` binding so each group can keep a dedicated subagent persona.

## Best Possible Solution

Make an explicit Feishu plugin policy for plain group current-conversation binding, then either support it with delivery and ambiguity tests or document static group bindings and topic-only thread binding as the supported path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Feishu requester with `to: "chat:..."`, `threadRequested: true`, and no topic/thread id resolves to null and emits the reported direct-message-or-topic error.

## Solution Assessment

Unclear. Restoring plain group binding is plausible, but maintainers first need to choose whether the binding key is the group id, sender-scoped id, topic-scoped id, static group binding, or intentionally unsupported.
