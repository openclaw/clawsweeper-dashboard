## Summary

Keep open: current main still intentionally rejects Feishu plain group-chat `thread=true` subagent binding, and the related Feishu group-subagents issue is broader rather than a clean duplicate for this specific binding policy.

## What This Changes

The issue requests restoring Feishu/Lark plain group-chat `thread=true` subagent binding so different group chats can bind to dedicated subagent sessions.

## Best Possible Solution

Make an explicit Feishu plugin policy, then either support plain-group current-conversation subagent binding with delivery and ambiguity tests or document static group bindings and broader group-subagent design as the supported path.

## Reproduction Assessment

Yes. Source inspection shows a Feishu `to: "chat:..."` request with `threadRequested` and no thread id returns null and emits the reported direct-message-or-topic error, and the current test suite asserts that unsupported path.

## Solution Assessment

Unclear. Adding plain-group binding is plausible, but maintainers first need to choose whether the binding key is the group id, sender-scoped id, topic-scoped id, static group binding, or intentionally unsupported.
