## Summary

Keep open. Current main still rejects Feishu plain group `sessions_spawn({ thread: true, mode: "session" })` current-conversation binding; the code and tests only support Feishu DMs or topic/thread conversations, and inbound runtime binding resolution is skipped for non-topic groups. The closed related item #44823 covered Feishu ACP topic bindings, while open #37967 is broader Feishu group subagent work rather than a full duplicate.

## What This Changes

Request restoring Feishu/Lark plain group-chat `sessions_spawn` thread/session current-conversation binding so group-specific subagents can remain attached to business groups.

## Best Possible Solution

Define the Feishu product policy for whole-group current-conversation subagent binding. If supported, implement it inside the Feishu plugin by allowing explicit plain-group binding only when channel policy permits it, routing follow-up group messages to the bound session, preserving completion delivery to the originating group, and adding focused regression coverage for spawn, routing, delivery, and unbind behavior.

## Reproduction Assessment

Yes. The current unit test at `extensions/feishu/src/subagent-hooks.test.ts:488` reproduces the reported path by spawning from `to: "chat:oc_group_chat"` with `threadRequested: true` and expecting the same error that the attachment reports.

## Solution Assessment

Unclear. A code fix can be narrow, but the best direction depends on maintainer product judgment about whole-group runtime binding versus documented static group `bindings`, plus the related trust and routing semantics.
