## Summary

Keep open: current main and the latest v2026.5.22 release still reject Feishu/Lark plain group-chat `thread=true` subagent binding, while adjacent Feishu group-subagent and channel-threading requests do not cleanly supersede this narrower binding-policy decision.

## What This Changes

Request to restore `sessions_spawn` current-conversation subagent binding with `thread: true` for plain Feishu/Lark group chats so separate business groups can use distinct agent personas.

## Best Possible Solution

Make an explicit Feishu plugin policy, then either support plain-group current-conversation subagent binding with delivery and ambiguity tests or document static group bindings and topic-only thread binding as the supported path.

## Reproduction Assessment

Yes, at source level. A Feishu `to: "chat:..."` request with `threadRequested` and no thread id returns null, emits the reported error, and is asserted by the current test suite.

## Solution Assessment

Unclear. Restoring plain-group binding is plausible, but maintainers first need to choose whether the binding key is the group id, sender-scoped id, topic-scoped id, static group binding, or intentionally unsupported.
