## Summary

Keep open: current main and v2026.6.8 still reject Feishu/Lark plain group-chat `thread=true` subagent binding. The remaining work is a maintainer product/routing decision because group-level, sender-scoped, topic-scoped, and static configured bindings can conflict.

## What This Changes

The issue requests restoring Feishu/Lark plain group-chat `sessions_spawn({ thread: true })` binding so different business groups can bind to dedicated subagent personas.

## Best Possible Solution

Define the Feishu policy for plain group current-conversation subagent binding, then either support it end-to-end with inbound routing, delivery, ambiguity tests, and docs, or document static group bindings/topic-only binding as the supported path.

## Reproduction Assessment

Yes, at source level: current main rejects plain Feishu group `chat:` targets for `threadRequested` subagent binding, and the test suite asserts that unsupported path.

## Solution Assessment

Unclear. Restoring group-level binding is plausible, but maintainers need to choose group-level, sender-scoped, topic-scoped, or static configured binding semantics before code changes.
