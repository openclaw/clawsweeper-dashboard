## Summary

Keep open: current main and v2026.6.6 still do not provide full Feishu/Lark plain group-chat `thread=true` subagent binding. The remaining gap is a product/routing decision because group-level, sender-scoped, topic-scoped, and configured binding semantics can conflict.

## What This Changes

Restore Feishu/Lark `thread=true` subagent binding for plain group chats so different group chats can bind to different subagent personas.

## Best Possible Solution

Define the Feishu plugin policy for plain group current-conversation subagent binding, then either support it with inbound routing, delivery, ambiguity tests, and docs, or document static group bindings/topic-only binding as the supported path.

## Reproduction Assessment

Yes, at source level. A Feishu requester with `to: "chat:..."`, `threadRequested: true`, and no topic/thread id still hits the unsupported path, and plain group inbound routing does not apply runtime conversation bindings.

## Solution Assessment

Unclear. Restoring plain group binding is plausible, but maintainers need to choose the binding semantics before code changes because group, sender-scoped, topic-scoped, static binding, and delivery behavior overlap.
