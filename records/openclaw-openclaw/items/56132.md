## Summary

Keep open: current main and v2026.6.8 have partially improved Feishu/Lark `thread: true` subagent binding by moving spawn-time binding into core, but plain group follow-up routing still skips runtime conversation bindings unless the group is topic-scoped. The remaining work is a maintainer product decision because group-level, sender-scoped, topic-scoped, and static binding semantics can conflict.

## What This Changes

The issue asks to restore Feishu/Lark `sessions_spawn(..., thread: true)` current-conversation subagent binding for plain group chats.

## Best Possible Solution

Define the Feishu policy for plain group current-conversation subagent binding, then align spawn binding, inbound runtime routing, ambiguity tests, docs, and upgrade expectations to that policy.

## Reproduction Assessment

Yes, at source level: current main gates Feishu runtime conversation binding resolution away from default plain groups, while docs only promise `--thread here` for DMs and thread messages. I did not run a live Feishu/Lark tenant scenario in this read-only review.

## Solution Assessment

Unclear: a small code change could enable group-level binding, but it would bake in product semantics before maintainers choose between group-level, sender-scoped, topic-scoped, static binding, or unsupported behavior.
