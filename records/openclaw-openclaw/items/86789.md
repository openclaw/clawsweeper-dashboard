## Summary

This issue is still necessary because current main still has the eager save-side preserve-key call path, and an open linked PR explicitly owns the proposed fix with closing syntax. The issue should stay open until that PR lands or maintainers choose a different session-store maintenance fix.

## What This Changes

The issue reports that session-store saves eagerly collect maintenance preserve keys and trigger subagent registry snapshot reads even when stale-prune, cap, or disk-budget maintenance does not need provider state.

## Best Possible Solution

Keep this issue paired with the linked PR and land a narrow session-store change that defers preserve-provider collection until stale pruning, cap enforcement, or disk-budget entry removal actually needs preserve keys, with regression coverage for both skipped-provider and preserved-stale-prune behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main calls preserve-key providers in `saveSessionStoreUnlocked`, and the registered subagent provider reaches the disk-aware registry snapshot path. I did not run a live profiler replay in this read-only review.

## Solution Assessment

Yes, deferring preserve-key collection until the maintenance branch actually needs those keys is the narrow maintainable direction; the linked PR still needs normal maintainer review and CI before the issue can close.
