## Summary

Keep open. Current main and v2026.5.2 still keep the Slack Socket Mode retry cap in place, and open PR #73162 remains the canonical closing PR for the narrow cap removal; current gateway restart recovery mitigates provider exits but does not remove the Slack-local cap or address reconnect-window delivery loss.

## What This Changes

Slack Socket Mode can stop answering after long-running reconnect instability, and the report asks OpenClaw to recover instead of silently losing Slack delivery.

## Best Possible Solution

Land #73162 or an equivalent Slack-owned change if maintainers want Socket Mode retries to be unlimited, then split reconnect-window delivery loss into a narrower follow-up if it still reproduces.

## Reproduction Assessment

Yes for the Slack-local failure path: force repeated recoverable Socket Mode disconnects or start failures and the twelfth retry exits the provider in current source. The exact days-long silent outage is less direct on current main because gateway channel auto-restart and health-monitor logic now add an outer recovery loop.

## Solution Assessment

Unclear. Removing the Slack-local cap is the narrowest fix for the retry-exhaustion path, but maintainers should account for the newer outer restart mitigation and the separate reconnect-window delivery failures described in the comments.
