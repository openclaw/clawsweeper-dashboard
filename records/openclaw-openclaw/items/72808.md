## Summary

Keep open. Current main still contains the hardcoded Slack Socket Mode reconnect cap, the latest release still ships that cap, and open PR #73162 remains the linked implementation candidate for this issue.

## What This Changes

Slack Socket Mode should keep replying instead of silently stopping after repeated long-running ping/pong or reconnect failures.

## Best Possible Solution

Land #73162 or an equivalent Slack plugin fix that makes recoverable Socket Mode retries unlimited while preserving fail-fast auth errors, then validate transient not-ready delivery loss separately if it still reproduces.

## Reproduction Assessment

Yes. Source inspection shows current main reaches a hardcoded twelve-attempt Slack Socket Mode cap and throws; the issue discussion also includes current-main reproduction notes and matching live logs.

## Solution Assessment

Yes for the permanent dead-Slack state. Removing the recoverable reconnect cap is the narrowest maintainable fix, while transient reconnect-window delivery loss should remain a separate validation or follow-up path.
