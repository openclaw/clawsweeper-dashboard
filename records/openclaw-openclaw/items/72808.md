## Summary

Keep open. Current main and the latest tagged release still contain the hardcoded finite Slack Socket Mode reconnect cap, the provider still throws after that cap, and the issue has a linked open fix PR plus maintainer discussion asking to keep this open while the upstream Slack SDK pong-stall path is validated.

## What This Changes

Slack Socket Mode should keep replying after long-running connection instability instead of silently becoming non-responsive while the gateway and web interface remain usable.

## Best Possible Solution

Keep this issue open until #73162 or a replacement lands with Slack-owned coverage for repeated disconnect/start-failure recovery, then validate a Slack SDK update for the pong-stall path. If reconnect-window delivery loss remains, split that into a narrower follow-up.

## Reproduction Assessment

Yes. Repeated Socket Mode disconnect or start-failure events deterministically increment the hardcoded 12-attempt policy until the provider throws, and the discussion includes current-main reproduction plus live logs for the same failure family.

## Solution Assessment

Unclear whether the open uncap is the complete solution. It is the narrowest fix for permanent reconnect exhaustion, but maintainer discussion points to upstream Slack SDK pong-stall behavior and transient delivery failures that may need validation or follow-up.
