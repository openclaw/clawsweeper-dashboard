## Summary

Keep this issue open because current main and the latest stable release still have the finite Slack Socket Mode reconnect cap, and an open linked PR already targets the source-backed root cause. The discussion also has fresh field reports showing the problem recurred after the earlier upstream Slack SDK pong fix, so this is not ready for cleanup closure.

## What This Changes

Slack Socket Mode can stop responding after repeated reconnects while the gateway and web UI remain usable, leaving Slack users without replies until restart or repair.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73162 or an equivalent maintainer replacement, then close this issue with the merged commit and release or main-only provenance once recoverable Slack Socket Mode disconnects retry indefinitely.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: force twelve recoverable Slack Socket Mode disconnects or start failures in a row and the provider throws from the maxAttempts guard while the rest of the gateway can remain alive.

## Solution Assessment

Yes. Disabling the finite reconnect cap for recoverable Slack socket failures is the narrowest maintainable fix because the loop already treats zero as unlimited and still fast-fails permanent Slack auth errors.
