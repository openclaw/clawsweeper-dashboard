## Summary

Keep open. Current main still has the hardcoded Slack Socket Mode reconnect cap, the provider still throws when that cap is reached, and the linked implementation PR that references this issue with closing syntax is still open rather than merged.

## What This Changes

Slack Socket Mode can silently stop responding after a long-running connection experiences repeated disconnects or pong-timeout instability while the gateway and web interface remain usable.

## Best Possible Solution

Keep this issue open until #73162 or a replacement lands with Slack-owned coverage for repeated disconnect/start-failure recovery, then validate the `@slack/socket-mode` 2.0.7 pong-stall fix against the reported logs. If reconnect-window delivery loss remains, split that into a narrower follow-up.

## Reproduction Assessment

Yes. Repeated Slack Socket Mode `disconnected` or `unable_to_socket_mode_start` events deterministically increment the hardcoded 12-attempt policy until the provider throws, and the discussion includes current-main reproduction plus live logs for the same failure family.

## Solution Assessment

Unclear whether the open uncap is the complete solution. It is the narrowest maintainable fix for permanent reconnect exhaustion, but the upstream Slack SDK pong-stall behavior and transient delivery failures still need validation or follow-up.
