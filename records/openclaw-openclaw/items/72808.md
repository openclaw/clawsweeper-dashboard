## Summary

Keep open. Current main still contains the Slack Socket Mode finite reconnect policy and throws after 12 consecutive disconnect/start-failure attempts, while #73162 is an open closing PR for the exact permanent Slack silence failure and the discussion still calls out upstream pong-stall and reconnect-window delivery behavior that need validation.

## What This Changes

The issue reports that Slack stopped receiving OpenClaw replies after several days while the web interface still worked, leaving the Slack bot apparently online but silently non-responsive.

## Best Possible Solution

Land #73162 or a replacement with Slack-owned coverage for repeated disconnect/start-failure recovery, then validate current `@slack/socket-mode` pong-stall behavior. If messages or slash commands are still dropped during reconnect windows after the permanent exhaustion fix, split that into a narrower follow-up.

## Reproduction Assessment

Yes. A focused fake Socket Mode disconnect/start-failure loop can drive the current provider through 12 consecutive attempts and hit the throw path, and the issue discussion includes current-main reproduction plus live logs for the same failure family.

## Solution Assessment

Unclear whether the open cap removal is the complete fix. It is the narrowest maintainable repair for permanent reconnect exhaustion, but Slack SDK pong-stall behavior and transient reconnect-window delivery failures still need maintainer validation.
