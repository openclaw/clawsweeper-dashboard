## Summary

Keep open. Current main and the latest tagged release still contain the finite Slack Socket Mode reconnect cap, and open PR #73162 is the active unmerged implementation candidate linked to this issue, so the issue should not be closed before that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to stop silently losing Slack delivery after a long-lived Socket Mode connection disconnects or stalls while the gateway and web chat remain usable.

## Best Possible Solution

Keep this issue open until #73162 or a replacement lands with Slack-owned coverage for repeated disconnect/start-failure recovery, then validate the Slack SDK ping/pong stall path; if reconnect-window message loss remains, split that into a narrower follow-up.

## Reproduction Assessment

Yes. Repeated Socket Mode disconnect or start-failure events deterministically increment the hardcoded 12-attempt policy until the provider throws, and the discussion includes current-main reproduction plus live logs for the same failure family.

## Solution Assessment

Unclear. Removing the reconnect cap is the narrowest fix for the permanent silent-dead state, but the comments also point to upstream Slack SDK behavior and transient delivery failures that need validation before declaring the whole issue solved.
