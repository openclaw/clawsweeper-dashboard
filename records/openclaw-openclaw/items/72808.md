## Summary

Keep open. Current main and v2026.5.7 still retain the Slack Socket Mode 12-attempt fallback cap, the provider still throws when that cap is reached, and an open closing PR targets this exact failure mode after the reporter saw it recur post-Slack-SDK pong fix.

## What This Changes

The issue asks OpenClaw to keep Slack Socket Mode responsive after long-running disconnect/reconnect failures instead of silently leaving Slack non-responsive while the gateway stays up.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/73162 so recoverable Slack Socket Mode outages cannot permanently exhaust the channel, then track reconnect-window delivery loss separately if it still reproduces.

## Reproduction Assessment

Yes. Current main has a source-visible path where repeated Slack disconnect or start-failure events hit the 12-attempt guard and throw, and the issue discussion includes matching `retry 1/12` field logs plus a recurrence after the Slack SDK pong fix.

## Solution Assessment

Yes. Keeping this issue open behind the linked reconnect-cap PR is the narrowest maintainable path; closing before that PR lands would lose the active implementation candidate.
