## Summary

Keep open. Current main and the latest v2026.5.2 tag still hard-cap Slack Socket Mode reconnects at 12 attempts, and open PR #73162 already uses closing syntax for the narrow cap-removal fix, so this issue should remain open until that PR or an equivalent change lands.

## What This Changes

Slack Socket Mode should keep replying after long-running reconnect churn instead of silently dropping Slack messages while the gateway and web chat remain usable.

## Best Possible Solution

Land #73162 or an equivalent Slack-owned change that makes recoverable Socket Mode retries unlimited while preserving fail-fast handling for permanent auth errors; track transient not-ready delivery loss separately if it still reproduces afterward.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path by forcing repeated recoverable Slack Socket Mode disconnects or start failures until the twelfth attempt, and the issue discussion includes matching current-main reproduction notes and live logs.

## Solution Assessment

Yes for the confirmed cap-exhaustion symptom. The narrowest maintainable fix is to remove the recoverable reconnect cap while keeping non-recoverable auth failures terminal; upstream pong-stall and transient not-ready drops should be validated as follow-up behavior.
