## Summary

Keep this PR open. Current main has already covered the narrow Mattermost DM single-session behavior and has separate websocket liveness work, but it still lacks the PR’s structured replied-post context, DM reply-reference fetching, explicit DM reply-target preservation, root typing `parent_id` compatibility, and Mattermost-DM typing TTL behavior; that makes the PR non-obsolete cleanup material, though it needs maintainer review because the branch now spans several behaviors.

## What This Changes

The PR changes Mattermost client, monitor, websocket, typing, and tests to keep DM replies in one direct session, fetch quoted DM post context into `ReplyToBody`/`ReplyToSender`, add websocket heartbeat handling, send root typing with `parent_id: ""`, and disable the generic typing TTL for Mattermost DMs.

## Best Possible Solution

Keep the shipped DM single-session behavior, then rebase or split this PR so the remaining Mattermost-owned pieces land as focused changes: structured DM reply context with direct-only post lookup, explicit reply-target handling that preserves current threading contracts, and separate websocket/typing updates that compose with current main’s liveness code and tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main routes a direct Mattermost post with `root_id` through `resolveMattermostEffectiveReplyToId` to `undefined`, and there is no Mattermost path that fetches the referenced post or populates `ReplyToBody`/`ReplyToSender`; live Mattermost replay was not run under the read-only constraint.

## Solution Assessment

No. The Mattermost owner module is the right boundary, but this single PR is no longer the best final shape because current main already solved part of the problem and the remaining reply-context, websocket, and typing changes should be rebased and likely split for safer review.
