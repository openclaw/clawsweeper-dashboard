## Summary

Keep open: current main still sends `timeoutSeconds` as an ACP `session/set_config_option` control before the turn starts, and the bundled acpx wrapper only suppresses unsupported timeout controls for Codex, not OpenCode. The report is a narrow ACP runtime-control bug with clear source evidence and a focused repair path.

## What This Changes

ACP OpenCode child sessions should not fail when the adapter rejects the optional timeout `session/set_config_option` sent from OpenClaw runtime options.

## Best Possible Solution

Patch ACP runtime-control application so unsupported adapter-level timeout config is skipped or tolerated for OpenCode-like adapters while preserving Gateway turn timeout enforcement and strict failures for essential controls.

## Reproduction Assessment

Yes at source level: a mock ACP runtime that advertises `session/set_config_option` but rejects `timeout` will fail before `runTurn` on current main. I did not run a live OpenCode binary, so this is not a full live current-main reproduction.

## Solution Assessment

Yes, a narrow fix in ACP runtime-control handling is the best direction: treat `timeoutSeconds` as an optional adapter hint and rely on the existing Gateway turn timeout. Avoid broadly swallowing model/thinking/permission config failures, because those are user-visible controls that should still report unsupported backends.
