## Summary

Keep open: current main still has the reported ACP timeout control path, and the acpx wrapper only drops timeout for Codex while forwarding it for non-Codex agents such as OpenCode. I found no duplicate or open paired fix PR, so this remains a narrow ACP runtime-control bug.

## What This Changes

Make ACP OpenCode child sessions tolerate unsupported timeout session config while OpenClaw continues enforcing its own turn timeout.

## Best Possible Solution

Teach the ACP control plane to treat the canonical timeout runtime option as optional backend config when unsupported or rejected, while preserving OpenClaw's local turn timeout and strict errors for explicit backend-only options.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: spawn an ACP OpenCode session with a positive run timeout, then current main persists `timeoutSeconds`, applies ACP controls before the turn, and forwards `timeout` for non-Codex acpx agents. I did not run the live OpenCode harness in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is in OpenClaw's ACP runtime-control application, not a broad product change: skip or tolerate unsupported `timeout` config only where OpenClaw can still enforce the turn timeout locally, and add regression coverage for advertised-missing and adapter-rejected timeout cases.
