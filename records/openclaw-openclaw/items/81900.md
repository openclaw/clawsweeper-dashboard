## Summary

Keep open: this targets a real Ollama `num_ctx` regression, but the patch restores fallback from the resolved runtime model window rather than only from an explicitly configured user budget. That can send `options.num_ctx` for discovered/default Ollama windows and reintroduce the resource/latency behavior current main intentionally avoided.

## What This Changes

The PR changes Ollama native `/api/chat` request construction, runtime tests, and provider docs so `options.num_ctx` falls back from `contextWindow` or `maxTokens` when `params.num_ctx` is absent.

## Best Possible Solution

Preserve compatibility for users who explicitly configured an Ollama budget while avoiding implicit native `num_ctx` from discovered/default catalog metadata, likely through an explicit migration or a carried provenance flag.

## Reproduction Assessment

Yes from source with high confidence: current main only forwards explicit `params.num_ctx`, while the linked issue and PR exercise a config where `contextWindow` is present and `params.num_ctx` is absent. I did not run a live Ollama daemon in this read-only review.

## Solution Assessment

No for the current patch as written: restoring fallback from the final runtime model window is broader than the reported explicit-config regression. A narrower solution should distinguish explicit user budget from discovered/default model metadata or migrate existing configs to `params.num_ctx`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not use discovered windows as native num_ctx:** `extensions/ollama/src/stream.ts:312-316`
  - body: This fallback runs on the final `ProviderRuntimeModel`, where `contextWindow` can come from Ollama discovery or OpenClaw's default model definition, not from user config. That means native requests can start sending `options.num_ctx` for unconfigured models again, contradicting the new docs' "no model budget" promise and reopening the latency/resource behavior that current main intentionally fixed; only an explicit user budget or migrated `params.num_ctx` should drive this path.
