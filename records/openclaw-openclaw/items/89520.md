Keep open: current main and the latest release still lack an automatic load/startup sweep for stale session-store atomic-write temp files, and the remaining fix is already owned by the open linked PR with closing syntax.

## What This Changes

The issue asks OpenClaw to automatically reclaim orphaned `sessions.json.<pid>.<uuid>.tmp` files left behind by interrupted atomic session-store writes.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90503, or an equivalent load/startup sweep that reuses the existing session-store temp matcher and preserves fresh in-flight writes.

## Reproduction Assessment

Yes, source-level: session writes create identifiable temp artifacts, current cleanup can reclaim stale ones only through explicit cleanup/disk-budget paths, and load/startup does not run that sweep. I did not run a kill-based filesystem reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is an automatic load/startup sweep using the existing temp filename contract and five-minute freshness guard. The open linked PR targets that shape, so this issue should stay open until that PR lands or maintainers choose an equivalent implementation.
