## Summary

Keep open. Current main still has the reported write-before-validation behavior for both `plugins enable` and `plugins disable`, and open PR #73554 explicitly targets this issue, so the issue should stay open until that PR is reviewed, landed, replaced, or closed.

## What This Changes

The issue asks for `openclaw plugins enable <id>` and `plugins disable <id>` to reject nonexistent plugin IDs before mutating `plugins.entries` config.

## Best Possible Solution

Add a pre-write discovered-plugin lookup in the `plugins enable` and `plugins disable` handlers, preserve canonical id/channel normalization for valid ids, add focused no-write regression tests, and leave cleanup of already-stale entries to doctor/fix unless maintainers intentionally expand scope.

## Reproduction Assessment

Yes. I did not run the mutating CLI because this review is read-only, but the current source path directly reproduces the report: the handlers pass the raw id into config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The requested registry validation before mutation is the narrowest maintainable fix; the active linked PR is the right surface to review, land, or replace before closing this issue.
