## Summary

Keep open. Current main still has the reported write-before-validation behavior for both plugin policy commands, and open PR #73554 is already the active closing implementation candidate.

## What This Changes

The issue asks `plugins enable` and `plugins disable` to reject nonexistent plugin ids before mutating `plugins.entries` config.

## Best Possible Solution

Add a registry-backed pre-write guard in `plugins enable` and `plugins disable`, preserve warn-only validation for legacy stale config, and cover unknown ids with no-write regression tests.

## Reproduction Assessment

Yes. I did not run the mutating CLI in this read-only review, but the current source path directly reproduces the bug: an unknown id flows into the config mutation helper before any discovered-plugin lookup.

## Solution Assessment

Yes. Pre-write validation against the same discovered plugin snapshot used by list/inspect is the narrow maintainable fix, with canonical channel-id normalization preserved for valid ids.
