## Summary

Keep open. Current main still has the reported write-before-validation behavior in `plugins enable` and `plugins disable`, and open PR #73554 is already the active closing implementation candidate.

## What This Changes

Request that `plugins enable/disable <id>` reject nonexistent plugin ids before writing `plugins.entries.<id>` and exit non-zero.

## Best Possible Solution

Add a registry-backed pre-write guard in `plugins enable` and `plugins disable`, preserve warn-only handling for old stale config, and land the existing closing PR or an equivalent focused patch with no-write regression tests.

## Reproduction Assessment

Yes. The read-only source path reproduces it: enable/disable accept the raw id, call config mutation/write before any discovered-plugin lookup, and stale-entry validation remains warn-only after persistence; I did not run the mutating CLI.

## Solution Assessment

Yes. Pre-write validation against the same discovered plugin snapshot used by list/inspect is the narrow maintainable fix, with canonical channel-id normalization preserved for valid ids.
