## Summary

Keep open. Current main still has the reported write-before-validation behavior in both plugin policy commands, and open PR #73554 is an active closing implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Request to make `openclaw plugins enable/disable <id>` reject undiscovered plugin IDs with a nonzero exit and no config mutation.

## Best Possible Solution

Land or replace PR #73554 with a narrow fix that resolves the requested id against the discovered plugin registry before mutation, preserves valid canonical/channel id normalization, and adds enable/disable regression tests proving unknown ids do not write config or refresh the registry.

## Reproduction Assessment

Yes. I did not run the mutating CLI in this read-only review, but current main source directly reproduces the report: both handlers pass the supplied id into config mutation helpers before any discovered-plugin lookup, and the helper creates `plugins.entries.<id>`.

## Solution Assessment

Yes. Pre-write registry validation in the `plugins enable` and `plugins disable` handlers is the narrowest maintainable fix; existing warning-only startup validation should remain resilient for stale entries created by older versions.
