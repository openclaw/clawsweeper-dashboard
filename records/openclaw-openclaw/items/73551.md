## Summary

Keep open. Current main still routes unknown plugin ids through the `plugins enable` and `plugins disable` config write paths before any discovered-registry validation, and open PR #73554 is already the paired implementation candidate that uses closing syntax for this bug.

## What This Changes

Validate `openclaw plugins enable/disable <id>` against discovered plugins before writing `plugins.entries` so nonexistent ids fail without mutating config.

## Best Possible Solution

Land or replace #73554 with registry-backed pre-write guards for both policy commands plus no-write regression tests, while keeping startup stale-entry handling warn-only for resilience.

## Reproduction Assessment

Yes. The issue gives deterministic CLI steps, and current-main source inspection shows the raw id flowing into config mutation helpers and `replaceConfigFile` before any registry lookup or not-found exit.

## Solution Assessment

Yes. A pre-write check against the discovered plugin snapshot used by plugin list/inspect is the narrow maintainable fix; stale-entry cleanup can remain in doctor/fix paths.
