## Summary

Keep open. Current main still has the reported write-before-validation behavior for both `plugins enable` and `plugins disable`, and open PR #73554 explicitly targets this issue with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests that `plugins enable` and `plugins disable` reject nonexistent plugin ids before mutating `plugins.entries`, exit non-zero, and leave config unchanged.

## Best Possible Solution

Add a pre-write plugin lookup in the CLI enable/disable handlers using the discovered plugin snapshot, preserve canonical id/channel normalization for valid ids, and add regression tests that assert missing ids exit 1 without config writes or registry refreshes.

## Reproduction Assessment

Yes. I did not run the mutating CLI because this review is read-only, but the current source path directly reproduces the report: the handlers pass the raw id into config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. Registry validation before mutation is the narrowest maintainable fix; review, land, or replace the open linked PR rather than closing this issue now.
