## Summary

Keep open. Current main still has the reported write-before-validation behavior for both `plugins enable` and `plugins disable`, and open PR #73554 explicitly closes this issue, so the right cleanup path is to review, land, rebase, or replace that PR rather than closing this issue now.

## What This Changes

The issue requests that `plugins enable <id>` and `plugins disable <id>` reject nonexistent plugin ids before mutating plugin config, exit nonzero, and leave `openclaw.json` unchanged.

## Best Possible Solution

Add a pre-write discovered-plugin lookup in the `plugins enable` and `plugins disable` command handlers, keep valid id normalization for built-in channels, add no-write regression tests in the plugin CLI policy suite, and leave cleanup of already-stale entries to the existing doctor/fix path unless maintainers intentionally expand scope.

## Reproduction Assessment

Yes. I did not run the mutating CLI in this read-only review, but the current source path directly reproduces the reported behavior: the handlers pass the raw id into config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. Registry validation before config mutation is the narrowest maintainable fix for this issue; the linked open PR is the active implementation candidate to review, land, or replace.
