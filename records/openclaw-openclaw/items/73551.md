## Summary

Keep open. Current main still has the reported write-before-validation behavior for both plugin policy commands, and open PR #73554 is already the active closing implementation candidate for https://github.com/openclaw/openclaw/issues/73551.

## What This Changes

The issue asks `openclaw plugins enable <id>` and `plugins disable <id>` to reject nonexistent plugin ids before mutating `plugins.entries`, exit non-zero, and leave config unchanged.

## Best Possible Solution

Land or replace PR #73554 with a narrow fix that resolves the requested id against the discovered plugin registry before mutation, preserves canonical/channel id normalization for valid ids, and adds enable/disable regression tests proving unknown ids do not write config or refresh the registry.

## Reproduction Assessment

Yes. I did not run the mutating CLI because this review is read-only, but current main source directly shows both handlers pass the raw id into config mutation helpers before any discovered-plugin lookup, and the helper creates `plugins.entries.<id>`.

## Solution Assessment

Yes. Pre-write registry validation in the `plugins enable` and `plugins disable` handlers is the narrowest maintainable fix; the existing warn-only stale-entry validation should remain for legacy config resilience.
