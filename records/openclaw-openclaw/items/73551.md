## Summary

Keep open. Current main still has the reported write-before-validation behavior for `plugins enable` and `plugins disable`, and open PR #73554 already references this issue with closing syntax, so maintainers should review, land, or replace that PR before closing.

## What This Changes

The issue asks for `openclaw plugins enable/disable <id>` to reject undiscovered plugin ids without writing stale `plugins.entries` config and with a non-zero exit.

## Best Possible Solution

Add a pre-write lookup in `plugins enable` and `plugins disable` using the discovered plugin registry snapshot, resolve valid aliases to the canonical plugin id, and exit 1 with a clear `Plugin not found` message before any config write or registry refresh on a miss.

## Reproduction Assessment

Yes. I did not run the CLI because this review is read-only, but the current source path directly reproduces the reported behavior: both commands pass the raw id into config mutation helpers before any registry lookup, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The narrow maintainable fix is to validate the operator-supplied id against the discovered plugin registry before mutation; PR #73554 appears to target that path and should be reviewed, landed, or replaced.
