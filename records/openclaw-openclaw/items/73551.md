## Summary

Keep open. Current main still has the reported write-before-validation behavior for both `plugins enable` and `plugins disable`, and open PR #73554 explicitly targets this issue with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for `openclaw plugins enable/disable <id>` to reject nonexistent plugin ids before writing `plugins.entries`, exit non-zero, and leave config unchanged.

## Best Possible Solution

Land or replace PR #73554 with a narrow fix that resolves the requested id against the discovered plugin snapshot before mutation, preserves canonical id/channel normalization for valid ids, and adds regression tests that prove unknown enable/disable requests do not write config or refresh the registry.

## Reproduction Assessment

Yes. I did not run the mutating CLI in this read-only review, but the current source path directly reproduces the report: both command handlers pass the raw id into config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. Pre-write registry validation in the `plugins enable` and `plugins disable` handlers is the narrowest maintainable fix; existing stale-entry cleanup can remain in doctor/fix or a separate follow-up.
