## Summary

Keep open. Current main still mutates plugin config before checking whether the requested id is discovered, and open PR #73554 explicitly targets this issue with closing syntax, so this should stay open until that PR lands or is replaced.

## What This Changes

Request that `openclaw plugins enable/disable <id>` reject undiscovered plugin ids with a non-zero exit before modifying plugin config.

## Best Possible Solution

Land or replace PR #73554 with a narrow fix that resolves the requested plugin id from the discovered registry before any config mutation, keeps valid built-in channel normalization working, and adds regression coverage proving unknown enable/disable requests exit non-zero without writing config or refreshing the registry.

## Reproduction Assessment

Yes. I did not run the mutating CLI in this read-only review, but the current source path directly reproduces the report: enable/disable pass the raw id into config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. Pre-write registry validation in the `plugins enable` and `plugins disable` handlers is the narrowest maintainable fix; cleanup of stale entries already created by older versions can stay in doctor/fix or a separate follow-up.
