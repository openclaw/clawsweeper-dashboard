## Summary

Keep open. Current main still accepts unknown plugin ids in the `plugins enable` and `plugins disable` write paths, and open PR #73554 is already the paired implementation candidate using closing syntax for this issue.

## What This Changes

Require `plugins enable <id>` and `plugins disable <id>` to reject nonexistent plugin ids without writing stale `plugins.entries.<id>` config.

## Best Possible Solution

Land or replace #73554 with registry-backed pre-write guards for both policy commands plus no-write regression tests, leaving stale-entry cleanup to doctor/fix paths.

## Reproduction Assessment

Yes. The issue gives deterministic CLI steps, and current main shows the requested id flowing into the config mutation helpers and `replaceConfigFile` before any discovered-registry lookup.

## Solution Assessment

Yes. A pre-write check against the discovered plugin snapshot used by plugin list/inspect is the narrow maintainable fix, while stale-entry validation can remain warn-only for startup resilience.
