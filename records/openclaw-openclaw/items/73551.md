## Summary

Keep open. Current main still has the reported write-before-validation behavior for both `plugins enable` and `plugins disable`, and open PR #73554 explicitly targets this issue with closing syntax, so maintainers should review, land, or replace that PR before closing this issue.

## What This Changes

The issue requests that `plugins enable <id>` and `plugins disable <id>` reject nonexistent plugin ids without writing stale `plugins.entries.<id>` config entries.

## Best Possible Solution

Add a pre-write plugin lookup in the `plugins enable` and `plugins disable` handlers using the discovered plugin snapshot, preserve canonical id/channel normalization for valid ids, and leave cleanup of already-stale entries to doctor/fix unless maintainers intentionally expand scope.

## Reproduction Assessment

Yes. I did not run the mutating CLI in this read-only review, but the current source path directly reproduces the reported behavior: the command handlers pass the raw id into config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The requested registry validation before mutation is the narrowest maintainable fix; the open linked PR is the right surface to review, land, or replace rather than closing this issue now.
