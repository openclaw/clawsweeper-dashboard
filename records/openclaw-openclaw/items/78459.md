## Summary

Keep open: this PR targets a valid current-main bug, but the patch drops `fastMode` instead of passing the existing resolved fast-mode state, which would make hidden commitment extraction ignore `agents.list[].fastModeDefault`; the supplied proof is also source-structure output rather than a real post-fix extraction run.

## What This Changes

The PR removes `fastMode: true` from the commitment extraction embedded-run payload, adds a regression assertion that the payload omits `fastMode`, and adds a changelog fix entry for the MiniMax highspeed-plan failure.

## Best Possible Solution

Pass `resolveFastModeState({ cfg, provider: modelRef.provider, model: modelRef.model, agentId: first.agentId }).enabled` into commitment extraction, cover default-false plus configured-fast cases, and merge after real extraction proof is added.

## Reproduction Assessment

Yes, at source level. Current main always passes `fastMode: true` for hidden commitment extraction, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true; the linked bug report supplies matching live logs.

## Solution Assessment

No. Removing the key fixes the default non-highspeed symptom, but the maintainable fix is to pass the existing fast-mode resolver output so per-agent and per-model fast-mode defaults remain consistent.
