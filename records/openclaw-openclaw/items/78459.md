## What This Changes

The PR removes `fastMode: true` from the hidden commitment-extraction embedded-agent request and adds a regression assertion that the request omits `fastMode`.

## Best Possible Solution

Commitment extraction should remove the unconditional true default, pass the shared resolved fast-mode value only when a non-default source configures it, and then land with exact-head redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main hardcodes `fastMode: true` in hidden commitment extraction, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true; I did not run a live MiniMax OAuth request.

## Solution Assessment

No. Removing the field fixes the forced-highspeed default, but it is not the best fix because it fails to preserve the shared `fastModeDefault` configuration path; the narrow safer fix is to pass the resolved non-default fast-mode state.
