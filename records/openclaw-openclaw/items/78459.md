## Summary

Keep open: the PR addresses a real source-backed MiniMax commitments bug, but the patch is incomplete because it drops the fast-mode field instead of using the existing resolver, and the after-fix proof is still only a mocked unit-test run.

## What This Changes

The branch removes `fastMode: true` from hidden commitment extraction, asserts the payload omits `fastMode`, and adds a changelog fix entry.

## Best Possible Solution

Commitment extraction should compute fast mode with `resolveFastModeState` for the selected extractor model, cover default-false and agent-default cases in regression tests, and include after-fix real MiniMax evidence before merge.

## Reproduction Assessment

Yes, source-backed: current main passes `fastMode: true` in hidden commitment extraction, and the MiniMax provider wrapper rewrites M2.7 to highspeed when that extra param is true. I did not run a live non-highspeed MiniMax account.

## Solution Assessment

No. Removing the field fixes the default-false symptom, but the maintainable fix is to pass the existing resolved fast-mode state so configured per-agent defaults keep working.
