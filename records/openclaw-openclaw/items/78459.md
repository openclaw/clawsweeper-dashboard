## Summary

Keep open: this PR addresses a real source-reproducible bug, but the patch drops `fastMode` entirely instead of passing the resolved fast-mode state, so configured per-agent fast mode would be skipped; the supplied proof is also structural source inspection rather than an after-fix extraction run.

## What This Changes

The PR removes `fastMode: true` from the commitment extraction embedded-agent payload, adds a regression assertion that the payload omits `fastMode`, and adds a changelog entry.

## Best Possible Solution

Pass `resolveFastModeState(...).enabled` for the selected extractor model, cover default-false and configured-fast cases, then merge after redacted real extraction proof is added.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main passes `fastMode: true`, the provider hook maps `extraParams.fastMode` to the MiniMax highspeed wrapper, and the linked bug report includes matching live HTTP 500 logs.

## Solution Assessment

No. Dropping the key fixes the default non-highspeed MiniMax symptom, but the maintainable fix is to pass the existing resolver output so configured-fast agents keep their documented behavior.
