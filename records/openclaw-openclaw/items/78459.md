## Summary

Keep open: the PR addresses a real source-reproducible MiniMax commitments bug, but the current patch fixes it by omitting `fastMode` entirely, which bypasses configured fast-mode opt-ins that normal agent runs preserve.

## What This Changes

The PR removes the hardcoded `fastMode: true` from hidden commitment extraction and adds a regression assertion that the embedded-agent request omits `fastMode`.

## Best Possible Solution

Keep the MiniMax highspeed regression fix, but resolve fast mode with the shared selected-model/agent resolver and pass that explicit value, with coverage for default-off and configured opt-in extraction plus exact-head proof.

## Reproduction Assessment

Yes, source-reproducible: current main sends `fastMode: true` from hidden commitment extraction, and the MiniMax stream wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when fast mode is true. I did not run a live MiniMax OAuth account in this read-only review.

## Solution Assessment

No. Removing the hardcoded true is the right direction, but the PR should pass the shared resolved fast-mode state instead of omitting the field and losing per-agent or per-model opt-ins.
