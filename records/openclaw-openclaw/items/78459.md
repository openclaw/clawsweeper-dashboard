## Summary

Keep open: current main still has the MiniMax-triggering hardcoded fast-mode path, but this PR is not merge-ready because dropping the field entirely would bypass documented configured fast-mode opt-ins for hidden commitment extraction.

## What This Changes

The PR removes a hardcoded commitment-extraction `fastMode: true` request field and adds a regression assertion that the embedded request omits `fastMode`.

## Best Possible Solution

Replace the hardcoded true with the shared resolved fast-mode boolean for the selected extractor model, then cover default-off MiniMax behavior plus explicit per-agent and per-model opt-ins.

## Reproduction Assessment

Yes, from source: current main passes `fastMode: true` for hidden commitment extraction, and the MiniMax wrapper rewrites MiniMax-M2.7 to MiniMax-M2.7-highspeed when fast mode is true; the linked issue supplies matching live HTTP 500 logs.

## Solution Assessment

No, not as currently shaped. Removing the hardcode is the right direction, but the PR should forward `resolveFastModeState(...).enabled` so configured opt-ins remain intact.
