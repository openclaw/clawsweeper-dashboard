## Summary

Keep open: the reported MiniMax commitment-extraction bug is real and this PR targets the right call site, but the patch currently drops configured per-agent fast-mode opt-ins and the supplied proof is structural output from an older head rather than exact-head runtime or live MiniMax proof.

## What This Changes

The branch removes the hardcoded `fastMode: true` from hidden commitment extraction and adds a regression assertion that the embedded request omits `fastMode`.

## Best Possible Solution

Replace the hardcoded true with the shared resolved fast-mode boolean for the selected extractor model, then cover default-off MiniMax behavior plus explicit per-agent and per-model opt-ins.

## Reproduction Assessment

Yes, from source. Current main passes `fastMode: true` for hidden commitment extraction, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when fast mode is true; the reporter supplied matching live HTTP 500 logs.

## Solution Assessment

No, not as currently shaped. Removing the hardcode is the right direction, but the PR should forward `resolveFastModeState(...).enabled` so configured opt-ins remain intact.
