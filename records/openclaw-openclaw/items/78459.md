## What This Changes

The PR removes the hardcoded `fastMode: true` field from commitment extraction embedded-agent requests and adds a regression assertion that the request omits `fastMode`.

## Best Possible Solution

Replace the hardcoded true with the shared resolved fast-mode behavior for the selected extractor model, cover default-off plus explicit per-agent and per-model opt-ins, then add exact-head real behavior proof or a maintainer waiver.

## Reproduction Assessment

Yes. Source inspection shows current main passes `fastMode: true` for hidden commitment extraction, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true; the linked issue supplies matching live HTTP 500 logs.

## Solution Assessment

No, not as currently shaped. Removing the hardcoded true is the right direction, but the PR should preserve the shared resolved fast-mode contract instead of omitting the field entirely.
