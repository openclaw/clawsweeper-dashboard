## Summary

Keep open. The bug is source-backed on current main, but this PR drops `fastMode` entirely, which fixes the default-false path while skipping the documented per-agent `fastModeDefault` layer; the after-fix proof is also structural source inspection rather than a real commitment extraction run.

## What This Changes

The PR removes the hardcoded `fastMode: true` from hidden commitment extraction, asserts the embedded payload omits `fastMode`, and adds a changelog fix entry.

## Best Possible Solution

Commitment extraction should pass the resolved fast-mode boolean for the selected extractor model and agent, with regression coverage for default-false plus configured true cases, then merge after real runtime proof is supplied.

## Reproduction Assessment

Yes, source-backed: current main passes `fastMode: true` into hidden commitment extraction, and the MiniMax wrapper rewrites `MiniMax-M2.7` to highspeed when fast mode is true. The linked issue includes live failure logs, but I did not run a non-highspeed MiniMax account.

## Solution Assessment

No. Removing the field fixes the default-false symptom, but the narrower maintainable fix is to pass `resolveFastModeState({ cfg, provider: modelRef.provider, model: modelRef.model, agentId: first.agentId }).enabled` so commitments follow the documented resolver order.

## Review Findings
