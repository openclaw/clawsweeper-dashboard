## Summary

Keep this PR open: it targets a real source-backed MiniMax commitments bug, but the diff drops fast-mode resolution instead of preserving the documented per-agent fast-mode default path, and the supplied proof is structural terminal output rather than an after-fix runtime run.

## What This Changes

The PR removes `fastMode: true` from hidden commitment extraction, adds a regression assertion that the payload omits `fastMode`, and adds a changelog fix entry.

## Best Possible Solution

Commitment extraction should pass the same resolved fast-mode boolean used by normal agent runs for the selected extractor model, with regression coverage for default-false plus per-agent and per-model fast-mode defaults, then merge after real behavior proof is added.

## Reproduction Assessment

Yes, source-backed: current main passes `fastMode: true` into hidden commitment extraction, and the MiniMax wrapper rewrites M2.7 to highspeed only when `extraParams.fastMode` is true. The linked issue also includes live failure logs, but I did not run a MiniMax account.

## Solution Assessment

No. Removing the field fixes the default-false symptom, but the narrow maintainable fix is to pass `resolveFastModeState({ cfg, provider: modelRef.provider, model: modelRef.model, agentId: first.agentId }).enabled` so commitments follow the documented resolver order.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass the resolved fast-mode state:** `src/commitments/runtime.ts:243-247`
  - body: Dropping `fastMode` entirely skips `agents.list[].fastModeDefault`, because only `resolveFastModeState` reads that field; extra params merge `params`, not the per-agent default. An agent configured with `fastModeDefault: true` would keep normal turns fast but make hidden commitment extraction default-speed, contrary to the documented fast-mode order. Please pass the resolver's `.enabled` value for the selected extractor model and cover that case.
  - confidence: 0.87
