## Summary

Keep open. The PR targets a real source-reproducible MiniMax commitments bug, but dropping the payload key entirely would skip the documented per-agent fast-mode default path, and the supplied proof is structural source inspection rather than an after-fix extraction run.

## What This Changes

The PR removes the hardcoded `fastMode: true` from commitment extraction, adds a regression assertion that the embedded payload omits `fastMode`, and adds a changelog entry.

## Best Possible Solution

Commitment extraction should pass the existing resolved fast-mode state for the selected extractor model, cover default-false and configured-fast cases, then merge after redacted real extraction proof is added.

## Reproduction Assessment

Yes. Source inspection on current main shows commitments extraction always passes `fastMode: true`, and the MiniMax wrapper rewrites `MiniMax-M2.7` to highspeed when that flag is true; the linked issue includes matching live HTTP 500 logs.

## Solution Assessment

No. Dropping the key fixes the default non-highspeed symptom, but the maintainable fix is to pass `resolveFastModeState(...).enabled` so configured-fast agents keep their documented behavior.
