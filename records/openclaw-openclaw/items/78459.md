## Summary

Keep open: current main still has the source-backed MiniMax commitment-extraction bug, but this PR drops `fastMode` instead of passing the resolved fast-mode state and the supplied proof is static source inspection rather than a real post-fix extraction run.

## What This Changes

This PR removes `fastMode: true` from hidden commitment extraction, adds a unit assertion that the embedded payload omits `fastMode`, and adds a changelog fix entry for the MiniMax non-highspeed plan failure.

## Best Possible Solution

Pass `resolveFastModeState({ cfg, provider: modelRef.provider, model: modelRef.model, agentId: first.agentId }).enabled` into the commitment extraction embedded run, cover default-false plus configured-fast cases, and merge after real extraction proof is added.

## Reproduction Assessment

Yes. Source inspection shows current main hardcodes `fastMode: true` for hidden commitment extraction, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when fast mode is true; the linked issue also has live failure logs, though I did not run a MiniMax account.

## Solution Assessment

No. Removing the key fixes the default non-highspeed symptom, but the narrower maintainable fix is to pass the existing fast-mode resolver output so per-agent and per-model fast-mode defaults keep working consistently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass the resolved fast-mode state:** `src/commitments/runtime.ts:243-247`
  - body: Dropping `fastMode` entirely means hidden commitment extraction no longer follows `agents.list[].fastModeDefault`, because that field is only read by `resolveFastModeState`; the extra-params merge only sees `params`. A user with normal agent turns configured fast via `fastModeDefault: true` would silently get default-speed commitment extraction. Pass the resolver's `.enabled` for the selected extractor model and cover default-false plus configured-fast cases.
  - confidence: 0.87
