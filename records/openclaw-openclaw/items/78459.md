## Summary

Keep open: the PR targets a real source-backed bug, but the patch should preserve existing fast-mode resolution instead of relying on an undefined field, and the external PR still needs after-fix real behavior proof.

## What This Changes

The PR removes the hardcoded commitment-extraction `fastMode: true`, adds a runtime test assertion, and adds a changelog fix entry.

## Best Possible Solution

Use `resolveFastModeState` for the extractor run so the default stays off, explicit false wins, and existing agent/model fast-mode defaults are honored with regression coverage.

## Reproduction Assessment

Yes, source-backed: current main passes `fastMode: true` in hidden commitment extraction, and MiniMax rewrites M2.7 to highspeed when that extra param is true. I did not exercise a live non-highspeed MiniMax account.

## Solution Assessment

No, not quite: deleting the field fixes the default-false case but skips `agents.list[].fastModeDefault`, unlike normal and cron agent runs. The narrower maintainable fix is to pass the existing resolved fast-mode state.
