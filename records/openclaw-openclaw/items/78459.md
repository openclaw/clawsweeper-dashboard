## What This Changes

The PR removes the hardcoded commitment-extraction `fastMode: true` payload field and adds a regression assertion that the embedded-agent request omits `fastMode`.

## Best Possible Solution

Resolve commitment extraction fast mode through the shared resolver after selecting the commitment model, pass the resolved mode and auto cutoff into `runEmbeddedAgent`, and require current-head proof for default-off and configured opt-in behavior before merge.

## Reproduction Assessment

Yes, source-reproducible: current main sends `fastMode: true` from hidden commitment extraction, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when fast mode is true. I did not run a live MiniMax Coding Plan account in this read-only review.

## Solution Assessment

No. Removing forced `true` is directionally right, but the best fix should use the shared resolver so default-off MiniMax plans and explicit configured fast-mode opt-ins both behave as documented.
