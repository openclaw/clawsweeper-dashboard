## Summary

Keep open: the PR addresses a real MiniMax commitments bug, but the patch is not merge-ready because omitting `fastMode` bypasses the shared fast-mode resolver for hidden extraction and leaves explicit configured defaults under-proven.

## What This Changes

The PR removes the hardcoded `fastMode: true` commitment-extraction request field and adds a regression assertion that the embedded-agent payload omits `fastMode`.

## Best Possible Solution

Resolve commitment extraction fast mode through `resolveFastModeState`, pass the resolved mode and auto cutoff into `runEmbeddedAgent`, and require exact-head real behavior proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main hardcodes `fastMode: true`, and the MiniMax stream wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when fast mode is true; I did not run a live MiniMax account.

## Solution Assessment

No. Removing forced `true` is directionally right, but the best fix is to use the shared resolver so default-off and documented configured opt-ins both work.
