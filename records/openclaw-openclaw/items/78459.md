Keep open: the PR targets a real source-backed bug, but the current diff drops `fastMode` instead of passing the existing resolved fast-mode state, so per-agent defaults are not preserved. The external after-fix proof is also still mock-only.

## What This Changes

This PR removes `fastMode: true` from the hidden commitments extractor payload, adds an assertion that the payload omits `fastMode`, and adds a changelog fix entry.

## Best Possible Solution

Commitment extraction should pass the existing resolved fast-mode state for the selected extractor model, cover default-false and configured fast-mode cases, and merge after real MiniMax proof is added.

## Reproduction Assessment

Yes, source-backed: current main passes `fastMode: true` into hidden commitment extraction, and the MiniMax wrapper rewrites M2.7 to highspeed when that extra param is true. The linked issue includes live failure logs, though I did not run a MiniMax account.

## Solution Assessment

No. Removing the field fixes the default-false symptom, but the narrow maintainable fix is to pass `resolveFastModeState({ cfg, provider: modelRef.provider, model: modelRef.model, agentId: first.agentId }).enabled` and test the configured-default cases.
