## What This Changes

The issue asks doctor/session cleanup to stop reporting live transcript files as orphaned when `sessions.json` is a keyed object whose entries contain `sessionId` and `sessionFile`.

## Best Possible Solution

Add a reporter-shaped regression for keyed object stores with absolute `sessionFile` values, then patch only the specific resolver/reference path if that fixture fails.

## Reproduction Assessment

Unclear: the issue gives concrete observed output and a representative keyed store entry, but current and v2026.5.12 source already iterate `Object.entries(store)` and resolve `sessionFile`/`sessionId`. I did not run a filesystem repro because this review is read-only.

## Solution Assessment

Unclear: changing orphan cleanup before a failing fixture would be risky. The narrow maintainable approach is to reproduce the reported keyed-store shape first, then adjust the shared transcript reference resolver only if it actually misclassifies live files.
