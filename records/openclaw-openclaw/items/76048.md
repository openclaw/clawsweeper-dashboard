## Summary

Keep open. Current main still has a source-level path where Z.AI `reasoning_content` becomes a hidden thinking block, so a GLM-5 stream with empty `content` can finish with no visible assistant text.

## What This Changes

The issue asks OpenClaw to make Z.AI GLM-5 replies visible when the provider sends answer text in `reasoning_content` while `content` is empty.

## Best Possible Solution

Add a provider-aware Z.AI output normalization path that promotes fallback answer text from `reasoning_content` only when visible content is absent in disabled-thinking mode, while preserving opted-in thinking replay as hidden reasoning.

## Reproduction Assessment

Yes. A mocked OpenAI-completions stream chunk for `zai/glm-5.1` with empty `delta.content` and non-empty `delta.reasoning_content` currently produces a thinking block and no visible text block by source inspection.

## Solution Assessment

No, not exactly. The issue’s proposed always-route-Z.AI check is too broad; the maintainable fix should key off Z.AI plus disabled/off thinking or an explicit compat fallback so preserved thinking remains hidden.
