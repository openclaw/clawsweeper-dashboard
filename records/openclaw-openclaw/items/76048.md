## Summary

Keep open. Current main still routes Z.AI GLM-5 `reasoning_content` into hidden thinking blocks, while the directly related fix PR #76078 is closed unmerged.

## What This Changes

The issue asks OpenClaw to surface Z.AI GLM-5 OpenAI-compatible `reasoning_content` as visible assistant text when `content` is empty instead of treating it as hidden thinking.

## Best Possible Solution

Land a narrow transport fix that uses the existing Z.AI compat signal to classify GLM-5 `reasoning_content` as visible text, with regression coverage and no provider/model hardcoding.

## Reproduction Assessment

Yes, at source level: feed `processOpenAICompletionsStream` a Z.AI OpenAI-completions chunk with empty `delta.content` and non-empty `delta.reasoning_content`. Current main classifies that delta as `thinking`, routes it through `appendThinkingDelta`, and later visible extraction keeps only `text` blocks.

## Solution Assessment

Yes, the narrowest maintainable fix is to consume the existing `compat.thinkingFormat === "zai"` signal in the stream extractor or equivalent routing point. Hardcoding GLM model IDs or making all `reasoning_content` visible would be less safe.
