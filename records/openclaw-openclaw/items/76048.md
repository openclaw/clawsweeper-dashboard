## Summary

Keep open. Current main and v2026.5.3 still parse `reasoning_content` as hidden thinking and do not pass the existing Z.AI `thinkingFormat` compat signal into the stream extractor; the directly related PR #76078 is closed unmerged.

## What This Changes

The issue asks OpenClaw to surface Z.AI GLM-5 visible output when the provider returns empty `content` and non-empty `reasoning_content`.

## Best Possible Solution

Land a narrow transport fix that threads the existing Z.AI compat/request context into completions stream parsing, surfaces GLM-5 empty-content `reasoning_content` as assistant text, and adds regression/control coverage.

## Reproduction Assessment

Yes, at source level: a Z.AI completions stream chunk with empty `delta.content` and non-empty `delta.reasoning_content` is emitted as a thinking block, and later visible extraction keeps only text blocks. I did not run a live Z.AI endpoint trace.

## Solution Assessment

Yes, a narrow transport/compat fix is the right path. The implementation should preserve non-Z.AI reasoning semantics and avoid exposing true Z.AI thinking when thinking is intentionally enabled or preserved.
