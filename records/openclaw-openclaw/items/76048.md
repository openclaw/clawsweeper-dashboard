## Summary

Keep open. Current main still has the reported Z.AI stream-routing bug: `reasoning_content` is classified as hidden thinking, while the existing Z.AI compat signal is not used at that routing point; the directly related fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to surface Z.AI GLM-5 `reasoning_content` as visible assistant text instead of hidden thinking when `content` is empty.

## Best Possible Solution

Land a narrow transport fix that consumes the existing Z.AI compat signal when classifying completions `reasoning_content`, with regression coverage for Z.AI and a control case for other providers.

## Reproduction Assessment

Yes, at source level: feed the completions stream a Z.AI chunk with empty `delta.content` and non-empty `delta.reasoning_content`; current main classifies it as `thinking`, routes it through `appendThinkingDelta`, and later visible extraction keeps only `text` blocks. I did not run live Z.AI API calls or tests in this read-only review.

## Solution Assessment

Yes, using the existing `compat.thinkingFormat === "zai"` signal in the stream extractor is the narrow maintainable fix. The implementation should avoid hardcoded GLM IDs and keep non-Z.AI reasoning output hidden unless already classified as visible text.
