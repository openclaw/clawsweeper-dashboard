## Summary

This should stay open: current main still has a source-reproducible path where a preserved or explicit `OPENCLAW_WRAPPER` resolving to the generated Windows `gateway.cmd` is written back into that same task script, producing the reported recursive command.

## What This Changes

The issue reports that Windows `gateway install` or update can generate a self-recursive `gateway.cmd` wrapper instead of invoking the OpenClaw CLI entrypoint directly.

## Best Possible Solution

Add a focused Windows gateway install guard that rejects or clears wrapper paths resolving to the generated task script, while preserving legitimate external wrappers and regenerating `gateway.cmd` with the direct Node/Bun CLI entrypoint.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: preserve or set `OPENCLAW_WRAPPER` to the resolved Windows task script path, then the install plan writes `gateway.cmd gateway --port 18789` into `gateway.cmd`. I did not execute a live Windows install in this read-only review.

## Solution Assessment

No, current main is not yet the best solved state because it still allows the generated task script to become its own wrapper. The narrow maintainable fix is an install-plan or Windows Scheduled Task guard plus regression coverage for preserved wrapper env and legitimate external wrappers.
