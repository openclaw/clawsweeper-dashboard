## Summary

This should stay open: current main still lets a persisted or requested `OPENCLAW_WRAPPER` resolve to the generated Windows task script, and the task writer then serializes that wrapper path back into `gateway.cmd`, producing the reported self-recursive command. I did not find a duplicate, open fixing PR, or post-release change on the implicated files.

## What This Changes

The issue asks for the Windows gateway install/update flow to avoid generating a `gateway.cmd` wrapper that invokes itself instead of the OpenClaw CLI entrypoint.

## Best Possible Solution

Add a focused Windows gateway install guard that rejects or clears wrapper paths resolving to the generated task script, then regenerates `gateway.cmd` with the direct Node/Bun CLI entrypoint while preserving legitimate external wrappers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: preserve or pass `OPENCLAW_WRAPPER` as the resolved Windows task script path, then the install plan writes `gateway.cmd gateway --port ...` into that same script.

## Solution Assessment

No, current main is not the solved state because the self-wrapper guard is missing. The narrow maintainable repair is to detect the generated Windows task script path before wrapper passthrough and cover reinstall, update, and doctor repair inputs.

Current Windows gateway install/update can preserve or accept `OPENCLAW_WRAPPER` resolving to the generated task script (`gateway.cmd`), then write that same path into `gateway.cmd` as the command, causing recursion. Add a focused guard in the Windows gateway install planning/install path so wrapper paths equal to the resolved task script or hidden launcher are rejected or cleared before building `programArguments`, while preserving legitimate external wrappers such as Doppler wrappers.
