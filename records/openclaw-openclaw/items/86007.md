## Summary

This should stay open: current main still accepts and persists a gateway wrapper path, then writes that wrapper path directly into the Windows task script, so a wrapper path equal to the resolved gateway.cmd still produces the reported self-recursive command. I did not find a current-main fix, duplicate, or shipped release that resolves this exact Windows install/update failure.

## What This Changes

The issue reports that the Windows gateway install/update flow in OpenClaw 2026.5.22 generated a gateway.cmd wrapper that calls itself recursively instead of launching the Node CLI entrypoint directly.

## Best Possible Solution

Keep this open and add a focused Windows gateway install guard that rejects or clears a wrapper path resolving to the generated task script, then regenerates gateway.cmd with the direct Node/Bun CLI entrypoint.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: set or preserve OPENCLAW_WRAPPER to the resolved Windows task script path, then the install plan writes gateway.cmd as gateway.cmd gateway --port 18789. I did not execute a live Windows install in this read-only review.

## Solution Assessment

No, current main is not the best solved state because it still allows the task script to become its own wrapper. The maintainable fix is a narrow install-plan guard and regression coverage around the Windows task script path and persisted wrapper env.

Fix the Windows managed gateway install/update self-recursive wrapper bug reported in https://github.com/openclaw/openclaw/issues/86007. In current main, OPENCLAW_WRAPPER or a preserved service OPENCLAW_WRAPPER can resolve to the generated Windows task script path, causing src/daemon/schtasks.ts to write gateway.cmd with a final command like gateway.cmd gateway --port 18789. Add a focused guard so the gateway install plan rejects or clears wrappers that resolve to the Windows task script path (including OPENCLAW_TASK_SCRIPT/OPENCLAW_TASK_SCRIPT_NAME and preserved update/doctor environments), then falls back to the direct Node/Bun CLI entrypoint rather than self-wrapping. Keep legitimate external wrappers working, do not add a new config option, and add regression coverage for Windows gateway install/update planning plus task script generation. Add a changelog fix entry because this is user-visible.
