## Summary

Keep open: the report has credible production evidence and current main still has pending-final metadata paths where `attemptCount=1` and `lastError=null` are not durably tied to a confirmed Telegram adapter send.

## What This Changes

The issue reports a Telegram DM final reply that was generated and persisted as pending with one clean attempt, but no outbound `sendMessage` and no pending-state cleanup.

## Best Possible Solution

Use the existing durable outbound delivery queue as the source of truth for final-delivery attempts, recording sent or failed outcomes on the pending-final marker and clearing it only after a confirmed adapter send.

## Reproduction Assessment

Yes at source level, but not as a live Telegram repro: current code can write pending-final attempt metadata with `lastError=null` without that write being a durable Telegram adapter receipt. The reporter's 2026.5.18 logs supply the real-world no-send observation.

## Solution Assessment

Yes in direction, but the maintainable path is to reuse the existing durable delivery queue rather than add a separate parallel outbox. The fix should make pending-final session state reflect confirmed send, explicit failure, or still-queued delivery.
