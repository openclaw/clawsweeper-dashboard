## Summary

Keep open: the reported Telegram message-loss/session-state bug is credible on current main, and an open fix PR already references this issue for closure, so the issue should wait for that PR to merge or be rejected.

## What This Changes

Make Telegram final replies record confirmed delivery or a non-null pending-final failure instead of staying stuck with `attempt=1`, `lastError=null`, and no observed `sendMessage`.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84382; the desired end state is pending-final session state that reflects confirmed send, explicit failure, or still-pending delivery without a second parallel outbox design.

## Reproduction Assessment

Yes, at source level: current main can persist pending-final attempt metadata with `lastError=null` without that write being a durable Telegram adapter receipt. I did not run a live Telegram repro, but the reporter's logs provide credible production evidence for the no-send outcome.

## Solution Assessment

Yes, the best current path is the focused linked PR that changes final-delivery accounting and restart recovery rather than adding a separate retry worker in this issue. Broader durable outbox retry design should stay separate unless the narrow fix proves insufficient.
