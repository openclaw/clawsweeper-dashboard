## Summary

Keep open: current main still has a source-level pending-final delivery accounting gap, and https://github.com/openclaw/openclaw/pull/84382 is the open focused fix linked from the discussion. The issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to stop leaving Telegram final replies in pendingFinalDelivery with attempt metadata but no durable send success or non-null failure.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84382 or an equivalent narrow accounting fix that records sent, failed, or still-pending state from durable delivery proof.

## Reproduction Assessment

Yes at source level, but not as a live Telegram repro in this review. Current main has paths that can write pending-final attempt metadata with lastError=null without durable adapter send proof, and the reporter supplied production Telegram logs for the no-send observation.

## Solution Assessment

Yes, the linked PR is the best current path because it narrows the fix to no-proof final delivery accounting and restart recovery rather than adding a separate broad retry worker. Broader durable outbox design can remain separate follow-up work.
