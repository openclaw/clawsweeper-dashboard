## Summary

Keep open: current main and v2026.5.18 still have a Telegram final-delivery accounting gap where final text is persisted as pending before delivery, but the Telegram custom final path can finish without clearing the pending marker or recording a non-null no-send error. This is adjacent to the open heartbeat cleanup work, but it is not fully covered by that PR because this report is about normal Telegram DM final delivery.

## What This Changes

The issue asks OpenClaw to make Telegram final replies behave like a durable outbox, clearing pending state only after confirmed delivery and recording a failure when no send occurs.

## Best Possible Solution

Make Telegram final delivery update the existing pending-final contract: clear pending state only after a confirmed visible send or finalized preview, and record a non-null retryable failure/no-send reason when durable delivery exits without an adapter send.

## Reproduction Assessment

Yes, at source level: current main persists visible final text as pending and the Telegram durable final path can return handled_no_send or visible delivery without updating the pending-final session fields. I did not establish a live current-main Telegram reproduction for the exact production no-send incident.

## Solution Assessment

Yes for a narrow bug fix: update Telegram's existing final delivery accounting rather than adding a new retry worker or broad outbox state machine first. A larger durable intent store can remain the long-term lifecycle direction, but the immediate fix should use the existing pendingFinalDelivery fields safely.
