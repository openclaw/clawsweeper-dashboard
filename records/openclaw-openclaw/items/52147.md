## Summary

Keep open. Current main mitigates the original no-custom-config 600s case by defaulting agent runtime timeouts to 48 hours, but the reported misclassification still exists for any configured run timeout: non-compaction run timeouts still feed assistant profile rotation/model fallback as `timeout`, with no active-tool timeout cause.

## What This Changes

The issue requests that embedded-agent run timeouts that fire during long tool execution not be classified as LLM request timeouts or trigger model fallback.

## Best Possible Solution

Add phase-aware timeout state to the embedded runner, preferably a small `timeoutCause` or `timedOutDuringToolExecution` signal derived from active tool lifecycle state, and have assistant failover treat tool-phase run timeouts as run/tool timeouts instead of model failover signals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set a short `agents.defaults.timeoutSeconds`, run a long tool or `process(poll)` with a fallback chain, and observe that current failover policy still treats `timedOut && !timedOutDuringCompaction` as rotation/fallback; no live long-running repro was run because this review is read-only.

## Solution Assessment

Yes, the reporter's `timeoutCause` or `timedOutDuringToolExecution` direction is the narrow maintainable fix. Raising the default runtime timeout reduces frequency but does not solve configured shorter run deadlines or the misleading fallback classification.
