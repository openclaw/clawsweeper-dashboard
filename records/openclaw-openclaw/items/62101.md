## Summary

Current main supersedes this BlueBubbles-only PR by removing the bundled BlueBubbles channel and documenting migration to the supported iMessage plugin; the remaining branch diff targets deleted code and includes a shared allowlist transform that would regress numeric-ID channels.

## What This Changes

The PR prefixes all-digit BlueBubbles handles, changes the shared numeric allowFrom schema branch to emit +E.164 strings, and adds tests for those normalization paths.

## Best Possible Solution

Keep the BlueBubbles removal and iMessage migration path; any remaining phone allowlist normalization should stay in the iMessage plugin path, not in the shared allowFrom schema.

## Reproduction Assessment

No. Current main no longer contains the BlueBubbles runtime or `extensions/bluebubbles/src/targets.ts`, so the old BlueBubbles allowlist failure cannot be reproduced against main; the stale branch diff is still source-reviewable.

## Solution Assessment

No. The old branch is no longer the best fix because the supported product path removed BlueBubbles, and the shared schema transform would still be too broad for channels with numeric sender IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep phone coercion out of the shared allowlist schema:** `src/channels/plugins/config-schema.ts:20`
  - body: `AllowFromEntrySchema` is shared by plugins beyond BlueBubbles. Rewriting every numeric entry to `+...` would make QQ Bot configs such as `allowFrom: [12345]` fail to authorize sender `12345`, so phone-number coercion must stay in a channel-specific normalizer.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The PR changes shared allowlist authorization semantics and could deny legitimate numeric-ID channel senders.

Concerns:

- **[medium] Shared allowlist transform changes authorization semantics:** `src/channels/plugins/config-schema.ts:20`
  - body: The shared `AllowFromEntrySchema` is used by plugins such as QQ Bot, where numeric sender IDs are valid bare IDs. Prefixing them with `+` during config parsing would break allowlist authorization for those channels.
  - confidence: 0.91
