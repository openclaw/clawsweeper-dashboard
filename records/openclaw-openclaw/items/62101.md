## Summary

Keep this PR open: the BlueBubbles allowlist bug is still present on current main, but the submitted patch has a blocking regression because it moves BlueBubbles phone-number coercion into a shared allowlist schema used by numeric-ID channels such as QQ Bot.

## What This Changes

The PR prefixes all-digit BlueBubbles handles, changes the shared numeric `AllowFromEntrySchema` branch to emit `+`-prefixed strings, and adds tests for those normalization paths.

## Best Possible Solution

Keep the shared `AllowFromEntrySchema` as string-or-number passthrough, normalize bare numeric phone handles inside the BlueBubbles plugin path, add cross-channel regression coverage for QQ Bot numeric IDs, and include the required changelog entry.

## Reproduction Assessment

Yes. Current main can be traced by code inspection: dashboard-saved `19175551234` flows through string normalization, `normalizeBlueBubblesHandle` leaves it bare, and parsed-chat matching compares it against incoming `+19175551234`; the PR regression is likewise reproducible by parsing QQ Bot `allowFrom: [12345]` through the changed shared schema and matching sender `12345`.

## Solution Assessment

No. The BlueBubbles all-digit runtime guard is directionally correct, but changing the shared allowlist schema is not the narrowest maintainable fix because other plugins intentionally use numeric sender IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict E.164 coercion to BlueBubbles allowlists:** `src/channels/plugins/config-schema.ts:19`

Keep the shared primitive as passthrough and do the phone-number coercion in BlueBubbles-specific schema or normalization.
