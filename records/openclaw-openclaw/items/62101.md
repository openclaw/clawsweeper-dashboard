## Summary

Keep this PR open. The BlueBubbles allowlist bug is valid and current main still has the relevant mismatch, but the proposed schema change is too broad because it rewrites shared numeric allowlist entries for other channel plugins such as QQ Bot.

## What This Changes

The PR changes BlueBubbles handle normalization, changes shared numeric `allowFrom` schema entries into `+`-prefixed strings, and adds tests for the new normalization behavior.

## Best Possible Solution

Keep the BlueBubbles runtime normalization and schema coercion local to the BlueBubbles plugin, preserve the shared `AllowFromEntrySchema` as string/number passthrough, add cross-channel regression coverage for numeric-ID allowlists, and include the required changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence path: current main maps numeric BlueBubbles `allowFrom` entries to bare strings, `normalizeBlueBubblesHandle` does not add `+`, and access matching compares that value against incoming `+E.164` handles. The PR regression is also reproducible by parsing QQ Bot numeric `allowFrom` through the changed shared schema and then matching sender `12345`.

## Solution Assessment

No. The BlueBubbles-specific runtime guard is directionally right, but changing the shared allowlist schema is not the narrowest maintainable fix because numeric IDs are valid for other plugins. The safer solution is BlueBubbles-specific coercion plus regression tests and changelog coverage.
