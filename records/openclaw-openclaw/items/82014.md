- `impact:message-loss`: The affected turn can leave duplicate visible channel artifacts: the stale typing placeholder plus the actual Google Chat reply.

## Summary

Keep open: current main still keeps the Google Chat typing placeholder lifecycle on the automatic reply path, while current-source message-tool sends go through generic outbound delivery without the stored placeholder name.

## What This Changes

The issue asks Google Chat current-source message-tool replies to edit or consume the channel's typing placeholder instead of leaving a stale typing message beside the real reply.

## Best Possible Solution

Add a Google Chat-owned current-source delivery lifecycle path that consumes the typing placeholder after a successful message-tool reply, preferably editing the first text when safe and deleting the placeholder as the fallback.

## Reproduction Assessment

Yes, by source inspection: with Google Chat `typingIndicator: "message"` and group `visibleReplies: "message_tool"`, the monitor posts a typing message but message-tool sends use the generic Google Chat send path without that placeholder state. I did not perform a live Google Chat run.

## Solution Assessment

Yes on the requested behavior: the model should stay unaware of the Google Chat placeholder, and the channel/runtime lifecycle should consume it. The safer implementation choice still needs maintainer review because edit-in-place versus delete-after-send affects visible Google Chat behavior.
