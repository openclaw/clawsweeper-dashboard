## Summary

Keep open: current main still leaves Google Chat automatic reply delivery and configured current-source message-tool sends on separate paths, so the message-tool reply does not receive or consume the monitor-owned typing placeholder.

## What This Changes

The issue asks Google Chat current-source `message(action=send)` replies to edit or clear the typing placeholder instead of leaving a stale visible typing message beside the real reply.

## Best Possible Solution

Keep the fix in the Google Chat plugin boundary and add a typed same-source message-tool delivery lifecycle path that lets current-source sends edit or clear the monitor-owned placeholder without exposing Google Chat edit APIs to the model.

## Reproduction Assessment

Yes from source inspection, not from a live run: Google Chat creates a typing message before the turn, but configured current-source `message(action=send)` goes through the plugin send path that has no access to `typingMessageName`. The helper that edits or clears the placeholder is only called by automatic final-reply delivery.

## Solution Assessment

No, current main is not sufficient: the recent message-tool final-reply work still routes configured Google Chat sends through the normal plugin path. The maintainable fix is a plugin-owned lifecycle seam for same-source message-tool delivery, not a model-visible Google Chat edit workaround.
