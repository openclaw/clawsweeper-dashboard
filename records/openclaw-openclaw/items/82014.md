## Summary

Keep open: current main still has separate Google Chat automatic-delivery and message-tool send paths, and the message-tool path does not receive or consume the monitor-owned typing placeholder. The related thread-routing work explicitly treats typing-placeholder editing as separate, so this remains a distinct source-supported bug.

## What This Changes

The issue requests Google Chat current-source `message(action=send)` replies to edit or clear the existing typing placeholder instead of leaving a stale `_BotName is typing..._` message.

## Best Possible Solution

Keep the model-facing message tool unchanged and add a Google Chat-owned lifecycle hook so current-source message-tool sends reuse or explicitly clear the existing typing placeholder, with focused regression coverage and redacted real Google Chat proof.

## Reproduction Assessment

Yes, source-level. With Google Chat `typingIndicator: "message"` and group visible replies in `message_tool` mode, the monitor creates `typingMessageName`, while the current-source message-tool send path calls `sendGoogleChatMessage` directly and never invokes the placeholder edit/delete helper.

## Solution Assessment

Yes. The requested direction is the right layer: integrate the Google Chat message-tool delivery path with the existing channel-owned placeholder lifecycle instead of adding a model-visible edit API or new config option.
