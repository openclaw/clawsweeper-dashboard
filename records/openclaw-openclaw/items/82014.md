## Summary

Keep open: current main and v2026.6.10 still keep Google Chat placeholder-aware automatic delivery separate from current-source message-tool sends, so the stale typing-placeholder problem remains. The related thread-routing work is adjacent and explicitly leaves typing-placeholder edit/cleanup out of scope.

## What This Changes

The issue asks OpenClaw to consume or edit Google Chat typing-placeholder messages when the visible reply is delivered through the current-source message tool.

## Best Possible Solution

Add a Google Chat-owned lifecycle bridge so current-source message-tool sends can update or clear the monitor-created typing placeholder without exposing Google Chat edit APIs to the model.

## Reproduction Assessment

Yes, source-level: Google Chat `typingIndicator: "message"` creates `typingMessageName` only in the monitor delivery closure, while current-source message-tool sends go through direct action/adapter paths that do not receive that state. I have not run a live Google Chat room reproduction in this read-only review.

## Solution Assessment

Yes, the requested fix boundary is the narrow maintainable one: bridge the Google Chat placeholder lifecycle into current-source message-tool delivery. Asking the model to call a Google Chat-specific edit API would put channel internals in the wrong layer.
