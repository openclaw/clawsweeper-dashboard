## Label Justifications

- `P2`: This is a normal-priority Google Chat delivery bug with visible duplicate/stale chat artifacts but limited channel-specific blast radius.
- `impact:message-loss`: The issue is about visible message delivery state: a real reply is delivered, but the stale typing placeholder remains as an extra misleading message in the same conversation.

## Summary

Keep open: current main still keeps Google Chat typing-placeholder state inside the normal monitor reply-delivery path, while current-source `message(action=send)` replies go through the generic outbound/message adapter and send a separate Google Chat message without receiving or consuming `typingMessageName`. The related thread-routing PRs are useful context but do not prove this placeholder lifecycle bug is fixed.

## What This Changes

The issue requests that Google Chat current-source message-tool replies edit or clear the existing typing placeholder instead of leaving `_BotName is typing..._` visible beside the real reply.

## Best Possible Solution

Add a Google Chat-owned lifecycle bridge so current-source `message(action=send)` can edit or clear the monitor-created typing placeholder while keeping Google Chat resource details inside the plugin boundary and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: configure Google Chat with `typingIndicator: "message"` and group visible replies set to `message_tool`, then the monitor-created `typingMessageName` is only available to normal delivery while the current-source message-tool send bypasses that path. I did not run a live Google Chat repro in this read-only review.

## Solution Assessment

No current implementation fully solves it. The maintainable fix should share or hand off the Google Chat placeholder lifecycle to the current-source message-tool path, not ask the model to call a Google Chat-specific edit API.
