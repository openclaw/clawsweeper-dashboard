## Label Justifications

- `P2`: This is a normal-priority Google Chat delivery bug with visible duplicate/stale chat artifacts and limited channel-specific blast radius.
- `impact:message-loss`: The issue concerns channel message delivery state: a real reply is delivered while a stale placeholder remains as an extra misleading message.

## Summary

Keep open: current main and the latest release still keep Google Chat placeholder-aware automatic delivery separate from current-source `message(action=send)`, and the open thread-delivery PR explicitly leaves typing-placeholder cleanup out of scope.

## What This Changes

Ensure Google Chat current-source `message(action=send)` replies edit or clear the typing placeholder instead of leaving a stale placeholder beside the real reply.

## Best Possible Solution

Add a Google Chat-owned lifecycle bridge so current-source message-tool sends can update or clear the monitor-created typing placeholder while keeping Google Chat resource details hidden from the model.

## Reproduction Assessment

Yes, at source level: configure Google Chat with `typingIndicator: "message"` and `messages.groupChat.visibleReplies: "message_tool"`; the monitor-created `typingMessageName` is only available to automatic delivery while the current-source message-tool send bypasses it.

## Solution Assessment

No current-main implementation fully solves it. The maintainable fix should share or hand off the Google Chat placeholder lifecycle to the current-source message-tool path, not ask the model to call a Google Chat-specific edit API.
