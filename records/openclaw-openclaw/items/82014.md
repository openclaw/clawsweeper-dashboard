## Label Justifications

- `P2`: This is a normal-priority Google Chat delivery bug with visible duplicate/stale chat artifacts but limited channel-specific blast radius.
- `impact:message-loss`: The issue is about visible message delivery state: a real reply is delivered, but a stale typing placeholder remains as an extra misleading message.

## Summary

Keep open: source inspection shows current main still separates Google Chat placeholder-aware automatic reply delivery from the current-source `message(action=send)` path, and the related thread-target PRs do not close this placeholder lifecycle gap.

## What This Changes

The issue requests that Google Chat current-source message-tool replies consume or edit the typing placeholder instead of leaving a stale placeholder plus a separate reply.

## Best Possible Solution

Add a Google Chat-owned lifecycle bridge so current-source `message(action=send)` can edit or clear the monitor-created typing placeholder while keeping Google Chat resource details inside the plugin boundary.

## Reproduction Assessment

Yes, at source level: configure Google Chat with `typingIndicator: "message"` and group visible replies set to `message_tool`, then the monitor-created `typingMessageName` is only available to normal delivery while the current-source message-tool send bypasses that path. I did not run a live Google Chat repro in this read-only review.

## Solution Assessment

No current-main implementation fully solves it. The maintainable fix should share or hand off the Google Chat placeholder lifecycle to the current-source message-tool path, not ask the model to call a Google Chat-specific edit API.
