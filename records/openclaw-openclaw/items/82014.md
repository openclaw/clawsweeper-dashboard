## Summary

Keep open: current main still has separate Google Chat automatic reply delivery and message-tool send paths, and only the automatic path receives the monitor-owned typing placeholder. The related open work covers thread placement or NO_REPLY cleanup, but not placeholder consumption for current-source message-tool sends.

## What This Changes

Google Chat current-source message-tool sends should update or clear the typing placeholder instead of leaving a stale visible typing message.

## Best Possible Solution

Add a Google Chat-owned lifecycle hook so current-source message-tool sends reuse, update, or clear the monitor-created placeholder, with focused regression coverage and redacted live Google Chat proof.

## Reproduction Assessment

Yes, source-level. With Google Chat message typing indicators and `messages.groupChat.visibleReplies` in message-tool mode, current main creates `typingMessageName` in the monitor while the Google Chat message-tool send path calls `sendGoogleChatMessage()` directly without consuming that placeholder.

## Solution Assessment

Yes. The maintainable fix is to keep the model-facing message tool unchanged and integrate current-source Google Chat sends with the channel-owned placeholder lifecycle, sequenced with the adjacent thread-routing and NO_REPLY cleanup work.
