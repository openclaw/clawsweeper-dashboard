## Label Justifications

- `impact:message-loss`: The issue concerns channel message delivery state: a real reply is delivered while a stale placeholder remains as an extra misleading message.

## Summary

Keep open: current main and the latest release still keep Google Chat typing-placeholder editing inside the monitor-owned automatic delivery path, while current-source `message(action=send)` replies go through the generic outbound adapter without any `typingMessageName` handoff.

## What This Changes

Requests Google Chat current-source message-tool replies to edit or clear the monitor-created typing placeholder instead of leaving a stale typing message.

## Best Possible Solution

Add a Google Chat-owned lifecycle bridge so current-source message-tool sends can update or clear the monitor-created typing placeholder while keeping Google Chat resource details hidden from the model.

## Reproduction Assessment

Yes, source-level: configure Google Chat with `typingIndicator: "message"` and `messages.groupChat.visibleReplies: "message_tool"`; the monitor-created `typingMessageName` is only available to automatic delivery while the current-source message-tool send bypasses it. I did not run a live Google Chat scenario in this read-only review.

## Solution Assessment

No current-main implementation fully solves it. The maintainable fix should share or hand off the Google Chat placeholder lifecycle to the current-source message-tool path, not ask the model to call a Google Chat-specific edit API.
