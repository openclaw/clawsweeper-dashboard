## Summary

Keep open: current main still has separate Google Chat automatic-delivery and message-tool send paths, and the message-tool path does not receive or consume the monitor-owned `typingMessageName`. The active related threading work explicitly scopes typing-placeholder editing out, so this remains a distinct source-supported bug.

## What This Changes

Google Chat current-source `message(action=send)` replies should edit or clear the existing typing placeholder instead of leaving a stale typing message.

## Best Possible Solution

Keep the model-facing message tool unchanged and add a Google Chat-owned lifecycle hook so current-source message-tool sends reuse or explicitly clear the existing typing placeholder.

## Reproduction Assessment

Yes, source-level. With Google Chat `typingIndicator: "message"` and group visible replies in `message_tool` mode, the monitor creates `typingMessageName`, while the current-source message-tool send path calls `sendGoogleChatMessage` directly and never invokes the placeholder edit/delete helper.

## Solution Assessment

Yes, the requested direction is the right layer: integrate the Google Chat message-tool delivery path with the existing channel-owned placeholder lifecycle. A model-visible edit API or new config option would be a less maintainable fix.
