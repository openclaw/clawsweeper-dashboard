## Label Justifications

- `P2`: The report is a source-supported Google Chat channel bug with visible duplicate/stale messages but limited blast radius.
- `impact:message-loss`: The affected path can leave duplicated visible Google Chat artifacts by sending the real reply while the provisional typing message remains.

## Summary

Keep open: current main still creates a Google Chat typing placeholder in the monitor-owned delivery lifecycle, while current-source `message(action=send)` uses the Google Chat action path and sends a separate message without receiving, editing, or clearing that placeholder.

## What This Changes

The issue asks Google Chat current-source message-tool replies to edit or consume the typing placeholder instead of leaving a stale `_BotName is typing..._` message.

## Best Possible Solution

Add a Google Chat-owned lifecycle hook so current-source message-tool sends reuse, update, or clear the monitor-created typing placeholder, with focused regression coverage and redacted live Google Chat proof.

## Reproduction Assessment

Yes, with source-level confidence. With Google Chat `typingIndicator: "message"` and group visible replies in `message_tool` mode, current main stores `typingMessageName` for automatic delivery while the message-tool action sends directly without that lifecycle state.

## Solution Assessment

Yes. The maintainable fix is to keep the model-facing message tool unchanged and integrate Google Chat current-source sends with the channel-owned placeholder lifecycle rather than exposing a special edit API or adding configuration.
