## Summary

Keep open: current `origin/main` and v2026.6.8 still keep Google Chat placeholder-aware automatic delivery separate from current-source `message(action=send)`, and the related thread-routing work explicitly leaves placeholder cleanup out of scope.

## What This Changes

The issue asks Google Chat current-source `message(action=send)` replies to edit or consume the typing placeholder instead of leaving a stale placeholder beside the real reply.

## Best Possible Solution

Add a Google Chat-owned lifecycle bridge so current-source message-tool sends can update or clear the monitor-created typing placeholder while keeping Google Chat resource details hidden from the model.

## Reproduction Assessment

Yes, source-level: with Google Chat `typingIndicator: "message"` and `messages.groupChat.visibleReplies: "message_tool"`, the monitor-created `typingMessageName` is available only to automatic delivery while the current-source message-tool send bypasses it. I did not run a live Google Chat room repro in this read-only review.

## Solution Assessment

Yes, the requested fix boundary is the right one: the Google Chat plugin should bridge or share its placeholder lifecycle with current-source message-tool delivery. Asking the model to call a Google Chat-specific edit API would be the wrong layer.
