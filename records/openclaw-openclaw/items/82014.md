- `impact:message-loss`: The issue is about channel message delivery state producing an extra stale placeholder alongside the real message-tool reply.

## Summary

Keep open: current main and the latest release still keep Google Chat automatic reply delivery and message-tool sends on separate paths, so a current-source `message(action=send)` does not receive or consume the monitor-owned typing placeholder.

## What This Changes

Ensure Google Chat current-source message-tool replies edit or clear the typing placeholder instead of leaving a stale `_BotName is typing..._` message beside the real reply.

## Best Possible Solution

Keep this open until Google Chat current-source message-action sends consume, edit, or clear the monitor-owned typing placeholder through a plugin-owned lifecycle seam with focused regression coverage.

## Reproduction Assessment

Yes, source-level: with Google Chat `typingIndicator: "message"` and tool-only visible replies, the monitor creates `typingMessageName`, the final reply stays private, and `message(action=send)` routes through the plugin action path that never receives the placeholder. I did not run a live Google Chat turn in this read-only review.

## Solution Assessment

Unclear for the exact implementation shape: the right outcome is to integrate current-source Google Chat message-action delivery with the existing placeholder lifecycle, but maintainers should sequence it with the open thread-routing work rather than adding a model-facing special API.
