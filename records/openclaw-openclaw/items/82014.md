## Summary

Keep open: current main still has separate Google Chat paths for automatic delivery and message-tool delivery. The automatic path edits or clears `typingMessageName`, while `message(action=send)` routes through the plugin send action and calls `sendGoogleChatMessage` directly, so the reported stale placeholder is source-supported.

## What This Changes

Google Chat current-source message-tool replies should consume or edit the existing typing placeholder instead of leaving a stale placeholder plus a separate reply.

## Best Possible Solution

Keep the model-facing message tool unchanged and integrate Google Chat current-source sends with the existing channel-owned provisional-message lifecycle, ideally updating the placeholder with the first text chunk and otherwise deleting or clearing it without adding a config knob.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with Google Chat `typingIndicator: "message"` and group visible replies set to `message_tool`, a current-source `message(action=send)` follows `actions.ts` instead of `deliverGoogleChatReply`, leaving the monitor-owned `typingMessageName` unconsumed. I did not run a live Google Chat turn in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow Google Chat delivery integration that reuses the existing placeholder edit/delete behavior. A new model-visible API or config option would be the wrong layer; any core seam should be additive and channel-owned.
