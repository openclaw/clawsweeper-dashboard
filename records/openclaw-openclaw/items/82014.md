## Summary

Keep open: current main still has a source-supported Google Chat UX bug. The documented typing-message mode says the placeholder is edited with the response, but message-tool sends for the current Google Chat conversation bypass the reply-delivery path that owns `typingMessageName`, so the placeholder can remain visible.

## What This Changes

Google Chat current-source `message(action=send)` replies should edit or clear the existing typing placeholder instead of leaving a stale `_BotName is typing..._` message.

## Best Possible Solution

Keep the model-facing message tool unchanged and make Google Chat current-source sends participate in the channel-owned provisional-message lifecycle, preferably editing the first text chunk into the placeholder and otherwise deleting or clearing it.

## Reproduction Assessment

Yes, with source inspection rather than a live turn. A Google Chat group turn with `typingIndicator: "message"` and `messages.groupChat.visibleReplies: "message_tool"` posts a monitor-owned typing message, suppresses automatic delivery, and routes `message(action=send)` through `actions.ts` without `typingMessageName`.

## Solution Assessment

Yes. The narrowest maintainable direction is to reuse the existing Google Chat placeholder edit/delete lifecycle for current-source tool sends; a model-visible edit API or new config knob would expose a channel implementation detail.
