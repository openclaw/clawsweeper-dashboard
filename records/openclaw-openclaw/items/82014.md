## Summary

Keep open: current main still source-supports the reported Google Chat placeholder lifecycle gap, and the related open Google Chat threading and NO_REPLY cleanup PRs do not close this distinct message-tool placeholder case.

## What This Changes

Google Chat current-source `message(action=send)` replies should edit or consume the existing typing placeholder instead of leaving a stale `_BotName is typing..._` message.

## Best Possible Solution

Add a Google Chat-owned lifecycle hook so current-source message-tool sends reuse the monitor-owned typing placeholder for the first visible reply chunk or explicitly clear it, while keeping the model-facing message tool unchanged.

## Reproduction Assessment

Yes, source-level. With Google Chat `typingIndicator: "message"` and group visible replies in `message_tool` mode, current main creates `typingMessageName`, suppresses automatic delivery, and sends the visible reply through direct message-tool paths that never receive or consume that placeholder.

## Solution Assessment

Yes. The requested direction is the maintainable layer: integrate Google Chat current-source message-tool delivery with the channel-owned placeholder lifecycle instead of exposing Google Chat edit/delete details to the model or adding a config workaround.
