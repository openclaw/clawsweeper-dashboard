## Summary

Keep open: current main still has separate Google Chat automatic delivery and message-tool send paths, and the message-tool path does not receive or consume the monitor-owned typing placeholder. The related open work addresses thread placement or NO_REPLY cleanup, but this remains a distinct source-supported bug.

## What This Changes

The issue requests that Google Chat current-source `message(action=send)` replies edit, consume, or clear the monitor-created typing placeholder instead of leaving a stale `_BotName is typing..._` message.

## Best Possible Solution

Add a Google Chat-owned lifecycle handoff so current-source message-tool sends reuse, update, or clear the monitor-created typing placeholder without exposing placeholder details to the model.

## Reproduction Assessment

Yes, source-level. With Google Chat `typingIndicator: "message"` and group visible replies in `message_tool` mode, current main creates `typingMessageName` in the monitor while the Google Chat message-tool send action calls `sendGoogleChatMessage()` directly without using the placeholder lifecycle.

## Solution Assessment

Yes. The maintainable fix is to keep the model-facing message tool unchanged and integrate Google Chat current-source sends with the channel-owned placeholder lifecycle instead of exposing a model-visible edit API or adding a config option.
