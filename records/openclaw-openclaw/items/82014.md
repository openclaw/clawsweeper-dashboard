## Summary

Keep open: current main still creates a Google Chat typing placeholder in the monitor, but current-source `message(action=send)` replies go through the Google Chat action path without receiving or consuming that placeholder. The related open Google Chat work covers thread placement or NO_REPLY cleanup, not this message-tool placeholder lifecycle.

## What This Changes

Google Chat should consume or edit the typing placeholder when current-source replies are delivered through `message(action=send)`.

## Best Possible Solution

Add a Google Chat-owned lifecycle hook so current-source message-tool sends reuse, update, or clear the monitor-created placeholder, with focused regression coverage and redacted live Google Chat proof.

## Reproduction Assessment

Yes, source-level. With Google Chat `typingIndicator: "message"` and group visible replies in `message_tool` mode, current main creates `typingMessageName` in the monitor while the Google Chat message-tool send action calls `sendGoogleChatMessage()` directly without using the placeholder lifecycle.

## Solution Assessment

Yes. The maintainable fix is to keep the model-facing message tool unchanged and integrate Google Chat current-source sends with the channel-owned placeholder lifecycle, rather than exposing a model-visible edit API or adding another config option.
