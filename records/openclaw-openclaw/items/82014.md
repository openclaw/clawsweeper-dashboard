- `P2`: This is a normal-priority Google Chat delivery bug with visible duplicate or stale chat artifacts and limited channel-specific blast radius.
- `impact:message-loss`: The issue concerns channel message delivery state: a real reply is delivered while a stale placeholder remains as an extra misleading message.

## Summary

Keep open: current main and the latest release still keep Google Chat typing-placeholder editing inside automatic delivery, while current-source message-tool sends call the Google Chat send path directly and do not consume the placeholder.

## What This Changes

The issue asks Google Chat current-source `message(action=send)` replies to edit or clear the monitor-created typing placeholder like automatic final replies do.

## Best Possible Solution

Implement a Google Chat-owned lifecycle bridge so current-source message-tool sends update or clear the existing typing message, with regression coverage for `message_tool` visible replies and no model-facing Google Chat edit API.

## Reproduction Assessment

Yes, source-level: Google Chat `typingIndicator: "message"` creates `typingMessageName` only in the monitor delivery closure, while current-source message-tool sends go through direct send paths that do not receive that state. I did not run a live Google Chat room reproduction in this read-only review.

## Solution Assessment

Yes, the right solution boundary is a Google Chat-owned bridge between current-source message-tool delivery and the existing placeholder lifecycle. Asking the model to call a Google Chat-specific edit API would put channel internals in the wrong layer.
