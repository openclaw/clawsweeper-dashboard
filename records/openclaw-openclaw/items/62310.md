## Summary

Keep open: current main still lacks the heartbeat burst-throttling behavior, and this PR is the canonical implementation path for #62294, but the latest branch still blocks a documented explicit plugin wake path by throttling no-argument `requestHeartbeatNow()` calls inside the interval.

## What This Changes

The PR adds heartbeat last-run cooldown state, scheduler overdue handling, regression tests, and a changelog entry for throttling ordinary activity-driven heartbeat wake bursts while preserving selected explicit wake paths.

## Best Possible Solution

Keep this PR open as the canonical fix path for #62294, repair the throttle predicate so omitted/default explicit plugin wakes remain immediate, add a focused regression test for that contract, then land the branch or an equivalent replacement after rebase and changed-gate validation.

## Reproduction Assessment

Yes. On the PR branch, run one heartbeat, then call `api.runtime.system.requestHeartbeatNow()` or `requestHeartbeatNow()` with no options inside the configured interval; the omitted reason resolves to `other`, the new predicate enforces the interval, and the wake is skipped as `too-soon` before `runOnce`.

## Solution Assessment

No, not as-is. The cooldown and overdue-recovery direction is narrow and maintainable for activity-driven bursts, but explicit no-argument plugin/system wake requests need to be exempted or normalized to manual semantics before merge.
