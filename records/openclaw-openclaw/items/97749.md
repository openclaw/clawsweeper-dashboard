## Summary

Keep open: current main and the latest release still only support global managed plugin install records and no `--agent` install/uninstall flag, while the requested per-agent install lifecycle is a valid but security-sensitive product/API design change.

## What This Changes

Add agent-scoped plugin install and uninstall support so the same plugin id can have independent per-agent install roots, versions, records, and cleanup behavior.

## Best Possible Solution

Define and implement a maintainer-approved per-agent plugin install contract covering record identity, install roots, discovery precedence, update/uninstall isolation, security policy, docs, and upgrade/doctor behavior.

Yes, at source level: current main lacks the requested `--agent` CLI flags, installs through the default global extensions directory, and persists/removes records by flat `pluginId`. I did not run a mutating CLI repro because this review is read-only.

## Solution Assessment

Unclear. The `--agent` CLI shape is plausible, but the best solution needs maintainer decisions on install identity, workspace/global precedence, security policy, update/uninstall isolation, and upgrade behavior.
