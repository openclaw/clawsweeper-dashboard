## Summary

Keep open: current main still only supports global plugin install/uninstall records and roots, while agent-scoped plugin installation would change core plugin lifecycle, discovery, persisted install identity, and security boundaries.

## What This Changes

The issue requests `--agent <id>` support for `openclaw plugins install` and `openclaw plugins uninstall` so plugin installs, versions, directories, and uninstall behavior can be isolated per agent.

## Best Possible Solution

Define a maintainer-approved agent-scoped plugin lifecycle contract covering CLI flags, install record identity, install roots, runtime discovery scope, config/allowlist/slot behavior, uninstall/update isolation, trust/security behavior, docs, and upgrade tests.

## Reproduction Assessment

Yes. Source inspection shows `openclaw plugins install` and `openclaw plugins uninstall` have no `--agent` option, use global plugin roots, and persist flat plugin-id install records on current main.

## Solution Assessment

Unclear. The requested `--agent` UX is plausible, but the proposed compound-key patch shape is not enough to judge as the best fix because duplicate plugin-id discovery, runtime activation scope, persisted config cleanup, and security semantics need a maintainer-owned design first.
