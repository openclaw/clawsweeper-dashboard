## Summary

Keep this PR open: it targets a valid auth-status display bug, but the implementation is broader than the OAuth marker case and would reclassify existing non-resolvable markers as effective auth; it also still needs real behavior proof beyond mocked command tests.

## What This Changes

The PR changes the models status auth overview to treat non-secret models.json markers as effective auth and adds a command regression test for `oauth:openai-codex`.

## Best Possible Solution

Keep the intended OAuth delegation fix, but narrow the effective-auth branch to OAuth markers, preserve existing SecretRef/local marker behavior, add focused coverage for both paths, and require redacted real CLI proof.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the new predicate accepts `secretref-managed`, while the unchanged auth-overview test expects that marker to remain `effective=missing`. The original issue is also source-reproducible from current main because OAuth markers fall through after `resolveUsableCustomProviderApiKey` returns null.

## Solution Assessment

No as written; the idea is right for OAuth delegation markers, but `isNonSecretApiKeyMarker` is too broad for the effective-auth contract. The safer fix is an OAuth-specific check, plus coverage that non-OAuth unresolved markers still display as markers without becoming effective auth.
