## Summary

Keep open: the macOS menubar symptom is plausible and current main does not implement this watchdog, but the PR is not merge-ready because the timeout eviction can hide active quiet work, one new test expects fresh work to be evicted, and real macOS behavior proof is still missing.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that clears stale menubar activity state, refreshes job timestamps on streaming/tool events, recomputes job role after main-session changes, and expands Swift tests for that behavior.

## Best Possible Solution

Keep this as a repairable PR, but merge only after the stale-state cleanup is lifecycle-aware, tests deterministically age stale activity without evicting fresh active work, and the branch includes redacted real macOS proof of the frozen-menubar scenario.

## Reproduction Assessment

Yes for the PR regression at source level: a started job or tool with no later streaming/tool events can exceed the watchdog threshold and be removed without a terminal event. I did not run a live macOS provider-overload reproduction in this read-only review.

## Solution Assessment

No. A watchdog can be an acceptable mitigation, but the current unconditional age-based eviction is not the best fix because it cannot distinguish orphaned UI state from active quiet work.

Full review comments:

- **[P2] Preserve quiet active work before evicting stale state:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:338-339`
  - body: This loop removes every job or tool whose `lastUpdate` is older than the fixed threshold, but the store only refreshes that timestamp on job start/streaming and tool start. A valid non-streaming model call or long shell tool that emits no intermediate events will be deleted while still running, so the menubar can flip to idle even though the session is active.
  - confidence: 0.93
- **[P2] Age the activity before expecting eviction:** `apps/macos/Tests/OpenClawIPCTests/WorkActivityStoreTests.swift:171-173`
  - body: The test starts a fresh job and immediately calls `evictStaleActivities()`, but production only evicts entries older than the 180-second threshold. As written, this assertion fails or encourages the implementation to clear fresh active work; the test needs a deterministic way to create an old activity before expecting idle.
  - confidence: 0.95
