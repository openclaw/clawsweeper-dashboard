## Summary

Keep open: the macOS menubar symptom is valid, but this PR is not merge-ready because the patch can false-idle active quiet work, does not wire current lifecycle events into the activity model, has a contradictory stale-eviction test, and lacks real macOS behavior proof.

## What This Changes

Adds a macOS WorkActivityStore watchdog, timestamp refresh logic, clear-all helper, and tests intended to clear stale menubar work activity after provider overload or timeout scenarios.

## Best Possible Solution

Repair the macOS activity model to consume current lifecycle/tool events and clear only provably orphaned display state, with tests for quiet active runs, lost terminal recovery, and real macOS proof.

## Reproduction Assessment

Yes at source level: a fresh started job cannot satisfy the greater-than-180-second eviction check, while any quiet active run older than that can be removed without terminal proof. I did not run a live macOS provider-overload scenario in this read-only review.

## Solution Assessment

No. A watchdog may be useful, but the best fix needs lifecycle-aware activity tracking and stale-orphan detection rather than unconditional age-based eviction.

Full review comments:

- **[P2] Handle lifecycle activity, not job-only state:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:63-64`
  - body: Current gateway events use the `lifecycle` stream for run start/end/error and `tool` for tool activity, while the macOS router only sends stream `job` into `handleJob`. This PR adds a watchdog around job state but does not wire lifecycle events into that state, so the claimed lost-run-end menubar freeze can remain unsolved on current main.
  - confidence: 0.83
- **[P2] Do not evict quiet active work by age alone:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:337-344`
  - body: This loop removes any job or tool whose `lastUpdate` is older than the threshold, but timestamps are only refreshed on streaming/tool-start events. A valid non-streaming model call or long shell command with no progress events can be cleared from the menubar while it is still running.
  - confidence: 0.9
- **[P2] Age activity before expecting stale eviction:** `apps/macos/Tests/OpenClawIPCTests/WorkActivityStoreTests.swift:171-172`
  - body: The test starts a job and immediately calls `evictStaleActivities()`, but production eviction requires `now - lastUpdate > 180s`. As written, this assertion should fail and does not prove stale cleanup; inject a clock or test hook to create an actually stale activity first.
  - confidence: 0.95
