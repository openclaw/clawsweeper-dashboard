## Summary

This PR should stay open because the macOS menu-bar freeze symptom is plausible from the current event-driven state path, but the proposed fixed-timeout watchdog can make legitimate quiet work look idle and still lacks real macOS proof after the fix.

## What This Changes

The PR adds a macOS `WorkActivityStore` watchdog that evicts stale job/tool display state, refreshes some activity timestamps, exposes `clearAllActivities()`, and adds Swift tests for menu-bar activity state.

## Best Possible Solution

Preserve visible activity for known active quiet work while adding a state-aware recovery path for orphaned or disconnected activity, with real macOS proof for the overload path.

## Reproduction Assessment

Yes at source level, but not as a live macOS reproduction: current main routes gateway job/tool events into `WorkActivityStore`, and a started or streaming job remains active until a terminal event arrives. I did not establish a real provider-overload run in this read-only review.

## Solution Assessment

No. A fixed three-minute timeout conflates lost terminal events with valid quiet active runs; explicit orphan/disconnect handling, a stale/degraded display state, or a gateway event-contract fix would be safer.
