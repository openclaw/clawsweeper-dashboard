## Summary

Keep open: the macOS menubar symptom is plausible and still not implemented on current main, but this branch is not merge-ready because the watchdog can hide active quiet runs, one new stale-eviction test asserts behavior the implementation does not perform, and the external PR still lacks real macOS behavior proof.

## What This Changes

The PR adds a macOS `WorkActivityStore` watchdog that evicts stale job/tool display state after a fixed timeout and adds Swift tests for the activity-store behavior.

## Best Possible Solution

Revise the fix to be lifecycle-aware, preserve quiet active work unless the gateway proves the UI state is orphaned, and add redacted real macOS proof of the overload/timeout recovery path.

## Reproduction Assessment

Yes at source level for the PR regression: any job/tool whose `lastUpdate` exceeds the 180-second threshold can be evicted without a terminal event, so quiet active work can be hidden. I did not run a live macOS provider-overload scenario in this read-only review.

## Solution Assessment

No. A watchdog may be a useful mitigation, but the best fix should distinguish orphaned UI state from quiet active work using lifecycle/disconnect/error ownership instead of unconditional age-based eviction.

- **[P1] Preserve quiet active runs before stale eviction:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:339`
