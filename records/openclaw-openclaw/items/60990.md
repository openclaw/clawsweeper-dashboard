## Summary

Keep open: current main still has event-driven menu-bar activity with no stale recovery, but this PR is not ready because the fixed 180-second eviction can show idle for healthy quiet work and one new test asserts behavior the implementation cannot produce.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that evicts job/tool UI activity after 180 seconds, refreshes timestamps on streaming and tool-start events, exposes all-activity cleanup, and adds Swift tests.

## Best Possible Solution

Recover orphaned macOS menu-bar activity without clearing known-active quiet jobs or tools solely for lack of progress events, preferably through an explicit orphan/disconnect signal or stale-state UI plus deterministic clock-based tests.

## Reproduction Assessment

Yes, at source level. Current main can be driven with a job started event and no terminal event through ControlChannel to WorkActivityStore, and the PR regression is a quiet job or tool whose lastUpdate exceeds 180 seconds without progress events.

## Solution Assessment

No. The fixed timeout is not the narrowest maintainable fix because it conflates lost terminal events with valid quiet work; a safer path is explicit orphan/disconnect handling, gateway-side terminal-event repair, or a visible stale activity state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve quiet active work during stale cleanup:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:337-344`
  - body: The watchdog removes any job/tool whose `lastUpdate` is older than 180 seconds, but timestamps are only refreshed on streaming job events and tool starts. A valid non-streaming model call or long shell command can run quietly past that threshold, so the menu can flip to idle while work is still active.
  - confidence: 0.9
- **[P2] Age the activity before expecting stale eviction:** `apps/macos/Tests/OpenClawIPCTests/WorkActivityStoreTests.swift:170-172`
  - body: This test calls `evictStaleActivities()` immediately after a fresh `started` event, but production only removes entries older than `jobStaleThreshold`. Use an injected clock or another test seam to make the activity genuinely stale before expecting idle.
  - confidence: 0.88
