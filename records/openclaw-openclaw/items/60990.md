## Summary

Keep open for maintainer review. Current main still has event-driven macOS menu-bar activity that can stay working if a terminal job event is lost, but this PR's fixed 3-minute timeout would also clear healthy quiet long-running work and one new stale-eviction test does not actually age the activity before expecting idle.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that clears job/tool UI activity after a 3-minute stale threshold, refreshes timestamps on streaming and tool-start events, exposes all-activity cleanup, and adds WorkActivityStore tests.

## Best Possible Solution

Recover orphaned macOS activity UI without clearing known-active quiet jobs or tools solely because no progress event arrived, add a deterministic clock/test seam, and include the required user-facing changelog entry before merge.

## Reproduction Assessment

Yes. Current main can be reproduced at source level by sending a job started event without any terminal event and observing the icon remain working; the PR regression can be reproduced by letting a valid quiet job or tool exceed 180 seconds with no progress event and watching the watchdog flip the UI to idle.

## Solution Assessment

No. A fixed timeout is not the narrowest maintainable fix because it conflates lost terminal events with valid quiet work; a safer design would use an explicit orphan/disconnect signal, a conservative stale state, or gateway-side terminal-event repair.
