## Summary

Keep open: current main still has the source-level orphaned menu-bar activity path, but this PR should not merge as-is because the fixed timeout can hide valid quiet work, one new test asserts a fresh job is idle, and after-fix real macOS proof is still missing.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that evicts stale job/tool activity after a fixed age and adds Swift tests for watchdog startup, timestamp refresh, role refresh, clear-all, and stale eviction.

## Best Possible Solution

Repair orphaned menu-bar activity through an explicit terminal/disconnect/orphan signal or state-aware cleanup that preserves valid quiet jobs/tools, with deterministic clock-based coverage and real macOS overload proof.

## Reproduction Assessment

Yes, at source level: current main routes agent job events into WorkActivityStore and keeps started/streaming activity active until a non-start state arrives, so omitting the terminal event leaves menu-bar activity active indefinitely. I did not run a live macOS provider-overload setup.

## Solution Assessment

No. A fixed three-minute timeout is not the narrowest maintainable fix because it conflates orphaned terminal events with valid quiet active work; explicit orphan/disconnect handling, gateway-side terminal repair, or a visible stale state would be safer.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve quiet active work during stale cleanup:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:337-344`
  - body: The watchdog removes jobs/tools whenever lastUpdate is older than 180 seconds, but lastUpdate is only refreshed on streaming job events and tool starts.   - confidence: 0.9
