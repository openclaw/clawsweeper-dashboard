## Summary

Keep open for maintainer review. Current main still has the menu-bar activity state driven only by gateway job/tool events, so the PR is not obsolete, but the proposed fixed 3-minute eviction can misreport healthy quiet long-running work as idle and one new stale-eviction test does not actually create stale activity before asserting idle.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that clears job/tool UI activity after a 3-minute stale threshold, refreshes timestamps on streaming and tool-start events, exposes all-activity cleanup, and adds WorkActivityStore tests.

## Best Possible Solution

Revise the macOS activity-state cleanup so orphaned UI state can recover without clearing known-active quiet jobs or tools solely because no progress event arrived for 180 seconds, make stale behavior deterministic with a clock or test seam, and add the required user-facing changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start a WorkActivityStore job or tool, let no streaming/tool events arrive for more than 180 seconds, and observe the PR watchdog evict it before any terminal event; the stale-eviction unit test also lacks an aged timestamp or test clock.

## Solution Assessment

No. The current proposed fixed timeout is not the narrowest maintainable solution because it conflates missing terminal events with valid quiet long-running provider calls or shell tools; a safer fix needs an explicit orphan/disconnect signal, a much more conservative stale policy, or a visible stale state that does not pretend work is idle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve quiet active work during stale cleanup:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:337-344`
  - body: The watchdog removes any job/tool whose lastUpdate is older than 180 seconds, but lastUpdate is only refreshed on streaming job events and tool starts. A valid non-streaming model call or long shell command can run quietly past that threshold, so the menu flips to idle while work is still active, regressing the current event-driven activity state.
  - confidence: 0.9
- **[P2] Age the activity before expecting stale eviction:** `apps/macos/Tests/OpenClawIPCTests/WorkActivityStoreTests.swift:170-172`
  - body: This test calls evictStaleActivities() immediately after a fresh started event, but production only removes entries older than jobStaleThreshold. Use an injected clock or another test seam to make the activity genuinely stale before expecting idle.
  - confidence: 0.88

  - body: This is a user-facing macOS fix, but the PR changes only Swift source and tests. Repo policy requires a single-line CHANGELOG.md entry for user-facing fixes, with appropriate contributor credit.
