## Summary

Keep open: the reported macOS menu-bar stale activity symptom is plausible, but the current PR head can incorrectly show idle during valid quiet work and its stale-eviction test contradicts the production threshold; the PR also lacks required after-fix real macOS behavior proof.

## What This Changes

Adds a macOS WorkActivityStore watchdog that clears stale menu-bar job/tool activity after three minutes, plus Swift tests for watchdog startup, role refresh, clear-all behavior, and stale eviction.

## Best Possible Solution

Recover orphaned macOS menu-bar activity with a signal or state that distinguishes lost terminal events from valid quiet work, backed by deterministic clock-based tests and real macOS overload proof.

## Reproduction Assessment

Yes, at source level. Current main can process a job started event without a terminal event through ControlChannel into WorkActivityStore, leaving the menu-bar activity active indefinitely; I did not run a live macOS overload setup in this read-only review.

## Solution Assessment

No. A fixed idle timeout is not the narrowest maintainable fix because it conflates lost terminal events with valid quiet work; explicit orphan/disconnect handling, gateway-side terminal-event repair, or a visible stale state would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve quiet active work during stale cleanup:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:338-344`
  - body: The watchdog removes jobs/tools whenever `lastUpdate` is older than 180 seconds, but `lastUpdate` only refreshes on streaming job events and tool starts. A valid non-streaming model call or long shell command can run quietly past that threshold, so the menu can flip to idle while work is still active.
  - confidence: 0.9
- **[P2] Age the activity before expecting stale eviction:** `apps/macos/Tests/OpenClawIPCTests/WorkActivityStoreTests.swift:171-173`
  - body: This test calls `evictStaleActivities()` immediately after a fresh `started` event, but production only removes entries older than `jobStaleThreshold`. Use an injected clock or another test seam to make the activity genuinely stale before expecting idle.
  - confidence: 0.88
