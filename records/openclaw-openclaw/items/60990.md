## Summary

Keep open: this PR addresses a plausible macOS menu-bar state bug, but the proposed fixed-timeout cleanup can hide legitimate active work and one new test contradicts the production threshold.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that evicts stale job/tool display state, refreshes selected activity timestamps, exposes clear-all cleanup, and adds Swift tests for menu-bar activity state.

## Best Possible Solution

Preserve visible activity for known active quiet work while adding a state-aware recovery path for orphaned or disconnected display state, with tests for both long quiet work and missing terminal events.

## Reproduction Assessment

Yes at source level, but not as a live macOS reproduction: current main routes gateway job/tool events into WorkActivityStore and a started or streaming job remains active until a terminal event arrives. I did not establish a real provider-overload run in this read-only review.

## Solution Assessment

No. A watchdog can be part of the fix, but unconditional age-based eviction is not the safest implementation because it conflicts with documented active-session display semantics for quiet long-running work.

  - confidence: 0.88

As written, the test expects `.idle` for a job that should still be active, so it will fail or pressure the implementation toward premature eviction.
  - confidence: 0.95
