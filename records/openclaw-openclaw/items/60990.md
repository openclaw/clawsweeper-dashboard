## Summary

Keep open. Current main still has the event-driven macOS activity state that can stay working if a terminal job event is lost, but this PR's fixed 180-second eviction would also clear valid quiet work and its stale-eviction test contradicts the implementation.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that evicts stale job/tool UI state after 180 seconds, refreshes timestamps on streaming and tool-start events, exposes clearAllActivities(), and adds WorkActivityStore tests.

## Best Possible Solution

Recover orphaned macOS activity UI without clearing known-active quiet jobs solely because no progress event arrived, add a deterministic clock/test seam, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main can be reproduced at source level by sending a started job event without a terminal event; the PR regression can be reproduced by letting a valid quiet job or tool exceed 180 seconds without progress and observing the watchdog return the menu to idle.

## Solution Assessment

No. A fixed timeout is not the narrowest maintainable fix because it conflates lost terminal events with valid quiet work; a safer approach should use an explicit orphan/disconnect signal, a stale-working state, or gateway-side terminal-event repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve quiet active work during stale cleanup:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:337-344`
  - body: The watchdog removes any job/tool whose `lastUpdate` is older than 180 seconds, but timestamps are only refreshed on streaming job events and tool starts. A valid non-streaming model call or long shell command can run quietly past that threshold, so the menu can flip to idle while work is still active.
  - confidence: 0.9

  - body: This test calls `evictStaleActivities()` immediately after a fresh `started` event, but production only removes entries older than `jobStaleThreshold`. Use an injected clock or another test seam to make the activity genuinely stale before expecting idle.
  - confidence: 0.88

  - body: This is a user-facing macOS fix, but the PR changes only Swift source and tests. Repo policy requires a single-line `CHANGELOG.md` entry for user-facing fixes, with appropriate contributor credit.
