## Summary

Keep open: this PR targets a plausible macOS activity-indicator failure path, but it still needs real macOS proof and the fixed-timeout cleanup can make valid quiet work appear idle.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that periodically clears stale job/tool display state, refreshes some activity timestamps, exposes clearAllActivities(), and adds Swift tests.

## Best Possible Solution

Use state-aware orphan/disconnect handling or a stale/degraded display state that preserves quiet active work, with deterministic Swift coverage and redacted macOS proof.

## Reproduction Assessment

Yes, at source level: current main routes gateway job/tool events into WorkActivityStore and a started or streaming job remains active until a terminal event arrives. I did not establish a live macOS provider-overload reproduction in this read-only review.

## Solution Assessment

No. A fixed three-minute timeout conflates lost terminal events with valid quiet active runs; explicit orphan/disconnect handling or a state-aware stale display would be safer.

- **[P2] Preserve quiet active work during stale cleanup:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:337-344`

As written, the test fails the intended behavior or pressures the implementation toward evicting fresh active work.
