## Summary

Keep open. Current main has adjacent sandbox noVNC observer and Codex/Peekaboo/cua desktop-control paths, but it still lacks the requested Control UI embedded VNC viewer and dashboard-mediated full-desktop takeover flow; the linked draft PR is closed unmerged, and the discussion raises valid product/security design questions.

## What This Changes

Request adds a Control UI-embedded noVNC viewer and later agent-initiated full-desktop takeover flow with dashboard permissioning and temporary VNC control tokens.

## Best Possible Solution

Define a maintainer-approved desktop-control design that keeps agent actions on semantic permissioned provider paths and adds any Control UI viewer only as an authenticated, consent-gated visual preview with short-lived credentials, tests, docs, and threat-model coverage.

## Reproduction Assessment

Not applicable; this is a feature request. Source and docs inspection on current main confirms no Control UI VNC/RFB route or request_desktop_takeover RPC, while sandbox noVNC and Codex/Peekaboo desktop-control paths are separate.

## Solution Assessment

No for a straight VNC-first takeover as proposed. The safer direction is a maintainer product/security design that separates visual preview from semantic desktop-control actions and reuses permissioned provider/MCP paths where possible.
