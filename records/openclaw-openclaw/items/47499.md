## Summary

Keep this issue open. Current main has adjacent desktop-control paths through Codex Computer Use, PeekabooBridge, direct MCP, node screen commands, and sandbox-browser noVNC observer URLs, but it does not implement the requested Control UI embedded VNC/remote-desktop viewer or a permissioned dashboard-mediated agent takeover flow. The linked VNC PR was closed unmerged as dirty and had concrete security concerns, so this needs maintainer product/security review rather than cleanup closure.

## What This Changes

Request to add a Control UI embedded VNC/remote-desktop viewer and later a permissioned agent full-desktop takeover flow using VNC-like access.

## Best Possible Solution

Define a maintainer-approved desktop-control design that separates visual preview from action semantics: keep existing Codex/Peekaboo/CUA paths for agent control, and only add a Control UI viewer if it has a narrow authenticated transport, no persisted VNC secrets, explicit user consent, target allowlisting, and docs/tests for the security model.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: the Control UI docs and source search show no embedded VNC/remote-desktop viewer, while adjacent desktop-control backends are documented separately.

## Solution Assessment

Unclear/no for the proposed VNC-first path. It is useful as a visual preview idea, but current evidence and the issue discussion point toward semantic accessibility/Codex/Peekaboo-style control for agent actions, with VNC only after a security/product design is settled.
