## Summary

Keep open: current main and the latest release still block Gateway-mode TUI sends during an active run before Gateway queue handling, and the linked implementation PRs are open rather than merged fixes.

## What This Changes

The issue asks for TUI messages sent during an active run to reach Gateway queue handling instead of being blocked locally by the busy guard.

## Best Possible Solution

Pick one canonical candidate, repair its queue-state and proof blockers, and land a TUI/Gateway fix that forwards allowed active-run sends to Gateway queue policy while preserving pending-submit and stop behavior.

## Reproduction Assessment

Yes. Source inspection on current `origin/main` shows both the typed-submit helper and `sendMessage` reject Gateway active-run messages before `client.sendChat`, and current tests assert that blocking behavior.

## Solution Assessment

Yes for the requested direction, but no merged fix exists yet. The maintainable solution is to propagate effective Gateway queue mode into TUI session state and locally block only steer-mode or duplicate in-flight sends.
