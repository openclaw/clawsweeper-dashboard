## Summary

Keep open: current main and v2026.6.8 still block gateway-mode TUI active-run sends before Gateway queue handling, and the linked implementation PRs are open rather than merged fixes.

## What This Changes

The issue requests that TUI active-run sends reach Gateway queue handling instead of being rejected locally with the busy warning regardless of `/queue` mode.

## Best Possible Solution

Land or replace a single queue-mode-aware TUI/Gateway fix that forwards allowed active-run sends to Gateway policy, preserves stop/Esc and duplicate-submit safeguards, then close this issue and consolidate the duplicate report.

## Reproduction Assessment

Yes, source-level: current main rejects gateway-mode active-run sends in both the typed-submit helper and `sendMessage` before `client.sendChat` can reach Gateway queue policy. I did not run a live TUI/Gateway reproduction in this read-only review.

## Solution Assessment

Yes for the requested direction, but no merged fix exists yet. The maintainable solution is to propagate effective queue mode to the TUI and locally block only steer-mode or duplicate in-flight sends.
