## Label Justifications

- `P2`: This is a valid TUI/Gateway queue bug with limited blast radius to active-run TUI submissions rather than an outage of the whole runtime.
- `impact:message-loss`: The TUI suppresses user prompts before they reach Gateway queue handling, so followup/collect/interrupt messages are not delivered into the session.

## Summary

Keep open: current main still blocks gateway-mode TUI submissions before they reach `chat.send`, while the docs and Gateway queue code define active-run `steer`, `followup`, `collect`, and `interrupt` behavior as Gateway-owned. This is a narrow, source-reproducible TUI bug, not a cleanup close.

## What This Changes

The issue asks for gateway-mode TUI submissions during active runs to reach Gateway queue handling instead of being blocked with `agent is busy`.

## Best Possible Solution

Let gateway-mode TUI forward active-run prompts to `chat.send` so Gateway queue policy handles them, while preserving stop/Esc abort handling and duplicate-submit protection during pending registration.

## Reproduction Assessment

Yes from source inspection: current gateway-mode TUI blocks normal active-run submits in `canSubmitTuiChatMessage` and blocks slash chat sends in `sendMessage` before `client.sendChat`. I did not run a live TUI/Gateway scenario in this read-only review.

## Solution Assessment

Yes, the best fix is to remove or narrow the TUI’s gateway-mode active-run block so the existing Gateway queue implementation remains the single decision point. Adding another TUI-side queue-mode reader would duplicate policy and risk drift.
