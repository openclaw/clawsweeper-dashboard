## Summary

Keep open: current main and the latest release still block gateway-mode TUI sends before they reach Gateway queue policy, and an open closing PR already owns the direct fix path.

## What This Changes

The issue requests that TUI gateway-mode sends during an active run be forwarded to Gateway queue policy instead of being rejected locally as busy.

## Best Possible Solution

Land a queue-mode-aware TUI/Gateway fix from the linked PR path or a proven replacement, then close this issue and consolidate duplicate https://github.com/openclaw/openclaw/issues/90012.

## Reproduction Assessment

Yes. Source inspection shows both typed submit and command-handler paths reject gateway-mode active-run sends before `client.sendChat`, while the queue docs define active-run queue modes.

## Solution Assessment

Yes for the requested direction. The maintainable fix is to let Gateway queue policy handle non-steer active-run sends while preserving stop/abort and in-flight RPC guards; no merged implementation exists yet.
