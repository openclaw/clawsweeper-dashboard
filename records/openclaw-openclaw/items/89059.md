## Label Justifications

- `P2`: This is a normal-priority TUI/Gateway queue bug with a bounded active-run submission blast radius.
- `impact:message-loss`: The TUI suppresses prompts before they reach Gateway queue handling, so followup/collect/interrupt messages can be lost from the session.

## Summary

Keep open: current main and the latest release still block gateway-mode TUI active-run sends before they reach Gateway queue policy, and the linked fixing PRs are open/unmerged. The issue remains valid and should close only after a queue-mode-aware TUI/Gateway fix lands.

## What This Changes

The issue requests that gateway-mode TUI submissions during active runs reach Gateway queue handling instead of being blocked locally by the TUI busy guard.

## Best Possible Solution

Land a queue-mode-aware TUI/Gateway fix that forwards active-run gateway sends to Gateway queue policy while preserving stop/Esc abort handling and pending-registration duplicate protection, then close this issue.

## Reproduction Assessment

Yes from source inspection: the typed-submit helper and command handler both reject gateway-mode active-run sends before `client.sendChat` reaches Gateway queue policy. I did not run a live TUI/Gateway scenario in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to stop the TUI from preempting Gateway queue policy, while keeping local stop/abort handling and duplicate-submit protection in the TUI.
