## Summary

Keep open. Current main still has the reported managed-markdown replacement behavior, and the issue is paired with an open implementation PR that references this issue as its closing fix, so the correct cleanup path is to review and land or replace that PR rather than close this issue early.

## What This Changes

The issue requests that Dreaming inline managed markdown replacement stop appending duplicate Light Sleep or REM Sleep blocks when line endings or existing duplicate blocks are present.

## Best Possible Solution

Land a focused fix, preferably through PR #75495 or a replacement if review finds issues, that makes the shared managed-markdown helper CRLF/whitespace tolerant, collapses all existing blocks for a marker pair to one canonical block, and adds helper plus memory-core inline regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection and a no-write helper reproduction both show that CRLF before the start marker appends a second block and that a file with two existing blocks still contains two marker starts after replacement.

## Solution Assessment

Yes at the issue level. Fixing the shared managed-markdown helper is the narrow maintainable solution because Dreaming inline output uses that helper; a bare global flag is not sufficient unless the implementation also handles CRLF/whitespace and self-heals existing duplicates.
