## Summary

Keep open. Current main still has the reported literal-LF, non-global managed block replacement behavior, and two open PRs are already linked as closing implementation candidates, so this issue should stay open until one fix lands or the PRs are consolidated.

## What This Changes

The issue reports that `replaceManagedMarkdownBlock` appends or preserves duplicate managed dreaming blocks when CRLF or whitespace mismatches prevent matching and pre-existing duplicates are not all replaced.

## Best Possible Solution

Land one focused helper-level fix through #75495, #75810, or a replacement that matches CRLF line endings, collapses all pre-existing duplicate managed blocks to one canonical block, preserves unrelated markdown, adds regression coverage for both failure modes, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection plus a no-write Node check against the current helper reproduce both reported failure modes: CRLF input appends a second block, and duplicate input still leaves two marker starts with old content preserved.

## Solution Assessment

Yes at the issue level. Fixing the shared managed-markdown helper is the narrow maintainable path because Dreaming and memory-wiki callers already route through it; a bare `g` flag is not sufficient unless the implementation also handles CRLF and self-heals existing duplicates.
