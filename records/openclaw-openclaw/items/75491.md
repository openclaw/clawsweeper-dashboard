## Summary

Keep open. Current main still has the reported managed-markdown behavior, and the issue is already paired with an open fix PR, so the right cleanup path is to review and land or replace that linked PR rather than close this issue early.

## What This Changes

The issue requests that inline Dreaming managed markdown blocks be replaced in place without accumulating duplicate Light Sleep or REM Sleep blocks when line endings, whitespace, or pre-existing duplicates are present.

## Best Possible Solution

Land a focused fix, preferably through the linked PR or a replacement if needed, that makes the shared helper CRLF/whitespace-tolerant, collapses all existing blocks for the same marker pair to one canonical block, and adds helper plus memory-core inline regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection plus the no-write Node check reproduce both reported failure modes on current main: CRLF causes an appended second block, and a file with two existing blocks still has two marker starts after replacement.

## Solution Assessment

Yes at the issue level: fixing the shared managed-markdown helper is the narrow maintainable path because Dreaming and other memory surfaces reuse it. A bare global flag is not sufficient by itself unless the implementation also collapses duplicate managed blocks to one canonical copy.
