## Summary

Keep open. Current main still has the reported managed-markdown behavior: the helper matches only an exact LF heading separator and replaces a single match, while inline Dreaming writes through that helper. The issue is also linked to open, unmerged PR #75495, so it should stay open until that fix or a replacement lands.

## What This Changes

Fix replaceManagedMarkdownBlock so inline managed Dreaming blocks are replaced in place across line-ending or whitespace mismatches and pre-existing duplicate blocks rather than appending extra copies.

## Best Possible Solution

Land a focused fix that makes the shared managed-markdown helper tolerant of CRLF/line-ending variation, replaces all matching blocks for a marker pair, self-heals pre-existing duplicates down to one canonical block, and adds helper plus memory-core inline regression coverage with the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection and a no-write Node check against the current helper reproduce the two important cases: CRLF causes a second block to be appended, and an input with two existing blocks still has two start markers after replacement.

## Solution Assessment

Yes. Fixing the shared helper is the narrow maintainable solution because Dreaming and other memory surfaces use it; a bare `g` flag is not enough unless the implementation also handles line endings and duplicate collapse.
