## Summary

Keep open. Current main still has the reported managed-markdown behavior: the helper matches only an exact heading newline, replaces only one existing block, and the inline dreaming writer uses that helper for daily light/REM blocks. The report is narrow, source-backed, and suitable for a focused fix PR.

## What This Changes

The issue requests that managed inline dreaming blocks be replaced in place and deduplicated instead of accumulating duplicate light/REM markdown blocks in daily memory files.

## Best Possible Solution

Update the shared managed-markdown helper to tolerate CRLF/nearby marker whitespace, replace or remove all existing blocks for the same marker pair while keeping one canonical block at the first managed-block location, and add focused helper plus memory-core inline regression coverage and a user-facing changelog fix entry.

## Reproduction Assessment

Yes. Source inspection plus a no-write regex check against the current helper behavior reproduces both key paths: CRLF after the heading fails to match, and a file with two existing managed blocks still has two marker starts after replacement.

## Solution Assessment

No, the current implementation is not the best solution. Adding only a `g` flag would update every duplicate block but still leave multiple copies, so the safer fix is an explicit collapse-to-one managed-block replacement with CRLF-tolerant matching.
