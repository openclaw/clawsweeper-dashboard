## Summary

Keep open. Current main still has the reported managed-markdown replacement behavior, and open unmerged PRs #75495 and #75810 are already linked as closing implementation candidates, so the issue should remain open until one lands or is replaced.

## What This Changes

The issue requests that `replaceManagedMarkdownBlock` replace and deduplicate existing inline Dreaming managed blocks across CRLF or whitespace variants instead of appending duplicate blocks.

## Best Possible Solution

Land one focused helper-level fix, through #75495, #75810, or a replacement, that matches CRLF line endings, collapses all pre-existing duplicate managed blocks to one canonical block, preserves unrelated markdown, adds regression coverage, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection plus a no-write Node check against the current helper reproduce both reported failure modes: CRLF input appends a second block, and duplicate input still leaves two marker starts with old content preserved.

## Solution Assessment

Yes at the issue level. Fixing the shared managed-markdown helper is the narrow maintainable path because Dreaming and memory-wiki callers already route through it; a bare `g` flag is only sufficient if the implementation also handles CRLF and self-heals existing duplicates.
