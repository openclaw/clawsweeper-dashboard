## Summary

Keep open. Current main still has the reported managed-markdown replacement behavior, and open PR #75495 is already the linked implementation candidate, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests fixing managed markdown replacement so inline Dreaming memory blocks are replaced or deduplicated instead of accumulating duplicates when line endings differ or duplicate blocks already exist.

## Best Possible Solution

Land a focused helper-level fix through PR #75495 or a replacement that accepts CRLF/line-ending variants, rewrites every matching managed block into one canonical block, self-heals existing duplicates, adds regression coverage for CRLF and duplicate blocks, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection plus a no-write Node reproduction against the current helper show CRLF produces two start markers and pre-existing duplicate blocks still leave duplicate marker starts after replacement.

## Solution Assessment

Yes. Fixing the shared replaceManagedMarkdownBlock helper is the narrow maintainable path because inline Dreaming and other managed markdown writers call through that helper; a bare global flag is incomplete unless existing duplicate blocks also self-heal.
