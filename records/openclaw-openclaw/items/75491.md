## Summary

Keep open. Current main still has the reported managed-markdown replacement behavior, and an open unmerged PR is already linked as the implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for inline dreaming managed markdown replacement to tolerate line-ending/whitespace variants and remove duplicate blocks instead of appending or preserving stale copies.

## Best Possible Solution

Land a focused helper-level fix, through PR #75495 or a replacement if needed, that matches CRLF/line-ending variants, rewrites all blocks for the same marker pair into one canonical block, adds regression coverage for CRLF and pre-existing duplicates, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection plus a no-write Node check reproduce both reported failure modes on current main: CRLF causes a second marker block to be appended, and two existing blocks still leave two marker starts after replacement.

## Solution Assessment

Yes at the issue level. Fixing the shared managed-markdown helper is the narrow maintainable path because inline Dreaming calls through it; a bare global flag is not sufficient unless the implementation also self-heals existing duplicate managed blocks.
