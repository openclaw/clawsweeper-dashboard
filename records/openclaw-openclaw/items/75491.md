## Summary

Keep open. Current main still has the reported managed-markdown behavior: the shared helper matches only a single exact-LF block and the inline dreaming writer still uses that helper for daily light/REM output. An open, unmerged PR (#75495) is already linked as the implementation candidate, so this issue should stay open until that or an equivalent fix lands.

## What This Changes

The issue requests fixing `replaceManagedMarkdownBlock` so inline dreaming managed blocks are replaced or collapsed instead of duplicated when line endings differ or duplicate blocks already exist.

## Best Possible Solution

Land a focused helper-level fix that tolerates CRLF or nearby marker whitespace, collapses all existing blocks for the same marker pair to one canonical block, and adds both SDK helper regression tests and memory-core inline dreaming coverage, plus a user-facing changelog entry if the change ships in a release.

## Reproduction Assessment

Yes. Source inspection plus the read-only harness reproduce the two important paths on current main: CRLF after the heading appends a second block, and files with two existing blocks still retain two marker starts after replacement.

## Solution Assessment

No for current main. A plain `g` flag is not enough if it still leaves multiple duplicate blocks; the maintainable fix should normalize matching and deliberately collapse the managed block set to one canonical block.
