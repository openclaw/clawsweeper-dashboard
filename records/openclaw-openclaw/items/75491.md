## Summary

Keep open. Current main still has the reported managed-markdown replacement behavior, and two open PRs already reference this issue as closing implementation candidates, so this should remain open until one fix lands or is consolidated.

## What This Changes

Fix `replaceManagedMarkdownBlock` so inline Dreaming managed blocks are replaced once, tolerate CRLF/whitespace variation, and collapse pre-existing duplicate marker blocks.

## Best Possible Solution

Land one focused helper-level fix, through #75495, #75810, or a replacement, that matches CRLF line endings, collapses all pre-existing duplicate managed blocks to one canonical block, preserves unrelated markdown, adds regression coverage, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection plus a no-write Node check against the current helper reproduce both reported cases: CRLF input appends a second block, and duplicate input leaves stale managed blocks behind.

## Solution Assessment

Yes at the issue level. Fixing the shared managed-markdown helper is the narrow maintainable path because Dreaming and memory-wiki callers already route through it; a bare `g` flag is only sufficient if the implementation also handles CRLF and self-heals existing duplicates.
