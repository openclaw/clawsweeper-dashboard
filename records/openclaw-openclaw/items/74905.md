## Summary

Keep open. The PR is pointed in the right direction for the APNs raw HTTP/2 proxy bypass and the previous live-shard wiring feedback is addressed, but the latest head regresses managed-proxy lifecycle semantics by rejecting same-URL nested handles instead of preserving handle-count safety.

## What This Changes

The PR routes direct APNs HTTP/2 delivery through a managed-proxy CONNECT tunnel and adds APNs proxy validation, docs, lint/OpenGrep guardrails, release-live wiring, and regression coverage.

## Best Possible Solution

Keep the APNs CONNECT wrapper and release-live wiring, but restore same-URL handle-counted active proxy registrations while rejecting only conflicting proxy URLs.

## Reproduction Assessment

Yes. Current main's APNs bypass is source-reproducible at `src/infra/push-apns.ts:662`, and the PR-head blocker is source-reproducible from `src/infra/net/proxy/proxy-lifecycle.ts:393` rejecting any active proxy despite current-main overlapping-handle coverage.

## Solution Assessment

No. The APNs wrapper is the right narrow fix and the prior release-shard blocker is addressed, but the active-proxy lifecycle should preserve handle-counted same-URL nesting instead of throwing on every second activation.
