## Summary

Keep open. Current main still has the APNs managed-proxy bypass this PR is trying to fix, but the latest PR head remains incorrect for security-sensitive managed egress code because the custom CONNECT tunnel still has unbounded response buffering and an exception path that bypasses cleanup. The protected `security` and `maintainer` labels plus the MEMBER author association also mean this should stay in maintainer/security review rather than cleanup-close.

## What This Changes

The PR routes direct APNs HTTP/2 delivery through a new APNs connection wrapper and active managed-proxy state, then adds CONNECT tunnel code, raw-HTTP/2 lint/OpenGrep guardrails, APNs proxy regression tests, docs, and a changelog entry.

## Best Possible Solution

Keep the APNs-specific proxy seam, but harden the CONNECT layer before merge: catch request-build/write failures through the normal cleanup path, enforce a fixed CONNECT response header limit, and either use the maintained `https-proxy-agent` path described in the PR or document and test the custom tunnel contract clearly.

## Reproduction Assessment

Yes. Current main’s bypass is reproducible by source inspection of `sendApnsRequest`, and the PR defects are reproducible with malformed percent-encoded proxy credentials or a proxy that streams an unterminated CONNECT response header.

## Solution Assessment

No, not as-is. The PR direction is narrow and maintainable, but custom managed-egress tunneling needs bounded parsing and predictable cleanup before it is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route CONNECT write errors through fail():** `src/infra/net/http-connect-tunnel.ts:116`
  - body: `writeConnectRequest()` decodes proxy credentials and can throw for malformed percent escapes. Because `onConnected()` calls it directly from the socket event handler, that exception bypasses `fail()`, skips socket cleanup, and can crash the process instead of rejecting the APNs send.
  - confidence: 0.91
- **[P2] Cap CONNECT response headers before buffering:** `src/infra/net/http-connect-tunnel.ts:121-123`
  - body: `onData()` appends every chunk to `responseBuffer` until it sees `\r\n\r\n`, with no maximum size. A bad proxy can keep sending unterminated header bytes and grow memory until timeout or OOM; enforce a fixed CONNECT header limit and fail the tunnel when exceeded.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR introduces custom CONNECT tunneling for managed egress and still has concrete memory-growth and cleanup/error-path concerns.

Concerns:

- **[medium] Unbounded CONNECT response buffering:** `src/infra/net/http-connect-tunnel.ts:121`
  - body: The tunnel accumulates proxy response bytes without a maximum header size, so a bad proxy response can consume memory before timeout; this matters because the PR changes managed egress security behavior.
  - confidence: 0.89
- **[low] Malformed proxy credentials can escape cleanup:** `src/infra/net/http-connect-tunnel.ts:116`
  - body: Malformed percent-encoded proxy credentials can throw during CONNECT request construction from an event handler, bypassing the Promise rejection path and socket cleanup.
  - confidence: 0.91
