## Summary

Keep open. Current main still bypasses managed proxy for direct APNs delivery, but this member-authored PR has protected `security`/`maintainer` context and the latest head still has concrete CONNECT tunnel hardening issues before merge.

## What This Changes

The PR routes direct APNs HTTP/2 sends through an active managed proxy via a new APNs connection wrapper/CONNECT tunnel, active proxy state, proxy validation flags/docs, lint/OpenGrep guardrails, tests, and a changelog entry.

## Best Possible Solution

Keep the APNs-specific proxy seam, but harden the CONNECT tunnel with bounded parsing and cleanup-safe error handling before maintainer/security review decides whether to merge this protected PR.

## Reproduction Assessment

Yes. Current main’s bypass is reproducible by source inspection of `sendApnsRequest`, and the PR defects are reproducible with malformed percent-encoded proxy credentials or a proxy that streams an unterminated CONNECT response header.

## Solution Assessment

No, not as-is. The APNs wrapper direction is narrow, but the custom CONNECT layer needs predictable cleanup and a fixed response-header limit before it is the safest solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route CONNECT write failures through fail():** `src/infra/net/http-connect-tunnel.ts:116`
  - body: `writeConnectRequest()` decodes proxy credentials and can throw for malformed percent escapes. Because `onConnected()` calls it directly from the socket event handler, that exception bypasses `fail()`, skips the normal cleanup/rejection path, and can crash instead of returning an APNs/proxy validation failure.
  - confidence: 0.91
- **[P2] Cap CONNECT response headers before buffering:** `src/infra/net/http-connect-tunnel.ts:121-123`
  - body: `onData()` appends every chunk to `responseBuffer` until it sees `\r\n\r\n`, with no maximum size. A bad proxy can keep sending unterminated header bytes and grow memory until timeout or OOM, so the tunnel should enforce a fixed CONNECT header limit and fail closed.
  - confidence: 0.9
- **[P3] Allowlist the test-only PEM fixture:** `src/infra/push-apns.test.ts:18`
  - body: The new test embeds a private-key PEM without the repo's `pragma: allowlist secret` marker used for known fake secrets. Add the marker or move the material into an existing fixture pattern so secret scanning does not produce noisy alerts for this security PR.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention: the PR changes managed egress behavior and still has concrete CONNECT parsing/cleanup issues plus an unallowlisted test key fixture.

Concerns:

  - body: The tunnel accumulates proxy response bytes without a maximum header size, so a bad proxy response can consume memory before timeout; this matters because the PR changes managed egress security behavior.
