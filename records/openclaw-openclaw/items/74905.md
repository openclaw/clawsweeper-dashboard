## Summary

Keep open: the PR addresses a real managed-proxy APNs bypass, but the latest head introduces a blocking APNs validation false positive by accepting any HTTP/2 403 without APNs-specific proof.

## What This Changes

The PR routes direct APNs HTTP/2 delivery through a managed-proxy CONNECT tunnel, adds active proxy lifecycle state, APNs proxy validation CLI/docs, raw HTTP/2 guardrails, live workflow coverage, changelog text, and regression tests.

## Best Possible Solution

Fix the APNs validation false positive, rebase onto current main, then land after exact-head CI and maintainer/security review while keeping broader raw-socket classification in #77126.

## Reproduction Assessment

Yes. Current main source shows direct APNs sends use raw http2.connect(), and the latest PR head source shows a 403 without apns-id is accepted as reachable.

## Solution Assessment

No. The APNs tunnel wrapper is the right narrow direction, but the validation fallback should require APNs-specific evidence before treating a 403 as success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Require APNs proof before passing 403 reachability:** `src/infra/net/proxy/proxy-validation.ts:425`
  - body: The latest check treats any HTTP/2 403 as APNs-reachable when `apns-id` is missing. A proxy or WAF can generate a 403 without tunneling to APNs, so `openclaw proxy validate --apns-reachable` can report success even though the docs describe APNs' `InvalidProviderToken` response as the success signal. Keep the no-`apns-id` fallback tied to APNs-specific proof, such as the response body reason, or fail it.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff is security-sensitive and has one concrete validation false positive that can make APNs proxy reachability look proven when it is not.

Concerns:

- **[medium] APNs reachability can pass on arbitrary 403 responses:** `src/infra/net/proxy/proxy-validation.ts:425`
  - body: `runApnsReachabilityCheck()` accepts status 403 without `apns-id` or body validation, so an intercepting proxy response can satisfy a security preflight meant to prove APNs is reachable through CONNECT.
