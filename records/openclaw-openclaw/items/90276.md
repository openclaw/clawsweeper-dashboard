## Summary

Keep open: current main and v2026.6.8 still reject a 300000ms top-level TTS timeout, and the configured-provider fallback/stickiness behavior still needs a maintainer policy decision.

## What This Changes

The issue asks OpenClaw to support longer local TTS timeouts, honor timeout precedence, and avoid silent voice-provider fallback when local TTS generation exceeds 120s.

## Best Possible Solution

Make slow configured local TTS timeouts valid and effective, then make fallback diagnostics or provider-stickiness explicit under maintainer-approved TTS semantics.

## Reproduction Assessment

Yes by source inspection: current main rejects `messages.tts.timeoutMs: 300000`, local CLI still has a 120s default, and speech-core still falls through to fallback providers after primary failure. I did not run the reporter's local Apple Silicon TTS model.

## Solution Assessment

Unclear as a complete fix: raising the cap and aligning timeout precedence are narrow repairs, but fallback/stickiness needs a maintainer decision because automatic fallback is documented current behavior.
