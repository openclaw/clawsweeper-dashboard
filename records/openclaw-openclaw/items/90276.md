## Summary

Keep open: current main and v2026.6.9 still reject a 300000 ms top-level TTS timeout, and automatic fallback/provider-stickiness semantics still need maintainer policy before changing user-visible voice behavior.

## What This Changes

The issue requests raising or removing the 120s `messages.tts.timeoutMs` cap, making slow local TTS timeouts effective, and making fallback diagnostics or provider switching explicit.

## Best Possible Solution

Choose the accepted TTS timeout ceiling and fallback/stickiness policy, then align schema, docs, and regression coverage so slow local providers do not unexpectedly change voice identity.

## Reproduction Assessment

Yes, by source inspection: current main rejects `messages.tts.timeoutMs: 300000` and still automatically tries fallback providers after primary failure. I did not run the reporter's local Apple Silicon TTS model.

## Solution Assessment

Unclear as a complete fix: raising the cap is narrow, but fallback/stickiness semantics need maintainer direction because automatic provider fallback is documented current behavior.
