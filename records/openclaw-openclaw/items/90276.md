## Summary

This should stay open: current main still rejects `messages.tts.timeoutMs` above 120000ms, and the local CLI provider has its own 120s default while automatic fallback can continue to another speech provider after failure.

## What This Changes

The issue asks OpenClaw to make slow local TTS timeouts valid and effective, avoid silent cross-provider voice fallback, and surface timeout failures clearly.

## Best Possible Solution

Make slow configured local TTS timeouts valid and effective across schema, speech-core, and `tts-local-cli`, then make the fallback behavior explicit enough that configured voice identity is not silently replaced.

## Reproduction Assessment

Yes, by source inspection: current schema rejects `messages.tts.timeoutMs: 300000`, `tts-local-cli` has a 120s provider default, and speech-core can continue to fallback providers after a failed primary attempt. I did not run the reporter's local voice-cloning model.

## Solution Assessment

Unclear as a complete fix: raising the timeout cap and making local CLI honor the resolved timeout is a narrow repair, but changing default fallback semantics for explicit voice identity needs maintainer policy judgment.
