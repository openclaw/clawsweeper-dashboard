## Summary

Keep open: current main and v2026.6.8 still reject a 300000ms top-level TTS timeout, and the remaining fallback/stickiness behavior is a maintainer policy decision rather than a completed fix.

## What This Changes

The issue asks OpenClaw to support longer local TTS timeouts, make configured provider timeouts effective, and avoid silent voice-identity fallback to a different TTS provider.

## Best Possible Solution

Make slow configured local TTS timeouts valid and effective, then make fallback diagnostics or fallback policy explicit so voice identity does not silently switch without maintainer-approved semantics.

## Reproduction Assessment

Yes, by source inspection: current main still rejects `messages.tts.timeoutMs: 300000`, the local CLI provider still carries a 120s default, and speech-core can still continue to fallback providers after a primary failure. I did not run the reporter's local Apple Silicon TTS model.

## Solution Assessment

Unclear as a complete fix: raising the cap and aligning timeout precedence are narrow repairs, but the fallback/stickiness behavior needs a maintainer decision because automatic fallback is documented current behavior.
