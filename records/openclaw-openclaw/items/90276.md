## Summary

This should stay open: current main and the latest release still keep the 120s top-level TTS timeout cap, and the local CLI provider still uses its own provider-config/default timeout rather than the resolved request timeout. The related timeout PR is closed unmerged, so this issue remains the canonical place for the remaining timeout and fallback policy decision.

## What This Changes

The issue asks OpenClaw to support longer and effective TTS timeouts for slow local CLI voice-cloning providers, and to avoid silently falling back to a different voice after timeout.

## Best Possible Solution

Make slow configured local TTS timeouts valid and effective across schema, speech-core, and `tts-local-cli`, then make fallback diagnostics or policy explicit before silently switching voice identity.

## Reproduction Assessment

Yes, by source inspection: current schema rejects `messages.tts.timeoutMs: 300000`, `tts-local-cli` still has a 120s default and uses provider config for the child process, and speech-core can continue to fallback providers after failure. I did not run the reporter's local model.

## Solution Assessment

Unclear as a complete fix: raising the cap and making local CLI honor the resolved request timeout are narrow repairs, but fallback behavior for voice identity needs maintainer policy judgment.
