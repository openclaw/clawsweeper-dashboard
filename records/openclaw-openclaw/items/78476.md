## Summary

Keep open: current main and v2026.6.8 still have no Android Google Home/Home APIs bridge, and the request is not fully covered by the broader node-tool design thread because it also needs native mobile SDK, OAuth/consent, and physical-device safety decisions.

## What This Changes

Request native Google Home/Home APIs support in the Android companion app so Talk can discover and control authorized smart-home devices without requiring Home Assistant.

## Best Possible Solution

Maintainers should first choose the product boundary: a generic Android/local capability contract plus plugin/ClawHub ownership is safer unless direct Google Home support earns a core Android exception with OAuth, revocation, and high-risk-device confirmation requirements.

## Reproduction Assessment

Yes, at source level: current main has no Android Google Home/Home APIs capability, command registry entries, Gateway node policy, or docs for this bridge. This is a missing feature rather than a broken existing behavior with a runtime repro.

## Solution Assessment

Unclear; a direct Google-specific Android core bridge is plausible but not proven best. Maintainers should compare a generic Android local-capability or node-tool contract plus external plugin/ClawHub ownership before adding vendor-specific core behavior.
