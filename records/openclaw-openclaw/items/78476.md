## Summary

Keep open: current main and the latest release still lack any Google Home/Home APIs bridge in the Android app, and the request is not fully superseded by the broader node-tool issue because it also needs mobile SDK, OAuth/permission, and physical-device safety decisions.

## What This Changes

Add native Google Home API support to the Android app so Talk can discover, read, and control a user's authorized Google Home devices directly.

## Best Possible Solution

Maintainers should first choose the product shape: a generic Android/local capability contract with explicit node policy and plugin/ClawHub ownership is safer unless direct Google Home support earns a core Android exception with OAuth, consent, revocation, and high-risk-device confirmation requirements.

## Reproduction Assessment

Yes at source level for the missing capability: Android advertises current node commands and Gateway policy has no Google Home/Home APIs command or capability. This is a feature gap, not a broken existing behavior with a runtime repro.

## Solution Assessment

Unclear; a direct Google-specific Android core bridge is plausible but not proven best. The safer design pass should compare a generic Android local-capability or node-tool contract plus external plugin/ClawHub ownership before adding vendor-specific core behavior.
