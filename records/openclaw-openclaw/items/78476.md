## Summary

Keep open. Current main has Android node and Talk plumbing, but no Google Home/Home APIs dependency, permission flow, Android capability, or smart-home command surface; this is broad first-party mobile/protocol work with consent, safety, and routing decisions still unresolved.

## What This Changes

Request native Google Home/Home APIs support in the Android app so Talk can list, read, and control authorized smart-home devices without requiring Home Assistant.

## Best Possible Solution

Define a maintainer-approved smart-home capability boundary first, then decide whether Google Home belongs in the first-party Android app, a separate node/plugin path, or after node-registered agent tools land, with explicit consent, revocation, routing, and dangerous-action confirmation semantics.

## Reproduction Assessment

Not applicable; this is a feature request, not broken existing behavior. Source inspection on current main shows Android node/Talk support exists, but no Google Home bridge or smart-home command surface exists to reproduce.

## Solution Assessment

Unclear as an implementation path. The user problem is valid, but the maintainable solution needs product design around Android SDK ownership, normalized smart-home schema, node/Talk routing, and sensitive-action policy before a patch is safe.
