The PR adds memory dreaming cron/timezone validation in the memory host SDK, wires memory-core reconciliation/runtime guards around invalid scheduling config, updates SDK metadata, and adds focused tests.

## Best Possible Solution

Rebase onto current main and land the validator with preserved schedule state represented as every retained managed timezone, or an explicit ambiguous state, with maintainer-owned release notes handled during landing.

## Reproduction Assessment

Yes from source inspection. A static reproduction is two retained managed dreaming jobs with different schedule timezones, an invalid parse-only frequency, and runtime config timezone matching the non-primary retained job; the PR stores only the primary timezone and gates runtime against that single value.

## Solution Assessment

No, not yet. The validator direction is narrow and maintainable, but the last-known-good fallback needs to track all retained managed schedule timezones or mark mixed duplicates ambiguous before runtime promotion can safely compare timezone state.
