## Summary

Keep open. Current main still has the underlying validation gap, but the latest PR head is not merge-ready because preserved duplicate schedules are tracked through only one timezone and can misgate runtime execution.

## What This Changes

The PR adds a Memory Dreaming cron/timezone validator, wires memory-core reconciliation/runtime handling around invalid scheduling config, updates SDK metadata/baseline hashes, and adds focused tests.

## Best Possible Solution

Land the validator on top of current main after carrying preserved schedule state in a way that handles all retained managed jobs, preserving current cron-trigger/runtime-retry behavior and adding the required user-facing changelog entry.

## Reproduction Assessment

Yes. A static reproduction is two retained managed dreaming jobs with different schedule timezones, an invalid parse-only frequency, and a runtime config timezone matching the non-primary job; the PR records only the primary timezone and can skip or allow promotion for the wrong preserved schedule.

## Solution Assessment

No, not yet. The validator direction is narrow and maintainable, but the preserved-schedule timezone state should represent all retained jobs or explicitly treat mixed duplicates as ambiguous before this is safe to merge.
