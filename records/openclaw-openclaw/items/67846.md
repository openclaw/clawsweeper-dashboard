## Summary

Keep open: current main still lacks upfront dreaming cron validation, but this PR is not merge-ready because the invalid-config fallback records only the primary preserved timezone while intentionally retaining duplicate managed jobs, and external real behavior proof is missing.

## What This Changes

The PR adds memory dreaming cron/timezone validation in the memory host SDK, wires memory-core reconciliation/runtime guards around invalid scheduling config, updates SDK metadata, and adds focused tests.

## Best Possible Solution

Rebase onto current main and land the validator with preserved schedule state covering all retained managed job timezones, or an explicit ambiguous state, plus focused tests and maintainer-owned release notes.

## Reproduction Assessment

Yes, from source inspection. A static reproduction is two retained managed dreaming jobs with different schedule timezones, an invalid parse-only frequency, and runtime config timezone matching the non-primary retained job; the PR preserves both but compares only the primary timezone.

## Solution Assessment

No, not yet. The validator direction is narrow, but the last-known-good fallback needs to track every retained managed schedule timezone or mark mixed duplicates ambiguous before runtime promotion can compare timezone state safely.
