## Summary

Keep open: the validator is a useful bug fix that is not on current main, but the latest PR head still has a correctness bug in the preserved-schedule fallback and lacks required real behavior proof from the contributor.

## What This Changes

The PR adds memory dreaming cron/timezone validation in the memory host SDK, wires memory-core reconciliation/runtime guards for invalid scheduling config, updates SDK documentation metadata/baseline hashes, and adds focused tests.

## Best Possible Solution

Rebase onto current main and land the validator with preserved schedule state that covers every retained managed job timezone, or explicitly treats mixed retained schedules as ambiguous, plus focused regression tests and maintainer-owned release notes.

## Reproduction Assessment

Yes, from source inspection. A static reproduction is two retained managed dreaming jobs with different schedule timezones, an invalid parse-only frequency, and runtime config timezone matching the non-primary retained job; the PR preserves both but compares only the primary timezone.

## Solution Assessment

No, not yet. The validator direction is narrow and appropriate, but the fallback must track all retained schedule timezones or explicitly block ambiguous mixed duplicates before runtime promotion compares timezone state.
