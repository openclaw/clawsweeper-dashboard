## Summary

Keep open: this is a plausible narrow fix for a real session-store corruption path, but the latest head still has two merge blockers around backup-file validation and fs-safe temp fixture coverage.

## What This Changes

The PR adds load-time gateway session-store recovery from `.bak` and residual `.tmp` artifacts, synchronous self-heal writeback, recovery tests, and a changelog entry.

## Best Possible Solution

Keep the load-time recovery direction, but require one shared recovery-candidate safety gate before every read and tests that exercise the pinned `.fs-safe-replace.<pid>.<uuid>.tmp` artifact path.

## Reproduction Assessment

Yes, source-reproducible: current main initializes an empty store and catches the final read/parse failure without recovery, so a zero-byte or malformed `sessions.json` can come back as an empty store. I did not run a mutating repro because this review was read-only.

## Solution Assessment

No, not yet. Load-time recovery is a narrow maintainable fix, but the proposed patch needs backup candidate pre-read validation and corrected fs-safe fixtures before it is mergeable.
