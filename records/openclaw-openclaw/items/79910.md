## Summary

Keep open: this is a plausible data-loss recovery fix with sufficient real-behavior proof, but the current PR still discards fresh valid temp artifacts, which can be the only complete copy immediately after a crash-before-rename restart.

## What This Changes

The PR adds load-time recovery for corrupt `sessions.json` from `.bak` and `.tmp` artifacts, self-heals the primary store, adds recovery tests, and updates the changelog.

## Best Possible Solution

Keep the loader-level recovery, but rank fresh valid tmp artifacts as last-resort candidates after `.bak` and stale tmp while preserving the existing normalization and file-safety gates.

## Reproduction Assessment

Yes, source-reproducible: current main returns an empty store after a zero-byte or malformed `sessions.json`, and the linked incident provides a real Linux 0-byte store report. I did not run a mutating filesystem repro in this read-only review.

## Solution Assessment

No, not yet. The loader boundary and safety checks are the right direction, but discarding fresh complete tmp files leaves an immediate post-crash recovery gap; they should be accepted only as the lowest-priority valid candidates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep fresh tmp artifacts as last-resort recovery:** `src/config/sessions/store-load.ts:499-500`
  - body: This skips every `.tmp` younger than 10 seconds.
