## Label Justifications

## Summary

Keep this PR open for maintainer review: current main still treats a zero-byte or malformed primary session store as an empty store, and this branch is a plausible narrow recovery path, but the session-state fallback semantics and coordination with stale-temp deletion need maintainer acceptance before merge.

## What This Changes

The PR changes session-store loading to recover in memory from stale `sessions.json.<pid>.<uuid>.tmp` and legacy temp artifacts when the primary store is zero-byte or malformed.

## Best Possible Solution

Land the narrowed stale-temp recovery only after maintainers confirm the fallback semantics, coordinate it with stale-temp deletion behavior, and track doctor/rebuild diagnostics separately if still required.

## Reproduction Assessment

Yes. Source inspection shows current `loadSessionStore` returns the initialized empty store after a zero-byte or malformed `sessions.json`, though this read-only review did not run a local filesystem repro.

## Solution Assessment

Yes for the narrowed stale-temp failure, with maintainer acceptance required. The PR uses the existing atomic-writer artifact contract and avoids `.bak` promotion or load-time primary rewrites, but it is not the full doctor/rebuild recovery requested by the canonical issue.
