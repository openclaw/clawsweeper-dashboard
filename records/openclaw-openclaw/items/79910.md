## Summary

Keep open: the recovery direction is useful and the supplied logs are adequate real-behavior proof, but the latest head still accepts any object-shaped `.bak` as a recovered session store and can mask a later valid temp candidate.

## What This Changes

The PR adds load-time session-store recovery from `.bak` and stale temp artifacts, synchronous self-heal, recovery tests, and a changelog entry.

## Best Possible Solution

Keep the load-time recovery boundary, but apply the same session-entry-shaped validation to `.bak` candidates, continue scanning when a candidate is empty or unrelated, and format the touched files before merge.

## Reproduction Assessment

Yes, source-reproducible: current main catches a failed `sessions.json` read/parse and returns an empty store with no recovery path. The linked report and PR proof provide concrete zero-byte corruption and after-fix recovery evidence, but this read-only review did not run a mutating repro.

## Solution Assessment

No, not yet. Load-time recovery from existing local artifacts is the narrow maintainable solution, but `.bak` candidates need the same content-shape validation already added for tmp candidates.
