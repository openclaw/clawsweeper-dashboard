## Label Justifications

- `P1`: The proposed recovery can silently replace newer canonical session metadata during a concurrent save.
- `merge-risk: 🚨 session-state`: Merging can roll session state back by promoting stale sibling data over a newer valid primary.

## Summary

This recovery remains necessary because current `main` still returns an empty store for readable zero-byte or malformed JSON, but the patch is not merge-safe: its unconditional self-heal can overwrite a newer valid store installed by a concurrent normal save.

## What This Changes

The PR adds load-time recovery of an unusable session store from validated backup or stale temporary artifacts and atomically self-heals the primary file.

## Best Possible Solution

Run recovery through the canonical session-store writer boundary and require the primary to still match the observed unusable state immediately before atomic replacement; accept only recovery artifacts with a documented writer or migration lifecycle.

## Reproduction Assessment

Yes. The existing corrupt-store symptom is source-reproducible, and the introduced race is deterministic by installing a newer valid primary after candidate selection but before the unconditional recovery rename.

## Solution Assessment

No. Artifact recovery addresses a real failure mode, but load-time replacement outside the existing writer synchronization boundary is not the safest implementation.
