## Summary

Keep open: this is a plausible sessions data-loss fix with sufficient real-behavior proof, but the latest branch still has a recovery-candidate normalization bug that should block merge.

## What This Changes

The PR adds load-time recovery from `sessions.json.bak` and stale session-store `.tmp` artifacts, self-heals the primary store, and adds focused recovery tests.

## Best Possible Solution

Keep the loader-level recovery, but normalize each candidate with the existing store contract, drop rejected entries, and only self-heal non-empty normalized candidates.

## Reproduction Assessment

Yes. Current-main source shows a malformed or zero-byte `sessions.json` falls through as an empty store, and the linked report plus PR proof describe the crash-mid-write scenario; I did not execute a live repro in this read-only review.

## Solution Assessment

No, not yet. The loader boundary is the right place, but the candidate-normalization repair should reuse or exactly mirror `normalizeSessionStore` semantics before the PR is mergeable.
