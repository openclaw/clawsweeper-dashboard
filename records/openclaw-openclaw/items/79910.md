## Summary

Keep open: the prior `.bak` validation blocker is addressed at the latest head, but the temp recovery scan still restores the oldest stale valid artifact first, which can resurrect outdated session metadata when a newer completed temp artifact is also present.

## What This Changes

The PR adds load-time session-store recovery from validated `.bak` and stale temp artifacts, synchronous self-heal writeback, recovery tests, and a changelog entry.

## Best Possible Solution

Keep the load-time recovery boundary, but choose the newest stale valid temp artifact after the safety and freshness gates, and add a regression for competing stale valid temp files.

## Reproduction Assessment

Yes, source-reproducible: current main catches failed `sessions.json` reads/parses and returns an empty store with no recovery path. The linked issue provides zero-byte file evidence and the PR body provides after-fix recovery logs, but this read-only pass did not run a mutating repro.

## Solution Assessment

No, not yet. Validated load-time recovery from local artifacts is the narrow maintainable direction, but temp candidate ordering should prefer the newest stale valid artifact to avoid restoring older session metadata.
