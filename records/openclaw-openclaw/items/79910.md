## Label Justifications

## Summary

Keep open: current main and the latest release still return an empty session store for zero-byte or malformed primary JSON, while this branch fixes that live problem and appears to have addressed the previous cache-path blocker. The remaining gate is maintainer acceptance of the session-state recovery policy for promoting validated `.bak` or stale `.tmp` artifacts.

## What This Changes

The branch teaches `loadSessionStore` to recover corrupted primary session stores from validated `.bak` and stale session-store temp artifacts, shares the cleanup stale window, self-heals the primary file, and adds recovery tests.

## Best Possible Solution

Land the bounded recovery only if maintainers accept promoting validated sibling artifacts after readable-but-unusable primary content; otherwise narrow the policy before merge rather than leaving corrupted stores to silently load empty.

## Reproduction Assessment

Yes from source inspection: current main and the latest release initialize an empty store and swallow the final read/parse failure for zero-byte or malformed `sessions.json`. The PR body and tests also exercise the same failure shape with real files.

## Solution Assessment

Yes, with maintainer policy acceptance: the implementation keeps valid stores and primary read errors from triggering recovery, validates candidate shape, shares the cleanup stale window, and adds focused regression coverage. The safer alternative would be a narrower recovery source if maintainers do not want automatic stale artifact promotion.
