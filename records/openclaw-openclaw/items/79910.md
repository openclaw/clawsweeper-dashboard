## Summary

Keep open: the recovery direction is valid and the PR has useful real-behavior proof, but the latest diff still accepts backup/tmp candidates before applying the current session-store validators, so a corrupted artifact can block fallback to a valid recovery source.

## What This Changes

The PR adds load-time session-store recovery from `.bak` and stale `.tmp` artifacts, self-heals `sessions.json`, and adds focused recovery tests plus a changelog entry.

## Best Possible Solution

Keep the loader-level recovery design, but accept and self-heal from a candidate only after the existing session-store migration and normalization rules leave recoverable entries.

## Reproduction Assessment

Yes. Current main is source-reproducible: a zero-byte or malformed existing `sessions.json` makes the final read/parse fail and `loadSessionStore` proceeds with the initialized empty store; the PR body also includes after-fix runtime logs for the recovery path.

## Solution Assessment

No, not yet. The loader boundary is the right place, but recovery candidate selection should reuse the current validator pipeline before choosing `.bak` or `.tmp` and before self-healing the primary file.
