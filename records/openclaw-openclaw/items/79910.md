## Summary

Keep open: the recovery direction is valid and the PR body now has sufficient real behavior proof, but the latest head still has merge blockers. The fs-safe temp tests use a filename that does not match the pinned writer, and `.bak` recovery still reads the candidate before applying the stated lstat-based symlink/hardlink gate.

## What This Changes

The PR adds load-time recovery for corrupted Gateway session stores from `.bak` and `.tmp` artifacts, plus recovery tests and a changelog entry.

## Best Possible Solution

Keep the load-time recovery approach, but make every recovery candidate pass the same local-file safety gate and align tests with the actual pinned fs-safe temp naming contract before merge.

## Reproduction Assessment

Yes, source-reproducible: current `main` returns an empty store after the final `sessions.json` read/parse failure, and the linked issue gives a concrete zero-byte incident. I did not run a mutating repro in this read-only review.

## Solution Assessment

No. Load-time recovery is a narrow maintainable fix, but this PR is not yet the best mergeable form until the real fs-safe temp path is tested and `.bak` recovery is protected by the same pre-read safety checks.
