## Summary

Keep open: this is a useful recovery-only PR with sufficient real-behavior proof, but the current diff still drops the immediate post-crash case where the only valid recovery artifact is a fresh temp file, so it can return an empty store before the artifact becomes stale.

## What This Changes

The branch adds load-time session-store recovery from validated `.bak` and stale `.tmp` artifacts, synchronous self-heal writeback, recovery tests, and a changelog entry.

## Best Possible Solution

Land a recovery-only fix that tries `.bak`, stale temp artifacts, and a valid fresh orphan temp as a last-resort fallback while preserving active-writer safety and the existing save hot path.

## Reproduction Assessment

Yes, source-reproducible: current main catches zero-byte or malformed `sessions.json` read/parse failures and returns `{}`. The remaining PR-specific gap is also source-reproducible because fresh matching temp artifacts are skipped before any candidate recovery runs.

## Solution Assessment

No. The recovery-only direction is right, but this implementation should not return an empty store when the only valid post-crash artifact is a fresh temp; use a lowest-priority fresh-temp fallback or an explicit retry/defer path instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recover fresh orphan temps before returning empty:** `src/config/sessions/store-load.ts:236`
  - body: When the primary store is corrupt and the only recovery artifact is the temp left by the interrupted write, that temp will often be younger than 10 seconds on an immediate gateway restart. This line skips it, then `loadSessionStore` returns and caches `{}`, so the next session save can overwrite the store before the artifact ever becomes eligible. Treat a valid fresh temp as a last-resort fallback or defer/retry instead of discarding the only recoverable copy.
  - confidence: 0.9
