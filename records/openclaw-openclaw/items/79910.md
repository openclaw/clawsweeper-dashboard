## Summary

Keep open: the bug is valid and the PR is close, but the latest head can still treat valid non-object `sessions.json` content such as `[]` as recoverable corruption and resurrect stale artifacts.

## What This Changes

The PR adds load-time recovery of a corrupted gateway `sessions.json` from validated `.bak` and stale `.tmp` artifacts, writes the recovered store back, and adds recovery tests plus a changelog entry.

## Best Possible Solution

Keep the load-time recovery boundary, but gate artifact recovery to actual primary read/parse failures or explicitly preserve valid reset-store semantics such as `[]`, with a regression test.

## Reproduction Assessment

Yes, source-reproducible: current main catches failed `sessions.json` reads/parses and returns an empty store with no artifact recovery. The linked issue provides zero-byte file evidence and the PR body provides after-fix recovery logs, but this read-only pass did not run a mutating repro.

## Solution Assessment

No, not yet. Validated load-time recovery from local artifacts is the right direction, but the gate should not resurrect stale data when the primary file parsed successfully as a valid reset/empty shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
