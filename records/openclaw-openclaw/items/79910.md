## Summary

Keep open: the reported no-recovery failure mode still exists on current main, and this PR is directionally useful, but the latest head still has two merge blockers in recovery safety and test coverage.

## What This Changes

Adds load-time session-store recovery from `.bak` and temp artifacts, synchronous self-heal writeback, focused recovery tests, and a changelog entry.

## Best Possible Solution

Keep the load-time recovery direction, but make every recovery candidate pass one local-file safety gate before reading and align fixtures with the pinned fs-safe temp naming before merge.

## Reproduction Assessment

Yes, source-reproducible: current main returns an empty store after the final `sessions.json` read/parse failure, and the linked bug gives a concrete zero-byte incident. I did not run a mutating repro in this read-only review.

## Solution Assessment

No, not yet. Load-time recovery is a narrow maintainable fix, but this PR is not mergeable until backup recovery is protected by the same pre-read validation and the fs-safe tests exercise the actual pinned temp path.
