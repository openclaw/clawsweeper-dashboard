## Summary

Keep open: this is a narrow replacement for the earlier dirty branch and targets the right linked outbound bug, but the new predicate treats any `actions.handleAction` plugin as send-capable and can bypass the private source-reply sink for plugins that do not actually support `send`. The supplied screenshots show local/mock test output rather than real external-channel delivery proof, so the PR is not merge-ready.

## What This Changes

The PR changes `src/infra/outbound/internal-source-reply.ts` so `message_tool_only` targetless source replies can bypass the internal sink when the current loaded channel plugin has `actions.handleAction` or `outbound`.

## Best Possible Solution

Use one shared send-capability predicate for this sink escape, preserving the internal-source fallback for non-send action plugins, then add focused regression coverage and real external-channel proof.

## Reproduction Assessment

Yes at source level. The linked issue reproduces from current main because `shouldUseInternalSourceReplySink` runs before plugin dispatch, and the PR-introduced regression is source-reproducible with a configured action plugin whose `supportsAction` declines `send`.

## Solution Assessment

No. The PR is in the right area, but the best fix should reuse the actual send-support contract instead of treating any `handleAction` or `outbound` object as sufficient.
