## Summary

Keep open: this is a fresh, detailed regression report and current main still has a source-level race where follow-up drain work can start before the prior reply operation has cleared the same session key.

## What This Changes

Fix alternating ReplyRunAlreadyActiveError failures for sequential gateway chat.send calls on the same session.

## Best Possible Solution

Repair the reply-run/follow-up lifecycle so queued same-session chat turns drain only after the previous reply operation is idle, with regression coverage for sequential chat.send traffic on one session.

## Reproduction Assessment

Yes, at source level: current main schedules follow-up draining before replyOperation.complete clears activeRunsByKey, and the follow-up runner immediately calls createReplyOperation for the same key. I did not run the live gateway/provider loop in this read-only pass.

## Solution Assessment

No implemented fix is present yet. The narrow maintainable fix is to order or guard follow-up draining around reply operation cleanup, not add a force-clear workaround or config knob.

## Repair Work Prompt

Fix openclaw/openclaw#77485: sequential gateway chat.send calls on the same session can alternate into ReplyRunAlreadyActiveError because follow-up draining is scheduled before the previous reply operation clears its active session key. Keep the fix inside the reply-run/follow-up lifecycle; do not add a user config knob, do not use forceClearReplyRunBySessionId as a normal-path workaround, and do not change plugin or provider policy. Likely repair options are to delay/schedule follow-up drain after replyOperation.complete, have the drain wait for replyRunRegistry.waitForIdle before beginning a queued same-session operation, or otherwise make the lifecycle ordering explicit. Add regression coverage for a queued same-session follow-up/chat.send path that proves the second turn reaches the model path instead of returning the canned 'Previous run is still shutting down' reply. Add a CHANGELOG.md fix entry because this is user-facing.
