## Summary

Keep open: current main and the latest shipped tag still derive QQ Bot media storage from the process home directory, while OpenClaw’s path contract says `OPENCLAW_HOME` should replace that home for internal paths. The linked fixing PR is still open, so this issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue requests that QQ Bot media paths honor `OPENCLAW_HOME` instead of always resolving under `~/.openclaw/media/qqbot/`.

## Best Possible Solution

Align the QQ Bot media/data path helper with OpenClaw’s `OPENCLAW_HOME`-aware state path semantics, keep the local media containment guard, and close this issue only after a merged fix.

## Reproduction Assessment

Yes, from source: with different `HOME` and `OPENCLAW_HOME`, QQ Bot validates local media against the HOME-derived `.openclaw/media` tree while core path helpers and docs make `OPENCLAW_HOME` the effective internal home. I did not run a live QQ Bot send in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the QQ Bot-owned path helper to reuse the existing OpenClaw effective-home contract and add custom-`OPENCLAW_HOME` media regression coverage, not to add a new config option or workaround.
