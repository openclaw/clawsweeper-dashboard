## Summary

Keep open: current main and the latest release still derive QQ Bot media roots from the OS home while OpenClaw’s documented path contract says `OPENCLAW_HOME` replaces that home for internal paths. The only linked fix PR was closed unmerged, so no canonical implementation is currently available.

## What This Changes

The issue asks for QQ Bot media paths and local media validation to honor `OPENCLAW_HOME` instead of resolving under the HOME-derived `~/.openclaw/media/qqbot`.

## Best Possible Solution

Update the QQ Bot media root and local-file allowlist to follow the existing `OPENCLAW_HOME` path contract, keep persisted QQ Bot data compatibility separate, and add focused regression coverage for custom-home media sends.

## Reproduction Assessment

Yes, at source level: with `HOME` different from `OPENCLAW_HOME`, QQ Bot validates local media under the HOME-derived `.openclaw/media` root while core/docs define `OPENCLAW_HOME` as the internal path base. I did not run a live QQ Bot send in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing OpenClaw home/state path semantics at the QQ Bot media boundary, with tests for custom `OPENCLAW_HOME` and containment behavior.
