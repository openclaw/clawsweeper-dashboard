## Summary

Keep open. Current main still prints the WhatsApp login QR through direct console/stdout output and `loginWeb` does not pass the active runtime into QR rendering, so the active CLI runtime can still miss the usable terminal QR described here.

## What This Changes

Ensure `openclaw channels login --channel whatsapp` reliably exposes the WhatsApp Linked Devices QR through the active CLI runtime.

## Best Possible Solution

Route WhatsApp terminal QR text and ASCII output through the active runtime, pass that runtime through the login socket path including reconnect replacement, and keep any Baileys patch refresh separate unless it is independently required.

## Reproduction Assessment

Yes. A focused unit test can mock a Baileys `connection.update` QR event during `loginWeb` or `createWaSocket(true, ...)` and show `RuntimeEnv.writeStdout` is not used while direct console/stdout output is used.

## Solution Assessment

Yes for a narrow source fix. The maintainable path is to pass `RuntimeEnv` into WhatsApp socket QR rendering and add focused CLI/channel-auth regression coverage, without bundling dependency patch changes into the same repair.

## Repair Work Prompt

Fix openclaw/openclaw#76213. On current main, WhatsApp CLI login can reach `Waiting for WhatsApp connection...` while QR rendering is written directly through `console.log`/`process.stdout.write` instead of the active `RuntimeEnv`. Route the WhatsApp terminal QR heading and ASCII QR through `RuntimeEnv.writeStdout` when available, with existing logging/stdout fallback behavior preserved where appropriate. Pass the active runtime from `loginWeb()` into `createWaSocket()` and through the 515 restart/replacement path if QR output can recur there. Add focused regression tests that mock a Baileys QR `connection.update` event and assert runtime-routed terminal output. Also reduce `channels login/logout` startup only as far as needed for explicit WhatsApp login/logout without breaking inferred/external channel auth; keep the seam generic where possible and respect the extension boundary. Do not refresh or edit the Baileys patch, `pnpm-lock.yaml`, or dependency versions in this fix; PR #74349 tracks the separate dependency-patch direction. Add a user-facing CHANGELOG.md fix entry if code behavior changes.
