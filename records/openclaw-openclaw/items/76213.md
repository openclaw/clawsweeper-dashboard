## Summary

Keep open: current main and the latest release tag still route WhatsApp terminal QR output through direct console/stdout printing, and #76556 is an open closing PR that should be reviewed or landed before this issue is closed.

## What This Changes

Fix WhatsApp channel login so `openclaw channels login --channel whatsapp` reliably exposes a scan-ready terminal QR through the active CLI runtime.

## Best Possible Solution

Keep the fix in the WhatsApp/channel-auth boundary: route terminal QR rendering through the injected runtime, add regression coverage, and review #76556 or an equivalent narrow patch before closing this issue.

## Reproduction Assessment

Yes, by source inspection: current `loginWeb` still opts into socket-owned QR printing, and the socket prints QR output through `console.log` and `process.stdout.write` outside the injected runtime. This was not live-reproduced with a WhatsApp pairing session.

## Solution Assessment

Yes, the narrow maintainable direction is to disable socket-owned terminal QR printing for explicit login and render QR output from an `onQr` callback through the active runtime, with focused tests. #76556 is the active implementation candidate, but it should be reviewed before this issue closes.
