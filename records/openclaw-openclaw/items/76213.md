## Summary

Keep open: current main still routes explicit WhatsApp CLI login QR output through socket-owned console/stdout writes instead of the injected runtime, while docs and CLI help expect a visible QR from `openclaw channels login`. The related fix PR was closed unmerged, and the latest stable source still has the same path despite one commenter reporting a beta resolved their local case.

## What This Changes

The issue asks for WhatsApp channel login to expose the Linked Devices terminal QR through the active CLI runtime during re-authentication.

## Best Possible Solution

Land a narrow WhatsApp plugin fix that renders explicit CLI login QR output through the injected runtime, including restart sockets, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source-level: `loginWeb` still opts into socket-owned QR printing, and the QR handler still writes through `console.log` and `process.stdout.write` instead of the active runtime. I did not run a live WhatsApp pairing session.

## Solution Assessment

Yes, the best fix is a narrow owner-module change: disable socket-owned terminal QR printing for explicit CLI login and render the QR from an `onQr` callback through `RuntimeEnv`, including the 515 restart path. Broader channel startup policy changes should be included only if the replacement implementation proves they are required for this failure.
