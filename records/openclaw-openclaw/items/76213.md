## Summary

Keep open. Current main still matches the reported failure mode: WhatsApp QR output is written through `console.log`/`process.stdout.write`, `loginWeb` does not pass the active runtime into QR rendering, and `channels login` still uses the broad configured-channel plugin startup policy despite the helpful beta-resolution comment.

## What This Changes

Request to make WhatsApp channel login reliably expose a usable terminal QR through the active CLI runtime during re-authentication.

## Best Possible Solution

Thread `RuntimeEnv` into WhatsApp QR rendering, write the QR heading and ASCII QR through `runtime.writeStdout`, and narrow explicit channel-auth startup without bundling an unrelated Baileys patch refresh.

## Reproduction Assessment

Yes. A high-confidence source-level unit repro can invoke `loginWeb` with a runtime mock, emit a Baileys `connection.update` QR event from `createWaSocket(true, ...)`, and observe QR output going to `console`/`process.stdout` instead of `runtime.writeStdout`.

## Solution Assessment

Yes. Passing the active `RuntimeEnv` into `createWaSocket`/`printTerminalQr` and adding exact startup-policy regression coverage is the narrowest maintainable fix; dependency patch updates should stay separate unless independently required.
