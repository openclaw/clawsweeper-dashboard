## Summary

Keep open. Current main still has the WhatsApp CLI login QR path bypassing the injected runtime writer, and #76556 is an open closing PR for this same bug, so this issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

Ensure WhatsApp channel login reliably exposes a terminal QR through the active CLI runtime during re-authentication.

## Best Possible Solution

Review and land or replace #76556's narrow RuntimeEnv-routed QR change with regression coverage for injected-runtime output.

## Reproduction Assessment

Yes. Source inspection shows `loginWeb()` calls `createWaSocket(true, ...)`; when Baileys emits a QR, the heading and ASCII art bypass the injected runtime through `console.log` and `process.stdout.write`.

## Solution Assessment

Yes. The narrow maintainable fix is to stop the CLI login path from using the built-in printQr path and route the heading plus rendered QR through the injected runtime, as the linked PR proposes.
