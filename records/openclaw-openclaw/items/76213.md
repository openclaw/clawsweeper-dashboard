## Summary

Keep open. Current main still bypasses the active runtime writer for WhatsApp terminal QR output and has no narrow/minimal explicit WhatsApp login path, so the reported v2026.4.29 behavior is not implemented or superseded.

## What This Changes

Ensure WhatsApp `channels login` reliably displays a scannable terminal QR through the active CLI runtime during re-authentication.

## Best Possible Solution

Implement the WhatsApp-owned login fix by routing QR heading and ASCII QR output through `RuntimeEnv.writeStdout`, adding a narrow explicit bundled WhatsApp login path/minimal startup policy, and covering both with focused regression tests.

## Reproduction Assessment

Yes. A focused unit reproduction can mock a Baileys `connection.update` QR event during `loginWeb` and show that `RuntimeEnv.writeStdout` is not called while direct stdout/console output is used; the current source also matches the reported release path.

## Solution Assessment

Yes for the non-dependency part of the requested approach: passing `RuntimeEnv` into the WhatsApp login socket and narrowing channel-auth startup is the smallest maintainable fix. The Baileys patch refresh should stay separate unless a focused failing case proves it is required here.
