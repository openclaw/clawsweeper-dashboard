## Summary

Keep open. Current main still prints the WhatsApp terminal QR through direct stdout/console paths and leaves `channels login` on broad configured-channel plugin startup, so the reported CLI login stall remains a valid narrow bug.

## What This Changes

The issue asks to make WhatsApp channel login reliably expose the Linked Devices QR through the active CLI runtime and avoid unnecessary startup work during channel auth.

## Best Possible Solution

Implement the WhatsApp-owned login fix by routing QR heading and ASCII QR output through `RuntimeEnv.writeStdout`, adding a narrow explicit bundled WhatsApp login/minimal startup path, and covering both with focused regression tests.

## Reproduction Assessment

Yes. A focused unit reproduction can mock a Baileys `connection.update` QR event during `loginWeb` and assert that the active runtime writer is not called while direct stdout/console output is used.

## Solution Assessment

Yes for the non-dependency part. Passing the active runtime into WhatsApp socket QR rendering and narrowing explicit channel auth startup is the smallest maintainable fix; dependency patch refresh should be separate unless proven necessary.
