## Summary

Keep open: current main still routes the WhatsApp login QR through socket-owned console/stdout output, and #76556 is an open unmerged closing PR for the same runtime-routing fix.

## What This Changes

The issue asks for WhatsApp channel login to reliably expose the terminal QR through the active CLI runtime during re-authentication.

## Best Possible Solution

Review and land #76556 or an equivalent narrow WhatsApp/channel-auth patch that routes terminal QR rendering through the injected runtime and adds regression coverage.

## Reproduction Assessment

Yes, by source inspection: `openclaw channels login --channel whatsapp` reaches `loginWeb`, which passes `printQr: true`, and QR output then uses `console.log`/`process.stdout.write` outside `RuntimeEnv`. I did not run a live WhatsApp pairing session.

## Solution Assessment

Yes, the best direction is a narrow WhatsApp-owned fix: disable socket-owned QR printing for explicit login, render the QR from an `onQr` callback through the active runtime, and keep focused regression tests. The issue should remain open until #76556 or equivalent code lands.
