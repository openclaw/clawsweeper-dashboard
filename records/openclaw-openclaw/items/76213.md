## Summary

Keep open: current `main` still has the reported runtime-bypass path, and the related fix PR was closed unmerged rather than shipped. The issue is narrow and actionable enough for a replacement repair PR.

## What This Changes

The issue requests that WhatsApp CLI login expose the terminal QR through the active runtime instead of stalling behind direct stdout/plugin startup behavior.

## Best Possible Solution

Land a narrow WhatsApp login fix that renders terminal QR output through the injected runtime, carries the same `onQr` callback into the 515 restart path, and adds focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection: `loginWeb` still enables socket-owned QR printing, and the socket still emits QR output through `console.log` and `process.stdout.write` outside the injected runtime. I did not run a live WhatsApp pairing session.

## Solution Assessment

Yes, the best narrow fix is to disable socket-owned terminal QR printing for explicit login and render QR output from a shared `onQr` callback through `RuntimeEnv`, including restart sockets. Broader channel startup-policy changes should stay out unless source review proves they are required for the same failure.
