## Summary

Keep open. Current main still matches the reported failure mode: WhatsApp QR rendering bypasses the active runtime writer, `loginWeb` does not pass the runtime into socket QR rendering, and `channels login` still uses the broad configured-channel plugin startup path despite a later beta-resolution comment.

## What This Changes

The issue asks for WhatsApp CLI login to reliably expose the Linked Devices QR through the invoking terminal/runtime during re-authentication.

## Best Possible Solution

Route QR heading and ASCII output through `RuntimeEnv.writeStdout`, pass the runtime into WhatsApp socket creation for login, and add focused regression tests while keeping Baileys patch refresh separate unless proven necessary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence unit reproduction: emit a Baileys `connection.update` QR event during `loginWeb`/`createWaSocket(true, ...)` and assert the active runtime writer is bypassed while direct stdout/console output is used.

## Solution Assessment

Yes for the source-level fix. Passing `RuntimeEnv` into socket QR rendering and narrowing explicit channel-auth startup is the smallest maintainable repair; dependency patch changes should be split unless independently required.
