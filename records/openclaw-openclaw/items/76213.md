## Summary

Keep open: current `main` and the latest stable source still route explicit WhatsApp login QR output through socket-owned `console`/`process.stdout` paths, and the directly related fix PR #76556 was closed unmerged.

## What This Changes

Ensure WhatsApp CLI login and re-authentication expose the scanable terminal QR through the active runtime instead of bypassing it with direct console/stdout output.

## Best Possible Solution

Route explicit WhatsApp login QR heading and ASCII art through the injected runtime on both the initial and 515 restart sockets, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection: `loginWeb` still enables socket-owned QR printing, and the QR handler still writes through `console.log`/`process.stdout` outside the active runtime. I did not run a live WhatsApp pairing session.

## Solution Assessment

Yes, the narrow maintainable fix is to disable socket-owned terminal QR printing for explicit login, provide a shared runtime-routed `onQr` callback, and pass that callback into the 515 restart path. Broader channel startup-policy changes should be included only if source review proves they are required for this same failure.

## Repair Work Prompt

Fix the WhatsApp CLI login QR runtime-bypass bug tracked by https://github.com/openclaw/openclaw/issues/76213, using https://github.com/openclaw/openclaw/pull/76556 as source context but creating a fresh narrow implementation on current main. In `extensions/whatsapp/src/login.ts`, route the QR heading and `renderQrTerminal(..., { small: true })` output through the injected runtime, preferring `writeStdout` when present and falling back to `runtime.log`; call `createWaSocket(false, ...)` for explicit login and pass the same `onQr` callback into `waitForWhatsAppLoginResult` so the 515 restart socket preserves QR output. Keep `createWaSocket` generic unless a small helper is needed; do not change Baileys dependency patches, package resolution, or broad channel startup policy unless source review proves it is required for this same failure. Add focused regression coverage for initial login QR output and restart QR output through the runtime, and add a user-facing `CHANGELOG.md` fix entry. Preserve existing WhatsApp login/logout behavior outside this QR routing fix.
