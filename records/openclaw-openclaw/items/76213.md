## Summary

Keep open. Current main still routes WhatsApp CLI login QR rendering through direct console/stdout output instead of the active RuntimeEnv writer, and the latest local release tag contains the same path despite a useful beta-success comment.

## What This Changes

The issue asks for WhatsApp CLI login to reliably expose the Linked Devices QR through the active terminal runtime and avoid unnecessary broad startup work during explicit channel auth.

## Best Possible Solution

Route the QR heading and rendered QR through `RuntimeEnv.writeStdout` with a conservative fallback, pass the runtime into WhatsApp socket creation, and add focused regression coverage for CLI login output and channel-auth startup policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: emit a Baileys `connection.update` event with `qr` during `loginWeb()` and assert the active runtime writer is bypassed while direct console/stdout output is used.

## Solution Assessment

Yes. The narrow maintainable fix is to wire `RuntimeEnv` into the existing WhatsApp socket QR path and tighten the explicit channel-auth startup policy; Baileys patch refreshes should stay separate unless independently required.

## Repair Work Prompt

Fix the WhatsApp CLI login QR runtime-output bug tracked at https://github.com/openclaw/openclaw/issues/76213. On current main, `openclaw channels login --channel whatsapp` reaches `extensions/whatsapp/src/channel.ts` auth login, then `loginWeb()`, then `createWaSocket(true, ...)`; QR output is emitted via `console.log` and `process.stdout.write` instead of the active `RuntimeEnv.writeStdout`. Update the WhatsApp session/login plumbing so the QR heading and rendered ASCII QR go through the active runtime writer when present, with a conservative fallback for callers that do not provide a runtime. Also tighten explicit `channels login`/`channels logout` startup policy only as needed to avoid broad channel/plugin startup work while preserving inferred-channel behavior when no channel is provided. Add regression tests for runtime-routed terminal QR output and CLI channel-auth startup policy. Add a user-facing `CHANGELOG.md` fix entry. Do not refresh Baileys patches or change WhatsApp Web version handling unless a targeted test proves it is required for this bug.
