## Summary

Keep open: current main still has the source-level QR routing bug, and the linked fixing PR #76556 is open and unmerged. The issue should stay open until that PR or an equivalent fix lands.

## What This Changes

WhatsApp channel login should consistently print a scanable terminal QR through the active CLI runtime during re-authentication.

## Best Possible Solution

Review and land #76556 or an equivalent WhatsApp fix that routes the QR heading and ASCII art through the injected runtime and adds regression coverage.

## Reproduction Assessment

Yes, source-level: current main calls `createWaSocket(true, ...)` from `loginWeb`, and the QR handler writes with `console.log` plus `process.stdout.write` instead of the injected `RuntimeEnv`. I did not run a live WhatsApp pairing in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make the explicit WhatsApp login path handle QR events through the active runtime while preserving the existing socket callback seam. #76556 appears to target that path, so the issue should remain open pending PR review and merge.
