## Summary

Keep this open because current main still routes the WhatsApp login QR through direct console/stdout output, and an open linked PR (#76556) uses closing syntax for this issue but has not merged yet.

## What This Changes

The issue asks for WhatsApp channel login to expose the Linked Devices QR through the active CLI runtime instead of appearing stuck after entering the connection flow.

## Best Possible Solution

Land #76556 or an equivalent WhatsApp-owned fix that renders the QR through the injected runtime, with regression coverage for terminal QR output and any accepted channel-auth startup policy cleanup.

## Reproduction Assessment

Yes, from source inspection: the documented `channels login --channel whatsapp` path reaches `loginWeb`, which asks `session.ts` to print QR output directly to console/stdout instead of the active runtime. I did not perform live WhatsApp pairing, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes, routing the QR heading and ASCII art through the injected runtime is the narrowest maintainable fix for the reported terminal visibility problem. If maintainers want the startup-policy bullets too, they should be folded into the linked WhatsApp/channel-auth fix before merge.
