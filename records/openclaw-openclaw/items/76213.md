## Summary

Keep open. Current main still routes the WhatsApp terminal QR around the injected CLI runtime, and the issue is paired with open closing PR #76556, so it should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue requests reliable WhatsApp channel login QR output through the active CLI runtime during re-authentication.

## Best Possible Solution

Land or replace the narrow runtime-routed WhatsApp QR fix, then let the merged PR close this issue once current main no longer bypasses the active CLI runtime.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: emit a Baileys `connection.update` QR event during `loginWeb()` and the QR heading/art go through `console.log` and `process.stdout.write` while the injected runtime receives only the waiting message.

## Solution Assessment

Yes for the proposed direction. The narrow maintainable fix is to route the terminal QR heading and ASCII art through the injected runtime, and #76556 is already the open implementation candidate for that path.
