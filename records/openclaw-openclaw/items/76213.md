## Summary

Keep open: current `main` and the latest release still route explicit WhatsApp CLI login QR rendering through socket-owned console/stdout output instead of the injected runtime, while the directly related fix PR was closed unmerged.

## What This Changes

The issue requests that WhatsApp channel login reliably display the Linked Devices QR through the active CLI runtime.

## Best Possible Solution

Land a narrow WhatsApp login fix that renders terminal QR output through the injected runtime, preserves QR callbacks across the 515 restart socket, adds focused regression coverage, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection: `loginWeb` still enables socket-owned QR printing, and the socket path still writes the QR heading/art through `console.log` and `process.stdout` outside the injected runtime. I did not run a live WhatsApp pairing session.

## Solution Assessment

Yes, the narrow maintainable fix is to disable socket-owned terminal QR printing for explicit CLI login and render the QR from a shared `onQr` callback through `RuntimeEnv`, including restart sockets. Broader channel startup-policy changes should stay separate unless source review proves they are required for the same failure.

Fix the WhatsApp CLI login QR runtime-routing bug tracked by https://github.com/openclaw/openclaw/issues/76213. In `extensions/whatsapp/src/login.ts`, stop explicit `loginWeb` from relying on `createWaSocket(true, ...)`; instead pass `printQr: false` plus an `onQr` callback that writes the Linked Devices heading and rendered ASCII QR through the active `RuntimeEnv` output path when available, with a safe fallback.
