## Summary

Keep open. Current main still renders the WhatsApp terminal QR through direct console/stdout output, `loginWeb` does not pass the active `RuntimeEnv` into QR rendering, and `channels login` still follows the generic configured-channel plugin startup path, so the reported CLI login stall remains actionable.

## What This Changes

The issue asks for WhatsApp CLI login to reliably expose the Linked Devices QR through the active CLI runtime and avoid unnecessary broad startup work for explicit bundled WhatsApp auth.

## Best Possible Solution

Route the CLI QR heading and ASCII QR through `RuntimeEnv.writeStdout`, pass the runtime into WhatsApp socket creation for login, add a minimal explicit WhatsApp login/logout startup path, and pin the behavior with focused regression tests.

## Reproduction Assessment

Yes. A focused unit reproduction can mock a Baileys `connection.update` QR event during `loginWeb` or `createWaSocket(true, ...)` and show that `RuntimeEnv.writeStdout` is not called while direct console/stdout output is used.

## Solution Assessment

Yes for the non-dependency fix. Passing `RuntimeEnv` into WhatsApp socket QR rendering and narrowing channel-auth startup is the smallest maintainable source change; a Baileys patch refresh should be separate unless proven necessary here.
