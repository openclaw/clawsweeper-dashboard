## Summary

Keep open: current main still routes explicit WhatsApp CLI login through socket-owned QR printing, and the only PR that directly proposed routing the terminal QR through the injected runtime was closed unmerged. The report is recent, narrow, source-reproducible, and suitable for a replacement repair PR.

## What This Changes

The issue requests reliable terminal QR rendering for `openclaw channels login --channel whatsapp` through the active CLI runtime during WhatsApp re-authentication.

## Best Possible Solution

Land a narrow WhatsApp login fix that renders terminal QR heading and ASCII art through the injected runtime for initial and restart sockets, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection: `loginWeb` still requests socket-owned QR printing, and the QR handler still writes through `console.log` and `process.stdout.write` instead of the injected runtime. I did not perform a live WhatsApp scan.

## Solution Assessment

Yes, the narrowest maintainable fix is to keep WhatsApp-owned login behavior in the WhatsApp plugin, disable socket-owned terminal QR output for explicit CLI login, and route a shared `onQr` renderer through `RuntimeEnv` including restart replacement sockets.

## Repair Work Prompt

Fix the WhatsApp CLI login QR runtime-bypass bug tracked by https://github.com/openclaw/openclaw/issues/76213. Keep the change scoped to the WhatsApp plugin and only touch CLI channel startup policy if source review proves it is required for the same failure. Expected behavior: `openclaw channels login --channel whatsapp` should render the QR heading and ASCII QR through the injected `RuntimeEnv` used by the active CLI runtime, not through direct `console.log`/`process.stdout.write`; the same QR callback must be preserved if the login flow replaces the socket after a WhatsApp 515 restart. Likely approach: in `extensions/whatsapp/src/login.ts`, call `createWaSocket(false, ...)` with an `onQr` callback that uses existing terminal QR rendering helpers and `runtime.writeStdout`/runtime logging consistently; thread that callback through `waitForWhatsAppLoginResult` restart socket creation in `extensions/whatsapp/src/connection-controller.ts` if needed. Add regression coverage proving the heading and ASCII QR are routed through the runtime and that the restart path carries the callback. Add a single-line user-facing `CHANGELOG.md` fix entry. Do not broaden channel/plugin startup behavior, alter Baileys dependency patches, or change unrelated QR login web/control surfaces unless necessary for this issue.
