## Summary

Keep open. Current main still sends the WhatsApp terminal QR heading/body through `console.log` and direct `process.stdout.write` instead of the active runtime writer, and `loginWeb()` still does not pass the runtime into socket QR rendering; the beta-resolution comment is useful signal, but tag/source inspection does not show this source-level path fixed on main.

## What This Changes

Fix WhatsApp channel login so the invoking CLI runtime reliably exposes a usable terminal QR for re-authentication.

## Best Possible Solution

Route WhatsApp QR heading/body output through the active `RuntimeEnv.writeStdout`, pass the runtime into socket QR rendering during login, and tighten channel login/logout startup policy with focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence unit reproduction: emit a Baileys `connection.update` QR event during `loginWeb()` and assert the provided runtime writer is bypassed while direct console/stdout output is used. I did not run a live WhatsApp pairing flow.

## Solution Assessment

Yes. Passing the runtime into the WhatsApp socket QR path and writing the QR heading/body through `RuntimeEnv.writeStdout` is the narrowest maintainable repair; dependency patch refreshes should stay separate unless independently required.
