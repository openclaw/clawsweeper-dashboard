## Summary

Keep open. Current main still routes the WhatsApp terminal QR through direct console/stdout output, does not pass the active RuntimeEnv into createWaSocket during loginWeb, and keeps channels login on the generic configured-channel plugin startup path; the beta success comment is useful signal but not enough to close against this main tree.

## What This Changes

Make WhatsApp CLI login reliably expose a scan-ready terminal QR during re-authentication.

## Best Possible Solution

Route the WhatsApp terminal QR heading and ASCII QR through RuntimeEnv.writeStdout, pass RuntimeEnv into socket creation for CLI login, and narrow explicit WhatsApp channel auth startup without changing unrelated channel behavior.

Yes. A focused unit reproduction can mock a Baileys connection.update QR event during loginWeb/createWaSocket and assert the active RuntimeEnv.writeStdout is not called while console/process.stdout is used.

## Solution Assessment

Yes for the source fix boundary. Passing RuntimeEnv into WhatsApp QR rendering and reducing explicit channel-auth startup work is the narrow maintainable fix; dependency patch refreshes should stay separate unless proven necessary.
