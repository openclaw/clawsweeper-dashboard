## Summary

Keep open: current main still has the WhatsApp login QR runtime-bypass path, and the directly related fix PR #76556 is closed unmerged. The comment that 2026.5.2-beta.1 helped is useful context, but current source and the latest stable tag still show the reported path.

## What This Changes

The issue asks for WhatsApp `channels login` re-authentication to expose a scannable terminal QR through the active CLI runtime instead of stalling after the waiting message.

## Best Possible Solution

Land a narrow WhatsApp login fix that renders the QR heading and ASCII art through the injected runtime, carries the same QR handler into the 515 restart socket, and adds focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection: current main still calls `createWaSocket(true, ...)` from `loginWeb`, and the QR printer still writes through `console.log`/`process.stdout` outside `RuntimeEnv`. I did not run a live WhatsApp pairing session.

## Solution Assessment

Yes, the best narrow repair is to disable socket-owned QR printing for explicit login and use a shared runtime-routed `onQr` callback, including restart sockets. Broader channel startup changes should stay out unless the patch proves they are required for the same failure.
