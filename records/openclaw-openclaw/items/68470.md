## Summary

Keep open. Current main still lacks the reported MiniMax `prompt_cache_hit_tokens` normalization, while timeout compaction still derives its high-context decision from additive prompt-token math; the linked closing PR remains open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

MiniMax cache-hit prompt usage should be de-duplicated so `/status`, session context accounting, and timeout compaction use actual prompt-side context pressure instead of inflated cached-token totals.

## Best Possible Solution

Keep this issue open until #68750 or an equivalent narrow fix lands. The desired end state is a confirmed MiniMax cache-hit payload normalized at `src/agents/usage.ts`, focused regression coverage in the usage tests, and no changes to compaction thresholds or memory-flush policy unless the normalization evidence proves insufficient.

## Reproduction Assessment

Yes for the source-level accounting path: represent the reported MiniMax cache-hit shape as prompt tokens plus cache-read breakdown, then pass it through `normalizeUsage` and `derivePromptTokens` to verify whether cached prompt tokens are de-duplicated before timeout-compaction ratio calculation. A live MiniMax run would still be useful to confirm the exact upstream field name before merging a provider-specific alias.

## Solution Assessment

Yes, the narrowest maintainable direction is to fix the shared usage-normalization boundary and add a regression test for the confirmed MiniMax payload shape. Changing compaction thresholds, disabling memory flush, or special-casing timeout recovery would treat the symptom rather than the accounting contract.
