## Summary

Keep open: current main still lacks the reported MiniMax cache-hit alias, timeout compaction still depends on derived prompt-token usage, and two open PRs already reference this issue with closing syntax, so it should not be closed before a canonical fix lands.

## What This Changes

The issue reports MiniMax cache-hit usage being counted into prompt pressure in a way that can trigger premature timeout compaction and memory flushes.

## Best Possible Solution

Keep this issue open until one canonical PR normalizes the MiniMax cache-hit payload at all relevant usage ingestion points, adds focused regression coverage, and includes redacted real MiniMax/OpenClaw proof.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection does show the relevant accounting path: preserved prompt-side `input` plus `cacheRead` is additive in `derivePromptTokens`, and timeout compaction uses that derived value.

## Solution Assessment

Yes, the narrow maintainable direction is usage normalization and ingestion-path coverage, not threshold changes or disabling compaction. The issue should be resolved by reviewing or replacing the linked implementation PRs, then closing after merge.
