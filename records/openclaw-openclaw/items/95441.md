## What This Changes

The issue reports GitHub Copilot GPT-5.5 channel/direct sessions replaying persisted Responses reasoning signatures and encrypted content, causing later turns to fail before a deliverable assistant reply.

## Best Possible Solution

Add a provider-owned Copilot GPT/Responses replay policy that prevents stale encrypted reasoning/signature replay in channel/direct histories, plus focused regression coverage and a repair path for already polluted session transcripts if runtime still reads those entries.

## Reproduction Assessment

Yes, source-level: current main still leaves Copilot GPT replay history unchanged and both shared and embedded Responses converters can replay `thinkingSignature` JSON. I did not establish a live provider repro because the failing setup needs private Copilot credentials and session history.

## Solution Assessment

Unclear until the provider owner chooses the replay contract. The narrowest maintainable direction appears to be a provider-owned Copilot GPT replay policy with regression tests and polluted-session repair coverage, rather than relying only on the existing encrypted-content retry.
