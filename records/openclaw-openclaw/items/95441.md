## Summary

Keep open: current main still has an intentional path that stores and can replay OpenAI Responses encrypted reasoning for same-context github-copilot GPT sessions, while the Copilot plugin only drops overlong/invalid reasoning IDs and only strips thinking for Copilot Claude models. The issue is newer than the shipped prior fixes and no open or merged PR now owns this exact remaining Copilot/channel-direct gap.

## What This Changes

The issue reports that github-copilot/gpt-5.5 channel/direct sessions can still persist and replay encrypted reasoning/signature fields after earlier replay fixes, causing later turns to fail before an assistant reply.

## Best Possible Solution

Keep this open for a provider-owner fix that proves the failing Copilot channel/direct replay path live, then either strips or provenance/expiry-gates stale Copilot encrypted reasoning before replay and repairs existing polluted transcripts without changing unrelated Responses providers.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main persists OpenAI Responses encrypted reasoning into `thinkingSignature`, preserves it through transcript redaction, and can replay it for same-context Copilot GPT responses. I did not run the private live Copilot/channel setup, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear: the current preservation/provenance/retry approach is a plausible general Responses strategy, but this report shows Copilot may need a provider-specific expiry/repair rule. The best fix should be chosen by the provider owners with live Copilot continuation proof and regression coverage.
