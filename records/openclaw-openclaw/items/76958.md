## Summary

Keep open for normal PR review/merge: current main has a source-inspectable Discord SecretRef failure path, and this PR uses the documented runtime snapshot helper at the credential-read boundary with focused regression coverage. I found no blocking correctness or security findings.

## What This Changes

The PR makes Discord account resolution select the active runtime config snapshot before token reads, passes that resolved config into provider startup, and adds Discord tests plus a changelog entry for SecretRef-backed named accounts.

## Best Possible Solution

Land this PR or an equivalent narrow Discord-plugin fix after normal CI, keeping token normalization strict and resolving credentials only through the existing active runtime snapshot contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, calling Discord account resolution with a source config containing an env SecretRef reaches strict token normalization before selecting the active runtime snapshot, which throws the unresolved SecretRef error reported in #76889.

## Solution Assessment

Yes. Selecting the applicable runtime config before Discord credential reads is the documented SDK pattern and is narrower than adding ad hoc process.env fallback or weakening token validation.
