## Summary

Keep open: current main still lacks a dedicated zero-output idle-timeout cost breaker, and the open PR #76345 uses closing syntax for this issue, so the right cleanup path is to review and land or close that PR rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to cap or back off LLM idle-timeout retries so heartbeat/model calls cannot loop into paid provider cost spikes.

## Best Possible Solution

Land a run-loop-level breaker that caps consecutive zero-output idle-timeout attempts across retries, profile rotation, and model fallback, with focused regression tests and the existing changelog entry.

## Reproduction Assessment

Yes by source inspection, but not by a live run in this review: current main recreates the stream idle-timeout wrapper per attempt and only has broad 32-160 run-loop protection, not a 3-5 consecutive idle-timeout cost cap.

## Solution Assessment

Yes, the open PR's revised direction is the right boundary because the breaker state lives in the outer run loop where it survives attempts and failover; this issue should remain open for that PR to close on merge.
