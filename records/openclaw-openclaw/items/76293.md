## Summary

Current main already fixes the runaway class: automatic heartbeat wakes now pass through a shared cooldown/flood guard, and the embedded LLM runner has bounded idle-timeout retry behavior. The proof commit is on main after v2026.4.29 and is not contained in a release tag yet.

## What This Changes

The issue asks OpenClaw to cap/back off LLM idle-timeout retries so heartbeat activity cannot generate hundreds or thousands of paid provider calls.

## Best Possible Solution

Keep the current main cooldown and bounded retry guards, then ship them in the next release so affected users can upgrade past v2026.4.29-era behavior.

## Reproduction Assessment

Yes for the root-cause class: the regression test simulates repeated automatic heartbeat wakes within the configured interval and verifies current main only runs once. The exact Anthropic billing incidents are external, but they are not needed to verify the current code path is bounded.

## Solution Assessment

Yes. A shared heartbeat wake cooldown plus the existing bounded LLM idle retry is the narrow maintainable fix; provider-specific retry knobs alone would not stop wake-driven heartbeat self-feeding.
