## Summary

Current main bounds the reported failure mode: heartbeat wakes now go through cooldown/flood guards, LLM idle-timeout same-model retry is capped at one, and the embedded run loop has a defensive max-attempt cap. The proof commit is on current main, but this checkout does not show it in a shipped release tag yet.

## What This Changes

The issue requests bounded retry/backoff behavior so LLM idle timeouts during heartbeat runs cannot trigger runaway API calls and cost spikes.

## Best Possible Solution

Keep the current main safeguards and ship them forward: heartbeat wake cooldown/flood protection plus bounded LLM idle-timeout retry is the right product boundary for preventing this cost spike.

## Reproduction Assessment

Yes. The issue provides concrete production symptoms, and current regression tests model the relevant heartbeat re-fire loop by sending repeated event-driven wakes and asserting only one run within the cooldown window.

## Solution Assessment

Yes. The current fix addresses both plausible loop sources in the narrowest maintainable places: scheduler-level re-fire prevention for heartbeats and runner-level caps for idle-timeout retries.
