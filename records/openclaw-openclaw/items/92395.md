## Summary

Current main and the v2026.6.6 release no longer hold the embedded session write lock across provider streaming: the lock is released before the model stream and transcript writes reacquire it only around the write operation, with focused tests covering prompt-stream writes and later session-locked writes.

## What This Changes

The issue reports user follow-up Discord messages timing out behind a session JSONL write lock held across long agent turns, causing message loss without channel-visible errors.

## Best Possible Solution

Keep the v2026.6.6 lock-lifetime behavior and let the broader subagent delivery timeout cluster continue under https://github.com/openclaw/openclaw/issues/86538 and https://github.com/openclaw/openclaw/pull/86540.

## Reproduction Assessment

Yes for the historical bug path: the report includes concrete production log timing, and current source/tests show the lock-lifetime behavior that addresses it. I did not establish a failing current-main repro because current main already has the release/reacquire path.

## Solution Assessment

Yes: releasing the retained session file lock around provider streaming while keeping transcript writes behind withSessionWriteLock is the narrow maintainable fix for this report. Broader delivery fallback and diagnostics work should stay in the linked cluster instead of reopening this specific issue.
