## Label Justifications

## Summary

This remains a credible P1 message-loss report: current code reduces contention by releasing the session lock during model streaming, but reacquires it afterward and can retain it across other turn phases, so the broader long tool-chain failure is not proven fixed. The production trace is from a 2026.5.x-derived build, so a focused current-release reproduction is needed before choosing the safest lock-lifecycle repair.

## What This Changes

Prevent queued human follow-up messages from silently failing when a long agent turn causes session transcript write-lock acquisition to time out.

## Best Possible Solution

Add a focused regression that runs a queued follow-up against a deliberately long non-write phase, identify the exact retained-lock interval, then shorten or yield that interval through the existing fenced lock controller and guarantee a user-visible error if acquisition still times out.

## Reproduction Assessment

No, not yet at high confidence on current main: source inspection supports a plausible retained-lock contention path, but the concrete logs come from a 2026.5.x-derived deployment and current releases contain several intervening lock-lifecycle fixes.

## Solution Assessment

No, the proposed blanket per-append locking is not yet the best-proven repair; the safer direction is to preserve the existing transcript fence and narrow the retained-lock interval shown by a current-version regression.
