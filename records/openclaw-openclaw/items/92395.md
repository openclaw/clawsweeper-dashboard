## Label Justifications

- `P1`: The issue describes a current broken channel workflow where real follow-up user messages can time out and receive no reply during long agent turns.
- `impact:message-loss`: The reported failure mode is silently dropped or undelivered human channel responses after queued messages hit the session lock timeout.
- `impact:session-state`: The root surface is session transcript locking, queued turn state, and write/finalization ownership around a shared session JSONL file.

## Summary

Keep this open: current main has narrowed the initial session-lock lifetime around provider streaming, but source inspection and related live reports still show follow-up channel turns can hit the 60s session transcript lock timeout while another in-process turn owns write, hook, finalization, fallback, or compaction work.

## What This Changes

The issue reports that queued human follow-up messages can time out on the session JSONL write lock and disappear while a long orchestrator turn is still running.

## Best Possible Solution

Choose one canonical session-lock and queue recovery path that keeps transcript writes fenced, lets queued human messages wait or retry safely, and surfaces a channel-visible error when a lock timeout is unavoidable.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main still throws SessionWriteLockTimeoutError after 60s, and related live reports show gateway-owned locks timing out follow-up channel turns during long model, fallback, or subagent work; no live Discord reproduction was run in this read-only review.

## Solution Assessment

No patch is proposed; the requested short-lived write-lock direction is directionally right, but the best fix must account for owned transcript fences, compaction/event hooks, and queued message retry or error delivery instead of only shortening one acquisition scope.
