## Label Justifications

- `P0`: The report is about credential material crossing an isolation boundary and silently changing which QQBot identity a gateway uses.
- `impact:security`: The issue concerns a same-host gateway state isolation bypass involving reusable credential backups.
- `impact:auth-provider`: The affected data is QQBot `appId` and `clientSecret` material that changes channel authentication and runtime identity.

## Summary

Current main still stores QQBot credential backups under a home-global path and restores them into gateway config during startup, while the documented multi-gateway model expects per-instance state isolation. This issue should stay open because the linked fixing PR is still open and references this issue with closing syntax, so the right cleanup path is PR review rather than closing the report early.

## What This Changes

The issue reports that QQBot credential backups are stored under a home-global path, allowing one gateway instance to restore another gateway's QQBot app credentials despite separate gateway state directories.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/84314 and land a fix that scopes QQBot credential backups to the active gateway state/profile boundary with regression coverage for two same-home, different-state gateways.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: one gateway can save `credential-backup-default.json` under the shared home-global QQBot data directory, and a second gateway with the same account id and missing credentials can load it and write those credentials into its own config.

## Solution Assessment

Yes, the best direction is to fix the existing QQBot credential-backup storage and restore path so it follows the active gateway state/profile boundary; the linked PR is the current implementation candidate and should be reviewed rather than closing this issue first.
