## Label Justifications

- `P0`: The report describes credential material crossing a documented gateway isolation boundary and silently changing the QQBot identity used by another gateway.
- `impact:security`: The issue is about a same-host state isolation bypass involving reusable credential backups.
- `impact:auth-provider`: The affected data is QQBot `appId` and `clientSecret` material used to authenticate the channel runtime identity.

## Summary

Current main still has the QQBot credential-backup isolation problem, and the same contributor has an open linked fix PR with closing syntax. The issue should stay open until that PR or a replacement fix lands.

## What This Changes

The issue reports that QQBot credential backups are stored under a home-global path and can be restored into another same-home gateway with a different state directory.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84314, or a replacement, that routes QQBot credential backups through the canonical active state resolver with regression coverage for two same-home gateways using different state directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: one gateway saves `credential-backup-default.json` under the shared QQBot home-global data directory, and another same-home gateway with the same account id can load it and write those credentials into its own config.

## Solution Assessment

Yes for the requested direction. The maintainable fix is to scope QQBot credential backup save/load paths to the active gateway state/profile boundary; the open linked PR is the current implementation candidate and should be reviewed rather than closing this issue early.
