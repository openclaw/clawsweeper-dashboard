## Label Justifications

- `P0`: The report concerns credential material crossing a documented gateway isolation boundary and silently changing a gateway's QQBot identity.
- `impact:security`: The issue is about a same-host gateway state isolation bypass involving reusable credential backups.
- `impact:auth-provider`: The affected values are QQBot `appId` and `clientSecret` credentials that determine channel authentication and runtime identity.

## Summary

Current main still stores QQBot credential backups under a home-global path and can restore them into gateway config during startup, while the documented multi-gateway model expects per-instance config/state isolation. The issue should stay open because the linked fixing PR is open and uses closing syntax, so cleanup should happen through PR review rather than closing the report early.

## What This Changes

The issue reports that QQBot credential backups are scoped to `~/.openclaw/qqbot/data` instead of the active gateway state directory, allowing credential reuse across isolated gateway profiles with the same account id.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/84314 with a fix that scopes QQBot credential backups to the active gateway state/profile boundary and keeps regression coverage for same-home, different-state gateways.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: one gateway can save `credential-backup-default.json` under the shared home-global QQBot data directory, and a second gateway with the same account id and missing credentials can load it and write those credentials into its own config.

## Solution Assessment

Yes in direction. The maintainable fix is to keep the existing QQBot recovery behavior but scope backup storage and restore lookup to the active gateway state/profile boundary; the linked PR is the current implementation candidate and should be reviewed before this issue closes.
