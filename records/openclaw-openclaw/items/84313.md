## Label Justifications

- `P0`: The report describes credential material crossing a documented gateway isolation boundary and silently changing which QQBot identity a gateway uses.
- `impact:security`: The issue concerns a same-host gateway state isolation bypass involving reusable credential backups.
- `impact:auth-provider`: The affected data is QQBot appId/clientSecret material that controls channel authentication and runtime identity.

## Summary

Keep open: current main still stores QQBot credential backups under a home-global path and restores them into gateway config, while an open linked PR uses closing syntax for this issue and should be reviewed before the issue is closed.

## What This Changes

The issue reports that QQBot credential backup files are saved and restored from a home-global path, allowing one isolated gateway to import another gateway's appId/clientSecret.

## Best Possible Solution

Review and land the linked PR or a replacement that routes QQBot credential backups through the canonical state resolver and tests two same-home, different-state gateways.

## Reproduction Assessment

Yes. Source inspection shows one gateway can save credential-backup-default.json under the shared QQBot data directory, and a second gateway with the same account id and missing credentials can load it and write those credentials into its own config.

## Solution Assessment

Yes. The maintainable fix is to scope QQBot credential backup save/restore to the active gateway state boundary through the canonical state resolver; the open linked PR is the current implementation candidate and should be reviewed rather than closing this issue first.
