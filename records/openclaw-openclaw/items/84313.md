## Label Justifications

- `P0`: The report concerns reusable QQBot credentials crossing a documented gateway isolation boundary and silently changing a gateway's authenticated identity.
- `impact:security`: The issue is about a same-host state isolation bypass involving credential material.
- `impact:auth-provider`: The affected values are QQBot `appId` and `clientSecret`, which control channel authentication and runtime identity.

## Summary

Current main still stores QQBot credential backups under a home-global `~/.openclaw/qqbot/data` path and restores them into gateway config during startup, while the documented multi-gateway model expects per-instance state isolation. This issue should stay open because https://github.com/openclaw/openclaw/pull/84314 is still the open implementation candidate and references this issue with closing syntax.

## What This Changes

The issue reports that QQBot credential backups are stored outside the active gateway state directory and can be restored across same-home, different-state gateway instances.

## Best Possible Solution

Land or replace the linked PR with a fix that scopes QQBot credential backup save and restore to the active gateway state/profile boundary and includes regression coverage for two same-home gateways with different state directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: one gateway can save `credential-backup-default.json` under the shared home-global QQBot data directory, and a second same-home gateway with the same account id and missing credentials can load it and write those credentials into its own config.

## Solution Assessment

Yes. The best direction is to fix the existing QQBot credential-backup storage and restore path so it follows the active gateway state/profile boundary; the linked PR is the current implementation candidate and should be reviewed rather than closing this issue first.
