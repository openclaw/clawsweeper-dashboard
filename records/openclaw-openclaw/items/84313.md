## Label Justifications

- `P0`: The report describes reusable QQBot credential material crossing a documented isolation boundary and changing a gateway's runtime identity.
- `impact:security`: The issue concerns a same-host gateway state isolation bypass involving credential backups shared across gateway profiles.
- `impact:auth-provider`: The affected values are QQBot `appId` and `clientSecret`, which determine channel authentication and bot identity.

## Summary

Current main still stores and restores QQBot credential backups through a home-global QQBot data path, so the reported cross-gateway credential import remains valid; the issue should stay open because the linked fixing PR is open and uses closing syntax for this issue.

## What This Changes

The issue reports that QQBot credential backups are written under a shared home-global path and can be restored into another gateway profile with a different state directory.

## Best Possible Solution

Review and land a fix that scopes QQBot credential backups to the active gateway state/profile boundary and adds regression coverage for two same-home gateways with different state directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Gateway A can save `credential-backup-default.json` through the home-global QQBot data helper, and Gateway B with the same account id can restore it and write those credentials into its own config.

## Solution Assessment

Yes. The narrow maintainable solution is to move QQBot credential backup storage and restore through the active gateway state/profile boundary; the linked PR is the current implementation candidate and should be reviewed before this issue closes.
