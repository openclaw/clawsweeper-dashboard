## Label Justifications

- `P0`: The report describes reusable credential material crossing a documented gateway isolation boundary and changing the QQBot identity a gateway uses.
- `impact:security`: The issue concerns a same-host gateway isolation bypass involving sensitive QQBot credential backups.
- `impact:auth-provider`: The affected data is QQBot appId and clientSecret material that controls channel authentication and runtime identity.

## Summary

Current main still stores QQBot credential backups under a home-global path and can restore them into another gateway config with the same account id. This issue should stay open because the linked fixing PR is still open and uses closing syntax, so the right cleanup path is to review and land or replace that PR rather than close the report early.

## What This Changes

Fix QQBot credential backup storage and restore so credentials cannot cross documented multi-gateway state isolation boundaries.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/84314 and land a fix that scopes QQBot credential backups to the active resolved state/profile boundary with regression coverage for same-home, different-state gateways.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: one gateway can save a home-global QQBot credential backup, and a second gateway with the same account id and missing credentials can load it and write those credentials into its own config.

## Solution Assessment

Yes for the issue direction. The maintainable fix is to route QQBot credential backup save and restore through the active OpenClaw state/profile boundary; the linked PR is the current implementation candidate and needs review rather than early issue closure.
