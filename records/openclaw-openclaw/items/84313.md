## Label Justifications

- `P0`: The report concerns reusable credential material crossing a documented gateway isolation boundary and silently changing the QQBot identity used by another gateway.
- `impact:security`: The issue is a same-host gateway state isolation bypass involving persisted credential backups.
- `impact:auth-provider`: The affected data is QQBot appId/clientSecret material that controls channel authentication and runtime identity.

## Summary

Current main still stores QQBot credential backups under a home-global QQBot data path and restores them into gateway config during startup. This issue should stay open because https://github.com/openclaw/openclaw/pull/84314 is the open implementation candidate and references this issue with closing syntax.

## What This Changes

The issue reports that QQBot credential backups can be saved under a shared home-global path and restored into a different gateway despite separate OPENCLAW_STATE_DIR values.

## Best Possible Solution

Review and land a fix that scopes QQBot credential backups to the active gateway state/profile boundary, with regression coverage for two same-home gateways using different state directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: one gateway saves a QQBot credential backup under the OS-home QQBot data directory, and another same-home gateway with the same account id can load it and write those credentials into its own config.

## Solution Assessment

Yes. The maintainable direction is to fix the existing QQBot backup storage and restore path so it follows the active state/profile boundary; the linked PR is the current implementation candidate and should be reviewed before closing this issue.
