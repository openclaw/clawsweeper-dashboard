## Label Justifications

- `P0`: The issue reports reusable QQBot credentials crossing a documented gateway isolation boundary and silently changing runtime identity.
- `impact:security`: The affected path is a same-host gateway state isolation bypass involving secret credential backups.
- `impact:auth-provider`: The affected data is QQBot appId/clientSecret material that controls channel authentication and identity.

## Summary

Current main and the latest release still store QQBot credential backups in a home-global path and restore them into gateway config, so this security issue should stay open. The issue is already paired with an open closing PR, so the right path is to review and land or replace that PR rather than close this report early.

## What This Changes

The issue asks OpenClaw to scope QQBot credential backups to the active gateway state/profile boundary so isolated gateways cannot reuse another instance's QQBot appId/clientSecret.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/84314 with a fix that routes QQBot credential backups through the canonical active state resolver and covers two same-home, different-state gateways.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: one gateway saves a QQBot backup under the shared home-global data directory, and another gateway with the same account id and missing credentials can load it and write those credentials into its own config.

## Solution Assessment

Yes for the issue direction. The maintainable fix is to scope QQBot credential backup save/restore to the active gateway state/profile boundary; the linked PR should be reviewed or adjusted before this issue closes.
