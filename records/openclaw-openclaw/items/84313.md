## Label Justifications

- `P0`: The issue reports credential material crossing a documented gateway isolation boundary and silently changing the QQBot identity a gateway uses.
- `impact:security`: The report concerns a same-host gateway state isolation bypass involving reusable credential backups.
- `impact:auth-provider`: The affected data is QQBot appId/clientSecret material that controls channel authentication and runtime identity.

## Summary

Current main still stores QQBot credential backups under a home-global path and restores them into gateway config during startup, so the reported isolation bypass remains valid; this issue should stay open while the linked fixing PR is reviewed.

## What This Changes

The issue reports that QQBot credential backups can cross same-user gateway state boundaries because appId/clientSecret backups are saved and restored from a home-global QQBot path.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/84314 or land an equivalent fix that scopes QQBot credential backups to the active gateway state/profile boundary with regression coverage for two same-home, different-state gateways.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: one gateway saves credential-backup-default.json under the shared home-global QQBot data directory, and a second gateway with the same account id and missing credentials can load it and write those credentials into its own config.

## Solution Assessment

Yes. The maintainable fix direction is to route the QQBot credential backup save and restore path through the active gateway state/profile boundary; the linked PR is the current implementation candidate.
