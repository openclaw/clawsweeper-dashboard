## Summary

Keep open: current main and v2026.6.8 still block doctor repair modes up front in Nix mode, while auth paste/login paths can still reach guarded openclaw.json updates even though credential storage is separate. The remaining work is valid, but it needs maintainer policy confirmation because current docs and tests intentionally disable doctor repair in Nix mode.

## What This Changes

The issue asks OpenClaw to allow Nix-mode auth-profile writes and non-config doctor session-state repairs when the operation does not need to mutate openclaw.json.

## Best Possible Solution

Define a Nix-mode write policy that keeps openclaw.json immutable while allowing separate auth-store and session-store repairs, then align the doctor/auth command paths, docs, and regression tests to that boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows the doctor Nix guard runs before repair sequencing, and auth paste/login paths can still reach guarded updateConfig calls after separate auth-store writes; no live CLI run was attempted because this review is read-only.

## Solution Assessment

Unclear until maintainers confirm the policy carve-out. The maintainable direction is to guard actual openclaw.json writes while permitting separate auth/session state writes, but current docs and tests deliberately ban doctor repair in Nix mode.
