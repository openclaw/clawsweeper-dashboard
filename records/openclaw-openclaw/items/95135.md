- `P2`: This is a real Nix-mode workflow bug affecting auth and repair paths, but it is limited to Nix-managed installs rather than a general runtime outage.
- `impact:auth-provider`: The report covers model auth login and pasted credential flows that write auth profiles and may be blocked or error in Nix mode.
- `impact:session-state`: The report covers doctor repairs for stale persisted session routing/model state that should live outside openclaw.json.

## Summary

Keep open: current main and v2026.6.8 still fail Nix-mode doctor repair at command entry before knowing whether the repair touches openclaw.json. Auth login is partly improved, but provider config patches and paste-token/API-key paths still couple credential writes to config writes, so the central state-versus-config boundary is not fully solved.

## What This Changes

The issue asks OpenClaw to block only openclaw.json mutations in Nix mode while allowing auth-profile and session-state writes that live in mutable state.

## Best Possible Solution

Refactor Nix mode so config writes remain fail-closed, but auth-store and session-state mutations can complete with focused tests and updated Nix/doctor docs for the allowed and blocked repair classes.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.8 call the Nix guard before doctor repair selection, and auth paste paths still call updateConfig after writing profiles. I did not run a live Nix/TTY OAuth repro in this read-only review.

## Solution Assessment

No, the current implementation is too coarse because it guards whole commands instead of actual openclaw.json writes. The maintainable fix is to preserve low-level config write refusal while letting state-only auth/session repairs proceed.
