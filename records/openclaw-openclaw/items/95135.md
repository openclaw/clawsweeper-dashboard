## Summary

Keep open: current main and the latest release still block the reported Nix-mode doctor repair path before repair selection, and the related candidate PRs remain open and unmerged. ## What This Changes

Allow Nix-mode doctor and model-auth operations to perform auth-profile and session-state writes that do not mutate `openclaw.json`, while still blocking actual config-file mutations.

## Best Possible Solution

Confirm the intended Nix-mode boundary, then land or replace the open candidate PR with doctor/auth tests and docs that allow non-config auth/session-state repairs while preserving refusal for actual `openclaw.json` writes.

Yes. Source inspection shows current main and the latest release check Nix mode before doctor repair selection, while auth paths still combine auth-store writes with guarded config updates in some flows; I did not run a live Nix/OAuth command in this read-only review.

## Solution Assessment

Unclear. Moving enforcement to actual `openclaw.json` writes is the likely maintainable direction, but current docs and tests deliberately encode broader Nix-mode doctor refusal, so maintainers need to confirm the product boundary before this is solved.
