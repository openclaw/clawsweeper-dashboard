## Repair Work Prompt

Fix this issue by reducing repeated plugin metadata scans in hot Gateway/subagent runtime paths. Current main calls loadPluginMetadataSnapshot synchronously from loadEnabledBundlePiSettingsSnapshot during embedded Pi settings/resource-loader setup, and many production call sites still call it directly. Do not add a wall-clock TTL or hidden persistent cache to loadPluginMetadataSnapshot; docs/plugins/architecture-internals.md forbids that for discovery/manifest answers. Prefer getCurrentPluginMetadataSnapshot or explicit PluginMetadataSnapshot/PluginLookUpTable threading so compatible default-workspace runtime calls reuse the Gateway startup snapshot, while incompatible workspace/config/index cases still fall back to a fresh load. Add focused regression coverage proving the hot bundle settings/resource-loader path avoids loadPluginMetadataSnapshot when a compatible current snapshot exists and falls back when it is incompatible. Add a user-facing changelog fix entry if behavior changes. Validate with targeted tests first, then the changed gate.
