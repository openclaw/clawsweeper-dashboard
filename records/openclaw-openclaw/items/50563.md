## Summary

Current main and the latest release still have the reported WhatsApp ack-before-process window: accepted inbound messages can be marked read before durable storage, dedupe finalization, or handler delivery. Ryce's KeepMyClaw comment is useful complementary crash-state recovery context, but it does not replace fixing the bundled WhatsApp plugin's acknowledgment ordering.

## What This Changes

The issue asks for WhatsApp inbound messages to be durably stored or processed before the gateway sends a read receipt, so a crash cannot silently drop an already-acked message.

## Best Possible Solution

Delay `sock.readMessages` until accepted inbound turns are either durably stored for replay or have successfully reached `onMessage` and inbound dedupe finalization, with focused tests and docs for the new crash-recovery boundary.

## Reproduction Assessment

Yes, as a source-level reproduction path. A WhatsApp `notify` upsert reaches `sock.readMessages` before append filtering, enrichment, dedupe claim/commit, and debounced `onMessage` delivery on current main.

## Solution Assessment

Yes. The issue's delayed-ack or persist-before-ack direction is the narrow maintainable fix; full gateway state snapshotting is complementary, not a substitute for correcting the plugin-owned read-receipt boundary.
