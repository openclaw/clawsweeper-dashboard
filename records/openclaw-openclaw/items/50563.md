## Label Justifications

- `P1`: The issue described a real broken WhatsApp channel workflow where a gateway crash could silently lose an acknowledged inbound message.
- `impact:message-loss`: The affected behavior was an inbound WhatsApp message being acknowledged before OpenClaw had durably accepted or delivered it.
- `impact:crash-loop`: The failure mode depends on a gateway crash or restart during the acknowledgment-to-processing window.

## Summary

Close as implemented on main: replacement PR https://github.com/openclaw/openclaw/pull/85506 merged a plugin-state durable receive journal for WhatsApp, persists accepted inbound messages before handler dispatch, replays pending records, and delays read receipts until finalization. The fix is current-main-only; v2026.5.22 still has the old eager read-receipt path.

## What This Changes

The issue asks for WhatsApp inbound messages to survive gateway crashes by avoiding acknowledgment before durable processing.

## Best Possible Solution

Keep the merged https://github.com/openclaw/openclaw/pull/85506 implementation and let the next release ship the current-main WhatsApp durable receive fix.

## Reproduction Assessment

Yes, source-level: v2026.5.22 calls WhatsApp readMessages before enrichment, dedupe, enqueue, and handler completion. Current main no longer reproduces that path because accepted inbound messages enter the durable journal before handler dispatch and read receipts are delayed until finalization.

## Solution Assessment

Yes: the merged plugin-state durable receive journal is the maintainable fix because it handles accepted-message persistence, retryable release, replay, dedupe completion, and delayed read receipts together. A read-receipt reorder alone would not have covered crash recovery.
