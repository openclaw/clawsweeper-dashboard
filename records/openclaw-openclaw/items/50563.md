## Summary

Keep open. Current main still has the reported WhatsApp ack-before-process window: accepted inbound messages are marked read before enrichment, inbound dedupe claim/commit, debounced `onMessage` delivery, or any durable inbound replay boundary. Ryce's KeepMyClaw comment is useful complementary recovery context for gateway state snapshots, but it does not replace fixing the bundled WhatsApp plugin's acknowledgment boundary.

## What This Changes

Request to prevent WhatsApp inbound message loss when the gateway crashes after acknowledging a message but before processing it, by using WAL, persist-before-ack, or delayed ack behavior.

## Best Possible Solution

Fix the bundled WhatsApp plugin by introducing a durable or successfully-processed inbound acknowledgment boundary: either persist accepted inbound messages before `readMessages`, or delay `readMessages` until `onMessage` succeeds and inbound dedupe is finalized. Keep the change plugin-owned unless multiple channel plugins need a shared durable inbound SDK seam.

## Reproduction Assessment

Yes. A default accepted WhatsApp `notify` upsert follows a static path where `sock.readMessages` is awaited before dedupe claim, debounced `onMessage`, and session recording; a crash in that interval matches the reported loss window.

## Solution Assessment

Yes. The issue's persist-before-ack or delayed-ack direction is the narrow maintainable fix; a broad core migration is not needed for the minimal WhatsApp plugin repair unless implementation proves a reusable channel SDK seam is required.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/50563 in the bundled WhatsApp plugin. Current main sends WhatsApp read receipts in `extensions/whatsapp/src/inbound/monitor.ts` before accepted inbound messages are durably recorded or delivered to `onMessage`; a gateway crash in that window can lose the message from OpenClaw's side. Implement a plugin-owned acknowledgment boundary: either persist accepted inbound messages before `sock.readMessages`, or delay `readMessages` until successful `onMessage` delivery plus inbound dedupe finalization. Keep the scope to WhatsApp unless a small, backwards-compatible SDK seam is clearly required. Do not change unrelated channels or broad gateway startup behavior. Update focused tests that currently assert eager read receipts, add ordering coverage for failure/crash-like `onMessage` behavior, update WhatsApp docs if user-visible read receipt timing changes, and add a single-line user-facing `CHANGELOG.md` fix entry without inventing bot/maintainer thanks credit.
