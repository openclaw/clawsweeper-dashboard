## Summary

Keep open. Current main still sends WhatsApp read receipts before durable inbound storage, inbound dedupe commit, debounced handler delivery, or downstream session/agent processing, so the reported crash-loss window remains present. Ryce's KeepMyClaw comment is useful complementary gateway-state recovery context, but it does not replace fixing the bundled WhatsApp plugin acknowledgment boundary.

## What This Changes

The issue asks for WhatsApp gateway inbound acknowledgments to be delayed or made durable so a crash after receipt does not silently lose a message.

## Best Possible Solution

Keep the issue open and fix the bundled WhatsApp plugin's acknowledgment boundary: either persist accepted inbound messages before calling `readMessages`, or delay `readMessages` until the accepted turn has successfully reached the chosen durable or processed boundary, with focused tests and docs covering the new ordering.

## Reproduction Assessment

Yes. A default accepted WhatsApp `notify` upsert follows a static current-main path where `sock.readMessages` is awaited before enrichment, dedupe claim/commit, debounced `onMessage`, and downstream session/agent processing; a crash in that interval matches the reported loss window.

## Solution Assessment

Yes. The requested persist-before-ack or delayed-ack direction is the maintainable fix; the narrowest first attempt should stay in the WhatsApp plugin unless implementation proves multiple channel plugins need a shared durable inbound seam.
