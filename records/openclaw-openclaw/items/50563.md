## Summary

Keep this open: current main still sends WhatsApp read receipts before accepted inbound message storage, dedupe finalization, or handler delivery, so the reported crash-loss window remains. Ryce's KeepMyClaw comment is useful complementary crash-state recovery context, but it does not replace a plugin-owned acknowledgment ordering fix.

## What This Changes

The issue requests preventing WhatsApp inbound message loss when the gateway crashes after acknowledging receipt but before processing completes.

## Best Possible Solution

Fix the bundled WhatsApp plugin by delaying `readMessages` until each accepted inbound turn is durably stored for replay or has successfully reached `onMessage` and inbound dedupe finalization, with focused regression tests and docs for the new ordering.

## Reproduction Assessment

Yes, as a source-level reproduction path. A WhatsApp `notify` upsert reaches `sock.readMessages` before append filtering, enrichment, dedupe claim/commit, and debounced `onMessage` delivery on current main; I did not run a live crash harness in this read-only review.

## Solution Assessment

Yes. The issue's delayed-ack or persist-before-ack direction is the narrow maintainable fix; full gateway state snapshotting is complementary, not a substitute for correcting the plugin-owned read-receipt boundary.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/50563 in the bundled WhatsApp plugin. Current main calls `sock.readMessages` in `extensions/whatsapp/src/inbound/monitor.ts` before append filtering, enrichment, inbound dedupe claim/commit, and `onMessage` delivery. Move the read-receipt boundary so accepted inbound turns are marked read only after they are durably stored for replay or after successful `onMessage` delivery plus inbound dedupe finalization. Keep the fix plugin-local unless a genuinely generic SDK seam is required. Preserve self-chat no-read-receipt behavior and `sendReadReceipts: false`; handle append/history messages, media/enrichment failures, debounced batches, and `WhatsAppRetryableInboundError` without introducing duplicate replies or premature read receipts. Add focused regression coverage for the new ordering and update WhatsApp docs/changelog for the user-visible reliability change.
