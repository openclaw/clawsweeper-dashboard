## Label Justifications

- `P1`: The issue describes a broken channel workflow where a gateway crash can silently lose an inbound user message after the sender sees acknowledgment state.
- `impact:message-loss`: The affected behavior is an inbound WhatsApp message being acknowledged before OpenClaw has safely delivered or recorded it.
- `impact:crash-loop`: The failure mode is specifically about process crash or restart during the acknowledgment-to-processing window.

## Summary

Keep open: current main and the latest shipped release still send WhatsApp read receipts before accepted inbound messages reach enrichment, dedupe commit, or `onMessage` delivery, and there is no durable WhatsApp inbound replay boundary. Dependency inspection also shows Baileys read-message calls are read receipts, so the fix should use local durable handoff/replay rather than assuming WhatsApp will redeliver after a crash.

## What This Changes

The issue asks for WhatsApp inbound messages to be persisted or processed before the gateway acknowledges them so a crash cannot silently drop an already-acked message.

## Best Possible Solution

Add a WhatsApp plugin-owned durable inbound handoff before read receipts and handler delivery, then replay or safely finalize accepted messages so a crash cannot turn an acknowledged inbound message into silence.

## Reproduction Assessment

Yes, as a source-level path: a WhatsApp notify upsert reaches `sock.readMessages` before append filtering, enrichment, dedupe claim/commit, and debounced `onMessage` delivery. I did not run a live crash harness in this read-only review.

## Solution Assessment

Yes for the durable handoff direction, with one caveat: moving `readMessages` later is not enough if Baileys has already acknowledged delivery upstream. The maintainable fix should persist or replay accepted inbound messages in the WhatsApp plugin before any user-visible read receipt or handler side effect.
