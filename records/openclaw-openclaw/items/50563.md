## Summary

Keep open. Current main still sends WhatsApp read receipts before the accepted inbound message is enriched, deduped, delivered to `onMessage`, or persisted for replay; Ryce's KeepMyClaw comment is useful crash-state recovery context, but it does not replace fixing the bundled WhatsApp plugin's acknowledgement boundary.

## What This Changes

The issue requests a durable or delayed WhatsApp inbound acknowledgement boundary so gateway crashes cannot lose messages after a read receipt but before processing.

## Best Possible Solution

Fix this in the bundled WhatsApp plugin by delaying `readMessages` until accepted inbound messages are durably stored or have successfully reached `onMessage` and inbound dedupe finalization, with docs and focused tests for the new ordering.

## Reproduction Assessment

Yes. A default accepted WhatsApp `notify` upsert still follows a static path where `sock.readMessages` is awaited before enrichment, dedupe claim/commit, and debounced `onMessage`; a crash in that interval matches the reported loss window.

## Solution Assessment

Yes. The issue's persist-before-ack or delayed-ack direction is the narrow maintainable fix; KeepMyClaw-style state snapshots are complementary and a broad core migration is not required for the minimal WhatsApp plugin repair.
