## Summary

Keep open: current main still sends WhatsApp read receipts before durable inbound storage, inbound dedupe finalization, or handler delivery, so the reported crash-loss window remains. Ryce's KeepMyClaw context is useful complementary crash-state recovery, but it does not replace a plugin-owned acknowledgment ordering fix.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp inbound message loss when the gateway crashes after acknowledging the message but before processing completes.

## Best Possible Solution

Fix the bundled WhatsApp plugin by delaying readMessages until each accepted inbound turn is durably stored for replay or has successfully reached onMessage and inbound dedupe finalization, with focused regression coverage and docs for the new ordering.

## Reproduction Assessment

Yes, as a source-level path: a WhatsApp notify upsert reaches sock.readMessages before append filtering, enrichment, dedupe claim/commit, and debounced onMessage delivery. I did not run a live crash harness in this read-only review.

## Solution Assessment

Yes. The narrow maintainable direction is plugin-owned delayed-read-receipt or persist-before-read behavior; full gateway snapshotting such as KeepMyClaw is complementary recovery context, not a substitute for correcting the WhatsApp acknowledgment boundary.
