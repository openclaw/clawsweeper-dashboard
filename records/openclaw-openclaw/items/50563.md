## Label Justifications

- `P1`: The source path shows a real WhatsApp channel workflow where a crash can silence an inbound message after read acknowledgment.
- `impact:message-loss`: The affected behavior is an inbound WhatsApp message being acknowledged before OpenClaw has safely delivered or durably recorded it.

## Summary

Keep open: current main and the latest release still mark accepted WhatsApp inbound messages read before filtering append history, enriching content, claiming durable dedupe, or delivering to `onMessage`. Ryce's KeepMyClaw comment is useful crash-state context, and samzong's latest comment describes the right durable-acceptance direction, but there is not yet a merged or open closing PR for this issue.

## What This Changes

The issue reports that WhatsApp inbound messages can be acknowledged before OpenClaw durably accepts or processes them, allowing crash-time message loss.

## Best Possible Solution

Add a WhatsApp plugin-owned durable accepted-inbound handoff and replay/finalization path so accepted messages survive restart before read receipts or handler side effects are finalized.

## Reproduction Assessment

Yes, as a high-confidence source-level path: a `messages.upsert(type="notify")` item reaches `sock.readMessages` before append filtering, enrichment, dedupe claim/commit, and `onMessage` delivery. I did not run a live WhatsApp crash harness in this read-only review.

## Solution Assessment

Yes for the WAL/persist-before-ack direction, with one caveat: delaying read receipts alone is not enough unless implementation proves reliable platform redelivery. The maintainable fix should durably accept and replay inbound WhatsApp records in the plugin, then finalize dedupe/read receipts after handler success.
