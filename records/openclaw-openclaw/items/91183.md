## Summary

Keep open: the reporter restored service by selecting a registered provider and rebuilding, but current stable still intentionally treats pre-identity indexes as unsafe, pauses vector recall, and uses a file-swap rebuild that can fail on Windows while the gateway holds the database. The remaining upgrade and repair behavior is therefore a real compatibility decision, not fully resolved by the installation-specific workaround.

## What This Changes

Address an upgrade-time memory search outage where newly required index identity metadata is absent, provider resolution degrades to FTS-only, and rebuilding can fail on Windows due to an open SQLite file.

## Best Possible Solution

Keep index-identity validation fail-closed, but define and prove a supported upgrade path that diagnoses legacy missing metadata, activates the canonical configured provider, and coordinates an explicit Windows-safe gateway stop/rebuild/restart without silently spending embedding cost.

## Reproduction Assessment

Partly yes: current source deterministically reproduces the dirty/paused result for missing identity metadata and the rename-based Windows lock failure is consistent with the reported EBUSY. The exact beta.1-to-beta.2 metadata transition has not been independently executed, so full live reproduction remains incomplete.

## Solution Assessment

No, the reporter's manual provider change and rebuild are an effective recovery but not the best repository-level resolution. The maintainable solution is an explicit upgrade/doctor or CLI repair contract that preserves safety, avoids automatic embedding spend, and coordinates the live gateway on Windows.
