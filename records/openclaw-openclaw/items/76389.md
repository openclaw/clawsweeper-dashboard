## Summary

Keep open. The linked security gap is valid, but this PR still has blocking parser bypasses, a build-shape error, a broad exec behavior regression, and a stale changelog overwrite.

## What This Changes

The PR adds a gateway exec SSRF filter for selected network tools, wires it into `createExecTool`, adds unit coverage, and rewrites the top of `CHANGELOG.md`.

## Best Possible Solution

Land a maintainer/security-owner-approved exec egress design that either provides robust isolation/proxy enforcement or a parser mitigation that preserves existing exec semantics and has complete curl/wget URL coverage.

## Reproduction Assessment

Yes, at source level. Current main lacks an exec SSRF destination guard, and the PR head still permits direct localhost/private URL forms such as query-string URLs and common curl/wget standalone flags while overblocking unrelated shell syntax.

## Solution Assessment

No. The PR addresses a real gap, but this parser-first patch is not the best mergeable solution until the bypasses, build error, normal exec semantics, and security-boundary decision are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Validate URL arguments before splitting on equals:** `src/infra/exec-ssrf-filter.ts:615-624`
  - body: A normal URL can contain `=`, so `curl 'http://127.0.0.1/?x=1'` enters this option-assignment branch and is ignored unless the token is `--url` or `-u`. That leaves direct localhost/private URLs reachable; the same shape is skipped in the wget branch.
  - confidence: 0.95
- **[P1] Classify curl standalone flags before consuming values:** `src/infra/exec-ssrf-filter.ts:608-610`
  - body: The value-taking set is checked before standalone flags, and it still contains no-value curl flags such as `--compressed`, `-S`, `-v`, and `--help`. Commands like `curl --compressed http://127.0.0.1/` still skip the URL as an option value and pass the filter.
  - confidence: 0.94
- **[P1] Classify wget standalone flags before consuming values:** `src/infra/exec-ssrf-filter.ts:655-658`
  - body: `WGET_OPTIONS_WITH_VALUE` is also checked before standalone flags and still includes no-value wget flags such as `-d`, `-v`, `-r`, and `--spider`. `wget -d http://localhost/file` therefore still skips the blocked URL.
  - confidence: 0.94
- **[P1] Return the required fields for parse-denied results:** `src/infra/exec-ssrf-filter.ts:759-762`
  - body: `ExecSsrfFilterResult` requires `blockedHost` and `blockedUrl` whenever `allowed` is false, but this parse-failure branch returns only `reason`. The PR will fail type checking, and `validateExecCommandSsrF` would format `undefined` fields if it compiled.
  - confidence: 0.96
- **[P1] Avoid rejecting unrelated gateway exec syntax:** `src/agents/bash-tools.exec.ts:1618-1619`
  - body: This call runs the SSRF parser for every gateway exec command before knowing whether a network tool is present. Since the parser returns false for supported shell syntax such as redirection, command substitution, and empty commands, ordinary non-network exec calls would now fail with an SSRF error even in the default gateway full/off mode.
  - confidence: 0.91
- **[P2] Preserve existing Unreleased changelog entries:** `CHANGELOG.md:7-16`
  - body: The changelog change replaces the current Unreleased section and removes the existing highlight plus many unrelated fixes. This PR should add the gateway exec SSRF entry without dropping current-main release notes.
  - confidence: 0.98
