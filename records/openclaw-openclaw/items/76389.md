## Summary

Keep open. Current main still has the host-exec SSRF gap, but this PR is not merge-ready because the new guard misses common curl/wget forms and fails open on unsupported shell syntax in a security-sensitive path.

## What This Changes

The PR adds a gateway-host exec SSRF filter for curl/wget/httpx/aria2c-style commands, wires it into `createExecTool`, adds tests, and updates the changelog.

## Best Possible Solution

Land a security-owner-approved exec egress policy that either uses a robust sandbox/egress boundary or a clearly documented fail-closed argv mitigation with regression tests for common network-tool forms and known bypass classes.

## Reproduction Assessment

Yes, source-reproducible. Current main has no gateway-host exec SSRF check, and the PR branch source shows direct bypasses such as `curl -L http://127.0.0.1:18760/status/500` and `wget -q http://localhost/file`.

## Solution Assessment

No, not as written. A parser-based mitigation may be acceptable as a stopgap, but this implementation must not skip common URL forms or fail open on shell syntax it cannot analyze; maintainers should also decide whether egress isolation is required.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Do not skip URLs after standalone curl flags:** `src/infra/exec-ssrf-filter.ts:623-625`
  - body: Because `CURL_OPTIONS_WITH_VALUE` is checked before the standalone set and includes flags such as `-L`, `-k`, `-I`, `-4`, and `--compressed`, `curl -L http://127.0.0.1:18760/status/500` advances past the URL as if it were an option value and returns allowed. That leaves a common curl fallback path unblocked.
  - confidence: 0.94
- **[P1] Validate URL-bearing option assignments:** `src/infra/exec-ssrf-filter.ts:634-637`
  - body: The blanket `--option=value` skip means direct curl URL forms such as `curl --url=http://127.0.0.1:18760/status/500` are never checked. Parse URL-bearing options explicitly instead of discarding every long option assignment before hostname validation.
  - confidence: 0.91
- **[P1] Classify wget flags before consuming values:** `src/infra/exec-ssrf-filter.ts:663-665`
  - body: `WGET_OPTIONS_WITH_VALUE` also contains standalone flags that are checked first, including `-q`. As a result, `wget -q http://localhost/file` skips the blocked URL and is allowed, which leaves the obvious wget fallback path open.
  - confidence: 0.93
- **[P2] Fail closed when network command parsing is unsupported:** `src/infra/exec-ssrf-filter.ts:762-765`
  - body: When `analyzeShellCommand` rejects syntax such as command substitution, the filter returns allowed even though gateway exec still runs the original shell command. Unsupported network-tool commands need a conservative scan or denial path.
  - confidence: 0.88
