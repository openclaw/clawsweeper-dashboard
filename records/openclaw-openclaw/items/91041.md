## What This Changes

The PR propagates imsg status messages into iMessage private-API blocked-action errors and adds a 150s iMessage send default when no explicit or probe timeout is configured.

## Best Possible Solution

Land as a maintainer-approved staged mitigation only if maintainers accept the documented 150s availability tradeoff and track the companion imsg rollout; otherwise defer or make the longer wait opt-in until the dependency ships.

## Reproduction Assessment

Partly: the blocked-action diagnostics path has high-confidence live CLI proof from the PR discussion, and current source shows why the old error lacked imsg's reason. The delayed 60-124s send stall is not reproducible on demand, so that half rests on source, unit, dependency-contract proof, and a recorded maintainer waiver.

## Solution Assessment

Yes as an acceptable staged mitigation, not a fully self-contained dependency fix. The code keeps explicit and probe timeouts authoritative, scopes the longer default to sends, and relies on https://github.com/openclaw/imsg/pull/139 for the bridge-layer default before the timeout benefit fully activates.
