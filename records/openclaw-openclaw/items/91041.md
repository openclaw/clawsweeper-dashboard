## What This Changes

This PR propagates imsg status messages into iMessage private-API blocked-action errors and gives iMessage sends a 150s default timeout when no explicit or probe timeout is configured.

## Best Possible Solution

Land this only with maintainer acceptance of the 150s send wait and dependency rollout plan, keeping the OpenClaw patch as a partial mitigation for diagnostics while openclaw/imsg#139 ships.

## Reproduction Assessment

Yes, source inspection of current main shows iMessage sends fall back to the 10s RPC default and private-API blocked actions do not include imsg's status message. The macOS 26 delayed-send stall itself is not reproducible on demand, so live proof covers the statusMessage half while timeout behavior is source/unit/dependency-proven.

## Solution Assessment

Yes, this is an acceptable mitigation rather than a complete standalone fix: OpenClaw preserves fast probes while giving sends a longer wait, and openclaw/imsg#139 supplies the matching bridge-layer timeout. The remaining question is maintainer acceptance of the 150s wait window before that dependency ships.
