## Summary

Keep open: the bridge is a reasonable fix direction and the updated body supplies live copied proof for the bridge/resolver path, but the patch still misses Codex bound conversation startup paths that already carry config and do not pass it into the shared app-server client.

## What This Changes

The PR injects resolved Gateway auth into isolated Codex stdio app-server environments, exports Gateway auth helpers through the plugin SDK, and adds tests, docs, and changelog coverage.

## Best Possible Solution

Keep the bridge stdio-only and parent-resolved, then make every Codex app-server launch path that can run OpenClaw CLI subprocesses pass the parent runtime config into the shared client.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main isolates Codex app-server `HOME`, the new bridge only injects Gateway auth with config, and bound conversation callers omit that config. A live repro would configure file-backed `gateway.auth.*`, launch the bound Codex stdio app-server path, then run an OpenClaw CLI child from that environment.

## Solution Assessment

No, not yet: the helper placement is reasonable, but the patch must consistently pass parent config through the remaining stdio startup paths before it solves the reported behavior.

## Review Findings
