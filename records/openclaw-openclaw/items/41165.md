## Summary

Keep open: current main and the latest release still let an event-originating forced heartbeat session win before an explicit `heartbeat.session`, while the requested Telegram-DM default isolation remains a compatibility/product decision. The open linked fork PR uses closing syntax, so the issue should wait for maintainer review of that fix path rather than be closed early.

## What This Changes

The issue asks OpenClaw to prevent Telegram DM, heartbeat, and system-event traffic from sharing or polluting `agent:main:main`, especially when heartbeat is configured for a dedicated session.

## Best Possible Solution

Make explicit `heartbeat.session` win over event-originating forced session keys with safe system-event queue handling, then separately decide whether Telegram DMs should keep the documented `dmScope=main` default or migrate to stricter isolation.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: event wake session keys flow into `forcedSessionKey`, and `resolveHeartbeatSession` handles that branch before `heartbeat.session`. I did not run a fresh live Telegram/exec reproduction.

## Solution Assessment

Unclear as one combined change. The heartbeat-session precedence fix is narrow, but changing Telegram's default DM-to-main behavior needs a maintainer compatibility decision.
