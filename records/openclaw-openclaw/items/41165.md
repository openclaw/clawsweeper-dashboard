## Summary

Keep open: current main and the latest release still let an exec/system-event wake carry an originating session into heartbeat resolution before `heartbeat.session` is considered, and the linked fix PR is open but unmerged/conflicting. The Telegram default-DM isolation part is still a product/docs decision because current docs and tests still describe main-session sharing by default.

## What This Changes

The issue reports Telegram direct-message and heartbeat/session-routing contamination between isolated Telegram DM sessions and the main heartbeat session.

## Best Possible Solution

Land or replace a narrow fix that makes configured heartbeat sessions win for system-event-triggered wakes, preserves pending events safely, and separately documents or decides any Telegram default-DM isolation change.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the system-event variant: an exec completion in a DM session requests heartbeat with the originating session key, and heartbeat resolution returns that forced session before honoring `heartbeat.session`. The broader Telegram default-DM isolation request is not a current contract because docs/tests still default DMs to main.

## Solution Assessment

No single close/fix is ready yet. The linked PR targets the right source-level heartbeat precedence bug, but maintainers still need to rebase or replace it and decide whether Telegram default-DM-to-main behavior should change or remain documented.
