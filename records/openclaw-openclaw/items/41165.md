## Label Justifications

- `P1`: Real Telegram workflows can execute heartbeat duties in the wrong persistent conversation and can silently lose expected heartbeat delivery.
- `impact:session-state`: Heartbeat and direct-message context can be persisted under the wrong session identity.
- `impact:message-loss`: A corroborating reproduction reported outbound heartbeat delivery being silently swallowed after the session became polluted.

## Summary

Current main still gives a valid wake-triggered originating session precedence over an explicit heartbeat session, matching the transcript-backed exec-completion contamination path; an open fork PR already owns a proposed fix, but its cross-session event migration semantics require maintainer review.

## What This Changes

Prevent Telegram direct-message and heartbeat work from reusing or polluting each other’s persistent sessions, including heartbeats triggered by system events.

## Best Possible Solution

Honor an explicit canonical heartbeat session for wake-triggered runs while leaving genuinely origin-scoped events in their source session, with focused coverage for exec completion, configured sessions, Telegram delivery, skipped runs, deduplication, and concurrent event arrival.

## Reproduction Assessment

Yes, at source level: exec/system-event wakes can supply the originating DM as `forcedSessionKey`, and current main resolves that before the configured heartbeat session; multiple transcript-backed reports match this path. A fresh live current-main reproduction was not run.

## Solution Assessment

The isolation goal is correct, but a simple precedence swap is not sufficient because queued events have different ownership; the linked migration-and-restore approach is plausible but needs maintainer agreement and validation against current event-routing invariants.
