## Summary

Keep open. Current main still has a source-reproducible path where exec/system-event heartbeat wakes preserve the originating session key and `resolveHeartbeatSession` accepts that forced key before `heartbeat.session`; the linked implementation PR is still open and unmerged.

## What This Changes

The issue asks for Telegram DM and heartbeat/system-event traffic to stay isolated so ordinary DMs and heartbeat turns do not pollute each other’s sessions.

## Best Possible Solution

Make configured `heartbeat.session` win for system-event-triggered wakes, preserve or migrate pending system events without cross-session pollution, and separately decide or document Telegram default DM-to-main behavior.

## Reproduction Assessment

Yes, source-level. Current main lets exec notify-on-exit enqueue and wake heartbeat with the originating session key, and heartbeat resolution accepts that forced key before the documented `heartbeat.session` setting.

## Solution Assessment

Yes for keeping this issue open behind the linked implementation candidate. The narrow fix is configured heartbeat-session precedence for system-event-triggered wakes; default Telegram DM isolation should remain a separate maintainer product decision because prior review rejected an implicit Telegram-only policy shift.
