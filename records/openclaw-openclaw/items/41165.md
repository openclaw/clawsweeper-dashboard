## Summary

Keep open: current main still lets a forced/originating heartbeat session resolve before an explicit `heartbeat.session`, while Telegram private DMs still follow the documented global `session.dmScope` default rather than forced Telegram isolation. Newer merged work narrows related event-wake queue drift, but it does not close the configured-heartbeat-session precedence or Telegram DM product-policy questions.

## What This Changes

The issue reports Telegram direct messages and system-event-triggered heartbeats crossing session boundaries, polluting heartbeat or direct-message session history.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers either land a bounded current-main fix that makes configured heartbeat sessions win for system-event-triggered wakes with safe event handling, or explicitly document/decide the remaining Telegram DM default behavior.
