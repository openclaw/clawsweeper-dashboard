## Summary

Keep open: current main still lets event-triggered heartbeat wakes carry an originating session into heartbeat session resolution before explicit `heartbeat.session`, and Telegram DMs still follow the documented main-session default unless `session.dmScope` is configured. The linked fork PR is an implementation candidate, but it is open, conflicting, and not a safe reason to close this issue before maintainer review.

## What This Changes

Reports that Telegram DM and heartbeat/system-event routing can reuse `agent:main:main` or another wrong session, polluting heartbeat/main context after the duplicate-reply fix.

## Best Possible Solution

Make explicit heartbeat session configuration win for system-event-triggered wakes with safe system-event handling, then separately decide whether Telegram DMs should keep the documented main-session default or move to an isolated default with tests and docs.

## Reproduction Assessment

Yes for source-level reproduction: exec/system-event wakes still preserve an originating session key, the wake queue forwards it, and current heartbeat resolution considers that forced session before configured `heartbeat.session`. I did not run a fresh live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear as one combined change: the heartbeat precedence repair is narrow, but changing Telegram's documented default DM-to-main behavior needs maintainer product judgment. The best path is to review or replace the linked heartbeat implementation and split the Telegram policy decision if needed.
