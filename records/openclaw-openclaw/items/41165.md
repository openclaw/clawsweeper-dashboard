## Summary

Keep open: current main and the latest release still let a wake-triggered originating session take precedence over an explicit `heartbeat.session`, while the broader Telegram-DM-to-main default remains an intentional product-policy decision rather than an already-merged fix.

## What This Changes

The issue reports Telegram DM and heartbeat/system-event routing contamination between `agent:main:main`, isolated Telegram direct sessions, and explicitly configured heartbeat sessions.

## Best Possible Solution

Honor explicit configured heartbeat sessions for wake-triggered runs while preserving origin-scoped events correctly, and handle any Telegram default-DM isolation change as a separate cross-channel product decision.

## Reproduction Assessment

Yes, at source level: exec/system-event wakes can supply an originating session as `forcedSessionKey`, and current main resolves that before an explicit `heartbeat.session`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No final fix is in place yet. The best direction is a focused heartbeat/event-routing repair, not a broad implicit Telegram `dmScope` policy change bundled into the same fix.
