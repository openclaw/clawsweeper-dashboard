## Summary

Keep open. Current main still lets exec/system-event heartbeat wakes carry an originating session that is resolved before an explicit `heartbeat.session`, and an open closing PR already tracks that fix; Telegram DMs also still default to main by documented and tested behavior, so changing that default needs maintainer product judgment.

## What This Changes

Telegram DM and heartbeat traffic should not pollute each other by reusing `agent:main:main` when a direct or configured heartbeat session should be isolated.

## Best Possible Solution

Land or supersede #56092 with a narrow fix that makes configured heartbeat sessions win for system-event-triggered wakes and handles pending system events without cross-session pollution; treat Telegram default DM isolation as a separate docs/product decision with matching tests if maintainers want to change it.

## Reproduction Assessment

Yes for the confirmed exec-completion subcase: source shows `notifyOnExit` scopes the wake to the originating session, the wake scheduler forwards that session key, and `resolveHeartbeatSession` resolves it before `heartbeat.session`. The original Telegram-only path remains unclear without a fresh live repro and overlaps documented default `dmScope=main` behavior.

## Solution Assessment

No for current main. The maintainable fix is to review and land #56092 or an equivalent narrow heartbeat session-precedence/event-migration patch, then separately decide whether Telegram default DMs should stop sharing main sessions.
