## Summary

Keep open: current main and the latest release still let a valid event-originating forced heartbeat session resolve before an explicit `heartbeat.session`, and Telegram DM-to-main sharing remains current documented behavior rather than a completed isolation fix. The issue is paired with an open closing PR, but that fork PR is currently conflicting, so maintainers should review or replace that fix path instead of closing this early.

## What This Changes

The issue requests strict isolation between Telegram DM sessions and heartbeat/main sessions so ordinary DMs and system-event-triggered heartbeats cannot pollute each other’s transcripts or delivery context.

## Best Possible Solution

Make explicit `heartbeat.session` win for system-event-triggered heartbeats with safe pending-event handling, then separately decide whether Telegram DMs should continue the documented main-session default or migrate to stricter isolation with tests and docs.

## Reproduction Assessment

Yes for source-level reproduction: exec/system-event wake paths still pass an originating session key into `forcedSessionKey`, and `resolveHeartbeatSession` resolves that before `heartbeat.session`. I did not run a fresh live Telegram/exec reproduction.

## Solution Assessment

Unclear as one combined fix: the heartbeat-session precedence repair is narrow, but changing Telegram’s default DM-to-main behavior would alter documented session continuity and needs maintainer product judgment.
