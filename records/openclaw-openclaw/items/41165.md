## Summary

Keep open. Current main still has the confirmed exec/system-event heartbeat routing path where a wake-supplied originating session can win before an explicit `heartbeat.session`, and Telegram default-DM behavior is still documented and tested as main-session sharing; #56092 remains the open closing PR to review rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to keep Telegram DM traffic and heartbeat/main-session traffic isolated so direct chats cannot pollute `agent:main:main` after the duplicate-reply fix in #40519.

## Best Possible Solution

End state should make configured heartbeat sessions win for system-event-triggered wakes, preserve or migrate pending system events without cross-session pollution, and separately settle whether Telegram default DMs should remain documented main-session sharing or move to isolated direct sessions with matching tests and docs.

## Reproduction Assessment

Yes for the confirmed exec-completion subcase: current code enqueues `notifyOnExit` system events on the originating session, forwards that session through the wake queue, and resolves it before `heartbeat.session`. The original Telegram-only path remains unclear without a fresh minimal live repro.

## Solution Assessment

No for current main. The maintainable path is to review and land #56092 or an equivalent narrow fix for heartbeat wake session precedence, then handle Telegram DM default isolation as a separate product/code decision if desired.
