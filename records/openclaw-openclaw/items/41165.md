## Summary

Keep open. Current main still has the source path for exec/system-event heartbeats to prefer an originating session before an explicit `heartbeat.session`, and the linked open PR #56092 is a closing implementation candidate that should not be preempted.

## What This Changes

The issue asks for Telegram direct messages and heartbeat/system-event traffic to avoid sharing or polluting `agent:main:main`.

## Best Possible Solution

Configured heartbeat sessions should win for system-event-triggered wakes, pending exec/system events should not pollute the originating DM/main transcript, and Telegram default DM isolation should be either intentionally preserved in docs or changed with explicit tests and migration guidance.

## Reproduction Assessment

Yes by source inspection for the heartbeat path: exec completion can enqueue a session-scoped event, forward that session into the wake queue, and have `resolveHeartbeatSession` consume it before explicit `heartbeat.session`. I did not run a live Telegram E2E reproduction.

## Solution Assessment

No, closing is not the best current outcome because the relevant source path remains and #56092 is still open with closing syntax. The narrow maintainable fix is to make configured heartbeat sessions outrank originating system-event sessions, then separately decide whether Telegram DMs should stop defaulting to main.
