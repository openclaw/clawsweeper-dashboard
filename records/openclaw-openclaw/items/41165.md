## Summary

Keep open: current main and the latest release still leave the combined session-routing concern unresolved. The merged event-routing work improves single-owner `dmScope=main` exec/cron wakes, but `resolveHeartbeatSession` still lets an originating wake session win before an explicit `heartbeat.session`, and Telegram DM-to-main behavior remains the documented/default path rather than the requested strict DM isolation.

## What This Changes

The issue requests that Telegram direct messages and heartbeat/system-event-triggered turns stay isolated from `agent:main:main` so heartbeat/main session context is not polluted by ordinary DM traffic.

## Best Possible Solution

Fix the source-level heartbeat precedence so explicit `heartbeat.session` wins for system-event-triggered wakes, then separately decide whether Telegram's default DM-to-main behavior should remain documented or change behind a compatibility-conscious migration/test plan.

## Reproduction Assessment

Yes for a source-level reproduction path: current main still passes wake session keys into heartbeat as `forcedSessionKey`, and `resolveHeartbeatSession` handles that before `heartbeat.session`. I did not establish a fresh live Telegram reproduction, and one commenter reported no recent hits.

## Solution Assessment

Unclear as one single change. The narrow best fix is to make explicit heartbeat session configuration win for event-triggered wakes, while changing Telegram's default DM-to-main behavior needs separate maintainer product and compatibility judgment.
