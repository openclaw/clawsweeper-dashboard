## Label Justifications

- `P1`: The source-backed path can run heartbeat duties inside an active conversation and has caused user-visible delivery failures and costly session contamination.
- `impact:session-state`: Heartbeat work and ordinary direct-message history can be associated with the wrong session and transcript context.
- `impact:message-loss`: A confirming report observed an outbound heartbeat being silently swallowed after the session became polluted.

## Summary

Current main includes later heartbeat isolation and wake-event hardening, but it still gives a valid originating `forcedSessionKey` precedence over explicit `heartbeat.session`; this matches the source-backed exec-completion reproduction, and an open paired PR still proposes the remaining contract change.

## What This Changes

Prevent Telegram DM and system-event-triggered heartbeat traffic from contaminating the wrong agent session.

## Best Possible Solution

Define one generic heartbeat ownership rule in which explicit or isolated heartbeat configuration controls wake-triggered execution, move only the appropriate pending events without loss, and preserve intentional DM-scope behavior with focused regression coverage.

## Reproduction Assessment

Yes at source level: a regular originating `forcedSessionKey` wins before configured heartbeat-session selection, matching the reported exec `notifyOnExit` trigger chain. This read-only review did not run a live Telegram reproduction against current main.

## Solution Assessment

No, forcing all Telegram DMs away from the main session is not the best general solution because it overrides an intentional DM-scope contract. The narrower maintainable fix is to define heartbeat-session precedence and cross-session event ownership generically.
