## Summary

Keep open. Current main still has the confirmed exec/system-event heartbeat routing problem: a wake-supplied non-subagent session key is resolved before an explicit `heartbeat.session`, and that wake path is fed by `notifyOnExit` from the originating session. Current Telegram tests and docs also still describe default direct messages sharing `agent:main:main`, so the requested default DM isolation has not landed. The related open PR #56092 uses closing syntax for this issue, so maintainers should review, land, or supersede that PR instead of closing the report early.

## What This Changes

The issue asks OpenClaw to keep Telegram direct-message sessions and heartbeat/main sessions isolated so DM traffic and system-event-triggered heartbeat work cannot pollute `agent:main:main` or a configured heartbeat session.

## Best Possible Solution

End state should make configured heartbeat sessions win for system-event-triggered wakes, preserve or intentionally migrate pending system events without cross-session pollution, and separately settle Telegram DM default behavior by either documenting intentional main-session sharing or changing Telegram DM isolation with matching tests and docs.

## Reproduction Assessment

Yes for the confirmed exec-completion subcase: `notifyOnExit` queues a system event on the originating session, the wake path forwards that session key, and current heartbeat resolution accepts it before `heartbeat.session`. The original Telegram-only pollution path remains unclear without a fresh minimal live repro.

## Solution Assessment

No for current main. #56092 appears to target the confirmed root cause, but maintainers still need to review whether its event migration and skip-restore behavior are the narrowest durable fix and whether Telegram DM default isolation should be handled separately.
