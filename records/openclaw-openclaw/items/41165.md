## Summary

Keep open: current main still has a source-level path where exec/system-event heartbeat wakes can force the originating session before an explicit `heartbeat.session`, and the remaining narrow fix is already represented by an open closing PR.

## What This Changes

Issue reports Telegram DM and heartbeat/session routing contamination where direct-chat or system-event-triggered heartbeat traffic can reuse the wrong session instead of staying isolated from `agent:main:main` or a configured heartbeat session.

## Best Possible Solution

Make configured `heartbeat.session` win for system-event-triggered wakes, preserve pending system events without cross-session contamination, and handle Telegram default DM sharing through an explicit docs/config/product decision rather than an accidental routing side effect.

## Reproduction Assessment

Yes, source-level. Current main queues exec notify-on-exit events with the originating session key, forwards that key through the wake scheduler, and resolves it before the documented `heartbeat.session` setting.

## Solution Assessment

Yes for keeping this issue open behind the linked implementation candidate. The narrow maintainable fix is heartbeat-session precedence for system-event-triggered wakes; changing Telegram's default DM sharing should remain a separate compatibility decision.
