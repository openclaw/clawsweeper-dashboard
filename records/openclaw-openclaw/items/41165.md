## Summary

Keep open: current main still resolves a forced/originating heartbeat session before an explicit heartbeat session, while Telegram private DMs still follow the documented global DM-scope default unless configuration isolates them. A linked fix PR exists but is open and conflicting, so it is not safe cleanup evidence for closing this issue.

## What This Changes

The issue reports Telegram DM and heartbeat/session routing mixing direct-chat traffic with the main or configured heartbeat session.

## Best Possible Solution

Land a bounded fix that makes configured heartbeat sessions win for system-event-triggered wakes and preserves pending event delivery, then separately document or decide the Telegram DM default isolation policy.

## Reproduction Assessment

Yes for a source-level current-main reproduction: exec completion can carry an originating session into heartbeat wake handling, and `resolveHeartbeatSession` resolves that forced session before `heartbeat.session`. I did not establish a fresh live Telegram reproduction in this read-only review.

## Solution Assessment

No final implementation is settled. A narrow heartbeat-session precedence fix appears maintainable, but changing default Telegram DM isolation remains a compatibility/product decision because current docs and tests still define main-session sharing as the default.
