## Summary

Keep open: current main and v2026.6.8 still let event-triggered heartbeat wakes carry an originating session before explicit heartbeat session config is considered, and Telegram DMs still default to the main session unless session scope is configured otherwise. The linked open fork PR is a candidate fix path, but it is conflicting, so this issue should remain open for maintainer review or replacement of that fix path.

## What This Changes

The issue asks for Telegram DM and heartbeat/main session isolation so user DMs and event-triggered heartbeats cannot pollute each other’s session context.

## Best Possible Solution

Make explicit heartbeat session configuration win for event-triggered wakes with safe pending-event handling, then separately decide whether Telegram’s default DM-to-main behavior should remain documented or change with migration-quality tests and docs.

## Reproduction Assessment

Yes for source-level reproduction: exec/system-event wakes still preserve an originating session key, and current resolveHeartbeatSession resolves that forced session before explicit heartbeat.session. I did not run a fresh live Telegram reproduction.

## Solution Assessment

Unclear as one combined fix: the heartbeat-session precedence repair is narrow, but changing Telegram’s default DM-to-main behavior would alter documented session continuity and needs maintainer product judgment.
