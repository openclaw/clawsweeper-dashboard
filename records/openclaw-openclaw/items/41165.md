## Summary

Keep open. Current main still lets a non-subagent forced heartbeat session resolve before an explicit `heartbeat.session`, Telegram DM tests/docs still preserve default DM-to-main behavior, and an open conflicting PR already targets the system-event heartbeat part of this issue.

## What This Changes

The issue requests stricter Telegram DM and heartbeat session isolation so direct-chat traffic and system-event heartbeats cannot pollute `agent:main:main`.

## Best Possible Solution

Land or replace the linked heartbeat routing fix so explicit `heartbeat.session` wins for system-event wakes, then separately decide whether Telegram default `dmScope: main` should remain documented behavior or change to isolated DMs with tests and docs.

## Reproduction Assessment

Yes at source level, but not by a live current-main run in this review. The current code path threads an originating exec/DM session into heartbeat wake handling, and `resolveHeartbeatSession` consumes that forced session before explicit `heartbeat.session`.

## Solution Assessment

No, closing now is not the best way to handle this issue. The best path is to review and land a conflict-free heartbeat-session fix, while treating Telegram default DM isolation as a separate product/docs decision if maintainers want to change the documented default.
