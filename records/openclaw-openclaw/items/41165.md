## Summary

Keep open: current main still lets a forced wake session resolve before configured `heartbeat.session`, and the linked open PR with closing syntax is the right implementation candidate rather than closing this issue early.

## What This Changes

The issue requests stricter isolation so Telegram DM traffic and system-event-triggered heartbeat runs do not pollute `agent:main:main` or the configured heartbeat session.

## Best Possible Solution

Land a reviewed fix where configured `heartbeat.session` wins for system-event-triggered wakes without losing queued system events, then separately settle or document Telegram DM default-to-main behavior with compatibility tests.

## Reproduction Assessment

Yes, source-level. Current main forwards exec/system-event wake session keys into heartbeat resolution, and `resolveHeartbeatSession` consumes that forced key before the documented `heartbeat.session` setting.

## Solution Assessment

Yes for keeping this issue open behind the linked implementation candidate. The narrow maintainable fix is heartbeat-session precedence for event wakes; changing Telegram's default DM sharing remains a separate product/compatibility decision.
