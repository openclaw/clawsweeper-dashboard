## Summary

Keep open: current main still allows event-triggered heartbeat wakes to carry an originating session that is resolved before explicit `heartbeat.session`, and Telegram DMs still use the documented main-session default unless DM scope is configured otherwise. The open linked fork PR is the candidate implementation path, but it is conflicting, so maintainers should review, refresh, or replace that PR rather than close this issue now.

## What This Changes

Telegram DM traffic and event-triggered heartbeats should stay isolated from the agent main/heartbeat session instead of polluting `agent:main:main` or a configured heartbeat session.

## Best Possible Solution

Make explicit heartbeat session configuration win for system-event-triggered wakes with safe pending-event handling, then separately settle whether Telegram DMs should keep the documented main-session default or change with tests, docs, and migration-quality review.

## Reproduction Assessment

Yes for source-level reproduction: exec/system-event wakes still preserve an originating session key, the wake queue forwards it, and current heartbeat resolution considers that forced session before configured `heartbeat.session`. I did not run a fresh live Telegram reproduction in this read-only review.

## Solution Assessment

No single combined fix is clearly best yet: the heartbeat precedence repair is narrow, but changing Telegram's documented default DM-to-main behavior needs maintainer product judgment. The best next step is to review or refresh the linked implementation path and split the product question if needed.
