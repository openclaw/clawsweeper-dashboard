## Summary

Keep open: current main and v2026.6.10 still resolve event-triggered heartbeat forced sessions before `heartbeat.session`, and Telegram DMs still default to the agent main session unless configured otherwise. The linked fork PR at https://github.com/BingqingLyu/openclaw/pull/1542 is open but conflicting/unmerged, so it is an implementation candidate rather than a reason to close this issue.

## What This Changes

The issue asks OpenClaw to prevent Telegram DM traffic and heartbeat/system-event runs from reusing or polluting the same main or heartbeat session context.

## Best Possible Solution

Make explicit `heartbeat.session` win for system-event-triggered wakes with safe system-event handling, then separately decide whether Telegram DMs should keep the documented main-session default or move to isolated sessions with tests and docs.

## Reproduction Assessment

Yes, at source level: exec/system-event wakes preserve an originating session key, the wake queue forwards it, and current heartbeat resolution considers that forced session before configured `heartbeat.session`. I did not establish a fresh live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear as one combined change: the heartbeat precedence repair is narrow, but changing Telegram's documented default DM-to-main behavior needs maintainer product judgment. The maintainable path is to fix or replace the heartbeat implementation candidate and split the Telegram policy decision if needed.
