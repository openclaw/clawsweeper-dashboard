## Summary

Keep open: current main and the latest release still leave the managed browser stuck when an untracked or unhealthy Chrome process holds the CDP port, and the closed related PR is unsafe and obsolete rather than a fix path.

## What This Changes

This issue asks the browser plugin to recover automatically when an OpenClaw-managed Chrome process becomes unresponsive while still holding the local CDP port.

## Best Possible Solution

Implement browser-plugin recovery only for local managed profiles: probe CDP health, verify ownership through port/process/user-data facts, terminate only stale owned Chrome, wait for the port to clear, and preserve remote plus attachOnly behavior.

## Reproduction Assessment

Yes, at source level. A local managed loopback profile with no tracked RunningChrome still reaches the occupied-port launch failure, and an HTTP-only/CDP-dead listener still surfaces as unowned instead of being safely recovered.

## Solution Assessment

Yes, if the repair stays narrow and ownership-checked in the browser plugin. The closed broad process-kill PR is not the best path because it was unsafe and targeted obsolete structure.
