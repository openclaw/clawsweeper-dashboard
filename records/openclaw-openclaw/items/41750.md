## Summary

Keep open: current main still lacks safe automatic recovery for an untracked unhealthy OpenClaw-managed Chrome process holding the browser CDP port, and the related attempted fix PR is closed unmerged.

## What This Changes

The issue requests automatic recovery when an OpenClaw-managed Chrome process becomes unhealthy while still holding the browser tool CDP port.

## Best Possible Solution

Add browser-plugin-owned stale managed Chrome recovery that probes CDP health, verifies OpenClaw-managed Chromium ownership, terminates with escalation only after verification, waits for the port to clear, and preserves healthy reuse plus attachOnly/remote behavior.

Yes, at source/test level: simulate an occupied loopback CDP endpoint with no tracked RunningChrome and failed readiness, and current main reaches the existing refusal or PortInUse path. I did not run a real long-lived zombie Chrome process in this read-only review.

## Solution Assessment

Yes: the narrow probe-and-recover direction is the maintainable fix, provided it is constrained to verified OpenClaw-managed Chromium processes. A generic kill-whatever-owns-the-port cleanup would be unsafe.
