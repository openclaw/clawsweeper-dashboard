## Summary

Keep open: current main still lacks safe recovery for an untracked unhealthy managed Chrome process or CDP listener on the browser port; the source path still rejects/fails, and the linked attempted fix PR is closed unmerged.

## What This Changes

The issue asks the browser plugin to detect an unhealthy OpenClaw-managed Chrome/CDP port holder and restart it automatically instead of timing out across gateway restarts.

## Best Possible Solution

Add browser-plugin-owned CDP health recovery that reuses healthy listeners, verifies OpenClaw-managed Chromium ownership for stale listeners, terminates only verified owned processes with graceful escalation, waits for the port to clear, and preserves attachOnly/remote refusal behavior.

## Reproduction Assessment

Yes at source level: simulate an occupied loopback CDP endpoint with no tracked RunningChrome and failed CDP readiness, or an occupied port before launch, and current main reaches the refusal/PortInUse path. I did not run a live zombie Chrome process in this read-only review.

## Solution Assessment

Yes: a narrow probe-and-recover path inside the browser plugin is the best maintainable direction if termination is constrained to verified OpenClaw-managed Chromium processes. A generic kill-whatever-owns-the-port cleanup would be unsafe.
