## Summary

Keep open: current `main` and `v2026.5.7` still fail the reported stale managed-Chrome case, and the direct fix attempt was closed unmerged after review found it unsafe. The code still either aborts on an occupied CDP port before recovery or refuses an untracked unhealthy loopback CDP listener instead of verifying OpenClaw ownership and restarting it.

## What This Changes

The issue asks the browser tool to detect and recover an unresponsive OpenClaw-managed Chrome process that still holds the loopback CDP port.

## Best Possible Solution

Add browser-plugin-owned stale managed Chrome recovery with a short CDP health probe, strict managed-process ownership checks, graceful termination with escalation, port-clear waiting, and unchanged remote/attachOnly behavior.

## Reproduction Assessment

Yes, at source level: simulate an occupied loopback CDP endpoint with no tracked `RunningChrome` and failed CDP readiness, and current `main` reaches the launch `PortInUseError` or the untracked local-port refusal branch. I did not run a live zombie Chrome process in this read-only review.

## Solution Assessment

Yes, if implemented narrowly: the maintainable fix is owner-checked stale managed Chrome recovery in the browser plugin. A broad `lsof`/port-holder kill is not the best solution because it can terminate non-OpenClaw CDP services.
