## Summary

Keep open: current main and the latest v2026.5.3 release still do not automatically recover an untracked unhealthy managed Chrome process holding the browser CDP port, and the directly related fix PR was closed unmerged.

## What This Changes

The issue requests browser-plugin recovery for stale or zombie OpenClaw-managed Chrome processes that keep the CDP port bound but stop responding.

## Best Possible Solution

Add browser-plugin-owned stale managed Chrome recovery that probes CDP health, verifies Chromium ownership by port plus expected OpenClaw user-data-dir, terminates gracefully with escalation, waits for the port to clear, and preserves attachOnly/remote behavior.

## Reproduction Assessment

Yes, at source level: simulate an occupied loopback CDP endpoint with no tracked RunningChrome and failed readiness, and current main reaches the existing refusal or PortInUse path. I did not run a live zombie Chrome in this read-only review.

## Solution Assessment

Yes: the narrow probe-and-recover direction is the best fix if it is guarded by OpenClaw-managed ownership checks. PR #41758 is useful context but is not sufficient as-is because its diff killed generic port owners without enough ownership proof.
