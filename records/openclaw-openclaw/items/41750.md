## Summary

Keep open: current main still refuses or fails the untracked unhealthy managed-Chrome CDP listener path instead of safely proving ownership, terminating the stale process, and relaunching.

## What This Changes

The issue requests automatic recovery when an OpenClaw-managed Chrome process becomes unresponsive while still holding the browser CDP port.

## Best Possible Solution

Add browser-plugin-owned stale managed-Chrome recovery that probes CDP health, verifies the listener is the expected OpenClaw-managed Chromium process, terminates with escalation only after verification, waits for the port to clear, and preserves attachOnly/remote behavior.

## Reproduction Assessment

Yes, at source level: simulate a loopback CDP endpoint where HTTP responds but CDP readiness fails with no tracked `RunningChrome`, and current main reaches the existing refusal path. I did not run a live long-lived zombie Chrome process in this read-only review.

## Solution Assessment

Yes: the narrow probe-and-recover direction is the maintainable fix if it is constrained to verified OpenClaw-managed Chromium processes. A generic kill-whatever-owns-the-port cleanup is not safe.
