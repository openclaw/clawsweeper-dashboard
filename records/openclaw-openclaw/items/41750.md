## Summary

Keep open: current main and the latest release still route an unresponsive local managed Chrome holding the CDP port into launch failure or an untracked-port refusal, and the only direct fix PR was closed unmerged with safety defects.

## What This Changes

The issue asks OpenClaw to detect and recover from an owned Chrome process that becomes unresponsive while still holding the browser tool CDP port.

## Best Possible Solution

Implement narrow browser-plugin recovery for local managed OpenClaw profiles: probe CDP health, prove OpenClaw ownership of the stale process/user data dir, terminate only that owned process, wait for the port to clear, relaunch, and preserve remote plus attachOnly behavior.

## Reproduction Assessment

Yes, at source level. An occupied or unresponsive loopback CDP port with no tracked RunningChrome reaches launchOpenClawChrome's PortInUseError path, and an HTTP-only unhealthy endpoint reaches the untracked local-port refusal branch.

## Solution Assessment

Yes, if the fix stays narrow and ownership-checked in the browser plugin. The closed broad process-kill PR is not the best solution because it targeted an obsolete path and introduced unsafe process termination and shell-command handling.
