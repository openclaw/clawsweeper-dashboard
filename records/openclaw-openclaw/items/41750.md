## Summary

Keep open: current main still fails the managed-browser recovery path when a stale or unresponsive process holds the local CDP port, and the only direct fix PR was closed unmerged with concrete safety defects.

## What This Changes

The issue asks OpenClaw to detect an unhealthy managed Chrome process holding the browser CDP port, terminate only the stale owned process, and relaunch browser control automatically.

## Best Possible Solution

Implement narrow browser-plugin recovery for local managed OpenClaw profiles: probe CDP health, verify OpenClaw ownership of the stale process/user data dir, terminate only that owned process, wait for the port to clear, relaunch, and preserve remote plus attachOnly behavior.

## Reproduction Assessment

Yes, at source level. An occupied or unresponsive loopback CDP port with no tracked RunningChrome reaches launchOpenClawChrome's PortInUseError path, and an HTTP-only unhealthy endpoint reaches the untracked local-port refusal branch.

## Solution Assessment

Yes, if the fix is narrow and ownership-checked in the browser plugin. The closed broad process-kill PR is not the best solution because it targeted an obsolete path and introduced unsafe process termination and shell-command handling.
