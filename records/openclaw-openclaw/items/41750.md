## Summary

## What This Changes

The issue requests automatic recovery when an OpenClaw-managed Chrome process becomes unresponsive while still holding the browser CDP port, causing persistent browser tool timeouts across gateway restarts.

## Best Possible Solution

Land one narrow loopback-only recovery path that proves the blocker is the same OpenClaw-managed browser profile and process before signaling or cleaning up state, and otherwise preserve the current safe failure behavior.

## Reproduction Assessment

Yes, at source level: a listener holding the managed CDP port makes `ensurePortAvailable(profile.cdpPort)` reject before any CDP health or managed-process ownership recovery can run. I did not run a live zombie-Chrome reproduction in this read-only review.

## Solution Assessment

Yes, the best fix direction is a guarded owned-process recovery path like the linked implementation, not a broad kill-by-port workaround. The issue should remain open until that PR lands or is replaced.
