## Summary

Keep open: current main and v2026.5.19 still route an unresponsive local managed Chrome holding the CDP port into launch failure or local-port ownership errors, and the only direct fix PR was closed unmerged with safety defects.

## What This Changes

The issue requests automatic recovery when a managed Chrome process remains alive but unresponsive while holding the browser CDP port.

## Best Possible Solution

Implement browser-plugin recovery only for local managed OpenClaw profiles: probe CDP health, prove process ownership, terminate only the stale owned Chrome, wait for the port to clear, then relaunch while preserving remote, attachOnly, and existing-session behavior.

## Reproduction Assessment

Yes, at source level. An occupied or unresponsive loopback CDP port with no tracked RunningChrome still reaches launchOpenClawChrome's PortInUseError path or the untracked local-port refusal branch on current main.

## Solution Assessment

Yes, if the fix stays narrow and ownership-checked in the browser plugin. The closed broad process-kill PR is not the best solution because it targeted an obsolete path and introduced unsafe termination and shell-command handling.
