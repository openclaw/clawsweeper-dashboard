## Summary

Keep open: current main and the latest release still do not recover an untracked unhealthy OpenClaw-managed Chrome process holding the loopback browser CDP port, and the only direct fix PR is closed unmerged with safety blockers.

## What This Changes

Recover an OpenClaw-managed Chrome process that is alive but unresponsive on the browser CDP port so browser tool calls stop timing out after gateway restarts.

## Best Possible Solution

Add browser-plugin-owned stale managed Chrome recovery with a short CDP health probe, strict user-data-dir and remote-debugging-port ownership checks, graceful termination with escalation, port-clear waiting, and unchanged remote/attachOnly behavior.

## Solution Assessment

Yes: the narrow owner-checked probe-and-recover path is the maintainable fix. A broad port-kill workaround is not safe enough because it can terminate non-OpenClaw CDP services.
