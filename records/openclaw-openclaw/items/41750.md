## Label Justifications

- `P1`: A stale managed Chrome listener can disable all browser calls, repeatedly time out, and remain broken across gateway restarts.
- `impact:crash-loop`: The issue concerns persistent process-level browser unavailability that currently requires manual process termination.

## Summary

The central failure remains in current main and v2026.6.5: managed Chrome launch rejects an occupied CDP port before checking whether the listener is a stale OpenClaw-owned browser. Keep the issue open while the linked closing PR is rebased and reviewed; it has not merged and GitHub currently reports conflicts.

## What This Changes

Detect an unresponsive OpenClaw-managed Chrome process holding the configured CDP port, safely terminate it after proving ownership, and relaunch the browser automatically.

## Best Possible Solution

Land one narrow loopback-only recovery path that waits for CDP readiness, proves the blocker is the same OpenClaw-managed profile and process, revalidates identity before every destructive action, and otherwise preserves the current safe failure behavior.

## Reproduction Assessment

Yes, at source level. On current main, a listener holding the managed CDP port makes `ensurePortAvailable` reject before any CDP health or managed-process ownership recovery can run; the issue logs and linked PR proof supply the matching real-world state.

## Solution Assessment

Yes, the linked PR’s direction is the best fix shape: recover only after strong ownership proof rather than killing whichever process holds the port. It still needs rebase and fresh maintainer review because process termination is availability-sensitive.
