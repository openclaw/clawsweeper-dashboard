## Summary

Current main still leaves the managed-browser recovery bug open. The browser plugin can reuse a healthy pre-existing loopback CDP endpoint, restart a tracked child Chrome process, and clear stale Chromium Singleton lock files, but it still does not owner-check and terminate an untracked unhealthy OpenClaw-managed Chrome process holding the CDP port. The ClawAid/manual cleanup comments are useful operational workarounds, and related PR #41758 was an attempted fix, but that PR is closed unmerged and no replacement is currently open.

## What This Changes

The issue requests automatic recovery when an OpenClaw-managed Chrome process becomes unhealthy while still holding the browser tool CDP port.

## Best Possible Solution

Implement browser-plugin-owned stale managed Chrome recovery: before managed launch, probe the loopback CDP endpoint with a short timeout; if the port is occupied but unhealthy, inspect the listener, verify it is a Chrome/Chromium-family process with the expected --remote-debugging-port and OpenClaw-managed --user-data-dir for that profile, terminate it gracefully with forced escalation if needed, wait for the port to clear, and relaunch. Preserve healthy pre-existing browser reuse and attachOnly/remote external-CDP refusal behavior.

## Reproduction Assessment

Yes for the current failure path at source/test level: an occupied loopback CDP port with no tracked RunningChrome either reaches launchOpenClawChrome where ensurePortAvailable throws, or reaches the availability path that throws 'not by openclaw' without cleanup when HTTP responds but CDP readiness fails. I did not run a real long-lived zombie Chrome process in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is the issue's probe-and-recover direction, but only with strict ownership verification inside the browser plugin. A generic kill-anything-on-the-CDP-port approach would not be acceptable because it could terminate external CDP services or unrelated local listeners.
