## Summary

Keep open: current main still has the same local managed-browser gap, where an unhealthy or untracked process can hold the CDP port and block recovery before OpenClaw proves ownership and restarts it safely.

## What This Changes

The issue requests automatic recovery when the managed browser tool's Chrome process becomes unresponsive while still holding the CDP port.

## Best Possible Solution

Implement browser-plugin recovery only for local managed profiles: probe CDP health, verify ownership through port/process/user-data facts, terminate only stale owned Chrome, wait for the port to clear, and preserve remote plus attachOnly behavior.

## Reproduction Assessment

Yes, at source level. An unhealthy loopback CDP port with no tracked RunningChrome still flows into launchOpenClawChrome's port-in-use failure, and a HTTP-only/CDP-dead listener still surfaces as unowned instead of being safely recovered.

## Solution Assessment

Yes, if the repair stays narrow and ownership-checked in the browser plugin. The closed broad process-kill PR is not the best path because it was unsafe and targeted obsolete structure.
