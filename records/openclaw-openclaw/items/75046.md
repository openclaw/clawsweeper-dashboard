## Summary

Keep open. Current main does not include Cursor SDK support, and this is an active feature PR, but the patch still has blocking auth/error-handling issues, insufficient OpenClaw-path proof, and dependency/native build review outstanding.

## What This Changes

Adds a bundled `cursor-sdk` plugin registering a Cursor SDK agent harness/provider, live/unit tests, `@cursor/sdk`, lockfile changes, a scoped `undici` override, and `sqlite3` build permission.

## Best Possible Solution

Keep the integration plugin-shaped only if maintainers accept bundling it; otherwise publish it through ClawHub, and before merge fix credential resolution, guarded SDK startup, and real OpenClaw-path proof.

## Reproduction Assessment

Yes, source-reproducible for the blocking PR behavior: current main skips generic auth bootstrap for plugin-owned harnesses, while the PR runner requires `params.resolvedApiKey` before calling the SDK. I did not run a live OpenClaw Cursor turn.

## Solution Assessment

No. A self-contained plugin is the right technical boundary if maintainers accept the bundle, but this implementation still needs plugin-safe credential resolution, guarded SDK startup, OpenClaw-path proof, and supply-chain review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve Cursor credentials before checking resolvedApiKey:** `extensions/cursor-sdk/src/run-attempt.ts:62-66`
  - body: For plugin-owned harnesses, current main skips PI auth bootstrap and only sets `params.resolvedApiKey` from `apiKeyInfo`; the auth plan only forwards a harness auth provider for Codex. A normal `cursor-sdk` run with `CURSOR_API_KEY` or a profile can therefore hit this missing-key branch before the SDK call. Resolve the plugin credential from the forwarded auth/profile/env contract first.
  - confidence: 0.91
- **[P2] Move SDK startup inside the guarded lifecycle:** `extensions/cursor-sdk/src/run-attempt.ts:115`
  - body: `Agent.create()` runs before the `try` block. If Cursor rejects during startup, auth validation, or network setup, the harness throws out to the selection layer instead of returning a classified prompt error result, bypassing the normal OpenClaw harness failure path.
  - confidence: 0.86
