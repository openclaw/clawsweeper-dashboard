## Summary

Keep this PR open. Current main does not implement the Cursor SDK local/cloud harness requested here, and the existing Cursor ACP path is a separate CLI-based runtime. The PR is directionally aligned with the plugin/harness boundary, but the latest head still leaves Cursor send/wait outside a reliable abort/timeout race and introduces a trusted runtime plus dependency/override footprint that needs maintainer security review before merge.

## What This Changes

The PR adds a bundled `extensions/cursor-sdk` plugin that registers a Cursor SDK provider and AgentHarness, stages `@cursor/sdk`, adds unit tests, updates the lockfile, and adds a workspace `undici` override.

## Best Possible Solution

If maintainers want this bundled, land it as a trusted provider-plus-harness plugin only after lifecycle cancellation is fixed, Cursor-owned tool/sandbox/approval boundaries are documented, dependency policy is reviewed, and tests cover send, stream, wait, timeout, and upstream abort behavior. If that product/security bar is not met, keep the implementation external through ClawHub/npm while preserving the existing plugin API.

## Reproduction Assessment

Yes for the review findings. Static tracing of the PR head plus the `@cursor/sdk@1.0.7` README/types shows first runtime initialization happens in `agent.send()`, while the PR only cancels after a `run` exists and removes cancellation before `run.wait()`.

## Solution Assessment

No as-is. The plugin/provider shape is a reasonable direction, but the implementation is not yet the narrowest maintainable solution because it does not fully preserve OpenClaw timeout/abort semantics and has unresolved trusted-runtime security policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Race first send against abort and timeout:** `extensions/cursor-sdk/src/run-attempt.ts:124`
  - body: `Agent.create()` defers runtime startup until the first `send`, but this awaits `agent.send(params.prompt)` without racing it against `runAbortController.signal` or `params.timeoutMs`. If Cursor local startup or cloud run creation stalls, the timer only marks the controller aborted and cannot cancel anything because no `run` exists yet, so failover can hang past the configured timeout.
  - confidence: 0.9
- **[P2] Keep cancellation active until wait completes:** `extensions/cursor-sdk/src/run-attempt.ts:141-159`
  - body: The abort listener that calls `run.cancel()` is removed immediately after `run.stream()` finishes, before `run.wait()` runs. If the final wait stalls or the upstream abort arrives in that phase, the timeout/abort signal no longer cancels the Cursor run and this attempt can block instead of returning a timed-out or aborted result.
  - confidence: 0.88
