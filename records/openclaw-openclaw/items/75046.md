## What This Changes

This PR adds a bundled `cursor-sdk` plugin with provider/auth metadata, an AgentHarness runner backed by `@cursor/sdk`, unit/live tests, lockfile changes, an `undici` override, and `sqlite3` build permission.

## Best Possible Solution

Keep the Cursor integration plugin-shaped only if maintainers accept bundling it; resolve credentials through the plugin-safe auth/profile contract, handle SDK lifecycle errors inside the harness result contract, and prove a real OpenClaw `cursor-sdk` turn before merge.

## Reproduction Assessment

Yes, source-reproducible: current main skips generic auth bootstrap for plugin-owned harnesses, while the PR runner only reads `params.resolvedApiKey`. I did not run a live OpenClaw Cursor turn.

## Solution Assessment

No. A self-contained plugin is the right boundary, but this implementation still needs plugin-safe auth resolution, guarded SDK startup/error handling, OpenClaw-path proof, and maintainer security approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve Cursor credentials before checking resolvedApiKey:** `extensions/cursor-sdk/src/run-attempt.ts:62-66`
  - body: Current main skips `initializeAuthProfile()` for plugin-owned harnesses and only sets `params.resolvedApiKey` from `apiKeyInfo`. This runner immediately fails when that field is unset, so normal `CURSOR_API_KEY` or profile setup for `cursor-sdk` can hit the missing-key branch before the SDK is called. Resolve the forwarded auth/profile/env credential through the plugin-safe contract first.
  - confidence: 0.92
- **[P2] Catch SDK creation failures inside the harness result:** `extensions/cursor-sdk/src/run-attempt.ts:115`
  - body: `Agent.create()` runs before the `try` block. If Cursor rejects during startup, auth validation, or network setup, the plugin harness throws out to current harness selection, which rethrows instead of returning an attempt result; that bypasses error classification and normal prompt failure handling. Move creation into the guarded lifecycle and only dispose the agent after it exists.
  - confidence: 0.86
