## Summary

Keep this PR open: the plugin boundary is plausible, but the current implementation appears unable to receive Cursor auth through OpenClaw's plugin-harness path, and the supplied proof bypasses that path.

## What This Changes

This PR adds a bundled `extensions/cursor-sdk` agent harness/provider plugin, `@cursor/sdk` runtime dependency, Cursor SDK tests/live test, and workspace dependency policy changes for `undici` and `sqlite3`.

## Best Possible Solution

Keep the self-contained plugin approach, but wire Cursor auth through the plugin-harness/runtime auth contract, add an OpenClaw-level proof that exercises the selected harness, and get maintainer approval for the native dependency build surface.

## Reproduction Assessment

Yes, source-reproducible: current main skips generic auth initialization for plugin-owned transports, while the PR harness only accepts `params.resolvedApiKey`. I did not run a live Cursor turn through OpenClaw.

## Solution Assessment

No. A bundled plugin is the right boundary, but this implementation needs to resolve/receive auth through the actual OpenClaw harness path before it can be considered the best fix.
