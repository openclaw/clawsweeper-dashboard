## Summary

Keep this PR open: the plugin boundary is appropriate, but the latest head still appears to fail the real OpenClaw auth path before Cursor runs, and the supplied proof exercises the SDK directly rather than OpenClaw selecting the new harness.

## What This Changes

This PR adds a bundled `cursor-sdk` provider/agent-harness plugin with `@cursor/sdk`, Cursor unit/live tests, lockfile updates, an `undici` override, and `sqlite3` build permission.

## Best Possible Solution

Keep the self-contained plugin approach, but resolve Cursor credentials through the plugin-safe OpenClaw harness auth contract and prove an actual OpenClaw `cursor-sdk` run before merge.

## Reproduction Assessment

Yes, source-reproducible: current main skips generic `apiKeyInfo` resolution for plugin-owned harnesses, while the PR returns a missing-key error unless `params.resolvedApiKey` is already set. I did not run a live OpenClaw Cursor turn.

## Solution Assessment

No. A bundled plugin is the right boundary for an agent harness, but this implementation needs to consume the forwarded auth profile/runtime auth contract rather than only `resolvedApiKey`.
