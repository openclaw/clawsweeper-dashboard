## Summary

Keep open. The plugin boundary is plausible, but the current patch still appears to miss the normal OpenClaw auth path, lacks after-fix proof through OpenClaw itself, and adds supply-chain-sensitive dependency/build surface that needs maintainer review.

## What This Changes

The PR adds a bundled `cursor-sdk` provider and AgentHarness plugin with plugin config, auth metadata, Cursor SDK runner/tests, lockfile updates, an `undici` override, and `sqlite3` build permission.

## Best Possible Solution

Land only a maintainer-approved plugin-scoped Cursor SDK harness that consumes OpenClaw’s forwarded auth contract, honors attempt timeout boundaries, and includes redacted real OpenClaw proof; otherwise redirect the integration to ClawHub/npm distribution.

## Reproduction Assessment

Yes, source-reproducible: current main skips generic auth bootstrap for plugin harnesses, while the PR reads only `params.resolvedApiKey`, which is derived from `apiKeyInfo`. I did not run a live OpenClaw Cursor turn.

## Solution Assessment

No. The AgentHarness plugin seam is the right technical boundary, but this implementation needs forwarded auth/profile handling, first-send timeout enforcement, real OpenClaw behavior proof, and maintainer security approval before it is a mergeable solution.

  - body: Current main skips generic auth initialization for plugin-owned harnesses, so `params.resolvedApiKey` is normally unset on this path.   - body: The timer only aborts a local controller until `agent.send()` returns a `Run`, but the SDK send options have no AbortSignal and `run.cancel()` is unavailable before that return. If the first send/runtime initialization stalls, this attempt can block past `params.timeoutMs`; race `agent.send()` against the timeout or otherwise fail the OpenClaw attempt on time.
  - confidence: 0.82
