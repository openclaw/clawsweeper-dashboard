## Summary

Keep open: the PR addresses the visible user-input injection, but it routes per-turn MEMORY/workspace context through Codex collaboration-mode developer instructions, which upstream treats as persistent thread settings and developer-role context. It needs a safer transient-context approach plus real multi-turn app-server proof before merge.

## What This Changes

This PR removes OpenClaw runtime context from Codex turn user input, forwards it through turn collaboration developer instructions, and updates Codex prompt snapshots and tests.

## Best Possible Solution

Keep only the actual request in `turn/start.input` while delivering OpenClaw runtime context through a non-persistent, non-authoritative turn reference channel or by sanitizing replayed decorated user items, with multi-turn and compaction coverage plus real app-server proof.

## Reproduction Assessment

Yes. Current main source inspection plus the linked issue's structured logs show the decorated `OpenClaw runtime context for this turn:` block being stored as normal `role=user` Codex history; I did not run a live multi-turn session in this read-only review.

## Solution Assessment

No. Keeping the real prompt out of `turn/start.input` is the right direction, but `collaborationMode.settings.developer_instructions` is not a safe transient reference channel under the upstream contract.
