## Summary

Keep open: current main still appears to reject schema-expected array/object parameters when the model emits them as JSON strings, and the related older fix attempts are closed unmerged rather than canonical fixes.

## What This Changes

The issue reports MCP tool calls failing when providers return array/object arguments as JSON strings instead of native JSON values.

## Best Possible Solution

Add schema-aware JSON parsing in the shared tool-argument validator for string values only when an object or array branch is expected, with focused regression tests for MCP-style nullable and union schemas.

## Reproduction Assessment

Yes from source inspection, but not from an executed live repro: a string under an array/object schema is never JSON-parsed before validator failure.

Yes; a narrow shared validator fix is the best fit because the issue is not specific to one MCP server or one provider and older provider-specific attempts would leave sibling paths drifting.

## Repair Work Prompt

Fix the MCP/tool-argument validation bug where model-emitted string values that encode JSON arrays or objects are rejected when the tool JSON schema expects an array/object, including nullable or anyOf/oneOf branches. Keep the change in the shared validation layer unless source inspection proves a narrower owner boundary is required. Add regression tests in `packages/llm-core/src/validation.test.ts` for an array property such as `tags: '["test","debug"]'`, an object property such as `params: '{}'`, and a nullable/union schema like `anyOf: [{type:'array', items:{type:'string'}}, {type:'null'}]`. Do not add new config flags or provider-specific policy. Validate with `node scripts/run-vitest.mjs packages/llm-core/src/validation.test.ts`. Reference https://github.com/openclaw/openclaw/issues/96916 and note that older related PRs https://github.com/openclaw/openclaw/pull/70882 and https://github.com/openclaw/openclaw/pull/57455 were closed unmerged.
