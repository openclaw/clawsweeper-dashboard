## Summary

Keep open: current main still has the nested approval-metadata bug, but this PR is not merge-ready because it broadens approval wrapper parsing on a path that also consumes generic command output and lacks after-fix real behavior proof.

## What This Changes

The PR replaces regex-based exec approval result parsing with balanced metadata parsing, changes one gateway denial reason literal to colon-separated text, strips parentheses from node exec denial reasons, adds focused tests, and adds a changelog entry.

## Best Possible Solution

Land the nested-metadata parsing fix with approval-result parsing limited to structured or trusted lifecycle data, plus redacted runtime proof that the denied followup branch is used.

## Reproduction Assessment

Yes, at source level: current main emits nested `approval-timeout (allowlist-miss)` metadata and the current `[^)]*` parser rejects that denied payload. I did not establish a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No. Balanced parsing and colon-separated producer text are useful, but the PR should first gate the broader parser to trusted approval lifecycle data or structured fields instead of generic command output.
