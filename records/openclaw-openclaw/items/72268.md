## Summary

Keep open. Current main still has the nested exec-approval metadata bug, and this PR targets a real regression, but the diff is not merge-ready because it broadens approval-wrapper parsing while the parser is still consumed from generic command output and it still carries unrelated generated SDK hash churn.

## What This Changes

The PR replaces exec approval denied/finished regex parsing with a balanced-parentheses metadata parser, adds nested metadata parser and followup tests, and modifies the generated plugin SDK API hash file.

## Best Possible Solution

Land a narrowed version that keeps the balanced metadata parser and nested denied/finished regression coverage, constrains approval-resolution parsing to trusted structured exec lifecycle data or a trusted source guard, removes the generated SDK hash change, and adds the required changelog entry for the user-facing followup fix.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: the existing denied/finished regexes reject nested metadata, while the gateway emits `approval-timeout (allowlist-miss)` and followup routing depends on the denied parser result.

## Solution Assessment

No, not as currently proposed. The balanced-parser direction is the right narrow fix for nested metadata, but the merge-ready solution should first address the trusted-source boundary and remove unrelated generated baseline churn.
