## Summary

Keep open: the reported parser bug is real, but the latest head still relies on a guessable metadata-prefix check while parsing raw exec output, and the supplied proof is diff inspection rather than an after-fix runtime or Telegram followup run.

## What This Changes

The PR replaces exec approval result regex parsing with balanced metadata parsing, adjusts denial metadata producers, adds parser/followup tests, and updates the changelog.

## Best Possible Solution

Feed approval resolution from structured or otherwise trusted exec lifecycle data instead of generic command output, while keeping the balanced metadata parser and proving the visible followup path with redacted runtime evidence.

## Reproduction Assessment

Yes, at source level: current main emits `approval-timeout (allowlist-miss)` and the current `[^)]*` parser rejects that nested metadata. I did not run a live gateway flow in this read-only review.

## Solution Assessment

No. Balanced parsing fixes the nested metadata symptom, but the current PR should not treat a guessable `gateway id=` or `node=` prefix as provenance while approval parsing still runs over raw exec output.
