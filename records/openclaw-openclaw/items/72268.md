## Summary

Keep open. The nested-metadata parser bug is real on current main, but the latest PR still widens approval-wrapper parsing while generic exec output is parsed into approval resolution events, so the security trust-boundary blocker remains before merge.

## What This Changes

The PR replaces denied/finished exec approval regex parsing with balanced-parentheses parsing, adds nested-metadata parser and followup tests, and records a changelog fix.

## Best Possible Solution

Land balanced metadata parsing together with a trust-boundary fix that emits approval resolution only from structured or otherwise trusted exec lifecycle data.

## Reproduction Assessment

Yes. A source-aligned Node regex check against current main shows nested `Exec denied` and `Exec finished` metadata fail the existing parser, and the gateway emits the nested `approval-timeout (allowlist-miss)` denial reason.

## Solution Assessment

No. The balanced parser is the right narrow parser change for nested metadata, but the PR is not merge-ready until approval resolution is scoped away from arbitrary tool output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope approval parsing to trusted exec results:** `src/agents/exec-approval-result.ts:88`
  - body: `parseExecApprovalResultText()` is still applied to generic exec output before emitting approval events. With the broader balanced parser, a command that prints `Exec denied (... (nested) ...): ...` can be surfaced as a resolved approval denial, so gate this parsing to structured or trusted exec lifecycle data before widening the wrapper grammar.
  - confidence: 0.86

## Security Review

Status: needs_attention

Concerns:
