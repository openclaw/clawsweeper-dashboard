## Summary

Keep open. The PR addresses the nested-metadata parser bug in the right area, but it still lacks external real behavior proof, leaves the trusted-output parsing concern unresolved, and the latest force-push changes node denial reason formatting without updating the existing Gateway test expectation.

## What This Changes

The PR replaces exec approval result regex parsing with balanced metadata parsing, adds nested denial/finished coverage, changes one gateway denial reason to colon-separated text, strips parentheses from node denial reasons, and adds a changelog entry.

## Best Possible Solution

Land the nested/colon metadata handling with a trusted-source approval-result gate, update the affected Gateway node-event test coverage, and require redacted terminal/log output or a short Telegram proof before merge.

## Reproduction Assessment

Yes, at source level: current main emits a nested `approval-timeout (allowlist-miss)` denial string and the current regex probe rejects nested denied/finished metadata. I did not run a live Telegram/Gateway scenario in this read-only review.

## Solution Assessment

No. Balanced parsing is the right parser direction, but the PR should also gate approval-result parsing to trusted lifecycle data, update the node-event sanitizer test, and add real behavior proof before merge.
