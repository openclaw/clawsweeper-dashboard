## Summary

Keep open: the parser fix addresses a real source-reproducible nested-metadata bug, but the PR still broadens approval-wrapper parsing while an existing caller applies that parser to generic exec output. The same trust-boundary concern is already flagged by the prior ClawSweeper and Aisle comments, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR replaces regex parsing of `Exec denied` and `Exec finished` metadata with a balanced-parentheses parser, adds parser/followup regression tests, and adds a changelog fix.

## Best Possible Solution

Land the balanced metadata parser together with a trusted-source or structured approval-result path, plus regression coverage for both nested metadata and spoofed command output.

## Reproduction Assessment

Yes. Source inspection and a read-only regex check against current main show nested denied/finished metadata fails the existing parser, and current gateway code emits `approval-timeout (allowlist-miss)` for this path.

## Solution Assessment

No. The balanced parser is the right parser direction, but it is not merge-ready until approval resolution is scoped to structured or otherwise trusted exec lifecycle data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope approval parsing to trusted exec results:** `src/agents/exec-approval-result.ts:88`
  - body: `parseExecApprovalResultText()` is also applied to generic command output before emitting approval events. With the broader balanced parser, a command that prints `Exec denied (... (nested) ...): ...` can now be surfaced as a resolved approval denial, so gate this parsing to structured or otherwise trusted exec lifecycle data before widening the accepted wrapper grammar.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff broadens approval-wrapper parsing while generic exec output is still treated as approval status input.

Concerns:

- **[medium] Spoofable exec approval status from command output:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1154`
  - body: `rawOutput` can come from command output, but the handler parses it with `parseExecApprovalResultText()` and emits a resolved approval event when it parses as denied. Accepting nested metadata expands the set of command-produced strings that can look like trusted approval lifecycle results.
