## Summary

Keep open. The nested-parentheses parser bug is valid, but the latest PR still widens approval-wrapper parsing while a current caller applies that parser to generic command output, so it should not merge until approval status is derived only from trusted exec lifecycle data.

## What This Changes

The PR replaces exec approval result regex parsing with balanced-parentheses metadata parsing, adds nested denied/finished parser and followup tests, and adds a changelog fix entry.

## Best Possible Solution

Land the balanced metadata parser together with a trust-boundary fix that derives approval resolution from structured or otherwise trusted exec lifecycle data, not arbitrary command stdout/stderr.

## Reproduction Assessment

Yes. Source inspection plus a read-only regex check against current main shows nested `Exec denied` and `Exec finished` metadata fail the existing parser, and the gateway can emit the nested denial reason `approval-timeout (allowlist-miss)`.

## Solution Assessment

No. The balanced parser is the right narrow parser direction, but the PR is not merge-ready until approval-event emission is scoped to trusted exec lifecycle data instead of generic command output.

## Review Findings

Full review comments:
