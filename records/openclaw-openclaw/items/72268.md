## Summary

Keep open. The nested-metadata bug is real and the parser direction is useful, but this PR is not merge-ready because it broadens approval-wrapper parsing while generic command output is still parsed as approval status, still carries generated SDK hash churn, lacks the required user-facing changelog entry, and is currently dirty against main.

## What This Changes

The PR replaces exec approval denied/finished regex parsing with balanced metadata parsing, adds nested-metadata parser and followup tests, and modifies the generated plugin SDK API hash file.

## Best Possible Solution

Land a rebased, narrowed version that keeps the balanced parser and regression tests, gates approval-result parsing to trusted or structured exec lifecycle data, removes SDK hash churn, and adds a changelog fix entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: the regex rejects nested approval metadata, the gateway emits `approval-timeout (allowlist-miss)`, and followup routing depends on the denied parser result.

## Solution Assessment

No. The balanced parser is the right narrow fix for nested metadata, but broadening a shared text parser before scoping approval status parsing to trusted lifecycle data is not the safest merge-ready solution.
