## Summary

Keep this PR open: the nested-metadata parser bug is real and the focused tests are useful, but the latest branch still broadens approval-wrapper parsing while generic exec output feeds that parser, and the supplied proof is code-diff inspection rather than after-fix runtime behavior.

## What This Changes

The PR replaces regex-based exec approval result parsing with balanced metadata parsing, changes one gateway denial reason to colon-separated text, strips parentheses from node denial reasons, adds focused tests, and updates the changelog.

## Best Possible Solution

Land the nested-denial fix only after approval-result parsing is scoped to structured or trusted lifecycle data, with regression coverage for spoofed stdout and redacted runtime or Telegram proof of the user-visible followup.

## Reproduction Assessment

Yes, at source level: current main emits nested `approval-timeout (allowlist-miss)` metadata and the current `[^)]*` parser rejects nested denied/finished payloads. I did not establish a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No. Balanced parsing and producer sanitization are useful, but the broader parser should be gated to trusted approval lifecycle data or structured fields before it is applied to generic command output.
