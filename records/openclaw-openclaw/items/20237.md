## Summary

The remaining user problem is real: current main has shipped partial coverage for context usage/compaction, cron editing/history, one-shot jobs, and Web Push, but it still lacks a generic event-driven Control UI notification queue, actionable cron-result popups, and defined snooze/reschedule semantics. The issue should remain open for maintainer product decomposition rather than an automated implementation.

## What This Changes

Add generic Control UI notifications, actionable cron-result popups and rescheduling, inline cron management, and context-pressure monitoring with save/flush actions.

## Best Possible Solution

Decompose the work into a generic connected-Control-UI notification contract, portable typed actions, explicit non-destructive cron snooze semantics, and a separate decision on whether the existing inline context/compact notice also emits notifications.

## Reproduction Assessment

## Solution Assessment

No, not as one bundled implementation. The maintainable solution is to define focused notification, action, cron-rescheduling, and context-state contracts while reusing the already-shipped cron, context, and Web Push foundations.
