## Summary

Keep open. Current main still does not show a Z.AI/AutoGLM expired-token recovery path for the reported Feishu embedded-agent route: the generic runtime-auth retry remains gated on prepared runtime auth state, while the Z.AI plugin is still ordinary API-key auth with no runtime exchange hook.

## What This Changes

The issue asks OpenClaw to retry Feishu embedded-agent requests after an expired Z.AI/AutoGLM JWT returns HTTP 401, and to avoid sending the raw HTTP error back to the Feishu user.

## Best Possible Solution

Add a narrow provider-owned runtime auth recovery path if Z.AI/AutoGLM exposes a deterministic source-credential-to-runtime-token exchange, then keep the generic embedded-run fallback as safe re-authentication guidance rather than raw HTTP status text when refresh cannot happen.

## Reproduction Assessment

Unclear. The issue gives concrete steps and logs, and static source review shows the missing refresh path for plain Z.AI API-key auth, but there is no live Feishu plus expired AutoGLM token run or current-main regression test for this exact route.

## Solution Assessment

Unclear. The existing prepared-runtime-auth retry is the right pattern for providers that implement `prepareRuntimeAuth`, but applying generic API-key refresh in core without Z.AI/AutoGLM semantics would be riskier than a provider-owned hook plus safe error copy.
