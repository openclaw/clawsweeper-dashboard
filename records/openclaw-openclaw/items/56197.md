## Summary

Current `main` still exposes the reported narrow formatter path: plain HTTP 401 token failures have no dedicated invalid-token classifier and fall through to raw HTTP formatting. The issue should stay open because https://github.com/openclaw/openclaw/pull/77394 is the active open implementation path for the safe user-facing masking, while actual refresh-and-retry behavior still needs provider-auth ownership judgment.

## What This Changes

The issue asks OpenClaw to avoid returning raw Feishu embedded-agent `HTTP 401: "Invalid token"` provider errors to users and preferably refresh the provider token before retrying.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/77394 for the safe user-facing 401 masking, then handle any Z.AI/AutoGLM refresh-and-retry behavior through the provider-owned `prepareRuntimeAuth` contract if maintainers choose to support it.

## Reproduction Assessment

Yes for the narrow formatter path from source: current main routes a plain `HTTP 401: "Invalid token"` string through raw HTTP formatting, preserving the reported text. No live Feishu/Z.AI expired-token session was reproduced.

## Solution Assessment

Unclear for the full retry request because token refresh crosses provider runtime-auth ownership. The active scoped PR is the best current path for the already-defined user-visible raw-error bug.
