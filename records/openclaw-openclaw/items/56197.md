## Summary

Keep open. Current main still has the raw HTTP 401 fallthrough and no Z.AI/AutoGLM runtime-auth retry path, and the issue is now paired with open PR #77394 using closing syntax, so the right next step is PR review rather than cleanup closure.

## What This Changes

Feishu embedded-agent sessions using Z.AI/AutoGLM should refresh or retry expired provider tokens, and at minimum should not show raw HTTP 401 `Invalid token` text to chat users.

## Best Possible Solution

Review and either land or revise PR #77394 for the safe user-facing error text, then separately decide whether Z.AI/AutoGLM can implement provider-owned `prepareRuntimeAuth` refresh.

## Reproduction Assessment

No, not as a live high-confidence reproduction. The issue gives concrete logs and current source shows why plain HTTP 401 invalid-token text can fall through and why Z.AI API-key auth will not enter runtime-auth refresh, but there is no current-main live Feishu/AutoGLM expiry run here.

## Solution Assessment

Unclear for the full retry request. The narrow friendly-error fix in the linked PR is a maintainable first step, while automatic token refresh should stay provider-owned through `prepareRuntimeAuth` only if Z.AI/AutoGLM has a deterministic source-credential exchange.
