## Summary

Keep open. Current main still has a plausible gap: runtime-auth retry is gated on existing runtime auth state, the Z.AI provider registers ordinary API-key auth without a runtime exchange hook, and Feishu sends embedded-run final payloads through the generic reply dispatcher. I did not find current-main source, test, or release evidence proving the expired AutoGLM token path refreshes before a user-visible reply.

## What This Changes

Reporter requests Feishu embedded runs using Z.AI/AutoGLM to refresh expired model tokens and retry instead of replying with raw HTTP 401 "Invalid token".

## Best Possible Solution

Add a narrow recovery path for the reported Z.AI/AutoGLM credential shape, preferably provider-owned if the short-lived JWT exchange is specific to that provider. The embedded runner should retry auth-classified 401s only when refreshed credentials are available, and Feishu should receive sanitized re-authentication guidance if recovery cannot happen.

## Reproduction Assessment

Unclear. The issue gives concrete steps and logs, and current source shows a no-refresh path for plain Z.AI API-key auth, but I did not have a live Feishu plus expired AutoGLM token run or focused current-main regression test for this exact route.

## Solution Assessment

Unclear. The existing prepared-runtime-auth retry is a good solution for providers that implement `prepareRuntimeAuth`, but it does not cover the reported Z.AI API-key route; maintainers should choose the provider/core boundary before changing auth semantics.
