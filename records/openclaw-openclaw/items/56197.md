## Summary

Keep open: current main still has the source-level raw HTTP 401 formatter leak for the reported Feishu/Z.AI path, and the linked narrow fix remains an open, unmerged PR. Current runtime-auth retry only helps providers with runtime-auth state; Z.AI has no `prepareRuntimeAuth` hook on main, so the broader refresh-and-retry behavior still needs provider-boundary judgment.

## What This Changes

The issue asks OpenClaw to handle expired Z.AI/AutoGLM tokens in Feishu embedded-agent runs by retrying after refresh, or at least by not sending raw `HTTP 401: "Invalid token"` text to the user.

## Best Possible Solution

Rebase, review, and land the linked safe-401 masking PR if maintainers accept its scope, then handle automatic Z.AI/AutoGLM refresh-and-retry as a separate provider-owned runtime-auth design.

## Reproduction Assessment

Yes for the narrow raw formatter path: current main routes plain `HTTP 401: "Invalid token"` text through raw HTTP formatting. No live Feishu/Z.AI expired-token session was reproduced in this read-only review.

## Solution Assessment

Unclear for the full retry request because it crosses provider runtime-auth ownership; yes for the scoped masking path, where the linked PR is the narrow maintainable fix for the already-defined user-visible raw-error bug.
