## What This Changes

The PR adds a shared proxy-capture header-redaction helper, uses it from runtime capture, and redacts standalone proxy request, response, and CONNECT headers before JSON persistence.

## Best Possible Solution

Land storage-time redaction on the current shared-state capture path, with maintainer-visible context that old capture rows are not scrubbed by this patch.

## Reproduction Assessment

Yes. Source inspection on current main shows standalone HTTP response, request, and CONNECT capture paths persist raw `headersJson`; a live repro would start `openclaw proxy start`, send sensitive headers, and query `capture_events`.

## Solution Assessment

Yes. Reusing the runtime capture predicate at the standalone persistence sites is the narrow maintainable fix; retroactive scrubbing of existing rows is separate operator/migration scope.
