## Summary

Keep open: current main and the latest release still store raw standalone debug-proxy headers, while this PR is the focused open candidate fix with sufficient real behavior proof and no blocking patch finding.

## What This Changes

The PR adds a shared proxy-capture header redaction helper, applies it to standalone HTTP request, HTTP response, and CONNECT capture persistence, and points runtime sensitivity detection at the shared predicate.

## Best Possible Solution

Land the exact merge result so future standalone proxy captures redact all header persistence sites, while treating old capture-row cleanup as separate operator guidance unless maintainers expand the scope.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw proxy start` persists raw HTTP response, HTTP request, and CONNECT headers into `headersJson`; a live repro would start the proxy, send sensitive headers, and query `capture_events`.

## Solution Assessment

Yes. Reusing the existing sensitive-header policy at the standalone proxy persistence sites is the narrow maintainable fix; retroactive scrubbing of old rows is a separate operator or migration decision.
