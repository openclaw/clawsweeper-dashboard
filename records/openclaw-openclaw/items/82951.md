## Summary

Keep open for maintainer merge review: current main and v2026.6.8 still store raw standalone debug-proxy headers, while this PR appears to apply the needed redaction cleanly with sufficient real behavior proof.

## What This Changes

The PR adds a shared proxy-capture header redaction helper and applies it before standalone debug proxy request, response, and CONNECT headers are serialized to capture storage.

## Best Possible Solution

Land storage-time redaction for new standalone proxy captures while making clear that future captures intentionally store redacted sensitive header values and old capture rows are out of scope.

## Reproduction Assessment

Yes. Current main source stores standalone proxy HTTP response, HTTP request, and CONNECT headers directly in capture records, while the sibling runtime path already redacts sensitive names before persistence.

## Solution Assessment

Yes. Reusing one shared sensitive-header predicate at all standalone proxy persistence sites is the narrow maintainable fix; retroactive cleanup of old local captures is separate scope.
