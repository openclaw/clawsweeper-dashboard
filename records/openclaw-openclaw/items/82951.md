## Summary

Keep open: the underlying leak is valid, but this PR is not ready because its copied standalone redaction list omits a sensitive-header class already covered by the runtime path, and the supplied proof is static/typecheck output rather than an after-fix proxy capture run.

## What This Changes

The PR adds standalone debug proxy header redaction for request, response, and CONNECT capture events, plus a changelog entry.

## Best Possible Solution

Use one shared proxy-capture redaction contract, apply it to standalone request, response, and CONNECT capture events, and add focused standalone coverage for sensitive and safe headers.

## Reproduction Assessment

Yes from source: current main serializes raw standalone request, response, and CONNECT headers into persisted capture events. A live repro would be `openclaw proxy start` with fake Authorization/session headers followed by inspecting `headers_json` in the capture DB, but I did not run it because this review is read-only.

## Solution Assessment

No: redacting before persistence is the right direction, but copying the policy misses `session` and should either share the runtime helper or exactly match it with standalone regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Keep standalone redaction in parity with runtime:** `src/proxy-capture/proxy-server.ts:28-35`
  - body: `runtime.ts` redacts any header name containing `session`, but this copied standalone list stops at `credential`. That means captures from `openclaw proxy start` can still persist headers such as `X-Session-Id` while the runtime path redacts them, leaving the security fix incomplete. Add the missing fragment or share the runtime helper so both paths use the same policy.
