## Summary

Keep open: the security/privacy fix is valid, but the PR head still has duplicate declarations that should block build validation, only unit-test proof, and a copied credential-redaction policy that can drift from runtime capture.

## What This Changes

The PR adds standalone debug proxy header redaction, focused redaction unit tests, and a changelog entry for proxy capture secrets.

## Best Possible Solution

Use one shared proxy-capture header-redaction helper in both runtime and standalone paths, then prove fake secrets are redacted in persisted standalone capture rows.

## Reproduction Assessment

Yes from source: current main persists raw standalone request, response, and CONNECT headers while runtime capture already redacts sensitive header names. I did not run a live proxy capture because this review is read-only.

## Solution Assessment

No as-is: redacting before persistence is the right direction, but the PR should remove duplicate declarations and avoid a second redaction policy that can drift from `runtime.ts`.
