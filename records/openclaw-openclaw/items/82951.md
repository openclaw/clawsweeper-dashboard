## Summary

Keep open for normal PR review: this is a valid security bugfix for standalone proxy captures, and the earlier session-fragment gap has been fixed on the PR head, but the external PR still lacks real after-fix proxy-capture proof from the actual changed path.

## What This Changes

The PR adds standalone debug proxy header redaction before persisting HTTP request, HTTP response, and CONNECT capture headers, plus a changelog entry.

## Best Possible Solution

Use one proxy-capture redaction contract, apply it consistently before SQLite persistence in runtime and standalone paths, and land after real standalone capture proof is supplied.

## Reproduction Assessment

Yes from source, though I did not run a live proxy capture: current main and the latest release serialize standalone request, response, and CONNECT headers directly into `headers_json`. A live path would run `openclaw proxy start` with fake Authorization/Cookie/X-Session-Id headers and query `capture_events.headers_json`.

## Solution Assessment

Mostly yes: redacting before persistence at all three standalone storage points matches the runtime sensitive header policy after the follow-up commit. It would be more maintainable with a shared helper or direct standalone tests, but I do not see a remaining concrete code blocker.
