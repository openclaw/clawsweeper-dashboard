## Summary

## What This Changes

The issue requests replacing or supplementing character-based workspace bootstrap limits and doctor reporting with token-based limits or estimates.

## Best Possible Solution

Pick the product contract for token accounting first, then implement token-aware reporting or limits across config, runtime budget analysis, doctor output, docs, and any required doctor migration.

## Reproduction Assessment

Yes for the current source state: config, runtime truncation, budget analysis, doctor output, and docs all use character-based bootstrap limits. This is not a failing bug repro; the missing token-based behavior is a feature/product request.

## Solution Assessment

Unclear. Token-based reporting is directionally useful, but the best permanent fix depends on maintainer decisions about estimator contracts, model/provider variance, and migration from the existing char-based config surface.
