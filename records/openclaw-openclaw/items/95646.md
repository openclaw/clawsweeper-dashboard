## Summary

Keep open: current main and v2026.6.10 still have the source-proven Mattermost private-channel split where authoritative `P` maps to `group`, but target-string/session derivation can still classify `channel:<id>` as `channel`; the linked open PRs are fix candidates, not closure proof.

## What This Changes

The issue requests deterministic Mattermost private-channel `chat_type` routing by adding Mattermost target chat-type inference and fixing lossy generic target-string inference.

## Best Possible Solution

Land or replace a focused Mattermost fix that keeps channel-specific classification in the plugin, consults the hook before lossy `channel:` defaults, and adds regression coverage for private-channel `group` session namespaces across outbound/session and announce-delivery paths.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: Mattermost private type `P` maps to `group`, while `channel:<id>` target/session inference still defaults to `channel` before a plugin can classify it.

## Solution Assessment

Yes. The best fix is a narrow plugin-boundary repair: Mattermost should expose authoritative target chat-type inference, and shared inference paths should let that hook override lossy target-prefix defaults while preserving public channels as `channel`.
