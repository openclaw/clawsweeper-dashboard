## Label Justifications

- `P2`: This is a normal-priority Mattermost routing bug with limited plugin-specific blast radius but real session and delivery impact.
- `impact:session-state`: The same Mattermost private-channel thread can be split across `group` and `channel` session namespaces.
- `impact:message-loss`: Threaded or scheduled deliveries can fail closed or look up the wrong conversation when the session namespace is inferred inconsistently.

## Summary

Keep open: current main and the latest release still have the Mattermost private-channel split where authoritative inbound type `P` maps to `group`, but outbound/session paths can still classify the same `channel:<id>` target as `channel`; the open linked PRs are implementation candidates, not merged fixes.

## What This Changes

The issue requests deterministic Mattermost private-channel `chat_type` derivation so private channels keep the same `group` session namespace across inbound and delivery paths.

## Best Possible Solution

Review and land a narrow Mattermost-owned routing fix that preserves private-channel sessions under `group` while keeping the Mattermost wire target as `channel:<id>`, then close this issue after the chosen PR merges.

## Reproduction Assessment

Yes for source reproduction: current `main` maps Mattermost `P` channels to `group`, while current outbound/session inference still turns `channel:<id>` paths into `channel`. I did not run a live Mattermost server in this read-only pass.

## Solution Assessment

Yes: the best fix is a bounded Mattermost plugin-boundary repair that carries or resolves authoritative channel kind into outbound session routing; generic inference ordering should change only where needed to let plugin-owned semantics override lossy target strings.
