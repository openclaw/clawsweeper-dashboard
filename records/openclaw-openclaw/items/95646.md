## Summary

Keep open: current `main` and the latest release still have a real Mattermost outbound/session-route mismatch where private-channel inbound context is classified as `group`, but outbound target/session derivation can still reconstruct the same `channel:<id>` target as `channel`. The report is source-reproducible and narrow enough for a focused fix PR.

## What This Changes

Mattermost private-channel chat type inference reportedly disagrees between authoritative channel metadata and lossy delivery target strings, splitting session namespaces for the same conversation.

## Best Possible Solution

Mattermost should keep chat-type classification at the plugin boundary and make outbound session routing derive `group` vs `channel` from authoritative Mattermost channel type or resolved target metadata, with public channels staying `channel` and private channels staying `group`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: inbound Mattermost private-channel posts use channel type `P -> group`, while outbound Mattermost session routing rebuilds non-user `channel:<id>` targets as `channel`. I did not run a live Mattermost server.

## Solution Assessment

Yes, the best fix is a narrow plugin-owned route/classification repair, not a new config option or broad core policy change. The repair should also check the generic heartbeat inference order only if Mattermost cannot otherwise return authoritative route metadata before the fallback.
