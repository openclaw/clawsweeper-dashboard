## Summary

Keep open: current main and v2026.6.9 still have the source-proven Mattermost private-channel split, and the open candidate PRs that close this issue have not merged.

## What This Changes

Mattermost private channels should keep the authoritative `group` chat_type consistently across inbound, delivery, and session routing instead of falling back to `channel` from `channel:<id>` target strings.

## Best Possible Solution

Land a corrected candidate PR that preserves authoritative Mattermost `P`/`G`/`O` channel typing through target inference, session-route peer/from/chatType, shared hook-before-fallback inference, focused tests, and redacted live Mattermost proof.

## Reproduction Assessment

Yes, from source inspection: current main maps Mattermost `P` to `group`, but delivery/session paths can still derive the same `channel:<id>` target as `channel`. I did not run a live Mattermost workspace.

## Solution Assessment

Yes, the best fix is a narrow plugin-boundary repair plus shared inference ordering so authoritative Mattermost channel type, not a lossy target prefix, owns session chat_type.
