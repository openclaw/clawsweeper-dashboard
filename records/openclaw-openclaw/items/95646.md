## Summary

Keep open: current main and v2026.6.9 still have the Mattermost private-channel split where inbound type `P` maps to `group`, while outbound/session derivation can still turn the same `channel:<id>` target into `channel`. The issue already has open candidate fix PRs at https://github.com/openclaw/openclaw/pull/95655 and https://github.com/openclaw/openclaw/pull/95669, so it should remain open until one fix lands or maintainers choose a canonical path.

## What This Changes

The issue requests deterministic Mattermost private-channel chat type/session routing so type-based `group` and target-string `channel` inference no longer split the same conversation.

## Best Possible Solution

Land one focused fix that keeps Mattermost chat-type classification at the plugin boundary, honors resolved `group` kind in outbound session routes, and lets plugin inference override lossy `channel:` fallback paths where needed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Mattermost `P` maps to `group` inbound, while outbound/session code still builds `channel:` routes and generic target inference returns `channel` before plugin hooks. I did not run a live Mattermost workspace.

## Solution Assessment

Yes, the best fix is a narrow plugin-boundary repair: derive Mattermost private-channel chat type from authoritative channel kind or resolved target metadata, and adjust shared fallback order only where plugin-owned semantics must beat lossy prefixes.
