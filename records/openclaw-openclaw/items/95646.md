## Summary

Keep open: current main and v2026.6.10 still classify Mattermost private-channel inbound state as `group` while outbound/session target paths can still key the same `channel:<id>` target as `channel`; https://github.com/openclaw/openclaw/pull/96645 and https://github.com/openclaw/openclaw/pull/95669 are open candidate fixes rather than merged resolution.

## What This Changes

The issue requests deterministic Mattermost private-channel `chat_type` and session routing by deriving outbound/private-channel chat type from authoritative channel kind instead of lossy target-string prefixes.

## Best Possible Solution

Land one canonical Mattermost-owned fix that derives private-channel outbound/session type from authoritative channel kind or equivalent resolved metadata, preserves `channel:<id>` wire delivery targets, and proves private and public channel behavior before closing this issue.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main maps Mattermost `P`/`G` inbound channel types to `group`, while outbound/session reconstruction and opaque target resolution still build non-user targets as `channel`. I did not run a live Mattermost workspace.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Mattermost plugin boundary, using authoritative channel type or current route metadata to keep private channels in the `group` namespace without changing the Mattermost wire target.
