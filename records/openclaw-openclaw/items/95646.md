## Summary

Keep open: current main still has divergent Mattermost chat-type derivation for private channels, with inbound metadata mapping `P` to `group` while outbound/session-target paths still classify `channel:<id>` as `channel` or hard-code non-user Mattermost routes as `channel`.

## What This Changes

The issue requests deterministic Mattermost `chat_type` derivation for private channels so inbound metadata, delivery targets, and session keys use the same `group` namespace.

## Best Possible Solution

Mattermost target/session routing should classify channel IDs from authoritative Mattermost channel type and carry that same `direct|group|channel` value through outbound delivery, heartbeat, cron, and subagent announcement session-key paths without adding config or core Mattermost policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a Mattermost `P` channel maps to `group` inbound, but current outbound/session inference treats `channel:<id>` and all non-user Mattermost route targets as `channel`. I did not run a live Mattermost workspace scenario in this read-only review.

## Solution Assessment

The report identifies the right owner boundary, but a hook-only fix is probably too narrow because Mattermost needs channel metadata to distinguish public and private `channel:<id>` targets. The best fix should use the existing async Mattermost target resolver/session-route seam where possible, then make generic prefix fallback order consistent for plugins that can classify synchronously.
