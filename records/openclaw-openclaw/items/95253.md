- **[P2] Return filenames from the service-env mock:** `test/scripts/postinstall-bundled-plugins.test.ts:1245-1246`
  - body: This mock returns a Dirent-like object, but `readGatewayPortFromServiceEnv()` calls `f.endsWith('.env')` because real `readdirSync()` without `withFileTypes` returns strings. The test will catch the resulting TypeError, fall through, and fail instead of proving launchd service-env resolution.
  - confidence: 0.96
- **[P2] Align the systemd precedence test with the resolver:** `test/scripts/postinstall-bundled-plugins.test.ts:1284-1296`
  - body: This fixture makes both `gateway.systemd.env` and the unit file exist, but the resolver reads `gateway.systemd.env` first and current systemd parsing lets EnvironmentFile values override inline Environment values. The assertion expects `24680`, while the code path returns `13579`; either the test fixture should remove the env file or the resolver contract must change deliberately.
  - confidence: 0.94
- **[P2] Use the installed service metadata contract:** `scripts/postinstall-bundled-plugins.mjs:1226-1233`
  - body: This hard-codes a default unit name/path and regexes one inline Environment line instead of following the existing systemd service reader that resolves unit identity, EnvironmentFile specs, and file-overrides-inline precedence. That can probe the wrong port for profile/custom services or future supervisor metadata even though the PR claims installed service custom ports are covered.
  - confidence: 0.82
