## Summary

Keep open: current `main` and the latest release still contain the reported abort path, and the related issues do not own this exact stale Codex thread/preflight compaction failure. The issue is source-reproducible enough to queue a focused repair rather than close as duplicate or stale.

## What This Changes

The issue reports Telegram inbound dispatch failing before an assistant run starts when reply-resolver preflight compaction hits a stale or missing Codex thread id.

## Best Possible Solution

Teach inbound preflight compaction to recover only missing or stale Codex thread bindings, using the existing binding cleanup and context-engine or fresh-session path while preserving terminal errors for unsafe failures.

## Reproduction Assessment

Yes, source-reproducible: current `main` turns a stale Codex `thread not found` compaction result into the exact preflight error reported by the issue. I did not run a live Telegram group repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable direction is to reuse the existing recoverable missing/stale-binding distinction instead of adding a new config mode or swallowing all compaction failures. The fix should include regression coverage for stale binding recovery and for non-recoverable failures staying terminal.

Implement a bounded recovery for missing/stale Codex thread bindings during inbound preflight compaction, reusing the existing failure.reason contract and the CLI compaction precedent where possible.
