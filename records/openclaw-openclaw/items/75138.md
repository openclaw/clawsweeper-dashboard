## Summary

Keep open. Current main still has the daemon/runtime env gap, and the proposed code path is a narrow functional fix that matches existing dotenv precedence. The remaining actionable cleanup is release/docs metadata, not a functional blocker.

## What This Changes

The PR adds `<stateDir>/gateway.env` to the global runtime dotenv load order after `<stateDir>/.env` and adds a unit test for loading `GATEWAY_AUTH_TOKEN` while preserving `.env` precedence.

## Best Possible Solution

Land the functional loader/test change after adding the required Unreleased fix note and updating the environment docs to show `<stateDir>/gateway.env` after `<stateDir>/.env` and before the legacy XDG fallback.

## Reproduction Assessment

Yes. Current main's loader has no path that reads `<stateDir>/gateway.env`; a source-level reproduction is to put `GATEWAY_AUTH_TOKEN` only in the state-dir `gateway.env` and follow the service-env regeneration path that reads from `process.env`.

## Solution Assessment

Yes for the code path. Reading `path.join(path.dirname(stateEnvPath), "gateway.env")` after `stateEnvPath` is the narrowest maintainable fix because it preserves existing `.env` precedence and keeps the XDG fallback as compatibility behavior.
