Keep open. The functional patch addresses a real current-main gap and keeps the intended precedence, but it is not merge-ready until the user-facing changelog and environment documentation are updated.

## What This Changes

The PR adds `<stateDir>/gateway.env` to the global runtime dotenv load chain after `<stateDir>/.env` and adds a unit test for loading `GATEWAY_AUTH_TOKEN` while preserving `.env` precedence.

## Best Possible Solution

Land the loader and unit-test change after adding a single-line Unreleased fix note and updating the environment docs/FAQ to show `$OPENCLAW_STATE_DIR/gateway.env` after `$OPENCLAW_STATE_DIR/.env` and before the legacy XDG fallback, keeping this in the shared trusted runtime dotenv loader rather than daemon-specific repair code.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: put `GATEWAY_AUTH_TOKEN` only in `$OPENCLAW_STATE_DIR/gateway.env`, then run a service-env regeneration path; the loader never sources that file and gateway auth SecretRefs are not persisted into service-env metadata.

## Solution Assessment

Yes for the code path. Reading sibling `<stateDir>/gateway.env` through the trusted global dotenv loader after `<stateDir>/.env` is the narrowest maintainable fix because it preserves existing precedence and keeps the legacy XDG fallback as compatibility behavior.
