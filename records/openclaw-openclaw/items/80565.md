Keep open. Current `main` still runs the reported Docker A2UI step through `pnpm` after the full source tree is copied, and the source path plus reporter logs support the claim that pnpm can re-evaluate optional native packages before the existing stub fallback runs.

## What This Changes

The issue asks for Linux arm64 Docker builds to avoid or fast-fail unrelated optional native binary downloads during the `canvas:a2ui:bundle` layer.

## Best Possible Solution

Keep the existing non-fatal A2UI stub behavior, but make the Docker A2UI layer avoid pnpm workspace re-evaluation and unrelated optional-platform fetches on Linux arm64.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80565. In the root Dockerfile, avoid invoking `pnpm canvas:a2ui:bundle` after `COPY . .` in a way that lets pnpm re-evaluate the workspace and fetch optional native packages for unrelated platforms. Prefer calling the existing Node wrapper directly, or an equivalent Docker-only path, while preserving the current non-fatal A2UI stub fallback. If `extensions/canvas/scripts/bundle-a2ui.mjs` can still fall back to `pnpm exec rolldown`, make the Docker path use installed local tooling or fail fast into the stub rather than downloading packages. Add focused coverage that guards the Dockerfile command shape, update `CHANGELOG.md` under Fixes, and do not change Docker base images, package manager/runtime choice, bundled plugin selection policy, or add a new build flag unless the direct-invocation path cannot solve the stall.
