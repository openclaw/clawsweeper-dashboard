## Summary

Current main already addresses the reported Docker/A2UI optional-binary fetch path by constraining Docker dependency phases to the target Linux architecture and disabling pnpm's pre-run dependency verification for the A2UI/build steps.

## What This Changes

The issue reports arm64 Docker builds stalling during A2UI bundling because pnpm fetches optional native packages for unrelated platforms.

## Best Possible Solution

Keep the current-main Dockerfile implementation and let the next release carry it; reopen with fresh logs only if the same stall occurs from a checkout at or after 6066e878f690955100c6847d12b29015002a24a7.

## Reproduction Assessment

No current-main failing path was established in this review. The reported v2026.5.7-style Dockerfile path is source-reproducible, but current main has the dependency-check guard that should prevent the optional binary fetch before A2UI bundling.

## Solution Assessment

Yes. Using pnpm's supportedArchitectures and verifyDepsBeforeRun=false settings is a narrow maintainable fix that preserves the existing A2UI fallback without adding a new build flag or bypassing the package-manager contract.
