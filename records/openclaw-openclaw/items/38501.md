## Summary

Keep open: current main still lacks active-run-aware Ctrl+C feedback, and #38502 is an open focused PR paired with this issue. #44223 separately raises the broader question of whether Ctrl+C should abort active runs rather than only clarify status text.

## What This Changes

The issue asks for TUI Ctrl+C feedback to make clear that an active run continues unless stopped with Escape or `/abort`.

## Best Possible Solution

Review and land the focused status-text fix if existing Ctrl+C semantics should remain, or deliberately choose the broader abort-on-Ctrl+C behavior and land that with matching tests and docs.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: during an active run, Ctrl+C reaches `handleCtrlC` and `resolveCtrlCAction` without reading `activeChatRunId`, while Escape and `/abort` use `abortActive`.

## Solution Assessment

Unclear pending maintainer policy. The narrow fix in #38502 best matches the reported wording mismatch, while #44223 changes Ctrl+C behavior and should be reviewed as a deliberate UX decision.
