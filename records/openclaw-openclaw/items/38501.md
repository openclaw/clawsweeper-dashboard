## Summary

Keep open: current main still routes TUI Ctrl+C through clear/warn/exit handling without active-run-aware feedback, and a focused open closing PR plus a broader Ctrl+C behavior PR are already awaiting maintainer review.

## What This Changes

The issue asks for TUI Ctrl+C feedback during an active run to say the run is still active and can be stopped with Esc or `/abort`.

## Best Possible Solution

Choose the intended Ctrl+C policy, then land either the focused status-text fix in #38502 or a reconciled broader abort-priority fix from #44223 with regression coverage and matching docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: during an active run, Ctrl+C reaches `handleCtrlC` and `resolveCtrlCAction` without reading `activeChatRunId`, while Esc and `/abort` use `abortActive`.

## Solution Assessment

Unclear pending maintainer policy. #38502 is the narrowest fix for the reported misleading status text, while #44223 deliberately changes Ctrl+C behavior and should be reviewed as a broader UX decision.
