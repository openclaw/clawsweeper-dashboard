## Summary

Keep open: current main still routes TUI Ctrl+C through clear/warn/exit status handling without active-run awareness, while Escape and `/abort` use the abort path. A focused closing PR and a broader Ctrl+C behavior PR are already open, so this should stay with maintainer review rather than be closed or queued as another repair.

## What This Changes

The issue requests clearer TUI Ctrl+C feedback during an active run so users know the run is still active and can stop it with Escape or `/abort`.

## Best Possible Solution

Choose one TUI Ctrl+C policy and land the matching patch with focused regression coverage. If current clear/exit semantics remain intended, merge #38502 or equivalent active-run hint text; if Ctrl+C should abort active runs, review #44223 deliberately and align tests plus docs.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: during an active run, Ctrl+C reaches `handleCtrlC` and `resolveCtrlCAction`, which ignore `activeChatRunId`, while Escape and `/abort` route through `abortActive`.

## Solution Assessment

Unclear pending maintainer policy. The narrowest maintainable fix for the reported UX mismatch is the focused status-text hint, while changing Ctrl+C to abort active runs is a broader behavior decision.
