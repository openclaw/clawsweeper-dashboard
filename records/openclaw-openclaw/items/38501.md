## Summary

Keep open: current main still routes TUI Ctrl+C through clear/warn/exit feedback without checking the active run, and an open closing PR plus a broader Ctrl+C behavior PR are already pending maintainer review.

## What This Changes

The issue requests clearer TUI Ctrl+C status text when an active run continues after Ctrl+C.

## Best Possible Solution

Choose the intended Ctrl+C policy, then land either the focused active-run status hint from #38502 or a reconciled broader abort-precedence change from #44223 with matching regression coverage.

## Reproduction Assessment

Yes, at source level. During an active run, Ctrl+C reaches handleCtrlC and resolveCtrlCAction without reading activeChatRunId, while Esc and /abort use abortActive.

## Solution Assessment

Unclear pending maintainer UX policy. #38502 is the narrow status-text fix for the reported confusion, while #44223 changes Ctrl+C behavior and should be reviewed as a broader interaction decision.
