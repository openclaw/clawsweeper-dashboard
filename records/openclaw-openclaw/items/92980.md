## Summary

Keep open: current main still dead-letters missing-harness failures but does not handle `ERR_MODULE_NOT_FOUND` or repeated same-signature poison failures, and two open PRs now explicitly close this issue so the issue should remain until a fix PR lands.

## What This Changes

The issue asks Telegram spooled update processing to dead-letter non-retryable `ERR_MODULE_NOT_FOUND` and missing-harness failures, add a poison retry policy for repeated same-signature failures, and avoid freezing the gateway behind one bad update.

## Best Possible Solution

Review and land the canonical Telegram PR that handles `ERR_MODULE_NOT_FOUND` plus scoped poison retries, or land the narrow classifier fix while keeping a separate tracked follow-up for poison retry policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: an unclassified `ERR_MODULE_NOT_FOUND` from `bot.handleUpdate` reaches `#releaseFailedSpooledUpdate`, releases the claim, and is kept for retry. I did not run a live Telegram gateway reproduction.

## Solution Assessment

Is this the best way to solve the issue? The dead-letter policy belongs in `extensions/telegram/src/polling-session.ts`, but the env-var retry limit is not the best first step without maintainer policy because OpenClaw keeps config surface intentionally tight.
