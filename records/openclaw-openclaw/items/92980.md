## Summary

Keep open: current main and v2026.6.8 still do not classify `ERR_MODULE_NOT_FOUND` or repeated same-signature poison failures as non-retryable Telegram spooled-update errors, and the broader candidate fix is still an open PR needing maintainer review.

## What This Changes

The issue reports a Telegram ingress spool retry loop where non-retryable spooled update failures are requeued instead of dead-lettered, blocking gateway message processing.

## Best Possible Solution

Implement the Telegram-owned dead-letter policy for module-missing and same-signature poison failures, with scoped poison state and tests, and avoid new env/config surface unless maintainers explicitly choose it.

## Reproduction Assessment

Yes, at source level: an `ERR_MODULE_NOT_FOUND` thrown from `bot.handleUpdate` is not classified by `resolveNonRetryableSpooledUpdateFailure`, so current main releases the claim for retry. I did not run a live Telegram gateway reproduction.

## Solution Assessment

No final implementation is merged yet. The right fix belongs in the Telegram polling owner module, but the open PR needs maintainer agreement on poison scope/config policy and real transport proof before the issue should close.
