## Summary

This issue is still necessary: current main and v2026.6.8 dead-letter missing-harness and dispatch-dedupe rollback failures, but ERR_MODULE_NOT_FOUND and repeated same-signature poisoned spooled failures still fall through to retry.

## What This Changes

The issue requests Telegram spooled updates to dead-letter non-retryable module/harness failures and repeated same-signature poison loops instead of freezing gateway message processing.

## Best Possible Solution

Keep this issue open as the canonical report until a Telegram-owned fix lands that classifies ERR_MODULE_NOT_FOUND as non-retryable, adds scoped poison retry protection or an approved equivalent policy, preserves transient retries, and includes focused tests plus real Telegram proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, at source level: current main routes unclassified spooled handler failures through releaseTelegramSpooledUpdateClaim, and the classifier does not recognize ERR_MODULE_NOT_FOUND. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Unclear as a final shape: the Telegram polling classifier is the right boundary for known permanent errors, but the configurable poison counter is retry-policy/config surface that needs maintainer judgment and scoping proof.
