## Summary

This issue is still necessary: current main and the latest release still requeue unclassified Telegram spooled-update failures, and the two linked closing PRs are open rather than merged.

## What This Changes

The issue requests Telegram spooled updates to dead-letter permanent failures such as `ERR_MODULE_NOT_FOUND`, add a poison retry cap, and optionally expose the cap through configuration.

## Best Possible Solution

Review and land a Telegram plugin fix that dead-letters permanent module-missing failures and safely scopes any poison retry state, while adding configuration only if maintainers explicitly accept that surface.

## Reproduction Assessment

Yes, at source level: an `ERR_MODULE_NOT_FOUND` thrown from `bot.handleUpdate` is not classified as non-retryable on current main, so the failure path releases the claim for retry. I did not run a live Telegram gateway reproduction.

## Solution Assessment

No final solution has landed yet. The Telegram-owned classifier/dead-letter path is the right boundary, but the poison cap, account/spool scoping, and requested config knob need maintainer acceptance before this issue can close.
