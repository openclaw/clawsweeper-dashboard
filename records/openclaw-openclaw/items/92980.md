## Summary

Keep open: current main already dead-letters missing-harness and dispatch-dedupe rollback failures, but stale `ERR_MODULE_NOT_FOUND` replay failures and repeated same-signature poison retries still fall through to the generic Telegram spool retry path.

## What This Changes

The issue requests Telegram ingress spool retry handling to dead-letter non-recoverable stale module/harness failures and cap repeated poison retries.

## Best Possible Solution

Extend the Telegram-owned spool failure policy for non-recoverable stale module errors and poison retries while preserving transient retry behavior.

## Reproduction Assessment

Yes at source level: current main routes unclassified spooled update errors through `releaseTelegramSpooledUpdateClaim`, while `ERR_MODULE_NOT_FOUND` is absent from the non-retryable classifier. I did not run a live stale-package Telegram gateway reproduction.

## Solution Assessment

Partly: dead-lettering known non-recoverable stale-module signatures is the narrow bug fix, but adding an env-tunable poison retry limit needs maintainer approval because repository policy keeps the config/env surface high-bar.
