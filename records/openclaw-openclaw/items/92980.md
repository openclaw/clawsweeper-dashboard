## Summary

Keep open: current main and v2026.6.8 still requeue unclassified Telegram spooled update failures, while the open candidate PR is not yet a safe merged fix for the reported retry loop.

## What This Changes

The issue asks Telegram spooled update handling to dead-letter permanent module-missing and missing-harness failures, add poison retry handling, and optionally expose a retry-limit env var.

## Best Possible Solution

Land or replace the open candidate with a Telegram-owned fix that dead-letters ERR_MODULE_NOT_FOUND, scopes poison retry state by account or spool, preserves transient retries, and avoids a new env var unless maintainers explicitly approve that config surface.

## Reproduction Assessment

Yes at source level: an ERR_MODULE_NOT_FOUND thrown from bot.handleUpdate is not classified as non-retryable on current main, so the failed claim falls through to releaseTelegramSpooledUpdateClaim and is retried. I did not establish a live Telegram gateway reproduction in this read-only review.

## Solution Assessment

Unclear as proposed. The Telegram polling-session failure classifier is the right owner boundary, but the env-var surface and generic poison retry policy need maintainer direction, and the current open PR needs account/spool scoping before it is the safest fix.
