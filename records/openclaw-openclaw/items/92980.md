## Label Justifications

- `impact:message-loss`: The issue is about Telegram ingress messages not draining while the spool retries the same poisoned update.

## Summary

Keep this open: current main and the latest release still dead-letter missing-harness and dispatch-dedupe failures only, while `ERR_MODULE_NOT_FOUND` and repeated same-signature poison failures still fall through to the Telegram spool retry path. The active path is to review the open fix candidate rather than close this issue early.

## What This Changes

The issue asks for Telegram spooled updates with permanent module/harness failures or repeated poison signatures to be dead-lettered instead of retrying forever and freezing gateway message processing.

## Best Possible Solution

Implement the fix in the Telegram polling owner module by classifying permanent module-missing failures and adding a spool/account-scoped poison backstop with focused regression coverage and live Telegram proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: an unclassified `ERR_MODULE_NOT_FOUND` from `bot.handleUpdate` reaches `#releaseFailedSpooledUpdate`, returns no non-retryable reason, and is released for retry. I did not run a live Telegram gateway reproduction.

## Solution Assessment

No complete solution is on main yet. The right layer is Telegram polling/spool failure policy, but the open candidate still needs maintainer agreement on poison retry scope/config and real transport proof before it can close the issue.
