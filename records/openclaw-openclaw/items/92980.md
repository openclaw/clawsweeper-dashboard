## Label Justifications

- `P1`: A poisoned Telegram spooled update can block real channel message processing until external intervention.
- `impact:message-loss`: The reported loop prevents later Telegram messages from being processed while the poisoned update keeps retrying.
- `impact:crash-loop`: The gateway remains process-alive but effectively hangs in a tight retry loop, creating an availability failure.

## Summary

Keep open: current main and v2026.6.8 still retry unclassified Telegram spooled update failures, so `ERR_MODULE_NOT_FOUND` can still fall through to the retry path. The missing-harness part is already dead-lettered, but the module-missing and same-signature poison cap work is only represented by unmerged PRs, with https://github.com/openclaw/openclaw/pull/93024 still needing maintainer review and proof.

## What This Changes

The issue requests Telegram spooled update handling that dead-letters non-retryable failures such as missing modules or unknown harnesses, adds a poison retry limit, and optionally exposes that limit through an environment variable.

## Best Possible Solution

Land or rework the Telegram-owned failure policy so `ERR_MODULE_NOT_FOUND` is dead-lettered and repeated same-signature poison failures are capped with account-scoped state, focused regression tests, and live Telegram proof; add an env knob only if maintainers explicitly accept the config surface.

## Reproduction Assessment

Yes, at source level: an `ERR_MODULE_NOT_FOUND` thrown from `bot.handleUpdate` is not classified as non-retryable, so the fallback path releases the claim for retry. I did not run a live Telegram gateway reproduction.

## Solution Assessment

Partly. Dead-lettering permanent module-missing failures in the Telegram polling owner module is the right direction, but the poison retry threshold and requested env var need maintainer policy judgment before the issue is fully solved.
