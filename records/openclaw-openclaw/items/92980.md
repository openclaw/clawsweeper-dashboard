## Summary

Keep open: current main and the latest release still dead-letter only missing-harness and dispatch-dedupe rollback failures, so an ERR_MODULE_NOT_FOUND or repeated same-signature poison failure still falls through to retry. The missing-harness part is already fixed by a merged PR, but the central stale-module retry loop remains owned by the open candidate PR and still needs maintainer review.

## What This Changes

The issue reports Telegram spooled updates being retried indefinitely after non-retryable ERR_MODULE_NOT_FOUND or missing-harness failures, freezing gateway delivery, and requests dead-lettering plus a poison retry cap.

## Best Possible Solution

Land or replace the open candidate with a Telegram-owned fix that dead-letters ERR_MODULE_NOT_FOUND, scopes poison retry state by account or spool plus update id, preserves transient retries, and avoids a new env var unless maintainers explicitly approve that config surface.

## Reproduction Assessment

Yes, at source level: an ERR_MODULE_NOT_FOUND thrown from bot.handleUpdate is not classified as non-retryable on current main, so the failed claim falls through to releaseTelegramSpooledUpdateClaim and is retried. I did not run a live Telegram gateway reproduction in this read-only review.

## Solution Assessment

Unclear as proposed. The Telegram polling-session classifier is the right owner boundary, but the env-var surface and generic poison retry policy need maintainer direction, and the open candidate PR needs account/spool scoping before it is the safest fix.
