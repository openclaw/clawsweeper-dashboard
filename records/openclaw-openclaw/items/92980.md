## Label Justifications

- `P1`: A poisoned Telegram spooled update can keep a real gateway unavailable for message processing until external intervention.
- `impact:crash-loop`: The report describes a tight retry loop that makes the gateway appear frozen even though the process stays alive.
- `impact:message-loss`: Messages are not processed while the Telegram spool is blocked behind the poisoned update.

## Summary

The issue is still necessary: current main dead-letters missing-harness and dispatch-dedupe failures, but `ERR_MODULE_NOT_FOUND` and repeated same-signature poisoned spooled updates still fall through to the generic retry path.

## What This Changes

The issue asks Telegram isolated ingress to dead-letter non-retryable spooled update failures and cap repeated poison retries instead of looping indefinitely.

## Best Possible Solution

Extend the Telegram-owned spooled update failure policy to dead-letter `ERR_MODULE_NOT_FOUND` and repeated same-signature poison failures while preserving retry for transient Telegram, filesystem, and network errors.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at the source level: an `ERR_MODULE_NOT_FOUND` thrown from `bot.handleUpdate` is not classified by `resolveNonRetryableSpooledUpdateFailure`, so it falls through to `releaseTelegramSpooledUpdateClaim` and is kept for retry.

## Solution Assessment

Is this the best way to solve the issue? The dead-letter direction is right, but the env-var retry limit is a product/config decision; a fixed poison policy plus focused tests may be the safer first implementation.
