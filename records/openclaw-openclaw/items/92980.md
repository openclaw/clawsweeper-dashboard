## Label Justifications

- `P1`: The report describes a gateway freeze that stops Telegram message processing for real users.
- `impact:message-loss`: The poisoned spool entry can block later Telegram updates from being processed.
- `impact:crash-loop`: The same update can repeatedly fail and retry fast enough to make the gateway unavailable even though the process stays alive.

## Summary

Keep open: current main and the latest release still requeue unclassified Telegram spooled handler failures, so `ERR_MODULE_NOT_FOUND` can still hit the retry path; two open closing PRs now own candidate fixes and should land or be replaced before this issue closes.

## What This Changes

The issue requests Telegram spooled updates to dead-letter permanent failures such as missing modules and unknown harnesses, add a bounded same-signature poison retry cap, reset that state on success, and optionally expose the cap through an environment variable.

## Best Possible Solution

Land a reviewed Telegram-owned fix, likely the broader PR or a replacement, that dead-letters permanent module failures and scopes bounded poison retry state correctly without adding env/config surface unless maintainers explicitly approve it.

## Reproduction Assessment

Yes at source level: an unclassified `ERR_MODULE_NOT_FOUND` thrown by `bot.handleUpdate` reaches `#releaseFailedSpooledUpdate`, then `releaseTelegramSpooledUpdateClaim`, so the same queued update can be claimed again. I did not run a live Telegram gateway reproduction.

## Solution Assessment

Partly: dead-lettering permanent Telegram spool failures is the right owner-boundary fix, and the broader linked PR is closer to the requested behavior than the narrower classifier-only PR. The env-var cap is not automatically the best solution because repository policy sets a high bar for new config/env surfaces.
