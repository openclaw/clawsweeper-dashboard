## Summary

Keep open: current main and latest release still do not support `OPENCODE_GO_API_KEY` for the active OpenCode Go provider auth path, and the prior implementation PR is closed unmerged. The request is aligned with provider support, but it changes public credential precedence and shared-profile behavior, so it needs maintainer product approval before another fix attempt.

## What This Changes

The issue requests separate OpenCode Zen and OpenCode Go API-key environment variables with shared `OPENCODE_API_KEY` fallback.

## Best Possible Solution

Maintainers should decide the OpenCode credential model, then land an additive plugin-local split-key implementation that preserves shared-key fallback/profile compatibility and updates provider metadata, auth lookup, docs, and focused tests together.

## Reproduction Assessment

Yes. Source inspection shows that setting only `OPENCODE_GO_API_KEY` is not a supported current-main path because the Go provider runtime, manifest, docs, and legacy env map do not advertise or resolve that env var.

## Solution Assessment

Unclear until maintainers decide the public credential model. Provider-specific keys with shared fallback are plausible, but the best fix must preserve current shared-key behavior or explicitly document and approve the upgrade impact.
