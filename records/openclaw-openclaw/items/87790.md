## Summary

Keep open: current main and the latest release still treat OpenCode Go as a shared `OPENCODE_API_KEY`/`OPENCODE_ZEN_API_KEY` credential path, and the linked split-key PR is closed unmerged. The request remains useful but needs maintainer approval for public env-var precedence and shared-key compatibility before another implementation attempt.

## What This Changes

Request support for provider-specific `OPENCODE_ZEN_API_KEY` and `OPENCODE_GO_API_KEY` environment variables while preserving `OPENCODE_API_KEY` fallback for OpenCode Zen and Go.

## Best Possible Solution

Maintainers should decide the public credential model, then land an additive split-key implementation that preserves `OPENCODE_API_KEY` shared fallback/profile compatibility and updates provider metadata, auth lookup, docs, and focused tests together.

## Reproduction Assessment

Yes, from source inspection: setting only `OPENCODE_GO_API_KEY` is not a supported current-main path because the Go provider runtime and manifest do not advertise or resolve that env var. I did not run a live OpenCode call in this read-only review.

## Solution Assessment

Unclear as a maintainer decision: provider-specific env vars with shared fallback are a plausible narrow shape, but the best solution must preserve the current shared-key behavior or explicitly approve and document the upgrade impact.
