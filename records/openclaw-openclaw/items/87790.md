## Summary

Keep open: current main and v2026.6.9 still use the shared `OPENCODE_API_KEY`/`OPENCODE_ZEN_API_KEY` path for both OpenCode catalogs, while the only split-key PR closed unmerged. Because this changes credential precedence and shared profile fan-out, it needs maintainer product review before another fix attempt.

## What This Changes

Requests separate primary OpenCode Zen and Go API-key environment variables with shared `OPENCODE_API_KEY` fallback.

## Best Possible Solution

Land an additive plugin-local split-key design only after maintainers choose credential precedence and compatibility rules, preserving `OPENCODE_API_KEY` fallback/profile fan-out or documenting and migrating any change.

## Reproduction Assessment

Yes. Source inspection shows `OPENCODE_GO_API_KEY` is absent from the Go provider registration, manifest setup metadata, current docs, and legacy env-key map, so setting only that key is not a supported current-main path.

## Solution Assessment

Unclear until maintainers decide the public credential model. Provider-specific keys with shared fallback are plausible, but the best fix must preserve shared-key behavior or explicitly document and approve the upgrade impact.
