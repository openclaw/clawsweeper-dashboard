## Summary

Keep open: current main and the latest release still do not support `OPENCODE_GO_API_KEY` as the OpenCode Go provider auth env var, and the issue-linked implementation PR is closed unmerged. The request is useful, but it changes public credential precedence and shared-profile behavior, so it needs maintainer product approval before another fix attempt.

## What This Changes

The issue requests provider-specific OpenCode Zen and Go API-key environment variables, with `OPENCODE_API_KEY` retained as a backward-compatible shared fallback.

## Best Possible Solution

Maintainers should decide the OpenCode credential model, then land an additive plugin-local split-key implementation that preserves shared-key fallback/profile compatibility and updates provider metadata, auth lookup, docs, and focused tests together.

## Reproduction Assessment

Yes, from source inspection: setting only `OPENCODE_GO_API_KEY` is not a supported current-main path because the Go provider runtime and manifest do not advertise or resolve that env var. I did not run a live OpenCode setup flow because this review is read-only.

## Solution Assessment

Unclear as an implementation direction until maintainers decide the public credential model. Provider-specific keys with shared fallback are plausible, but the best fix must preserve current shared-key behavior or explicitly document and approve the upgrade impact.
