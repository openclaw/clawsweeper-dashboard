## Summary

Keep open: current main and the latest release still use shared OpenCode credential metadata for Zen and Go, `OPENCODE_GO_API_KEY` is not part of the normal Go provider env lookup, and the linked split-key PR is closed unmerged. The remaining work is useful but needs maintainer approval for public env-var precedence and shared-key compatibility before another implementation attempt.

## What This Changes

The issue requests provider-specific `OPENCODE_ZEN_API_KEY` and `OPENCODE_GO_API_KEY` env vars for OpenCode Zen and Go while retaining shared `OPENCODE_API_KEY` fallback.

## Best Possible Solution

Maintain a single approved OpenCode credential contract, then land an additive provider-owned implementation that preserves `OPENCODE_API_KEY` fallback/profile fan-out or documents and tests any deliberate migration.

## Reproduction Assessment

Yes, from source inspection: setting only `OPENCODE_GO_API_KEY` is not a supported current-main path because the Go provider runtime and manifest do not advertise that env var. I did not run live provider auth because this read-only review does not use secrets.

## Solution Assessment

Unclear as a product decision: provider-specific env vars with shared fallback are a plausible narrow shape, but current code and docs intentionally present one shared OpenCode key. The best fix needs an approved precedence rule plus coverage for provider metadata, onboarding, auth lookup, docs, and shared-profile compatibility.
