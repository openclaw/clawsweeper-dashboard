## Summary

Keep this PR open, but it is not merge-ready: current main still lacks Telegram HTTP 421 retry handling, and the PR now reaches the strict send predicate, but it lacks required real Telegram/transport proof and does not yet ensure the retry happens on a different/fresh connection as the HTTP 421 contract requires.

## What This Changes

The PR adds HTTP 421/Misdirected Request retry classification to the shared Telegram retry runner and strict Telegram send retry helper, with regression tests.

## Best Possible Solution

Handle Telegram 421 by detecting it in strict outbound send retries and forcing the retry onto a different or fresh Telegram transport connection, with focused tests plus redacted real Telegram or transport-level proof.

## Reproduction Assessment

Yes at source level: current main omits 421 from both the generic retry regex and the strict Telegram send predicate while outbound sends use strictShouldRetry. I did not live-reproduce Telegram's intermittent 421 response path in this read-only review.

## Solution Assessment

No for the current PR as written: adding 421 to the retry predicate reaches the right send path, but HTTP 421 should be retried over a different/fresh connection and the patch does not rotate or close the cached Telegram transport before retrying.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Retry 421 on a different Telegram connection:** `extensions/telegram/src/network-errors.ts:196-198`
  - body: HTTP 421 is a misdirected-connection response; RFC 9110 says clients may retry it over a different connection. This patch only makes `isSafeToRetrySendError` return true, so the strict retry runner can replay through the same cached Telegram fetch/undici dispatcher. In the reported coalesced-connection case, attempts can keep hitting the same misdirected connection and exhaust without delivering; rotate, close, or promote the Telegram transport before retrying 421.
  - confidence: 0.84
