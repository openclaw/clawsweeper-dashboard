## Summary

Keep open: the linked Telegram 421 retry bug is real and this PR is scoped to the right area, but it still lacks required real Telegram proof and the patch appears to retry 421 without guaranteeing the different/fresh connection that 421 requires.

## What This Changes

The PR adds HTTP 421/Misdirected Request matching to the generic Telegram retry policy and strict Telegram send retry predicate, with regression tests for both paths.

## Best Possible Solution

Implement 421 handling as a Telegram transport retry that uses a different or fresh connection for outbound sends, keep the strict non-idempotent safeguards, and add regression coverage plus redacted live Telegram proof.

## Reproduction Assessment

Yes at source level, not by live Telegram: current main routes non-idempotent sends through the strict predicate and that predicate does not classify 421 as retryable. A focused unit repro could inject a 421 grammY/HTTP error into that path, but this read-only review did not execute tests.

## Solution Assessment

No. Adding 421 to the retry predicates is the right direction, but the best fix should also ensure the retry uses a different/fresh Telegram connection and should include real transport proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Retry 421 on a fresh Telegram transport:** `extensions/telegram/src/network-errors.ts:196-198`
  - body: HTTP 421 is only safe to retry over a different connection, but this change only makes the strict send predicate return true. The retry runner will call the same grammY request again through the cached Telegram transport, and the current client fetch wrapper only promotes `forceFallback` for request timeouts, so a misdirected keep-alive connection can be reused until all attempts fail. Please make the 421 path promote/rebuild the Telegram transport, or otherwise guarantee a fresh connection, and cover that behavior.
  - confidence: 0.8
