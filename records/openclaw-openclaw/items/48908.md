## Summary

Keep this PR open, but it is not ready to merge: it targets a valid Telegram message-loss bug, yet the current patch only classifies 421 as retryable and does not ensure the retry uses a different or fresh Telegram connection.

## What This Changes

The PR adds HTTP 421/Misdirected Request matching to the shared retry regex and Telegram strict send retry predicate, with focused tests.

## Best Possible Solution

Revise the PR so 421 detection triggers retry on a different or fresh Telegram transport connection, keep focused regression coverage, and require redacted real Telegram or transport-level proof before merge.

## Reproduction Assessment

Yes at source level: current main omits 421 from both the generic retry regex and the strict Telegram send predicate while outbound sends use strictShouldRetry. I did not live-reproduce Telegram's intermittent 421 response path in this read-only review.

## Solution Assessment

No for the current PR as written: adding 421 to the retry predicate reaches the right send path, but HTTP 421 should be retried over a different or fresh connection and the patch does not rotate or close the cached Telegram transport before retrying.
