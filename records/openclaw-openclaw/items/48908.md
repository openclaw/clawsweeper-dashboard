## What This Changes

The PR adds HTTP 421/Misdirected Request retry matching to the generic channel retry policy and Telegram strict send retry predicate, with regression tests.

## Best Possible Solution

Land a rebased version that proves real Telegram 421 retry behavior, covers both strict send and default retry paths, and preserves the duplicate-send safety boundary.

## Reproduction Assessment

Yes at source level: current main's strict Telegram send path uses isSafeToRetrySendError with strictShouldRetry true, and that predicate only accepts pre-connect codes, so a 421 status/error_code falls through. I did not run a live Telegram network reproduction in this read-only review.

## Solution Assessment

Mostly yes: adding 421 to the strict Telegram send predicate is the right boundary for outbound sends, while the generic retry regex covers non-strict callers. The remaining proof gap is whether the real observed 421 shape is fully covered or needs fetch/dispatcher-level retry handling.
