## Summary

Keep open. Current main still does not retry Telegram HTTP 421/Misdirected Request on either the shared retry matcher or the strict non-idempotent send path, while this PR is a narrow code/test fix for that gap. It should not be closed as cleanup; it needs the small public docs and changelog alignment before maintainer review/merge.

## What This Changes

The PR adds HTTP 421/Misdirected Request recognition to the shared channel retry policy and Telegram safe-send retry predicate, with regression coverage for the default and strict retry paths.

## Best Possible Solution

Land the narrow Telegram 421 retry behavior after adding the retry policy docs and Unreleased changelog entry, keeping the strict send predicate conservative for non-idempotent Telegram sends.

## Reproduction Assessment

Yes. On current main, a Telegram send that raises HTTP 421/Misdirected Request will not match the strict send retry predicate, and strictShouldRetry prevents the shared fallback regex from being OR'd in.

## Solution Assessment

Yes, with small documentation and changelog follow-through. Adding 421 handling to both the shared retry matcher and the Telegram safe-send predicate is the narrow maintainable fix without broadly retrying ambiguous send failures.
