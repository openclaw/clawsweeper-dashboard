## Summary

Keep open. Current main still lacks HTTP 421/Misdirected Request retry coverage in both the shared retry matcher and the strict Telegram send predicate, and this PR addresses that behavior but still needs docs and changelog follow-through before merge.

## What This Changes

The PR adds HTTP 421/Misdirected Request handling to the shared channel retry matcher and Telegram safe-send predicate, with regression coverage for default retries and strict send retries.

## Best Possible Solution

Add the missing retry docs and Unreleased changelog entry, then land the narrow retry fix once the targeted Telegram/retry tests pass.

## Reproduction Assessment

Yes. On current main, a Telegram send that raises HTTP 421/Misdirected Request will not match isSafeToRetrySendError, and strictShouldRetry prevents the shared fallback regex from being OR'd in for non-idempotent sends.

## Solution Assessment

Yes, with docs and changelog follow-through. Adding 421 handling to both the shared retry matcher and the strict Telegram safe-send predicate is narrower than disabling strict mode or broadly retrying ambiguous send failures.
