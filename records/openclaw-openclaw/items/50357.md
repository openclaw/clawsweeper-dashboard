## Summary

Keep open: the Feishu docs problem is still present on current main, but this PR currently introduces inaccurate documentation for permission level, requester context, and result shape, and it overlaps with the open create-with-content direction in #48377.

## What This Changes

The PR edits the Feishu document skill docs to remove `owner_open_id` from create examples, state that `create` is title-only, and add a create-then-write workflow.

## Best Possible Solution

Align the Feishu skill docs with the chosen contract: either correct this docs-only patch for current title-only create behavior, or land coordinated create-with-content support with schema, runtime, tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection on current main shows `create` lacks `content`, returns payload data under `details`, and grants only trusted Feishu requesters `edit` access; copying the PR example would read an undefined `document_id`.

## Solution Assessment

No. A docs-only clarification can be the narrow fix if the current contract is kept, but this patch needs factual corrections and maintainers should first choose between it and #48377's create-with-content direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Describe requester grants as edit access:** `extensions/feishu/skills/feishu-doc/SKILL.md:61`
  - body: The new bullet says the requester gets `full_access`, but `createDoc()` grants `perm: "edit"` and returns `requester_perm_type: "edit"`. This would mislead users about the sharing controls available after document creation.

  - confidence: 0.91

  - body: The example uses `result.document_id`, but tool results wrap payloads under `details` and the tests read `result.details.document_id`. Copying this workflow would pass `undefined` as the follow-up `doc_token`.
  - confidence: 0.96
