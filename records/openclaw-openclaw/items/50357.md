## Summary

Keep open. The current Feishu skill docs gap is real, but this PR is not mergeable as written because it overstates requester permissions, omits the trusted Feishu requester limitation, and uses the wrong result path while #48377 remains an alternate behavior direction.

## What This Changes

The PR updates the Feishu document skill docs to remove `owner_open_id` from create examples, state that `create` only creates a titled document, and add a create-then-write workflow.

## Best Possible Solution

Align the Feishu skill docs with the chosen contract: either correct this docs-only PR to match current runtime, or land a coordinated create-with-content change and document that behavior instead.

## Reproduction Assessment

Yes. Source inspection on current main shows the create schema lacks `content`, runtime dispatch calls `createDoc()` only, and tests assert `details.document_id`, `edit` permission, and trusted-context grants.

## Solution Assessment

No. The docs-only direction is useful for current main, but this patch is not the best fix until its examples and permission wording match the runtime and maintainers settle the competing create-with-content direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Describe requester grants as edit access:** `extensions/feishu/skills/feishu-doc/SKILL.md:61`
  - body: The new bullet says the requester gets `full_access`, but `createDoc()` grants `perm: "edit"` and returns `requester_perm_type: "edit"`. This would mislead users about the sharing controls available after document creation.
  - confidence: 0.97
- **[P2] Qualify grants to trusted Feishu requester context:** `extensions/feishu/skills/feishu-doc/SKILL.md:61`
  - body: This wording reads as if create always grants requester access, but the runtime only derives a requester open id from Feishu-channel context. Calls from other contexts skip the grant, so the docs need to name that limitation.
  - confidence: 0.91
- **[P2] Read the created token from details:** `extensions/feishu/skills/feishu-doc/SKILL.md:70`
  - body: The example uses `result.document_id`, but tool results wrap payloads under `details` and the tests read `result.details.document_id`. Copying this workflow would pass `undefined` as the follow-up `doc_token`.
  - confidence: 0.96
