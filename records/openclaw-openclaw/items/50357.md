## Summary

Keep this PR open. Current main still has the Feishu skill documentation gap, but the proposed docs change is not merge-ready because it overstates requester permissions, omits the trusted-Feishu-requester limitation, and gives an invalid result access path in the example.

## What This Changes

The PR updates `extensions/feishu/skills/feishu-doc/SKILL.md` to remove `owner_open_id` from create examples, state that create does not accept content, and add a create-then-write workflow example.

## Best Possible Solution

Revise the Feishu skill docs so they match the shipped contract: create accepts title/folder/grant options only, content is added with write or append unless #48377 lands, requester grants require trusted Feishu channel context, permission is `edit`, and examples read the created token from `result.details.document_id`.

## Reproduction Assessment

Yes. The mismatch is reproducible by comparing the PR patch to current main: schema/runtime/tests show `create` has no `content`, returns payload under `details`, grants `edit`, and skips grants without trusted Feishu requester context.

## Solution Assessment

No. The PR is aimed at the right docs gap, but as written it is not the best fix because it documents `full_access`, unconditional requester grants, and `result.document_id`; the safer solution is a narrower docs correction aligned with the current runtime and tests, coordinated with #48377.

## Review Findings

Overall confidence: 0.93

Full review comments:

- **[P2] Describe the requester grant as edit access:** `extensions/feishu/skills/feishu-doc/SKILL.md:61`
  - body: The new bullet says the requester gets `full_access`, but `createDoc()` grants `perm: "edit"` and returns `requester_perm_type: "edit"`. This would give agents the wrong expectation about the access level after document creation.

  - confidence: 0.91
- **[P2] Read the created document id from details:** `extensions/feishu/skills/feishu-doc/SKILL.md:70`
  - body: The workflow example uses `result.document_id`, but the tool wrapper returns payloads under `details`. As written, copied code would pass `undefined` as the follow-up `write` token; use `result.details.document_id`.
  - confidence: 0.96
