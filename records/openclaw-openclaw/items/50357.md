## Summary

Keep open. Current main still has the Feishu skill documentation gap this PR targets, but the patch should not merge as-is because it adds inaccurate requester-permission wording and an example that reads the created document id from the wrong result shape. The open behavior-change PR #48377 also needs maintainer coordination before settling the public docs direction.

## What This Changes

The PR updates `extensions/feishu/skills/feishu-doc/SKILL.md` to remove `owner_open_id` from create examples, document that `create` does not write content, and add a create-then-write workflow example.

## Best Possible Solution

Keep the PR open until maintainers choose the Feishu contract direction. If current runtime remains, revise the skill docs to remove `owner_open_id`, state that create accepts title/folder/grant options only, say trusted Feishu requester grants are `edit` permission when a trusted requester identity exists, and use `result.details.document_id` in examples; if #48377 lands, document the new create-with-content behavior instead.

## Reproduction Assessment

Yes. The mismatch is reproducible from source without a live Feishu account by comparing the PR patch to current `doc-schema.ts`, `docx.ts`, and `docx.test.ts`, which pin no create `content`, `details.document_id`, trusted-context-only grants, and `edit` permission.

## Solution Assessment

No. The PR addresses a valid docs gap, but as written it is not the best fix because it overstates requester permissions, omits the trusted-context limitation, and shows the wrong result path; the safer solution is a docs correction aligned with current runtime or a coordinated update after #48377.
