## Summary

Keep open: this is a fresh, concrete report against current memory indexing, and current main still routes `openai-compatible` memory batches through the same batch-start, retry/split, and guarded `/embeddings` path without evidence that the no-completion hang is already fixed or owned by a safer canonical item.

## What This Changes

The issue reports that `openclaw memory status --index --agent main --verbose` hangs with OpenAI-compatible embeddings, repeatedly logging batch starts while the index stays dirty at zero chunks.

## Best Possible Solution

Keep this issue open and first build a deterministic repro around `openai-compatible` memory indexing that distinguishes true hung fetches, provider-limit retries/splits, slow normal indexing, and missing progress/completion logging before changing batching or timeout policy.

## Reproduction Assessment

Partially: the reported command maps to the documented reindex path and current main contains the exact batch-start call site. The external endpoint hang itself was not reproduced live, so confidence is source-level rather than high-confidence runtime reproduction.

## Solution Assessment
