## Label Justifications

- `P1`: The report describes a valid macOS Gateway being SIGTERM'd during an active assistant/channel turn, which is a broken live agent workflow.
- `impact:message-loss`: The unexpected Gateway restart can interrupt in-flight app-server/channel work and create a silent delivery gap.
- `impact:session-state`: The issue affects active assistant turn continuity and restart recovery state after the Gateway is killed.
- `impact:crash-loop`: PortGuardian terminates and launchd respawns the Gateway, making this a process-availability/restart problem.

## Summary

Keep open: current main and latest release still use the local PortGuardian allowlist that misses the reported `node .../openclaw/dist/index.js gateway --port 18789` launchd shape, and an open fixing PR already references this issue with closing syntax.

## What This Changes

The issue requests that macOS PortGuardian preserve launchd-managed OpenClaw Gateway processes whose listener command is a runtime like Node but whose full command line runs an OpenClaw entrypoint with the `gateway` subcommand.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94492, or an equivalent narrow PortGuardian matcher and regression test, then close this issue as fixed.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main's local-mode matcher returns false for the reported Node full-command shape while the macOS command resolver can legitimately produce that shape. I did not run a live macOS launchd reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to extend `PortGuardian.isExpected()` to recognize canonical OpenClaw entrypoints plus the `gateway` subcommand, with conservative negative tests; the open fixing PR follows that shape.
