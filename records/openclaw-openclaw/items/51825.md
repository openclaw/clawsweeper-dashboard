- `impact:session-state`: The issue is about TUI session/run activity state not reflecting actual in-flight agent work.

## Summary

Keep this open. Current main and the latest release still drop unknown-run TUI agent lifecycle/tool events before session matching, and the focused linked implementation path is still an open, unmerged PR.

## What This Changes

The issue asks for the TUI to show a visible activity indicator when same-session work is triggered externally by bridge notifications, webhooks, or cron rather than by a TUI-submitted message.

## Best Possible Solution

TUI should adopt same-session lifecycle/tool events using the gateway-provided sessionKey before chat registration, preserve cross-session filtering, and add focused regression coverage for lifecycle/tool events arriving before chat deltas.

## Reproduction Assessment

Yes, from source inspection: a same-session agent lifecycle start for a run that is not already active, pending, tracked, or finalized returns at the known-run guard before the running status path executes. I did not run a live bridge/webhook reproduction in this read-only review.

## Solution Assessment

Yes in direction: a sessionKey-aware adoption path in the TUI event handler is the narrowest maintainable fix. The broader session-activity API tracked at https://github.com/openclaw/openclaw/issues/39127 is complementary, not required to fix this TUI regression.
