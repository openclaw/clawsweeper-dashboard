- `impact:session-state`: The issue is about the TUI session view drifting from actual agent run state during system-injected work.

## Summary

Keep this open. Current main and the latest release still drop same-session TUI agent lifecycle/tool events when they arrive before a chat event has registered the run, and the focused fix candidate remains open at https://github.com/openclaw/openclaw/pull/66400.

## What This Changes

The issue asks for the TUI to show running/activity state for system-injected agent work such as bridge notifications, webhooks, and cron triggers.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/66400 or an equivalent TUI handler change that pre-registers same-session agent events before unknown-run filtering while preserving cross-session, BTW, and optimistic-message protections.

## Reproduction Assessment

Yes, at source level. Deliver a current-session `agent` lifecycle start or tool start event with a new runId before any `chat` event registers that run, and current main returns at the unknown-run guard before updating activity or rendering tool state.

## Solution Assessment

Yes. Using the gateway-provided `sessionKey` to adopt same-session early agent events in the TUI is the narrowest maintainable fix and avoids a new config surface or gateway rewrite.
