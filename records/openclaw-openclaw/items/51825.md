## Summary

Keep open: current main and the latest release still ignore unknown `agent` lifecycle-start events in the TUI, so externally started work can remain invisible until a `chat` delta or final arrives.

## What This Changes

TUI should show an activity indicator for agent work triggered by system-injected events such as bridge notifications, webhooks, or cron events.

## Best Possible Solution

Teach the TUI to claim same-session lifecycle-start events as active runs, then clear them on terminal lifecycle or chat events without regressing optimistic local-send and concurrent-run behavior.

## Reproduction Assessment

Yes, at source level: the TUI ignores unknown `agent` lifecycle-start events and only claims a run after a matching `chat` event. I did not run a live bridge/webhook/TUI reproduction in this read-only pass.

## Solution Assessment

Yes, the best fix appears to be in the TUI event handler rather than bridge/webhook-specific code, because Gateway already broadcasts session-owned lifecycle events. The repair should be transport-agnostic and covered by a focused TUI handler regression test.

## Repair Work Prompt

Fix the TUI activity indicator for system-injected or externally started agent runs. Current main claims a run from `chat` events in `src/tui/tui-event-handlers.ts`, but `handleAgentEvent` returns for an unknown lifecycle `phase: "start"` even when the event carries the current session key, so bridge/webhook/cron-started work can look idle until a chat delta/final arrives. Add the smallest TUI-side run-claiming path for same-session lifecycle start events, preserve existing optimistic local send behavior and concurrent-run protections, and add regression coverage in `src/tui/tui-event-handlers.test.ts`. Do not add config flags or bridge-specific special cases; keep the fix transport-agnostic. Suggested validation: `node scripts/run-vitest.mjs src/tui/tui-event-handlers.test.ts`. No CHANGELOG.md edit.
