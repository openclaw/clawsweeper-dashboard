## Summary

Keep this open. Current main still drops same-session agent lifecycle/tool events when they arrive before any chat event has registered the run, and https://github.com/openclaw/openclaw/pull/66400 remains the focused open implementation candidate.

## What This Changes

Show TUI activity for agent work started by system-injected events such as bridge notifications, webhooks, or cron triggers.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/66400 or an equivalent TUI handler change that pre-registers same-session agent events before unknown-run filtering while preserving cross-session, BTW, and optimistic-message protections.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: deliver a current-session `agent` lifecycle `start` or tool `start` event before any `chat` event registers that run ID, and current main returns at the unknown-run guard before setting activity or rendering a tool card.

## Solution Assessment

Is this the best way to solve the issue? Yes: using the gateway-provided `sessionKey` to adopt current-session agent events in the TUI before filtering is narrower than a gateway rewrite or a new configuration surface.
