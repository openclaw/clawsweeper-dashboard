## Summary

Keep this open. Current main still ignores early agent lifecycle/tool events for unknown run IDs before a chat event registers the run, and #66400 remains the focused open implementation candidate.

## What This Changes

The TUI should show activity for agent work started by bridge notifications, webhooks, or cron instead of only TUI-originated messages.

## Best Possible Solution

Land #66400 or an equivalent narrow TUI fix that adopts current-session agent events before the known-run guard while preserving cross-session and `/btw` isolation.

## Reproduction Assessment

Yes. A high-confidence source path is to deliver a current-session lifecycle `start` or tool `start` agent event before any chat event registers that run ID; current main returns at the unknown-run guard and leaves the TUI idle.

## Solution Assessment

Yes. The narrowest maintainable fix is to use the gateway-provided `sessionKey` to adopt current-session run IDs in the TUI before filtering, not a gateway rewrite or new configuration option.
