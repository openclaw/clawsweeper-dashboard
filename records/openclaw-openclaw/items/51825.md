## Summary

Keep open. Current main has the TUI status loader and direct-send busy states, but a system-injected run can arrive first as an `agent` lifecycle event with a `sessionKey` and no known `activeChatRunId`; the TUI filters that event out before it can set `running`, so the reported idle-looking gap remains.

## What This Changes

The issue requests a TUI activity indicator for bridge, webhook, and cron-triggered agent work that was not initiated from the TUI.

## Best Possible Solution

Make gateway-mode TUI bind current-session lifecycle starts to an active run and clear on terminal events, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, source-reproducible: a current-session lifecycle `agent` event for an untracked system run has enough `sessionKey` data, but `handleAgentEvent` returns before setting `running` when `activeChatRunId` is unset. I did not execute an end-to-end live bridge/webhook/cron scenario.

## Solution Assessment

Yes. The narrow fix is to let the TUI accept session-keyed lifecycle starts for the current session, rather than adding bridge-, webhook-, or cron-specific handling.

Preserve existing cross-session filtering, local BTW behavior, user-send optimistic state, and watchdog semantics.
