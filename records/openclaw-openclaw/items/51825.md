## Summary

Keep this open. Current main still drops the first session-matching agent lifecycle/tool event for an externally started run because TUI only accepts agent events after the run is already active, recorded in `sessionRuns`, or finalized; the only current adoption path is a matching chat event. The supplied context also shows #66400 as an open, unmerged implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for the TUI to show running/elapsed activity for agent work triggered by bridge notifications, webhooks, or cron, not only work started from the TUI input.

## Best Possible Solution

Land #66400 or an equivalent narrow TUI fix that accepts session-matching agent lifecycle/tool events before the known-run guard, records the run as active for the current session, preserves cross-session and /btw isolation, and adds regression coverage for lifecycle/tool events arriving before the first chat delta.

## Reproduction Assessment

Yes. A unit-level reproduction is clear: call `handleAgentEvent` with a new `runId`, `stream: "lifecycle"`, `data.phase: "start"`, and the current session key before any chat event; current main returns at the unknown-run guard and never sets activity to running.

## Solution Assessment

Yes, the best direction is the targeted TUI adoption fix described by #66400 rather than a gateway rewrite, because gateway already emits session keys and session-scoped lifecycle/tool signals.
