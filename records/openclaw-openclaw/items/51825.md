- `P2`: This is a valid user-visible TUI run-state bug with limited blast radius and a focused open fix candidate.
- `impact:session-state`: The TUI session view can drift from actual agent run state during externally injected work.

## Summary

Keep this open. Current main and the latest release still return from the TUI agent-event handler when a same-session lifecycle or tool event arrives with a new run ID before any chat event registers that run, and the focused open fix path remains https://github.com/openclaw/openclaw/pull/66400.

## What This Changes

The issue reports that OpenClaw TUI appears idle when agent work starts from bridge, webhook, or cron-injected events instead of a local TUI send.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/66400 or an equivalent TUI handler change that adopts same-session early agent lifecycle and tool events using gateway `sessionKey` while preserving existing guards.

## Reproduction Assessment

Yes, at source level. Deliver a current-session `agent` lifecycle start or tool start event with a new `runId` before any `chat` event registers that run, and current main returns at the unknown-run guard before updating activity.

## Solution Assessment

Yes. Using the gateway-provided `sessionKey` to adopt same-session early agent events in the TUI is the narrowest maintainable fix and avoids a new config surface or gateway rewrite.
