- `impact:session-state`: The issue is about the TUI session view drifting from actual agent run state during externally injected work.

## Summary

Keep this open. Current `main` and `v2026.5.20` still drop same-session TUI agent lifecycle/tool events for new run IDs before a chat event registers the run, and the focused open fix path remains https://github.com/openclaw/openclaw/pull/66400.

## What This Changes

The issue requests that the TUI show running and elapsed activity for agent work started by bridge, webhook, or cron events instead of only TUI-originated messages.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/66400 or an equivalent TUI handler change that uses gateway `sessionKey` to adopt same-session early agent events while preserving cross-session, BTW, and optimistic-message guards.

## Reproduction Assessment

Yes, at source level. Deliver a current-session `agent` lifecycle start or tool start event with a new `runId` before any `chat` event registers that run, and current main returns at the unknown-run guard before updating activity.

## Solution Assessment

Yes. Using gateway-provided session identity to adopt same-session early agent events in the TUI is the narrowest maintainable fix and avoids a new config surface or gateway rewrite.
