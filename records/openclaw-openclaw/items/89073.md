## Summary

Keep open. Current main still lacks a duplicate running-session lookup for repeated background-capable `exec` calls, and this PR is a focused implementation with sufficient real runtime proof; the remaining blocker is maintainer acceptance of the compatibility tradeoff for intentional identical concurrent background exec launches.

## What This Changes

The PR adds execution-identity hashing and a scoped running-session lookup so duplicate background/yield `exec` retries return the existing process session, with registry/runtime tests for reuse and non-reuse cases.

## Best Possible Solution

Merge after maintainers explicitly accept exact duplicate background exec coalescing as the default, or add a documented duplicate-launch escape hatch if independent identical launches must remain supported.

## Reproduction Assessment

Yes, as a source-level reproduction: current main sends repeated background-capable `exec` calls through `runExecProcess`, which creates a fresh registered session before any reuse decision. I did not run a live Xiaomi/MiMo replay in this read-only review.

## Solution Assessment

Yes, this is the right code layer for the reported bug because it fixes the generic exec/process runtime boundary using exact resolved execution identity rather than provider-specific policy or fuzzy command matching. The unresolved part is maintainer acceptance of the compatibility tradeoff.
