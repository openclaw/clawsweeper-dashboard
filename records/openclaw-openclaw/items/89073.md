## Summary

Keep open: the PR addresses a real duplicate-background-session fanout, but the new reuse key can collapse non-identical exec calls that share command/cwd/scope while changing spawn-affecting options such as env, PTY, or timeout.

## What This Changes

The PR adds a process-registry lookup so repeated scoped background exec calls with the same command and cwd return the existing running session instead of spawning another child process.

## Best Possible Solution

Keep the duplicate-session reuse idea, but match on the resolved spawn-affecting execution identity or skip reuse when per-call execution options differ, with regression tests for env, PTY, and timeout drift.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR returns an existing session before runExecProcess receives documented spawn-affecting params such as env, pty, and timeout. I did not run tests because this was a read-only review.

## Solution Assessment

No as written; the process registry is a plausible owner boundary, but the match key is too broad. The maintainable fix is to reuse only truly identical resolved exec requests or conservatively start a new session when execution options differ.
