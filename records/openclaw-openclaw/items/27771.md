## Summary

Keep open. The requested Signal config-directory feature is valid, but this PR is not merge-ready against current main: it wires the runtime change into obsolete `src/signal` files while the shipped Signal channel now runs from `extensions/signal`, and current main still lacks `configPath` in the strict config schema, generated metadata, docs, account resolution, monitor propagation, and daemon argv builder. The author has also said they cannot continue the branch, so this is a narrow replacement-PR candidate rather than a close candidate.

## What This Changes

The PR adds optional Signal `configPath` config/schema/help/docs entries and wires legacy Signal monitor/daemon code to pass a trimmed `--config` argument when autostarting `signal-cli`.

## Best Possible Solution

Land a narrow replacement patch that implements optional `channels.signal.configPath` and `channels.signal.accounts.<id>.configPath` in the bundled Signal plugin end to end, passing the trimmed value as the global `signal-cli --config` option before `daemon` while preserving existing defaults and multi-account behavior.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: a config containing `channels.signal.configPath` is not part of the strict Signal schema/types, and the current Signal monitor/daemon has no path for that value to reach the `signal-cli` argv.

## Solution Assessment

No. The feature direction is the right solution to the user problem, but this PR applies it to obsolete runtime files; the maintainable fix is a small replacement against `extensions/signal` plus the current config/docs/generated surfaces.
