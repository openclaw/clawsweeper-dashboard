## Summary

Keep open: current main has plugin session lifecycle hooks and cron-backed scheduled turns, but it does not expose a cron event schedule or supported session archive/rotation trigger that can fire an agent turn at the archive boundary.

## What This Changes

The issue requests a session archive or rotation lifecycle event that can trigger cron-style automation or an agent turn when a session naturally rolls over.

## Best Possible Solution

Define one supported contract for archive-boundary automation: either add a generic session lifecycle event that cron can consume, or document and extend the plugin hook/scheduled-turn path so post-session automation can run with explicit security and timeout rules.

## Reproduction Assessment

Yes, at source level: current main's cron contract accepts only time-based schedules, while the existing `session_end` lifecycle signal is a plugin hook and not a cron-triggering event schedule. I did not run a live gateway scenario because the missing behavior is visible in the current source contracts.

## Solution Assessment

Unclear. The requested event schedule is plausible, but the existing plugin `session_end` hook and bundled-only scheduled-turn API mean maintainers should choose the supported lifecycle/cron boundary before code is queued.
