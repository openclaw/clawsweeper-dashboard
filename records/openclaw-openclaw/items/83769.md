## Summary

Keep open: the PR is aimed at the right bug, but it hardcodes a small Discord-only parent-path list in the generic config CLI and leaves supported numeric-string record paths under the existing array heuristic. The supplied proof is also mock-only test output, so it still needs real CLI proof before merge.

## What This Changes

The PR updates `openclaw config set` path writing so known Discord record parents keep numeric-looking child segments as object keys, with regression coverage for `channels.discord.guilds.<snowflake>.requireMention`.

## Best Possible Solution

Drive missing-container selection from config schema or schema-lookup metadata, preserve true array paths, and require real CLI proof against a temp config file before merge.

## Reproduction Assessment

Yes at source level: current `setAtPath` creates array parents for all-digit next segments in `config set`, while current schemas define record/object paths such as Discord `guilds` and Telegram `dms`. I did not run tests or the CLI because this review was required to keep the checkout read-only.

## Solution Assessment

No: the PR fixes the shown top-level Discord guild path, but the hardcoded allowlist is not the narrow maintainable solution for schema-backed record paths. The safer fix is to derive object-versus-array container creation from the config schema or existing schema lookup surface and cover representative records plus true arrays.
