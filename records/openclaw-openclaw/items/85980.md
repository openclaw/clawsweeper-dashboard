## Summary

Keep open: current main already has SecretRef/env-ref coverage for static `api_key`/`token` auth profiles, but the requested `access`/`refresh` behavior remains intentionally excluded for OAuth credentials and the missing-env fallback would conflict with fail-fast SecretRef semantics.

## What This Changes

The issue asks for auth profile token-like fields to support `${ENV_VAR}` references so secrets can be kept out of `auth-profiles.json`.

## Best Possible Solution

Keep static credentials on the existing SecretRef path, and have maintainers decide whether OAuth `access`/`refresh` should remain excluded or gain a carefully documented SecretRef-compatible model.

## Reproduction Assessment

Yes, by source inspection: static `token` env refs are handled by the secrets runtime, while OAuth `access`/`refresh` refs are explicitly rejected by policy. I did not run tests because this review is read-only.

## Solution Assessment

No, not as proposed: falling back to literal strings on missing env vars is weaker than the current SecretRef fail-fast contract. The maintainable path is to extend or reaffirm the SecretRef policy rather than add a parallel loader substitution rule.

## Risks / Open Questions

- Adding a separate auth-loader `${ENV_VAR}` path could duplicate or bypass the existing SecretRef activation model for static credentials.
- Allowing env refs for OAuth `access` and `refresh` would reverse an explicit policy that treats OAuth material as runtime-mutable rather than read-only SecretRef-backed config.
- The proposed fallback to the literal string when an env var is missing conflicts with current fail-fast behavior and could accidentally send `${ENV_VAR}` as a credential.
