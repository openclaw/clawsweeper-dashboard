## Summary

Keep open: current main and v2026.6.8 already support env SecretRefs for static `api_key` and `token` auth profiles, but OAuth `access`/`refresh` SecretRefs remain explicitly unsupported and need maintainer security/product direction.

## What This Changes

The issue requests `${ENV_VAR}` references for `token`, `access`, and `refresh` credential fields in agent auth-profile stores.

## Best Possible Solution

Keep the existing static `keyRef`/`tokenRef` path, and have maintainers either explicitly keep OAuth material out of SecretRefs or design a documented OAuth-compatible SecretRef model with fail-closed resolution and refresh persistence semantics.

## Reproduction Assessment

Yes, at source level: static `${ENV}` `api_key` and `token` profiles resolve, while OAuth-mode or `type: "oauth"` SecretRef-like credential material is rejected by policy. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as proposed: adding loader substitution with literal fallback is weaker than the existing SecretRef contract. The maintainable path is a maintainer-approved OAuth SecretRef design or an explicit decision to keep OAuth access/refresh material unsupported.
