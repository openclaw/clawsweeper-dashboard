## Summary

Keep open: current main and v2026.6.8 already support SecretRef/env references for static `api_key` and `token` auth profiles, but OAuth `access`/`refresh` refs remain explicitly unsupported and changing that mutable-credential boundary needs maintainer security/product review.

## What This Changes

Request support for `${ENV_VAR}` or external secret references in `auth-profiles.json` token, access, and refresh fields to reduce plaintext credential storage.

## Best Possible Solution

Keep static credentials on the existing `keyRef`/`tokenRef` SecretRef path, and make an explicit maintainer decision on whether OAuth `access`/`refresh` should gain a fail-closed SecretRef-compatible model with documented refresh persistence semantics.

## Reproduction Assessment

Yes. Source inspection shows static `api_key`/`token` env refs resolve through SecretRef handling, while OAuth `access`/`refresh` refs are rejected by current policy; I did not run tests because this pass is read-only.

## Solution Assessment

No, not as proposed. Adding literal env substitution fallback in the loader is weaker than the existing SecretRef contract; the maintainable path is either reaffirming the OAuth exclusion or extending SecretRef with explicit OAuth refresh/writeback rules.
