## Summary

Keep open: current main and the latest release already support SecretRef/env references for static `api_key` and `token` auth profiles, but the requested `access`/`refresh` OAuth behavior remains explicitly unsupported and needs maintainer security/product judgment.

## What This Changes

The issue requests `${ENV_VAR}` resolution for `token`, `access`, and `refresh` fields in auth profile credential storage to reduce plaintext credentials on disk.

## Best Possible Solution

Keep static credentials on the existing SecretRef path, and have maintainers decide whether OAuth `access`/`refresh` should gain a documented SecretRef-compatible model with fail-closed resolution and explicit refresh persistence semantics.

## Reproduction Assessment

Yes: source inspection shows static `api_key`/`token` env refs resolve through SecretRef handling, while OAuth `access`/`refresh` refs are rejected by policy. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as proposed: adding a parallel loader substitution with literal fallback is weaker than the existing SecretRef contract. The maintainable path is either to reaffirm the current OAuth exclusion or extend the SecretRef model with explicit OAuth refresh/writeback rules.
