## Summary

Keep open: current main and v2026.6.9 already support SecretRef/env references for static `api_key` and `token` auth profiles, but OAuth `access`/`refresh` remains explicitly unsupported and needs a maintainer security/product decision before changing mutable credential semantics.

## What This Changes

The issue requests environment-variable references for `token`, `access`, and `refresh` fields in agent auth-profile token storage so credentials do not need to remain plaintext on disk.

## Best Possible Solution

Keep static credentials on the existing `keyRef`/`tokenRef` SecretRef path, and make an explicit maintainer security/product decision on whether OAuth access/refresh should get a fail-closed SecretRef-compatible model with documented refresh persistence semantics.

## Reproduction Assessment

Yes. Source inspection shows static `api_key`/`token` env refs resolve through SecretRef handling, while OAuth `access`/`refresh` refs are rejected by current policy; I did not run tests because this pass is read-only.

## Solution Assessment

No, not as originally proposed. Adding literal env substitution fallback in the loader is weaker than the existing SecretRef contract; the maintainable path is either reaffirming the OAuth exclusion or extending SecretRef with explicit OAuth refresh/writeback rules.
