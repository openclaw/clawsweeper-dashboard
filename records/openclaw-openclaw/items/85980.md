- `impact:auth-provider`: The affected behavior is provider credential loading, SecretRef resolution, and OAuth/token auth-profile handling.

## Summary

Keep open: current main and v2026.6.9 already support env/SecretRef resolution for static `api_key` and `token` auth profiles, but OAuth `access`/`refresh` remains explicitly unsupported and needs a maintainer security/product decision before changing mutable credential ownership.

## What This Changes

Support env-var or SecretRef-style references in `auth-profiles.json` credential token fields, including OAuth `access`/`refresh`, to reduce plaintext credential storage.

## Best Possible Solution

Keep static credentials on the existing `keyRef`/`tokenRef` path; if OAuth refs are desired, design a fail-closed OAuth-specific SecretRef/writeback contract with docs and regression coverage before implementation.

## Reproduction Assessment

Yes. Source inspection shows static `api_key`/`token` env refs resolve through SecretRef handling, while OAuth `access`/`refresh` refs are rejected by current policy; I did not run tests because this review is read-only.

## Solution Assessment

No, not as originally proposed. Literal loader substitution with fallback is weaker than the existing fail-closed SecretRef contract; the maintainable path is either reaffirming the OAuth exclusion or extending SecretRef with explicit OAuth refresh/writeback rules.
