## Summary

Keep this PR open. Current main still lacks the intended Windows `windowsHide` propagation for the shared direct and shell-fallback spawn resolution paths, and the production hunk is still the right narrow direction, but the branch is dirty against main because its main regression coverage lives in an ACPX test file that has since been removed.

## What This Changes

The PR adds `windowsHide: true` to the plugin SDK Win32 direct and shell-fallback spawn resolutions, plus ACPX regression coverage and test-only hidden-spawn options.

## Best Possible Solution

Rebase the PR, keep the shared `src/plugin-sdk/windows-spawn.ts` production change, move the direct `.exe` and shell-fallback regression coverage into `src/plugin-sdk/windows-spawn.test.ts`, preserve non-Windows `undefined` behavior and fail-closed unresolved-wrapper policy, then handle the raw ACPX MCP proxy console-flash surface separately if it remains in scope.

## Reproduction Assessment

Yes. Current main can be checked by resolving an absolute `.exe` on a simulated Win32 runtime or an opt-in unresolved wrapper shell fallback; the shared helper currently returns/materializes `windowsHide: undefined`, and the existing shell-fallback test asserts that value.

## Solution Assessment

Yes for the production hunk, but not for the current branch state. Setting `windowsHide` in the shared plugin SDK resolver is the narrow maintainable fix because ACP, Codex app-server, Docker, lobster, ACPX, and memory-host wrappers forward that resolved value; the tests need to move to the current SDK test file before merge.
