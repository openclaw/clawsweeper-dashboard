## Summary

Keep open. Current main and the latest release still retain `storeLocks` entries indefinitely, while this PR's identity-checked cleanup is narrow and matches an existing keyed-tail cleanup pattern. I found no functional or security blocker, but the PR should add the required changelog entry before merge.

## What This Changes

The PR exports a test-only cron `storeLocks` accessor, deletes the per-store lock tail when the completing promise is still current, and adds unit coverage for drained and queued lock behavior.

## Best Possible Solution

Ship the identity-checked cleanup with its regression tests and an Unreleased Fixes changelog entry, preserving same-store serialization while pruning only the final current per-store lock entry after the queue drains.

## Reproduction Assessment

Yes. The source-level reproduction is direct: repeated `locked()` calls using unique `state.deps.storePath` values add entries to the module-level `storeLocks` Map, and current main has no code path that removes those keys after settlement.

## Solution Assessment

Yes, with the changelog addition. The proposed identity check is the narrowest maintainable code fix because older queued operations cannot delete a newer tail, future calls already fall back to `Promise.resolve()`, and the same keyed-tail cleanup shape exists in `enqueueKeyedTask()`.
