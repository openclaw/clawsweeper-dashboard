## Summary

Keep open. Current main still has the unbounded cron `storeLocks` Map behavior, and the PR's identity-checked cleanup matches an existing keyed-tail cleanup pattern; the remaining merge blocker is the required changelog entry.

## What This Changes

The PR adds a test-only cron lock accessor, prunes the per-store lock tail after the final queued operation settles, and adds focused unit coverage for drained and queued lock behavior.

## Best Possible Solution

Ship the identity-checked cleanup with its regression tests and a single-line Unreleased Fixes changelog entry, preserving same-store serialization while pruning only the final current per-store tail.

## Reproduction Assessment

Yes. On current main, repeated `locked()` calls with unique `state.deps.storePath` values add entries to the module-level `storeLocks` Map, and there is no code path that removes those keys after settlement.

## Solution Assessment

Yes, with the changelog addition. The proposed identity check is the narrowest maintainable fix because older queued operations cannot delete a newer tail, future calls already fall back to `Promise.resolve()`, and the same keyed-tail cleanup shape exists in `enqueueKeyedTask()`.
