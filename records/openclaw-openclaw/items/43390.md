## Summary

Keep this PR open. Current main still retains each cron `storePath` tail in the module-level `storeLocks` Map with no cleanup path, while the PR is a focused identity-checked cleanup plus regression test and has no definite blocking correctness issue from the diff review.

## What This Changes

The PR adds a test-only accessor for the cron `storeLocks` Map, deletes the per-store entry in `locked()` only when the completing promise is still the current tail, and adds unit coverage for pruning after the final queued operation while retaining the entry during a queued chain.

## Best Possible Solution

Review and ship the identity-checked cleanup with the regression test, preserving same-store serialization while pruning only the final current per-store lock entry after the queue drains.

## Reproduction Assessment

Yes. The source-level reproduction is direct: repeated calls with unique `storePath` values write entries into the module-level `storeLocks` Map, and current main has no path that removes those keys after the operation settles.

## Solution Assessment

Yes. The proposed `finally` identity check is the narrowest maintainable fix because earlier queued operations cannot delete a newer tail, future calls already fall back to a resolved promise, and the same keyed-tail cleanup shape exists in `enqueueKeyedTask()`.
