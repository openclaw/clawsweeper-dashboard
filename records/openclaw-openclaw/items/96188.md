## Summary

Keep this open: current main and v2026.6.10 still lack the Matrix to-device accept-event normalization, and the open linked PR is the active candidate fix for the exact reported failure.

## What This Changes

Matrix SAS verification should tolerate or clearly recover from Element `m.key.verification.accept` to-device events that omit `content.method`.

## Best Possible Solution

Land a narrow Matrix plugin normalization at the Rust-crypto to-device preprocessing boundary, with regression coverage and redacted live Matrix verification proof, then close this issue from the merged fix.

## Reproduction Assessment

Yes for source reproduction: current main lacks the preprocessor normalizer, and matrix-js-sdk 41.6.0 sends to-device accept events through Rust crypto before compatibility events are emitted. I did not run a live Element/Synapse reproduction in this read-only review.

## Solution Assessment

Yes, the active fix direction is the right layer: a higher-level MatrixEvent or verification-manager workaround would run after Rust crypto has already rejected the malformed accept event. The issue should stay open until the linked PR merges or is replaced by an equivalent fix.
