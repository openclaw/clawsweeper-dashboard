## Summary

Keep open: current main still leaves remote generic custom OpenAI-compatible providers such as USTC without streamed usage unless the model explicitly opts into `supportsUsageInStreaming`, and the reporter confirmed that opt-in restores token tracking. Because current tests intentionally avoid broadening the generic custom default, this needs provider-contract or product review before a repo fix.

## What This Changes

The issue reports direct-session token usage being stored as null and displayed as `unknown/<ctx> (?%)` for USTC OpenAI-compatible sessions on v2026.5.18.

## Best Possible Solution

Verify USTC's streaming-usage contract, then add provider-owned compat metadata or clear docs/config guidance for that endpoint without broadening the generic custom-provider default blindly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for a remote custom OpenAI-compatible provider without `compat.supportsUsageInStreaming`; endpoint-specific live reproduction still needs USTC credentials or a captured request/response trace.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the reporter's compat workaround fits current contracts, but a repo fix needs a maintainer decision between provider-specific compat, docs guidance, or a broader fallback for missing usage.
