## Summary

Keep open: the report is plausible on current main because custom OpenAI-compatible providers only request streamed usage when compat says they support it, and direct-session status falls back to unknown when no fresh usage or prompt snapshot reaches the session store.

## What This Changes

The issue asks OpenClaw to persist and display direct-session token usage instead of showing unknown context usage when `totalTokens` is missing for v2026.5.18 direct sessions.

## Best Possible Solution

Verify the USTC streaming-usage contract, then either declare provider-owned streaming usage compatibility for that endpoint or reuse the existing transcript prompt-estimate path in CLI/session status with clear freshness semantics.

## Reproduction Assessment

Source-reproducible, but not live-reproduced: a custom OpenAI-compatible provider without streaming-usage compat will not request `include_usage`, and the direct-session store/status path then has no fresh total to display. A live USTC credentialed run would be needed for high-confidence endpoint-specific reproduction.

## Solution Assessment

Unclear: a blanket core default change is risky because tests intentionally disable streaming usage for generic custom providers. The safer fix needs a maintainer decision between provider-specific compat/plugin metadata, docs/config guidance, or a transcript-estimate fallback for CLI status.
