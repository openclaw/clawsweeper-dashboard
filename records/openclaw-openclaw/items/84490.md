## Summary

Keep open: the reporter's USTC follow-up makes the source path plausible, but current main intentionally leaves streamed usage off for generic custom OpenAI-compatible endpoints unless compatibility metadata opts in, so this needs maintainer direction rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to restore accurate direct-session token totals for a USTC OpenAI-compatible provider that stores `totalTokens: null` and renders `unknown/1049k (?%)`.

## Best Possible Solution

Verify USTC's streaming-usage behavior, then either add provider-owned compatibility metadata for that endpoint or document the explicit model-level compat setting; only add a core status fallback if maintainers accept estimated freshness semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live USTC run was performed, but the source path is clear: a generic custom OpenAI-compatible provider without `compat.supportsUsageInStreaming` will not request `include_usage`, leaving direct session totals unavailable.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the reporter's compat workaround is supported, but deciding whether USTC should be provider-owned metadata, docs guidance, or a broader fallback requires maintainer product judgment.
