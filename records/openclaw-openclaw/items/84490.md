## Summary

Keep open: current main and v2026.6.8 include the shipped estimated-context fallback, but current source still withholds `stream_options.include_usage` for generic custom non-native OpenAI-compatible endpoints unless compatibility is declared, so exact `totalTokens` persistence remains unresolved for the reported USTC-style setup.

## What This Changes

The issue reports direct-session token usage showing `unknown/1049k (?%)` because `totalTokens` is persisted as null for directly managed sessions on v2026.5.18.

## Best Possible Solution

Verify whether the USTC endpoint supports streamed usage, then encode that decision through provider/plugin metadata or explicit docs/config guidance while keeping the shipped estimate fallback as a display mitigation.

## Reproduction Assessment

Yes at source level for generic custom OpenAI-compatible providers without streaming-usage compat: current code does not send `stream_options.include_usage`, so no exact usage reaches session persistence. The USTC-specific endpoint still needs live contract proof.

## Solution Assessment

No, closing this as implemented would be premature: the shipped fallback improves `/status` output but does not persist exact provider totals. The safer solution is a maintainer-approved provider compatibility decision backed by USTC endpoint proof.
