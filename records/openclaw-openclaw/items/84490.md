## Summary

Keep open: current main and v2026.6.6 include a shipped estimated-context fallback that reduces the visible unknown status, but they still do not make a generic custom USTC OpenAI-compatible provider persist real `totalTokens` unless streaming-usage compatibility is explicitly declared.

## What This Changes

The issue reports that direct sessions on v2026.5.18 persist `totalTokens: null` and show unknown context usage for a USTC OpenAI-compatible provider while subagent sessions keep numeric usage.

## Best Possible Solution

Verify whether the USTC endpoint supports streamed usage, then either declare that compatibility through provider/plugin metadata or document the existing model-level compat setting while keeping the shipped estimate fallback as a display mitigation.

## Reproduction Assessment

Yes at source level for generic custom OpenAI-compatible providers without streaming-usage compat: current code will not send `stream_options.include_usage`, so no exact usage reaches session persistence. The USTC-specific path still needs live endpoint proof.

## Solution Assessment

Unclear as a final fix: the shipped estimated fallback is useful but does not persist exact provider totals. The maintainable path depends on verified USTC behavior and a maintainer decision about provider metadata versus docs/config guidance.
