## Summary

Keep open: current main still leaves arbitrary remote OpenAI-compatible providers without streamed usage unless `compat.supportsUsageInStreaming` is set, and the reporter confirmed that explicit compat fixes USTC. The remaining choice is whether USTC should get provider-owned compat metadata/docs or whether session status needs a broader fallback, so this should stay with maintainers rather than close as already implemented.

## What This Changes

Direct sessions using a USTC OpenAI-compatible provider on v2026.5.18 should persist and display numeric `totalTokens` instead of `unknown/1049k (?%)` with null or missing totals.

## Best Possible Solution

Verify USTC's streaming-usage contract, then either add provider-owned compatibility metadata/docs for that endpoint or implement a clearly labeled session-status fallback without changing the default for all generic custom providers.

## Reproduction Assessment

No, not at high endpoint-specific confidence: this review did not run a live USTC credentialed session. The source path is reproducible for any remote custom OpenAI-compatible provider without `supportsUsageInStreaming`, and the reporter's workaround confirms the likely mechanism.

## Solution Assessment

Unclear: the existing compat flag is a narrow workaround, but the repo-level fix needs a maintainer choice between USTC provider metadata, docs/config guidance, or a broader fallback. A blanket default change for all custom providers is not the safest solution because current tests intentionally keep that disabled.
