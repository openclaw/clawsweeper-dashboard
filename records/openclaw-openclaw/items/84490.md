## Summary

Keep open: current main and v2026.5.18 still gate streamed usage for generic custom OpenAI-compatible providers, and the reporter's follow-up confirms that the existing `compat.supportsUsageInStreaming: true` override restores session token tracking for USTC.

## What This Changes

The issue asks OpenClaw to stop direct/main/dashboard/dreaming sessions from persisting missing `totalTokens` and showing `unknown/1049k (?%)` when a custom USTC OpenAI-compatible provider can return streaming usage.

## Best Possible Solution

Verify USTC streaming usage live, then either add a narrow provider compatibility/defaulting path for that endpoint or make an explicit maintainer decision to restore broader OpenAI-compatible usage requests with regression coverage for incompatible providers.

## Reproduction Assessment

Yes for a high-confidence source-level path, but not for a live USTC run here. A custom OpenAI-compatible provider without `compat.supportsUsageInStreaming: true` will not request streamed usage on current main, and the reporter confirmed the override fixes the real setup.

## Solution Assessment

Unclear. The existing config override is a valid workaround, but choosing between USTC-specific provider metadata, documentation, a transcript fallback, or a broader default change needs maintainer product and compatibility judgment.
