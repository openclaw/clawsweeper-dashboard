## Summary

Keep open. Current main has useful partial fixes, especially abort-aware `process.poll`, but it still lacks the broader runtime contract requested here: no framework-level per-tool interruption result, no explicit partial-result/transcript policy for live aborts, and no caller `AbortSignal` path for `sessions_send` gateway waits.

## What This Changes

The issue requests a core runtime design for abort-aware tool execution, including framework-level interruption, partial-result/transcript policy, steer semantics, and cancellable gateway-backed tools.

## Best Possible Solution

Define the core agent-runtime interruption contract first, then implement it consistently: a framework fallback for stuck tools, opt-in per-tool partial results, transcript handling for aborted live tool calls, and cancellable gateway plumbing for `sessions_send` without killing resumable background processes.

## Reproduction Assessment

Yes for the remaining gaps by static reproduction against current main: inspect `sessions_send`, `run-wait`, `callGateway`, and `GatewayClient.request` to see timeout-only waits, and inspect the tool wrapper/adapter to see signal forwarding without a per-tool execution race. The broad partial-result and transcript behavior is a design contract question rather than a single runnable failure.

## Solution Assessment

No current implementation fully solves the issue. The `process.poll` fix is the right narrow optimization for that tool, but the broader issue needs an explicit runtime contract before more piecemeal tool behavior changes.
