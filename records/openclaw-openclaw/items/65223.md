## Summary

Keep open. Current main fixes the narrow `process.poll` abort example, but the broader framework-level interruption contract is still not implemented: tool execution is still awaited directly, `sessions_send` remains timeout-only, gateway calls do not accept caller abort signals, and docs still define Pi steering at the post-tool-call-batch model boundary.

## What This Changes

The issue asks for a framework-level tool abort/interruption contract, with opt-in per-tool partial-result handling for cases like `process.poll` and gateway-backed `sessions_send`.

## Best Possible Solution

Define one core interruption contract, then implement it consistently across the agent tool adapter, transcript/result handling, opt-in per-tool partial results, gateway cancellation plumbing, and docs while preserving resumable background processes.

## Reproduction Assessment

Yes, for the remaining gap by source inspection: a long-running tool that ignores the passed signal is still awaited directly, and `sessions_send` waits through timeout-only gateway calls. No live runtime reproduction was run because this review is read-only.

## Solution Assessment

No current implementation fully solves the issue. The `process.poll` abort fix and outer prompt abort wrapper are useful partial measures, but the maintainable solution needs an explicit runtime contract for abort, steer, partial results, and gateway cancellation before broad implementation.
