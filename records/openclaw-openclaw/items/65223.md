## Summary

Keep open: current main has fixed the narrow `process.poll` abort wait, but the broader framework interruption contract remains unresolved because generic tool execution is still awaited directly, `sessions_send` still ignores the tool `AbortSignal`, gateway calls are timeout-only, and steering is still documented at the post-tool-batch boundary.

## What This Changes

Request a design and implementation direction for framework-level tool interruption, per-tool abort handling, partial-result policy, and gateway cancellation while preserving resumable background work.

## Best Possible Solution

Define one explicit runtime interruption contract, then implement it across the tool adapter, transcript/result handling, opt-in per-tool partial results, gateway cancellation plumbing, and steering docs while preserving resumable background processes.

## Reproduction Assessment

Yes, by source inspection: a long-running tool that ignores the passed signal is still awaited directly, and `sessions_send` waits through timeout-only gateway calls. I did not run a live runtime reproduction because this review is read-only.

## Solution Assessment

No current implementation fully solves the issue. The `process.poll` abort fix and related steering/run-generation PRs are useful partial pieces, but the maintainable solution needs a single maintainer-approved contract for abort, steer, partial results, transcript history, and gateway cancellation.
