## Summary

Keep open. Current main addresses the narrow `process.poll` abort example, but the broader framework-level interruption contract is still missing: the tool adapter still awaits tool execution directly, `sessions_send` remains timeout-only, gateway requests have no caller `AbortSignal`, and docs still describe Pi steering at the post-tool-call-batch model boundary.

## What This Changes

The issue requests a framework-level interruption contract for long-running tool execution, including abort fallback behavior, opt-in partial results, steer semantics, transcript policy, and cancellable gateway-backed tools.

## Best Possible Solution

Define one core interruption contract, then implement it consistently across the agent tool adapter, transcript/result handling, opt-in per-tool partial results, gateway cancellation plumbing, and docs while preserving resumable background processes.

## Reproduction Assessment

Yes for the remaining gap by source inspection: a long-running tool that ignores the passed signal is still awaited directly, and `sessions_send` waits through timeout-only gateway calls. No live runtime reproduction was run because this review is read-only.

## Solution Assessment

No current implementation fully solves the issue. The `process.poll` abort fix and outer prompt abort wrapper are useful partial measures, but the maintainable solution needs an explicit runtime contract before adding more isolated tool behavior.
