## Summary

Keep open. Current main fixes the narrow `process.poll` busy-wait example and aborts the outer embedded prompt, but it still does not implement the broader framework-level tool execution interruption contract requested here: tool execution is still awaited directly, `sessions_send` has no `AbortSignal` path, gateway requests are timeout-only, and the docs still describe Pi steering as waiting until the model boundary after the current tool-call batch.

## What This Changes

The issue requests a design and implementation contract for abort-aware tool execution, framework-level interruption, steer semantics, partial tool results, and cancellable gateway-backed tool waits.

## Best Possible Solution

Define one core interruption contract first, then implement it consistently: framework fallback behavior for stuck tools, opt-in per-tool partial results, transcript handling for aborted tool calls, and cancellable gateway plumbing for `sessions_send` without killing resumable background processes.

## Reproduction Assessment

Yes for the remaining gaps by source inspection: a long-running tool that ignores its signal is still awaited directly, and `sessions_send` waits through timeout-only gateway calls. No live runtime test was run because this review is read-only.

## Solution Assessment

No current implementation fully solves the issue. The `process.poll` abort fix and outer prompt abort wrapper are useful partial measures, but the best fix is an explicit maintainer-approved runtime contract before adding more isolated tool behavior.
