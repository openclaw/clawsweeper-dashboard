## Label Justifications

- `P2`: This is a normal-priority runtime design improvement with real user impact but broad product semantics rather than an emergency fix.
- `impact:session-state`: The report is about stale tool results, ReplyOperation lifetime, and old-run state leaking into later conversation context after abort.
- `impact:crash-loop`: The issue covers active runs remaining blocked or hung behind long tool and gateway waits after abort.

## Summary

Keep open: current main has partial cooperative abort improvements, including `process.poll` and MCP signal forwarding, but the broader framework-level interruption contract is still missing because generic tool execution is awaited directly, `sessions_send` still has no tool-signal path, gateway calls remain timeout-only, and steering is still documented as tool-boundary behavior.

## What This Changes

The issue asks OpenClaw to define and implement abort-aware framework-level interruption semantics for long-running tools, including process polling, `sessions_send`, steer handling, partial results, and cleanup.

## Best Possible Solution

Define one maintainer-approved interruption contract, then implement it in staged PRs across the tool adapter, cooperative tool partial results, gateway cancellation plumbing, transcript/result handling, steering docs, and focused plus real-runtime proof while preserving resumable background processes.

## Reproduction Assessment

Yes, by source inspection: a long-running tool that ignores the passed signal is still awaited directly, and `sessions_send` waits through timeout-only gateway calls. I did not run a live runtime reproduction because this review is read-only.

## Solution Assessment

No current implementation fully solves the issue. The existing `process.poll` and MCP signal-forwarding fixes are useful partial pieces, but the maintainable solution needs one explicit contract for abort, steer, partial results, transcript history, and gateway cancellation.
