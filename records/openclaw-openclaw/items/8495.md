## Summary

Current main and v2026.6.5 still suppress unauthorized WhatsApp group traffic without leaving the group, reporting group membership changes, or exposing a leave action. The request remains useful and is not superseded, but group-admission identity rules and destructive-action authorization require an explicit maintainer product and security decision.

## What This Changes

Add opt-in unauthorized WhatsApp group auto-leave behavior, a group-join event or webhook, and a manual leave-group action.

## Best Possible Solution

Implement one plugin-owned WhatsApp group-admission lifecycle: detect the linked account being added, normalize Baileys identities, compare the group itself against the configured membership policy, notify the owner through an explicit privacy-scoped event, and allow departure only through an opt-in policy or owner-authorized action.

## Reproduction Assessment

Not applicable as a bug reproduction. Current source and the v2026.6.5 tag clearly establish the baseline: unauthorized traffic is suppressed, but membership is not changed and no join event or leave action exists.

## Solution Assessment

No—not as three independent additions. The narrowest maintainable solution is a single WhatsApp plugin-owned admission model that separates group membership policy from sender policy and centralizes notification and leave authorization.

## Risks / Open Questions

- Baileys group participant events include multiple identity forms, so an implementation that assumes stable phone-number JIDs could authorize or classify a join incorrectly.
