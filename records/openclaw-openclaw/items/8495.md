## Summary

Current main and v2026.6.5 still suppress unauthorized WhatsApp group traffic without leaving the group, emitting a group-membership event, or exposing a leave action. The request remains useful and unsuperseded, but its group-admission and destructive-action security semantics require maintainer judgment.

## What This Changes

Add opt-in WhatsApp unauthorized-group auto-leave, group-join notification or event handling, and a manual leave-group action.

## Best Possible Solution

Add an explicit WhatsApp group-membership response policy keyed to the existing `groups` admission surface, subscribe to Baileys membership events, and expose leaving only through a separately gated account-and-group action while sharing the owner-notification contract with the broader group-pairing design.

## Reproduction Assessment

Not applicable: current main follows its documented split between group membership and sender authorization; this issue requests new membership actions and events rather than reporting broken existing behavior.

## Solution Assessment

No, not exactly as proposed: auto-leave should not be driven directly by `groupAllowFrom`, because that field authorizes senders. The safer design uses the existing `groups` membership boundary plus explicit authorization for destructive leave operations.
