## Summary

Current main and v2026.6.5 still suppress unauthorized WhatsApp group traffic without leaving the group, emitting a membership event, or exposing a leave action. The feature remains useful and unsuperseded, but needs maintainer product and security design because group admission and sender authorization are intentionally separate contracts.

## What This Changes

Add opt-in unauthorized-group auto-leave, a WhatsApp group-join notification/event, and a manual leave-group action.

## Best Possible Solution

Add one plugin-owned WhatsApp membership-change event keyed by group JID, use the existing group-admission model rather than sender allowlisting, and route opt-in auto-leave plus any manual leave action through one audited, target-authorized operation coordinated with https://github.com/openclaw/openclaw/issues/11489.

## Reproduction Assessment

Not applicable; current source gives high confidence that these optional capabilities are absent, but no existing documented behavior is failing.

## Solution Assessment

No, not exactly as proposed: the useful outcome should remain, but group admission must not be inferred from sender-oriented `groupAllowFrom`, and join notification plus leave operations should share one canonical membership boundary.

- Implementing automatic policy, lifecycle notification, and manual action as separate ad hoc paths could create conflicting membership decisions.
