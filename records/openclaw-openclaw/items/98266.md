## Summary

Keep open. Current main and the latest release still create a direct Discord Gateway WebSocket agent unless explicit Discord proxy config is set; related proxy work covers broader or adjacent REST/metadata behavior, not this remaining WebSocket env-proxy fallback.

## What This Changes

Discord Gateway WebSocket should honor HTTP_PROXY/HTTPS_PROXY as a fallback when no explicit Discord proxy is configured.

## Best Possible Solution

Add a narrow Discord Gateway WebSocket env-proxy fallback through the existing plugin-safe proxy helper, preserving explicit Discord proxy precedence and NO_PROXY semantics.

## Reproduction Assessment

Yes. Source inspection shows the Discord Gateway WebSocket path creates a direct agent unless explicit Discord proxy config is present, while Discord REST and WhatsApp WebSocket already have env proxy fallback.

## Solution Assessment

Yes, with maintainer security confirmation. Reusing `createNodeProxyAgent` for the Discord Gateway target is the narrow maintainable fix because it avoids new config surface and preserves existing proxy helper semantics.
