## Summary

Keep open: current main still sends Matrix markdown tables through the shared table converter with the fallback `code` mode, so the reported fenced-code behavior remains source-reproducible. The prior Matrix table issue and attempted PR are both closed/unmerged, making this issue the active follow-up rather than something already implemented.

## What This Changes

The issue requests Matrix outbound markdown tables stop becoming fenced code blocks by setting a better Matrix table default and, if needed, allowing HTML table rendering for explicit opt-in.

## Best Possible Solution

Matrix should carry an explicit owner-approved table-mode default plus regression tests for default send/edit formatting; table HTML support should be added only if dependency/source validation confirms explicit opt-in currently needs it.

## Reproduction Assessment

Yes, at source level: Matrix send/edit paths resolve `channel: "matrix"`, the resolver falls back to `"code"` without a plugin default, and code mode is covered by tests as fenced-code table rendering. I did not run a live homeserver/Element flow.

## Solution Assessment

Unclear: the report identifies the right source path, but the best default, bullets versus raw HTML/off, needs Matrix maintainer judgment because client rendering support is mixed. Once that policy is chosen, the implementation should be narrow in the Matrix plugin plus focused tests.
