## Summary

Keep open: current main and v2026.6.10 still provide SKILL.md/frontmatter skills, ClawHub verification/scanning, operator install policy, and generic hooks, but not the requested skill.yaml permission manifest standard with install-time consent, audit/trust grades, signing, or attestation.

## What This Changes

The issue requests a standardized skill permission manifest format for skills, including install-time review, audit/trust reporting, signing, and attestation.

## Best Possible Solution

Keep this open for maintainer-sponsored product/security design that defines one versioned skill permission manifest contract and coordinates ClawHub, install/update UX, audit/trust output, signing/attestation, and runtime hook enforcement semantics.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/security design request. Source and release inspection give high confidence that the requested skill.yaml manifest, consent, audit/trust, signing, and attestation contract is still absent.

## Solution Assessment

Unclear as a final implementation path: the direction addresses a real security gap, but the proposed full manifest/trust/signing/attestation stack needs maintainer design before code should land. A phased approach should start with an accepted schema and explicit declared-versus-enforced wording, then layer consent, audit, signing, and runtime enforcement.
