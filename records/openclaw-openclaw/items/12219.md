## Summary

Keep open. Current main and v2026.6.10 still have SKILL.md/frontmatter skills, ClawHub verification/provenance, install policy, and generic hooks, but not the requested versioned `skill.yaml` permission manifest with install consent, skills audit/trust grades, signing, and attestation; related open items are partial overlaps rather than replacements.

## What This Changes

The issue requests a standardized `skill.yaml` permission manifest for skills, including install-time review, audit/trust scoring, signing, and attestation integration.

## Best Possible Solution

Keep this open for maintainer security/product design around one versioned skill permission and trust contract, coordinated with ClawHub verification, install/update consent, audit/trust reporting, signing, attestation, and runtime enforcement hooks.

## Reproduction Assessment

Yes, as a source-level feature gap: current `main` and v2026.6.10 expose SKILL.md metadata, ClawHub verification, install policy, and hooks, but no standardized `skill.yaml` permission manifest or skills audit/consent workflow.

## Solution Assessment

Unclear as a final implementation path. The direction is well motivated, but maintainers still need to choose schema placement, trust roots, declared-versus-enforced semantics, signing/attestation authority, and coordination with broader capability work.
