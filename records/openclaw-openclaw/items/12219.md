## Summary

Keep open. Current main still uses SKILL.md/frontmatter skills, ClawHub scan visibility, and generic install/tool-call hooks, but it does not implement the requested skill.yaml permission manifest, install-time consent review, skills audit/trust grading, signing, or attestation; recent maintainer discussion also kept this as product/security follow-up.

## What This Changes

The issue requests a standardized skill permission manifest for skills, including declared permissions, install-time review, audit/trust scoring, signing, and attestation.

## Best Possible Solution

A maintainer-designed, versioned skills-first capability manifest should land with parser/types, backward-compatible SKILL.md integration, inspect/audit or install/update visibility, ClawHub display policy, documented signing and attestation expectations, and a clear relationship to before_install and before_tool_call enforcement.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata types, public docs, hook contracts, tests, and focused searches for the requested manifest/audit/trust/signing surfaces.

## Solution Assessment

Unclear whether the full external npm package model is the best direct implementation. The safer path is explicit maintainer RFC/design, likely starting with the smaller visibility-first vocabulary in #74594 before deciding consent, runtime enforcement, trust grading, signing, and attestation policy.
