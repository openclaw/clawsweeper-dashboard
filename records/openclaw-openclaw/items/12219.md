## Summary

Keep open. Current main still lacks the requested standardized skill permission manifest, install-time consent review, audit/trust reporting, signing, and attestation contract; the linked external implementations are useful prior art but do not settle the OpenClaw core/API design.

## What This Changes

Request to standardize a `skill.yaml` permission manifest for skills, with install-time review, audit/trust scoring, signing, attestation, and ClawHub visibility.

## Best Possible Solution

Keep this open for maintainer-designed, versioned skill permission metadata that works with existing SKILL.md skills, exposes pre-install inspection and audit in CLI/ClawHub, documents signing/attestation expectations, and clearly separates declaration from runtime enforcement.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a regression. Source and docs inspection on current main verifies that the requested manifest, consent, audit/trust, signing, and attestation surfaces are absent.

## Solution Assessment

Unclear whether the exact external `@ellistevo/openclaw-secure` schema is the best implementation. The safer path is explicit maintainer product/security design coordinated with the open capability-manifest, capability-enforcement, and supply-chain security threads.
