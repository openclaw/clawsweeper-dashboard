## Summary

Keep open. Current main still lacks the requested standardized skill permission manifest and the related install consent, audit/trust, signing, and attestation contract; the latest maintainer context also treats this as product/security design rather than cleanup-closeable.

## What This Changes

Request a versioned `skill.yaml` permission manifest for skills, with install-time review plus audit, trust scoring, signing, and attestation integration.

## Best Possible Solution

Keep this open for maintainer-designed, versioned skill permission metadata that works with existing `SKILL.md` skills, exposes pre-install inspection and audit in CLI/ClawHub, and clearly separates visibility, consent, signing, attestation, and runtime enforcement.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a regression. The missing behavior is high-confidence from current source and docs inspection of the skills CLI, ClawHub installer, skill metadata, docs, and hook contracts.

## Solution Assessment

Unclear whether adopting the external `@ellistevo/openclaw-secure` model wholesale is the best implementation. The safer path is explicit maintainer product/security design coordinated with https://github.com/openclaw/openclaw/issues/74594 and the broader capability/enforcement threads.
