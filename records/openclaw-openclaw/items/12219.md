## Summary

Keep open. Current main still uses SKILL.md/frontmatter skills, ClawHub scan visibility, and generic install/tool-call hooks, but it does not implement the requested skill.yaml permission manifest, install-time consent review, skills audit/trust grading, signing, or attestation; the recent maintainer discussion also explicitly left this as product/security follow-up.

## What This Changes

Requests a standardized skill.yaml permission manifest for skills, with pre-install permission review, audit/trust scoring, signing/attestation, and ClawHub integration.

## Best Possible Solution

A maintainer-designed, versioned skills-first capability manifest should land with parser/types, backward-compatible SKILL.md integration, visible inspect/audit or install/update output, ClawHub policy/display support, documented signing and attestation expectations, and a clear relationship to before_install and before_tool_call enforcement.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata types, plugin hook docs/tests, public skills docs, and focused searches for manifest, audit, trust, signature, and attestation surfaces.

## Solution Assessment

Unclear whether the full proposed external npm package model is the best direct implementation. The safer path is explicit maintainer RFC/design, likely starting with the smaller visibility-first manifest vocabulary from #74594 before deciding consent, enforcement, trust grading, signing, and attestation policy.
