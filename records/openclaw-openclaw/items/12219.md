## Summary

Keep open. Current main still lacks the requested standardized skill permission manifest and related consent/audit/trust/signing contract, and recent maintainer context explicitly treats this as product/security design rather than cleanup-closeable.

## What This Changes

Request a `skill.yaml` permission manifest standard for skills, with install-time review, audit/trust scoring, signing, attestation, and ClawHub visibility.

## Best Possible Solution

Keep this open for a maintainer-designed, versioned skills permission manifest that remains compatible with `SKILL.md`, exposes pre-install inspection/consent/audit in CLI and ClawHub, and defines how signing, attestation, and runtime hooks relate.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a reported regression. The absence of the requested standard is high-confidence from current source/docs inspection of skills CLI, ClawHub install, skill metadata, docs, and hook contracts.

## Solution Assessment

Unclear whether adopting the external `@ellistevo/openclaw-secure` model wholesale is the best implementation. The safer path is explicit maintainer product/security design, likely coordinated with the narrower visibility-first RFC in #74594 and the broader capability/enforcement threads.
