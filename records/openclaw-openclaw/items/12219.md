## Summary

Keep open. Current main and the latest release still use SKILL.md/frontmatter skills with generic hooks and ClawHub scan visibility, but they do not implement the requested skill.yaml permission manifest, install-time consent review, skills audit/trust reporting, signing, or attestation contract; related open threads are complementary rather than superseding.

## What This Changes

The issue requests a standardized skill.yaml permission manifest for skills, including permission review before install, audit/trust reporting, signing, and attestation.

## Best Possible Solution

Keep this open for a maintainer-designed, versioned skill permission manifest that integrates with SKILL.md skills, ClawHub install/update review, audit/trust reporting, signing/attestation expectations, and documented runtime-enforcement boundaries.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a runtime regression. Static inspection on current main and v2026.5.7 gives a high-confidence absence check for the requested manifest, consent, audit/trust, signing, and attestation surfaces.

## Solution Assessment

Unclear whether the exact proposed skill.yaml schema is the best final contract. The safer path is explicit maintainer product/security design coordinated with the related inspect-only capability RFC, runtime enforcement, provenance, and signing threads.
