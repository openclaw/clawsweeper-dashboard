## Summary

Keep open. Current main and v2026.6.10 have SKILL.md/frontmatter skills, ClawHub verification/provenance, install policy, and generic hooks, but not the requested versioned skill.yaml permission manifest with install consent, skills audit/trust grades, signing, and attestation; the linked openclaw-secure package, shell-security-ultimate skill, and tinkerclaw fork are useful prior art but do not obsolete the core/API design question.

## What This Changes

The issue requests a standard skill permission manifest format plus install-time review, audit/trust reporting, signing, and attestation for OpenClaw skills.

## Best Possible Solution

Keep this open for maintainer security/product design around one versioned skill permission contract, coordinated with ClawHub verification, install/update consent, audit/trust reporting, signing, attestation, and runtime enforcement hooks.

## Reproduction Assessment

Yes, as a source-level feature gap: current main and v2026.6.10 expose SKILL.md metadata, ClawHub verification, install policy, and hooks, but no standardized skill.yaml permission manifest or skills audit/consent workflow.

## Solution Assessment

Unclear as a final implementation path. The direction is well motivated, but maintainers still need to choose schema location, declared-versus-enforced semantics, signing/attestation trust roots, and how this coordinates with broader capability-grant work.
