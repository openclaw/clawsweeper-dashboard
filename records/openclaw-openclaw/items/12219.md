## Summary

Keep open. Current main still lacks the requested skill permission-manifest, consent/audit/trust, signing, and attestation contract, and the recent maintainer context treats this as product/security design rather than cleanup-closeable.

## What This Changes

Define a standard `skill.yaml` permission manifest with install-time review, audit/trust scoring, signing, and attestation for OpenClaw skills.

## Best Possible Solution

Keep this open for a maintainer-designed, versioned skills-first permission manifest that stays compatible with `SKILL.md`, exposes pre-install inspection/consent/audit in CLI and ClawHub, and defines how signing, attestation, and runtime hooks relate.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a reported regression. The absence of the requested standard is high-confidence from current source/docs inspection of skills CLI, ClawHub install, skill metadata, and hook contracts.

## Solution Assessment

Unclear whether adopting the external package model wholesale is the best direct implementation. The safer path is explicit maintainer product/security design, likely starting from #74594's visibility-first vocabulary before deciding consent, enforcement, signing, and attestation.
