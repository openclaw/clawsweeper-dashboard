## Summary

Current main adds ClawHub scan/provenance verification and generic install/tool-call hooks, but it still lacks a standardized per-skill capability declaration, pre-install permission review, trust-grade contract, or runtime authorization model; the issue remains useful but requires explicit product/security consolidation with the overlapping open RFCs.

## What This Changes

Requests a standardized skill permission manifest with install-time consent, auditing, trust scoring, signing, and attestation.

## Best Possible Solution

Designate one versioned skills-first capability schema, expose it through skill inspection and install/update permission diffs, keep ClawHub provenance/signature verdicts separate, and add runtime enforcement only through explicit generic hook and policy contracts.

## Reproduction Assessment

Not applicable; this is a proposed security and ecosystem capability rather than broken behavior with a failing current-main reproduction.

## Solution Assessment

No, not as a single wholesale integration of the reference package; the maintainable direction is a shared capability vocabulary with separately owned ClawHub provenance, install consent, and runtime enforcement layers.
