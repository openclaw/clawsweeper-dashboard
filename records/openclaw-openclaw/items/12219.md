## Summary

Keep open. Current main still lacks the requested standardized skill permission manifest, install-time consent review, audit/trust reporting, signing, and attestation contract, and the newer capability-manifest RFCs are narrower related design threads rather than a complete replacement.

## What This Changes

The issue requests a `skill.yaml` permission manifest standard for skills, including declared permissions, install-time review, audit/trust reporting, signing, and attestation.

## Best Possible Solution

Keep this open for maintainer-designed, versioned skill permission metadata that works with SKILL.md skills, exposes pre-install inspection and audit in CLI/ClawHub, documents signing/attestation expectations, and clearly separates declaration from runtime enforcement.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a regression. Static inspection of current main gives a high-confidence verification path that the requested manifest, consent, audit/trust, signing, and attestation surfaces are absent.

## Solution Assessment

Unclear whether the exact `skill.yaml` schema from `@ellistevo/openclaw-secure` is the best implementation. The safer path is explicit maintainer product/security design coordinated with https://github.com/openclaw/openclaw/issues/74594, https://github.com/openclaw/openclaw/issues/12678, and https://github.com/openclaw/openclaw/issues/10827.
