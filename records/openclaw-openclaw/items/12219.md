## Summary

Keep open. Current main still supports SKILL.md/frontmatter skills, ClawHub scan visibility, and generic plugin hooks, but not the requested permission manifest, install consent, audit/trust scoring, signing, or attestation surfaces; the maintainer discussion also frames this as product/security follow-up.

## What This Changes

The issue requests a standardized `skill.yaml` permission manifest with install-time review, audit/trust scoring, signing, and attestation for skills.

## Best Possible Solution

Keep this open for a maintainer-designed, versioned skills-first capability manifest that integrates with SKILL.md/frontmatter compatibility, ClawHub/direct install visibility, inspect/audit UX, and documented enforcement hooks.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata parser/types, public docs, hook contracts, tests, and focused searches for the requested manifest/audit/trust/signing surfaces.

## Solution Assessment

Unclear whether the full external npm package model is the best direct implementation. The safer path is explicit maintainer product/security design, likely starting with the smaller visibility-first vocabulary in #74594 before deciding consent, runtime enforcement, trust grading, signing, and attestation policy.
