## Summary

Keep open. Current main still lacks the requested standardized skill permission manifest, install-time consent, audit/trust reporting, signing, and attestation contract; the external `@ellistevo/openclaw-secure`, shell-security-ultimate ClawHub skill, and globalcaos/tinkerclaw references are useful prior art but do not make the OpenClaw core/API decision obsolete.

## What This Changes

Request a standardized `skill.yaml` permission manifest for skills, with validation, signing, auditing, trust metadata, and install-review integration.

## Best Possible Solution

Keep this open for maintainer-designed, versioned skill permission metadata that works with existing SKILL.md skills, exposes pre-install inspection and audit in CLI/ClawHub, documents signing/attestation expectations, and clearly separates declaration from runtime enforcement.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a regression. Source and docs inspection on current main gives high-confidence evidence that the requested manifest, consent, audit, signing, and attestation surfaces are absent.

## Solution Assessment

Unclear whether the exact external `@ellistevo/openclaw-secure` schema is the best implementation. The safer path is explicit maintainer product/security design coordinated with the open capability-manifest and capability-enforcement RFCs.
