## Summary

Keep open: current main has ClawHub trust-envelope verification, persisted install provenance, operator install policy, and generic install/tool hooks, but it still lacks the requested standardized skill permission/capability manifest, install-time consent review, skill audit/trust scoring, and skill-scoped runtime authorization. Related open trackers split the declaration and enforcement pieces, so this remains product/security design work rather than cleanup-close material.

## What This Changes

The issue requests a `skill.yaml` permission manifest standard for skills, including declared network/filesystem/shell/credential needs, install review, audit/trust scoring, signing, attestations, and ClawHub trust badges.

## Best Possible Solution

Keep this open for maintainer-led security/product consolidation, then adopt one versioned skill capability vocabulary and phase inspection, consent, persisted grants, runtime enforcement, and attestation into separately reviewable milestones.

## Reproduction Assessment

Not applicable as a bug reproduction: this requests a new skill security, consent, and trust contract. Source inspection does provide high-confidence evidence that current main lacks the requested standardized permission manifest and consent/audit flow.

## Solution Assessment

Unclear as a final implementation shape: the defense-in-depth direction is sound, but the all-in-one proposal should be split after maintainers approve the manifest taxonomy, consent/grant storage, enforcement boundary, and ClawHub/signing trust model.
