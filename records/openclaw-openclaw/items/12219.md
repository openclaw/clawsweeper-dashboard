## Summary

Keep open. Current main and the latest release still use `SKILL.md` metadata, ClawHub scan visibility, and generic install/tool-call hooks, but they do not implement the requested standardized `skill.yaml` permission manifest, install-time consent/audit flow, trust grades, signing, or attestation; the linked external package, ClawHub skill, and fork are useful prior art rather than a core implementation.

## What This Changes

The issue requests a standardized `skill.yaml` permission manifest with install-time review, audit/trust scoring, signing, and attestation for OpenClaw skills.

## Best Possible Solution

Keep this open for a maintainer-designed, versioned skill permission/capability manifest integrated with SKILL.md skills, ClawHub install/update review, audit/trust reporting, signing or attestation expectations, and documented runtime-enforcement boundaries.

Unclear whether a separate `skill.yaml` file is the final best contract. The maintainable path is explicit product/security design coordinated with the related capability-manifest, runtime-enforcement, provenance, signing, and ClawHub ecosystem threads.
