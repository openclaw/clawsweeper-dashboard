## Summary

Keep open. Current main and v2026.5.22 still use SKILL.md/frontmatter, ClawHub scan visibility, direct install/update, and generic hooks, but they do not implement the requested standardized skill.yaml permission manifest, install-time consent, skills audit/trust grades, signing, attestation, or ClawHub trust badges. Related open issues cover narrower or broader slices, and the existing discussion explicitly treats the external packages and ClawHub skills as useful prior art rather than a replacement for a maintainer-owned OpenClaw contract.

## What This Changes

Request for a versioned skill permission manifest standard with install-time review, audit/trust reporting, signing, attestation, and ClawHub trust presentation.

## Best Possible Solution

Design a maintainer-owned, versioned skill permission or capability manifest that integrates with SKILL.md skills, ClawHub install/update review, audit/trust reporting, signing or attestation expectations, and documented runtime-enforcement boundaries.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a runtime regression. Static inspection of current main and v2026.5.22 gives a high-confidence absence check for the requested manifest, consent, audit/trust, signing, and attestation surfaces.

## Solution Assessment

Unclear whether a separate skill.yaml file is the final best contract. The maintainable path is explicit product/security design coordinated with related capability-manifest, runtime-enforcement, provenance, signing, and ClawHub ecosystem threads.
