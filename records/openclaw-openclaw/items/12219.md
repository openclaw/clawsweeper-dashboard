## Summary

Keep open. Current main has SKILL.md/frontmatter skills, ClawHub scan visibility, source-install scanning, and generic before_install/before_tool_call hooks, but not a standardized skill.yaml permission manifest with install-time consent, skills audit/trust grades, signing, attestation, or ClawHub trust badges. The linked openclaw-secure package, shell-security-ultimate ClawHub skill, globalcaos/tinkerclaw fork, and related open RFCs are useful prior art rather than a settled core implementation.

## What This Changes

Request to define a standard skill.yaml permission manifest for skills, including declared permissions, install-time review, audit/trust reporting, signing, attestation, and ClawHub badges.

## Best Possible Solution

Keep this open for a maintainer-designed, versioned skill permission/capability manifest integrated with SKILL.md skills, ClawHub install/update review, audit/trust reporting, signing or attestation expectations, and documented runtime-enforcement boundaries.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a runtime regression. Static inspection of current main gives a high-confidence absence check for the requested manifest, consent, audit/trust, signing, and attestation surfaces.

## Solution Assessment

Unclear whether a separate skill.yaml file is the final best contract. The maintainable path is explicit product/security design coordinated with related capability-manifest, runtime-enforcement, provenance, signing, and ClawHub ecosystem threads.
