## Summary

Keep open. Current main still has SKILL.md/frontmatter skills, ClawHub scan visibility, and generic install/tool-call hooks, but it does not provide the requested standardized skill permission manifest, install-time consent review, audit/trust reporting, signing, or attestation contract; the linked prior art is useful but not a merged replacement.

## What This Changes

Requests a standardized `skill.yaml` permission manifest with install-time review, audit/trust reporting, signing, and attestation for skills.

## Best Possible Solution

Keep this open for a maintainer-designed, versioned skill permission manifest that integrates with SKILL.md skills, ClawHub install/update review, CLI audit/inspection, signing/attestation expectations, and documented runtime-enforcement boundaries.

## Reproduction Assessment

Not applicable: this is a feature/security design request, not a runtime regression. Static inspection on current main gives a high-confidence absence check for the requested manifest, consent, audit/trust, signing, and attestation surfaces.

## Solution Assessment

Unclear whether the exact proposed `skill.yaml` schema is the best final contract. The safer path is explicit maintainer product/security design coordinated with https://github.com/openclaw/openclaw/issues/74594, https://github.com/openclaw/openclaw/issues/12678, and https://github.com/openclaw/openclaw/issues/10827.
