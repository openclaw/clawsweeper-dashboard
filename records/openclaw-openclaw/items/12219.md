## Summary

Keep open for maintainer product and security design. Current main has useful ClawHub verification and generic install/tool-call policy hooks, but it still lacks the requested standardized permission declaration, consent, authorization, signing, and attestation contract; related RFCs remain open, and the security label also requires explicit maintainer handling.

## What This Changes

Define a standard skill permission manifest and integrate capability review, auditing, provenance, signing, attestation, and runtime enforcement across OpenClaw and ClawHub.

## Best Possible Solution

Adopt one canonical staged design: define a versioned, inspectable skill capability vocabulary with explicit non-enforcement semantics first, then separately specify install/update consent and runtime authorization, adding ClawHub signing or attestation only after its publisher trust model is agreed.

## Reproduction Assessment

## Solution Assessment

No, not as one combined implementation; capability declaration, consent, enforcement, scoring, signing, and attestation should be staged under one canonical RFC so each compatibility and trust boundary can be reviewed independently.
