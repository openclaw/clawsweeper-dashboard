## Summary

Keep this open: current main and the latest release still enforce a fixed 15-second `memory_search` wrapper deadline while the QMD search path and docs honor `memory.qmd.limits.timeoutMs`. The issue already has an open linked fix PR, but that PR is not landed and is currently dirty.

## What This Changes

The issue requests aligning the hardcoded `memory_search` 15-second tool timeout with the existing QMD timeout configuration so slower `qmd query` reranking can complete.

## Best Possible Solution

Land or replace the linked fix with a memory-core change that derives the QMD path's outer deadline from the resolved QMD timeout with bounded overhead, while retaining the default finite guard for non-QMD and hung operations.

## Reproduction Assessment

Yes at source level: with QMD query mode and `memory.qmd.limits.timeoutMs` set above 15 seconds, the current outer wrapper can still time out at 15 seconds before QMD's configured budget completes. I did not run a live slow QMD reranker in this read-only review.

## Solution Assessment

Yes for reusing the existing QMD timeout contract; adding a separate timeout knob should not be the first fix. The narrow maintainable path is to derive the QMD-specific outer deadline from resolved QMD config with bounded overhead.
