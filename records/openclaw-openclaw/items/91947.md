## Summary

Keep open: current main and the latest release still use a fixed 15s memory_search wrapper deadline, while QMD search itself honors memory.qmd.limits.timeoutMs. The issue is already paired with open fixing PRs, so it should stay open until one lands or maintainers choose a replacement path.

## What This Changes

The issue asks memory_search to stop failing slow QMD query-mode searches at a fixed 15s tool deadline by honoring or configuring the QMD search timeout.

## Best Possible Solution

Land or replace a narrow memory-core fix that derives the outer deadline from the resolved QMD timeout plus bounded overhead while keeping the existing 15s guard for default and builtin paths.

## Reproduction Assessment

Yes, at source level: with QMD query mode and memory.qmd.limits.timeoutMs above 15000, createMemorySearchTool still races the operation against a fixed 15s wrapper. I did not run a live QMD backend in this read-only pass, but the reporter's 46.7s rerank trace matches the current code path.

## Solution Assessment

Partly: adding a new timeout knob is not the best first fix because the existing QMD timeout is already documented and resolved. The maintainable path is to reuse that resolved QMD budget with bounded wrapper overhead and focused regression coverage.
