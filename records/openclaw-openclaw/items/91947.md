## Summary

Keep this issue open: current main and v2026.6.8 still put a fixed 15-second outer deadline around `memory_search`, while the documented QMD search timeout is only enforced inside QMD calls. Maintainer discussion on the closest landed-attempt PR confirms the bug is real, but the simple single-timeout-plus-overhead fix shape was rejected as unsafe, so this remains canonical work rather than cleanup-close material.

## What This Changes

The issue requests making the `memory_search` tool deadline configurable or aligned with QMD search timeout settings so slow `qmd query` runs can complete instead of timing out at 15 seconds.

## Best Possible Solution

Keep this as the canonical tracker and land a memory-core/QMD fix where QMD exposes or enforces a whole-search budget while the tool wrapper keeps finite default, runtime-loading, and supplement deadlines.

## Reproduction Assessment

Yes, at source level: configure QMD with `memory.qmd.limits.timeoutMs` above 15 seconds and run a QMD search that completes after the fixed outer 15-second wrapper. I did not run a live QMD reranker in this read-only pass, but current source and the reporter's 46.7-second trace match the failure path.

## Solution Assessment

No, the issue should not be closed and the simple single-QMD-timeout-plus-overhead PR shape is not the best fix. The maintainable path is a canonical QMD whole-search budget with separate bounded deadlines for runtime loading and supplement work.
