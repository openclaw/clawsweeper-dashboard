## Summary

Keep this issue open: current main and v2026.6.8 still use a fixed 15-second memory_search outer deadline while QMD searches use memory.qmd.limits.timeoutMs separately, and the active fix path is an open linked PR rather than merged code.

## What This Changes

The issue asks for QMD-backed memory_search to stop timing out at a fixed 15 seconds and instead honor the configured QMD search timeout for slow qmd query runs.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/95347 as the canonical configured-timeout fix, then close this issue only after the accepted fix reaches main.

## Reproduction Assessment

Yes, at source level: current main wraps memory_search in a fixed 15-second deadline while QMD search uses memory.qmd.limits.timeoutMs separately. The reporter and a later commenter also supplied live timing evidence showing QMD query work exceeding that wrapper deadline.

## Solution Assessment

Yes for keeping this issue open, but not for queueing another fix. The maintainable path is to review the open linked PR that keeps the timeout decision in the memory/QMD owner path, with maintainer acceptance of the longer-wait tradeoff.
