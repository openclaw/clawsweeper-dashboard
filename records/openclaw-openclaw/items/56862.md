## Label Justifications

- `P2`: Builtin memory_search can silently miss relevant persisted memory until a manual reindex, but the blast radius is limited to memory-core search freshness.
- `impact:session-state`: The report concerns stale memory index and sync state causing recall/search state drift rather than missing source memory files.

## Summary

Keep open. Current main and the latest release still let builtin memory_search query a populated dirty or reindex-needed index while search-triggered sync runs in the background, and the related open rollback/cache PR is adjacent rather than a query-path fix.

## What This Changes

The issue asks builtin memory_search to avoid returning empty results from stale or transitional builtin SQLite indexes and to wait for or expose reindex state more clearly.

## Best Possible Solution

Add a builtin memory search preflight that detects dirty/session-dirty or full-reindex-needed state before querying, waits for bounded sync/reindex or returns an explicit transient failure, and leaves QMD/provider behavior unchanged.

## Reproduction Assessment

Yes by source inspection: a populated builtin index skips the synchronous search bootstrap, then dirty/session-dirty search sync starts in the background while the current query proceeds. I did not run a live BigModel endpoint repro in this read-only review.

## Solution Assessment

No. The current manual reindex workaround and background search sync are not the best fix; the narrow maintainable solution is a builtin memory-core query preflight/guard with regression coverage.
