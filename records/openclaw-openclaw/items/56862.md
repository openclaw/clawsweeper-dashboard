## Summary

Keep open. Current main still matches the reported gap: builtin search only forces synchronous sync when the index has no content, while populated dirty or reindex-needed indexes still proceed to query execution with search sync running in the background. Related PR #59137 is adjacent rollback/cache durability work and explicitly does not change query behavior.

## What This Changes

The issue requests that builtin `memory_search` avoid returning empty results while the builtin SQLite memory index is dirty, stale, or awaiting reindex/sync completion.

## Best Possible Solution

Fix this in the `memory-core` plugin query path: before querying an existing builtin index, detect dirty/session-dirty and full-reindex-needed metadata state, then await bounded sync/reindex or return an explicit transient indexing state instead of an empty result caused by pending sync.

## Reproduction Assessment

Yes. A focused regression can create a populated builtin SQLite/FTS index, mark memory or sessions dirty or make metadata require full reindex, delay sync, and assert that `search()` currently proceeds before sync completes.

## Solution Assessment

No. Current main still relies on background search sync plus manual `openclaw memory index --force`; the narrower maintainable fix is a builtin memory-core query preflight/guard with regression coverage, leaving QMD behavior unchanged.
