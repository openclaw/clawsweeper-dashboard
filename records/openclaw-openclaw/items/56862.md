## Summary

Keep open. Current main still matches the reported gap: builtin search only does a synchronous bootstrap when the index has no content, then lets a populated dirty/stale index proceed to FTS/vector query while search-triggered sync runs in the background. The documented force-reindex/status commands are useful workarounds, and related PR #59137 is adjacent rollback/cache durability work, but neither removes the query-path empty-result failure mode.

## What This Changes

The issue requests that builtin `memory_search` avoid returning empty results while the builtin SQLite memory index is dirty, stale, or awaiting reindex/sync completion.

## Best Possible Solution

Keep the issue open and fix it in `extensions/memory-core`: before builtin search queries an existing index, detect dirty memory/session state and metadata/config changes that require a full reindex, then await a bounded sync/reindex or surface an explicit transient indexing/unavailable state instead of returning an empty result solely because sync is pending.

## Reproduction Assessment

Yes. A focused regression can create a populated builtin SQLite/FTS index, mark the manager dirty or make stored metadata require full reindex, delay the sync promise, and assert that `search()` currently proceeds before sync completes; the reporter's force-reindex workaround and current non-blocking test make the path high-confidence even without live provider reproduction.

## Solution Assessment

No, current main is not the best solution because it relies on background sync plus manual `openclaw memory index --force` recovery. The narrow maintainable solution is a memory-core query-path guard with focused tests, leaving QMD and core plugin contracts unchanged unless a shared status surface is intentionally added.
