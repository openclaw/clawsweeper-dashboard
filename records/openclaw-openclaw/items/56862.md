## Summary

Keep open. Current main still lets builtin `memory_search` query a populated dirty or reindex-needed index while search-triggered sync runs in the background, and the related open PR at https://github.com/openclaw/openclaw/pull/59137 is rollback/cache durability work that explicitly does not change query behavior.

## What This Changes

The issue asks builtin `memory_search` to avoid returning empty results from a dirty or reindex-needed builtin SQLite memory index and to provide clearer reindex/status handling.

## Best Possible Solution

Add a builtin memory search preflight that detects dirty/session-dirty or full-reindex-needed state before querying, waits for a bounded sync/reindex or returns an explicit transient failure, and leaves QMD/provider behavior unchanged.

## Reproduction Assessment

Yes. A focused source-level regression can create a populated builtin SQLite/FTS index, mark memory or sessions dirty or make metadata require full reindex, delay sync, and assert that current `search()` proceeds before sync completes.

## Solution Assessment

No. Current main's manual reindex workaround and background search sync are not the best fix; the narrow maintainable solution is a builtin memory-core query preflight/guard with regression coverage.
