## Summary

Keep open. Current main still matches the reported gap: builtin search only waits for a synchronous sync when the index has no content; a populated dirty or reindex-needed index can still be queried while search sync runs in the background, and the related open PR at https://github.com/openclaw/openclaw/pull/59137 is adjacent rollback/cache durability work rather than a query-path fix.

## What This Changes

The issue asks builtin memory search to avoid returning empty results from a stale or transitional builtin index and to provide safer sync/reindex behavior.

## Best Possible Solution

Add a memory-core search preflight guard that detects dirty/session-dirty or full-reindex-needed builtin index state before querying, waits for a bounded sync/reindex or surfaces an explicit transient failure, and leaves QMD/provider behavior unchanged.

## Reproduction Assessment

Yes, source-reproducible with high confidence. A focused regression can create a populated builtin SQLite/FTS index, mark memory or sessions dirty or make metadata require full reindex, delay sync, and assert that current `search()` proceeds before sync completes.

## Solution Assessment

No, current main is not the best solution because manual `openclaw memory index --force` is only a workaround. The narrow maintainable fix is a builtin memory-core query preflight/guard with regression coverage, not a provider or QMD change.
