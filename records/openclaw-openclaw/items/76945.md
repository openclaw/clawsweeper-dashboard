## Summary

Keep open: current main still source-reproduces the no-delivery preview confusion and the status session table still has no cron/history distinction for cron session-store rows.

## What This Changes

Cron no-delivery jobs should show clear no-delivery preview/status output and distinguish historical cron sessions from ordinary recent sessions.

## Best Possible Solution

Make no-delivery preview text explicitly say no runner fallback delivery, then add an additive cron/history marker in status while preserving runtime delivery semantics and session retention.

## Reproduction Assessment

Yes, from source inspection. A job with `delivery: { mode: "none" }` produces a non-requested plan, but the preview code only short-circuits when the delivery object is absent, and status rows classify `agent:*:cron:*` store keys as ordinary direct/group sessions.

## Solution Assessment

Yes, a narrow presentation fix is the best path. Keep `resolveCronDeliveryPlan` and runner delivery behavior intact, but make preview/status rendering reflect no fallback delivery and cron session history clearly.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76945. On current main, explicit `delivery: { mode: "none" }` in `resolveCronDeliveryPreview` falls through because the not-requested branch requires `!job.delivery`, so `cron list/show` can report `none -> last` and fail-closed route errors even though docs say mode none means no runner fallback delivery and runtime state reports `lastDeliveryStatus: "not-requested"`. Adjust preview rendering so no runner fallback delivery is clear for `mode: none`; preserve existing message-tool target context semantics for explicitly authored channel/to/account/thread fields and do not change runner delivery behavior. Also distinguish cron session-store rows in `openclaw status` without deleting old rows; prefer an additive marker/flag or CLI label that avoids unnecessary status JSON breakage. Add focused regression tests and a user-facing CHANGELOG fix entry. Do not change session retention or delivery execution semantics.
