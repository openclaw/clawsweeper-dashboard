## Summary

Keep open: current main and the latest release still route `feishu_doc` create directly through `createDoc()` without writing or explicitly rejecting a supplied `content` value. Current schema and skill text no longer advertise `content` on create, so the remaining choice is a maintainer product decision: land or replace the linked implementation at https://github.com/openclaw/openclaw/pull/48377, or intentionally fail fast for unsupported `content`.

## What This Changes

The issue asks `feishu_doc` create to stop silently discarding a supplied `content` value, either by writing initial markdown during create or rejecting unsupported content.

## Best Possible Solution

Resolve the contract in the linked Feishu PR path: either land a tested create-plus-write implementation with recovery semantics, or update the tool to fail fast when `content` is supplied to create.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a create call with extra `content` reaches a branch that calls only `createDoc()` and never reads or rejects `p.content`. I did not run a live Feishu API test in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. Implementing create-plus-write is plausible and better UX, but a clear fail-fast error is also a valid fix if create-time content should remain unsupported.
