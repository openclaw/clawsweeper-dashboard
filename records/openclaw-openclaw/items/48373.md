## Summary

Keep this open. Current main still creates Feishu documents through `createDoc()` only for the `create` action, so any supplied `content` is not written; the issue is also paired with open fix PRs, so it should remain open until one is reviewed and merged or deliberately closed.

## What This Changes

The issue asks for `feishu_doc` `create` calls with `content` to either populate the new document or fail clearly instead of accepting input that is not applied.

## Best Possible Solution

Review the linked open fix PRs and land one focused change that either chains `createDoc()` plus `writeDoc()` with regression coverage, or explicitly rejects `content` on `create` and aligns the provider-facing schema/docs to remove the implication.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass `{ action: "create", title, content }` to `feishu_doc` and the current switch path calls only `createDoc()`, leaving `writeDoc()` unused for that call.

## Solution Assessment

Unclear until maintainers choose the intended contract. Implementing create-with-content is the better UX if it preserves created-document recovery on write failure; otherwise the safer narrow fix is to reject `content` on `create` with a clear error and schema/docs alignment.
