## Summary

Keep this issue open. Current main still has no explicit handling for the reported inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant text, and the existing WhatsApp delivery path can still pass normalized assistant text to `msg.reply` after sanitizers that do not cover this syntax.

## What This Changes

The issue reports that WhatsApp replies can expose raw pseudo tool-call text instead of executing the intended tool call and returning results.

## Best Possible Solution

Add shared user-visible sanitizer and diagnostic coverage for this exact pseudo-call format so it cannot leak through WhatsApp or history, and only consider executable promotion if maintainers approve a deterministic, registered-tool-only parser consistent with the existing tool-payload contract.

## Reproduction Assessment

Yes at source level: feed `[TOOL_CALL]{tool => "web_search", args => {...}}[/TOOL_CALL]` through the current assistant-visible sanitizer and WhatsApp normalization path, and the existing patterns do not remove it before `msg.reply`. A live WhatsApp/modelrelay reproduction was not run.

## Solution Assessment

Unclear for full tool execution: the safest narrow fix is to suppress and warn on this unsupported pseudo-call text, while converting it into real tool calls would need explicit maintainer approval because docs warn against blindly executing arbitrary assistant text.
