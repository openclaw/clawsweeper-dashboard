## Summary

Keep open. Current main and the latest release still drop the node-host exit signal, and the PR’s narrow helper-based fix looks correct, but the external contributor still needs after-fix real behavior proof and a fresh green validation pass before merge.

## What This Changes

The PR adds a signal-aware `resolveExitResult` helper in `src/node-host/invoke.ts` and tests signal kills, timeout kills, normal exits, and null-code/null-signal exits.

## Best Possible Solution

Land the narrow signal-observability fix after redacted real runtime proof is added and the branch is rebased with fresh green validation; leave broader kill-cause telemetry to https://github.com/openclaw/openclaw/issues/72240.

## Reproduction Assessment

Yes at source level: current main’s node-host `runCommand` ignores the exit `signal` argument, while a focused Node probe shows a signal-killed child emits `{code:null, signal:"SIGKILL"}`. I did not run the PR’s full OpenClaw behavior in a real contributor setup.

## Solution Assessment

Yes. Extracting a small exit-result helper and gating signal errors on `!timedOut` is the narrow maintainable fix for this result contract; broader OOM and scan diagnostics belong in the related diagnostics issue.
