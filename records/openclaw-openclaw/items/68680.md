## Summary

Keep open. Current main still drops signal-killed child process exits into `exitCode: undefined` with `error: null`, while this PR implements the narrow signal-aware result mapping and addresses the earlier timeout/test-isolation review concerns; it still needs the required changelog entry and a green rebased CI state before merge.

## What This Changes

The PR adds a `resolveExitResult` helper and tests so node-host `system.run` reports signal-killed subprocesses as structured errors while preserving intentional timeout-kill semantics.

## Best Possible Solution

Land this narrow signal-observability fix after adding the required changelog entry and getting a green rebased CI run, keeping intentional timeout results as `timedOut: true` with `error: null`.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction: a child process that emits `exit(null, "SIGKILL")` or `exit(null, "SIGTERM")` follows the current `code`-only handler and resolves with no error; Node's documented `exit` event contract confirms that signal-killed children produce exactly that shape.

## Solution Assessment

Mostly yes. The helper plus `!timedOut` guard is the narrow maintainable fix for the node-host result contract; the safer completion path is to add the required changelog entry and rebase for green CI, not broaden this into OOM policy or process supervision work.
