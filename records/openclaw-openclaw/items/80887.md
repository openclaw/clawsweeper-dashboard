## Summary

Keep open. The PR addresses a real open feature request, but the proposed contract is incomplete by source inspection: configured keys are only merged into save-time stale pruning/count capping, while load-time maintenance, `openclaw sessions cleanup`, and disk-budget eviction can still remove them; the external proof is also insufficient and the branch bundles unrelated installer/template work.

## What This Changes

The PR adds `session.maintenance.preserveKeys`, wires it into save-time session pruning/capping, and also adds PR-template files plus a PowerShell installer git-pull catch.

## Best Possible Solution

Implement the preserve-key contract narrowly across resolved config, every session entry-removal and disk-budget path, config help/docs, and focused regression tests; split installer/template changes into separate PRs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable for a current-main bug because this is a new config feature, but the PR defects are source-reproducible: the new field is only consumed by save-time prune/cap while other removal paths ignore it.

## Solution Assessment

Is this the best way to solve the issue? No. The direction fits https://github.com/openclaw/openclaw/issues/80853, but the implementation is partial and should be narrowed to the session-maintenance contract with docs/tests and without unrelated installer/template edits.
