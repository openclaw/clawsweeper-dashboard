## What This Changes

The PR adds `agents.list[].contextInjection` schema/type/generated metadata, threads agent IDs into embedded attempt and compaction context-injection resolution, and adds resolver tests.

## Best Possible Solution

Land the per-agent override after the resolver preserves normalized agent-id semantics, public docs and changelog describe the new config surface, and the branch is rebased cleanly.

Yes. A static reproduction is enough: configure an entry such as `{ id: "Strict Agent", contextInjection: "always" }` and run with normalized agent id `strict-agent`; existing per-agent config can match that entry, while this PR's raw equality lookup falls back to defaults.

## Solution Assessment

No, not yet. The schema and runtime plumbing are the right narrow direction, but the implementation should reuse the normalized per-agent config lookup and complete docs/changelog alignment before merge.
