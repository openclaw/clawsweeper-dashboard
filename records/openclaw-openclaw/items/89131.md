## Summary

Keep open: current main and v2026.6.8 still expose Talk relay as audio/tool transport plus `supportsVideoFrames` metadata, but not the requested provider/gateway `VideoFrame` contract, bridge append method, or append-video relay RPC. The related open PRs are partial, security-sensitive candidates and do not replace the maintainer product/API decision for this issue.

## What This Changes

The issue requests a provider/gateway realtime video-frame transport contract and relay RPC for active Talk sessions.

## Best Possible Solution

Settle the product/security/API shape first, then implement a narrow additive one-shot frame contract with provider mappings, gateway protocol validation, docs, and focused relay/provider tests.

## Reproduction Assessment

Not applicable: this is a feature/API proposal, not a broken existing behavior report. Source, tests, live related-PR state, and release inspection give high confidence that current OpenClaw lacks the requested provider/gateway video-frame submission path.

## Solution Assessment

Unclear until maintainers decide the product/security direction. A provider/gateway seam is plausible for relay clients, but the permanent SDK/RPC contract needs approved consent, validation, logging, and provider-handling rules before implementation.
