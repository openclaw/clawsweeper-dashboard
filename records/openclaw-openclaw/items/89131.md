## Summary

Keep open: current main and latest release still expose realtime Talk relay as audio/tool transport plus `supportsVideoFrames` metadata, but not the requested provider/gateway `VideoFrame` contract or relay RPC. The related open PRs are partial, security-sensitive candidates and do not replace the remaining product/API decision.

## What This Changes

Request a provider/gateway-level realtime video-frame transport for relay sessions, including a shared frame contract, bridge method, gateway RPC, and provider-specific delivery mapping.

## Best Possible Solution

Settle the product/security/API shape first, then implement a narrow additive one-shot frame contract with provider mappings, gateway protocol validation, docs, and focused relay/provider tests.

## Reproduction Assessment

Not applicable: this is a feature/API proposal, not a broken existing behavior report. Source and release inspection give high confidence that current OpenClaw lacks the requested provider/gateway video-frame submission path.

## Solution Assessment

Unclear until maintainers decide the product/security direction. A provider/gateway seam is plausible for relay clients, but the permanent SDK/RPC contract needs approved consent, validation, logging, and provider-handling rules before implementation.
