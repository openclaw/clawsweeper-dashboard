## Summary

Keep open: current main and v2026.6.6 still expose realtime relay audio/tool surfaces but no shared VideoFrame contract, RealtimeVoiceBridge video method, or talk.session.appendVideo-style RPC. The related browser-direct draft at https://github.com/openclaw/openclaw/pull/89127 is still open and explicitly leaves gateway-relay video transport to this issue, so this remains valid security-sensitive product/API work.

## What This Changes

The issue requests a provider/gateway-level contract for submitting one base64 JPEG video frame to an active realtime voice session, usable by relay clients and non-WebRTC providers.

## Best Possible Solution

Settle the product/security direction around the browser-direct draft, then add an additive one-shot frame contract with explicit consent, size/MIME limits, provider mappings, docs, and focused gateway/provider tests.
