## Summary

Keep this PR open. Current main still lacks pre-persistence hostname/IP blocking for cron webhook URLs, so the hardening remains useful, but the branch is not merge-ready until it updates the existing gateway cron test that currently expects private webhook URLs to be accepted and blocked only at dispatch time.

## What This Changes

The PR imports the existing SSRF hostname/IP classifier into `normalizeHttpWebhookUrl` and adds a dedicated normalizer test file for protocol validation, malformed input, blocked private/internal targets, and accepted public URLs.

## Best Possible Solution

Land the pre-persistence guard only with aligned tests: keep the shared runtime `fetchWithSsrFGuard` protection, reject obvious private/internal literal targets in `normalizeHttpWebhookUrl`, update the existing gateway cron test contract, and make an explicit maintainer call on whether #58971's operator opt-in should coexist or be superseded.

## Reproduction Assessment

Yes. Current main has a focused path: `src/gateway/server-cron.test.ts:363` creates a cron webhook targeting `127.0.0.1` and verifies runtime blocking; with this PR's normalizer guard, that same add path would reject before dispatch through `src/cron/service/jobs.ts:203`.

## Solution Assessment

No, not as submitted. The core approach is narrow and reuses the right shared SSRF classifier, but the PR needs to update existing gateway cron test expectations before it is the best mergeable solution.
