## Summary

Keep open. Current main still accepts private cron webhook URLs into job config and relies on the runtime fetch guard, so the PR addresses a real hardening gap, but the submitted patch invalidates the existing gateway cron SSRF regression test and needs a maintainer policy call against the related private-network opt-in PR.

## What This Changes

The PR imports the shared SSRF hostname/IP classifier into cron webhook URL normalization and adds focused tests for protocol validation, malformed values, blocked private/internal targets, and accepted public targets.

## Best Possible Solution

Land the pre-persistence SSRF guard only after aligning the gateway cron test contract and making an explicit maintainer decision on whether a future operator private-network opt-in should bypass normalization, bypass only runtime fetch policy, or be rejected as incompatible.

## Reproduction Assessment

Yes. Current main has a focused path: `src/gateway/server-cron.test.ts:363` creates a cron webhook targeting `127.0.0.1` and verifies runtime blocking; with this PR's normalizer guard, the add path would reject before dispatch.

## Solution Assessment

No, not as submitted. Reusing the shared SSRF classifier is the right narrow implementation direction, but the PR needs to update existing gateway cron expectations and resolve the open #58971 policy overlap before it is mergeable.
