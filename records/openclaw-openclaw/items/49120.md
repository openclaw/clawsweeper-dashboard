## Summary

Keep this PR open: current main still lacks pre-persistence private/internal cron webhook URL rejection, and the patch is a focused hardening change, but it needs real behavior proof and maintainer/security policy reconciliation with the open private-network opt-in proposal.

## What This Changes

The PR adds `isBlockedHostnameOrIp` validation to `normalizeHttpWebhookUrl` and a new test file covering private/internal webhook URL rejection and public URL acceptance.

## Best Possible Solution

Choose the cron webhook network posture explicitly, then land strict pre-persistence rejection with the runtime guard preserved, corrected comment, real behavior proof, and focused validation, or supersede it with a documented private-network opt-in.

## Reproduction Assessment

Yes, from source inspection. Current main accepts private http(s) webhook URLs through `normalizeHttpWebhookUrl` and persists delivery/failure-destination targets through that helper; I did not run a live cron create/patch flow in this read-only review.

## Solution Assessment

Mostly yes for a strict security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow and maintainable, but it is not the final answer if maintainers choose a documented private-network opt-in.
