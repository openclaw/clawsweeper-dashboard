## Summary

Keep open: the security hardening is plausible, but the patch is not merge-ready because it changes cron webhook rejection from runtime to create time without updating the existing runtime SSRF test contract, and the external contributor has not provided real behavior proof.

## What This Changes

The PR imports the shared SSRF host/IP classifier into cron webhook URL normalization and adds tests rejecting private/internal webhook targets while accepting public HTTP(S) URLs.

## Best Possible Solution

Choose the cron webhook SSRF posture explicitly, then land create/patch-time rejection with updated runtime coverage or pursue a documented operator opt-in while keeping the runtime DNS-pinned guard.

## Reproduction Assessment

Yes. Source inspection shows the PR guard would make state.cron.add reject the 127.0.0.1 target in src/gateway/server-cron.test.ts before the existing runtime-fetch assertion can run.

## Solution Assessment

No, not as submitted. Reusing the shared classifier is a reasonable hardening seam, but the patch must reconcile runtime coverage and the private-network opt-in policy first.
