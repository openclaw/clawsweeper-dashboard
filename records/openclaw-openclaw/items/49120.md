## Summary

## What This Changes

The PR adds `isBlockedHostnameOrIp` to cron webhook URL normalization and adds tests rejecting private/internal webhook hosts before cron webhook URLs are persisted or used.

## Best Possible Solution

Maintainers should choose the cron webhook network posture explicitly, then either land strict pre-persistence rejection with proof and the comment fix or supersede it with a documented private-network opt-in design.

## Reproduction Assessment

Yes, at source level. Current main accepts private HTTP(S) webhook URLs through `normalizeHttpWebhookUrl` and routes delivery/failure targets through that helper; I did not run a live cron create or patch flow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow for strict rejection, but it conflicts with the open private-network opt-in design.
