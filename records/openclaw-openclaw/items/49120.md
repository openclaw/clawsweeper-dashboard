## Summary

Keep open: current main still lacks pre-persistence SSRF rejection for cron webhook URLs, and the PR is a focused security hardening. It should not merge yet because the external contributor has not provided real behavior proof, and maintainers still need to choose between strict rejection and the related private-network opt-in direction.

## What This Changes

The PR imports the existing SSRF hostname/IP guard into cron webhook URL normalization and adds webhook URL validation tests.

## Best Possible Solution

Choose the cron webhook SSRF posture explicitly, then land strict pre-persistence rejection with corrected comments, runtime guard preserved, focused tests, and real behavior proof, or supersede it with a documented opt-in direction.

## Reproduction Assessment

Yes, at source level. Current main accepts private HTTP(S) webhook URLs through normalizeHttpWebhookUrl and persists delivery and failure-destination targets through that helper; I did not run a live cron create/patch flow in this read-only review.

## Solution Assessment

Mostly yes for a strict security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow and maintainable, but it is not the final answer if maintainers choose a documented private-network opt-in instead.
