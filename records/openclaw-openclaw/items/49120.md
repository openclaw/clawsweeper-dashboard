## Summary

Keep this PR open. Current main still accepts private/internal cron webhook URLs at the storage-normalization boundary and relies on runtime SSRF blocking later; the branch is a focused strict hardening, but it still needs contributor real behavior proof and maintainer/security reconciliation with https://github.com/openclaw/openclaw/pull/58971.

## What This Changes

The branch adds an early `isBlockedHostnameOrIp` check to cron webhook URL normalization and adds regression tests for private/internal host rejection and public URL acceptance.

## Best Possible Solution

Choose and document the cron webhook network posture, then land either strict pre-persistence rejection with the runtime guard preserved or a cohesive opt-in design that governs both normalization and delivery.

## Reproduction Assessment

Yes, from source inspection. Current main accepts private `http(s)` webhook URLs through `normalizeHttpWebhookUrl` and stores delivery/failure-destination targets through that helper; I did not run a live create/patch flow in this read-only review.

## Solution Assessment

Mostly yes for a strict security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow and maintainable, but it is not the final answer if maintainers choose to support a documented private-network opt-in.
