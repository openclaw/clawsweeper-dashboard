## Summary

Keep open. The security-hardening direction is sound, but the branch is not mergeable as submitted because it changes cron webhook rejection from runtime to create time without updating the existing gateway cron SSRF test, lacks the required changelog entry, and overlaps the open private-network opt-in policy discussion in #58971.

## What This Changes

The PR adds the shared SSRF hostname/IP classifier to cron webhook URL normalization and adds focused normalizer tests for blocked private/internal targets and accepted public URLs.

## Best Possible Solution

Land a coordinated cron webhook policy that either rejects blocked targets consistently at creation time or supports an explicit operator opt-in through both normalization and runtime dispatch, with gateway tests and changelog updated together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: applying this guard makes state.cron.add reject the existing 127.0.0.1 webhook in src/gateway/server-cron.test.ts before the runtime fetch assertion can run.

## Solution Assessment

No, not as submitted. Reusing the shared SSRF classifier is the narrow implementation direction, but the branch needs to update the gateway cron test contract, add the required changelog entry, and be reconciled with #58971 before merge.
