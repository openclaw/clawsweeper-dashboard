## Summary

Keep open. The patch is focused and appears source-correct, but this external PR still lacks after-fix real behavior proof and the strict pre-persistence rejection changes compatibility for private/internal cron webhook configs while a competing opt-in design remains open.

## What This Changes

The PR adds an `isBlockedHostnameOrIp` check to cron webhook URL normalization and adds unit coverage for private/internal hostname rejection and public URL acceptance.

## Best Possible Solution

Maintainers should explicitly choose the cron webhook network posture, then either land strict pre-persistence rejection with proof and the comment fix or supersede it with a documented private-network opt-in design.

## Reproduction Assessment

Yes, at source level. Current main accepts private HTTP(S) webhook URLs through `normalizeHttpWebhookUrl` and persists delivery and failure-destination targets through that helper; I did not run a live cron create/patch flow in this read-only review.

## Solution Assessment

Mostly yes if maintainers choose a strict security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow and maintainable, but it is not the final answer if OpenClaw wants an explicit private-network opt-in for cron webhooks.
