## Summary

Keep open. The strict reject-at-save hardening is plausible, and the author addressed the earlier edge-case test suggestion, but the branch still needs real behavior proof and a test-path fix because the new guard conflicts with current runtime SSRF regression coverage.

## What This Changes

Adds `isBlockedHostnameOrIp` to cron webhook URL normalization and adds webhook URL tests that reject private/internal targets before persistence.

## Best Possible Solution

Land the strict hardening only after preserving runtime SSRF coverage for already-persisted or legacy webhook URLs and after maintainers settle the cron webhook private-network policy.

## Reproduction Assessment

Yes. Source inspection shows the PR makes `normalizeHttpWebhookUrl("http://127.0.0.1:8080/cron-finished")` return null during cron add, while current `src/gateway/server-cron.test.ts` expects that exact URL to reach `fetchWithSsrFGuard`.

## Solution Assessment

Unclear. Reusing the existing SSRF guard is a maintainable strict-hardening path, but the PR should preserve runtime SSRF regression coverage and be reconciled with https://github.com/openclaw/openclaw/pull/58971 before merge.
