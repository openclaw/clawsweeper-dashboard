## Summary

Keep open: this PR is a focused security hardening change and current main still accepts private/internal cron webhook URLs at the normalization boundary, but it needs contributor real behavior proof and an explicit maintainer decision on strict save-time rejection versus the open private-network opt-in direction.

## What This Changes

The PR imports the existing SSRF hostname/IP classifier into cron webhook URL normalization and adds tests for rejecting private, loopback, link-local, metadata, `.local`, and `.internal` webhook targets.

## Best Possible Solution

Maintainers should choose the cron webhook network posture explicitly, then either land strict pre-persistence rejection with proof and the comment fix or supersede it with a documented private-network opt-in design.

## Reproduction Assessment

Yes, at source level. Current main accepts private HTTP(S) webhook URLs through `normalizeHttpWebhookUrl` and persists delivery and failure-destination targets through that helper; I did not run a live cron create/patch flow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow and maintainable for strict rejection, but it conflicts with the open private-network opt-in design.
