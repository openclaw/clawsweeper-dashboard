## Summary

Keep open: current main still accepts private/internal cron webhook URLs at the normalization and storage boundary, and this PR is a focused security hardening. It still needs contributor real behavior proof and maintainer/security reconciliation with the open private-network opt-in proposal.

## What This Changes

The PR applies the existing SSRF hostname/IP guard during cron webhook URL normalization and adds focused tests for private/internal rejection and public URL acceptance.

## Best Possible Solution

Land strict pre-persistence SSRF rejection with the shared guard after correcting the comment and adding real cron create/patch proof, or supersede it with an explicitly approved opt-in policy.

## Reproduction Assessment

Yes, at source level. Current main returns a trimmed http(s) URL from normalizeHttpWebhookUrl without hostname/IP classification, and create/patch validation stores that normalized target; I did not run a live cron flow in this read-only review.

## Solution Assessment

Mostly yes for a strict security posture. Reusing the shared SSRF classifier at the normalization boundary is narrow and maintainable, but maintainers still need to decide whether strict rejection or an opt-in private-network policy is the intended product direction.
