## Summary

Keep open: current main still accepts private/internal cron webhook URLs at persistence time, and this PR is a plausible narrow hardening, but it needs real behavior proof and maintainer security/product alignment with the open private-network opt-in proposal before merge.

## What This Changes

The PR adds the existing SSRF hostname/IP classifier to cron webhook URL normalization and adds focused tests for blocked private/internal targets plus public URL acceptance.

## Best Possible Solution

Land a default-deny pre-persistence guard only after maintainers decide how it should interact with any explicit private-network cron webhook opt-in, then require focused tests and redacted real behavior proof for the create/patch path.

## Reproduction Assessment

Yes. Source inspection shows current main accepts any parsed http(s) webhook URL in normalizeHttpWebhookUrl, and cron create/patch validation stores that normalized target through service/jobs.ts.

## Solution Assessment

Yes, for the default-deny hardening path: reusing isBlockedHostnameOrIp in normalizeHttpWebhookUrl is narrow and avoids duplicating SSRF rules. The remaining question is whether maintainers want this composed with an explicit cron private-network policy hook before merge.
