## Summary

Keep open: current main does not implement early cron webhook host/IP rejection, and the PR is a plausible security hardening direction, but it is not merge-ready because the changed validator conflicts with an existing gateway cron test/contract and the external PR lacks real behavior proof.

## What This Changes

The PR adds the shared SSRF host/IP blocker to cron webhook URL normalization and adds focused normalizer tests for private/internal webhook targets.

## Best Possible Solution

Keep the runtime DNS-pinned guard, add early persistence-time validation only after updating current-main cron regression expectations and settling the private-network opt-in policy.

## Reproduction Assessment

Yes. Source inspection shows current main accepts a loopback cron webhook URL at validation time and relies on `fetchWithSsrFGuard` at runtime; current-main `src/gateway/server-cron.test.ts:458` encodes that behavior.

## Solution Assessment

Not yet. Reusing the shared SSRF classifier is a good narrow implementation direction, but the PR needs current-main test/contract updates and real behavior proof before it is merge-ready.
