## Summary

Keep open. The PR is a focused security hardening for cron webhook URL persistence and current main still lacks the pre-persistence private/internal host check, but it needs contributor real behavior proof plus maintainer/security reconciliation with the private-network opt-in proposal before merge.

## What This Changes

The branch adds `isBlockedHostnameOrIp` to `normalizeHttpWebhookUrl` and adds cron webhook URL tests for blocked private/internal hosts and accepted public URLs.

## Best Possible Solution

Maintainers should choose one cron webhook network policy: strict reject-before-storage with runtime DNS-pinned SSRF preserved, or a documented private-network opt-in that also governs normalization.

## Reproduction Assessment

Yes. Source inspection on current main shows `normalizeHttpWebhookUrl` accepts private `http(s)` URLs after only protocol validation, and cron create/patch validation persists webhook delivery and failure-destination URLs through that helper.

## Solution Assessment

Mostly yes for the strict security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow and maintainable, but maintainers still need to decide how it interacts with private-network opt-in proposals.
