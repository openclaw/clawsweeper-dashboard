## Summary

Keep open: the PR is a focused security hardening for cron webhook URL persistence, and current main still only blocks these targets at the runtime fetch layer. It needs contributor real behavior proof plus maintainer/security reconciliation with the open private-network opt-in direction before merge.

## What This Changes

The PR applies the existing SSRF hostname/IP classifier during cron webhook URL normalization and adds tests for private/internal rejection and public URL acceptance.

## Best Possible Solution

Choose the cron webhook network posture explicitly, then land either strict pre-persistence rejection with the runtime guard preserved or a documented private-network opt-in that governs both persistence and delivery.

## Reproduction Assessment

Yes, from source inspection. Current main accepts private http(s) webhook URLs through normalizeHttpWebhookUrl and stores delivery/failure-destination targets through that helper; I did not run a live create/patch flow in this read-only review.

## Solution Assessment

Mostly yes for a strict security posture. Reusing the existing SSRF classifier at the normalization boundary is narrow and maintainable, but it is not the final answer if maintainers choose a documented private-network opt-in.
