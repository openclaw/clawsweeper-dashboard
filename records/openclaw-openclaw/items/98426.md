## Summary

Keep open. Current main has persistent compaction checkpoints plus manual branch/restore, but I did not find automatic restoration of the pre-compaction transcript when fallback later routes to a larger-context model, so the requested behavior still needs a maintainer policy decision.

## What This Changes

The issue requests preserving, rehydrating, or warning about pre-compaction context when fallback switches between models with smaller and larger context windows.

## Best Possible Solution

Define one fallback/context-window policy that uses checkpoint metadata to warn, branch, or explicitly restore larger-window context without silently mutating the canonical active transcript.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a medium-confidence path: fallback attempts reuse the active session and current compaction is persistent, but I did not run a live large-to-small-to-large provider fallback scenario.

## Solution Assessment

Is this the best way to solve the issue? Unclear: preserving pre-compaction context is plausible, but automatic rehydration of the active session needs an owner decision because manual checkpoint branch/restore already exists and config growth should be avoided unless necessary.
