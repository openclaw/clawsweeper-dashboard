## Summary

Keep open: the feature direction is valid, but the latest PR head still has a blocking styling mismatch. The renderer emits highlighted `<code>` inside `.code-block-wrapper`, while the new token color rules are scoped to `.code-block`, so visible WebChat syntax colors will not apply; the supplied terminal proof does not cover that browser behavior.

## What This Changes

The PR adds highlight.js-backed syntax highlighting to the Control UI markdown renderer, theme token styles, regression tests, a type shim, and a direct UI dependency.

## Best Possible Solution

Keep the renderer-level highlight.js approach, fix the CSS/DOM contract, then require a redacted WebChat browser screenshot or recording plus green focused checks before merge.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR emits `hljs` on `<code>` inside `.code-block-wrapper`, while the added selectors require a `.code-block` ancestor. That mismatch is enough to reproduce why visible token colors will not apply.

## Solution Assessment

No. The renderer-level highlighter is the right surface, but the CSS must match the generated DOM or the renderer must add the expected class before this is mergeable.
