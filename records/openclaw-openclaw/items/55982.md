## Summary

Keep open. Current main still lacks this cron option, and the PR is not mergeable as-is because normal add/edit flows would not persist the option and production Gateway wiring would skip configured jobs every time.

## What This Changes

The PR adds a `skipWhenIdle` cron option across CLI flags, protocol schema, cron types, timer skip logic, and service tests for main-session jobs.

## Best Possible Solution

Land a corrected version that persists `skipWhenIdle` as a cron job field, wires main-session user activity into Gateway cron construction, covers add/edit persistence and production wiring with tests, and documents the new CLI/API behavior before merging.

## Reproduction Assessment

Yes for the PR defects. The source-level reproduction is to follow the new CLI/protocol fields into `createJob` and `applyJobPatch`, then follow the timer's missing-data skip branch into production `buildGatewayCronService`, which provides no `getLastInboundAtMs` dependency.

## Solution Assessment

No, not as currently proposed. The concept is narrow and useful, but the implementation must persist the field and wire a real activity timestamp before the idle skip can be safe in production.
