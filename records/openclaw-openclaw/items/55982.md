- **[P1] Persist skipWhenIdle in cron create and patch paths:** `src/cli/cron-cli/register.cron-add.ts:255`
  - body: The CLI/schema now send `skipWhenIdle`, but `createJob` constructs persisted jobs field-by-field and `applyJobPatch` only handles known top-level fields. Without copying this field in both service paths, `cron add --skip-when-idle` and `cron edit --skip-when-idle` are accepted but the option is dropped before the timer can observe it.
  - confidence: 0.94
- **[P1] Wire real activity data before failing closed:** `src/cron/service/timer.ts:1064-1072`
  - body: This branch treats missing `getLastInboundAtMs` data as idle and skips the run, but production `buildGatewayCronService` does not provide that callback. Any persisted job with `skipWhenIdle` would therefore skip every run in the normal gateway service until a real main-session activity timestamp is supplied or the missing-data policy changes.
  - confidence: 0.92
- **[P2] Regenerate the protocol artifacts for the new cron field:** `src/gateway/protocol/schema/cron.ts:268`
  - body: Adding `skipWhenIdle` to the gateway cron schema changes the protocol contract, but the generated Swift protocol models and schema artifacts are not updated in this PR. `protocol:check` expects those generated files to match, and Swift clients will not expose the new field until the generated models are refreshed.
  - confidence: 0.86
- **[P3] Document the skipWhenIdle cron behavior:** `src/cli/cron-cli/register.cron-add.ts:100-104`
  - body: This adds a user-facing CLI/API option, but the cron docs do not explain the default threshold, main-session-only scope, skipped-run status, activity timestamp source, or missing-data behavior. Users need those details to operate the option safely.
  - confidence: 0.84
