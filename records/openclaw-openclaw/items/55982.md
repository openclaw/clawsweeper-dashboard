## Summary

Keep this PR open: it is useful source work, but the current patch still has two source-level P1 blockers before the new cron option can work through normal CLI/API paths.

## What This Changes

The PR adds a `skipWhenIdle` cron job option with CLI add/edit flags, gateway schema support, timer skip logic, and focused tests for skipping main-session jobs during idle periods.

## Best Possible Solution

Complete the PR by persisting the new field, wiring idle checks to the main session's `lastInteractionAt`, documenting the option and changelog entry, then revalidating the cron service/gateway schema paths.

## Reproduction Assessment

Yes, source-level. The PR accepts and sends `skipWhenIdle`, while the service create/patch paths still do not persist that field and production cron construction does not provide the activity callback used by the new skip branch.

## Solution Assessment

No. The direction is maintainable, but the patch is incomplete until the option survives add/edit and uses real main-session activity data instead of failing closed with an unwired optional callback.

- **[P1] Persist skipWhenIdle through cron create and update:** `src/gateway/protocol/schema/cron.ts:299-314`
  - body: This schema change accepts `skipWhenIdle`, and the CLI sends it, but `createJob` still builds `CronJob`s field-by-field and `applyJobPatch` has no `skipWhenIdle` branch. As written, `cron add --skip-when-idle` and `cron edit --skip-when-idle/--no-skip-when-idle` drop the option before the timer can observe it.
  - confidence: 0.94

Any persisted job with `skipWhenIdle` would therefore skip every run until the callback is wired to the target main session's activity timestamp or the unavailable-data behavior changes.
  - confidence: 0.91

  - body: This adds a user-facing CLI/API cron feature, but the patch does not update cron docs or `CHANGELOG.md`. Add the option, default threshold, main-session scope, and skipped-run behavior so users can discover and operate it.
  - confidence: 0.86
