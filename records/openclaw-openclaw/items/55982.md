Keep open: current main does not implement cron idle skipping, and the PR still has blocking correctness and proof gaps around persistence, production activity wiring, and protocol client drift.

## What This Changes

The PR adds a cron `skipWhenIdle` job option across CLI add/edit, cron types, timer execution, gateway protocol schema, and cron service tests.

## Best Possible Solution

Land a complete idle-skip feature only after it persists the job option, wires a canonical main-session activity timestamp, updates generated protocol clients, and settles the missing-data policy.

## Reproduction Assessment

Yes, source-level. The PR sends and validates `skipWhenIdle`, while current cron create/patch persistence omits the field and production cron construction does not provide the activity callback used by the new timer branch.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best mergeable shape until persistence, canonical activity wiring, generated protocol artifacts, and missing-data semantics are handled together.

  - body: The CLI now sends `skipWhenIdle`, but `createJob` builds stored jobs field by field and `applyJobPatch` never handles this new field. Since this PR does not update `src/cron/service/jobs.ts`, `cron add` and `cron edit` accept the option but drop it before the timer can observe it.

  - confidence: 0.92

  - confidence: 0.86
