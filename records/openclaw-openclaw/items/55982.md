## Summary

Keep this PR open. Current main at f3145f6db865d118c0042b8adfa11abad22622c7 still does not implement the proposed `skipWhenIdle` cron option, so the PR is not obsolete, but the branch has blocking correctness issues: normal add/edit paths do not persist the option, and Gateway does not wire the activity callback even though the timer treats missing activity data as idle. The earlier Greptile findings for one-shot jobs and `idleMs` validation appear addressed in the branch's second commit, but the later persistence and production wiring blockers still need follow-up.

## What This Changes

The PR adds a `skipWhenIdle` cron job option across cron types, Gateway protocol schema, CLI add/edit flags, timer skip logic, and skip-when-idle regression tests.

## Best Possible Solution

Keep the feature on this PR track, but require the branch to make `skipWhenIdle` a real persisted cron field, wire a reliable main-session last-inbound activity source into Gateway cron construction, document the new CLI/API behavior, and add regression coverage for add/edit persistence, direct protocol validation, production Gateway wiring, recurring main-session skips, and one-shot schedule behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on the PR branch, `cron add/edit` can set `skipWhenIdle`, but `createJob` and `applyJobPatch` do not store it; separately, `buildGatewayCronService` does not pass `getLastInboundAtMs` while the timer skips when that dependency is missing.

## Solution Assessment

No. The proposed direction is useful, and the one-shot guard plus `idleMs >= 1000` validation are good, but exposing the CLI/schema and timer check before persistence and production activity wiring makes the feature ineffective or permanently skipping in real Gateway runs.
