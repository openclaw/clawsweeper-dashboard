## Summary

Keep open. The PR addresses a real cron/main-session contamination class, but current source review still shows blocking regressions in no-relay cron wakes, heartbeat response-tool mode, legacy job maintenance, and public cron contract alignment, with contributor real behavior proof still missing.

## What This Changes

The PR constrains main-session cron systemEvent text to short internal wake tokens, changes cron heartbeat prompts to a SYSTEM_WAKE shape, updates cron tool/system prompt guidance, and adjusts focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased cron hardening that removes duplicate/rich main-session reminder prose while preserving no-relay and heartbeat response-tool semantics, keeps legacy jobs operable, aligns docs/tool guidance, and includes redacted real cron or Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main enqueues main-session cron systemEvent text, heartbeat classifies cron-tagged events and builds a cron-event prompt, and reply setup can drain queued events into System lines; I did not run a live Telegram or cron flow in this read-only review.

## Solution Assessment

No, not as submitted. Tokenizing main-session cron prose is a useful direction, but this branch weakens relay-disabled and response-tool behavior, can block legacy job maintenance, and changes public cron semantics without real behavior proof.
