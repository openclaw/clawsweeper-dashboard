## Summary

Keep open. The underlying cron prompt contamination path is still source-reproducible on current main, but this PR is not mergeable as-is: it drops no-relay and response-tool semantics, blocks benign edits on legacy jobs, conflicts with the current documented cron contract, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR tokenizes/redacts main-session cron systemEvent text, rejects rich main-session reminder prose, updates cron tool/system prompt guidance, and adds focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased cron hardening that tokenizes main-session cron text while preserving no-relay and heartbeat response-tool semantics, permits disabling or repairing legacy jobs, updates the public/tool contracts, and includes redacted real Telegram or cron proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main accepts main-session systemEvent text unchanged and wraps cron events into scheduled-reminder relay prose, but I did not run a live cron or Telegram job in this read-only review.

## Solution Assessment

No, not as-is. The tokenization direction is appropriate, but the PR drops current no-relay and response-tool invariants, blocks legacy operational edits, conflicts with documented usage, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore internal-only no-relay cron wakes:** `src/infra/heartbeat-events-filter.ts:23`
  - body: `buildCronEventPrompt` ignores `_opts.deliverToUser` and always tells the model to follow up when context requires it. For target-none or alerts-disabled cron wakes, that can leave user-facing follow-up text in the main-session transcript even though delivery was disabled, so preserve the strict internal `HEARTBEAT_OK` path for no-relay cron wakes.
  - confidence: 0.91
- **[P2] Preserve response-tool cron prompts:** `src/infra/heartbeat-events-filter.ts:11-13`
  - body: The new `_opts` shape only carries `deliverToUser` and the function never accepts or reads `useHeartbeatResponseTool`. Current main passes that option into cron prompt construction, so a rebase would regress structured heartbeat response-tool mode back to literal text sentinel behavior.
  - confidence: 0.86
- **[P2] Permit benign updates on legacy main jobs:** `src/cron/service/jobs.ts:147`
  - body: The new safety assertion runs inside `assertSupportedJobSpec`, which is called after patches as well as creation. A persisted legacy main-session reminder job with rich text could become impossible to disable or repair through ordinary metadata patches even though execution now degrades unsafe text to a fallback token.
  - confidence: 0.89
- **[P2] Align docs before rejecting documented reminders:** `src/cron/main-session-system-event.ts:15-16`
  - body: The validator rejects reminder-style main-session `systemEvent` text, but current docs and tool guidance still show `--session main --system-event` reminder prose. Landing this without updating the public/tool contract would make documented usage fail or silently downgrade.
  - confidence: 0.84
