## Summary

Keep open. The remediation targets a real cron/main-session safety problem, but this PR is stale against current main and has blocking regressions around no-relay cron wakes, exec-event prompts, legacy job edits, and documented cron examples.

## What This Changes

The PR adds main-session cron systemEvent token validation/redaction, rewrites cron heartbeat prompts to a minimal wake token, updates cron tool/system-prompt guidance, and adds focused tests.

## Best Possible Solution

Rebase or replace the remediation so main-session cron wakes use token-only text while preserving current no-relay and exec-event invariants, allowing legacy job disable/repair edits, and aligning docs, changelog, tool guidance, and regression tests.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: main-session systemEvent text is accepted unchanged, passed into `buildCronEventPrompt`, and wrapped with relay-style reminder prose; the PR-specific blockers are visible directly in the PR diff and current-main tests.

## Solution Assessment

No. The direction is the right narrow boundary, but this implementation is not the best fix as-is because it regresses no-relay behavior, current exec-event handling, legacy job operability, and documented examples.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore internal-only no-relay cron wakes:** `src/infra/heartbeat-events-filter.ts:25`
  - body: `buildCronEventPrompt` now ignores `_opts.deliverToUser` and always tells the model to follow up if session context requires it. In no-target or alerts-disabled heartbeat runs, delivery is skipped but the model turn can still be generated and retained, so disabling relay no longer forces the internal-only `HEARTBEAT_OK` behavior that avoids main-session residue.
  - confidence: 0.91
- **[P1] Preserve current exec event prompt handling:** `src/infra/heartbeat-events-filter.ts:29`
  - body: This PR head carries the older one-argument `buildExecEventPrompt` shape. Current main passes pending exec events into this helper and relies on structured completion filtering, missing-output safeguards, and truncation, so landing this stale version would drop command output and failure details from exec heartbeats.
  - confidence: 0.86
- **[P2] Allow operational edits to legacy main jobs:** `src/cron/service/jobs.ts:144`
  - body: `assertSupportedJobSpec` now rejects every main-session job whose stored text is rich reminder prose, and `applyJobPatch` calls this assertion even for unrelated updates like toggling `enabled`. Existing legacy jobs can become impossible to disable or repair even though execution is designed to degrade unsafe text to a fallback token.
  - confidence: 0.88
- **[P2] Update docs before rejecting documented reminders:** `src/cron/main-session-system-event.ts:43-45`
  - body: The new validator rejects reminder-style main-session `systemEvent` text, but the current cron docs still show `--session main --system-event "Reminder: ..."` examples. Users following the published examples would hit the new validation error unless the docs and examples move rich reminders to isolated `agentTurn` jobs or use short wake tokens.
  - confidence: 0.84
