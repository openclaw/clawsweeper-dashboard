## Best Possible Solution

Full review comments:

- **[P1] Restore internal-only no-relay cron wakes:** `src/infra/heartbeat-events-filter.ts:23`
  - body: `buildCronEventPrompt` now ignores `deliverToUser`, so target-none or alerts-disabled cron wakes still ask the model for follow-up when session context requires it. Those turns can remain in the main-session transcript even when no user delivery occurs, so relay-disabled cron wakes need the strict internal `HEARTBEAT_OK` path preserved.
  - confidence: 0.91

  - body: Current main passes `useHeartbeatResponseTool` into cron prompt construction, but this branch accepts `_opts` and always emits a literal `Reply HEARTBEAT_OK` instruction. In response-tool mode, cron wakes should continue using the structured heartbeat response tool instruction instead of falling back to text-only sentinel behavior.
  - confidence: 0.86
- **[P2] Permit operational edits on legacy main jobs:** `src/cron/service/jobs.ts:147`
  - body: The new safety assertion runs from `assertSupportedJobSpec`, which `applyJobPatch` calls after every patch. A persisted legacy main-session reminder job with rich text would become impossible to disable or repair through ordinary metadata patches, even though execution degrades unsafe text to a fallback token.
  - confidence: 0.89
- **[P2] Align docs before rejecting documented reminders:** `src/cron/main-session-system-event.ts:15-16`
  - body: The validator rejects reminder-style main-session `systemEvent` text, but the public cron docs still show `--session main --system-event` reminder prose. Landing this without updating the public CLI/docs contract makes documented usage fail or silently downgrade.
  - confidence: 0.84
