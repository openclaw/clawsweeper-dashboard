## Summary

Keep open: the remediation is directionally useful, but this PR is still incorrect against current main because it drops no-relay and response-tool cron semantics, can block legacy main-job edits, lacks real behavior proof, and is merge-conflicting.

## What This Changes

The PR adds a main-session cron systemEvent token validator, rewrites cron heartbeat prompts to a minimal SYSTEM_WAKE form, updates cron tool/system prompt guidance, and adjusts focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased hardening that removes duplicate/rich main-session cron prose while preserving no-relay and response-tool semantics, keeps legacy jobs operable, aligns docs/tool guidance, and includes real cron or Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main enqueues main-session cron systemEvent text with a cron context key, heartbeat selects and wraps cron events, and reply setup drains queued events into System lines; I did not run a live Telegram or cron flow in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing main-session cron prose is a useful direction, but this branch weakens relay-disabled and response-tool behavior, blocks legacy job maintenance, and changes public cron semantics without enough docs or real behavior proof.
