## Summary

Keep open: the PR targets a real current-main cron/main-session contamination path, but it is not mergeable as-is because the branch regresses no-relay cron semantics, is stale against current exec-event handling, and would make some legacy main-session cron jobs hard to disable or repair.

## What This Changes

The PR adds main-session cron systemEvent token validation/redaction, rewrites cron heartbeat prompts to minimal wake tokens, updates agent cron guidance, and adds focused tests.

## Best Possible Solution

Rebase or replace the remediation so main-session cron wakes use token-only text while preserving no-relay and exec-event invariants, allowing legacy job disable/repair edits, and updating docs, changelog, tool guidance, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows main-session systemEvent text is accepted unchanged, routed into `buildCronEventPrompt`, and wrapped with relay-style reminder prose.

## Solution Assessment

No. The direction is the right narrow boundary, but this implementation is not the best fix as-is because it weakens no-relay handling, risks dropping current exec-event safeguards, blocks benign legacy-job edits, and leaves published cron examples inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore internal-only no-relay cron wakes:** `src/infra/heartbeat-events-filter.ts:25-27`
  - body: `buildCronEventPrompt` ignores `deliverToUser` and always allows follow-up when session context requires it. In no-target or alerts-disabled heartbeat runs, that can still create and retain a main-session turn, so relay-disabled cron wakes need the strict internal `HEARTBEAT_OK` path preserved.
  - confidence: 0.91
- **[P1] Preserve current exec event prompt handling:** `src/infra/heartbeat-events-filter.ts:29`
  - body: The PR branch is stale relative to current main's structured exec completion parsing, missing-output safeguards, response-tool prompts, and truncation. Landing or resolving conflicts with this older helper shape would drop command output and failure details from exec heartbeats.
  - confidence: 0.86
- **[P2] Allow benign edits to legacy main jobs:** `src/cron/service/jobs.ts:144`
  - body: `assertSupportedJobSpec` now rejects every main-session job whose stored text is rich reminder prose, and `applyJobPatch` calls it even for unrelated updates like toggling `enabled`. Existing legacy jobs can become impossible to disable or repair even though execution degrades unsafe text to a fallback token.
  - confidence: 0.88
- **[P2] Update docs before rejecting documented reminders:** `src/cron/main-session-system-event.ts:43-45`
  - body: The validator rejects reminder-style main-session `systemEvent` text, but the published cron guide still tells users to create main-session reminders with rich `--system-event` prose. The docs and examples need to move those reminders to isolated `agentTurn` jobs or show short wake tokens.
  - confidence: 0.84
