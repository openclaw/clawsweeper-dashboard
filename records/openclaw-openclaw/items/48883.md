## Summary

Keep open: current main still lacks the central cron/main-session hardening, but this branch is not merge-ready because it weakens no-relay and response-tool cron wake semantics, can strand legacy main-session jobs, and lacks required real behavior proof for a privacy-sensitive Telegram-visible path.

## What This Changes

The PR adds a main-session cron systemEvent token validator, rewrites cron heartbeat prompts to a minimal SYSTEM_WAKE form, updates cron prompt guidance, and adjusts cron/heartbeat tests.

## Best Possible Solution

Land a rebased hardening that removes rich main-session cron prose while preserving no-relay and response-tool semantics, keeps legacy jobs operable, aligns docs/tool guidance, and includes real cron or Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main enqueues main-session cron systemEvent text, heartbeat selects cron events and wraps them into the model prompt; I did not run a live Telegram or cron flow in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing main-session cron prose is a useful direction, but this branch drops existing no-relay and response-tool behavior, blocks benign legacy-job edits, and needs docs plus real behavior proof.

  - confidence: 0.86

  - confidence: 0.89
