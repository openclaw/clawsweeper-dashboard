## Summary

Keep open: the underlying cron/main-session contamination path is still source-reproducible on current main, but this PR is not merge-ready because it weakens no-relay and response-tool semantics, blocks benign legacy-job updates, changes the documented cron contract, and lacks real behavior proof.

## What This Changes

The PR adds main-session cron systemEvent token validation, rewrites cron heartbeat prompts to a `SYSTEM_WAKE` shape, updates cron tool/system prompt guidance, and adds focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased hardening that tokenizes main-session cron text while preserving no-relay and response-tool behavior, keeps legacy jobs operable, aligns docs/tool guidance, and includes redacted real Telegram or cron proof.

## Reproduction Assessment

Yes. Source inspection on current main shows rich main-session systemEvent text is accepted and cron events are wrapped into scheduled-reminder relay prose; I did not run a live Telegram or cron job in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing main-session cron text is the right direction, but this patch drops established no-relay and response-tool behavior, makes legacy main jobs harder to disable or repair, and does not update the public contract it changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore internal-only no-relay cron wakes:** `src/infra/heartbeat-events-filter.ts:23`
  - body: `buildCronEventPrompt` ignores `_opts.deliverToUser` and always permits follow-up when session context requires it. For target-none or alerts-disabled cron wakes, that can leave user-facing follow-up text in the main-session transcript even though delivery was disabled.
  - confidence: 0.91
- **[P2] Preserve response-tool cron prompts:** `src/infra/heartbeat-events-filter.ts:11-13`
  - body: The new options shape no longer accepts or reads `useHeartbeatResponseTool`. Current main passes that option into cron prompt construction, so this would regress structured heartbeat response-tool mode back to literal text sentinel behavior.
  - confidence: 0.86
- **[P2] Allow benign edits to legacy main jobs:** `src/cron/service/jobs.ts:147`
  - body: The new safety assertion runs inside `assertSupportedJobSpec`, which `applyJobPatch` calls even for unrelated patches. A persisted legacy main-session reminder job with rich text could become impossible to disable or repair even though execution now degrades unsafe text.
  - confidence: 0.89
- **[P2] Align public cron docs before rejecting reminders:** `src/cron/main-session-system-event.ts:15-16`
  - body: The validator rejects reminder-style main-session `systemEvent` text, while current public docs and tool guidance still show that usage. Landing this without a docs/tool contract update would make documented reminder examples fail or silently downgrade.
  - confidence: 0.84
