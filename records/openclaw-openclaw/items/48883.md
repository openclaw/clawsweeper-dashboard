## Summary

Keep open: this PR targets a real main-session cron contamination/privacy class, but the current diff still regresses cron wake semantics, can block legacy job maintenance, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a main-session cron systemEvent validator, rewrites cron heartbeat prompts to a minimal SYSTEM_WAKE token, shifts rich reminders toward isolated agentTurn jobs, and updates focused tests/tool guidance.

## Best Possible Solution

Land a rebased hardening that strips rich main-session cron prompt content while preserving no-relay and response-tool behavior, keeps legacy jobs operable, updates the public cron/tool contract, and includes redacted real cron or Telegram proof.

## Reproduction Assessment

Yes, at source level: a main-session systemEvent cron job on current main enqueues rich text, heartbeat selects cron events, and buildCronEventPrompt wraps that text into the main-session prompt. I did not run a live Telegram or cron flow in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing rich main-session cron prose is directionally useful, but this patch drops no-relay and response-tool semantics, validates legacy jobs too broadly, and lacks aligned docs plus real behavior proof.

Overall correctness: patch is incorrect

Overall confidence: 0.91

  - confidence: 0.91

  - body: The new cron prompt options shape drops `useHeartbeatResponseTool`, while current main passes that option into cron prompt construction. Structured heartbeat response-tool mode would regress to literal HEARTBEAT_OK sentinel behavior for cron events.
  - confidence: 0.86
- **[P2] Allow benign edits to legacy main jobs:** `src/cron/service/jobs.ts:147`
  - body: `assertSupportedJobSpec` now validates existing main-job text for every patch after merge. Persisted rich main-session reminder jobs can become impossible to disable or repair even though execution would otherwise degrade unsafe text to a fallback wake token.
  - confidence: 0.89
- **[P2] Align the public cron contract before rejecting reminders:** `src/cron/main-session-system-event.ts:15-16`
  - body: The validator rejects reminder-style main-session `systemEvent` text, but current public docs still show main-session reminder/system-event usage, including punctuation this validator rejects. Landing this without docs and tool contract alignment would make documented examples fail or silently downgrade.
  - confidence: 0.84
