## Summary

The PR rewrites main-session cron event prompts into minimal SYSTEM_WAKE tokens, adds validation for main-session systemEvent text, changes cron tool guidance/context handling, and updates focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased hardening that removes duplicate or rich main-session cron prose while preserving no-relay and heartbeat response-tool semantics, keeping legacy jobs operable through compatible repair paths, and adding real cron or Telegram proof.

Yes, at source level. Current main still queues cron-tagged main-session system events and wraps cron pending events in heartbeat prompt construction; I did not run a live Telegram or cron scenario in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing main-session cron prose is a plausible mitigation, but this branch needs a compatibility-preserving rebase, maintainer agreement on the public cron contract, and real behavior proof before it is the best fix.
