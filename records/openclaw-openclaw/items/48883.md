## Summary

Keep open: current main still has the cron/main-session contamination path and this PR targets a real bug class, but the branch is not mergeable as-is because it lacks real behavior proof and introduces no-relay, response-tool, legacy-job, and docs/contract regressions. The narrower duplicate-wrapper work at https://github.com/openclaw/openclaw/pull/44973 does not fully supersede this PR's rich-text hardening.

## What This Changes

The PR adds a main-session cron systemEvent token validator, rewrites cron heartbeat prompts to a SYSTEM_WAKE form, updates cron tool/system prompt guidance, and adjusts focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased hardening that removes duplicate/rich main-session cron prose while preserving no-relay and response-tool semantics, keeping legacy jobs operable, updating docs/tool guidance, and adding real Telegram or cron proof.

## Reproduction Assessment

Yes, at source level. Current main enqueues main-session cron systemEvent text with a `cron:` context key, heartbeat peeks and wraps it, and reply setup drains queued events into `System:` lines; I did not run a live Telegram or cron flow in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing main-session cron prose is directionally useful, but this branch weakens no-relay and response-tool behavior, blocks legacy job maintenance, and changes public cron semantics without sufficient docs or proof.
