## Summary

Keep open: the safety direction addresses a source-reproducible cron/main-session contamination bug, but this PR is not mergeable as-is because it weakens no-relay cron wake semantics, is stale against current exec-event prompt safeguards, blocks some legacy job maintenance, conflicts with current docs/tool guidance, and lacks required real behavior proof.

## What This Changes

The PR constrains main-session cron systemEvent text to short wake tokens, redacts rich reminder prose from cron heartbeat prompts, updates agent cron guidance, and adds focused tests.

## Best Possible Solution

Land a rebased narrow cron hardening that tokenizes main-session cron text, preserves no-relay and exec response-tool invariants, allows legacy jobs to be disabled or repaired, aligns docs/tool guidance, and includes redacted real cron or Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current main shows main-session systemEvent text is accepted unchanged and then wrapped into scheduled-reminder relay prose by the cron heartbeat prompt path; I did not run a live cron job because this was a read-only review.

## Solution Assessment

No. The remediation direction is appropriate, but this branch is not the best fix as-is because it drops no-relay semantics, is stale against current exec-event prompt handling, blocks legacy operational edits, conflicts with current docs/tool guidance, and lacks real proof.
