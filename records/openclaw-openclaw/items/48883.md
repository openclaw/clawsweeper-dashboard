## Summary

Keep open. The PR addresses a real source-reproducible cron/main-session contamination path, but the current branch still has blocking behavior regressions around no-relay wakes, response-tool mode, legacy job edits, public cron contract alignment, and missing real behavior proof.

## What This Changes

The PR adds a main-session cron systemEvent token validator, rewrites cron heartbeat prompts to minimal SYSTEM_WAKE tokens, updates cron tool/system-prompt guidance, and adjusts focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased hardening that removes duplicate or rich main-session cron prose while preserving no-relay and response-tool semantics, keeps legacy jobs operable, aligns docs/tool guidance, and includes redacted real cron or Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main enqueues main-session cron systemEvent text with a cron context key, heartbeat selects and wraps cron events, and reply setup can drain queued events into System lines; I did not run a live Telegram or cron flow in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing main-session cron prose is a useful direction, but this branch weakens relay-disabled and response-tool behavior, blocks legacy job maintenance, and changes public cron semantics without enough docs or real behavior proof.
