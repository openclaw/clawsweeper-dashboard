## Summary

Keep open. The PR targets a real current-main cron prompt contamination path, but the branch is conflicting, lacks after-fix real behavior proof, and introduces blocking regressions in no-relay, response-tool, and legacy-job behavior while changing the documented cron contract.

## What This Changes

The PR adds a main-session cron systemEvent validator/tokenizer, rewrites cron heartbeat prompts to `SYSTEM_WAKE`, updates cron tool/system prompt guidance, and adds focused cron/heartbeat tests.

## Best Possible Solution

Land a rebased hardening that tokenizes main-session cron text while preserving no-relay and response-tool semantics, allows disabling or repairing legacy jobs, aligns public docs/tool guidance, and includes redacted real Telegram or cron proof.

## Reproduction Assessment

Yes. Source inspection shows current main accepts rich main-session systemEvent text and wraps cron events into scheduled-reminder relay prose, but I did not run a live Telegram or cron job in this read-only review.

## Solution Assessment

No, not as-is. Tokenizing main-session cron text is the right direction, but this patch drops existing no-relay and response-tool invariants, blocks benign legacy-job edits, conflicts with current docs, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore internal-only no-relay cron wakes:** `src/infra/heartbeat-events-filter.ts:23`
  - body: `buildCronEventPrompt` ignores `_opts.deliverToUser` and always tells the model to follow up when context requires it. For target-none or alerts-disabled cron wakes, that can leave user-facing follow-up text in the main-session transcript even though delivery was disabled, so preserve the strict internal `HEARTBEAT_OK` path for no-relay cron wakes.
  - confidence: 0.91
- **[P2] Preserve response-tool cron prompts:** `src/infra/heartbeat-events-filter.ts:11-13`
  - body: The new `_opts` shape only carries `deliverToUser`, and the function never accepts or reads `useHeartbeatResponseTool`. Current main passes that option into cron prompt construction, so a rebase would regress structured heartbeat response-tool mode back to literal text sentinel behavior.
  - confidence: 0.86
- **[P2] Allow benign edits to legacy main jobs:** `src/cron/service/jobs.ts:147`
  - body: The new safety assertion runs inside `assertSupportedJobSpec`, which `applyJobPatch` calls even for unrelated patches. A persisted legacy main-session reminder job with rich text could become impossible to disable or repair through ordinary metadata patches even though execution now degrades unsafe text to a fallback token.
  - confidence: 0.89
- **[P2] Align public cron docs before rejecting reminders:** `src/cron/main-session-system-event.ts:15-16`
  - body: The validator rejects reminder-style main-session `systemEvent` text, but the current public cron docs still show `--session main --system-event` reminder prose. Landing this without a docs/tool contract update would make documented usage fail or silently downgrade.
  - confidence: 0.84
