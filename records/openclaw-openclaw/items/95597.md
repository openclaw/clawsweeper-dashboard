## Label Justifications

- `P2`: The issue affects bundled plugin middleware/artifact capture for Codex native tool results, but it does not make the core runtime unavailable.
- `impact:data-loss`: Tokenjuice metadata artifacts for completed Codex native tool results can be silently missing when the middleware-only plugin is skipped.

## Summary

Keep open. Current main and the latest shipped release still gate Codex native `post_tool_use` work on legacy `after_tool_call` hooks and dispatch only that hook path, while Tokenjuice declares `agentToolResultMiddleware` for `codex` and the Codex dynamic-tool path already runs the middleware runner.

## What This Changes

The issue reports that Codex native `post_tool_use` relay skips `codex` `agentToolResultMiddleware`, preventing Tokenjuice artifact metadata from advancing after the v2026.6.9 upgrade.

## Best Possible Solution

Add a native `post_tool_use` middleware bridge for runtime `codex` that treats middleware owners as relay work, adapts Codex tool responses into the OpenClaw tool-result shape for middleware, preserves legacy `after_tool_call`, and tests the middleware-only case.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: register a Codex native relay with only a `codex` `agentToolResultMiddleware` owner and no `after_tool_call` hook, then `shouldRelayEvent("post_tool_use")` remains false on current main. I did not run the full live Tokenjuice artifact workflow.

## Solution Assessment

Yes, the best fix belongs in the native relay rather than Tokenjuice because the relay currently decides whether Codex should send the event and owns the native payload adapter. The fix should be narrow and avoid changing dynamic-tool or Copilot hook semantics.
