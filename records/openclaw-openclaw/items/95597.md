## Summary

Keep open. Current main and v2026.6.9 still relay Codex native `post_tool_use` only when legacy `after_tool_call` hooks exist, and the native post-tool handler never invokes the `codex` tool-result middleware contract that Tokenjuice registers.

## What This Changes

The issue requests that Codex native `post_tool_use` relay treat `agentToolResultMiddleware` registrations as local work and run Codex native tool results through the middleware pipeline.

## Best Possible Solution

Extend the generic native Codex relay to detect `codex` tool-result middleware owners as post-tool work, run native tool results through the existing middleware runner, and cover the middleware-only case without Tokenjuice-specific logic.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 gate native `post_tool_use` on legacy `after_tool_call`, while Tokenjuice declares `agentToolResultMiddleware` for `codex` and the native post handler never invokes that runner. I did not run the live Codex/Tokenjuice artifact command during this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make the native Codex relay honor the existing generic tool-result middleware contract while preserving the no-work relay optimization. A Tokenjuice-specific workaround would be the wrong layer because the skipped contract is in core relay dispatch.
