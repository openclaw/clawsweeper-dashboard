## Summary

Keep open. Current main and v2026.6.9 still skip middleware-only Codex native post-tool work, and an open draft fix PR now references this issue but has not merged.

## What This Changes

Restore Codex native PostToolUse relay participation in the existing codex agentToolResultMiddleware contract so Tokenjuice can capture or compact native tool results.

## Best Possible Solution

Land a narrow native relay fix that treats codex tool-result middleware as local post-tool work, invokes the existing middleware runner for native PostToolUse, and preserves the no-work relay optimization.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 gate native post_tool_use on legacy after_tool_call, while Tokenjuice declares codex agentToolResultMiddleware and the native handler never invokes that runner. I did not run a live Codex/Tokenjuice artifact capture in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to honor the existing generic tool-result middleware contract in the native Codex relay; a Tokenjuice-specific workaround would be the wrong layer.
