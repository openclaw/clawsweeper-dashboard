## Summary

Keep open: this is a useful candidate fix for the no-finish OpenAI-compatible tool-call issue, but the current patch is not merge-ready because it promotes native tool calls after normal iterator EOF without proving that the stream reached a clean `[DONE]` terminal marker.

## What This Changes

The PR changes OpenAI-compatible transport stream finalization so native streamed `delta.tool_calls` can be promoted to `toolUse` even when no final `finish_reason` chunk is present, with adjacent regression tests.

## Best Possible Solution

Carry a positive clean-terminal signal, such as observed `[DONE]`, into the transport finalizer before promoting no-finish native tool calls, while keeping EOF-only streams fail-closed.

## Reproduction Assessment

Yes, at source level: current main has a finalizer branch and regression test that drop no-finish native streamed tool calls, matching the linked issue. I did not run a live Evolink route in this read-only review.

## Solution Assessment

No. The transport finalizer is the plausible layer, but the best fix needs positive clean-terminal evidence or an explicit maintainer decision instead of treating SDK iterator EOF as sufficient.

Full review comments:

- **[P1] Require a clean stream terminal before no-finish tool use:** `src/agents/openai-transport-stream.ts:3261`
  - body: The new `sawNativeToolCallDelta` path still promotes after ordinary async-iterator EOF. In OpenAI SDK 6.39.1, `[DONE]` is consumed internally and ordinary body EOF also ends the iterator, so a response that closes after a parseable native tool-call delta but before `[DONE]` or `finish_reason` can now execute that tool.
