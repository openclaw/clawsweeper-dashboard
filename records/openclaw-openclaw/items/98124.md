## Summary

Keep open: the linked bug is real and this PR targets the right stream finalizer, but the patch removes the only terminal-signal guard and can promote no-finish tool calls after ordinary EOF even when the SDK never exposed a `[DONE]` marker.

## What This Changes

The PR changes the OpenAI-compatible transport stream finalizer to preserve silent parsed tool calls when no `finish_reason` is received, and updates adjacent parser tests for no-finish and visible-text cases.

## Best Possible Solution

Preserve the reported no-finish provider compatibility only when the transport has positive clean-terminal evidence, such as a tracked `[DONE]` marker or an explicit provider finish reason, while keeping EOF-only streams fail-closed.

## Reproduction Assessment

Yes at source level: current main has a named test and finalizer branch that drop no-finish streamed tool-call blocks, matching the linked issue. I did not run tests or a live Evolink route because this review is read-only.

## Solution Assessment

No. The target finalizer is the right layer, but the PR uses iterator EOF as proof of clean stream completion; the safer fix must distinguish `[DONE]` from EOF before promoting tool calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require a real DONE signal before promoting no-finish tools:** `src/agents/openai-transport-stream.ts:3249`
  - body: This condition now treats any normal iterator EOF as a clean stream, but the OpenAI SDK consumes `[DONE]` internally and also marks the stream done after `_iterSSEMessages` simply exhausts the response body. A response that closes after a parseable tool-call chunk but before `[DONE]` or `finish_reason` will now run the tool instead of staying fail-closed. Please carry an explicit clean-terminal signal from the SSE layer, or keep EOF-only no-finish streams stripped.
  - confidence: 0.87
