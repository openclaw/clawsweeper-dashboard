## Summary

Keep open: current main still contains a source-visible Telegram polling liveness gap matching the report, and the transport-rebuild silence is plausible enough to need a targeted fix rather than cleanup close.

## What This Changes

The issue reports Telegram long-poll stalls in 2026.5.4 and 2026.5.5 where outbound API activity masks stall detection and dirty transport rebuild can go silent before a new polling cycle starts.

## Best Possible Solution

Adjust Telegram polling liveness so successful outbound API calls cannot indefinitely mask missing getUpdates progress, while preserving in-flight reply protection and adding focused diagnostics or bounded handling for the next-cycle startup path after dirty transport rebuild.

## Reproduction Assessment

Yes for the masked watchdog part: current source and tests show stale getUpdates plus recent sendMessage returns no stall. The full post-rebuild live wedge is not directly reproduced here and needs a focused regression or live proof.

## Solution Assessment

Unclear as written: dropping apiElapsed entirely may be too blunt because it can regress the in-flight outbound reply protection added earlier. The better fix is to separate completed outbound success from active in-flight delivery protection, then add diagnostics or a bounded guard around the rebuild-to-next-cycle path.

## Repair Work Prompt

Fix the Telegram polling stall masking reported in https://github.com/openclaw/openclaw/issues/78473. On current main, `TelegramPollingLivenessTracker.detectStall` returns null when getUpdates is stale but a recent successful non-getUpdates API call updated API liveness; change the watchdog logic so completed outbound API success cannot indefinitely mask missing getUpdates progress. Preserve the earlier protection for active in-flight outbound delivery from https://github.com/openclaw/openclaw/issues/56065, so do not blindly reintroduce reply abortion during long work. Add/update focused regression tests in `extensions/telegram/src/polling-liveness.test.ts` and/or `extensions/telegram/src/polling-session.test.ts` for stale getUpdates plus successful sendMessage. Also inspect the dirty transport rebuild path in `extensions/telegram/src/polling-session.ts` and `extensions/telegram/src/polling-transport-state.ts`; add bounded diagnostics or error handling around the transition from rebuild to the next polling cycle if a focused source-level gap is found, but do not make broad transport architecture or config changes. Add a user-facing changelog fix entry because this affects Telegram inbound reliability.
