## What This Changes

The issue asks for Telegram polling stall recovery to depend on `getUpdates` liveness and to make post-stall transport rebuild failures visible and recoverable.

## Best Possible Solution

Land a focused Telegram plugin fix that makes stale `getUpdates` alone drive the watchdog, then add bounded diagnostics or timeout handling for the post-stall rebuild path without redesigning channel lifecycle.

## Repair Work Prompt

Fix the Telegram polling stall issue in openclaw/openclaw. Current main still lets unrelated non-`getUpdates` API activity suppress `TelegramPollingLivenessTracker.detectStall()`, contradicting docs that polling restarts after 120s without completed long-poll liveness. Change the watchdog so stale or stuck `getUpdates` controls stall detection even when `sendMessage` or another Bot API call recently succeeded or is in flight; keep generic API elapsed data only as diagnostics if useful. Update regression tests that currently expect no restart for stale polling plus recent non-polling API activity. Also inspect the post-stall rebuild path and add narrowly scoped bounded diagnostics or timeout handling where the issue reports silence after `rebuilding transport for next polling cycle`; coordinate with https://github.com/openclaw/openclaw/pull/76153 and https://github.com/openclaw/openclaw/issues/71412 and do not redesign channel lifecycle, add new config, or duplicate an open transport-close fix. Add a user-facing CHANGELOG.md fix entry unless the final change is test-only. Stop and report instead of broadening into event-loop worker threads, wake detection, provider-wide lifecycle changes, or dependency upgrades.
