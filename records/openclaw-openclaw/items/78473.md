## Summary

Keep open: current main still contains the source-level watchdog suppression and fire-and-forget transport rebuild cleanup, while later comments add fresh macOS re-wedge logs, so this is not safely implemented or superseded.

## What This Changes

The issue reports Telegram polling wedges where inbound `getUpdates` stalls are masked by unrelated Bot API activity and post-stall transport rebuilds fail or immediately re-wedge.

## Best Possible Solution

Land a Telegram-plugin-scoped fix that makes the polling watchdog depend on completed or stuck `getUpdates` liveness and bounds dirty transport teardown/rebuild with awaited cleanup or clear diagnostics, while coordinating related transport lifecycle work.

## Reproduction Assessment

Yes, at source level: make `getUpdates` stale beyond the threshold, refresh or start a non-`getUpdates` API call, and the current watchdog path returns no stall. I did not run a live multi-hour Telegram socket wedge in this read-only review.

## Solution Assessment

Yes for the watchdog fix: removing unrelated API liveness from the stall gate is the narrow maintainable direction. For the rebuild silence, the safer path is focused awaited teardown, timeout handling, and diagnostics around the existing Telegram transport state rather than a broad wake-detection or core channel redesign.
