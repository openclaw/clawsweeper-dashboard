## Summary

Keep open: current main still contains the reported Telegram polling watchdog suppression path, and the later 2026.5.7 macOS comment plus https://github.com/openclaw/openclaw/issues/71412 and https://github.com/openclaw/openclaw/pull/76153 show the restart/rebuild recovery side is not settled. The merged https://github.com/openclaw/openclaw/pull/78646 and changelog entry are useful provenance, but they are not enough to close while the source still has the old gate.

## What This Changes

The issue reports Telegram polling wedges where stale inbound `getUpdates` liveness can be masked by unrelated Bot API activity and restart/rebuild recovery fails to restore polling reliably.

## Best Possible Solution

Land a narrow Telegram plugin fix that makes the polling watchdog depend on completed/stuck `getUpdates` liveness, keeps unrelated Bot API activity diagnostic-only, and adds focused rebuild/start diagnostics while leaving broader channel lifecycle recovery coordinated with https://github.com/openclaw/openclaw/issues/71412 and https://github.com/openclaw/openclaw/pull/76153.

## Reproduction Assessment

Yes, for a source-level reproduction: make `getUpdates` stale past the threshold while a recent or in-flight non-`getUpdates` API call keeps `apiElapsed` fresh, and current `detectStall()` returns `null`. I did not rerun a live multi-hour Telegram socket wedge in this read-only review.

## Solution Assessment

Yes for the watchdog half: removing generic Bot API liveness from the restart gate is the narrowest maintainable fix and matches the docs. The rebuild/re-wedge half should be handled with focused transport lifecycle diagnostics and coordinated with the existing related reports instead of a broad redesign in this issue.
