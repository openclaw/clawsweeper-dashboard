## Summary

Keep open: current main still has the reported native `sessions_spawn` warm-reuse gap. Native subagent spawns use a fresh UUID-backed child session key, `resumeSessionId` remains ACP-only, and neither config nor docs expose a native reuse key, pool, or stable child-session selector.

## What This Changes

The issue requests an opt-in or automatic warm-reuse path for repeated native `sessions_spawn` delegations so subagents can avoid repeated cold-start cache costs.

## Best Possible Solution

Define an explicit native subagent reuse contract that keeps fresh isolated sessions as the default, then wire the chosen API through tool schema, config/types, docs, session key selection, cleanup, and focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is a native `sessions_spawn` call: the tool reaches `spawnSubagentDirect`, which always computes a new UUID-backed `childSessionKey` and has no native reuse parameter to select a warm session.

## Solution Assessment

Unclear. The issue’s opt-in key, automatic pool, and main-session routing options are plausible, but the best fix needs maintainer judgment on isolation, shared history, concurrency locking, archive cleanup, and thread-bound behavior.
