## Summary

Keep open: current main still lacks the requested native warm-reuse path for `sessions_spawn`. Source, tests, config, and docs all show native subagent spawns use isolated UUID-backed session keys, while resume support remains ACP-only; related merged PRs #64468 and #72331 fixed adjacent cache-marker and ACP-field behavior but do not implement native reuse.

## What This Changes

Request a native `sessions_spawn` warm-reuse path so repeated delegations can reuse a subagent session instead of always creating a new UUID-backed isolated session.

## Best Possible Solution

Define an explicit native subagent reuse contract that preserves isolated UUID sessions as the default, then wire the chosen API through tool schema, config/types if needed, docs, session key selection, cleanup, and focused tests.

## Reproduction Assessment

Yes. A native `sessions_spawn` call reaches `spawnSubagentDirect`, whose current params/schema expose no reuse selector and whose session key construction still uses a fresh UUID.

## Solution Assessment

Unclear. The issue's opt-in key, automatic pooling, and main-session routing ideas are plausible, but maintainers need to choose the isolation, history, locking, archive cleanup, and thread semantics before implementation.
