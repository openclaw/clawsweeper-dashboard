## Summary

Keep open: current main still creates native `sessions_spawn` subagent runs as fresh UUID-backed isolated sessions, while the only resume path is ACP-only and no native warm-reuse, pooling, or reuse-key contract exists. Related merged PRs #64468 and #72331 fixed adjacent cache-marker and ACP-field behavior, but they do not implement this request.

## What This Changes

The issue requests a native `sessions_spawn` warm-reuse path so repeated delegation can reuse an existing child session instead of creating a fresh subagent session every time.

## Best Possible Solution

Define an explicit native subagent reuse contract that keeps isolated UUID sessions as the default, then wire the chosen API through tool schema, config/types if needed, docs, key selection, cleanup, locking, and focused tests.

## Reproduction Assessment

Yes for the source-level gap. A native `sessions_spawn` call reaches `spawnSubagentDirect`, which still creates a UUID-backed child session key and receives no native reuse, pool, or resume selector from the tool path.

## Solution Assessment

Unclear. The issue's opt-in key, automatic pool, and main-session routing options are plausible, but maintainers need to choose isolation, history sharing, locking, archive cleanup, and thread semantics before implementation.

## Risks / Open Questions

- A warm-reuse path needs a maintainer decision on isolation versus shared history, especially for routine delegation compared with `context: "fork"` and thread-bound sessions.
- Automatic pooling would need concurrency/liveness locking so two spawns do not reuse the same active child session unsafely.
- Reuse must be reconciled with `archiveAfterMinutes`, cleanup mode, orphan recovery, requester/target policy checks, and subagent registry semantics.
