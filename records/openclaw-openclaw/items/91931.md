## Label Justifications

- `P2`: The deterministic bug silently skips first-run setup in managed deployments but does not crash or disable the core runtime.
- `impact:data-loss`: OpenClaw removes a user-provided BOOTSTRAP.md before its onboarding instructions can be consumed.
- `impact:session-state`: OpenClaw records setupCompletedAt before the intended onboarding flow has completed.

## Summary

This is a valid current bug: both current main and the latest release treat customized SOUL.md, IDENTITY.md, or USER.md as bootstrap-completion evidence while BOOTSTRAP.md exists, then persist completion and remove that file. No merged fix, viable fixing PR, or canonical duplicate was found, so the issue remains necessary.

## What This Changes

Prevent managed workspaces with preseeded profile files from being falsely marked bootstrap-complete before their one-time BOOTSTRAP.md flow runs.

## Best Possible Solution

Keep an existing BOOTSTRAP.md authoritative while setup is incomplete, and limit profile-difference inference to legacy migration cases where no active bootstrap file exists; avoid adding a new public config mode unless that state distinction proves insufficient.

## Reproduction Assessment

Yes at source level: current main has a deterministic path from customized profile files plus an existing BOOTSTRAP.md to premature completion state and file removal, matching the reporter's real Kubernetes timeline. This read-only review did not execute a live current-main deployment.

## Solution Assessment

No, the proposed config-first approach is not the best initial solution because it adds a public setting for a state-machine false positive. The narrowest maintainable fix is to preserve active BOOTSTRAP.md state while retaining profile evidence only for legacy workspaces without an active bootstrap file.
