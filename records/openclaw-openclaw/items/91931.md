## Label Justifications

- `P2`: Affected deployments consistently lose their intended first-run onboarding flow, but the agent runtime remains operational.
- `impact:data-loss`: OpenClaw silently deletes a user-provided `BOOTSTRAP.md` before its instructions can be consumed.
- `impact:session-state`: The persisted workspace state is advanced to completed even though the intended onboarding ritual has not run.

## Summary

Current main intentionally treats customized `SOUL.md`, `IDENTITY.md`, or `USER.md` as stale-bootstrap completion evidence, but that heuristic cannot distinguish completed onboarding from managed pre-seeding and can silently delete a still-needed `BOOTSTRAP.md`. The issue is valid and distinct from the closed persistent-bootstrap request, but preserving both workflows requires an explicit maintainer decision about completion evidence rather than a mechanical repair.

## What This Changes

Support managed or preseeded workspaces without treating customized profile files as proof that the one-time bootstrap ritual has completed.

## Best Possible Solution

Define an explicit canonical completion signal for managed workspaces—such as an opt-in completion-evidence mode or setup-state API—while retaining the current legacy profile heuristic by default, then cover both fresh managed pre-seeding and completed legacy workspace upgrades.

## Reproduction Assessment

Yes at high confidence from source and field evidence: a pending workspace whose profile files differ from templates reaches the completion-and-delete path, and independent Linux/Kubernetes and macOS reports match that transition. This read-only review did not execute a new local filesystem reproduction.

## Solution Assessment

Unclear pending maintainer policy choice. The suggested opt-in mode is safer than changing the default heuristic, but maintainers must decide whether the permanent contract should be configuration, an explicit completion marker/API, or a provisioning workflow.
