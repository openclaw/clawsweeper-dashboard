## Summary

Keep open. The keyboard-scroll part is still a useful current-main WebChat bug fix, but the overflow-clipping claim appears stale from source inspection and the PR needs a scoped DOM lookup before it should land.

## What This Changes

The PR wraps the WebChat slash menu in a new input wrapper and calls `scrollIntoView` after slash-menu ArrowUp/ArrowDown navigation.

## Best Possible Solution

Land a rebased current-main patch that adds scoped active-option auto-scroll with regression coverage, and only change composer layout if overflow clipping is reverified.

## Reproduction Assessment

Yes for the keyboard-scroll sub-issue: current source shows ArrowUp/ArrowDown update the active slash-menu index inside a scrollable menu without a `scrollIntoView` path. The original overflow-clipping sub-claim is not reproduced from current source because `.agent-chat__input` does not set `overflow: hidden`.

## Solution Assessment

No, not as-is. The auto-scroll direction is right, but the active-item lookup should be scoped to the current composer/listbox and stale wrapper/layout churn should be dropped unless current-main clipping is reverified.
