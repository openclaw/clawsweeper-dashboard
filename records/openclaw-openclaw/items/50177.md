## Summary

Keep open: current main already avoids the original overflow clipping, but the remaining keyboard auto-scroll gap is still source-reproducible and the external PR needs clearer real behavior proof plus a scoped rebase before merge.

## What This Changes

The PR wraps the WebChat composer so the slash menu renders outside the input container and adds `scrollIntoView` after slash-menu ArrowUp/ArrowDown navigation.

## Best Possible Solution

Land or replace this with a rebased, narrow WebChat fix that scrolls the active slash-menu option within the current composer/listbox, adds focused regression coverage, and leaves the current clipping layout alone unless it reproduces again.

## Reproduction Assessment

Yes, source-level: current main has a scrollable slash menu, and ArrowUp/ArrowDown update the active index without scrolling the active option into view. The original overflow-clipping symptom is not reproduced from current source because `.agent-chat__input` no longer hides overflow.

## Solution Assessment

No, not as-is: the auto-scroll direction is useful, but the lookup should be scoped to the current composer/listbox and the obsolete wrapper layout changes should be dropped unless clipping is reverified on main.
