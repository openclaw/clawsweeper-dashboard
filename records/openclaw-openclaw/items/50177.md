## Summary

Keep open: the PR still points at a real WebChat keyboard-navigation gap, but the overflow-clipping half is stale on current main and the external branch needs clearer real behavior proof plus a rebase/scope cleanup before merge.

## What This Changes

The PR wraps the WebChat input, renders the slash menu outside that input container, and adds a helper that calls `scrollIntoView` after ArrowUp/ArrowDown slash-menu navigation.

## Best Possible Solution

Land or replace this with a rebased narrow patch that scrolls the active slash-menu option within the current composer/listbox, preserves the current no-clipping layout, and adds focused coverage plus real browser proof.

## Reproduction Assessment

Yes, source-level: current main has a scrollable slash menu and ArrowUp/ArrowDown handlers only update the active index without scrolling the active option into view. The original clipping symptom is not supported by current main because `.agent-chat__input` no longer has the claimed `overflow: hidden` rule.

## Solution Assessment

No, not as-is: auto-scroll is the right narrow behavior, but the lookup should be scoped to the current composer/listbox and the stale wrapper layout changes should be dropped unless clipping is reverified.
