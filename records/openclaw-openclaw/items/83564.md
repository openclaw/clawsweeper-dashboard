## Summary

Keep open: current main still renders the configured `adaptive` value as the only `/status` thinking value while runtime/provider policy can map supported effort separately. This is a narrow diagnostic enhancement, but it needs a maintainer choice about which resolved layer should be exposed.

## What This Changes

Expose the resolved or effective thinking level in `/status` when `thinkingDefault` is `adaptive`, for example `adaptive -> medium`.

## Best Possible Solution

Define one effective-thinking metadata contract for `/status`, keep the configured value visible, and add the resolved canonical or provider effort only where that value is well-defined and covered by focused status/thinking tests.

## Reproduction Assessment

Yes, source-level. Current main can pass configured `adaptive` directly into the single `/status` `Think:` field while the thinking resolver has separate provider-profile fallback behavior; no live DeepSeek turn was run in this read-only review.

## Solution Assessment

Unclear. Showing `adaptive -> medium` is plausible, but maintainers should first choose whether “resolved” means OpenClaw canonical fallback, provider-specific payload effort, or last-turn request shaping.
