## Summary

Keep open: current `main` and latest release still render a single `/status` thinking value, and a configured `adaptive` value can still short-circuit the model-aware fallback resolver before any `adaptive -> resolved` display is built.

## What This Changes

The issue requests that `/status` show both the configured `adaptive` thinking default and the actual resolved thinking level, such as `adaptive -> medium`.

## Best Possible Solution

Define one maintained effective-thinking status contract, then show the configured value plus the chosen resolved value only where that layer is unambiguous and covered by focused status tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `thinkingDefault: "adaptive"`, then `/status` can render the configured value before the model-aware fallback resolver is consulted.

## Solution Assessment

Unclear. Showing `adaptive -> medium` is plausible for canonical fallback, but provider payload semantics can differ, so maintainers should choose the status layer before implementation.
