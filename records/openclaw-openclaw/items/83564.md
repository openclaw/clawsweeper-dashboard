## Summary

Keep open: current main still shows the configured thinking value in `/status`, while the runtime has separate fallback/provider mapping behavior that is not exposed there. This is a narrow diagnostic enhancement, but the exact meaning of “resolved” needs maintainer direction before implementation.

## What This Changes

Request to make `/status` show both configured `adaptive` thinking and the effective thinking level used for the current turn.

## Best Possible Solution

Define a single effective-thinking metadata contract for `/status`, keep the configured value visible, and add the resolved canonical or provider effort only where that value is well-defined and tested.

## Reproduction Assessment

Yes. Source inspection shows `/status` can render the configured `adaptive` value while the thinking fallback code can remap unsupported adaptive to `medium`; I did not run a live DeepSeek turn in this read-only review.

## Solution Assessment

Unclear. Showing `adaptive -> medium` is plausible, but DeepSeek V4 transport mapping can turn lower non-off levels into provider `high`, so maintainers should first choose which resolved layer `/status` should expose.
