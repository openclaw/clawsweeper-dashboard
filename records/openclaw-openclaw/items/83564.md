## Summary

Keep open: current `main` and the latest release still render one `/status` thinking value, so `thinkingDefault: "adaptive"` can still be shown as the configured value without a configured-versus-effective display. A prior hardcoded `adaptive -> medium` status PR was closed after feedback that the effective value varies by provider, so this needs maintainer product direction rather than cleanup closure.

## What This Changes

Request to have `/status` show both configured `thinkingDefault: "adaptive"` and the effective resolved thinking level for the current model or turn.

## Best Possible Solution

Define one maintained effective-thinking status contract, then show configured and effective values only where that layer is unambiguous and covered by focused status tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `thinkingDefault: "adaptive"`, then `/status` can render the configured value before the model-aware fallback resolver is consulted.

## Solution Assessment

Unclear. Showing `adaptive -> medium` is plausible for one canonical fallback case, but provider payload semantics can differ, so maintainers should choose the status layer before implementation.
