## Summary

Keep open: current main and the latest release still render a single `/status` thinking value, and the requested `adaptive -> resolved` display remains a product/status-contract decision rather than an implemented behavior.

## What This Changes

The issue asks `/status` to expose both the configured `adaptive` thinking value and the actual resolved thinking level, or to attach that resolved value as assistant metadata.

## Best Possible Solution

Define one effective-thinking status contract, keep the configured value visible, and add a resolved canonical or provider-effort value only where that layer is well-defined and covered by focused status tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `thinkingDefault: "adaptive"`, then `/status` can render the configured value before the fallback resolver is consulted. I did not run a live DeepSeek turn in this read-only review.

## Solution Assessment

Unclear. Showing `adaptive -> medium` is plausible for canonical OpenClaw fallback, but Direct DeepSeek V4 can send provider `reasoning_effort: "high"`, so maintainers should choose the status layer before implementation.
