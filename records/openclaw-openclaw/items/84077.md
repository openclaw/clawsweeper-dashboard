## Label Justifications

- `P1`: An unbounded plugin-owned compaction can leave an interactive agent turn unresponsive until the broad run timeout or longer.
- `impact:crash-loop`: The report is about agent availability and hangs during compaction, even though the process may not crash.
- `impact:session-state`: The hang occurs inside session compaction/retry handling and can leave the active session lane stuck behind in-flight work.

## Summary

Keep open: current main still awaits plugin-owned `ContextEngine.compact()` directly on the queued, timeout-recovery, and overflow-recovery lanes, while native compaction is the only path wrapped by the existing safety timeout.

## What This Changes

Plugin-owned context-engine compaction should be bounded by the same safety timeout and abort path as native embedded compaction.

## Best Possible Solution

Reuse the existing compaction timeout helper for every host-invoked `ContextEngine.compact` path and extend the context-engine compact params with an optional abort signal so plugin work can stop cooperatively.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an ownsCompaction engine whose `compact()` never resolves will be awaited in queued, timeout-recovery, or overflow-recovery compaction without the existing safety wrapper.

## Solution Assessment

Yes. The narrow maintainable fix is to wrap plugin compact calls with the existing timeout helper, pass the active abort signal through an additive compact parameter, and cover the ownsCompaction paths with regression tests.
