## Label Justifications

- `P1`: A plugin-owned compaction promise that never settles can leave an interactive agent turn unresponsive in real channel workflows.
- `impact:crash-loop`: The report is about agent availability and hangs during compaction, even though the process may not literally crash.
- `impact:session-state`: The failure occurs inside session compaction and retry handling, leaving the active session lane blocked behind in-flight work.

## Summary

Keep open: current main still directly awaits plugin-owned `compact()` on the reported pi-embedded and Codex harness lanes, and https://github.com/openclaw/openclaw/pull/84083 is the open linked fix that should land before this issue closes.

## What This Changes

Plugin-owned context engines that advertise `ownsCompaction` can block agent turns because current host and Codex harness compaction paths await `ContextEngine.compact()` without a safety timeout or abort signal.

## Best Possible Solution

Land a reviewed fix that reuses the existing compaction safety timeout for every plugin-owned `compact()` path and adds cooperative abort support without requiring existing context-engine plugins to change.

## Reproduction Assessment

Yes. Source inspection is high-confidence: an `ownsCompaction` engine whose `compact()` never resolves will block queued/manual, timeout-recovery, overflow-recovery, and Codex-owned compaction because current main awaits that promise directly; I did not run a live wedged plugin in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing compaction safety timeout for plugin-owned `compact()` calls and thread optional abort support; this issue should remain open until the linked PR lands.
