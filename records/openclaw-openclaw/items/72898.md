## Summary

The PR is a plausible workaround for the original iMessage failure mode, but the later overlapping review established that raw startup banners are an `imsg` JSON-RPC contract bug, and that fix has shipped upstream rather than belonging in OpenClaw-side stdout classification.

## What This Changes

The PR adds iMessage `chat.db` permission-denial text detection in the OpenClaw iMessage RPC client/probe and a regression test for fatal probe mapping.

## Best Possible Solution

Keep the startup-framing fix in `imsg`, let OpenClaw consume JSON-RPC error frames and surface concise diagnostics, and track any separate fatal-probe policy as a new OpenClaw issue if maintainers still want it.

## Reproduction Assessment

Yes for the historical failure path: pre-v0.8.2 `imsg` could emit a Full Disk Access banner on stdout while OpenClaw parsed every non-empty stdout line as JSON. For the current dependency contract, `imsg` v0.8.2 now returns JSON-RPC error frames, so the OpenClaw-side workaround is no longer the target fix.

## Solution Assessment

No. The PR works around contaminated stdout inside OpenClaw, but the maintainable solution is the shipped upstream `imsg` JSON-RPC framing fix plus OpenClaw diagnostic surfacing.
