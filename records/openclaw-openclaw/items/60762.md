## Summary

Keep open: the reconnect-herd problem is still visible in current main by source inspection, but this branch targets a stale Carbon/baseDelay wrapper surface and lacks after-fix real Discord proof, so it should not merge as-is.

## What This Changes

The PR threads accountId into Discord monitor gateway creation and sets a deterministic reconnect.baseDelay in the gateway wrapper, with unit coverage for wrapper options.

## Best Possible Solution

Thread accountId into the current internal Discord GatewayPluginOptions and apply deterministic additive jitter inside scheduleReconnect while preserving the existing reconnect floor, cap, invalid-session delay, and maxAttempts behavior.

## Reproduction Assessment

Yes by source inspection. Current main's internal scheduler has no account-specific input and computes reconnect delay only from minDelayMs and reconnectAttempts, so equal-attempt multi-account reconnects remain synchronized.

## Solution Assessment

No. The goal is sound, but this PR writes baseDelay in the monitor wrapper while current main computes reconnect timing in extensions/discord/src/internal/gateway.ts and does not read baseDelay.
