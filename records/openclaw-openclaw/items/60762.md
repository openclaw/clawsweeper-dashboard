## What This Changes

The PR passes Discord account IDs into gateway plugin creation, sets deterministic per-account reconnect baseDelay jitter, and adds monitor option tests.

## Best Possible Solution

Thread account identity into the active internal Discord gateway options and apply small deterministic additive jitter inside scheduleReconnect while preserving the existing backoff, cap, maxAttempts, and invalid-session cooldown behavior.

## Reproduction Assessment

Yes, by source inspection: current main's scheduleReconnect derives delay only from minDelayMs and reconnect attempt count, so multiple accounts at the same attempt schedule the same delay. I did not run a live multi-account Discord outage reproduction.

## Solution Assessment

No. The PR targets stale Carbon baseDelay plumbing, while current main's active path is the internal GatewayPlugin scheduler; the maintainable fix belongs where scheduleReconnect schedules the reconnect timer.
