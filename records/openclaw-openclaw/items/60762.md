## Summary

Keep this PR open, but it is not ready to merge: the thundering-herd problem is plausible from current source, while the branch still targets the old Carbon gateway surface instead of current main's internal Discord reconnect scheduler and lacks after-fix real behavior proof.

## What This Changes

The PR threads Discord `accountId` into monitor gateway creation and sets a deterministic per-account Carbon `reconnect.baseDelay`, with provider proxy unit coverage.

## Best Possible Solution

Thread account identity into the internal Discord gateway options, apply deterministic small additive jitter inside `scheduleReconnect` while preserving the reconnect floor, invalid-session floor, cap, and `maxAttempts`, then require redacted real multi-account reconnect proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main's Discord reconnect scheduler has no account-specific input and computes the timer from only `minDelayMs` and `reconnectAttempts`, so equal-attempt multi-account reconnects remain synchronized.

## Solution Assessment

No. The branch sets `baseDelay` on the old Carbon wrapper; the maintainable fix is to move account-based additive jitter into current main's internal scheduler and cover that timer behavior directly.
