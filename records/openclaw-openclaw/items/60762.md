## Summary

Keep open because the reported reconnect-herd problem is still source-reproducible, but this PR targets the old Carbon/baseDelay surface instead of the current internal Discord gateway scheduler and still lacks after-fix real behavior proof.

## What This Changes

This PR forwards Discord accountId into monitor gateway creation and sets a deterministic per-account reconnect baseDelay with tests for the gateway wrapper options.

## Best Possible Solution

Thread accountId into the internal Discord gateway options and apply deterministic additive jitter inside scheduleReconnect, with focused tests that preserve the existing backoff floor, cap, and maxAttempts behavior.

## Reproduction Assessment

Yes. Source inspection shows two Discord accounts at the same reconnect attempt count still reach scheduleReconnect with no account-specific input, so current main computes identical reconnect delays.

## Solution Assessment

No. The goal is right, but this PR writes Carbon baseDelay in the monitor wrapper; the maintainable fix belongs in the current internal gateway scheduler.
