## Summary

Keep open: the Discord reconnect thundering-herd problem still appears source-reproducible, but this branch is conflicting and targets the old Carbon `baseDelay` surface instead of current main's internal gateway scheduler. It also lacks after-fix real Discord multi-account reconnect proof.

## What This Changes

The PR passes Discord `accountId` into monitor gateway creation, computes deterministic reconnect `baseDelay` jitter in the gateway plugin wrapper, and adds gateway option unit tests.

## Best Possible Solution

Thread account identity into the current internal Discord gateway options and apply small deterministic additive jitter inside `scheduleReconnect` while preserving `minDelayMs`, the 30-second cap, and `maxAttempts`.

## Reproduction Assessment

Yes, by source inspection: current main's `scheduleReconnect` derives delay only from `minDelayMs` and reconnect attempt count, so multiple accounts at the same attempt schedule the same delay. I did not run a live multi-account Discord outage reproduction.

## Solution Assessment

No. The PR writes Carbon `baseDelay` in the monitor wrapper, but current main uses the internal gateway scheduler, so the maintainable fix belongs in `extensions/discord/src/internal/gateway.ts` after passing `accountId` through startup.
