## Summary

Keep this PR open, but it still needs changes before merge: the Discord multi-account reconnect issue is source-reproducible on current main, while the branch targets the old Carbon `baseDelay` surface that current main no longer uses for reconnect scheduling. The external PR also lacks sufficient after-fix real behavior proof.

## What This Changes

The PR adds deterministic per-account Discord gateway reconnect staggering by passing `accountId` into gateway plugin creation and setting a reconnect `baseDelay` with unit coverage.

## Best Possible Solution

Thread `accountId` into the current monitor gateway creation, add a typed deterministic per-account additive jitter inside the internal Discord gateway scheduler while preserving existing backoff floors and caps, cover the scheduler behavior with focused tests, and let maintainers add the user-facing changelog entry during landing.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: two Discord accounts with the same reconnect attempt count reach `scheduleReconnect` with no account-specific input, so current main schedules the same delay for both. I did not run a live gateway or test command because this review is read-only.

## Solution Assessment

No. The intended fix is useful, but setting Carbon `reconnect.baseDelay` is no longer the best current solution because current main schedules reconnects in `extensions/discord/src/internal/gateway.ts`; the safer fix is typed additive jitter there.
