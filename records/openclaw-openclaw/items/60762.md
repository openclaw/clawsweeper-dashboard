## Summary

## What This Changes

The PR threads Discord `accountId` into monitor gateway creation, sets deterministic per-account Carbon reconnect `baseDelay` jitter, and adds provider proxy unit coverage.

## Best Possible Solution

Thread account identity into the current internal Discord gateway options and apply small deterministic additive jitter inside `scheduleReconnect` while preserving `minDelayMs`, invalid-session floors, the 30s cap, and `maxAttempts`.

## Reproduction Assessment

Yes, by source inspection: current main computes Discord reconnect delay from only `minDelayMs` and reconnect attempt count, so equal-attempt multi-account reconnects remain synchronized. I did not run a live multi-account Discord outage repro.

## Solution Assessment

No. The branch writes an old Carbon `baseDelay` option; the maintainable fix is to move account-based additive jitter into current main's internal scheduler and prove it with focused timer tests plus redacted real reconnect evidence.
