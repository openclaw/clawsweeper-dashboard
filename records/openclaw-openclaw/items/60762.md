## Summary

Keep open: the PR addresses a real Discord multi-account reconnect problem, but the branch is stale against current main's internal gateway scheduler and does not currently solve the synchronized reconnect timing there.

## What This Changes

The PR adds account-aware Discord reconnect baseDelay calculation, forwards accountId into gateway plugin creation, and adds unit coverage for deterministic per-account reconnect values.

## Best Possible Solution

Implement deterministic per-account additive jitter inside the current internal Discord gateway scheduler, thread accountId from monitor startup into the gateway plugin, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The source-level reproduction is clear: two Discord accounts with the same reconnect attempt count both reach scheduleReconnect with no account-specific input, so current main schedules the same delay for both.

## Solution Assessment

No. The intended behavior is useful, but monitor-level Carbon baseDelay is no longer the best current implementation; the safer fix is a typed additive jitter in the internal gateway scheduler while preserving the existing backoff baseline and cap.
