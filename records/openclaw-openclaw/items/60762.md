## Summary

Keep open: the user problem is still plausible, but this PR is stale against current main's Discord gateway implementation and the proposed `baseDelay` wiring no longer reaches the reconnect scheduler.

## What This Changes

The PR adds deterministic per-account Discord gateway reconnect jitter by passing `accountId` into monitor gateway creation and setting `reconnect.baseDelay` in the gateway options.

## Best Possible Solution

Thread `accountId` into the internal Discord gateway options and apply deterministic additive jitter inside `scheduleReconnect` while preserving the reconnect floor, invalid-session floor, 30-second cap, and `maxAttempts` behavior.

## Reproduction Assessment

Yes, by source inspection. Current main schedules reconnect delay from `minDelayMs` and `reconnectAttempts` only, so equal-attempt multi-account reconnects remain synchronized.

## Solution Assessment

No. Setting `reconnect.baseDelay` in the monitor wrapper is not the best fix because current main's active reconnect scheduler is in `extensions/discord/src/internal/gateway.ts` and does not read that option.
