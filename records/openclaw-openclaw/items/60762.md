## Summary

Keep open: the multi-account Discord reconnect herd still appears valid on current main, but this PR is stale against the current internal gateway scheduler because monitor-level `reconnect.baseDelay` is not consumed by `scheduleReconnect`.

## What This Changes

This PR attempts to pass an account-aware Discord reconnect `baseDelay` from monitor startup into the gateway plugin and adds tests for deterministic per-account delay values.

## Best Possible Solution

Move deterministic per-account additive jitter into the current Discord internal gateway reconnect scheduler, preserve the existing backoff baseline and cap, thread `accountId` from monitor startup, and ship focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: two Discord accounts with the same reconnect attempt count reach `scheduleReconnect` with no account-specific input, so both schedule the same delay on current main.

## Solution Assessment

No. The intent is useful, but monitor-layer `reconnect.baseDelay` is no longer the best current-main implementation; the narrow maintainable fix is a typed jitter option applied inside `extensions/discord/src/internal/gateway.ts`.
