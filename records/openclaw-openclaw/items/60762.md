## Summary

Keep open: the Discord reconnect-herd problem is source-reproducible on current main, but this branch is not correct because it targets the stale Carbon/monitor wrapper path while reconnect scheduling now lives in the internal Discord gateway scheduler; it also still lacks after-fix real behavior proof.

## What This Changes

The PR adds optional Discord account identity to monitor gateway creation, sets a deterministic per-account reconnect baseDelay in the monitor gateway wrapper, forwards accountId from monitor startup, and adds provider proxy tests.

## Best Possible Solution

Thread accountId into the current Discord internal gateway scheduler, add deterministic additive reconnect jitter while preserving existing delay floors and caps, and cover it with focused internal gateway and monitor startup tests.

## Reproduction Assessment

Yes. Source inspection shows two configured Discord accounts at the same reconnect attempt count reach scheduleReconnect with no account-specific input, so current main schedules identical reconnect delays.

## Solution Assessment

No. The goal is valid, but this PR writes reconnect baseDelay into a stale monitor/Carbon wrapper; the maintainable fix belongs in extensions/discord/src/internal/gateway.ts and its current callers/tests.
