## Summary

Keep open: the Discord multi-account reconnect herd is still source-reproducible on current main, but this PR branch targets the old Carbon gateway surface and lacks after-fix real behavior proof, so it should be rewritten/rebased rather than merged as-is.

## What This Changes

The PR threads Discord `accountId` into monitor gateway creation and sets a deterministic per-account Carbon reconnect `baseDelay` with unit coverage.

## Best Possible Solution

Thread `accountId` into the current Discord gateway scheduler, add deterministic per-account additive reconnect jitter while preserving existing floors and caps, and cover it with focused internal gateway and monitor startup tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two configured Discord accounts that reconnect at the same attempt count reach `scheduleReconnect` with no account-specific input, so current main schedules the same delay for both.

## Solution Assessment

No. The goal is right, but this patch writes Carbon `baseDelay` in a stale monitor wrapper while current main computes reconnect delays in `extensions/discord/src/internal/gateway.ts`; the safer fix belongs in that scheduler.
