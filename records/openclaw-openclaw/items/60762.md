## Summary

Keep open: the Discord reconnect-stagger goal remains valid, but this branch still targets the obsolete `reconnect.baseDelay` surface while current main computes reconnect timing inside the internal gateway scheduler and does not read that option. The PR also still lacks after-fix real Discord reconnect proof.

## What This Changes

The PR passes Discord `accountId` into monitor gateway creation and adds deterministic per-account `reconnect.baseDelay` coverage to stagger reconnect timing.

## Best Possible Solution

Thread `accountId` into the internal Discord gateway options and apply deterministic additive jitter inside `scheduleReconnect` while preserving the reconnect floor, invalid-session floor, 30-second cap, and `maxAttempts` behavior.

## Reproduction Assessment

Yes, by source inspection. Current main's reconnect scheduler has no account-specific input and computes delay only from `minDelayMs` and `reconnectAttempts`, so equal-attempt multi-account reconnects remain synchronized.

## Solution Assessment

No. The branch should not set `baseDelay` in the monitor wrapper; the maintainable fix is to move account-based additive jitter into the internal gateway scheduler that actually schedules reconnects.
