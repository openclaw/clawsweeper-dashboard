## Summary

Keep open: the Discord reconnect herd problem still appears valid on current main, but this branch is not mergeable as-is because it targets the stale Carbon/baseDelay wrapper while current reconnect scheduling lives in the internal gateway, and it still lacks real behavior proof.

## What This Changes

The branch forwards Discord accountId into monitor gateway creation, sets a deterministic per-account Carbon reconnect baseDelay in the gateway wrapper, and adds option assertions in provider proxy tests.

## Best Possible Solution

Implement deterministic additive jitter in extensions/discord/src/internal/gateway.ts, thread accountId from monitor startup into the internal gateway options, preserve the existing reconnect floor/cap behavior, and cover it with focused internal gateway and monitor startup tests.

## Reproduction Assessment

Yes. Source inspection shows two Discord accounts at the same reconnect attempt count still reach scheduleReconnect with no account-specific input, so current main computes identical reconnect delays.

## Solution Assessment

No. The goal is right, but this PR writes a baseDelay option into a stale wrapper; the narrow maintainable fix is to carry account identity into the internal scheduler and add jitter to the computed reconnect delay.
