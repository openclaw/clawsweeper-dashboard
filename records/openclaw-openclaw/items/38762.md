## Summary

Keep open. Current main still drops top-level `ref` when `action="act"` also supplies an object `request`, so the mixed payload reaches `/act` without `ref`; the prior compatibility PR is closed unmerged and targeted the old pre-plugin path.

## What This Changes

The issue asks for `browser action="act"` to handle top-level `ref` consistently with nested `request.ref`, or to return a request-shape error instead of misleading browser-service guidance.

## Best Possible Solution

Add a narrow browser tool entry-point normalizer that fills missing nested act fields from supported legacy top-level params while preserving the `/act` service contract and nested-request precedence.

## Reproduction Assessment

Yes, by source inspection. Call `browser` with `action: "act"`, `request: { "kind": "click" }`, and top-level `ref`; the parser returns the nested request unchanged, and `/act` normalization then sees no `ref` or `selector`.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/browser/src/browser-tool.ts` before transport, because the browser service route never receives top-level tool params.
