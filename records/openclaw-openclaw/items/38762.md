## Summary

Keep open. Current main still returns an object `request` before top-level `ref`, `targetId`, or `timeoutMs` can be merged, so the report's mixed `act` payload drops the top-level `ref`; the related compatibility PR #38790 is closed unmerged. The old reachability-style error path appears improved, but the parameter compatibility bug remains.

## What This Changes

Allow or clearly diagnose `browser action="act"` calls that put `ref` at the top level while `request` contains only the act kind.

## Best Possible Solution

Add a narrow browser tool entry-point normalizer that fills only missing nested act fields from supported top-level compatibility params while preserving the `/act` service contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `browser` with `action: "act"`, `request: { kind: "click" }`, and top-level `ref`; the parser forwards only the request object, and `/act` normalization then sees no `ref` or `selector`.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/browser/src/browser-tool.ts` before transport, filling missing nested fields from the supported top-level compatibility params while leaving nested values authoritative.
