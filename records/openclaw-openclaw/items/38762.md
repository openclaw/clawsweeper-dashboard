## Summary

Current `main` and the latest release still drop top-level browser `act` fields when a nested `request` object is present, so the reported bug remains necessary. Keep this issue open because https://github.com/openclaw/openclaw/pull/81076 is an open linked implementation candidate using closing syntax for the same behavior.

## What This Changes

The issue reports that browser `act` handles `ref` inside `request` but ignores a top-level `ref`, causing misleading browser-service error text for a malformed internal request shape.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81076, or an equivalent narrow parser-layer fix, so supported top-level act fields are backfilled only when missing while nested request fields remain authoritative.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current `main`: `action: "act"` with top-level `ref` and `request: { kind: "click" }` returns the nested request before copying `ref`, then the act service receives a body without `ref`.

## Solution Assessment

Yes for the current fix direction. Backfilling only allowlisted top-level act fields in the browser tool parser is narrower than changing the `/act` service contract and preserves nested-field precedence.
