## Summary

Keep open. Current main and the latest shipped release still return the nested `request` object unchanged for `action: "act"`, so top-level `ref` is dropped in the mixed payload the reporter described; the related compatibility PRs are closed, and none currently owns the remaining plugin-path fix.

## What This Changes

The issue asks for consistent `browser action="act"` handling when `ref` is supplied at the top level while `request` contains the act payload.

## Best Possible Solution

Backfill missing nested act request fields from supported top-level act params in the browser tool entrypoint, preserve nested `request` precedence, and add a focused mixed-payload regression test.

## Reproduction Assessment

Yes, by source inspection. With `action: "act"`, top-level `ref`, and `request: { kind: "click" }`, `readActRequestParam` returns the nested request unchanged, so `/act` sees no `ref` or `selector`.

## Solution Assessment

Yes. A narrow normalizer in `extensions/browser/src/browser-tool.ts` is the best fix because it keeps the `/act` service contract intact while honoring the tool schema’s accepted compatibility shape.
