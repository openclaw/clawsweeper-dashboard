## Label Justifications

- `P1`: The bug breaks a real browser-agent workflow for model-generated tool calls that place `ref` at the advertised top level.
- `impact:auth-provider`: The failure is tied to model/provider payload behavior: OpenAI-style tool calls fail while nested-ref payloads continue to work.

## Summary

Keep open: current main still drops top-level browser `act` fields whenever a nested `request` object is present, and the open closing PR is the focused fix path.

## What This Changes

The issue requests that browser `act` accept `ref` consistently when supplied beside a nested `request`, or return a clearer parameter-shape error.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81076 with the same narrow parser backfill and regression tests, preserving nested request precedence and the `/act` service contract.

## Reproduction Assessment

Yes, at source level. Current main returns the nested `request` before copying top-level `ref`, then the act normalizer rejects click/type bodies without `ref` or `selector`.

## Solution Assessment

Yes. Backfilling existing allowlisted top-level act fields into `request` only when missing is the narrowest maintainable fix and keeps nested fields authoritative.
