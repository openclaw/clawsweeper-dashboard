## Summary

Keep open: current main and the latest release still return a nested browser `act` request unchanged before copying advertised top-level fields, so the reported top-level `ref` loss remains source-reproducible. The focused linked fix PR is still open and should remain the canonical implementation path until it lands or is rejected.

## What This Changes

The issue reports that browser `act` drops top-level `ref` when a nested `request` object is present, causing ref-based browser actions from OpenAI-style tool calls to fail with misleading errors.

## Best Possible Solution

Review and land, revise, or explicitly reject https://github.com/openclaw/openclaw/pull/81076 so the advertised top-level and nested browser act shapes behave consistently without changing the `/act` service contract.

## Reproduction Assessment

Yes by source inspection: a current-main call with `action: "act"`, top-level `ref`, and `request: { kind: "click" }` reaches `browserAct` with only the nested request object. I did not run a live browser scenario because this review is read-only.

## Solution Assessment

Yes: the best current path is the focused linked PR that backfills only allowed top-level act fields into a nested request when missing, preserving nested-field precedence and keeping the `/act` service contract unchanged.
