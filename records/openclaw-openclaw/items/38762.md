## Label Justifications

- `P1`: The bug breaks a real browser-agent workflow for model-generated tool calls that place `ref` at the advertised top level.
- `impact:auth-provider`: The failure is tied to model/provider payload shape: OpenAI-style calls fail while nested-ref payloads continue to work.

## Summary

Keep open: current main and the latest release still drop top-level browser `act` fields when a nested `request` object is present, and https://github.com/openclaw/openclaw/pull/81076 remains the focused closing PR for this bug.

## What This Changes

The issue reports that browser `act` accepts `ref` inside `request` but drops the same `ref` when it is supplied at the top level beside a nested `request`, producing a misleading browser-service error.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/81076 with an equivalent narrow parser backfill and regression tests, while keeping nested request fields authoritative and the `/act` service contract unchanged.

## Reproduction Assessment

Yes, at source level. Current main returns the nested `request` before copying top-level `ref`, then the act normalizer rejects click/type bodies without `ref` or `selector`.

## Solution Assessment

Yes. Backfilling existing allowlisted top-level act fields into `request` only when missing is the narrowest maintainable fix and preserves nested-field precedence.
