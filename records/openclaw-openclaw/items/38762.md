## Label Justifications

- `P1`: The bug breaks a real browser-agent workflow for OpenAI-family model tool calls that place `ref` at the advertised top level.
- `impact:auth-provider`: The failure is tied to model/provider payload behavior: OpenAI-style tool calls fail while nested-ref payloads from other model families can continue working.

## Summary

Keep open: current main and the latest release still drop top-level browser `act` fields when a nested `request` object is present, and the open closing PR is the focused fix path.

## What This Changes

Browser `act` calls with `ref` at the top level beside a nested `request` should either work consistently or return a request-shape error instead of dropping `ref` and failing downstream.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81076 with the same narrow parser backfill and regression tests, preserving nested request precedence and the `/act` service contract.

## Reproduction Assessment

Yes, at source level. Current main returns the nested `request` before copying top-level `ref`, then the act normalizer rejects click/type bodies without `ref` or `selector`.

## Solution Assessment

Yes. Backfilling existing allowlisted top-level act fields into `request` only when missing is the narrowest maintainable fix and keeps nested fields authoritative.
