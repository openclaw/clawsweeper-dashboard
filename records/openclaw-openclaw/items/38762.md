## Summary

Keep open. Current main still preserves the unsupported mixed call shape: an object `request` is returned before top-level `ref`, `targetId`, or `timeoutMs` can be merged, so the browser act call can still drop the top-level ref before `/act` validation. The service-error wrapping has improved, but the argument-position inconsistency remains, and the related compatibility PR #38790 is closed unmerged.

## What This Changes

The issue asks for `browser action="act"` to accept `ref` at either the top level or inside `request`, or to return an accurate argument-shape error.

## Best Possible Solution

Add a narrow compatibility normalizer at the browser tool entry point that fills only missing `request.ref`, `request.targetId`, and finite numeric `request.timeoutMs` from top-level act params, while preserving nested request precedence and leaving the browser control service contract unchanged.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from current source: call `browser action="act"` with `request: { kind: "click" }` and top-level `ref`; the parser forwards only the request object, then `/act` rejects the click because the forwarded body has no `ref` or `selector`.

## Solution Assessment

Yes. The best fix is a scoped compatibility normalizer in `extensions/browser/src/browser-tool.ts`, not a route-level contract change, because the schema already exposes the mixed tool-call shape and the dropped fields occur before transport.
