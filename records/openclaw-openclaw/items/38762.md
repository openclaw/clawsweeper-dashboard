## Label Justifications

- `impact:other`: The concrete impact is browser tool parameter normalization and misleading browser-control diagnostics, which does not fit the more specific owned impact labels.

## Summary

Keep open: current `main` and the latest release still drop top-level browser `act` fields when a nested `request` object is present, and the open linked PR remains the active implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests consistent browser `act` handling, or clearer diagnostics, when `ref` is passed at the top level beside a nested `request` object.

## Best Possible Solution

Land `https://github.com/openclaw/openclaw/pull/81076` or an equivalent parser-layer fix that backfills only allowlisted top-level act fields into nested requests while preserving nested-field precedence.

## Reproduction Assessment

Yes, source-level. On current `main`, `readActRequestParam` returns the nested `request` unchanged, the `act` branch forwards that stripped object, and `/act` then rejects click/type requests without `ref` or `selector`.

## Solution Assessment

Yes for the fix shape. Backfilling allowlisted top-level act fields at the browser tool parser is narrower than changing the `/act` service contract, and it preserves nested request values as authoritative.
