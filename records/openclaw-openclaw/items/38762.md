## Summary

Keep open: current `main` and v2026.6.6 still drop top-level browser `act` fields when a nested `request` is present, and the active fix path is the open linked pull request https://github.com/openclaw/openclaw/pull/81076.

## What This Changes

The issue reports that browser `action=act` accepts `ref` inside `request` but drops `ref` when supplied at the top level beside a nested request, producing a misleading browser-service error.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81076, or an equivalent narrow browser-plugin parser fix that backfills only allowlisted top-level act fields while keeping nested fields authoritative.

## Reproduction Assessment

Yes, at source level. A browser `act` call with top-level `ref` and `request: { kind: "click" }` reaches `readActRequestParam`, which returns the nested request before copying `ref`, and the `/act` normalizer then rejects click/type bodies without `ref` or `selector`.

## Solution Assessment

Yes for the fix direction. The narrow maintainable solution is to backfill only allowlisted top-level act fields into a nested request when missing, while preserving nested-field precedence.
