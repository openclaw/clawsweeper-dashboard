## Label Justifications

- `P1`: The bug breaks a real browser-agent workflow for model-generated tool calls, especially OpenAI-family payloads that place `ref` at the advertised top level.
- `impact:auth-provider`: The failure is tied to model/provider behavior: OpenAI-style tool payloads fail while nested-ref payloads from other model families continue to work.

## Summary

Keep open: current main and the latest release still drop top-level browser `act` fields when a nested `request` object is present, and the open closing PR is the focused fix path.

## What This Changes

The issue asks for browser `act` to accept `ref` consistently when it is provided at the top level beside `request`, or to return a request-shape error instead of a browser-service reachability error.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81076 with the same narrow parser backfill and regression tests, preserving nested request precedence and the `/act` service contract.

## Reproduction Assessment

Yes, at source level. On current main, `action: "act"` with top-level `ref` and `request: { kind: "click" }` returns only the nested request, then `/act` rejects it because `ref` or `selector` is absent from that body.

## Solution Assessment

Yes. Backfilling existing allowlisted top-level act fields into the nested request only when missing is the narrowest maintainable fix and keeps nested request fields authoritative.
