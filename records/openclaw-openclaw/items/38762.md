## Summary

Keep open. Current main still drops top-level act fields whenever `request` is an object, so the mixed payload from the report reaches `/act` without `ref`; the related compatibility PR #38790 is closed unmerged.

## What This Changes

The issue requests consistent handling for `browser action="act"` when `ref` is supplied at the top level alongside a nested `request`.

## Best Possible Solution

Add a narrow browser tool entry-point normalizer that fills missing nested act fields from supported legacy top-level params while preserving the existing `/act` service contract and nested-request precedence.

## Reproduction Assessment

Yes, by source inspection. Call `browser` with `action: "act"`, `request: { "kind": "click" }`, and top-level `ref`; the parser returns the nested request unchanged, and `/act` validation then sees no `ref` or `selector`.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/browser/src/browser-tool.ts` before transport, because the browser service route never receives top-level tool params.
