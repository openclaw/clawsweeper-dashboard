## Label Justifications

- `P1`: The mismatch can block browser click, type, and form workflows for real model/provider-generated tool calls using the advertised schema shape.
- `impact:auth-provider`: Different model/provider tool-call shapes determine whether the required browser action reference reaches execution.

## Summary

Keep open: current main and v2026.6.5 still drop the advertised top-level browser `act` `ref` whenever a nested `request` object is present. The misleading reachability wrapper has been addressed, but the central parameter-shape bug remains and is already owned by the open closing PR https://github.com/openclaw/openclaw/pull/81076.

## What This Changes

The issue asks for browser `act` to handle `ref` consistently when it is passed either inside `request` or at the top level, or to report a clearer parameter error.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/81076 so allowlisted top-level act fields fill only missing nested request fields, nested values stay authoritative, and focused regression tests cover flattened, nested, and mixed payloads.

## Reproduction Assessment

Yes, at source level. Current main returns object-valued `params.request` before copying allowlisted top-level `ref`, then the act normalizer rejects click/type bodies without `ref` or `selector`.

## Solution Assessment

Yes. Normalizing both advertised shapes at the browser tool parser boundary is the narrowest maintainable fix and avoids changing the browser service contract; nested request fields should keep precedence.
