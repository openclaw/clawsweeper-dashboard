## Label Justifications

- `P1`: The issue breaks a real browser agent workflow for model-generated tool calls, especially OpenAI-family models that place `ref` at the advertised top level.
- `impact:auth-provider`: The failure is tied to model-choice behavior: OpenAI-style tool payloads use the top-level field shape while nested-ref payloads continue to work.

## Summary

Keep open because current main and the latest release still drop top-level browser `act` fields when a nested `request` object is present, while the open closing PR is already the focused fix path.

## What This Changes

The issue asks browser `act` to handle top-level `ref` beside a nested `request` consistently, or at least return a request-shape error instead of a browser-service error.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81076 with the same narrow parser backfill and regression tests, preserving nested request precedence and the `/act` service contract.

## Reproduction Assessment

Yes. Source inspection on current main shows `action: "act"` with top-level `ref` and `request: { kind: "click" }` returns only the nested request, then `/act` rejects it because `ref` or `selector` is absent from that body.

## Solution Assessment

Yes. Backfilling only existing allowlisted top-level act fields into the nested request when missing is the narrowest maintainable fix and keeps nested request fields authoritative.
