## Label Justifications

- `P1`: The bug breaks a real browser-agent workflow for model-generated tool calls, especially OpenAI-family payloads that place ref at the advertised top level.
- `impact:auth-provider`: The failure is tied to model-choice behavior: OpenAI-style tool payloads fail while nested-ref payloads from other model families continue to work.

## Summary

Keep open because current main and v2026.5.19 still drop top-level browser act fields when a nested request object is present, and the open closing PR is the right focused fix path.

## What This Changes

The issue requests consistent browser act handling for top-level ref fields beside a nested request, or at least a clearer request-shape error.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81076 with the same narrow parser backfill and regression tests, preserving nested request precedence and the /act service contract.

## Reproduction Assessment

Yes. Source inspection on current main shows action: "act" with top-level ref and request: { kind: "click" } returns only the nested request, then /act rejects it because ref or selector is absent from that body.

## Solution Assessment

Yes. Backfilling only existing allowlisted top-level act fields into the nested request when missing is the narrowest maintainable fix and keeps nested request fields authoritative.
