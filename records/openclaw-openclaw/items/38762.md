## Summary

Keep open: current main still returns the nested browser `act` request before copying top-level act fields, while the tool schema advertises `ref` both at top level and inside `request`. The linked fix PR at https://github.com/openclaw/openclaw/pull/81076 is still open and uses closing syntax for this issue, so this issue should remain the canonical tracker until that PR or an equivalent fix lands.

## What This Changes

The issue asks browser `act` to handle top-level `ref` consistently with nested `request.ref`, or at least report a request-shape error instead of a misleading browser-service error.

## Best Possible Solution

Merge https://github.com/openclaw/openclaw/pull/81076 or an equivalent parser-level backfill that preserves nested request precedence and adds regression coverage, then close this issue.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call browser `act` with `request: { kind: "click" }` and top-level `ref`, and current `readActRequestParam` returns only the nested request before the route normalizer checks for `ref`. I did not run a live Chrome Relay scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to normalize the already-advertised top-level act fields into the nested request only when missing, while keeping nested request fields authoritative. The open linked PR appears to target that parser boundary rather than changing the browser service contract.
