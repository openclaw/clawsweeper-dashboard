## Label Justifications

- `P1`: The issue breaks a real browser-agent workflow for model-generated tool calls that place `ref` at the advertised top level.
- `impact:auth-provider`: The failure is tied to provider/model payload behavior: OpenAI-style tool calls fail while nested-ref payloads continue to work.

## Summary

Keep open: current main and the latest release still return the nested browser `act` request before copying top-level `ref`/`selector` fields, and the open closing PR is the focused fix path.

## What This Changes

The issue asks browser `act` to accept or accurately diagnose top-level `ref` fields when a nested `request` object is present.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81076 with the same narrow parser backfill and regression tests, keeping nested request fields authoritative.

## Reproduction Assessment

Yes, at source level. Current main returns the nested `request` before copying top-level `ref`, then the act normalizer rejects click/type bodies without `ref` or `selector`.

## Solution Assessment

Yes. Backfilling existing allowlisted top-level act fields into `request` only when missing is the narrowest maintainable fix and keeps nested fields authoritative.
