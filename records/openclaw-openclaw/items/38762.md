## Summary

Keep open. Current main still drops top-level `ref` when `action="act"` also supplies an object `request`, and the only linked compatibility PR is closed unmerged against the old pre-plugin path.

## What This Changes

The issue asks for `browser action="act"` to accept top-level `ref` alongside an object `request`, or at least report a request-shape error instead of a browser-service reachability error.

## Best Possible Solution

Add a narrow browser tool normalizer that fills missing nested act fields from supported top-level params while preserving nested `request` precedence and the `/act` service contract.

## Reproduction Assessment

Yes, by source inspection. With `action: "act"`, top-level `ref`, and `request: { kind: "click" }`, `readActRequestParam` returns the nested request unchanged and `/act` receives no `ref` or `selector`.

## Solution Assessment

Yes. Backfilling missing nested act fields in `extensions/browser/src/browser-tool.ts` before `executeActAction` is narrower than changing `/act` route validation or browser control service error wrapping.
