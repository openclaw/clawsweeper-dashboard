## Summary

Keep open. Current `main` still drops top-level `ref` when `action: "act"` also has a nested `request`, and the closed compatibility PR targeted the old core tool path rather than the live browser plugin path.

## What This Changes

Issue reports inconsistent `browser action="act"` handling where `ref` works inside `request` but fails when supplied at the top level alongside `request: { kind: "click" }`.

## Best Possible Solution

Backfill supported top-level act fields into the nested request only when missing, preserve nested request precedence, and add a focused mixed-payload regression test.

## Reproduction Assessment

Yes, by source inspection. With `action: "act"`, top-level `ref`, and `request: { kind: "click" }`, `readActRequestParam` returns the nested request unchanged, so `/act` receives no `ref` or `selector`.

## Solution Assessment

Yes. A narrow normalizer in `extensions/browser/src/browser-tool.ts` is the maintainable fix because it keeps the `/act` service contract intact while honoring the tool schema’s accepted compatibility shape.
