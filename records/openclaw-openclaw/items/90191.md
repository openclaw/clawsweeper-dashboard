## Summary

Keep open: current main and v2026.6.8 still preserve native Codex `spawn_agent` as the primary Codex subagent surface while keeping OpenClaw `sessions_spawn` searchable for explicit OpenClaw/ACP delegation, so the reported competing-surface/product-decision problem remains unresolved.

## What This Changes

The issue asks OpenClaw to make Codex-harness delegation unambiguous when Codex-native `spawn_agent` and OpenClaw `sessions_spawn` are both reachable with different limits and ownership.

## Best Possible Solution

A maintainer should choose the Codex-harness delegation authority, then land one coherent policy across developer instructions, tool descriptions, docs, tests, and real Codex-harness proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 expose the split policy, and sibling Codex source proves native `spawn_agent` has a separate thread cap. I did not establish a deterministic live model turn that chooses the wrong tool.

## Solution Assessment

No single proposed fix is clearly best yet. Hiding, namespacing, stronger descriptions, or bridging native spawning all change ownership semantics, so maintainers need to choose the contract before implementation.
