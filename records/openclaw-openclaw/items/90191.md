## Summary

Keep open: current main and the latest release still intentionally keep Codex-native `spawn_agent` as the primary Codex subagent surface while leaving OpenClaw `sessions_spawn` searchable, so the competing-surface problem remains a real product decision rather than a stale fixed issue.

## What This Changes

The issue asks OpenClaw to remove or clearly disambiguate overlapping Codex-native `spawn_agent` and OpenClaw `sessions_spawn` delegation surfaces in the Codex app-server harness.

## Best Possible Solution

Choose the Codex delegation contract explicitly, then align Codex app-server prompt guidance, dynamic-tool exposure, docs, and tests so operators see one clear authority for limits and completion behavior.

## Reproduction Assessment

Yes, at source level: current main exposes OpenClaw `sessions_spawn` as a searchable Codex dynamic tool while instructing Codex to use native `spawn_agent` for Codex subagents, and sibling Codex source enforces a separate native thread limit. I did not run a live model turn to reproduce the model-selection mistake directly.

## Solution Assessment

No settled best fix yet: hiding, renaming, documenting, or bridging the two surfaces would all change the current shipped Codex delegation contract. The open guardrail PR is only a candidate and is blocked on maintainer product direction.
