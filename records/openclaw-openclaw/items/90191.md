## Summary

Keep open: current main and v2026.6.9 still intentionally keep Codex-native `spawn_agent` as the primary Codex subagent surface while OpenClaw `sessions_spawn` remains searchable, so the reported competing-surface/product-decision problem is not resolved. The open draft PR at https://github.com/openclaw/openclaw/pull/79518 is a candidate direction, but it is not merged or safe to treat as the canonical fix.

## What This Changes

The issue asks OpenClaw to remove, clarify, or mediate the overlapping Codex-native `spawn_agent` and OpenClaw `sessions_spawn` delegation surfaces in Codex app-server harness sessions.

## Best Possible Solution

Maintainers should choose one Codex-harness delegation contract, then align developer instructions, tool descriptions, docs, tests, and real Codex-harness proof around that contract.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 expose both reachable delegation surfaces, and sibling Codex source proves native `spawn_agent` has a separate thread cap. I did not establish a deterministic live model turn that chooses the wrong tool.

## Solution Assessment

No single implementation is clearly best yet. Hiding, namespacing, stronger descriptions, or bridging native spawning all change ownership semantics, so maintainers need to choose the contract before implementation.
