## Summary

Keep open: current main and the latest release still contain the Slack DM U-ID direct-send path alongside the D-channel live-reply path, so the reported routing/session-attribution split is not resolved on main.

## What This Changes

The issue asks for Slack DM normal replies and Control attribution to converge after v2026.5.2, where inbound/live DM routing and explicit message-tool sends can diverge between `user:U...` and `D...` targets.

## Best Possible Solution

Unify Slack-owned DM target/session handling so live replies, default current-conversation sends, and explicit user sends use consistent direct-session metadata while choosing the Slack API target form required by each delivery path.

## Reproduction Assessment

No high-confidence live reproduction of the full issue is available from this review. Source inspection does reproduce the core split: a Slack DM can carry `user:U...` persisted routing, `channel:D...` live replies, and D-channel default message-tool attribution.

## Solution Assessment

No, current main is not the best end state because it preserves divergent DM target forms without converging session metadata. The narrow maintainable fix is in the Slack plugin routing/send surfaces with regression coverage and live Slack proof, not a broad core rewrite.
