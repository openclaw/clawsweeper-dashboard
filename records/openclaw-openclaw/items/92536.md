## Summary

This issue should stay open because the reported Dreaming sidecar activation bug is still present on current main and is already paired with an open closing PR. Current source still lets a restrictive `plugins.allow` reject the bundled Dreaming engine as `not-in-allowlist`, while https://github.com/openclaw/openclaw/pull/92541 is the canonical active fix path with before/after CLI proof.

## What This Changes

The issue reports that Dreaming sweeps do not run when a Dreaming-compatible memory plugin owns the memory slot and a restrictive `plugins.allow` omits `memory-core`.

## Best Possible Solution

Review and land or reject the existing linked PR, keeping any allowlist exception narrowly scoped to the bundled Dreaming sidecar while preserving denylist and explicit `entries.memory-core.enabled: false` behavior.

## Reproduction Assessment

Yes. Source inspection shows that a non-empty `plugins.allow` omitting `memory-core` reaches the shared `not-in-allowlist` decision before the Dreaming sidecar can load, matching the reporter's config and the linked PR's before evidence.

## Solution Assessment

Yes. The best current path is the existing narrow activation-policy fix in the linked PR, not requiring users to add a separately disabled `memory-core` entry or opening a duplicate repair lane.
