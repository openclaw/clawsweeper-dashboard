## Summary

This issue remains necessary: current `main` supports Dreaming with external memory-slot owners, but a restrictive `plugins.allow` still disables the required `memory-core` sidecar when it is omitted.

## What This Changes

Ensure Dreaming sweeps run when a compatible external memory plugin owns the memory slot and a restrictive plugin allowlist omits the `memory-core` sidecar.

## Best Possible Solution

Extend the existing plugin auto-enable/materialization planner to admit `memory-core` only as the required Dreaming sidecar when a non-core selected memory capability has Dreaming enabled, while keeping explicit deny/disable behavior and UI runtime status consistent.

## Reproduction Assessment

Yes, at source level. Current code derives `memory-core` as the Dreaming sidecar and then deterministically filters it when the restrictive allowlist omits it; existing coverage only tests both plugins allowlisted.

## Solution Assessment

Yes. A narrow auto-enable/materialization repair uses the existing activation architecture and is safer than weakening global allowlist precedence or requiring a misleading disabled `memory-core` entry.
