Keep this issue open: current main and the latest release still show the allowlist activation gap, and https://github.com/openclaw/openclaw/pull/92541 is an open closing PR that now owns the focused fix path.

## What This Changes

The issue reports that Dreaming sweeps do not run when a Dreaming-compatible memory plugin owns the memory slot and restrictive `plugins.allow` omits `memory-core`.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/92541; the desired end state is a narrowly authorized `memory-core` Dreaming sidecar for enabled non-core memory plugins without widening the operator allowlist boundary.

## Reproduction Assessment

Yes. Source inspection shows that with a restrictive allowlist omitting `memory-core`, the shared activation gate returns `not-in-allowlist` before the `memory-core` entry can register the Dreaming `before_agent_reply` hook.

## Solution Assessment

Yes, the best fix shape is a narrow sidecar activation path in shared plugin activation state while preserving explicit deny and disabled gates. The open PR follows that direction, with maintainer review needed for the allowlist boundary and any separate cron-status signal follow-up.
