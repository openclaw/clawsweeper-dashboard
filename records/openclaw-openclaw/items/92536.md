## Summary

Keep open: current main and the latest release still show the Dreaming sidecar activation gap, and the open linked PR is the canonical fix path rather than a reason to close the issue before merge.

## What This Changes

The issue reports that Dreaming sweeps do not run when a Dreaming-compatible memory plugin owns the memory slot while a restrictive `plugins.allow` omits `memory-core`.

## Best Possible Solution

Review and land or revise the linked PR so `memory-core` activates only as the resolved bundled Dreaming sidecar, with regression coverage for restrictive allowlists that omit it.

## Reproduction Assessment

Yes, at source level: current main computes the Dreaming sidecar but the shared activation gate still returns `not-in-allowlist` when `memory-core` is omitted from a restrictive allowlist. I did not run a live gateway sweep in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to model the enabled Dreaming engine as a sidecar dependency in activation state, not to require users to add a disabled `memory-core` entry manually.
