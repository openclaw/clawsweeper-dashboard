## Label Justifications

- `P2`: A supported memory configuration silently loses scheduled Dreaming sweeps, but explicitly allowlisting memory-core provides a bounded workaround.
- `impact:session-state`: The missing sweeps prevent configured memory consolidation, reflection, and promotion state from advancing.

## Summary

Current main intends memory-core to run as the Dreaming engine beside another selected memory plugin, but restrictive plugin allowlists disable that sidecar before the loader and gateway-startup exceptions can take effect. This is a valid, bounded activation bug rather than intended operator behavior, and no canonical open fix was found.

## What This Changes

Ensure Dreaming sweeps work with a Dreaming-compatible memory plugin when a restrictive plugin allowlist omits the memory-core sidecar.

## Best Possible Solution

Represent the enabled Dreaming engine as a narrowly authorized runtime dependency of the selected memory-slot plugin, then cover both loader and gateway-startup behavior without permitting unrelated plugins to bypass restrictive allowlists.

## Reproduction Assessment

Yes. Current-main source shows that a non-empty `plugins.allow` omitting memory-core produces `not-in-allowlist` before the Dreaming-sidecar exceptions can activate it, matching the reported missing schedule.

## Solution Assessment

No, requiring users to allowlist a separately disabled memory-core entry is not the best solution. Dependency-aware activation is narrower, preserves the allowlist security model, and matches the existing sidecar design.
