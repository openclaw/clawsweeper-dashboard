## Summary

Keep open: the branch still has blocking behavior conflicts with current WebChat reload, gateway session lookup, and cron lifecycle contracts, and the PR lacks after-fix real behavior proof.

## What This Changes

The PR changes WebChat final-event reload handling, gateway session entry resolution, and cron session reaping/tests to suppress no-message final reloads, prefer exact session keys, and mark old running cron run rows done.

## Best Possible Solution

Rework as narrower fixes: distinguish command-generated finals without disabling terminal reconciliation, fix archived-session selection while preserving freshest duplicate semantics, and recover cron rows only with runtime ownership or timeout-aware lifecycle proof.

## Reproduction Assessment

Yes. Source and current tests show the PR-introduced regressions: no-message final reload coverage, duplicate-row session lookup/history tests, and cron timeout plus lifecycle code make the unsafe paths clear.

## Solution Assessment

No. The proposed patch uses broad heuristics that conflict with established behavior; the safer fix is a narrower command/session-target distinction plus ownership-aware cron recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve no-message final history reloads:** `ui/src/ui/chat-event-reload.ts:10-12`
  - body: Returning false whenever `payload.message` is absent disables the fallback reload path for final events that reconcile deferred `session.message` updates or no-payload terminal events. Current main explicitly expects no-message finals to reload history, so `/new` handling needs a command-specific signal instead of a blanket skip.
  - confidence: 0.9
- **[P1] Keep freshest duplicate session rows:** `src/gateway/session-utils.ts:400-408`
  - body: `loadSessionEntry` now prefers `store[trimmedKey]` or `store[canonicalKey]` before checking duplicate store keys. That can return stale metadata and history when a fresher legacy or case-variant duplicate exists, which current gateway tests cover.
  - confidence: 0.92
- **[P1] Prove cron runs are unowned before marking done:** `src/cron/session-reaper.ts:106-109`
  - body: The reaper marks any running cron row older than two hours as done without checking whether a job still owns it. Agent-turn cron jobs can disable or extend timeouts, and lifecycle `updatedAt` is not a heartbeat, so active runs can be finalized and later pruned.
  - confidence: 0.93
- **[P2] Use recovery time when updatedAt is missing:** `src/cron/session-reaper.ts:107`
  - body: When `entry.updatedAt` is missing, `updatedAt` becomes 0 and the recovered row is persisted with `endedAt: 0`, which would display the Unix epoch instead of the recovery time.
  - confidence: 0.84
