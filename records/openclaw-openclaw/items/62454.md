## Summary

Keep this PR open for contributor or maintainer rework: the reported session-management bugs are plausible, but the current branch introduces regressions against existing reload, duplicate-session, and cron lifecycle contracts, and it still lacks after-fix real behavior proof.

## What This Changes

The branch changes WebChat final-event reload decisions, gateway session-store entry resolution, and cron run-session reaper behavior/tests.

## Best Possible Solution

Rework into targeted fixes that preserve current reload and duplicate-session contracts, use command/session-specific signals for the UI cases, and recover cron rows only with ownership-aware lifecycle proof and maintainer-approved terminal semantics.

## Reproduction Assessment

Yes for the proposed patch regressions by source inspection: current tests assert no-message final events reload history and duplicate session reads prefer the freshest row. The original user symptoms still do not have high-confidence real-setup proof in the PR.

## Solution Assessment

No. The branch uses broad no-message, exact-key, and age-only heuristics where current main needs narrower command/session targeting, freshest duplicate preservation, and ownership-aware cron recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve no-message final history reloads:** `ui/src/ui/chat-event-reload.ts:10-12`
  - body: Returning false whenever payload.message is absent disables fallback reloads for final events that intentionally omit message payloads. Current main explicitly tests no-message finals as reloading, so /new handling needs a command- or session-specific signal instead of changing this global predicate.
  - confidence: 0.9
- **[P1] Keep freshest duplicate session rows:** `src/gateway/session-utils.ts:405-413`
  - body: loadSessionEntry now prefers store[trimmedKey] or store[canonicalKey] before checking duplicate store keys. With a stale canonical row and a fresher case-variant duplicate, this returns stale metadata/history and regresses the current duplicate-resolution tests.
  - confidence: 0.94
- **[P1] Prove cron runs are abandoned before marking done:** `src/cron/session-reaper.ts:106-109`
  - body: The reaper marks any running cron run row older than two hours as done without proving the job no longer owns that session. Explicit cron timeouts can exceed that threshold or be disabled, and updatedAt is not a heartbeat, so active runs can be finalized and later pruned.
  - confidence: 0.92
- **[P1] Recover the stable cron session row:** `src/cron/session-reaper.ts:106-109`
  - body: The recovery loop only updates :run: alias rows, but current isolated cron state is persisted under the stable base cron key and session listing hides run aliases. A sweep can therefore report recovery while the visible cron session still shows running.
  - confidence: 0.91
- **[P2] Run recovery when pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: The new recovery scan still sits behind the retentionMs === null early return. Users who disable retention to keep history would also lose crash-status recovery, which conflicts with the recovery behavior this patch is adding.
  - confidence: 0.88
- **[P3] Avoid epoch endedAt values:** `src/cron/session-reaper.ts:107`
  - body: When a running row lacks updatedAt, this branch falls back to 0 and persists endedAt: 0, which can surface as the Unix epoch. Use the sweep time or another valid terminal timestamp instead.
  - confidence: 0.82
