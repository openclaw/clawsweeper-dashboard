## Summary

Keep open: the PR is a focused bug-fix attempt, but current main source and tests show the branch would regress established WebChat reload and session duplicate-resolution behavior, while the cron recovery path needs ownership-aware semantics and real behavior proof before merge.

## What This Changes

The PR changes WebChat final-event reload logic, gateway session-entry lookup, and cron reaper handling/tests for old running cron run rows.

## Best Possible Solution

Rework into targeted fixes that preserve current reload and duplicate-session contracts, use command/session-specific signals for the UI cases, and recover cron rows only with ownership-aware lifecycle proof.

## Reproduction Assessment

Yes for the proposed patch regressions by source inspection: current tests assert no-message finals reload history and duplicate session reads prefer the freshest row. The original user symptoms still do not have high-confidence real-setup proof in the PR.

## Solution Assessment

No. The branch uses global no-message, exact-key, and age-only heuristics where current main needs narrower command/session targeting and ownership-aware cron lifecycle handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve no-message final history reloads:** `ui/src/ui/chat-event-reload.ts:10-12`
  - body: Returning false whenever `payload.message` is absent disables the fallback reload path for final events that intentionally omit a message payload. Current main explicitly tests that no-message finals reload history, so `/new` handling needs a command-specific signal instead of changing this global predicate.
  - confidence: 0.9
- **[P1] Keep freshest duplicate session rows:** `src/gateway/session-utils.ts:405-413`
  - body: `loadSessionEntry` now prefers `store[trimmedKey]` or `store[canonicalKey]` before checking duplicate store keys. With a stale canonical row and fresher case-variant duplicate, this returns stale metadata/history and regresses the current duplicate-resolution tests.
  - confidence: 0.94
- **[P1] Prove cron runs are unowned before marking done:** `src/cron/session-reaper.ts:106-109`
  - body: The reaper marks any running cron run row older than two hours as done without proving the job no longer owns that session. Explicit cron timeouts can exceed that threshold or be disabled, and lifecycle `updatedAt` is not a heartbeat, so active runs can be finalized and later pruned.
  - confidence: 0.92
- **[P1] Recover the visible base cron row:** `src/cron/session-reaper.ts:106-109`
  - body: The recovery loop only considers `isCronRunSessionKey` entries and writes `store[key]`, but current isolated cron state is surfaced through the stable base cron key while session listing hides `:run:` aliases. A sweep can therefore report recovery while the visible cron session stays running.
  - confidence: 0.9
- **[P2] Run recovery when pruning is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: The new recovery scan still sits behind the `retentionMs === null` early return. Users who disable retention to keep history would also lose crash recovery, which conflicts with the recovery behavior this patch is adding.
  - confidence: 0.86
- **[P3] Use recovery time when updatedAt is missing:** `src/cron/session-reaper.ts:107`
  - body: When a running row lacks `updatedAt`, the branch falls back to `0` and persists `endedAt: 0`, which can display or compare as the Unix epoch. Use the sweep time as the fallback terminal timestamp.
  - confidence: 0.82
