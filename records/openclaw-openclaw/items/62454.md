## Summary

Keep open, but the current PR patch is not merge-ready. Current main does not fully implement the three intended fixes, yet the PR conflicts with existing WebChat and session-store contracts and uses a time-only cron recovery heuristic that can finalize legitimate long-running work.

## What This Changes

The PR changes WebChat final-event reload handling, gateway session entry resolution, and cron session reaping/tests to suppress no-message final reloads, prefer exact session keys, and mark old running cron run rows done.

## Best Possible Solution

Rework the fixes as narrower changes: keep the current WebChat final reload reconciliation from main, add any archived-session click fix without breaking freshest duplicate alias handling, and base cron recovery on runtime/task ownership or timeout-aware lifecycle proof rather than a fixed two-hour session-row age heuristic.

## Reproduction Assessment

Yes. The PR-introduced regressions have high-confidence source/test reproductions through the current WebChat final reload tests, session duplicate-row tests, direct history tests, and the 48-hour agent timeout plus lifecycle timestamp contract.

## Solution Assessment

No. The PR addresses plausible symptoms with broad heuristics that conflict with existing contracts; a safer solution should distinguish /new or /reset finals at the command/run level, scope exact-key selection to true archived keys, and make cron recovery prove the run is no longer owned before marking it terminal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve no-message final history reloads:** `ui/src/ui/chat-event-reload.ts:10-12`
  - body: Returning false for every final event without a message treats all such finals as /new or /reset, but current WebChat uses missing-message finals as the fallback signal to reload history and replay deferred session.message updates. This can leave the active chat stale for cross-run or no-payload terminal events.
  - confidence: 0.9
- **[P1] Keep freshest duplicate session rows:** `src/gateway/session-utils.ts:400-408`
  - body: loadSessionEntry now chooses store[trimmedKey] or store[canonicalKey] before checking the freshest duplicate. Current gateway tests require the fresher duplicate row to win for legacy/case-variant aliases, so this can return stale session metadata and history when duplicate rows exist.
  - confidence: 0.92
- **[P1] Do not finalize active cron runs by age alone:** `src/cron/session-reaper.ts:107-109`
  - body: The reaper marks any running cron session older than two hours as done without checking whether the run is still owned by the gateway. Cron agent runs can legitimately run far longer, and lifecycle updatedAt is only guaranteed at start/end, so this can incorrectly finalize and later prune an active run.
  - confidence: 0.93
- **[P2] Use a non-epoch endedAt fallback:** `src/cron/session-reaper.ts:108`
  - body: When a running entry has no updatedAt, updatedAt is coerced to 0 and the recovered row is persisted with endedAt: 0. Any UI or API that displays the terminal timestamp would show the Unix epoch instead of the recovery time.
  - confidence: 0.84
