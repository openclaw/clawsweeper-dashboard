## Summary

Keep open. The PR targets a real hook diagnostics gap, but the proposed change only adds stack and event details as structured logger metadata; current compact/pretty console output, including the Docker-style non-TTY path described by the linked bug, still renders the message string without that metadata.

## What This Changes

The PR changes `triggerInternalHook` so caught internal hook errors are logged with `Error.stack` and limited `{ type, action, sessionKey }` event metadata.

## Best Possible Solution

Revise the hook failure path so stack traces remain in structured file/JSON metadata and also appear on the intended operator console surface, using the existing `consoleMessage` mechanism or an equivalent message-level format while avoiding full `event.context` or `messages`. Add focused regression coverage for the thrown-hook log output and include a changelog entry if this remains a user-facing `fix`.

## Reproduction Assessment

Yes. Register an internal hook handler that throws and call `triggerInternalHook`; current main logs only the formatted error message, and the PR's added metadata would still be hidden by default compact/pretty console formatting.

## Solution Assessment

No. Structured metadata is useful, but by itself it is not the narrowest complete fix for the reported Docker/operator log problem; the stack must be included in the emitted console message while preserving limited structured metadata for file and JSON logs.
