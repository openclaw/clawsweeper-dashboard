## Summary

Keep open. Current main still has message-only internal-hook error logging, and this PR is useful source work, but metadata-only stack/event fields do not solve the default compact/pretty console logs described by the linked Docker report.

## What This Changes

The PR changes `triggerInternalHook` to pass `Error.stack` and `{ type, action, sessionKey }` event metadata to `log.error` when an internal hook handler throws.

## Best Possible Solution

Revise the PR so hook failures keep narrow structured stack/event metadata for file and JSON logs, include a redacted stack on the default console surface, add focused regression coverage, and add the required changelog entry.

## Reproduction Assessment

Yes. Register an internal hook handler that throws and call `triggerInternalHook`; current main logs only the formatted message, and the PR’s metadata would still be invisible in default compact/pretty console output.

## Solution Assessment

No. Structured metadata is useful, but metadata-only logging is not the narrowest complete fix for the reported operator log problem; using `consoleMessage` while preserving file/JSON metadata is safer.
