## Summary

Keep open. Current main still logs internal hook handler failures with only the formatted error message, and this PR is useful source work, but the proposed patch does not put the stack trace on the default operator console surface where the linked Docker report needs it. It should be revised before merge rather than closed as obsolete or implemented.

## What This Changes

The PR changes `triggerInternalHook` to pass an `Error.stack` value and `{ type, action, sessionKey }` event metadata to `log.error` when an internal hook handler throws.

## Best Possible Solution

Revise the PR so hook failures keep narrow structured stack/event metadata for file and JSON logs, also include a redacted stack on the default console surface via `consoleMessage` or the emitted message, add focused regression coverage, and add the required changelog entry if this remains a user-facing diagnostics fix.

## Reproduction Assessment

Yes. Register an internal hook handler that throws and call `triggerInternalHook`; current main reaches the catch block that logs only the formatted message, and the PR’s metadata would still be invisible in default compact/pretty console output.

## Solution Assessment

No. Structured metadata is useful, but metadata-only logging is not the narrowest complete fix for the reported operator log problem; using the existing console-message path while preserving file/JSON metadata is safer.
