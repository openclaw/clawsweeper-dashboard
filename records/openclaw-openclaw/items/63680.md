## Summary

Keep open: current main still lacks the self-approval check, but this PR is not ready to merge because its guard conflicts with current approval visibility/trusted-backend replay behavior, its new tests are stale against current main, and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds a same-connection rejection to `exec.approval.resolve`, adds focused self-approval tests, and updates node.invoke approval-bypass tests to use separate requester and resolver sockets.

## Best Possible Solution

Refresh the gateway approval hardening so requester self-allow is rejected while device/client/trusted-backend approval flows remain intentional, covered, and proven in a real setup.

## Reproduction Assessment

Yes. By source inspection, current main records `requestedByConnId` during `exec.approval.request` and resolves approvals without comparing it to the resolving `client.connId`; I did not run tests because this review is read-only.

## Solution Assessment

No as submitted. The resolve validation hook is the right place for the guard, but this branch needs to align tests with current visibility, preserve or explicitly revise trusted-backend replay behavior, and clean up rejected approval timers.
