## Summary

The security hardening target is valid by source inspection, but this PR is not merge-ready on current main: the new tests do not match approval visibility, the guard would break trusted-backend chat approval replay, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a same-connection self-approval guard to `exec.approval.resolve`, adds focused self-approval tests, and updates node.invoke approval-bypass tests to use a separate resolver socket.

## Best Possible Solution

Refresh the PR against current main and enforce requester self-allow rejection while preserving intentional device, approval-runtime, and trusted-backend turn-source approval flows with realistic regression coverage.

## Reproduction Assessment

Yes. Current main records `requestedByConnId` during `exec.approval.request` and resolves approvals without comparing it to the resolving `client.connId`; I did not run tests because this pass is read-only.

## Solution Assessment

No as submitted. The resolve validation path is the right place for the guard, but the patch needs visibility-aware tests and an explicit trusted-backend/chat replay policy before merge.
