## Summary

Keep open: the PR targets a real exec approval security gap on current main, but the submitted patch still needs real behavior proof and has merge-blocking issues around current approval visibility and trusted-backend chat approval replay.

## What This Changes

The PR adds a same-connection self-approval guard to `exec.approval.resolve`, adds focused self-approval tests, and updates node.invoke approval-bypass tests to resolve approvals from a second socket.

## Best Possible Solution

Refresh the gateway approval hardening so requester self-allow is rejected while current device/client/trusted-backend approval flows remain intentional, tested, and proved in a real setup.

## Reproduction Assessment

Yes, by source inspection: current main records `requestedByConnId` during `exec.approval.request` and resolves approvals without comparing it to the resolving `client.connId`. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted. The guard belongs in the resolve validation path, but the branch needs to align tests with current visibility, preserve or explicitly revise trusted-backend chat replay, clean rejected-path timers, and provide real behavior proof.
