## Summary

This PR should stay open: current main still lacks requester/resolver separation for exec approval resolution, and the branch is a focused fix for that remaining security-sensitive gateway path. The production guard is directionally sound for the reported same-connection bypass, but the new unit test helper leaves rejected approvals pending until their real 30-second timeout, so the patch is not merge-ready as-is.

## What This Changes

The PR adds an `exec.approval.resolve` guard that rejects non-deny decisions from the same WebSocket connection that created the exec approval, adds focused unit tests, and updates node-invoke approval bypass tests to use separate requester and resolver sockets.

## Best Possible Solution

Keep the PR open, fix the new test cleanup issue, and have a gateway/security maintainer confirm the intended identity boundary before landing a narrow access-control change with the focused regression tests.

## Reproduction Assessment

Yes. Current main shows a high-confidence reproduction path by inspection: a pending exec approval can be created with `requestedByConnId`, then the same `client.connId` can reach `manager.resolve` because resolve validation does not compare requester and resolver identity.

## Solution Assessment

Unclear as submitted. The production change is the narrowest fix for the reported same-WebSocket bypass, but the branch should not merge until the new rejected-path tests clear their pending approval timers and maintainers confirm that `connId` is the right identity boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clean up rejected approval timers:** `src/gateway/server-methods/exec-approval-self-approval.test.ts:39`
  - body: `createPending()` registers each fake approval with a real 30-second timer. In the same-connection rejection cases the new validation returns before `manager.resolve()` clears that timer, so the pending record and handle stay alive until timeout; this can leave open handles or add a 30s stall to the Vitest process. Clean these records in `afterEach`, use fake timers, or make the helper use a controlled short timeout.
  - confidence: 0.87
