## Summary

Keep this PR open. Current main still lacks a requester/resolver identity check for exec approval resolution, so the PR targets a real gateway access-control gap, but it should not merge until the new rejected-path tests clean up pending approval timers, the required changelog entry is added, and a gateway/security maintainer confirms that `connId` is the intended boundary.

## What This Changes

The PR adds an `exec.approval.resolve` guard that rejects non-deny decisions from the same WebSocket connection that created the exec approval, adds focused unit tests, and updates node-invoke approval bypass tests to use separate requester and resolver sockets.

## Best Possible Solution

Land a narrow gateway access-control hardening only after the rejected-path test cleanup and changelog are fixed, and after a gateway/security maintainer confirms whether same-connection blocking is sufficient or whether device/client identity should also be considered.

## Reproduction Assessment

Yes. Current main can be reproduced by inspection: create an exec approval on a connection with `requestedByConnId`, then resolve it with `allow-once` on the same `client.connId`; the current validation and manager paths do not compare those identities.

## Solution Assessment

Unclear as submitted. The production guard is the narrowest fix for the same-WebSocket bypass described by the PR, but the patch should not merge until the test timer leak and changelog omission are fixed and the `connId` boundary is explicitly accepted.
