## Summary

Keep open: current main still lacks same-connection self-approval enforcement, but this external security-boundary PR is not ready because real behavior proof is missing and the branch conflicts with current approval visibility and trusted-backend chat replay behavior.

## What This Changes

The PR adds a same-connId guard to `exec.approval.resolve`, adds focused self-approval tests, and updates node.invoke approval-bypass tests to use a separate resolver socket.

## Best Possible Solution

Refresh the gateway approval hardening against current main so requester self-allow is rejected while existing device, approval-runtime, and trusted-backend chat approval flows remain intentional and covered.

## Reproduction Assessment

Yes. By source inspection, current main records `requestedByConnId` during `exec.approval.request` and resolves approvals without comparing it to the resolving `client.connId`; I did not run tests because this review is read-only.

## Solution Assessment

No as submitted. The guard belongs in the resolve validation path, but the branch needs current-main test fixtures, trusted-backend replay handling or an explicit policy change, rejected-path cleanup, and real behavior proof.
