## Summary

Keep open: current main still lacks same-connection self-approval enforcement, but the submitted PR remains blocked by missing real behavior proof and two correctness issues around trusted-backend chat replay and rejected-path timer cleanup.

## What This Changes

The PR adds a connId-based self-approval guard to `exec.approval.resolve`, adds focused self-approval tests, and updates some node-invoke approval-bypass tests to use separate requester and resolver sockets.

## Best Possible Solution

Land a refreshed gateway approval hardening that rejects requester self-allow decisions, preserves or explicitly revises trusted-backend chat replay semantics, cleans timer lifecycle in tests, and includes redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main stores `requestedByConnId` on exec approval requests, then resolves approvals without comparing it to the resolving `client.connId`, so a same-connection requester can attempt an allow decision.

## Solution Assessment

No as submitted. The guard is the right narrow seam for the same-WebSocket bypass, but the branch needs to handle current trusted-backend chat replay, clean up rejected-path timers, and provide real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trusted-backend chat replay approvals:** `src/gateway/server-methods/exec-approval.ts:358-366`
  - body: This guard rejects every non-deny resolve from the same `connId`, but current chat replay coverage still creates and resolves chat approvals on one trusted-backend WebSocket before replaying through another backend connection. Update that flow to resolve from a distinct approver connection, or explicitly preserve/revise the trusted-backend policy, otherwise the existing chat replay path will fail before `node.invoke` is exercised.
  - confidence: 0.86
- **[P2] Clean up rejected approval timers:** `src/gateway/server-methods/exec-approval-self-approval.test.ts:39-41`
  - body: `createPending()` registers each fake approval with a real 30-second timer, and the same-connection rejection path returns before `manager.resolve()` or `expire()` clears it. Clean up those records in the tests, use controlled timers, or avoid registering long-lived timers for rejection-only cases so the suite does not leave pending handles until timeout.
  - confidence: 0.87
