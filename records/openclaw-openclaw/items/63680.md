## Summary

Keep open: current main still has the same-connection approval gap, but this external security PR is not ready to merge because it lacks real behavior proof and is stale against current gateway approval visibility and trusted-backend chat replay behavior.

## What This Changes

The PR adds a connId self-approval guard to `exec.approval.resolve`, adds focused self-approval tests, and updates node.invoke approval-bypass tests to use a separate resolver socket.

## Best Possible Solution

Refresh the gateway approval hardening against current main so requester self-allow is rejected while current device/client/trusted-backend approval flows remain intentional and covered.

## Reproduction Assessment

Yes, by source inspection. Current main records `requestedByConnId` on exec approval requests, then resolves approvals without comparing it to the resolving `client.connId`; I did not run tests because this review is read-only.

## Solution Assessment

No as submitted. The guard belongs in the resolve validation path, but the branch needs to align with current approval visibility, preserve or explicitly revise trusted-backend chat replay, clean timer lifecycle in tests, and provide real behavior proof.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P2] Align allow-path tests with current approval visibility:** `src/gateway/server-methods/exec-approval-self-approval.test.ts:72-75`
  - body: `createPending()` only sets `requestedByConnId`, and the resolver mock only supplies a different `connId`. On current main, a connId-bound approval with no device/client/admin metadata is visible only to that same connId, so this allow-path test will get an unknown/expired approval instead of proving cross-connection approval; populate the fixture with current device/client/admin/internal approval metadata before expecting success.
  - confidence: 0.88
- **[P2] Preserve trusted-backend chat replay approvals:** `src/gateway/server-methods/exec-approval.ts:362-368`
  - body: This guard rejects every non-deny resolve from the same `connId`, but current main still creates chat approvals by requesting and resolving on one trusted-backend WebSocket before replaying through another backend connection. Update that flow to resolve from a distinct approver connection, or explicitly preserve/revise the trusted-backend policy, otherwise the current chat replay path fails before `node.invoke` is exercised.
  - confidence: 0.87
- **[P2] Clean up rejected approval timers:** `src/gateway/server-methods/exec-approval-self-approval.test.ts:40-41`
  - body: `createPending()` registers each fake approval with a real 30-second timer, and the same-connection rejection path returns before `manager.resolve()` or `expire()` clears it. Clean these records up with fake timers, `afterEach`, or explicit expiry so rejection tests do not leave pending handles until timeout.
  - confidence: 0.86
