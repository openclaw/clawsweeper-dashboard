## Summary

Keep open: current main still has the self-approval gap, and this PR is a plausible fix path, but it is not merge-ready because it lacks real behavior proof and conflicts with current approval visibility/trusted-backend replay behavior.

## What This Changes

The PR adds a same-connId self-approval rejection in `exec.approval.resolve`, adds focused self-approval tests, and updates node.invoke approval-bypass tests to use a separate resolver socket.

## Best Possible Solution

Refresh the gateway approval hardening against current main so requester self-allow is rejected while current device/client/trusted-backend approval flows remain intentional and covered.

## Reproduction Assessment

Yes, by source inspection: current main records `requestedByConnId` during `exec.approval.request` and resolves approvals without comparing it to the resolving `client.connId`. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted. The guard belongs in the resolve validation path, but the branch needs to align tests with current visibility, preserve or explicitly revise trusted-backend chat replay, clean rejected-path timers, and provide real behavior proof.
