## Summary

Keep open. The linked bug is still present on current main, and this PR is a focused implementation with regression coverage and no blocking correctness or security finding; it needs maintainer review and current-head validation rather than cleanup closure.

## What This Changes

The PR adds opt-in retry support to `fetchRemoteMedia`, enables three-attempt retry in `MediaAttachmentCache`, covers retry/fail-fast cases in tests, and adds a changelog fix entry.

## Best Possible Solution

Land this shared retry implementation after current-head validation, keeping the retry policy at the media fetch/cache boundary and preserving SSRF, 4xx, abort/timeout, and maxBytes fail-fast behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a remote Discord audio/media-understanding attachment through the shared cache and make the first guarded fetch fail transiently or return 5xx; current main performs one fetch without retry options.

## Solution Assessment

Yes. The shared fetch/cache retry is the narrowest maintainable fix because it covers Discord audio preflight and other media-understanding consumers without adding Discord-specific behavior.
