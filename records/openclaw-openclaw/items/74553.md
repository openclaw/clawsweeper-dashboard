## Summary

Keep open. Current main still has single-shot remote media fetching, while this PR is a focused implementation with tests and no blocking review findings; it should get normal maintainer review and current-head validation before landing.

## What This Changes

Adds opt-in retry support to `fetchRemoteMedia`, enables three-attempt retry in `MediaAttachmentCache`, updates media retry/fail-fast tests, and adds a changelog fix.

## Best Possible Solution

Land this shared fetch/cache retry implementation after current-head validation, preserving SSRF, 4xx, abort/timeout, and maxBytes fail-fast behavior so #74316 closes on merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a remote Discord audio or media-understanding attachment through the shared cache and make the first guarded fetch fail transiently or return 5xx; current main performs one fetch without retry options.

## Solution Assessment

Yes. The shared `fetchRemoteMedia` / `MediaAttachmentCache` boundary is the narrowest maintainable fix because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy.
