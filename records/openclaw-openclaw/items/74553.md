## Summary

Keep open: current main still lacks the requested retry behavior, and this PR is the active focused implementation candidate with tests, changelog coverage, and green CI. I found no blocking correctness or security issue in the proposed patch.

## What This Changes

The PR adds opt-in retry options to remote media fetching, enables bounded retry for media-understanding attachment downloads, expands retry/fail-fast tests, and adds a changelog fix entry.

## Best Possible Solution

Land the narrow shared retry implementation after maintainer review and current-head validation, while preserving SSRF, 4xx, abort/timeout, and maxBytes fail-fast behavior so the linked bug closes on merge.

## Reproduction Assessment

Yes. Against current main, a URL-backed Discord audio/media-understanding attachment whose first fetch throws a transient network/read error or returns 5xx flows through `MediaAttachmentCache.getBuffer` to one `fetchRemoteMedia` attempt and fails before STT can run.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable place to solve this because both Discord audio preflight and normal media-understanding reuse it, avoiding Discord-specific retry policy.
