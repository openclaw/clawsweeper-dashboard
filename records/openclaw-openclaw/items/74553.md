## Summary

Keep open. Current main still has the single-shot media-understanding remote fetch behavior, while this focused PR addresses the linked reliability bug with bounded shared retries and no blocking review finding found.

## What This Changes

Adds opt-in retry support to `fetchRemoteMedia`, enables a three-attempt retry policy in `MediaAttachmentCache`, expands fetch/cache tests, and adds a changelog fix for remote media-understanding attachments.

## Best Possible Solution

Land this shared retry implementation after current-head validation, preserving SSRF, 4xx, abort/timeout, and maxBytes fail-fast behavior so the linked reliability issue closes on merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a remote Discord audio/media-understanding attachment through the shared cache and make the first guarded fetch fail transiently or return 5xx; current main performs one fetch without retry options.

## Solution Assessment

Yes. The shared fetch/cache retry is the narrowest maintainable solution because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy.
