## Summary

Keep open. Current main still performs the shared remote media attachment fetch as a single operation, while this focused PR adds bounded retry at the shared fetch/cache boundary with tests for the retryable and fail-fast cases; I found no blocking diff finding, so it should proceed through maintainer review and exact-head validation.

## What This Changes

The PR adds opt-in retry support to `fetchRemoteMedia`, enables a three-attempt retry policy in `MediaAttachmentCache`, expands media retry/cache tests, and adds a changelog fix for #74316.

## Best Possible Solution

Land this shared retry implementation after required current-head checks, keeping retry bounded at the fetch/cache boundary so #74316 closes on merge.

## Reproduction Assessment

Yes. On current main, route a remote Discord audio/media-understanding attachment through `MediaAttachmentCache` and make the first guarded fetch fail transiently or return 5xx; the cache calls `fetchRemoteMedia` once and the attachment never reaches transcription.

## Solution Assessment

Yes. The PR fixes the shared media fetch/cache boundary instead of adding Discord-specific retry logic, and preserves fail-fast handling for SSRF blocks, 4xx responses, aborts/timeouts, and maxBytes failures.
