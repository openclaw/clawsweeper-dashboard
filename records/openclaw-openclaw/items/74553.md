## Summary

Keep open. Current main still lacks general transient retry for media-understanding remote attachment fetches, while this PR is a focused implementation with tests and no blocking correctness or security finding from this review.

## What This Changes

Adds opt-in retry support to `fetchRemoteMedia`, enables three-attempt retry in `MediaAttachmentCache`, expands media fetch/cache tests, and records a changelog fix.

## Best Possible Solution

Land this shared retry implementation after current-head validation, preserving SSRF, 4xx, abort/timeout, and maxBytes fail-fast behavior so #74316 closes on merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a remote Discord audio/media-understanding attachment through the shared cache and make the first guarded fetch fail transiently or return 5xx; current main performs one fetch without retry options.

## Solution Assessment

Yes. The shared fetch/cache retry is the narrowest maintainable fix because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy.
