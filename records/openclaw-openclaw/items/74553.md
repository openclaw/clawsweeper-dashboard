## Summary

Keep open. Current main still has the reported single-shot remote media fetch behavior, while this PR is a focused implementation candidate with bounded retry, focused tests, a changelog entry, and no blocking correctness or security finding from this pass.

## What This Changes

The PR adds opt-in retry support to `fetchRemoteMedia`, enables a three-attempt bounded retry policy in `MediaAttachmentCache`, updates fetch/cache tests, and adds a changelog fix entry.

## Best Possible Solution

Land this narrow shared media-fetch retry after maintainer review, preserving fail-fast behavior for 4xx responses, SSRF/policy blocks, caller aborts/timeouts, and maxBytes failures; then let linked #74316 close after merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, make a remote media attachment fetch fail transiently through `MediaAttachmentCache.getBuffer`; the cache calls `fetchRemoteMedia` once and rethrows. The linked issue supplies real Discord conditions, and the PR adds synthetic failing/passing coverage for the retry cases.

## Solution Assessment

Yes. The shared `fetchRemoteMedia` / `MediaAttachmentCache` boundary is the narrowest maintainable solution because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy.
