## Summary

Keep open: the recovery goal is plausible, but the current patch can count one failed WebSocket reconnect twice and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch adds a node-host gateway transport watchdog that exits after repeated 1006 gateway closes or 502 connection failures, with colocated runner tests.

## Best Possible Solution

Keep the supervisor-recovery behavior, but count one classified transport failure per reconnect attempt, preferably at the GatewayClient boundary or by suppressing the close event that belongs to an already-counted connect error.

## Reproduction Assessment

Yes for the PR-introduced blocker: `ws@8.20.0` turns a 502 handshake response into an error followed by close, and current `GatewayClient` forwards both callbacks. I did not run a live node-host/systemd reproduction because this review was read-only.

## Solution Assessment

No. Exiting on repeated unhealthy transport attempts may be the right recovery direction, but this implementation counts callbacks rather than reconnect attempts and can terminate too early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Count each reconnect attempt only once:** `src/node-host/runner.ts:319`
  - body: A 502 WebSocket handshake failure can emit both `onConnectError` and a later close, and this patch records both into the same three-event watchdog. That lets two failed reconnects terminate the node host, so please count one transport failure per reconnect attempt or suppress the close after an already-counted connect error.
  - confidence: 0.9
