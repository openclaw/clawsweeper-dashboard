## Summary

Keep open: this is a plausible focused node-host recovery fix and current main does not include the proposed repeated-1006/502 watchdog, but the external PR still lacks after-fix real behavior proof and changes a user-visible node-host exit policy that needs documentation and maintainer judgment.

## What This Changes

Adds a node-host gateway drop watchdog that exits with code 1 after repeated abnormal WebSocket closes or 502 handshake failures and adds unit coverage for the watchdog and double-count guard.

## Best Possible Solution

Land a supervised node-host recovery policy only after redacted real node-host/supervisor proof, with docs that clearly distinguish repeated 1006/502 exit behavior from ordinary in-process retry behavior.

## Reproduction Assessment

Yes at source level: current main logs 502 connect failures and 1006 closes while GatewayClient retries, and ws 8.20.0 supplies the error-plus-close sequence this PR handles. I did not establish a live systemd/launchd zombie reproduction in this read-only review.

## Solution Assessment

Unclear until proof and policy review are supplied. The code shape is narrow and the double-count guard matches the callback path, but the 3-in-60s process-exit policy is user-visible and should be documented and validated in a real supervised setup.
