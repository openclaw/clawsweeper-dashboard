## Best Possible Solution

Retire stale aborted runs through the same registry finalization/pruning path in both restore and scheduled recovery, add focused scheduled-recovery coverage, and record maintainer acceptance of the stale session cutoff before merge.

## Reproduction Assessment

Yes, source-reproducible: on the PR branch, a stale active run with `abortedLastRun: true` enters the new `isStaleUnendedSubagentRun` branch and is counted only as skipped, while the scheduler finalizes only `failedRuns`. I did not run tests because this was a read-only review.

## Solution Assessment

No. Removing the restore-time `abortedLastRun` exemption is a plausible part of the fix, but the scheduled orphan-recovery path should retire the stale run instead of only skipping it.
