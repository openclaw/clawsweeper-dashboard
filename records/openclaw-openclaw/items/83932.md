## Summary

Keep open: current main has a source-visible gap for the preflight compaction path that injects the post-compaction refresh prompt. It increments the session compaction count, but it does not route through the same user-notice or normal compaction event/log path used by in-turn Codex `contextCompaction` events.

## What This Changes

Request Codex-runtime compactions that trigger a post-compaction refresh to expose the same user-visible, `/status`, or logged observability as Pi/embedded compactions.

## Best Possible Solution

Route preflight/Codex refresh compactions through the same canonical compaction event, notice, log, and idempotent count accounting path used by in-turn compactions, honoring the existing `notifyUser` setting without adding a new config knob.

## Reproduction Assessment

Yes for the source-level path: the only code path that appends the reported post-compaction refresh prompt performs preflight compaction and count updates without reaching the existing event-driven user notice handler. I did not replay a live Discord/Codex compaction session.

## Solution Assessment

Yes: using the existing compaction event/notice/accounting contract for the preflight refresh path is narrower than adding another notification mode or a Codex-only special case.
