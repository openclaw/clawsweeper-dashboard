## Summary

Keep open. Current main has Codex app-server `contextCompaction` events flowing into compaction callbacks and session-count accounting, but the Codex path still does not emit the Pi-style gateway success log, and visible chat notices remain explicitly gated by `notifyUser`, so the reported observability parity gap is only partially covered.

## What This Changes

Request Codex runtime compactions to surface the same user-visible, `/status`, or gateway-log observability as Pi/embedded compactions after post-compaction refresh.

## Best Possible Solution

Add narrow Codex app-server compaction observability parity: log completed `contextCompaction` events and verify `/status` count accounting, while keeping Discord notices behind the existing `notifyUser` or hook-message paths.

## Reproduction Assessment

No high-confidence live repro was run. Source inspection does reproduce the narrow log-parity gap: Codex `contextCompaction` events feed counts and callbacks, but the Codex projection path lacks the Pi-style gateway success log while chat notices are opt-in.

## Solution Assessment

Yes, if scoped to non-chat observability parity. The maintainable fix is to add Codex compaction logging and regression coverage for count propagation, not to make user-visible Discord notices unconditional.
