## Summary

Keep open. Current main still carries node command declarations as string allowlists and validates only command membership/policy before forwarding untyped params/results, so the requested per-command input/output schema mechanism remains missing. Related #8287 is adjacent node-tool registration work and remains open; it does not supersede this narrower gateway command-schema protocol gap.

## What This Changes

Add a mechanism for nodes to declare per-command input/output schemas so the gateway can validate invocations, route safely, and expose discoverable APIs.

## Best Possible Solution

Define a backward-compatible gateway protocol extension that preserves `commands: string[]`, adds optional per-command descriptor/schema metadata, persists and exposes it through pairing plus `node.list`/`node.describe`, updates generated clients and node-host/plugin runtime APIs, and specifies exactly when `node.invoke` validates input and output payloads.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature/protocol gap. We do have a high-confidence verification path by inspecting current main: connect, pairing, catalog, docs, and invoke code all model commands as strings with untyped params/results.

## Solution Assessment

Unclear for the literal proposed `node.userdefine` or `camera.userdefine` shape. The maintainable solution is an additive command descriptor/schema contract aligned with the existing TypeBox protocol and current command allowlist semantics.
