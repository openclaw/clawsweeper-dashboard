## Summary

Keep open. Current main still models node command declarations as string allowlists and forwards `node.invoke` params/results without per-command input or output schemas; related #8287 is adjacent dynamic tool-registration work, but it remains open and does not supersede this narrower gateway command-schema protocol gap.

## What This Changes

Add a gateway/node protocol mechanism for connected nodes to declare per-command input and output schemas for validation, routing, and API exposure.

## Best Possible Solution

Define a backward-compatible command descriptor contract that preserves `commands: string[]`, adds optional per-command schema metadata, carries it through pairing and node discovery, updates generated clients/docs, and specifies when `node.invoke` validates input and output payloads.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature/protocol gap. Source inspection gives a high-confidence verification path: handshake, pairing, catalog, docs, and invoke code all still model commands as strings with untyped params/results.

## Solution Assessment

Unclear for the literal proposed `node.userdefine` or `camera.userdefine` shape. The maintainable solution is an additive TypeBox-backed command descriptor/schema contract aligned with existing allowlist and pairing semantics.
