## Summary

Keep open. Current main has fixed important exec-event cooldown/payload and lightweight-bootstrap pieces, but it still does not satisfy the central durable heartbeat receipt/status contract: `last-heartbeat` and status read an in-process singleton, while the protocol docs still describe that RPC as persisted, and exec-event heartbeat runs can still emit onto the same last-heartbeat surface.

## What This Changes

The issue asks for reliable isolated heartbeat cadence, durable actual-heartbeat receipts/status, effective lightweight context, and clarified `memory/heartbeat-state.json` ownership.

## Best Possible Solution

Define a runtime-owned durable receipt/status contract for actual scheduled/manual heartbeat runs, make CLI/status read or clearly label that source, keep exec-event wake handling separate from cadence proof, and update docs to distinguish runtime receipts from agent-managed `memory/heartbeat-state.json` bookkeeping.

## Reproduction Assessment

Yes for the remaining source-level problem: `last-heartbeat` traces to an in-memory singleton, and status/CLI read that same RPC. The live cadence and token-cost observations are corroborated in comments but were not live-reproduced during this read-only review.

## Solution Assessment

No, the current implementation is not the complete best solution because it still lacks a durable scheduled/manual heartbeat receipt source. A CLI-only filter would be too narrow; the maintainable path is a canonical receipt/status contract with docs and status semantics updated together.
