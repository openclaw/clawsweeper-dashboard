## Summary

Keep open. Current main has partial fixes for exec-event payload handling, cooldown gating, and lightweight bootstrap filtering, but the heartbeat status contract is still unresolved: `last-heartbeat`, CLI, and deep status read a process-local singleton while the protocol docs describe a persisted heartbeat event.

## What This Changes

The issue asks OpenClaw to make isolated heartbeat cadence, `last-heartbeat` reporting, `lightContext` cost, and heartbeat state or receipt behavior reliable and auditable.

## Best Possible Solution

Define a canonical runtime-owned heartbeat receipt/status contract for scheduled and manual heartbeat runs, wire CLI/status/docs to that source, and document `memory/heartbeat-state.json` separately as agent-managed workspace bookkeeping.

## Reproduction Assessment

Yes for the remaining source-level status problem: `getLastHeartbeatEvent()` returns process-local state, and both RPC/CLI/status read that surface. I did not live-reproduce the reported cadence stall or token-cost behavior against current main.

## Solution Assessment

No, a CLI-only filter or another narrow exec-event patch would be too small because it would not create a durable receipt source or resolve the docs/status contract. The maintainable fix is a shared heartbeat receipt/status API with storage, retention, and docs semantics decided together.
