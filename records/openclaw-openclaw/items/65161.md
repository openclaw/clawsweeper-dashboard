## Summary

Keep open. Current main contains partial fixes for exec-event prompt handling, event consumption, and cooldown gating, but the remaining heartbeat status contract is still unresolved: `last-heartbeat`, CLI, and deep status still read a process-local singleton while the gateway protocol docs still describe a persisted heartbeat event.

## What This Changes

The issue requests reliable isolated heartbeat cadence, durable and accurate heartbeat visibility, effective lightweight heartbeat context, and clarification or removal of the `memory/heartbeat-state.json` runtime expectation.

## Best Possible Solution

Define a canonical runtime-owned heartbeat receipt/status contract for actual scheduled and manual heartbeat runs, make CLI/status read or clearly label that source, and update docs to distinguish runtime receipts from agent-managed workspace bookkeeping.

## Reproduction Assessment

Yes for the remaining source-level status problem: `last-heartbeat` still traces to an in-memory singleton while docs describe a persisted event. The original live cadence-stall and token-cost symptoms are corroborated in comments but were not live-reproduced in this read-only review.

## Solution Assessment

No, the current implementation is only a partial solution because it still lacks a durable or clearly documented scheduled/manual heartbeat status source. A CLI-only filter would be too narrow; the maintainable fix is a shared receipt/status contract plus aligned docs and status semantics.
