## Summary

Keep open. Current main has partial heartbeat fixes, but the central status/receipt contract is still unresolved: `last-heartbeat`, CLI status, and protocol wording do not agree on whether the heartbeat event is durable.

## What This Changes

The issue reports unreliable isolated heartbeat cadence, exec-event contamination in heartbeat status, ineffective lightweight context behavior, and an unclear heartbeat state/receipt contract.

## Best Possible Solution

Define one canonical runtime-owned heartbeat receipt/status contract for actual scheduled and manual heartbeat runs, make CLI/status read or explicitly label that source, and keep `memory/heartbeat-state.json` documented as agent-managed workspace bookkeeping only.

## Reproduction Assessment

Yes for the remaining source-level problem: current main still routes `last-heartbeat` and status through an in-memory singleton while protocol docs describe a persisted heartbeat event. The original live cadence and token-cost symptoms remain corroborated by comments but were not live-reproduced in this read-only review.

## Solution Assessment

No, current main is only a partial solution. A durable receipt/status contract with aligned docs and CLI semantics is the maintainable endpoint; a CLI-only filter would leave the underlying status source ambiguous.
