## Summary

Keep open. Current main still has the central heartbeat visibility gap: the CLI/status path reads `last-heartbeat` from an in-process singleton, docs still describe that RPC as persisted, and exec-event wakes can still share the same last-heartbeat surface. Narrower exec-event routing/payload fixes and lightweight bootstrap support exist, but they do not satisfy the requested durable receipt and cadence-proof behavior.

## What This Changes

The issue requests trustworthy isolated-heartbeat cadence/status behavior, including durable actual-heartbeat receipts, separation from exec-event wakes, effective lightweight context, and clear `memory/heartbeat-state.json` ownership.

## Best Possible Solution

Define a runtime-owned heartbeat receipt contract for actual scheduled/manual heartbeat runs, persist it durably, make `system heartbeat last` and status read or clearly label that source, keep exec-event wakeups separate from cadence proof, and update docs to distinguish runtime receipts from agent-managed `memory/heartbeat-state.json` bookkeeping.

## Reproduction Assessment

Yes for the remaining source-level problem: `openclaw system heartbeat last` can be traced to `last-heartbeat` -> `getLastHeartbeatEvent()` -> process-global memory, and exec-event wakes can emit to that same surface. The live cadence-stall and token-cost symptoms are plausible and corroborated in comments, but were not live-reproduced in this read-only review.

## Solution Assessment

No, the current implementation is not the best complete solution because it still lacks a durable scheduled/manual heartbeat receipt source. A CLI-only filter would be too narrow; the maintainable fix is a canonical receipt/status contract with docs and status semantics updated together.
