## Summary

Keep open. Current main has shipped partial fixes for exec-event cooldown and task-state handling, but the remaining status/receipt contract is still unresolved: `last-heartbeat` is process-local while the protocol docs say persisted, and recent comments still separate `sessions.json` task state from a trustworthy heartbeat receipt surface.

## What This Changes

The issue reports isolated heartbeat cadence stalls, misleading `last-heartbeat` status, unexpectedly heavy `lightContext` runs, and confusion over whether `memory/heartbeat-state.json` is runtime-owned.

## Best Possible Solution

Define a runtime-owned heartbeat receipt/status source for actual scheduled/manual runs, make CLI/status/protocol read or clearly label that source, and keep docs explicit that `memory/heartbeat-state.json` is workspace bookkeeping while `sessions.json` holds task timestamps.

## Reproduction Assessment

Yes for the remaining source-level status problem: current main exposes `last-heartbeat` through an in-process singleton while the protocol says persisted. The broader cadence stall and token-cost claims are supported by issue comments but were not live-reproduced here.

## Solution Assessment

No for closing as implemented: current main has partial fixes, but the status/receipt contract remains inconsistent. A CLI-only filter would be too narrow; the maintainable fix is to align runtime receipts, status surfaces, and docs together.
