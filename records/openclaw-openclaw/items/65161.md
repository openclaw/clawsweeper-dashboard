## Summary

Keep open. Current main has partial fixes for exec-event prompting, cooldown, and lightweight bootstrap behavior, but the remaining heartbeat status contract is still not satisfied: `last-heartbeat`, CLI, and status use an in-process singleton while the protocol docs still describe a persisted heartbeat event, and exec-event wakes can still share that visibility surface.

## What This Changes

The issue requests reliable isolated heartbeat cadence, durable scheduled/manual heartbeat receipts, clearer separation from exec-event wakes, effective lightweight context, and either runtime ownership or clearer documentation for `memory/heartbeat-state.json`.

## Best Possible Solution

Define a canonical runtime-owned heartbeat receipt/status contract for actual scheduled/manual heartbeat runs, make CLI/status read or clearly label that source, and update docs to distinguish runtime receipts from agent-managed workspace bookkeeping.

## Reproduction Assessment

Yes for the remaining source-level status problem: `last-heartbeat` traces to an in-memory singleton and CLI/status read that RPC. The live cadence-stall and token-cost observations are corroborated in comments but were not live-reproduced during this read-only pass.

## Solution Assessment

No, current main is only a partial solution because it still lacks a durable scheduled/manual heartbeat receipt source. A CLI-only filter would be too narrow; the maintainable fix is a canonical receipt/status contract with docs and status semantics updated together.
