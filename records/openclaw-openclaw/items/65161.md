## Summary

Keep open. Current main has partial fixes for exec-event payload handling, lightweight bootstrap filtering, and event-wake cooldowns, but the core receipt/status problem remains: `last-heartbeat` still returns a process-memory singleton while the gateway protocol describes a persisted event, and `memory/heartbeat-state.json` is still only documented as workspace bookkeeping with no runtime writer.

## What This Changes

The issue asks for reliable isolated heartbeat cadence evidence, non-misleading heartbeat status, effective lightweight context behavior, and a clear runtime-vs-workspace heartbeat state contract.

## Best Possible Solution

Define a runtime-owned durable receipt/status contract for actual scheduled/manual heartbeat runs, wire CLI/status/protocol to that source, keep exec-event wake results clearly separate from cadence proof, and clarify whether `memory/heartbeat-state.json` is agent bookkeeping or runtime state.

## Reproduction Assessment

Yes for the remaining source-level issue: current main still routes `last-heartbeat` through an in-memory singleton while docs call it persisted, and source search finds no runtime `memory/heartbeat-state.json` writer. The live cadence stall and token-cost observations remain corroborated by reports but were not reproduced in this read-only pass.

## Solution Assessment

No, the current partial fixes are not the complete solution because they do not create a canonical durable heartbeat receipt/status source. A CLI-only filter would be too narrow; the durable runtime contract and docs/status semantics should be resolved together.
