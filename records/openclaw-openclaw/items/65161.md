## Summary

Keep open: current main and the latest release have fixed some related exec-event prompt and cooldown subpaths, but the central heartbeat observability contract is still not satisfied because `last-heartbeat` remains process-local while docs describe a persisted event, and the heartbeat failure path still leaves inspected events unconsumed.

## What This Changes

The issue asks OpenClaw to make isolated heartbeat cadence and receipts trustworthy, separate exec-event wakes from canonical heartbeat runs, prove lightweight mode is cheap, and clarify the heartbeat-state bookkeeping contract.

## Best Possible Solution

Define the canonical heartbeat receipt/status surface in the existing session or state DB owner, fix failure-path event consumption to match rendered prompt content, and keep already-fixed exec prompt/cooldown subpaths linked to their narrower issues.

## Reproduction Assessment

Yes, partially from current source: `last-heartbeat` still returns a process-local singleton despite docs saying persisted, and the failure catch does not consume selected inspected events. No live current-main run proved the cadence or token-cost claims.

## Solution Assessment

Unclear: a narrow failure-path event-consumption repair may be straightforward, but the durable receipt/status contract needs maintainer direction on storage and docs semantics.
