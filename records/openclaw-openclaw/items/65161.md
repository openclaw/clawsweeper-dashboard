## Summary

Keep open. Current main still routes `system heartbeat last` and status through the in-process last-heartbeat singleton while docs describe a persisted event, so the remaining visibility/receipt-contract part of the report is not implemented. The narrower exec-event prompt-body/replay issues appear improved in current code and related issues, but they do not resolve the durable scheduled/manual heartbeat receipt gap.

## What This Changes

The issue requests reliable isolated-heartbeat cadence evidence, separation of exec-event wakeups from canonical heartbeat status, effective lightweight context behavior, and clarity around `memory/heartbeat-state.json`.

## Best Possible Solution

Define a runtime-owned heartbeat receipt contract for actual scheduled/manual heartbeat runs, persist it durably, make CLI/status read or clearly label that source, keep exec-event wakeups separate from cadence proof, and update docs to distinguish runtime receipts from agent-managed `memory/heartbeat-state.json` bookkeeping.

## Reproduction Assessment

Yes for the remaining source-level visibility problem: `system heartbeat last` traces to `last-heartbeat` -> `getLastHeartbeatEvent()` -> process-global `state.lastHeartbeat`, and exec-event wakes can still write that surface. The full cadence-stall and token-cost symptoms remain live-observation reports rather than reproduced in this read-only pass.

## Solution Assessment

No, the current implementation is not the best complete solution because it still lacks a durable scheduled/manual heartbeat receipt source. More CLI filtering would be weaker than designing the receipt/status contract and then routing CLI/status/docs through it.
