## Summary

Keep open. Current main has partial fixes for exec-event prompt payloads, event consumption, and wake cooldowns, but the remaining heartbeat status/receipt contract is still unresolved: `last-heartbeat`, CLI status, and deep status still read a process-local singleton while docs describe a persisted heartbeat event, and there is still no canonical durable scheduled/manual receipt surface.

## What This Changes

The issue asks OpenClaw to make isolated heartbeat cadence, exec-event reporting, lightContext cost, and heartbeat state/receipt tracking reliable.

## Best Possible Solution

Define one canonical runtime-owned heartbeat receipt/status contract for actual scheduled and manual heartbeat runs, make CLI/status read or clearly label that source, and keep workspace `memory/heartbeat-state.json` documented as agent-managed bookkeeping only.

## Reproduction Assessment

Yes for the remaining source-level status problem: current main still routes `last-heartbeat` through an in-memory singleton while protocol docs describe a persisted event. The original live cadence-stall and token-cost symptoms are corroborated by comments but were not live-reproduced here.

## Solution Assessment

No, current main is only a partial solution. A CLI-only filter would be too narrow; the maintainable fix is a shared receipt/status contract plus aligned docs and status semantics.
