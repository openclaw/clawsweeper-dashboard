## Label Justifications

- `P1`: The issue describes a deterministic production Teams channel break where inbound messages reach the agent but outbound replies fail for managed-identity deployments.
- `impact:message-loss`: The affected path can drop Teams replies after model work completes, leaving the Teams user with no delivered response.
- `impact:auth-provider`: The root cause is token acquisition and bot identity selection in Teams federated managed-identity auth.

## Summary

Keep open: current main and the latest shipped release still use the raw Teams managed-identity credential path, and an open closing PR already owns the concrete fix. Closing this before https://github.com/openclaw/openclaw/pull/85169 lands would hide a source-backed Teams auth/provider bug.

## What This Changes

The issue reports that Microsoft Teams federated managed-identity auth ignores the bot app FIC exchange and sends Bot Framework a token for the managed identity app, causing outbound replies to fail.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85169 or an equivalent patch that uses `ClientAssertionCredential` for the bot-app FIC exchange and safely logs delivery failure status/cause, then close this after merge.

## Reproduction Assessment

Yes, with high-confidence source reproduction. The reporter provided Azure VM/FIC steps and logs, and current main plus Azure Identity source show the managed-identity path requests the Bot Framework token directly instead of performing the documented bot-app assertion exchange.

## Solution Assessment

Yes for the reported bug. The narrow maintainable fix is to keep the existing `authType: "federated"` plus `useManagedIdentity` contract and exchange the managed-identity assertion through `ClientAssertionCredential`, as the open fixing PR does.
