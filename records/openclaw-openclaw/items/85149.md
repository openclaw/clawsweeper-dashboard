## Label Justifications

- `P1`: The issue describes a deterministic production Teams channel auth break where inbound messages can reach the agent but outbound replies fail for managed-identity deployments.
- `impact:message-loss`: The failure mode leaves Teams replies undelivered after model work completes.
- `impact:auth-provider`: The root cause is Teams federated managed-identity token acquisition and bot app identity selection.

## Summary

Keep open: current main and the latest release still use the raw managed-identity token path for Teams federated auth, and the issue is already paired with an open closing PR that has not merged. The later maintainer comment also explicitly says to leave this open while the broader Teams SDK migration is evaluated.

## What This Changes

The issue requests that Microsoft Teams federated managed-identity auth exchange the managed-identity assertion through the bot app FIC instead of sending a raw managed-identity Bot Framework token, and asks for safer delivery-failure diagnostics.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85169 or an equivalent narrow fix; if https://github.com/openclaw/openclaw/pull/76262 lands first and removes the affected Bot Framework path, close this issue with that merged commit as the implementation proof.

## Reproduction Assessment

Yes for source-level reproduction: current main passes the Teams SDK requested scope directly to `ManagedIdentityCredential`, while the documented setup depends on a federated identity credential with `api://AzureADTokenExchange`. I did not run a live Azure/Teams environment in this read-only review.

## Solution Assessment

Yes: the narrow `ClientAssertionCredential(tenantId, appId, assertion)` path in the linked fix PR matches the existing documented managed-identity contract without adding a new config mode. The issue should remain open until that PR or the broader Teams SDK migration actually lands.
