## Label Justifications

- `P1`: The report describes a deterministic production Microsoft Teams channel failure where inbound messages reach the agent but outbound replies fail for managed-identity deployments.
- `impact:message-loss`: The affected path drops Teams replies after model work completes, leaving the Teams user with no delivered response.
- `impact:auth-provider`: The root cause is token acquisition and bot identity selection for Teams federated managed-identity authentication.

## Summary

Keep open: current main and the latest shipped release still use the raw managed-identity credential for Teams Bot Framework tokens, and the issue is now paired with an open fix PR that uses closing syntax. The issue should close only after that PR merges or maintainers choose a different fix path.

## What This Changes

The issue reports that Microsoft Teams federated managed-identity auth bypasses the bot-app FIC exchange, causing outbound Bot Framework replies to fail with 401 and sparse logging.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85169, or replace it with the same narrow token-exchange fix, then close this issue after the merged commit ships or is confirmed on main.

## Reproduction Assessment

Yes, with high confidence at the source level: current main requests the Teams SDK scope from `ManagedIdentityCredential` directly, while the documented setup depends on a bot-app FIC exchange. I did not rerun the live Azure/Teams scenario, but the reporter supplied concrete deployment steps, decoded-token evidence, and after-patch success logs.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve the existing managed-identity config contract and wrap the managed-identity assertion in `ClientAssertionCredential(tenantId, appId, getAssertion)`, with focused tests and safe delivery-failure metadata.
