## Label Justifications

- `P1`: The report describes a deterministic production Teams channel workflow break where inbound messages run the agent but outbound replies fail for managed-identity deployments.
- `impact:message-loss`: The affected path drops Teams replies after model work completes, leaving users with no delivered response.

## Summary

Keep open: current main and the latest shipped tag still route Teams federated managed-identity auth through a raw `ManagedIdentityCredential`, while the documented setup requires a federated credential on the bot app. The root cause is narrow enough for a focused fix PR, with live Azure/Teams proof still needed before landing.

## What This Changes

The issue reports that Microsoft Teams federated managed identity auth obtains a raw managed-identity token instead of exchanging it through the bot app federated credential, causing outbound Bot Framework replies to fail.

## Best Possible Solution

Keep the existing federated managed-identity config contract, but implement the token exchange with `ClientAssertionCredential`, add focused regression coverage, and include safe delivery-failure status logging.

## Reproduction Assessment

Yes. The reporter provided concrete Azure VM/FIC steps and logs, and current main/source plus Azure Identity contracts show the managed-identity path requests the Bot Framework token directly instead of performing the documented federated exchange.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing managed-identity federated mode exchange an MI assertion through `ClientAssertionCredential`, rather than adding a new config mode or changing unrelated Teams auth behavior.

## Repair Work Prompt

Fix the Microsoft Teams federated managed-identity auth path reported in https://github.com/openclaw/openclaw/issues/85149. In `extensions/msteams/src/sdk.ts`, keep the existing `authType: "federated"` + `useManagedIdentity` contract but change `createManagedIdentityApp` so the raw managed identity is used only to obtain an assertion for `api://AzureADTokenExchange`, then wrap that assertion callback in `@azure/identity` `ClientAssertionCredential(creds.tenantId, creds.appId, ...)` for the Teams SDK bot token provider. Add the `ClientAssertionCredential` constructor to the local Azure identity module type and preserve system-assigned and user-assigned MI behavior. Add focused tests in `extensions/msteams/src/sdk.test.ts` that prove the MI credential requests `api://AzureADTokenExchange`, the Teams token provider uses the client assertion credential for `https://api.botframework.com/.default`, and certificate/client-secret flows are unchanged. Also improve `extensions/msteams/src/reply-dispatcher.ts` warning details so delivery failures include safe status/cause context without logging tokens or private identifiers. Do not add a new config option or move ownership into core. Add a changelog entry because this fixes user-visible Teams reply delivery.
