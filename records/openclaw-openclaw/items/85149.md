## Label Justifications

- `P1`: The report describes a deterministic production Teams channel workflow break where inbound messages can run the agent but outbound replies fail for managed-identity deployments.
- `impact:message-loss`: The affected path can drop Teams replies after model work completes, leaving users with no delivered response.
- `impact:auth-provider`: The root cause is token acquisition and provider identity selection for Teams federated managed-identity auth.

## Summary

Keep open: current main and the latest shipped tag still route Teams federated managed-identity auth through raw `ManagedIdentityCredential`, while the documented setup requires the managed identity to be linked to the bot app through a federated credential. The source path, tests, Azure Identity source, and closed prior fix attempt all support a narrow repair PR rather than cleanup closure.

## What This Changes

The issue asks to make Microsoft Teams federated managed-identity auth exchange a managed-identity assertion for a bot-app token and to expose safer delivery-failure details when outbound Bot Framework sends fail.

## Best Possible Solution

Keep the existing federated managed-identity config contract, implement the MI assertion exchange through `ClientAssertionCredential`, add focused regression coverage, and include safe warning-level delivery failure details.

## Reproduction Assessment

Yes, at source level. The reporter supplied concrete Azure VM/FIC steps and current main shows the managed-identity path requests the Bot Framework token directly from `ManagedIdentityCredential` instead of exchanging an MI assertion for a bot-app token.

## Solution Assessment

Yes. The narrow maintainable fix is to keep the existing config surface and make only the managed-identity federated mode use `ClientAssertionCredential`, plus add regression tests and safe failure logging.

Add safe warning-level delivery failure details in `extensions/msteams/src/reply-dispatcher.ts` and tests in `extensions/msteams/src/reply-dispatcher.test.ts` if the logging ask is included; never log bearer tokens, client assertions, full JWTs, or sensitive tenant/user identifiers.
