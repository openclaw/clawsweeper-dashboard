## Label Justifications

- `P1`: The report describes a deterministic production Teams channel break where inbound messages reach the agent but outbound replies fail for managed-identity deployments.
- `impact:message-loss`: The affected path drops Teams replies after model work completes, leaving the Teams user with no delivered response.
- `impact:auth-provider`: The root cause is token acquisition and bot identity selection for Teams federated managed-identity auth.

## Summary

Keep open: current main and the latest shipped tag still send the Teams SDK a raw managed-identity token for the Bot Framework scope, while the documented managed-identity setup depends on a federated credential on the bot app. The issue is source-backed, narrow, and suitable for a focused fix PR with live Azure/Teams proof before landing.

## What This Changes

Fix Teams federated managed-identity auth so OpenClaw exchanges a managed-identity assertion for a bot-app token and surfaces useful outbound delivery failure details.

## Best Possible Solution

Keep the existing `authType: "federated"` plus `useManagedIdentity` contract, implement the managed-identity assertion exchange with `ClientAssertionCredential`, add focused regression tests, and safely include delivery failure status/cause in the Teams warning path.

## Reproduction Assessment

Yes for source-level reproduction. The reporter provided concrete Azure VM/FIC steps and logs, and current main plus Azure Identity source show the managed-identity path requests the Bot Framework token directly instead of performing the documented federated exchange.

## Solution Assessment

Yes. The narrow maintainable fix is to wrap the managed-identity assertion in `ClientAssertionCredential(tenantId, appId, getAssertion)` for the existing managed-identity mode, not add a parallel config mode or change unrelated Teams auth behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/85149. In `extensions/msteams/src/sdk.ts`, keep the existing federated managed-identity config contract but stop returning a raw `ManagedIdentityCredential` token for the Bot Framework scope. Use `@azure/identity` `ClientAssertionCredential` with `creds.tenantId` and `creds.appId`; supply the assertion by asking the managed identity credential for `api://AzureADTokenExchange`, then let `ClientAssertionCredential.getToken(scope)` acquire the bot-app token. Preserve client-secret and certificate auth behavior, keep credential caching, and add focused tests proving the managed-identity path constructs/uses `ClientAssertionCredential`, requests the MI assertion with `api://AzureADTokenExchange`, supports system- and user-assigned MI, and throws a useful error when the assertion is missing. Also improve the Teams delivery warning in `extensions/msteams/src/reply-dispatcher.ts` to include safe status/cause details for failed sends without logging bearer tokens, app secrets, or sensitive tenant details. Add a user-facing `CHANGELOG.md` fix entry. Do not introduce a new config option or switch unrelated Teams auth paths to `DefaultAzureCredential` unless the existing contract cannot be preserved.
