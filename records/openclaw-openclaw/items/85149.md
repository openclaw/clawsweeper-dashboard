## Label Justifications

- `P1`: The report describes a deterministic production Teams channel break where inbound work completes but outbound replies fail for documented managed-identity deployments.
- `impact:message-loss`: The affected path can drop Teams replies after model work completes, leaving the Teams user with no delivered response.
- `impact:auth-provider`: The root cause is Teams bot token acquisition and identity selection for federated managed-identity auth.

## Summary

Keep open: current main and v2026.5.20 still use a raw `ManagedIdentityCredential` for the Teams Bot Framework token path, while the documented setup depends on a bot-app federated identity exchange. A narrow fix PR was closed unmerged, and the broader Teams SDK migration remains open, so no shipped implementation resolves this yet.

## What This Changes

The issue requests that Microsoft Teams federated managed-identity auth exchange the MI assertion through the configured bot app FIC and expose safe outbound delivery failure details.

## Best Possible Solution

Land a focused `ClientAssertionCredential` exchange plus safe warning metadata for the existing Teams managed-identity contract, unless the broader Teams SDK migration ships first with equivalent live-proven behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main requests the Bot Framework scope from `ManagedIdentityCredential` directly while the documented FIC setup requires an `api://AzureADTokenExchange` assertion for the bot app. I did not run the live Azure VM/Teams environment in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to wrap the managed-identity assertion in `ClientAssertionCredential(tenantId, appId, getAssertion)` for the existing managed-identity mode. If https://github.com/openclaw/openclaw/pull/76262 lands first and proves equivalent behavior, that shipped path can become the implementation evidence instead.
