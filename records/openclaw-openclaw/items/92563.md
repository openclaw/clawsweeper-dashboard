## Summary

Keep this issue open: current main and the latest release still read transcript-only assistant rows into session-memory output, and several open PRs use closing syntax for this exact bug. The right cleanup is to pick and land one canonical fix rather than close the issue before merge.

## What This Changes

The issue reports duplicate assistant lines in session-memory files when thinking-enabled sessions contain both raw assistant output and cleaned transcript-only assistant rows.

## Best Possible Solution

Land one canonical consumer-layer fix that filters transcript-only OpenClaw assistant rows from session-memory extraction, then let that merged PR close this issue.

## Reproduction Assessment

Yes, from source inspection: a JSONL transcript containing a normal assistant text block followed by an OpenClaw `delivery-mirror` or `gateway-injected` assistant row with text will produce two `assistant:` lines. I did not execute a live provider repro to keep the checkout read-only.

## Solution Assessment

Yes on direction: filtering transcript-only assistant rows at the session-memory consumer boundary matches existing replay/history predicates and avoids changing storage. The storage-layer deletion suggested in the issue would be broader than necessary because other transcript consumers rely on persisted bookkeeping rows.
