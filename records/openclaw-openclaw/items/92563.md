## Summary

Keep open: current session-memory extraction still emits any persisted raw-plus-mirror assistant pair, the latest release has the same reader behavior, and three open closing PRs remain candidate fixes rather than merged implementation.

## What This Changes

The issue reports duplicate assistant lines in session-memory files when thinking-enabled responses are persisted as both raw assistant content and a cleaned/mirrored assistant row.

## Best Possible Solution

Land one canonical session-memory fix that filters only actual duplicate transcript mirror rows while preserving legitimate channel-final and standalone gateway-injected assistant content, then close this issue after the PR merges.

## Reproduction Assessment

Yes, at source level. A JSONL transcript containing a raw assistant text block followed by a cleaned/mirrored assistant copy is read twice by current getRecentSessionContent because it extracts and pushes text from every assistant row.

## Solution Assessment

No, there is not yet a complete merged solution for this issue. The best fix should be a narrow consumer-layer or lineage-aware filter that removes duplicate mirror rows without dropping legitimate visible assistant content.
