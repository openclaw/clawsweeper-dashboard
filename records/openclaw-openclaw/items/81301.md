## Summary

Keep open: the PR is a plausible focused OpenAI Codex OAuth fix, but it is an external auth-flow PR without after-fix proof from a real remote/SSH-forwarded setup. The diff is mostly sound from source review, with one low-priority overlap edge in the temporary fetch monitor.

## What This Changes

The PR defers the OpenAI Codex manual paste prompt in remote/VPS OAuth flows and adds tests for delayed fallback, callback completion, and refresh-token traffic.

## Best Possible Solution

Land this direction after redacted real remote OAuth proof is posted and the fetch monitor is tightened so overlapping logins cannot leak or suppress the wrong prompt.

## Reproduction Assessment

Source-reproducible: current main immediately creates the remote paste prompt in both the shared OAuth handler and the Codex remote manual handler. I did not live-reproduce the SSH-forwarded callback path.

## Solution Assessment

Mostly yes: deferring only the OpenAI Codex remote prompt keeps the shared handler's immediate default for other providers and matches Pi's callback/manual race. The fetch-monitoring mechanism should be made stack-safe or request-scoped before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.78

Full review comments:

- **[P3] Guard the fetch monitor against overlapping logins:** `src/plugins/provider-openai-codex-oauth.ts:117-120`
  - body: This wrapper is process-global, so two overlapping Codex OAuth logins can observe each other's token-exchange fetches: the inner wrapper calls the previous wrapper, which marks the outer login as having started token exchange and can suppress its manual fallback. If the outer login exits first, its restore is skipped and the wrapper can also remain installed; use a stack-safe/ref-counted monitor or a more local signal, and mirror the fix in the OpenAI plugin runtime file.
