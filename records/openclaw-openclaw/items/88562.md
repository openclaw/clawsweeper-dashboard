## Summary

Keep open. Current main and latest release v2026.6.8 still have source-backed paths that persist env-backed provider auth as `apiKey` strings, flag arbitrary `apiKey` strings in generated `models.json`, and can write provider `apiKey` into generated plugin catalog sidecars, so the central security/audit problem is not implemented on main.

## What This Changes

The issue requests that generated model/provider catalog files stop writing or misclassifying SecretRef-managed provider `apiKey` values as plaintext strings.

## Best Possible Solution

Land a maintainer/security-reviewed fix that covers generated root `models.json`, plugin `catalog.json` sidecars, stale generated files, and secrets-audit classification while preserving provider auth and live discovery.

## Reproduction Assessment

Yes at source level: current main still returns env-backed auth-profile `keyRef` ids as generated provider `apiKey` strings, can persist provider `apiKey` fields into plugin catalog sidecars, and audits unrecognized `apiKey` strings as plaintext. I did not run a live CLI repro because this read-only review cannot write generated agent state.

## Solution Assessment

No for closing as implemented. The requested structured SecretRef object is one possible shape, but the maintainable fix needs a security-approved contract across `models.json`, plugin `catalog.json`, stale sidecars, runtime auth, and audit behavior.
