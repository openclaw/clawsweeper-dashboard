## Summary

Keep open. Current main and the latest release still have source-level evidence for the generated `models.json` env-marker audit problem, and the later comments make this the canonical tracker for generated plugin `catalog.json` `apiKey` persistence too.

## What This Changes

The issue reports that generated per-agent model and plugin catalog files write provider auth markers or keys as plain `apiKey` strings, causing secrets-audit warnings and possible credential exposure.

## Best Possible Solution

Land one security-reviewed fix that covers both root `models.json` and plugin-owned `catalog.json` sidecars, preserves runtime provider auth through the canonical auth store, and keeps secrets-audit warnings for real plaintext residue.

## Reproduction Assessment

Yes at source level. Current main returns env keyRef ids as generated provider `apiKey` strings, and the secrets audit still flags unrecognized `apiKey` strings; I did not run a live CLI reproduction in this read-only pass.

## Solution Assessment

No for closing as implemented. The best fix direction is a maintainer-selected security/compatibility shape that avoids leaking resolved keys while preserving provider/catalog runtime behavior.
