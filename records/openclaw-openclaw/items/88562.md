## Summary

Keep open. Current main and latest release v2026.6.9 still have source-backed paths that return env-backed auth-profile SecretRefs as bare `apiKey` strings, serialize provider maps into generated plugin catalog sidecars, and flag arbitrary generated `apiKey` strings as plaintext, while related fix PRs remain open and unmerged.

## What This Changes

The issue reports that generated per-agent `models.json` and related plugin `catalog.json` snapshots can serialize provider auth as plaintext or bare env-name `apiKey` strings instead of scanner-safe SecretRef or marker data.

## Best Possible Solution

Land one maintainer/security-reviewed fix that covers generated root `models.json`, plugin `catalog.json` sidecars, stale generated files, runtime provider auth, and secrets-audit classification without adding a new config surface.

## Reproduction Assessment

Yes at source level: current main still resolves env-backed auth-profile keyRefs to bare env-var-name `apiKey` markers and serializes provider maps into generated root/plugin catalog outputs. I did not run a mutating CLI repro because this review is read-only.

## Solution Assessment

No for closing as implemented. The requested structured SecretRef object is one possible shape, but the best fix still needs a maintainer-selected security contract across generated catalogs, runtime auth, migration/cleanup, and audit behavior.
