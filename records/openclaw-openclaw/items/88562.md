## Summary

Keep open. Current main still has source-level evidence for the custom-provider env marker audit problem, and the later comments made this issue the canonical tracker for generated plugin catalog `apiKey` persistence as well.

## What This Changes

The issue asks OpenClaw to stop writing bare provider `apiKey` strings into generated `models.json` and related catalog snapshots when the credential source is a SecretRef/keyRef.

## Best Possible Solution

Land a security-reviewed fix that covers both generated `models.json` and plugin `catalog.json` sidecars, with regression coverage for custom env-backed keyRefs, legacy plaintext snapshots, and audit behavior.

## Reproduction Assessment

Yes at source level. Current main returns env keyRef ids as generated provider apiKey strings, and the secrets audit still flags unrecognized apiKey strings; I did not run a live CLI reproduction in this read-only pass.

## Solution Assessment

No for closing as implemented. The best fix direction is not merely accepting structured SecretRef objects in generated files; maintainers need a security-reviewed shape that avoids leaking resolved keys while preserving provider/catalog compatibility.
