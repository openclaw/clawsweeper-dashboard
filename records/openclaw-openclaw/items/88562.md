## Summary

This issue is still necessary: current main directly serializes plugin-owned provider configuration into generated `catalog.json` files, and recent v2026.6.5 production evidence reports that this includes real plaintext API keys. The linked closing PR is open and viable, so the issue should remain open while that PR is expanded or paired with canonical follow-up work for plugin catalogs.

- `steipete` — high confidence; introduced the recent plugin model-catalog state refactor and frequently maintains the model-config path.

## What This Changes

Prevent generated per-agent model and plugin catalog files from persisting provider API-key material as plaintext strings.

## Best Possible Solution

Use one credential-safe serialization boundary for root and plugin-owned generated catalogs, resolve SecretRefs only in memory at request time, scan every generated catalog location, and define cleanup or regeneration behavior for existing plaintext snapshots.

## Reproduction Assessment

Yes. Current main shows plugin-owned provider objects being serialized directly into `catalog.json`, and the discussion provides recent v2026.6.5 production artifacts containing real plaintext `apiKey` values.

## Solution Assessment

No, the current linked PR is not yet the complete solution because its diff does not cover plugin catalog serialization or equivalent audit coverage; the same invariant must protect every generated catalog path.
