## Summary

Keep open. Current main and v2026.6.8 still have source-backed paths that serialize env-backed provider markers as `apiKey` strings and can persist provider `apiKey` fields into generated plugin catalog sidecars, so the central security/audit problem is not implemented on main.

## What This Changes

The issue reports generated per-agent `models.json` serializing env-backed provider auth `keyRef`s as bare `apiKey` strings, with follow-up scope covering generated plugin `catalog.json` sidecars.

## Best Possible Solution

Land a security-reviewed fix that prevents generated `models.json` and plugin `catalog.json` from persisting resolved provider keys while preserving runtime discovery, provider auth, and accurate secrets-audit results.

## Reproduction Assessment

Yes at source level: current main returns env-backed auth-profile `keyRef` ids as generated provider `apiKey` strings, and the audit still flags unrecognized `apiKey` strings. I did not run a live CLI reproduction because this read-only review cannot write generated agent state.

## Solution Assessment

No for closing as implemented. The requested structured SecretRef object is one possible shape, but the best fix needs a maintainer-approved security contract across `models.json`, plugin `catalog.json`, stale sidecars, runtime auth, and audit behavior.
