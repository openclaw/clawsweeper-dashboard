- `impact:auth-provider`: The affected path is model provider auth flowing from auth profiles into generated `models.json` entries.
- `impact:security`: The user-visible failure is in `openclaw security audit --deep` credential scanning for SecretRef-managed provider credentials.

## Summary

Keep open: current main and v2026.5.28 still generate bare env-var `apiKey` markers from env-backed auth-profile `keyRef` entries, while the deep audit only exempts known marker strings, so custom provider env names can still produce the reported false-positive. Persisting raw SecretRef objects is not the best fix because `models.json` SecretRef objects are currently treated as unresolved generated state.

## What This Changes

Fix generated per-agent `models.json` so env-backed auth-profile keyRefs are persisted and audited as non-secret SecretRef markers instead of producing `PLAINTEXT_FOUND` warnings.

## Best Possible Solution

Keep this open and align the generated marker contract with the audit/runtime contract so arbitrary env-backed auth-profile keyRefs remain non-secret, do not trigger false plaintext findings, and still resolve for custom providers.

## Reproduction Assessment

Yes, source-reproducible: current code returns the env `keyRef` id as the generated provider `apiKey`, and the audit flags unknown string apiKey values as plaintext. I did not run the CLI because this review is read-only.

## Solution Assessment

No, not if the fix is to write raw SecretRef objects into generated `models.json`, because current audit code treats those objects as unresolved generated state. The safer fix is to make the generated marker, audit exemption, and runtime resolution contract handle arbitrary env-backed keyRefs consistently.

## Repair Work Prompt

Current generated `models.json` writes env-backed auth-profile `keyRef` values to `providers.<id>.apiKey` as a bare env-var string, but `openclaw security audit --deep` treats arbitrary custom env-var strings as plaintext unless they are known non-secret markers.
