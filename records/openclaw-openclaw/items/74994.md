## Summary

Keep open. The PR improves several early deleted-cwd callers, but it still does not cover the original `openclaw tui` path from related #73676: current startup still reaches raw `process.cwd()` calls in PATH bootstrap and TUI setup after the patched dotenv preflight.

## What This Changes

The PR adds a `safeCwd()` helper, routes home-dir, main-module, runtime dotenv, and CLI dotenv startup callers through it, and adds home-dir deleted-cwd regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value through dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup, while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 gives a concrete create/cd/delete/run `openclaw tui` repro, and current source tracing shows the PR still leaves later raw `process.cwd()` calls on that same command path.

## Solution Assessment

No. The `safeCwd()` helper is a good direction, but the proposed patch is not the complete fix until PATH bootstrap and TUI startup share the same deleted-cwd handling and the original TUI path has regression coverage.
