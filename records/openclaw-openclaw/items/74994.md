## Summary

Keep open. The PR moves several early cwd probes behind a safe helper, but it still does not cover the original `openclaw tui` startup path because PATH bootstrap and TUI initialization continue to call `process.cwd()` directly after the patched dotenv preflight.

## What This Changes

The PR exports a `safeCwd()` helper, uses it in home-dir, `isMainModule`, runtime dotenv, and CLI dotenv startup code, and adds home-dir deleted-cwd regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a captured safe cwd through dotenv loading, PATH bootstrap, TUI initial workspace selection, autocomplete setup, and local auth spawn setup, while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 provides a concrete create/cd/delete/run `openclaw tui` reproduction, and current source tracing shows the PR still leaves raw `process.cwd()` calls on that command path.

## Solution Assessment

No. The helper approach is a maintainable direction, but this patch is incomplete until the remaining PATH bootstrap and TUI startup cwd reads use the same safe value and the original TUI path has regression coverage.
