## Summary

Keep open. The branch moves several early startup callers toward deleted-cwd safety, but the original `openclaw tui` path still reaches raw `process.cwd()` through PATH bootstrap and later TUI startup, and the user-facing CLI crash fix is missing the required changelog entry.

## What This Changes

The PR exports a `safeCwd()` helper, routes home-dir, main-module, runtime dotenv, and CLI dotenv startup callers through it, and adds home-dir deleted-cwd regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value through dotenv loading, PATH bootstrap, TUI initial workspace selection, autocomplete setup, and local auth spawn setup while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 gives a concrete `openclaw tui` reproduction from a deleted current directory, and source tracing on current main shows the PR still leaves raw cwd calls on that command path.

## Solution Assessment

No. The helper approach is maintainable, but this patch is not complete until the remaining PATH bootstrap and TUI startup callers share the same safe-cwd behavior and the original TUI path has regression coverage.
