## Summary

Keep open. This PR moves several early cwd callers toward a safe helper, but it still does not fully fix the linked `openclaw tui` deleted-working-directory crash because the same startup path continues into PATH bootstrap and TUI code that still read `process.cwd()` directly.

## What This Changes

The branch exports a `safeCwd()` helper, routes home-dir, main-module, runtime dotenv, and CLI dotenv callers through it, and adds home-dir deleted-cwd regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value through dotenv loading, PATH bootstrap, TUI initial workspace selection, autocomplete setup, and local auth spawn setup while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 gives a concrete create/cd/delete/run `openclaw tui` repro, and current source tracing shows this PR still leaves raw `process.cwd()` calls on that command path.

## Solution Assessment

No. The `safeCwd()` helper is the right direction, but the proposed patch is incomplete until PATH bootstrap and TUI startup share the same deleted-cwd handling and the original `openclaw tui` path has regression coverage.
