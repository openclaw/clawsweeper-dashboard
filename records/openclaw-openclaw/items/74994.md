## Summary

Keep open. The PR is aimed at the right deleted-cwd bug, but it does not yet cover the full `openclaw tui` startup path from the linked report: after the patched dotenv preflight, startup still reaches raw `process.cwd()` calls in PATH bootstrap and TUI initialization.

## What This Changes

This PR adds a `safeCwd()` helper, uses it in home-dir, dotenv, CLI dotenv, and `isMainModule` startup callers, and adds home-dir regression tests for deleted-current-working-directory fallback behavior.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd/default through CLI dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup, with focused regressions for the original `openclaw tui` repro.

## Reproduction Assessment

Yes. Related #73676 provides concrete steps for `openclaw tui` from a deleted current directory, and current-main source inspection shows that path still reaches unguarded cwd reads after the PR’s patched dotenv preflight.

## Solution Assessment

No. The `safeCwd()` helper direction is maintainable, but this patch is not complete until it also guards the PATH bootstrap and TUI startup cwd reads reached by the original command.
