## Summary

Keep open. The PR improves several early deleted-cwd callers, but it does not yet cover the original `openclaw tui` path because CLI PATH bootstrap and later TUI startup still have raw `process.cwd()` calls that can throw `uv_cwd`; it also lacks the required changelog entry for a user-facing CLI crash fix.

## What This Changes

The PR exports a `safeCwd()` helper, uses it in home-dir, main-module, runtime dotenv, and CLI dotenv startup paths, and adds home-dir deleted-cwd regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a captured safe cwd through CLI dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup, while preserving cwd-local `.env` precedence whenever cwd is available and adding targeted regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. Related #73676 gives a concrete deleted-working-directory `openclaw tui` reproduction, and source tracing on current main shows the PR still leaves raw `process.cwd()` calls on that same startup path.

## Solution Assessment

No. The `safeCwd()` helper is a maintainable direction, but the proposed patch is not the best complete fix until the remaining PATH bootstrap and TUI startup cwd readers share the same deleted-cwd handling.
