## Label Justifications

- `P1`: The PR targets a regression that can block active CLI-backed agent sessions once compaction preflight starts.
- `merge-risk: 🚨 compatibility`: The global selector change can alter existing fail-closed behavior for direct harness-selector callers outside compaction.
- `merge-risk: 🚨 auth-provider`: CLI-backed configs can be routed through PI/API auth paths instead of the intended local CLI backend if helper callers bypass `runCliAgent`.
- `merge-risk: 🚨 security-boundary`: Prompts expected to remain on a local CLI backend may be sent through embedded provider execution when current helper paths call `runEmbeddedPiAgent` directly.

## Summary

Keep open with a blocking finding: the PR addresses the compaction symptom, but the fix is installed in the global harness selector and can silently route other CLI-backed helper paths through PI/API instead of preserving CLI dispatch or failing closed.

## What This Changes

The PR routes CLI backend aliases such as `claude-cli` and `google-gemini-cli` to the PI harness during agent harness selection and adds selector/compaction regression tests.

## Best Possible Solution

Keep `selectAgentHarness` fail-closed for CLI aliases and add a compaction-scoped guard in `maybeCompactAgentHarnessSession` that returns `undefined` so queued context-engine compaction can continue.

## Reproduction Assessment

Yes. Current main shows the compaction helper calls the harness selector directly, and the selector throws for model-scoped `claude-cli`; the PR also supplies live logs and a Telegram screenshot showing the before/after compaction symptom.

## Solution Assessment

No. Routing CLI aliases through PI only for compaction is the right direction, but putting that rule in the global selector is broader than the bug and can mask remaining CLI-helper routing defects.
