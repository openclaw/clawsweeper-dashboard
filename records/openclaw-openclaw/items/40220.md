## Summary

Keep open: current main still computes a local-offset top-level `time` field while serializing the original log object unchanged, so nested `_meta.date` can still appear as UTC. The directly related fix attempt in #48717 was closed unmerged, so the reported defect remains actionable.

## What This Changes

The issue requests consistent file-log timestamp fields so `_meta.date` and top-level `time` do not show different timezones for the same event.

## Best Possible Solution

Normalize `_meta.date` in the file-log JSONL payload to the already computed top-level `time` string, with focused regression coverage and a changelog fix entry.

## Reproduction Assessment

Yes. Current source computes a local-offset `time` string and JSON-serializes the unmodified `Date` under `_meta.date`; a minimal serialization check produces the reported UTC `Z` plus local-offset pair.

## Solution Assessment

Yes. Aligning `_meta.date` inside the file-log serializer is narrower than changing CLI/UI rendering or redefining `agents.defaults.userTimezone`, and it preserves the existing top-level timestamp behavior.
