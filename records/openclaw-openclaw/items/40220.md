## Summary

Keep open: current main still adds a local-offset top-level `time` while serializing the original log object unchanged, so `_meta.date` can remain a UTC `Z` string; the direct fix attempt was closed unmerged.

## What This Changes

The issue asks for file-log `_meta.date` and top-level `time` to use a consistent timestamp representation when local time differs from UTC.

## Best Possible Solution

Patch only the file-log JSONL serializer to shallow-copy `_meta` and set `_meta.date` to the computed `time`, with focused regression coverage and no change to `agents.defaults.userTimezone` or envelope timestamp semantics.

## Reproduction Assessment

Yes. Source inspection plus a minimal serialization check reproduces the mixed UTC `_meta.date` and local-offset `time` shape on current main.

## Solution Assessment

Yes. Aligning `_meta.date` inside the file-log serializer is the narrowest maintainable fix; changing CLI display, UI display, or timezone configuration semantics would be broader than the reported defect.
