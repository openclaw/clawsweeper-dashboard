## Summary

Keep open: current main still writes a local-offset top-level `time` while serializing the original tslog `_meta.date` unchanged, so file logs can still contain the mixed UTC/local timestamp shape reported here.

## What This Changes

The issue asks for file-log `_meta.date` to be consistent with the local-offset `time` field instead of showing a UTC `Z` timestamp.

## Best Possible Solution

Patch only the file-log JSONL serializer to shallow-copy `_meta` and set `_meta.date` to the computed `time`, then add focused regression coverage for that invariant.

## Reproduction Assessment

Yes. Source inspection plus a minimal serialization check show the current file-log path can still produce UTC `_meta.date` beside local-offset `time` on current main.

## Solution Assessment

Yes. Aligning `_meta.date` inside the file-log serializer is the narrowest maintainable fix; changing CLI display, UI display, `userTimezone`, or envelope timestamp semantics would be broader than the reported defect.
