## Summary

Keep open: current `main` and the latest release still serialize the original tslog `_meta.date` beside a separately formatted local-offset `time`, so the reported mixed timestamp shape remains present.

## What This Changes

The issue reports inconsistent JSON log timestamps where `_meta.date` stays UTC while top-level `time` uses local-offset formatting.

## Best Possible Solution

Update the file-log JSONL serializer to shallow-copy `_meta` and set `_meta.date` to the already computed file-log `time`, then lock that invariant with focused logging tests.

## Reproduction Assessment

Yes. Source inspection plus a minimal serialization check show the current file-log path can still produce UTC `_meta.date` beside local-offset `time` on current `main`.

## Solution Assessment

Yes. Aligning `_meta.date` inside the file-log serializer is the narrowest maintainable fix; changing CLI display, UI display, `userTimezone`, or envelope timestamp semantics would be broader than the reported defect.
