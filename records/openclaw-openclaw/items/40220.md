- `impact:other`: The concrete impact is ambiguous operational timestamps, which does not fit the more specific owned impact classes.

## Summary

Current main and the latest release still serialize tslog’s UTC `_meta.date` beside OpenClaw’s host-local `time`. The report remains valid, but changing dependency-owned metadata requires a maintainer decision because the CLI already treats `time` as the preferred display timestamp and raw-log consumers may rely on `_meta.date` as canonical UTC metadata.

## What This Changes

Define and make consistent the timezone semantics of raw file-log `_meta.date` and top-level `time` fields.

## Best Possible Solution

Explicitly define raw JSONL timestamps as either canonical UTC metadata plus a host-local display field, or normalize both only at OpenClaw’s file-serialization boundary; document that logging uses host timezone or `TZ`, preserve CLI `--local-time`/`--utc`, and add regression tests for the chosen field types and values.

## Reproduction Assessment

Yes. Current source deterministically writes host-local top-level `time` while tslog’s `Date` metadata serializes `_meta.date` in UTC on non-UTC systems; a focused file-log write can verify the two strings represent the same instant with different offsets.

## Solution Assessment

Unclear. Making the strings equal is mechanically small, but maintainers must first decide whether `_meta.date` remains dependency-owned canonical UTC metadata or becomes an OpenClaw-owned local display field; coupling it to `userTimezone` is not the narrowest solution.
