## Summary

Keep open: current `origin/main` and the latest release still write offset-formatted `time` while preserving the original tslog `_meta.date`, so the mixed timestamp shape reported by the user remains present.

## What This Changes

The issue asks for JSONL log `_meta.date` to use the same timezone-adjusted timestamp as the top-level `time` field.

## Best Possible Solution

Update the file-log JSONL serializer to shallow-copy `_meta` and set `_meta.date` to the already computed `time`, then add focused regression coverage for that invariant.

## Reproduction Assessment

Yes. Source inspection plus a minimal serialization check show the current file-log path can still produce UTC `_meta.date` beside local-offset `time` on current main.

## Solution Assessment

Yes. Aligning `_meta.date` inside the file-log serializer is the narrowest maintainable fix; changing CLI display, UI display, user timezone, or envelope timestamp semantics would be broader than the reported defect.
