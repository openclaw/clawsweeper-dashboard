## Summary

Keep open: current main still rewrites only the top-level file-log `time` field while spreading the original log object, so nested `_meta.date` can still serialize as UTC; the directly related fix PR was closed unmerged.

## What This Changes

The issue reports inconsistent JSON file-log timestamps where `_meta.date` remains UTC while the top-level `time` uses a local-offset timestamp.

## Best Possible Solution

Normalize `_meta.date` in the serialized file-log JSONL payload to the already computed `time` string while preserving redaction, structured fields, trace fields, and parser fallback behavior.

## Reproduction Assessment

Yes. Static inspection plus a minimal `JSON.stringify` check reproduce the mismatch: nested `_meta.date` serializes as UTC `Z`, while sibling `time` can be a local-offset string.

## Solution Assessment

Yes. Aligning `_meta.date` inside the file-log record is narrower and more maintainable than changing CLI rendering, envelope timestamps, or the meaning of `agents.defaults.userTimezone`.
