## Summary

Keep open: current `main` and latest release still compute a local-offset top-level `time` while serializing tslog's nested `_meta.date` unchanged, so raw JSON file logs can still show two timestamp zones. The direct fix PRs are closed unmerged, and the related open logs issue tracks CLI display defaults rather than this raw-field consistency bug.

## What This Changes

Make raw JSON file-log `_meta.date` use the same local-offset timestamp as top-level `time` instead of preserving tslog's UTC `Date` serialization.

## Best Possible Solution

Update the file-log serializer to clone `_meta` and serialize `_meta.date` as the already computed `time`, with regression coverage asserting equality; leave CLI display-default behavior to https://github.com/openclaw/openclaw/issues/46748.

## Reproduction Assessment

Yes, at source level. Current `main` computes a local-offset top-level `time` while serializing unchanged tslog `_meta.date`, whose dependency contract is a `Date` that JSON-renders as UTC.

## Solution Assessment

Yes. Aligning `_meta.date` inside `src/logging/logger.ts` is the narrowest maintainable fix; changing `openclaw logs` display defaults is separate work tracked by https://github.com/openclaw/openclaw/issues/46748.
