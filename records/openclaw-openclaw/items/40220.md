## Summary

Keep open: current main still writes a local-offset top-level `time` field while serializing the original log object unchanged, so `_meta.date` can remain a UTC `Z` string. The directly related #48717 fix attempt was closed unmerged, leaving the reported logging inconsistency actionable.

## What This Changes

The issue requests consistent timezone formatting between file-log `_meta.date` and top-level `time` fields.

## Best Possible Solution

Normalize `_meta.date` in the file-log JSONL payload to the already computed top-level `time` value, and add focused regression coverage without changing `agents.defaults.userTimezone` or envelope timestamp semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows the file logger serializes a `Date` under `_meta.date` unchanged while computing a separate local-offset `time`, and a minimal JSON serialization check reproduces the reported pair.

## Solution Assessment

Yes. Aligning `_meta.date` inside the file-log serializer is the narrowest maintainable fix; changing CLI rendering, UI rendering, or `agents.defaults.userTimezone` semantics would be broader than the defect.
