## Summary

Keep open: current main still has the reported trusted-proxy media-read path by source inspection, and existing security docs make the intended isolation boundary a maintainer/security decision rather than an auto-close.

## What This Changes

The issue asks OpenClaw to require requester-session ownership, or an explicit elevated admin path, for trusted-proxy outgoing media downloads.

## Best Possible Solution

Decide the trusted-proxy media-read contract explicitly, then either enforce requester-session ownership for non-admin trusted-proxy reads or document and test the elevated operator-read behavior.

## Reproduction Assessment

Yes, from source inspection on current main: trusted-proxy auth sets `trustDeclaredOperatorScopes`, the media route treats that as privileged, and the requester-session ownership guard is skipped. I did not run a live Gateway request in this read-only review.

## Solution Assessment

Unclear. Removing trusted-proxy from the media-route privileged path is the narrow code direction, but it must be reconciled with docs that describe `chat.history` and session keys as gateway-wide operator surfaces.
