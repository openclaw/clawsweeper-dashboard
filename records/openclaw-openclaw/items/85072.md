This PR deprecates memory-specific embedding provider registration in SDK/docs metadata, adds plugin-inspect compatibility notices for non-bundled usage, and adds guardrail tests for bundled plugins.

## Best Possible Solution

Land this as part of the generic embedding-provider stack after deriving diagnostics from actual runtime memory embedding registrations as well as manifest metadata, while keeping the deprecated compatibility path documented and tested.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a user bug, but the remaining review finding is source-reproducible: the PR’s notice condition reads manifest-derived plugin fields while runtime registrations are stored in `registry.memoryEmbeddingProviders`.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the deprecation direction is sound, but the implementation should warn from actual runtime registrations and land with the generic replacement API stack.
