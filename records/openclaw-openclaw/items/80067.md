## Summary

Keep open: current head still has blocking installer issues, including an unverified portable Git download, unscoped CLI cleanup, and a regression of the documented Windows `-Tag main` npm flow. The protected `maintainer` label also means this installer/sync PR needs explicit maintainer handling rather than cleanup closure.

## What This Changes

The PR makes OpenClaw own and sync website installer scripts, adds cross-platform installer verification, adds `install-cli.sh` and `install.cmd`, updates changed-scope routing, and adds installer tests.

## Best Possible Solution

Keep the source-owned installer sync direction, but pin or verify the portable Git path, preserve the `main` branch install mapping, remove the unconditional CLI cleanup, and merge only after maintainer/security review plus exact-head CI.

## Reproduction Assessment

Yes. The review findings are reproducible by source inspection on the PR head: the Windows installer downloads/extracts mutable portable Git, `install-cli.sh` runs cleanup before method dispatch, and `-Tag main` now resolves through the generic package-tag path.

## Solution Assessment

No. The source-owned installer sync is a reasonable direction, but the current patch is not the best solution until the mutable binary trust path and two installer regressions are fixed narrowly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify portable Git before extracting it:** `scripts/install.ps1:287-288`
  - body: This fallback resolves the latest Git-for-Windows release, downloads a MinGit ZIP, and extracts it without pinning the release or checking a digest/signature. Fresh Windows installs without Git would trust a mutable third-party binary path, so either keep the manual/package-manager path or pin and verify the archive before extraction.
  - confidence: 0.88
- **[P2] Remove the unconditional legacy cleanup:** `scripts/install-cli.sh:711`
  - body: `main` invokes `cleanup_legacy_submodules` before it knows whether this is a git install, and without passing the parsed `GIT_DIR`; the helper then falls back to `OPENCLAW_GIT_DIR` or `~/openclaw`. Since the git path already cleans the selected checkout, this call can delete `Peekaboo` outside the requested target, including during npm installs.
  - confidence: 0.9
- **[P2] Preserve main as the GitHub branch install spec:** `scripts/install.ps1:495`
  - body: `Resolve-NpmOpenClawInstallSpec` no longer special-cases `main`, so `install.ps1 -Tag main` resolves to `openclaw@main` instead of the documented GitHub branch spec `github:openclaw/openclaw#main`. That can fail or install the wrong artifact when no `main` npm dist-tag exists.
  - confidence: 0.9
