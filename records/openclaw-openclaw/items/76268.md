## Summary

Current main implements the requested stable plugin registry cache-key behavior: runtime plugin lookup keys off structural plugin/discovery/activation inputs rather than the full root config, and the earlier paired hot-reload concern is now covered by the config reload dedupe path.

## What This Changes

The issue asks for the plugin registry cache key to ignore volatile config bookkeeping so per-request runtime plugin lookup can reuse the active registry instead of synchronously reloading plugins.

## Best Possible Solution

Keep the current structural cache-key and active-registry reuse design, with any future performance work focused on narrower measured misses rather than reintroducing full-config hashing.

## Reproduction Assessment

No, not as reported on current main. Source inspection shows volatile root fields like meta.lastTouchedAt are outside the loader cache-key inputs, and compatible runtime lookup returns the active registry before loading plugins again.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution because it preserves invalidation for real plugin/discovery/runtime changes while excluding unrelated root-config bookkeeping from the cache key.
