## Summary

The PR is still useful and should remain open: it fixes the browser-profile restart problem with the right ownership shape, but the browser docs still describe all browser config changes as restart-required, so the behavior/documentation contract needs one narrow update before landing.

## What This Changes

The PR adds browser plugin hot-reload metadata for `browser.profiles`, changes reload-rule matching to prefer the longest matching prefix, and adds regression coverage for profile hot reload versus service-level browser restarts.

## Best Possible Solution

Land the plugin-owned hot-prefix plus longest-prefix planner fix after narrowing the browser docs to say profile fields hot-reload while service-level browser settings still require restart.

## Reproduction Assessment

Yes from source: current main routes `config.patch` restart scheduling through `buildGatewayReloadPlan(...).restartGateway`, and the browser plugin's broad `browser` restart prefix covers `browser.profiles.*` before this PR.

## Solution Assessment

Yes: browser-owned reload metadata plus generic longest-prefix matching is the best fit because it avoids a browser-specific core exception, preserves service-level restarts, and keeps `browser.defaultProfile` restart-scoped until its selection order is changed.
