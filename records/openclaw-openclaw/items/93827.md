## Summary

This PR should stay open: it is the current viable fix candidate for the canonical browser-profile config restart bug, and the runtime approach is source-backed. The only concrete remaining issue is that public browser docs still claim all browser config changes require a Gateway restart.

## What This Changes

The PR adds browser profile hot-reload metadata, changes reload rule matching to prefer the longest matching prefix, adds gateway/browser regression coverage, and removes one unused agents import.

## Best Possible Solution

Land the browser-owned reload metadata and longest-prefix planner behavior with a small docs update that says `browser.profiles.*` hot-reloads while service-level browser settings still restart.

## Reproduction Assessment

Yes, at source level. Current main maps `browser.profiles.*` through the broad `browser` restart prefix, and the hot-mode `config.patch` path schedules SIGUSR1 when the resulting reload plan requires a gateway restart.

## Solution Assessment

Yes, with the docs gap fixed. The fix belongs in browser-owned reload metadata plus generic longest-prefix matching; a browser-specific core exception or making all `browser.*` paths hot would be too broad.
