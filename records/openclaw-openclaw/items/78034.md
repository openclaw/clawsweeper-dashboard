## What This Changes

The branch skips the Control UI local-attachment client allowlist hard-deny while `localMediaPreviewRoots` is empty, mirrors that behavior for renderable images, adds grouped-render regression coverage, and adds a changelog line.

## Best Possible Solution

Land the focused empty-roots fallback after redacted Control UI runtime proof is supplied and the changelog entry is removed, while preserving populated-root blocking and server-side media authorization.

## Reproduction Assessment

Yes, at source level: current main can render local `MEDIA:` attachments while `localMediaPreviewRoots` is still `[]`, and the client returns `Outside allowed folders` before the server metadata request can run. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

Yes with cleanup: falling through to the existing authenticated server metadata check only while roots are empty is the narrowest maintainable fix. Adding a new config path or widening client roots would be broader and less aligned with the existing server authority.
