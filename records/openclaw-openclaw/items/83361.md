## Summary

Keep open: current main still has a credible DeepSeek/MCP schema failure path because bundle MCP tools are appended after provider tool-schema normalization, and the DeepSeek plugin/manifest has no anyOf/oneOf compatibility rule.

## What This Changes

Request provider-compatible handling for MCP tool schemas with anyOf/oneOf so DeepSeek turns do not fail when MCP servers expose union input schemas.

## Best Possible Solution

Keep provider-specific schema policy in provider-owned compatibility hooks, and make sure materialized bundle MCP tools pass through that compatibility step before any DeepSeek request is sent.

## Reproduction Assessment

Yes. Source inspection shows bundle MCP schemas with anyOf/oneOf can reach DeepSeek without provider-specific normalization, matching the reporter's DeepSeek 400 error path.

## Solution Assessment

No to a generic gateway-level strip as the first choice. The narrower maintainable fix is to route MCP tools through the existing provider-owned schema compatibility boundary and add DeepSeek-specific coverage there.
