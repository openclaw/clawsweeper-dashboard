## Summary

Keep open: current main still has a clear DeepSeek/MCP schema failure path because bundle MCP tools can carry nested anyOf/oneOf schemas into the final tool set after provider-owned normalization has already run, and the DeepSeek provider does not register a tool-schema compatibility hook.

## What This Changes

The issue reports DeepSeek rejecting MCP tool schemas that contain anyOf/oneOf and asks for schema sanitization before provider requests.

## Best Possible Solution

Add provider-owned DeepSeek tool-schema compatibility and ensure all effective tools, including bundle MCP tools, pass through that compatibility step before any DeepSeek request is sent.

## Reproduction Assessment

Yes, source-reproducible with high confidence: configure a bundle MCP server whose inputSchema contains nested anyOf/oneOf and run a DeepSeek turn. Current main preserves nested oneOf in MCP materialization and appends bundle MCP tools after provider normalization.

## Solution Assessment

No to a generic gateway-level strip as the first choice; the narrower maintainable fix is provider-owned DeepSeek schema compatibility plus a request-boundary change so bundle MCP tools are included in normalization.
