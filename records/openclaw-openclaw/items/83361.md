## Summary

Keep open: current main still has a source-reproducible DeepSeek/MCP schema failure path because bundled MCP tools are materialized after provider tool-schema normalization, nested MCP anyOf/oneOf remains intact, and the DeepSeek plugin does not register a schema compatibility hook. Duplicate search found no canonical issue or PR taking over this work.

## What This Changes

The issue requests that MCP tool schemas containing anyOf/oneOf be made DeepSeek-compatible before those tools are sent to the provider.

## Best Possible Solution

Add DeepSeek-owned tool-schema compatibility and ensure bundled MCP tools pass through provider normalization after materialization and before provider submission, with regression coverage for nested MCP anyOf/oneOf schemas.

## Reproduction Assessment

Yes, source-reproducible with high confidence: configure a bundled MCP server whose inputSchema contains a nested anyOf/oneOf property and run a DeepSeek turn. Current main appends that MCP tool schema after provider normalization, and DeepSeek has no tool-schema normalizer.

## Solution Assessment

No to a generic gateway-level strip as the first choice; the better fit is provider-owned DeepSeek schema compatibility plus routing bundled MCP tools through that provider boundary. That keeps provider policy out of generic gateway code while fixing the observed failure.
