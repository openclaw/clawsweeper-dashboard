## Summary

Keep open: current main still has a source-backed Mattermost final-delivery gap where the inbound final path can finish without proving that a visible Mattermost post was sent, and the related wake/thread-routing items do not cover this last-mile delivery contract.

## What This Changes

The issue requests that Mattermost group/channel/thread runs which produce a final answer must either post a visible reply in the originating conversation or surface an explicit delivery failure.

## Best Possible Solution

Keep the issue open and make the Mattermost final-delivery path return/check visible delivery status, preserving reasoning-only suppression while treating non-reasoning no-send finals as explicit failures or fallbacks.

## Reproduction Assessment

No high-confidence live Mattermost reproduction was established. Source inspection does show a current-main path where a final delivery helper can complete with no visible-send result checked, which matches the reported failure class.

## Solution Assessment

Yes. The narrow maintainable fix is to enforce the existing visible-or-fail delivery contract in the Mattermost final path, not to fold this into the separate wake detection or thread-root routing items.
