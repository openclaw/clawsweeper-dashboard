## Summary

Keep open: current main still lacks the requested Discord PluralKit canonical inbound ID behavior, and this PR remains useful but is not merge-ready because the PR head runs the webhook-capable PK lookup after the bot-drop gate and is stale against the current process-context refactor.

## What This Changes

The PR looks up PluralKit data for webhook-authored Discord messages, carries a canonical original ID through preflight, uses it as inbound `MessageSid`, preserves the provider Discord ID as `MessageSidFull`, and adds regression tests.

## Best Possible Solution

Land a rebased Discord fix that suppresses known OpenClaw bound-thread webhook echoes before external PK calls, performs PK lookup for every other PK-enabled inbound message before bot and mention gates, carries a trimmed `pluralkitInfo.original` through the current preflight context, and emits canonical `MessageSid` plus provider `MessageSidFull` from the current context builder with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. On current main, a webhook-authored PK message with `discord.pluralkit.enabled` hits the helper's `webhookId` early return, so the sender is not marked PluralKit and the inbound context still uses the provider `message.id` as `MessageSid`.

## Solution Assessment

Is this the best way to solve the issue? No. The provenance-driven canonical ID approach is the right narrow direction, but the current PR head needs the PK lookup ordered before bot-drop gates and needs its `MessageSid` mapping rebased into `message-handler.context.ts`.
