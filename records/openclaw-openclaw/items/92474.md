## Summary

Keep this open: current `main` and the latest release still accept blocked stdio MCP env keys at write time and warn during runtime resolution, while two linked fixing PRs remain open and unmerged.

## What This Changes

The issue asks OpenClaw to stop silently accepting blocked stdio MCP environment keys such as `PYTHONPATH` and to avoid repeated per-spawn warning noise in gateway logs.

## Best Possible Solution

Keep this issue open until a canonical fix lands, preferably the full-boundary approach from https://github.com/openclaw/openclaw/pull/92484 or an equivalent that aligns write validation, schema diagnostics, runtime warning cadence, and doctor recovery.

## Reproduction Assessment

Yes, at source level: current main saves arbitrary MCP stdio `env` keys and the runtime resolver logs when it drops blocked keys on each transport resolution. I did not run the CLI because this review is read-only.

## Solution Assessment

Yes, the best direction is a shared contract fix rather than a local warning cache only: enforce or diagnose blocked stdio env keys at the MCP config boundary, retain runtime filtering for defense in depth, and provide doctor recovery for released configs.
