## Summary

Keep open. Current main and v2026.6.6 still save blocked stdio MCP env keys at the config boundary and only drop them during runtime transport resolution, while the public MCP docs say these keys are rejected with a configuration error; two live closing PRs now own the repair path.

## What This Changes

The issue reports that blocked stdio MCP environment keys such as PYTHONPATH can be saved and then repeatedly warn during later MCP operations.

## Best Possible Solution

Review and land the compatibility-safe linked MCP fix that enforces the documented write/validation error, dedupes runtime warnings for legacy configs, and provides doctor recovery for shipped saved configs.

## Reproduction Assessment

Yes at the source level. Current main writes arbitrary MCP server env keys in `setConfiguredMcpServer`, then `resolveMcpTransportConfig` drops dangerous stdio env keys and logs the reported warning whenever that server is resolved; no live command was run because this review was read-only.

## Solution Assessment

Yes: the best path is a narrow config-boundary fix, not a new feature. It should reuse the existing stdio env safety predicate, align schema/CLI/docs, keep runtime filtering as defense in depth, and handle shipped configs through doctor recovery.
