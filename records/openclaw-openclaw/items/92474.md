## Summary

Current main accepts blocked stdio MCP environment keys and repeatedly logs when resolving them, while the public MCP documentation promises a configuration error; no related issue or PR currently owns this mismatch.

## What This Changes

The issue asks OpenClaw to prevent repeated gateway warnings when a blocked stdio MCP environment key such as `PYTHONPATH` remains in saved configuration.

## Best Possible Solution

Centralize blocked stdio environment-key diagnostics, enforce the documented configuration error at write and validation boundaries, surface the exact key through status or doctor, and eliminate repeated per-resolution logging.

## Reproduction Assessment

Yes at the source level. Current main saves arbitrary MCP environment keys, and runtime spawning plus multiple MCP CLI operations call the resolver that emits the exact warning; no commands were executed because this review was required to remain read-only.

## Solution Assessment

Yes, with a contract-aligned implementation: shared validation and structured diagnostics are preferable to a process-global warning cache because the documentation already defines blocked keys as configuration errors.
