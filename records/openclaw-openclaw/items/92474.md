## Summary

Current main silently saves blocked stdio MCP environment keys, then repeatedly warns from a resolver used by several CLI and runtime paths, contradicting the documented configuration-error contract. No related issue or pull request currently owns the repair.

## What This Changes

Reject or clearly diagnose blocked stdio MCP environment keys such as `PYTHONPATH` when saving server config, and prevent the same fallback warning from flooding logs during repeated resolution or spawn.

## Best Possible Solution

Use the existing stdio launch-policy helper at the MCP configuration control plane to reject blocked keys before persistence, while retaining at most one lifecycle-scoped diagnostic for legacy or externally sourced raw definitions and leaving the safety blocklist unchanged.

## Reproduction Assessment

Yes at source level: the writer accepts the definition, and existing focused tests prove the exact dropped-key warning emitted by the resolver used across repeated MCP operations. A live gateway reproduction was not run during this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is shared configuration-time validation plus lifecycle-scoped fallback diagnostics; weakening the blocklist, adding a config option, or retaining per-call warnings would be inferior.
