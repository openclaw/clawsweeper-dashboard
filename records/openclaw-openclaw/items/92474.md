## Summary

Current main silently persists blocked stdio MCP environment keys, then logs each dropped key whenever the shared transport configuration is resolved; this conflicts with the documented configuration-error contract, and no related issue or pull request currently owns the repair.

## What This Changes

Reject or clearly diagnose blocked stdio MCP environment keys during configuration and prevent repeated per-resolution gateway warnings.

## Best Possible Solution

Apply the existing stdio safety policy at every canonical MCP server write boundary, return a clear blocked-key configuration error, and retain runtime sanitization as defense in depth while reporting manually introduced invalid config without per-call log repetition.

## Reproduction Assessment

Yes at source level with high confidence: persist a command-based MCP server containing `PYTHONPATH`, then invoke any path that calls `resolveMcpTransportConfig`; the key is dropped and the callback logs again. A live packaged reproduction was not run during this read-only review.

## Solution Assessment

Yes. Shared validation at the canonical config boundary is narrower and more maintainable than warning-and-write behavior, while runtime sanitization should remain as defense against manually edited or legacy invalid config.
