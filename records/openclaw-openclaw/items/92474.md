## Summary

Current main still matches the reported bug: MCP config writes accept blocked stdio env keys, while transport resolution drops them and logs the warning from each resolving call. Keep this issue open because two live linked PRs still target the fix and none has merged.

## What This Changes

The issue asks OpenClaw to stop flooding gateway logs when a stdio MCP server config contains blocked env keys such as `PYTHONPATH`, by rejecting, surfacing, or throttling the policy earlier.

## Best Possible Solution

Land one canonical fix that uses the existing stdio env safety predicate at config write and validation boundaries, keeps bounded runtime defense for old or hand-edited configs, and provides upgrade recovery for released configs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: save a command-bearing MCP server env containing `PYTHONPATH` through `openclaw mcp set`/`setConfiguredMcpServer`, then resolve it through `resolveMcpTransportConfig` to drop the key and log the warning.

## Solution Assessment

Yes, the best fix shape is clear: enforce the existing stdio env safety policy at write/validation boundaries while preserving a bounded runtime defense and migration/recovery path for already-saved configs.
