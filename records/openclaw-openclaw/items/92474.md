## Summary

Keep this issue open: current main and latest stable still allow blocked stdio MCP env keys to be saved and still log the dropped-env warning from the runtime resolver, while the open linked PR is the canonical fix path.

## What This Changes

The issue asks OpenClaw to stop silently saving blocked MCP stdio env keys such as PYTHONPATH and to avoid per-spawn gateway journal warning floods.

## Best Possible Solution

Land or reject the canonical open PR; the desired end state is one shared blocked-env policy across MCP write validation, schema validation, runtime fallback, doctor repair, docs, and regression tests.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: current main persists arbitrary MCP env records, then each stdio transport resolution drops blocked keys through the warning-producing resolver path. I did not run the CLI because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The best fix is not a new policy knob; it is to enforce the existing stdio startup-safety policy at the write/config-validation boundary while keeping a de-duplicated runtime fallback and doctor recovery for released configs.
