## Summary

Keep open: current main and v2026.6.8 still contain a source-visible zero-reply/no-resolver path after an active user abort, while the Feishu-specific CardKit state-machine part still needs live confirmation.

## What This Changes

The issue reports that after `/stop` aborts an in-progress Feishu streaming-card response, the next message is dispatched but the agent is not invoked and no reply is produced.

## Best Possible Solution

Keep this issue open until maintainers live-confirm the Feishu `/stop` flow and fix the owner path so an aborted same-chat reply/card lifecycle cannot suppress the next visible turn before the resolver runs.

## Reproduction Assessment

Yes at source level, but not with high-confidence live Feishu proof. Current main and v2026.6.8 still contain a test and code path where a running reply operation aborted by the user can make the next dispatch return `queuedFinal:false` with zero counts and no resolver call.

## Solution Assessment

Unclear. Resetting or settling state after `/stop` is the right end state, but the best fix must be pinned to the owner path that remains active after abort rather than broadly weakening abort suppression or Feishu streaming-card closeout.
