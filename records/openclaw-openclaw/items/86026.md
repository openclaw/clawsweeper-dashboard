## What This Changes

The issue reports that Feishu `/stop` aborts an active streaming-card response but leaves state stale so the next same-chat message dispatches with `replies=0` and no agent invocation.

## Best Possible Solution

Keep this issue open until a Feishu-capable maintainer captures a current-main `/stop` then follow-up trace and fixes the earliest owner path so an aborted same-chat reply/card lifecycle cannot suppress the next visible turn before the resolver runs.

## Reproduction Assessment

Yes at source level, but not with high-confidence live Feishu proof. Current main and v2026.6.8 still contain a test and code path where a running reply operation aborted by the user can make the next dispatch return zero counts without calling the resolver.

## Solution Assessment

Unclear as an implementation path. Resetting or settling state after `/stop` is the right end state, but the best fix must be pinned to the owner path that remains active after abort rather than broadly weakening abort suppression or Feishu streaming-card closeout.
