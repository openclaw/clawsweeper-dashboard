## Summary

Keep open. Current main still has the shared OpenAI-compatible collision path: provider wrappers can write `payload.thinking`, then the later `extra_body` wrapper logs and overwrites the same key; native Ollama `think` handling has improved, but the framework-level warning path remains.

## What This Changes

The issue asks OpenClaw to stop `extra_body.thinking` from colliding with provider-generated thinking payload fields across OpenAI-compatible providers and to keep Ollama thinking-off behavior reliable.

## Best Possible Solution

Define one shared OpenAI-compatible request rule that preserves documented `extra_body` override precedence, suppresses or avoids equivalent `thinking` collisions, normalizes disabled thinking aliases, and keeps native Ollama `think` handling in the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: source inspection shows a DeepSeek V4 wrapper can write `payload.thinking`, after which configured `extra_body.thinking` collides, logs, and overwrites it on current main. The reported live Ollama timeout/stuck-session impact remains unproven from the inspected source.

## Solution Assessment

Unclear because this issue has no patch and the linked open PRs solve narrower slices. A coordinated request-precedence and thinking-alias fix is safer than a provider-by-provider workaround.
