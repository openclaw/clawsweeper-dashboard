## Summary

Keep open. Current main still has the shared OpenAI-compatible collision path: provider wrappers can write `payload.thinking`, then the later `extra_body` merge warns and overwrites the same key; related open PRs address pieces but do not yet settle the shared request-precedence contract.

## What This Changes

Request to eliminate the OpenAI-compatible `extra_body.thinking` collision and downstream thinking-mode failures reported across DeepSeek, Ollama, and related providers.

## Best Possible Solution

Define one shared OpenAI-compatible request rule that preserves documented `extra_body` override precedence, suppresses noisy equivalent `thinking` collisions, normalizes supported thinking-off aliases before provider wrappers, and keeps native Ollama `think` mapping in the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: a provider wrapper such as DeepSeek V4 can write `payload.thinking`, then `params.extra_body.thinking` still collides and overwrites it on current main. The live Ollama timeout/stuck-session impact remains unproven from the available source-only evidence.

## Solution Assessment

Unclear because this issue has no patch and the related open PRs solve narrower slices. A provider-by-provider workaround would be weaker than a coordinated fix for request precedence, alias normalization, and provider replay cleanup.
