## Summary

Keep open. Current main still applies `params.extra_body` after provider wrappers, logs the reported collision, and overwrites existing payload keys, so the `thinking` collision remains; the related DeepSeek issue and PRs address narrower disabled-thinking replay/alias gaps rather than the shared collision and warning contract.

## What This Changes

The issue asks OpenClaw to stop `params.extra_body.thinking` from colliding with provider-generated top-level `thinking` request fields across OpenAI-compatible providers, with DeepSeek and Ollama examples.

## Best Possible Solution

Define a documented OpenAI-compatible request-payload rule for `extra_body` collisions: preserve predictable explicit override precedence, suppress noisy no-op `thinking` collisions when the final payload is equivalent, reconcile DeepSeek final-payload cleanup after overrides, and keep native Ollama `think` behavior in the Ollama plugin unless a focused core reproduction proves a shared gap.

## Reproduction Assessment

Yes for the core collision path: configure a DeepSeek V4 OpenAI-compatible model so the provider wrapper writes top-level `payload.thinking`, and also set `params.extra_body.thinking`; current main warns and overwrites that key later. The broader timeout impact is not yet reproducible from source alone.

## Solution Assessment

Unclear because this issue has no proposed patch and the related DeepSeek work does not settle the shared `extra_body` precedence contract. The narrowest maintainable path is maintainer scoping of the precedence/warning rule before changing request shaping across providers.
