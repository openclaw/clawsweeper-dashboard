## Summary

Keep open. Current main still has the reported shared request-shaping path: a provider wrapper can write top-level `thinking`, then the later `extra_body` wrapper logs a collision and overwrites that same key. The core warning path is clear, but the correct fix needs a maintainer decision because docs currently define `extra_body` as the final override contract for OpenAI-compatible payloads.

## What This Changes

The issue asks OpenClaw to stop colliding on top-level `thinking` between provider request wrappers and `params.extra_body`, removing the repeated warning and related provider reliability failures.

## Best Possible Solution

Codify a narrow request-payload rule: keep explicit `extra_body` final precedence, avoid noisy duplicate `thinking` collisions when the final value is already intentional or equivalent, and keep native Ollama `think` handling inside the Ollama plugin unless a separate shared-core reproduction proves otherwise.

## Reproduction Assessment

Yes for the core warning path: configure an OpenAI-compatible DeepSeek V4 model with `params.extra_body.thinking` while the provider wrapper writes top-level `payload.thinking`; current main will warn and overwrite. The timeout and stuck-session impacts remain unclear without live provider reproduction.

## Solution Assessment

Unclear because this issue has no proposed patch and the fix touches a documented shared payload precedence rule. The safer path is maintainer scoping before changing cross-provider request shaping.
