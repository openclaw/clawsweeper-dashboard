## Summary

Keep open because current main still has the shared OpenAI-compatible `extra_body` collision path and the related open PRs only address narrower slices of the thinking/extra-body interaction.

## What This Changes

The issue reports repeated OpenAI-compatible request warnings where `params.extra_body.thinking` collides with provider-generated `thinking` payload fields across providers.

## Best Possible Solution

Define one shared OpenAI-compatible request rule that preserves explicit `extra_body` precedence, suppresses noisy equivalent `thinking` collisions, normalizes common thinking-off aliases before provider wrapping, and keeps native Ollama `think` behavior in the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: a DeepSeek V4-style provider wrapper can write `payload.thinking`, and the later `extra_body` merge will warn and overwrite the same key on current main. The reported live Ollama timeout impact remains unclear without a live reproduction or logs tied to the native/OpenAI-compatible route actually used.

## Solution Assessment

Unclear because this issue has no patch and the related PRs solve narrower pieces. A provider-by-provider workaround is less maintainable than a coordinated fix for request precedence, alias normalization, and provider replay cleanup.
