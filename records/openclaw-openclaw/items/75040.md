## Summary

Keep open. Current main still has the shared OpenAI-compatible collision path: a provider wrapper can write `payload.thinking`, then the later `extra_body` wrapper logs and overwrites the same key; native Ollama `think` handling has improved, but the reported framework-level warning path is still present.

## What This Changes

The issue requests a framework-level fix for duplicated `thinking` request fields so `extra_body` does not noisily overwrite generated provider payloads or destabilize thinking-disabled providers.

## Best Possible Solution

Define one shared OpenAI-compatible request rule that preserves documented `extra_body` override precedence, avoids or suppresses equivalent `thinking` collisions, normalizes disabled thinking aliases, and keeps native Ollama `think` handling in the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: source inspection shows a DeepSeek V4 wrapper can write `payload.thinking`, after which `extra_body.thinking` collides and overwrites it on current main. The reported live Ollama timeout/stuck-session impact remains unproven from the inspected source.

## Solution Assessment

Unclear because this issue has no patch and the open related PRs solve narrower slices. A coordinated request-precedence and thinking-alias fix is safer than a provider-by-provider workaround.
