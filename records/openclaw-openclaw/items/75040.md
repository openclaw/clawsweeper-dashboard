## Summary

Keep open. Current main still has the shared `extra_body.thinking` collision path: a provider wrapper can write top-level `payload.thinking`, then the later `extra_body` wrapper warns and overwrites that key. Current main has useful DeepSeek replay cleanup and native Ollama `think` handling, but it does not settle the broader collision/noisy-warning contract or the `params.thinking=false`/`disabled` alias gap discussed by the related open PR.

## What This Changes

The issue asks OpenClaw to stop duplicate `thinking` writes in OpenAI-compatible request payload construction so `extra_body` does not produce noisy collision warnings or inconsistent thinking controls across providers.

## Best Possible Solution

Define a narrow request-payload rule for OpenAI-compatible `extra_body` collisions: keep explicit `extra_body` precedence, normalize supported thinking-off aliases before provider wrappers, avoid noisy no-op `thinking` collisions when the final value is equivalent, and keep native Ollama `think` handling inside the Ollama plugin unless a separate shared-core reproduction proves otherwise.

## Reproduction Assessment

Yes for the core warning path: use a DeepSeek V4 OpenAI-compatible model whose provider wrapper writes top-level `payload.thinking`, then configure `params.extra_body.thinking`; current main warns and overwrites the key. No high-confidence local reproduction proves the reported live timeout or native Ollama stuck-session impact.

## Solution Assessment

Unclear because this issue has no patch and current main only partially addresses related DeepSeek replay behavior. The safest solution is a maintainer-scoped change that coordinates the related PRs with the documented `extra_body` precedence contract rather than a broad provider-by-provider workaround.
