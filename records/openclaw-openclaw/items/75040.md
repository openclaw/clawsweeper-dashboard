## Summary

Keep open. Current main still has the reported source-level collision path: provider wrappers can write top-level `thinking`, then the post-plugin `extra_body` wrapper warns and overwrites colliding keys. The related DeepSeek issue and PR are useful context, but they target `reasoning_content` replay behavior and do not supersede this broader request-payload precedence problem.

## What This Changes

The issue asks OpenClaw to prevent configured `params.extra_body.thinking` from colliding with provider-generated `thinking` request payload fields and causing WARN noise or reliability problems across providers.

## Best Possible Solution

Define a maintainer-approved request-payload precedence and logging policy for OpenAI-compatible providers: preserve explicit override behavior, avoid noisy no-op `thinking` collisions where appropriate, and keep native Ollama `think` handling inside the Ollama plugin unless a focused reproduction proves a shared core gap.

## Reproduction Assessment

Yes for the core collision path: configure an OpenAI-compatible provider wrapper that writes top-level `thinking`, such as DeepSeek V4, and also configure `params.extra_body.thinking`; current main will warn and overwrite the generated key when the payload is emitted. The broader timeout impact is not proven by source inspection alone.

## Solution Assessment

Unclear because this issue has no proposed patch and the right fix depends on the intended `extra_body` precedence contract. The narrow DeepSeek `reasoning_content` work is useful but is not the best complete solution for this shared collision path.
