## Summary

Keep open. Current main still has the shared OpenAI-compatible collision path: provider wrappers can write `payload.thinking`, and the later `extra_body` merge still warns and overwrites that key; related PRs are open but not merged.

## What This Changes

The issue asks OpenClaw to stop duplicate `thinking` writes in OpenAI-compatible request payloads while preserving explicit `params.extra_body` override behavior.

## Best Possible Solution

Define one shared OpenAI-compatible request rule that preserves documented `extra_body` override precedence, suppresses noisy equivalent `thinking` collisions, normalizes supported thinking-off aliases before provider wrappers, and keeps native Ollama mapping on top-level `think`.

## Reproduction Assessment

Yes for the core warning path: a provider wrapper such as DeepSeek V4 can write `payload.thinking`, then `params.extra_body.thinking` collides and overwrites it on current main. The live timeout/stuck-session impact remains unclear from source-only evidence.

## Solution Assessment

Unclear because this issue has no patch and the related open PRs solve narrower pieces. A coordinated shared request-precedence and alias-normalization fix is safer than provider-by-provider workarounds.
