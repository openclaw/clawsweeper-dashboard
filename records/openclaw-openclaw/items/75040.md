## Summary

Keep open. Current main still has the reported shared `extra_body.thinking` collision path: generated/provider payload wrappers can set top-level `thinking`, then the later generic `extra_body` wrapper warns and overwrites the same key. The related DeepSeek PRs are useful context but do not yet settle the cross-provider precedence and warning contract.

## What This Changes

The issue requests fixing duplicate `thinking` request payload writes so `extra_body` does not generate noisy overwrite warnings or inconsistent thinking behavior across providers.

## Best Possible Solution

Define a small documented request-payload rule for OpenAI-compatible `extra_body` collisions: preserve explicit override precedence where intended, suppress noisy no-op `thinking` collisions when final values are equivalent, and keep native Ollama `think` semantics inside the Ollama plugin unless a focused shared-core reproduction proves otherwise.

## Reproduction Assessment

Yes for the core warning path: configure a DeepSeek V4 OpenAI-compatible model so the provider wrapper writes top-level `payload.thinking`, and also set `params.extra_body.thinking`; current main will warn and overwrite that key later. The live timeout impact still needs provider-level reproduction.

## Solution Assessment

Unclear because there is no proposed patch for this issue, and related PR #74418 only addresses disabled-thinking alias resolution. The safest solution is a maintainer-scoped request-shaping rule before changing shared behavior across providers.
