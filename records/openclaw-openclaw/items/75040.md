## Summary

Keep open. Current main still has the shared OpenAI-compatible `extra_body` collision path: provider/plugin wrappers can write `payload.thinking`, then the later `extra_body` wrapper warns and overwrites the same key; related open PRs cover overlapping DeepSeek alias/replay symptoms but do not fully settle the shared collision contract.

## What This Changes

The issue asks OpenClaw to stop writing `thinking` twice in OpenAI-compatible request payload shaping so `extra_body` no longer causes warning/noisy collisions across providers.

## Best Possible Solution

Define a narrow shared rule for OpenAI-compatible `extra_body` collisions that preserves documented explicit override precedence, treats equivalent `thinking` values as non-noisy, normalizes supported thinking-off aliases before provider wrappers, and leaves native Ollama `think` mapping in the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: an OpenAI-compatible provider wrapper such as DeepSeek V4 can write `payload.thinking`, then `params.extra_body.thinking` still collides and overwrites it on current main. The live Ollama timeout/stuck-session impact is not reproducible from the available source-only evidence.

## Solution Assessment

Unclear because this issue has no patch and the related open PRs solve overlapping but narrower pieces. The maintainable fix should coordinate request precedence, alias normalization, and provider replay cleanup instead of landing a provider-by-provider workaround that leaves the shared collision behavior ambiguous.
