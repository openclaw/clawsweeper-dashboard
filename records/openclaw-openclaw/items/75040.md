## Summary

Keep open. Current `main` and the latest release still contain the shared OpenAI-compatible path where a provider wrapper can write `payload.thinking` before `extra_body` logs and overwrites the same key; the related open PRs address narrower DeepSeek disabled-thinking slices but do not settle the broader precedence/warning contract.

## What This Changes

The issue reports a framework-level `extra_body` collision on the `thinking` request key across OpenAI-compatible provider payloads.

## Best Possible Solution

Define one shared OpenAI-compatible thinking precedence rule that normalizes disabled aliases, preserves final `extra_body` override semantics, avoids or suppresses equivalent `thinking` collisions, cleans up disabled-thinking replay state, and keeps native Ollama `think` mapping inside the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: current source shows a provider wrapper can write `payload.thinking`, then the later `extra_body` wrapper logs and overwrites the same key. The live Ollama timeout/stuck-session impact remains unproven by this read-only source review.

## Solution Assessment

No patch is attached, and the two related open PRs are narrower than this issue by themselves. The best fix is a coordinated request-precedence and disabled-thinking cleanup rather than separate provider workarounds that leave the noisy collision contract ambiguous.
