## Summary

Keep open. Current main still has the shared OpenAI-compatible `extra_body` collision path for `thinking`, and related open PRs only cover narrower DeepSeek alias/replay cases rather than the broader request-payload precedence contract.

## What This Changes

The issue requests a framework-level fix so `params.extra_body.thinking` and provider-generated `thinking` payload fields do not collide noisily or leave inconsistent thinking state across providers.

## Best Possible Solution

Define and test a narrow shared rule for OpenAI-compatible `extra_body` collisions: keep explicit `extra_body` precedence, normalize supported thinking-off aliases before provider wrappers, suppress no-op equivalent `thinking` collisions, and keep native Ollama `think` behavior in the Ollama plugin unless a separate shared-core reproduction proves otherwise.

## Reproduction Assessment

Yes for the core warning path: configure an OpenAI-compatible model whose provider wrapper writes top-level `payload.thinking`, then configure `params.extra_body.thinking`; current main still warns and overwrites that key. The live Ollama idle-timeout impact remains unclear from source-only review.

## Solution Assessment

Unclear because this issue has no patch and related open PRs solve narrower symptoms. The maintainable path is a coordinated request-payload precedence and thinking-alias fix, not a provider-by-provider workaround that leaves the shared contract ambiguous.
