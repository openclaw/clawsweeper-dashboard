## Summary

Keep open. Current main still has the shared OpenAI-compatible collision path: provider wrappers can write `payload.thinking`, then the later `extra_body` merge warns and overwrites the same key; related open PRs address pieces but do not settle the shared precedence contract.

## What This Changes

The issue asks OpenClaw to stop `params.extra_body.thinking` from colliding with generated thinking payload fields across OpenAI-compatible providers, while keeping DeepSeek and Ollama thinking-off behavior reliable.

## Best Possible Solution

Define one shared OpenAI-compatible request rule that preserves documented `extra_body` override precedence, suppresses noisy equivalent `thinking` collisions, normalizes supported thinking-off aliases before provider wrappers, and keeps native Ollama `think` mapping in the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: configure an OpenAI-compatible provider wrapper such as DeepSeek V4 to write `payload.thinking`, then set `params.extra_body.thinking`; current main will warn and overwrite during the later extra-body merge. The live Ollama timeout/stuck-session impact is plausible but not proven by source-only review.

## Solution Assessment

Unclear because this issue has no patch and the related open PRs solve narrower slices. A provider-by-provider workaround would be weaker than a coordinated fix for request precedence, alias normalization, and provider replay cleanup.
