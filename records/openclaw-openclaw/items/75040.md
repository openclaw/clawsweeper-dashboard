## Summary

Keep open. Current main and the latest release still contain the shared OpenAI-compatible path where a provider wrapper can write `payload.thinking`, then `extra_body` logs and overwrites the same key; adjacent open PRs cover narrower DeepSeek disabled-thinking slices but do not close the broader warning/precedence issue.

## What This Changes

The issue asks OpenClaw to stop duplicated `thinking` writes in the OpenAI-compatible request payload pipeline so `extra_body` no longer repeatedly collides with provider-generated payload fields.

## Best Possible Solution

Define one shared OpenAI-compatible thinking precedence rule that normalizes disabled aliases, preserves final `extra_body` override semantics, avoids or suppresses equivalent `thinking` collisions, and keeps native Ollama `think` mapping in the Ollama plugin.

## Reproduction Assessment

Yes for the core warning path: current source shows a provider wrapper can write `payload.thinking`, then the later `extra_body` wrapper logs and overwrites the same key. The live Ollama timeout/stuck-session impact remains unproven by this source-only review.

## Solution Assessment

No patch is attached, and the adjacent PRs are narrower than this issue by themselves. The maintainable fix should be a coordinated request-precedence and disabled-thinking cleanup rather than provider-by-provider workarounds that preserve the noisy collision path.
