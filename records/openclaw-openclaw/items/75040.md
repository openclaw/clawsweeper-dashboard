## Summary

Keep open. Current main still has the reported `extra_body.thinking` collision path, and the related DeepSeek issue/PRs cover narrower disabled-thinking replay gaps rather than the shared `extra_body` precedence and warning contract.

## What This Changes

The issue requests a framework-level fix for `params.extra_body.thinking` colliding with provider-generated `thinking` request fields across OpenAI-compatible providers, with DeepSeek and Ollama examples.

## Best Possible Solution

Define a small, documented request-payload rule for OpenAI-compatible `extra_body` collisions: keep explicit override precedence predictable, avoid noisy no-op `thinking` collisions when the final payload is already equivalent, and keep native Ollama `think` behavior inside the Ollama plugin unless a focused shared-core reproduction proves otherwise.

## Reproduction Assessment

Yes for the core collision path: configure a DeepSeek V4 OpenAI-compatible model so the provider wrapper writes top-level `payload.thinking`, and also set `params.extra_body.thinking`; current main warns and then overwrites that key later. The broader timeout impact remains unclear without live provider reproduction.

## Solution Assessment

Unclear because there is no proposed patch for this issue, and the related DeepSeek work does not settle the shared `extra_body` precedence and warning contract. The safer path is maintainer scoping before changing request-shaping behavior across providers.
