## Summary

Keep open. Current main still has a concrete `extra_body.thinking` collision path: provider wrappers can write top-level `thinking`, then the generic OpenAI-compatible `extra_body` wrapper warns and overwrites that key afterward. The related DeepSeek report and PR cover a narrower replay failure, while the Ollama evidence is partly provider-native `think` handling, so this issue is not fully implemented or superseded.

## What This Changes

The issue requests a framework-level fix for `params.extra_body.thinking` colliding with provider-generated `thinking` payload fields across OpenAI-compatible providers, with DeepSeek and Ollama examples.

## Best Possible Solution

Define a canonical request-payload precedence rule for OpenAI-compatible providers, keep explicit overrides predictable, suppress noisy no-op `thinking` collisions where appropriate, and leave native Ollama `think` semantics in the Ollama plugin unless a focused reproduction proves a shared core gap.

## Reproduction Assessment

Yes for the core collision path: configure a DeepSeek V4 OpenAI-compatible model so the provider wrapper writes top-level `payload.thinking`, and also set `params.extra_body.thinking`; current main will warn and then overwrite that key in the later wrapper. The all-provider timeout impact is less certain, but the collision itself has a high-confidence source reproduction.

## Solution Assessment

Unclear because there is no proposed patch on this issue, and the narrower DeepSeek work does not settle the shared `extra_body` precedence contract. The safest solution needs maintainer scoping before changing core request-shaping behavior across providers.
