## Summary

Keep open. Current main still has a source-reproducible shared collision path where a provider wrapper can write `payload.thinking` before the later `extra_body` wrapper warns and overwrites the same key; native Ollama `think` handling has improved, but the central warning/precedence issue remains unresolved.

## What This Changes

The issue reports that generated provider thinking parameters and user `params.extra_body.thinking` can both write the same OpenAI-compatible request key, producing repeated collision warnings and possibly inconsistent provider behavior.

## Best Possible Solution

Define a shared OpenAI-compatible request-precedence and thinking-normalization rule that preserves documented `extra_body` override precedence, avoids repeated warning spam for equivalent generated/user thinking values, and keeps native Ollama `think` handling inside the Ollama plugin.

## Reproduction Assessment

Yes for the source-level warning path: current main shows a provider wrapper can write `payload.thinking`, after which `params.extra_body.thinking` collides and overwrites it. The reported live Ollama timeout/stuck-session impact remains unproven from the inspected source.

## Solution Assessment

Unclear because this issue has no patch and the related PRs solve narrower slices. A coordinated request-precedence and disabled-thinking alias fix is safer than a provider-by-provider workaround.
