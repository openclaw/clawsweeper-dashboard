## Summary

Keep open. Current main still has the shared OpenAI-compatible `extra_body` collision path: provider wrappers can write `payload.thinking`, then the later `extra_body` wrapper warns and overwrites the same key. Native Ollama `think` handling exists, and DeepSeek has some disabled-thinking replay cleanup, but the broader collision/noisy-precedence contract and per-model thinking alias gap remain unresolved.

## What This Changes

The issue asks OpenClaw to stop writing conflicting `thinking` fields through provider wrappers and `params.extra_body`, with reliable disabled-thinking behavior across providers such as DeepSeek and Ollama.

## Best Possible Solution

Define one shared OpenAI-compatible request-payload rule: preserve documented `extra_body` override precedence, normalize supported thinking-off aliases before provider wrappers run, avoid noisy warnings for equivalent no-op thinking collisions, and keep provider-specific replay cleanup in the owning provider/plugin where possible.

## Reproduction Assessment

Yes for the confirmed warning path: configure an OpenAI-compatible DeepSeek V4 model where the provider wrapper writes top-level `payload.thinking`, then also configure `params.extra_body.thinking`; current main warns and overwrites that key. No high-confidence reproduction from this read-only review proves the reported live Ollama timeout/stuck-session impact.

## Solution Assessment

Not applicable as a proposed patch because this issue has no diff. The best solution is a coordinated shared payload-precedence and thinking-alias fix, not a provider-by-provider workaround that leaves `extra_body` collision behavior ambiguous.
