## Summary

Keep open. Current main still has the reported `extra_body.thinking` collision path: provider wrappers can write top-level `thinking`, then the generic OpenAI-compatible `extra_body` wrapper warns and overwrites that key afterward. The related DeepSeek issue/PR address a narrower `reasoning_content` replay failure, while the Ollama evidence is mostly native `think` handling, so this is not implemented or superseded.

## What This Changes

The issue requests a framework-level fix for `params.extra_body.thinking` colliding with provider-generated `thinking` request fields across OpenAI-compatible providers, with DeepSeek and Ollama examples.

## Best Possible Solution

Define a canonical request-payload precedence rule for OpenAI-compatible providers, keep explicit overrides predictable, suppress noisy no-op `thinking` collisions where appropriate, and leave native Ollama `think` semantics inside the Ollama plugin unless a focused reproduction proves a shared core gap.

## Reproduction Assessment

Yes for the core collision path: configure a DeepSeek V4 OpenAI-compatible model so the provider wrapper writes top-level `payload.thinking`, and also set `params.extra_body.thinking`; current main will warn and then overwrite that key in the later wrapper. The broader all-provider timeout impact is unclear from source inspection alone.

## Solution Assessment

Unclear because this issue has no proposed patch, and the narrower DeepSeek work does not settle the shared `extra_body` precedence contract. The safer path is maintainer scoping before changing request-shaping behavior across providers.
