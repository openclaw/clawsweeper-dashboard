## Summary

Keep open. Current main still has the reported `extra_body.thinking` collision path: provider wrappers can set top-level `thinking`, then the later generic `extra_body` wrapper warns and overwrites the same key; related DeepSeek work only addresses narrower replay behavior, not the shared precedence and warning contract.

## What This Changes

The issue requests a framework-level fix for `params.extra_body.thinking` colliding with provider-generated `thinking` request fields across OpenAI-compatible providers, with DeepSeek and Ollama examples.

## Best Possible Solution

Define a small documented request-payload rule for OpenAI-compatible `extra_body` collisions: preserve explicit override precedence where intended, avoid noisy no-op `thinking` collisions when final values are equivalent, and keep native Ollama `think` semantics inside the Ollama plugin unless a focused shared-core reproduction proves otherwise.

## Reproduction Assessment

Yes for the core warning path: configure a DeepSeek V4 OpenAI-compatible model so the provider wrapper writes top-level `payload.thinking`, and also set `params.extra_body.thinking`; current main will warn and overwrite that key later. The timeout impact remains unclear without live provider reproduction.

## Solution Assessment

Unclear because this issue has no proposed patch and the related DeepSeek work does not settle the shared `extra_body` precedence contract. The safer path is maintainer scoping before changing request-shaping behavior across providers.
