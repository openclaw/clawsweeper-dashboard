## Summary

Keep open. Current main still has a source-backed `extra_body.thinking` collision path: provider wrappers can set top-level `thinking`, then the generic OpenAI-compatible `extra_body` wrapper logs a collision and overwrites it. The related DeepSeek issue and PRs cover narrower disabled-thinking replay failures, not the broader framework-level precedence/noisy-warning contract reported here.

## What This Changes

The issue requests a framework-level fix for `params.extra_body.thinking` colliding with provider-generated `thinking` payload fields across OpenAI-compatible providers, with DeepSeek and Ollama examples.

## Best Possible Solution

Define a canonical OpenAI-compatible request-payload precedence and warning policy: preserve predictable explicit overrides, avoid noisy no-op `thinking` collisions where appropriate, and keep native Ollama `think` behavior inside the Ollama plugin unless a focused reproduction proves a shared core gap.

## Reproduction Assessment

Yes for the core collision path: configure a provider wrapper such as DeepSeek V4 to write top-level `payload.thinking`, then set `params.extra_body.thinking`; current main will warn and overwrite the key in the later `extra_body` wrapper. The live timeout/stuck-session impact is not fully reproduced from source alone.

## Solution Assessment

Unclear. There is no proposed patch on this issue, and the linked DeepSeek PRs do not settle the shared `extra_body` precedence contract; the safer path is maintainer scoping before automation changes shared request shaping across providers.
