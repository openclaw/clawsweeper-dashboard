## Summary

Keep open. Current main still has the shared OpenAI-compatible extra_body collision path: provider/plugin request wrappers can set top-level payload.thinking first, then the later extra_body wrapper warns and overwrites the same key. Current main has useful DeepSeek replay handling and native Ollama think handling, but it does not resolve the broader noisy collision contract or the per-model params.thinking=false/disabled/none alias gap covered by related open work.

## What This Changes

The issue reports that OpenAI-compatible extra_body handling overwrites an existing request payload thinking key, producing WARN logs and possible provider-specific thinking-mode failures.

## Best Possible Solution

Define a narrow request-payload rule for OpenAI-compatible extra_body collisions: preserve explicit extra_body precedence, normalize supported thinking-off aliases before provider wrappers, avoid noisy no-op thinking collisions when the final value is equivalent, and keep native Ollama think handling inside the Ollama plugin unless a separate shared-core reproduction proves otherwise.

## Reproduction Assessment

Yes for the core warning path: configure an OpenAI-compatible model such as DeepSeek V4 where the provider wrapper writes top-level payload.thinking, then also configure params.extra_body.thinking; current main warns and overwrites the key. No high-confidence reproduction in this review proves the reported live Ollama timeout/stuck-session impact.

## Solution Assessment

Unclear because this issue has no patch and related open PRs address overlapping but narrower pieces. The best fix should coordinate the alias normalization and collision-warning/precedence behavior rather than land a provider-by-provider workaround that leaves the shared contract ambiguous.
