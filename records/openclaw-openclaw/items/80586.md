## Summary

Keep open: current main still has the data-loss path the reporter described, because the Stop/abort handler clears the composer draft before aborting and the running-state Stop button occupies the far-right composer action slot used by Send when idle.

## What This Changes

The issue asks the web UI to prevent accidental Stop taps during a running turn from wiping a composed follow-up prompt.

## Best Possible Solution

Preserve the composer draft when the visible Stop button aborts a run, while keeping intentional typed `/stop` handling clean if maintainers want that command to clear itself.

## Reproduction Assessment

Yes, at source level: with an active abortable run and a non-empty `chatMessage`, the Stop button invokes `handleAbortChat`, which clears the draft before aborting. Existing tests also assert the current clear-on-abort behavior.

## Solution Assessment

Yes: the narrowest maintainable fix is to change the Stop-button abort path so it does not discard the composer, then update the abort/control tests; moving the button can be a follow-up UI polish if needed.
