## Summary

Keep open. Current main still clears the composer draft when Stop is pressed, and https://github.com/openclaw/openclaw/pull/80606 is an open closing PR for the same issue, so this should remain open until that fix or a focused replacement lands.

## What This Changes

The issue asks the web UI to avoid losing typed composer text when Stop is tapped during an active run, either by preserving/recovering the draft or moving Stop away from Send.

## Best Possible Solution

Land a focused web UI fix that preserves or restores the composer draft on Stop, with regression coverage in `app-chat` tests and review of https://github.com/openclaw/openclaw/pull/80606 or a replacement branch.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Stop calls `handleAbortChat`, and current main clears `host.chatMessage` before aborting or queueing the abort.

## Solution Assessment

Yes, preserving the draft in `handleAbortChat` is the narrowest maintainable fix; moving Stop can be a follow-up UX improvement, but it is not required to stop the data loss.
