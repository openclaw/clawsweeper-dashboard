## Summary

Keep open: current main and v2026.6.6 still make `memory_search` feed short-term recall only for eligible short-term paths, while the reporter's durable `MEMORY.md` and non-date `memory/*.md` hits are intentionally filtered out and still have no current-main diagnostic. The linked diagnostic PR is open and unmerged, so this issue should stay open until maintainers review that PR or make the promotion-source policy decision.

## What This Changes

The issue reports that dreaming deep sleep never promotes durable memory because the short-term recall store is populated only by session-corpus daily entries while live `memory_search` hits do not add recall signals for the user's durable memory files.

## Best Possible Solution

Review the linked diagnostic PR, then explicitly settle whether durable memory hits should stay excluded from short-term promotion or get a separate grounded promotion path without lowering the existing quality thresholds.

## Reproduction Assessment

Source-reproducible, but not live-reproduced: current main and v2026.6.6 filter recall tracking to short-term paths and silently return for durable memory-only hits. I did not establish a high-confidence live replay of the reporter's exact MCP `memory_search` path.

## Solution Assessment

Unclear for the full issue: the linked diagnostic PR is the narrowest current implementation path, but maintainers still need to decide whether the intended product behavior is diagnostics-only or a changed promotion-source contract.
