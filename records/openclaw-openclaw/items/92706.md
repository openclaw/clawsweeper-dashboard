## Summary

Keep open: current main and v2026.6.6 still filter short-term recall tracking to date-based daily memory paths and `.dreams/session-corpus`, while this issue now has an open linked PR that proposes a diagnostic fix without resolving the broader promotion-source policy question.

## What This Changes

The issue asks whether live `memory_search` should feed short-term recall and deep promotion, because the reporter's v2026.6.6 recall store only contains session-corpus daily entries and deep promotion returns zero candidates.

## Best Possible Solution

Review the linked diagnostic PR, then make an explicit maintainer decision on the durable-memory versus short-term-source promotion contract before changing thresholds or eligibility.

## Reproduction Assessment

Partly. Source inspection shows why the reported durable paths do not create short-term recall entries and why session-corpus daily entries do not promote, but I did not run a live v2026.6.6 MCP/search replay.

## Solution Assessment

Unclear. The linked diagnostic PR is a good narrow observability step, but deciding whether durable memory files should feed short-term promotion is a maintainer product decision.
