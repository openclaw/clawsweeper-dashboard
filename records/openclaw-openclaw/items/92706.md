## Summary

Keep open: current main and v2026.6.6 already feed live memory_search into recall for eligible daily short-term paths, but durable MEMORY.md and general memory files are intentionally filtered, and an open Fixes PR owns the diagnostic that would answer this report.

## What This Changes

The issue asks whether live memory_search results should increment short-term recall signals for deep dreaming promotion because the reporter’s store only contains session-corpus daily entries with recallCount 0.

## Best Possible Solution

Review and either land or revise https://github.com/openclaw/openclaw/pull/92745 so skipped durable-memory hits are observable, then separately decide whether durable memory paths should ever feed short-term promotion.

## Reproduction Assessment

Yes at source level: current main queues recall tracking from memory_search, but recordShortTermRecalls drops MEMORY.md and general memory files because they fail isShortTermMemoryPath. I did not run the reporter’s multi-day live workspace or MCP path.

## Solution Assessment

Unclear as a final product answer: the linked diagnostic PR is the narrowest near-term fix for observability, but it does not settle whether durable memory paths should become promotion sources.
