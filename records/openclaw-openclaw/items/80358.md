## Summary

Keep this PR open: the Google transport fix is aimed at a real Gemini replay regression, but the latest head still has a source-level replay-cache bug and the current production head lacks fresh real behavior proof.

## What This Changes

The PR updates the Google transport parser and serializer to recover, route-gate, sanitize, and replay Gemini tool-call thought signatures, with focused regression tests.

## Best Possible Solution

Scope recovered Gemini tool-call signatures to the active function-calling turn, fall back to the documented skip validator for unsigned transferred calls, add the missing cross-user-turn regression, then refresh current-head remote or live proof.

## Reproduction Assessment

Yes, at source level. The linked issue has a direct Gemini replay matrix for the 400, and current main still has parser/serializer paths that can lose or fail to recover required Gemini tool-call signatures.

## Solution Assessment

No, not yet. The owner-boundary fix direction is right, but replay recovery should be scoped to the current Gemini turn and the latest head needs real behavior proof after that correction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear replay signatures when a new user turn starts:** `extensions/google/transport-stream.ts:592-593`
  - body: At PR head the replay cache is initialized once for the whole converted history and updated after every assistant message, but it is never cleared when a normal user message starts a new Gemini turn. If a later turn reuses the same tool-call id/name/args and its stored block is unsigned, the older turn's signature remains available and is serialized instead of the skip-validator fallback. Gemini requires signatures to be returned in the exact part where received, so scope this cache to the active turn and add a regression with a separating user text message.
  - confidence: 0.84
