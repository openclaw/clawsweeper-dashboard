## Summary

Keep open: this is a focused, plausible Google transport bug fix, but the latest PR head still has a blocking replay-precedence bug and lacks current-head real behavior proof after the final production change.

## What This Changes

The PR updates native Google transport parsing and replay so Gemini tool-call thought signatures are recovered from sibling SSE parts, sanitized, route-gated, keyed by tool-call shape, and covered by regression tests.

## Best Possible Solution

Prefer the current block's sanitized same-route signature before any replay-cache value, add a regression for repeated identical tool-call keys with different signatures, then refresh current-head real behavior proof.

## Reproduction Assessment

Yes. The underlying bug has direct Gemini API evidence in the linked issue, and the remaining PR bug is source-reproducible from the all-history replay collector plus `retainThoughtSignature` preferring the replay-cache value.

## Solution Assessment

No, not as-is. The owner-boundary fix direction is right, but the serializer must preserve a valid stored same-route signature before consulting recovered replay signatures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Prefer block-owned signatures before replay cache:** `extensions/google/transport-stream.ts:575`
  - body: The replay cache is built from all assistant history before serialization, and `Map#set` keeps the later signature for a repeated tool-call key. Because `retainThoughtSignature(ownSignature, replayedThoughtSignature)` prefers the cache hit, an earlier same-route block that already has its own valid signature can be serialized with a later repeated call's signature; Gemini requires returning the signature in the exact part where it was received. Prefer `ownSignature ?? replayedThoughtSignature` here and cover the repeated same id/name/args case with different signatures.
  - confidence: 0.82
