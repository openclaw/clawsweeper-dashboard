## Summary

Keep open. Current main and v2026.6.5 still expose Telegram sender id/name/username to agent context but not the sender bot classification, and the related open bot-targeting PR is distinct work about which bot a message targets.

## What This Changes

The issue requests exposing Telegram sender metadata such as sender id, username, and is_bot to the agent-facing context.

## Best Possible Solution

Add one optional transport-neutral sender-bot fact to the shared inbound context, populate it from Telegram msg.from.is_bot, and render it as untrusted agent metadata without changing requireMention, authorization, routing, or raw Telegram object exposure.

## Reproduction Assessment

Yes, at source level. Telegram receives msg.from.is_bot and the shared sender contract can carry isBot, but current main and v2026.6.5 do not render that fact into agent-facing context.

## Solution Assessment

Yes. Reusing the existing SenderFacts.isBot path is the narrowest maintainable solution; exposing raw msg.from or changing mention/routing policy would be broader than the issue needs.
