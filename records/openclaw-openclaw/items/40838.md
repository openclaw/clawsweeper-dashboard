## Summary

Keep open. Current main and v2026.5.19 still have the Telegram sender identity and the shared `SenderFacts.isBot` contract, but the agent-visible Telegram context and rendered inbound metadata do not expose whether the sender is a bot.

## What This Changes

Expose Telegram sender bot/source metadata, especially `msg.from.is_bot`, to agent-visible context for message classification.

## Best Possible Solution

Add an optional sender-bot boolean through Telegram `msg.from?.is_bot`, `SenderFacts`, finalized `MsgContext`, and inbound metadata with focused regression coverage, while leaving broader bot-to-bot routing policy to the adjacent Telegram issues.

## Reproduction Assessment

Yes for the source-level gap: grammY exposes `message.from.is_bot`, `SenderFacts.isBot` already exists, and current main does not carry it into agent-visible Telegram context or sender metadata. I did not run a live Telegram turn because the missing plumbing is clear in source.

## Solution Assessment

Yes. The maintainable path is to plumb the existing sender fact into finalized context and rendered metadata, not to expose raw Telegram `msg.from` objects or solve the broader bot-to-bot policy in this issue.
