## Summary

Keep open. Current main still exposes Telegram sender name, id, and username to the agent-visible context, but it does not expose `msg.from.is_bot` or any equivalent sender bot/source classification field; the related voice-message metadata issue is adjacent and does not supersede this request.

## What This Changes

Expose Telegram sender/source metadata, especially `msg.from.is_bot`, to the agent context so agents can route bot and user messages differently.

## Best Possible Solution

Add a narrow Telegram-owned sender classification field populated from `msg.from.is_bot`, preserve the existing sender identity fields, render the field as untrusted metadata for agent routing, and document it as provider-supplied metadata rather than trusted authorization state.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence verification path: construct Telegram inbound messages with `from.is_bot` true and false, then inspect the reply payload and rendered inbound metadata. Current code and tests show only `SenderName`, `SenderId`, and `SenderUsername` are surfaced.

## Solution Assessment

Yes for a narrow bounded field; it is the maintainable way to satisfy the request without dumping raw Telegram user objects into shared context. A broader raw `msg.from` export would be a less safe and less stable public contract.
