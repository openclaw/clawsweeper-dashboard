## Summary

Keep open. Current main and latest v2026.5.22 expose Telegram sender id/name/username to agent context but still omit Telegram `msg.from.is_bot`; the existing shared `SenderFacts.isBot` contract makes this a bounded additive context-plumbing task.

## What This Changes

Expose Telegram sender metadata, especially `msg.from.is_bot`, to agent-visible context so agents can classify bot versus human senders.

## Best Possible Solution

Add an optional sender bot boolean derived from `msg.from?.is_bot` through `SenderFacts.isBot`, `MsgContext`/finalized context, and untrusted inbound sender metadata without changing Telegram bot-routing policy.

## Reproduction Assessment

Yes for source-level reproduction: grammY exposes `message.from.is_bot`, while current Telegram assembly, core context mapping, `MsgContext`, and rendered metadata do not carry that boolean. I did not run a live Telegram turn because the missing plumbing is clear in source.

## Solution Assessment

Yes. The maintainable path is additive plumbing through the existing `SenderFacts.isBot` contract and untrusted prompt metadata, not a broader `requireMention` redesign or raw Telegram object exposure.
