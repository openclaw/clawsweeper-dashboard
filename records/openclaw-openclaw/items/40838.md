## Summary

Keep open. Current main already exposes Telegram sender id, name, and username to the agent context, but it does not expose the requested bot-origin flag, so the central feature request remains open and should be handled deliberately with the broader inbound metadata contract.

## What This Changes

Expose additional Telegram sender/message metadata, especially `msg.from.is_bot`, in agent-visible inbound context.

## Best Possible Solution

Add a documented, generic inbound metadata extension for sender classification and message type, then populate Telegram from `msg.from.is_bot` with focused Telegram and shared prompt-renderer coverage.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection confirms current main lacks the requested bot-origin context field even though it already carries sender id/name/username.

## Solution Assessment

Unclear. Exposing the raw Telegram field directly would work narrowly, but the more maintainable fix is a generic channel context field rendered through the existing trusted/untrusted inbound metadata paths.
