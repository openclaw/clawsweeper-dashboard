## Summary

Keep open. Current main and the latest release still carry Telegram sender id/name/username into agent context, but not the Telegram `message.from.is_bot` fact; the existing shared `SenderFacts.isBot` slot makes this a narrow additive context-plumbing task.

## What This Changes

Expose Telegram sender bot classification and related sender facts in agent-visible inbound context so agents can classify Telegram message sources.

## Best Possible Solution

Add an optional sender bot boolean derived from `msg.from?.is_bot` through `SenderFacts`, finalized `MsgContext`, and untrusted inbound sender metadata, without changing Telegram bot-routing policy or exposing raw Telegram message objects.

## Reproduction Assessment

Yes for source-level reproduction: grammY types expose `message.from.is_bot`, while current Telegram assembly, core context mapping, `MsgContext`, and rendered metadata do not carry that boolean. I did not run a live Telegram turn because this review is read-only and the missing plumbing is clear in source.

## Solution Assessment

Yes. The narrow maintainable path is additive plumbing through the existing `SenderFacts.isBot` contract and untrusted prompt metadata, not a broader `requireMention` redesign or raw Telegram object exposure.
