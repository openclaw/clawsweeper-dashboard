## Summary

Keep open. Current main still exposes Telegram sender name/id/username but does not surface `msg.from.is_bot` or an equivalent sender bot classification to the agent, and related #50482 tracks message-type metadata rather than this sender-classification gap.

## What This Changes

Request to expose Telegram sender fields, especially bot status from `msg.from.is_bot`, to agent context for message classification and routing.

## Best Possible Solution

Add a bounded sender bot boolean sourced from Telegram `msg.from.is_bot`, carry it through the shared context path, render it as sender metadata, and avoid exporting the raw Telegram user object.

## Reproduction Assessment

Not applicable as a runtime bug; this is an additive feature request. Source inspection gives a high-confidence verification path: a Telegram message with `from.is_bot` would currently produce sender name/id/username but no bot classification field in finalized context or prompt metadata.

## Solution Assessment

Yes, the narrow maintainable direction is a bounded boolean such as `SenderIsBot`/`sender_is_bot`. Exporting raw `msg.from` would unnecessarily broaden the prompt/API contract and mix provider-specific untrusted data into agent context.
