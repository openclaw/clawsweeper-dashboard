## Summary

Keep open. Current main still has access to Telegram `msg.from` and surfaces sender name/id/username, but it does not expose `msg.from.is_bot` or an equivalent sender bot classification field to the agent context; related #50482 is adjacent message-type metadata work, not a replacement for sender classification.

## What This Changes

Expose key Telegram `msg.from` sender metadata, especially whether the sender is a bot, to the agent context for routing and classification.

## Best Possible Solution

Add a bounded sender bot flag sourced from Telegram `msg.from.is_bot`, surface it through the shared context/prompt metadata as untrusted provider metadata, and avoid exporting the raw Telegram user object.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature gap. Source inspection gives a high-confidence verification path: create a Telegram inbound message with `from.is_bot` set and confirm current main only exposes sender name/id/username.

## Solution Assessment

Yes, a narrow boolean/classification field is the best direction. Exporting raw `msg.from` would broaden the prompt/API contract and mix untrusted provider data into the agent context unnecessarily.
