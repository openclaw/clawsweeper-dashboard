## Summary

Keep open. Current main still has the Telegram sender source data and a shared `SenderFacts.isBot` field, but the Telegram turn builder, finalized message context, and inbound prompt metadata expose sender name/id/username without the sender bot classification; the broader Telegram bot-to-bot design in https://github.com/openclaw/openclaw/issues/79077 is adjacent, not a replacement for this focused metadata gap.

## What This Changes

Request to expose Telegram sender fields such as `msg.from.is_bot`, sender id, and username to agent context so agents can classify inbound messages by source.

## Best Possible Solution

Add a bounded sender-bot boolean sourced from Telegram `msg.from?.is_bot`, carry it through `SenderFacts` and finalized agent context, render it in inbound sender metadata, and document it without exposing raw Telegram user objects.

## Reproduction Assessment

Not applicable as a runtime bug; this is an additive metadata request. Source verification is high-confidence: Telegram has `from.is_bot`, but current main only carries sender name/id/username into the agent context and inbound metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to plumb a boolean such as `SenderIsBot`/`sender.is_bot`, while leaving broader bot-to-bot policy, loop prevention, and peer routing to https://github.com/openclaw/openclaw/issues/79077.
