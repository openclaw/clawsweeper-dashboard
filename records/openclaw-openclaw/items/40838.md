## Summary

Keep open. Current `main` still has Telegram sender access and surfaces sender name/id/username, but it does not expose `msg.from.is_bot` or any equivalent sender bot classification to agent context; related #50482 tracks message-type metadata and does not supersede this sender-classification request.

## What This Changes

The issue requests exposing Telegram sender metadata such as `msg.from.is_bot` to the agent context so agents can classify sender/source type.

## Best Possible Solution

Add a bounded sender bot boolean sourced from Telegram `msg.from.is_bot`, carry it through the shared context path, render it as untrusted sender metadata, and avoid exporting the raw Telegram user object.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature gap. Source inspection gives a high-confidence verification path: create a Telegram inbound message with `from.is_bot` set and confirm current `main` only exposes sender name/id/username.

## Solution Assessment

Yes, the narrow maintainable direction is to expose a bounded bot/classification boolean. Exporting raw `msg.from` would broaden the prompt/API contract and mix provider-specific untrusted data into agent context unnecessarily.
