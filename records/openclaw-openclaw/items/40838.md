## Summary

Keep open. Current main still has the Telegram sender source object and a shared `SenderFacts.isBot` contract, but the Telegram turn builder, finalized message context, and rendered inbound metadata expose only sender id/name/username without bot classification; the broader Telegram bot-to-bot design in https://github.com/openclaw/openclaw/issues/79077 and open bot-targeting PR https://github.com/openclaw/openclaw/pull/72510 are adjacent, not replacements for this focused metadata gap.

## What This Changes

The issue requests exposing Telegram sender fields such as `msg.from.is_bot`, sender id, and username in the agent context for metadata-based routing.

## Best Possible Solution

Expose a bounded boolean such as `SenderIsBot`/`sender.is_bot` sourced from Telegram `msg.from?.is_bot`, carry it through `SenderFacts`, `MsgContext`, and inbound metadata, and leave broader bot-to-bot routing policy to https://github.com/openclaw/openclaw/issues/79077.

## Reproduction Assessment

Not applicable as a runtime bug; this is an additive metadata request. Source inspection gives high-confidence verification that Telegram has the sender object but current main only carries id/name/username into agent-visible context.

## Solution Assessment

Yes. The narrow maintainable solution is metadata plumbing through the existing shared sender facts contract, not a raw Telegram payload or the broader bot-to-bot policy requested in https://github.com/openclaw/openclaw/issues/79077.
