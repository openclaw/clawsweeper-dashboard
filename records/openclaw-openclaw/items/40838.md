## Summary

Keep open. Current main and v2026.5.20 still carry Telegram sender id/name/username into agent-visible context, but not the Telegram `from.is_bot` sender classification; the shared sender contract already has an `isBot` slot, so the remaining work is narrow additive plumbing rather than a broad routing redesign. The related open PR https://github.com/openclaw/openclaw/pull/72510 addresses group bot-targeting history signals, not this current-sender bot flag.

## What This Changes

Expose Telegram sender bot classification to the agent-visible inbound context and rendered metadata.

## Best Possible Solution

Add an optional sender bot boolean derived from Telegram `msg.from?.is_bot` through `SenderFacts`, finalized `MsgContext`, and inbound metadata, with focused tests and no broader bot-to-bot policy changes.

Yes for the source-level gap: grammY exposes `message.from.is_bot`, current Telegram code reads the sender but does not pass that boolean into finalized context or rendered metadata. I did not run a live Telegram turn because the missing plumbing is clear in source.

## Solution Assessment

Yes. The maintainable fix is additive context plumbing from the existing Telegram field through the existing `SenderFacts` contract, not exposing raw `msg.from` or solving the broader bot-to-bot routing policy here.
