## Summary

Keep open. Current main still exposes Telegram sender name/id/username to the agent, but it does not surface `msg.from.is_bot` or an equivalent sender bot classification; the related voice-message metadata issue and adjacent bot-targeting PRs do not replace this sender-classification request.

## What This Changes

The issue requests exposing Telegram sender metadata such as `msg.from.is_bot` to agent context so agents can classify bot versus human senders.

## Best Possible Solution

Add a bounded, documented sender bot flag sourced from Telegram `msg.from.is_bot` through the existing sender/context metadata path, with regression tests and without exposing the raw Telegram user object.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: an inbound Telegram message can contain `from.is_bot`, but current main only exposes sender name/id/username to the agent context.

## Solution Assessment

Yes, the narrow maintainable direction is a boolean/classification field in the existing sender metadata path. Exporting raw `msg.from` would broaden the prompt/API contract and mix provider-specific untrusted data into agent context unnecessarily.
