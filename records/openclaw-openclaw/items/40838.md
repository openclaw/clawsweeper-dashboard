## Summary

Keep open. Current main still has access to Telegram sender identity and a shared `SenderFacts.isBot` field, but the finalized agent context and rendered inbound metadata only expose sender name/id/username, not sender bot status. The related voice-message request at https://github.com/openclaw/openclaw/issues/50482 is adjacent metadata work, not a substitute for this sender-classification gap.

## What This Changes

Expose Telegram sender metadata such as `msg.from.is_bot` in agent-visible context so agents can classify message sources.

## Best Possible Solution

Add a bounded sender bot boolean sourced from Telegram `msg.from.is_bot`, carry it through shared context, render it as sender metadata, and avoid exposing the raw Telegram user object.

## Reproduction Assessment

Not applicable as a runtime bug; this is an additive metadata request. Source inspection gives a high-confidence verification path: a Telegram message with `from.is_bot` currently reaches code that exports sender name/id/username but no bot classification field.

## Solution Assessment

Yes. The narrow maintainable path is a bounded boolean such as `SenderIsBot`/`sender_is_bot`; exporting raw `msg.from` would broaden the prompt/API contract and mix provider-specific untrusted data into agent context.
