## Summary

Keep open. Current main still has Telegram sender data and a shared `SenderFacts.isBot` contract, but the Telegram inbound payload, finalized `MsgContext`, and rendered sender metadata only expose sender id/name/username, so the agent still cannot classify bot-sourced Telegram messages from context.

## What This Changes

Expose Telegram sender metadata such as `msg.from.is_bot` to agent-visible inbound context so agents can distinguish bot and user message sources.

## Best Possible Solution

Expose a bounded sender bot boolean from `msg.from?.is_bot` through `SenderFacts`, `MsgContext`, and inbound metadata, while leaving broader Telegram bot-to-bot routing policy to https://github.com/openclaw/openclaw/issues/79077.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is an additive metadata request. Source inspection gives a high-confidence verification path: Telegram has `msg.from` data and the shared sender contract has `isBot`, but current main only carries id/name/username into agent-visible context.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable solution is metadata plumbing through the existing shared sender facts contract, not raw Telegram payload exposure or the broader bot-to-bot policy requested in https://github.com/openclaw/openclaw/issues/79077.
