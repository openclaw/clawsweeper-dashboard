## Summary

Keep open. Current main still has the Telegram sender data and a shared `SenderFacts.isBot` contract, but the Telegram inbound context, finalized `MsgContext`, and rendered agent metadata only expose sender id/name/username/tag fields, so agents still cannot classify bot-sourced Telegram messages from context.

## What This Changes

Expose Telegram sender fields such as `msg.from.is_bot` to agent-visible inbound context so agents can classify message source/type without ad hoc routing logic.

## Best Possible Solution

Expose a bounded sender bot boolean from `msg.from?.is_bot` through `SenderFacts`, `MsgContext`, and inbound metadata, with docs/tests, while leaving broader Telegram bot-to-bot policy to https://github.com/openclaw/openclaw/issues/79077.

## Reproduction Assessment

Not applicable as a runtime bug; this is an additive metadata request. Source inspection gives a high-confidence verification path: Telegram has sender data available in the inbound path and the shared sender contract has `isBot`, but current main only carries id/name/username/tag into agent-visible context.

## Solution Assessment

Yes. The narrow maintainable solution is metadata plumbing through the existing shared sender facts contract, not raw Telegram payload exposure or the broader bot-to-bot policy work.
