## Summary

Keep open. Current main and v2026.6.5 still omit the Telegram sender’s bot classification from agent-facing context, while the shared channel-turn contract already provides the appropriate transport-neutral fact; the related open bot-targeting PR addresses which bot a message targets, not whether the current sender is a bot.

## What This Changes

Expose Telegram sender metadata such as bot identity and sender fields in the agent’s inbound context for message classification.

## Best Possible Solution

Add one optional transport-neutral bot-sender context fact backed by `SenderFacts.isBot`, populate it from Telegram, and render it as untrusted metadata with focused tests and no mention, authorization, or routing-policy changes.

## Reproduction Assessment

Yes, at source level. Telegram receives `msg.from.is_bot`, the shared turn contract supports `SenderFacts.isBot`, and both current main and v2026.6.5 omit that boolean from agent-facing context.

## Solution Assessment

Yes. Reusing the shared sender-facts seam is the narrowest maintainable direction; exposing raw Telegram objects or changing `requireMention` and routing behavior would unnecessarily mix transport data with product policy.
