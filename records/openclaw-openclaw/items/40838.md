## Summary

Keep open. Current main still carries Telegram sender name/id/username into agent-visible context, and the shared turn contract has `SenderFacts.isBot`, but the Telegram path does not export current-sender bot status into `MsgContext` or rendered inbound metadata. The related voice metadata issue at https://github.com/openclaw/openclaw/issues/50482 and open bot-targeting/debounce PRs are adjacent, not substitutes for this sender-classification request.

## What This Changes

The issue requests exposing Telegram sender fields, especially bot status, to agent context so agents can classify inbound messages by source.

## Best Possible Solution

Add a bounded sender bot boolean sourced from Telegram `msg.from.is_bot`, carry it through shared context, and render it as untrusted sender metadata without exposing raw Telegram user objects.

## Reproduction Assessment

Not applicable as a runtime bug; this is an additive context metadata request. Source inspection gives a high-confidence verification path: Telegram has `from.is_bot`, but current main exports only sender name/id/username into agent-visible context.

## Solution Assessment

Yes in direction: a narrow bounded boolean is the maintainable fix. Maintainers should settle the public field name and placement before implementation because raw `msg.from` exposure would be too provider-specific and broad.
