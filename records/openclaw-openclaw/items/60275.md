## Summary

Keep open. Current main has per-agent heartbeat scheduling, explicit session routing, targeted wake dispatch, isolated heartbeat sessions, and heartbeat-delivered commitments, but it still does not provide durable native per-session heartbeat entries with independent schedules, enable/disable state, concurrency policy, run history, or operator CRUD/list surfaces.

## What This Changes

Requested native session-scoped heartbeat entries so one agent can manage multiple durable per-session heartbeat schedules with predictable routing, safety boundaries, and operator visibility.

## Best Possible Solution

Keep this as a core automation design issue: choose whether heartbeat entries reuse cron's durable scheduler/run-history substrate or get a separate versioned store, then add session-bound entry schema, explicit concurrency semantics, restart-safe state, operator controls, and docs separating scheduling from gating, commitments, and cron.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current schema, runner state, wake dispatch, RPC/CLI methods, commitments, and docs all show no durable per-session heartbeat-entry model.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native heartbeat entries are plausible, but the maintainable path needs a design decision that composes with cron persistence, current targeted wakes, session routing, and inferred commitments before implementation.
