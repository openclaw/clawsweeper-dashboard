## Summary

Keep open. Current main has session-aware heartbeat routing, targeted wakes, isolated heartbeat sessions, and heartbeat-delivered commitments, but it still lacks native durable per-session heartbeat entries with independent schedules, enablement state, concurrency policy, run history, and operator CRUD/list surfaces.

## What This Changes

Request native session-scoped heartbeat entries so one agent can manage multiple durable per-session heartbeat schedules with predictable routing, safety boundaries, and operator visibility.

## Best Possible Solution

Keep this as a core automation design issue. A complete solution should decide whether heartbeat entries reuse cron's durable scheduler/run-history substrate or introduce a separate versioned store, then add session-bound entry schema, explicit skip/queue/parallel semantics, restart-safe enabled state, recent-run observability, Gateway RPC/CLI or UI controls, and docs that separate scheduling from gating, commitments, and cron.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current schema, runner state, wake dispatch, RPC methods, and docs all show one heartbeat config/run path rather than durable per-session heartbeat entries.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native heartbeat entries are a plausible solution, but the safest path is a design decision that composes with cron persistence, current targeted wakes, session routing, and inferred commitments before implementation.
