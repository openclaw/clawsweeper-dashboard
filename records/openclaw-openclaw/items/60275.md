## Summary

Keep open. Current main still exposes heartbeat as one per-agent scheduler/config object with optional session targeting; it does not provide durable per-session heartbeat entries, per-entry enabled state, or first-class operator run status.

## What This Changes

Request native, durable heartbeat entries scoped to explicit sessions so one agent can manage multiple scheduled heartbeat targets predictably.

## Best Possible Solution

Define a durable core heartbeat-entry contract that composes with heartbeat.session, isolatedSession, wake events, task batching, receipts/status, and operator APIs before implementing the runtime surface.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence verification path: current main exposes one heartbeat object and per-agent scheduler state, not native durable session-scoped heartbeat entries.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native entries are plausible, but the design must reconcile the current per-agent scheduler contract with explicit multi-session entries, persistence, concurrency, migration, and operator controls.
