Keep open: current main and v2026.6.10 still model heartbeat as one effective heartbeat configuration and scheduler state per agent, while the requested native per-session heartbeat entries remain a product, storage, and operator-surface decision.

## What This Changes

Add native session-scoped heartbeat entries so one agent can schedule multiple explicit session-bound heartbeat runs with persistence and operator visibility.

## Best Possible Solution

Maintainers should choose one canonical product path: extend heartbeat with persisted session-bound entries and operator surfaces, or route multi-session proactive scheduling through cron/plugin APIs with clear docs and migration guidance.

## Reproduction Assessment

## Solution Assessment

Unclear: native heartbeat entries are plausible, but cron already provides persisted session-targeted scheduled work, so maintainers should decide the canonical API and product boundary first.
