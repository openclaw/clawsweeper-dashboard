## Summary

Keep open: current main and v2026.6.10 still model heartbeat as one effective schedule/config per agent with a single optional session override; cron can schedule explicit sessions, but that is not the requested native multi-entry heartbeat surface.

## What This Changes

The issue requests native per-session heartbeat entries so one agent can schedule multiple explicit heartbeat sessions with persistence and operator visibility.

## Best Possible Solution

Keep this open for maintainer product/design review of native heartbeat-entry state and APIs, explicitly deciding whether entries live in config, session metadata, or a scheduler store and how they coexist with cron.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new native heartbeat-entry concept. Source inspection does give high-confidence proof that current main lacks a multi-entry per-session heartbeat scheduler.

## Solution Assessment

Unclear until maintainers choose the product shape. Native entries may be the right solution, but it should not be implemented as a narrow patch to the single `heartbeat.session` override or as an uncoordinated cron duplicate.
