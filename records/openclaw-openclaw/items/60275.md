## Summary

Keep open. Current main has per-agent heartbeat scheduling, explicit session routing, targeted wakes, isolated heartbeat sessions, and cron-backed durable scheduling, but it still does not provide durable native per-session heartbeat entries with independent schedules, enable/disable state, concurrency policy, run history, or operator CRUD/list surfaces.

## What This Changes

The issue requests durable session-scoped heartbeat entries so one agent can own multiple independently scheduled heartbeat targets.

## Best Possible Solution

Keep this as a core automation design issue: decide whether heartbeat entries reuse cron's durable scheduler/run-history substrate or need a separate versioned store, then add session-bound entry schema, explicit concurrency semantics, restart-safe state, operator controls, tests, and docs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, and current main shows no durable per-session heartbeat-entry model or operator CRUD surface.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native heartbeat entries are plausible, but the maintainable path needs a design decision that composes with cron persistence, targeted wakes, current session routing, and inferred commitments before implementation.
