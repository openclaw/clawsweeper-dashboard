## Summary

Keep open. Current main still models heartbeat as one per-agent scheduler/config unit with an optional session override, and I did not find a merged implementation or canonical duplicate for durable session-scoped heartbeat entries with operator controls.

## What This Changes

The issue requests native session-scoped heartbeat entries so one agent can maintain multiple explicit heartbeat schedules with safe routing, persistence, and operator visibility.

## Best Possible Solution

Define a core durable heartbeat-entry contract that composes with heartbeat.session, isolatedSession, task batching, commitments, cron wakes, delivery routing, and operator APIs before implementation.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence verification path: current main exposes one heartbeat object and one scheduler state per agent, not native durable session-scoped heartbeat entries.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. Native entries are plausible, but the design must first settle state ownership, concurrency, routing, migration, and operator controls.
