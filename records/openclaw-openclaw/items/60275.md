## Summary

Keep this open: current main still models heartbeat as one effective heartbeat config/schedule per agent, while cron already covers some persisted session-targeted scheduled work but is not the native heartbeat-entry surface requested here. The request is therefore still a real product/design item, not an implemented bug fix or a duplicate of the related heartbeat observability report.

## What This Changes

Requested native per-session heartbeat scheduling so one agent can own multiple explicit, persistent, observable heartbeat entries bound to individual sessions.

## Best Possible Solution

Have maintainers decide the product boundary first, then implement one canonical persisted heartbeat-entry model or explicitly document cron/plugin APIs as the supported path without creating competing scheduler surfaces.

## Solution Assessment

Unclear as an implementation direction: native heartbeat entries are plausible, but cron already provides persisted session-targeted scheduling, so maintainers should first choose the canonical product/API boundary.
