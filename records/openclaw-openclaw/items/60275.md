## Summary

Keep open: current main and v2026.6.8 still model heartbeat as one effective config and scheduler state per agent; cron is adjacent but does not provide the requested native heartbeat-entry control plane.

## What This Changes

Request native session-scoped heartbeat entries so one agent can manage multiple explicit scheduled heartbeat targets with persisted state and operator visibility.

## Best Possible Solution

Maintainers should choose one canonical product path: extend heartbeat with persisted session-bound entries and operator surfaces, or route multi-session proactive scheduling through cron/plugin APIs with docs and migration guidance.

## Reproduction Assessment

Yes for source-level feature-gap proof: current main lacks a multi-entry heartbeat model, though this is not a runtime bug reproduction.

## Solution Assessment

Unclear: native heartbeat entries are plausible, but cron already provides persisted session-targeted scheduling, so maintainers should decide the canonical API and product boundary first.
