## Summary

Keep open: current main and v2026.6.10 still model heartbeat as one effective config and scheduler state per agent; the requested native per-session heartbeat entries remain a product, storage, and control-plane decision.

## What This Changes

The issue requests native, persistent, session-scoped heartbeat entries so one agent can schedule multiple explicit heartbeat sessions with observable run state.

## Best Possible Solution

Maintainers should choose one canonical product path: extend heartbeat with persisted session-bound entries and operator surfaces, or route multi-session proactive scheduling through cron/plugin APIs with docs and migration guidance.

## Reproduction Assessment

Yes for a source-level feature-gap proof: current main models one effective heartbeat schedule per agent rather than multiple native session entries. This is not a runtime bug reproduction.

## Solution Assessment

Unclear: native heartbeat entries are plausible, but cron already provides persisted session-targeted scheduled work, so maintainers should decide the canonical API and product boundary first.
