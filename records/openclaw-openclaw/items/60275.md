## Summary

Keep open. Current main still has one heartbeat configuration and scheduler record per agent, plus explicit session routing and ephemeral targeted wakes, but it does not provide durable native per-session heartbeat entries with independent schedules, per-entry enablement, run history, concurrency semantics, or operator CRUD/list surfaces.

## What This Changes

Request native session-scoped heartbeat entries so one agent can schedule multiple explicit sessions with persisted state and operator visibility.

## Best Possible Solution

Define a core heartbeat-entry model with durable session-bound schedules, per-entry enabled/run state, documented concurrency policy, operator CRUD/list/status surfaces, and clear separation from cron, commitments, and session-local gating.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows routing and wake support but no durable per-session heartbeat-entry model.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native entries are plausible, but the maintainable path must compose with the current heartbeat runner, cron persistence, commitments, and Gateway operator APIs.
