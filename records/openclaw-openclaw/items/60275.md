## Summary

Keep open: current main and the latest release still model heartbeat as one effective heartbeat config and scheduler state per agent; targeted wakes and cron are adjacent, but they do not provide the requested native persisted session-scoped heartbeat entries and operator control plane.

## What This Changes

Request native session-scoped heartbeat entries so one agent can own multiple persisted heartbeat schedules bound to explicit sessions.

## Best Possible Solution

Choose one canonical design: either extend heartbeat with persisted session-bound entries and operator RPC/CLI surfaces, or document cron/plugin APIs as the supported multi-session scheduler path.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request; source inspection gives high-confidence proof that current main lacks a multi-entry heartbeat model.

## Solution Assessment

Unclear as an implementation direction: native heartbeat entries are plausible, but cron already provides persisted session-targeted scheduling, so maintainers should choose the canonical product/API boundary first.
