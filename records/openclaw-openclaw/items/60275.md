## Summary

Keep open. Current main still has per-agent heartbeat scheduling, explicit session routing, targeted wakes, and heartbeat task mode, but not durable native per-session heartbeat entries with independent schedules, state, run history, or CRUD/list operator surfaces.

## What This Changes

The issue requests native session-scoped heartbeat entries so one agent can own multiple durable, explicit heartbeat schedules bound to different sessions.

## Best Possible Solution

Keep this as a core automation design item and define a durable session-bound heartbeat-entry model with explicit scheduling semantics, persisted enabled/run state, operator CRUD/list surfaces, and docs separating scheduling from session-local gating, cron, and commitments.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main has no durable per-session heartbeat-entry model or operator CRUD/list surface.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native heartbeat entries are plausible, but the maintainable path needs design alignment with cron persistence, targeted wakes, session routing, commitments, and control-plane APIs before implementation.
