## Summary

Keep open. Current main still exposes heartbeat as one per-agent config/scheduler unit with optional session routing, while the request is for durable session-scoped heartbeat entries with independent schedules, state, observability, and operator control surfaces.

## What This Changes

Request native heartbeat entries so one agent can schedule and manage multiple explicit session-bound heartbeat runs.

## Best Possible Solution

Define a core durable heartbeat-entry contract that composes with heartbeat.session, isolatedSession, task batching, commitments, cron, and operator APIs before implementation.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current main exposes one heartbeat object and per-agent scheduler state, not native durable heartbeat entries.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native entries are plausible, but implementing them safely requires decisions about storage, concurrency, gating versus scheduling, migration, and control-plane shape.
