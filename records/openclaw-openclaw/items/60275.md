## Summary

Keep open. Current main has per-agent heartbeat config, one configured or forced heartbeat session, targeted in-memory wakes, isolated heartbeat runs, commitments, and adjacent multi-agent scheduler work, but it still lacks native durable per-session heartbeat entries with independent schedules, per-entry state, run history, concurrency semantics, and operator CRUD/list surfaces.

## What This Changes

Request native durable heartbeat entries scoped to explicit sessions, with per-entry scheduling, state, run visibility, and operator control.

## Best Possible Solution

Keep this as a core automation design issue and define a durable session-bound heartbeat-entry model that composes with cron, commitments, targeted wakes, session routing, and existing heartbeat docs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source/docs/test inspection: current main exposes one heartbeat config object and targeted wake routing, but no durable per-session heartbeat-entry collection or CRUD/run-history surface.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native heartbeat entries are plausible, but this needs a design decision that composes with cron persistence, commitments, wake coalescing, current session routing, and operator APIs before implementation.
