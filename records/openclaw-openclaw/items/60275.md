## Summary

Keep open. Current main supports one heartbeat config object per agent, explicit/forced heartbeat session routing, isolated heartbeat runs, and in-memory targeted wakes, but it still lacks durable session-scoped heartbeat entries with independent schedules, per-entry state, recent-run history, concurrency semantics, and operator CRUD/list surfaces. The related open heartbeat PR addresses adjacent multi-agent runner bugs, not this native per-session scheduling model.

## What This Changes

The issue requests native session-scoped heartbeat entries so one agent can manage multiple durable heartbeat schedules with explicit routing, state, and operator visibility.

## Best Possible Solution

Keep this as a core heartbeat scheduler design issue and define a durable session-bound heartbeat-entry model that composes with existing heartbeat.session, isolatedSession, targeted wakes, commitments, cron, and operator APIs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current main exposes one heartbeat object and per-agent scheduler state, not durable per-session heartbeat-entry enrollment.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native heartbeat entries are plausible, but this needs explicit decisions for persistence, concurrency, gating versus scheduling, migration, and control-plane shape before implementation.
