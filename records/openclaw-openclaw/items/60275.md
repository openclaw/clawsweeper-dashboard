## Summary

Keep open. Current main still models heartbeat as one per-agent scheduler/config object with optional session routing, not durable session-scoped heartbeat entries with independent schedules, persistence, and operator controls.

## What This Changes

Request native session-scoped heartbeat entries so one agent can schedule multiple durable per-session heartbeat runs with predictable routing and control-plane visibility.

## Best Possible Solution

Define a durable core heartbeat-entry contract, storage location, concurrency policy, pause/resume semantics, migration path, and operator API before implementing the scheduler changes.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main gives a high-confidence verification path: current heartbeat exposes one config object and one per-agent scheduler state, not native durable session-scoped entries.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native entries are plausible, but the best implementation depends on storage, concurrency, routing, migration, and operator-control decisions.
