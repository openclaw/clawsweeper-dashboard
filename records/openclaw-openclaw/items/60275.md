## Summary

Keep open. Current main has per-agent heartbeat scheduling, explicit session routing, targeted wakes, isolated heartbeat sessions, heartbeat task mode, and commitment delivery, but it still lacks durable native per-session heartbeat entries with independent schedules, enable/disable state, concurrency policy, run history, and operator CRUD/list surfaces.

## What This Changes

Request native session-scoped heartbeat entries so one agent can schedule, persist, inspect, and control multiple explicit heartbeat targets.

## Best Possible Solution

Keep this as a core automation design issue: define a durable session-bound heartbeat-entry model with explicit concurrency semantics, persisted enabled/run state, operator CRUD/list surfaces, and docs that separate scheduling from session-local gating, cron, and commitments.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current schema, runner state, wake dispatch, RPC/CLI methods, cron comparison, and docs all show no durable per-session heartbeat-entry model.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native heartbeat entries are plausible, but the maintainable path needs a design decision that composes with cron persistence, current targeted wakes, session routing, and inferred commitments before implementation.
