## Summary

Keep open: this PR targets a valid update-path bug and the installed-newer case is useful, but it needs real `openclaw update` proof and a code repair for the newer-pin-not-installed path, which can keep stale integrity metadata from the older requested version and fail verification after npm installs the preserved pin.

## What This Changes

The branch adds update-mode handling in managed npm plugin installs to preserve newer exact direct dependency pins, plus regression tests for the managed root writer and npm-spec install path.

## Best Possible Solution

Land a narrow update-mode fix that preserves newer exact direct pins with correct resolution metadata, compatibility coverage, and redacted real `openclaw update` proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main writes the requested dependency spec into the managed npm root, and the linked issue gives a concrete `openclaw update --tag` downgrade path. I did not run the mutating real update flow locally because this review is read-only.

## Solution Assessment

No. The installed-newer case is a good direction, but the proposed helper keeps stale integrity when preserving a newer pin that npm still has to install; the safer fix is to re-resolve or clear version-specific metadata and prove the real update path.
