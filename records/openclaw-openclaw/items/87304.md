## Summary

Keep open: the central downgrade fix is on the right path, but the new fast path still trusts package-lock metadata as proof of the actual installed package version, so it can report a preserved newer npmResolution while using a different node_modules package.

## What This Changes

The PR changes managed npm plugin update installs to preserve newer exact direct dependency pins, re-resolve preserved-version metadata when lock metadata is missing, fail closed on preserved metadata lookup failure, and add focused regression tests.

## Best Possible Solution

Keep the preserved-newer-pin behavior, but require the fast reuse path to prove package-lock metadata and the actual package directory agree before skipping npm install.

## Reproduction Assessment

Yes. Source inspection shows current main writes the requested older dependency spec into the managed root manifest during update, and the linked issue provides a concrete 0.11.2 to 0.10.0 downgrade reproduction; I did not run the full live update flow in this read-only review.

## Solution Assessment

No, not quite. The proposed direction is narrow and maintainable, but the current fast path must verify the actual package directory version before treating lockfile metadata as proof that the newer package is installed.
