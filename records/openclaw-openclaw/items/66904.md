## Summary

Keep open. Current main still lacks a WhatsApp `messaging-history.set` listener, reconnect-history config, or Baileys history-sync filter; the existing WhatsApp path handles live `messages.upsert` and deliberately skips stale append backlog.

## What This Changes

Add opt-in WhatsApp reconnect history sync so Baileys `messaging-history.set` messages can populate agent context buffers after reconnect.

## Best Possible Solution

Add a WhatsApp-plugin-owned, opt-in reconnect-history path that subscribes to typed `messaging-history.set`, enables the needed Baileys history filter, normalizes and dedupes synced messages through existing inbound helpers, and populates bounded DM/group context without broad message-store/search behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: emit `messaging-history.set` into the current monitor or reconnect with Baileys history sync; current main has no listener, so those messages cannot enter OpenClaw context, while stale `append` upserts are intentionally skipped.

## Solution Assessment

Yes, if kept narrow and opt-in. The maintainable fix belongs in the WhatsApp plugin runtime/config/docs/tests, not in core startup migration or a broad persistent history/search feature.

## Risks / Open Questions

- Baileys/WhatsApp Web history behavior can vary by account, linked-device state, and sync type; a fix should not rely solely on unreliable on-demand history fetch behavior noted in #56088.
- Replayed history can duplicate live delivery or trigger unwanted replies unless the implementation dedupes by message key and keeps stale backlog context-only by default.
- A reconnect-history feature exposes older private/group messages to agent context, so opt-in config, tight limits, and documentation are important.
