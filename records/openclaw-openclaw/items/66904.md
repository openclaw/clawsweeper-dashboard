## Summary

Keep open. Current main still lacks a WhatsApp `messaging-history.set` listener, reconnect-history config, or Baileys history-sync filter path; existing code handles live `messages.upsert` and intentionally skips stale append backlog. Related history/offline-message attempts are closed or unmerged, so there is no merged replacement to close against.

## What This Changes

The issue asks OpenClaw's WhatsApp plugin to capture Baileys `messaging-history.set` events on reconnect and make missed messages available through existing DM and group context history buffers under opt-in config.

## Best Possible Solution

Add a WhatsApp-plugin-owned, opt-in reconnect-history path: configure the Baileys history filter for the selected sync types, subscribe to typed `messaging-history.set`, normalize and dedupe synced `WAMessage`s through existing inbound helpers, populate DM/group context history without default stale-backlog auto-replies, detach listeners on close, and document the limitations.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit `messaging-history.set` with messages on the current mock socket: current main has no listener, so `onMessage` cannot receive those messages; existing tests only exercise `messages.upsert` behavior.

## Solution Assessment

Yes, the requested direction is the right shape if kept narrow, plugin-owned, and opt-in. A broad persistent message store/search feature or core migration is not needed to solve this reconnect-context gap.
