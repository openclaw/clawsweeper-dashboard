## Summary

Keep open: current main still lacks explicit Claude CLI `rate_limit_event.status` handling, and an open unmerged fixing PR already references this issue with closing syntax, so the issue should not be closed before that PR lands or is replaced.

## What This Changes

The issue reports that Claude CLI `allowed_warning` rate-limit events from successful Pro Max runs are recorded as billing failures, causing long auth-profile cooldowns and fallback to another model.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80528 or an equivalent narrow parser fix that ignores `allowed`/`allowed_warning`, maps SDK `rejected` to `rate_limit`, and includes regression fixtures plus redacted real Claude CLI proof.

## Reproduction Assessment

No. The issue gives detailed live evidence and current source shows the relevant parser gap, but this review did not run a Claude Pro Max >=75% live session or observe the auth-state write on current main.

## Solution Assessment

Yes, directionally. Handling the SDK `rate_limit_event.status` contract in the Claude CLI JSONL parser is the narrow maintainable fix, with `rejected` rather than the issue body's `denied` spelling for the hard-cap case.
