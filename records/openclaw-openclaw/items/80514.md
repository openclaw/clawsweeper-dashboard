## Summary

Keep open: the report describes a concrete user-visible auth-state failure in a supported Claude CLI runtime path, and current main has the relevant stream-json parser and billing-disable machinery but no status-aware regression for Claude `rate_limit_event` warnings. I did not prove the failure live on current main because it requires a Pro/Max account in the `allowed_warning` state, so this is not safe to close.

## What This Changes

The issue asks OpenClaw to treat Claude CLI `rate_limit_event.status` values `allowed` and `allowed_warning` as successful soft warnings instead of billing failures that disable the `anthropic:claude-cli` profile.

## Best Possible Solution

Add a focused Claude CLI stream-json/auth-state regression and make `allowed` or `allowed_warning` cap events non-failing while preserving short `rate_limit` handling for denied cap events and long billing disables for explicit credit/billing failures.
