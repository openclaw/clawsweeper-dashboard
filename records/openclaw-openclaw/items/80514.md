## Summary

Keep open: the report is concrete and recent, the related fix PR was closed unmerged, and current `main` plus the latest release still lack explicit Claude CLI `rate_limit_event` status handling or regression coverage.

## What This Changes

The issue reports that Claude CLI `allowed_warning` rate-limit events are recorded as billing failures, causing a long auth cooldown and unwanted fallback to another provider.

## Best Possible Solution

Add focused Claude CLI JSONL handling and regression coverage: ignore `allowed` and `allowed_warning`, map `rejected` to `rate_limit`, and leave billing reserved for explicit billing or credit failures.

## Reproduction Assessment

Unclear. The reporter provides a detailed live reproduction and auth-state observation, and current source lacks explicit status handling, but this review did not establish a failing current-main repro for the successful `allowed_warning` stream.

## Solution Assessment

Yes, the parser-level direction is the narrow maintainable fix, with the SDK-contract correction that hard denial is `rejected` rather than `denied`.
