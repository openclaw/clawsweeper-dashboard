Keep open: the report is detailed and the current code still has no explicit Claude CLI `rate_limit_event` status handling or regression coverage, while the linked fix PR was closed unmerged. ## What This Changes

The issue asks OpenClaw to stop treating Claude CLI `rate_limit_event` `allowed_warning` JSONL records as billing failures that disable the `claude-cli` auth profile and trigger fallback models.

## Best Possible Solution

Add a repro-first Claude CLI JSONL fixture around `allowed`, `allowed_warning`, and `rejected`; keep soft warnings out of `usageStats`, map `rejected` to `rate_limit`, and investigate the actual auth-state writer first if the soft-warning fixture already passes.

## Reproduction Assessment

Unclear. The issue gives a detailed live report, but I did not establish a current-main path where an `allowed_warning` success stream writes `disabledReason: billing`; source inspection shows the success path ignores non-result records and only classifies errors after failed runs.

## Solution Assessment

Yes, the requested parser/status direction is the narrow maintainable fix if a current-main fixture proves the failure. The safer implementation should use the SDK status spelling `rejected` for hard denial and avoid changing cooldown defaults or fallback policy.
