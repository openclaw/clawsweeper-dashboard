## Summary

Keep open: current main still lacks structured Claude CLI `rate_limit_event.status` handling, and the linked fix PR was closed without merging. The report remains a narrow bug candidate, although this read-only pass did not live-reproduce the exact Pro Max auth-state transition.

## What This Changes

Treat Claude CLI `allowed`/`allowed_warning` rate-limit events as successful and hard-cap rejections as `rate_limit`, not `billing`, so profiles are not put into a long billing cooldown after served requests.

## Best Possible Solution

Land a focused parser/classifier fix with fixtures for `allowed`, `allowed_warning`, and `rejected`, while preserving the existing billing lane for explicit payment or credit failures.

## Reproduction Assessment

No live high-confidence current-main reproduction was established. The reporter supplied a detailed real payload and current source shows the missing structured handler/runtime path, but I did not run a Pro Max `allowed_warning` session or observe the auth-state write.

## Solution Assessment

Yes, the narrow maintainable direction is to handle Claude CLI `rate_limit_event` status in the JSONL/error parser and cover it with fixtures. A broader cooldown or fallback-policy change is not needed for this bug.
