## Summary

Keep open: current main still lacks explicit Claude CLI `rate_limit_event.status` handling or regression coverage, and the linked closing-syntax fix PR was closed without merging. The SDK contract confirms `allowed_warning` is a served request and `rejected` is the hard-cap status, but the exact current-main transition into a `billing` auth-profile write still needs a focused fixture or live reproduction.

## What This Changes

The issue asks OpenClaw to treat Claude CLI `rate_limit_event.status=allowed_warning` as a served request, avoid billing cooldowns, and classify hard cap rejection as `rate_limit`.

## Best Possible Solution

Add a focused Claude CLI JSONL fixture for `allowed`, `allowed_warning`, and `rejected`; keep served warning statuses out of auth failure recording, map `rejected` to `rate_limit`, and leave true direct-API billing errors on the existing billing backoff path.

No high-confidence current-main reproduction yet. The issue provides a detailed live report and source inspection confirms the missing structured handler, but I did not prove the exact write from a successful `allowed_warning` stream into `disabledReason: billing`.

## Solution Assessment
