## Summary

Keep open: current main still lacks explicit Claude CLI `rate_limit_event.status` handling or regression coverage, the only linked fix PR was closed unmerged, and the SDK/docs support treating `allowed_warning` as non-billing.

## What This Changes

The issue asks OpenClaw to treat Claude CLI `rate_limit_event` `allowed`/`allowed_warning` as successful output and hard-cap rejection as `rate_limit`, not a billing disable.

## Best Possible Solution

Add status-aware Claude CLI JSONL fixtures and parser behavior that ignores served warning events, maps SDK `rejected` cap events to `rate_limit`, and leaves `billing` for explicit payment or credit failures.

## Reproduction Assessment

No high-confidence live current-main reproduction. The issue gives detailed live evidence and current source lacks the status-aware parser contract, but I did not observe the auth-state write and the exact failing edge still needs a focused fixture.

## Solution Assessment

Yes, with one correction: the narrow maintainable fix is to key off the SDK `rate_limit_info.status` values, including `rejected` rather than `denied`. That keeps the change in the Claude CLI JSONL parser/failover path instead of weakening billing backoff globally.
