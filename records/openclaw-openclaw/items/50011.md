## Summary

Keep open. Current main still accepts arbitrary `gateway.nodes.denyCommands` strings through schema and the `openclaw config set` write path, while the existing warning only runs later in security audit. The issue is also paired with open PR #50076 using closing syntax and overlapping open PR #50320, so this should remain open until maintainers review and land one focused fix.

## What This Changes

The issue requests config-write validation or warning when `gateway.nodes.denyCommands` contains unknown node command names that will not affect runtime command policy.

## Best Possible Solution

Land a narrow post-mutation validator for config writes that touch `gateway.nodes.denyCommands`, `gateway.nodes.allowCommands`, or ancestor `gateway.nodes` objects. It should reuse the built-in and dangerous node-command vocabulary, account for same-update custom `allowCommands`, preserve plugin/custom extensibility with warnings rather than hard schema rejection, and add focused config CLI tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a typo such as `sendd` or `system.runx` can pass the current config schema and reach `replaceConfigFile`, while the security audit helper would later flag the same value as unknown or ineffective. I did not run a live config write because this review is read-only.

## Solution Assessment

Yes. The best fix belongs in the config mutation path rather than a static schema enum, because command names can be extended by runtime/plugin configuration. The direction in open PR #50076 matches that shape, but maintainers still need to resolve overlap with #50320.
