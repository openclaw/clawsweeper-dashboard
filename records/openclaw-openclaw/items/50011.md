## Summary

Keep this open: current main and v2026.5.22 still accept `gateway.nodes.denyCommands` as a free-form string array at config-write time, while runtime only removes exact command IDs and the security audit reports ineffective entries after the config is already saved.

## What This Changes

The issue requests write-time warning or validation for invalid `gateway.nodes.denyCommands` entries so typoed command names are not silently accepted.

## Best Possible Solution

Land a maintainer-approved config-mutation warning that reuses the shared runtime/audit node-command vocabulary for CLI and Gateway config writes, with regression coverage for leaf writes, ancestor-object writes, and custom/plugin command IDs.

## Reproduction Assessment

Yes. Source inspection shows current main accepts any string-array `gateway.nodes.denyCommands`, runtime only removes exact command IDs, and audit detection runs later; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes in direction. The narrow maintainable fix is a config-write-time warning based on the shared node command catalog; audit-only remediation leaves the misleading successful write in place.
