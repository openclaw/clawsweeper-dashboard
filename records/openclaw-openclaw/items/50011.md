## Summary

Keep open: current main and v2026.6.6 still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while the ineffective-entry warning only appears later in `security audit`. The remaining work is real, but warning-versus-failure behavior and custom/plugin command handling need maintainer direction, especially with several unmerged related PR attempts already in the discussion.

## What This Changes

Request warning or validation for invalid `gateway.nodes.denyCommands` values at config-write time instead of relying only on later security audit findings.

## Best Possible Solution

Add a narrow warning-first shared config mutation validator that reuses the audit/runtime command vocabulary for direct and ancestor writes, while leaving strict rejection or `security audit --fix` autocorrection as explicit maintainer decisions.

## Reproduction Assessment

Yes. Source inspection shows the schema accepts arbitrary string-array `gateway.nodes.denyCommands`, the config-write path lacks a denyCommands-specific warning, runtime exact matching makes invalid entries no-ops, and only the audit path flags them later; I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear until maintainers choose warning versus hard failure. A warning-only validator in the shared config mutation path appears safer than schema rejection because node commands can be custom or plugin-provided.
