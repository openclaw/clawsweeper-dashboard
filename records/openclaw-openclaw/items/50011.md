- `impact:security`: Invalid denylist entries can make users believe sensitive node commands are blocked when runtime will ignore the typo.

## Summary

Keep this open: current main still accepts `gateway.nodes.denyCommands` as a plain string array and only the security audit path detects ineffective entries later. The focused open fix at https://github.com/openclaw/openclaw/pull/50076 references this issue with closing syntax, so the issue should remain open until that PR or a replacement lands.

## What This Changes

The issue asks for write-time warning or validation when `gateway.nodes.denyCommands` contains command names that will not match the gateway command registry.

## Best Possible Solution

Land a rebased focused validator in the shared config mutation path that warns on ineffective deny entries using the audit/runtime command catalog while accepting exact custom/plugin IDs, then close this issue after merge.

## Reproduction Assessment

Yes, at source level. Current main accepts any string array for `gateway.nodes.denyCommands`, writes it through `config set`, and only later audit logic reports unknown or pattern-like exact entries.

## Solution Assessment

Yes in direction. The maintainable fix is write-time warning/validation in the config mutation path using the same command vocabulary as runtime/audit; `security audit --fix` alone would leave the misleading successful write behavior in place.
