## Summary

Keep open. Current main still models `gateway.nodes.denyCommands` as free-form strings and the config mutation path reaches the file write without node-command registry validation; the existing audit catches unknown or pattern-like entries only after persistence. The issue is also directly paired with open PR #50076 using closing syntax, with overlapping work in #50320, so it should remain open until one focused fix lands or maintainers choose a superseding path.

## What This Changes

Warn or fail during `openclaw config set` when `gateway.nodes.denyCommands` contains unknown or ineffective node command names instead of only surfacing them later in `openclaw security audit`.

## Best Possible Solution

Land a narrow config mutation validator for writes that touch `gateway.nodes.denyCommands`, `gateway.nodes.allowCommands`, or ancestor `gateway.nodes` objects. It should reuse the audit command vocabulary and suggestion logic, preserve plugin/custom command extensibility, warn rather than hard-block unless maintainers decide otherwise, and add focused CLI regression tests plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a typo such as `system.runx` or `camera.snapp` satisfies the current schema and reaches `replaceConfigFile`, while the security audit helper would later flag the same value as unknown. I did not run a live config write because this review is read-only.

## Solution Assessment

Yes. Runtime validation in the config mutation path is the narrowest maintainable fix because node command names can be influenced by platform defaults, dangerous built-ins, plugin commands, and same-update `allowCommands`; a static schema enum would be too rigid.
