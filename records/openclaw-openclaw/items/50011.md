Keep open: current main and v2026.6.10 still leave this as audit-time detection rather than config-write validation, and the remaining behavior needs maintainer security/product direction before another fix path is promoted.

## What This Changes

Issue requests write-time warning or validation for invalid `gateway.nodes.denyCommands` values so operators do not assume ineffective deny entries are active.

## Best Possible Solution

Add a narrow warning-first config mutation validator for direct and ancestor `gateway.nodes` writes that reuses the audit/runtime command vocabulary and preserves exact custom command flexibility.

## Reproduction Assessment

Yes at source level: current main accepts arbitrary `gateway.nodes.denyCommands` strings, runtime exact matching makes unknown entries ineffective, and security audit reports them only after the write. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the product/security semantics. A warning-first validator in the shared config mutation path looks safest, while hard failure and plugin/custom command treatment need an explicit decision.
