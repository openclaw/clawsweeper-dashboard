## Summary

Keep open: current main and v2026.6.10 still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while ineffective entries are only reported later by security audit. Same-repo repair PRs are closed unmerged, and the open fork PRs are not merged target-repository fixes.

## What This Changes

The issue requests write-time warning or failure when `gateway.nodes.denyCommands` contains invalid or ineffective node command names.

## Best Possible Solution

Add a narrow warning-first config mutation validator for direct and ancestor `gateway.nodes` writes that reuses the audit/runtime command vocabulary and preserves custom/plugin exact command flexibility.

## Reproduction Assessment

Yes at source level. Current main accepts arbitrary `gateway.nodes.denyCommands` strings, exact runtime matching makes unknown entries ineffective, and security audit reports them only after the write.

## Solution Assessment

Unclear until maintainers choose warning versus failure semantics. A warning-first validator in the shared config mutation path looks like the narrowest maintainable fix because it avoids rejecting valid plugin/custom command IDs while still warning on likely mistakes.
