## Summary

Keep open. Current main still lacks write-time validation for `gateway.nodes.denyCommands`, and the issue is directly paired with open PR #50076 plus overlapping PR #50320, so it should remain open until one focused fix lands or maintainers choose a superseding path.

## What This Changes

`openclaw config set` should warn or reject writes that put unknown or pattern-like command IDs in `gateway.nodes.denyCommands` before persisting config.

## Best Possible Solution

Land one focused config mutation validator for writes touching `gateway.nodes.denyCommands`, `gateway.nodes.allowCommands`, or `gateway.nodes`, reusing the audit command vocabulary and suggestion logic while preserving custom/plugin command extensibility.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an unknown string satisfies the current schema and reaches `replaceConfigFile`, while the same value is only flagged later by security audit. I did not run a live config write because this review is read-only.

## Solution Assessment

Yes, write-time validation in the config mutation path is the narrowest maintainable fix. A static schema enum would be too rigid because custom commands can be introduced through `allowCommands` and plugins.
