## Summary

Keep open: current main and v2026.6.6 still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while the ineffective-entry diagnostic only runs later in `security audit`. The remaining work is real, but warning-versus-failure behavior and plugin/custom command handling need maintainer direction before another fix lane.

## What This Changes

The issue asks for config-write-time validation or warning when `gateway.nodes.denyCommands` contains invalid command names.

## Best Possible Solution

Add a narrow warning-first validator in the shared config mutation path that reuses the audit/runtime command vocabulary for direct and ancestor writes, leaving hard failure or `security audit --fix` autocorrection as explicit maintainer decisions.

## Reproduction Assessment

Yes. Source inspection shows the schema accepts arbitrary string-array `gateway.nodes.denyCommands`, the config-write path lacks a denyCommands-specific warning, runtime exact matching makes invalid entries no-ops, and only the audit path flags them later.

## Solution Assessment

Unclear until maintainers choose warning versus hard failure. A warning-only validator in the shared config mutation path appears safer than schema rejection because node commands can be custom or plugin-provided.
