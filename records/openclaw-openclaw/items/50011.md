## Summary

Keep open: current main and the latest release still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while ineffective entries are only surfaced later by security audit. The report is valid, but warning-versus-failure semantics and custom/plugin command handling still need maintainer security/product direction before another fix lane.

## What This Changes

The issue requests write-time validation or warnings for invalid `gateway.nodes.denyCommands` entries so config writes do not silently persist ineffective denylist entries.

## Best Possible Solution

Add a narrow warning-first config mutation validator for direct and ancestor `gateway.nodes` writes that reuses the audit/runtime command vocabulary, preserves custom/plugin command flexibility, and leaves strict failure or audit autofix as explicit maintainer choices.

## Reproduction Assessment

Yes. Source inspection shows the schema accepts arbitrary string-array `gateway.nodes.denyCommands`, `runConfigOperations` writes after generic checks without a denyCommands validator, runtime exact matching makes invalid entries no-ops, and only audit flags them later.

## Solution Assessment

Unclear. A warning-only validator in the shared config mutation path looks like the safest maintainable fix, but maintainers still need to choose warning versus hard failure and how custom/plugin command IDs should be treated.
