## Label Justifications

- `P2`: This is a bounded CLI/config hardening bug with real operator-security impact but limited blast radius.
- `impact:security`: Invalid denylist entries can make operators believe sensitive node commands are blocked when runtime will ignore the typo.

## Summary

Keep open: current `main` still stores `gateway.nodes.denyCommands` as free-form strings at config-write time, while runtime only applies exact command IDs and the audit reports ineffective entries later. An open PR already references this issue with closing syntax, so this should stay open until that PR or a replacement lands.

## What This Changes

The issue requests a write-time warning or validation path for invalid `gateway.nodes.denyCommands` entries so typos are not silently accepted.

## Best Possible Solution

Land a focused config-mutation warning that reuses the audit/runtime command vocabulary, handles direct and parent-object writes, and preserves intentional custom or plugin command IDs.

## Reproduction Assessment

Yes. Source inspection shows current `main` accepts any string array for `gateway.nodes.denyCommands`, runtime only removes exact matching command IDs, and audit detection runs later; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes in direction. The narrow maintainable fix is config-write-time warning using the shared audit/runtime command catalog; `security audit --fix` alone would leave the misleading write-time success behavior in place.
