## Summary

Keep open: current main and v2026.6.1 still accept `gateway.nodes.denyCommands` as a free-form string array at config-write time, while the unknown/pattern-like command warning only runs later in `openclaw security audit`; linked fix attempts are still unmerged or closed.

## What This Changes

The issue asks OpenClaw to warn or fail during config writes when `gateway.nodes.denyCommands` contains unknown or pattern-like command names instead of only reporting them during security audit.

## Best Possible Solution

Add warning-only config set/patch validation for direct and ancestor `gateway.nodes.denyCommands` writes, reusing the security audit command catalog and suggestions while preserving custom/plugin command escape paths, then keep audit as the backstop.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: current schema accepts arbitrary string arrays, config writes do not validate `denyCommands`, and audit tests show invalid entries are only caught later by security audit. I did not run a live CLI repro because this review is read-only and must not mutate config files.

## Solution Assessment

Unclear until maintainers choose the write-time behavior. The narrowest maintainable fix appears to be a warning-only validator shared with audit suggestions, with tests for direct leaf writes and parent-object writes, rather than a hard schema enum that would block plugin/custom commands.
