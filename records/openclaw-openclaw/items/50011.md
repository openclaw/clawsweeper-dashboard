## Summary

Keep this open: current main still accepts `gateway.nodes.denyCommands` as plain strings, writes config changes without a deny-command warning, and only the later security-audit path detects ineffective unknown entries. A focused open PR already references this issue with closing syntax, so the issue should stay open until that PR or a replacement lands.

## What This Changes

Request config-write-time warning or validation when `gateway.nodes.denyCommands` contains invalid command names.

## Best Possible Solution

Land a focused, rebased config-mutation warning that shares the audit/runtime command vocabulary, handles direct and parent-object writes, and preserves explicit custom or plugin command IDs.

## Reproduction Assessment

Yes. Source inspection shows current main accepts any string array for `gateway.nodes.denyCommands`, runtime only removes exact trimmed command IDs, and audit detection runs later.

## Solution Assessment

Yes in direction. The narrow maintainable fix is config-write-time warning using the shared audit/runtime command catalog; `security audit --fix` alone would leave the misleading write-time success behavior in place.
