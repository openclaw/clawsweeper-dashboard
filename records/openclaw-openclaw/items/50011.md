## Summary

Keep this open: current main and the latest release still accept `gateway.nodes.denyCommands` as a free-form string array at config-write time, while runtime exact-matches only and the security audit catches ineffective entries later. The focused target-repo fix PR was closed unmerged, so there is no merged or open target-repo PR that owns the remaining fix.

## What This Changes

`gateway.nodes.denyCommands` should warn or fail during `openclaw config set` when entries are unknown or pattern-like command names.

## Best Possible Solution

Land a maintainer-approved config-mutation warning that reuses the audit/runtime node-command vocabulary, handles leaf and ancestor writes, and preserves valid custom/plugin command IDs.

## Reproduction Assessment

Yes. Source inspection shows current main accepts any string-array `gateway.nodes.denyCommands`, runtime removes only exact command IDs, and audit detection runs later; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes in direction. The narrow maintainable fix is a config-write-time warning based on the shared node command catalog; audit-only remediation leaves the misleading successful write in place.
