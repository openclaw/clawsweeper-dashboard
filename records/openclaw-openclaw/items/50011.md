## Summary

Keep open: current main and v2026.6.10 still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while runtime exact matching makes unknown entries ineffective and security audit only reports them later. The remaining work needs maintainer direction on warning-versus-fail behavior and custom/plugin command IDs before another repair PR is queued.

## What This Changes

The issue requests write-time warning or validation for invalid `gateway.nodes.denyCommands` values instead of silently accepting them until security audit.

## Best Possible Solution

Add a narrow warning-first validator in the shared config mutation path for direct and ancestor `gateway.nodes` writes, reusing the audit/runtime command vocabulary while preserving custom/plugin exact command flexibility.

## Reproduction Assessment

Yes at source level: current main accepts arbitrary `gateway.nodes.denyCommands` strings, runtime removes only exact command IDs, and audit reports unknown or pattern-like entries only after the write. I did not run the CLI because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the product/security semantics. A warning-first validator in `runConfigOperations` looks like the safest fix; hard failure and plugin/custom command treatment need explicit direction.
