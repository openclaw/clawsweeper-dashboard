## Summary

Keep open: current main and v2026.6.10 still validate ineffective `gateway.nodes.denyCommands` only during security audit, not at config-write time. The remaining behavior needs maintainer direction on warning versus failure semantics and custom/plugin command handling.

## What This Changes

The issue asks OpenClaw to warn or fail when `gateway.nodes.denyCommands` contains unknown command names at config-write time instead of only during security audit.

## Best Possible Solution

Add a maintainer-approved warning-first config mutation validator for direct and ancestor `gateway.nodes` writes that reuses the audit/runtime command vocabulary and preserves custom command flexibility.

## Reproduction Assessment

Yes at source level: current main accepts arbitrary `gateway.nodes.denyCommands` strings, runtime exact matching makes unknown entries ineffective, and security audit reports them only after the write. I did not run the CLI because this review is read-only and avoids commands that can mutate config files.

## Solution Assessment

Unclear until maintainers choose warning versus failure semantics. A warning-first validator in the shared config mutation path looks safest because plugin/custom commands may be valid even when they are absent from the built-in vocabulary.
