## Summary

Keep open: current main and v2026.6.6 still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while unknown or ineffective entries are only warned about later by `security audit`. The remaining work is real, but warning-versus-failure behavior and plugin/custom command handling need maintainer direction, especially with linked implementation attempts still unmerged.

## What This Changes

The issue asks for `openclaw config set` to warn or fail when `gateway.nodes.denyCommands` contains command names that are not known to the gateway command registry.

## Best Possible Solution

Add a narrow warning-first validator in the shared config mutation path for direct and ancestor `gateway.nodes` writes, reusing the audit/runtime command vocabulary while preserving custom/plugin command flexibility.

## Reproduction Assessment

Yes by source inspection. Current main accepts arbitrary string-array `gateway.nodes.denyCommands`, the runtime only deletes exact allowlist entries, and the unknown-command warning exists in the later security audit path rather than in config writes; I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear. Warning-only validation in the shared config mutation path looks like the narrowest maintainable fix, but warning versus hard failure and custom/plugin command handling need maintainer direction.
