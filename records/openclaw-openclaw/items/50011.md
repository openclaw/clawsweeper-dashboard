## Summary

Keep open: current main and the latest release still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while ineffective entries are only detected later by security audit; the remaining decision is warning-versus-failure behavior for custom/plugin command IDs.

## What This Changes

The issue asks for `openclaw config set gateway.nodes.denyCommands` to warn or fail when invalid node command names are written instead of deferring detection to security audit.

## Best Possible Solution

Add a narrow warning-first config mutation validator for direct and ancestor `gateway.nodes` writes that reuses the audit/runtime command vocabulary and preserves plugin/custom command flexibility.

## Reproduction Assessment

Yes. Source inspection shows current main accepts arbitrary `gateway.nodes.denyCommands` strings, writes them without a denyCommands-specific validator, and only reports ineffective entries later through security audit.

## Solution Assessment

Unclear. A warning-first validator in the shared config mutation path looks like the safest maintainable fix, but maintainers still need to decide warning versus hard failure and custom/plugin command treatment.
