## Summary

Keep this open: current main still stores `gateway.nodes.denyCommands` as a free-form string array at config-write time, while runtime only applies exact command IDs and the security audit reports bad entries afterward. A focused open PR already references this issue with closing syntax, so the right cleanup path is to review, rebase, or replace that PR rather than close the issue or queue duplicate repair.

## What This Changes

The issue requests config-write-time warning or validation when `gateway.nodes.denyCommands` contains unknown or ineffective node command names.

## Best Possible Solution

Land the linked focused fix, or a replacement, that warns during config mutation using the shared audit/runtime command vocabulary while preserving explicit custom or plugin command IDs.

## Reproduction Assessment

Yes. Source inspection shows current main accepts any string array for `gateway.nodes.denyCommands`, runtime only removes exact matching IDs, and audit detection runs later; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes in direction. The narrow maintainable solution is config-write-time warning based on the shared command catalog; `security audit --fix` alone would not address the misleading successful write.
