## Summary

Keep open: current main and the latest release still accept arbitrary `gateway.nodes.denyCommands` strings during config writes, while invalid or ineffective entries are only detected later by security audit. The remaining blocker is maintainer security/product direction on warning versus failure behavior and plugin/custom command IDs.

## What This Changes

Request that `openclaw config set` warn or fail when `gateway.nodes.denyCommands` contains invalid command names.

## Best Possible Solution

Add a narrow warning-first config mutation validator for direct and ancestor `gateway.nodes` writes that reuses the audit/runtime command vocabulary and preserves plugin/custom exact command flexibility.

## Reproduction Assessment

Yes at source level. Current main accepts arbitrary `gateway.nodes.denyCommands` strings, exact runtime matching makes unknown entries no-ops, and security audit reports them only after the write.

## Solution Assessment

Unclear until maintainers choose the product/security semantics. A warning-first validator in the shared config mutation path looks safest, but hard failure and plugin/custom command treatment need an explicit decision.
