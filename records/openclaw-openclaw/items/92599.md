## Label Justifications

- `P1`: A scheduled model workflow can miss many runs under provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported cron turns lose their expected final output because fallback is never attempted before the job timeout.
- `impact:auth-provider`: The issue is about provider/model timeout handling and configured model fallback routing under stalled provider connections.
- `impact:crash-loop`: The affected model calls hang until the caller watchdog kills the job, making this an availability failure rather than only a bad response.

## Summary

Keep open: current main has fixes for some timeout/fallback pieces, but source review still leaves a current path where a stalled model stream can consume the same cron/provider timeout budget before fallback gets a useful attempt, and provider-owned Anthropic paths still need timeout propagation proof.

## What This Changes

The issue asks OpenClaw to turn stalled model-provider connections into failed model calls so `model_call_ended` fires and configured fallbacks run before cron or job timeouts kill the turn.

## Best Possible Solution

Define a canonical per-candidate request or first-byte timeout that is shorter than the caller wall-clock budget, always emits `model_call_ended`, and enters fallback for stalled streams across OpenAI-compatible and Anthropic-compatible transports.

## Reproduction Assessment

Yes at source level, but not with a live provider replay; a synthetic provider that accepts a stream and never yields while cron/provider timeouts match should exercise the same fallback race.

## Solution Assessment

Unclear until maintainers choose timeout hierarchy; the safe fix should separate retryable candidate-idle failure from the whole cron/job budget instead of only shortening existing timeouts.
