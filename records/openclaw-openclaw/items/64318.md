## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Rebase without reverting root package metadata:** `package.json:3`
  - body: The PR head changes root `package.json` back to version `2026.4.26` and an older `files` allowlist while current main is `2026.5.19` with newer package entries. That stale package churn can break published contents and dependency resolution before the Amazon plugin code is even evaluated.
  - confidence: 0.97
- **[P1] Gate Transcribe when disabled:** `extensions/amazon/transcribe/media-understanding-provider.ts:53-58`
  - body: `transcribe.enabled` is parsed, but the provider returned here always advertises audio capability and always exposes `transcribeAudio`. Current media-understanding routing has no provider `isConfigured` hook and selects audio providers by capability/handler plus auth, so `amazon.transcribe.enabled: false` can still be auto-selected and invoked.
  - confidence: 0.96
- **[P1] Follow Nova Sonic's bidirectional event schema:** `extensions/amazon/nova-sonic/bridge.ts:184-197`
  - body: The bridge sends a custom `sessionStart` shape and then `audioInput` chunks without the required `promptStart`/`contentStart` names or close sequence. AWS documents those identifiers as part of the Nova Sonic bidirectional contract, so this stream shape is unlikely to work against the service.
  - confidence: 0.94
- **[P1] Defer error close until reconnect fails:** `extensions/amazon/nova-sonic/bridge.ts:289-299`
  - body: On a stream error, `processOutputStream()` starts `attemptReconnect()` and returns, but the `finally` block immediately emits `onClose("error")`. Voice-call treats an error close as a call-ending failure, so transient Bedrock stream interruptions can tear down the call before reconnect has a chance to succeed.
  - confidence: 0.93
- **[P1] Resample telephony audio before Nova I/O:** `extensions/amazon/nova-sonic/bridge.ts:67-72`
  - body: This path converts 8 kHz telephony mu-law to PCM bytes but sends it as 16 kHz Nova input without resampling; the output path likewise converts 24 kHz PCM to mu-law without downsampling to 8 kHz. Calls will be time-scaled or distorted even when the codec label is right.
  - confidence: 0.92
- **[P2] Use a valid Polly PCM telephony rate:** `extensions/amazon/polly/speech-provider.ts:235-241`
  - body: `synthesizeTelephony` requests `outputFormat: "pcm"` with sample rate `22050`, but Polly's `SynthesizeSpeech` API only accepts `8000` or `16000` for PCM output. Normal telephony synthesis can fail with an invalid sample-rate error.
  - confidence: 0.9
- **[P2] Declare the true FLAC sample rate:** `extensions/amazon/transcribe/stt.ts:110-112`
  - body: The FLAC branch passes the original bytes through but hard-codes `MediaSampleRateHertz` to `16000`. Transcribe Streaming requires the specified sample rate to match the input audio, so common 44.1/48 kHz FLAC files can fail or transcribe poorly.
  - confidence: 0.88
