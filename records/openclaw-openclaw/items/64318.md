## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Drop stale root package rewrites:** `package.json:3`
  - body: The branch reverts root package metadata to an older `2026.4.26` snapshot and removes current package file/export entries while adding the Amazon plugin. That unrelated root churn can regress published package contents and SDK dependency resolution, so rebase/regenerate from current main and keep only intentional package changes.
  - confidence: 0.94
- **[P1] Send Nova Sonic's documented event shapes:** `extensions/amazon/nova-sonic/bridge.ts:130-168`
  - body: The bridge sends simplified `{ event: { type }, data }` objects and starts audio without the required `promptStart`/`contentStart` hierarchy or shared `promptName`/`contentName` identifiers. AWS documents a structured bidirectional flow, so this session is unlikely to work against Bedrock as written.
  - confidence: 0.93
- **[P2] Honor the Transcribe enabled flag:** `extensions/amazon/transcribe/media-understanding-provider.ts:53-58`
  - body: `transcribe.enabled` is parsed but never applied to provider registration: the provider still advertises audio capability, auto priority, and `transcribeAudio`. Since the current media-understanding contract has no provider `isConfigured` hook, disabled Amazon Transcribe can still be selected and invoked.
  - confidence: 0.95
- **[P2] Use a valid Polly PCM sample rate:** `extensions/amazon/polly/speech-provider.ts:235-241`
  - body: `synthesizeTelephony` requests `outputFormat: "pcm"` with sample rate `22050`, but Polly only accepts `8000` or `16000` for PCM. Normal telephony synthesis can fail with an AWS validation error instead of returning audio.
  - confidence: 0.94
- **[P2] Declare the real FLAC sample rate:** `extensions/amazon/transcribe/stt.ts:110-112`
  - body: The FLAC path passes audio through unchanged while declaring `MediaSampleRateHertz: 16000`. Amazon Transcribe requires the request sample rate to match the input audio, so common FLAC files at 44.1 or 48 kHz can fail or transcribe incorrectly.
  - confidence: 0.9
- **[P2] Resample telephony input before Nova Sonic:** `extensions/amazon/nova-sonic/bridge.ts:67-72`
  - body: This path only converts 8 kHz mu-law bytes to PCM samples, but the bridge session declares 16 kHz input. Without 8k to 16k resampling, Nova Sonic receives time-scaled speech and recognition quality can break.
  - confidence: 0.88
- **[P2] Downsample Nova output before telephony:** `extensions/amazon/nova-sonic/bridge.ts:318-321`
  - body: The session requests 24 kHz PCM output, then converts it directly to mu-law and emits it to telephony callbacks without 24k to 8k downsampling. Voice-call media streams expect 8 kHz mu-law, so playback can be wrong-speed even after codec conversion.
  - confidence: 0.88
- **[P2] Defer error close until reconnect retries fail:** `extensions/amazon/nova-sonic/bridge.ts:289-299`
  - body: `processOutputStream` starts `attemptReconnect()` on stream errors and returns, but the `finally` block immediately emits `onClose("error")`. Realtime call flows that treat error close as teardown can end a recoverable call before reconnect completes.
  - confidence: 0.9
