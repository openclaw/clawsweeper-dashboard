## Summary

Keep open: current main and v2026.6.6 handle the adjacent stalled-run abort/lane-release path, but they still do not rotate the wedged hot session, retry the original Telegram direct-chat turn once, or write the repair record requested here before the generic assistant failure path can surface `LLM request failed.`

## What This Changes

Request automatic reset-and-retry for a stalled Telegram direct-chat model call after stuck-session recovery aborts the embedded run.

## Best Possible Solution

Define a narrow direct-session recovery contract that resets only the unsafe hot session, preserves durable memory, retries the same user turn once, prevents duplicate delivery, and logs a structured repair record.

Yes at source level: current diagnostics can classify stale model-call work and abort the embedded run, while assistant failover and payload synthesis can still surface the generic failure; this review did not replay the exact Telegram incident live.

## Solution Assessment

Unclear until maintainers choose the recovery policy; reset-and-retry is plausible, but it must reuse existing reset primitives and prove it will not duplicate turns or reset true provider failures.
