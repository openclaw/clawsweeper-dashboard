## Summary

Keep open: current main still aborts/releases diagnostic stuck-session work and can surface the generic aborted-assistant failure, but it does not reset the wedged hot session, retry the original direct-chat turn once, or write the repair record requested here. The open adjacent fix at https://github.com/openclaw/openclaw/pull/92303 targets lane release for https://github.com/openclaw/openclaw/issues/92270 and does not supersede this reset/retry policy decision.

## What This Changes

The issue requests automatic reset-and-retry recovery for wedged long-lived direct chat sessions after stalled embedded model calls abort and would otherwise surface a generic LLM failure.

## Best Possible Solution

Define a narrow direct-session recovery contract that resets only the unsafe hot session, preserves durable memory, retries the same user turn once, prevents duplicate delivery, and logs a structured repair record.

## Reproduction Assessment

Yes at source level: current main can classify stale model-call work, abort the embedded run, and still synthesize the generic aborted assistant failure; the live incident also showed manual reset restoring the exact session key. This review did not replay the exact Telegram production incident live.

## Solution Assessment

Unclear until maintainers choose the recovery policy; reset-and-retry is plausible, but it must reuse existing reset primitives and prove it will not duplicate turns or reset true provider failures.
