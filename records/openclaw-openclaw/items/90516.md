## Summary

Keep open: current main still detects and aborts stalled model-call sessions and can surface the generic failure reply, but it does not reset the wedged hot session, retry the original direct-chat turn once, or record the requested repair evidence. The adjacent open PR #92303 only targets the lane-release subproblem from #92270 and does not supersede this reset/retry policy work.

## What This Changes

A Telegram direct-chat session with a long compacted history wedged during a model call and returned `LLM request failed.` until manual `sessions.reset`; the requested change is automatic safe reset, one retry, and operator repair evidence.

## Best Possible Solution

Define a narrow recovery contract for channel-backed direct sessions: detect locally unsafe stalled model-call state, reset only the hot session while preserving durable memory, retry the same user turn once with idempotent delivery, and log a structured repair record.

## Reproduction Assessment

Yes at source level and from production logs: current main can classify stale model-call work, abort the embedded run, and still synthesize the generic aborted assistant failure; this review did not replay the exact Telegram incident live.

## Solution Assessment

Unclear pending maintainer policy; reset-and-retry is plausible only if it reuses `performGatewaySessionReset`, guards duplicate delivery/tool replay, and distinguishes local session wedge from provider failure.
