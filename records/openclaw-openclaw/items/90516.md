## Summary

Keep open: current main and the latest release still abort/release stalled active model-call work, but they do not implement the requested hot-session reset, one retry of the original direct-chat turn, or structured repair record. The adjacent open work at https://github.com/openclaw/openclaw/pull/92303 targets lane release for https://github.com/openclaw/openclaw/issues/92270 and does not supersede this reset/retry policy issue.

## What This Changes

The issue requests automatic recovery for wedged long-lived direct chat sessions by resetting unsafe hot session state, retrying the same user turn once, and recording the repair before surfacing a generic LLM failure.

## Best Possible Solution

Define a narrow core recovery contract that resets only the unsafe hot session, preserves durable memory, retries the same user turn once, prevents duplicate delivery, and logs a structured repair record.

## Reproduction Assessment

Yes, at source level: current main can classify stale model-call work, abort the embedded run, and still follow the generic failure path without resetting and retrying the original turn. I did not replay the exact Telegram production incident live in this read-only review.

## Solution Assessment

Unclear: reset-and-retry is a plausible recovery, but maintainers still need to choose the duplicate-turn guard, unsafe-session criteria, and repair-record contents before implementation.
