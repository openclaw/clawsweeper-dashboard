## Summary

Keep open: current main and the latest release handle the detected stall by aborting/reclaiming the active embedded run, but I found no reset-and-retry path for the wedged channel session before the normal assistant failure path can surface a generic error.

## What This Changes

The issue requests automatic reset-and-retry for a wedged channel-backed direct chat session after stalled model-call recovery aborts the embedded run and currently surfaces a generic failure.

## Best Possible Solution

Define a narrow recovery contract for channel-backed direct sessions: after a stale model-call abort with otherwise healthy routing/provider evidence, rotate the hot session through the existing reset primitive, retry the user turn once, and log whether reset/retry was attempted and succeeded.

## Reproduction Assessment

Yes at source level, but not as a minimal live repro: the issue logs match current code that can classify stale model-call work, abort the embedded run, and then rely on normal assistant failure handling without session reset/retry.

## Solution Assessment

Unclear until maintainers choose the recovery policy; reset-and-retry is plausible, but the safe implementation should reuse existing session reset primitives and prove it cannot duplicate turns or reset healthy provider failures.
