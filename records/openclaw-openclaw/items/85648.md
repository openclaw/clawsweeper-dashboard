## Summary

Keep open: the report is detailed, current main still lacks a fresh-session cutoff or whole-announce-exchange exclusion in the `sessions_history` path, and no merged fix or linked closing PR covers this narrower `/new` contamination case. The broader related report at https://github.com/openclaw/openclaw/issues/78055 remains open, but this issue adds distinct, actionable Feishu/session-history evidence.

## What This Changes

The issue reports that `/new` can hydrate a fresh Feishu session with stale subagent announce tail messages from an older session sharing the same sessionKey.

## Best Possible Solution

Default `sessions_history` should honor the active session boundary after `/new` and exclude one-shot subagent announce delivery exchanges from normal context hydration, while keeping explicit prior-session or announce inspection available.

## Reproduction Assessment

Yes at source level, but not as a live Feishu replay in this review. A focused regression can construct an inter-session `subagent_announce` user message followed by an assistant result under a rotated session boundary and show current history projection still preserves the assistant tail.

## Solution Assessment

Unclear until a patch exists, but the maintainable direction is narrow: enforce the fresh-session boundary and default announce-delivery exclusion in history hydration rather than adding a broad history API redesign or a user setting.
