## Summary

This PR is a focused implementation of a real CLI capability gap: current main already accepts `sessionKey` in the Gateway agent protocol, but `openclaw agent` does not expose that selector. I did not find a blocking correctness issue in the diff; the remaining maintainer decision is whether the documented bare-key/default-agent and sentinel behavior is the right CLI contract before landing.

## What This Changes

The PR adds `--session-key <key>` to `openclaw agent`, normalizes and validates explicit keys across Gateway dispatch, embedded local runs, and fallback paths, and documents the new selector with focused routing tests.

## Best Possible Solution

Land the CLI flag if maintainers agree with the documented session-key contract, keeping the normalization shared across Gateway, embedded, and fallback paths with the focused regression coverage in this PR.

## Reproduction Assessment

Yes. Current main's `openclaw agent` registration lacks `--session-key`, while the Gateway protocol already has a `sessionKey` field, so the CLI gap is source-reproducible without running the command.

## Solution Assessment

Yes. Reusing the existing Gateway `sessionKey` contract and central session-key helpers is the narrow maintainable path; the only policy point is whether maintainers accept the documented bare-key and sentinel semantics.
