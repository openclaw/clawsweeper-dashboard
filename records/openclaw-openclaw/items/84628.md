## Label Justifications

- `P1`: This is a startup-path bug fix for restrictive plugin configurations, but the PR still has a merge-blocking compatibility defect in current metadata reuse.
- `merge-risk: 🚨 compatibility`: The patch can change upgraded runtime behavior by exposing a scoped startup registry to existing config-based metadata callers that expect full plugin metadata.

## Summary

Keep this PR open: the startup metadata scoping direction is useful, but the latest head still lets scoped startup snapshots leak into config-only current metadata callers, which can make later registry, provider, setup, or contract queries see only the startup subset.

## What This Changes

The PR scopes gateway startup plugin metadata snapshot loading to a conservative plugin ID set, persists installed-index config-path activation hints, and adds regression coverage for scoped startup manifests and fallback cases.

## Best Possible Solution

Keep the scoped startup manifest optimization, but make current metadata snapshot reuse exact-scope-only unless a caller explicitly opts into scoped reuse, with regression coverage for post-startup unscoped metadata queries.

## Reproduction Assessment

Yes, at source level. The PR head stores a scoped plugin lookup table as current metadata, and config-only callers still pass through getCurrentPluginMetadataSnapshot without exact pluginIds.

## Solution Assessment

No, not yet. The startup scoping approach is maintainable, but current snapshot reuse must be scope-aware before this is safe to merge.
