## Label Justifications

- `P1`: The PR targets a P1 startup stability issue where unscoped plugin discovery can cause boot stalls, high memory use, or crash-loop-like availability failures.
- `merge-risk: 🚨 compatibility`: The diff changes which plugin manifests are available during startup, so a missed owner in the scope could break existing configured plugin behavior after upgrade.

## Summary

Keep open. The scoped metadata lookup is the right direction, but the PR is not merge-ready because the normal gateway startup path can still receive and reuse an unscoped config-validation metadata snapshot before the new scoped fallback runs, so the linked startup discovery problem may remain in the real boot path.

## What This Changes

The PR adds Installed Plugin Index-derived plugin-id scoping to metadata snapshot loading and persists startup config-path metadata, with regression tests for restrictive plugin allowlists and fallback cases.

## Best Possible Solution

Carry the conservative scope into the first gateway config/startup metadata snapshot, or otherwise ensure startup does not build or reuse a full manifest registry before scoped lookup, while retaining unscoped fallbacks for legacy or ambiguous indexes.

## Reproduction Assessment

Yes, at source level. Current main builds plugin metadata before startup narrowing, and the linked issue provides real user evidence of manifest reads scaling with installed plugin count; I did not run a live heap or boot profile in this read-only review.

## Solution Assessment

No, not yet. The scoped lookup fallback is a maintainable shape, but it must apply before or instead of the unscoped config-validation metadata snapshot for this PR to fix the real startup path it closes.
