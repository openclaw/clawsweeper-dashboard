## Summary

Keep open: the startup-scope optimization is useful and the proof is strong, but the latest head still lets a scoped startup metadata snapshot satisfy ordinary config-based callers, which can hide non-startup installed plugins after gateway startup.

## What This Changes

The PR adds scoped plugin metadata snapshot loading for restrictive gateway startup allowlists, persists installed-index config-path activation hints, and adds regression coverage for startup scoping and fallback cases.

## Best Possible Solution

Keep the scoped startup manifest optimization, but make current metadata snapshot reuse scope-aware so unscoped/control-plane callers reload full metadata unless they request an exact scoped set, with regression coverage for post-startup manifest and contract queries.

## Reproduction Assessment

Yes, at source level. Current main loads the installed-index manifest registry without a pluginIds scope, and the PR body provides before/after live diagnostic output; the remaining blocker is traceable from the scoped current snapshot handoff to unscoped config-based callers.

## Solution Assessment

No, not yet. The scoped startup direction is the right boundary, but the patch needs exact-scope current snapshot reuse before it is safe to merge.
