## Summary

Keep open. The PR targets a real startup stability bug tracked at https://github.com/openclaw/openclaw/issues/70533 and supplies useful before/after proof, but the new scoped metadata path can drop manifest-only `activation.onConfigPaths` startup owners such as the browser plugin when `plugins.allow` is restrictive.

## What This Changes

The PR adds scoped plugin-id metadata snapshots and wires gateway startup lookup to read only a conservative installed-index manifest subset when a startup allowlist can be proven.

## Best Possible Solution

Keep the metadata read scoping, but only when all startup activation contracts are represented in the installed index; otherwise fall back unscoped or add indexed config-path metadata, with regression coverage for `browser.enabled` plus a restrictive `plugins.allow`.

## Reproduction Assessment

Yes. The central issue is source-reproducible from the current lookup path and supported by reporter data, and the introduced regression is source-reproducible from current tests/docs that require `activation.onConfigPaths` to bypass restrictive allowlists.

## Solution Assessment

No, not as written. The scoping direction is the right maintainable fix, but the scope must preserve all startup activation contracts or fall back unscoped when the installed index cannot prove them.
