## Summary

Keep this PR open. The second commit appears to fix the earlier full-startup snapshot gap and the proof is now useful, but the new scoped snapshot memo can still reuse the wrong plugin scope when env-derived channel configuration changes between calls.

## What This Changes

The PR adds Installed Plugin Index-derived plugin-id scoping for gateway startup metadata snapshots, persists startup config-path metadata, and adds regression coverage for restrictive plugin allowlists and fallback paths.

## Best Possible Solution

Keep the Installed Plugin Index scoping direction, but make the cache key or memo lookup depend on the resolved startup scope, including env-derived channel signals, and add regression coverage for that cache transition.

## Reproduction Assessment

Yes, at source level. The PR body gives before/after manifest-count proof for the linked startup issue, and the remaining cache bug follows from env-sensitive channel detection feeding a pluginIdScope whose memo key ignores those env signals.

## Solution Assessment

No, not yet. The scoping approach is directionally right, but the memoization must be keyed by the resolved scope or its env-sensitive inputs before this is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Include env-derived scope inputs in the memo key:** `src/plugins/gateway-startup-plugin-ids.ts:691-696`
  - body: `createGatewayStartupMetadataPluginIdScope.resolve()` uses `params.env` to add env-configured channels, but the hashed key here excludes those env signals. Since `loadPluginMetadataSnapshot` memoizes by `pluginIdScope.key`, a call with a restrictive allowlist and no channel env var can cache a scoped snapshot, then a later same-process call with an env token for a channel can reuse that narrower snapshot and never load the channel manifest. Key the memo by the resolved scope or by the env-derived channel presence inputs, and cover that transition.
  - confidence: 0.86
