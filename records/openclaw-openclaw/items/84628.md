## Summary

Keep open. The startup-scoping optimization is well targeted and the PR body now includes useful before/after gateway-path output, but the patch is not merge-ready because a scoped startup snapshot can still be reused by ordinary config-based metadata callers as if it were complete.

## What This Changes

The PR adds startup plugin-id scoping to plugin metadata snapshot loading, persists installed-index config-path activation hints, and adds regression coverage for restrictive gateway startup allowlists and fallback cases.

## Best Possible Solution

Keep the scoped startup manifest optimization, but make current metadata snapshot reuse scope-aware so unscoped/control-plane callers reload full metadata unless they request an exact scoped set, with regression coverage for post-startup manifest and contract queries.

## Reproduction Assessment

Yes, at source level. Current main builds startup metadata before narrowing, and the PR body provides before/after diagnostic output; the remaining review finding is also traceable from the new scoped snapshot field through existing current-snapshot reuse callers.

## Solution Assessment

No, not yet. The startup scoping approach is the right direction, but the implementation must prevent partial startup snapshots from serving unscoped or differently scoped metadata callers.

- **[P1] Require exact scope before reusing current snapshots:** `src/plugins/current-plugin-metadata-snapshot.ts:109-113`
