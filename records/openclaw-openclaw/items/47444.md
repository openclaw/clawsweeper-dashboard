## Summary

Keep this open. Current main still has a shared sanitizer gap for the reported workflow leak shape: nested plural XML `<function_calls><invoke ...>` payloads are preserved on the shared assistant-visible path unless a per-call opt-in is passed, and `<function_response>` is not handled.

## What This Changes

The issue requests hiding workflow-generated `<function_calls>`, `<invoke>`, `<function_response>`, raw command text, and intermediate execution output from user-visible workflow/chat replies.

## Best Possible Solution

Extend the shared assistant-visible sanitizer and extraction coverage so standalone workflow envelopes with plural XML wrappers and paired response blocks are stripped before any user-facing delivery surface sees them, while preserving literal documentation/prose examples.

## Reproduction Assessment

Yes. Current tests and source provide a high-confidence static reproduction: the exact nested plural XML shape is expected to survive by default, the shared profiles do not enable the opt-in, and no `<function_response>` handler exists.

## Solution Assessment

Yes. The maintainable fix belongs in the shared sanitizer/extraction path, not in another channel-local workaround; WhatsApp’s opt-in helper proves the narrower behavior is feasible but does not cover Telegram or shared history delivery.
