## Summary

Keep open. Current main strips the reported nested `<function_calls><invoke ...>` wrapper, but the paired `<function_response>...</function_response>` envelope from the report is still not recognized by the shared/user-facing sanitizer or covered by regression tests, so raw workflow response content can still reach visible replies.

## What This Changes

The issue asks OpenClaw to prevent workflow/agent replies from showing raw `<function_calls>` and `<function_response>` tool envelopes, command text, and intermediate output in user-visible chat.

## Best Possible Solution

Extend the shared visible-text sanitizer to remove complete workflow response envelopes such as `<function_response>...</function_response>` at the delivery/history boundary, with regression tests that preserve literal prose and code examples.

## Reproduction Assessment

Yes, by source inspection. A visible reply containing `<function_response>...</function_response>` has no matching sanitizer tag handler or regression test on current main, while the adjacent `<function_calls><invoke ...>` case is already stripped.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the shared sanitizer and regression coverage rather than add a Telegram- or workflow-local workaround, because the affected delivery path already funnels through the shared user-facing sanitization layer.
