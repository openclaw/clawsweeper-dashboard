## Summary

Keep this open. Current main partially covers the reported nested `<function_calls><invoke ...>` wrapper in the agent user-facing sanitizer, but the paired `<function_response>` envelope from the report has no sanitizer or regression coverage, so raw workflow response content can still reach user-visible text.

## What This Changes

The issue asks OpenClaw to strip internal workflow tool-call and tool-response envelopes from user-visible agent replies so only the final human-readable answer is shown.

## Best Possible Solution

Extend the shared/user-facing sanitizer to remove complete workflow response envelopes, including `<function_response>...</function_response>`, while preserving literal code/prose examples.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a user-visible reply containing `<function_response>...</function_response>` reaches the current sanitizer paths without any matching tag handler or regression test, while the sibling `<function_calls><invoke ...>` block is now stripped by the agent user-facing sanitizer.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the shared/user-facing sanitizer and extraction coverage, not add another channel-local workaround; the current opt-in function-call wrapper coverage shows the right layer and preserves literal examples.
