## Summary

Keep this open. Current main still only partially covers the reported leak: it strips the call-side XML wrappers but does not recognize the reported `<function_response>` response envelope, so execution output can still remain visible.

## What This Changes

The issue asks OpenClaw to prevent workflow replies from exposing internal tool-call and tool-response XML scaffolding in user-visible chat.

## Best Possible Solution

Extend the shared assistant-visible/user-facing sanitizer to strip machine-generated response/result XML envelopes, including `<function_response>`, while preserving literal code examples.

## Reproduction Assessment

Yes, at source level. Current main removes the reported `<function_calls>` wrapper but does not recognize `<function_response>`, and Telegram’s plain-text sanitizer would not remove that block as internal response content.

## Solution Assessment

Yes, the narrow maintainable solution is to extend the existing shared sanitizer and regression tests rather than adding a Telegram-only workaround or changing workflow behavior.
