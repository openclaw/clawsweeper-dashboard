## Summary

Keep this PR open, but it should not merge as-is: the new bounded Slack replies call can return the oldest messages in the time range while the existing contract needs the latest messages before the current reply.

## What This Changes

The PR changes Slack thread-history preparation to use a single `conversations.replies` call bounded by `latest` when the current message timestamp is known, with tests for bounded and fallback pagination.

## Best Possible Solution

Preserve the latest-N-before-current thread-context contract while reducing Slack calls only with an implementation and regression test that model Slack's earliest-first time pagination, then add redacted real Slack proof.

## Reproduction Assessment

Yes, source inspection plus Slack's API contract gives a high-confidence reproduction path: use a thread with more than `limit` earlier replies, pass `currentMessageTs`, and the PR stops after Slack's earliest page before the current message. I did not run a live Slack workspace repro.

## Solution Assessment

No. The single `latest`-bounded call is a useful optimization idea, but it does not preserve the existing latest-N thread-context behavior; the safer path is to keep paginating or otherwise prove the selected window is the newest prior window.
