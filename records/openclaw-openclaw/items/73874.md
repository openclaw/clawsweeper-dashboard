## Summary

Keep open. The reporter has continued reproducing the Windows Docker event-loop stall through v2026.4.29, and current main contains adjacent runtime-deps, readiness, and diagnostic improvements rather than a proven root-cause fix for the post-ready Slack/HTTP/WS starvation.

## What This Changes

The issue requests investigation and repair for a Windows Docker Gateway stall where HTTP probes, internal WebSocket RPC, Slack Socket Mode, and session delivery become unresponsive after startup.

## Best Possible Solution

Keep this as an active Windows Docker gateway regression until maintainers capture a CPU profile or main-thread stack during a liveness warning, move or bound the blocking work off the dispatch path, and verify /healthz, /readyz, Gateway WS RPC, and Slack Socket Mode stay responsive through startup and post-ready quiet periods.
