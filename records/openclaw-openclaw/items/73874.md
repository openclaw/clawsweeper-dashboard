The reporter’s v2026.4.29 retest still separates fixed runtime-deps/bind-mount symptoms from remaining 135s startup and 38s post-ready event-loop blocks, while current main shows diagnostics and adjacent mitigations rather than proof that Windows Docker HTTP/WS/Slack dispatch starvation is fixed.

## What This Changes

The issue reports Windows Docker Desktop gateway event-loop stalls that leave HTTP, WebSocket, Slack, and session dispatch intermittently unresponsive after the listener is bound.

## Best Possible Solution

Keep this as the focused tracker until maintainers capture a current-main Windows Docker Desktop CPU profile or main-thread stack during an event-loop-delay spike, then fix the specific startup or periodic work that blocks the event loop and validate /healthz, /readyz, Gateway WS RPC, Slack Socket Mode, and a Slack DM through the same setup.
