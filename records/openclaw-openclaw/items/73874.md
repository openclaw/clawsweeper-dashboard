## Summary

Keep open. The reporter's latest update says v2026.4.29 still reproduces long Windows Docker event-loop stalls after the named-volume/runtime-deps fixes, and current main shows additional mitigations and diagnostics rather than a proven root-cause fix for the 38-135s blocked-loop windows.

## What This Changes

The issue reports a Windows Docker Gateway regression where HTTP probes, WebSocket RPC, Slack delivery, and plugin startup can stall behind long event-loop blocks after upgrading from 2026.4.23.

## Best Possible Solution

Keep this as an active Windows Docker Gateway liveness regression. Maintainers should reproduce with the reporter's named-volume compose layout, capture a Node CPU profile or main-thread stack during a liveness warning, move or bound the blocking work off the dispatch path, then verify /healthz, /readyz, Gateway WS RPC, and Slack Socket Mode through startup and a post-ready quiet period.

## Reproduction Assessment

Yes. The issue provides a concrete Windows 11 + Docker Desktop compose setup, version comparisons, named-volume retests, and measured v2026.4.29 event-loop delay logs; this review did not independently run that Windows Docker environment.

## Solution Assessment

Unclear. The current mitigation direction is appropriate for runtime-deps pressure and observability, but the best final fix should be chosen after profiling the blocking stack rather than adding more waits or route-level timeouts.
