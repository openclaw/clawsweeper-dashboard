## Summary

Keep open. The reporter has continued to reproduce the Windows Docker Gateway event-loop starvation through v2026.4.29, and current main shows useful mitigations and diagnostics but not a proven root-cause fix for the post-ready HTTP/WS and Slack starvation.

## What This Changes

The issue reports a Windows 11 Docker Desktop Gateway regression where HTTP and WebSocket requests accept connections but can stop responding after startup, causing Slack delivery and control-plane RPCs to stall.

## Best Possible Solution

Keep this as an active Windows Docker Gateway regression until maintainers capture a CPU profile or main-thread stack during a liveness spike, isolate the blocking subsystem, move or bound that work off the dispatch path, and verify /healthz, /readyz, Gateway WS RPC, and Slack Socket Mode remain responsive through startup and post-ready quiet periods.

## Reproduction Assessment

Yes. The reporter provided a concrete Windows 11 Docker Desktop compose setup, release-by-release retests through v2026.4.29, named-volume follow-up, endpoint probes, and measured event-loop delay values; this review did not independently run that Windows environment.

## Solution Assessment

Unclear. The current diagnostics, early probe routing, runtime-deps volume staging, client event-loop readiness waits, and readiness event-loop health are useful mitigations, but the best fix should remove or bound the blocking main-thread work once the stack is known.
