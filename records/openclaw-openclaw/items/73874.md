## Summary

Keep open. The latest reporter update still reproduces the Windows Docker event-loop stall on v2026.4.29, and current main shows adjacent mitigations and diagnostics rather than a proven root-cause fix. The remaining failure needs maintainer-led profiling on the reporter's Windows Docker setup before choosing a narrow code change.

## What This Changes

The issue asks maintainers to diagnose and fix a Windows Docker Desktop gateway regression where HTTP/WS dispatch, Slack Socket Mode, and queued agent sessions stall behind severe event-loop blocking after startup.

## Best Possible Solution

Keep this as an active Windows Docker gateway performance regression. Maintainers should capture a CPU profile or main-thread stack when liveness warnings exceed a threshold, identify the synchronous or starvation source, move or bound that work off the dispatch path, and verify HTTP probes, WebSocket RPC, and Slack Socket Mode stay responsive through startup and post-ready quiet periods.

## Reproduction Assessment

Yes. The reporter provides a concrete Windows 11 + Docker Desktop compose layout and repeated measurements through v2026.4.29, including event-loop delay values and Slack/HTTP probe symptoms; this review did not independently run that Windows environment.

## Solution Assessment

Unclear. The current direction of diagnostics, client-side readiness waits, runtime-deps cleanup, and stuck-session recovery is useful, but the best final fix should remove or offload the blocking main-thread work once the stack is known rather than adding more waits or route-level timeouts.
