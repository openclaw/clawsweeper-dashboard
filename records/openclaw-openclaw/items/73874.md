## Summary

Keep open. Current main has important adjacent mitigations, including async session transcript I/O and removal of bundled runtime-deps startup installs, but the Windows Docker event-loop block reported through v2026.4.29 is not proven fixed on the reporter-style setup and still needs profile-backed root-cause confirmation.

## What This Changes

The issue reports a Windows Docker Desktop Gateway regression where HTTP/WS dispatch, internal WebSocket RPC, Slack Socket Mode, and session delivery starve after the listener reports ready.

## Best Possible Solution

Keep this open until maintainers run a current-main package on the reporter-style Windows Docker setup, capture a main-thread stack or CPU profile during an event-loop spike, and bound or move the identified startup/periodic work off the dispatch path.

## Reproduction Assessment

Yes. The issue has concrete Windows 11 Docker Desktop compose steps, release-by-release rollback/retest evidence through v2026.4.29, and measured eventLoopDelayMaxMs spikes; this review did not independently run that Windows environment.

## Solution Assessment

Unclear. Async transcript I/O is the right fix class for one suspected blocker, but this issue also reproduced on a cleaner bot and during quiet post-ready windows, so the final fix should be selected from profile evidence rather than inferred from adjacent mitigations.
