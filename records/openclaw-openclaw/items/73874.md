## Summary

Keep open. Current main has removed the active bundled `plugin-runtime-deps` startup/reload machinery and contains several diagnostics and responsiveness mitigations, but the reporter’s v2026.4.29 retest and the still-open umbrella discussion both say the Windows Docker event-loop/dispatch wedge is not proven fixed. The remaining problem needs a profile or main-thread stack from the affected setup before a narrow fix can be chosen safely.

## What This Changes

The issue reports a Windows Docker Desktop Gateway regression where HTTP/WS requests, internal WebSocket RPC, Slack Socket Mode, and session handling starve after the listener reports ready, with adjacent runtime-deps and permission symptoms partly mitigated by later releases.

## Best Possible Solution

Keep this as an active Windows Docker Gateway regression until maintainers run a current-main package against the reporter-style setup, capture a CPU profile or main-thread stack during a liveness spike, identify the blocking startup or periodic task, and move, bound, or defer that work so `/healthz`, `/readyz`, Gateway WS RPC, Slack Socket Mode, and DMs remain responsive through startup and quiet post-ready periods.

## Reproduction Assessment

Yes. The issue has a concrete Windows 11 Docker Desktop compose setup, release-by-release rollback/retest evidence through v2026.4.29, and measured `eventLoopDelayMaxMs` spikes; this review did not independently run that Windows environment.

## Solution Assessment

Unclear. Current main’s dependency rearchitecture and session-yielding work are plausible partial mitigations, but the best fix should be selected from profile evidence of the blocked stack rather than treating runtime-deps cleanup or probe ordering as sufficient.
