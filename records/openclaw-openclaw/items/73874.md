## Summary

Keep open. The reporter continued to reproduce the Windows Docker Gateway starvation through v2026.4.29 after adopting the documented named-volume runtime-deps layout, and current main contains diagnostics/runtime-deps mitigations rather than a proven root-cause fix for the post-ready event-loop blocks that starve HTTP, Gateway WS RPC, Slack Socket Mode, and session lanes.

## What This Changes

The issue requests investigation and a fix for Windows Docker Gateway HTTP/WS dispatch starvation that makes probes, Slack internal WebSocket handshakes, and Slack DM processing hang after the Gateway logs ready.

## Best Possible Solution

Keep this as an active Windows Docker Gateway regression until maintainers capture a CPU profile or main-thread stack during a liveness spike, identify the synchronous or effectively blocking startup/post-ready work, move or bound that work off the dispatch path, and verify /healthz, /readyz, Gateway WS RPC, and Slack Socket Mode stay responsive on the reporter-style setup.

## Reproduction Assessment

Yes. The reporter gives a concrete Windows 11 + Docker Desktop compose setup, release-by-release rollback/retest evidence through v2026.4.29, and measurable eventLoopDelayMaxMs spikes; this review did not independently run that Windows environment.

## Solution Assessment

Unclear. Current main adds useful diagnostics, readiness visibility, Docker runtime-deps layout, and staging mitigations, but the best final fix should be chosen after profiling the blocked main-thread stack rather than adding more waits or treating adjacent runtime-deps fixes as sufficient.
