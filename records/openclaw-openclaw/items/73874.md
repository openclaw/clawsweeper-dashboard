## Summary

Keep open. The latest reporter follow-up says the blocker still reproduces on the latest shipped release, and current main shows several useful mitigations and diagnostics but not a source-backed proof that the Windows Docker event-loop stall has been removed. The remaining symptom is broader than stale runtime-deps locks or Slack timeout tuning: the event loop is blocked for tens to hundreds of seconds, making both HTTP probes and WebSocket dispatch unreachable while Slack DMs stall.

## What This Changes

The issue requests investigation and a fix for Windows Docker Desktop gateway event-loop stalls that leave HTTP probes, WebSocket RPCs, Slack Socket Mode, and queued sessions unresponsive after startup despite adjacent runtime-deps mitigations.

## Best Possible Solution

Keep this as an active Windows Docker gateway starvation investigation. Maintainers should reproduce on current main with the named runtime-deps volume, capture startup trace plus diagnostics timeline and a CPU profile during the liveness warning window, then land a narrow fix that removes or offloads the blocking startup/periodic work rather than broadening timeouts around it.

## Reproduction Assessment

Yes. The reporter provided a concrete Windows 11 + Docker Desktop compose layout and reconfirmed the remaining failure on v2026.4.29 with named runtime-deps storage, event-loop delay metrics, hanging probes, and Slack DM impact; this review did not independently run that Windows setup.

## Solution Assessment

Unclear. Current main's named-volume runtime-deps storage, event-loop-aware client starts, readiness diagnostics, stale-lock handling, and Slack timeout tuning are useful mitigations, but the best final fix should be profile-driven and remove the blocking hot path rather than only adding route or handshake waits.
