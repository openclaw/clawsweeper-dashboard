## Summary

This PR fixes the documented map lookup divergence, but it does not fix the linked issue’s central active-memory behavior: an explicit `thinking: "off"` is still lost, the legacy path still resolves the request as `high`, and the new fallback then translates that to provider-native `on`. The branch should remain open for a complete fix and real LM Studio proof.

## What This Changes

The PR makes the legacy OpenAI completions request builder fall back to `compat.reasoningEffortMap` when `model.thinkingLevelMap` is absent across its reasoning payload formats.

## Best Possible Solution

Carry the caller’s explicit thinking selection into the legacy completions options, resolve provider-native mappings through one typed normalization contract, and add focused tests plus a redacted LM Studio request/log proving that off stays off and enabled levels map correctly.

## Reproduction Assessment

Yes, at source level: active-memory passes `thinkLevel: "off"`, but the affected legacy request path receives/defaults a separate reasoning effort and currently serializes `high`; the linked issue also supplies the resulting LM Studio warning. This review did not run a live LM Studio reproduction.

## Solution Assessment

No. The fallback corrects one unreachable-map divergence, but mapping the defaulted `high` value to `on` cannot honor the caller’s explicit off selection and therefore does not solve the complete reported behavior.
