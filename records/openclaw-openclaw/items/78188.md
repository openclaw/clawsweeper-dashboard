## Summary

Current main already has the supported switch to make Telegram DMs truly silent: allow direct silent replies via `silentReply.direct`, rather than changing only the rewrite text behavior.

## What This Changes

The issue reports Telegram DMs repeatedly receiving the silent-reply fallback text despite attempts to disable silent replies.

## Best Possible Solution

Keep the conservative default for direct chats, and use `surfaces.telegram.silentReply.direct: "allow"` when an operator wants Telegram DMs to stay quiet on `NO_REPLY` turns.

## Reproduction Assessment

Yes from source for the default behavior: a no-visible-response Telegram DM flows through the shared outbound silent-reply plan and becomes visible fallback text while direct silence is disallowed. The specific user setup was not live-tested because the issue has no logs or runnable Telegram credentials.

## Solution Assessment

Yes; the existing policy switch is the narrowest maintainable solution because it changes only the operator-selected Telegram/direct silence behavior and does not relax the default direct-chat empty-response guard for everyone.
