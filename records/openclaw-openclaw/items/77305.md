## Summary

Keep open. Current main still contains the reported restart-recovery and WebChat delivery constraints, and the WebChat docs describe replies as returning to WebChat, so this is a focused bug rather than cleanup closure.

## What This Changes

The issue reports that WebChat restart auto-resume completes internally after a gateway restart but no longer delivers the resumed response to the WebChat UI in 5.3.

## Best Possible Solution

Make restart-recovered WebChat sessions visibly complete through the internal chat event and durable transcript/history path while preserving WebChat's non-outbound channel boundary.

## Reproduction Assessment

No live end-to-end reproduction was run. Source inspection matches the reported path: restart recovery resumes with `deliver: false`, and the agent method keeps WebChat session-only even when delivery is requested.

## Solution Assessment

No, the issue's broad `deliver: true` or removal of the WebChat exclusion is not clearly the safest fix. The maintainable repair is WebChat-specific resume visibility through existing internal chat events and `chat.history`, without making WebChat an external delivery target.
