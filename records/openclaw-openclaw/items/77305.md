## Summary

Keep open. Current main still has a source-reproducible WebChat restart-recovery visibility gap: recovery resumes a canonical main-session agent run, but reconnecting WebChat state can lose the tracked run and ignore untracked events unless the session key matches exactly, while the docs promise WebChat replies and reconnect continuity.

## What This Changes

WebChat restart recovery should resume interrupted sessions and show the recovered assistant response after gateway reconnect without requiring a manual follow-up message.

## Best Possible Solution

Make restart recovery surface recovered main-session output through WebChat’s internal chat event/history path, including canonical/alias session matching after reconnect, while keeping WebChat non-deliverable externally.

## Reproduction Assessment

Yes, at source level: restart recovery starts a session-only agent run, reconnect clears the UI’s tracked run id, and the UI ignores untracked chat events unless the session key matches exactly. I did not run a live end-to-end gateway restart/WebChat session.

## Solution Assessment

No, the broad suggested fix of setting `deliver: true` or removing the WebChat exclusion is not the safest path. The narrower maintainable fix is WebChat-internal resume visibility through canonical session matching, chat events, and history refresh.
