## Summary

Keep open. Current main still resumes restart-aborted main sessions with session-only agent delivery and keeps WebChat out of outbound delivery, while the WebChat contract expects replies to be visible through its internal event/history path.

## What This Changes

Restore WebChat restart auto-resume so a recovered main-session response after gateway restart is visible in the WebChat UI, matching the previously reported 5.2 behavior.

## Best Possible Solution

Make restart-recovered WebChat turns visibly complete through the existing internal chat event and transcript/history mechanisms, with regression coverage, while preserving WebChat’s non-outbound delivery boundary.

## Reproduction Assessment

Unclear for a live end-to-end run because this read-only pass did not restart a real Gateway/WebChat session. Source inspection gives a strong reproduction path: restart recovery resumes with `deliver: false`, and current Gateway tests lock WebChat delivery to session-only.

## Solution Assessment

No, the issue’s broadest possible fix is not the safest path. The better fix is targeted WebChat resume visibility through the existing internal event/history path, without turning WebChat into an outbound delivery channel.
