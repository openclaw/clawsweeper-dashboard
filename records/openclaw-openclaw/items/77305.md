## Summary

Current main now implements the WebChat restart-resume path through WebChat chat events and durable transcript/history projection rather than generic outbound delivery, so the reported `deliver: false` and WebChat non-deliverable guard are no longer a close blocker for normal resumed replies.

## What This Changes

Report that WebChat sessions stopped showing auto-resumed agent responses after a gateway restart in 5.3 because restart recovery runs with `deliver: false` and WebChat is not a generic outbound channel.

## Best Possible Solution

Keep the current implementation: WebChat remains an internal channel, while restarted main-session agent runs become visible through chat events and durable `chat.history`.

## Reproduction Assessment

No, there is not a high-confidence current-main failing reproduction path. I did not run a live gateway restart, and current source plus regression coverage now show resumed main-session agent events are projected back to WebChat without making WebChat a generic outbound channel.

## Solution Assessment

Yes. The best fix is not to set broad `deliver: true` for WebChat, but to keep WebChat internal-only and route resumed agent runs through the existing chat event and durable transcript/history model, which current main now does.
