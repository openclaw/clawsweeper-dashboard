## Summary

Keep this open. Current main still does not pass the requested `reply_to_guid` to legacy imsg; it strips visible directive tags and sends `reply_to` metadata instead, while public imsg main still treats `reply_to_guid` as output metadata and does not consume it in `send`.

## What This Changes

Request legacy iMessage sends to turn OpenClaw `replyToId` into imsg native `reply_to_guid` threading metadata while keeping the delivered text free of inline reply tags.

## Best Possible Solution

Keep this issue open until the imsg send contract is pinned. If imsg supports native replies, update `extensions/imessage/src/send.ts` to send the documented field and add a fake-client regression test for both sanitized text and JSON-RPC params; if not, coordinate the imsg API first or steer users to the already-documented BlueBubbles native reply path.

## Reproduction Assessment

Yes for the OpenClaw plumbing gap: a stubbed `sendMessageIMessage` call with `replyToId` would show current params include `reply_to` but not `reply_to_guid`. Native Messages UI threading still needs a live macOS/imsg verification path after the dependency contract is confirmed.

## Solution Assessment

Unclear. Passing `reply_to_guid` directly is only the best fix if imsg documents and ships that as a send-side field; otherwise the safer solution is to first add or pin upstream imsg support, then wire OpenClaw to that contract.
