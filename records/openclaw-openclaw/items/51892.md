## Summary

Keep this open. Current main still forwards iMessage reply metadata as `reply_to`, not the requested `reply_to_guid`, and the public imsg send contract I checked does not yet document or consume either field for native threaded sends.

## What This Changes

The issue asks the legacy iMessage send path to pass `replyToId` to imsg as `reply_to_guid` so outbound replies become native iMessage threaded replies.

## Best Possible Solution

Keep this open until the imsg send contract is pinned; then either wire OpenClaw to the documented native reply field with a fake-client regression test, or coordinate upstream imsg support and direct new deployments to BlueBubbles.

## Reproduction Assessment

Yes for the OpenClaw plumbing gap: source inspection shows a `replyToId` send currently produces `reply_to`, not `reply_to_guid`. Native Messages UI threading still needs a live macOS/imsg check after the dependency contract is confirmed.

## Solution Assessment

Unclear. Passing `reply_to_guid` is only the right fix if imsg documents and consumes that field for `send`; public imsg sources currently present it as read-only output metadata.
