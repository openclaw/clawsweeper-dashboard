## Summary

Keep this open. Current main has several building blocks, including dashboard token bootstrap, daemon install in non-interactive onboarding, and Telegram username resolution, but it still does not provide the requested one-line local bootstrap that configures Telegram owner access end to end. The external sentinel-stack-public prompt shared in the discussion is useful reference material, but it is not a replacement for an official OpenClaw installer/onboarding surface.

## What This Changes

Request a supported one-line local setup that installs or refreshes the daemon, opens or prints authenticated dashboard access, configures Telegram, and preauthorizes the intended owner without weakening pairing.

## Best Possible Solution

Design a first-class bootstrap/onboarding path that composes daemon setup, dashboard token delivery, Telegram token setup, and numeric-owner allowlist or explicit preapproval with tests and clear docs that pairing remains manual unless preauthorized.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs checks give high-confidence current behavior: users still need separate onboarding, channel configuration, and Telegram pairing or allowlist steps even though some building blocks now exist.

## Solution Assessment

Unclear as an implementation path. The requested goal is valid, but maintainers should first choose whether the supported surface belongs in `openclaw onboard --non-interactive`, `openclaw channels add`, the installer, or a dedicated bootstrap command.
