## Summary

Keep open. Current main has useful building blocks, including Telegram target resolution and clearer pairing/owner docs, but it still does not provide the requested single supported local bootstrap surface for daemon install, dashboard auth, Telegram bot setup, and owner allowlisting.

## What This Changes

The issue requests a supported one-line local setup that installs or refreshes the gateway, provides dashboard auth, configures Telegram, and pre-authorizes the Telegram owner non-interactively.

## Best Possible Solution

Add a first-class local bootstrap surface that composes daemon install or refresh, dashboard auth URL output, Telegram bot setup, Telegram username-to-ID resolution where possible, durable numeric `allowFrom`, `dmPolicy: "allowlist"`, and `commands.ownerAllowFrom`, with docs that explicitly say unattended pairing is not bypassed.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The high-confidence static check is that current `onboard`, `channels add`, `dashboard`, Telegram setup, and pairing/owner code remain separate surfaces without one composed local bootstrap path.

## Solution Assessment

Unclear because there is no patch to evaluate. The maintainable direction is to reuse the existing Telegram resolver, setup adapter, dashboard token helper, and pairing/owner semantics rather than duplicating lookup logic or bypassing owner approval boundaries.
