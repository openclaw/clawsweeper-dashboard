## Summary

Keep open: current main and the latest release have separate pieces for non-interactive gateway/daemon setup, dashboard token URL delivery, Telegram token setup, numeric allowlists, pairing, and Telegram username resolution, but no supported one-command local bootstrap that safely configures Telegram owner access.

## What This Changes

The issue requests a supported one-line local bootstrap path that installs OpenClaw, starts or refreshes the gateway daemon, provides dashboard auth, configures Telegram, and pre-authorizes the intended owner safely.

## Best Possible Solution

Define a maintainer-approved bootstrap/onboard path that composes gateway, daemon, dashboard, Telegram token, resolver, and numeric-owner allowlist/preapproval without weakening pairing, then document and test fresh-install and rerun behavior.

## Reproduction Assessment

Yes from source inspection: current non-interactive onboarding stops at workspace/auth/gateway/daemon/health and does not accept Telegram owner enrollment inputs; Telegram username resolution now exists, so the remaining missing behavior is the absent composition surface.

## Solution Assessment

Unclear as filed: a one-line path is reasonable, but maintainers need to choose the product/security shape before deciding between onboard flags, a bootstrap command, or a documented preapproval flow.
