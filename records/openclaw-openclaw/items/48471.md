## Summary

Keep open: current main has several underlying pieces, including non-interactive local gateway setup, dashboard token handoff, Telegram allowlist setup, and a Telegram target resolver, but it still does not provide the requested single supported bootstrap surface. The recent external sentinel-stack-public comment is useful reference evidence for the allowFrom-preseed approach, not a replacement for an OpenClaw-owned product path.

## What This Changes

The issue requests a supported one-line local bootstrap path that installs OpenClaw, starts or refreshes the local gateway, bootstraps dashboard auth, configures Telegram, and pre-authorizes the intended owner.

## Best Possible Solution

Define a first-class bootstrap/onboarding surface that composes daemon setup, dashboard token handoff, Telegram bot config, numeric owner allowlist, and explicit docs that DM pairing remains the guest/on-demand approval path.

## Reproduction Assessment

Not applicable: this is a feature request, not a report of broken existing behavior. Source inspection confirms the underlying pieces exist separately but not as the requested one-line flow.

## Solution Assessment

Unclear: the issue gives plausible implementation directions, but choosing installer extension versus onboard flags versus a dedicated bootstrap command needs maintainer product and security judgment. The safest direction is to preserve numeric allowFrom/preapproval as the unattended owner path and avoid weakening DM pairing.
