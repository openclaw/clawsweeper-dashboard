## Summary

Keep open: current main and v2026.5.22 still provide separate onboarding, daemon, dashboard, channel setup, and Telegram owner allowlist pieces, but not the requested supported one-line bootstrap that composes them safely. The external sentinel-stack-public guide in the discussion is useful validation of the numeric allowlist-preseed approach, and https://github.com/openclaw/openclaw/issues/7679 covers a narrower Telegram default/owner-capture decision, but neither supersedes the broader bootstrap feature.

## What This Changes

The issue requests a supported one-line local bootstrap that installs or refreshes OpenClaw, starts the gateway, provides dashboard auth, configures Telegram, and preauthorizes the intended Telegram owner.

## Best Possible Solution

Define an explicit bootstrap/onboarding path that composes daemon setup, dashboard token delivery, Telegram bot setup, and numeric owner allowlist or token-verified preapproval, with docs and tests preserving the DM pairing trust model.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Current source and docs give high-confidence current-state proof that the one-line daemon plus dashboard auth plus Telegram owner bootstrap still requires separate setup surfaces.

## Solution Assessment

Unclear as an implementation path. The goal is valid, but maintainers should choose the official bootstrap surface and Telegram preauthorization model before implementation rather than letting automation pick flags, a wrapper, or a trust-capture mechanism ad hoc.
