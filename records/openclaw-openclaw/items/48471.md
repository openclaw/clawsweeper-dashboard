## Summary

Keep this open. Current main has daemon install, dashboard token delivery, Telegram token setup, and a Telegram username resolver as separate building blocks, but it still lacks a supported one-line bootstrap that configures Telegram owner access end to end without weakening pairing semantics.

## What This Changes

Request a documented one-line local setup path that installs OpenClaw, starts or refreshes the daemon, opens dashboard auth, configures Telegram, and preauthorizes the intended Telegram owner.

## Best Possible Solution

Design a first-class bootstrap or onboarding path that composes daemon setup, dashboard token delivery, Telegram token setup, and numeric-owner allowlist or explicit preapproval, with docs and regression tests that keep pairing manual unless preauthorized.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs checks give a high-confidence current-state proof that the requested end-to-end bootstrap surface is still missing on current main.

## Solution Assessment

Unclear as an implementation path. The goal is valid, but maintainers should choose whether the supported surface belongs in non-interactive onboard flags, channel add flags, installer handoff, a dedicated bootstrap command, or a combination.
