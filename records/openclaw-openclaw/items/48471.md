## Summary

Keep this open. Current main has several pieces of the requested flow, including daemon install in non-interactive onboarding, dashboard token delivery, and a Telegram username resolver, but it still lacks a supported one-line bootstrap that configures Telegram owner access end to end.

## What This Changes

Request a documented one-command local setup path that installs/refreshes OpenClaw, starts the daemon, bootstraps dashboard auth, configures Telegram, and preauthorizes the intended Telegram owner safely.

## Best Possible Solution

Design a first-class bootstrap/onboarding path that composes daemon setup, dashboard token delivery, Telegram token setup, and numeric-owner allowlist or explicit preapproval, with docs and regression tests that keep pairing manual unless preauthorized.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs checks give a high-confidence current-state proof that users still need separate setup surfaces for onboarding, dashboard auth, Telegram config, and owner allowlisting or pairing.

## Solution Assessment

Unclear as an implementation path. The goal is valid, but maintainers should choose whether the supported surface belongs in non-interactive onboard flags, channel add flags, installer handoff, a dedicated bootstrap command, or a combination.
