## Summary

Keep open: current main now covers the Telegram resolver building block, but it still does not provide the requested first-party one-line local bootstrap across daemon install, dashboard auth URL, Telegram bot setup, durable owner allowlist, and command-owner authority.

## What This Changes

Request a supported single-command local bootstrap path that installs or refreshes OpenClaw, starts the gateway, exposes dashboard auth, configures Telegram, and pre-authorizes the intended owner safely.

## Best Possible Solution

Add a deliberate first-class bootstrap/onboarding path that composes the existing daemon, dashboard token URL, Telegram resolver, numeric allowlist, and command-owner surfaces while documenting that unattended DM pairing is not bypassed.

## Reproduction Assessment

Not applicable as a bug reproduction. The feature gap is statically verifiable on current main by inspecting onboard flags/options, non-interactive local setup, channels add/resolve, Telegram access-control docs, and the Telegram resolver surface.

## Solution Assessment

Unclear because there is no patch to evaluate. The maintainable direction is to reuse the now-existing Telegram resolver and current access-control contracts inside a product-approved bootstrap surface rather than weakening pairing or adding an external-only workaround.
