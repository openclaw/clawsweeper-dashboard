## Summary

Keep open: current main has useful pieces, including non-interactive gateway setup, dashboard token URLs, Telegram numeric allowlists, and a Telegram username resolver, but it still lacks one supported local bootstrap path that composes daemon setup, dashboard auth, Telegram bot setup, and owner preauthorization end to end. The external sentinel-stack-public guide is useful reference work, not an OpenClaw shipped setup surface.

## What This Changes

Request a supported one-line local bootstrap flow for installing OpenClaw, starting the daemon, opening an authenticated dashboard, configuring Telegram, and preauthorizing the intended Telegram owner.

## Best Possible Solution

Define an official bootstrap/onboarding path that composes daemon setup, dashboard token delivery, Telegram bot setup, and numeric owner allowlist or explicit preapproval, with docs and tests preserving the DM pairing trust model.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs give high-confidence current-state proof that the requested one-line bootstrap still requires separate setup surfaces.

## Solution Assessment

Unclear as an implementation path. The goal is valid, but maintainers should choose the official bootstrap surface and Telegram preauthorization model before implementation rather than letting automation pick flags, a wrapper command, or state-manifest behavior ad hoc.
