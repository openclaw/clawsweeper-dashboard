## Summary

Keep open. Current main has useful building blocks, including daemon-capable non-interactive onboarding, dashboard token URL support, Telegram numeric allowlist guidance, and a Telegram username resolver, but it still does not provide a first-class one-line localhost bootstrap that configures daemon startup, dashboard auth, Telegram bot credentials, Telegram DM owner access, and command-owner state together.

## What This Changes

Request a supported one-line local bootstrap path that installs or refreshes the gateway daemon, exposes a working dashboard auth URL, configures Telegram, and pre-authorizes the intended Telegram owner non-interactively without weakening pairing security.

## Best Possible Solution

Add a deliberate local bootstrap surface that composes daemon install/refresh, dashboard auth URL output, Telegram bot token setup, username-to-ID resolution where possible, durable numeric allowFrom with dmPolicy="allowlist", and commands.ownerAllowFrom seeding, with docs that state unattended pairing is not bypassed and tests covering the supported automation path.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. There is a high-confidence static verification path: inspect onboard flags/types, non-interactive local setup, channels add, Telegram resolver/setup, and docs on current main, which shows the composed one-line bootstrap is still missing.

## Solution Assessment

Unclear because there is no patch to evaluate. The narrow maintainable direction is to reuse the existing Telegram resolver and setup adapters behind a first-class bootstrap/onboarding surface instead of weakening pairing or duplicating Telegram lookup logic.
