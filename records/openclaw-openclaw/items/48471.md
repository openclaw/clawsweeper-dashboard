## Summary

Keep open: current main has several building blocks, including daemon install, dashboard URL handling, Telegram allowlist setup, and a Telegram resolver, but it still does not provide a first-class one-line local bootstrap that wires daemon, dashboard auth, Telegram bot setup, and owner access together. The Ryce and azfarh95 comments add useful manifest and external-guide context, but they do not satisfy the requested OpenClaw product surface.

## What This Changes

Request a supported one-line local bootstrap path for installing OpenClaw, starting or refreshing the gateway daemon, opening dashboard auth, configuring Telegram, and pre-authorizing the Telegram owner.

## Best Possible Solution

Add a deliberate core bootstrap/onboarding surface that reuses existing gateway daemon, dashboard, Telegram resolver, allowlist, and command-owner primitives, with docs that state unattended pairing is not bypassed.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Static verification on current main shows the requested composed bootstrap surface is still missing even though several lower-level pieces exist.

## Solution Assessment

Unclear because there is no patch to evaluate. The maintainable direction is a first-class onboarding/bootstrap path that composes existing secure primitives instead of weakening Telegram pairing or duplicating access-control logic.
