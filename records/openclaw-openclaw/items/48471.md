## Summary

Keep open: current main and the latest release have the separate pieces for local gateway/daemon setup, dashboard token URL delivery, Telegram channel add, Telegram username resolution, numeric allowlists, and pairing, but not the requested supported one-command local bootstrap for Telegram owner setup. The remaining work is a real setup/auth product gap and needs maintainer security direction.

## What This Changes

Request a supported one-command local setup path that installs/refreshes OpenClaw, starts the daemonized gateway, exposes dashboard auth, configures Telegram, and pre-authorizes the intended Telegram owner.

## Best Possible Solution

Define a maintainer-approved bootstrap/onboard path that composes the existing gateway, daemon, dashboard, Telegram token, and numeric-owner allowlist/preapproval pieces without weakening pairing, then document and test fresh-install and rerun behavior.

## Reproduction Assessment

Yes from source inspection for the missing capability: current onboarding exposes gateway/daemon flags, while dashboard auth, Telegram channel add, Telegram owner allowlist, and resolver flows remain separate surfaces.

## Solution Assessment

Unclear as filed: a one-line path is reasonable, but maintainers need to choose the product/security shape before deciding between onboard flags, a higher-level bootstrap command, or a documented preapproval flow.
