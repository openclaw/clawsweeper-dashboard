## Summary

Keep open. Current main still has a source-level path where a post-tool synthetic compaction marker does not pause subagent output selection, allowing the previous tool/toolResult text to remain eligible as the completion reply; the issue is also already paired with open implementation PRs, including one with closing syntax.

## What This Changes

The issue requests preventing subagent completion announcements from using stale raw tool JSON when auto-compaction completes before a post-compaction assistant reply is available.

## Best Possible Solution

Land one consolidated agents fix that treats synthetic compaction markers as continuation boundaries for subagent output selection and frozen completion capture, preserves timeout partial-progress behavior, and updates regression tests plus the user-facing changelog together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a child history with toolResult text followed by a synthetic compaction system marker and no later assistant text still leaves latestRawText selectable as subagent output on current main.

## Solution Assessment

No for the issue's narrow retry-only suggestion. The safer solution should cover the snapshot selector's compaction marker handling and the capture/frozen completion path, not only recheck delayed assistant text in runSubagentAnnounceFlow.
