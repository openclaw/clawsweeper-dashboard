## Summary

Keep open. Current main still has a source-level path where a compaction marker after a tool result does not pause subagent output selection, so stale tool/toolResult text can still be selected before the delayed assistant reply; the issue is also paired with open implementation PRs, including one with closing syntax, so it should remain open until one consolidated fix lands.

## What This Changes

The issue requests fixing a subagent auto-compaction race so delayed post-compaction assistant replies are selected instead of stale raw tool JSON during completion announcement and capture.

## Best Possible Solution

Land one consolidated agents fix that treats compaction system markers as continuation boundaries for both announce output selection and frozen completion capture, preserves timeout partial-progress behavior, and adds regression coverage for toolResult-then-compaction limbo, delayed assistant recovery, and completion freeze capture.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a child history with raw toolResult text followed by a synthetic compaction system marker and no later assistant text can still leave latestRawText selectable as the subagent output on current main.

## Solution Assessment

No for a narrow unified-retry-only fix. The maintainable solution should cover the output summarizer's compaction marker handling and the captureSubagentCompletionReplyUsing freeze path, not only recheck delayed assistant text inside runSubagentAnnounceFlow.
