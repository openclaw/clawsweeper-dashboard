## Summary

Keep open. Current main still has a source-level path where a synthetic compaction marker is not treated as a continuation boundary, so stale `latestRawText` can be selected before the post-compaction assistant reply; the issue is also paired with open implementation PRs that should be consolidated rather than bypassed: https://github.com/openclaw/openclaw/pull/74113, https://github.com/openclaw/openclaw/pull/73783, and https://github.com/openclaw/openclaw/pull/74340.

## What This Changes

The issue reports that subagent auto-compaction can let `readSubagentOutput` freeze or announce stale tool-result JSON instead of waiting for the post-compaction assistant reply.

## Best Possible Solution

Land one consolidated agents fix that treats compaction markers and successful completion capture as non-terminal until assistant output, timeout partial progress, or an explicit no-output state exists, with regression tests across direct announce and frozen-result capture.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction shape: a child history with tool output followed by a synthetic compaction system marker and no later assistant text still leaves `latestRawText` selectable, and the May 8 comment adds a real runtime confirmation of the same class.

## Solution Assessment

No, closing or landing only the narrow retry branch is not the best resolution. The maintainable fix needs to cover the shared output selector and frozen completion capture path while preserving timeout partial-progress behavior.
