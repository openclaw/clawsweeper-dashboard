## Summary

Keep open. Current main still allows stale raw tool/toolResult text to be selected after a compaction marker, and an open PR with closing syntax already targets this issue, with overlapping open PRs covering adjacent announce and frozen-capture paths.

## What This Changes

The issue reports that subagent announce/completion output can select stale raw tool-result JSON after auto-compaction instead of waiting for the post-compaction assistant reply.

## Best Possible Solution

Land one consolidated agents fix that treats compaction system markers as continuation boundaries for announce output selection and frozen completion capture, while preserving timeout partial-progress behavior and adding regression coverage for toolResult-then-compaction limbo, delayed assistant recovery, and frozen-result capture.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: a child history containing toolResult text, then the synthetic compaction system marker, and no later assistant text leaves latestRawText selectable as the subagent output.

## Solution Assessment

No for the narrow unified-retry-only direction. The safer maintainable fix should cover compaction-marker handling in output selection and the captureSubagentCompletionReplyUsing freeze path, not only recheck delayed assistant text in runSubagentAnnounceFlow.
