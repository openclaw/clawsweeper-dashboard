## Summary

Keep open. The WebChat transcript-loss bug is a valid regression and this PR targets the right Codex app-server mirror path, but the proposed thread-wide role/content fingerprint would drop legitimate later turns that repeat the same text, and the PR is maintainer-protected.

## What This Changes

The PR changes Codex app-server transcript mirroring to use thread-scoped SHA-256 role/content fingerprints for idempotency and adds regression tests for rotated scopes and snapshot reordering.

## Best Possible Solution

Use a stable logical-message identity that survives re-mirroring and snapshot reordering while still distinguishing separate same-content turns, with regression coverage for both cases.

## Reproduction Assessment

Yes. Source inspection plus the linked report give a high-confidence path: turn-scoped index keys let re-emitted prior messages bypass mirror dedupe, then WebChat history follows only the newest parent-linked branch.

## Solution Assessment

No. The direction is right, but thread-scoped role/content fingerprints are not the best fix because they collapse distinct repeated turns into one idempotency key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Keep repeated same-content turns distinct:** `extensions/codex/src/app-server/transcript-mirror.ts:51`
  - body: With the call site changing the scope to the whole thread, this key only varies by role and content. If a user sends the same message twice in one thread, the second turn gets the same idempotency key as the first and is skipped by the mirror, so WebChat can lose a real user turn. The key needs a stable logical-message discriminator that survives re-mirroring without collapsing separate same-content messages.
