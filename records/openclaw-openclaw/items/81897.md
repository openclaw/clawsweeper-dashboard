## Summary

Keep open: the PR is not ready to merge because the new registry snapshot cache can reuse results across different plugin load-path scopes, and the external contributor has not supplied after-fix real behavior proof.

## What This Changes

The PR changes child-process env sanitization, SQLite WAL checkpoints, WhatsApp auth-key reads, QMD read-stream cleanup, and plugin registry snapshot caching plus related tests.

## Best Possible Solution

Keep the narrowly proven fixes, make any registry cache key cover every behavior-affecting input or defer registry caching to the canonical registry-performance PRs, and require redacted real setup proof before merge.

## Reproduction Assessment

Yes for the PR-introduced cache bug: source shows `plugins.loadPaths` affects discovery while the new cache key hashes only the narrower policy hash. The original ops findings were not live-reproduced here because the PR body provides tests/typechecks rather than runtime proof.

## Solution Assessment

No, not as written. The cache should include a stable digest of load paths or skip caching whenever config load paths are present, and maintainers should decide whether this registry mitigation belongs beside the existing open registry-performance PRs.
