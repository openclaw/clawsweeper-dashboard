## Summary

Keep this PR open. It has the protected `security` label, lacks contributor-supplied real behavior proof, and the current head still introduces a plugin registry cache key that omits `plugins.loadPaths`, which can cross local plugin visibility scopes during the TTL window.

## What This Changes

The PR changes skill-secret env inheritance, SQLite WAL checkpointing, plugin-registry snapshot caching, WhatsApp auth-key reads, QMD stream cleanup, exec-approval reads, secret comparison, and an OpenGrep workflow guard.

## Best Possible Solution

Keep only narrowly proven fixes, make any registry cache key cover every behavior-affecting input or defer that part to the canonical registry-performance PRs, and require redacted real runtime proof before merge.

## Reproduction Assessment

Yes for the PR-introduced cache defect: source inspection shows current discovery uses `plugins.loadPaths` while the PR cache key hashes only narrower policy inputs. No high-confidence real setup reproduction was supplied for the original ops findings in this review.

## Solution Assessment

No, not as written. The registry cache should include a stable load-path digest or bypass caching when custom load paths are present, and the remaining runtime fixes need real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Key the registry cache by load paths:** `src/plugins/plugin-registry-snapshot.ts:122`
