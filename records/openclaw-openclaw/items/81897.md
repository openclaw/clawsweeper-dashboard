## Summary

Keep open: the PR contains plausible ops fixes, but the new plugin-registry cache has a correctness/security scoping bug and the external contributor has only supplied tests/typechecks, not after-fix real behavior proof.

## What This Changes

The PR changes child-process env sanitization, SQLite WAL checkpoints, WhatsApp auth-key reads, QMD stream cleanup, and plugin registry snapshot caching plus tests.

## Best Possible Solution

Keep only the narrowly proven fixes, make any registry cache key cover every behavior-affecting input or defer that piece to the canonical registry-performance PRs, and require redacted real setup proof before merge.

## Reproduction Assessment

Yes for the PR-introduced cache bug: source shows `plugins.loadPaths` affects discovery while the new cache key hashes only the narrower policy hash. The original ops findings were not live-reproduced here because the contributor provided tests/typechecks rather than runtime proof.

## Solution Assessment

No, not as written. The cache should include a stable digest of load paths or skip caching whenever config load paths are present, and maintainers should decide whether this registry mitigation still belongs beside the existing open registry-performance PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key the registry cache by load paths:** `src/plugins/plugin-registry-snapshot.ts:119`
  - body: The cache key only includes `resolveInstalledPluginIndexPolicyHash(params.config)`, but that hash omits `plugins.loadPaths` while registry assembly passes those load paths into plugin discovery. Two callers with different load paths but the same state/workspace/config policy can receive the first caller's plugin registry during the TTL window, so include a stable load-path digest or bypass caching for custom load paths.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a short-lived cache that can cross plugin registry trust scopes when callers differ only by configured load paths.

  - body: Plugin load paths decide which local plugins are discoverable, but the cache key omits them. A caller using one load-path set can populate a result that another caller with different load paths reuses during the TTL window.
