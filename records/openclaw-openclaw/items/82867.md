## Summary

Keep open. The PR is a plausible fix for the linked doctor diagnostic request and has adequate after-fix proof, but it carries the protected `maintainer` label and the diff has a correctness gap: the new doctor check only scans default state-dir agent stores, so it misses supported `session.store` overrides.

## What This Changes

This PR adds a read-only `openclaw doctor` health contribution that scans cached session snapshot metadata for stale bundled skill runtime paths and adds focused tests.

## Best Possible Solution

Keep the doctor diagnostic direction, but pass the doctor config into the contribution and scan cfg-aware session store targets so default, templated, and custom stores are covered.

## Reproduction Assessment

Yes. The linked issue and proof artifacts show stale `skillsSnapshot.prompt` paths are currently undetected, and the review finding is source-reproducible because the PR scans only default state-dir agent stores despite the documented `session.store` contract.

## Solution Assessment

No, not yet. The proposed diagnostic is the right shape, but it should use cfg-aware session store discovery before it can reliably solve the reported stale-session metadata problem for supported layouts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Honor configured session stores:** `src/commands/doctor-session-snapshots.ts:250`
  - body: The doctor contribution currently discovers stores by walking only `resolveStateDir(...)/agents/*/sessions/sessions.json`. Users can set `session.store` to a custom or templated path, and those are the stores the runtime reads, so `openclaw doctor` would still report nothing for stale snapshots in a supported custom store. Pass the doctor config into this health check and use the existing session-store target resolver, or an equivalent cfg-aware path list, before scanning.
  - confidence: 0.86
