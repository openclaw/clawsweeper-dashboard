## Summary

Keep open: this PR targets the valid canonical write-tool data-loss problem and has sufficient real behavior proof, but it should not merge while malformed append intent can fall through to the destructive overwrite path.

## What This Changes

Adds an optional `append` parameter to the agent `write` tool for host and sandbox writes, expands append/abort/workspace regression tests, and adds a changelog entry.

## Best Possible Solution

Land a maintainer-approved append-capable file primitive that preserves existing workspace and sandbox guards and treats malformed append intent as a validation error, never as permission to overwrite.

## Reproduction Assessment

Yes, by source inspection. Current main routes ordinary host and sandbox writes through upstream overwrite-only `createWriteTool`, while append behavior is limited to memory-flush runs; I did not run a live cron in this read-only review.

## Solution Assessment

No, not as-is. The optional flag is a plausible narrow implementation if maintainers approve it, but the current fall-through for non-boolean append values can still trigger the destructive overwrite path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed append values before overwriting:** `src/agents/pi-tools.read.ts:883-884`
  - body: When the model sends an `append` key with a non-boolean value like `"true"` or `1`, `doAppend` is false and this delegates to the original overwrite implementation. That can truncate the exact shared log the feature is meant to protect, so treat any present non-boolean `append` value as a validation error before calling the base write tool.
  - confidence: 0.88
