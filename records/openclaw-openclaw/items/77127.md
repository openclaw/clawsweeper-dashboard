- `merge-risk: 🚨 session-state`: If consumers rely on the new `append` type through the exported shared factory, existing memory or session files can still be overwritten instead of appended.

## Summary

Keep open: the core append wrapper now has good proof and the prior completed-append abort issue appears fixed, but the patch still mismatches the public SDK/session write contract because `WriteToolInput` advertises `append` while the exported shared write factory remains overwrite-only.

## What This Changes

This PR adds an optional `append` flag to OpenClaw agent write-tool wrappers, updates exported write input/event types, and adds host/sandbox regression tests for append behavior.

## Best Possible Solution

Align one canonical write append contract before merge: either implement append in the shared session write factory/schema/operations and tests, or keep `append` out of SDK-visible types until only the OpenClaw wrapper supports it.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.6.6 expose ordinary `write` as path/content overwrite-only, and the linked issue includes cron plus single-agent truncation evidence. I did not run a destructive live cron reproduction in this read-only review.

## Solution Assessment

No, not yet. The append wrapper is a plausible direction, but the merge shape is not best until the SDK-visible type, shared write schema, exported factory, runtime behavior, and maintainer-approved API shape all agree.
