## Summary

Keep open. The append feature is still a useful narrow fix for the canonical write-tool data-loss problem, but PR head still has an active-operation cancellation gap in the new append path that should block merge.

## What This Changes

Adds an optional `append: true` parameter to the agent `write` tool for host/workspace and sandbox writes, with queueing, regression tests, and a changelog entry.

## Best Possible Solution

Land append-capable writes only after the new path preserves the existing write tool safety envelope: path policy, validation, queue serialization, parent creation, and active cancellation behavior.

## Reproduction Assessment

Yes. Source inspection shows current main is overwrite-only for general `write`, and the PR-specific abort gap is visible at PR head where the append wrapper awaits `ops.appendFile` without active abort handling.

## Solution Assessment

No. The optional append flag is a plausible narrow direction, but this implementation should match upstream `write` cancellation behavior before merge, preferably with active abort handling or a signal-aware append operation plus regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject append calls aborted during active writes:** `src/agents/pi-tools.read.ts:882`
  - body: The append path checks `signal` before queue entry and again when the queued callback starts, but once `ops.appendFile` begins it only awaits completion. If the run is cancelled during sandbox read/mkdir/write or host append, this can still mutate and report success, unlike upstream `createWriteTool`, which rejects while the active queued mutation is in flight. Add active-operation abort handling plus a regression that aborts after `appendFile` starts.
  - confidence: 0.8
